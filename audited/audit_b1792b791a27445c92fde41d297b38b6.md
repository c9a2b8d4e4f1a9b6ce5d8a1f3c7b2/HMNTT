### Title
Unauthenticated Cache Miss Storm via `/block` Endpoint Exhausts DB Connection Pool in `updateOperationAccountAlias()`

### Summary
The `updateOperationAccountAlias()` function in `rosetta/app/services/block_service.go` issues one synchronous `GetAccountAlias()` database query per operation whose account ID is absent from the shared LRU `entityCache`. Because the Rosetta `/block` and `/block/transaction` endpoints require no authentication and have no rate limiting, an unprivileged attacker can drive a sustained stream of cache misses by querying blocks that collectively reference more than `MaxSize` (default 524,288) unique account IDs, continuously evicting warm entries and forcing repeated DB queries that saturate the bounded connection pool and deny service to legitimate callers.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 104–130:

```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // DB query
    ...
    s.entityCache.Set(result.GetId(), result)
}
```

The `entityCache` is a single process-level LRU instance created at startup with capacity `entityCacheConfig.MaxSize` (default **524,288**):

```go
cache.AsLRU[int64, types.AccountId](lru.WithCapacity(entityCacheConfig.MaxSize))
```

`GetAccountAlias()` in `rosetta/app/persistence/account.go` (lines 109–136) executes a raw SQL query against the `entity` table for every miss:

```go
db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity)
```

The middleware stack in `rosetta/main.go` (lines 217–219) applies only metrics, tracing, and CORS wrappers — **no rate limiting, no per-IP throttling, no authentication**:

```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```

**Root cause / failed assumption:** The design assumes the LRU cache will stay warm because real-world traffic reuses a bounded set of popular account IDs. This assumption fails when an attacker deliberately cycles through blocks that collectively reference more unique account IDs than `MaxSize`, keeping the cache perpetually cold.

**Exploit flow:**

1. Attacker enumerates high-activity blocks on the public chain (all block data is public).
2. Attacker issues concurrent `POST /block` requests for blocks whose union of operation account IDs exceeds 524,288 unique values.
3. Each request enters `updateOperationAccountAlias()`, finds no cache entry for each account ID, and fires a synchronous `GetAccountAlias()` DB query.
4. LRU eviction ensures that entries added by one request are evicted by the next batch of unique IDs, so the cache never warms up.
5. The DB connection pool (`maxOpenConnections = 100`, `rosetta/app/config/types.go` Pool struct, documented default) is saturated by concurrent queries.
6. Legitimate requests queue behind the pool limit and eventually time out (DB statement timeout = 20 s).

### Impact Explanation

The DB connection pool is the hard resource ceiling. With 100 open connections and each `GetAccountAlias()` query holding a connection for up to 20 seconds under load, an attacker sending ~5–10 concurrent `/block` requests against blocks with many operations can fully occupy the pool. All other Rosetta API endpoints (`/account/balance`, `/construction/*`, etc.) share the same DB client and are equally starved. The result is effective denial of service of the entire Rosetta API for the duration of the attack — no privileged access required.

### Likelihood Explanation

The attack requires only:
- Network access to the Rosetta port (default 5700, publicly exposed by design for exchange integrations).
- Knowledge of which blocks contain many unique account IDs (derivable from any Hedera block explorer or the mirror node's own REST API).
- A simple HTTP client capable of concurrent POST requests.

No credentials, no special protocol knowledge, and no on-chain funds are needed. The attack is trivially repeatable and can be sustained indefinitely.

### Recommendation

1. **Add per-IP / global request rate limiting** as a middleware layer before the router in `rosetta/main.go`, e.g. using `golang.org/x/time/rate` or a reverse-proxy rule.
2. **Batch the alias lookup**: replace the per-operation loop with a single `SELECT … WHERE id IN (…)` query for all uncached IDs in a block, reducing round-trips from O(operations) to O(1) per block request.
3. **Negative-cache misses**: store a sentinel for account IDs that have no alias so repeated queries for the same non-aliased account do not hit the DB.
4. **Limit concurrent in-flight DB queries** with a semaphore in `updateOperationAccountAlias()` to prevent pool exhaustion even if rate limiting is bypassed.
5. **Consider increasing `maxOpenConnections`** only as a short-term palliative; it does not fix the root cause.

### Proof of Concept

```bash
# 1. Identify a block with many unique account IDs (e.g. via mirror REST API)
BLOCK=12345678   # block with 200+ transactions, each with multiple operations

# 2. Fire concurrent /block requests with no authentication
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$((BLOCK + i))}}" &
done
wait

# 3. Observe: DB connection pool saturated, subsequent legitimate requests
#    return errors or hang until the 20-second statement timeout fires.
# Repeat continuously to sustain the denial of service.
```
