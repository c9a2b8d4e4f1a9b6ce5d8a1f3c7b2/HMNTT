### Title
Unauthenticated gRPC Subscription Flood Exhausts `boundedElastic` Scheduler, Disrupting Legitimate Subscribers

### Summary
The `subscribeTopic` gRPC endpoint accepts connections from any unauthenticated external caller with no global subscription cap. Each subscription routes its downstream message processing through `publishOn(Schedulers.boundedElastic())` in `SharedTopicListener.listen()`, consuming a thread from the JVM-wide bounded elastic pool (capped at `10 × CPU cores` by default). An attacker opening many connections and subscribing to active topics can saturate this pool, queuing or stalling message delivery for all legitimate subscribers on the same scheduler.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` — `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java` lines 43–53 — accepts any inbound `ConsensusTopicQuery` with zero authentication or identity checks. It maps the request to a `TopicMessageFilter` and calls `topicMessageService.subscribeTopic(filter)`.

`TopicMessageServiceImpl.subscribeTopic()` — `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java` lines 59–91 — tracks a `subscriberCount` gauge but enforces **no maximum**; it is a metric only.

`SharedTopicListener.listen()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedTopicListener.java` lines 21–26:

```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)
            .doOnSubscribe(s -> log.info("Subscribing: {}", filter))
            .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
            .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
}
```

Every individual subscriber — regardless of whether it shares the upstream Redis channel with others — receives its own `publishOn(Schedulers.boundedElastic())` operator. In Project Reactor, `publishOn` dispatches downstream signal processing (each `onNext`, `onError`, `onComplete`) to a worker thread from the named scheduler. When messages are actively flowing, each subscriber holds a worker thread for the duration of its processing window.

`Schedulers.boundedElastic()` is a JVM-wide singleton with a hard thread cap of `10 × Runtime.getRuntime().availableProcessors()` (e.g., 40 threads on a 4-core host). When the cap is reached, new tasks are queued (up to 100,000 by default). Tasks queued behind attacker-owned subscriptions delay `onNext` delivery for every other subscriber sharing the same scheduler — including legitimate ones.

**Why the sharing in `RedisTopicListener` does not mitigate this:**

`RedisTopicListener.getSharedListener()` (lines 59–62) uses `computeIfAbsent` + `.share()` to create one upstream Redis channel per topic. This reduces Redis connections, but each subscriber still gets its own independent `publishOn` call in `listen()`. One hundred subscribers to the same topic produce one Redis subscription but one hundred separate scheduler work items per message burst.

**Why existing guards are insufficient:**

`NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14) limits calls per TCP connection, but an attacker can open an arbitrary number of connections — there is no per-IP connection limit or global connection cap configured in `GrpcConfiguration` (lines 28–35). With 200 connections the attacker controls 1,000 concurrent subscriptions.

### Impact Explanation

On a typical 4-core deployment the bounded elastic pool holds 40 threads. An attacker with 40+ active subscriptions to topics receiving real message traffic saturates the pool. All subsequent `onNext` dispatches — including those of legitimate subscribers — are queued. Message delivery latency grows unboundedly until the attacker disconnects. At higher subscription counts (hundreds), the 100,000-task queue itself can fill, causing `RejectedExecutionException` and stream termination for legitimate subscribers. Additionally, each subscription allocates an `onBackpressureBuffer` of up to 16,384 message slots; 500 subscriptions can consume hundreds of MB of heap.

Severity: **High** — complete disruption of real-time topic message delivery for all users of the affected mirror node instance, with no privileged access required.

### Likelihood Explanation

The gRPC port (default 5600) is publicly reachable by design (it is the consumer-facing HCS streaming API). No credentials, API keys, or tokens are required to call `subscribeTopic`. The attack requires only a gRPC client library and the ability to open TCP connections — both freely available. It is fully repeatable and scriptable. The attacker does not need to know valid topic IDs in advance; subscribing to any topic that receives messages (e.g., high-traffic mainnet topics) is sufficient.

### Recommendation

1. **Enforce a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable threshold, returning `RESOURCE_EXHAUSTED` to the caller.
2. **Add per-IP or per-connection subscription limits** at the Netty layer or via a gRPC interceptor, complementing the existing `maxConcurrentCallsPerConnection` setting.
3. **Isolate subscriber scheduler resources**: use a named, separately-sized `Schedulers.newBoundedElastic(...)` instance for topic listener dispatch rather than the JVM-wide default, so exhaustion does not affect other reactor pipelines.
4. **Consider token-bucket rate limiting** on new subscription establishment (analogous to the `ThrottleManagerImpl` used in the web3 module).

### Proof of Concept

```
# Prerequisites: grpcurl or any gRPC client; a mirror node with active topic traffic.

# Step 1 – open many connections and subscribe from each (bash loop):
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{
    "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 1234},
    "consensusStartTime": {"seconds": 0}
  }' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 2 – observe on the server:
#   - bounded elastic thread pool saturates (visible via JVM thread dump or
#     micrometer executor metrics)
#   - legitimate subscriber latency increases; at high counts streams terminate
#     with RejectedExecutionException or backpressure ERROR signal

# Step 3 – kill attacker processes; legitimate subscriber latency recovers,
#   confirming the scheduler was the bottleneck.
```
