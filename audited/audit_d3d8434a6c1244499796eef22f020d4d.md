### Title
Unauthenticated Contradictory Filter Combination Causes Full Table Scan DoS on `/api/v1/blocks`

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts both `block.number` and `timestamp` query parameters simultaneously but performs no cross-filter contradiction detection. An unprivileged attacker can submit requests with logically impossible filter combinations (e.g., `block.number=gte:0&timestamp=lte:1`) that force PostgreSQL to perform a full sequential index scan of the `record_file` table on every request, returning zero rows. Because the REST API's rate-limiting middleware is disabled by default, this can be repeated at scale to degrade database performance.

### Finding Description

**Code path:**

`getBlocks()` at [1](#0-0)  calls `utils.buildAndValidateFilters()`, then `extractSqlFromBlockFilters()`, then `RecordFileService.getBlocks()`.

`buildAndValidateFilters()` at [2](#0-1)  validates each filter individually and calls `filterDependencyCheck`. That dependency checker at [3](#0-2)  only rejects `block.hash + block.number` combinations and `transaction.index` without a block anchor. It contains **no check for contradictory `block.number` + `timestamp` combinations**.

`filterValidityChecks` at [4](#0-3)  confirms that `block.number` accepts all `basicOperators` (eq, gt, gte, lt, lte) with any positive-or-zero long value, and `timestamp` at [5](#0-4)  accepts any valid timestamp value. Both are independently valid.

`extractSqlFromBlockFilters()` at [6](#0-5)  blindly maps `block.number` → `index` column condition and `timestamp` → `consensus_end` column condition with no cross-filter sanity check.

`RecordFileService.getBlocks()` at [7](#0-6)  constructs and executes:

```sql
SELECT ... FROM record_file
WHERE index >= 0 AND consensus_end <= 1
ORDER BY index DESC
LIMIT 25
```

Because `consensus_end` is always a large positive nanosecond timestamp (never ≤ 1), PostgreSQL must scan the entire `record_file` index on `index` before concluding zero rows satisfy both predicates. The `LIMIT 25` clause provides no early exit because no row ever matches. The indexes involved are confirmed at [8](#0-7) .

**Root cause:** The system assumes that individually valid filter parameters are collectively meaningful. There is no guard that detects when the intersection of `index` and `consensus_end` ranges is provably empty before issuing the query.

### Impact Explanation

The `record_file` table grows at approximately one row per consensus round (~1/second on mainnet), reaching tens of millions of rows over years of operation. A full index scan on such a table is orders of magnitude more expensive than a normal paginated query (which uses the index to return 25 rows immediately). Each crafted request forces the DB to do maximum work and return nothing. Sustained at even modest request rates (hundreds/second from a single host or distributed botnet), this can saturate DB I/O and CPU, degrading or denying service to all other API consumers.

### Likelihood Explanation

The endpoint is public and requires no credentials. The attack requires only knowledge of the public API spec (documented in `rest/api/v1/openapi.yml` at [9](#0-8) ). The Traefik rate-limiting middleware for the REST service is **opt-in and disabled by default** (`global.middleware: false`) as seen at [10](#0-9) , unlike the Rosetta service which has explicit `rateLimit` middleware. The application-level throttle manager at [11](#0-10)  is in the `web3` module and does not apply to the REST API. The attack is trivially scriptable, repeatable, and requires no special tooling.

### Recommendation

1. **Add cross-filter contradiction detection** in `filterDependencyCheck` or `extractSqlFromBlockFilters`: if both `block.number` and `timestamp` range filters are present, validate that the implied ranges are not provably disjoint (e.g., reject if `block.number=gte:X` combined with `timestamp=lte:Y` where Y is below the minimum possible timestamp for block X).
2. **Enable rate limiting by default** for the REST API Helm chart (`global.middleware: true`) with per-IP request rate caps, consistent with the Rosetta service configuration.
3. **Add a query timeout** at the DB connection pool level so runaway full-scan queries are killed before they exhaust DB resources.

### Proof of Concept

```bash
# Single contradictory request - forces full record_file index scan, returns []
curl "https://<mirror-node-host>/api/v1/blocks?block.number=gte:0&timestamp=lte:1"

# At scale - flood with contradictory requests from a single unauthenticated client
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/blocks?block.number=gte:0&timestamp=lte:1" &
done
wait
```

Expected result: each request returns `{"blocks":[],"links":{"next":null}}` with HTTP 200, while the database executes a full scan of the `record_file` table for every request. DB CPU and I/O climb proportionally with request rate, degrading all concurrent API consumers.

const filterDependencyCheck = (query) => {
  const badParams = [];
  let containsBlockNumber = false;
  let containsBlockHash = false;
  let containsTransactionIndex = false;
  for (const key of Object.keys(query)) {
    if (key === constants.filterKeys.TRANSACTION_INDEX) {
      containsTransactionIndex = true;
    } else if (key === constants.filterKeys.BLOCK_NUMBER) {
      containsBlockNumber = true;
    } else if (key === constants.filterKeys.BLOCK_HASH) {
      containsBlockHash = true;
    }
  }

  if (containsTransactionIndex && !(containsBlockNumber || containsBlockHash)) {
    badParams.push({
      key: constants.filterKeys.TRANSACTION_INDEX,
      error: 'transaction.index requires block.number or block.hash filter to be specified',
      code: 'invalidParamUsage',
    });
  }

  if (containsBlockHash && containsBlockNumber) {
    badParams.push({
      key: constants.filterKeys.BLOCK_HASH,
      error: 'cannot combine block.number and block.hash',
      code: 'invalidParamUsage',
    });
  }

  if (badParams.length) {
    throw InvalidArgumentError.forRequestValidation(badParams);
  }
