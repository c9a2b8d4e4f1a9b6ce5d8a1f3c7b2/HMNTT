### Title
Unbounded DB Queries via Non-Existent Historical Block Numbers Due to Cache Miss on Empty Optional

### Summary
`ContractCallService.callContract()` sets a lazy `blockSupplier` that calls `recordFileService.findByBlockType()` → `recordFileRepository.findByIndex()` on every request for a numeric block parameter. The `@Cacheable` annotation on `findByIndex()` uses `unless = "#result == null"`, but Spring Cache unwraps `Optional` return types and treats `Optional.empty()` as null — so non-existent block numbers are **never cached**. Combined with the gas throttle being restored for failed calls, an unprivileged attacker can sustain a stream of DB queries at the full request-rate limit with near-zero gas cost.

### Finding Description

**Code path:**

`ContractController.call()` → `throttleManager.throttle(request)` → `contractExecutionService.processCall()` → `ContractCallService.callContract()` (lines 100–107) sets:

```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)));
```

`RecordFileServiceImpl.findByBlockType()` (lines 19–26) routes any numeric block to:

```java
return recordFileRepository.findByIndex(block.number());
```

`RecordFileRepository.findByIndex()` is annotated:

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
```

**Root cause — cache miss on empty Optional:**
Spring Cache (4.3+) unwraps `Optional` return values before caching. An `Optional.empty()` result is treated as `null` by the caching infrastructure. Because the condition is `unless = "#result == null"`, an empty Optional (non-existent block) is **never stored in the cache**. Every subsequent request for the same non-existent block number re-executes the SQL query against the database.

**Gas throttle bypass:**
In `ContractCallService.doProcessCall()` (lines 127–135), the `finally` block calls `restoreGasToBucket(result, params.getGas())`. When `BlockNumberNotFoundException` is thrown (result is null), the method restores `gasLimitToRestoreBaseline` tokens:

```java
if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
    throttleManager.restore(gasLimitToRestoreBaseline);
}
```

This means the gas bucket is replenished for every failed historical-block call, leaving only the `requestsPerSecond` rate limit as an effective throttle.

**Existing checks reviewed and shown insufficient:**
- `CACHE_MANAGER_RECORD_FILE_INDEX` cache (10-minute TTL, max 10,000 entries) only caches successful hits; misses are uncached.
- Gas throttle is restored for failed calls, so it does not bound the DB query rate.
- `requestsPerSecond` rate limit is the only remaining control, and it applies globally across all callers.

### Impact Explanation
Each request with a distinct non-existent block number (e.g., `0xFFFFFFFF`, `0xFFFFFFFE`, …) executes `SELECT r FROM RecordFile r WHERE r.index = ?1` against the database without any cache benefit. At the `requestsPerSecond` ceiling, an attacker can sustain a continuous stream of uncached DB queries. On a node where the `record_file` table is large or DB connection pool is constrained, this produces measurable I/O amplification. Because gas is restored, the attacker's effective cost per DB query approaches zero, making the attack cheap to sustain over the 24-hour window required to demonstrate 30%+ resource increase.

### Likelihood Explanation
No authentication or special privilege is required — the `/api/v1/contracts/call` endpoint is public. The attacker only needs to supply a valid JSON body with a numeric `block` field set to a non-existent block number. Rotating through a large space of non-existent block numbers (e.g., `Long.MAX_VALUE - N`) trivially avoids any future cache population. The attack is fully scriptable and repeatable.

### Recommendation
1. **Cache negative results explicitly:** Remove `unless = "#result == null"` from `findByIndex()` and replace with `unless = "#result != null && !#result.isPresent()"` — or use a sentinel value — so that non-existent block lookups are also cached (with a short TTL, e.g., 30 seconds).
2. **Validate block range before DB query:** In `RecordFileServiceImpl.findByBlockType()`, reject block numbers that exceed the current chain tip (obtainable from the already-cached `findLatest()`) before issuing the `findByIndex()` query.
3. **Do not restore gas for pre-execution failures:** Distinguish between EVM execution failures (gas should be restored) and pre-execution validation failures like `BlockNumberNotFoundException` (gas should not be restored), so the gas throttle remains effective.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
