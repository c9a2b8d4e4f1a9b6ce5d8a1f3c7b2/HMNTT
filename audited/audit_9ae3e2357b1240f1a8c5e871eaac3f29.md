### Title
Account Enumeration via Differential 404 Response on EVM Address Lookup in `getNftAllowances`

### Summary
The `getNftAllowances` endpoint at `/api/v1/accounts/{id}/allowances/nfts` returns a distinct HTTP 404 with the message `"No account found for the given ID"` when an EVM address is supplied as `{id}` and no matching entity exists in the database. Because no authentication or rate limiting is enforced on this public endpoint, any unauthenticated attacker can systematically probe arbitrary EVM addresses and use the 200 vs. 404 differential to determine which addresses correspond to registered Hedera accounts.

### Finding Description

**Exact code path:**

1. `AllowancesController.getNftAllowances()` receives `{id}` as an `EntityIdParameter` path variable. [1](#0-0) 

2. The request is forwarded to `NftAllowanceServiceImpl.getNftAllowances()`, which calls `entityService.lookup(request.getAccountId())`. [2](#0-1) 

3. Inside `EntityServiceImpl.lookup()`, when the parameter is an `EntityIdEvmAddressParameter`, it queries the database via `entityRepository.findByEvmAddress(p.evmAddress())`. If the result is empty, it unconditionally throws `EntityNotFoundException("No account found for the given ID")`, which Spring maps to HTTP 404. [3](#0-2) 

**Root cause and failed assumption:**

The code assumes that a missing EVM address entity should be surfaced as a 404 error. This is intentional for usability but creates a binary oracle: the HTTP status code directly reveals whether an EVM address is a registered account. By contrast, numeric IDs bypass the DB lookup entirely (by design for partial mirror nodes), returning 200 with empty allowances regardless of whether the account exists. [4](#0-3) 

**Why existing checks are insufficient:**

- No authentication is required on the endpoint.
- No rate limiting is applied at the controller or service layer for this endpoint.
- The test suite explicitly confirms the 404 behavior for EVM addresses with no entity, treating it as expected behavior rather than a security concern. [5](#0-4) 

### Impact Explanation
An attacker can enumerate which EVM addresses correspond to active Hedera accounts by observing the HTTP response code:
- **200** → account exists (allowances list returned, possibly empty)
- **404 + "No account found for the given ID"** → account does not exist

This leaks account existence information for any EVM address, enabling targeted phishing, social engineering, or pre-attack reconnaissance. The impact is **information disclosure** (account existence oracle), rated **Medium** severity — it does not directly allow fund theft but meaningfully reduces attacker effort for follow-on attacks.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. An attacker needs only a list of candidate EVM addresses (trivially generated or sourced from public blockchain data) and a simple HTTP client. The attack is fully automatable, repeatable, and requires no special privileges or insider knowledge. No exploit code or special tooling is needed beyond `curl` or a script.

### Recommendation
1. **Uniform response for missing EVM address accounts**: Return HTTP 200 with an empty allowances list (matching the behavior for numeric IDs) instead of 404 when an EVM address has no corresponding entity. This eliminates the oracle.
2. **Alternatively**, if 404 is required for EVM addresses, apply rate limiting (e.g., via a gateway or Spring's `RateLimiter`) on the `/api/v1/accounts/{id}/allowances/*` endpoints to make bulk enumeration impractical.
3. **Do not** rely solely on rate limiting — the root fix is response normalization.

### Proof of Concept

```bash
# Step 1: Probe a known-existing EVM address (account exists)
curl -s -o /dev/null -w "%{http_code}" \
  "https://<mirror-node>/api/v1/accounts/0x000000000000000000000000000000000000abcd/allowances/nfts"
# → 200

# Step 2: Probe a non-existing EVM address (account does not exist)
curl -s -o /dev/null -w "%{http_code}" \
  "https://<mirror-node>/api/v1/accounts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/allowances/nfts"
# → 404, body: {"_status":{"messages":[{"message":"No account found for the given ID"}]}}

# Step 3: Automate over a list of candidate addresses
for addr in $(cat evm_addresses.txt); do
  code=$(curl -s -o /dev/null -w "%{http_code}" \
    "https://<mirror-node>/api/v1/accounts/0x${addr}/allowances/nfts")
  echo "$addr -> $code"
done
# 200 = registered account, 404 = not registered
```

No authentication, no rate limit, no special precondition required. Any internet-accessible mirror node deployment is affected.

```java
    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
```
