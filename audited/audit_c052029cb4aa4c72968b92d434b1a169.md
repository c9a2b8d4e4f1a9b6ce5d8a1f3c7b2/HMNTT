### Title
Unauthenticated Timestamp-Filtered Contract State Query Triggers Unsatisfiable Mixed-Direction Sort, Enabling Resource Exhaustion

### Summary
Any unauthenticated user can request `GET /api/v1/contracts/{contractId}/state?timestamp=lte:{ts}`, causing `getContractStateByIdAndFilters()` to execute `contractStateTimestampQuery` with `DISTINCT ON (slot)` and `ORDER BY slot ASC, consensus_timestamp DESC`. The existing index `(contract_id, slot, consensus_timestamp)` is entirely ASC and cannot satisfy this mixed-direction sort, forcing PostgreSQL to perform a full in-memory sort of every `contract_state_change` row for the target contract before applying the LIMIT. The REST API's 20-second statement timeout and 10-connection pool amplify the impact: a small number of concurrent requests against a high-volume contract can saturate the database CPU and exhaust the connection pool.

### Finding Description

**Exact code path:**

`rest/service/contractService.js` lines 111–122 define `contractStateTimestampQuery`: [1](#0-0) 

`getContractStateByIdAndFilters()` at lines 245–267 detects a timestamp filter and builds the ORDER BY as `slot ASC, consensus_timestamp DESC`: [2](#0-1) 

The resulting SQL is:
```sql
WITH entity AS (...)
SELECT DISTINCT ON (slot)
    contract_id, slot, evm_address,
    coalesce(value_written, value_read) AS value,
    consensus_timestamp AS modified_timestamp
FROM contract_state_change csc
LEFT JOIN entity e ON e.id = csc.contract_id
WHERE csc.contract_id = $1 AND csc.consensus_timestamp <= $2
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT $3
```

**Root cause — mixed-direction sort defeats the index:**

The only relevant index is: [3](#0-2) 

This index stores `(contract_id ASC, slot ASC, consensus_timestamp ASC)`. PostgreSQL's B-tree index can be scanned forward (`slot ASC, consensus_timestamp ASC`) or entirely backward (`slot DESC, consensus_timestamp DESC`), but **not** in mixed directions (`slot ASC, consensus_timestamp DESC`). The planner therefore falls back to a sequential scan of all rows matching `contract_id = $1 AND consensus_timestamp <= $2`, followed by an O(N log N) sort. Because `DISTINCT ON` must see every row in each slot group before it can emit a result, the `LIMIT $3` clause is applied only after the full sort completes — it provides zero protection against the sort cost.

**Failed assumption:** The index added in migration `V1.77.1` and carried into `V2.0.3` was intended to support this query, but the query's `ORDER BY` direction was later changed to include `consensus_timestamp DESC` (to retrieve the latest state per slot), making the index unusable for the sort. [4](#0-3) 

**Controller entry point** (no authentication required): [5](#0-4) 

### Impact Explanation

The REST API database pool has `maxConnections = 10` and `statementTimeout = 20,000 ms`: [6](#0-5) [7](#0-6) 

A popular DeFi contract on Hedera mainnet accumulates millions of `contract_state_change` rows. Each timestamp-filtered request holds a connection for up to 20 seconds while PostgreSQL sorts all matching rows. With 10 connections, 10 concurrent requests fully exhaust the pool, blocking all other REST API queries. Simultaneously, the sort operations drive database CPU to saturation. Because the `contract_state_change` table is partitioned by `consensus_timestamp`, a request with `timestamp=lte:<current_time>` spans all partitions, maximising row count. [8](#0-7) 

### Likelihood Explanation

**Preconditions:** None. No account, API key, or authentication is required. The attacker only needs to know (or enumerate) a contract ID with a large history — publicly visible on any Hedera block explorer.

**Feasibility:** A single HTTP request is sufficient to trigger the expensive sort. Sending 10 concurrent requests (well below any brute-force threshold) exhausts the connection pool. The 20-second timeout means each "wave" of 10 requests occupies the pool for up to 20 seconds, during which the node is unresponsive to all REST clients.

**Repeatability:** The attack is trivially repeatable with a simple script or even `curl` in a loop. No special tooling is required.

### Recommendation

1. **Fix the index direction** to match the query's actual sort order. Replace the existing index with a composite index that covers the mixed-direction sort:
   ```sql
   CREATE INDEX contract_state_change__id_slot_desc_ts_desc
       ON contract_state_change (contract_id, slot ASC, consensus_timestamp DESC);
   ```
   With this index, PostgreSQL can satisfy `DISTINCT ON (slot) ORDER BY slot ASC, consensus_timestamp DESC` via an index-only scan, making the query O(distinct_slots) instead of O(N log N).

2. **Reduce `statementTimeout`** for the REST API database user from 20,000 ms to a value consistent with the web3 API (3,000 ms), limiting the blast radius of any slow query. [9](#0-8) 

3. **Add per-endpoint rate limiting** for `/contracts/:contractId/state` with a timestamp parameter, similar to the Traefik `inFlightReq` middleware already used for the Rosetta API.

4. **Increase the connection pool size** or add a circuit breaker so that slow queries on one endpoint cannot starve all other REST API consumers.

### Proof of Concept

```bash
# Identify a high-volume contract (e.g., a known DeFi contract on mainnet)
CONTRACT_ID="0.0.1234567"
MIRROR_NODE="https://mainnet-public.mirrornode.hedera.com"

# Send 10 concurrent timestamp-filtered requests
for i in $(seq 1 10); do
  curl -s "${MIRROR_NODE}/api/v1/contracts/${CONTRACT_ID}/state?timestamp=lte:9999999999.999999999&limit=100" &
done
wait

# Observe: all 10 requests hold DB connections for ~20 seconds (statement timeout),
# exhausting the 10-connection pool. Concurrent REST API requests return 503 or hang.
# Database CPU spikes due to parallel full-table sorts on contract_state_change.
```
