### Title
Unauthenticated `getNodes()` gRPC Endpoint Lacks Rate Limiting, Enabling Resource Exhaustion DoS

### Summary
The `getNodes()` handler in `NetworkController` accepts unlimited, unauthenticated requests with no per-IP or global rate limiting. Each call drives multiple synchronous database queries and holds a HikariCP connection for the full paging duration. An attacker opening many parallel gRPC connections can exhaust the database connection pool and degrade or deny service for all users.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/controller/NetworkController.java`, `getNodes()`, lines 33–43:

```java
public void getNodes(final AddressBookQuery request, final StreamObserver<NodeAddress> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(networkService::getNodes)   // ← unbounded DB pipeline, no throttle
            ...
``` [1](#0-0) 

No authentication, no rate-limiting interceptor, and no per-IP connection cap is applied before this call.

**Root cause — what `networkService::getNodes` does per call:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/NetworkServiceImpl.java`, lines 55–77:

1. `addressBookRepository.findLatestTimestamp()` — DB query
2. `nodeStakeRepository.findLatestTimestamp()` — DB query
3. `nodeStakeRepository.findAllStakeByConsensusTimestamp()` — DB query
4. Iterative paging loop via `addressBookEntryRepository.findByConsensusTimestampAndNodeId()` with a **250 ms sleep between pages** (`pageDelay`), holding a HikariCP connection for the entire duration. [2](#0-1) 

With `limit=0` (the default when the field is omitted), `take(Long.MAX_VALUE)` is used, meaning all nodes are returned and the connection is held until the full address book is streamed. [3](#0-2) 

**Why existing checks are insufficient:**

The only server-side guard is `maxConcurrentCallsPerConnection = 5` (default): [4](#0-3) 

This is enforced per HTTP/2 connection, not per IP and not globally: [5](#0-4) 

An attacker opens N connections → N × 5 concurrent `getNodes` calls, each holding a DB connection for seconds.

The GCP gateway `maxRatePerEndpoint: 250` is the only other candidate, but it is **disabled by default** (`global.gateway.enabled: false`): [6](#0-5) 

The gRPC module has **no application-level throttle** — the `ThrottleConfiguration`/`ThrottleManagerImpl` infrastructure exists only in the `web3` module and is never wired into the gRPC service.

The sole gRPC interceptor (`GrpcInterceptor`) only sets an endpoint context label; it performs no rate limiting: [7](#0-6) 

### Impact Explanation

Each concurrent `getNodes(limit=0)` call holds a HikariCP database connection for the full paging duration (number\_of\_pages × 250 ms + query time). With enough parallel connections, the pool is exhausted. Legitimate `subscribeTopic` and `getNodes` calls from real clients receive connection-timeout errors. The `GrpcHighDBConnections` alert fires at 75% utilization — by that point service is already degraded. This is a full availability impact (DoS) against the public mirror node gRPC API. [8](#0-7) 

### Likelihood Explanation

Preconditions: none. The endpoint is publicly reachable on port 5600 with no authentication. Any standard gRPC client (e.g., `grpcurl`, the Hedera Java/Go SDK) can open many HTTP/2 connections. The attack is trivially scriptable, repeatable, and requires no privileged access or leaked credentials. The `AddressBookQuery` proto requires only a valid `file_id` (either `0.0.101` or `0.0.102`), which is publicly documented. [9](#0-8) 

### Recommendation

1. **Add a global gRPC rate-limiting interceptor** (`@GlobalServerInterceptor`) that enforces a per-IP and/or global RPS cap on `getNodes` calls, analogous to the `ThrottleManagerImpl` in the `web3` module.
2. **Enforce a maximum `limit` value** in `AddressBookFilter` (e.g., cap at 1000) to bound per-request DB work; reject `limit=0` or treat it as the cap rather than `Long.MAX_VALUE`.
3. **Set `maxInboundConnections`** on the `NettyServerBuilder` to cap total simultaneous connections regardless of source IP.
4. **Enable the GCP gateway** with `maxRatePerEndpoint` in production deployments and ensure it is not optional for public-facing instances.

### Proof of Concept

```bash
# Open 50 parallel gRPC connections, each sending getNodes with limit=0
# (returns all nodes, holds DB connection for full paging duration)
for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes \
    > /dev/null &
done
wait

# Expected result: DB connection pool exhausted (HikariCP active connections
# approach max), legitimate requests begin timing out or receiving UNAVAILABLE
# gRPC status codes. GrpcHighDBConnections alert fires.
```

    public Flux<AddressBookEntry> getNodes(AddressBookFilter filter) {
        var fileId = filter.getFileId();
        if (!getValidFileIds().contains(fileId)) {
            throw new IllegalArgumentException(INVALID_FILE_ID);
        }

        long addressBookTimestamp = addressBookRepository
                .findLatestTimestamp(fileId.getId())
                .orElseThrow(() -> new EntityNotFoundException(fileId));
        long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp().orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
        var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
        var context = new AddressBookContext(addressBookTimestamp, nodeStakeMap);

        return Flux.defer(() -> page(context))
                .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                        .jitter(0.5)
                        .withFixedDelay(addressBookProperties.getPageDelay())
                        .withScheduler(Schedulers.boundedElastic()))
                .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Querying for address book: {}", filter))
                .doOnComplete(() -> log.info("Retrieved {} nodes from the address book", context.getCount()));
    }
