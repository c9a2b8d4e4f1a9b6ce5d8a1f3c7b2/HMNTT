### Title
GraphQL `account(evmAddress)` Endpoint Lacks Application-Level Rate Limiting, Enabling DB Connection Pool Exhaustion via Valid-Format Non-Existent Address Flood

### Summary
The `account(input: {evmAddress: ...})` GraphQL query accepts any address matching `^(0x)?[a-fA-F0-9]{40}$` and unconditionally issues a native SQL query against the `entity` table for every request. The GraphQL module has no application-level rate limiting (unlike the `web3` module which has `ThrottleManager`), and the only protection is a Traefik `inFlightReq: amount: 5` per-IP middleware that is trivially bypassed with multiple source IPs. An unprivileged attacker can sustain enough concurrent DB queries to exhaust the HikariCP connection pool, degrading or denying service to legitimate users.

### Finding Description

**Code path:**

1. Schema: `evmAddress: String @Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")` — validates format only, not existence. [1](#0-0) 

2. Controller dispatches to service unconditionally when `evmAddress != null`: [2](#0-1) 

3. Service decodes the address and, for any non-zero-prefix address (i.e., a real EVM address like `0xffffffffffffffffffffffffffffffffffffffff`), calls `findByEvmAddress`: [3](#0-2) 

4. Repository issues a native SQL query that acquires a HikariCP connection for its duration: [4](#0-3) 

**Root cause — missing application-level rate limiting in the GraphQL module:**

The `web3` module has a full `ThrottleManager` with bucket4j rate limiting: [5](#0-4) 

The GraphQL module has **no equivalent**. A search of `graphql/**` for `rateLimit`, `throttle`, or `inFlightReq` returns zero matches in application code. The only protection is the Traefik middleware: [6](#0-5) 

This limits 5 concurrent requests **per source IP**, which is bypassable with ≥2 source IPs. Furthermore, `global.middleware: false` is the default, meaning the middleware may not be active in all deployments: [7](#0-6) 

**Why existing GraphQL-level controls are insufficient:**

- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` limit query structure, not request rate: [8](#0-7) 
- The `PreparsedDocumentProvider` caches parsed query documents, not query results — every unique EVM address still hits the DB.
- The `@Pattern` directive only rejects malformed addresses; a valid-format non-existent address passes all checks and reaches the DB.

### Impact Explanation

HikariCP's default maximum pool size is 10 connections (no override is set in the GraphQL module's configuration): [9](#0-8) 

With 10 concurrent DB queries holding connections (achievable from just 2 IPs at 5 concurrent each), the pool is saturated. New legitimate requests queue for a connection and time out. The `statementTimeout` for the GraphQL module is 10,000 ms, meaning each connection can be held for up to 10 seconds per query in a slow-DB scenario. This constitutes a denial-of-service against the GraphQL service's database connectivity. The Prometheus alert `GraphQLHighDBConnections` (threshold: 75% utilization for 5 minutes) confirms the operators themselves recognize this as a critical failure mode: [10](#0-9) 

### Likelihood Explanation

- **No authentication required**: the `/graphql/alpha` endpoint is public.
- **No special knowledge needed**: any valid hex string of 40 characters suffices.
- **Bypass is trivial**: the per-IP `inFlightReq: 5` limit requires only 2 source IPs (e.g., two cheap VPS instances or a small botnet) to saturate a 10-connection pool.
- **Middleware may be disabled**: `global.middleware: false` is the default, meaning many deployments may have no Traefik protection at all.
- **Repeatable and automatable**: a simple script with `curl` or `ab` from two hosts is sufficient.

### Recommendation

1. **Add application-level rate limiting to the GraphQL module**, mirroring the `web3` module's `ThrottleManager` / bucket4j pattern. Apply a global requests-per-second cap and optionally a per-IP cap.
2. **Set an explicit HikariCP `maximumPoolSize`** in the GraphQL application configuration and tune it with the rate limit in mind.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) and add a rate-limit middleware (not just `inFlightReq`) to bound total request throughput, not just per-IP concurrency.
4. **Add a result cache** (e.g., Caffeine with short TTL) for `findByEvmAddress` lookups so repeated misses for the same address do not each hit the DB.

### Proof of Concept

```bash
# From two separate hosts (host1 and host2), run simultaneously:
# This bypasses the per-IP inFlightReq:5 limit and saturates a 10-connection pool.

for i in $(seq 1 5); do
  curl -s -X POST https://<target>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{account(input:{evmAddress:\\\"$(printf '%040x' $RANDOM$RANDOM$RANDOM)\\\"}){id}}\"}" &
done
wait
```

Run the above loop continuously from 2+ hosts. Each request passes `@Pattern` validation, reaches `EntityRepository.findByEvmAddress`, acquires a HikariCP connection, and executes `SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE`. With the pool at 10 connections and no application-level rate limiting, the pool exhausts within seconds, causing `SQLTransientConnectionException: Connection is not available, request timed out` for legitimate users.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
