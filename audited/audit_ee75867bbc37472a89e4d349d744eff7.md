### Title
Unauthenticated gRPC `getNodes()` Allows N-fold DB Amplification via Concurrent Connections

### Summary
The `getNodes()` handler in `NetworkController` accepts requests from any unauthenticated caller and immediately fans out to `NetworkServiceImpl.getNodes()`, which executes multiple independent database queries per invocation with no per-client rate limiting. Because the only concurrency guard (`maxConcurrentCallsPerConnection = 5`) is scoped per-connection rather than globally, an attacker opening N connections can drive 5N simultaneous full address-book queries, each independently hitting the database, causing N-fold amplification of DB load.

### Finding Description
**Code path:**

`NetworkController.getNodes()` (lines 33–43) calls `networkService::getNodes` directly with no rate-limiting gate, no deduplication, and no authentication check.

`NetworkServiceImpl.getNodes()` (lines 55–77) then executes, for every single invocation:
1. `addressBookRepository.findLatestTimestamp(fileId.getId())` — uncached DB query
2. `nodeStakeRepository.findLatestTimestamp()` — uncached DB query
3. `nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp)` — uncached DB query that loads the full node-stake map

Only after these three uncached setup queries does it enter the paged loop, where `addressBookEntryRepository.findByConsensusTimestampAndNodeId()` is `@Cacheable` with a 2-second expiry. The cache does not cover the three setup queries above.

**Root cause / failed assumption:** The design assumes that the per-connection call cap (`maxConcurrentCallsPerConnection = 5`, `NettyProperties` line 14, applied in `GrpcConfiguration` line 33) is sufficient to bound total DB load. This assumption fails because the limit is per-connection, not per-IP or global. An attacker opens N HTTP/2 connections and issues 5 calls on each, yielding 5N concurrent invocations, each independently executing the three uncached DB queries.

Additionally, the 2-second `cacheExpiry` on page queries creates a thundering-herd window: when the cache expires, all in-flight concurrent requests simultaneously miss the cache and issue fresh DB queries.

**Existing checks reviewed:**
- `maxConcurrentCallsPerConnection = 5`: per-connection only; trivially bypassed by opening more connections.
- GCP gateway `maxRatePerEndpoint: 250`: a backend-pod-level cap, not a per-client-IP cap; a single attacker can consume the full budget.
- `@Cacheable` on `findByConsensusTimestampAndNodeId`: covers only the paged entry queries, not the three setup queries; expires every 2 seconds.
- No `ThrottleManager`, no `@RateLimiter`, no IP-based throttle exists anywhere in the `grpc` module.

### Impact Explanation
Each `getNodes()` call unconditionally issues at minimum 3 uncached SQL queries against the mirror-node PostgreSQL database. With N open connections each carrying 5 concurrent calls, the DB receives 15N simultaneous queries. On a network with ~30 nodes and default `pageSize = 10`, each call also triggers 3 paged queries (cache-miss scenario), raising the per-call DB query count to 6. A single attacker with 100 open connections can therefore drive 600 simultaneous DB queries, saturating the connection pool and degrading or denying service to all legitimate users of the gRPC API. No funds are at risk, but availability of the mirror node's address-book service is directly impacted.

### Likelihood Explanation
The attack requires no credentials, no tokens, and no special knowledge beyond the publicly documented gRPC endpoint and proto definition. Opening hundreds of HTTP/2 connections is trivial with standard tooling (`grpcurl`, any gRPC client library). The attack is fully repeatable and can be sustained indefinitely. The only practical barrier is network bandwidth to the attacker, which is negligible given the small request payload size.

### Recommendation
1. **Add a global per-IP rate limiter** at the gRPC server interceptor layer (analogous to the `ThrottleManagerImpl` in the `web3` module) that rejects `getNodes()` calls exceeding a configurable threshold per source IP per second.
2. **Cache the three setup queries** (`findLatestTimestamp` on both repositories and `findAllStakeByConsensusTimestamp`) using the already-configured `NODE_STAKE_CACHE` / `ADDRESS_BOOK_ENTRY_CACHE` Caffeine managers, so that concurrent requests share a single DB round-trip during the cache window.
3. **Add a global concurrent-call semaphore** for `getNodes()` (e.g., via a `Semaphore` or Reactor's `flatMap(maxConcurrency)`) to cap total in-flight DB queries regardless of connection count.
4. Consider increasing `cacheExpiry` beyond 2 seconds to reduce thundering-herd frequency on cache expiry.

### Proof of Concept
```bash
# Open 50 parallel gRPC connections, each sending 5 concurrent getNodes calls (250 total)
for i in $(seq 1 250); do
  grpcurl -plaintext \
    -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes &
done
wait
# Each of the 250 concurrent calls independently executes 3+ uncached DB queries.
# Monitor DB connection pool saturation and query latency on the mirror node.
```

Each background call independently reaches `NetworkServiceImpl.getNodes()` lines 61–65, executing `findLatestTimestamp` and `findAllStakeByConsensusTimestamp` against the database with no shared state or deduplication, producing the N-fold amplification described above.
