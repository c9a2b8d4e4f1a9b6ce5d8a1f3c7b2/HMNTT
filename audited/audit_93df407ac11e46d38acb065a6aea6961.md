### Title
Unauthenticated DoS via Unbounded High-Frequency Requests to `POST /api/v1/network/fees` with STATE Mode Triggering Live DB Queries

### Summary
The `estimateFees()` endpoint in `NetworkController` is publicly accessible with no rate limiting applied in the `rest-java` module. When called with `mode=STATE`, each request triggers live database queries through `FeeTopicStore` and `FeeTokenStore` with no caching. An unauthenticated attacker can flood this endpoint to exhaust the database connection pool and thread pool, causing denial-of-service for all legitimate users of the mirror node REST API.

### Finding Description

**Exact code path:**

`NetworkController.estimateFees()` (lines 110â€“124) accepts any unauthenticated POST request and delegates to `FeeEstimationService.estimateFees()`:


In `FeeEstimationService.estimateFees()` (lines 95â€“116), when `mode == STATE`, a `FeeEstimationFeeContext` is constructed wrapping live-DB-backed stores:


The `FeeEstimationFeeContext.readableStore()` returns `FeeTopicStore` and `FeeTokenStore`. Each call to `FeeTopicStore.getTopic()` (lines 28â€“34) issues **two live DB queries** â€” one to `topicRepository.findById()` and one to `customFeeRepository.findById()`. Similarly, `FeeTokenStore.get()` (lines 30â€“35) issues two live DB queries. The integration test `stateModeTopicCustomFeesLive` explicitly confirms: *"store reads live from DB, no refresh needed."*

**Root cause â€” no rate limiting on this endpoint:**

The `ThrottleManager` (with `rateLimitBucket`, `gasLimitBucket`, `opcodeRateLimitBucket`) lives entirely in the `web3` module and is wired only to `ContractController` and `OpcodesController`. The `rest-java` module's `NetworkController` has **zero throttle infrastructure**. No `@RateLimiter`, no bucket4j filter, no Spring Security filter, and no per-IP limiting is applied to `POST /api/v1/network/fees`.

The `JacksonConfiguration` in `rest-java` (lines 20â€“27) limits JSON parsing (`maxDocumentLength=1000`, etc.) but the `estimateFees` endpoint accepts raw protobuf `byte[]` â€” Jackson limits are entirely bypassed for this content type.

**Failed assumption:** The developers applied rate limiting to the computationally expensive `web3` endpoints but omitted it for the new `POST /api/v1/network/fees` endpoint, assuming it would be lightweight. STATE mode makes it DB-bound per request.

### Impact Explanation

Each STATE-mode request with a `ConsensusSubmitMessage` or token-transfer transaction body issues 2 synchronous DB queries against the shared PostgreSQL connection pool. At high request rates (e.g., 500+ req/s from a single attacker), the DB connection pool is saturated, causing all other REST API endpoints (account lookups, transaction queries, etc.) to queue or fail. This is a complete availability impact for all legitimate users of the mirror node REST API. Severity: **High**.

### Likelihood Explanation

The endpoint requires no authentication, no API key, and no prior account. A valid protobuf-encoded `ConsensusSubmitMessage` transaction body is trivially constructable using the public Hedera SDK or by hand-crafting a minimal protobuf. The attack is repeatable, scriptable, and requires no special knowledge beyond the public OpenAPI spec. A single attacker with a modest botnet or even a single high-throughput client can sustain the attack indefinitely.

### Recommendation

1. **Apply rate limiting to `POST /api/v1/network/fees`** using the same bucket4j pattern already used in the `web3` module â€” add a dedicated `feeEstimateRateLimitBucket` in `rest-java` and enforce it at the controller or filter level.
2. **Differentiate limits by mode**: Apply a stricter per-second limit for `mode=STATE` requests (e.g., 10â€“50 RPS) vs. `mode=INTRINSIC` (which is CPU-only and cheaper).
3. **Add per-IP rate limiting** at the reverse proxy/ingress layer (e.g., nginx `limit_req`) as a defense-in-depth measure.
4. **Enforce a request body size limit** explicitly for the protobuf endpoint (e.g., via `spring.servlet.multipart.max-request-size` or a `ContentSizeLimitFilter`) to prevent large-body amplification.

### Proof of Concept

```bash
# Build a minimal valid ConsensusSubmitMessage protobuf transaction (using hedera-sdk or protoc)
# Field 1 (signedTransactionBytes): a SignedTransaction containing a TransactionBody with
# consensusSubmitMessage set to any TopicID.

# Then flood the endpoint:
while true; do
  curl -s -X POST \
    "https://<mirror-node>/api/v1/network/fees?mode=STATE" \
    -H "Content-Type: application/protobuf" \
    --data-binary @consensus_submit_message.bin &
done
# Run N parallel instances (e.g., N=200) to saturate the DB connection pool.
# Observe: legitimate GET /api/v1/transactions requests begin timing out or returning 503.
```

Each iteration of the loop triggers 2 synchronous DB queries (`topicRepository.findById` + `customFeeRepository.findById`) with no rate gate, exhausting the connection pool and starving legitimate traffic.
