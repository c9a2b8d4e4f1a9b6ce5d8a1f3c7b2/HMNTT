### Title
Unauthenticated 7-Query DB Chain in `getContractResultsByTransactionIdOrHash` Enables Resource Exhaustion Against Mirror Node REST API

### Summary
The `GET /api/v1/contracts/results/:transactionIdOrHash` endpoint, routed through `rest/routes/contractRoute.js` and handled by `getContractResultsByTransactionIdOrHash` in `rest/controllers/contractController.js`, executes up to 7 database queries per request with no rate limiting on the REST API layer. Any unauthenticated caller supplying a syntactically valid transaction ID that resolves to a contract transaction triggers this full chain. Under concurrent load, this exhausts the shared PostgreSQL connection pool, degrading or halting the mirror node REST service.

### Finding Description

**Exact code path** (`rest/controllers/contractController.js`, lines 1120–1220):

1. **Query 1** (line 1142): `TransactionService.getTransactionDetailsFromTransactionId(transactionId, nonce)` — sequential, always executes for a syntactically valid transaction ID.
2. **Query 2** (line 1156): `ContractService.getInvolvedContractsByTimestampAndContractId(...)` — sequential, executes if Query 1 returns a result.
3. **Queries 3–7** (lines 1205–1219): `getDetailedContractResults()` fires five queries in parallel via `Promise.all`:
   - `ContractService.getContractResultsByTimestamps(...)`
   - `TransactionService.getEthTransactionByTimestampAndPayerId(...)`
   - `RecordFileService.getRecordFileBlockDetailsFromTimestamp(...)`
   - `ContractService.getContractLogsByTimestamps(...)`
   - `ContractService.getContractStateChangesByTimestamps(...)`

**Root cause — no rate limiting on the REST API layer**: The `rest/middleware/` directory contains `authHandler.js`, `httpErrorHandler.js`, `openapiHandler.js`, `requestHandler.js`, `responseCacheHandler.js`, and `responseHandler.js`. None implement per-IP or global request-rate throttling. The `ThrottleConfiguration` / `ThrottleManagerImpl` (bucket4j) exists only in the `web3` Java module and does not protect the Node.js REST service. The DB pool (`rest/dbpool.js`) is bounded by `config.db.pool.maxConnections`, but with 7 concurrent queries per request, pool exhaustion is reachable at modest concurrency.

**Failed assumption**: The design assumes an external load balancer or infrastructure-level rate limiter is always present. No such control is enforced in the application code itself.

**Why the response cache does not mitigate this**: `responseCacheHandler` caches by request URL. An attacker cycling through distinct valid transaction IDs (all publicly observable on the Hedera ledger) produces cache misses on every request, forcing the full 7-query chain each time.

### Impact Explanation

The mirror node REST API is the primary read interface for dApps, wallets, and explorers querying Hedera contract data. Exhausting the PostgreSQL connection pool (`maxConnections`) causes all subsequent queries across all endpoints to queue or fail with connection-timeout errors, taking the entire REST service offline. Because the pool is shared globally, this affects every endpoint, not just `/contracts/results`. The mirror node is not a Hedera consensus node, so ledger consensus is unaffected; however, the availability of the public data API — which many ecosystem participants depend on — is fully compromised.

### Likelihood Explanation

Preconditions are minimal: the attacker needs only syntactically valid Hedera transaction IDs that correspond to contract transactions. These are freely enumerable from the public ledger (e.g., via `/api/v1/transactions?type=CONTRACTCALL`). No credentials, tokens, or privileged access are required. The attack is trivially scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, `hey`) and is repeatable indefinitely. The absence of any application-level rate limiting makes this a low-skill, high-impact attack.

### Recommendation

1. **Add rate limiting middleware to the REST API**: Integrate a per-IP and global rate limiter (e.g., `express-rate-limit` with a Redis store for multi-instance deployments) applied before route handlers in `rest/server.js`.
2. **Apply specifically to expensive endpoints**: The `/contracts/results/:transactionIdOrHash` route should have a tighter per-IP limit than simpler endpoints, given its 7-query cost.
3. **Add a DB query concurrency guard**: Consider a semaphore or queue in `getDetailedContractResults` to cap simultaneous in-flight DB query sets.
4. **Do not rely solely on infrastructure-level controls**: Application-level rate limiting should be a defense-in-depth layer, not an assumed external dependency.

### Proof of Concept

```bash
# Step 1: Obtain valid contract transaction IDs from the public ledger
curl "https://<mirror-node>/api/v1/transactions?type=CONTRACTCALL&limit=100" \
  | jq -r '.transactions[].transaction_id' > tx_ids.txt

# Step 2: Flood the endpoint with concurrent requests using distinct IDs
# (each triggers the full 7-query chain, bypassing the response cache)
cat tx_ids.txt | xargs -P 200 -I{} \
  curl -s -o /dev/null "https://<mirror-node>/api/v1/contracts/results/{}"

# Step 3: Observe REST API degradation
# Subsequent requests to any endpoint return connection timeout errors
# as the PostgreSQL pool (maxConnections) is exhausted
curl "https://<mirror-node>/api/v1/transactions"
# Expected: timeout or 503 Service Unavailable
```
