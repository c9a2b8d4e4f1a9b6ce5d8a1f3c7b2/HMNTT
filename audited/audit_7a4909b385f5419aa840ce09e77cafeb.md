### Title
Inconsistent Deleted-Entity Filter in `getByEvmAddressAndType()` Exposes Deleted Account State via Long-Form EVM Address

### Summary
`getByEvmAddressAndType()` in `EntityServiceImpl` routes "long-form" EVM addresses (first 12 bytes zero) through `entityRepository.findById()`, which is the bare Spring Data `CrudRepository.findById()` with no deleted-entity filter. The two custom query methods — `findByEvmAddress()` and `findByAlias()` — both enforce `deleted is not true` at the SQL level, but `findById()` does not. An unprivileged caller can therefore retrieve the full persisted state of a deleted account, including its `pendingReward` field, by supplying a crafted 20-byte address.

### Finding Description

**Exact code path:**

`EntityServiceImpl.java` lines 34–41: [1](#0-0) 

The branch condition at line 37 checks whether bytes 0–3 (int) and bytes 4–11 (long) are all zero. If so, it reads bytes 12–19 as a `long` entity ID and calls `entityRepository.findById(buffer.getLong())` at line 38.

**Root cause — `findById()` has no deleted filter:**

`EntityRepository.java` (graphql module): [2](#0-1) 

`findByAlias()` and `findByEvmAddress()` both carry `deleted is not true` in their native SQL. `findById()` is inherited from `CrudRepository<Entity, Long>` and issues a plain `SELECT * FROM entity WHERE id = ?` — no deleted predicate.

**Exploit flow:**
1. Attacker identifies a deleted account with a known numeric entity ID (e.g., `12345`).
2. Attacker constructs a 20-byte hex string: 12 zero bytes followed by the 8-byte big-endian encoding of `12345`.
3. Attacker sends a GraphQL `account` query with `evmAddress` set to that hex string.
4. `getByEvmAddressAndType()` detects the zero prefix, calls `findById(12345)`, and returns the full `Entity` row — including `deleted = true` and the `pendingReward` value — to the caller.

**Why existing checks fail:**

The `findByEvmAddress` path (line 40) correctly filters deleted entities: [3](#0-2) 

But the long-form branch at line 38 bypasses this entirely by using the unfiltered `findById`. There is no post-fetch `filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` guard applied before the type check.

The `AccountController` passes the result directly to `accountMapper::map` with no deleted check: [4](#0-3) 

### Impact Explanation

A deleted account's full persisted state — including `pendingReward`, balance, keys, memo, and staking metadata — is returned to any unauthenticated caller who supplies the crafted address. The `pendingReward` field is part of the `Accountable` GraphQL interface: [5](#0-4) 

Because this is a read-only mirror node, no funds can be moved. The concrete impact is **unauthorized information disclosure** of deleted-account state that the API is designed to suppress (as evidenced by the `deleted is not true` guards on every other lookup path). API consumers relying on the absence of a response to infer account non-existence are misled.

### Likelihood Explanation

Preconditions are minimal: the attacker needs only a valid numeric entity ID (trivially enumerable from public ledger data) and the ability to send a GraphQL HTTP request. No authentication, no special role, no on-chain transaction required. The crafted address is a deterministic 20-byte value. The attack is repeatable for any deleted account ID.

### Recommendation

Add an explicit deleted-entity guard in the long-form branch of `getByEvmAddressAndType()`, consistent with the other two lookup paths:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
        .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))
        .filter(e -> e.getType() == type);
}
```

Alternatively, introduce a `findByIdAndDeletedIsFalse(Long id)` method in `EntityRepository` (mirroring the pattern already used in the `web3` module's `EntityRepository`) and use it in both `getByIdAndType()` and the long-form branch of `getByEvmAddressAndType()`.

### Proof of Concept

```
# Target: deleted account with entity ID 98765 (0x000000000000000000000000000000000000000000181CD)
# Long-form EVM address: 0x0000000000000000000000000000000000000000000181CD
# (bytes 0-11 = 0x000000000000000000000000, bytes 12-19 = 0x000000000000181CD)

curl -X POST https://<mirror-node>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "{ account(input: { evmAddress: \"0x000000000000000000000000000000000000181cd\" }) { deleted pendingReward balance } }"
  }'

# Expected (correct) behavior: empty/null response — account is deleted
# Actual behavior: returns the deleted account with pendingReward populated
```
