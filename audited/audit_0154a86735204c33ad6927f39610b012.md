### Title
Unbounded Multi-Connection Subscription Exhaustion via Unauthenticated `subscribeTopic()` gRPC Endpoint

### Summary
`ConsensusController.subscribeTopic()` accepts unlimited concurrent streaming subscriptions from any unauthenticated caller with no IP-based, identity-based, or global subscription count enforcement. The only guard — `maxConcurrentCallsPerConnection = 5` — is scoped per HTTP/2 connection, so an attacker opening N connections from N source ports trivially multiplies their subscription count to N×5, exhausting R2DBC connection pool, Redis listener slots, and JVM thread/memory resources.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any `ConsensusTopicQuery` and immediately delegates to `topicMessageService.subscribeTopic()` with no authentication check, no IP check, and no global subscription cap check:

```java
// ConsensusController.java lines 43-48
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`TopicMessageServiceImpl.subscribeTopic()` (lines 48, 89–90) maintains an `AtomicLong subscriberCount` that is incremented on subscribe and decremented on finish — but it is **only registered as a metrics gauge** (lines 52–55). There is no guard of the form `if (subscriberCount.get() >= MAX) reject()`. The counter is purely observational.

```java
// TopicMessageServiceImpl.java lines 48, 52-55, 89-90
private final AtomicLong subscriberCount = new AtomicLong(0L);

Gauge.builder("hiero.mirror.grpc.subscribers", () -> subscriberCount)  // metrics only
     ...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())   // no cap check before this
.doFinally(s -> subscriberCount.decrementAndGet())
```

**The only server-side limit** is `maxConcurrentCallsPerConnection = 5` configured in `GrpcConfiguration.java` (line 33) via `NettyProperties` (line 14):

```java
// GrpcConfiguration.java line 33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// NettyProperties.java line 14: default = 5
```

This is a **per-HTTP/2-connection** limit. No `maxConnections` total limit is set on the `NettyServerBuilder`. No IP-based connection limit is configured.

**Infrastructure-level checks are also insufficient:**
- The nginx proxy config (docker-compose.yml line 226) routes `subscribeTopic` with only a `grpc_read_timeout 600s` — no `limit_conn`, no `limit_req`.
- The GCP backend policy `maxRatePerEndpoint: 250` (charts/hedera-mirror-grpc/values.yaml line 69) is a new-request rate limit per backend pod, not a per-client or global concurrent-stream cap, and does not apply to long-lived streaming connections already established.

**Root cause:** The design assumes a single-connection rate limit is sufficient, but HTTP/2 multiplexing means the per-connection call limit is trivially bypassed by opening additional connections.

### Impact Explanation

Each active `subscribeTopic` stream holds:
1. A reactive Flux pipeline with a `TopicContext` object and associated `AtomicLong`/`AtomicReference` state.
2. An R2DBC database connection slot (for the historical retriever polling loop).
3. A Redis pub/sub listener registration (via `RedisTopicListener` or `SharedPollingTopicListener`).
4. A `boundedElastic` scheduler thread slot for the safety-check Flux (line 70 of `TopicMessageServiceImpl`).

With no global cap, an attacker can exhaust the R2DBC connection pool, starving legitimate subscribers and the importer of database access, causing a full service outage for the gRPC API. This is a non-network-based DoS (resource exhaustion) affecting the mirror node's HCS subscription service.

### Likelihood Explanation

**Preconditions:** None. No account, no API key, no authentication of any kind is required. The gRPC port (5600) is publicly exposed.

**Attacker capability:** Any unprivileged external user with a standard gRPC client (e.g., `grpcurl`, any HCS SDK). A single machine with ephemeral port range ~28,000 ports can open ~28,000 connections × 5 streams = ~140,000 concurrent subscriptions. A small botnet or cloud VM fleet makes this trivially repeatable.

**Repeatability:** Fully repeatable. Connections can be re-opened immediately after being closed. The 600-second nginx read timeout means connections stay alive for up to 10 minutes per cycle without any message activity.

### Recommendation

1. **Enforce a global concurrent subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount.get()` against a configurable maximum before allowing a new subscription, returning `RESOURCE_EXHAUSTED` status if exceeded.
2. **Add a per-IP concurrent stream limit** via a gRPC `ServerInterceptor` that tracks active calls per remote address and rejects new calls beyond a threshold.
3. **Add `maxConnections` to the Netty server builder** in `GrpcConfiguration.java` to cap total TCP connections server-wide.
4. **Add nginx `limit_conn`** for the `subscribeTopic` gRPC location block to enforce per-IP connection limits at the proxy layer.
5. Add a configurable `maxSubscriptionsPerIp` property to `NettyProperties` or `GrpcProperties`.

### Proof of Concept

```bash
# Open 1000 connections, each with 5 concurrent subscribeTopic streams (5000 total)
# Using grpcurl in a loop (or any HCS gRPC client)
for i in $(seq 1 1000); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# Each connection uses a different ephemeral source port, bypassing maxConcurrentCallsPerConnection=5
# Monitor: hiero.mirror.grpc.subscribers metric will climb unboundedly
# Result: R2DBC pool exhausted, new subscriptions and importer DB queries begin failing
```
