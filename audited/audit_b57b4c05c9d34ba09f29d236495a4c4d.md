### Title
HikariCP Connection Pool Exhaustion via Synchronous Blocking DB Calls During Network Partition

### Summary
`TopicServiceImpl.findById()` issues a synchronous, blocking JDBC call with no query/socket timeout configured. During a network partition between the REST-Java service and PostgreSQL, in-flight connections hang for the OS TCP timeout duration (hours by default), exhausting the HikariCP pool. An unprivileged attacker flooding `GET /api/v1/topics/{id}` during such a partition can hold all pool connections blocked, denying service to all users.

### Finding Description

**Exact code path:**

`TopicServiceImpl.java` line 20:
```java
return topicRepository.findById(id.getId()).orElseThrow(() -> new EntityNotFoundException("Topic not found"));
``` [1](#0-0) 

`TopicRepository` is a plain Spring Data `CrudRepository` — fully synchronous, blocking JDBC. [2](#0-1) 

`TopicController.getTopic()` is a standard synchronous Spring MVC handler with no rate limiting, no async dispatch, and no timeout annotation. [3](#0-2) 

**Root cause — two compounding failures:**

1. **No JDBC socket/statement timeout.** HikariCP is configured only via `@ConfigurationProperties("spring.datasource.hikari")` with no `socketTimeout`, `connectionTimeout` override, or statement-level timeout visible in the codebase. [4](#0-3) 
   HikariCP's `connectionTimeout` (default 30 s) governs only *acquiring* a connection from the pool, not how long an already-acquired, in-flight query can run. Without a JDBC-level `socketTimeout`, a query that is mid-execution when a network partition occurs will block for the OS TCP keepalive timeout — typically 2+ hours on Linux.

2. **No application-level rate limiting.** A search across all `rest-java/src/main/java/**` finds zero `RateLimiter`, `@RateLimit`, `Semaphore`, or request-throttling constructs protecting this endpoint.

**Exploit flow:**

- Attacker sends ≥10 concurrent `GET /api/v1/topics/{id}` requests (HikariCP default pool size = 10).
- Each request acquires one HikariCP connection and issues the blocking `findById` query.
- A network partition (or one the attacker induces/times with) occurs between the service and PostgreSQL.
- All 10 in-flight queries hang indefinitely at the TCP socket layer — no timeout fires.
- HikariCP pool is fully exhausted (`hikaricp_connections_active == hikaricp_connections_max`).
- Every subsequent request from any user waits up to `connectionTimeout` (30 s default) then receives a `SQLTimeoutException` / 500 error.
- Service is effectively unavailable for the entire duration of the partition.

**Why existing checks are insufficient:**

- The GCP Gateway `maxRatePerEndpoint: 250` only applies when `gateway.gcp.enabled: true`; the default deployment uses Traefik ingress, where this limit is absent. [5](#0-4) 
- The Traefik circuit breaker (`NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25`) triggers on *error ratios*, not on slow/hanging requests. Blocked connections do not produce errors until the 30 s `connectionTimeout` fires, by which time the pool is already exhausted. [6](#0-5) 
- The Prometheus alert `RestJavaHighDBConnections` fires after 5 minutes at >75% utilization — far too slow to prevent the DoS. [7](#0-6) 

### Impact Explanation
Full denial of service for all users of the REST-Java service for the duration of the network partition. Every endpoint sharing the same HikariCP pool (accounts, network, topics) is affected, not just `/topics/{id}`. Severity: **High** — complete service unavailability, no authentication required to trigger.

### Likelihood Explanation
The attacker requires no credentials and no special knowledge beyond the public API path. The attack is trivially repeatable with any HTTP load tool (`curl`, `ab`, `wrk`). Network partitions between application pods and database pods occur naturally in cloud environments (node eviction, rolling restarts, AZ failures) and can also be induced by an attacker with network-adjacent access. The combination of a naturally occurring partition and a concurrent flood is realistic.

### Recommendation

1. **Set a JDBC socket timeout** via HikariCP's `dataSourceProperties` or connection string: `socketTimeout=30` (seconds) for PostgreSQL JDBC, or `spring.datasource.hikari.connection-timeout` combined with `spring.datasource.hikari.data-source-properties.socketTimeout=30000`.
2. **Set a statement/query timeout** at the PostgreSQL session level: `spring.datasource.hikari.data-source-properties.options=-c statement_timeout=10000`.
3. **Add application-level concurrency limiting** (e.g., Resilience4j `Bulkhead` or `TimeLimiter`) on `TopicServiceImpl.findById()` to cap in-flight DB calls.
4. **Enable HPA** (`hpa.enabled: true`) so the service can scale under load, reducing per-instance pool pressure.
5. **Ensure the GCP rate limit or equivalent Traefik middleware** is enforced unconditionally, not only when the GCP gateway is active.

```bash
# Step 1: Induce or wait for a network partition between rest-java pod and PostgreSQL
# (e.g., apply a NetworkPolicy blocking egress to DB port 5432, or use tc/iptables)

for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/topics/0.0.1234" &
done
wait

# Step 3: Observe — all 10 HikariCP connections are now blocked.
# Any subsequent request from any user will hang for ~30s then receive HTTP 500.
