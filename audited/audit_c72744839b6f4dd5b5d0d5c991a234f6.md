### Title
Synchronous CPU-Intensive Mapping in `formatTransactionRows()` Blocks Node.js Event Loop Without Rate Limiting

### Summary
`doGetTransactions()` in `rest/transactions.js` allows any unauthenticated user to request up to 100 transactions (the configured max), each of which may carry large `crypto_transfer_list` and `token_transfer_list` JSON aggregates. The subsequent call to `formatTransactionRows()` performs all per-row and per-transfer processing synchronously inside `rows.map()`, blocking the single-threaded Node.js event loop. Because the REST API has no per-IP rate limiting or concurrency cap, an attacker can flood the endpoint with concurrent max-limit requests to continuously starve other in-flight requests.

### Finding Description

**Code path:**

`getTransactions()` → `doGetTransactions()` → `cache.get(…, loader, …)` → `loader` = `getTransactionsDetails(keys, order).then((result) => formatTransactionRows(result.rows))` [1](#0-0) 

`formatTransactionRows` is declared `async` but its body is entirely synchronous after the one awaited DB call: [2](#0-1) 

Inside the synchronous `rows.map()`, for every row the following nested synchronous iterations execute:

- `createCryptoTransferList(row.crypto_transfer_list)` — `.map()` over every crypto transfer, calling `EntityId.parse()` per entry. [3](#0-2) 
- `createTokenTransferList(row.token_transfer_list)` — `.map()` over every token transfer, two `EntityId.parse()` calls per entry. [4](#0-3) 
- `createNftTransferList(row.nft_transfer)` — `.map()` constructing `NftTransfer` + `NftTransferViewModel` per entry. [5](#0-4) 
- `createAssessedCustomFeeList(row.assessed_custom_fees)` — `.map()` constructing model + viewmodel per entry. [6](#0-5) 

Additionally, `getStakingRewardTimestamps()` — called before the map — iterates all rows and calls `.some()` on each `crypto_transfer_list`, adding another O(N×M) synchronous pass: [7](#0-6) 

**Root cause / failed assumption:** The design assumes that the per-row formatting work is negligible. It is not bounded per request beyond the 100-row limit, and no async yield point (e.g., `setImmediate`, worker thread) is inserted to release the event loop between rows.

**Why existing checks are insufficient:**

- The `limit` parameter is capped at `response.limit.max = 100` (default). [8](#0-7)   This bounds rows per request but does not bound the total transfer-list entries across those 100 rows, nor does it limit concurrent requests.
- The server registers no rate-limiting or concurrency-limiting middleware for the `/transactions` route. The middleware stack is: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`, `responseHandler`, `handleError`. [9](#0-8) 
- Redis caching (`cache.get`) only helps for cache hits on previously seen timestamp keys. A cache miss (e.g., a fresh time window) always falls through to `loader` and the full synchronous mapping. [10](#0-9) 
- The throttling infrastructure found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3` Java service, not the Node.js REST API. [11](#0-10) 

### Impact Explanation
Node.js processes all JavaScript on a single thread. While the DB queries are async and do not block the event loop, the synchronous `rows.map()` in `formatTransactionRows` does. For 100 transactions each with, say, 50 crypto transfers + 20 token transfers, the event loop is held for the duration of ~7,000 synchronous object constructions and `EntityId.parse` calls per request. An attacker sending tens of concurrent requests keeps the event loop continuously occupied, causing latency spikes or timeouts for all other API consumers on the same process. Because this is a public mirror-node API with no authentication required for read access, the attack surface is fully exposed to the internet.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no protocol-level access — only the ability to send HTTP GET requests to `/api/v1/transactions?limit=100`. An attacker can identify time windows containing high-transfer transactions (e.g., airdrop or batch-transfer periods) by browsing public Hedera explorer data, then repeatedly query those windows. The attack is trivially scriptable and repeatable. The absence of any rate limiting or IP-based concurrency cap in the Node.js REST server makes sustained exploitation straightforward.

### Recommendation
1. **Yield the event loop** inside `formatTransactionRows` by chunking the `rows.map()` with `setImmediate` between chunks, or offload the work to a worker thread pool.
2. **Add rate limiting** to the REST API (e.g., `express-rate-limit`) scoped per IP, specifically on the `/transactions` endpoint.
3. **Add a concurrency cap** per IP or globally using a semaphore/queue to limit simultaneous in-flight `formatTransactionRows` executions.
4. **Cap total transfer entries** returned per response (not just the number of transactions), so that a single request cannot trigger unbounded nested iteration.

### Proof of Concept

**Preconditions:** Public mirror-node REST API accessible. Identify a timestamp range containing transactions with large transfer lists (e.g., a known airdrop event on mainnet).

**Steps:**

```bash
# Step 1: Identify a timestamp range with high-transfer transactions
# (use public Hedera explorer or any prior knowledge of airdrop timestamps)
RANGE_START="1700000000"
RANGE_END="1700001000"

# Step 2: Launch concurrent max-limit requests targeting that range
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/transactions?limit=100&timestamp=gte:${RANGE_START}&timestamp=lte:${RANGE_END}&order=asc" &
done
wait

# Step 3: Simultaneously measure latency of a lightweight request
time curl -s "https://<mirror-node-host>/api/v1/transactions?limit=1"
```

**Expected result:** The lightweight request in Step 3 experiences significantly elevated latency (seconds instead of milliseconds) because the Node.js event loop is occupied processing the synchronous `rows.map()` calls from the 50 concurrent max-limit requests. In a sustained attack, the service becomes unresponsive to all clients.

  async get(keys, loader, keyMapper = (k) => (k ? k.toString() : k)) {
    if (isEmpty(keys)) {
      return [];
    }
    if (!this.ready) {
      return loader(keys);
    }

    const buffers =
      (await this.redis
        .mgetBuffer(map(keys, keyMapper))
        .catch((err) => logger.warn(`Redis error during mget: ${err.message}`))) || new Array(keys.length);
    const values = buffers.map((t) => JSONParse(t));

    let i = 0;
    const missingKeys = keys.filter(() => isNil(values[i++]));

    if (missingKeys.length > 0) {
      const missing = await loader(missingKeys);
      const newValues = [];
      let j = 0;

      missing.forEach((value) => {
        // Update missing values in Redis array
        for (; j < values.length; j++) {
          if (isNil(values[j])) {
            values[j] = value;
            newValues.push(keyMapper(keys[j]));
            newValues.push(JSONStringify(value));
            break;
          }
        }
      });

      if (newValues.length > 0) {
        this.redis.mset(newValues).catch((err) => logger.warn(`Redis error during mset: ${err.message}`));
      }
    }

    if (logger.isDebugEnabled()) {
      const count = keys.length - missingKeys.length;
      logger.debug(`Redis returned ${count} of ${keys.length} keys`);
    }

    return values;
  }

import io.github.bucket4j.Bandwidth;
import io.github.bucket4j.Bucket;
import io.github.bucket4j.local.SynchronizationStrategy;
import java.time.Duration;
import lombok.RequiredArgsConstructor;
import org.hiero.mirror.web3.throttle.ThrottleProperties;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
