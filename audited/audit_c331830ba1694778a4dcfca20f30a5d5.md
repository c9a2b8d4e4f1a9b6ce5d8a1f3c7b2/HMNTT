### Title
Unauthenticated DB Query Exhaustion via Syntactically Valid Non-Existent Transaction Hash

### Summary
Any unauthenticated external user can supply a syntactically valid but non-existent transaction hash to `GET /api/v1/transactions/:transactionIdOrHash`, causing `extractSqlFromTransactionsByIdOrHashRequest()` to unconditionally execute a database query via `getTransactionHash()` before throwing `NotFoundError`. With no application-level rate limiting present, an attacker can sustain a high request rate that continuously consumes DB connection pool slots, degrading or denying service to legitimate users.

### Finding Description
**Code path:**

- `rest/transactions.js` lines 763–775 (`extractSqlFromTransactionsByIdOrHashRequest`):
  ```js
  const isTransactionHash = isValidTransactionHash(transactionIdOrHash); // regex only
  if (isTransactionHash) {
    ...
    const rows = await getTransactionHash(Buffer.from(transactionIdOrHash, encoding)); // DB query
    if (rows.length === 0) {
      throw new NotFoundError(); // thrown AFTER the query
    }
  ```
- `rest/transactionHash.js` lines 21–37 (`getTransactionHash`):
  ```js
  const {rows} = await pool.queryQuietly(query, params); // executes get_transaction_info_by_hash($1)
  ```

**Root cause:** `isValidTransactionHash()` (`rest/transactionHash.js` line 44) is a pure regex check — `/^([\dA-Za-z+\-\/_]{64}|(0x)?[\dA-Fa-f]{96})$/` — that validates only format, not existence. Any string matching this regex (e.g., 96 hex chars prefixed with `0x`) unconditionally triggers a full DB stored-procedure call. The `NotFoundError` is thrown only *after* the query completes and returns zero rows.

**Why existing checks are insufficient:**
- `isValidTransactionHash()` is format-only; it cannot distinguish a real hash from a crafted one.
- `rest/server.js` registers no rate-limiting middleware (confirmed: `express-rate-limit` or equivalent is absent from the middleware chain at lines 67–144).
- The `authHandler` (`rest/middleware/authHandler.js`) only adjusts response-size limits for authenticated users; it does not throttle unauthenticated requests.
- The response cache (`responseCacheCheckHandler`) caches successful responses only; 404 error paths are not cached, so every repeated non-existent hash hits the DB fresh.

### Impact Explanation
Each crafted request holds a PostgreSQL connection pool slot for the duration of the `get_transaction_info_by_hash` stored-procedure call. Under sustained parallel requests with distinct valid-format but non-existent hashes, the connection pool can be saturated, causing legitimate queries to queue or time out. This constitutes a non-network-layer DoS: the attacker does not need to flood bandwidth — a modest request rate targeting this endpoint is sufficient to degrade the DB tier for all API consumers.

### Likelihood Explanation
Preconditions are minimal: no authentication, no special knowledge, no privileged access. A valid-format hash is trivially generated (e.g., `0x` + 96 random hex characters). The attack is fully repeatable, scriptable, and requires only HTTP access to the public REST API port. The absence of any application-level rate limiting makes sustained exploitation straightforward.

### Recommendation
1. **Add application-level rate limiting** (e.g., `express-rate-limit`) in `rest/server.js` scoped to the `/api/v1/transactions/:transactionIdOrHash` route, or globally, before the route handlers.
2. **Cache negative (404) results** for hash lookups in Redis with a short TTL so repeated lookups of the same non-existent hash do not re-hit the DB.
3. **Enforce a DB query timeout** on `get_transaction_info_by_hash` to bound the maximum connection-hold time per request.

### Proof of Concept
```bash
# Generate a syntactically valid but non-existent 48-byte hex hash
FAKE_HASH="0x$(python3 -c "import secrets; print(secrets.token_hex(48))")"

# Send concurrent requests — each triggers a DB query and returns 404
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/transactions/$FAKE_HASH" &
done
wait
```

Each request passes `isValidTransactionHash()` (regex match), executes `get_transaction_info_by_hash($1)` against the DB, receives 0 rows, and throws `NotFoundError` — consuming a DB connection slot for the full query duration. With 500 concurrent requests and no rate limiting, the connection pool is saturated.
