### Title
Unprivileged User Can Force Three-Part UNION ALL Query via Dual-Range NFT Filter, Enabling DB CPU Griefing

### Summary
Any unauthenticated caller can supply `token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B` to `/api/v1/accounts/{id}/nfts`, which passes all server-side validation and causes `NftService.getQuery()` to emit a three-part `UNION ALL` SQL query instead of a single query. Because each of the three sub-queries carries its own `LIMIT` clause, the database must execute three independent index scans and then sort and re-limit the merged result set, making every such request roughly 3× more expensive than a plain query. No authentication, no special privilege, and no rate-limiting on this REST endpoint is required.

### Finding Description

**Code path:**

1. `GET /api/v1/accounts/{id}/nfts?token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B&limit=100`
2. `AccountController.getNftsByAccountId()` → `extractNftMultiUnionQuery()` (`rest/controllers/accountController.js` lines 34–82)
3. `validateFilters()` → `validateBounds()` (`rest/controllers/baseController.js` lines 56–123)
4. `getLowerFilters()` / `getInnerFilters()` / `getUpperFilters()` → all three return non-empty arrays
5. `NftService.getQuery()` (`rest/service/nftService.js` lines 84–126) → emits three-part `UNION ALL`

**Root cause – validation explicitly permits the triggering combination:**

`validateLowerBounds()` (baseController.js lines 83–92) only rejects a secondary lower bound when the primary uses strict `gt`; `gte` is allowed. `validateUpperBounds()` (lines 100–108) mirrors this: `lte` is allowed. Therefore `token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B` clears every check.

`getLowerFilters()` (lines 131–146) returns `[{token_id = X}, {serialnumber >= A}]`.  
`getInnerFilters()` (lines 154–168) returns `[{token_id > X}, {token_id < Y}]`.  
`getUpperFilters()` (lines 176–183) returns `[{token_id = Y}, {serialnumber <= B}]`.

All three arrays are non-empty, so `NftService.getQuery()` (lines 93–123) takes the `subQueries.length > 1` branch and emits:

```sql
(SELECT … FROM nft … WHERE account_id=$1 AND token_id=$3 AND serial_number>=$4 … LIMIT $2)
UNION ALL
(SELECT … FROM nft … WHERE account_id=$1 AND token_id>$5 AND token_id<$6 … LIMIT $2)
UNION ALL
(SELECT … FROM nft … WHERE account_id=$1 AND token_id=$7 AND serial_number<=$8 … LIMIT $2)
ORDER BY token_id desc, serial_number desc LIMIT $2
```

This is confirmed verbatim by the project's own test at `rest/__tests__/service/nftService.test.js` lines 251–296 ("token closed range and serial closed range").

**Why existing checks are insufficient:**

- `validateBounds()` is a correctness guard (prevents logically contradictory filters), not a resource guard.
- The `limit` parameter is capped at 100 (`rest/__tests__/config.test.js` line 324: `max: 100`), but this cap applies *per sub-query*; the database still executes three separate scans of up to 100 rows each before the outer sort.
- The throttle/rate-limit infrastructure found in the codebase (`web3/src/main/java/…/ThrottleConfiguration.java`) belongs to the **web3 Java module**, not to the **REST Node.js service**. No equivalent middleware was found protecting the `/accounts/{id}/nfts` route.

### Impact Explanation
An attacker can continuously issue the crafted request, causing the PostgreSQL backend to execute three index scans per request instead of one. At scale (e.g., hundreds of concurrent connections), this multiplies DB CPU and I/O load by ~3×, potentially degrading response times for all users of the mirror-node REST API. No funds are at risk, but service availability and latency are impacted — consistent with the "griefing / no economic damage" classification.

### Likelihood Explanation
The exploit requires zero privileges, zero tokens, and zero on-chain state. The URL is public, the parameter names are documented in the OpenAPI spec (`rest/api/v1/openapi.yml`), and the triggering combination is a natural-looking pagination query. Any script-kiddie with `curl` or a load-testing tool can reproduce it repeatedly and indefinitely.

### Recommendation
1. **Add per-IP or global rate limiting** to the REST Node.js service for the `/accounts/{id}/nfts` endpoint (analogous to the bucket4j throttle already present in the web3 module).
2. **Detect and reject the three-part-union trigger at the application layer**: if both `primary.hasLower() && primary.hasUpper() && secondary.hasLower() && secondary.hasUpper()` is true, either reject the request with HTTP 400 or collapse it into a single-range query.
3. **Alternatively**, enforce a tighter `limit` cap specifically when the three-part union path is taken (e.g., cap at 25 instead of 100) to reduce per-request cost.

### Proof of Concept

```bash
# Single crafted request – forces 3-part UNION ALL
curl "https://<mirror-node>/api/v1/accounts/0.0.1000/nfts?\
token.id=gte:0.0.1&token.id=lte:0.0.999999&\
serialnumber=gte:1&serialnumber=lte:999999&\
limit=100"

# Griefing loop – no authentication required
while true; do
  curl -s -o /dev/null "https://<mirror-node>/api/v1/accounts/0.0.1000/nfts?\
  token.id=gte:0.0.1&token.id=lte:0.0.999999&\
  serialnumber=gte:1&serialnumber=lte:999999&limit=100" &
done
```
