### Title
Cache Poisoning via `Optional.empty()` Caching in `findRuntimeBytecode()` — Negative-Result Entries Evict Legitimate Bytecode

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"` to suppress caching of empty results, but the method returns `Optional<byte[]>`. When a contract does not exist, the method returns `Optional.empty()`, which is a non-null object — so the `unless` guard never fires and `Optional.empty()` is stored in the Caffeine cache. An unprivileged attacker can flood the cache (capacity: 1 000 entries) with negative results for non-existent contract IDs, evicting all legitimate bytecode entries and forcing the EVM to re-query the database on every subsequent call.

    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));

**Root cause:** Spring's `unless` SpEL expression `#result == null` is evaluated against the actual return value of the method. The method signature is `Optional<byte[]>`, so when no contract row exists the repository returns `Optional.empty()`. `Optional.empty()` is a singleton non-null object; the condition `#result == null` evaluates to `false`, so Spring proceeds to store the value in the cache. The developer's intent was clearly to skip caching of "no result" cases, but the guard is written for a nullable return type, not an `Optional` return type.

**Exploit flow:**
1. Attacker sends `eth_call` (or `eth_estimateGas`) requests targeting 1 000+ distinct contract addresses that do not exist in the database.
2. Each request reaches `ContractBytecodeReadableKVState.readFromDataSource()`, which calls `contractRepository.findRuntimeBytecode(entityId.getId())`.
3. The DB returns no row; the repository returns `Optional.empty()`.
4. Because `Optional.empty() != null`, the `unless` guard does not suppress caching; Caffeine stores the entry keyed by the non-existent `contractId`.
5. Once 1 000 distinct non-existent IDs have been queried, Caffeine's size-based eviction removes the oldest entries — which are the legitimate bytecode entries for real contracts.
6. All subsequent EVM calls to real contracts miss the cache and must re-query the database.
7. If the attacker continuously rotates through non-existent IDs at the allowed rate (up to 500 RPS per the throttle configuration), legitimate bytecodes can never re-populate the cache.

**Why existing checks are insufficient:**
- The `unless = "#result == null"` guard is the only protection against caching empty results. It is structurally incorrect for an `Optional`-returning method.
- There is no separate negative-result cache with a shorter TTL or smaller capacity.
- The rate limiter (500 RPS) is far too permissive to prevent filling a 1 000-entry cache; an attacker saturates it in ~2 seconds.

### Impact Explanation
The immediate effect is **cache thrashing**: every call to a real contract misses the cache and hits the database, increasing latency and database load. The more serious protocol-level effect is **stale negative caching**: because `expireAfterAccess=1h`, a cached `Optional.empty()` entry for a given `contractId` persists for one hour after the last access. If the attacker continuously re-accesses the same non-existent IDs (keeping entries alive), and one of those IDs corresponds to a contract that is subsequently deployed and indexed by the importer, the EVM will continue to see `Optional.empty()` for that contract for up to one hour. `ContractBytecodeReadableKVState.readFromDataSource()` maps `Optional.empty()` to `null` (via `.orElse(null)`), which the EVM interprets as "no bytecode / not a contract." Any `eth_call` or `eth_estimateGas` targeting that newly deployed contract will behave as if the contract does not exist — unintended smart contract behavior with no direct fund loss but incorrect execution results returned to callers.

### Likelihood Explanation
No authentication or special privilege is required. The web3 JSON-RPC endpoint is publicly accessible. Hedera contract IDs are sequential integers, making it trivial to enumerate both existing and non-existing IDs. The attack requires only standard `eth_call` requests and can be sustained indefinitely within the 500 RPS throttle. The 1 000-entry cache limit makes the attack cheap to execute and maintain.

### Recommendation
Fix the `unless` condition to correctly handle the `Optional` return type:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

This prevents `Optional.empty()` from ever being stored in the cache, eliminating the negative-result pollution vector entirely.

### Proof of Concept
```
# Fill the cache with 1 000 non-existent contract IDs (sequential IDs starting above the known max)
for i in $(seq 9000000 9001000); do
  curl -s -X POST http://<mirror-node-web3>:8545 \
    -H 'Content-Type: application/json' \
    -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_call\",\"params\":[{\"to\":\"0x$(printf '%040x' $i)\",\"data\":\"0x\"},\"latest\"],\"id\":$i}" &
done
wait

# Now call a known real contract — observe cache miss and DB re-query in metrics/logs
curl -X POST http://<mirror-node-web3>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_call","params":[{"to":"<real_contract_address>","data":"0x"},"latest"],"id":1}'
```
