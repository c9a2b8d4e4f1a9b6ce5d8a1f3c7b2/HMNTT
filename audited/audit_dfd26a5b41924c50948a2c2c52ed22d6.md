### Title
Log Amplification via Unchecked Pool Queries During Network Partition Enables Disk Exhaustion

### Summary
`BaseService.getRows()` unconditionally calls `pool().queryQuietly()` with no pool health check or application-level circuit breaker. During a network partition, every inbound HTTP request to `getAccountTokenAllowances()` attempts a DB query that immediately fails, and the Traefik ingress retry middleware (configured for 10 attempts per request) amplifies each external request into 10 DB failure attempts, each emitting a `logger.error()` with a full stack trace. An unprivileged attacker sending a sustained flood of requests during a partition can exhaust log disk storage and trigger secondary service failures.

### Finding Description

**Exact code path:**

`rest/service/tokenAllowanceService.js:88` → `super.getRows(sqlQuery, params)` → `rest/service/baseService.js:55-56`:
```js
async getRows(query, params) {
  return (await this.pool().queryQuietly(query, params)).rows;
}
```
No pool health check, no circuit breaker, no back-off.

`rest/utils.js:1518-1539` — `queryQuietly` catches every DB error and re-throws as `DbError`:
```js
} catch (err) {
  ...
  throw new DbError(err.message);
}
```

`rest/middleware/httpErrorHandler.js:21-22,38-39` — `DbError` maps to HTTP 503 and unconditionally calls `logger.error()` with the full error object (stack trace included, because `shouldPrintStacktrace` only suppresses logging for `statement timeout` errors, not connection errors):
```js
} else if (err instanceof DbError) {
  statusCode = httpStatusCodes.SERVICE_UNAVAILABLE;
...
logger.error(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode}`, detailedMessage);
```

**Retry amplification:**

`charts/hedera-mirror-rest/values.yaml:137-139`:
```yaml
- retry:
    attempts: 10
    initialInterval: 100ms
```
Traefik retries each 503 response up to 10 times. Each retry is a new request to the Node.js process, which re-enters `getRows()` → `queryQuietly()` → fails → `logger.error()`. One external request produces **10 DB connection attempts and 10 full-stack-trace error log entries**.

**No application-level rate limiting:** The REST chart has no `inFlightReq` or `rateLimit` Traefik middleware (unlike the Rosetta chart). An unauthenticated attacker can send unlimited concurrent requests.

**Traefik circuit breaker is insufficient:** `ResponseCodeRatio(500, 600, 0, 600) > 0.25` trips only after 25% of responses are 5xx, measured over a sliding window. During the window before it trips, and across any circuit-breaker reset cycles, the retry amplification continues to generate log entries at 10× the request rate.

### Impact Explanation

Each attacker request during a partition generates 10 `logger.error()` calls, each containing a full Node.js stack trace (several KB). At modest request rates (e.g., 1,000 req/s), this produces ~10,000 error log entries per second with multi-KB payloads, potentially writing tens of MB/s to disk. Log disk exhaustion causes the Node.js process to stall or crash on log writes, and can cascade to other services sharing the same storage volume. The service returns 503 to all legitimate users for the duration, constituting a complete availability failure.

### Likelihood Explanation

The precondition (network partition) is an infrastructure event outside attacker control, but attackers routinely monitor for degraded service states (e.g., via public status pages or error-rate monitoring) and time amplification attacks accordingly. Once a partition is detected, the exploit requires only unauthenticated HTTP GET requests to `/api/v1/accounts/{id}/tokens` — no credentials, no special knowledge, no elevated privileges. The attack is trivially repeatable with any HTTP load tool.

### Recommendation

1. **Application-level circuit breaker**: Implement a circuit breaker (e.g., `opossum`) in `BaseService.getRows()` or `queryQuietly()` that opens after a configurable number of consecutive DB failures and fast-fails subsequent requests without attempting a DB query or emitting a full error log.
2. **Rate-limit the REST ingress**: Add `inFlightReq` and `rateLimit` Traefik middleware to `charts/hedera-mirror-rest/values.yaml` (as already done for the Rosetta chart) to cap per-IP request rates.
3. **Reduce retry amplification**: Lower `retry.attempts` from 10 to 2–3, or disable retries for 503 responses specifically, to reduce the log amplification factor.
4. **Log deduplication / sampling**: Throttle `logger.error()` for repeated identical DB errors (e.g., emit at most one error log per second per error type during a degraded state).

### Proof of Concept

**Preconditions:** Network partition between the mirror-node REST service and its PostgreSQL backend is active (e.g., firewall rule blocking port 5432).

**Steps:**
1. Confirm the partition: `curl -s https://<mirror-node>/api/v1/accounts/0.0.1/tokens` returns HTTP 503.
2. Launch a sustained request flood from an unprivileged client:
   ```bash
   ab -n 100000 -c 200 "https://<mirror-node>/api/v1/accounts/0.0.1/tokens"
   ```
3. Each of the 100,000 requests is retried 10 times by Traefik → 1,000,000 calls to `getRows()` → 1,000,000 `logger.error()` entries with full stack traces written to disk.
4. Monitor log volume: `watch -n1 'du -sh /var/log/mirror-node/'` — observe rapid growth.
5. **Result:** Log partition fills; Node.js process stalls on log writes; service becomes unresponsive to all requests including health checks; Kubernetes liveness probe fails; pod is restarted, repeating the cycle.
