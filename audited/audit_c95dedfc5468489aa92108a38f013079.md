### Title
Unauthenticated DB Connection Pool Exhaustion via Alias-Based Crypto Allowance Endpoint

### Summary
The `/accounts/:idOrAliasOrEvmAddress/allowances/crypto` endpoint in `getAccountCryptoAllowances()` unconditionally calls `EntityService.getEncodedId()`, which issues an uncached database query for every request containing an account alias or EVM address. The REST service has no application-level rate limiting and the Traefik ingress middleware for the REST chart omits both `inFlightReq` and `rateLimit` controls (unlike other services such as Rosetta). An unprivileged attacker can flood the endpoint with distinct valid aliases, saturating the DB connection pool and rendering mirror node REST processing nodes unresponsive.

**No caching in EntityService:** A grep of `rest/service/entityService.js` for `cache`, `Cache`, `redis`, or `Redis` returns zero matches. The response-layer cache in `responseCacheHandler.js` keys on `req.originalUrl` (MD5 hash), so each distinct alias produces a unique cache key and a cache miss, bypassing the only caching layer entirely. [4](#0-3) 

**No application-level rate limiting:** The full REST middleware stack exported from `rest/middleware/index.js` contains `authHandler`, `handleError`, `openApiValidator`, `requestLogger`, `requestQueryParser`, `responseCacheCheckHandler`, `responseCacheUpdateHandler`, and `responseHandler` — no rate-limit or throttle middleware. [5](#0-4) 

The `authHandler` only sets a custom per-user query limit for authenticated users; unauthenticated requests pass through without any restriction. [6](#0-5) 

**No Traefik-level rate limiting for REST:** The `charts/hedera-mirror-rest/values.yaml` middleware block contains only `circuitBreaker` and `retry`. It is missing both `inFlightReq` and `rateLimit` entries that are present in the Rosetta chart. [7](#0-6) [8](#0-7) 

### Impact Explanation
Each alias-based request consumes one DB connection for the duration of the `entity` table lookup. With a high-concurrency flood of requests using distinct aliases (all cache misses), the `pg` connection pool in `rest/dbpool.js` is exhausted. Queued requests time out or error, causing the Node.js REST processes to return 5xx responses. The Traefik `circuitBreaker` expression `ResponseCodeRatio(500, 600, 0, 600) > 0.25` will then open the circuit breaker, taking affected REST pods out of rotation. With enough concurrent attack traffic across multiple pods, 30%+ of REST processing nodes can be shut down, matching the stated severity tier. [9](#0-8) 

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public API schema, and no brute force. Valid account aliases are publicly observable on-chain. An attacker needs only a script generating distinct valid base32-encoded aliases and a moderate-bandwidth connection. The attack is repeatable and can be sustained indefinitely. The absence of both application-level and infrastructure-level rate limiting for this service makes it straightforward to execute.

### Recommendation
1. **Add `inFlightReq` and `rateLimit` Traefik middleware** to `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta chart configuration.
2. **Add application-level rate limiting** (e.g., `express-rate-limit`) in the REST middleware stack, keyed on client IP, applied before route handlers.
3. **Cache alias→entityId resolutions** inside `EntityService.getEncodedId()` using a short-TTL in-process or Redis cache, so repeated or concurrent alias lookups do not each consume a DB connection.
4. **Enforce a DB query concurrency cap** in `dbpool.js` with a bounded pool size and explicit request queuing limits to prevent pool exhaustion under load.

### Proof of Concept
```bash
# Generate N distinct valid base32 aliases and flood the endpoint concurrently
# Valid aliases are base32-encoded public keys observable on-chain

for i in $(seq 1 5000); do
  # Each alias is a distinct valid base32 string (e.g., from observed on-chain accounts)
  ALIAS="KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HM${i}"
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/allowances/crypto" &
done
wait
```

Each request:
1. Passes `AccountAlias.isValid()` in `getEncodedId()` (line 125 of `entityService.js`)
2. Triggers `getAccountFromAlias()` → SQL query against `entity` table (line 43 of `entityService.js`)
3. Produces a unique `responseCacheHandler` cache key → always a cache miss
4. Holds a DB pool connection for the query duration

Under sustained concurrency the pool is exhausted, REST pods begin returning 500s, and the Traefik circuit breaker removes them from rotation.

    try {
      if (EntityId.isValidEntityId(entityIdString)) {
        const entityId = EntityId.parseString(entityIdString, {paramName});
        return entityId.evmAddress === null
          ? entityId.getEncodedId()
          : await this.getEntityIdFromEvmAddress(entityId, requireResult);
      } else if (AccountAlias.isValid(entityIdString)) {
        return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
      }

      if (ex instanceof InvalidArgumentError) {
        throw InvalidArgumentError.forParams(paramName);
      }
      // rethrow
      throw ex;
    }

    throw InvalidArgumentError.forParams(paramName);
  }
