### Title
Unauthenticated EVM Address `from` Parameter Triggers Unconditional Extra DB Lookup per Request on `GET /contracts/results`

### Summary
Any unauthenticated user can send `GET /contracts/results?from=<evm_address>` to force an extra database query (`entityFromEvmAddressQuery` against the `entity` table) per request inside `extractContractResultsByIdQuery`, before the main contract results query executes. Because the endpoint is fully public, no authentication is required, and no rate limiting is applied in the REST middleware, an attacker can flood this endpoint with concurrent requests — each carrying one or more valid EVM address `from` values — to amplify database load proportionally.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 19 registers the endpoint with no authentication: [1](#0-0) 

`getContractResults` (the handler) calls `extractContractResultsByIdQuery(filters)` unconditionally: [2](#0-1) 

Inside `extractContractResultsByIdQuery`, for every filter with key `filterKeys.FROM`, if the value passes `EntityId.isValidEvmAddress()`, `EntityService.getEncodedId()` is awaited — a live DB call: [3](#0-2) 

`getEncodedId` routes to `getEntityIdFromEvmAddress`, which always executes `entityFromEvmAddressQuery` — a `SELECT id FROM entity WHERE evm_address = $1` — regardless of whether the address exists: [4](#0-3) 

The query itself: [5](#0-4) 

**Root cause:** The EVM address resolution is performed eagerly and unconditionally for every `from` filter value that is a valid 20-byte hex address, with no caching, no deduplication, and no rate limiting at the REST layer. The `isValidEvmAddress` guard only gates *which branch* of `getEncodedId` is taken — it does not prevent the DB query.

**Why existing checks fail:** The `isValidEvmAddress` check at line 443 is a format check, not a guard against DB access. The `requireResult = true` default means the DB query always runs; a `NotFoundError` is only thrown *after* the query returns 0 rows. The `requestHandler.js` middleware supports repeated query parameters up to `config.query.maxRepeatedQueryParameters`, meaning multiple `from` EVM addresses per request are accepted, each triggering a separate DB lookup. [6](#0-5) 

No rate-limiting middleware is applied to this route in `server.js`: [7](#0-6) 

### Impact Explanation
Each request to `GET /contracts/results?from=<evm_address>` causes at minimum one extra synchronous DB query before the main query. With N repeated `from` EVM address parameters per request, the attacker forces N extra DB queries per request. Under high concurrency, this exhausts the PostgreSQL connection pool and degrades or denies service for all users of the mirror node REST API. Because the mirror node serves as critical read infrastructure for the Hedera/Hiero ecosystem, degradation affects all downstream consumers (wallets, explorers, dApps).

### Likelihood Explanation
The attack requires zero privileges, zero tokens, and zero knowledge of the system beyond the public API spec. The endpoint is documented and publicly accessible. The attacker only needs to know that EVM addresses (any syntactically valid 20-byte hex string) are accepted as `from` values. The attack is trivially scriptable with `curl` or any HTTP load tool, is repeatable indefinitely, and can be amplified by sending multiple `from` parameters per request.

### Recommendation
1. **Cache EVM address → entity ID lookups** (e.g., in-process LRU or Redis) so repeated lookups for the same address do not hit the DB.
2. **Apply rate limiting** at the middleware layer for all `/contracts/results` endpoints, keyed by IP or API key.
3. **Resolve EVM addresses lazily** or batch-resolve after input validation, rather than resolving each filter value individually in a loop before the main query.
4. **Cap the number of repeated `from` parameters** accepted per request to a small value (e.g., 1–5) to limit per-request amplification.

### Proof of Concept

```bash
# Single request — forces 1 extra DB query (entity lookup) + 1 main query
curl "https://<mirror-node-host>/api/v1/contracts/results?from=0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"

# Amplified — forces N extra DB queries per request (N repeated from params)
curl "https://<mirror-node-host>/api/v1/contracts/results?from=0xaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa&from=0xbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb&from=0xcccccccccccccccccccccccccccccccccccccccc"

# DoS flood — concurrent requests, each triggering extra DB queries
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/results?from=0x$(openssl rand -hex 20)" &
done
wait
```

Each request in the flood causes `EntityService.getEncodedId` → `getEntityIdFromEvmAddress` → `SELECT id FROM entity WHERE evm_address = $1` to execute before the main `contract_result` query, doubling (or more) the DB query rate relative to the request rate, exhausting the connection pool under sustained load.

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
