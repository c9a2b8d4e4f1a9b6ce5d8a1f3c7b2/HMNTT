### Title
HikariCP Connection Pool Exhaustion via Unbounded `findById()` Calls During Partial Network Partition

### Summary
`TopicServiceImpl.findById()` delegates directly to `TopicRepository.findById()` (a plain Spring Data `CrudRepository`) with no per-request query timeout, no application-level circuit breaker, and no in-flight request cap. During a partial network partition where TCP connections to PostgreSQL hang rather than reset, concurrent unauthenticated requests to `GET /api/v1/topics/{id}` will hold open every HikariCP connection indefinitely, starving all other services sharing the same pool and rendering the rest-java service completely unavailable.

### Finding Description

**Exact code path:**

`TopicController.getTopic()` → `TopicServiceImpl.findById()` → `TopicRepository.findById()` (Spring Data CrudRepository) → HikariCP → JDBC → PostgreSQL. [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — three absent controls:**

1. **No statement/query timeout on the JDBC layer for rest-java.** The `web3` module has a `StatementInspector` that enforces `requestTimeout` per query, but rest-java has no equivalent. The HikariCP bean is built from `spring.datasource.hikari` properties with no default `connectionTimeout` or `statementTimeout` set in code. [4](#0-3) 

2. **No application-level circuit breaker.** No `@CircuitBreaker`, no Resilience4j, no `@Timeout` annotation anywhere in the rest-java service layer or repository layer. The `ImporterConfiguration` uses `@EnableResilientMethods` but rest-java does not. [5](#0-4) 

3. **No `inFlightReq` middleware.** The Traefik middleware for rest-java only has a `circuitBreaker` (error-ratio based) and `retry`. Compare with the Rosetta chart which adds `inFlightReq: amount: 5` per IP. [6](#0-5) 

**Why the Traefik circuit breaker is insufficient:**

The expression `NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25` only trips on *errors*. During a partial network partition, TCP connections to PostgreSQL hang silently — they produce no errors and no 5xx responses until HikariCP's `connectionTimeout` (default 30 s) expires. The circuit breaker never sees the error ratio threshold crossed during the hang window. [7](#0-6) 

**The retry middleware amplifies the attack:**

`retry: attempts: 3` means Traefik retries each hanging request up to 3 times, tripling the number of held connections per user request. [8](#0-7) 

### Impact Explanation

HikariCP's default pool size is 10 connections. With 10 concurrent hanging requests (each retried 3× by Traefik = effectively 30 connection-hold attempts), the pool is exhausted. All subsequent requests — including `entityService.findById()` and `customFeeService.findById()` called in the same `getTopic()` handler, plus every other endpoint in the rest-java service — block waiting for a connection until `connectionTimeout` expires, returning 500s. This constitutes a full service-level denial of service for the rest-java pod for the duration of the partition.

### Likelihood Explanation

No authentication is required to call `GET /api/v1/topics/{id}`. The endpoint is publicly routed via ingress. [9](#0-8) 

A partial network partition (e.g., asymmetric packet loss, firewall rule blocking DB RST packets, cloud provider transient routing issue) is a realistic infrastructure event. An attacker does not need to induce the partition — they only need to detect elevated latency and flood the endpoint with concurrent requests. Even without malicious intent, a natural traffic spike during a partition achieves the same result. The attack is trivially repeatable with a simple HTTP load tool (`ab`, `wrk`, `hey`) sending ~15 concurrent requests.

### Recommendation

1. **Add a JDBC statement timeout** via HikariCP `connectionInitSql` or Spring Boot property `spring.datasource.hikari.connection-timeout` and `spring.datasource.hikari.validation-timeout`, and set a PostgreSQL-level `statement_timeout` on the rest-java DB role (e.g., `SET statement_timeout = '5000'`).
2. **Add `inFlightReq` Traefik middleware** to the rest-java chart (matching the Rosetta pattern) to cap concurrent requests per source IP.
3. **Add an application-level timeout** using Spring's `@Transactional(timeout = N)` on `TopicServiceImpl.findById()`, or adopt Resilience4j `@TimeLimiter` / `@CircuitBreaker` as already used in the importer module.
4. **Separate HikariCP pools** per service if multiple services share one pool, to prevent cross-service starvation.

### Proof of Concept

**Preconditions:**
- rest-java is deployed and `GET /api/v1/topics/0.0.1` returns 200.
- Simulate partial partition: use `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the rest-java host (drops packets silently, TCP connections hang).

**Steps:**
```bash
# 1. Induce silent partition to PostgreSQL
iptables -A OUTPUT -p tcp --dport 5432 -j DROP

# 2. Send 15 concurrent requests (exceeds default pool of 10, with retry amplification)
hey -n 100 -c 15 http://<rest-java-host>/api/v1/topics/0.0.1

# 3. Observe: first ~10 requests hang for 30s (HikariCP connectionTimeout)
#    Subsequent requests immediately return 500 (pool exhausted)
#    All other endpoints (e.g., /api/v1/network/supply) also return 500

# 4. Restore
iptables -D OUTPUT -p tcp --dport 5432 -j DROP
```

**Expected result:** During the partition window, 100% of requests to all rest-java endpoints fail with 500 or timeout. The Traefik circuit breaker does not open until after the 30-second hang window produces enough 5xx responses, by which time the service has already been unavailable for ~30 seconds per wave of requests.
