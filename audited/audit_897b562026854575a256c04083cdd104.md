### Title
Unenforced `startTime` Validation Allows Perpetual No-Op DB Polling via `Long.MAX_VALUE` Timestamp Injection

### Summary
An unprivileged gRPC client can send a `ConsensusTopicQuery` with `consensusStartTime.seconds >= 9223372035L`, which `ConsensusController.convertTimestamp()` explicitly converts to `Long.MAX_VALUE`. The `TopicMessageFilter.isValidStartTime()` guard exists but is never enforced because `subscribeTopic()` lacks `@Valid` on its parameter. When the listener type is `POLL`, `PollingTopicListener.poll()` then issues `WHERE consensus_timestamp >= Long.MAX_VALUE` to the database on every poll interval (default 500 ms) indefinitely, since no message is ever returned and `last` stays `null`.

### Finding Description

**Exact code path:**

1. **`ConsensusController.java` lines 76–81** — `convertTimestamp()` explicitly maps any `seconds >= 9223372035L` to `Long.MAX_VALUE`:
   ```java
   if (timestamp.getSeconds() >= 9223372035L) {
       return Long.MAX_VALUE;
   }
   ```
   This value is then stored via `filter.startTime(Long.MAX_VALUE)` and `filter.build()` is called with no validation.

2. **`TopicMessageFilter.java` lines 48–51** — The guard exists:
   ```java
   @AssertTrue(message = "Start time must be before the current time")
   public boolean isValidStartTime() {
       return startTime <= DomainUtils.now();
   }
   ```
   But `TopicMessageFilter` is not a Spring bean, so `@Validated` on it is inert. `TopicMessageServiceImpl.subscribeTopic(TopicMessageFilter filter)` has no `@Valid` on the parameter, so Spring AOP method validation is never triggered. No explicit `Validator.validate()` call exists anywhere in the call chain. The constraint is dead code.

3. **`PollingTopicListener.java` line 58** — In `poll()`:
   ```java
   long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
   ```
   On every poll iteration while `last == null` (which is permanent when no messages match), `startTime = Long.MAX_VALUE`. The `toBuilder().limit(pageSize).startTime(startTime).build()` on line 59 also bypasses validation.

4. **`TopicMessageRepositoryCustomImpl.java` lines 38–40** — `findByFilter()` generates:
   ```java
   cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime())
   ```
   → `WHERE consensus_timestamp >= 9223372036854775807` — always empty, always executed.

5. **`PollingTopicListener.java` lines 40–43** — The repeat loop runs `Long.MAX_VALUE` times with a 500 ms fixed delay, meaning each subscription fires ~2 DB queries/second forever.

**Root cause:** `convertTimestamp()` intentionally produces `Long.MAX_VALUE` as a sentinel, but the downstream validation that should reject it (`isValidStartTime()`) is never invoked due to a missing `@Valid` annotation on the service method parameter.

**Note on the `last + 1` path:** The question's framing of `last.getConsensusTimestamp() + 1 = Long.MAX_VALUE` would require a DB message with `consensusTimestamp = Long.MAX_VALUE - 1`, which is not user-controllable. The actual exploitable path is the direct one above (`last == null`, uses `filter.getStartTime()` directly).

### Impact Explanation

Each malicious subscription generates one DB query every 500 ms indefinitely against the `topic_message` table. `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls per connection but not total connections — an attacker opens many TCP connections. With hundreds of connections × 5 calls each, the DB receives thousands of queries per second. These queries hit the `(topic_id, consensus_timestamp)` index but still consume DB worker threads, connection pool slots, and I/O bandwidth. This degrades or blocks the importer's write path (same DB), causing the mirror node to fall behind ingesting new transactions. The mirror node itself becomes unavailable to legitimate subscribers. This is a mirror-node-level DoS, not a main-network shutdown — the Hedera consensus nodes are unaffected.

**Precondition:** The deployment must use `hiero.mirror.grpc.listener.type=POLL` (non-default; default is `REDIS`). With `REDIS` or `SHARED_POLL`, `PollingTopicListener` is not invoked, but `PollingTopicMessageRetriever.retrieve()` (always active for historical retrieval) has the identical `last.getConsensusTimestamp() + 1` pattern and the same validation bypass, so a related DoS applies regardless of listener type.

### Likelihood Explanation

The attack requires only a standard gRPC client (e.g., `grpcurl`) and knowledge of the protobuf schema — no credentials, no special network position. The trigger value (`seconds >= 9223372035`) is a single field in a public protobuf message. The attack is trivially repeatable and scriptable. The only deployment constraint is `listener.type=POLL`, which is a documented configuration option.

### Recommendation

1. **Enforce validation at the service boundary:** Add `@Valid` to the `filter` parameter in `TopicMessageService.subscribeTopic(@Valid TopicMessageFilter filter)` so Spring AOP triggers `isValidStartTime()`.

2. **Guard `poll()` against sentinel values:** In `PollingTopicListener.poll()`, add an explicit check:
   ```java
   if (startTime == Long.MAX_VALUE) return Flux.empty();
   ```

3. **Reconsider `convertTimestamp` sentinel:** Instead of returning `Long.MAX_VALUE` for out-of-range timestamps, reject the request with a `INVALID_ARGUMENT` gRPC status before building the filter.

4. **Add per-IP or per-client subscription limits** to bound the total number of concurrent polling subscriptions.

### Proof of Concept

```bash
# Using grpcurl against a mirror node configured with listener.type=POLL
# Send consensusStartTime.seconds = 9223372036 (>= 9223372035)
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 9223372036, "nanos": 0}
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

Repeat this in a loop across many connections. Each open stream causes `PollingTopicListener.poll()` to execute `SELECT ... FROM topic_message WHERE topic_id = 1 AND consensus_timestamp >= 9223372036854775807 ORDER BY consensus_timestamp ASC` every 500 ms. Monitor DB `pg_stat_activity` to observe the query flood. The stream never completes (no `endTime` set, no `limit` set), so the polling continues until the client disconnects.
