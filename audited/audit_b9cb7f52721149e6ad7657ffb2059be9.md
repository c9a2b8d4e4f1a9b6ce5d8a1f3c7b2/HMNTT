### Title
Unbounded Mirror Node Response Accepted Without Size or Address Validation in `fetchMirrorNodeNetwork()`

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` decodes the HTTP response from the configured mirror node into a `nodesEnvelope` struct with no body-size cap, no limit on the number of `Nodes` or `ServiceEndpoints`, and no validation of the returned host/IP values. A party controlling the mirror node endpoint (or in a MITM position on the unencrypted HTTP channel) can return a crafted response that floods the SDK network map with attacker-controlled gRPC addresses, causing the SDK client — which has the operator's private key set — to route signed transactions to those addresses.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–132.

The only guard is the empty-string/zero-port check (line 118). There is:
- No cap on `len(payload.Nodes)`
- No cap on `len(n.ServiceEndpoints)`
- No allowlist or format validation of `host` values (private RFC-1918 addresses, loopback, attacker-owned domains all accepted)
- No TLS certificate pinning or HTTPS enforcement (config example is `http://mirror-rest:5551`)

The resulting `network` map is passed directly to `hiero.ClientForNetwork(netmap)` and then `client.SetOperator(opID, opKey)` is called, binding the operator's private key to a client whose node list is entirely attacker-supplied. [3](#0-2) 

**Root cause:** Blind trust of the mirror node HTTP response with no size bound, no entry count limit, and no address validation before the data is used to configure a credentialed SDK client.

### Impact Explanation

1. **Memory exhaustion / OOM**: A single crafted response with millions of node entries causes the Go process to allocate unbounded memory, crashing the pinger pod.
2. **Transaction routing hijack**: The SDK client selects gRPC endpoints from the attacker-supplied map. Every periodic transfer (signed with the operator's private key) is submitted to attacker-controlled infrastructure. The attacker receives the fully-signed transaction bytes, can observe them, selectively drop them (liveness DoS), or forward them to the real network at a chosen time.
3. **No owner approval for endpoint selection**: The operator configured a mirror node URL, not individual consensus node addresses. The implicit assumption is that the mirror node returns legitimate network topology; there is no secondary approval step before the returned addresses are trusted.

### Likelihood Explanation

The `network=other` mode is explicitly designed for non-standard deployments where the operator supplies a custom mirror node URL. In such environments:
- The HTTP channel is commonly unencrypted (`http://`), making MITM feasible for any on-path network actor (same Kubernetes namespace, compromised sidecar, cloud VPC route injection).
- DNS poisoning of the hostname in `mirrorRest` requires no credentials on the target system.
- A supply-chain compromise of the mirror node service itself (a separate, lower-privilege component) is sufficient.

No authentication of the mirror node response is required; the attack is repeatable on every pinger restart.

### Recommendation

1. **Limit response body size**: Wrap `resp.Body` with `io.LimitReader` before decoding (e.g., 1 MB).
2. **Cap node and endpoint counts**: Reject or truncate responses where `len(payload.Nodes)` or per-node `len(ServiceEndpoints)` exceeds a reasonable bound (e.g., 100 nodes, 10 endpoints each).
3. **Validate returned addresses**: Reject RFC-1918, loopback, link-local, and unresolvable addresses unless explicitly allowlisted.
4. **Enforce HTTPS**: Reject `mirrorRest` URLs that do not use `https://`, or pin the TLS certificate/CA.
5. **Allowlist expected `NodeAccountID` values**: Cross-check returned account IDs against a known-good set from config before adding them to the network map.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other`
- Attacker controls the host at `HIERO_MIRROR_PINGER_REST` (via DNS poisoning, MITM on HTTP, or direct server compromise)

**Steps:**

1. Stand up an HTTP server that responds to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        {"ip_address_v4": "<attacker-IP>", "port": 50211},
        ... (repeat thousands of times across thousands of node entries)
      ]
    }
  ],
  "links": {"next": null}
}
```
2. Point `HIERO_MIRROR_PINGER_REST` at the malicious server (or intercept the HTTP request in transit).
3. The pinger starts, calls `fetchMirrorNodeNetwork()`, decodes the unbounded payload, and builds a `network` map containing only attacker-controlled `<attacker-IP>:50211` entries.
4. `hiero.ClientForNetwork(netmap)` is called with this map; `client.SetOperator(opID, opKey)` binds the operator key.
5. On every ticker tick, `submitWithRetry` submits a signed `CryptoTransfer` transaction to `<attacker-IP>:50211`. The attacker receives the signed transaction bytes over gRPC.
