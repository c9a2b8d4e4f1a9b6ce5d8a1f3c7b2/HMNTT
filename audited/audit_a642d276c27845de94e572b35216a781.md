### Title
Global Rate Limit Bucket Exhaustion via Distributed Clients (Application-Layer DoS)

### Summary
`rateLimitBucket()` creates a single JVM-local token bucket shared across all incoming requests with no per-IP, per-session, or per-client partitioning. An unprivileged attacker controlling a modest botnet can collectively exhaust the global 500 req/s default capacity, causing all legitimate users to receive HTTP 429 responses while the attacker's requests dominate the allowed throughput.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` (lines 24–32) instantiates one `Bucket` singleton with capacity equal to `throttleProperties.getRequestsPerSecond()` (default: 500):

```java
// ThrottleConfiguration.java:25-31
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

This single bean is injected into `ThrottleManagerImpl` and consumed at line 38:

```java
// ThrottleManagerImpl.java:38-39
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

Every request from every client — regardless of source IP — consumes exactly one token from the same shared bucket. There is no per-IP bucket, no per-session bucket, and no IP field in `RequestFilter.FilterField` (which only exposes `BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, `VALUE`). The `FROM` field is the EVM sender address supplied in the JSON body, not the network-layer client IP, so it can be freely spoofed per-request.

**Root cause / failed assumption:** The design assumes the global limit is sufficient to protect the service, implicitly assuming that no single actor can generate a significant fraction of the total allowed throughput. This assumption fails against distributed clients.

### Impact Explanation
With the default 500 req/s global limit, a botnet of 50 nodes each sending 11 req/s (550 req/s total) fully saturates the bucket. All subsequent requests from legitimate users receive HTTP 429 `Too Many Requests` until the bucket refills. Because the bucket refills greedily at 500 tokens/second and the attacker continuously drains it, legitimate users are effectively locked out for the duration of the attack. This constitutes a complete availability denial for the `/api/v1/contracts/call` endpoint, which is the primary interface for querying historical Hashgraph EVM state.

### Likelihood Explanation
No authentication, API key, or network-layer credential is required to send requests to the endpoint. The attacker needs only HTTP access and enough distributed clients to exceed 500 req/s collectively — trivially achievable with a small botnet or even a single high-bandwidth host using concurrent connections. The attack is repeatable indefinitely at zero cost per request (each request is rejected before any EVM execution occurs, so the attacker pays no gas). The Rosetta service has Traefik-level per-IP rate limiting configured (`sourceCriterion: requestHost: true`), confirming the project is aware of this class of risk, but no equivalent protection exists for the web3 module.

### Recommendation
1. **Per-IP rate limiting at the application layer**: Use Bucket4j's distributed or keyed bucket support to maintain a separate bucket per client IP (extracted from `X-Forwarded-For` or `RemoteAddr`), in addition to the global bucket.
2. **Infrastructure-level protection**: Apply a Traefik/nginx/API-gateway rate-limit middleware keyed on source IP for the web3 service, mirroring the existing Rosetta configuration.
3. **Configurable per-IP limit**: Add a `requestsPerSecondPerClient` property to `ThrottleProperties` and enforce it in `ThrottleManagerImpl.throttle()` using the request's source IP.

### Proof of Concept
**Preconditions:** Network access to the web3 `/api/v1/contracts/call` endpoint; no credentials required.

**Steps:**
1. Spin up 50 concurrent HTTP clients (e.g., using `wrk`, `hey`, or a simple script).
2. Each client sends POST requests to `/api/v1/contracts/call` with a minimal valid JSON body at ~11 req/s.
3. Collective throughput: ~550 req/s, exceeding the default 500-token bucket.
4. **Result:** Legitimate single-client requests begin receiving `HTTP 429 Too Many Requests` with body `{"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}`.
5. The attack sustains indefinitely; the bucket never accumulates tokens because the attacker continuously drains it faster than the 500/s refill rate.
