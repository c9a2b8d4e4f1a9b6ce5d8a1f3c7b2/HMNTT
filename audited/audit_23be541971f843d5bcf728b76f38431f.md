### Title
Unauthenticated HTTP Mirror-Node Response Allows Full Network-Map Replacement via MITM

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` builds the entire Hiero SDK network map from a single unauthenticated, plaintext HTTP response. No TLS is enforced, no response is authenticated, and no returned IP address is validated against any allowlist. An attacker with a MITM position on the path between the pinger and the mirror-REST endpoint can inject a crafted `nodesEnvelope` containing exactly one entry pointing to an attacker-controlled gRPC server, causing every subsequent HBAR transfer transaction to be submitted exclusively to that server.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes` (lines 36-72) calls `fetchMirrorNodeNetwork` (lines 74-132).

The HTTP client is constructed with no TLS configuration and no authentication:
```go
// mirror_node_client.go line 46
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

The response is decoded and iterated with only format-level checks:
```go
// lines 102-124
for _, n := range payload.Nodes {
    if n.NodeAccountID == "" { continue }
    nodeAccountId, err := hiero.AccountIDFromString(n.NodeAccountID)
    if err != nil { continue }
    for _, ep := range n.ServiceEndpoints {
        host := strings.TrimSpace(ep.DomainName)
        if host == "" { host = strings.TrimSpace(ep.IPAddressV4) }
        if host == "" || ep.Port == 0 { continue }
        addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
        network[addr] = nodeAccountId
    }
}
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

**Root cause / failed assumption:** The code assumes the HTTP response originates from a trusted mirror node. There is no TLS enforcement, no HMAC/signature on the response, and no allowlist of valid node IPs. The only liveness check is `len(network) == 0` (line 127), which an attacker trivially satisfies with a single entry.

**Exploit flow:**

An attacker crafts this JSON response:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "domain_name": "", "ip_address_v4": "ATTACKER_IP", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```

All legitimate node entries in the real response have their `DomainName` and `IPAddressV4` stripped (set to `""`). The loop at line 118 skips every entry where `host == ""`, so they are silently dropped. The single attacker entry passes all checks. `len(network) == 1 > 0`, so the guard at line 127 passes. `hiero.ClientForNetwork(netmap)` (sdk_client.go line 22) is called with a map containing only the attacker's address.

**Why existing checks are insufficient:**

| Check | What it validates | What it misses |
|---|---|---|
| `n.NodeAccountID == ""` | Non-empty string | Whether the account ID is a real Hedera node |
| `AccountIDFromString` | Parseable format | Whether the account ID is authoritative |
| `host == "" \|\| ep.Port == 0` | Non-empty host + non-zero port | Whether the IP belongs to a legitimate node |
| `len(network) == 0` | At least one entry | Whether entries are authentic |

### Impact Explanation

Once the network map is replaced, `hiero.ClientForNetwork(netmap)` (sdk_client.go line 22) routes every `TransferTransaction.Execute(client)` call (transfer.go line 33) to the attacker's gRPC server. The attacker receives fully-signed transaction payloads. Direct HBAR theft via signature forgery is not possible because transactions are already signed with the operator key. However:

- **Denial of service**: The attacker drops all transactions; no transfers are ever confirmed on-chain.
- **Transaction data harvesting**: The attacker collects signed transaction bodies, operator account IDs, destination account IDs, and amounts on every tick.
- **Selective replay**: The attacker can selectively forward transactions to the real network, controlling which transfers succeed or fail.

Severity: **High** — complete loss of transaction availability and full visibility into all transfer activity for the pinger operator.

### Likelihood Explanation

The `network=other` path is the only one that calls `fetchMirrorNodeNetwork`; standard named networks use hardcoded SDK node lists. For `other`, the mirror REST URL is operator-supplied via `HIERO_MIRROR_PINGER_REST`. The documented example value is `http://mirror-rest:5551` — plaintext HTTP. In a Kubernetes deployment (evidenced by the `/tmp/alive` liveness probe and Dockerfile), any attacker who has compromised a pod in the same namespace can perform DNS spoofing or ARP poisoning to intercept this HTTP call. No application-level credentials are required; only network-layer access is needed, which is a low bar in shared-cluster environments. The attack is repeatable on every pinger restart.

### Recommendation

1. **Enforce HTTPS**: Require `cfg.mirrorRest` to use `https://` and reject plaintext URLs at config validation time (`config.go`).
2. **Pin or validate the TLS certificate**: Configure the `http.Client` with a `tls.Config` that pins the expected CA or leaf certificate for the mirror node.
3. **Validate returned node IPs against a known-good allowlist**: Maintain a configurable set of expected node account IDs and/or CIDR ranges; reject any `nodesEnvelope` entry that falls outside them.
4. **Enforce a minimum node count**: Reject any response that returns fewer than N nodes (e.g., N=3) to prevent single-node substitution attacks.
5. **Add response authentication**: If the mirror node is operator-controlled, sign responses with an HMAC key shared at deployment time and verify it in `fetchMirrorNodeNetwork`.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
- Attacker has network-level access to intercept traffic to `mirror-rest:5551` (e.g., compromised pod in same Kubernetes namespace, or ARP spoofing on the same L2 segment).

**Steps:**

1. Attacker starts a fake HTTP server on their controlled host listening on port 80 (or intercepts via transparent proxy).

2. Attacker's server responds to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "domain_name": "", "ip_address_v4": "192.0.2.1", "port": 50211 }
      ],
      "grpc_proxy_endpoint": null
    }
  ],
  "links": { "next": null }
}
```

3. Attacker starts a fake gRPC server on `192.0.2.1:50211` that accepts `CryptoTransfer` RPCs and logs all received transaction bytes (or drops them).

4. Pinger starts, calls `buildNetworkFromMirrorNodes` → `fetchMirrorNodeNetwork`, receives the injected response.

5. `network` map is built as `{"192.0.2.1:50211": AccountID{0,0,3}}`.

6. `hiero.ClientForNetwork(network)` is called; all subsequent `TransferTransaction.Execute(client)` calls in `submitWithRetry` (transfer.go line 33) are directed to `192.0.2.1:50211`.

7. **Result**: All HBAR transfer transactions are routed exclusively to the attacker's gRPC endpoint. The attacker observes every signed transaction payload; no transfers are confirmed on the real network.
