### Title
Unauthenticated Flood of `/network/status` Causes Repeated DB Queries with No Application-Level Rate Limiting

### Summary
The `NetworkStatus()` handler in `rosetta/app/services/network_service.go` completely ignores the `NetworkRequest` parameter and unconditionally executes multiple database queries on every invocation. The Rosetta service has no application-level rate limiting — only optional, infrastructure-level Traefik middleware that is not enforced at the process boundary. Any unauthenticated external user who supplies the publicly discoverable correct network identifier can flood the endpoint and drive sustained DB load.

### Finding Description

**Code location:** `rosetta/app/services/network_service.go`, `NetworkStatus()`, lines 59–88.

```go
func (n *networkAPIService) NetworkStatus(
    ctx context.Context,
    _ *rTypes.NetworkRequest,   // ← parameter is entirely ignored
) (*rTypes.NetworkStatusResponse, *rTypes.Error) {
    if !n.IsOnline() {
        return nil, errors.ErrEndpointNotSupportedInOfflineMode
    }
    genesisBlock, err := n.RetrieveGenesis(ctx)   // DB query (once, via sync.Once)
    ...
    currentBlock, err := n.RetrieveLatest(ctx)    // DB query on EVERY call
    ...
    peers, err := n.addressBookEntryRepo.Entries(ctx) // DB query on EVERY call
    ...
}
```

**Root cause — two compounding issues:**

1. **`NetworkRequest` is silently discarded.** The blank identifier `_` means the service performs zero per-request validation of the caller-supplied network identifier. The only guard that touches the network identifier is the Rosetta SDK `asserter`, which runs *before* the service method and rejects requests whose `network_identifier` does not match the configured network. However, the correct network identifier is trivially obtained by any caller via the unauthenticated `/network/list` endpoint (`NetworkList` also ignores its parameter and returns the single configured identifier unconditionally). Once an attacker has the valid identifier, every subsequent `/network/status` request passes the asserter and reaches the DB.

2. **No application-level rate limiting exists in the Rosetta process.** The middleware chain assembled in `main.go` (lines 217–219) is:
   ```
   MetricsMiddleware → TracingMiddleware → CorsMiddleware
   ```
   None of these throttle requests. The rate-limiting infrastructure found in the repository (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j) lives exclusively in the `web3` Spring Boot module and is never wired into the Go Rosetta service. The Helm chart (`charts/hedera-mirror-rosetta/values.yaml`, lines 157–160) defines a Traefik `rateLimit` middleware (`average: 10` per host), but this is an optional Kubernetes ingress configuration — it is not enforced when the service is accessed directly, when Traefik is not deployed, or when `global.middleware: false` (the default, line 95).

**Per-request DB cost:** Every call that reaches `NetworkStatus()` executes:
- `blockRepository.RetrieveLatest` → `SELECT … FROM record_file ORDER BY index DESC LIMIT 1` (always hits DB)
- `addressBookEntryRepository.Entries` → a multi-table JOIN across `address_book`, `address_book_entry`, `node`, and `address_book_service_endpoint` (always hits DB, up to two queries)

`RetrieveGenesis` is amortized via `sync.Once` after the first call, but the other two queries run on every request.

### Impact Explanation

An attacker can sustain a high request rate against `/network/status` with no credential, no token, and no per-IP enforcement at the application layer. Each request issues at least two non-trivial SQL queries against the shared PostgreSQL mirror-node database. Sustained flooding degrades query latency for all other Rosetta endpoints (block, account, construction) and for the importer that writes to the same database, causing service degradation or outage for legitimate users. Because the endpoint is stateless and the payload is tiny, the attack is cheap to mount and hard to distinguish from legitimate polling at the network level.

### Likelihood Explanation

The attack requires zero privileges: the attacker needs only the correct network identifier (one unauthenticated call to `/network/list`) and the ability to send HTTP POST requests. The endpoint is publicly reachable on port 5700 (default). No exploit code, no cryptographic material, and no prior knowledge of the deployment is required. The attack is trivially repeatable and scriptable with a single `curl` loop or any HTTP load-generation tool.

### Recommendation

1. **Add application-level rate limiting inside the Go process.** Introduce a per-IP (or global) token-bucket middleware — analogous to the `ThrottleManagerImpl` pattern already used in the `web3` module — and apply it to all Rosetta routes before the service handler is reached. This must be in-process so it is enforced regardless of how the service is exposed.

2. **Do not rely solely on infrastructure middleware.** The Traefik `rateLimit` in the Helm chart is a useful defence-in-depth layer but must not be the only control; direct access to the pod port bypasses it entirely.

3. **Consider caching the `/network/status` response** for a short TTL (e.g., 1–2 seconds) since the data (latest block, peer list) changes infrequently relative to the polling rate an attacker would use.

### Proof of Concept

```bash
# Step 1 – discover the valid network identifier (no credentials required)
NETWORK_ID=$(curl -s -X POST http://<rosetta-host>:5700/network/list \
  -H 'Content-Type: application/json' \
  -d '{"metadata":{}}' | jq '.network_identifiers[0]')

# Step 2 – flood /network/status; each request triggers 2+ DB queries
while true; do
  curl -s -o /dev/null -X POST http://<rosetta-host>:5700/network/status \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\": $NETWORK_ID, \"metadata\": {}}" &
done
```

Running this from a single host with modest concurrency (e.g., 50 parallel `curl` processes) will saturate the database connection pool and degrade all dependent Rosetta and importer operations. No authentication, no special tooling, and no prior access to the system is required.
