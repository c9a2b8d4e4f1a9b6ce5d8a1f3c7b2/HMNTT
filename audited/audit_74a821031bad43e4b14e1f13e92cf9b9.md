### Title
Integer Narrowing Cast Overflow in `poll()` Enables Unbounded Query or Infinite Retry DoS via Crafted `limit = Long.MAX_VALUE`

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` performs an unchecked narrowing cast from `long` to `int`. When an external user supplies `limit = Long.MAX_VALUE` (permitted by the `@Min(0)`-only constraint), the cast produces `-1`, which propagates as the page size into the downstream repository query. Depending on the repository implementation, this either removes the LIMIT clause entirely (unbounded data fetch / resource exhaustion) or causes an exception that triggers the infinite retry loop, effectively halting message delivery.

### Finding Description

**Exact location:** `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `poll()`, lines 68–71.

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())   // line 69 — overflow here
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());        // line 71 — min of negative = negative
```

**Root cause — arithmetic:**
- `filter.getLimit()` = `Long.MAX_VALUE` = `0x7FFFFFFFFFFFFFFF`
- `context.getTotal().get()` = `0L` (first poll)
- Subtraction result = `Long.MAX_VALUE` (still `0x7FFFFFFFFFFFFFFF`)
- `(int) Long.MAX_VALUE` truncates to the low 32 bits: `0xFFFFFFFF` = **`-1`** (signed)
- `Math.min(-1, maxPageSize)` = **`-1`** (maxPageSize is always positive: 1000 or 5000)

**Propagation:**
```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
// newFilter.limit == -1
// newFilter.hasLimit() == (-1 > 0) == false  ← limit guard bypassed
``` [1](#0-0) 

`hasLimit()` is defined as `return limit > 0;`, so with `limit = -1` it returns `false`. [2](#0-1) 

The internally-constructed `newFilter` is built via Lombok's `toBuilder()`, which bypasses Spring Bean Validation entirely — the `@Min(0)` constraint on `limit` is **not re-evaluated** on the derived object. [3](#0-2) 

**Why existing checks fail:**
- `@Min(0)` on the original filter only enforces a floor of 0; there is no `@Max` constraint, so `Long.MAX_VALUE` is accepted.
- `hasLimit()` guards the *original* filter's limit check in `isComplete()`, but the overflow happens before that guard is reached.
- The retry operator `.retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))` means any exception thrown by the repository on a negative limit will cause indefinite retries. [4](#0-3) 

### Impact Explanation

Two concrete outcomes depending on repository behavior:

1. **Unbounded query (most likely with PostgreSQL `LIMIT -1` = no limit):** `findByFilter(newFilter)` runs without a row cap, returning the entire `topic_message` table for the given topic. This causes memory exhaustion and extreme DB load per subscription.
2. **Exception path:** If the JDBC driver rejects a negative LIMIT, an exception is thrown, caught by the retry operator, and the subscription enters an infinite backoff loop — consuming a thread from the bounded elastic scheduler indefinitely and starving legitimate subscribers.

In both cases the attacker can trigger the condition with a single gRPC `subscribeTopic` call, with no authentication required.

### Likelihood Explanation

The gRPC `ConsensusService.subscribeTopic` endpoint is publicly accessible. The `limit` field in the protobuf request maps to a `long`/`uint64`, so any client can trivially set it to `Long.MAX_VALUE` (e.g., `9223372036854775807`). No special privileges, credentials, or knowledge of internal state are required. The attack is repeatable: each new subscription with this limit value triggers the overflow independently.

### Recommendation

1. **Add an upper-bound guard before the cast** in `poll()`:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = filter.hasLimit()
           ? (remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining)
           : Integer.MAX_VALUE;
   ```
2. **Add `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) to `TopicMessageFilter.limit`** so the original filter is rejected at the validation layer before reaching the retriever.
3. **Validate the derived filter** or avoid re-using `TopicMessageFilter` for internal pagination state — use a plain `int pageSize` variable instead of embedding it back into a `TopicMessageFilter`.

### Proof of Concept

```
Precondition: Mirror node gRPC endpoint is reachable (no auth required).

1. Connect to the gRPC ConsensusService.subscribeTopic endpoint.
2. Send a SubscribeTopicRequest with:
     topicID = <any valid topic>
     limit   = 9223372036854775807  (Long.MAX_VALUE)
3. The server calls PollingTopicMessageRetriever.retrieve() → poll().
4. Inside poll():
     filter.getLimit()          = 9223372036854775807L
     context.getTotal().get()   = 0L
     subtraction                = 9223372036854775807L
     (int) cast                 = -1
     Math.min(-1, 1000)         = -1
     newFilter.limit            = -1
     newFilter.hasLimit()       = false
5. findByFilter(newFilter) executes with no effective LIMIT →
   full table scan returned, OR exception → infinite retry loop.
6. Repeat with N concurrent connections to amplify resource exhaustion.
```
