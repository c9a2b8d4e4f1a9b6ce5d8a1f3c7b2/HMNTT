### Title
Silent Pagination Truncation in `buildNetworkFromMirrorNodes` Allows Mirror-Node-Controlled Node Set Injection

### Summary
`fetchMirrorNodeNetwork` fetches exactly one page from the mirror REST API and returns immediately, never checking or following `payload.Links.Next`. An attacker who controls the mirror REST endpoint (or performs a MITM on an HTTP-configured deployment) can return a first page containing only attacker-operated consensus nodes, set `Links.Next` to the legitimate remainder, and the pinger will silently build its entire network map from the attacker's subset. All subsequent real HBAR transfers are then routed exclusively through those nodes.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go` — `nodesEnvelope` struct declares `Links.Next` at lines 19–21: [1](#0-0) 

`fetchMirrorNodeNetwork` decodes the envelope at line 96 and then iterates `payload.Nodes` at lines 102–125, but `payload.Links.Next` is **never read again after decoding**: [2](#0-1) 

`buildNetworkFromMirrorNodes` calls `fetchMirrorNodeNetwork` once per retry attempt and returns on the first success, so no outer pagination loop exists either: [3](#0-2) 

**Root cause:** The field `Links.Next` is modeled in the struct (signaling the developer was aware of pagination) but the follow-up fetch is simply never implemented. The failed assumption is that the first page always contains the complete node set.

**Exploit flow:**
1. Attacker operates or intercepts the mirror REST endpoint pointed to by `HIERO_MIRROR_PINGER_REST` / `-mirror-rest`.
2. On the initial `GET /api/v1/network/nodes` request, attacker returns a valid JSON body containing only attacker-controlled node entries, with `"links": {"next": "/api/v1/network/nodes?limit=25&node.id=gt:X"}` pointing to the legitimate nodes.
3. `fetchMirrorNodeNetwork` decodes the body, builds `network` from the attacker's nodes only, passes the `len(network) > 0` guard at line 127, and returns.
4. `newClient` calls `hiero.ClientForNetwork(netmap)` with this poisoned map: [4](#0-3) 
5. Every subsequent `cryptoTransfer.Execute(client)` in `submitWithRetry` is dispatched to attacker-controlled gRPC endpoints: [5](#0-4) 

**Why existing checks fail:** The only guard is `len(network) == 0` (line 127), which only ensures at least one node was returned — it says nothing about completeness. There is no signature, count, or cross-check against a trusted source.

### Impact Explanation
The pinger performs real HBAR transfers on a configurable interval (default every 1 second) from the operator account. With the network map fully controlled by the attacker:
- Attacker nodes can drop transactions while returning fabricated success receipts, causing the pinger to log false `transfer success` entries while funds are never settled on the real network.
- Transaction fees are still deducted from the operator account for each attempt, causing incremental fund drain.
- The pinger's health-monitoring signal becomes entirely attacker-controlled, masking real network outages or manufacturing false ones.
- All transaction IDs, amounts, and operator account details are visible to the attacker's nodes.

This path is only reachable when `network=other`, which is the custom/private-network deployment mode — precisely the environment where the mirror node is least likely to be a well-known, certificate-pinned public endpoint.

### Likelihood Explanation
The default example URL in the config comment is `http://mirror-rest:5551` (plain HTTP): [6](#0-5) 

The HTTP client uses no custom TLS configuration: [7](#0-6) 

In a Kubernetes/Docker deployment using plain HTTP (the documented default for `network=other`), any pod or process on the same network segment can intercept or spoof the mirror REST responses with no credentials required — matching the "unprivileged external user" precondition. Even over HTTPS, a compromised or malicious mirror node operator requires no special Hedera-network privileges to exploit this.

### Recommendation
Replace the single-fetch call with a pagination loop inside `fetchMirrorNodeNetwork` (or a new helper). After decoding each page, check `payload.Links.Next`; if non-nil and non-empty, construct the next URL and repeat the fetch, accumulating nodes into the same `network` map. Apply a maximum-page guard to prevent infinite loops. Example structure:

```go
currentURL := url
for {
    // ... fetch and decode into payload ...
    // accumulate payload.Nodes into network
    if payload.Links.Next == nil || *payload.Links.Next == "" {
        break
    }
    currentURL = resolveNext(base, *payload.Links.Next)
    if pageCount++; pageCount > maxPages {
        return nil, false, fmt.Errorf("mirror node pagination exceeded %d pages", maxPages)
    }
}
```

Additionally, enforce HTTPS with certificate validation for the mirror REST endpoint in production deployments, and consider a minimum-node-count threshold before accepting the network map.

### Proof of Concept
1. Stand up a mock HTTP server at `http://attacker-mirror:5551`.
2. Configure the pinger: `HIERO_MIRROR_PINGER_NETWORK=other`, `HIERO_MIRROR_PINGER_REST=http://attacker-mirror:5551`.
3. On `GET /api/v1/network/nodes`, return:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.999",
      "service_endpoints": [{"ip_address_v4": "attacker-ip", "port": 50211}]
    }
  ],
  "links": { "next": "/api/v1/network/nodes?limit=25&node.id=gt:999" }
}
```
4. On `GET /api/v1/network/nodes?limit=25&node.id=gt:999`, return the real nodes (this endpoint will never be called).
5. Start the pinger. Observe via logs that it connects only to `attacker-ip:50211` and all `cryptoTransfer.Execute` calls go to the attacker's gRPC server. The attacker's server can return any receipt status, including `SUCCESS`, while never forwarding the transaction to the real network.
