### Title
Cache TTL Asymmetry Enables Low-Rate Amplified Batch DB Query Attack via `findStorageBatch()`

### Summary
In `ContractStateServiceImpl.findStorage()`, the `contractStateCache` uses `expireAfterWrite=2s` while the per-contract slot key accumulator (`slotsPerContract`) uses `expireAfterAccess=5m`. An unprivileged attacker can pre-populate the slot key accumulator with up to 1,500 keys over time, then send a single request every 2+ seconds to guarantee a cache miss and trigger a full 1,500-slot batch DB query each time. This creates a sustained, high-amplification DB load with a request rate of 0.5 RPS — far below any throttle threshold.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` at [1](#0-0)  checks `contractStateCache` (configured `expireAfterWrite=2s` at [2](#0-1) ). On any miss it calls `findStorageBatch()`.

`findStorageBatch()` at [3](#0-2)  does the following unconditionally:
1. Retrieves the per-contract slot key cache (`slotsPerContract`, configured `expireAfterAccess=5m,maximumSize=1500` at [4](#0-3) ).
2. Adds the current key via `putIfAbsent`.
3. Reads **all** keys currently in that per-contract cache.
4. Issues `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` with the full accumulated list.

After the batch query, results are written back into `contractStateCache` at [5](#0-4) .

**Root cause — failed assumption:**

The design assumes that once slot values are batch-loaded into `contractStateCache`, subsequent requests will hit the cache and avoid re-querying. This holds only within the 2-second write TTL window. The `slotsPerContract` accumulator has a 5-minute access TTL, so slot keys persist across many `contractStateCache` expiry cycles. Every 2 seconds, the entire accumulated batch becomes stale simultaneously, and the very next request for any slot of that contract re-issues the full batch query.

**Exploit flow:**

1. **Accumulation phase (low rate):** Attacker sends requests for N distinct slot keys of a target contract (e.g., N=1,500, the `slotsPerContract` maximum). Each request adds a key to the per-contract accumulator. At 5 req/s this takes 5 minutes — well within the 5-minute access TTL that keeps keys alive.
2. **Exploitation phase (ultra-low rate):** Every 2+ seconds the attacker sends exactly 1 request for any slot of that contract. The `contractStateCache` has expired, so `findStorageBatch()` is called with all 1,500 accumulated keys, issuing a `SELECT … WHERE slot IN (…1500 values…)` query.
3. **Sustain:** The attacker sends 1 request every 2 seconds indefinitely. Each request resets the `expireAfterAccess` TTL on the slot keys, keeping the accumulator full. The DB is hit with a 1,500-slot batch query every 2 seconds from a single attacker at 0.5 RPS.

**Why existing checks fail:**

The global rate limiter (`requestsPerSecond=500`) at [6](#0-5)  and gas throttle (`gasPerSecond`) are global counters. The attacker's 0.5 RPS is orders of magnitude below the 500 RPS ceiling. There is no per-contract, per-user, or per-batch-size throttle. The `slotsPerContract` max size of 1,500 is a capacity bound, not a security control — it merely caps the amplification factor at 1,500×.

### Impact Explanation

Each attacker request at 0.5 RPS causes a DB query scanning 1,500 contract storage slots. An attacker targeting 10 contracts simultaneously causes 15,000 slot lookups per second from 5 total requests per second. The `contract_state` table query `SELECT slot, value FROM contract_state WHERE contract_id = :contractId AND slot IN (:slots)` with 1,500 bind parameters is a non-trivial indexed scan. Sustained over time this can saturate DB connection pool resources, increase query latency for all users, and push node resource consumption well above the 30% threshold without any brute-force request volume. The `db.statementTimeout=3000ms` at [7](#0-6)  provides a backstop but does not prevent the amplification — it only caps individual query duration.

### Likelihood Explanation

The attack requires no authentication, no special privileges, and no high request rate. The accumulation phase is indistinguishable from normal usage (legitimate EVM contract calls read many storage slots). The exploitation phase requires only 1 request every 2 seconds. Any public Ethereum-compatible JSON-RPC client can execute this. The attacker needs only knowledge of a contract address and the ability to craft `eth_call` requests that read different storage slots — both trivially obtained from on-chain data or block explorers.

### Recommendation

1. **Decouple batch size from TTL expiry:** When `contractStateCache` entries expire, do not re-query the entire accumulated slot list. Instead, query only the single requested slot (fall back to `contractStateRepository.findStorage()`) and lazily re-populate the batch cache on subsequent misses.
2. **Bound the batch query size:** Cap `cachedSlots` passed to `findStorageBatch()` to a configurable maximum (e.g., 50–100) per invocation, regardless of how many keys are in the accumulator.
3. **Increase `contractStateCache` TTL or use `expireAfterAccess`:** A 2-second write TTL guarantees periodic full-batch re-queries. Switching to `expireAfterAccess` or increasing the TTL to 30–60 seconds would dramatically reduce re-query frequency.
4. **Add per-contract batch query rate limiting:** Track the last batch query timestamp per contract and skip the batch re-query if one occurred within a configurable window.

```
# (spread over ~5 minutes at low rate to avoid gas throttle)
for i in $(seq 1 1500); do
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0xTARGET\",\"data\":\"0x$(printf '%064x' $i)\",\"gas\":50000}"
  sleep 0.2
done

# Step 2: Every 2.1 seconds, send 1 request to trigger a 1500-slot batch DB query
while true; do
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0xTARGET","data":"0x0000...0001","gas":50000}'
  sleep 2.1
done

# Observable effect: DB query log shows repeated
# "SELECT slot, value FROM contract_state WHERE contract_id=X AND slot IN (...1500 values...)"
# every ~2 seconds, triggered by a single attacker at 0.5 RPS.
```

```java
        if (!cacheProperties.isEnableBatchContractSlotCaching()) {
```

```java
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
