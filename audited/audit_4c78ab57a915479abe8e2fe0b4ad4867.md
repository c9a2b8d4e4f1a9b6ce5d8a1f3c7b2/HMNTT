### Title
Cache Eviction Race in `findStorageBatch()` Enables Unauthenticated DB Amplification DoS

### Summary
`ContractStateServiceImpl.findStorageBatch()` inserts a new slot key via `putIfAbsent`, then immediately reads `asMap().keySet()` to verify the key is still present. Caffeine's Window TinyLFU policy can evict the newly inserted key synchronously during the same `putIfAbsent` maintenance run when the per-contract slot cache is full of high-frequency entries. When the key is absent from the snapshot, `isKeyEvictedFromCache` remains `true`, triggering both a full batch query (up to 1500 keys in an `IN` clause) and a separate individual fallback `findStorage` DB query — every single time. An unauthenticated attacker can sustain this condition indefinitely, causing amplified DB load at the global rate limit of 500 RPS.

### Finding Description

**Exact code path:**
`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 85–122.

**Cache architecture:**
- `contractSlotsCache` (`CACHE_MANAGER_CONTRACT_SLOTS`, `maximumSize=3000`): outer cache mapping `contractId → CaffeineCache` (per-contract slot key cache).
- Per-contract slot key cache (`CACHE_MANAGER_SLOTS_PER_CONTRACT`, `maximumSize=1500`): maps `ByteBuffer(slotKey) → EMPTY_VALUE`.
- `contractStateCache` (`CACHE_MANAGER_CONTRACT_STATE`, `expireAfterWrite=2s, maximumSize=25000`): maps `(contractId, slotKey) → value`.

**Root cause — unsafe post-insert presence check:**

```java
// line 90: insert new key
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
// line 91: immediately snapshot the cache
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

boolean isKeyEvictedFromCache = true;   // line 94 — starts true

for (var slot : cachedSlotKeys) {
    if (wrappedKey.equals(slot)) {
        isKeyEvictedFromCache = false;  // only cleared if key survives in snapshot
    }
}
// line 118-119: fallback individual DB query if key not found
if (isKeyEvictedFromCache) {
    return contractStateRepository.findStorage(contractId.getId(), key);
}
```

Caffeine's Window TinyLFU runs its maintenance task **synchronously on the calling thread** when the write buffer fills. During maintenance, a brand-new key (frequency = 0 in the frequency sketch) is the weakest candidate and is evicted immediately if the cache is at capacity with high-frequency incumbents. After `putIfAbsent` returns, the key is already gone. `asMap().keySet()` does not contain it, `isKeyEvictedFromCache` stays `true`, and the fallback fires.

**Two DB queries are issued per affected request:**
1. `findStorageBatch(contractId, cachedSlots)` — a `SELECT ... WHERE slot IN (...)` with up to 1500 keys (line 103).
2. `findStorage(contractId, key)` — individual point query (line 119), annotated `@Cacheable` with `expireAfterWrite=2s`, so it re-hits the DB every 2 seconds per unique key.

**Why existing checks fail:**
- The `contractStateCache` `@Cacheable` on `findStorage` only caches the result for 2 seconds. An attacker cycling through unique keys faster than the TTL bypasses it entirely.
- The global rate limiter (500 RPS, `ThrottleProperties.requestsPerSecond`) is a single shared bucket with no per-IP or per-contractId partitioning. It limits request count, not DB query amplification.
- `enableBatchContractSlotCaching` defaults to `true` and there is no mechanism to detect or suppress the eviction-triggered fallback path.

### Impact Explanation

At 500 RPS (the global cap), each request in the attack state issues:
- 1 batch query with up to 1500 slot keys in an `IN` clause.
- 1 individual fallback query.

This yields up to **750,500 DB key lookups per second** from a single attacker operating at the rate limit. The `contract_state` table is queried with large `IN` predicates continuously, exhausting DB connection pool capacity (monitored alert threshold: 75% utilization) and query execution threads. The web3 service becomes unable to serve legitimate contract call requests, constituting a total service outage for the JSON-RPC endpoint. No authentication or special privilege is required.

### Likelihood Explanation

The web3 JSON-RPC endpoint (`eth_call`) is a public, unauthenticated API. Any external user can:
1. Enumerate or generate arbitrary 32-byte slot keys (EVM storage slots are public knowledge).
2. Pre-warm the per-contract cache with 1500 repeated accesses to build TinyLFU frequency scores.
3. Continuously submit requests with fresh unique slot keys.

The attack requires no exploit chain, no credentials, and no special network position. It is fully repeatable and automatable with a simple script. The 500 RPS global limit is the only barrier, and it is shared across all users — meaning the attacker consumes the entire quota while amplifying DB load by ~1500×.

### Recommendation

1. **Remove the post-insert eviction check entirely.** The `isKeyEvictedFromCache` pattern is fundamentally unsafe with any evicting cache. Instead, after `putIfAbsent`, always include the key in the batch query list regardless of whether it appears in the snapshot. If the batch result does not contain the key, return `Optional.empty()` (or query individually only when the batch result is empty for that key).

2. **Bound the batch query size.** Cap `cachedSlots` to a configurable maximum (e.g., 100) before issuing `findStorageBatch`, preventing large `IN` clause amplification.

3. **Add per-contractId rate limiting** in addition to the global bucket, so a single contract cannot monopolize the DB query budget.

4. **Increase `contractState` TTL** or use `expireAfterAccess` instead of `expireAfterWrite` to reduce repeated DB hits for the same keys.

while True:
    POST /api/v1/contracts/call
