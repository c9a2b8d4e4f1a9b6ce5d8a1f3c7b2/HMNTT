### Title
Unbounded Subscription Limit Enables Indefinite Connection Hold and Server Resource Exhaustion

### Summary
`TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` when a limit is set, but `TopicMessageFilter` places no upper bound on the `limit` field — only `@Min(0)`. An unprivileged caller can supply `limit = Long.MAX_VALUE` with no `endTime`, causing the Flux to never terminate by count or by time, holding the gRPC connection and all associated server resources open indefinitely. Repeating this across many connections constitutes a practical denial-of-service.

### Finding Description
**Code path:**

- `TopicMessageFilter.java` lines 25–26: `@Min(0) private long limit;` — no `@Max` constraint exists.
- `TopicMessageFilter.java` lines 39–41: `hasLimit()` returns `true` for any `limit > 0`, including `Long.MAX_VALUE`.
- `TopicMessageServiceImpl.java` lines 83–85:
  ```java
  if (filter.hasLimit()) {
      flux = flux.take(filter.getLimit()); // take(Long.MAX_VALUE) never fires
  }
  ```
- `TopicMessageServiceImpl.java` lines 79–81: the `takeWhile` on `endTime` is skipped entirely when `endTime == null`.
- `TopicMessageServiceImpl.java` lines 123–131: `pastEndTime()` returns `Flux.never()` when `endTime == null`, so the `takeUntilOther` terminator never fires.
- `TopicMessageServiceImpl.java` lines 203–206: `isComplete()` unconditionally returns `false` when `endTime == null`.

**Root cause:** The failed assumption is that callers will supply a reasonable limit. There is no server-enforced maximum on `limit`, no per-connection wall-clock timeout in the reactive pipeline, and no enforcement of the `subscriberCount` gauge as an admission-control gate (lines 48, 89–90 — it is metrics-only).

**Exploit flow:**
1. Attacker constructs a `ConsensusSubscribeRequest` (standard gRPC, no auth required) targeting any valid topic, with `limit = Long.MAX_VALUE` and no `consensusEndTime`.
2. Server builds a Flux that: (a) never terminates via `take` (Long.MAX_VALUE messages will never arrive), (b) never terminates via `endTime` checks, (c) keeps the live listener active indefinitely via `incomingMessages()`.
3. Each such connection increments `subscriberCount` and holds a gRPC server stream, a Reactor subscription chain, and a database polling path open forever.
4. Attacker repeats with many concurrent connections to exhaust thread pools, connection pools, or file descriptors.

### Impact Explanation
Each zombie subscription consumes a gRPC server stream slot, a Reactor scheduler thread (bounded elastic pool), and periodic database polling resources. With enough concurrent connections the server becomes unresponsive to legitimate subscribers. Because the mirror node is a public, unauthenticated gRPC endpoint, this is a straightforward remote DoS against the entire topic subscription service.

### Likelihood Explanation
No authentication or rate-limiting is required. The gRPC `ConsensusService/subscribeTopic` endpoint is publicly reachable. The attacker needs only a standard gRPC client and knowledge of one valid topic ID (trivially obtained from public ledger data). The attack is fully repeatable and scriptable.

### Recommendation
1. Add a server-side maximum limit in `TopicMessageFilter`: `@Max(GrpcProperties.MAX_LIMIT) private long limit;` and enforce a sensible cap (e.g., 10,000 or configurable via `GrpcProperties`).
2. Add a mandatory wall-clock timeout to every subscription Flux regardless of `endTime`, e.g., `flux.timeout(grpcProperties.getMaxSubscriptionDuration())`.
3. Enforce `subscriberCount` as an admission-control gate: reject new subscriptions when the count exceeds a configured maximum.
4. Consider requiring a non-null `endTime` for subscriptions that specify a `limit`, or defaulting `endTime` to `now + maxDuration` when absent.

### Proof of Concept
```python
import grpc
from proto import consensus_service_pb2_grpc, mirror_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

request = mirror_pb2.ConsensusTopicQuery(
    topic_ID=<any_valid_topic>,
    limit=9223372036854775807,  # Long.MAX_VALUE, no consensus_end_time set
)

# Open N concurrent streams; each holds server resources indefinitely
responses = stub.subscribeTopic(request)
for _ in responses:
    pass  # blocks forever, server-side Flux never completes
```
Repeat in parallel threads/processes to exhaust server resources.
