### Title
Unauthenticated DB Connection Pool Exhaustion via Alias Lookup on `/accounts/:id/rewards`

### Summary
The `listStakingRewardsByAccountId` handler in `rest/controllers/accountController.js` performs an uncached, unauthenticated database query for every request containing an account alias. With no rate limiting on the Node.js REST API and a default pool of only 10 DB connections, a high-concurrency flood of requests with distinct non-existent aliases can saturate the shared connection pool, causing all REST API endpoints on the affected mirror node instance to queue or timeout.

### Finding Description
**Exact code path:**

`GET /api/v1/accounts/{alias}/rewards` → `listStakingRewardsByAccountId` (line 170) → `EntityService.getEncodedId(alias)` (line 171) → `AccountAlias.isValid()` returns true → `getAccountIdFromAlias(alias, requireResult=true)` (line 126) → `getAccountFromAlias(alias)` (line 72) → `super.getRows(entityFromAliasQuery, [alias])` (line 43) — **DB connection acquired and query executed** → empty result → returns `null` → `NotFoundError` thrown (line 75).

**Root cause:** No caching of alias-to-ID resolution in the Node.js `EntityService` (confirmed: no cache object in `rest/service/entityService.js`), combined with no rate limiting middleware in `rest/server.js`, and a shared DB pool with a default of 10 connections (`rest/dbpool.js` line 14: `max: config.db.pool.maxConnections`).

**Why checks fail:** The `entityId.js` LRU cache only caches parsed `shard.realm.num` / encoded-ID strings — not alias-to-entity-ID resolution results. Negative results (non-existent aliases) are never cached. The `authHandler` middleware only enforces response `limit` values for authenticated users; it does not throttle request rates. ### Impact Explanation
Exhausting the 10-connection pool causes all REST API endpoints sharing that pool (accounts, transactions, tokens, balances, etc.) to queue requests up to `connectionTimeout` (20 seconds default) before failing. This degrades the entire mirror node REST API instance, not just the rewards endpoint. Since the pool is global (`global.pool` in `dbpool.js`), there is no isolation between endpoints. The impact is a full denial-of-service of the mirror node REST API for the duration of the attack.

### Likelihood Explanation
No authentication or API key is required. The attack requires only HTTP GET requests to a public endpoint with valid-format but non-existent aliases (e.g., valid base32 strings). Distinct aliases prevent any future caching benefit. A single attacker with modest bandwidth can sustain the attack indefinitely. The attack is trivially scriptable and repeatable.

### Recommendation
1. **Add rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit`) applied globally or specifically to alias-resolving endpoints.
2. **Cache negative alias lookup results** in `EntityService.getAccountFromAlias` with a short TTL (e.g., 30–60 seconds) to prevent repeated DB queries for the same non-existent alias.
3. **Increase the default DB connection pool size** or add a request concurrency limiter to bound the number of in-flight DB queries.
4. **Add a fast-path rejection** before the DB query: validate alias format strictly and reject structurally invalid aliases before touching the DB.

                                 from ${Entity.tableName}
                                 where coalesce(${Entity.DELETED}, false) <> true
                                   and ${Entity.ALIAS} = $1`;

  static entityFromEvmAddressQuery = `select ${Entity.ID}
                                      from ${Entity.tableName}
                                      where ${Entity.DELETED} <> true
                                        and ${Entity.EVM_ADDRESS} = $1`;

  // use a small column in existence check to reduce return payload size
  static entityExistenceQuery = `select ${Entity.TYPE}
                                 from ${Entity.tableName}
                                 where ${Entity.ID} = $1`;

  static missingAccountAlias = 'No account with a matching alias found';
  static multipleAliasMatch = `Multiple alive entities matching alias`;
  static multipleEvmAddressMatch = `Multiple alive entities matching evm address`;

  /**
   * Retrieves the entity containing matching the given alias
   *
   * @param {AccountAlias} accountAlias accountAlias
   * @return {Promise<Entity>} raw entity object
   */
  async getAccountFromAlias(accountAlias) {
    const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);

    if (isEmpty(rows)) {
      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching alias ${accountAlias}`);
      throw new Error(EntityService.multipleAliasMatch);
    }

    return new Entity(rows[0]);
  }

        throw InvalidArgumentError.forParams(paramName);
      }
      // rethrow
      throw ex;
