### Title
Plaintext Database Connections Due to Explicitly Disabled PostgreSQL SSL and Absent HikariCP SSL Enforcement

### Summary
PostgreSQL SSL is explicitly disabled at the cluster level in the StackGres deployment manifest, and the `HikariConfig` bean in `CommonConfiguration.java` applies no SSL parameters. With Kubernetes network policies also disabled, any attacker with intra-cluster network access can passively intercept or actively tamper with all database traffic in plaintext, including transaction and signer-related query results.

### Finding Description
**Code locations:**

1. `charts/hedera-mirror/templates/stackgres/stackgres-cluster.yaml`, lines 61–62 — SSL is **explicitly disabled** at the PostgreSQL server level: [1](#0-0) 

```yaml
ssl:
  enabled: false  # Disable SSL to work around issue 9143
```

This is not a missing configuration — it is an active opt-out. The server will not accept TLS connections regardless of what the client requests.

2. `common/src/main/java/org/hiero/mirror/common/CommonConfiguration.java`, lines 62–64 and 88–94 — `hikariConfig()` returns a bare `new HikariConfig()` with no SSL properties, and `dataSource()` copies that state verbatim: [2](#0-1) [3](#0-2) 

No `sslMode`, `ssl=true`, or equivalent JDBC parameter is set anywhere in the chain.

3. `common/src/main/java/org/hiero/mirror/common/util/DatabaseWaiter.java`, lines 26–43 — the startup probe connection also carries no SSL properties: [4](#0-3) 

4. `charts/hedera-mirror-common/values.yaml`, lines 217–218 — Kubernetes `NetworkPolicy` is disabled cluster-wide: [5](#0-4) 

```yaml
networkPolicy:
  enabled: false
```

**Root cause:** The PostgreSQL server actively rejects TLS (server-side `ssl.enabled: false`), the JDBC connection pool never requests it, and no network-layer segmentation compensates. All database traffic flows in plaintext over the pod network.

**Exploit flow:**
- Attacker gains intra-cluster network access (compromised sidecar, vulnerable co-tenant pod, supply-chain attack on a container image, or misconfigured RBAC allowing pod creation).
- With `NetworkPolicy` disabled, the attacker's pod can reach the database pod's port directly.
- Passive sniffing (e.g., `tcpdump` on the pod's veth interface or via a CNI-level tap) captures all PostgreSQL wire-protocol frames in plaintext.
- Active ARP/DNS spoofing within the flat cluster network allows MITM: the attacker relays and modifies query responses (e.g., returning forged rows for signer/approval queries) without either endpoint detecting the tampering.

### Impact Explanation
All data exchanged between the mirror node services and PostgreSQL — including transaction records, signer approval state, credentials transmitted during connection setup, and query results — is exposed in plaintext. An attacker performing MITM can forge query responses, causing the mirror node to act on falsified transaction or signer data. Severity is **High**: confidentiality and integrity of the entire database layer are compromised for any attacker with pod-level cluster access.

### Likelihood Explanation
The precondition is intra-cluster network access, not external internet access. This is achievable via: a compromised container image in the supply chain, exploitation of any other service running in the same namespace, or misconfigured RBAC granting pod-exec or pod-create rights. With `NetworkPolicy` disabled, lateral movement to the database network segment requires no additional privilege escalation. The SSL opt-out is permanent and unconditional (`# Disable SSL to work around issue 9143`), so no configuration path re-enables it without a manifest change.

### Recommendation
1. **Re-enable PostgreSQL SSL** in `charts/hedera-mirror/templates/stackgres/stackgres-cluster.yaml`: remove or set `ssl.enabled: true` and resolve the underlying issue 9143 through a proper fix rather than disabling TLS.
2. **Enforce SSL on the client side** in `CommonConfiguration.java` by adding `sslMode=verify-full` (or at minimum `require`) to the JDBC URL or as a HikariCP data-source property, so the pool refuses to connect to a non-TLS endpoint.
3. **Enable Kubernetes NetworkPolicy** (`networkPolicy.enabled: true`) to restrict which pods can reach the database port, providing defense-in-depth even if TLS is temporarily unavailable.
4. **Add SSL properties to `DatabaseWaiter`** probe connections to match the pool's security posture.

### Proof of Concept
**Preconditions:** Attacker has exec access to any pod in the same Kubernetes namespace (e.g., `kubectl exec` via leaked credentials or a compromised CI runner).

```bash
# 1. From attacker pod, identify the database pod IP
kubectl get pods -o wide | grep stackgres

# 2. Passive capture of plaintext PostgreSQL wire traffic
# (no privileges beyond pod exec needed; NetworkPolicy is disabled)
tcpdump -i eth0 -A 'tcp port 5432'
# Output: plaintext SQL queries and result rows visible immediately

# 3. Active MITM (ARP spoofing within flat pod network)
arpspoof -i eth0 -t <app-pod-ip> <db-pod-ip> &
arpspoof -i eth0 -t <db-pod-ip> <app-pod-ip> &
# Forward and selectively modify PostgreSQL responses with a tool
# such as mitmproxy with a PostgreSQL protocol plugin,
# returning forged rows for signer/approval queries.
```

The absence of TLS means the PostgreSQL wire protocol provides no integrity protection; forged server responses are indistinguishable from legitimate ones.
