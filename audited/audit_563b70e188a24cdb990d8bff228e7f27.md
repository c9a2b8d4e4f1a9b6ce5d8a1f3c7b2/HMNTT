### Title
Unbounded `getTransactionHash()` Await Exhausts DB Connection Pool During Network Partition via Unprivileged `transaction_hash` Filter

### Summary
Any unauthenticated caller of `GET /api/v1/contracts/results/logs?transaction_hash=<valid_hex>` causes `extractContractLogsMultiUnionQuery` to unconditionally `await getTransactionHash()`, which issues a PostgreSQL query through `pool.queryQuietly`. The pool is configured with only a server-side `statement_timeout`; no client-side query timeout exists. During a network partition the server's timeout error is never delivered to the Node.js process, so the Promise never settles, the async handler stalls indefinitely, and the borrowed pool connection is never released. Ten concurrent such requests (the default pool maximum) permanently exhaust the connection pool for all other API consumers.

### Finding Description

**Code path:**

`getContractLogs` / `getContractLogsById` → `extractContractLogsMultiUnionQuery` (line 582) → unconditional `await getTransactionHash(...)` at line 671 → `pool.queryQuietly` (line 35 of `transactionHash.js`) → `this.query(query, params)` (line 1520 of `utils.js`).

**Root cause – no client-side query timeout:**

`rest/dbpool.js` configures the `pg.Pool` with:
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // 20 000 ms – new-connection only
max: config.db.pool.maxConnections,                         // 10
statement_timeout: config.db.pool.statementTimeout,         // 20 000 ms – server-side only
```
`statement_timeout` is a PostgreSQL session-level parameter. It is enforced by the database server and communicated back to the client as an error message over the TCP connection. During a network partition the TCP socket remains in ESTABLISHED state (no RST/FIN), the server's error packet is never delivered, and the `pg` library has no independent client-side deadline. The `pool.queryQuietly` call therefore suspends on `await this.query(...)` forever.

**Why the existing checks are insufficient:**

| Mechanism | Scope | Effective during partition? |
|---|---|---|
| `statement_timeout = 20 000 ms` | Server-side | No – response never arrives |
| `connectionTimeoutMillis = 20 000 ms` | Acquiring a *new* connection | No – connection already held |
| `query_timeout` (pg client-side) | Not configured | N/A |
| Express request timeout middleware | Not present in `server.js` | N/A |

**Trigger condition:**

The `transaction_hash` branch at line 669 is reached whenever the parsed filter array contains a `TRANSACTION_HASH` key. Input validation only checks format (64-char base64url or 96-char hex); any syntactically valid hash reaches `getTransactionHash`.

### Impact Explanation

With the default pool of 10 connections, 10 concurrent requests with valid-format `transaction_hash` values permanently occupy every connection. Subsequent requests from all users queue for a connection; after `connectionTimeoutMillis` (20 s) they receive a pool-acquisition error. The REST API is effectively offline for the duration of the partition. Because Node.js is single-threaded and the event loop is not blocked (the awaits are async), the attacker can sustain the attack with a trivially small number of keep-alive HTTP connections. Memory also grows linearly with pending requests because each stalled handler retains its closure state.

### Likelihood Explanation

No authentication or special privilege is required. The endpoint is publicly reachable. A network partition between the REST service and its PostgreSQL replica is a realistic operational event (cloud AZ failure, maintenance, misconfiguration). An attacker who monitors public infrastructure status pages or who can induce a partition (e.g., via BGP manipulation or a targeted volumetric attack against the DB host) can time the exploit precisely. Even without deliberate timing, any accidental partition immediately becomes a DoS amplifier for any user who happens to send a `transaction_hash` query.

### Recommendation

1. **Add a client-side `query_timeout`** to the `pg.Pool` configuration in `rest/dbpool.js`:
   ```js
   query_timeout: config.db.pool.statementTimeout, // same value, enforced client-side
   ```
   This causes `node-postgres` to reject the Promise after the deadline regardless of network state.

2. **Add a per-request HTTP timeout** in `rest/server.js` (e.g., via `connect-timeout` or a custom middleware) so that any handler that does not complete within a bounded wall-clock time has its response forcibly terminated.

3. **Wrap the `getTransactionHash` call** with an explicit `Promise.race` against a timeout sentinel as a defence-in-depth measure at the call site (line 671 of `contractController.js`).

### Proof of Concept

**Preconditions:** Network partition between the REST service host and the PostgreSQL host (e.g., `iptables -I OUTPUT -d <db_host> -p tcp --dport 5432 -j DROP` on the REST host, or a firewall rule that silently drops packets without RST).

**Steps:**
```bash
# 1. Induce silent packet drop toward DB (simulates partition)
sudo iptables -I OUTPUT -d <db_host> -p tcp --dport 5432 -j DROP

# 2. Send 10 concurrent requests with a syntactically valid transaction_hash
for i in $(seq 1 10); do
  curl -s "http://<mirror-rest>/api/v1/contracts/results/logs?\
transaction_hash=0x$(python3 -c 'import secrets; print(secrets.token_hex(48))')" &
done
wait

# 3. Immediately send a normal request (no transaction_hash)
curl -v "http://<mirror-rest>/api/v1/contracts/results/logs"
# Expected: hangs for 20 s then returns a 500 pool-acquisition error
# (or hangs indefinitely if connectionTimeoutMillis is also not enforced)
```

**Result:** Step 3 demonstrates that the entire connection pool is exhausted and the API is unavailable to all users for the duration of the partition.
