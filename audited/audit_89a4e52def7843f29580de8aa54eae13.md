### Title
Silent Stale Balance Data Returned as HTTP 200 During Read Replica Lag / Network Partition

### Summary
`getBalances()` in `rest/balances.js` queries the read replica via `pool.queryQuietly()` with no check of replica lag, wall-clock freshness, or partition state. The response `timestamp` field is derived directly from the stale `consensus_timestamp` stored in the lagging replica, not from wall-clock time. Any unauthenticated caller receives HTTP 200 with silently stale balance data and no signal that the data is degraded.

### Finding Description

**Code path:**

`getBalances()` (`rest/balances.js` lines 83–156) unconditionally calls:

```js
const result = await pool.queryQuietly(pgSqlQuery, sqlParams);          // line 153
res.locals[constants.responseDataLabel] = formatBalancesResult(req, result, limit, order);  // line 154
```

`pool` is the read-replica pool configured in `rest/dbpool.js` lines 7–16. There is no call to `pg_last_xact_replay_timestamp()`, `pg_is_in_recovery()`, or any lag metric before issuing the query.

`formatBalancesResult()` (`rest/balances.js` lines 16–56) sets the response `timestamp` as:

```js
const maxTimestamp = rows
  .map((r) => r.consensus_timestamp)
  .reduce((result, current) => (result > current ? result : current), 0);
ret.timestamp = utils.nsToSecNs(maxTimestamp);   // line 30
```

This value is whatever `consensus_timestamp` the lagging replica holds — it is never compared against wall-clock time.

**Root cause / failed assumption:** The code assumes the read replica is always sufficiently current. No middleware, no request handler, and no response handler (`rest/middleware/`) sets a `Warning`, `Age`, or any degraded-state header. A grep for `pg_last_xact_replay_timestamp`, `pg_is_in_recovery`, and `replication_lag` across all `rest/**/*.js` returns zero matches.

**Why existing checks are insufficient:**

- `checkBalanceFreshness` in `rest/monitoring/balance_tests.js` (lines 110–117) is an *external* monitoring probe that compares the returned timestamp against wall-clock time. It has no effect on what the API returns to real callers.
- `getOptimizedTimestampRange()` (`rest/balances.js` lines 229–276) calls `utils.nowInNs()` only to optimise partition pruning when a `timestamp` query parameter is supplied. It is not invoked in the default (no-timestamp) path and does not gate the response.
- The OpenAPI spec (`rest/api/v1/openapi.yml` lines 394–400) states "The latest balance information is returned when there is no timestamp query parameter" — a guarantee the API cannot honour during replica lag.

**Exploit flow:**

1. Read replica falls behind primary (network partition, replication delay, or slow WAL replay).
2. Attacker (or any consumer) issues `GET /api/v1/balances` — no authentication required.
3. `getBalances()` queries the lagging replica; rows contain `consensus_timestamp` values that are N seconds/minutes old.
4. `formatBalancesResult()` sets `ret.timestamp` to the stale snapshot time.
5. Response: HTTP 200, `{"timestamp":"<stale_time>","balances":[...],"links":{...}}`.
6. No `Warning`, no `X-Data-Age`, no `503`, no `Retry-After`. The caller has no protocol-level signal that the data is stale.

### Impact Explanation

Any downstream system that uses `/api/v1/balances` for security-sensitive decisions (payment verification, DeFi collateral checks, fraud detection) will silently act on outdated balances. Because Hedera balance snapshots are taken every ~15 minutes in normal operation, a consumer cannot distinguish "last snapshot was 14 minutes ago (normal)" from "replica is 3 hours behind (partition)" — both produce the same HTTP 200 with an old `timestamp`. This enables double-spend-style race conditions and incorrect payment confirmations without any attacker privilege.

### Likelihood Explanation

No attacker capability is required to trigger the condition — the attacker only needs to query the public API while a partition or replication lag event is in progress. Replication lag events are common in cloud-hosted PostgreSQL (maintenance windows, network blips, high write load). The attack is repeatable, requires zero authentication, and is invisible to the caller.

### Recommendation

1. **Replica lag gate:** Before executing the balance query, run `SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp()))` on the read replica. If the lag exceeds a configured threshold (e.g., 30 s), return HTTP 503 with a `Retry-After` header.
2. **Response header:** Unconditionally emit an `X-Data-Timestamp` or `Age` header so callers can independently detect staleness.
3. **Staleness guard in `formatBalancesResult`:** Compare `maxTimestamp` against `utils.nowInNs()`. If the delta exceeds the configured freshness threshold, set a `Warning: 110` header or return a degraded status.
4. **OpenAPI contract:** Document the maximum staleness guarantee and the conditions under which a 503 is returned.

```javascript
const getOptimizedTimestampRange = (tsQuery, tsParams) => {
  let lowerBound = 0n;
  const neParams = [];
  let upperBound = constants.MAX_LONG;

  // Find the lower bound and the upper bound from tsParams if present
  tsQuery
    .split('?')
    .filter(Boolean)
    .forEach((query, index) => {
      const value = BigInt(tsParams[index]);
      // eq operator has already been converted to the lte operator
      if (query.includes(utils.opsMap.lte)) {
        // lte operator includes the lt operator, so this clause must before the lt clause
        upperBound = utils.bigIntMin(upperBound, value);
      } else if (query.includes(utils.opsMap.lt)) {
        // Convert lt to lte to simplify query
        const ltValue = value - 1n;
        upperBound = utils.bigIntMin(upperBound, ltValue);
      } else if (query.includes(utils.opsMap.gte)) {
        // gte operator includes the gt operator, so this clause must come before the gt clause
        lowerBound = utils.bigIntMax(lowerBound, value);
      } else if (query.includes(utils.opsMap.gt)) {
        // Convert gt to gte to simplify query
        const gtValue = value + 1n;
        lowerBound = utils.bigIntMax(lowerBound, gtValue);
      } else if (query.includes(utils.opsMap.ne)) {
        neParams.push(value);
      }
    });

  if (lowerBound > upperBound) {
    return {};
  }

  // The optimized range of [lower, upper] should overlap with at most two months, with the exception that when upper
  // is more than 1 month in the future, the range may cover more months. Since the partition maintenance job will
  // create at most one monthly partition ahead, it's unnecessary to adjust the upper bound.
  // With the assumption that the data in db is in sync with the network, in other words, the balance information is
  // update-to-date as of NOW in wall clock, the algorithm below sets lower bound to
  //   max(lowerBound from user, first day of the month before the month min(now, upperBound) is in)
  const nowInNs = utils.nowInNs();
  const effectiveUpperBound = utils.bigIntMin(upperBound, nowInNs);
  const optimalLowerBound = utils.getFirstDayOfMonth(effectiveUpperBound, -1);
  lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);

  return {lowerBound, upperBound, neParams};
};
```
