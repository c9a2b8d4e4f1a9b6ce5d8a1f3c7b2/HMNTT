### Title
`buildOpcodesResponse()` Uses Current State Instead of Historical Timestamp, Returning Incorrect `contractId` for Deleted or Address-Reused Contracts

### Summary
In `OpcodeServiceImpl.buildOpcodesResponse()`, the recipient entity is resolved via `commonEntityAccessor.get(recipientAddress, Optional.empty())`, which queries the **current (latest) database state** rather than the state at the original transaction's `consensusTimestamp`. Any unprivileged user can supply a transaction hash for a transaction whose recipient contract was subsequently deleted or whose EVM address was reused by a different contract, causing the response to return a `contractId` that is either `null` (deleted) or belongs to a completely different contract than the one actually executed.

### Finding Description

**Exact code path:**

`OpcodeServiceImpl.buildOpcodesResponse()` at lines 120–133 (`web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`):

```java
private OpcodesResponse buildOpcodesResponse(@NonNull OpcodesProcessingResult result) {
    final var recipientAddress = result.recipient();
    Entity recipientEntity = null;
    if (recipientAddress != null && !recipientAddress.equals(EMPTY_ADDRESS)) {
        recipientEntity =
                commonEntityAccessor.get(recipientAddress, Optional.empty()).orElse(null);  // ← BUG
    }
    ...
    contractId = recipientEntity.toEntityId().toString();
    ...
}
```

**Root cause:** `buildOpcodesResponse()` receives only an `OpcodesProcessingResult` (record fields: `transactionProcessingResult`, `recipient`, `opcodes`) — the `consensusTimestamp` is not included in this record and is not passed to the method. As a result, `Optional.empty()` is hardcoded, which causes `CommonEntityAccessor.get()` to branch into the current-state path:

- Long-zero address → `entityRepository.findByIdAndDeletedIsFalse(entityId)` (line 77 of `CommonEntityAccessor.java`)
- EVM address → `entityRepository.findByEvmAddressAndDeletedIsFalse(addressBytes)` (line 83 of `CommonEntityAccessor.java`)

Both queries exclude deleted entities and reflect the present database state, not the state at `consensusTimestamp`.

**Contrast with correct historical path:** When a timestamp is provided, the same accessor calls `entityRepository.findActiveByIdAndTimestamp(entityId, t)` or `findActiveByEvmAddressAndTimestamp(addressBytes, t)`, which correctly scope the lookup to the historical state.

**The `consensusTimestamp` is available** in `processOpcodeCall()` (line 64–74) via `params.getConsensusTimestamp()` (set at line 169 of `buildCallServiceParameters()`), but it is never threaded into `buildOpcodesResponse()`.

**Exploit flow:**
1. Contract C is deployed at EVM address X (entity `0.0.1234`) at block B1.
2. A transaction T executes against C at block B2 (B2 > B1).
3. C is deleted after B2.
4. Attacker (no privileges required) calls `GET /api/v1/contracts/results/{hash_of_T}/opcodes`.
5. `buildOpcodesResponse()` calls `commonEntityAccessor.get(X, Optional.empty())`.
6. `findByIdAndDeletedIsFalse(1234)` returns empty → `contractId = null`, `address = 0x000...000`.
7. **Alternatively**, if a new contract D (`0.0.5678`) is deployed at the same address X after deletion, `findByEvmAddressAndDeletedIsFalse(X)` returns D → `contractId = "0.0.5678"` (wrong contract attributed).

**Existing checks reviewed:** There are no guards in `buildOpcodesResponse()` that compare the current entity against the historical record. The `ContractDebugParameters` correctly carries the historical `block` and `consensusTimestamp` for EVM replay, but this context is discarded before `buildOpcodesResponse()` is called. The `OpcodesProcessingResult` record itself carries no timestamp field.

### Impact Explanation
The `contractId` and `address` fields in the `OpcodesResponse` are the primary identifiers that consumers use to attribute the opcode trace to a specific contract. Returning `null` or a wrong `contractId` means:
- Debugging tools and indexers that consume this API will attribute the opcode trace to the wrong contract or no contract at all.
- For `CREATE2`-based address reuse (valid in Hedera EVM), the trace is silently attributed to a completely different contract entity, corrupting audit trails and forensic analysis.
- No funds are directly at risk; severity is medium per the stated scope ("unintended smart contract behavior with no concrete funds at direct risk").

### Likelihood Explanation
- **No privileges required.** Any caller with network access to the mirror node REST API can trigger this.
- **Precondition:** A contract that was the recipient of a historical transaction must have been deleted (or its EVM address reused) after that transaction. Contract deletion is a normal Hedera operation (`ContractDeleteTransaction`), and `CREATE2` address reuse is a standard EVM pattern.
- **Repeatability:** The bug is deterministic and reproducible for any qualifying transaction hash. It requires no race condition or timing attack.

### Recommendation
Pass the `consensusTimestamp` into `buildOpcodesResponse()` and use it as the timestamp argument:

```java
// In OpcodesProcessingResult, add consensusTimestamp, OR pass it as a separate parameter:
private OpcodesResponse buildOpcodesResponse(
        @NonNull OpcodesProcessingResult result, long consensusTimestamp) {
    ...
    recipientEntity = commonEntityAccessor
            .get(recipientAddress, Optional.of(consensusTimestamp))
            .orElse(null);
    ...
}
```

The same fix should be applied to `getSenderAddress()` (line 178) and `getReceiverAddress()` (lines 191, 203), which also pass `Optional.empty()` when building `ContractDebugParameters` — those affect the EVM replay inputs, not just response metadata.

### Proof of Concept

**Preconditions:**
1. Deploy contract C at EVM address `0xABCD...` (entity `0.0.1234`) at block B1.
2. Execute a contract call transaction T against C at block B2; record its transaction hash `H`.
3. Delete contract C (`ContractDeleteTransaction`) at block B3 (B3 > B2).

**Trigger:**
```
GET /api/v1/contracts/results/H/opcodes
Accept-Encoding: gzip
```

**Expected result:** `contractId = "0.0.1234"`, `address = "0x000000000000000000000000000000000000004d2"` (long-zero form of entity 1234).

**Actual result:** `contractId = null`, `address = "0x0000000000000000000000000000000000000000"` — because `findByIdAndDeletedIsFalse(1234)` returns empty for the deleted entity.
