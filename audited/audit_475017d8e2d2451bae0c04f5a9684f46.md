### Title
Missing `return` After Block Number Mismatch Allows Data Integrity Corruption or DoS via Malicious Block Node

### Summary
In `BlockAssembler.onEndOfBlock()`, after detecting a mismatch between `BlockHeader.number` and `EndOfBlock.blockNumber`, there is no `return` statement following the `handleRecoverableError()` call. In the default configuration (`HALT_ON_ERROR=false`), execution falls through and the block is assembled and dispatched using the attacker-controlled `blockNumber` from `EndOfBlock` as the filename, corrupting the mirror node's block index. When `HALT_ON_ERROR=true`, `handleRecoverableError()` throws `ParserException`, which terminates the streaming session and prevents all transactions in that block from being processed.

### Finding Description

**Exact code location**: `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, inner class `BlockAssembler`, method `onEndOfBlock()`, lines 239–272.

```java
void onEndOfBlock(final BlockEnd blockEnd) {
    final long blockNumber = blockEnd.getBlockNumber();   // attacker-controlled
    if (pending.isEmpty()) {
        Utility.handleRecoverableError(...);
        return;   // <-- correctly returns here
    }

    final var blockHeader = pending.getFirst().getFirst().getBlockHeader();
    if (blockHeader.getNumber() != blockNumber) {
        Utility.handleRecoverableError(
                "Block number mismatch in BlockHeader({}) and EndOfBlock({})",
                blockHeader.getNumber(),
                blockNumber);
        // *** NO return HERE — execution falls through unconditionally ***
    }

    // ... block assembled from pending items (correct block content) ...
    final var filename = BlockFile.getFilename(blockNumber, false); // uses attacker value
    blockStreamConsumer.accept(new BlockStream(block, null, filename, loadStart));
}
```

**`handleRecoverableError()` behavior** (`Utility.java` lines 220–231):
- `HALT_ON_ERROR=false` (default): logs the error and **returns normally** → execution continues past the mismatch check.
- `HALT_ON_ERROR=true`: throws `ParserException` → propagates up through `onEndOfBlock()` → caught by `catch (Exception ex)` in `streamBlocks()` (line 162) → `onError()` called → `BlockStreamException` thrown → streaming session terminated.

**Root cause**: The mismatch check at line 248 calls `handleRecoverableError()` but has no `return` statement after it. The failed assumption is that `handleRecoverableError()` always halts execution; it only does so under a non-default system property.

**Exploit flow (default mode — data integrity)**:
1. Attacker operates a block node that the mirror node is configured to subscribe to.
2. Attacker sends valid `BlockItemSet` messages for block N (with `BlockHeader.number = N`).
3. Attacker sends `EndOfBlock` with `blockNumber = M` (M ≠ N, e.g., a future block number).
4. `handleRecoverableError()` logs the mismatch and returns.
5. `BlockFile.getFilename(M, false)` generates filename for block M.
6. `blockStreamConsumer.accept(new BlockStream(block, null, filename_M, loadStart))` dispatches block N's content under block M's identity.
7. The mirror node records block N's transactions as belonging to block M, and `getNextBlockNumber()` advances past M, permanently skipping block N.

**Exploit flow (HALT_ON_ERROR=true — DoS)**:
1. Same steps 1–3 above.
2. `handleRecoverableError()` throws `ParserException`.
3. Caught by `catch (Exception ex)` in `streamBlocks()` (line 162) → `onError()` increments error counter.
4. After `maxSubscribeAttempts` (default 3) consecutive triggers, the node is marked inactive (`active = false`) with a readmit delay.
5. `BlockStreamException` is thrown, terminating the streaming session.
6. All transactions in block N are not processed by the mirror node for the duration of the cooldown.

**Existing checks reviewed and shown insufficient**:
- The mismatch check at line 248 exists but is purely diagnostic — it does not prevent downstream processing.
- The `pending.isEmpty()` guard at line 241 correctly uses `return`, but the mismatch guard does not.
- `streamBlocks()` error handling (lines 159–164) catches the thrown exception but this terminates the entire session rather than just rejecting the malformed block.

### Impact Explanation

**Default configuration (HALT_ON_ERROR=false)**: Block N's transactions are permanently recorded under block M's number. The mirror node's sequential block tracking advances past M, causing block N to be silently skipped. This is a persistent data integrity corruption — transactions are either lost or misattributed to the wrong block number, affecting all downstream consumers of the mirror node API.

**HALT_ON_ERROR=true**: Repeated triggering across multiple blocks causes the block node to be marked inactive after `maxSubscribeAttempts` errors. If the attacker controls all configured block nodes, the mirror node cannot process any blocks until the readmit delay expires, at which point the attack can be repeated indefinitely. This is a repeatable DoS against block ingestion.

### Likelihood Explanation

An attacker must operate a block node that the mirror node is configured to subscribe to. Block nodes are permissioned network participants, but the mirror node's trust model does not cryptographically authenticate the `EndOfBlock.blockNumber` field against the `BlockHeader.number` in the block items — it only logs the discrepancy. The default configuration (HALT_ON_ERROR=false) makes the data integrity path exploitable without any special operator configuration. The HALT_ON_ERROR=true path requires a non-default but documented and supported configuration. The attack is trivially repeatable on every block.

### Recommendation

Add a `return` statement immediately after the mismatch `handleRecoverableError()` call to prevent processing a block with a mismatched number:

```java
if (blockHeader.getNumber() != blockNumber) {
    Utility.handleRecoverableError(
            "Block number mismatch in BlockHeader({}) and EndOfBlock({})",
            blockHeader.getNumber(),
            blockNumber);
    return;  // <-- add this
}
```

This ensures that regardless of the `HALT_ON_ERROR` setting, a block with a mismatched `EndOfBlock.blockNumber` is never dispatched to the consumer. The pending state should also be cleared on mismatch to avoid stale data accumulation.

### Proof of Concept

1. Stand up a gRPC server implementing `BlockStreamSubscribeService`.
2. Configure the mirror node to subscribe to this server as its block node.
3. On receiving a `SubscribeStreamRequest`, respond with:
   - A `BLOCK_ITEMS` response containing a valid `BlockHeader` with `number = 100` plus transaction items.
   - An `END_OF_BLOCK` response with `BlockEnd.blockNumber = 999`.
4. **Default mode** (`HALT_ON_ERROR=false`): Observe in the mirror node database that block 100's transactions are stored under block number 999, and block 100 is never re-fetched.
5. **HALT_ON_ERROR=true mode**: Observe `ParserException` in logs, `onError()` increments, and after 3 repetitions the block node is marked inactive. Block 100's transactions are never recorded during the cooldown window. Repeat after readmit delay to sustain the DoS. [1](#0-0) [2](#0-1) [3](#0-2)

    public static void handleRecoverableError(String message, Object... args) {
        var haltOnError = Boolean.parseBoolean(System.getProperty(HALT_ON_ERROR_PROPERTY));

        if (haltOnError) {
            var formattingTuple = MessageFormatter.arrayFormat(message, args);
            var throwable = formattingTuple.getThrowable();
            var formattedMessage = formattingTuple.getMessage();
            throw new ParserException(formattedMessage, throwable);
        } else {
            log.error(RECOVERABLE_ERROR + message, args);
