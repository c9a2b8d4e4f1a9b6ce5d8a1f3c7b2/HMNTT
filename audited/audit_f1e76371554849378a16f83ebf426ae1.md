### Title
Permanent Cache Poisoning of Earliest Block Anchor via `Optional.empty()` Bypass of `unless` Guard in `findEarliest()`

### Summary
The `CACHE_MANAGER_RECORD_FILE_EARLIEST` cache has no TTL and a `maximumSize` of 1. The `@Cacheable` guard `unless = "#result == null"` only prevents caching a literal `null`, but the method returns `Optional<RecordFile>`. When the database returns no rows, the method returns `Optional.empty()` â€” which is not `null` â€” so the empty result is permanently cached for the process lifetime. Any unprivileged user who sends a single `block=earliest` request during a window where the database has no record files (startup, brief partition, or replica lag) permanently poisons the earliest-block anchor, causing all subsequent `earliest`-block queries to return empty for the entire process lifetime.

### Finding Description
**Exact code locations:**

Cache definition â€” no TTL, `maximumSize(1)`: [1](#0-0) 

Repository method with insufficient `unless` guard: [2](#0-1) 

Service dispatch path (any user can reach this via `block=earliest`): [3](#0-2) 

**Root cause:** Spring's `@Cacheable` evaluates `#result` as the actual Java return value. The method signature returns `Optional<RecordFile>`. When the database has no rows, the JPA query returns `Optional.empty()`. `Optional.empty() != null`, so `unless = "#result == null"` evaluates to `false`, and Spring **does** store the empty Optional in the cache. Because the cache has no TTL (`Caffeine.newBuilder().maximumSize(1)` with no `expireAfterWrite`/`expireAfterAccess`), this entry is never evicted for the entire JVM process lifetime.

**Exploit flow:**
1. Attacker identifies a window where `record_file` table is empty or returns no rows â€” this occurs naturally during mirror node startup before the importer has persisted any record files, or during a brief replica-lag event on a read replica.
2. Attacker sends any JSON-RPC request that resolves to `block=earliest` (e.g., `eth_call` with `"block":"earliest"`, `eth_getBlockByNumber` with `"earliest"`).
3. `BlockType.of("earliest")` returns `BlockType.EARLIEST`; `RecordFileServiceImpl.findByBlockType` calls `recordFileRepository.findEarliest()`.
4. DB returns no rows â†’ method returns `Optional.empty()`.
5. `unless = "#result == null"` â†’ `Optional.empty() == null` is `false` â†’ Spring caches `Optional.empty()` under key `CACHE_NAME` in `CACHE_MANAGER_RECORD_FILE_EARLIEST`.
6. All future calls to `findEarliest()` hit the cache and return `Optional.empty()` without touching the database â€” permanently, until process restart.

**Why the existing check is insufficient:** The guard `unless = "#result == null"` is the only protection. It is semantically wrong for a method returning `Optional<T>`: the correct guard must also exclude the empty-Optional case, e.g., `unless = "#result == null || !#result.isPresent()"`.

### Impact Explanation
Every consumer of `findEarliest()` â€” including all `block=earliest` JSON-RPC calls and any historical range validation that uses the earliest block as a lower bound â€” permanently receives `Optional.empty()`. This causes:
- All `eth_call`/`eth_getBlockByNumber`/`eth_getLogs` requests with `block=earliest` to fail or return "block not found" errors for the entire process lifetime.
- Historical range queries that rely on the earliest block anchor to use an incorrect (absent) lower bound, potentially rejecting valid historical queries or silently returning wrong results.
- No recovery is possible without a full process restart, making this a persistent, operator-invisible denial-of-service on historical query functionality.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no knowledge beyond the public JSON-RPC API. The vulnerable window (empty `record_file` table) occurs on every fresh mirror node deployment during the startup period before the importer has written its first record file â€” a window that can last minutes to hours. An attacker monitoring a new deployment or probing during maintenance windows can reliably trigger this with a single HTTP request. The attack is also repeatable after each process restart.

### Recommendation
Fix the `unless` condition in `RecordFileRepository.findEarliest()` to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_EARLIEST,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures `Optional.empty()` is never stored in the no-TTL cache. As defense-in-depth, consider adding a short TTL (e.g., 60 seconds) to `cacheManagerRecordFileEarliest` so that any transient empty result self-heals without a restart.

### Proof of Concept
**Preconditions:** Mirror node web3 service is running; `record_file` table is empty (fresh deployment, before importer has persisted any files).

**Steps:**
1. Send a single unauthenticated JSON-RPC request:
   ```bash
   curl -X POST http://<mirror-node-web3>:8545 \
     -H "Content-Type: application/json" \
     -d '{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["earliest",false],"id":1}'
   ```
2. The DB returns no rows â†’ `Optional.empty()` is cached permanently in `CACHE_MANAGER_RECORD_FILE_EARLIEST`.
3. Wait for the importer to populate `record_file` (records now exist in DB).
4. Send the same request again:
   ```bash
   curl -X POST http://<mirror-node-web3>:8545 \
     -H "Content-Type: application/json" \
     -d '{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["earliest",false],"id":2}'
   ```
5. **Expected (correct):** Returns the genesis block. **Actual (vulnerable):** Returns `null`/not-found, because `findEarliest()` returns the cached `Optional.empty()` without querying the DB. This persists until the JVM is restarted.
