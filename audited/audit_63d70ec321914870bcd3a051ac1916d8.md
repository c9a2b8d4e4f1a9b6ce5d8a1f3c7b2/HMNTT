### Title
LATEST Sentinel Collision: `0x7fffffffffffffff` Bypasses Reference-Equality Guard in `findByBlockType`, Returning Empty Optional

### Summary
`BlockType.of("0x7fffffffffffffff")` produces a **new** `BlockType` instance with `number = Long.MAX_VALUE` and `name = "7fffffffffffffff"`, which is structurally distinct from the `LATEST` singleton (`name = "latest"`). Because `findByBlockType` guards the `findLatest()` path with a reference-equality check (`block == BlockType.LATEST`), this new instance falls through to `findByIndex(Long.MAX_VALUE)`, which returns an empty `Optional` since no block with that index exists. Any unprivileged caller that controls the block parameter can trigger this path, causing transaction processing to fail or fall back to incorrect state.

### Finding Description

**Root cause — sentinel collision in `BlockType`:**

`LATEST` is defined as:
```java
public static final BlockType LATEST = new BlockType("latest", Long.MAX_VALUE);
``` [1](#0-0) 

The hex-number branch of `BlockType.of()` accepts any 1–16 hex-digit string prefixed with `0x`:
```java
final var hexNum = matcher.group(GROUP_HEX_NUM);   // group 4: 0x([0-9a-f]{1,16})
if (hexNum != null) {
    return new BlockType(hexNum, Long.parseLong(hexNum, 16));
}
``` [2](#0-1) 

`0x7fffffffffffffff` is exactly 16 hex digits, matches `GROUP_HEX_NUM`, and `Long.parseLong("7fffffffffffffff", 16)` returns `Long.MAX_VALUE` without overflow. The result is `new BlockType("7fffffffffffffff", Long.MAX_VALUE)` — a **different object** from `LATEST`.

The existing test suite even documents this as accepted input:
```
"0x7fffffffffffffff," + MAX_VALUE   // listed under valid()
``` [3](#0-2) 

**Bypass in `findByBlockType`:**

```java
} else if (block == BlockType.LATEST) {      // reference equality — fails for the new instance
    return recordFileRepository.findLatest();
}
...
return recordFileRepository.findByIndex(block.number());  // called with Long.MAX_VALUE
``` [4](#0-3) 

`findByIndex(Long.MAX_VALUE)` queries the database for a block whose index is `9223372036854775807`. No such block exists, so the method returns `Optional.empty()` instead of the actual latest block.

**Why existing checks are insufficient:**
- The `==` guard is a reference-equality check, not a value/number check. Java records auto-generate `equals()` based on all fields; `LATEST.equals(new BlockType("7fffffffffffffff", MAX_VALUE))` is `false` because `name` differs. Neither `equals` nor a numeric sentinel check (`block.number() == Long.MAX_VALUE`) is used.
- The `isHash()` guard (`number == BLOCK_HASH_SENTINEL`) uses `-1L` as sentinel, so it correctly passes through; there is no analogous guard for the LATEST sentinel value.

### Impact Explanation
Any API endpoint that accepts a user-supplied block parameter (e.g., `eth_call`, `eth_getBalance`, `eth_getCode`, `eth_estimateGas`) and routes it through `BlockType.of()` → `findByBlockType()` will receive `Optional.empty()` instead of the latest `RecordFile`. Callers that throw on empty Optional produce a 4xx/5xx error for the requesting user. Callers that fall back to a default or cached state may execute the transaction against stale or zero-initialized EVM state, producing incorrect results (wrong balances, wrong contract storage). The attacker needs no credentials — only the ability to send a JSON-RPC request.

### Likelihood Explanation
The input `0x7fffffffffffffff` is a syntactically valid Ethereum hex block number accepted by every standard JSON-RPC client library. Any user who knows (or discovers by fuzzing) that Long.MAX_VALUE is the LATEST sentinel can reproduce this deterministically on every request. No rate-limiting or authentication is required. The attack is repeatable and stateless.

### Recommendation
Replace the reference-equality guards in `findByBlockType` with value-based checks, or normalize the sentinel in `BlockType.of()`:

**Option A — normalize in `BlockType.of()`:** After parsing a hex/decimal number, if the result equals `Long.MAX_VALUE`, return the `LATEST` singleton instead of a new instance:
```java
long n = Long.parseLong(hexNum, 16);
return (n == Long.MAX_VALUE) ? LATEST : new BlockType(hexNum, n);
```

**Option B — guard by number in `findByBlockType`:**
```java
} else if (block == BlockType.LATEST || block.number() == Long.MAX_VALUE) {
    return recordFileRepository.findLatest();
}
```

Option A is preferable because it fixes the invariant at the source and makes `toString()`, caching, and any future callers consistent without requiring each call site to be patched.

### Proof of Concept
1. Start the mirror-node web3 service locally.
2. Send a standard `eth_getBalance` JSON-RPC call using `"0x7fffffffffffffff"` as the block parameter:
   ```json
   {"jsonrpc":"2.0","method":"eth_getBalance",
    "params":["0x<any_address>","0x7fffffffffffffff"],"id":1}
   ```
3. Observe that the service returns an error or zero balance instead of the balance at the latest block.
4. Repeat with `"latest"` as the block parameter and observe the correct latest-block balance is returned.
5. Confirm in debug logs that `findByIndex(9223372036854775807)` is called (not `findLatest()`), and that the repository returns `Optional.empty()`.
