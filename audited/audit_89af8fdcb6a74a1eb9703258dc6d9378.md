### Title
Unbounded Concurrent Subscriptions via Multi-Connection Bypass of Per-Connection Call Limit in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` creates an independent, perpetually-repeating DB polling loop for every subscription accepted. The only server-side guard is `maxConcurrentCallsPerConnection = 5`, which is scoped to a single TCP connection. Because there is no per-IP connection limit and no global subscription cap, an attacker opens an arbitrary number of TCP connections — each carrying 5 subscriptions — causing DB query rate and memory to grow linearly with the number of connections, with no server-side ceiling.

### Finding Description

**Code path — the unbounded polling loop:**

`PollingTopicListener.listen()` ( [1](#0-0) ) constructs a `Flux` that calls `topicMessageRepository.findByFilter()` on every tick and repeats `Long.MAX_VALUE` times with a fixed 500 ms interval. Each call to `listen()` is fully independent — it allocates its own `PollingContext`, its own `AtomicLong`, its own `AtomicReference`, and its own scheduler slot. [2](#0-1) 

**The only guard — per-connection call limit:**

`GrpcConfiguration` applies `maxConcurrentCallsPerConnection` to the Netty server builder: [3](#0-2) 

The default value is 5: [4](#0-3) 

This is a **per-TCP-connection** limit. It does not restrict how many TCP connections a single IP may open. There is no `maxConnectionsPerIp`, no `maxConnections` (global), and no IP-based interceptor anywhere in the gRPC configuration.

**The subscriber counter is a metric, not a gate:**

`TopicMessageServiceImpl` tracks `subscriberCount` as a Micrometer gauge: [5](#0-4) 

It is incremented on subscribe and decremented on completion: [6](#0-5) 

There is no `if (subscriberCount.get() >= MAX) { reject }` check anywhere. The counter is observability-only.

**No per-topicId or per-IP subscription accounting exists** anywhere in `TopicMessageServiceImpl.subscribeTopic()`, `PollingTopicListener.listen()`, `ListenerProperties`, or `GrpcProperties`. [7](#0-6) 

### Impact Explanation

When `hiero.mirror.grpc.listener.type = POLL`, each accepted subscription drives one independent DB query every 500 ms (default `interval`). With N attacker-controlled TCP connections and 5 calls each:

- **Active subscriptions**: 5N
- **DB queries/second**: 5N / 0.5 = **10N**
- **Memory**: 5N `PollingContext` objects + 5N Reactor `Flux` chains + 5N scheduler tasks

At N = 500 connections (easily achievable from a single host with non-blocking I/O), the server sustains 5,000 DB queries/second from attacker traffic alone, exhausting the database connection pool and starving legitimate users. The `boundedElastic` scheduler used by the polling loops has a thread ceiling, but thread starvation itself causes cascading latency for all subscribers. The service becomes unavailable to legitimate clients — a non-network-based DoS against the mirror node's gRPC API.

### Likelihood Explanation

- **Precondition**: Listener type must be `POLL` (`hiero.mirror.grpc.listener.type=POLL`). Default is `REDIS`, but POLL is a documented, supported mode.
- **Attacker capability**: Zero privileges required. The `subscribeTopic` RPC is unauthenticated. A single machine with a non-blocking gRPC client (e.g., `grpc-java` async stub) can open thousands of connections.
- **Repeatability**: Fully repeatable. Connections can be re-opened after server-side timeouts. No authentication, no CAPTCHA, no token.
- **Detection difficulty**: The `subscriberCount` gauge will spike, but there is no alerting threshold enforced in code, and the attack is indistinguishable from a legitimate burst of subscribers at the protocol level.

### Recommendation

1. **Add a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount.get() >= configuredMax` with `RESOURCE_EXHAUSTED` status.
2. **Add a per-IP connection limit** in `GrpcConfiguration` via a gRPC `ServerInterceptor` that tracks active connections per remote address and rejects new ones above a threshold.
3. **Add a per-IP subscription limit** using a `ConcurrentHashMap<InetAddress, AtomicInteger>` in the interceptor or service layer.
4. **Enforce a minimum `limit` or `endTime`** on `subscribeTopic` requests to prevent infinite open-ended subscriptions from a single client.
5. Consider exposing `maxConcurrentCallsPerConnection` as insufficient documentation — clarify in operator docs that it does not bound total load.

### Proof of Concept

```python
# Requires: grpcio, protobuf, hedera-mirror proto stubs
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor
# proto-generated stub: ConsensusServiceStub, ConsensusTopicQuery, TopicID

TARGET = "mirror-node-grpc-host:5600"
TOPIC_NUM = 1  # any valid topic
NUM_CONNECTIONS = 500
CALLS_PER_CONNECTION = 5  # matches maxConcurrentCallsPerConnection default

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = ConsensusServiceStub(channel)
    query = ConsensusTopicQuery(
        topicID=TopicID(topicNum=TOPIC_NUM),
        limit=0  # unlimited — subscription never self-terminates
    )
    threads = []
    for _ in range(CALLS_PER_CONNECTION):
        t = threading.Thread(
            target=lambda: list(stub.subscribeTopic(query)),
            daemon=True
        )
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as pool:
    list(pool.map(open_subscriptions, range(NUM_CONNECTIONS)))

# Result: 2500 active subscriptions, 5000 DB queries/second sustained.
# Monitor hiero.mirror.grpc.subscribers metric — will show unbounded growth.
# DB connection pool exhausted; legitimate subscribers receive UNAVAILABLE errors.
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    @Override
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
