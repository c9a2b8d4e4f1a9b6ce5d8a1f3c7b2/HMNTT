### Title
Unauthenticated Cache-Miss DoS via Unbounded `getByEvmAddressAndType()` Database Queries in GraphQL Module

### Summary
The GraphQL module's `EntityServiceImpl.getByEvmAddressAndType()` calls `entityRepository.findByEvmAddress()` on every invocation with no caching layer and no application-level rate limiting. An unprivileged attacker can flood the `/graphql` endpoint with `account` queries using unique EVM addresses, causing one direct database query per request, exhausting the connection pool and denying service to legitimate users.

### Finding Description

**Exact code path:**

`AccountController.account()` routes `evmAddress`-based lookups to `EntityServiceImpl.getByEvmAddressAndType()`: [1](#0-0) 

`getByEvmAddressAndType()` unconditionally calls `entityRepository.findByEvmAddress()`: [2](#0-1) 

The GraphQL module's `EntityRepository.findByEvmAddress()` has **no `@Cacheable` annotation**: [3](#0-2) 

This is in direct contrast to the `web3` module's `EntityRepository`, which applies `@Cacheable` on its equivalent method: [4](#0-3) 

**No application-level rate limiting exists in the graphql module.** The `ThrottleConfiguration` and `ThrottleManagerImpl` are scoped exclusively to the `web3` module: [5](#0-4) 

The only protections in `GraphQlConfiguration` are query complexity and depth limits, which do not bound request rate: [6](#0-5) 

**Why existing checks fail:**

The Helm chart configures a Traefik `inFlightReq` middleware limiting concurrent requests to 5 per source IP: [7](#0-6) 

This is insufficient because:
- It is per-IP, not global — a distributed attacker with N IPs can sustain N×5 concurrent DB queries simultaneously.
- It limits concurrency, not request rate — an attacker can continuously cycle requests.
- `ipStrategy.depth: 1` uses the outermost `X-Forwarded-For` header, which is trivially spoofable if the load balancer is not the first hop.
- It is an infrastructure-layer control absent from the application itself, meaning direct access bypasses it entirely.

### Impact Explanation

Each request with a unique EVM address triggers a full `SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE` query against the database. With no caching and no rate limiting, a sustained flood of requests with unique addresses exhausts the database connection pool, causing connection timeouts for all concurrent users of the graphql service. This is a complete denial of service for entity lookups, including legitimate transaction entity lookups.

### Likelihood Explanation

No authentication or API key is required to call the GraphQL endpoint. The attack requires only HTTP access and the ability to generate unique 20-byte hex strings (trivially done programmatically). A single attacker with one machine and multiple source IPs (e.g., via a VPN, proxy rotation, or a small botnet) can bypass the per-IP Traefik limit. The attack is fully repeatable and requires no special knowledge of the system.

### Recommendation

1. **Add `@Cacheable` to `EntityRepository.findByEvmAddress()`** in the graphql module, mirroring the pattern used in the `web3` module. Cache negative results (empty `Optional`) with a short TTL to prevent cache-miss amplification.
2. **Add application-level rate limiting** to the graphql module (e.g., a global bucket4j `requestsPerSecond` filter analogous to `ThrottleConfiguration` in `web3`).
3. **Enforce a global (not per-IP) concurrent request limit** at the application layer so it cannot be bypassed by IP rotation.

### Proof of Concept

```bash
# Generate and send 10,000 unique EVM address queries concurrently
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s -X POST http://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {evmAddress: \\\"$ADDR\\\"}) { id } }\"}" &
done
wait
```

Each request hits `EntityServiceImpl.getByEvmAddressAndType()` → `entityRepository.findByEvmAddress()` → a direct DB query with no cache. With sufficient concurrency, the database connection pool is exhausted and legitimate requests begin receiving connection timeout errors.
