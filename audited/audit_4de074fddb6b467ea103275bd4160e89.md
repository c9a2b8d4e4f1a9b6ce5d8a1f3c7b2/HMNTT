### Title
Unauthenticated `/metrics` Endpoint Enables Route Enumeration and Operational Intelligence Disclosure

### Summary
The Rosetta API's `/metrics` endpoint is registered via `Routes()` in `rosetta/app/middleware/metrics.go` using `promhttp.Handler().ServeHTTP` with no authentication wrapper. Any unprivileged external user who can reach port 5700 can retrieve the full Prometheus text output, which includes `route` label sets across all instrumented metric families — revealing which API routes have been accessed, their traffic volumes, error rates, and response sizes. No network policy restricts access to this port in the Rosetta Helm chart.

### Finding Description
**Code path:**

In `rosetta/app/middleware/metrics.go` lines 64–73, `Routes()` registers the `/metrics` endpoint directly with `promhttp.Handler().ServeHTTP`:

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,          // "/metrics"
            promhttp.Handler().ServeHTTP,  // no auth wrapper
        },
    }
}
``` [1](#0-0) 

The `MetricsMiddleware` (lines 76–83) instruments every request with four metric families, all carrying a `route` label:
- `hiero_mirror_rosetta_request_inflight` (GaugeVec — currently inflight per route)
- `hiero_mirror_rosetta_request_duration` (HistogramVec — accumulates per route over time)
- `hiero_mirror_rosetta_request_bytes` / `hiero_mirror_rosetta_response_bytes` (HistogramVec — accumulates per route) [2](#0-1) 

In `rosetta/main.go` lines 217–219, the `/metrics` route is part of the same router served on the single public port (default 5700), with no authentication middleware inserted before it:

```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
``` [3](#0-2) 

**Root cause:** `promhttp.Handler()` is used bare — no `promhttp.HandlerOpts` with auth, no middleware guard, no separate internal-only port. The failed assumption is that `/metrics` is an internal-only scrape target, but it is co-hosted on the public API port with no access control.

**Why existing checks fail:** The REST API's metrics endpoint defaults to `authentication: true` (documented in `docs/configuration.md` line 569), but the Rosetta API has no equivalent configuration option or enforcement. [4](#0-3) 

No Kubernetes `NetworkPolicy` exists for the Rosetta chart to restrict ingress to the metrics path:


The `serviceMonitor.yaml` confirms `/metrics` is expected to be scraped on the same `http` port as the API, with no bearer token or TLS client auth configured: [5](#0-4) 

### Impact Explanation
An external attacker reading `/metrics` obtains:
1. **Route enumeration** — the `route` label in the histogram families (`_request_duration`, `_request_bytes`, `_response_bytes`) accumulates permanently after the first request to each route, revealing all routes that have ever been hit (including non-standard ones like `/health/liveness`, `/health/readiness`, and any future custom routes).
2. **Traffic pattern intelligence** — per-route request counts, error rates (via `status_code` label on `_request_duration`), and response sizes reveal which endpoints are active, how heavily they are used, and which are returning errors.
3. **Reconnaissance for targeted attacks** — knowing which routes are active and their error rates helps an attacker focus probing on high-value or error-prone endpoints.

### Likelihood Explanation
Precondition: the attacker can reach port 5700 (the default Rosetta API port). Since the Rosetta API is a public blockchain interface by design, this port is routinely internet-exposed. No credentials, tokens, or special headers are required. The exploit is a single unauthenticated HTTP GET, repeatable at will, with zero side effects on the server.

### Recommendation
1. **Serve `/metrics` on a separate internal-only port** (e.g., 9090) not exposed through the gateway or load balancer.
2. **If a separate port is not feasible**, wrap `promhttp.Handler()` with an authentication check (e.g., Bearer token or Basic Auth via `promhttp.HandlerOpts` or a custom middleware guard) before registering it in `Routes()`.
3. **Add a Kubernetes `NetworkPolicy`** to the Rosetta Helm chart that restricts ingress on the metrics port to only the Prometheus scraper pod's namespace/label selector.
4. **Add a `bearerTokenSecret` or `tlsConfig`** to the `ServiceMonitor` so Prometheus authenticates when scraping.

### Proof of Concept
```bash
# No credentials required. Replace <host> with the Rosetta API host.
curl -s http://<host>:5700/metrics | grep 'hiero_mirror_rosetta_request_duration'

# Expected output (example) — reveals all routes ever accessed:
# hiero_mirror_rosetta_request_duration_bucket{application="rosetta",method="POST",route="/account/balance",status_code="200",ws="false",le="0.1"} 42
# hiero_mirror_rosetta_request_duration_bucket{application="rosetta",method="POST",route="/block",status_code="200",ws="false",le="0.1"} 17
# hiero_mirror_rosetta_request_duration_bucket{application="rosetta",method="GET",route="/health/liveness",status_code="200",ws="false",le="0.1"} 300
# ...

# To observe inflight routes in real time, send a slow request in background then immediately scrape:
curl -s -X POST http://<host>:5700/network/list -d '{"metadata":{}}' &
curl -s http://<host>:5700/metrics | grep 'hiero_mirror_rosetta_request_inflight'
# hiero_mirror_rosetta_request_inflight{application="rosetta",method="POST",route="/network/list"} 1
```
