### Title
BufferUnderflowException via Short EVM Address in `getByEvmAddressAndType()`

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` wraps the decoded EVM address bytes in a `ByteBuffer` and unconditionally reads 4 + 8 + 8 = 20 bytes. Because `decodeEvmAddress()` performs no length validation, an unauthenticated attacker can supply a 16-byte (32 hex char) all-zero string that satisfies the first two reads but causes the third `buffer.getLong()` to throw an unchecked `BufferUnderflowException`, crashing the resolver for every such request.

### Finding Description

**Exact code path:**

`decodeEvmAddress()` in `GraphQlUtils.java` strips the optional `0x` prefix and calls `Hex.decodeHex()` with no length check: [1](#0-0) 

The returned byte array — of any even length — is immediately wrapped in a `ByteBuffer` in `getByEvmAddressAndType()`: [2](#0-1) 

The three sequential reads require exactly 20 bytes:
- `buffer.getInt()` — consumes bytes 0–3 (4 bytes)
- `buffer.getLong()` — consumes bytes 4–11 (8 bytes)
- `buffer.getLong()` — consumes bytes 12–19 (8 bytes) **← crashes if fewer than 8 bytes remain**

**Root cause:** `decodeEvmAddress()` assumes the caller will only ever pass a 20-byte (40 hex char) address, but enforces no such constraint. The test suite confirms inputs as short as 5 bytes are accepted: [3](#0-2) 

**Why existing checks fail:** The only upstream guard is `@Valid` on `AccountInput` in `AccountController`, but no `@Pattern` or `@Size` constraint exists on the `evmAddress` field to enforce a 40-character length. The `invalidEvmAddress` test only rejects odd-length and non-hex strings — not short even-length strings: [4](#0-3) 

### Impact Explanation
Every GraphQL query that reaches `getByEvmAddressAndType()` with a crafted short address throws an unchecked `BufferUnderflowException`. Spring GraphQL propagates this as an unhandled resolver error, returning a GraphQL error response. Because the condition is trivially and repeatably triggerable with zero authentication, an attacker can flood the endpoint to degrade or exhaust the GraphQL service's thread pool, causing effective denial of service for all account/contract lookups via the GraphQL API.

### Likelihood Explanation
The GraphQL endpoint is publicly reachable with no authentication required. The exploit requires only a single malformed GraphQL query with a 32-character all-zero hex string — no special knowledge, credentials, or tooling beyond a standard HTTP client. It is fully repeatable and automatable.

### Recommendation
Add an explicit length check inside `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
// In GraphQlUtils.decodeEvmAddress(), after decoding:
byte[] decoded = Hex.decodeHex(evmAddress);
if (decoded.length != 20) {
    throw new IllegalArgumentException(
        "evmAddress must be exactly 20 bytes, got: " + decoded.length);
}
return decoded;
```

Alternatively, add a `@Pattern(regexp = "^(0x)?[0-9a-fA-F]{40}$")` constraint to the `evmAddress` field in `AccountInput` so the violation is rejected at the validation layer before reaching the service.

### Proof of Concept

Send the following GraphQL request to the mirror node GraphQL endpoint (no authentication needed):

```graphql
query {
  account(input: { evmAddress: "00000000000000000000000000000000" }) {
    id
  }
}
```

**Step-by-step execution:**
1. `decodeEvmAddress("00000000000000000000000000000000")` → `byte[16]` (all zeros), no exception
2. `ByteBuffer.wrap(byte[16])` → buffer with 16 bytes capacity
3. `buffer.getInt()` → returns `0`, position = 4, remaining = 12 ✓
4. `buffer.getLong()` → returns `0`, position = 12, remaining = 4 ✓
5. Condition `0 == 0 && 0 == 0` is `true` → enters the `if` branch
6. `buffer.getLong()` → needs 8 bytes, only 4 remain → **`java.nio.BufferUnderflowException`** thrown
7. Exception propagates unhandled through `AccountController` → GraphQL error response

Repeat indefinitely to sustain denial of service against the GraphQL resolver.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
