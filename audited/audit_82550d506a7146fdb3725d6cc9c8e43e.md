### Title
Unauthenticated DoS via Uncached, Unthrottled EVM Address Lookups in GraphQL `EntityServiceImpl`

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` in the GraphQL module routes zero-prefix EVM addresses directly to `entityRepository.findById()`, which has no `@Cacheable` annotation and no application-level rate limiting. An unprivileged attacker submitting concurrent GraphQL queries with crafted zero-prefix addresses (each with a unique entity-num suffix) generates a unique, uncached database query per request, enabling connection pool exhaustion and denial of service.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {
        return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
```

Any 20-byte EVM address whose first 4 bytes (shard) and next 8 bytes (realm) are all zero triggers the `findById()` branch. The attacker controls the final 8 bytes (entity num), making it trivial to generate an unbounded set of distinct inputs.

**Root cause — no caching on the graphql `EntityRepository`:**

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 12–18:

```java
@GraphQlRepository
public interface EntityRepository extends CrudRepository<Entity, Long> {
    @Query(value = "select * from entity where alias = ?1 ...", nativeQuery = true)
    Optional<Entity> findByAlias(byte[] alias);

    @Query(value = "select * from entity where evm_address = ?1 ...", nativeQuery = true)
    Optional<Entity> findByEvmAddress(byte[] evmAddress);
}
```

Neither `findById()` (inherited from `CrudRepository`) nor `findByEvmAddress()` carries any `@Cacheable` annotation. Contrast this with the `web3` module's `EntityRepository`, which annotates both methods with `@Cacheable` backed by a Caffeine cache. Every call to `getByEvmAddressAndType()` in the GraphQL module unconditionally hits the database.

**No application-level rate limiting in the graphql module:**

A search across all Java sources under `graphql/src/main/java/` finds zero occurrences of `@RateLimiter`, `@Cacheable`, `bucket4j`, `resilience4j`, or any throttle mechanism. The `ThrottleManagerImpl` and `ThrottleConfiguration` exist only in the `web3` module. `GraphQlConfiguration` enforces only query complexity (max 200) and depth (max 10) limits — neither of which prevents high-frequency, low-complexity single-field lookups.

**Retry middleware amplifies load:**

`charts/hedera-mirror-graphql/values.yaml`, lines 143–145:

```yaml
- retry:
    attempts: 3
    initialInterval: 100ms
```

Each request that fails (e.g., due to DB timeout) is automatically retried up to 3 times by the Traefik middleware, multiplying effective DB query load by up to 4× per original request.

**The only infrastructure control is per-IP in-flight limiting:**

`charts/hedera-mirror-graphql/values.yaml`, lines 138–142:

```yaml
- inFlightReq:
    amount: 5
    sourceCriterion:
      ipStrategy:
        depth: 1
```

This limits each source IP to 5 concurrent in-flight requests. It is ineffective against distributed attacks (multiple source IPs) and does not prevent sequential high-rate requests from a single IP.

### Impact Explanation

Each crafted request with a unique zero-prefix address (e.g., `0x000000000000000000000000<incrementing_num>`) produces a distinct `SELECT * FROM entity WHERE id = ?` query that cannot be served from any cache. Under concurrent load from multiple IPs, the database connection pool is exhausted, causing all GraphQL queries (including legitimate ones) to queue or fail. The circuit breaker (`ResponseCodeRatio(500, 600, 0, 600) > 0.25`) only activates after 25% of responses are already 5xx — meaning significant degradation occurs before any automatic mitigation engages. This is a full availability denial for the GraphQL service with no authentication required.

### Likelihood Explanation

The GraphQL endpoint (`/graphql/alpha`) is publicly accessible with no authentication. The crafted address pattern (`0x000000000000000000000000XXXXXXXXXXXXXXXX`) is trivially constructable. A single attacker with a modest botnet (or even rotating proxies) bypasses the per-IP `inFlightReq: 5` limit. The attack is repeatable, stateless, and requires no prior knowledge of the system beyond the public API schema.

### Recommendation

1. **Add `@Cacheable` to the graphql `EntityRepository`**: Mirror the pattern used in the `web3` module — annotate `findById()` and `findByEvmAddress()` with `@Cacheable` backed by a bounded Caffeine cache with a short TTL (e.g., 1–5 seconds).
2. **Add application-level rate limiting to the GraphQL module**: Introduce a `bucket4j` or Resilience4j rate limiter (global and/or per-IP) analogous to `ThrottleManagerImpl` in the `web3` module.
3. **Remove or scope the retry middleware**: Retrying on DB-timeout responses amplifies load; limit retries to network-level errors only, not application errors.
4. **Lower the `inFlightReq` threshold or add a global RPS cap** at the Traefik layer to complement per-IP limits.

### Proof of Concept

```bash
# Generate 1000 unique zero-prefix EVM addresses (shard=0, realm=0, num=1..1000)
# and send them concurrently from multiple IPs/threads

for i in $(seq 1 1000); do
  NUM=$(printf '%016x' $i)
  EVM_ADDR="000000000000000000000000${NUM}"
  curl -s -X POST https://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {evmAddress: \\\"${EVM_ADDR}\\\"}) { id } }\"}" &
done
wait
```

Each request routes through `getByEvmAddressAndType()` → `findById(i)` with no cache, issuing 1000 distinct `SELECT * FROM entity WHERE id = ?` queries. With concurrent execution across multiple source IPs (bypassing `inFlightReq: 5` per IP), the database connection pool is exhausted and legitimate requests begin timing out.
