### Title
Missing Rate Limiting on `getOutstandingAirdrops()` Enables Unauthenticated Resource Exhaustion

### Summary
The `GET /api/v1/accounts/{id}/airdrops/outstanding` endpoint in `TokenAirdropsController` accepts up to 100 results per request with no per-IP or global rate limiting in the `rest-java` module. An unauthenticated attacker can flood this endpoint with concurrent requests, exhausting the Tomcat thread pool and HikariCP database connection pool, degrading or denying service to legitimate users at zero economic cost.

### Finding Description
**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TokenAirdropsController.java`, `getOutstandingAirdrops()`, lines 66–75.

```java
@GetMapping(value = "/outstanding")
TokenAirdropsResponse getOutstandingAirdrops(
        @PathVariable EntityIdParameter id,
        @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
        ...
```

`MAX_LIMIT = 100` (`Constants.java` line 35). Each request invokes `entityService.lookup()` + `repository.findAll()` — two synchronous blocking DB operations per request (`TokenAirdropServiceImpl.java` lines 20–21).

**Root cause:** The `rest-java` module has no rate-limiting mechanism. Its filter chain contains only `LoggingFilter` (logging only) and `MetricsFilter` (metrics only), registered in `RestJavaConfiguration.java`. There is no `ThrottleConfiguration`, `ThrottleManager`, or bucket4j integration anywhere under `rest-java/src/main/java/`. By contrast, the `web3` module has a full `ThrottleConfiguration` + `ThrottleManagerImpl` using bucket4j (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`), but this is entirely absent from `rest-java`.

**Exploit flow:**
1. Attacker sends N concurrent `GET /api/v1/accounts/0.0.1/airdrops/outstanding?limit=100` requests with no credentials.
2. Each request blocks a Tomcat worker thread while executing two DB queries.
3. With enough concurrent requests, all Tomcat threads (default 200) and all HikariCP connections are occupied.
4. Legitimate requests queue and time out; the server returns 503 or hangs.

**Why existing checks fail:**
- `@Max(MAX_LIMIT)` only caps the result size at 100; it does not limit request rate.
- `LoggingFilter` and `MetricsFilter` are observability-only; neither rejects or throttles requests.
- `ShallowEtagHeaderFilter` (registered in `RestJavaConfiguration`) only handles ETags.
- No Spring Security, no servlet filter, no interceptor in `rest-java` enforces a rate limit.

### Impact Explanation
An attacker can saturate the server's thread pool and database connection pool, causing legitimate API users to experience timeouts or 503 errors across all `rest-java` endpoints (not just airdrops). The impact is availability degradation of the mirror node REST API. Classified as medium/griefing because there is no direct economic damage to on-chain state, but it disrupts network observability for all users.

### Likelihood Explanation
Requires no credentials, no on-chain funds, no special knowledge — only the ability to send HTTP requests. A single machine with a modest HTTP flood tool (e.g., `ab`, `wrk`, `hey`) can saturate a default Tomcat thread pool. The attack is trivially repeatable and automatable. The absence of any rate limiting in `rest-java` (unlike `web3`) makes this a realistic, low-effort attack.

### Recommendation
Add rate limiting to the `rest-java` module mirroring the pattern already used in `web3`:
1. Introduce a `ThrottleConfiguration` bean in `rest-java/src/main/java/org/hiero/mirror/restjava/config/` using bucket4j, scoped to requests-per-second per IP or globally.
2. Add a servlet filter (extending `OncePerRequestFilter`) that consumes a token from the bucket and returns HTTP 429 when exhausted, registered before the controller dispatch.
3. Alternatively, enforce rate limiting at the ingress/load-balancer layer (e.g., nginx `limit_req`, Kubernetes ingress annotations) as a defense-in-depth measure.

### Proof of Concept
```bash
# Flood the endpoint with 500 concurrent requests, 10 at a time
ab -n 500 -c 50 \
  "http://<mirror-node-host>/api/v1/accounts/0.0.1/airdrops/outstanding?limit=100"

# Observe: legitimate requests begin timing out or returning 503
# while the attack is in progress, with zero authentication required.
```
