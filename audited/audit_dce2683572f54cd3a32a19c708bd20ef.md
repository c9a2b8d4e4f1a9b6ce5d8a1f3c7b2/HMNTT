### Title
Unauthenticated Timestamp-Range DoS via Unconditional DB Query in `getAccountBalanceTimestampRange()`

### Summary
The `/api/v1/balances` endpoint executes a database query inside `getAccountBalanceTimestampRange()` for every request containing a timestamp filter, regardless of whether the query will return results. Because no rate limiting exists on the REST API and no caching guards the empty-result path, an unprivileged attacker can flood the endpoint with timestamp ranges that always miss balance snapshots, generating sustained, unbounded database load.

### Finding Description

**Exact code path:**

`getBalances()` → `getTsQuery()` → `getAccountBalanceTimestampRange()` → `pool.queryQuietly()`

In `rest/balances.js` lines 113–116, whenever a `timestamp` query parameter is present, `getTsQuery()` is called unconditionally:

```js
if (tsQuery) {
  const tsQueryResult = await getTsQuery(tsQuery, tsParams);   // line 114
  if (!tsQueryResult.query) { return; }                        // line 115-117
```

Inside `getTsQuery()` (line 278), `getAccountBalanceTimestampRange()` is called, which at line 189 always fires a database query before inspecting the result:

```js
const {rows} = await pool.queryQuietly(query, params);   // line 189
if (rows.length === 0) { return {}; }                    // line 190-192
```

The query is:
```sql
SELECT consensus_timestamp FROM account_balance
WHERE account_id = $1 AND consensus_timestamp >= $2 AND consensus_timestamp <= $3
ORDER BY consensus_timestamp DESC LIMIT 1
```

**Root cause:** The DB query is issued before any check that the timestamp range contains a snapshot. There is no short-circuit, no result cache, and no request-level rate limit on this endpoint.

**Why existing checks are insufficient:**

1. `getOptimizedTimestampRange()` (lines 229–276) limits the scan to ~2 monthly partitions, but does **not** prevent the query from executing — it only narrows the range.
2. The response cache (`responseCacheCheckHandler`, line 97) is only active when **both** `config.cache.response.enabled` and `config.redis.enabled` are true (line 54). It is optional infrastructure, not a guaranteed guard. Cache misses (the attacker's target) still reach the DB.
3. `server.js` (lines 68–144) shows the full middleware stack for `/api/v1/balances` (line 106): `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, optional cache check — **no rate limiter**.
4. The only rate limiting in the codebase (`web3/ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) applies exclusively to the `web3` module's contract-call endpoints, not to the REST API.

### Impact Explanation
Every timestamp-filtered request to `/api/v1/balances` that misses a snapshot consumes a DB connection and executes a query against the `account_balance` table. Under sustained flood, the DB connection pool is exhausted, causing all mirror node REST API queries to queue or fail. Because the importer and REST API share the same PostgreSQL instance, extreme pool exhaustion can also degrade ingestion of new network data, making the mirror node unable to serve current state. The impact is denial-of-service of the mirror node REST API and potential degradation of data ingestion.

### Likelihood Explanation
The endpoint is public and unauthenticated. The attack requires only a standard HTTP client. The attacker needs to identify a 1-second window between two 15-minute balance snapshots (trivially deterministic from any prior valid response's `timestamp` field) and send requests with `timestamp=gte:T&timestamp=lte:T+1` where no snapshot exists. The attack is fully automatable, repeatable, and requires no credentials or special knowledge.

### Recommendation
1. **Add rate limiting to the REST API.** Apply a per-IP or global request-rate limiter (e.g., `express-rate-limit`) to all routes in `server.js`, mirroring the pattern already used in the `web3` module.
2. **Cache the empty-result path.** When `getAccountBalanceTimestampRange()` returns `{}`, cache that negative result (keyed on the optimized `[lowerBound, upperBound]` pair) for a short TTL (e.g., 15 seconds) to collapse repeated identical misses.
3. **Fail fast before the DB query.** In `getBalances()`, validate that the requested timestamp range is plausible (e.g., not in a known gap) before issuing the probe query, or add a lightweight in-process bloom/LRU cache of recently-queried empty ranges.

### Proof of Concept

**Precondition:** Identify any 1-second window between two consecutive 15-minute balance snapshots. A prior valid response gives the last snapshot timestamp `T`.

**Trigger:**
```bash
# T_gap is any second strictly between two snapshots
T_gap_ns=<nanosecond timestamp between two snapshots>
T_gap_end=$((T_gap_ns + 1000000000))  # +1 second

# Flood with requests that always miss
while true; do
  curl -s "https://<mirror-node>/api/v1/balances?timestamp=gte:${T_gap_ns}&timestamp=lte:${T_gap_end}" &
done
```

**Result:** Each request executes the probe query in `getAccountBalanceTimestampRange()`, returns empty, and exits early.
```javascript
const getOptimizedTimestampRange = (tsQuery, tsParams) => {
  let lowerBound = 0n;
  const neParams = [];
  let upperBound = constants.MAX_LONG;

  // Find the lower bound and the upper bound from tsParams if present
  tsQuery
    .split('?')
    .filter(Boolean)
    .forEach((query, index) => {
      const value = BigInt(tsParams[index]);
      // eq operator has already been converted to the lte operator
      if (query.includes(utils.opsMap.lte)) {
        // lte operator includes the lt operator, so this clause must before the lt clause
        upperBound = utils.bigIntMin(upperBound, value);
      } else if (query.includes(utils.opsMap.lt)) {
        // Convert lt to lte to simplify query
        const ltValue = value - 1n;
        upperBound = utils.bigIntMin(upperBound, ltValue);
      } else if (query.includes(utils.opsMap.gte)) {
        // gte operator includes the gt operator, so this clause must come before the gt clause
        lowerBound = utils.bigIntMax(lowerBound, value);
      } else if (query.includes(utils.opsMap.gt)) {
        // Convert gt to gte to simplify query
        const gtValue = value + 1n;
        lowerBound = utils.bigIntMax(lowerBound, gtValue);
      } else if (query.includes(utils.opsMap.ne)) {
        neParams.push(value);
      }
    });

  if (lowerBound > upperBound) {
    return {};
  }

  // The optimized range of [lower, upper] should overlap with at most two months, with the exception that when upper
  // is more than 1 month in the future, the range may cover more months. Since the partition maintenance job will
  // create at most one monthly partition ahead, it's unnecessary to adjust the upper bound.
  // With the assumption that the data in db is in sync with the network, in other words, the balance information is
  // update-to-date as of NOW in wall clock, the algorithm below sets lower bound to
  //   max(lowerBound from user, first day of the month before the month min(now, upperBound) is in)
  const nowInNs = utils.nowInNs();
  const effectiveUpperBound = utils.bigIntMin(upperBound, nowInNs);
  const optimalLowerBound = utils.getFirstDayOfMonth(effectiveUpperBound, -1);
  lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);

  return {lowerBound, upperBound, neParams};
};
```
