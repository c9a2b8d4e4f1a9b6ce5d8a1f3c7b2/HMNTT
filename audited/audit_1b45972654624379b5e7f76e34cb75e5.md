### Title
Unbounded `req.originalUrl` Logging in `handleError` Enables Unauthenticated Log-Flooding DoS

### Summary
The `handleError` middleware unconditionally interpolates the full `req.originalUrl` into every `logger.warn` / `logger.error` call with no length cap, no rate limiting, and no sanitization. Because any unauthenticated HTTP request can trivially trigger an error path (404, 400, 413, etc.), an attacker can sustain a high-volume flood of large log messages, exhausting disk space, filling the stdout pipe, and stalling the Node.js event loop.

### Finding Description
**Exact code path:**

`rest/middleware/httpErrorHandler.js`, `handleError`, lines 33–39:

```js
logger.warn(
  `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ...`
);
// and
logger.error(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ...`, detailedMessage);
``` [1](#0-0) 

`req.originalUrl` is embedded verbatim. Node.js's HTTP parser accepts request-line URLs up to ~8 KB by default (configurable higher); Express imposes no additional cap. There is no truncation, no `maxLength` guard, and no per-IP or global rate limit anywhere in the middleware chain.

**Logger implementation** (`rest/monitoring/logger.js`, lines 13–23) calls `process.stdout.write(text)` synchronously. When stdout is a pipe (the normal container/production case), this call blocks the Node.js event loop once the OS pipe buffer is saturated. [2](#0-1) 

**Middleware stack** (`rest/server.js`, lines 67–144) contains no rate-limiting, no `helmet`, no URL-length middleware, and no `express-rate-limit` before `handleError` is registered. [3](#0-2) 

**Root cause / failed assumption:** The code assumes log volume is bounded by legitimate traffic. It does not account for an adversary who deliberately maximises URL length and request rate to amplify log output.

**Error triggering is trivial for an unauthenticated user:**
- Any path not matching a registered route → `NotFoundError` → `handleError` → `logger.warn` with full URL.
- Any request with an invalid query parameter → `InvalidArgumentError` → same path.
- `requestQueryParser` uses `throwOnLimitExceeded: true`, so exceeding `maxRepeatedQueryParameters` also throws into `handleError`. [4](#0-3) 

### Impact Explanation
1. **Disk exhaustion** – if stdout is redirected to a file or a log aggregator that writes to disk, each ~8 KB URL produces an ~8 KB log line. At 10 000 req/s this is ~80 MB/s of log data, filling disk within minutes.
2. **Event-loop stall** – `process.stdout.write()` blocks when the pipe buffer is full. A sustained flood can cause the single-threaded Node.js process to stop serving legitimate requests entirely.
3. **Log aggregator overload** – downstream systems (Elasticsearch, Loki, Splunk) can be overwhelmed, causing log loss across the entire platform.

Severity: **Medium–High** (availability impact, no authentication required, no resource cap in application code).

### Likelihood Explanation
The attack requires only an HTTP client and a loop. No credentials, no special knowledge of the API, and no exploit chain are needed. A single machine with a modest connection rate (hundreds of req/s) is sufficient to produce measurable impact. The attack is fully repeatable and can be automated trivially.

### Recommendation
1. **Truncate `req.originalUrl` before logging** – cap at a safe length (e.g., 512 characters):
   ```js
   const safeUrl = req.originalUrl?.slice(0, 512) ?? '';
   logger.warn(`${req.ip} ${req.method} ${safeUrl} in ${elapsed} ms: ...`);
   ```
2. **Add application-level rate limiting** – use `express-rate-limit` (or equivalent) before route handlers to cap requests per IP.
3. **Set `--max-http-header-size`** – lower the Node.js HTTP parser URL limit via the `--max-http-header-size` flag or enforce it at the reverse-proxy layer.
4. **Use asynchronous logging** – replace synchronous `process.stdout.write()` with a non-blocking logger (e.g., `pino`) to prevent event-loop stalls under log pressure.

### Proof of Concept
```bash
# Generate a ~7 KB URL path and flood the error handler
LONG_PATH=$(python3 -c "print('/api/v1/' + 'A'*7000)")

# Send 5000 concurrent requests (adjust -c/-n as needed)
ab -n 50000 -c 200 "http://<target>:5551${LONG_PATH}"

# Observe: each request returns 404 and logs a ~7 KB line via logger.warn
# Monitor disk usage / stdout pipe saturation on the server
watch -n1 'df -h /var/log && ls -lh /proc/$(pgrep node)/fd/1'
```

Each request is unauthenticated, hits no valid route, triggers `NotFoundError`, and causes `handleError` to emit a log line containing the full 7 KB URL. At 200 concurrent connections the log output rate exceeds 1 MB/s, exhausting typical pipe buffers and disk quotas within minutes.
