### Title
Unbounded `account.id` Range Parameter Enables Wide Index Range Scan DoS on NFT Allowances Endpoint

### Summary
The `getNftAllowances()` endpoint accepts `account.id=gte:X&account.id=lte:Y` with no restriction on the width of the range `[X, Y]`. An unprivileged user can supply `account.id=gte:0.0.1&account.id=lte:0.0.9999999` with `limit=100`, causing the database to perform a wide index range scan on `NFT_ALLOWANCE.SPENDER` (or `NFT_ALLOWANCE.OWNER`). Repeated concurrent requests of this form constitute a low-cost, unauthenticated denial-of-service vector against the database.

### Finding Description

**Code path:**

`AllowancesController.java` line 60–61 accepts up to two `account.id` parameters with no range-width constraint:

```java
@RequestParam(name = ACCOUNT_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] accountIds,
@RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
``` [1](#0-0) 

`MAX_LIMIT` is 100, so `limit=100` is fully valid. [2](#0-1) 

The `Bound` constructor (called with `primarySortField=true` for `ownerOrSpenderIds`) only checks that `adjustedLower <= adjustedUpper`; it does **not** check the width of the range:

```java
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException("Invalid range provided for %s".formatted(parameterName));
}
``` [3](#0-2) 

`NftAllowanceServiceImpl.checkOwnerSpenderParamValidity` validates operator type, single-occurrence, EQ/range mixing, and token.id co-presence — but **never** validates the numeric span of the range: [4](#0-3) 

**Resulting query shape:**
```sql
SELECT * FROM nft_allowance
WHERE owner = {path_id}
  AND spender >= 1
  AND spender <= 9999999
ORDER BY spender ASC, token_id ASC
LIMIT 100;
```
With an index on `(owner, spender, token_id)`, PostgreSQL seeks to `(owner={path_id}, spender=1)` and scans forward. If the account has fewer than 100 allowances in the range, the engine scans the entire range before returning. An attacker targeting a high-value account (e.g., a known NFT marketplace with many allowances) can force the engine to traverse millions of index entries per request.

### Impact Explanation
Each crafted request forces a full index range scan across up to ~10 million spender values for the targeted account. With no rate limiting visible in the controller layer and `limit=100` only bounding the result set (not the scan depth), an attacker sending tens of concurrent requests can saturate database I/O and CPU, degrading availability for all users of the mirror node. Severity: **Medium–High** (unauthenticated, repeatable, targets shared database infrastructure).

### Likelihood Explanation
No authentication is required. The endpoint is publicly documented in `openapi.yml`. [5](#0-4) 
The exploit requires only a standard HTTP client. The attacker needs only to know a target account ID with a non-trivial number of NFT allowances (discoverable via the same API). Repeatability is unlimited.

### Recommendation
1. **Enforce a maximum range width** in `Bound` or `NftAllowanceServiceImpl`: reject requests where `adjustedUpper - adjustedLower` exceeds a configurable threshold (e.g., 1,000 or 10,000 entity IDs).
2. **Apply rate limiting** at the API gateway or Spring filter layer, keyed on client IP, for this endpoint.
3. Consider requiring a minimum specificity (e.g., `account.id=eq:X` or a narrow `gte/lte` window) before allowing secondary `token.id` range filters.

### Proof of Concept
```
# Unauthenticated wide-range scan
GET /api/v1/accounts/0.0.5000/allowances/nfts?account.id=gte:0.0.1&account.id=lte:0.0.9999999&limit=100&owner=true

# Repeat concurrently (e.g., 50 parallel requests) to amplify DB load:
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.5000/allowances/nfts?account.id=gte:0.0.1&account.id=lte:0.0.9999999&limit=100" &
done
wait
```
All 50 requests pass every validation gate (`@Size(max=2)`, `@Max(100)`, lower ≤ upper) and each triggers a full index range scan on `NFT_ALLOWANCE.SPENDER` for the targeted account.
