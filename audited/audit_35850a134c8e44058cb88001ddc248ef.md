### Title
Missing `responseTimeout` on `streamBlocks()` Allows Malicious Block Node to Hold Streaming Thread Indefinitely

### Summary
`StreamProperties.responseTimeout` (400 ms) is applied only to the unary `serverStatus` gRPC call inside `getBlockRange()`, but the long-lived `streamBlocks()` subscription is opened with `CallOptions.DEFAULT` (no gRPC deadline). The only guard inside the streaming loop is a per-block stopwatch that resets to the full 30-second `commonDownloaderProperties.timeout` after every completed block. A malicious block node can exploit this by completing syntactically valid blocks just before the 30-second window expires, resetting the timer indefinitely and permanently occupying the importer's single streaming thread.

### Finding Description

**Exact code path:**

`getBlockRange()` correctly applies `responseTimeout`:
```java
// BlockNode.java lines 101-102
final var blockNodeService = BlockNodeServiceGrpc.newBlockingStub(channel)
        .withDeadlineAfter(streamProperties.getResponseTimeout());   // 400 ms
```

`streamBlocks()` opens the subscription with **no deadline**:
```java
// BlockNode.java lines 129-133
final var grpcCall = ClientCalls.blockingV2ServerStreamingCall(
        channel,
        BlockStreamSubscribeServiceGrpc.getSubscribeBlockStreamMethod(),
        CallOptions.DEFAULT,   // ← no deadline, no timeout
        request);
```

The only timeout guard is `assembler.timeout()` used as the per-`read()` wait:
```java
// BlockNode.java line 138
while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
```

`BlockAssembler.timeout()` (lines 274-281) starts a `Stopwatch` on the first block item and returns the remaining milliseconds of `commonDownloaderProperties.getTimeout()` (default **30 seconds**). Critically, `stopwatch.reset()` is called inside `onEndOfBlock()` (line 268) every time a block is completed, giving the attacker a fresh 30-second window per block.

**Root cause:** The per-block stopwatch is a liveness guard for a single block, not a wall-clock deadline for the entire streaming session. Because it resets on every `END_OF_BLOCK`, there is no cumulative upper bound on how long a single `streamBlocks()` invocation can run.

**Exploit flow:**
1. Attacker controls (or compromises) a block node that the importer is configured to contact.
2. Importer calls `streamBlocks(blockNumber, …)`.
3. Malicious node sends `BLOCK_HEADER` → stopwatch starts (30 s window).
4. Malicious node sends up to `maxBlockItems - 1` (799,999) arbitrary block items, each arriving well within the remaining window.
5. Malicious node sends `END_OF_BLOCK` at ~29 s → `onEndOfBlock()` fires, `stopwatch.reset()` is called, timer resets to 30 s.
6. Repeat from step 3 indefinitely.

Because `BlockNodeSubscriber` uses a **single-threaded executor** (`Executors.newSingleThreadExecutor()`, line 48) and `doGet()` calls `node.streamBlocks(…)` synchronously (lines 66-69), the single streaming thread is permanently occupied. No other block node can be tried while this call is blocking.

**Why existing checks are insufficient:**

| Check | Why it fails |
|---|---|
| `maxBlockItems = 800_000` | Resets to 0 on every `END_OF_BLOCK`; attacker just completes each block before the limit |
| `maxSubscribeAttempts = 3` / `readmitDelay` | Only triggered by exceptions; a node that completes blocks successfully never increments `errors` |
| `maxStreamResponseSize = 36 MB` | Per-message size cap, not a session cap |
| `responseTimeout = 400 ms` | Applied only to `serverStatus`, never to `streamBlocks()` |

### Impact Explanation
The importer's streaming thread is permanently blocked on the malicious node's connection. No blocks from any other node (or from cloud storage fallback) can be processed while the thread is held. This halts mirror-node ingestion entirely for as long as the attacker sustains the attack, constituting a complete availability failure for the mirror node and all downstream consumers (REST API, gRPC API, etc.).

### Likelihood Explanation
Any operator of a block node that appears in the importer's configuration (static or dynamically discovered via `BlockNodeDiscoveryService`) can execute this attack without any cryptographic material or special privilege beyond being a reachable block node. The attack is trivially repeatable: after a restart or node-list refresh, the attacker simply reconnects. No exploit tooling beyond a custom gRPC server is required.

### Recommendation
1. **Apply a gRPC call deadline to `streamBlocks()`**: pass a `CallOptions` with a reasonable wall-clock deadline (e.g., a configurable `streamTimeout`, separate from the per-block timeout) so the entire RPC is bounded:
   ```java
   CallOptions.DEFAULT.withDeadlineAfter(streamProperties.getStreamTimeout().toMillis(), TimeUnit.MILLISECONDS)
   ```
2. **Add a cumulative session timeout**: track total elapsed time across all blocks in a single `streamBlocks()` invocation and abort if it exceeds a configurable maximum (e.g., 5 minutes).
3. **Apply `responseTimeout` (or a similarly short value) as the per-read timeout** when no block is in progress (stopwatch not running), so an idle connection between blocks is also bounded.

### Proof of Concept
```
1. Stand up a gRPC server implementing BlockStreamSubscribeService.
2. Configure the importer to use this server as its sole block node.
3. In subscribeBlockStream(), for each incoming request:
   a. Send SubscribeStreamResponse{BLOCK_ITEMS: [BLOCK_HEADER(blockN)]}
   b. Every 100 ms, send SubscribeStreamResponse{BLOCK_ITEMS: [EVENT_HEADER]}
   c. At t=29s, send SubscribeStreamResponse{END_OF_BLOCK(blockN)}
   d. Immediately increment blockN and go to step (a).
4. Observe: the importer's streaming thread never returns from streamBlocks().
5. Observe: no blocks are ingested; mirror node REST/gRPC APIs return stale data indefinitely.
6. The attack continues until the importer process is restarted and the node is removed from config.
```
