### Title
Asymmetric Health-Check Failure via DB Resource Exhaustion on `/network/status` Enables Readiness Denial-of-Service

### Summary
`checkNetworkStatus` in `rosetta/app/middleware/health.go` makes two sequential HTTP calls — `NetworkList` (in-memory, infallible) then `NetworkStatus` (DB-dependent, fallible) — with no circuit breaker and `SkipOnErr: false`. Because both calls share the same process port and DB connection pool as public traffic, an attacker who exhausts DB connections via the public `/network/status` endpoint causes the health check's own `NetworkStatus` call to fail while `NetworkList` continues to succeed, marking the pod not ready. Repeating this across the replica set can take ≥30% of nodes out of the load balancer without any brute-force or privileged access.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/health.go`, `checkNetworkStatus()`, lines 80–101:

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
    serverUrl := fmt.Sprintf("http://localhost:%d", port)   // same port as public API
    cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
    rosettaClient := client.NewAPIClient(cfg)

    return func(ctx context.Context) (checkErr error) {
        networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, ...)
        // NetworkList returns static in-memory data — never fails
        if err != nil { return err }

        network := networkList.NetworkIdentifiers[0]
        _, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, ...)
        // NetworkStatus makes 3 DB queries — fails under DB pressure
        if err != nil { return err }   // ← no circuit breaker, no SkipOnErr
        return
    }
}
```

**Root cause — asymmetric failure modes:**

`NetworkList` (`network_service.go` line 31) returns a static in-memory struct and **never** touches the database. `NetworkStatus` (`network_service.go` lines 67–80) issues three sequential DB queries (`RetrieveGenesis`, `RetrieveLatest`, `Entries`). Both share the same GORM connection pool (`db/db.go` lines 31–33, bounded by `Pool.MaxOpenConnections`). The health check calls `localhost:<port>` directly (bypassing Traefik), so it competes with public traffic for the same finite pool.

**Failed assumption:** The design assumes `NetworkStatus` is as reliable as `NetworkList`. It is not — one is in-memory, the other is DB-bound.

**Exploit flow:**

1. Attacker sends concurrent POST requests to the public `/rosetta/network/status` endpoint. Each request causes `NetworkStatus` to hold a DB connection for the duration of `RetrieveGenesis` + `RetrieveLatest` + `Entries`.
2. With `inFlightReq: amount: 5` per source IP (Traefik), the attacker uses N source IPs to maintain N×5 concurrent in-flight requests, each holding a DB connection.
3. The DB connection pool (`MaxOpenConnections`) becomes saturated. New connection requests block or time out.
4. The health check's own `NetworkStatus` call (via `localhost`) cannot acquire a DB connection within the 10-second `health.Config.Timeout` (`health.go` line 46).
5. `checkNetworkStatus` returns an error; `SkipOnErr: false` (`health.go` line 47) causes the readiness check to report `StatusUnavailable`.
6. Kubernetes readiness probe (`failureThreshold: 5`, `values.yaml` line 237) marks the pod not ready after 5 consecutive failures, removing it from the load balancer.
7. Repeating across replicas takes ≥30% of nodes out of service.

**Why existing checks are insufficient:**

- **Traefik `rateLimit: average: 10` per `requestHost`** (`values.yaml` line 157–160): This is a token-bucket average, not a hard cap on concurrent connections. Burst traffic can exceed it momentarily. More critically, `global.middleware: false` (`values.yaml` line 95) means the entire middleware stack — including rate limiting — can be disabled at deploy time, leaving the application with zero application-level protection.
- **Traefik `circuitBreaker: NetworkErrorRatio() > 0.25`** (`values.yaml` line 150–151): This only gates *external* traffic through Traefik. The health check calls `localhost` directly and is completely unaffected by this circuit breaker.
- **`podDisruptionBudget: enabled: false`** (`values.yaml` line 175): No PDB prevents multiple pods from being simultaneously marked not ready.
- **No application-level circuit breaker or retry** in `checkNetworkStatus`: a single transient DB timeout immediately propagates to a not-ready verdict.

### Impact Explanation

When ≥30% of rosetta pods fail readiness checks, Kubernetes removes them from the Service endpoints. The load balancer routes all traffic to the remaining pods, increasing their DB load, potentially cascading the failure to additional pods. This matches the stated scope: "Shutdown of greater than or equal to 30% of network processing nodes without brute force actions." The `RosettaNoPodsReady` alert (`values.yaml` line 202–212) fires only after all pods are gone, meaning partial degradation goes undetected until it becomes total outage.

### Likelihood Explanation

The attacker needs only:
- Public network access to the `/rosetta/network/status` endpoint (no credentials)
- Multiple source IPs (available via cloud VMs, proxies, or a small botnet) to bypass per-IP `inFlightReq` limits
- Knowledge that the endpoint is DB-backed (trivially discoverable from open-source code or by observing response latency)

The attack is repeatable and self-sustaining: as long as the attacker maintains concurrent requests, the health check continues to fail. The `statementTimeout` of 20 seconds (`docs/configuration.md` line 662) exceeds the health check timeout of 10 seconds (`health.go` line 46), meaning a query that is merely *slow* (not erroring) is sufficient to trigger the failure. If Traefik middleware is disabled (`global.middleware: false`), the attack requires no special resources at all.

### Recommendation

1. **Add a circuit breaker inside `checkNetworkStatus`**: use a library such as `sony/gobreaker` or `afex/hystrix-go` to stop calling `NetworkStatus` after consecutive failures, preventing cascading readiness failures.
2. **Set `SkipOnErr: true` for the `network` check** or use a separate, lighter liveness signal (e.g., a dedicated `/health/network` endpoint that caches the last successful `NetworkStatus` result with a TTL) so a transient DB hiccup does not immediately mark the pod not ready.
3. **Isolate the health-check DB connection**: give `checkNetworkStatus` a dedicated, minimal-size connection pool separate from the public-traffic pool, so public traffic exhaustion cannot starve the health check.
4. **Enable `podDisruptionBudget`** (`minAvailable: 70%`) to prevent Kubernetes from simultaneously removing more than 30% of pods.
5. **Apply application-level rate limiting** on `/network/status` independent of Traefik, so the protection is not bypassable by disabling `global.middleware`.

  curl -s -X POST http://<rosetta-host>/rosetta/network/status \
    -H "Content-Type: application/json" \

done

# Step 2: Simultaneously poll the readiness endpoint:
watch -n1 'curl -s -o /dev/null -w "%{http_code}" \
  http://<rosetta-host>/health/readiness'

# Expected result:
# - While concurrent /network/status requests hold DB connections,
#   /health/readiness returns 503 (StatusServiceUnavailable)
