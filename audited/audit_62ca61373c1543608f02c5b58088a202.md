### Title
Negative Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode()` Enables Denial-of-Service for Newly Deployed Contracts

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"` to prevent caching empty results. However, when a contract does not exist, the method returns `Optional.empty()` — a non-null Java object — causing the SpEL condition to evaluate to `false` and the empty result to be cached. Because the cache uses `expireAfterAccess=1h` with no eviction mechanism, an unprivileged attacker can pre-poison the cache for a predictable future contract ID, causing all subsequent bytecode lookups for that contract to return empty for up to one hour (or indefinitely with repeated access), effectively denying service to any newly deployed contract.

### Finding Description
**Exact code location:**
- `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18
- Cache configuration: `web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 22

**Root cause:**
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```
The `unless` guard is `#result == null`. When the SQL query `select runtime_bytecode from contract where id = :contractId` finds no matching row, Spring Data JPA returns `Optional.empty()`. `Optional.empty()` is a singleton non-null Java object, so `#result == null` evaluates to `false`, and Spring's caching infrastructure proceeds to store the empty `Optional` in the Caffeine cache under the key `contractId`.

The cache is configured as `expireAfterAccess=1h,maximumSize=1000,recordStats` — entries expire only after 1 hour of inactivity. There is no `@CacheEvict` or any other invalidation path in the web3 module for this cache.

**Exploit flow:**
1. Attacker calls any web3 endpoint (e.g., `eth_call`, `eth_getCode`) targeting a contract address whose Hedera entity ID is `X` — before the contract is deployed.
2. `ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode(X)`.
3. DB has no row for `X` → returns `Optional.empty()`.
4. `unless = "#result == null"` → `false` → `Optional.empty()` is stored in cache under key `X`.
5. The contract is subsequently deployed on-chain and the importer writes its bytecode to the `contract` table.
6. All future calls to `findRuntimeBytecode(X)` return the cached `Optional.empty()` without hitting the DB.
7. `readFromDataSource()` maps `Optional.empty()` to `null` (via `.orElse(null)`), so the EVM sees no bytecode for the contract.
8. Every `eth_call`, `eth_estimateGas`, or contract interaction targeting contract `X` fails as if the contract does not exist.

**Why existing checks fail:**
The only guard is `unless = "#result == null"`. It does not account for `Optional.empty()`. There is no secondary check, no `@CacheEvict` on write paths, and no TTL short enough to self-heal quickly (1 hour access-based expiry, renewable by the attacker).

### Impact Explanation
Any newly deployed contract whose entity ID was pre-queried by an attacker becomes unreachable through the mirror node's web3 API for up to 1 hour. Since the cache uses `expireAfterAccess`, the attacker can extend the DoS indefinitely by periodically re-querying the same ID, resetting the 1-hour timer. This affects all callers — dApps, wallets, and other integrations — that rely on the mirror node for EVM execution. The impact is targeted and repeatable per contract ID.

### Likelihood Explanation
Hedera entity IDs are sequential integers, publicly observable via the mirror node REST API (e.g., `/api/v1/contracts`). An attacker can trivially predict the next contract ID by observing the current highest ID. The trigger requires only a standard unauthenticated HTTP request to any web3 endpoint (e.g., `eth_getCode` with the target address). No authentication, no special role, no on-chain funds are required. The attack is fully automatable and can be applied to every new contract deployment.

### Recommendation
Change the `unless` condition to also exclude `Optional.empty()` from being cached:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

This ensures only successful (non-empty) bytecode lookups are cached, and any miss always falls through to the database on the next call.

### Proof of Concept
**Preconditions:** Mirror node web3 API is accessible. Current highest contract entity ID is `N` (observable via `GET /api/v1/contracts?order=desc&limit=1`).

**Steps:**
1. Predict next contract ID: `TARGET = N + 1` (or any near-future ID).
2. Pre-poison the cache:
   ```
   curl -X POST http://<mirror-node>/api/v1/contracts/call \
     -H "Content-Type: application/json" \
     -d '{"to":"0x0000000000000000000000000000000000000<TARGET_HEX>","data":"0x"}'
   ```
   This triggers `findRuntimeBytecode(TARGET)` → `Optional.empty()` → cached.
3. Wait for a legitimate user to deploy a contract that receives entity ID `TARGET`.
4. Immediately after deployment, call the contract:
   ```
   curl -X POST http://<mirror-node>/api/v1/contracts/call \
     -H "Content-Type: application/json" \
     -d '{"to":"0x0000000000000000000000000000000000000<TARGET_HEX>","data":"<any_selector>"}'
   ```
5. **Observed:** The call fails (contract not found / no bytecode) despite the contract being deployed.
6. **Expected:** The call succeeds and executes the contract.
7. To extend the DoS beyond 1 hour, repeat step 2 every ~50 minutes to reset the `expireAfterAccess` timer.
