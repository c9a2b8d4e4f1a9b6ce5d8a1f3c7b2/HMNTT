### Title
GraphQL Field Aliasing Bypasses Complexity Guard, Enabling DB Connection Pool Exhaustion

### Summary
The `MaxQueryComplexityInstrumentation(200)` in `GraphQlConfiguration` uses the default `FieldComplexityCalculator`, which assigns a uniform weight of 1 to every field regardless of whether it performs a database lookup. An unauthenticated attacker can pack ~50 aliased `account(...)` calls into a single query document — each alias triggers a separate `entityService` DB call — while the computed complexity (≈104) stays well below the 200 limit, and no rate limiting or authentication guards the endpoint.

### Finding Description

**Exact code path:**

- Schema: `graphql/src/main/resources/graphql/query.graphqls` line 5 — `account(input: AccountInput!): Account`
- Complexity guard: `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java` line 43 — `new MaxQueryComplexityInstrumentation(200)` with no custom `FieldComplexityCalculator`
- Resolver: `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java` lines 33–58 — every invocation of `account(...)` calls `entityService.getByIdAndType(...)` (or alias/evmAddress variants), each issuing a DB query

**Root cause — failed assumption:**

`MaxQueryComplexityInstrumentation` with its default calculator assigns complexity 1 to *every* field node in the AST, treating a trivial scalar (`id`) identically to a top-level resolver that hits the database (`account`). The assumption that "complexity ≤ 200 bounds resource consumption" is false when a single field selection maps to a full DB round-trip.

**Exploit flow:**

1. Each alias `aN: account(input:{entityId:{num:N}}){id}` consumes ≈19 GraphQL tokens and 2 complexity points.
2. The `maxTokens=1000` parser limit (set at line 34 of `GraphQlConfiguration.java`) allows ≈52 such aliases per document: `(1000 − 2) / 19 ≈ 52`.
3. Complexity for 52 aliases: `52 × 2 = 104 < 200` → **complexity check passes**.
4. GraphQL-java executes all 52 `account` field selections, each invoking `AccountController.account(...)` → `entityService` → DB query.
5. The `CachedPreparsedDocumentProvider` (lines 24–27 of `CachedPreparsedDocumentProvider.java`) caches the parsed+validated document keyed on the query string. Repeated identical queries skip validation entirely and go straight to execution, removing even the parse/validate overhead from subsequent requests.
6. No authentication and no rate limiting exist on the `/graphql/alpha` endpoint.

**Why existing checks fail:**

| Check | Limit | Actual constraint on aliases |
|---|---|---|
| `MaxQueryComplexityInstrumentation(200)` | 200 | Allows 100 aliases (`account{id}` = 2 pts each) |
| `maxTokens=1000` | 1000 tokens | Allows ≈52 aliases (binding limit) |
| `maxCharacters=10000` | 10000 chars | Allows ≈400+ aliases |
| JSON `maxTokenCount=100` | 100 JSON tokens | Irrelevant — entire GraphQL doc is 1 JSON string token |
| Authentication | None | No auth on GraphQL endpoint |
| Rate limiting | None | Not implemented for GraphQL |

### Impact Explanation
Each HTTP request with 52 aliased `account(...)` calls issues 52 synchronous DB queries. An attacker sending N concurrent such requests generates 52×N simultaneous DB connections. A typical R2DBC/HikariCP pool of 10–20 connections is exhausted with fewer than 1 concurrent attacker request per pool slot. Pool exhaustion causes all GraphQL queries (and potentially other DB-dependent services sharing the same PostgreSQL instance) to queue or fail, resulting in full service unavailability for legitimate users. The `statementTimeout=10000ms` (10 s) means each connection is held for up to 10 seconds per request, maximizing pool starvation duration.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable at `/graphql/alpha`. No special knowledge, credentials, or tooling beyond `curl` is required. The attack is trivially scriptable, repeatable, and requires no prior reconnaissance beyond knowing the schema (which is self-describing via introspection). A single attacker with a modest number of concurrent HTTP connections can sustain the attack indefinitely.

### Recommendation
1. **Assign higher complexity weight to DB-hitting resolvers.** Provide a custom `FieldComplexityCalculator` to `MaxQueryComplexityInstrumentation` that assigns a cost of, e.g., 10 to `account` (and any future top-level query fields) and 1 to scalar sub-fields. This reduces the maximum number of `account` aliases to 20 before the limit is hit.
2. **Add per-IP or global rate limiting** on the `/graphql/alpha` endpoint (e.g., via Spring's `HandlerInterceptor` or an API gateway).
3. **Limit top-level field repetition.** Reject queries where the same top-level field name appears more than once (aliases included), or set the effective alias limit to 1 for root query fields.
4. **Consider requiring authentication** for the GraphQL endpoint, consistent with its "alpha" status.

### Proof of Concept

```bash
# Build a query with 50 aliased account lookups (complexity = 100, tokens ≈ 952 — both under limits)
QUERY='{'
for i in $(seq 1 50); do
  QUERY="${QUERY} a${i}: account(input:{entityId:{num:${i}}}){id}"
done
QUERY="${QUERY} }"

# Send concurrently — each request triggers 50 DB lookups
for c in $(seq 1 20); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"${QUERY}\"}" &
done
wait
# Result: 20 concurrent requests × 50 DB queries = 1000 simultaneous DB queries
# → DB connection pool exhausted → subsequent legitimate requests time out or fail
```
