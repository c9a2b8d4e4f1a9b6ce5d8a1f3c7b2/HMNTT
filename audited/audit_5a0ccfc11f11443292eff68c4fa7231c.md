### Title
Integer Overflow in `PollingTopicListener.poll()` Causes Unbounded Database Query via Silently Dropped Limit

### Summary
An unprivileged user can submit a gRPC `subscribeTopic` request with a `limit` value between `Integer.MAX_VALUE + 1` and `Long.MAX_VALUE` (e.g., `2^31 = 2147483648`). This passes the `@Min(0)` bean validation but, inside `PollingTopicListener.poll()`, the narrowing cast `(int)(filter.getLimit() - context.getCount().get())` overflows to a negative integer. Because `TopicMessageFilter.hasLimit()` returns `false` for any negative value, the downstream `findByFilter` call omits `setMaxResults` entirely, issuing an unbounded `SELECT` that loads every matching row into the JVM heap on every poll cycle.

### Finding Description

**Code path:**

1. `ConsensusController.toFilter()` — `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java` line 56 — copies the raw `uint64` proto field directly into `TopicMessageFilter.limit` (a Java `long`).

2. `TopicMessageService.subscribeTopic(@Valid filter)` — `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java` line 59 — Spring AOP validates the filter; `@Min(0)` passes because `2147483648L > 0`.

3. `TopicMessageServiceImpl.incomingMessages()` — lines 115–118 — computes `long limit = filter.getLimit() - topicContext.getCount().get()` (no overflow, still `2147483648L`) and builds a new `TopicMessageFilter` with that value, passing it to `topicListener.listen(newFilter)`.

4. **`PollingTopicListener.poll()`** — `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java` lines 54–57:
   ```java
   int limit = filter.hasLimit()
           ? (int) (filter.getLimit() - context.getCount().get())
           //  (int)(2147483648L - 0L) = (int)2147483648L = -2147483648  ← overflow
           : Integer.MAX_VALUE;
   int pageSize = Math.min(-2147483648, listenerProperties.getMaxPageSize());
   // = -2147483648  (always less than any positive maxPageSize)
   ```
   The narrowing cast silently wraps to `Integer.MIN_VALUE`.

5. Line 59 builds `newFilter` with `limit = -2147483648` via `filter.toBuilder().limit(pageSize)…build()`. This is an internal builder call — it is **not** routed through the Spring `@Valid`-annotated `subscribeTopic` method, so `@Min(0)` is never re-evaluated.

6. **`TopicMessageRepositoryCustomImpl.findByFilter()`** — `grpc/src/main/java/org/hiero/mirror/grpc/repository/TopicMessageRepositoryCustomImpl.java` lines 51–53:
   ```java
   if (filter.hasLimit()) {   // -2147483648 > 0 → false
       typedQuery.setMaxResults(…);
   }
   ```
   `hasLimit()` returns `false` for any negative value, so `setMaxResults` is never called. Line 60 then executes `typedQuery.getResultList().stream()`, which materialises **all matching rows** into the JVM heap.

**Root cause:** The `long`→`int` narrowing cast at line 55 of `PollingTopicListener.java` is unchecked. Any user-supplied limit in the range `(Integer.MAX_VALUE, Long.MAX_VALUE]` survives input validation (it is a valid positive `long`) but overflows to a negative `int`, which `hasLimit()` treats as "no limit."

**Why existing checks fail:**
- `@Min(0)` on `TopicMessageFilter.limit` only rejects negative input; it does not bound the value to `Integer.MAX_VALUE`.
- The `@Valid` annotation on `subscribeTopic` is enforced only at the Spring bean boundary; the internal `toBuilder().limit(pageSize).build()` call inside `poll()` bypasses it entirely.
- `Math.min(limit, listenerProperties.getMaxPageSize())` is intended as a safety cap but operates on the already-overflowed negative value, making it ineffective.

### Impact Explanation
On every poll interval (default configurable, e.g. 500 ms) the affected subscription triggers a full table scan of `topic_message` for the targeted topic with no `LIMIT` clause. `getResultList()` loads the entire result set into heap memory. On a busy topic with millions of messages this causes:
- Repeated heap pressure / GC storms on the gRPC service JVM.
- Database connection saturation and query-plan degradation (the `random_page_cost` hint is still applied, worsening planner choices for other queries).
- Throughput degradation for all other subscribers sharing the same `boundedElastic` scheduler and database connection pool, constituting a denial-of-service against the polling pipeline.

### Likelihood Explanation
The gRPC endpoint is publicly reachable with no authentication required. The `ConsensusTopicQuery` proto field `limit` is `uint64`; any client library can trivially set it to `2147483648` (one above `Integer.MAX_VALUE`). The attack is repeatable, requires no special knowledge beyond the public proto definition, and a single subscription is sufficient to trigger the condition on every poll cycle indefinitely.

### Recommendation
1. **Clamp before casting** in `PollingTopicListener.poll()`:
   ```java
   long rawLimit = filter.getLimit() - context.getCount().get();
   int limit = rawLimit > listenerProperties.getMaxPageSize()
           ? listenerProperties.getMaxPageSize()
           : (int) rawLimit;
   ```
2. **Add an upper-bound constraint** on `TopicMessageFilter.limit`, e.g. `@Max(Integer.MAX_VALUE)` or a domain-specific maximum, so values that cannot safely be narrowed to `int` are rejected at the validation boundary.
3. **Guard `findByFilter`**: in `TopicMessageRepositoryCustomImpl.findByFilter()`, treat any `limit <= 0` as an error or as "apply a safe default cap" rather than silently omitting `setMaxResults`.

### Proof of Concept
```python
# Using the official Hedera/Mirror gRPC proto
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    limit=2147483648,          # 2^31 — passes @Min(0), overflows (int) cast
)

# Each streamed response triggers poll() → unbounded SELECT on every interval
for response in stub.subscribeTopic(query):
    pass   # attacker keeps stream open; server hammers DB with full scans
```
