### Title
Unauthenticated EVM Address Cache-Miss DB Connection Pool Exhaustion DoS in `getAccountCryptoAllowances`

### Summary
The `getAccountCryptoAllowances` handler in `rest/controllers/cryptoAllowanceController.js` accepts any valid EVM address as a path parameter and unconditionally issues a live database query via `getEntityIdFromEvmAddress()` with no caching and no rate limiting on the Node.js REST service. An unauthenticated attacker can flood the endpoint with syntactically valid but non-existent EVM addresses, saturating the finite `pg` connection pool and causing legitimate queries to queue or fail, resulting in denial of service of the mirror node REST API.

### Finding Description

**Code path:**

`CryptoAllowanceController.getAccountCryptoAllowances` (`rest/controllers/cryptoAllowanceController.js:77`) calls `EntityService.getEncodedId(req.params[...])`. [1](#0-0) 

`getEncodedId` (`rest/service/entityService.js:118-124`) detects a valid EVM address (`entityId.evmAddress !== null`) and delegates to `getEntityIdFromEvmAddress`. [2](#0-1) 

`getEntityIdFromEvmAddress` (`rest/service/entityService.js:90-91`) executes a live DB query every single call with no in-process cache:
```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
  [Buffer.from(entityId.evmAddress, 'hex')]);
``` [3](#0-2) 

The SQL issued is:
```sql
select id from entity where deleted <> true and evm_address = $1
``` [4](#0-3) 

`getRows` acquires a connection from the global `pg` pool (`pool.queryQuietly`), holds it for the duration of the query, then releases it. [5](#0-4) 

The pool is bounded by `config.db.pool.maxConnections` (the `max` parameter of `pg.Pool`): [6](#0-5) 

**Root cause:** `getEntityIdFromEvmAddress` has no caching layer (contrast with the web3 Java service's `@Cacheable` on `findByEvmAddressAndDeletedIsFalse`). Every request with a valid-format but absent EVM address hits the database. The Node.js REST service has no per-endpoint rate limiting (the throttle infrastructure found in the codebase belongs exclusively to the web3 Java service). [7](#0-6) 

**Why existing checks fail:**
- `statement_timeout` limits individual query duration but does not prevent concurrent pool saturation — an attacker sending requests at a rate exceeding `maxConnections / avg_query_duration` keeps every slot occupied.
- pgbouncer's `max_user_connections: 250` for `mirror_rest` is a ceiling, not a rate limit; it does not reject requests before they reach the application pool.
- No authentication or IP-based rate limiting is applied to this route.

### Impact Explanation

Exhausting the `pg` connection pool causes all subsequent REST API requests that require a DB connection to either queue indefinitely or fail with a connection-timeout error. This makes the entire mirror node REST API unavailable to legitimate users for the duration of the attack. Because the mirror node REST API is the primary interface for wallets, explorers, and dApps to query account state and transaction history, sustained unavailability constitutes a significant denial-of-service. Note: this does not affect Hedera consensus-node transaction confirmation (which is independent), but it does deny service to all REST API consumers.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no prior state. Any internet-accessible deployment is reachable. The attacker needs only to generate random 40-hex-character strings and send concurrent HTTP GET requests. This is trivially scriptable with standard tools (`ab`, `wrk`, `curl` in a loop). The attack is repeatable and stateless — each request is independent.

### Recommendation

1. **Add a negative-result cache** in `getEntityIdFromEvmAddress`: cache `null` results for unknown EVM addresses for a short TTL (e.g., 30–60 s) to absorb repeated misses without hitting the DB.
2. **Add rate limiting** to the Node.js REST service for the `/accounts/:id/allowances/*` family of endpoints, consistent with the throttle infrastructure already present in the web3 service.
3. **Consider request coalescing** (deduplicate in-flight queries for the same EVM address) to prevent parallel requests for the same unknown address from each consuming a connection slot.

### Proof of Concept

```bash
# Generate 500 concurrent requests with random non-existent EVM addresses
for i in $(seq 1 500); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/accounts/0x${ADDR}/allowances/crypto" &
done
wait
# Observe: legitimate requests begin timing out or returning 503/connection errors
# as the pg pool (maxConnections slots) is saturated by in-flight DB lookups
```

Each request triggers `getEntityIdFromEvmAddress` → `Buffer.from(addr, 'hex')` → `SELECT id FROM entity WHERE evm_address = $1` with no cache, holding a pool slot until `statement_timeout` fires. With enough concurrent requests the pool is fully occupied and new requests cannot acquire a connection.
