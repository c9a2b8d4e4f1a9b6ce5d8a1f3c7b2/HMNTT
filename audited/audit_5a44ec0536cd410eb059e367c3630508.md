### Title
Unauthenticated Unbounded `contract_state_change` Table Scan via Timestamp `eq`→`lte` Conversion in `getContractStateById`

### Summary
The `GET /contracts/:contractId/state` endpoint accepts a `timestamp` query parameter with any comparison operator. When the `eq` operator is used, `extractContractStateByIdQuery()` silently converts it to `lte` and routes the query to the `contract_state_change` table (instead of the compact `contract_state` table), with no upper-bound range validation. An unauthenticated attacker can supply a far-future timestamp (e.g., `timestamp=eq:9999999999`), causing a full historical scan of `contract_state_change` for the target contract, exhausting the DB connection pool and rendering the mirror node API unavailable.

### Finding Description

**Code path:**

1. `getContractStateById` (line 964–994, `rest/controllers/contractController.js`) calls `extractContractIdAndFiltersFromValidatedRequest`, which invokes `utils.buildAndValidateFilters` with `contractResultsFilterValidityChecks`. [1](#0-0) 

2. `contractResultsFilterValidityChecks` (lines 273–279) only adds special handling for `BLOCK_NUMBER`; it performs no timestamp range validation. [2](#0-1) 

3. Inside `extractContractStateByIdQuery` (lines 909–918), when `filter.operator === opsMap.eq`, the operator is mutated to `opsMap.lte` and `timestampPresent` is set to `true`. No call to `parseTimestampFilters` (which enforces `maxTimestampRange`) is made anywhere in this path. [3](#0-2) 

4. `getContractStateByIdAndFilters` (lines 256–263, `rest/service/contractService.js`) branches on `timestamp === true` and executes `contractStateTimestampQuery` against `contract_state_change` instead of the compact `contract_state` table. [4](#0-3) 

5. `contractStateTimestampQuery` (lines 111–122) issues:
   ```sql
   SELECT DISTINCT ON (slot) ...
   FROM contract_state_change
   WHERE contract_id = $1 AND consensus_timestamp <= $2
   ORDER BY slot, consensus_timestamp DESC
   LIMIT $3
   ```
   The `LIMIT` is applied after `DISTINCT ON`, so the DB must process all rows matching `contract_id = $1 AND consensus_timestamp <= $2` before returning results. [5](#0-4) 

6. The primary key of `contract_state_change` is `(consensus_timestamp, contract_id, slot)`, which is not ordered to efficiently serve `WHERE contract_id = $1 AND consensus_timestamp <= $2 ORDER BY slot, consensus_timestamp DESC`. [6](#0-5) 

**Root cause:** The `extractContractStateByIdQuery` function converts `eq` to `lte` without applying the `maxTimestampRange` guard (default 7 days) that `parseTimestampFilters` enforces for other endpoints. A single unbounded `lte` condition against `contract_state_change` can scan the entire history of a contract.

### Impact Explanation

An attacker can exhaust the DB connection pool (default max 10 connections, 20-second statement timeout per query) by issuing ~10 concurrent requests. During the 20-second window per query, all connections are held, blocking legitimate API requests. This causes mirror node API unavailability (HTTP 503 / connection timeout for all endpoints).

**Important correction on severity classification:** The mirror node is a read-only observer; it does not participate in Hedera network consensus. Exhausting the mirror node DB does **not** prevent the Hedera network from confirming transactions. The actual impact is mirror node API DoS, not "total network shutdown." [7](#0-6) 

### Likelihood Explanation

- No authentication or API key is required.
- The endpoint is publicly documented and accessible.
- The exploit requires only a single HTTP GET request with a crafted timestamp parameter.
- Repeatable at will; the attacker needs no special knowledge beyond the contract ID (which is enumerable from other public endpoints).
- Contracts with large state histories (e.g., heavily-used DeFi contracts) amplify the scan cost.

### Recommendation

1. **Apply `maxTimestampRange` validation** in `extractContractStateByIdQuery` for the timestamp filter, consistent with other endpoints. Call `utils.parseTimestampFilters` on the collected timestamp filters before building conditions, rejecting open-ended or excessively wide ranges.
2. **Reject bare `lte`/`gte` without a paired bound** for the state endpoint, or require an explicit upper bound when a lower bound is provided.
3. **Add a composite index** on `contract_state_change (contract_id, slot, consensus_timestamp DESC)` to make the `DISTINCT ON` query efficient even when it must run.
4. Consider **rate-limiting** the `/contracts/:contractId/state` endpoint with a timestamp filter at the API gateway level.

### Proof of Concept

```bash
# Step 1: Identify any valid contract ID (publicly enumerable)
CONTRACT_ID="0.0.1234"

# Step 2: Send a single request with eq timestamp far in the future
# eq is silently converted to lte, scanning all historical state changes
curl "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}/state?timestamp=eq:9999999999"

# Step 3: Send ~10 concurrent requests to exhaust the DB connection pool
for i in $(seq 1 10); do
  curl "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}/state?timestamp=eq:9999999999" &
done
wait

# Result: All DB connections held for up to 20 seconds each.
# Legitimate API requests receive connection errors or timeouts.
# Repeat to maintain sustained unavailability.
```

The same effect is achievable with `timestamp=lte:9999999999` directly, which bypasses even the `ne`-operator check and goes straight to the unbounded scan. [8](#0-7)
