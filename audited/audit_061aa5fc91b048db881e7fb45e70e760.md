### Title
Unbounded DB Query Amplification via Crafted EVM CALL Opcodes in `ContractController.call()`

### Summary
An unprivileged user can submit a single `/api/v1/contracts/call` request whose `data` field encodes bytecode that executes EVM `CALL` opcodes to thousands of unique non-existent addresses. Because the entity cache explicitly skips caching null/empty results (`unless = "#result == null"`), every such CALL triggers a separate synchronous database query. The gas-based throttle limits total gas per second but does not bound the number of DB queries generated per unit of gas, creating a large amplification factor that can saturate the database connection pool and degrade service for all users.

### Finding Description

**Exact code path:**

1. `ContractController.call()` — no authentication, open to any caller. [1](#0-0) 

2. The only pre-execution guards are a gas-bucket throttle and a max-gas-limit check: [2](#0-1) [3](#0-2) 

3. The gas throttle consumes `scaleGas(gas) = gas / 10_000` tokens per request. With `maxGasLimit = 15,000,000`, each max-gas request consumes 1,500 tokens from a bucket of 750,000 tokens/s, allowing up to ~500 max-gas requests per second: [4](#0-3) [5](#0-4) 

4. During EVM execution, each `CALL` opcode to a non-long-zero address resolves the target through `CommonEntityAccessor.get(Address, Optional<Long>)` → `entityRepository.findByEvmAddressAndDeletedIsFalse()` (for latest-block calls) or `entityRepository.findActiveByEvmAddressAndTimestamp()` (for historical calls): [6](#0-5) [7](#0-6) 

5. **Root cause — cache misses are never stored.** The `@Cacheable` annotation on `findByEvmAddressAndDeletedIsFalse` uses `unless = "#result == null"`. When the target address does not exist in the database, the result is `Optional.empty()` (treated as null by Spring Cache), so the miss is **not** written to the cache. Every subsequent CALL to the same or a different non-existent address hits the database again: [8](#0-7) 

6. For historical block queries, `findActiveByEvmAddressAndTimestamp` carries **no `@Cacheable` annotation at all**, making every lookup unconditionally hit the database: [9](#0-8) 

7. The entity cache TTL is only 1 second with a maximum of 10,000 entries, so even hits expire quickly: [10](#0-9) 

**Exploit flow:**

- Attacker crafts bytecode that loops `CALL` to N unique non-existent EVM addresses (e.g., sequentially generated addresses `0xdead0001`, `0xdead0002`, …).
- With `maxGasLimit = 15,000,000` and a minimum CALL cost of ~700 gas, a single request can execute ~21,000 CALL opcodes.
- Each CALL to a unique non-existent address triggers one synchronous DB query.
- The gas throttle permits up to ~500 such requests per second (gas bucket is the binding constraint).
- **Amplification: 500 req/s × 21,000 DB queries/req = ~10,500,000 DB queries/second** from a single attacker.

**Why existing checks fail:**

- The gas throttle limits gas throughput, not DB query count. The ratio of DB queries to gas consumed is unbounded by the throttle.
- The rate limit (500 req/s) is a global shared bucket, not per-IP, so a single attacker can consume the entire budget.
- Cache miss non-storage (`unless = "#result == null"`) is the direct enabler: the cache provides zero protection for non-existent addresses.

### Impact Explanation

A single attacker with no credentials or economic stake can generate millions of database queries per second, saturating the connection pool and causing query timeouts or connection exhaustion for all legitimate users of the web3 module. The `statementTimeout` is 3,000 ms, meaning legitimate requests will begin timing out quickly under load. This is a pure griefing attack — no funds are at risk, but service availability is severely degraded.

### Likelihood Explanation

The endpoint is publicly accessible with no authentication. The attack requires only an HTTP client and knowledge of the API. The crafted bytecode is trivial to construct (a simple loop with `CALL` opcodes). The attack is repeatable indefinitely at zero cost to the attacker (no gas fees, no tokens required). Any motivated party can execute this.

### Recommendation

1. **Cache negative results (null/empty) for non-existent addresses** with a short TTL (e.g., 1–5 seconds). Change `unless = "#result == null"` to `unless = "false"` or use a sentinel value, so repeated lookups for the same non-existent address are served from cache.
2. **Add caching to `findActiveByEvmAddressAndTimestamp`** for historical queries, keyed on `(evmAddress, blockTimestamp)`.
3. **Bound DB queries per request** by tracking the number of entity lookups within a single EVM execution context (`ContractCallContext`) and aborting execution if a configurable threshold is exceeded.
4. **Apply per-IP rate limiting** in addition to the global gas bucket, so a single source cannot consume the entire global budget.
5. Consider **limiting EVM call depth / external CALL count** at the application level, independent of gas.

### Proof of Concept

```
# 1. Construct bytecode that loops CALL to unique addresses:
#    PUSH1 0x00 (retSize) PUSH1 0x00 (retOffset) PUSH1 0x00 (argsSize)
#    PUSH1 0x00 (argsOffset) PUSH1 0x00 (value)
#    PUSH20 <unique_address_N> (to)
#    PUSH3 0x0F4240 (gas=1000000)
#    CALL
#    ... repeated for addresses 0xdead0001 through 0xdead5000

# 2. Send the request (no auth required):
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "data": "<crafted_bytecode_hex>",
    "to": "0x0000000000000000000000000000000000000000",
    "gas": 15000000,
    "estimate": false
  }'

# 3. Repeat in a tight loop from multiple threads.
# Each request triggers ~21,000 DB queries for non-existent addresses.
# Cache misses are not stored, so every request re-queries the DB.
# DB connection pool exhaustion occurs within seconds.
```
