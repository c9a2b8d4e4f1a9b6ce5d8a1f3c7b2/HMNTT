### Title
Unauthenticated EVM Address Flood Exhausts DB Connection Pool via `getEncodedId()` on `/accounts/:idOrAliasOrEvmAddress/nfts`

### Summary
The `/api/v1/accounts/{evmAddress}/nfts` endpoint accepts any syntactically valid EVM address as a path parameter and unconditionally executes a database query (`entityFromEvmAddressQuery`) against the `entity` table for each request. No rate limiting exists for unauthenticated callers in the REST API middleware stack. An attacker flooding this endpoint with unique EVM addresses can sustain a continuous stream of DB queries, exhausting the shared pgbouncer connection pool and degrading or denying service to all mirror node REST instances sharing that pool.

### Finding Description

**Exact code path:**

1. Route registration — `rest/routes/accountRoute.js` line 15:
   ```js

   ```

2. Handler — `rest/controllers/accountController.js` line 91:
   ```js

   ```

3. `getEncodedId` — `rest/service/entityService.js` lines 120–124:

   Any 40-hex-char string (with or without `0x` prefix, optionally prefixed with `0.0.`) passes `isValidEntityId` and produces a non-null `evmAddress`, routing unconditionally to the DB path.

4. `getEntityIdFromEvmAddress` — `rest/service/entityService.js` lines 91:
   ```js

     [Buffer.from(entityId.evmAddress, 'hex')]);
   ```

5. Query executed — `rest/service/entityService.js` lines 22–25:
   ```sql
   select id from entity
   where deleted <> true and evm_address = $1
   ```

**Root cause:** There is no result cache, no per-IP rate limit, and no authentication requirement. Every syntactically valid EVM address — including ones that do not exist in the database — causes a full round-trip DB query. The REST server's middleware stack (`rest/server.js` lines 68–103) applies only `authHandler`, which passes unauthenticated requests through without any throttle.

**Why existing checks fail:**

- `authHandler` (`rest/server.js` line 86) only sets a custom response limit for authenticated users; unauthenticated requests proceed freely.
- The `statement_timeout` in `rest/dbpool.js` line 15 limits individual query duration but does not bound the number of concurrent connections consumed.
- The pgbouncer configuration (`charts/hedera-mirror/values.yaml` lines 371–373) caps `mirror_rest` at `max_user_connections: 250`. Once all 250 server-side connections are occupied by in-flight EVM address lookups, new legitimate requests queue or time out.
- No caching layer sits in front of `getEntityIdFromEvmAddress`; every request, even for the same address, hits the DB.

### Impact Explanation

Exhausting the 250 pgbouncer server-side connections allocated to `mirror_rest` blocks all REST API queries across every mirror node instance sharing that pgbouncer. Because the pgbouncer is a shared resource (single deployment per cluster per the Helm chart), a sustained flood degrades or denies service to all REST API replicas simultaneously — consistent with the ≥30% network processing node impact threshold. Legitimate users receive connection-timeout errors on all account, NFT, token, and transaction endpoints.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no prior account setup. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`, or a small botnet) can generate thousands of requests per second with randomly generated 40-hex-char addresses. Each address is syntactically valid and unique, bypassing any hypothetical response cache. The endpoint is publicly documented in the OpenAPI spec and requires no authentication.

### Recommendation

1. **Add per-IP rate limiting** to the REST API middleware (e.g., `express-rate-limit`) before route handlers, targeting path-parameter endpoints specifically.
2. **Cache EVM address → entity ID lookups** (short TTL, e.g., 30s) in an in-process LRU cache to absorb repeated or near-repeated lookups.
3. **Validate EVM address existence before DB lookup** is not sufficient alone, but combining a short-circuit 404 cache for recently-missed addresses reduces DB load.
4. **Increase pgbouncer `max_user_connections`** or deploy per-instance pgbouncer to limit blast radius.
5. Consider **deploying an API gateway** (e.g., Kong, Nginx rate-limit module) in front of the REST service to enforce global RPS limits per IP.

### Proof of Concept

```bash
# Generate unique random EVM addresses and flood the endpoint
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ADDR}/nfts" &
done
wait
```

Each request passes `EntityId.isValidEntityId()`, sets `evmAddress` to the random hex string, and triggers `entityFromEvmAddressQuery` against the DB. With sufficient concurrency (e.g., 500+ parallel requests), the pgbouncer `max_user_connections: 250` limit for `mirror_rest` is saturated, causing all subsequent REST API requests — from any client — to receive connection timeout errors until the flood subsides.   async getEntityIdFromEvmAddress(entityId, requireResult = true) {
```javascript
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
```

```javascript
  async getEncodedId(entityIdString, requireResult = true, paramName = filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS) {
    try {
      if (EntityId.isValidEntityId(entityIdString)) {
        const entityId = EntityId.parseString(entityIdString, {paramName});
        return entityId.evmAddress === null
          ? entityId.getEncodedId()
          : await this.getEntityIdFromEvmAddress(entityId, requireResult);
      } else if (AccountAlias.isValid(entityIdString)) {
        return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
      }
    } catch (ex) {
      if (ex instanceof InvalidArgumentError) {
        throw InvalidArgumentError.forParams(paramName);
      }
      // rethrow
      throw ex;
    }

    throw InvalidArgumentError.forParams(paramName);
  }
```
