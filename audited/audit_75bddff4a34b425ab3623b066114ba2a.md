### Title
Shared `applicationTaskExecutor` with Unauthenticated gRPC Reflection Service Enables Executor Saturation

### Summary
`grpcServerConfigurer()` assigns the single shared `applicationTaskExecutor` to the Netty gRPC server, meaning all service types — ConsensusService, NetworkService, and the publicly-exposed Reflection service — compete for the same thread pool. The only concurrency control is `maxConcurrentCallsPerConnection=5`, which is a per-connection limit only. An unprivileged attacker can open an unbounded number of connections and flood the shared executor with reflection requests, starving legitimate service calls and driving CPU consumption above 30%.

### Finding Description
**Code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35.

```java
serverBuilder.executor(applicationTaskExecutor);                          // line 32 — single shared executor
serverBuilder.maxConcurrentCallsPerConnection(                            // line 33 — per-connection only
    nettyProperties.getMaxConcurrentCallsPerConnection());                //           default = 5
```

`applicationTaskExecutor` is Spring Boot's default `ThreadPoolTaskExecutor` (core=8, max=200, unbounded queue). It is injected as the sole executor for the entire Netty gRPC server, so every RPC — including `grpc.reflection.v1alpha.ServerReflection` and `grpc.reflection.v1.ServerReflection` — dispatches tasks into this pool.

The reflection service is explicitly exposed in the production ingress/gateway configuration (`charts/hedera-mirror-grpc/values.yaml`, lines 93–95):
```yaml
- method:
    service: grpc.reflection.v1alpha.ServerReflection
- method:
    service: grpc.reflection.v1.ServerReflection
```

No gRPC-layer rate limiter, no per-IP connection cap, and no authentication interceptor exist in the production `grpc/` module. The only interceptor found (`grpc/src/test/java/.../GrpcInterceptor.java`) is test-only and only sets endpoint context — it performs no access control.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no connection-per-IP limit).
2. Each connection sends 5 concurrent `ServerReflection/ServerReflectionInfo` streaming calls (the per-connection limit).
3. With N=200 connections → 1,000 concurrent reflection tasks queued in the shared executor.
4. The executor's 200 threads are fully occupied serializing protobuf file descriptors in a tight loop.
5. Legitimate `subscribeTopic` and `getNodes` calls queue behind the reflection flood, and CPU spins at high utilization.

**Why the existing check fails:** `maxConcurrentCallsPerConnection=5` (NettyProperties line 14) is enforced by Netty per TCP connection. It does not bound the total number of concurrent calls across all connections, so it provides no protection against a multi-connection flood.

### Impact Explanation
The shared executor is the single chokepoint for all gRPC processing. Saturating it with reflection calls directly degrades ConsensusService and NetworkService availability — topic subscriptions stall and node-address queries time out. CPU consumption rises proportionally to the number of attacker connections; with 200 connections (each at the 5-call limit) the full 200-thread pool is consumed by reflection work, representing a complete executor starvation. This satisfies the 30%+ CPU increase threshold without any subscription-based calls.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond standard gRPC reflection (widely documented and tooled via `grpcurl`), and no brute-force. The reflection endpoints are publicly routed. A single attacker machine with a modest number of persistent TCP connections can sustain the flood indefinitely. The attack is repeatable and scriptable.

### Recommendation
1. **Isolate the reflection executor**: Configure a separate, bounded `Executor` for the reflection service rather than sharing `applicationTaskExecutor` with business-critical services.
2. **Add a gRPC-layer rate limiter**: Implement a `ServerInterceptor` (production, not test-only) that applies per-IP or global rate limiting using a token-bucket (e.g., Bucket4j, as already used in `web3/`).
3. **Restrict or disable reflection in production**: If reflection is only needed for tooling/debugging, gate it behind an IP allowlist at the gateway level or disable it entirely via `spring.grpc.server.reflection.enabled=false`.
4. **Add a global `maxConcurrentCalls` limit**: Use `NettyServerBuilder.maxConcurrentCallsPerConnection` in conjunction with a server-wide connection limit (e.g., `maxConnectionAge`, `maxConnectionIdle`) to bound total resource consumption.

### Proof of Concept
```bash
# Open 200 parallel connections, each streaming reflection requests
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"list_services":""}' \
    <mirror-node-host>:5600 \
    grpc.reflection.v1alpha.ServerReflection/ServerReflectionInfo &
done
wait
# Monitor CPU on the mirror node: expect >30% increase vs. baseline
# Legitimate subscribeTopic calls will queue and time out during the flood
```
