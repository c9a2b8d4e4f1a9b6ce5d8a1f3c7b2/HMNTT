### Title
Unauthenticated Sequential Dual-DB-Query Chain in `getContractActions()` Enables Connection Pool Exhaustion DoS

### Summary
The `GET /api/v1/contracts/results/:transactionIdOrHash/actions` endpoint is publicly accessible with no rate limiting and executes two sequential database queries per request when a valid Ethereum hash is supplied. Under concurrent load, this dual-query chain holds connection pool slots for the combined duration of both queries, allowing an unprivileged attacker to exhaust the finite connection pool and deny service to all other users of the REST API.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 22 registers the route with no authentication or rate-limiting middleware: [1](#0-0) 

`rest/server.js` shows the full middleware stack applied before route handlers — it contains `authHandler` (optional basic-auth, no-op for unauthenticated requests), logging, metrics, and optional cache check, but **zero rate-limiting middleware**: [2](#0-1) 

Inside `getContractActions()` in `rest/controllers/contractController.js`, when `transactionIdOrHash` is a valid 32-byte Ethereum hash, two **sequential** (not parallel) DB queries are issued:

- **Query 1** (line 1249): `ContractService.getContractTransactionDetailsByHash(...)` — acquires a pool connection, executes `ethereumTransactionsByHashQuery`, holds the connection until the result returns.
- **Query 2** (line 1263): `ContractService.getContractActionsByConsensusTimestamp(...)` — acquires a second pool connection (or the same one after release), executes the actions query. [3](#0-2) 

The connection pool is a single shared `pg.Pool` with a hard cap of `config.db.pool.maxConnections`: [4](#0-3) 

**Root cause:** No rate limiting exists at the application layer for this endpoint. The two sequential queries mean each concurrent request occupies a pool slot for the combined wall-clock time of both round-trips. With `N` concurrent requests where `N ≥ maxConnections`, all pool slots are occupied simultaneously, and any further request that needs a DB connection will block until `connectionTimeoutMillis` elapses, then fail — this is the DoS condition.

**Why existing checks are insufficient:**

- `statement_timeout` limits individual query execution time but does not prevent `N` concurrent requests from each holding a slot simultaneously.
- `connectionTimeoutMillis` causes waiting requests to error out — this is the observable DoS, not a mitigation.
- `authHandler` is a no-op for unauthenticated requests (returns immediately without blocking): [5](#0-4) 
- The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` Java module, not in the REST Node.js service: [6](#0-5) 

### Impact Explanation
When the pool is exhausted, every other REST API endpoint that requires a DB query (accounts, transactions, tokens, etc.) also fails with connection timeout errors. The blast radius is the entire REST API, not just the contracts endpoint. The service becomes unavailable to all legitimate users for as long as the attacker sustains the flood. This is a non-network DoS: it does not require saturating bandwidth — only enough concurrent HTTP connections to fill the pool.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. Any attacker who knows a single valid Ethereum transaction hash (trivially obtained from a public block explorer) can trigger the chain. The attack is repeatable and scriptable with standard tools (`ab`, `wrk`, `hey`). The required concurrency level equals `maxConnections` (default pool size), which is a small integer in typical deployments.

### Recommendation
1. **Add application-level rate limiting** to the REST API (e.g., `express-rate-limit`) scoped per IP, applied globally or specifically to the contracts routes in `rest/server.js` before route registration.
2. **Parallelize the two queries** where possible (they are independent when the hash lookup succeeds) to reduce the total connection-hold time per request.
3. **Set a low per-IP concurrency limit** at the ingress/proxy layer (pgbouncer already has `max_user_client_connections: 1000` for `mirror_rest` but no per-IP cap).
4. Consider a **request queue with backpressure** so that pool exhaustion returns `503` immediately rather than blocking until `connectionTimeoutMillis`.

### Proof of Concept
```bash
# Obtain any valid Ethereum tx hash from a public explorer, e.g.:
HASH="0xabc123...def456"  # 32-byte hex

# Fire maxConnections+N concurrent requests (adjust -c to exceed pool size)
wrk -t8 -c50 -d30s \
  "https://<mirror-node-host>/api/v1/contracts/results/${HASH}/actions"

# Expected result:
# - During attack: legitimate requests to /api/v1/accounts, /api/v1/transactions, etc.
#   begin returning 500 errors (DB connection timeout)
# - After attack stops: service recovers automatically
# - No authentication, no special headers required
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
