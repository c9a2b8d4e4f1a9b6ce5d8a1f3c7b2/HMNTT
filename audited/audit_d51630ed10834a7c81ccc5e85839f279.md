### Title
Per-Opcode `log.info()` Amplification via Tight Opcode Loop When Tracing Enabled

### Summary
When `hiero.mirror.web3.evm.trace.enabled=true` is configured, `MirrorOperationActionTracer.tracePostExecution()` emits a `log.info()` call for every single EVM opcode executed. An unprivileged external user can submit a single contract call containing a tight JUMPDEST loop at the maximum gas limit (15,000,000), triggering up to ~1,250,000 `log.info()` invocations per request — each performing multiple hex string conversions — with no per-opcode logging rate limit in place.

### Finding Description

**Exact code path:**

`MirrorOperationActionTracer.tracePostExecution()` at lines 51–86:

```java
if (!traceProperties.isEnabled()) {
    return;                          // only guard: a single boolean flag
}
if (traceProperties.stateFilterCheck(frame.getState())) {
    return;                          // skipped when status set is empty (default)
}
// contractFilterCheck skipped when contract set is empty (default)
log.info(
    "type={} operation={}, callDepth={}, contract={}, sender={}, recipient={}, " +
    "remainingGas={}, revertReason={}, input={}, output={}, return={}",
    ...
    frame.getContractAddress().toShortHexString(),
    frame.getSenderAddress().toShortHexString(),
    frame.getRecipientAddress().toShortHexString(),
    ...
    frame.getInputData().toShortHexString(),
    frame.getOutputData().toShortHexString(),
    frame.getReturnData().toShortHexString());
``` [1](#0-0) 

**Root cause and failed assumption:**

`TraceProperties.stateFilterCheck()` and `contractFilterCheck()` only filter when their respective sets are non-empty. Both default to empty `HashSet`, so with default filter configuration all opcodes pass through to `log.info()`. [2](#0-1) 

The gas throttle in `ThrottleManagerImpl.throttle()` consumes `scaleGas(request.getGas()) = gas / 10_000` tokens from the bucket. For a 15M gas request this is only 1,500 tokens. The bucket holds 750,000 tokens/second (from `gasPerSecond = 7,500,000,000 / 10,000`). The throttle is calibrated to EVM computation cost, not to the number of side-effect log calls generated per opcode. [3](#0-2) [4](#0-3) 

**Exploit flow:**

A tight loop in EVM bytecode:
```
JUMPDEST   ; 1 gas
PUSH1 0x00 ; 3 gas
JUMP       ; 8 gas
; total: 12 gas per iteration
```
With `maxGasLimit = 15,000,000`: `15,000,000 / 12 ≈ 1,250,000` iterations, each triggering one `log.info()` with ~10 string formatting operations (multiple `toShortHexString()` / `toHexString()` calls on Bytes objects). [5](#0-4) 

**Why existing checks fail:**

| Check | Why insufficient |
|---|---|
| `gasLimitBucket` | Throttles by declared gas value, not opcode count; 15M gas = 1,500 tokens, well within 750,000/s budget |
| `requestsPerSecond = 500` | Limits HTTP requests, not per-opcode log calls; 500 req/s × 1.25M logs/req = 625M log calls/s |
| `requestTimeout = 10,000ms` | Limits wall-clock time but 1.25M log.info() calls still execute within that window |
| `stateFilterCheck` / `contractFilterCheck` | Both no-ops with default empty sets | [6](#0-5) 

### Impact Explanation

A single HTTP request to `POST /api/v1/contracts/call` with a tight JUMPDEST loop and `gas=15000000` generates ~1,250,000 `log.info()` calls, each performing multiple heap allocations and hex string conversions on `Bytes` objects. This saturates the logging I/O pipeline and drives CPU usage well above the 30% threshold from a single unauthenticated request. The effect is repeatable and requires no privileged access — only that the operator has enabled tracing, which is a documented and expected operational configuration.

### Likelihood Explanation

The `hiero.mirror.web3.evm.trace.enabled` flag is documented in `docs/configuration.md` as a supported operator feature for debugging. Any operator enabling it for production diagnostics immediately exposes the node to this amplification. The attacker requires no credentials, no on-chain funds (the mirror node simulates calls without requiring payment), and no special knowledge beyond the public API. The attack is trivially repeatable. [7](#0-6) 

### Recommendation

1. **Add a per-request opcode trace count cap**: Introduce a `maxTracedOpcodes` property in `TraceProperties`. In `tracePostExecution()`, maintain a per-request counter (stored in `ContractCallContext`) and stop logging once the cap is reached.
2. **Throttle log emission rate**: Use a token-bucket or sampling strategy inside `tracePostExecution()` (e.g., log every Nth opcode, or use a time-based rate limiter) rather than logging unconditionally on every invocation.
3. **Warn in documentation**: Explicitly document that `trace.enabled=true` should never be used in production without also configuring `trace.contract` or `trace.status` filters to bound the logging volume.

### Proof of Concept

**Preconditions:**
- Mirror node running with `hiero.mirror.web3.evm.trace.enabled=true` and default empty `contract`/`status` filter sets.
- A deployed contract containing the tight loop bytecode (or use `eth_call` with raw bytecode via `data` field and empty `to`).

**Steps:**

1. Deploy (or reference) a contract whose bytecode is:
   ```
   5b 6000 56  ; JUMPDEST PUSH1 0x00 JUMP (infinite loop, 12 gas/iter)
   ```

2. Submit:
   ```bash
   curl -X POST http://<mirror-node>/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     -d '{
       "to": "<deployed_contract_address>",
       "gas": 15000000,
       "estimate": false
     }'
   ```

3. **Observed result**: The mirror node emits ~1,250,000 `log.info()` lines of the form `type=MESSAGE_CALL operation=JUMPDEST callDepth=0 contract=... sender=... recipient=... remainingGas=... revertReason= input=... output=... return=...` before the gas is exhausted, causing a measurable spike in CPU and log I/O that exceeds the 30% resource increase threshold.
