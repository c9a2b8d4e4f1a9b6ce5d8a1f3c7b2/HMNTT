### Title
Unauthenticated Full-Table Scan DoS via `GET /accounts?balance=true` Without `account.id` Filter

### Summary
The `getAccounts()` handler in `rest/accounts.js` accepts unauthenticated requests with `balance=true` (the default) and no `account.id` filter, causing `getEntityBalanceQuery()` to execute a full scan of the `entity` table joined against `entity_stake`, plus a CTE that scans all of `token_account where associated is true`. With a default DB connection pool of only 10 connections and a 20-second statement timeout, an attacker sending ~10 concurrent requests can exhaust the pool and cause connection timeout errors for all legitimate users.

### Finding Description

**Code path:**

`rest/accounts.js` → `getAccounts()` (lines 335–388) → `getAccountQuery()` (lines 254–304) → `getEntityBalanceQuery()` (lines 133–237)

**Root cause — three compounding weaknesses:**

1. **No rate limiting on the REST API.** `rest/server.js` registers the route at line 101 with no throttle middleware. The rate-limiting code found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) belongs exclusively to the `web3` Java service and does not apply here.

2. **Expensive query with no mandatory filter.** When `balance=true` (the default, enforced at line 323) and no `account.id` is supplied, `getEntityBalanceQuery()` builds:
   ```sql
   with latest_token_balance as (
     select account_id, balance, token_id
     from token_account          -- full scan
     where associated is true
   )
   select ...
   from entity as e              -- full scan, type-filtered
   left join entity_stake as es on es.id = e.id   -- no extra predicate
   where e.type in ('ACCOUNT','CONTRACT') and ...
   order by e.id asc
   limit $N
   ```
   The `getEntityStakeAccountCondition()` helper (line 307–308) returns an empty string when `entityAccountQuery.query` is empty, so the `entity_stake` join has no selective predicate. The `LIMIT` clause caps rows *returned* but does not prevent the planner from scanning the full `entity` and `token_account` tables to satisfy the join and ordering.

3. **Tiny default connection pool.** `rest/dbpool.js` (line 14) sets `max: config.db.pool.maxConnections`, whose documented default is **10** connections. The `statement_timeout` is 20 000 ms (20 s). During those 20 seconds each slow query holds one pool slot.

**Why existing checks fail:**

- The `limit` parameter is capped at 100 rows (default 25), but this only limits the result set, not the scan cost.
- The `statement_timeout` of 20 s is a partial mitigation: it eventually kills the query, but during those 20 s the connection is held, and the attacker only needs to keep re-issuing requests faster than the timeout.
- Redis response caching is **disabled by default** (`hiero.mirror.rest.cache.response.enabled = false`).
- There is no per-IP, per-user, or global request-rate limit on this endpoint.

### Impact Explanation
An attacker who saturates the 10-slot pool causes every subsequent legitimate request to wait up to `connectionTimeoutMillis` (20 s default) before receiving a connection-timeout error. This renders the entire REST API unavailable for the duration of the attack. Because the attack targets a shared infrastructure component (the DB pool), all endpoints — not just `/accounts` — are affected. The impact is classified as availability degradation (DoS) with no direct economic loss to network participants, consistent with the "griefing" severity tier.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero cost. Any public internet client can issue `GET /api/v1/accounts` (no parameters needed — `balance=true` is the default). Maintaining ~10 concurrent HTTP connections is trivial with any scripting tool (`curl`, `ab`, `wrk`). The attack is repeatable indefinitely and self-sustaining as long as the attacker keeps sending requests.

### Recommendation
1. **Add a per-IP (or global) rate limiter** on the `/accounts` route in `rest/server.js`, e.g. using `express-rate-limit` or an upstream API gateway.
2. **Require at least one selective filter** (e.g. `account.id`) when `balance=true` and the query would otherwise scan the full table, or add a query-cost guard.
3. **Increase the default pool size** or add a request-queue depth limit so pool exhaustion does not immediately translate to user-visible errors.
4. **Enable Redis response caching** by default for the `/accounts` list endpoint to absorb repeated identical queries.
5. **Add a DB-level work_mem or query-timeout guard** specifically for unfiltered list queries.

### Proof of Concept
```bash
# Send 15 concurrent unfiltered requests (balance=true is the default)
# No authentication, no account.id filter
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts" &
done
wait

# Legitimate request issued concurrently will receive a connection-timeout
# error after connectionTimeoutMillis (default 20 000 ms) because all 10
# pool slots are occupied by the expensive full-table-scan queries.
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
# Expected: hangs ~20 s then returns 500 / connection pool exhausted error
```
