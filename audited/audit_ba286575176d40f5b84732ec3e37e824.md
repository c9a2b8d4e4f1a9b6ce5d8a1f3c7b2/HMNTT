### Title
Unauthenticated Indefinite gRPC Stream Exhaustion via Unbounded `subscribeTopic` Connections

### Summary
Any unauthenticated external caller can open an unlimited number of TCP connections to the gRPC port and, on each connection, hold up to 5 indefinite `subscribeTopic` streams (by omitting both `consensusEndTime` and `limit`). Because `grpcServerConfigurer()` sets only a per-connection call cap with no global connection limit, no max stream duration, and no authentication, an attacker can exhaust the shared `applicationTaskExecutor` thread pool and the topic-listener buffer, denying legitimate subscribers access to submitted transaction data.

### Finding Description

**Code path and root cause**

`GrpcConfiguration.java` `grpcServerConfigurer()` configures the Netty server with exactly two settings:

```
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default = 5
``` [1](#0-0) 

`NettyProperties` hard-codes the default at 5 and exposes no other server-level guards: [2](#0-1) 

No `maxConnectionAge`, `maxConnectionIdle`, `keepAliveTimeout`, global concurrent-call cap, per-IP connection limit, or any `ServerInterceptor` for authentication is present anywhere in the gRPC module. The grep across all `grpc/src/main/java/**/*.java` for those identifiers returns zero matches.

The proto contract explicitly documents the indefinite-stream behaviour:

```
// If not set it will receive indefinitely.
.proto.Timestamp consensusEndTime = 3;
// If not set or set to zero it will return messages indefinitely.
uint64 limit = 4;
``` [3](#0-2) 

`ConsensusController.toFilter()` passes `limit=0` and `endTime=null` straight through when the client omits them: [4](#0-3) 

`TopicMessageServiceImpl.subscribeTopic()` applies neither `takeWhile` nor `take` when both fields are absent, so the reactive `Flux` never self-terminates: [5](#0-4) 

`pastEndTime()` returns `Flux.never()` when `endTime == null`, so the `takeUntilOther` guard on the live stream also never fires: [6](#0-5) 

**Why the existing check is insufficient**

`maxConcurrentCallsPerConnection=5` is scoped to a single TCP connection. It prevents one connection from holding more than 5 simultaneous RPCs, but it places no cap on the number of connections an attacker may open. An attacker opening *N* connections holds *5N* indefinite streams with no server-side enforcement stopping them.

### Impact Explanation

Each open stream increments `subscriberCount`, holds a live subscription in the shared `topicListener` (default `maxBufferSize=16384`), and occupies a slot in the shared `applicationTaskExecutor` thread pool. Flooding the server with thousands of indefinite streams exhausts the thread pool and/or the listener buffer. Legitimate subscribers either queue indefinitely or receive `RESOURCE_EXHAUSTED` errors and cannot receive HCS topic messages (submitted transaction data). This is a complete denial-of-service against the mirror node's primary streaming API with no authentication barrier.

### Likelihood Explanation

The attack requires zero privileges, zero credentials, and a trivial gRPC client (e.g., `grpcurl` or any generated stub). The proto documentation itself advertises the indefinite-stream feature, making the attack surface self-documenting. It is fully repeatable and scriptable; a single attacker machine can open thousands of connections. No exploit code beyond a standard gRPC client is needed.

### Recommendation

1. **Set `maxConnectionAge` and `maxConnectionIdle`** in `grpcServerConfigurer()` via `NettyServerBuilder` (e.g., `serverBuilder.maxConnectionAge(1, TimeUnit.HOURS).maxConnectionIdle(5, TimeUnit.MINUTES)`) to bound stream lifetime and reclaim idle connections.
2. **Add a global concurrent-call cap** — Netty gRPC does not expose a single `maxCallsPerServer` knob, but a counting `ServerInterceptor` that rejects new calls above a threshold achieves the same effect.
3. **Add a per-IP connection limit** at the load-balancer or via a Netty `ChannelHandler` before the gRPC layer.
4. **Expose `maxConcurrentCallsPerConnection` alongside new `maxConnectionAge` / `maxConnectionIdle` fields** in `NettyProperties` so operators can tune them without code changes.
5. **Consider requiring authentication** (even a simple shared API key via metadata) for streaming endpoints to raise the cost of abuse.

### Proof of Concept

```bash
# Install grpcurl: https://github.com/fullstorydev/grpcurl
# Replace TOPIC_ID with any valid topic (e.g., 0.0.1234)

# Open 200 connections × 5 streams = 1000 indefinite streams
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1234}}' \
      -proto consensus_service.proto \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Legitimate subscriber now blocked:
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 1234}, "limit": 1}' \
  -proto consensus_service.proto \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected result: hangs or returns RESOURCE_EXHAUSTED / UNAVAILABLE
```
