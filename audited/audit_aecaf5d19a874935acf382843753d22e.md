### Title
Deleted Contract Bytecode Returned in EVM Simulation Due to Missing Entity Deletion Check in `findRuntimeBytecode`

### Summary
`ContractRepository.findRuntimeBytecode()` queries only the `contract` table without joining the `entity` table to verify `deleted = false`. Since the `deleted` column was migrated exclusively to the `entity` table (migration `V1.64.2`), the `contract` table has no deletion flag, so the query unconditionally returns bytecode for logically deleted contracts. Any unprivileged caller can trigger this by issuing an `eth_call` or `eth_estimateGas` against a deleted contract's numeric ID or long-zero EVM address, causing the mirror node's EVM simulation to treat the deleted contract as active and export incorrect execution records.

### Finding Description

**Exact code path:**

`ContractRepository.java` line 17: [1](#0-0) 

The query `select runtime_bytecode from contract where id = :contractId` has no join to `entity` and no `deleted` predicate.

**Root cause — schema split:**

Migration `V1.64.2__merge_contract_entity.sql` moved all lifecycle metadata (including `deleted`) from the `contract` table into the `entity` table and then dropped those columns from `contract`: [2](#0-1) 

The `Contract` domain object confirms no `deleted` field exists: [3](#0-2) 

**Call chain — no deletion guard for contractNum path:**

`ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode` directly after resolving the entity ID, with zero deletion check for the `contractNum` and long-zero EVM address branches: [4](#0-3) [5](#0-4) 

**Asymmetric protection:** The non-long-zero EVM address branch routes through `commonEntityAccessor.getEntityByEvmAddressAndTimestamp()`, which calls `findByEvmAddressAndDeletedIsFalse` — a deletion-aware query: [6](#0-5) 

The `contractNum` and long-zero EVM address branches have no equivalent guard, so they bypass deletion entirely.

**Exploit flow:**
1. Contract `0.0.X` is deleted on-chain → `entity.deleted = true`, but the `contract` row with `runtime_bytecode` remains untouched.
2. Attacker (no privileges required) submits `eth_call` or `eth_estimateGas` referencing contract `0.0.X` by numeric ID.
3. `ContractBytecodeReadableKVState.readFromDataSource()` resolves the numeric ID directly and calls `findRuntimeBytecode(X)`.
4. The query returns the stored bytecode because it only checks `contract.id = X` — no deletion filter.
5. The EVM simulation receives non-null `Bytecode`, treats the address as a live contract, and executes its code.
6. The mirror node exports the simulation result as if the contract were active, diverging from consensus-node behavior (which returns empty bytecode for deleted contracts).

### Impact Explanation
The mirror node's EVM simulation (`eth_call`, `eth_estimateGas`, opcode tracing) returns non-empty bytecode for deleted contracts. This means `EXTCODESIZE` returns a non-zero value, `EXTCODECOPY` copies real code, and direct calls execute the deleted contract's logic. Records exported by the mirror node diverge from what the consensus network would produce, violating the mirror node's core correctness guarantee. Downstream tooling (wallets, dApps, indexers) that rely on mirror node simulation results will receive incorrect state information about deleted contracts.

### Likelihood Explanation
The precondition — a contract being deleted — is a normal, observable on-chain event requiring no attacker involvement. Any unprivileged user who knows a deleted contract's numeric ID (publicly visible on-chain) can immediately trigger the incorrect behavior. The trigger is a standard JSON-RPC call (`eth_call`) requiring no special credentials. The condition is persistent: the `contract` row is never cleaned up after deletion, so the window is indefinite. Repeatability is unlimited.

### Recommendation
Add a join to the `entity` table in `findRuntimeBytecode` to filter out deleted contracts:

```sql
SELECT c.runtime_bytecode
FROM contract c
JOIN entity e ON e.id = c.id
WHERE c.id = :contractId
  AND (e.deleted IS NULL OR e.deleted = FALSE)
```

Alternatively, add a deletion check in `ContractBytecodeReadableKVState.readFromDataSource()` for the `contractNum` and long-zero EVM address branches by calling `commonEntityAccessor.get(entityId, timestamp)` and returning `null` if the entity is absent or deleted — consistent with how the non-long-zero EVM address branch already behaves.

### Proof of Concept
1. Deploy a contract to the Hedera network; note its numeric ID `N`.
2. Delete the contract via `ContractDeleteTransaction` (requires admin key — one-time setup, not attacker action).
3. Confirm deletion: `GET /api/v1/contracts/0.0.N` returns `"deleted": true`.
4. As an unprivileged user, issue:
   ```
   POST /api/v1/contracts/call
   { "to": "0x000000000000000000000000000000000000000N", "data": "0x..." }
   ```
5. Observe: the mirror node executes the call against the deleted contract's bytecode and returns a result, whereas a consensus node would return empty bytecode and treat the address as a plain account.
6. Confirm root cause: query `SELECT runtime_bytecode FROM contract WHERE id = N` returns non-null bytes; query `SELECT deleted FROM entity WHERE id = N` returns `true`.
