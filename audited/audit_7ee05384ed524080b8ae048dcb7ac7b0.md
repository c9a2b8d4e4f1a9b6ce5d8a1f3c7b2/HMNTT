### Title
Unbounded In-Memory Opcode Accumulation in EVM Replay Enables JVM Heap Exhaustion via `getContractOpcodes()`

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint replays a historical EVM transaction and accumulates every executed opcode as a heap-allocated `Opcode` object in an unbounded `ArrayList`. The per-request throttle enforces only a request-rate limit (one token consumed regardless of transaction gas), so a single request referencing a maximum-gas transaction can force the JVM to allocate hundreds of millions of bytes of opcode trace data, exhausting heap and crashing the node.

### Finding Description

**Endpoint entry point** — `OpcodesController.getContractOpcodes()`: [1](#0-0) 

The only guards applied before delegating to `opcodeService.processOpcodeCall()` are:
1. A feature-flag check (`properties.isEnabled()`).
2. A gzip header validation.
3. `throttleManager.throttleOpcodeRequest()`.

**Throttle is request-rate only, not gas-bounded** — `ThrottleManagerImpl.throttleOpcodeRequest()`: [2](#0-1) 

It consumes exactly one token from `opcodeRateLimitBucket` regardless of the gas limit of the replayed transaction. The gas-based bucket used by the regular `throttle()` path is never consulted for opcode requests.

**Unbounded accumulation** — `OpcodeContext`: [3](#0-2) 

`opcodes` is a plain `ArrayList` with no size cap. `addOpcodes()` appends unconditionally.

**Called once per EVM opcode** — `OpcodeActionTracer.tracePostExecution()`: [4](#0-3) 

Every opcode executed during the replay — including every LOG0–LOG4, every PUSH, every JUMPDEST — appends a new `Opcode` object. When `stack=true` (the default) or `memory=true`, each object also carries a full snapshot of the EVM stack/memory at that point.

**Initial capacity hint does not cap growth** — `OpcodeServiceImpl.processOpcodeCall()`: [5](#0-4) 

`gas / 3` is only the initial `ArrayList` capacity hint; the list grows without bound beyond it.

**Root cause / failed assumption**: The design assumes the response size is bounded by gzip compression and that a rate limit is sufficient. Neither is true: gzip compresses the serialised response but the in-memory `List<Opcode>` is fully materialised before serialisation; and the rate limit does not account for the per-request work proportional to gas consumed.

### Impact Explanation
Hedera's maximum transaction gas is 15,000,000. With cheap opcodes (JUMPDEST = 1 gas, ADD = 3 gas) a single transaction can execute ~5–15 million opcodes. With `stack=true` each `Opcode` object holds a `List<String>` of 32-byte hex stack items. A conservative estimate of 500 bytes per entry × 5 million entries = ~2.5 GB of heap per request. A single such request can trigger an `OutOfMemoryError`, crashing the JVM and taking the entire web3 mirror-node service offline. Concurrent requests multiply the effect. This satisfies the stated severity threshold of shutting down ≥30% of network processing nodes without brute force.

### Likelihood Explanation
**Precondition**: The opcode tracer must be enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`; it defaults to `false`). When enabled for production debugging or public use, any unauthenticated caller who knows (or can enumerate) the hash of any high-gas historical transaction can trigger the attack. No special privileges, no on-chain action, and no brute force are required — only a valid transaction hash and the `Accept-Encoding: gzip` header. The attack is repeatable at the configured request rate.

### Recommendation
1. **Add a hard cap on the opcode list**: In `OpcodeContext.addOpcodes()`, throw or silently truncate once `opcodes.size()` exceeds a configurable maximum (e.g., 100,000 entries).
2. **Apply gas-based throttling to opcode requests**: In `ThrottleManagerImpl.throttleOpcodeRequest()`, also consume tokens from the gas bucket proportional to the gas limit of the transaction being replayed (requires passing gas to the method).
3. **Add a configurable gas-limit ceiling for opcode replay**: Reject requests whose stored `gas_limit` exceeds a threshold (e.g., 2,000,000) before starting the replay.
4. **Stream/paginate the response**: Instead of materialising the full list in heap, stream opcode entries directly to the HTTP response writer to avoid the full in-memory accumulation.

### Proof of Concept
1. Ensure `hiero.mirror.web3.opcode.tracer.enabled=true` on the target mirror node.
2. Identify (or submit) a Hedera contract transaction that executes a tight loop emitting LOG opcodes up to the gas limit (e.g., a Solidity loop: `for(uint i=0;i<N;i++) emit Evt(i);` with gas limit = 15,000,000).
3. Obtain the transaction hash from the mirror node REST API.
4. Send:
   ```
   GET /api/v1/contracts/results/<txHash>/opcodes?stack=true&memory=true
   Accept-Encoding: gzip
   ```
5. Observe the JVM heap grow unboundedly during replay; the node throws `OutOfMemoryError` and crashes before returning a response.
6. Repeat at the configured opcode rate limit to sustain the outage.

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
