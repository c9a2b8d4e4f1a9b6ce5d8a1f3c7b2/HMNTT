### Title
Mirror Node Stores Duplicate Chunk Numbers, Corrupting HCS Chunked Message Reassembly

### Summary
`doUpdateTransaction()` in `ConsensusSubmitMessageTransactionHandler.java` blindly persists every successful `CONSENSUSSUBMITMESSAGE` transaction into `topic_message` with no deduplication check on `(initial_transaction_id, chunk_num)`. Because the Hedera consensus network validates only that `chunkNum` is in range `[1, total]` — not that a given `(initialTransactionId, chunkNum)` pair is unique — an unprivileged attacker can submit two transactions carrying the same `chunkNum` for the same `initialTransactionId`, both of which succeed at the consensus layer. The mirror node stores both rows, permanently corrupting the chunk sequence for any downstream consumer that relies on the mirror node to reassemble the fragmented message.

### Finding Description

**Exact code path:**

`importer/src/main/java/org/hiero/mirror/importer/parser/record/transactionhandler/ConsensusSubmitMessageTransactionHandler.java`, `doUpdateTransaction()`, lines 52–93.

```java
// Lines 52-55: only guard is "topics enabled" and "transaction succeeded"
if (!entityProperties.getPersist().isTopics() || !recordItem.isSuccessful()) {
    return;
}
// Lines 76-85: chunk fields are copied verbatim from the transaction body
if (transactionBody.hasChunkInfo()) {
    ConsensusMessageChunkInfo chunkInfo = transactionBody.getChunkInfo();
    topicMessage.setChunkNum(chunkInfo.getNumber());      // no uniqueness check
    topicMessage.setChunkTotal(chunkInfo.getTotal());
    if (chunkInfo.hasInitialTransactionID()) {
        topicMessage.setInitialTransactionId(
                chunkInfo.getInitialTransactionID().toByteArray());
    }
}
// Line 93: stored unconditionally
entityListener.onTopicMessage(topicMessage);
```

**Root cause / failed assumption:**

The handler assumes the consensus network enforces `(initialTransactionId, chunkNum)` uniqueness. It does not. The network only rejects chunk numbers outside `[1, total]` (`INVALID_CHUNK_NUMBER`, result code 163) and mismatched `initialTransactionID` for chunk 1 (`INVALID_CHUNK_TRANSACTION_ID`, result code 164). There is no network-level deduplication of `(initialTransactionId, chunkNum)` pairs across separate transactions.

**Database layer — no constraint either:**

`topic_message` is keyed solely on `consensus_timestamp` (each transaction has a unique timestamp). There is no unique constraint on `(initial_transaction_id, chunk_num)` anywhere in the schema:

```sql
-- V2.0.0__create_tables.sql lines 637-650
create table if not exists topic_message (
    chunk_num              integer,          -- nullable, no unique constraint
    initial_transaction_id bytea,            -- nullable, no unique constraint
    consensus_timestamp    bigint   not null, -- sole key
    ...
) partition by range (consensus_timestamp);
```

Indexes on `topic_message` are `(topic_id, consensus_timestamp)` and `(topic_id, sequence_number)` — neither prevents duplicate `(initial_transaction_id, chunk_num)` rows.

**Exploit flow:**

1. Attacker observes chunk 1 of N being submitted to a topic (learns `initialTransactionId = TX1`).
2. Attacker submits a legitimate-looking chunk 2 of N with `initialTransactionId = TX1` → consensus returns `SUCCESS`.
3. Attacker immediately submits an identical or different-content chunk 2 of N with the same `initialTransactionId = TX1` → consensus again returns `SUCCESS` (no duplicate check).
4. Mirror node calls `doUpdateTransaction()` for both; both pass `isSuccessful()`.
5. Two `TopicMessage` rows are inserted with `chunk_num = 2` and `initial_transaction_id = TX1` but different `consensus_timestamp` values.
6. Any consumer querying the mirror node for chunks of `TX1` receives two rows with `chunk_num = 2` and no row with `chunk_num = 3` (if the attacker raced ahead of the legitimate chunk 3), permanently breaking reassembly.

### Impact Explanation

Applications that use HCS chunked messages to coordinate fund-release or other critical state transitions and rely on the mirror node's REST/gRPC API for reassembly will receive an ambiguous or incomplete chunk set. Because the mirror node is append-only and there is no mechanism to retract or supersede a stored `TopicMessage`, the corruption is permanent for that `initialTransactionId`. The legitimate message can never be cleanly reassembled from the mirror node's data. Severity is high for any protocol that treats mirror-node chunk data as authoritative.

### Likelihood Explanation

**Precondition:** The target topic must have no `submitKey` (or the attacker must hold it). Topics used for open/public messaging have no `submitKey` by design. The attacker needs only a funded Hedera account (small HBAR amount for transaction fees) and knowledge of the `initialTransactionId` of an in-flight chunked message — observable from the public ledger in real time. The attack is repeatable and requires no privileged access.

### Recommendation

1. **In `doUpdateTransaction()`**: Before calling `entityListener.onTopicMessage()`, query whether a `TopicMessage` with the same `(initial_transaction_id, chunk_num)` already exists for the same topic. If a duplicate is detected, log a recoverable error and skip persistence (or apply a last-writer-wins policy with explicit logging).
2. **At the database layer**: Add a partial unique index:
   ```sql
   create unique index topic_message__initial_tx_chunk
       on topic_message (initial_transaction_id, chunk_num)
       where initial_transaction_id is not null;
   ```
   This enforces the invariant at the storage level and prevents silent corruption even if the application layer misses a case.
3. **Consider idempotency**: If the design intent is that each `(initialTransactionId, chunkNum)` maps to exactly one message fragment, document and enforce that invariant explicitly rather than relying on the consensus network.

  T1 | chunk_num=1 | (legitimate)
  T2 | chunk_num=2 | (legitimate)

  T4 | chunk_num=3 | (legitimate)
