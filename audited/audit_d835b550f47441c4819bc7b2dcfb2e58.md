### Title
Gas Bucket Under-Restoration on Pre-EVM Failures When `gasLimitRefundPercent < 100`

### Summary
When `transactionExecutionService.execute()` throws a `MirrorEvmTransactionException` without an attached result (i.e., the call never reached the EVM — e.g., sender not found), `result` remains `null` in `doProcessCall()`. The `finally` block then calls `restoreGasToBucket(null, gasLimit)`, which restores only `gasLimitRefundPercent%` of the consumed gas. If `gasLimitRefundPercent` is configured below 100, each such pre-EVM failure permanently under-restores the gas bucket, and a sufficiently high request rate can deplete it faster than it refills, throttling legitimate users.

### Finding Description

**Code path:**

In `doProcessCall()`, `result` is initialized to `null`: [1](#0-0) 

The `MirrorEvmTransactionException` catch block sets `result = e.getResult()`. When the exception was constructed without a result (as in pre-EVM failures), `e.getResult()` returns `null`, leaving `result` still `null`: [2](#0-1) 

The `finally` block unconditionally calls `restoreGasToBucket(result, params.getGas())` with `result == null`: [3](#0-2) 

Inside `restoreGasToBucket`, the `result == null` branch restores only `gasLimitRefundPercent%` of `gasLimit`: [4](#0-3) 

**Trigger path — unprivileged user:**

In `TransactionExecutionService.getSenderAccountIDAsNum()`, if an alias address resolves to `null` in `aliasesReadableKVState`, `throwPayerAccountNotFoundException()` is called: [5](#0-4) 

This throws `MirrorEvmTransactionException(PAYER_ACCOUNT_NOT_FOUND, message, StringUtils.EMPTY)` — **no result attached** — before the EVM is ever invoked: [6](#0-5) 

Similarly, `handleFailedResult()` throws a no-result exception when the call did not reach the EVM: [7](#0-6) 

**Root cause:** The code conflates two distinct failure modes under the same `result == null` branch: (a) EVM-level failures where gas was actually consumed, and (b) pre-EVM failures where no EVM gas was consumed at all. Both receive the same partial restoration of `gasLimitRefundPercent%`, even though pre-EVM failures should logically restore 100% of the gas limit since no computation occurred.

**Failed assumption:** The comment at line 141–142 states "If the transaction fails, gasUsed is equal to gasLimit" — this is only true for EVM-level failures. For `result == null` (pre-EVM), no gas was consumed by the EVM, yet the same reduced restoration is applied.

### Impact Explanation

With `gasLimitRefundPercent` set below 100 (a valid, documented operator configuration): [8](#0-7) 

Each pre-EVM failure with gas limit `G` permanently removes `G × (100 − gasLimitRefundPercent) / 100` tokens from the bucket until the bucket's time-based refill recovers them. If the attacker's request rate × per-request gas loss exceeds the bucket's refill rate, the bucket is kept depleted, causing all legitimate requests to receive `GAS_PER_SECOND_LIMIT_EXCEEDED` throttle errors. The gas bucket uses `SynchronizationStrategy.SYNCHRONIZED`, so depletion is globally visible across all threads: [9](#0-8) 

### Likelihood Explanation

**Preconditions:**
1. `gasLimitRefundPercent` must be configured below 100 (non-default; default is 100, which makes `gasLimitToRestoreBaseline == gasLimit` and causes no net loss).
2. The attacker must be able to send requests at a rate where `rate × gasLimit × (1 − refundPercent/100) > gasPerSecond`.

**Feasibility:** Triggering the pre-EVM failure is trivial — any request with a non-existent alias sender address reliably produces `PAYER_ACCOUNT_NOT_FOUND` before EVM execution. No special privileges are required. The attacker only needs to craft requests with a random/non-existent EVM alias as the sender. With `gasLimitRefundPercent = 0` and default `requestsPerSecond = 500` and `maxGasLimit = 15,000,000`, the attacker can consume `500 × 15,000,000 = 7.5B` gas/s, exactly matching the default `gasPerSecond = 7.5B`, meaning the bucket can be kept at zero. Any configuration with lower `gasPerSecond` or higher `requestsPerSecond` makes this easier. [10](#0-9) 

### Recommendation

Distinguish pre-EVM failures from EVM failures in `restoreGasToBucket`. When `result == null`, the EVM was never invoked and no gas was consumed; the full `gasLimit` should be restored regardless of `gasLimitRefundPercent`:

```java
private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
    if (result == null) {
        // Pre-EVM failure: no gas was consumed by the EVM, restore everything
        throttleManager.restore(gasLimit);
        return;
    }
    final var gasLimitToRestoreBaseline =
        (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
    if (!result.isSuccessful() && gasLimit == result.gasUsed()) {
        throttleManager.restore(gasLimitToRestoreBaseline);
    } else {
        final var gasRemaining = gasLimit - result.gasUsed();
        throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
    }
}
```

### Proof of Concept

**Precondition:** Deploy mirror-node with `hiero.mirror.web3.throttle.gasLimitRefundPercent=50` (or any value < 100).

**Steps:**
1. Identify a valid contract address on the mirror node.
2. Craft a `eth_call` / contract call request with:
   - `from`: a random 20-byte alias address that does not exist in the mirror node state (e.g., `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`)
   - `to`: any valid contract address
   - `gas`: `15000000` (maximum allowed)
3. Send this request repeatedly in a loop at the maximum allowed rate (up to `requestsPerSecond`).
4. Each request: consumes `15,000,000` gas from the bucket, restores only `7,500,000` (50%), net loss = `7,500,000` gas per request.
5. Observe that after enough requests, legitimate `eth_call` requests from valid users begin receiving HTTP 429 / `Gas per second rate limit exceeded` errors.
6. The bucket refills over time, but while the attacker maintains the request rate, legitimate traffic is continuously throttled.
