### Title
Thundering Herd via Unjittered `delaySubscription` in `PollingTopicListener` Exhausts DB Connection Pool

### Summary
`PollingTopicListener.listen()` applies a fixed `delaySubscription(interval, scheduler)` with no randomization on the initial delay. An unprivileged attacker who opens many gRPC connections simultaneously causes all first polls to fire at exactly the same moment (T + 500ms), producing a synchronized burst of blocking JPA queries that exhausts the HikariCP connection pool and starves legitimate subscribers.

### Finding Description
**Exact location**: `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 38–43.

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // line 39 — fixed, no jitter
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)                             // jitter only on repeat, not on first fire
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

**Root cause**: The `jitter(0.1)` is attached exclusively to `RepeatSpec` (subsequent polls). The initial `delaySubscription(interval, scheduler)` uses the raw `interval` value (default 500 ms) with no randomization. Every subscription opened at time T fires its first `poll()` call at exactly T + 500 ms.

**Exploit flow**:
1. Attacker opens N gRPC connections simultaneously (no per-IP or total-connection limit enforced by the gRPC layer — `maxConcurrentCallsPerConnection = 5` is per-connection only, not a global cap).
2. Each connection opens 5 `subscribeTopic` streams → N × 5 independent `PollingTopicListener` instances, each with its own `delaySubscription(500ms)`.
3. At T + 500 ms all N × 5 subscriptions call `poll()` → `topicMessageRepository.findByFilter()` → `typedQuery.getResultList()` (blocking JPA, holds a HikariCP connection for up to `statementTimeout` = 10 s).
4. HikariCP default pool size is 10 connections. With N × 5 >> 10 concurrent blocking queries, the pool is exhausted; legitimate queries queue and time out.

**Why existing checks fail**:
- `jitter(0.1)` on `repeatWhen` spreads out polls #2, #3, … but does nothing for poll #1.
- `maxConcurrentCallsPerConnection = 5` limits calls per TCP connection but imposes no ceiling on the number of connections an attacker may open.
- No per-IP rate limit, no global subscription count limit, and no admission control exists in `PollingTopicListener` or `TopicMessageServiceImpl`.
- `Schedulers.boundedElastic()` queues excess tasks rather than rejecting them, so all queries eventually execute — they just pile up.

### Impact Explanation
DB connection pool exhaustion causes `HikariPool-1 - Connection is not available, request timed out` errors for all concurrent users of the gRPC service. Legitimate subscribers receive no messages or stream errors. The `statementTimeout` of 10 s means each wave of attacker queries holds the pool saturated for up to 10 s; the attacker can sustain this indefinitely by keeping connections open (the `repeatWhen` loop re-fires every 500 ms with only ±50 ms jitter). Severity: Medium — service-level DoS (griefing) with no on-chain economic damage.

### Likelihood Explanation
No authentication is required to open a gRPC `subscribeTopic` stream. The attack requires only a script that opens many TCP connections simultaneously — trivially achievable with any gRPC client library. The 500 ms synchronization window is wide enough that even connections opened tens of milliseconds apart will fire their first poll within the same scheduler tick. The attack is repeatable and persistent as long as the attacker holds connections open.

### Recommendation
1. **Add jitter to the initial delay**: Replace `.delaySubscription(interval, scheduler)` with a randomized delay, e.g. `interval` × (1 + random[0, 0.5]), mirroring the `jitter` already applied to `repeatWhen`.
2. **Enforce a global subscription limit**: Track active `POLL`-type subscriptions in an `AtomicInteger`; reject new subscriptions above a configurable ceiling.
3. **Add per-IP connection rate limiting** at the Netty/load-balancer layer (the GCP `maxRatePerEndpoint` in `values.yaml` applies to HTTP, not raw gRPC streams).
4. **Consider using `SHARED_POLL` or `REDIS`** (the default) instead of `POLL` in production, as `SharedPollingTopicListener` issues a single shared DB query regardless of subscriber count.

### Proof of Concept
```python
import grpc, threading, time
from proto import consensus_service_pb2_grpc, mirror_pb2

ADDR = "mirror-node:5600"
N_CONNECTIONS = 50   # 50 connections × 5 streams = 250 simultaneous first polls

def open_streams(stub):
    for _ in range(5):
        req = mirror_pb2.ConsensusTopicQuery(topic_id=..., consensus_start_time=...)
        stub.subscribeTopic(req)  # non-blocking; stream fires first poll at T+500ms

stubs = [consensus_service_pb2_grpc.ConsensusServiceStub(
             grpc.insecure_channel(ADDR)) for _ in range(N_CONNECTIONS)]

# Open all streams simultaneously — all first polls fire at T+500ms
threads = [threading.Thread(target=open_streams, args=(s,)) for s in stubs]
for t in threads: t.start()
for t in threads: t.join()

# At T+500ms: 250 blocking JPA queries hit a ~10-connection HikariCP pool.
# Legitimate subscribers receive HikariPool timeout errors.
time.sleep(60)  # hold connections open to sustain the attack
```
