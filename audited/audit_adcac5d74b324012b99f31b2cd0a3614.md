### Title
Unbounded `string_agg` Hash-Aggregate in `latestFileContentsQuery` Enables DB Memory Exhaustion via Crafted Hedera File Reference

### Summary
The outer query of `latestFileContentsQuery` in `rest/service/fileDataService.js` uses `GROUP BY entity_id` combined with `string_agg(file_data, '' ORDER BY consensus_timestamp)` with no row or byte limit. PostgreSQL must buffer every qualifying `file_data` row for the entity in a hash aggregate before returning a single result row, consuming memory proportional to the total concatenated size of all blobs. An attacker who controls a Hedera account (no special privileges required) can pre-load a target entity with arbitrarily many large `FILEAPPEND` chunks, then repeatedly trigger this query through the public contract-results REST endpoint, causing sustained database memory pressure that degrades service for all users.

### Finding Description

**Exact code location:**
`rest/service/fileDataService.js`, `latestFileContentsQuery` static field, lines 18–41; called from `getLatestFileDataContents` (line 84–87), which is invoked at `rest/controllers/contractController.js` lines 1022–1023 and 1183–1184.

**The query (rendered):**
```sql
WITH latest_create AS (
    SELECT MAX(consensus_timestamp) AS consensus_timestamp
    FROM file_data
    WHERE entity_id = $1 AND transaction_type IN (17, 19)
    GROUP BY entity_id
    ORDER BY consensus_timestamp DESC
)
SELECT
    MAX(fd.consensus_timestamp)          AS consensus_timestamp,
    MIN(fd.consensus_timestamp)          AS first_consensus_timestamp,
    string_agg(fd.file_data, ''
               ORDER BY fd.consensus_timestamp) AS file_data
FROM file_data fd
JOIN latest_create l
  ON fd.consensus_timestamp >= l.consensus_timestamp
WHERE fd.entity_id = $1
  AND fd.transaction_type IN (16, 17, 19)
  AND fd.consensus_timestamp >= l.consensus_timestamp
GROUP BY fd.entity_id          -- ← forces full hash-aggregate of all blobs
```

**Root cause:** The `GROUP BY fd.entity_id` + `string_agg(...)` aggregate has no `LIMIT`, no `WHERE` clause bounding the number of rows, and no PostgreSQL `work_mem` guard at the application layer. PostgreSQL's hash-aggregate node must materialise every qualifying row's `file_data` column in memory before it can emit the single output row. The total memory consumed equals the sum of all `file_data` byte values for the entity since its last `FILECREATE`/`FILEUPDATE` (transaction types 17/19).

**Trigger path:**
1. `contractController.getContractResultsByTimestamp` or `getContractResultsByTransactionIdOrHash` receives a public HTTP request.
2. It resolves `ethTransaction.callDataId` from the database (the file entity ID stored in the `ethereum_transaction` record).
3. `utils.isValidUserFileId(ethTransaction.callDataId)` passes if the ID is a non-system file entity.
4. `FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []})` is called with an empty filter, meaning the outer query aggregates **all** rows from the latest create-timestamp to the present.

**Why existing checks fail:**
- `isValidUserFileId` only validates that the ID is a non-system file entity; it imposes no size or row-count constraint.
- `statement_timeout` (configured in `rest/dbpool.js` line 15 via `config.db.pool.statementTimeout`) limits wall-clock query time but does **not** prevent the memory spike that occurs while the hash aggregate is being built before the timeout fires. Under concurrent load, many in-flight aggregates each hold their own copy of the blobs.
- There is no `work_mem` cap, no `MAX(length(file_data))` guard, and no pagination or streaming of the blob assembly. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

### Impact Explanation
Each request that resolves to a large-file `callDataId` causes the PostgreSQL backend process to allocate memory equal to the sum of all `file_data` blobs for that entity. With N concurrent requests targeting the same entity, N independent hash-aggregate nodes each hold the full blob set simultaneously. A file with 1 000 × 6 KB `FILEAPPEND` chunks (6 MB total, well within Hedera's per-file limits) causes each query to allocate ~6 MB of PostgreSQL working memory. Fifty concurrent requests = ~300 MB of database memory consumed by a single attacker-controlled entity. Scaling to larger files or higher concurrency can exhaust the database server's available memory, causing OOM kills of backend processes, connection failures, and full service unavailability for all mirror-node consumers. This directly satisfies the "increasing network processing node resource consumption by at least 30%" threshold stated in the scope.

### Likelihood Explanation
**Preconditions (all achievable without privilege):**
1. A Hedera account — free to create on mainnet/testnet.
2. A small amount of HBAR — available from public faucets on testnet; cheap on mainnet.
3. Ability to submit `FileCreate` + repeated `FileAppend` transactions — standard public Hedera SDK operations.
4. Ability to submit one `EthereumTransaction` with `callDataId` set to the created file — standard public operation.

Once the data is on-chain and ingested by the mirror node, the attacker needs only to send repeated unauthenticated HTTP GET requests to `/api/v1/contracts/results/{hash}` — zero additional cost. The attack is fully repeatable and can be automated. No authentication, no API key, and no special network access is required to query the REST endpoint.

### Recommendation
1. **Add a row/byte cap in the query:** Introduce a subquery that counts or sums `length(file_data)` for the entity and short-circuits (returns NULL or raises an error) if it exceeds a configured threshold (e.g., 10 MB).
2. **Replace unbounded `string_agg` with a streaming/chunked approach** or enforce a `LIMIT` on the number of `FILEAPPEND` rows assembled.
3. **Set a per-query `work_mem` limit** at the session level before executing this query (`SET LOCAL work_mem = '16MB'`) so PostgreSQL spills to disk rather than consuming unbounded RAM.
4. **Rate-limit the contract-results endpoints** per IP/account to reduce the amplification factor of concurrent requests.
5. **Validate `callDataId` file size** before issuing the aggregation query — query `pg_column_size` or a pre-computed `file_size` column and reject requests above a threshold.
