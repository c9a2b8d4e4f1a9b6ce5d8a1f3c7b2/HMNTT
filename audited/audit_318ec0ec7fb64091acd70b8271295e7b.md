### Title
Gas Token Neutralization via Band-Boundary Exploit in `scaleGas()` Enables Zero-Net-Cost EVM Execution

### Summary
The `scaleGas()` floor-division in `ThrottleProperties` creates 10,000-gas-wide bands where all values within a band cost the same number of tokens. An attacker who sets `gas=19,999` and calls a contract consuming fewer than 10,000 gas will have the full consumed token restored post-execution, resulting in **zero net gas tokens** deducted from the `gasLimitBucket` for real EVM work. This allows any unprivileged user to sustain EVM execution at the full rate-limit rate (500 req/s by default) while never depleting the gas throttle bucket.

### Finding Description

**Root cause — `ThrottleProperties.scaleGas()`, lines 42–47:**

```java
private static final long GAS_SCALE_FACTOR = 10_000L;

public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {
        return 0L;   // ← dead zone: any gas ≤ 10,000 is free
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
``` [1](#0-0) 

`Math.floorDiv` maps the entire range `[10,001 … 19,999]` to **1 token** and `[20,000 … 29,999]` to **2 tokens**.

**Consume path — `ThrottleManagerImpl.throttle()`, line 40:**
```java
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
``` [2](#0-1) 

**Restore path — `ThrottleManagerImpl.restore()`, lines 59–63:**
```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);
    }
}
``` [3](#0-2) 

**Restore trigger — `ContractCallService.restoreGasToBucket()`, lines 149–150:**
```java
final var gasRemaining = gasLimit - result.gasUsed();
throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
``` [4](#0-3) 

**The arithmetic that breaks the throttle:**

| gasLimit | gasUsed | gasRemaining | Consumed tokens | Restored tokens | **Net tokens** |
|---|---|---|---|---|---|
| 19,999 | 5,000 | 14,999 | `scaleGas(19999)=1` | `scaleGas(14999)=1` | **0** |
| 19,999 | 9,998 | 10,001 | 1 | `scaleGas(10001)=1` | **0** |
| 19,999 | 9,999 | 10,000 | 1 | `scaleGas(10000)=0` (≤ threshold) | 1 |
| 10,001 | 5,000 | 5,001 | 1 | `scaleGas(5001)=0` | 1 |

The zero-cost window is: **`gas=19,999` with any contract consuming ≤ 9,998 gas**. The consumed token is fully refunded because `gasRemaining ≥ 10,001`, which `scaleGas` maps back to 1.

**Failed assumption:** The design assumes `scaleGas(gasLimit) - scaleGas(gasRemaining)` approximates `scaleGas(gasUsed)`. Due to floor-division band boundaries, this equality breaks whenever `gasLimit` and `gasRemaining` fall in the same band, yielding a net cost of 0 regardless of actual EVM work performed.

### Impact Explanation

With `gasLimitRefundPercent=100` (default) and `gasPerSecond=7,500,000,000` (default), the gas bucket capacity is `scaleGas(7_500_000_000) = 750,000` tokens/s. [5](#0-4) 

An attacker making 500 req/s (the rate-limit ceiling) with `gas=19,999` targeting contracts that use ~5,000–9,998 gas:
- Consumes **0 net gas tokens** per second from the gas bucket
- Executes up to **500 × 9,998 ≈ 5,000,000 gas/s** of real EVM computation
- The gas bucket is **never depleted**, so the throttle never fires

If baseline legitimate traffic is ~100 req/s × 50,000 gas = 5,000,000 gas/s, the attacker adds an equal load (100% increase, far above the 30% threshold) while the gas throttle reports zero consumption. The rate-limit bucket (500 req/s) is the only remaining constraint, and it is a global shared bucket — the attacker can consume all of it. [6](#0-5) 

### Likelihood Explanation

- **No privileges required.** Any caller of `POST /api/v1/contracts/call` can set an arbitrary `gas` field.
- **Trivially discoverable.** The band boundaries are deterministic from `GAS_SCALE_FACTOR=10_000`. An attacker reads the open-source code or empirically probes with gas values near multiples of 10,000.
- **Repeatable and sustainable.** The exploit requires no state, no authentication, and no special contract — any cheap view function (ERC-20 `name()`, `symbol()`, `balanceOf()`) uses well under 10,000 gas in simulation.
- **Automatable.** A simple script sending `gas=19999` at 500 req/s sustains the attack indefinitely.

### Recommendation

Replace the floor-division scaling with a **ceiling-division** so that the token cost always rounds up, never down:

```java
public long scaleGas(long gas) {
    if (gas <= 0) return 0L;
    return Math.ceilDiv(gas, GAS_SCALE_FACTOR); // Java 18+, or (gas + GAS_SCALE_FACTOR - 1) / GAS_SCALE_FACTOR
}
```

This ensures `scaleGas(gasLimit) - scaleGas(gasRemaining) ≥ scaleGas(gasUsed)` — i.e., the net token cost is never less than the actual gas consumed scaled down. Additionally, consider charging tokens based on **actual `gasUsed`** post-execution rather than the upfront `gasLimit`, which would eliminate the band-boundary artifact entirely. [1](#0-0) 

### Proof of Concept

**Precondition:** Any ERC-20 token contract deployed on the mirror node (e.g., a token whose `name()` call uses ~3,000–5,000 gas in simulation).

**Steps:**

```bash
# Step 1: Confirm the zero-net-cost window
# gas=19999 → scaleGas=1 token consumed
# After call using ~5000 gas: gasRemaining=14999 → scaleGas=1 token restored
# Net = 0 tokens

# Step 2: Send 500 requests/second with gas=19999
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{
      "to": "<erc20_contract_address>",
      "data": "0x06fdde03",   // name() selector
      "gas": 19999
    }' &
done
wait
```

**Expected result:** All 500 requests succeed. The `gasLimitBucket` token count is unchanged (net 0 tokens consumed). Node CPU/DB load increases proportionally to the 500 EVM executions per second while the gas throttle never triggers. Repeating this every second sustains >30% higher resource consumption compared to a baseline with no attacker traffic.
