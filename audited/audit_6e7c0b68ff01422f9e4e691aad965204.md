### Title
Unauthenticated Multi-Phase DB Amplification DoS via `from=` EVM Address on `GET /contracts/results`

### Summary
The `getContractResults()` handler in `rest/controllers/contractController.js` performs up to four sequential/parallel database queries per unauthenticated HTTP request when a valid EVM address is supplied as the `from=` filter. No rate limiting exists on the REST API for unauthenticated users, allowing any external actor to flood the mirror node's database connection pool with amplified query load using only publicly discoverable EVM addresses.

### Finding Description

**Exact code path:**

**Phase 1 — EVM address resolution DB query** (`contractController.js` lines 441–444):
```js
case filterKeys.FROM:
  if (EntityId.isValidEvmAddress(filter.value)) {
    filter.value = await EntityService.getEncodedId(filter.value);
  }
```
`EntityService.getEncodedId()` calls `getEntityIdFromEvmAddress()` (`entityService.js` lines 90–103), which executes:
```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
```
This DB round-trip happens before any main query, for every request carrying an EVM address in `from=`.

**Phase 2 — Main contract results query** (`contractController.js` line 1072):
```js
const rows = await ContractService.getContractResultsByIdAndFilters(conditions, params, order, limit);
```

**Phase 3 — Two parallel DB queries** (`contractController.js` lines 1083–1086):
```js
const [ethereumTransactionMap, recordFileMap] = await Promise.all([
  ContractService.getEthereumTransactionsByPayerAndTimestampArray(payers, timestamps),
  RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps),
]);
```

**Root cause:** The handler unconditionally resolves EVM addresses via a live DB lookup before the main query, then always issues two additional parallel DB queries against `ethereum_transaction` and `record_file` tables for every non-empty result set. There is no per-IP or per-user rate limiting on the REST API layer — `authHandler.js` only adjusts the response row limit for authenticated users; it does not throttle request rate for unauthenticated callers.

**Why existing checks are insufficient:**
- `buildAndValidateFilters()` validates format only; it explicitly does not resolve EVM addresses (noted in the comment at line 442).
- `authHandler.js` sets a custom `limit` for authenticated users but does not block or throttle unauthenticated requests.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` (JSON-RPC) module, not in the `rest` module.
- The `maxRepeatedQueryParameters` guard limits repeated `from=` values per request but does not limit concurrent requests.

### Impact Explanation
Each unauthenticated `GET /contracts/results?from=<evm_address>` request causes 3–4 database queries (entity lookup → contract_result scan → parallel ethereum_transaction + record_file scans). An attacker sending N concurrent requests generates up to 4N simultaneous DB queries. Because the PostgreSQL connection pool is shared across all mirror node request handlers, sustained amplified load saturates the pool, causing query queuing, timeout cascades, and effective unavailability of the mirror node REST API. Mirror node instances are part of the Hedera network's data-serving infrastructure; degrading them disrupts block explorers, wallets, dApps, and any service relying on mirror node data, consistent with the ≥30% network processing degradation threshold.

### Likelihood Explanation
Preconditions are minimal: no account, no credentials, no privileged access. Valid EVM addresses are publicly enumerable from the mirror node's own `/api/v1/contracts` and `/api/v1/accounts` endpoints or from public block explorers. The attack is trivially scriptable with standard HTTP tooling (`curl`, `ab`, `wrk`). It is repeatable indefinitely because there is no connection-level or IP-level throttle in the REST service. A single attacker with modest bandwidth can sustain the load.

### Recommendation
1. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit`) applied globally before route handlers, with a low per-IP ceiling for unauthenticated callers.
2. **Cache EVM-address-to-entity-ID resolutions** in a short-lived in-process or Redis cache so repeated lookups for the same address do not hit the DB.
3. **Short-circuit on EVM address not found**: the current code throws `NotFoundError` from `getEncodedId` (with `requireResult=true`) which aborts before Phase 2/3, but Phase 1 still consumes a DB query. Pass `requireResult=false` and return an empty result immediately if the address is unknown, avoiding the main query entirely.
4. **Enforce a global request concurrency limit** per client at the ingress/load-balancer layer.

### Proof of Concept
```bash
# Step 1: obtain any valid EVM address from the public API
EVM=$(curl -s 'https://<mirror-node>/api/v1/contracts?limit=1' \
      | jq -r '.contracts[0].evm_address')

# Step 2: flood with concurrent unauthenticated requests
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/contracts/results?from=${EVM}" &
done
wait
```
Each of the 500 concurrent requests triggers:
1. `SELECT id FROM entity WHERE evm_address = $1` (Phase 1)
2. `SELECT … FROM contract_result cr LEFT JOIN entity e … WHERE cr.sender_id = $1 … LIMIT $2` (Phase 2)
3. Parallel: `SELECT … FROM ethereum_transaction WHERE payer_account_id = ANY($1) AND consensus_timestamp = ANY($2) …` and `SELECT … FROM record_file WHERE consensus_end IN (…)` (Phase 3)

Resulting in up to 2,000 simultaneous DB queries from a single unauthenticated source, exhausting the connection pool and degrading or halting mirror node REST API availability.
