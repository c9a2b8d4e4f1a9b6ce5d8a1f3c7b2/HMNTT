### Title
Unauthenticated DB Connection Pool Exhaustion via `/accounts/:id/rewards` Endpoint

### Summary
The `listStakingRewardsByAccountId` handler in `rest/controllers/accountController.js` issues at least two sequential, uncached database queries per request (`isValidAccount` + `getRewards`). The REST API has no rate limiting middleware, and the default connection pool size is 10. An unauthenticated attacker maintaining modest concurrent request volume can saturate the pool, causing all subsequent requests to queue or fail, effectively partitioning the application from the database.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 16 registers the handler with no rate-limiting wrapper: [1](#0-0) 

`rest/controllers/accountController.js` lines 170–203 — `listStakingRewardsByAccountId` issues three sequential async DB operations: [2](#0-1) 

1. `EntityService.getEncodedId()` — for numeric IDs this is pure computation (no DB hit); for aliases/EVM addresses it issues one DB query.
2. `EntityService.isValidAccount()` — **always** issues a DB query with no caching: [3](#0-2) 
3. `StakingRewardTransferService.getRewards()` — always issues a DB query. [4](#0-3) 

**Root cause — no rate limiting in the REST API middleware stack:**

The middleware index exports no throttle or rate-limit handler: [5](#0-4) 

A `grep` across all `rest/**/*.js` for `rateLimit`, `rateLimiter`, `express-rate-limit`, or `throttle` returns zero matches in production code. The `ThrottleConfiguration`/`ThrottleManagerImpl` classes exist only in the `web3` (Java) module and are entirely absent from the Node.js REST service.

**Root cause — tiny default pool:**

The pool is configured with `max: config.db.pool.maxConnections`, whose documented default is **10**: [6](#0-5) [7](#0-6) 

**Why existing timeouts are insufficient:**

`connectionTimeoutMillis: 20000` and `statement_timeout: 20000` bound individual query duration but do not prevent concurrent requests from holding all 10 pool slots simultaneously. They only determine how long a queued request waits before failing — they do not shed load.

### Impact Explanation

With a pool of 10 and 2–3 sequential DB queries per request, an attacker maintaining ≥10 concurrent in-flight requests keeps every pool slot occupied. All legitimate requests then queue behind `connectionTimeoutMillis` (20 s) before receiving an error. The application cannot execute any database query — including health checks and unrelated endpoints — until the attacker stops. This is a complete, unauthenticated denial-of-service against the database layer.

### Likelihood Explanation

No authentication, API key, or IP-based throttle is required. The attacker needs only a standard HTTP client capable of issuing ~10–50 concurrent connections (trivially achievable with `curl --parallel`, `ab`, `wrk`, or any scripting language). Valid numeric account IDs (e.g., `0.0.1` through `0.0.1000`) are publicly known from the Hedera ledger. The attack is repeatable indefinitely and requires no special knowledge of the system internals.

### Recommendation

1. **Add rate limiting to the REST API** — integrate `express-rate-limit` (or equivalent) as middleware applied globally or specifically to the `/accounts/:id/rewards` route, keyed by IP address.
2. **Cache `isValidAccount` results** — the existence check result for a given `accountId` changes infrequently; a short-lived in-process cache (e.g., using the existing `quickLru` pattern already present in `rest/entityId.js`) would eliminate the per-request DB round-trip.
3. **Increase pool size or add a concurrency limiter** — raise `db.pool.maxConnections` in production deployments and/or use a semaphore to cap the number of simultaneous in-flight DB operations from the REST service.
4. **Deploy an ingress-level rate limiter** — the Traefik `Middleware` CRD already exists in the Helm chart (`charts/hedera-mirror-rest/templates/middleware.yaml`) and can be configured with `rateLimit` to enforce per-IP request caps before traffic reaches the Node.js process.

### Proof of Concept

```bash
# Saturate the 10-connection pool with 20 concurrent requests
# using publicly known account IDs (no auth required)
seq 1 20 | xargs -P 20 -I{} \
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.{}/rewards" &

# Simultaneously observe that legitimate requests time out or return 503
curl -v "https://<mirror-node-host>/api/v1/accounts/0.0.800/rewards"
# Expected: hangs for up to 20 s then fails with connection timeout error
```

Reproducible steps:
1. Identify any 20+ valid Hedera account IDs (e.g., `0.0.1`–`0.0.20`).
2. Fire 20 concurrent GET requests to `/api/v1/accounts/<id>/rewards` with no credentials.
3. Observe that the pool (`maxConnections=10`) is saturated; subsequent requests from any client stall for up to 20 seconds or fail.
4. Maintain the flood to sustain the outage indefinitely.
