### Title
Unbounded Prometheus Label Cardinality via Arbitrary HTTP Method Injection in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates instrumentation to `github.com/weaveworks/common/middleware.Instrument`, which records `r.Method` verbatim as the `method` label on all four `HistogramVec`/`GaugeVec` metrics. Go's `net/http` server does not reject non-standard HTTP method tokens, so an unauthenticated attacker can flood the server with requests bearing unique arbitrary method strings, creating an unbounded number of Prometheus time series and exhausting process memory.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76â€“83:
```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,   // labels: method, route, status_code, ws
        InflightRequests: requestInflightGauge,       // labels: method, route
        RequestBodySize:  requestBytesHistogram,      // labels: method, route
        ResponseBodySize: responseBytesHistogram,     // labels: method, route
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

The `weaveworks/common/middleware.Instrument.Wrap()` implementation reads `r.Method` directly and passes it as the `method` label when calling `With(labels).Observe(...)` on each `HistogramVec`. There is no allowlist, normalization, or truncation of the method string before it becomes a label value.

**Middleware chain** (`rosetta/main.go`, lines 217â€“219):
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware    := server.CorsMiddleware(tracingMiddleware)
```

`MetricsMiddleware` wraps the router, so it executes on every inbound request before routing. Even if the inner router returns 405, the metrics observation with the attacker-controlled method string has already been committed to the Prometheus registry.

**Root cause:** Go's `net/http` server accepts any syntactically valid HTTP token as a method (RFC 7230 Â§3.1.1 allows any token). It does not reject `FUZZ`, `INJECT`, `EXPLOIT`, or any other non-standard string. The application adds zero validation between the raw `r.Method` value and the Prometheus label.

**Why existing checks fail:**
- The Traefik `rateLimit` (10 req/s) in `charts/hedera-mirror-rosetta/values.yaml` lines 157â€“160 is an optional Helm/Kubernetes deployment artifact â€” it is not enforced by the Go application itself and is absent in bare-metal or non-Kubernetes deployments.
- No application-level rate limiter, method allowlist, or label sanitizer exists anywhere in the rosetta middleware stack.
- The `rosettaAsserter` validates request *bodies*, not HTTP methods at the transport layer.

### Impact Explanation
Each unique `(method, route, status_code, ws)` tuple creates a new time series in `requestDurationHistogram` (6 buckets + `_count` + `_sum` = 8 series). Across all four metric vecs, each new method string materializes ~20â€“30 new in-memory series. With no bound on cardinality, an attacker sending N distinct method strings causes O(N) memory growth in the Prometheus default registry. Sustained flooding leads to OOM, crashing the rosetta process and denying service to all legitimate users of the mirror node's Rosetta API.

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge of the application internals. It is executable with standard tools (`curl`, `netcat`, or any HTTP fuzzer). The attacker needs only network access to the rosetta port. The attack is repeatable and automatable: a simple loop generating unique method tokens (e.g., `METHOD1`, `METHOD2`, â€¦) is sufficient. Without infrastructure-level rate limiting deployed, a single attacker machine can exhaust memory within minutes.

### Recommendation
1. **Normalize the method label** before recording: replace any method not in the standard set (`GET`, `POST`, `PUT`, `DELETE`, `PATCH`, `HEAD`, `OPTIONS`, `CONNECT`, `TRACE`) with a fixed sentinel value such as `"UNKNOWN"`.
2. **Reject non-standard methods early** in the middleware chain (before `MetricsMiddleware`) with a 405 response, preventing the label from ever being recorded.
3. **Apply application-level rate limiting** (e.g., `golang.org/x/time/rate`) unconditionally in the Go process, independent of infrastructure configuration.

### Proof of Concept
```bash

for i in $(seq 1 10000); do
  printf "METHOD${i} /network/list HTTP/1.1\r\nHost: localhost\r\nContent-Length: 0\r\n\r\n" \
    | nc -q1 <rosetta-host> <rosetta-port> &
done

# Observe memory growth in the rosetta process
watch -n1 'ps -o rss= -p $(pgrep rosetta)'

# Confirm cardinality explosion via metrics endpoint
curl http://<rosetta-host>:<rosetta-port>/metrics | grep hiero_mirror_rosetta_request_duration | wc -l
```
Each iteration adds new `method="METHODn"` label combinations to all four metric vecs, with memory usage growing proportionally until OOM.
