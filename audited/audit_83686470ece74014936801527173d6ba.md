### Title
Unauthenticated `getAirdrops()` Endpoint Lacks Rate Limiting, Enabling DB Buffer Pool Cache Thrashing

### Summary
The `getAirdrops()` endpoint in the `rest-java` module has no rate limiting, no per-IP throttling, and no application-level caching. An unprivileged attacker can issue unlimited concurrent requests cycling through different high-volume account IDs, forcing the database to continuously load cold pages for each distinct account's airdrop data, evicting warm pages used by legitimate queries and degrading DB buffer pool efficiency for all concurrent users.

### Finding Description

**Code path:**

`TokenAirdropsController.getOutstandingAirdrops()` / `getPendingAirdrops()` (lines 66–86) → `processRequest()` (lines 88–114) → `TokenAirdropServiceImpl.getAirdrops()` (lines 19–22) → `TokenAirdropRepositoryCustomImpl.findAll()` (lines 58–72).

The repository executes a direct, uncached SQL query on every call:

```java
// TokenAirdropRepositoryCustomImpl.java lines 66-71
return dslContext
    .selectFrom(TOKEN_AIRDROP)
    .where(condition)
    .orderBy(order)
    .limit(request.getLimit())
    .fetchInto(TokenAirdrop.class);
```

There is no `@Cacheable`, no application-level result cache, and no query deduplication. Every HTTP request results in a live DB round-trip.

**Failed assumption — no rate limiting in rest-java:**

The `web3` module has a full throttle stack (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j rate buckets). The `rest-java` module has **none of this**. Its filter chain contains only `LoggingFilter` (logging) and `MetricsFilter` (metrics recording). `RestJavaConfiguration` registers only an ETag filter and a protobuf converter. There is no `RateLimiter`, no `Bucket`, no per-IP or per-user throttle anywhere in `rest-java`.

**Only existing guard:**

```java
// TokenAirdropsController.java line 69
@RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
```

`MAX_LIMIT` is 100 (per the design doc). This caps result size per request but does **not** limit request frequency or total request volume.

**Exploit flow:**

1. Attacker enumerates or discovers N high-volume account IDs (accounts with large numbers of pending/outstanding airdrops — publicly observable on-chain).
2. Attacker issues sustained parallel GET requests cycling through those accounts:
   ```
   GET /api/v1/accounts/{account_1}/airdrops/outstanding?limit=100
   GET /api/v1/accounts/{account_2}/airdrops/outstanding?limit=100
   ...
   GET /api/v1/accounts/{account_N}/airdrops/outstanding?limit=100
   (repeat round-robin)
   ```
3. Each distinct account ID causes the DB to read a different set of `token_airdrop` index and data pages (indexed on `sender_account_id` / `receiver_account_id`).
4. With N large enough relative to the DB buffer pool size, each new account's pages evict the previously loaded pages, preventing any page from staying warm.
5. Legitimate queries (from other users or internal processes) find their working set continuously evicted, causing physical I/O on every access.

### Impact Explanation

The `token_airdrop` table is append-heavy and can grow large on a production network. Each account's airdrop rows are clustered by the index (`token_airdrop__sender_id`, `token_airdrop__receiver_id`), so rows for different accounts reside on different physical pages. Cycling through many accounts forces the DB buffer pool into a thrashing state: hit rate drops, physical reads spike, and query latency increases for **all** concurrent users — not just the attacker's requests. This satisfies the "30% resource increase" threshold without brute-force volume, since each individual request is a legitimate, well-formed query that the DB must fully service.

### Likelihood Explanation

Preconditions are minimal: no account, no API key, no authentication of any kind is required. Account IDs are public and enumerable from the ledger. The attack is trivially scriptable with any HTTP client. The attacker does not need to send an abnormally high volume — a modest concurrency level (e.g., 10–20 parallel clients cycling through 50–100 high-volume accounts) is sufficient to keep the buffer pool in a cold state if the airdrop dataset is large relative to available RAM.

### Recommendation

1. **Add rate limiting to rest-java airdrop endpoints.** Port the existing `bucket4j`-based throttle from the `web3` module (`ThrottleConfiguration` / `ThrottleManagerImpl`) to `rest-java`, or add a Spring Cloud Gateway / reverse-proxy rate-limit layer in front of the service.
2. **Add per-IP or per-client-token rate limiting** at the controller or filter level (e.g., using `bucket4j` with a `ConcurrentHashMap<String, Bucket>` keyed on remote IP).
3. **Add a short-lived application-level cache** (e.g., Caffeine with a 5–10 second TTL) in `TokenAirdropServiceImpl.getAirdrops()` keyed on `(accountId, type, limit, order, bounds)` to absorb repeated identical queries.
4. **Consider DB-level connection pool limits** per client IP as a defense-in-depth measure.

### Proof of Concept

```bash
# Enumerate high-volume sender accounts (publicly observable)
ACCOUNTS=(1001 1002 1003 ... 1100)

# Launch round-robin requests with concurrency
while true; do
  for ACCT in "${ACCOUNTS[@]}"; do
    curl -s "https://<mirror-node>/api/v1/accounts/${ACCT}/airdrops/outstanding?limit=100" &
  done
  wait
done
```

No authentication headers are needed. Monitor DB `pg_stat_bgwriter` (`buffers_alloc`, `buffers_clean`) and `pg_stat_database` (`blks_read` vs `blks_hit`) — the buffer hit ratio will drop measurably as the attack runs, confirming cache eviction pressure on the shared `token_airdrop` working set.
