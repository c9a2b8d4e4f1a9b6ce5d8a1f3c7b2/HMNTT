### Title
Unbounded Response Body in `fetchMirrorNodeNetwork` Enables Memory Exhaustion via MITM

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` decodes the HTTP response body from the mirror node REST API directly into a `nodesEnvelope` struct with no size cap. A network-adjacent attacker who can intercept the HTTP connection (MITM) can serve an arbitrarily large JSON body, causing the pinger process to allocate unbounded memory, crash with OOM, and halt transaction submissions — creating gaps in the mirror node transaction history.

### Finding Description
**Exact location:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, line 96:

```go
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```

`resp.Body` is passed directly to `json.NewDecoder` with no `io.LimitReader` or `http.MaxBytesReader` wrapper. The only guard present is the `http.Client` timeout set at line 46:

```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

This timeout governs the total round-trip duration but does **not** cap the number of bytes read from the body. An attacker who can respond within the timeout window (e.g., by streaming a large payload at high bandwidth) bypasses this control entirely.

**Exploit flow:**
1. Attacker positions themselves as a MITM between the pinger and the configured `mirrorRest` endpoint (feasible when the URL is plain HTTP, or via DNS spoofing/ARP poisoning on the same network segment).
2. Attacker intercepts the `GET /api/v1/network/nodes` request.
3. Attacker responds with a valid HTTP 200 and a JSON body containing a `nodes` array with millions of deeply nested entries, sized to exhaust available RAM (e.g., several GB).
4. `json.NewDecoder(...).Decode(&payload)` allocates memory proportional to the payload size.
5. The Go runtime triggers OOM; the pinger process is killed by the OS or stalls under GC pressure.

**Why existing checks fail:**
- `http.Client.Timeout` only limits wall-clock time, not bytes transferred. A fast network link can deliver gigabytes within a typical timeout.
- There is no `http.MaxBytesReader`, no `io.LimitReader`, and no maximum-node-count validation before or after decoding.
- The retry loop in `buildNetworkFromMirrorNodes` (lines 52–69) would re-attempt the request, repeatedly triggering the same allocation if the attacker sustains the MITM.

### Impact Explanation
The pinger's sole purpose is to submit periodic crypto-transfer transactions to populate the mirror node's transaction history. If the pinger process is killed or stalled by OOM, no transactions are submitted, creating observable gaps in the mirror node's transaction record. This is a Denial-of-Service against the pinger's core function. Severity is **Medium-High** given the targeted, availability-focused impact on a monitoring/health-check component.

### Likelihood Explanation
The precondition is network-level MITM capability — not zero-effort, but realistic in several scenarios: (a) the `mirrorRest` URL is configured as plain HTTP (no TLS), which is common in internal/testnet deployments; (b) the attacker is on the same LAN/container network and can perform ARP poisoning; (c) DNS is not DNSSEC-protected and the attacker controls a resolver. No application-level credentials or API keys are required. The attack is repeatable as long as the MITM position is maintained.

### Recommendation
Wrap `resp.Body` with a size-limiting reader before decoding:

```go
const maxBodyBytes = 10 * 1024 * 1024 // 10 MB
limited := http.MaxBytesReader(nil, resp.Body, maxBodyBytes)
if err := json.NewDecoder(limited).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```

Additionally:
- Enforce HTTPS for `mirrorRest` and validate the server certificate.
- Add a post-decode sanity check: reject responses with an unreasonably large number of nodes (e.g., `> 1000`).

### Proof of Concept
1. Configure the pinger with `mirrorRest = "http://mirror.example.com"` (plain HTTP).
2. On the same network, run a rogue HTTP server that responds to `GET /api/v1/network/nodes` with:
   ```python
   import json, socket
   # Generate ~2 GB JSON: {"nodes": [{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"x"*10000,"ip_address_v4":"","port":50211}]} * 200000], "links":{}}
   ```
3. ARP-spoof or DNS-spoof the pinger host to redirect `mirror.example.com` to the rogue server.
4. Start (or restart) the pinger. Observe `buildNetworkFromMirrorNodes` → `fetchMirrorNodeNetwork` allocating memory until OOM kill.
5. Confirm no transactions appear in the mirror node history for the duration of the attack. [1](#0-0) [2](#0-1) [3](#0-2)
