### Title
Unauthenticated DoS via Repeated `IllegalArgumentException` Stack Trace Generation in `getNftAllowances()`

### Summary
Any unauthenticated caller can send `GET /api/v1/accounts/0.0.1/allowances/nfts?account.id=ne:0.0.1` at arbitrary rate. The `ne` operator is a valid `RangeOperator` enum value and passes parameter deserialization cleanly, but is rejected only at the service layer by throwing a new `IllegalArgumentException` — including a full JVM stack trace — on every single request. No rate limiting exists on this endpoint, so an attacker can sustain this indefinitely.

### Finding Description
`EntityIdRangeParameter.valueOf()` calls `RangeOperator.of("ne")` which successfully returns `RangeOperator.NE` (line 30 of `EntityIdRangeParameter.java`). The parsed parameter is stored in a `Bound` object (controller line 71). The controller then calls `service.getNftAllowances(request)`, which calls `checkOwnerSpenderParamValidity()` → `verifyRangeId()` → `ids.verifyUnsupported(RangeOperator.NE)`. This method executes `throw new IllegalArgumentException("Unsupported range operator ne for account.id")` (`Bound.java` lines 148–153) on every invocation. Java's `new IllegalArgumentException(...)` always calls the native `fillInStackTrace()`, which walks the entire call stack and allocates a `StackTraceElement[]` array on the heap. `GenericControllerAdvice.badRequest()` catches it and returns HTTP 400 without any deduplication or suppression. No rate limiter, token bucket, or connection throttle is applied to the `rest-java` REST endpoints.

### Impact Explanation
Each malicious request forces: (a) full call-stack capture via native `fillInStackTrace()`, (b) heap allocation of a `StackTraceElement[]` array, (c) GC pressure proportional to request rate. Under sustained high-rate bombardment this degrades JVM throughput for all legitimate users sharing the same process. The scope is explicitly classified as "Medium: griefing with no economic damage" in the audit scope, which matches exactly — no funds are at risk but service quality degrades.

### Likelihood Explanation
The attack requires zero authentication, zero tokens, and zero knowledge beyond the public API path. The request is a single HTTP GET with a trivially crafted query string. It is fully repeatable, automatable with any HTTP load tool (e.g., `wrk`, `ab`, `curl` in a loop), and requires no state. The absence of any rate limiting in the `rest-java` module means there is no server-side barrier.

### Recommendation
Apply one or more of the following:
1. **Reject `ne` at parameter deserialization**: In `EntityIdRangeParameter.valueOf()`, explicitly reject `RangeOperator.NE` before the object is constructed, so the exception is thrown before `Bound` is allocated and before the service is entered.
2. **Add rate limiting to `rest-java` endpoints**: Port the bucket4j throttle infrastructure from the `web3` module to `rest-java`, applying a per-IP or global request-per-second cap on `/api/v1/accounts/*/allowances/nfts`.
3. **Use a static, pre-allocated exception**: For known-invalid-operator rejections, throw a pre-constructed static exception instance (with `fillInStackTrace` overridden to be a no-op) to eliminate per-request stack trace allocation.
