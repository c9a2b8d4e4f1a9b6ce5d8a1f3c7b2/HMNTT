### Title
Missing Query Timeout on `entityRepository.findById()` in GraphQL `getByIdAndType()` Enables Connection Pool Exhaustion During DB Unavailability

### Summary
`EntityServiceImpl.getByIdAndType()` calls `entityRepository.findById()` with no query timeout configured at the JPA, Hibernate, or JDBC socket level for the GraphQL module. During a network partition or DB slowdown, each in-flight GraphQL request holds a HikariCP connection indefinitely. With no application-level rate limiting on the GraphQL endpoint and only a weak per-IP Traefik concurrency cap, an attacker with multiple source IPs can exhaust the connection pool and render the service unavailable.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 25:
```java
public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
    return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
}
```

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 12–17: `EntityRepository` extends `CrudRepository<Entity, Long>` and inherits `findById()` with no `@QueryHints` or `javax.persistence.query.timeout` annotation.

**Root cause — three compounding absences:**

1. **No JPA/Hibernate query timeout** on `findById()`. The `web3` module explicitly enforces this via `HibernateConfiguration.statementInspector()` (web3/src/main/java/org/hiero/mirror/web3/config/HibernateConfiguration.java, lines 31–47), which throws `QueryTimeoutException` when elapsed time exceeds `web3Properties.getRequestTimeout()`. No equivalent exists in the graphql module.

2. **No JDBC socket timeout on runtime connections.** `CommonConfiguration` builds the HikariCP pool from `@ConfigurationProperties("spring.datasource.hikari")` (common/src/main/java/org/hiero/mirror/common/CommonConfiguration.java, lines 61–95). The `DatabaseStartupProperties` timeouts (`connectTimeout=2s`, `socketTimeout=2s`) are consumed only by `DatabaseWaiter.waitForDatabase()` during startup (common/src/main/java/org/hiero/mirror/common/util/DatabaseWaiter.java, lines 36–39) and are never applied to the runtime pool. No `spring.datasource.hikari.*` properties file exists under `graphql/src/main/resources/`.

3. **No application-level rate limiting for the GraphQL module.** The `web3` module has `ThrottleManagerImpl` with a bucket-based rate limiter. The graphql module has no equivalent — only GraphQL parser/complexity limits (`MaxQueryComplexityInstrumentation(200)`, `MaxQueryDepthInstrumentation(10)`) in `GraphQlConfiguration` (graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java, lines 43–48), which do not bound concurrency or connection hold time.

**Why existing checks fail:**

- The Traefik middleware `inFlightReq: amount: 5` with `ipStrategy.depth: 1` (charts/hedera-mirror-graphql/values.yaml, lines 138–141) limits 5 concurrent requests per source IP. An attacker with ≥2 source IPs can saturate the default HikariCP pool (10 connections). A botnet or cloud VMs trivially provide this.
- The `GraphQLHighDBConnections` Prometheus alert (charts/hedera-mirror-graphql/values.yaml, lines 204–214) fires at 75% utilization after 5 minutes — far too slow to prevent exhaustion.

**Exploit flow:**

1. Attacker identifies the public GraphQL endpoint (no authentication required).
2. Attacker sends concurrent `account { ... }` queries that resolve through `getByIdAndType()` from ≥2 source IPs (5 concurrent each = 10 total).
3. A network partition between the graphql pod and PostgreSQL occurs (naturally or induced by a co-located attacker controlling network infrastructure).
4. Each `findById()` call blocks on the JDBC socket with no timeout. HikariCP connections are held open.
5. After 10 connections are held, HikariCP's `connectionTimeout` (default 30 s) causes all new requests to fail with `SQLTimeoutException`.
6. The GraphQL service is fully unavailable for the duration of the partition plus the JDBC default socket timeout (PostgreSQL JDBC default: 0 = infinite).

### Impact Explanation
Complete denial of service for the GraphQL API. All entity-lookup queries (`account`, `contract` by ID) fail. Because the connection hold is indefinite (no socket timeout), recovery requires either the partition to heal or a pod restart. Severity: **High** (availability impact, no authentication required, no per-request cost to attacker).

### Likelihood Explanation
The attacker needs: (a) access to the public GraphQL endpoint (zero privilege), (b) ≥2 source IPs (trivially obtained via cloud VMs or proxies), and (c) a network partition event. Partitions occur naturally in cloud environments; an attacker co-located in the same cloud region can also induce them via ARP poisoning or BGP manipulation on shared infrastructure. The attack is repeatable and requires no special knowledge beyond the GraphQL schema.

### Recommendation

1. **Add a JDBC socket timeout to the runtime HikariCP pool** for the graphql module via `spring.datasource.hikari.connection-timeout` and a PostgreSQL JDBC URL parameter `socketTimeout=N` (e.g., 10 seconds).
2. **Add a JPA query timeout** either via `spring.jpa.properties.jakarta.persistence.query.timeout=10000` or by annotating `EntityRepository.findById()` with `@QueryHints(@QueryHint(name = "jakarta.persistence.query.timeout", value = "10000"))`.
3. **Port the `StatementInspector` pattern** from `web3/HibernateConfiguration` to the graphql module to enforce a per-request wall-clock timeout.
4. **Add application-level concurrency limiting** (e.g., a Semaphore or Bucket4j rate limiter) in the graphql module, not relying solely on Traefik.

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
    }

    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
