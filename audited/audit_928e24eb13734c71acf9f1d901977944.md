### Title
Unauthenticated DoS via Uncached 404 Responses Causing Repeated DB Queries in `getTokenRelationships()`

### Summary
The `getTokenRelationships()` handler unconditionally calls `EntityService.isValidAccount()` for every request, which issues a live database query (`SELECT type FROM entity WHERE id = $1`) to validate the account. Because the response cache only stores successful (2xx) responses and never caches 404s, and because no IP-based rate limiting exists in the middleware stack, any unauthenticated attacker can flood the endpoint with non-existent numeric account IDs and force a new DB query on every single request, sustaining unnecessary database load that degrades service for legitimate users.

### Finding Description

**Exact code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 66–71:
```js
getTokenRelationships = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);
  const isValidAccount = await EntityService.isValidAccount(accountId);  // DB hit every time
  if (!isValidAccount) {
    throw new NotFoundError();  // 404 — never cached
  }
```

`rest/service/entityService.js`, `isValidAccount()`, lines 60–63:
```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
```

`entityExistenceQuery` (lines 28–30):
```sql
SELECT type FROM entity WHERE id = $1
```

**Root cause — two failed assumptions:**

1. **`getEncodedId()` for a plain numeric ID (e.g. `0.0.999999999`) performs no DB lookup** — it simply encodes the number (`entityId.getEncodedId()`). So the only DB work is `isValidAccount()`, and it fires unconditionally on every request.

2. **The response cache (`responseCacheUpdateHandler`) only stores successful responses.** Line 95 of `rest/middleware/responseCacheHandler.js`:
   ```js
   if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
   ```
   A 404 `NotFoundError` is not a success code, so it is never written to Redis. Identical repeated requests for the same non-existent account ID each bypass the cache and each trigger a fresh DB query.

**No rate limiting exists.** `rest/server.js` registers no IP-based throttle or rate-limit middleware — only `cors`, `compression`, `httpContext`, `authHandler`, and the response cache check. The `authHandler` only adjusts the response *limit* parameter; it does not restrict request *rate*.

### Impact Explanation
An attacker can saturate the PostgreSQL connection pool with cheap `SELECT type FROM entity WHERE id = $1` queries against non-existent IDs. Because these queries always miss (no row returned), they cannot be short-circuited by any index bloom filter or result cache. At sufficient request rate, the DB thread pool fills, query latency rises for all concurrent legitimate queries, and the mirror node REST API degrades or becomes unresponsive. No economic damage to on-chain state occurs, but service availability for legitimate users is impaired — consistent with the "griefing" severity classification.

### Likelihood Explanation
**Preconditions:** None. No account, no token, no authentication required.  
**Trigger:** A single `curl` loop or any HTTP flood tool targeting `/api/v1/accounts/0.0.<random>/tokens`.  
**Repeatability:** Unlimited. Each request is stateless and costs the attacker nothing. The attacker can rotate account IDs trivially (any large integer not in the entity table) to prevent any future negative-result caching if it were ever added.

### Recommendation
1. **Cache negative (404) results** for a short TTL (e.g. 5–30 s) in `responseCacheUpdateHandler` by relaxing the `httpStatusCodes.isSuccess` guard to also include 404 responses for this endpoint. This ensures repeated identical requests for the same non-existent ID are served from Redis without a DB round-trip.
2. **Add IP-based rate limiting** (e.g. `express-rate-limit`) as a global middleware in `rest/server.js`, applied before route handlers, to bound the per-IP request rate regardless of response outcome.
3. **Short-circuit before the DB call** when the numeric account ID is out of a plausible range (e.g. exceeds the known maximum entity num), returning 404 immediately without querying the database.

### Proof of Concept
```bash
# No authentication, no existing account needed.
# Each request triggers SELECT type FROM entity WHERE id = $1 with no cache hit.

for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.$((RANDOM + 9000000))/tokens" &
done
wait
```

Each iteration: `getEncodedId("0.0.<N>")` → encoded ID (no DB) → `isValidAccount(encodedId)` → DB query → 0 rows → `NotFoundError` (404) → response cache skips write → next request repeats identically.
