### Title
Entity ID Enumeration via Differential Error Responses in `topicExists()`

### Summary
The `topicExists()` method in `TopicMessageServiceImpl` returns two semantically distinct exceptions depending on whether a queried entity ID does not exist at all versus exists but is not of type `TOPIC`. These exceptions are mapped to different gRPC status codes by `ProtoUtil`, allowing any unauthenticated caller to distinguish between "entity does not exist" and "entity exists but is a non-TOPIC type" — enabling full entity ID enumeration with no privileges required.

### Finding Description
**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 94–106:

```java
private Mono<?> topicExists(TopicMessageFilter filter) {
    var topicId = filter.getTopicId();
    return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
            .switchIfEmpty(
                    grpcProperties.isCheckTopicExists()
                            ? Mono.error(new EntityNotFoundException(topicId))   // path A
                            : Mono.just(Entity.builder()...))
            .filter(e -> e.getType() == EntityType.TOPIC)
            .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));  // path B
}
```

**Root cause:** Two structurally different exceptions encode two different states:
- **Path A** (`EntityNotFoundException`): entity ID not present in the database at all.
- **Path B** (`IllegalArgumentException`): entity ID *is* present but belongs to a non-TOPIC type (ACCOUNT, CONTRACT, FILE, TOKEN, etc.).

In `grpc/src/main/java/org/hiero/mirror/grpc/util/ProtoUtil.java`, lines 47–52, these are mapped to distinct gRPC status codes:

```java
} else if (t instanceof ConstraintViolationException
        || t instanceof IllegalArgumentException   // → INVALID_ARGUMENT (code 3)
        ...
} else if (t instanceof EntityNotFoundException) {
    return clientError(t, Status.NOT_FOUND, t.getMessage());  // → NOT_FOUND (code 5)
```

The message text is also included verbatim in the gRPC status description: `"0.0.X does not exist"` vs `"Not a valid topic"`.

**Why existing checks are insufficient:** There is no authentication or authorization on the gRPC endpoint — `GrpcConfiguration` configures only concurrency limits, and the only `ServerInterceptor` present sets an endpoint context label with no access control. `checkTopicExists` defaults to `true` (line 19 of `GrpcProperties.java`), so path A is active in all standard deployments.

### Impact Explanation
An attacker can iterate over entity IDs (shard.realm.num tuples) and classify each one as: (1) does not exist, (2) exists and is a non-TOPIC entity, or (3) exists and is a TOPIC. This leaks the full shape of the entity namespace — which IDs are allocated, and what broad category they belong to. For a public mirror node this reveals information about accounts, contracts, tokens, and files that the operator may not intend to expose through this interface.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a standard gRPC client. The gRPC `subscribeTopic` RPC is the public HCS (Hedera Consensus Service) subscription endpoint, reachable by any network client. Enumeration is trivially scriptable: send one request per candidate ID, inspect the returned status code. The default configuration (`checkTopicExists=true`) means the oracle is on in every standard deployment.

### Recommendation
Normalize the error response so both cases return the same gRPC status code and the same opaque message. The simplest fix is to replace the `IllegalArgumentException` on line 105 with an `EntityNotFoundException` using the same `topicId` argument, or to map both to `Status.NOT_FOUND` with a single generic message (e.g., `"Topic not found"`). Do not include the entity ID or type information in the error description returned to the caller.

```java
// Replace line 105:
.switchIfEmpty(Mono.error(new EntityNotFoundException(topicId)));
// instead of: Mono.error(new IllegalArgumentException("Not a valid topic"))
```

This collapses both "not found" and "wrong type" into a single indistinguishable `NOT_FOUND` response, eliminating the oracle.

### Proof of Concept
**Preconditions:** A running mirror node gRPC service (default config). Entity `0.0.1000` is an ACCOUNT; entity `0.0.9999` does not exist.

**Steps:**
1. Send `subscribeTopic` for entity `0.0.9999` (non-existent):
   ```
   grpcurl -plaintext <host>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
   → Status: NOT_FOUND, description: "0.0.9999 does not exist"
   ```
2. Send `subscribeTopic` for entity `0.0.1000` (exists, type=ACCOUNT):
   ```
   grpcurl -plaintext <host>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
   → Status: INVALID_ARGUMENT, description: "Not a valid topic"
   ```
3. Observe the distinct status codes (5 vs 3) and messages — confirming entity `0.0.1000` exists in the ledger as a non-TOPIC entity.
4. Repeat across the full numeric ID space to enumerate all allocated entity IDs and their broad type classification. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
