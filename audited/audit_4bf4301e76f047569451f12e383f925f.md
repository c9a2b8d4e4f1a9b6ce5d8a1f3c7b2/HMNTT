### Title
Unbounded `X-Forwarded-For` Header Logged Verbatim Enables Log-Flood Disk-Exhaustion DoS

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the `X-Forwarded-For` header with no length cap and returns the raw value, which `TracingMiddleware` then embeds directly into a `log.Info` call. Because Go's `net/http` default `MaxHeaderBytes` is 1 MB and no explicit limit is set in the server configuration, an unauthenticated attacker can send a stream of requests each carrying a ~1 MB `X-Forwarded-For` value, writing ~1 MB per request to the log sink and exhausting available disk (or container-log storage), causing a denial-of-service.

### Finding Description

**Exact code path:**

`getClientIpAddress()` — [1](#0-0)  — reads `X-Forwarded-For` at line 67 with a plain `r.Header.Get()` and returns the raw string with zero length validation.

`TracingMiddleware` — [2](#0-1)  — interpolates that raw string directly into `message` and passes it to `log.Info`.

**Server has no `MaxHeaderBytes` override:** [3](#0-2)  — the `http.Server` struct sets only timeout fields; `MaxHeaderBytes` is absent, so Go's default of `1 << 20` (1 MB) applies per request.

**Log sink is `os.Stdout`:** [4](#0-3)  — in Docker/Kubernetes deployments stdout is captured to a JSON log file on the node's disk (e.g., `/var/lib/docker/containers/<id>/<id>-json.log`), which is subject to disk exhaustion.

**Root cause / failed assumption:** The code assumes the `X-Forwarded-For` value is a short, well-formed IP address. There is no `len()` guard, no truncation, and no sanitization before the value reaches the logger.

### Impact Explanation

Each malicious request can inject up to ~1 MB into the log stream. A sustained flood of such requests (trivially automated with `curl` or any HTTP client) fills the host's disk or container-log partition. Once disk is full:
- The Go process itself may block on the next `log.Info` write (synchronous `os.Stdout` write blocks when the pipe buffer is full and the consumer is stalled by a full disk).
- Other node processes sharing the same disk (database, OS) also fail.
- The Rosetta API becomes unavailable, partitioning the node from the network.

Severity: **High** (unauthenticated, remotely triggerable, complete availability loss).

### Likelihood Explanation

- **No authentication required** — any external host that can reach the Rosetta port can exploit this.
- **No rate limiting** visible in the middleware stack.
- **Trivially repeatable** — a single `while true; do curl -H "X-Forwarded-For: $(python3 -c "print('1.2.3.4, ' * 50000)")" http://target:5700/network/list; done` loop is sufficient.
- Deployments that expose the Rosetta port directly (common in validator/mirror-node setups) are immediately vulnerable.

### Recommendation

1. **Truncate the header value** before logging — cap `ipAddress` to a safe maximum (e.g., 45 characters for a full IPv6 address):
   ```go
   const maxIpLen = 45
   if len(ipAddress) > maxIpLen {
       ipAddress = ipAddress[:maxIpLen]
   }
   ```
2. **Take only the first token** of `X-Forwarded-For` (the leftmost IP is the client's):
   ```go
   if idx := strings.IndexByte(ipAddress, ','); idx != -1 {
       ipAddress = strings.TrimSpace(ipAddress[:idx])
   }
   ```
3. **Set an explicit `MaxHeaderBytes`** on the `http.Server` (e.g., `8192`) to reduce the per-request attack surface. [3](#0-2) 
4. **Add rate limiting** middleware upstream of `TracingMiddleware`.

### Proof of Concept

```bash
# Generate a ~900 KB X-Forwarded-For value
PAYLOAD=$(python3 -c "print(', '.join(['1.2.3.4']*130000))")

# Flood the node — each request writes ~900 KB to the log
while true; do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: $PAYLOAD" \
    http://<rosetta-host>:5700/network/list
done
```

Each iteration causes `TracingMiddleware` to call `log.Info` with a ~900 KB string. [2](#0-1)  After a few thousand iterations the host log partition is exhausted, the process blocks on the next stdout write, and the node stops responding to network requests.
