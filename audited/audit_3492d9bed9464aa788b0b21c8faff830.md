### Title
Synchronous On-Event-Loop Decompression in `responseCacheCheckHandler` Enables CPU Exhaustion by Unauthenticated Clients

### Summary
When the REST API response cache is enabled with compression (`cache.response.compress=true`, default), any unauthenticated client can omit `accept-encoding: gzip` on requests to cached endpoints, forcing the Node.js event loop to execute `unzipSync(Buffer.from(this.body, 'base64url'))` synchronously for every such cache hit. Because Node.js is single-threaded and the REST API layer has no per-IP or global rate limiting, an attacker can sustain a stream of these requests to block the event loop and drive CPU consumption well above baseline.

### Finding Description

**Exact code path:**

`rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler()`, lines 70–72:
```js
const useCompressed = negotiate(cachedResponse, req, res);
body = useCompressed ? cachedResponse.getBody() : cachedResponse.getUncompressedBody();
```

`negotiate()` (lines 121–134) returns `false` when `negotiator.encoding(['gzip'])` does not match — i.e., when the client omits or explicitly excludes `accept-encoding: gzip`. This unconditionally routes to `getUncompressedBody()`.

`rest/model/cachedApiResponse.js`, line 30:
```js
getUncompressedBody() {
  return this.compressed ? unzipSync(Buffer.from(this.body, 'base64url')) : this.body;
}
```

Two synchronous, CPU-bound operations execute on the event loop for every such request:
1. `Buffer.from(this.body, 'base64url')` — base64url decode of the stored compressed blob
2. `unzipSync(...)` — synchronous zlib decompression

**Root cause / failed assumption:** The implementation assumes that clients will generally accept gzip (the common browser/HTTP client default), so the decompression path is treated as a rare fallback. In reality, any client can deliberately omit `accept-encoding`, and there is no guard (rate limit, async offload, or pre-computed uncompressed cache) to prevent repeated synchronous decompression.

**Why existing checks fail:**
- `shouldCompress()` (line 138) uses a threshold of only 150 bytes (default `compressThreshold`), meaning virtually all real API responses are stored compressed.
- The `negotiate()` function has no short-circuit or caching of the uncompressed form.
- The REST API (Node.js) layer has **no rate limiting** — the `ThrottleConfiguration` / `ThrottleManagerImpl` classes exist only in the `web3` Java component, not in the REST Node.js server (`rest/server.js` confirms no rate-limit middleware is registered).
- `unzipSync` is a blocking call; Node.js cannot process any other request while it runs.

### Impact Explanation

Every cache hit for a compressed response that arrives without `accept-encoding: gzip` forces a full synchronous base64url-decode + zlib-decompress cycle on the single Node.js event loop thread. For responses near the API's maximum page size (100 items, potentially tens of KB of JSON), each decompression cycle is measurable CPU work. Because the event loop is blocked during this work, all other in-flight requests are stalled. Sustained at moderate concurrency (tens of requests/second, well below what a single machine can generate), this raises CPU utilization and increases latency for all users of the node. The REST API is a public, unauthenticated endpoint, so the blast radius is the entire node's availability.

### Likelihood Explanation

The attack requires zero privileges, zero credentials, and no special knowledge beyond the public API surface. The attacker needs only to:
- Identify any endpoint whose response exceeds 150 bytes (nearly all of them)
- Send GET requests to that endpoint without `accept-encoding: gzip`

This is trivially scriptable with `curl --no-compressed` or any HTTP client. The absence of rate limiting in the REST layer means there is no server-side throttle to overcome. The attack is fully repeatable and can be sustained indefinitely.

### Recommendation

1. **Cache the uncompressed form alongside the compressed form** in `CachedApiResponse`, or pre-compute and store the uncompressed body at write time so `getUncompressedBody()` is a simple field access rather than a synchronous decode+decompress.
2. **Use the async `zlib.unzip()` (promise/callback form)** instead of `unzipSync` so the event loop is not blocked.
3. **Add rate limiting** to the REST API Node.js layer (e.g., `express-rate-limit`) to bound the number of requests per IP per second, independent of the web3 Java throttle.
4. Consider storing the body in the cache in its raw (uncompressed) form and applying compression only at response-send time using the existing `compression()` middleware, eliminating the need for manual decompression entirely.

### Proof of Concept

**Preconditions:**
- Mirror node REST API running with `cache.response.enabled=true` and `cache.response.compress=true` (compress is `true` by default).
- Redis available and connected.

**Steps:**

1. Warm the cache for a large-response endpoint (any endpoint returning >150 bytes):
   ```
   curl -s -H "accept-encoding: gzip" \
     "http://<mirror-node>:5551/api/v1/transactions?limit=100" > /dev/null
   ```

2. Confirm the response is cached and compressed (200 with `content-encoding: gzip` on repeat).

3. Launch concurrent requests deliberately omitting `accept-encoding`:
   ```bash
   # Using Apache Bench or similar
   ab -n 10000 -c 50 \
     -H "accept-encoding: identity" \
     "http://<mirror-node>:5551/api/v1/transactions?limit=100"
   ```
   Or with curl in a loop:
   ```bash
   while true; do
     curl -s --no-compressed \
       "http://<mirror-node>:5551/api/v1/transactions?limit=100" > /dev/null &
   done
   ```

4. **Observe:** Node.js process CPU climbs significantly; latency for all other concurrent requests increases; event loop lag metrics spike. Each request triggers `unzipSync(Buffer.from(..., 'base64url'))` synchronously on the event loop thread, confirmed by the code path at `responseCacheHandler.js:72` → `cachedApiResponse.js:30`. const negotiate = (cachedResponse, req, res) => {
  res.setHeader(VARY_HEADER, 'accept-encoding');

  if (cachedResponse.compressed) {
    const negotiator = new Negotiator(req);
    if (negotiator.encoding([GZIP_ENCODING]) === GZIP_ENCODING) {
      res.setHeader(CONTENT_ENCODING_HEADER, GZIP_ENCODING);
      res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getLength());
      return true;
    }
  }

  res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getUncompressedLength());
  return false;
