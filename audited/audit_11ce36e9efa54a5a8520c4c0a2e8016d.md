### Title
Unbounded Per-Request Log Flooding via `tracePostExecution()` When Tracing Enabled with Default Empty Status Filter

### Summary
When `hiero.mirror.web3.evm.trace.enabled=true`, `MirrorOperationActionTracer.tracePostExecution()` emits a `log.info()` line for every single EVM opcode executed. Because `stateFilterCheck()` returns `false` (do not skip) for all states when the `status` set is empty (the default), and no per-request log-count cap exists, an unprivileged user submitting a single max-gas contract call can force the logging subsystem to emit millions of lines — potentially hundreds of megabytes — of log data per request.

### Finding Description
**Exact code path:**

`TraceProperties.java` lines 29–31:
```java
public boolean stateFilterCheck(State state) {
    return !getStatus().isEmpty() && !getStatus().contains(state);
}
```
When `status` is the default empty `HashSet`, `!getStatus().isEmpty()` evaluates to `false`, so the entire expression is `false` — meaning "do not filter, proceed to log." This is the default out-of-the-box behaviour when tracing is enabled.

`MirrorOperationActionTracer.java` lines 51–86:
```java
if (!traceProperties.isEnabled()) { return; }          // only guard: enabled flag
if (traceProperties.stateFilterCheck(frame.getState())) { return; }  // no-op with empty status
// ... falls through to:
log.info("type={} operation={}, callDepth={}, contract={}, sender={}, recipient={}, "
       + "remainingGas={}, revertReason={}, input={}, output={}, return={}", ...);
```
Every opcode execution in the EVM calls `tracePostExecution()`. With the default empty `status` and `contract` sets, every opcode unconditionally reaches `log.info()`.

**Root cause:** `stateFilterCheck()` is designed as an opt-in filter: it only filters when the operator has explicitly populated `status`. With the default empty set it is a no-op, so enabling tracing with default settings means zero filtering — one `log.info()` per opcode, unbounded.

**Exploit flow:**
1. Operator sets `hiero.mirror.web3.evm.trace.enabled=true` (leaves `status` and `contract` at defaults).
2. Attacker deploys a contract containing a tight loop using cheap opcodes (e.g., `JUMPDEST` at 1 gas, `PUSH1`/`POP` at 3 gas each).
3. Attacker POSTs to `/api/v1/contracts/call` with `gas=15000000` (the documented `maxGasLimit`).
4. The EVM executes up to ~5 million opcodes (at 3 gas each) within that single request.
5. Each opcode triggers one `log.info()` line containing type, operation name, call depth, contract address, sender, recipient, remaining gas, revert reason, input hex, output hex, and return hex — roughly 200–400 bytes per line.
6. Result: ~1–2 GB of log output generated by a single unauthenticated HTTP request.

**Why existing checks are insufficient:**
- The `enabled` flag is the only global guard; once set by the operator it applies to all callers equally.
- `stateFilterCheck()` with empty `status` is a no-op (returns `false` for every state).
- `contractFilterCheck()` with empty `contract` is likewise a no-op.
- There is no per-request opcode-log counter, no log-rate limiter, and no log-size cap inside `tracePostExecution()`.
- The throttle (`gasPerSecond`, `requestsPerSecond`) limits request rate but does not limit log output per request.

### Impact Explanation
A single request can saturate disk I/O, fill the log partition, or overwhelm a centralised log aggregator (e.g., Elasticsearch, Loki). This can degrade or crash the web3 service for all users (availability impact) and may cause loss of other operational log data (integrity impact on observability). The scope classification of "griefing with no economic damage" is accurate: the attacker pays nothing (contract calls on the mirror node are read-only simulations with no on-chain cost) while the operator bears disk and I/O costs.

### Likelihood Explanation
Any user who can reach the `/api/v1/contracts/call` endpoint can trigger this. No authentication, no special role, and no on-chain funds are required. The only prerequisite is that an operator has enabled tracing — a documented, supported configuration option intended for debugging and monitoring. Operators enabling tracing for production diagnostics are the realistic target. The attack is trivially repeatable at the rate allowed by `requestsPerSecond` (default 500 RPS), multiplying the impact further.

### Recommendation
1. **Add a per-request log-line cap** inside `tracePostExecution()`: maintain a counter in `ContractCallContext` and stop logging once a configurable threshold (e.g., 10,000 lines) is reached per request, emitting a single warning that the limit was hit.
2. **Warn on dangerous default configuration**: if `enabled=true` and both `status` and `contract` are empty, log a startup warning that all opcodes for all contracts will be traced.
3. **Consider making `status` default to `{CODE_EXECUTING}` or similar** so that the filter is non-trivially restrictive by default when tracing is enabled.
4. **Add log-rate limiting** (e.g., using SLF4J rate-limited logger or a token-bucket guard) inside `tracePostExecution()` to bound aggregate log throughput.
