### Title
Unauthenticated Account Enumeration via Unbounded `EntityIdInput.num` Enables DB Exhaustion DoS on GraphQL Endpoint

### Summary
The `account(input: AccountInput!)` GraphQL query accepts an `EntityIdInput` with `num: Long! @Min(value: 0)` — no upper bound — and requires no authentication. The graphql module contains zero rate limiting (unlike the web3 module which uses bucket4j), so any unauthenticated external caller can issue unlimited sequential queries with incrementing `num` values, each triggering a synchronous `entityRepository.findById()` database call, exhausting the DB connection pool and causing the service to lose its database connection.

### Finding Description

**Schema — no upper bound on `num`:**
`graphql/src/main/resources/graphql/common.graphqls`, lines 169–178:
```graphql
input EntityIdInput {
    shard: Long! = 0 @Min(value: 0)
    realm: Long! = 0 @Min(value: 0)
    num: Long! @Min(value: 0)   // ← no @Max, no upper bound
}
``` [1](#0-0) 

**Controller — no authentication, no rate limiting:**
`graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java`, lines 32–45: the `account()` method is a plain `@QueryMapping` with only `@Valid` bean validation. No `@PreAuthorize`, no throttle check, no IP guard. [2](#0-1) 

**Service — direct synchronous DB hit per request:**
`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 25:
```java
return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
```
Every query issues a `SELECT * FROM entity WHERE id = ?` against the database. [3](#0-2) 

**GraphQL configuration — only structural per-query limits, no request-rate limit:**
`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 42–48: only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` are configured. These are per-query parse-time checks; they do not limit how many requests per second a single IP can send. [4](#0-3) 

**No rate limiting exists in the graphql module at all:**
A grep across all `graphql/**/*.java` files returns zero matches for `throttle`, `rateLimit`, `RateLimit`, `bucket4j`, or `Bucket`. The bucket4j-based `ThrottleConfiguration` / `ThrottleManagerImpl` exist exclusively in the `web3` module and are not wired into the graphql module. [5](#0-4) 

The design document for GraphQL subscriptions explicitly notes: *"We should implement rate limiting in the load balancer"* — confirming that application-level rate limiting was never implemented. [6](#0-5) 

**Root cause:** The failed assumption is that an external load balancer will always be present and correctly configured to rate-limit the `/graphql/alpha` endpoint. The application itself provides no defence-in-depth.

### Impact Explanation

An attacker can:
1. **Enumerate all existing accounts** — timing/response differences between `null` (account absent) and a populated `Account` object reveal which `num` values correspond to live accounts. This is an information-disclosure issue on a public ledger.
2. **Exhaust the DB connection pool** — each request holds a JDBC connection for the duration of the `findById` call. At high request rates the HikariCP pool is saturated, causing all subsequent requests (including legitimate ones) to queue and time out, effectively partitioning the service from its database.
3. **Saturate DB CPU** — even though `findById` is a primary-key lookup, at tens of thousands of requests per second the aggregate I/O and CPU load on the PostgreSQL instance can degrade or crash it.

Severity: **High** (unauthenticated, no preconditions, full DoS achievable, account enumeration as a secondary impact).

### Likelihood Explanation

- **Preconditions:** None. No account, no API key, no special network position required.
- **Tooling:** A single `curl` loop or any HTTP benchmarking tool (e.g., `wrk`, `hey`, `ab`) suffices.
- **Repeatability:** The attack is stateless and trivially repeatable from multiple source IPs to bypass any upstream IP-based rate limiting.
- **Detection difficulty:** Each individual request is valid and returns HTTP 200; anomaly detection requires volume-based alerting.

### Recommendation

1. **Application-level rate limiting:** Add a bucket4j filter to the graphql module mirroring the pattern in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java` — a per-IP token bucket on the `/graphql/alpha` endpoint.
2. **Upper bound on `num`:** Add `@Max(value = <network_max_entity_num>)` to `EntityIdInput.num` in `common.graphqls` to reject obviously out-of-range values before they reach the DB.
3. **Query cost accounting:** Assign a non-trivial cost to the `account` resolver in `MaxQueryComplexityInstrumentation` so that batched alias queries are also bounded.
4. **Do not rely solely on the load balancer** for rate limiting; defence-in-depth requires the application to enforce its own limits.

### Proof of Concept

```bash
# Enumerate accounts 1–100000 and record which exist
for i in $(seq 1 100000); do
  curl -s -X POST https://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {entityId: {num: $i}}) { id } }\"}" &
done
# Running concurrently saturates the DB connection pool within seconds.
# Responses containing {"data":{"account":{"id":"..."}}} reveal live accounts.
# Responses containing {"data":{"account":null}} reveal absent accounts.
```

No credentials, no special headers, no prior knowledge required. The attack is reproducible against any deployment of the graphql module that lacks an external rate limiter.
