### Title
Connection Pool Exhaustion via Unauthenticated Concurrent Hook Storage Requests

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint in `HooksController` accepts up to 100 `key=eq:` parameters per request with no rate limiting. An unprivileged attacker can flood this endpoint with concurrent requests, each holding a HikariCP connection for up to 10 seconds while executing a `WHERE key IN (100 values)` query, exhausting the connection pool and making the database unreachable to all other services sharing it.

### Finding Description
**Code path:**
- `rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, lines 104–130 (`getHookStorage`) — accepts `List<SlotRangeParameter> keys` with `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` (100), no authentication, no rate limiting
- `rest-java/src/main/java/org/hiero/mirror/restjava/repository/HookStorageRepository.java`, lines 13–14 — `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse(long ownerId, long hookId, List<byte[]> key, Pageable pageable)` — Spring Data JPA generates `SELECT ... WHERE key IN (?, ?, ..., ?)` with up to 100 bind parameters
- `rest-java/src/main/java/org/hiero/mirror/restjava/common/Constants.java`, line 36 — `MAX_REPEATED_QUERY_PARAMETERS = 100`

**Root cause:** The `rest-java` module has no rate limiting on the hooks endpoint. The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`, `ThrottleProperties`) lives entirely in the `web3` module and is wired only to `ContractCallRequest` processing — it is never applied to `HooksController`. The only input bound is `@Size(max = 100)` on the key list, which still allows 100 keys per request. With `statementTimeout = 10000ms` and HikariCP's default pool of 10 connections, an attacker needs only 10 concurrent requests to saturate the pool.

**Failed assumption:** The design assumes that the `@Size(max = 100)` constraint and the statement timeout together are sufficient to prevent resource exhaustion. They are not: 10 concurrent requests × 10-second hold time = continuous pool saturation with no mechanism to reject excess traffic.

### Impact Explanation
When the HikariCP pool is fully occupied, all other services that share the same database connection pool (importer, other rest-java endpoints) receive `SQLTransientConnectionException: Connection is not available, request timed out` errors. This is functionally equivalent to a network partition from the application's perspective — the database becomes unreachable to legitimate traffic. The `RestJavaHighDBConnections` alert fires only after 5 minutes at >75% utilization, meaning the attacker has a wide window before any automated response.

### Likelihood Explanation
No privileges, authentication, or special knowledge are required. Any internet-accessible deployment is vulnerable. The attack is trivially repeatable with a basic HTTP concurrency tool (e.g., `ab`, `wrk`, or a simple async script). The attacker only needs to discover a valid `ownerId` and `hookId` (both are enumerable from the public hooks listing endpoint). The attack is sustainable indefinitely as long as the attacker maintains the concurrent request rate.

### Recommendation
1. **Add rate limiting to `HooksController`** — apply a per-IP or global token-bucket rate limiter (analogous to the `web3` `ThrottleManagerImpl`) to the `/api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint.
2. **Reduce `statementTimeout`** for the hook storage query — 10 seconds is excessive for a keyed lookup; 1–2 seconds is more appropriate.
3. **Increase pool size or add connection acquisition timeout** — configure `spring.datasource.hikari.maximum-pool-size` explicitly and set a short `connectionTimeout` so waiting threads fail fast rather than queuing indefinitely.
4. **Consider authentication or API key requirements** for the hooks endpoints if they are not intended for fully public access.
