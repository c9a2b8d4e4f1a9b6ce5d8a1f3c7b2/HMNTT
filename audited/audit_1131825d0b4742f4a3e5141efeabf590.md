### Title
Label Cardinality Explosion via Unmatched Routes Enables Synchronous Event-Loop Block in `metricsMiddleware`

### Summary
An unprivileged attacker can flood the mirror-node REST API with requests to arbitrary, never-matched paths, causing unbounded label cardinality growth in all four per-route OpenTelemetry instruments. When the metrics endpoint is subsequently scraped, `serializer.serialize(resourceMetrics)` at line 163 executes synchronously on the Node.js event loop, blocking it proportionally to the accumulated metric series count. This causes latency spikes and request timeouts for all concurrent connections, constituting a practical denial-of-service.

### Finding Description

**Cardinality explosion root cause — `toOpenApiPath`, lines 16–18:**

```js
if (!path) {
  if (!req.route) {
    path = req.path;   // raw attacker-controlled path for every 404
  } else {
    path = (req.baseUrl ?? '') + req.route?.path;
  }
}
``` [1](#0-0) 

For any request that does not match a registered Express route (`req.route` is `undefined`), the raw `req.path` — fully attacker-controlled — is used verbatim as the `path` label. There is no sanitisation, truncation, or cardinality cap applied at the application level.

**Label recording — lines 192, 207–210:**

```js
const labels = {method, path, code};
requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
``` [2](#0-1) 

Each unique `(method, path, code)` triple creates a new time-series in all four instruments. With `durationHistogram` configured with explicit bucket boundaries (default 4 buckets), each unique path produces 6 data points (4 buckets + sum + count) in that instrument alone, multiplying serialization work.

**Synchronous serialization — lines 161–164:**

```js
return exporter.collect().then(({resourceMetrics}) => {
  res.set('Content-Type', 'text/plain; charset=utf-8');
  res.send(serializer.serialize(resourceMetrics));   // ← synchronous, CPU-bound
});
``` [3](#0-2) 

`PrometheusSerializer.serialize()` is a fully synchronous, CPU-bound string-building loop over every accumulated metric series. It runs inside a microtask (`.then()` callback) on the single Node.js event loop thread. No `setImmediate`, worker thread, or streaming is used.

**Authentication is optional and disabled by default — lines 138–143:**

```js
const authenticate = (req) => {
  const {authentication, ...} = config.metrics.config;
  if (!authentication) {
    return true;   // no auth required when disabled
  }
  ...
};
``` [4](#0-3) 

When `authentication` is falsy (the default unless explicitly configured), any external client can directly request the metrics endpoint and trigger serialization. Even when authentication is enabled, the attacker still controls cardinality growth through normal API requests; the legitimate Prometheus scraper then triggers the blocking serialization.

**No existing cardinality guard in application code.** The OpenTelemetry SDK has a default cardinality limit of 2000 per instrument, but this is not configured or enforced at the application layer. With 4 instruments × up to 2000 series × 6 data points each = up to 48,000 data points serialized synchronously per scrape.

### Impact Explanation

The Node.js event loop is single-threaded. A synchronous CPU-bound operation of sufficient size starves all pending I/O callbacks, timer callbacks, and incoming connection handlers. With the cardinality ceiling reached across all four instruments, each metrics scrape serializes tens of thousands of data points in a tight string-concatenation loop, blocking the event loop for tens to hundreds of milliseconds. During this window, all in-flight HTTP requests stall, TCP keepalive probes go unanswered, and new connections queue up. Repeated scrapes (Prometheus default: every 15 seconds) or attacker-triggered scrapes (when auth is disabled) make this a sustained, repeatable DoS against the entire REST API service.

### Likelihood Explanation

- **Precondition 1**: The attacker needs only unauthenticated HTTP access to the public REST API — no credentials, no special headers.
- **Precondition 2**: The metrics endpoint must be reachable (same port, same process). In the default configuration `authentication = false`, the attacker can also directly trigger serialization.
- **Repeatability**: The cardinality explosion is persistent (metrics are never pruned within a process lifetime). A one-time burst of ~2000 unique-path requests saturates the cardinality limit permanently. Subsequent scrapes continue to block the event loop indefinitely.
- **Automation**: A trivial script generating `GET /api/v1/<random_string>` requests achieves full cardinality saturation in seconds.

### Recommendation

1. **Normalize unmatched-route paths**: Replace the `req.path` fallback with a fixed sentinel label (e.g., `"unknown"` or `"unmatched"`) to prevent attacker-controlled strings from becoming label values:
   ```js
   if (!req.route) {
     path = 'unmatched';  // fixed label, no cardinality growth
   }
   ```
2. **Enforce explicit cardinality limits** via the OTel SDK `cardinalityLimit` option on each instrument, set to a small, known-safe value (e.g., 100).
3. **Offload serialization** to a `worker_thread` or use streaming serialization so the event loop is not blocked regardless of series count.
4. **Restrict metrics endpoint access** by enabling `authentication` by default and/or binding the metrics endpoint to a separate internal-only port/interface.

### Proof of Concept

```bash
# Step 1: Saturate label cardinality with unique 404 paths (no auth required)
for i in $(seq 1 2000); do
  curl -s "http://<mirror-node-host>/api/v1/exploit_path_$i" &
done
wait

# Step 2: Trigger synchronous serialization (no auth if authentication=false)
# Measure event-loop block by timing a concurrent legitimate request
time curl -s "http://<mirror-node-host>/api/v1/accounts" &
curl -s "http://<mirror-node-host>/swagger/metrics/"

# Expected: the /api/v1/accounts request stalls for tens–hundreds of ms
# while serializer.serialize() runs synchronously on the event loop.

# Step 3: Confirm persistence — repeat Step 2 every 15s to match Prometheus
# scrape interval; the block recurs on every scrape indefinitely.
```
