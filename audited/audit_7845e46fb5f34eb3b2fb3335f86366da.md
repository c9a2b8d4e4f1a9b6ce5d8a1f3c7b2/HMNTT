### Title
Unbounded Cache Eviction via Unauthenticated GraphQL Query Flooding Degrades Service Availability

### Summary
The `CachedPreparsedDocumentProvider` uses the raw query string as the cache key with a hard cap of 1000 entries and no per-client or rate-limiting controls on the GraphQL endpoint. An unauthenticated attacker can continuously submit unique query strings to saturate and continuously evict the parse cache, forcing every legitimate user request to incur full parse-and-validate overhead on every call.

### Finding Description
**Code locations:**

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15: cache spec is `expireAfterWrite=1h,maximumSize=1000,recordStats` — hard cap of 1000 entries, 1-hour TTL.
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26: `cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput))` — cache key is the raw, unnormalized query string.
- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 42–48: the `PreparsedDocumentProvider` is wired in with no rate-limiting bean; the only instrumentation is `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`.

**Root cause:** The cache key is the literal query string. Any syntactic variation (inlined literal values, whitespace differences, alias names, field ordering) produces a distinct key. The GraphQL module has no rate-limiting or request-throttling mechanism — the `ThrottleManagerImpl`/`ThrottleConfiguration` beans exist only in the `web3` module and are not applied here. No authentication is required to reach the `/graphql/alpha` endpoint.

**Exploit flow:**
1. Attacker generates 1001+ structurally valid but syntactically unique queries, e.g., by inlining different numeric literals or varying whitespace within the 10,000-character parser limit.
2. Each request is accepted (passes `maxCharacters`, `maxTokens`, complexity, and depth checks) and inserts a new entry into the Caffeine cache.
3. Once the cache reaches 1000 entries, Caffeine evicts the least-recently-used legitimate entries.
4. Attacker continues at high throughput (no rate limit); the cache churns continuously.
5. Every legitimate user query is now a cache miss and must go through full parse + schema validation on every request.

**Why existing checks fail:**
- `maxCharacters(10000)` / `maxTokens(1000)` / `maxRuleDepth(100)` bound individual query size but do not prevent high-volume unique-query submission.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` fire *after* parsing and do not prevent cache insertion.
- There is no per-IP, per-connection, or global request-rate limit on the GraphQL endpoint.

### Impact Explanation
Every cache eviction forces a full parse-and-validate cycle for legitimate users. Under sustained attack the effective cache hit rate drops to zero, increasing per-request CPU cost and latency for all concurrent users. Because the attack is stateless and requires no credentials, it can be maintained indefinitely from a single host or distributed across many IPs. The result is measurable service degradation (increased latency, elevated CPU) that can escalate to effective denial of service under moderate load.

### Likelihood Explanation
The attack requires no privileges, no account, and no special knowledge beyond the public GraphQL schema. The endpoint is documented and accessible at `/graphql/alpha`. Generating 1000+ syntactically unique but valid queries is trivial (e.g., a loop varying a single inline integer argument). The absence of any rate-limiting bean in the `graphql` module means the attack can be executed at network speed with a simple script.

### Recommendation
1. **Add a request rate limiter** to the GraphQL module (e.g., a Bucket4j filter analogous to `ThrottleConfiguration` in `web3`) to cap requests per second per IP or globally.
2. **Normalize query strings before caching** (strip comments, canonicalize whitespace, sort fields) so syntactic variants map to the same cache key.
3. **Enforce query variables** — require clients to use GraphQL variables (`$id: ID!`) rather than inline literals; the query template is then stable and cache-friendly.
4. **Add a per-client cache-entry quota** or use a bounded cache with a shorter `expireAfterWrite` (e.g., 5–10 minutes) to reduce the eviction window.

### Proof of Concept
```bash
# Fill and continuously evict the 1000-entry cache with unique inline-literal queries
for i in $(seq 1 2000); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { balance } }\"}" \
    &
done
wait
# Legitimate users now always miss the cache; every request incurs full parse+validate overhead.
