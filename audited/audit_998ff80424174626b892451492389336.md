### Title
IP Spoofing via Unvalidated `X-Real-IP`/`X-Forwarded-For` Headers in `TracingMiddleware` Destroys Audit Trail for `/construction/submit`

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` and `X-Forwarded-For` HTTP headers without verifying the request originated from a trusted reverse proxy. Because `TracingMiddleware` is applied to every request including `/construction/submit`, any unprivileged external attacker can forge the source IP that appears in every audit log entry, making it impossible to attribute tampered transaction submissions to the real attacker.

### Finding Description
**Exact code path:**

`getClientIpAddress()` (lines 63–74) resolves the client IP in priority order:
1. `X-Real-IP` header — taken verbatim from the request
2. `X-Forwarded-For` header — taken verbatim from the request
3. `r.RemoteAddr` — the actual TCP peer address (only used as last resort) [1](#0-0) 

`TracingMiddleware` calls `getClientIpAddress()` before dispatching the request and logs the result for every non-internal path at `Info` level: [2](#0-1) 

In `main.go`, the server binds directly to a TCP port with no upstream proxy that would strip or validate these headers before they reach the middleware: [3](#0-2) 

**Root cause:** The function assumes that if `X-Real-IP` or `X-Forwarded-For` is present, it was set by a trusted reverse proxy. There is no allowlist of trusted proxy CIDRs, no check that `RemoteAddr` belongs to a known proxy, and no stripping of these headers for direct connections. The header values are accepted from any TCP peer, including the attacker themselves.

**Failed assumption:** The code assumes these headers are only reachable after passing through a proxy that controls their values. In reality the server is directly internet-facing (or reachable without a mandatory proxy), so any client controls these headers.

### Impact Explanation
Every `/construction/submit` request is logged at `Info` level with the forged IP as the attributed source. Operators relying on these logs to investigate tampered or malicious transaction submissions will see a fabricated IP (e.g., a Tor exit node, a competitor's IP, or `127.0.0.1`) instead of the real attacker address. This completely destroys the audit trail for the highest-risk endpoint in the Rosetta API — the one that broadcasts signed transactions to the Hedera network. Severity: **Medium–High** (audit integrity loss; no direct fund theft, but enables cover for on-chain attacks).

### Likelihood Explanation
Precondition: the attacker can reach the Rosetta port (default behavior; no authentication is required by the Rosetta spec). The exploit requires only adding a single HTTP header — trivially repeatable with any HTTP client (`curl`, Python `requests`, etc.). No special privileges, credentials, or network position are needed. Every single submission request can carry a different forged IP, making correlation across requests impossible.

### Recommendation
1. **Maintain a trusted-proxy allowlist.** Only honour `X-Real-IP`/`X-Forwarded-For` when `r.RemoteAddr` (the actual TCP peer) falls within a configured set of trusted proxy CIDRs (e.g., the cluster's ingress CIDR).
2. **Default to `r.RemoteAddr`.** If no trusted proxy is configured, always use `net.SplitHostPort(r.RemoteAddr)` and ignore the forwarded headers entirely.
3. **Validate header format.** If the headers are used, parse and validate them as IP addresses with `net.ParseIP`; reject non-IP values to prevent log injection.
4. Consider adopting a well-tested library (e.g., `realip` middleware from `chi` or `gorilla/handlers`) that implements RFC 7239 correctly with proxy trust configuration.

### Proof of Concept
```bash
# Attacker submits a transaction while forging the logged source IP
curl -s -X POST http://<rosetta-host>:<port>/construction/submit \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 8.8.8.8" \
  -d '{"network_identifier":{...},"signed_transaction":"..."}'

# Resulting log line (observed in stdout / Loki):
# level=info ... msg="8.8.8.8 POST /construction/submit (200) in 12ms"
# Real attacker IP (r.RemoteAddr) is never logged.
```

The forged IP `8.8.8.8` appears in the audit log; the real `RemoteAddr` is silently discarded. Repeating with a different `X-Real-IP` value on every request makes the attacker completely untraceable in the logs.

```go
func getClientIpAddress(r *http.Request) string {
	ipAddress := r.Header.Get(xRealIpHeader)

	if len(ipAddress) == 0 {
		ipAddress = r.Header.Get(xForwardedForHeader)
	}

	if len(ipAddress) == 0 {
		ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
	}

	return ipAddress
```
