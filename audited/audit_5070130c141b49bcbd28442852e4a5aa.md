### Title
Cache Miss Amplification via `unless = "#result == null"` on Optional-Returning `findRuntimeBytecode()` Enables Unbounded DB Hammering

### Summary
`findRuntimeBytecode()` in `ContractRepository` uses `@Cacheable` with `unless = "#result == null"`, but the method returns `Optional<byte[]>`. Spring's `CacheAspectSupport` unwraps `Optional` before evaluating the `unless` SpEL expression, so `#result` is the unwrapped content — `null` for `Optional.empty()`. This means every lookup for a non-existent `contractId` evaluates `unless` as `true` and is never stored in the Caffeine cache. An unprivileged attacker can repeatedly query non-existent contract IDs, forcing every request to execute a live SQL query against the database, bypassing the cache entirely.

### Finding Description
**Exact location:**
- File: `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18
- Cache config: `web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 22 (`expireAfterAccess=1h,maximumSize=1000`)

**Root cause:**
Spring's `CacheAspectSupport.execute()` unwraps `Optional` return values before evaluating `unless`:
```java
// Spring internals (CacheAspectSupport)
if (returnValue instanceof Optional<?> optional) {
    cacheValue = optional.orElse(null);   // null for Optional.empty()
}
// ...
if (!isUnless(context, cacheValue)) {    // #result == cacheValue == null → unless=true → skip cache
    cache.put(key, cacheValue);
}
```
For a non-existent `contractId`, the JPA query returns no rows → Spring Data returns `Optional.empty()` → unwrapped to `null` → `unless = "#result == null"` evaluates to `true` → the result is **never written to Caffeine**.

**Exploit flow:**
1. Attacker sends `eth_call` / `eth_estimateGas` targeting any contract address that resolves to a non-existent `contractId` (e.g., any random 20-byte EVM address not in the DB).
2. `ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode(entityId.getId())`.
3. Cache miss → SQL: `SELECT runtime_bytecode FROM contract WHERE id = :contractId` executes against PostgreSQL.
4. `Optional.empty()` returned → `unless` fires → result not cached.
5. Identical or varied non-existent IDs on the next request → same DB hit, no cache benefit.

**Why existing checks fail:**
- The `unless` condition was intended to prevent caching of "not found" results, but it achieves the opposite of what is needed: it permanently prevents negative-result caching, so the DB is hit on every single miss.
- The Caffeine cache (`maximumSize=1000, expireAfterAccess=1h`) only ever holds entries for *existing* contracts; it provides zero protection for the miss path.

### Impact Explanation
Every request for a non-existent contract ID executes a full indexed lookup (`WHERE id = :contractId`) on the `contract` table in PostgreSQL. At the default global rate limit of 500 req/s (`requestsPerSecond = 500` in `ThrottleProperties`), an attacker can sustain 500 uncached DB queries per second indefinitely. On a system with moderate baseline DB load (e.g., a few hundred queries/second from legitimate traffic), this easily exceeds a 30% increase in DB CPU and I/O. The attack also degrades cache effectiveness for legitimate users by consuming connection pool threads and DB I/O bandwidth.

### Likelihood Explanation
No authentication or account creation is required — the JSON-RPC endpoint (`/api/v1/contracts/call`) is publicly accessible. The attacker needs only to craft requests with addresses that do not correspond to deployed contracts (trivially achievable with random addresses). The global rate limiter (`rateLimitBucket`, 500 req/s) is a single shared token bucket with no per-IP partitioning, so a single attacker can consume the entire budget. The attack is fully repeatable and requires no special knowledge of the system beyond the public API.

### Recommendation
Replace `unless = "#result == null"` with `unless = "#result != null && #result.isEmpty()"` — or, more idiomatically for `Optional` return types, use `unless = "!#result.isPresent()"`. This caches negative results (empty Optionals) so that repeated lookups for non-existent contract IDs are served from Caffeine rather than hitting the database. Alternatively, change the return type to a nullable `byte[]` and keep `unless = "#result == null"` — but the Optional-based fix is less invasive.

```java
// Fix: cache empty Optional results too
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "!#result.isPresent()"   // only skip caching if Optional itself is absent (never true)
)
// Better: cache everything, including misses
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT
    // remove 'unless' entirely — let Caffeine store Optional.empty() as a negative-cache entry
)
```
