### Title
Unbounded `Timer` Registration via Mutable `Timer.Builder` Accumulation in `Downloader.verifySigsAndDownloadDataFiles`

### Summary
`streamVerificationMetric` is stored as a shared mutable `Timer.Builder` field. Each call to `.tag("success", ...)` in the processing loop mutates the builder by appending a new tag entry to its internal `Tags` list rather than replacing it. Because the accumulated tag set is unique on every iteration, each `.register(meterRegistry)` call creates a brand-new `Timer` instance in the registry, and old instances are never evicted, leading to unbounded memory growth.

### Finding Description
**Exact location:** `importer/src/main/java/org/hiero/mirror/importer/downloader/Downloader.java`

- Line 97: `streamVerificationMetric` is declared as `Timer.Builder` (a mutable object).
- Lines 141–143 (constructor): the builder is initialized with a fixed `"type"` tag but `.register()` is intentionally deferred.
- Lines 313–316 (`verifySigsAndDownloadDataFiles`, called on every polling cycle for every file in the batch):

```java
streamVerificationMetric
        .tag("success", String.valueOf(valid))   // mutates shared builder
        .register(meterRegistry)                 // registers a NEW Timer each time
        .record(Duration.between(startTime, Instant.now()));
```

**Root cause:** Micrometer's `Timer.Builder.tag(String, String)` calls `this.tags = this.tags.and(key, value)` internally, which appends to the builder's `Tags` field and returns `this`. It does **not** replace an existing tag with the same key. Because `streamVerificationMetric` is a shared instance field, each invocation of the loop body permanently grows the builder's tag list:

| Iteration | Builder's accumulated tags | Registered Meter.Id |
|---|---|---|
| 1 | `[type=RECORD, success=true]` | unique ID #1 |
| 2 | `[type=RECORD, success=true, success=false]` | unique ID #2 |
| 3 | `[type=RECORD, success=true, success=false, success=true]` | unique ID #3 |
| N | `[type=RECORD, success=true, …×N]` | unique ID #N |

Micrometer's `MeterRegistry` uses a `ConcurrentHashMap` keyed by `Meter.Id` (name + sorted tag list). Since every call produces a distinct `Meter.Id`, a new `Timer` is inserted into the map on every call and never removed.

**Contrast with the correct pattern** used in `BlockStreamVerifier` (lines 92–95), which uses `MeterProvider` via `.withRegistry()` and then `.withTags()` per call — this is the safe API that does not mutate a shared builder.

**Existing checks:** None. There is no guard on the number of registered meters, no deduplication of the `"success"` tag, and no use of `MeterProvider` or pre-registered `Timer` instances.

### Impact Explanation
The `MeterRegistry`'s internal `ConcurrentHashMap` grows by one entry per stream file processed. The Hedera network produces approximately one record file every two seconds (~43,200 files/day). Each `Timer` object carries histogram state, thread-local accumulators, and tag metadata — typically several hundred bytes to kilobytes. Over hours to days of continuous operation the heap fills with unreachable `Timer` instances, ultimately causing an `OutOfMemoryError` and crashing the importer process. Loss of the importer means the mirror node stops ingesting new blockchain state, breaking all downstream consumers (REST API, gRPC, web3 API).

### Likelihood Explanation
This is triggered by **normal, unattended operation** — no external actor is required. Every polling cycle that finds at least one new stream file exercises the vulnerable code path. A deployment that has been running for more than a few days will accumulate tens of thousands of orphaned `Timer` objects. An attacker with write access to the cloud storage bucket (e.g., a compromised consensus node or a misconfigured public bucket) could accelerate exhaustion by injecting large numbers of synthetic signature files, but the vulnerability is self-inflicted even without any adversarial input.

### Recommendation
Replace the shared mutable `Timer.Builder` with two pre-registered `Timer` instances (one per `success` value), exactly as `AbstractStreamFileParser` does for `parseDurationMetricFailure`/`parseDurationMetricSuccess`, or use the `MeterProvider` pattern already adopted in `BlockStreamVerifier`:

```java
// Constructor — safe, idiomatic pattern
private final MeterProvider<Timer> streamVerificationMetric =
    Timer.builder("hiero.mirror.importer.stream.verification")
         .description("...")
         .tag("type", streamType.toString())
         .withRegistry(meterRegistry);

// Usage in loop
streamVerificationMetric
    .withTag("success", String.valueOf(valid))
    .record(Duration.between(startTime, Instant.now()));
```

`MeterProvider.withTag()` looks up or creates a meter by its full tag set without mutating any shared state, so repeated calls with the same tags return the same `Timer` instance.

### Proof of Concept
1. Deploy the mirror node importer against any Hedera network (mainnet, testnet, or a local test environment).
2. Let it run continuously. Each polling cycle calls `verifySigsAndDownloadDataFiles` → `streamVerificationMetric.tag("success", ...).register(meterRegistry)`.
3. Attach a JVM heap profiler (e.g., `jmap -histo` or VisualVM). After N files processed, observe N distinct `io.micrometer.core.instrument.Timer` instances registered under the name `hiero.mirror.importer.stream.verification`, each with a unique tag combination containing N copies of the `"success"` key.
