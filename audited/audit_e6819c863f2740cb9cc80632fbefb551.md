### Title
Unauthenticated `account.publickey` Filter Forces Expensive Entity JOIN in `getTokenBalances()`, Enabling DB Griefing

### Summary
Any unauthenticated external user can supply an `account.publickey` query parameter to `GET /api/v1/tokens/:tokenId/balances`, which unconditionally triggers a JOIN between `token_account` and the `entity` table in `extractSqlFromTokenBalancesRequest()`. No authentication, privilege check, or rate limit guards this code path. An attacker flooding this endpoint with arbitrary public key values forces repeated expensive JOIN queries, degrading database performance for all users.

### Finding Description
In `rest/tokens.js`, `extractSqlFromTokenBalancesRequest()` (lines 630–635) builds a `joinEntityClause` purely based on whether the `publicKey` variable is truthy:

```js
// rest/tokens.js lines 630-635
const joinEntityClause = publicKey
  ? `join entity as e
    on e.type = '${entityTypes.ACCOUNT}' and
      e.id = ${tokenBalancesSqlQueryColumns.ACCOUNT_ID} and
      ${tokenBalancesSqlQueryColumns.ACCOUNT_PUBLICKEY} = $${params.push(publicKey)}`
  : '';
``` [1](#0-0) 

`publicKey` is set at line 611 directly from the raw filter value with no privilege check:

```js
case filterKeys.ACCOUNT_PUBLICKEY:
  publicKey = filter.value;
  break;
``` [2](#0-1) 

This `joinEntityClause` is then embedded into the final SQL at lines 677 (non-timestamp path) and 655 (timestamp path), forcing a JOIN on every execution when the parameter is present. [3](#0-2) 

The endpoint is registered publicly with no authentication gate:

```js
app.getExt(`${apiPrefix}/tokens/:tokenId/balances`, tokens.getTokenBalances);
``` [4](#0-3) 

The `authHandler` middleware only sets a custom result limit for authenticated users; it does not block or throttle unauthenticated requests:

```js
if (!credentials) {
  return;  // proceeds without any restriction
}
``` [5](#0-4) 

No rate-limiting middleware is applied to this route in `server.js`. [6](#0-5) 

### Impact Explanation
The JOIN between `token_account` (potentially millions of rows for popular tokens) and the `entity` table is materially more expensive than the non-JOIN path. Even with a LIMIT clause, the planner must evaluate the join condition across the working set before applying the limit. Sustained flooding of this endpoint with arbitrary `account.publickey` values (which need not match any real key) forces repeated expensive JOIN executions, consuming DB CPU and I/O, and degrading response times for all other API consumers. No economic damage occurs to on-chain state, placing this squarely in the griefing/DoS category.

### Likelihood Explanation
The attack requires zero privileges — no account, no API key, no authentication. The parameter is documented in the OpenAPI spec (`accountPublicKeyQueryParam`) and in test fixtures, making it trivially discoverable. A single attacker with a basic HTTP flood tool (e.g., `ab`, `wrk`, or a simple script) can sustain the attack indefinitely. The absence of any rate limiting or IP-based throttling in the middleware stack means there is no server-side defense. [7](#0-6) 

### Recommendation
1. **Rate-limit the endpoint** at the reverse-proxy or application layer, especially for requests containing `account.publickey`.
2. **Require authentication or an API key** to use the `account.publickey` filter, consistent with how `authHandler` already supports user-specific limits.
3. **Add a short-circuit**: if the supplied public key does not match any entity (a cheap pre-check query), return an empty result immediately rather than executing the full JOIN.
4. Consider **caching** public-key-to-account-id lookups (similar to how `balances.js` resolves public keys via `getAccountIdsByPublicKey`) so repeated identical keys hit a cache rather than the DB.

### Proof of Concept
```bash
# No credentials required. Flood with a random (non-matching) public key.
TOKEN_ID="0.0.1000"
FAKE_KEY="aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"
BASE="https://<mirror-node-host>/api/v1"

# Single request demonstrating the JOIN path is taken
curl -s "${BASE}/tokens/${TOKEN_ID}/balances?account.publickey=${FAKE_KEY}"

# Flood to degrade DB performance (adjust -n/-c as needed)
ab -n 10000 -c 50 \
  "${BASE}/tokens/${TOKEN_ID}/balances?account.publickey=${FAKE_KEY}"
```

Each request causes `extractSqlFromTokenBalancesRequest()` to build and execute the entity JOIN query. With 50 concurrent connections and a large `token_account` table, DB CPU and query latency for all users will increase measurably.
