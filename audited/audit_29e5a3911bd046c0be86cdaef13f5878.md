### Title
Unauthenticated HTTP Mirror-Node Response Allows MITM Node-Set Truncation in `newClient()`

### Summary
When `network=other`, `newClient()` calls `buildNetworkFromMirrorNodes()`, which issues a plain, unauthenticated HTTP GET to the operator-configured `mirrorRest` URL. The response JSON is accepted without any integrity check or minimum-node-count guard beyond "not empty." An attacker who can intercept the network path between the pinger and the mirror REST API can substitute a crafted `nodesEnvelope` containing only 1–2 entries, causing `hiero.ClientForNetwork(netmap)` to build a consensus-node map that omits ≥30% of real nodes, which then receive zero transactions from this pinger.

### Finding Description

**Code path:**

`pinger/config.go` line 37 — the `mirrorRest` URL is operator-supplied with no scheme enforcement; the documented example is `http://mirror-rest:5551` (plaintext HTTP). [1](#0-0) 

`pinger/mirror_node_client.go` lines 46–46 — the HTTP client is constructed with zero TLS configuration (no `Transport`, no certificate pinning, no HTTPS requirement). [2](#0-1) 

`pinger/mirror_node_client.go` lines 79–98 — the GET request is issued and the response body is decoded directly into `nodesEnvelope` with no signature, HMAC, or content-hash verification. [3](#0-2) 

`pinger/mirror_node_client.go` lines 127–129 — the **only** guard is `len(network) == 0`; returning 1 or 2 valid-looking node entries bypasses this check entirely. [4](#0-3) 

`pinger/sdk_client.go` lines 18–22 — the unvalidated `netmap` is passed directly to `hiero.ClientForNetwork(netmap)` with no cross-check against a known-good node list or minimum cardinality. [5](#0-4) 

**Root cause:** The code assumes the mirror REST endpoint is trusted and reachable only over a secure channel. Neither assumption is enforced in code. There is no TLS enforcement, no certificate pinning, no response signature, and no lower-bound on the number of returned nodes.

**Failed assumption:** The developer assumes `mirrorRest` will always be an HTTPS endpoint on a trusted internal network. The config flag description explicitly advertises `http://` as a valid value, and no runtime check rejects it. [1](#0-0) 

### Impact Explanation
The Hiero SDK's `ClientForNetwork` uses the supplied map as the **complete** set of consensus nodes for transaction routing. If the map contains only 1–2 entries, every transaction submitted by the pinger is sent exclusively to those nodes; all omitted nodes receive zero traffic from this pinger instance. For a network with, say, 10 consensus nodes, injecting 2 fake entries means 8 real nodes (80%) are starved of pinger-generated transactions. This satisfies the ≥30% processing-node shutdown threshold described in the scope without any brute-force action. [5](#0-4) 

### Likelihood Explanation
**Preconditions (all realistic):**
1. `HIERO_MIRROR_PINGER_NETWORK=other` — required for any custom/private deployment, which is the exact use-case this code path serves.
2. `HIERO_MIRROR_PINGER_REST` set to an `http://` URL — explicitly documented as the example value; common in internal Kubernetes service meshes where TLS termination is assumed to happen elsewhere.
3. Attacker controls a network path — achievable via ARP spoofing on the same L2 segment, a compromised sidecar/proxy, a rogue DNS entry, or a BGP hijack for routable addresses.

No credentials, no privileged access, and no brute force are required. The attack is repeatable on every pinger restart (startup is the only time `newClient()` is called). [6](#0-5) 

### Recommendation
1. **Enforce HTTPS:** Reject any `mirrorRest` URL whose scheme is not `https` at config-load time in `loadConfig()`.
2. **Pin or validate the TLS certificate:** Supply a custom `http.Transport` with a pinned CA or leaf certificate for the mirror REST endpoint.
3. **Enforce a minimum node count:** After `fetchMirrorNodeNetwork` returns, assert `len(network) >= expectedMinNodes` (configurable) and treat a smaller result as a retriable error.
4. **Cross-check against a static bootstrap list:** For `network=other`, allow operators to supply a known-good node list; reject any mirror response that omits nodes from that list.

### Proof of Concept

**Setup:**
- Deploy pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
- The real mirror REST API returns 10 node entries.

**Attack steps:**
1. Position a transparent HTTP proxy (e.g., `mitmproxy`) on the network path between the pinger pod and `mirror-rest:5551`.
2. Intercept the GET request to `/api/v1/network/nodes`.
3. Return a crafted response:
```json
{
  "nodes": [
    {"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"attacker-ip","port":50211}]},
    {"node_account_id":"0.0.4","service_endpoints":[{"ip_address_v4":"attacker-ip","port":50211}]}
  ],
  "links":{"next":null}
}
```
4. Restart (or wait for the first start of) the pinger process.

**Result:**
- `fetchMirrorNodeNetwork` returns a 2-entry map; `len(network) == 2 > 0` so no error is raised.
- `hiero.ClientForNetwork(netmap)` builds a client aware of only 2 nodes.
- All subsequent transactions are routed exclusively to `attacker-ip`; the 8 real consensus nodes receive zero transactions from this pinger for the lifetime of the process. [7](#0-6)
