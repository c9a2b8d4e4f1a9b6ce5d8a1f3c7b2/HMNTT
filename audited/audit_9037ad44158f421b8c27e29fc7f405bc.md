### Title
Unbounded ArrayList Pre-allocation via Gas-Derived Capacity Enables Heap Exhaustion in Opcode Tracer Endpoint

### Summary
In `OpcodeServiceImpl.processOpcodeCall()`, an `OpcodeContext` is constructed with `new ArrayList<>((int) params.getGas() / 3)` where `params.getGas()` is sourced directly from the stored transaction's gas limit with no upper-bound cap applied. An attacker who first submits a Hedera contract transaction with the maximum gas limit (15,000,000) can then repeatedly invoke the opcode tracer endpoint with that transaction hash, causing each request to pre-allocate an `ArrayList` backed by a 5,000,000-entry `Object[]` array (~20–40 MB per request). A small number of concurrent requests is sufficient to exhaust JVM heap memory.

### Finding Description

**Exact code path:**

- `OpcodeServiceImpl.java`, line 67:
  ```java
  final var opcodeContext = new OpcodeContext(opcodeRequest, (int) params.getGas() / 3);
  ``` [1](#0-0) 

- `OpcodeContext.java`, lines 47–52:
  ```java
  public OpcodeContext(final OpcodeRequest opcodeRequest, final int opcodesSize) {
      ...
      this.opcodes = new ArrayList<>(opcodesSize);
  }
  ``` [2](#0-1) 

**Root cause:** `params.getGas()` is populated from the stored transaction record via `getGasLimit()`:
```java
private Long getGasLimit(EthereumTransaction ethereumTransaction, ContractResult contractResult) {
    return ethereumTransaction != null ? ethereumTransaction.getGasLimit() : contractResult.getGasLimit();
}
``` [3](#0-2) 

There is no cap applied to this value before it is divided by 3 and passed as the `ArrayList` initial capacity. The failed assumption is that gas values from stored transactions are "safe" sizes for memory pre-allocation — they are not, because Hedera's maximum gas limit is 15,000,000, yielding `new ArrayList<>(5_000_000)`.

**Exploit flow:**
1. Attacker submits a Hedera contract transaction with `gasLimit = 15,000,000` (a normal, permitted operation).
2. Attacker obtains the resulting transaction hash from the Hedera network.
3. Attacker sends concurrent GET requests to `/api/v1/contracts/results/{hash}/opcodes` with `Accept-Encoding: gzip`.
4. Each request reaches `processOpcodeCall()`, which calls `new ArrayList<>(5_000_000)` — allocating a backing `Object[]` of 5M references (~20 MB with compressed oops, ~40 MB without).
5. With enough concurrent requests, JVM heap is exhausted, causing `OutOfMemoryError` and service crash.

**Why existing checks fail:**

- `throttleManager.throttleOpcodeRequest()` is called before processing, but `ThrottleManager` is only an interface: [4](#0-3) 
  The throttle operates on request rate, not on per-request memory footprint. Multiple concurrent requests from different source IPs or within the throttle window can all reach the allocation simultaneously.

- The `Accept-Encoding: gzip` header check is trivially satisfied by any HTTP client. [5](#0-4) 

- `OpcodesProperties.enabled` defaults to `false`, but when operators enable the feature (which is its intended use), the vulnerability is fully exposed with no additional protection. [6](#0-5) 

### Impact Explanation
Each request with a max-gas transaction pre-allocates ~20–40 MB of heap before any EVM execution begins. A typical mirror node JVM heap of 512 MB–2 GB can be exhausted with 13–100 concurrent requests. This causes `OutOfMemoryError`, crashing or severely degrading the web3 service, which processes all contract calls and queries — not just opcode tracing. This matches the stated scope of "shutdown of ≥30% of network processing nodes."

### Likelihood Explanation
The attacker requires no special privileges — only the ability to submit a standard Hedera contract transaction (costs a small amount of HBAR, publicly available) and make HTTP GET requests to the mirror node. The transaction hash is publicly observable on the Hedera network. The attack is repeatable: the same transaction hash can be used indefinitely. The only barrier is that the operator must have enabled the opcode tracer feature, which is a documented operational configuration.

### Recommendation
Apply a hard cap on the `ArrayList` initial capacity before construction:

```java
private static final int MAX_OPCODES_CAPACITY = 100_000; // tunable

final int capacity = (int) Math.min(params.getGas() / 3, MAX_OPCODES_CAPACITY);
final var opcodeContext = new OpcodeContext(opcodeRequest, capacity);
```

Additionally, consider enforcing a per-IP or global concurrency limit specifically for the opcode endpoint (separate from the general throttle), and document the memory implications of enabling this feature so operators can size JVM heap accordingly.

### Proof of Concept
1. Submit a Hedera smart contract call transaction with `gasLimit = 15_000_000` using the Hedera SDK or REST API. Record the resulting transaction hash `TX_HASH`.
2. Start 20 concurrent HTTP requests:
   ```bash
   for i in $(seq 1 20); do
     curl -s -H "Accept-Encoding: gzip" \
       "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes" &
   done
   wait
   ```
3. Monitor JVM heap via JMX or GC logs. Each request allocates `new ArrayList<>(5_000_000)` (~20–40 MB) before EVM replay begins.
4. Observe heap exhaustion (`OutOfMemoryError`) and service unavailability within seconds of the concurrent burst.
