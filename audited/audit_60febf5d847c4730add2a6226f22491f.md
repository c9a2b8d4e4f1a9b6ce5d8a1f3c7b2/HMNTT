### Title
Unauthenticated High-Frequency Flooding of `/api/v1/network/stake` Causes Repeated SQL Subquery Execution and Database CPU Exhaustion

### Summary
The `GET /api/v1/network/stake` endpoint in the `rest-java` module is publicly accessible with no rate limiting, no per-IP throttling, and no result caching. Every request triggers `NetworkStakeRepository.findLatest()`, which executes a native SQL query containing a correlated subquery (`select max(consensus_timestamp) from network_stake`) against the live database. An unprivileged attacker can flood this endpoint to exhaust database CPU and connection pool resources, degrading service for all legitimate users.

### Finding Description
**Exact code path:**

`NetworkController.getNetworkStake()` (lines 126–130) calls `networkService.getLatestNetworkStake()`, which calls `networkStakeRepository.findLatest()` (lines 52–55 of `NetworkServiceImpl`). `findLatest()` executes the following native SQL on every invocation with no caching layer:

```sql
select *
from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
``` [1](#0-0) 

**Root cause — three compounding failures:**

1. **No rate limiting in `rest-java`**: The entire throttle/rate-limit infrastructure (`ThrottleManager`, `ThrottleConfiguration`, `ThrottleProperties`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is wired only into `ContractController`. The `NetworkController` in `rest-java` has no `ThrottleManager` dependency and no equivalent mechanism. [2](#0-1) [3](#0-2) 

2. **No result caching**: A grep across all `rest-java/src/main/java/**/*.java` finds zero `@Cacheable` annotations on `NetworkServiceImpl.getLatestNetworkStake()` or `NetworkStakeRepository.findLatest()`. Compare this to the `grpc` module's `NodeStakeRepository`, which uses `@Cacheable` on an equivalent query. [4](#0-3) [5](#0-4) 

3. **ShallowEtagHeaderFilter does not protect the database**: `RestJavaConfiguration` registers a `ShallowEtagHeaderFilter` for `/api/*`. This filter computes an ETag *after* the full response is generated — meaning the database query executes unconditionally on every request before the ETag comparison occurs. An attacker simply omits the `If-None-Match` header. [6](#0-5) 

**Exploit flow:**
- Attacker sends `GET /api/v1/network/stake` at high frequency (thousands of RPS) from one or more IPs.
- Each request passes through `NetworkController` → `NetworkServiceImpl` → `NetworkStakeRepository.findLatest()`.
- Each call issues two sequential database operations: `select max(consensus_timestamp) from network_stake` (subquery) followed by the outer `select *`.
- Under sustained load, the database connection pool is saturated, CPU spikes, and query latency increases for all other endpoints sharing the same database.

### Impact Explanation
The `network_stake` table is updated only once per staking period (approximately daily), yet the query re-executes the `max()` subquery on every HTTP request. Under a flood attack, this creates sustained, unnecessary database load. Because the `rest-java` service shares a PostgreSQL instance with other mirror node components, connection pool exhaustion and CPU saturation degrade all API endpoints — not just `/network/stake`. The impact is classified as a medium-severity availability issue (griefing/DoS with no economic damage to network participants).

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero knowledge of the system beyond the public API specification (documented in `openapi.yml`). The endpoint is listed in the public OpenAPI spec with no security requirements. Any attacker with a basic HTTP flood tool (e.g., `wrk`, `ab`, `k6`) can reproduce this. The absence of any IP-based or global rate limiting in the `rest-java` stack makes this trivially repeatable and sustainable. [7](#0-6) 

### Recommendation
Apply one or more of the following mitigations:

1. **Add `@Cacheable` to `getLatestNetworkStake()`** in `NetworkServiceImpl` with a short TTL (e.g., 60 seconds). Since `network_stake` is updated only once per staking period, caching is safe and eliminates the per-request DB hit entirely.
2. **Add a global rate limiter to `rest-java`** analogous to the `web3` `ThrottleConfiguration`, using Bucket4j or Spring's `HandlerInterceptor`, applied at the servlet filter level.
3. **Rewrite the query** to use `ORDER BY consensus_timestamp DESC LIMIT 1` instead of the correlated subquery, and ensure an index exists on `consensus_timestamp`. This reduces per-query cost but does not eliminate the flooding vector.
4. **Deploy an ingress-level rate limiter** (e.g., HAProxy, nginx, or a Kubernetes ingress rate-limit annotation) as a defense-in-depth measure.

Each of the 500 concurrent connections triggers a fresh execution of the `select max(consensus_timestamp) from network_stake` subquery with no caching or throttle gate to stop it.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)

        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();

                .capacity(rateLimit)

                .build();
