### Title
Unauthenticated EVM Address Lookup Exhausts DB Connection Pool via NFT Endpoint

### Summary
The `/api/v1/accounts/:idOrAliasOrEvmAddress/nfts` endpoint accepts any valid 40-hex-character EVM address from any unauthenticated caller and unconditionally issues a database query against the `entity` table for each request. With no application-level rate limiting on the REST API and a default DB connection pool of only 10 connections, an attacker flooding the endpoint with distinct non-existent EVM addresses can saturate the pool, causing all other REST API queries — including those serving transaction data — to fail or queue indefinitely.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 15 registers the route with no authentication or rate-limiting middleware: [1](#0-0) 

`rest/controllers/accountController.js` line 91 unconditionally calls `EntityService.getEncodedId()` with the raw user-supplied path parameter: [2](#0-1) 

`rest/service/entityService.js` lines 120–124: when the input is a valid EVM address (passes `EntityId.isValidEntityId()` and has a non-null `evmAddress`), `getEncodedId()` calls `getEntityIdFromEvmAddress()`: [3](#0-2) 

`rest/service/entityService.js` lines 90–94: `getEntityIdFromEvmAddress()` executes a live DB query (`SELECT id FROM entity WHERE deleted <> true AND evm_address = $1`) and throws `NotFoundError` when no row is found — but only after the query completes: [4](#0-3) 

The static query used: [5](#0-4) 

**Root cause:** Every EVM-address-shaped request triggers a real DB round-trip with no short-circuit, no negative-result cache, and no rate gate.

**Failed assumption:** The design assumes that callers are well-behaved or that infrastructure-level rate limiting is always present. Neither is guaranteed for the REST API.

### Impact Explanation

The default DB connection pool for the REST API is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`) with a `statement_timeout` of 20 000 ms and a `connectionTimeoutMillis` of 20 000 ms: [6](#0-5) [7](#0-6) 

An attacker holding all 10 connections with slow or queued queries blocks every other REST API handler — including `/api/v1/transactions` — from obtaining a connection. Legitimate clients receive connection-timeout errors or indefinitely queued responses, effectively denying access to gossip transaction data served by the mirror node.

### Likelihood Explanation

- **No authentication required**: the route is fully public.
- **Trivial to trigger**: any syntactically valid 40-hex string (e.g., `0xdeadbeef...`) is accepted as an EVM address.
- **Easily automated**: a single script sending 10–20 concurrent HTTP GET requests with distinct random EVM addresses per second is sufficient to saturate the pool continuously.
- **No existing application-level rate limiting**: the REST API `server.js` middleware stack contains only `authHandler` (for response-limit customization), `requestLogger`, optional metrics, and optional Redis cache — no request-rate throttle: [8](#0-7) 

The Traefik rate-limit middleware exists only in the Rosetta chart, not the REST API chart, and is an optional infrastructure concern that may not be deployed.

### Recommendation

1. **Add application-level rate limiting** to the REST API (e.g., `express-rate-limit`) scoped per IP, applied globally or specifically to the accounts sub-router.
2. **Cache negative EVM address lookups**: extend the existing `entityId` cache (already configured with `maxAge=1800`, `maxSize=100000`) to store negative results for a short TTL (e.g., 60 s), preventing repeated DB hits for the same non-existent address.
3. **Increase the default pool size** or add a per-request concurrency cap so that EVM-address lookups cannot monopolize all connections.
4. **Validate EVM address format strictly** before issuing a DB query — addresses that decode to a known-invalid pattern (e.g., all-zero prefix indicating a numeric entity ID) can be short-circuited without a DB round-trip, as partially done in the GraphQL layer: [9](#0-8) 

### Proof of Concept

```bash
# Generate 200 distinct non-existent EVM addresses and flood the endpoint concurrently
for i in $(seq 1 200); do
  addr=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/accounts/0x${addr}/nfts" &
done
wait
# Result: DB pool saturated; concurrent legitimate requests to /api/v1/transactions
# receive connection timeout errors or hang for up to 20 seconds
```

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
