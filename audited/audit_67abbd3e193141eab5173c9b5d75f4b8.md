### Title
Unbounded Storage Trace Response Causes Server-Side Memory Exhaustion in `getContractOpcodes()`

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` accepts a `storage=true` parameter that causes `captureStorage()` to attach a full copy of all accumulated storage accesses to **every single opcode entry** in the trace. There is no cap on the number of opcodes returned, no cap on storage slots per opcode, and no pagination. The sole size-mitigation — requiring `Accept-Encoding: gzip` — only validates the client's request header declaration; it does not guarantee server-side HTTP compression is actually applied, and even when compression is active the server must first materialize the entire response in heap memory before compressing it.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (lines 52–68) accepts `storage` as an unauthenticated boolean query parameter with no privilege check: [1](#0-0) 

The only size-related guard is the `Accept-Encoding: gzip` header check: [2](#0-1) 

This check validates only that the **client declares** it accepts gzip. Spring Boot HTTP response compression requires separate server-side configuration (`server.compression.enabled=true`). No such configuration was found in the repository. Even when server-side compression is active, the full response object is built in JVM heap before compression occurs.

**Root cause — `captureStorage()` in `AbstractOpcodeTracer`:** [3](#0-2) 

For every opcode step during the EVM replay, `captureStorage()` iterates over **all** `getTxStorageUsage(true).accesses()` — the cumulative set of every storage slot touched so far — and builds a new `TreeMap<String,String>`. This map is then embedded in each `Opcode` object: [4](#0-3) 

The final response is assembled with no truncation: [5](#0-4) 

**Throttle is rate-only, not size-bounded:**

The default `opcodeRequestsPerSecond` is 1 RPS: [6](#0-5) 

This limits frequency but a single request can still trigger unbounded allocation.

**Exploit flow:**

1. Attacker identifies (or previously submitted) a historical transaction that: (a) executes many opcodes (e.g., a loop with 50 000 iterations) and (b) writes to many distinct storage slots (e.g., 500 slots via `SSTORE`).
2. Attacker sends: `GET /api/v1/contracts/results/{txHash}/opcodes?storage=true` with header `Accept-Encoding: gzip`.
3. The EVM replay fires `captureStorage()` at every opcode step. With 50 000 opcodes × 500 storage entries × ~130 bytes per entry ≈ **3.25 GB of `TreeMap` data** allocated in heap before any serialization.
4. Jackson serializes the full `OpcodesResponse` to JSON (potentially 5–10 GB uncompressed).
5. If server-side gzip is not configured, the full payload traverses the network. If it is configured, the server still OOMs during construction.

### Impact Explanation

- **Server-side OOM / JVM crash**: Constructing O(opcodes × storage_slots) `TreeMap` objects in heap can exhaust the JVM heap, triggering `OutOfMemoryError` and crashing the web3 service for all users.
- **Network buffer exhaustion**: Without confirmed server-side gzip, a multi-gigabyte uncompressed JSON response saturates the server's TCP send buffer and the client's receive buffer.
- **Client-side memory issues**: Even with gzip, the client must decompress and parse the response, potentially exhausting client memory.
- Severity: **High** — single unauthenticated request can crash the service.

### Likelihood Explanation

- No authentication or authorization is required; the endpoint is public when `hiero.mirror.web3.opcode.tracer.enabled=true`.
- The attacker only needs to know a valid transaction hash for a storage-heavy contract (publicly visible on-chain).
- The `Accept-Encoding: gzip` header is trivially added by any HTTP client (`curl -H "Accept-Encoding: gzip"`).
- The 1 RPS throttle does not prevent a single oversized request from causing OOM.
- Repeatability: the same transaction hash can be replayed indefinitely (it is a historical replay, not a state-changing call).

### Recommendation

1. **Enforce a hard cap on opcodes returned**: Add a configurable `maxOpcodes` property (e.g., 10 000) and truncate or paginate the `opcodes` list before serialization.
2. **Enforce a hard cap on storage entries per opcode**: Limit the `TreeMap` in `captureStorage()` to a configurable maximum (e.g., 256 slots).
3. **Confirm and enforce server-side gzip**: Add `server.compression.enabled=true` and `server.compression.min-response-size=1024` to the Spring Boot configuration. The current header check is a client-declaration check only and provides no server-side protection.
4. **Stream the response**: Use `StreamingResponseBody` or chunked transfer to avoid materializing the entire response in heap before sending.
5. **Add a response-size circuit breaker**: Track serialized byte count during Jackson serialization and abort with HTTP 413 if a threshold is exceeded.

### Proof of Concept

```bash
# 1. Find a transaction hash for a contract that performs many SSTORE operations
TX_HASH="0x<hash_of_storage_heavy_tx>"

# 2. Send the request (no authentication required, only the gzip header)
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?storage=true" \
  -o /dev/null

# 3. Observe: server JVM heap exhaustion (OutOfMemoryError in logs),
#    or multi-GB response body transferred over the wire if gzip is not
#    configured server-side, or client process killed by OOM killer.
```

A contract suitable for triggering this can be deployed with a function like:
```solidity
function massStore(uint256 n) external {
    for (uint256 i = 0; i < n; i++) {
        assembly { sstore(i, i) }
    }
}
```
Calling `massStore(500)` produces a transaction with ~500 distinct storage slots and thousands of opcodes, sufficient to generate a response exceeding available heap.

```java
    protected Map<String, String> captureStorage(
            final MessageFrame frame, final OpcodeContext options, final ContractCallContext context) {
        if (!options.isStorage()) {
            return Collections.emptyMap();
        }

        try {
            if (context.getOpcodeContext().getRootProxyWorldUpdater() == null) {

                var worldUpdater = frame.getWorldUpdater();
                var parent = worldUpdater.parentUpdater().orElse(null);
                while (parent != null) {
                    worldUpdater = parent;
                    parent = worldUpdater.parentUpdater().orElse(null);
                }

                if (!(worldUpdater instanceof RootProxyWorldUpdater rootProxyWorldUpdater)) {
                    // The storage updates are kept only in the RootProxyWorldUpdater.
                    // If we don't have one -> something unexpected happened and an attempt to
                    // get the storage changes from a ProxyWorldUpdater would result in a
                    // NullPointerException, so in this case just return an empty map.
                    return Collections.emptyMap();
                }

                context.getOpcodeContext().setRootProxyWorldUpdater(rootProxyWorldUpdater);
            }

            final var rootProxyWorldUpdater = context.getOpcodeContext().getRootProxyWorldUpdater();
            final var updates = rootProxyWorldUpdater
                    .getEvmFrameState()
                    .getTxStorageUsage(true)
                    .accesses();

            if (updates.isEmpty()) {
                return Collections.emptyMap();
            }

            final var result = new TreeMap<String, String>();
            for (final var storageAccesses : updates) {
                for (final var access : storageAccesses.accesses()) {
                    final var key = hexCache.get(access.key(), Bytes::toHexString);
                    if (!result.containsKey(key)) {
                        final var value = access.writtenValue() != null
                                ? hexCache.get(access.writtenValue(), Bytes::toHexString)
                                : hexCache.get(access.value(), Bytes::toHexString);
                        result.put(key, value);
                    }
                }
            }
            return result;

        } catch (final ModificationNotAllowedException e) {
            return Collections.emptyMap();
        }
    }
```
