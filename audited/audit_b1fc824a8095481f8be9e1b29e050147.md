### Title
Unthrottled Per-Gap DB Query Amplification via Unbounded Concurrent Subscriptions on Gappy Topics

### Summary
In `TopicMessageServiceImpl.java`, the `incomingMessages()` method pipes every live message through `concatMap(t -> missingMessages(topicContext, t))`. Whenever a live message arrives with a sequence-number gap, `missingMessages()` issues an unthrottled DB query (`retrieve(newFilter, false)`). Because there is no limit on the number of concurrent subscriptions an unprivileged caller may open, an attacker can open N subscriptions to any topic that experiences gaps, causing N unthrottled DB queries per gapped message — a multiplicative amplification of database load with no privilege required.

### Finding Description
**Exact code path:**

`incomingMessages()` — line 120:
```java
return topicListener.listen(newFilter).concatMap(t -> missingMessages(topicContext, t));
```
Every live message emitted by `topicListener.listen()` is synchronously passed to `missingMessages()`.

`missingMessages()` — lines 152–177:
```java
if (last == null || topicContext.isNext(current)) {
    return Flux.just(current);          // no DB query — consecutive message
}
long numMissingMessages = current.getSequenceNumber() - last.getSequenceNumber() - 1;
if (numMissingMessages <= -1) {
    return Flux.empty();                // no DB query — duplicate
}
// GAP DETECTED — issues an unthrottled DB query:
return topicMessageRetriever.retrieve(newFilter, false).concatWithValues(current);
```
The `false` argument to `retrieve()` explicitly bypasses throttling. The initial historical retrieval at line 63 uses `retrieve(filter, true)` (throttled); gap-fill queries do not.

**Root cause and failed assumption:**
The design assumes gaps are rare, transient events (importer briefly down). It does not account for an attacker multiplying the per-gap query cost by holding many concurrent subscriptions open. There is no subscription-count cap, no per-subscriber query rate limit, and no back-pressure on gap-fill queries.

**Exploit flow:**
1. Attacker opens N gRPC connections (e.g., N = 500–2000) and calls `subscribeTopic` on the same topic ID on each.
2. Each connection independently runs `incomingMessages()` → `concatMap(t -> missingMessages(..., t))`.
3. When the live stream delivers a message with a sequence-number gap (a normal occurrence during importer lag or catch-up), each of the N subscriptions independently detects the gap and fires an unthrottled `retrieve()` DB query.
4. Result: N simultaneous unthrottled DB queries per gapped message, sustained for as long as the attacker holds connections open and gaps continue to arrive.

**Why existing checks fail:**
- `subscriberCount` (line 48) is a Micrometer gauge for observability only — it enforces no limit.
- `GrpcProperties` exposes no `maxSubscribers` or per-IP connection cap.
- The `throttled=false` path in `retrieve()` has no internal rate gate.
- The `filter` on line 74–77 (sequence deduplication) operates per-subscription and does not reduce cross-subscription query fan-out.

### Impact Explanation
Each gapped live message causes N unthrottled DB queries, where N is the number of open subscriptions. At N = 1000 and a topic receiving 10 gapped messages per second, this is 10,000 unthrottled DB queries/second from a single attacker with no credentials. This degrades database response times for all mirror-node consumers (REST API, other gRPC subscribers), constituting a griefing-class denial-of-service against shared infrastructure. No economic value is stolen, consistent with the Medium scope classification.

### Likelihood Explanation
Preconditions are minimal: any caller who can reach the gRPC port can open arbitrarily many subscriptions. Gaps in the live stream are a normal operational condition (importer restart, network hiccup, catch-up after lag). The attacker does not need to create gaps — they only need to be present when gaps naturally occur, or subscribe to a high-volume topic where the importer periodically lags. The attack is fully repeatable and requires no special knowledge beyond the topic ID, which is public.

### Recommendation
1. **Cap concurrent subscriptions per IP / per topic** in `subscribeTopic()` using the existing `subscriberCount` infrastructure or a `ConcurrentHashMap<topicId, AtomicLong>` counter, rejecting new subscriptions above a configurable threshold.
2. **Apply throttling to gap-fill queries**: change the `retrieve(newFilter, false)` call at line 177 to `retrieve(newFilter, true)` so gap-fill queries are subject to the same rate gate as historical retrieval.
3. **Debounce or coalesce gap-fill queries per topic**: if multiple subscriptions detect the same gap simultaneously, a shared cache or in-flight deduplication layer should serve one DB query and fan out the result, rather than issuing N independent queries.
4. **Add a configurable `maxSubscribersPerTopic`** property to `GrpcProperties` and enforce it in `subscribeTopic()` before incrementing `subscriberCount`.
