### Title
Missing `length(file_data) <> 0` Guard in `latestFileContentsQuery` Inner CTE Enables Unprivileged User to Force Stale Exchange Rate via Empty FileAppend Targeting

### Summary
The `latestFileContentsQuery` CTE in `rest/service/fileDataService.js` omits the `length(file_data) <> 0` guard for `transaction_type = 19` (FileAppend) rows in its inner CTE, unlike the equivalent Java query in `FileDataRepository.java`. An unprivileged user can supply a `timestamp=lte:T` query parameter targeting a timestamp window where an empty FileAppend is the highest-ranked row, causing the inner CTE to anchor on that empty row's timestamp, the outer `string_agg` to produce a zero-byte buffer, protobuf parsing to throw, and `fallbackRetry` to silently serve an older exchange rate.

### Finding Description

**Exact code location:**

`rest/service/fileDataService.js`, lines 18–41 — the `latestFileContentsQuery` static field:

```sql
with latest_create as (
    select max(consensus_timestamp) as consensus_timestamp
    from file_data
    where entity_id = $1 and transaction_type in (17, 19)
    <filterInnerPlaceHolder>          -- ← user timestamp filter injected here
    group by entity_id
    ...
)
select
    max(f.consensus_timestamp) as consensus_timestamp,
    min(f.consensus_timestamp) as first_consensus_timestamp,
    string_agg(f.file_data, '' order by f.consensus_timestamp) as file_data
from file_data f
join latest_create l on f.consensus_timestamp >= l.consensus_timestamp
where f.entity_id = $1 and f.transaction_type in (16, 17, 19)
    and f.consensus_timestamp >= l.consensus_timestamp
    <filterOuterPlaceHolder>          -- ← same filter, table-alias-prefixed
group by f.entity_id
``` [1](#0-0) 

**Root cause — failed assumption:**

The inner CTE assumes that any row with `transaction_type in (17, 19)` is a meaningful anchor point. This is false: Hedera allows FileAppend transactions with zero-length `file_data` (e.g., the final chunk of a multi-chunk update). The Java equivalent explicitly excludes them:

```sql
and (transaction_type = 17 or (transaction_type = 19 and length(file_data) <> 0))
``` [2](#0-1) 

The JS query has no such guard in either the inner CTE or the outer `WHERE` clause.

**How `whereQuery` reaches the CTE:**

`getLatestFileDataContents` calls `buildWhereSqlStatement` on the caller-supplied `filterQueries.whereQuery`, producing a parameterized `where` string, then passes it to `getLatestFileContentsQuery` which string-replaces both placeholders: [3](#0-2) [4](#0-3) 

`buildWhereSqlStatement` uses positional `$N` parameters (no SQL injection), but the attacker controls the *value* of the timestamp parameter, which is sufficient.

**Exploit flow:**

1. Attacker queries `/api/v1/transactions?transactiontype=FILEAPPEND&account.id=0.0.112` (exchange rate file) to enumerate FileAppend transactions.
2. Attacker identifies timestamp `T` of a FileAppend row where `file_data` is empty (zero-length). Such rows are a normal on-chain occurrence.
3. Attacker issues `GET /api/v1/network/exchangerate?timestamp=lte:T`.
4. `buildWhereSqlStatement` produces `and consensus_timestamp <= $2` with param `T`.
5. Inner CTE evaluates `max(consensus_timestamp) … where transaction_type in (17,19) and consensus_timestamp <= T`. Because the empty FileAppend at `T` is the highest-ranked row in that window, the CTE returns `T`.
6. Outer query selects rows where `consensus_timestamp >= T and consensus_timestamp <= T` — exactly one row: the empty FileAppend at `T`.
7. `string_agg(f.file_data, '')` over a single empty buffer produces a zero-byte result.
8. `ExchangeRate` constructor calls `fromBinary(ExchangeRateSetSchema, exchangeRate.file_data)` on zero bytes → throws `FileDecodeError`. [5](#0-4) 

9. `fallbackRetry` catches the error, appends `consensus_timestamp < row.first_consensus_timestamp` (= `T`) to `whereQuery`, and retries — returning the exchange rate from the version *before* `T`. [6](#0-5) 

**Why existing checks are insufficient:**

`fallbackRetry` is designed to recover from *accidental* corruption, not adversarial targeting. It does not detect that the failure was caused by a user-supplied timestamp; it simply walks backward through history. The 10-attempt cap means a chain of empty FileAppend rows could exhaust retries and return `null` (HTTP 404/500 to callers).

### Impact Explanation

An attacker can force any consumer of `/api/v1/network/exchangerate` to receive an arbitrarily older exchange rate by selecting a timestamp window that straddles an empty FileAppend. Exchange rates are used to compute HBAR-to-USD fee conversions; a stale rate can cause fee underestimates or overestimates. If the attacker can identify a window where the prior rate was significantly different (e.g., during a rate update), they can systematically serve that stale rate to clients relying on the mirror node API for fee estimation, enabling economic manipulation of transaction cost calculations. Additionally, if multiple consecutive empty FileAppend rows exist, `fallbackRetry` exhausts its 10 attempts and returns `null`, causing a denial-of-service for the exchange rate endpoint.

### Likelihood Explanation

The endpoint is public and unauthenticated. Empty FileAppend rows are a legitimate on-chain artifact (multi-chunk file updates, no-op appends). The attacker needs only: (a) enumerate FileAppend transactions for file `0.0.112` via the public transactions API, (b) identify one with empty data, and (c) issue a single HTTP GET with `timestamp=lte:T`. No special privileges, no cryptographic material, and no rate-limiting bypass is required. The attack is repeatable and scriptable.
