### Title
Negative-Result Cache Poisoning in `findRuntimeBytecode()` Allows DoS on Newly Deployed Contracts

### Summary
`ContractRepository.findRuntimeBytecode()` uses `@Cacheable` with `unless = "#result == null"`, but the method returns `Optional<byte[]>`. When a contract does not exist, the return value is `Optional.empty()` — a non-null object — which passes the `unless` guard and is stored in the Caffeine cache with a TTL of up to 1 hour (`expireAfterAccess=1h`). An unprivileged attacker who can predict a future contract's entity ID (Hedera assigns sequential IDs) can pre-populate this cache with an empty-Optional entry before the contract is deployed, causing the EVM to treat the newly deployed contract as non-existent for the full cache lifetime.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring's `unless` SpEL expression `"#result == null"` evaluates to `false` when the result is `Optional.empty()`, because `Optional.empty()` is a non-null singleton object. The intent was to avoid caching "not found" results, but the condition only guards against a literal `null` return. Since the method signature returns `Optional<byte[]>`, a missing contract produces `Optional.empty()`, which is unconditionally stored in the Caffeine cache.

**Cache configuration** (`CacheProperties.java`, line 22):
```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```
`expireAfterAccess=1h` means the poisoned entry survives for 1 hour from the last access — and repeated attacker queries reset the timer, potentially extending the window indefinitely.

**Exploit flow:**

1. Attacker observes the current highest Hedera entity ID (publicly visible on-chain or via the mirror node REST API).
2. Attacker predicts the entity ID `X` that will be assigned to the next contract deployment.
3. Attacker submits a `eth_call` / `contractCall` request that causes the EVM to resolve bytecode for entity ID `X` (e.g., a call to a contract that internally calls `X`, or a direct call to address `X`).
4. `ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode(X)`. The DB has no row for `X`, so the query returns `Optional.empty()`. The Spring cache stores this `Optional.empty()` under key `X`.
5. The legitimate owner deploys a contract; it receives entity ID `X`.
6. Any subsequent EVM execution that needs bytecode for `X` calls `findRuntimeBytecode(X)`, which returns the cached `Optional.empty()` without hitting the DB.
7. `ContractBytecodeReadableKVState.readFromDataSource()` receives `Optional.empty()` and returns `null` (via `.orElse(null)`). The EVM interprets `null` bytecode as a non-existent contract, causing all calls to `X` to fail or behave as if calling an EOA.

**Why existing checks fail:**

The only guard is `unless = "#result == null"`. `Optional.empty() != null` in Java, so this check never fires for the "not found" case. There is no secondary validation in `ContractBytecodeReadableKVState.readFromDataSource()` or in `ReadableKVStateBase.get()` that would bypass the Spring cache and re-query the DB.

### Impact Explanation

Any contract deployed at a predictable entity ID can be rendered invisible to the EVM for up to 1 hour (or longer if the attacker keeps refreshing the cache entry). All `eth_call`, `eth_estimateGas`, and internal EVM `CALL`/`STATICCALL` operations targeting the contract will fail as if the contract does not exist. This is a targeted, repeatable denial-of-service against specific contracts with no cost beyond submitting read-only API calls.

### Likelihood Explanation

Hedera entity IDs are strictly sequential and publicly observable via the mirror node REST API and on-chain receipts. No privileged access, private keys, or gas is required — only unauthenticated read-only `eth_call` requests. The attack is trivially scriptable: poll the current max entity ID, fire a pre-poisoning call, and repeat to keep the cache warm. The 1-hour `expireAfterAccess` window is long enough to cause significant disruption to any newly deployed contract.

### Recommendation

Change the `unless` condition to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

This ensures that only successful (non-empty) bytecode lookups are cached, while "not found" results always fall through to the database. Alternatively, switch the cache TTL policy from `expireAfterAccess` to a short `expireAfterWrite` (e.g., 1–2 seconds, consistent with the entity cache) so that stale negative entries expire quickly regardless.

### Proof of Concept

1. Query the mirror node REST API to find the current highest entity ID, e.g., `N`.
2. Send an `eth_call` to the web3 endpoint targeting address `0.0.(N+1)` (the next expected contract ID):
   ```json
   {"jsonrpc":"2.0","method":"eth_call","params":[{"to":"0x000000000000000000000000000000000000<N+1>","data":"0x"},"latest"],"id":1}
   ```
3. This triggers `findRuntimeBytecode(N+1)` → returns `Optional.empty()` → cached for 1 hour.
4. Deploy a contract; it receives entity ID `N+1`.
5. Immediately call the newly deployed contract:
   ```json
   {"jsonrpc":"2.0","method":"eth_call","params":[{"to":"0x000000000000000000000000000000000000<N+1>","data":"<valid_selector>"},"latest"],"id":2}
   ```
6. Observe that the call fails or returns empty data as if the contract does not exist, despite the contract being live on-chain.
