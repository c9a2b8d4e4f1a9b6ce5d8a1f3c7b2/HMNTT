### Title
Unauthenticated Multi-Query Amplification DoS via Timestamp-Filtered GET /contracts/:contractId

### Summary
An unauthenticated external user can send `GET /api/v1/contracts/:contractId?timestamp=lte:<value>` to trigger a deterministic 3-query database chain per request: a UNION across `entity` and `entity_history`, followed by a correlated subquery against `file_data`. The REST API has no rate limiting, and the response cache is trivially bypassed by varying the timestamp value. Under concurrent load, this exhausts the database connection pool and degrades mirror node availability.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 13 registers the endpoint with no authentication: [1](#0-0) 

`getContractById` in `rest/controllers/contractController.js` lines 707–737 is the handler: [2](#0-1) 

**Query 1+2 — UNION trigger:**
At line 194, when `timestampConditions.length !== 0` (i.e., any `?timestamp=` filter is present), `getContractByIdOrAddressContractEntityQuery` builds a UNION across both `entity` and `entity_history` tables: [3](#0-2) 

**Query 3 — FileDataService trigger:**
At line 731, if `contract.file_id` passes `isValidUserFileId()` (num > 1000, which is true for all user-deployed contracts), `FileDataService.getFileData()` fires a third, correlated subquery against the `file_data` table: [4](#0-3) 

The `getFileDataQuery` is a correlated subquery that scans `file_data` twice — once in the subquery to find the latest create/update timestamp, and once in the outer aggregation: [5](#0-4) 

**`isValidUserFileId` is not a security gate:** [6](#0-5) 
It only excludes null/empty values and system entities (num ≤ 1000). Any user-deployed contract with a file-based bytecode passes this check.

**Cache bypass:**
The response cache key is `md5(req.originalUrl)`: [7](#0-6) 
Varying the timestamp by 1 nanosecond per request (e.g., `timestamp=lte:1000000000.000000001`, `...000000002`, etc.) produces a unique cache key each time, bypassing the cache entirely.

**No rate limiting on the REST API:**
The REST middleware stack contains only: `authHandler`, `httpErrorHandler`, `openapiHandler`, `requestHandler`, `responseCacheHandler`, `responseHandler`: [8](#0-7) 
The throttle/rate-limit infrastructure found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3` module and does not apply to the REST API: [9](#0-8) 

**Root cause:** The design assumes that the response cache will absorb repeated requests, but the cache key is URL-based and trivially bypassed. There is no per-IP or global request rate limit on the REST API, so each unique timestamp value causes a full 3-query chain against the database.

### Impact Explanation
Each request with a timestamp filter causes: (1) a UNION scan of `entity` + `entity_history`, (2) a correlated double-scan of `file_data` via `string_agg`. The `file_data` table can be large for contracts with chunked bytecode. Under concurrent load, the database connection pool (`rest/dbpool.js`) is exhausted, causing all REST API endpoints to queue or fail — not just the contracts endpoint. This degrades the mirror node's availability for all consumers (wallets, dApps, block explorers).

### Likelihood Explanation
Preconditions are minimal: the attacker needs only a valid contract ID with a `file_id` (num > 1000), which is publicly discoverable via `GET /api/v1/contracts`. No credentials, API keys, or special network access are required. The attack is repeatable, scriptable, and requires no brute force — a single loop varying the timestamp nanosecond field is sufficient to defeat the cache. The attack is stateless and can be distributed across multiple source IPs.

### Recommendation
1. **Add rate limiting to the REST API** at the middleware layer (e.g., express-rate-limit per IP, or a token-bucket at the load balancer/ingress level) specifically for endpoints that trigger multi-query chains.
2. **Normalize the cache key** for timestamp-filtered requests by rounding or bucketing the timestamp to a coarser granularity (e.g., nearest second), so that requests for the same contract within the same second share a cache entry.
3. **Decouple the `FileDataService.getFileData()` call** from the synchronous request path, or add a short-lived in-process cache keyed on `(file_id, timestamp_bucket)` to avoid repeated DB hits for the same file.
4. **Set a DB query timeout** on `pool.queryQuietly` calls to bound the worst-case duration of each request.

### Proof of Concept

```bash
# Step 1: Discover a contract with a file_id (num > 1000)
CONTRACT_ID=$(curl -s "https://<mirror-node>/api/v1/contracts?limit=1" \
  | jq -r '.contracts[0].contract_id')

# Step 2: Flood with unique timestamp values to bypass cache
# Each request triggers: UNION(entity, entity_history) + file_data correlated subquery
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}?timestamp=lte:1000000000.00000${i}" &
done
wait

# Step 3: Observe REST API degradation / connection pool exhaustion
# Other endpoints (e.g., /api/v1/transactions) will also begin timing out
curl -v "https://<mirror-node>/api/v1/transactions?limit=1"
```
