### Title
Unauthenticated Balance Snapshot Timestamp Enumeration via `getOneAccount()` Timestamp Filter

### Summary
`getOneAccount()` in `rest/accounts.js` accepts an open-ended `timestamp=lte:X` filter with no range-size validation (`validateRange=false`). Each response unconditionally leaks the exact `consensus_timestamp` of the matched `account_balance` snapshot as `balance.timestamp`. By iterating with a decreasing upper-bound, any unauthenticated caller can walk backwards through every snapshot row in the `account_balance` table, reconstructing the complete snapshot schedule and the full balance history of any account.

### Finding Description

**Code path:**

`getOneAccount()` calls `parseTimestampFilters` with `allowOpenRange=true` and `validateRange=false`: [1](#0-0) 

This means a single `timestamp=lte:X` parameter is accepted with no 7-day window enforcement (contrast with other endpoints where `validateRange=true`).

The resulting query is passed to `getAccountBalanceTimestampRange()`: [2](#0-1) 

Inside `getAccountBalanceTimestampRange()`, `getOptimizedTimestampRange()` sets `lowerBound` to the first day of the month before `min(now, upperBound)` and `upperBound` to the user-supplied value. The function then executes: [3](#0-2) 

The returned `upper` is the **exact `consensus_timestamp`** of a real row in `account_balance`. This value is then wired directly into the SQL as `balance_timestamp`: [4](#0-3) 

And `processRow()` emits it verbatim in the JSON response: [5](#0-4) 

**Enumeration loop:**
1. `GET /api/v1/accounts/0.0.2?timestamp=lte:<very_large_ns>` → response contains `balance.timestamp = T1` (the most recent snapshot).
2. Convert `T1` to nanoseconds, subtract 1: `GET /api/v1/accounts/0.0.2?timestamp=lte:<T1_ns - 1>` → response contains `balance.timestamp = T2`.
3. Repeat until the response returns no balance (no more snapshots found).

**On `lower == upper`:** `lower = getFirstDayOfMonth(upper)` and `upper` is the snapshot timestamp. They are equal only when the snapshot falls exactly on the first nanosecond of a UTC month — a rare but possible condition. The enumeration attack does not require this; it works in all cases because `upper` is always leaked.

**Existing checks that fail:**
- `validateRange=false` explicitly disables the 7-day range guard.
- `allowOpenRange=true` removes the requirement for a lower bound, making a single `lte` filter sufficient.
- `getOptimizedTimestampRange` limits each DB query to ~2 months, but this does not prevent iterative month-by-month traversal.
- No authentication, no rate limiting, no pagination cursor required. [6](#0-5) 

### Impact Explanation
An attacker learns the precise nanosecond timestamp of every balance snapshot ever recorded in `account_balance`. Combined with the balance value returned in the same response, this reconstructs the complete HBAR balance history of any account on the network. It also reveals the internal snapshot cadence of the importer, which could be used to time other attacks or to detect anomalies in snapshot frequency.

### Likelihood Explanation
No credentials, no special network access, and no prior knowledge are required. The attack is a simple loop of HTTP GET requests with a decreasing integer parameter. It can be fully automated in under 20 lines of code. The only practical friction is the ~2-month window per query imposed by `getOptimizedTimestampRange`, which means traversing years of history requires O(months) requests — still entirely feasible.

### Recommendation
1. **Enforce a range limit for `getOneAccount`**: Change the `validateRange` argument to `true` (or a dedicated configurable limit) so that open-ended or arbitrarily large timestamp windows are rejected.
2. **Do not expose the raw snapshot timestamp**: Instead of returning the exact `upper` value as `balance.timestamp`, return the user-supplied upper bound or the account's `balance_timestamp` from the entity table, so the internal snapshot grid is not disclosed.
3. **Add rate limiting** on the `/accounts/:id` endpoint to slow automated enumeration.

### Proof of Concept

```bash
# Step 1 – get the most recent snapshot timestamp
curl "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=lte:9999999999999999999"
# Response: { "balance": { "timestamp": "1700000000.123456789", ... } }

# Step 2 – convert to nanoseconds: 1700000000123456789
# Subtract 1: 1700000000123456788
curl "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=lte:1700000000123456788"
# Response: { "balance": { "timestamp": "1699913600.000000000", ... } }  ← previous snapshot

# Repeat until balance is null (no more snapshots)
# A script can automate this to enumerate the full account_balance table
```

```javascript
const getOptimizedTimestampRange = (tsQuery, tsParams) => {
  let lowerBound = 0n;
  const neParams = [];
  let upperBound = constants.MAX_LONG;

  // Find the lower bound and the upper bound from tsParams if present
  tsQuery
    .split('?')
    .filter(Boolean)
    .forEach((query, index) => {
      const value = BigInt(tsParams[index]);
      // eq operator has already been converted to the lte operator
      if (query.includes(utils.opsMap.lte)) {
        // lte operator includes the lt operator, so this clause must before the lt clause
        upperBound = utils.bigIntMin(upperBound, value);
      } else if (query.includes(utils.opsMap.lt)) {
        // Convert lt to lte to simplify query
        const ltValue = value - 1n;
        upperBound = utils.bigIntMin(upperBound, ltValue);
      } else if (query.includes(utils.opsMap.gte)) {
        // gte operator includes the gt operator, so this clause must come before the gt clause
        lowerBound = utils.bigIntMax(lowerBound, value);
      } else if (query.includes(utils.opsMap.gt)) {
        // Convert gt to gte to simplify query
        const gtValue = value + 1n;
        lowerBound = utils.bigIntMax(lowerBound, gtValue);
      } else if (query.includes(utils.opsMap.ne)) {
        neParams.push(value);
      }
    });

  if (lowerBound > upperBound) {
    return {};
  }

  // The optimized range of [lower, upper] should overlap with at most two months, with the exception that when upper
  // is more than 1 month in the future, the range may cover more months. Since the partition maintenance job will
  // create at most one monthly partition ahead, it's unnecessary to adjust the upper bound.
  // With the assumption that the data in db is in sync with the network, in other words, the balance information is
  // update-to-date as of NOW in wall clock, the algorithm below sets lower bound to
  //   max(lowerBound from user, first day of the month before the month min(now, upperBound) is in)
  const nowInNs = utils.nowInNs();
  const effectiveUpperBound = utils.bigIntMin(upperBound, nowInNs);
  const optimalLowerBound = utils.getFirstDayOfMonth(effectiveUpperBound, -1);
  lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);

  return {lowerBound, upperBound, neParams};
```
