### Title
Unauthenticated Connection Pool Exhaustion via Concurrent GET /schedules Requests

### Summary
The `getSchedules()` handler in `rest/schedules.js` issues up to three concurrent `pool.queryQuietly` calls per request (one main query plus two parallel follow-up queries via `Promise.all`). The Node.js REST API has no rate-limiting middleware, and the default database connection pool is only 10 connections. An unauthenticated attacker sending a small number of concurrent requests can fully exhaust the pool, causing all subsequent legitimate requests to fail with a `DbError` after the 20-second connection timeout expires.

### Finding Description
**Exact code path:**

In `rest/schedules.js`, `getSchedules()` (lines 234–282):
- Line 241: `await pool.queryQuietly(schedulesQuery, params)` — first connection acquired
- Lines 261–264: `await Promise.all([pool.queryQuietly(entityQuery, entityIds), pool.queryQuietly(signatureQuery, entityIds)])` — two more connections acquired simultaneously

Each non-empty `getSchedules` request therefore holds up to **3 pool connections concurrently** during the `Promise.all` phase.

**Pool configuration** (`rest/dbpool.js` lines 7–16):
```
max: config.db.pool.maxConnections   // default = 10
connectionTimeoutMillis: 20000       // 20 seconds
statement_timeout: 20000             // 20 seconds
```
Default `maxConnections` is **10** (confirmed in `docs/configuration.md` line 556).

**`queryQuietly` behavior** (`rest/utils.js` lines 1518–1520): when `preQueryHint` is absent (all schedule queries), it calls `this.query(query, params)` — `node-postgres` pool.query() acquires a connection from the pool, runs the query, and releases it. The `Promise.all` at lines 261–264 causes two simultaneous `pool.query()` calls, each competing for a pool slot.

**No rate limiting exists**: A `grep` across all `rest/**/*.js` files for `rateLimit`, `rate-limit`, `express-rate`, `throttle`, or `limiter` returns **zero matches**. The throttle mechanism found (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the separate web3 Java API, not the Node.js REST API.

**Root cause**: The only guard in `getSchedules()` is `utils.buildAndValidateFilters()` (line 236), which validates parameter types/formats but imposes no request-rate constraint. There is no per-IP, per-endpoint, or global request-rate control on the Node.js REST API path.

**Exploit flow**:
1. Attacker opens ~4–5 concurrent HTTP connections and sends `GET /api/v1/schedules?account.id=0.0.X` with varying `X` values (to defeat any hypothetical caching).
2. Each request reaches `getSchedules()`, acquires 1 connection for the main query, then 2 more for the `Promise.all` — consuming ~3 pool slots for up to 20 seconds (statement timeout).
3. With 10 pool slots and ~3 consumed per request, only ~3–4 concurrent attacker requests saturate the pool.
4. All subsequent requests (legitimate or otherwise) queue waiting for a free connection. After `connectionTimeoutMillis` (20 s) they receive a `DbError`.
5. Legitimate gossip-monitoring clients polling `/schedules` for pending scheduled transactions receive errors and miss execution windows.

### Impact Explanation
The database connection pool is a shared, finite resource for the entire Node.js REST API process. Exhausting it denies service to **all** REST API endpoints, not just `/schedules`. Gossip-monitoring clients that rely on polling `/schedules` to detect pending scheduled transactions will miss the execution window (schedules expire after ~31 minutes by default, per `constants.THIRTY_ONE_MINUTES`). This constitutes a complete, unauthenticated denial-of-service against the mirror node REST API with a very low attack cost (~4–5 concurrent HTTP requests).

### Likelihood Explanation
No special privileges, credentials, or network position are required. The attack is trivially reproducible with `curl`, `ab`, `wrk`, or any HTTP load tool. The attacker does not need to know valid account IDs — any syntactically valid entity ID (e.g., `0.0.1`, `0.0.2`, …) is accepted. The 20-second statement timeout means each attacker request holds pool slots for up to 20 seconds, making sustained exhaustion easy to maintain. The attack is repeatable indefinitely.

### Recommendation
1. **Add rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit`) applied globally or specifically to expensive endpoints like `/schedules`.
2. **Increase `maxConnections`** from the default of 10 to a value appropriate for expected concurrency, or deploy a connection pooler (PgBouncer is already configured in the Helm chart for production but may not be present in all deployments).
3. **Reduce `statementTimeout`** for the `/schedules` endpoint to limit how long each connection is held.
4. **Add a concurrency limiter** (e.g., `p-limit` or a semaphore) inside `getSchedules()` to cap simultaneous in-flight DB queries.
5. Consider moving the two follow-up queries (entity + signature) into a single SQL query or CTE to reduce the per-request connection demand from 3 to 1.

### Proof of Concept
```bash
# Saturate the 10-connection pool with 5 concurrent long-running schedule requests
for i in $(seq 1 5); do
  curl -s "http://<mirror-node>:5551/api/v1/schedules?account.id=0.0.$((RANDOM % 10000))" &
done
wait

# Immediately attempt a legitimate request — will timeout after 20s with DbError
curl -v "http://<mirror-node>:5551/api/v1/schedules"
# Expected: 500 Internal Server Error with DbError after ~20 seconds
```

  const schedulesResponse = {schedules: [], links: {next: null}};
  res.locals[constants.responseDataLabel] = schedulesResponse;

  if (schedules.length === 0) {
    return;
  }

  const entityIds = schedules.map((s) => s.schedule_id);
  const positions = range(1, entityIds.length + 1)
    .map((i) => `$${i}`)
    .join(',');

    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;

    pool.queryQuietly(entityQuery, entityIds),
    pool.queryQuietly(signatureQuery, entityIds),
