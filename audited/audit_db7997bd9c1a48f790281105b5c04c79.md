### Title
Connection Pool Exhaustion DoS via Concurrent Requests Against Fixed-Size Pool with No Application-Level Rate Limiting

### Summary
`ConnectToDb()` in `rosetta/app/db/db.go` configures a fixed-size connection pool (`MaxOpenConnections` default: 100) with no application-level per-IP or per-request concurrency limiting. An unprivileged attacker can send 100 concurrent HTTP requests to any DB-backed Rosetta endpoint, holding all pool connections for up to the `statementTimeout` (default: 20 seconds), causing all subsequent requests to queue indefinitely and rendering the node unresponsive. The only infrastructure-level mitigations (Traefik `inFlightReq`/`rateLimit` middleware) are disabled by default.

### Finding Description

**Code path:**

`rosetta/app/db/db.go`, `ConnectToDb()`, line 33:
```go
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)  // default: 100
```

`rosetta/app/db/client.go`, `GetDbWithContext()`, lines 36–37:
```go
childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
return d.db.WithContext(childCtx), cancel
```

Every DB-backed repository method (e.g., `account.go` lines 113, 146, 229, 272, 317; `block.go`, `transaction.go`) calls `GetDbWithContext(ctx)` which creates a child context bounded by `statementTimeout` (default: 20 seconds). When all 100 pool connections are in use, Go's `database/sql` blocks new callers waiting for a free connection — there is no wait timeout configured and no queue depth limit.

**Root cause:** The pool size is a hard cap on concurrency, but there is no mechanism to limit how many concurrent HTTP requests can simultaneously acquire DB connections. The `statementTimeout` bounds individual query duration but does not bound concurrent connection occupancy.

**Exploit flow:**
1. Attacker sends 100 concurrent POST requests to any online-mode endpoint (e.g., `/block`, `/account/balance`) with parameters that trigger non-trivial queries (e.g., large block index, complex account alias lookups).
2. Each request calls `GetDbWithContext`, acquires a pool connection, and holds it for up to `min(statementTimeout=20s, httpWriteTimeout=10s)` = 10 seconds.
3. All 100 connections are occupied; the pool is exhausted.
4. Any new incoming request that reaches a DB call blocks indefinitely waiting for a connection.
5. Attacker repeats every ~10 seconds to maintain exhaustion.

**Why existing checks fail:**

- `statementTimeout = 20s` (Go context timeout, not PostgreSQL `SET statement_timeout`): Bounds individual query duration but does not prevent 100 simultaneous connections from being held concurrently.
- Helm Traefik middleware (`inFlightReq: amount: 5`, `rateLimit: average: 10`): Gated by `{{ if and .Values.global.middleware .Values.middleware }}` in `charts/hedera-mirror-rosetta/templates/middleware.yaml` lines 3–4. The chart default is `middleware: false` (`values.yaml` line 95), so these CRDs are **not created** in default deployments.
- HTTP server middleware chain in `main.go` (lines 217–219): Only `MetricsMiddleware`, `TracingMiddleware`, `CorsMiddleware` — no concurrency limiting.

### Impact Explanation
Complete denial of service for the Rosetta node. All API endpoints backed by the database (`/block`, `/account/balance`, `/network/status`, etc.) become unresponsive. Legitimate Coinbase Rosetta clients (exchanges, validators, indexers) cannot query chain state. For a network where Rosetta is used for exchange integrations, this disrupts deposit/withdrawal processing. The attack requires no authentication and no special privileges.

### Likelihood Explanation
Any external user can reach the Rosetta port (default 5700). Sending 100 concurrent HTTP requests is trivially achievable from a single machine using tools like `wrk`, `hey`, or a simple script. The attacker does not need to know valid account IDs — any syntactically valid request body triggers a DB query. The attack is repeatable and sustainable indefinitely at low cost. The absence of default infrastructure-level rate limiting makes this exploitable in any default deployment.

### Recommendation
1. **Application-level semaphore**: Add a `golang.org/x/sync/semaphore`-based middleware in the Rosetta HTTP handler that limits the number of concurrent in-flight DB-bound requests (e.g., to `MaxOpenConnections * 0.8`), returning HTTP 503 when the limit is exceeded.
2. **Pool wait timeout**: Call `sqlDb.SetConnMaxIdleTime(...)` and configure a connection wait timeout so requests waiting for a pool connection fail fast rather than queuing indefinitely.
3. **Enable Helm middleware by default**: Change `middleware: false` to `middleware: true` in `charts/hedera-mirror-rosetta/values.yaml` so the Traefik `inFlightReq` and `rateLimit` middleware are deployed by default.
4. **PostgreSQL-level timeout**: Embed `statement_timeout` in the DSN (e.g., `options=-c statement_timeout=20000`) in `types.go` `GetDsn()` so the timeout is enforced server-side, ensuring connections are released even if the Go context cancellation is delayed.

### Proof of Concept
```bash
# Requires: wrk or hey installed, Rosetta running at localhost:5700

# Step 1: Identify a valid network identifier
NETWORK=$(curl -s -X POST http://localhost:5700/network/list \
  -H 'Content-Type: application/json' \
  -d '{}' | jq -r '.network_identifiers[0]')

# Step 2: Flood with 100 concurrent block requests (triggers DB query)
hey -n 10000 -c 100 -m POST \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier": '"$NETWORK"', "block_identifier": {"index": 1}}' \
  http://localhost:5700/block &

# Step 3: In a separate terminal, observe that legitimate requests hang
time curl -s -X POST http://localhost:5700/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier": '"$NETWORK"'}' 
# Expected: request hangs or times out instead of returning immediately
# Confirms: pool exhausted, node unresponsive to new requests
```
