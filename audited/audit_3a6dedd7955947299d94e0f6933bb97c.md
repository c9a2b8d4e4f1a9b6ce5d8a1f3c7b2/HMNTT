### Title
Unauthenticated Alias-Based DB Query Flood via `getHooks()` — Mirror Node DoS

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks` endpoint accepts alias-format `ownerId` path parameters and, for every request, unconditionally issues a live SQL query against the `entity` table via `entityRepository.findByAlias()`. There is no caching and no rate limiting on this code path in the `rest-java` module. An unprivileged attacker can flood this endpoint with alias-format owner IDs to saturate the database connection pool, starving the importer of DB connections and halting ingestion of new transactions.

### Finding Description

**Exact code path:**

`HooksController.getHooks()` (line 81–102) accepts any `EntityIdParameter` as `{ownerId}`, including alias format (e.g., `0.0.HIQQEXWKW53RKN4W6XXC4Q232SYNZ3SZANVZZSUME5B5PRGXL663UAQA`). It delegates to `HookServiceImpl.getHooks()`:

```
// HookServiceImpl.java line 35
final var id = entityService.lookup(request.getOwnerId());
```

`EntityServiceImpl.lookup()` (lines 30–38) switches on the parameter type:

```java
case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
```

`EntityRepository.findByAlias()` (lines 13–14) executes a raw SQL query with no `@Cacheable` annotation:

```java
@Query(value = "select id from entity where alias = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByAlias(byte[] alias);
```

**Root cause:** Unlike the `web3` module's `EntityRepository` (which has `@Cacheable(cacheNames = CACHE_NAME_ALIAS, ...)` on its alias lookup) and unlike the `importer`'s `EntityIdServiceImpl` (which uses a `cacheLookup()` wrapper), the `rest-java` `EntityRepository.findByAlias()` has zero caching. Additionally, the `rest-java` module has no rate-limiting infrastructure — the bucket4j throttle (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not applied to any `rest-java` controller.

**Why existing checks fail:**
- `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on `hookId` query params limits query parameter count, but does nothing to limit request rate.
- `@Positive @Max(MAX_LIMIT)` on `limit` constrains result set size, not DB lookup cost.
- No `@RateLimiter`, no servlet filter, no Spring Security throttle is present in `rest-java` for this endpoint.
- Each request with a distinct alias value bypasses any hypothetical connection-level caching.

### Impact Explanation

Every alias-format request to `GET /api/v1/accounts/{alias}/hooks` consumes one database connection for the duration of the `SELECT id FROM entity WHERE alias = ?` query. Under sustained flood (e.g., thousands of requests/second from a botnet or even a single high-throughput client), the shared PostgreSQL connection pool is exhausted. The mirror node importer — which writes all ingested Hedera transactions to the same database — is then starved of connections and halts. New transactions stop being recorded by the mirror node. The REST API also becomes fully unresponsive. This is a complete mirror node DoS, not a Hedera network-level shutdown (the Hedera consensus nodes are unaffected), but it renders the mirror node unable to track new transactions.

### Likelihood Explanation

No authentication, no API key, no rate limit, and no CAPTCHA protects this endpoint. The alias format is publicly documented in the OpenAPI spec. Any attacker with a script and a moderate network connection can issue thousands of requests per second. The attack is trivially repeatable and requires zero privileges or prior knowledge beyond the public API spec.

### Recommendation

1. **Add caching** to `rest-java`'s `EntityRepository.findByAlias()` using `@Cacheable` (mirroring the `web3` module's pattern with a bounded TTL cache).
2. **Add rate limiting** to the `rest-java` module for all public endpoints, particularly those that trigger DB lookups by alias or EVM address. Apply a bucket4j or similar filter at the servlet/controller level.
3. **Short-circuit on cache miss** for non-existent aliases: return 404 immediately without a DB round-trip if the alias is syntactically valid but not in cache.

### Proof of Concept

```bash
# Generate a valid base32 alias (any 32-byte base32-encoded string)
ALIAS="HIQQEXWKW53RKN4W6XXC4Q232SYNZ3SZANVZZSUME5B5PRGXL663UAQA"

# Flood the endpoint with alias-format ownerIds (vary the alias to defeat any
# hypothetical connection-level caching)
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.${ALIAS}/hooks" &
done
wait
```

Each concurrent request causes `EntityServiceImpl.lookup()` → `entityRepository.findByAlias()` → a live `SELECT` against the `entity` table.
