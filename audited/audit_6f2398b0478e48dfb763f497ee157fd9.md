### Title
Unbounded Topic Message Collection in RedisPublisher Causes Main Parse Thread Blocking via Queue Saturation

### Summary
`RedisPublisher.onEnd()` enqueues an entire, unbounded `Collection<TopicMessage>` from a single stream file into a fixed-capacity `ArrayBlockingQueue` (default capacity 8). The single-threaded consumer calls `redisOperations.executePipelined()` over every message in the collection with no size cap. If the executor falls behind — due to a large collection or Redis latency — the queue fills and `topicMessagesQueue.put()` blocks the main parse thread indefinitely, stalling the entire mirror node importer.

### Finding Description

**Exact code path:**

`RedisPublisher.onEnd()` at [1](#0-0)  retrieves all `TopicMessage` objects accumulated for the current stream file from `parserContext.get(TopicMessage.class)` with no size limit, then attempts a non-blocking `offer()`. If the queue is full, it falls back to an **indefinitely blocking** `topicMessagesQueue.put(topicMessages)` on the main parse thread, with only a warning log.

The single-threaded executor at [2](#0-1)  calls `publish()`, which calls `redisOperations.executePipelined(callback(messages))`. The `callback()` at [3](#0-2)  iterates over every `TopicMessage` in the collection, issuing one `convertAndSend()` per message — all within a single pipeline call with no batching limit.

The queue capacity is hardcoded to a default of **8**: [4](#0-3) 

The main parse thread is the `synchronized` `RecordFileParser.parse()` method: [5](#0-4)  — blocking it halts all record file processing.

**Root cause:** There is no upper bound on the size of the collection enqueued per stream file, no timeout on `put()`, and the single-threaded consumer processes each collection in one unbounded `executePipelined()` call. The failed assumption is that Redis pipelining will always drain the queue faster than new stream files arrive.

**Exploit flow:**
1. Attacker submits a sustained flood of `ConsensusSubmitMessage` transactions to a public HCS topic, maximizing messages per record file.
2. Each record file's `onEnd()` enqueues a large `Collection<TopicMessage>` as a single queue item.
3. The single-threaded executor is occupied for an extended period processing the large pipeline for each item.
4. While the executor is busy, subsequent stream files' collections fill the 8-slot queue.
5. On the 9th enqueue attempt, `offer()` returns false and `put()` blocks the main parse thread indefinitely.

**Why existing checks fail:**
- The `offer()` → `put()` fallback at line 81–83 is not a guard; it is the mechanism that causes the block. [6](#0-5) 
- The `@SneakyThrows` on `onEnd()` means an `InterruptedException` from `put()` would be rethrown as an unchecked exception, potentially crashing the parse thread rather than recovering gracefully. [7](#0-6) 
- There is no per-file message count limit anywhere in the pipeline. [8](#0-7) 

### Impact Explanation
Blocking the main parse thread stalls the entire mirror node importer. No further record files are processed, causing the mirror node to fall behind the network. All downstream consumers — REST API, gRPC subscriptions, and any system depending on mirror node data — stop receiving updates. The `@Transactional` annotation on `parse()` means an open database transaction is held for the duration of the block, potentially exhausting the DB connection pool. This constitutes a denial-of-service against mirror node processing, matching the stated scope of "shutdown of ≥30% of network processing nodes without brute force."

### Likelihood Explanation
Any Hedera account holder can submit `ConsensusSubmitMessage` transactions to any public topic without special permissions. Transaction fees are low (fractions of a cent in HBAR). At the Hedera mainnet TPS ceiling, hundreds of topic messages can appear in a single ~2-second record file. The attacker does not need to compromise Redis or any infrastructure component — they only need to sustain a high message submission rate. The attack is repeatable and can be automated. The only cost barrier is HBAR for transaction fees, which is a low bar for a motivated attacker.

### Recommendation
1. **Drop instead of block**: Replace `topicMessagesQueue.put()` with a bounded-wait `offer(topicMessages, timeout, TimeUnit)` or simply drop and log an error, so the main parse thread is never blocked by a slow Redis consumer.
2. **Cap collection size per pipeline call**: In `publish()`, chunk the incoming collection into fixed-size batches (e.g., 1000 messages) and process each chunk as a separate `executePipelined()` call, preventing any single call from monopolizing the executor.
3. **Increase queue capacity or make it configurable** with a documented upper bound, and add a metric/alert when the queue approaches capacity.
4. **Decouple Redis publishing from the parse transaction**: Consider dropping messages rather than blocking the parse thread when Redis is slow, since Redis publishing is a best-effort notification layer, not a durability requirement.

### Proof of Concept
1. Create a public HCS topic on Hedera testnet/mainnet.
2. Using the Hedera SDK, submit a sustained burst of `ConsensusSubmitMessage` transactions at maximum allowed TPS (e.g., using multiple accounts to bypass per-account throttles).
3. Observe that record files processed by the mirror node importer contain large numbers of `TopicMessage` entries.
4. Monitor the mirror node importer logs: after 8 record files are queued, the log line `"topicMessagesQueue is full, will block until space is available"` appears and the main parse thread stops advancing (no further `"Successfully batch processed"` log lines).
5. Confirm by observing that the mirror node REST API `/api/v1/transactions` endpoint stops returning new transactions, and the mirror node's consensus timestamp lag grows unboundedly.
