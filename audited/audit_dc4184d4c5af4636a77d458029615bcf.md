### Title
Unvalidated `X-Forwarded-For` Header Logged Verbatim Enables IP Spoofing in Rosetta Tracing Middleware

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` accepts the raw `X-Forwarded-For` header value without parsing, splitting on commas, or validating that it is a single IP address. An unprivileged external attacker can supply an arbitrary string (e.g., `victim_ip, attacker_ip`) that is logged verbatim as the client IP, spoofing transaction attribution in audit logs. The deployment-level proxy configurations actively allow this header to pass through from untrusted clients.

### Finding Description
**Code path:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74.

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67 — raw, unvalidated
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }

    return ipAddress   // returned and logged as-is
}
```

`r.Header.Get()` in Go returns the entire header value as a single string. When the value is `victim_ip, attacker_ip`, the function returns that whole string. No comma-splitting, no IP format validation, and no sanitization is applied before the value is interpolated into the log message at line 52–53:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```

**Why proxy checks are insufficient:**

- **Traefik** (Kubernetes deployment): configured with `--entryPoints.web.forwardedHeaders.insecure` and `--entryPoints.websecure.forwardedHeaders.insecure` — this explicitly instructs Traefik to trust `X-Forwarded-For` from *all* sources, including external clients, passing the header through unmodified.
- **nginx** (docker-compose): uses `$proxy_add_x_forwarded_for`, which *appends* the real client IP to any existing `X-Forwarded-For` value. A client sending `X-Forwarded-For: victim_ip` causes nginx to forward `victim_ip, <real_client_ip>`. The Rosetta service receives and logs the entire string `victim_ip, <real_client_ip>`.

The test suite (`trace_test.go`, lines 56–58) only tests with a single clean IP, confirming multi-value handling was never considered.

### Impact Explanation
Every Rosetta API request log entry can have its source IP replaced with an arbitrary attacker-controlled string. This directly corrupts the audit trail used to attribute which IP address initiated which Rosetta transaction query. Log-based transaction history attribution tools (SIEM, forensic pipelines, compliance dashboards) will record the spoofed IP as the originator. Additionally, if the injected string contains newline characters (`\n`), an attacker can forge entirely new log lines (log injection), further corrupting the audit record. The impact is confined to log integrity — no funds are at risk — but the audit trail becomes unreliable.

### Likelihood Explanation
No authentication or special privilege is required. Any client that can reach the Rosetta HTTP endpoint can send an arbitrary `X-Forwarded-For` header. The Traefik `insecure` forwarded-headers configuration is the default in the provided Helm chart values, making this exploitable in standard deployments. The attack is trivially repeatable with a single `curl` command.

### Recommendation
Parse the `X-Forwarded-For` header as a comma-separated list and extract only the first (leftmost) element, then validate it is a well-formed IP address:

```go
if len(ipAddress) == 0 {
    xff := r.Header.Get(xForwardedForHeader)
    if xff != "" {
        // XFF is a comma-separated list; leftmost is the original client
        parts := strings.SplitN(xff, ",", 2)
        candidate := strings.TrimSpace(parts[0])
        if net.ParseIP(candidate) != nil {
            ipAddress = candidate
        }
    }
}
```

Additionally, configure Traefik to only trust `X-Forwarded-For` from known internal proxy CIDRs (remove `forwardedHeaders.insecure` and use `forwardedHeaders.trustedIPs` instead), and configure nginx to unconditionally overwrite (not append to) the header with `proxy_set_header X-Forwarded-For $remote_addr`.

### Proof of Concept
```bash
# Attacker spoofs victim's IP in Rosetta audit log
curl -H "X-Forwarded-For: 203.0.113.99" \
     http://<rosetta-host>/network/list

# Resulting log entry (logged by TracingMiddleware):
# level=info msg="203.0.113.99 GET /network/list (200) in 3ms"
# — victim IP 203.0.113.99 is recorded as the requester

# With nginx proxy (proxy_add_x_forwarded_for appends real IP):
curl -H "X-Forwarded-For: victim_ip" http://<nginx-host>/network/list
# Header forwarded to Rosetta: "X-Forwarded-For: victim_ip, <attacker_real_ip>"
# Logged: "victim_ip, <attacker_real_ip> GET /network/list (200) in 3ms"
# — malformed entry; attribution tools may parse "victim_ip" as the client

# Log injection via newline (if logger does not strip control chars):
curl -H $'X-Forwarded-For: 1.2.3.4\nlevel=info msg="forged log entry"' \
     http://<rosetta-host>/network/list
