### Title
Log Injection via Unsanitized `req.ip` from Attacker-Controlled `X-Forwarded-For` Header in `responseCacheCheckHandler()`

### Summary
The `responseCacheCheckHandler()` function logs `req.ip` directly into a log message without any sanitization. Because the Express application sets `trust proxy: true`, `req.ip` is derived from the attacker-controlled `X-Forwarded-For` request header. An unprivileged external attacker can inject arbitrary newlines, ANSI escape codes, or fake log entries by crafting this header, corrupting log output and potentially defeating log-based alerting or forensic analysis.

### Finding Description

**Exact code path:**

`rest/server.js` line 58 sets `trust proxy: true`: [1](#0-0) 

This causes Express to resolve `req.ip` from the leftmost value in the `X-Forwarded-For` request header — a value fully controlled by any external client.

`rest/middleware/responseCacheHandler.js` lines 84–86 interpolate `req.ip` directly into a log message with no sanitization: [2](#0-1) 

The custom `Logger` class in `rest/logger.js` constructs the log line at line 69 and writes it directly to `process.stdout` via a buffer at line 56, with zero sanitization of the message content: [3](#0-2) [4](#0-3) 

**Root cause:** The failed assumption is that `req.ip` is a trusted, well-formed IP address. With `trust proxy: true`, it is not — it is the raw, unsanitized leftmost value from `X-Forwarded-For`, which any client can set to an arbitrary string including newlines (`\n`), carriage returns (`\r`), and ANSI escape sequences.

**Exploit flow:**
1. Attacker sends a GET request to any cached endpoint with a crafted header:
   ```
   X-Forwarded-For: 1.2.3.4\n2024-01-01T00:00:00.000Z INFO req-fake 10.0.0.1 GET /api/v1/accounts from cache (ttl: 3600) in 1 ms: 200
   ```
2. Express resolves `req.ip` = `1.2.3.4\n2024-01-01T00:00:00.000Z INFO req-fake 10.0.0.1 GET /api/v1/accounts from cache (ttl: 3600) in 1 ms: 200`
3. The logger writes this verbatim to stdout, producing a forged log line indistinguishable from a real one.

**Why checks fail:** There are no checks. No IP format validation, no log message sanitization, and the custom logger performs no escaping of any kind before writing to stdout.

### Impact Explanation
An attacker can forge arbitrary log entries, making malicious activity appear as normal traffic or hiding real events. ANSI escape codes can corrupt terminal-based log viewers and break regex-based log parsers or SIEM ingestion pipelines. Injected fake "success" lines can cause alerting rules that look for error patterns to miss real incidents. This is a medium-severity finding: it does not directly compromise data integrity or confidentiality, but it undermines the reliability of the audit trail and incident response capability.

### Likelihood Explanation
Exploitation requires no authentication, no special privileges, and no prior knowledge beyond knowing the service is running. The only precondition is that a cached response exists for the targeted endpoint, which is the normal operational state when caching is enabled. The attack is trivially repeatable with a single HTTP request using any standard tool (curl, Python requests, etc.). Any internet-facing deployment is exposed.

### Recommendation
1. **Sanitize `req.ip` before logging**: strip or replace newline characters (`\n`, `\r`) and ANSI escape sequences before interpolating into log messages. A minimal fix:
   ```js
   const safeIp = (req.ip ?? '').replace(/[\r\n\x1b]/g, '_');
   logger.info(`${safeIp} ${req.method} ...`);
   ```
2. **Restrict `trust proxy`**: instead of `trust proxy: true` (trust all), configure it to trust only the specific number of known proxy hops (e.g., `app.set('trust proxy', 1)`), so `req.ip` cannot be freely spoofed by end clients.
3. **Sanitize in the logger**: add a sanitization step inside `Logger.#log()` in `rest/logger.js` to strip control characters from `msg` before writing, providing defense-in-depth for all log call sites.

### Proof of Concept

**Precondition:** Response caching is enabled and at least one endpoint response is cached in Redis.

```bash
# Inject a fake log line via X-Forwarded-For
curl -H "X-Forwarded-For: 1.2.3.4"$'\n'"2024-01-01T00:00:00.000Z INFO req-fake 10.0.0.1 GET /api/v1/accounts from cache (ttl: 3600) in 1 ms: 200" \
  http://<mirror-node-rest>/api/v1/accounts
```

**Expected result in stdout/logs:**
```
2024-01-01T00:00:01.000Z INFO req-abc123 1.2.3.4
2024-01-01T00:00:00.000Z INFO req-fake 10.0.0.1 GET /api/v1/accounts from cache (ttl: 3600) in 1 ms: 200 GET /api/v1/accounts from cache (ttl: 30) in 5 ms: 200
```

The second line is a fully forged log entry injected by the attacker, indistinguishable from a legitimate log line to automated parsers.
