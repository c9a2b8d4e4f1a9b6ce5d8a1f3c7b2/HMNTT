### Title
Gas Bucket Exhaustion via User-Controlled Gas Limit Enables Temporary Denial-of-Service

### Summary
An unprivileged user can submit a `POST /api/v1/contracts/call` request with a gas value calibrated to exactly consume the throttle bucket's remaining token capacity. Because gas tokens are deducted upfront from the shared `gasLimitBucket` before execution and only restored in the `finally` block of `doProcessCall()` after execution completes, a window exists during which the bucket is at zero and all concurrent legitimate requests are rejected with "Gas per second rate limit exceeded." This attack requires no authentication and can be repeated to sustain the denial-of-service.

### Finding Description

**Exact code path:**

`ContractController.call()` (lines 38–51) is the entry point:

```
throttleManager.throttle(request);          // (1) consumes tokens upfront
validateContractMaxGasLimit(request);       // (2) validates gas <= maxGasLimit
contractExecutionService.processCall(params); // (3) executes
``` [1](#0-0) 

Inside `ThrottleManagerImpl.throttle()`, the gas bucket is atomically drained:

```java
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
``` [2](#0-1) 

`scaleGas()` maps raw gas to bucket tokens via integer division by 10,000:

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) { return 0L; }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
``` [3](#0-2) 

Gas is only restored in the `finally` block of `doProcessCall()`, after execution completes:

```java
} finally {
    if (!estimate) {
        restoreGasToBucket(result, params.getGas());
    }
}
``` [4](#0-3) 

`restoreGasToBucket()` restores `min(gasLimit - gasUsed, gasLimitToRestoreBaseline)` tokens — but only after the EVM execution finishes: [5](#0-4) 

**Root cause and failed assumption:**

The design assumes that the execution window is short enough that temporary bucket depletion is harmless. This assumption fails because:

1. The bucket capacity is `scaleGas(7,500,000,000)` = **750,000 tokens**.
2. The maximum gas per request is `maxGasLimit = 15,000,000` → **1,500 tokens** per request.
3. The rate limit is **500 req/s**. [6](#0-5) [7](#0-6) 

An attacker who knows (or probes) the remaining bucket capacity R can send a single request with `gas = R × 10,000`, atomically draining the bucket to exactly 0. Any concurrent or immediately subsequent legitimate request then fails the `tryConsume` check and receives a throttle error — before the attacker's execution completes and gas is restored.

**Why existing checks are insufficient:**

- `validateContractMaxGasLimit` (gas ≤ 15,000,000) runs *after* `throttle()` has already consumed tokens. It does not prevent the bucket from being drained. [8](#0-7) 
- The rate-limit bucket (500 req/s) only limits request count, not gas-based exhaustion. [9](#0-8) 
- The `gasLimitBucket` uses `SynchronizationStrategy.SYNCHRONIZED`, making the `tryConsume` atomic but providing no protection against intentional exhaustion. [10](#0-9) 

### Impact Explanation

During the execution window of the attacker's request, the gas bucket is at zero. Any legitimate user whose request arrives in this window receives `ThrottleException("Gas per second rate limit exceeded.")` and their transaction is rejected. By chaining multiple high-gas requests (up to 500/s), the attacker can extend this window continuously. The impact is a targeted, sustained denial-of-service against all users of the `/api/v1/contracts/call` endpoint, with no cost to the attacker beyond sending HTTP requests.

### Likelihood Explanation

The attack requires no credentials, no on-chain assets, and no special knowledge beyond the public API. The remaining bucket capacity can be probed by binary-searching gas values until a request is throttled. The attack is fully automatable with a simple HTTP client. The `gasLimitBucket` is a single shared in-process bucket (not per-user), so one attacker affects all users. Repeatability is high.

### Recommendation

1. **Move `validateContractMaxGasLimit` before `throttle()`** so that invalid requests never consume bucket tokens.
2. **Throttle on actual gas used, not gas limit**: restore the full gas limit immediately after the throttle check, and only deduct actual gas used after execution. This eliminates the depletion window.
3. **Introduce per-IP or per-caller gas quotas** so a single source cannot exhaust the shared bucket.
4. **Cap the gas limit used for throttle accounting** to a lower value (e.g., a configurable `throttleGasCap`) independent of `maxGasLimit`.

### Proof of Concept

**Preconditions:** Access to `POST /api/v1/contracts/call`. Default configuration (`gasPerSecond=7_500_000_000`, `maxGasLimit=15_000_000`, `requestsPerSecond=500`).

**Steps:**

1. **Probe remaining capacity**: Send requests with decreasing gas values until `tryConsume` succeeds. The last successful gas value approximates the remaining bucket capacity R (in units of 10,000 gas per token).

2. **Drain the bucket**: Send a single request:
   ```json
   POST /api/v1/contracts/call
   { "to": "0x<any_valid_contract>", "gas": <R * 10000>, "data": "0x" }
   ```
   This atomically consumes all R remaining tokens. The bucket is now at 0.

3. **Observe throttling**: Immediately send a legitimate request from any other client:
   ```json
   POST /api/v1/contracts/call
   { "to": "0x<contract>", "gas": 21000, "data": "0x<calldata>" }
   ```
   **Result**: HTTP 429 / `ThrottleException("Gas per second rate limit exceeded.")` — the legitimate user is denied service.

4. **Sustain the DoS**: Repeat step 2 in a loop at the rate limit (500 req/s) to keep the bucket continuously depleted. Each new request re-drains any tokens restored by completed executions.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
```
