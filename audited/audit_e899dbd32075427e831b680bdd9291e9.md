### Title
Cache TTL Mismatch Enables Low-Rate DB Amplification via Batch Slot Query

### Summary
The `findStorage()` method in `ContractStateServiceImpl` uses a 2-second `expireAfterWrite` TTL for `contractStateCache` while the per-contract slot key cache (`slotsPerContract`) retains up to 1,500 slot keys for 5 minutes. An unprivileged attacker who pre-populates the slot key cache can send one request every 2 seconds to a single contract, causing each request to miss the `contractStateCache` and trigger a batch DB query fetching all 1,500 cached slots — a 1,500× DB amplification factor at 0.5 RPS, well below the global 500 RPS rate limit.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (lines 58–70): [1](#0-0) 

If `contractStateCache` returns null (or `EMPTY_VALUE`), it unconditionally calls `findStorageBatch()`.

`findStorageBatch()` (lines 85–103): [2](#0-1) 

It reads **all** slot keys currently in the per-contract `slotsPerContract` cache and issues a single `SELECT ... WHERE slot IN (all_cached_slots)` query against the DB.

**Root cause — TTL mismatch:**

- `contractState` cache: `expireAfterWrite=2s` — entries expire 2 seconds after being written. [3](#0-2) 

- `slotsPerContract` cache: `expireAfterAccess=5m,maximumSize=1500` — slot keys survive for 5 minutes as long as they are accessed. [4](#0-3) 

When `findStorageBatch()` runs, it writes all fetched slot values back to `contractStateCache` (line 109). All those entries share the same write timestamp and therefore expire together ~2 seconds later. The next request after that 2-second window finds the entire `contractStateCache` cold for that contract, while the `slotsPerContract` cache still holds all 1,500 slot keys (5-minute TTL, refreshed on each access). This triggers another full 1,500-slot batch query. [5](#0-4) 

**Exploit flow:**

1. **Populate slot key cache**: Attacker sends 1,500 requests to distinct slot keys of contract X. Each call to `findStorageBatch()` executes `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` (line 90), accumulating all 1,500 keys in the per-contract cache. [6](#0-5) 

2. **Sustain the attack**: Every ~2 seconds, send one request to any slot key of contract X. The `contractStateCache` entry for that key has expired; `findStorageBatch()` is invoked; it reads all 1,500 keys from the still-live `slotsPerContract` cache and issues a 1,500-row `IN` query to the DB. [7](#0-6) 

3. **Maintain slot key cache**: Each request also refreshes the `expireAfterAccess` TTL on the accessed slot key, keeping the slot key cache alive indefinitely.

**Why existing checks fail:**

The rate limiter is a single global token bucket at 500 RPS shared across all users: [8](#0-7) [9](#0-8) 

At 0.5 RPS the attacker consumes 0.1% of the global budget. There is no per-IP, per-contract, or per-batch-size throttle. The gas-based throttle only applies to EVM execution gas, not to storage slot query volume. [10](#0-9) 

### Impact Explanation

Each attacker request at 0.5 RPS causes a DB query scanning up to 1,500 rows. With N contracts pre-loaded, N × 0.5 RPS generates N × 1,500 DB row reads per second. For example, 20 contracts → 10 RPS → 30,000 DB row reads/second, sustained indefinitely. This can saturate DB I/O, increase query latency for all users, and degrade overall node throughput — meeting the stated threshold of ≥30% resource increase with a very low, rate-limit-evading request rate.

### Likelihood Explanation

The attack requires no credentials, no special network position, and no brute force. The setup phase (1,500 requests) is a one-time cost. The sustain phase is a single HTTP call every 2 seconds per contract. Any public Ethereum-compatible JSON-RPC client can execute this. The attack is fully repeatable and automatable with a simple script.

### Recommendation

1. **Align TTLs**: Set `contractState` TTL ≥ `slotsPerContract` TTL, or use `expireAfterAccess` instead of `expireAfterWrite` for `contractState` so that actively-used entries are not mass-expired simultaneously.
2. **Cap batch query size**: In `findStorageBatch()`, limit `cachedSlots` to a configurable maximum (e.g., 50–100) before issuing the `IN` query, regardless of how many keys are in the slot cache.
3. **Per-contract or per-IP rate limiting**: Introduce a secondary rate limit keyed on the target contract address or source IP to prevent a single actor from repeatedly triggering batch queries for the same contract.
4. **Decouple batch write timing**: Stagger `contractStateCache` write timestamps (e.g., add a small random jitter) so that all 1,500 entries do not expire simultaneously.

### Proof of Concept

```python
import requests, time, random

RPC = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract_address>"

# Phase 1: populate slotsPerContract cache with 1500 distinct slot keys
for i in range(1500):
    slot = hex(i).zfill(66)  # 32-byte slot key
    requests.post(RPC, json={
        "to": CONTRACT,
        "data": "0x<eth_getStorageAt_calldata_for_slot_" + str(i) + ">",
        "block": "latest"
    })

# Phase 2: sustain — one request per 2 seconds, each triggers 1500-slot batch DB query
while True:
    slot = hex(random.randint(0, 1499)).zfill(66)
    requests.post(RPC, json={
        "to": CONTRACT,
        "data": "0x<eth_getStorageAt_calldata_for_slot_" + slot + ">",
        "block": "latest"
    })
    time.sleep(2.0)  # just after contractStateCache TTL expires
```

Each iteration of the loop causes `findStorageBatch()` to issue `SELECT slot, value FROM contract_state WHERE contract_id = X AND slot IN (<1500 slots>)` against the database, sustained indefinitely at 0.5 RPS per contract.
