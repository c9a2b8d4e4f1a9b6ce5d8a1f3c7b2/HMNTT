### Title
Unauthenticated DB Connection Pool Exhaustion via Combined Filter Query with `zeroRandomPageCostQueryHint` in `getAccounts()`

### Summary
An unprivileged external user can combine all available query filters on `GET /api/v1/accounts` — including `account.publickey` — to trigger the `zeroRandomPageCostQueryHint` code path, which acquires a **dedicated, non-pooled client connection** per request. With the REST API's default pool of only 10 connections and no application-level rate limiting, 10 concurrent crafted requests exhaust the connection pool, rendering the REST API unavailable for up to 20 seconds per wave. The attack is trivially repeatable, causing sustained denial of service.

### Finding Description

**Code path:**

In `rest/accounts.js` `getAccounts()` (lines 335–388), when `account.publickey` is present in the request, the condition at line 366 evaluates to `true`:

```js
const preQueryHint = pubKeyQuery.query !== '' && constants.zeroRandomPageCostQueryHint;
const result = await pool.queryQuietly(query, params, preQueryHint);
``` [1](#0-0) 

`constants.zeroRandomPageCostQueryHint` is `'set local random_page_cost = 0'`. [2](#0-1) 

Inside `queryQuietly` (lines 1518–1527 of `rest/utils.js`), when `preQueryHint` is truthy, the code **acquires a dedicated client** from the pool (not the pool's shared `query()` method) and holds it for the full duration of the transaction:

```js
client = await this.connect();
client.on('error', clientErrorCallback);
await client.query(`begin; ${preQueryHint}`);
result = await client.query(query, params);
await client.query('commit');
``` [3](#0-2) 

The pool is configured with `max: config.db.pool.maxConnections` (default **10**) and `statement_timeout: config.db.pool.statementTimeout` (default **20,000 ms**): [4](#0-3) [5](#0-4) 

The query generated by `getEntityBalanceQuery()` (lines 133–237) for the combined-filter case includes:
- A `with latest_token_balance as (select ... from token_account where associated is true)` CTE
- A `LEFT JOIN entity_stake` with the account condition replicated
- A multi-condition WHERE clause: `e.type in ('ACCOUNT','CONTRACT')`, `e.id >= $N and e.id <= $M` (wide range), `e.balance >= $P`, `public_key >= $Q` [6](#0-5) [7](#0-6) 

With `set local random_page_cost = 0`, the PostgreSQL planner is forced to prefer index scans. A range predicate on `public_key` (e.g., `gte:0000...`) combined with a maximally wide `account.id` range and `balance >= 0` causes the planner to scan the public_key index across all accounts, then join and compute token balances — a very expensive operation on a large dataset.

**Why existing checks are insufficient:**

- `validateReq` only checks parameter names and format validity — it does not limit query cost or concurrency. [8](#0-7) 
- `maxRepeatedQueryParameters = 100` limits repetition of a single parameter but does not prevent combining all filters in one request. [9](#0-8) 
- `statementTimeout = 20,000 ms` kills individual queries after 20 seconds but does **not** prevent pool exhaustion: 10 concurrent requests each hold a dedicated connection for up to 20 seconds, fully saturating the 10-connection pool.
- There is **no application-level rate limiting** for the REST API (rate limiting exists only in the `web3` module). [10](#0-9) 

### Impact Explanation

With the default pool of 10 connections, 10 concurrent crafted requests exhaust the pool. All subsequent REST API requests fail with connection timeout errors (waiting up to `connectionTimeout = 20,000 ms`). The REST API — the primary query interface of the mirror node — becomes completely unavailable. Since the attack is repeatable every ~20 seconds (the statement timeout), sustained DoS is achievable. The importer and gRPC components share the same PostgreSQL instance; under heavy query load from these expensive scans, DB CPU/IO pressure can also degrade those components.

### Likelihood Explanation

The attack requires zero privileges — only a valid public key (obtainable from any public Hedera account lookup). The request is a single valid HTTP GET. Sending 10 concurrent requests is trivial with any HTTP client (`curl`, `ab`, `wrk`). No brute force is needed. The attack is repeatable indefinitely at ~0.5 req/s to maintain sustained DoS.

### Recommendation

1. **Add per-IP or global rate limiting** to the REST API (e.g., via an Express middleware like `express-rate-limit` or at the ingress/load-balancer level) specifically for expensive filter combinations.
2. **Increase `maxConnections`** or use a separate, smaller sub-pool for the `zeroRandomPageCostQueryHint` path to prevent full pool exhaustion.
3. **Require `account.publickey` to use only the equality operator** (`eq`) rather than range operators (`gte`, `lte`, etc.), which dramatically reduces scan cost.
4. **Add a query complexity guard**: if `account.publickey` is combined with a wide `account.id` range and/or `account.balance` filter, reject or require a narrower `account.id` range.
5. **Reduce `statementTimeout`** for the public-key query path to limit connection hold time.

### Proof of Concept

```bash
# Obtain any valid public key from the network, e.g.:
PUBKEY="302a300506032b6570032100<hex>"

# Send 10 concurrent exhausting requests
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/accounts?\
account.id=gte:0.0.1&\
account.id=lte:0.0.9999999999&\
account.balance=gte:0&\
account.publickey=gte:${PUBKEY}" &
done
wait

# All subsequent REST API requests now fail or time out for up to 20 seconds.
# Repeat the loop every 18 seconds to maintain sustained DoS.
curl -v "http://<mirror-node>/api/v1/accounts"
# Expected: connection timeout or 503
```

import httpContext from 'express-http-context';
import qs from 'qs';

import config from '../config';
import {httpStatusCodes, requestIdLabel, requestStartTime} from '../constants';
import {lowerCaseQueryValue, randomString} from '../utils';

const queryCanonicalizationMap = {
  order: lowerCaseQueryValue,
  result: lowerCaseQueryValue,
};

const queryOptions = {
  arrayLimit: config.query.maxRepeatedQueryParameters,
  depth: 1,
  strictDepth: true,
  throwOnLimitExceeded: true,
};

const requestLogger = async (req, res) => {
  const requestId = await randomString(8);
  httpContext.set(requestIdLabel, requestId);

  // set default http OK code for reference
  res.locals.statusCode = httpStatusCodes.OK.code;
  res.locals[requestStartTime] = Date.now();
};

/**
 * Manage request query params to support case insensitive keys and some values (queryCanonicalizationMap above).
 * Express default query parser uses qs, other option is querystring, both are case sensitive
 * Parse using default qs logic and use to populate a new map in which all keys are lowercased
 * @param queryString
 * @returns Query string map object
 */
const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);
    if (lowerKey in caseInsensitiveQueryString) {
      // handle repeated values, merge into an array
      caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;
};

const canonicalizeValue = (key, value) => {
  const canonicalizationFunc = queryCanonicalizationMap[key];
  if (canonicalizationFunc === undefined) {
    return value;
  }

  return Array.isArray(value) ? value.map((v) => canonicalizationFunc(v)) : canonicalizationFunc(value);
};

export {requestLogger, requestQueryParser};
