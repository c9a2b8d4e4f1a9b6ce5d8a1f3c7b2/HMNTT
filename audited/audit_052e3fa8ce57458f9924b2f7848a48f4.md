### Title
Unconditional Double DB Query via `mayMissLongTermScheduledTransaction()` for Any Successful SCHEDULECREATE Transaction ID

### Summary
`getTransactionsByIdOrHash()` in `rest/transactions.js` unconditionally issues a second database query whenever the first query returns a successful SCHEDULECREATE (type=42) with no corresponding scheduled execution row, regardless of how old the transaction is. Because `mayMissLongTermScheduledTransaction()` contains no timestamp guard — unlike the sibling function `getTransactionsByIdOrHashCacheControlHeader()` which does — any unprivileged user who supplies a transaction ID for a historical, expired SCHEDULECREATE permanently triggers 2× DB load per request. No authentication, no special privilege, and no rate-limiting on the REST API path is required.

### Finding Description

**Exact code path:**

`rest/transactions.js`, `getTransactionsByIdOrHash()`, lines 930–938:

```js
const {rows} = await pool.queryQuietly(query, params).then((result) => {
  if (!mayMissLongTermScheduledTransaction(isTransactionHash, scheduled, result.rows)) {
    return result;
  }
  params[params.upperConsensusTimestampIndex] =
    params[params.lowerConsensusTimestampIndex] + maxScheduledTransactionConsensusTimestampRangeNs;
  return pool.queryQuietly(query, params);   // ← second DB query
});
```

`mayMissLongTermScheduledTransaction()`, lines 974–996:

```js
const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  if (isTransactionHash || scheduled === false) { return false; }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId &&
          SUCCESS_PROTO_IDS.includes(transaction.result)) {
        scheduleExists = true;           // ← no timestamp check here
      } else if (transaction.scheduled) {
        return false;
      }
    }
    return scheduleExists;              // ← returns true for ANY age
  }
  ...
};
```

**Root cause — missing timestamp guard:**

`getTransactionsByIdOrHashCacheControlHeader()` (lines 904–909) correctly checks whether the SCHEDULECREATE is recent enough to still be pending:

```js
const elapsed = utils.nowInNs() - transaction.consensus_timestamp;
if (elapsed < maxScheduledTransactionConsensusTimestampRangeNs) {
  header = SHORTER_CACHE_CONTROL_HEADER;
}
```

`mayMissLongTermScheduledTransaction()` performs no equivalent check. It returns `true` — triggering the second query — for every successful SCHEDULECREATE that has no scheduled-execution row, whether the schedule expired one second ago or three years ago.

**Exploit flow:**

1. Attacker enumerates any successful SCHEDULECREATE transaction ID from the public ledger (e.g., via `/api/v1/transactions?type=SCHEDULECREATE`). Thousands exist on mainnet.
2. Attacker selects one whose schedule has already expired (no scheduled execution row will ever appear).
3. Attacker sends `GET /api/v1/transactions/0.0.X-XXXXXXXXXX-XXXXXXXXX` (no `scheduled` query param, no hash form).
4. First `pool.queryQuietly` executes; result contains the SCHEDULECREATE row, no `scheduled=true` row.
5. `mayMissLongTermScheduledTransaction(false, undefined, rows)` → `scheduleExists = true` → returns `true`.
6. Second `pool.queryQuietly` executes with an extended upper timestamp bound.
7. Both queries return the same data; the second is wasted.
8. Attacker repeats at high concurrency; every request costs 2 DB round-trips.

**Why existing checks fail:**

- `isTransactionHash` is `false` (attacker uses transaction ID form) — bypass confirmed.
- `scheduled === false` short-circuit is not hit because attacker omits the `scheduled` query parameter — bypass confirmed.
- The `scheduled === true` branch (`transactions.length === 0`) is not reached because the first query does return the SCHEDULECREATE row.
- The application-level Redis cache is **optional** (`applicationCacheEnabled = config.cache.response.enabled && config.redis.enabled`, `server.js` line 54); if disabled, no caching shields the DB.
- The `SHORTER_CACHE_CONTROL_HEADER` (`cache-control: public, max-age=5`) is only set for *recent* SCHEDULECREATE transactions by `getTransactionsByIdOrHashCacheControlHeader()`; for expired ones it returns `{}`, so no short-TTL HTTP cache hint is emitted either.
- There is no application-level rate limiter on the REST `/transactions/:id` route (the `ThrottleManagerImpl` is web3-only).

### Impact Explanation

Every request to a qualifying transaction ID costs exactly 2× the normal DB query budget. An attacker with a list of expired SCHEDULECREATE IDs (freely available from the public ledger) can sustain a permanent 2× DB amplification with zero authentication. Under high concurrency this doubles connection pool pressure, query queue depth, and I/O on the PostgreSQL backend, potentially degrading service for all users. The condition is **permanent** — expired schedules never gain a scheduled-execution row, so the retry path fires on every request forever.

### Likelihood Explanation

- Requires zero privileges; the endpoint is fully public.
- SCHEDULECREATE transaction IDs are trivially enumerable from the same mirror-node API.
- The attacker needs only a standard HTTP client and a list of IDs.
- The condition is stable and reproducible indefinitely.
- No exploit tooling beyond `curl` or `ab` is needed.

### Recommendation

Add the same elapsed-time guard used in `getTransactionsByIdOrHashCacheControlHeader()` to `mayMissLongTermScheduledTransaction()`. Only return `true` when the SCHEDULECREATE's `consensus_timestamp` is recent enough that the scheduled transaction could still be pending:

```js
const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  if (isTransactionHash || scheduled === false) { return false; }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId &&
          SUCCESS_PROTO_IDS.includes(transaction.result)) {
        // Only retry if the schedule could still be pending
        const elapsed = utils.nowInNs() - transaction.consensus_timestamp;
        if (elapsed < maxScheduledTransactionConsensusTimestampRangeNs) {
          scheduleExists = true;
        }
      } else if (transaction.scheduled) {
        return false;
      }
    }
    return scheduleExists;
  } else if (scheduled && transactions.length === 0) {
    return true;
  }
  return false;
};
```

Additionally, consider adding per-IP rate limiting to the `/transactions/:id` route independent of the Redis cache.

### Proof of Concept

```bash
# Step 1: find a successful, expired SCHEDULECREATE transaction ID
TXID=$(curl -s "https://<mirror>/api/v1/transactions?type=SCHEDULECREATE&result=SUCCESS&limit=1" \
  | jq -r '.transactions[0].transaction_id')

# Step 2: confirm no scheduled execution exists (schedule expired)
curl -s "https://<mirror>/api/v1/transactions/$TXID?scheduled=true"

# Step 3: flood with unauthenticated requests omitting scheduled=false
# Each request triggers 2 DB queries instead of 1
