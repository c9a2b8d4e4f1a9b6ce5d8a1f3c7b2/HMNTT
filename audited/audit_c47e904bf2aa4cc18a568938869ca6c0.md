### Title
Unauthenticated Alias-Resolution DB Query Per Request Enables Connection Pool Exhaustion DoS

### Summary
The `GET /:idOrAliasOrEvmAddress/allowances/crypto` endpoint is publicly accessible with no rate limiting or caching. When the path parameter is a valid account alias, `EntityService.getEncodedId()` unconditionally executes a live database query on every request. A single unprivileged attacker can flood this endpoint with a valid alias, saturating the database connection pool and partitioning the API layer from the database for all users.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 17 registers the route with no authentication middleware:
```js
router.getExt(getPath('allowances/crypto'), CryptoAllowanceController.getAccountCryptoAllowances);
```

`rest/controllers/cryptoAllowanceController.js` line 77 calls `getEncodedId` unconditionally on every request:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```

`rest/service/entityService.js` lines 125–126 branch into alias resolution when the path parameter is a valid alias:
```js
} else if (AccountAlias.isValid(entityIdString)) {
  return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
```

`rest/service/entityService.js` lines 42–43 execute a live DB query with no caching:
```js
const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
```

The query is:
```sql
SELECT id FROM entity
WHERE coalesce(deleted, false) <> true
  AND alias = $1
```

**Root cause:** There is no in-process or distributed cache for alias→ID resolution, and no rate limiting is applied to this route. `extendExpress.js` adds only async error-wrapping; no throttle or IP-based limiter is present. Every alias-based request unconditionally consumes one database connection for the duration of the query.

**Why existing checks fail:** The `responseCacheHandler` middleware (if applied) caches full HTTP responses keyed on the URL. An attacker trivially bypasses it by varying query parameters (e.g., `?limit=1`, `?limit=2`, `?spender.id=0.0.1`, etc.), each producing a distinct cache key while still triggering the alias-resolution query before any cached response can be served.

### Impact Explanation
The database connection pool is a finite shared resource. Exhausting it blocks all other API endpoints that require DB access, not just this one. The result is a full API-to-database partition: all reads and writes across the mirror node REST API fail until the flood stops and connections drain. Severity is **High** (availability impact across the entire service, no authentication required).

### Likelihood Explanation
Preconditions are minimal: the attacker needs only one valid account alias (publicly discoverable via `GET /api/v1/accounts`). The attack is stateless, trivially parallelizable with standard HTTP load tools (`wrk`, `ab`, `hey`), and repeatable indefinitely. No credentials, tokens, or special network position are required.

### Recommendation
1. **Cache alias resolution**: Introduce a short-lived (e.g., 30–60 s) in-process or Redis cache keyed on the alias string inside `EntityService.getAccountIdFromAlias`. A cache hit returns the encoded ID with zero DB cost.
2. **Rate limit by IP**: Apply a per-IP request rate limit (e.g., via `express-rate-limit`) specifically on the `/accounts/:id/allowances/*` family of routes, or globally on the router.
3. **Connection pool timeout + queue depth**: Configure the DB pool with a maximum queue depth and a short acquire timeout so that pool exhaustion returns a `503` quickly rather than blocking indefinitely.

### Proof of Concept
```bash
# 1. Discover a valid alias (unauthenticated)
ALIAS=$(curl -s 'https://<mirror-node>/api/v1/accounts?limit=1' \
  | jq -r '.accounts[0].alias')

# 2. Flood the alias-resolution path with varied query params to bypass response cache
seq 1 500 | xargs -P 100 -I{} \
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/allowances/crypto?limit={}" \
  -o /dev/null

# 3. Observe: subsequent requests to any /api/v1/* endpoint time out or return 503
curl -v 'https://<mirror-node>/api/v1/transactions?limit=1'
# Expected: connection timeout or "503 Service Unavailable"
```

Each of the 500 concurrent requests triggers `SELECT id FROM entity WHERE alias = $1`, holding a pool connection for the query duration. With a typical pool size of 10–20 connections, the pool saturates within the first wave of requests.
