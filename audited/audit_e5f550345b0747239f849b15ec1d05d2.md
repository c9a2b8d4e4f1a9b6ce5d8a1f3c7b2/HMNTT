### Title
Unbounded `content-length` Header Poisons `api_request_size_bytes` Histogram in `metricsMiddleware`

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware` function reads the raw `content-length` request header and passes it directly to `requestSizeHistogram.record()` with no upper-bound validation. Any unauthenticated external user can send a GET request with `Content-Length: 9007199254740991` (Number.MAX_SAFE_INTEGER), causing the OpenTelemetry histogram's cumulative `_sum` to be permanently skewed for the lifetime of the process, rendering the `api_request_size_bytes` metric useless for capacity planning.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, function `metricsMiddleware`, line 209:

```javascript
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** The only guard is `|| 0`, which handles `NaN`/falsy results from `parseInt` (e.g., non-numeric strings). It does **not** impose any upper bound. `parseInt('9007199254740991', 10)` returns the exact JavaScript integer `9007199254740991` — a valid, non-NaN, non-zero number — so the `|| 0` fallback never triggers.

**Exploit flow:**
1. Attacker sends a GET request (no body required) to any API endpoint with the header `Content-Length: 9007199254740991`.
2. Node.js HTTP server populates `req.headers['content-length']` with the attacker-supplied string verbatim.
3. Express routes the request normally; the route handler sends a response.
4. `res.emit('finish')` fires; the `metricsMiddleware` finish handler executes.
5. `parseInt('9007199254740991', 10)` → `9007199254740991`; the `|| 0` guard is bypassed.
6. `requestSizeHistogram.record(9007199254740991, labels)` is called.
7. The OpenTelemetry SDK adds `9007199254740991` (~9 petabytes) to the histogram's cumulative `_sum` counter. This value dominates all legitimate observations for the process lifetime.

**Why existing checks fail:** The `|| 0` guard only catches `NaN` and `0`. There is no `Math.min`, no `Number.isFinite` range check, no cap against a maximum expected request size, and no server-level enforcement that the declared `Content-Length` matches an actual body for GET requests.

### Impact Explanation
The OpenTelemetry histogram's `_sum` is a monotonically increasing cumulative counter. A single injection of `9007199254740991` (~9×10¹⁵ bytes) permanently dominates the sum until the process restarts. The derived average request size (`_sum / _count`) becomes astronomically wrong, and any Prometheus/Grafana dashboards or alerting rules based on `api_request_size_bytes` (e.g., for capacity planning, anomaly detection, or SLO tracking) produce meaningless data. The `_count` and bucket distribution remain accurate, but the `_sum`-dependent metrics are corrupted. Severity matches the stated scope: griefing with no economic damage to network users, but operational observability is degraded.

### Likelihood Explanation
No authentication, no rate limiting, and no special privileges are required. A single HTTP GET request with a crafted header is sufficient. The attack is trivially repeatable with `curl`:
```
curl -H "Content-Length: 9007199254740991" https://<mirror-node>/api/v1/accounts
```
Any internet-accessible mirror node REST API is exposed. The attacker does not need to send a body, does not need to maintain a connection, and the server returns a normal response (the header mismatch is irrelevant for GET requests in Node.js).

### Recommendation
Cap the recorded value to a sane maximum before passing it to the histogram. Replace line 209 with:

```javascript
const MAX_RECORDED_REQUEST_BYTES = 100 * 1024 * 1024; // 100 MB upper bound
const contentLength = Math.min(
  Math.max(0, parseInt(req.headers['content-length'] ?? '0', 10) || 0),
  MAX_RECORDED_REQUEST_BYTES
);
requestSizeHistogram.record(contentLength, labels);
```

Alternatively, use the actual bytes received (e.g., by accumulating `data` events on `req`) rather than trusting the client-supplied header.

### Proof of Concept
```bash
