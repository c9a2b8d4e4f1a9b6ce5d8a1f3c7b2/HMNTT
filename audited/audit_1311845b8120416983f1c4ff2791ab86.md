### Title
Unauthenticated Health Endpoint Triggers Unbounded Outbound Prometheus Queries, Enabling Monitoring Infrastructure DoS

### Summary
The `ImporterLagHealthIndicator.health()` method unconditionally issues a synchronous HTTP request to Prometheus on every invocation with no caching, throttling, or rate limiting. The `/actuator/health/cluster` endpoint is exposed publicly via the Kubernetes ingress with no authentication, allowing any unprivileged external attacker to flood the health endpoint and generate a proportional stream of PromQL queries against the Prometheus server, degrading or disabling monitoring infrastructure.

### Finding Description

**Exact code path:**

`monitor/src/main/java/org/hiero/mirror/monitor/health/ImporterLagHealthIndicator.java`, lines 48–62:

```java
@Override
public Health health() {
    if (!properties.isEnabled()) {
        return up();
    }
    final var query = buildLagQuery();
    try {
        final var resp = prometheusClient.query(query);   // outbound HTTP on every call
        return evaluate(resp);
    } catch (final Exception e) {
        log.warn("Importer lag health check failed; returning UP: {}", e.getMessage());
        return up();
    }
}
```

`monitor/src/main/java/org/hiero/mirror/monitor/health/PrometheusApiClient.java`, lines 46–53:

```java
PrometheusQueryResponse query(final String query) {
    final var encodedQuery = UriUtils.encodeQuery(query, StandardCharsets.UTF_8);
    return prometheusClient
            .get()
            .uri(builder -> builder.queryParam("query", encodedQuery).build())
            .retrieve()
            .body(PrometheusQueryResponse.class);   // blocking HTTP call, no cache
}
```

**Root cause:** There is no response cache, no per-IP rate limit, and no authentication on the health endpoint. `ImporterLagHealthIndicator` is a plain `HealthIndicator` bean; Spring Boot invokes `health()` synchronously on every request to the health endpoint. `PrometheusApiClient` constructs a new `RestClient` request and blocks on the Prometheus HTTP API for each call (up to `timeout`, default 5 s).

**Why existing checks fail:**

- `properties.isEnabled()` (line 49) only guards against misconfiguration (missing URL/cluster), not against request frequency.
- The ingress in `charts/hedera-mirror-monitor/values.yaml` (lines 91–99) exposes `/actuator/health/cluster` with `enabled: true` and **no authentication annotations**.
- No `SecurityFilterChain` or Spring Security configuration exists anywhere in the monitor module (confirmed by directory scan of `monitor/src/main/java/org/hiero/mirror/monitor/config/`).
- The monitor uses Spring WebFlux (`spring-boot-starter-webflux`), so the custom `ActuatorHttpServer` (which is `@ConditionalOnMissingClass({"reactor.netty.http.server.HttpServer",...})`) does **not** activate; the standard WebFlux actuator endpoint is used instead, with no additional access controls.

### Impact Explanation

An attacker can drive Prometheus query load at an arbitrary rate by flooding `/actuator/health/cluster`. Each inbound request maps 1:1 to a PromQL query execution on Prometheus. At sufficient request rates this exhausts Prometheus query concurrency, increases scrape/query latency for all consumers, and can render the monitoring layer blind — preventing operators from detecting real incidents (importer lag, node failures, etc.) on the Hiero network. This directly undermines the safety monitoring layer for the mirror node infrastructure.

### Likelihood Explanation

The attack requires zero privileges and zero authentication. The ingress endpoint is reachable from the public internet by default configuration. The exploit is trivially repeatable with any HTTP load tool (`curl` in a loop, `ab`, `wrk`, etc.). No special knowledge of the system is required beyond knowing the hostname. The `timeout` of 5 s per Prometheus call means even a modest 20 req/s from a single attacker saturates 100 concurrent Prometheus connections.

### Recommendation

1. **Add response caching** to `ImporterLagHealthIndicator` using Spring Boot's built-in health caching (`management.endpoint.health.cache.time-to-live`, e.g. 10–30 s), so repeated calls within the TTL return the cached result without hitting Prometheus.
2. **Restrict actuator access** via Spring Security: add a `SecurityWebFilterChain` bean in the monitor module that requires authentication (or IP allowlist) for all `/actuator/**` paths, or move the actuator to a separate management port not exposed by the ingress.
3. **Add ingress-level rate limiting** (e.g., Traefik `RateLimit` middleware) on the `/actuator/health/cluster` path as a defense-in-depth measure.

### Proof of Concept

```bash
# Precondition: monitor deployed with importerLag enabled and ingress accessible
MONITOR_HOST="https://<monitor-ingress-host>"

# Flood the health endpoint with no credentials required
while true; do
  curl -s -o /dev/null "$MONITOR_HOST/actuator/health/cluster" &
done
# Each request triggers ImporterLagHealthIndicator.health() -> PrometheusApiClient.query()
# Prometheus receives unbounded PromQL queries; query latency rises; alerting degrades
```

Observe in Prometheus metrics (`prometheus_engine_query_duration_seconds`) that query latency and active query count increase proportionally to the flood rate, with no server-side throttle applied.
