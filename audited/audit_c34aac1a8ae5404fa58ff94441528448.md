### Title
Unauthenticated DoS via Unbounded LIKE Prefix Scan Flood on GET /blocks/:hashOrNumber

### Summary
The `getByHashOrNumber` handler in `rest/routes/blockRoute.js` is registered with no rate-limiting middleware. Any unauthenticated caller who supplies a valid-format (but non-matching) hash causes `RecordFileService.getByHashOrNumber` to execute a `LIKE` prefix scan against the `record_file` table on every request. A sustained flood of such requests exhausts the database connection pool and I/O capacity, producing a denial-of-service condition without requiring any credentials.

### Finding Description

**Route registration** — no rate-limiting middleware is applied: [1](#0-0) 

**Controller** — format validation passes for any well-formed hex hash (64 or 96 chars, optional `0x` prefix), then immediately issues a DB call: [2](#0-1) 

**Service** — when a hash is supplied, the query uses a `LIKE` prefix pattern (`hash || '%'`) against the full `record_file` table with no short-circuit: [3](#0-2) 

The underlying `blocksQuery` selects many columns including a `coalesce(size, length(bytes))` expression: [4](#0-3) 

**Middleware inventory** — the server registers `authHandler`, `requestLogger`, optional `metricsHandler`, and optional `responseCacheCheckHandler`. None of these implement per-IP or global request-rate limiting: [5](#0-4) 

The `authHandler` only sets a custom *response-result limit* for authenticated users; it does not throttle request frequency: [6](#0-5) 

A grep for `rateLimit`, `throttle`, and `express-rate` across `rest/**` returns hits only in `openapi.yml` (documentation) and `integrationUtils.js` (test setup) — confirming no runtime rate-limiting exists.

**Root cause**: The design assumes infrastructure-level rate limiting (e.g., an API gateway or reverse proxy) that is not enforced in the application code itself. An attacker who bypasses or is not subject to such infrastructure controls can drive unlimited DB queries.

### Impact Explanation
Each request with a valid-format, non-matching hash causes PostgreSQL to execute a prefix `LIKE` scan on `record_file.hash`. Even with a B-tree index, a high-volume flood saturates the database connection pool (bounded by `pg` pool size), increases I/O wait, and degrades or blocks all other API consumers. The `coalesce(size, length(bytes))` projection adds per-row CPU cost. The result is a full or partial service outage — a network-partition-like effect — for all mirror node REST API users.

**Severity: High** — complete availability impact, no authentication required.

### Likelihood Explanation
- **Precondition**: None. No account, token, or credential is needed.
- **Attacker capability**: Any script that generates random 64-char hex strings and sends HTTP GET requests. Trivially automated with `curl`, `ab`, `wrk`, or a simple Python loop.
- **Repeatability**: Continuous; the attacker simply keeps sending requests.
- **Detection evasion**: Requests look syntactically valid (pass OpenAPI schema validation), making WAF signature-based blocking difficult.

**Likelihood: High.**

### Recommendation
1. **Application-layer rate limiting**: Add `express-rate-limit` (or equivalent) as middleware on the blocks router, keyed by IP, before the route handler. A conservative default of ~10–30 req/s per IP is appropriate for a lookup endpoint.
2. **Database-side guard**: Enforce a short `statement_timeout` on the pool used by `RecordFileService` so runaway queries are killed automatically.
3. **Index verification**: Confirm `record_file.hash` has a B-tree index created with `text_pattern_ops` (PostgreSQL) so prefix `LIKE` queries use the index rather than a sequential scan.
4. **Connection pool cap**: Ensure the `pg` pool `max` setting is sized to prevent a single flood from consuming all available DB connections.

### Proof of Concept

```bash
# Generate and flood with valid-format, non-matching 64-char hex hashes
python3 -c "
import os, requests, threading

def flood():
    while True:
        fake_hash = os.urandom(32).hex()  # 64 hex chars, valid format
        requests.get(f'http://<mirror-node-host>/api/v1/blocks/{fake_hash}')

threads = [threading.Thread(target=flood) for _ in range(100)]
for t in threads: t.start()
for t in threads: t.join()
"
```

Each request passes `validateHashOrNumber` (valid 64-char hex), reaches `RecordFileService.getByHashOrNumber`, and executes:

```sql
SELECT count, hash, name, prev_hash, ... , coalesce(size, length(bytes)) as size
FROM record_file
WHERE hash LIKE '<random_64_hex>%'
```

With 100 concurrent threads, the database connection pool is saturated within seconds, causing all subsequent API requests (blocks, transactions, accounts) to queue or fail with connection timeout errors.

```javascript
  async getByHashOrNumber(hash, number) {
    let whereStatement = '';
    const params = [];
    if (hash) {
      hash = hash.toLowerCase();
      whereStatement += `${RecordFile.HASH} like $1`;
      params.push(hash + '%');
    } else {
      whereStatement += `${RecordFile.INDEX} = $1`;
      params.push(number);
    }

    const query = `${RecordFileService.blocksQuery} where ${whereStatement}`;
    const row = await super.getSingleRow(query, params);
    return row ? new RecordFile(row) : null;
  }
```
