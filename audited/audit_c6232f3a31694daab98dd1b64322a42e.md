### Title
Unauthenticated Two-Query DB Chain in `getContractResultsById` Enables Connection Pool Starvation

### Summary
Any unauthenticated user can supply a valid `block_hash` query parameter to `GET /contracts/:contractId/results`, causing the REST API to execute two sequential database queries per request with no application-level rate limiting. Under sufficient concurrent load, this exhausts the shared connection pool, denying service to all REST API consumers.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 14 registers the endpoint with no authentication: [1](#0-0) 

`getContractResultsById` in `rest/controllers/contractController.js` (lines 856–892) calls `extractContractResultsByIdQuery`, which — when a `block_hash` filter is present — performs **DB Query 1** synchronously before returning: [2](#0-1) 

`RecordFileService.getRecordFileBlockDetailsFromHash` executes a `LIKE`-prefix scan on `record_file`: [3](#0-2) 

backed by the query: [4](#0-3) 

After that `await` resolves, control returns to `getContractResultsById`, which then issues **DB Query 2** — a full `contract_result` scan bounded only by the block's timestamp range and the default page limit: [5](#0-4) 

The two queries are **sequential** (`await` on each), meaning each in-flight request holds a pool connection for the combined duration of both round-trips.

**Root cause / failed assumption:** The design assumes that the `statement_timeout` configured in the pool is sufficient to bound per-request resource consumption: [6](#0-5) 

This assumption fails under concurrent load: even with fast individual queries, N concurrent requests each holding a connection for two sequential round-trips can saturate a pool of size `maxConnections`.

**Why existing throttle is insufficient:** The only throttle/rate-limit mechanism in the codebase is in the `web3` Java module (`ThrottleManagerImpl`, `ThrottleConfiguration`), which guards `/contracts/call` and gas-estimation endpoints: [7](#0-6) 

There is **no equivalent rate-limiting middleware** applied to the Node.js REST API routes, including `/:contractId/results`.

### Impact Explanation
When the connection pool (`maxConnections`) is exhausted, every subsequent REST API request — across all endpoints — queues or fails with a connection-timeout error. This is a non-network-based DoS: a single attacker with a modest number of concurrent HTTP connections can render the entire mirror-node REST API unavailable for legitimate users. The impact is proportional to pool size and query latency; a `block_hash` that maps to a block with many contract results maximises per-request hold time.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. A valid `block_hash` value is publicly observable from any block explorer or from the `/api/v1/blocks` endpoint on the same mirror node. The attack is trivially scriptable with standard HTTP concurrency tools (`ab`, `wrk`, `hey`). It is repeatable and stateless — the attacker does not need to maintain any session.

### Recommendation
1. **Add application-level rate limiting** to the Node.js REST API (e.g., `express-rate-limit` or an equivalent middleware) applied globally or specifically to the `/:contractId/results` route, mirroring the pattern already used in the `web3` module.
2. **Parallelize the two queries** where possible (the block-hash lookup and the contract-results query can be restructured into a single JOIN or CTE), eliminating the sequential connection-hold pattern.
3. **Enforce a short, explicit `statement_timeout`** per query rather than relying on a pool-level default, so that slow queries release connections faster.
4. Consider deploying an infrastructure-level rate limiter (e.g., nginx `limit_req`, API gateway) in front of the REST service as a defence-in-depth measure.

### Proof of Concept
```bash
# 1. Obtain a valid block hash (no auth required)
BLOCK_HASH=$(curl -s https://<mirror-node>/api/v1/blocks?limit=1 \
  | jq -r '.blocks[0].hash')

# 2. Obtain any contract ID
CONTRACT_ID=$(curl -s https://<mirror-node>/api/v1/contracts?limit=1 \
  | jq -r '.contracts[0].contract_id')

# 3. Flood with concurrent requests — each triggers 2 sequential DB queries
hey -n 10000 -c 200 \
  "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}/results?block_hash=${BLOCK_HASH}"

# Expected result: after pool exhaustion, all /api/v1/* endpoints begin
# returning 503 / connection-timeout errors for legitimate users.
```
