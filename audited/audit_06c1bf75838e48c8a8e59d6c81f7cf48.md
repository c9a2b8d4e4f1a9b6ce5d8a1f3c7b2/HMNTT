### Title
Unauthenticated DB Resource Exhaustion via Uncached `FindByIndex` Correlated Subquery with Disabled Rate Limiting

### Summary
`FindByIndex` in `rosetta/app/persistence/block.go` executes a correlated subquery against the `record_file` table on every invocation with no application-level result cache. The only infrastructure-level rate limiting (Traefik middleware) is disabled by default via `global.middleware: false`. Any unauthenticated external caller can flood the endpoint with valid block index values, causing unbounded DB query load and measurable resource exhaustion on the database node.

### Finding Description
**Code path:**

`FindByIndex` (line 171) validates only that `index >= 0`, then calls `initGenesisRecordFile` (cached via `sync.Once`) and immediately delegates to `findBlockByIndex` (line 180).

`findBlockByIndex` (line 210) checks `index < br.genesisBlock.Index` — trivially bypassed with any valid block index — then executes `selectRecordBlockByIndex` directly against the DB with no result cache:

```go
// block.go line 219
if err := db.Raw(selectRecordBlockByIndex, sql.Named("index", index)).First(rb).Error; err != nil {
```

`selectRecordBlockByIndex` (lines 75–85) contains a correlated subquery that fires on every call:

```sql
select consensus_start,
       coalesce((
         select consensus_start-1
         from record_file
         where index = @index + 1::bigint   -- correlated subquery, every request
       ), consensus_end) as consensus_end,
       ...
from record_file
where index = @index
```

The `record_file.index` column has a non-unique B-tree index (`record_file__index`, created in `V1.58.5__remove_record_files_index_unique_constraint.sql`), so each individual query is fast. However, with no caching and no rate limiting, N concurrent requests produce N full query executions (outer lookup + correlated subquery) against the DB.

**Why existing checks fail:**

The Traefik middleware chain (`rateLimit: average: 10`, `inFlightReq: amount: 5`) is defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 but is gated by the condition in `templates/middleware.yaml` line 3:

```yaml
{{ if and .Values.global.middleware .Values.middleware -}}
```

`global.middleware` defaults to `false` (line 95 of `values.yaml`), so the entire middleware chain — including rate limiting — is **not deployed** in a default installation. There is zero application-level caching or throttling in the Go code itself.

### Impact Explanation
With no rate limiting active by default, an attacker can sustain thousands of requests per second to `/rosetta/block`. Each request causes two indexed DB lookups. On a production mirror node with a large `record_file` table (millions of rows after years of operation), the aggregate I/O and CPU load on the PostgreSQL instance increases proportionally with request volume. This can degrade or deny service to legitimate users and to other mirror node components sharing the same DB. The `record_file` table grows continuously, making the attack surface larger over time.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and knowledge of only one valid block index (the genesis index is publicly known and discoverable via `/rosetta/network/status`). The attacker does not need to enumerate or brute-force values — a single valid index repeated at high concurrency is sufficient. The attack is trivially scriptable, repeatable, and requires no special tooling beyond standard HTTP clients.

### Recommendation
1. **Application-level result cache**: Add a short-lived (e.g., 1–2 second) in-process LRU cache keyed on `index` inside `findBlockByIndex`. Blocks are immutable once written, so cache correctness is not a concern for existing blocks.
2. **Enable middleware by default**: Change `global.middleware` default to `true` in `values.yaml` so the Traefik `rateLimit` and `inFlightReq` middleware are deployed in all standard installations.
3. **Application-level concurrency limit**: Add a semaphore or token-bucket rate limiter in the Go service layer (e.g., in `RetrieveBlock` in `base_service.go`) independent of infrastructure configuration.

### Proof of Concept
**Preconditions:** Default Helm deployment (i.e., `global.middleware: false`); Rosetta API reachable at `$HOST`.

**Step 1 — Discover genesis index** (public, no auth):
```bash
curl -s -X POST $HOST/rosetta/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq .genesis_block_identifier.index
# e.g., returns 0
```

**Step 2 — Flood FindByIndex with genesis+1** (valid index, always hits DB):
```bash
seq 1 500 | xargs -P 100 -I{} curl -s -X POST $HOST/rosetta/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index":1}}'
```

**Result:** Each of the 500 requests (100 concurrent) executes `selectRecordBlockByIndex` with its correlated subquery against the DB. Monitor PostgreSQL `pg_stat_activity` and CPU metrics during the flood — active query count and CPU utilization on the DB host increase proportionally, with no server-side throttle applied. func (br *blockRepository) findBlockByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
```go
	if index < br.genesisBlock.Index {
		return nil, hErrors.ErrBlockNotFound
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectRecordBlockByIndex, sql.Named("index", index)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	return rb.ToBlock(br.genesisBlock), nil
}
```
