### Title
Unauthenticated GraphQL `account()` Query Flooding via Non-Zero Shard/Realm EntityId Causes Sustained DB Miss Load and Connection Pool Exhaustion

### Summary
The GraphQL `account(input: AccountInput!)` endpoint accepts `entityId` inputs with attacker-controlled `shard` and `realm` values. These are encoded into large non-standard 64-bit IDs that are guaranteed to be absent from the database. Because the GraphQL `EntityRepository` has no result caching and the application has no rate limiting, an unauthenticated attacker can flood the endpoint with unique non-existent IDs, driving sustained DB connection pool exhaustion that degrades availability for legitimate users.

### Finding Description

**Code path:**

`AccountController.account()` (line 41–44) receives an `EntityIdInput` and calls `toEntityId(entityId)`: [1](#0-0) 

`GraphQlUtils.toEntityId()` (line 64–66) passes the raw user-supplied shard/realm/num directly to `EntityId.of()`: [2](#0-1) 

`EntityId.encode()` (line 75–85) accepts any shard in 0–1023 and realm in 0–65535. When shard or realm is non-zero, it bit-shifts them into the upper bits of a 64-bit long, producing IDs in a range that is entirely disjoint from the real entity ID space (which stores shard=0, realm=0 entities as plain `num` values): [3](#0-2) 

`EntityServiceImpl.getByIdAndType()` (line 24–26) calls `entityRepository.findById(entityId.getId())` — a direct, uncached DB lookup: [4](#0-3) 

The GraphQL `EntityRepository` has **no `@Cacheable` annotations** — every `findById()` call issues a live SQL query: [5](#0-4) 

**Root cause:** The `EntityIdInput` schema enforces only `@Min(value: 0)` on shard and realm — no upper-bound rejection at the GraphQL layer, no shard/realm allowlist, and no application-level rate limiting. The `EntityId.CACHE` (line 42–45) caches only the Java `EntityId` object keyed by encoded long, not the DB result: [6](#0-5) [7](#0-6) 

**Why non-zero shard/realm is worse than shard=0/realm=0 flooding:** With shard=0, realm=0, encoded IDs equal `num` and overlap with the real entity ID space — some may reside in PostgreSQL's shared buffer cache. With shard≥1 or realm≥1, encoded IDs have high bits set and are guaranteed to be absent from the DB and from any DB-level buffer cache. The attacker has a space of 1023 × 65535 × 274,877,906,943 ≈ 1.8×10¹⁶ unique guaranteed-miss IDs to cycle through, making cache-warming defenses irrelevant.

### Impact Explanation

Each attacker request consumes one DB connection from the pool for the duration of the `SELECT * FROM entity WHERE id = ?` query. With no application-level rate limiting and no result caching, a sustained flood of unique non-existent IDs exhausts the connection pool, causing legitimate queries to queue and time out. This results in full service unavailability for the GraphQL API and potentially for other components sharing the same DB connection pool.

### Likelihood Explanation

No credentials, API keys, or privileged access are required — the endpoint is fully public. The attack requires only an HTTP client capable of sending POST requests to `/graphql/alpha`. The only deployment-level mitigation is the Traefik `inFlightReq: amount: 5` per source IP: [8](#0-7) 

This is bypassable with a modest botnet or by rotating source IPs. The `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` configured in `GraphQlConfiguration` limit per-query structure, not request rate: [9](#0-8) 

The attack is repeatable, requires no state, and can be automated trivially.

### Recommendation

1. **Reject non-system shard/realm values at the application layer**: Add a validator in `AccountController` or `GraphQlUtils.toEntityId()` that rejects `entityId` inputs where `shard` or `realm` differ from the configured system shard/realm (`CommonProperties.getShard()`/`getRealm()`), returning a GraphQL validation error before any DB call is made.
2. **Add result caching to the GraphQL `EntityRepository`**: Apply `@Cacheable` to `findById()` (including negative/miss caching) as is done in the web3 `EntityRepository`, so repeated lookups for the same non-existent ID do not hit the DB.
3. **Enforce application-level rate limiting**: Add a token-bucket or fixed-window rate limiter in the Spring filter chain for `/graphql/alpha`, independent of the optional Traefik middleware.

### Proof of Concept

```bash
# Flood with guaranteed-miss IDs (shard=1, realm=0, num cycling)
for i in $(seq 1 100000); do
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {entityId: {shard: 1, realm: 0, num: $i}}) { id }}\"}" &
done
```

Each request encodes to a unique large ID (e.g., shard=1, realm=0, num=1 → `(1 << 54) | 1 = 18014398509481985`) that is absent from the `entity` table. With no caching and no rate limiting, each request issues a live `SELECT` against the DB. Sustained at sufficient concurrency from multiple source IPs (to bypass the per-IP Traefik limit), this exhausts the DB connection pool and causes HTTP 503/timeout responses for all legitimate users.

```java
    private static long encode(long shard, long realm, long num) {
        if (shard > SHARD_MASK || shard < 0 || realm > REALM_MASK || realm < 0 || num > NUM_MASK || num < 0) {
            throw new InvalidEntityException("Invalid entity ID: " + shard + "." + realm + "." + num);
        }

        if (shard == 0 && realm == 0) {
            return num;
        }

        return (num & NUM_MASK) | (realm & REALM_MASK) << NUM_BITS | (shard & SHARD_MASK) << (REALM_BITS + NUM_BITS);
    }
```
