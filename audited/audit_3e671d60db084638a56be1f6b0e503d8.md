### Title
Unthrottled Binary Search Amplification in `estimateGas` Allows 20x EVM Execution Multiplication Per Request

### Summary
An unprivileged user can set `estimate=true` with `gas=15_000_000` in a `ContractCallRequest` to trigger `ContractExecutionService.estimateGas()`, which runs one initial EVM call followed by up to 20 additional EVM executions inside `BinaryGasEstimator.search()`. The `ThrottleManagerImpl` charges the gas bucket only once per HTTP request (for the declared gas limit), while the binary search iterations bypass all throttle accounting entirely, creating a 20x CPU amplification factor with no additional throttle cost.

### Finding Description

**Code path:**

`ContractExecutionService.estimateGas()` (lines 81–98) first calls `callContract(params, context)`, which internally calls `doProcessCall(params, params.getGas(), false)` — with `estimate=false`. Then it calls `binaryGasEstimator.search(...)`, which loops up to `maxGasEstimateRetriesCount = 20` times calling `doProcessCall(params, gas, true)` — with `estimate=true`. [1](#0-0) 

**Root cause — throttle charges once, binary search runs 20 more times for free:**

`ThrottleManagerImpl.throttle()` consumes `scaleGas(request.getGas())` tokens from the gas bucket exactly once per HTTP request. [2](#0-1) 

Inside `doProcessCall`, the `restoreGasToBucket` call and metric recording are guarded by `if (!estimate)`. When the binary search calls `doProcessCall(..., true)`, neither gas restoration nor any throttle charge occurs for those iterations. [3](#0-2) 

**Why existing checks fail:**

The gas bucket is charged `scaleGas(15_000_000)` = 1500 tokens once. The initial `callContract` call (estimate=false) restores unused gas back to the bucket, so the net cost for the initial call is only the gas actually consumed. The subsequent 20 binary search iterations — each a full EVM execution — consume zero gas tokens from the bucket and are completely invisible to the throttle. [4](#0-3) 

The `maxGasEstimateRetriesCount = 20` cap limits amplification but does not prevent it. [5](#0-4) 

### Impact Explanation
A single HTTP request with `estimate=true` and `gas=15_000_000` causes up to 21 EVM executions (1 initial + 20 binary search iterations) while the throttle accounts for only 1. Against a contract that consumes near the gas limit, each iteration is maximally expensive. With the default `requestsPerSecond = 500`, an attacker can sustain 500 such requests per second, each multiplying CPU work by up to 21x, easily exceeding a 30% increase in node resource consumption. No authentication or special privilege is required. [6](#0-5) 

### Likelihood Explanation
The attack requires only a standard HTTP POST to the public `/api/v1/contracts/call` endpoint with `estimate=true` and a valid `to` address pointing to any deployed contract. No credentials, tokens, or privileged access are needed. The `estimate` field is a plain boolean in `ContractCallRequest` with no access control. [7](#0-6) 

The attack is trivially repeatable with a simple HTTP client loop and is fully automatable.

### Recommendation
1. **Charge the gas bucket proportionally for estimate requests.** Before entering `binaryGasEstimator.search()`, consume additional gas tokens from the bucket proportional to `maxGasEstimateRetriesCount * gasUsedByInitialCall`, or consume tokens inside each binary search iteration.
2. **Apply a separate, lower rate limit for estimate requests.** Add a dedicated bucket for `estimate=true` requests with a significantly lower `requestsPerSecond` than regular calls.
3. **Reduce `maxGasEstimateRetriesCount`** from 20 to a smaller value (e.g., 10) to bound the worst-case amplification.
4. **Apply the `RequestProperties` filter** to detect and throttle or reject high-gas estimate requests via the existing `request` filter list in `ThrottleProperties`. [8](#0-7) 

### Proof of Concept

**Preconditions:** A deployed contract on the network that performs a moderately expensive computation (e.g., a loop consuming ~1M gas).

**Steps:**

```bash
# Single amplified request — triggers 1 initial EVM call + up to 20 binary search iterations
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to": "0x<expensive_contract_address>",
    "data": "0x<expensive_function_selector>",
    "gas": 15000000,
    "estimate": true
  }'

# Sustained attack — 500 req/s (within rate limit), each causing 21x EVM work
while true; do
  for i in $(seq 1 500); do
    curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
      -H "Content-Type: application/json" \
      -d '{"to":"0x<addr>","data":"0x<selector>","gas":15000000,"estimate":true}' &
  done
  sleep 1
done
```

**Result:** The node performs up to 10,500 EVM executions per second (500 requests × 21 executions) while the throttle accounts for only 500 requests worth of gas, increasing CPU consumption by over 20x compared to equivalent non-estimate calls.

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
