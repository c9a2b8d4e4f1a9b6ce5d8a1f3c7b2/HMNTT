### Title
Hardcoded `sslmode=disable` in Rosetta `GetDsn()` Enables Unauthenticated MITM/DNS-Spoofing Against PostgreSQL, Allowing Fabrication of `staking_reward_transfer` Records

### Summary
`GetDsn()` in `rosetta/app/config/types.go` unconditionally hardcodes `sslmode=disable` in the PostgreSQL DSN with no configurable override, unlike every other mirror-node component. An attacker with network-level access (DNS poisoning, ARP spoofing, or BGP hijacking) can redirect the Rosetta service to a malicious PostgreSQL server and return fabricated `staking_reward_transfer` rows with inflated reward amounts, which Rosetta will serve as authoritative balance data to API consumers.

### Finding Description

**Exact code location:**

`rosetta/app/config/types.go`, `GetDsn()`, line 51:
```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        ...
    )
}
``` [1](#0-0) 

This DSN is consumed directly in `ConnectToDb()`:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), ...)
``` [2](#0-1) 

And also in the health check:
```go
Check: postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
``` [3](#0-2) 

**Root cause:** The `Db` struct contains no SSL/TLS fields whatsoever:
```go
type Db struct {
    Host, Name, Password string
    Pool             Pool
    Port             uint16
    StatementTimeout int
    Username         string
}
``` [4](#0-3) 

There is no `SslMode`, `SslCert`, `SslKey`, or `SslRootCert` field. The string `sslmode=disable` is a compile-time constant with no runtime override path. The configuration documentation confirms no SSL property exists for Rosetta: [5](#0-4) 

**Contrast with every other component:** `graphql`, `web3`, `restJava`, and `importer` all expose a configurable `sslMode` (defaulting to `DISABLE` but overridable to `REQUIRE`, `VERIFY_CA`, or `VERIFY_FULL`): [6](#0-5) [7](#0-6) 

**Exploit flow:**

1. In production Kubernetes deployments, `HIERO_MIRROR_ROSETTA_DB_HOST` is set to a DNS name (e.g., the cluster service name `db`): [8](#0-7) 

2. An attacker on the same cluster network (compromised pod, rogue container, or lateral movement) poisons CoreDNS or performs ARP spoofing to redirect the resolved IP of `db` to an attacker-controlled host on port 5432.

3. Because `sslmode=disable` is hardcoded, the PostgreSQL client sends the startup packet in plaintext with no TLS negotiation. The attacker's server accepts the connection without needing any certificate.

4. The attacker's server authenticates using the credentials transmitted in plaintext (or accepts `trust` auth), then responds to the query in `transaction.go`:
```sql
select json_agg(json_build_object('account_id', account_id, 'amount', amount) ...)
from staking_reward_transfer
where consensus_timestamp = t.consensus_timestamp
``` [9](#0-8) 

5. The malicious server returns fabricated rows with arbitrarily inflated `amount` values. Rosetta processes these as `staking_reward_payouts` and includes them in block/transaction responses served to downstream clients.

**Why existing checks fail:** There are none. `sslmode=disable` instructs the `lib/pq` / `pgx` driver to skip TLS entirely — no certificate is requested, no hostname is verified, no channel binding occurs. The Rosetta `Db` struct provides zero mechanism to harden this.

### Impact Explanation

An attacker who can intercept the Rosetta→PostgreSQL TCP stream (DNS spoofing, ARP poisoning, BGP hijack, or compromised network infrastructure) can inject arbitrary `staking_reward_transfer` records. Rosetta is used by exchanges and custodians (Coinbase Rosetta standard) to verify HBAR balances and transaction history. Fabricated staking reward amounts could cause exchanges to credit users with HBAR they did not earn, or cause reconciliation failures that halt withdrawals. The credentials themselves are also exposed in plaintext over the wire, enabling further database compromise.

### Likelihood Explanation

The precondition is network-level access between the Rosetta pod and the PostgreSQL service — not OS-level privilege on either host. In Kubernetes, this is achievable by: (a) compromising any pod in the same namespace/node, (b) exploiting a misconfigured NetworkPolicy, or (c) DNS cache poisoning of CoreDNS (CVE-class attacks exist). In cloud deployments where the DB host is an external FQDN, off-path DNS poisoning (Kaminsky-style) is feasible without any cluster access. The attack is repeatable on every new connection (pool reconnects, health checks).

### Recommendation

1. Add SSL fields to the `Db` struct in `rosetta/app/config/types.go`:
```go
type Db struct {
    ...
    SslMode     string `yaml:"sslMode"`     // default: "require"
    SslRootCert string `yaml:"sslRootCert"`
    SslCert     string `yaml:"sslCert"`
    SslKey      string `yaml:"sslKey"`
}
```
2. Update `GetDsn()` to use the configured value instead of the hardcoded literal:
```go
sslmode=%s sslrootcert=%s sslcert=%s sslkey=%s
```
3. Default `SslMode` to `verify-full` (not `disable`) in the embedded default config, consistent with security best practice.
4. Document the new properties in `docs/configuration.md` alongside the other components' `sslMode` entries.

### Proof of Concept

```bash
# 1. On attacker machine (192.168.1.100), start a rogue PostgreSQL server
#    that accepts any credentials and returns fabricated staking rewards:
docker run -e POSTGRES_HOST_AUTH_METHOD=trust -e POSTGRES_DB=mirror_node \
  -p 5432:5432 postgres:15

# Inject fake staking_reward_transfer rows with inflated amounts:
psql -h 127.0.0.1 -U postgres mirror_node -c "
  CREATE TABLE IF NOT EXISTS staking_reward_transfer (
    account_id bigint, amount bigint,
    consensus_timestamp bigint, payer_account_id bigint
  );
  INSERT INTO staking_reward_transfer VALUES (1234, 999999999999, 1700000000000000000, 800);
"

# 2. Poison DNS for the Rosetta pod (e.g., via CoreDNS configmap or ARP spoof)
#    so that 'db' resolves to 192.168.1.100

# 3. Rosetta connects with sslmode=disable — no TLS negotiation, plaintext auth:
#    The startup packet containing the password is visible on the wire:
tcpdump -i eth0 -A port 5432 | grep -a "password"

# 4. Query Rosetta's block endpoint — it returns the fabricated reward amount:
curl -X POST http://rosetta-host:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{"index": <target_block>}}'
# Response will contain staking_reward_payout with amount=999999999999
```
