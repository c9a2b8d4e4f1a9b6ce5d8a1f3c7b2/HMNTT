### Title
Unbounded Concurrent Historical Subscriptions Exhaust DB Connection Pool (Non-Network DoS)

### Summary
`TopicMessageServiceImpl.subscribeTopic()` unconditionally starts a long-lived database polling loop (`topicMessageRetriever.retrieve(filter, true)`) for every subscription without any global subscriber cap or per-user/per-IP rate limit. An unprivileged attacker can open arbitrarily many gRPC connections (each carrying up to `maxConcurrentCallsPerConnection=5` streams) and subscribe to a high-volume topic with `startTime` at network genesis and no `limit`/`endTime`, causing hundreds of concurrent DB polling loops that exhaust the R2DBC connection pool and deny service to all legitimate users.

### Finding Description

**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()` at line 63:

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
```

This is called unconditionally for every subscriber before any global concurrency check. The `subscriberCount` field (line 48) is a Micrometer gauge metric only — it is never compared against any threshold to reject new subscriptions.

**Root cause in the retriever:**

In `PollingTopicMessageRetriever.retrieve()` (line 45–63), when `throttled=true` (the historical path):

```java
numRepeats = Long.MAX_VALUE;          // line 99 — polls forever
frequency  = retrieverProperties.getPollingFrequency();
maxPageSize = retrieverProperties.getMaxPageSize();
```

`isComplete()` (line 121–128) returns `false` as long as the last page was full — which is always true for a topic with millions of messages and no `limit` set. So each subscription drives an infinite DB polling loop (`Flux.defer(() -> poll(context)).repeatWhen(...)`) that issues `topicMessageRepository.findByFilter(newFilter)` queries continuously.

**Why existing checks fail:**

| Check | Why insufficient |
|---|---|
| `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14) | Limits streams *per HTTP/2 connection*, not globally. Attacker opens K connections → 5K concurrent subscriptions. |
| `subscriberCount` (line 48) | Metric gauge only; never enforced as a cap. |
| `.timeout(retrieverProperties.getTimeout(), scheduler)` (line 59) | Attacker reconnects immediately after timeout; no backoff or ban. |
| `@Min(0)` on `limit` in `TopicMessageFilter` (line 25) | `limit=0` means *no limit* (`hasLimit()` returns false); no minimum enforced. |
| `isValidStartTime()` (line 48–50 of filter) | Only requires `startTime <= now()`; attacker sets it to nanosecond 0 (network genesis). |

**Exploit flow:**

1. Attacker identifies a topic with large historical message volume (publicly visible on-chain).
2. Attacker opens K gRPC connections to the mirror node's port 5600.
3. On each connection, attacker opens 5 `subscribeTopic` streams with `startTime=0`, `limit=0`, `endTime=null`.
4. Each stream triggers `retrieve(filter, true)` → infinite polling loop → continuous R2DBC connection acquisition.
5. R2DBC connection pool (typically 10 connections by default in Spring R2DBC) is saturated.
6. All other gRPC and REST queries that need DB access queue indefinitely or fail with connection timeout errors.

### Impact Explanation

Complete denial of service for all users of the mirror node's gRPC API (and potentially REST/Web3 APIs sharing the same DB pool). Legitimate `subscribeTopic`, `getAccountBalance`, and other queries fail or hang. The mirror node becomes operationally useless. Given the mirror node's role in serving Hedera network data to wallets, exchanges, and dApps, this constitutes a high-severity availability impact.

### Likelihood Explanation

No authentication or API key is required to call `subscribeTopic` — it is a fully public gRPC endpoint. The attacker needs only a gRPC client (e.g., `grpcurl`, any HCS SDK) and knowledge of one high-volume topic ID (trivially discoverable from the public ledger). The attack is repeatable, scriptable, and requires no special hardware. A single attacker machine can open hundreds of connections.

### Recommendation

1. **Enforce a global subscriber cap**: Compare `subscriberCount.get()` against a configurable maximum before creating a new subscription; return `RESOURCE_EXHAUSTED` gRPC status if exceeded.
2. **Per-IP/per-client rate limiting**: Limit the number of concurrent subscriptions per source IP using a token bucket or semaphore at the gRPC interceptor layer.
3. **Require `endTime` or `limit` for historical queries**: Reject subscriptions where `startTime` is far in the past and neither `endTime` nor `limit` is set, or enforce a maximum lookback window.
4. **Bound the retriever's DB concurrency**: Use a `Semaphore` or `Schedulers.newBoundedElastic` with a cap tied to the R2DBC pool size to limit simultaneous `retrieve()` calls.
5. **Add a maximum `startTime` age**: Reject filters where `now() - startTime > configurable_max_age`.
