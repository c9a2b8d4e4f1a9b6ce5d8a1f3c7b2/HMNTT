### Title
Integer Overflow in `poll()` Causes Unlimited Database Query via `hasLimit()` Bypass

### Summary
In `PollingTopicMessageRetriever.poll()`, the remaining-limit calculation `(int)(filter.getLimit() - context.getTotal().get())` silently overflows to a negative integer when `filter.getLimit()` exceeds `Integer.MAX_VALUE` (2,147,483,647). The resulting negative `pageSize` is passed into a rebuilt `TopicMessageFilter`, where `hasLimit()` returns `false` (since `limit > 0` is false for negatives), causing `TopicMessageRepositoryCustomImpl.findByFilter()` to issue an unbounded SQL query with no `LIMIT` clause — loading the entire topic message table into JVM heap on every poll cycle.

### Finding Description

**Entry point** — `ConsensusController.toFilter()`: [1](#0-0) 

The protobuf field `uint64 limit` maps to Java `long`. It is assigned directly to `TopicMessageFilter.limit` with no upper-bound check. The only constraint on `TopicMessageFilter.limit` is `@Min(0)`: [2](#0-1) 

A value such as `3_000_000_000L` (> `Integer.MAX_VALUE`) passes `@Min(0)` validation without error.

**Overflow site** — `PollingTopicMessageRetriever.poll()` lines 68–71: [3](#0-2) 

With `filter.getLimit() = 3_000_000_000L` and `context.getTotal().get() = 0`:
- `3_000_000_000L - 0L = 3_000_000_000L`
- `(int) 3_000_000_000L = -1_294_967_296` (Java silent narrowing overflow)
- `Math.min(-1_294_967_296, maxPageSize) = -1_294_967_296`

The negative `pageSize` is embedded into a new filter via `toBuilder().limit(pageSize).build()`: [4](#0-3) 

Lombok's `build()` is a plain constructor — it does **not** trigger Spring Bean Validation. The negative limit reaches `findByFilter()` unchecked.

**Limit bypass** — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51–53: [5](#0-4) 

`hasLimit()` is defined as `return limit > 0`: [6](#0-5) 

A negative limit fails `limit > 0`, so `setMaxResults()` is never called. The JPA query executes with **no SQL LIMIT**, returning every row for the topic.

**Partial mitigation that does NOT prevent the DB attack** — `TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` on the original filter: [7](#0-6) 

This caps what is *delivered to the client* at 3 billion messages, but it does not prevent each individual `poll()` invocation from issuing an unlimited DB query and loading all results into JVM heap. In throttled mode, `numRepeats = Long.MAX_VALUE`: [8](#0-7) 

The `isComplete()` check `filter.getLimit() == total.get()` will never fire (3B messages would need to exist), so polling repeats indefinitely, each time issuing an unbounded full-table scan.

The identical overflow exists in `PollingTopicListener.poll()` line 54–56: [9](#0-8) 

### Impact Explanation

Each poll cycle fetches the entire `topic_message` table for the targeted topic into JVM heap with no row limit. On a topic with millions of messages this causes severe memory pressure or OOM crash of the gRPC service. Multiple concurrent subscriptions with this crafted limit amplify the effect. The server's database connection pool is also saturated by long-running full-table scans. This is a reliable, unauthenticated denial-of-service against the mirror node gRPC service.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. The protobuf `uint64 limit` field accepts any 64-bit unsigned value. Setting `limit = 2_147_483_648` (Integer.MAX_VALUE + 1) is the minimum value to trigger the overflow and requires no special privileges or knowledge beyond the public API schema. The attack is trivially repeatable with any gRPC client.

### Recommendation

1. **Immediate fix**: In `poll()` (both `PollingTopicMessageRetriever` and `PollingTopicListener`), guard against overflow before the cast:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
2. **Add an upper-bound validator** on `TopicMessageFilter.limit`, e.g. `@Max(Integer.MAX_VALUE)` or a domain-appropriate cap, so oversized values are rejected at the validation layer before reaching retrieval logic.
3. **Validate rebuilt filters**: Consider adding a static factory or validation step when `toBuilder().build()` is used internally so negative limits cannot propagate to the repository layer.

### Proof of Concept

**Preconditions**: A topic exists with at least one message. The gRPC mirror node is reachable.

**Steps**:
1. Connect to the gRPC mirror node using any gRPC client (e.g., `grpcurl` or the Hedera Java SDK).
2. Send a `subscribeTopic` request:
   ```
   topicID: <valid topic ID>
   consensusStartTime: { seconds: 0 }
   limit: 2147483648   // Integer.MAX_VALUE + 1
   ```
3. **Trigger**: `poll()` computes `(int)(2_147_483_648L - 0L) = -2_147_483_648`. `Math.min(-2_147_483_648, maxPageSize) = -2_147_483_648`. The rebuilt filter has `limit = -2_147_483_648`. `hasLimit()` returns `false`. `setMaxResults()` is not called.
4. **Result**: The JPA query executes `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ? ORDER BY consensus_timestamp` with no LIMIT clause. All rows are loaded into heap. This repeats on every poll interval for the lifetime of the subscription.
5. **Observable impact**: JVM heap exhaustion / OOM error in the mirror node gRPC process, or severe latency degradation. Multiple concurrent connections amplify the effect.
