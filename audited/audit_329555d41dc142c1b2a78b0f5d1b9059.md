### Title
Unauthenticated Opcode Replay Endpoint Exposes Full EVM Stack/Memory of All Nested Frames Including STATICCALL Return Values

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController.java` is publicly accessible with no authentication or authorization check. Any unprivileged user who knows a transaction ID or hash can trigger a full EVM replay with `stack=true&memory=true`, receiving the complete opcode trace — including stack and memory snapshots at every step of every nested frame (STATICCALL, CALL, DELEGATECALL). This exposes intermediate computation values, STATICCALL return data, and memory-resident sensitive values that are not surfaced through any other public API.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (lines 52–68) performs three checks before serving the trace:
1. `properties.isEnabled()` — a feature flag (`hiero.mirror.web3.opcode.tracer.enabled`, default `false`; when deployed it is `true`)
2. `validateAcceptEncodingHeader()` — requires `Accept-Encoding: gzip` (trivially satisfied)
3. `throttleManager.throttleOpcodeRequest()` — a global token-bucket rate limit of **1 request/second** shared across all callers, not per-IP or per-user

There is **no Spring Security configuration** in the web3 module (no `SecurityFilterChain`, no `@PreAuthorize`, no `@Secured`), confirmed by the absence of any such annotations across all web3 Java sources.

Once past these checks, `OpcodeServiceImpl.processOpcodeCall()` replays the transaction and `OpcodeActionTracer.tracePostExecution()` is invoked for **every opcode in every message frame**, including frames spawned by STATICCALL:

```java
// OpcodeActionTracer.java:44-61
public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
    final var options = context.getOpcodeContext();
    final var memory = captureMemory(frame, options);   // full 32-byte word dump
    final var stack  = captureStack(frame, options);    // full stack dump
    final var storage = captureStorage(frame, options, context);
    context.getOpcodeContext().addOpcodes(createOpcode(..., stack, memory, storage));
}
```

`captureMemory()` reads the entire EVM memory of the current frame word-by-word; `captureStack()` reads every item on the EVM stack. Both are called unconditionally for every frame, including nested STATICCALL frames at any depth. The `traceContextEnter` / `traceContextReEnter` hooks ensure the tracer follows into and out of nested frames.

**Root cause:** The endpoint is designed as a developer debugging tool but was deployed without any access-control gate. The failed assumption is that the feature flag alone is sufficient protection — but once enabled (which it must be for the feature to work), the endpoint is world-readable.

**Exploit flow:**
1. Attacker observes a transaction hash on-chain (public) for a transaction that internally STATICCALLs a view function (e.g., `balanceOf`, `getSecret`, any function that loads sensitive storage slots into memory/stack).
2. Attacker sends:
   ```
   GET /api/v1/contracts/results/{txHash}/opcodes?stack=true&memory=true
   Accept-Encoding: gzip
   ```
3. The server replays the transaction and returns the full opcode trace. Every `MLOAD`, `MSTORE`, `PUSH`, `RETURN` inside the STATICCALL frame is captured with its full stack and memory snapshot.
4. The attacker reads the `memory` and `stack` arrays from the opcode entries at `depth > 1` (the STATICCALL frame) to recover the return value and all intermediate computation of the view function — data that is never emitted in events or returned to the outer caller.

**Why existing checks fail:**
- The global 1-req/sec throttle (`opcodeRequestsPerSecond = 1`) is shared across all clients; a single attacker can still issue one request per second continuously, and the data is returned on the first request.
- The gzip header requirement is a one-line HTTP header, not a security control.
- The feature flag is binary: off means 404, on means full access to everyone.

### Impact Explanation
Any transaction that involved a STATICCALL to a view function — including Hedera system contracts (HTS precompiles, token balance queries, etc.) — can be replayed to expose the full EVM memory and stack of the callee frame. This includes:
- Return values of view functions that the outer contract consumed internally but never emitted
- Intermediate values computed inside the STATICCALL (e.g., hash preimages loaded into memory, decoded ABI parameters)
- Storage slot values read via SLOAD within the STATICCALL frame (visible in the stack after the SLOAD opcode)

Severity: **High** — it is an information-disclosure vulnerability on a public, unauthenticated HTTP endpoint that exposes execution internals of any historical transaction.

### Likelihood Explanation
- **Precondition**: The feature flag must be enabled (required for the feature to function at all in production).
- **Attacker capability**: Zero — only a known transaction hash is needed, which is public on-chain.
- **Repeatability**: Fully repeatable; the replay is deterministic and the endpoint is stateless.
- **Rate limit bypass**: The 1-req/sec global throttle does not prevent a single attacker from extracting data; one request is sufficient per transaction.

### Recommendation
1. **Add authentication/authorization**: Protect the endpoint with Spring Security. Require an API key, JWT, or role (e.g., `ROLE_ADMIN` or `ROLE_DEBUGGER`) before serving opcode traces.
2. **Per-caller rate limiting**: If the endpoint must remain semi-public, enforce per-IP or per-authenticated-user rate limits rather than a single global bucket.
3. **Restrict `stack` and `memory` flags**: Consider requiring elevated privilege to enable `stack=true` or `memory=true`, since these expose the most sensitive intermediate data.
4. **Audit the feature flag**: Document that enabling `hiero.mirror.web3.opcode.tracer.enabled` opens a public debugging endpoint and ensure it is never enabled in production without accompanying access controls.

### Proof of Concept
```bash
# Step 1: Observe any transaction hash involving a STATICCALL (e.g., from Hedera explorer)
TX_HASH="0xabc123..."

# Step 2: Call the unauthenticated endpoint (no credentials required)
curl -s -H "Accept-Encoding: gzip" \
  "https://<mirror-node-web3>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true" \
  | gunzip \
  | python3 -c "
import json, sys
data = json.load(sys.stdin)
# Find all opcodes from nested frames (depth > 1 = inside STATICCALL)
for op in data['opcodes']:
    if op['depth'] > 1:
        print(f\"op={op['op']} depth={op['depth']} stack={op['stack']} memory={op['memory']}\")
"
# Output: full stack and memory snapshots at every opcode step inside the STATICCALL frame,
# including the RETURN opcode which has the callee's return value on the stack.
```

The `stack` array at the `RETURN` opcode of the STATICCALL frame contains the memory offset and length of the return data; the `memory` array at that point contains the actual return value bytes — data that was never exposed through any other API.
