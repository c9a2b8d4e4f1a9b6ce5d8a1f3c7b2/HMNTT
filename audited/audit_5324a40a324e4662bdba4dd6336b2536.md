### Title
Cache Stampede via Async TOCTOU in `getCachedTokens()` Enables DB Connection Pool Exhaustion (DoS)

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` performs a synchronous cache check followed by an `await`-ed DB query with no in-flight deduplication. Because Node.js yields at every `await`, N concurrent requests for the same uncached token IDs will each independently pass the cache-miss check before any of them writes results back, issuing N redundant DB queries. With the default pool capped at 10 connections, a modest flood of concurrent requests exhausts the pool and renders the REST API unresponsive.

### Finding Description

**Exact code path:**

`rest/service/tokenService.js`, `getCachedTokens()`, lines 136–161:

```
136: async getCachedTokens(tokenIds) {
139:   tokenIds.forEach((tokenId) => {          // ← synchronous cache check
140:     const cachedToken = tokenCache.get(tokenId);
141:     if (cachedToken) { ... } else {
144:       uncachedTokenIds.push(tokenId);       // ← token X added here
       }
     });
148:   if (uncachedTokenIds.length === 0) return cachedTokens;
152:   const rows = await super.getRows(...);    // ← YIELD POINT
153:   rows.forEach((row) => {
156:     tokenCache.set(tokenId, cachedToken);   // ← cache write happens AFTER yield
     });
```

**Root cause:** There is no promise-coalescing or in-flight map. The `await` on line 152 surrenders the event loop. Any concurrent invocation that already passed the `tokenCache.get()` check on line 140 (and found a miss) will also reach line 152 and fire its own independent DB query for the same token IDs. The failed assumption is that the synchronous check-then-query sequence is atomic — it is not across `await` boundaries.

**Exploit flow:**

1. Attacker sends N concurrent HTTP requests to any endpoint that calls `getTokenAccounts()` (e.g., `GET /api/v1/accounts/{id}/tokens`) for accounts sharing the same token IDs that are not yet in `tokenCache`.
2. All N coroutines execute the `forEach` cache check synchronously and find misses — `tokenCache` is still empty for those IDs.
3. All N coroutines reach `await super.getRows(...)` and yield.
4. All N DB queries are dispatched simultaneously against the pool.
5. The first response writes to `tokenCache`; subsequent responses redundantly overwrite it — but all N connections were consumed.

**Why existing checks fail:**

- `tokenCache.get()` (line 140) and the guard in `putTokenCache` (line 123) are synchronous and only prevent duplicate work *within a single execution context*. They provide zero protection across concurrent async invocations.
- There is no `Map<tokenId, Promise>` or equivalent in-flight registry to coalesce duplicate pending queries. [1](#0-0) 

### Impact Explanation

The default DB connection pool is capped at **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`). [2](#0-1) 

Each concurrent request that hits a cache miss consumes one pool connection for the duration of the `tokenCacheQuery`. With N ≥ 10 concurrent requests targeting the same uncached token IDs, the pool is fully saturated. All subsequent requests queue behind `connectionTimeoutMillis` (default 20 s), after which they throw `DbError`, causing HTTP 500 responses. Sustained attack keeps the pool perpetually exhausted, making the mirror node REST API unavailable — a complete denial of service for all API consumers. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

No authentication or privilege is required — the `/api/v1/accounts/{id}/tokens` endpoint is public. The attacker needs only:
- Knowledge of any account ID that holds tokens not yet in the warm cache (trivially discoverable from the public API).
- The ability to send ~10–20 concurrent HTTP requests (achievable with `curl --parallel`, `ab`, `wrk`, or any scripting language).

The attack is fully repeatable: the cache is an LRU with `maxSize = 100000` entries; eviction or a process restart resets it, re-opening the window. The attack can be sustained indefinitely by rotating through different account IDs or waiting for cache eviction. [5](#0-4) 

### Recommendation

Introduce an **in-flight promise registry** (promise coalescing) keyed on the set of uncached token IDs. Before issuing a DB query, check whether an identical query is already in flight and await its result instead of issuing a new one:

```js
const inFlight = new Map(); // module-level, alongside tokenCache

async getCachedTokens(tokenIds) {
  // ... existing forEach cache check ...

  if (uncachedTokenIds.length === 0) return cachedTokens;

  const key = uncachedTokenIds.slice().sort().join(',');
  if (!inFlight.has(key)) {
    const promise = super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds])
      .then((rows) => {
        rows.forEach((row) => {
          tokenCache.set(row.token_id, new CachedToken(row));
        });
        inFlight.delete(key);
        return rows;
      })
      .catch((err) => { inFlight.delete(key); throw err; });
    inFlight.set(key, promise);
  }

  const rows = await inFlight.get(key);
  // populate cachedTokens from tokenCache for uncachedTokenIds
  ...
}
```

Alternatively, use a library such as `p-memoize` with a short TTL to deduplicate concurrent identical async calls.

### Proof of Concept

**Preconditions:** Mirror node REST API running with default config (pool `maxConnections = 10`). Identify an account ID (e.g., `0.0.1234`) associated with tokens not yet in the warm cache (restart the service or wait for LRU eviction).

**Steps:**

```bash
# Send 15 concurrent requests for the same account's token relationships
seq 1 15 | xargs -P 15 -I{} \
  curl -s -o /dev/null -w "%{http_code}\n" \
  "http://<mirror-node-host>:5551/api/v1/accounts/0.0.1234/tokens"
```

**Expected result without fix:** The first 10 requests each consume a DB connection for `tokenCacheQuery`. Requests 11–15 (and any other concurrent API requests) queue waiting for a free connection. If the query takes > 20 s or the pool stays saturated, subsequent requests return HTTP 500 (`DbError: timeout`). Repeating the burst keeps the pool exhausted.

**Observed DB side:** In PostgreSQL `pg_stat_activity`, you will see N identical copies of the `tokenCacheQuery` (`where token_id = any ($1)`) executing simultaneously with the same parameter values — confirming the duplicate-query amplification. [6](#0-5) [7](#0-6)
