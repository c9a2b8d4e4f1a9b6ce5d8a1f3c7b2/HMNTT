### Title
Unauthenticated High-Volume Enumeration of `getAirdrops()` Causes DB Buffer Pool Eviction Due to Missing Rate Limiting in rest-java Module

### Summary
The `/api/v1/accounts/{id}/airdrops/outstanding` and `/api/v1/accounts/{id}/airdrops/pending` endpoints are publicly accessible with no rate limiting in the `rest-java` module. Each request triggers a direct database query against the `token_airdrop` table keyed by a different `account_id`. An unprivileged attacker enumerating sequential Hedera account IDs (which are public and monotonically increasing) can continuously evict and refill DB buffer pool pages, degrading cache hit rates and increasing I/O.

### Finding Description

**Code path:**

`TokenAirdropsController` (lines 66–86) accepts unauthenticated GET requests and calls `service.getAirdrops(request)` with no throttle check: [1](#0-0) 

`TokenAirdropServiceImpl.getAirdrops()` (lines 19–22) performs an entity lookup then immediately delegates to the repository: [2](#0-1) 

`TokenAirdropRepositoryCustomImpl.findAll()` (lines 58–72) executes a live SQL query against `TOKEN_AIRDROP` for every call, with no caching layer: [3](#0-2) 

**Root cause — missing rate limiting in rest-java:** The `web3` module has a full `ThrottleManagerImpl` with bucket4j-based per-second request and gas limits: [4](#0-3) 

The `rest-java` module has no equivalent. Its `RestJavaConfiguration` registers only an ETag filter and a metrics filter — no rate-limiting filter: [5](#0-4) 

The `MetricsFilter` only records byte counts; it does not throttle: [6](#0-5) 

**Failed assumption:** The developers assumed infrastructure-level controls (load balancer, reverse proxy) would protect these endpoints, but no application-layer rate limiting exists for the rest-java airdrop API.

### Impact Explanation
On a production Hedera mirror node with a large `token_airdrop` table (millions of rows across hundreds of thousands of accounts), each request for a distinct `account_id` loads a different set of index and heap pages into the PostgreSQL shared buffer pool. Continuous enumeration across thousands of distinct account IDs forces constant page eviction, reducing cache hit ratio and increasing physical I/O. The `limit` cap of 100 rows per request: [7](#0-6) 
means each request is cheap enough to sustain at high concurrency, but collectively the working set far exceeds the buffer pool, causing measurable I/O amplification and latency degradation for all other queries sharing the same database.

### Likelihood Explanation
Hedera account IDs are sequential integers publicly visible on any block explorer. No authentication, API key, or proof-of-work is required to call these endpoints. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple script) can issue thousands of requests per second from a single IP or distributed across multiple IPs. The attack is repeatable indefinitely and requires no special knowledge beyond the public API specification: [8](#0-7) 

### Recommendation
1. Add a bucket4j-based rate-limiting filter to the `rest-java` module mirroring the pattern in `ThrottleConfiguration` and `ThrottleManagerImpl` in the `web3` module, applying a global requests-per-second cap and optionally a per-IP cap.
2. Consider adding a short-lived application-level cache (e.g., Caffeine with `expireAfterWrite=1s`) for `getAirdrops()` results keyed by `(accountId, type, params)` to absorb repeated identical queries.
3. Enforce connection-level rate limiting at the reverse proxy/ingress layer as a defense-in-depth measure.

### Proof of Concept
```bash
# Enumerate 10,000 sequential Hedera account IDs at high concurrency
# No authentication required
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/accounts/$i/airdrops/outstanding?limit=100" &
done
wait

# Alternatively with wrk for sustained load:
wrk -t 16 -c 200 -d 300s \
  --script <(echo 'math.randomseed(os.time()); \
    request = function() \
      local id = math.random(1, 5000000) \
      return wrk.format("GET", "/api/v1/accounts/"..id.."/airdrops/outstanding?limit=100") \
    end') \
  https://<mirror-node>/
```
Each request hits a different `sender_account_id` index partition, continuously displacing cached pages. Monitor PostgreSQL `pg_stat_bgwriter` (`buffers_clean`, `maxwritten_clean`) and `pg_statio_user_tables` (`heap_blks_read` vs `heap_blks_hit`) before and during the attack to observe the cache hit ratio drop and I/O increase.

```java
    public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
        var type = request.getType();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, type.getBaseField())
                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

        var order = SORT_ORDERS.getOrDefault(type, Map.of()).get(request.getOrder());
        return dslContext
                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())
                .fetchInto(TokenAirdrop.class);
    }
```
