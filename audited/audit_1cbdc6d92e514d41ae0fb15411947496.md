### Title
Cache-Miss DoS via Non-Existent ContractId Lookups with Gas-Throttle Bypass

### Summary
`ContractRepository.findRuntimeBytecode()` uses `unless = "#result == null"` on a method returning `Optional<byte[]>`. Spring's caching layer unwraps `Optional` before evaluating `unless`, so `Optional.empty()` (returned for every non-existent contract) is treated as a null result and is **never cached**. Simultaneously, the gas-based throttle is completely bypassed when `gas <= 10,000` because `scaleGas()` returns `0` and `bucket4j`'s `tryConsume(0)` always succeeds. An unauthenticated attacker can therefore sustain 500 DB queries per second indefinitely with no gas cost.

### Finding Description

**Root cause 1 — cache never populated for non-existent contracts** [1](#0-0) 

Spring's `CacheAspectSupport` calls `unwrapReturnValue()` on the method's return value before binding `#result` in the `unless` SpEL expression. For `Optional.empty()`, the unwrapped value is `null`. Therefore `#result == null` evaluates to `true` and the result is **excluded from the cache** on every invocation for a non-existent contract. The same contractId can be queried repeatedly and will hit the database every time.

**Root cause 2 — gas throttle bypass for gas ≤ 10,000** [2](#0-1) 

`scaleGas(gas)` returns `0L` for any `gas <= 10_000`. This value is passed directly to: [3](#0-2) 

`bucket4j`'s `tryConsume(0)` always returns `true` regardless of bucket state, so the gas-per-second throttle is completely inoperative for these requests. The only remaining guard is the RPS bucket: [4](#0-3) 

Default is 500 requests/second.

**Call path from eth_call to DB** [5](#0-4) 

→ `ContractExecutionService.processCall()` → EVM execution → `ContractBytecodeReadableKVState.readFromDataSource()`: [6](#0-5) 

→ `contractRepository.findRuntimeBytecode(entityId.getId())` — cache miss every time for non-existent IDs → raw SQL against the `contract` table.

### Impact Explanation

Every sustained 500 RPS flood of `eth_call` requests targeting non-existent contract addresses generates 500 uncached SQL queries per second against the mirror node's PostgreSQL database. The database connection pool exhausts, query latency spikes, and legitimate `eth_call`, `eth_estimateGas`, and REST API traffic degrades or fails entirely. Because the mirror node is the sole EVM-simulation layer for dApps relying on Hedera's JSON-RPC relay, sustained degradation effectively takes the web3 API offline without touching consensus nodes.

### Likelihood Explanation

No account, token, or privileged credential is required — only the ability to send HTTP POST requests to `/api/v1/contracts/call`. The attacker sets `gas` to any value ≤ 10,000 (e.g., `"gas": "0x0"`) and rotates the `to` field across arbitrary 20-byte addresses that do not correspond to deployed contracts. A single machine with a modest HTTP client can sustain 500 RPS. The attack is repeatable indefinitely and requires no prior knowledge of the system beyond the public API endpoint.

### Recommendation

1. **Fix the `unless` condition** to correctly handle `Optional` return types. Replace:
   ```java
   unless = "#result == null"
   ```
   with:
   ```java
   unless = "#result == null || !#result.isPresent()"
   ```
   This ensures `Optional.empty()` (non-existent contracts) is also cached, preventing repeated DB hits for the same absent contractId.

2. **Fix the gas-throttle bypass** in `scaleGas()`. When `gas == 0`, `tryConsume(0)` should not be called; instead treat zero-gas requests as consuming a minimum token count (e.g., 1), or enforce a minimum gas floor before throttle evaluation.

3. **Add a minimum gas floor** at the controller validation layer so requests with `gas < 21_000` are rejected before reaching the throttle.

### Proof of Concept

```bash
# Requires: curl, bash
# Sends 500 req/s with gas=0 targeting random non-existent contract addresses
# Each request bypasses gas throttle and causes an uncached DB query

for i in $(seq 1 500); do
  ADDR=$(printf "0x%040x" $RANDOM$RANDOM$RANDOM)
  curl -s -X POST http://<mirror-node>:8545/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$ADDR\",\"gas\":\"0x0\",\"data\":\"0x\",\"estimate\":false}" &
done
wait
```

Each request: (a) passes the gas throttle because `scaleGas(0) = 0` and `tryConsume(0)` always succeeds; (b) misses the cache because `Optional.empty()` is unwrapped to null and excluded by `unless = "#result == null"`; (c) issues a live SQL query `SELECT runtime_bytecode FROM contract WHERE id = ?` against the database. Sustained at 500 RPS this exhausts DB connections and degrades all mirror node services.
