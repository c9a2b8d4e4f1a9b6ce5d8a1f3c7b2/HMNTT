### Title
Unbounded Blocking Thread Exhaustion via `PollingTopicListener.poll()` — No Application-Level Query Timeout

### Summary
`PollingTopicListener.poll()` calls `topicMessageRepository.findByFilter()`, which executes a synchronous blocking JDBC `getResultList()` call on a `Schedulers.boundedElastic()` thread. Unlike `PollingTopicMessageRetriever`, which applies a `.timeout()` operator, `PollingTopicListener` has no application-level timeout. An unauthenticated attacker can open enough concurrent subscriptions with `startTime=0` to saturate all bounded-elastic threads, stalling the entire polling infrastructure for all subscribers.

### Finding Description

**Exact code path:**

`PollingTopicListener.poll()` at line 61:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [1](#0-0) 

`findByFilter` in `TopicMessageRepositoryCustomImpl` at line 60 executes a fully synchronous, blocking JDBC call:
```java
return typedQuery.getResultList().stream();
``` [2](#0-1) 

This call blocks the calling thread for the entire duration of the DB query — it does not return until PostgreSQL finishes and all rows are fetched into memory.

**Scheduler context:** The `poll()` lambda is scheduled on `Schedulers.boundedElastic()` via `delaySubscription(interval, scheduler)` and `repeatWhen(...withScheduler(scheduler))`. This is a finite thread pool capped at `10 × availableProcessors` threads. [3](#0-2) 

**Root cause — missing timeout:** `PollingTopicMessageRetriever` (the historical retriever) correctly applies `.timeout(retrieverProperties.getTimeout(), scheduler)` at line 59. `PollingTopicListener` has no equivalent operator anywhere in its reactive chain. [4](#0-3) 

**No global subscription cap:** `TopicMessageServiceImpl` tracks `subscriberCount` as a Micrometer gauge only — it does not enforce any maximum. `maxConcurrentCallsPerConnection = 5` is a per-connection limit only; an attacker opens many connections. [5](#0-4) [6](#0-5) 

**`startTime=0` is valid:** `TopicMessageFilter.isValidStartTime()` only rejects future timestamps. `startTime=0` (Unix epoch in nanoseconds) passes validation and forces the query to scan from the very beginning of the topic's history. [7](#0-6) 

**No authentication:** `ConsensusController.subscribeTopic()` performs no authentication or authorization check before invoking `topicMessageService.subscribeTopic()`. [8](#0-7) 

### Impact Explanation

When the POLL listener type is configured (`hiero.mirror.grpc.listener.type=POLL`), every active subscription occupies one `boundedElastic` thread for up to 10 seconds per poll cycle (the DB `statementTimeout`). On an 8-core server the pool caps at ~80 threads. An attacker needs only ~16 TCP connections (each carrying 5 concurrent gRPC streams) to saturate all threads. Once saturated:

- All legitimate subscribers' poll tasks queue behind the attacker's tasks.
- The `repeatWhen(RepeatSpec.times(Long.MAX_VALUE)...)` ensures each attacker subscription immediately re-queues after the DB kills the query, maintaining continuous saturation.
- New topic message notifications are never delivered, effectively halting the HCS streaming service.

Severity: **Critical** when POLL mode is active — complete denial of the gRPC topic subscription service.

### Likelihood Explanation

- No credentials or special privileges are required; the gRPC `subscribeTopic` endpoint is unauthenticated.
- Topic IDs with large message histories are publicly discoverable on Hedera mainnet/testnet.
- The attack requires only a standard gRPC client (e.g., `grpcurl`) and ~16 TCP connections — trivially scriptable.
- The attack is fully repeatable and self-sustaining due to `Long.MAX_VALUE` repeat cycles.
- The only prerequisite is that the operator has configured `listener.type=POLL` (a documented, supported option).

### Recommendation

1. **Add a reactive timeout to `PollingTopicListener.listen()`**, mirroring `PollingTopicMessageRetriever`:
   ```java
   return Flux.defer(() -> poll(context))
       .delaySubscription(interval, scheduler)
       .repeatWhen(...)
       .timeout(listenerProperties.getInterval().multipliedBy(10), scheduler)
       ...
   ```
2. **Enforce a global concurrent-subscription limit** in `TopicMessageServiceImpl`, rejecting new subscriptions when `subscriberCount` exceeds a configurable threshold.
3. **Add a JPA query timeout hint** (`HibernateHints.HINT_QUERY_TIMEOUT`) in `TopicMessageRepositoryCustomImpl.findByFilter()` so the application layer — not just the DB — enforces a bound.
4. **Consider IP-level rate limiting** at the load balancer for `subscribeTopic` calls (the Helm chart already has `maxRatePerEndpoint` for GCP gateway but this is not universally enforced).

### Proof of Concept

```bash
# Step 1: Identify a topic with a large message history (public info on Hedera mainnet)
# e.g., topic 0.0.41110

# Step 2: Open 16 connections × 5 streams = 80 concurrent subscriptions,
#         each with startTime=0 (epoch) to maximize query scan depth.
for i in $(seq 1 80); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":41110},"consensusStartTime":{"seconds":0,"nanos":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 3: Observe that legitimate subscribers receive no messages.
# Each attacker stream blocks a boundedElastic thread for ~10s (DB statementTimeout),
# then immediately re-polls, maintaining saturation indefinitely.
# The polling infrastructure stalls; no new topic messages are delivered to any subscriber.
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)

                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
