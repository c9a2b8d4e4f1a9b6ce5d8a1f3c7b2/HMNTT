### Title
Rate Token Permanently Consumed Without Restoration on Gas Limit Failure Enables Accelerated Rate-Bucket Depletion DoS

### Summary
In `ThrottleManagerImpl.throttle()`, when `rateLimitBucket.tryConsume(1)` succeeds but `gasLimitBucket.tryConsume()` subsequently fails, the consumed rate token is never restored. The `restore()` method only replenishes gas tokens, not rate tokens. An unprivileged attacker can exploit this asymmetry to drain the rate bucket at full speed while the gas bucket is empty, denying service to legitimate users even after the gas bucket refills.

### Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–42:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
``` [1](#0-0) 

When the rate check passes (1 token deducted from `rateLimitBucket`) and the gas check fails (bucket4j's `tryConsume` returns `false` without consuming any gas tokens), a `ThrottleException` is thrown. Control never reaches any restoration path for the rate token.

**`restore()` only touches the gas bucket:**

```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);   // rate bucket untouched
    }
}
``` [2](#0-1) 

**The controller only calls `restore()` for `InvalidParametersException`, never for `ThrottleException`:**

```java
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
``` [3](#0-2) 

**Root cause / failed assumption:** The design assumes that a request rejected by the gas check should still "count" against the rate limit. This is incorrect: the gas-rejected request never executes, so consuming a rate slot for it allows an attacker to exhaust rate tokens without ever consuming gas tokens, breaking the independence of the two throttle dimensions.

**`scaleGas` boundary:** Requests with `gas <= 10,000` return `scaleGas = 0`, so `tryConsume(0)` always succeeds. The attacker must use `gas > 10,000` to trigger the failure path. [4](#0-3) 

### Impact Explanation
Default configuration: `requestsPerSecond = 500`, `gasPerSecond = 7,500,000,000`. An attacker who exhausts the gas bucket with a single max-gas request can then fire 499 additional requests per second that each pass the rate check but fail the gas check. All 500 rate tokens are consumed per second. When the gas bucket refills one second later, legitimate users still cannot proceed because the rate bucket is simultaneously drained by the attacker's gas-failing requests. This creates a sustained, self-reinforcing DoS: the attacker keeps the rate bucket empty while the gas bucket cycles, permanently blocking legitimate traffic. No privileged access, API key, or authentication is required.

### Likelihood Explanation
Any unprivileged external user can reach the `/api/v1/contracts/call` endpoint. The attack requires only HTTP POST requests with a high `gas` field value — no special tooling beyond a basic HTTP client or load generator (e.g., `wrk`, `ab`, `curl` in a loop). The attacker needs to sustain ~500 req/s, which is trivially achievable from a single machine or a small botnet. The attack is repeatable indefinitely and requires no prior knowledge of the system beyond the public API.

### Recommendation
1. **Restore the rate token on gas-check failure.** Add `rateLimitBucket.addTokens(1)` inside the gas-check failure branch before throwing, or restructure the check to be atomic (check both buckets before consuming either).
2. **Alternatively, check gas first.** Reorder the checks so the gas bucket is checked before the rate bucket. A gas-rejected request never consumes a rate token.
3. **Extend `restore()` to accept a rate-token count** and call it from all rejection paths, including `ThrottleException` from the gas branch.
4. **Add a test** that verifies the rate bucket token count is unchanged after a gas-limit rejection.

### Proof of Concept
**Preconditions:** Default config (`requestsPerSecond=500`, `gasPerSecond=7_500_000_000`). No authentication required.

**Step 1 — Exhaust the gas bucket (1 request):**
```
POST /api/v1/contracts/call
{"gas": 7500000000, "to": "0x...", "data": "0x"}
→ HTTP 200 (rate token consumed, gas bucket emptied)
```

**Step 2 — Drain rate tokens without consuming gas (loop at 500 req/s):**
```
for i in 1..499:
  POST /api/v1/contracts/call
  {"gas": 7500000000, "to": "0x...", "data": "0x"}
  → HTTP 429 "Gas per second rate limit exceeded."
    (rate token consumed, gas token NOT consumed)
```

**Step 3 — Observe legitimate user denial:**
```
POST /api/v1/contracts/call  # legitimate user, small gas
{"gas": 100000, "to": "0x...", "data": "0x"}
→ HTTP 429 "Requests per second rate limit exceeded"
  (rate bucket empty, even though gas bucket has capacity)
```

**Step 4 — Repeat Steps 1–3 every second.** The gas bucket refills, but the attacker immediately re-exhausts it and drains the rate bucket again, sustaining the DoS indefinitely.
