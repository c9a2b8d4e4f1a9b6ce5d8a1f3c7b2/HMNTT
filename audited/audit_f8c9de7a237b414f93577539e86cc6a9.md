### Title
`Optional.empty()` Incorrectly Cached in `CACHE_MANAGER_RECORD_FILE_INDEX` Due to Flawed `unless` Condition on `findByIndex()`

### Summary
`RecordFileRepository.findByIndex()` is annotated with `@Cacheable(unless = "#result == null")`, but the method returns `Optional<RecordFile>`. When no record exists, Spring Data JPA returns `Optional.empty()` â€” a non-null object â€” so the `unless` guard never fires, and the empty result is stored in `CACHE_MANAGER_RECORD_FILE_INDEX` for 10 minutes. Any unprivileged caller who queries a block index before it is ingested will receive a stale "not found" response for the full TTL window even after the block lands in the database.

### Finding Description
**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 27â€“29:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
```

**Root cause:** The `unless` SpEL expression `#result == null` is evaluated against the actual return value of the method. Spring Data JPA never returns a raw `null` from a method declared `Optional<T>`; it returns `Optional.empty()`, which is a non-null singleton. Therefore `#result == null` is always `false` for a miss, the `unless` condition never suppresses caching, and `Optional.empty()` is written into the Caffeine cache backed by `CACHE_MANAGER_RECORD_FILE_INDEX`.

**Cache TTL:** `expireAfterWrite(10, TimeUnit.MINUTES)` with `maximumSize(10000)` â€” confirmed at `EvmConfiguration.java` lines 156â€“158.

**Call path:** `RecordFileServiceImpl.findByBlockType(BlockType)` (line 28) delegates to `recordFileRepository.findByIndex(block.number())` for any numeric or hex block identifier supplied by the caller. This is the public JSON-RPC surface (e.g., `eth_getBlockByNumber`, `eth_getTransactionByBlockNumberAndIndex`).

**Why the existing `@CachePut` on `findByTimestamp` does not mitigate this:** The `@CachePut` at line 47 writes to `CACHE_MANAGER_RECORD_FILE_INDEX` only when `findByTimestamp` is invoked. A caller querying by block number never triggers `findByTimestamp`, so the poisoned index-cache entry is never overwritten until it naturally expires.

### Impact Explanation
Any block that is queried by index before it is fully ingested will be invisible to all callers for up to 10 minutes after ingestion. This directly causes "missing records exported to mirror nodes" as described in the scope: downstream consumers relying on `eth_getBlockByNumber` or any service path through `findByBlockType` will receive an empty response and may skip or misreport the block. With `maximumSize(10000)`, an attacker can pre-poison up to 10,000 distinct future block indices in a single sweep, extending the denial-of-visibility window across a large range of upcoming blocks.

### Likelihood Explanation
No authentication or special privilege is required. The JSON-RPC endpoint is publicly accessible. The attacker only needs to know (or guess) upcoming block numbers, which are strictly sequential and trivially predictable from the current chain tip. The attack is repeatable: after each 10-minute expiry the attacker can re-query to re-poison the cache. A single automated script issuing `eth_getBlockByNumber` for the next N block numbers is sufficient.

### Recommendation
Change the `unless` condition on `findByIndex` (and the analogous conditions on `findEarliest`, `findLatest`, and `findByTimestamp`) to also exclude empty Optionals:

```java
// Before (broken):
@Cacheable(..., unless = "#result == null")

// After (correct):
@Cacheable(..., unless = "#result == null || !#result.isPresent()")
```

Alternatively, use `unless = "#result?.isEmpty() ?: true"` or restructure the return type so that a genuine `null` (never returned by Spring Data) is the sentinel, and cache only confirmed hits.

### Proof of Concept
1. Observe the current latest block index N via `eth_blockNumber`.
2. Send `eth_getBlockByNumber` with parameter `N+1` (block not yet ingested). The mirror node calls `findByIndex(N+1)`, gets `Optional.empty()`, and caches it for 10 minutes.
3. Wait for the importer to ingest block `N+1` (typically within seconds on mainnet cadence).
4. Immediately send `eth_getBlockByNumber` with `N+1` again. The cached `Optional.empty()` is returned; the database is not queried.
5. Repeat step 4 for up to 10 minutes â€” every call returns "block not found" despite the block existing in the database.
6. After the 10-minute TTL expires, the next call correctly returns the block data.
