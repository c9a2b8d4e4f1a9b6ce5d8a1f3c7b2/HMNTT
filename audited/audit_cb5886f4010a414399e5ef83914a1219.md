### Title
Historical eth_call Uses Current Contract Bytecode Instead of Historical Version Due to Missing Timestamp Filter in `findRuntimeBytecode()`

### Summary
`ContractRepository.findRuntimeBytecode()` queries only the `contract` table (current/latest state) with no block timestamp parameter, while the `contract_history` table — which stores historical bytecode versions with `timestamp_range` — is never consulted. When a historical `eth_call` is processed, the EVM executes the current (post-update) bytecode against historical storage state, producing incorrect simulation results. This asymmetry with `findStorageByBlockTimestamp()` (which correctly filters by timestamp) means any unprivileged user can trigger historically inaccurate execution by specifying a past block number.

### Finding Description

**Exact code path:**

`ContractRepository.java` line 17–18:
```java
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```
This queries only the `contract` table — the current live state — with no timestamp predicate.

`ContractBytecodeReadableKVState.java` line 43:
```java
return contractRepository
        .findRuntimeBytecode(entityId.getId())
```
Called unconditionally during EVM execution, including for historical block queries, with no timestamp passed.

**Contrast with storage:**

`ContractStateRepository.java` line 44–54 correctly uses:
```java
select coalesce(value_written, value_read) as value
from contract_state_change
where contract_id = ?1 and slot = ?2 and consensus_timestamp <= ?3
order by consensus_timestamp desc limit 1
```

**Root cause:** The `contract_history` table (confirmed to have `runtime_bytecode` via `V1.64.0__contract_action.sql` lines 24–25) is never queried for historical bytecode lookups. The design assumes bytecode is immutable, but Hedera supports `ContractUpdateTransaction` which changes bytecode and archives the old version to `contract_history`.

**Exploit flow:**
1. Attacker (or any party) deploys contract with bytecode A at block N.
2. Contract bytecode is updated to bytecode B at block M > N. Old bytecode A moves to `contract_history` with `timestamp_range = [N, M)`.
3. Attacker sends `eth_call` with `block = N` (historical).
4. `ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode()`, which returns bytecode B (current) from the `contract` table.
5. EVM executes bytecode B against historical storage state from block N — a hybrid that never existed.

**Why existing checks fail:** There is no timestamp guard anywhere in the bytecode retrieval path. The `@Cacheable` annotation on `findRuntimeBytecode()` further entrenches the current-state result, as a cached current-bytecode response will be served for all historical queries for the same contract ID.

### Impact Explanation
Historical `eth_call` and `eth_estimateGas` queries execute the wrong bytecode. If bytecode B contains different fee-related logic (e.g., different calls to the exchange rate precompile at `0x168`, different gas-consuming operations, or modified HTS precompile interactions), the returned values and gas estimates are incorrect relative to what actually occurred at the historical block. This breaks the integrity guarantee of historical simulation and can cause `eth_estimateGas` to return gas figures that do not reflect the historical state, affecting any tooling or protocol logic that relies on historical call simulation for fee estimation.

### Likelihood Explanation
Any unprivileged user can trigger this with a standard JSON-RPC `eth_call` specifying a historical block number — no special permissions required. The precondition (a contract having been updated) is a normal operational event on Hedera. The attack is trivially repeatable and requires no on-chain transaction by the attacker.

### Recommendation
Add a timestamp-aware query that consults `contract_history` for historical blocks, mirroring the pattern used for storage:

```sql
select runtime_bytecode from contract
where id = :contractId
union all
select runtime_bytecode from contract_history
where id = :contractId
  and timestamp_range @> :blockTimestamp::bigint
limit 1
```

Add a new repository method `findRuntimeBytecodeByBlockTimestamp(Long contractId, Long blockTimestamp)` and update `ContractBytecodeReadableKVState.readFromDataSource()` to pass the block timestamp from `ContractCallContext` when a historical query is in progress, consistent with how other historical repositories operate.

### Proof of Concept
1. Deploy contract with bytecode A (e.g., a contract that calls `tinycentsToTinybars` and returns a hardcoded multiplier of 1). Record block number N.
2. Update the contract via `ContractUpdateTransaction` to bytecode B (same interface, but multiplier of 100). Record block number M.
3. Issue: `eth_call { to: <contract>, data: <tinycentsToTinybars selector>, block: N }`
4. **Expected:** Result uses bytecode A (multiplier 1), consistent with block N state.
