### Title
Unbounded Request Body Parsing in `/mempool` Enables GC-Pressure Griefing

### Summary
The `Mempool()` handler in `rosetta/app/services/mempool_service.go` discards its `*types.NetworkRequest` argument entirely, but the rosetta-sdk-go framework still fully JSON-decodes the request body — including an arbitrarily large `Metadata map[string]interface{}` — into heap memory before the handler is invoked. No `http.MaxBytesReader` or application-level body-size cap exists anywhere in the rosetta Go codebase, and the Traefik rate-limiting middleware is disabled by default (`global.middleware: false`). An unprivileged attacker can flood the endpoint with large bodies, forcing repeated large heap allocations that are immediately abandoned, driving frequent GC cycles and degrading latency for all concurrent users.

### Finding Description
**Exact code path:**

- `rosetta/app/services/mempool_service.go` lines 22–27: `Mempool()` accepts `_ *types.NetworkRequest` (blank identifier) and immediately returns `nil, errors.ErrNotImplemented`. Zero application logic runs; the only work done is the framework-level JSON decode that already happened.
- `rosetta/main.go` lines 220–227: the `http.Server` is constructed with only timeout fields (`ReadTimeout`, `ReadHeaderTimeout`, etc.). No `http.MaxBytesReader` wrapper is applied to `http.Request.Body` at any point in the middleware chain (`MetricsMiddleware` → `TracingMiddleware` → `CorsMiddleware` → router).
- `grep` across all `rosetta/**/*.go` for `MaxBytesReader|MaxBytes|LimitReader` returns **zero matches**.

**Root cause:** The framework (rosetta-sdk-go) reads and decodes the full request body before dispatching to the handler. `types.NetworkRequest.Metadata` is typed `map[string]interface{}`, which the JSON decoder will populate with arbitrarily nested, arbitrarily large data. Because the handler discards the result, every byte parsed is a pure waste allocation that the GC must later collect.

**Why existing checks fail:**
- `ReadTimeout: 5 s` (5 000 000 000 ns, `docs/configuration.md` line 664) limits *time*, not *bytes*. On a 100 Mbit/s link an attacker can deliver ~62 MB per request within that window.
- The Traefik `inFlightReq` (5 per IP) and `rateLimit` (10 req/s) middleware in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 is conditioned on `{{ if and .Values.global.middleware .Values.middleware }}` (`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3); `global.middleware` defaults to `false` (values.yaml line 95), so this protection is **off by default**.

### Impact Explanation
Each oversized request forces the Go runtime to allocate a large `map[string]interface{}` tree, immediately making it garbage. Under sustained concurrent bombardment this produces stop-the-world (or concurrent-mark) GC pauses that increase tail latency for every legitimate request being processed at the same time. Because the endpoint requires no authentication and returns a deterministic `ErrNotImplemented` response, the attacker receives instant confirmation that each request was processed, enabling tight loop automation. The impact is availability degradation (latency spikes, potential timeout cascades) rather than data loss, consistent with the Medium/griefing classification.

### Likelihood Explanation
No privilege, credential, or special network position is required — only the ability to send HTTP POST requests to port 5700 (or whatever the exposed ingress resolves to). The attack is trivially scriptable with `curl` or any HTTP load tool. Because the endpoint is listed in the ingress path table (`/rosetta/mempool`, values.yaml line 131) it is intentionally internet-reachable in standard deployments. Repeatability is unlimited given the absence of application-level rate limiting in the default configuration.

### Recommendation
1. **Wrap `http.Request.Body` with `http.MaxBytesReader`** in a middleware applied before the router, e.g.:
   ```go
   func maxBodyMiddleware(next http.Handler, limit int64) http.Handler {
       return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           r.Body = http.MaxBytesReader(w, r.Body, limit)
           next.ServeHTTP(w, r)
       })
   }
   ```
   Insert it in the chain in `main.go` before `metricsMiddleware`. A limit of 8–16 KB is sufficient for all legitimate Rosetta `NetworkRequest` payloads.
2. **Enable the Traefik middleware by default** by changing `global.middleware` to `true` in `charts/hedera-mirror-rosetta/values.yaml` line 95, so the `inFlightReq` and `rateLimit` guards are active in all Kubernetes deployments.
3. Consider returning early with a `413 Request Entity Too Large` before JSON decoding for any endpoint whose handler is known to be a stub (`ErrNotImplemented`).

### Proof of Concept
```bash
# Generate a ~10 MB metadata payload
python3 -c "
import json, sys
payload = {'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
           'metadata': {'x': 'A' * 10_000_000}}
sys.stdout.write(json.dumps(payload))
" > big_payload.json

# Flood the endpoint from a single unprivileged client
for i in $(seq 1 200); do
  curl -s -o /dev/null -X POST http://<rosetta-host>:5700/mempool \
       -H 'Content-Type: application/json' \
       --data-binary @big_payload.json &
done
wait
```
Expected result: Go runtime GC pause metrics (`go_gc_duration_seconds`) spike; `hiero_mirror_rosetta_request_duration` p99 latency for concurrent legitimate requests increases measurably. The endpoint returns `{"code":501,...}` for every request, confirming full body parsing occurred before the handler ran.
