### Title
Unauthenticated DoS via Unbounded DB Query Flooding on GET /api/v1/network/stake

### Summary
The `getNetworkStake()` handler in `NetworkController` delegates directly to `networkStakeRepository.findLatest()` on every request with no caching and no rate limiting. Because the endpoint is fully public and the data it serves changes at most once per day, an unprivileged attacker can trivially saturate the database connection pool by issuing high-frequency requests, starving all other concurrent queries and causing service-wide degradation of the mirror node REST API.

### Finding Description
**Exact code path:**

`NetworkController.getNetworkStake()` (line 126–130) calls `networkService.getLatestNetworkStake()`: [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56) calls `networkStakeRepository.findLatest()` unconditionally on every invocation: [2](#0-1) 

`NetworkStakeRepository.findLatest()` (lines 12–19) executes a native SQL query with a correlated subquery (`SELECT max(consensus_timestamp) FROM network_stake`) against the live database on every call: [3](#0-2) 

**Root cause:** No `@Cacheable` annotation, no Caffeine/Redis cache, and no rate-limiting interceptor (Bucket4j, Resilience4j, etc.) exist anywhere in the `rest-java` service layer for this endpoint. The only registered filter is `ShallowEtagHeaderFilter`, which only adds ETag headers and does not prevent repeated database round-trips: [4](#0-3) 

`WebMvcConfiguration` adds only argument resolvers and formatters — no rate-limiting interceptors: [5](#0-4) 

A search across all `rest-java` configuration and service classes confirms zero occurrences of `@Cacheable`, `CacheManager`, `caffeine`, `redis`, `RateLimiter`, `Bucket4j`, or `resilience4j`.

**Exploit flow:**
1. Attacker (no credentials required) opens many concurrent HTTP clients.
2. Each client hammers `GET /api/v1/network/stake` in a tight loop.
3. Each request acquires a HikariCP connection and executes the two-phase SQL query.
4. With Spring Boot's default pool size (~10 connections), a modest flood exhausts the pool.
5. All other in-flight queries (account lookups, transaction queries, etc.) queue behind the pool semaphore or receive `Connection pool timeout` errors.
6. The REST API becomes unresponsive service-wide.

**Why existing checks fail:** The `ShallowEtagHeaderFilter` only computes a response hash for conditional GET support — it does not short-circuit the service call or the database query. The ETag value changes only when the underlying data changes (once per day), but the filter still calls through to the controller and the DB on every cache miss or unconditional request.

### Impact Explanation
The `network_stake` table is updated at most once per day (driven by `NodeStakeUpdateTransactionHandler`). Serving this endpoint from a short-lived in-memory cache would eliminate all database load. Without it, every request hits the database. Exhausting the HikariCP connection pool causes all other REST endpoints — including those serving transaction, account, and token data — to fail or time out, resulting in full service degradation of the mirror node REST API. The mirror node is a critical read path for wallets, explorers, and dApps.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero cost beyond a single machine running `wrk`, `ab`, or a simple script. The endpoint is publicly documented in the OpenAPI spec: [6](#0-5) 
The k6 load-test script for this endpoint already exists in the repository, demonstrating the attack pattern is trivially reproducible: [7](#0-6) 
Any attacker can adapt it. The attack is repeatable indefinitely and requires no persistence.

### Recommendation
1. **Add application-level caching** on `NetworkServiceImpl.getLatestNetworkStake()` using `@Cacheable` with a TTL of 60–300 seconds (the data changes once per day). A Caffeine cache with `maximumSize(1).expireAfterWrite(60, SECONDS)` is sufficient.
2. **Add rate limiting** on the `/api/v1/network/stake` endpoint (and all public endpoints) using Bucket4j or Spring's `resilience4j-ratelimiter`, keyed by client IP.
3. **Tune the HikariCP pool** with a `connectionTimeout` and `maximumPoolSize` appropriate for the expected load, and configure a query timeout on the `findLatest()` query to bound worst-case connection hold time.

### Proof of Concept
```bash
# Requires wrk (https://github.com/wg/wrk)
# Flood the endpoint with 100 concurrent connections for 30 seconds
wrk -t 10 -c 100 -d 30s http://<mirror-node-host>/api/v1/network/stake

# Simultaneously observe that other endpoints begin timing out:
curl -v http://<mirror-node-host>/api/v1/transactions?limit=1
# Expected: connection pool timeout / 500 / hung response
```
