### Title
Unbounded Polling Loop DoS via Non-Existent Topic Subscription When `checkTopicExists=false`

### Summary
When `hiero.mirror.grpc.checkTopicExists` is set to `false`, any unprivileged user can subscribe to an arbitrary number of non-existent topic IDs. Each subscription spawns an infinite polling loop in `PollingTopicListener.listen()` that issues a database query every 500 ms indefinitely. With enough concurrent subscriptions, the HikariCP database connection pool is exhausted, causing legitimate subscribers to time out or fail.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicExists()` → `incomingMessages()` → `PollingTopicListener.listen()`

**Root cause — `topicExists()` bypass:** [1](#0-0) 

When `checkTopicExists=false`, a non-existent topic ID causes `entityRepository.findById()` to return empty, but instead of erroring, the code synthesises a fake `EntityType.TOPIC` entity and allows the subscription to proceed. The subscription is never rejected.

**Root cause — infinite polling loop:** [2](#0-1) 

`RepeatSpec.times(Long.MAX_VALUE)` (≈ 9.2 × 10¹⁸ iterations) with a fixed 500 ms delay means the loop is effectively infinite. Each iteration calls: [3](#0-2) 

`topicMessageRepository.findByFilter(newFilter)` — a real database query — which always returns zero rows for a non-existent topic, yet the loop never terminates.

**No global subscriber cap:** `subscriberCount` is a metric gauge only; there is no enforced ceiling on concurrent subscriptions. [4](#0-3) 

**Per-connection limit is insufficient:** `maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, but an attacker opens many connections. [5](#0-4) 

### Impact Explanation
Each attacker-controlled subscription issues 2 DB queries per second indefinitely. With a typical HikariCP pool of 10–20 connections and queries that take even a few milliseconds, a few hundred concurrent ghost subscriptions saturate the pool. The Helm chart already monitors for this exact condition (pool utilisation > 75% triggers a critical alert). [6](#0-5) 

Legitimate subscribers receive connection-acquisition timeouts or errors. While sequence-number ordering within a topic is maintained in the database, subscribers experience stalls and missed delivery windows, effectively breaking the real-time delivery guarantee.

### Likelihood Explanation
**Preconditions (both non-default):**
- `hiero.mirror.grpc.checkTopicExists=false` — must be explicitly set
- `hiero.mirror.grpc.listener.type=POLL` — default is `REDIS` [7](#0-6) [8](#0-7) 

Both settings are documented and operators do configure them (e.g., for environments without Redis). Once set, any unauthenticated gRPC client can trigger the attack. The gRPC `subscribeTopic` endpoint requires no credentials. The attack is trivially scriptable and repeatable.

### Recommendation
1. **Enforce a maximum concurrent subscriber limit** in `TopicMessageServiceImpl.subscribeTopic()` — reject new subscriptions when `subscriberCount` exceeds a configurable threshold.
2. **Add a no-data timeout** to `PollingTopicListener.listen()`: if zero messages are returned for N consecutive polls (configurable), terminate the flux with an appropriate error rather than repeating `Long.MAX_VALUE` times.
3. **Rate-limit subscriptions per source IP** at the Netty/gRPC interceptor layer, analogous to the `bucket4j` throttle already used in the web3 module. [9](#0-8) 

4. Consider making `checkTopicExists=true` immutable or adding a warning when it is disabled alongside `type=POLL`.

### Proof of Concept
**Preconditions:** server configured with `checkTopicExists=false` and `listener.type=POLL`.

```python
import grpc, threading
from com.hedera.hashgraph.sdk.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hedera.hashgraph.sdk.proto import basic_types_pb2

channel = grpc.insecure_channel("mirror-node:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

def ghost_subscribe(topic_num):
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_num)
    )
    # Consume the stream (blocks until server closes or error)
    for _ in stub.subscribeTopic(req):
        pass

# Open 500 subscriptions to non-existent topics across many threads
threads = [threading.Thread(target=ghost_subscribe, args=(9_000_000 + i,))
           for i in range(500)]
for t in threads: t.start()
# Each subscription now polls the DB every 500 ms indefinitely.
# Monitor: hikaricp_connections_active / hikaricp_connections_max → approaches 1.0
# Legitimate subscribers begin timing out on DB connection acquisition.
```

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
