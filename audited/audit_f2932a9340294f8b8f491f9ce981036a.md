### Title
Silent `BlockType.LATEST` Fallback in Opcode Replay Produces Incorrect Historical Execution Environment

### Summary
In `OpcodeServiceImpl.buildCallServiceParameters()`, when `recordFileService.findByTimestamp(consensusTimestamp)` returns empty, the code silently falls back to `BlockType.LATEST` instead of failing. Because `ContractCallContext.useHistorical()` gates all historical state reads on `block != BlockType.LATEST`, this fallback causes the entire EVM replay to execute against the current (latest) chain state rather than the state at the time of the original transaction, producing a fabricated opcode trace.

### Finding Description

**Exact code location:**

`OpcodeServiceImpl.java` lines 158–161: [1](#0-0) 

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);   // ← silent fallback
```

**Root cause — the `useHistorical()` gate:**

`ContractCallContext.java` lines 103–116: [2](#0-1) 

```java
public boolean useHistorical() {
    return callServiceParameters != null && callServiceParameters.getBlock() != BlockType.LATEST;
}

public Optional<Long> getTimestamp() {
    if (useHistorical()) {
        return getTimestampOrDefaultFromRecordFile();
    }
    return Optional.empty();   // ← returned when block == LATEST
}
```

Every state-reading KV-state (contract storage, entities, balances, etc.) branches on `ctx.getTimestamp()`:
- non-empty → historical query (e.g. `findStorageByBlockTimestamp`)
- empty → latest query (e.g. `findStorage`)

`ContractDebugService.processOpcodeCall()` does call `ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1))` before invoking `callContract()`: [3](#0-2) 

However, `callContract()` immediately overwrites `callServiceParameters` with the `BlockType.LATEST` params: [4](#0-3) 

After that, `useHistorical()` returns `false`, so `getTimestamp()` returns `Optional.empty()` regardless of the previously set timestamp field. The explicitly set `consensusTimestamp - 1` is silently discarded.

**When does `findByTimestamp()` return empty?**

The repository query is: [5](#0-4) 

```sql
select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1
```

This returns empty when no record file has `consensusEnd >= timestamp`. This occurs when:
1. The record file containing the transaction has been pruned AND it was the most recent record file at that time (i.e., no later record files exist in the DB).
2. The `record_file` table has been fully pruned while `contract_result` rows survive (different retention policies).

The only guard before reaching this code is `contractResultRepository.findById(consensusTimestamp)`: [6](#0-5) 

This confirms the contract result exists, but says nothing about whether the corresponding record file still exists.

### Impact Explanation

When the fallback fires, the opcode trace returned to the caller reflects the **current** EVM state (latest storage slots, latest account balances, latest bytecode) rather than the state at the time of the original transaction. A contract whose storage or code changed after the original transaction will produce a completely different execution path, different SLOAD/SSTORE values, and potentially a different success/failure outcome. This corrupts the forensic value of the `debug_traceTransaction`-equivalent endpoint and can be used to produce a trace that misrepresents what actually happened on-chain — directly matching the "reorganizing transaction history" threat model described in the question.

### Likelihood Explanation

An unprivileged user needs only:
1. A valid transaction hash or ID for a transaction whose record file has been pruned (but whose `contract_result` row still exists — possible when retention policies differ between tables or when the importer prunes `record_file` rows more aggressively).
2. A standard HTTP GET to `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` with `Accept-Encoding: gzip`.

No elevated privileges, no special tokens, no on-chain interaction required. The condition is uncommon in default deployments but is realistic in long-running mirror nodes with storage-saving pruning enabled.

### Recommendation

Replace the silent `.orElse(BlockType.LATEST)` with an explicit error:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElseThrow(() -> new EntityNotFoundException(
                "Record file not found for consensus timestamp: " + consensusTimestamp));
```

This ensures that if the record file has been pruned, the endpoint returns a clear 404 rather than silently replaying against the wrong state. Additionally, `ContractDebugService.processOpcodeCall()` should assert `useHistorical()` is true after setting parameters, as a defense-in-depth check.

### Proof of Concept

1. Deploy a contract and invoke it, recording the transaction hash `TX_HASH` and `consensusTimestamp`.
2. Modify the contract's storage (e.g., call a setter) so the current state differs from the state at `TX_HASH`.
3. Delete (or simulate pruning of) the `record_file` row whose `consensus_end` covers `consensusTimestamp`, while leaving the `contract_result` row intact.
4. Call:
   ```
   GET /api/v1/contracts/results/TX_HASH/opcodes
   Accept-Encoding: gzip
   ```
5. Observe that the returned opcode trace shows SLOAD values from the **current** storage (post-setter call) rather than the storage at the time of `TX_HASH`, confirming the replay used the latest state.
