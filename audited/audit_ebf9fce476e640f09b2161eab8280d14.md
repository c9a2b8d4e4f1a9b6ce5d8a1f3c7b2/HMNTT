### Title
Unbounded TCP Connection Acceptance Exhausts Shared `applicationTaskExecutor` via Infinite gRPC Streaming Subscriptions

### Summary
`GrpcConfiguration.grpcServerConfigurer()` configures the Netty gRPC server with only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`) but imposes no limit on the total number of accepted TCP connections. Because all gRPC call dispatch is routed through the shared Spring Boot `applicationTaskExecutor`, an unauthenticated attacker who opens many connections and saturates each with long-lived streaming subscriptions can occupy all executor threads (or fill its queue), starving legitimate callers and internal processing tasks that share the same pool.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:
```java
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);                                    // shared pool
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // per-conn only
    };
}
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `NettyServerBuilder` is never called with `maxConnections(...)`. The only guard is `maxConcurrentCallsPerConnection = 5`, which is a *per-connection* ceiling. There is no ceiling on the number of simultaneous TCP connections, so the total concurrent call budget is `5 × N` where N is unbounded. All of those calls are dispatched to the single shared `applicationTaskExecutor` (Spring Boot's `ThreadPoolTaskExecutor`, default core-size = 8, queue-capacity = `Integer.MAX_VALUE`). Long-lived server-streaming RPCs (topic subscriptions) hold executor resources for their entire lifetime. Once the 8 core threads are occupied by attacker-controlled streams, all subsequent tasks — including legitimate subscriber calls and any internal scheduled work sharing the same executor — are queued indefinitely.

**Why existing checks fail:** `maxConcurrentCallsPerConnection` only limits calls *within* a single connection. An attacker using 2 connections already saturates the 8 default core threads (2 × 5 = 10 > 8). There is no IP-based rate limit, no authentication gate, no `maxConnectionAge`, and no subscription timeout for live (non-historical) streams (`retriever.timeout = 60s` applies only to historical retrieval).

### Impact Explanation

The gRPC mirror node is the sole interface for Hedera topic subscription consumers. Saturating its executor causes:
- All new `subscribeTopic` and `getNodes` calls to queue indefinitely, appearing as hangs to clients.
- Internal periodic tasks sharing `applicationTaskExecutor` (e.g., scheduled polling) to stall.
- No self-recovery: attacker streams persist until the attacker disconnects or the JVM is restarted.

Severity: **High** — complete denial of the gRPC API surface with no authentication barrier.

### Likelihood Explanation

Preconditions: network access to port 5600 (publicly exposed per Helm chart ingress configuration). No credentials, tokens, or special protocol knowledge required beyond standard gRPC client libraries. The attack is trivially scriptable, repeatable, and requires minimal bandwidth (streaming subscriptions send no data after the initial request). A single attacker machine with 2 open connections is sufficient to saturate the default 8-thread core pool.

### Recommendation

1. **Add a total connection limit** to the `NettyServerBuilder` in `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnectionAge(Duration.ofMinutes(10), TimeUnit.MILLISECONDS);
   serverBuilder.maxConnectionIdle(Duration.ofMinutes(5), TimeUnit.MILLISECONDS);
   ```
   And expose `maxConnections` in `NettyProperties` with a safe default (e.g., 200).

2. **Add `maxConnections` to `NettyProperties`** and call `serverBuilder.maxConnections(nettyProperties.getMaxConnections())`.

3. **Bound the `applicationTaskExecutor`** via `spring.task.execution.pool.max-size` and `spring.task.execution.pool.queue-capacity` to a finite value so overload is rejected rather than queued forever.

4. **Isolate the gRPC executor** from other Spring-managed async tasks by defining a dedicated `Executor` bean for gRPC rather than reusing `applicationTaskExecutor`.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

def open_saturating_connection():
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    # Open 5 concurrent streaming calls per connection (maxConcurrentCallsPerConnection limit)
    for _ in range(5):
        req = consensus_service_pb2.ConsensusTopicQuery(topic_id=..., consensus_start_time=0)
        threading.Thread(target=lambda: list(stub.subscribeTopic(req))).start()

# 2 connections × 5 calls = 10 tasks → saturates 8 core threads, remainder queues
for _ in range(2):
    threading.Thread(target=open_saturating_connection).start()

# All subsequent legitimate subscribeTopic calls now queue indefinitely
```

After running this, new gRPC clients connecting to the mirror node will receive no response until the attacker disconnects.
