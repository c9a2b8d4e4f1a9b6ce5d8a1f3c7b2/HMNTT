### Title
Permanent Processing Stall via Malformed Signature File Injection into Cloud Storage Bucket

### Summary
An attacker with write access to the cloud storage bucket (e.g., a misconfigured or publicly-writable S3 bucket) can inject malformed signature files for a target timestamp, causing `ConsensusValidatorImpl.validate()` to throw `SignatureVerificationException` on every poll cycle. Because `Downloader.downloadNextBatch()` catches and swallows this exception without advancing the `lastStreamFile` cursor, the downloader permanently re-attempts the same failing files on every scheduled tick, halting all record stream processing indefinitely.

### Finding Description

**Code path:**

`RecordFileDownloader.download()` is a `@Scheduled` method that calls `downloadNextBatch()` on every tick: [1](#0-0) 

Inside `downloadNextBatch()`, a `SignatureVerificationException` is caught and only logged — no cursor advancement, no skip, no circuit-breaker: [2](#0-1) 

The `lastStreamFile` cursor is **only** updated inside `onVerified()`, which is only reached on successful verification: [3](#0-2) 

On the next scheduled tick, `getStartAfterFilename()` returns the same unchanged cursor value, causing the downloader to re-fetch the same malformed files: [4](#0-3) 

In `verifySigsAndDownloadDataFiles()`, when the last (or only) signature group fails, the exception is re-thrown up to `downloadNextBatch()`: [5](#0-4) 

`ConsensusValidatorImpl.validate()` throws `SignatureVerificationException` when no injected signature reaches the 1/3 stake threshold, because `NodeSignatureVerifier.verifySignature()` returns `false` for all malformed files (cryptographic verification fails), leaving `signatureHashMap` empty: [6](#0-5) 

**Root cause:** The failed assumption is that all files present in the bucket are legitimate. There is no mechanism to skip past a permanently-failing signature file — the cursor only advances on success, and the exception is silently swallowed on failure.

### Impact Explanation
All record stream processing halts permanently at the injected timestamp. No subsequent record files are downloaded or parsed. The mirror node stops reflecting new network state, breaking all downstream consumers (REST API, GRPC, etc.). This matches the stated scope: shutdown of ≥30% of network processing nodes without brute force, as a single injected file per stream type is sufficient to stall the entire importer.

### Likelihood Explanation
The precondition is write access to the cloud storage bucket. This is realistic in:
- Self-hosted mirror node deployments where operators configure their own S3/GCS buckets with overly permissive ACLs
- Deployments using a public bucket with write permissions inadvertently enabled
- Any environment where the bucket policy has not been hardened

The attack is trivially repeatable: the attacker uploads one malformed `.rcd_sig` file per node folder for the target timestamp. No cryptographic material is needed — any byte sequence that fails RSA signature verification suffices. The stall is permanent until an operator manually intervenes.

### Recommendation
1. **Advance the cursor on repeated failure**: After N consecutive `SignatureVerificationException`s for the same filename, log an error and advance `lastStreamFile` past the offending file to unblock processing.
2. **Failure counter per filename**: Track per-filename failure counts in `verifySigsAndDownloadDataFiles()`; after a configurable threshold, skip the file and emit an alert metric.
3. **Bucket write-access hardening**: Enforce read-only IAM/bucket policies for the mirror node's cloud storage credentials; document this as a required security control.
4. **Alerting**: Emit a distinct metric/alert when the same filename fails repeatedly, enabling operators to detect and respond to injection attempts.

### Proof of Concept

**Preconditions:**
- Attacker has write access to the S3/GCS bucket used by the mirror node importer
- Mirror node is running and has successfully processed files up to timestamp `T-1`

**Steps:**
1. Identify the next expected record file timestamp `T` (observable from the bucket's existing file listing)
2. For each node folder (e.g., `recordstreams/record0.0.3/`, `recordstreams/record0.0.4/`, ...), upload a file named `<T>.rcd_sig` containing random bytes (e.g., `dd if=/dev/urandom bs=256 count=1 > malformed.rcd_sig`)
3. Observe the importer logs: every `fixedDelay` interval, the importer logs `WARN: Consensus not reached for file <T>.rcd_sig with 0/<totalStake> stake`
4. Confirm that `lastStreamFile` never advances past `T-1` — no new record files are processed
5. The stall persists indefinitely until the malformed files are removed from the bucket or an operator manually resets the importer state

            return;
        }

            // Following is a cost optimization to not unnecessarily list the public demo bucket once complete

                log.warn("Disabled polling after downloading all files in demo bucket");
            }

            // Verify signature files and download corresponding files of valid signature files
            verifySigsAndDownloadDataFiles(sigFilesMap);
        } catch (SignatureVerificationException e) {
            log.warn(e.getMessage());
        } catch (Exception e) {

        }
    }
