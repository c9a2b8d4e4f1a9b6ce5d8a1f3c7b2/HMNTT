### Title
Unbounded Concurrent Goroutine Accumulation via Unauthenticated `/health/readiness` Endpoint with Self-Referential 10-Second Timeout Checks

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` executes two health checks per request — a PostgreSQL check and a self-referential network check — each with a 10-second timeout and no concurrency cap. Because the Traefik `inFlightReq`/`rateLimit` middleware is scoped exclusively to `/rosetta/*` ingress paths and never covers `/health/readiness`, any caller with network access to the pod can flood the endpoint, causing unbounded goroutine accumulation, amplified internal HTTP traffic, and memory pressure that exceeds the 30% threshold without triggering brute-force detection.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/health.go`, `NewHealthController()` lines 37–50 registers two checks, each with `Timeout: time.Second * 10`. The `checkNetworkStatus()` closure (lines 80–101) issues two outbound HTTP calls back to `http://localhost:{port}` — `NetworkList` (line 86) and `NetworkStatus` (line 93) — using the caller-supplied context, which carries that 10-second deadline.

The `hellofresh/health-go/v4` library (`health.HandlerFunc`) executes all registered checks on **every** incoming request with no result caching and no in-flight deduplication. Each request spawns at least two goroutines (one per check), each of which may block for up to 10 seconds.

**Root cause / failed assumption:**
The design assumes the health endpoint is only called by the Kubernetes kubelet at a controlled rate (every 10 s per `readinessProbe`). No application-level semaphore, rate limiter, or concurrency gate is placed on the handler.

**Why existing checks fail:**
The Traefik middleware block in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 is attached to the ingress, whose `paths` list (lines 125–133) enumerates only `/rosetta/account`, `/rosetta/block`, `/rosetta/call`, `/rosetta/construction`, `/rosetta/events`, `/rosetta/mempool`, `/rosetta/network`, `/rosetta/search`. The path `/health/readiness` is absent. The `inFlightReq: amount: 5` and `rateLimit: average: 10` rules therefore never fire for health requests. The Go application itself (`rosetta/main.go` lines 217–227) applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which throttle requests.

**Exploit flow:**
1. Attacker obtains network access to the pod on `rosettaConfig.Port` (5700 per gateway config line 64; ClusterIP service, so reachable from any pod in the cluster or via a misconfigured network policy).
2. Attacker sends a continuous stream of `GET /health/readiness` requests, e.g. 50 concurrent connections sustained indefinitely.
3. Each request causes `health.HandlerFunc` to launch goroutines for the `postgresql` check and the `network` check simultaneously.
4. The `network` check calls `rosettaClient.NetworkAPI.NetworkList` and then `NetworkAPI.NetworkStatus` on localhost — two additional inbound requests to the same server, each consuming a goroutine and an HTTP connection from the shared `rosettaClient` pool.
5. If the upstream DB or network is slow, each goroutine blocks for up to 10 seconds. At 50 req/s, this yields ≥500 live goroutines and ≥1000 pending internal HTTP requests at steady state.
6. Go goroutine stacks start at 2–8 KB and grow; combined with HTTP response buffers and the shared `rosettaClient` state, heap pressure climbs rapidly against the 200 Mi memory limit (`charts/hedera-mirror-rosetta/values.yaml` line 249).

### Impact Explanation
- **Goroutine exhaustion / OOM**: The container memory limit is 200 Mi. Sustained goroutine accumulation with 10-second hold times can exhaust heap, triggering OOM kills and pod restarts, making the node unavailable.
- **CPU amplification**: Each health request generates 2 internal HTTP requests to the same process, doubling CPU work for JSON serialisation, routing, and DB queries.
- **Self-reinforcing degradation**: As the server slows under load, the 10-second timeouts are more likely to be reached, keeping goroutines alive longer, worsening the spiral.
- **No brute-force signal**: The requests are individually valid, low-bandwidth GET calls with no authentication requirement; they produce no 4xx/5xx responses under normal conditions and will not trigger `NetworkErrorRatio` or `ResponseCodeRatio` circuit-breaker rules.

### Likelihood Explanation
Precondition is network access to the pod port — achievable from any co-tenant pod in the same namespace or cluster with default Kubernetes network policies (which allow all intra-cluster traffic). No credentials, tokens, or special headers are required. The attack is trivially scriptable (`ab -n 100000 -c 100 http://<pod-ip>:5700/health/readiness`) and fully repeatable. The 30% resource threshold is reachable with modest concurrency (~20–50 parallel connections) given the 10-second check duration and 200 Mi memory cap.

### Recommendation
1. **Add a concurrency gate in the handler**: Wrap `readinessHealth.HandlerFunc` with a semaphore (e.g., `golang.org/x/sync/semaphore`) limiting simultaneous in-flight readiness checks to a small number (e.g., 3).
2. **Cache check results**: Use `hellofresh/health-go/v4`'s `health.WithMaxConcurrent` option (if available in the version in use) or implement a short-lived result cache (e.g., 5 s TTL) so concurrent requests share a single check execution.
3. **Extend Traefik middleware scope**: Add `/health/readiness` to the ingress paths list in `charts/hedera-mirror-rosetta/values.yaml` so the existing `inFlightReq` and `rateLimit` rules apply to it.
4. **Apply a Kubernetes NetworkPolicy**: Restrict access to the pod port to only the kubelet CIDR and trusted internal services, preventing arbitrary cluster pods from reaching the health endpoint.
5. **Reduce check timeouts**: 10 seconds is excessive for a liveness/readiness probe; 2–3 seconds aligns with the `timeoutSeconds: 2` already set on the Kubernetes probe itself.

### Proof of Concept

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
	serverUrl := fmt.Sprintf("http://localhost:%d", port)
	cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
	rosettaClient := client.NewAPIClient(cfg)

	return func(ctx context.Context) (checkErr error) {
		networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
		if err != nil {
			log.Errorf("Readiness check, /network/list failed: %v", err)
			return err
		}

		network := networkList.NetworkIdentifiers[0]
		_, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{NetworkIdentifier: network})
		if err != nil {
			log.Errorf("Readiness check, /network/status failed: %v", err)
			return err
		}

		return
	}
}
```
