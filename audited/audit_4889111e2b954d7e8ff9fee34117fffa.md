### Title
Unauthenticated DoS via Uncached Aggregation Query on `/api/v1/network/supply`

### Summary
The `getSupply()` method in `EntityRepository` executes a full `unnest+join` aggregation query against the `entity` table on every single HTTP request to `/api/v1/network/supply`. No response caching exists anywhere in the rest-java module, and no rate limiting is applied to this endpoint. An unprivileged attacker sending concurrent requests from multiple source IPs can exhaust the database connection pool, degrading or halting mirror node database processing.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (line 133–150) calls `networkService.getSupply(bound)` on every request with no caching: [1](#0-0) 

`NetworkServiceImpl.getSupply()` (line 59–89) calls `entityRepository.getSupply(lowerBounds, upperBounds)` directly with no `@Cacheable` or memoization: [2](#0-1) 

`EntityRepository.getSupply()` executes a native SQL aggregation — `SUM`, `MAX`, `unnest`, and a range `JOIN` over the entire `entity` table — on every invocation: [3](#0-2) 

**Root cause:** No `@Cacheable` annotation, no `CacheManager`, and no `@EnableCaching` exists anywhere in the rest-java module. A search across all rest-java Java sources confirms zero cache annotations on any supply-related code path. The endpoint is a plain unauthenticated `@GetMapping` with no throttle guard. [4](#0-3) 

**Why checks fail:** The `HIGH_VOLUME_THROTTLE` constant referenced in `NetworkController` is wired only to the fee-estimation `POST /network/fees` endpoint (line 116), not to `GET /network/supply`. No per-IP or global rate limiter (bucket4j, resilience4j, etc.) is present in the rest-java module. [5](#0-4) 

### Impact Explanation
Each concurrent request to `/api/v1/network/supply` holds a JDBC connection from the HikariCP pool (Spring Boot default: 10 connections) for the duration of the aggregation query. On a large `entity` table, this query is non-trivial. Flooding with concurrent requests from multiple IPs saturates the pool, causing all other mirror node database operations (ingestion, other API endpoints) to queue or time out. This directly maps to the stated scope: shutdown of ≥30% of network processing nodes without brute force, since the mirror node's DB layer becomes unavailable to legitimate consumers.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero privileged access — only the ability to send HTTP GET requests. Distributing requests across multiple source IPs trivially bypasses any hypothetical per-IP network-layer rate limiting (e.g., nginx `limit_req_zone`). The endpoint is publicly documented as part of the Hedera Mirror Node REST API. Any script kiddie with a botnet or even a single machine with multiple egress IPs (cloud NAT, VPN rotation) can execute this.

### Recommendation
1. **Add response caching** on `NetworkServiceImpl.getSupply()` with a short TTL (e.g., 5–15 seconds) using Spring's `@Cacheable` with a dedicated cache name, so repeated requests within the TTL window skip the DB query entirely.
2. **Apply endpoint-level rate limiting** on `GET /api/v1/network/supply` using a token-bucket limiter (e.g., Resilience4j `RateLimiter` or Bucket4j), independent of source IP, to cap total QPS hitting the DB.
3. **Set an explicit HikariCP `connectionTimeout`** and a low `maximumPoolSize` for read-only supply queries, or route supply queries to a read replica with its own pool, to isolate blast radius.

### Proof of Concept
```bash
# Send 200 concurrent requests from a single machine (simulating multi-IP via proxies/VPN)
seq 1 200 | xargs -P 200 -I{} curl -s \
  "https://<mirror-node-host>/api/v1/network/supply" \
  -o /dev/null -w "%{http_code}\n"
```
Expected result: DB connection pool exhausted within seconds; subsequent requests to any mirror node REST endpoint return 500 or hang until pool timeout. Monitor with `SELECT count(*) FROM pg_stat_activity WHERE state = 'active'` on the mirror node DB — active connections will spike to pool maximum and stay there.
