### Title
Alias-Based Path Parameter Causes Uncached Double DB Query Per Request, Enabling DB Pool Exhaustion DoS

### Summary
Every call to `getAccountCryptoAllowances` with an alias path parameter unconditionally executes two sequential database queries — one to resolve the alias to an account ID and one to fetch allowances — with no caching, coalescing, or rate limiting at the controller or service layer. An unprivileged attacker sending N concurrent alias-based requests causes 2N database queries, which is double the load of equivalent numeric-ID requests, and can saturate the small default DB connection pool (10 connections), queuing or timing out legitimate traffic.

### Finding Description

**Exact code path:**

In `rest/controllers/cryptoAllowanceController.js`, `getAccountCryptoAllowances` (lines 76–98):

```js
const accountId = await EntityService.getEncodedId(req.params[...]);  // line 77 — DB query #1 for alias
const allowances = await CryptoAllowanceService.getAccountCryptoAllowances(...);  // line 80 — DB query #2
```

`EntityService.getEncodedId` in `rest/service/entityService.js` (lines 118–137) branches on input type. For a numeric ID it returns immediately with no DB call. For an alias it calls `getAccountIdFromAlias` → `getAccountFromAlias` → `super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias])` (line 43), which is a live database query every time with no caching.

**Root cause:** There is no in-process or Redis-backed cache for alias→accountId resolution in the JS REST service. The configuration keys `hiero.mirror.rest.cache.entityId.*` appear in documentation but are not implemented in `rest/service/entityService.js`. The response cache (`responseCacheHandler.js`) uses `req.originalUrl` as the cache key (line 152), so concurrent requests arriving before the first completes all miss the cache and each independently execute both queries. The response cache is also disabled by default (`config.cache.response.enabled && config.redis.enabled`, `server.js` line 54).

**No rate limiting:** `rest/server.js` and `rest/middleware/index.js` confirm the REST API has no per-IP or global rate-limiting middleware. The throttle configuration found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the web3 module, not the Node.js REST API.

**DB pool constraint:** `rest/dbpool.js` (line 14) sets `max: config.db.pool.maxConnections`, which defaults to 10 connections. With 10 connections and 2 sequential queries per alias request, an attacker sustaining 10+ concurrent alias requests keeps the pool fully occupied, forcing all other requests to queue behind a `connectionTimeoutMillis` of 20 seconds.

### Impact Explanation

An attacker can continuously saturate the 10-connection DB pool using alias-based requests, causing legitimate requests to queue and eventually time out with connection errors. The alias path doubles the DB cost per request compared to numeric-ID requests, meaning the attacker achieves the same pool saturation with half the request rate. In a default deployment (response cache disabled, no rate limiting), there is no server-side mitigation. All API endpoints sharing the same pool — transactions, balances, tokens — are collaterally degraded.

### Likelihood Explanation

The endpoint `/api/v1/accounts/{idOrAliasOrEvmAddress}/allowances/crypto` is public and unauthenticated. Any external user can supply a valid alias (e.g., a base32-encoded public key visible on-chain). Sending 50–100 concurrent HTTP requests per second from a single machine is trivial and sufficient to keep the 10-connection pool saturated indefinitely. The attack is repeatable, requires no credentials, and leaves no exploitable state — it is purely a resource exhaustion pattern.

### Recommendation

1. **Cache alias→accountId resolution** in `EntityService.getEncodedId`: implement an in-process LRU cache (or use the existing Redis layer) keyed on the alias string, with a TTL matching `hiero.mirror.rest.cache.entityId.maxAge` (default 1800s). Aliases are immutable once assigned, so stale reads are not a concern.
2. **Enable the response cache by default** or document clearly that it must be enabled in production to prevent thundering-herd cache misses.
3. **Add per-IP rate limiting** to the REST API middleware stack (e.g., `express-rate-limit`) to bound the request rate from any single source.
4. **Increase the DB pool size** or add a concurrency limiter (e.g., `p-limit`) in the service layer to prevent a single request type from monopolizing all connections.

### Proof of Concept

```bash
# Precondition: a valid alias for an existing account, e.g. KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ
# (visible in test fixtures: rest/__tests__/specs/accounts/{id}/allowances/crypto/no-params.json)

ALIAS="KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ"
BASE="http://<mirror-node-host>:5551/api/v1"

# Send 200 concurrent requests — each causes 2 DB queries
for i in $(seq 1 200); do
  curl -s "$BASE/accounts/$ALIAS/allowances/crypto" &
done
wait

# Observe: DB pool (10 connections) is saturated; legitimate requests to
# /api/v1/transactions, /api/v1/balances, etc. begin timing out with
# "connection timeout" errors after 20 seconds (default statementTimeout).
```
