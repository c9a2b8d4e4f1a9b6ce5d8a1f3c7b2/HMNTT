### Title
Gas Throttle Fully Bypassed for Failing Transactions via 100% Default Refund — Sustained Max-Gas EVM Execution

### Summary
The `gasLimitBucket` in `ThrottleConfiguration.gasLimitBucket()` is designed to cap total gas processed per second, but the default `gasLimitRefundPercent = 100` causes `restoreGasToBucket()` to return every token consumed by a failing transaction (where `gasUsed == gasLimit`). An unprivileged attacker can craft transactions that always cause an exceptional halt (OOG), consuming zero net tokens from the gas bucket while forcing full max-gas EVM execution on every request, sustained indefinitely at the rate limit.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.java` lines 59–63 — `restore()` calls `gasLimitBucket.addTokens(scaleGas(gas))`: [3](#0-2) 

`ThrottleManagerImpl.java` line 40 — `throttle()` consumes `scaleGas(request.getGas())` tokens: [4](#0-3) 

**Root cause and failed assumption:**

The design assumes that failed transactions (exceptional halt) should be refunded because they "didn't really use" the gas. With `gasLimitRefundPercent = 100` (the default and maximum allowed value), the tokens consumed by `throttle()` are exactly equal to the tokens restored by `restore()`:

- Consumed: `scaleGas(gasLimit)` tokens
- Restored: `scaleGas(gasLimit * 100 / 100)` = `scaleGas(gasLimit)` tokens
- **Net: 0 tokens consumed from the gas bucket**

The EVM still executes the full max-gas transaction, but the gas throttle registers no cost. The bucket is never depleted by failing transactions, so the gas throttle never fires for this attack pattern.

**Why existing checks fail:**

1. The `rateLimitBucket` (500 req/s default) is the only remaining guard, but it is a global bucket — not per-IP — so a single attacker can occupy all 500 slots. [5](#0-4) 

2. The `gasLimitBucket` capacity (`gasPerSecond / GAS_SCALE_FACTOR = 750,000` tokens by default) is irrelevant because the attacker never depletes it. [6](#0-5) 

3. The `scaleGas` floor (`gas <= 10,000 → 0`) does not help because max-gas requests are far above 10,000. [7](#0-6) 

### Impact Explanation

With 500 req/s (rate limit) each carrying the maximum EVM gas limit (~15,000,000 gas), the attacker drives 7.5 billion gas/second of EVM execution — exactly the `gasPerSecond` ceiling — while the gas throttle never engages. This saturates the EVM execution thread pool, causing severe latency or OOM on the mirror node's web3 service. Because the gas bucket is never depleted, legitimate requests that do consume tokens are still throttled while the attacker's failing requests are not, creating an asymmetric denial-of-service. This can degrade or halt ≥30% of the node's processing capacity.

### Likelihood Explanation

No privileges are required. The attacker needs only a publicly accessible `eth_call` or `eth_estimateGas` endpoint and a contract (or bytecode) that reliably causes an OOG exceptional halt. This is trivially constructed (e.g., an infinite loop or a call that consumes all forwarded gas). The attack is fully repeatable, stateless, and requires no on-chain funds since mirror node calls are simulated. A single HTTP client can sustain it.

### Recommendation

1. **Lower the default `gasLimitRefundPercent`** to a value significantly below 100 (e.g., 10–25%) so that failing transactions still consume a meaningful portion of the gas budget.
2. **Cap `addTokens` to never exceed the tokens originally consumed** — i.e., `restore` should add at most `min(tokensToRestore, tokensConsumedByThisRequest)`, preventing any net-zero or net-positive effect on the bucket.
3. **Add per-IP / per-caller rate limiting** so a single source cannot monopolize the global `rateLimitBucket`.
4. Consider tracking "gas consumed by failing transactions" separately and applying a stricter sub-limit for that category.

### Proof of Concept

**Preconditions:**
- Mirror node web3 endpoint accessible (no auth required)
- A deployed contract (or raw bytecode) containing an infinite loop, or any call that exhausts all forwarded gas

**Steps:**

```
1. Deploy (or identify) a contract whose fallback/function always OOGs:
   e.g., bytecode: 0x5B600056  (JUMPDEST, PUSH1 0x00, JUMP — infinite loop)

2. In a tight loop (up to 500 req/s), send:
   POST /api/v1/contracts/call
   {
     "to": "<oog_contract>",
     "gas": 15000000,   // maximum gas
     "data": "0x"
   }

3. Observe:
   - Each request: throttle() consumes scaleGas(15_000_000) = 1_500 tokens
   - EVM executes full 15M gas → exceptional halt → gasUsed == gasLimit
   - restoreGasToBucket(result, 15_000_000):
       gasLimitToRestoreBaseline = 15_000_000 * 100 / 100 = 15_000_000
       condition: !isSuccessful() && gasLimit == gasUsed → TRUE
       restore(15_000_000) → addTokens(1_500)
   - Net tokens consumed: 0

4. The gas bucket is never depleted. The attacker sustains 500 × 15M = 7.5B gas/s
   of EVM execution indefinitely. Legitimate requests experience severe degradation.
```
