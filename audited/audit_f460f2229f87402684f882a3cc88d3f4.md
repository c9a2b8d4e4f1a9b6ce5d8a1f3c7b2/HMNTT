### Title
Shared Stopwatch Timeout Budget in `BlockAssembler.timeout()` Allows Malicious Block Node to Abort Mid-Block Assembly via Deliberate Response Delay

### Summary
`BlockAssembler.timeout()` uses a single `Stopwatch` that starts on the first `grpcCall.read()` call and is only reset after a complete block is assembled via `onEndOfBlock()`. A block node operator can deliberately delay the first response to consume nearly the entire timeout budget, then send a `BLOCK_HEADER` item to begin block assembly, leaving zero or negative milliseconds for all subsequent `grpcCall.read()` calls. This causes an immediate `TimeoutException` on the next read, aborting assembly of the in-progress block and discarding all its pending transactions.

### Finding Description

**Exact code path:**

In `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, the `streamBlocks` method drives the read loop:

```java
// Line 138
while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
```

`assembler.timeout()` is defined at lines 274–281:

```java
long timeout() {
    if (!stopwatch.isRunning()) {
        stopwatch.start();          // starts on first call
        return timeout.toMillis();  // returns full budget once
    }
    return timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS); // shrinks every call
}
```

The stopwatch is only reset at line 268, inside `onEndOfBlock()`:

```java
stopwatch.reset();
```

**Root cause:** The timeout budget (`timeout.toMillis()`, default 30 000 ms per `CommonDownloaderProperties` line 83) is a single shared countdown across every `grpcCall.read()` call for the entire block assembly session. There is no per-item or per-response timeout reset. Once the stopwatch has elapsed past `timeout`, `timeout()` returns a negative value. `BlockingClientCall.read(negative, MILLISECONDS)` throws `TimeoutException` immediately (confirmed by the `streamTimeout` test at `BlockNodeTest.java` lines 337–357).

**Exploit flow:**

1. Attacker controls a block node that is in the mirror node's configured node list.
2. Mirror node calls `streamBlocks(blockNumber, ...)`, which calls `grpcCall.read(assembler.timeout(), MILLISECONDS)` — stopwatch starts, full 30 000 ms budget returned.
3. Attacker's block node holds the connection open for 29 999 ms (just under the timeout), then sends a single `BLOCK_ITEMS` response containing only a `BLOCK_HEADER` item.
4. `onBlockItemSet()` is called; `pending` now contains the header. `loadStart` is set. Block assembly has begun.
5. Loop iterates; `assembler.timeout()` now returns `30000 - 29999 = 1 ms` (or negative if any processing overhead consumed the remaining millisecond).
6. `grpcCall.read(≤0, MILLISECONDS)` throws `TimeoutException` immediately.
7. `catch (Exception ex)` at line 162 calls `onError()` and re-throws as `BlockStreamException`.
8. `pending` is never cleared via `onEndOfBlock()`; `blockStreamConsumer` is never called; all transactions in the partially assembled block are silently dropped from this assembly attempt.

**Why existing checks are insufficient:**

- `maxSubscribeAttempts` / `readmitDelay` (lines 196–206): These only deactivate the node after repeated consecutive failures. The attacker can interleave occasional successful blocks to stay below the threshold, or simply accept the cooldown and resume after `readmitDelay`.
- `maxBlockItems` guard (line 293): Unrelated; only limits item count, not timing.
- The `@DurationMin(seconds = 1)` constraint on `timeout` (line 81 of `CommonDownloaderProperties`) guarantees the attacker always has at least 1 000 ms of budget to consume before triggering the condition.
- No gRPC-level deadline is set on the `CallOptions.DEFAULT` used at line 130–133; the only timeout enforcement is the application-level `assembler.timeout()`.

### Impact Explanation

Every block whose assembly is aborted mid-stream causes the mirror node to discard all pending `BlockItem`s for that block. The mirror node must retry the block from another configured node or fall back to cloud storage (`BlockFileSource`). During the retry window, all transactions in the affected block are unavailable to downstream consumers (REST API, gRPC subscribers). An attacker who is the sole or primary configured block node can sustain this indefinitely, causing a persistent denial-of-service against block ingestion. Even with multiple block nodes, the attacker can target the primary node to force repeated failovers and introduce multi-second delays per block.

### Likelihood Explanation

The attacker must control a block node that is included in the mirror node's configuration. In a permissioned deployment this requires operator-level access to the node list; in deployments using dynamic discovery via `BlockNodeDiscoveryService`, the bar may be lower. The attack requires no cryptographic material, no special protocol knowledge beyond the gRPC streaming API, and is trivially repeatable: simply delay the first response to `(timeout - epsilon)` ms, then send a `BLOCK_HEADER`. The attack is deterministic and requires no race condition.

### Recommendation

Replace the shared-countdown design with a per-response deadline reset. Specifically, reset the stopwatch after each successfully received response item, not only after a complete block:

```java
long timeout() {
    stopwatch.reset().start();   // reset on every call
    return timeout.toMillis();
}
```

Or, more precisely, reset the stopwatch inside the read loop after each successful `grpcCall.read()` returns (before dispatching to `onBlockItemSet`/`onEndOfBlock`), so each individual response is given the full timeout budget independently of how long prior responses took. Additionally, consider setting a gRPC-level deadline on `CallOptions` as a defense-in-depth measure.

### Proof of Concept

1. Stand up a malicious gRPC server implementing `BlockStreamSubscribeService`.
2. Configure the mirror node to use this server as its sole block node.
3. In `subscribeBlockStream`, upon receiving a request:
   - Sleep for `(timeout - 1)` ms (e.g., 29 999 ms for the default 30 s timeout).
   - Send one `SubscribeStreamResponse` containing a `BlockItemSet` with a single `BLOCK_HEADER` item.
   - Sleep for 5 ms (to exhaust the remaining budget).
   - Send the remaining block items and `END_OF_BLOCK`.
4. Observe that `BlockNode.streamBlocks()` throws `BlockStreamException` wrapping `TimeoutException` after step 3's second sleep, before any `END_OF_BLOCK` is processed.
