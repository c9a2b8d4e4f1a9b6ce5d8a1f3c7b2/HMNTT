### Title
Accept-Encoding Substring Check Bypass Allows Uncompressed Opcode Replay Responses

### Summary
The `validateAcceptEncodingHeader()` method in `OpcodesController` uses a naive `String.contains("gzip")` check that can be bypassed by sending `Accept-Encoding: gzip;q=0, deflate`. This causes the server to pass validation, perform the expensive EVM opcode replay, and return a large uncompressed response — defeating the compression enforcement that exists specifically to protect server and network resources on this heavy endpoint. The "staking reward data disclosure" framing in the question is incorrect; the actual impact is resource exhaustion / compression bypass, not confidentiality breach.

### Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, line 76

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
        throw HttpClientErrorException.create(...);
    }
}
``` [1](#0-0) 

The check only tests for the substring `"gzip"` anywhere in the header value. Per RFC 7231, `q=0` means the encoding is explicitly **not acceptable**. The value `gzip;q=0, deflate` contains the substring `"gzip"` and therefore passes the check, but it signals to the HTTP stack that gzip is explicitly rejected. Spring's embedded server (Tomcat) correctly honors `q=0` and will not apply gzip compression, returning the full uncompressed opcode trace payload.

The test suite at line 507 only tests `{"deflate", "identity", "br", ""}` as rejection cases — `gzip;q=0` and `gzip;q=0, deflate` are not tested and would not be rejected. [2](#0-1) 

The code comment at line 71–73 explicitly states the gzip requirement exists because "the response from this endpoint is huge and without compression this will result in big network latency," confirming this is a resource-protection control, not a confidentiality control. [3](#0-2) 

**Clarification on "staking reward data" framing:** The gzip requirement is not a confidentiality control. The opcodes endpoint returns the same data whether compressed or not; compression does not hide any values. The "theft of pending_reward values" framing in the question is not a valid impact — those values are equally accessible in a legitimate gzip-compressed response.

### Impact Explanation
An attacker can bypass the compression enforcement gate and force the server to:
1. Perform the full, expensive EVM transaction replay (CPU/DB intensive).
2. Serialize and transmit the full uncompressed opcode trace (potentially megabytes of stack/memory/storage data).

This amplifies the resource cost per request on both server and network. The throttle (`opcodeRequestsPerSecond = 1` by default) still applies, but the bypass means each allowed request now consumes significantly more resources than intended. [4](#0-3) 

### Likelihood Explanation
Any unauthenticated external user can craft this header. No privileges, tokens, or special access are required. The bypass requires only knowledge of HTTP `q`-values (public RFC 7231 knowledge) and a single crafted header. It is trivially repeatable up to the rate limit.

### Recommendation
Replace the substring check with proper RFC 7231 content negotiation using Spring's built-in `ContentNegotiationStrategy` or a library like `Negotiator`. Specifically, parse the `Accept-Encoding` header and verify that `gzip` has a quality factor `q > 0` (or is present without an explicit `q=0`). Example fix:

```java
import org.springframework.http.HttpHeaders;
// Use a proper parser:
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    boolean gzipAccepted = Arrays.stream(acceptEncodingHeader.split(","))
        .map(String::trim)
        .anyMatch(token -> {
            String[] parts = token.split(";");
            if (!parts[0].trim().equalsIgnoreCase("gzip")) return false;
            for (int i = 1; i < parts.length; i++) {
                String[] kv = parts[i].trim().split("=");
                if (kv.length == 2 && kv[0].trim().equals("q")) {
                    return Double.parseDouble(kv[1].trim()) > 0;
                }
            }
            return true; // no q param means q=1.0
        });
    if (!gzipAccepted) {
        throw HttpClientErrorException.create(MISSING_GZIP_HEADER_MESSAGE, ...);
    }
}
```

Also add `gzip;q=0` and `gzip;q=0, deflate` to the `rejectsWhenAcceptEncodingIsNotGzip` test cases.

### Proof of Concept
```
GET /api/v1/contracts/results/{validTxHash}/opcodes?stack=true&memory=true&storage=true HTTP/1.1
Host: <mirror-node-web3>
Accept-Encoding: gzip;q=0, deflate
```

**Expected (intended behavior):** HTTP 406 Not Acceptable — "Accept-Encoding: gzip header is required"

**Actual behavior:** HTTP 200 OK — full uncompressed opcode trace returned (no `Content-Encoding: gzip` header), server has performed the full EVM replay and transmitted the uncompressed payload.

The bypass is confirmed by the logic at line 76: `"gzip;q=0, deflate".toLowerCase().contains("gzip")` evaluates to `true`, so the exception is never thrown. [5](#0-4)
