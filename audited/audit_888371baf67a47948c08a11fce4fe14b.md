### Title
Unbounded Concurrent Historical Subscriptions Exhaust Database Connection Pool via `subscribeTopic()`

### Summary
The `subscribeTopic()` endpoint in `ConsensusController` accepts unauthenticated gRPC calls with no global subscriber cap or rate limit. Each subscription with a `startTime` in the past independently and indefinitely polls the database via `PollingTopicMessageRetriever.retrieve()`, holding HikariCP connections. An attacker opening many connections (each with up to 5 calls) can exhaust the database connection pool, denying service to all legitimate users.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any unauthenticated request and delegates directly to `topicMessageService.subscribeTopic()` with no rate limiting, no IP throttle, and no subscriber count enforcement. [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` (line 63) unconditionally calls `topicMessageRetriever.retrieve(filter, true)` for every subscriber. The `subscriberCount` field (line 48) is a Micrometer gauge for observability only — it is never checked to reject new subscriptions. [2](#0-1) 

`PollingTopicMessageRetriever.retrieve()` (lines 45–63) creates a `PollingContext` per subscriber. When `throttled=true`, `numRepeats = Long.MAX_VALUE` and `frequency = pollingFrequency` (default 2 seconds), meaning each subscriber polls the database indefinitely every 2 seconds until cancelled or timed out (default timeout: 60 seconds with no emissions). [3](#0-2) [4](#0-3) 

Each poll cycle calls `topicMessageRepository.findByFilter()`, which executes a synchronous JPA query (`getResultList()`) against the database, consuming a HikariCP connection for the duration of the query. [5](#0-4) 

**Root cause:** No global limit on concurrent subscriptions, no per-IP rate limiting in the gRPC module (throttling exists only in the `web3` module), and each historical subscription independently holds and repeatedly acquires database connections. [6](#0-5) 

**Why existing checks fail:**

`maxConcurrentCallsPerConnection = 5` (default) limits calls *per TCP connection*, not globally. An attacker opens M connections × 5 calls = 5M concurrent subscriptions with no server-side rejection. [7](#0-6) [8](#0-7) 

The gRPC server configuration sets only `maxConcurrentCallsPerConnection`; there is no `maxConnectionAge`, no `maxConnections`, and no IP-based throttle configured. [9](#0-8) 

The HikariCP pool is configured via `spring.datasource.hikari` with Spring Boot's default maximum pool size of 10 connections. The Grafana dashboard alert fires only at 75% utilization — it does not prevent exhaustion. [10](#0-9) [11](#0-10) 

### Impact Explanation
When the HikariCP pool is exhausted, all database operations in the gRPC service queue and eventually time out. This affects not only historical retrieval but also the `topicExists()` check (which queries `entityRepository`), the live listener path, and any shared database infrastructure. The result is complete unavailability of the `subscribeTopic` endpoint for legitimate users — a non-network DoS requiring no special privileges.

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl`, any HCS SDK). No authentication, no tokens, no special knowledge beyond the public API schema. The attacker needs to open ~20–50 connections with 5 subscriptions each (100–250 total) with `consensusStartTime` set to epoch 0 to saturate a default 10-connection HikariCP pool. The attack is trivially scriptable, repeatable, and can be sustained indefinitely since each subscription polls every 2 seconds for up to 60 seconds before timing out — at which point the attacker simply reconnects.

### Recommendation

1. **Enforce a global concurrent subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection rate limiting** at the gRPC server level using a `ServerInterceptor` with a token-bucket or sliding-window counter.
3. **Configure `maxConnectionAge` and `maxConnections`** on the `NettyServerBuilder` to bound total server-side connections.
4. **Increase HikariCP pool size** and set a `connectionTimeout` appropriate for the expected load, so pool exhaustion fails fast rather than queuing indefinitely.
5. **Add a `maxConcurrentCallsPerConnection` complement at the global level** — e.g., `serverBuilder.maxConcurrentCallsPerConnection()` combined with a global semaphore in the service layer.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools hedera-sdk or raw proto stubs
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

TARGET = "mainnet-public.mirrornode.hedera.com:443"
TOPIC_ID = 0  # any valid topic

def open_subscription():
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_ID),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),  # epoch = far past
        # No limit set → subscription runs indefinitely
    )
    try:
        for _ in stub.subscribeTopic(request):
            pass  # drain but keep connection alive
    except Exception:
        pass

# Open 50 connections × 5 calls each = 250 concurrent historical retrievals
threads = []
for _ in range(250):
    t = threading.Thread(target=open_subscription, daemon=True)
    t.start()
    threads.append(t)

# HikariCP pool (default 10) is now saturated; legitimate queries time out
input("Attack running. Press Enter to stop.")
```

**Expected result:** After ~10–20 concurrent subscriptions with `startTime=0` on a topic with historical data, `hikaricp_connections_active` reaches `hikaricp_connections_max`, and subsequent `subscribeTopic` calls from legitimate clients receive `DEADLINE_EXCEEDED` or hang indefinitely waiting for a pool connection.

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }

    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }

        config.setUsername(username);
