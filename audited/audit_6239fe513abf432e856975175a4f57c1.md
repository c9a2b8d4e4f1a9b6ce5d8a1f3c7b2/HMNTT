### Title
Unbounded JSON Body Decode in `fetchMirrorNodeNetwork` Enables OOM-Based DoS

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` passes `resp.Body` directly to `json.NewDecoder(...).Decode(...)` with no byte-size cap. A mirror node endpoint that is compromised or attacker-controlled can stream an arbitrarily large JSON body, exhausting the pinger process's heap memory and crashing it. Because the pinger relies on this function to bootstrap its network map, a crash renders it permanently unable to submit liveness transactions until manually restarted.

### Finding Description
**Exact code path:**
`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, line 96:
```go
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {   // line 96
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```
`resp.Body` is read without any size guard. The only protection present is the `http.Client` timeout configured at line 46:
```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```
`http.Client.Timeout` is a **wall-clock time limit**, not a byte-count limit. On a modern network (1 Gbps LAN or even a 100 Mbps WAN), a malicious server can deliver several gigabytes of valid JSON within a typical 30-second timeout window, long before the timer fires. There is no `Content-Length` pre-check, no `http.MaxBytesReader`, and no `io.LimitReader` wrapping `resp.Body` before the decoder is created.

**Root cause / failed assumption:** The code assumes the mirror node REST endpoint is trusted and will return a bounded, well-formed response. This assumption fails when the endpoint is compromised.

**Exploit flow:**
1. Attacker controls the mirror node REST endpoint (e.g., via server compromise, DNS hijacking, or MITM on an unencrypted HTTP connection).
2. Attacker crafts a response with HTTP 200 and a JSON body containing an enormous `nodes` array — millions of entries, each with long `node_account_id` and `service_endpoints` strings.
3. `json.NewDecoder(resp.Body).Decode(&payload)` streams and materialises the entire body into the `nodesEnvelope` struct in heap memory.
4. The Go runtime OOMs and the pinger process is killed by the OS.
5. If the pinger is not auto-restarted, liveness transactions stop permanently.

**Why existing checks are insufficient:**
- `http.Client.Timeout` (line 46): time-based only; a fast server saturates memory before the timer fires.
- Status code check (lines 90–93): only rejects non-2xx; a 200 response bypasses it entirely.
- No `Content-Length` validation, no `http.MaxBytesReader`, no `io.LimitReader`.

### Impact Explanation
The pinger's sole purpose is to submit periodic liveness transactions to the Hedera network. If it crashes during `buildNetworkFromMirrorNodes` (called at startup and on network refresh), it cannot build the node map and halts all transaction submission. This is a complete, persistent denial-of-service of the pinger component. Given the pinger's role in network health monitoring, this directly impairs the ability to detect and respond to network outages.

### Likelihood Explanation
The mirror node REST URL is a single operator-configured string (`cfg.mirrorRest`). An attacker who can compromise that one HTTP endpoint — via server exploit, DNS poisoning, BGP hijacking, or MITM on a plaintext HTTP deployment — can trigger this with a single crafted HTTP response. The attack is repeatable: even if the pinger is restarted, each restart re-calls `fetchMirrorNodeNetwork`, so the attacker can crash it on every boot. No authentication or cryptographic verification of the mirror node response is performed.

### Recommendation
Wrap `resp.Body` with a size-limiting reader before decoding. A reasonable upper bound (e.g., 10 MB) is far larger than any legitimate `/api/v1/network/nodes` response:

```go
const maxBodyBytes = 10 * 1024 * 1024 // 10 MB
limited := io.LimitReader(resp.Body, maxBodyBytes+1)
if err := json.NewDecoder(limited).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```

Alternatively use `http.MaxBytesReader(nil, resp.Body, maxBodyBytes)` which also causes the HTTP transport to abort the connection once the limit is hit. Additionally, validate `resp.ContentLength` before reading if it is non-negative.

### Proof of Concept
1. Stand up a minimal HTTP server that returns HTTP 200 with `Content-Type: application/json` and streams an infinite (or multi-GB) JSON body:
   ```python
   # python3 poc_server.py
   import socket, time
   s = socket.socket(); s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
   s.bind(('0.0.0.0', 8080)); s.listen(1)
   conn, _ = s.accept()
   conn.sendall(b'HTTP/1.1 200 OK\r\nContent-Type: application/json\r\nTransfer-Encoding: chunked\r\n\r\n')
   # Stream a huge nodes array
   chunk = b'{"nodes":[' + (b'{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"' + b'A'*4096 + b'","port":50211}]},' * 10000)
   while True:
       conn.sendall(hex(len(chunk)).encode() + b'\r\n' + chunk + b'\r\n')
   ```
2. Configure the pinger with `mirrorRest: http://<attacker-host>:8080`.
3. Start the pinger. Observe RSS memory growing unboundedly in `top`/`htop` until the process is OOM-killed.
