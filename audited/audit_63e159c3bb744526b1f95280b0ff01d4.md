### Title
Unbounded Prometheus Label Cardinality via Arbitrary URL Paths in MetricsMiddleware

### Summary
The `MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates route label resolution to the weaveworks `middleware.Instrument` with a gorilla/mux-backed `RouteMatcher`. When an incoming request does not match any registered route, the weaveworks library falls back to using the raw `r.URL.Path` as the `route` label value. Because Prometheus stores a separate time-series entry for every unique label combination, an unprivileged attacker can flood the server with requests to unique URL paths, causing unbounded growth of in-process Prometheus metric storage and driving node memory consumption well above 30%.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76–83:

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

`next` is the gorilla/mux-based router returned by `server.NewRouter(...)` (coinbase rosetta-sdk-go, confirmed by `github.com/gorilla/mux v1.8.1` in `go.mod`). The weaveworks `middleware.Instrument.Wrap` method calls `RouteMatcher.Match(r, &routeMatch)` at request start. When a route **matches**, it uses `routeMatch.Route.GetPathTemplate()` (a fixed string like `/network/list`) as the label. When **no route matches** (HTTP 404), it falls back to `r.URL.Path` — the raw, attacker-controlled request path.

All four metric vectors (`requestBytesHistogram`, `requestDurationHistogram`, `requestInflightGauge`, `responseBytesHistogram`) use `"route"` as a label dimension. Each unique `(method, route)` combination causes Prometheus to allocate a new set of internal time-series objects (one per histogram bucket + `_sum` + `_count`). These objects are never garbage-collected during the process lifetime.

**Root cause / failed assumption:** The code assumes `RouteMatcher` will always resolve to a finite, pre-registered set of route templates. It does not sanitize or cap the `route` label value before it is passed to Prometheus.

**No existing checks:** There is no URL length limit, no label allowlist, no rate-limiting, and no `MaxBytesReader` on the URL path visible anywhere in the middleware chain for the rosetta server.

### Impact Explanation
Each unique URL path sent by an attacker creates new Prometheus time-series objects across all four metric vectors. With `sizeBuckets` having 5 buckets, each `requestBytesHistogram` entry alone allocates ~7 time-series objects per unique `(method, route)` pair. Sending 100,000 unique paths (trivially achievable with a script) creates ~2.8 million time-series objects. This directly inflates the Go heap of the rosetta process. Because the objects are never freed, memory usage grows monotonically, easily exceeding a 30% increase over baseline within minutes. In extreme cases this leads to OOM-kill of the node process.

### Likelihood Explanation
The rosetta HTTP port is publicly accessible by design (it serves the Rosetta API). No authentication is required to send arbitrary GET or POST requests. The attack requires only a simple HTTP client loop generating unique path strings. It is fully repeatable, requires no special knowledge of the system, and can be executed from a single machine with modest bandwidth. The attacker does not need to receive a valid response — even 404 responses trigger the label allocation.

### Recommendation
1. **Normalize unmatched routes to a fixed sentinel label** (e.g., `"unmatched"`) before recording metrics. Wrap the `RouteMatcher` with a custom implementation that returns a fixed string when `Match` returns false.
2. **Alternatively**, record metrics only for matched routes and skip instrumentation for 404 responses.
3. **Add a URL path length limit** at the HTTP server level (`http.MaxBytesReader` or a custom middleware) to bound the size of any label value that could leak through.
4. Consider using `prometheus.MustCurryWith` with a fixed label allowlist, or switch to a metrics library that supports label cardinality limits.

### Proof of Concept
```bash
# Sends 50,000 requests to unique paths, each creating a new Prometheus label entry
for i in $(seq 1 50000); do
  curl -s -o /dev/null "http://<rosetta-host>:<port>/exploit-path-$(cat /dev/urandom | tr -dc 'a-z0-9' | head -c 32)" &
done
wait

# Observe memory growth via /metrics endpoint
curl http://<rosetta-host>:<port>/metrics | grep "go_memstats_heap_inuse_bytes"
# Compare before/after: heap will have grown by hundreds of MB
```

Each request to a unique path that does not match any registered Rosetta route causes `middleware.Instrument` to record `r.URL.Path` as the `route` label, permanently allocating new Prometheus time-series storage for all four metric vectors.
