### Title
Unbounded Concurrent Subscriptions with Full-Page In-Memory Materialization Enable JVM Heap Exhaustion

### Summary
The `poll()` method in `PollingTopicMessageRetriever` bounds each individual database query to `maxPageSize` rows, but `findByFilter()` materializes the entire result page into a `List<TopicMessage>` via `getResultList()` before streaming. Because there is no global limit on concurrent subscriptions (only a per-connection cap of 5), an unprivileged attacker can open arbitrarily many connections, each with 5 subscriptions using `startTime=0` and no `endTime` or `limit`, causing continuous parallel page loads that accumulate in JVM heap until OOM.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` accepts any unauthenticated gRPC request and maps it to a `TopicMessageFilter` with no authentication check. [1](#0-0) 

`startTime=0` passes validation because `@Min(0)` permits it and `isValidStartTime()` only requires `startTime <= DomainUtils.now()`. [2](#0-1) 

`poll()` computes `pageSize = Math.min(limit, context.getMaxPageSize())` and passes it as the query `limit`. For throttled mode this is 1000; for unthrottled, 5000. [3](#0-2) 

`findByFilter()` applies `setMaxResults()` correctly, but then calls `typedQuery.getResultList().stream()` — the comment explicitly acknowledges this materializes the full page into a `List<TopicMessage>` in heap before returning a `Stream`. [4](#0-3) 

`isComplete()` in throttled mode returns `true` only when the last page returned fewer rows than `maxPageSize`. On a topic with billions of messages, every page is full, so polling continues indefinitely (`numRepeats = Long.MAX_VALUE`). [5](#0-4) 

The only concurrency guard is `maxConcurrentCallsPerConnection = 5`, which is per-connection only. There is no global subscription count limit enforced — `subscriberCount` is a metrics gauge only, never checked against a cap. [6](#0-5) [7](#0-6) 

The `timeout` of 60 seconds applies between emissions; with a topic that has billions of messages, messages flow continuously and the timeout never fires. [8](#0-7) 

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection` is sufficient to bound aggregate memory pressure. It is not — it only limits calls per TCP connection, not the total number of connections or subscriptions system-wide.

### Impact Explanation
Each concurrent subscription with `startTime=0` on a large topic loads up to 1,000 `TopicMessage` objects (each containing message bytes, running hash, timestamps, etc.) into a `List` in heap every 2 seconds. With N connections × 5 subscriptions = 5N concurrent subscriptions, heap pressure is `5N × 1000 × avg_message_size`. At 1 KB average message size and 200 attacker connections (1,000 subscriptions), this is ~1 GB of live objects per poll cycle, with GC unable to reclaim them while subscriptions are active. This causes JVM OOM and service crash, constituting a non-network DoS against the gRPC mirror node.

### Likelihood Explanation
No authentication is required. Any client with network access to port 5600 can open gRPC connections. Opening hundreds of TCP connections is trivial from a single machine. The attack is repeatable: after a crash, the attacker can immediately reconnect. The only friction is knowing a valid `topicId`, which is public on-chain data.

### Recommendation
1. **Enforce a global subscription limit**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and reject new subscriptions with `RESOURCE_EXHAUSTED` when the cap is reached.
2. **Per-IP or per-client connection rate limiting**: Add a connection limiter at the Netty layer or via an ingress proxy.
3. **Lazy streaming instead of `getResultList()`**: Replace `typedQuery.getResultList().stream()` with a cursor-based or `ScrollableResults`-based approach to avoid materializing the full page into heap. The existing comment acknowledges this trade-off; it should be revisited.
4. **Require a minimum `startTime`**: Reject or warn on `startTime` values older than a configurable threshold (e.g., 30 days) to limit historical scan depth per subscription. [9](#0-8) 

### Proof of Concept
```
# Prerequisites: gRPC client (e.g., grpcurl), valid topicId with billions of messages

# Open 200 parallel connections, each with 5 subscriptions (1000 total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{
      "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": TARGET_TOPIC},
      "consensusStartTime": {"seconds": 0, "nanos": 0}
    }' <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Expected result:
# - JVM heap fills with List<TopicMessage> pages (1000 objects × 1000 subscriptions per poll cycle)
# - GC overhead increases, then OOM error
# - gRPC service becomes unresponsive / crashes
# - Observable via: jvm_memory_used_bytes metric spiking to heap max before crash
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
