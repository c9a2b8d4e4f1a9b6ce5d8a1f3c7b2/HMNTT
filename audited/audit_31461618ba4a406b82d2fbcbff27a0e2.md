### Title
Silent Treasury Account Substitution for Smart Contract Senders Produces Incorrect `msg.sender` in EVM Simulation

### Summary
In `TransactionExecutionService.getSenderAccountID()`, when a caller supplies a sender address that resolves to an account with `smartContract() == true`, the function silently substitutes the treasury account as the transaction payer/sender instead of rejecting the request. Because `transactionID.accountID` is the sole source of the EVM sender identity in the constructed `ContractCallTransactionBody`, the EVM executes the simulation with the treasury account's address as `msg.sender`, not the address the caller supplied. The consensus node would reject the same request with `PAYER_ACCOUNT_NOT_FOUND`; the mirror node instead returns a successful but incorrect simulation result.

### Finding Description

**Exact code path:**

`getSenderAccountID()` at [1](#0-0)  performs the following logic:

1. If sender is zero-address and value is 0 → return treasury (intended default).
2. Otherwise resolve the `AccountID` via alias or long-zero lookup.
3. If `account.smartContract()` is `true` → **silently return treasury account** (line 251).
4. Otherwise return the resolved `AccountID`.

The returned `AccountID` is placed directly into `transactionID.accountID` at [2](#0-1)  which is the only sender identity field in the `ContractCallTransactionBody` / `ContractCreateTransactionBody` passed to the executor. In Hedera's EVM execution model, `transactionID.accountID` is the payer and the EVM sender; its EVM address becomes `msg.sender` inside the called contract.

**Root cause / failed assumption:**

The code comment at [3](#0-2)  explicitly acknowledges that the consensus node returns `PAYER_ACCOUNT_NOT_FOUND` when the payer is a smart contract. The mirror node code, however, does not throw that error for smart-contract senders — it substitutes the treasury account and proceeds. This is a deliberate but security-relevant divergence: the simulation succeeds and returns data computed under the treasury's identity, while the real network would reject the transaction outright.

**Exploit flow:**

1. Attacker identifies any on-chain smart contract address `C` (publicly visible).
2. Attacker sends `POST /api/v1/contracts/call` with `"from": "<address of C>"` and any `"to"` contract address.
3. `getSenderAccountID()` resolves `C` → finds `account.smartContract() == true` → returns treasury `AccountID`.
4. The `ContractCallTransactionBody` is built with `transactionID.accountID = treasury`.
5. The EVM executes the call with `msg.sender = treasury_evm_address`.
6. The mirror node returns `SUCCESS` with results computed from the treasury's perspective.

**Why existing checks are insufficient:**

The only check for smart-contract senders is the branch at line 250 that returns the treasury account — it is itself the source of the problem, not a guard against it. There is no downstream check that detects or rejects the substitution before the EVM runs. The test `testExecuteContractCallSenderIsSmartContract` [4](#0-3)  and the integration test `ethCallWithValueAndSenderContractSucceeds` [5](#0-4)  both confirm the substitution path reaches the EVM and returns `SUCCESS`.

### Impact Explanation

Any contract whose behavior depends on `msg.sender` (access-control gates, ownership checks, allowance logic, role-based branching) will return results computed as if the treasury account initiated the call. This produces:

- **False positive simulations**: calls that would be rejected on the real network (`PAYER_ACCOUNT_NOT_FOUND`) appear successful.
- **Incorrect return data**: `msg.sender`-dependent read paths return treasury-specific state, misleading callers about what their actual transaction would do.
- **Privilege escalation in simulation**: the treasury account holds special roles in many Hedera system contracts; simulating as the treasury can reveal privileged state or produce results that are impossible for the real caller to reproduce.

No funds are directly at risk because the mirror node is read-only, placing this at Medium severity consistent with "unintended smart contract behavior with no concrete funds at direct risk."

### Likelihood Explanation

The precondition is trivially satisfied: any publicly visible contract address on the network qualifies. No authentication, no special role, no prior on-chain interaction is required. The attacker only needs to know one contract address (trivially obtained from any block explorer) and issue a standard `eth_call`-equivalent POST request. The behavior is deterministic and repeatable on every request.

### Recommendation

Replace the silent substitution with an explicit rejection, consistent with what the consensus node does and what the existing comment already documents:

```java
} else if (account.smartContract()) {
    throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
}
```

Remove the line `return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());` at line 251. Update the test `testExecuteContractCallSenderIsSmartContract` to assert `PAYER_ACCOUNT_NOT_FOUND` is thrown (matching `testExecuteContractCallInvalidSenderContract`). The zero-address default path (lines 241-242) is unaffected and should remain.

### Proof of Concept

**Preconditions:** Mirror node running; contract `C` exists at address `0xAAA...` on the network.

**Steps:**

```bash
# 1. Identify any deployed contract address (e.g. from mirror node REST API)
CONTRACT_ADDR="0xaabbccdd..."   # address of contract C (smartContract=true in state)

# 2. Send eth_call with the contract address as the sender
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "from": "'"$CONTRACT_ADDR"'",
    "to":   "0x<any_target_contract>",
    "data": "0x<function_selector>",
    "gas":  15000000
  }'

# Expected (correct) behavior: HTTP 400 / PAYER_ACCOUNT_NOT_FOUND
# Actual behavior:             HTTP 200 / SUCCESS, result computed with
#                              msg.sender = treasury_evm_address
```

**Verification:** Deploy or use a contract that returns `msg.sender` (e.g. `function getSender() external view returns (address) { return msg.sender; }`). The response will contain the treasury account's EVM address, not `CONTRACT_ADDR`.

```java
    private AccountID getSenderAccountID(final CallServiceParameters params) {
        // Set a default account to keep the sender parameter optional.
        if (params.getSender().isZero() && params.getValue() == 0L) {
            return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
        }
        final var senderAddress = params.getSender();
        final var accountIDNum = getSenderAccountIDAsNum(senderAddress);

        final var account = accountReadableKVState.get(accountIDNum);
        if (account == null) {
            throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
        } else if (account.smartContract()) {
            return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
        } else if (!account.hasKey() || account.key().equals(IMMUTABILITY_SENTINEL_KEY)) {
            // If the account is hollow, complete it in the state as a workaround
            // as this happens in HandleWorkflow in hedera-app but calling the
            // transaction executor directly skips this account completion and
            // this results in failed transactions that would otherwise succeed
            // against the consensus node.
            final var writableAccountCache =
                    ContractCallContext.get().getWriteCacheState(AccountReadableKVState.STATE_ID);
            final var completedAccount = account.copyBuilder().key(DEFAULT_KEY).build();
            writableAccountCache.put(account.accountId(), completedAccount);
        }

        return accountIDNum;
    }
```
