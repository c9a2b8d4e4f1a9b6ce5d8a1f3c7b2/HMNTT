### Title
Unbounded In-Memory Transaction Fetch in `/block` Endpoint Enables Resource Exhaustion by Unprivileged Users

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` calls `FindBetween()` to fetch **all** transactions in a block's consensus timestamp range before applying the `maxTransactionsInBlock` cap. The underlying `FindBetween()` implementation in `rosetta/app/persistence/transaction.go` uses a paginated loop with no total-count ceiling, accumulating every transaction row (including expensive correlated JSON subqueries) into a single in-memory slice. Any unauthenticated caller who requests a block containing more transactions than `maxTransactionsInBlock` forces the server to allocate and process the full unbounded set before truncating it for the response.

### Finding Description
**Exact code path:**

`rosetta/app/services/block_service.go`, `Block()`, line 56:
```go
if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
```
No limit is passed to `FindBetween`. The cap is applied only after the call returns, at lines 61–66:
```go
if len(block.Transactions) > s.maxTransactionsInBlock {
    ...
    block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
}
```

`rosetta/app/persistence/transaction.go`, `FindBetween()`, lines 123–147:
```go
transactions := make([]*transaction, 0)
for start <= end {
    transactionsBatch := make([]*transaction, 0)
    err := db.Raw(...).Limit(batchSize).Find(&transactionsBatch).Error
    ...
    transactions = append(transactions, transactionsBatch...)
    if len(transactionsBatch) < batchSize {
        break
    }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```
`batchSize = 2000` (line 22) is a **per-query page size**, not a total cap. The loop continues until the entire timestamp range is exhausted, appending every batch into the `transactions` slice. Each row carries two correlated JSON aggregation subqueries (`crypto_transfer` and `staking_reward_transfer`), making each batch expensive at both the DB and Go heap levels. After the loop, a full in-memory hash-deduplication map is built (lines 149–158) before the result is returned to `Block()`.

**Root cause / failed assumption:** The design assumes `maxTransactionsInBlock` is a response-shaping parameter, but it is applied post-fetch. There is no guard that stops `FindBetween` from loading an arbitrarily large result set into the Go process heap.

### Impact Explanation
An attacker who repeatedly requests a single high-transaction block forces the server to:
1. Execute multiple expensive paginated SQL queries with correlated subqueries.
2. Accumulate all rows in a growing Go slice (heap pressure proportional to total transactions × row size).
3. Build a full `map[string][]*transaction` deduplication structure.
4. Discard most of the data only after all of the above is complete.

On Hedera mainnet, individual record files (blocks) routinely contain thousands of transactions. A sustained stream of such requests from a single unauthenticated client can push memory and CPU well above the 30% baseline threshold. Concurrent requests multiply the effect linearly. There is no per-endpoint rate limiting visible in the router setup (`main.go` lines 111–119).

### Likelihood Explanation
The `/block` endpoint is public and unauthenticated (Rosetta spec requirement). Block indices are sequential and publicly enumerable. An attacker needs only to identify a historically dense block (trivially discoverable via any Hedera block explorer or by scanning sequentially) and issue repeated POST requests to `/block`. No credentials, tokens, or special network position are required. The attack is fully repeatable and scriptable.

### Recommendation
Apply the transaction limit **before** the database fetch, not after. Pass `maxTransactionsInBlock` (plus one, to detect overflow) as the `LIMIT` in the SQL query, or add an early-exit condition inside the `FindBetween` loop:

```go
// In FindBetween, accept an optional maxRows int parameter.
// Break as soon as len(transactions) >= maxRows.
if maxRows > 0 && len(transactions) >= maxRows {
    break
}
```

Alternatively, push the limit into the SQL itself by adding `LIMIT @maxRows` to `selectTransactionsInTimestampRangeOrdered` and passing `maxTransactionsInBlock + 1` from `Block()`. Additionally, add per-IP or global rate limiting on the `/block` endpoint.

### Proof of Concept
1. Identify a Hedera mainnet block with a large transaction count (e.g., any high-throughput period; block indices are public).
2. Send repeated POST requests to the Rosetta `/block` endpoint with that block's index:
```json
POST /block
{
  "network_identifier": {"blockchain": "Hiero", "network": "mainnet"},
  "block_identifier": {"index": <HIGH_TX_BLOCK_INDEX>}
}
```
3. Monitor the rosetta process RSS and CPU. Each request causes `FindBetween` to page through all N transactions (N >> `maxTransactionsInBlock`) and build the full in-memory map before truncating.
4. With a concurrency of ~10 parallel clients issuing this request in a tight loop, observe memory and CPU usage climb well above 30% of the idle baseline, sustained for as long as the requests continue. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
