### Title
Missing Server-Side gRPC Keepalive Enables Zombie Connection Accumulation Leading to Executor Exhaustion (DoS)

### Summary
`grpcServerConfigurer()` in `GrpcConfiguration.java` configures the `NettyServerBuilder` with only an executor and `maxConcurrentCallsPerConnection`, omitting `keepAliveTime`, `keepAliveTimeout`, `maxConnectionIdle`, and `maxConnectionAge`. Without server-initiated HTTP/2 PING frames, the server cannot detect dead TCP connections during a network partition. An unprivileged attacker can open many long-lived `subscribeTopic` streaming connections, silently drop packets to simulate a partition, and cause zombie streams to accumulate until the `applicationTaskExecutor` is exhausted, denying service to legitimate clients.

### Finding Description

**Exact code location:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 27–35.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // NO keepAliveTime, keepAliveTimeout, maxConnectionIdle, maxConnectionAge
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–14: only `maxConcurrentCallsPerConnection = 5` is modeled — no keepalive fields exist in the properties class at all.

**Root cause:**
gRPC-Java's `NettyServerBuilder` defaults `keepAliveTime` to `Long.MAX_VALUE` (effectively disabled). With no server-side PING configured, the server relies entirely on OS-level TCP keepalive (~2 hours on Linux) to detect dead peers. During a network partition where the client silently drops packets (no TCP RST/FIN), the server-side TCP connection stays in `ESTABLISHED` state indefinitely.

**Exploit flow:**

1. The `subscribeTopic` RPC (`ConsensusController.java` lines 43–53) is a server-streaming call with no authentication. When called with no `endTime` and no `limit`, `TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) creates a `Flux` that runs indefinitely — `pastEndTime()` returns `Flux.never()` when `endTime == null`.

2. Each active stream holds a Reactor subscription, a `TopicListener` registration, and — critically — an `applicationTaskExecutor` thread whenever the server attempts to write data to the client.

3. `maxConcurrentCallsPerConnection = 5` limits calls *per connection*, not total connections. There is no `maxConnections` or rate-limit on new TCP connections configured anywhere in the server builder.

4. An attacker opens `N` TCP connections × 5 `subscribeTopic` calls = `5N` zombie streams, then installs a firewall rule (`iptables -A OUTPUT -p tcp --dport 5600 -j DROP`) to silently black-hole packets. The server never receives a TCP RST/FIN.

5. Without `keepAliveTime` set, the server never sends HTTP/2 PING frames. Without `maxConnectionIdle`, idle connections are never reaped. The zombie streams persist until OS TCP keepalive fires (~2 hours).

6. The `applicationTaskExecutor` thread pool fills with threads blocked on writes to dead connections. New legitimate `subscribeTopic` calls queue and eventually time out or are rejected.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5`: Scoped per-connection; attacker opens unbounded connections.
- `retriever.timeout = 60s`: Applies to the message retriever gap between emissions, not to the transport write path.
- No `maxConnectionAge` / `maxConnectionIdle`: No forced connection recycling.
- No authentication/authorization on the gRPC endpoint: any external client can connect.

### Impact Explanation

Complete denial of service for the `subscribeTopic` and `getNodes` gRPC endpoints. Once the `applicationTaskExecutor` is saturated with threads blocked on writes to dead connections, no new RPC calls can be dispatched. The mirror node's primary public API (HCS topic subscription) becomes unavailable to all legitimate clients. The effect persists for up to ~2 hours per zombie connection batch (OS TCP keepalive interval) unless the server is restarted.

Severity: **High** — unauthenticated, remotely triggerable, full service disruption.

### Likelihood Explanation

The `subscribeTopic` endpoint is publicly documented and accessible without credentials (plaintext on port 5600 per `docker-compose.yml` line 190 and `docs/grpc/README.md` line 16). The attacker needs only a standard gRPC client (`grpcurl`) and a local firewall rule — no exploit code, no vulnerability in a dependency, no privileged access. The attack is repeatable: after a server restart, the attacker can immediately re-establish zombie connections. The `maxConcurrentCallsPerConnection = 5` default actually makes the attack more efficient by allowing 5 streams per TCP connection, multiplying resource consumption per socket.

### Recommendation

Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Expose `keepAliveTime`, `keepAliveTimeout`, `maxConnectionIdle`, and `maxConnectionAge` as configurable fields in `NettyProperties.java` so operators can tune them without code changes. Additionally, consider adding a connection-level rate limiter or total connection cap at the load balancer (nginx `grpc_pass` layer already present in `docker-compose.yml`).

### Proof of Concept

**Preconditions:** Network access to port 5600; `grpcurl` installed; Linux host with `iptables`.

```bash
# Step 1: Open 20 connections, each with 5 infinite subscribeTopic streams (100 zombie streams total)
for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}}' \
      target-host:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Step 2: Simulate network partition — silently drop all outbound packets to the server
# (server never receives RST/FIN, connections stay ESTABLISHED on server side)
iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# Step 3: Wait ~30 seconds for the server's write buffers to fill and executor threads to block

# Step 4: Restore connectivity and attempt a legitimate subscription — it will hang/fail
iptables -D OUTPUT -p tcp --dport 5600 -j DROP
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}, "limit": 1}' \
  target-host:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: timeout or RESOURCE_EXHAUSTED — executor has no free threads
```

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
```

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
