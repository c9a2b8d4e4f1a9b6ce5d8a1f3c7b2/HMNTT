### Title
Unauthenticated DB Connection Pool Exhaustion via Uncached `isValidAccount()` in `/accounts/:id/rewards`

### Summary
The `listStakingRewardsByAccountId()` handler in `rest/controllers/accountController.js` unconditionally calls `EntityService.isValidAccount()` on every request, which always issues a live `SELECT` against the database with no caching, no rate limiting, and no concurrency control. An unprivileged external attacker can flood the endpoint with concurrent requests using publicly known valid account IDs, exhausting the finite `pg` connection pool shared across all REST API handlers and causing widespread request failures across the mirror node service.

### Finding Description

**Exact code path:**

`listStakingRewardsByAccountId()` at `rest/controllers/accountController.js` lines 170–203 executes two sequential database queries per request with no guard:

```js
// Line 171 – may hit DB for alias/EVM address lookups
const accountId = await EntityService.getEncodedId(req.params[...]);
// Line 172 – ALWAYS hits DB, no cache
const isValidAccount = await EntityService.isValidAccount(accountId);
// Line 180 – ALWAYS hits DB
const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(...);
```

`EntityService.isValidAccount()` at `rest/service/entityService.js` lines 60–63 issues:

```js
static entityExistenceQuery = `select ${Entity.TYPE} from ${Entity.tableName} where ${Entity.ID} = $1`;
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
```

There is no result cache, no memoization, and no deduplication. Every HTTP request triggers a fresh `SELECT type FROM entity WHERE id = $1`.

**Connection pool:**

`rest/dbpool.js` lines 7–16 creates a single `pg.Pool` with `max: config.db.pool.maxConnections`. This pool is shared by every endpoint. With N concurrent requests each holding a connection for the duration of two sequential queries, the pool is exhausted when N ≥ `maxConnections / 2`.

**No rate limiting:**

A full audit of `rest/middleware/` confirms there is no rate-limiting middleware. `rest/middleware/authHandler.js` only enforces response-size limits for authenticated users; it does not throttle request rates. `rest/server.js` lines 67–99 show no IP-based throttling or concurrency limiter is registered.

**Root cause:** The failed assumption is that database existence checks are cheap enough to be called unconditionally on every unauthenticated request. With a bounded connection pool and no admission control, this assumption breaks under concurrent load.

### Impact Explanation

When the pool is exhausted, every subsequent request to any endpoint on the same mirror node instance fails with a connection-timeout error (`connectionTimeoutMillis` from `rest/dbpool.js` line 13). This includes health checks, transaction queries, token queries, and all other REST endpoints. The mirror node REST API becomes entirely unresponsive for the duration of the attack. Because the pool is per-process and the attack is cheap to sustain (valid account IDs are permanently on-chain and publicly enumerable), the attacker can maintain the degraded state indefinitely with a modest number of concurrent HTTP connections.

### Likelihood Explanation

No privileges, API keys, or special knowledge are required. Valid account IDs (e.g., `0.0.98`, `0.0.800`) are publicly known from the Hedera ledger. The attacker needs only a standard HTTP client capable of issuing concurrent requests (e.g., `curl --parallel`, `ab`, `wrk`). The attack is repeatable, stateless, and requires no prior reconnaissance beyond knowing the endpoint path. The default pool size is small (the config test at `rest/__tests__/config.test.js` line 275 shows `maxConnections: 5` as a representative value), meaning even a handful of concurrent connections suffices to saturate it.

### Recommendation

1. **Add a result cache for `isValidAccount()`**: Cache the existence result keyed on `accountId` with a short TTL (e.g., 30–60 seconds). The entity table changes infrequently enough that stale reads are acceptable.
2. **Add per-IP or global rate limiting**: Introduce a rate-limiting middleware (e.g., `express-rate-limit`) before route handlers, applied globally or specifically to the `/rewards` endpoint.
3. **Increase pool size or add a concurrency semaphore**: Ensure the pool is sized relative to expected concurrent load, or add a semaphore that rejects requests when the pool is near exhaustion rather than queuing them indefinitely.
4. **Merge existence check into the rewards query**: Replace the two-query pattern (existence check + data fetch) with a single query that returns an empty result set for non-existent accounts, eliminating one pool acquisition per request.

### Proof of Concept

```bash
# Use 50 valid account IDs (publicly known from Hedera mainnet)
# Send 200 concurrent requests to exhaust the pool

for i in $(seq 1 200); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.98/rewards" &
done
wait

# Immediately probe another endpoint to confirm degradation
curl -v "https://<mirror-node-host>/api/v1/transactions"
# Expected: connection timeout or 503 due to pool exhaustion
```

Reproducible steps:
1. Identify any valid account ID (e.g., `0.0.98` is the Hedera fee collection account, always present).
2. Launch ≥ `maxConnections` concurrent unauthenticated GET requests to `/api/v1/accounts/0.0.98/rewards`.
3. While those are in-flight, issue a request to any other endpoint (e.g., `/api/v1/transactions`).
4. Observe that the second request times out or returns an error, confirming pool exhaustion.
5. Sustain the flood to maintain the degraded state indefinitely.
