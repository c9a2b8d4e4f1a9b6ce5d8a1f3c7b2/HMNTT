### Title
Unbounded Concurrent `TransactionExecute` Calls in `ConstructionSubmit` Enable Resource Exhaustion DoS

### Summary
`ConstructionSubmit` in `rosetta/app/services/construction_service.go` applies no application-level rate limiting or concurrency control before calling the blocking `hiero.TransactionExecute(transaction, c.sdkClient)` on a single shared SDK client. An unprivileged attacker can flood the endpoint with syntactically valid, semantically invalid signed transactions, causing goroutine accumulation and gRPC HTTP/2 stream exhaustion on the shared `sdkClient`, preventing legitimate transactions from being submitted to the network.

### Finding Description

**Exact code path:**

`ConstructionSubmit` (lines 332–368) performs three cheap local operations before reaching the blocking network call:
1. `unmarshallTransactionFromHexString` — deserializes protobuf bytes and checks the type is `AccountCreateTransaction` or `TransferTransaction` (lines 340–343)
2. `TransactionGetTransactionHash` — local hash computation (line 345)
3. `hiero.TransactionExecute(transaction, c.sdkClient)` — **blocking gRPC call to a Hedera network node** (line 355)

Critically, **signatures are never re-verified in `ConstructionSubmit`**. The function accepts any syntactically valid transaction bytes and unconditionally forwards them to the network. An attacker only needs to produce valid protobuf-encoded `AccountCreateTransaction` or `TransferTransaction` bytes (with arbitrary/random signatures) to reach `TransactionExecute`.

**Root cause — failed assumption:**

The code assumes that upstream infrastructure (Traefik) will enforce concurrency and rate limits before requests reach the application. This assumption fails because:

- The application middleware stack (`main.go` lines 217–219) contains only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — **zero rate limiting or concurrency control at the application layer**.
- The Traefik middleware in `charts/hedera-mirror-rosetta/values.yaml` (lines 149–166) is an optional Helm chart default, not an application-level guarantee. It is absent in bare-metal, Docker Compose, or custom Kubernetes deployments.

**Why the Traefik protections are insufficient even when deployed:**

| Traefik control | Configured value | Bypass |
|---|---|---|
| `inFlightReq: amount: 5` | Per source IP (`ipStrategy: depth: 1`) | Attacker uses ≥N IPs (botnet/proxies); each IP gets 5 concurrent slots |
| `rateLimit: average: 10` | Per `requestHost` (Host header) | Applies to the Host header value, not source IP; trivially manipulated |
| `retry: attempts: 3` | On network errors | Amplifies load 3× for any request that causes a 5xx from the backend |

**Goroutine and gRPC stream exhaustion mechanism:**

Go's `net/http` server spawns one goroutine per request. Each goroutine calling `TransactionExecute` blocks on a gRPC unary RPC to a Hedera node, waiting for a response (even a rejection like `INVALID_SIGNATURE` requires a full network round-trip). The shared `sdkClient` uses gRPC over HTTP/2; the default gRPC maximum concurrent streams per connection is 100. With enough concurrent requests from distributed IPs:

1. All 100 HTTP/2 stream slots on each node's gRPC channel are occupied.
2. New `TransactionExecute` calls queue or fail immediately.
3. Goroutines accumulate in memory (~8 KB stack each, growing under load).
4. Legitimate `ConstructionSubmit` calls either queue indefinitely or receive gRPC resource-exhaustion errors.

`sdkClient.SetMaxAttempts(1)` (line 636) prevents SDK-level retries, but does not bound concurrency — it only ensures each blocked call completes after exactly one round-trip rather than multiple.

### Impact Explanation

When the gRPC stream limit is saturated, `hiero.TransactionExecute` returns an error for all new calls. `ConstructionSubmit` maps this to `ErrTransactionSubmissionFailed` (error code 118, `retriable: false`). No legitimate transaction can be submitted through this Rosetta node for the duration of the attack. Because the Rosetta construction workflow is the only path for external parties to submit transactions via this node, this constitutes a complete transaction submission outage — matching the "total network shutdown" severity classification for this node's role.

### Likelihood Explanation

**Preconditions:** The attacker needs only to produce syntactically valid protobuf bytes for an `AccountCreateTransaction` or `TransferTransaction`. No valid account, no funds, and no correct signature are required — the network will reject the transaction semantically, but the gRPC call still completes a full round-trip. Standard Hedera SDK tooling or raw protobuf construction suffices.

**Feasibility:** A single machine with multiple source IPs (or a small botnet) can sustain the attack. Each "slot" is held for the duration of one Hedera network round-trip (~1–3 seconds on mainnet/testnet). With 5 IPs × 5 concurrent slots = 25 concurrent blocking calls, the 100-stream gRPC limit is approached quickly. Without Traefik, a single machine can open hundreds of concurrent connections.

**Repeatability:** The attack is stateless and trivially repeatable. No authentication, no account balance, no prior state is required.

### Recommendation

1. **Application-level concurrency semaphore on `ConstructionSubmit`:** Add a `chan struct{}` semaphore (e.g., capacity 10–20) acquired before `TransactionExecute` and released after. Requests that cannot acquire the semaphore immediately return HTTP 429 / `ErrTransactionSubmissionFailed`.

2. **Application-level rate limiter:** Use `golang.org/x/time/rate` to enforce a per-IP token-bucket rate limit inside the handler, independent of any infrastructure proxy.

3. **Context timeout on `TransactionExecute`:** Pass a `context.WithTimeout` (e.g., 5 s) to bound how long each blocked goroutine can hold a gRPC stream slot.

4. **Do not rely solely on Traefik defaults:** The Helm chart middleware is not a substitute for in-process protection. Document that the Traefik middleware is required for production and add a startup check or warning when running without it.

### Proof of Concept

```python
import threading, requests, struct

# Minimal valid AccountCreateTransaction protobuf (hex) with random signature
# Craft using hiero-sdk or raw protobuf; must pass isSupportedTransactionType check
VALID_TX_HEX = "<hex-encoded AccountCreateTransaction protobuf bytes>"

TARGET = "http://<rosetta-online-host>/construction/submit"
PAYLOAD = {
    "network_identifier": {"blockchain": "Hiero", "network": "testnet"},
    "signed_transaction": "0x" + VALID_TX_HEX
}

def flood():
    while True:
        try:
            requests.post(TARGET, json=PAYLOAD, timeout=10)
        except Exception:
            pass

# Launch from multiple source IPs to bypass per-IP inFlightReq limit
threads = [threading.Thread(target=flood) for _ in range(200)]
for t in threads:
    t.start()

# Observe: legitimate ConstructionSubmit calls begin returning
# "Transaction submission failed" as gRPC streams are exhausted
```

**Steps:**
1. Craft a valid protobuf `AccountCreateTransaction` or `TransferTransaction` (any signatures, even random bytes).
2. Hex-encode and POST to `/construction/submit` from ≥6 source IPs with high concurrency.
3. Simultaneously attempt a legitimate signed transaction via `/construction/submit`.
4. Observe the legitimate transaction receiving `ErrTransactionSubmissionFailed` (error 118) while the flood is active.
