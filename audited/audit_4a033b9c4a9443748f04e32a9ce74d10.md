### Title
Unauthenticated Full-History Scan via Timestamp Parameter Triggers Expensive `DISTINCT ON` Query in Hook Storage API

### Summary
Any unauthenticated external user can supply `timestamp=gte:0.000000000` to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, which unconditionally routes execution to `getHookStorageChange()` and issues a `SELECT DISTINCT ON (key)` query against the entire `hook_storage_change` table for that hook. Because `DISTINCT ON` requires a full scan of all matching rows before the `LIMIT` is applied, and because no rate limiting exists in the `rest-java` module for this endpoint, an attacker can repeatedly trigger this expensive code path to consume significantly more CPU than the non-historical path.

### Finding Description

**Code path:**

1. **Controller entry** — `HooksController.getHookStorage()` accepts `timestamps` with no authentication and only a `@Size(max = 2)` constraint (no minimum-value check): [1](#0-0) 

2. **Bound construction** — `hookStorageChangeRequest()` passes the raw timestamp directly to `Bound.of()` with no floor validation: [2](#0-1) 

3. **`isHistorical()` gate** — any non-empty timestamp bound (including `gte:0`) returns `true`, routing to the expensive path: [3](#0-2) 

4. **Timestamp bounds** — with `gte:0`, `timestampLowerBound = 0` and `timestampUpperBound = Long.MAX_VALUE` (or current time), spanning the entire history: [4](#0-3) 

5. **Expensive query** — `findByKeyBetweenAndTimestampBetween()` uses `SELECT DISTINCT ON (key)` which forces PostgreSQL to scan **all** rows matching `(owner_id, hook_id, timestamp BETWEEN 0 AND MAX)` before deduplicating and applying `LIMIT`: [5](#0-4) 

**Root cause:** The `isHistorical()` predicate treats any timestamp value — including `gte:0` — as a valid historical query, and the `DISTINCT ON (key)` query has no early-exit mechanism; the database must read every qualifying row before the `LIMIT` clause can be applied. The non-historical path queries the `hook_storage` table via a simple primary-key index scan, which is orders of magnitude cheaper.

**Why existing checks fail:**
- `@Size(max = 2)` limits the number of timestamp parameters, not their values.
- `@Max(MAX_LIMIT)` on `limit` caps output rows at 100 but does not reduce the number of rows the database must scan.
- The `ThrottleConfiguration`/`ThrottleManagerImpl` rate limiter exists only in the `web3` module; the `rest-java` module has no equivalent throttle for this endpoint. [6](#0-5) 

### Impact Explanation
A single request with `timestamp=gte:0.000000000` forces a full sequential scan of `hook_storage_change` for the targeted `(owner_id, hook_id)` pair, deduplicating across all historical partitions. For a hook with a large write history, this can consume substantially more CPU and I/O than the non-historical path. Repeated requests from one or more unauthenticated clients can degrade database CPU by ≥30% relative to baseline, satisfying the stated impact threshold. The `hook_storage_change` table is partitioned by `consensus_timestamp`, so a range of `[0, MAX_LONG]` crosses all partitions. [7](#0-6) 

### Likelihood Explanation
The exploit requires zero privileges — no API key, no authentication, no special account. The endpoint is publicly documented and the `timestamp` parameter is explicitly described as accepting `gte:` operators. The attack is trivially repeatable with a simple HTTP client loop. Any attacker who can reach the mirror node's REST API can execute this. [8](#0-7) 

### Recommendation
1. **Enforce a minimum timestamp lower bound**: Reject or clamp `timestamp=gte:0` to a reasonable floor (e.g., no more than N days in the past), consistent with how other historical endpoints are protected.
2. **Add rate limiting to `rest-java`**: Apply a per-IP or global request-rate limiter (e.g., Bucket4j, as used in `web3`) specifically for the historical hook storage endpoint.
3. **Rewrite the query to use `ORDER BY … LIMIT` with an index**: Replace `DISTINCT ON (key)` with a lateral join or a recursive CTE that leverages the `(owner_id, hook_id, key, consensus_timestamp)` primary key index to fetch the latest value per key without a full scan.
4. **Validate that `timestamp` lower bound is not zero or near-zero**: Add a `@Min` constraint or explicit service-layer validation rejecting timestamps below a configurable threshold.

### Proof of Concept

```bash
# Step 1: Identify any valid account/hook pair (public information from the hooks list endpoint)
OWNER_ID="0.0.123"
HOOK_ID="1"
BASE_URL="https://<mirror-node-host>"

# Step 2: Issue the full-history scan request as an unauthenticated user
curl -s "${BASE_URL}/api/v1/accounts/${OWNER_ID}/hooks/${HOOK_ID}/storage?timestamp=gte:0.000000000"

# Step 3: Repeat in a loop to amplify CPU consumption
for i in $(seq 1 200); do
  curl -s "${BASE_URL}/api/v1/accounts/${OWNER_ID}/hooks/${HOOK_ID}/storage?timestamp=gte:0.000000000" &
done
wait
```

Each request triggers `getHookStorageChange()` → `findByKeyBetweenAndTimestampBetween()` with `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`, executing `SELECT DISTINCT ON (key) … FROM hook_storage_change WHERE … AND consensus_timestamp BETWEEN 0 AND 9223372036854775807` across all time partitions, with no rate limiting to prevent concurrent execution.
