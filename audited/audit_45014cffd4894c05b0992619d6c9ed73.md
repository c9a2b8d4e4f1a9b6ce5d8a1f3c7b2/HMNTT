### Title
Missing JDBC `socketTimeout` in rest-java HikariCP Pool Causes Indefinite Thread/Connection Blocking During Network Partition

### Summary
The `findByAlias()` method in `rest-java`'s `EntityRepository` executes a native SQL query over a HikariCP JDBC connection that has no `socketTimeout` configured at the driver level. When a network partition silences the TCP connection to PostgreSQL, the calling thread blocks indefinitely on the broken socket. The server-side `statementTimeout` (10 s) cannot be delivered across a dead TCP link, so threads and pooled connections accumulate until the OS TCP-keepalive timer (default ~2 h on Linux) finally reaps them, enabling a connection/thread-pool exhaustion DoS.

### Finding Description

**Exact code location**

`findByAlias()` issues a native query: [1](#0-0) 

The query is executed on a connection drawn from the Spring Boot auto-configured HikariCP pool. The documented rest-java database properties contain no `socketTimeout` entry: [2](#0-1) 

The only timeout present is `statementTimeout` (10 000 ms), which is a PostgreSQL server-side `SET statement_timeout` parameter.

**Root cause and failed assumption**

The PostgreSQL JDBC driver's `socketTimeout` defaults to `0` (infinite) when not supplied in the JDBC URL or as a `DataSource` property. HikariCP forwards driver-level properties only when they are explicitly set. Because no `socketTimeout` is configured for the rest-java pool, every `Socket.read()` call inside the driver has no OS-level deadline.

The failed assumption is that `statementTimeout` protects against broken connections. It does not: `statement_timeout` is enforced by the PostgreSQL backend, which sends a cancellation notice over the same TCP socket. If that socket is silenced by a network partition, the notice never arrives and the client-side `read()` blocks forever.

**Why the startup-only `socketTimeout` is irrelevant**

`DatabaseWaiter` sets `socketTimeout` only on a temporary `DriverManager` connection used during startup health-checking: [3](#0-2) 

This has no effect on the production HikariCP pool.

**Exploit flow**

1. A network partition silences the path between the rest-java pod and PostgreSQL (e.g., a firewall rule, a cloud-provider network event, or an attacker with network-segment access performing ARP/BGP manipulation).
2. An unprivileged external user sends repeated HTTP requests to any rest-java endpoint that resolves an account by alias (triggering `findByAlias()`).
3. Each request acquires a HikariCP connection and blocks in `Socket.read()` with no timeout.
4. HikariCP's pool is exhausted; subsequent requests queue or fail immediately with "connection timeout."
5. The Tomcat/Netty thread pool fills with blocked threads.
6. The service becomes unresponsive to all traffic until the OS TCP-keepalive timer (Linux default: `tcp_keepalive_time` = 7 200 s, `tcp_keepalive_intvl` = 75 s × `tcp_keepalive_probes` = 9 ≈ 2 h) detects the dead connections and unblocks them.

### Impact Explanation
Complete service-level DoS for the rest-java API for the duration of the partition plus up to ~2 hours of recovery time. All endpoints sharing the same HikariCP pool are affected, not only alias-lookup paths. Severity: **High** (availability impact, no authentication required to trigger once the partition exists).

### Likelihood Explanation
The precondition is a network partition between the application tier and the database. In cloud-native deployments this occurs naturally (transient network events, rolling DB failovers, cloud provider incidents). An attacker with access to the same network segment (e.g., a compromised adjacent pod in Kubernetes, or a cloud tenant with shared-network access) can induce the partition via ARP poisoning or firewall manipulation. Once the partition exists, any unauthenticated HTTP client can amplify the impact by flooding alias-lookup requests. The combination of a realistic precondition and a trivial trigger makes this a credible, repeatable attack.

### Recommendation

1. **Set `socketTimeout` in the JDBC URL or HikariCP `dataSourceProperties`** for the rest-java pool (e.g., `socketTimeout=30` seconds for the PostgreSQL JDBC driver). This causes `Socket.read()` to throw after 30 s regardless of TCP state.
2. **Enable HikariCP `keepaliveTime`** (e.g., 30 000 ms) so the pool proactively validates idle connections and evicts dead ones before they are handed to a request thread.
3. **Set HikariCP `connectionTimeout`** to a low value (e.g., 5 000 ms) so requests fail fast when the pool is exhausted rather than queuing indefinitely.
4. **Add an application-level query timeout** via Spring Data's `@QueryHints` (`javax.persistence.query.timeout`) as a defence-in-depth layer.

### Proof of Concept

```
# 1. Simulate a network partition (on the host running rest-java, or via a network policy):
iptables -I OUTPUT -d <postgres-host> -p tcp --dport 5432 -j DROP

# 2. Flood alias-lookup requests from an unprivileged client:
for i in $(seq 1 50); do
  curl -s "http://<rest-java-host>:8080/api/v1/accounts/<alias-encoded>" &
done
wait

# 3. Observe: all 50 threads block; HikariCP pool exhausted within seconds.
# 4. Attempt a new request:
curl -v "http://<rest-java-host>:8080/api/v1/accounts/0.0.1"
# Result: connection refused or HTTP 503 — service unresponsive.

# 5. Restore network; service remains degraded for up to ~2 hours
#    until OS TCP-keepalive reaps the dead connections.
iptables -D OUTPUT -d <postgres-host> -p tcp --dport 5432 -j DROP
```
