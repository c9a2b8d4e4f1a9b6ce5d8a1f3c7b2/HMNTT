### Title
Unbounded Timestamp Scan via Missing Upper Bound Cap in `getHookStorageChange`

### Summary
When a caller supplies only a lower-bound timestamp parameter (e.g., `timestamp=gte:X`) with no upper bound, `Bound.adjustUpperBound()` returns `Long.MAX_VALUE`, which is passed verbatim as `timestampUpperBound` to the repository queries. The resulting `DISTINCT ON (key)` SQL must process every matching row from timestamp X to `Long.MAX_VALUE` for the given owner/hook before the page limit is applied, enabling an unprivileged caller to trigger arbitrarily large DB scans.

### Finding Description
**Code path:**

`HookServiceImpl.getHookStorage` (line 57) delegates to the private `getHookStorageChange` (line 84) whenever `request.isHistorical()` is true. `isHistorical()` returns `!timestamp.isEmpty()`, and `isEmpty()` is `lower == null && upper == null` — so supplying only `timestamp=gte:X` makes `isHistorical()` return `true`.

Inside `getHookStorageChange` (lines 97–99):
```java
final var timestamp = request.getTimestamp();
final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
final long timestampUpperBound = timestamp.adjustUpperBound();   // ← Long.MAX_VALUE
```

`Bound.adjustUpperBound()` (lines 63–74 of `Bound.java`):
```java
public long adjustUpperBound() {
    if (this.upper == null) {
        return Long.MAX_VALUE;   // ← no cap applied
    }
    ...
}
```

`Long.MAX_VALUE` is then forwarded directly to both repository queries (lines 104–114):
```java
hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
// or
hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
        ..., timestampLowerBound, timestampUpperBound, page);
```

Both queries use `consensus_timestamp between :timestampLowerBound and :timestampUpperBound` combined with `DISTINCT ON (key) … ORDER BY key, consensus_timestamp DESC`. PostgreSQL must read and sort **all** rows matching `owner_id`, `hook_id`, key range, and the open-ended timestamp range before it can emit the first page of 25 results. The `Pageable` limit only caps the output rows, not the rows the engine must examine.

The only guard in `Bound`'s constructor (lines 56–60) is `if (primarySortField && adjustedLower > adjustedUpper)` — with lower=X and upper=`Long.MAX_VALUE`, the condition is never true, so no exception is thrown.

### Impact Explanation
An attacker who knows (or enumerates) a valid `ownerId`/`hookId` pair for a hook with a large history can repeatedly issue requests with `timestamp=gte:0`, forcing the database to perform a full historical scan of that hook's storage-change table on every request. This can exhaust DB CPU and I/O, degrade response times for all users, and delay transaction gossip processing that depends on the same database.

### Likelihood Explanation
The endpoint is reachable by any unprivileged external caller. The only precondition is knowing a valid `ownerId` and `hookId`, which are enumerable via the public `/hooks` endpoint. The attack is trivially repeatable with a single HTTP parameter and requires no authentication or special tooling.

### Recommendation
Cap `timestampUpperBound` to the current wall-clock time (or a configurable maximum range) before passing it to the repository. For example, in `getHookStorageChange`:
```java
final long now = System.currentTimeMillis() * 1_000_000L; // nanos
final long timestampUpperBound = Math.min(timestamp.adjustUpperBound(), now);
```
Alternatively, enforce a maximum timestamp range (e.g., reject requests where `upper - lower > MAX_RANGE_NS`) and require an explicit upper bound when a lower bound is provided. A similar pattern already exists in the Node.js REST layer (`bindTimestampRange` in `rest/timestampRange.js`).

### Proof of Concept
1. Enumerate a valid `ownerId` and `hookId` via `GET /api/v1/hooks`.
2. Issue the following request with no upper-bound timestamp:
   ```
   GET /api/v1/hooks/{ownerId}/{hookId}/storage?timestamp=gte:0
   ```
3. Observe that the DB executes `consensus_timestamp between 0 and 9223372036854775807` for the given hook, scanning the entire history.

```java
    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
```

```java
    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
```
