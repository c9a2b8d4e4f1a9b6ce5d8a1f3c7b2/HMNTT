### Title
`@CachePut` in `findByTimestamp()` Bypasses `@Cacheable` and Pollutes Index Cache on Every Request

### Summary
In `RecordFileRepository.findByTimestamp()`, combining `@Cacheable` (timestamp cache) and `@CachePut` (index cache) inside a single `@Caching` annotation triggers a well-documented Spring Framework behavior: because `hasCachePut()` returns `true`, Spring's `CacheAspectSupport` **always invokes the underlying method** (DB query) even when the timestamp cache has a hit. Additionally, the `@CachePut` writes to the index cache using the raw `timestamp` value as the key — not the block index — polluting the index cache with entries that `findByIndex()` can never look up. An unprivileged attacker can exploit this to force sustained DB load and evict legitimate index-keyed cache entries.

### Finding Description

**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 38–46.

```java
@Caching(
        cacheable =
                @Cacheable(
                        cacheNames = CACHE_NAME,
                        cacheManager = CACHE_MANAGER_RECORD_FILE_TIMESTAMP,
                        unless = "#result == null"),
        put = @CachePut(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX))
@Query("select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1")
Optional<RecordFile> findByTimestamp(long timestamp);
```

**Root cause — Spring `hasCachePut()` bypass:**
Spring's `CacheAspectSupport.execute()` contains the following guard:

```java
if (cacheHit != null && !hasCachePut(contexts)) {
    // return cached value, skip method
} else {
    returnValue = invokeOperation(invoker);  // always reached when @CachePut present
}
```

Because `@CachePut` is present with no `condition` attribute, `hasCachePut()` always returns `true`. The `else` branch is always taken, meaning **the DB query is executed on every single call**, regardless of whether the timestamp cache already holds the result. The `@Cacheable` annotation provides zero short-circuit benefit.

**Root cause — wrong key in index cache:**
The `@CachePut` specifies no `key` attribute, so Spring's default key generator uses the method parameter: `timestamp` (a `long`). The index cache (`CACHE_MANAGER_RECORD_FILE_INDEX`, `maximumSize=10000`, `expireAfterWrite=10m`) is also used by `findByIndex(long index)`, which caches by block index number. These are entirely different key spaces. Every `findByTimestamp(T)` call writes `(key=T, value=RecordFile)` into the index cache — entries that `findByIndex` will never retrieve — polluting the cache with useless entries.

**Exploit flow:**
1. Attacker sends a stream of `findByTimestamp(T_i)` requests with distinct timestamps `T_1 … T_N` (all within a valid record file range).
2. For each request: Spring detects `@CachePut` → invokes DB query → writes `(T_i, RecordFile)` to index cache.
3. After ~10,000 unique timestamps, the index cache (`maximumSize=10000`) is saturated with timestamp-keyed entries.
4. Legitimate `findByIndex(I)` calls now miss the index cache and fall through to the DB.
5. Combined DB pressure: up to 500 QPS (rate limit) of `findByTimestamp` queries + increased `findByIndex` DB fallback queries.

### Impact Explanation

- **DB overload:** The `@Cacheable` optimization is completely negated; every `findByTimestamp` call issues a `SELECT … WHERE consensusEnd >= ? ORDER BY consensusEnd ASC LIMIT 1` query. At the 500 RPS rate limit, this is 500 DB queries/second sustained.
- **Index cache pollution:** The index cache (10,000 entries, 10-minute TTL) fills with timestamp-keyed entries. Legitimate block-number lookups via `findByIndex` lose their cache entries, increasing DB load further.
- **No authentication required:** Any caller of the public JSON-RPC/REST endpoint that internally invokes `findByTimestamp` can trigger this.
- Severity matches the stated scope: griefing / infrastructure degradation with no direct economic damage to users.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond knowing the endpoint exists, and no on-chain transactions. Any attacker with network access can send HTTP requests at the 500 RPS rate limit. The timestamps need only be valid `long` values within the record file range; they do not need to correspond to real transactions. The attack is fully repeatable and automatable with a simple loop.

### Recommendation

1. **Remove the `@CachePut` or add a correct `key` expression.** If the intent is to pre-populate the index cache after a timestamp lookup, the key must be the block index of the returned `RecordFile`, not the input timestamp:
   ```java
   put = @CachePut(
       cacheNames = CACHE_NAME,
       cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
       key = "#result?.get()?.index",
       unless = "#result == null || !#result.isPresent()")
   ```
2. **Alternatively, remove `@CachePut` entirely** if cross-cache pre-population is not required, restoring the `@Cacheable` short-circuit behavior.
3. **Add an `unless` condition to `@CachePut`** to prevent writes when the result is empty.
4. Consider adding per-IP or per-key rate limiting at the `findByTimestamp` call site in addition to the global 500 RPS limit.

### Proof of Concept

```bash
# Generate 10,000 unique timestamps all within a known record file window
# (e.g., consensusEnd of a known block = 1700000000000000000)
# Send them as eth_getBlockByNumber or eth_call requests that internally
# resolve to findByTimestamp()

for i in $(seq 1 10000); do
  TIMESTAMP=$((1700000000000000000 - i))
  curl -s -X POST http://<mirror-node-web3>:8545 \
    -H 'Content-Type: application/json' \
    -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_getBlockByNumber\",\"params\":[\"0x$(printf '%x' $TIMESTAMP)\",false],\"id\":$i}" &
done
wait

# After this burst:
# 1. Index cache is saturated with 10,000 timestamp-keyed entries
# 2. findByIndex() calls now miss cache and hit DB
# 3. DB receives sustained query load from both findByTimestamp and findByIndex
```
