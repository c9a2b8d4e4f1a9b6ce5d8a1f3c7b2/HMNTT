### Title
Missing Rate Limit on GraphQL Middleware Enables Sustained Max-Complexity Query Exhaustion

### Summary
The GraphQL Traefik middleware stack defines only `inFlightReq: amount: 5` per IP as its sole throughput control, with no `rateLimit` directive — unlike the Rosetta service which adds `rateLimit: average: 10`. An unprivileged attacker can continuously cycle 5 concurrent maximum-complexity queries (complexity=200, depth=10) against `account(input: AccountInput!)`, each triggering multiple recursive DB lookups, with no temporal rate gate to interrupt the cycle. The `retry: attempts: 3` middleware further amplifies backend load when errors begin to occur.

### Finding Description

**Exact code path:**

`graphql/src/main/resources/graphql/query.graphqls` line 5 exposes the sole query entry point: [1](#0-0) 

`graphql/src/main/resources/graphql/account.graphqls` lines 9, 55, 67 define three recursive/relational fields on `Account` — `autoRenewAccount: Account`, `obtainer: Accountable`, and `stakedAccount: Account` — each resolved by a separate DB query: [2](#0-1) [3](#0-2) 

`GraphQlConfiguration.java` lines 43–44 set the complexity ceiling at 200 and depth at 10. `MaxQueryComplexityInstrumentation` in graphql-java rejects queries that *exceed* the limit; queries *at* the limit (complexity=200, depth=10) pass validation and execute fully, triggering all nested resolvers: [4](#0-3) 

`charts/hedera-mirror-graphql/values.yaml` lines 135–145 define the full middleware chain. It contains `inFlightReq: amount: 5` and `retry: attempts: 3` but **no `rateLimit`**: [5](#0-4) 

Compare with the Rosetta middleware at lines 149–163, which adds `rateLimit: average: 10`: [6](#0-5) 

The middleware CRD is rendered only when `global.middleware: true`; the default is `false`: [7](#0-6) [8](#0-7) 

This creates two attack surfaces:
- **Default deployment** (`global.middleware: false`): the ingress annotation still references the middleware CRD but the CRD is absent, meaning Traefik applies no middleware at all — zero rate control.
- **Hardened deployment** (`global.middleware: true`): `inFlightReq: amount: 5` limits concurrency but imposes no temporal rate gate; an attacker immediately re-submits each completed request, maintaining a continuous 5-query saturation loop.

There is no application-level throttling in the GraphQL module. The `ThrottleManagerImpl` / `ThrottleConfiguration` rate-limiting beans exist only in the `web3` module: [9](#0-8) 

**Root cause:** The failed assumption is that `inFlightReq: amount: 5` is sufficient to bound resource consumption. It bounds *concurrency* but not *rate*. Without a `rateLimit`, an attacker can sustain 5 concurrent expensive queries indefinitely, and the `retry: attempts: 3` middleware multiplies each failing request into up to 4 backend hits when the server begins returning errors.

### Impact Explanation

Each max-complexity query at depth=10 resolves `autoRenewAccount → autoRenewAccount → stakedAccount → ...` chains, issuing multiple sequential DB queries per request via `EntityService`. With 5 concurrent such queries and HikariCP's default pool of 10 connections (no explicit override in the GraphQL module), 5 concurrent queries each holding 2–3 connections can consume 50–100% of the pool. CPU is consumed by graphql-java field resolution, JPA/Hibernate query building, and result mapping for every nested level. The `statementTimeout: 10000` (10 s) means each query holds a connection for up to 10 seconds, keeping the pool saturated continuously. This directly causes the >30% resource increase threshold to be crossed and degrades or denies service to legitimate users. [10](#0-9) [11](#0-10) 

### Likelihood Explanation

No authentication or API key is required; the endpoint `/graphql/alpha` is publicly reachable. Crafting a query at exactly complexity=200 and depth=10 requires only reading the public schema (available via introspection or the repo). The attack is fully scriptable with `curl` or any HTTP client sending 5 parallel POST requests in a loop. The attacker needs a single IP (or can rotate IPs to bypass even the `inFlightReq` limit). The attack is repeatable indefinitely and requires no special knowledge beyond the public GraphQL schema.

### Recommendation

1. **Add `rateLimit` to the GraphQL middleware**, mirroring the Rosetta configuration:
   ```yaml
   - rateLimit:
       average: 10
       sourceCriterion:
         ipStrategy:
           depth: 1
   ```
   in `charts/hedera-mirror-graphql/values.yaml` lines 135–145.

2. **Add application-level throttling** in the GraphQL module analogous to `ThrottleManagerImpl` in `web3`, enforcing a per-IP or global requests-per-second cap via a `WebGraphQlInterceptor`.

3. **Lower the complexity ceiling** or assign higher per-field costs to recursive resolver fields (`autoRenewAccount`, `stakedAccount`, `obtainer`) using a custom `FieldComplexityCalculator` so that deeply nested queries are rejected before execution.

4. **Enable `global.middleware: true` by default** or document that the default-off state leaves the service completely unprotected at the ingress layer.

### Proof of Concept

```bash
# Craft a max-depth nested account query (complexity ~200, depth 10)
QUERY='{"query":"{ account(input:{entityId:{shard:0,realm:0,num:2}}) { autoRenewAccount { autoRenewAccount { autoRenewAccount { autoRenewAccount { autoRenewAccount { autoRenewAccount { autoRenewAccount { autoRenewAccount { autoRenewAccount { id memo deleted } } } } } } } } } } }"}'

TARGET="https://<mirror-node-host>/graphql/alpha"

# Send 5 concurrent requests in a tight loop (no rate limit stops re-submission)
for i in 1 2 3 4 5; do
  while true; do
    curl -s -X POST "$TARGET" \
      -H 'Content-Type: application/json' \
      -d "$QUERY" > /dev/null
  done &
done
wait
```

Expected result: CPU utilization and HikariCP active connection count climb above 30% of baseline within seconds. Prometheus alerts `GraphQLHighCPU` (>80%) and `GraphQLHighDBConnections` (>75%) fire within the configured `for: 5m` window. Legitimate requests begin timing out or receiving 503 responses from the circuit breaker once `ResponseCodeRatio(500, 600, 0, 600) > 0.25`.

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
```
