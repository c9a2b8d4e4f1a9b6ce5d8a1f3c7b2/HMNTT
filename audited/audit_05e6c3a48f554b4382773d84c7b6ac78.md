### Title
Unauthenticated HTTP Mirror Node Response Poisoning Causes Permanent Network Partition

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` fetches the Hiero node list over a plain, unauthenticated HTTP connection with no TLS enforcement, no response signature verification, and no IP allowlist validation. A network-adjacent unprivileged attacker who can intercept or spoof the HTTP response can return a crafted `nodesEnvelope` containing only unreachable `ServiceEndpoints`, causing the pinger to permanently build its entire network map from dead nodes. Because the client is constructed once at startup and never refreshed, the pinger is irreversibly partitioned from the real Hiero network for its entire process lifetime.

### Finding Description

**Code path:**

`pinger/sdk_client.go` line 18 calls `buildNetworkFromMirrorNodes()` exactly once at startup: [1](#0-0) 

`buildNetworkFromMirrorNodes()` constructs a plain `http.Client` with no TLS configuration and no certificate pinning: [2](#0-1) 

The URL is derived directly from `cfg.mirrorRest`, whose documented default example is `http://mirror-rest:5551` — plain HTTP: [3](#0-2) 

Inside `fetchMirrorNodeNetwork()`, the only validation performed on the response is:
1. HTTP status code in 200–299 range (line 90–93) — trivially satisfied by attacker.
2. Valid JSON decode into `nodesEnvelope` (line 96–98) — trivially satisfied.
3. `len(network) == 0` guard (line 127–129) — trivially bypassed by including one entry with a valid-looking but unreachable host/port. [4](#0-3) 

There is **no**:
- TLS/HTTPS enforcement
- Certificate pinning
- Cryptographic signature on the response
- Validation that returned IPs belong to a known-good set
- Periodic refresh of the network map after startup

The poisoned `network` map is passed directly to `hiero.ClientForNetwork(netmap)`: [5](#0-4) 

**Root cause:** Blind, unauthenticated trust of an HTTP response to construct the entire network topology, with no integrity guarantee and no recovery mechanism.

### Impact Explanation

The pinger's sole purpose is to submit periodic transfers to the Hiero network as a liveness/health signal. If the network map is poisoned with unreachable endpoints, every call to `submitWithRetry` fails permanently. The `/tmp/ready` readiness file is written before the ticker starts, so Kubernetes probes may not detect the failure immediately. The `/tmp/alive` heartbeat file continues to be touched regardless of transfer success, masking the partition from liveness probes. The result is a silently broken pinger that reports healthy while performing no actual network monitoring — a complete, undetected denial of the pinger's monitoring function.

### Likelihood Explanation

The attack requires only network-adjacency — no credentials, no special privileges. Realistic attacker positions include:

- **Same Kubernetes namespace/cluster**: Any compromised pod can perform ARP spoofing or DNS spoofing against the `mirror-rest` service name, since the default URL uses plain HTTP with an internal hostname.
- **DNS cache poisoning**: If the mirror REST hostname resolves via a poisonable DNS resolver, an off-path attacker can redirect the HTTP request.
- **BGP hijack**: For internet-facing deployments, a BGP-level attacker can redirect the HTTP traffic.

The plain-HTTP default (`http://mirror-rest:5551`) makes the network-adjacency vector the most realistic and lowest-privilege path. The attack is fully repeatable: if the pinger is restarted (e.g., by Kubernetes after a crash), the attacker can re-poison the next startup fetch.

### Recommendation

1. **Enforce HTTPS** for `cfg.mirrorRest` and reject any `http://` URL at config validation time.
2. **Pin the TLS certificate or CA** for the mirror REST endpoint, or at minimum enforce standard TLS certificate validation.
3. **Validate returned IPs** against a configurable allowlist of known Hiero node address ranges.
4. **Periodically refresh** the network map (e.g., on each tick or on consecutive transfer failures) so a poisoned startup does not cause permanent partition.
5. **Fail closed on empty or suspicious responses**: if all returned endpoints are RFC-1918/loopback/documentation addresses, treat the response as invalid.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (plain HTTP, internal hostname).
- Attacker has a pod in the same Kubernetes namespace (zero privileges required beyond pod execution).

**Steps:**

1. Attacker deploys a rogue HTTP server in the same namespace responding to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "192.0.2.1", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```
(`192.0.2.1` is a TEST-NET address guaranteed unreachable.)

2. Attacker poisons DNS for `mirror-rest` to resolve to the rogue server's ClusterIP (or performs ARP spoofing).

3. Pinger starts (or restarts). `buildNetworkFromMirrorNodes()` fetches from the rogue server. All validation checks pass: HTTP 200, valid JSON, `len(network) == 1 > 0`.

4. `hiero.ClientForNetwork({"192.0.2.1:50211": AccountID{0,0,3}})` is called. The client is permanently configured with a dead node.

5. Every subsequent `submitWithRetry` call times out or fails. The pinger emits no successful transfers for its entire lifetime. The real Hiero network receives no health signals from this pinger instance.
