### Title
Connection Pool Exhaustion via Untimed JDBC Socket Hang in `getByIdAndType()` During Network Partition

### Summary
`EntityServiceImpl.getByIdAndType()` calls `entityRepository.findById()` with no JDBC-level socket timeout configured in the graphql module. The only timeout protection is a PostgreSQL server-side `statement_timeout` (10 seconds), which is ineffective during a network partition because the server never receives or returns the query. An unprivileged attacker sending concurrent GraphQL requests during a partition can hold all HikariCP connections open for the full OS TCP timeout duration, exhausting the pool and denying service to all subsequent users.

### Finding Description
**Exact code path:**

`EntityServiceImpl.getByIdAndType()` at line 25:
```java
return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
``` [1](#0-0) 

This is invoked from `AccountController.account()` via the public GraphQL endpoint `/graphql/alpha`, reachable by any unauthenticated user: [2](#0-1) 

**Root cause — two-layer timeout gap:**

1. The graphql module's `EntityRepository` is a plain Spring Data JPA `CrudRepository` with no `@QueryHints`, no `@Lock`, and no `@Timeout` annotation: [3](#0-2) 

2. The `CommonConfiguration` builds the HikariCP `DataSource` from `spring.datasource.hikari` properties. No `socketTimeout` JDBC property is set anywhere in the graphql module's configuration: [4](#0-3) 

3. The only timeout protection is a PostgreSQL role-level `statement_timeout = 10000` ms applied via Helm chart SQL: [5](#0-4) 

**Why the existing check fails during a network partition:**
`statement_timeout` is enforced server-side by PostgreSQL. During a network partition between the graphql pod and the database, the TCP socket blocks at the kernel level — the JDBC driver never sends the query (or the response never arrives). PostgreSQL's server-side timer never starts. The JDBC socket hangs until the OS TCP keepalive/retransmit timeout fires, which under default Linux settings is 2–15 minutes. HikariCP's `connectionTimeout` (default 30 s) governs only pool acquisition, not an already-checked-out, in-flight socket.

The web3 module has an explicit per-request `StatementInspector` timeout and a `requestTimeout` property; the graphql module has neither: [6](#0-5) 

### Impact Explanation
HikariCP's default pool size is 10 connections. With 10 concurrent GraphQL `account` queries in flight during a partition, the entire pool is exhausted. All subsequent requests block waiting for a connection until HikariCP's `connectionTimeout` (30 s default) expires, at which point they receive a `SQLTimeoutException`. The graphql service is effectively unavailable for the duration of the partition plus the OS TCP timeout tail. Because the endpoint is public and unauthenticated, no credentials are required to trigger this condition.

### Likelihood Explanation
The precondition is a network partition between the graphql application pod and the PostgreSQL database. This can arise from:
- Transient cloud networking events (common in multi-AZ deployments)
- Database failover/switchover windows
- Misconfigured network policies or firewall rules

An attacker does not need to cause the partition — they only need to detect degraded DB response (e.g., by observing increased latency on the GraphQL endpoint) and then flood the endpoint with concurrent `entityId`-based account queries. The attack requires no authentication, no special knowledge beyond the public GraphQL schema, and is trivially scriptable. The graphql README explicitly documents the query format: [7](#0-6) 

### Recommendation
1. **Add a JDBC `socketTimeout`** to the graphql datasource JDBC URL (e.g., `?socketTimeout=15` for PostgreSQL JDBC, in seconds). This ensures the JDBC driver closes the socket if no data arrives within the timeout, regardless of server-side state.
2. **Set HikariCP `connectionTimeout`** explicitly and conservatively (e.g., 5 s) so pool exhaustion fails fast rather than queuing.
3. **Mirror the web3 pattern**: add a `StatementInspector` or Spring `@Transactional(timeout=...)` to bound query execution at the application layer, independent of the network path.
4. **Add a concurrency limit** on the GraphQL endpoint (e.g., via a Servlet filter or Spring WebMVC `HandlerInterceptor`) to cap simultaneous in-flight DB queries.

### Proof of Concept
```bash
# During a network partition (or simulated with iptables DROP on DB port):
# iptables -I OUTPUT -p tcp --dport 5432 -j DROP  (on the graphql pod)

# Send 15 concurrent account queries (exceeds default HikariCP pool of 10):
for i in $(seq 1 15); do
  curl -s -X POST http://<graphql-host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:'"$i"'}}){balance}}"}' &
done
wait

# All 15 requests hang. Pool is exhausted.
# New requests now fail with connection timeout after 30 s:
curl -X POST http://<graphql-host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:999}}){balance}}"}'
# Returns: 500 / Unable to acquire JDBC Connection
```

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
    }
