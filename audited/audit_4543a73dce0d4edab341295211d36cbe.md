### Title
Integer Overflow in `poll()` Limit Cast Enables Unbounded Database Query via Unprivileged gRPC Request

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` narrows a `long` subtraction result to `int` without bounds checking. An unprivileged caller can supply a `limit` value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`) via the gRPC `ConsensusTopicQuery`, causing the cast to produce a negative `int`. This negative value bypasses the `hasLimit()` guard in the repository layer, resulting in a query with no `setMaxResults` applied — returning every matching row from the database.

### Finding Description

**Code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `poll()`, lines 68–71:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```

**Root cause:** `filter.getLimit()` is a `long`. When it holds a value such as `Long.MAX_VALUE` (`9223372036854775807L`) and `context.getTotal().get()` is `0`, the subtraction yields `Long.MAX_VALUE`. The narrowing cast `(int)` retains only the lower 32 bits: `(int)(0x7FFFFFFFFFFFFFFF)` = `0xFFFFFFFF` = **`-1`**.

`Math.min(-1, maxPageSize)` = `-1`, so `pageSize = -1`.

The rebuilt filter is passed directly to the repository:

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

No Spring validation is triggered on this internally-constructed `newFilter`. In `TopicMessageRepositoryCustomImpl.findByFilter()` (lines 51–53):

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
```

`hasLimit()` is `return limit > 0;` — with `limit = -1` this is `false`, so `setMaxResults` is **never called**. The JPA query executes with no row cap and returns the entire matching result set.

**Why existing checks fail:**

- `TopicMessageFilter.limit` carries `@Min(0)` but **no `@Max`**, so `Long.MAX_VALUE` passes validation at the service boundary.
- The `@Valid` annotation on `TopicMessageService.subscribeTopic()` validates the user-supplied filter, not the internally rebuilt `newFilter`.
- `hasLimit()` is designed to distinguish "no limit" (`0`) from "has limit" (`> 0`), but a negative overflow value is indistinguishable from "no limit" by this predicate.

### Impact Explanation
Any authenticated (but otherwise unprivileged) gRPC client can trigger an unbounded `SELECT` against the `topic_message` table on every polling interval. For a high-volume topic this can return millions of rows per poll cycle, causing heap exhaustion or severe GC pressure on the mirror node's JVM, degrading or crashing the gRPC service for all subscribers. The polling loop repeats with `numRepeats = Long.MAX_VALUE` for throttled subscribers, so the unbounded query fires continuously until the connection times out (default 60 s).

### Likelihood Explanation
The exploit requires only a standard gRPC client and knowledge of any valid topic ID. No credentials, elevated roles, or special network access are needed. The `limit` field is a plain `uint64` in the protobuf wire format; setting it to `Long.MAX_VALUE` is a single-field change. The condition is 100% reproducible and requires no race or timing dependency.

### Recommendation
1. **Add an upper-bound constraint** on `TopicMessageFilter.limit`:
   ```java
   @Min(0)
   @Max(Integer.MAX_VALUE)   // or a domain-appropriate cap
   private long limit;
   ```
2. **Guard the cast explicitly** in `poll()`:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (remaining > Integer.MAX_VALUE || remaining < 0)
           ? Integer.MAX_VALUE
           : (int) remaining;
   ```
3. **Validate the rebuilt filter** before passing it to the repository, or assert `pageSize > 0` before building `newFilter`.

### Proof of Concept
```java
// Build a gRPC ConsensusTopicQuery with limit = Long.MAX_VALUE

    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
    .setLimit(Long.MAX_VALUE)   // passes @Min(0), overflows (int) cast to -1
    .build();
