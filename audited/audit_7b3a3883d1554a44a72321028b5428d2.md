### Title
Unauthenticated `/health/readiness` Endpoint Triggers Uncached Dual Internal API Calls Per Request, Enabling Sustained Resource Amplification

### Summary
The `checkNetworkStatus()` function in `rosetta/app/middleware/health.go` returns a closure that unconditionally executes two full HTTP round-trips to the local Rosetta server (`/network/list` then `/network/status`) on every invocation. Because no result caching is applied to the health check and no rate limiting protects the `/health/readiness` endpoint, any unprivileged external caller can flood the endpoint and force a 2× amplification of internal API calls — each of which likely involves database queries — sustaining elevated resource consumption indefinitely.

### Finding Description

**Exact code location**: `rosetta/app/middleware/health.go`, lines 80–101 (`checkNetworkStatus`) and lines 37–54 (`NewHealthController`).


**Root cause**: The `hellofresh/health-go v4` library supports a `WithCacheDuration` option that would memoize check results for a configurable TTL. It is never passed to `health.New()`. Consequently, `HandlerFunc` re-executes every registered check synchronously on every HTTP request. Each execution of the `checkNetworkStatus` closure makes two outbound HTTP calls back to `localhost:<port>` — `/network/list` and `/network/status` — which are full Rosetta API handlers backed by database queries (via `networkAPIService`, `addressBookEntryRepo`, `blockRepo`).

**Middleware stack** (`main.go` lines 217–219): `MetricsMiddleware → TracingMiddleware → CorsMiddleware`. None implement rate limiting or concurrency throttling on any path, including `/health/readiness`.

**Exploit flow**:
1. Attacker sends a continuous stream of `GET /health/readiness` requests (no credentials required).
2. Each request causes the `hellofresh/health-go` handler to invoke `checkNetworkStatus`'s closure.
3. The closure issues `NetworkList` → `NetworkStatus` to `localhost`, two full internal HTTP round-trips.
4. Those internal calls hit the live Rosetta handlers, executing DB queries against PostgreSQL.
5. With no caching and no rate limiting, N external requests produce 2N internal API calls and ≥2N DB queries.

**Why existing checks fail**:
- `Timeout: time.Second * 10` only caps per-check duration; it does not limit concurrency or request rate.
- `SkipOnErr: false` is irrelevant to caching.
- CORS middleware does not restrict request rate.
- No IP-based throttle, token bucket, or semaphore exists anywhere in the middleware chain. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Every `/health/readiness` request amplifies into two internal API calls with associated DB I/O. On a lightly-to-moderately loaded node, an attacker sustaining even ~5–10 req/s produces 10–20 additional DB-backed API calls per second continuously. Over 24 hours this is 864,000–1,728,000 extra internal requests. This can exhaust DB connection pool slots (default `maxConnections: 10` for the REST layer), inflate CPU usage, and degrade legitimate Rosetta API response times — meeting the ≥30% resource increase threshold without any brute-force volume. The attack requires zero privileges and zero authentication. [4](#0-3) 

### Likelihood Explanation
The endpoint is publicly reachable on the configured port with no authentication, no API key, and no rate limit. A single attacker with a basic HTTP client (e.g., `curl` in a loop, `ab`, `wrk`) can sustain the attack indefinitely. The attack is trivially repeatable, requires no special knowledge of the system beyond the port number, and produces no error that would cause the server to reject subsequent requests. [5](#0-4) [6](#0-5) 

### Recommendation
1. **Add result caching** to the health check by passing `health.WithCacheDuration(time.Second * 30)` (or a suitable TTL) to `health.New()` in `NewHealthController`. This amortizes the cost of `NetworkList` + `NetworkStatus` across all requests within the TTL window.
2. **Add rate limiting** on `/health/readiness` (and `/health/liveness`) using a token-bucket or fixed-window middleware (e.g., `golang.org/x/time/rate`) to cap the number of health evaluations per second regardless of caching.
3. **Restrict network exposure**: if `/health/readiness` is only needed by Kubernetes probes or internal load balancers, bind it to a separate internal-only port or restrict it via network policy. [1](#0-0)

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
	serverUrl := fmt.Sprintf("http://localhost:%d", port)
	cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
	rosettaClient := client.NewAPIClient(cfg)

	return func(ctx context.Context) (checkErr error) {
		networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
		if err != nil {
			log.Errorf("Readiness check, /network/list failed: %v", err)
			return err
		}

		network := networkList.NetworkIdentifiers[0]
		_, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{NetworkIdentifier: network})
		if err != nil {
			log.Errorf("Readiness check, /network/status failed: %v", err)
			return err
		}

		return
	}
}
```
