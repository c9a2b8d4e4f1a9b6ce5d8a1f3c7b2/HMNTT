### Title
Unbounded Prometheus Label Cardinality via Unmatched Route Paths in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates route resolution to the gorilla/mux-backed `RouteMatcher`. When a request path does not match any registered route, the weaveworks `middleware.Instrument` falls back to using the raw `r.RequestURI` as the `route` label value. Because all four Prometheus metrics (`requestBytesHistogram`, `requestDurationHistogram`, `requestInflightGauge`, `responseBytesHistogram`) use an unbounded `route` label, an unprivileged attacker can exhaust process memory by sending requests to arbitrarily many unique paths.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83:
```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

All four metrics declare `"route"` as a label with no cardinality bound:
- `requestBytesHistogram` — labels: `["method", "route"]`
- `requestDurationHistogram` — labels: `["method", "route", "status_code", "ws"]`
- `requestInflightGauge` — labels: `["method", "route"]`
- `responseBytesHistogram` — labels: `["method", "route"]`

**Root cause:**

The `weaveworks/common/middleware.Instrument.Wrap` implementation calls `i.RouteMatcher.MatchRoute(r)` to obtain the route template. The `RouteMatcher` is the gorilla/mux router produced by `server.NewRouter(...)` (coinbase/rosetta-sdk-go, which uses gorilla/mux v1.8.1 per `go.mod`). For any request path that does not match a registered route, gorilla/mux returns an empty string from `MatchRoute`. The weaveworks middleware then falls back to using `r.RequestURI` verbatim as the `route` label value. Each unique URI becomes a distinct label combination, allocating a new histogram/gauge time series in the Prometheus default registry — permanently, for the lifetime of the process.

**Middleware chain (rosetta/main.go lines 217–219):**
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```

The metrics middleware sits before the router, so every request — including 404s — is instrumented before routing occurs.

**Why existing checks fail:**

The Traefik-level rate limiting (`average: 10`, `inFlightReq: 5`) defined in `charts/hedera-mirror-rosetta/values.yaml` lines 157–161 is:
1. Optional — only applied when the Helm chart is deployed with Traefik; not enforced at the application layer.
2. Insufficient — even at 10 req/s, an attacker can create 864,000 unique label values per day. Each new `HistogramVec` label combination allocates a `Histogram` struct with 5 buckets plus `_count`/`_sum` series. At scale this exhausts heap memory.

There is no application-level path normalization, no allowlist of valid route prefixes before metrics recording, and no cardinality cap on any of the four registered metrics.

### Impact Explanation

Each unique `route` label value permanently allocates memory in the Prometheus default registry for the process lifetime. With 5 size buckets per histogram and 4 metrics vectors, each unique path creates approximately 20+ new time series. Sending ~50,000 unique paths (achievable in under 90 minutes at 10 req/s) can consume hundreds of MB of heap, leading to OOM kill of the rosetta process. This constitutes a non-network-based DoS (memory exhaustion) requiring no authentication, no valid request body, and no high concurrency — a single sequential client suffices.

### Likelihood Explanation

The rosetta API is a public HTTP endpoint (port 5700, exposed via ingress). No authentication is required to send arbitrary GET/POST requests. The attacker needs only an HTTP client and the ability to generate unique path strings. The attack is fully repeatable across process restarts (each restart resets the registry, but the attacker can immediately re-flood). The Traefik rate limit, when present, only slows the attack — it does not prevent it.

### Recommendation

1. **Normalize unmatched routes to a fixed label value** — wrap `RouteMatcher` to return a constant (e.g., `"unmatched"`) when gorilla/mux returns an empty string, preventing raw URIs from becoming label values.
2. **Apply a cardinality limiter** — use `prometheus.WrapRegistererWithPrefix` combined with a cardinality-capping registerer, or use `promauto` with explicit label allowlists.
3. **Move metrics recording after routing** — instrument only requests that match a registered route, discarding 404s from metric recording entirely.
4. **Enforce application-level rate limiting** — add a middleware before `MetricsMiddleware` that enforces per-IP request rate limits regardless of deployment topology.

### Proof of Concept

```bash
# Send 100,000 requests with unique paths to the rosetta server
# No authentication, no valid body required
for i in $(seq 1 100000); do
  curl -s -o /dev/null "http://<rosetta-host>:5700/exploit-path-$i" &
done
wait

# Observe memory growth in the rosetta process
# Each unique path creates new Prometheus time series permanently
# Monitor via: curl http://<rosetta-host>:5700/metrics | grep hiero_mirror_rosetta
# Distinct "route" label values will equal the number of unique paths sent
```

After sufficient unique paths, the rosetta process will be OOM-killed or become unresponsive due to heap exhaustion, denying service to legitimate Rosetta API consumers.
