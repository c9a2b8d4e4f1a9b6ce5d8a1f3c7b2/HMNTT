### Title
Missing Format Validation on `HIERO_MIRROR_PINGER_OPERATOR_ID` Causes Startup Crash via `log.Fatalf`

### Summary
`loadConfig()` only checks that `operatorID` is non-empty, performing no format validation. Any non-empty but malformed account ID string (e.g., `"not-valid"`) passes this check, is stored in `cfg.operatorID`, and is later passed to `hiero.AccountIDFromString` in `newClient()`. When that call fails, `main()` calls `log.Fatalf`, terminating the process.

### Finding Description

**Code path:**

1. `pinger/config.go:102–103` — only guard is an empty-string check: [1](#0-0) 

2. `pinger/sdk_client.go:35–38` — `hiero.AccountIDFromString` is called with the unvalidated string; any parse error is returned to the caller: [2](#0-1) 

3. `pinger/main.go:41–44` — the returned error triggers `log.Fatalf`, which calls `os.Exit(1)`: [3](#0-2) 

**Root cause:** `loadConfig()` treats any non-empty string as a valid account ID. Format validation (shard.realm.account) is deferred entirely to the SDK call inside `newClient()`, and errors there are fatal.

**Exploit flow:**
- Attacker sets `HIERO_MIRROR_PINGER_OPERATOR_ID=garbage` (any non-empty, non-`shard.realm.account` string).
- `loadConfig()` stores it without error.
- `newClient()` calls `hiero.AccountIDFromString("garbage")` → returns parse error.
- `main()` receives the error and calls `log.Fatalf("client error: %v", err)` → `os.Exit(1)`.

**Why existing checks are insufficient:** The only check at `config.go:102` is `cfg.operatorID == ""`. There is no regex, no structural validation, and no attempt to pre-parse the account ID inside `loadConfig()`.

### Impact Explanation
The pinger process crashes at startup and never becomes ready (the `/tmp/ready` file is never written). Any liveness/readiness probe will fail. This is a complete Denial of Service for the pinger component: no transfers are submitted, no heartbeat is written, and the service remains permanently unavailable as long as the malformed value persists.

### Likelihood Explanation
The precondition is write access to `HIERO_MIRROR_PINGER_OPERATOR_ID`. In Kubernetes deployments this value typically comes from a ConfigMap or environment injection. A user with `update` rights on the relevant ConfigMap (a common misconfiguration in shared namespaces) can trigger this with a single `kubectl edit configmap` or equivalent. The attack is trivially repeatable: the process will crash on every restart attempt until the value is corrected.

### Recommendation
Validate the account ID format inside `loadConfig()` immediately after the empty-string check, before returning `cfg`:

```go
if _, err := hiero.AccountIDFromString(cfg.operatorID); err != nil {
    return cfg, fmt.Errorf("invalid operator id %q: %w", cfg.operatorID, err)
}
```

Apply the same pattern to `cfg.toAccountID`. This converts a fatal startup crash into a clean, early configuration error that is surfaced at the same layer as all other config validation, and does not require `newClient()` to be the first place format errors are caught.

### Proof of Concept
```bash
# Set a malformed operator ID
export HIERO_MIRROR_PINGER_OPERATOR_ID="not-a-valid-account-id"
export HIERO_MIRROR_PINGER_OPERATOR_KEY="302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137"

# Run the pinger
./pinger
# Output:
# 2024/xx/xx xx:xx:xx pinger starting (version=development)
# 2024/xx/xx xx:xx:xx client error: invalid operator id: ...
# exit status 1

# Process exits immediately; /tmp/ready is never created.
# Any Kubernetes readiness probe will report the pod as not ready indefinitely.
```
