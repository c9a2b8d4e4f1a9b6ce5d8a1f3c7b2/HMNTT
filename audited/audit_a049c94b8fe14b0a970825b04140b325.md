### Title
Liveness Probe Spoofing via Unprivileged `/tmp/alive` File Creation

### Summary
The liveness healthcheck in `pinger/cmd/healthcheck/main.go` relies solely on `os.Stat("/tmp/alive")` to verify the pinger is alive, checking only file existence and mtime. Because `stat(2)` requires no permissions on the target file itself — only execute permission on the parent directory — any local user who can write to `/tmp` can create `/tmp/alive` with mode `0000` and a fresh mtime, causing the probe to report healthy even when the real pinger process is dead.

### Finding Description
**Code location:** `pinger/cmd/healthcheck/main.go`, lines 27–38.

```go
fi, err := os.Stat("/tmp/alive")   // line 27 — succeeds on mode-0000 files
if err != nil { ... os.Exit(1) }

age := time.Since(fi.ModTime())    // line 34
if age > 2*time.Minute { ... os.Exit(1) }
os.Exit(0)                         // passes if file exists and mtime < 2 min
```

The legitimate heartbeat writer (`pinger/main.go`, line 36) creates the file with mode `0644` every 15 seconds. The healthcheck never validates:
- File ownership (UID/GID)
- File permissions
- File content

`os.Stat` maps directly to the `stat(2)` syscall. POSIX specifies that `stat(2)` requires only execute (`x`) permission on each directory component of the path — it requires **zero** permissions on the file itself. `/tmp` is universally mode `1777` (world-writable + sticky), so any UID can create files there. A file created as `0000` is fully `stat`-able by all users.

### Impact Explanation
A dead or hung pinger process will not be restarted by Kubernetes because the liveness probe continues to return exit code 0. The pinger's purpose is to submit periodic Hedera network transactions; a silently dead pinger means those transactions stop while the operator believes the service is healthy. This masks a complete service outage at the monitoring layer.

### Likelihood Explanation
Preconditions are realistic in several common Kubernetes configurations:
1. **emptyDir tmpfs at `/tmp`** — pods frequently mount `emptyDir` at `/tmp` for scratch space; this volume is writable by all UIDs in the pod.
2. **Sidecar or init containers** — any co-located container running as a different UID (e.g., a log shipper, metrics agent, or init container) can pre-create or overwrite `/tmp/alive`.
3. **RCE in another pod component** — if an attacker achieves code execution in any container sharing the same `/tmp` volume, they can sustain the spoof indefinitely with a simple loop: `while true; do touch /tmp/alive; sleep 60; done`.

No elevated privileges are required. The attack is repeatable and silent.

### Recommendation
Replace the bare `os.Stat` check with ownership and content validation:

1. **Check file ownership** — after `os.Stat`, call `os.Lstat` and cast `fi.Sys()` to `*syscall.Stat_t`; assert `Uid == uint32(os.Getuid())`.
2. **Validate content** — have the heartbeat writer embed a secret token or PID; have the healthcheck read and verify it (`os.ReadFile`). A file with mode `0000` cannot be read, so a read-based check would fail on a spoofed `0000` file.
3. **Use a private directory** — write the heartbeat file to a directory created with mode `0700` owned by the pinger UID, preventing other users from creating files there at all.

### Proof of Concept
```bash
# From any UID with access to the pod's /tmp (e.g., a sidecar shell):

# 1. Kill or wait for the real pinger to die
kill $(pgrep pinger)

# 2. Create the spoof file with mode 0000 and current mtime
install -m 0000 /dev/null /tmp/alive
# or: touch /tmp/alive && chmod 0000 /tmp/alive

# 3. Verify os.Stat succeeds (stat(2) ignores file permission bits)
stat /tmp/alive   # succeeds, shows mtime = now

# 4. Run the healthcheck — exits 0 (healthy) despite pinger being dead
/healthcheck live
echo $?   # prints 0

# 5. Sustain the spoof to keep mtime fresh (< 2 minutes)
while true; do touch /tmp/alive; sleep 60; done
```
