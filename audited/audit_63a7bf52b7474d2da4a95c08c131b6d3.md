### Title
Unauthenticated EVM Address Input Forces Unconditional Extra DB Lookup in `getContractLogsById`

### Summary
Any unauthenticated caller can supply a valid 40-hex-character EVM address as the `:contractId` path parameter to `GET /contracts/:contractId/results/logs`. This unconditionally triggers an extra database query (`getContractIdByEvmAddress`) to resolve the EVM address to an encoded entity ID before the main log query executes. No caching exists for this resolution path, so every such request doubles the database load for that endpoint.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js:16` registers the route: [1](#0-0) 

`rest/controllers/contractController.js:788` unconditionally calls `computeContractIdFromString` with the raw user-supplied path parameter: [2](#0-1) 

`rest/service/contractService.js:468-476` — `computeContractIdFromString` calls `EntityId.computeContractIdPartsFromContractIdValue`, and if the input matches `isEvmAddressAlias` (i.e., is a 40-char hex string), it immediately calls `getContractIdByEvmAddress`: [3](#0-2) 

`rest/service/contractService.js:452-466` — `getContractIdByEvmAddress` executes a live DB query (`ContractService.contractIdByEvmAddressQuery`) with no caching: [4](#0-3) 

`rest/entityId.js:284-299` — `computeContractIdPartsFromContractIdValue` classifies any 40-char hex string as a `create2_evm_address`, which is the branch that triggers the DB lookup: [5](#0-4) 

**Root cause:** The `computeContractIdFromString` function has two branches: numeric/shard.realm.num IDs are resolved purely in memory, while EVM addresses always trigger a live DB query. There is no caching layer between the HTTP handler and `getContractIdByEvmAddress`. The `quickLru` cache in `entityId.js` covers only the `parseCached` path (numeric IDs), not EVM address → entity ID resolution. [6](#0-5) 

### Impact Explanation
Every request with a valid EVM address as `:contractId` forces one extra synchronous DB query before the main log query. An attacker sending a high volume of such requests (with arbitrary valid 40-char hex strings, including addresses that do not exist in the DB) causes the database to execute twice the number of queries compared to numeric-ID requests. Since the EVM address lookup hits the `entity` table filtered by `evm_address`, and non-existent addresses still complete the full index scan, this is a reliable amplification primitive. The impact is DB resource exhaustion / griefing with no economic damage to end users, but potential degradation of service for all consumers of the mirror node REST API.

### Likelihood Explanation
No authentication, API key, or privilege is required. The endpoint is publicly accessible. The attacker only needs to know the URL pattern (documented in the OpenAPI spec at `rest/api/v1/openapi.yml:760`) and supply any syntactically valid 40-char hex string. The attack is trivially scriptable, repeatable, and requires no prior knowledge of actual contract addresses. Rate limiting is absent at the route or controller layer. [7](#0-6) 

### Recommendation
1. **Cache EVM address → entity ID resolutions** using the existing `quickLru` infrastructure (or a dedicated short-TTL cache) so repeated lookups for the same EVM address do not hit the DB.
2. **Apply rate limiting** at the route or middleware level for endpoints that accept EVM address path parameters.
3. Optionally, **validate that the EVM address exists** before proceeding to the main log query and return a `404` immediately (the current code already does this via `NotFoundError` in `getContractIdByEvmAddress`, but the DB round-trip still occurs).

### Proof of Concept

```bash
# Single request forcing extra DB lookup (address need not exist)
curl "https://<mirror-node>/api/v1/contracts/0xDEADBEEFDEADBEEFDEADBEEFDEADBEEFDEADBEEF/results/logs"

# High-volume griefing loop (no auth required)
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/contracts/0x${ADDR}/results/logs" &
done
wait
```

Each iteration causes `getContractIdByEvmAddress` to execute a DB query against the `entity` table for a non-existent EVM address, returning a `404` after the full lookup completes. At scale this saturates DB connection pool capacity.

  async getContractIdByEvmAddress(evmAddressFilter) {
    const create2EvmAddress = evmAddressFilter.create2_evm_address;
    const rows = await super.getRows(ContractService.contractIdByEvmAddressQuery, [
      Buffer.from(create2EvmAddress, 'hex'),
    ]);
    if (rows.length === 0) {
      throw new NotFoundError(`No contract with the given evm address 0x${create2EvmAddress} has been found.`);
    }
    // since evm_address is not a unique index, it is important to make this check.
    if (rows.length > 1) {
      throw new Error(`More than one contract with the evm address 0x${create2EvmAddress} have been found.`);
    }

    return rows[0].id;
  }

const computeContractIdPartsFromContractIdValue = (contractId) => {

    realm: realm,
  };
