### Title
Global Gas Bucket Exhaustion via Concurrent Max-Gas ETH_CALL Requests Causes DoS for All Users

### Summary
The `ThrottleManagerImpl` uses a single global `gasLimitBucket` shared across all users with no per-IP or per-user isolation. Gas tokens are consumed upfront in `throttle()` and only restored after `doProcessCall()` completes in its `finally` block. An unprivileged attacker can continuously saturate the shared bucket with max-gas ETH_CALL requests, blocking all legitimate users — including fund-transfer callers — from passing the gas throttle check for the duration of the attack.

### Finding Description

**Code path:**

1. `ContractController.call()` calls `throttleManager.throttle(request)` which immediately deducts `scaleGas(request.getGas())` tokens from the global `gasLimitBucket` via `gasLimitBucket.tryConsume(...)`. [1](#0-0) 

2. The bucket is configured with capacity = `scaleGas(gasPerSecond)` = `7_500_000_000 / 10_000` = **750,000 tokens**, refilling at 750,000 tokens/second, using `SynchronizationStrategy.SYNCHRONIZED` (no distributed isolation). [2](#0-1) 

3. Gas is only returned to the bucket inside `restoreGasToBucket()`, which is called from the `finally` block of `doProcessCall()` — **after** EVM execution completes. [3](#0-2) 

4. `scaleGas()` maps gas values below 10,000 to 0 (no cost), and divides by 10,000 otherwise. Max gas of 15,000,000 → 1,500 tokens per request. [4](#0-3) 

**Root cause:** Gas is consumed atomically at request ingress but restored asynchronously after execution. The bucket is global with no per-user partitioning. The `requestsPerSecond` limit (500/sec, also global) does not prevent a single attacker from consuming the entire gas budget within that limit.

**Exploit flow:**
- Attacker sends 500 concurrent ETH_CALL requests each with `gas = 15,000,000` (max allowed by `evmProperties.getMaxGasLimit()`).
- Each request deducts 1,500 tokens: 500 × 1,500 = 750,000 tokens → bucket fully drained.
- While these requests execute (even for milliseconds), `gasLimitBucket.tryConsume(...)` returns `false` for every new request from any user.
- Legitimate fund-transfer callers receive `ThrottleException("Gas per second rate limit exceeded.")`.
- Attacker repeats at 500 req/sec to maintain depletion continuously.

**Why existing checks fail:**
- The `requestsPerSecond` limit (500/sec) is also global — the attacker consumes it, but this does not prevent gas bucket exhaustion; it merely sets the pace of the attack. [5](#0-4) 
- `validateContractMaxGasLimit` only enforces an upper bound on gas per request; it does not limit how many max-gas requests a single IP can submit. [6](#0-5) 
- `gasLimitRefundPercent = 100` (default) means gas is eventually restored, but the restore is deferred — the depletion window is real and exploitable. [7](#0-6) 

### Impact Explanation
All users share one gas bucket. A sustained attack blocks every ETH_CALL and ETH_ESTIMATE_GAS request — including fund-transfer simulation calls — from passing the gas throttle. Legitimate users receive HTTP 429-equivalent `ThrottleException` responses for the entire duration of the attack. Because the attacker operates within the published rate limit (500 req/sec), standard rate-limit defenses at the load-balancer level do not distinguish attack traffic from normal traffic.

### Likelihood Explanation
No privileges, API keys, or on-chain funds are required. The attacker needs only an HTTP client capable of sending 500 concurrent requests per second to the `/api/v1/contracts/call` endpoint. This is trivially achievable with commodity tooling (e.g., `wrk`, `hey`, `ab`). The attack is repeatable indefinitely and requires no state between iterations.

### Recommendation
1. **Per-IP gas sub-bucket or token reservation**: Introduce per-source-IP gas accounting so a single client cannot monopolize the global budget.
2. **Cap per-request gas contribution to the throttle**: Limit the gas tokens deducted per request to a fraction of the total bucket (e.g., 10%), regardless of the declared `gas` field.
3. **Separate read (ETH_CALL) and write (fund-transfer) gas pools**: Allocate distinct buckets for read-only calls and state-changing calls so read-only exhaustion cannot block fund transfers.
4. **Restore gas immediately on throttle failure**: If `tryConsume` fails (gas bucket already empty), no tokens are consumed — this is already correct. The gap is concurrent in-flight requests; address this with the per-IP approach above.
5. **Add IP-level rate limiting at the ingress layer** (e.g., nginx, API gateway) before requests reach the shared bucket.

### Proof of Concept

```bash
# Drain the global gas bucket with 500 concurrent max-gas ETH_CALL requests
# (within the 500 req/sec rate limit)

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{
      "to": "0x0000000000000000000000000000000000000001",
      "gas": 15000000,
      "data": "0x",
      "estimate": false
    }' &
done
wait

# Immediately after, a legitimate fund-transfer simulation call fails:
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to": "0x<victim_contract>",
    "gas": 50000,
    "data": "0x<transfer_calldata>",
    "estimate": false
  }'
# Expected response: 429 / ThrottleException "Gas per second rate limit exceeded."

# Repeat the 500-request flood every second to maintain continuous DoS.
```

The attacker's requests consume all 750,000 gas tokens from the `gasLimitBucket`. During EVM execution of those requests, the bucket is empty. Any concurrent legitimate call hits `gasLimitBucket.tryConsume(...) == false` at line 40 of `ThrottleManagerImpl` and is rejected. [8](#0-7)

    Bucket gasLimitBucket() {

        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();

                .addLimit(limit)
