### Title
Cache Pollution via Error Entry Accumulation in `CachedPreparsedDocumentProvider`

### Summary
`CachedPreparsedDocumentProvider` caches `PreparsedDocumentEntry` objects unconditionally, including entries that contain parse/validation errors. Because the cache has a fixed `maximumSize=1000`, an unprivileged attacker can flood it with 1000 unique malformed queries, evicting all legitimate cached documents and forcing every subsequent legitimate query to be re-parsed on every request.

### Finding Description

**Exact code path:**

`CachedPreparsedDocumentProvider.getDocumentAsync` (line 24–27) stores the result of `parseAndValidateFunction.apply(executionInput)` directly into the Caffeine `AsyncCache`, keyed by the raw query string, with no check on whether the returned `PreparsedDocumentEntry` contains errors:

```java
// CachedPreparsedDocumentProvider.java:24-27
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

The cache is configured in `CacheProperties.java:15`:
```
"expireAfterWrite=1h,maximumSize=1000,recordStats"
```

**Root cause:** The failed assumption is that only valid, reusable documents will be cached. In reality, `PreparsedDocumentEntry` wraps both valid documents and error results. The graphql-java library's own documentation notes that callers should avoid caching error entries. There is no `entry.hasErrors()` guard before the cache insertion.

**Exploit flow:**
1. Attacker sends 1000 HTTP POST requests to the GraphQL endpoint, each with a syntactically unique but malformed query (e.g., `{ __typename_1 }`, `{ __typename_2 }`, …, varying field names or adding junk tokens within the `maxTokens=1000` / `maxCharacters=10000` parser limits).
2. Each unique query string becomes a distinct cache key. The `parseAndValidateFunction` returns a `PreparsedDocumentEntry` with errors, which is stored in the cache.
3. Once `maximumSize=1000` is reached, Caffeine evicts entries using its W-TinyLFU policy. Legitimate queries that were previously cached are displaced.
4. All subsequent legitimate queries miss the cache and must be fully re-parsed and re-validated on every request for up to 1 hour (`expireAfterWrite=1h`), until the error entries naturally expire.
5. The attacker can sustain the attack by periodically sending new unique malformed queries to replace any legitimate entries that reclaim cache slots.

**Why existing checks are insufficient:**
- Parser limits (`maxCharacters=10000`, `maxTokens=1000`, `maxRuleDepth=100`) constrain the *size* of each individual query but do not limit the *number* of unique malformed queries an attacker can submit. Within those limits, there are effectively unlimited unique malformed query strings.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` only run post-parse; they do not prevent error entries from being cached.
- There is no rate limiting, authentication requirement, or error-entry filtering visible in the GraphQL configuration.

### Impact Explanation
Legitimate queries are evicted from the preparsed document cache and must be fully re-parsed and re-validated on every request. This increases CPU consumption and per-request latency for all legitimate users. Under sustained attack, the cache provides zero benefit to legitimate traffic, effectively degrading the service to its un-cached baseline performance. This is a non-network-based DoS: the service remains reachable but operates at degraded throughput and latency.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public GraphQL endpoint URL, and no sophisticated tooling — a simple script sending 1000 HTTP POST requests with unique query strings suffices. The 1000-entry cache limit is small enough that a single attacker with a basic script can fill it in seconds. The `expireAfterWrite=1h` TTL means error entries persist for up to an hour, requiring only periodic re-flooding to sustain the attack. This is highly repeatable and automatable.

### Recommendation
Filter out error entries before caching. In `CachedPreparsedDocumentProvider.getDocumentAsync`, check `entry.hasErrors()` and return the error entry without storing it:

```java
@Override
public CompletableFuture<PreparsedDocumentEntry> getDocumentAsync(
        ExecutionInput executionInput, Function<ExecutionInput, PreparsedDocumentEntry> parseAndValidateFunction) {
    return cache.get(executionInput.getQuery(), key -> {
        var entry = parseAndValidateFunction.apply(executionInput);
        if (entry.hasErrors()) {
            // Do not cache error entries; return directly without populating the cache
            // Remove the just-inserted key to avoid storing the error
            cache.synchronous().invalidate(key);
        }
        return entry;
    });
}
```
A cleaner alternative is to use `cache.synchronous().get(key, ...)` with an explicit conditional put, or to wrap the async loader to invalidate error entries immediately after computation. Additionally, consider adding per-IP rate limiting at the ingress/API gateway layer.

### Proof of Concept

```bash
#!/bin/bash
# Fill the 1000-entry preparsed document cache with unique error entries
ENDPOINT="http://<graphql-host>/graphql"
for i in $(seq 1 1000); do
  curl -s -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -d "{\"query\": \"{ invalidField_$i }\"}" &
done
wait

# Now send a legitimate query — it will miss the cache and be re-parsed every time
for i in $(seq 1 10); do
  time curl -s -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -d '{"query": "{ __typename }"}'
done
```

Observe that the legitimate `{ __typename }` query is re-parsed on every request (measurable via increased latency and `recordStats` cache miss counters) rather than being served from the preparsed cache. [1](#0-0) [2](#0-1) [3](#0-2)
