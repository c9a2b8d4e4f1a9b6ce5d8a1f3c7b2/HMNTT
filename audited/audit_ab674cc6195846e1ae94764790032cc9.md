### Title
Unauthenticated Repeated `/block` Requests Cause Unbounded DB Work via Uncached Correlated-Subquery Batching in `FindBetween()`

### Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` issues K sequential batch queries (each fetching up to 2000 rows) for a block containing 2000×K transactions, and each row triggers two correlated subqueries against `crypto_transfer` and `staking_reward_transfer`. No result cache exists for block transaction data, and no rate-limiting middleware is present, so any unauthenticated caller can repeatedly request the same historically large block and force the database to re-execute the full O(K × 4000) subquery workload on every request.

### Finding Description

**Code path:**

`POST /block` → `blockAPIService.Block()` (block_service.go:56) → `BaseService.FindBetween()` (base_service.go:54-59) → `transactionRepository.FindBetween()` (transaction.go:112-171).

**Batching loop** (transaction.go:124-147):
```go
for start <= end {
    err := db.Raw(selectTransactionsInTimestampRangeOrdered, ...).Limit(batchSize).Find(&transactionsBatch).Error
    ...
    if len(transactionsBatch) < batchSize { break }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```
For a block with N = 2000×K transactions, this loop executes K full DB queries.

**Correlated subqueries per row** (transaction.go:41-55, `selectTransactionsInTimestampRange`):
Each of the 2000 rows returned per batch triggers two correlated subqueries:
- `SELECT json_agg(...) FROM crypto_transfer WHERE consensus_timestamp = t.consensus_timestamp`
- `SELECT json_agg(...) FROM staking_reward_transfer WHERE consensus_timestamp = t.consensus_timestamp`

Total DB operations per request: **K × 2000 × 2 = 4000K correlated subquery executions**.

**No caching of results:** The `entityCache` in `blockAPIService` (block_service.go:22) caches only account aliases via `updateOperationAccountAlias()`, not the transaction query results from `FindBetween()`. Every `/block` request re-executes the full query chain from scratch.

**`maxTransactionsInBlock` does not help:** The truncation at block_service.go:61-67 happens *after* `FindBetween()` returns, so all K batches are already executed before any response trimming occurs.

**No rate limiting:** The middleware stack (rosetta/app/middleware/) contains only `health.go`, `metrics.go`, and `trace.go` — no rate-limiting or request-deduplication layer exists.

**No authentication:** The Rosetta API is a public, unauthenticated HTTP API by design.

### Impact Explanation
A single request to a block with 10,000 transactions (K=5) causes 20,000 correlated subquery executions against indexed but potentially large tables. An attacker sending N concurrent or sequential requests to the same block multiplies this linearly. Because historical block data is immutable and the query result is never cached, the DB performs identical full work on every request. This can saturate DB connection pools, increase query latency for all users, and degrade or deny service to legitimate callers — meeting the threshold of ≥30% resource increase with a small number of targeted requests against a known large block.

### Likelihood Explanation
Block indices and hashes are public information on the Hedera network. An attacker can trivially identify historically large blocks by querying the public mirror node REST API or block explorers. No credentials, special access, or brute force are required — only knowledge of a block index and the ability to send HTTP POST requests. The attack is fully repeatable and stateless from the attacker's perspective.

### Recommendation
1. **Add a response cache** keyed on `(consensusStart, consensusEnd)` for `FindBetween()` results, with a TTL appropriate for finalized (immutable) historical blocks.
2. **Rewrite the query** to use a single `JOIN` or `LEFT JOIN LATERAL` instead of correlated subqueries, eliminating the per-row subquery fan-out.
3. **Add rate limiting middleware** (e.g., token bucket per IP) to the Rosetta HTTP server, specifically for the `/block` endpoint.
4. **Add a concurrency limiter** (semaphore) on in-flight `FindBetween()` calls to bound simultaneous DB load regardless of request rate.

### Proof of Concept
```
# 1. Identify a large block (public info, e.g. via mirror node REST API)
curl https://<mirror-node>/api/v1/blocks?limit=1&order=desc

# 2. Find a block with many transactions (e.g., block index 12345678 with ~4000 txns)
# 3. Repeatedly POST to the Rosetta /block endpoint (no auth required):

for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-node>:5700/block \
    -H "Content-Type: application/json" \
    -d '{
      "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
      "block_identifier": {"index": 12345678}
    }' &
done
wait

# Each of the 50 concurrent requests triggers 2 batch queries × 2000 rows × 2 correlated
# subqueries = 8000 DB subquery executions, for a total of 400,000 subquery executions
# across the 50 concurrent requests, with zero caching or deduplication.
