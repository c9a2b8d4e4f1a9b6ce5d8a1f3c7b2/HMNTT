### Title
One-Shot `/tmp/ready` Creation with No Recreation Logic Enables Persistent Readiness Probe Failure

### Summary
`pinger/main.go` writes `/tmp/ready` exactly once during initialization with no mechanism to recreate it if deleted. Any process with write access to `/tmp` (and the ability to unlink the file) can delete it after creation, causing every subsequent Kubernetes readiness probe invocation to permanently return exit code 1, keeping the pod perpetually out of the ready state.

### Finding Description
**Exact code path:**

`pinger/main.go` line 47 — single, non-repeating write:
```go
if err := os.WriteFile("/tmp/ready", []byte("ok\n"), 0o644); err != nil {
    log.Fatalf("failed to create readiness file /tmp/ready: %v", err)
}
``` [1](#0-0) 

`pinger/cmd/healthcheck/main.go` lines 20–23 — existence-only check:
```go
if _, err := os.Stat("/tmp/ready"); err != nil {
    fmt.Fprintln(os.Stderr, "not ready")
    os.Exit(1)
}
``` [2](#0-1) 

**Root cause:** The design assumes `/tmp/ready` is immutable once written. There is no periodic re-touch, no recreation loop, and no file-integrity check. The file is created with mode `0o644`, which allows the owning user to delete it; if `/tmp` is mounted as a Kubernetes `emptyDir` volume (which does not set the sticky bit by default), any user with directory write permission can `unlink` the file regardless of ownership.

**Exploit flow:**
1. Attacker process (same container or sidecar sharing an `emptyDir` at `/tmp`) opens an inotify watch on `/tmp` for `IN_CREATE` events.
2. `pinger/main.go` reaches line 47 and calls `os.WriteFile("/tmp/ready", ...)`.
3. The inotify event fires; attacker calls `os.Remove("/tmp/ready")` (or `unlink(2)`) within microseconds.
4. `pinger/main.go` continues into its ticker loop — it never revisits the readiness file.
5. Every Kubernetes readiness probe execution of `healthcheck ready` calls `os.Stat("/tmp/ready")`, receives `ENOENT`, prints "not ready", and exits 1.
6. The pod is never marked Ready; Kubernetes removes it from all Service endpoints.

**Why existing checks are insufficient:** There are none. The liveness probe (`/tmp/alive`) is refreshed every 15 seconds via a goroutine (lines 28–39 of `pinger/main.go`), but the readiness file has no equivalent refresh. [3](#0-2) 

### Impact Explanation
The pod is permanently excluded from Kubernetes Service load-balancing. The pinger continues running internally (liveness probe passes), so Kubernetes does not restart it, making the failure self-sustaining. The pinger's function — submitting periodic Hiero network transfers to verify mirror-node health — silently stops being observable from the cluster's readiness perspective. Severity matches the stated classification: griefing / availability disruption with no direct economic damage.

### Likelihood Explanation
**Preconditions:**
- Write access to `/tmp` in the same filesystem namespace as the pinger (achievable via a sidecar container sharing an `emptyDir` volume, or via code execution within the container).
- `/tmp` must lack the sticky bit — standard for Kubernetes `emptyDir` volumes, which are plain `tmpfs` or host-directory mounts initialized without `chmod +t`.

The attack requires no privileges, no kernel exploits, and no cryptographic material. The inotify-based race is deterministic at the kernel level (the `IN_CREATE` event is delivered before the creating process returns from `openat(2)`), making the race reliably winnable. The attack is repeatable: even if the pinger is restarted, the attacker can re-arm the watch and win the race again.

### Recommendation
Replace the one-shot write with a periodic re-creation loop, mirroring the liveness heartbeat pattern already present in the codebase:

```go
go func() {
    t := time.NewTicker(10 * time.Second)
    defer t.Stop()
    for {
        select {
        case <-ctx.Done():
            return
        case <-t.C:
            _ = os.WriteFile("/tmp/ready", []byte("ok\n"), 0o644)
        }
    }
}()
```

Additionally, mount `/tmp` with the sticky bit (`chmod 1777`) in the container's entrypoint or Dockerfile, and consider using a dedicated directory (e.g., `/run/pinger/`) with restrictive permissions (`0700`) owned exclusively by the pinger's UID, preventing any other user from unlinking files within it.

### Proof of Concept
```bash
# In a sidecar or second shell within the same /tmp namespace:

# 1. Arm inotify watcher and delete on creation
python3 - <<'EOF'
import inotify.adapters, os
i = inotify.adapters.Inotify()
i.add_watch('/tmp')
for event in i.event_gen(yield_nones=False):
    (_, type_names, path, filename) = event
    if 'IN_CREATE' in type_names and filename == 'ready':
        os.unlink('/tmp/ready')
        print("Deleted /tmp/ready")
        break
EOF

# 2. Start (or restart) the pinger in another shell.
#    The watcher fires, deletes /tmp/ready immediately after creation.

# 3. Verify readiness probe fails:
/healthcheck ready
# stderr: "not ready"
# exit code: 1

# 4. Confirm pinger is still running (liveness passes):
/healthcheck live
# exit code: 0

# The pod stays alive but never becomes Ready.
```
