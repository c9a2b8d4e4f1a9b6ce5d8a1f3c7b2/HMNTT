### Title
Unbounded Timestamp Range in Historical Hook Storage Query Causes Expensive Full-Table Sort Before DISTINCT ON Deduplication

### Summary
An unprivileged external user can trigger the historical code path in `getHookStorageChange()` by supplying any timestamp parameter (e.g., `timestamp=gte:0`). Because no maximum timestamp span is enforced, the lower bound defaults to `0` and the upper bound to `Long.MAX_VALUE`, forcing PostgreSQL to sort every historical row for the targeted hook by `(key ASC, consensus_timestamp DESC)` before applying `DISTINCT ON (key)` deduplication. The `LIMIT 100` cap only restricts the output rows, not the work the database must perform to produce them, enabling repeated requests to spike DB CPU well above 30%.

### Finding Description

**Code path:**

`HooksController.getHookStorage()` accepts up to two `TimestampParameter` values with `@Size(max = 2)` but imposes no restriction on the span between them. [1](#0-0) 

`Bound.of()` is called with `primarySortField = false`, so the constructor skips the `adjustedLower > adjustedUpper` range-validation guard. [2](#0-1) 

When no upper timestamp is supplied, `adjustUpperBound()` returns `Long.MAX_VALUE`. [3](#0-2) 

`isHistorical()` returns `true` whenever the `Bound` is non-empty (i.e., any timestamp parameter was provided). [4](#0-3) 

`getPageRequest()` then builds a dual-sort `(key <order>, consensus_timestamp DESC)` for the historical branch. [5](#0-4) 

Spring Data JPA appends this sort as `ORDER BY key ASC, consensus_timestamp DESC` to the native query. PostgreSQL's `DISTINCT ON (key)` semantics require the entire qualifying result set to be sorted by `(key, consensus_timestamp DESC)` before the first row per key can be selected — the `LIMIT 100` is applied only after that sort completes. [6](#0-5) 

**Failed assumption:** The design assumes callers will supply a narrow timestamp window. No server-side enforcement of a maximum range exists anywhere in the request-processing chain.

### Impact Explanation
For a hook with a large history (e.g., a frequently-updated EVM hook), a single request with `timestamp=gte:0` causes the database to read and sort every row in `hook_storage_change` for that `(owner_id, hook_id)` pair — potentially millions of rows — before returning 100 results. Concurrent identical requests from multiple source IPs compound the effect, driving sustained DB CPU above 30% and degrading query latency for all other mirror-node consumers. No data is exfiltrated, but availability is directly threatened.

### Likelihood Explanation
The attack requires zero privileges: `ownerId` and `hookId` are public path parameters visible in any hook-listing response. The trigger is a single extra query parameter (`timestamp=gte:0`). The endpoint has no rate limiting in the rest-java service. [7](#0-6) 

The attack is trivially scriptable and repeatable.

### Recommendation
1. **Enforce a maximum timestamp span** in `hookStorageChangeRequest()` or inside `Bound.of()` when `primarySortField = false` for the timestamp dimension — reject or clamp requests whose `upper - lower` exceeds a configurable threshold (e.g., 7 days in nanoseconds).
2. **Add a composite index** on `(owner_id, hook_id, key, consensus_timestamp DESC)` so PostgreSQL can satisfy `DISTINCT ON (key)` via an index scan rather than a sort.
3. **Apply a DB-level statement timeout** for this query class to bound worst-case execution time.
4. **Add per-IP or per-account rate limiting** to the `/hooks/{hookId}/storage` endpoint in the rest-java filter chain.

### Proof of Concept

```
# Step 1 – discover a valid ownerId/hookId from the public listing endpoint
GET /api/v1/accounts/0.0.1001/hooks?limit=1

# Step 2 – trigger the unbounded historical sort (no auth required)
GET /api/v1/accounts/0.0.1001/hooks/1/storage?timestamp=gte:0&limit=100

# Step 3 – repeat in a loop from multiple clients
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/hooks/1/storage?timestamp=gte:0&limit=100" &
done
wait
```

Each concurrent request forces PostgreSQL to sort the full `hook_storage_change` history for hook `1` owned by `0.0.1001` before returning 100 rows. With a sufficiently large history, DB CPU climbs sharply; with 10–50 concurrent requests it exceeds the 30% threshold sustained over the 24-hour window.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
