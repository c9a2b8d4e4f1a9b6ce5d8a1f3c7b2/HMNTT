### Title
Missing Port Range Validation in `fetchMirrorNodeNetwork` Allows Malformed gRPC Addresses via Attacker-Controlled Mirror Endpoint

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` deserializes `port` from the mirror REST response into a plain Go `int` with no range validation beyond a zero-check. An attacker who can serve an arbitrary HTTP response from the configured mirror REST endpoint can supply port values such as `-1` or `99999`, which pass the only guard (`ep.Port == 0`), get formatted into syntactically invalid address strings, and are fed directly into the Hiero SDK network map — causing the pinger to be unable to dial any consensus node and permanently fail to submit transactions.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–124.

```
// line 33 — Port is a plain int, no bounds enforced at unmarshal time
type serviceEndpoint struct {
    DomainName  string `json:"domain_name"`
    IPAddressV4 string `json:"ip_address_v4"`
    Port        int    `json:"port"`          // accepts any JSON integer
}

// lines 113–124 — the only guard is == 0
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← only rejects zero; -1 / 99999 pass
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))  // ← malformed addr
    network[addr] = nodeAccountId
}
```

**Root cause:** `Port int` accepts any JSON integer. `json.Unmarshal` will happily decode `"port": -1` or `"port": 99999` into the field without error. The sole filter `ep.Port == 0` does not enforce the valid TCP port range `[1, 65535]`. `fmt.Sprintf("%d", ep.Port)` then serialises the raw integer (e.g. `"-1"`, `"99999"`) and `net.JoinHostPort` accepts any string as the port component, producing addresses like `"10.0.0.1:-1"`. These are inserted into the `network` map and returned to `newClient`, which passes them verbatim to `hiero.ClientForNetwork(netmap)`.

**Why the check fails:** The guard `ep.Port == 0` was written to skip "unset" endpoints. It does not constitute port-range validation and explicitly allows every non-zero integer, including all negative values and all values above 65535. [1](#0-0) [2](#0-1) 

### Impact Explanation

When the SDK client is initialised with a network map whose every entry has an invalid port string, every gRPC dial attempt fails immediately (the OS rejects the address before a TCP SYN is sent). The pinger can never submit a transaction. Because `buildNetworkFromMirrorNodes` returns a non-empty map (the length check at line 127 only fires when the map is empty), no error is surfaced at startup — the client is created successfully and then silently fails on every tick. This matches the stated scope: **total network shutdown** from the pinger's perspective, with no self-healing unless the process is restarted against a clean mirror endpoint. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

**Precondition:** The attacker must be able to serve an HTTP response to the pinger's mirror REST URL. This is realistic in several scenarios:

1. The `mirrorRest` URL is plain HTTP (no TLS) — a network-adjacent attacker (same LAN, cloud VPC, or BGP peer) can MITM the connection with no credentials.
2. The mirror node server itself is compromised or misconfigured.
3. In a Kubernetes/Docker deployment the internal service DNS name resolves to an attacker-controlled pod (e.g. via a compromised sidecar or DNS spoofing).

No credentials, no special privileges within the pinger system, and no prior knowledge of the operator key are required. The attack is fully repeatable: every time the pinger restarts and calls `buildNetworkFromMirrorNodes`, it re-fetches and re-poisons the network map. [5](#0-4) [6](#0-5) 

### Recommendation

Add an explicit port-range guard immediately after the existing zero-check:

```go
if host == "" || ep.Port <= 0 || ep.Port > 65535 {
    continue
}
```

This rejects negative values, zero, and values above the maximum valid TCP port in a single expression, with no performance cost. Optionally, log a warning when an out-of-range port is encountered so operators can detect a poisoned mirror response.

### Proof of Concept

1. Stand up an HTTP server that responds to `GET /api/v1/network/nodes` with:

```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "10.0.0.3", "port": -1 }
      ]
    }
  ],
  "links": { "next": null }
}
```

2. Start the pinger with:

```
HIERO_MIRROR_PINGER_NETWORK=other \
HIERO_MIRROR_PINGER_REST=http://<attacker-server> \
HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.2 \
HIERO_MIRROR_PINGER_OPERATOR_KEY=<any-valid-key> \
./pinger
```

3. Observe: `newClient` returns without error (the network map contains `"10.0.0.3:-1" → 0.0.3`). Every subsequent `submitWithRetry` call fails because the SDK cannot dial `10.0.0.3:-1`. The pinger logs `transfer failed` on every tick and never successfully submits a transaction.
