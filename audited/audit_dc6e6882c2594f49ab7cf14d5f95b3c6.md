### Title
DB Connection Pool Exhaustion via Unauthenticated Concurrent Requests to `getAccountCryptoAllowances` Using HTTP/2 Multiplexing

### Summary
The `getAccountCryptoAllowances()` handler in `rest/controllers/cryptoAllowanceController.js` performs two sequential database queries per request with no per-client rate limiting or concurrency control at either the application or ingress layer. The REST API's default DB connection pool is capped at 10 connections. An unauthenticated attacker can open a single HTTP/2 TCP connection and multiplex hundreds of concurrent requests, exhausting the pool and rendering the REST API unavailable to all users.

### Finding Description
**Code path:**

`rest/controllers/cryptoAllowanceController.js`, `getAccountCryptoAllowances()`, lines 76–98: [1](#0-0) 

Line 77 calls `EntityService.getEncodedId()`, which — when the path parameter is an alias or EVM address — issues a live DB query against the `entity` table: [2](#0-1) 

Line 80 then calls `CryptoAllowanceService.getAccountCryptoAllowances()`, issuing a second DB query against `crypto_allowance`: [3](#0-2) 

Each in-flight request holds a DB pool connection for the duration of both sequential queries.

**DB pool size:** The default `maxConnections` is **10**: [4](#0-3) [5](#0-4) 

**No application-level rate limiting:** A `grep` across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, `express-rate-limit`, and `throttle` returns zero hits in production code. The middleware stack exports no throttling component: [6](#0-5) 

**No ingress-level per-client limiting:** The Traefik middleware block for the REST chart contains only `circuitBreaker` and `retry` — no `inFlightReq` and no `rateLimit`: [7](#0-6) 

Compare to the Rosetta chart, which explicitly configures both `inFlightReq` (per-IP) and `rateLimit`: [8](#0-7) 

**HTTP/2 multiplexing bypass:** HTTP/2 allows an unlimited number of concurrent streams over a single TCP connection. Any IP-level connection-count control is irrelevant because all requests share one TCP socket. The GCP Gateway `maxRatePerEndpoint: 250` is a per-backend-pod aggregate, not a per-client control: [9](#0-8) 

**Root cause / failed assumption:** The design assumes that network-layer controls (IP rate limiting, connection limits) will bound per-client concurrency. HTTP/2 multiplexing invalidates this assumption entirely, and no such controls exist for this service anyway.

### Impact Explanation
With 10 pool connections and a `connectionTimeout` of 20 seconds, an attacker sending ≥10 concurrent alias-based requests saturates the pool. All subsequent requests from any user queue and time out after 20 seconds, returning errors. Because the REST API is the primary read interface for Hedera Mirror Node consumers (wallets, explorers, dApps), sustained pool exhaustion constitutes a full REST API denial of service. The `statementTimeout` of 20 seconds means each attacker request holds a connection for up to 20 seconds, giving the attacker a very favorable amplification ratio. [10](#0-9) 

### Likelihood Explanation
The attack requires zero authentication, zero special privileges, and only a standard HTTP/2 client (e.g., `curl --http2`, Python `httpx`, or `h2load`). The alias-based path parameter (`/api/v1/accounts/{alias}/allowances/crypto`) is publicly documented and forces the two-query code path. The attack is trivially repeatable and scriptable. No existing control detects or blocks it.

### Recommendation
1. **Add `inFlightReq` and `rateLimit` Traefik middleware** to `charts/hedera-mirror-rest/values.yaml`, mirroring the Rosetta configuration, with `sourceCriterion.ipStrategy` to enforce per-client limits.
2. **Increase `maxConnections`** from the default of 10 to a value commensurate with expected concurrency (e.g., 50–100), and set it explicitly in production configuration.
3. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or a semaphore on DB pool acquisition) so that a single client cannot hold more than N connections simultaneously.
4. **Cache `EntityService.getEncodedId()` results** for numeric/alias lookups to eliminate the first DB query on cache hits, reducing per-request DB pressure.

### Proof of Concept
```bash
# Install h2load (part of nghttp2)
# Send 200 concurrent requests over a single HTTP/2 connection using an alias-based path
h2load \
  -n 200 \
  -c 1 \
  -m 200 \
  "https://<mirror-node-host>/api/v1/accounts/AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ/allowances/crypto"
```

**Expected result:** After the first 10 requests acquire all pool connections, subsequent requests from all clients begin timing out with 500/503 errors or hang for up to 20 seconds (`connectionTimeout`). Monitor with:
```bash
watch -n1 'curl -s https://<mirror-node-host>/api/v1/accounts/0.0.1/allowances/crypto | head -c 200'
```
The endpoint becomes unresponsive for the duration of the attack, which can be sustained indefinitely by repeating the `h2load` invocation.
