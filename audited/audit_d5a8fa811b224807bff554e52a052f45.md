### Title
Shared Per-Filter Token Bucket Exhaustion Enables Targeted DoS Against Legitimate Users

### Summary
The per-filter `THROTTLE` bucket in `RequestProperties` is a single shared instance (one per configured filter), with no per-IP or per-user isolation. An unprivileged attacker who can craft requests matching a configured filter can continuously drain all tokens from that shared bucket, causing every subsequent legitimate request matching the same filter to receive a `ThrottleException` — while the attacker stays well within the global rate limit.

### Finding Description

**Code path:**

- `ThrottleManagerImpl.java` lines 44–48: iterates all configured `RequestProperties` filters and calls `action()` for every matching request.
- `ThrottleManagerImpl.java` lines 70–73: the `THROTTLE` branch calls `filter.getBucket().tryConsume(1)` on the single shared bucket instance.
- `RequestProperties.java` lines 41–42: `@Getter(lazy = true)` — Lombok generates double-checked locking, so the bucket is initialized exactly once and then reused for every request that matches this filter, for the lifetime of the application.
- `RequestProperties.java` lines 63–68: `createBucket()` builds a bucket with `capacity = rate` (default 100) and a greedy refill of `rate` tokens per second.

**Root cause:** The bucket object is a singleton per `RequestProperties` instance. All concurrent HTTP requests that match the filter share the same token pool. There is no per-source-IP, per-session, or per-user sub-bucket. The lazy initialization is thread-safe (volatile + synchronized), so there is no initialization race — the real problem is that once initialized, the single bucket is the only gate for all callers.

**Failed assumption:** The design assumes that the aggregate request rate across all callers will stay within `rate` tokens/second. It does not account for a single hostile caller consuming the entire token budget, starving all other callers.

**Why existing checks are insufficient:**

- The global `rateLimitBucket` (default 500 req/s, `ThrottleProperties.requestsPerSecond`) is checked first (line 38), but it is a separate, larger bucket. An attacker can exhaust the per-filter bucket (100 tokens) while consuming only 100 of the 500 global tokens — leaving the global check green and the per-filter bucket empty.
- There is no IP-based or authenticated-user-based sub-limiting anywhere in the throttle stack.

### Impact Explanation
Any operator-configured `THROTTLE` filter becomes a denial-of-service vector. An attacker who identifies (or guesses) a filter predicate — e.g., a specific `to` address, method selector, or gas range — can continuously saturate that filter's bucket. All legitimate callers whose requests match the same predicate receive HTTP 429 / `ThrottleException("Requests per second rate limit exceeded")` for as long as the attacker sustains the flood. Because the bucket refills at only `rate` tokens/second (default 100), even a modest sustained attack of 100 req/s is sufficient to keep the bucket perpetually empty.

### Likelihood Explanation
No authentication or special privilege is required. The attacker only needs network access to the JSON-RPC endpoint and knowledge of (or ability to brute-force) a filter predicate. The attack is trivially repeatable with any HTTP load-generation tool (e.g., `wrk`, `hey`, `curl` in a loop). The default `rate = 100` means 100 requests per second — well within reach of a single machine. The attack is also stealthy: the attacker's own requests are throttled at the same rate, but they can accept that cost while denying service to all others.

### Recommendation
1. **Per-source rate limiting:** Replace or augment the single shared bucket with a per-IP (or per-authenticated-caller) bucket map, e.g., using Bucket4j's `ProxyManager` backed by a `ConcurrentHashMap` keyed on the client IP extracted from the servlet request.
2. **Fair-share enforcement:** Apply a per-caller sub-limit so no single source can consume more than a configurable fraction of the filter's total budget.
3. **Burst protection:** Set `initialTokens` lower than `capacity` to prevent a cold-start burst from a single caller immediately draining the bucket.

### Proof of Concept

**Preconditions:**
- A filter is configured with `action: THROTTLE` and `rate: 100` (the default).
- The filter predicate matches, e.g., all calls to a specific contract address.

**Steps:**
```bash
# Terminal 1 – attacker: sustain 100 req/s to the matching endpoint
while true; do
  for i in $(seq 1 100); do
    curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d '{"to":"0x<target>","gas":50000,"data":"0x"}' &
  done
  wait
done

# Terminal 2 – legitimate user: single request per second
while true; do
  curl -v -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x<target>","gas":50000,"data":"0x"}'
  sleep 1
done
```

**Expected result:** Terminal 2 consistently receives `429 Too Many Requests` / `"Requests per second rate limit exceeded"` because the attacker in Terminal 1 keeps the shared per-filter bucket at zero tokens. The global `rateLimitBucket` is not exhausted (attacker uses 100/500 of global budget), so the global check passes and the per-filter check is reached — and fails — for every legitimate request.
