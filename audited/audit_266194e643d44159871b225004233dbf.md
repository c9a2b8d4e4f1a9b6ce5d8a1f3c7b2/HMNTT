### Title
Unbounded Concurrent Request Amplification via ShallowEtagHeaderFilter Causes Heap and Thread Pool Exhaustion in rest-java

### Summary
The `etagFilter()` bean in `RestJavaConfiguration.java` registers Spring's `ShallowEtagHeaderFilter` across all `/api/*` endpoints with no rate limiting, no concurrency cap, and no per-IP throttle. Because `ShallowEtagHeaderFilter` wraps every response in a `ContentCachingResponseWrapper` that buffers the full response body in a `ByteArrayOutputStream` before computing the ETag hash, an unprivileged attacker who floods the service with concurrent requests that each trigger a slow database query will simultaneously hold open hundreds of in-memory response buffers while exhausting the Tomcat thread pool, degrading or crashing the node.

### Finding Description
**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, lines 42–46:
```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
```

**Root cause:**
`ShallowEtagHeaderFilter.doFilterInternal()` wraps the `HttpServletResponse` in a `ContentCachingResponseWrapper` and calls `filterChain.doFilter(request, wrappedResponse)`. The entire response body is written into an internal `ByteArrayOutputStream` before the filter computes the MD5 ETag and flushes to the real response. This means every in-flight request holds a heap-allocated buffer for the full duration of request processing — including the time spent waiting for a database connection from HikariCP and the time the database query runs.

**No rate limiting in rest-java:**
The `hiero.mirror.restJava` configuration table (docs/configuration.md lines 623–636) exposes no `requestsPerSecond`, throttle, or concurrency property. The bucket4j-based `ThrottleConfiguration` and `ThrottleManagerImpl` exist exclusively in the `web3` module and are not wired into rest-java. `WebMvcConfiguration.java` adds only argument resolvers and formatters — no request limiting. `LoggingFilter.java` only logs.

**DB statement timeout is 10 seconds:**
`hiero.mirror.restJava.db.statementTimeout` defaults to 10,000 ms (docs/configuration.md line 630). The Helm chart sets the PostgreSQL-level `statement_timeout` to 20,000 ms for the `restJavaUsername` (charts/hedera-mirror/templates/secret-passwords.yaml line 125). During those 10–20 seconds, each thread holds its `ContentCachingResponseWrapper` buffer.

**Exploit flow:**
1. Attacker opens N concurrent HTTP connections (N ≥ Tomcat's default max-threads of 200) to any `/api/v1/*` endpoint (e.g., `GET /api/v1/network/nodes?limit=100`).
2. Each request passes through `ShallowEtagHeaderFilter`, which immediately wraps the response in a `ContentCachingResponseWrapper`.
3. Each request blocks waiting for a HikariCP connection (default pool size ~10) or executing a slow query (up to 10 s).
4. All 200 Tomcat threads are occupied; new connections queue and eventually time out.
5. As queries complete and responses are serialized (up to 100 records × response size per page), each `ByteArrayOutputStream` fills with the full JSON payload. With 200 concurrent large responses, heap pressure accumulates.
6. The node becomes unresponsive to legitimate traffic.

### Impact Explanation
Thread pool exhaustion renders the rest-java service completely unavailable to all clients, including legitimate mirror node consumers and monitoring infrastructure. Because rest-java serves network-level data (nodes, fees, supply, NFT allowances), sustained unavailability across ≥30% of deployed nodes constitutes a medium-severity network processing disruption. The `ContentCachingResponseWrapper` heap amplification compounds the thread exhaustion with GC pressure, potentially triggering OOM kills.

### Likelihood Explanation
No authentication or API key is required to reach `/api/*` endpoints. A single attacker with a modest number of persistent HTTP connections (200–500) and knowledge of any slow query endpoint can trigger this. The attack is trivially repeatable, requires no exploit code, and is not detectable until the thread pool is already saturated. The 10-second statement timeout means each "slot" is occupied for a long window, making the attack highly efficient.

### Recommendation
1. **Add rate limiting to rest-java**: Introduce a bucket4j or Resilience4j `RateLimiter` filter in `RestJavaConfiguration` analogous to the `ThrottleConfiguration` in the web3 module, enforcing a per-IP or global requests-per-second cap before the ETag filter runs.
2. **Reduce DB statement timeout**: Lower `hiero.mirror.restJava.db.statementTimeout` to 3,000–5,000 ms to reduce the window each thread holds a buffer.
3. **Cap Tomcat threads and add a request queue limit**: Configure `server.tomcat.max-threads` and `server.tomcat.accept-count` to bound the number of queued connections.
4. **Consider disabling or scoping the ETag filter**: If ETag support is not required for all `/api/*` endpoints, restrict the URL pattern or disable the filter for high-volume endpoints.

### Proof of Concept
```bash
# Send 250 concurrent slow requests to exhaust the thread pool
# (replace HOST with the rest-java service address)
for i in $(seq 1 250); do
  curl -s "http://HOST/api/v1/network/nodes?limit=100" &
done
wait

# Verify service is unresponsive to new requests
curl -v "http://HOST/api/v1/network/nodes" --max-time 5
# Expected: connection timeout or HTTP 503
```

The attack works because each of the 250 goroutines holds a Tomcat thread blocked on a DB query for up to 10 seconds, with `ShallowEtagHeaderFilter` holding a `ContentCachingResponseWrapper` buffer in heap for each, while no rate-limiting filter exists in the rest-java filter chain to reject excess requests early.
