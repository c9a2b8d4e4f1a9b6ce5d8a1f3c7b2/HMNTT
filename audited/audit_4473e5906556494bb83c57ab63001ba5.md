### Title
Unbounded `crypto_transfer` Aggregation in `AccountBalance` Enables Unauthenticated Resource Exhaustion

### Summary
The `AccountBalance()` handler in `rosetta/app/services/account_service.go` accepts unauthenticated requests and unconditionally calls `accountRepo.RetrieveBalanceAtBlock()`, which executes an unbounded `SUM()` aggregation over the `crypto_transfer` table via `getBalanceChange()`. No application-level rate limiting, caching, or query result cap exists. An attacker can repeatedly query a high-volume account at a recent block, forcing the database to aggregate an arbitrarily large number of transfer rows on every request, driving DB CPU and I/O well above 30% of baseline.

### Finding Description

**Code path:**

`AccountBalance()` (`account_service.go:41-79`) → `RetrieveBalanceAtBlock()` (`persistence/account.go:161-223`) → `getBalanceChange()` (`persistence/account.go:313-338`) → executes `balanceChangeBetween` SQL (`persistence/account.go:21-29`):

```sql
select coalesce((
  select sum(amount) from crypto_transfer
  where
    consensus_timestamp > @start and
    consensus_timestamp <= @end and
    entity_id = @account_id and
    (errata is null or errata <> 'DELETE')
), 0) as value
```

`@start` is `snapshotTimestamp` (the last balance snapshot for the account) and `@end` is `block.ConsensusEndNanos` (attacker-controlled via `request.BlockIdentifier`). There is **no LIMIT, no pagination, and no result cache**. The query must scan and aggregate every `crypto_transfer` row for the account in that window.

**Root cause / failed assumption:** The design assumes the window between the last snapshot and the requested block is small. In practice, for high-volume accounts (exchanges, treasury, DeFi contracts) this window can contain millions of rows. The application makes no attempt to bound, cache, or throttle this aggregation.

**Why existing checks fail:**

The only rate-limiting protection is Traefik middleware defined in `charts/hedera-mirror-rosetta/values.yaml:149-166`:
```yaml
middleware:
  - inFlightReq:
      amount: 5          # per source IP
  - rateLimit:
      average: 10        # per requestHost
  - retry:
      attempts: 3        # multiplies DB load 3×
```
This middleware is **conditionally applied** only when `global.middleware` is `true` (`middleware.yaml:3`). It is **not enforced at the application layer**. Even when deployed, `average: 10` req/s per host still allows 10 expensive aggregation queries per second, and the `retry: attempts: 3` middleware actually **triples** the DB load per logical request. There is no application-level caching, no query timeout specific to this path, and no deduplication of identical in-flight requests.

### Impact Explanation
Each request for a high-volume account (e.g., treasury `0.0.2`, a major exchange account, or any account with dense transfer history) forces a full sequential scan + aggregation over potentially millions of `crypto_transfer` rows. At 10 req/s (the configured rate limit, when enabled), with each query scanning e.g. 500k rows, the DB is performing 5M row reads per second continuously. This causes sustained CPU and I/O saturation on the database, degrading all other mirror node services (REST API, gRPC, web3) that share the same DB. The 30% resource increase threshold is easily exceeded with a single attacker IP.

### Likelihood Explanation
No special privileges are required. The `/account/balance` endpoint is publicly accessible per the Rosetta API specification. The attacker only needs to know (or enumerate) a high-volume account ID — treasury account `0.0.2` is always a valid target and is guaranteed to have the densest transfer history. The attack is trivially scriptable with `curl` or any HTTP client. The Traefik middleware is an optional deployment concern and is absent in bare/non-Helm deployments.

### Recommendation
1. **Application-level result caching**: Cache `RetrieveBalanceAtBlock()` results keyed on `(accountId, consensusEnd)` with a short TTL (e.g., 5–30 seconds). Identical requests for the same account+block should hit the cache, not the DB.
2. **Query timeout**: Apply a per-query DB deadline (the `ctx` is already threaded through but the DB client should enforce a short timeout, e.g., 5s, on the aggregation query).
3. **Enforce middleware unconditionally**: Move rate limiting and in-flight request caps into the application layer (e.g., Go middleware) rather than relying on optional Traefik configuration.
4. **Remove the `retry` middleware** or ensure it does not retry on 5xx responses caused by expensive queries, as it currently multiplies DB load 3× per logical request.

### Proof of Concept

```bash
# Target: treasury account 0.0.2 (guaranteed high transfer volume)
# Request balance at the latest block (maximizes the snapshot-to-block window)

ROSETTA_URL="http://<rosetta-host>:5700"
NETWORK='{"blockchain":"Hedera","network":"mainnet"}'
ACCOUNT='{"address":"0.0.2"}'

# Fire 50 concurrent requests (bypasses per-IP inFlightReq=5 with 10 IPs)
for i in $(seq 1 50); do
  curl -s -X POST "$ROSETTA_URL/account/balance" \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":$NETWORK,\"account_identifier\":$ACCOUNT}" &
done
wait
```

Each request triggers `SELECT SUM(amount) FROM crypto_transfer WHERE entity_id = 2 AND consensus_timestamp > <last_snapshot> AND consensus_timestamp <= <latest_block>` — an unbounded aggregation over the densest account in the network. Monitor DB CPU with `pg_stat_activity` or `top` to observe the spike exceeding 30% above the 24-hour baseline.
