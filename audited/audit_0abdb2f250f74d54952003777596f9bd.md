### Title
Unauthenticated DoS via Unbounded Database Queries in `getEntityIdFromEvmAddress()`

### Summary
`getEntityIdFromEvmAddress()` in `rest/service/entityService.js` executes a raw database query (`entityFromEvmAddressQuery`) on every invocation with no caching layer for any result — including misses for non-existent addresses. The Node.js REST service has no application-level rate limiting. An unprivileged attacker can flood the service with unique, syntactically valid but non-existent EVM addresses, exhausting the database connection pool (default: 10 connections) and denying service to all legitimate users.

### Finding Description
**Exact code path:**

`getEntityIdFromEvmAddress()` at `rest/service/entityService.js` lines 90–104 unconditionally calls `this.getRows(EntityService.entityFromEvmAddressQuery, [...])` for every invocation:

```js
// rest/service/entityService.js:22-25
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where ${Entity.DELETED} <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;

// rest/service/entityService.js:90-104
async getEntityIdFromEvmAddress(entityId, requireResult = true) {
  const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
    [Buffer.from(entityId.evmAddress, 'hex')]);
  if (rows.length === 0) {
    if (requireResult) { throw new NotFoundError(); }
    return null;
  }
  ...
}
``` [1](#0-0) [2](#0-1) 

**Root cause:** There is no in-process or distributed cache wrapping this function. Every call — including for non-existent addresses — issues a live SQL query. The function is called from `getEncodedId()` whenever the parsed `EntityId` has a non-null `evmAddress`: [3](#0-2) 

**Why existing checks fail:**

1. **Response cache (Redis):** Disabled by default (`hiero.mirror.rest.cache.response.enabled = false`). Even when enabled, `responseCacheUpdateHandler` only stores responses where `httpStatusCodes.isSuccess(res.statusCode)` is true — 404 responses from non-existent EVM addresses are never cached. [4](#0-3) 

2. **No application-level rate limiting:** The REST middleware stack (`rest/middleware/index.js`) exports `authHandler`, `httpErrorHandler`, `openapiHandler`, `requestHandler`, `requestNormalizer`, `responseCacheHandler`, `responseHandler` — none implement rate limiting. The `grep` for `rateLimit` in `rest/**/*.js` returns only a test utility file. [5](#0-4) 

3. **DB pool:** The default `maxConnections` is 10. This is a shared resource across all concurrent requests — it provides a natural ceiling but not protection; saturating it denies service to all users. [6](#0-5) [7](#0-6) 

4. **Infrastructure rate limiting (Traefik/GCP):** Present only in the Rosetta chart values and optional GCP gateway config — not enforced by the REST application itself and not enabled by default for the REST service. [8](#0-7) 

### Impact Explanation
An attacker can exhaust the 10-connection database pool by sending concurrent requests with unique, syntactically valid 40-hex-character EVM addresses that do not exist in the database. Each request holds a DB connection for the duration of the query (up to `statementTimeout`, default 20 seconds). With 10 or more concurrent attack requests, all legitimate REST API users experience connection timeouts or errors. The attack is global — it degrades every endpoint that shares the pool, not just EVM address lookups.

### Likelihood Explanation
No authentication, no API key, no application-level rate limit is required. Any internet-accessible deployment is vulnerable. The attacker needs only an HTTP client and the ability to generate random 40-character hex strings. The attack is trivially scriptable, repeatable, and cheap to sustain. The only mitigations are optional infrastructure-layer controls (Traefik, GCP gateway) that are not enforced by default in the REST application.

### Recommendation
1. **Add a negative-result cache** inside `getEntityIdFromEvmAddress()` (e.g., using the existing `entityId` cache infrastructure documented at `hiero.mirror.rest.cache.entityId.*`) so repeated lookups for the same non-existent address do not hit the database.
2. **Add application-level rate limiting** to the Node.js REST middleware stack (e.g., `express-rate-limit` or similar), keyed per IP, for all endpoints that trigger EVM address DB lookups.
3. **Cache 404 results** in the response cache for a short TTL (e.g., 5–30 seconds) to prevent repeated identical miss queries.
4. **Enforce infrastructure rate limiting** (Traefik `rateLimit` / GCP `maxRatePerEndpoint`) as a mandatory default, not an optional deployment concern.

### Proof of Concept
```bash
# Generate 50 concurrent requests with unique random non-existent EVM addresses
for i in $(seq 1 50); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/accounts/0x${ADDR}" &
done
wait
# Result: DB connection pool (max 10) saturated; legitimate requests receive
# connection timeout errors or 503 responses for the duration of the attack.
```
