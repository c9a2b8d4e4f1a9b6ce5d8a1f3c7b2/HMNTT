### Title
Unbounded Concurrent Subscriptions via `subscribeTopic()` Enable Resource Exhaustion DoS Against gRPC Topic Listeners

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no global cap on concurrent subscriptions and no per-IP or per-client connection limit at the application layer. The only guard — `maxConcurrentCallsPerConnection = 5` — is a per-TCP-connection limit trivially bypassed by opening multiple connections. An unprivileged attacker can flood the server with thousands of long-lived subscriptions, exhausting the bounded elastic scheduler and database connection pool, causing legitimate subscribers to miss or delay receiving topic messages.

### Finding Description

**Code location:** `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()`, lines 59–92.

`subscribeTopic()` unconditionally creates a new `TopicContext`, schedules a `safetyCheck` Flux on `Schedulers.boundedElastic()`, and registers a live listener — all per subscription, with no enforcement of any global or per-client cap:

```java
// Line 48 — only a metric gauge, never checked as a limit
private final AtomicLong subscriberCount = new AtomicLong(0L);

// Lines 89-90 — increments/decrements the gauge but never rejects
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
``` [1](#0-0) [2](#0-1) 

The only configured limit is `maxConcurrentCallsPerConnection = 5` (default), applied in `GrpcConfiguration.java`:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [3](#0-2) [4](#0-3) 

This is a **per-connection** limit, not a global or per-IP limit. There are zero interceptors, filters, or application-level guards that limit total connections or total subscriptions. A grep across all gRPC Java sources confirms no `maxConnections`, `connectionLimit`, `subscriptionLimit`, or IP-based interceptor exists.

Each subscription also schedules a safety-check Flux on `Schedulers.boundedElastic()`:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
``` [5](#0-4) 

And each subscription may trigger `topicMessageRetriever.retrieve()` database calls for historical and gap-recovery queries, consuming JDBC connection pool slots.

The optional GCP gateway config `maxRatePerEndpoint: 250` is an infrastructure-level Helm value that (a) only applies when the GCP gateway is deployed, (b) limits request initiation rate — not concurrent open streaming connections — and (c) is not present in the application itself. [6](#0-5) 

### Impact Explanation

An attacker who saturates `Schedulers.boundedElastic()` and the JDBC connection pool causes:
- Safety-check and gap-recovery queries for legitimate subscribers to queue indefinitely or time out.
- Live listener fan-out to degrade under memory/thread pressure.
- Legitimate HCS topic subscribers (e.g., payment-confirmation listeners) to miss or receive delayed messages, which for applications that gate fund release on topic message receipt constitutes a direct operational impact.

The mirror node is read-only and holds no funds itself, but HCS is explicitly used for payment confirmation workflows; a sustained DoS against the gRPC endpoint disrupts those workflows.

### Likelihood Explanation

- **No authentication required**: port 5600 is publicly exposed.
- **No special tooling**: standard gRPC client libraries (grpc-java, grpcurl) suffice.
- **Trivially repeatable**: open N TCP connections × 5 calls each = 5N concurrent subscriptions. With N=200 connections, that is 1,000 concurrent long-lived streaming RPCs.
- **Subscriptions are long-lived by design** (no `endTime` means `Flux.never()` termination), so connections accumulate without natural cleanup. [7](#0-6) 

### Recommendation

1. **Global subscription cap**: Check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if a configurable maximum is exceeded.
2. **Per-IP connection limit**: Add a gRPC `ServerInterceptor` that tracks active streaming calls per remote IP and rejects new ones above a threshold.
3. **`maxConnectionAge` / `maxConnectionIdle`**: Configure Netty `maxConnectionAge` and `maxConnectionIdle` in `GrpcConfiguration` to reclaim idle or long-lived attacker connections.
4. **Infrastructure enforcement**: Make the GCP gateway `maxRatePerEndpoint` mandatory and add a concurrent-streams limit at the load balancer level, not just rate.

### Proof of Concept

```bash
# Open 200 parallel connections, each with 5 subscriptions (1000 total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
# Server's boundedElastic scheduler and DB pool are now saturated.
# Legitimate subscriber's safety-check queries queue indefinitely.
```

Observe via Micrometer metrics that `hiero.mirror.grpc.subscribers` climbs without bound and that legitimate subscriber latency spikes or messages are dropped.
