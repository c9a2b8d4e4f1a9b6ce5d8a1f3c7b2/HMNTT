### Title
Unbounded Indefinite gRPC Subscriptions via `Flux.never()` Enable File Descriptor Exhaustion DoS

### Summary
When an unprivileged client subscribes to a valid topic without specifying `endTime`, `pastEndTime()` returns `Flux.never()`, causing the subscription pipeline to remain open indefinitely with no server-side termination. There is no global cap on concurrent subscriptions — `subscriberCount` is a metric gauge only, never enforced — so an attacker opening many connections (each with up to 5 concurrent calls) can accumulate thousands of idle, never-completing gRPC streams, exhausting OS file descriptors and crashing the node.

### Finding Description

**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`:

`pastEndTime()` (lines 123–131):
```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // ← never emits, never completes
    }
    ...
}
```

`subscribeTopic()` (lines 72–73):
```java
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
```

When `endTime == null`, `takeUntilOther(Flux.never())` never fires. The `live` sub-flux (`incomingMessages → topicListener.listen`) waits indefinitely for messages that never arrive on an empty topic. The two additional termination guards:

- Line 79–81: `if (filter.getEndTime() != null) { flux = flux.takeWhile(...) }` — skipped when `endTime` is null.
- Line 83–85: `if (filter.hasLimit()) { flux = flux.take(...) }` — skipped when `limit == 0` (default).

**No global subscriber cap:**

`subscriberCount` (line 48) is an `AtomicLong` exposed only as a Micrometer `Gauge` (lines 52–55). It is incremented on subscribe (line 89) and decremented on completion (line 90), but **never checked** to reject new subscriptions. There is no code path that returns an error when `subscriberCount` exceeds any threshold.

**Per-connection limit is insufficient:**

`NettyProperties.maxConcurrentCallsPerConnection = 5` (default) limits calls per TCP connection, but there is no limit on the number of TCP connections an attacker may open. An attacker with N connections can hold 5N simultaneous open streams.

**Topic existence check passes:**

`topicExists()` (lines 94–106) validates the topic exists and is of type `TOPIC`. A valid topic with zero messages passes this check cleanly, allowing the subscription to proceed into the indefinite-wait state.

**`TopicMessageFilter` validation** (lines 43–51) only enforces `startTime <= now` and `endTime > startTime` (if set). No per-user, per-IP, or global subscription count limit exists.

### Impact Explanation

Each open gRPC stream holds:
- One OS file descriptor (TCP socket)
- A Reactor `Flux` chain with associated scheduler threads/tasks (safetyCheck on `boundedElastic`)
- A `TopicContext` object and associated `AtomicReference`/`AtomicLong` heap allocations

Linux default `ulimit -n` is typically 1024–65536 per process. With `maxConcurrentCallsPerConnection = 5`, an attacker needs only ~200–13,000 TCP connections to exhaust file descriptors. At that point, the gRPC server cannot accept new connections or open new sockets, causing `UNAVAILABLE` errors for all clients. If the JVM heap is also exhausted by accumulated Flux state, the node crashes entirely. Crashing 30%+ of mirror nodes degrades the network's data availability layer.

### Likelihood Explanation

- **No authentication required**: The gRPC `subscribeTopic` endpoint is publicly accessible; no credentials or privileged role is needed.
- **Trivially scriptable**: A single attacker machine can open thousands of TCP connections using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`) in a loop.
- **Stable and repeatable**: Each subscription stays open until the attacker disconnects or the process crashes; the attacker can maintain pressure continuously.
- **Low cost**: Empty-topic subscriptions consume no bandwidth (no messages flow), making this cheap to sustain.

### Recommendation

1. **Enforce a global (or per-IP) concurrent subscription cap**: Check `subscriberCount` (or a per-IP counter) before allowing a new subscription and return `RESOURCE_EXHAUSTED` if the limit is exceeded.
2. **Require or enforce a maximum `endTime`**: Reject subscriptions with `endTime == null` that also have `limit == 0`, or impose a server-side maximum subscription duration (e.g., 24 hours) even when the client omits `endTime`.
3. **Replace `Flux.never()` with a bounded timeout**: Instead of `Flux.never()`, use a configurable maximum-lifetime signal (e.g., `Flux.never().timeout(maxSubscriptionDuration)`) so all subscriptions eventually terminate server-side.
4. **Rate-limit connections per IP** at the Netty/load-balancer layer to prevent a single source from opening unbounded TCP connections.

### Proof of Concept

```python
# Pseudocode — requires grpcio and the mirror node proto stubs
import grpc, threading
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc

TARGET = "mirror-node-grpc:5600"
VALID_TOPIC_NUM = 1234  # a topic that exists but has no messages

def open_subscription():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_pb2.ConsensusTopicQuery(
        topicID=...,   # VALID_TOPIC_NUM, no endTime, no limit
    )
    for _ in stub.subscribeTopic(query):  # blocks indefinitely
        pass

# Open thousands of threads, each holding one persistent subscription
threads = [threading.Thread(target=open_subscription) for _ in range(5000)]
for t in threads:
    t.start()
# After ~ulimit connections, the target node stops accepting new gRPC calls
```
