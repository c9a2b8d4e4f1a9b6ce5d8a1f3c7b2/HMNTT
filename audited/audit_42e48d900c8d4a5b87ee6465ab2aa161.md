### Title
Global-Only Rate Limit on Opcode Endpoint Allows Single Unauthenticated Attacker to Sustain Multiple Concurrent EVM Re-Executions

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint, when enabled, requires no authentication and enforces only a single global (not per-IP) token-bucket rate limit of 1 request/second. Because the throttle gates only request *entry* and not concurrent *execution count*, and because EVM re-execution with full stack/memory/storage tracing can run for up to the 10-second server timeout, a single unauthenticated attacker can accumulate up to ~10 concurrent full EVM re-executions simultaneously, causing sustained CPU and memory exhaustion.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (lines 52–68): [1](#0-0) 

There is no `@PreAuthorize`, no Spring Security filter, and no IP-based check. Any unauthenticated caller reaching the endpoint passes through two checks:
1. `validateAcceptEncodingHeader` — trivially satisfied with `Accept-Encoding: gzip`
2. `throttleManager.throttleOpcodeRequest()` — consumes one token from the global `opcodeRateLimitBucket`

`ThrottleManagerImpl.throttleOpcodeRequest()` (lines 52–56): [2](#0-1) 

`ThrottleConfiguration.opcodeRateLimitBucket()` (lines 47–55) — a single Spring singleton `Bucket` bean, shared across all callers, with default capacity=1, refill=1/second: [3](#0-2) 

Default `opcodeRequestsPerSecond = 1` (documented as intentionally low because "this endpoint is heavy"): [4](#0-3) 

`requestTimeout = 10000ms`: [5](#0-4) 

**Root cause:** The throttle is a global rate limiter, not a per-IP or per-user limiter, and it controls only the *rate of admission* (1 req/sec), not the *number of concurrently executing* EVM re-runs. Once a request passes the token-bucket check, it proceeds to `opcodeService.processOpcodeCall(request)` which triggers a full EVM re-execution with opcode tracing. There is no semaphore, no concurrency cap, and no mechanism to prevent multiple long-running re-executions from overlapping.

`OpcodeServiceImpl.processOpcodeCall()` (lines 64–74) — unconditionally starts EVM re-execution: [6](#0-5) 

`OpcodesProperties.enabled` defaults to `false`: [7](#0-6) 

The `enabled=false` default is the primary deployment-time guard, but once an operator enables the endpoint (its intended production use), the above gap is fully exposed.

### Impact Explanation
With a 10-second request timeout and a 1 req/sec admission rate, an attacker can have up to 10 concurrent full EVM re-executions running simultaneously. Each re-execution with `stack=true&memory=true&storage=true` on a complex transaction (up to `maxGasLimit=15,000,000` gas) allocates large in-memory opcode trace structures (`OpcodeContext` pre-allocates `gas/3` opcode slots) and saturates CPU with EVM interpretation. Ten such concurrent executions can exhaust JVM heap and CPU cores. Additionally, because the bucket is global, the attacker monopolizes the entire quota, denying service to all legitimate users.

### Likelihood Explanation
The attack requires zero privileges — only a valid transaction hash (publicly observable on-chain) and the `Accept-Encoding: gzip` header. The attacker needs only a script that sends one request per second continuously. The endpoint must be operator-enabled, which reduces the exposed population, but any deployment that enables it for debugging or tooling purposes is immediately vulnerable. The attack is trivially repeatable and requires no special knowledge beyond knowing the endpoint exists.

### Recommendation
1. **Add a per-IP (or per-client) rate limit** in addition to the global bucket, so a single source cannot monopolize the global quota.
2. **Add a concurrency semaphore** (e.g., `Semaphore` with a small bound like 2–3 permits) inside `throttleOpcodeRequest()` or `OpcodeServiceImpl.processOpcodeCall()` to cap the number of simultaneously executing EVM re-runs, independent of the admission rate.
3. **Require authentication** (e.g., an API key or operator-configured token) for this endpoint, since it is explicitly documented as a heavy, non-public debugging tool.

### Proof of Concept
```bash
# Precondition: endpoint enabled (hiero.mirror.web3.opcode.tracer.enabled=true)
# Obtain any valid complex transaction hash from the chain (publicly visible)
TX_HASH="0x<known_complex_tx_hash>"
BASE_URL="https://<mirror-node-host>"

# Send 1 request/second continuously; each takes ~10s to complete
# After ~10 seconds, ~10 concurrent EVM re-executions are running simultaneously
while true; do
  curl -s -o /dev/null \
    -H "Accept-Encoding: gzip" \
    "${BASE_URL}/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" &
  sleep 1
done
```

After ~10 iterations, `jstack`/heap dumps on the server will show 10 concurrent threads blocked inside EVM execution with large `OpcodeContext` allocations, causing CPU saturation and potential OOM.

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
