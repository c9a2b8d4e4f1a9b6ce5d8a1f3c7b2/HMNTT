### Title
Prometheus Label Cardinality Explosion via Unbounded `route` Label in `MetricsMiddleware`

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` uses `github.com/weaveworks/common/middleware.Instrument` with a gorilla/mux `RouteMatcher`. For HTTP requests that do not match any registered route, the `RouteMatcher.MatchRoute(r)` call falls back to the raw `req.URL.Path`, causing every distinct URL path sent by an attacker to become a unique `route` label value across all four `HistogramVec`/`GaugeVec` metrics. An unprivileged external attacker can flood the server with requests to unique paths, causing unbounded Prometheus time-series growth and eventual process memory exhaustion (OOM).

### Finding Description

**Exact code location:** `rosetta/app/middleware/metrics.go`, lines 76–83, function `MetricsMiddleware`.

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,   // labels: method, route, status_code, ws
        InflightRequests: requestInflightGauge,       // labels: method, route
        RequestBodySize:  requestBytesHistogram,      // labels: method, route
        ResponseBodySize: responseBytesHistogram,     // labels: method, route
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

**Root cause:** `middleware.Instrument.Wrap` (weaveworks) calls `i.RouteMatcher.MatchRoute(r)` on every incoming request to derive the `route` label value. The `RouteMatcher` is the gorilla/mux router returned by `server.NewRouter(...)` (rosetta-sdk-go). Gorilla/mux's `MatchRoute` implementation returns the matched route template (e.g., `/account/balance`) for registered routes, but **falls back to `req.URL.Path` (the raw request path) when no route matches**. There is no sanitization, normalization, or cardinality cap applied to this value before it is passed to `WithLabelValues(...)` on the Prometheus vectors.

**All registered Rosetta routes are fixed POST paths** (e.g., `POST /account/balance`, `POST /block`, `POST /network/list`, etc.) with no URL path parameters — account identifiers are in the POST body. Any request to a path not in this fixed set (e.g., `GET /account/0.0.12345`, `POST /foo/bar/0.0.99999`) will not match, causing the raw path to become the `route` label.

**Exploit flow:**
1. Attacker sends HTTP requests to the Rosetta server with unique URL paths: `/account/0.0.1`, `/account/0.0.2`, … `/account/0.0.N`.
2. None of these match any registered route (registered route is `POST /account/balance`, not `/account/{id}`).
3. `MatchRoute` returns the raw path for each request.
4. `requestDurationHistogram.WithLabelValues(method, "/account/0.0.1", statusCode, ws)` allocates a new histogram (6 buckets + sum + count = 8 time series per unique label combination).
5. With 4 metric vectors and 8 series per histogram bucket set, each unique path creates ~30+ new in-memory time series.
6. Prometheus client library stores all label combinations in memory indefinitely (no TTL, no eviction).

**Why existing checks fail:** There are no checks in `MetricsMiddleware` or anywhere in the middleware chain (at the Go application level) that cap, normalize, or reject unknown route labels. The Traefik-level rate limiting (`rateLimit: average: 10`, `inFlightReq: amount: 5`) defined in `charts/hedera-mirror-rosetta/values.yaml` is an optional Kubernetes deployment artifact — it is absent in bare-metal, Docker, or non-Traefik deployments, and even when present, 10 req/s × 86400 s/day = 864,000 unique paths/day is sufficient to exhaust memory.

### Impact Explanation
Each unique URL path permanently allocates new Prometheus time series in the Go process heap. With `requestDurationHistogram` having 4 label dimensions and 6 buckets, a sustained attack of ~100,000 unique paths creates millions of in-memory objects. The Go process will OOM-crash, taking down the Rosetta API entirely. This constitutes a non-network-based DoS (memory exhaustion within the process) requiring no authentication, no valid Rosetta payload, and no special privileges. The Rosetta API is a critical integration point for exchanges interacting with the Hedera network, making availability impact significant.

### Likelihood Explanation
The attack requires only the ability to send HTTP GET or POST requests to the Rosetta server's port — no credentials, no valid JSON body, no knowledge of the API. A single attacker with a script generating sequential paths (`/x/0.0.1`, `/x/0.0.2`, …) can trigger this. The attack is repeatable, persistent (allocations are never freed), and low-cost. Rate limiting at the application layer is absent; infrastructure-level mitigations are deployment-dependent and bypassable with distributed sources.

### Recommendation
1. **Normalize unmatched routes to a fixed sentinel value** before recording metrics. Override or wrap `RouteMatcher.MatchRoute` to return a constant (e.g., `"unmatched"`) when no route matches, instead of the raw path.
2. **Apply the fix at the middleware level**, not at the infrastructure level, so it is deployment-agnostic:

```go
type safeRouteMatcher struct {
    inner middleware.RouteMatcher
    known map[string]struct{}
}

func (s safeRouteMatcher) MatchRoute(r *http.Request) string {
    route := s.inner.MatchRoute(r)
    if _, ok := s.known[route]; ok {
        return route
    }
    return "unmatched"
}
```

3. Alternatively, use `promhttp.InstrumentHandlerDuration` with a fixed set of label values and a `prometheus.Labels` allowlist.
4. Add an application-level rate limiter (e.g., `golang.org/x/time/rate`) independent of infrastructure.

### Proof of Concept

```bash
# Send 100,000 requests with unique paths to the Rosetta server
for i in $(seq 1 100000); do
  curl -s -o /dev/null "http://<rosetta-host>:<port>/account/0.0.$i" &
done
wait

# Observe memory growth in the Rosetta process
# Then query Prometheus to confirm cardinality explosion:
curl http://<rosetta-host>:<port>/metrics | grep hiero_mirror_rosetta_request_duration | wc -l
# Expected: thousands of unique route label values
```

Each iteration creates a new `route` label value (`/account/0.0.1`, `/account/0.0.2`, …) permanently stored in the Prometheus client's in-memory registry, growing heap usage until OOM.
