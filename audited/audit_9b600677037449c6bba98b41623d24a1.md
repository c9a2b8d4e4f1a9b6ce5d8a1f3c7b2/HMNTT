### Title
Unauthenticated, Unthrottled gRPC `getNodes()` Endpoint Enables Amplification-Based DoS via Unbounded Address Book Streaming

### Summary
The `getNodes()` handler in `NetworkController` accepts requests from any unauthenticated caller with no application-level rate limiting. A caller sending a minimal request with `limit=0` forces the server to page through the entire address book, execute the nested `serviceEndpoints` for-loop in `toNodeAddress()` for every node, serialize each `NodeAddress` protobuf, and stream all results back. With no per-IP or global throttle on the gRPC module, an attacker opening many concurrent connections can sustain this amplification continuously, exhausting serialization CPU and outbound bandwidth across mirror node processing instances.

### Finding Description
**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) accepts any `AddressBookQuery` with no authentication check and no rate-limiting interceptor: [1](#0-0) 

`toFilter()` (lines 45–53) passes `query.getLimit()` directly into `AddressBookFilter`. When `limit=0`, `NetworkServiceImpl.getNodes()` substitutes `Long.MAX_VALUE`: [2](#0-1) 

This causes the service to page through the entire address book indefinitely until exhausted. For each `AddressBookEntry`, `toNodeAddress()` executes an unbounded for-loop over all `serviceEndpoints`: [3](#0-2) 

There is no cap on the number of service endpoints per node in this loop. Each iteration allocates a `ServiceEndpoint` protobuf, calls `InetAddress.getByName()`, and appends to the `NodeAddress` builder before the whole object is serialized and written to the wire.

**Root cause — failed assumption:** The design assumes that the address book is small and queries are infrequent. There is no application-level rate limiter in the `grpc` module. The only throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`, `ThrottleProperties`) lives exclusively in the `web3` module and is never wired into the gRPC service: [4](#0-3) 

The sole gRPC server interceptor only sets an endpoint-context label and performs no access control: [5](#0-4) 

**Existing checks reviewed and shown insufficient:**

- `maxConcurrentCallsPerConnection=5` limits concurrent streams *per TCP connection*, not total connections or total requests per second. An attacker opens N connections and gets 5N concurrent streams. [6](#0-5) 

- The GCP backend policy `maxRatePerEndpoint: 250` is optional infrastructure (`global.gateway.enabled: false` by default) and is not an application-level control: [7](#0-6) 

- `AddressBookFilter.limit` has `@Min(0)`, making `limit=0` (meaning "no limit") a valid, accepted value: [8](#0-7) 

### Impact Explanation
Each `getNodes(limit=0)` call causes: repeated DB reads (paged at `pageSize=10` with 250 ms delay between pages), CPU-bound protobuf serialization of every node's full `NodeAddress` including all service endpoints, and streaming of the complete result set over the wire. An attacker sustaining many concurrent connections drives CPU utilization and outbound bandwidth proportionally. The `GrpcHighCPU` alert threshold is 80% CPU for 5 minutes, meaning a sustained flood can trigger it and degrade or crash ≥30% of mirror node processing pods before autoscaling (HPA disabled by default) can respond. [9](#0-8) 

### Likelihood Explanation
The endpoint is publicly documented and trivially reachable with `grpcurl` — no credentials, no tokens, no prior state required: [10](#0-9) 

A single attacker with a modest number of machines can open hundreds of connections and sustain the flood indefinitely. The attack is repeatable, requires no privileged access, and is not detectable until CPU/bandwidth alerts fire.

### Recommendation
1. Add a gRPC `ServerInterceptor` in the `grpc` module that enforces a per-IP and global request rate limit for `NetworkService/getNodes`, mirroring the `ThrottleManagerImpl` pattern already used in `web3`.
2. Enforce a hard minimum for `limit` (e.g., `@Min(1)`) or cap the maximum nodes returned per call, preventing unbounded full-book streaming.
3. Add a per-node cap on the number of `serviceEndpoints` serialized in `toNodeAddress()` to bound per-response size regardless of DB content.
4. Enable the GCP backend `maxRatePerEndpoint` policy unconditionally, or add an equivalent Traefik `rateLimit` middleware for the `NetworkService` path (as already done for Rosetta).

### Proof of Concept
```bash
# Open 50 concurrent connections, each issuing a no-limit getNodes stream
for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes \
    > /dev/null &
done
wait
```
Each background process holds a streaming gRPC call open, forcing the server to continuously page the address book, execute the `serviceEndpoints` for-loop for every node, serialize all `NodeAddress` protos, and flush them to the network. Repeating this from multiple source IPs bypasses the per-connection concurrency limit and drives CPU and bandwidth toward exhaustion.
