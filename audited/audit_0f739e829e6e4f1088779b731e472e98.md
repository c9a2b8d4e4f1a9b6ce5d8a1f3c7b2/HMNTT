### Title
Unauthenticated DB Query Amplification via Uncached, Unthrottled `getHookStorage()` in rest-java

### Summary
The `getHookStorage()` method in `HookServiceImpl` executes a live database query on every invocation with no application-level caching and no rate limiting in the `rest-java` module. An unprivileged external user can enumerate valid `(ownerId, hookId)` pairs and issue a sustained flood of requests, each triggering a separate DB query, with no in-process mechanism to reject or coalesce them.

### Finding Description
**Code path:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`, `getHookStorage()`, lines 56â€“82.

Every call unconditionally resolves the owner entity via `entityService.lookup()` (line 61) and then executes one of two repository queries:
- `hookStorageRepository.findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse()` (line 66â€“67)
- `hookStorageRepository.findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse()` (line 78â€“79)

Neither `HookStorageRepository` nor `HookStorageChangeRepository` carries a `@Cacheable` annotation (contrast with `ContractRepository`, `ContractStateRepository`, and `EntityRepository` in the `web3` module, which all use `@Cacheable`). There is no `@Cacheable` on the service method either.

The `rest-java` module has no equivalent of the `web3` module's `ThrottleManagerImpl` / `ThrottleConfiguration` / `rateLimitBucket`. The only application-level throttle constant visible in `rest-java` is `HIGH_VOLUME_THROTTLE` in `Constants.java` (line 19), which is wired to the fee-estimation endpoint, not to hook storage. The GCP gateway `maxRatePerEndpoint: 250` in the Helm chart is infrastructure-level and absent in non-GCP or bare-metal deployments.

**Root cause:** The failed assumption is that infrastructure-level rate limiting is always present and sufficient. At the application layer, `getHookStorage()` is a fully open, uncached, per-request DB operation.

### Impact Explanation
Each HTTP request to the hook storage endpoint causes at minimum one SQL query against the `hook_storage` or `hook_storage_change` table. With no caching, identical or varied `hookId` values each produce a fresh query. At high concurrency (e.g., 200â€“500 req/s from a single client or botnet), the DB connection pool saturates, query latency rises across all mirror-node services sharing the same database, and CPU/IO on the DB host increases proportionally. Because `hookId` is a `long` (line 23 of `HookStorageRequest.java`) and owner IDs are enumerable entity IDs, an attacker has a large valid input space without needing any privileged credential.

### Likelihood Explanation
No authentication is required. Valid `ownerId` values are publicly discoverable via the accounts API. `hookId` values are sequential longs starting from a known range. A single attacker with a modest HTTP client (e.g., `wrk`, `hey`, or a simple script) can sustain hundreds of requests per second. The attack is repeatable, stateless, and requires no special knowledge beyond a valid account ID.

### Recommendation
1. Add `@Cacheable` to `HookStorageRepository` query methods (short TTL, e.g., 2â€“5 s) to absorb repeated identical queries.
2. Introduce an application-level rate limiter in `rest-java` for hook storage endpoints, mirroring the `ThrottleManagerImpl` pattern in `web3`, with a configurable `requestsPerSecond` property.
3. Consider a per-IP or per-`ownerId` rate limit to prevent a single client from monopolising DB capacity.
4. Do not rely solely on infrastructure-level gateways as the only line of defence.

    public HookStorageResult getHookStorage(HookStorageRequest request) {
        if (request.isHistorical()) {
            return getHookStorageChange(request);
        }

        final var ownerId = entityService.lookup(request.getOwnerId());
        final var page = request.getPageRequest();
        final var keys = request.getKeys();

        if (keys.isEmpty()) {
            final var hookStorage = hookStorageRepository.findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse(
                    ownerId.getId(), request.getHookId(), request.getKeyLowerBound(), request.getKeyUpperBound(), page);

            return new HookStorageResult(ownerId, hookStorage);
        }

        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty()) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var hookStorage = hookStorageRepository.findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse(
                ownerId.getId(), request.getHookId(), keysInRange, page);

        return new HookStorageResult(ownerId, hookStorage);
    }
