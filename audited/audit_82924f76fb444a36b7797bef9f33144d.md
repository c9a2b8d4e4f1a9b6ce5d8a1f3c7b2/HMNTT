### Title
Unauthenticated BLOCK_HASH Filter Forces Uncached DB Query on Every Request in `getContractResultsById`

### Summary
Any unauthenticated user can supply a `block.hash` query parameter to `GET /api/v1/contracts/:contractId/results`, which unconditionally triggers `RecordFileService.getRecordFileBlockDetailsFromHash()` — a raw SQL `LIKE` query against the `record_file` table — on every single request. There is no caching, no rate limiting at the code level, and no authentication requirement, making this a reliable, low-effort resource exhaustion vector.

### Finding Description

**Code path:**

`getContractResultsById` (line 856) calls `extractContractIdAndFiltersFromValidatedRequest` (line 857), which calls `utils.buildAndValidateFilters` with `contractResultsFilterValidityChecks`. `acceptedContractResultsParameters` explicitly includes `filterKeys.BLOCK_HASH` (line 1338). [1](#0-0) 

In `filterValidityChecks` (line 298–300), `BLOCK_HASH` is validated as:

```js
ret = isValidEthHashOrHederaHash(val) && op === constants.queryParamOperators.eq;
``` [2](#0-1) 

This means any well-formed Ethereum or Hedera hash passes validation. Once past validation, in `extractContractResultsByIdQuery`, the `blockFilter` is set and the following branch executes unconditionally:

```js
if (blockFilter) {
  let blockData;
  if (blockFilter.key === filterKeys.BLOCK_NUMBER) {
    blockData = await RecordFileService.getRecordFileBlockDetailsFromIndex(blockFilter.value);
  } else {
    blockData = await RecordFileService.getRecordFileBlockDetailsFromHash(blockFilter.value);
  }
  ...
}
``` [3](#0-2) 

`getRecordFileBlockDetailsFromHash` executes:

```sql
SELECT consensus_start, consensus_end, hash, index
FROM record_file
WHERE hash LIKE $1
LIMIT 1
```

with `${hash}%` as the parameter — a prefix `LIKE` scan. [4](#0-3) [5](#0-4) 

**Root cause:** The `BLOCK_HASH` filter path has no result caching and no per-IP or per-endpoint rate limiting in the application layer. Every HTTP request with a syntactically valid `block.hash` value unconditionally issues a database query. The `contractResultsFilterValidityChecks` function adds no additional restriction for `BLOCK_HASH` beyond format validation. [6](#0-5) 

### Impact Explanation
An attacker can drive a sustained, high-rate stream of DB queries against the `record_file` table using only HTTP GET requests — no credentials, no special privileges. Each request causes one `LIKE`-prefix scan. If the `hash` column lacks a covering index for prefix patterns, this degrades to a sequential scan. Under sufficient concurrency this exhausts the DB connection pool and/or saturates I/O, degrading or halting query processing for all mirror node consumers. This aligns with the stated severity: ≥30% processing node degradation without brute-force.

### Likelihood Explanation
The endpoint is public, requires no authentication, and accepts any valid 32-byte hex hash. The attacker needs only a valid contract ID (trivially enumerable) and any syntactically valid hash value (does not need to exist in the DB — a non-matching hash still triggers the query and returns `skip: true`). The attack is trivially scriptable, repeatable, and requires no prior knowledge of the system state.

### Recommendation
1. **Cache** the result of `getRecordFileBlockDetailsFromHash` (and `getRecordFileBlockDetailsFromIndex`) with a short TTL (e.g., 5–10 seconds) keyed on the hash value, since block data is immutable once written.
2. **Rate-limit** the `/contracts/:contractId/results` endpoint at the API gateway or middleware layer, especially when `block.hash` or `block.number` filters are present.
3. **Ensure a B-tree index** exists on `record_file.hash` to make the prefix `LIKE` scan index-bound rather than a sequential scan.
4. Optionally enforce a minimum hash prefix length before issuing the query.

### Proof of Concept

**Preconditions:** Mirror node REST API is publicly accessible. Any valid contract ID is known (e.g., `0.0.1000`).

**Steps:**

```bash
# Single trigger — forces one DB LIKE query, no auth needed
curl "https://<mirror-node>/api/v1/contracts/0.0.1000/results?block.hash=0xaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"

# Sustained flood — each request independently triggers getRecordFileBlockDetailsFromHash()
while true; do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1000/results?block.hash=0xaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa" &
done
```

Each request passes format validation, sets `blockFilter`, and unconditionally calls `RecordFileService.getRecordFileBlockDetailsFromHash()` before any short-circuit can occur. Under load, DB connection pool exhaustion and query latency spikes are observable.
