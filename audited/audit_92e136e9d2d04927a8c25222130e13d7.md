### Title
Cache Stampede (Thundering Herd) on `findLatest()` via Unprotected 500ms Caffeine TTL Expiry

### Summary
`RecordFileRepository.findLatest()` is protected only by Spring's `@Cacheable` annotation backed by a Caffeine cache with a 500ms `expireAfterWrite` TTL and no stampede-prevention mechanism. Spring's `@Cacheable` does not serialize concurrent cache-miss computations: when the entry expires, every thread that simultaneously observes a miss proceeds to execute the underlying SQL query. An unprivileged attacker sustaining ≥500 concurrent requests/second (the default rate limit) can reliably trigger this condition every 500ms, exhausting the HikariCP connection pool and causing cascading request queuing or rejection across all web3 API nodes.

### Finding Description

**Exact code path:**

`RecordFileServiceImpl.findByBlockType()` (lines 22-23) unconditionally delegates to `recordFileRepository.findLatest()` for every `BlockType.LATEST` request with no additional guard: [1](#0-0) 

`RecordFileRepository.findLatest()` is annotated with `@Cacheable` using `CACHE_MANAGER_RECORD_FILE_LATEST`: [2](#0-1) 

The backing `CacheManager` is configured with `expireAfterWrite(500, TimeUnit.MILLISECONDS)`, `maximumSize(1)`, and **no** `refreshAfterWrite`, `AsyncLoadingCache`, or any other stampede-prevention primitive: [3](#0-2) 

**Root cause:** Spring's `@Cacheable` proxy performs a non-atomic check-then-act: it reads the cache, and if the entry is absent (expired), it calls the real method and stores the result. There is no locking between the check and the store. When N threads simultaneously observe a miss, all N call `findLatest()` and issue `SELECT * FROM record_file ORDER BY consensus_end DESC LIMIT 1` against the database concurrently.

**Exploit flow:**
1. Attacker sends a sustained burst of concurrent `eth_call` / `eth_blockNumber` requests with `block = "latest"` at the maximum allowed rate (default 500 req/s).
2. Every 500ms the single cache entry expires.
3. All threads that arrived during the 500ms window and are still in-flight simultaneously observe a cache miss.
4. Each thread independently acquires a HikariCP connection and executes `findLatest()`.
5. With a default HikariCP pool (typically 10 connections), hundreds of threads queue for connections; `connectionTimeout` (default 30s) causes cascading `SQLTransientConnectionException` / request timeouts.
6. The cycle repeats every 500ms for as long as the attacker sustains the load.

### Impact Explanation

The HikariCP pool for the web3 service is finite. The Grafana alert threshold of 75% active connections already signals criticality: [4](#0-3) 

When the stampede fires, all available connections are consumed by duplicate `findLatest()` queries. Every other in-flight request (contract calls, gas estimates) that also needs a DB connection is blocked. The `requestTimeout` of 10,000ms means requests pile up until they time out, producing 5xx responses. Because `findByBlockType(LATEST)` is called on virtually every contract call execution path, the blast radius covers the entire web3 API surface, not just block-number lookups. Across a horizontally scaled deployment, every replica is independently vulnerable and will independently stampede, amplifying the effect.

### Likelihood Explanation

No authentication is required. The attacker needs only an HTTP client capable of sustaining ~500 concurrent connections to the public web3 endpoint. The default `requestsPerSecond = 500` throttle is a per-node, in-memory bucket4j counter: [5](#0-4) 

500 requests/second × 500ms TTL = up to 250 threads simultaneously hitting the DB on each expiry cycle. The attack is trivially repeatable with tools like `wrk`, `hey`, or `ab`. No exploit code, no credentials, no special knowledge of the system is required beyond knowing the public JSON-RPC endpoint.

### Recommendation

1. **Use Caffeine's `refreshAfterWrite`** instead of (or in addition to) `expireAfterWrite`. With `refreshAfterWrite`, Caffeine returns the stale value to all concurrent readers while a single background thread refreshes the entry, eliminating the stampede window entirely.
2. **Alternatively**, replace the `CaffeineCacheManager` for this cache with a `LoadingCache` / `AsyncLoadingCache` built directly via `Caffeine.newBuilder().refreshAfterWrite(400, MILLISECONDS).build(loader)`, bypassing Spring's non-atomic `@Cacheable` proxy.
3. **Short-term mitigation**: Wrap the cache-miss path in a per-key `ReentrantLock` or use a `ConcurrentHashMap.computeIfAbsent` pattern so only one thread executes `findLatest()` at a time.
4. **Increase the TTL** to reduce stampede frequency; 500ms is extremely aggressive for a "latest block" value that changes only every ~2 seconds on Hedera.

  http://<web3-node-host>/api/v1/contracts/call
