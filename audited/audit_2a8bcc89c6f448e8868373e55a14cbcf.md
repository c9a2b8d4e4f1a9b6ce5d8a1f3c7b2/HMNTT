### Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Unique `contractId` Requests Causes JVM Heap Exhaustion

### Summary
`ContractStateServiceImpl.findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` for every unique `contractId` not already in the outer `contractSlotsCache`. Because `cacheManagerSlotsPerContract` is a `CaffeineCacheManager` with no `setCacheNames()` pre-registration (dynamic mode), its internal `ConcurrentHashMap<String, Cache> cacheMap` grows without any bound. The outer `contractSlotsCache` (capped at 3,000 entries) does not trigger any cleanup in `cacheManagerSlotsPerContract.cacheMap` when it evicts entries, so every unique `contractId` ever seen permanently occupies a slot in that map. An unauthenticated attacker can exhaust JVM heap by sending requests that reference a large number of distinct contract addresses.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageBatch()` — lines 85–87: [1](#0-0) 

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

The outer `contractSlotsCache` is the `"default"` cache from `cacheManagerContractSlots`, configured with `maximumSize=3000`: [2](#0-1) 

The inner `cacheManagerSlotsPerContract` bean is declared **without** `setCacheNames()`: [3](#0-2) 

Omitting `setCacheNames()` leaves `CaffeineCacheManager` in *dynamic* mode. Spring's implementation of `CaffeineCacheManager.getCache(name)` stores every newly created cache in an internal `ConcurrentHashMap<String, Cache> cacheMap` with no capacity limit. Once a `CaffeineCache` is inserted for a given name it is **never removed** from that map — there is no eviction listener, no TTL on the map itself, and no size bound.

**Root cause and failed assumption:**

The design assumes the outer `contractSlotsCache` (bounded to 3,000) acts as the sole growth limiter. It does not. When the outer cache evicts an entry for `contractId=X`, it removes the `EntityId → CaffeineCache` mapping from its own Caffeine table, but `cacheManagerSlotsPerContract.cacheMap` still holds the strong reference `"X" → CaffeineCache`. The `slotsPerContract` spec `"expireAfterAccess=5m,maximumSize=1500"` governs entries *within* each per-contract cache, not the number of caches registered in the manager. [4](#0-3) 

**Exploit flow:**

1. Attacker sends `POST /api/v1/contracts/call` with a unique `to` address per request.
2. Each request reaches `ContractStorageReadableKVState.readFromDataSource()`, which calls `contractStateService.findStorage(entityId, keyBytes)`: [5](#0-4) 
3. `findStorage` checks `contractStateCache` (miss for a new contractId), then calls `findStorageBatch`. [6](#0-5) 
4. `findStorageBatch` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())`, inserting a new `CaffeineCache` object into the unbounded internal map.
5. After the outer `contractSlotsCache` fills to 3,000 and starts evicting, the inner map continues to grow monotonically — one entry per unique contractId, never reclaimed.

**Why existing checks are insufficient:**

- The outer `contractSlotsCache` `maximumSize=3000` only bounds the outer map; it has no eviction listener wired to `cacheManagerSlotsPerContract`.
- The `slotsPerContract` TTL/size spec applies to slot-key entries *inside* each per-contract cache, not to the cache objects themselves.
- The rate limiter (500 req/s, 7.5 B gas/s) slows accumulation but does not cap the total number of unique contractIds seen over the lifetime of the process: [7](#0-6) 
- No authentication is required on the `/api/v1/contracts/call` endpoint: [8](#0-7) 

### Impact Explanation
Each `CaffeineCache` instance allocated by `cacheManagerSlotsPerContract.getCache()` carries its own Caffeine internal data structures (segment arrays, reference queues, stats counters). At 500 req/s sustained, an attacker accumulates ~1.8 million cache objects per hour. This leads to progressive JVM heap exhaustion, ultimately triggering `OutOfMemoryError` and crashing the web3 service — a complete denial of service for all users of the mirror node's EVM/JSON-RPC interface.

### Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special knowledge beyond the public API. The attacker only needs to vary the `to` field across requests. The default rate limit of 500 req/s is generous; even at a fraction of that rate the leak is persistent and cumulative across the process lifetime. The attack is fully repeatable and automatable with a simple script.

### Recommendation
1. **Register cache names statically**: Call `cacheManagerSlotsPerContract.setCacheNames(Set.of(...))` with a fixed set of names, or switch to a design that does not create one named cache per contractId.
2. **Use a bounded map for per-contract caches**: Replace the dynamic `CaffeineCacheManager` with a single Caffeine cache whose keys are `(contractId, slotKey)` tuples, eliminating the two-level structure entirely.
3. **Add an eviction listener on the outer cache**: If the two-level design is kept, register a `RemovalListener` on `contractSlotsCache` that calls `cacheManagerSlotsPerContract`'s internal cleanup (e.g., invalidates the named cache) when an entry is evicted.
4. **Cap unique contractIds at the API layer**: Reject or rate-limit requests whose resolved contract address has never been seen before, or validate that the target contract exists before entering the caching path.

### Proof of Concept

```python
import requests, threading, itertools, time

URL = "http://<mirror-node-host>/api/v1/contracts/call"
HEADERS = {"Content-Type": "application/json"}

def make_address(n):
    # Generate a unique valid EVM address for each n
    return "0x" + hex(n)[2:].zfill(40)

def send(n):
    payload = {
        "to": make_address(n),
        "data": "0x",
        "gas": 50000,
        "estimate": False,
        "block": "latest"
    }
    try:
        requests.post(URL, json=payload, headers=HEADERS, timeout=5)
    except Exception:
        pass

# Flood with unique contract addresses at ~500 req/s
counter = itertools.count(1)
while True:
    threads = [threading.Thread(target=send, args=(next(counter),)) for _ in range(50)]
    for t in threads: t.start()
    for t in threads: t.join()
    time.sleep(0.1)  # ~500 req/s
    # Monitor mirror node JVM heap: jcmd <pid> VM.native_memory
    # cacheManagerSlotsPerContract internal cacheMap grows without bound
    # OOM expected within hours depending on JVM heap size
```

**Observable evidence**: Attach a JVM profiler (e.g., `jmap -histo`) and observe the count of `com.github.benmanes.caffeine.cache.BoundedLocalCache` instances growing monotonically with no upper bound, correlated with the number of unique `to` addresses sent.
