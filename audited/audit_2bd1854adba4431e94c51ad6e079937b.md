### Title
Unauthenticated EVM Address Path Parameter Forces Uncached DB Lookup on Every Request, Enabling DoS Amplification Against Shared Database

### Summary
Any unauthenticated user can supply a valid EVM address as the `idOrAliasOrEvmAddress` path parameter to `/api/v1/accounts/{evmAddress}/allowances/crypto`. Unlike numeric entity IDs (which are resolved locally with zero DB queries), EVM addresses unconditionally trigger `entityFromEvmAddressQuery` against the database on every request with no result caching in the JavaScript `EntityService`. An attacker can exploit this asymmetry to amplify DB query load at low request rates, exhausting connection pool slots shared with the importer and other services.

### Finding Description

**Exact code path:**

`cryptoAllowanceController.js` line 77 calls `EntityService.getEncodedId()` with the raw path parameter: [1](#0-0) 

Inside `getEncodedId()` (`entityService.js` lines 118–137), the branch at lines 122–124 is the critical fork: [2](#0-1) 

- If `entityId.evmAddress === null` (numeric/encoded ID) → `getEncodedId()` is a pure in-memory computation, **zero DB queries**.
- If `entityId.evmAddress !== null` (opaque EVM address) → `getEntityIdFromEvmAddress()` is called, which unconditionally executes `entityFromEvmAddressQuery`: [3](#0-2) [4](#0-3) 

**Root cause — failed assumption:** The JS `EntityService` has no result cache for EVM address DB lookups. The `parseCached` LRU in `entityId.js` only caches the *parsing* of the string into an `EntityId` object (lines 314–332), not the DB resolution result: [5](#0-4) 

This is in direct contrast to the Java `EntityIdServiceImpl` which wraps the same lookup in a Spring `Cache`: [6](#0-5) 

**No application-level rate limiting:** The only auth middleware (`authHandler.js`) adjusts response *limits* for known users; it provides no IP-based or per-endpoint request throttling for anonymous callers: [7](#0-6) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts/0x<random_40_hex>/allowances/crypto` repeatedly.
2. `EntityId.isValidEntityId('0x<40hex>')` returns `true` (confirmed by tests at lines 73–76 of `entityId.test.js`).
3. `EntityId.parseString()` produces an `EntityId` with `evmAddress !== null`.
4. `getEntityIdFromEvmAddress()` fires `entityFromEvmAddressQuery` against the DB.
5. DB returns empty → `NotFoundError` thrown → HTTP 404 returned.
6. Each request consumes one DB connection slot for the duration of the query.

For **existing** EVM addresses, the attacker gets two DB queries per request (EVM lookup + allowances query), doubling the load. For non-existent addresses, one DB query fires per request (vs. zero for numeric IDs), still a net amplification.

### Impact Explanation

The `entity` table is shared across the mirror node's REST API, importer, and other services. Saturating the DB connection pool (typically bounded at tens to low hundreds of connections) blocks the importer from writing newly confirmed transactions, stalling the mirror node's view of the network. The impact is a sustained degradation or halt of transaction confirmation visibility. The severity is high because the amplification is asymmetric (attacker cost: HTTP request; defender cost: DB query + connection slot) and requires no credentials.

### Likelihood Explanation

The endpoint is publicly documented in the OpenAPI spec and requires no authentication. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`) can generate thousands of EVM-address requests per second from a single host. The attack is trivially repeatable, requires no prior knowledge of the system, and is not detectable until DB saturation occurs. The only barrier is network-level rate limiting (load balancer / WAF), which is infrastructure-dependent and not enforced in the application code.

### Recommendation

1. **Add a result cache to `EntityService.getEntityIdFromEvmAddress()`** mirroring the Java `EntityIdServiceImpl` pattern — cache successful lookups (and optionally negative results with a short TTL) using the existing `quickLru` infrastructure already present in `entityId.js`.
2. **Add application-level rate limiting** (e.g., `express-rate-limit`) scoped to path-parameter-bearing endpoints, keyed by IP, before the controller handler runs.
3. **Short-circuit on long-form (account-num-alias) EVM addresses** — these can be decoded locally without a DB query; only opaque EVM addresses should reach the DB. Verify `isEvmAddressAlias()` is correctly applied before dispatching to `getEntityIdFromEvmAddress()`.

### Proof of Concept

```bash
# Flood with random opaque EVM addresses (no account needed)
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/accounts/0x${ADDR}/allowances/crypto" &
done
wait
```

Each iteration causes one `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` query. With sufficient concurrency, DB connection slots are exhausted, blocking the importer's write path and halting transaction confirmation updates.
