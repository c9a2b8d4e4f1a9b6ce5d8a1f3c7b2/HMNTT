### Title
Cache Key Bypass via Un-normalized Path Traversal Sequences in `cacheKeyGenerator()`

### Summary
`cacheKeyGenerator()` in `rest/middleware/responseCacheHandler.js` hashes `req.originalUrl` directly with MD5 without any path normalization. An unprivileged attacker can send a URL containing `..` segments (e.g., `/api/v1/transactions/../transactions/0.0.10-123`) that Express routes identically to the canonical path but produces a distinct MD5 hash, causing a guaranteed cache miss and forcing a fresh database read on every such request. The code itself acknowledges this gap with a TODO comment referencing a future normalizer.

### Finding Description
**Exact location:** `rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator()`, line 151–153.

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
``` [1](#0-0) 

The comment immediately above this function explicitly acknowledges the missing normalization:

> *"In the future, this will utilize Edwin's request normalizer (9113)."* [2](#0-1) 

**Root cause / failed assumption:** The code assumes `req.originalUrl` always arrives in canonical form. In Express.js, `req.originalUrl` is set from the raw HTTP request line and is **never** normalized by the framework — `..` segments are preserved verbatim. Express's router resolves `..` internally for route matching (so the correct handler is still invoked), but `req.originalUrl` retains the attacker-supplied form.

**Exploit flow:**

1. A legitimate client requests `GET /api/v1/transactions/0.0.10-123`. The response (e.g., `result: FAILURE`) is cached under key `MD5("/api/v1/transactions/0.0.10-123") + "-v1"`.
2. The attacker sends `GET /api/v1/transactions/../transactions/0.0.10-123` using a raw HTTP client (e.g., `curl --path-as-is`).
3. `responseCacheCheckHandler` calls `cacheKeyGenerator(req)`, which hashes the un-normalized string, producing a **different** key — cache miss. [3](#0-2) 
4. The request falls through to the route handler, which queries the database fresh.
5. `responseCacheUpdateHandler` stores the fresh response under the traversal-variant key, polluting Redis with a duplicate entry. [4](#0-3) 

**Why existing checks are insufficient:**

- `requestNormalizer.js` (`normalizeRequestQueryParams`) only normalizes **query parameters** (sorting, defaults, collapsing), not path segments, and is not called from `cacheKeyGenerator` at all. [5](#0-4) 
- `requestQueryParser` in `requestHandler.js` only lowercases query keys/values; it does not touch the URL path. [6](#0-5) 
- There is no middleware in `server.js` between request receipt and `responseCacheCheckHandler` that normalizes `req.originalUrl`. [7](#0-6) 

### Impact Explanation
- **Cache bypass:** Every request with a `..`-variant URL bypasses the Redis cache entirely, forcing a live database query. For a read-heavy mirror node API, this degrades performance and defeats the purpose of caching.
- **Cache pollution:** Each unique traversal variant (e.g., `/../transactions/`, `/./transactions/`, `/%2e%2e/transactions/`) creates a separate Redis entry for the same logical resource, exhausting Redis memory over time.
- **Stale-data bypass:** If a cached entry reflects a specific transaction state (e.g., `FAILURE`), the attacker can force a fresh read at any time without waiting for TTL expiry, potentially observing a state change that the cache would have hidden during the TTL window.

Severity: **Medium** — no direct data corruption, but cache integrity and availability are compromised.

### Likelihood Explanation
- Requires no authentication or privileges — any HTTP client that can send raw requests (e.g., `curl --path-as-is`, Python `requests` with `urllib3`, custom scripts) can exploit this.
- Fully repeatable and automatable.
- The TODO comment in the source confirms the developers are aware of the missing normalization, meaning it is a known-incomplete control.
- Reverse proxies (nginx, Envoy) often normalize `..` before forwarding, which would mitigate this in some deployments, but this is not guaranteed and is not enforced by the application itself.

### Recommendation
Normalize `req.originalUrl` (or derive the cache key from already-normalized components) before hashing in `cacheKeyGenerator()`. The simplest fix is to use Node's built-in `path.normalize` or the WHATWG `URL` API:

```js
import { URL } from 'url';

const cacheKeyGenerator = (req) => {
  // Normalize the path to collapse any '..' or '.' segments
  const parsed = new URL(req.originalUrl, 'http://localhost');
  const normalized = parsed.pathname + parsed.search;
  return crypto.createHash('md5').update(normalized).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

Alternatively, integrate the existing `normalizeRequestQueryParams` utility (already referenced in the TODO) and extend it to also normalize the path component using `path.posix.normalize`.

### Proof of Concept
**Preconditions:** Cache is enabled (`config.cache.response.enabled = true`, `config.redis.enabled = true`). A transaction `0.0.10-1234567890-000000000` exists and has been queried at least once (populating the cache).

**Steps:**

```bash
# Step 1: Populate the cache with the canonical URL
curl -s "http://mirror-node/api/v1/transactions/0.0.10-1234567890-000000000"
# Response cached under MD5("/api/v1/transactions/0.0.10-1234567890-000000000")-v1

# Step 2: Bypass the cache using a path traversal variant
# --path-as-is prevents curl from normalizing the URL
curl --path-as-is -s "http://mirror-node/api/v1/transactions/../transactions/0.0.10-1234567890-000000000"
# cacheKeyGenerator hashes "/api/v1/transactions/../transactions/0.0.10-1234567890-000000000"
# Different MD5 → cache MISS → fresh DB query → new Redis entry created

# Step 3: Confirm two distinct Redis keys exist for the same resource
redis-cli KEYS "*-v1" | grep -c ""
# Returns 2 (or more) entries for the same logical transaction
```

**Observed result:** The traversal request always produces a cache miss regardless of how recently the canonical URL was cached, and a duplicate Redis entry is created for each unique traversal variant used.

const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
};

const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);
    if (lowerKey in caseInsensitiveQueryString) {
      // handle repeated values, merge into an array
      caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;
};
