### Title
Unauthenticated KYC Status Disclosure via `/accounts/{id}/tokens` Endpoint

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` serves the `kyc_status` field of any account's token relationships to any unauthenticated caller with no ownership or authorization check. Any external user can enumerate the KYC compliance status of arbitrary victim accounts across all associated tokens, revealing which accounts have passed regulated KYC verification without the account owner's knowledge or consent.

### Finding Description
**Code path:**

- Route registration: `rest/routes/accountRoute.js` line 19 — `router.getExt(getPath('tokens'), TokenController.getTokenRelationships)` — no auth middleware attached.
- Handler: `rest/controllers/tokenController.js` lines 66–92, `getTokenRelationships()`. The only guard is `EntityService.isValidAccount(accountId)` (line 68–71), which checks account existence, not caller identity.
- SQL query: `rest/service/tokenService.js` lines 20–29 — `kyc_status` is unconditionally selected from `token_account` for the queried account.
- Response serialization: `rest/viewmodel/tokenRelationshipViewModel.js` lines 23–25 — `kyc_status` is always included in the JSON response.

**Root cause:** The endpoint performs no caller authentication and no ownership verification. The `authHandler` middleware (`rest/middleware/authHandler.js` lines 15–36) only sets a custom response-size limit for known users; it does not block or restrict unauthenticated access to the endpoint. There is no check that `req.user` or any session identity matches the queried `accountId`.

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts/0.0.<VICTIM>/tokens` with no credentials.
2. Server resolves the account, queries `token_account` for all associated tokens, and returns the full list including `kyc_status: "GRANTED"` or `"REVOKED"` per token.
3. Attacker learns which tokens the victim has passed KYC for, with no rate-limiting beyond the default page size.

**Why existing checks fail:** `isValidAccount` only confirms the account exists. `authHandler` is opt-in (no credentials → passes through silently). No middleware enforces that the caller owns or has permission to view the queried account's compliance data.

### Impact Explanation
`kyc_status: "GRANTED"` is a regulated compliance signal indicating an account has been identity-verified for a specific token. Exposing this to arbitrary unauthenticated callers leaks private financial compliance data at scale. An attacker can enumerate all accounts on the network, identify KYC-granted accounts for high-value regulated tokens, and use this information for targeted phishing, social engineering, or to map the network's verified participant graph. This is a direct information disclosure of sensitive compliance state with no barrier to exploitation.

### Likelihood Explanation
Exploitation requires only an HTTP client and knowledge of any account ID (which are sequential integers, trivially enumerable). No credentials, special network access, or on-chain privileges are needed. The attack is fully automated, repeatable, and leaves no distinguishable trace beyond normal API traffic. Any actor with internet access to the mirror node REST API can perform this at scale.

### Recommendation
1. **Require caller authentication** for the `/accounts/{id}/tokens` endpoint, or at minimum enforce that the authenticated caller's account ID matches the queried account ID before returning `kyc_status`.
2. **Redact or omit `kyc_status`** from unauthenticated responses, returning `null` or omitting the field entirely unless the caller is the account owner or an authorized operator.
3. If public read of KYC status is intentional by protocol design, document it explicitly and assess whether it conflicts with applicable data-privacy or compliance regulations.

### Proof of Concept
```
# No credentials required. Replace 0.0.1001 with any valid account ID.
curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.1001/tokens" | jq '.tokens[] | {token_id, kyc_status}'

# Expected response (HTTP 200, no auth):
{
  "token_id": "0.0.1135",
  "kyc_status": "GRANTED"
}
{
  "token_id": "0.0.2500",
  "kyc_status": "REVOKED"
}
```

  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
