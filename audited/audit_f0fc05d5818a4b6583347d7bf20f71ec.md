### Title
Redis Subscription Teardown Race via `share()` Refcount Drop Causes Message Gap for Subsequent Subscribers

### Summary
In `RedisTopicListener.subscribe()`, the `share()` operator (equivalent to `publish().refCount()`) cancels the upstream Redis subscription when the last subscriber disconnects. The `doOnCancel` hook placed upstream of `share()` then removes the topic entry from `topicMessages`, creating a window during which any messages published to the Redis channel are silently dropped. Any unprivileged gRPC client can trigger this by being the last subscriber and cancelling, forcing a full re-subscription cycle for the next legitimate subscriber.

### Finding Description

**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()` (lines 68–80):

```java
return container
    .flatMapMany(r -> r.receive(...))
    .map(Message::getMessage)
    .doOnCancel(() -> unsubscribe(topic))   // line 74 — fires on upstream cancel
    .doOnComplete(() -> unsubscribe(topic)) // line 75
    ...
    .retryWhen(...)                         // line 78
    .share();                               // line 79 — refCount(1)
```

`unsubscribe()` (lines 82–85):
```java
private void unsubscribe(Topic topic) {
    topicMessages.remove(topic.getTopic()); // line 83 — evicts the shared Flux
}
```

`getSharedListener()` (lines 59–62):
```java
return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
```

**Root cause:**

`Flux.share()` is `publish().refCount()`. When the subscriber count drops to zero, `refCount` cancels the upstream. The cancel signal propagates through `retryWhen` and triggers `doOnCancel`, which calls `unsubscribe(topic)` → `topicMessages.remove(...)`. The entry is now gone from the map. The next call to `getSharedListener` via `computeIfAbsent` creates a brand-new `subscribe(topic)` call, which must re-establish the Redis `SUBSCRIBE` command from scratch. Any messages published to the Redis channel during the interval between `topicMessages.remove` and the new `r.receive(...)` completing its subscription handshake are permanently lost — Redis pub/sub has no replay.

**Failed assumption:** The design assumes the shared Flux in `topicMessages` is always live while at least one subscriber exists. But the `doOnCancel` hook fires *before* the map entry is replaced, creating a zero-coverage window.

**Why existing checks are insufficient:**

- `ConcurrentHashMap.computeIfAbsent` is thread-safe for map operations but does not prevent the gap between `remove` (line 83) and the next `computeIfAbsent` call.
- `retryWhen` only handles upstream *errors*, not cancellations; it does not intercept the cancel signal from `share()`.
- There is no buffering or replay mechanism between teardown and re-subscription.

### Impact Explanation

Any messages published to a Redis topic channel during the re-subscription window are silently dropped. For smart contract event log streams (the GraphQL `contractLogs` subscription described in `docs/design/contract-log-subscription.md`), this means contract log events are permanently missed by all subsequent subscribers. Downstream consumers relying on complete event streams (e.g., DeFi indexers, bridge relayers watching contract events) will observe gaps with no error signal — the stream simply resumes from the next message after re-subscription. No funds are directly at risk, but event-driven contract logic that depends on complete log delivery can behave incorrectly.

### Likelihood Explanation

No privileges are required. Any gRPC client can subscribe to a topic and then cancel (disconnect). In a low-traffic deployment or for a topic with few subscribers, an attacker can reliably be the last subscriber. The attack is repeatable: subscribe → cancel → repeat, continuously forcing re-subscription cycles and maximizing the gap window. The attacker does not need to know the exact subscriber count; they can simply subscribe, wait briefly, and cancel, relying on the probability that they are the last subscriber. The re-subscription window is bounded by the Redis connection handshake latency (typically milliseconds to tens of milliseconds), but under load or network jitter it can be longer.

### Recommendation

Replace `share()` with `publish().refCount(1, <grace-period>)` (Reactor's overload that keeps the upstream alive for a configurable grace period after the last subscriber cancels). This prevents teardown during transient subscriber-count-zero moments:

```java
.publish().refCount(1, Duration.ofSeconds(5))
```

Alternatively, move the `unsubscribe` cleanup out of `doOnCancel`/`doOnComplete` and into a scheduled cleanup that only removes the entry after confirming no new subscriber has arrived (e.g., using `computeIfPresent` with a staleness check). Additionally, consider using `publish().autoConnect(1)` if teardown on zero subscribers is not required, keeping the upstream Redis subscription permanently alive per topic.

### Proof of Concept

**Preconditions:** Mirror node running with `REDIS` listener type; attacker has network access to the gRPC endpoint (no authentication required).

**Steps:**

1. Attacker opens a gRPC stream: `subscribeTopic` for topic `0.0.X`. This causes `getSharedListener` → `computeIfAbsent` → `subscribe(topic)` → Redis `SUBSCRIBE topic.X`. The shared Flux is stored in `topicMessages`.

2. Attacker immediately cancels the gRPC stream (closes the connection or sends RST). Since the attacker is the only subscriber, `share()` refcount drops to 0 → upstream cancel → `doOnCancel` fires → `topicMessages.remove("topic.X")`.

3. A legitimate subscriber connects at this moment (or the attacker reconnects). `computeIfAbsent` creates a new `subscribe(topic)` call. The new Redis `SUBSCRIBE` handshake is in progress.

4. During the window between steps 2 and 3, the importer publishes a contract log event to Redis channel `topic.X`. Redis delivers it to no subscriber (the old subscription is torn down, the new one is not yet active). The message is permanently lost.

5. The legitimate subscriber's stream starts receiving messages only from after the re-subscription completes, with no indication that messages were missed.

**Repeatability:** Steps 1–4 can be looped continuously to cause persistent message gaps. [1](#0-0) [2](#0-1) [3](#0-2)

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
