### Title
Unauthenticated `/network/list` Flood Causes Availability Degradation Due to Disabled Rate-Limiting Middleware

### Summary
The `NetworkList()` handler returns a static in-memory value with zero I/O, making it trivially cheap to serve individually but also trivially cheap to flood at high concurrency. Go's `net/http` spawns an unbounded goroutine per accepted connection with no application-level concurrency cap, and the Traefik rate-limiting/in-flight middleware that would otherwise protect the server is **disabled by default** (`global.middleware: false`). An unprivileged attacker can open thousands of concurrent connections to `/network/list`, exhausting OS file descriptors and goroutine memory, starving all other endpoints.

### Finding Description

**Code path — handler (zero cost, zero protection):**

`rosetta/app/services/network_service.go` lines 27–32: the handler does nothing but return a pre-built struct pointer. No DB call, no lock, no allocation beyond the response object.

**Code path — HTTP server (unbounded goroutine spawn):**

`rosetta/main.go` lines 217–227: the middleware chain applied to the server is only `MetricsMiddleware → TracingMiddleware → CorsMiddleware`. None of these limit request rate or concurrency. Go's `net/http.Server` accepts each TCP connection and immediately spawns a new goroutine; there is no maximum goroutine count or connection queue depth configured.

**Code path — Traefik middleware (disabled by default):**

`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3:
```
{{ if and .Values.global.middleware .Values.middleware -}}
```
The entire Traefik `Middleware` CRD (which contains `inFlightReq: amount: 5` and `rateLimit: average: 10`) is only rendered when `global.middleware: true`. The default in `charts/hedera-mirror-rosetta/values.yaml` line 95 is `global.middleware: false`. The ingress annotation at line 121 still references the middleware chain by name, but because the CRD is never created, Traefik silently skips it and routes traffic with no rate limiting applied.

**Root cause:** The application relies entirely on an external Traefik middleware for rate limiting, but that middleware is opt-in and off by default, leaving the Go HTTP server exposed with no concurrency controls.

### Impact Explanation
With no in-flight request cap and no rate limit, an attacker can hold thousands of concurrent connections open. Each goroutine consumes ~4–8 KB of stack plus a file descriptor. At ~65 k concurrent connections (typical OS fd limit), the server stops accepting new connections entirely, making every endpoint — including DB-backed ones like `/network/status` and `/block` — unreachable. Because `NetworkList()` completes in microseconds, the attacker needs only a high-concurrency HTTP client (e.g., `wrk`, `hey`, or a simple Go program) to keep the connection count saturated. No authentication, API key, or special network position is required.

### Likelihood Explanation
Any external actor with a single machine and a standard HTTP benchmarking tool can reproduce this. The endpoint is publicly listed in the ingress path `/rosetta/network`, requires no credentials, and accepts a POST with an empty JSON body `{}`. The attack is repeatable, stateless, and requires no prior knowledge of the deployment beyond the hostname.

### Recommendation
1. **Application level (primary fix):** Add a concurrency-limiting middleware directly in `main.go` before the router, e.g. using `golang.org/x/net/netutil.LimitListener` or a semaphore-based handler wrapper, so the limit is enforced regardless of infrastructure configuration.
2. **Helm chart (secondary fix):** Change the default value of `global.middleware` to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the Traefik `inFlightReq` and `rateLimit` middlewares are active in all default deployments.
3. **Per-source in-flight cap:** The existing `inFlightReq: amount: 5` is per-source IP; ensure `ipStrategy.depth` is set correctly for deployments behind a load balancer to avoid IP spoofing bypass.

### Proof of Concept
```bash
# Flood /network/list with 5000 concurrent connections, 100k total requests
# (default Helm deployment with global.middleware: false)
hey -n 100000 -c 5000 -m POST \
    -H "Content-Type: application/json" \
    -d '{}' \
    http://<rosetta-host>/network/list

# In a second terminal, observe that /network/status becomes unresponsive:
curl -s -m 5 -X POST -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
    http://<rosetta-host>/network/status
# Expected during flood: connection timeout or "connection refused"
