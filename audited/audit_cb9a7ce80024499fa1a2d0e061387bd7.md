### Title
Unbounded Repeated DB Queries via Uncached, Unthrottled `TopicServiceImpl.findById()` Enable Mirror Node DB Throughput Exhaustion

### Summary
`TopicServiceImpl.findById()` in the rest-java module issues a direct database query on every invocation with no result caching and no request-rate limiting. An unprivileged attacker can flood `GET /api/v1/topics/{id}` with concurrent requests, exhausting the shared DB connection pool and degrading all mirror node database throughput by 30% or more.

### Finding Description
**Exact code path:**
- `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java` line 33: `topicService.findById(id.id())`
- `rest-java/src/main/java/org/hiero/mirror/restjava/service/TopicServiceImpl.java` line 20: `topicRepository.findById(id.getId())`
- `rest-java/src/main/java/org/hiero/mirror/restjava/repository/TopicRepository.java`: plain `CrudRepository<Topic, Long>` — no `@Cacheable`

**Root cause:** The service layer applies no `@Cacheable` annotation and the repository has no cache decoration. Every HTTP request results in a synchronous JDBC round-trip to the database. The rest-java module registers no rate-limiting filter or bucket (unlike the `web3` module which has `ThrottleConfiguration` / `ThrottleManagerImpl`). The only registered filter is `ShallowEtagHeaderFilter`, which only signals cooperative HTTP clients to use their local cache; it does not suppress the server-side DB query.

**Why existing checks fail:** `ShallowEtagHeaderFilter` computes an ETag from the response body and returns HTTP 304 only if the client sends a matching `If-None-Match` header. An attacker simply omits that header, forcing a full DB query on every request. There is no server-side guard.

### Impact Explanation
The mirror node's rest-java service shares a bounded JDBC connection pool with all other DB-backed operations (entity lookups, custom fee lookups, message queries, etc.). Flooding `GET /api/v1/topics/{id}` with high concurrency saturates the pool, causing queuing and timeouts across all mirror node DB operations. This degrades overall mirror node DB throughput well beyond the 30% threshold, effectively making the mirror node unavailable for all consumers (wallets, explorers, dApps) that depend on it.

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly reachable. A single attacker with a modest botnet or even a single machine with async HTTP tooling (e.g., `wrk`, `hey`, `ab`) can sustain thousands of requests per second. The attack is trivially repeatable and requires no special knowledge beyond the public API path `/api/v1/topics/{id}`.

### Recommendation
1. **Add `@Cacheable`** to `TopicRepository.findById` (or at the `TopicServiceImpl` level) using a short TTL (e.g., 5–30 seconds), consistent with how `grpc/EntityRepository` and `web3/EntityRepository` are already protected.
2. **Add rate limiting** to the rest-java module for the `/api/v1/topics/{id}` endpoint, mirroring the `ThrottleConfiguration` / `ThrottleManagerImpl` pattern already present in the `web3` module.
3. Consider deploying an API gateway or reverse proxy with per-IP rate limiting as a defense-in-depth layer.

### Proof of Concept
```bash
# No authentication required. Run from any machine.
# Install: apt install apache2-utils  (or use wrk/hey)

# Flood the endpoint with 500 concurrent connections for 30 seconds
ab -n 1000000 -c 500 http://<mirror-node-host>/api/v1/topics/0.0.1234

# Observe: DB connection pool exhaustion in mirror node logs,
# increased latency / errors on ALL other mirror node API endpoints,
# e.g., GET /api/v1/accounts/{id} begins timing out.
```

Each request bypasses the ETag filter by omitting `If-None-Match`, forcing a full `SELECT * FROM topic WHERE id = ?` on every call, with no server-side throttle to stop it.
