### Title
Unauthenticated Timestamp Filter Forces Expensive UNION Historical Table Scan in `getContractByIdOrAddressContractEntityQuery()`

### Summary
Any unauthenticated external user can supply a `timestamp=` query parameter to `GET /api/v1/contracts/:contractId`, causing `getContractByIdOrAddressContractEntityQuery()` to execute a UNION across `entity` and `entity_history` with `ORDER BY timestamp_range DESC LIMIT 1` instead of the cheaper single-table path. Because the REST API has no rate limiting, an attacker can flood this endpoint with timestamp-filtered requests, forcing repeated historical-table joins and sorts that degrade database performance for all other users.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js`, the handler `getContractById` (lines 707–724) accepts any filters validated by `buildAndValidateFilters` against `acceptedContractByIdParameters`, then calls `utils.extractTimestampRangeConditionFilters(filters)` to produce `timestampConditions` and `timestampParams`, and passes them directly to `getContractByIdOrAddressContractEntityQuery`. [1](#0-0) 

Inside `getContractByIdOrAddressContractEntityQuery` (lines 179–208), the branch at line 194 is taken whenever `timestampConditions.length !== 0`: [2](#0-1) 

This produces:
```sql
SELECT ... FROM entity e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND c.timestamp_range && $2 AND e.id = $3
UNION
SELECT ... FROM entity_history e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND c.timestamp_range && $2 AND e.id = $3
ORDER BY timestamp_range DESC
LIMIT 1
```

Without a timestamp filter, only the `entity` table is queried (single fast lookup). With any timestamp filter, both `entity` and `entity_history` are scanned, joined to `contract`, sorted, and limited — a materially more expensive plan.

**Root cause:** The decision to UNION `entity_history` is gated solely on `timestampConditions.length !== 0`, which is controlled entirely by user-supplied query parameters. No rate limiting exists in the REST API layer. [3](#0-2) 

**Why existing checks are insufficient:**

`buildAndValidateFilters` validates the format of each filter value (timestamp syntax, operator legality) but does not cap the number of timestamp parameters or the rate of requests. [4](#0-3) 

`extractTimestampRangeConditionFilters` in `utils.js` iterates all timestamp filters and emits one condition and one range param per filter — no count cap. [5](#0-4) 

A search of the REST Node.js service confirms zero rate-limiting middleware is registered: [6](#0-5) 

(The throttling found in `web3/` applies only to the separate Java web3 service, not the REST mirror-node API.)

### Impact Explanation
An attacker sending a high volume of requests such as `GET /api/v1/contracts/0.0.1?timestamp=gte:1&timestamp=lte:9999999999` forces the database to repeatedly execute the UNION + sort plan against `entity_history`, which grows unboundedly as the network ages. Under sustained load this degrades query latency for all concurrent users of the mirror node REST API. No funds are at risk; the impact is availability degradation (griefing).

### Likelihood Explanation
The exploit requires no credentials, no special knowledge beyond the public API spec, and is trivially scriptable with any HTTP client. The `entity_history` table is expected to be large on a production mirror node, making the cost differential between the two query paths significant. Repeatability is unlimited given the absence of rate limiting.

### Recommendation
1. Add a per-IP or global rate limiter (e.g., `express-rate-limit`) to the REST API, specifically for endpoints that can trigger historical queries.
2. Cap the number of accepted `timestamp` filter repetitions per request (e.g., reject requests with more than 2 timestamp parameters).
3. Consider adding a database-level statement timeout for REST queries to bound worst-case execution time.
4. Ensure `entity_history` has a composite index on `(id, timestamp_range)` so the UNION branch uses index scans rather than sequential scans.

### Proof of Concept
```bash
# No authentication required
# Trigger the expensive UNION path repeatedly
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1?timestamp=gte:1&timestamp=lte:9999999999" &
done
wait
```

Each request causes the database to execute:
```sql
SELECT ... FROM entity ... WHERE ... AND c.timestamp_range && $1 AND e.id = $3
UNION
SELECT ... FROM entity_history ... WHERE ... AND c.timestamp_range && $1 AND e.id = $3
ORDER BY timestamp_range DESC LIMIT 1
```
instead of the cheap single-table lookup, with no server-side throttle to prevent flooding.

```javascript
const extractTimestampRangeConditionFilters = (
  filters,
  offset = 0,
  timestampRangeColumn = Entity.getFullName(Entity.TIMESTAMP_RANGE)
) => {
  const conditions = [];
  const params = [];

  filters
    .filter((filter) => filter.key === constants.filterKeys.TIMESTAMP)
    .forEach((filter) => {
      const position = `$${params.length + offset + 1}`;
      let condition;
      let range;

      if (filter.operator === opsMap.ne) {
        // handle ne filter differently
        condition = `not ${timestampRangeColumn} @> ${position}`; // @> is the pg range "contains" operator
        range = Range(filter.value, filter.value, '[]');
      } else {
        condition = `${timestampRangeColumn} && ${position}`; // && is the pg range "overlaps" operator

        switch (filter.operator) {
          case opsMap.lt:
            range = Range(null, filter.value, '()');
            break;
          case opsMap.eq:
          case opsMap.lte:
            range = Range(null, filter.value, '(]');
            break;
          case opsMap.gt:
            range = Range(filter.value, null, '()');
            break;
          case opsMap.gte:
            range = Range(filter.value, null, '[)');
            break;
        }
      }

      conditions.push(condition);
      params.push(range);
    });

  return {
    conditions,
    params,
  };
```
