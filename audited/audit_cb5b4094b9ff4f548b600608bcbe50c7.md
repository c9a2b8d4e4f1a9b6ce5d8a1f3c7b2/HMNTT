### Title
Reference Identity Check Bypass in `findByBlockType` Allows `"0x0"` to Silently Skip Genesis Block Lookup

### Summary
`RecordFileServiceImpl.findByBlockType()` guards the `findEarliest()` path with a reference-equality check (`block == BlockType.EARLIEST`). `BlockType.of("0x0")` constructs a **new** `BlockType` object with `number=0` that is not the `EARLIEST` singleton, bypassing the guard and routing to `findByIndex(0)` instead. If the `record_file` table has ever had its indices shifted via `updateIndex()`, `findByIndex(0)` returns `Optional.empty()` while `findEarliest()` would have returned the true genesis block.

### Finding Description

**Root cause — reference equality on a record type:** [1](#0-0) 

Line 20 uses `==` (reference identity). `BlockType` is a Java `record`; two records with identical fields are `.equals()` but are **not** `==` unless they are the same object.

**How `BlockType.of("0x0")` produces a non-singleton:** [2](#0-1) [3](#0-2) 

`"0x0"` does not match `"earliest"` in the switch, so it falls to `extractNumericBlock`. That method strips `"0x"`, parses `"0"` as hex → `0L`, and returns `new BlockType("0x0", 0L)` — a **fresh heap object**, not `EARLIEST`. The `==` guard on line 20 therefore evaluates to `false`, and execution falls through to:

```java
return recordFileRepository.findByIndex(block.number()); // findByIndex(0)
```

**Semantic difference between the two repository methods:** [4](#0-3) 

- `findEarliest()`: `SELECT * FROM record_file ORDER BY index ASC LIMIT 1` — always returns the record with the **lowest index**, regardless of its numeric value.
- `findByIndex(0)`: `SELECT … WHERE r.index = 0` — returns a record only if one exists with `index` exactly equal to `0`.

**The re-indexing precondition is real and present in the codebase:** [5](#0-4) 

`updateIndex(long offset)` executes `UPDATE record_file SET index = index + ?1`. After any such call, the genesis block's `index` is no longer `0`; `findByIndex(0)` returns `Optional.empty()` while `findEarliest()` still returns the genesis block.

**Exploit flow:**
1. Attacker (no privileges required) sends an Ethereum JSON-RPC or REST request with block parameter `"0x0"` (e.g., `eth_getBlockByNumber("0x0", false)`).
2. `BlockType.of("0x0")` is invoked via `@JsonCreator`.
3. A new `BlockType("0x0", 0)` is returned — not `EARLIEST`.
4. `findByBlockType` routes to `findByIndex(0)`.
5. If indices have been shifted, the call returns `Optional.empty()`.
6. The caller receives a "block not found" response for the genesis block.

### Impact Explanation
Any consumer of the mirror node API (explorer, wallet, bridge, audit tool) that queries block `"0x0"` to retrieve the genesis/earliest block will receive an empty response instead of the true genesis block. This hides the start of Hashgraph history from external observers and can break applications that rely on genesis-block data for chain validation, trust anchoring, or historical replay. The impact is a silent data-integrity failure, not a crash, making it harder to detect.

### Likelihood Explanation
The trigger requires no authentication or special privilege — any caller of the public JSON-RPC endpoint can supply `"0x0"`. The re-indexing precondition (`updateIndex`) is an internal importer operation, but it exists in production code and can be invoked during maintenance or migration. Once indices are shifted, every subsequent `"0x0"` query silently fails. The bug is repeatable and deterministic.

### Recommendation
Replace the reference-equality checks with value-equality checks using the record's auto-generated `equals()`, or compare by `name` field:

```java
// Instead of:
if (block == BlockType.EARLIEST)

// Use:
if (BlockType.EARLIEST.equals(block))
// or equivalently, since BlockType is a record:
if (block.name().equalsIgnoreCase("earliest") && block.number() == 0)
```

Alternatively, normalize `BlockType.of("0x0")` to return the `EARLIEST` singleton directly:

```java
// In extractNumericBlock, after parsing:
if (blockNumber == 0L) return EARLIEST;
```

Also add a unit test asserting `BlockType.of("0x0") == BlockType.EARLIEST` (or `.equals(EARLIEST)`) to prevent regression.

### Proof of Concept
```

{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["0x0",false],"id":1}

# 4. Observed result: null / block-not-found

# 5. Same request with "earliest" returns the genesis block correctly,

```
