### Title
Missing Timestamp Filtering in `ContractBytecodeReadableKVState` Causes Historical `eth_call` to Execute Wrong Contract Bytecode

### Summary
`ContractBytecodeReadableKVState.readFromDataSource()` unconditionally calls `ContractRepository.findRuntimeBytecode()`, a query with no timestamp predicate (`select runtime_bytecode from contract where id = :contractId`). Every other `ReadableKVState` implementation in the same package reads `ContractCallContext.get().getTimestamp()` and branches to a timestamp-filtered query for historical calls. The bytecode state does not, so any `eth_call` at a past block executes the **current** runtime bytecode against historical storage/account state, producing results that never occurred on-chain.

### Finding Description

**Exact code path:**

`ContractBytecodeReadableKVState.readFromDataSource()` (line 39–47) never consults `ContractCallContext`:

```java
// ContractBytecodeReadableKVState.java lines 39-47
@Override
protected Bytecode readFromDataSource(@NonNull ContractID contractID) {
    final var entityId = toEntityId(contractID);
    return contractRepository
            .findRuntimeBytecode(entityId.getId())   // ← always current, no timestamp
            .map(Bytes::wrap)
            .map(Bytecode::new)
            .orElse(null);
}
``` [1](#0-0) 

The underlying query is:

```sql
select runtime_bytecode from contract where id = :contractId
``` [2](#0-1) 

**Contrast with every other state reader.** `ContractStorageReadableKVState.readFromDataSource()` (the storage slot reader for the same contract service) explicitly reads the historical timestamp and dispatches to a timestamp-filtered query:

```java
final var timestamp = ContractCallContext.get().getTimestamp();
return timestamp
    .map(t -> contractStateService.findStorageByBlockTimestamp(..., t))
    .orElse(contractStateService.findStorage(...))
``` [3](#0-2) 

The same pattern is used by `NftReadableKVState`, `TokenReadableKVState`, `FileReadableKVState`, `AccountReadableKVState`, `AliasesReadableKVState`, `ScheduleReadableKVState`, and `TokenRelationshipReadableKVState` — all call `ContractCallContext.get().getTimestamp()` and branch accordingly. [4](#0-3) [5](#0-4) 

**`ContractCallContext` provides the historical timestamp** when `block != BlockType.LATEST`:

```java
public boolean useHistorical() {
    return callServiceParameters != null && callServiceParameters.getBlock() != BlockType.LATEST;
}
public Optional<Long> getTimestamp() {
    if (useHistorical()) { return getTimestampOrDefaultFromRecordFile(); }
    return Optional.empty();
}
``` [6](#0-5) 

`ContractBytecodeReadableKVState` never calls `ContractCallContext.get()` at all.

**Secondary issue — `@Cacheable` cache poisoning.** `findRuntimeBytecode` is annotated `@Cacheable`, so the first call (whether historical or current) populates the cache keyed only on `contractId`. A subsequent call for the same contract at a different block returns the cached (wrong) bytecode. [7](#0-6) 

Also, `toEntityId()` passes `Optional.empty()` to `getEntityByEvmAddressAndTimestamp` unconditionally, ignoring the historical timestamp even for EVM-address-keyed lookups:

```java
return commonEntityAccessor
    .getEntityByEvmAddressAndTimestamp(evmAddress, Optional.empty())  // ← always empty
``` [8](#0-7) 

### Impact Explanation
When a contract's bytecode is updated (Hedera supports `ContractUpdateTransaction` to replace a contract's code file), any `eth_call` at a block prior to the update will execute the **post-update** bytecode against **pre-update** storage and account state. The EVM simulation produces results that never occurred on-chain. Applications using historical simulations for auditing, debugging, or financial reconstruction (e.g., replaying past transactions to verify outcomes) receive silently incorrect data. The design document explicitly states that historical state must be loaded via the timestamp from `ContractCallContext`; bytecode is the only state type that violates this contract.



### Likelihood Explanation
Any unprivileged user can trigger this by issuing a standard JSON-RPC `eth_call` with a `blockNumber` parameter set to any past block. No authentication, special role, or privileged access is required. The precondition — a contract whose bytecode was updated after deployment — is uncommon in pure EVM chains but is a supported and documented Hedera operation (`ContractUpdateTransaction`). The bug is deterministic and 100% reproducible whenever the precondition is met.

### Recommendation
1. In `ContractBytecodeReadableKVState.readFromDataSource()`, read `ContractCallContext.get().getTimestamp()` and, when a historical timestamp is present, query a timestamp-filtered source (e.g., a `contract_history` table or a query like `select runtime_bytecode from contract_history where id = :contractId and lower(timestamp_range) <= :timestamp order by lower(timestamp_range) desc limit 1`).
2. Remove the `@Cacheable` annotation from `findRuntimeBytecode()`, or make the cache key include the timestamp so historical and current results are stored separately.
3. Fix `toEntityId()` to pass the historical timestamp from `ContractCallContext` to `getEntityByEvmAddressAndTimestamp` instead of `Optional.empty()`.

### Proof of Concept
**Preconditions:**
- Contract `0xABCD` is deployed at block 100 with bytecode `V1` (e.g., a function returning `1`).
- At block 200, the contract's bytecode is updated to `V2` (function returns `2`).
- Mirror node DB: `contract` table row for `0xABCD` holds `runtime_bytecode = V2`.

**Steps:**
1. Send `eth_call { to: "0xABCD", data: "<call selector>", blockNumber: "0x64" }` (block 100, before the update).
2. `ContractCallContext.useHistorical()` returns `true`; `getTimestamp()` returns the consensus timestamp of block 100.
3. `ContractBytecodeReadableKVState.readFromDataSource()` is invoked; it calls `findRuntimeBytecode(contractId)` with no timestamp.
4. The query returns `V2` (current bytecode).
5. The EVM executes `V2` against block-100 storage state.
6. The call returns `2` instead of the historically correct `1`.

**Expected result:** `eth_call` at block 100 should return `1` (V1 bytecode).
**Actual result:** `eth_call` at block 100 returns `2` (V2 bytecode), silently incorrect.
