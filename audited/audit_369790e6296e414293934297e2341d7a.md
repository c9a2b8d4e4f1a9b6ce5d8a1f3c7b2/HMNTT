### Title
Unauthenticated Alias-Based DB I/O Waste via Missing Entity-Type Filter in SQL Query

### Summary
`EntityRepository.findByAlias()` executes a database query that does not filter by entity type. The type check is applied in Java *after* the database round-trip completes. An unprivileged attacker who knows the alias of any non-ACCOUNT entity (e.g., a CONTRACT) can repeatedly trigger real database hits that always produce a discarded result, wasting DB I/O at will with no rate limiting in the GraphQL module.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 48) calls: [1](#0-0) 

which delegates to `EntityServiceImpl.getByAliasAndType()`: [2](#0-1) 

which calls the repository: [3](#0-2) 

**Root cause:** The SQL query `select * from entity where alias = ?1 and deleted is not true` contains no `type` predicate. The entity type check (`.filter(e -> e.getType() == type)`) is applied in the Java layer *after* the database row has already been fetched. When the alias belongs to a CONTRACT entity, the DB performs a full index lookup, returns the row, and then Java silently discards it, returning `Optional.empty()`. The database work is entirely wasted.

**Exploit flow:**
1. Attacker enumerates or observes a known CONTRACT alias (aliases are public Hedera network data).
2. Attacker sends repeated unauthenticated POST requests to `/graphql/alpha`:
   ```graphql
   { account(input: { alias: "KNOWNCONTRACTALIAS" }) { id } }
   ```
3. Each request: decodes the alias → executes the DB query → finds the CONTRACT row → Java filter discards it → returns `null` to the caller.
4. The attacker loops this at high volume. Every iteration is a real DB round-trip with zero useful output.

### Impact Explanation
The mirror node's PostgreSQL instance absorbs repeated indexed reads against the `entity` table that produce no useful result. Under sustained high-volume attack, this degrades DB throughput for all legitimate queries. Since the mirror node is a shared read service, degraded DB performance affects all consumers. The impact is limited to the mirror node infrastructure (no on-chain economic damage), consistent with the Medium/griefing classification.

### Likelihood Explanation
No authentication or authorization is required to reach the GraphQL endpoint. [4](#0-3) 

The only protections present are query complexity/depth limits and parser token limits — none of which limit the *rate* of distinct requests: [5](#0-4) 

The rate-limiting infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not applied to the GraphQL module at all. [6](#0-5) 

CONTRACT aliases are observable on-chain, making the precondition trivially satisfiable by any attacker.

### Recommendation
1. **Push the type filter into SQL**: Change `findByAlias` to accept an `EntityType` parameter and add `and type = ?2` to the query, eliminating the wasted DB round-trip entirely.
2. **Add rate limiting to the GraphQL module**: Apply a per-IP or global request-rate bucket (analogous to `ThrottleConfiguration` in `web3`) to the `/graphql/alpha` endpoint.
3. **Optional caching**: Cache negative results (alias → empty) for a short TTL to absorb repeated identical queries.

### Proof of Concept
```bash
# 1. Identify a CONTRACT entity alias from public Hedera mirror data
CONTRACT_ALIAS="KNOWNBASE32ALIAS"   # e.g. from /api/v1/contracts

# 2. Fire high-volume requests (no auth required)
for i in $(seq 1 10000); do
  curl -s -X POST https://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: { alias: \\\"$CONTRACT_ALIAS\\\" }) { id } }\"}" &
done

# Result: each request hits the DB (findByAlias executes), finds the CONTRACT row,
# Java discards it (type != ACCOUNT), returns null.
# DB I/O is consumed for every request with zero useful output.
```
