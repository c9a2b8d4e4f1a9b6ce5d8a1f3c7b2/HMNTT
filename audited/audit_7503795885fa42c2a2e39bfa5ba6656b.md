### Title
GraphQL `findByAlias` Leaks Entities with `deleted = NULL` Due to `IS NOT TRUE` vs `<> true` SQL Predicate Inconsistency

### Summary
The GraphQL `EntityRepository.findByAlias` uses `deleted is not true` which, under SQL three-valued logic, evaluates to TRUE for both `deleted = false` AND `deleted = NULL`, causing entities in an indeterminate deletion state to be returned. Every other service (importer, rest-java) uses `deleted <> true`, which correctly excludes NULL rows. An unprivileged external user can query the public GraphQL endpoint with a known alias and retrieve an entity that all other services treat as non-existent, producing inconsistent ledger-state visibility across the mirror node stack.

### Finding Description
**Exact code locations:**

- GraphQL: `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 13:
  `"select * from entity where alias = ?1 and deleted is not true"`
- Importer: `importer/src/main/java/org/hiero/mirror/importer/repository/EntityRepository.java`, line 16:
  `"select id from entity where alias = ?1 and deleted <> true"`
- Rest-Java: `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java`, line 13:
  `"select id from entity where alias = ?1 and deleted <> true"`

**Root cause — SQL three-valued logic:**

| `deleted` value | `deleted IS NOT TRUE` | `deleted <> true` |
|---|---|---|
| `false` | TRUE (row returned) | TRUE (row returned) |
| `NULL` | TRUE **(row returned)** | NULL → FALSE **(row excluded)** |
| `true` | FALSE (row excluded) | FALSE (row excluded) |

`IS NOT TRUE` is a boolean test that treats NULL as "not true"; `<> true` is a comparison that propagates NULL as unknown/false. The GraphQL query therefore surfaces any entity whose `deleted` column is NULL.

**Failed assumption:** The GraphQL repository assumes `deleted is not true` is equivalent to `deleted <> true`. It is not when `deleted` can be NULL, which is a valid database state (e.g., entities ingested before a deletion-status field was populated, or entities mid-import where the importer has not yet written a definitive `deleted` value).

**Exploit flow:**
1. An entity row exists in the `entity` table with `alias = <known_bytes>` and `deleted = NULL`.
2. Attacker calls the public GraphQL endpoint: `{ account(alias: "<base32_alias>") { ... } }` via `EntityServiceImpl.getByAliasAndType` → `EntityRepository.findByAlias`.
3. The query `deleted is not true` matches the NULL row; the entity is returned in full.
4. The same alias queried through the importer's `EntityIdServiceImpl.findByAlias` → importer `EntityRepository.findByAlias` returns empty (`deleted <> true` excludes NULL).
5. Rest-Java similarly returns empty for the same alias.

**Why existing checks are insufficient:**
- The GraphQL test suite (`EntityRepositoryTest`, lines 22–34) only tests a normally-created entity (default `deleted = false`). There is no test for `deleted = null`, unlike the rest-java test suite which explicitly asserts `findByAlias(entityDeletedNull.getAlias())` is empty (lines 23–28).
- No application-layer guard in `EntityServiceImpl` checks the `deleted` field after retrieval; it only filters on `EntityType` (line 30).

### Impact Explanation
An attacker can observe entities via GraphQL that the importer and rest-java treat as non-existent. This creates a split-brain view of ledger state: GraphQL reports an account as resolvable by alias while the transaction-processing pipeline (importer) cannot resolve the same alias to any entity. Downstream consumers relying on GraphQL for account lookups will see a different set of active accounts than consumers relying on the REST or importer APIs, enabling inconsistent transaction history visibility — e.g., an account appears to exist and have a history in GraphQL but is invisible to alias-based resolution in all other services. Severity is medium: no funds can be moved, but ledger-state integrity guarantees are violated for affected entities.

### Likelihood Explanation
Entities with `deleted = NULL` are a realistic database state: any entity row written before the `deleted` column was introduced, or any entity whose import processing was interrupted before a definitive deletion status was committed, will have `deleted = NULL`. No special privilege is required — the GraphQL endpoint is public. The attacker only needs to know (or enumerate) a valid alias, which is derivable from public key material. The condition is repeatable for every such entity in the database.

### Recommendation
Replace `deleted is not true` with `deleted <> true` (or equivalently `deleted = false`) in both queries in the GraphQL `EntityRepository`:

```java
// graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java

@Query(value = "select * from entity where alias = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

Add test coverage mirroring the rest-java pattern:
```java
var entityDeletedNull = domainBuilder.entity().customize(b -> b.deleted(null)).persist();
assertThat(entityRepository.findByAlias(entityDeletedNull.getAlias())).isEmpty();
```

### Proof of Concept
**Precondition:** Insert an entity with `deleted = NULL` and a known alias into the mirror node database:
```sql
INSERT INTO entity (id, alias, deleted, type, ...)
VALUES (9999, '\xDEADBEEF...', NULL, 'ACCOUNT', ...);
```

**Trigger:** Query the GraphQL endpoint (no authentication required):
```graphql
query {
  account(input: { alias: "<base32_of_DEADBEEF...>" }) {
    id
    alias
    deleted
  }
}
```

**Result:** GraphQL returns the entity with `deleted: null`.

**Contrast:** Query the importer's alias resolution (internal) or rest-java endpoint with the same alias → entity not found (empty result), confirming the split-brain inconsistency.
