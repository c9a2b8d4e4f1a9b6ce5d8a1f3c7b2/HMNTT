### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `getScheduleById` Requests Blocks Co-Signers from Retrieving `transaction_body`

### Summary
The `getScheduleById` handler in `rest/schedules.js` issues a complex three-table aggregation query against a shared `pg` connection pool whose default maximum is 10 connections, with no rate limiting, no per-IP concurrency cap, and no response caching for active (non-expired, non-executed) schedules. An unprivileged attacker can saturate all pool connections with concurrent requests, causing legitimate co-signers' requests to queue and time out, preventing them from retrieving the `transaction_body` they need to submit a `ScheduleSign` transaction before the schedule expires.

### Finding Description

**Exact code path:**

`rest/schedules.js`, `getScheduleById()`, line 120: [1](#0-0) 

The function calls `pool.queryQuietly(getScheduleByIdQuery, scheduleId)` unconditionally. `queryQuietly` with no `preQueryHint` resolves to `this.query(query, params)` — the standard `pg` Pool method — which acquires a physical connection, executes the query, and releases it: [2](#0-1) 

The query itself is a three-table LEFT JOIN with a `json_agg` window aggregation over `transaction_signature`: [3](#0-2) 

**Pool size:**

The pool is initialized with `max: config.db.pool.maxConnections`: [4](#0-3) 

The documented default is **10 connections**, with a `connectionTimeoutMillis` of **20 000 ms** and `statement_timeout` of **20 000 ms**: [5](#0-4) 

**No rate limiting on the REST API:**

`server.js` registers the endpoint with no throttle middleware: [6](#0-5) 

The middleware stack contains only logging, optional basic-auth, optional metrics, and optional response cache — no per-IP or per-endpoint concurrency control: [7](#0-6) 

The throttling that does exist (bucket4j `ThrottleManagerImpl`) is scoped exclusively to the `web3` Java service's `/contracts/call` endpoint and does not apply here: [8](#0-7) 

**Response cache does not protect active schedules:**

`applicationCacheEnabled` is `false` by default (`cache.response.enabled` defaults to `false`). Even when enabled, `getScheduleCacheControlHeader` returns `{}` (no cache header) for schedules that have not yet executed or expired — precisely the schedules that need co-signing: [9](#0-8) 

**Root cause / failed assumption:** The design assumes that the database pool and the upstream infrastructure (reverse proxy, CDN) will absorb burst traffic. Neither assumption is enforced in code for this endpoint.

### Impact Explanation

A scheduled transaction on Hedera requires co-signers to submit `ScheduleSign` transactions before the schedule expires. Co-signers use the Mirror Node REST API to retrieve the `transaction_body` field (base64-encoded protobuf) to know what they are signing. If the pool is exhausted, their GET requests queue up and eventually fail with a `DbError` wrapping a `connection timeout` after 20 seconds. Schedules auto-expire after 31 minutes (no explicit expiration) or at their configured `expiration_time + 60 s`. A sustained attack during the signing window causes the schedule to expire without reaching the required signature threshold, permanently preventing execution of the scheduled transaction. This is a direct, protocol-level denial of the co-signing workflow.

### Likelihood Explanation

The endpoint is public and unauthenticated. Saturating 10 pool connections requires only ~10–20 concurrent HTTP requests whose queries take ≥50 ms each (realistic for the three-table `json_agg` join on a loaded database). This is achievable with a single machine using `curl --parallel`, `ab`, `wrk`, or a trivial script. The attacker needs only to know a valid `scheduleId` (all schedule IDs are publicly enumerable via `GET /api/v1/schedules`). The attack is repeatable, requires no credentials, and can be sustained indefinitely.

### Recommendation

1. **Add a per-IP rate limiter** (e.g., `express-rate-limit`) to the REST API, applied globally or specifically to the `/schedules/:scheduleId` route, to cap concurrent requests per source IP.
2. **Increase the pool size** or configure a maximum queue depth (`pg` supports `maxQueue` via `pg-pool`) so that overflow requests are rejected immediately with HTTP 503 rather than queuing silently.
3. **Enable the Redis response cache** for active schedules with a short TTL (e.g., 5–10 s). The `getScheduleCacheControlHeader` logic should be extended to emit a short `max-age` for non-terminal schedules so the cache layer absorbs repeated reads of the same `scheduleId`.
4. **Add a query result cache** (in-process LRU, similar to the existing `entityId` cache) keyed on `scheduleId` with a short TTL to avoid hitting the database on every request for the same schedule.

### Proof of Concept

```bash
# 1. Enumerate a valid scheduleId
SCHEDULE_ID=$(curl -s "https://<mirror-node>/api/v1/schedules?limit=1" \
  | jq -r '.schedules[0].schedule_id')

# 2. Saturate the pool with 50 concurrent requests (well above the default max of 10)
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/schedules/${SCHEDULE_ID}" &
done
wait

# 3. While the above is running, a legitimate co-signer attempts to retrieve transaction_body
curl -v "https://<mirror-node>/api/v1/schedules/${SCHEDULE_ID}"
# Expected result: connection timeout error (DbError) or HTTP 500 after ~20 s
# The co-signer cannot retrieve transaction_body and cannot submit ScheduleSign
```

const getScheduleCacheControlHeader = (schedule) => {
  const nowNs = utils.nowInNs();
  const executedTimestamp = schedule.executed_timestamp;
  const expirationTime = schedule.expiration_time;
  const consensusTimestamp = schedule.consensus_timestamp;

  const hasExecuted = executedTimestamp !== null || schedule.deleted;
  const hasAutoExpired =
    expirationTime === null && nowNs >= consensusTimestamp + constants.THIRTY_ONE_MINUTES * constants.NANOS_PER_SECOND;
  const hasExpired =
    expirationTime !== null && nowNs >= expirationTime + constants.SIXTY_SECONDS * constants.NANOS_PER_SECOND;

  if (hasExecuted || hasAutoExpired || hasExpired) {
    return LONGER_SCHEDULE_CACHE_CONTROL_HEADER;
  }
  return {};
};
