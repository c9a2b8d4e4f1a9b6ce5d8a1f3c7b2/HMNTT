### Title
Unbounded Timestamp Range in Historical Hook Storage Query Causes Full-Table Scan DoS

### Summary
An unprivileged external user can supply a single timestamp query parameter (e.g., `timestamp=gte:0`) to the `/api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint, triggering the historical code path with an effective timestamp range of `[0, Long.MAX_VALUE]`. This forces the database to perform a full scan of every `hook_storage_change` row for the targeted `(owner_id, hook_id)` pair, followed by an in-memory `DISTINCT ON (key)` sort, with no intermediate row-count limit. Repeated requests can exhaust PostgreSQL sort memory (`work_mem`) and saturate DB CPU.

### Finding Description

**Trigger path:**

`HookStorageRequest.isHistorical()` returns `!timestamp.isEmpty()`, and `Bound.isEmpty()` is `lower == null && upper == null`. [1](#0-0) 

Supplying a single timestamp parameter (e.g., `timestamp=gte:0`) creates a `Bound` with only a lower value set, making `isEmpty()` false and `isHistorical()` true.

**Default bounds when one side is absent:**

- `getAdjustedLowerRangeValue()` returns `0` when `lower == null`.
- `adjustUpperBound()` returns `Long.MAX_VALUE` when `upper == null`. [2](#0-1) 

So `timestamp=gte:0` alone produces `timestampLowerBound = 0`, `timestampUpperBound = Long.MAX_VALUE` — the entire network history.

**Service delegates to the expensive query:** [3](#0-2) 

**The native SQL query:**
```sql
select distinct on (key)
     owner_id, hook_id, key, value_written as "value", ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id  = :hookId
  and key >= :keyLowerBound
  and key <= :keyUpperBound
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
``` [4](#0-3) 

PostgreSQL's `DISTINCT ON (key)` must sort **all** rows matching the WHERE clause by `(key, consensus_timestamp DESC)` (injected by `Pageable`) before it can emit the first deduplicated row. The `LIMIT` from `Pageable` is applied only after this full sort; it does not reduce the intermediate scan. With `consensus_timestamp BETWEEN 0 AND Long.MAX_VALUE`, every row for the targeted hook is scanned and sorted.

**Why existing checks are insufficient:**

- `@Size(max = 2)` on `timestamps` limits the number of timestamp parameters to 2, not the width of the range.
- `@Max(MAX_LIMIT)` (`limit ≤ 100`) caps output rows, not the intermediate sort set.
- `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on `keys` is irrelevant when no keys are supplied (the broader `findByKeyBetweenAndTimestampBetween` path is taken).
- No statement timeout, no `MAX ROWS` guard, and no range-width validation exist anywhere in the path. [5](#0-4) 

### Impact Explanation
Every request with a wide timestamp range forces PostgreSQL to sort the entire `hook_storage_change` partition for the targeted `(owner_id, hook_id)`. A hook with millions of storage-change records (normal for an active EVM hook) will cause repeated disk spills when the sort exceeds `work_mem`, consuming I/O, CPU, and memory. Sending this request in a tight loop from a single client can increase DB processing load by well over 30% compared to baseline, meeting the stated impact threshold. Because the query is scoped to one `(owner_id, hook_id)` pair, the attacker only needs to identify one high-activity hook — information that is publicly observable on-chain.

### Likelihood Explanation
No authentication or authorization is required. `ownerId` and `hookId` are public blockchain identifiers discoverable by any observer. The endpoint is reachable by any HTTP client. The exploit requires a single crafted GET request and is trivially repeatable in a loop. No special tooling, credentials, or elevated privileges are needed.

### Recommendation
1. **Enforce a maximum timestamp range width**: Reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 7 days in nanoseconds).
2. **Require both bounds for historical queries**: If `isHistorical()` is true, validate that both a lower and an upper timestamp are explicitly provided.
3. **Add a DB statement timeout**: Configure `statement_timeout` at the connection or query level for REST-Java DB sessions to cap runaway queries.
4. **Add a `LIMIT` inside the subquery**: Wrap the `DISTINCT ON` in a subquery with an inner `LIMIT` guard before the outer `LIMIT` is applied, or use a `FETCH FIRST N ROWS ONLY` hint.
5. **Index**: Ensure a covering index on `(owner_id, hook_id, key, consensus_timestamp DESC)` exists so PostgreSQL can use an index scan for `DISTINCT ON` rather than a full sort.

#   SELECT DISTINCT ON (key) ... FROM hook_storage_change

#     AND consensus_timestamp BETWEEN 0 AND 9223372036854775807

done

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }

    public RangeParameter<Long> adjustLowerRange() {
        if (this.hasEqualBounds()) {
            // If the primary param has a range with a single value, rewrite it to EQ
            lower = new NumberRangeParameter(EQ, this.getAdjustedLowerRangeValue());
            upper = null;
        }

        return lower;
    }

    public long getAdjustedLowerRangeValue() {
        if (this.lower == null) {
            return 0;
        }

        long lowerBound = this.lower.value();
        if (this.lower.operator() == RangeOperator.GT) {
            lowerBound++;
        }

        return lowerBound;
    }

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
