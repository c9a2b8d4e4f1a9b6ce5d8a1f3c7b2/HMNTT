### Title
Missing Query Timeout in `findByFilter()` Enables Unbounded Thread and Connection Exhaustion via Repeated Polling Subscriptions

### Summary
`TopicMessageRepositoryCustomImpl.findByFilter()` creates a `TypedQuery` with no query timeout hint. When `PollingTopicListener` is active (type=POLL), each subscriber drives a repeating poll loop on a `boundedElastic` scheduler thread. An unprivileged user opening many concurrent subscriptions can saturate the DB connection pool and scheduler thread pool with long-running queries, blocking all other subscriptions indefinitely.

### Finding Description

**Exact code location:**

`TopicMessageRepositoryCustomImpl.findByFilter()` ŌĆö [1](#0-0) 

The `TypedQuery` is created at line 48 and only receives `HINT_READ_ONLY` (line 49). No `jakarta.persistence.query.timeout` or equivalent hint is ever set:

```java
TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);
// ŌåÉ no timeout hint here
```

`poll()` in `PollingTopicListener` calls this synchronously on the `boundedElastic` scheduler: [2](#0-1) 

The scheduler is shared across all subscriptions: [3](#0-2) 

**Root cause:** No application-level query timeout is enforced. `setMaxResults` (line 51ŌĆō53) limits the *row count* returned but does not bound *execution time*. The `random_page_cost = 0` hint (line 56ŌĆō58) encourages index use but provides no time guarantee under DB stress. [4](#0-3) 

**Exploit flow:**

1. Attacker opens N concurrent gRPC `subscribeTopic` streams (no rate limiting visible in `PollingTopicListener`).
2. Each subscription spawns a repeating poll loop (every 500 ms by default) on the shared `boundedElastic` scheduler. [5](#0-4) 
3. Each poll issues a synchronous `getResultList()` call, blocking a scheduler thread and holding a DB connection for the full query duration. [6](#0-5) 
4. Under DB load (table bloat, lock contention, or simply the attacker's own concurrent queries competing for I/O), query latency rises.
5. With no timeout, threads and connections are held until the DB responds. The `boundedElastic` pool is exhausted; legitimate subscriptions queue indefinitely.

### Impact Explanation

Complete denial of service for the gRPC topic subscription service when running in POLL mode. All subscriber threads stall; no new or existing subscriptions can make progress. DB connection pool exhaustion can cascade to other services sharing the same pool. Severity is high because the resource being exhausted (scheduler threads + DB connections) is shared across all tenants/subscribers.

### Likelihood Explanation

Precondition: operator must have configured `type: POLL` (not the default `REDIS`). Once that is true, any unauthenticated gRPC client can open arbitrarily many `subscribeTopic` streams. No special knowledge or privilege is required. The attack is repeatable and self-amplifying: more concurrent subscriptions increase DB load, which increases query latency, which holds threads longer, which allows even fewer threads to service new polls.

### Recommendation

Set a query timeout hint on the `TypedQuery` in `findByFilter()`:

```java
typedQuery.setHint("jakarta.persistence.query.timeout", timeoutMillis);
```

Expose the timeout as a configurable property in `ListenerProperties` (e.g., `queryTimeout: 5s`). Additionally, consider enforcing a per-subscriber concurrency limit or a global cap on concurrent POLL subscriptions to prevent thread pool exhaustion independently of query duration.

### Proof of Concept

1. Deploy mirror-node gRPC service with `hiero.mirror.grpc.listener.type=POLL`.
2. From an unprivileged client, open 200+ concurrent `subscribeTopic` gRPC streams targeting any valid `topicId` with `startTime` set to a timestamp far in the past (maximizing rows scanned per poll).
3. Observe via metrics or thread dumps that `boundedElastic` threads are all blocked in `getResultList()`.
4. Attempt a new `subscribeTopic` from a separate client ŌĆö it will stall or time out at the gRPC layer because no scheduler thread is available to process its first poll.
5. DB connection pool metrics will show all connections checked out with no idle connections available.

        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;

        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
