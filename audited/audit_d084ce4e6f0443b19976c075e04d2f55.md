### Title
Uncached Alias-Based Entity Lookup Enables Sustained DB Query Load via Hooks Endpoint

### Summary
The `rest-java` module's `EntityServiceImpl.lookup()` performs a direct, uncached database query via `entityRepository.findByAlias()` on every request when the `ownerId` path parameter is an alias-based identifier. There is no application-level rate limiting on the `/api/v1/accounts/{ownerId}/hooks` endpoint in the `rest-java` module. An unprivileged attacker can flood this endpoint with valid alias-based owner IDs, forcing a DB query per request and sustaining elevated database load.

### Finding Description

**Exact code path:**

`HooksController.getHooks()` → `HookServiceImpl.getHooks()` (line 35) → `EntityServiceImpl.lookup()` (line 33) → `EntityRepository.findByAlias()` (line 13–14). [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — absent caching:**

The `rest-java` `EntityRepository.findByAlias()` carries no `@Cacheable` annotation. Compare this to:
- `web3` module's `EntityRepository`, which annotates its alias lookup with `@Cacheable(cacheNames = CACHE_NAME_ALIAS, ...)` [4](#0-3) 
- `grpc` module's `EntityRepository`, which annotates `findById` with `@Cacheable` [5](#0-4) 
- `importer`'s `EntityIdServiceImpl`, which wraps alias lookups in a `cacheLookup()` helper [6](#0-5) 

The `rest-java` `EntityServiceImpl` has no equivalent caching layer. [7](#0-6) 

**No application-level rate limiting on the hooks endpoint:**

The `rest-java` module has no `ThrottleManager` or equivalent bean applied to `HooksController`. The only rate-limiting visible is a GCP backend policy (`maxRatePerEndpoint: 250`) in the Helm chart, which is infrastructure-specific and absent in non-GCP deployments. [8](#0-7) 

**Exploit flow:**

1. Attacker enumerates or knows valid Base32-encoded aliases matching `ALIAS_PATTERN` (`^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$`). [9](#0-8) 
2. Attacker issues high-rate GET requests to `/api/v1/accounts/{alias}/hooks` using valid aliases for existing entities (to avoid early 404 short-circuit).
3. Each request unconditionally executes `SELECT id FROM entity WHERE alias = ? AND deleted <> true` against the database.
4. Because there is no caching and no application-level throttle, every request hits the DB.

### Impact Explanation

Each alias-based request to the hooks endpoint issues at least one uncached SQL query against the `entity` table. Under sustained high-rate requests from one or more clients, this directly increases database CPU, I/O, and connection pool utilization. Given that the baseline mirror node DB serves many other services (importer, web3, grpc), a sustained flood of alias lookups can push DB resource consumption well above the 30% threshold relative to the preceding 24-hour baseline, degrading service for all consumers of the shared database.

### Likelihood Explanation

No authentication or API key is required to call the hooks endpoint. The alias format is publicly documented and trivially constructable. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple script) can sustain hundreds of requests per second. The GCP rate limit of 250 RPS per endpoint is deployment-specific and absent in bare-metal or non-GCP Kubernetes deployments. The attack is repeatable, requires no special privileges, and leaves no persistent side effects.

### Recommendation

1. **Add `@Cacheable` to `rest-java`'s `EntityRepository.findByAlias()`** (and `findByEvmAddress()`), mirroring the pattern already used in the `web3` module with a short TTL (e.g., 60 seconds).
2. **Add application-level rate limiting** to the `rest-java` hooks (and related) endpoints, independent of infrastructure-level GCP policies, so protection applies in all deployment environments.
3. Consider a negative-result cache (cache misses for unknown aliases) to prevent cache-bypass via non-existent aliases.

### Proof of Concept

```bash
# 1. Obtain a valid alias for an existing account (e.g., from public mirror node API)
ALIAS="CIQAAAH4AY2OFK2FL37TSPYEQGPPUJRP4XTKWHD62HKPQX543LTJPBA"

# 2. Flood the hooks endpoint with alias-based requests (no auth required)
wrk -t8 -c200 -d60s \
  "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/hooks"

# 3. Observe DB query rate on the mirror node's PostgreSQL instance:
#    SELECT count(*), query FROM pg_stat_activity
#    WHERE query LIKE '%alias%' GROUP BY query;
# Expected: sustained high rate of:
#   SELECT id FROM entity WHERE alias = $1 AND deleted <> true
# with no cache hits, directly proportional to HTTP request rate.
```

final class EntityServiceImpl implements EntityService {

    private final EntityRepository entityRepository;

    @Override
    public Entity findById(EntityId id) {
        return entityRepository.findById(id.getId())
                .orElseThrow(() -> new EntityNotFoundException("Entity not found"));
    }

    @Override
    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
