### Title
Unbounded Growth of `cacheManagerSlotsPerContract` Internal Cache Registry Leading to Memory Exhaustion

### Summary
`ContractStateServiceImpl.findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` for every unique `contractId`, dynamically registering a new named `CaffeineCache` in the manager's internal `ConcurrentHashMap`. Because `cacheManagerSlotsPerContract` is configured without `setCacheNames()` (dynamic mode) and has no size limit on its internal cache registry, this map grows without bound. The outer `contractSlotsCache` (capped at 3000 entries) evicts old entries but never removes the corresponding named caches from `cacheManagerSlotsPerContract`, creating a permanent memory leak exploitable by any unprivileged caller.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageBatch()` — lines 86–87: [1](#0-0) 

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

`this.contractSlotsCache` is a Caffeine cache (the `CACHE_MANAGER_CONTRACT_SLOTS` "default" cache) configured with `maximumSize=3000`: [2](#0-1) 

The value loader calls `cacheManagerSlotsPerContract.getCache(contractId.toString())`. This `CaffeineCacheManager` bean is configured **without** `setCacheNames()`: [3](#0-2) 

All other `CaffeineCacheManager` beans call `setCacheNames(Set.of(...))` to pre-register a fixed set of caches. `cacheManagerSlotsPerContract` does not. In Spring's `CaffeineCacheManager`, omitting `setCacheNames()` enables **dynamic mode**: every call to `getCache(name)` with a previously unseen name creates a new `CaffeineCache` and stores it permanently in the manager's internal `ConcurrentHashMap<String, Cache> cacheMap`. There is no eviction mechanism on this map.

**Root cause and failed assumption:**

The design assumes the outer `contractSlotsCache` (max 3000 entries) bounds the total number of per-contract caches. This assumption is wrong. When the outer cache evicts an entry `(contractId → CaffeineCache)`, the `CaffeineCache` object is no longer referenced by the outer cache, but `cacheManagerSlotsPerContract.cacheMap` retains a permanent reference keyed by `contractId.toString()`. The two data structures are never synchronized on eviction.

**Exploit flow:**

1. Attacker sends `POST /api/v1/contracts/call` requests targeting N distinct contract addresses (or any EVM execution path that reads storage from distinct contracts).
2. Each unique `contractId` triggers `findStorageBatch()` → `cacheManagerSlotsPerContract.getCache(contractId.toString())`.
3. A new `CaffeineCache` object is inserted into `cacheManagerSlotsPerContract.cacheMap` for each new name.
4. The outer `contractSlotsCache` evicts entries once it exceeds 3000, but `cacheManagerSlotsPerContract.cacheMap` retains all N entries.
5. Memory grows linearly with the number of unique contractIds seen over the lifetime of the process.

**Existing checks reviewed and shown insufficient:**

The rate limiter (`requestsPerSecond = 500`, `gasPerSecond = 7,500,000,000`) slows the attack but does not prevent it: [4](#0-3) 

At 500 req/s, an attacker can register 500 new contractIds per second. Over 24 hours: ~43 million entries. Each empty `CaffeineCache` object (with its internal Caffeine data structures) consumes hundreds of bytes to low KB. At scale this exhausts heap. The `maximumSize=3000` on the outer cache does not help because it does not propagate eviction to `cacheManagerSlotsPerContract.cacheMap`. [5](#0-4) 

### Impact Explanation

The JVM heap is exhausted over time, causing `OutOfMemoryError` and service crash. This is a denial-of-service against the mirror node's web3 API. All users of the node are affected. The leak is permanent (survives cache TTL expiry) and accumulates across restarts only if the attacker resumes. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation

The `/api/v1/contracts/call` endpoint is publicly accessible with no authentication. The attacker only needs to vary the `to` field across requests. Hedera contract IDs are sequential 64-bit integers, making enumeration trivial. The global rate limit of 500 req/s means a single attacker consumes the full capacity while also degrading service for legitimate users. The attack is fully repeatable and requires no special privileges or on-chain state.

### Recommendation

1. **Fix the root cause**: Register `cacheManagerSlotsPerContract` with a fixed set of cache names, or implement a custom eviction listener on the outer `contractSlotsCache` that calls `cacheManagerSlotsPerContract` to invalidate/remove the corresponding named cache when an entry is evicted.
2. **Bound the registry**: Use a size-limited map (e.g., a Caffeine-backed map with eviction) instead of `CaffeineCacheManager`'s internal unbounded `ConcurrentHashMap` for the per-contract cache registry.
3. **Alternatively**: Replace the two-level cache design with a single Caffeine cache keyed by `(contractId, slotKey)` composite key, eliminating the need for dynamic named caches entirely.

### Proof of Concept

```bash
# Send requests with N distinct contract addresses (0x...0001 through 0x...FFFF)
for i in $(seq 1 100000); do
  addr=$(printf "0x%040x" $i)
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$addr\",\"data\":\"0x\",\"gas\":50000}" &
done
wait
# Monitor JVM heap: cacheManagerSlotsPerContract.cacheMap grows by ~1 entry per unique address.
# After sustained attack, heap exhaustion causes OutOfMemoryError.
```

Each unique `to` address maps to a unique `contractId`, triggering `cacheManagerSlotsPerContract.getCache(contractId.toString())` and permanently growing the internal registry.
