### Title
Global Shared Rate-Limit Bucket Enables Single-Client Starvation DoS on `/api/v1/contracts/call`

### Summary
`ThrottleConfiguration.rateLimitBucket()` creates a single JVM-level Bucket4j token bucket shared across all clients with no per-IP or per-user partitioning. A single unprivileged attacker can exhaust the entire `requestsPerSecond` capacity (default 500 req/s) from one HTTP client, causing every other user to receive HTTP 429 responses for the duration of the attack. No authentication or special privilege is required.

### Finding Description

**Exact code path:**

`ThrottleConfiguration.java` lines 24–32 create one Spring singleton `Bucket`:

```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

`ThrottleManagerImpl.java` lines 37–42 consume one token from this single shared bucket per request, regardless of caller identity:

```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
    ...
}
```

`ContractController.java` line 40 invokes this unconditionally for every POST to `/api/v1/contracts/call`:

```java
throttleManager.throttle(request);
```

**Root cause:** The bucket is a single shared object. `tryConsume(1)` is called identically for every client. There is no keying by IP address, session, API key, or any other client identifier anywhere in the throttle path. The `LoggingFilter` reads `request.getRemoteAddr()` only for logging; it plays no role in throttling. The `RequestProperties` per-filter throttling (`ThrottleProperties.request[]`) is keyed on request content fields (`DATA`, `BLOCK`, `GAS`, `TO`, `VALUE`) — not on client identity — and is disabled by default (`List.of()`).

**Failed assumption:** The design assumes the global bucket will be shared "fairly" across many clients. In reality, nothing prevents one client from consuming all tokens.

### Impact Explanation

With the default `requestsPerSecond = 500`, a single attacker sending ≥500 HTTP POST requests per second to `/api/v1/contracts/call` drains the entire bucket. All concurrent legitimate users receive `HTTP 429 Too Many Requests` with body `"Requests per second rate limit exceeded"` for as long as the attack continues. Because the bucket refills greedily at 500 tokens/second and the attacker can re-consume them at the same rate, the attack is self-sustaining with zero recovery window for other users. This constitutes a complete, sustained denial of access to historical Hashgraph state simulation for all users of the endpoint.

### Likelihood Explanation

**Preconditions:** None. No account, API key, or authentication is required to call `/api/v1/contracts/call`. The endpoint is publicly accessible.

**Capability required:** Any HTTP client capable of sustaining ~500 req/s — achievable with a single machine using tools like `wrk`, `hey`, `ab`, or a simple async Python/Go script.

**Repeatability:** The attack is trivially repeatable and can be sustained indefinitely. The attacker does not need to maintain state between requests; each request is a minimal POST with an empty or minimal body.

**Detection/mitigation gap:** There is no IP-based blocking, no connection-rate limiting at the application layer, and no per-client quota. Unless an external load balancer or WAF enforces per-IP limits (not present in this codebase), the attack succeeds unconditionally.

### Recommendation

Replace the single global bucket with per-client buckets keyed on the client IP address (extracted from `X-Forwarded-For` or `HttpServletRequest.getRemoteAddr()`). Use Bucket4j's `ProxyManager` with a `ConcurrentHashMap` or distributed cache backend:

```java
// Per-IP bucket map
private final ConcurrentHashMap<String, Bucket> buckets = new ConcurrentHashMap<>();

public Bucket resolveBucket(String clientIp) {
    return buckets.computeIfAbsent(clientIp, k -> buildBucket(perIpLimit));
}
```

Additionally, enforce a global cap to prevent resource exhaustion from the map itself (e.g., evict stale entries with Caffeine). Consider also adding a global bucket as a secondary guard against distributed attacks across many IPs.

### Proof of Concept

```bash
# Requires: wrk (https://github.com/wg/wrk) or equivalent
# Target: default requestsPerSecond = 500

# Terminal 1 – Attacker: saturate the global bucket from a single client
wrk -t4 -c50 -d60s -s post.lua http://<mirror-node-host>/api/v1/contracts/call
# post.lua: wrk.method="POST"; wrk.body='{"data":"0x","to":"0x0000000000000000000000000000000000000001"}'; wrk.headers["Content-Type"]="application/json"

# Terminal 2 – Victim: legitimate user making a single request
curl -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000001"}'

# Expected victim response while attacker runs:
# HTTP 429 Too Many Requests
# {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

The victim receives HTTP 429 for the entire duration of the attacker's flood. Stopping the attacker immediately restores service, confirming the single shared bucket is the sole control point. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
