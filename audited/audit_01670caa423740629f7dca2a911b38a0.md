### Title
Unauthenticated Alias Lookup DoS via Uncached DB Queries in `getBalances()`

### Summary
The `/api/v1/balances` endpoint accepts a syntactically valid but non-existent account alias via `account.id=<alias>`, which unconditionally triggers a live PostgreSQL query against the `entity` table with no caching of negative (miss) results and no rate limiting on the REST Node.js service. An unprivileged attacker can flood the endpoint with unique valid-format aliases that never match, generating sustained DB load from queries that always fail, degrading or exhausting the database connection pool and starving legitimate traffic.

### Finding Description

**Exact code path:**

`rest/balances.js` `getBalances()` (line 87) calls `parseAccountIdQueryParam()`:

```js
const [accountQuery, accountParamsPromise] = parseAccountIdQueryParam(req.query, 'ab.account_id');
const accountParams = await Promise.all(accountParamsPromise);
``` [1](#0-0) 

Inside `parseAccountIdQueryParam()` (lines 331–332), any value passing `AccountAlias.isValid()` (a pure format check) immediately triggers a live DB call with `requireResult = false`:

```js
if (AccountAlias.isValid(value, true) && ++evmAliasAddressCount === 1) {
  return EntityService.getAccountIdFromAlias(AccountAlias.fromString(value), false);
}
``` [2](#0-1)

```js

                                 and ${Entity.ALIAS} = $1`;

```js
async getAccountFromAlias(accountAlias) {
  const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
  if (isEmpty(rows)) {

``` [4](#0-3) 

**Root cause — two failed assumptions:**

1. **No caching of negative results.** A `grep` for `cache|Cache` in `rest/service/entityService.js` returns zero matches. Unlike the Java importer's `EntityIdServiceImpl`, which uses a Caffeine cache keyed on the alias bytes, the Node.js `EntityService` has no cache at all. Every alias lookup — hit or miss — goes directly to PostgreSQL. [5](#0-4) 

2. **No rate limiting on the REST service.** The `rest/server.js` middleware stack registers only `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optional response-cache handlers — no rate-limiting middleware is applied to the `/balances` route. The `authHandler` does not block unauthenticated requests; it only sets a custom limit for authenticated users. [6](#0-5) 

The response cache (`responseCacheCheckHandler`) does not help because each unique alias produces a distinct cache key; an attacker using different aliases per request bypasses it entirely.

### Impact Explanation

Each request with a unique non-existent alias forces a full index scan (or index seek) on the `entity` table's `alias` column. Under sustained flood, the PostgreSQL connection pool is saturated with queries that always return empty, starving legitimate reads. This degrades or halts the mirror node's ability to serve balance, transaction, and account queries — all of which downstream clients (wallets, explorers, SDKs) rely on to confirm transaction finality. The severity is **Critical** because the mirror node is the primary off-chain source of truth for transaction confirmation on the Hedera/Hiero network.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge of the system beyond the public API spec. The alias format is documented in the OpenAPI spec (`(?:[A-Z2-7]{8})*(?:[A-Z2-7]{2}|[A-Z2-7]{4,5}|[A-Z2-7]{7,8})`), making it trivial to generate millions of syntactically valid but non-existent aliases programmatically. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`, `k6`) can sustain thousands of requests per second. The attack is fully repeatable and stateless. [7](#0-6) 

### Recommendation

1. **Cache negative alias lookups** in `rest/service/entityService.js` `getAccountFromAlias()`: use an in-process LRU/TTL cache (e.g., `lru-cache`) keyed on the alias bytes. Cache both hits and misses (null results) with a short TTL (e.g., 30–60 s) to prevent repeated DB queries for the same non-existent alias.

2. **Add rate limiting** to the REST Node.js service. Apply a per-IP rate limiter (e.g., `express-rate-limit`) globally or specifically to endpoints that trigger alias/EVM-address DB lookups.

3. **Short-circuit early**: if `getAccountIdFromAlias()` returns `null` (alias not found), skip the main balance query entirely and return an empty result immediately, avoiding the second DB round-trip.

### Proof of Concept

```bash
# Generate unique valid Base32 aliases and flood the endpoint
# Each alias is syntactically valid but does not exist in the DB

for i in $(seq 1 10000); do
  # Vary the alias to defeat response caching
  ALIAS=$(python3 -c "
import base64, os
raw = os.urandom(20)
print(base64.b32encode(raw).decode().rstrip('='))
")
  curl -s "https://<mirror-node>/api/v1/balances?account.id=${ALIAS}" &
done
wait
```

Each request hits `parseAccountIdQueryParam` → `EntityService.getAccountIdFromAlias(..., false)` → `getAccountFromAlias()` → `SELECT id FROM entity WHERE coalesce(deleted, false) <> true AND alias = $1` with a unique alias that returns 0 rows. With no caching and no rate limiting, all queries reach PostgreSQL, exhausting the connection pool and degrading service for legitimate users.

class EntityService extends BaseService {
  static entityFromAliasQuery = `select ${Entity.ID}
                                 from ${Entity.tableName}
                                 where coalesce(${Entity.DELETED}, false) <> true
                                   and ${Entity.ALIAS} = $1`;

  static entityFromEvmAddressQuery = `select ${Entity.ID}
                                      from ${Entity.tableName}
                                      where ${Entity.DELETED} <> true
                                        and ${Entity.EVM_ADDRESS} = $1`;

  // use a small column in existence check to reduce return payload size
  static entityExistenceQuery = `select ${Entity.TYPE}
                                 from ${Entity.tableName}
                                 where ${Entity.ID} = $1`;

  static missingAccountAlias = 'No account with a matching alias found';
  static multipleAliasMatch = `Multiple alive entities matching alias`;
  static multipleEvmAddressMatch = `Multiple alive entities matching evm address`;

  /**
   * Retrieves the entity containing matching the given alias
   *
   * @param {AccountAlias} accountAlias accountAlias
   * @return {Promise<Entity>} raw entity object
   */
  async getAccountFromAlias(accountAlias) {
    const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);

    if (isEmpty(rows)) {
      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching alias ${accountAlias}`);
      throw new Error(EntityService.multipleAliasMatch);
    }

    return new Entity(rows[0]);
  }
```
