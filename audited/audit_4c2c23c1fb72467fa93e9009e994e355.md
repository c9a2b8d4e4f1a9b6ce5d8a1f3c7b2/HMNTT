### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent Rosetta API Requests

### Summary
The Rosetta API application in `rosetta/app/db/db.go` configures a shared `database/sql` pool capped at 100 open connections via `SetMaxOpenConns` with no application-level concurrency or rate limiting. Any unauthenticated external user can flood the server with concurrent DB-backed requests, saturating the pool and causing all subsequent legitimate requests to block waiting for a free connection for up to the `statementTimeout` window (default 20 s). The only potential guard — Traefik middleware — is optional, not built into the application, and bypassable with multiple source IPs.

### Finding Description

**Exact code path:**

`rosetta/app/db/db.go`, `ConnectToDb()`, line 33:
```go
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)  // default: 100
``` [1](#0-0) 

`rosetta/app/db/client.go`, `GetDbWithContext()`, line 36:
```go
childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
``` [2](#0-1) 

`rosetta/main.go`, `main()`, lines 220–227 — the HTTP server has no concurrency-limiting middleware; only metrics, tracing, and CORS wrappers are applied:
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
httpServer := &http.Server{ ... }
``` [3](#0-2) 

**Root cause:** The application assumes an external proxy (Traefik) will enforce concurrency limits. No in-process guard exists. Go's `database/sql` blocks the calling goroutine when `SetMaxOpenConns` is reached and all connections are in use; it waits until the context passed to `QueryContext` is cancelled. The context timeout is `statementTimeout` seconds (default 20 s). [4](#0-3) 

**Exploit flow:**
1. Attacker opens 100 concurrent HTTP connections to any DB-backed endpoint (e.g., `/block`, `/account/balance`, `/network/status`).
2. Each goroutine calls `GetDbWithContext`, acquires one of the 100 pool slots, and holds it for up to 20 s (statementTimeout).
3. Pool is fully saturated. All new legitimate requests block inside `db.QueryContext` waiting for a slot.
4. Attacker sustains the attack by sending ≥ 5 new requests/second (100 connections ÷ 20 s hold time).
5. Legitimate requests either time out after 20 s with `context deadline exceeded` or hang until the attacker stops.

**Why existing checks fail:**

- `statementTimeout = 20 s` bounds per-connection hold time but does not prevent pool exhaustion; it only sets the pace the attacker must maintain (~5 req/s). [5](#0-4) 

- HTTP `WriteTimeout = 10 s` / `ReadTimeout = 5 s` close the underlying TCP connection when exceeded, which cancels the request context and thus the DB wait. This reduces the hold time to ~10 s, raising the required attack rate to ~10 req/s — still trivially achievable. [6](#0-5) 

- Traefik `inFlightReq: amount: 5` per source IP is **conditional** (`{{ if and .Values.global.middleware .Values.middleware }}`), not part of the application binary, and bypassable: 20 source IPs × 5 in-flight = 100 connections. The `retry: attempts: 3` middleware further amplifies the attack by tripling request volume on errors. [7](#0-6) 

### Impact Explanation
All DB-backed Rosetta endpoints (`/block`, `/account/balance`, `/network/status`, `/search/transactions`, etc.) become unavailable to legitimate users for the duration of the attack. Requests either block for up to 20 s and return an error, or fail immediately if the pool wait context expires. No authentication, tokens, or funds are required. The impact is full service denial with no economic damage to any on-chain participant — consistent with the griefing classification.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no tooling beyond a standard HTTP load generator (e.g., `wrk`, `hey`, `ab`). Sustaining 10 req/s against a single public endpoint is well within the capability of any script kiddie. The Rosetta API is a public-facing exchange integration endpoint, making it an attractive target for griefing. The attack is repeatable and stateless.

### Recommendation
1. **Application-level semaphore:** Add an in-process `golang.org/x/sync/semaphore` (or `chan struct{}`) capped at `MaxOpenConnections` before any DB call, returning HTTP 503 immediately when the semaphore is full rather than blocking.
2. **Per-IP concurrency limit:** Implement middleware (e.g., using `golang.org/x/time/rate`) inside the Go application, independent of any external proxy.
3. **Pool wait timeout:** Set a short context deadline specifically for the pool-acquisition phase (separate from the query execution timeout), so requests fail fast with 503 rather than blocking for 20 s.
4. **Reduce `MaxOpenConnections`:** Align the pool size with realistic concurrency expectations and document that Traefik middleware is **required** for production deployments, not optional.

### Proof of Concept
```bash
# Step 1: Saturate the pool with 100 concurrent long-running block queries
# (replace HOST with the Rosetta API address)
for i in $(seq 1 100); do
  curl -s -X POST http://HOST:5700/block \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}' &
done

# Step 2: Immediately attempt a legitimate request — it will block for ~20 s then fail
time curl -s -X POST http://HOST:5700/network/status \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'

# Expected: the legitimate request hangs for ~20 s and returns a context deadline error
# while the 100 attacker goroutines hold all pool connections.

# Step 3: Sustain the attack at ~10 req/s to keep the pool permanently exhausted
wrk -t10 -c100 -d60s -s post.lua http://HOST:5700/block
```
