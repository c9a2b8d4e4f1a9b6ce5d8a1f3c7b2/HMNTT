### Title
Unauthenticated `getNodes()` gRPC Endpoint Lacks Global Connection Limit, Enabling DB Connection Pool Exhaustion DoS

### Summary
The `getNodes()` gRPC handler in `NetworkController.java` accepts requests from any unauthenticated caller and delegates to `NetworkServiceImpl.getNodes()`, which internally uses a `Flux.defer().repeatWhen()` pagination loop that issues repeated database queries per stream. The server enforces only a per-connection call cap (`maxConcurrentCallsPerConnection=5`) but imposes no global connection limit, no per-IP throttle, and no rate limit on `getNodes()`. An attacker opening thousands of connections can flood the database connection pool and the `boundedElastic` scheduler, starving legitimate `subscribeTopic` calls.

### Finding Description

**Exact code path:**

`NetworkController.java` lines 33–43 — `getNodes()` subscribes a reactive pipeline with no authentication or rate-limiting guard:

```java
public void getNodes(final AddressBookQuery request, final StreamObserver<NodeAddress> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(networkService::getNodes)
            ...
            .subscribe(...);
    if (responseObserver instanceof ServerCallStreamObserver s) {
        s.setOnCancelHandler(disposable::dispose);
    }
}
```

`NetworkServiceImpl.java` lines 55–77 — each subscription triggers two upfront DB queries (`findLatestTimestamp`, `findAllStakeByConsensusTimestamp`) and then enters a `Flux.defer().repeatWhen()` pagination loop:

```java
return Flux.defer(() -> page(context))
        .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                .jitter(0.5)
                .withFixedDelay(addressBookProperties.getPageDelay())   // 250 ms default
                .withScheduler(Schedulers.boundedElastic()))
        .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE);
```

With `limit=0` the `.take()` becomes `.take(Long.MAX_VALUE)`. The loop repeats until `context.isComplete()` is true, which is set only when a page returns fewer rows than `pageSize` (default 10). Each page iteration calls `transactionOperations.execute()` — a synchronous DB round-trip — on the `boundedElastic` scheduler.

**Server-side connection configuration** (`GrpcConfiguration.java` lines 28–35):

```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
```

No `maxConnections`, no `maxInboundConnections`, no IP-level throttle is configured anywhere in the gRPC module. The `maxConcurrentCallsPerConnection=5` cap is per-TCP-connection only.

**Root cause / failed assumption:** The design assumes that the per-connection call cap is sufficient to bound total server load. It is not: an attacker controls the number of connections, so total concurrent streams = (number of connections) × 5, with no upper bound on connections.

### Impact Explanation

Each concurrent `getNodes()` stream holds a `boundedElastic` scheduler thread slot for the duration of its pagination (multiple 250 ms-delayed DB round-trips). With thousands of concurrent streams:

1. **DB connection pool exhaustion** — each `page()` call acquires a JDBC connection via `transactionOperations.execute()`. Saturating the pool causes `subscribeTopic` and all other DB-backed gRPC calls to queue or fail with connection-timeout errors.
2. **`boundedElastic` scheduler saturation** — the scheduler has a bounded thread cap (default 10 × CPU cores) and a task queue. Under flood conditions, queued tasks back up, increasing latency for all reactive pipelines sharing the scheduler.
3. **Legitimate gossip queries blocked** — `subscribeTopic` (ConsensusService) shares the same executor (`applicationTaskExecutor`) and DB pool, so it is directly impacted.

Severity: **High** — complete denial of service to the mirror node's gRPC API is achievable without any credentials.

### Likelihood Explanation

- No authentication is required; any internet-reachable mirror node is a target.
- Standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`) trivially open many parallel connections.
- The attack is repeatable and scriptable: open N connections, issue 5 `getNodes({file_id: {fileNum: 102}, limit: 0})` calls per connection, loop.
- The GCP gateway `maxRatePerEndpoint: 250` (in `values.yaml`) is a backend-level rate limit per endpoint instance, not a per-client connection limit, and does not prevent this attack from a single client opening many connections to one pod.

### Recommendation

1. **Add a global server connection limit** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.maxConnectionAge(Duration.ofSeconds(30), TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(Duration.ofSeconds(10), TimeUnit.SECONDS);
   // Netty-level: set SO_BACKLOG and consider a ChannelHandler that counts active connections
   ```
2. **Add per-IP or global rate limiting** for `getNodes()` — mirror the `ThrottleManagerImpl` pattern already used in the `web3` module (Bucket4j token bucket), applied as a gRPC interceptor.
3. **Add a `maxConnections` guard** — implement a `ServerInterceptor` that tracks active connection count and returns `RESOURCE_EXHAUSTED` when a configurable ceiling is reached.
4. **Cap `pageDelay` stream lifetime** — add a `.timeout(Duration)` to the `Flux` in `NetworkServiceImpl.getNodes()` so a single stream cannot hold resources indefinitely if the address book is large or the DB is slow.

### Proof of Concept

```bash
# Open 200 connections, each with 5 concurrent getNodes streams (1000 total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes \
      > /dev/null &
  done
done
wait

# Observe: subscribeTopic calls now time out or return UNAVAILABLE
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}, "limit": 1}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected result:** `subscribeTopic` returns `UNAVAILABLE` or hangs indefinitely while the flood is active, because the DB connection pool and/or executor is saturated by the concurrent `getNodes` pagination loops.

```java
    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
```
