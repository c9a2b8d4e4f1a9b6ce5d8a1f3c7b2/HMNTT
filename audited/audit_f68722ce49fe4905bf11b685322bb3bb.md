### Title
Unbounded In-Memory Opcode Trace Accumulation via Unauthenticated Request Causes JVM OOM

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint accepts `stack=true`, `memory=true`, and `storage=true` from any unauthenticated caller. For a historically large contract transaction, the EVM re-execution traces every opcode and accumulates unbounded `Opcode` objects — each carrying full stack, memory, and storage snapshots — into a heap-resident `ArrayList` with no per-request size cap. A single such request can exhaust JVM heap memory, crashing the mirror-node process.

### Finding Description

**Entry point** — `OpcodesController.getContractOpcodes()`: [1](#0-0) 

No authentication is required. The only guards are:
1. `validateAcceptEncodingHeader` — trivially satisfied by sending `Accept-Encoding: gzip`.
2. `throttleManager.throttleOpcodeRequest()` — rate-limits to **1 request per second** by default. [2](#0-1) 

Neither check bounds the *size* of a single request.

**Pre-allocation proportional to gas** — `OpcodeServiceImpl.processOpcodeCall()`: [3](#0-2) 

`new OpcodeContext(opcodeRequest, (int) params.getGas() / 3)` pre-allocates an `ArrayList` with capacity `gasLimit / 3`. For a 15 M-gas transaction this is a 5 000 000-slot initial allocation before a single opcode is traced. [4](#0-3) 

**Per-opcode unbounded capture** — `OpcodeActionTracer.tracePostExecution()`: [5](#0-4) 

For every EVM opcode executed, `captureMemory`, `captureStack`, and `captureStorage` are called and the resulting `Opcode` object is appended to the list. With `stack=true`, `memory=true`, `storage=true`:
- **Stack**: up to 1 024 × 32-byte hex strings per opcode.
- **Memory**: EVM memory grows during execution; each snapshot is a full copy.
- **Storage**: all modified storage slots for the current frame.

There is no maximum opcode count, no maximum per-opcode data size, and no heap-budget check anywhere in the call chain.

### Impact Explanation
A single request targeting a historically large transaction (e.g., a complex DeFi interaction that consumed 10–15 M gas) with all three flags set can force the JVM to allocate several gigabytes of heap for the in-flight `ArrayList<Opcode>`. This triggers an `OutOfMemoryError`, crashing the mirror-node web3 process. Because the endpoint is unauthenticated and the throttle is per-second rate-based (not size-based), one request per second is sufficient to keep the node in a crash-restart loop, effectively denying service to all web3 API consumers. This matches the stated severity: shutdown of ≥30 % of network processing nodes without brute force.

### Likelihood Explanation
- **No privilege required**: any HTTP client can call the endpoint.
- **Trivial bypass of gzip check**: add `Accept-Encoding: gzip` header.
- **Throttle is ineffective**: 1 req/s is a rate limit, not a memory guard; one large request is enough.
- **Historical transactions are permanent**: an attacker can identify a known large transaction once and replay the attack indefinitely.
- **Repeatability**: after the process restarts, the same request can be re-sent immediately.

### Recommendation
1. **Add a hard opcode-count cap**: reject or truncate the response when the accumulated `opcodes` list exceeds a configurable maximum (e.g., 100 000 entries).
2. **Add a per-request memory budget**: track estimated bytes allocated during tracing and abort with HTTP 413 / 507 when the budget is exceeded.
3. **Cap per-opcode data sizes**: limit the number of stack items, memory words, and storage entries captured per opcode.
4. **Require authentication/authorization** for this endpoint, since it is explicitly a debug/tracing facility.
5. **Remove the gas-proportional pre-allocation** in `OpcodeContext` or cap it at a safe maximum, rather than using `(int) params.getGas() / 3` directly.

### Proof of Concept
```
# 1. Identify a historically large contract transaction hash from the mirror node REST API
#    (e.g., one with gasUsed close to the network limit ~15 000 000)
TX_HASH="0x<large_historical_tx_hash>"

# 2. Send the crafted request — no credentials needed
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"

# 3. Observe: the mirror-node JVM allocates unbounded heap for the opcode list,
#    eventually throwing OutOfMemoryError and crashing the process.
#    Repeat once per second to prevent recovery.
```

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
