### Title
Unsanitized `X-Forwarded-For` / `X-Real-IP` Header Values Logged Directly — ANSI Escape Sequence Injection in Access Logs

### Summary
`getClientIpAddress()` returns the raw, unsanitized value of the `X-Forwarded-For` or `X-Real-IP` header with no validation or character filtering. This value is directly interpolated into a log message string and emitted via logrus. An unprivileged external attacker can inject ANSI terminal escape sequences (e.g., `\x1b[...m`) or other control characters into these headers, which Go's HTTP parser does not strip, causing them to appear verbatim in log output and corrupt terminal-based log viewers and regex-based log parsers.

### Finding Description
**Exact code path:**

- `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74: reads `X-Real-IP` then `X-Forwarded-For` with zero sanitization and returns the raw string.
- `TracingMiddleware`, lines 46 and 52–53: the raw return value is directly interpolated into `fmt.Sprintf("%s %s %s (%d) in %s", clientIpAddress, ...)` and passed to `log.Info(message)` / `log.Debug(message)`.

**Root cause:** The function makes the failed assumption that HTTP header values are well-formed IP address strings. No allowlist validation, no regex check, no character stripping is applied at any point in the call chain.

**Why existing checks fail:** Go's `net/http` parser rejects bare `\r\n` sequences (CRLF injection), but it does **not** strip other control characters including `\x1b` (ESC, used to begin ANSI escape sequences). The test suite (`trace_test.go` lines 52–58) only exercises valid IP strings and provides no coverage for malicious input. There is no proxy trust list that would restrict which clients can set these headers.

**Exploit flow:**
1. Attacker sends an HTTP request directly to the Rosetta endpoint (no authentication required) with a crafted header:
   ```
   X-Forwarded-For: \x1b[2J\x1b[H\x1b[32m[INFO]\x1b[0m 2026-05-01T00:00:00Z fake-ip GET /block/transaction (200) in 1ms
   ```
2. `getClientIpAddress()` returns the raw string including the ESC bytes.
3. `TracingMiddleware` logs it verbatim via logrus.
4. On any terminal or log viewer that renders ANSI codes, the injected sequence clears the screen (`\x1b[2J`) and repositions the cursor (`\x1b[H`), making the real log entry invisible and replacing it with the attacker-controlled fake entry.
5. Regex-based log parsers that split on the expected `IP METHOD PATH (STATUS) in DURATION` format will misparse the line, potentially attributing the request to a spoofed IP or misreading the status/path fields.

### Impact Explanation
The direct impact is **log integrity corruption**: operators relying on access logs to audit which Rosetta API endpoints were called (e.g., `/block`, `/block/transaction`) and by which clients cannot trust the IP-address field of any log line. For deployments where access logs feed into SIEM tools or audit pipelines using pattern-based parsing, injected ANSI sequences or embedded fake log lines can cause misattribution of API calls to wrong source IPs, silent suppression of real log entries in terminal viewers, and confusion in incident-response workflows. This does not affect the underlying blockchain data but undermines the operational audit trail.

### Likelihood Explanation
Exploitation requires only the ability to send an HTTP request to the Rosetta API — no credentials, no special network position beyond TCP reachability. The only precondition is that the service is deployed without a sanitizing reverse proxy that strips or overwrites `X-Forwarded-For` before it reaches the Go application. Many Kubernetes/Docker deployments pass these headers through from untrusted clients. The attack is trivially repeatable with a single `curl` command.

### Recommendation
Sanitize the header value in `getClientIpAddress()` before returning it. The minimal fix is to validate that the returned string is a well-formed IP address using `net.ParseIP` (for single IPs) or by taking only the first comma-separated token and validating it:

```go
func getClientIpAddress(r *http.Request) string {
    raw := r.Header.Get(xRealIpHeader)
    if len(raw) == 0 {
        raw = r.Header.Get(xForwardedForHeader)
    }
    // Take only the first entry (leftmost client IP)
    if idx := strings.IndexByte(raw, ','); idx != -1 {
        raw = raw[:idx]
    }
    raw = strings.TrimSpace(raw)
    if ip := net.ParseIP(raw); ip != nil {
        return ip.String()
    }
    // Fall back to RemoteAddr
    ipAddress, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ipAddress
}
```

This ensures only valid IP address strings ever reach the logger, eliminating the injection surface entirely.

### Proof of Concept
```bash
# Send a request with ANSI escape sequences in X-Forwarded-For
# \x1b[2J clears the terminal screen; \x1b[H repositions cursor to top-left
curl -s http://<rosetta-host>:<port>/network/list \
  -H $'X-Forwarded-For: \x1b[2J\x1b[H\x1b[32m[INFO]\x1b[0m 2026-05-01T00:00:00Z 1.2.3.4 GET /block/transaction (200) in 1ms' \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"}}'

# Observe the server's log output in a terminal:
# - The real log line is hidden (screen cleared)
# - The fake log line appears in green as if it were a legitimate entry
# - A log parser splitting on the pattern "IP METHOD PATH (STATUS)" will
#   parse the injected fake fields instead of the real ones
```
