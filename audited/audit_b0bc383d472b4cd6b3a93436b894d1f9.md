### Title
Unauthenticated Spring Boot Actuator Health Endpoint Exposes Internal Network State via Public Ingress

### Summary
The `SubscriberHealthIndicator.health()` method exposes real-time network state (UP/DOWN/UNKNOWN), publishing rates, subscription rates, and REST API error details through the Spring Boot Actuator endpoint `/actuator/health/cluster`. No authentication, rate limiting, or access control is configured anywhere in the monitor module, and the Helm chart explicitly exposes this endpoint via a public ingress with TLS disabled by default. Any unauthenticated external user can continuously poll this endpoint to infer the live operational state of the Hiero network.

### Finding Description
**Code location:** `monitor/src/main/java/org/hiero/mirror/monitor/health/SubscriberHealthIndicator.java`, `health()` method, lines 59–64; `restNetworkStakeHealth()`, lines 93–121.

**Root cause:** No Spring Security configuration exists anywhere in the monitor module (confirmed: no `SecurityConfig`, `SecurityFilterChain`, `WebSecurityConfigurerAdapter`, or `spring.security` properties). The `health()` method is registered as a `ReactiveHealthIndicator` bean and is served by Spring Boot Actuator without any access control.

**Exploit flow:**
1. The Helm chart at `charts/hedera-mirror-monitor/values.yaml` lines 91–99 configures an ingress with `enabled: true`, path `/actuator/health/cluster`, and `tls.enabled: false` — making the endpoint reachable over plain HTTP from the internet.
2. `health()` calls `restNetworkStakeHealth()` (line 60), which returns the HTTP status code of the internal REST API in the health detail body (line 103: `"Network stake status is %s with status code %s"`), and on connection/timeout errors returns the raw error message (line 117–118: `"Network stake status is %s with error: %s"`).
3. If the REST API is UP, `health()` then calls `publishing()` and `subscribing()` (lines 61–62), which aggregate live transaction publish/subscribe rates and return DOWN or UNKNOWN when rates drop to zero.
4. An attacker polls `GET /actuator/health/cluster` at any interval with zero credentials and receives a JSON body containing `{"status":"DOWN","details":{"reason":"..."}}` or similar, directly reflecting the live network state.

**Why existing checks fail:** The `LoggingFilter` (lines 60–61 of `LoggingFilter.java`) only downgrades actuator path log verbosity to DEBUG — it provides no access control. There is no IP allowlist, no authentication header check, and no rate limiting anywhere in the request pipeline.

### Impact Explanation
An attacker learns in real time whether the Hiero network is processing transactions (UP), has stopped (DOWN), or is in an indeterminate state (UNKNOWN), along with internal HTTP status codes and error strings from the REST API. This enables precise timing of attacks (e.g., launching a double-spend attempt, exploiting a degraded consensus window, or amplifying a denial-of-service) to coincide with periods of network instability. The error detail strings also leak internal service connectivity information (e.g., `ConnectException` root causes), narrowing the attack surface further.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and only a standard HTTP client. The ingress is enabled by default in the Helm chart with TLS off. Any internet-accessible deployment of the monitor service is immediately exploitable. Polling can be automated trivially (e.g., `watch -n 1 curl http://<host>/actuator/health/cluster`). The attacker needs only to know the monitor service hostname, which may be discoverable via DNS enumeration or public documentation.

### Recommendation
1. **Restrict actuator exposure:** In `application.yml` (or equivalent config), set `management.endpoints.web.exposure.include=health` and `management.endpoint.health.show-details=never` (or `when-authorized`) to suppress detail leakage.
2. **Add authentication:** Introduce a `SecurityFilterChain` bean in the monitor module that requires authentication (e.g., HTTP Basic or token) for all `/actuator/**` paths except liveness/readiness probes used by Kubernetes internally.
3. **Restrict ingress:** Remove `/actuator/health/cluster` from the public ingress or place it behind an authenticated ingress annotation (e.g., `nginx.ingress.kubernetes.io/auth-type: basic`).
4. **Enable TLS:** Set `tls.enabled: true` in the Helm chart to prevent plaintext interception.
5. **Add rate limiting:** Apply ingress-level rate limiting to `/actuator/**` paths.

### Proof of Concept
```bash
# Precondition: monitor service exposed via ingress (default Helm values)
# No credentials required

# Single poll
curl -s http://<monitor-ingress-host>/actuator/health/cluster
# Response example:
# {"status":"DOWN","details":{"reason":"Network stake status is DOWN with status code 503"}}

# Continuous polling to track network state transitions
while true; do
  STATUS=$(curl -s http://<monitor-ingress-host>/actuator/health/cluster)
  echo "$(date): $STATUS"
  sleep 5
done
# Attacker observes UP -> DOWN transition and times their attack accordingly
```
