### Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses `maxPageSize` Cap, Enabling Unlimited DB Queries

### Summary
In `PollingTopicListener.poll()`, the expression `(int)(filter.getLimit() - context.getCount().get())` silently truncates a `long` to `int`. Any user-supplied `limit` value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`) produces a negative `int` after the cast. `Math.min(negativeValue, maxPageSize)` then returns the negative value, which is stored as a negative `long` in the rebuilt filter. Because `hasLimit()` checks `limit > 0`, the negative limit is treated as "no limit," causing `findByFilter()` to execute an unbounded DB query on every poll interval — indefinitely.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` maps the gRPC `uint64 limit` field directly to `TopicMessageFilter.limit` with no upper-bound cap: [1](#0-0) 

The proto field is `uint64`, accepting values up to `2^64-1`; Java maps it as a signed `long`, so values up to `Long.MAX_VALUE` pass the `@Min(0)` constraint on `TopicMessageFilter.limit`: [2](#0-1) 

`TopicMessageServiceImpl.incomingMessages()` forwards the limit as a `long` to `topicListener.listen()`: [3](#0-2) 

Inside `PollingTopicListener.poll()`, the unsafe narrowing cast occurs: [4](#0-3) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getCount().get() = 0`:
- `(int)(Long.MAX_VALUE)` → lower 32 bits of `0x7FFFFFFFFFFFFFFF` = `0xFFFFFFFF` = **`-1`**
- `Math.min(-1, 5000)` = **`-1`**
- `filter.toBuilder().limit(-1)` builds a new `TopicMessageFilter` with `limit = -1L` — **no Spring validation fires** because the builder call is not a Spring-managed method boundary

In `TopicMessageRepositoryCustomImpl.findByFilter()`, `hasLimit()` returns `false` for `-1 > 0`, so `setMaxResults` is never called: [5](#0-4) 

The query runs with **no row limit**, returning every matching topic message. This repeats on every poll interval (default 500 ms) for the lifetime of the subscription.

**Why existing checks fail:**
- `@Min(0)` on `TopicMessageFilter.limit` only rejects negative input; `Long.MAX_VALUE` passes.
- `@Validated` / `@Valid` on `subscribeTopic` validates the original filter, not the internally rebuilt filter in `poll()`.
- `Math.min(limit, maxPageSize)` is intended to cap page size but is defeated when `limit` is already negative.
- `hasLimit()` (`limit > 0`) inadvertently treats the overflowed negative value as "unlimited."

### Impact Explanation
Every poll cycle (default every 500 ms) issues a full table scan against `topic_message` for the subscribed topic with no `LIMIT` clause. On a busy topic with millions of messages, this saturates DB I/O, memory, and connection pool resources. Multiple concurrent subscriptions with this payload amplify the effect linearly, constituting a practical denial-of-service against the mirror node's database. The `maxPageSize` protection — the only intended safeguard against large result sets in the polling path — is completely bypassed.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. The attacker only needs to send a `ConsensusTopicQuery` with `limit` set to any value greater than `2147483647` (e.g., `limit: 9223372036854775807`). No special privileges, tokens, or prior knowledge beyond a valid topic ID are required. The attack is trivially repeatable and scriptable with any gRPC client (e.g., `grpcurl`).

### Recommendation
Replace the unsafe narrowing cast with a saturating clamp before the cast:

```java
// In PollingTopicListener.poll()
long remaining = filter.getLimit() - context.getCount().get();
int limit = filter.hasLimit()
        ? (int) Math.min(remaining, Integer.MAX_VALUE)   // clamp before cast
        : Integer.MAX_VALUE;
```

Additionally, add an upper-bound validator on `TopicMessageFilter.limit` (e.g., `@Max(Integer.MAX_VALUE)`) or enforce a configurable maximum subscription limit at the controller layer in `ConsensusController.toFilter()` to reject unreasonably large values before they propagate.

### Proof of Concept

```bash
# Using grpcurl against a running mirror node with POLL listener enabled
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 41110},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Steps:**
1. Start mirror node with `hiero.mirror.grpc.listener.type=POLL`.
2. Subscribe to any existing topic with `limit = Long.MAX_VALUE` (`9223372036854775807`).
3. The request passes `@Min(0)` validation.
4. `PollingTopicListener.poll()` computes `(int)(Long.MAX_VALUE) = -1`; `Math.min(-1, 5000) = -1`.
5. `findByFilter()` receives a filter with `limit = -1`, `hasLimit()` returns `false`, no `setMaxResults` is set.
6. Every 500 ms, a full unbounded `SELECT` is issued against `topic_message` for the topic.
7. Observe DB CPU/IO spike; repeat with multiple concurrent connections to amplify.
