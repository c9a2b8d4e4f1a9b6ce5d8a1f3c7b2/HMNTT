### Title
Unauthenticated Bandwidth Exhaustion via Unbounded Bytecode Retrieval in `getContractById`

### Summary
Any unauthenticated user can repeatedly call `GET /api/v1/contracts/:contractId` for contracts whose database-stored `file_id` passes `isValidUserFileId()`. Each such request triggers `FileDataService.getFileData()`, which executes an unbounded `string_agg` SQL aggregation over all file data chunks and returns the full bytecode payload — potentially hundreds of kilobytes — with no response size cap and no per-IP rate limiting on this endpoint.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js` lines 731–732: [1](#0-0) 

```js
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(contract.file_id, contract.created_timestamp);
```

The `file_id` is read from the database row for the attacker-chosen `:contractId`. `isValidUserFileId()` returns `true` for any `file_id > 1000` (user-created files), as confirmed by tests: [2](#0-1) 

When true, `FileDataService.getFileData` is called: [3](#0-2) 

The SQL query uses `string_agg(file_data, '' order by consensus_timestamp)` with no `LIMIT` or size cap on the aggregated result — it concatenates every chunk of the file into a single byte string and returns it. The result is placed directly into `contract.bytecode`, then serialized into the response via `ContractBytecodeViewModel` with only a hex prefix added and no truncation: [4](#0-3) 

The route is a public, unauthenticated GET: [5](#0-4) 

**Root cause:** The assumption that `isValidUserFileId()` is a security gate is false — it is only a routing decision (file-based vs. initcode-based bytecode). There is no size cap on the returned bytecode, no per-IP rate limiting on this endpoint in the REST Node.js API, and no caching in `FileDataService.getFileData` (unlike the Java importer's `FileDataServiceImpl` which has a bounded cache).

**Why existing checks fail:**
- `isValidUserFileId` only checks `file_id > 1000`; it does not limit payload size.
- The `response.limit.default/max` configuration (25/100) applies to pagination, not response body size.
- The `ThrottleManager` / bucket4j rate limiting exists only in the `web3` Java module for `/contracts/call` POST requests — it does not apply to the REST Node.js API's GET endpoints. [6](#0-5) 

### Impact Explanation
An attacker can identify one or more deployed contracts with large file-backed bytecode (common for complex contracts on Hedera mainnet/testnet, where bytecode > 6 KB must be uploaded as a file). Repeatedly requesting `GET /api/v1/contracts/{contractId}` for such a contract forces the server to: (1) execute a full `string_agg` DB aggregation on every request, (2) serialize and transmit the full bytecode payload (potentially hundreds of KB per response) to the attacker. With concurrent connections and no rate limiting, this exhausts both outbound bandwidth and database I/O, degrading service for all users.

### Likelihood Explanation
No privileges, authentication, or special knowledge are required. Large-bytecode contracts are publicly enumerable via `GET /api/v1/contracts`. The attack is trivially scriptable with `curl` or any HTTP client in a loop. The endpoint is stateless and idempotent, making it trivially repeatable from a single IP or distributed across many IPs.

### Recommendation
1. **Add a bytecode size cap**: Truncate or reject `FileDataService.getFileData` results exceeding a configurable maximum (e.g., 128 KB) before placing them in the response.
2. **Add per-IP rate limiting** on the REST API's `/contracts/:contractId` endpoint, consistent with the Traefik `rateLimit` middleware already used for other services.
3. **Cache `getFileData` results** in the REST service layer (similar to the Java `FileDataServiceImpl` cache) to avoid repeated DB aggregation for the same `file_id`.

### Proof of Concept
```bash
# 1. Find a contract with a large file_id (file-backed bytecode)
curl "https://<mirror-node>/api/v1/contracts?limit=100" | jq '.contracts[] | select(.file_id != null) | .contract_id'

# 2. Repeatedly request the large-bytecode contract (no auth required)
while true; do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.<large_contract_id>" > /dev/null &
done
# Each request triggers string_agg over all file_data chunks and returns
# the full hex-encoded bytecode in the response body with no size limit.
```

  /**
   * The function returns the data for the fileId at the provided consensus timestamp.
   * @param fileId
   * @param timestamp
   * @return {data: string}
   */
  getFileData = async (fileId, timestamp) => {
    const params = [fileId, timestamp];
    const query = FileDataService.getFileDataQuery;
