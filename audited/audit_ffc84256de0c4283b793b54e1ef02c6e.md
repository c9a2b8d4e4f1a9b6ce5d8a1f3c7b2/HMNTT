### Title
Readiness Spoofing via Directory Creation at `/tmp/ready`

### Summary
The `ready` healthcheck in `main()` uses `os.Stat("/tmp/ready")` and treats any non-error result as "ready." Because `os.Stat` succeeds for both regular files **and** directories, an unprivileged user can `mkdir /tmp/ready` before the legitimate init process creates the marker file, causing the readiness probe to permanently return exit 0 (ready) even though the service has never completed initialization.

### Finding Description
**Exact location:** `pinger/cmd/healthcheck/main.go`, `main()`, lines 20–23.

```go
if _, err := os.Stat("/tmp/ready"); err != nil {
    fmt.Fprintln(os.Stderr, "not ready")
    os.Exit(1)
}
os.Exit(0)
``` [1](#0-0) 

**Root cause:** The code discards the `fs.FileInfo` return value (`_`) and only checks whether `err != nil`. `os.Stat` returns `nil` error for any existing filesystem object — regular file, directory, symlink target, etc. There is no call to `fi.Mode().IsRegular()` or any equivalent type assertion.

**Failed assumption:** The developer assumes only the legitimate init process can create `/tmp/ready`, and that it will always be a regular file.

**Exploit flow:**
1. `/tmp` is world-writable with the sticky bit on standard Linux. Any user can create new entries; the sticky bit only prevents *deleting* entries owned by others.
2. Before the legitimate init process runs (or if it has not yet created `/tmp/ready`), the attacker executes `mkdir /tmp/ready`.
3. `os.Stat("/tmp/ready")` now returns `nil` error because the directory exists.
4. The healthcheck exits 0 — signalling "ready" — for every subsequent probe call, indefinitely.
5. The sticky bit prevents the legitimate init process from removing the attacker-created directory (different owner), so it cannot replace it with the real marker file, permanently blocking correct readiness signalling.

The same logic flaw exists for the `live` probe at line 27–38: `mkdir /tmp/alive` would also satisfy `os.Stat` and produce a fresh `ModTime()`, passing the 2-minute staleness check. [2](#0-1) 

### Impact Explanation
A Kubernetes readiness probe returning exit 0 causes the orchestrator to add the pod to the load-balancer endpoint pool immediately. If the actual service inside the container has not finished initialization, real user traffic is routed to it and will receive errors or incomplete responses. The attacker achieves a persistent false-ready state with no ability for the legitimate process to correct it (sticky-bit lock-out). Severity matches the stated scope: griefing / availability degradation with no direct economic loss.

### Likelihood Explanation
Requires only unprivileged code execution in the same container (e.g., a compromised sidecar, a shared-namespace debug container, or a supply-chain injected process). The attack is a single shell command (`mkdir /tmp/ready`), is repeatable across restarts if the directory persists, and requires no special privileges or kernel exploits. The race window (before init creates the file) is wide during cold starts.

### Recommendation
Capture and validate the `FileInfo` returned by `os.Stat`:

```go
fi, err := os.Stat("/tmp/ready")
if err != nil || !fi.Mode().IsRegular() {
    fmt.Fprintln(os.Stderr, "not ready")
    os.Exit(1)
}
```

Apply the same fix to the `live` probe. Additionally, consider placing the marker files outside world-writable `/tmp` (e.g., a directory owned exclusively by the service user), or using a Unix domain socket / named pipe whose creation is atomic and type-checked.

### Proof of Concept
```bash
# 1. In the container, before init completes:
mkdir /tmp/ready

# 2. Invoke the healthcheck (as any user):
/usr/local/bin/healthcheck ready
echo $?   # prints 0  — falsely "ready"

# 3. Confirm the legitimate init cannot overwrite it:
touch /tmp/ready   # fails: "cannot overwrite directory"
# or if init tries: os.WriteFile("/tmp/ready", ...) → EISDIR

# 4. Every subsequent probe returns 0 indefinitely.
```
