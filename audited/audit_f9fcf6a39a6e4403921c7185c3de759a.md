### Title
Hash Collision in `CACHE_NAME_ALIAS` Cache Allows Wrong Entity Resolution via `Arrays.hashCode` Key Truncation

### Summary
`findByEvmAddressOrAliasAndDeletedIsFalse` uses `Arrays.hashCode(byte[])` — a 32-bit integer — as the sole cache key for the `CACHE_NAME_ALIAS` cache. Because this method handles **both** alias-based and evm_address-based lookups through the same cache, two distinct byte arrays (one an alias for entity A, one an evm_address for entity B) that produce the same `Arrays.hashCode()` value will collide: the second lookup returns entity A from cache instead of querying the database for entity B. An unprivileged attacker who can register a Hedera account with a crafted EVM address can trigger this collision deterministically.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 39–49:

```java
@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",   // ← 32-bit int key
        unless = "#result == null")
@Query(value = """
        select *
        from entity
        where (evm_address = ?1 or alias = ?1) and deleted is not true
        """, nativeQuery = true)
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

`common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22:

```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);   // ← standard Java 32-bit polynomial hash
}
```

**Root cause:** `Arrays.hashCode(byte[])` maps an arbitrary-length byte array to a 32-bit signed integer using the polynomial `result = 31*result + b[i]`. The cache key space is therefore only 2³² ≈ 4 billion values. Because `findByEvmAddressOrAliasAndDeletedIsFalse` is the single entry point for **both** alias lookups and evm_address lookups (the SQL `WHERE evm_address = ?1 OR alias = ?1` handles both), both lookup types share the same `CACHE_NAME_ALIAS` cache and the same 32-bit key. No secondary equality check on the actual byte array is performed before returning the cached value.

**Failed assumption:** The cache assumes that `Arrays.hashCode(X) == Arrays.hashCode(Y)` implies `X == Y`. This is false for a 32-bit hash over 20-byte inputs.

**Exploit flow:**
1. Attacker knows victim entity A has alias `X` (e.g., a compressed Ed25519 public key, 32 bytes).
2. Attacker computes `h = Arrays.hashCode(X)` offline.
3. Attacker iterates ECDSA key pairs, deriving EVM addresses (20 bytes each), until finding address `Y` where `Arrays.hashCode(Y) == h`. Expected iterations: ~2³² ≈ 4 billion — feasible in seconds on modern hardware since the computation is a simple polynomial evaluation per candidate.
4. Attacker registers a Hedera account with EVM address `Y` (entity B).
5. Attacker triggers a lookup of entity A via its alias `X` (e.g., by calling any web3 API endpoint that resolves alias `X`). Entity A is stored in `CACHE_NAME_ALIAS` under key `h`.
6. Within the 1-second cache TTL (`expireAfterWrite=1s`), attacker triggers a lookup of entity B via its EVM address `Y`. `Arrays.hashCode(Y) == h`, so the cache returns entity A instead of querying the database.
7. The caller receives entity A's data (account ID, EVM address, balance, etc.) when it asked for entity B.

**Why existing checks are insufficient:**
- `unless = "#result == null"` only prevents caching empty results; it does not prevent a collision between two non-null results.
- The `CACHE_NAME_ALIAS` and `CACHE_NAME_EVM_ADDRESS` caches are separate named caches within the same `CaffeineCacheManager` (`cacheManagerEntity`, line 102 of `EvmConfiguration.java`), so there is no cross-cache collision — but the intra-cache collision within `CACHE_NAME_ALIAS` is real because the same method handles both lookup types.
- The 1-second TTL (`ENTITY_CACHE_CONFIG = "expireAfterWrite=1s,maximumSize=10000"` in `CacheProperties.java` line 19) narrows the window but does not eliminate it; two rapid sequential API calls are sufficient.

### Impact Explanation
The mirror node's web3 JSON-RPC API is used by DApps and wallets to resolve Hedera aliases and EVM addresses to account entities (balance, EVM address, account ID). A cache collision causes the API to return entity A's data in response to a query for entity B's address. DApps that use this resolution to determine transfer destinations (e.g., resolving an alias before constructing a `CryptoTransfer`) will direct funds to entity A instead of entity B. This constitutes indirect but concrete misdirection of HBAR or token transfers. Severity is high given the financial context.

### Likelihood Explanation
The attacker requires no privileges — only the ability to:
1. Register a Hedera account (permissionless on mainnet/testnet).
2. Make two sequential unauthenticated HTTP requests to the mirror node's web3 API within 1 second.

The preimage search (finding `Y` with `Arrays.hashCode(Y) == h`) is a trivial offline computation (~4 billion iterations of a 5-operation loop, completable in under a minute on commodity hardware). The attack is fully repeatable and requires no special network position.

### Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with the full byte array wrapped in a collision-resistant key type. Options:
1. Use `Arrays.toString(#alias)` or `new String(#alias, StandardCharsets.ISO_8859_1)` as the SpEL key expression — this produces a string that is structurally equal iff the byte arrays are equal, eliminating collisions entirely.
2. Alternatively, wrap the byte array in a `ByteBuffer` or a custom value object that implements `equals`/`hashCode` correctly (using `Arrays.equals` for equality and a 64-bit or cryptographic hash for bucketing).
3. The `SpelHelper.hashCode` method should be removed or renamed to make clear it is not suitable as a cache key.

// Step 1: Compute hash of victim alias X
byte[] X = /* alias of entity A, e.g. 32-byte Ed25519 public key */;

// Step 2: Find EVM address Y with same hash (offline brute-force)
byte[] Y = null;
KeyPairGenerator gen = KeyPairGenerator.getInstance("EC");
gen.initialize(new ECGenParameterSpec("secp256k1"));
for (long i = 0; i < 0xFFFFFFFFL; i++) {
    KeyPair kp = gen.generateKeyPair();
    byte[] pubKey = ((ECPublicKey) kp.getPublic()).getEncoded();

    if (Arrays.hashCode(addr) == h) { Y = addr; break; }

// GET /api/v1/contracts/call  with `to` = hex(Y)
