### Title
Stale Token-Level Freeze Status Permanently Applied to Accounts via Unbounded LRU Cache in `getTokenAccounts()`

### Summary
`getTokenAccounts()` in `rest/service/tokenService.js` uses a nullish-coalescing fallback (`??`) to apply a cached token-level `freezeStatus` to any `token_account` row whose `freeze_status` column is `null` — a valid post-V1.94.0 state. The in-process `tokenCache` (a `quickLru` instance) has **no TTL and no invalidation path**: once a token's freeze status is cached, it is never refreshed until the entry is evicted by LRU pressure. Any unprivileged user can anchor a FROZEN entry in the cache by issuing a single read-only API request while the token is frozen; after the token is unfrozen on-chain, every subsequent query for an account with `freeze_status = null` will receive the stale FROZEN status indefinitely.

### Finding Description

**Exact code path**

`rest/service/tokenService.js` lines 12–14 — cache construction:
```js
const tokenCache = new quickLru({
  maxSize: config.cache.token.maxSize,   // size-only eviction, no maxAge / TTL
});
```

`getTokenAccounts()` lines 103–114 — fallback application:
```js
const cachedTokens = await this.getCachedTokens(tokenIds);   // line 104
return rows.map((row) => {
  const cachedToken = cachedTokens.get(row.token_id);
  if (cachedToken) {
    row.decimals        = cachedToken.decimals;
    row.freeze_status   = row.freeze_status ?? cachedToken.freezeStatus;  // line 109 ← root cause
    row.kyc_status      = row.kyc_status    ?? cachedToken.kycStatus;
  }
  return new TokenAccount(row);
});
```

`getCachedTokens()` lines 136–161 — cache write, no refresh:
```js
tokenIds.forEach((tokenId) => {
  const cachedToken = tokenCache.get(tokenId);
  if (cachedToken) {
    cachedTokens.set(tokenId, cachedToken);   // cache HIT → DB never queried again
  } else {
    uncachedTokenIds.push(tokenId);
  }
});
// DB is only queried for uncachedTokenIds; result is written once and never updated
```

**Root cause**

`quickLru` is configured with `maxSize` only. The `quick-lru` library supports a `maxAge` option for TTL-based expiry, but it is not used here. There is no explicit invalidation call anywhere in the codebase for `tokenCache`. Once a `CachedToken` with `freezeStatus = 1` (FROZEN) is stored, it persists until the LRU evicts it due to size pressure — which may never happen in a low-traffic deployment or for a popular token that keeps the entry "hot".

**Failed assumption**

The code assumes that `token.freeze_status` (the token-level default, sourced from the `token` table) is immutable after first cache population. It is not: a token admin can call `TokenUpdate` to change `freeze_default`, which updates `token.freeze_status` in the DB. The cache is never notified.

**V1.94.0 interaction**

Migration `V1.94.0__update_token_freeze_kyc_status.sql` lines 36–38 explicitly drops `NOT NULL` and the default from `token_account.freeze_status`. Any account associated after this migration (or whose row was not explicitly updated by a freeze/unfreeze transaction) will have `freeze_status = null`. The `??` operator on line 109 treats `null` as "missing" and substitutes the cached token-level value — exactly the stale FROZEN value.

**Exploit flow**

1. Token T is created with `freeze_default = true` → `token.freeze_status = 1` (FROZEN) in DB.
2. Attacker (no privileges) calls `GET /api/v1/accounts/{any}/tokens?token.id=T`. `getCachedTokens` finds T uncached, queries DB, stores `CachedToken{freezeStatus:1}` in `tokenCache`.
3. Token admin calls `TokenUpdate` to set `freeze_default = false` → `token.freeze_status = 2` (UNFROZEN) in DB. Cache is **not** invalidated.
4. Account B associates with token T. Because no explicit freeze/unfreeze transaction has been issued for B, `token_account.freeze_status = null`.
5. Any user calls `GET /api/v1/accounts/B/tokens`. `getCachedTokens` returns the cached entry (cache hit). Line 109 evaluates `null ?? 1 → 1`. `TokenRelationshipViewModel` line 24 serialises this as `"freeze_status": "FROZEN"` in the JSON response.
6. Account B is permanently reported as FROZEN to all API consumers until the LRU entry is evicted.

### Impact Explanation

The mirror node REST API is the canonical read layer used by wallets, exchanges, and DeFi integrations to gate token transfers. `TokenRelationshipViewModel` directly exposes the computed `freeze_status` field to clients. An account incorrectly reported as FROZEN will be refused by any application that checks this field before submitting a `CryptoTransfer`. While the Hedera consensus layer enforces the true state, most client-side UIs and SDKs short-circuit on the mirror node response and never attempt the transfer. The result is a **denial-of-service on token transfers** for affected accounts, with no on-chain action required from the attacker and no self-healing until the LRU entry is evicted.

### Likelihood Explanation

- **Attacker capability**: zero — a single unauthenticated GET request suffices to anchor the cache entry.
- **Precondition**: the token must have been frozen at some point (a routine administrative action). The attacker does not need to control or influence the token admin.
- **Persistence**: indefinite in the absence of LRU eviction. In production deployments with large `maxSize` and a popular token, the entry will remain hot and never be evicted.
- **Repeatability**: if the entry is eventually evicted, the attacker re-anchors it with another GET request during any future freeze window.

### Recommendation

1. **Add a TTL to `tokenCache`**: pass `maxAge` (milliseconds) to `quickLru` so entries expire and are re-fetched from the DB:
   ```js
   const tokenCache = new quickLru({
     maxSize: config.cache.token.maxSize,
     maxAge: config.cache.token.maxAge,   // e.g. 60_000 ms
   });
   ```
2. **Never fall back to token-level freeze status for account-level display**: if `token_account.freeze_status` is `null`, the API response should return `null` (already handled correctly in `TokenRelationshipViewModel` line 24 — the fallback on line 109 is what breaks this). Remove or guard the fallback so that a `null` account-level status is not silently replaced by a potentially stale token-level default.
3. **Invalidate the cache on token update events**: wherever `token.freeze_status` is updated (e.g. in the token-update ingestion path), call `tokenCache.delete(tokenId)`.
