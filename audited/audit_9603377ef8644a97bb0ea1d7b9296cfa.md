### Title
Cache Key Pollution via Unnormalized URL Flooding in `responseCacheCheckHandler`

### Summary
The `cacheKeyGenerator` function hashes `req.originalUrl` directly without any normalization, meaning every unique query string (e.g., `?timestamp=<unique_value>`) produces a distinct Redis key. An unprivileged attacker can flood the `/api/v1/transactions` or `/api/v1/transactions/{id}` endpoints with high-cardinality parameters, filling Redis with single-use entries, driving the cache hit rate to near zero, and forcing constant eviction churn and DB fallback load well above the 30% threshold.

### Finding Description

**Exact code location:**

`rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator()`, line 151–153:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
The comment at line 148–149 explicitly acknowledges the absence of normalization: *"In the future, this will utilize Edwin's request normalizer (9113)."*

**Root cause:** `req.originalUrl` is used verbatim. No parameter sorting, deduplication, or canonicalization is applied. Every semantically equivalent but syntactically distinct URL (e.g., `?timestamp=1700000000.000000001` vs `?timestamp=1700000000.000000002`) produces a different MD5 hash and a different Redis key.

**Exploit flow:**

1. `responseCacheCheckHandler` (line 42–43) calls `cacheKeyGenerator(req)` and immediately queries Redis with the result. On a cache miss (line 45–47), it stores the key in `res.locals` and falls through to the DB handler.
2. `responseCacheUpdateHandler` (line 90–116) then writes the response to Redis via `getCache().setSingle(responseCacheKey, ttl, cachedResponse)` (line 116).
3. The `/api/v1/transactions` list endpoint sets `cache-control: public, max-age=2` (confirmed in `rest/__tests__/specs/transactions/responseHeaders.json`). The `/api/v1/transactions/{id}` endpoint sets `max-age=600` (confirmed in `rest/__tests__/specs/transactions/{id}/responseHeaders.json`).
4. An attacker sends a continuous stream of requests with unique `timestamp` values. Each request: (a) misses the cache, (b) hits the DB, (c) writes a new unique Redis key. For the `{id}` endpoint, each entry persists for 600 seconds.

**Why existing checks are insufficient:**

- **`allkeys-lfu` eviction policy** (configured at `rest/cache.js` line 49, default documented as `allkeys-lfu`): This caps Redis memory at 250 MB and evicts least-frequently-used keys. However, it does not prevent the attack — it only bounds memory growth. Every new unique write at maxmemory triggers an eviction cycle. The attacker's single-use entries (LFU score = 1) are evicted, but the eviction CPU overhead is real and continuous. More critically, legitimate high-frequency entries are also displaced during the eviction scan.
- **`maxmemory=250MB`**: Caps memory but does not cap the rate of write+evict cycles. At high request rates, Redis spends a significant fraction of CPU on eviction bookkeeping.
- **`DEFAULT_REDIS_EXPIRY = 1`** (line 24): Only applies when no `cache-control` header is present. Both transaction endpoints set explicit `max-age` values (2s and 600s), so this fallback does not apply.
- **No rate limiting** on cache writes: There is no per-IP, per-endpoint, or global rate limit on how many unique keys can be written to Redis per second.

### Impact Explanation

The attack has two compounding effects:

1. **Redis eviction overhead**: With `maxmemory=250MB` and `allkeys-lfu`, once Redis is full, every incoming unique-key write triggers an eviction scan. At sustained high request rates, this can consume >30% of Redis CPU compared to a baseline where the cache is warm and evictions are rare.
2. **Cache hit rate collapse**: The effective cache hit rate drops to ~0% for the targeted endpoint. Every request falls through to the PostgreSQL database. With a DB pool of only 10 connections (`maxConnections: 10`, documented default), even moderate flood rates saturate the pool, causing query queuing and latency spikes across all endpoints — a denial-of-service effect on the broader API.

The `/api/v1/transactions/{id}` endpoint is the highest-impact target: its 600-second TTL means each poisoned entry occupies Redis for 10 minutes, maximizing the ratio of attacker-controlled entries to legitimate entries.

### Likelihood Explanation

No authentication, API key, or rate limiting is required. Any client with network access to the public REST API can execute this attack. The attack is trivially scriptable:

```bash
for i in $(seq 1 100000); do
  curl -s "https://<mirror-node>/api/v1/transactions?timestamp=1700000000.$(printf '%09d' $i)" &
done
```

The attack is repeatable, stateless, and requires no prior knowledge of the system beyond the public API documentation. The timestamp parameter accepts nanosecond-precision values, providing effectively unbounded cardinality.

### Recommendation

1. **Implement URL normalization before hashing**: Sort query parameters canonically and strip parameters that do not affect the response (or use a whitelist of cacheable parameter combinations). This is already tracked as issue #9113 per the code comment.
2. **Add a per-IP or global rate limit on cache writes**: Limit the number of distinct cache keys that can be written per IP per time window.
3. **Validate/bound timestamp parameters**: Enforce that `timestamp` query values fall within a reasonable recent window, reducing the effective cardinality of cacheable URLs.
4. **Consider `volatile-lfu` or a separate Redis instance** for response caching to isolate blast radius from other cache uses.
