### Title
Unbounded Concurrent Subscription Resource Exhaustion via Missing gRPC Keepalive and No Global Connection Cap

### Summary
`TopicMessageServiceImpl.subscribeTopic()` increments `subscriberCount` on `doOnSubscribe` and decrements it only in `doFinally`. Because no gRPC keepalive is configured and no maximum connection count is enforced, an attacker can open arbitrarily many connections (each carrying up to 5 streams), trigger or simulate a network partition, and leave server-side reactive pipelines permanently hung — `doFinally` is never invoked on a hung pipeline, so `subscriberCount` never decrements and every associated resource (Reactor pipeline objects, `boundedElastic` scheduler tasks, Redis/DB listener registrations, gRPC stream contexts) accumulates indefinitely.

### Finding Description

**Exact code path:**

`subscribeTopic()` at lines 87–91:
```java
return topicExists(filter)
        .thenMany(flux.doOnNext(topicContext::onNext)
                .doOnSubscribe(s -> subscriberCount.incrementAndGet())   // line 89
                .doFinally(s -> subscriberCount.decrementAndGet())       // line 90
                .doFinally(topicContext::finished));                     // line 91
``` [1](#0-0) 

`doFinally` in Project Reactor is guaranteed to fire on `onComplete`, `onError`, and `CANCEL`. A `CANCEL` signal is only propagated when the gRPC layer detects the client connection is gone. During a true network partition (no TCP FIN/RST, just silence), the server-side Netty/gRPC layer cannot detect the dead connection without keepalive probes.

**Missing keepalive configuration:**

`GrpcConfiguration` configures only `maxConcurrentCallsPerConnection` and the executor. No `keepAliveTime`, `keepAliveTimeout`, `maxConnectionAge`, `maxConnectionIdle`, or `maxConnections` is set:
```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
``` [2](#0-1) 

Without keepalive, the OS TCP keepalive default (Linux: `tcp_keepalive_time = 7200 s`) governs when a dead connection is detected — meaning hung pipelines can persist for up to 2 hours per batch.

**No global subscriber cap:**

`subscriberCount` is registered solely as a Micrometer gauge for observability:
```java
Gauge.builder("hiero.mirror.grpc.subscribers", () -> subscriberCount)
        .description("The number of active subscribers")
        ...
        .register(meterRegistry);
``` [3](#0-2) 

There is no guard of the form `if (subscriberCount.get() >= MAX) return Mono.error(...)` anywhere in `subscribeTopic()`. The counter is never consulted before accepting a new subscription.

**Per-connection limit is insufficient:**

`maxConcurrentCallsPerConnection` defaults to 5: [4](#0-3) 

This limits 5 streams per TCP connection, but there is no `maxConnections` call in `GrpcConfiguration`, so an attacker can open an unbounded number of TCP connections, each carrying 5 subscriptions.

**Each hung subscription holds real resources:**

- A `TopicContext` object (per-subscription state, `AtomicLong`, `AtomicReference`, `Stopwatch`)
- A `Flux.merge(safetyCheck, live)` pipeline chain with a `Schedulers.boundedElastic()` task scheduled 1 s after subscription
- A registration with `topicListener` (Redis or DB poll listener) that fans out messages to the subscriber
- A gRPC `ServerCall` context and associated HTTP/2 stream [5](#0-4) 

### Impact Explanation

An attacker who opens `C` connections × 5 streams each and then causes or simulates a network partition accumulates `5C` permanently hung pipelines. Each pipeline holds memory (Reactor chain + `TopicContext`), a `boundedElastic` thread slot (safety-check delay), and a listener registration. With enough connections the JVM heap is exhausted, the `boundedElastic` scheduler queue fills, and the shared `topicListener` fan-out list grows without bound — resulting in a full denial of service for all legitimate subscribers. The `subscriberCount` metric also becomes permanently inflated, masking the real subscriber count and defeating the `GrpcNoSubscribers` alert.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`; any client that knows a valid topic ID (all topic IDs are public on-chain) can subscribe. Opening thousands of TCP connections from a single host or a small botnet is trivial with standard gRPC client libraries. A network partition can be simulated by the attacker simply dropping all inbound packets from the server after the subscription is established (e.g., via a firewall rule or by using a raw socket that never sends ACKs). This requires no special privilege and is repeatable.

### Recommendation

1. **Configure gRPC server keepalive** in `GrpcConfiguration`:
   ```java
   serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
   serverBuilder.maxConnectionAge(5, TimeUnit.MINUTES);
   serverBuilder.maxConnectionIdle(2, TimeUnit.MINUTES);
   ```
2. **Add a `maxConnections` limit** to `NettyProperties` and apply it via `serverBuilder.maxConnectionAge` / connection-count tracking or an external load-balancer policy.
3. **Enforce a global subscriber cap** inside `subscribeTopic()`:
   ```java
   if (subscriberCount.get() >= grpcProperties.getMaxSubscribers()) {
       return Flux.error(new StatusRuntimeException(Status.RESOURCE_EXHAUSTED));
   }
   ```
4. **Add a per-subscription idle timeout** so that subscriptions receiving no messages for a configurable period are terminated server-side.

### Proof of Concept

```python
import grpc, threading
# Assume proto stubs are generated
stubs = []
for i in range(2000):          # 2000 connections
    ch = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(ch)
    for _ in range(5):         # 5 streams per connection = 10,000 subscriptions
        req = ConsensusTopicQuery(topic_id=TopicID(topic_num=1))
        it = stub.subscribeTopic(req)
        threading.Thread(target=lambda i=it: list(i), daemon=True).start()

# Now block all return traffic from the server (simulate partition):
# iptables -A INPUT -p tcp --sport 5600 -j DROP

# Server-side subscriberCount now at 10,000 and never decrements.
# Repeat until OOM / thread exhaustion.
```
