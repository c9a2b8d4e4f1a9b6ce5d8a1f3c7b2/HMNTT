### Title
Missing Rate Limiting on `GET /api/v1/network/registered-nodes` Enables Sustained Resource Exhaustion

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module has no application-level rate limiting. Any unauthenticated external user can send an unbounded stream of individually valid requests, each triggering a full-range database query against the `registered_node` table. The only throttling infrastructure in the codebase (`ThrottleConfiguration`/`ThrottleManagerImpl`) is scoped exclusively to the `web3` module and is never applied to `rest-java` endpoints.

### Finding Description

**Code path:**

`NetworkController.java` line 173–187 handles `GET /api/v1/network/registered-nodes` with no rate-limiting annotation or filter: [1](#0-0) 

It delegates to `NetworkServiceImpl.getRegisteredNodes()` (lines 139–152), which resolves bounds defaulting to `lowerBound=0L, upperBound=Long.MAX_VALUE` when no `registerednode.id` parameter is supplied: [2](#0-1) [3](#0-2) 

This calls the repository query: [4](#0-3) 

The SQL issued is:
```sql
SELECT * FROM registered_node
WHERE registered_node_id >= 0
  AND registered_node_id <= 9223372036854775807
  AND deleted IS FALSE
  AND (NULL IS NULL OR type @> ARRAY[NULL]::smallint[])
ORDER BY registered_node_id ASC
LIMIT 100
```

The `@Max(MAX_LIMIT)` constraint on `RegisteredNodesRequest.limit` caps rows returned at 100, but imposes no constraint on request frequency: [5](#0-4) 

**Why existing checks fail:**

The only rate-limiting infrastructure in the repository (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) is declared under the `web3` module and is wired exclusively to `ContractCallRequest` processing: [6](#0-5) 

The `rest-java` config package contains only `LoggingFilter`, `MetricsFilter`, `ShallowEtagHeaderFilter`, and `WebMvcConfiguration` — none of which implement rate limiting: [7](#0-6) 

No `@RateLimiter`, `bucket4j`, or servlet filter for rate limiting is present anywhere in the `rest-java` module. The `statementTimeout` of 10,000 ms documented for `rest-java` only kills individual slow queries, not high-frequency valid ones.

### Impact Explanation
Each request causes PostgreSQL to execute a range scan on `registered_node` with an array containment check (`type @> array[:type]::smallint[]`), sort, and LIMIT. Under sustained concurrent load (e.g., thousands of requests/second from a single or distributed source), the cumulative effect is: repeated index scans, buffer pool pressure, connection pool saturation, and CPU usage from query planning and execution. Even for a small table, at sufficient request rates this can push DB I/O and CPU well above the 30% baseline threshold. The `rest-java` DB user (`mirror_rest_java`) has no query-level throttle, and the connection pool is shared across all concurrent requests.

### Likelihood Explanation
No authentication, API key, or credential is required. The endpoint is publicly documented in `openapi.yml`. A single attacker with a commodity machine or a small botnet can sustain thousands of requests per second. The attack is trivially repeatable, requires no special knowledge beyond the public API spec, and leaves no distinguishing fingerprint beyond high request volume.

### Recommendation
1. Add application-level rate limiting to `rest-java` using bucket4j (already a dependency in `web3`) or Spring's `@RateLimiter`, scoped per IP or globally, applied via a servlet filter registered in `RestJavaConfiguration`.
2. Alternatively, configure infrastructure-level rate limiting (e.g., Traefik `rateLimit` middleware) for the `rest-java` service, analogous to what is already configured for the `rosetta` service.
3. Consider adding a database-level connection pool limit per endpoint or per client IP.

Monitor PostgreSQL CPU and I/O (`pg_stat_activity`, `pg_stat_bgwriter`) before and during the attack. With no rate limiting in `rest-java`, all requests reach the database. At sufficient concurrency, cumulative DB CPU and I/O will exceed 30% above the 24-hour baseline.

    private static Range<Long> resolveRegisteredNodeIdBounds(List<NumberRangeParameter> registeredNodeIdRanges) {
        long lowerBound = 0L;
        long upperBound = MAX_VALUE;

        for (final var range : registeredNodeIdRanges) {
            if (range.operator() == RangeOperator.EQ) {
                if (registeredNodeIdRanges.size() > 1) {
                    throw new IllegalArgumentException("The 'eq' operator cannot be combined with other operators");
                }
                return Range.closed(range.value(), range.value());
            } else if (range.hasLowerBound()) {
                lowerBound = Math.max(lowerBound, range.getInclusiveValue());
            } else if (range.hasUpperBound()) {
                upperBound = Math.min(upperBound, range.getInclusiveValue());
            }
        }

        if (lowerBound > upperBound) {
            throw new IllegalArgumentException("Invalid range: lower bound exceeds upper bound");
        }

        return Range.closed(lowerBound, upperBound);
    }

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();

                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();

                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
