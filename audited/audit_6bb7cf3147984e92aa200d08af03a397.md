### Title
Unbounded Indefinite Subscription Accumulation via Missing `endTime` Leads to Memory Exhaustion DoS

### Summary
When a client subscribes to a topic without setting `endTime`, `TopicMessageServiceImpl.pastEndTime()` unconditionally returns `Flux.never()`, causing the reactive pipeline's `takeUntilOther()` termination signal to never fire. Because no global subscription cap exists and the only per-connection limit (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening many connections, an unauthenticated attacker can accumulate thousands of indefinitely-live subscriptions, each holding JVM heap state, until the process runs out of memory.

### Finding Description

**Exact code path:**

In `TopicMessageServiceImpl.java`, `pastEndTime()` at lines 123–131:

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // ← never emits, never completes
    }
    ...
}
``` [1](#0-0) 

This return value is consumed in `subscribeTopic()` at line 73:

```java
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
``` [2](#0-1) 

`takeUntilOther(Flux.never())` means the operator's termination branch never fires. The two remaining termination paths — `takeWhile` (line 79–81) and `take(limit)` (line 83–85) — are both guarded by null/zero checks and are skipped when `endTime` is null and `limit` is 0: [3](#0-2) 

**Root cause:** The design assumes that either `endTime` or `limit` will bound every subscription. There is no enforcement of this assumption — `TopicMessageFilter` validation only checks that `endTime > startTime` *if* `endTime` is provided, and `limit` defaults to 0 (unlimited): [4](#0-3) 

**Per-subscription state that accumulates:** Each call to `subscribeTopic()` allocates a `TopicContext` containing `AtomicLong count`, `AtomicReference<TopicMessage> last`, a Guava `Stopwatch`, an `EntityId`, and the full `TopicMessageFilter`. The Reactor pipeline itself (operator chain, scheduler hooks, `Mono.delay` on `boundedElastic`) also retains heap references for the lifetime of the subscription: [5](#0-4) 

**Why the only guard is insufficient:** `NettyProperties.maxConcurrentCallsPerConnection` defaults to 5 and is enforced per TCP connection: [6](#0-5) [7](#0-6) 

There is no global cap on total connections or total subscriptions. The `subscriberCount` `AtomicLong` is a Micrometer gauge only — it is never checked against a maximum: [8](#0-7) 

### Impact Explanation
An attacker who opens *N* TCP connections and sends 5 subscriptions per connection (each with no `endTime`, no `limit`, targeting any valid topic) accumulates 5*N* indefinitely-live reactive pipelines. Each pipeline holds JVM heap objects for its entire lifetime. At sufficient scale this exhausts heap, triggering `OutOfMemoryError` and crashing the gRPC service. Because the gRPC mirror node is the only interface for HCS topic subscriptions, its unavailability directly impacts all consumers of Hedera Consensus Service data, qualifying as a high-severity non-network DoS against a critical infrastructure component.

### Likelihood Explanation
No authentication or API key is required to call `subscribeTopic`. The gRPC port (default 5600) is publicly reachable. The attacker needs only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or a raw HTTP/2 client). The attack is repeatable and can be automated in a loop. The per-connection limit of 5 is trivially bypassed by opening many connections — a single machine with default OS socket limits can sustain thousands of concurrent TCP connections. The attack is slow and stealthy (memory grows gradually), making it harder to detect before impact.

### Recommendation
1. **Enforce a maximum subscription lifetime or require `endTime`:** Reject subscriptions where `endTime` is null, or impose a server-side maximum duration (e.g., 24 hours) applied unconditionally in `subscribeTopic()` regardless of what the client sends.
2. **Add a global concurrent-subscription cap:** Check `subscriberCount` against a configurable maximum at the start of `subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Add a per-IP or per-connection subscription limit** at the gRPC interceptor layer to prevent a single client from monopolizing capacity.
4. **Apply a server-side `timeout()` operator** to every subscription pipeline so that subscriptions with no activity for a configurable period are automatically cancelled.

### Proof of Concept

**Preconditions:** gRPC mirror node running with default config; at least one valid topic ID known (or `checkTopicExists = false`).

**Steps:**

```python
# Pseudocode – repeat with any gRPC client
import grpc, threading
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc

def open_subscription(channel):
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_pb2.ConsensusTopicQuery(
        topicID=...,          # any valid topic
        consensusStartTime=0, # no endTime field set
        # limit omitted → defaults to 0 (unlimited)
    )
    for _ in stub.subscribeTopic(req):
        pass  # keep pipeline alive, never terminates

threads = []
for _ in range(2000):                        # 2000 connections
    ch = grpc.insecure_channel("mirror-node:5600")
    for _ in range(5):                       # 5 calls per connection
        t = threading.Thread(target=open_subscription, args=(ch,))
        t.start()
        threads.append(t)

# 10,000 indefinite subscriptions now accumulate JVM heap
# Monitor: watch JVM heap grow until OOM / service crash
```

**Expected result:** JVM heap grows monotonically; after sufficient subscriptions the process throws `OutOfMemoryError` or becomes unresponsive due to GC pressure, denying service to all legitimate subscribers.

        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }
