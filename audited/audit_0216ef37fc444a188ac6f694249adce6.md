### Title
Unauthenticated Mirror REST Bootstrap Allows Network-Adjacent Node Topology Forgery Leading to Fee Manipulation

### Summary
When `network=other` is configured, `newClient()` in `pinger/sdk_client.go` bootstraps the Hiero SDK client by fetching node topology from a plain HTTP mirror REST endpoint with no cryptographic verification of the response. A network-adjacent attacker who can intercept or spoof that HTTP response can inject an arbitrary `map[string]hiero.AccountID` into `hiero.ClientForNetwork(netmap)`, redirecting all subsequent gRPC transaction traffic — including fee-schedule queries — to attacker-controlled nodes, enabling fee inflation outside design parameters.

### Finding Description

**Root cause:** The design assumes the mirror REST endpoint is trusted and its responses are authentic. No TLS is enforced, no HMAC/signature covers the JSON payload, and no allowlist of valid node addresses is consulted. The failed assumption is that a plain HTTP channel inside a cluster is equivalent to an authenticated, integrity-protected channel.

**Exploit flow:**
1. Attacker positions themselves to intercept traffic between the pinger pod and `http://mirror-rest:5551` (ARP spoofing, DNS poisoning, rogue pod on the same overlay network, or a compromised sidecar).
2. Attacker intercepts the GET `/api/v1/network/nodes` request and returns a crafted JSON body listing attacker-controlled gRPC endpoints with syntactically valid account IDs (e.g., `0.0.3`, `0.0.4`).
3. `fetchMirrorNodeNetwork` accepts the response (HTTP 200, valid JSON, parseable account IDs) and returns the forged `netmap`.
4. `hiero.ClientForNetwork(netmap)` configures the SDK to route all consensus-node gRPC calls to the attacker's endpoints.
5. When `submitWithRetry` calls `cryptoTransfer.Execute(client)`, the SDK first queries the attacker's node for the current fee schedule. The attacker returns an inflated fee schedule.
6. The SDK sets `maxTransactionFee` based on the inflated schedule and signs the transaction with the operator key. The attacker can then proxy the transaction to the real network with the inflated fee field intact, causing the operator to overpay, or simply drop transactions for DoS.

**Why existing checks are insufficient:**

- The only response validation is an HTTP status-code range check (`200–299`) and JSON parseability — neither provides any authenticity guarantee.
- `hiero.AccountIDFromString` validates format only, not that the account ID belongs to a legitimate consensus node.
- There is no TLS `RootCAs`, `InsecureSkipVerify=false` enforcement, or certificate pinning on the `http.Client`.
- The `network=other` branch deliberately bypasses the hardcoded node lists used by `hiero.ClientForName` for mainnet/testnet/previewnet, so those SDK-level protections do not apply here.

### Impact Explanation

An attacker who successfully forges the node topology gains full control over which gRPC endpoints the pinger uses for every transaction. Concretely:

- **Fee inflation:** The attacker's node returns an inflated fee schedule; the SDK uses it to set `maxTransactionFee`; the operator's account is debited more than the legitimate network fee on every periodic transfer.
- **Transaction suppression (DoS):** The attacker's node can silently drop all submitted transactions, causing the pinger to report failures and breaking the liveness/readiness probes that depend on successful transfers.
- **Credential exposure risk:** The operator private key signs transactions destined for the attacker's node; the attacker sees the signed transaction bytes (though the key itself is not directly exposed).

Severity: **High** — direct financial impact (fee drain) on the operator account, repeatable on every tick of the transfer loop.

### Likelihood Explanation

- **Precondition 1 (`network=other`):** This is an explicit operator choice for private/custom deployments, which is exactly the scenario where an internal HTTP mirror URL is most likely to be used.
- **Precondition 2 (plain HTTP):** The documented example in `config.go` (`http://mirror-rest:5551`) is plain HTTP. Kubernetes internal service communication commonly uses plain HTTP.
- **Precondition 3 (network adjacency):** In a Kubernetes cluster, any pod in the same namespace (or with network policy gaps) can perform ARP spoofing or DNS poisoning against cluster-internal DNS names. A compromised dependency or a misconfigured network policy is sufficient — no privileges on the pinger pod are required.
- **Repeatability:** The bootstrap happens once at startup; a single successful interception permanently redirects all traffic for the lifetime of the process.

### Recommendation

1. **Enforce HTTPS with certificate validation** for `cfg.mirrorRest` when `network=other`. Reject `http://` URLs at config-load time or configure the `http.Client` with a strict `tls.Config`.
2. **Pin the mirror node's TLS certificate or CA** in the `http.Client.Transport` so that a forged certificate cannot satisfy the handshake.
3. **Validate returned node addresses against an operator-supplied allowlist** of known consensus node account IDs before passing `netmap` to `hiero.ClientForNetwork`.
4. **Re-fetch and re-validate the node list periodically** rather than trusting a single bootstrap response for the entire process lifetime.

### Proof of Concept

```
# 1. Deploy pinger with:
#    HIERO_MIRROR_PINGER_NETWORK=other
#    HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551

# 2. On a pod in the same K8s namespace, poison DNS for "mirror-rest"
#    to resolve to attacker-controlled IP, or use ARP spoofing.

# 3. Serve a crafted HTTP response on port 5551:
cat <<'EOF' > fake_nodes.json
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{
      "domain_name": "attacker.example.com",
      "ip_address_v4": "",
      "port": 50211
    }]
  }],
  "links": {"next": null}
}
EOF
python3 -m http.server 5551  # serve fake_nodes.json at /api/v1/network/nodes

# 4. On attacker.example.com:50211, run a gRPC server that:
#    - Responds to CryptoGetInfo/FeeSchedule queries with 10x inflated fees
#    - Proxies signed TransferTransaction to the real network

# 5. Observe pinger logs: every transfer uses the inflated maxTransactionFee,
#    draining the operator account at 10x the legitimate rate.
```
