### Title
Unbounded Indefinite Polling from Epoch Timestamp Enables DB Griefing via `PollingTopicListener.listen()`

### Summary
An unprivileged user can open an arbitrary number of concurrent gRPC `subscribeTopic` streams with `startTime=0` (epoch), no `endTime`, and no `limit`. Each subscription triggers `PollingTopicListener.listen()`, which schedules `Long.MAX_VALUE` DB poll iterations with no termination condition, each querying `topic_message` from `consensus_timestamp >= 0` every 500 ms. With no per-client subscription cap or rate limit, an attacker can saturate the DB connection pool and `boundedElastic` thread pool, degrading service for all legitimate subscribers.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `incomingMessages()` → `topicListener.listen(newFilter)` → `PollingTopicListener.listen()` → `poll()` → `TopicMessageRepositoryCustomImpl.findByFilter()`

**Root cause — `isValidStartTime()` only enforces an upper bound, not a lower bound:**

```java
// TopicMessageFilter.java line 49-51
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();
}
```

`startTime = 0` (epoch, nanoseconds) satisfies `0 <= now()` and also satisfies `@Min(0)`. There is no minimum-age guard.

**Infinite polling loop with no termination:**

```java
// PollingTopicListener.java line 38-43
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

`Long.MAX_VALUE` ≈ 9.2 × 10¹⁸ repeats. With `interval = 500 ms` (default), this runs for ~146 billion years. No `endTime` means `pastEndTime()` in `TopicMessageServiceImpl` returns `Flux.never()` (line 124–126). No `limit` means `flux.take()` is never applied (line 83–85).

**Each poll queries from epoch:**

```java
// PollingTopicListener.java line 58-59
long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
```

When no messages have been received yet (`last == null`), every poll uses `filter.getStartTime() = 0`. The resulting JPA query is:

```sql
SELECT * FROM topic_message
WHERE topic_id = X AND consensus_timestamp >= 0
ORDER BY consensus_timestamp
LIMIT 5000
```

This is the maximum historical scan range for that topic.

**No subscription cap exists:**

`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge metric (line 48–55) — it is never checked against a maximum. `ConsensusController` applies no per-IP or per-connection limit before calling `subscribeTopic`.

### Impact Explanation

Each malicious subscription generates 2 DB queries/second (one per 500 ms interval), each scanning `topic_message` from `consensus_timestamp = 0`. With N concurrent subscriptions:
- DB query rate: 2N queries/second
- Each query fetches up to `maxPageSize = 5000` rows
- The `boundedElastic` scheduler (default: 10× CPU cores threads) and the DB connection pool are finite shared resources

Saturating either resource causes legitimate subscribers to queue or time out. The `@Transactional(readOnly = true)` annotation on `findByFilter` holds a DB connection for the duration of each poll, amplifying connection pool exhaustion. This is a pure griefing DoS: no funds at risk, but service availability is degraded for all users of the mirror node's HCS subscription endpoint.

### Likelihood Explanation

- **Precondition**: The server must be configured with `hiero.mirror.grpc.listener.type = POLL` (not the default `REDIS`). This is a valid and documented configuration.
- **Attacker capability**: Any unauthenticated client with network access to the gRPC port. No credentials, tokens, or special knowledge required.
- **Repeatability**: Fully scriptable. A single attacker machine can open hundreds of concurrent gRPC streams using any standard gRPC client library.
- **Detection difficulty**: Each individual subscription looks legitimate (valid `topicId`, valid `startTime`). The only anomaly is volume.

### Recommendation

1. **Enforce a minimum `startTime` age**: Add a configurable lower bound (e.g., reject `startTime` older than 7 days) in `isValidStartTime()` or at the service layer.
2. **Cap concurrent subscriptions per remote address**: Track active subscriptions by client IP in `ConsensusController` or `TopicMessageServiceImpl` and reject new ones above a threshold.
3. **Enforce a maximum subscription lifetime**: Add a server-side timeout (e.g., 24 hours) to the polling `Flux`, replacing `Long.MAX_VALUE` with a bounded repeat count or a `timeout()` operator.
4. **Add back-pressure / rate limiting**: Apply a token-bucket or semaphore on the number of concurrent `PollingTopicListener` instances to bound total DB query rate.

### Proof of Concept

```python
import grpc
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

# Open N concurrent subscriptions with startTime=0, no endTime, no limit
channels = []
for _ in range(200):  # 200 concurrent subscriptions
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no consensusEndTime, no limit
    )
    # Each stream triggers PollingTopicListener with startTime=0, Long.MAX_VALUE repeats
    stream = stub.subscribeTopic(query)
    channels.append((channel, stream))

# Each subscription now polls DB every 500ms from consensus_timestamp=0
# 200 subscriptions × 2 polls/sec = 400 queries/sec, each scanning from epoch
import time
time.sleep(60)  # sustain for 60 seconds to exhaust DB connection pool
```
