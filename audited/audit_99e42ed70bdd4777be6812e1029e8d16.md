### Title
Unbounded Non-Parameterized IN-Clause in `getInClauseSubQuery()` Enables DB Connection Exhaustion via Concurrent Requests

### Summary
An unprivileged external user can send concurrent requests to `GET /:idOrAliasOrEvmAddress/tokens` with the maximum allowed `token.id=eq:` filters, causing `getInClauseSubQuery()` to build a non-parameterized IN-clause SQL string. Each unique combination of token IDs generates a distinct query string that bypasses PostgreSQL's query plan cache, and concurrent flooding of such requests holds DB connections open, contributing to resource exhaustion.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js` line 19 → `TokenController.getTokenRelationships()` (`rest/controllers/tokenController.js` lines 66–92) → `extractTokensRelationshipQuery()` (lines 22–58) → `TokenService.getTokenAccounts()` → `getInClauseSubQuery()` (`rest/service/tokenService.js` lines 78–87).

**Root cause — non-parameterized IN clause:** [1](#0-0) 

```js
getInClauseSubQuery(inConditions, params) {
  const tokenIdInParams = [];
  inConditions.forEach((condition) => {
    tokenIdInParams.push(condition.value);
  });
  if (!isEmpty(tokenIdInParams)) {
    return ` and ${TokenAccount.getFullName(TokenAccount.TOKEN_ID)} in (${tokenIdInParams})`;
  }
}
```

The values in `tokenIdInParams` are **interpolated directly into the SQL string** — they are never pushed into the `params` array and never become `$N` placeholders. Every unique set of token IDs produces a unique query string. PostgreSQL cannot reuse a cached plan for any two requests with different token ID combinations.

**How inConditions grows:** [2](#0-1) 

Every `token.id=eq:X` filter appends one entry to `inConditions` with no per-request cap inside `extractTokensRelationshipQuery()`.

**The only guard — `maxRepeatedQueryParameters`:** [3](#0-2) 

`buildFilters()` checks `isRepeatedQueryParameterValidLength(values)` and rejects a single parameter array that exceeds `config.query.maxRepeatedQueryParameters`. This bounds the IN-list size per request, but does **not** prevent:
- Many concurrent requests each at the maximum allowed count
- Query plan cache pollution (each unique token-ID combination is a new query string)
- DB connection pool exhaustion under concurrent load

### Impact Explanation

Each request with the maximum allowed `token.id=eq:` filters executes a unique, non-parameterized SQL query against the `token_account` table. PostgreSQL must parse and plan each distinct query string independently. Under concurrent flooding, this causes:
1. DB connection pool saturation (each slow query holds a connection)
2. Query plan cache thrashing (unique strings evict cached plans)
3. Elevated CPU on the DB server for repeated query planning

This can degrade or deny service to all API consumers sharing the same DB pool — a network partition effect from the application's perspective.

### Likelihood Explanation

No authentication or rate limiting is required. Any external user knowing a valid account ID (publicly discoverable via the `/accounts` endpoint) can immediately exploit this. The attack is trivially scriptable: send hundreds of concurrent requests each carrying the maximum number of `token.id=eq:` filters. The `maxRepeatedQueryParameters` guard only limits the IN-list size per request, not the request rate or concurrency.

### Recommendation

1. **Parameterize the IN clause**: Push each token ID into `params` and use `$N` placeholders, matching the pattern used elsewhere (e.g., `contractController.js` lines 126–131). [4](#0-3) 
2. **Add a strict, low cap on `token.id=eq:` filter count** specific to this endpoint, independent of the global `maxRepeatedQueryParameters`.
3. **Apply request-level rate limiting** on the `/accounts/:id/tokens` route.

### Proof of Concept

```bash
# Discover a valid account ID
ACCOUNT=$(curl -s https://<mirror-node>/api/v1/accounts?limit=1 | jq -r '.accounts[0].account')

# Build a request with N token.id=eq: filters (up to maxRepeatedQueryParameters)
FILTERS=$(python3 -c "print('&'.join(['token.id=eq:' + str(i) for i in range(1, 101)]))")

# Flood concurrently
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT}/tokens?${FILTERS}" &
done
wait
```

Each concurrent request executes a unique non-parameterized SQL query, exhausting the DB connection pool and causing query plan cache thrashing. Monitor `pg_stat_activity` to observe connection saturation.
