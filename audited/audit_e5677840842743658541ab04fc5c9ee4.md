### Title
Unauthenticated Parallel Paginated Requests to `getBlocks()` Exhaust the Default 10-Connection Database Pool (DoS)

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts unauthenticated, unbounded concurrent requests. The REST API's database connection pool defaults to only 10 connections with a 20-second statement timeout. An attacker sending 10 or more concurrent `GET /api/v1/blocks?limit=1&block.number=gte:0` requests — and continuously repeating them — saturates the pool, causing all subsequent legitimate requests to wait up to 20 seconds before receiving a connection-timeout error, effectively denying service to all users.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js` line 12 registers the route with no authentication or rate-limiting middleware:
```
router.getExt('/', BlockController.getBlocks);
```

`rest/controllers/blockController.js` lines 101–112 — `getBlocks()` — performs no concurrency check, no per-IP throttle, and no request queuing before issuing a database query:
```js
getBlocks = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedBlockParameters);
  const formattedFilters = this.extractSqlFromBlockFilters(filters);
  const blocks = await RecordFileService.getBlocks(formattedFilters);   // acquires a pool connection
  ...
};
```

`rest/dbpool.js` lines 7–16 configures the pool:
```js
const poolConfig = {
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20 000 ms
  max: config.db.pool.maxConnections,                         // default 10
  statement_timeout: config.db.pool.statementTimeout,         // default 20 000 ms
};
```

The documented default for `maxConnections` is **10** and `statementTimeout` is **20 000 ms** (20 s).

`generateNextLink` (lines 90–98) always emits a `next` cursor (`block.number=gt:<last_index>`) as long as any blocks exist, giving the attacker an infinite chain of valid follow-up URLs.

**Root cause:** The REST API middleware stack (`rest/middleware/index.js`) contains only `authHandler`, `requestHandler`, `responseCacheHandler`, `responseHandler`, `metricsHandler`, and `openapiHandler`. None implement per-IP rate limiting or in-flight concurrency caps for the blocks endpoint. The only throttling in the codebase (`web3/src/main/java/…/ThrottleConfiguration.java`) applies exclusively to the web3 `/contracts/call` endpoint.

### Impact Explanation
With a default pool of 10 connections and a 20-second statement timeout, an attacker holding 10 concurrent open queries monopolises every available connection. All other users' requests queue waiting for a free connection; after `connectionTimeoutMillis` (20 s) they receive a 500 error. The attack is global — it degrades every REST API endpoint that touches the same pool, not just `/blocks`. Recovery requires the attacker to stop sending requests; as long as they maintain the flood the service remains unavailable.

### Likelihood Explanation
No authentication, API key, or network-level credential is required. The attack requires only a standard HTTP client capable of issuing 10 concurrent GET requests in a loop. The `next` link in each response provides the exact URL for the next page, making automation trivial. The attacker does not need to know any block numbers in advance (`gte:0` covers the entire chain). This is repeatable indefinitely from a single IP or distributed across multiple IPs to evade any IP-based infrastructure controls.

### Recommendation
1. **Add an application-level concurrency/rate-limit middleware** to the REST API (e.g., `express-rate-limit` or a custom semaphore) that caps in-flight DB-bound requests per IP before they reach the pool.
2. **Increase the default pool size** or, better, use a connection-queue with a maximum wait queue depth so that excess requests are rejected with HTTP 429 rather than timing out after 20 s.
3. **Add a Traefik/nginx `inFlightReq` middleware** for the REST API similar to the one already present for the Rosetta API (`charts/hedera-mirror-rosetta/values.yaml` lines 152–155: `inFlightReq: amount: 5`).
4. Consider adding a minimum `block.number` lower-bound or requiring a tighter filter to prevent open-ended full-table scans.

### Proof of Concept
```bash
# Exhaust the 10-connection pool with 12 concurrent requests, repeated continuously
for i in $(seq 1 12); do
  curl -s "https://<mirror-node>/api/v1/blocks?limit=1&block.number=gte:0" &
done
wait

# Legitimate request now times out (connectionTimeoutMillis = 20 000 ms → HTTP 500)
curl -v "https://<mirror-node>/api/v1/blocks?limit=1"
```
While the 12 background requests hold all 10 pool connections (each for up to 20 s), the final `curl` will block for 20 seconds and then receive a 500 Internal Server Error.
