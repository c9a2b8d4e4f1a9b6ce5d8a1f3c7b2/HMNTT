### Title
Cache Exhaustion via Unbounded Contract ID Flooding in `findRuntimeBytecode()` Enabling Sustained DB Query Amplification

### Summary
`ContractRepository.findRuntimeBytecode()` is backed by a Caffeine cache with a hard `maximumSize=1000` and `expireAfterAccess=1h`. The `@Cacheable` annotation's `unless = "#result == null"` guard is ineffective because the method returns `Optional<byte[]>`, which is never `null` — meaning even lookups for non-existent contracts are cached. An unprivileged attacker can flood the endpoint with requests targeting more than 1000 unique contract IDs, exhausting the cache and forcing continuous DB queries for every legitimate user's contract bytecode lookup, degrading EVM execution for all users.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Cache configuration** (`CacheProperties.java`, line 22):
```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```

Wired in `EvmConfiguration.java` lines 67–73:
```java
@Bean(CACHE_MANAGER_CONTRACT)
CacheManager cacheManagerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getContract());
    return caffeineCacheManager;
}
```

**Root cause — failed `unless` guard:**
Spring Cache evaluates `unless = "#result == null"` after method execution. The return type is `Optional<byte[]>`. Spring Data JPA never returns a raw `null` for an `Optional`-typed query method — it returns `Optional.empty()` for missing rows. `Optional.empty() != null`, so the condition is always `false`, and every result (including lookups for non-existent contracts) is unconditionally cached. This means an attacker does not need valid contract addresses; any arbitrary `Long` ID works.

**Exploit flow:**
1. Attacker sends `eth_call` (POST `/api/v1/contracts/call`) targeting contract address `0x0000...0001`, `0x0000...0002`, …, `0x0000...03E9` (1000 unique IDs). Cache fills to capacity.
2. Attacker sends request for ID `0x0000...03EA` (1001st unique ID). Caffeine evicts the LRU entry (ID 1).
3. Attacker continues cycling through IDs 1–1001 in a round-robin pattern. Every cycle evicts one legitimate entry.
4. Any legitimate user whose contract was evicted now incurs a DB query (`select runtime_bytecode from contract where id = :contractId`) on every subsequent `eth_call` to that contract.
5. The attacker sustains this indefinitely, keeping the cache in a permanent thrashing state.

**Why existing checks are insufficient:**

- **Global rate limit** (`ThrottleProperties.java` line 35, `ThrottleManagerImpl.java` lines 38–42): `requestsPerSecond = 500` is a **global** bucket shared across all clients, not per-IP. A single attacker consuming 500 RPS fills the 1000-entry cache in 2 seconds and then sustains evictions at the full allowed rate.
- **Gas throttle** (`ThrottleManagerImpl.java` line 40): Gas throttling applies to gas consumed per request. Minimal-gas `eth_call` requests (e.g., simple `STATICCALL` to a contract) consume very little gas, allowing the attacker to stay well within the gas budget while maximizing unique contract ID coverage.
- **`unless = "#result == null"`**: As explained above, this guard never fires for `Optional<byte[]>` return types, so negative results (non-existent contracts) are cached and the attacker needs no knowledge of real contract IDs.

### Impact Explanation

Every `eth_call` targeting a contract whose cache entry has been evicted falls through to a synchronous DB query. Under sustained attack, the effective cache hit rate for the contract bytecode cache approaches zero for legitimate traffic. This increases DB connection pool pressure and query latency for all concurrent EVM executions. Since `findRuntimeBytecode()` is called on the hot path of every contract execution (bytecode must be loaded before the EVM can execute), degraded cache performance directly translates to degraded `eth_call` and `eth_estimateGas` throughput for all users of the mirror node. The `db.statementTimeout = 3000ms` means each cache miss can hold a DB connection for up to 3 seconds, compounding under load.

### Likelihood Explanation

The attack requires zero privileges, zero on-chain assets, and no special knowledge — only the ability to send HTTP POST requests to the public `/api/v1/contracts/call` endpoint. The attacker needs no valid contract addresses (arbitrary `Long` IDs suffice due to the `unless` guard bypass). The attack is trivially scriptable, repeatable indefinitely, and self-sustaining once the cache is exhausted. The global (non-per-IP) rate limit does not prevent a single attacker from monopolizing the allowed request budget.

### Recommendation

1. **Fix the `unless` guard** to correctly exclude empty Optionals:
   ```java
   @Cacheable(..., unless = "#result == null || !#result.isPresent()")
   ```
   This prevents non-existent contract IDs from consuming cache slots, eliminating the ability to fill the cache with garbage IDs.

2. **Increase cache size** or make it configurable with a higher default (e.g., `maximumSize=10000`) to raise the cost of exhaustion.

3. **Add per-IP/per-client rate limiting** at the ingress layer (e.g., via a reverse proxy or API gateway) so a single client cannot consume the entire global request budget.

4. **Consider `expireAfterWrite` instead of `expireAfterAccess`** for the contract cache to bound the maximum staleness and reduce the window during which an attacker can keep entries alive by re-accessing them.

### Proof of Concept

```bash
# Fill the 1000-entry cache with non-existent contract IDs, then sustain evictions
for i in $(seq 1 10000); do
  ADDR=$(printf "0x%040x" $i)
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$ADDR\",\"data\":\"0x\",\"gas\":21000}" &
  # Throttle to stay under 500 RPS
  if (( i % 50 == 0 )); then sleep 0.1; fi
done
# After i > 1000, every new unique ID evicts a legitimate entry.
# Legitimate users calling real contracts now miss cache and hit DB on every request.
```
