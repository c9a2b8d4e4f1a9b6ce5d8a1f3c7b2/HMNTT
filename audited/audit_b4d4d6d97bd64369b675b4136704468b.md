### Title
Silent Treasury Substitution as `msg.sender` When Sender Is a Smart Contract in `getSenderAccountID()`

### Summary
When an unprivileged user supplies a non-long-zero EVM address that resolves to an account with `account.smartContract() == true`, `getSenderAccountID()` silently returns the treasury account ID instead of rejecting the request. This treasury account ID is placed into `transactionID.accountID` and used by the Hedera `TransactionExecutor` to derive the EVM sender, causing the EVM simulation to execute with the treasury's EVM address as `msg.sender` rather than the user-supplied contract address. Any contract that branches on `msg.sender` will produce incorrect simulation results.

### Finding Description

**Exact code path:**

In `TransactionExecutionService.java`, `getSenderAccountID()` (lines 239–265):

```java
} else if (account.smartContract()) {
    return EntityIdUtils.toAccountId(systemEntity.treasuryAccount()); // line 251
}
``` [1](#0-0) 

The returned treasury `AccountID` is then placed into `transactionID.accountID` in `defaultTransactionBodyBuilder()`:

```java
.accountID(getSenderAccountID(params))  // line 167
``` [2](#0-1) 

In Hedera's `TransactionExecutor`, the `transactionID.accountID` (the payer) is the account whose EVM address is used as the EVM-level sender (`msg.sender`). The treasury account (e.g., `0.0.2`) maps to the long-zero EVM address `0x0000000000000000000000000000000000000002`.

**Root cause / failed assumption:**

The code's own comment at lines 287–288 acknowledges that the real consensus node (`SolvencyPreCheck#getPayerAccount()`) returns `PAYER_ACCOUNT_NOT_FOUND` when the payer is a smart contract: [3](#0-2) 

However, for non-long-zero addresses, the mirror node does **not** throw this error — it silently substitutes the treasury. The failed assumption is that substituting the treasury as the Hedera-level payer is harmless; in reality, the payer identity propagates directly into the EVM as `msg.sender`.

**Inconsistency confirmed by tests:**

- `testExecuteContractCallSenderIsSmartContract` (non-long-zero address, `smartContract=true`) → expects `SUCCESS` with treasury substituted.
- `testExecuteContractCallInvalidSenderContract` (long-zero address, `smartContract=true`) → expects `PAYER_ACCOUNT_NOT_FOUND`. [4](#0-3) [5](#0-4) 

The same smart-contract sender is rejected for long-zero addresses but silently accepted (with treasury substitution) for non-long-zero addresses — an inconsistency that exposes the substitution path.

### Impact Explanation

Any `eth_call` or `eth_estimateGas` request where `from` is set to a deployed contract's EVM address will execute with `msg.sender = <treasury EVM address>` instead of the supplied address. Contracts that gate logic on `msg.sender` (ownership checks, access control, allowance checks, role-based guards) will return results that diverge from what would happen on the real consensus node. This can cause:

- dApps to receive incorrect simulation results and make wrong decisions (e.g., believing a call would succeed when it would fail on-chain, or vice versa)
- Incorrect gas estimates for transactions that depend on `msg.sender`-gated branches
- Misleading opcode traces (`eth_debugTraceTransaction`) that show the treasury as the initiator

Severity is medium: no real funds are at risk (mirror node is read-only), but the simulation fidelity guarantee is broken in a user-controllable way.

### Likelihood Explanation

Exploitation requires no privileges. Any user of the JSON-RPC API (`eth_call`, `eth_estimateGas`) can set the `from` field to any deployed contract address. Deployed contracts are publicly enumerable. The trigger is trivially repeatable and requires no special knowledge beyond knowing a contract address exists on the network.

### Recommendation

Replace the silent treasury substitution with a rejection consistent with the consensus node's behavior. When `account.smartContract() == true`, throw `PAYER_ACCOUNT_NOT_FOUND` regardless of whether the address is long-zero or not:

```java
} else if (account.smartContract()) {
    throwPayerAccountNotFoundException(SENDER_NOT_FOUND); // instead of returning treasury
}
```

This aligns the mirror node's behavior with `SolvencyPreCheck#getPayerAccount()` on the consensus node, as the existing comment at line 287 already documents as the intended behavior.

### Proof of Concept

1. Identify any deployed smart contract on the network with EVM address `0xABCD...` (non-long-zero).
2. Send an `eth_call` request to the mirror node's JSON-RPC endpoint:
   ```json
   {
     "method": "eth_call",
     "params": [{
       "from": "0xABCD...",
       "to": "<target contract that returns msg.sender>",
       "data": "<selector for getSender()>"
     }, "latest"]
   }
   ```
3. Observe that the returned value is the treasury's long-zero EVM address (e.g., `0x0000000000000000000000000000000000000002`), not `0xABCD...`.
4. Confirm: the same call on the real consensus node would return `PAYER_ACCOUNT_NOT_FOUND` rather than executing. [6](#0-5)

    private AccountID getSenderAccountID(final CallServiceParameters params) {
        // Set a default account to keep the sender parameter optional.
        if (params.getSender().isZero() && params.getValue() == 0L) {
            return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
        }
        final var senderAddress = params.getSender();
        final var accountIDNum = getSenderAccountIDAsNum(senderAddress);

        final var account = accountReadableKVState.get(accountIDNum);
        if (account == null) {
            throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
        } else if (account.smartContract()) {
            return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
        } else if (!account.hasKey() || account.key().equals(IMMUTABILITY_SENTINEL_KEY)) {
            // If the account is hollow, complete it in the state as a workaround
            // as this happens in HandleWorkflow in hedera-app but calling the
            // transaction executor directly skips this account completion and
            // this results in failed transactions that would otherwise succeed
            // against the consensus node.
            final var writableAccountCache =
                    ContractCallContext.get().getWriteCacheState(AccountReadableKVState.STATE_ID);
            final var completedAccount = account.copyBuilder().key(DEFAULT_KEY).build();
            writableAccountCache.put(account.accountId(), completedAccount);
        }

        return accountIDNum;
    }
