### Title
Gas Throttle Burst Capacity Allows Concurrent CPU-Exhausting EVM Executions Without Concurrency Limit

### Summary
The gas throttle in `ContractCallService` / `ThrottleManagerImpl` is a **rate limiter**, not a **concurrency limiter**. Because the bucket4j `gasLimitBucket` capacity equals its per-second refill rate, an unprivileged attacker can burst up to 500 concurrent max-gas requests simultaneously, all of which pass the throttle check before any EVM execution begins. All admitted requests then execute the EVM in parallel on server threads, saturating CPU with no mechanism to bound concurrent in-flight executions.

### Finding Description

**Code path:**

`ContractController.java:40` calls `throttleManager.throttle(request)` synchronously before any EVM work. Inside `ThrottleManagerImpl.throttle()` (lines 37–42), two `tryConsume` calls are made — both non-blocking:

```java
if (!rateLimitBucket.tryConsume(1)) { throw ... }
else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) { throw ... }
```

The `gasLimitBucket` is configured in `ThrottleConfiguration.java:34–44`:

```java
long gasLimit = throttleProperties.getGasPerSecond(); // scaleGas(7_500_000_000) = 750_000 tokens
final var limit = Bandwidth.builder()
        .capacity(gasLimit)          // capacity == refill rate
        .refillGreedy(gasLimit, Duration.ofSeconds(1))
        .build();
```

**Root cause — capacity equals refill rate:** Setting `capacity = refillGreedy rate` means the bucket starts full and allows a complete burst equal to the entire per-second allowance before any refill is needed. With `scaleGas(15_000_000) = 1_500` tokens per max-gas request and bucket capacity `750_000`, exactly **500 concurrent max-gas requests** can all pass `tryConsume` simultaneously before the bucket empties.

The `rateLimitBucket` has the same pattern: `capacity = requestsPerSecond = 500`, so 500 requests burst through simultaneously.

After passing the throttle, control flows to `ContractExecutionService.processCall()` → `callContract()` → `doProcessCall()` (lines 109–138), where `transactionExecutionService.execute(params, estimatedGas)` runs the EVM synchronously on the calling thread. Gas tokens are only restored in the `finally` block of `doProcessCall` (lines 127–135) via `restoreGasToBucket()` — **after** EVM execution completes. During the entire execution window, all 500 admitted threads are running EVM concurrently.

**Why the gas restore does not help:** For a tight-loop contract that consumes all gas (successful or out-of-gas), `restoreGasToBucket` (lines 140–151) restores either `gasLimitToRestoreBaseline` (on failure) or `Math.min(gasRemaining, gasLimitToRestoreBaseline)` (on success). For a loop that uses all gas, `gasRemaining ≈ 0`, so nearly zero tokens are restored to the bucket. The bucket refills at 750,000 tokens/second independently, but this does not prevent the concurrent burst already in flight.

**No concurrency cap exists anywhere in the path.** There is no semaphore, no `maxConcurrentRequests` setting, and no reactive back-pressure on EVM execution.

### Impact Explanation

An attacker sending 500 concurrent POST requests to `/api/v1/contracts/call` with `gas=15000000` and calldata encoding a tight loop (e.g., `JUMPDEST; JUMP` loop) causes all 500 to pass the throttle simultaneously and execute EVM in parallel. The Spring WebFlux bounded-elastic thread pool (default cap ~200 threads) becomes fully occupied with CPU-bound EVM work. Legitimate requests queue behind the saturated thread pool and either time out (after `requestTimeout=10s`) or receive no response. This constitutes a complete, non-network-based DoS of the web3 API endpoint with no authentication required.

### Likelihood Explanation

The attack requires no credentials, no on-chain state, and no special knowledge beyond the public API. The attacker needs only an HTTP client capable of sending ~200–500 concurrent requests. The tight-loop contract bytecode is trivial (`0x5b600056` — `JUMPDEST PUSH1 0x00 JUMP`). The attack is repeatable: after the first burst completes and gas is restored to the bucket (out-of-gas failures restore the full gas limit), the attacker can immediately launch another burst. This makes sustained DoS feasible with a single machine.

### Recommendation

1. **Add a concurrency limiter** (e.g., a `Semaphore` or `bulkhead` pattern) on EVM execution, separate from the rate limiter, to cap the number of simultaneously executing EVM calls (e.g., `maxConcurrentEvmExecutions = 2 * availableProcessors`).
2. **Decouple bucket capacity from refill rate**: set `capacity` to a fraction of `refillGreedy` (e.g., `capacity = gasPerSecond / 10`) so the burst window is bounded to 100ms of gas budget rather than a full second.
3. **Enforce a CPU-time deadline** on EVM execution (not just a wall-clock request timeout) so a single tight-loop call cannot hold a thread for the full 10-second `requestTimeout`.
4. Consider deploying an external rate limiter (e.g., nginx `limit_req`) in front of the service to enforce per-IP concurrency limits before requests reach the JVM.

# Then send 500 concurrent max-gas calls:

# - All 500 requests pass throttle simultaneously (bucket capacity allows it)
# - Server thread pool saturates with CPU-bound EVM executions

    public long scaleGas(long gas) {

            return 0L;

    protected final EvmTransactionResult doProcessCall(
            CallServiceParameters params, long estimatedGas, boolean estimate) throws MirrorEvmTransactionException {
        EvmTransactionResult result = null;
        var status = ResponseCodeEnum.SUCCESS.toString();

        try {
            result = transactionExecutionService.execute(params, estimatedGas);

            if (!estimate) {
                validateResult(result, params);
            }
        } catch (IllegalStateException | IllegalArgumentException e) {
            throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
        } catch (MirrorEvmTransactionException e) {
            // This result is needed in case of exception to be still able to call restoreGasToBucket method
            result = e.getResult();
            status = e.getMessage();
            throw e;
        } finally {
            if (!estimate) {
                restoreGasToBucket(result, params.getGas());

                // Only record metric if EVM is invoked and not inside estimate loop
                if (result != null) {
                    updateMetrics(params, result.gasUsed(), 1, status);
                }
            }
        }
        return result;
    }

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
