### Title
Sequential Blocking `getBlockRange()` Calls on All Inactive Nodes Enables Thread-Stall DoS via Malicious Block Node

### Summary
In `BlockNodeSubscriber.getNode()`, when all configured block nodes are inactive, the fallback loop calls `hasBlock()` — which internally calls `getBlockRange()`, a blocking gRPC call capped at `responseTimeout` (400 ms) — for every inactive node sequentially with no aggregate timeout budget. A block node operator (a network-participant role reachable without admin privileges) can deliberately cause their node(s) to become inactive via 3 subscribe failures, then delay `serverStatus` responses to near 400 ms, causing `getNode()` to block for N × 400 ms per invocation and stalling the 100 ms-scheduled block-processing loop well beyond 30%.

### Finding Description

**Exact code path:**

`BlockNodeSubscriber.getNode()` — `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNodeSubscriber.java`, lines 108–131:

```java
private BlockNode getNode(final AtomicLong nextBlockNumber) {
    final var nodeList = getBlockNodes();
    final var inactiveNodes = new ArrayList<BlockNode>();
    for (final var node : nodeList) {
        if (!node.tryReadmit(false).isActive()) {   // inactive → no getBlockRange() here
            inactiveNodes.add(node);
            continue;
        }
        if (hasBlock(nextBlockNumber, node)) {       // active → getBlockRange() called
            return node;
        }
    }

    // find the first inactive node with the block and force activating it
    for (final var node : inactiveNodes) {
        if (hasBlock(nextBlockNumber, node)) {       // ← getBlockRange() called for EVERY inactive node
            node.tryReadmit(true);
            return node;
        }
    }
    throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
}
```

`hasBlock()` (lines 133–145) unconditionally calls `node.getBlockRange()`.

`BlockNode.getBlockRange()` — `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, lines 99–112:

```java
public Range<Long> getBlockRange() {
    try {
        final var blockNodeService = BlockNodeServiceGrpc.newBlockingStub(channel)
                .withDeadlineAfter(streamProperties.getResponseTimeout()); // 400 ms per call
        final var response = blockNodeService.serverStatus(SERVER_STATUS_REQUEST);
        ...
    } catch (Exception ex) {
        return EMPTY_BLOCK_RANGE;   // timeout → empty → loop continues to next node
    }
}
```

`StreamProperties.responseTimeout` defaults to 400 ms (`importer/src/main/java/org/hiero/mirror/importer/downloader/block/StreamProperties.java`, line 35).

**Root cause:** There is no aggregate timeout budget across the inactive-node fallback loop. Each `getBlockRange()` call is individually capped at 400 ms, but the loop iterates all N inactive nodes sequentially. If all N nodes time out, total blocking = N × 400 ms.

**How nodes become inactive** — `BlockNode.onError()` (lines 196–207): after `maxSubscribeAttempts` (default 3) consecutive `streamBlocks()` failures, `active` is set to `false` and `readmitTime` is set to `now + readmitDelay` (default 1 minute). `tryReadmit(false)` (lines 184–190) will not re-activate the node until that time passes.

**Exploit flow:**
1. Attacker operates a registered block node (network-participant role, no admin keys required; auto-discovery reads from `registeredNodeRepository` — `BlockNodeDiscoveryService.java`, lines 79–87).
2. Attacker deliberately causes 3 consecutive `streamBlocks()` failures (e.g., by returning a malformed `SubscribeStreamResponse` or closing the connection) → node marked inactive for 1 minute.
3. During the inactive window, every `getNode()` call reaches the second loop and calls `getBlockRange()` on the attacker's node.
4. Attacker's `serverStatus` handler sleeps for ~395 ms before responding (or simply never responds, forcing the full 400 ms deadline to expire).
5. `getBlockRange()` returns `EMPTY_BLOCK_RANGE` after 400 ms; loop continues to the next inactive node.
6. With N=5 inactive nodes all timing out: `getNode()` blocks for 5 × 400 ms = 2,000 ms per invocation.
7. After `readmitDelay` (1 minute), attacker re-triggers 3 failures to reset the inactive state and repeat indefinitely.

### Impact Explanation

The `CompositeBlockSource.get()` is scheduled with `fixedDelay = 100 ms` (`importer/src/main/java/org/hiero/mirror/importer/downloader/block/CompositeBlockSource.java`, line 44). Under normal operation, `getNode()` completes in single-digit milliseconds. Under attack with N=5 nodes each timing out, `getNode()` blocks for 2,000 ms — a 20× increase in per-invocation latency, far exceeding the 30% threshold. Block ingestion stalls entirely during the blocking period, causing the mirror node to fall behind the chain. With a continuous attack (re-triggering failures every minute), block processing is effectively halted for the duration of the attack. This is a sustained service-degradation DoS against the mirror node's core function.

### Likelihood Explanation

A block node operator is a network-participant role. With `autoDiscoveryEnabled = true` (default), any node registered in the `registered_node` table as `BLOCK_NODE` type is auto-discovered and added to the pool. The attacker needs only to operate a reachable gRPC endpoint and register it as a block node — no admin keys, no leaked credentials, no internal network access. Causing 3 subscribe failures is trivial (return an error status or close the connection). Delaying `serverStatus` responses to ~395 ms is equally trivial. The attack is repeatable every `readmitDelay` (1 minute) and requires no special tooling beyond a standard gRPC server.

### Recommendation

1. **Add an aggregate timeout budget** for the entire inactive-node fallback loop. Track elapsed time across all `getBlockRange()` calls and abort the loop if the total exceeds a configurable cap (e.g., `N × responseTimeout` or a fixed `maxNodeSelectionTimeout`).
2. **Skip `getBlockRange()` for nodes whose `readmitTime` has not yet passed** in the fallback loop — only call it after `tryReadmit(true)` succeeds, not before. Currently `getBlockRange()` is called on still-inactive nodes to decide whether to force-readmit them; inverting this (force-readmit first, then check range) would eliminate the per-node blocking for nodes that will not be readmitted.
3. **Rate-limit or parallelize** the inactive-node status checks using a bounded async executor with a shared deadline, so N slow nodes do not multiply latency linearly.
4. **Cap the number of auto-discovered nodes** that can be added to the pool, or require operator approval for new block node registrations, to limit the attacker's ability to inflate N.

### Proof of Concept

**Preconditions:**
- Mirror node configured with `hiero.mirror.importer.block.autoDiscoveryEnabled=true` (default) and `hiero.mirror.importer.block.stream.maxSubscribeAttempts=3` (default), `responseTimeout=400ms` (default).
- Attacker registers 5 block node endpoints (e.g., `attacker-node-{1..5}:40840`) in the `registered_node` table as `BLOCK_NODE` type.

**Steps:**
1. Each attacker-controlled gRPC server accepts `SubscribeBlockStream` connections and immediately returns a `BlockStreamException`-triggering error status (e.g., `SubscribeStreamResponse.Code.READ_STREAM_INVALID_START_BLOCK_NUMBER`).
2. After 3 such failures per node, all 5 nodes are marked inactive (`active=false`, `readmitTime = now + 60s`).
3. The mirror node's next `getNode()` call finds 0 active nodes; the fallback loop begins.
4. Each attacker gRPC server's `serverStatus` handler sleeps for 395 ms before returning any response (or simply never responds, forcing the 400 ms deadline).
5. `getNode()` blocks for 5 × 400 ms = 2,000 ms.
6. The 100 ms-scheduled `CompositeBlockSource.get()` is delayed by 2,000 ms per cycle — a 2,000% latency increase, far exceeding the 30% threshold.
7. After 60 seconds, nodes are eligible for readmission; attacker immediately re-triggers 3 failures on each to reset the inactive state and repeat from step 3.

**Expected result:** Block ingestion stalls for 2 seconds per 100 ms cycle, effectively halting mirror node block processing for the duration of the attack.
