### Title
Negative Timeout in `BlockAssembler.timeout()` Enables Silent Block-Drop Griefing Without Error Penalty

### Summary
`BlockAssembler.timeout()` computes remaining budget as `timeout.toMillis() - stopwatch.elapsed(MILLISECONDS)` with no floor guard, so it can return a negative value. When passed to `grpcCall.read(negative, MILLISECONDS)`, the gRPC blocking call returns `null` immediately (Java's `LinkedBlockingQueue.poll` with non-positive timeout is a non-blocking poll), causing the streaming loop to exit as if the stream ended normally. Because the exit path is not a caught exception, `onError()` is never called, the error counter is never incremented, and the attacker's block node is never penalized.

### Finding Description

**Code locations:**

- `BlockNode.java` line 138 — the while-loop that drives streaming:
  ```java
  while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
  ```
- `BlockNode.java` lines 274–281 — `BlockAssembler.timeout()`:
  ```java
  long timeout() {
      if (!stopwatch.isRunning()) {
          stopwatch.start();
          return timeout.toMillis();
      }
      return timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS);
  }
  ```
- `BlockNode.java` lines 159–164 — `onError()` is only called inside catch blocks, never on a `null` return from `read()`.
- `BlockNode.java` lines 196–207 — `onError()` increments the error counter and enforces the readmit delay.
- `StreamProperties.java` line 35 — `responseTimeout = 400 ms` (used only for `getBlockRange()`).
- `CommonDownloaderProperties.java` line 83 — `timeout = 30 s` (used for `BlockAssembler`).

**Root cause:** `timeout()` performs an unbounded subtraction. The stopwatch starts on the first `timeout()` call and is only reset inside `onEndOfBlock()`. If the attacker delivers a `BLOCK_ITEMS/BLOCK_HEADER` response at time `T − ε` (just before the 30-second budget expires), the second call to `timeout()` at time `T − ε + δ` (where `δ` is processing overhead, easily > 1 ms) returns `30000 − (T − ε + δ)`, which is negative. No guard exists to clamp this to zero.

**Why existing checks fail:**
- The `@DurationMin(seconds = 1)` constraint on `CommonDownloaderProperties.timeout` only validates the configured value, not the computed remainder.
- The `@DurationMin(millis = 100)` on `StreamProperties.responseTimeout` is irrelevant here; `responseTimeout` is only used in `getBlockRange()`.
- The `catch (Exception ex) { onError(); }` block is never reached when `grpcCall.read()` returns `null`; it is only reached when `read()` throws (e.g., `TimeoutException` on a positive timeout expiry).

**Exploit flow:**
1. Attacker's block node responds to `serverStatus()` within 400 ms → passes `getBlockRange()`.
2. `streamBlocks()` is called; first `timeout()` starts the stopwatch and returns 30 000 ms.
3. `grpcCall.read(30000, MILLISECONDS)` blocks.
4. At `t = 29 999 ms`, attacker sends a single `BLOCK_ITEMS` message containing only a `BLOCK_HEADER` item.
5. `onBlockItemSet()` runs (adds to `pending`, sets `loadStart`); `errors.set(0)` resets the error counter.
6. Second `timeout()` call at `t ≈ 30 000 ms` returns `30000 − 30000 = 0` or a small negative value.
7. `grpcCall.read(≤0, MILLISECONDS)` — backed by `LinkedBlockingQueue.poll(≤0, unit)` — returns `null` immediately (non-blocking poll on an empty queue).
8. While-loop exits normally; no exception, no `onError()`, no error-counter increment.
9. The partial block (BLOCK_HEADER with no EndOfBlock) is silently discarded; `blockStreamConsumer` is never called.
10. `streamBlocks()` returns normally to `BlockNodeSubscriber.doGet()`, which reports no block delivered. The same block number is retried on the next scheduling cycle.
11. Repeat indefinitely: the attacker's node is never marked inactive, never subject to `readmitDelay`.

### Impact Explanation
The attacker can permanently stall block ingestion from a targeted mirror node without ever being penalized. Because `onError()` is bypassed, the `errors` counter stays at zero and the node is never deactivated. The mirror node will keep selecting the same malicious block node in round-robin, retrying the same block number forever (or until an operator intervenes). No economic damage occurs, but block processing halts, making the mirror node's data stale — a griefing denial-of-service against the mirror node operator.

### Likelihood Explanation
Any operator of a registered block node can execute this attack. The only technical requirement is the ability to delay a gRPC server-streaming response by a configurable amount (trivially done with `Thread.sleep` or equivalent). The 30-second window is large enough that millisecond-precision timing is not required; the attacker simply needs to send the `BLOCK_HEADER` response within the last few milliseconds of the window. The attack is repeatable on every polling cycle with no cooldown imposed on the attacker.

### Recommendation
1. **Clamp the return value of `timeout()` to a minimum of 0 (or 1) ms:**
   ```java
   return Math.max(1L, timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS));
   ```
   This ensures `grpcCall.read()` always receives a positive timeout and will throw `TimeoutException` when the budget is exhausted, routing through `onError()`.

2. **Treat a `null` return from `grpcCall.read()` as an error when a block is partially assembled.** After the while-loop, check `!assembler.pending.isEmpty()` and throw `BlockStreamException` if true, so `onError()` is always called for incomplete blocks.

3. **Alternatively, use a deadline-based approach** (absolute `Instant` rather than a shrinking remainder) so that clock skew and processing overhead cannot push the remaining budget below zero.

### Proof of Concept
```
1. Stand up a gRPC server implementing BlockStreamSubscribeService.
2. In subscribeBlockStream(), sleep for 29 900 ms, then send one SubscribeStreamResponse
   containing a BlockItemSet with a single BLOCK_HEADER item.
3. After sending, do NOT send EndOfBlock or any further message; hold the stream open.
4. Configure the mirror node to use this block node.
5. Observe: streamBlocks() returns without throwing; no block is delivered to onBlockStream;
   the error counter on the BlockNode remains 0; the node is not marked inactive.
6. On the next polling cycle the mirror node contacts the same node and the cycle repeats.
7. Confirm by adding a log line in BlockAssembler.timeout() printing the return value —
   it will print a value <= 0 on the second call.
