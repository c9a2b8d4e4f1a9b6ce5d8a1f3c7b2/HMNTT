### Title
Global Opcode Rate Bucket Monopolization via Predictable Greedy Refill Timing

### Summary
The `opcodeRateLimitBucket` is a single process-wide token bucket with no per-user or per-IP isolation. Its greedy refill strategy with a default capacity of 1 token/second makes refill timing fully deterministic. Any unauthenticated caller can continuously consume the sole available token immediately after each refill, starving all other users of debug/trace access indefinitely.

### Finding Description

**Code locations:**

- `ThrottleConfiguration.java` lines 47–55: the bucket is created with `capacity(rateLimit)` and `refillGreedy(rateLimit, Duration.ofSeconds(1))`, where `rateLimit` defaults to `opcodeRequestsPerSecond = 1` (ThrottleProperties.java line 29). No `SynchronizationStrategy` is set (unlike `gasLimitBucket`).
- `ThrottleManagerImpl.java` lines 52–56: `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` — no caller identity, no IP, no session, no authentication.
- `OpcodesController.java` line 61: `throttleManager.throttleOpcodeRequest()` is called before processing, with no authentication gate in front of it.

**Root cause:** The bucket is a single JVM-global object shared across all callers. There is no per-source-IP or per-user sub-bucket. The greedy refill strategy means tokens accumulate continuously and deterministically: with capacity=1 and refillGreedy(1, 1s), exactly 1 token becomes available exactly 1 second after the previous one was consumed. This timing is not secret — it is derivable from the HTTP 429 response itself (the caller knows when they were rejected and can compute when the next token arrives).

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/{id}/opcodes` with `Accept-Encoding: gzip` — consumes the 1 available token.
2. Attacker waits exactly 1000 ms (or polls with tryConsume semantics by observing 200 vs 429).
3. Attacker immediately sends the next request, consuming the freshly refilled token before any legitimate user can.
4. Repeat indefinitely in a tight loop with ~1 s sleep.

**Why existing checks fail:** The only guard is `opcodeRateLimitBucket.tryConsume(1)` — a global counter. There is no IP-level throttle, no authenticated identity check, and no fairness mechanism. The `rateLimitBucket` (500 req/s) is not checked for opcode requests — only `throttleOpcodeRequest()` is called, which hits only the opcode bucket.

### Impact Explanation
The opcode/debug/trace endpoint (`/api/v1/contracts/results/{id}/opcodes`) becomes exclusively owned by a single attacker. All other users receive HTTP 429 for every request. Developers and operators lose the ability to debug or trace contract executions for the duration of the attack. The attack requires no funds, no credentials, and no special network position — just a simple HTTP client with a 1-second timer.

### Likelihood Explanation
Trivially feasible. The attacker needs only:
- Public HTTP access to the mirror node web3 API (no authentication required).
- A script that sends one request per second with `Accept-Encoding: gzip`.
- No prior knowledge beyond the public API documentation.

The attack is fully repeatable, requires no setup cost, and can be sustained indefinitely from a single machine or even a browser. The default `opcodeRequestsPerSecond = 1` makes the monopolization especially easy since there is only ever one token to race for.

### Recommendation
1. **Per-IP or per-client rate limiting**: Maintain a `ConcurrentHashMap<String, Bucket>` keyed by client IP (or authenticated identity) inside `throttleOpcodeRequest()`, so each source gets its own sub-bucket rather than sharing one global bucket.
2. **Increase default capacity**: Even raising `opcodeRequestsPerSecond` to a higher value (e.g., 10) with per-IP limits reduces the monopolization window.
3. **Use `refillIntervally` instead of `refillGreedy`**: Interval refill adds tokens at fixed wall-clock boundaries rather than continuously, making burst-timing attacks harder (though not sufficient alone without per-IP isolation).
4. **Require authentication** for the opcode endpoint, since it is a resource-intensive debug facility.

### Proof of Concept

```bash
#!/bin/bash
# Monopolize the opcode endpoint — no credentials needed
TXID="0.0.1234-1234567890-000000000"
URL="http://<mirror-node>/api/v1/contracts/results/${TXID}/opcodes"

while true; do
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
    -H "Accept-Encoding: gzip" \
    "$URL")
  echo "$(date +%T) -> HTTP $STATUS"
  # On 200: token consumed; sleep just under 1s to race the refill
  # On 429: token not yet available; retry immediately
  if [ "$STATUS" = "200" ]; then
    sleep 0.95
  fi
done
```
