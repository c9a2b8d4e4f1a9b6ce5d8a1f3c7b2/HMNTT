### Title
Thread Pool Exhaustion via Unauthenticated Flood on `GET /api/v1/topics/{id}` During Database Degradation

### Summary
`TopicController.getTopic()` makes three sequential synchronous blocking database calls per request with no rate limiting, no circuit breaker, and no per-IP throttle. An unprivileged attacker sending concurrent requests during any database slowdown (including a network partition) can hold all Tomcat worker threads blocked on JDBC waits, rendering every endpoint of the rest-java service unresponsive until the partition resolves or the JVM is restarted.

### Finding Description
**Exact code path:**

`TopicController.getTopic()` (lines 32–37) executes three sequential, synchronous, blocking repository calls:
```java
var topic     = topicService.findById(id.id());   // DB call 1
var entity    = entityService.findById(id.id());  // DB call 2
var customFee = customFeeService.findById(id.id()); // DB call 3
``` [1](#0-0) 

Each call blocks the Tomcat worker thread until the database responds or the statement timeout fires. The configured `statementTimeout` for rest-java is **10 000 ms** (10 s): [2](#0-1) 

With three sequential calls, a single request can hold a thread for up to **30 seconds** under DB degradation.

**Root cause — no rate limiting in rest-java:**
A grep across the entire `rest-java/` source tree for `RateLimiter`, `bucket4j`, `throttle`, and `RequestRateLimiter` returns zero hits in production code. The REST Java API configuration table lists **no rate-limiting properties whatsoever**: [3](#0-2) 

This is in direct contrast to the `web3` module, which has a full `ThrottleConfiguration` using `bucket4j` with a `requestsPerSecond` cap of 500: [4](#0-3) 

**Thread pool:** Spring Boot's embedded Tomcat defaults to 200 worker threads. No override is present in rest-java configuration. With 200 concurrent requests each blocking for up to 30 s, the pool is fully occupied.

**HikariCP pool:** The rest-java module uses HikariCP (evidenced by the `hikaricp_connections_active` alert expression). No explicit `maximumPoolSize` is set in the rest-java config table, so HikariCP defaults to 10 connections. Requests beyond 10 concurrent DB users queue inside HikariCP, further extending thread hold time beyond the statement timeout. [5](#0-4) 

### Impact Explanation
When the thread pool is saturated, Spring/Tomcat queues incoming connections up to `acceptCount` (default 100) and then begins refusing new TCP connections. All endpoints — not just `/api/v1/topics/{id}` — become unreachable, including health probes. The Kubernetes readiness probe (`/actuator/health/readiness`, timeout 2 s) will fail, causing the pod to be removed from the load-balancer service, amplifying the outage to the entire replica set if the attack is sustained across pods. [6](#0-5) 

Severity: **High** — complete, unauthenticated denial of service against a public API with no authentication requirement.

### Likelihood Explanation
The endpoint requires no credentials. Any internet-accessible deployment is reachable. The attack requires only a standard HTTP load tool (`wrk`, `ab`, `hey`, or a simple shell loop). A network partition is not required to trigger thread exhaustion; any transient DB slowdown (high query latency, connection pool saturation from another source) is sufficient. The attack is repeatable and can be sustained indefinitely. The monitoring alert `RestJavaHighDBConnections` fires only after **5 minutes** at 75% utilization — far too slow to prevent the outage. [5](#0-4) 

### Recommendation
1. **Add rate limiting to rest-java** using the same `bucket4j`-based pattern already present in the `web3` module (`ThrottleConfiguration`). Apply a global RPS cap and a per-IP cap via a servlet filter or Spring MVC interceptor registered in `WebMvcConfiguration`.
2. **Parallelize the three DB calls** in `getTopic()` using `CompletableFuture` or Spring's `@Async` so a single request holds one thread for `max(t1, t2, t3)` instead of `t1 + t2 + t3`.
3. **Add a circuit breaker** (Resilience4j) around the repository calls so that when the DB is degraded, requests fail fast (503) instead of blocking threads.
4. **Tune Tomcat** `server.tomcat.max-threads` and `server.tomcat.accept-count` to values appropriate for the expected load, and set `server.tomcat.connection-timeout` to a value shorter than the statement timeout.
5. **Set HikariCP `connectionTimeout`** explicitly in rest-java to a value shorter than the statement timeout so threads fail fast when the pool is exhausted rather than waiting the full 10 s per call.

### Proof of Concept
```bash
# Simulate a DB partition by adding iptables DROP rules on the DB port,
# then flood the endpoint from an external host:

# Step 1 – on the DB host, simulate partition:
# iptables -A INPUT -p tcp --dport 5432 -j DROP

# Step 2 – from attacker host (no credentials needed):
wrk -t 20 -c 500 -d 60s \
  http://<mirror-node-rest-java-host>/api/v1/topics/0.0.1234

# Step 3 – observe that all other endpoints become unresponsive:
curl -v http://<mirror-node-rest-java-host>/api/v1/network/supply
# Expected: connection refused or timeout after thread pool exhaustion

# Step 4 – observe readiness probe failure in Kubernetes:
kubectl get endpoints <rest-java-service>
# Expected: pod removed from endpoints after readiness probe times out (2s)
```

The attack requires no authentication, no special knowledge of the system, and is executable with freely available tools.
