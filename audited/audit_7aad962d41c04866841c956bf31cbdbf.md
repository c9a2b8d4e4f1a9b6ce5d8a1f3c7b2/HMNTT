### Title
Unauthenticated Connection Pool Exhaustion via Parallel DB Queries in `getContractResultsByTimestamp`

### Summary
The public endpoint `GET /contracts/:contractId/results/:consensusTimestamp` triggers 6 sequential/parallel database queries per request with no rate limiting in the REST API layer. With a default connection pool of only 10 connections, a small number of concurrent unauthenticated requests can exhaust the pool, causing all subsequent REST API database operations to queue and time out, resulting in a full service-level DoS.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 18 registers the route with no authentication middleware: [1](#0-0) 

`getContractResultsByTimestamp()` in `rest/controllers/contractController.js` (lines 1002–1042) executes sequentially:

1. **Query 1** — `getInvolvedContractsByTimestampAndContractId()` (line 1011): one synchronous DB query against `contract_transaction`. [2](#0-1) 

2. **Queries 2–6** — `getDetailedContractResults()` (lines 1205–1220) fires **5 queries in parallel** via `Promise.all()`:
   - `getContractResultsByTimestamps` → `contract_result`
   - `getEthTransactionByTimestampAndPayerId` → `ethereum_transaction`
   - `getRecordFileBlockDetailsFromTimestamp` → `record_file`
   - `getContractLogsByTimestamps` → `contract_log`
   - `getContractStateChangesByTimestamps` → `contract_state_change` [3](#0-2) 

3. **Optional Query 7** — `FileDataService.getLatestFileDataContents()` (line 1023) fires if `ethTransaction.callDataId` is a valid file ID. [4](#0-3) 

**Root cause — connection pool size:**

`rest/dbpool.js` initializes the pool with `max: config.db.pool.maxConnections`: [5](#0-4) 

The documented default for `maxConnections` is **10**: [6](#0-5) 

**Math:** The `Promise.all()` in `getDetailedContractResults` holds 5 connections simultaneously per request. With a pool of 10, just **2 concurrent requests** consume all 10 connections. Any further requests queue and wait up to `connectionTimeout` (default 20,000 ms) before failing.

**No rate limiting in the REST API:** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` Java module — it is entirely absent from the Node.js REST service that serves this endpoint. [7](#0-6) 

### Impact Explanation
With a default pool of 10 connections and 5–6 connections consumed per request, an attacker sending ~10–20 concurrent requests continuously will exhaust the pool. All other REST API endpoints that require DB access (accounts, transactions, tokens, etc.) will also fail because they share the same pool. This is a full REST API denial of service, not limited to the contracts endpoint. The `statementTimeout` (20s) does not help — it limits query duration, not pool slot occupancy under concurrent load.

### Likelihood Explanation
No authentication, API key, or IP-based rate limiting protects this endpoint in the REST layer. The `contractId` and `consensusTimestamp` path parameters accept any valid-format values; a 404 is returned only after the first DB query completes, meaning even non-existent IDs consume one connection before short-circuiting. An attacker needs only a valid-format timestamp (e.g., `1234567890.000000000`) and any contract ID string. The attack is trivially scriptable with `curl` or any HTTP load tool and is fully repeatable.

### Recommendation
1. **Increase pool size** or deploy a connection pooler (PgBouncer) in front of the REST service with per-user connection limits — the Helm chart already configures `mirror_rest` with `max_user_connections: 250` for the Citus path, but the default application pool of 10 is the binding constraint. [8](#0-7) 
2. **Add request-level rate limiting** to the Node.js REST service (e.g., `express-rate-limit`) scoped to this and similar high-fan-out endpoints.
3. **Reduce query fan-out** by combining the 5 parallel queries into fewer queries or using a single CTE-based query where possible.
4. **Add a concurrency limiter** (e.g., `p-limit`) around `getDetailedContractResults` to cap simultaneous in-flight DB connections per endpoint.

### Proof of Concept
```bash
# Requires: any valid-format contractId and consensusTimestamp
# Even non-existent values consume 1 DB connection before 404

ENDPOINT="https://<mirror-node-host>/api/v1/contracts/0.0.1234/results/1234567890.000000000"

# Fire 20 concurrent requests in a tight loop
for i in $(seq 1 20); do
  curl -s "$ENDPOINT" &
done
wait

# Repeat in a loop to sustain pool exhaustion:
while true; do
  for i in $(seq 1 20); do curl -s "$ENDPOINT" & done
  wait
done
```

With a default pool of 10 and 5 connections held per request by `Promise.all`, the pool is exhausted after the first 2 concurrent requests reach `getDetailedContractResults`. All subsequent requests — including unrelated endpoints — will receive connection timeout errors (after 20s) or `503`-equivalent failures until the attack traffic stops.
