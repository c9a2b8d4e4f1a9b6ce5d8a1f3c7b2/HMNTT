### Title
Missing Entity Type Validation in `isValidAccount()` Allows Schedule Entity IDs to Bypass Account Check in `getTokenRelationships()`

### Summary
`EntityService.isValidAccount()` only checks whether an entity row exists in the database, but never validates that the entity's `type` column equals `ACCOUNT`. Because schedule entities are stored in the same `entity` table with `type = 'SCHEDULE'`, an unprivileged attacker can supply a valid schedule entity ID to `GET /api/v1/accounts/:id/tokens`, receive a `200 OK` response, and cause the mirror node to export token-relationship records attributed to a schedule entity â€” a semantically incorrect association that should never occur.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenController.js`, `getTokenRelationships()` (lines 66â€“92):

```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) {
  throw new NotFoundError();
}
``` [1](#0-0) 

`EntityService.isValidAccount()` in `rest/service/entityService.js` (lines 60â€“63):

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
``` [2](#0-1) 

The query used is:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;
``` [3](#0-2) 

**Root cause:** Although `entityExistenceQuery` selects the `type` column, `isValidAccount()` discards it entirely â€” it only checks `!isNil(entity)`. The returned row object (containing `type`) is never inspected for its value. The function name `isValidAccount` implies type validation, but the implementation only performs an existence check.

**Why schedule entities are affected:** Schedule entities are upserted into the shared `entity` table with `type = 'SCHEDULE'` by `ScheduleCreateTransactionHandler`:

```java
entity.setType(EntityType.SCHEDULE);
entityListener.onEntity(entity);
``` [4](#0-3) 

The DB schema confirms `SCHEDULE` is a valid `entity_type` enum value stored in the same `entity` table: [5](#0-4) 

The `constants.js` also defines `entityTypes.SCHEDULE = 'SCHEDULE'` as a known type in the REST layer, confirming the type is accessible: [6](#0-5) 

**Exploit flow:**
1. Attacker identifies a schedule entity ID (e.g., `0.0.500`) from the `/api/v1/schedules` endpoint (public, no auth).
2. Attacker sends `GET /api/v1/accounts/0.0.500/tokens`.
3. `getEncodedId("0.0.500")` resolves to the encoded numeric ID â€” no type check.
4. `isValidAccount(encodedId)` queries the `entity` table, finds the schedule row, returns `true`.
5. `TokenService.getTokenAccounts(query)` executes with the schedule's encoded ID as `ownerAccountId`.
6. Server responds `200 OK` with `{"tokens": [], "links": {"next": null}}` â€” a valid account-tokens response shape attributed to a schedule entity.

**Why existing checks fail:** The only guard is `isValidAccount()`, which is documented as checking "if provided accountId maps to a valid entity" â€” not "a valid *account* entity." The test suite confirms this: the `isValidAccount` tests only verify existence behavior, never type filtering: [7](#0-6) 

### Impact Explanation
The `/accounts/:id/tokens` endpoint is semantically scoped to account-type entities. Returning `200 OK` for a schedule entity causes the mirror node to export token-relationship records (even if empty) attributed to a non-account entity. Downstream consumers (indexers, wallets, analytics pipelines) that trust the mirror node's REST API may incorrectly classify schedule entities as accounts with token state. Additionally, the `200` vs `404` behavioral difference leaks entity-type information â€” an attacker can enumerate which numeric IDs are schedules vs. non-existent entities without any privileges.

### Likelihood Explanation
No authentication or special privileges are required. Schedule entity IDs are publicly discoverable via `/api/v1/schedules`. The exploit requires only a standard HTTP GET request with a known schedule ID as the path parameter. It is trivially repeatable and automatable.

### Recommendation
Fix `isValidAccount()` to validate the entity type, not just existence. The `type` column is already selected by `entityExistenceQuery` â€” use it:

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity) && (entity.type === 'ACCOUNT' || entity.type === 'CONTRACT');
}
```

Alternatively, modify `entityExistenceQuery` to add a `WHERE type IN ('ACCOUNT', 'CONTRACT')` clause, making the type constraint enforced at the DB level. The `Entity.TYPE` constant is already defined in `rest/model/entity.js` (line 33) and the valid type strings are defined in `rest/constants.js` `entityTypes`. [8](#0-7) 

### Proof of Concept
```
# Step 1: Discover a schedule entity ID
GET /api/v1/schedules
# Response includes schedule_id: "0.0.500"

# Step 2: Query token relationships using the schedule ID as an account
GET /api/v1/accounts/0.0.500/tokens

# Expected (correct): HTTP 404 Not Found
# Actual (vulnerable): HTTP 200 OK
# {
#   "tokens": [],
#   "links": { "next": null }
# }
```

The `200` response confirms `isValidAccount()` passed for a `SCHEDULE`-type entity, and the mirror node has exported a token-relationship record set for a non-account entity.
