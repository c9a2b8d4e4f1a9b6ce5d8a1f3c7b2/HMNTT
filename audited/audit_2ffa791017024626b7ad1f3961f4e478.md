### Title
Unbounded Concurrent Subscriptions in `PollingTopicMessageRetriever.retrieve()` Enable Unauthenticated Resource Exhaustion DoS

### Summary
`PollingTopicMessageRetriever.retrieve()` allocates a new `PollingContext` and a full Reactor Flux pipeline per call with no global subscription counter or admission check. The only per-connection cap (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening many TCP connections, since no global connection limit is configured on the Netty server. An unauthenticated attacker can accumulate thousands of live subscriptions, each polling the database every 2 seconds, exhausting JVM heap and the DB connection pool and causing a full service outage.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.retrieve()` at line 50 unconditionally allocates a `PollingContext`:

```java
// grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java, line 50
PollingContext context = new PollingContext(filter, throttled);
```

The `PollingContext` inner class (lines 82–147) holds `AtomicLong`, `AtomicReference`, a `Stopwatch`, and a `TopicMessageFilter`. The surrounding Flux pipeline (lines 51–62) adds Reactor operator chains, a `RepeatSpec` with `numRepeats = Long.MAX_VALUE` in throttled mode (line 99), and an infinite-retry policy:

```java
.retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
.timeout(retrieverProperties.getTimeout(), scheduler)   // 60 s idle timeout only
```

The only server-side admission control is in `GrpcConfiguration.java` line 33:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// default = 5
```

No `maxConnections` call is made on the `NettyServerBuilder`. The `subscriberCount` field in `TopicMessageServiceImpl` (line 48) is wired only to a Micrometer `Gauge` (lines 52–55) — it is read-only telemetry with zero enforcement logic.

**Root cause:** The design assumes that `maxConcurrentCallsPerConnection` is a sufficient global guard. It is not: it limits parallelism *per TCP connection* but places no ceiling on the number of connections, so the effective global subscription cap is `maxConcurrentCallsPerConnection × (number of open connections)` — unbounded.

**Exploit flow:**

1. Attacker opens *C* TCP connections to port 5600 (no server-side connection limit).
2. On each connection, issues 5 concurrent `subscribeTopic` RPCs targeting a topic with a steady message stream (so the 60 s idle timeout never fires).
3. Each RPC reaches `TopicMessageServiceImpl.subscribeTopic()` → `topicMessageRetriever.retrieve(filter, true)` → `PollingTopicMessageRetriever.retrieve()`.
4. Each call allocates a `PollingContext` + Flux pipeline and schedules a repeating DB poll every 2 s (`pollingFrequency` default).
5. With *C* = 2 000 connections: 10 000 live subscriptions, each issuing a DB query returning up to 1 000 `TopicMessage` objects every 2 s.

**Why checks fail:**

| Check | Why insufficient |
|---|---|
| `maxConcurrentCallsPerConnection = 5` | Per-connection only; no `maxConnections` set |
| `subscriberCount` in `TopicMessageServiceImpl` | Gauge metric only, no rejection path |
| `timeout = 60 s` | Idle-emission timeout; never fires on an active topic |
| `retryWhen(Long.MAX_VALUE, ...)` | Keeps subscriptions alive indefinitely on transient DB errors, compounding load |
| Circuit breaker (Traefik middleware) | Triggers only after errors accumulate; does not prevent subscription creation |

### Impact Explanation

With 10 000 concurrent subscriptions each buffering up to 1 000 `TopicMessage` objects (~500 bytes each) per poll cycle, the in-flight heap demand reaches ~5 GB/cycle before GC can reclaim it. The `boundedElastic()` scheduler (shared singleton, line 41) queues tasks without bound when threads are saturated, growing its internal queue until `OutOfMemoryError`. Simultaneously, the DB connection pool is saturated by 5 000 queries/second, starving the importer and REST API of connections. The result is a full gRPC service crash and cascading degradation of dependent services.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC port (5600) is publicly exposed. Opening thousands of TCP connections is trivial from a single host with a modest script (e.g., `grpcurl` in a loop, or any gRPC client library). The attacker does not need to know any secret, hold any account, or submit any transaction. The attack is repeatable and can be sustained indefinitely by reconnecting after the 60 s idle timeout.

### Recommendation

1. **Add a global subscription counter with enforcement** in `PollingTopicMessageRetriever.retrieve()` (or at the `TopicMessageServiceImpl` layer where `subscriberCount` already exists): reject new subscriptions when the counter exceeds a configurable `maxSubscriptions` threshold, returning `RESOURCE_EXHAUSTED` status.
2. **Set a global connection limit** on the Netty server builder in `GrpcConfiguration`:
   ```java
   serverBuilder.maxConnectionAge(Duration.ofMinutes(10), TimeUnit.MILLISECONDS);
   serverBuilder.maxConnectionIdle(Duration.ofSeconds(60), TimeUnit.MILLISECONDS);
   // and consider a custom connection limiter
   ```
3. **Add a `maxSubscriptions` property** to `RetrieverProperties` (or `GrpcProperties`) and enforce it atomically before allocating `PollingContext`.
4. **Apply IP-based rate limiting** at the ingress/proxy layer for `subscribeTopic` RPCs.

### Proof of Concept

```bash
# Requires grpcurl and a running mirror-node gRPC service at localhost:5600
# with a topic (e.g., 0.0.41110) that receives continuous messages.

for i in $(seq 1 2000); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    > /dev/null 2>&1 &
done

# Each background process holds one TCP connection with one active subscription.
# Monitor JVM heap via JMX or /actuator/metrics; observe OOM or DB pool exhaustion
# within minutes as the 2-second polling cycles accumulate.
# To reach maxConcurrentCallsPerConnection=5 per connection, use a client that
# multiplexes 5 streams per HTTP/2 connection (standard gRPC behavior).
```
