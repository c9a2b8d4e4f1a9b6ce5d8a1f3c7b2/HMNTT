### Title
Unauthenticated GraphQL Complexity-Ceiling DoS: No Rate Limiting Allows CPU Saturation via Maximum-Complexity Queries

### Summary
The `graphQlCustomizer()` method configures `MaxQueryComplexityInstrumentation(200)`, which rejects queries with complexity **above** 200 but explicitly allows queries at exactly 200 — the maximum permitted work per request. The GraphQL module contains no application-level rate limiting (unlike the `web3` module which uses `ThrottleManager`/Bucket4j). Any unauthenticated external user can craft queries at the complexity ceiling and flood the service concurrently, driving CPU saturation across processing nodes.

### Finding Description

**Exact code location:**
`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, `graphQlCustomizer()`, lines 42–48:

```java
var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
var instrumentation = new ChainedInstrumentation(maxQueryComplexity, maxQueryDepth);
```

**Root cause — failed assumption:** The design assumes the complexity ceiling (200) is low enough to prevent abuse. It is not, because there is no concurrent-request limit or per-IP rate limit at the application layer. The complexity check is a *per-query* gate, not a *per-second throughput* gate.

**Schema amplification surface:** The schema (`account.graphqls`, `common.graphqls`) exposes recursive `Account` references:
- `Account.autoRenewAccount: Account`
- `Account.stakedAccount: Account`

With `MaxQueryDepthInstrumentation(10)`, an attacker can nest 10 levels of `Account` resolution. Each level triggers a separate `EntityRepository` database call. A single complexity-200 query can therefore cause O(10) chained DB lookups plus scalar field resolution across all levels.

**Preparsed cache does not help:** `CachedPreparsedDocumentProvider` caches the *parsed/validated document* keyed by query string. Parsing and complexity validation are skipped on cache hits, but **execution (resolver calls, DB queries) runs on every request**. Sending the same complexity-200 query string repeatedly bypasses parsing overhead while still executing all resolvers.

**No application-level rate limiting in the GraphQL module:** The `web3` module has `ThrottleManagerImpl`, `ThrottleConfiguration` (Bucket4j), and `ThrottleProperties`. The `graphql` module has none of these. There is no `requestsPerSecond` guard, no per-IP token bucket, and no concurrent-request cap in the application code.

**Exploit flow:**
1. Attacker crafts a GraphQL query with complexity exactly 200 using nested `autoRenewAccount` chains and many scalar fields.
2. Attacker sends N concurrent HTTP POST requests to `/graphql` (no authentication required — no `SecurityConfiguration` exists in the graphql module).
3. Each request passes the complexity check (200 ≤ 200), passes depth check, and is executed.
4. Each request triggers multiple DB round-trips and CPU-bound field resolution.
5. With sufficient N (e.g., hundreds of concurrent connections from a single machine or botnet), JVM thread pools and DB connection pools saturate.

### Impact Explanation
CPU and DB connection pool exhaustion on the GraphQL processing nodes. Because there is no per-IP or global request-rate cap, a single attacker with a moderate number of concurrent connections can monopolize all available worker threads. If the deployment runs multiple replicas, the attacker scales the attack proportionally. The severity matches the stated scope: ≥30% of processing nodes can be rendered unresponsive without brute-force volume — a moderate number of sustained concurrent connections suffices because each request performs maximum allowed work.

### Likelihood Explanation
- **No authentication required:** The GraphQL endpoint is publicly accessible; zero credentials needed.
- **No rate limiting at application layer:** Unlike `web3`, the `graphql` module has no Bucket4j throttle.
- **Trivial to craft:** The attacker only needs to construct a valid GraphQL query that scores exactly 200 in graphql-java's default field-count complexity model.
- **Repeatable and automatable:** A simple script with an HTTP client library suffices. The preparsed cache actually *helps* the attacker by reducing server-side parsing overhead, making each attack request cheaper to process at the parse stage while still executing all resolvers.
- **No forensic barrier:** HTTP requests leave logs but no blocking mechanism exists to stop the flood in real time.

### Recommendation
1. **Add application-level rate limiting to the GraphQL module** — mirror the `web3` module's `ThrottleConfiguration`/`ThrottleManagerImpl` pattern using Bucket4j, enforcing a `requestsPerSecond` cap per IP and globally.
2. **Reduce the complexity ceiling** — 200 is high for a schema with only one query root (`account`) and recursive types. Audit the maximum legitimate complexity needed and lower the limit accordingly.
3. **Add a concurrent-request cap** — configure Spring's embedded Tomcat/Jetty `maxThreads` and a Semaphore-based guard to bound simultaneous GraphQL executions.
4. **Consider infrastructure-level rate limiting** — the Helm chart for `hedera-mirror-graphql` has a `middleware.yaml` template; ensure `rateLimit` and `inFlightReq` Traefik middlewares are configured (as done for the Rosetta chart).

### Proof of Concept

```python
import concurrent.futures
import requests

# Complexity-200 query: 10-level nested autoRenewAccount (each level ~20 scalar fields)
QUERY = """
query {
  account(input: {entityId: {num: 1}}) {
    alias balance createdTimestamp deleted entityId { shard realm num }
    expirationTimestamp id memo type
    autoRenewAccount {
      alias balance createdTimestamp deleted entityId { shard realm num }
      expirationTimestamp id memo type
      autoRenewAccount {
        alias balance createdTimestamp deleted entityId { shard realm num }
        expirationTimestamp id memo type
        autoRenewAccount {
          alias balance createdTimestamp deleted entityId { shard realm num }
          expirationTimestamp id memo type
        }
      }
    }
  }
}
"""

URL = "http://<graphql-host>/graphql"
CONCURRENT = 200  # adjust to available connections

def send():
    return requests.post(URL, json={"query": QUERY}, timeout=30)

with concurrent.futures.ThreadPoolExecutor(max_workers=CONCURRENT) as ex:
    futures = [ex.submit(send) for _ in range(1000)]
    for f in concurrent.futures.as_completed(futures):
        print(f.result().status_code)
```

**Expected result:** Server CPU climbs to saturation; response times degrade or connections time out. No 429 or blocking response is returned because no rate limiting exists in the application.
