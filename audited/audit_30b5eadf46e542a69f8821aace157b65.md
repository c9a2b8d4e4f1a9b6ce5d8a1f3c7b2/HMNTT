### Title
`validateAcceptEncodingHeader()` Bypassed via `Accept-Encoding: gzip;q=0` â€” Gzip Requirement Circumvented on Expensive EVM Replay Endpoint

### Summary
The `validateAcceptEncodingHeader()` method in `OpcodesController.java` uses a naive `.contains("gzip")` substring check to enforce that clients accept gzip compression before triggering an expensive EVM transaction replay. An unprivileged attacker can send `Accept-Encoding: gzip;q=0`, which contains the substring `"gzip"` and passes the check, while simultaneously signaling to the HTTP stack that gzip is explicitly **not acceptable** (RFC 7231 Â§5.3.4, q=0 means "not acceptable"). This causes the server to perform the full, costly EVM replay and then emit a large uncompressed response â€” exactly the resource-waste scenario the check was designed to prevent.

### Finding Description
**Exact location:**
- File: `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`
- Function: `validateAcceptEncodingHeader()`, line 76

**Root cause:**

```java
// Line 76
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
```

The check only tests whether the literal string `"gzip"` appears anywhere in the header value. It does not parse quality factors (`q=`). Per RFC 7231, `Accept-Encoding: gzip;q=0` means the client explicitly refuses gzip. The string `"gzip;q=0"` still contains `"gzip"` as a substring, so `.contains("gzip")` returns `true` and no exception is thrown.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/contracts/results/{txHash}/opcodes?stack=true&memory=true&storage=true` with header `Accept-Encoding: gzip;q=0`.
2. `validateAcceptEncodingHeader("gzip;q=0")` evaluates `"gzip;q=0".toLowerCase().contains("gzip")` â†’ `true` â†’ check passes, no 406 thrown.
3. `throttleManager.throttleOpcodeRequest()` is called (rate-limited, but not blocked).
4. `opcodeService.processOpcodeCall(request)` triggers a full EVM replay â€” the most expensive operation on this endpoint.
5. Spring/Tomcat's compression layer performs proper RFC-compliant content negotiation: it sees `q=0` and does **not** compress the response.
6. A large, uncompressed opcode trace is sent over the wire â€” the exact outcome the gzip gate was designed to prevent.

**Why existing checks fail:**
- The test suite at line 507 covers `{"deflate", "identity", "br", ""}` but does **not** include `"gzip;q=0"`, leaving this bypass untested and undetected.
- The throttle manager limits request rate but does not prevent the bypass itself; an attacker operating within the rate limit can still repeatedly trigger expensive uncompressed responses. [1](#0-0) [2](#0-1) 

### Impact Explanation
The opcodes endpoint is explicitly documented as potentially taking "a significant amount of time" due to EVM re-execution. The gzip gate exists solely to prevent large uncompressed payloads from being served. Bypassing it means:
- The server performs a full, CPU/memory-intensive EVM replay.
- The response (which can be megabytes of opcode trace data) is transmitted uncompressed, consuming significant server-side I/O and network bandwidth.
- Under the rate limit, an attacker can sustain a stream of expensive, uncompressed responses, degrading throughput for legitimate users. [3](#0-2) 

### Likelihood Explanation
No authentication or special privilege is required. The bypass requires only crafting a single HTTP header value (`gzip;q=0`), which is trivially done with any HTTP client (`curl`, Python `requests`, etc.). The technique is well-known to anyone familiar with HTTP content negotiation. The rate limiter reduces but does not eliminate impact, since each allowed request still triggers the full expensive path. [4](#0-3) 

### Recommendation
Replace the substring check with proper RFC-compliant quality-factor parsing. Use Spring's built-in `org.springframework.http.MediaType` / `ContentNegotiationStrategy`, or the `com.github.mizosoft.methanol` / Apache HttpComponents `Accept-Encoding` parser, to verify that gzip has a quality factor `> 0`:

```java
// Correct approach: parse quality factors
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw notAcceptable();
    }
    // Split on comma, parse each token for q-value
    boolean gzipAccepted = Arrays.stream(acceptEncodingHeader.split(","))
        .map(String::trim)
        .anyMatch(token -> {
            String[] parts = token.split(";");
            if (!parts[0].trim().equalsIgnoreCase("gzip")) return false;
            for (int i = 1; i < parts.length; i++) {
                String p = parts[i].trim();
                if (p.startsWith("q=")) {
                    try { return Double.parseDouble(p.substring(2)) > 0; }
                    catch (NumberFormatException e) { return false; }
                }
            }
            return true; // no q= means q=1 (fully acceptable)
        });
    if (!gzipAccepted) throw notAcceptable();
}
```

Also add `"gzip;q=0"` and `"gzip;q=0,deflate"` to the `@ValueSource` in `rejectsWhenAcceptEncodingIsNotGzip` to prevent regression. [1](#0-0) 

### Proof of Concept
```bash
# Bypass the gzip gate with q=0 (explicitly rejecting gzip)
curl -v \
  -H "Accept-Encoding: gzip;q=0" \
  "https://<mirror-node-host>/api/v1/contracts/results/<tx-hash>/opcodes?stack=true&memory=true&storage=true"

# Expected (correct) behavior: HTTP 406 Not Acceptable
# Actual behavior:             HTTP 200 OK with large UNCOMPRESSED body
```

Alternatively, with `gzip;q=0,deflate` (gzip rejected, deflate offered):
```bash
curl -v \
  -H "Accept-Encoding: gzip;q=0,deflate" \
  "https://<mirror-node-host>/api/v1/contracts/results/<tx-hash>/opcodes"
# Also returns 200 OK, bypassing the gate, with uncompressed response
```
