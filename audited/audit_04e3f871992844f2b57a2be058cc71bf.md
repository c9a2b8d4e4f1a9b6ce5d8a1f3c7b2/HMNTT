### Title
Unbounded INFO Log Flooding via Unauthenticated Mass Subscription Completion in `PollingTopicMessageRetriever.onComplete()`

### Summary
The `onComplete()` method in `PollingTopicMessageRetriever` emits an unconditional `log.info(...)` call on every subscription termination (both cancel and complete signals) with no rate limiting or global connection cap enforced at the server level. An unauthenticated attacker can open an unbounded number of gRPC connections, each carrying up to 5 concurrent `subscribeTopic` calls, and rapidly cycle them through completion to flood the INFO log. During network partition recovery, all accumulated subscriptions complete simultaneously, producing a synchronous burst that can fill disk storage and suppress legitimate error-level log entries from being recorded.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java` lines 60–61 register `onComplete()` on both cancel and complete signals:
```java
.doOnCancel(context::onComplete)
.doOnComplete(context::onComplete)
```

`PollingContext.onComplete()` at lines 137–146 calls `log.info(...)` unconditionally:
```java
void onComplete() {
    var elapsed = stopwatch.elapsed(TimeUnit.MILLISECONDS);
    var rate = elapsed > 0 ? (int) (1000.0 * total.get() / elapsed) : 0;
    log.info(
            "[{}] Finished retrieving {} messages in {} ({}/s)",
            filter.getSubscriberId(),
            total,
            stopwatch,
            rate);
}
```

**Root cause — failed assumption:** The design assumes the number of concurrent subscriptions is bounded by infrastructure controls. In practice, `GrpcConfiguration.java` (lines 28–35) only configures `maxConcurrentCallsPerConnection` (default 5); it sets **no** `maxConnectionAge`, no total connection count limit, and no global subscriber cap. The `subscriberCount` field in `TopicMessageServiceImpl` (line 48) is a Micrometer gauge only — it is never checked against a ceiling to reject new subscriptions.

**Amplification via `missingMessages()`:** `TopicMessageServiceImpl.missingMessages()` (lines 138–177) calls `topicMessageRetriever.retrieve(gapFilter, false)` (unthrottled) for every detected gap. Each such call creates a new independent `PollingContext` with its own `onComplete()` log emission. A single top-level subscription can therefore produce multiple INFO log lines.

**Network partition burst:** The retriever is configured with `Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))` (line 58). During a partition, all open subscriptions stall in retry loops. When connectivity is restored, every stalled `PollingContext` completes simultaneously, producing a synchronous burst of `log.info` calls proportional to the total number of open subscriptions across all connections.

**Existing checks reviewed and shown insufficient:**
- `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14, GrpcConfiguration line 33): limits calls *per connection*, not total connections. An attacker opens N connections to get N×5 concurrent subscriptions.
- `maxRatePerEndpoint: 250` (values.yaml line 69): applies only to GCP gateway deployments; absent in direct-access or non-GCP deployments. It is also a request-initiation rate limit, not a connection count limit.
- No log-level rate limiting (e.g., Logback's `DuplicateMessageFilter` or `ThresholdFilter`) is present in the grpc module configuration.

### Impact Explanation
Continuous INFO log flooding can fill the filesystem partition hosting log files. Once the partition is full: (1) the JVM may fail to write subsequent log entries including ERROR-level messages from legitimate failures; (2) the `GrpcLogErrors` Grafana alert (rules.tf lines 187–213) monitors `logback_events_total{level="error"}` — flooding INFO does not trigger this alert, so operators receive no automated warning while disk fills; (3) application stability may degrade if the logging framework blocks on a full disk. The attack requires no credentials and no knowledge beyond a valid topic ID (public on the Hedera network).

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. A valid topic ID is obtainable from the public REST API. An attacker needs only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or a raw HTTP/2 client) and the ability to open many TCP connections. Subscribing to a topic with `startTime=0` on a topic with no messages causes immediate completion and immediate `onComplete()` logging. The attack is fully scriptable, repeatable, and requires no privileged access. The partition-recovery burst scenario requires maintaining open connections during a partition event, which is achievable by any persistent attacker.

### Recommendation
1. **Enforce a global connection/subscriber ceiling:** In `GrpcConfiguration`, add `serverBuilder.maxConnectionAge(...)` and/or a global active-subscriber guard in `TopicMessageServiceImpl.subscribeTopic()` that rejects new subscriptions when `subscriberCount` exceeds a configurable threshold.
2. **Downgrade or rate-limit the `onComplete()` log:** Change `log.info(...)` in `PollingContext.onComplete()` to `log.debug(...)`, or apply a Logback `DuplicateMessageFilter` / token-bucket rate limiter so that repeated completions within a short window are suppressed.
3. **Add per-IP connection limiting** at the load-balancer or Netty level to prevent a single source from opening unbounded connections.
4. **Cap unthrottled `retrieve()` invocations** per top-level subscription to prevent the `missingMessages()` path from multiplying log output.

### Proof of Concept
```
# Prerequisites: grpcurl installed, mirror node reachable at TARGET, TOPIC_ID is any valid topic

# Step 1: Open 200 parallel connections, each with 5 concurrent subscribeTopic calls
# (1000 total PollingContexts), all subscribing from startTime=0 on an empty topic
# so each completes immediately and calls onComplete() → 1000 INFO log lines per cycle

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{
      "topicID": {"topicNum": TOPIC_ID},
      "consensusStartTime": {"seconds": 0}
    }' TARGET:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait

# Step 2: Repeat in a tight loop to continuously generate log lines
# Each cycle produces ≥1000 INFO entries in PollingTopicMessageRetriever

# Step 3 (partition burst): Keep connections open during a simulated DB outage.
# On DB recovery, all 1000 PollingContexts complete simultaneously,
# producing a synchronous burst of 1000+ log.info() calls.
```
