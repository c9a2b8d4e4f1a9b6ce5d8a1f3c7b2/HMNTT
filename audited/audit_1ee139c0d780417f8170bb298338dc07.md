### Title
Unbounded Label Cardinality via Attacker-Controlled `req.path` in `toOpenApiPath()` Metrics Middleware

### Summary
In `rest/middleware/metricsHandler.js`, the `toOpenApiPath()` function falls back to the raw, attacker-controlled `req.path` when a request does not match any registered Express route (`req.route` is `undefined`). This raw path is used directly as the `path` label value in four OpenTelemetry instruments. An unprivileged external attacker can send requests to an unbounded number of unique paths, causing label cardinality explosion that degrades or disables the metrics subsystem.

### Finding Description

**Exact code path:**

In `toOpenApiPath()` ( [1](#0-0) ), when `res.locals[requestPathLabel]` is not set and `req.route` is `undefined` (i.e., the request matched no registered Express route), the function assigns `path = req.path` — the raw, unparameterized, attacker-supplied URL path — with no normalization, truncation, or allowlist check.

The `recordRequestPath` middleware in `rest/routes/index.js` only populates `res.locals[requestPathLabel]` for matched routes: [2](#0-1) . For unmatched routes it is never set, so `toOpenApiPath()` always falls through to `req.path`.

In the `res.on('finish', ...)` handler, this raw path is assembled into a label set together with `req.method` and `res.statusCode`, then recorded against four instruments: [3](#0-2) 

```
const path   = toOpenApiPath(req, res);   // ← attacker-controlled for 404s
const code   = String(res.statusCode);
const method = req.method;
const labels = {method, path, code};

requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
```

**Root cause / failed assumption:** The code assumes the `path` label will be drawn from the finite set of registered Express route patterns. For unmatched routes this assumption is false; the label value is whatever the attacker placed in the URL.

**Why existing checks fail:** There is no path length cap, no allowlist, no "unknown" sentinel substitution, and no OpenTelemetry cardinality limit configured. The `@opentelemetry/sdk-metrics` default cardinality limit is 2 000 attribute sets per instrument; with four instruments running in parallel, the effective budget is consumed four times as fast, and even approaching the limit causes linear scan overhead on every `add()`/`record()` call.

### Impact Explanation

Each unique `{method, path, code}` triple creates a new time-series entry inside the SDK's in-process attribute-set registry. Flooding with N unique paths × M HTTP methods produces up to N×M new entries per instrument per scrape interval. Consequences:

- **CPU/memory growth** inside the Node.js process proportional to cardinality, degrading API response latency.
- **Metrics data loss** once the SDK's cardinality cap is hit; legitimate route metrics are silently dropped.
- **Prometheus scrape bloat**: the `/metrics` endpoint serializes every live time series; a large cardinality causes scrape timeouts and monitoring blind spots.
- No economic damage to network users, but the observability plane is rendered unreliable — matching the stated "griefing" severity.

### Likelihood Explanation

- **No authentication required.** Any internet-reachable instance is exploitable.
- **Trivially automatable.** A simple loop generating random path segments (e.g., `GET /api/v1/<uuid>`) is sufficient.
- **Persistent effect.** OpenTelemetry attribute sets are retained for the lifetime of the process; a one-time burst of ~2 000 unique paths saturates the default cap permanently until restart.
- **Method multiplier is real.** Node.js's HTTP server and Express middleware accept all standard methods (GET, POST, PUT, DELETE, PATCH, OPTIONS, HEAD, TRACE); the `metricsMiddleware` is registered unconditionally and fires for every method, so each path can be replayed with 9 methods for 9× the cardinality at no extra cost.

### Recommendation

1. **Substitute a sentinel for unmatched routes.** In `toOpenApiPath()`, replace the `path = req.path` fallback with a fixed string such as `"unknown"` or `"/api/v1/unknown"`:

   ```js
   if (!req.route) {
     path = 'unknown';   // ← fixed sentinel, not req.path
   }
   ```

2. **Normalize the HTTP method label** to an allowlist (`GET`, `POST`, `PUT`, `DELETE`, `PATCH`, `OPTIONS`, `HEAD`; map anything else to `OTHER`) to prevent method-dimension cardinality abuse.

3. **Set an explicit OpenTelemetry cardinality limit** via `MeterProviderOptions` to cap worst-case memory growth even if other mitigations are bypassed.

### Proof of Concept

```bash
# Bash one-liner: send 5000 requests to unique paths using multiple methods
METHODS=(GET POST PUT DELETE PATCH OPTIONS HEAD)
for i in $(seq 1 5000); do
  METHOD=${METHODS[$((i % ${#METHODS[@]}))]}
  curl -s -o /dev/null -X "$METHOD" \
    "http://<mirror-node-host>/api/v1/griefing-$(uuidgen)"
done

# Then scrape metrics and observe cardinality explosion:
curl http://<mirror-node-host>/swagger/metrics/ | grep api_request_total | wc -l
# Expected: thousands of unique {method, path, code} label combinations
```

After the burst, every subsequent `requestTotalCounter.add()` call performs a linear scan over thousands of stored attribute sets, measurably increasing per-request latency and eventually causing the SDK to silently drop new label combinations once the 2 000-entry cap is reached.
