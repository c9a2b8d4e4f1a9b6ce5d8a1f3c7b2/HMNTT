### Title
Unauthenticated DoS via Unbounded DB Query Amplification on `/account/balance` Due to Disabled Rate Limiting by Default

### Summary
The `AccountBalance()` handler in `rosetta/app/services/account_service.go` accepts unauthenticated POST requests and triggers 4–5 sequential database queries per call via `RetrieveBalanceAtBlock()`. No application-level rate limiting exists, and the Traefik middleware chain (which includes `rateLimit` and `inFlightReq`) is disabled by default because `global.middleware: false` in the Helm chart values. An unprivileged attacker can flood this endpoint with attacker-controlled block identifiers, exhausting the DB connection pool and degrading or blocking gossip transaction ingestion by the mirror node importer.

### Finding Description

**Exact code path:**

`AccountBalance()` at `rosetta/app/services/account_service.go` lines 41–79 accepts any POST to `/account/balance` with no authentication or rate check. When `request.BlockIdentifier != nil` (line 52), it calls `RetrieveBlock()` (one DB query), then unconditionally calls `a.accountRepo.RetrieveBalanceAtBlock(ctx, accountId, block.ConsensusEndNanos)` at line 61.

`RetrieveBalanceAtBlock()` at `rosetta/app/persistence/account.go` lines 161–222 issues:
1. `getCryptoEntity()` — queries `entity` + `entity_history` (UNION ALL with range operator `@>`)
2. `getLatestBalanceSnapshot()` — first queries `mirror_node_time_partitions` (`selectPreviousPartitionLowerBound`), then queries `account_balance` with a correlated subquery (`latestBalanceBeforeConsensus`)
3. `getBalanceChange()` — queries `crypto_transfer` with a `SUM(amount)` aggregation (`balanceChangeBetween`)

Total: **4–5 DB queries per request**, each with range scans on large partitioned tables.

**Root cause — rate limiting disabled by default:**

The Traefik middleware CRD is gated at `charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3:
```
{{ if and .Values.global.middleware .Values.middleware -}}
```
`global.middleware` defaults to `false` in `charts/hedera-mirror-rosetta/values.yaml` line 95. The middleware CRD (which includes `rateLimit: average: 10` and `inFlightReq: amount: 5`) is therefore **never created** in a default deployment.

The application-level middleware stack in `rosetta/main.go` lines 217–219 applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no rate limiting. The `rosetta/app/middleware/` directory contains only `health.go`, `metrics.go`, and `trace.go`.

The DB connection pool in `rosetta/app/db/db.go` lines 31–33 sets `SetMaxOpenConns()`, which limits concurrent DB connections but does not limit the rate of incoming HTTP requests. Requests queue in the Go HTTP server goroutine pool, and each goroutine holds a DB connection for the duration of all 4–5 queries.

**Why existing checks fail:**

The ingress annotation at `charts/hedera-mirror-rosetta/values.yaml` line 121 always references the Traefik middleware by name, but since `global.middleware: false`, the CRD object is never created. Traefik silently drops the middleware reference and routes traffic without it, meaning the `rateLimit` and `inFlightReq` rules are never enforced.

### Impact Explanation

An attacker sending thousands of requests/second with varying `block_identifier.index` values forces unique `ConsensusEndNanos` values into each `RetrieveBalanceAtBlock()` call, preventing any query result caching. The DB connection pool becomes saturated; new requests block waiting for connections. The mirror node importer, which shares the same PostgreSQL instance, experiences write contention and lock waits, delaying or stalling gossip transaction ingestion. At sufficient request rates, the Rosetta API becomes fully unavailable (HTTP 500/timeout) and the importer falls behind the gossip stream.

Severity: **High** — complete service disruption achievable with a single attacker machine.

### Likelihood Explanation

The endpoint is publicly reachable (ingress `enabled: true`, no authentication). The attack requires only a valid account identifier (trivially obtained from any public Hedera explorer) and a loop sending POST requests. No credentials, tokens, or special network position are needed. The attack is repeatable and automatable with standard HTTP tooling (curl, wrk, k6 — the project itself ships a k6 load test for this endpoint at `tools/k6/src/rosetta/test/accountBalance.js`). The default-off rate limiting means most production deployments are vulnerable unless an operator explicitly sets `global.middleware: true`.

### Recommendation

1. **Enable the Traefik middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml` line 95, or make the ingress annotation conditional on the middleware being enabled.
2. **Add application-level rate limiting**: Implement a per-IP token-bucket middleware in `rosetta/main.go` (e.g., using `golang.org/x/time/rate`) applied before the router, so rate limiting is enforced regardless of infrastructure configuration.
3. **Add per-request DB query timeout**: Ensure `GetDbWithContext` uses a short deadline (e.g., 5s) so a flood of slow queries does not hold connections indefinitely.
4. **Separate DB connection pools**: Use a dedicated read replica connection pool for the Rosetta API, isolated from the importer's write pool, to prevent cross-contamination.

### Proof of Concept

**Preconditions:** Default Helm deployment with `global.middleware: false` (the default). Rosetta API reachable at `$HOST`.

**Steps:**

```bash
# 1. Confirm no rate limiting is active
curl -s -o /dev/null -w "%{http_code}" -X POST $HOST/account/balance \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "account_identifier":{"address":"0.0.98"},
       "block_identifier":{"index":1}}'
# Returns 200

# 2. Flood with varying block indices (attacker-controlled ConsensusEndNanos)
for i in $(seq 1 10000); do
  curl -s -o /dev/null -X POST $HOST/account/balance \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"account_identifier\":{\"address\":\"0.0.98\"},
         \"block_identifier\":{\"index\":$i}}" &
done
wait

# 3. Observe: DB connection pool exhausted, subsequent requests return 500 or timeout.
#    Mirror node importer metrics show increasing lag in gossip ingestion.
```

**Result:** DB read replicas saturated with concurrent `account_balance`, `crypto_transfer`, `entity`, and `mirror_node_time_partitions` range scans. Importer write throughput degrades. Rosetta API unavailable for legitimate users.
