### Title
Unbounded Database Query Amplification via Opaque EVM Address Flooding in `getEntityIdFromEvmAddress()`

### Summary
`EntityService.getEntityIdFromEvmAddress()` issues a direct, uncached database query for every opaque EVM address lookup. Because the response cache is keyed on the full request URL, flooding with unique opaque EVM addresses produces a cache miss on every request, resulting in one DB query per HTTP request with no bound. An unprivileged attacker can exhaust the database connection pool and saturate DB CPU, causing a non-network DoS against all users of the mirror node REST API.

### Finding Description

**Code path:**

`rest/service/entityService.js`, `getEncodedId()` (lines 118–137): [1](#0-0) 

When `entityId.evmAddress !== null` (the opaque EVM address branch), it unconditionally calls `getEntityIdFromEvmAddress()`: [2](#0-1) 

This issues a raw SQL query against the `entity` table on every invocation: [3](#0-2) 

**Root cause — failed assumption about caching:**

There is a `parseCached` LRU cache in `rest/entityId.js` (lines 301–333): [4](#0-3) 

This cache stores only the parsed `EntityId` object (including the `evmAddress` string). It does **not** cache the result of the database lookup. For a unique opaque EVM address, `parseCached` will store the `EntityId` object, but `getEntityIdFromEvmAddress` will still execute a fresh DB query on every call.

The response-level Redis cache (`responseCacheCheckHandler`, `server.js` lines 94–98) is keyed on the full request URL: [5](#0-4) 

Each unique EVM address produces a unique URL → perpetual cache miss.

**What makes an address "opaque":**

In `parseFromString()` (`rest/entityId.js` lines 250–256), any 40-hex address whose first 12 bytes are not all-zero, or whose numeric suffix exceeds `maxNum`, is classified as opaque (`evmAddress` set, `num = null`): [6](#0-5) 

Any address like `0xdeadbeef...` (non-long-zero) qualifies.

**Exposed endpoints (no authentication required):**

- `GET /api/v1/accounts/:idOrAliasOrEvmAddress` → calls `EntityService.getEncodedId()`
- `GET /api/v1/accounts/:idOrAliasOrEvmAddress/rewards` → calls `EntityService.getEncodedId()`
- `GET /api/v1/accounts/:idOrAliasOrEvmAddress/allowances/tokens` → calls `EntityService.getEncodedId()` [7](#0-6) [8](#0-7) [9](#0-8) 

No rate-limiting middleware is present in the server stack: [10](#0-9) 

### Impact Explanation
Each unique opaque EVM address request forces a full index scan (or index lookup) on the `entity` table's `evm_address` column. At high request rates, this exhausts the PostgreSQL connection pool, starves legitimate queries of connections, and saturates DB CPU. The result is a complete or partial denial of service for all mirror node REST API consumers. Because the mirror node serves as critical read infrastructure for the Hedera ecosystem, this affects all downstream applications relying on account/contract data.

### Likelihood Explanation
The attack requires zero privileges — no API key, no authentication, no on-chain funds. An attacker needs only HTTP access to the public REST API. Generating unique 40-hex addresses is trivially scriptable (e.g., sequential counter encoded as hex). The attack is repeatable, cheap, and can be sustained indefinitely. A single machine with a modest HTTP request rate (hundreds of requests/second) is sufficient to saturate a typical DB connection pool.

### Recommendation
1. **Add a service-layer result cache** for EVM address → entity ID lookups (e.g., a bounded LRU/TTL in-memory cache or Redis lookup before the DB query), keyed on the hex EVM address string.
2. **Add IP-based rate limiting** middleware (e.g., `express-rate-limit`) on all public endpoints, applied before route handlers.
3. **Consider returning a fast 404 without a DB query** for addresses that fail a pre-check (e.g., a Bloom filter of known EVM addresses), or at minimum add a DB query timeout.
4. **Enforce connection pool limits** with a short statement timeout on the `entityFromEvmAddressQuery` to bound per-query DB impact.

### Proof of Concept
```bash
# Generate and flood with unique opaque EVM addresses (non-long-zero)
for i in $(seq 1 100000); do
  # Produce a unique address where first byte is non-zero (never long-zero)
  ADDR=$(printf "aa%038x" $i)
  curl -s "https://<mirror-node-host>/api/v1/accounts/0x${ADDR}" &
done
wait
```

Each request passes input validation (valid 40-hex EVM address), is classified as opaque (first byte `0xaa` ≠ `0x00`), bypasses `parseCached` DB-result caching (none exists), bypasses the response cache (unique URL), and triggers one `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` query. At scale, the DB connection pool is exhausted and legitimate requests begin timing out.
