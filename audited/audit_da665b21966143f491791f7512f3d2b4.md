### Title
Cache Amplification via Batch Slot Query: Unprivileged DB Load Amplification Through contractSlots/contractState Cache Interaction

### Summary
An unprivileged external user can prime the per-contract slot key cache (`slotsPerContract`, max 1500 entries/contract) by sending 1500 distinct `eth_call` requests against a single contract, then exploit the 2-second `contractState` cache TTL to repeatedly trigger batch DB queries of up to 1500 slots with a single request. The batch query in `findStorageBatch()` always fetches ALL cached slot keys for a contract — not just the cache-missed ones — creating up to a 1500× amplification in DB query size relative to the triggering request. This can be sustained indefinitely by any unauthenticated caller.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (line 58–70) checks `contractStateCache` (TTL: `expireAfterWrite=2s`). On a miss, it calls `findStorageBatch()` (line 85–122).

Inside `findStorageBatch()`:
```java
// line 86-87: retrieves or creates the per-contract slot key cache
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
// line 90: adds the new key
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
// line 91: reads ALL currently cached slot keys
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
// ...
// line 103: batch DB query with ALL cached slot keys, not just the missed one
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

**Root cause:** The batch query at line 103 sends every slot key currently in the per-contract cache to the DB, regardless of how many are actually cache-missed. The `slotsPerContract` cache holds up to 1500 slot keys per contract (`expireAfterAccess=5m,maximumSize=1500`), while `contractState` values expire every 2 seconds (`expireAfterWrite=2s`). This mismatch means: slot keys accumulate for 5 minutes, but their values expire every 2 seconds, causing every post-expiry read to re-issue the full 1500-slot batch query.

**Failed assumption:** The design assumes the batch query is a net optimization (preloading). It fails to account for an attacker who deliberately accumulates a large slot key set and then exploits the short value TTL to repeatedly trigger full-size batch queries with minimal per-request gas.

**Exploit flow:**

1. Attacker sends 1500 `eth_call` requests to contract C, each reading a distinct storage slot (slot keys 0x0001 through 0x05DC). Over 5 minutes, the per-contract `slotsPerContract` cache fills to its 1500-entry maximum.
2. After 2 seconds, all values in `contractStateCache` for contract C expire.
3. Attacker sends a single `eth_call` reading slot 0x0001 from C.
4. `contractStateCache.get()` returns null (expired) → `findStorageBatch()` is called.
5. `findStorageBatch()` issues `SELECT slot, value FROM contract_state WHERE contract_id = C AND slot IN (0x0001, 0x0002, ..., 0x05DC)` — a 1500-row IN-clause query.
6. Attacker waits 2 seconds and repeats step 3.

**Why existing checks are insufficient:**

The rate limiter (`requestsPerSecond=500`, `gasPerSecond=7,500,000,000`) operates on request count and gas, not on DB query row count. A single request with 21,000 gas (minimum) consumes `scaleGas(21000) = 2` gas-throttle tokens out of 750,000 available per second. The attacker can sustain 500 requests/second, each triggering a 1500-slot batch query, for a total of 750,000 DB slot lookups/second — with no mechanism to detect or limit this amplification.

### Impact Explanation

Each attacker request (1 slot read) causes a DB query scanning up to 1500 rows. With 500 RPS allowed, the attacker can drive 750,000 DB row lookups/second sustained. The `contractSlots` cache (max 3000 contracts) allows this to be spread across up to 3000 contracts simultaneously, for a theoretical ceiling of 4,500,000 DB row lookups per 2-second cycle. This causes sustained elevated PostgreSQL CPU and I/O load, degrading response times for all users of the mirror node. The 30% resource consumption threshold is readily exceeded given the 1500× query amplification factor.

### Likelihood Explanation

No authentication or special privilege is required. The attacker only needs:
- Knowledge of a contract address with many storage slots (publicly visible on-chain)
- The ability to send standard `eth_call` JSON
