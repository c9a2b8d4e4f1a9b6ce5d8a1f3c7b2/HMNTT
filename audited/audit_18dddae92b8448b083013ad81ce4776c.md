### Title
Unbounded TCP Connection Count Enables DB Connection Pool Exhaustion via Concurrent `getNodes()` Calls

### Summary
The gRPC server enforces `maxConcurrentCallsPerConnection=5` per TCP connection but imposes no limit on the total number of TCP connections from a single client or IP. An unauthenticated attacker can open an arbitrary number of TCP connections and saturate each with 5 concurrent `getNodes()` calls, causing each call to synchronously acquire HikariCP database connections and allocate per-call `nodeStakeMap` heap objects, exhausting the database connection pool and JVM memory.

### Finding Description

**Code path:**

`GrpcConfiguration.java` configures the Netty server with only one constraint:

```java
// GrpcConfiguration.java:31-34
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

`NettyProperties.java:14` sets the default to `5`. No `maxConnections()`, no per-IP limit, and no `maxConnectionAge` is set anywhere in the gRPC module (confirmed by grep across all `grpc/**/*.java`).

**Per-call resource acquisition in `NetworkServiceImpl.getNodes()` (lines 55–77):**

Every `getNodes()` invocation, before returning the `Flux`, synchronously executes three database queries on the calling thread:
- `addressBookRepository.findLatestTimestamp()` — DB query
- `nodeStakeRepository.findLatestTimestamp()` — DB query
- `nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp)` — loads the **entire node stake table** into a `Map<Long,Long>` held in the `AddressBookContext` for the lifetime of the call

Then the paging loop (`page()`, lines 79–108) calls `transactionOperations.execute()` per page, acquiring a HikariCP connection each time. With `pageDelay=250ms` and `pageSize=10`, a typical ~30-node address book requires 3 pages × 250ms ≈ 750ms per call, during which the pipeline, the `nodeStakeMap`, and the per-page DB connection are all live.

**Why the existing check is insufficient:**

`maxConcurrentCallsPerConnection=5` only limits streams per TCP connection. It does not bound the number of TCP connections. An attacker opening `N` connections gets `5N` concurrent `getNodes()` pipelines. With `N=200` connections (trivially achievable from a single host), that is 1000 simultaneous calls, each competing for HikariCP connections. The default HikariCP pool is typically 10–20 connections; 1000 concurrent acquirers will queue, time out, and propagate errors to all legitimate callers.

### Impact Explanation

- **Database connection pool exhaustion**: 1000 concurrent `transactionOperations.execute()` calls against a pool of ~10–20 connections causes all legitimate queries (including `subscribeTopic` and REST endpoints sharing the same pool) to fail with `Connection is not available, request timed out`.
- **Heap pressure**: Each call allocates a `nodeStakeMap` (one `Long` entry per network node, ~30–100 entries). At 1000 concurrent calls this is manageable alone, but combined with the reactive pipeline objects and Netty channel state, it contributes to GC pressure.
- **`boundedElastic` thread pool saturation**: The `repeatWhen` scheduler uses `Schedulers.boundedElastic()`. Thousands of delayed tasks queued there can exhaust its bounded thread count, stalling all reactive work on that scheduler.

Severity: **High** — complete denial of service to all gRPC and potentially REST clients sharing the DB pool, achievable with no authentication.

### Likelihood Explanation

- Requires no credentials, no API key, no prior knowledge beyond the public gRPC port (default 5600).
- Standard tooling (`grpcurl`, any HTTP/2 client library) can open hundreds of TCP connections and issue 5 concurrent streaming RPCs per connection.
- The attack is repeatable and can be sustained indefinitely by re-issuing calls as they complete (~750ms cycle time).
- The `getNodes` endpoint is publicly documented and requires only a valid `file_id` (either `0.0.101` or `0.0.102`, both public knowledge).

### Recommendation

1. **Add a total connection limit** to the Netty server builder in `GrpcConfiguration.java`:
   ```java
   serverBuilder.maxConnectionAge(30, TimeUnit.SECONDS);
   serverBuilder.maxConnectionAgeGrace(5, TimeUnit.SECONDS);
   ```
   and expose a `maxConnections` property in `NettyProperties` mapped to `serverBuilder.maxConnections(n)` (available in grpc-netty ≥ 1.47).

2. **Add a per-IP connection limit** at the load balancer / ingress layer (e.g., Kubernetes `NetworkPolicy` + Envoy `connection_limit` filter).

3. **Add gRPC-level rate limiting** analogous to the `ThrottleConfiguration` already present in the `web3` module — a `ServerInterceptor` that tracks concurrent active calls per remote IP and rejects with `RESOURCE_EXHAUSTED` above a threshold.

4. **Cache the `nodeStakeMap`** across concurrent calls (a `Caffeine` cache keyed on `nodeStakeTimestamp` already exists for other data per `CacheConfiguration.java`) to eliminate per-call heap allocation.

### Proof of Concept

```bash
# Open 200 TCP connections, 5 concurrent getNodes streams each = 1000 concurrent calls
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
      <host>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes \
      > /dev/null &
  done
done
wait
```

Expected result: HikariCP pool exhaustion visible in logs (`Connection is not available, request timed out after Xms`), Prometheus alert `GrpcHighDBConnections` fires (threshold >75% pool utilization), and all subsequent legitimate gRPC and REST calls return errors until the attack stops.
