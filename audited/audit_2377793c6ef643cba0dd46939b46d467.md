### Title
Race Between `computeIfAbsent` and `unsubscribe()` in `RedisTopicListener` Causes Orphaned Flux and Incorrect Map Eviction

### Summary
`RedisTopicListener.getSharedListener()` uses `ConcurrentHashMap.computeIfAbsent` to retrieve or create a shared `Flux`, but `unsubscribe()` removes the map entry by key only (`topicMessages.remove(topic.getTopic())`), not by key+value. When a new subscriber races with the asynchronous `doOnCancel` callback, a stale Flux can be returned to the new subscriber, and when that orphaned Flux later cancels, it incorrectly evicts the *current* canonical Flux from the map — causing all future subscribers to create yet another disconnected stream and miss messages.

### Finding Description

**Exact code locations:**

- `getSharedListener`: [1](#0-0) 
- `subscribe` / `doOnCancel` hook: [2](#0-1) 
- `unsubscribe` (removes by key only): [3](#0-2) 

**Root cause:** `topicMessages.remove(topic.getTopic())` removes the map entry unconditionally by key. It does not verify that the value being removed is the same `Flux` instance that triggered the cancel. This is not atomic with respect to `computeIfAbsent` inserting a new entry.

**Exploit flow (step-by-step):**

1. `Flux_A` is the active shared stream for `"topic.1"` in `topicMessages`.
2. The last subscriber to `Flux_A` cancels (e.g., gRPC client disconnects). Reactor's `share()` operator detects zero subscribers and cancels its upstream. The `doOnCancel` callback is scheduled asynchronously on a Reactor scheduler thread.
3. **Before** `doOnCancel` executes, an attacker-controlled gRPC client immediately re-subscribes to `"topic.1"`. `computeIfAbsent` finds the key present and returns `Flux_A` (still in the map). `share()` re-subscribes to the upstream Redis channel — a new Redis connection is opened.
4. `doOnCancel` now fires (from step 2's cancellation event) and calls `topicMessages.remove("topic.1")` — the map entry is deleted, even though `Flux_A` now has an active subscriber again.
5. A second legitimate subscriber calls `computeIfAbsent("topic.1", ...)` — key is absent — a new `Flux_B` is created and inserted into the map.
6. The attacker from step 3 eventually unsubscribes from `Flux_A`. `Flux_A`'s `doOnCancel` fires and calls `topicMessages.remove("topic.1")` — this removes `Flux_B` (the current canonical entry), not `Flux_A`.
7. All subsequent subscribers create yet another new `Flux_C`, and the cycle repeats. Subscribers on `Flux_A` and `Flux_B` are now orphaned — they receive messages from their own independent Redis subscriptions, invisible to each other and to the map.

**Why existing checks are insufficient:**

- `ConcurrentHashMap.computeIfAbsent` is atomic for the map operation itself, but the `share()` lifecycle (cancel → callback → remove) is asynchronous and not coordinated with the map. [4](#0-3) 
- The `retryWhen` on the subscribe chain handles Redis reconnection but does not prevent the map from being corrupted. [5](#0-4) 
- There is no version/identity check in `unsubscribe()` — it always removes by key regardless of which `Flux` instance triggered the callback. [3](#0-2) 

### Impact Explanation

- **Message loss:** Subscribers on orphaned `Flux` instances receive messages from their own private Redis subscription. Messages published after the split are not visible across all subscribers for the same topic.
- **Resource exhaustion:** Each race cycle opens an additional Redis pub/sub connection for the same topic channel. Under repeated triggering, this exhausts Redis connection limits and server-side file descriptors.
- **Cascading eviction:** Each orphaned Flux's eventual cancel incorrectly evicts the current canonical entry, perpetuating the fragmentation indefinitely.
- Severity: **High** — message delivery integrity is broken for a core HCS (Hedera Consensus Service) streaming feature.

### Likelihood Explanation

An unprivileged external user needs only the ability to open gRPC streams to the mirror node's `subscribeTopic` endpoint (publicly accessible). The attack requires:
1. Subscribing to a topic.
2. Immediately cancelling the gRPC stream.
3. Immediately re-subscribing.

This is trivially scriptable with any gRPC client. The race window is normally narrow but widens significantly during partition recovery (elevated scheduler latency, Redis reconnection backoff delays), exactly the scenario described in the question. No credentials, no elevated permissions, and no knowledge of internal state are required. The attack is repeatable and can be automated.

### Recommendation

Replace the key-only `remove` in `unsubscribe()` with the two-argument `ConcurrentHashMap.remove(key, value)` form, passing the specific `Flux` instance being cancelled:

```java
// In subscribe(), capture the Flux reference before returning:
Flux<TopicMessage> shared = container
        .flatMapMany(...)
        ...
        .share();
// Use a holder/wrapper so doOnCancel can reference the exact instance:
topicMessages.computeIfAbsent(topic.getTopic(), key -> {
    Flux<TopicMessage>[] holder = new Flux[1];
    holder[0] = container
            .flatMapMany(r -> r.receive(...))
            .map(Message::getMessage)
            .doOnCancel(() -> topicMessages.remove(topic.getTopic(), holder[0]))
            .doOnComplete(() -> topicMessages.remove(topic.getTopic(), holder[0]))
            ...
            .share();
    return holder[0];
});
```

The critical fix is `topicMessages.remove(topic.getTopic(), specificFluxInstance)` so that a stale cancel callback cannot evict a newer entry. Additionally, consider using `Sinks` with explicit lifecycle management or a `ReplayProcessor` with a coordinated lock around the subscribe/unsubscribe pair to eliminate the TOCTOU window entirely.

### Proof of Concept

**Preconditions:** Mirror node running with `REDIS` listener type; Redis accessible; gRPC endpoint reachable.

**Steps:**

1. Open gRPC stream: `subscribeTopic(topicId=1, startTime=now)` — this creates `Flux_A` in the map.
2. Immediately cancel the stream from the client side (simulate last-subscriber-leaves).
3. Within the same millisecond (before `doOnCancel` fires), open a second gRPC stream for the same `topicId=1`.
4. Open a third gRPC stream for `topicId=1` from a different client (legitimate subscriber).
5. Publish a message to `topic.1` in Redis: `PUBLISH topic.1 <serialized_TopicMessage>`.
6. Observe: the third client (step 4) may receive the message on `Flux_B` while the second client (step 3) is on orphaned `Flux_A` and misses it — or vice versa, depending on which Flux Redis delivers to.
7. Cancel the second client's stream. Observe that the map entry for `topic.1` is now gone (evicted by `Flux_A`'s cancel callback removing `Flux_B`'s entry).
8. Publish another message — the third client receives nothing (its Flux was evicted from the map; no new canonical Flux exists until the next `computeIfAbsent` call).

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
