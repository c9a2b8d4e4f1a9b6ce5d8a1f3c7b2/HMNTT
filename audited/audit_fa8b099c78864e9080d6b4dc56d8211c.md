### Title
Permanently Cached Zombie `ReactiveRedisMessageListenerContainer` Causes Silent Subscription Hang for All Users

### Summary
The `.cache()` operator on the `container` Mono in `RedisTopicListener` permanently caches the first successfully created `ReactiveRedisMessageListenerContainer`. When the underlying Redis connection transitions to a zombie state (TCP-connected but non-functional), the cached container is returned to every subsequent `subscribe()` call. Because `receive()` on a zombie connection returns a `Flux` that never emits and never errors, the `retryWhen` in `subscribe()` never fires, and every new subscription — regardless of the caller's privilege level — silently hangs indefinitely.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`

- **Lines 47–55** (constructor): The `container` Mono is built with `Mono.defer(() -> Mono.just(new ReactiveRedisMessageListenerContainer(connectionFactory)))`. The constructor call is pure Java object instantiation — it never touches the network — so it always succeeds immediately. The `.retryWhen(...)` at line 54 therefore never has a chance to fire (there is nothing to retry), and `.cache()` at line 55 locks in this container object permanently on the very first subscription.

- **Lines 71–79** (`subscribe()`): Every call to `subscribe()` resolves the cached container via `container.flatMapMany(r -> r.receive(...))`. The `retryWhen` at line 78 only reacts to a terminal `onError` signal. A zombie TCP connection produces neither `onError` nor `onComplete` — it produces silence. The `Flux` returned by `r.receive()` simply never emits, and the retry operator is never invoked.

- **Lines 61–62** (`getSharedListener()`): `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))` stores the first hanging `Flux` (wrapped in `.share()`) in the map. Every subsequent subscriber to the same topic receives this same hanging shared `Flux`.

**Root cause / failed assumption:**

The design assumes that a successfully instantiated `ReactiveRedisMessageListenerContainer` implies a functional Redis connection. This is false: the container object is created without any network I/O, so `.cache()` caches a shell that may later hold a zombie connection. Additionally, the design assumes that a broken connection will surface as an `onError` signal, enabling `retryWhen` to recover. A zombie TCP connection violates this assumption — it produces no signal at all.

**Exploit flow:**

1. Service starts; first `subscribe()` call causes `container` Mono to emit a freshly constructed `ReactiveRedisMessageListenerContainer`. `.cache()` stores it.
2. A network partition occurs that leaves the Redis TCP connection in a zombie state (socket open, no data flows, no RST/FIN sent).
3. An unprivileged gRPC client calls `subscribeTopic` (the public API). This reaches `getSharedListener` → `subscribe()` → `container.flatMapMany(r -> r.receive(...))`.
4. The cached zombie container is returned. `r.receive()` returns a `Flux` that never emits and never errors.
5. The `retryWhen(Retry.backoff(...))` at line 78 is never triggered.
6. The resulting `Flux` is stored in `topicMessages` via `computeIfAbsent` and wrapped with `.share()`.
7. Every subsequent subscriber to any topic on this node receives a `Flux` backed by the same zombie container. All subscriptions hang silently — no error, no timeout, no message.

### Impact Explanation

All active and future topic subscriptions on the affected node are silently starved of messages. gRPC streams remain open but deliver nothing. Clients have no way to distinguish a zombie subscription from a legitimately quiet topic. This constitutes a full availability loss (Denial of Service) for the mirror node's streaming API. Because the `topicMessages` map retains the hanging `Flux` entries and `.cache()` retains the zombie container, the condition is self-perpetuating: even after the network partition heals, the cached zombie container is never replaced, so recovery requires a full service restart.

### Likelihood Explanation

Network partitions that produce zombie TCP connections are a well-known operational reality (cloud provider link flaps, NAT/firewall idle-timeout drops without RST, kernel TCP keepalive disabled or set too long). No authentication, authorization, or special network access is required from the gRPC client side — any client that can open a `subscribeTopic` stream after the partition occurs will land on the hanging `Flux`. The condition is also sticky: it does not self-heal, making it easy to reproduce and confirm.

### Recommendation

1. **Remove `.cache()` or replace it with a cache that can be invalidated.** Use `Mono.defer(...)` without `.cache()` so that each retry in `subscribe()` can obtain a fresh container. Alternatively, use `.cache(Duration)` with a short TTL so the container is periodically re-evaluated.

2. **Add a timeout to the `receive()` Flux.** Apply `.timeout(Duration)` after `r.receive(...)` so that a connection that produces no data within a configurable window is treated as an error, allowing `retryWhen` to fire and obtain a new connection.

3. **Enable TCP keepalive on the Redis connection factory** so that the OS detects and closes zombie connections, converting the silent hang into an `onError` that the existing `retryWhen` can handle.

4. **Invalidate the `topicMessages` cache entry on error/completion.** The `unsubscribe()` method (lines 82–85) already removes the entry on cancel/complete, but it is never called when the `Flux` hangs silently. A timeout-triggered error path would allow `retryWhen` to restart and `unsubscribe()` to clean up the map.

### Proof of Concept

**Preconditions:**
- Mirror node running with `ListenerType.REDIS`.
- A Redis instance reachable from the node.
- Ability to open a gRPC `subscribeTopic` stream (no authentication required in default config).

**Steps:**

1. Start the mirror node; confirm Redis connectivity.
2. Open one gRPC `subscribeTopic` stream to establish the cached container and the shared `Flux` in `topicMessages`.
3. Simulate a zombie connection: use `iptables -A OUTPUT -p tcp --dport 6379 -j DROP` (drops packets silently without RST) on the mirror node host, or use a tool like `tc netem` to black-hole traffic. Do **not** close the existing TCP socket.
4. Open a new gRPC `subscribeTopic` stream from any unprivileged client.
5. Observe: the stream is accepted, no error is returned, but no messages are ever delivered — even if topic messages are published to Redis from another host that can still reach Redis.
6. Restore network connectivity (`iptables -D ...`). Observe: the stream **still** delivers nothing, because the zombie container remains cached and the hanging `Flux` remains in `topicMessages`. A service restart is required to recover.
