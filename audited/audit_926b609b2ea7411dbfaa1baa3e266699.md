### Title
Unbounded Indefinite gRPC Streaming Subscriptions Enable Resource Exhaustion via Griefing

### Summary
An unprivileged external user can open a `subscribeTopic` gRPC stream with no `endTime` and no `limit`, then simply hold the TCP connection open without ever cancelling. Because the only cleanup path is the client-triggered cancel handler (`disposable::dispose`), and no server-side timeout or connection-age limit is configured, the Reactor `Flux` subscription and `topicListener` registration remain alive indefinitely per connection. An attacker opening many such connections exhausts server-side resources (threads, memory, listener slots) with no authentication required.

### Finding Description

**Code path — `ConsensusController.java` lines 43–52:**

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);

if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
    serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
}
``` [1](#0-0) 

The subscription is started at line 48 and the cancel handler is registered at line 51. There is a narrow race window where a client cancel between these two lines leaves the `disposable` un-disposed. More critically, if the client simply **never** cancels, `disposable::dispose` never fires at all — there is no server-initiated teardown.

**No server-side timeout exists:**

`TopicMessageServiceImpl.subscribeTopic()` constructs the Flux as:

```java
Flux<TopicMessage> flux = historical
        .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
``` [2](#0-1) 

`pastEndTime()` returns `Flux.never()` when `filter.getEndTime() == null`:

```java
if (topicContext.getFilter().getEndTime() == null) {
    return Flux.never();
}
``` [3](#0-2) 

No `timeout()` operator is applied anywhere in the chain. No `take()` is applied when `limit == 0`:

```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
``` [4](#0-3) 

**The only connection-level control is `maxConcurrentCallsPerConnection = 5`:**

```java
private int maxConcurrentCallsPerConnection = 5;
``` [5](#0-4) 

This limits calls *per connection* but there is no limit on the number of connections from a single client, no `maxConnectionAge`, no `maxConnectionIdle`, and no `keepAliveTimeout` configured anywhere in `GrpcProperties` or `NettyProperties`. [6](#0-5) 

**Root cause:** The design assumes the client will eventually cancel or that an `endTime`/`limit` will bound the stream. Neither is enforced server-side.

### Impact Explanation

Each held-open subscription keeps alive:
- A Reactor `Flux` subscription (thread/scheduler slot)
- A `topicListener` registration (shared listener state)
- A gRPC server call slot
- Incremented `subscriberCount` with no enforcement ceiling

An attacker opening `N` TCP connections × 5 calls each = `5N` indefinitely-live subscriptions. At scale this exhausts the bounded-elastic scheduler, listener memory, and gRPC server call capacity, degrading or denying service to legitimate users. No authentication or payment is required.

### Likelihood Explanation

Any network-reachable client can issue a `subscribeTopic` gRPC call. The request requires only a valid `topicID` (publicly known) and omitting `endTime` and `limit` (both optional fields). The attack is trivially scriptable, repeatable, and requires no special knowledge or credentials. A single attacker with a modest number of outbound TCP connections can sustain the attack indefinitely.

### Recommendation

Apply multiple layers of defense:

1. **Server-side stream timeout**: Add a `.timeout(grpcProperties.getMaxStreamDuration())` operator to the Flux chain in `TopicMessageServiceImpl.subscribeTopic()` to force-terminate streams that exceed a configurable maximum duration.

2. **Netty connection-age limits**: Configure `maxConnectionAge` and `maxConnectionIdle` on the Netty gRPC server (e.g., via `NettyServerBuilder`) to recycle long-lived connections.

3. **Per-IP or global subscriber cap**: Enforce a maximum on `subscriberCount` (currently only a metric gauge) and reject new subscriptions when the cap is reached.

4. **Fix the cancel-handler race**: Move `setOnCancelHandler` registration *before* calling `.subscribe()`, or use `doOnCancel`/`doFinally` inside the Flux chain itself so cleanup is guaranteed regardless of client behavior.

### Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_topic_query_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# No endTime, no limit — subscription lives forever
query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1)
    # consensusEndTime intentionally omitted
    # limit intentionally omitted (defaults to 0)
)

# Open many connections, each with up to 5 streams
streams = []
for _ in range(1000):
    streams.append(stub.subscribeTopic(query))

# Never cancel — hold all streams open indefinitely
import time
time.sleep(86400)
```

Each stream holds a live Flux subscription and `topicListener` registration on the server. With 1000 connections × 5 calls = 5000 indefinitely-live subscriptions, server resources are exhausted.
