### Title
Unauthenticated `getNodes()` gRPC Stream Exhausts DB Connection Pool via Unbounded Concurrent Connections

### Summary
The `getNodes()` gRPC endpoint in `NetworkServiceImpl` has no global limit on concurrent streams across connections. The only throttle is `maxConcurrentCallsPerConnection = 5`, which is a per-connection limit. An unauthenticated attacker opening many connections can drive hundreds of concurrent streams, each issuing a database query every ~250 ms (the `pageDelay`), collectively exhausting the HikariCP connection pool and denying service to all other users.

### Finding Description

**Code path:**

`NetworkController.getNodes()` → `NetworkServiceImpl.getNodes()` → `page()` (repeated)

`NetworkServiceImpl.getNodes()` (lines 68–76):
```java
return Flux.defer(() -> page(context))
    .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
        .jitter(0.5)
        .withFixedDelay(addressBookProperties.getPageDelay())   // 250 ms
        .withScheduler(Schedulers.boundedElastic()))
    .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
```

`page()` (lines 80–107) calls `transactionOperations.execute()` synchronously, acquiring a HikariCP connection for each page query and releasing it after the query completes. The connection is **not** held for the full stream duration; it is acquired and released once per 250 ms cycle.

**Root cause — failed assumption:**

`GrpcConfiguration` applies `maxConcurrentCallsPerConnection = 5` (default) as the only concurrency guard:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```
This is a **per-connection** limit. There is no global cap on the number of connections, no per-IP connection limit, no authentication, and no stream-level timeout on `getNodes()`. An attacker can open arbitrarily many TCP connections, each carrying 5 concurrent `getNodes()` streams.

**Exploit flow:**

1. Attacker opens *C* TCP connections to port 5600 (no auth required).
2. On each connection, sends 5 concurrent `getNodes(AddressBookQuery{file_id: 0.0.102, limit: 0})` requests.
3. Each stream loops: `page()` → DB query → 250 ms delay → repeat, until all address book entries are returned.
4. With *C* = 200 connections → 1 000 concurrent streams → ~4 000 DB queries/second aggregate.
5. HikariCP default pool size is 10 connections; the pool is saturated, causing all other DB-dependent operations (HCS subscriptions, REST queries) to queue or time out.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` only limits per-connection concurrency; it does not bound total server-wide streams.
- `AddressBookFilter.limit` has `@Min(0)` with no maximum; `limit=0` maps to `Long.MAX_VALUE` (line 73), so the stream runs until the entire address book is consumed.
- No IP-based rate limiting or connection-count limit exists anywhere in the gRPC stack as configured.
- The `pageDelay` minimum of 100 ms (`@DurationMin(millis = 100L)`) is a configuration guard, not a runtime enforcement against many concurrent callers.

### Impact Explanation

Each active stream issues one synchronous DB query per ~250 ms cycle. With enough concurrent streams the HikariCP pool (default 10 connections) is fully saturated. New DB operations block waiting for a connection, causing:
- HCS `subscribeTopic` streams to stall (they also use the same pool).
- All other gRPC and internal DB operations to queue or fail with connection-timeout errors.
- The gRPC server's own Prometheus alert fires at 75 % pool utilization (`hikaricp_connections_active / hikaricp_connections_max > 0.75`), confirming the operational impact.

Severity: **Medium–High** (unauthenticated DoS against a public infrastructure API).

### Likelihood Explanation

The attack requires only a standard gRPC client (e.g., `grpcurl`) and the ability to open many TCP connections — no credentials, no special knowledge, no exploit chain. The address book file IDs (`0.0.101`, `0.0.102`) are publicly documented. The attack is trivially scriptable, repeatable, and can be sustained indefinitely since each stream re-subscribes automatically. A single attacker machine with a modest number of open sockets is sufficient.

### Recommendation

1. **Add a global concurrent-stream limit** for `getNodes()` using a server-side semaphore or a Reactor `flatMap` with `maxConcurrency`, or a dedicated rate-limiter (e.g., Resilience4j `RateLimiter`/`Bulkhead`) applied in `NetworkController.getNodes()`.
2. **Add a per-IP connection limit** at the Netty/load-balancer layer.
3. **Enforce a maximum `limit` value** in `AddressBookFilter` (e.g., `@Max(1000)`) so streams cannot run indefinitely regardless of address book growth.
4. **Add a stream-level timeout** (e.g., `.timeout(Duration.ofSeconds(30))`) in `NetworkServiceImpl.getNodes()` to bound the maximum lifetime of any single stream.
5. Consider requiring authentication (mTLS or an API key) for the `getNodes()` endpoint if it is not intended for fully anonymous public access.

### Proof of Concept

```bash
# Open 200 connections, each with 5 concurrent getNodes streams (1000 total)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.NetworkService/getNodes &
  done
done
wait
```

```java
    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
```
