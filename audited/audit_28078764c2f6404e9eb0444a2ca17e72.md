### Title
Unauthenticated DoS via Unbounded DB Query Flooding Through `getAccountIdsByPublicKey()`

### Summary
The `getAccountIdsByPublicKey()` function in `rest/balances.js` unconditionally executes a database query against the `entity` table for every request containing a `public_key` parameter combined with a `timestamp` parameter. Because the REST API has no rate limiting middleware, an unauthenticated attacker can flood the database with unique-key queries, exhausting the connection pool and degrading or halting mirror node availability.

### Finding Description

**Exact code path:**

`rest/balances.js`, lines 58–76 (`entityPublicKeyQuery` + `getAccountIdsByPublicKey`):

```js
const entityPublicKeyQuery = `select id from entity where type in ('ACCOUNT', 'CONTRACT') and public_key = $1 limit $2`;

const getAccountIdsByPublicKey = async (publicKey, limit) => {
  if (isEmpty(publicKey)) {
    return null;
  }
  const params = [...publicKey, limit];
  const result = await pool.queryQuietly(entityPublicKeyQuery, params);
  ...
  return null;
};
``` [1](#0-0) 

This is called unconditionally from `getBalances()` whenever `tsQuery` is truthy:

```js
const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit);
if (pubKeyQuery && !accountIdsQuery) {
  return;
}
``` [2](#0-1) 

**Root cause:** There is no rate limiting middleware anywhere in the REST API stack. `server.js` registers the `/api/v1/balances` route with no throttling layer: [3](#0-2) 

The only protection present is an optional Redis response cache (`responseCacheCheckHandler`), gated on `config.cache.response.enabled && config.redis.enabled`: [4](#0-3) 

**Why the cache is insufficient:** The cache key includes the full request URL. An attacker who varies the `account.publickey` value on each request produces a unique cache key every time, guaranteeing a cache miss and a live DB query for every single request.

**Exploit flow:**
1. Attacker generates thousands of syntactically valid but non-existent 64-hex-char public keys.
2. For each key, sends: `GET /api/v1/balances?account.publickey=<key>&timestamp=<valid_ts>`
3. Each request passes `validateReq`, passes `parsePublicKeyQueryParam`, enters the `tsQuery` branch, calls `getTsQuery` (one DB query), then calls `getAccountIdsByPublicKey` (second DB query).
4. Both queries return empty results; the handler returns early with an empty response.
5. The attacker receives a cheap 200 response while the DB absorbs two queries per request.

The throttling infrastructure that exists (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` module (contract calls) and has no effect on the REST API: [5](#0-4) 

### Impact Explanation
Sustained flooding exhausts the PostgreSQL connection pool shared between the REST API and the mirror node importer. When the importer cannot acquire DB connections, it falls behind in processing network transactions, causing the mirror node to serve stale or unavailable data. At sufficient request volume, the REST API becomes entirely unresponsive. While Hedera consensus nodes are not directly affected, the mirror node is the primary data access layer for wallets, explorers, and dApps; its unavailability constitutes a critical availability failure for the ecosystem.

### Likelihood Explanation
The attack requires zero authentication, zero privileged access, and zero knowledge of existing accounts. Valid 64-character hex strings are trivially generated. The endpoint is publicly documented in the OpenAPI spec: [6](#0-5) 
The attack is repeatable, automatable, and requires only a modest number of concurrent HTTP clients to saturate a typical DB connection pool.

### Recommendation
1. **Add rate limiting middleware to the REST API** (e.g., `express-rate-limit`) applied globally before route handlers in `server.js`, with per-IP limits.
2. **Add a result cache at the `getAccountIdsByPublicKey` level** (e.g., short-lived in-memory or Redis cache keyed on the normalized public key) so repeated lookups for the same non-existent key do not hit the DB.
3. **Ensure Redis response caching is enabled** in production deployments; document it as a required security control, not an optional feature.
4. **Consider connection pool limits per query type** to prevent read-heavy API traffic from starving the importer's write path.

### Proof of Concept

```bash
# Generate 10,000 unique valid-format but non-existent public keys and flood the endpoint
for i in $(seq 1 10000); do
  KEY=$(openssl rand -hex 32)
  curl -s "https://<mirror-node>/api/v1/balances?account.publickey=${KEY}&timestamp=1234567890.000000000" &
done
wait
```

Each iteration triggers two DB queries (`account_balance` timestamp lookup + `entity` public key lookup) with no rate limiting or caching protection. At sufficient concurrency, the DB connection pool is exhausted and the mirror node becomes unresponsive.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
