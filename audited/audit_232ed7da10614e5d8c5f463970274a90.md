### Title
Unauthenticated Bulk Hook ID Lookup Enables Amplified Database Resource Exhaustion

### Summary
The public `GET /api/v1/accounts/{ownerId}/hooks` endpoint accepts up to `MAX_REPEATED_QUERY_PARAMETERS` (100) `hook.id=eq:X` parameters per request, each of which is passed directly to `findByOwnerIdAndHookIdIn()` as a 100-element `IN` clause. No rate limiting exists in the `rest-java` module for this endpoint. An unauthenticated attacker can flood the endpoint with maximum-size requests targeting a non-existent `ownerId`, forcing the database to perform 100 primary-key index point-lookups per request that all miss, at an unbounded request rate.

### Finding Description

**Exact code path:**

`HooksController.java` lines 83–86 accept up to `MAX_REPEATED_QUERY_PARAMETERS = 100` (defined in `Constants.java` line 36) `hook.id=eq:X` query parameters with only a `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` annotation:

```java
@RequestParam(defaultValue = "", name = HOOK_ID, required = false)
    @Size(max = MAX_REPEATED_QUERY_PARAMETERS)
    NumberRangeParameter[] hookId,
``` [1](#0-0) [2](#0-1) 

`HookServiceImpl.java` lines 43–51 filter the IDs by range bounds (defaulting to `[0, MAX_VALUE]`), meaning all 100 EQ-filter IDs pass through and are forwarded to the repository:

```java
final var idsInRange = request.getHookIds().stream()
        .filter(hookId -> hookId >= lowerBound && hookId <= upperBound)
        .toList();
...
return hookRepository.findByOwnerIdAndHookIdIn(id.getId(), idsInRange, page);
``` [3](#0-2) 

`HookRepository.java` line 13 issues a Spring Data JPA-derived query that becomes `SELECT ... FROM hook WHERE owner_id = ? AND hook_id IN (?, ?, ..., ?)` with all 100 values: [4](#0-3) 

**Root cause — failed assumption:** The `@Size(max = 100)` guard limits the *per-request* parameter count but makes no attempt to limit the *request rate*. The rest-java module has no rate-limiting filter or throttle bean for this endpoint. The `ThrottleConfiguration` and `ThrottleManagerImpl` that exist in the codebase are scoped exclusively to the `web3` module and are not applied here: [5](#0-4) 

The `rest-java` config directory contains only `LoggingFilter`, `MetricsFilter`, `ShallowEtagHeaderFilter`, and MVC configuration — no rate limiter:


**Why existing checks fail:** `@Size(max = 100)` is a per-request input-size guard, not a resource-consumption guard. It does not prevent an attacker from sending thousands of such maximum-size requests per second. The `entityService.lookup(request.getOwnerId())` call resolves the owner entity before the repository call, but does not short-circuit when the owner has no hooks — the 100-element `IN` query is still issued. [6](#0-5) 

### Impact Explanation

Each request with 100 `hook.id=eq:X` parameters causes the database to perform 100 primary-key index point-lookups (`(owner_id, hook_id)` composite PK) that all return empty. This is a 100× amplification of index I/O per request compared to a zero-parameter call. With no rate limiting, a single attacker process can sustain thousands of such requests per second, driving database CPU (index traversal) and I/O (buffer pool pressure) well above the 30% threshold relative to baseline. The hook table is distributed by `owner_id` (Citus), so targeting a non-existent `owner_id` still routes to a shard and forces index work on that node. [7](#0-6) 

### Likelihood Explanation

The endpoint is fully public — no authentication, no API key, no IP-based throttle in the `rest-java` stack. Any external user can craft the request with a trivial script:

```
GET /api/v1/accounts/0.0.999999999/hooks?hook.id=eq:1&hook.id=eq:2&...&hook.id=eq:100
```

This requires zero privileges, zero knowledge of the system beyond the public API spec, and is trivially repeatable in a loop. The attacker does not need to know valid owner IDs — targeting a non-existent one maximises miss-rate and avoids any result-caching benefit.

### Recommendation

1. **Add a rate-limiting filter to `rest-java`** analogous to the `ThrottleConfiguration` in `web3`, applied globally or specifically to `/api/v1/accounts/*/hooks`.
2. **Charge cost proportional to collection size**: count each element in the `hookIds` collection as one rate-limit token, so a 100-element request consumes 100 tokens.
3. **Short-circuit before the DB call** if the owner entity does not exist (the `entityService.lookup` already resolves the entity; throw 404 before issuing the `IN` query).
4. Consider capping `MAX_REPEATED_QUERY_PARAMETERS` for the hooks endpoint specifically to a lower value (e.g., 10–25).

### Proof of Concept

```bash
# Craft a request with 100 distinct hook IDs against a non-existent owner
PARAMS=$(seq 1 100 | awk '{printf "hook.id=eq:%d&", $1}')
while true; do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.999999999/hooks?${PARAMS}" -o /dev/null
done
```

Each iteration issues one SQL query with a 100-element `IN` clause against the `hook` table's primary-key index. Sustained over seconds, database CPU and I/O metrics will rise proportionally to the request rate with no server-side throttle to stop it.
