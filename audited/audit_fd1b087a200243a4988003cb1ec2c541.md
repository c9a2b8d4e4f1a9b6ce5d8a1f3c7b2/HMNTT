### Title
`getEncodedId()` Priority Ordering Causes Alias Lookup Bypass for Inputs Satisfying Both `EntityId.isValidEntityId()` and `AccountAlias.isValid()`

### Summary
The `getEncodedId()` function in `rest/service/entityService.js` uses a strict if-else chain that always evaluates `EntityId.isValidEntityId()` first. A non-empty intersection exists between the character sets accepted by `evmAddressRegex` and `accountAliasRegex`, meaning a 40-character string composed entirely of uppercase `{A, B, C, D, E, F, 2, 3, 4, 5, 6, 7}` satisfies both validators simultaneously. For any such input, the alias resolution branch is permanently unreachable, causing the system to perform an EVM-address lookup instead of an alias lookup, potentially returning a different entity or a spurious `NotFoundError`.

### Finding Description

**Code location:** `rest/service/entityService.js`, `getEncodedId()`, lines 118–137.

```
// entityService.js lines 120-127
if (EntityId.isValidEntityId(entityIdString)) {   // checked FIRST
    const entityId = EntityId.parseString(entityIdString, {paramName});
    return entityId.evmAddress === null
      ? entityId.getEncodedId()
      : await this.getEntityIdFromEvmAddress(entityId, requireResult);
} else if (AccountAlias.isValid(entityIdString)) { // NEVER reached for ambiguous input
    return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
}
```

**Root cause — validator overlap:**

`EntityId.isValidEntityId()` (entityId.js line 133) accepts EVM addresses via:
```
evmAddressRegex = /^(0x)?[A-Fa-f0-9]{40}$/
```
`AccountAlias.isValid()` (accountAlias.js line 10) accepts:
```
accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/
```

The intersection of `[A-Fa-f0-9]` (uppercase only) and `[A-Z2-7]` is `{A, B, C, D, E, F, 2, 3, 4, 5, 6, 7}`. A 40-character string drawn entirely from this set — e.g., `ABCDEF2345672345672345672345672345672345` — passes **both** validators.

**Divergent resolution paths:**

- EVM address path (entityService.js line 91): queries `entity.evm_address = Buffer.from(input, 'hex')` → 20 decoded bytes.
- Alias path (entityService.js line 43): queries `entity.alias = base32.decode(input)` → 25 decoded bytes (40 × 5 / 8).

These are structurally different byte sequences; they cannot resolve to the same DB row.

**Why existing checks are insufficient:**

There is no disambiguation step between the two validators. The if-else ordering at lines 120 and 125 is the sole routing mechanism. Once `EntityId.isValidEntityId()` returns `true`, the alias branch at line 125 is dead code for that input.

### Impact Explanation

An account whose alias base32-encodes to a 40-character string from `{A–F, 2–7}` becomes unreachable via `getEncodedId()` when queried by alias: the function silently reroutes to an EVM-address lookup. If no entity holds that EVM address, the caller receives a `NotFoundError` for an account that genuinely exists. If a different entity holds that EVM address, the caller receives the wrong entity's encoded ID — a cross-account misdirection. All downstream callers of `getEncodedId()` (account, token, contract, schedule, balance, and topic-message controllers) inherit this misbehavior.

### Likelihood Explanation

Standard Hedera public-key aliases (ED25519 = 32 bytes → 52 base32 chars; ECDSA secp256k1 = 33 bytes → 53 base32 chars) are not 40 characters and do not trigger this path. However, the Hedera protocol permits arbitrary-byte aliases. An account with a 25-byte alias whose base32 representation falls entirely within `{A–F, 2–7}` would trigger the bug. An attacker who can observe or predict such an alias can reliably cause the wrong resolution on every request — no special privilege is required; only the ability to supply a query parameter.

### Recommendation

Replace the flat if-else priority chain with explicit format detection that is mutually exclusive:

1. Check for `shard.realm.num` / encoded-integer forms first (no overlap with aliases).
2. Check for EVM address forms **only when the input is exactly 40 hex characters and contains at least one character outside `[A-Z2-7]`**, or when it carries a `0x` prefix.
3. Fall through to `AccountAlias.isValid()` for all remaining inputs.

Alternatively, add a pre-check: if `AccountAlias.isValid(input)` is true, always prefer the alias path regardless of whether `EntityId.isValidEntityId()` is also true, since a genuine EVM address supplied by a caller would normally carry a `0x` prefix or contain lowercase hex digits or digits `{0,1,8,9}` that disqualify it from the alias regex.

### Proof of Concept

**Precondition:** An account exists in the mirror node DB with alias bytes equal to `base32.decode("ABCDEF2345672345672345672345672345672345")` (25 bytes). No entity has `evm_address` equal to `0xABCDEF2345672345672345672345672345672345` (20 bytes).

**Trigger:**
```
GET /api/v1/accounts/ABCDEF2345672345672345672345672345672345
```

**Execution trace:**
1. `getEncodedId("ABCDEF2345672345672345672345672345672345")` is called.
2. `EntityId.isValidEntityId("ABCDEF2345672345672345672345672345672345")` → `true` (matches `evmAddressRegex`).
3. `EntityId.parseString(...)` → `evmAddress = "ABCDEF2345672345672345672345672345672345"`.
4. `getEntityIdFromEvmAddress(entityId)` queries `entity.evm_address = \xABCDEF...` → 0 rows.
5. `requireResult=true` → throws `NotFoundError`.
6. `AccountAlias.isValid(...)` is **never evaluated**.

**Result:** The account is unreachable by alias despite existing in the database. Substituting an entity that does hold that EVM address in step 4 produces cross-account misdirection instead.
