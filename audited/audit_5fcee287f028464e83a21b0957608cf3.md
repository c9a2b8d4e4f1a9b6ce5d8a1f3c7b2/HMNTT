### Title
Unauthenticated `block_hash` Filter Triggers Extra DB Lookup, Enabling Connection Pool Exhaustion DoS on Mirror Node REST API

### Summary
Any unauthenticated caller can send `GET /api/v1/contracts/{contractId}/results?block_hash=<value>` to force `extractContractResultsByIdQuery()` to issue an extra `LIKE`-based DB query via `RecordFileService.getRecordFileBlockDetailsFromHash()` before the main contract-results query. Because the REST API has no rate-limiting middleware and the default DB connection pool is only 10 connections, an attacker holding 10 concurrent requests saturates the pool within the 20-second statement timeout window, making the REST API unresponsive to all legitimate users.

### Finding Description

**Route registration** — `rest/routes/contractRoute.js` line 14 registers the endpoint with no authentication: [1](#0-0) 

**Handler** — `getContractResultsById` calls `extractContractResultsByIdQuery` unconditionally: [2](#0-1) 

**Extra DB lookup** — inside `extractContractResultsByIdQuery`, when `filter.key === filterKeys.BLOCK_HASH`, a separate DB round-trip is issued before the main query: [3](#0-2) 

**The hash query** — `getRecordFileBlockDetailsFromHash` executes a `LIKE` prefix scan (`where hash like $1`) with no short-circuit other than `limit 1`: [4](#0-3) [5](#0-4) 

**Connection pool** — the pool is capped at `maxConnections` (default **10**) with a `statementTimeout` of **20 000 ms** and a `connectionTimeout` of **20 000 ms**: [6](#0-5) [7](#0-6) 

**No rate limiting** — a grep across `rest/**` for `rateLimit`, `rateLimiter`, or `express-rate-limit` returns only a test-utility match. The throttling code found (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) belongs exclusively to the **web3** Java service, not the Node.js REST API: [8](#0-7) 

**Exploit flow:**
1. Attacker opens 10 concurrent HTTP connections to `GET /api/v1/contracts/0.0.1/results?block_hash=<random_64_hex_chars>`.
2. Each request enters `extractContractResultsByIdQuery`, hits the `BLOCK_HASH` branch, and calls `getRecordFileBlockDetailsFromHash`, issuing a `LIKE` query that holds a DB connection for up to 20 seconds (statement timeout).
3. All 10 pool slots are occupied. New legitimate requests queue for a connection.
4. After 20 seconds the `connectionTimeout` fires and legitimate requests receive a DB error, surfaced as a 500 to the caller.
5. Attacker re-issues the 10 requests immediately after each timeout cycle, maintaining continuous saturation.

### Impact Explanation
The mirror node REST API becomes fully unresponsive to legitimate users for the duration of the attack. Every endpoint sharing the same `pg` pool is affected, not just the contracts endpoint. Because the REST API is the primary read interface for dApps, wallets, and explorers querying Hedera state, sustained unavailability constitutes a meaningful service disruption. The default pool size of 10 makes the threshold trivially low.

### Likelihood Explanation
No credentials, API keys, or special network position are required. The attack needs only 10 concurrent HTTP requests, achievable from a single machine with `curl --parallel` or any scripting language. The 20-second statement timeout means the attacker must sustain ~0.5 requests/second per slot — well within the capability of any consumer-grade internet connection. The attack is fully repeatable and requires no prior knowledge of valid contract IDs (any syntactically valid ID works; the hash lookup fires regardless of whether the contract exists).

### Recommendation
1. **Add per-IP rate limiting** to the Node.js REST API (e.g., `express-rate-limit`) before any DB-touching handler.
2. **Increase the default pool size** or deploy a connection pooler (PgBouncer) in front of the REST API to absorb burst traffic without exhausting backend connections.
3. **Cache the block-hash → timestamp-range mapping** (e.g., in the existing Redis layer) so repeated or concurrent lookups for the same or non-existent hashes do not each consume a DB connection.
4. **Validate block hash format strictly** before issuing the DB query and return 400 immediately for malformed values, reducing the attack surface.

### Proof of Concept
```bash
# Exhaust the 10-connection pool with concurrent block_hash requests
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/contracts/0.0.1/results?block_hash=$(openssl rand -hex 32)" &
done
wait

# Legitimate request now times out or returns 500
curl -v "http://<mirror-node>/api/v1/contracts/0.0.1/results"
```
Repeat the first loop every ~19 seconds to maintain saturation. Legitimate requests will receive connection-timeout errors for the duration.
