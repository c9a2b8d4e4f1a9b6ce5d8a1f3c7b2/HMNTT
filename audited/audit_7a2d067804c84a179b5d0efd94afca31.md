### Title
Cache Stampede via Unsynchronized `@Cacheable` on `findRuntimeBytecode` Allows Concurrent DB Query Flood

### Summary
`ContractRepository.findRuntimeBytecode()` uses Spring's `@Cacheable` without `sync=true`, meaning multiple concurrent threads that simultaneously encounter a cache miss for the same `contractId` will all proceed to execute the underlying database query before any result is stored. An unprivileged external user can deliberately trigger this by sending many concurrent requests for the same uncached contract, causing a burst of identical database queries that exhausts connection pool capacity and degrades service availability.

### Finding Description
**Exact location:**
- `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18
- Cache manager bean: `web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java`, lines 67–73

**Root cause:**

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

Spring's `@Cacheable` without `sync=true` uses a non-atomic check-then-act pattern: it checks the cache, finds a miss, and then calls the underlying method — but this is not done under any lock. Caffeine supports atomic single-loader semantics via `cache.get(key, loader)`, which Spring only uses when `sync=true` is specified. Without it, N concurrent threads all observing a cache miss for the same key will all invoke the DB query concurrently.

The `CaffeineCacheManager` is configured with:
```
expireAfterAccess=1h, maximumSize=1000, recordStats
```
No stampede protection is configured.

The `unless = "#result == null"` condition does not help: the method returns `Optional<byte[]>`, which is never `null` (it is either `Optional.of(...)` or `Optional.empty()`), so this condition is always false and does not affect caching behavior.

**Exploit flow:**
1. Attacker identifies any valid `contractId` whose bytecode is not yet in the cache (e.g., a newly deployed contract, or after the 1-hour `expireAfterAccess` window).
2. Attacker sends a burst of concurrent `eth_getCode` (or equivalent contract call) requests all targeting that `contractId`.
3. All concurrent threads simultaneously miss the cache and each independently executes `SELECT runtime_bytecode FROM contract WHERE id = :contractId`.
4. The first thread to complete populates the cache; all others have already issued their DB queries.

**Why existing checks are insufficient:**

The global rate limiter (`requestsPerSecond = 500`, `gasPerSecond = 7,500,000,000`) is a shared bucket across all clients. An attacker can consume the entire 500 req/s budget with a single burst, all targeting the same uncached key. The gas throttle (`scaleGas` divides by 10,000 and returns 0 for gas ≤ 10,000) means low-gas calls like `eth_getCode` consume zero gas tokens and are not gas-throttled at all:

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {  // GAS_SCALE_FACTOR = 10_000
        return 0L;                  // zero gas consumed — no gas throttle applied
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
```

This means the gas bucket provides no protection for bytecode-retrieval calls.

### Impact Explanation
Up to 500 concurrent identical `SELECT` queries can be issued to the database for a single cache miss event. Under sustained attack (attacker repeatedly targets contracts whose cache entries have expired or are newly uncached), the database connection pool can be saturated, causing query queuing, latency spikes, and eventual request timeouts for all users. The impact is a partial or full denial of service of the web3 API's contract execution path.

### Likelihood Explanation
No authentication or per-IP rate limiting is required. Any external user can send HTTP requests to the public JSON-RPC endpoint. The attack is trivially scriptable (e.g., 500 concurrent `eth_getCode` calls in a tight loop). The attacker can sustain the stampede by targeting contracts whose cache entries expire (1-hour TTL) or by targeting many different uncached contracts simultaneously. The gas throttle bypass (zero gas consumed for small-gas calls) removes the primary intended rate-control mechanism.

### Recommendation
Add `sync=true` to the `@Cacheable` annotation:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null",
    sync = true
)
```

With `sync=true`, Spring delegates to Caffeine's `cache.get(key, loader)` which guarantees that only one thread executes the loader for a given key; all other concurrent threads block and receive the same result once it is computed. This eliminates the stampede entirely. Additionally, consider adding per-IP rate limiting at the ingress/controller layer to prevent a single client from consuming the entire global request budget.

### Proof of Concept
```bash
# Identify any contractId not currently in cache (e.g., after server restart or 1h expiry)
CONTRACT_ID="0x000000000000000000000000000000000000abcd"

# Send 500 concurrent eth_getCode requests for the same address
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$CONTRACT_ID\",\"data\":\"0x\",\"gas\":\"0x2710\"}" &
done
wait
