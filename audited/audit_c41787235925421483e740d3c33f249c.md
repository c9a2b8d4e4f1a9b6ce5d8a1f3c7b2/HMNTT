### Title
Cache Bypass via Historical Block Queries Enabling Disproportionate DB Load Amplification

### Summary
`ContractStateServiceImpl.findStorageByBlockTimestamp()` has no caching at any layer, while the current-state path `findStorage()` benefits from both `@Cacheable` at the repository level and batch caching at the service level. Any unprivileged user can force all contract storage reads through the uncached historical path by specifying a historical block in `POST /api/v1/contracts/call`, causing every storage slot read to execute a live DB query against `contract_state_change`. The global (not per-IP) rate limiter does not prevent a single attacker from monopolizing the entire 500 req/s budget with uncached queries, and the gas throttle is bypassable with low-gas requests.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageByBlockTimestamp()` (lines 73–76) unconditionally delegates to the repository with zero cache interaction:

```java
public Optional<byte[]> findStorageByBlockTimestamp(
        final EntityId entityId, final byte[] slotKeyByteArray, final long blockTimestamp) {
    return contractStateRepository.findStorageByBlockTimestamp(entityId.getId(), slotKeyByteArray, blockTimestamp);
}
``` [1](#0-0) 

The repository method carries no `@Cacheable` annotation (unlike `findStorage()` at line 20–21 which does): [2](#0-1) [3](#0-2) 

**How the historical path is selected:**

`ContractStorageReadableKVState.readFromDataSource()` checks `ContractCallContext.get().getTimestamp()`. When a timestamp is present (historical block), it calls `findStorageByBlockTimestamp()`; otherwise it calls the cached `findStorage()`: [4](#0-3) 

**How an external user sets the timestamp:**

`ContractController.call()` accepts a `block` field in the request body. Any value other than `latest` (e.g., a specific block number or hash) causes the EVM execution context to carry a historical timestamp, routing every storage read through the uncached path: [5](#0-4) 

**Why rate limiting is insufficient:**

`ThrottleManagerImpl` enforces a single global `rateLimitBucket` (default 500 req/s) and a `gasLimitBucket`. The gas throttle is bypassable: `scaleGas()` returns `0` for any `gas <= 10_000`, meaning requests with minimal gas consume zero gas tokens and are only subject to the 500 req/s request limit: [6](#0-5) [7](#0-6) 

The bucket is global (not per-IP), so a single attacker can consume the entire 500 req/s budget: [8](#0-7) 

**Correction on "full table scan":** The query is NOT a full table scan. A composite index `contract_state_change__id_slot_timestamp` on `(contract_id, slot, consensus_timestamp)` exists: [9](#0-8) 

The query performs an efficient index range scan. However, the absence of any result caching means every historical request still executes a live DB round-trip, whereas the current-state path serves repeated identical queries entirely from the in-process Caffeine cache with zero DB I/O.

### Impact Explanation

The asymmetry is concrete: under the cached `findStorage()` path, a warm cache serves repeated slot reads with zero DB queries. Under `findStorageByBlockTimestamp()`, every single read — even for the same `(contractId, slot, blockTimestamp)` triple — executes a live DB query. An attacker saturating the 500 req/s global budget with historical queries (using `gas=0` to bypass the gas throttle) forces the DB to handle up to 500 index-range-scan queries per second that would otherwise be cache hits. On a node where the current-state cache absorbs the majority of storage reads, this represents a large relative increase in DB query volume, consistent with the 30%+ threshold described. Additionally, legitimate users are crowded out of the global rate limit bucket.

### Likelihood Explanation

No account, authentication, or special privilege is required. Any user with network access to `POST /api/v1/contracts/call` can set `"block": "<historical_block_number>"` and `"gas": 0` in the JSON body. The attack is trivially scriptable, repeatable, and requires no knowledge of contract internals — any valid contract address and slot key suffices. The attacker can sustain the attack indefinitely within the 500 req/s global limit.

### Recommendation

1. **Add caching for historical queries** using a bounded cache keyed on `(contractId, slot, blockTimestamp)`. Since historical state is immutable, cache entries never need invalidation. A size-bounded Caffeine cache (e.g., 10,000 entries) would absorb repeated queries for the same historical slot.
2. **Enforce per-IP rate limiting** in addition to the global bucket, so a single client cannot monopolize the entire request budget.
3. **Remove the gas-throttle bypass** by setting a minimum gas floor (e.g., treat `gas < 21,000` as `21,000` for throttle accounting) so the gas bucket cannot be bypassed with `gas=0`.

### Proof of Concept

```bash
# Attacker script: saturate the global rate limit with uncached historical queries
# No authentication required. gas=0 bypasses the gas throttle bucket.
while true; do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{
      "to":    "0x0000000000000000000000000000000000000001",
      "data":  "0x",
      "gas":   0,
      "block": "0x1"
    }' &
done
```

Each request routes through `ContractStorageReadableKVState.readFromDataSource()` → `findStorageByBlockTimestamp()` → live `contract_state_change` DB query. Repeating with varying `block` values (e.g., `0x1`, `0x2`, … `0xN`) ensures no accidental cache reuse even if caching were added naively without the timestamp dimension. DB query rate rises proportionally to request rate while the cached current-state path would generate near-zero DB queries at the same request rate.
