### Title
Reverted Transactions Recorded as SUCCESS in Metrics via ContractDebugService.validateResult() Exception Swallowing

### Summary
`ContractDebugService` overrides `validateResult()` and silently swallows the `MirrorEvmTransactionException` that would otherwise update the `status` variable in `doProcessCall()`. Because `status` is initialized to `"SUCCESS"` and only updated inside the `catch (MirrorEvmTransactionException e)` block — which is never reached when the exception is swallowed — `updateMetrics()` is called with `status="SUCCESS"` for every reverted transaction replayed through the opcode-trace endpoint. Any unauthenticated user can trigger this by calling the publicly accessible `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint with the hash of a reverted transaction.

### Finding Description

**Root cause — `ContractDebugService.validateResult()` swallows the exception:**

`ContractDebugService.java` lines 62–72:
```java
@Override
protected void validateResult(final EvmTransactionResult txnResult, final CallServiceParameters params) {
    try {
        super.validateResult(txnResult, params);   // throws MirrorEvmTransactionException for reverts
    } catch (MirrorEvmTransactionException e) {
        log.warn(...);   // exception is swallowed here — never re-thrown
    }
}
```

**How `status` is set in `doProcessCall()`:**

`ContractCallService.java` lines 111–136:
```java
var status = ResponseCodeEnum.SUCCESS.toString();   // initialized to "SUCCESS"
try {
    result = transactionExecutionService.execute(params, estimatedGas);
    if (!estimate) {
        validateResult(result, params);   // ContractDebugService version swallows the exception
    }
} catch (MirrorEvmTransactionException e) {
    result = e.getResult();
    status = e.getMessage();   // NEVER REACHED when exception is swallowed
    throw e;
} finally {
    if (!estimate) {
        if (result != null) {
            updateMetrics(params, result.gasUsed(), 1, status);   // called with "SUCCESS"
        }
    }
}
```

**Why `result` is non-null for the opcode path:**

`TransactionExecutionService.java` lines 154–159 explicitly returns a non-null `EvmTransactionResult` (instead of throwing) when `getOpcodeContext() != null`:
```java
} else {
    // opcode trace scenario — return failed result instead of throwing
    return new EvmTransactionResult(status, result);
}
```

**Full exploit flow:**
1. `execute()` returns a non-null `EvmTransactionResult` with a failed `responseCodeEnum` (e.g., `CONTRACT_REVERT_EXECUTED`).
2. `validateResult()` is called (`estimate=false`).
3. `ContractDebugService.validateResult()` calls `super.validateResult()`, which throws `MirrorEvmTransactionException`.
4. The exception is caught and swallowed; `validateResult()` returns normally.
5. The `catch (MirrorEvmTransactionException e)` block in `doProcessCall()` is never entered; `status` stays `"SUCCESS"`.
6. `finally` block: `result != null` → `updateMetrics(params, result.gasUsed(), 1, "SUCCESS")` is called for a reverted transaction.

### Impact Explanation
Every reverted transaction replayed via the opcode endpoint is counted under the `hiero.mirror.web3.evm.invocation` metric with `status=SUCCESS` and its gas is added to the `hiero.mirror.web3.evm.gas.used` SUCCESS bucket. This corrupts the operational metrics used to monitor EVM health, success/failure ratios, and gas consumption patterns. Operators relying on these metrics for alerting or capacity planning receive a systematically inflated SUCCESS count, masking the true revert rate. The gas throttle bucket (`restoreGasToBucket`) is also affected: because `result.isSuccessful()` returns `false` but no exception propagates, the throttle restoration logic at line 144 takes the wrong branch for a reverted transaction.

### Likelihood Explanation
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint is publicly accessible with no authentication requirement — only an `Accept-Encoding: gzip` header and the feature flag being enabled are needed. Any user who knows (or can discover) the hash of a reverted transaction can trigger this repeatedly. The opcode feature is designed for debugging and is expected to be called against failed/reverted transactions, making this the normal usage pattern rather than an edge case.

### Recommendation
`ContractDebugService.validateResult()` must re-throw the exception after logging, or the `status` variable must be derived directly from `txnResult.responseCodeEnum()` in the `finally` block rather than relying on the exception path to update it:

```java
// Option A: re-throw after logging
@Override
protected void validateResult(final EvmTransactionResult txnResult, final CallServiceParameters params) {
    try {
        super.validateResult(txnResult, params);
    } catch (MirrorEvmTransactionException e) {
        log.warn("Transaction failed with status: {}, detail: {}, revertReason: {}",
                txnResult.responseCodeEnum(), e.getDetail(), e.getData());
        throw e;   // re-throw so doProcessCall() updates status correctly
    }
}
```

Or, in `doProcessCall()`, derive `status` from the result object in the `finally` block independently of the exception path:

```java
finally {
    if (!estimate && result != null) {
        var finalStatus = result.isSuccessful()
                ? ResponseCodeEnum.SUCCESS.toString()
                : result.responseCodeEnum().protoName();
        updateMetrics(params, result.gasUsed(), 1, finalStatus);
    }
}
```

### Proof of Concept
**Preconditions:** Mirror node web3 service running with `opcodes.enabled=true`; a reverted contract transaction exists with known hash `0xABCD...`.

**Steps:**
1. Send `GET /api/v1/contracts/results/0xABCD.../opcodes` with header `Accept-Encoding: gzip`.
2. Observe the response: `"failed": true` (correctly reported in the HTTP response).
3. Query the Micrometer metric `hiero.mirror.web3.evm.invocation` filtered by `status=SUCCESS`.
4. Observe the counter has been incremented — the reverted transaction is counted as a SUCCESS invocation.
5. Repeat to accumulate an arbitrarily large SUCCESS count for reverted transactions, corrupting the metric dashboard and any alerting thresholds based on it.
