### Title
Unbounded `CaffeineCacheManager` Internal Cache Map Causes Memory Exhaustion via Distinct ContractId Flooding

### Summary
In `ContractStateServiceImpl.findStorageBatch()`, each unique `contractId` causes `cacheManagerSlotsPerContract.getCache(contractId.toString())` to be called, which stores a new `CaffeineCache` entry in the `CaffeineCacheManager`'s internal `ConcurrentHashMap` (`cacheMap`). This `cacheMap` is unbounded and never cleaned up — even when the outer `contractSlotsCache` (maximumSize=3000) evicts entries, the corresponding entries in `cacheMap` are permanently retained. An unprivileged attacker can exploit this by sending requests with a continuous stream of distinct contract addresses, causing monotonically growing memory consumption until the JVM heap is exhausted.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, function `findStorageBatch()`, lines 86–87:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Two-level cache structure:**
- **Outer cache** (`contractSlotsCache`): configured via `contractSlots = "expireAfterAccess=5m,maximumSize=3000,recordStats"` — a Caffeine cache keyed by `EntityId`, values are `CaffeineCache` instances. Bounded at 3000 entries.
- **Inner cache manager** (`cacheManagerSlotsPerContract`): configured via `slotsPerContract = "expireAfterAccess=5m,maximumSize=1500"` — a `CaffeineCacheManager` that creates named `CaffeineCache` instances on demand and stores them in its internal `ConcurrentHashMap<String, Cache> cacheMap`. **This map has no size bound.**

**Root cause — failed assumption:**
The design assumes that eviction from the outer `contractSlotsCache` is sufficient to bound memory. It is not. Spring's `CaffeineCacheManager.getCache(name)` stores every created `CaffeineCache` permanently in its internal `cacheMap`. When the outer cache evicts an entry for contractId X, the `CaffeineCache` for X remains in `cacheMap`. On the next miss for X, `cacheManagerSlotsPerContract.getCache(X.toString())` returns the existing (stale) `CaffeineCache` from `cacheMap` — it is not re-created, but it is also never removed. The `cacheMap` grows monotonically with every distinct contractId ever seen.

**Exploit flow:**
1. Attacker sends `POST /api/v1/contracts/call` (or JSON-RPC `eth_call`) requests targeting distinct contract addresses (e.g., `0x0000...0001`, `0x0000...0002`, …).
2. Each unique `to` address maps to a distinct `EntityId` / `contractId.toString()`.
3. `findStorage()` → `findStorageBatch()` is called for each.
4. `cacheManagerSlotsPerContract.getCache(contractId.toString())` creates a new `CaffeineCache` and inserts it into `cacheMap`.
5. After 3000 distinct contractIds, the outer cache begins evicting. The `cacheMap` is unaffected and continues growing.
6. After N distinct contractIds, `cacheMap` holds N `CaffeineCache` objects. Each wraps a Caffeine `BoundedLocalCache` with internal data structures (hash table, read/write buffers, queues) consuming memory even when empty.

**Why existing checks fail:**
- The outer `contractSlotsCache` eviction (maximumSize=3000) only removes entries from the outer Caffeine cache. It has no eviction listener that removes the corresponding entry from `cacheManagerSlotsPerContract.cacheMap`.
- The `enableBatchContractSlotCaching` flag is `true` by default, so this path is always active.
- The throttle manager (`ThrottleManagerImpl`) applies a global requests-per-second and gas-per-second rate limit, but these are shared across all callers and do not prevent a slow, sustained attack accumulating thousands of distinct contractIds over minutes or hours.

### Impact Explanation

Each `CaffeineCache` wrapping a Caffeine `BoundedLocalCache` (maximumSize=1500) consumes several KB of heap even when empty (internal arrays, queues, and counters). At 10–50 KB per instance, accumulating 100,000 distinct contractIds consumes 1–5 GB of heap. Mirror nodes running with typical JVM heap settings (2–8 GB) will experience `OutOfMemoryError` or severe GC pressure, causing service degradation or crash. Because mirror nodes serve the Hedera network's read path (contract calls, state queries), crashing multiple nodes degrades or halts network processing capacity, consistent with the ≥30% node shutdown impact classification.

### Likelihood Explanation

The attack requires no authentication, no privileged access, and no on-chain transactions. The attacker only needs HTTP access to the mirror node's web3 API (`/api/v1/contracts/call` or JSON-RPC endpoint). Hedera entity IDs are 64-bit integers, providing an effectively unlimited supply of distinct contractIds. The rate limiter slows the attack but does not prevent it — at 100 requests/second, 100,000 distinct contractIds are accumulated in ~17 minutes. The attack is repeatable, automatable with a simple script, and leaves no on-chain trace.

### Recommendation

1. **Add an eviction listener to the outer `contractSlotsCache`** that removes the corresponding entry from `cacheManagerSlotsPerContract` when an entry is evicted:
   ```java
   Caffeine.newBuilder()
       .maximumSize(3000)
       .removalListener((contractId, cache, cause) ->
           cacheManagerSlotsPerContract.getCache(contractId.toString()) /* invalidate or remove */)
       .build();
   ```
   Since `CaffeineCacheManager` does not expose a `removeCache(name)` method, consider replacing it with a direct `ConcurrentHashMap<EntityId, com.github.benmanes.caffeine.cache.Cache<ByteBuffer, byte[]>>` with a bounded Caffeine cache as the outer structure and explicit eviction cleanup.

2. **Replace the two-level structure** with a single bounded Caffeine cache keyed by `(EntityId, ByteBuffer)` composite key, eliminating the unbounded `cacheMap` entirely.

3. **Add a size cap or TTL-based cleanup** to `cacheManagerSlotsPerContract` by subclassing `CaffeineCacheManager` to override `getCache()` with an LRU-bounded map of at most N named caches.

### Proof of Concept

```python
import requests, threading

BASE_URL = "http://<mirror-node>:8545"  # or /api/v1/contracts/call

def send_call(contract_id_int):
    # Hedera contract address: shard.realm.num -> EVM address (last 4 bytes = num)
    addr = "0x" + hex(contract_id_int)[2:].zfill(40)
    payload = {
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": addr, "data": "0x06fdde03"}, "latest"],
        "id": contract_id_int
    }
    requests.post(BASE_URL, json=payload, timeout=5)

# Send requests for 100,000 distinct contract addresses
# Each unique address causes a new CaffeineCache entry in cacheMap
threads = []
for i in range(1, 100001):
    t = threading.Thread(target=send_call, args=(i,))
    threads.append(t)
    t.start()
    if i % 100 == 0:
        for t in threads: t.join()
        threads = []

# Expected result: cacheManagerSlotsPerContract.cacheMap grows to 100,000 entries
# Monitor JVM heap: heap usage grows monotonically, GC cannot reclaim cacheMap entries
# Eventually: OutOfMemoryError or severe GC pause causing node unresponsiveness
```
