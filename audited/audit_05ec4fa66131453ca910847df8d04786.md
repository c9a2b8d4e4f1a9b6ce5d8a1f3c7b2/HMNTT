### Title
Unauthenticated Pagination Loop on `/accounts/{id}/rewards` Exhausts DB Connection Pool

### Summary
The `listStakingRewardsByAccountId()` handler in `rest/controllers/accountController.js` issues two DB queries per request (account validation + reward fetch) with no application-level rate limiting. The REST API's default DB connection pool is 10 connections with a 20-second statement timeout. An unauthenticated attacker following `next` pagination links in a tight loop with concurrent connections can saturate the pool, blocking all other REST API users from obtaining DB connections.

### Finding Description

**Code path:** `rest/controllers/accountController.js`, `listStakingRewardsByAccountId`, lines 170–203.

Each invocation executes two sequential DB queries:
- Line 172: `EntityService.isValidAccount(accountId)` — one DB round-trip
- Lines 180–185: `StakingRewardTransferService.getRewards(...)` — one DB round-trip [1](#0-0) 

When `rewards.length === query.limit`, a `next` link is emitted with a shifted timestamp bound (e.g., `gt:<last_timestamp>`): [2](#0-1) 

The DB pool is configured with a default of **10 connections** and a **20-second statement timeout**: [3](#0-2) [4](#0-3) 

**No application-level rate limiting exists in the REST API.** A `grep` across all `rest/**/*.js` files for `rateLimit`, `throttle`, or `rateLimiter` returns only a test utility match. The `server.js` middleware stack contains: `authHandler` (optional Basic Auth for custom limits only), `requestLogger`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, and `handleError` — no rate-limiting middleware: [5](#0-4) 

The throttling code (`ThrottleManagerImpl`, `ThrottleProperties`) is exclusively in the `web3` module and applies only to contract call endpoints, not to the REST API: [6](#0-5) 

**Root cause:** The REST API assumes infrastructure-level rate limiting (e.g., Traefik) will be present, but this is optional and not enforced by the application. The endpoint itself has no concurrency guard, no per-IP limit, and no cap on total pagination depth.

### Impact Explanation

With a pool of 10 connections and 2 queries per request, 5 concurrent paginating clients can hold the entire pool. The 20-second statement timeout means each "wave" of 5 concurrent requests can block the pool for up to 20 seconds. During this window, all other REST API endpoints (transactions, balances, tokens, etc.) that require a DB connection will queue or fail with connection timeout errors (default 20-second connection timeout). A high-volume staking account (e.g., a validator node account on mainnet) may have thousands of reward records, providing hundreds of pages to iterate through, sustaining the attack indefinitely.

### Likelihood Explanation

No privileges, tokens, or special knowledge are required. The attacker only needs a valid account ID with many staking rewards (publicly discoverable via the same API). The attack is trivially scriptable: fetch page 1, extract `links.next`, fetch that URL, repeat — with 5–10 concurrent workers. The attack is repeatable from a single IP or distributed across IPs. Infrastructure-level Traefik rate limiting is present in the Rosetta chart but is not guaranteed to be deployed in front of the REST API in all configurations. [7](#0-6) 

### Recommendation

1. Add application-level rate limiting middleware to the REST API (e.g., `express-rate-limit`) scoped per IP, applied globally before route handlers in `server.js`.
2. Reduce the number of DB queries per request: combine the `isValidAccount` check and the reward fetch into a single query (return empty results if account not found, or use a `LEFT JOIN`).
3. Add a configurable maximum page depth or cursor expiry to prevent indefinite pagination chains.
4. Increase the default `maxConnections` pool size or add a per-route concurrency limiter to bound the blast radius.

### Proof of Concept

```bash
# Step 1: Find a high-volume staking account (e.g., node account 0.0.3)
ACCOUNT="0.0.3"
BASE="https://<mirror-node-host>/api/v1"

# Step 2: Launch 5 concurrent pagination workers
for i in $(seq 1 5); do
  (
    URL="${BASE}/accounts/${ACCOUNT}/rewards?limit=25&order=asc"
    while [ -n "$URL" ]; do
      RESP=$(curl -s "$URL")
      NEXT=$(echo "$RESP" | jq -r '.links.next // empty')
      URL="${BASE}${NEXT}"
    done
  ) &
done
wait

# Result: All 10 DB pool connections are saturated.
# Other REST API requests (e.g., /api/v1/transactions) begin
# returning 500 errors or timing out due to pool exhaustion.
```
