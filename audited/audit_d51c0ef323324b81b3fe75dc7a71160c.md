### Title
Unbounded In-Memory Transaction Accumulation via Unauthenticated `/block` Endpoint

### Summary
The `/block` Rosetta API endpoint is unauthenticated and triggers `FindBetween()`, which accumulates all transactions in a block's timestamp range into a single in-memory slice with no total-row cap. The `maxTransactionsInBlock` truncation occurs only after the full result set is already buffered in memory. An attacker can repeatedly request high-transaction blocks to drive heap allocation and GC pressure well above baseline.

### Finding Description
**Code path:**

1. `rosetta/app/db/db.go:35` — `ConnectToDb()` creates a `DbClient` with `statementTimeout`. If `statementTimeout <= 0` (the zero-value default when unconfigured), `GetDbWithContext()` at `rosetta/app/db/client.go:23-29` returns a DB handle with **no timeout at all**.

2. `rosetta/app/services/block_service.go:56` — The unauthenticated `/block` handler calls `s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` with the full block timestamp range.

3. `rosetta/app/persistence/transaction.go:123-147` — `FindBetween()` runs an unbounded accumulation loop: it fetches `batchSize=2000` rows per iteration and appends every batch to a single `transactions` slice with no total-row ceiling:
   ```go
   transactions = append(transactions, transactionsBatch...)
   // loop continues until len(batch) < 2000
   ```

4. `rosetta/app/services/block_service.go:61-66` — `maxTransactionsInBlock` truncation happens **after** `FindBetween()` returns, meaning the entire result set (potentially millions of rows) is already resident in heap memory before any trimming occurs.

**Root cause / failed assumption:** The design assumes blocks contain a bounded number of transactions, but the DB query layer enforces no such bound. The `statementTimeout` from `ConnectToDb()` limits individual SQL statement duration, not the total number of rows accumulated across loop iterations. There is no per-request row cap, no streaming/cursor-based response, and no authentication gate.

### Impact Explanation
Each `/block` request for a high-transaction block forces the server to allocate a `[]*transaction` slice proportional to the total transaction count in that block's nanosecond range. With correlated subqueries per row (crypto_transfer and staking_reward_transfer aggregations in `selectTransactionsInTimestampRange`), each row is also a non-trivial object. Concurrent requests targeting the same or different large blocks multiply heap pressure, triggering frequent GC cycles. On a live Hedera mainnet mirror node where blocks can contain thousands of transactions, a small number of concurrent requests (e.g., 10–20) is sufficient to sustain >30% heap growth above baseline.

### Likelihood Explanation
The Rosetta API is designed to be publicly accessible (no authentication by spec). Any external party can POST to `/block` with a valid block index or hash. Block identifiers are enumerable from the chain. No rate limiting, no authentication, and no row cap exist at the handler or repository layer. The attack is trivially repeatable with a simple HTTP client loop.

### Recommendation
1. **Enforce a DB-level row cap in `FindBetween()`**: add a hard maximum total-row limit (e.g., equal to `maxTransactionsInBlock`) and return an error or partial result if exceeded, rather than accumulating unboundedly.
2. **Move truncation before full fetch**: pass `maxTransactionsInBlock` into `FindBetween()` and apply `LIMIT` at the SQL level so the DB never returns more rows than the response will use.
3. **Ensure `statementTimeout` is always configured**: in `ConnectToDb()` / `GetDbWithContext()`, treat `statementTimeout <= 0` as a misconfiguration and apply a safe default, rather than silently disabling timeouts.
4. **Add rate limiting** at the HTTP layer for the `/block` endpoint.

### Proof of Concept
```bash
# Identify a block with many transactions (e.g., from public Hedera mirror REST API)
BLOCK_INDEX=<high_tx_block_number>
BLOCK_HASH=<corresponding_hash>

# Hammer the /block endpoint concurrently with no credentials required
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-node>:5700/block \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"block_identifier\": {\"index\": $BLOCK_INDEX, \"hash\": \"$BLOCK_HASH\"}
    }" &
done
wait
# Monitor heap via pprof: GET http://<rosetta-node>:5700/debug/pprof/heap
# Observe alloc_objects and heap_inuse spike >30% above idle baseline
```
