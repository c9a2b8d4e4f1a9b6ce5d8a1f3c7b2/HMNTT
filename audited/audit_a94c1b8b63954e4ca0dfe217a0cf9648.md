### Title
Unauthenticated Thread Pool Exhaustion via Unbounded Concurrent Requests to NFT Allowance Endpoint

### Summary
The `getNftAllowances()` method in `NftAllowanceServiceImpl` executes blocking database operations on Tomcat worker threads with no rate limiting applied in the `rest-java` module. An unprivileged attacker can flood the `/api/v1/accounts/{id}/allowances/nfts` endpoint with concurrent requests that pass in-memory validation instantly but hold Tomcat threads for up to the full 10-second DB statement timeout, exhausting the thread pool and rendering the node unresponsive.

### Finding Description
**Code path:**
- `AllowancesController.getNftAllowances()` — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, lines 57–83
- `NftAllowanceServiceImpl.getNftAllowances()` — `rest-java/src/main/java/org/hiero/mirror/restjava/service/NftAllowanceServiceImpl.java`, lines 21–31

The execution sequence is:
1. `checkOwnerSpenderParamValidity()` (lines 26, 33–54) — pure in-memory logic, completes in microseconds.
2. `entityService.lookup(request.getAccountId())` (line 28) — for `EntityIdNumParameter` inputs (numeric IDs like `0.0.12345`), the `EntityServiceImpl.lookup()` switch at line 31–35 of `EntityServiceImpl.java` returns immediately via `Optional.of(p.id())` with **no database call**.
3. `repository.findAll(request, id)` (line 30) — always issues a database query.

**Root cause:** The `rest-java` module has **no rate limiting** on this endpoint. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists exclusively in the `web3` module and is not wired into `rest-java`. The only filter in `rest-java` is `MetricsFilter`, which only records metrics and does not throttle. The `rest-java` module runs on Tomcat (confirmed by `MetricsFilter` importing `org.apache.catalina.connector.ResponseFacade` and Grafana dashboards tracking `tomcat_threads_busy_threads`), which uses a blocking thread-per-request model.

**Failed assumption:** The `hiero.mirror.restJava.db.statementTimeout = 10000` ms setting is assumed to be sufficient protection. It is not — it only bounds the duration of each held thread, it does not prevent an attacker from holding all threads simultaneously for that full duration.

### Impact Explanation
Tomcat's default maximum thread pool is 200 threads. With a 10-second statement timeout, an attacker sustaining ~20–25 concurrent requests per second per node will keep all 200 threads occupied, causing HTTP 503 / connection refusal for all legitimate users of that mirror node. Since mirror nodes are independent deployments, an attacker targeting multiple nodes simultaneously (e.g., via a botnet or distributed script) can degrade 30%+ of the network's mirror node capacity. All endpoints on the affected `rest-java` instance become unavailable, not just the NFT allowance endpoint.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no brute force. A single attacker with a modest number of concurrent HTTP connections (achievable with tools like `ab`, `wrk`, or `hey`) can sustain the required request rate. The endpoint is publicly documented and reachable. The parameter format is simple: a valid numeric account ID (e.g., `0.0.2`) as the path variable passes `checkOwnerSpenderParamValidity()` instantly and skips the entity DB lookup, making the attack maximally efficient. This is repeatable and automatable.

### Recommendation
1. **Add rate limiting to `rest-java`**: Implement a `bucket4j`-based filter (mirroring the `web3` module's `ThrottleConfiguration`) applied globally or specifically to the `/api/v1/accounts/{id}/allowances/nfts` endpoint. Enforce a per-IP or global requests-per-second cap.
2. **Reduce statement timeout**: Lower `hiero.mirror.restJava.db.statementTimeout` from 10,000 ms to a value appropriate for this query (e.g., 2,000–3,000 ms) to reduce the window during which threads are held.
3. **Consider async/reactive I/O**: Migrating the `rest-java` module from Tomcat blocking threads to a reactive stack (e.g., Spring WebFlux + R2DBC) would eliminate the thread-per-request exhaustion vector entirely.
4. **Add connection-level limits**: Configure Tomcat's `acceptCount` and `maxConnections` and consider deploying an API gateway or reverse proxy (e.g., nginx, Envoy) with per-IP connection limits in front of the service.

### Proof of Concept
```bash
# Precondition: mirror node running rest-java on localhost:8080
# No authentication required. Use a valid numeric account ID (e.g., 0.0.2).

# Step 1: Confirm the endpoint responds normally
curl "http://localhost:8080/api/v1/accounts/0.0.2/allowances/nfts?limit=100"

# Step 2: Flood with concurrent requests to exhaust Tomcat thread pool
# Using 'hey' (https://github.com/rakyll/hey):
hey -n 100000 -c 250 -q 0 \
  "http://localhost:8080/api/v1/accounts/0.0.2/allowances/nfts?limit=100"

# Step 3: In a separate terminal, observe that legitimate requests now hang or return 503:
curl "http://localhost:8080/api/v1/accounts/0.0.3/allowances/nfts"

# Expected result: legitimate requests time out or receive connection refused.
# The attack sustains ~250 concurrent blocking DB queries, exceeding Tomcat's
# default 200-thread pool. Each thread is held for up to 10 seconds (statement timeout).
# Repeat against additional mirror node instances to reach 30%+ network impact.
```
