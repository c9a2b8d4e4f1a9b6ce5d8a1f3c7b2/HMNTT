### Title
Unauthenticated Metrics Endpoint Exposes Per-Route Timing Histogram When Authentication Is Disabled

### Summary
When `hiero.mirror.rest.metrics.config.authentication` is set to `false`, the `authenticate()` function in `metricsHandler.js` unconditionally returns `true`, granting any unauthenticated external user full read access to the Prometheus metrics endpoint. This endpoint exposes `api_request_duration_milliseconds` histogram data broken down by route, HTTP method, and status code, along with in-flight request counts and process-level CPU/memory gauges. An attacker can use this timing distribution to infer database query latency, cache hit rates, and system load patterns, enabling precisely timed transaction submission attacks.

### Finding Description
**File:** `rest/middleware/metricsHandler.js`
**Function:** `authenticate()` ŌĆö lines 137ŌĆō144; `metricsMiddleware()` ŌĆö lines 152ŌĆō165

```js
// lines 137-144
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;          // ŌåÉ unconditional bypass; no credential check at all
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

When `authentication` is falsy, the function short-circuits and returns `true` before ever inspecting the request. The caller at lines 156ŌĆō160 only blocks access when `authenticate()` returns `false`:

```js
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) {
    res.set('WWW-Authenticate', 'Basic realm="Metrics"');
    return res.status(401).send('Unauthorized');
  }
  return exporter.collect().then(({resourceMetrics}) => {
    res.set('Content-Type', 'text/plain; charset=utf-8');
    res.send(serializer.serialize(resourceMetrics));   // full metrics dump
  });
}
```

The metrics endpoint path is `${config.metrics.config.uriPath}/metrics/` (default: `/swagger/metrics/`).

The data serialized and returned includes (lines 91ŌĆō134):
- `api_request_duration_milliseconds` histogram with labels `{method, path, code}` ŌĆö per-route timing buckets (default boundaries: 25 ms, 100 ms, 250 ms, 500 ms)
- `api_all_request_in_processing_total` ŌĆö live in-flight request count
- `nodejs_process_cpu_usage_percentage` ŌĆö process CPU load
- `nodejs_process_memory_*` ŌĆö heap/RSS gauges
- `api_request_size_bytes` / `api_response_size_bytes` ŌĆö per-route payload sizes

**Root cause:** The design treats `authentication = false` as "open to everyone" rather than "deny all unauthenticated access." There is no network-layer or middleware-layer fallback restriction.

**Failed assumption:** The code assumes that when an operator disables authentication, the endpoint is only reachable from trusted internal scrapers (e.g., Prometheus). No such network restriction is enforced in the application itself.

### Impact Explanation
An attacker who can reach the REST API port (default 5551) can poll `/swagger/metrics/` at will and obtain:

1. **Timing side-channel for transaction routes** ŌĆö histogram bucket counts for `/api/v1/transactions/{transactionId}` reveal whether responses are served from cache (sub-25 ms bucket) or require a full DB query (250ŌĆō500 ms bucket). Watching bucket transitions in real time reveals cache warm/cold state.
2. **System load fingerprinting** ŌĆö `api_all_request_in_processing_total` and CPU gauges reveal when the node is under load, allowing an attacker to submit transactions during low-load windows to maximize processing speed or during high-load windows to exploit race conditions.
3. **Cache hit rate inference** ŌĆö comparing `api_all_success` counts against duration bucket distributions over time reveals Redis cache effectiveness, exposing predictable cache-miss windows.
4. **Reconnaissance for DoS timing** ŌĆö response size histograms for expensive routes (e.g., `/api/v1/accounts/{id}`) reveal which queries return large payloads, useful for amplification-style load attacks.

Severity: **MediumŌĆōHigh** (information disclosure enabling targeted follow-on attacks; no direct data exfiltration, but materially reduces attacker effort for timing and load-based attacks).

### Likelihood Explanation
- **Precondition:** `authentication` must be set to `false`. The documented default is `true`, so this requires an explicit operator misconfiguration or a deployment that intentionally disables auth for internal Prometheus scraping without network-level isolation.
- **Feasibility:** Once the precondition is met, exploitation requires only a single unauthenticated HTTP GET ŌĆö no credentials, no special tooling, no rate-limit bypass.
- **Repeatability:** Fully repeatable; the attacker can poll continuously to build a time-series of system behavior.
- **Detection difficulty:** The request hits the metrics path, which is typically excluded from alerting (it is a monitoring endpoint). The `requestLogger` middleware logs it, but at a low sample rate (`log.sampleRate` default 0.1), so 90% of polling requests may not be logged.

### Recommendation
1. **Remove the unconditional bypass.** Change `authenticate()` so that `authentication = false` still requires the request to originate from a trusted source (e.g., localhost or a configured CIDR), rather than returning `true` for everyone:
   ```js
   if (!authentication) {
     // Only allow loopback when auth is disabled
     const ip = req.ip || req.connection.remoteAddress;
     return ip === '127.0.0.1' || ip === '::1';
   }
   ```
2. **Alternatively, remove the `authentication` toggle entirely** and always require credentials, relying on Kubernetes/network policy to restrict scraper access.
3. **Add a rate limit** on the metrics path independent of authentication state.
4. **Document the security implication** of setting `authentication: false` in `docs/configuration.md` with an explicit warning that the endpoint must be network-isolated.
