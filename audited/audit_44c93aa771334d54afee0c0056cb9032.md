### Title
Parallel DB Query Fan-Out in `getOneAccount()` Enables Connection Pool Exhaustion at Half the Request Rate

### Summary
`getOneAccount()` unconditionally fires an entity query and a full `doGetTransactions()` call in parallel via `Promise.all()` when `transactions=true` (the default). Because `doGetTransactions` itself issues a second sequential DB query (`getTransactionTimestamps` → `getTransactionsDetails`), each inbound HTTP request holds up to 2 simultaneous pool connections. With the default pool size of 10, an attacker needs only 5 concurrent unauthenticated requests to saturate the pool, half the rate required against any single-query endpoint. No rate-limiting middleware exists on the REST API.

### Finding Description
**Code location:** `rest/accounts.js`, `getOneAccount()`, lines 487–495.

```js
const entityPromise = pool.queryQuietly(entityQuery, entityParams);          // line 487

const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)             // line 492
  : emptyTransactionsPromise;

const [entityResults, transactionResults] = await Promise.all([entityPromise, transactionsPromise]); // line 495
```

`includeTransactions` defaults to `true` (line 402: `let includeTransactions = true`). `doGetTransactions` (`rest/transactions.js` lines 695–723) first `await`s `getTransactionTimestamps` (issues `pool.queryQuietly` at line 483), then conditionally issues a second `pool.queryQuietly` inside `getTransactionsDetails` (line 662). Because `entityPromise` and `transactionsPromise` are both started before `Promise.all` is awaited, the entity query and the first transaction query run concurrently, consuming 2 pool connections simultaneously per request.

**Root cause / failed assumption:** The design assumes the connection pool is large enough relative to concurrent traffic. The default `maxConnections` is 10 (`docs/configuration.md` line 556). No per-IP or global rate-limiting middleware is registered in `rest/server.js` for this route — confirmed: `authHandler` (`rest/middleware/authHandler.js`) only sets a custom response-size limit for authenticated users; it performs no request-rate throttling. A `grep` across all `rest/**/*.js` files finds zero references to any rate-limit library.

### Impact Explanation
With `maxConnections = 10`, an attacker holding 5 concurrent long-running requests to `GET /api/v1/accounts/<id>` (default `transactions=true`) occupies all 10 pool slots (2 per request). Subsequent legitimate requests queue behind `connectionTimeoutMillis` (default 20 s) and then fail with a `DbError`, producing 500 responses for all users. The `statement_timeout` (default 20 s) bounds how long each slot is held, but an attacker simply re-issues requests as soon as slots free up, maintaining continuous saturation. The impact is service degradation / denial for all REST API consumers — no economic damage, pure griefing.

### Likelihood Explanation
The endpoint is fully public, requires no credentials, and accepts any valid account ID (e.g., `0.0.2` always exists on mainnet). The attacker needs only a standard HTTP client capable of issuing ~5–10 concurrent requests in a tight loop. No special knowledge, tokens, or on-chain interaction is required. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add a rate-limiting middleware** (e.g., `express-rate-limit`) to the REST API, applied globally or specifically to `GET /accounts/:id`, to cap requests per IP per second before they reach the DB pool.
2. **Increase `maxConnections`** or front the Node.js pool with pgBouncer in transaction-pooling mode (the production Helm chart already configures pgBouncer with `max_user_connections: 250` for `mirror_rest`, but the application-level pool default of 10 remains the bottleneck).
3. **Consider sequentializing** the entity and transaction queries (await entity first, then fetch transactions) to halve peak connection usage per request, at the cost of slightly higher latency.
4. **Add a concurrency limiter** (e.g., `p-limit`) around `pool.queryQuietly` calls to cap in-flight DB queries regardless of HTTP concurrency.

### Proof of Concept

const getTransactionsDetails = async (payerAndTimestamps, order) => {
  if (payerAndTimestamps.length === 0) {
    return {rows: []};
  }

  const payerAccountIds = new Set();
  const timestamps = [];
  payerAndTimestamps.forEach((row) => {
    timestamps.push(row.consensus_timestamp);
    payerAccountIds.add(row.payer_account_id);
  });

  const params = [];
  const payerAccountIdsCondition = getQueryWithEqualValues('payer_account_id', params, Array.from(payerAccountIds));
  const timestampsCondition = getQueryWithEqualValues('consensus_timestamp', params, timestamps);
  const outerPayerAccountIdsCondition = 't.' + payerAccountIdsCondition;
  const outerTimestampsCondition = 't.' + timestampsCondition;

  const query = `with c_list as (select consensus_timestamp,
                                        payer_account_id,
                                        ${cryptoTransferJsonAgg} as crypto_transfer_list
                                 from crypto_transfer
                                 where ${payerAccountIdsCondition}
                                   and ${timestampsCondition}
                                 group by consensus_timestamp, payer_account_id),
                      t_list as (select consensus_timestamp,
                                        payer_account_id,
                                        ${tokenTransferJsonAgg} as token_transfer_list
                                 from token_transfer
                                 where ${payerAccountIdsCondition}
                                   and ${timestampsCondition}
                                 group by consensus_timestamp, payer_account_id)
                 select ${transactionFullFields},
                        (select crypto_transfer_list
                         from c_list
                         where consensus_timestamp = t.consensus_timestamp
                           and payer_account_id = t.payer_account_id),
                        (select token_transfer_list
                         from t_list
                         where consensus_timestamp = t.consensus_timestamp
                           and payer_account_id = t.payer_account_id)
                 from transaction as t
                 where ${outerPayerAccountIdsCondition}
                   and ${outerTimestampsCondition}
                 order by t.consensus_timestamp ${order}`;

  return pool.queryQuietly(query, params);
};

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
