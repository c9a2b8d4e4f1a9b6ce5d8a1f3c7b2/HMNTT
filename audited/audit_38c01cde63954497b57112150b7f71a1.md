### Title
Missing File ID Allowlist Validation in REST Java `getNodes()` Allows Arbitrary Address Book Enumeration

### Summary
The REST Java `NetworkController.getNodes()` endpoint accepts a user-supplied `file.id` query parameter but only validates that the operator is `EQ`; it never checks that the value is one of the two permitted file IDs (101 or 102). This allows any unauthenticated caller to query the `address_book` table for any arbitrary file ID, exposing historical node configurations stored under non-standard file IDs. The sibling gRPC path has an explicit allowlist guard that the REST Java path is missing.

### Finding Description

**Code path:**

`NetworkController.getNodes()` (lines 152–157) checks only the operator type:

```java
if (fileId != null && fileId.operator() != RangeOperator.EQ) {
    throw new IllegalArgumentException("Only equality operator is supported for file.id");
}
``` [1](#0-0) 

The request is then forwarded to `NetworkServiceImpl.getNetworkNodes()`, which resolves the file ID via `getAddressBookFileId()`:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    return request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();
}
``` [2](#0-1) 

The raw caller-controlled value is passed directly to `networkNodeRepository.findNetworkNodes()`, whose SQL CTE filters `address_book` with `where file_id = :fileId` — no further restriction. [3](#0-2) 

**Contrast with the gRPC path**, which has an explicit allowlist guard:

```java
if (!getValidFileIds().contains(fileId)) {
    throw new IllegalArgumentException(INVALID_FILE_ID);
}
``` [4](#0-3) 

where `validFileIds` is `{addressBookFile101, addressBookFile102}`. [5](#0-4) 

The REST Java path has no equivalent guard.

### Impact Explanation
An unauthenticated attacker can enumerate the `address_book` table for any file ID ever stored in the Hashgraph history. This leaks historical node configurations (IP addresses, public keys, service endpoints, cert hashes, account IDs) that were associated with non-standard or retired file IDs and were never intended to be publicly queryable via this API. The data returned includes sensitive network topology information.

### Likelihood Explanation
Exploitation requires zero privileges and zero special tooling — a plain HTTP GET request with a crafted `file.id` query parameter suffices. The endpoint is publicly reachable, the parameter is documented in the proto spec as accepting any `FileID`, and the missing check is trivially discoverable by comparing the gRPC and REST behaviors. Repeatability is unlimited.

### Recommendation
Add an allowlist check in `NetworkServiceImpl.getNetworkNodes()` (or in `getAddressBookFileId()`) mirroring the gRPC guard:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    long file101 = systemEntity.addressBookFile101().getId();
    long file102 = systemEntity.addressBookFile102().getId();
    if (id != file101 && id != file102) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return id;
}
```

This mirrors the existing gRPC-layer protection and closes the gap. [2](#0-1) 

### Proof of Concept

```
# Query address book for an arbitrary non-standard file ID (e.g., 0.0.1)
GET /api/v1/network/nodes?file.id=0.0.1

# Query for file 0.0.101 (permitted) vs 0.0.1 (not permitted but accepted)
GET /api/v1/network/nodes?file.id=0.0.101   # intended
GET /api/v1/network/nodes?file.id=0.0.1     # should be rejected, is not

# Any numeric entity ID accepted:
GET /api/v1/network/nodes?file.id=0.0.999
```

No authentication header is required. If any `address_book` rows exist for the supplied `file_id`, they are returned in full, including node IP addresses, public keys, cert hashes, and service endpoints.
