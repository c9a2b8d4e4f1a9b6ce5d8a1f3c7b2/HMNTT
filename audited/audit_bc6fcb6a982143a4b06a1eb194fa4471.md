### Title
Integer Overflow in `PollingTopicListener.poll()` Causes Unbounded Database Query via `limit > Integer.MAX_VALUE`

### Summary
In `PollingTopicListener.poll()`, the expression `(int)(filter.getLimit() - context.getCount().get())` performs an unchecked narrowing cast from `long` to `int`. Any unprivileged gRPC client can set `limit` to a value greater than `Integer.MAX_VALUE` (e.g., `2147483648L`), causing the cast to overflow to a large negative integer. The resulting negative `pageSize` is passed to an internally-built `TopicMessageFilter` whose `hasLimit()` returns `false`, causing `TopicMessageRepositoryCustomImpl.findByFilter()` to execute an unbounded database query on every poll cycle — a repeatable resource-exhaustion DoS.

### Finding Description

**Entry point** — `ConsensusController.toFilter()` ( [1](#0-0) ) reads `query.getLimit()` directly from the protobuf field, which is declared `uint64` ( [2](#0-1) ) and stored as a Java `long`. A value of `2147483648L` is valid, positive, and passes the `@Min(0)` constraint on `TopicMessageFilter.limit`. ( [3](#0-2) )

**Overflow site** — Inside `PollingTopicListener.poll()`:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())  // line 55
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());  // line 57
``` [4](#0-3) 

With `filter.getLimit() = 2147483648L` and `context.getCount().get() = 0L`:
- `(int) 2147483648L` → `-2147483648` (Integer.MIN_VALUE, Java narrowing-cast overflow)
- `Math.min(-2147483648, maxPageSize)` → `-2147483648` (any positive `maxPageSize` loses)

**Validation bypass** — The negative `pageSize` is used to build an internal `newFilter`:

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [5](#0-4) 

This builder call is internal and never passes through a Spring `@Valid`-annotated method, so the `@Min(0)` Bean Validation constraint is never evaluated. `newFilter.getLimit()` is `-2147483648L`.

**Unbounded query** — In `TopicMessageRepositoryCustomImpl.findByFilter()`:

```java
if (filter.hasLimit()) {          // hasLimit() returns limit > 0 → FALSE for -2147483648
    typedQuery.setMaxResults(...); // never called
}
``` [6](#0-5) 

`setMaxResults` is never called, so the JPA query returns every matching row for the topic with no upper bound, on every poll interval.

**Why the outer `flux.take()` does not mitigate this** — `TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` at the Flux level ( [7](#0-6) ) but this only limits how many items are *emitted downstream*. It does not prevent the underlying `findByFilter` call from loading all rows into memory on each poll cycle before the Flux layer can cancel.

### Impact Explanation
Each poll cycle (default interval: configurable, typically 500 ms) executes a full table scan for the topic with no `LIMIT` clause. On a topic with millions of messages, this causes:
- Heap exhaustion / OOM in the mirror node JVM (all rows loaded via `getResultList().stream()`)
- Excessive database load (full sequential scan repeated every poll interval)
- Denial of service for all other subscribers sharing the same JVM and database connection pool

A single malicious subscription is sufficient to trigger this continuously until the connection is dropped.

### Likelihood Explanation
The precondition is trivial: any unauthenticated gRPC client can send `ConsensusTopicQuery` with `limit = 2147483648` (one above `Integer.MAX_VALUE`). No credentials, no special role, no prior knowledge beyond the public proto schema is required. The attack is repeatable and persistent for the lifetime of the subscription. The proto field is `uint64`, so the value is semantically valid and passes all wire-level checks.

### Recommendation
Replace the unsafe narrowing cast with an explicit overflow-safe clamp before casting:

```java
long remaining = filter.getLimit() - context.getCount().get();
// Clamp to [0, Integer.MAX_VALUE] before narrowing
int limit = filter.hasLimit()
        ? (int) Math.min(remaining, Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
```

Additionally, add a `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) constraint on `TopicMessageFilter.limit` to reject oversized values at the API boundary before they propagate into internal logic.

### Proof of Concept

1. Start the mirror node gRPC service.
2. Send a `ConsensusTopicQuery` with a valid `topicID` and `limit = 2147483648` (i.e., `Integer.MAX_VALUE + 1`).
3. Observe in `PollingTopicListener.poll()` that `(int)(2147483648L - 0L)` evaluates to `-2147483648`.
4. Observe `Math.min(-2147483648, maxPageSize)` returns `-2147483648`.
5. Observe the `newFilter` built with `limit = -2147483648`; `hasLimit()` returns `false`.
6. Observe `TopicMessageRepositoryCustomImpl.findByFilter()` executes without `setMaxResults`, returning all rows.
7. Confirm the query repeats on every poll interval for the duration of the subscription, exhausting heap and DB resources.
