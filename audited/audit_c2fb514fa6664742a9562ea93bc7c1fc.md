### Title
Unauthenticated Timestamp Parameter Forces Expensive `DISTINCT ON` Historical Query, Enabling Low-Rate DoS

### Summary
Any unauthenticated external user can supply a `timestamp` query parameter to the public `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint, causing `isHistorical()` to return `true` and redirecting execution to `getHookStorageChange()`. This method executes native SQL queries using `SELECT DISTINCT ON (key)` against the append-only, time-partitioned `hook_storage_change` table — a significantly more expensive operation than the simple primary-key lookup on `hook_storage` used in the non-historical path. With no authentication or rate limiting in place, an attacker can sustain elevated database load at a low request rate, degrading processing capacity.

### Finding Description

**Exact code path:**

1. **Controller entry point** — `HooksController.java` line 104–130: The endpoint is a public `@GetMapping("/{hookId}/storage")` with no authentication annotation. The `timestamps` parameter is `@RequestParam(required = false, defaultValue = "")` with only `@Size(max = 2)` validation. Any caller can supply `?timestamp=lte:9999999999`.

2. **Bound construction** — `HooksController.java` line 186:
   ```java
   final var bound = Bound.of(timestamps, TIMESTAMP, HookStorageChange.HOOK_STORAGE_CHANGE.CONSENSUS_TIMESTAMP);
   ```
   `Bound.of()` (`Bound.java` lines 169–182) returns `Bound.EMPTY` only when the array is null or empty. A single user-supplied timestamp produces a non-empty `Bound` with a populated `lower` or `upper` field.

3. **`isHistorical()` gate** — `HookStorageRequest.java` lines 66–68:
   ```java
   public boolean isHistorical() {
       return !timestamp.isEmpty();
   }
   ```
   `isEmpty()` (`Bound.java` lines 126–128) returns `true` only when both `lower` and `upper` are null. One timestamp parameter is sufficient to make this return `false`, i.e., `isHistorical() == true`.

4. **Branch redirect** — `HookServiceImpl.java` lines 56–58:
   ```java
   if (request.isHistorical()) {
       return getHookStorageChange(request);
   }
   ```

5. **Expensive query** — `HookStorageChangeRepository.java` lines 15–39: Both `findByKeyBetweenAndTimestampBetween` and `findByKeyInAndTimestampBetween` use:
   ```sql
   SELECT DISTINCT ON (key) ...
   FROM hook_storage_change
   WHERE owner_id = :ownerId
     AND hook_id = :hookId
     AND key >= :keyLowerBound AND key <= :keyUpperBound
     AND consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound
   ```
   The `hook_storage_change` table is a historical, append-only table partitioned by `consensus_timestamp` (migration `V2.17.1__add_hooks_support.sql` lines 37–54). `DISTINCT ON (key)` requires the database to sort and deduplicate across potentially many rows and partitions for each key, unlike the non-historical path which performs a direct primary-key lookup on `hook_storage` (PK: `owner_id, hook_id, key`).

**Root cause:** The failed assumption is that the `isHistorical()` branch is only reachable by legitimate callers with a genuine need for historical data. In reality, the timestamp parameter is entirely user-controlled, unauthenticated, and requires no privilege to supply.

**Why existing checks are insufficient:**
- `@Size(max = 2)` only limits the number of timestamp parameters, not their presence.
- `@Max(MAX_LIMIT)` limits result count but not query scan cost — `DISTINCT ON` must still scan all matching rows before applying the limit.
- No `@PreAuthorize`, no rate limiter, no IP throttle, and no connection-pool guard is present in the filter chain (`LoggingFilter`, `MetricsFilter`, `ShallowEtagHeaderFilter` — none enforce rate limits).

### Impact Explanation
Each request with a timestamp parameter forces a `DISTINCT ON (key)` scan over the `hook_storage_change` table across all time partitions within the supplied range. With `?timestamp=lte:9999999999` (effectively unbounded upper), the query scans the entire history for the given `owner_id`/`hook_id`. Sustained at even a modest rate (e.g., tens of concurrent connections), this can saturate database CPU and I/O, degrading the mirror node's ability to serve other queries and process incoming data — consistent with ≥30% processing degradation without requiring brute-force volume.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and zero knowledge beyond the public API schema. The endpoint path and parameter names are documented. Any attacker with HTTP access to the mirror node REST API can reproduce this immediately and sustain it indefinitely. The attack is repeatable, low-cost per request, and difficult to distinguish from legitimate historical queries without behavioral analysis.

### Recommendation
1. **Require authentication or authorization** for the historical (`timestamp`-bearing) query path, or restrict it to privileged callers.
2. **Enforce a maximum timestamp range width** — reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configured threshold (e.g., one day in nanoseconds).
3. **Add per-IP or per-client rate limiting** at the API gateway or Spring Security layer for this endpoint.
4. **Add a query timeout** on the `hook_storage_change` repository queries to bound worst-case DB impact per request.
5. **Require a lower-bound timestamp** when an upper-bound is supplied, preventing unbounded historical scans.
