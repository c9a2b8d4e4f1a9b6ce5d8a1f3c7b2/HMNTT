### Title
Unbounded Duplicate `token.id=eq:X` Parameters Accumulate in IN Clause Without Deduplication

### Summary
`extractTokensRelationshipQuery()` in `rest/controllers/tokenController.js` pushes every `eq`-operator `TOKEN_ID` filter directly into `inConditions` without deduplication. `getInClauseSubQuery()` in `rest/service/tokenService.js` then interpolates all of them verbatim into the SQL IN clause. An unprivileged attacker can supply up to `maxRepeatedQueryParameters` identical `token.id=eq:X` values, causing the database to receive a redundant `IN (X,X,X,…,X)` clause on every request.

### Finding Description

**Code path:**

`rest/controllers/tokenController.js`, `extractTokensRelationshipQuery()`, lines 34–35: [1](#0-0) 

Every filter whose operator is `eq` is unconditionally pushed to `inConditions`:
```js
inConditions.push({key: TokenAccount.TOKEN_ID, operator: filter.operator, value: filter.value});
```
No set-membership check or deduplication is performed.

`rest/service/tokenService.js`, `getInClauseSubQuery()`, lines 78–86: [2](#0-1) 

All entries are pushed into `tokenIdInParams` and then **directly string-interpolated** into the SQL:
```js
return ` and ${TokenAccount.getFullName(TokenAccount.TOKEN_ID)} in (${tokenIdInParams})`;
```
No deduplication, no parameterization of the IN values.

**Root cause:** The failed assumption is that callers will not supply the same `eq` value multiple times. The code never normalises or deduplicates `inConditions` before building the query.

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts/0.0.X/tokens?token.id=eq:5&token.id=eq:5&…` (repeated up to `maxRepeatedQueryParameters` times).
2. `buildFilters()` in `rest/utils.js` (lines 1240–1253) accepts the array as long as its length ≤ `maxRepeatedQueryParameters`. [3](#0-2) 
3. Each duplicate passes through `extractTokensRelationshipQuery()` and lands in `inConditions`.
4. `getInClauseSubQuery()` emits `WHERE … token_id IN (5,5,5,…,5)` — a semantically redundant clause that the database planner must still parse and evaluate.

### Impact Explanation
Each request forces the database to process an IN clause with up to `maxRepeatedQueryParameters` identical literal values. While the query result is identical to `IN (5)`, the repeated literals increase parse time, planning overhead, and query string size proportionally. Because the endpoint is unauthenticated and the IN values are directly interpolated (not bound parameters), query-plan caching is also defeated for each unique repetition pattern. The impact is bounded by `maxRepeatedQueryParameters` per request but is freely repeatable, enabling sustained low-cost DB griefing.

### Likelihood Explanation
No authentication or rate-limiting is required beyond what the API gateway provides. Any external user can craft the request with a standard HTTP client. The attack is trivially scriptable and repeatable at high frequency.

### Recommendation
Deduplicate `inConditions` values before passing them to `getInClauseSubQuery()`. The simplest fix is to use a `Set` when collecting `eq` values in `extractTokensRelationshipQuery()`:

```js
const inValues = new Set();
// inside the TOKEN_ID / eq branch:
inValues.add(filter.value);
// after the loop:
inConditions.push(...[...inValues].map(v => ({key: TokenAccount.TOKEN_ID, operator: opsMap.eq, value: v})));
```

Additionally, the IN clause values should be passed as a PostgreSQL array parameter (`= ANY($N)`) rather than string-interpolated, which also restores query-plan caching.

token.id=eq:5&token.id=eq:5&token.id=eq:5&token.id=eq:5&\
token.id=eq:5&token.id=eq:5&token.id=eq:5&token.id=eq:5&\
