### Title
Unauthenticated `limit=100` Requests Exhaust DB Connection Pool During Degraded-DB Conditions

### Summary
Any unauthenticated external user can supply `limit=100` (the configured maximum) to any paginated REST endpoint. During a database partition or degradation event, each such request holds a `pg` pool connection open for the full `statementTimeout` window (default 20 seconds). With a pool capped at 10 connections, only 10 concurrent requests are needed to exhaust the pool and deny service to all legitimate callers for the duration of the timeout window. The attack is trivially repeatable with no credentials required.

### Finding Description

**Limit validation — `rest/utils.js` lines 330–332:**
`filterValidityChecks` accepts any positive long integer for `filterKeys.LIMIT`:
```js
case constants.filterKeys.LIMIT:
  ret = isPositiveLong(val);
  break;
```
`isPositiveLong` accepts values up to `9223372036854775807`. No upper-bound rejection occurs here.

**Limit capping — `rest/utils.js` lines 544–553 (`getLimitParamValue`):**
```js
const parsed = Number(value);
const maxLimit = getEffectiveMaxLimit();
ret = parsed > maxLimit ? maxLimit : parsed;
```
For an unauthenticated user `getEffectiveMaxLimit()` returns `responseLimit.max` (default **100**). The value is silently capped, not rejected. The resulting SQL query is issued with `LIMIT 100`.

**Pool configuration — `rest/dbpool.js` lines 7–16:**
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20000 ms
max: config.db.pool.maxConnections,                         // default 10
statement_timeout: config.db.pool.statementTimeout,         // default 20000 ms
```
The pool holds at most **10** connections; each query is allowed to run for up to **20 seconds** before the statement timeout fires.

**Exploit flow:**
1. Attacker sends 10 concurrent `GET /api/v1/transactions?limit=100` (or any paginated endpoint) with no credentials.
2. Each request acquires one of the 10 pool connections and issues `SELECT … LIMIT 100` against the degraded DB.
3. The degraded DB stalls; each connection is held open for the full 20-second `statement_timeout`.
4. The pool is exhausted. All subsequent legitimate requests block on `connectionTimeoutMillis` (20 s) and then fail.
5. Attacker re-fires 10 requests every ~20 seconds to sustain the outage indefinitely.

**Why existing checks are insufficient:**
- The limit cap (100) is a ceiling, not a floor; it does not prevent the query from being issued.
- The statement timeout (20 s) is the *only* bound on how long a connection is held, and it is long enough to sustain pool exhaustion with a small number of concurrent requests.
- There is no per-IP rate limiting, no request queue depth limit, and no circuit-breaker visible in the REST layer code.
- The DB-level `statement_timeout` set via `ALTER USER` (also 20 s per `charts/hedera-mirror/templates/secret-passwords.yaml` line 124) matches the pool setting, providing no additional protection.

### Impact Explanation
During any DB degradation event (network partition, replica lag, maintenance), an attacker with no credentials can sustain a complete denial of service against the REST API by holding all 10 pool connections open. Legitimate monitoring, health checks, and user queries all fail. The attack requires only 10 concurrent HTTP requests repeated every 20 seconds — well within the capability of a single machine or a trivial script.

### Likelihood Explanation
No authentication, no rate limiting, and no special knowledge of the system are required. The attacker only needs to know the public API endpoint (documented) and that `limit=100` is the maximum (also documented). The attack is effective only when the DB is already degraded, but that is precisely the moment when availability is most critical. The 10-connection pool and 20-second timeout are default values that many deployments will not change.

### Recommendation
1. **Reduce `statementTimeout`** to the minimum operationally acceptable value (e.g., 5 s) to shrink the window each connection is held during degradation.
2. **Increase `maxConnections`** or implement a request queue with a short wait timeout so pool exhaustion fails fast rather than blocking for 20 s.
3. **Add per-IP or global rate limiting** at the HTTP layer (e.g., via an ingress controller or Express middleware) so a single source cannot saturate the pool.
4. **Implement a circuit breaker** that returns `503` immediately when the pool is exhausted rather than waiting for `connectionTimeoutMillis`.
5. Consider **lowering the default `response.limit.max`** or requiring authentication for values above a lower threshold (e.g., 25) to reduce per-query cost.

### Proof of Concept
```bash
# Requires: curl, GNU parallel or xargs, a degraded/slow DB instance

# Fire 10 concurrent max-limit requests (no credentials needed)
seq 10 | xargs -P10 -I{} \
  curl -s "https://<mirror-node-host>/api/v1/transactions?limit=100" &

# Immediately attempt a legitimate request — expect connection timeout or 503
curl -v "https://<mirror-node-host>/api/v1/transactions?limit=1"

# Repeat the 10-request burst every 19 seconds to sustain pool exhaustion
while true; do
  seq 10 | xargs -P10 -I{} \
    curl -s "https://<mirror-node-host>/api/v1/transactions?limit=100" &
  sleep 19
done
```
Expected result: the legitimate single-row request fails with a connection-pool timeout error or HTTP 503 for as long as the attack loop runs, even though the DB is only degraded (not fully down).
