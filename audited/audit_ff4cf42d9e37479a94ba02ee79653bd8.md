### Title
Reference Equality Bypass: Hex-Encoded `Long.MAX_VALUE` Block Treated as Historical Block, Always Failing DB Lookup

### Summary
`BlockType.of("0x7fffffffffffffff")` successfully parses to a new `BlockType` object with `number = Long.MAX_VALUE`, but `RecordFileServiceImpl.findByBlockType()` uses reference equality (`==`) to detect `BlockType.LATEST`. Because the user-supplied object is a distinct instance, it bypasses the "latest" fast path and falls through to `recordFileRepository.findByIndex(Long.MAX_VALUE)`, a DB query that always returns empty. Any unprivileged caller can trigger this path repeatedly, causing every such request to fail with `BlockNumberNotFoundException` after a wasted DB round-trip.

### Finding Description

**Root cause — reference equality in `RecordFileServiceImpl`:**

```java
// RecordFileServiceImpl.java lines 20-28
if (block == BlockType.EARLIEST) {          // reference equality
    return recordFileRepository.findEarliest();
} else if (block == BlockType.LATEST) {     // reference equality — MISSED for user-supplied value
    return recordFileRepository.findLatest();
} else if (block.isHash()) { ... }

return recordFileRepository.findByIndex(block.number()); // always called for 0x7fffffffffffffff
```

`BlockType.LATEST` is the singleton `new BlockType("latest", Long.MAX_VALUE)`.

**Parsing path in `BlockType.of()`:**

The regex group `GROUP_HEX_NUM` matches `0x([0-9a-f]{1,16})`. The value `0x7fffffffffffffff` (16 hex digits) matches and is parsed:

```java
// BlockType.java lines 73-79
final var hexNum = matcher.group(GROUP_HEX_NUM);
if (hexNum != null) {
    return new BlockType(hexNum, Long.parseLong(hexNum, 16)); // new object, NOT == LATEST
}
```

`Long.parseLong("7fffffffffffffff", 16)` = `Long.MAX_VALUE` = `9223372036854775807`. This succeeds and returns a **new** `BlockType` instance — not the `LATEST` singleton.

**Exploit flow:**

1. Attacker sends an `eth_call` (or `eth_estimateGas`) JSON-RPC request with `"blockNumber": "0x7fffffffffffffff"`.
2. The controller deserializes this via `BlockType.of("0x7fffffffffffffff")` → new `BlockType("7fffffffffffffff", Long.MAX_VALUE)`.
3. `callContract()` sets the lazy block supplier:
   ```java
   // ContractCallService.java lines 103-104
   ctx.setBlockSupplier(Suppliers.memoize(() ->
       recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)));
   ```
4. `RecordFileServiceImpl.findByBlockType()` skips both `EARLIEST` and `LATEST` checks (reference equality fails), skips hash check, and calls `recordFileRepository.findByIndex(Long.MAX_VALUE)`.
5. No record file exists at index `Long.MAX_VALUE` → `Optional.empty()` → `BlockNumberNotFoundException` thrown.
6. The EVM execution is aborted; the request returns an error.

**Why existing checks are insufficient:**

- The throttle manager (`ThrottleManager`) limits gas consumption but does not block structurally invalid block numbers before the DB is hit.
- The `BLOCK_PATTERN` regex (`[0-9a-f]{1,16}`) correctly accepts `7fffffffffffffff` as a valid hex number — no input validation rejects it.
- There is no guard comparing `block.number() == Long.MAX_VALUE` as a value-equality fallback to the LATEST path.

Note: `0xFFFFFFFFFFFFFFFF` (as mentioned in the question) would throw `NumberFormatException` → `IllegalArgumentException` because it exceeds `Long.MAX_VALUE`. The working payload is `0x7fffffffffffffff`.

### Impact Explanation
Every request with `block = "0x7fffffffffffffff"` unconditionally hits the database with a query for a non-existent index and then fails. While the individual DB query is cheap (indexed lookup), an attacker can flood the endpoint with such requests. The throttle mechanism partially mitigates this but does not prevent the bypass itself. The practical impact is griefing: increased DB load, inflated error-rate metrics, and degraded availability for legitimate callers sharing the same throttle bucket — with no economic cost to the attacker.

### Likelihood Explanation
No authentication or special privilege is required. The payload is a single JSON field in a standard `eth_call` request. The pattern is trivially repeatable from any HTTP client. Any public-facing mirror node deployment is exposed.

### Recommendation
Replace reference equality with value-based detection in `RecordFileServiceImpl.findByBlockType()`:

```java
if (block == BlockType.EARLIEST) {
    return recordFileRepository.findEarliest();
} else if (block == BlockType.LATEST || block.number() == Long.MAX_VALUE) {
    return recordFileRepository.findLatest();
}
```

Alternatively, normalize in `BlockType.of()`: if the parsed hex number equals `Long.MAX_VALUE`, return the `LATEST` singleton directly instead of creating a new instance.

### Proof of Concept

```bash
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "data": "0x",
    "to": "0x0000000000000000000000000000000000000001",
    "block": "0x7fffffffffffffff"
  }'
```
