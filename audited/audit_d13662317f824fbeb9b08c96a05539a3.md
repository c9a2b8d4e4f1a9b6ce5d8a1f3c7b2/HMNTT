### Title
Unbounded Subscription Amplification via Missing Global Connection/Subscription Limit in `subscribeTopic()`

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no global limit on the total number of active subscriptions. The only throttle is `maxConcurrentCallsPerConnection = 5`, but there is no limit on the number of TCP connections. An unprivileged attacker can open arbitrarily many connections, each carrying 5 subscriptions, causing each subscription to independently allocate its own backpressure buffer and independently fire historical and safety-check DB queries, exhausting memory and database connection pool resources.

### Finding Description

**Code path and root cause:**

`subscriberCount` at [1](#0-0)  is a Micrometer gauge for observability only — it is never checked against any maximum before accepting a new subscription. [2](#0-1) 

Every call to `subscribeTopic()` unconditionally allocates a new `TopicContext`, a new `historical` DB query Flux, a new `safetyCheck` Flux, and a new `live` pipeline: [3](#0-2) 

The `safetyCheck` Flux fires an independent DB query per subscription after 1 second on `Schedulers.boundedElastic()`: [4](#0-3) 

Each subscriber gets its own `onBackpressureBuffer` (up to `maxBufferSize = 16384` entries) and its own `publishOn` task allocation in `SharedTopicListener.listen()`: [5](#0-4) 

**The only throttle — `maxConcurrentCallsPerConnection = 5` — is per-connection, not global:** [6](#0-5) 

It is applied in `GrpcConfiguration` with no companion limit on the number of connections: [7](#0-6) 

**Exploit flow:**

1. Attacker opens `C` TCP connections to the gRPC port (no connection limit enforced).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streams to the same high-traffic topic with no `endTime` and no `limit` (infinite subscriptions).
3. Total active subscriptions = `5 × C`, each independently:
   - Allocates up to `16384 × sizeof(TopicMessage)` bytes in its own backpressure buffer.
   - Fires an independent historical DB query at subscription start.
   - Fires an independent `missingMessages` DB query after 1 second via `safetyCheck`.
   - Independently processes every live message through its own reactive pipeline stage.

**Why the shared listener does not fully mitigate this:**

`SharedPollingTopicListener` uses `.share()` so the underlying DB poll is shared: [8](#0-7)  However, each subscriber still independently buffers messages and independently fires `historical` and `safetyCheck` queries — these are not shared.

### Impact Explanation

With `C` connections and 5 subscriptions each:
- **Memory**: `5C × 16384 × ~6 KB` (max HCS message size) = gigabytes of buffer at modest connection counts.
- **DB load**: `5C` concurrent historical queries at subscription time, plus `5C` `safetyCheck` queries after 1 second, plus per-message `missingMessages` gap-fill queries — all independent, multiplying DB connection pool pressure.
- **CPU**: `5C` independent reactive pipeline stages processing every live message.

This can exhaust JVM heap, saturate the `boundedElastic` scheduler thread pool, and overwhelm the database connection pool, causing the service to become unavailable for all legitimate subscribers.

### Likelihood Explanation

No authentication or authorization is required to call `subscribeTopic` — it is a public gRPC endpoint. Opening many TCP connections is trivial from a single host. The attacker needs only a valid topic ID (publicly observable on-chain). The attack is repeatable and requires no special knowledge beyond the gRPC proto definition.

### Recommendation

1. **Enforce a global subscription limit**: Check `subscriberCount` against a configurable maximum before accepting a new subscription in `subscribeTopic()`, returning `RESOURCE_EXHAUSTED` if exceeded.
2. **Enforce a per-IP connection limit**: Configure `NettyServerBuilder` with a per-IP connection throttle (e.g., via a Netty `ChannelHandler` or a reverse proxy rule).
3. **Enforce a per-connection subscription limit globally**: Add a `maxConnections` setting alongside `maxConcurrentCallsPerConnection` in `NettyProperties`.
4. **Rate-limit `safetyCheck` DB queries**: Deduplicate or coalesce `missingMessages` queries across subscribers for the same topic rather than firing one per subscriber.

### Proof of Concept

```python
import grpc
import threading
from hiero.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = (0, 0, 12345)  # any valid high-traffic topic
NUM_CONNECTIONS = 200      # 200 connections × 5 streams = 1000 subscriptions

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,  # set TOPIC_ID
        consensusStartTime=Timestamp(seconds=0),
        # no limit, no endTime → infinite subscription
    )
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    # drain streams to keep them alive
    for s in streams:
        threading.Thread(target=lambda st=s: list(st), daemon=True).start()

threads = [threading.Thread(target=flood_connection, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: 1000 concurrent subscriptions, each with independent 16384-entry buffer
# and independent DB queries → OOM / DB pool exhaustion
```

        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });
