### Title
Unauthenticated `from` Field Enables Per-Filter Bucket Exhaustion DoS in `ThrottleManagerImpl`

### Summary
The `from` field in `ContractCallRequest` is fully user-controlled and carries no authentication. When an operator configures a `THROTTLE` action with a `FilterField.FROM` / `FilterType.CONTAINS` filter, any unprivileged caller can craft a request whose `from` address matches the filter expression, consuming tokens from the shared per-filter `Bucket`. By sending requests at or above the bucket's refill rate, the attacker exhausts the bucket and causes every subsequent request whose `from` address legitimately matches the pattern to receive a `ThrottleException`, effectively denying service to those users.

### Finding Description
**Code path:**

- `ThrottleManagerImpl.throttle()` — [1](#0-0)  iterates every configured `RequestProperties` and calls `action()` for each one whose predicate matches the incoming request.
- `action()` — [2](#0-1)  for `THROTTLE` calls `filter.getBucket().tryConsume(1)`; if the bucket is empty it throws `ThrottleException`.
- The per-filter `Bucket` — [3](#0-2)  is created with `capacity = rate` (default 100) and refills at `rate` tokens per second. This single bucket is **shared** across every request that matches the filter, regardless of caller identity.
- `RequestFilter.test()` — [4](#0-3)  extracts `ContractCallRequest::getFrom` and applies `Strings.CI::contains` (case-insensitive substring match) against the configured expression. [5](#0-4) 
- The `from` field — [6](#0-5)  is a plain hex string accepted from the HTTP request body with no signature or ownership verification.

**Root cause:** The system assumes the `from` field reflects the true caller identity. In reality it is arbitrary user input. The per-filter bucket is a single shared resource keyed only on whether the field value contains the expression, so any caller who deliberately sets `from` to a matching value drains the same bucket that legitimate users with naturally matching addresses depend on.

**Failed assumption:** The random-sampling bypass (`action != ActionType.THROTTLE`) is explicitly skipped for `THROTTLE` actions at [7](#0-6) , meaning every matching request unconditionally consumes a token — there is no jitter or per-IP isolation to slow exhaustion.

**Exploit flow:**
1. Operator configures: `action=THROTTLE`, `field=FROM`, `type=CONTAINS`, `expression=<substring>`, `rate=100`.
2. Attacker sends ≥100 POST `/api/v1/contracts/call` requests per second, each with `from` set to any address containing `<substring>` (e.g. `0x000000000000000000000000000000000000<substring>`).
3. The global `rateLimitBucket` (default 500 req/s) and `gasLimitBucket` are consumed but not exhausted by this traffic alone. [8](#0-7) 
4. The per-filter bucket empties within one second.
5. All subsequent requests from legitimate users whose real `from` addresses contain `<substring>` receive `ThrottleException("Requests per second rate limit exceeded")`.
6. The attacker's own "real" work uses a `from` address that does **not** contain `<substring>`, bypassing the filter entirely.

### Impact Explanation
Legitimate users whose Ethereum addresses happen to contain the filtered substring are completely denied access to the `/api/v1/contracts/call` endpoint for as long as the attacker sustains the flood. Because the `from` field is used for EVM simulation (not authentication), the operator cannot distinguish the attacker's spoofed addresses from real ones. The impact is targeted availability loss (DoS) against a specific address pattern, with no fund theft but potential disruption of any application relying on contract-call simulation for that address class.

### Likelihood Explanation
No privileges are required. The attacker needs only HTTP access to the public endpoint and knowledge of (or ability to guess) the configured filter expression — which may be inferred from error responses or operator documentation. The attack is trivially repeatable with a simple HTTP client loop and costs the attacker nothing beyond bandwidth. The default bucket capacity of 100 tokens/second means exhaustion is achievable with a modest request rate well within the global 500 req/s ceiling.

### Recommendation
1. **Do not use unauthenticated request fields as throttle-filter keys for shared buckets.** If `FROM`-based throttling is needed, key the per-filter bucket on the verified network identity (e.g., authenticated API key or source IP), not the user-supplied `from` value.
2. **Per-source-IP sub-buckets:** Introduce a secondary bucket keyed on `X-Forwarded-For` / remote IP so one source cannot exhaust the shared filter bucket.
3. **Isolate attacker impact:** Use a `ConcurrentHashMap<String, Bucket>` keyed on the actual `from` value so exhaustion of one address's bucket does not affect others.
4. **Validate or ignore `from` for throttle purposes:** If the filter intent is to rate-limit a known abusive address, require an authenticated session token that binds the `from` value, or use IP-based throttling instead.

### Proof of Concept
```
# Assume operator config: action=THROTTLE, field=FROM, type=CONTAINS, expression=dead, rate=100

# Step 1 – exhaust the per-filter bucket (run in background)
for i in $(seq 1 200); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "from":"0x000000000000000000000000000000000000dead",
      "to":"0x0000000000000000000000000000000000000001",
      "gas":21000,
      "data":"0x"
    }' &
done
wait

# Step 2 – legitimate user with a real address containing "dead" is now throttled
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{
    "from":"0xabc000000000000000000000000000000000dead",
    "to":"0x0000000000000000000000000000000000000001",
    "gas":21000,
    "data":"0x"
  }'
# Expected response: 429 / "Requests per second rate limit exceeded"

# Step 3 – attacker's own requests use a non-matching from and succeed normally
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{
    "from":"0x0000000000000000000000000000000000001234",
    "to":"0x0000000000000000000000000000000000000001",
    "gas":21000,
    "data":"0x"
  }'
# Expected response: 200 OK
```
