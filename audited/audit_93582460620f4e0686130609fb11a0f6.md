### Title
Unauthenticated Mirror Node Response Allows Targeted Network Map Manipulation in `fetchMirrorNodeNetwork()`

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` builds the entire SDK node network map by blindly trusting the JSON response from the configured mirror REST endpoint, with no integrity verification, no minimum node-count enforcement, and no cross-validation against an expected node set. An attacker who can influence the HTTP response (MITM on plaintext HTTP, DNS poisoning, or BGP hijacking) can serve a crafted `nodesEnvelope` containing only a targeted subset of real nodes, causing the Hiero SDK client to route all subsequent transactions exclusively to those nodes.

### Finding Description
**Code path:**

`sdk_client.go` `newClient()` (line 18–22) calls `buildNetworkFromMirrorNodes()` → `fetchMirrorNodeNetwork()` (`mirror_node_client.go` lines 74–131) when `cfg.network == "other"`. [1](#0-0) 

Inside `fetchMirrorNodeNetwork()`, the HTTP client is constructed with no TLS pinning and no transport-level integrity controls: [2](#0-1) 

The response body is decoded directly into `nodesEnvelope` and iterated. The only per-entry checks are:

1. `n.NodeAccountID == ""` → skip (line 103)
2. `hiero.AccountIDFromString(n.NodeAccountID)` must parse (line 107–110)
3. `host != ""` and `ep.Port != 0` (lines 118–119) [3](#0-2) 

The sole post-loop guard is: [4](#0-3) 

This is trivially satisfied by including even a single valid entry. There is **no check** that the returned set covers the full expected node roster, no minimum cardinality threshold, and no signature/HMAC on the response. The resulting `network` map is passed verbatim to `hiero.ClientForNetwork(netmap)`: [5](#0-4) 

**Root cause:** The function assumes the mirror node REST endpoint is a trusted, authenticated source. No such trust is established in code.

**Exploit flow:**
1. Attacker positions themselves to intercept or replace the HTTP response (see Likelihood section).
2. Attacker crafts a `nodesEnvelope` JSON body containing only the `NodeAccountID` values of the targeted nodes (e.g., `0.0.3`, `0.0.4`) with their real or attacker-controlled `service_endpoints`.
3. `hiero.AccountIDFromString()` succeeds for all entries (they are syntactically valid).
4. `fetchMirrorNodeNetwork()` returns a map with only those entries; `len(network) > 0` passes.
5. `hiero.ClientForNetwork(netmap)` configures the SDK to send all transactions exclusively to those nodes.

### Impact Explanation
All SDK-level `CryptoTransfer` transactions submitted by the pinger are routed only to the targeted subset of consensus nodes. Depending on attacker goals:

- **Traffic concentration / DoS:** Funneling all pinger traffic (and potentially other clients sharing the same mirror node) to ≥30% fewer nodes degrades or overloads those nodes without brute-force.
- **Traffic interception:** If the attacker also controls the `service_endpoints` values in the crafted response, gRPC connections go to attacker-controlled hosts, enabling transaction interception and operator key exposure via replayed challenges.
- **Monitoring evasion:** Transactions never reach non-targeted nodes, breaking health-check coverage for those nodes.

### Likelihood Explanation
- **HTTP deployments (highest risk):** `mirrorRest` is operator-configured with no enforcement of HTTPS. Internal/dev/staging deployments commonly use `http://`. A network-adjacent attacker (same subnet, cloud VPC, or Kubernetes namespace) can perform ARP/ICMP redirect MITM with zero privileges.
- **DNS poisoning:** An external attacker with access to the DNS resolver used by the pinger container can redirect the mirror node hostname to an attacker-controlled server. No credentials required.
- **BGP hijacking:** For internet-facing mirror nodes, prefix hijacking is a known, documented attack requiring no target-side privileges.
- The attack is **repeatable**: `buildNetworkFromMirrorNodes` is called once at startup; a single poisoned response permanently misconfigures the client for the lifetime of the process.

### Recommendation
1. **Enforce HTTPS with certificate verification** for `mirrorRest`; reject `http://` URLs at config validation time.
2. **Validate response cardinality:** Require the returned network map to contain at least a configurable minimum number of nodes (e.g., `minNodes` config option); fail hard if the count falls below it.
3. **Cross-validate against a pinned node set:** Accept an optional operator-supplied list of expected `NodeAccountID` values; reject any response that omits entries from that list.
4. **Add response signing:** Mirror node responses should be signed (e.g., JWS); the pinger should verify the signature before consuming the payload.

### Proof of Concept
```bash
# 1. Start a fake mirror node returning only node 0.0.3
cat > fake_mirror.json <<'EOF'
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        {"ip_address_v4": "35.237.200.180", "port": 50211}
      ]
    }
  ],
  "links": {"next": null}
}
EOF

python3 -m http.server 8080 &   # serves fake_mirror.json at /api/v1/network/nodes

# 2. Configure pinger to use the fake mirror node (HTTP, no auth)
export HIERO_MIRROR_PINGER_NETWORK=other
export HIERO_MIRROR_PINGER_REST=http://attacker-controlled-host:8080

# 3. Start pinger — fetchMirrorNodeNetwork() fetches the crafted response,
#    hiero.AccountIDFromString("0.0.3") succeeds,
#    len(network)==1 passes the only guard,
#    ClientForNetwork routes ALL transactions to node 0.0.3 only.
./pinger
# All CryptoTransfer transactions now go exclusively to 0.0.3.
```
