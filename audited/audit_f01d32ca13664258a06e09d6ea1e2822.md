### Title
FIFO Pre-Creation at `/tmp/ready` Causes Pinger Deadlock While Readiness Probe Reports Healthy

### Summary
An unprivileged user with write access to `/tmp` can pre-create `/tmp/ready` as a named pipe (FIFO). When the pinger process starts, `os.WriteFile("/tmp/ready", ...)` opens the FIFO for writing and blocks indefinitely (no reader), deadlocking the main goroutine. Simultaneously, the Kubernetes readiness probe calls `os.Stat("/tmp/ready")`, which succeeds immediately on a FIFO, causing the pod to be reported as ready while it is actually deadlocked and submitting no transfers.

### Finding Description

**Code path 1 — readiness healthcheck** (`pinger/cmd/healthcheck/main.go`, `main()`, line 20):
```go
if _, err := os.Stat("/tmp/ready"); err != nil {
    fmt.Fprintln(os.Stderr, "not ready")
    os.Exit(1)
}
os.Exit(0)
```
`os.Stat` calls the `stat(2)` syscall, which **never blocks on FIFOs** — it returns immediately with file metadata. If `/tmp/ready` is a FIFO, `err` is `nil`, and the probe exits 0 (healthy). There is no check on `fi.Mode()&os.ModeNamedPipe`.

**Code path 2 — pinger startup** (`pinger/main.go`, `main()`, line 47):
```go
if err := os.WriteFile("/tmp/ready", []byte("ok\n"), 0o644); err != nil {
    log.Fatalf("failed to create readiness file /tmp/ready: %v", err)
}
```
`os.WriteFile` internally calls `os.OpenFile(name, O_WRONLY|O_CREATE|O_TRUNC, perm)`. On Linux, `open(2)` on a FIFO with `O_WRONLY` and without `O_NONBLOCK` **blocks indefinitely** until a reader opens the other end. No reader ever does, so the main goroutine hangs here permanently. `log.Fatalf` is never reached, so no error is logged.

**Why the liveness probe also passes:** The heartbeat goroutine (lines 28–39 of `pinger/main.go`) is launched *before* the blocking `os.WriteFile` call. It continues writing `/tmp/alive` every 15 seconds, so the liveness probe also reports healthy. The pod appears both live and ready while doing nothing.

**Root cause:** `os.Stat` does not distinguish between regular files and special files (FIFOs, sockets, devices). The code assumes the existence of `/tmp/ready` implies the pinger successfully initialized, but a FIFO satisfies the existence check without requiring the pinger to have written it.

### Impact Explanation
The pinger pod is marked ready by Kubernetes and receives no restart/remediation. The main goroutine is permanently blocked; no Hiero transfers are submitted for the lifetime of the pod. The liveness probe also passes, so Kubernetes never kills and restarts the pod. The griefing is silent and persistent until the pod is manually restarted or the attacker's access is revoked. No economic damage occurs to the network, but monitoring/pinging functionality is completely disabled.

### Likelihood Explanation
**Precondition:** The attacker needs code execution inside the pod as UID 1000 (e.g., via RCE in the pinger binary, a compromised dependency, or `kubectl exec` access). The `/tmp` volume is a pod-specific `emptyDir`, so host-level `/tmp` access is insufficient. Once inside the pod, the attack is a single command (`mkfifo /tmp/ready`) and requires no elevated privileges. The `emptyDir` is cleared on pod restart, but an attacker with persistent exec access can re-create the FIFO after each restart. The attack is repeatable and leaves no obvious log trace (no error is printed).

### Recommendation
1. **Check file type after `os.Stat`** in `pinger/cmd/healthcheck/main.go`:
   ```go
   fi, err := os.Stat("/tmp/ready")
   if err != nil || (fi.Mode()&os.ModeType) != 0 {
       // not a regular file
       os.Exit(1)
   }
   ```
2. **Use `O_NONBLOCK` when writing the readiness file** in `pinger/main.go`, or check whether the path is a regular file before writing:
   ```go
   if fi, err := os.Lstat("/tmp/ready"); err == nil && (fi.Mode()&os.ModeType) != 0 {
       os.Remove("/tmp/ready")
   }
   ```
3. **Set a write deadline/timeout** around `os.WriteFile("/tmp/ready", ...)` using a goroutine with a channel, so a blocked open is detected and causes a fatal error rather than a silent hang.

### Proof of Concept

**Precondition:** Attacker has shell access inside the pinger pod (UID 1000).

```bash
# Step 1: Before or during pinger startup, create /tmp/ready as a FIFO
mkfifo /tmp/ready

# Step 2: Pinger starts (or is restarted). Observe it hangs silently:
# pinger/main.go line 47: os.WriteFile("/tmp/ready", ...) blocks forever.
# No log output after "pinger starting" and config/client init.

# Step 3: Verify readiness probe reports healthy:
/healthcheck ready
echo $?   # prints 0 — pod reported as ready

# Step 4: Verify liveness probe also reports healthy:
/healthcheck live
echo $?   # prints 0 — heartbeat goroutine is still running

# Step 5: Confirm no transfers are being submitted:
# No "Starting transfer ticker" log line is ever printed.
# No Hiero transactions appear on-chain from this pinger.
```

The pod remains in `Running/Ready` state indefinitely with zero transfer activity.
