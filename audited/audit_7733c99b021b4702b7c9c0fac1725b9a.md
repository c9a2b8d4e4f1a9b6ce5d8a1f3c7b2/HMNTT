### Title
Log Flooding via Unbounded Subscription Cycling in `PollingContext.onComplete()`

### Summary
Every gRPC topic subscription — regardless of how quickly it is opened and closed — unconditionally emits multiple `log.info()` calls through `PollingContext.onComplete()` and `TopicContext.finished()`. There is no per-client connection rate limit or subscription-cycling throttle, so an unprivileged attacker opening and immediately cancelling subscriptions at high volume can saturate the logging subsystem, burying legitimate operator signals in noise.

### Finding Description

**Code locations:**

- [1](#0-0)  — `PollingContext.onComplete()` emits one `log.info(...)` per retriever-level subscription termination (cancel **or** complete), wired unconditionally via:
- [2](#0-1)  — `.doOnCancel(context::onComplete).doOnComplete(context::onComplete)`
- [3](#0-2)  — `TopicContext.finished()` emits a second `log.info(...)` on every `SignalType` (cancel, complete, error) via `.doFinally(topicContext::finished)`
- [4](#0-3)  — A third `log.info("Subscribing to topic: {}", filter)` fires at subscription start

Each open→cancel cycle therefore produces **3 unconditional `log.info()` entries** with no guard, throttle, or deduplication.

**Existing check — insufficient:**
The only visible rate control is `maxConcurrentCallsPerConnection = 5` (default): [5](#0-4) 

This limits concurrent streams **per TCP connection** but imposes no limit on the number of connections an attacker may open, no per-IP connection cap, and no minimum subscription lifetime. An attacker opens N connections × 5 concurrent streams, cancels them immediately, and repeats — the per-connection cap does not bound the aggregate log rate.

### Impact Explanation
Sustained log flooding at INFO level fills log buffers and disk, buries legitimate error/warning signals, and can cause synchronous logging backends to introduce latency into the gRPC serving path. Operators lose real-time observability of the node's health. No economic damage occurs, but incident response and debugging are severely degraded — consistent with the stated Medium/griefing scope.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. A single attacker with a modest number of TCP connections (e.g., 200 connections × 5 streams = 1 000 concurrent subscriptions, each cycled every few hundred milliseconds) can produce tens of thousands of `log.info()` lines per second. No special privileges, tokens, or on-chain resources are required.

### Recommendation
1. **Rate-limit subscription opens per source IP** at the gRPC interceptor layer (e.g., a token-bucket interceptor).
2. **Downgrade terminal log calls to `log.debug()`** in `PollingContext.onComplete()` and `TopicContext.finished()`, reserving `log.info()` for subscriptions that actually delivered messages (`total.get() > 0`).
3. **Add a global connection limit per remote address** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionsPerIp` or an equivalent Netty handler.
4. **Enforce a minimum subscription lifetime** before the completion log is emitted, or use a rate-limited logger (e.g., SLF4J's `RateLimitingLogger`).

### Proof of Concept
```python
import grpc, threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 200   # each allowed 5 concurrent streams
CYCLES = 10_000

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1)
    )
    for _ in range(CYCLES):
        call = stub.subscribeTopic(query)
        call.cancel()   # triggers doOnCancel -> onComplete() -> log.info x3

threads = [threading.Thread(target=flood) for _ in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: up to 200 * 10_000 * 3 = 6_000_000 log.info() lines emitted
```
