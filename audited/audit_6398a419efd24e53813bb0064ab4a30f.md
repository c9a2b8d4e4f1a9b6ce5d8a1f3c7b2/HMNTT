### Title
Unbounded `spender.id=eq` Filter Count Enables Unauthenticated DB Resource Exhaustion via Three-Subquery UNION ALL in `getNfts()`

### Summary
The `getNfts()` function in `rest/service/nftService.js` builds up to three `UNION ALL` subqueries, each of which appends an unbounded `spender IN (...)` clause derived from an uncapped list of `spender.id=eq:X` query parameters. With no rate limiting and a `statement_timeout` of 20,000 ms, an unauthenticated attacker can flood the endpoint with maximally complex queries, holding database connections for up to 20 seconds each and saturating CPU — a condition that is severely amplified during a partition recovery when the database is already resource-constrained.

### Finding Description

**Code path:**

`accountController.js` `validateFilters()` (lines 18–32) validates `spenderIdFilters` (range operators) but performs **no validation whatsoever on `spenderIdInFilters`** — the list of `spender.id=eq:X` values: [1](#0-0) 

`eq`-operator spender filters are pushed into `spenderIdInFilters` without any count cap: [2](#0-1) 

`getQuery()` in `nftService.js` produces up to three subqueries (lower / inner / upper), passing the full `spenderIdInFilters` array into **each** subquery: [3](#0-2) 

`getSubQuery()` appends every element of `spenderIdInFilters` as a `spender IN ($5, $6, ..., $N)` condition to each subquery: [4](#0-3) 

The final UNION ALL query is assembled with no additional guard: [5](#0-4) 

The `statement_timeout` is set to 20,000 ms at the pool level — the only DB-side protection: [6](#0-5) 

No rate limiting exists on the REST layer for this endpoint: [7](#0-6) 

**Root cause:** The failed assumption is that `spender.id=eq` repetitions are implicitly bounded. They are not. The validation gate in `validateFilters` only guards against multiple range operators (`lt/lte/gt/gte`) for `spender.id`, leaving the `eq` accumulator (`spenderIdInFilters`) completely uncapped. Combined with the three-subquery UNION ALL path (triggered by a token.id range + serialnumber boundary), each HTTP request can inject an arbitrarily large `IN (...)` list into three independent subqueries, each of which the planner must evaluate independently.

### Impact Explanation

During normal operation, the `statement_timeout=20000ms` cap limits damage per query. During a partition recovery — when replica lag is high, autovacuum is catching up, and buffer cache is cold — even queries that would normally resolve in milliseconds can approach the 20-second ceiling. An attacker sending concurrent requests, each with 3 subqueries × N spender IDs, can:

- Exhaust the database connection pool (`max` connections configured in `rest/dbpool.js`)
- Saturate CPU on the recovering replica, delaying WAL replay and prolonging the outage
- Cause legitimate queries to queue behind 20-second-timeout slots, effectively making the API unavailable

No authentication is required. The endpoint `/api/v1/accounts/{id}/nfts` is fully public.

**Severity: High** — unauthenticated, repeatable, amplified by the exact failure mode described (partition recovery).

### Likelihood Explanation

The attacker needs only:
1. Knowledge of the public API (documented in `openapi.yml`)
2. A valid account ID (any account with NFTs, trivially discoverable)
3. The ability to send HTTP GET requests

The exploit is fully scriptable, requires no credentials, and is repeatable at will. The three-subquery path is documented behavior (the OpenAPI spec describes range filtering), so the trigger conditions are not obscure. [8](#0-7) 

### Recommendation

1. **Cap `spenderIdInFilters` count**: In `extractNftMultiUnionQuery()` or `validateFilters()`, enforce a maximum number of `spender.id=eq` values (e.g., ≤ 10), throwing `InvalidArgumentError` if exceeded — consistent with how other repeated parameters are handled elsewhere in the codebase.

2. **Add per-IP or per-endpoint rate limiting** at the application or ingress layer for the `/accounts/{id}/nfts` endpoint.

3. **Reduce `statement_timeout`** for this specific query class, or introduce a query-complexity budget (e.g., cap the total number of bind parameters).

4. **Ensure indexes** on `(account_id, token_id, serial_number, spender)` exist so that even worst-case queries use index scans rather than sequential scans.

      accountIdCondition,
      ...filters.map((filter) => {
        params.push(filter.value);
        const column = NftService.columns[filter.key];

    }

  getQuery(query) {
    const {lower, inner, upper, order, ownerAccountId, limit, spenderIdInFilters, spenderIdFilters} = query;
    const params = [ownerAccountId, limit];
    const accountIdCondition = `${Nft.ACCOUNT_ID} = $1`;
    const limitClause = super.getLimitQuery(2);
    const orderClause = super.getOrderByQuery(
      ...NftService.orderByColumns.map((column) => OrderSpec.from(column, order))
    );

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) =>
        this.getSubQuery(
          filters,
          params,
          accountIdCondition,
          limitClause,
          orderClause,
          spenderIdInFilters,
          spenderIdFilters
        )
      );

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = this.getSubQuery(
        [],
        params,
        accountIdCondition,
        limitClause,
        orderClause,
        spenderIdInFilters,
        spenderIdFilters
      );
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
    }

    return {sqlQuery, params};
  }
