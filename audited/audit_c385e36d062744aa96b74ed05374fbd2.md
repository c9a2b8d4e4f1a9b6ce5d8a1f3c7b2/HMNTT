### Title
Unbounded Non-Terminating gRPC Stream via Non-Existent Topic When `checkTopicExists` Is Disabled

### Summary
When `hiero.mirror.grpc.checkTopicExists` is set to `false`, any unauthenticated external caller can invoke `subscribeTopic()` with an arbitrary non-existent `topicID` and no `endTime`, causing the server to open a Reactive stream that never emits and never terminates. Each such call holds a live server-side subscription, a Reactor scheduler thread slot, and a gRPC HTTP/2 stream open indefinitely. An attacker repeating this across many TCP connections can exhaust server resources and deny service to legitimate subscribers.

### Finding Description

**Code path and root cause:**

In `TopicMessageServiceImpl.subscribeTopic()`, the existence check is gated behind the `checkTopicExists` flag: [1](#0-0) 

When the topic is not found in the repository and `checkTopicExists=false`, the method returns a synthetic `Entity` of type `TOPIC` instead of raising `EntityNotFoundException`. This allows the subscription pipeline to proceed as if the topic were valid.

The main flux pipeline is: [2](#0-1) 

The termination signal for the live portion comes from `pastEndTime()`: [3](#0-2) 

When no `endTime` is provided (the attacker omits it), `pastEndTime()` returns `Flux.never()`. The `takeUntilOther(Flux.never())` call therefore never fires. `isComplete()` also always returns `false` when `endTime == null`: [4](#0-3) 

The `live` flux calls `topicListener.listen(newFilter)` for a non-existent topic — it never emits. The `safetyCheck` fires once after 1 second, queries the DB (returns empty for a non-existent topic), and completes. The merged flux then sits idle forever, holding the subscription open.

**The test suite explicitly confirms this behavior:** [5](#0-4) 

The test uses `expectNoEvent(WAIT).thenCancel()` — meaning the stream produces nothing and requires external cancellation to terminate.

**Controller layer — no authentication, no per-client limit:** [6](#0-5) 

Any unauthenticated caller can invoke this. The only server-side limit is `maxConcurrentCallsPerConnection = 5`: [7](#0-6) 

This is a **per-connection** cap, not a global cap. An attacker opens many TCP connections, each carrying 5 zombie streams.

The nginx proxy sets a 600-second read timeout for `subscribeTopic`: [8](#0-7) 

Each zombie stream lives for up to 10 minutes before the proxy closes it, but the attacker can continuously replenish them.

**The default value of `checkTopicExists` is `true`:** [9](#0-8) 

However, the documentation explicitly exposes this as a configurable operator option: [10](#0-9) 

Any deployment that sets this to `false` is fully exposed.

### Impact Explanation
Each zombie subscription consumes: one gRPC HTTP/2 stream, one Reactor `boundedElastic` thread slot (for the safety check), one `topicListener` subscription, and one entry in the `subscriberCount` gauge. With enough connections (each carrying 5 streams), the attacker saturates the bounded elastic scheduler, exhausts file descriptors, and starves legitimate subscribers. The `GrpcHighFileDescriptors` and `GrpcHighDBConnections` alerts confirm the operators are aware these are real resource ceilings: [11](#0-10) 

Severity: **High** — full denial of service against the gRPC topic subscription service with no authentication required.

### Likelihood Explanation
Preconditions: operator has set `checkTopicExists=false` (a documented, supported configuration). The attacker needs no credentials, no special knowledge beyond the public gRPC proto schema, and no privileged network position. The attack is trivially scriptable: open TCP connections in a loop, send `subscribeTopic` with a random non-existent `topicID` and no `endTime`, never close the stream. The 600-second proxy timeout means each wave of connections occupies resources for 10 minutes, requiring only ~6 requests/minute per connection to maintain saturation.

### Recommendation

1. **Enforce a server-side stream timeout regardless of `checkTopicExists`**: Add a `timeout()` operator to the flux pipeline in `TopicMessageServiceImpl.subscribeTopic()` so that streams with no activity are terminated after a configurable idle period.
2. **Add a global concurrent-subscription cap**: Track `subscriberCount` (already instrumented) and reject new subscriptions above a configured maximum, returning `RESOURCE_EXHAUSTED`.
3. **Deprecate or remove `checkTopicExists=false`**: The only documented purpose of this flag is to allow subscriptions to topics that may not yet exist. A safer alternative is to allow the subscription but terminate it with `NOT_FOUND` after a short grace period if the topic never appears.
4. **Add per-source-IP connection rate limiting** at the ingress/proxy layer.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc
from com.hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
NUM_CONNECTIONS = 50
STREAMS_PER_CONNECTION = 5  # matches maxConcurrentCallsPerConnection

def open_zombie_streams(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=999999999)
        # No consensusEndTime set — stream never terminates
    )
    streams = []
    for _ in range(STREAMS_PER_CONNECTION):
        # Each call returns an iterator that blocks indefinitely
        it = stub.subscribeTopic(query)
        streams.append(it)
    # Hold streams open forever
    import time; time.sleep(86400)

threads = [threading.Thread(target=open_zombie_streams, args=(i,))
           for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
# Result: 50 * 5 = 250 zombie streams held open on the server,
# each consuming scheduler threads, listener subscriptions, and
# file descriptors, with no server-side termination.
```

**Preconditions**: Server configured with `hiero.mirror.grpc.checkTopicExists=false`. No authentication required. Attacker has network access to port 5600 (or 8080 via nginx proxy).

**Trigger**: Send `subscribeTopic` with a non-existent `topicID` and no `consensusEndTime`.

**Result**: Server-side Reactor flux never completes; `subscriberCount` gauge climbs; file descriptors, scheduler threads, and listener slots are exhausted; legitimate subscribers receive `RESOURCE_EXHAUSTED` or connection timeouts.

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))

                                        .build()))

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
