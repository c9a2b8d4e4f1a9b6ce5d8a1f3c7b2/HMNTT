### Title
Circular Symlink at `/tmp/alive` Permanently Breaks Liveness Probe via Unhandled `ELOOP` in `os.Stat`

### Summary
In `pinger/cmd/healthcheck/main.go`, the liveness check calls `os.Stat("/tmp/alive")`, which follows symlinks. Any process with write access to the `/tmp` emptyDir volume can replace `/tmp/alive` with a self-referential symlink, causing `os.Stat` to return `ELOOP`. The error is treated identically to a missing file, causing the liveness probe to permanently fail. Compounding this, the heartbeat writer in `pinger/main.go` silently discards the `os.WriteFile` error, so it cannot recover the file once the symlink is in place.

### Finding Description
**Code path:**

- `pinger/cmd/healthcheck/main.go`, `main()`, lines 27–32: `os.Stat("/tmp/alive")` follows symlinks; any non-nil error causes `os.Exit(1)`.
- `pinger/main.go`, line 36: `_ = os.WriteFile("/tmp/alive", ...)` — the error return is explicitly discarded with `_`, so a failed write (e.g., due to ELOOP) is silently swallowed.
- `charts/hedera-mirror-pinger/templates/deployment.yaml`, lines 43–44, 51–53: `/tmp` is an `emptyDir` volume, world-writable within the pod.
- `pinger/Dockerfile`, line 42: container runs as `USER 1000:1000`.

**Root cause:** `os.Stat` resolves symlinks before stat-ing. A circular symlink (`/tmp/alive → /tmp/alive`) causes the kernel to return `ELOOP` after the symlink-follow limit is hit. The healthcheck treats every non-nil error from `os.Stat` as "not alive" with no distinction between `ENOENT`, `ELOOP`, or any other error. Once the symlink is planted, `os.WriteFile` also fails with `ELOOP` but the error is discarded, so the legitimate heartbeat file is never restored.

**Exploit flow:**
1. Attacker gains any write capability to `/tmp` inside the container (same UID 1000, sidecar, or init-container escape).
2. `rm /tmp/alive && ln -sf /tmp/alive /tmp/alive` (or equivalent via a two-hop loop).
3. Every subsequent `healthcheck live` invocation calls `os.Stat("/tmp/alive")`, gets `ELOOP`, prints the error, and exits 1.
4. The heartbeat goroutine calls `os.WriteFile("/tmp/alive", ...)` every 15 s, which also fails with `ELOOP`; the `_` discard means the symlink is never replaced.
5. Kubernetes liveness probe fails on every poll → pod is killed and restarted in a continuous loop.

### Impact Explanation
The pod enters a permanent crash-restart loop. The pinger service — responsible for submitting Hedera network transactions and monitoring mirror-node liveness — is rendered unavailable for as long as the symlink persists across restarts (emptyDir is cleared on pod restart, so the attack must be re-applied after each restart, but it is trivially repeatable). No data is exfiltrated and no funds are at risk; the impact is pure availability griefing.

### Likelihood Explanation
The precondition is write access to `/tmp` inside the container. The container runs as a single non-root UID (1000) with no read-only filesystem enforcement on `/tmp`. Any secondary process sharing the pod (sidecar, debug container, or a compromised dependency) running as the same UID can perform the attack. The attack is one shell command, requires no privileges, and is repeatable after every pod restart.

### Recommendation
1. **Use `os.Lstat` instead of `os.Stat`** to avoid following symlinks, then explicitly reject non-regular files:
   ```go
   fi, err := os.Lstat("/tmp/alive")
   if err != nil || fi.Mode()&os.ModeSymlink != 0 || !fi.Mode().IsRegular() {
       fmt.Fprintln(os.Stderr, "invalid or missing heartbeat file")
       os.Exit(1)
   }
   ```
2. **Handle `os.WriteFile` errors** in `pinger/main.go` line 36 — log or alert instead of silently discarding with `_`, so symlink interference is visible.
3. **Mount `/tmp` read-only or use a named pipe / in-memory channel** between the pinger and healthcheck binaries to eliminate the shared-filesystem attack surface entirely.
