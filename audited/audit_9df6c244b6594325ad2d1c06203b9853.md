### Title
Unbounded ConcurrentHashMap Growth in RedisTopicListener Enables Heap Exhaustion via Topic Subscription Flooding

### Summary
`RedisTopicListener.getSharedListener()` inserts one entry per unique topic ID into an unbounded `ConcurrentHashMap` with no capacity cap. Cleanup only fires on `doOnCancel`/`doOnComplete`, both of which are suppressed during a Redis network partition because `retryWhen` intercepts errors before they propagate. An attacker who subscribes to a large number of distinct valid topic IDs while a partition is active can accumulate map entries indefinitely, exhausting JVM heap.

### Finding Description

**Exact code path:**

`getSharedListener()` at line 59–62: [1](#0-0) 

```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`topicMessages` is declared as an unbounded `ConcurrentHashMap` with no size limit: [2](#0-1) 

The cleanup callbacks are placed **before** `retryWhen` in the operator chain: [3](#0-2) 

```java
return container
    .flatMapMany(r -> r.receive(...))
    .map(Message::getMessage)
    .doOnCancel(() -> unsubscribe(topic))   // only fires on cancellation
    .doOnComplete(() -> unsubscribe(topic)) // only fires on normal completion
    .doOnError(t -> log.error(...))         // logs only, does NOT remove from map
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval)...) // swallows errors, retries forever
    .share();
```

**Root cause and failed assumption:**

The design assumes that every entry inserted via `computeIfAbsent` will eventually be removed by `unsubscribe()`. This holds under normal conditions because `.share()` (which is `publish().refCount()`) cancels the upstream when the last subscriber leaves, triggering `doOnCancel`. However, during a Redis network partition:

1. The inner stream **errors** (not completes), so `doOnComplete` never fires.
2. `retryWhen` intercepts the error and schedules a retry — `doOnError` only logs.
3. The shared Flux stays alive in retry-backoff state.
4. If the attacker's gRPC TCP connections also hang (common during a partition — TCP keepalive is not enforced by default in the Netty config), `doOnCancel` is also never triggered.
5. The map entry persists for every distinct topic subscribed, with no eviction path.

**Existing check — `checkTopicExists` — and why it is insufficient:**

`TopicMessageServiceImpl.topicExists()` validates that the topic exists in the entity repository before the listener is invoked: [4](#0-3) 

This is the primary mitigation. However:

- `checkTopicExists` defaults to `true` but is a runtime-configurable flag (`grpcProperties.isCheckTopicExists()`). If set to `false`, any arbitrary topic ID is accepted with no validation.
- Even with `checkTopicExists = true`, Hedera mainnet contains millions of valid topic IDs. An attacker can enumerate and subscribe to a large number of real topics, each creating a distinct map entry.
- The `maxConcurrentCallsPerConnection = 5` limit in `NettyProperties` applies per connection, not globally. An attacker opens many connections in parallel. [5](#0-4) 

### Impact Explanation

Each `topicMessages` entry holds a `Flux<TopicMessage>` backed by a live (or retrying) Redis subscription. Under a sustained partition with many open subscriptions, the map grows proportionally to the number of distinct topics. At sufficient scale this causes `OutOfMemoryError`, crashing the gRPC service and denying all subscribers access to topic message streaming. Because the `container` Mono is `.cache()`-d and the retry loop runs with `Long.MAX_VALUE` attempts, the entries are effectively immortal until the JVM dies or the partition resolves and clients cleanly disconnect.

### Likelihood Explanation

- The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required.
- Opening many parallel gRPC connections (each with up to 5 concurrent streams) is trivial with standard tooling (`grpcurl`, Hedera SDK).
- Hedera mainnet has millions of valid topic IDs, making the `checkTopicExists = true` guard bypassable without any privileged knowledge.
- Network partitions between the mirror node and Redis are a realistic operational event (Redis restart, network flap, cloud AZ failure), and the retry-forever policy ensures entries are never cleaned up during the partition window.
- The attack is repeatable: after a JVM restart the attacker can immediately re-flood.

### Recommendation

1. **Cap the map size**: Replace `ConcurrentHashMap` with a size-bounded cache (e.g., Caffeine `Cache` with a maximum entry count and TTL-based eviction). Reject new subscriptions when the cap is reached.

2. **Move cleanup outside `retryWhen`**: Place `doOnCancel`/`doOnComplete` **after** `retryWhen` so they fire when the outer (retrying) Flux is cancelled or terminates, not just the inner attempt.

3. **Add a global subscriber cap**: Enforce a maximum number of concurrent live subscriptions server-wide, returning `RESOURCE_EXHAUSTED` when exceeded.

4. **Enable TCP keepalive**: Configure Netty to detect dead gRPC connections promptly so cancellation propagates and `doOnCancel` fires even during network partitions.

5. **Periodic map audit**: Add a background task that removes entries whose shared Flux has no active subscribers (refCount == 0).

### Proof of Concept

**Preconditions:**
- Mirror node running with `REDIS` listener type (default).
- Redis accessible (or a partition is induced after subscriptions are established).
- `checkTopicExists = false` **or** access to a list of valid topic IDs on the target network.

**Steps:**

```bash
# 1. Open N parallel gRPC connections, each subscribing to a unique topic ID.
#    With checkTopicExists=false, any numeric ID works.
for i in $(seq 1 10000); do
  grpcurl -plaintext \
    -d "{\"topicID\": {\"topicNum\": $i}}" \
    mirror-node:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 2. Induce or wait for a Redis network partition (e.g., block Redis port via iptables).
#    retryWhen keeps all Flux instances alive; doOnComplete/doOnCancel never fire.

# 3. Observe heap growth via JMX / heap dump:
#    topicMessages map contains ~10,000 entries, each holding a live Flux + Redis state.

# 4. Continue adding subscriptions. JVM OOM occurs when heap is exhausted.
```

**Observable result:** `java.lang.OutOfMemoryError: Java heap space` in the gRPC service logs; all topic subscriptions fail for legitimate users.

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))

                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
