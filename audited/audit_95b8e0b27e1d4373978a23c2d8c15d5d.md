### Title
Finite Retry Budget in `submitWithRetry()` Exhausted by Unprivileged Node-Level Throttle Flooding

### Summary
`submitWithRetry()` in `pinger/transfer.go` treats all errors identically with a fixed retry budget and exponential backoff, performing no error-type discrimination. Because Hedera's `BUSY` and `PLATFORM_TRANSACTION_NOT_CREATED` responses are **node-level** throttles (not account-level), any unprivileged attacker holding ordinary Hedera accounts can flood network nodes with their own transactions, sustaining throttle responses for the entire retry window and causing the pinger's transaction to be silently dropped for that tick.

### Finding Description
**Code location:** `pinger/transfer.go`, `submitWithRetry()`, lines 14–60.

```
// lines 20-59
var lastErr error
attempts := cfg.maxRetries + 1          // default: 11 attempts

for i := 1; i <= attempts; i++ {
    ...
    resp, err := cryptoTransfer.Execute(client)
    if err == nil { ... return nil }

    lastErr = err
    log.Printf("attempt %d/%d failed: %v", i, attempts, err)

    if i < attempts {
        sleep := backoff(cfg.baseBackoff, i)   // 2s, 4s, 8s, 16s, 30s, 30s...
        ...
    }
}
return fmt.Errorf("all attempts failed: %w", lastErr)
```

**Root cause / failed assumption:** The loop applies identical handling to every error class. There is no inspection of whether the returned error carries a `BUSY` (code 12) or `PLATFORM_TRANSACTION_NOT_CREATED` (code 69) status. The code assumes throttle conditions are transient and will clear within the retry window. That assumption is externally controllable.

**Why node-level throttling matters:** `BUSY` and `PLATFORM_TRANSACTION_NOT_CREATED` are emitted by a Hedera node when its inbound transaction backlog is saturated. They are not scoped to the payer/operator account — any transaction hitting an overloaded node receives them. An attacker with ordinary Hedera accounts (no special permissions) can submit a high volume of transactions to the same node(s) the pinger targets, keeping those nodes in a throttled state.

**Retry window calculation (defaults):** `baseBackoff=2s`, `maxRetries=10` → total wait before final failure: 2+4+8+16+30×6 = **210 seconds**. An attacker must sustain flooding for ~3.5 minutes per pinger tick to exhaust all retries.

**Existing mitigations reviewed and found insufficient:**
- Exponential backoff (`backoff()`, lines 62–65): helps against momentary spikes but provides no relief if the attacker sustains the flood across the entire window.
- Context cancellation check (line 24): only stops retries on shutdown, not on throttle errors.
- No error-type filtering exists anywhere in the retry loop.
- The Hiero SDK's internal node-rotation on `BUSY` may shift the target node, but an attacker flooding all nodes in the client's node list negates this.

After all attempts fail, `main.go` line 67 logs the error and the tick is silently skipped — no alerting, no circuit-breaker, no escalation.

### Impact Explanation
The pinger's sole purpose is to submit periodic liveness/monitoring transactions to the Hedera network and verify they are gossiped and recorded. Exhausting the retry budget means the transaction for that tick is never submitted to consensus. Sustained attacks across multiple ticks produce a gap in the pinger's transaction history, defeating its monitoring function. On a private or staging network (`network=other`) with a small node set, this can be achieved with modest transaction volume. On testnet/mainnet the attacker must flood more nodes but the code provides no additional defense.

### Likelihood Explanation
- **Precondition:** Attacker needs one or more funded Hedera accounts — permissionless on testnet, low-cost on mainnet.
- **No privileged access required:** The attack uses the attacker's own accounts; the operator's private key is never needed.
- **Repeatability:** The attack can be scripted and repeated every pinger tick (default: every 1 second), causing continuous monitoring blackout.
- **Feasibility by network:** High on private/`other` networks (1–3 nodes); Medium on testnet; Low-to-Medium on mainnet (30+ nodes, higher cost).

### Recommendation
1. **Inspect error type before retrying.** Parse the Hedera response code from the error. For `BUSY` / `PLATFORM_TRANSACTION_NOT_CREATED`, apply a shorter, immediate retry against a *different* node rather than burning the shared retry budget.
2. **Separate throttle retries from hard-error retries.** Throttle errors should not count against `maxRetries`; they should have their own counter with a tighter cap and immediate node-switch.
3. **Emit a metric/alert on throttle errors** so operators can detect an ongoing flooding attack rather than seeing only a silent log line.
4. **Consider a circuit-breaker** that raises an alarm if throttle errors persist beyond a configurable threshold across multiple ticks.

### Proof of Concept
**Preconditions:**
- Pinger deployed against a testnet or private network with `maxRetries=10`, `baseBackoff=2s` (defaults).
- Attacker has ≥1 funded Hedera account on the same network.

**Steps:**
1. Identify the Hedera node(s) in the pinger's client node list (observable from SDK logs or network address book).
2. From the attacker's client, submit a continuous flood of `CryptoTransfer` transactions (or any transaction type) to those nodes at a rate exceeding the node's TPS throttle limit.
3. The targeted node(s) begin returning `BUSY` or `PLATFORM_TRANSACTION_NOT_CREATED` to all submitters, including the pinger.
4. `submitWithRetry()` logs `attempt 1/11 failed: ...BUSY...`, waits 2 s, retries — receives `BUSY` again. Repeats through all 11 attempts over ~210 seconds.
5. After attempt 11, `submitWithRetry()` returns `"all attempts failed"`. `main.go` logs the failure and waits for the next tick.
6. The pinger's transaction is never gossiped to the network for that tick. Repeat the flood across subsequent ticks to produce a continuous monitoring gap.
