### Title
GraphQL Preparsed Query Cache Pollution via Unbounded Unique Query Flooding

### Summary
The `CachedPreparsedDocumentProvider` uses a Caffeine cache with a hard `maximumSize=1000` and no per-client rate limiting or admission control. Any unauthenticated caller can submit 1001+ syntactically distinct but valid GraphQL queries, filling the cache and triggering eviction of legitimate cached queries. Subsequent legitimate users whose queries were evicted must re-parse and re-validate their queries on every request until the cache re-warms.

### Finding Description
**Exact code path:**

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15: default spec is `"expireAfterWrite=1h,maximumSize=1000,recordStats"`.
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 20: `cache = Caffeine.from(properties.getQuery()).buildAsync();`
- Same file, line 26: `cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));` — the cache key is the **raw query string**.

**Root cause:** The cache is keyed on the verbatim query string with no normalization, no per-IP admission guard, and no rate limiting at the GraphQL layer. The `graphQlCustomizer()` bean in `GraphQlConfiguration.java` wires this provider directly into the GraphQL engine without any request-source throttle.

**Exploit flow:**
1. Attacker crafts 1001 unique but syntactically valid GraphQL queries (e.g., varying field aliases, argument names, or adding distinct inline fragments — all within the `maxCharacters=10000` / `maxTokens=1000` parser limits set in the static initializer of `GraphQlConfiguration.java` lines 33–38).
2. Attacker POSTs each query to the unauthenticated `/graphql/alpha` endpoint.
3. Each unique string gets its own cache slot. Once 1000 slots are filled, Caffeine's W-TinyLFU eviction begins displacing existing entries.
4. Legitimate queries that were previously cached are evicted; subsequent legitimate requests miss the cache and trigger full parse + validate cycles.

**Why existing checks are insufficient:**
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` run **after** parsing; they do not prevent cache insertion of the parsed result.
- Parser limits (`maxCharacters`, `maxTokens`) constrain individual query size but still allow thousands of distinct valid queries to be constructed.
- Caffeine's W-TinyLFU does protect high-frequency entries somewhat, but an attacker sending each unique query even once at a rate exceeding legitimate query frequency will still cause evictions of low-frequency legitimate queries.
- There is no per-IP, per-session, or authenticated-user rate limit on the GraphQL endpoint.

### Impact Explanation
Legitimate users whose queries are evicted experience re-parsing and re-validation latency on every subsequent request until the cache re-warms. While re-parsing a single GraphQL query is fast (sub-millisecond to low-millisecond), under sustained flooding the cache never re-warms for legitimate queries, turning every legitimate request into a cold-parse. This constitutes persistent griefing/latency degradation with no data loss or economic damage — consistent with the Medium classification.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no exploit tooling beyond a simple HTTP client loop. Generating 1001 unique valid GraphQL queries is trivial (vary one alias name per query). The attack is repeatable and can be sustained indefinitely. Any public-facing deployment of the GraphQL module is exposed.

### Recommendation
1. **Add per-IP or per-client rate limiting** at the HTTP layer (e.g., Spring's `HandlerInterceptor` or a reverse-proxy rule) before queries reach the preparsed document cache.
2. **Normalize query strings** before using them as cache keys (strip comments, canonicalize whitespace) to reduce the attacker's ability to generate distinct cache keys from semantically identical queries.
3. **Increase `maximumSize`** or make it configurable with a higher default to raise the cost of flooding.
4. **Add cache admission control**: reject or shadow-drop requests from sources that have caused more than N cache misses in a sliding window.

### Proof of Concept
```bash
# Generate 1001 unique valid GraphQL queries and POST each one
for i in $(seq 1 1001); do
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { balance }}\"}" \
    > /dev/null
done

# Now a legitimate query that was previously cached (e.g., num: 1) will miss the cache
# and incur full parse+validate latency on every subsequent request
curl -X POST http://localhost:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{account(input: {entityId: {shard: 0, realm: 0, num: 1}}) { balance }}"}'
```

Each iteration of the loop inserts a new unique cache entry (keyed on the distinct query string). After 1000 iterations the cache is full; iteration 1001 onward evicts existing entries. The final `curl` for `num: 1` will be a cache miss and require re-parsing.
