### Title
Unauthenticated Mirror Node Response Injection via Unvalidated HTTP Endpoint Enables Full gRPC Node Hijacking

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` fetches the consensus node list over a plain, unauthenticated HTTP connection with no TLS enforcement, no certificate pinning, and no validation of the returned gRPC endpoints. A network-positioned attacker (DNS poisoning, ARP spoofing, or BGP hijack) who intercepts the mirror REST request can inject a crafted `nodesEnvelope` that maps every node account ID to attacker-controlled gRPC addresses, causing all subsequent transaction submissions to be silently dropped or rejected, halting network confirmation for the pinger.

### Finding Description
**Exact code path:**

- `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, lines 36–72: constructs a bare `http.Client` with no TLS configuration and no scheme enforcement. The `mirrorRest` URL is operator-supplied; the documented default example is `http://mirror-rest:5551` — plain HTTP.
- Lines 46: `httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}` — no `Transport` override, no TLS pinning, no HTTPS-only enforcement.
- `fetchMirrorNodeNetwork()`, lines 74–132: issues a GET to the URL, decodes the JSON body into `nodesEnvelope`, and builds the network map directly from `ServiceEndpoints[].DomainName` / `IPAddressV4` / `Port` with zero validation of those values.
- Lines 113–124: every `(host, port)` pair from the response is inserted verbatim into the `network` map that is returned to `sdk_client.go`.
- `sdk_client.go` line 22: `client = hiero.ClientForNetwork(netmap)` — the injected map becomes the sole source of gRPC targets.

**Root cause:** The code makes an unauthenticated, unencrypted HTTP request to an operator-configured URL and unconditionally trusts the JSON body to define the consensus network topology. There is no HTTPS enforcement, no TLS certificate pinning, no HMAC/signature on the response, and no cross-check of returned endpoints against a static allowlist or known-good address range.

**Why existing checks are insufficient:**
- The only response check is the HTTP status code (lines 90–93): a 200 OK with a malicious body passes.
- The only per-entry check is that `NodeAccountID` is non-empty and parseable (lines 103–109): the attacker keeps valid account IDs and only replaces the endpoint addresses.
- The `len(network) == 0` guard (line 127) is bypassed because the injected response contains entries.

### Impact Explanation
All transaction submissions by the pinger are routed exclusively to the attacker-controlled gRPC endpoints. Those endpoints can drop, reject, or silently discard every `CryptoTransfer`, causing 100% transaction failure for the pinger. Because the pinger is a liveness/health-check component, this produces false network-down signals and halts confirmation monitoring. The network map is set once at startup (`newClient` → `buildNetworkFromMirrorNodes`) and never refreshed, so a single successful injection persists for the lifetime of the process.

### Likelihood Explanation
When `mirrorRest` is plain HTTP (the documented default), no TLS stripping is required at all — passive interception or DNS poisoning suffices. In a Kubernetes cluster, DNS spoofing of the internal service name (`mirror-rest`) is achievable by any pod with `CAP_NET_RAW` or via a compromised CoreDNS plugin. In cloud environments, BGP hijacking or a compromised load balancer achieves the same result externally. The attack is repeatable on every pinger restart and requires no credentials.

### Recommendation
1. **Enforce HTTPS**: Reject any `mirrorRest` URL that does not use the `https://` scheme at config load time (`config.go`, `loadConfig()`).
2. **Pin or verify the TLS certificate**: Supply a custom `http.Transport` with a pinned CA or leaf certificate for the mirror node.
3. **Validate returned endpoints**: Cross-check returned `(host, port)` pairs against a configured allowlist or CIDR range before inserting them into the network map.
4. **Sign the response**: Have the mirror node sign the `/network/nodes` payload; verify the signature before trusting the body.

### Proof of Concept
**Preconditions:**
- Pinger is deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (plain HTTP, default example).
- Attacker can poison DNS for `mirror-rest` within the cluster (e.g., via a compromised pod or CoreDNS misconfiguration).

**Steps:**
1. Stand up a malicious HTTP server on attacker-controlled IP that responds to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {"node_account_id": "0.0.3", "service_endpoints": [{"ip_address_v4": "attacker.ip", "port": 50211}]},
    {"node_account_id": "0.0.4", "service_endpoints": [{"ip_address_v4": "attacker.ip", "port": 50211}]}
  ],
  "links": {"next": null}
}
```
2. Poison DNS so `mirror-rest` resolves to the attacker's IP.
3. Restart (or wait for initial startup of) the pinger process.
4. `buildNetworkFromMirrorNodes()` fetches the injected response, builds a network map pointing entirely to `attacker.ip:50211`, and passes it to `hiero.ClientForNetwork()`.
5. All subsequent `CryptoTransfer` submissions are sent to `attacker.ip:50211`, which drops them.
6. The pinger reports 100% transaction failure; network confirmation monitoring halts.
