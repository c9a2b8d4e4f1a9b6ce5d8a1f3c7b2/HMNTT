### Title
Cache Bypass via Reference-Equality Singleton Check on `BlockType.LATEST` Allows Repeated Database Hits

### Summary

`findByBlockType()` in `RecordFileServiceImpl` uses Java reference equality (`==`) to detect the `BlockType.LATEST` singleton. A user-supplied hex block parameter `0x7fffffffffffffff` parses to `Long.MAX_VALUE` via the `GROUP_HEX_NUM` branch of `BlockType.of()`, producing a **new** `BlockType` instance that is structurally identical to `LATEST` but fails the `==` check. This diverts every such request to `findByIndex(Long.MAX_VALUE)`, which queries the database for a non-existent block index, returns `Optional.empty()`, and — because Spring Cache treats an unwrapped empty Optional as null and the `unless = "#result == null"` guard prevents caching null — the result is never stored in cache. The database is hit on every such request.

### Finding Description

**Code path:**

`BlockType.LATEST` is defined as a singleton with `number = Long.MAX_VALUE`: [1](#0-0) 

`BlockType.of()` routes the input `"0x7fffffffffffffff"` through the `GROUP_HEX_NUM` regex branch (group 4: `0x([0-9a-f]{1,16})`), which calls `Long.parseLong("7fffffffffffffff", 16)` = `Long.MAX_VALUE` and returns `new BlockType("7fffffffffffffff", Long.MAX_VALUE)` — a **fresh heap object**, not the `LATEST` singleton: [2](#0-1) 

Only string tags (`"latest"`, `"pending"`, `"safe"`, `"finalized"`) are routed to the singleton via `blockTypeForTag()`: [3](#0-2) 

In `findByBlockType()`, the guard `block == BlockType.LATEST` is a reference comparison. The crafted instance fails it, `isHash()` also returns false (number ≠ `BLOCK_HASH_SENTINEL`), so execution falls through to `findByIndex(Long.MAX_VALUE)`: [4](#0-3) 

`findByIndex` is annotated with `@Cacheable(... unless = "#result == null")`. Spring Cache unwraps `Optional` return values before evaluating `unless`; `Optional.empty()` unwraps to `null`, making `#result == null` true, so the empty result is **never stored in the index cache**: [5](#0-4) 

The intended `findLatest()` path uses `CACHE_MANAGER_RECORD_FILE_LATEST` (500 ms TTL, max 1 entry), which absorbs high-frequency "latest block" polling. The bypassed path has no effective cache protection for this input: [6](#0-5) 

### Impact Explanation

Every request carrying `block = "0x7fffffffffffffff"` issues a live `SELECT` against the `record_file` table (`WHERE index = 9223372036854775807`). No block with that index exists, so the query always returns empty and is never cached. Under sustained load this saturates database connections and degrades or denies service to all other mirror-node consumers. Additionally, any caller expecting the latest block receives an empty response, causing incorrect behavior in downstream EVM simulation, gas estimation, and contract-call endpoints.

### Likelihood Explanation

The exploit requires zero authentication or privileges — any unauthenticated JSON-RPC or REST caller can supply a `blockNumber` parameter. The value `0x7fffffffffffffff` is a valid, well-formed hex string accepted by the parser without error. The attack is trivially repeatable in a tight loop from a single client. The `BlockTypeTest` already documents that `0x7fffffffffffffff` parses to `Long.MAX_VALUE` as a numeric block (not the LATEST singleton), confirming the behavior is stable and not an edge case. [7](#0-6) 

### Recommendation

Replace the reference-equality checks in `findByBlockType()` with value-based checks on `block.number()`:

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (block.number() == 0L && !block.isHash()) {          // EARLIEST
        return recordFileRepository.findEarliest();
    } else if (block.number() == Long.MAX_VALUE && !block.isHash()) {  // LATEST
        return recordFileRepository.findLatest();
    } else if (block.isHash()) {
        return recordFileRepository.findByHash(block.name());
    }
    return recordFileRepository.findByIndex(block.number());
}
```

Alternatively, normalize `BlockType.of()` so that any numeric input whose parsed value equals `Long.MAX_VALUE` returns the `LATEST` singleton rather than a new instance. Also consider adding an upper-bound validation in `BlockType.of()` that rejects hex numbers ≥ a realistic maximum block index.

### Proof of Concept

```bash
# Repeated requests — each hits the database, none are served from cache
for i in $(seq 1 1000); do
  curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"blockNumber":"0x7fffffffffffffff","data":"0x","to":"0x0000000000000000000000000000000000000001"}'
done
```

1. `BlockType.of("0x7fffffffffffffff")` → `new BlockType("7fffffffffffffff", 9223372036854775807L)` (not the LATEST singleton).
2. `findByBlockType()` line 22: `block == BlockType.LATEST` → `false`.
3. `findByIndex(9223372036854775807L)` executes `SELECT r FROM RecordFile r WHERE r.index = 9223372036854775807` → empty.
4. Spring Cache unwraps `Optional.empty()` → null; `unless = "#result == null"` → true → result not cached.
5. Step 3 repeats on every request; database load grows linearly with request rate.
