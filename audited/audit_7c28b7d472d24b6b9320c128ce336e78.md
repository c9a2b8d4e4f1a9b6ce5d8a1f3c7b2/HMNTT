### Title
Unprivileged User Can Trigger Misleading ERROR-Level Log Spam via Pre-Genesis Block Hash Lookup

### Summary
In `findBlockByHash()`, when a queried block exists in the database but predates the genesis block, the function logs a user-supplied hash value at `ERROR` level. Because pre-genesis block hashes are public blockchain data and the `/block` Rosetta endpoint requires no authentication, any external user can repeatedly trigger this error log, flooding operator monitoring with false system-error signals.

### Finding Description
**Code location:** `rosetta/app/persistence/block.go`, `findBlockByHash()`, lines 235–237.

```go
if rb.Index < br.genesisBlock.Index {
    log.Errorf("The block with hash %s is before the genesis block", hash)
    return nil, hErrors.ErrBlockNotFound
}
```

**Root cause:** The `hash` parameter is user-supplied and flows from the public HTTP endpoint through `RetrieveBlock` (`base_service.go:84–86`) → `FindByHash` (`block.go:135–145`) → `findBlockByHash` with no sanitization and no rate limiting. The only guard in `FindByHash` is an empty-string check (`block.go:136–138`). When the DB query succeeds (the hash matches a real pre-genesis record file) but `rb.Index < br.genesisBlock.Index`, the code logs at `ERROR` level — a level that implies a system fault — even though this is a fully expected, normal user query for a block that simply predates the configured genesis.

**Exploit flow:**
1. Attacker enumerates one or more pre-genesis `record_file` hashes from the public Hedera ledger (they are stored in the `record_file` table and are part of the public chain history).
2. Attacker sends repeated unauthenticated POST requests to `/block` with `block_identifier.hash` set to a known pre-genesis hash.
3. Each request causes `log.Errorf("The block with hash %s is before the genesis block", <attacker-hash>)` to fire.

**Why existing checks are insufficient:**
- `FindByHash` only rejects empty strings; any non-empty hash passes through.
- `RetrieveBlock` in `base_service.go` applies no authentication, rate limiting, or hash-format validation before calling `FindByHash`.
- The Rosetta router (`main.go:111–119`) exposes the block endpoint with no middleware-level rate limiting visible in the codebase.

### Impact Explanation
An attacker can generate a continuous stream of `ERROR`-level log entries containing attacker-chosen hash strings. This causes: (1) alert fatigue — monitoring/alerting systems fire on ERROR logs, masking real failures; (2) log storage exhaustion if log retention is volume-based; (3) misleading incident response, since operators see ERROR messages that look like data-integrity problems ("block is before genesis") when no actual fault exists. Severity is **Medium** — no data is leaked or corrupted, but operational integrity of the monitoring plane is degraded.

### Likelihood Explanation
Pre-genesis block hashes are publicly available on the Hedera public ledger. The `/block` endpoint requires zero authentication (standard Rosetta API design). The attack is trivially scriptable with a single `curl` loop. It is repeatable indefinitely with no credential or privilege requirement.

### Recommendation
1. **Downgrade log level:** Change `log.Errorf` to `log.Warnf` or `log.Debugf` at line 236 — a user querying a pre-genesis block is not a system error.
2. **Sanitize before logging:** Strip or truncate the hash to a fixed expected length (e.g., 64 hex chars) before embedding it in any log message, preventing log-injection via crafted hash strings.
3. **Add rate limiting:** Apply per-IP or global rate limiting middleware on the Rosetta HTTP router to bound the request rate to the `/block` endpoint.

### Proof of Concept
```bash
# 1. Obtain a pre-genesis record_file hash (public chain data), e.g.:
PRE_GENESIS_HASH="<known pre-genesis block hash>"

# 2. Flood the endpoint (no auth required):
while true; do
  curl -s -X POST http://<rosetta-host>/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"hash\":\"${PRE_GENESIS_HASH}\"}}" \
    > /dev/null
done

# 3. Observe server logs filling with:
# ERROR: The block with hash <PRE_GENESIS_HASH> is before the genesis block
```

Each iteration produces one `ERROR` log entry. The server returns `ErrBlockNotFound` to the caller (correct), but the ERROR-level log is emitted regardless, misleading operators.
