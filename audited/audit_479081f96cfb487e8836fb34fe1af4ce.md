### Title
Unauthenticated Cache-Bypass DoS via Timestamp-Parameterized Historical UNION ALL Queries in `getTokenInfoRequest`

### Summary
The `getTokenInfoRequest` handler in `rest/tokens.js` only populates the in-process `TokenService` LRU cache when no filters are present. Any request carrying a `?timestamp=lte:<value>` filter bypasses this cache entirely and executes three nested `UNION ALL` subqueries against both current and history tables. Because the HTTP response cache key is derived from the full URL, an attacker can trivially generate unique cache-miss requests by incrementing the timestamp value, forcing a fresh set of expensive DB queries on every request with no per-IP rate limiting in the Node.js REST service.

### Finding Description

**Code path:**

`getTokenInfoRequest` (rest/tokens.js, lines 548–565) calls `extractSqlFromTokenInfoRequest`, which — when `filters.length !== 0` — calls `buildHistoryQuery` three times (for `token`, `entity`, and `custom_fee` tables). [1](#0-0) 

`buildHistoryQuery` (lines 530–546) generates a subquery of the form:
```sql
(SELECT ... FROM tableName WHERE conditions
 UNION ALL
 SELECT ... FROM tableName_history WHERE conditions ORDER BY lower(timestamp_range) DESC LIMIT 1
 ORDER BY modified_timestamp DESC LIMIT 1)
```
This is repeated for `token`/`token_history`, `entity`/`entity_history`, and `custom_fee`/`custom_fee_history`, all joined together in a single outer query. [2](#0-1) 

**Root cause — cache bypass:**

The `TokenService` in-process LRU cache is only written when `filters.length === 0`:
```js
if (filters.length === 0) {
  TokenService.putTokenCache(token);
}
``` [3](#0-2) 

The HTTP-layer Redis response cache (`responseCacheHandler.js`) keys on `req.originalUrl` via MD5:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
``` [4](#0-3) 

Every distinct timestamp value produces a distinct cache key, so an attacker who increments the timestamp by 1 nanosecond per request generates an unbounded stream of cache misses, each hitting the database.

**No per-IP rate limiting in the REST service:**

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the Java `web3` module, not in the Node.js REST service. [5](#0-4) 

The `requestHandler.js` middleware performs no rate limiting; it only logs and sets timing metadata.

**DB connection pool is the only backstop:**

The pool is bounded by `config.db.pool.maxConnections` and has a `statement_timeout`. [6](#0-5) 

Under sustained attack, all pool connections become occupied waiting for the UNION ALL queries to complete or time out, causing all other API requests to queue or fail with connection-timeout errors.

### Impact Explanation

An attacker can exhaust the PostgreSQL connection pool by flooding `GET /api/v1/tokens/<popularTokenId>?timestamp=lte=<incrementing_value>`. Each request executes three nested UNION ALL scans across both current and history tables. With the pool saturated, all other mirror-node REST endpoints (transactions, accounts, balances, etc.) become unavailable or severely degraded. The `statement_timeout` provides partial mitigation but does not prevent pool exhaustion when the request rate exceeds the timeout-and-release cycle.

### Likelihood Explanation

No authentication or API key is required. The only input needed is a valid token ID (publicly enumerable from `GET /api/v1/tokens`) and a monotonically incrementing nanosecond timestamp. A single attacker with a modest HTTP client (e.g., `ab`, `wrk`, or a simple script) can sustain hundreds of unique-URL requests per second. The attack is stateless, trivially scriptable, and requires no privileged access.

### Recommendation

1. **Apply rate limiting per IP/client** in the Node.js REST middleware for the `/tokens/:tokenId` route (e.g., via `express-rate-limit` or an ingress-level policy).
2. **Normalize/clamp the timestamp parameter** to a coarser granularity (e.g., nearest second) before generating the cache key, so that near-identical requests share a Redis cache entry.
3. **Cache historical responses**: extend `TokenService.putTokenCache` (or the Redis response cache TTL logic) to also cache responses for timestamp-filtered requests, keyed on `(tokenId, normalizedTimestamp)`.
4. **Enforce a maximum pool wait queue depth** so that excess requests are rejected with HTTP 503 rather than queuing indefinitely.

### Proof of Concept

```bash
# Enumerate a valid token ID
TOKEN_ID=$(curl -s https://<mirror-node>/api/v1/tokens?limit=1 | jq -r '.tokens[0].token_id')

# Flood with unique timestamps — each request bypasses both caches
BASE_TS=1700000000000000000
for i in $(seq 1 5000); do
  curl -s "https://<mirror-node>/api/v1/tokens/${TOKEN_ID}?timestamp=lte:$((BASE_TS + i))" &
done
wait
# Observe: subsequent requests to /api/v1/transactions or /api/v1/accounts return
# 503 / connection timeout as the DB pool is exhausted.
```
