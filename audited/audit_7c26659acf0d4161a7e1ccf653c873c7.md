### Title
LRU Cache Poisoning via Token ID Cycling Causes Sustained DB Overload (Non-Network DoS)

### Summary
The `tokenCache` in `rest/service/tokenService.js` is initialized with `quick-lru` using only a `maxSize` bound and no TTL. An unauthenticated attacker can cycle through more than `maxSize` unique token IDs via the public `/api/v1/accounts/{id}/tokens` endpoint, continuously evicting legitimate tokens from the cache and forcing repeated DB queries for every legitimate user request, saturating the small DB connection pool (default: 10 connections).

### Finding Description

**Exact code location:**

`rest/service/tokenService.js`, lines 12–14 — cache initialization with no TTL:
```js
const tokenCache = new quickLru({
  maxSize: config.cache.token.maxSize,  // default: 100,000; no maxAge/TTL
});
```

`rest/service/tokenService.js`, lines 136–161 — `getCachedTokens()`:
```js
async getCachedTokens(tokenIds) {
  const cachedTokens = new Map();
  const uncachedTokenIds = [];
  tokenIds.forEach((tokenId) => {
    const cachedToken = tokenCache.get(tokenId);  // LRU access — updates recency
    if (cachedToken) {
      cachedTokens.set(tokenId, cachedToken);
    } else {
      uncachedTokenIds.push(tokenId);             // cache miss → DB query
    }
  });
  if (uncachedTokenIds.length === 0) return cachedTokens;
  const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]); // DB hit
  rows.forEach((row) => {
    tokenCache.set(tokenId, cachedToken);         // fills cache, triggers LRU eviction
    ...
  });
  return cachedTokens;
}
```

**Root cause and failed assumption:**

`quick-lru` implements a two-bucket approximation of LRU. Internally it maintains `_cache` (new bucket) and `_oldCache` (old bucket). When `_cache` reaches `maxSize`, `_oldCache` is discarded and `_cache` becomes the new `_oldCache`. Any item not re-accessed before the next rotation is permanently evicted. This means inserting `2 × maxSize` (200,000) unique token IDs is sufficient to evict **all** previously cached entries, regardless of how recently they were accessed by legitimate users.

The failed assumption is that `maxSize = 100,000` is large enough to resist cache flooding. With no TTL, no per-IP rate limiting on the endpoint, and no authentication requirement, the cache boundary is the only protection — and it is bypassable.

**Exploit flow:**

1. Attacker enumerates Hedera accounts with many token associations (public data via the same mirror node API: `/api/v1/accounts`).
2. Attacker scripts rapid GET requests to `/api/v1/accounts/{id}/tokens` for a rotating set of accounts whose collective unique token associations exceed `2 × maxSize` (200,000 token IDs on Hedera mainnet — feasible given the network's token count).
3. Each request triggers `getTokenAccounts()` → `getCachedTokens()`. Cache misses cause `tokenCacheQuery` DB queries. Results are stored in `tokenCache`, filling the new bucket.
4. Once the new bucket reaches `maxSize`, `quick-lru` rotates: the old bucket (containing legitimate tokens) is discarded.
5. Attacker continues cycling through the next `maxSize` unique token IDs, rotating the bucket again.
6. All legitimate tokens are now permanently evicted. Every subsequent legitimate user request for token relationships results in a DB query.
7. With the DB pool capped at `maxConnections = 10` (default), concurrent cache-miss-induced queries from legitimate users saturate the pool, causing query queuing, timeouts (`statementTimeout = 20,000ms`), and effective service degradation.

**Why existing checks fail:**

- No TTL on `tokenCache` — once evicted, tokens are never automatically re-warmed; every miss is a DB round-trip.
- No per-IP or per-user rate limiting on `/api/v1/accounts/{id}/tokens` — the attacker can send requests at wire speed.
- No authentication — the endpoint is fully public.
- DB pool is only 10 connections — a small number of concurrent cache-miss requests is sufficient to saturate it.
- The `statementTimeout = 20,000ms` means each saturated connection holds a slot for up to 20 seconds, amplifying the impact.

### Impact Explanation

Sustained DB connection pool exhaustion causes legitimate API requests to queue or time out. Since `tokenCache` is a process-level singleton shared across all requests, the attack degrades the entire REST API service, not just a single user. The DB pool default of 10 connections means as few as 10 concurrent cache-miss-inducing requests can fully saturate the pool. This constitutes a non-network-based DoS affecting the mirror node's REST API availability, which serves as critical infrastructure for applications querying Hedera state.

### Likelihood Explanation

The attack requires no credentials, no on-chain transactions, and no special tooling — only the ability to send HTTP GET requests. Hedera mainnet already has hundreds of thousands of tokens and accounts with diverse token associations, so the attacker does not need to create any on-chain state. A simple script cycling through known account IDs is sufficient. The attack is repeatable and self-sustaining: as long as the attacker maintains the cycling rate, the cache never recovers. Detection is difficult because each individual request is indistinguishable from legitimate traffic.

### Recommendation

1. **Add a TTL (`maxAge`) to `tokenCache`**: Initialize `quick-lru` with both `maxSize` and `maxAge` (e.g., 60–300 seconds), consistent with how `entityId.js` configures its cache (see `rest/entityId.js` line 301–304 which uses `maxAge`). This bounds the window of cache poisoning and ensures organic re-warming.

2. **Add rate limiting** on the `/api/v1/accounts/{id}/tokens` endpoint at the application or ingress layer to limit requests per IP per second.

3. **Consider a minimum cache floor**: Track access frequency (LFU instead of LRU) for the token cache, or use a segmented cache that protects a "protected" segment of high-frequency tokens from eviction by new entries.

4. **Increase DB pool size** or add a circuit breaker that rejects new requests when the pool is near saturation, preventing cascading timeouts.

### Proof of Concept

```python
import requests
import threading

MIRROR_NODE = "https://<mirror-node-host>"
# Pre-enumerate 200,001 account IDs with token associations
# (obtainable from /api/v1/accounts?limit=100 pagination)
ACCOUNT_IDS = [f"0.0.{i}" for i in range(1000, 201001)]

def flood():
    for account_id in ACCOUNT_IDS:
        try:
            requests.get(
                f"{MIRROR_NODE}/api/v1/accounts/{account_id}/tokens",
                params={"limit": 100},
                timeout=5
            )
        except Exception:
            pass

# Launch concurrent threads to sustain eviction pressure
threads = [threading.Thread(target=flood) for _ in range(20)]
for t in threads:
    t.start()
for t in threads:
    t.join()

# After the above runs, legitimate token lookups will all be cache misses.
# Monitor DB connection pool saturation and response latency degradation.
```

**Expected result**: After cycling through >200,000 unique token IDs, all previously cached tokens are evicted. Subsequent legitimate requests to `/api/v1/accounts/{id}/tokens` for any account will trigger DB queries for every token ID. With 10 concurrent legitimate requests each touching uncached tokens, the DB pool saturates and response times spike or requests time out.
