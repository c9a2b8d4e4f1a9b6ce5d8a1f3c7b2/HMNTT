### Title
Double Uncached DB Query Per Request on `GET /api/v1/network/fees` Enables Amplified Connection Pool Exhaustion

### Summary
`NetworkController.getFees()` issues two sequential, uncached database queries per request — one for the fee schedule and one for the exchange rate — with no rate limiting applied to the `rest-java` module's endpoint. Any unauthenticated external user can flood this endpoint at high concurrency to exhaust the HikariCP connection pool at twice the rate of any single-query endpoint, degrading or denying service to all other database-backed endpoints.

### Finding Description

**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 100–108.

```java
@GetMapping("/fees")
NetworkFeesResponse getFees(
        @RequestParam(required = false) @Size(max = 2) TimestampParameter[] timestamp,
        @RequestParam(required = false, defaultValue = "ASC") Sort.Direction order) {
    final var bound = Bound.of(timestamp, TIMESTAMP, FileData.FILE_DATA.CONSENSUS_TIMESTAMP);
    final var feeSchedule = fileService.getFeeSchedule(bound);   // DB query #1
    final var exchangeRate = fileService.getExchangeRate(bound); // DB query #2
    return feeScheduleMapper.map(feeSchedule, exchangeRate, bound, order);
}
``` [1](#0-0) 

Both calls delegate to `FileServiceImpl.getSystemFile()`, which unconditionally calls `fileDataRepository.getFileAtTimestamp()` — a live DB query — with no `@Cacheable` annotation or any in-memory cache on the `rest-java` `FileServiceImpl`. [2](#0-1) [3](#0-2) 

**No rate limiting exists for this endpoint.** The bucket4j throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives entirely in the `web3` module and is wired only to `ContractController` requests. [4](#0-3) 

The `rest-java` module has no equivalent throttle bean or filter applied to `NetworkController`.

**Retry amplification:** `FileServiceImpl` configures a `RetryTemplate` with `maxRetries(9)`. If the latest file is corrupt (which the `fallbackRecovers` test explicitly exercises), each of the two calls can issue up to 10 DB queries, for a worst-case of 20 DB queries per single HTTP request. [5](#0-4) 

By contrast, `GET /api/v1/network/exchangerate` issues only one DB query per request, and `GET /api/v1/network/stake` issues none via the file path. [6](#0-5) 

### Impact Explanation
The HikariCP connection pool shared by the `rest-java` service has a finite size. Each concurrent `GET /api/v1/network/fees` request holds two pool connections (or blocks waiting for them) for the duration of both queries. At sufficient concurrency, the pool is exhausted, causing all other endpoints (`/api/v1/network/nodes`, `/api/v1/network/supply`, etc.) to queue or time out on connection acquisition. This is a targeted, amplified resource exhaustion: the attacker gets 2× DB load per request compared to any other single-query endpoint, with zero authentication required. [7](#0-6) 

### Likelihood Explanation
The endpoint is fully public — no API key, no session, no authentication. The only parameter needed is a valid timestamp format (e.g., `?timestamp=gte:0`), which is documented in the OpenAPI spec. A single attacker with a modest number of concurrent HTTP clients (e.g., 50–100 goroutines or threads) can sustain the attack indefinitely. The attack is stateless, trivially repeatable, and requires no special knowledge beyond the public API documentation. [8](#0-7) 

### Recommendation
1. **Add caching to `FileServiceImpl`**: Apply a short-lived (e.g., 30–60 second) in-memory cache (Caffeine) to `getExchangeRate()` and `getFeeSchedule()` for the common case of no timestamp parameter (i.e., `Bound.EMPTY`). The `web3` module's `SystemFileLoader` already demonstrates this pattern with `CACHE_MANAGER_EXCHANGE_RATES_SYSTEM_FILE`.
2. **Add rate limiting to `rest-java` endpoints**: Introduce a bucket4j or Spring rate-limiter filter for the `rest-java` module, analogous to the `web3` `ThrottleManagerImpl`, with a per-IP or global RPS cap on `/api/v1/network/fees`.
3. **Merge the two queries**: Since both calls use the same `bound`, consider a single repository method that fetches both system files in one query or transaction to halve the connection usage.

### Proof of Concept
```bash
# Flood with 100 concurrent requests, each supplying two timestamp params
# to ensure the bound is non-trivial and bypasses any future empty-bound cache
seq 1 10000 | xargs -P 100 -I{} \
  curl -s -o /dev/null \
  "https://<mirror-node-host>/api/v1/network/fees?timestamp=gte:0&timestamp=lte:9999999999999999999"
```

Each request triggers two uncached `fileDataRepository.getFileAtTimestamp()` calls. At 100 concurrent clients, 200 DB connections are demanded simultaneously. Once the HikariCP pool (default size 10) is saturated, all subsequent requests — including from legitimate users hitting other endpoints — will block on connection acquisition until timeout.

    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();

    }
