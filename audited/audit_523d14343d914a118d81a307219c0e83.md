### Title
Cache-Bypass DoS via Unbounded Token Cache Queries on `/accounts/{id}/tokens` Endpoint

### Summary
The `getTokenAccounts()` function in `rest/service/tokenService.js` issues a secondary `token_id = any($1)` database query for every uncached token ID returned by the primary query. Because the endpoint has no rate limiting and the in-memory LRU cache can be continuously bypassed using novel token IDs, an unauthenticated attacker can sustain a high rate of large array queries against the database, exhausting the 10-connection pool and causing a non-network-based DoS.

### Finding Description

**Code path:**

`rest/service/tokenService.js`, `getTokenAccounts()` (lines 96–115) → `getCachedTokens()` (lines 136–161).

```
getTokenAccounts(query)
  → getRows(tokenRelationshipsQuery, params)          // primary query: token_account table
  → tokenIds = rows.reduce(... new Set())             // up to 100 unique IDs at limit=100
  → getCachedTokens(tokenIds)
      → for each tokenId: check quickLru cache
      → uncachedTokenIds.push(tokenId) for misses
      → getRows(tokenCacheQuery, [uncachedTokenIds])  // secondary: WHERE token_id = any($1)
``` [1](#0-0) [2](#0-1) 

**Root cause — three compounding weaknesses:**

1. **No rate limiting on the endpoint.** `rest/server.js` registers the route via `AccountRoutes.router` with no throttle middleware. The only middleware chain is: cors → compression → httpContext → requestLogger → authHandler → metricsHandler → responseCacheCheck → route handlers. There is no IP-based or request-rate limiter for the REST API (unlike the web3 API which has `ThrottleManagerImpl`). [3](#0-2) 

2. **The `quickLru` cache has no TTL.** It is constructed with only `maxSize: config.cache.token.maxSize` (default 100,000). Entries never expire; they are only evicted by LRU pressure. An attacker querying accounts associated with token IDs not yet in the cache will always produce `uncachedTokenIds.length > 0` and trigger the secondary DB query. [4](#0-3) 

3. **The secondary query has no LIMIT clause.** `tokenCacheQuery` is `WHERE token_id = any($1)` with no row cap. With `limit=100` on the primary query, up to 100 IDs are passed in the array per request. [5](#0-4) 

**Why the cache fails to protect:** The Hedera mainnet has hundreds of thousands of tokens. The cache holds 100,000 entries. An attacker can rotate through accounts associated with different tokens, continuously producing cache misses. Once the cache fills, LRU eviction restores the attack surface. The cache is also per-process and provides no protection across horizontally scaled instances.

### Impact Explanation

Each request to `/api/v1/accounts/{id}/tokens?limit=100` with uncached token IDs causes two DB queries: one against `token_account` and one `token_id = any($1)` against the `token` table with up to 100 IDs. The database connection pool is capped at **10 connections** with a `statementTimeout` of **20 seconds**. [6](#0-5) 

Sustained parallel requests can exhaust the pool, causing all subsequent queries (including unrelated endpoints) to queue or timeout. This degrades or denies service for all users of the mirror node REST API without any network-layer amplification.

### Likelihood Explanation

- **No authentication required.** The endpoint is fully public; `authHandler` only sets a custom response limit for known users, it does not block or throttle anonymous callers. [7](#0-6) 
- **No privileges needed.** Any HTTP client can call `/api/v1/accounts/{id}/tokens`.
- **Trivially repeatable.** A single script cycling through account IDs with diverse token associations (publicly discoverable via the same API) is sufficient.
- **Low cost to attacker.** Each request is a standard GET with no body; the cost is asymmetric — cheap for the attacker, expensive for the DB.

### Recommendation

1. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit` or a token-bucket per IP) specifically covering the `/accounts/*/tokens` route.
2. **Add a LIMIT clause to `tokenCacheQuery`** matching the maximum response limit (100) to bound the secondary query cost.
3. **Add a TTL to the token cache** (e.g., via `quick-lru`'s `maxAge` option or replacing with a TTL-aware cache) so that the cache size alone does not determine the attack surface.
4. **Consider capping `uncachedTokenIds`** before issuing the secondary query, as a defense-in-depth measure.

### Proof of Concept

```bash
# 1. Discover accounts with many token associations (publicly available)
curl "https://<mirror-node>/api/v1/accounts?limit=100"

# 2. For each account, fire requests at maximum limit with novel token.id ranges
# to ensure cache misses on each iteration:
while true; do
  for ACCOUNT in 0.0.1234 0.0.5678 0.0.9012 ...; do
    curl -s "https://<mirror-node>/api/v1/accounts/$ACCOUNT/tokens?limit=100" &
  done
  wait
done

# 3. Observe: DB connection pool exhaustion, increasing query latency,
#    and eventual timeout errors (HTTP 500 / 503) for all API consumers.
```

The attack requires no credentials, no special network position, and no payload — only repeated HTTP GET requests to a public endpoint.

  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });
  }

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
