### Title
N+1 DB Lookup Amplification via Multiple EVM-Address `from` Parameters in `extractContractResultsByIdQuery`

### Summary
`extractContractResultsByIdQuery` in `rest/controllers/contractController.js` iterates over all `from` filters and, for each one that is a valid EVM address, issues a separate synchronous `EntityService.getEncodedId` database call. An unauthenticated attacker can supply up to `maxRepeatedQueryParameters` (default: 100) distinct EVM addresses in a single request, forcing up to 100 sequential DB lookups against a connection pool capped at 10 connections.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js`, `extractContractResultsByIdQuery` (lines 439–454):

```js
for (const filter of filters) {
  switch (filter.key) {
    case filterKeys.FROM:
      if (EntityId.isValidEvmAddress(filter.value)) {
        filter.value = await EntityService.getEncodedId(filter.value); // DB call per EVM address
      }
      ...
``` [1](#0-0) 

Each call to `EntityService.getEncodedId` with an EVM address resolves to `getEntityIdFromEvmAddress`, which executes:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
  from ${Entity.tableName}
  where ${Entity.DELETED} <> true
    and ${Entity.EVM_ADDRESS} = $1`;
``` [2](#0-1) 

This is a direct DB query with no caching, no batching, and no deduplication. [3](#0-2) 

**Root cause:** The loop processes each `from` filter independently and `await`s each DB call sequentially. There is no batching of EVM address lookups into a single `WHERE evm_address IN (...)` query, and no in-process cache for EVM-address-to-entity-id resolution in this service.

**Failed assumption:** The code assumes the number of `from` EVM address filters will be small. The only guard is `maxRepeatedQueryParameters` (default **100**), which is enforced in `buildFilters`: [4](#0-3) 

This check allows up to 100 repeated `from` parameters — it is a ceiling, not a protection against amplification.

**Validation check for `FROM`** in `filterValidityChecks` only validates format, not count: [5](#0-4) 

Both public endpoints that call `extractContractResultsByIdQuery` are unauthenticated:
- `GET /api/v1/contracts/results` (`getContractResults`, line 1050)
- `GET /api/v1/contracts/:contractId/results` (`getContractResultsById`, line 856) [6](#0-5) 

### Impact Explanation
The DB connection pool is capped at **10 connections** by default: [7](#0-6) 

A single request with 100 unique EVM-address `from` values issues 100 sequential DB queries. A modest number of concurrent such requests (e.g., 10–20) can saturate the pool, causing all other API requests to queue or time out (20 s statement timeout). This is a low-cost, unauthenticated denial-of-service amplification: 1 HTTP request → up to 100 DB queries.

### Likelihood Explanation
No authentication or rate limiting is required. The attack is trivially reproducible with a single `curl` command using repeated `from=` query parameters. Any external user who can reach the REST API can exploit this. The default `maxRepeatedQueryParameters` of 100 is documented publicly: [8](#0-7) 

### Recommendation
1. **Batch the EVM address lookups**: Collect all EVM-address `from` values first, then resolve them in a single `WHERE evm_address = ANY($1)` query instead of N sequential queries.
2. **Reduce `maxRepeatedQueryParameters` for the `from` key specifically**, or add a separate, lower cap for parameters that trigger DB lookups.
3. **Add an in-process LRU cache** for EVM-address → encoded-id resolution (the config already defines `cache.entityId` settings but they are not wired into `EntityService.getEncodedId` in the JS REST layer). [9](#0-8) 

### Proof of Concept

```bash
# Generate 100 unique fake EVM addresses and send as from= params
PARAMS=$(python3 -c "
import random, sys
addrs = ['from=' + ''.join(['{:02x}'.format(random.randint(0,255)) for _ in range(20)]) for _ in range(100)]
print('&'.join(addrs))
")

curl -s "http://<mirror-node-host>:5551/api/v1/contracts/results?$PARAMS"
```

**Expected result:** The server issues up to 100 `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` queries sequentially before returning a response. Sending this request concurrently from multiple clients exhausts the DB connection pool and degrades or denies service to all other API consumers.
