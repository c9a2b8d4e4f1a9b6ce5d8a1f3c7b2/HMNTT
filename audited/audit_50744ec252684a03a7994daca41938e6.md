### Title
Unauthenticated GraphQL Variable-Driven DB Exhaustion via Unbounded `findByAlias()` Invocations

### Summary
The GraphQL module has no application-level rate limiting, no result caching, and no authentication requirement. An unprivileged attacker can send a high volume of requests using a constant query string (exploiting `CachedPreparsedDocumentProvider`'s parse-only cache) while varying the `alias` variable, causing `entityRepository.findByAlias()` to execute a raw SQL query against the database on every single request. This generates sustained, unbounded DB load while the parse/validate overhead is eliminated by the cache, making the attack maximally efficient.

### Finding Description

**Code path:**

`CachedPreparsedDocumentProvider.getDocumentAsync()` caches keyed on `executionInput.getQuery()` — the raw query string only, not variables: [1](#0-0) 

This means parse+validate is skipped on cache hit, but **execution always proceeds**. The resolver calls: [2](#0-1) 

Which calls `decodeBase32(alias)` and passes the resulting `byte[]` directly to: [3](#0-2) 

This is a native SQL query with no result caching (`@Cacheable` or similar) anywhere in `EntityServiceImpl` or `EntityRepository`. There is **zero application-level rate limiting** in the entire `graphql` module:



The only throttling in the codebase is in the `web3` module (`ThrottleConfiguration`, `ThrottleManagerImpl`), which is entirely separate and does not apply to GraphQL endpoints. [4](#0-3) 

**Root cause:** The `CachedPreparsedDocumentProvider` is a parse-cache, not an execution-cache. The failed assumption is that caching the document provides any protection against DB-level resource exhaustion when variables change per request.

### Impact Explanation

Every request with a unique alias value issues a full table scan / index lookup against the `entity` table (`WHERE alias = $1 AND deleted IS NOT TRUE`). With no rate limiting, an attacker can sustain thousands of DB queries per second from a single IP. The Traefik `inFlightReq: amount: 5` middleware is an optional Helm chart value (not enforced in application code) and only limits concurrency per IP — with 5 concurrent requests each completing in ~5–10ms, that still allows ~500–1000 DB queries/second per attacker IP. The `retry: attempts: 3` Traefik config further multiplies effective DB load by up to 3x per request. [5](#0-4) 

This satisfies the >30% DB load increase threshold with a single attacker IP, and scales linearly with additional IPs.

### Likelihood Explanation

No privileges, credentials, or special knowledge are required. The GraphQL endpoint is publicly documented and accessible. The attack requires only a script sending repeated POST requests with a constant query string and incrementing alias values. The parse cache actually makes the attack more efficient by reducing server-side CPU overhead, allowing higher request throughput from the attacker.

### Recommendation

1. **Add application-level rate limiting to the GraphQL module** — implement a `bucket4j`-based filter (mirroring `web3`'s `ThrottleConfiguration`) applied to the `/graphql/alpha` endpoint, limiting requests per IP per second.
2. **Add result-level caching** — apply `@Cacheable` (with a short TTL, e.g., 5–30 seconds) to `EntityServiceImpl.getByAliasAndType()` keyed on the alias string, to absorb repeated lookups of the same alias.
3. **Do not rely on Traefik middleware as a security control** — infrastructure-level config is optional and bypassable; defense must be in the application layer.

### Proof of Concept

```bash
# Single attacker, no credentials required
# Constant query string → parse cache hit every time
# Unique alias variable → new DB query every time

QUERY='query GetAccount($alias: String!) { account(input: { alias: $alias }) { balance } }'

for i in $(seq 1 100000); do
  ALIAS=$(python3 -c "import base64, os; print(base64.b32encode(os.urandom(20)).decode().rstrip('='))")
  curl -s -X POST http://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"$QUERY\", \"variables\": {\"alias\": \"$ALIAS\"}}" &
done
```

Each iteration: parse step skipped (cache hit on constant `$QUERY` string), `findByAlias()` executes with a unique `byte[]`, issuing a new SQL query. Running this in parallel loops from one or more IPs generates sustained DB load well above the 30% baseline threshold with no authentication or special access required.
