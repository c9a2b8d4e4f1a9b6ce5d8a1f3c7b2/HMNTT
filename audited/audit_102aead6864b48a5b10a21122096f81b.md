### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Concurrent `getSupply()` Requests with Timestamp Parameters

### Summary
The `GET /api/v1/network/supply` endpoint in the rest-java module accepts timestamp query parameters that, when present, unconditionally execute a heavy native SQL query (`getSupplyHistory`) against the `account_balance` table with no rate limiting, no caching, and no concurrency guard. An unprivileged attacker flooding this endpoint with many concurrent requests using varying timestamp values can exhaust the HikariCP database connection pool, denying service to all other API consumers.

### Finding Description
**Code path:**

In `NetworkController.java` (lines 132–150), the `GET /api/v1/network/supply` endpoint accepts up to two `TimestampParameter` values (`@Size(max = 2)`) with no authentication or rate limiting applied. [1](#0-0) 

This delegates to `NetworkServiceImpl.getSupply(Bound timestamp)`. When the bound is non-empty (i.e., any timestamp is supplied), the code unconditionally calls `accountBalanceRepository.getSupplyHistory(...)` at line 81. [2](#0-1) 

`getSupplyHistory` executes a heavy native PostgreSQL query: a CTE with `DISTINCT ON (ab.account_id)`, a `JOIN unnest(...)` over account ranges, a `WHERE consensus_timestamp BETWEEN :lowerTimestamp AND :upperTimestamp` filter, `ORDER BY`, and a final `SUM`/`MAX` aggregation over the entire matching result set. [3](#0-2) 

**Root cause — failed assumptions:**
1. No rate limiting exists in the rest-java module for this endpoint. The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`, `ThrottleProperties`) is scoped entirely to the `web3` module and is not wired into rest-java. [4](#0-3) 

2. No `@Cacheable` annotation is present on `getSupplyHistory()` or on `getSupply()`. Every request with a timestamp triggers a fresh DB query regardless of whether the same timestamp was queried moments before. [5](#0-4) 

3. The only input validation is `@Size(max = 2)` (limits parameters per request, not request rate) and a range sanity check (`minTimestamp > maxTimestamp` throws). Neither prevents concurrent flooding. [6](#0-5) 

4. The `statementTimeout` for rest-java is 10,000 ms (10 seconds), meaning each connection can be held for up to 10 seconds per query. [7](#0-6) 

The production monitoring alert fires when HikariCP active connections exceed 75% of pool max, confirming the pool is a finite, shared resource. [8](#0-7) 

### Impact Explanation
An attacker who exhausts the HikariCP connection pool causes all subsequent DB-backed requests across the entire rest-java service to queue or fail with connection timeout errors. This denies service to all legitimate API consumers (wallets, explorers, dApps) querying any rest-java endpoint, not just `/network/supply`. The impact is a full API-level DoS for the duration of the attack. There is no economic damage to the Hedera network itself, placing this squarely in the griefing/medium severity category.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a standard HTTP client. The attacker needs to sustain enough concurrent connections to fill the pool — achievable with a single machine using async HTTP libraries (e.g., `curl --parallel`, Python `aiohttp`, `wrk`). Varying the timestamp value (e.g., incrementing by 1 nanosecond each request) defeats any hypothetical query-level deduplication. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add rate limiting to rest-java**: Implement a per-IP or global request-rate limiter (e.g., Bucket4j, as already used in the `web3` module) applied to the `/api/v1/network/supply` endpoint, or globally across all rest-java endpoints.
2. **Cache `getSupplyHistory` results**: Apply `@Cacheable` with a short TTL (e.g., 15–30 seconds) keyed on `(lowerTimestamp, upperTimestamp)`. Supply data changes only at consensus intervals, so stale-by-seconds results are acceptable.
3. **Limit timestamp range granularity**: Snap the requested timestamp to the nearest balance snapshot interval before querying, reducing the effective key space and increasing cache hit rate.
4. **Reduce statement timeout or add a query semaphore**: Limit the number of concurrent `getSupplyHistory` executions in-flight at any time.
