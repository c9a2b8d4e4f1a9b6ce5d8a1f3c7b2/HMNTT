### Title
Missing Entity Existence Check for Numeric Account IDs in Token Allowance Endpoint

### Summary
`getAccountTokenAllowances()` calls `EntityService.getEncodedId()` which, for numeric-only inputs (e.g., `12345`), resolves the encoded ID purely in-memory without any database existence check. As a result, querying `/accounts/12345/allowances/tokens` for a non-existent account returns HTTP 200 with an empty allowance list instead of HTTP 404, making it impossible for callers to distinguish "account exists with no allowances" from "account does not exist."

### Finding Description

**Exact code path:**

In `getAccountTokenAllowances()`, the account ID is resolved via:

```js
// tokenAllowanceController.js:69
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```

Inside `getEncodedId()` (`entityService.js:118-137`), the branching logic is:

```js
if (EntityId.isValidEntityId(entityIdString)) {
    const entityId = EntityId.parseString(entityIdString, {paramName});
    return entityId.evmAddress === null
      ? entityId.getEncodedId()                                        // ← pure in-memory, NO DB check
      : await this.getEntityIdFromEvmAddress(entityId, requireResult); // ← DB check
} else if (AccountAlias.isValid(entityIdString)) {
    return await this.getAccountIdFromAlias(...);                       // ← DB check
}
```

`EntityId.isValidEntityId()` (`entityId.js:133-136`) accepts numeric-only strings via `encodedEntityIdRegex = /^-?\d{1,19}$/` (`entityId.js:38`). For input `"12345"`, this regex matches, `evmAddress` is `null`, and `getEncodedId()` returns the encoded integer directly — **no database lookup occurs**.

The `isValidAccount()` method (`entityService.js:60-63`) exists and performs the correct existence check via `entityExistenceQuery`, but it is **never called** in this flow.

`TokenAllowanceService.getAccountTokenAllowances()` then executes `SELECT * FROM token_allowance WHERE owner = $1` (`tokenAllowanceService.js:57`), which returns zero rows for a non-existent owner. The controller returns HTTP 200 with `{ allowances: [], links: { next: null } }`.

**Why existing checks fail:**
- EVM address inputs (`0x...`) → DB existence check via `getEntityIdFromEvmAddress()` ✓
- Alias inputs → DB existence check via `getAccountIdFromAlias()` ✓
- Numeric/encoded ID inputs → **no DB check** ✗

### Impact Explanation
Smart contracts or off-chain systems integrating with this API endpoint cannot distinguish between two semantically different states: (1) an account that exists but holds no token allowances, and (2) an account that does not exist at all. Authorization logic that relies on a 404 to gate behavior (e.g., "if account not found, reject the operation") will silently receive a 200 with an empty list, potentially allowing operations to proceed under false assumptions about account state. This is a logic-level integrity issue in any system that treats the mirror node as a source of truth for allowance state.

### Likelihood Explanation
No authentication or special privileges are required. Any external user can craft a GET request with an arbitrary numeric account ID. The attack is trivially repeatable, requires no tooling beyond a standard HTTP client, and is completely silent — the server returns a well-formed 200 response with no error indicators. Automated smart contract integrations are particularly susceptible since they typically rely on HTTP status codes for control flow.

### Recommendation
After resolving the encoded ID for numeric inputs, add an existence check before proceeding. The `isValidAccount()` method already implements this correctly:

```js
// In getEncodedId(), after: return entityId.getEncodedId()
// Add:
const encodedId = entityId.getEncodedId();
if (requireResult && !(await this.isValidAccount(encodedId))) {
  throw new NotFoundError();
}
return encodedId;
```

Alternatively, call `isValidAccount()` directly in `getAccountTokenAllowances()` after resolving the ID, consistent with how other endpoints handle this pattern.
