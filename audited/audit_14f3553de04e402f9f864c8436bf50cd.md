### Title
Unbounded Timestamp Range in Hook Storage Historical Query Enables Full Table Scan DoS

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts timestamp parameters with no maximum range validation. Any unauthenticated user can supply a single open-ended timestamp (e.g., `timestamp=gte:0`) or a range spanning the entire chain history, causing `HookServiceImpl.getHookStorageChange()` to invoke `hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` with `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`. This forces a full sequential scan of the `hook_storage_change` table, driving node I/O well above normal operating levels.

### Finding Description

**Entry point** — `HooksController.getHookStorage()`: [1](#0-0) 

The `timestamps` array is constrained only to `@Size(max = 2)` — no range-width limit exists.

**Bound construction** — `hookStorageChangeRequest()` calls: [2](#0-1) 

`Bound.of()` is invoked with `primarySortField=false`: [3](#0-2) 

**Critical gap** — the only range-validity check inside `Bound`'s constructor is gated on `primarySortField`: [4](#0-3) 

Because `primarySortField=false` for timestamp bounds, no exception is thrown regardless of range width.

**Default bounds** — when only one side is supplied (or neither), the missing side defaults to the extreme: [5](#0-4) [6](#0-5) 

`lower == null → 0`; `upper == null → Long.MAX_VALUE`.

**Service layer** — `getHookStorageChange()` uses these values directly: [7](#0-6) 

**Repository** — the resulting SQL performs `consensus_timestamp BETWEEN 0 AND 9223372036854775807` with `DISTINCT ON (key)`, which requires a full scan and sort of every row for the given `owner_id`/`hook_id`: [8](#0-7) 

The `Pageable` limit (max 100 rows returned) does **not** prevent the database from scanning all matching rows before `DISTINCT ON` deduplication is applied.

### Impact Explanation
A single HTTP GET with `timestamp=gte:0` forces a full historical scan of `hook_storage_change` for the targeted `owner_id`/`hook_id`. On a chain with significant history, this translates to unbounded disk I/O and CPU for sort/dedup. Repeated in parallel across multiple `owner_id`/`hook_id` combinations, this can saturate I/O and raise node resource consumption well above 30% without any brute-force volume of requests. The endpoint requires no authentication.

### Likelihood Explanation
The exploit requires zero privileges — no wallet, no token, no API key. The attack surface is a standard public REST endpoint. The payload is a single query parameter (`timestamp=gte:0`). Any attacker who reads the API documentation (which explicitly lists `gte:` as a supported operator) can reproduce this immediately. It is repeatable and parallelizable.

### Recommendation
1. **Enforce a maximum timestamp range width** in `Bound.of()` when called for the hook storage timestamp parameter, analogous to the JavaScript REST API's `maxTimestampRange` config check.
2. Alternatively, require **both** a lower and upper timestamp bound when the historical path is taken, rejecting open-ended ranges with a `400 Bad Request`.
3. Add a database-level query timeout or a query cost guard for `hook_storage_change` queries.

### Proof of Concept

```
# Full history scan — lower bound only, upper defaults to Long.MAX_VALUE
GET /api/v1/accounts/0.0.1/hooks/1/storage?timestamp=gte:0

# Equivalent full-span range
GET /api/v1/accounts/0.0.1/hooks/1/storage?timestamp=gte:0&timestamp=lte:9999999999

# Upper-bound only — lower defaults to 0
GET /api/v1/accounts/0.0.1/hooks/1/storage?timestamp=lte:9999999999
```

All three forms pass `@Size(max=2)` validation, produce a non-empty `Bound` (so `isHistorical()` returns `true`), and route to `getHookStorageChange()` with `timestampLowerBound=0` / `timestampUpperBound=Long.MAX_VALUE` (or near-max), triggering the full-table `DISTINCT ON` scan.
