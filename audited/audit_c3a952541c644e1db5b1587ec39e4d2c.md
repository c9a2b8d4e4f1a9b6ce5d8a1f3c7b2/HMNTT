### Title
Finite Retry Budget Exhaustion via Mirror REST 5xx Responses Permanently Blocks `newClient()` in `network=other` Mode

### Summary
When the pinger is configured with `network=other`, `buildNetworkFromMirrorNodes()` fetches node topology from a mirror REST endpoint. An attacker who controls that endpoint can return HTTP 500 on every request, causing the finite retry loop to exhaust all attempts and return a fatal error. Because `newClient()` propagates this error with no recovery path, the pinger is permanently unable to submit transactions until manually restarted.

### Finding Description
In `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()` computes a fixed attempt budget:

```go
attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)  // default: 11
``` [1](#0-0) 

Inside `fetchMirrorNodeNetwork()`, any HTTP status ≥ 500 sets `retry = true` and returns an error:

```go
retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
``` [2](#0-1) 

The retry loop in `buildNetworkFromMirrorNodes()` re-enters on `retry=true` until `attempt == attempts`, then falls through and returns `nil, lastErr` — a hard failure with no subsequent recovery attempt: [3](#0-2) 

`newClient()` propagates this error directly to the caller with no fallback:

```go
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
if err != nil {
    return nil, err
}
``` [4](#0-3) 

**Root cause**: HTTP 5xx is treated as a retryable transient error, but the retry budget is finite and bounded by `cfg.mirrorNodeClientMaxRetries` (default 10). A mirror REST server that consistently returns 500 will consume every retry slot, and the resulting error is treated as fatal with no re-initialization path.

### Impact Explanation
When `newClient()` fails, the pinger cannot build a Hiero SDK client and cannot submit any transactions. This constitutes a complete denial-of-service for transaction submission — the exact scenario described as "Critical: Network not being able to confirm new transactions (total network shutdown)" in the scope. The pinger remains broken until an operator manually intervenes and restarts the process. No in-process recovery or periodic re-initialization exists. [5](#0-4) 

### Likelihood Explanation
The precondition is controlling the mirror REST server pointed to by `HIERO_MIRROR_PINGER_REST`. In `network=other` deployments this is typically a self-hosted or third-party mirror node. An attacker who has compromised the mirror node (which is a read-only indexer, a lower-value target than consensus nodes), or who can perform DNS poisoning or a MITM attack on the unencrypted HTTP endpoint (the default example uses `http://mirror-rest:5551`), can trivially return HTTP 500 on every request. The attack is repeatable: even if the pinger is restarted, the attacker simply continues returning 500 to re-trigger the failure. No authentication or integrity check on the mirror REST response exists. [6](#0-5) 

### Recommendation
1. **Retry at the `newClient()` / main-loop level**: Do not treat `buildNetworkFromMirrorNodes` failure as a one-shot fatal error. Wrap `newClient()` in a persistent retry loop with a longer backoff so the pinger can recover when the mirror node becomes available again.
2. **Cap per-attempt timeout and add a global deadline**: Pass a context with a deadline into `buildNetworkFromMirrorNodes` so a hung or slow mirror node cannot stall the process indefinitely.
3. **Enforce TLS / server authentication** for the mirror REST endpoint to prevent MITM-based 500 injection.
4. **Distinguish permanent vs. transient failures**: A mirror node returning 500 should not be treated identically to a network timeout. Consider a separate, longer-lived retry strategy for infrastructure-level errors vs. application-level errors.

### Proof of Concept
1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>/api/v1`.
2. Configure the attacker-controlled HTTP server to return `HTTP 500 Internal Server Error` on every `GET /api/v1/network/nodes` request.
3. Start the pinger. Observe that `buildNetworkFromMirrorNodes` retries 11 times (default), each returning `retry=true` due to the 5xx status.
4. After the 11th attempt, `buildNetworkFromMirrorNodes` returns `nil, lastErr`.
5. `newClient()` returns `nil, err`; the pinger exits or enters a broken state with no Hiero client.
6. No transactions are submitted. The pinger remains non-functional until manually restarted — and will fail again immediately on restart as long as the attacker's server continues returning 500. [7](#0-6)

func buildNetworkFromMirrorNodes(ctx context.Context, cfg config) (map[string]hiero.AccountID, error) {
	base := strings.TrimRight(strings.TrimSpace(cfg.mirrorRest), "/")

	var url string
	if strings.HasSuffix(base, "/api/v1") {
		url = base + "/network/nodes"
	} else {
		url = base + "/api/v1/network/nodes"
	}

	httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}

	attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)

	var lastErr error

	for attempt := 1; attempt <= attempts; attempt++ {
		network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
		if err == nil {
			return network, nil
		}

		lastErr = fmt.Errorf("attempt %d/%d: %w", attempt, attempts, err)
		if !retry || attempt == attempts {
			break
		}

		backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
		select {
		case <-ctx.Done():
			return nil, ctx.Err()
		case <-time.After(backoff):
		}
	}

	return nil, lastErr
}
```
