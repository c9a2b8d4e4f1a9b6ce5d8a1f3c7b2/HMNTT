### Title
Unauthenticated Log-Flooding DoS via Unbounded `InvalidArgumentError` Logging in `handleError()`

### Summary
The Node.js REST API has no rate-limiting middleware. Every request that triggers an `InvalidArgumentError` (HTTP 400) causes `handleError()` to call `logger.warn()` with the full, attacker-controlled `req.originalUrl` and `errorMessage`. When a crafted URL produces a log line exceeding the logger's 4096-byte internal buffer, the logger falls through to a synchronous `process.stdout.write()` call per request, creating unbounded synchronous I/O pressure on the single-threaded Node.js event loop and degrading REST API availability.

### Finding Description

**Exact code path:**

`rest/middleware/httpErrorHandler.js`, lines 19 and 31–35:
```js
} else if (err instanceof InvalidArgumentError || err instanceof RangeError) {
    statusCode = httpStatusCodes.BAD_REQUEST;          // line 19-20
}
if (shouldReturnMessage(statusCode)) {                 // true for all 4xx
    errorMessage = err.message;
    logger.warn(
      `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode} ${err.constructor.name} ${errorMessage}`
    );                                                 // lines 33-35
}
```

`req.originalUrl` is the raw, attacker-supplied URL string with no length cap applied before logging.

**Logger internals** (`rest/logger.js`, lines 73–88):
```js
#write(line) {
    const bytes = Buffer.byteLength(line);
    if (this.#position + bytes > this.#bufferSize) {
      this.#flush();                          // flush existing buffer
    }
    if (bytes > this.#bufferSize) {           // line > 4096 bytes
      process.stdout.write(line);             // SYNCHRONOUS write, bypasses buffer
      return;
    }
    this.#buffer.write(line, this.#position);
    this.#position += bytes;
}
```

The buffer is a fixed 4096-byte pre-allocated `Buffer.allocUnsafe(4096)` (line 33). It cannot grow. When a log line exceeds 4096 bytes, the code falls through to a **synchronous** `process.stdout.write(line)` call. In a high-concurrency flood, each oversized log line blocks the event loop for the duration of the syscall.

**Root cause / failed assumption:** The code assumes that log lines will be short and that the 4096-byte buffer will absorb bursts. It does not account for an attacker deliberately crafting long URLs to force the synchronous write path, nor does any middleware enforce a per-IP or global request rate limit.

**No rate limiting in the REST API stack** (`rest/server.js`, lines 68–144): The middleware chain is: `httpContext` → `requestLogger` → `authHandler` → optional metrics → optional cache check → routes → `responseHandler` → `handleError`. There is no rate-limiting middleware anywhere. The throttle configuration found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the separate Spring Boot web3 service, not to this Node.js REST API.

### Impact Explanation
An attacker flooding the REST API with requests containing URLs longer than ~4000 bytes and an invalid parameter value (e.g., `GET /api/v1/transactions?timestamp=AAAA...AAAA[x4000]`) will cause every request to trigger a synchronous `process.stdout.write()` in the single-threaded Node.js process. This saturates the event loop, increasing latency and eventually causing request timeouts for all legitimate clients of the mirror node REST API. Because the mirror node REST API is the primary interface through which downstream applications (wallets, explorers, dApps) query transaction data, sustained flooding degrades the availability of transaction query services.

### Likelihood Explanation
No authentication or rate limiting is required. Any external user with network access to the REST API port can execute this attack with a simple HTTP client loop. The attack is trivially repeatable and scriptable (e.g., `while true; do curl "https://host/api/v1/transactions?timestamp=$(python3 -c 'print("A"*5000)')"; done`). The invalid parameter path is guaranteed to reach `handleError()` without touching the database, making each request cheap for the attacker but expensive for the server's I/O path.

### Recommendation
1. **Add rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit`) before the route handlers in `rest/server.js`, keyed on `req.ip`.
2. **Truncate `req.originalUrl` in log output** inside `handleError()` to a safe maximum (e.g., 512 characters) before interpolating it into the log string, preventing oversized log lines from triggering the synchronous write path.
3. **Cap URL/query-string length** at the HTTP server or reverse-proxy layer (e.g., nginx `large_client_header_buffers`) so malformed long URLs are rejected before reaching Express.

### Proof of Concept
```bash
# Craft a URL with an invalid timestamp and a query string > 4096 bytes
PADDING=$(python3 -c "print('&x=' + 'A'*4100)")
while true; do
  curl -s "http://<mirror-node-host>/api/v1/transactions?timestamp=INVALID${PADDING}" &
done
```

**Expected result:** Each request returns HTTP 400, and `handleError()` calls `logger.warn()` with a log line >4096 bytes, triggering `process.stdout.write()` synchronously on every iteration. Under sustained concurrency, Node.js event-loop latency climbs, legitimate REST API requests time out, and the mirror node REST service becomes unavailable.
