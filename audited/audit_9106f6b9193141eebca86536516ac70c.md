### Title
Unbounded DB Query Amplification via Uncached Negative Lookups in `findRuntimeBytecode()`

### Summary
`ContractRepository.findRuntimeBytecode()` uses `@Cacheable` with `unless = "#result == null"`, but Spring Cache unwraps `Optional<byte[]>` before evaluating the `unless` SpEL expression — meaning `Optional.empty()` (returned for any non-existent contract) is treated as a null result and is **never stored in the cache**. An unprivileged attacker can craft a single `eth_call` that triggers EVM opcodes (`EXTCODESIZE`/`CALL`) against many distinct non-existent contract addresses, causing one uncached DB query per address per call, with no per-query gas accounting to bound the amplification.

### Finding Description

**Exact code location:**

`ContractRepository.java` lines 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

`ContractBytecodeReadableKVState.java` lines 39–47 — the EVM calls this on every bytecode lookup:
```java
protected Bytecode readFromDataSource(@NonNull ContractID contractID) {
    final var entityId = toEntityId(contractID);
    return contractRepository
            .findRuntimeBytecode(entityId.getId())
            .map(Bytes::wrap)
            .map(Bytecode::new)
            .orElse(null);
}
```

**Root cause — failed assumption:**

Spring Cache with `Optional<T>` return type unwraps the Optional before evaluating `unless`. When the method returns `Optional.empty()` (contract not found), Spring sees the unwrapped value as `null`, so `#result == null` evaluates to `true`, and the result is **excluded from the cache**. The developer's intent was to cache all results except hard nulls, but the actual effect is that every lookup for a non-existent contract bypasses the cache and hits the database.

The cache configuration (`CacheProperties.java` line 22) sets `maximumSize=1000`, so even for existing contracts the cache is bounded — but for non-existent IDs the cache is never populated at all.

**Exploit flow:**

1. Attacker deploys (or references) a contract whose bytecode loops over `EXTCODESIZE` calls to a large set of distinct, non-existent addresses (e.g., sequential IDs not in the DB).
2. Each `EXTCODESIZE` cold-access costs 2,600 EVM gas. With the 15M gas cap (`maxGasLimit = 15_000_000`), a single call can issue ≈5,769 such operations.
3. Each operation reaches `ContractBytecodeReadableKVState.readFromDataSource()` → `findRuntimeBytecode()` → cache miss → live SQL query: `select runtime_bytecode from contract where id = :contractId`.
4. Because the result is `Optional.empty()`, it is never cached. The next call with the same IDs repeats all queries.
5. The gas throttle (`gasPerSecond = 7_500_000_000`, `requestsPerSecond = 500`) allows up to 500 max-gas calls per second, yielding up to **~2.88 million uncached DB queries per second** from a single attacker.

**Why existing checks are insufficient:**

- The gas throttle (`ThrottleManagerImpl.java` lines 37–48) accounts for EVM gas consumed, not for the number of DB queries triggered per gas unit. The EVM gas cost for `EXTCODESIZE` (2,600 gas) is fixed and does not scale with DB query latency or load.
- The request-rate limit (500 req/s) bounds the number of API calls but not the DB fan-out within each call.
- The `unless = "#result == null"` guard was intended to prevent caching of absent results but, combined with Spring's Optional unwrapping, it achieves the opposite of negative caching — it guarantees every absent-contract lookup is a DB round-trip.

### Impact Explanation

An attacker can sustain a high rate of uncached DB queries against the PostgreSQL backend, causing CPU and I/O saturation on the database, increased query latency for all users, and potential connection pool exhaustion. Because the mirror node is a read-only query service (no on-chain state), the attacker pays no real cost (no HBAR, no privileged access required) and can repeat the attack indefinitely. Severity: **High** for availability, **None** for confidentiality/integrity.

### Likelihood Explanation

The attack requires only the ability to submit `eth_call` requests to the public `/api/v1/contracts/call` endpoint — no authentication, no privileged account, no on-chain transaction. The attacker needs only to know or enumerate non-existent contract IDs (sequential integers work). The attack is fully repeatable and automatable. Any public Hedera mirror node deployment is exposed.

### Recommendation

1. **Fix negative caching:** Change the `unless` condition to also cache empty Optional results. Use `unless = "#result?.isEmpty() ?: true"` or switch the return type to `@Nullable byte[]` with `unless = "#result == null"` so the semantics are unambiguous.
2. **Alternatively:** Cache a sentinel value for absent contracts (e.g., an empty byte array) and distinguish it from a real empty bytecode at the call site.
3. **Rate-limit DB fan-out:** Introduce a per-request cap on the number of distinct contract bytecode lookups (e.g., via a request-scoped counter in `ContractCallContext`), rejecting or short-circuiting calls that exceed the limit.
4. **Gas cost calibration:** Consider charging higher EVM gas for cold bytecode lookups that miss the cache, aligning the EVM cost model with actual backend cost.

### Proof of Concept

**Preconditions:** Public access to `POST /api/v1/contracts/call`. No account or privileged access needed.

**Steps:**

1. Identify a range of contract IDs that do not exist in the mirror node DB (e.g., IDs 999000001–999005769).
2. Deploy (or simulate via `eth_call` with `to: null`) a contract whose constructor/function executes:
   ```solidity
   for (uint i = 999000001; i < 999005769; i++) {
       address target = address(uint160(i));
       uint size;
       assembly { size := extcodesize(target) }
   }
   ```
3. Submit the call:
   ```json
   POST /api/v1/contracts/call
   { "data": "<encoded_loop_bytecode>", "gas": 15000000 }
   ```
4. Observe via DB monitoring (e.g., `pg_stat_activity`) that ~5,769 queries `select runtime_bytecode from contract where id = ?` are issued against the DB for this single API call.
5. Repeat at the maximum allowed rate (500 req/s). Observe DB CPU/IO saturation and degraded response times for all other users.
6. Confirm no caching occurs by repeating the same call with the same IDs — DB query count is identical on every invocation.
