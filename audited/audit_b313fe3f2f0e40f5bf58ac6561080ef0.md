### Title
Unbounded `endTime` Allows Indefinite gRPC Subscription Causing Resource Exhaustion

### Summary
An unprivileged user can set `endTime` to `Long.MAX_VALUE` in a `TopicMessageFilter`, which passes all existing validation checks. This causes `takeWhile`, `isComplete()`, and `pastEndTime()` to never terminate the subscription, allowing the caller to hold a live streaming connection open indefinitely. With no per-client subscription limit enforced in application code, many such connections can be opened simultaneously, exhausting server-side resources.

### Finding Description

**Exact code path:**

`TopicMessageFilter.java` — validation on `endTime`: [1](#0-0) 

The only constraint is `endTime > startTime`. There is no upper bound. `Long.MAX_VALUE` satisfies `Long.MAX_VALUE > startTime` trivially.

`TopicMessageServiceImpl.java` — `subscribeTopic()`, the `takeWhile` guard: [2](#0-1) 

With `endTime = Long.MAX_VALUE`, `t.getConsensusTimestamp() < Long.MAX_VALUE` is always `true` for any real consensus timestamp (nanoseconds since epoch; `Long.MAX_VALUE` nanoseconds ≈ year 2262). The stream never terminates via this check.

`TopicMessageServiceImpl.java` — `isComplete()`: [3](#0-2) 

`Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to approximately year 2262. `Instant.now().isAfter(year_2262 + 30s)` is always `false`, so `isComplete()` never returns `true`.

`TopicMessageServiceImpl.java` — `pastEndTime()`, the only other termination signal for the live flux: [4](#0-3) 

Because `isComplete()` never returns `true`, `RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)` repeats up to `Long.MAX_VALUE` times (effectively forever), polling every 30 seconds and never emitting a termination signal.

`TopicMessageServiceImpl.java` — limit check: [5](#0-4) 

`limit` has `@Min(0)` but no `@Max`. Setting `limit = 0` disables the limit entirely (`hasLimit()` returns `false`). Setting `limit = Long.MAX_VALUE` requires delivering ~9.2×10¹⁸ messages before terminating — also effectively never. [6](#0-5) 

**Root cause:** No upper bound is enforced on `endTime` or `limit`, and no per-client or global concurrent-subscription cap exists in application code. The `isComplete()` and `takeWhile` termination paths both silently become no-ops when `endTime = Long.MAX_VALUE`.

### Impact Explanation

Each open subscription holds:
- A live gRPC/Netty connection and its associated thread/buffer resources
- A `TopicContext` object with `AtomicLong`, `AtomicReference`, and `Stopwatch`
- A `pastEndTime()` repeating flux that fires a DB/listener poll every 30 seconds indefinitely
- An incremented `subscriberCount` gauge (masking the true load)

An attacker opening hundreds or thousands of such subscriptions exhausts connection pool capacity, heap memory, and scheduled-task threads, degrading or denying service to legitimate subscribers. There is no application-level rate limit or max-subscriptions-per-IP guard visible in this code path. [7](#0-6) 

### Likelihood Explanation

No authentication or authorization is required to call `subscribeTopic`. The gRPC endpoint is publicly accessible. The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK), setting two fields (`endTime = Long.MAX_VALUE`, `limit = 0`), and opening connections in a loop. This is trivially scriptable and repeatable with no special privileges.

### Recommendation

1. **Cap `endTime`**: Add a `@Max` constraint or a custom validator in `TopicMessageFilter` that rejects `endTime` values beyond a reasonable horizon (e.g., current time + configurable max duration such as 1 hour or 24 hours).
2. **Cap `limit`**: Add a `@Max` constraint on the `limit` field (e.g., `@Max(10_000)` or a configurable property).
3. **Enforce a maximum subscription duration**: In `TopicMessageServiceImpl`, apply a `flux.timeout(grpcProperties.getMaxSubscriptionDuration())` regardless of `endTime`.
4. **Per-client connection throttling**: Enforce a maximum number of concurrent subscriptions per client IP at the Netty/gRPC interceptor layer.
5. **Validate `endTime` upper bound in `isValidEndTime()`**: Reject values that would make `isComplete()` unreachable within a reasonable operational window.

### Proof of Concept

```java
// Using Hedera Java SDK or any gRPC client:
TopicMessageFilter filter = TopicMessageFilter.builder()
    .topicId(EntityId.of(0, 0, 1234))   // any valid topic
    .startTime(DomainUtils.now() - 1)    // just before now (passes isValidStartTime)
    .endTime(Long.MAX_VALUE)             // passes isValidEndTime: MAX_VALUE > startTime
    .limit(0)                            // no limit; hasLimit() returns false
    .build();

// Open N connections in a loop:
for (int i = 0; i < 10_000; i++) {
    stub.subscribeTopic(filter, noopObserver());
}
// Each connection: takeWhile never fires, isComplete() never returns true,
// pastEndTime() polls every 30s forever. Server resources exhausted.
```
