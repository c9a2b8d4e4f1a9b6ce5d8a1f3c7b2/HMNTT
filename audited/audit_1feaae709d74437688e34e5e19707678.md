### Title
Multi-Rule Match Amplification in `ThrottleManagerImpl.throttle()` — Unbounded `action()` Invocations Per Request

### Summary
The `throttle()` method iterates over every configured `RequestProperties` entry and calls `action()` for each one whose `test()` returns true, with no "first-match-wins" or deduplication guard. An unprivileged caller can craft a single `ContractCallRequest` whose fields simultaneously satisfy multiple configured rules, causing `log.info()` to fire N times (log amplification) or N separate per-rule `Bucket.tryConsume(1)` calls (multi-bucket depletion) for a single inbound request.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.java`, `throttle()`, lines 44–48:
```java
for (var requestFilter : throttleProperties.getRequest()) {
    if (requestFilter.test(request)) {
        action(requestFilter, request);   // called once per matching rule
    }
}
```
`action()` at lines 66–75 dispatches on `ActionType`:
- `LOG` → `log.info("{}", request)` — one log line per matching rule.
- `THROTTLE` → `filter.getBucket().tryConsume(1)` — one token consumed from each matching rule's independent bucket.

**Root cause / failed assumption:**
The design assumes each `RequestProperties` entry is an independent, non-overlapping rule. There is no `break`, no `return`, and no "already-handled" flag after the first match. The `RequestProperties.test()` method (lines 45–61) evaluates its own `filters` list with OR semantics and returns `true` if any inner `RequestFilter` matches. Because `ContractCallRequest` exposes seven independently filterable fields (`BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, `VALUE`), a single request can trivially satisfy rules keyed on different fields simultaneously.

**Exploit flow:**

Suppose the operator has configured (a realistic multi-rule setup):
```yaml
hiero.mirror.web3.throttle.request:
  - action: LOG
    filters: [{field: DATA, type: CONTAINS, expression: "0x"}]
  - action: LOG
    filters: [{field: FROM, type: CONTAINS, expression: "0x"}]
  - action: LOG
    filters: [{field: TO,   type: CONTAINS, expression: "0x"}]
  - action: THROTTLE
    rate: 50
    filters: [{field: ESTIMATE, type: EQUALS, expression: "false"}]
  - action: THROTTLE
    rate: 50
    filters: [{field: GAS, type: CONTAINS, expression: "100"}]
```
An attacker sends:
```json
{"data":"0xdeadbeef","from":"0xAAA...","to":"0xBBB...","estimate":false,"gas":100000}
```
Result per request:
- 3 × `log.info()` calls (rules 1–3 all match).
- 2 × `bucket.tryConsume(1)` calls (rules 4–5 both match), depleting two independent THROTTLE buckets simultaneously.

**Why existing checks are insufficient:**

- The global `rateLimitBucket` (line 38) and `gasLimitBucket` (line 40) fire once per request and are unaffected.
- Each `RequestProperties.test()` has its own `counter` and `rate` guard (lines 46–51), but these are per-rule and do not prevent the same request from matching multiple rules.
- The test `requestMultipleActions` (test file lines 160–166) confirms the multi-fire behavior is untested for abuse: it only asserts that at least one log line appears, not that exactly one fires.

### Impact Explanation

**Log amplification (griefing):** With N overlapping LOG rules and the global rate limit of 500 req/s, an attacker sustains up to 500 × N `log.info()` calls per second. With N=10 rules (a plausible production config), this is 5 000 log lines/second from a single attacker at the rate cap, potentially exhausting disk space or overwhelming log aggregation pipelines.

**THROTTLE bucket depletion:** A request matching M THROTTLE rules drains M independent per-rule buckets simultaneously. Legitimate traffic that only matches one rule is then throttled by a bucket that was depleted by requests targeting a different rule, causing unintended cross-rule rate-limit interference.

Severity matches the stated scope: griefing with no direct economic damage, but measurable operational impact.

### Likelihood Explanation

- No authentication or special privilege is required; any caller of the JSON-RPC endpoint can send a crafted request.
- Overlapping filter configurations are natural and likely in production (e.g., one rule per field type, or a catch-all rule plus specific rules).
- The attacker does not need to know the exact filter configuration; sending requests with all fields populated with common substrings (e.g., `"0x"`) maximises the chance of matching multiple rules.
- The attack is fully repeatable and automatable up to the global `requestsPerSecond` cap (default 500/s).

### Recommendation

1. **First-match-wins semantics:** Break out of the loop after the first `RequestProperties` whose `action()` does not throw:
   ```java
   for (var requestFilter : throttleProperties.getRequest()) {
       if (requestFilter.test(request)) {
           action(requestFilter, request);
           break; // stop after first matching rule
       }
   }
   ```
2. **Alternatively, deduplicate by action type:** If multi-rule evaluation is intentional, ensure at most one `LOG` and one `THROTTLE` action fires per request.
3. **Document the intended semantics** explicitly so operators understand that overlapping rules multiply side-effects.

### Proof of Concept

**Preconditions:**
Configure two or more `RequestProperties` entries whose filters can be simultaneously satisfied by a single request (e.g., one matching `DATA CONTAINS "0x"` and one matching `FROM CONTAINS "0x"`).

**Steps:**
1. Start the web3 mirror node with the overlapping rule configuration above.
2. Send a single `eth_call` / `eth_estimateGas` JSON-RPC request:
   ```json
   {
     "jsonrpc":"2.0","method":"eth_call","id":1,
     "params":[{"from":"0xAAAA...","to":"0xBBBB...","data":"0xdeadbeef","gas":"0x186A0"},"latest"]
   }
   ```
3. Observe the application log: **two or more** `ContractCallRequest(...)` lines appear for the single request, confirming `log.info()` was called once per matching LOG rule.
4. For THROTTLE rules: configure two THROTTLE rules each with `rate=1`. Send two requests. The second request is rejected even though each individual bucket should still have capacity, because the first request drained both buckets simultaneously. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    private void action(RequestProperties filter, ContractCallRequest request) {
        switch (filter.getAction()) {
            case LOG -> log.info("{}", request);
            case REJECT -> throw new ThrottleException("Invalid request");
            case THROTTLE -> {
                if (!filter.getBucket().tryConsume(1)) {
                    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
                }
            }
        }
