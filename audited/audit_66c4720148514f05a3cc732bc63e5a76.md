### Title
SharedPollingTopicListener Timestamp Reset on Re-subscription Causes Permanent Message Loss After Network Partition

### Summary
`SharedPollingTopicListener` constructs a single shared `Flux<TopicMessage> topicMessages` with `.share()` and a single `PollingContext` instance. When all subscribers disconnect (e.g., during a network partition), `.share()` cancels the upstream; when the next subscriber connects, `doOnSubscribe(context::onStart)` fires and unconditionally resets `lastConsensusTimestamp` to `DomainUtils.now()`, causing all subsequent polls to start from the reconnection instant and permanently skipping every message produced during the partition window. No privilege is required — any gRPC client that reconnects after the last subscriber has left triggers the reset.

### Finding Description

**Exact code path:**

`SharedPollingTopicListener` constructor (`SharedPollingTopicListener.java` lines 39–52):
```java
PollingContext context = new PollingContext();   // single shared instance, line 39

topicMessages = Flux.defer(() -> poll(context).subscribeOn(scheduler))
        .repeatWhen(...)
        .doOnCancel(() -> log.info("Cancelled polling"))
        .doOnError(...)
        .doOnSubscribe(context::onStart)          // line 50 — fires on every upstream re-subscription
        .retryWhen(...)
        .share();                                 // line 52 — refCount; upstream cancelled at refcount=0
```

`PollingContext.onStart` (`SharedPollingTopicListener.java` lines 101–106):
```java
void onStart(Subscription subscription) {
    lastConsensusTimestamp.set(DomainUtils.now());   // unconditional reset to wall-clock "now"
    ...
}
```

`poll()` uses that timestamp as the exclusive lower bound for the DB query (`SharedPollingTopicListener.java` lines 65–66):
```java
return Flux.fromIterable(topicMessageRepository.findLatest(
        context.getLastConsensusTimestamp().get(), listenerProperties.getMaxPageSize()))
```

`findLatest` SQL (`TopicMessageRepository.java` lines 12–25):
```sql
where consensus_timestamp > ?1   -- ?1 = lastConsensusTimestamp (now after reset)
```

**Root cause:** `.share()` is `publish().refCount()`. When subscriber count drops to 0 the upstream is cancelled. On the next subscription a brand-new upstream subscription is created, which fires `doOnSubscribe` → `onStart` → `lastConsensusTimestamp = DomainUtils.now()`. The single `PollingContext` is never re-created (it is captured in the constructor closure), so the reset permanently advances the polling cursor past the partition window.

**Failed assumption:** The design assumes the upstream is never cancelled while messages of interest are being produced. During a network partition this assumption is violated: all clients drop, the upstream is cancelled, messages continue to be written to the DB, and the cursor is then jumped forward past all of them.

**Why the `filterMessage` check in `CompositeTopicListener` (lines 61–64) is insufficient:**
```java
private boolean filterMessage(TopicMessage message, TopicMessageFilter filter) {
    return message.getTopicId().equals(filter.getTopicId())
            && message.getConsensusTimestamp() >= filter.getStartTime();
}
```
This filter operates only on messages already emitted by the shared Flux. Because `poll()` queries `consensus_timestamp > lastConsensusTimestamp` (which is now set to the reconnection instant), messages from the partition window are never fetched from the database at all — the filter never sees them.

### Impact Explanation
Any subscriber using `SHARED_POLL` mode who reconnects after a complete subscriber drain permanently loses every topic message whose `consensus_timestamp` falls between the last successful poll before the partition and the reconnection instant. Because `findLatest` uses a strict `>` predicate on the reset timestamp, those rows are skipped forever — there is no catch-up mechanism. For a consensus-message streaming service this constitutes silent, undetectable data loss: the subscriber's sequence numbers will have gaps with no error signal.

### Likelihood Explanation
Network partitions (even brief ones — a rolling pod restart, a load-balancer timeout flush, a Kubernetes node eviction) routinely drain all active gRPC streams simultaneously. No attacker capability is required; the vulnerability fires automatically under normal operational conditions. An adversary who can cause even a brief network interruption (e.g., a targeted TCP RST injection or a short-lived DDoS against the gRPC port) can reliably trigger the reset on demand. The default `REDIS` listener type reduces exposure, but any deployment configured with `type: SHARED_POLL` is fully affected.

### Recommendation
1. **Do not reset `lastConsensusTimestamp` on re-subscription.** Remove `doOnSubscribe(context::onStart)` from the shared Flux chain, or guard `onStart` so it only initialises the timestamp when it is still at its zero/unset value:
   ```java
   void onStart(Subscription subscription) {
       lastConsensusTimestamp.compareAndSet(0L, DomainUtils.now());
   }
   ```
2. **Alternatively, use `publish().autoConnect(0)`** (warm-start) instead of `.share()` so the upstream is never cancelled when subscribers leave, eliminating the re-subscription entirely.
3. **Add a gap-detection / catch-up path** analogous to `PollingTopicListener`, which correctly uses `filter.getStartTime()` as the initial cursor and advances only by observed message timestamps.

### Proof of Concept

**Preconditions:**
- Mirror node configured with `hiero.mirror.grpc.listener.type=SHARED_POLL`
- At least one topic with active message production

**Steps:**

1. Client A subscribes to topic T via gRPC `subscribeTopic`. The shared Flux starts; `onStart` sets `lastConsensusTimestamp = T0`.
2. Messages M1, M2, M3 are produced and delivered to Client A (cursor advances to `T3`).
3. Simulate a network partition: terminate all active gRPC connections (e.g., `kill -9` the client process, or block the gRPC port with a firewall rule). `.share()` sees refcount → 0 and cancels the upstream.
4. During the partition, messages M4, M5, M6 are written to the database with timestamps `T4 < T5 < T6`.
5. Restore connectivity. Client B (any unprivileged user) subscribes to topic T.
6. `.share()` creates a new upstream subscription → `doOnSubscribe` fires → `onStart` sets `lastConsensusTimestamp = DomainUtils.now()` ≈ `T_reconnect > T6`.
7. `poll()` queries `findLatest(T_reconnect, pageSize)` → returns 0 rows (M4–M6 have timestamps < `T_reconnect`).
8. **Result:** Client B never receives M4, M5, M6. The gap is permanent and silent.
