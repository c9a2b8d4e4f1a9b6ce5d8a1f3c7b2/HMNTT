### Title
Safety-Check Flux Exhausts `boundedElastic` Thread Pool via Blocking `Flux.fromStream()` on Partitioned Database

### Summary
Every unprivileged gRPC subscriber triggers a safety-check Flux after 1 second of inactivity. That check calls `PollingTopicMessageRetriever.poll()`, which wraps a synchronous blocking JDBC `Stream` with `Flux.fromStream()` on a `boundedElastic` thread. During a network partition the JDBC call hangs indefinitely; the `.timeout()` operator cannot interrupt a thread blocked in synchronous I/O, so the thread is held permanently. An attacker opening many concurrent subscriptions can exhaust the `boundedElastic` pool and starve all other reactive operations.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` constructs the safety-check Flux:

```java
// TopicMessageServiceImpl.java lines 67-70
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());   // ← pins a boundedElastic thread
``` [1](#0-0) 

After 1 second with no data, `missingMessages(topicContext, null)` is called. Because `current == null`, it immediately delegates to the retriever:

```java
// TopicMessageServiceImpl.java lines 142-149
if (current == null) {
    ...
    return topicMessageRetriever.retrieve(gapFilter, false);
}
``` [2](#0-1) 

`PollingTopicMessageRetriever.retrieve()` eventually calls `poll()`:

```java
// PollingTopicMessageRetriever.java line 78
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [3](#0-2) 

`Flux.fromStream()` accepts a `java.util.stream.Stream<T>` — a **synchronous, blocking** construct backed by a JDBC `ResultSet`. The subscribing thread (a `boundedElastic` thread, pinned by `subscribeOn`) iterates the stream inline. If the database is unreachable, the JDBC driver blocks that thread until the OS-level TCP timeout fires (default: minutes to hours on Linux with no `tcp_keepalive` tuning).

**Why the `.timeout()` is ineffective:**

```java
// PollingTopicMessageRetriever.java lines 51-62
return Flux.defer(() -> poll(context))
        .repeatWhen(...)
        .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
        .timeout(retrieverProperties.getTimeout(), scheduler)   // ← outer operator
        ...
``` [4](#0-3) 

Reactor's `.timeout()` works by racing the source against a timer and sending a cancellation signal when the timer wins. However, a cancellation signal is cooperative — it cannot preempt a thread that is blocked inside a synchronous `Stream.iterator()` / JDBC `ResultSet.next()` call. The timer fires, the cancel is enqueued, but the blocked thread never checks it. The `boundedElastic` thread remains held.

**Root cause:** `Flux.fromStream()` wrapping a blocking JDBC stream is fundamentally incompatible with Reactor's non-blocking cancellation model. The safety-check's `subscribeOn(Schedulers.boundedElastic())` routes this blocking call onto a managed thread pool that has a finite upper bound.

### Impact Explanation

`Schedulers.boundedElastic()` has a default cap of `10 × CPU cores` threads. Each concurrent subscriber that has received no data for 1 second holds one thread for the duration of the JDBC hang. On a 4-core node the pool cap is 40 threads. An attacker opening 40 concurrent subscriptions to a low-traffic (or non-existent) topic exhausts the pool entirely. All other reactive operations that require a `boundedElastic` thread — including other subscribers, health checks, and internal maintenance tasks — queue indefinitely, causing a full service hang. This is a denial-of-service against the gRPC mirror node service.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is unauthenticated by default (standard Hedera mirror node deployment). An attacker needs only a gRPC client and knowledge of any valid topic ID (or even an invalid one if `checkTopicExists` is disabled). The 1-second trigger is deterministic and requires no race condition. The attack is trivially repeatable: open connections, wait 1 second, partition fires. No special privileges, credentials, or insider knowledge are required.

### Recommendation

1. **Replace `Flux.fromStream()` with a non-blocking R2DBC query** in `PollingTopicMessageRetriever.poll()`. The rest of the stack already uses R2DBC; the historical retriever is the only outlier using a blocking JDBC stream.
2. **If blocking JDBC must be retained**, wrap the call with `Mono.fromCallable(() -> ...).subscribeOn(Schedulers.boundedElastic())` and apply a `Mono.timeout()` that calls `Thread.interrupt()` on the blocking thread, or use a dedicated, bounded thread pool separate from the shared `boundedElastic` scheduler.
3. **Set a JDBC socket timeout** (`socketTimeout` / `connectTimeout` on the datasource) to a low value (e.g., 5–10 seconds) so the JDBC driver self-terminates hung connections independently of Reactor's cancellation.
4. **Rate-limit concurrent safety-check retrievals** per subscriber or globally to prevent pool exhaustion even if the above fixes are not immediately applied.

### Proof of Concept

**Preconditions:**
- Mirror node gRPC service is running and reachable.
- A valid topic ID exists (or `grpcProperties.checkTopicExists = false`).
- Network partition between the mirror node and its database is simulated (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the mirror node host, applied after the JDBC connection pool is established so connections hang rather than fail fast).

**Steps:**

1. Establish the network partition (drop DB packets silently so TCP connections hang rather than RST).
2. Open `N` concurrent gRPC streams to `subscribeTopic` for a topic with no recent messages, where `N ≥ boundedElastic thread cap` (default: `10 × CPU cores`):
   ```
   for i in $(seq 1 50); do
     grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
       mirror-node:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
   done
   ```
3. Wait 1 second. Each stream's safety-check fires, calls `PollingTopicMessageRetriever.poll()`, and blocks a `boundedElastic` thread on the hung JDBC connection.
4. Attempt any other operation that requires a `boundedElastic` thread (e.g., a new subscription, a health-check endpoint backed by reactive code). Observe it hangs indefinitely.
5. **Result:** The mirror node gRPC service is fully unresponsive until the OS-level TCP timeout clears the hung connections (minutes to hours).

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
