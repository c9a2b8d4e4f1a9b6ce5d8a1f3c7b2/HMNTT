### Title
Indefinite Scheduler Thread Exhaustion via Full-Page Poll Manipulation in Throttled `PollingTopicMessageRetriever`

### Summary
The `isComplete()` heuristic in `PollingTopicMessageRetriever` returns `false` whenever the last poll returned exactly `maxPageSize` messages, and the throttled path sets `numRepeats = Long.MAX_VALUE`. The `.timeout()` operator applied to the flux is a Reactor **per-item inactivity timeout**, not a total-duration cap — so as long as items keep flowing (one full page every `pollingFrequency`), the timeout never fires. An unprivileged subscriber targeting any topic with a large backlog of historical messages can hold a `boundedElastic` scheduler thread open indefinitely, and by opening many concurrent subscriptions can exhaust the shared scheduler pool, denying service to all other subscribers.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java`, `isComplete()`, lines 121–129:
```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;
}
```
When `pageSize.get() == maxPageSize` (last poll was exactly full) and no limit is set, this returns `false`.

**Root cause — failed assumption:** The design assumes that a full page implies "more messages may exist, keep polling," and that eventually a partial page will signal completion. This assumption fails when a topic has a large historical backlog: every poll returns exactly `maxPageSize` messages, so `isComplete()` never returns `true`.

**Throttled path wires `numRepeats = Long.MAX_VALUE`** (lines 98–101):
```java
if (throttled) {
    numRepeats = Long.MAX_VALUE;
    ...
}
```

**The `repeatWhen` loop** (lines 51–55):
```java
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                .withFixedDelay(context.getFrequency())
                .withScheduler(scheduler))
```
With `isComplete()` always `false` and `numRepeats = Long.MAX_VALUE`, this loop never exits.

**Why the timeout does not save it** (line 59):
```java
.timeout(retrieverProperties.getTimeout(), scheduler)
```
`Flux.timeout(Duration)` in Reactor fires only if **no item is emitted** within the duration. Default `pollingFrequency = 2s` and `maxPageSize = 1000` means 1000 items are emitted every 2 seconds — far below the 60-second inactivity threshold. The timeout never triggers.

**`pageSize` reset per poll** (line 73):
```java
context.getPageSize().set(0L);
```
`pageSize` is reset to 0 at the start of each poll and incremented by `onNext`. So `pageSize.get()` at the time `isComplete()` is evaluated reflects only the most recent poll's count, not a cumulative total. A topic with ≥ `maxPageSize` messages available at every poll window keeps `pageSize == maxPageSize` indefinitely.

### Impact Explanation
Each stuck subscription holds a thread on the shared `Schedulers.boundedElastic()` pool (default cap: `10 × CPU cores`). An attacker opening `N` concurrent gRPC subscriptions to a high-backlog topic can exhaust the pool, causing all subsequent `retrieve()` calls to queue indefinitely. This is a full **Denial of Service** against the historical message retrieval path for all users of the mirror node gRPC service. The `subscriberCount` gauge will show inflated counts, and legitimate subscribers will stall waiting for a scheduler thread.

### Likelihood Explanation
No authentication is required to open a gRPC subscription. The attacker does not need to publish any messages — they only need to target an existing topic with a large historical backlog (common on mainnet for popular topics). A single attacker can open many concurrent connections from one machine. The attack is repeatable: after the scheduler is exhausted, new connections queue indefinitely. The only cost to the attacker is network connections; no HBAR spend is required if targeting an existing high-traffic topic.

### Recommendation

1. **Add a hard wall-clock timeout** independent of item flow. Replace or supplement the inactivity `.timeout()` with a total-duration cap using `Flux.take(Duration)` or a `takeUntil` based on a `Mono.delay`:
   ```java
   .take(retrieverProperties.getTimeout())  // total duration, not inactivity
   ```

2. **Cap `numRepeats` for throttled mode** to a finite value (e.g., same as `unthrottled.getMaxPolls()` scaled up), preventing `Long.MAX_VALUE` repeat loops.

3. **Treat `pageSize == maxPageSize` as ambiguous, not as "keep going."** After a configurable number of consecutive full-page polls with no limit set, force completion or switch to the live listener path.

4. **Enforce a per-connection subscription limit** at the gRPC layer to bound the number of concurrent subscriptions per client IP.

### Proof of Concept

**Preconditions:**
- A topic exists on the mirror node with ≥ 10,000 historical messages (common on mainnet).
- Default config: `maxPageSize=1000`, `pollingFrequency=2s`, `timeout=60s`.

**Steps:**
1. Open a gRPC `subscribeTopic` stream with `startTime=0`, no `limit`, targeting the high-backlog topic.
2. `TopicMessageServiceImpl.subscribeTopic()` calls `topicMessageRetriever.retrieve(filter, true)` (throttled=true).
3. `PollingContext` is created with `numRepeats=Long.MAX_VALUE`, `maxPageSize=1000`.
4. Poll 1 returns 1000 messages → `pageSize=1000` → `isComplete()` = `(1000 < 1000)` = `false`.
5. Poll 2 returns messages 1001–2000 → `pageSize=1000` → `isComplete()` = `false`.
6. Items flow every 2 seconds; the 60-second inactivity timeout never fires.
7. Repeat step 4–6 in a loop: the subscription never completes, holding a `boundedElastic` thread.
8. Open 50+ concurrent such subscriptions from the same unprivileged client.
9. `boundedElastic` thread pool is exhausted; all new `retrieve()` calls block; legitimate subscribers receive no historical messages.
