### Title
Stale NFT Allowance Records Returned for Deleted Accounts via Numeric ID Bypass in `EntityService.lookup()`

### Summary
When a numeric entity ID (e.g., `0.0.12345`) is supplied to `GET /api/v1/accounts/{id}/allowances/nfts`, `EntityServiceImpl.lookup()` short-circuits the entity table lookup and directly returns the parsed `EntityId` without verifying the account exists or is not deleted. This is inconsistent with alias and EVM address lookups, which both enforce `deleted <> true`. Any unprivileged user can query NFT allowance records belonging to a deleted account by supplying its former numeric ID.

### Finding Description
In `EntityServiceImpl.lookup()` (line 32), the `EntityIdNumParameter` branch returns `Optional.of(p.id())` directly, bypassing the `entity` table entirely. In contrast, `EntityIdAliasParameter` and `EntityIdEvmAddressParameter` both query `EntityRepository` with `deleted <> true` guards (lines 13 and 16 of `EntityRepository.java`). After `lookup()` returns the raw `EntityId`, `NftAllowanceServiceImpl` passes it directly to `NftAllowanceRepository.findAll(request, id)` (line 30 of `NftAllowanceServiceImpl.java`). The repository queries the `nft_allowance` table using this ID with no secondary check on the entity's existence or deletion status. If an account was deleted but its allowance rows were not purged (or were only soft-deleted in the entity table), those rows are returned to the caller.

### Impact Explanation
An attacker can retrieve NFT allowance records associated with accounts that no longer exist on the network. This constitutes incorrect data export from the mirror node — the mirror node presents stale state as current state. Downstream consumers (wallets, dApps, indexers) relying on this endpoint may make incorrect authorization or ownership decisions based on revoked or invalid allowances. Severity is medium: no private data is exposed, but data integrity of the mirror node's public API is violated.

### Likelihood Explanation
Exploitation requires zero privileges — any HTTP client can issue a GET request. The attacker only needs to know (or enumerate) a formerly valid numeric account ID, which is trivially discoverable from historical ledger data or prior API responses. The condition (deleted account with residual allowance rows) is realistic in any active network where accounts are routinely deleted.

### Recommendation
In the `EntityIdNumParameter` branch of `EntityServiceImpl.lookup()`, perform the same entity table validation as the other branches. Specifically, call `entityRepository.findById(p.id().getId())` and check that the result is present and not deleted, throwing `EntityNotFoundException` otherwise. Alternatively, add a unified post-lookup existence check in `NftAllowanceServiceImpl.getNftAllowances()` using `entityService.findById(id)` before querying the allowance repository.

### Proof of Concept
1. Create account `0.0.99999` on the network and grant NFT allowances.
2. Delete account `0.0.99999`.
3. Send: `GET /api/v1/accounts/0.0.99999/allowances/nfts`
4. Observe: the response returns the previously granted NFT allowance records despite the account being deleted, because `EntityServiceImpl.lookup()` returns `EntityId(0,0,99999)` without any DB validation, and the `nft_allowance` table still contains the rows.
