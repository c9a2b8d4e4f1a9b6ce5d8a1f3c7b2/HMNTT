### Title
Unauthenticated DoS via Unindexed Full Entity Table Scan in `/balances` No-Timestamp Path

### Summary
In `rest/balances.js`, the `getBalances()` function takes a structurally different code path when no `timestamp` filter is supplied. This `else` branch builds a CTE that unconditionally scans the entire `entity` table, then applies the `public_key` filter on the CTE result rather than pushing it down to the indexed column. Because no application-level rate limiting exists on this endpoint, an unauthenticated attacker can flood the endpoint with `?account.publickey=` requests, each forcing a full entity-table scan, saturating database CPU and rendering the mirror node unavailable.

### Finding Description

**Exact code path — `rest/balances.js`, `getBalances()`, lines 113–149:**

When `tsQuery` is falsy (no `timestamp` parameter supplied), execution falls into the `else` branch:

```
} else {
    const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
    const whereClause = conditions && `where ${conditions}`;
    ...
    sqlQuery = `
      with entity_balance as (
        select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key

      )

      from entity_balance ab

      order by ab.account_id ${order}
      ${limitQuery}`;

**Root cause — two compounding issues:**

1. The `entity_balance` CTE always performs `SELECT … FROM entity WHERE type IN ('ACCOUNT','CONTRACT')`. PostgreSQL materializes CTEs as optimization fences in versions < 12; even in 12+ the planner may choose to materialize. Once materialized, the CTE result is an in-memory relation with no index. The `public_key` filter in the outer `WHERE` clause therefore requires a sequential scan of every row in the materialized CTE.

2. Contrast with the `if (tsQuery)` branch, which calls `getAccountIdsByPublicKey()` — a separate, bounded query that hits the entity table directly with `public_key = $1 LIMIT $2`, allowing the planner to use the `entity__public_key_type` index: [2](#0-1) 

The index `entity__public_key_type on entity(public_key, type)` exists: [3](#0-2) 

But it is unreachable from the outer query once the CTE is materialized.

**`pubKeyQuery` construction** — `parsePublicKeyQueryParam` produces a parameterized fragment like `public_key = ?` bound to the caller-supplied value. No SQL injection is possible, but the query shape itself is the weapon: [4](#0-3) 

**No application-level rate limiting** exists in the REST middleware stack for this endpoint. The throttle infrastructure found in the repository is scoped exclusively to the `web3` module (`ThrottleManagerImpl`, `ThrottleConfiguration`), not to the REST API: [5](#0-4) 

### Impact Explanation
Each request without a `timestamp` parameter and with a `public_key` parameter forces PostgreSQL to: (a) scan every ACCOUNT/CONTRACT row in the `entity` table to materialize the CTE, then (b) filter that materialized result by `public_key` without an index. On a production Hedera mirror node the entity table contains tens of millions of rows. Concurrent flooding of this endpoint saturates database CPU, causing query queues to back up across all mirror-node REST endpoints. This degrades or eliminates the mirror node's ability to serve balance, transaction, and account queries. While the mirror node itself is read-only and does not directly confirm network transactions, operators and wallets depend on it for balance lookups and transaction status; sustained unavailability constitutes a significant service disruption.

### Likelihood Explanation
Preconditions are minimal: no account, no token, no authentication. The attacker only needs to know the public API URL and supply any syntactically valid 64/66/88-character hex string as `account.publickey`. The attack is trivially scriptable (`curl` in a loop or any HTTP flood tool), repeatable indefinitely, and amplified by the absence of per-IP or per-endpoint rate limiting in the application layer.

### Recommendation
1. **Short-term**: In the `else` branch, replace the inline `pubKeyQuery` with the same two-step resolution already used in the `if (tsQuery)` branch — call `getAccountIdsByPublicKey()` first, return early if no match, and inject the resulting `account_id IN (…)` clause. This keeps the `public_key` lookup on the indexed entity table rather than on the unindexed CTE result.
2. **Medium-term**: Add application-level rate limiting (e.g., express-rate-limit or a Traefik middleware equivalent to what is already deployed for the rosetta service) to the `/api/v1/balances` route, keyed by source IP.
3. **Long-term**: Annotate the CTE with `NOT MATERIALIZED` (PostgreSQL 12+) to guarantee predicate push-down when the two-step approach is not used.

### Proof of Concept
```bash
# Any syntactically valid 64-hex-char public key; no account needed
PK="aabbccddaabbccddaabbccddaabbccddaabbccddaabbccddaabbccddaabbccdd"

# Single request — triggers full entity table scan
curl "https://<mirror-node>/api/v1/balances?account.publickey=${PK}"

# Flood — saturates DB CPU (no authentication, no rate limit)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/balances?account.publickey=${PK}" &
done
wait
```

Expected result: database CPU spikes to 100 %; subsequent queries (balances, transactions, accounts) time out or return 503.
