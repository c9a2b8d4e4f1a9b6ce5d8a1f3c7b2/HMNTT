### Title
TOCTOU Race in `initGenesisRecordFile` Allows Concurrent `selectGenesis` Query Flooding During Startup Window

### Summary
`initGenesisRecordFile` performs an unsynchronized early-exit check on `br.genesisBlock.ConsensusStart` before acquiring a GORM session and executing the expensive `selectGenesis` CTE query, with `sync.Once` placed only after the query completes. During the startup window (before genesis is initialized), any number of concurrent goroutines can pass the unsynchronized check simultaneously, each borrowing a physical DB connection from the pool and executing the full `selectGenesis` query. An unprivileged attacker who floods the Rosetta API with concurrent block-lookup requests during this window can exhaust the connection pool and spike DB CPU/memory well above baseline.

### Finding Description

**Exact code path:**

`rosetta/app/persistence/block.go`, `initGenesisRecordFile()`, lines 243–263:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {  // line 244 – unsynchronized read
        return nil
    }

    db, cancel := br.dbClient.GetDbWithContext(ctx)   // line 248 – GORM session + context
    defer cancel()

    var rb recordBlock
    if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {  // lines 252-255 – borrows pool conn + executes CTE
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
    }

    br.once.Do(func() {          // line 257 – sync.Once is AFTER the query
        br.genesisBlock = rb
    })
    ...
}
```

**Root cause — failed assumption:**
The developer assumed the early check at line 244 would act as a fast-path guard equivalent to `sync.Once`. It does not. The read of `br.genesisBlock.ConsensusStart` is an unsynchronized struct-field access with no mutex or atomic. `sync.Once` is invoked only after the DB query completes (line 257), so it guards only the write to `br.genesisBlock`, not the query execution itself.

**Exploit flow:**
1. During startup, `br.genesisBlock.ConsensusStart` equals `genesisConsensusStartUnset` (-1).
2. N concurrent goroutines all read line 244 simultaneously; all see `-1`; all proceed past the guard.
3. Each calls `GetDbWithContext` (creates a GORM session with context — `rosetta/app/db/client.go` lines 22–37), then `.First()` which borrows a physical connection from the `database/sql` pool (`rosetta/app/db/db.go` line 33: `sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)`).
4. All N goroutines execute the full `selectGenesis` CTE query (lines 49–72 of `block.go`) concurrently — a query involving a CTE, a JOIN across `account_balance` and `record_file`, a correlated subquery, and an `ORDER BY`.
5. Only one goroutine's result is committed via `sync.Once`; the rest discard their results but have already consumed pool connections and DB CPU.
6. If genesis initialization keeps failing (DB not yet populated), `br.genesisBlock` is never set, the early check never fires, and the window remains open indefinitely — every incoming request repeats steps 2–5.

**Why existing checks fail:**
- The early check (line 244) is a plain struct-field read with no `sync.Mutex`, `sync/atomic`, or `sync.Once` protection — it is a data race under the Go memory model.
- `sync.Once` (line 257) only serializes the write to `br.genesisBlock`; it does not prevent multiple concurrent query executions.
- `GetDbWithContext` applies only a statement-level timeout (`statementTimeout`), not a concurrency limit.

### Impact Explanation
Each concurrent request borrows one connection from the bounded pool (`MaxOpenConnections`, default 100 in test config at `rosetta/test/db/db.go` line 103) and issues the `selectGenesis` CTE. With the pool saturated, legitimate requests queue or time out. The `selectGenesis` query scans `account_balance` (filtered by `treasury_entity_id`, ordered, limited) and joins against `record_file` with a correlated subquery — under production data volumes this is non-trivial. Saturating the pool with this query concurrently causes measurable DB CPU and memory spikes. If the genesis record is absent (node still syncing), the condition persists across the entire uptime of the node, not just a brief startup window, making the attack continuously repeatable.

### Likelihood Explanation
No authentication is required to call any Rosetta block endpoint (`/block`, `/block/transaction`). All four public methods (`FindByHash`, `FindByIdentifier`, `FindByIndex`, `RetrieveGenesis`, `RetrieveLatest`) call `initGenesisRecordFile` before doing anything else. An attacker needs only HTTP access to the Rosetta port and the ability to issue concurrent requests — a trivial capability. The attack is especially effective during node startup or re-sync periods when genesis is not yet populated, keeping the vulnerable window open indefinitely.

### Recommendation
Move `sync.Once` to wrap both the guard check and the query execution, eliminating the TOCTOU window entirely:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })
    return initErr
}
```

Note: if the first attempt fails (DB not ready), `sync.Once` will not retry. A retry-aware pattern using a mutex + boolean flag, or `golang.org/x/sync/singleflight`, is preferable so that transient failures during startup do not permanently prevent initialization while still collapsing concurrent requests into a single in-flight query.

### Proof of Concept

**Preconditions:**
- Rosetta node is starting up (or DB does not yet contain the genesis `account_balance` row).
- Attacker has HTTP access to the Rosetta API port.

**Steps:**

```bash
# Send 200 concurrent requests to /block (index 0) during node startup
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:<port>/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":0}}' &
done
wait
```

**Expected result:**
- All 200 goroutines pass the unsynchronized check at line 244.
- Up to `MaxOpenConnections` (e.g., 100) concurrent `selectGenesis` CTE queries execute simultaneously.
- DB CPU and memory spike; connection pool is saturated; legitimate requests receive timeouts or `ErrNodeIsStarting` errors.

func (d *client) GetDbWithContext(ctx context.Context) (*gorm.DB, context.CancelFunc) {
	if d.statementTimeout <= 0 {
		db := d.db
		if ctx != nil {
			db = db.WithContext(ctx)
		}
		return db, noop
	}

	if ctx == nil {
		ctx = context.Background()
	}
