### Title
Global Rate-Limit Token Pre-Consumed Without Rollback When Per-Filter THROTTLE Bucket Rejects Request

### Summary
In `ThrottleManagerImpl.throttle()`, the global `rateLimitBucket` token is irrevocably consumed before the per-filter `ActionType.THROTTLE` bucket is checked. When the per-filter bucket is exhausted and throws a `ThrottleException`, the already-consumed global token is never refunded. An unprivileged attacker who knows (or probes) a contract address matching a `FilterField.TO` + `ActionType.THROTTLE` filter can flood requests at the global rate limit, exhausting the global bucket while the per-filter bucket rejects the majority of those requests, denying service to all users system-wide.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.throttle()` executes in this order:

1. **Line 38** — global token consumed unconditionally: [1](#0-0) 

2. **Lines 44–48** — per-filter evaluation happens *after* the global token is gone: [2](#0-1) 

3. **Lines 70–73** — per-filter THROTTLE bucket check; throws on exhaustion but does NOT refund the global token: [3](#0-2) 

**Root cause:** There is no rollback or `addTokens()` call on `rateLimitBucket` when the per-filter THROTTLE check fails. The `restore()` method only refunds gas tokens, not rate-limit tokens: [4](#0-3) 

**Per-filter bucket capacity is bounded at 0–100** (the `rate` field), far below the global default of 500 req/s: [5](#0-4) [6](#0-5) 

**The `to` field is fully attacker-controlled** — validation is only a hex-format check, no authentication or address allowlist: [7](#0-6) 

**`FilterField.TO` extracts the attacker-supplied value directly:** [8](#0-7) 

**Exploit flow (concrete numbers):**
- Global `rateLimitBucket` capacity: 500 tokens/s (default)
- Per-filter THROTTLE bucket capacity: 100 tokens/s (default `rate`)
- Attacker sends 500 req/s with `to = <filtered contract address>`
- Requests 1–100: consume 1 global token + 1 per-filter token each → per-filter bucket exhausted
- Requests 101–500: consume 1 global token each at line 38, then fail at line 71 → `ThrottleException` thrown, global token already gone
- Net: all 500 global tokens consumed per second; global bucket exhausted; 0 tokens remain for any other user

### Impact Explanation
Once the global `rateLimitBucket` is exhausted, every subsequent request from any user — regardless of which contract they target — is rejected with `REQUEST_PER_SECOND_LIMIT_EXCEEDED`. This is a complete, system-wide denial of service for the entire web3 API endpoint. The attacker achieves 5× amplification: spending 100 "legitimate" per-filter tokens to drain 500 global tokens, because the 400 rejected-but-already-consumed global tokens are wasted. Severity is **High** — full availability loss with no authentication barrier.

### Likelihood Explanation
The attack requires zero privileges. The attacker only needs to discover one contract address matching a configured `FilterField.TO` + `ActionType.THROTTLE` filter, which can be found by probing known popular contract addresses or reading public configuration. The attack is trivially automatable with any HTTP client sending parallel requests, is repeatable every second (buckets refill greedily), and leaves no persistent state to clean up. A single attacker machine can sustain the flood indefinitely.

### Recommendation
Refund the global `rateLimitBucket` token when a per-filter THROTTLE check rejects the request. In `ThrottleManagerImpl.action()`, before throwing the `ThrottleException` for the THROTTLE case, call `rateLimitBucket.addTokens(1)`. Alternatively, restructure `throttle()` to evaluate all per-filter predicates *before* consuming the global token, only consuming it once all checks are confirmed to pass. Also consider checking the per-filter bucket availability (without consuming) before consuming the global token, using `bucket.getAvailableTokens()` as a pre-flight guard.

### Proof of Concept

**Preconditions:**
- A `RequestProperties` entry is configured with `action: THROTTLE`, `rate: 100`, and a filter `field: TO`, `type: EQUALS`, `expression: "0x000000000000000000000000000000000000abcd"`.
- Global `requestsPerSecond: 500`.

**Steps:**
1. Start 500 concurrent threads, each sending in a tight loop:
   ```
   POST /api/v1/contracts/call
   { "to": "0x000000000000000000000000000000000000abcd", "gas": 21000 }
   ```
2. Observe that after the first ~100 requests succeed, requests 101–500 receive HTTP 429 from the per-filter THROTTLE check — but the global bucket is now empty.
3. Immediately send a request with a completely different `to` address (not matching any filter):
   ```
   POST /api/v1/contracts/call
   { "to": "0x000000000000000000000000000000000000ffff", "gas": 21000 }
   ```
4. Observe HTTP 429 with `REQUEST_PER_SECOND_LIMIT_EXCEEDED` — the global bucket is exhausted even though this request never touched the per-filter bucket.
5. Wait 1 second (bucket refill), repeat from step 1 to sustain the DoS indefinitely.
