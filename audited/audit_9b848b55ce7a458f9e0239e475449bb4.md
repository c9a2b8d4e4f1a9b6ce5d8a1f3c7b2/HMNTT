### Title
Binary Search Gas Estimator Bypasses `gasLimitBucket` Throttle — Up to 21x EVM Amplification per `eth_estimateGas` Request

### Summary
Every `eth_estimateGas` request consumes the `gasLimitBucket` exactly once (for the user-supplied gas value) at the controller layer, but then silently triggers up to 20 additional full EVM executions inside `BinaryGasEstimator.search()`. Because those iterations call `doProcessCall(..., true)` with `estimate=true`, the `finally` block in `ContractCallService` skips `restoreGasToBucket()` and never charges the bucket again. An unprivileged attacker can therefore drive the server to perform ~21× the EVM work that the throttle was designed to permit, causing sustained CPU/memory exhaustion.

### Finding Description

**Throttle entry point — charged once per request**

`ContractController.call()` calls `throttleManager.throttle(request)` before any processing:

```java
// ContractController.java:40-41
throttleManager.throttle(request);
```

`ThrottleManagerImpl.throttle()` deducts `scaleGas(request.getGas())` tokens from `gasLimitBucket` exactly once:

```java
// ThrottleManagerImpl.java:40-41
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
```

**Estimate path — 1 initial call + up to 20 binary-search calls**

`ContractExecutionService.estimateGas()` first runs one real EVM call, then hands control to `BinaryGasEstimator.search()`:

```java
// ContractExecutionService.java:82-95
final var processingResult = callContract(params, context);          // EVM call #1
...
final var estimatedGas = binaryGasEstimator.search(
        (totalGas, iterations) -> updateMetrics(...),
        gas -> doProcessCall(params, gas, true),                     // EVM calls #2–#21
        gasUsedByInitialCall,
        params.getGas());
```

`BinaryGasEstimator.search()` loops up to `maxGasEstimateRetriesCount = 20` times:

```java
// BinaryGasEstimator.java:35
while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount()) {
```

**The gate that exempts all binary-search iterations from the throttle**

`doProcessCall()` only calls `restoreGasToBucket()` when `estimate == false`:

```java
// ContractCallService.java:128-135
if (!estimate) {
    restoreGasToBucket(result, params.getGas());
    if (result != null) {
        updateMetrics(params, result.gasUsed(), 1, status);
    }
}
```

The binary-search lambda passes `estimate = true`, so all 20 iterations execute the EVM (`transactionExecutionService.execute(params, estimatedGas)`) without ever touching the `gasLimitBucket`. No tokens are consumed, no tokens are restored — the bucket is completely blind to those executions.

**Root cause:** The failed assumption is that one token deduction at the HTTP layer is sufficient to bound the total EVM work per request. For `eth_call` this holds (1 deduction → 1 EVM call). For `eth_estimateGas` it does not (1 deduction → up to 21 EVM calls).

### Impact Explanation

With default settings (`gasPerSecond = 7_500_000_000`, `requestsPerSecond = 500`, `maxGasLimit = 15_000_000`, `maxGasEstimateRetriesCount = 20`):

- The `gasLimitBucket` is sized to absorb 500 requests/s × 15 M gas = 7.5 B gas/s of EVM work.
- Each `eth_estimateGas` request at gas=15 M actually causes up to 21 EVM executions ≈ 315 M gas of real computation.
- 500 such requests/s → ~157.5 B gas/s of actual EVM work, roughly **21× the designed capacity**.

The result is CPU saturation, JVM heap pressure from repeated EVM state loading, and potential OOM or request-queue exhaustion — a sustained denial-of-service against the mirror-node web3 service. No funds are at risk, but service availability is fully compromised.

### Likelihood Explanation

- **No authentication required.** The `/api/v1/contracts/call` endpoint is public; any external user can set `"estimate": true`.
- **No special contract needed.** Any deployed contract whose execution approaches the gas limit maximises binary-search iterations (the search runs all 20 rounds when `lo` and `hi` are far apart).
- **Trivially repeatable.** A single attacker with a modest HTTP client (e.g., `wrk`, `hey`, or a simple script) can sustain 500 req/s indefinitely.
- **No per-IP throttle exists** in the visible code; the `requestsPerSecond` bucket is global, so one attacker can consume the entire allowance.

### Recommendation

1. **Charge the bucket proportionally for estimate requests.** Before entering `binaryGasEstimator.search()`, multiply the gas cost by the expected iteration count (or a conservative multiplier such as `maxGasEstimateRetriesCount`) and attempt to consume that many tokens from `gasLimitBucket`. Reject with `ThrottleException` if insufficient tokens are available.

2. **Alternatively, apply a separate, lower `requestsPerSecond` cap for estimate requests** (e.g., `estimateRequestsPerSecond`) so that the amplification factor is absorbed at the rate-limit layer.

3. **Restore consumed gas after the binary search completes**, accounting for the total gas used across all iterations (the `totalGasUsed` value already accumulated in `BinaryGasEstimator.search()` and passed to `metricUpdater` could be used for this).

4. **Add per-IP / per-caller rate limiting** so a single source cannot monopolise the global `requestsPerSecond` budget.

### Proof of Concept

```bash
# Target: mirror-node web3 service (default port 8545 or 8080)
# Contract: any deployed contract whose execution uses close to maxGasLimit gas

# 1. Craft a request body that sets estimate=true and gas at the maximum
BODY='{
  "from": "0x0000000000000000000000000000000000000001",
  "to":   "<deployed_contract_address>",
  "data": "<calldata_for_expensive_function>",
  "gas":  15000000,
  "estimate": true
}'

# 2. Flood the endpoint at the maximum allowed rate (500 req/s)
#    Each request triggers 1 initial EVM call + up to 20 binary-search EVM calls
wrk -t4 -c50 -d60s \
    -s <(echo "
        wrk.method = 'POST'
        wrk.headers['Content-Type'] = 'application/json'
        wrk.body = '$BODY'
    ") \
    http://<mirror-node-host>/api/v1/contracts/call

# 3. Observe: CPU pegged at 100%, heap usage climbing, response latency
#    increasing for all other clients — sustained DoS with no special privileges.
```
