### Title
Unbounded `endTime` Allows Indefinite Subscription Hold via `pastEndTime()` Timer Resource Exhaustion

### Summary
An unprivileged gRPC client can set `endTime` to an arbitrarily far future timestamp (e.g., year 2100 in nanoseconds). `pastEndTime()` creates a `RepeatSpec` with `Long.MAX_VALUE` repetitions and a 30-second fixed delay, keeping the reactive subscription — and its associated timer, gRPC stream, and listener resources — open until that future time arrives. No upper bound on `endTime` is enforced, and no concurrent-subscription cap exists in the service layer, enabling an attacker to accumulate thousands of such subscriptions.

### Finding Description

**Code path:**

`TopicMessageFilter` validates `endTime` only as: [1](#0-0) 

```java
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

There is **no upper bound** on `endTime`. A client may supply any nanosecond timestamp in the future (e.g., year 2100 = `4102444800000000000L` ns).

`pastEndTime()` then constructs: [2](#0-1) 

```java
return Flux.empty()
        .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                .withFixedDelay(grpcProperties.getEndTimeInterval()));
```

`Long.MAX_VALUE` is the repeat ceiling; the actual termination is governed by `isComplete()`: [3](#0-2) 

```java
return Instant.ofEpochSecond(0, filter.getEndTime())
        .plus(grpcProperties.getEndTimeInterval())
        .isBefore(Instant.now());
```

`isComplete()` returns `true` only after `endTime + endTimeInterval` (default 30 s) has elapsed in real wall-clock time. With `endTime` set to year 2100, this never returns `true` for ~75 years.

The subscription is wired as: [4](#0-3) 

```java
Flux<TopicMessage> flux = historical
        .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
```

`takeUntilOther(pastEndTime(...))` keeps the merged live flux alive until `pastEndTime` emits — which it never does while `isComplete()` is false.

`subscriberCount` is tracked but never used to reject new subscriptions: [5](#0-4) 

**Root cause:** Missing upper-bound validation on `endTime` combined with `Long.MAX_VALUE` repeat count and a termination condition that is purely wall-clock-driven.

### Impact Explanation
Each malicious subscription holds:
- A Reactor timer firing every 30 seconds (scheduler thread pool slot)
- An open gRPC/Netty stream
- A live `TopicListener` database listener connection

With thousands of concurrent subscriptions (each with `endTime` decades away), the server's bounded-elastic scheduler, Netty event loop, and database connection pool can be exhausted, causing legitimate subscribers to be rejected or starved. This is a non-network-based DoS requiring no special privileges.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible to any client that can reach the mirror node's gRPC port. The attack requires only a valid topic ID (discoverable via the REST API) and a crafted `ConsensusSubscribeRequest` with a far-future `endTime`. It is trivially scriptable and repeatable; a single attacker machine can open thousands of connections.

### Recommendation
1. **Enforce a maximum `endTime` horizon** in `TopicMessageFilter` — e.g., reject any `endTime` more than a configurable window (e.g., 1 hour) beyond `now()`:
   ```java
   @AssertTrue(message = "End time must not be more than maxEndTimeFuture beyond now")
   public boolean isValidEndTime() {
       return endTime == null || (endTime > startTime && endTime <= DomainUtils.now() + MAX_END_TIME_OFFSET_NS);
   }
   ```
2. **Enforce a maximum concurrent subscriber count** in `subscribeTopic()`, returning a gRPC `RESOURCE_EXHAUSTED` error when the limit is exceeded.
3. **Cap the `RepeatSpec` count** to a finite value derived from `(endTime - now) / endTimeInterval` rather than `Long.MAX_VALUE`.

### Proof of Concept
```
# Precondition: mirror node gRPC port reachable, valid topicId known
# endTime = year 2100 in nanoseconds = 4102444800000000000

for i in $(seq 1 5000); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0},"consensusEndTime":{"seconds":4102444800}}' \
    <mirror-node>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Each connection holds a timer + stream open for ~75 years.
# After ~N connections the bounded-elastic scheduler / DB pool is exhausted;
# legitimate subscribers receive UNAVAILABLE errors.
```
