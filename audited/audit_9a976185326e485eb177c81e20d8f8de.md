### Title
Unbounded Growth of `cacheManagerSlotsPerContract` Internal Cache Map via Unique `contractId` Flooding

### Summary
In `ContractStateServiceImpl.findStorageBatch()`, each unique `contractId` that misses the outer bounded `contractSlotsCache` triggers a call to `cacheManagerSlotsPerContract.getCache(contractId.toString())`. Because `cacheManagerSlotsPerContract` is a `CaffeineCacheManager` configured in dynamic mode (no fixed cache names), it permanently registers every new cache name in its internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no eviction mechanism, so it grows without bound as unique `contractId` values are introduced, while the outer `contractSlotsCache` (which is bounded) silently evicts entries without cleaning up the corresponding entries in `cacheManagerSlotsPerContract`.

### Finding Description

**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 85–87.

```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause — two-layer cache mismatch:**

- `this.contractSlotsCache` (from `cacheManagerContractSlots`) is configured at `EvmConfiguration.java` lines 75–81 with `setCacheNames(Set.of(CACHE_NAME))` and a `maximumSize` from `cacheProperties.getContractSlots()`. It is a **bounded** Caffeine cache that maps `contractId → CaffeineCache`.
- `cacheManagerSlotsPerContract` is configured at `EvmConfiguration.java` lines 107–112 **without** calling `setCacheNames()`, placing it in `CaffeineCacheManager`'s dynamic mode. In dynamic mode, Spring's `CaffeineCacheManager.getCache(name)` creates a new `CaffeineCache` and stores it permanently in an internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has **no size limit and no eviction**.

**Exploit flow:**

1. For each new unique `contractId` not already in `this.contractSlotsCache`, the value loader `() -> cacheManagerSlotsPerContract.getCache(contractId.toString())` is invoked.
2. `CaffeineCacheManager.getCache()` creates a new `CaffeineCache` and inserts it into its internal `cacheMap` under the key `contractId.toString()`.
3. The resulting `CaffeineCache` is stored in `this.contractSlotsCache` (bounded).
4. When `this.contractSlotsCache` reaches its `maximumSize` and evicts an old `contractId` entry, the corresponding entry in `cacheManagerSlotsPerContract.cacheMap` is **never removed** — it remains permanently.
5. Repeating with N unique `contractId` values causes `cacheManagerSlotsPerContract.cacheMap` to accumulate N entries indefinitely.

**Why existing checks are insufficient:**

The outer `contractSlotsCache` bound only limits how many `CaffeineCache` objects are *actively referenced* by the outer cache. It does not prevent `cacheManagerSlotsPerContract.cacheMap` from retaining all ever-created cache entries. There is no listener, removal callback, or cleanup path that removes entries from `cacheManagerSlotsPerContract` when the outer cache evicts them.

The entry point `ContractStorageReadableKVState.readFromDataSource()` (lines 39–44) performs no validation that the `contractId` corresponds to an existing on-chain contract before calling `contractStateService.findStorage(entityId, keyBytes)`, so non-existent contract addresses flow through to `findStorageBatch()` without rejection.

### Impact Explanation

Each entry permanently retained in `cacheManagerSlotsPerContract.cacheMap` holds a `String` key and a `CaffeineCache` wrapping a Caffeine cache instance. Even an empty Caffeine cache carries several kilobytes of internal data structures (`ConcurrentHashMap`, internal queues, etc.). With millions of unique `contractId` values, the JVM heap is exhausted, causing `OutOfMemoryError` and crashing the mirror node web3 service. This denies EVM query service (`eth_call`, `eth_estimateGas`, contract storage reads) to all users.

### Likelihood Explanation

An attacker needs only to send repeated JSON-RPC requests (e.g., `eth_call` or `eth_getStorageAt`) referencing different contract addresses. Since Hedera EVM addresses are 20-byte values and the mirror node does not validate contract existence before invoking `findStorage`, the attacker can enumerate arbitrary addresses (e.g., incrementing a counter) at zero on-chain cost. No privileged access, no HBAR spend, and no special tooling is required — a simple HTTP client loop suffices. The attack is repeatable and persistent across restarts if the JVM crashes and the cache is re-populated.

### Recommendation

1. **Remove the dynamic-mode `cacheManagerSlotsPerContract` entirely.** Instead of using `CaffeineCacheManager.getCache(contractId)` to create per-contract named caches, store the per-contract `Cache` objects directly in the outer bounded `contractSlotsCache` (which already maps `contractId → CaffeineCache`). The `CaffeineCache` objects can be constructed directly via `new CaffeineCache(name, caffeineInstance)` without registering them in any `CacheManager`.

2. **Alternatively**, if the `CaffeineCacheManager` pattern must be kept, replace it with a `LoadingCache<EntityId, Cache>` (a plain Caffeine cache with a `maximumSize` and `expireAfterAccess`) that constructs `CaffeineCache` instances on demand and evicts them automatically — bypassing Spring's `CacheManager` registration entirely.

3. **Add input validation** in `ContractStorageReadableKVState.readFromDataSource()` or `ContractStateServiceImpl.findStorage()` to reject or short-circuit requests for `contractId` values that do not correspond to known entities, preventing the cache path from being reached for non-existent contracts.

### Proof of Concept

```python
import requests, itertools, threading

URL = "http://<mirror-node-web3>:8545"

def flood(start, count):
    for i in range(start, start + count):
        # Craft a unique contract address for each i
        addr = "0x" + hex(i)[2:].zfill(40)
        payload = {
            "jsonrpc": "2.0", "method": "eth_getStorageAt",
            "params": [addr, "0x0", "latest"], "id": i
        }
        requests.post(URL, json=payload)

# Launch parallel threads to accelerate cache map growth
threads = [threading.Thread(target=flood, args=(i * 100000, 100000)) for i in range(10)]
for t in threads: t.start()
for t in threads: t.join()
# After ~millions of unique addresses, the mirror node JVM heap is exhausted
# and the web3 service crashes with OutOfMemoryError
```

Each unique address maps to a unique `EntityId`, triggering one permanent entry in `cacheManagerSlotsPerContract.cacheMap` per address, with no upper bound.
