### Title
Unbounded Query String Logged Per Request in `TracingMiddleware` Enables Log Storage Exhaustion

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` captures the full request URI via `request.URL.RequestURI()` — which includes the raw query string — and logs it unconditionally with no length truncation. Because Go's `net/http` server accepts request lines up to `MaxHeaderBytes` (default 1 MB) and the Rosetta server sets no lower bound, any unauthenticated caller can send repeated requests with ~100 KB query strings, writing ~100 KB to the log per request and exhausting disk storage.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 47–58:

```go
path := request.URL.RequestURI()          // line 47 — full URI, query string included, no cap
// ...
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)  // line 52-53 — path written verbatim

if internalPaths[path] {                  // line 55 — key lookup uses full URI
    log.Debug(message)
} else {
    log.Info(message)                     // line 58 — logged at Info for every non-internal path
}
```

**Root cause:** `request.URL.RequestURI()` returns the encoded path **plus** the raw query string (e.g. `/health/liveness?a=AAAA…100 KB`). The result is stored in `path` and interpolated into `message` without any truncation before being passed to `log.Info`.

**Secondary amplifier — `internalPaths` bypass:** The map `internalPaths` is keyed on exact strings (`/health/liveness`, `/health/readiness`, `/metrics`). When a query string is appended, the key no longer matches, so the request is logged at `Info` instead of `Debug`, regardless of whether log-level filtering would otherwise suppress it.

**Server-side size limit:** `rosetta/main.go` lines 220–227 construct the `http.Server` with no `MaxHeaderBytes` field set:

```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
    // MaxHeaderBytes not set → defaults to DefaultMaxHeaderBytes = 1 MB
}
```

Go's `DefaultMaxHeaderBytes` is 1,048,576 bytes (1 MB) and covers the request line (method + URL + HTTP version) plus all headers combined. A 100 KB query string is well within this limit and will be fully parsed and handed to the middleware.

**No other guards exist** in the Rosetta middleware stack (`MetricsMiddleware` → `TracingMiddleware` → `CorsMiddleware`) that truncate or reject oversized URIs before logging.

### Impact Explanation

An attacker sending a sustained stream of HTTP GET requests with ~900 KB query strings (staying under the 1 MB server limit) causes the Rosetta node to write ~900 KB to its log file per request. At a modest rate of 100 requests/second this produces ~90 MB/s of log output. On a typical deployment with a few GB of log partition, disk exhaustion occurs within seconds to minutes, causing the Rosetta process to crash or become unresponsive when the log sink (file, stdout, or a log aggregator) backs up. This is a denial-of-service against the Rosetta API node with no authentication required and no economic cost to the attacker.

### Likelihood Explanation

The Rosetta API port (default 5700) is reachable by any network peer that can connect to the node. No credentials, tokens, or special headers are required. The attack requires only a standard HTTP client (`curl`, `ab`, `wrk`) and a loop. It is trivially repeatable and scriptable. In deployments behind nginx (as shown in `docker-compose.yml`), nginx's default `large_client_header_buffers 4 8k` would cap individual header buffers at 8 KB, but the Rosetta port is also exposed directly in many configurations (Kubernetes `NodePort`, direct Docker port mapping), bypassing the proxy entirely.

### Recommendation

Truncate `path` to a safe maximum length before including it in the log message:

```go
const maxLoggedPathLen = 512

path := request.URL.RequestURI()
loggedPath := path
if len(loggedPath) > maxLoggedPathLen {
    loggedPath = loggedPath[:maxLoggedPathLen] + "…"
}
// use loggedPath in the message, keep path for internalPaths lookup
```

Additionally, set an explicit `MaxHeaderBytes` on the `http.Server` (e.g. 8192) to reject oversized request lines at the transport layer before they reach any middleware.

### Proof of Concept

```bash
# Build a 100 KB query string and send it repeatedly to the liveness endpoint
PAYLOAD=$(python3 -c "print('a=' + 'A'*102400)")
while true; do
  curl -s "http://<rosetta-host>:5700/health/liveness?${PAYLOAD}" -o /dev/null
done
```

Each request causes `TracingMiddleware` to log a line of ~100 KB at `Info` level (the `internalPaths` check fails because the key `/health/liveness?a=AAA…` does not match `/health/liveness`).
