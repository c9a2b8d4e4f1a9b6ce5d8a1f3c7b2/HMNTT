### Title
Unbounded Concurrent Database Queries in `NftAllowanceServiceImpl.getNftAllowances()` Enable DoS via Resource Exhaustion

### Summary
The `getNftAllowances()` method in `NftAllowanceServiceImpl` performs a direct, uncached, unthrottled database query for every incoming request. The `rest-java` module has no rate limiting, no per-IP throttling, and no request deduplication on the `/api/v1/accounts/{id}/allowances/nfts` endpoint. An unprivileged attacker can flood this endpoint with concurrent identical requests, exhausting the database connection pool and degrading or halting mirror node data ingestion.

### Finding Description
**Code path:**

`AllowancesController.getNftAllowances()` (lines 57–83) builds a request and calls `service.getNftAllowances(request)` with no guard. [1](#0-0) 

`NftAllowanceServiceImpl.getNftAllowances()` (lines 21–31) performs only parameter validation, then immediately delegates to `repository.findAll(request, id)`. [2](#0-1) 

`NftAllowanceRepositoryCustomImpl.findAll()` (lines 37–47) executes a live jOOQ SQL query against the database on every call, with no caching layer. [3](#0-2) 

**Root cause — failed assumption:** The `rest-java` module has no rate limiting. A grep across `rest-java/**` for `RateLimiter`, `Throttle`, `bucket4j` returns zero hits in production code. The only throttling in the codebase lives in the `web3` module (`ThrottleManagerImpl`, `ThrottleConfiguration`) and is scoped exclusively to contract-call endpoints. [4](#0-3) 

The `rest-java` build file includes `spring-boot-starter-cache` but no `@Cacheable` annotation is applied to the NFT allowance service or its repository, so every request is a cold database hit. [5](#0-4) 

**Exploit flow:**
1. Attacker identifies a high-activity account ID (e.g., a popular DeFi spender with many NFT allowances).
2. Attacker sends thousands of concurrent `GET /api/v1/accounts/{id}/allowances/nfts` requests with no authentication required.
3. Each request passes parameter validation and issues an independent `SELECT … FROM nft_allowance WHERE owner = ? AND approved_for_all = true ORDER BY … LIMIT ?` query.
4. The database connection pool is saturated; new requests queue or fail with connection-timeout errors.
5. The shared PostgreSQL instance also serves the importer module; connection starvation causes the importer to fall behind or stop writing new consensus data.

### Impact Explanation
The mirror node's REST API and importer share the same database. Saturating the connection pool degrades or halts data ingestion across all mirror node components. Because mirror nodes are the primary source of historical and real-time Hedera data for wallets, explorers, and dApps, sustained exhaustion can effectively take multiple mirror node deployments offline, meeting the ≥30% processing degradation threshold described in the scope. The `limit` parameter is capped at 100 rows, but this only bounds result size, not query concurrency or frequency. [6](#0-5) 

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. The endpoint is publicly documented in the OpenAPI spec. A single attacker with a modest botnet or even a single high-bandwidth host using async HTTP clients (e.g., `wrk`, `hey`, `ab`) can sustain thousands of requests per second. The attack is trivially repeatable and requires no special knowledge beyond the public API documentation. [7](#0-6) 

### Recommendation
1. **Add rate limiting to `rest-java`**: Introduce a bucket4j or Spring-based rate-limiting filter (mirroring the pattern already used in `web3/ThrottleConfiguration`) applied globally to all REST endpoints, with tighter per-IP limits for read-heavy endpoints like `/allowances/nfts`.
2. **Add response caching**: Annotate `NftAllowanceRepositoryCustomImpl.findAll()` or the service layer with `@Cacheable` using a short TTL (e.g., 1–5 seconds). This collapses concurrent identical requests into a single DB query.
3. **Enforce a global connection pool limit with fast-fail**: Configure HikariCP `maximumPoolSize` and `connectionTimeout` so that pool exhaustion returns HTTP 503 immediately rather than queuing indefinitely, protecting the importer's share of connections.

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg/wrk)

MIRROR_NODE_URL="https://<mirror-node-host>"
ACCOUNT_ID="0.0.98"

wrk -t 16 -c 500 -d 60s \
  "${MIRROR_NODE_URL}/api/v1/accounts/${ACCOUNT_ID}/allowances/nfts"
```
**Expected result:** Within seconds, database connection pool metrics spike to maximum; subsequent requests to other endpoints (including importer health checks) begin timing out or returning 500/503 errors, demonstrating cross-component impact from a single unauthenticated attacker.

    public Collection<NftAllowance> findAll(NftAllowanceRequest request, EntityId accountId) {
        boolean byOwner = request.isOwner();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, byOwner).and(getBoundConditions(bounds));
        return dslContext
                .selectFrom(NFT_ALLOWANCE)
                .where(condition)
                .orderBy(SORT_ORDERS.get(new OrderSpec(byOwner, request.getOrder())))
                .limit(request.getLimit())
                .fetchInto(NftAllowance.class);
    }

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
