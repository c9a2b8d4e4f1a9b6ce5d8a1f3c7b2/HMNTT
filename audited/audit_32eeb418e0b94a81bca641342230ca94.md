### Title
Unauthenticated DB Connection Pool Exhaustion via `GET /api/v1/transactions/:transactionIdOrHash`

### Summary
An unprivileged external attacker can send concurrent requests with syntactically valid but non-existent transaction IDs or hashes to `GET /api/v1/transactions/:transactionIdOrHash`. Because the REST API has no rate limiting, the default connection pool is only 10 connections, and each request holds a connection for up to the 20-second `statement_timeout`, a small number of concurrent requests is sufficient to saturate the pool and force all subsequent legitimate requests to queue or time out waiting for a connection.

### Finding Description

**Code path:**

`rest/server.js` line 133 registers the route with no authentication or rate-limiting middleware: [1](#0-0) 

The handler `getTransactionsByIdOrHash` in `rest/transactions.js` (lines 923–972) executes the following DB queries per request: [2](#0-1) 

1. **Hash path** — `getTransactionHash()` in `rest/transactionHash.js` issues `pool.queryQuietly` unconditionally for any syntactically valid 64-char base64url or 96-char hex string: [3](#0-2) 

2. **Transaction ID path** — `extractSqlFromTransactionsByIdOrHashRequest` builds and executes a parameterized query via `pool.queryQuietly` for any syntactically valid `shard.realm.num-seconds-nanos` string: [4](#0-3) 

3. A **conditional second query** is issued when `mayMissLongTermScheduledTransaction` returns true (extended 62-day timestamp range): [5](#0-4) 

4. A **third query** for inner transactions and a **fourth query** for staking rewards via `createStakingRewardTransferList` can follow: [6](#0-5) 

**Root cause — pool configuration:**

`rest/dbpool.js` initializes the pool with defaults from config: [7](#0-6) 

The documented defaults are:
- `maxConnections = 10`
- `statementTimeout = 20000` ms (20 s)
- `connectionTimeoutMillis = 20000` ms (20 s) [8](#0-7) 

**No rate limiting exists on the REST API.** The throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the web3 Java service, not the Node.js REST service: [9](#0-8) 

The `users` config only controls pagination `limit` per authenticated user, not request rate: [10](#0-9) 

**Why existing checks are insufficient:**

- `isValidTransactionHash` and `TransactionId.fromString()` reject malformed inputs (400), but any syntactically valid ID/hash that does not exist in the DB still reaches `pool.queryQuietly` and holds a connection until the query completes or `statement_timeout` fires.
- `statement_timeout = 20000 ms` limits individual query duration but does not prevent pool saturation: with 10 connections each holdable for 20 s, an attacker maintaining ≥10 concurrent requests continuously saturates the pool.
- `connectionTimeoutMillis = 20000 ms` means queued legitimate requests wait up to 20 s before receiving a connection-timeout error, effectively denying service.

### Impact Explanation

With the default pool of 10 connections and a 20-second statement timeout, all 10 connections can be held simultaneously by attacker-controlled requests. Every legitimate call to `getTransactionsByIdOrHash` (and any other endpoint sharing the same pool) must queue behind them. Queued requests either wait up to 20 seconds (degraded service) or fail with a connection-timeout error. Because `getTransactionsByIdOrHash` can issue up to 4 sequential queries per request, each holding a connection in turn, the effective throughput available to legitimate users approaches zero under sustained attack. This matches the stated critical scope: inability to confirm new transactions.

### Likelihood Explanation

No privileges, API keys, or special network access are required. A single attacker machine can trivially maintain 10–50 concurrent HTTP connections using standard tools (`curl`, `ab`, `wrk`, `hey`). The attack is repeatable indefinitely, requires no prior knowledge of valid transaction IDs (any syntactically valid string suffices), and produces no observable side-effects that would trigger automated blocking. The default pool size of 10 makes the threshold extremely low.

### Recommendation

1. **Add rate limiting** to the Node.js REST API (e.g., `express-rate-limit` or an ingress-level limiter) targeting the `/api/v1/transactions/:transactionIdOrHash` route, keyed by IP or API key.
2. **Increase `maxConnections`** to a value appropriate for expected concurrency, or deploy a connection pooler (PgBouncer is already referenced in the Helm chart) in front of the REST service.
3. **Reduce `statementTimeout`** for this endpoint to the minimum needed (e.g., 5 s) to release connections faster under load.
4. **Add a concurrency cap** at the application layer (e.g., a semaphore limiting simultaneous in-flight DB queries) so pool exhaustion cannot be triggered regardless of connection pool size.
5. Consider requiring authentication or an API key for this endpoint to raise the bar for unauthenticated abuse.

### Proof of Concept

```bash
# Generate 50 concurrent requests with valid-format but non-existent transaction IDs
# Each request will hold a DB connection until the query completes or times out (up to 20s)
# With pool.maxConnections=10, this saturates the pool immediately

for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions/0.0.$i-1234567890-000000000" &
done
wait

# Simultaneously, legitimate requests will queue and time out:
curl -v "http://<mirror-node-host>:5551/api/v1/transactions/0.0.2-1234567890-000000000"
# Expected: hangs for up to 20s then returns connection timeout error
```

The attack requires no authentication, no special knowledge of the system, and is repeatable indefinitely. Increasing concurrency beyond 10 ensures the pool remains saturated even as individual queries complete.

```javascript
const getTransactionHash = async (hash, {order = orderFilterValues.ASC, timestampFilters = []} = {}) => {
  const normalized = normalizeTransactionHash(hash);
  const params = [normalized];

  const timestampConditions = [];
  for (const filter of timestampFilters) {
    timestampConditions.push(`${TransactionHash.CONSENSUS_TIMESTAMP} ${filter.operator} $${params.push(filter.value)}`);
  }

  const query = `${mainQuery}
    ${timestampConditions.length !== 0 ? `where ${timestampConditions.join(' and ')}` : ''}
    ${orderClause} ${order}
    ${limitClause}`;

  const {rows} = await pool.queryQuietly(query, params);
  return normalized !== hash ? rows.filter((row) => row.hash.equals(hash)) : rows;
```
