### Title
Unbounded DB Query Rate via Missing Result Cache and Rate Limiting on GraphQL Account Endpoint

### Summary
The `account()` handler in `AccountController.java` dispatches every incoming request directly to the database through `EntityServiceImpl` with no result-level cache and no per-client rate limiting. The only caching present (`CachedPreparsedDocumentProvider`) covers query parsing only, not query results. An unprivileged external user can sustain arbitrarily high DB query rates by repeatedly issuing a single valid account lookup query.

### Finding Description
**Code path:**

- [1](#0-0) 
  Every invocation of `account()` unconditionally calls `entityService.getByIdAndType()`, `getByAliasAndType()`, or `getByEvmAddressAndType()`.

- [2](#0-1) 
  Each service method calls `entityRepository.findById()`, `findByAlias()`, or `findByEvmAddress()` directly — no `@Cacheable` or any result-level cache annotation is present anywhere in this class.

- [3](#0-2) 
  The cache keyed on `executionInput.getQuery()` stores only the parsed/validated `PreparsedDocumentEntry` (the AST), not the DB result. After the first request, parse overhead is eliminated for the attacker while the DB is still hit on every subsequent request.

- [4](#0-3) 
  `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` bound the cost of a *single* query but impose no constraint on *request rate*. No rate limiter, no per-IP throttle, and no authentication gate is configured anywhere in the graphql module.

**Root cause:** The system assumes that query-parse caching is sufficient to protect backend resources. It is not — parse caching only removes CPU overhead for the attacker, while the DB is still queried on every execution. There is no result cache and no request-rate control.

### Impact Explanation
A single unauthenticated client can issue thousands of identical `account` queries per second. Each request bypasses the parse cache benefit on the server side (the parse is cheap after caching) and lands a fresh `SELECT` on the database. With no result cache and no rate limiting, the DB connection pool and query throughput are the only natural limits. Sustained load can exceed 30% of normal DB capacity, degrading service for all users and potentially causing query timeouts or connection exhaustion.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The attacker needs only one valid account identifier (e.g., any public account ID visible on a block explorer). A simple script or `curl` loop is sufficient. The attack is trivially repeatable and can be automated at high frequency from a single host or distributed across multiple IPs to avoid network-level blocks.

### Recommendation
1. **Add result-level caching** in `EntityServiceImpl` using Spring's `@Cacheable` on `getByIdAndType`, `getByAliasAndType`, and `getByEvmAddressAndType` with a short TTL (e.g., 5–30 seconds) to absorb repeated identical lookups.
2. **Implement per-client rate limiting** at the HTTP or GraphQL instrumentation layer (e.g., a `WebFilter` or a custom `Instrumentation` that tracks request counts per IP/token and rejects excess requests with HTTP 429).
3. Consider adding a `GraphQL` query cost budget per time window per client identity.

### Proof of Concept
```bash
# One valid account ID is sufficient — no credentials needed
ACCOUNT_ID="0.0.1234"
QUERY='{"query":"{ account(input: {entityId: \"'$ACCOUNT_ID'\"}) { balance } }"}'

# Fire 500 requests in parallel — each hits the DB
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node-graphql-host>/graphql \
    -H "Content-Type: application/json" \
    -d "$QUERY" &
done
wait
```

After the first request, the GraphQL parse tree is cached (`CachedPreparsedDocumentProvider`), so subsequent requests skip parsing but still execute `entityRepository.findById()` on every call. Monitoring DB query rate will show a linear increase proportional to request concurrency, with no server-side throttle to stop it.

    public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
        return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
        return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
    }

    @Override
    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
