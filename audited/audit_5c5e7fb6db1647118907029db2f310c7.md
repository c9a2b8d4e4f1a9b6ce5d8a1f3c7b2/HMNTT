### Title
Concurrent Cache-Miss Race in `getTokenAccounts()` Causes DB Connection Pool Exhaustion (Non-Network DoS)

### Summary
`getTokenAccounts()` in `rest/service/tokenService.js` unconditionally issues a second DB query (`token_id = any($1)`) for any token IDs not found in the process-local LRU cache. Because the cache check and population are not atomic, concurrent requests arriving before any single request completes all observe a cache miss simultaneously and each independently saturate a DB connection with the secondary query. With a default DB pool of only 10 connections and no rate limiting on the endpoint, an unprivileged attacker can exhaust the pool and deny service to all other API consumers.

**Root cause — non-atomic cache check/populate:**
The cache is read synchronously, but the DB query and subsequent `tokenCache.set()` calls are asynchronous. Any number of concurrent requests that arrive before the first request's `getCachedTokens()` resolves will all observe an empty cache and each independently issue the `token_id = any($1)` query. There is no mutex, no deduplication of in-flight queries, and no coalescing of concurrent lookups for the same token IDs.

**Why existing checks fail:**

- The `quickLru` token cache (`maxSize: 100000`) only helps *after* the first request completes. It provides zero protection against concurrent requests in flight simultaneously. [6](#0-5) 

- The Redis response cache is **disabled by default** (`cache.response.enabled: false`), so repeated identical URLs are not short-circuited at the HTTP layer. [7](#0-6) 

- The DB connection pool defaults to **10 connections** with a 20-second timeout. [8](#0-7) 

- No per-IP or per-endpoint rate limiting is present in the middleware stack. [9](#0-8) 

- The `limit` parameter is user-controlled and capped at the configured max (default 100), accepted without any concurrency guard. [10](#0-9) 

### Impact Explanation

Each concurrent request in the attack wave consumes **two** DB connections sequentially (primary query, then secondary query). With a pool of 10 connections and N concurrent attack requests, the pool is saturated after 5 simultaneous attack requests (each holding 2 connections). All other API endpoints sharing the same pool — transactions, accounts, balances — begin queuing and eventually time out after 20 seconds (`connectionTimeout: 20000`). This constitutes a full non-network DoS against the mirror node REST API with no authentication required.

### Likelihood Explanation

**Preconditions:**
- One Hedera account with ≥100 token associations (a one-time on-chain cost, achievable for any participant on the network).
- The ability to send concurrent HTTP requests (trivially available to any internet user).

**Repeatability:** After the first attack wave completes and the cache is populated, the attacker can rotate through multiple accounts (each with 100 unique token IDs) to keep the cache cold, or simply wait for the LRU to evict entries under normal load. The attack is repeatable with minimal ongoing cost.

**Feasibility:** No special privileges, no authentication, no insider access required. The endpoint `/api/v1/accounts/{id}/tokens` is fully public.

### Recommendation

1. **Coalesce in-flight cache lookups:** Use a promise-based deduplication map (e.g., a `Map<tokenId, Promise>`) so that concurrent requests for the same uncached token IDs share a single in-flight DB query rather than each issuing their own.

2. **Enable the Redis response cache** for the `/accounts/{id}/tokens` endpoint, which would short-circuit repeated identical requests at the HTTP layer before any DB query is issued. [11](#0-10) 

3. **Add per-IP rate limiting** on the token relationships endpoint to bound the number of concurrent requests from a single source.

4. **Increase the DB connection pool** size or add a per-request concurrency semaphore to prevent pool exhaustion from a single endpoint.

### Proof of Concept

**Setup (one-time):**
```
# Create a Hedera account and associate it with 100 unique tokens
# (standard Hedera SDK operations, costs ~100 * association fee)
ACCOUNT_ID="0.0.12345"
```

**Attack:**
```bash
# Send 50 concurrent requests before any completes (cache cold)
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT_ID}/tokens?limit=100" &
done
wait
```

**Observed result:**
- All 50 requests simultaneously pass the empty cache check in `getCachedTokens()`.
- All 50 issue `SELECT ... FROM token WHERE token_id = any($1)` with 100 IDs each.
- The 10-connection DB pool is immediately saturated.
- All other API requests (transactions, accounts, etc.) begin returning 500/timeout errors within seconds.
- Repeating the loop continuously (rotating `ACCOUNT_ID` across multiple prepared accounts) sustains the outage indefinitely.
