### Title
Cache-Busting DoS via `order` Parameter Alternation — No Rate Limiting on REST API

### Summary
The REST API's response cache key is derived directly from the raw `req.originalUrl` with no query-parameter normalization or per-IP rate limiting. An unprivileged attacker can alternate between `?order=asc` and `?order=desc` on any cacheable endpoint to guarantee 100% cache-miss rate, forcing every request to hit the database and degrading performance for all other users.

### Finding Description

**Cache key generation** — `rest/middleware/responseCacheHandler.js`, lines 151–153:

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

The key is `MD5(req.originalUrl) + "-v1"`. `req.originalUrl` is the verbatim URL string as received by Express, set before any middleware runs. It is never rewritten.

The `requestQueryParser` middleware (`rest/middleware/requestHandler.js`, lines 10–13) does canonicalize the `order` key to lowercase in `req.query`:

```js
const queryCanonicalizationMap = {
  order: lowerCaseQueryValue,
  ...
};
```

However, this only affects `req.query` — it has **no effect on `req.originalUrl`**, which is what the cache key is computed from. The developers themselves acknowledge the gap in a comment at line 149:

> *"In the future, this will utilize Edwin's request normalizer (9113)."*

Because `orderFilterValues.ASC = 'asc'` and `orderFilterValues.DESC = 'desc'` are two distinct string values, the URLs `/api/v1/transactions?order=asc` and `/api/v1/transactions?order=desc` produce two different MD5 hashes and therefore two independent Redis keys. Neither request ever hits the other's cached entry.

The REST server middleware chain (`rest/server.js`, lines 67–141) contains no per-IP or per-client rate limiting for the REST API — the rate-limiting code found in the repository (`web3/src/main/java/.../ThrottleManagerImpl.java`) belongs to the separate Java `web3` service, not the Node.js REST service.

**Exploit flow:**

1. Attacker sends `GET /api/v1/transactions?order=asc` → cache miss → DB query executed → result stored under key A.
2. Attacker sends `GET /api/v1/transactions?order=desc` → cache miss → DB query executed → result stored under key B.
3. Attacker repeats steps 1–2 in a tight loop. Every request is a cache miss; the DB is hit on every single request.
4. Legitimate users querying the same endpoint also get cache misses because the attacker's alternation keeps the effective cache utilization at zero for the targeted query shape.

### Impact Explanation

Every cache miss causes a full database query. Under sustained alternation from even a single attacker, the PostgreSQL backend receives a continuous stream of queries that the cache was designed to absorb. This degrades response latency for all users sharing the same mirror-node instance and can exhaust DB connection pool slots or push query throughput beyond sustainable limits. The impact is availability degradation (griefing), not data exfiltration, consistent with the "Medium" severity label.

### Likelihood Explanation

The attack requires zero authentication, zero special knowledge, and only two HTTP GET requests repeated in a loop — trivially scriptable with `curl` or any HTTP client. The two valid `order` values (`asc`/`desc`) are publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml`, lines 5116–5130). No exploit tooling is needed. The attack is repeatable indefinitely and is not self-limiting.

### Recommendation

1. **Normalize `req.originalUrl` before hashing** (or build the cache key from a sorted, canonicalized representation of the parsed query parameters rather than the raw URL string). This is already planned per the inline comment referencing issue 9113.
2. **Add per-IP rate limiting** to the Node.js REST middleware chain (e.g., `express-rate-limit`) so that a single client cannot sustain a high enough request rate to keep the cache cold.
3. As a short-term mitigation, consider caching both `order=asc` and `order=desc` results under a single key and returning the appropriate ordering from the cached payload, or pre-populating both orderings on first miss.

### Proof of Concept

```bash
# Requires: mirror-node REST API running with Redis cache enabled

while true; do
  curl -s "https://<mirror-node>/api/v1/transactions?order=asc" > /dev/null
  curl -s "https://<mirror-node>/api/v1/transactions?order=desc" > /dev/null
done
```

Each iteration produces two cache misses and two full DB queries. Monitor PostgreSQL `pg_stat_activity` or the mirror-node metrics endpoint to observe the sustained query rate with zero cache hits.
