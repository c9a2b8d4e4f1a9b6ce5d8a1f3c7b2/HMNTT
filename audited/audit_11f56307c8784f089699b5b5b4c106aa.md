### Title
Unauthenticated Unbounded Subscription Exhausts Scheduler Threads and Database Connections via `PollingTopicListener.listen()`

### Summary
`PollingTopicListener.listen()` creates a per-subscriber, server-side infinite polling loop (`RepeatSpec.times(Long.MAX_VALUE)`) that issues blocking JDBC queries on a `boundedElastic` scheduler. Because the gRPC `subscribeTopic` endpoint has no authentication, no global subscriber cap, and the per-connection limit of 5 is trivially bypassed by opening multiple connections, an unauthenticated attacker can flood the server with long-lived subscriptions that permanently consume scheduler threads and database connections, denying service to legitimate users.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (no auth check) → `TopicMessageServiceImpl.subscribeTopic()` (no subscriber cap) → `TopicListener.listen()` → `PollingTopicListener.listen()` (when `listenerProperties.type = POLL`).

**Root cause — `PollingTopicListener.listen()` lines 34–49:**

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // ← infinite, server-hardcoded
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))             // ← boundedElastic
```

Each call to `listen()` creates an independent cold Flux. `RepeatSpec.times(Long.MAX_VALUE)` is hardcoded server-side; the subscriber triggers it simply by connecting. Every repeat cycle calls `poll()` → `topicMessageRepository.findByFilter()` → `typedQuery.getResultList()` (line 60 of `TopicMessageRepositoryCustomImpl`), a **blocking JDBC call** that holds a `boundedElastic` thread and a JDBC connection for its full duration.

**Failed assumptions / missing controls:**

1. **No authentication**: `ConsensusController` has no `@PreAuthorize`, no `ServerInterceptor` enforcing credentials. The only production interceptor found (`GrpcConfiguration`) only sets `maxConcurrentCallsPerConnection`.
2. **Per-connection limit is trivially bypassed**: `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, but an attacker opens N connections → N×5 concurrent subscriptions.
3. **No global subscriber cap**: `subscriberCount` in `TopicMessageServiceImpl` is a Micrometer gauge only — it is never checked against a maximum.
4. **No subscription timeout**: `TopicMessageFilter` requires neither `limit` nor `endTime`. With `limit=0` (`hasLimit()` returns false) and no `endTime`, `pastEndTime()` returns `Flux.never()` and `flux.take()` is never applied — the subscription runs indefinitely.
5. **`boundedElastic` is finite**: Reactor's default cap is `10 × CPU_cores` threads. Blocking JDBC polls consume these threads for the query duration; with enough concurrent subscriptions the pool saturates and new polls queue, cascading into DB connection pool exhaustion.

### Impact Explanation

When `hiero.mirror.grpc.listener.type = POLL` is configured, an attacker with network access to port 5600 can exhaust both the `boundedElastic` thread pool and the JDBC connection pool. Once saturated, all new `subscribeTopic` calls (legitimate or otherwise) block indefinitely waiting for a thread/connection, effectively causing a complete denial of service for the gRPC API. The DB connection pool exhaustion also impacts any other component sharing the same pool. Severity: **High** (full DoS of the gRPC service, no authentication required).

### Likelihood Explanation

The attack requires only a gRPC client (e.g., `grpcurl`, the Hedera SDK, or any HTTP/2 client) and network reachability to port 5600. No credentials, no special knowledge, and no privileged account are needed. The attacker opens many TCP connections in parallel (each contributing 5 subscriptions), which is trivial to script. The attack is repeatable and persistent — subscriptions survive until the server is restarted or the attacker's TCP connections are forcibly closed. The only prerequisite is that `POLL` mode is configured, which is a documented and supported deployment option.

### Recommendation

1. **Enforce a global subscriber limit**: In `TopicMessageServiceImpl.subscribeTopic()`, reject new subscriptions when `subscriberCount` exceeds a configurable maximum (e.g., `grpcProperties.getMaxSubscribers()`).
2. **Add per-IP or per-connection rate limiting**: Use a gRPC `ServerInterceptor` to count and cap active streams per remote address.
3. **Require a maximum subscription duration**: Enforce a server-side `endTime` ceiling (e.g., max 1 hour) so subscriptions cannot run indefinitely.
4. **Use non-blocking DB access**: Replace the blocking `getResultList()` in `TopicMessageRepositoryCustomImpl.findByFilter()` with a reactive/R2DBC query to avoid holding `boundedElastic` threads during DB I/O.
5. **Consider authentication**: Add token/API-key validation via a `ServerInterceptor` before allowing any subscription.

### Proof of Concept

```bash
# Open 20 parallel connections, each with 5 subscriptions (100 total infinite polling loops)
for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Each subscription triggers PollingTopicListener.listen() with RepeatSpec.times(Long.MAX_VALUE),
# issuing a blocking JDBC query every 500ms per subscription.
# After ~(boundedElastic_cap / avg_poll_duration_ms * 500) subscriptions,
# the scheduler thread pool saturates.
# Legitimate subscribeTopic calls then hang indefinitely.
```
