### Title
Unbounded Poll Flood via HCS Message Submission Exhausts Per-Subscriber `onBackpressureBuffer`, Terminating Slow Subscriber Streams

### Summary
Any Hedera account holder (no special privilege required) can submit a high volume of messages to a public HCS topic. The `SharedPollingTopicListener.poll()` method returns up to `maxPageSize` (default 5,000) messages per interval from the database and emits them through a shared hot publisher. Each subscriber's `onBackpressureBuffer` in `SharedTopicListener.listen()` is count-bounded (default 16,384 messages) with `BufferOverflowStrategy.ERROR`, which terminates the subscriber's stream when the buffer overflows. A slow subscriber (e.g., on a congested network) cannot drain the buffer fast enough, causing forced disconnection.

### Finding Description

**Code path:**

`SharedTopicListener.listen()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedTopicListener.java`, line 24:
```java
.onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
```
`maxBufferSize` defaults to 16,384 (bounded 8,192–65,536). `BufferOverflowStrategy.ERROR` propagates an `OverflowException` downstream, terminating the subscriber's gRPC stream.

`SharedPollingTopicListener.poll()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedPollingTopicListener.java`, lines 65–66:
```java
return Flux.fromIterable(topicMessageRepository.findLatest(
    context.getLastConsensusTimestamp().get(), listenerProperties.getMaxPageSize()))
```
`maxPageSize` defaults to 5,000. The result is emitted through a `.share()` hot publisher (line 52), meaning every active subscriber receives every polled message regardless of their individual consumption rate.

**Root cause:** The buffer overflow threshold is purely count-based. There is no per-topic rate limit, no per-subscriber admission control, and no message-size-aware throttling. The shared publisher emits to all subscribers simultaneously; a subscriber that cannot drain 5,000 messages within the poll interval (default 500 ms) accumulates backlog. After ≈3–4 poll intervals with no consumption, the 16,384-message buffer overflows and `BufferOverflowStrategy.ERROR` fires.

**Failed assumption:** The design assumes subscribers consume messages at least as fast as the poll rate. This assumption breaks for any slow consumer (slow network, slow client, GC pause) when the topic is flooded.

**Why existing checks are insufficient:**
- `maxBufferSize` (16,384) and `maxPageSize` (5,000) bound the attack parameters but do not prevent the attack — they only determine how many messages the attacker must submit.
- There is no per-topic or per-account rate limiting on message ingestion in the mirror node's gRPC layer.
- The `retryWhen` and `repeatWhen` on the shared publisher protect the publisher, not individual subscribers.
- Message content (including the `message` byte array up to 6 KiB) is stored and replayed without size-based flow control.

### Impact Explanation

Every subscriber watching a flooded topic that cannot drain messages at the poll rate is forcibly disconnected with an error. With `maxPageSize=5000` and `interval=500ms`, the effective emission rate is up to 10,000 messages/second into each subscriber's buffer. A subscriber that processes even 8,000 messages/second will accumulate 2,000 messages/interval of backlog, exhausting the 16,384-message buffer in approximately 8 seconds. Additionally, 16,384 buffered messages × 6 KiB each ≈ 96 MB of heap per subscriber; with many concurrent subscribers this creates significant GC pressure that further slows consumption, accelerating the overflow. The result is a sustained denial-of-service against all slow or moderately-paced subscribers on any targeted topic.

### Likelihood Explanation

Hedera HCS is a permissionless protocol: any funded account can submit messages to any public topic. Transaction fees on Hedera are low (fractions of a cent per message). Submitting 16,384 messages of 6 KiB each costs on the order of a few dollars at current fee schedules. The attack is repeatable indefinitely and requires no privileged access, no authentication bypass, and no exploit of a cryptographic weakness — only a funded account and a script that submits transactions in a loop.

### Recommendation

1. **Switch `BufferOverflowStrategy` to `DROP_LATEST` or `DROP_OLDEST`** for the per-subscriber buffer so slow subscribers are degraded (messages dropped) rather than terminated.
2. **Add per-topic ingestion rate limiting** in the gRPC subscription layer so a single topic cannot monopolize the shared poll budget.
3. **Make the buffer size-aware**: track buffered bytes, not just message count, and apply backpressure proportional to message size.
4. **Implement subscriber-level admission control**: if a subscriber's buffer exceeds a configurable high-water mark for more than a configurable duration, drop messages rather than erroring.
5. **Decouple slow subscribers** from the shared hot publisher using a per-subscriber bounded queue with drop semantics instead of `BufferOverflowStrategy.ERROR`.

### Proof of Concept

**Preconditions:**
- A funded Hedera account (no special role).
- A public HCS topic ID.
- Mirror node running with default config (`maxPageSize=5000`, `maxBufferSize=16384`, `interval=500ms`, `type=SHARED_POLL`).
- At least one gRPC subscriber connected on a slow link or under load.

**Steps:**

1. Using the Hedera SDK or `hedera-cli`, submit ≥16,384 `ConsensusSubmitMessage` transactions to the target topic, each with a `message` field of 6,144 bytes (maximum allowed), in rapid succession.
2. The mirror node importer ingests these messages into the database.
3. On the next poll cycle, `findLatest` returns 5,000 messages; the shared publisher emits them to all subscribers.
4. Repeat for 3–4 poll intervals (≈1.5–2 seconds total).
5. Any subscriber whose gRPC client cannot consume 5,000 messages per 500 ms will see its `onBackpressureBuffer` fill to 16,384 entries.
6. On the next emission past the buffer limit, `BufferOverflowStrategy.ERROR` fires, and the subscriber's gRPC stream is terminated with an `OverflowException`.
7. The subscriber must reconnect; the attacker can continue submitting messages to keep the topic flooded, causing repeated disconnections. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
