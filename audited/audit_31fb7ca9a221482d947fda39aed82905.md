### Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses Limit Guard, Causing Unbounded Repeated Database Queries

### Summary
An unprivileged gRPC subscriber can set `limit=Long.MAX_VALUE` in a `ConsensusTopicQuery`. Inside `PollingTopicListener.poll()`, the narrowing cast `(int)(Long.MAX_VALUE - 0L)` silently overflows to `-1`. Because `TopicMessageFilter.hasLimit()` returns `false` for negative values, the downstream `findByFilter()` call fires an unbounded database query with no `LIMIT` clause. This repeats on every poll interval for the lifetime of the subscription, constituting a sustained resource-exhaustion attack.

### Finding Description

**Entry point — no upper-bound check on `limit`:**

`ConsensusController.toFilter()` maps the protobuf `uint64 limit` field directly to `TopicMessageFilter.limit` with no maximum cap. [1](#0-0) 

The only validation on `TopicMessageFilter.limit` is `@Min(0)`, which `Long.MAX_VALUE` trivially satisfies. [2](#0-1) 

**Propagation to `PollingTopicListener`:**

`TopicMessageServiceImpl.incomingMessages()` computes `limit = Long.MAX_VALUE - 0 = Long.MAX_VALUE` (no overflow at `long` width) and passes it to `topicListener.listen(newFilter)`. [3](#0-2) 

**The overflow in `poll()`:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())  // (int)(Long.MAX_VALUE - 0) = -1
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // Math.min(-1, positive) = -1
``` [4](#0-3) 

`(int) Long.MAX_VALUE` = `(int) 0x7FFFFFFFFFFFFFFF` = `0xFFFFFFFF` = `-1`. `Math.min(-1, any_positive)` = `-1`. The new filter is built with `limit = -1`.

**Why `findByFilter` silently drops the limit:**

`hasLimit()` is defined as `limit > 0`. With `limit = -1`, it returns `false`, so `setMaxResults` is never called and the JPA query has no `LIMIT` clause — returning every matching row. [5](#0-4) [6](#0-5) 

**Why the polling loop never stops:**

`PollingTopicListener.listen()` repeats with `RepeatSpec.times(Long.MAX_VALUE)`. The outer `flux.take(Long.MAX_VALUE)` in `subscribeTopic` is effectively unlimited. No termination condition is ever satisfied, so the unbounded query fires on every poll interval indefinitely. [7](#0-6) [8](#0-7) 

**Note on the claimed impact:** The specific claim that `findByFilter` throws an exception is incorrect. The negative limit causes `hasLimit()` to return `false`, so `setMaxResults` is never called with `-1` and no exception is raised. The actual impact is silent limit bypass leading to unbounded queries, not a crash.

### Impact Explanation
Every poll cycle (default interval configurable via `listenerProperties.getInterval()`) issues a full table scan against `topic_message` for the subscribed topic with no row limit. On a busy topic with millions of messages, each query can consume significant DB CPU, memory, and I/O. Multiple concurrent attackers amplify this linearly. The polling loop runs for the entire subscription lifetime (effectively forever with `RepeatSpec.times(Long.MAX_VALUE)`), making this a sustained DoS against the database tier with no economic cost to the attacker.

### Likelihood Explanation
The attack requires only a valid gRPC client and knowledge of any existing topic ID. No authentication, no tokens, no privileged access. The protobuf `uint64` field accepts `Long.MAX_VALUE` natively. The exploit is trivially reproducible with a single `grpcurl` invocation and is repeatable by any number of concurrent clients.

### Recommendation
1. **Cap the limit at ingestion:** In `ConsensusController.toFilter()`, clamp `query.getLimit()` to a server-defined maximum (e.g., `Math.min(query.getLimit(), grpcProperties.getMaxLimit())`).
2. **Fix the cast:** In `PollingTopicListener.poll()` (and the identical pattern in `PollingTopicMessageRetriever.poll()`), guard against overflow before casting:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
3. **Validate `pageSize > 0`:** Assert `pageSize > 0` before building `newFilter`; if it is ≤ 0, return `Flux.empty()` immediately.

### Proof of Concept
```bash
# Subscribe with limit = Long.MAX_VALUE (9223372036854775807)
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": <valid_topic>},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected (buggy) behavior:**
- `(int)(9223372036854775807L - 0L)` = `-1`
- `Math.min(-1, maxPageSize)` = `-1`
- `newFilter.limit = -1` → `hasLimit()` = false → no SQL `LIMIT`
- Full table scan on `topic_message` fires every poll interval indefinitely
- DB CPU/memory spikes; observable via `EXPLAIN ANALYZE` showing sequential scan with no limit clause
