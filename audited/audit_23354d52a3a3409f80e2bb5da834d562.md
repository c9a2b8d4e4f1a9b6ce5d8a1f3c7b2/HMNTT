### Title
Unbounded Keep-Alive Connection Exhaustion via Missing Server-Level Connection Limits in `rest/server.js`

### Summary
The Node.js HTTP server in `rest/server.js` is started with `app.listen(port, '0.0.0.0')` but the returned server object is never configured with `maxConnections`, a hardened `keepAliveTimeout`, or any per-IP connection throttle. Because port 5551 is directly exposed to the host in `docker-compose.yml`, an unprivileged attacker can open a large number of HTTP/1.1 keep-alive connections and sustain them indefinitely by sending a new request on each connection before the 5-second default idle timeout expires, eventually exhausting OS-level file-descriptor or socket limits and preventing new connections from being accepted.

### Finding Description
**Exact code path:**

`rest/server.js` lines 149–166:
```js
const server = app.listen(port, '0.0.0.0', (err) => { ... });
createTerminus(server, { ... });
```
After `app.listen()` returns the `http.Server` instance, no hardening properties are set:
- `server.keepAliveTimeout` — never set; defaults to Node.js built-in **5000 ms**.
- `server.headersTimeout` — never set; defaults to **60 000 ms**.
- `server.maxConnections` — never set; defaults to **unlimited**.

**Root cause / failed assumption:** The code assumes that the 5-second default `keepAliveTimeout` will naturally reclaim idle connections. This assumption fails because an attacker can trivially reset the idle timer by sending a new valid HTTP/1.1 GET request on the same socket every ~4 seconds, keeping the connection alive indefinitely at negligible cost.

**No rate-limiting or connection-limiting middleware exists.** `rest/package.json` contains no `express-rate-limit`, `helmet`, or any connection-throttling dependency. The middleware stack registered in `server.js` (cors → compression → httpContext → requestLogger → authHandler → metricsHandler → responseCacheCheckHandler → route handlers → responseHandler → handleError) has no IP-based connection counter.

**Direct exposure:** `docker-compose.yml` line 117 maps `5551:5551`, making the Node.js process reachable directly from the host network, bypassing the nginx proxy on port 8080 entirely. The nginx config (lines 190–193) only applies `keepalive 16` to *upstream* connections from nginx to the backend — it provides zero protection against clients connecting directly to port 5551.

**Exploit flow:**
1. Attacker opens *N* TCP connections to `<host>:5551` with `Connection: keep-alive`.
2. On each connection, attacker sends a minimal valid request (e.g., `GET /api/v1/transactions HTTP/1.1\r\nHost: x\r\n\r\n`) every ~4 seconds.
3. Each response resets the 5-second `keepAliveTimeout` timer; the connection is never closed by the server.
4. Node.js holds one file descriptor per connection. Once the process or OS file-descriptor limit is reached (default `ulimit -n` is often 1024 in containers, or up to 65535 on tuned hosts), `accept()` calls fail and new legitimate connections are refused with `ECONNREFUSED` or silently dropped.
5. The mirror node REST API becomes unavailable for transaction confirmation queries.

### Impact Explanation
The REST API at `/api/v1/transactions` is the primary interface for confirming new Hedera transactions. If the server's connection table is saturated, no new TCP handshakes can complete, causing a total denial of service for transaction confirmation. Because the Node.js process is single-threaded and `maxConnections` is unlimited, the attacker does not need to trigger any expensive application logic — simply holding sockets open is sufficient. Severity: **High** (availability impact, no authentication required, directly reachable on a publicly mapped port).

### Likelihood Explanation
The attack requires only a standard HTTP client capable of maintaining many parallel persistent connections (e.g., `wrk`, `ab`, or a simple Python `asyncio` script). No credentials, special headers, or knowledge of the application are needed. The attacker must sustain ~4-second heartbeat requests per connection, which is trivially automatable. The direct exposure of port 5551 in the default `docker-compose.yml` means any operator who runs the reference deployment without additional firewall rules is immediately vulnerable. Repeatability is high: the attack can be re-launched instantly after any server restart.

### Recommendation
Immediately after `app.listen()` returns the server object, apply the following hardening in `rest/server.js`:

```js
const server = app.listen(port, '0.0.0.0', (err) => { ... });

// Limit idle keep-alive connections (e.g., 10 s)
server.keepAliveTimeout = 10_000;
// Ensure headers timeout exceeds keepAliveTimeout
server.headersTimeout = 15_000;
// Hard cap on simultaneous connections
server.maxConnections = 1000; // tune to expected load
```

Additionally:
- Add `express-rate-limit` middleware to throttle requests per IP.
- Use nginx `limit_conn` and `limit_req` directives on the proxy layer.
- Remove or firewall the direct `5551:5551` port mapping in production deployments so all traffic flows through the nginx proxy, which can enforce connection limits independently.

### Proof of Concept
```bash
# Open 2000 persistent connections, each sending a heartbeat every 4 s
python3 - <<'EOF'
import asyncio, aiohttp

TARGET = "http://<host>:5551/api/v1/transactions"

async def hold(session):
    while True:
        try:
            async with session.get(TARGET) as r:
                await r.read()
        except Exception:
            pass
        await asyncio.sleep(4)

async def main():
    connector = aiohttp.TCPConnector(limit=0)
    async with aiohttp.ClientSession(connector=connector) as session:
        await asyncio.gather(*[hold(session) for _ in range(2000)])

asyncio.run(main())
EOF
