### Title
Integer Overflow in `poll()` Bypasses `maxPageSize` Cap, Enabling Unlimited Database Queries

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` performs an unchecked narrowing cast from `long` to `int`. Any unprivileged caller who supplies a `limit` value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`) causes this cast to produce `-1`, which then bypasses the configured `maxPageSize` guard entirely and issues an unbounded SQL query against the `topic_message` table.

### Finding Description

**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `poll()`, lines 68–71.

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())   // ← narrowing cast
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());        // ← guard bypassed
```

**Root cause:** `filter.getLimit()` is a `long`; `context.getTotal().get()` is also a `long`. Their difference is computed as a `long`, then silently truncated to `int`. When `filter.getLimit() = Long.MAX_VALUE` and `total = 0`:

```
(int)(Long.MAX_VALUE - 0) == (int)(0x7FFFFFFFFFFFFFFF) == -1
```

`Math.min(-1, maxPageSize)` evaluates to `-1`, so `pageSize = -1`.

**Exploit flow:**

1. `poll()` builds an internal filter: `filter.toBuilder().limit(-1).startTime(startTime).build()` (line 75). This is a direct Java builder call — Spring AOP validation is **not** invoked here.
2. `TopicMessageFilter.hasLimit()` returns `limit > 0` (line 39–41). With `limit = -1`, `hasLimit()` returns `false`.
3. `topicMessageRepository.findByFilter(newFilter)` is called with a filter that reports no limit, so no `LIMIT` clause is applied to the SQL query.
4. The `maxPageSize` cap (`RetrieverProperties.maxPageSize`, default 1000 throttled / 5000 unthrottled) is completely bypassed.

**Why existing checks fail:**

- `TopicMessageFilter.limit` carries `@Min(0)`, which only rejects negative values. `Long.MAX_VALUE` is `> 0` and passes validation at the `subscribeTopic` entry point.
- The `@Validated` / `ConstraintViolationException` path (confirmed by the `invalidFilter` test) only fires for the externally supplied filter. The internally rebuilt filter (`toBuilder().limit(-1)`) is never passed through Spring's validation interceptor.
- `isComplete()` (lines 121–128) checks `filter.getLimit() == total.get()`, i.e., `Long.MAX_VALUE == total`, which is never true, so the retriever keeps polling indefinitely (throttled: until `pageSize < maxPageSize`; unthrottled: until `numRepeats` exhausted) — each poll issuing an unbounded query.

### Impact Explanation

Each invocation of `poll()` issues a SQL query with no `LIMIT` clause against the `topic_message` table. A single subscription with `limit = Long.MAX_VALUE` can force a full table scan on every polling interval (default every 2 s for throttled, 20 ms for unthrottled). Multiple concurrent subscriptions amplify the database load linearly. This constitutes a denial-of-service against the database tier. The only backstop is the 60-second stream timeout (`retrieverProperties.getTimeout()`), but within that window the damage is already done and the attack is trivially repeatable.

### Likelihood Explanation

The gRPC `ConsensusService` `subscribeTopic` RPC maps the protobuf `limit` field (a `uint64`) directly into `TopicMessageFilter.limit` (a Java `long`). No authentication or authorization is required to subscribe to a topic — any network-reachable client can send a `ConsensusTopicQuery` with `limit = 0xFFFFFFFFFFFFFFFF`. The attack requires zero privileges, zero knowledge of the topic's owner, and is trivially scriptable. It is repeatable on every new subscription.

### Recommendation

Replace the unchecked narrowing cast with a safe clamp:

```java
long remaining = filter.getLimit() - context.getTotal().get();
// Clamp to [0, Integer.MAX_VALUE] before narrowing
int limit = filter.hasLimit()
        ? (int) Math.min(remaining, Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```

Additionally, add an upper-bound validation on `TopicMessageFilter.limit` at the API entry point (e.g., `@Max(Long.MAX_VALUE)` is vacuous — use a domain-meaningful cap such as `@Max(10_000_000)`) so that absurdly large limits are rejected before they reach the retriever.

### Proof of Concept

```
# Using grpcurl against the mirror node gRPC endpoint:
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Preconditions:** Network access to the gRPC port; topic 0.0.1 exists (or `checkTopicExists=false`).

**Trigger:** The `limit` field is `Long.MAX_VALUE`. On the first `poll()` call, `(int)(Long.MAX_VALUE - 0) = -1`; `pageSize = -1`; `findByFilter` receives a filter with `hasLimit() == false`; a full table scan is issued.
