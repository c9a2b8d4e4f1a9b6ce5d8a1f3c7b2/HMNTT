### Title
Query String Bypass of `internalPaths` Debug-Level Suppression in `TracingMiddleware`

### Summary
`TracingMiddleware` uses `request.URL.RequestURI()` to obtain the request path, which returns the full URI including any query string. The `internalPaths` map contains only bare paths (e.g., `/health/readiness`), so a request to `/health/readiness?anything=1` produces a path string that does not match any map key, causing the request to be logged at `log.Info` level instead of `log.Debug`. Any unprivileged external user can trigger this by appending an arbitrary query string to any internal health/metrics path.

### Finding Description
**File:** `rosetta/app/middleware/trace.go`
**Function:** `TracingMiddleware()`, lines 47 and 55

```
path := request.URL.RequestURI()   // line 47 — includes query string
...
if internalPaths[path] {           // line 55 — exact-match map lookup
    log.Debug(message)
} else {
    log.Info(message)              // fires for /health/readiness?foo=bar
}
```

`internalPaths` is defined at line 19 as:
```go
var internalPaths = map[string]bool{livenessPath: true, metricsPath: true, readinessPath: true}
```
where `readinessPath = "/health/readiness"` (no query string).

`net/http`'s `URL.RequestURI()` is documented to return `path?query` when a query is present. Therefore `internalPaths["/health/readiness?block=999"]` evaluates to `false` (zero value for `bool`), and the `else` branch executes, emitting a `log.Info` entry.

The test suite in `trace_test.go` only tests exact bare paths (lines 37–46) and never tests a path-with-query-string variant against an internal path, so this bypass is untested and undetected.

### Impact Explanation
The info-level log is the operational transaction/request audit trail for the Rosetta API node. An attacker can flood it with synthetic health-probe entries (e.g., `GET /health/readiness?txid=<spoofed>`) that are indistinguishable in format from real API calls, because the logged message is `<ip> GET /health/readiness?txid=... (200) in Xms`. This pollutes the transaction history log with non-transaction noise, degrades log-based auditing and anomaly detection, and can inflate log storage. It does not allow fund theft or data exfiltration, placing it in the "integrity/availability of audit trail" impact class.

### Likelihood Explanation
No authentication or rate-limiting is required. Any network-reachable client can send `GET /health/readiness?x=1` in a tight loop. The Rosetta health endpoints are typically exposed to load balancers and monitoring systems, making them reachable from outside the cluster in many deployments. The exploit is one HTTP request and is trivially repeatable.

### Recommendation
Replace the `RequestURI()` call with `request.URL.Path` (which strips the query string) for the `internalPaths` lookup:

```go
path := request.URL.RequestURI()   // keep for the log message (full URI is useful)
logPath := request.URL.Path        // use bare path for the map lookup

if internalPaths[logPath] {
    log.Debug(message)
} else {
    log.Info(message)
}
```

This preserves the full URI in the log message while ensuring the suppression check is query-string-agnostic.

### Proof of Concept
```bash
# Normal readiness probe — logged at DEBUG, invisible at INFO level
curl http://<rosetta-host>/health/readiness

# Bypass — logged at INFO level, appears in transaction audit log
curl "http://<rosetta-host>/health/readiness?block=999"

# Flood the info log with fake health-probe entries
for i in $(seq 1 10000); do
  curl -s "http://<rosetta-host>/health/readiness?txid=$i" > /dev/null &
done
```
Each request produces an info-level log line such as:
```
level=info msg="1.2.3.4 GET /health/readiness?txid=42 (200) in 1.2ms"
```
