### Title
Missing `errorMessage` Sanitization Allows Null-Byte DoS in COPY-Based Batch Insert

### Summary
The `ContractResult.errorMessage` field is populated directly from the protobuf `ContractFunctionResult.getErrorMessage()` without calling `DomainUtils.sanitize()`, which is the project's own utility for stripping null bytes (`\u0000`) that PostgreSQL rejects in text columns. The `BatchInserter` writes all queued `ContractResult` rows to PostgreSQL via `COPY FROM STDIN WITH CSV`, so a single row containing a null byte in `errorMessage` causes the entire batch to fail with a `ParserException`, halting all contract-result writes for that flush cycle.

### Finding Description

**Exact code path:**

`ContractResultServiceImpl.processContractResult()` at line 251: [1](#0-0) 

```java
contractResult.setErrorMessage(functionResult.getErrorMessage());
```

No call to `DomainUtils.sanitize()` is made here. Compare with `AbstractEntity` and `AbstractToken`, where every user-influenced string field (e.g., `memo`) is passed through `DomainUtils.sanitize()` before storage. [2](#0-1) 

`DomainUtils.sanitize()` is explicitly documented and implemented to strip null bytes for PostgreSQL compatibility: [3](#0-2) 

The `BatchInserter.persistItems()` serializes the entire collection via Jackson `CsvMapper` and streams it into a `PGCopyOutputStream`: [4](#0-3) 

Any encoding error during the COPY stream causes the `copyIn` to be cancelled and the exception is re-thrown as a `ParserException`: [5](#0-4) 

**Root cause:** The `ContractResult.errorMessage` field [6](#0-5) 
has no custom setter, no `@Column` constraint, and no sanitization call in the service layer, unlike every other user-influenced string field in the domain model.

**Why existing checks fail:** `DomainUtils.sanitize()` only replaces `\u0000` and is never invoked for `errorMessage`. The `BatchInserter` has no per-field filtering; it serializes the entire POJO graph. PostgreSQL's COPY protocol rejects null bytes in `text`/`varchar` columns at the wire level, causing the entire in-flight batch to be aborted.

### Impact Explanation
A single poisoned `ContractResult` row aborts the entire COPY batch for that flush cycle. Because the `ParserException` propagates upward, the importer's record-stream parser stalls or retries indefinitely on the same poisoned record file, preventing **all** subsequent contract-result writes. This is a non-network DoS: contract execution history stops being recorded, breaking downstream REST API queries for contract results, Ethereum-compatible JSON-RPC calls, and any application relying on mirror-node contract data.

### Likelihood Explanation
Any account on the Hedera network can deploy a Solidity contract that executes `revert("\x00poisoned")`. The consensus node records the raw revert reason string in `ContractFunctionResult.error_message`. Because protobuf `string` fields in Java are decoded from the wire bytes and the Hedera consensus node does not strip null bytes from EVM revert data before encoding the record, a null byte in the revert payload reaches the mirror node verbatim. No special privilege is required; a standard `ContractCall` transaction with a fee payment is sufficient. The attack is repeatable: the attacker can re-submit the transaction every time the importer recovers.

### Recommendation
Apply `DomainUtils.sanitize()` to `errorMessage` at the point of assignment in `ContractResultServiceImpl.processContractResult()`:

```java
// line 251 – ContractResultServiceImpl.java
contractResult.setErrorMessage(DomainUtils.sanitize(functionResult.getErrorMessage()));
```

Additionally, extend `DomainUtils.sanitize()` to strip lone Unicode surrogates (`\uD800`–`\uDFFF`), which PostgreSQL UTF-8 also rejects but the current implementation does not handle. [7](#0-6) 

### Proof of Concept
1. Deploy a Solidity contract on Hedera testnet:
   ```solidity
   contract Poison {
       function trigger() external pure {
           revert("\x00null-byte-poison");
       }
   }
   ```
2. Call `trigger()` from any funded account. The consensus node records the revert reason (including `\x00`) in `ContractFunctionResult.error_message`.
3. The mirror node importer reads the record stream, calls `ContractResultServiceImpl.processContractResult()`, sets `errorMessage = "\x00null-byte-poison"` without sanitization.
4. At the next flush, `BatchInserter.persistItems()` streams the CSV row containing the null byte into `PGCopyOutputStream`. PostgreSQL rejects it; `copyIn.cancelCopy()` is called; a `ParserException` is thrown.
5. All `ContractResult` rows in the same batch are discarded. The importer stalls on the same record file. Contract result writes halt until the record is manually skipped or the code is patched.
