### Title
Unauthenticated Cache Exhaustion via Unique contractId Flooding in findStorage()

### Summary
`ContractStateServiceImpl.findStorage()` writes attacker-controlled `contractId` values into two shared, globally-bounded Caffeine caches (`contractSlotsCache` maximumSize=3000, `contractStateCache` maximumSize=25000) with no per-IP or per-user rate limiting. The only protection is a global 500 req/s bucket shared across all callers. An unauthenticated attacker can flood the endpoint with unique contract addresses, filling both caches with useless entries, evicting legitimate cached data, and forcing all other users' requests to fall back to expensive DB queries.

### Finding Description

**Code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` (global bucket only) → `ContractExecutionService` → EVM SLOAD opcode → `ContractStorageReadableKVState.readFromDataSource()` → `ContractStateServiceImpl.findStorage()` → `findStorageBatch()`

In `findStorageBatch()` (lines 85–122 of `ContractStateServiceImpl.java`):

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
// ...
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
// ...
contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);
```

Every unique `contractId` passed by the caller creates a new entry in `contractSlotsCache` (keyed by `EntityId`). Every unique `(contractId, slotKey)` pair creates an entry in `contractStateCache`. Both caches are global singletons shared across all concurrent requests.

**Cache limits (from `CacheProperties.java` lines 25–28):**
- `contractSlotsCache`: `expireAfterAccess=5m, maximumSize=3000`
- `contractStateCache`: `expireAfterWrite=2s, maximumSize=25000`

**Rate limiting (from `ThrottleProperties.java` line 35, `ThrottleManagerImpl.java` lines 37–42):**
- `requestsPerSecond=500` — a **single global token bucket**, not per-IP or per-user.

**Root cause:** The global rate limit is the only guard. It is shared across all callers, so an attacker consuming 500 req/s leaves zero capacity for legitimate users. Even at lower rates, an attacker can fill `contractSlotsCache` (3000 entries) in 6 seconds at 500 req/s, or in 50 seconds at 60 req/s. Once full, Caffeine's LRU eviction removes legitimate contracts' cached slot data, forcing DB fallback for every subsequent legitimate request.

**Why existing checks fail:**
- `ThrottleManagerImpl.throttle()` uses a single shared `rateLimitBucket` — no per-source identity is tracked.
- There is no IP-based, session-based, or authenticated-user-based rate limiting on the web3 endpoint.
- The `contractSlotsCache` has a 5-minute access TTL, so evicted legitimate entries do not recover quickly.

### Impact Explanation

1. **Cache thrashing:** Once `contractSlotsCache` is saturated with 3000 attacker-controlled contract IDs, every new legitimate request for a different contract evicts an existing entry. The `findStorageBatch()` optimization (batch DB query for all known slots of a contract) is defeated; every SLOAD for a legitimate contract hits the database.
2. **Rate limit starvation:** At 500 req/s the attacker consumes the entire global budget, causing all legitimate callers to receive HTTP 429.
3. **DB amplification:** `findStorageBatch()` issues a `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (...)` for every cache miss. Cache thrashing converts what should be rare DB calls into per-request DB calls, increasing database load proportionally.
4. **Severity:** Medium-High DoS. No data loss or privilege escalation, but sustained service degradation or unavailability for all users is achievable with no credentials.

### Likelihood Explanation

- **Preconditions:** None. The `/api/v1/contracts/call` endpoint is public and unauthenticated.
- **Attacker capability:** Any HTTP client. No special knowledge required beyond knowing the endpoint exists.
- **Feasibility:** Filling `contractSlotsCache` (3000 entries) requires only 3000 requests with distinct `to` addresses. At even 60 req/s this takes 50 seconds. Contract addresses can be synthetically generated (e.g., `0x0000000000000000000000000000000000000001` through `0x0000000000000000000000000000000000000BB8`).
- **Repeatability:** The attack is continuous and self-sustaining as long as the attacker keeps sending requests. The 5-minute TTL on `contractSlotsCache` means the attacker only needs to re-flood every 5 minutes to maintain cache pollution.

### Recommendation

1. **Add per-IP rate limiting** at the servlet filter or gateway level (e.g., using bucket4j keyed by `X-Forwarded-For` / remote address) in addition to the global bucket.
2. **Bound cache insertion by request identity:** Track how many unique contract IDs a single IP/session has inserted in a time window and reject or throttle excess.
3. **Reduce `contractSlotsCache` TTL** or add a write-rate limiter so a single burst cannot fill the cache before Caffeine's eviction kicks in.
4. **Consider authentication** for the `/api/v1/contracts/call` endpoint in production deployments, or deploy a WAF/API gateway with per-source rate limiting in front of the service.

### Proof of Concept

```bash
# Generate 3001 unique contract addresses and flood the endpoint
for i in $(seq 1 3001); do
  ADDR=$(printf "0x%040x" $i)
  curl -s -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$ADDR\",\"data\":\"0x\",\"gas\":21000}" &
done
wait

# After this, contractSlotsCache is full with attacker entries.
# Legitimate users querying real contracts now miss the cache and hit the DB on every SLOAD.
# Repeat every 5 minutes to maintain cache pollution.
```
