### Title
Unauthenticated Alias-Based DB Query Flooding Enables DoS via Unrated, Uncached 404 Path

### Summary
The `getAccountCryptoAllowances()` handler unconditionally issues a database query for every request containing a syntactically valid but non-existent account alias. Because 404 responses are explicitly excluded from the response cache and no rate-limiting middleware exists in the REST stack, an unprivileged attacker can flood the endpoint with unique valid-format alias strings, exhausting the DB connection pool and denying service to legitimate users.

### Finding Description

**Code path:**

`getAccountCryptoAllowances()` at [1](#0-0)  calls `EntityService.getEncodedId()` with the raw path parameter before any caching or rate-limiting check.

Inside `getEncodedId()`, when the input passes `AccountAlias.isValid()` (regex `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/`), it calls `getAccountIdFromAlias()`: [2](#0-1) 

`getAccountFromAlias()` unconditionally executes: [3](#0-2) 

When the alias does not exist, `rows` is empty, `getAccountIdFromAlias()` throws `NotFoundError`: [4](#0-3) 

**Why existing checks fail:**

The response cache (`responseCacheUpdateHandler`) only stores responses when the status code is a success: [5](#0-4) 

A 404 (`NotFoundError`) is never cached. Every unique alias string therefore hits the database unconditionally.

The `authHandler` middleware only adjusts the *result-set size limit* for authenticated users; it performs no request-rate throttling: [6](#0-5) 

No other rate-limiting middleware is registered in the REST stack. The DB pool is bounded by `maxConnections`: [7](#0-6) 

### Impact Explanation
An attacker who exhausts the DB connection pool blocks all other API endpoints that share the same pool. Because the mirror node REST API is the primary read interface for Hedera network state (balances, transactions, allowances), a sustained attack renders the entire public API unavailable. The alias query touches the `entity` table on every request; even with an index, connection-slot exhaustion occurs before query latency becomes the bottleneck.

### Likelihood Explanation
No authentication, CAPTCHA, or IP-based throttle is required. The alias regex accepts any uppercase base32 string (e.g., `AAAAAAAAAA`, `BBBBBBBBBB`, …), giving an attacker an effectively unbounded supply of unique cache-miss keys. A single machine with a modest HTTP concurrency tool (e.g., `wrk`, `ab`, `hey`) can sustain thousands of requests per second, each generating a distinct DB query. The attack is trivially repeatable and requires no special knowledge of the system.

### Recommendation
1. **Rate-limit by IP** at the application or reverse-proxy layer (e.g., `express-rate-limit`, nginx `limit_req`) for all `/accounts/:id/allowances/*` endpoints.
2. **Cache negative (404) responses** for alias lookups with a short TTL (e.g., 5–10 s) so repeated identical aliases are absorbed without hitting the DB.
3. **Add a DB-level statement timeout** (already configurable via `pool.statementTimeout`) and ensure it is set aggressively for alias-lookup queries.
4. **Consider a short-lived in-process negative cache** (e.g., an LRU map) for alias→notFound results to absorb burst traffic before it reaches the DB pool.

### Proof of Concept
```bash
# Generate 10 000 unique valid-format aliases and flood the endpoint
for i in $(seq 1 10000); do
  ALIAS=$(python3 -c "import random,string; print(''.join(random.choices('ABCDEFGHIJKLMNOPQRSTUVWXYZ234567',k=20)))")
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/allowances/crypto" &
done
wait
```
Each request passes `AccountAlias.isValid()`, triggers `SELECT id FROM entity WHERE alias = $1`, receives an empty result, and returns HTTP 404 — never cached. With sufficient concurrency the DB connection pool (`maxConnections`) is saturated, and subsequent legitimate requests receive connection-timeout errors.
