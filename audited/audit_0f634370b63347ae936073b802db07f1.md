### Title
Cache Stampede via 1-Second Default TTL and Absent Rate Limiting in `responseCacheCheckHandler`

### Summary
`responseCacheCheckHandler` in `rest/middleware/responseCacheHandler.js` falls back to `DEFAULT_REDIS_EXPIRY = 1` (one second) when no parseable `cache-control: max-age` header is present. Because the REST service has zero rate-limiting middleware and no cache-stampede protection (no mutex, no in-flight deduplication), any unauthenticated user can sustain a request rate of ≥1 req/s to force repeated cache misses, each of which incurs a Redis pipeline round-trip (`ttl` + `get`) **and** a full database query, driving node resource consumption well above the 30% threshold.

### Finding Description

**Exact code path:**

`rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler`, lines 40–87:

```js
// line 43 – unconditional Redis round-trip on every request
const cachedTtlAndValue = await getCache().getSingleWithTtl(responseCacheKey);

// lines 45-48 – cache miss: falls through to full backend processing
if (!cachedTtlAndValue) {
  res.locals[responseCacheKeyLabel] = responseCacheKey;
  return;
}
```

`getCacheControlExpiryOrDefault`, lines 155–164:

```js
return DEFAULT_REDIS_EXPIRY;   // = 1 (second) when no max-age header
```

`Cache.getSingleWithTtl`, `rest/cache.js` lines 69–83: issues a Redis `MULTI / TTL / GET / EXEC` pipeline on every call.

**Root cause and failed assumption:**

The design assumes that a 1-second TTL is an acceptable fallback and that upstream infrastructure (load balancer, CDN) will absorb high-frequency traffic. Neither assumption is enforced in code. The REST Express application (`rest/server.js`) registers `responseCacheCheckHandler` globally (line 97) with **no rate-limiting middleware anywhere in the chain** (confirmed: zero matches for `rateLimit`, `express-rate`, `rateLimiter` in `rest/**/*.js`). There is also no stampede guard: when the TTL expires, every concurrent in-flight request independently calls `getSingleWithTtl`, gets `undefined`, and proceeds to the database.

**Exploit flow:**

1. Attacker identifies any REST endpoint (e.g., `/api/v1/accounts?...`).
2. Attacker opens N concurrent HTTP connections and sends requests at ≥1 req/s per connection.
3. Every ~1 second the Redis key expires; all N concurrent requests simultaneously receive `cachedTtlAndValue === undefined`.
4. All N requests fall through to full backend processing (PostgreSQL queries, JSON serialisation).
5. `responseCacheUpdateHandler` eventually writes the result back, but the next second the cycle repeats.
6. Each miss also pays the Redis pipeline cost (`ttl` + `get`), compounding I/O load.

**Why existing checks are insufficient:**

- `Cache.getSingleWithTtl` (cache.js line 65) only skips the call when `this.ready === false` (Redis down). When Redis is healthy it always executes.
- `responseCacheUpdateHandler` writes back with `ttl > 0` guard (line 97), but with `DEFAULT_REDIS_EXPIRY = 1` the entry is gone within one second.
- No mutex, probabilistic early expiration, or request coalescing exists anywhere in the middleware chain.
- No rate-limiting middleware is registered in `rest/server.js`.

### Impact Explanation

Each cache miss forces: (a) one Redis `MULTI/TTL/GET/EXEC` pipeline, and (b) one full database query chain. With N=10 concurrent attackers at 2 req/s each, the server processes ≥10 full DB queries per second that would otherwise be served from cache. On a node that normally serves the majority of traffic from cache, this easily exceeds a 30% increase in CPU, memory, and database connection consumption. The attack is amplified by the thundering-herd effect: all concurrent misses land simultaneously, creating query bursts rather than a smooth load.

### Likelihood Explanation

No privileges, API keys, or special knowledge are required. The attacker only needs the public API URL. The attack is trivially scriptable with `curl`, `ab`, `wrk`, or any HTTP load tool. It is continuously repeatable (every second) and scales linearly with the number of attacker connections. The 1-second TTL fallback makes the cache nearly ineffective as a shield against even moderate request rates.

### Recommendation

1. **Increase `DEFAULT_REDIS_EXPIRY`**: Change the fallback from `1` to a value aligned with actual data freshness requirements (e.g., 30–60 seconds). This is the single highest-impact fix.
2. **Add rate limiting**: Register an IP-based rate-limiting middleware (e.g., `express-rate-limit`) in `rest/server.js` before `responseCacheCheckHandler`.
3. **Add stampede protection**: Implement a per-key in-flight lock (e.g., a `Map` of pending Promises) so that only one request triggers backend processing when a key is missing; all others await the same result.
4. **Probabilistic early expiration**: Refresh the cache entry before it expires (e.g., when remaining TTL < 10% of original) to avoid the hard expiry cliff.
