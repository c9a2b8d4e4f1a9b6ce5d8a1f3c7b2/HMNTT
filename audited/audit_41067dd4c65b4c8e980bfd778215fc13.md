### Title
Unbounded Connection Flooding on `getNodes()` gRPC Endpoint Enables Denial of Service

### Summary
The `getNodes()` endpoint in `NetworkController` is publicly accessible with no authentication, no global connection limit, and no per-IP or aggregate rate limiting. The only server-side guard is `maxConcurrentCallsPerConnection = 5`, which is a per-connection limit that an attacker trivially bypasses by opening many parallel TCP connections. Each stream triggers paginated database queries with inter-page delays, exhausting the DB connection pool, bounded-elastic scheduler threads, and OS file descriptors, causing complete denial of service for legitimate clients.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) accepts any unauthenticated `AddressBookQuery` and immediately subscribes a reactive pipeline:

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(networkService::getNodes)
        ...
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`NetworkServiceImpl.getNodes()` (lines 55–77) then:
- Queries the DB for the latest address book timestamp and all node stake data (lines 61–65)
- Enters a `Flux.defer(() -> page(context)).repeatWhen(...)` loop with a **250 ms inter-page delay** on `Schedulers.boundedElastic()` (lines 68–72)
- Each page calls `addressBookEntryRepository.findByConsensusTimestampAndNodeId()` inside a transaction (lines 80–107), consuming a DB connection for the duration of the query

**The only server-level guard** is in `GrpcConfiguration` (line 33):
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```
with `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14). No `maxConnections`, no `maxConnectionAge`, no IP-based throttle, and no global stream count limit are configured anywhere in `GrpcConfiguration`.

**Root cause:** `maxConcurrentCallsPerConnection` is a *per-connection* limit. The total number of TCP connections is unbounded. An attacker opening *N* connections gets *5N* concurrent streams with no server-side rejection.

**Why existing checks fail:**
- The web3 module has `ThrottleConfiguration` (bucket4j, `requestsPerSecond = 500`), but the gRPC module has no equivalent.
- `maxConcurrentCallsPerConnection = 5` does not bound the total number of connections or total concurrent RPCs.
- No authentication or API key is required to call `getNodes()`.

### Impact Explanation

Each concurrent stream holds:
1. A DB connection for each page query (default pool size is finite; exhaustion blocks all DB-dependent operations including `subscribeTopic`)
2. A thread from `Schedulers.boundedElastic()` (capped at `10 × CPU cores` by default) during each page fetch
3. An `AddressBookContext` object plus a `Map<Long, Long>` node stake map in heap memory
4. An OS file descriptor for the TCP connection

With `limit = 0` (unlimited nodes) and `pageDelay = 250 ms`, each stream stays alive for `ceil(nodeCount / pageSize) × 250 ms` — several seconds for a real address book. Thousands of concurrent streams exhaust the DB pool and bounded-elastic scheduler, making the gRPC service completely unresponsive to legitimate clients querying node topology or subscribing to topics.

### Likelihood Explanation

No privileges, accounts, or special tooling are required. A single attacker machine can open thousands of TCP connections to port 5600 using standard gRPC client libraries or `grpcurl` in a loop. The attack is trivially scriptable, repeatable, and requires no prior knowledge beyond the publicly documented endpoint (`grpcurl -plaintext -d '{"file_id": {"fileNum": 102}, "limit": 0}' host:5600 com.hedera.mirror.api.proto.NetworkService/getNodes`). It can be sustained indefinitely since each completed stream can be immediately reopened.

### Recommendation

1. **Add a global concurrent-stream limit** in `GrpcConfiguration`: call `serverBuilder.maxConcurrentCallsPerConnection(...)` alongside a global `serverBuilder.maxConnectionAge(...)` / `serverBuilder.maxConnectionIdle(...)` to recycle connections and bound total resource hold time.
2. **Add a global connection limit**: configure `serverBuilder` with a maximum number of simultaneous connections (e.g., via a Netty channel option or a gRPC `ServerInterceptor` that tracks and rejects connections above a threshold).
3. **Add per-IP or aggregate rate limiting** via a `ServerInterceptor` using the same bucket4j pattern already used in the web3 module (`ThrottleConfiguration`), applied to all gRPC methods including `getNodes`.
4. **Set a maximum stream lifetime**: configure `maxConnectionAge` and `maxConnectionIdle` on the `NettyServerBuilder` to force-close long-lived connections.
5. **Enforce a non-zero minimum `limit`** or cap the maximum number of nodes returned per stream to bound per-stream resource consumption.

### Proof of Concept

```bash
# Open 1000 parallel connections, each requesting unlimited nodes
for i in $(seq 1 1000); do
  grpcurl -plaintext \
    -d '{"file_id": {"fileNum": 102}, "limit": 0}' \
    <target>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes &
done
wait
```

**Preconditions:** Network access to port 5600; no credentials needed.
**Trigger:** Each of the 1000 background processes opens a TCP connection and starts a streaming RPC. With `maxConcurrentCallsPerConnection = 5`, an attacker using 200 connections already saturates 1000 concurrent streams.
**Result:** DB connection pool exhausted → all gRPC handlers stall waiting for a connection → legitimate `getNodes` and `subscribeTopic` calls time out or receive `RESOURCE_EXHAUSTED` errors → complete denial of service for all clients querying node topology.
