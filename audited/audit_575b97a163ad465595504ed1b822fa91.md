### Title
Unprivileged User Can Force Contract-Create Simulation Path via Explicit Zero-Address `to` Field, Bypassing Initcode Validation

### Summary
An unprivileged external user can submit a `/api/v1/contracts/call` request with `to` explicitly set to `"0x0000000000000000000000000000000000000000"` and arbitrary bytes in `data`. Because `ContractController` maps any non-null/non-empty `to` value through `Address.fromHexString()` — which returns `Address.ZERO` for the zero address string — and `TransactionExecutionService.execute()` branches solely on `params.getReceiver().isZero()`, the request is routed into `buildContractCreateTransactionBody()` with the attacker-controlled `data` field used verbatim as EVM initcode. This also bypasses the `hasTo()` bean-validation guard that was intended to require valid initcode when no real destination is provided.

### Finding Description

**Code path:**

`ContractController.constructServiceParameters()` ( [1](#0-0) ) uses `Address.ZERO` only as a sentinel for a *null or empty* `to` field. When the caller supplies the literal string `"0x0000000000000000000000000000000000000000"`, execution falls into the `else` branch and `Address.fromHexString(...)` still produces `Address.ZERO`.

`TransactionExecutionService.execute()` ( [2](#0-1) ) then evaluates:
```java
final var isContractCreate = params.getReceiver().isZero();
```
Because `Address.ZERO.isZero() == true`, the branch at line 86–87 is taken and `buildContractCreateTransactionBody()` is called, which places the caller-supplied `callData` directly into the `initcode` field: [3](#0-2) 

**Bypassed guard — `hasTo()` in `ContractCallRequest`:** [4](#0-3) 
```java
private boolean hasTo() {
    boolean isValidToField = value <= 0 || from == null || StringUtils.isNotEmpty(to);
    return BytecodeUtils.isValidInitBytecode(data) || isValidToField;
}
```
When `value > 0` and `from != null` and `to` is *empty*, the validator requires `data` to be valid initcode. However, setting `to = "0x0000000000000000000000000000000000000000"` makes `StringUtils.isNotEmpty(to)` return `true`, so `isValidToField = true` and the initcode check is skipped entirely — even though the downstream code will treat this as a contract-create and use `data` as initcode anyway.

**Root cause:** The design uses `Address.ZERO` as a dual-purpose sentinel (both "no destination supplied" and "contract create") but the controller does not distinguish between an absent `to` and an explicitly supplied zero address. The single `isZero()` check at line 74 of `TransactionExecutionService` is the sole branch condition, with no upstream normalization or rejection of an explicit zero-address `to`.

### Impact Explanation

The mirror node is a **simulation-only** service; no real Hedera transactions are submitted and no persistent state is mutated. Within that scope the concrete impacts are:

1. **Arbitrary EVM initcode execution in CREATE context** — the attacker runs any bytecode they choose under `ContractCreate` semantics (constructor execution, `SELFDESTRUCT` in constructor, `CREATE`/`CREATE2` sub-calls, etc.) without needing a pre-deployed contract. This is a broader execution surface than a normal `eth_call` to a known contract.
2. **Bypass of the `hasTo()` initcode-validity gate** — the guard that was meant to ensure only well-formed initcode triggers contract-create simulation is circumvented, so malformed or maximally gas-consuming bytecode can be submitted.
3. **Semantic divergence from Ethereum JSON-RPC spec** — in standard `eth_call`/`eth_estimateGas`, `to = 0x0` is a *call* to the zero address, not a deployment. Clients relying on spec-compliant behavior receive incorrect results.
4. **Amplified resource consumption** — contract-create simulation uses `CONTRACT_CREATE_TX_FEES = 1_000_000_000L` ( [5](#0-4) ) and runs initcode up to the full gas limit, potentially consuming more server-side EVM resources than an equivalent call.

Severity: **Medium** (simulation context limits blast radius, but the bypass is clean and repeatable).

### Likelihood Explanation

- **No authentication required** — the `/api/v1/contracts/call` endpoint is public. [6](#0-5) 
- **Trivial to trigger** — a single HTTP POST with `"to": "0x0000000000000000000000000000000000000000"` and arbitrary `"data"` is sufficient.
- **Repeatable at scale** — throttling is gas-bucket-based; a failed or reverted create simulation partially restores the bucket ( [7](#0-6) ), so an attacker can sustain a stream of requests.
- Any developer familiar with Ethereum JSON-RPC semantics would discover this within minutes of reading the API.

### Recommendation

1. **Distinguish absent `to` from explicit zero address at the controller layer.** Track whether `to` was omitted (`null`) vs. explicitly set. Only treat the absent case as contract-create:
   ```java
   boolean isContractCreate = request.getTo() == null || request.getTo().isEmpty();
   // Pass isContractCreate as a separate flag in ContractExecutionParameters
   ```
2. **Remove `Address.ZERO` as a sentinel.** Use an `Optional<Address>` or a dedicated boolean `isContractCreate` field in `ContractExecutionParameters` so `TransactionExecutionService` does not rely on `receiver.isZero()` alone.
3. **Reject explicit `to = 0x0` in `hasTo()` or a dedicated validator** if contract-create is not the caller's intent, or document and enforce that `0x0` is reserved for deployment simulation and requires valid initcode.
4. **Align with EIP-1474 / JSON-RPC spec**: `to` absent → CREATE; `to` present (including zero address) → CALL.

### Proof of Concept

```bash
# Force contract-create simulation with attacker-controlled initcode
# (PUSH1 0x60 PUSH1 0x40 MSTORE ... arbitrary bytecode)
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to":   "0x0000000000000000000000000000000000000000",
    "data": "0x6060604052600a8060106000396000f360606040526008565b00",
    "gas":  15000000
  }'
```

**Expected (spec-compliant):** call to the zero address, routed through `buildContractCallTransactionBody()`.

**Actual:** `receiver.isZero() == true` → `buildContractCreateTransactionBody()` is invoked; the `data` bytes are passed as `initcode` and executed in EVM CREATE context. The `hasTo()` bean-validation guard is satisfied because `to` is non-empty, so no `isValidInitBytecode` check is performed on `data`.
