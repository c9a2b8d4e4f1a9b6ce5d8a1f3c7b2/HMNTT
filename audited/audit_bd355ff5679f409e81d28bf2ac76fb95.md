### Title
Metric Label Cardinality Explosion via Unmatched Route Paths in `metricsMiddleware`

### Summary
In `rest/middleware/metricsHandler.js`, the `toOpenApiPath()` function falls back to `req.path` (the raw decoded request path) when no Express route matches a request. Because `req.path` is unique per arbitrary URL, an unprivileged attacker can flood the server with requests to distinct paths, causing the OpenTelemetry SDK to accumulate an unbounded number of unique label sets in heap memory — one per unique `{method, path, code}` combination — across three histograms and one counter, leading to heap exhaustion and process degradation.

### Finding Description

**Exact code path:**

In `toOpenApiPath()` ( [1](#0-0) ), the function resolves the metric `path` label via three priority levels:

1. `res.locals[requestPathLabel]` — set only by `recordRequestPath` in `rest/routes/index.js` for **matched** routes [2](#0-1) 
2. `req.route.path` — the route template, available only for **matched** routes [3](#0-2) 
3. **`req.path`** — the raw decoded request path, used as fallback when **no route matches** [4](#0-3) 

For any request that does not match a registered route (resulting in a 404), neither `res.locals[requestPathLabel]` nor `req.route` is populated. The function therefore uses `req.path` verbatim — the attacker-controlled, decoded URL path — as the metric label.

This label is then used in `metricsMiddleware`'s `res.on('finish', ...)` callback to record four instruments: [5](#0-4) 

```js
const labels = {method, path, code};
requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
```

Each unique `{method, path, code}` triple causes the OpenTelemetry SDK to allocate a new internal data point (with bucket arrays for each histogram). These accumulate indefinitely in the `MeterProvider`'s in-memory store — there is no eviction, cap, or normalization. [6](#0-5) 

**Root cause:** The failed assumption is that `toOpenApiPath()` will always resolve to a bounded set of route templates. This holds for matched routes but is completely violated for unmatched routes, where attacker-controlled path strings become metric labels.

**Why URL-encoding is a minor detail:** Express automatically decodes `req.path`, so `%41%42%43` → `ABC`. URL-encoding is not required for the attack; any distinct path string suffices. The encoding angle does not add or remove exploitability.

**No rate limiting:** A search across `rest/**/*.js` for rate-limiting middleware (`rateLimit`, `throttle`) found zero production matches — only a test utility reference. 

### Impact Explanation

Each unique path injects new bucket arrays into three histograms (default bucket counts: 4 for duration, 0 for request size, 4 for response size) plus a counter entry. With tens of thousands of unique paths, the Node.js heap grows without bound. At sufficient scale this triggers aggressive GC cycles, then OOM, crashing the REST API process. A crashed or severely degraded REST API prevents clients and downstream services from querying chain state, effectively stalling observable block ingestion from the external perspective. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation

The attack requires zero privileges — any HTTP client can send GET requests to arbitrary paths. It is trivially scriptable (e.g., `for i in $(seq 1 100000); do curl http://host/api/v1/x$i & done`). The absence of rate limiting makes it repeatable and sustainable. The only practical barrier is network access to the REST API port, which is typically public-facing.

### Recommendation

1. **Normalize unmatched-route labels:** In `toOpenApiPath()`, replace the `path = req.path` fallback with a fixed sentinel string such as `"unknown"` or `"unmatched"`:
   ```js
   if (!req.route) {
     path = 'unknown';  // instead of req.path
   }
   ```
2. **Add rate limiting:** Apply a per-IP rate limiter (e.g., `express-rate-limit`) before `metricsMiddleware` to bound the rate of unique label creation.
3. **Cap label cardinality:** Consider using an OTel view with a cardinality limit on the per-route instruments.

### Proof of Concept

```bash
# Send 50,000 requests to unique paths (no auth required)
for i in $(seq 1 50000); do
  curl -s "http://<mirror-node-host>/api/v1/nonexistent_path_$i" > /dev/null &
done
wait

# Each request creates a unique label {method=GET, path=/api/v1/nonexistent_path_N, code=404}
# across durationHistogram, requestSizeHistogram, responseSizeHistogram, requestTotalCounter.
# Monitor heap: watch -n1 "curl -s http://<host>/metrics | grep heap_used"
# Heap grows linearly; at sufficient N the process OOMs or GC stalls the event loop.
```

```javascript
const toOpenApiPath = (req, res) => {
  let path = res.locals[requestPathLabel];

  if (!path) {
    if (!req.route) {
      path = req.path;
    } else {
      path = (req.baseUrl ?? '') + req.route?.path;
    }
  }

  path = path.replace(/:([^/]+)/g, '{$1}');

  if (!path.startsWith(apiPrefix)) {
    return apiPrefix + '/' + path;
  }

  return path;
};
```

```javascript
const recordRequestPath = async (req, res) => {
```
