### Title
Thundering-Herd DoS via Unjittered `delaySubscription` in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` applies `delaySubscription(interval, scheduler)` with a fixed, unjittered delay before the first database poll. When an unprivileged attacker opens many gRPC subscriptions simultaneously, all of them schedule their first `poll()` to fire at exactly `t + interval`, producing a synchronized burst of concurrent database queries. No per-user, per-IP, or global subscription rate limit exists on the gRPC endpoint to prevent this. The resulting DB overload causes query timeouts that propagate as errors to all active subscribers, including legitimate ones.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` at lines 38–39:
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)   // ← fixed delay, zero jitter
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)                      // ← jitter only on repeat, not initial
                .withFixedDelay(interval)
                .withScheduler(scheduler))
``` [1](#0-0) 

`delaySubscription(interval, scheduler)` schedules the first `poll(context)` call after exactly `interval` milliseconds (default 500 ms) with no randomization. The `jitter(0.1)` on `RepeatSpec` only randomizes the delay between the second and subsequent polls — it has no effect on the initial subscription delay. [2](#0-1) 

Each call to `listen()` creates an independent cold publisher with its own `PollingContext`, so N simultaneous subscriptions produce N independent DB queries at `t + interval`. [3](#0-2) 

Each poll executes `topicMessageRepository.findByFilter(newFilter)` — a real database query — with no caching or deduplication across subscriptions. [4](#0-3) 

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (default) limits calls per single gRPC connection, but an attacker can open arbitrarily many connections, each contributing 5 subscriptions. [5](#0-4) 

- `TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge metric — it is never used to enforce a maximum or reject new subscriptions. [6](#0-5) 

- The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` module (EVM contract calls) and is entirely absent from the gRPC listener path. [7](#0-6) 

- `ConsensusController.subscribeTopic()` passes the request directly to `topicMessageService::subscribeTopic` with no admission control. [8](#0-7) 

### Impact Explanation
When the DB connection pool is saturated by the synchronized burst, queries time out and the reactive error propagates upstream through `Flux.fromStream(topicMessageRepository.findByFilter(...))` to every subscriber's `onError` handler. Legitimate subscribers — including any application polling for time-sensitive topic messages — receive errors and miss messages for the duration of the attack. Because `PollingTopicListener` is a cold publisher with no shared state across subscriptions, there is no circuit-breaker or back-pressure mechanism that would shed attacker load while protecting legitimate subscribers. Severity is high availability impact on the gRPC service.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. An attacker needs only a standard gRPC client (e.g., `grpcurl`, the Hedera SDK, or a trivial script) to open hundreds of simultaneous connections. The attack is repeatable: after each interval the attacker can reconnect and re-trigger the burst. The default interval of 500 ms means the synchronized storm recurs every 500 ms if subscriptions are kept alive. No special privileges, tokens, or on-chain assets are required.

### Recommendation
1. **Add jitter to the initial delay**: Replace `delaySubscription(interval, scheduler)` with a randomized delay, e.g.:
   ```java
   Duration jitteredDelay = interval.multipliedBy((long)(1000 + ThreadLocalRandom.current().nextInt(0, 200))) .dividedBy(1000);
   .delaySubscription(jitteredDelay, scheduler)
   ```
2. **Enforce a global subscription cap**: Use the existing `subscriberCount` atomic to reject new subscriptions above a configurable threshold.
3. **Apply per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer, analogous to the `ThrottleManagerImpl` used in `web3`.
4. **Consider a shared/hot publisher** (`SharedPollingTopicListener`) for the `POLL` mode so that N subscribers share a single DB poll cycle rather than each issuing independent queries.

### Proof of Concept
```bash
# Open 200 simultaneous gRPC subscriptions at t=0
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1000},"consensusStartTime":{"seconds":0}}' \
    mirror-node:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# At t=500ms all 200 subscriptions fire poll() simultaneously.
# Monitor DB connection pool exhaustion and subscriber error rates.
# Repeat every 500ms to sustain the attack.
```
All 200 `poll()` calls hit `topicMessageRepository.findByFilter()` at the same instant, exhausting the DB connection pool and causing timeouts that surface as gRPC errors to all subscribers.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
