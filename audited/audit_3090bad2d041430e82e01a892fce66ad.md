### Title
Deleted Entity Returned via Long-Zero EVM Address Branch in `getByEvmAddressAndType()`

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, when an EVM address has its first 12 bytes set to zero (the "long-zero" Hedera native address format), the code routes to `entityRepository.findById()` — a standard Spring Data `CrudRepository` method that issues a bare `SELECT * FROM entity WHERE id = ?` with no `deleted` filter. The other branch (`findByEvmAddress`) and all sibling methods (`findByAlias`) explicitly include `deleted is not true` in their SQL. Any unprivileged caller can craft such an address to retrieve entities that have been deleted, without requiring a network partition or any elevated privilege.

### Finding Description

**Code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0–11 all zero
        return entityRepository.findById(buffer.getLong())         // bytes 12–19 = entity ID
                               .filter(e -> e.getType() == type); // only type checked, NOT deleted
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
```

**Root cause — asymmetric deletion filtering:**

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 13–17:

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

Both custom queries guard against deleted entities. `findById()` is inherited from `CrudRepository` and generates `SELECT * FROM entity WHERE id = ?` — no `deleted` predicate. The post-query `.filter(e -> e.getType() == type)` only checks entity type, not deletion status.

**Failed assumption:** The developer assumed that the long-zero branch (Hedera native ID lookup) would only be reached for live entities, or that the type filter was sufficient. Neither is true.

### Impact Explanation
An attacker can query the GraphQL API for any deleted account/contract/topic/token by supplying a crafted long-zero EVM address. The service returns the full entity record (balance, keys, memo, type, timestamps, etc.) as if the entity is still active. This constitutes incorrect data disclosure: consumers of the API (wallets, explorers, DeFi integrations) may treat a deleted account as valid, enabling downstream logic errors such as crediting balances to non-existent accounts or accepting transactions targeting deleted contracts.

### Likelihood Explanation
No authentication or special privilege is required — the GraphQL endpoint (`/graphql/alpha`) is publicly accessible. The long-zero address format is documented Hedera behavior (shard=0, realm=0, num=N maps to `0x000000000000000000000000<N_as_8_bytes>`), so any developer familiar with Hedera addressing can construct the payload. Entity IDs are sequential integers, making enumeration trivial. The bug is deterministic and repeatable on every request; no race condition or timing dependency is needed.

### Recommendation
Replace the bare `findById()` call in the long-zero branch with a query that also enforces `deleted is not true`, consistent with the other two branches. The simplest fix is to add a dedicated repository method:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Then update `EntityServiceImpl` line 38:

```java
return entityRepository.findByIdAndNotDeleted(buffer.getLong())
                       .filter(e -> e.getType() == type);
```

Apply the same fix to `getByIdAndType()` (line 25), which has the identical problem via the `entityId` input path.

### Proof of Concept

1. Identify a deleted entity ID, e.g., `12345` (decimal) = `0x0000000000003039` (8-byte big-endian hex).
2. Construct the long-zero EVM address (20 bytes, first 12 bytes zero):
   ```
   0x0000000000000000000000000000000000003039
   ```
3. Send a GraphQL query to the public endpoint:
   ```bash
   curl -X POST http://<host>/graphql/alpha \
     -H 'Content-Type: application/json' \
     -d '{"query":"{account(input:{evmAddress:\"0x0000000000000000000000000000000000003039\"}){deleted entityId{num}balance}}"}'
   ```
4. Observe that the response returns the deleted entity's data (including `deleted: true` in the payload, confirming the entity was deleted but is still being served) instead of a null/empty result. [1](#0-0) [2](#0-1)
