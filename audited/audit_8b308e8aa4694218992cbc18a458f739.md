### Title
Unauthenticated DB Connection Pool Exhaustion via Uncached Alias Lookups in `getTokenRelationships()`

### Summary
Every request to `/api/v1/accounts/:idOrAliasOrEvmAddress/tokens` unconditionally executes one or two synchronous PostgreSQL queries with no result caching and no per-IP rate limiting on the Node.js REST service. An unprivileged attacker sending a high volume of concurrent requests with a valid account alias forces two DB queries per request — one in `EntityService.getEncodedId()` and one in `EntityService.isValidAccount()` — which can exhaust the finite connection pool and deny service to all consumers of the mirror node.

### Finding Description

**Code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 67–68:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
const isValidAccount = await EntityService.isValidAccount(accountId);
```

**Query 1 — `getEncodedId()` with an alias input** (`rest/service/entityService.js` lines 125–126):
```js
} else if (AccountAlias.isValid(entityIdString)) {
  return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
```
This calls `getAccountFromAlias()` (line 43), which calls `super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias])` — a live DB query every time.

**Query 2 — `isValidAccount()` is unconditional** (`rest/service/entityService.js` lines 60–62):
```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
```
This always calls `super.getSingleRow(...)` → `this.pool().queryQuietly(...)`, acquiring a connection from the shared pool on every single request, regardless of input type.

**Pool is finite** (`rest/dbpool.js` lines 13–14):
```js
max: config.db.pool.maxConnections,
```

**No caching of alias→ID or entity-existence results** in the REST service layer. The `parseCached` LRU in `rest/entityId.js` (lines 301–332) only caches the parsed `EntityId` struct (shard/realm/num), not the DB lookup result.

**No rate limiting** on the Node.js REST service for unauthenticated users. The throttle/rate-limit infrastructure found (`web3/src/main/java/.../ThrottleConfiguration.java`) belongs to the separate Java `web3` service. The REST server (`rest/server.js`) registers routes with no per-IP throttle middleware.

**Root cause:** The failed assumption is that `isValidAccount()` is cheap or cached. It is neither — it issues a live DB query on every request, and there is no guard preventing an unauthenticated user from triggering it at arbitrary concurrency.

### Impact Explanation
When the pool (`maxConnections`) is saturated, all other endpoints that require DB access — transactions, balances, contracts, NFTs, etc. — begin timing out or returning errors. The `connectionTimeoutMillis` setting causes queued requests to fail after the timeout, but does not prevent the pool from being held. This is a complete denial of service for all consumers of the mirror node REST API, not just the `/tokens` endpoint.

### Likelihood Explanation
No authentication, API key, or proof-of-work is required. Any internet-accessible mirror node deployment is reachable. The attacker needs only a valid alias string (publicly observable on-chain) and a tool capable of sending concurrent HTTP requests (e.g., `ab`, `wrk`, `hey`). The attack is repeatable indefinitely and requires no special knowledge beyond the public API spec. The two-query-per-request pattern doubles the pool pressure compared to a single-query endpoint.

### Recommendation
1. **Cache `isValidAccount()` results** with a short TTL (e.g., 5–30 seconds) keyed on `accountId`, using the existing `quick-lru` or a similar in-process cache already used in `rest/entityId.js`.
2. **Cache alias→ID resolution** in `getAccountIdFromAlias()` similarly, so repeated requests for the same alias do not re-query the DB.
3. **Add per-IP rate limiting** to the Node.js REST service (e.g., via `express-rate-limit`) for unauthenticated callers, consistent with the Traefik-level `inFlightReq`/`rateLimit` middleware already applied to other services in the Helm chart.
4. **Consider combining** the two queries (`getEncodedId` + `isValidAccount`) into a single DB round-trip when the input is an alias, since the alias lookup already retrieves the entity row.

### Proof of Concept
```bash
# Obtain a valid alias from the public API or chain explorer, e.g.:
ALIAS="KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ"
MIRROR_NODE="https://<mirror-node-host>"

# Send 500 concurrent requests (adjust -c to exceed maxConnections)
ab -n 10000 -c 500 \
  "${MIRROR_NODE}/api/v1/accounts/${ALIAS}/tokens"

# Expected result:
# - DB connection pool exhausted
# - Subsequent requests to ANY endpoint (e.g., /api/v1/transactions) begin
#   returning 500 errors or connection timeout errors
# - Pool saturation visible in mirror node logs:
#   "error event emitted on pool" or connection timeout messages
```

Each request in the flood triggers:
1. `SELECT id FROM entity WHERE alias = $1` (alias lookup)
2. `SELECT type FROM entity WHERE id = $1` (existence check)

With 500 concurrent requests and a default `maxConnections` pool, both queries hold connections simultaneously, starving all other API consumers.
