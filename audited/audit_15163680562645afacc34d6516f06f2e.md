### Title
Unauthenticated Metrics Endpoint Exposes Real-Time `inFlightCounter` Enabling Timed Connection Pool Exhaustion

### Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` unconditionally returns `true` when `config.metrics.config.authentication` is falsy, making the Prometheus metrics endpoint publicly accessible by default. This exposes the `api_all_request_in_processing_total` UpDownCounter (`inFlightCounter`) — a real-time gauge of concurrent in-flight requests — to any unauthenticated external caller. An attacker can poll this metric to precisely observe server load and time a flood of expensive API queries to coincide with near-capacity conditions, exhausting the PostgreSQL connection pool and causing a DB-level partition.

### Finding Description

**Code path:**

`rest/middleware/metricsHandler.js`, `authenticate()`, lines 137–144:
```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;          // ← open to all when authentication is falsy/unset
  }
  ...
};
```
When `config.metrics.config.authentication` is `undefined`, `false`, `null`, or `0` — all falsy — the guard short-circuits and grants access unconditionally. There is no default YAML file found in `rest/config/` that sets `authentication: true`, so the out-of-box posture is open.

**Exposed instrument:**

`rest/middleware/metricsHandler.js`, lines 83–85:
```js
inFlightCounter = meter.createUpDownCounter('api_all_request_in_processing_total', {
  description: 'Number of requests currently being processed',
});
```
Incremented at line 169 (`inFlightCounter.add(1)`) on every non-metrics request, decremented at line 186 (`inFlightCounter.add(-1)`) on `res.finish`. The current value is a live, accurate count of concurrent requests being processed by the Node.js process.

**Metrics endpoint serving:**

Lines 156–164 — any request whose normalized path equals `metricsPath` that passes `authenticate()` (which it always does by default) receives the full serialized Prometheus payload, including `api_all_request_in_processing_total`, `api_all_server_error_total`, CPU/memory gauges, and per-route histograms.

**Root cause / failed assumption:**

The design assumes operators will explicitly opt-in to authentication. In practice, the `authentication` key is absent from the default config, so the guard is never active. The metrics endpoint is registered on the same public-facing Express server (`server.js` lines 89–92) — not on a separate internal port — meaning it is reachable by any client that can reach the API.

**Exploit flow:**

1. Attacker discovers the metrics path (default: `/swagger/metrics/` per `uriPath` default, or trivially brute-forced).
2. Attacker polls `GET /swagger/metrics/` at ~1 s intervals with no credentials; receives 200 OK with full Prometheus text.
3. Attacker reads `api_all_request_in_processing_total` to observe real-time concurrency and `api_all_server_error_total` to detect when 5xx responses begin (indicating pool pressure).
4. When `inFlightCounter` approaches the DB pool's `maxConnections` ceiling (visible indirectly via rising 5xx rate), attacker floods the server with expensive, long-running queries (e.g., large timestamp-range scans on `/api/v1/transactions`).
5. Each query holds a DB connection for its full duration. With the pool exhausted, new requests queue and time out, causing the application to lose connectivity to PostgreSQL — a DB-level partition from the application's perspective.

**Why existing checks fail:**

- `authHandler` middleware (`rest/middleware/authHandler.js`) only governs API routes and enforces per-user rate limits; it does not protect the metrics endpoint.
- The metrics `authenticate()` check exists but is bypassed by the falsy-default condition.
- No rate limiting or IP allowlisting is applied to the metrics path.

### Impact Explanation

An attacker gains a real-time oracle of server concurrency at zero cost. This converts a blind volumetric DoS attempt into a precision-timed attack: the attacker can observe exactly when the server is near saturation and apply the final increment of load needed to exhaust the connection pool. Pool exhaustion causes all in-flight DB queries to fail, producing cascading 500 errors and effectively partitioning the mirror node from its database until load subsides. The additional memory/CPU gauges also leak process-level resource state useful for fingerprinting and capacity planning by an adversary.

### Likelihood Explanation

The metrics endpoint is enabled whenever `config.metrics.enabled` is `true` (a standard production setting for observability). Authentication is off by default. The metrics path is predictable (`/swagger/metrics/`). No credentials, special headers, or network position are required — any internet-reachable instance is vulnerable. Polling is cheap (one HTTP GET per second) and leaves minimal log footprint compared to the attack it enables. The attack is fully repeatable.

### Recommendation

1. **Default-deny**: Change the `authenticate()` logic so that the absence of an `authentication` config key causes the endpoint to reject requests, not permit them:
   ```js
   if (!authentication) {
     return false; // deny by default
   }
   ```
2. **Separate internal port**: Serve the metrics endpoint on a dedicated internal-only port (e.g., `9090`) not exposed to the public internet, consistent with standard Prometheus scraping architecture.
3. **Network-layer restriction**: If a single-port design is required, add an IP allowlist middleware before the metrics path check, restricting access to known scraper IPs.
4. **Rate-limit the metrics path**: Apply a strict rate limit (e.g., 1 req/10 s per IP) to prevent high-frequency polling even if authentication is misconfigured.

### Proof of Concept

```bash
# Step 1: Confirm unauthenticated access (no credentials sent)
curl -si http://<mirror-node-host>:<port>/swagger/metrics/

# Expected (vulnerable): HTTP 200 with Prometheus text including:
# api_all_request_in_processing_total 0
# api_all_server_error_total 0

# Step 2: Poll inFlightCounter while flooding with expensive queries
while true; do
  # Fire expensive query in background
  curl -s "http://<mirror-node-host>:<port>/api/v1/transactions?timestamp=gte:0&limit=100" &

  # Read current concurrency
  IN_FLIGHT=$(curl -s http://<mirror-node-host>:<port>/swagger/metrics/ \
    | grep '^api_all_request_in_processing_total' | awk '{print $2}')
  echo "In-flight: $IN_FLIGHT"

  # When approaching pool limit, intensify flood
  sleep 0.5
done

# Result: Once inFlightCounter reaches maxConnections, subsequent requests
# fail to acquire a DB connection → 500 errors → DB partition observed.
```
