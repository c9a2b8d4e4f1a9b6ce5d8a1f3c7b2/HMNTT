### Title
Unauthenticated Alias-Based DB Exhaustion via Unbounded `findByAlias()` Queries in GraphQL Service

### Summary
The `getByAliasAndType()` method in `EntityServiceImpl` issues an uncached, unbounded SQL query to the database for every unique alias value supplied by any caller. No application-level rate limiting exists in the GraphQL module, and the only infrastructure-level controls are either per-IP concurrency caps (Traefik-specific, trivially bypassed with multiple source IPs) or a global per-pod throughput ceiling that a single attacker can fully consume. An unprivileged external user can therefore drive sustained, high-volume `findByAlias()` queries that exhaust database connection pool capacity and degrade response times for all users.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 47–48) calls `entityService.getByAliasAndType(alias, EntityType.ACCOUNT)` for every request that supplies an `alias` input field. [1](#0-0) 

`EntityServiceImpl.getByAliasAndType()` (lines 29–31) immediately calls `entityRepository.findByAlias(decodeBase32(alias))` with no caching, no throttle check, and no deduplication. [2](#0-1) 

`EntityRepository.findByAlias()` (lines 13–14) executes a raw SQL `SELECT * FROM entity WHERE alias = ?1 AND deleted IS NOT TRUE` against the live database for every call. [3](#0-2) 

**Root cause — failed assumption:** The design assumes infrastructure-layer controls are sufficient to bound per-user query rates. They are not:

1. The `CachedPreparsedDocumentProvider` caches only the *parsed GraphQL document* (query structure), not query results. Every unique alias still produces a live database round-trip. [4](#0-3) 

2. The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module; it is entirely absent from the `graphql` module. [5](#0-4) 

3. `GraphQlConfiguration` enforces query complexity (max 200) and depth (max 10) limits, but these do not bound request *volume* — a simple single-field alias query has complexity 1. [6](#0-5) 

**Infrastructure controls and why they are insufficient:**

- `inFlightReq: amount: 5` with `ipStrategy: depth: 1` (Traefik middleware) limits concurrent in-flight requests to 5 per source IP. This is a *concurrency* cap, not a *rate* cap, and is trivially bypassed by distributing requests across even a handful of source IPs (e.g., a small VPS fleet or cloud NAT pool). [7](#0-6) 

- `maxRatePerEndpoint: 250` (GCP backend policy) is a *global* per-pod ceiling, not a per-IP limit. A single attacker can consume the entire 250 RPS budget, leaving zero capacity for legitimate users. [8](#0-7) 

- The `retry: attempts: 3` Traefik middleware can amplify load: a request that triggers a backend error is retried up to 3 times, multiplying database query volume by up to 4×. [9](#0-8) 

- The DB `statement_timeout = 10000` ms limits individual query duration but does not prevent many concurrent queries from saturating the connection pool. [10](#0-9) 

### Impact Explanation

Each unique alias value forces a full index/table scan on the `entity` table. At 250 RPS (the global pod ceiling), an attacker sending unique aliases generates 250 concurrent or near-concurrent DB queries per second. This exhausts the HikariCP connection pool (monitored via `GraphQLHighDBConnections` alert at 75% utilization), causes query queuing, and raises response latency above the 2-second SLA threshold tracked by `GraphQLRequestLatency`. All users sharing the same pod are affected. The impact is service degradation (DoS) with no economic damage to any individual account — consistent with the Medium severity classification in scope. [11](#0-10) 

### Likelihood Explanation

**Preconditions:** None. The GraphQL endpoint (`/graphql/alpha`) is publicly accessible with no authentication required, as confirmed by the Postman collection and documentation. [12](#0-11) 

**Attacker capability:** Any external user with a script and access to 2–3 source IPs (e.g., cloud VMs, Tor exit nodes, or a small botnet) can bypass the per-IP `inFlightReq` cap and sustain maximum throughput. Generating unique Base32-encoded alias strings is trivial. The attack is fully repeatable and requires no special knowledge of the system.

### Recommendation

1. **Add application-level per-IP rate limiting** to the GraphQL module, mirroring the `ThrottleConfiguration` pattern already used in `web3`. Apply it as a servlet filter or Spring interceptor before the GraphQL handler.
2. **Cache alias lookup results** (not just parsed documents) using a short-TTL cache (e.g., Caffeine with `expireAfterWrite=5s`) keyed on the decoded alias bytes, to collapse repeated queries for the same alias.
3. **Enforce a per-IP rate limit at the infrastructure layer** (e.g., GCP Cloud Armor rate-based ban rules per client IP) in addition to the global `maxRatePerEndpoint` cap.
4. **Remove or reconsider the `retry` middleware** for the GraphQL path, or scope it only to network-level errors, to prevent load amplification.

### Proof of Concept

```bash
# Requires: bash, curl, python3 (for unique alias generation)
# No credentials needed. Run from 2+ IPs to bypass inFlightReq.

generate_alias() {
  # Generate a random 20-byte value, Base32-encode it (valid alias format)
  python3 -c "import base64, os; print(base64.b32encode(os.urandom(20)).decode())"
}

flood() {
  while true; do
    ALIAS=$(generate_alias)
    curl -s -o /dev/null -X POST http://<TARGET>/graphql/alpha \
      -H 'Content-Type: application/json' \
      -d "{\"query\": \"{account(input: {alias: \\\"${ALIAS}\\\"}){balance}}\"}" &
  done
}

# Launch from multiple IPs simultaneously
flood
```

Each iteration sends a unique alias, bypassing any query-level caching, and forces a fresh `SELECT * FROM entity WHERE alias = ?` against the database. Running this from 2 IPs saturates the `inFlightReq` cap (5 × 2 = 10 concurrent queries) and approaches the `maxRatePerEndpoint: 250` global ceiling, causing measurable DB connection pool exhaustion and latency degradation for all concurrent users.
