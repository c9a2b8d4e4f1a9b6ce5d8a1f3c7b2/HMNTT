### Title
Unauthenticated EVM Address + Timestamp Combination Triggers Expensive UNION Query Across `entity` and `entity_history` Tables

### Summary
Any unauthenticated user can call `GET /api/v1/contracts/<40-char-hex-evm-address>?timestamp=gte:0.000000001` to force `getContractById()` to execute a UNION query spanning both the `entity` and `entity_history` tables filtered on the `evm_address` column. This is significantly more expensive than a normal entity-ID lookup (which hits only `entity` via primary key) and can be repeated at will with no rate limiting or authentication gate, enabling a griefing/DB-exhaustion attack.

### Finding Description

**Route registration** — no authentication middleware: [1](#0-0) 

**`acceptedContractByIdParameters` explicitly includes `TIMESTAMP`:** [2](#0-1) 

**`validateContractIdParam` explicitly allows any valid EVM address through without error:** [3](#0-2) 

**`getContractById` passes both the EVM address and the extracted timestamp conditions directly into `getContractByIdOrAddressContractEntityQuery`:** [4](#0-3) 

**Inside `getContractByIdOrAddressContractEntityQuery`, the UNION branch is taken whenever `timestampConditions.length !== 0`:** [5](#0-4) 

**The EVM address branch uses `evm_address` column (not the primary-key `id` column) for both sides of the UNION:** [6](#0-5) 

**Root cause:** The UNION-across-history path is gated solely on `timestampConditions.length !== 0`. Because `TIMESTAMP` is an accepted query parameter for this endpoint and EVM addresses are accepted path parameters, any anonymous caller can combine them to force the expensive code path. There is no check that prevents an EVM-address lookup from being paired with a timestamp, and no rate limiting is applied before the DB query is issued.

**Resulting SQL (attacker-triggered):**
```sql
SELECT ... FROM entity e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND e.evm_address = $3
UNION
SELECT ... FROM entity_history e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND e.evm_address = $3
ORDER BY timestamp_range DESC LIMIT 1
```

vs. the cheap path (entity-ID, no timestamp):
```sql
SELECT ... FROM entity e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND e.id = $3
```

The `entity_history` table accumulates every historical state of every entity; on a production network it is orders of magnitude larger than `entity`. Filtering it on `evm_address` (a non-primary-key column) without a covering index is a full or partial sequential scan.

### Impact Explanation
The `entity_history` table grows unboundedly with network activity. Each attacker request forces two full table scans (one per UNION branch) filtered on `evm_address`, plus a sort and limit. Sustained parallel requests from a single unauthenticated client can saturate DB CPU/IO, degrading or denying service to all legitimate API consumers. No funds are at risk, but mirror-node availability is directly impacted.

### Likelihood Explanation
The attack requires zero privileges, zero on-chain activity, and zero cost. The attacker needs only an HTTP client. The trigger URL is trivially constructable from public API documentation. The attack is fully repeatable and parallelizable. Any motivated party wishing to degrade mirror-node availability can execute it immediately.

### Recommendation
1. **Decouple the UNION trigger from the address type.** When the contractId is a numeric entity ID (not an EVM address), the `entity_history` UNION is already gated on timestamp presence. Apply the same logic consistently, but additionally ensure `evm_address` has a composite index with `type` and `timestamp_range` on `entity_history`.
2. **Add rate limiting** on the `/api/v1/contracts/:contractId` endpoint (and all public endpoints) keyed by IP or API token before the DB query is issued.
3. **Require a bounded timestamp range** (e.g., reject open-ended `gte:` without a corresponding `lte:`) when the UNION path would be taken, to limit the scan range on `entity_history`.

### Proof of Concept

```bash
# Single request — triggers UNION across entity + entity_history via evm_address filter
curl "https://<mirror-node>/api/v1/contracts/70f2b2914a2a4b783faefb75f459a580616fcb5e?timestamp=gte:0.000000001"

# Sustained flood — no auth, no cost, arbitrary EVM addresses
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/contracts/$(openssl rand -hex 20)?timestamp=gte:0.000000001" &
done
wait
```

Each iteration causes the server to execute the two-branch UNION query against `entity_history` filtered on a random `evm_address` value, maximising scan cost while guaranteeing a 404 response (no matching row), so the attacker incurs no observable side-effect beyond DB load.
