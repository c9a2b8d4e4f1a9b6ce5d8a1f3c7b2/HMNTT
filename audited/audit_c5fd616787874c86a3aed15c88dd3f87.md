### Title
Race Condition in `RedisTopicListener.getSharedListener()` Causes Permanent Message Loss for New Subscribers

### Summary
A non-atomic sequence between `topicMessages.remove()` (fired via `doOnCancel` when the last subscriber cancels) and `computeIfAbsent()` (called by the next subscriber) creates a race window during which messages published to Redis are silently dropped. The higher-level gap-detection and safety-check mechanisms in `TopicMessageServiceImpl` do not recover these messages because gap detection is skipped when `last == null` (first message for a new subscriber), and the safety check queries forward from the last-received message's timestamp — which is already past the lost messages if any message arrives before the 1-second timer fires.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`

```
Line 59-62  getSharedListener()  →  computeIfAbsent(key, k -> subscribe(topic))
Line 74     subscribe()          →  .doOnCancel(() -> unsubscribe(topic))
Line 82-85  unsubscribe()        →  topicMessages.remove(topic.getTopic())
```

**Root cause:**

`subscribe()` terminates with `.share()` (Reactor's `publish().refCount()`). When the last subscriber cancels, `.share()` cancels the upstream Redis subscription and then fires `doOnCancel`, which calls `unsubscribe()` → `topicMessages.remove()`. These two steps — upstream cancellation and map removal — are not atomic with respect to a concurrent `computeIfAbsent()` call.

**Race window (Thread interleaving):**

```
Thread A (last subscriber cancels):
  1. .share() refCount → 0, upstream Redis sub cancelled
  2. doOnCancel fires → topicMessages.remove("topic.X")   ← key gone

Thread B (new subscriber, concurrent):
  3. computeIfAbsent("topic.X", ...) → key absent → subscribe() called
  4. New Redis SUBSCRIBE command sent (async, not yet active)

Redis broker (between steps 2 and 4):
  → Messages M1, M2, M3 published to "topic.X"
  → No active subscriber on the mirror node → messages dropped
```

**Why existing checks fail:**

`TopicMessageServiceImpl` has two recovery mechanisms:

1. **Per-message gap detection** (`missingMessages`, line 152):
   ```java
   if (last == null || topicContext.isNext(current)) {
       return Flux.just(current);   // ← no gap query when last == null
   }
   ```
   When the new subscriber receives its first message M4 after the race window, `last == null`, so `isNext()` short-circuits to `true` and M1–M3 are never queried.

2. **Safety check** (line 67–70):
   ```java
   Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
       .filter(_ -> !topicContext.isComplete())
       .flatMapMany(_ -> missingMessages(topicContext, null))
       ...
   ```
   If M4 arrives within 1 second (common in any active topic), `topicContext.getLast()` is already M4 when the safety check fires. `missingMessages(ctx, null)` then sets `startTime = M4.consensusTimestamp + 1`, querying only forward — M1–M3 are permanently skipped.

### Impact Explanation
Messages published to a Redis topic channel during the unsubscribe/resubscribe gap are permanently lost for any subscriber that joins during that window and receives at least one subsequent message before the 1-second safety-check timer expires. This directly violates the mirror node's correctness guarantee of delivering all consensus-ordered `TopicMessage` records to gRPC subscribers. Affected subscribers receive a silently incomplete stream with no error signal.

### Likelihood Explanation
Any unauthenticated gRPC client can open and immediately cancel a `subscribeTopic` stream (zero privilege required). On a topic with even moderate traffic, an attacker who is the sole subscriber can reliably open the race window by cancelling, then immediately re-subscribing (or coordinating with a second connection). The window is narrow (milliseconds) but the attack is repeatable at will and requires no special knowledge beyond the topic ID, which is public on-chain data.

### Recommendation
Remove the entry from `topicMessages` atomically only when no new subscriber has raced in. Replace the `remove` + `computeIfAbsent` pattern with a single atomic `compute` that checks whether the stored `Flux` is still the one being cancelled before removing it:

```java
private void unsubscribe(Topic topic, Flux<TopicMessage> expected) {
    topicMessages.remove(topic.getTopic(), expected); // atomic conditional remove
}
```

Pass the specific `Flux` instance into `doOnCancel`/`doOnComplete` so only the originating shared Flux removes itself. Alternatively, use `publish().refCount(1, Duration.ZERO)` with a `replay(0)` warm-up so the map entry is never removed until a replacement is atomically installed.

### Proof of Concept

**Preconditions:** Mirror node running with Redis listener; topic `0.0.1001` exists; moderate message rate (≥1 msg/s).

**Steps:**

1. Client A opens `subscribeTopic(topicId=0.0.1001)` — becomes the sole subscriber; `topicMessages` map holds the shared `Flux`.
2. Client A cancels the stream. `doOnCancel` fires → `topicMessages.remove("topic.1001")`.
3. Immediately (same millisecond), Client B opens `subscribeTopic(topicId=0.0.1001)`. `computeIfAbsent` finds no entry → calls `subscribe()` → sends Redis `SUBSCRIBE` (async).
4. While the Redis `SUBSCRIBE` handshake is in flight, the importer publishes messages M1, M2, M3 to `topic.1001`. No active Redis subscriber on the mirror node → messages dropped.
5. Redis `SUBSCRIBE` completes; Client B starts receiving from M4 onward.
6. Client B's first received message M4: `last == null` → gap detection skipped → M1–M3 not queried.
7. Safety check fires at T+1s: `last == M4` → queries from `M4.timestamp+1` → M1–M3 permanently lost.
8. Client B's stream contains M4, M5, M6… with M1–M3 silently absent and no error emitted.
