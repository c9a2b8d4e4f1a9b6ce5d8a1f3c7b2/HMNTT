### Title
Unbounded Pagination Loop in `getTokenRelationships()` Enables Non-Network-Based DoS via Sustained DB Load

### Summary
The `/accounts/:idOrAliasOrEvmAddress/tokens` endpoint handled by `getTokenRelationships()` generates cursor-based `next` pagination links with no per-IP or global rate limiting applied anywhere in the REST service middleware stack. An unprivileged external attacker can follow these links in a tight automated loop, causing each page request to execute the full DB query pipeline (two sequential queries against `token_account` and `token` tables), sustaining arbitrarily high database load that degrades service for all users.

**No rate limiting exists in the REST service.** `rest/server.js` registers only: `cors`, `compression`, `httpContext.middleware`, `requestLogger`, `authHandler`, `metricsHandler`, and `responseCacheCheckHandler`: [4](#0-3)

**Failed assumption:** The design assumes that infrastructure-level controls (reverse proxy, CDN) will enforce rate limits. No such enforcement is present in the application layer, and the question scope is the application code.

### Impact Explanation

An account with N token associations requires âŒˆN / limitâŒ‰ page requests to exhaust. With `limit=100` (the configurable max), an account holding 10,000 token associations requires 100 sequential requests, each triggering 3â€“4 DB round-trips. A single attacker running this loop at 50 req/s generates ~150â€“200 DB queries per second continuously. Multiple concurrent attackers from different IPs multiply this linearly. The PostgreSQL connection pool (`maxConnections`) becomes the hard ceiling; once saturated, all API endpoints â€” not just this one â€” begin returning errors or timing out, constituting a full service outage for legitimate users.

### Likelihood Explanation

No special privileges, API keys, or accounts are required. Any public Hedera account with a large number of token associations (common for ecosystem participants) serves as the amplification target. The `next` link is machine-readable and self-describing. Automated pagination is a standard pattern in API clients and scrapers. The attack is trivially repeatable from any IP, and distributing it across multiple IPs bypasses any upstream IP-based rate limiting that might be configured at the infrastructure layer.

### Recommendation

1. **Add per-IP rate limiting middleware** to the REST Express application (e.g., `express-rate-limit`) applied globally before route handlers in `rest/server.js`.
2. **Add a DB-level statement timeout** (already partially present via `pool.statementTimeout` config) and ensure it is set aggressively enough to bound per-request DB cost.
3. **Cap the maximum `limit` parameter** at a lower value (e.g., 25) for unauthenticated requests to reduce per-request DB cost.
4. **Add a concurrency limiter** (e.g., `express-slow-down` or a semaphore) to bound simultaneous in-flight DB queries.

### Proof of Concept

```bash
# 1. Identify a target account with many token associations (e.g., 0.0.12345)
# 2. Fetch first page
URL="https://<mirror-node>/api/v1/accounts/0.0.12345/tokens?limit=100&order=asc"

while true; do
  RESPONSE=$(curl -s "$URL")
  NEXT=$(echo "$RESPONSE" | jq -r '.links.next // empty')
  if [ -z "$NEXT" ]; then
    # Reset to first page to loop indefinitely
    URL="https://<mirror-node>/api/v1/accounts/0.0.12345/tokens?limit=100&order=asc"
  else
    URL="https://<mirror-node>$NEXT"
  fi
  # No sleep â€” tight loop
done
```

Each iteration of the loop triggers 3â€“4 DB queries. Running this from multiple terminals or hosts sustains proportionally higher DB load with no application-layer mechanism to stop it.

  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });
  }
