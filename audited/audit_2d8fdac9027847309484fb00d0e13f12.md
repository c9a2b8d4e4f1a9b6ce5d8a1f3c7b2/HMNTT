### Title
Gas Throttle Bucket Drain via Future Block Number — Partial Restore on `BlockNumberNotFoundException`

### Summary
An unprivileged user can send `POST /api/v1/contracts/call` with a syntactically valid but future block number. The request passes all input validation, consumes gas tokens from the throttle bucket, and then throws `BlockNumberNotFoundException` deep inside the EVM layer. Because `BlockNumberNotFoundException` is not `InvalidParametersException`, the controller's `throttleManager.restore()` path is never reached, and only a small configured percentage of the consumed gas is returned to the bucket. Repeated requests drain the gas-per-second bucket, causing legitimate callers to receive `GAS_PER_SECOND_LIMIT_EXCEEDED` throttle errors.

### Finding Description

**Exact code path:**

`ContractController.call()` (lines 38–51) wraps all work in a single try/catch that only restores gas on `InvalidParametersException`:

```java
// ContractController.java lines 38-51
try {
    throttleManager.throttle(request);          // consumes up to 15 M gas tokens
    validateContractMaxGasLimit(request);
    final var params = constructServiceParameters(request);
    final var result = contractExecutionService.processCall(params);
    return new ContractCallResponse(result);
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas()); // ONLY path that fully restores
    throw e;
}
```

`constructServiceParameters()` (lines 76–79) passes the block value through without any future-block guard:

```java
final var block = request.getBlock();
return ContractExecutionParameters.builder().block(block)...build();
```

`BlockType.of()` (lines 18–55 of `BlockType.java`) accepts any non-negative integer, including values far beyond the current chain tip.

Inside `ContractCallService.callContract()` (lines 100–107), the block lookup is registered as a **lazy supplier**:

```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock())
        .orElseThrow(BlockNumberNotFoundException::new)));
return doProcessCall(params, params.getGas(), false);
```

When the EVM layer first dereferences the supplier during `transactionExecutionService.execute()`, `recordFileService.findByBlockType()` returns `Optional.empty()` for a future block, and `BlockNumberNotFoundException` is thrown.

`doProcessCall()` (lines 109–138) catches only `IllegalStateException`, `IllegalArgumentException`, and `MirrorEvmTransactionException`. `BlockNumberNotFoundException extends InvalidInputException` matches none of these, so it falls through to the `finally` block:

```java
} finally {
    if (!estimate) {
        restoreGasToBucket(result, params.getGas()); // result == null here
    }
}
```

`restoreGasToBucket(null, gasLimit)` (lines 140–152) restores only `gasLimitRefundPercent`% of the gas limit:

```java
final var gasLimitToRestoreBaseline =
    (long)(gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
if (result == null || ...) {
    throttleManager.restore(gasLimitToRestoreBaseline); // partial restore only
}
```

`BlockNumberNotFoundException` then propagates back to `ContractController.call()`, which does **not** catch it (it is not `InvalidParametersException`), so the controller's full-restore path is never executed. A global Spring exception handler converts it to HTTP 400, as confirmed by the controller test at line 411–420 of `ContractControllerTest.java`.

**Net effect per malicious request:** `(100 − gasLimitRefundPercent)`% of `request.getGas()` is permanently consumed from the gas-per-second bucket with no real EVM computation performed.

### Impact Explanation

The gas-per-second token bucket is a shared, finite resource. An attacker who sends requests at the maximum allowed request rate, each with `gas = maxGasLimit` (15 000 000) and `block` set to a far-future number, drains the bucket continuously as it refills. Legitimate callers then receive `ThrottleException("Gas per second rate limit exceeded.")` and cannot execute any contract calls. Because the gas bucket is global (not per-IP), a single attacker affects all users of the node. This maps directly to the stated impact of shutting down ≥30 % of network processing capacity without brute-force actions.

### Likelihood Explanation

No authentication or special privilege is required. The attack requires only the ability to send HTTP POST requests to `/api/v1/contracts/call` with a JSON body containing a large numeric `block` field (e.g., `"block": "0xFFFFFFFF"`). The request passes all `@Valid` bean-validation constraints because `BlockType.of()` accepts any non-negative hex/decimal integer. The attack is trivially scriptable, repeatable, and requires no knowledge of on-chain state.

### Recommendation

1. **Validate the block number before consuming throttle tokens.** In `ContractController.call()` (or in `constructServiceParameters()`), compare the requested block number against the current chain tip obtained from `RecordFileService`. If the block is in the future, throw `InvalidParametersException` before `throttleManager.throttle()` is called (or immediately after, inside the existing `catch (InvalidParametersException)` block).

2. **Alternatively, add `BlockNumberNotFoundException` (or its parent `InvalidInputException`) to the controller's catch clause** so that gas is fully restored when no EVM computation has occurred:
   ```java
   } catch (InvalidParametersException | BlockNumberNotFoundException e) {
       throttleManager.restore(request.getGas());
       throw e;
   }
   ```

3. **Longer term**, move future-block rejection into `BlockType.of()` or a dedicated validator so it is uniformly enforced across all entry points.

### Proof of Concept

**Preconditions:** Mirror node is running; current chain tip is block N.

**Steps:**

```bash
# Send a request with a block number far in the future (e.g., 0xFFFFFFFF = 4 294 967 295)
# Repeat at the maximum requests-per-second rate allowed by the rate-limit bucket.

for i in $(seq 1 100); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "block": "0xFFFFFFFF",
      "gas": 15000000,
      "to": "0x0000000000000000000000000000000000000001",
      "data": "0x"
    }' &
done
wait
```

**Expected result:** Each request returns HTTP 400 `{"_status":{"messages":[{"message":"Unknown block number"}]}}` but consumes `(100 − gasLimitRefundPercent)`% of 15 000 000 gas tokens from the shared bucket. After a small number of iterations the gas bucket is empty and all subsequent legitimate requests receive HTTP 429 `Gas per second rate limit exceeded.`

**Verification:** Send a normal `latest`-block call after the flood; observe it is throttled even though no real EVM work was performed by the attacker.
