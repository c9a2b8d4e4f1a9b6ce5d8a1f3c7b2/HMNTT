### Title
Unauthenticated Resource Exhaustion via `getBalancesQuery()` Correlated Token Subquery with Broad Account Filter

### Summary
The `/api/v1/balances` endpoint in `rest/balances.js` allows any unauthenticated caller to trigger a `SELECT DISTINCT ON (account_id)` query over the `account_balance` table combined with a per-row correlated `DISTINCT ON (token_id)` subquery against `token_balance`. With `account.id=gt:0.0.1`, `timestamp=lte:<recent>`, and `limit=100`, each request maximally exercises both the partition scan and the correlated subquery fan-out. The REST API has no rate-limiting middleware, so an attacker can saturate the 10-connection pool (default) with 20-second-duration queries, blocking all legitimate users.

### Finding Description

**Code path:**

`getBalances()` at [1](#0-0)  calls `getTsQuery()` then `getBalancesQuery()` when a `timestamp` filter is present.

`getBalancesQuery()` at [2](#0-1)  builds:

```sql
SELECT DISTINCT ON (account_id)
  ab.account_id, ab.balance,
  $1::bigint AS consensus_timestamp,
  (<correlated_token_subquery>) AS token_balances
FROM account_balance ab
WHERE ab.consensus_timestamp >= $4 AND ab.consensus_timestamp <= $5
  AND ab.account_id > 1          -- from account.id=gt:0.0.1
ORDER BY ab.account_id ASC, ab.consensus_timestamp DESC
LIMIT 100
```

The correlated subquery at [3](#0-2)  runs once per returned row:

```sql
SELECT DISTINCT ON (token_id) token_id, balance
FROM token_balance tb
WHERE tb.account_id = ab.account_id
  AND tb.consensus_timestamp >= $2 AND tb.consensus_timestamp <= $3
ORDER BY token_id ASC, consensus_timestamp DESC
LIMIT 50
```

**Root cause — failed assumption:** The code assumes that the timestamp partition optimization (capping the scan to ~2 monthly partitions via `getOptimizedTimestampRange()`) [4](#0-3)  is sufficient to bound query cost. It is not: on a production network with millions of accounts and 15-minute balance snapshots, a single monthly partition contains hundreds of millions of rows. The `DISTINCT ON (account_id)` with `ORDER BY account_id ASC, consensus_timestamp DESC` cannot be satisfied by a simple forward index scan on the `(account_id, consensus_timestamp)` primary key when the WHERE clause filters on `consensus_timestamp` range — the planner must scan the partition and sort. The correlated token subquery then multiplies this cost by up to 100 (the `limit` cap).

**No rate limiting on the REST API:** The Node.js server middleware stack at [5](#0-4)  includes cors, compression, logging, auth, metrics, and response cache — but no rate-limiting middleware. The `ThrottleManager` with per-second request limits exists only in the `web3` Java service, not in the REST API.

**DB pool is small:** The default `maxConnections` is 10 [6](#0-5)  with `statement_timeout` of 20 000 ms. [7](#0-6) 

### Impact Explanation

An attacker sending 10 concurrent requests with `account.id=gt:0.0.1&timestamp=lte:<recent>&limit=100` saturates all 10 DB connections. Each connection is held for up to 20 seconds (the statement timeout). During that window, every other API user — across all endpoints — receives connection-pool-exhaustion errors or hangs. The attacker can sustain this indefinitely by immediately re-issuing requests as connections are released. This degrades throughput for all users of the mirror node REST API with no authentication, no cost, and no special knowledge beyond the public API spec.

### Likelihood Explanation

The endpoint is publicly documented in the OpenAPI spec at [8](#0-7)  and requires no credentials. The parameters `account.id=gt:X`, `timestamp=lte:Y`, and `limit=100` are all standard, documented query parameters. Any attacker with HTTP access and a script can execute this. The attack is repeatable, stateless, and requires no prior knowledge of account IDs.

### Recommendation

1. **Add rate limiting to the REST API** — apply a per-IP (and global) request rate limiter middleware (e.g., `express-rate-limit`) in `rest/server.js`, analogous to the `ThrottleManager` already present in the `web3` service.
2. **Reduce `statement_timeout`** for the `mirror_rest` DB user from 20 s to a lower value (e.g., 5 s) for the balances endpoint specifically, or use advisory locks / query cancellation.
3. **Increase `maxConnections`** or use a connection pool proxy (PgBouncer is already configured in the Helm chart) with per-user connection limits to prevent a single client from monopolizing all connections. [9](#0-8) 
4. **Require a more selective `account.id` filter** (e.g., require `eq` or a bounded range) when `timestamp` is also provided, to prevent full-partition scans.

### Proof of Concept

```bash
# Saturate the 10-connection pool with concurrent max-cost requests
for i in $(seq 1 15); do
  curl -s "https://<mirror-node>/api/v1/balances?account.id=gt:0.0.1&timestamp=lte:$(date +%s)&limit=100" &
done
wait
# All legitimate API calls during this window will time out or receive 503
```

Each request triggers `getBalancesQuery()` [10](#0-9)  with a full-partition `DISTINCT ON` scan plus 100 correlated `token_balance` subqueries [11](#0-10) , holding a DB connection for up to 20 seconds per request.

```javascript
const getBalancesQuery = async (accountQuery, balanceQuery, accountIdsQuery, limitQuery, order, tsQueryResult) => {
  const tokenBalanceSubQuery = getTokenBalanceSubQuery(order, tsQueryResult.query);
  const whereClause = `
      where ${[tsQueryResult.query, accountQuery, accountIdsQuery, balanceQuery].filter(Boolean).join(' and ')}`;
  const {lower, upper} = tsQueryResult.timestampRange;
  // The first upper is for the consensus_timestamp in the select fields, also double the lower and the upper since
  // they are used twice, in the token balance subquery and in the where clause of the main query
  const tsParams = [upper, lower, upper, lower, upper];
  const sqlQuery = `
      select distinct on (account_id)
        ab.account_id,
        ab.balance,
        ?::bigint as consensus_timestamp,
        (${tokenBalanceSubQuery}) as token_balances
      from account_balance ab
      ${whereClause}
      order by ab.account_id ${order}, ab.consensus_timestamp desc
      ${limitQuery}`;
  return [sqlQuery, tsParams];
};
```

```javascript
const getOptimizedTimestampRange = (tsQuery, tsParams) => {
  let lowerBound = 0n;
  const neParams = [];
  let upperBound = constants.MAX_LONG;

  // Find the lower bound and the upper bound from tsParams if present
  tsQuery
    .split('?')
    .filter(Boolean)
    .forEach((query, index) => {
      const value = BigInt(tsParams[index]);
      // eq operator has already been converted to the lte operator
      if (query.includes(utils.opsMap.lte)) {
        // lte operator includes the lt operator, so this clause must before the lt clause
        upperBound = utils.bigIntMin(upperBound, value);
      } else if (query.includes(utils.opsMap.lt)) {
        // Convert lt to lte to simplify query
        const ltValue = value - 1n;
        upperBound = utils.bigIntMin(upperBound, ltValue);
      } else if (query.includes(utils.opsMap.gte)) {
        // gte operator includes the gt operator, so this clause must come before the gt clause
        lowerBound = utils.bigIntMax(lowerBound, value);
      } else if (query.includes(utils.opsMap.gt)) {
        // Convert gt to gte to simplify query
        const gtValue = value + 1n;
        lowerBound = utils.bigIntMax(lowerBound, gtValue);
      } else if (query.includes(utils.opsMap.ne)) {
        neParams.push(value);
      }
    });

  if (lowerBound > upperBound) {
    return {};
  }

  // The optimized range of [lower, upper] should overlap with at most two months, with the exception that when upper
  // is more than 1 month in the future, the range may cover more months. Since the partition maintenance job will
  // create at most one monthly partition ahead, it's unnecessary to adjust the upper bound.
  // With the assumption that the data in db is in sync with the network, in other words, the balance information is
  // update-to-date as of NOW in wall clock, the algorithm below sets lower bound to
  //   max(lowerBound from user, first day of the month before the month min(now, upperBound) is in)
  const nowInNs = utils.nowInNs();
  const effectiveUpperBound = utils.bigIntMin(upperBound, nowInNs);
  const optimalLowerBound = utils.getFirstDayOfMonth(effectiveUpperBound, -1);
  lowerBound = utils.bigIntMax(lowerBound, optimalLowerBound);

  return {lowerBound, upperBound, neParams};
};
```

```javascript
const getTokenBalanceSubQuery = (order, consensusTsQuery) => {
  consensusTsQuery = consensusTsQuery.replaceAll('ab.', 'tb.');
  return `
    select json_agg(json_build_object('token_id', token_id, 'balance', balance))
    from (
      select distinct on (token_id) token_id, balance
      from token_balance tb
      where tb.account_id = ab.account_id
        and ${consensusTsQuery}
      order by token_id ${order}, consensus_timestamp desc
      limit ${tokenBalanceLimit.multipleAccounts}
    ) as account_token_balance`;
};
```
