### Title
Unauthenticated Block-Existence Oracle via `BlockNumberNotFoundException` on `/api/v1/contracts/call`

### Summary
The `/api/v1/contracts/call` endpoint accepts an arbitrary user-supplied `block` parameter with no authentication requirement. When a block number does not exist in the mirror node's record files, `BlockNumberNotFoundException` is thrown and the controller returns a deterministic HTTP 400 `{"_status":{"messages":[{"message":"Unknown block number"}]}}` response. This binary oracle (400 vs. 200/other) allows any unauthenticated external caller to systematically probe block numbers and map which blocks are present or absent in the mirror node's historical record.

### Finding Description

**Code path:**

`ContractController.call()` (lines 37–51) calls `constructServiceParameters()` (lines 53–90), which reads `request.getBlock()` at line 76 and passes it directly into `ContractExecutionParameters` with no existence check. The block value is later resolved against the database by `RecordFileServiceImpl.findByBlockType()`, which calls `recordFileRepository.findByIndex(block.number())`. When the index is absent, `BlockNumberNotFoundException` is thrown.

`GenericControllerAdvice` handles `InvalidInputException.class` (the parent of `BlockNumberNotFoundException`) at lines 89–93 and returns HTTP 400 with the literal message `"Unknown block number"`. A block that *does* exist produces HTTP 200 (or a different 400 with a contract-execution error message, never `"Unknown block number"`).

**Root cause:** The endpoint is publicly accessible (CORS `allowedOrigins("*")`, no Spring Security guard visible), and the error response is a deterministic, distinguishable oracle for block existence.

**Why existing checks fail:**
- `ThrottleManager.throttle()` is gas-based. The `catch (InvalidParametersException e)` block at line 46–49 restores tokens only for `InvalidParametersException`. `BlockNumberNotFoundException` is an `InvalidInputException`, not an `InvalidParametersException`, so tokens are consumed — but with the minimum gas value (e.g., `gas: 21000`), an attacker can issue thousands of probing requests before hitting any throttle ceiling.
- No IP-rate-limiting or authentication is applied to this endpoint.

### Impact Explanation
An attacker can binary-search the full block-number space to determine: (a) the current chain tip known to the mirror node, (b) any gaps (missing blocks) in the mirror node's ingested history. Gaps indicate periods where the mirror node has no record of transactions, which can be used to identify windows where transaction history is unverifiable or where a mirror node operator may be serving incomplete data. This is an information-disclosure vulnerability enabling targeted reconnaissance of mirror node data integrity without any privileged access.

### Likelihood Explanation
The exploit requires only an HTTP client and knowledge of the public API. No credentials, keys, or internal access are needed. The endpoint is documented and publicly reachable. Probing the full 64-bit block space is impractical, but binary-searching for the chain tip (O(log N) ≈ ~47 requests) and then scanning for gaps around a known range is trivially automatable. The throttle does not prevent this at realistic gas values.

### Recommendation
1. Return a uniform error response (e.g., HTTP 400 `"Invalid block parameter"`) for both out-of-range and non-existent block numbers, eliminating the distinguishable oracle.
2. Apply IP-based rate limiting (e.g., via a gateway or Spring's `HandlerInterceptor`) to the `/api/v1/contracts/call` endpoint independent of gas consumption.
3. Consider returning HTTP 400 for block numbers beyond the current chain tip and HTTP 400 with the same generic message for gaps, so the two cases are indistinguishable to callers.

### Proof of Concept

```bash
# Step 1: Find the current chain tip (binary search, ~47 requests)
# Probe block 0x7FFFFFFFFFFFFFFF → 400 "Unknown block number"
curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"block":"0x7FFFFFFFFFFFFFFF","to":"0x0000000000000000000000000000000000000001","gas":21000}'
# → {"_status":{"messages":[{"message":"Unknown block number"}]}}  (absent)

# Probe block 0x1 → 200 or different error (present)
curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"block":"0x1","to":"0x0000000000000000000000000000000000000001","gas":21000}'
# → 200 or contract-execution error (NOT "Unknown block number") → present

# Step 2: Binary search between known-present and known-absent to find tip.
# Step 3: Scan sequentially around suspected gaps:
for block in $(seq $KNOWN_GAP_START $KNOWN_GAP_END); do
  result=$(curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"block\":\"$(printf '0x%x' $block)\",\"to\":\"0x0000000000000000000000000000000000000001\",\"gas\":21000}")
  echo "Block $block: $result"
done
# Blocks returning "Unknown block number" are absent from the mirror node's record.
