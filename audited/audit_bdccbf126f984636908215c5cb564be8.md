### Title
Unbounded `topicMessages` ConcurrentHashMap Enables Resource Exhaustion DoS via Distinct Topic Subscriptions

### Summary
`RedisTopicListener.getSharedListener()` inserts one entry per unique topic ID into an unbounded `ConcurrentHashMap` and opens a corresponding Redis channel subscription for each. There is no cap on the map size and no limit on total gRPC connections, so an attacker who opens many connections (each carrying up to 5 concurrent calls to distinct valid topics) can exhaust JVM heap and Redis subscription resources. Legitimate subscribers are then starved of service.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, line 61:
```java
return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
```
`subscribe()` (line 68–79) calls `r.receive(...)` on the shared `ReactiveRedisMessageListenerContainer`, creating a live Redis channel subscription, then wraps it with `.share()`. The entry is only removed from the map in `unsubscribe()` (lines 82–85), which fires via `doOnCancel`/`doOnComplete` — i.e., only when the last downstream subscriber disconnects.

**Root cause:** `topicMessages` is a plain `ConcurrentHashMap` with no size bound. While any subscriber to a given topic remains connected, the map entry and its Redis channel subscription are pinned in memory.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties.java:14`, `GrpcConfiguration.java:33`) limits calls *per connection*, not total connections. There is no `maxConnections` or global subscription cap anywhere in the gRPC server configuration.
- `checkTopicExists = true` (`GrpcProperties.java:19`, `TopicMessageServiceImpl.java:96–105`) requires a valid topic in the DB, but Hedera mainnet/testnet exposes millions of valid topic IDs publicly. This is friction, not a barrier.
- The gateway `maxRatePerEndpoint: 250` (`charts/hedera-mirror-grpc/values.yaml:69`) is a request-rate limit, not a concurrent-subscription cap, and is only present in GCP gateway deployments.

**Exploit flow:**
1. Attacker enumerates N valid topic IDs (publicly available on-chain).
2. Attacker opens ⌈N/5⌉ gRPC connections, each issuing 5 `subscribeTopic` calls to distinct topic IDs, and holds them open.
3. Each unique topic ID causes `computeIfAbsent` to insert a new entry and call `subscribe()`, which calls `r.receive(...)` on the Redis container.
4. The map grows to N entries; N Redis channel subscriptions are held open.
5. JVM heap fills with `Flux` state objects; Redis subscription overhead accumulates.
6. New legitimate `subscribeTopic` calls either fail with OOM errors or time out waiting for Redis.

### Impact Explanation
The mirror node's gRPC service becomes unavailable for legitimate subscribers. Applications that poll HCS topic messages via the mirror node (e.g., token-bridge relayers, DApp event listeners) stop receiving messages for as long as the attack is sustained. This is a sustained denial-of-service, not a permanent state change — entries are cleaned up if the attacker disconnects — but the attacker can trivially reconnect to re-establish the condition. The "permanent fund freeze" framing in the question is inaccurate: this is a read-only mirror node and holds no funds. The real severity is **high-impact DoS** against the gRPC subscription service.

### Likelihood Explanation
The attack requires no credentials, no privileged access, and no on-chain transactions. Any client with network access to port 5600 can execute it. Valid topic IDs are freely discoverable via the mirror node's own REST API. The attacker needs only a script that opens many TCP connections and issues `subscribeTopic` RPCs. This is low-skill and easily repeatable.

### Recommendation
1. **Cap the map size:** Replace `ConcurrentHashMap` with a bounded structure (e.g., Caffeine cache with a maximum entry count and eviction policy) so that the number of concurrent Redis channel subscriptions is bounded.
2. **Limit total concurrent subscriptions globally:** Track total active subscriptions with an `AtomicInteger` and reject new ones above a configurable threshold (analogous to the existing per-connection `maxConcurrentCallsPerConnection`).
3. **Limit connections per IP:** Configure Netty's `maxConnectionsPerIp` or use an external rate-limiter/firewall rule to cap the number of simultaneous gRPC connections from a single source.
4. **Require topic existence unconditionally:** Ensure `checkTopicExists` cannot be disabled in production, reducing the pool of usable topic IDs to those actually present in the local DB.

import grpc, threading

stubs = []

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
