### Title
Unauthenticated Concurrent `GET /contracts` Requests Cause Unbounded DB Load (No Rate Limiting)

### Summary
The `getContracts` handler in `rest/controllers/contractController.js` executes a full `entity LEFT JOIN contract` table scan filtered only by `type = 'CONTRACT'` and capped by `defaultLimit` rows. No application-level rate limiting exists in the REST server, so any unauthenticated external caller can flood the endpoint with concurrent requests, causing sustained and potentially exhausting database load.

### Finding Description

**Code path:**

`getContracts` (lines 745–771) calls `extractSqlFromContractFilters` (lines 83–144) then `getContractsQuery` (lines 217–229), producing:

```sql
SELECT <fields>
FROM entity e
LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT'
ORDER BY e.id DESC
LIMIT $1          -- defaultLimit, e.g. 25
``` [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — no rate limiting:**

`server.js` registers only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optional response-cache middleware. There is no `express-rate-limit` or equivalent per-IP/global request throttle for the REST API. [4](#0-3) 

The `authHandler` middleware provides optional authentication that grants authenticated users a custom `limit` cap; it does **not** block or throttle unauthenticated callers.

**Why `defaultLimit` is insufficient protection:**

`LIMIT 25` caps the *rows returned*, not the *scan cost*. PostgreSQL must still locate all `entity` rows where `type = 'CONTRACT'`, perform the `LEFT JOIN`, and sort by `e.id DESC` before it can return the first 25 rows. On a large entity table this is an expensive operation per request. [5](#0-4) 

Additionally, a caller may supply `?limit=<max>` to increase per-query cost up to the configured maximum. [6](#0-5) 

### Impact Explanation
An attacker sending hundreds of concurrent `GET /api/v1/contracts` requests (zero authentication required) forces the database to execute many simultaneous full-scan + JOIN + sort operations. This can saturate the DB connection pool (`maxConnections` in `db.pool`), spike CPU/IO on the database host, and degrade or deny service to all other API consumers. The impact is a practical denial-of-service against the mirror-node REST API and its underlying database.

### Likelihood Explanation
The exploit requires no credentials, no special knowledge, and no complex tooling — a simple `ab`, `wrk`, or `curl` loop suffices. The endpoint is publicly documented and reachable on any deployed mirror-node instance. The attack is trivially repeatable and can be sustained indefinitely.

### Recommendation
1. **Add per-IP rate limiting** at the Express application layer (e.g., `express-rate-limit`) before route handlers, covering all `/api/v1/*` routes.
2. **Set a DB-level statement timeout** (`statementTimeout` in `db.pool` config) to bound the maximum wall-clock cost of any single query.
3. **Ensure a partial index** exists on `entity(id) WHERE type = 'CONTRACT'` so the query uses an index scan rather than a sequential scan, reducing per-query cost.
4. Consider requiring authentication (or a CAPTCHA/token) for high-cost listing endpoints, or enforce stricter default limits for unauthenticated callers.

### Proof of Concept
```bash
# Send 200 concurrent requests with no filters, no authentication
ab -n 2000 -c 200 "https://<mirror-node-host>/api/v1/contracts"

# Or with wrk
wrk -t 10 -c 200 -d 30s "https://<mirror-node-host>/api/v1/contracts"
```

Observe: database CPU/IO spikes, connection pool saturation, increased query latency for all other endpoints, and potential HTTP 503/timeout responses from the REST API.

const getContractsQuery = (whereQuery, limitQuery, order) => {
  return [
    `select ${contractSelectFields}`,
    `from ${Entity.tableName} ${Entity.tableAlias}`,
    `left join ${Contract.tableName} ${Contract.tableAlias}`,
    `on ${Entity.getFullName(Entity.ID)} = ${Contract.getFullName(Contract.ID)}`,
    whereQuery,
    `order by ${Entity.getFullName(Entity.ID)} ${order}`,
    limitQuery,
  ]
    .filter((q) => q !== '')
    .join('\n');
};
```
