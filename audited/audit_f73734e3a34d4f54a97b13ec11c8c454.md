### Title
Unauthenticated Concurrent Request Amplification via Unrestricted `hook.id` EQ Filter Fanout in `getHooks()`

### Summary
The `getHooks()` endpoint in `HooksController` accepts up to 100 `hook.id=eq:X` parameters per request and translates them into a single 100-element SQL `IN` clause with no per-IP or global rate limiting in the `rest-java` module. An unauthenticated attacker sending 10+ concurrent requests each at the parameter maximum can force 10+ simultaneous large DB queries, exhausting connection pool slots and elevating CPU/DB load well above 30% of baseline without any single request being rejected.

### Finding Description
**Code path:**

- `HooksController.java` lines 83–86: `hookId` is bound with `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` (= 100) and `limit` with `@Max(MAX_LIMIT)` (= 100). No authentication, no rate-limit annotation, no IP throttle.
- `Constants.java` lines 35–36: `MAX_LIMIT = 100`, `MAX_REPEATED_QUERY_PARAMETERS = 100`.
- `HookServiceImpl.java` lines 39–52: when `hookIds` is non-empty, all IDs are passed directly to `hookRepository.findByOwnerIdAndHookIdIn(id.getId(), idsInRange, page)`, generating `WHERE hook_id IN (?, ?, … ×100)` against the DB.

**Root cause:** The `@Size(max = 100)` constraint is a *per-request parameter count* guard, not a rate limit. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is never wired into `rest-java`. `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter — no request-rate or concurrency controls exist for this endpoint.

**Exploit flow:**
1. Attacker opens 10 concurrent HTTP connections (no credentials needed).
2. Each sends `GET /api/v1/accounts/0.0.1/hooks?hook.id=eq:1&hook.id=eq:2&…&hook.id=eq:100&limit=100`.
3. Each request passes `@Size` validation (exactly 100 ≤ 100) and `@Max` validation (100 ≤ 100).
4. `hooksRequest()` builds a `TreeSet<Long>` of 100 IDs; `HookServiceImpl.getHooks()` calls `findByOwnerIdAndHookIdIn` with all 100 IDs.
5. 10 parallel `SELECT … WHERE hook_id IN (100 values)` queries hit the DB simultaneously.
6. Attacker repeats in a tight loop; no mechanism rejects or slows subsequent requests.

**Why existing checks fail:** `@Size(max = 100)` only prevents >100 parameters in a single request — it does not limit how many requests per second arrive or how many run concurrently. The throttle beans (`rateLimitBucket`, `gasLimitBucket`) are Spring beans scoped to the `web3` application context and are not present in `rest-java`.

### Impact Explanation
Each 100-element `IN` query is significantly more expensive than a point lookup: the DB must evaluate 100 bind parameters, potentially scan/index-merge, and return up to 100 rows. Ten concurrent such queries consume 10 DB connection pool slots simultaneously and impose proportionally higher CPU on both the JVM (parameter binding, result mapping) and the database (query planning, execution). Sustained at even modest concurrency (10–20 clients), this can push DB CPU and connection utilization 30–50%+ above idle baseline, degrading response times for all users of the mirror node.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a standard HTTP client. The request is syntactically valid and passes all input validation. It is trivially scriptable with `curl`, `ab`, `wrk`, or any load-testing tool. The attacker does not need to know valid hook IDs — arbitrary integers are accepted and simply return empty results, still executing the full DB round-trip. The attack is repeatable indefinitely with no lockout or backoff mechanism.

### Recommendation
1. **Add rate limiting to `rest-java`**: Introduce a `bucket4j`-based filter (mirroring `ThrottleManagerImpl` in `web3`) applied to all `/api/v1/**` routes, limiting requests per IP per second.
2. **Reduce `MAX_REPEATED_QUERY_PARAMETERS` for EQ filters**: Separate the cap for equality (IN-clause) filters from range filters; a value of 10–25 for EQ filters significantly reduces per-query cost.
3. **Cap concurrent DB connections per request origin**: Configure HikariCP with a per-client connection limit or use a semaphore in the service layer.
4. **Consider query cost budgeting**: Reject or queue requests whose estimated IN-clause size × limit exceeds a configurable threshold.

### Proof of Concept
```bash
# Generate 100 hook.id=eq: parameters
PARAMS=$(seq 1 100 | awk '{printf "hook.id=eq:%d&", $1}' | sed 's/&$//')

# Fire 10 concurrent requests
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1/hooks?${PARAMS}&limit=100" &
done
wait
```
