### Title
Missing JDBC Socket Timeout on Runtime HikariCP Pool Enables Thread Starvation via Silent Network Partition

### Summary
The `TopicServiceImpl.findById()` method calls `topicRepository.findById()` over a HikariCP connection pool that has no socket-level read timeout configured for runtime connections. The `socketTimeout` defined in `CommonProperties.DatabaseStartupProperties` is applied exclusively to the startup-probe `DatabaseWaiter`, not to the live pool. During a silent TCP partition (no RST, no FIN), every in-flight JDBC read blocks indefinitely, holding its HTTP worker thread until the OS-level TCP keepalive fires (hours by default), enabling an unprivileged attacker to exhaust the thread pool with concurrent requests.

### Finding Description
**Exact code path:**

`TopicController.getTopic()` → `TopicService.findById()` → `TopicServiceImpl.findById()` → `topicRepository.findById(id.getId())` (Spring Data `CrudRepository`, no timeout annotation). [1](#0-0) [2](#0-1) 

**Root cause — the `socketTimeout` is startup-only:**

`CommonProperties.DatabaseStartupProperties` defines a `socketTimeout` (default 2 s), but this is consumed exclusively by `DatabaseWaiter.waitForDatabase()`, which sets it as a JDBC property on a one-shot `DriverManager.getConnection()` call used only at startup. [1](#0-0) [3](#0-2) [4](#0-3) 

**The runtime HikariCP pool is built separately** in `CommonConfiguration.dataSource()` from `@ConfigurationProperties("spring.datasource.hikari")`. No `socketTimeout` (PostgreSQL JDBC driver property) is set programmatically on this pool, and no `application.yml` exists under `rest-java/src/main/resources/` that would supply one. [5](#0-4) 

**Why HikariCP's own `connectionTimeout` does not help:** HikariCP's `connectionTimeout` (default 30 s) governs only how long a caller waits to *acquire* a connection from the pool. Once a connection is checked out and a query is dispatched, HikariCP imposes no deadline on the blocking socket read. With no PostgreSQL-driver `socketTimeout` set, the thread blocks at `java.net.SocketInputStream.read()` until the OS TCP keepalive stack declares the connection dead — typically 2+ hours with default Linux kernel settings.

### Impact Explanation
An attacker who can send HTTP requests to `GET /api/v1/topics/{id}` (no authentication required) and who can induce or wait for a silent network partition between the mirror-node and its PostgreSQL backend (common in cloud/Kubernetes environments during node drain, network policy changes, or firewall rule updates) can hold every Tomcat/Netty worker thread indefinitely. With the default Spring Boot embedded server thread pool (~200 threads), a sustained burst of ~200 concurrent requests during the partition window is sufficient to render the entire `rest-java` service unresponsive — a complete denial of service. The same code path is replicated in `EntityServiceImpl.findById()` and `CustomFeeService`, meaning all three repository calls in `getTopic()` are affected. [6](#0-5) 

### Likelihood Explanation
No privilege is required — the endpoint is public. Silent partitions are a routine operational event in Kubernetes (pod eviction, rolling restarts of the database StatefulSet, cloud provider network blips). An external attacker cannot directly cause the partition but can pre-position many long-lived requests and wait; an internal attacker or a misconfigured network policy can trigger it on demand. The condition is repeatable: every new request during the partition window adds another stuck thread.

### Recommendation
1. **Set `socketTimeout` on the HikariCP pool's data-source properties.** For the PostgreSQL JDBC driver this is the `socketTimeout` property (in seconds). Add it to `spring.datasource.hikari.data-source-properties.socketTimeout` in the rest-java configuration (e.g., 30 s).
2. **Set HikariCP `connectionTimeout` and `keepaliveTime`** to detect dead connections in the pool before they are handed to a caller.
3. **Add a query-level timeout** via `@QueryHints({@QueryHint(name = "jakarta.persistence.query.timeout", value = "30000")})` on the repository, or via `spring.jpa.properties.jakarta.persistence.query.timeout`.
4. **Apply a `@Transactional(timeout = N)` annotation** on `TopicServiceImpl.findById()` as a defence-in-depth layer.

### Proof of Concept
```
# 1. Start the rest-java service connected to PostgreSQL.
# 2. Induce a silent partition (e.g., iptables DROP on the DB port, no REJECT):
#    iptables -I OUTPUT -p tcp --dport 5432 -j DROP
#
# 3. Flood the endpoint with concurrent requests:
for i in $(seq 1 250); do
  curl -s "http://<host>/api/v1/topics/0.0.1" &
done
wait
#
# 4. Observe: all 250 threads block in topicRepository.findById().
#    The service stops responding to any new request (thread pool exhausted).
#    Threads remain stuck until OS TCP keepalive fires (~2 hours default)
#    or the process is restarted.
#
# Root cause confirmed: no socketTimeout in HikariCP pool data-source properties,
# DatabaseWaiter socketTimeout applies only to startup probe connections.
```

```java
    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
```
