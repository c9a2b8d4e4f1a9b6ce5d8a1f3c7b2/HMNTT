### Title
Connection Pool Exhaustion DoS via Concurrent `GET /api/v1/schedules` Requests

### Summary
The `getSchedules()` handler in `rest/schedules.js` issues two parallel database queries via `Promise.all([pool.queryQuietly(entityQuery, ...), pool.queryQuietly(signatureQuery, ...)])`, each holding a pool connection for the full query duration. With the default pool size of 10 connections and no per-client concurrency or rate limiting on the Node.js REST API, five concurrent unauthenticated requests reaching the `Promise.all` phase simultaneously exhaust all available connections, starving every other API endpoint of database access for up to 20 seconds per attack wave.

### Finding Description
**Exact code location:** `rest/schedules.js`, `getSchedules()`, lines 241 and 261â€“264.

```
// Line 241 â€” sequential first query (1 connection, released before Promise.all)
const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);

// Lines 261â€“264 â€” two parallel queries, each holding a connection simultaneously
const [{rows: entities}, {rows: signatures}] = await Promise.all([
  pool.queryQuietly(entityQuery, entityIds),
  pool.queryQuietly(signatureQuery, entityIds),
]);
```

**Root cause:** `pool.queryQuietly()` without a `preQueryHint` calls `this.query()` (node-postgres `Pool.prototype.query`), which internally acquires a dedicated client from the pool, executes the query, and releases it only after the query completes. During `Promise.all`, both queries run concurrently, each holding one pool connection for the full execution duration.

**Pool configuration:** `rest/dbpool.js` line 14 sets `max: config.db.pool.maxConnections`. The documented default is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`, `docs/configuration.md` line 556). The `statementTimeout` is 20,000 ms (20 seconds), meaning a connection can be held for up to 20 seconds per query.

**No rate limiting on the REST API:** The Node.js REST middleware stack (`rest/middleware/index.js`, `rest/middleware/requestHandler.js`) contains no per-IP concurrency limit, no global in-flight request cap, and no connection semaphore for the `/api/v1/schedules` endpoint. The throttle configuration found (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the separate Java web3 service, not the Node.js REST API.

**Exploit flow:**
1. Attacker sends 5+ concurrent `GET /api/v1/schedules` requests (no authentication required).
2. Each request completes its first sequential query (line 241) and enters the `Promise.all` phase.
3. With 5 requests simultaneously in `Promise.all`, 10 connections (5 Ă— 2) are held concurrently â€” the pool is exhausted.
4. Any subsequent request to any endpoint (transactions, accounts, blocks, etc.) that calls `pool.queryQuietly()` blocks waiting for a free connection.
5. Waiting requests time out after `connectionTimeout` (default 20,000 ms), returning errors to legitimate users.
6. The attacker repeats the burst to maintain continuous exhaustion.

**Why existing checks fail:**
- `statementTimeout` (20 s) limits individual query duration but does not prevent simultaneous connection holding.
- `connectionTimeout` (20 s) causes waiting requests to fail with an error rather than hang, but this is the DoS effect itself â€” legitimate requests are denied service.
- The `authHandler` middleware only enforces response-size limits for authenticated users; it does not limit concurrency or connection usage per client.

### Impact Explanation
All API endpoints sharing the same `global.pool` instance are affected. During exhaustion, every call to `pool.queryQuietly()` across all routes blocks until a connection is freed or `connectionTimeout` elapses. This results in HTTP 500 errors (wrapped `DbError`) for all concurrent legitimate users. The attack is repeatable in a tight loop, enabling sustained denial of service against the entire REST API database layer with only 5 concurrent HTTP requests.

### Likelihood Explanation
The endpoint requires no authentication, no API key, and no special headers. The attack requires only an HTTP client capable of issuing 5 concurrent GET requests â€” trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The attacker does not need to know any valid schedule IDs; a bare `GET /api/v1/schedules` suffices. The attack is repeatable indefinitely and requires no sustained bandwidth, making it a low-cost, high-impact operation.

### Recommendation
1. **Increase default pool size** or document that `maxConnections` must be tuned relative to expected concurrency.
2. **Add a per-endpoint or global in-flight request concurrency limiter** in the Node.js middleware (e.g., using `express-rate-limit` or a semaphore) to cap simultaneous requests that reach the DB layer.
3. **Reduce `statementTimeout`** for the schedules queries to limit the window during which connections are held.
4. **Refactor `getSchedules()`** to avoid holding two connections simultaneously â€” e.g., use a single SQL query joining `entity` and `transaction_signature` instead of two parallel queries, or serialize the two queries sequentially.
5. **Deploy pgbouncer** in front of the Node.js pool (as done in the production Helm chart for `mirror_rest` with `max_user_connections: 250`) and ensure this is enforced in all deployment configurations, not just production.

```bash
  curl -s "http://<mirror-node-host>/api/v1/schedules" &

# Simultaneously, observe that other endpoints time out or return 500:

```
