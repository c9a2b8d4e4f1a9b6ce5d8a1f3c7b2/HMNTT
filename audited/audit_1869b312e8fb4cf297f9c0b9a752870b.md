### Title
Missing Timestamp Bound in `getLatestFileDataContents` Allows Post-Transaction File Mutation to Corrupt Historical Call Data in API Response

### Summary
`getContractResultsByTimestamp` extracts a `timestamp` from the request path at line 1010 but never passes it as a filter to `FileDataService.getLatestFileDataContents` at line 1023. The call uses `{whereQuery: []}`, which causes the underlying SQL to retrieve the **current latest** state of the referenced file rather than its state at the transaction's consensus timestamp. Any unprivileged user who owns the referenced file can update it after their transaction is confirmed, causing the mirror node API to return fabricated call data for that historical transaction.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js`, `getContractResultsByTimestamp` (lines 1002–1042):

```js
// line 1010 — timestamp is extracted from the path param
const {contractId, timestamp} = await getAndValidateContractIdAndConsensusTimestampPathParams(req);
...
// line 1022-1024 — timestamp is NEVER forwarded to the file data query
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
``` [1](#0-0) 

**Root cause — the SQL query has no upper timestamp bound:**

`FileDataService.latestFileContentsQuery` (lines 18–41 of `rest/service/fileDataService.js`) works by first finding `max(consensus_timestamp)` of the most recent `FileCreate`/`FileUpdate` (transaction types 17/19) for the given file ID, then aggregating all subsequent file chunks. With `{whereQuery: []}`, neither the inner CTE nor the outer aggregation has an upper bound on `consensus_timestamp`, so the query always resolves to the **current** latest version of the file. [2](#0-1) 

**Contrast with the correct pattern:**

`FileDataService.getFileData` (lines 70–75) is the timestamp-aware variant. It accepts an explicit `timestamp` argument and enforces `consensus_timestamp <= $2` in both the inner subquery and the outer filter, correctly bounding the result to the file's state at the moment of the transaction. [3](#0-2) 

**Failed assumption:** The code assumes the file referenced by `callDataId` is immutable after the transaction, or that "latest" equals "at-transaction-time." Neither is guaranteed on Hedera; files are mutable by their key holders.

**Same defect is present in `getContractResultsByTransactionIdOrHash` at line 1184**, which also calls `getLatestFileDataContents` with `{whereQuery: []}`. [4](#0-3) 

### Impact Explanation
The `call_data` field in the API response for `/contracts/{contractId}/results/{timestamp}` and `/contracts/results/{transactionIdOrHash}` will reflect the **post-mutation** file content rather than the bytes that were actually submitted in the transaction. This breaks the historical integrity guarantee of the mirror node: a verifier, auditor, or dApp relying on the API to reconstruct what was called in a past Ethereum transaction will receive falsified input data. An attacker can make it appear that a different function selector or different ABI-encoded arguments were used, enabling retroactive misrepresentation of on-chain activity without any privileged access.

### Likelihood Explanation
Preconditions are minimal and require no special privilege:
1. The attacker must hold the key to a Hedera file — achievable by any account via a standard `FileCreate` transaction.
2. The attacker must submit an Ethereum transaction that references that file as `callDataId` — a normal user operation.
3. After the transaction reaches consensus, the attacker issues a `FileUpdate` or `FileAppend` to overwrite the file content — again a normal user operation requiring only the file key.

The attack is fully repeatable, costs only normal Hedera transaction fees, and requires no knowledge of internal mirror node state. Any user who has ever submitted an Ethereum transaction with a file-based call data payload can perform this retroactively.

### Recommendation
Pass the transaction's consensus timestamp as an upper-bound filter when calling `getLatestFileDataContents`. Replace the empty `{whereQuery: []}` with a bounded query in both affected call sites:

```js
// getContractResultsByTimestamp — line 1023
fileData = await FileDataService.getLatestFileDataContents(
  ethTransaction.callDataId,
  {whereQuery: [{query: `${FileData.CONSENSUS_TIMESTAMP} <= `, param: timestamp}]}
);

// getContractResultsByTransactionIdOrHash — line 1184
// use contractResult.consensusTimestamp (already available) as the bound
fileData = await FileDataService.getLatestFileDataContents(
  ethTransaction.callDataId,
  {whereQuery: [{query: `${FileData.CONSENSUS_TIMESTAMP} <= `, param: contractResult.consensusTimestamp}]}
);
```

Alternatively, use the already-correct `FileDataService.getFileData(fileId, timestamp)` which enforces the timestamp bound natively. [5](#0-4) 

### Proof of Concept
1. Create a Hedera account (no special role needed).
2. Submit `FileCreate` with content `0xAABBCCDD` → receive `fileId = 0.0.X`.
3. Submit an Ethereum transaction with `callDataId = 0.0.X`; note the resulting `consensusTimestamp = T`.
4. Submit `FileUpdate` on `0.0.X` with new content `0x11223344`.
5. `GET /api/v1/contracts/{contractId}/results/{T}` — the response `call_data` field returns `0x11223344` (the post-update content), not `0xAABBCCDD` (the actual submitted call data).
6. The on-chain record still contains `0xAABBCCDD`; the mirror node API is now inconsistent with the ledger.
