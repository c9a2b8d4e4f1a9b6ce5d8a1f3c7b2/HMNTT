### Title
Unbounded Cache Eviction DoS via Unauthenticated Token Relationship Endpoint

### Summary
The `tokenCache` in `rest/service/tokenService.js` is a `quickLru` instance with a fixed `maxSize` of 100,000 and no TTL. An unauthenticated attacker can continuously query the public `/accounts/:id/tokens` endpoint with accounts associated to distinct token IDs, filling and then continuously evicting the LRU cache. This forces `getCachedTokens()` to issue a DB query on every request, exhausting the small DB connection pool (default `maxConnections: 10`) and degrading or halting the REST API service.

### Finding Description

**Exact code path:**

`tokenCache` is initialized with no TTL at module scope: [1](#0-0) 

`getCachedTokens()` unconditionally calls `tokenCache.set()` for every DB-fetched token, with no guard against re-insertion of a just-evicted entry: [2](#0-1) 

`putTokenCache()` has a `has()` guard that prevents overwriting existing entries, but this does not prevent LRU eviction of those entries when new ones are inserted: [3](#0-2) 

`getCachedTokens()` is called on every request to the public, unauthenticated `/accounts/:id/tokens` endpoint: [4](#0-3) 

The REST API server applies no rate-limiting middleware to this endpoint: [5](#0-4) 

The DB connection pool is capped at 10 connections by default: [6](#0-5) 

**Root cause:** `tokenCache` has no TTL and a bounded size of 100,000. Once full, every insertion of a new token ID silently evicts the LRU entry. `getCachedTokens()` always re-queries the DB for evicted IDs and re-inserts them, creating a perpetual eviction cycle under attacker-controlled load. The `putTokenCache()` guard (`has()` check) only prevents duplicate writes; it does not prevent eviction-driven cache misses.

**Exploit flow:**
1. Attacker enumerates accounts on the public Hedera ledger that collectively hold associations to >100,000 distinct token IDs (Hedera mainnet has millions of tokens; this is trivially achievable with public data).
2. Attacker sends a burst of requests to `GET /api/v1/accounts/{accountId}/tokens` cycling through these accounts.
3. `getCachedTokens()` fills `tokenCache` to `maxSize` (100,000).
4. Attacker continues sending requests with accounts associated to new, unseen token IDs. Each new token ID evicts an LRU entry.
5. Subsequent legitimate requests for the evicted token IDs miss the cache and each trigger `SELECT … FROM token WHERE token_id = any($1)` against the DB.
6. With the DB pool at 10 connections and no rate limiting, the pool saturates, causing connection timeouts (default 20,000 ms) and request failures across all REST API consumers.

### Impact Explanation
Every cache miss in `getCachedTokens()` issues a synchronous DB query. With the default pool of 10 connections and no rate limiting on the REST API, a sustained eviction attack saturates the pool, causing all concurrent requests that need DB access to queue or time out. This effectively takes the REST API offline for legitimate users. The mirror node REST API is the primary read interface for the Hedera network; its unavailability degrades ecosystem tooling, wallets, and explorers that depend on it.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no brute force. All required inputs (account IDs, token associations) are public on-chain data. The attacker needs only a standard HTTP client and a list of account IDs with diverse token associations, both freely available. The attack is repeatable and can be sustained indefinitely from a single machine given the absence of rate limiting on the REST API.

### Recommendation
1. **Add a TTL to `tokenCache`**: Pass `maxAge` to the `quickLru` constructor so entries expire and the cache self-heals after eviction pressure stops.
2. **Rate-limit the REST API**: Add a per-IP or global rate-limiting middleware (e.g., `express-rate-limit`) to the Node.js server for token-relationship endpoints.
3. **Increase DB pool size** or add a connection queue limit with fast-fail to prevent pool exhaustion from cascading.
4. **Cap the number of token IDs per `getCachedTokens()` call** to bound the per-request DB query cost.

### Proof of Concept
```bash
# 1. Collect >100,000 accounts with distinct token associations from public Hedera mirror data
# 2. Fill the cache
for i in $(seq 1 100000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.$i/tokens" &
done
wait

# 3. Continuously evict with new token IDs (accounts with unseen token associations)
while true; do
  for i in $(seq 100001 200000); do
    curl -s "https://<mirror-node>/api/v1/accounts/0.0.$i/tokens" &
  done
  wait
done
# Result: getCachedTokens() issues a DB query on every request;
# DB pool (maxConnections=10) saturates; REST API becomes unresponsive.
```
