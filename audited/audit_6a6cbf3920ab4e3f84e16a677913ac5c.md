### Title
Unbounded Account Alias Length Bypasses Database Index, Enabling Sequential Scan DoS via `getNftsByAccountId()`

### Summary
The `AccountAlias.isValid()` regex imposes no upper bound on the alias string length, allowing an unauthenticated attacker to supply arbitrarily long base32 alias strings that pass validation. When decoded and passed to the `entity` table query in `getAccountFromAlias()`, values exceeding PostgreSQL's btree index key size limit (~2704 bytes for 8KB pages) force a sequential scan on the full `entity` table. Flooding the `/accounts/:idOrAliasOrEvmAddress/nfts` endpoint with such requests exhausts database worker threads and causes sustained service degradation.

### Finding Description
**Code path:**

1. `getNftsByAccountId()` at [1](#0-0)  calls `EntityService.getEncodedId(req.params[...])` with the raw URL path parameter — no length check.

2. `getEncodedId()` at [2](#0-1)  calls `AccountAlias.isValid(entityIdString)`. The validation regex is:

   ```js
   const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;
   ``` [3](#0-2) 

   The `[A-Z2-7]+` quantifier is **unbounded** — a string of 10,000 uppercase base32 characters passes this check.

3. `AccountAlias.fromString()` calls `base32.decode(base32Alias)` and stores the decoded binary as `this.alias`. [4](#0-3) 

4. `getAccountFromAlias()` executes: [5](#0-4) 

   ```sql
   select id from entity
   where coalesce(deleted, false) <> true
     and alias = $1
   ```

   with the decoded binary blob as `$1`. [6](#0-5) 

**Root cause:** PostgreSQL btree indexes have a maximum key size of ~2704 bytes (for 8KB pages). A base32 string of ~4,400 characters decodes to ~2,750 bytes, exceeding this limit. PostgreSQL cannot use the btree index on the `alias` column for such a value and falls back to a **full sequential scan** of the `entity` table. In a production Hedera mirror node, the `entity` table contains millions of rows, making each such scan expensive.

**Failed assumption:** The code assumes `isValid()` is a sufficient gate before issuing the DB query. It only validates the character set, not the length.

### Impact Explanation
Each crafted request triggers a full sequential scan on the `entity` table. With a sufficiently large table (millions of entities in production), each scan holds a DB worker thread for seconds. Flooding the endpoint with concurrent requests (e.g., 50–100 parallel connections) exhausts the database connection pool, causing all queries — including transaction ingestion and other API endpoints — to queue or time out. This maps directly to the stated critical scope: **total network inability to confirm new transactions** if the mirror node's DB layer is saturated.

### Likelihood Explanation
The attack requires no authentication, no special account, and no prior knowledge of the system state. The endpoint is publicly accessible. The only precondition is knowing the API path (`/api/v1/accounts/<alias>/nfts`), which is documented. The crafted alias is trivially constructed (e.g., `AAAA...` repeated 5,000 times). The attack is fully repeatable and automatable with standard HTTP tooling (e.g., `ab`, `wrk`, `curl` in a loop).

### Recommendation
1. **Enforce a maximum alias length** in `AccountAlias.isValid()` before any decoding or DB access:
   ```js
   // Hedera public keys are at most 88 base32 chars (44 bytes); add margin
   const MAX_ALIAS_LENGTH = 128;
   const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;

   static isValid(accountAlias, noShardRealm = false) {
     if (typeof accountAlias !== 'string' || accountAlias.length > MAX_ALIAS_LENGTH) return false;
     const regex = noShardRealm ? noShardRealmAccountAliasRegex : accountAliasRegex;
     return regex.test(accountAlias);
   }
   ``` [7](#0-6) 

2. **Alternatively**, enforce the length limit directly in the regex: `/^(\d{1,5}\.){0,2}[A-Z2-7]{1,128}$/`.

3. **Add rate limiting** on the `/accounts/:id/nfts` endpoint at the API gateway or middleware layer as a defense-in-depth measure.

### Proof of Concept
```bash
# Generate a 5000-character valid base32 alias (all uppercase A, valid per regex)
ALIAS=$(python3 -c "print('A' * 5000)")

# Flood the endpoint with concurrent requests
for i in $(seq 1 100); do
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/nfts" &
done
wait
```

Each request passes `AccountAlias.isValid()`, decodes to ~3,125 bytes of binary (exceeding PostgreSQL's btree index key limit), and forces a sequential scan on the `entity` table. With 100 concurrent requests, the database connection pool is exhausted, blocking all other queries.
