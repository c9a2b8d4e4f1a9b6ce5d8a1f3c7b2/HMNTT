### Title
Cache Pollution via Whitespace-Varied GraphQL Queries (No Query Normalization Before Caching)

### Summary
`CachedPreparsedDocumentProvider` uses the raw query string as the cache key with no normalization. An unprivileged attacker can flood the bounded Caffeine cache (max 1000 entries) with whitespace-varied permutations of the same semantic query, evicting legitimate cached entries and forcing full parse+validate overhead on every subsequent legitimate request.

### Finding Description

**Exact code path:**

In `CachedPreparsedDocumentProvider.java` line 26, the cache key is `executionInput.getQuery()` — the raw, unnormalized query string: [1](#0-0) 

The cache is a Caffeine `AsyncCache<String, PreparsedDocumentEntry>` configured with `maximumSize=1000`: [2](#0-1) 

**Root cause:** There is no normalization (trimming, whitespace collapsing, or AST-fingerprinting) applied to the query string before it is used as a cache key. The strings `{ account { id } }` and `{  account  {  id  } }` are semantically identical but produce two distinct cache entries.

**Why existing checks are insufficient:**

`GraphQlConfiguration` sets `maxWhitespaceTokens(1000)` and `maxCharacters(10000)`: [3](#0-2) 

These limits bound the *size* of each individual query but do nothing to prevent an attacker from generating thousands of distinct whitespace permutations of the same query. With up to 1000 whitespace tokens allowed per query and a cache of only 1000 slots, an attacker can continuously cycle the entire cache with near-duplicate entries.

**Exploit flow:**
1. Attacker crafts a valid GraphQL query (e.g., `{ account { id } }`).
2. Attacker generates 1000+ whitespace permutations (extra spaces, tabs, newlines between tokens) — all within the `maxWhitespaceTokens=1000` and `maxCharacters=10000` limits.
3. Each permutation is a cache miss, triggers a full parse+validate cycle, and occupies one cache slot.
4. Once the cache is full (1000 entries), Caffeine evicts the least-recently-used entries — which are the legitimate application queries.
5. Attacker continuously sends new permutations to keep the cache saturated with near-duplicates.
6. All legitimate queries now miss the cache on every request and incur full parse+validate overhead indefinitely.

### Impact Explanation
With the cache hit rate for legitimate queries driven to ~0%, every production request must re-parse and re-validate its query. The parse+validate path in graphql-java is CPU-intensive (AST construction, type checking, rule validation). Under sustained attack, this multiplies CPU load on all GraphQL nodes proportionally to request rate, degrading throughput and increasing latency. Since no authentication is required to submit GraphQL queries, this is reachable by any external user.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the GraphQL endpoint URL, and no brute force — only trivially generated whitespace variations of any valid query. It is fully automatable with a simple script. The attacker does not need to know the cache internals; any whitespace variation that passes the parser limits will work. The attack is repeatable and sustainable indefinitely.

### Recommendation
Normalize the query string before using it as a cache key. Options in increasing robustness:
1. **Whitespace normalization**: Collapse all whitespace sequences to a single space and trim before caching — `key = query.replaceAll("\\s+", " ").trim()`.
2. **AST fingerprinting**: Hash the parsed AST (post-parse, pre-validate) rather than the raw string, so all semantically equivalent queries share one cache slot.
3. **Rate limiting per IP**: Limit the number of distinct cache-miss queries accepted per IP per time window to prevent cache flooding.

Option 2 is the most robust as it also handles comment variations and field-order differences.

### Proof of Concept
```python
import requests, itertools

ENDPOINT = "http://<graphql-host>/graphql"
BASE_QUERY = "{ account { id } }"

# Generate 1100 whitespace variants (all valid within maxWhitespaceTokens=1000)
def variants():
    for n in range(1, 1101):
        spaces = " " * n
        yield f"{{{spaces}account{spaces}{{{spaces}id{spaces}}}{spaces}}}"

for q in variants():
    requests.post(ENDPOINT, json={"query": q})
    # Each request is a cache miss, occupies one slot, evicts a legitimate entry

# Now send a legitimate query — it will be a cache miss every time
while True:
    requests.post(ENDPOINT, json={"query": BASE_QUERY})
    # Forces full parse+validate on every call — cache is always full of attacker entries
```
