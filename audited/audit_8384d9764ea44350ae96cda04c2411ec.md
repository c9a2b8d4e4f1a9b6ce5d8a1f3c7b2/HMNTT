### Title
`@Cacheable` `unless` Guard Ineffective for `Optional.empty()` — Cache Pollution via Non-Existent Contract IDs

### Summary
`findRuntimeBytecode()` in `ContractRepository` declares `unless = "#result == null"` to prevent caching of missing-contract results, but the method returns `Optional<byte[]>`. When no contract row exists, Spring Data JPA returns `Optional.empty()`, which is a non-null object. The `unless` predicate therefore evaluates to `false` and the empty Optional is stored in the cache. An unprivileged attacker can flood the 1,000-slot contract bytecode cache with empty entries by issuing `eth_call` requests targeting non-existent contract IDs, evicting all legitimate cached bytecodes and forcing repeated DB hits for every subsequent real contract call.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** The `unless` SpEL expression `#result == null` tests the `Optional` wrapper object for nullity, not the absence of a value. `Optional.empty()` is never `null`; it is a singleton non-null object. Spring's `@Cacheable` therefore caches every "not found" lookup under its `contractId` key.

**Call chain:**
1. `ContractBytecodeReadableKVState.readFromDataSource()` (line 42–46) calls `contractRepository.findRuntimeBytecode(entityId.getId())`.
2. For any `contractId` with no matching row, the native query returns zero rows; Spring Data wraps this as `Optional.empty()`.
3. `unless = "#result == null"` → `Optional.empty() == null` → `false` → result **is cached**.
4. The cache entry persists for up to 1 hour (`expireAfterAccess=1h`) with no `@CacheEvict` anywhere in the codebase.

**Cache configuration** (`CacheProperties.java`, line 22):
```
contract = "expireAfterAccess=1h,maximumSize=1000,recordStats"
```

**No eviction path exists:** a `grep` across the entire repository finds zero `@CacheEvict` annotations targeting the contract cache.

### Impact Explanation

The contract bytecode cache has a hard cap of 1,000 entries. An attacker who sends 1,000 `eth_call` requests each targeting a distinct non-existent contract address fills the entire cache with `Optional.empty()` entries. Caffeine's LRU eviction then removes all legitimate cached bytecodes. For the following hour, every `eth_call` to a real contract must execute a synchronous DB query to fetch bytecode, increasing database load and response latency proportionally to traffic. The attack is repeatable: as legitimate entries re-populate the cache, the attacker can re-flood it. There is no economic damage to end users (calls still succeed), but the infrastructure is degraded — consistent with the declared Medium / griefing scope.

### Likelihood Explanation

No authentication or rate-limiting specific to contract-ID diversity is enforced at the `eth_call` endpoint. The `ContractCallRequest` only validates that the `to` field is a 40-hex-character address (`@Hex(minLength=40, maxLength=40)`); any 20-byte value is accepted. A single attacker script generating 1,000 random or sequential long-zero EVM addresses (e.g., `0x0000…0001` through `0x0000…03E8`) is sufficient to saturate the cache. The gas throttle (`gasPerSecond=1,500,000,000`) limits throughput but does not prevent the attack — each call can use the minimum gas (21,000), allowing ~71,000 calls/second before throttling, far more than the 1,000 needed to fill the cache.

### Recommendation

Change the `unless` condition to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

Or equivalently using the `#result?.isEmpty()` form:

```java
unless = "#result == null || #result.isEmpty()"
```

This mirrors the correct pattern already used in `ContractStateRepository.findStorage()`, which should also be audited for the same issue.

### Proof of Concept

```bash
# Fill the 1000-slot cache with empty Optional entries
for i in $(seq 1 1000); do
  ADDR=$(printf "0x%040x" $i)   # long-zero address, no contract exists
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$ADDR\",\"gas\":21000,\"data\":\"0x\"}" &
done
wait

# Verify: subsequent calls to a REAL contract now miss the cache and hit the DB.
# Monitor DB query rate — it will spike for bytecode lookups on every eth_call.
```

**Preconditions:** No account, no privileges. Only public `eth_call` access required.
**Trigger:** 1,000 requests with distinct non-existent `to` addresses.
