### Title
Unauthenticated DoS via Unbounded Request Rate on GET /api/v1/blocks — No Rate Limiting in REST Middleware Stack

### Summary
The `GET /api/v1/blocks` endpoint in the Hiero Mirror Node REST service has no per-IP or global request rate limiting. Any unauthenticated external caller can flood the endpoint at arbitrary request rates, each triggering a full async PostgreSQL query against the `record_file` table. Because the Node.js DB connection pool is finite and no request-shedding mechanism exists, sustained flooding exhausts the pool and saturates the event loop's pending-callback queue, rendering the REST node unresponsive to all other clients.

**Root cause — the complete middleware stack in `rest/server.js` contains no rate-limiting layer:**
```
express.urlencoded / express.json / cors
httpContext.middleware
requestLogger          ← logging only
authHandler            ← Basic Auth for custom row-limits only, no throttle
metricsHandler         ← optional, metrics only
responseCacheCheckHandler ← optional cache read
[routes]
responseHandler / handleError
``` [4](#0-3) 

No `express-rate-limit`, no token-bucket, no concurrency limiter, and no IP-based throttle is present anywhere under `rest/middleware/`:


**Why the existing `maxLimit` cap is insufficient:**

`extractLimitFromFilters` caps the number of *rows returned per query* at the configured maximum, but places no constraint on *how many queries per second* a single IP may issue:
```js
extractLimitFromFilters = (filters) => {
  const limit = findLast(filters, {key: filterKeys.LIMIT});
  const maxLimit = getEffectiveMaxLimit();
  return limit ? (limit.value > maxLimit ? defaultLimit : limit.value) : defaultLimit;
};
``` [5](#0-4) 

The `authHandler` only sets a per-user row-limit for authenticated users; unauthenticated requests pass through with no restriction: [6](#0-5) 

The optional Traefik middleware chart (`charts/hedera-mirror-rest/templates/middleware.yaml`) is gated on `{{ if and .Values.global.middleware .Values.middleware }}` and is not guaranteed to be deployed, leaving the application layer unprotected in default or bare-metal deployments: [7](#0-6) 

### Impact Explanation
Each request at maximum `limit` causes a full sequential scan with `ORDER BY consensus_end` on the `record_file` table. At 50 req/s, 50 concurrent PostgreSQL queries compete for the finite connection pool (`db.pool.maxConnections`). Once the pool is exhausted, new requests queue as pending Promises in the Node.js event loop. The single-threaded event loop accumulates unresolved callbacks, response latency climbs, and the node stops processing legitimate requests — effectively a complete REST node outage. Because mirror node deployments typically run a small number of REST replicas, saturating even one node degrades availability for a significant fraction of API consumers.

### Likelihood Explanation
No authentication, no CAPTCHA, no API key, and no network-layer rate limit (in default deployments) is required. The attack requires only `curl` or any HTTP load tool. It is repeatable indefinitely from a single IP or trivially distributed across a botnet. The endpoint is publicly documented in the OpenAPI spec and is a standard, well-known path (`/api/v1/blocks`). [8](#0-7) 

### Recommendation
1. **Application layer**: Add `express-rate-limit` (or equivalent) as a global middleware in `rest/server.js` before the route registrations, keyed on `req.ip`, with a configurable window and max-requests ceiling.
2. **Concurrency cap**: Add a per-IP or global in-flight request counter that returns HTTP 429 when exceeded, preventing pool exhaustion independent of wall-clock rate.
3. **DB pool protection**: Configure `db.pool.maxConnections` and `db.pool.connectionTimeout` conservatively, and add a statement timeout so runaway queries are killed rather than held indefinitely.
4. **Infrastructure**: Ensure the Traefik rate-limit middleware (`charts/hedera-mirror-rest/templates/middleware.yaml`) is enabled by default rather than opt-in.

### Proof of Concept
```bash
# No credentials required. Run from any host.
# Replace HOST with the mirror node REST endpoint.

ab -n 5000 -c 50 \
  "https://HOST/api/v1/blocks?limit=100&order=asc"

# Or with wrk:
wrk -t 10 -c 50 -d 30s \
  "https://HOST/api/v1/blocks?limit=100"

# Expected result within ~10-30 seconds:
# - Response latency climbs from <100ms to multi-second timeouts
# - Legitimate requests to /api/v1/transactions, /api/v1/accounts, etc.
#   begin returning 503 or timing out
# - Node.js process memory grows as pending Promise callbacks accumulate
# - DB connection pool log shows "all connections in use" errors
```

  if (!credentials) {

  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({

        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
