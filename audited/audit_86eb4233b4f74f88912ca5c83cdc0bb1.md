### Title
Address Book Files Lack Timestamp Normalization, Rendering Cache Ineffective and Enabling Per-Request DB Amplification

### Summary
In `SystemFileLoader.load(FileID, long)`, exchange-rate and fee-schedule files have their consensus timestamp rounded down to the hour before being used as a cache key, but `addressBookFile101` and `addressBookFile102` receive no such normalization. Because `FileReadableKVState` always passes the current wall-clock nanosecond as the timestamp for system-file lookups, every request produces a unique cache key for address-book files, making the `defaultSystemFileCacheManager` permanently ineffective for those files. Any unprivileged caller sending requests that touch the address book therefore triggers a full DB query plus a `NodeAddressBook.PROTOBUF` parse on every single call.

### Finding Description

**Code path:**

`FileReadableKVState.readFromDataSource()` ( [1](#0-0) ) always calls `systemFileLoader.load(key, currentTimestamp)` for system files, where `currentTimestamp` is the current wall-clock time in nanoseconds.

Inside `SystemFileLoader.load(FileID, long)` ( [2](#0-1) ), the timestamp-rounding branch at lines 111–116 only fires for `exchangeRateFileId`, `feeSchedulesFileId`, and `simpleFeeSchedulesFileId`:

```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp);   // ← normalization
}
``` [3](#0-2) 

`addressBookFile101` and `addressBookFile102` fall through without rounding. The cache key is then built directly from the raw nanosecond timestamp:

```java
final var cacheKey = new CacheKey(fileId, consensusTimestamp);
``` [4](#0-3) 

Because wall-clock nanoseconds advance with every request, every lookup produces a previously-unseen key, so `cache.get(cacheKey, File.class)` always returns `null` and execution falls through to `loadFromDB()` → `loadWithRetry()`:

```java
fileDataRepository.getFileAtTimestamp(fileId, nanoSeconds.get())
    ...
    codec.parse(bytes.toReadableSequentialData());   // NodeAddressBook.PROTOBUF
``` [5](#0-4) 

The `defaultSystemFileCacheManager` is configured with `maximumSize=20` ( [6](#0-5) ), which prevents unbounded memory growth but does nothing to prevent the per-request DB + parse cost, since the cache is never hit.

**Root cause:** The assumption that the cache will absorb repeated address-book lookups is false. The missing `roundDownToHour` (or equivalent normalization) for address-book file IDs means the cache key space is effectively infinite.

### Impact Explanation

Every HTTP request that causes the EVM to read `addressBookFile101` or `addressBookFile102` unconditionally issues one SQL query (`getFileAtTimestamp`) and one protobuf deserialization of the full node address book. At the documented default rate limit of 500 RPS ( [7](#0-6) ), a sustained flood of such requests produces up to 500 DB queries/second and 500 protobuf parses/second for address-book data alone, amplifying both DB connection pool pressure and CPU load with no caching relief. The `statementTimeout=3000 ms` and `maximumSize=20` provide no mitigation against this throughput.

### Likelihood Explanation

No privilege is required. Any caller able to submit `eth_call` or equivalent JSON-RPC requests can trigger address-book reads. The exploit requires no special knowledge beyond knowing that the mirror-node exposes a public JSON-RPC endpoint. It is trivially repeatable and scriptable, and the rate limit (500 RPS) is the only throttle standing between the attacker and sustained DB amplification.

### Recommendation

Apply the same hour-granularity (or coarser) timestamp normalization to address-book file IDs that is already applied to exchange-rate and fee-schedule files:

```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)
        || fileId.equals(addressBook101FileId)      // add
        || fileId.equals(addressBook102FileId)) {   // add
    consensusTimestamp = roundDownToHour(consensusTimestamp);
}
```

Alternatively, since address books change far less frequently than exchange rates, a longer normalization window (e.g., round to day) would be appropriate. The `exchangeRatesCacheManager` vs `defaultSystemFileCacheManager` split can be retained; the critical fix is the missing normalization step.

### Proof of Concept

1. Deploy or connect to a mirror-node web3 instance with a populated `file_data` table containing entries for file IDs 101 and 102.
2. Send a continuous stream of `eth_call` requests that cause the EVM to read the address book (e.g., any call that internally resolves node addresses):
   ```bash
   for i in $(seq 1 10000); do
     curl -s -X POST http://<mirror-node>:8545 \
       -H 'Content-Type: application/json' \
       -d '{"jsonrpc":"2.0","method":"eth_call","params":[{...},"latest"],"id":'$i'}' &
   done
   ```
3. Monitor the DB with `pg_stat_activity` or slow-query logging. Observe that `getFileAtTimestamp` for entity IDs 101 and 102 is executed on every request with no cache hits, confirming the cache is permanently bypassed for address-book files.

    public @Nullable File load(FileID fileId, long consensusTimestamp) {
        // Skip database for network properties so that CN props can't override MN props and cause us to break.
        if (genesisNetworkProperties.fileId().equals(fileId)) {
            return genesisNetworkProperties;
        }

        var cacheManager = defaultSystemFileCacheManager;

        if (fileId.equals(exchangeRateFileId)
                || fileId.equals(feeSchedulesFileId)
                || fileId.equals(simpleFeeSchedulesFileId)) {
            cacheManager = exchangeRatesCacheManager;
            consensusTimestamp = roundDownToHour(consensusTimestamp);
        }

        final var cacheKey = new CacheKey(fileId, consensusTimestamp);
        log.debug("Looking up {}", cacheKey);
        final var cache = cacheManager.getCache(CACHE_NAME);

        if (cache == null) {
            return loadFromDB(fileId, consensusTimestamp);
        }

        // Try to return the value from the cache
        var file = cache.get(cacheKey, File.class);
        if (file != null) {
            return file;
        }

        // The value was not in cache -> try to load from DB
        var result = loadFromDB(fileId, consensusTimestamp);
        if (result != null) {
            log.info("Updating cache for key {}", cacheKey);
            cache.put(cacheKey, result);
        }

        return result;
    }
