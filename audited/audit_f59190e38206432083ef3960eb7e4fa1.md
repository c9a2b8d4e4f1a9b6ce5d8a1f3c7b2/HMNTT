### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded EVM Address Lookups in `getEncodedId()`

### Summary
Any unauthenticated caller can submit syntactically valid EVM address strings (40-char hex) to any endpoint that invokes `getEncodedId()`. Each such input unconditionally triggers a live database query via `getEntityIdFromEvmAddress()`. Because the DB-lookup result is never cached and 404 responses are excluded from the response cache, an attacker flooding the service with unique EVM addresses at high concurrency will exhaust the database connection pool, rendering the service unresponsive.

### Finding Description

**Code path:**

`rest/service/entityService.js` — `getEncodedId()` (lines 118–137):

```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()
    : await this.getEntityIdFromEvmAddress(entityId, requireResult); // ← DB hit
}
``` [1](#0-0) 

`getEntityIdFromEvmAddress()` (lines 90–104) issues a live SQL query on every call:

```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
  [Buffer.from(entityId.evmAddress, 'hex')]);
``` [2](#0-1) 

**Root cause — three compounding gaps:**

1. **`parseCached` LRU cache does not cover DB lookups.** The `quickLru` cache in `entityId.js` (lines 301–333) stores only the parsed `EntityId` object. `getEntityIdFromEvmAddress()` is called after the cache lookup and is never short-circuited by it. [3](#0-2) 

2. **Response cache excludes 404s.** `responseCacheUpdateHandler` only caches responses where `httpStatusCodes.isSuccess(res.statusCode)` is true (line 95). A non-existent EVM address returns `NotFoundError` → HTTP 404, which is never cached, so every repeat request re-hits the DB. [4](#0-3) 

3. **No application-level rate limiting.** The middleware stack (`requestHandler.js`, `authHandler.js`, `responseCacheHandler.js`, etc.) contains no per-IP or per-endpoint rate limiting. Infrastructure-level Traefik middleware is a deployment concern, not a code-level guarantee. [5](#0-4) 

**Why `isValidEntityId` passes EVM addresses:**

```js
const isValidEntityId = (entityId, allowEvmAddress = true, ...) => {
  ...
  return allowEvmAddress && isValidEvmAddress(entityId, evmAddressType);
};
``` [6](#0-5) 

Any 40-char hex string (e.g. `deadbeef...`) passes this check, is parsed with `evmAddress !== null`, and unconditionally triggers the DB path.

### Impact Explanation
The database connection pool is bounded (`db.pool.maxConnections` in config). With enough concurrent requests each holding a pool connection waiting for a DB response, the pool is exhausted. Subsequent legitimate requests queue indefinitely or time out (`connectionTimeout`), making the REST API fully unresponsive — a complete service-level DoS. Because the mirror node REST API is the primary data-access layer for the network, this affects all consumers. [7](#0-6) 

### Likelihood Explanation
No authentication is required. The attacker needs only an HTTP client and the ability to generate valid 40-char hex strings (trivial; the address space is 2^160). The attack is fully repeatable, requires no special knowledge of the system state, and can be automated with a single script. The absence of application-level rate limiting means there is no built-in throttle to slow the attack.

### Recommendation
1. **Cache DB lookup results for EVM addresses** (both hits and misses) with a short TTL (e.g., 5–30 s) using the existing Redis/LRU infrastructure, so repeated lookups for the same address do not hit the DB.
2. **Cache 404 responses** for EVM address lookups (with a short TTL) in `responseCacheUpdateHandler`, or add a dedicated negative-result cache in `getEntityIdFromEvmAddress`.
3. **Add application-level rate limiting** (e.g., `express-rate-limit`) on endpoints that invoke `getEncodedId()`, keyed by IP or API key, independent of infrastructure-level controls.
4. **Bound concurrency** of outgoing DB queries per request or globally using a semaphore/queue, so pool exhaustion cannot be triggered by a single burst.

### Proof of Concept

```bash
# Generate 10,000 unique valid EVM addresses and flood the accounts endpoint
for i in $(seq 1 10000); do
  addr=$(openssl rand -hex 20)  # 40-char hex, valid EVM address
  curl -s "https://<mirror-node>/api/v1/accounts/0x${addr}" &
done
wait
# Result: DB connection pool exhausted; legitimate requests time out or receive 503
```

Each request passes `isValidEntityId()`, reaches `getEntityIdFromEvmAddress()`, issues a DB query, gets 0 rows, returns 404 (uncached), and the connection is held for the full query duration. At sufficient concurrency, all pool connections are occupied simultaneously.
