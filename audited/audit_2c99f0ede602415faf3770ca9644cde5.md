### Title
Global Gas Bucket Burst Depletion via Concurrent Max-Gas Requests (No Per-Client Rate Limiting)

### Summary
The `/api/v1/contracts/call` endpoint enforces gas throttling using a single global `gasLimitBucket` with no per-IP or per-user partitioning. A single unauthenticated attacker can send up to 500 concurrent requests each specifying `gas=15000000` (the maximum), consuming all 750,000 scaled tokens in the global bucket within one second. This denies service to every other user for the entire 1-second refill period and is trivially repeatable.

### Finding Description

**Code path:**

`ContractController.call()` (line 40) delegates to `ThrottleManagerImpl.throttle()`:

```
// ThrottleManagerImpl.java:37-42
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Both buckets are application-wide singletons injected as Spring beans:

- `rateLimitBucket` — capacity 500 tokens, refills 500/second, no `SynchronizationStrategy` (defaults to `LOCK_FREE`). [1](#0-0) 
- `gasLimitBucket` — capacity `scaleGas(7_500_000_000) = 750_000` tokens, refills 750,000/second, uses `SynchronizationStrategy.SYNCHRONIZED`. [2](#0-1) 

**Token arithmetic:**

- `GAS_SCALE_FACTOR = 10_000` [3](#0-2) 
- `scaleGas(15_000_000) = 15_000_000 / 10_000 = 1_500` tokens per max-gas request [4](#0-3) 
- Gas bucket capacity = `scaleGas(7_500_000_000) = 750_000` tokens [5](#0-4) 
- Requests to deplete: `750_000 / 1_500 = 500` — exactly equal to `requestsPerSecond = 500` [6](#0-5) 
- `maxGasLimit = 15_000_000` [7](#0-6) 

**Root cause — failed assumption:**

The design assumes the global 500 req/s rate limit prevents any single client from depleting the gas bucket. This assumption fails because:

1. The rate limiter is global, not per-client. A single attacker can consume all 500 rate-limit tokens.
2. `SynchronizationStrategy.SYNCHRONIZED` on `gasLimitBucket` serializes individual `tryConsume` calls for thread safety, but does **not** prevent multiple concurrent threads from each successfully consuming tokens when the bucket has sufficient capacity. All 500 threads call `tryConsume(1500)` sequentially under the lock; each succeeds until the bucket reaches 0.
3. The two `tryConsume` calls (rate bucket, then gas bucket) are not atomic together — all 500 threads can pass the rate check before any of them hits the gas check.
4. No authentication, no IP extraction, no per-source fairness mechanism exists anywhere in the web3 throttle stack. [8](#0-7) 

**`restore()` does not help:** tokens are only restored on `InvalidParametersException` (gas > maxGasLimit). Requests with `gas=15000000` (exactly at the limit) pass `validateContractMaxGasLimit` and their tokens are never restored. [9](#0-8) 

### Impact Explanation
Every second an attacker runs the exploit, all legitimate users receive HTTP 429 "Gas per second rate limit exceeded" for the entire refill window. The endpoint is completely unavailable to other users during that window. Because the attack requires only standard HTTP clients and no credentials, it can be sustained indefinitely at zero cost to the attacker. The scope is the entire `/api/v1/contracts/call` surface — all callers, all contracts.

### Likelihood Explanation
The attacker needs no account, no token, no special knowledge — only the ability to open 500 parallel HTTP connections per second to the public endpoint. This is trivially achievable with tools like `ab`, `wrk`, `hey`, or a simple async HTTP client. The attack is repeatable every second with no cooldown or penalty. Any motivated party (competitor, vandal, stress-tester) can execute it.

### Recommendation
1. **Add per-IP (or per-authenticated-user) rate limiting** using a `ConcurrentHashMap<String, Bucket>` keyed on `request.getRemoteAddr()` (or a forwarded-IP header behind a proxy), so a single source cannot consume the global gas budget.
2. **Alternatively, cap per-request gas consumption** at a fraction of the global budget (e.g., `maxGasLimit / requestsPerSecond`) so no single request can consume a disproportionate share.
3. **Consider a token-bucket hierarchy**: a global bucket for aggregate enforcement plus a per-IP bucket for fairness.
4. If an API gateway (e.g., Nginx, Envoy, AWS API Gateway) sits in front, enforce per-IP rate limits there as a defense-in-depth layer.

### Proof of Concept
```bash
# Send 500 concurrent POST requests each with gas=15000000
# (adjust HOST to the target)
seq 1 500 | xargs -P500 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST https://HOST/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x"}'

# Expected: all 500 return 200 (or EVM error), gas bucket now at 0
# Immediately after, any new request returns 429 "Gas per second rate limit exceeded"
curl -X POST https://HOST/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000,"data":"0x"}'
# → HTTP 429

# Repeat every ~1 second to sustain the DoS
```

    ContractCallResponse call(@RequestBody @Valid ContractCallRequest request, HttpServletResponse response) {
        try {
            throttleManager.throttle(request);
            validateContractMaxGasLimit(request);

            final var params = constructServiceParameters(request);
            final var result = contractExecutionService.processCall(params);
            return new ContractCallResponse(result);
        } catch (InvalidParametersException e) {
            // The validation failed, but no processing occurred so restore the consumed tokens.
