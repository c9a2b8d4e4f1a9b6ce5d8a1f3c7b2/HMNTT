### Title
Unbounded Prometheus Label Cardinality via Unsanitized `req.method` in `metricsMiddleware`

### Summary
The `metricsMiddleware` function in `rest/middleware/metricsHandler.js` uses the raw, unvalidated `req.method` value directly as a Prometheus label in all four per-route instruments. Because Node.js's llhttp HTTP parser accepts any RFC 7230 token as a method name, an unauthenticated attacker can flood the server with requests bearing unique synthetic method strings (e.g., `FAKEMETHOD1`, `FAKEMETHOD2`, …), creating an unbounded number of unique label-set combinations in the OpenTelemetry SDK's in-memory store and exhausting the Node.js heap until the process crashes.

### Finding Description

**Exact code location:** `rest/middleware/metricsHandler.js`, lines 185–210, inside the `res.on('finish', …)` callback registered by `metricsMiddleware`.

```
191:  const method = req.method;          // raw, no validation
192:  const labels = {method, path, code};

207:  requestTotalCounter.add(1, labels);
208:  durationHistogram.record(duration, labels);
209:  requestSizeHistogram.record(..., labels);
210:  responseSizeHistogram.record(responseSize, labels);
```

**Root cause:** `req.method` is taken verbatim from the parsed HTTP request and used as a Prometheus label value. There is no allowlist of valid HTTP methods, no normalization, and no cardinality cap anywhere in the middleware or in the OpenTelemetry SDK configuration.

**Exploit flow:**
1. The attacker sends a stream of HTTP/1.1 requests where each request uses a distinct synthetic method token (`FAKEMETHOD1`, `FAKEMETHOD2`, …). Node.js's llhttp parser accepts any sequence of valid token characters as a method, so these requests are parsed successfully and reach Express.
2. The metrics middleware is registered globally at `rest/server.js` line 91 (`app.useExt(metricsHandler())`), before all route handlers. It immediately registers a `res.on('finish', …)` listener and calls `next()`.
3. When the response finishes (even with a 404 or 405 from a downstream handler), the listener fires and records all four instruments with `labels = {method: "FAKEMETHODn", path: "…", code: "…"}`.
4. Each unique `{method, path, code}` triple creates a new attribute-set entry in the OpenTelemetry SDK's in-memory `AttributesHashMap`. Each histogram entry additionally allocates per-bucket arrays (4 duration buckets + sum + count = 6 values per histogram × 3 histograms = 18 allocations per unique method per path).
5. With N unique methods × M paths × K status codes, the in-memory store grows without bound until the Node.js heap is exhausted and the process is killed by OOM.

**Why existing checks fail:**
- The `openApiValidator` (lines 63–65 of `server.js`) is gated on `config.openapi.validation.enabled`, which is not guaranteed to be true in production. Even when enabled, it is registered before the metrics middleware, so it would reject the request before the metrics listener is registered — but this is a fragile, configuration-dependent defense, not a deliberate cardinality guard.
- There is no HTTP method allowlist anywhere in the middleware stack.
- The OpenTelemetry SDK has no built-in cardinality limit for attribute sets by default.

### Impact Explanation
Successful exploitation causes the Node.js REST API process to exhaust heap memory and crash (OOM kill or V8 heap-limit abort). This takes down the entire REST API, preventing any client — including Hedera network participants — from querying mirror-node data. Recovery requires a process restart, but the attack can be repeated immediately, making the service permanently unavailable as long as the attacker sustains the request stream. This maps directly to the stated critical impact: network inability to confirm transactions via the mirror REST API.

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge of the application beyond the public REST endpoint. Any HTTP client capable of sending custom method strings (curl, Python `http.client`, raw TCP) can execute it. The attacker needs only a sustained stream of requests — even a single machine with modest bandwidth is sufficient. The attack is fully repeatable and automatable.

### Recommendation
1. **Normalize `req.method` to an allowlist** before using it as a label. Replace line 191 with:
   ```js
   const ALLOWED_METHODS = new Set(['GET','POST','PUT','PATCH','DELETE','HEAD','OPTIONS']);
   const method = ALLOWED_METHODS.has(req.method) ? req.method : 'UNKNOWN';
   ```
2. **Apply the same guard to `path`** in `toOpenApiPath` — unmatched routes fall back to `req.path`, which is also attacker-controlled.
3. **Enable OpenAPI validation unconditionally** (or at least enforce it for method filtering) so unknown methods are rejected before the metrics listener fires.
4. **Configure an OpenTelemetry cardinality limit** via `MeterProvider` view configuration to cap the number of unique attribute sets per instrument.

### Proof of Concept
```bash
# Send 10,000 requests with unique synthetic HTTP methods to any valid path
for i in $(seq 1 10000); do
  curl -s -o /dev/null -X "FAKEMETHOD${i}" http://<mirror-rest-host>/api/v1/transactions &
done
wait
# Monitor heap growth:
# watch -n1 'curl -s http://<mirror-rest-host>/metrics | grep nodejs_process_memory_heap_used'
```

Each iteration registers a new unique `method` label value across all four per-route instruments. Heap usage grows monotonically. At sufficient scale the process crashes with a JavaScript heap out-of-memory error. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
