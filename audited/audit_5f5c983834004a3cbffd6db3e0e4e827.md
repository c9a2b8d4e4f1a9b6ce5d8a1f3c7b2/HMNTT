### Title
Unbounded Cache Thrashing via Unique Query Flooding on Unauthenticated GraphQL Endpoint

### Summary
The `CachedPreparsedDocumentProvider` uses a Caffeine cache keyed on the raw query string with `maximumSize=1000` and no TTL-based eviction short enough to prevent abuse. Because the GraphQL module has no rate limiting, no authentication requirement, and no per-IP throttling, an unprivileged attacker can continuously submit more than 1000 distinct query strings to force perpetual cache eviction and re-parsing, causing sustained CPU overhead on the server.

### Finding Description
**Code locations:**

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15: default spec is `"expireAfterWrite=1h,maximumSize=1000,recordStats"` — hard cap of 1000 entries, 1-hour TTL.
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, lines 24–27: cache key is `executionInput.getQuery()` (the raw query string); on a cache miss, `parseAndValidateFunction.apply(executionInput)` is called synchronously.
- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 42–49: `graphQlCustomizer()` wires in the `PreparsedDocumentProvider` and chains `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — both of which execute only after parsing.

**Root cause:** The cache is bounded at 1000 entries. Once full, Caffeine evicts entries (W-TinyLFU policy). Every evicted entry whose query string is re-submitted triggers a full parse-and-validate cycle. There is no rate limiter, no per-IP throttle, and no authentication on the `/graphql` endpoint in the GraphQL module (confirmed: no `SecurityConfig`, no `ThrottleManager`, no Bucket4j — those exist only in the `web3` module).

**Exploit flow:**
1. Attacker generates 1001+ syntactically valid but lexically distinct query strings (e.g., varying field aliases, argument names, or adding semantically irrelevant whitespace variations that still pass `maxWhitespaceTokens=1000`).
2. Sends them in a tight loop. After the first 1000 fill the cache, every subsequent unique query evicts an existing entry.
3. By cycling through a set of >1000 unique queries, the attacker ensures the cache never serves a hit, forcing every request through `parseAndValidateFunction.apply()`.
4. The complexity and depth instrumentations run post-parse, so they do not prevent the parse cost.

**Why existing checks are insufficient:**
- `maxCharacters(10000)`, `maxTokens(1000)`, `maxRuleDepth(100)` bound the cost of a *single* parse but do not limit the *rate* of parse invocations.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` fire after parsing; they do not prevent the parse step.
- No HTTP-layer rate limiting exists in the GraphQL module.

### Impact Explanation
Sustained cache thrashing forces the GraphQL server to parse and validate every incoming request rather than serving from cache. Under concurrent load from a single attacker (or a small botnet), this can saturate CPU threads handling parse/validate work, degrading or denying service to legitimate users. The impact is a non-network-based DoS: the server remains reachable but becomes unresponsive due to CPU saturation.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge, and no elevated privileges — only the ability to send HTTP POST requests to the public `/graphql` endpoint. Generating 1001 distinct valid GraphQL query strings is trivial (e.g., vary a single field alias per request). The attack is repeatable, automatable, and can be sustained indefinitely from a single machine. No existing control in the GraphQL module blocks or slows this pattern.

### Recommendation
1. **Add rate limiting to the GraphQL endpoint** — integrate Bucket4j or Spring's `HandlerInterceptor`-based throttle (mirroring the `web3` module's `ThrottleManagerImpl`) to cap requests per IP per second.
2. **Normalize query strings before caching** — strip insignificant whitespace and sort field order so semantically equivalent queries share a cache key, reducing the effective unique-query space.
3. **Add a query allowlist or persisted queries** — only accept pre-registered query hashes; reject arbitrary query strings entirely.
4. **Reduce `maximumSize` or add a short `expireAfterAccess`** — a smaller cache with `expireAfterAccess` rather than `expireAfterWrite` naturally deprioritizes infrequently used entries, but this alone does not fix the root cause.

### Proof of Concept
```bash
# Generate 2000 unique but valid GraphQL queries and send them in a loop
for i in $(seq 1 2000); do
  curl -s -X POST http://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: {entityId: {shard: 0, realm: 0, num: $i}}) { alias$i: alias } }\"}" &
done
wait
# Repeat indefinitely; cache is capped at 1000, so entries 1001-2000 evict entries 1-1000,
# and the next cycle re-evicts 1001-2000, forcing continuous parse+validate on every request.
```
