### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Request Rate on `/api/v1/topics/{id}` in rest-java

### Summary
The rest-java module exposes `GET /api/v1/topics/{id}` with no per-IP or per-client rate limiting. Each request triggers three sequential database calls through `TopicServiceImpl.findById()`, `EntityService.findById()`, and `CustomFeeService.findById()`. An unprivileged attacker can flood this endpoint to exhaust the shared HikariCP connection pool, causing all other rest-java repository operations to time out or queue indefinitely.

### Finding Description
**Exact code path:**

`TopicController.getTopic()` (lines 31–37) calls three services per request with no guard: [1](#0-0) 

`TopicServiceImpl.findById()` (line 20) delegates directly to `TopicRepository.findById()` — a Spring Data `CrudRepository` method — with no caching, no circuit breaker, and no rate limiting: [2](#0-1) 

`TopicRepository` is a bare `CrudRepository` interface with no custom logic: [3](#0-2) 

**Root cause — missing rate limiting in rest-java:**

The only filters registered in `RestJavaConfiguration` are an ETag filter and a Protobuf converter. No throttle filter exists: [4](#0-3) 

`LoggingFilter` only logs; it does not gate requests: [5](#0-4) 

The `rest-java/src/main/resources/` directory contains only `banner.txt` — no `application.yml` with Hikari pool sizing or rate-limit configuration exists in the module.

The throttling infrastructure (bucket4j `ThrottleManagerImpl`, `ThrottleConfiguration`, `ThrottleProperties`) exists exclusively in the `web3` module: [6](#0-5) 

It is not wired into rest-java at all.

The shared HikariCP pool is configured via `CommonConfiguration` with `spring.datasource.hikari` properties: [7](#0-6) 

HikariCP's default `maximumPoolSize` is 10. The production Grafana alert confirms this is a known finite resource — it fires when active connections exceed 75% of max: [8](#0-7) 

**Exploit flow:**

1. Attacker sends concurrent GET requests to `/api/v1/topics/0.0.1` (or any valid/invalid topic ID) at high rate from one or more IPs.
2. Each request causes `TopicController.getTopic()` to invoke three service calls, each acquiring a HikariCP connection.
3. With pool size 10 and 3 connections per request, as few as 4 concurrent in-flight requests can saturate the pool.
4. Subsequent requests from all clients (including legitimate ones hitting `/api/v1/accounts`, `/api/v1/transactions`, etc.) block waiting for a connection up to `connectionTimeout` ms, then fail with `SQLTimeoutException` or `CannotGetJdbcConnectionException`.
5. The attacker sustains this with a simple loop: `while true; do curl -s http://mirror-node/api/v1/topics/1 & done`.

### Impact Explanation
All rest-java endpoints share the same HikariCP pool. Pool exhaustion causes a full service outage for the rest-java REST API — not just the topics endpoint. Legitimate users querying accounts, tokens, transactions, or network info all receive errors. The mirror node's read API becomes unavailable for the duration of the attack. This is a griefing/DoS with no economic damage to the Hedera network itself, consistent with the Medium scope classification.

### Likelihood Explanation
No authentication, API key, or network-level credential is required. The endpoint is publicly reachable. The attack requires only an HTTP client and a loop — no exploit code, no special knowledge. It is trivially repeatable and can be sustained indefinitely. A single machine with modest bandwidth can exhaust a 10-connection pool. The absence of any rate-limiting code in the rest-java module (confirmed by directory listing and grep) means there is no in-process defense.

### Recommendation
1. **Add a global rate-limiting servlet filter to rest-java** using bucket4j (already a dependency in the sibling `web3` module), mirroring `ThrottleConfiguration` and `ThrottleManagerImpl`. Apply a per-IP token-bucket limit on all `/api/v1/**` routes.
2. **Increase and explicitly configure the HikariCP pool size** in rest-java's application configuration, and set a short `connectionTimeout` so exhausted-pool requests fail fast rather than queuing.
3. **Add response caching** (the Redis cache infrastructure already exists per `RestJavaProperties`) for the `GET /api/v1/topics/{id}` endpoint to reduce DB round-trips per request.
4. **Deploy an API gateway or reverse proxy** (e.g., nginx, Envoy) in front of rest-java with per-IP rate limiting as a defense-in-depth layer.

### Proof of Concept
```bash
# Flood the topics endpoint from a single unauthenticated client
# Replace MIRROR_HOST with the target mirror node address

for i in $(seq 1 500); do
  curl -s "http://MIRROR_HOST:8080/api/v1/topics/0.0.1" &
done
wait

# Immediately probe a different endpoint to observe pool exhaustion impact
curl -v "http://MIRROR_HOST:8080/api/v1/accounts/0.0.2"
# Expected: connection timeout or HTTP 500 due to HikariCP pool exhaustion
```

Sustained execution (wrapping in `while true`) will keep the pool saturated. Monitor with the existing Grafana alert (`RestJavaHighDBConnections`) — it will fire within seconds showing >75% pool utilization, confirming the attack is effective.

    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
