### Title
Unauthenticated Alias-Based DB Query Amplification via `getHooks()` — No Rate Limiting or Caching in `rest-java`

### Summary
The `HookServiceImpl.getHooks()` method unconditionally calls `entityService.lookup(request.getOwnerId())`, which for alias-type parameters issues a raw, uncached SQL query against the `entity` table on every invocation. The `rest-java` module has no rate-limiting layer protecting the hooks endpoint, and the `EntityRepository.findByAlias()` in `rest-java` carries no `@Cacheable` annotation. An unprivileged attacker can flood the endpoint with concurrent alias-based requests, exhausting the DB connection pool and driving CPU above the 30% threshold.

### Finding Description

**Exact code path:**

`HooksController` → `HookServiceImpl.getHooks()` (line 35) → `EntityServiceImpl.lookup()` (line 33) → `EntityRepository.findByAlias()` (line 13–14, rest-java).

In `EntityServiceImpl.lookup()`:
```java
case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
```
This executes the native query:
```sql
select id from entity where alias = ?1 and deleted <> true
```
on every single request. There is **no `@Cacheable` annotation** on `rest-java`'s `EntityRepository.findByAlias()` (contrast with `web3`'s `EntityRepository`, which uses `@Cacheable(cacheNames = CACHE_NAME_ALIAS, ...)`).

**No rate limiting in `rest-java`:** The `ThrottleConfiguration` and `ThrottleManagerImpl` (bucket4j-based) exist exclusively in the `web3` module. The `rest-java` config directory contains only `LoggingFilter`, `MetricsFilter`, `WebMvcConfiguration`, etc. — no throttle bean, no per-IP limit, no global request-rate guard on the hooks endpoint.

**Root cause / failed assumption:** The design assumes that infrastructure-level rate limiting (e.g., an API gateway or reverse proxy) will protect the endpoint. No such protection is enforced in code, and the alias lookup path is not memoized, so every concurrent HTTP request maps 1:1 to a DB connection checkout and query execution.

### Impact Explanation
Each concurrent alias-based request holds a JDBC connection from the pool for the duration of the `findByAlias` query plus the subsequent `hookRepository` query. Under high concurrency, the pool is exhausted, causing queued threads to block, latency to spike, and CPU to rise due to thread contention and query scheduling overhead. Because the endpoint is public and requires no credentials, the attacker can sustain this with a modest botnet or even a single machine with async HTTP tooling. The impact is degraded availability for all mirror-node consumers sharing the same DB pool, meeting the >30% resource increase threshold.

### Likelihood Explanation
Preconditions are minimal: the attacker needs only to know one or more valid entity aliases (these are public Hedera account aliases, discoverable via the network). The attack is trivially repeatable with tools like `ab`, `wrk`, or `asyncio`-based scripts. No authentication, no CAPTCHA, no token is required. The absence of any in-process rate limiting makes sustained exploitation straightforward.

### Recommendation
1. **Add caching** to `rest-java`'s `EntityRepository.findByAlias()` and `findByEvmAddress()` mirroring the `@Cacheable` pattern already used in `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java` (lines 39–49).
2. **Add rate limiting** to the `rest-java` module for the hooks (and related) endpoints, using the same bucket4j pattern already present in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`.
3. **Enforce connection pool limits** with a short acquisition timeout so that pool exhaustion fails fast rather than queuing unboundedly.

### Proof of Concept

**Preconditions:** Mirror node is running; at least one valid entity alias is known (e.g., obtained from `/api/v1/accounts`).

**Steps:**
```bash
# Replace <ALIAS> with a valid base32-encoded alias, e.g. HIQQEAYB...
# Replace <HOST> with the mirror node REST-Java host

# Fire 500 concurrent requests using wrk or similar
wrk -t 50 -c 500 -d 60s \
  "http://<HOST>/api/v1/accounts/<ALIAS>/hooks"
```

**Expected result:** DB connection pool saturates; query latency for all other endpoints increases; CPU on the mirror node and DB host rises by >30% compared to baseline; connection pool wait metrics spike in application logs.
