### Title
Unauthenticated Connection Pool Exhaustion via Concurrent Requests in REST API

### Summary
The `initializePool()` function in `rest/dbpool.js` creates a global `pg` pool capped at **10 connections** by default with no per-IP rate limiting or concurrent-request throttling anywhere in the REST API middleware stack. An unprivileged attacker can send ≥10 concurrent long-running requests to any DB-backed route (e.g., `GET /api/v1/transactions`) to hold all pool connections for up to `statementTimeout` (20 s), causing every subsequent request to queue and then fail after `connectionTimeoutMillis` (20 s), effectively halting the server's ability to serve transaction queries.

### Finding Description

**Exact code path:**

`rest/dbpool.js` lines 7–16 and 35–37 — the pool is constructed with:
- `max: config.db.pool.maxConnections` → default **10**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` → default **20 000 ms**
- `statement_timeout: config.db.pool.statementTimeout` → default **20 000 ms** [1](#0-0) [2](#0-1) 

`rest/server.js` lines 67–98 — the full middleware chain applied to every route is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. **No rate-limit, no in-flight-request cap, no per-IP throttle.** [3](#0-2) 

`rest/middleware/authHandler.js` lines 15–36 — `authHandler` only sets a custom *response-row limit* for authenticated users; it does not reject, throttle, or count concurrent requests from any caller. [4](#0-3) 

**Root cause / failed assumption:** The design assumes that `statement_timeout` (20 s) and `connectionTimeoutMillis` (20 s) are sufficient safety valves. They are not: they only bound *how long* a single connection is held and *how long* a waiter queues, but they do not prevent an attacker from continuously re-filling all 10 slots faster than they drain.

**Exploit flow:**
1. Attacker opens ≥10 concurrent TCP connections and sends `GET /api/v1/transactions` (or any other DB-backed endpoint) on each.
2. Each request acquires one pool connection and executes a query that can run up to 20 s.
3. All 10 pool slots are occupied.
4. Every legitimate request that arrives now enters the `pg` internal wait queue.
5. After 20 s the waiters receive `Error: timeout exceeded when trying to connect` and the HTTP response is a 500.
6. Attacker re-sends requests immediately upon each slot release, maintaining saturation indefinitely with a trivial loop.

**Why existing checks fail:**
- `statement_timeout` kills the query server-side but the connection is still held until the error propagates back through the Node.js event loop.
- `connectionTimeoutMillis` causes queued callers to *fail*, not to be protected — legitimate users get errors.
- The throttling in `web3/ThrottleManagerImpl` is in a separate Java service and has no effect on the Node.js REST API.
- The Helm chart `inFlightReq` / `rateLimit` Traefik middleware is present for the Rosetta API but **absent** for the REST API. [5](#0-4) 

### Impact Explanation
The REST API is the primary interface for clients to confirm whether a submitted transaction has reached consensus. Exhausting the 10-connection pool causes every DB-backed endpoint (`/api/v1/transactions`, `/api/v1/accounts`, etc.) to return 500 errors for the duration of the attack. This constitutes a complete read-path denial of service for the mirror node, preventing any external observer from confirming new transactions — matching the stated critical severity of "total network shutdown" from the perspective of transaction visibility.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero special knowledge beyond the public API path. A single attacker machine with a trivial script (e.g., `ab -c 15 -n 100000 http://host:5551/api/v1/transactions`) is sufficient. The attack is repeatable, cheap, and fully automated. The default pool size of 10 makes the threshold extremely low.

### Recommendation
1. **Add a concurrent-request cap** at the Express layer (e.g., `express-rate-limit` or a custom semaphore) to limit in-flight DB-bound requests per IP before they reach the pool.
2. **Increase `maxConnections`** to a value commensurate with expected concurrency (the Helm chart already provisions `mirror_rest` with up to 250 server-side connections via pgBouncer).
3. **Add Traefik `inFlightReq` and `rateLimit` middleware** to the REST API Helm chart, mirroring what is already configured for the Rosetta API.
4. **Reduce `connectionTimeoutMillis`** so queued requests fail fast rather than holding event-loop slots for 20 s.

### Proof of Concept
```bash
# Requires: Apache Bench or any tool that supports concurrency
# Replace HOST with the mirror node REST API address

# Step 1: Saturate the pool (10 concurrent, long-running queries)
ab -c 15 -n 10000 "http://HOST:5551/api/v1/transactions?limit=100&order=asc"

# Step 2: In a second terminal, observe that legitimate requests now fail
curl -v "http://HOST:5551/api/v1/transactions"
# Expected: HTTP 500 with "timeout exceeded when trying to connect" or equivalent

# Step 3: Confirm recovery only after attacker stops
# Stop ab, then retry curl — requests succeed again
```

The attack is sustained as long as the attacker maintains ≥10 concurrent open requests. No credentials, tokens, or special headers are required.

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
