### Title
Unbounded Opcode List Accumulation in `OpcodeContext` Leads to OOM DoS via Unauthenticated `/opcodes` Endpoint

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` accepts `stack=true` by default from any unauthenticated caller. During EVM replay, `OpcodeActionTracer.tracePostExecution()` calls `OpcodeContext.addOpcodes()` for every executed opcode with no size cap on the backing `ArrayList`. A single request replaying a high-gas historical transaction can accumulate millions of `Opcode` objects in heap memory, causing an `OutOfMemoryError` and crashing the web3 service.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` accepts `stack=true` by default with no authentication: [1](#0-0) 

`OpcodeServiceImpl.processOpcodeCall()` creates an `OpcodeContext` with an initial `ArrayList` capacity of `gas / 3` — this is only a hint, not a hard cap: [2](#0-1) 

`OpcodeContext.addOpcodes()` appends to an unbounded `ArrayList` with zero size enforcement: [3](#0-2) 

`OpcodeActionTracer.tracePostExecution()` calls `addOpcodes()` unconditionally for every single EVM opcode executed during replay: [4](#0-3) 

**Root cause:** The failed assumption is that gas-bounded execution implies memory-bounded tracing. In reality, with `maxGasLimit = 15,000,000` and cheap opcodes (e.g., `JUMPDEST` at 1 gas, `PUSH1` at 3 gas), a single transaction replay can generate 5–15 million `Opcode` objects. With `stack=true`, each `Opcode` carries a `List<String>` snapshot of the EVM stack (up to 1024 × 66-byte hex strings). Total heap consumption can reach multiple gigabytes per request. [5](#0-4) 

**Why existing checks fail:**

The only guards are:
1. `Accept-Encoding: gzip` header check — trivially satisfied by any HTTP client
2. `throttleOpcodeRequest()` — enforces 1 RPS via a token bucket, but does **not** limit per-request memory consumption
3. `requestTimeout = 10000ms` — OOM can occur well within 10 seconds for large transactions [6](#0-5) [7](#0-6) 

### Impact Explanation
A single well-crafted request targeting a known high-gas transaction causes `OutOfMemoryError` in the JVM, crashing or severely degrading the web3 service for all users. Because the endpoint replays historical transactions from the database, the attacker does not need to submit any on-chain transaction — they only need to discover the hash of any existing high-gas transaction (trivially available from block explorers). This is a complete, unauthenticated denial-of-service against the web3 module.

### Likelihood Explanation
The endpoint is publicly accessible when `hiero.mirror.web3.opcode.tracer.enabled=true`. High-gas transactions are publicly visible on Hedera mainnet/testnet via block explorers. The attacker needs only a valid `Accept-Encoding: gzip` header and a known transaction hash. The 1 RPS throttle is irrelevant since a single request suffices. The attack is trivially repeatable after service recovery.

### Recommendation
1. **Hard cap on opcode count**: In `OpcodeContext.addOpcodes()`, enforce a maximum list size (e.g., 100,000 opcodes) and throw a controlled exception or truncate when exceeded.
2. **Per-request memory budget**: Track estimated memory usage during tracing and abort when a configurable threshold is exceeded.
3. **Gas-based pre-check**: Reject requests where the transaction's gas limit exceeds a configurable threshold for opcode tracing.
4. **Configurable stack depth limit**: When `stack=true`, limit the number of stack items captured per opcode.

### Proof of Concept
1. Ensure `hiero.mirror.web3.opcode.tracer.enabled=true` on the target instance.
2. Identify a historical transaction with gas usage near `maxGasLimit` (15,000,000) that executed many cheap opcodes (e.g., a tight loop contract). Such transactions are publicly visible on Hedera explorers.
3. Send the following request:
```
GET /api/v1/contracts/results/{high_gas_tx_hash}/opcodes?stack=true
Accept-Encoding: gzip
```
4. The server replays the transaction; `OpcodeActionTracer.tracePostExecution()` is called for each of the millions of opcodes, each appending a new `Opcode` with a full stack snapshot to the unbounded `OpcodeContext.opcodes` `ArrayList`.
5. JVM heap is exhausted → `OutOfMemoryError` → web3 service crash or severe degradation.

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
