### Title
Unbounded Redis Channel Subscription Exhaustion via Distinct Topic ID Flooding in `RedisTopicListener`

### Summary
`RedisTopicListener.getSharedListener()` creates one Redis pub/sub channel subscription per unique topic ID with no upper bound on the total number of active subscriptions. An unprivileged attacker who can enumerate valid topic IDs can open concurrent gRPC streams to thousands of distinct topics, exhausting Redis pub/sub capacity and severing the mirror node's real-time message delivery pipeline from Redis.

### Finding Description

**Exact code path:**

`RedisTopicListener.getSharedListener()` ( [1](#0-0) ) calls `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))` where `topicMessages` is an unbounded `ConcurrentHashMap<String, Flux<TopicMessage>>`. [2](#0-1) 

Each unique topic ID (derived from `filter.getTopicId().getId()` formatted as `"topic.%d"`) [3](#0-2)  triggers a call to `subscribe(topic)`, which issues a Redis `SUBSCRIBE` command via `ReactiveRedisMessageListenerContainer.receive()`. [4](#0-3) 

The subscription is `.share()`d, meaning multiple gRPC clients on the *same* topic share one Redis channel. However, N *distinct* topic IDs produce N distinct Redis channel subscriptions with no cap. [5](#0-4) 

`unsubscribe()` is only called on `doOnCancel` / `doOnComplete`. [6](#0-5)  With `limit=1` on a topic that never receives a message, the flux never completes and the Redis subscription stays open indefinitely.

**Root cause:** The `topicMessages` map has no size bound, and `ListenerProperties` exposes no `maxActiveSubscriptions` or `maxDistinctTopics` property. [7](#0-6) 

**Failed assumption:** The design assumes the number of distinct subscribed topics is small and bounded by legitimate usage. There is no enforcement of this assumption.

### Impact Explanation

Redis pub/sub has a hard limit on the number of concurrent channel subscriptions per connection (configurable, but finite). When this limit is reached, new `SUBSCRIBE` commands fail. Since `RedisTopicListener` uses a single `ReactiveRedisMessageListenerContainer` (one underlying connection) [8](#0-7) , exhausting its subscription capacity means all subsequent legitimate subscribers to new topics receive no real-time messages. The importer's `RedisPublisher` continues publishing to Redis channels [9](#0-8) , but the gRPC layer cannot subscribe to receive them — effectively partitioning the service from Redis for new topic subscriptions. Existing subscriptions on already-subscribed channels continue to work, but any new topic subscription silently fails or errors.

### Likelihood Explanation

**Preconditions:**
- `hiero.mirror.grpc.checkTopicExists` defaults to `true` [10](#0-9) , so the attacker needs valid topic IDs. On public Hedera networks (mainnet/testnet), topic IDs are publicly enumerable via the REST API.
- `hiero.mirror.grpc.netty.maxConcurrentCallsPerConnection` = 5 limits calls per TCP connection, but there is no limit on the number of TCP connections from a single client or in total. An attacker opens many connections (e.g., 200 connections × 5 calls = 1,000 concurrent streams).
- No authentication is required to call `subscribeTopic`.

**Feasibility:** High. The attacker needs only a list of valid topic IDs (trivially obtained from the public REST API) and the ability to open many TCP connections to port 5600. No special privileges, credentials, or insider knowledge are required.

**Repeatability:** After subscriptions are cleaned up (e.g., service restart), the attack can be repeated immediately.

### Recommendation

1. **Add a `maxDistinctTopics` cap** to `ListenerProperties` and enforce it in `getSharedListener()` — reject new topic subscriptions when the map size exceeds the cap.
2. **Add a global `maxActiveSubscriptions` limit** in `TopicMessageServiceImpl` analogous to the `canSubscribe()` check designed for `ContractLogTopicListener` (referenced in design docs). The existing `subscriberCount` AtomicLong [11](#0-10)  is metrics-only; add a hard rejection when it exceeds a configured threshold.
3. **Enforce per-IP or per-client connection limits** at the load balancer / Netty layer (the `maxRatePerEndpoint: 250` GCP gateway config [12](#0-11)  helps but is insufficient alone).
4. **Use a bounded map** (e.g., Caffeine `maximumSize`) for `topicMessages` with an eviction policy.

### Proof of Concept

```python
import grpc
import threading
# Enumerate N valid topic IDs from the public REST API
topic_ids = enumerate_valid_topics(count=5000)  # e.g., via GET /api/v1/topics

def subscribe(topic_id):
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(channel)
    req = ConsensusTopicQuery(
        topicID=TopicID(topicNum=topic_id),
        limit=1  # subscription stays open if topic never receives a message
    )
    try:
        for _ in stub.subscribeTopic(req, timeout=3600):
            pass
    except:
        pass

# Open 5 streams per connection across 1000 connections = 5000 distinct Redis subscriptions
threads = [threading.Thread(target=subscribe, args=(tid,)) for tid in topic_ids]
for t in threads:
    t.start()
# Redis pub/sub subscription limit exhausted; new legitimate topic subscriptions fail
```

After this runs, any new gRPC client subscribing to a topic not already in `topicMessages` will fail to receive real-time messages because `ReactiveRedisMessageListenerContainer.receive()` cannot issue new `SUBSCRIBE` commands on the saturated connection.

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
