### Title
Unauthenticated HTTP Node-List Poisoning in `newClient()` Enables Consensus-Node Redirection and Fee Manipulation

### Summary
When `cfg.network` is set to `"other"`, `newClient()` in `pinger/sdk_client.go` calls `buildNetworkFromMirrorNodes()`, which fetches the consensus node list over a plain, unauthenticated HTTP connection to `cfg.mirrorRest` with no TLS enforcement and no cryptographic integrity check on the response. An attacker with network-path access (e.g., same Kubernetes cluster, DNS poisoning, or ARP spoofing on the same segment) can intercept and replace the JSON node list, redirecting the `hiero.Client` to attacker-controlled gRPC endpoints that apply non-standard fee schedules, causing transaction fees to exceed design parameters.

### Finding Description
**Exact code path:**

- `pinger/sdk_client.go` lines 17–22: when `cfg.network == "other"`, `buildNetworkFromMirrorNodes(context.Background(), cfg)` is called and its result is passed directly to `hiero.ClientForNetwork(netmap)` with no further validation.
- `pinger/mirror_node_client.go` line 46: `httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}` — a vanilla Go HTTP client with no TLS configuration, no certificate pinning, and no HTTPS enforcement.
- `pinger/mirror_node_client.go` lines 79–98: `fetchMirrorNodeNetwork()` issues a plain `GET` to the URL derived from `cfg.mirrorRest`, decodes the JSON body, and returns the network map. There is no signature verification, no HMAC, no certificate check, and no comparison against a known-good node list.
- `pinger/config.go` line 37: the default example URL in the flag description is `http://mirror-rest:5551` — plaintext HTTP — and there is no validation that the supplied URL uses `https://`.

**Root cause:** The code assumes the network path to the mirror REST endpoint is trusted. It makes no attempt to enforce TLS, verify the server's identity, or authenticate the response payload.

**Why existing checks are insufficient:**
- Lines 102–124 of `mirror_node_client.go` only validate that `node_account_id` is non-empty and parseable, and that endpoints have a non-empty host and non-zero port. These are purely syntactic checks — they accept any well-formed attacker-supplied address.
- There is no allowlist of known-good node account IDs or IP ranges.
- There is no response signature or hash to verify authenticity.

**Exploit flow:**
1. Attacker positions themselves on the network path between the pinger and `cfg.mirrorRest` (e.g., DNS poisoning of `mirror-rest`, ARP spoofing on the same Kubernetes node, or compromising any pod in the same namespace that can intercept cluster-internal traffic).
2. Attacker intercepts the HTTP GET to `/api/v1/network/nodes`.
3. Attacker returns a crafted JSON payload with `service_endpoints` pointing to attacker-controlled gRPC servers and valid-looking `node_account_id` values (e.g., `"0.0.3"`).
4. `buildNetworkFromMirrorNodes()` parses the crafted response and returns a `map[string]hiero.AccountID` mapping attacker-controlled addresses to those account IDs.
5. `hiero.ClientForNetwork(netmap)` creates a client targeting attacker-controlled nodes.
6. Every subsequent `TransferTransaction` in `pinger/transfer.go` (lines 29–33) is submitted to the attacker's nodes, which can apply arbitrary fee schedules.

### Impact Explanation
The pinger's operator private key (`cfg.operatorKey`) signs and submits real `TransferTransaction`s to attacker-controlled consensus nodes. Those nodes can:
- Apply non-standard (inflated) fee schedules, draining the operator account beyond design parameters.
- Observe all transaction details (operator account ID, destination account ID, amounts, private key usage patterns).
- Return forged receipts, masking the attack from monitoring.

Severity is **High**: the operator account's funds are directly at risk, and the attack is silent (the pinger continues to log "transfer success" if the attacker returns plausible receipts).

### Likelihood Explanation
The deployment context (liveness probe at `/tmp/alive`, readiness probe at `/tmp/ready`, internal service name `mirror-rest:5551`) is characteristic of a Kubernetes cluster. In such environments:
- DNS poisoning within a namespace requires only a compromised pod (no cluster-admin privileges).
- ARP spoofing on the same node requires only host-network access or a compromised DaemonSet pod.
- If `cfg.mirrorRest` is operator-supplied and points to a public HTTP endpoint, any on-path ISP or cloud provider can intercept.

The attack is repeatable: `newClient()` is called once at startup, so a single poisoned response permanently redirects all subsequent transactions for the lifetime of the process.

### Recommendation
1. **Enforce HTTPS**: Validate at config load time (`pinger/config.go`) that `cfg.mirrorRest` begins with `https://`. Reject `http://` URLs with a fatal error.
2. **Use a TLS-aware HTTP client**: Configure `httpClient` with a `tls.Config` that enforces certificate verification and optionally pins the CA.
3. **Validate node account IDs**: Maintain a compile-time or config-time allowlist of expected node account IDs (e.g., `0.0.3`–`0.0.28` for mainnet) and reject any response containing IDs outside that set.
4. **Verify response integrity**: If the mirror node API supports signed responses or a known hash, verify it before consuming the payload.

        self.send_response(200)
        self.send_header('Content-Type','application/json')
