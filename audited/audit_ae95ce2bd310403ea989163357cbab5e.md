### Title
Hardcoded `sslmode=disable` Forces Plaintext PostgreSQL Connections, Enabling Network MITM Credential and Data Interception

### Summary
`GetDsn()` in `rosetta/app/config/types.go` hardcodes `sslmode=disable` unconditionally in the PostgreSQL DSN, with no configuration override possible. Every connection opened by `ConnectToDb()` in `rosetta/app/db/db.go` transmits credentials and all query data in plaintext over the wire. Any attacker with network access between the Rosetta service and the PostgreSQL host can passively capture or actively intercept the full PostgreSQL wire protocol stream.

### Finding Description
**Exact code path:**

`rosetta/app/config/types.go`, `GetDsn()`, line 49–58:
```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        db.Host, db.Port, db.Username, db.Name, db.Password,
    )
}
```
This DSN is consumed directly at `rosetta/app/db/db.go` line 18:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), &gorm.Config{...})
```

**Root cause:** `sslmode=disable` is a string literal baked into the format string. The `Db` struct (`types.go` lines 39–47) has no `SSLMode` field, so there is no configuration path — environment variable, YAML key, or otherwise — that can enable TLS. The test in `rosetta/app/config/types_test.go` line 24 explicitly asserts `sslmode=disable` as the expected output, confirming this is intentional and locked in.

**Why existing checks fail:** There are none. The `Db` struct exposes `Host`, `Port`, `Username`, `Password`, `Name`, and `Pool` — SSL mode is absent entirely. PostgreSQL's `sslmode=disable` instructs the client to never negotiate TLS even if the server supports or requires it.

**Exploit flow:**
1. Attacker gains a network-adjacent position (same subnet, compromised router/switch, ARP poisoning on a flat LAN, or a rogue pod in the same Kubernetes namespace).
2. Attacker runs a passive TCP capture (e.g., `tcpdump -i eth0 port 5432 -w capture.pcap`) or an active MITM proxy (e.g., `mitmproxy` with a TCP passthrough plugin).
3. When `ConnectToDb()` is called at Rosetta startup, the PostgreSQL startup message containing `user` and `database` fields is sent in plaintext, followed by the MD5 or SCRAM-SHA-256 password exchange — both visible in plaintext framing at the wire level. With MD5 auth the hash is trivially crackable offline; with SCRAM the session key material is exposed.
4. All subsequent `SELECT` queries and their result rows (transaction records, account data, etc.) flow in plaintext PostgreSQL wire protocol messages, readable without any decryption.

### Impact Explanation
- **Credential exposure:** The database username and password (or their wire-protocol equivalents) are transmitted in cleartext on every new connection. Captured credentials allow direct database access bypassing the Rosetta API entirely.
- **Data confidentiality breach:** All query results — blockchain transaction history, account balances, cryptographic keys stored in the DB — are readable to the interceptor.
- **Integrity:** An active MITM can inject or modify query responses, causing the Rosetta API to return falsified blockchain data to clients.
- **Severity: High** — complete loss of confidentiality and integrity of the database tier.

### Likelihood Explanation
- **Cloud/Kubernetes deployments:** The `Db.Host` field is configurable (not localhost-only), meaning the DB is routinely on a separate host. In Kubernetes, pods on the same node or namespace share a network bridge; a compromised sidecar or co-tenant pod can ARP-spoof or sniff without OS-level host privileges.
- **On-premise / VM deployments:** A compromised host on the same VLAN can passively sniff all PostgreSQL traffic with no privileges on the target machines.
- **No attacker privileges required on the Rosetta or DB hosts** — only network adjacency, which is a realistic attacker capability in cloud environments with lateral movement from any compromised workload.
- **Repeatability:** Passive sniffing is persistent and undetectable; every new DB connection re-exposes credentials.

### Recommendation
1. Add an `SSLMode` field to the `Db` struct in `rosetta/app/config/types.go` with a secure default (e.g., `"require"` or `"verify-full"`):
   ```go
   type Db struct {
       Host             string
       Name             string
       Password         string
       Pool             Pool
       Port             uint16
       SSLMode          string `yaml:"sslMode"`
       StatementTimeout int    `yaml:"statementTimeout"`
       Username         string
   }
   ```
2. Update `GetDsn()` to use the field, defaulting to `"require"` if empty:
   ```go
   func (db Db) GetDsn() string {
       sslMode := db.SSLMode
       if sslMode == "" {
           sslMode = "require"
       }
       return fmt.Sprintf(
           "host=%s port=%d user=%s dbname=%s password=%s sslmode=%s",
           db.Host, db.Port, db.Username, db.Name, db.Password, sslMode,
       )
   }
   ```
3. Configure the PostgreSQL server to require TLS (`ssl = on`, `hostssl` entries in `pg_hba.conf`).
4. For production, use `sslmode=verify-full` with a trusted CA certificate to prevent certificate spoofing.
