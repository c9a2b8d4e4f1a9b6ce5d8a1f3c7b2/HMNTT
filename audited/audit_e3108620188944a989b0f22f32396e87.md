### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded `/schedules` Endpoint Flooding

### Summary
The `getSchedules()` handler in `rest/schedules.js` issues up to 3 database queries per request with no rate limiting enforced at the application layer. An unauthenticated attacker can flood `GET /api/v1/schedules` to saturate the default 10-connection pool, causing all subsequent queries across the entire REST API to queue or timeout, effectively degrading the mirror node service.

### Finding Description
**Exact code path:**

`rest/schedules.js`, `getSchedules()`, lines 234–282:
- Line 241: `pool.queryQuietly(schedulesQuery, params)` — first query, always executed
- Lines 261–264: `pool.queryQuietly(entityQuery, entityIds)` and `pool.queryQuietly(signatureQuery, entityIds)` — two additional parallel queries executed when results are non-empty

**Root cause — no rate limiting:**

`rest/server.js` (lines 67–98) registers the full middleware stack: `urlencoded`, `json`, `cors`, optional `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. There is no rate-limiting middleware anywhere in this chain. No `express-rate-limit` or equivalent package is present in `rest/package.json`.

**`authHandler` is insufficient:**

`rest/middleware/authHandler.js` (lines 15–36) only processes requests that supply HTTP Basic credentials. Unauthenticated requests (`credentials` is null) return immediately at line 19 with no restriction applied. The `user.limit` field (line 32) controls only the DB result-set size, not request throughput.

**Response cache is off by default:**

`rest/server.js` line 54: `applicationCacheEnabled = config.cache.response.enabled && config.redis.enabled`. Documentation confirms `cache.response.enabled` defaults to `false`, so no caching absorbs repeated identical requests.

**Pool size:**

`rest/dbpool.js` line 14: `max: config.db.pool.maxConnections`. Documentation confirms the default is **10 connections**. With 3 queries per `getSchedules()` call (2 running in parallel via `Promise.all`), only ~4–5 concurrent requests are needed to saturate the pool. Thousands of requests per second will keep the pool permanently exhausted.

### Impact Explanation
With the pool saturated, every endpoint sharing the same `global.pool` — accounts, balances, tokens, transactions, topics — will stall waiting for a connection up to `connectionTimeoutMillis` (default 20 s), then fail. The mirror node REST API becomes entirely unresponsive to legitimate users. Because mirror nodes are the primary read interface for dApps, wallets, and explorers querying Hedera state, sustained exhaustion degrades the observable availability of the network's data layer across all deployed mirror node instances targeted by the attack.

### Likelihood Explanation
No authentication, API key, or IP-based throttle is required. Any attacker with internet access and a basic HTTP flood tool (e.g., `wrk`, `ab`, `hey`) can trigger this. The endpoint is publicly documented in `rest/api/v1/openapi.yml` (lines 1030–1050). The attack is trivially repeatable, requires no special knowledge of the system, and can be sustained indefinitely from a single machine or botnet.

### Recommendation
1. Add a per-IP rate-limiting middleware (e.g., `express-rate-limit`) globally in `rest/server.js` before route registration, with a conservative default (e.g., 100 req/min per IP).
2. Enforce a concurrency limit on in-flight DB queries (e.g., using a semaphore or `pg-pool`'s queue depth cap).
3. Enable the Redis response cache by default for read-only list endpoints like `/schedules` to serve repeated requests without hitting the DB.
4. Consider increasing `maxConnections` for production deployments and documenting a minimum recommended value.

  if (schedules.length === 0) {

  if (!credentials) {
