### Title
Unthrottled Safety-Check DB Query Amplification via Unbounded gRPC Subscriptions

### Summary
Any unprivileged client can open an arbitrary number of gRPC connections and subscribe to a topic without an `endTime` or `limit`. After one second, a built-in safety check fires for every active subscription and invokes `topicMessageRetriever.retrieve(filter, false)` — the unthrottled path. In unthrottled mode, `PollingContext.isComplete()` never returns `true` when no limit is set, so the retriever always exhausts all `maxPolls=12` iterations at `pollingFrequency=20ms`, issuing 12 DB queries per subscription in ~240ms. With no per-IP connection limit or global subscription cap, an attacker can multiply this into hundreds of concurrent DB query bursts, preventing a recovering database from stabilising after a network partition.

### Finding Description
**Code path:**

`TopicMessageServiceImpl.subscribeTopic()` (lines 67–70) schedules a one-shot safety check for every subscription:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
    .filter(_ -> !topicContext.isComplete())   // always true when endTime == null
    .flatMapMany(_ -> missingMessages(topicContext, null))
    .subscribeOn(Schedulers.boundedElastic());
```

`topicContext.isComplete()` (line 203–215) returns `false` whenever `filter.getEndTime() == null`, which is the default for any open-ended subscription. The safety check therefore fires unconditionally for every such subscription.

`missingMessages(topicContext, null)` (line 142–149) calls:

```java
return topicMessageRetriever.retrieve(gapFilter, false);   // throttled=false
```

Inside `PollingTopicMessageRetriever.retrieve()` (lines 50–55), the unthrottled `PollingContext` is constructed with:

```java
numRepeats = unthrottled.getMaxPolls();          // 12
frequency  = unthrottled.getPollingFrequency();  // 20 ms
maxPageSize = unthrottled.getMaxPageSize();      // 5000
```

The repeat predicate is `!context.isComplete()`. In unthrottled mode, `PollingContext.isComplete()` (lines 121–128) is:

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) { ... }
    return limitHit;   // always false when no limit is set
}
```

Because the safety-check filter is built without a limit (`topicContext.getFilter().toBuilder().startTime(startTime).build()` — no `.limit()` call), `limitHit` is always `false`, so `isComplete()` never short-circuits the loop. The retriever unconditionally fires all 12 polls at 20 ms intervals regardless of whether any rows are returned, issuing 12 DB queries in ~240 ms per subscription.

**Root cause:** The unthrottled `isComplete()` has no empty-page guard (unlike the throttled branch which checks `pageSize.get() < maxPageSize`). The design assumption is that the caller will always supply a `limit`; the safety-check caller does not.

### Impact Explanation
Each open-ended subscription from an unprivileged client generates 12 DB queries (up to 5 000 rows each) in 240 ms via the safety check. `maxConcurrentCallsPerConnection = 5` caps calls per TCP connection, but there is no limit on the number of connections from a single IP, no global subscription cap, and no rate limiting on the gRPC API (unlike the web3 module which has `requestsPerSecond`). An attacker holding *C* connections contributes `C × 5 × 12 = 60C` DB queries every ~240 ms. During a post-partition recovery window — exactly when the database is most sensitive to query load — this amplification can saturate the connection pool, starve legitimate queries, and prevent the database from reaching a healthy state, constituting a sustained denial-of-service against the mirror node's data tier.

### Likelihood Explanation
The gRPC port (default 5600) is publicly reachable. No authentication is required to call `subscribeTopic`. A single attacker machine can open hundreds of TCP connections and register 5 subscriptions each with a trivial gRPC client. The 1-second delay before the safety check fires is not a meaningful barrier; the attacker simply holds connections open. The attack is repeatable: after the 60-second retriever timeout expires, the client can re-subscribe and trigger the burst again. No special knowledge of the codebase is needed — subscribing to any valid topic ID suffices.

### Recommendation
1. **Add an empty-page guard to unthrottled `isComplete()`**: mirror the throttled branch — return `true` when the last page returned fewer rows than `maxPageSize`. This prevents the retriever from exhausting all polls when the DB is returning nothing.
2. **Enforce a global (or per-IP) concurrent-subscription limit** in `TopicMessageServiceImpl` using the existing `subscriberCount` gauge or a semaphore, rejecting new subscriptions with `RESOURCE_EXHAUSTED` when the cap is reached.
3. **Apply a circuit breaker** (e.g., Resilience4j) around `topicMessageRepository.findByFilter()` so that repeated DB failures during a recovery window open the circuit and stop issuing queries rather than retrying indefinitely.
4. **Propagate a limit into the safety-check filter**: when `missingMessages` builds `gapFilter` for the safety check, set a reasonable upper bound (e.g., `unthrottled.getMaxPageSize()`) so `isComplete()` can terminate early.

### Proof of Concept
```
# 1. Open 100 TCP connections; on each, subscribe 5 times to a valid topic
#    with no endTime and no limit (standard open-ended subscription).
for i in $(seq 1 100); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
    mirror-node:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 2. Wait ~1 second. The safety check fires for all 500 subscriptions.
#    Each triggers retrieve(filter, false) → 12 DB queries at 20 ms intervals.
#    Result: 500 × 12 = 6 000 DB queries issued within a 240 ms window.

#    On a recovering database this prevents stabilisation and extends the outage.
