### Title
Unbounded `keys` Collection in Historical Path Causes Large SQL IN Clause Leading to Resource Exhaustion

### Summary
In `HookServiceImpl.getHookStorageChange()`, the full user-supplied `keys` collection is passed directly to `findByKeyInAndTimestampBetween()` without any size cap. Combined with a user-controlled wide timestamp range, this generates a native SQL query with an arbitrarily large `IN` clause against `hook_storage_change`, which can force a full or near-full table scan and drive node resource consumption well above 30%.

### Finding Description

**Exact code path:**

`HookServiceImpl.java` lines 84–118, specifically lines 89–105:

```java
final var keys = request.getKeys();                    // line 89 – unbounded user input
final boolean requestHasKeys = !keys.isEmpty();
final var keysInRange = request.getKeysInRange();      // line 91 – filtered copy, NOT used below

if (keysInRange.isEmpty() && requestHasKeys) {         // line 93 – only exits if ALL keys out of range
    return new HookStorageResult(ownerId, List.of());
}
...
if (requestHasKeys) {
    changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
            ownerId.getId(), hookId, keys,             // line 104 – raw `keys`, not `keysInRange`
            timestampLowerBound, timestampUpperBound, page);
}
``` [1](#0-0) 

**Root cause – two compounding failures:**

1. `HookStorageRequest.keys` has no `@Size` or any other cardinality constraint: [2](#0-1) 

2. The filtered `keysInRange` list is computed (line 91) and used only for the early-exit guard (line 93). The actual repository call on line 104 passes the raw, unfiltered `keys` collection. So even if 9,999 of 10,000 supplied keys are outside the key range, all 10,000 still appear in the SQL `IN` clause.

**Repository query – no server-side guard:**

```sql
select distinct on (key) ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id  = :hookId
  and key in (:keys)                                          -- unbounded IN list
  and consensus_timestamp between :timestampLowerBound        -- user-controlled range
                              and :timestampUpperBound
``` [3](#0-2) 

The `Pageable` argument limits the *returned* rows but does not prevent the database engine from scanning the full matching set before applying `DISTINCT ON` and the page limit.

**Why existing checks are insufficient:**

- The `keysInRange` early-exit (line 93) only fires when *every* supplied key is outside the range. One in-range key bypasses it entirely, yet all keys still reach the database.
- The `limit` field in `HookStorageRequest` (default 25) caps result rows, not the IN-clause width or the timestamp scan range.
- No rate-limiting, query timeout, or maximum-keys validation is present in the DTO or service layer. [4](#0-3) 

### Impact Explanation
A single HTTP request carrying thousands of keys and a maximally wide timestamp range forces PostgreSQL to evaluate a huge IN predicate across a potentially large `hook_storage_change` table. The `DISTINCT ON (key)` clause requires sorting/hashing the full intermediate result set. Repeated requests from one or more clients can saturate CPU and I/O on the database node, degrading or denying service to all other users. The impact is a realistic >30% resource spike per the threat model, achievable without any privileged account.

### Likelihood Explanation
The historical path is triggered by any request that includes a `timestamp` parameter (`isHistorical()` returns `true` when `timestamp` is non-empty). No authentication or special privilege is required. The attacker only needs to know the public REST endpoint, supply a valid `ownerId`/`hookId`, include one in-range key alongside thousands of out-of-range keys (to bypass the early-exit guard), and set a wide timestamp window. The attack is trivially scriptable and repeatable.

### Recommendation
1. **Cap the keys collection size** at the DTO or controller layer (e.g., `@Size(max = 100)` on `HookStorageRequest.keys`).
2. **Pass `keysInRange` instead of `keys`** to `findByKeyInAndTimestampBetween()` (line 104), so the IN clause is already bounded by the intersection with the declared key range.
3. **Enforce a maximum timestamp window** (e.g., reject ranges wider than a configurable threshold).
4. **Set a statement timeout** on the database connection used by the REST service to bound worst-case query duration.

### Proof of Concept
```
# Precondition: know any valid ownerId (e.g. 0.0.1234) and hookId (e.g. 1).
# One key (0x01) is in-range; 9,999 additional arbitrary keys are appended.

GET /api/v1/hooks/0.0.1234/1/storage
    ?timestamp=gte:0&timestamp=lte:9999999999999999999
    &key=0x01
    &key=0xAAAA0001
    &key=0xAAAA0002
    ... (repeat for 10,000 total key= parameters)

# Result: SQL IN clause contains 10,000 entries; consensus_timestamp range spans
# the entire table; DISTINCT ON forces a full sort of all matching rows.
# Repeat in a tight loop from multiple clients to sustain >30% resource increase.
```
