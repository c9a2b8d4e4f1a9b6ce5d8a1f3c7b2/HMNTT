### Title
Unauthenticated Index-Only AccountBalance Requests Force Unbounded Uncached DB Correlated Subqueries, Enabling Resource Exhaustion

### Summary
The `AccountBalance()` endpoint in `rosetta/app/services/account_service.go` accepts a `BlockIdentifier` containing only an `Index` (no Hash) from any unauthenticated caller. This triggers `RetrieveBlock()` → `blockRepo.FindByIndex()`, which executes a correlated SQL subquery against the database on every call with zero application-level caching. Because the Rosetta service has no built-in rate limiting and the optional infrastructure-level middleware is disabled by default, an attacker cycling through sequential block indices can force sustained, unbounded DB load that materially increases node resource consumption.

### Finding Description

**Exact code path:**

1. `AccountBalance()` — `rosetta/app/services/account_service.go` lines 52–53: if `request.BlockIdentifier != nil`, calls `a.RetrieveBlock(ctx, request.BlockIdentifier)` unconditionally. [1](#0-0) 

2. `RetrieveBlock()` — `rosetta/app/services/base_service.go` lines 82–83: when `Hash == nil && Index != nil`, calls `b.blockRepo.FindByIndex(ctx, *bIdentifier.Index)` with no caching layer. [2](#0-1) 

3. `FindByIndex()` — `rosetta/app/persistence/block.go` lines 171–181: validates `index >= 0`, then delegates directly to `findBlockByIndex()` with no result cache. [3](#0-2) 

4. `findBlockByIndex()` — `rosetta/app/persistence/block.go` lines 210–223: executes `selectRecordBlockByIndex`, a **correlated subquery** that performs two index scans on `record_file` per call. [4](#0-3) 

5. The SQL itself — `rosetta/app/persistence/block.go` lines 75–85: the inner `SELECT consensus_start-1 FROM record_file WHERE index = @index + 1` is a correlated subquery executed for every outer row match. [5](#0-4) 

6. After a successful block lookup, `RetrieveBalanceAtBlock()` is also called (line 61), adding a second DB round-trip per request. [6](#0-5) 

**Root cause:** The `blockRepository` caches only the genesis block (via `sync.Once`). All other `FindByIndex` calls hit the database unconditionally. Block indices are sequential integers, trivially enumerable without any prior knowledge. There is no application-level rate limiting in the Rosetta Go service — the Rosetta configuration table lists no throttle/rate-limit properties at all. [7](#0-6) 

**Why existing checks fail:**

The only rate-limiting mechanism is the optional Traefik middleware chain (`inFlightReq: amount: 5`, `rateLimit: average: 10`), but it is **disabled by default** because `global.middleware` defaults to `false`. The middleware template is gated on `{{ if and .Values.global.middleware .Values.middleware }}`, so in a default deployment no middleware is deployed at all. [8](#0-7) [9](#0-8) 

Even when enabled, `rateLimit: average: 10` is keyed on `requestHost` (not source IP), making it trivially bypassable by varying the `Host` header or using multiple source IPs. [10](#0-9) 

The DB connection pool is capped at 100 open connections (`maxOpenConnections: 100`), meaning a sustained flood of index-only requests can saturate the pool and starve legitimate traffic. [11](#0-10) 

### Impact Explanation
Each `POST /account/balance` request with a unique `BlockIdentifier.Index` forces at minimum two uncached DB queries (one correlated subquery for block lookup, one for balance retrieval). An attacker sending requests with sequential indices (0, 1, 2, …) bypasses any result reuse. Sustained at moderate concurrency (e.g., 50–100 parallel connections filling the pool), this can increase DB CPU, I/O, and connection utilization well beyond 30% compared to baseline, degrading or denying service to legitimate users. The Rosetta endpoint is publicly accessible with no authentication.

### Likelihood Explanation
The attack requires zero privileges, zero prior knowledge (block indices are sequential integers), and only a standard HTTP client. The endpoint is publicly exposed via ingress at `/rosetta/account`. In a default deployment, no rate limiting is active. The attack is trivially repeatable and scriptable, and can be sustained indefinitely.

### Recommendation
1. **Add application-level rate limiting** inside the Rosetta Go service (e.g., using `golang.org/x/time/rate` or a middleware like `go-chi/httprate`) so it is enforced regardless of infrastructure configuration.
2. **Cache `FindByIndex` results** in `blockRepository` using a bounded LRU cache (similar to the entity cache already used in `blockAPIService`), keyed on block index, with a short TTL to avoid stale data.
3. **Enable middleware by default** (`global.middleware: true`) and change `rateLimit.sourceCriterion` from `requestHost` to `ipStrategy` to prevent trivial bypass.
4. **Validate block index range** against the known latest block index before issuing a DB query, returning an early error for out-of-range indices without touching the DB.

### Proof of Concept
```bash
# Enumerate sequential block indices, each forcing a fresh DB correlated subquery
# No authentication, no special headers required
for i in $(seq 1 500); do
  curl -s -X POST http://<rosetta-host>/account/balance \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"account_identifier\": {\"address\": \"0.0.2\"},
      \"block_identifier\": {\"index\": $i}
    }" &
done
wait
# Each concurrent request hits FindByIndex() with a distinct index,
# forcing 500 uncached correlated subqueries against record_file simultaneously.
# Monitor DB CPU/connection utilization to observe >30% increase vs. baseline.
```

```go
	}
```

```go
	}
```

```go
func (br *blockRepository) findBlockByIndex(ctx context.Context, index int64) (*types.Block, *rTypes.Error) {
	if index < br.genesisBlock.Index {
		return nil, hErrors.ErrBlockNotFound
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectRecordBlockByIndex, sql.Named("index", index)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	return rb.ToBlock(br.genesisBlock), nil
```
