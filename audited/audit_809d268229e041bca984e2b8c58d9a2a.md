### Title
Unbounded Exception Message Logging via `ERROR_EXCEPTION_ATTRIBUTE` Allows Log Storage Griefing

### Summary
`LoggingFilter.getMessage()` in `rest-java` reads the raw exception stored in `ERROR_EXCEPTION_ATTRIBUTE` by `GenericControllerAdvice.handleExceptionInternal()` and calls `ex.getMessage()` with no length truncation before writing it to the log. Because Spring exception messages (e.g., `MethodArgumentTypeMismatchException`) embed the user-supplied value verbatim, an unprivileged attacker can craft requests whose exception messages are as large as the server's URL/body size limit, generating arbitrarily oversized log entries at will.

### Finding Description

**Code path:**

`GenericControllerAdvice.handleExceptionInternal()` unconditionally stores the original exception object in the request attribute `ERROR_EXCEPTION_ATTRIBUTE` at line 156:

```java
// rest-java/.../controller/GenericControllerAdvice.java:156
request.setAttribute(WebUtils.ERROR_EXCEPTION_ATTRIBUTE, ex, SCOPE_REQUEST);
```

This happens for every exception handler, including `typeMismatch` (line 87–92), `badRequest` (line 104–106), `bindException` (line 95–97), and `defaultExceptionHandler` (line 109–113).

`LoggingFilter.getMessage()` then reads that attribute and calls `ex.getMessage()` with zero truncation:

```java
// rest-java/.../config/LoggingFilter.java:67-69
if (request.getAttribute(ERROR_EXCEPTION_ATTRIBUTE) instanceof Exception ex) {
    return ex.getMessage();
}
```

The returned string is placed directly into the log call:

```java
// rest-java/.../config/LoggingFilter.java:51,56-59
var params = new Object[] {request.getRemoteAddr(), request.getMethod(), uri, elapsed, status, message};
...
log.info(LOG_FORMAT, params);   // or log.warn(...)
```

**Root cause / failed assumption:**

The developer was aware that user input must not appear in logs — the `typeMismatch` handler carries the comment *"Explicitly don't log the parameter value to avoid cross-site scripting attacks"* (line 88) and constructs a sanitized `ProblemDetail` for the HTTP response. However, `handleExceptionInternal` still stores the *original* exception (not the sanitized body) in `ERROR_EXCEPTION_ATTRIBUTE` (line 156), and `LoggingFilter.getMessage()` reads that original exception and logs its raw message without any size cap.

Spring's `MethodArgumentTypeMismatchException.getMessage()` returns a string of the form:
```
Failed to convert value of type 'String' to required type 'Long'; For input string: "<USER_VALUE>"
```
The `<USER_VALUE>` portion is the verbatim attacker-controlled string.

### Impact Explanation

Every HTTP request that triggers a type-mismatch, bind, or `IllegalArgumentException` produces one log line whose size is proportional to the attacker-supplied value. With Tomcat's default 8 KB URL limit, a single request can produce a ~8 KB log entry. Sending thousands of such requests per second (no authentication required) can exhaust disk space allocated to log storage, fill log-shipping pipelines, or degrade log-aggregation systems. No economic damage to network users occurs, but service observability and operator storage are impacted.

### Likelihood Explanation

The attack requires no credentials, no special knowledge of the API, and no rate-limit bypass — only the ability to send HTTP GET/POST requests with long query-parameter values to any endpoint that performs type conversion (e.g., any endpoint accepting a numeric or UUID path/query parameter). It is trivially scriptable and repeatable.

### Recommendation

Truncate the exception message before logging in `getMessage()`:

```java
private static final int MAX_MSG_LEN = 512;

private String getMessage(HttpServletRequest request, Exception e) {
    if (e != null) {
        return StringUtils.truncate(e.getMessage(), MAX_MSG_LEN);
    }
    if (request.getAttribute(ERROR_EXCEPTION_ATTRIBUTE) instanceof Exception ex) {
        return StringUtils.truncate(ex.getMessage(), MAX_MSG_LEN);
    }
    return SUCCESS;
}
```

Additionally, `handleExceptionInternal` should store a sanitized wrapper or a fixed-length summary rather than the raw exception when the exception message may contain user input.

### Proof of Concept

```bash
# Assume any endpoint that accepts a numeric parameter, e.g. /api/v1/accounts/{id}
# Send a request with a very long non-numeric value for a Long/UUID parameter:

LONG_VALUE=$(python3 -c "print('A' * 7000)")

# Single request produces ~7 KB log entry:
curl "https://<host>/api/v1/accounts/$LONG_VALUE"

# Flood to exhaust log storage:
for i in $(seq 1 10000); do
  curl -s "https://<host>/api/v1/accounts/$LONG_VALUE" &
done
wait
```

The server logs a line matching `LOG_FORMAT` containing the full 7 000-character string for every request, with no truncation applied anywhere in `LoggingFilter.getMessage()` or `logRequest()`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    protected ResponseEntity<Object> handleExceptionInternal(
            Exception ex, @Nullable Object body, HttpHeaders headers, HttpStatusCode statusCode, WebRequest request) {

        Error errorResponse =
                switch (ex) {
                    case BindException e -> bindExceptionResponse(e);
                    case Errors errors -> errorResponse(errors.getAllErrors());
                    case MethodValidationResult errors -> errorResponse(errors.getAllErrors());
                    default -> {
                        var message =
                                statusCode instanceof HttpStatus hs ? hs.getReasonPhrase() : statusCode.toString();
                        var detail = body instanceof ProblemDetail pb ? pb.getDetail() : ex.getMessage();
                        var nonSensitiveDetail = !statusCode.is5xxServerError() ? detail : StringUtils.EMPTY;
                        yield errorResponse(message, nonSensitiveDetail);
                    }
                };

        request.setAttribute(WebUtils.ERROR_EXCEPTION_ATTRIBUTE, ex, SCOPE_REQUEST);
        return new ResponseEntity<>(errorResponse, headers, statusCode);
