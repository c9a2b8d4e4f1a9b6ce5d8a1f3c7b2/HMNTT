### Title
Unbounded Prometheus Label Cardinality via Arbitrary HTTP Method Strings in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates to the weaveworks `middleware.Instrument`, which uses `r.Method` directly as the `method` label value in all four Prometheus metric vectors. Go's `net/http` server accepts any RFC 7230 token string as an HTTP method without a length restriction (bounded only by the 1 MB default header buffer). An unauthenticated attacker can flood the server with requests bearing unique, arbitrarily long method strings, creating a new Prometheus time series per unique value and causing unbounded heap growth in the process.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–84 — `MetricsMiddleware` constructs a `middleware.Instrument` struct and calls `.Wrap(next)`. The weaveworks `Instrument.Wrap` implementation (public source: `github.com/weaveworks/common/middleware/instrument.go`) calls `r.Method` directly when invoking `.WithLabelValues(r.Method, route, ...)` on every histogram and gauge:

```
requestBytesHistogram    → labels: {method=r.Method, route}
requestDurationHistogram → labels: {method=r.Method, route, status_code, ws}
requestInflightGauge     → labels: {method=r.Method, route}
responseBytesHistogram   → labels: {method=r.Method, route}
```

**Root cause:** No normalization, allowlist check, or length cap is applied to `r.Method` before it is used as a Prometheus label. The failed assumption is that only standard HTTP verbs (GET, POST, …) will appear in `r.Method`.

**Exploit flow:**
1. Go's `net/http` parser accepts any sequence of valid token characters as the HTTP method (RFC 7230 §3.1.1). The only practical upper bound is the server's `ReadHeaderBytes` / header buffer (~1 MB).
2. The middleware chain is `corsMiddleware → tracingMiddleware → metricsMiddleware → router`. The metrics middleware fires **before** the router rejects the unknown method, so every request — matched or not — is instrumented.
3. For a request whose method does not match any registered route, the weaveworks `RouteMatcher` returns the fallback label `"other"` for `route`. The `method` label is still the raw attacker-supplied string.
4. `prometheus.NewHistogramVec.WithLabelValues(...)` allocates a new `*histogram` object and stores it in an internal `sync.Map` keyed by the full label tuple. This allocation is never freed during the process lifetime.

**Why existing checks are insufficient:**
- `ReadHeaderTimeout` limits connection duration but does not restrict method length or cardinality.
- The CORS middleware (`server.CorsMiddleware`) does not reject non-standard methods.
- The coinbase rosetta-sdk-go router rejects unknown methods with 405, but this happens *after* the metrics middleware has already recorded the label.
- There is no method allowlist or string-length guard anywhere in the middleware stack.

### Impact Explanation
Each unique method string permanently allocates one `*histogram` (or `*gauge`) object per metric vector, plus the label string itself, in the Prometheus default registry. With four metric vectors and six duration buckets, each unique method string consumes roughly:

- 4 vectors × (8 float64 counters/buckets + label map overhead) ≈ several hundred bytes of Go heap per unique method, **plus** the full method string stored as a map key.

Sending 10 000 requests with 1 KB unique method strings consumes on the order of tens of megabytes; scaling to millions of requests or kilobyte-length strings can exhaust available memory and crash the process (OOM). Because Prometheus never evicts label combinations, the damage is cumulative across the process lifetime. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation
The attack requires only the ability to open a TCP connection to the Rosetta port and send a raw HTTP/1.1 request line with a custom method — trivially achievable with `nc`, `curl --request`, or any HTTP library. No credentials, tokens, or prior knowledge of the API are needed. The attack is fully repeatable and automatable, and a single script can generate millions of unique method strings in minutes.

### Recommendation
1. **Allowlist HTTP methods** at the earliest possible point in the middleware chain (before `MetricsMiddleware`). Reject any request whose method is not in `{GET, POST, PUT, DELETE, OPTIONS, HEAD}` with HTTP 405 before metrics are recorded.
2. **Normalize the label value** inside `MetricsMiddleware`: replace any method not in the known set with a fixed sentinel string (e.g., `"unknown"`) before passing it to Prometheus:
   ```go
   func sanitizeMethod(m string) string {
       switch strings.ToUpper(m) {
       case "GET", "POST", "PUT", "DELETE", "OPTIONS", "HEAD", "PATCH":
           return strings.ToUpper(m)
       default:
           return "unknown"
       }
   }
   ```
3. **Set `http.Server.MaxHeaderBytes`** explicitly to a low value (e.g., 8 KB) to limit the maximum method string length that Go will parse.

### Proof of Concept
```bash
# Send 10 000 requests with unique method strings to the Rosetta endpoint
for i in $(seq 1 10000); do
  METHOD=$(head -c 512 /dev/urandom | tr -dc 'A-Z' | head -c 512)
  printf "${METHOD} /network/list HTTP/1.1\r\nHost: target\r\nContent-Length: 0\r\n\r\n" \
    | nc -q1 <rosetta-host> <rosetta-port> &
done
wait

# Observe heap growth via /metrics endpoint:
curl http://<rosetta-host>:<rosetta-port>/metrics | grep go_memstats_heap_alloc
```

Each iteration creates a new Prometheus time series keyed on the unique method string. Heap allocation grows monotonically and is never reclaimed.
