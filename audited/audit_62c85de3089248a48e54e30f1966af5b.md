### Title
Unauthenticated Single Request Triggers 3-Subquery UNION Scan on `contract_log` Table via `getContractLogsById`

### Summary
An unprivileged external user can send a single crafted GET request to `/api/v1/contracts/:contractId/results/logs` with both `index` and `timestamp` range parameters that pass all validation checks and cause `extractContractLogsMultiUnionQuery()` to produce three non-empty filter sets (lower, inner, upper). `ContractService.getContractLogsQuery()` then builds and executes three independent UNION subqueries against the `contract_log` table per HTTP request, with no application-level rate limiting on this endpoint.

### Finding Description

**Exact code path:**

1. Route `/:contractId/results/logs` → `getContractLogsById()` (`rest/controllers/contractController.js`, line 779) [1](#0-0) 

2. `acceptedContractLogsByIdParameters` accepts `INDEX` and `TIMESTAMP` filters (lines 1317–1326): [2](#0-1) 

3. `extractContractLogsMultiUnionQuery()` parses the filters into `bounds.primary` (timestamp) and `bounds.secondary` (index) (lines 595–698): [3](#0-2) 

4. `validateContractLogsBounds()` (lines 544–549) calls `validateBounds()`. For the payload `index=gte:0&index=lte:999999&timestamp=gte:0&timestamp=lte:99999999999`:
   - `validateBoundsRange`: neither bound has both range AND equal → passes
   - `validateLowerBounds`: `primary.lower.operator` is `gte` (not `gt`), so the guard `(!primary.hasLower() || primary.lower.operator === gt)` is **false** → no error
   - `validateUpperBounds`: `primary.upper.operator` is `lte` (not `lt`), so the guard `(!primary.hasUpper() || primary.upper.operator === lt)` is **false** → no error [4](#0-3) 

5. With both bounds having lower+upper, `getLowerFilters()` returns `[timestamp=0, index>=0]`, `getInnerFilters()` returns `[timestamp>0, timestamp<99999999999]`, and `getUpperFilters()` returns `[timestamp=99999999999, index<=999999]` — all three non-empty: [5](#0-4) 

6. `ContractService.getContractLogsQuery()` (lines 335–365) builds three independent `SELECT … FROM contract_log` subqueries joined with `UNION`, each with its own `LIMIT` clause, and executes them as a single compound query: [6](#0-5) 

**Root cause:** `validateLowerBounds` and `validateUpperBounds` only reject the combination when the primary bound uses strict operators (`gt`/`lt`). When `gte`/`lte` are used for both timestamp bounds alongside both index bounds, validation passes and all three UNION branches are populated. The inner subquery (`timestamp > 0 AND timestamp < 99999999999`) scans virtually the entire `contract_log` table for the given contract.

**No application-level rate limiting exists for this endpoint.** The throttle/rate-limit code (`ThrottleManagerImpl`, `ThrottleProperties`) is in the `web3` Java module and only applies to `POST /contracts/call`: [7](#0-6) 

The REST Node.js router has no per-IP or per-endpoint rate limiting middleware for this route.

### Impact Explanation
Each crafted request causes the database to execute three full scans of `contract_log` for the target contract instead of one, multiplying DB CPU and I/O by approximately 3x per request. The inner subquery (`timestamp > 0 AND timestamp < 99999999999`) is effectively unbounded and will scan all rows for that contract. An attacker sending a sustained stream of such requests (no authentication required, no rate limit) can increase database processing load by well over 30% compared to normal traffic, potentially causing query queue buildup, connection pool exhaustion, and degraded response times for all users.

### Likelihood Explanation
The attack requires zero privileges — it is a plain unauthenticated HTTP GET request. The parameter combination is trivially discoverable from the public OpenAPI spec (`rest/api/v1/openapi.yml`, lines 781–788) which documents `index=gte:X` and `index=lte:X` as supported operators. The request is repeatable at will with no per-IP throttle on the REST API layer. A single attacker with a script loop can sustain the amplified load indefinitely. [8](#0-7) 

### Recommendation
1. **Add application-level rate limiting** to the REST API for the `/:contractId/results/logs` endpoint (e.g., express-rate-limit middleware keyed by IP).
2. **Restrict the UNION path**: require that when both `index` lower and upper bounds are present, the timestamp range must be narrow (e.g., enforce a maximum timestamp span), or reject the combination that produces all three UNION branches.
3. **Add a DB statement timeout** for this query class to bound worst-case execution time.
4. **Tighten `validateUpperBounds`/`validateLowerBounds`** to also reject `gte`/`lte` combinations on the secondary bound when the primary bound is also a full range, unless the timestamp range is sufficiently narrow.

### Proof of Concept
```
# No authentication required
curl "https://<mirror-node-host>/api/v1/contracts/0.0.1234/results/logs?index=gte:0&index=lte:999999&timestamp=gte:0&timestamp=lte:99999999999"
```

This single request causes `extractContractLogsMultiUnionQuery` to return:
- `lower = [timestamp=0, index>=0]`
- `inner = [timestamp>0, timestamp<99999999999]`
- `upper = [timestamp=99999999999, index<=999999]`

`ContractService.getContractLogsQuery` then builds and executes:
```sql
(SELECT ... FROM contract_log cl WHERE cl.contract_id=$1 AND cl.index>=$4 AND cl.consensus_timestamp=$5 ORDER BY ... LIMIT $3)
UNION
(SELECT ... FROM contract_log cl WHERE cl.contract_id=$1 AND cl.consensus_timestamp>$6 AND cl.consensus_timestamp<$7 ORDER BY ... LIMIT $3)
UNION
(SELECT ... FROM contract_log cl WHERE cl.contract_id=$1 AND cl.index<=$8 AND cl.consensus_timestamp=$9 ORDER BY ... LIMIT $3)
ORDER BY consensus_timestamp desc, index desc LIMIT $3
```

Repeat in a loop to sustain amplified DB load. Confirmed by the positive test case `'index > & timestamp >= & timestamp <= & index <'` in the test suite at line 1256–1282. [9](#0-8)

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
