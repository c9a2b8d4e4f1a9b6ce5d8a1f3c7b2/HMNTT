### Title
Deleted Topic Exposes Stale Custom Fees via `GET /api/v1/topics/{id}`

### Summary
When a topic is deleted via `ConsensusDeleteTopicTransactionHandler`, only the entity's `deleted` flag is set to `true`; no corresponding empty `custom_fee` record is written to clear the fee schedule. Because `CustomFeeServiceImpl.findById()` queries the `custom_fee` table unconditionally and `TopicMapper` maps the result without inspecting the entity's `deleted` field, any unprivileged caller of `GET /api/v1/topics/{id}` receives a response where `deleted: true` coexists with a populated `custom_fees` block containing the pre-deletion fee schedule.

### Finding Description

**Importer side — deletion handler does not clear custom fees**

`ConsensusDeleteTopicTransactionHandler.doUpdateEntity()` only marks the entity deleted:

```java
// ConsensusDeleteTopicTransactionHandler.java lines 28-31
protected void doUpdateEntity(Entity entity, RecordItem recordItem) {
    entity.setType(EntityType.TOPIC);
    entityListener.onEntity(entity);   // sets deleted=true, nothing else
}
```

No call to `entityListener.onCustomFee()` is made, so the `custom_fee` table row for the topic retains its last fee schedule unchanged.

**API side — custom fee fetched and mapped without deleted check**

`getTopic()` calls all three services independently and passes results straight to the mapper:

```java
// TopicController.java lines 32-37
Topic getTopic(@PathVariable EntityIdNumParameter id) {
    var topic   = topicService.findById(id.id());
    var entity  = entityService.findById(id.id());
    var customFee = customFeeService.findById(id.id());   // no deleted guard
    return topicMapper.map(customFee, entity, topic);
}
```

`CustomFeeServiceImpl.findById()` performs a plain repository lookup with no awareness of the entity's lifecycle:

```java
// CustomFeeServiceImpl.java lines 19-23
public CustomFee findById(EntityId id) {
    return customFeeRepository
            .findById(id.getId())
            .orElseThrow(() -> new EntityNotFoundException("Custom fee for entity not found"));
}
```

`TopicMapper` maps the `customFee` argument directly to `customFees` in the response with no conditional logic:

```java
// TopicMapper.java line 16
@Mapping(source = "customFee", target = "customFees")
Topic map(CustomFee customFee, Entity entity, org.hiero.mirror.common.domain.topic.Topic topic);
```

**Exploit flow**

1. Topic `0.0.X` is created with a non-empty fixed fee schedule → `custom_fee` row inserted.
2. Topic `0.0.X` is deleted → `entity.deleted = true`; `custom_fee` row untouched.
3. Attacker (no credentials required) calls `GET /api/v1/topics/0.0.X`.
4. Response: `"deleted": true` AND `"custom_fees": { "fixed_fees": [{ "amount": 100, ... }] }`.

### Impact Explanation
The mirror node is the authoritative read layer for the Hedera network state. Returning a populated `custom_fees` block for a deleted topic contradicts the `deleted: true` flag in the same response. Any downstream application or wallet that reads `custom_fees` without also checking `deleted` will believe the fee schedule is still active, potentially causing incorrect fee calculations, erroneous user-facing warnings, or flawed smart-contract integrations that rely on mirror-node data. The severity is medium: no funds are directly at risk, but the exported record is factually wrong and observable by anyone.

### Likelihood Explanation
Exploitation requires zero privileges — a plain unauthenticated HTTP GET to a public mirror-node endpoint. The condition is triggered automatically for every topic that (a) had custom fees set and (b) was subsequently deleted. No race condition or timing window is involved; the stale data persists indefinitely after deletion.

### Recommendation
Two complementary fixes are needed:

1. **Importer**: In `ConsensusDeleteTopicTransactionHandler.doUpdateEntity()`, after calling `entityListener.onEntity(entity)`, insert a new `CustomFee` record with an empty `fixedFees` list (mirroring what `ConsensusUpdateTopicTransactionHandler.updateCustomFee()` does when fees are cleared) so the `custom_fee` table reflects the post-deletion state.

2. **API / Mapper**: In `TopicController.getTopic()` (or in `TopicMapper`), when `entity.getDeleted() == true`, substitute an empty `ConsensusCustomFees` object (or `null`) for the `customFees` field instead of passing the stale `CustomFee` from the repository.
