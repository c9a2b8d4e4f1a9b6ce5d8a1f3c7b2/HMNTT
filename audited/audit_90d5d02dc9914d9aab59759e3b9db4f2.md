### Title
Cache Neutralization via Unique `timestamp=eq` Values in `doGetTransactions()`

### Summary
In `doGetTransactions()`, the Redis cache is keyed on `consensus_timestamp` values returned from the database, not on request-level parameters. An unprivileged attacker can send requests with a distinct `timestamp=eq:X` value per request, ensuring every request maps to a unique cache key and always triggers the expensive `getTransactionsDetails` loader (a multi-CTE, multi-table DB query). This completely neutralizes the caching layer for the detail-fetch step.

### Finding Description

**Code path:**

`rest/transactions.js`, `doGetTransactions()`, lines 695–723:

```
getTransactionTimestamps(filters, timestampRange)   // line 701 — always hits DB, no cache
  → returns payerAndTimestamps rows

cache.get(payerAndTimestamps, loader, keyMapper)     // line 705
```

`keyMapper` (lines 679–685) maps each row to `transaction:${consensus_timestamp}`. The cache key is derived from the actual `consensus_timestamp` value returned by the DB, which directly corresponds to the user-supplied `timestamp=eq:X` filter value.

**Root cause:** There is no request-level cache (e.g., keyed on the full query string or a normalized filter set). The only cache is a per-transaction-record cache keyed by `consensus_timestamp`. Because each unique `timestamp=eq:X` value produces a unique cache key, an attacker who cycles through distinct real timestamps will always produce a Redis miss, causing `loader` (i.e., `getTransactionsDetails`, lines 616–663) to be invoked on every request.

**Why existing checks are insufficient:**

- Line 452 checks `timestampRange.eqValues.length > 1` — this only blocks multiple `eq` operators within a *single* request. It does nothing to prevent unique `eq` values across *different* requests.
- `getTransactionTimestamps` (line 483) always executes a DB query unconditionally — there is no caching at this stage either.
- `cache.get()` (cache.js line 96) short-circuits only on empty keys. With a real timestamp that matches a transaction, keys are non-empty and the full Redis lookup + loader path executes.

**Two DB queries fire per request when cache is bypassed:**
1. `getTransactionTimestamps` → always (no cache)
2. `getTransactionsDetails` (loader) → always on cache miss (unique timestamp per request)

### Impact Explanation
`getTransactionsDetails` (lines 634–662) is a heavy query: two CTEs (`c_list`, `t_list`) each joining `crypto_transfer` and `token_transfer` by `payer_account_id` and `consensus_timestamp`, plus a final join against the `transaction` table. Forcing this query on every request bypasses the only DB-load-reduction mechanism for this endpoint. Under sustained attack, this can saturate DB connection pools and degrade response times for all users of the mirror node REST API. The impact is griefing/availability degradation with no economic damage to network participants.

### Likelihood Explanation
No authentication or privilege is required. Transaction timestamps are public — they appear in every API response. An attacker can harvest a large set of real `consensus_timestamp` values from a single paginated scan of `/api/v1/transactions`, then replay them in a loop with `timestamp=eq:X` to guarantee perpetual cache misses. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond basic API usage.

### Recommendation
1. **Add a request-level cache** (e.g., keyed on the normalized query string or a hash of the filter set) in front of `doGetTransactions()`, so identical or equivalent queries are served from cache regardless of whether the underlying transaction records are cached.
2. **Rate-limit** the `/api/v1/transactions` endpoint per IP/client to bound the maximum DB query rate.
3. **Cache `getTransactionTimestamps` results** as well (not just `getTransactionsDetails`), since it unconditionally hits the DB on every request.
4. Consider enforcing a minimum timestamp range width for `eq` queries or throttling `eq`-operator requests more aggressively than range queries.

### Proof of Concept

```bash
# Step 1: Harvest real consensus_timestamps from the public API
curl "https://<mirror-node>/api/v1/transactions?limit=100&order=desc" \
  | jq -r '.transactions[].consensus_timestamp' > timestamps.txt

# Step 2: Replay each unique timestamp as an eq filter in a tight loop
while read ts; do
  curl -s "https://<mirror-node>/api/v1/transactions?timestamp=eq:${ts}" > /dev/null &
done < timestamps.txt
wait

# Each request maps to a unique cache key `transaction:<ts>`.
# Redis will miss on every request (assuming timestamps are not already cached
# or the attacker uses timestamps not yet in cache).
# getTransactionsDetails (multi-CTE query) fires on every single request,
# fully neutralizing the Redis caching layer for the detail-fetch step.
```
