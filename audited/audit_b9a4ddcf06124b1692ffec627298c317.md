### Title
Unsynchronized Cache-Aside Pattern in `SystemFileLoader.load()` Enables Thundering Herd DB Amplification

### Summary
`SystemFileLoader.load(FileID, long)` implements a manual cache-aside pattern using Spring's `Cache` abstraction over Caffeine without any synchronization between the cache miss check and the DB load. When the cache entry expires, N concurrent requests for the same system file key all observe a cache miss simultaneously, each independently executing a `fileDataRepository.getFileAtTimestamp()` DB query, multiplying DB load by N. No privilege is required to trigger this — any caller of the public JSON-RPC/web3 API suffices.

### Finding Description

**Exact code path** — `web3/src/main/java/org/hiero/mirror/web3/state/SystemFileLoader.java`, lines 129–139:

```java
var file = cache.get(cacheKey, File.class);   // line 129 — simple non-atomic lookup
if (file != null) {
    return file;
}
// The value was not in cache -> try to load from DB
var result = loadFromDB(fileId, consensusTimestamp);  // line 135 — DB query, no lock held
if (result != null) {
    log.info("Updating cache for key {}", cacheKey);
    cache.put(cacheKey, result);   // line 138 — store result, no lock held
}
```

**Root cause** — Spring's `Cache.get(key, type)` (backed by `CaffeineCacheManager`) delegates to Caffeine's `Cache.getIfPresent(key)`, which is a plain non-atomic read. It is categorically different from Caffeine's `Cache.get(key, mappingFunction)`, which serialises concurrent loaders for the same key. The code never uses `computeIfAbsent` semantics, holds no lock between lines 129 and 138, and has no per-key in-flight deduplication.

**Cache configuration** — both caches are `CaffeineCacheManager` instances configured via `EvmConfiguration.java` lines 137–151:
- `CACHE_MANAGER_SYSTEM_FILE`: `expireAfterWrite=10m,maximumSize=20` (address book, throttle definitions, HAPI permissions)
- `CACHE_MANAGER_EXCHANGE_RATES_SYSTEM_FILE`: `expireAfterWrite=60m,maximumSize=20` (exchange rates, fee schedules)

**Exploit flow:**
1. Attacker waits for (or observes via timing) a cache expiry boundary (every 10 or 60 minutes — fully predictable).
2. Attacker sends N concurrent `eth_call` / `eth_estimateGas` requests that each trigger a system file lookup for the same `CacheKey(fileId, roundedTimestamp)`.
3. All N threads reach line 129 before any of them reaches line 138; all observe `null`.
4. All N threads independently call `loadFromDB()` → `loadWithRetry()` → `fileDataRepository.getFileAtTimestamp()`.
5. Each `loadWithRetry` may itself retry up to `maxFileAttempts` times (default 2) on parse failure, so worst-case DB queries = N × `maxFileAttempts`.
6. All N threads then call `cache.put()`, redundantly writing the same value N times.

**Why existing checks are insufficient:**
- The `cache == null` guard at line 124 only handles a missing cache manager, not concurrent access.
- The `genesisNetworkProperties` fast-path at line 107 only covers the network-properties file; all other system files go through the racy path.
- Caffeine's thread-safety guarantees apply to individual cache operations, not to the compound read-check-load-write sequence here.

### Impact Explanation
Every EVM query that touches fee schedules, exchange rates, address books, or throttle definitions calls `SystemFileLoader.load()`. At each cache expiry boundary, a burst of N concurrent requests produces N simultaneous `SELECT` queries against the `file_data` table. With a moderate concurrency of 50–100 requests (easily achievable on a public endpoint), DB CPU spikes by 50–100× for the duration of the burst. Because the exchange-rate cache expires every 60 minutes and the system-file cache every 10 minutes, the attack surface recurs continuously and predictably. The `loadWithRetry` multiplier (up to `maxFileAttempts` queries per call) further amplifies the effect. This directly satisfies the ≥30% DB CPU increase threshold without any brute-force volume requirement — a single coordinated burst at expiry time is sufficient.

### Likelihood Explanation
No authentication or special privilege is required; the web3 JSON-RPC endpoint is publicly accessible. Cache expiry times are fixed and deterministic (10 min / 60 min), so an attacker can schedule bursts precisely. The attack is repeatable indefinitely at each expiry cycle. Standard HTTP concurrency tools (e.g., `wrk`, `ab`, `hey`) are sufficient to generate the required concurrent load. The attacker does not need to know internal state — only that the endpoint processes EVM calls that read system files.

### Recommendation
Replace the manual check-then-load-then-put pattern with Caffeine's atomic `get(key, mappingFunction)` via Spring's `Cache.get(key, valueLoader)` overload, which serialises concurrent loaders for the same key internally:

```java
// Replace lines 129–139 with:
return cache.get(cacheKey, () -> loadFromDB(fileId, consensusTimestamp));
```

This delegates to `CaffeineCache.get(Object key, Callable<T> valueLoader)`, which uses Caffeine's internal per-key locking to ensure only one thread executes the loader while others wait for the result — eliminating the thundering herd entirely. If a `null` result must be excluded from caching, wrap with an explicit null check inside the loader or use a sentinel value.

### Proof of Concept

**Preconditions:**
- Public web3 JSON-RPC endpoint is accessible.
- At least one system file (e.g., exchange rate file) has been cached and its 60-minute TTL is about to expire (or has just expired).

**Steps:**
1. Determine the approximate cache population time (first request after node start, or after the last expiry).
2. At T+60min (exchange rate cache expiry) or T+10min (system file cache expiry), fire N=50 concurrent `eth_estimateGas` requests:
   ```bash
   hey -n 50 -c 50 -m POST \
     -H "Content-Type: application/json" \
     -d '{"jsonrpc":"2.0","method":"eth_estimateGas","params":[{"to":"0x0000000000000000000000000000000000000167"}],"id":1}' \
     http://<mirror-node-web3>/api/v1/contracts/call
   ```
3. Monitor DB CPU (e.g., via `pg_stat_activity` or Prometheus metrics) during the burst window.

**Expected result:** 50 simultaneous `getFileAtTimestamp` queries appear in `pg_stat_activity` for the same `file_id` and `consensus_timestamp`, causing a measurable DB CPU spike proportional to N, compared to the single query that would occur with a properly serialised loader.
