### Title
Rate-Limit Token Leak on Gas-Check Failure Enables Sustained DoS Against Legitimate Callers

### Summary
In `ThrottleManagerImpl.throttle()`, a rate-limit token is irrevocably consumed from `rateLimitBucket` before the gas check is attempted. When `gasLimitBucket.tryConsume()` returns `false`, the already-consumed rate-limit token is never restored. An unprivileged attacker can exploit this to drain the entire rate-limit budget with requests that never consume any gas budget, permanently blocking legitimate callers for each refill window.

### Finding Description
**Exact code path**: `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, lines 38â€“42:

```java
if (!rateLimitBucket.tryConsume(1)) {           // line 38 â€“ token consumed here
    throw new ThrottleException(...);
} else if (!gasLimitBucket.tryConsume(          // line 40 â€“ may fail
        throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(...);            // line 41 â€“ rate token NOT restored
}
```

**Root cause**: The two-step check is not atomic as a compound operation. `rateLimitBucket.tryConsume(1)` succeeds and permanently deducts a token. If the subsequent `gasLimitBucket.tryConsume()` fails, execution jumps to line 41 and throws â€” but the rate-limit token is gone. The `restore()` method (lines 59â€“64) only adds tokens back to `gasLimitBucket`; it has no corresponding path for `rateLimitBucket`.

**`ThrottleProperties.scaleGas()`** (line 42â€“47 of `ThrottleProperties.java`) returns `0` for `gas â‰¤ 10 000`, meaning `gasLimitBucket.tryConsume(0)` always succeeds. For `gas > 10 000` the scaled value is `gas / 10 000`. With the default `gasPerSecond = 7 500 000 000` (scaled to 750 000 bucket tokens) and `requestsPerSecond = 500`:

**Exploit flow**:
1. Attacker sends 1 request with `gas = 7 500 000 000` â†’ passes rate check (âˆ’1 token), passes gas check (âˆ’750 000 tokens, gas bucket now empty).
2. Attacker sends 499 more requests with any `gas > 10 000` â†’ each passes rate check (âˆ’1 token each), each fails gas check (bucket empty), rate token NOT restored.
3. After 500 requests: `rateLimitBucket` is fully drained; `gasLimitBucket` is fully drained.
4. Every legitimate request now receives `REQUEST_PER_SECOND_LIMIT_EXCEEDED` for the remainder of the 1-second window.

Without this bug, step 2 would not drain rate-limit tokens (they would be restored), leaving 499 tokens available for legitimate callers.

**Existing checks are insufficient**: `ContractController.call()` calls `throttleManager.restore(request.getGas())` only on `InvalidParametersException` (line 48), not on `ThrottleException`. There is no code path that restores a rate-limit token under any condition.

### Impact Explanation
An unprivileged attacker can continuously exhaust the `rateLimitBucket` each second using only `requestsPerSecond` (default: 500) cheap HTTP requests, while consuming negligible gas budget after the first request. All legitimate `/api/v1/contracts/call` traffic is rejected with HTTP 429 for the duration of each window. Because the mirror node's web3 API is the interface through which contract-call results are exported, sustained blocking of this endpoint directly causes missing or delayed records â€” matching the stated scope of the finding.

### Likelihood Explanation
No authentication or special privilege is required. Any internet-accessible deployment is reachable. The attack is trivially scriptable (e.g., `ab`, `wrk`, or a simple loop), requires no knowledge of contract addresses, and is repeatable every second indefinitely. The default rate limit of 500 req/s is easily achievable from a single machine or a small botnet.

### Recommendation
Restore the rate-limit token when the gas check fails. The simplest fix is to add `rateLimitBucket.addTokens(1)` before throwing in the gas-failure branch:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);   // restore the token â€” request was not served
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, check the gas bucket first (without consuming) using `gasLimitBucket.getAvailableTokens()` as a pre-flight guard, or restructure the check to be atomic by using a single composite bucket or a synchronized block covering both `tryConsume` calls.

### Proof of Concept

```bash

```
