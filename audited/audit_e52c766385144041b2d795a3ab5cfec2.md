### Title
Unbounded `node.id` Parameter List in `getNetworkNodes()` Enables Database Connection Pool Exhaustion

### Summary
The `getNetworkNodes()` method in `NetworkServiceImpl` accepts an unbounded list of `node.id` equality parameters with no size constraint, no rate limiting, and no per-request cost control. An unauthenticated attacker can flood the `/api/v1/network/nodes` endpoint with requests carrying thousands of `node.id=eq:X` values, each holding a database connection while PostgreSQL evaluates a large `ANY(ARRAY[...])` clause, exhausting the connection pool and denying service to legitimate users.

### Finding Description

**Root cause — missing `@Size` on `nodeIds`:**

`NetworkNodeRequest.nodeIds` is declared as an unbounded `List<NumberRangeParameter>` with no `@Size` constraint:

```java
// NetworkNodeRequest.java line 36-38
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
private List<NumberRangeParameter> nodeIds = List.of();
``` [1](#0-0) 

Compare this to `RegisteredNodesRequest.registeredNodeIds`, which correctly applies `@Size(max = 2)`:

```java
// RegisteredNodesRequest.java line 41-44
@Builder.Default
@RestJavaQueryParam(name = REGISTERED_NODE_ID, required = false)
@Size(max = 2)
private List<NumberRangeParameter> registeredNodeIds = List.of();
``` [2](#0-1) 

The developers applied the guard on `registeredNodeIds` but omitted it for `nodeIds`.

**Code path — unbounded accumulation into SQL array:**

`getNetworkNodes()` iterates every element of `nodeIdParams` and accumulates EQ values into a `HashSet<Long>` with no size check: [3](#0-2) 

The entire set is then converted to a `Long[]` and passed directly to the native SQL query: [4](#0-3) 

The SQL query uses `abe.node_id = any(:nodeIds)`, meaning PostgreSQL receives the full unbounded array in every request: [5](#0-4) 

**`getEffectiveLimit()` does not help:** It caps the number of *rows returned* at 25, but the `nodeIds` array sent to the database is not bounded by this: [6](#0-5) 

**No rate limiting on this endpoint:** The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) lives in the `web3` module and is wired only to contract-call endpoints. The `rest-java` `NetworkController.getNodes()` handler has no rate-limiting interceptor: [7](#0-6) 

**`RequestParameterArgumentResolver` passes all values through:** The resolver collects all repeated query-parameter values into an array and hands them to the binder without any count check: [8](#0-7) 

### Impact Explanation
Each malicious request holds one database connection for the full duration of the PostgreSQL query. With a large `ANY(ARRAY[...])` clause (e.g., 10 000 elements), query latency increases significantly. Concurrent flooding of this endpoint exhausts the connection pool, causing all subsequent legitimate requests (to any endpoint sharing the pool) to queue indefinitely — a full application-level denial of service with no economic cost to the attacker and no authentication required.

### Likelihood Explanation
The endpoint is public, unauthenticated, and reachable by any internet user. HTTP query strings can carry thousands of repeated `node.id` parameters within normal URL-length limits (browsers allow ~8 KB; tools like `curl` or `ab` have no practical limit). The attack is trivially scriptable, requires no special knowledge, and can be sustained indefinitely. The absence of any rate limiting or parameter-count guard makes repeated exploitation straightforward.

### Recommendation
1. **Add `@Size(max = N)` to `NetworkNodeRequest.nodeIds`** — matching the pattern already used on `RegisteredNodesRequest.registeredNodeIds`. A reasonable cap is 2–10, consistent with the rest of the API.
2. **Apply rate limiting to the `rest-java` network endpoints** — extend the existing bucket4j throttle infrastructure (or an API-gateway rule) to cover `/api/v1/network/nodes`.
3. **Add a server-side guard in `getNetworkNodes()`** — reject or truncate requests where `nodeIdParams.size()` exceeds the allowed maximum before any DB interaction.

### Proof of Concept

```bash
# Build a URL with 5000 node.id equality parameters
PARAMS=$(python3 -c "print('&'.join(['node.id=eq:' + str(i) for i in range(5000)]))")

# Fire 50 concurrent requests to exhaust the connection pool
for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>/api/v1/network/nodes?$PARAMS" &
done
wait

# Legitimate request now queues or times out
curl -v "http://<mirror-node-host>/api/v1/network/nodes"
```

Each of the 50 concurrent requests sends a `Long[5000]` array to PostgreSQL via `abe.node_id = any(:nodeIds)`, holding a connection for the query duration. Once the pool is saturated, the final legitimate request receives no connection and hangs or fails.
