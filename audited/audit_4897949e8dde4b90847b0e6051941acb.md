### Title
Unauthenticated GraphQL Thread Exhaustion via `account` Query Causes Liveness Probe Timeout and Pod Restart Loop

### Summary
The GraphQL `account(input: AccountInput!)` query is publicly accessible with no application-level rate limiting or concurrency control in the `graphql` module. The Traefik `inFlightReq` middleware that would limit concurrent requests is disabled by default (`global.middleware: false`). An unprivileged attacker can flood the endpoint with concurrent requests, exhausting the Tomcat thread pool, causing the `/actuator/health/liveness` probe (configured with `timeoutSeconds: 2`) to time out and triggering Kubernetes pod restarts.

### Finding Description

**Code path 1 — Unauthenticated, rate-limit-free query handler:**
`graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java`, lines 32–58: The `account()` method carries no authentication annotation, no rate-limiting guard, and no concurrency semaphore. Each invocation calls `entityService.getByIdAndType()` → `entityRepository.findById()`, a blocking JDBC call that holds a Tomcat worker thread for the duration of the DB round-trip (up to the 10 000 ms `statementTimeout` documented in `docs/graphql/README.md`).

**Code path 2 — Liveness probe with 2-second timeout:**
`charts/hedera-mirror-graphql/values.yaml`, lines 127–133:
```yaml
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: http
  timeoutSeconds: 2
```
The probe shares the same HTTP port (8083) and the same Tomcat thread pool as the GraphQL endpoint.

**Code path 3 — Concurrency middleware disabled by default:**
`charts/hedera-mirror-graphql/values.yaml`, lines 135–145 defines an `inFlightReq` Traefik middleware capping 5 concurrent requests per source IP. However, `charts/hedera-mirror-graphql/templates/middleware.yaml`, line 3 gates the entire middleware chain on `{{ if and .Values.global.middleware .Values.middleware }}`, and `values.yaml` line 89 sets `global.middleware: false`. The middleware is therefore **not deployed in the default configuration**.

**Code path 4 — No application-level throttle in the graphql module:**
The only throttle implementation in the repository (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) belongs exclusively to the `web3` module. The `graphql` module contains no equivalent `ThrottleConfiguration`, `RateLimiter`, or `Semaphore`.

**Root cause:** The design assumes the Traefik middleware layer provides concurrency control, but that layer is opt-in (`global.middleware: false`) and absent in the default deployment. The application itself has no fallback concurrency limit.

**Why existing checks fail:**
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` (`GraphQlConfiguration.java` lines 43–44) reject structurally complex queries but do not limit the *number* of simultaneous simple queries.
- JSON/parser limits (`GraphQlConfiguration.java` lines 74–88) reject oversized payloads but not concurrent small ones.
- `maxRatePerEndpoint: 250` in the GCP backend policy (`values.yaml` line 56) is a requests-per-second throughput cap at the load-balancer tier, not a concurrency cap, and applies only when the GCP gateway is enabled.

### Impact Explanation
Spring Boot's embedded Tomcat defaults to 200 worker threads. Each `account` query holds a thread for the DB round-trip. With 200+ concurrent requests in flight, no thread is available to service the liveness probe within its 2-second window. After `failureThreshold` consecutive misses (default 3), Kubernetes marks the pod `NotReady` and restarts it. If the attacker sustains the flood across the restart cycle, the pod enters a restart loop, effectively partitioning the GraphQL service from the network. With `minAvailable: 50%` in the PodDisruptionBudget and HPA `minReplicas: 1`, a single-replica deployment has no redundancy to absorb the attack.

### Likelihood Explanation
The attack requires no credentials, no special tooling, and no knowledge beyond the public GraphQL schema (`/graphql/alpha`). A single attacker with a modest HTTP flood tool (e.g., `wrk`, `hey`, or a simple script) can sustain 200+ concurrent connections. The query payload is minimal (`{"query":"{account(input:{entityId:{num:1}}){id}}"}`), making it cheap to generate at scale. The attack is fully repeatable across pod restarts.

### Recommendation
1. **Enable the Traefik middleware by default**: Set `global.middleware: true` in the default `values.yaml`, or move the `inFlightReq` guard to a per-service default that does not require the global flag.
2. **Add application-level concurrency control**: Introduce a `Semaphore` or Bucket4j rate limiter in the `graphql` module (mirroring `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) to cap concurrent GraphQL requests independently of the ingress layer.
3. **Isolate the actuator thread pool**: Configure a dedicated Tomcat executor for `/actuator/**` paths so liveness/readiness probes are not starved by application traffic.
4. **Increase `timeoutSeconds`** on the liveness probe or add `failureThreshold` headroom to tolerate transient load spikes without triggering restarts.

### Proof of Concept
```bash
# Precondition: default deployment (global.middleware=false, no app-level rate limit)
# Step 1: saturate the Tomcat thread pool with concurrent account queries

  http://<graphql-service>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{entityId:{num:{}}}){id}}"}' &

# Step 2: observe liveness probe failures in pod events
kubectl get events --field-selector reason=Unhealthy \
  --watch -n <namespace>

    @QueryMapping
    Optional<Account> account(@Argument @Valid AccountInput input) {
        final var alias = input.getAlias();
        final var evmAddress = input.getEvmAddress();
        final var entityId = input.getEntityId();
        final var id = input.getId();

        validateOneOf(alias, entityId, evmAddress, id);

        if (entityId != null) {
            return entityService
                    .getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        if (alias != null) {
            return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
        }

        if (evmAddress != null) {
            return entityService
                    .getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        throw new IllegalStateException("Not implemented");
    }
```
