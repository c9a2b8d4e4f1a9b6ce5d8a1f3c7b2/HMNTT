### Title
Unauthenticated Mirror-Node Response Allows Attacker-Controlled gRPC Endpoint Injection, Exposing Signed CryptoTransfer Transactions

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` fetches node topology over a plain, unauthenticated HTTP connection and blindly trusts every `DomainName`/`IPAddressV4` field in the JSON response. An attacker who can intercept or spoof that HTTP response can inject an arbitrary gRPC endpoint; the resulting network map is passed directly to `hiero.ClientForNetwork(netmap)`, which then routes all signed `CryptoTransfer` transactions — carrying the operator's private-key signature — to the attacker's server. The attacker can replay those transactions within their validity window or selectively withhold them to manipulate the observable Hashgraph transaction sequence.

### Finding Description

**Code path:**

`pinger/sdk_client.go` `newClient()` (line 18-22) calls `buildNetworkFromMirrorNodes()`, which delegates to `fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go`.

The HTTP client is constructed with no TLS enforcement and no authentication: [1](#0-0) 

The example and default value for `cfg.mirrorRest` is explicitly a plain-HTTP URL (`http://mirror-rest:5551`): [2](#0-1) 

The JSON response is decoded and every `DomainName` (falling back to `IPAddressV4`) is accepted without any allowlist or format validation: [3](#0-2) 

The only "checks" are that `host` and `port` are non-empty strings — there is no validation against known-good node addresses, no HTTPS requirement, and no certificate pinning for the mirror-node connection: [4](#0-3) 

The fully attacker-controlled map is handed to the SDK: [5](#0-4) 

The real operator private key is then bound to this client: [6](#0-5) 

Every subsequent `CryptoTransfer` is signed with that key and dispatched to the attacker's gRPC endpoint: [7](#0-6) 

**Root cause:** The function assumes the HTTP response is trustworthy. There is no transport-layer authentication (TLS + cert pinning) for the mirror-node connection, and no application-layer validation of the returned node addresses.

### Impact Explanation

1. **Signed transaction exposure:** The attacker's gRPC server receives fully signed `CryptoTransfer` protobuf messages. Within Hedera's validity window (~180 s), the attacker can forward them to the real network — effectively replaying operator-signed transactions at will.
2. **Selective withholding:** By dropping some transactions and forwarding others, the attacker controls which transfers appear on-chain, directly manipulating the observable Hashgraph transaction sequence.
3. **Operator key oracle:** Repeated captures build a corpus of signed messages that can be used for offline analysis of the signing pattern.

Severity: **High** — confidentiality of signed transaction bytes is broken; integrity of the transaction sequence is broken.

### Likelihood Explanation

**Preconditions (all realistic for an unprivileged attacker):**

- The default/example deployment uses plain HTTP (`http://mirror-rest:5551`). In any shared-network environment (Kubernetes pod network, Docker bridge, cloud VPC without micro-segmentation) a co-tenant can ARP-spoof or DNS-poison the mirror-node hostname with no privileges on the pinger host.
- DNS poisoning of the mirror-node hostname requires only the ability to respond faster than the legitimate resolver — trivially achievable on the same subnet or via a rogue DNS server.
- The attack is repeatable on every startup of the pinger (network map is fetched once at init) and on any restart/redeploy.

No credentials, no access to the pinger host, and no Hedera account are required.

### Recommendation

1. **Enforce HTTPS with certificate validation** for the mirror-node REST call; reject plain-HTTP `cfg.mirrorRest` values at startup.
2. **Pin the mirror-node TLS certificate or CA** so a compromised DNS entry cannot substitute a different server.
3. **Validate returned addresses** against a configurable allowlist of known-good node IP ranges / domain suffixes (e.g., `*.hedera.com`).
4. **Re-fetch the network map periodically** only over the authenticated channel, and diff against the previous map; abort if unexpected endpoints appear.
5. Consider using `hiero.ClientForName` (which uses SDK-hardcoded, well-known node lists) even for "other" networks, rather than trusting a remote HTTP source.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
- Attacker can respond to HTTP on the `mirror-rest` hostname (DNS spoof, ARP spoof, or rogue service on same Docker network).

**Steps:**

1. Stand up a fake HTTP server that returns:
```json
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{
      "domain_name": "attacker.example.com",
      "ip_address_v4": "",
      "port": 50211
    }]
  }],
  "links": {"next": null}
}
```
2. Stand up a gRPC server at `attacker.example.com:50211` that accepts and logs `CryptoTransfer` requests (or forwards them selectively to the real Hedera testnet).
3. Poison DNS so `mirror-rest` resolves to the fake HTTP server's IP.
4. Start (or restart) the pinger. `buildNetworkFromMirrorNodes()` fetches the spoofed response; `hiero.ClientForNetwork` builds a client pointing to `attacker.example.com:50211`.
5. The pinger's `submitWithRetry` loop sends signed `CryptoTransfer` gRPC messages to the attacker's server every `cfg.interval` seconds.
6. The attacker can now replay each captured signed transaction to the real Hedera network within the ~180-second validity window, or withhold them entirely.
