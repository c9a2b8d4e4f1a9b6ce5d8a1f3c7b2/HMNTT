### Title
Historical Entity Resurrection via Missing Upper-Bound and Deleted-Flag Checks in `findActiveByEvmAddressAndTimestamp`

### Summary
The `entity_history` branch of the SQL query in `findActiveByEvmAddressAndTimestamp` only checks `lower(eh.timestamp_range) <= ?2` but neither verifies `upper(eh.timestamp_range) > ?2` nor filters `eh.deleted is not true`. When an entity is deleted at time T2, its pre-deletion `entity_history` record (with `deleted=false`, `timestamp_range=[T1,T2)`) is returned for any `blockTimestamp >= T2`, making the deleted account appear alive in the historical state view. The sibling query `findActiveByEvmAddressOrAliasAndTimestamp` carries the identical defect.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85 (`findActiveByEvmAddressAndTimestamp`) and lines 95–121 (`findActiveByEvmAddressOrAliasAndTimestamp`). [1](#0-0) 

The `entity_history` branch:
```sql
select *
from entity_history eh
where lower(eh.timestamp_range) <= ?2          -- only lower-bound checked
and eh.id = (select id from entity_cte)
order by lower(eh.timestamp_range) desc
limit 1
```

**Root cause — two missing predicates:**
1. No `upper(eh.timestamp_range) > ?2` (or `eh.timestamp_range @> ?2::int8`) — the range containment check.
2. No `eh.deleted is not true` — the deletion-flag check.

**Contrast with `findActiveByIdAndTimestamp`** (lines 136–155), which does include `and deleted is not true` in its `entity_history` branch: [2](#0-1) 

**Data model mechanics (from migration):** [3](#0-2) 

When entity X (EVM address `0xABC`) is deleted at T2:
- `entity_history` receives the pre-deletion snapshot: `(id=X, timestamp_range=[T1,T2), deleted=false)`
- `entity` is updated to: `(id=X, deleted=true, timestamp_range=[T2,∞))`

**Exploit flow — query at blockTimestamp = T2+1:**
1. CTE resolves X from `entity` where `evm_address=0xABC AND created_timestamp <= T2+1` → finds X ✓
2. First UNION branch: `entity WHERE deleted IS NOT TRUE AND id=X` → empty (entity is deleted) ✓
3. Second UNION branch: `entity_history WHERE lower([T1,T2)) = T1 <= T2+1 AND id=X` → returns `(deleted=false, timestamp_range=[T1,T2))` ✗
4. `ORDER BY timestamp_range DESC LIMIT 1` picks this record.
5. `CommonEntityAccessor.getEntityByEvmAddressTimestamp` (lines 80–84) returns it with no further `deleted` check: [4](#0-3) 

**Why existing tests do not catch this:**
The test `findHistoricalEntityByEvmAddressAndTimestampRangeAndDeletedTrueCall` (line 126–132) queries at `getTimestampLower() - 1` — before the entity was created — so the CTE returns nothing and the test passes vacuously: [5](#0-4) 

No test covers the critical scenario: entity deleted in `entity` table + pre-deletion record in `entity_history` + `blockTimestamp > upper(timestamp_range)`.

### Impact Explanation
A deleted Hashgraph account is presented as active (`deleted=false`) to any EVM historical-state query that uses an EVM address. Downstream consumers (`AliasesReadableKVState`, `ContractBytecodeReadableKVState`, balance/EXTCODEHASH opcodes, etc.) receive a non-empty `Optional<Entity>` with `deleted=false` and treat the account as live. This corrupts historical balance reads, EXTCODEHASH results, and any smart-contract logic that calls `eth_call` at a historical block referencing the deleted address — effectively rewriting Hashgraph history for that account.

### Likelihood Explanation
Exploitation requires zero privileges. The attacker only needs: (a) the EVM address of any previously deleted account (all deletions are public on-chain), and (b) the ability to issue a standard `eth_call` or `eth_getBalance` with a `blockNumber` parameter pointing to any block after the deletion. Both are freely available to any JSON-RPC client. The attack is deterministic and repeatable for every deleted account in the mirror node's history.

### Recommendation
Add both missing predicates to the `entity_history` branch in **both** affected queries:

```sql
(
    select *
    from entity_history eh
    where eh.timestamp_range @> ?2::int8        -- range must contain blockTimestamp
    and eh.deleted is not true                  -- must not be a deletion record
    and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

`timestamp_range @> ?2::int8` is equivalent to `lower(eh.timestamp_range) <= ?2 AND upper(eh.timestamp_range) > ?2` and uses the existing GiST index on `entity_history(timestamp_range)` for efficiency. Apply the same fix to `findActiveByEvmAddressOrAliasAndTimestamp`. Also add a regression test with the pattern: persist entity + persist pre-deletion entity_history record + query at `upper(timestamp_range) + 1` → assert empty.

### Proof of Concept

```
Preconditions (DB state):
  entity:         id=42, evm_address=0xDEAD..., created_timestamp=1000,
                  deleted=true, timestamp_range=[2000, ∞)
  entity_history: id=42, evm_address=0xDEAD..., timestamp_range=[1000, 2000),
                  deleted=false

Trigger:
  GET /api/v1/contracts/call
  {
    "to": "0xDEAD...",
    "blockNumber": <any block whose consensus_timestamp > 2000>
  }

SQL execution trace:
  entity_cte → returns id=42  (created_timestamp=1000 ≤ blockTimestamp)
  branch 1   → empty          (entity.deleted=true)
  branch 2   → returns entity_history row (lower=1000 ≤ blockTimestamp, no upper check)
  final row  → entity_history(deleted=false) returned to caller

Result:
  CommonEntityAccessor.get() returns Optional.of(entity{deleted=false})
  → deleted account treated as active in historical EVM state
```
