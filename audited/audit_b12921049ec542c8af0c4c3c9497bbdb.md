### Title
Unbounded Timestamp Range in `getHookStorageChange` Enables DB Resource Exhaustion via `DISTINCT ON` Full Scan

### Summary
The REST Java hook storage historical endpoint accepts an arbitrarily wide timestamp range with no server-side span validation. The underlying `DISTINCT ON (key)` native SQL query must scan and sort every row matching the timestamp range before deduplication and pagination, meaning the `limit` parameter (capped at 100) does not bound intermediate DB work. An unprivileged attacker can repeatedly issue requests spanning the full history of a high-churn storage key, forcing the database to process thousands of rows per request and materially increasing node resource consumption.

### Finding Description
**Code path:**
- Controller: `rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, `hookStorageChangeRequest()` (lines 158–198) — accepts up to 2 `TimestampParameter` values (`@Size(max = 2)`) but applies no range-width constraint.
- Service: `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`, `getHookStorageChange()` (lines 84–118) — passes `timestampLowerBound` and `timestampUpperBound` directly to the repository without checking `upperBound - lowerBound`.
- Repository: `rest-java/src/main/java/org/hiero/mirror/restjava/repository/HookStorageChangeRepository.java`, `findByKeyBetweenAndTimestampBetween()` (lines 15–39) and `findByKeyInAndTimestampBetween()` (lines 41–63).

**Root cause:** PostgreSQL's `SELECT DISTINCT ON (key) … WHERE consensus_timestamp BETWEEN :lo AND :hi ORDER BY key, consensus_timestamp DESC LIMIT N` must read and sort **all** rows satisfying the WHERE predicate before it can emit the first N distinct keys. The `LIMIT N` (max 100) only truncates the output; it does not short-circuit the full-range scan when the index cannot satisfy the `DISTINCT ON` ordering without a sequential pass. No maximum timestamp span is enforced anywhere in the REST Java path for this endpoint, unlike the Node.js REST API which rejects ranges wider than 7 days.

**Failed assumption:** The design assumes that `limit` bounds DB work. It does not — `DISTINCT ON` is a post-scan deduplication step.

### Impact Explanation
A high-churn storage key (one written on every block) accumulates O(millions) of rows over months. A single request with `timestamp=gte:0&timestamp=lte:9999999999.999999999` forces the DB to scan and sort all of those rows. Issuing tens of such requests concurrently (trivially scriptable) can saturate DB CPU and I/O, increasing processing node resource consumption well beyond 30% compared to baseline. The 10-second statement timeout limits per-query duration but does not prevent aggregate resource exhaustion from concurrent requests.

### Likelihood Explanation
No authentication or privilege is required — the endpoint is public. The attacker only needs a valid `ownerId` and `hookId`, both discoverable via the public `/api/v1/accounts/{id}/hooks` endpoint. The attack is fully repeatable and scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, `curl` in a loop). The only friction is identifying a high-churn key, which is observable from public blockchain data.

### Recommendation
1. **Enforce a maximum timestamp range span** in `HooksController.hookStorageChangeRequest()` or inside `Bound.of()` for the hook storage path — reject requests where `upperBound - lowerBound` exceeds a configured maximum (e.g., 7 days, matching the Node.js REST API's `maxTimestampRange`).
2. **Add a DB-level row scan limit** (e.g., via a CTE with `LIMIT` before the `DISTINCT ON`) to cap intermediate work regardless of the timestamp range.
3. **Apply rate limiting** per IP/account on the historical hook storage endpoint.
4. Consider adding a **connection-pool-level concurrency limit** for this query type.

    private HookStorageRequest hookStorageChangeRequest(
            EntityIdParameter ownerId,
            long hookId,
            List<SlotRangeParameter> keys,
            TimestampParameter[] timestamps,
            int limit,
            Direction order) {
        final var keyFilters = new ArrayList<byte[]>();

        var lowerBound = MIN_KEY_BYTES;
        var upperBound = MAX_KEY_BYTES;

        for (final var key : keys) {
            final byte[] value = key.value();

            if (key.hasLowerBound()) {
                if (key.operator() == RangeOperator.EQ) {
                    keyFilters.add(value);
                } else if (Arrays.compareUnsigned(value, lowerBound) > 0) {
                    lowerBound = value;
                }
            } else if (key.hasUpperBound()) {
                if (Arrays.compareUnsigned(value, upperBound) < 0) {
                    upperBound = value;
                }
            }
        }

        final var bound = Bound.of(timestamps, TIMESTAMP, HookStorageChange.HOOK_STORAGE_CHANGE.CONSENSUS_TIMESTAMP);

        return HookStorageRequest.builder()
                .hookId(hookId)
                .keys(keyFilters)
                .limit(limit)
                .keyLowerBound(lowerBound)
                .keyUpperBound(upperBound)
                .order(order)
                .ownerId(ownerId)
                .timestamp(bound)
                .build();
    }
