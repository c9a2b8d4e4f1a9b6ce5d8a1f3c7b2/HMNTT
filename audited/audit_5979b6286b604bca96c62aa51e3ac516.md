### Title
Unauthenticated Alias-Triggered DB Query Flood Exhausts REST Connection Pool (DoS)

### Summary
The `/api/v1/balances` endpoint accepts an `account.id` query parameter that, when supplied as a valid base32 alias, unconditionally triggers an asynchronous database lookup via `EntityService.getAccountIdFromAlias()`. Because the REST API has no rate limiting and the response cache is keyed per unique URL, an attacker flooding the endpoint with distinct valid alias values generates a sustained stream of concurrent DB queries that exhausts the finite connection pool, rendering the mirror node REST API unavailable.

### Finding Description

**Exact code path:**

`rest/balances.js` `getBalances()` (line 87) calls `parseAccountIdQueryParam(req.query, 'ab.account_id')`, which is the local function defined at lines 320РђЊ357: [1](#0-0) 

When `AccountAlias.isValid(value, true)` passes (regex `/^[A-Z2-7]+$/` Рђћ any uppercase base32 string, no authentication required), and `evmAliasAddressCount === 1` (first alias in this request), it calls:

```js
EntityService.getAccountIdFromAlias(AccountAlias.fromString(value), false)
```

This chains to `getAccountFromAlias()` Рєњ `super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias])`: [2](#0-1) 

Which executes a live DB query: [3](#0-2) 

The DB pool is initialized in `rest/dbpool.js` with a hard `max` from config: [4](#0-3) 

The Helm chart caps `mirror_rest` at `max_user_connections: 250`: [5](#0-4) 

**Root cause:** No rate limiting exists on the REST API. The response cache in `responseCacheHandler.js` is keyed by MD5 of `req.originalUrl`: [6](#0-5) 

Each distinct alias value produces a unique URL Рєњ unique cache key Рєњ cache miss Рєњ DB query. There is no in-process alias-to-ID cache in the REST service.

**Why the `evmAliasAddressCount` guard is insufficient:** It only prevents more than one alias *within a single request*. It does nothing to limit the rate of alias lookups *across* concurrent requests. [7](#0-6) 

**No rate limiting on the REST API:** `rest/server.js` registers the balances route with no throttle middleware: [8](#0-7) 

The throttle infrastructure found (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the web3 `contracts/call` endpoint, not the REST API.

### Impact Explanation

An attacker can exhaust the `mirror_rest` DB connection pool (capped at 250 connections). Once exhausted, all REST API endpoints that require DB access begin queuing or timing out (`connectionTimeoutMillis`). This causes complete unavailability of the mirror node REST API. While the mirror node does not directly participate in Hedera consensus, exhausting it denies all downstream consumers (wallets, explorers, dApps) access to transaction and balance data, and can indirectly stall the importer if it shares connection budget with the REST user.

### Likelihood Explanation

- **No privileges required:** The `/api/v1/balances` endpoint is fully public.
- **Trivial alias generation:** Valid aliases are any uppercase base32 strings (`[A-Z2-7]+`). An attacker can generate millions of distinct values programmatically (e.g., `AAAAAAA`, `AAAAAAB`, Рђд).
- **Amplification:** A single HTTP client with async concurrency (e.g., `curl`/`wrk`/`ab`) can saturate the pool. Each request holds a DB connection for the full query round-trip.
- **Repeatability:** The attack is stateless and trivially repeatable. No session, token, or account is needed.

### Recommendation

1. **Add rate limiting to the REST API** Рђћ apply a per-IP or global request-rate limiter (e.g., `express-rate-limit`) to all REST endpoints, especially those that trigger DB lookups.
2. **Cache aliasРєњentityId resolutions** Рђћ introduce an in-process LRU/TTL cache (e.g., using `lru-cache`) for alias lookups in `EntityService.getAccountIdFromAlias()` so repeated or concurrent lookups for the same alias do not each consume a DB connection.
3. **Limit concurrent in-flight alias lookups** Рђћ use a semaphore or concurrency limiter around the alias DB lookup path.
4. **Reject unknown aliases early** Рђћ consider returning a 400 or 404 immediately for aliases that fail a lightweight pre-check before hitting the DB.

### Proof of Concept

```bash
# Generate 500 distinct valid base32 aliases and flood the endpoint concurrently
python3 -c "
import string, itertools, subprocess, threading

chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'
aliases = [''.join(c) for c in itertools.product(chars[:8], repeat=3)][:500]

def req(alias):
    import urllib.request
    try:
        urllib.request.urlopen(
            f'http://<mirror-node-host>/api/v1/balances?account.id={alias}',
            timeout=10
        )
    except: pass

threads = [threading.Thread(target=req, args=(a,)) for a in aliases]
[t.start() for t in threads]
[t.join() for t in threads]
print('Done')
"

# Expected result: DB connection pool exhausted; subsequent legitimate API
# requests return 503 / connection timeout errors.
```

Each of the 500 concurrent requests passes `AccountAlias.isValid()`, increments `evmAliasAddressCount` to 1, and calls `EntityService.getAccountIdFromAlias()`, issuing a live `SELECT` against the `entity` table. With 250 max connections, the pool saturates and all other REST API traffic is blocked.
