### Title
Unchecked Slice Index Access on `GetNodeAccountIDs()[0]` in `ConstructionSubmit` Causes Panic-Based DoS

### Summary
`ConstructionSubmit()` in `rosetta/app/services/construction_service.go` unconditionally accesses `transaction.GetNodeAccountIDs()[0]` at line 353 without first verifying the slice is non-empty. An unprivileged external user can submit a crafted, structurally valid signed transaction (of a supported type) that carries no node account IDs, causing a Go runtime panic (index out of range) that crashes the service before `hiero.TransactionExecute` is ever reached. The error type `ErrNodeAccountIdsEmpty` already exists in the codebase but is never checked in this code path.

### Finding Description

**Exact code path:**

`rosetta/app/services/construction_service.go`, function `ConstructionSubmit()`, lines 331–368:

```go
transaction, rErr := unmarshallTransactionFromHexString(request.SignedTransaction)
if rErr != nil {
    return nil, rErr
}

hashBytes, err := hiero.TransactionGetTransactionHash(transaction)
if err != nil {

transactionId, _ := hiero.TransactionGetTransactionID(transaction)
log.Infof("Submitting transaction %s (hash %s) to node %s", transactionId,
    hash, transaction.GetNodeAccountIDs()[0])   // ← PANIC if slice is empty
```

**Root cause:** `transaction.GetNodeAccountIDs()` returns a `[]hiero.AccountID`. If the deserialized transaction carries no node account IDs, this slice is empty and `[0]` triggers a Go runtime panic.

**Why existing checks are insufficient:**

`unmarshallTransactionFromHexString` (lines 658–674) performs exactly three checks:
1. Valid hex decoding
2. `hiero.TransactionFromBytes` succeeds (protobuf deserialization)
3. `isSupportedTransactionType` — only allows `AccountCreateTransaction` or `TransferTransaction`

None of these checks require node account IDs to be present. The Hiero SDK's `TransactionFromBytes` deserializes a `SignedTransaction` protobuf; the `nodeAccountID` field in the transaction body is optional at the protobuf level, so deserialization succeeds even when it is absent. `TransactionGetTransactionHash` computes a hash over the raw `bodyBytes` field of the `SignedTransaction`, which is present regardless of whether node account IDs are set, so that call also succeeds.

The `ErrNodeAccountIdsEmpty` error (errors.go line 99–40) and its guard exist only in `getRandomNodeAccountId()` (lines 526–527), which is a completely separate code path used during transaction construction — it is never invoked in `ConstructionSubmit`.

**Exploit flow:**
1. Attacker constructs a `CryptoTransfer` or `CryptoCreateAccount` protobuf transaction body **without** setting any `nodeAccountID`.
2. Wraps it in a `SignedTransaction` protobuf with valid `bodyBytes` (and optionally a signature map).
3. Serializes to bytes, hex-encodes with `0x` prefix.
4. POSTs to the public `/construction/submit` endpoint with `signed_transaction` set to that hex string.
5. `unmarshallTransactionFromHexString` returns successfully.
6. `TransactionGetTransactionHash` returns successfully.
7. `transaction.GetNodeAccountIDs()[0]` panics: **index out of range [0] with length 0**.
8. Go runtime terminates the goroutine (or the whole process depending on recovery configuration), crashing the online Rosetta service.

### Impact Explanation
A successful exploit crashes the Rosetta service process, making the `/construction/submit` endpoint (and all other endpoints served by the same process) unavailable to all legitimate users. Because the panic occurs before any network I/O to the Hedera network, no transaction is ever submitted, so the attacker achieves a pure denial-of-service with no side effects on the ledger. The crash is repeatable on every restart as long as the code is unpatched.

### Likelihood Explanation
The `/construction/submit` endpoint is a public HTTP endpoint requiring no authentication or privileged credentials — any network-reachable client can call it. Crafting a minimal valid protobuf `SignedTransaction` with an empty node account ID list requires only basic protobuf knowledge and is trivially automatable. The attack is repeatable indefinitely and requires no prior knowledge of the system beyond the endpoint URL and the Rosetta API schema.

### Recommendation
Add an explicit bounds check immediately after `unmarshallTransactionFromHexString` returns (or inside it), before the `log.Infof` call:

```go
nodeIds := transaction.GetNodeAccountIDs()
if len(nodeIds) == 0 {
    return nil, errors.ErrNodeAccountIdsEmpty  // already defined in errors.go
}
log.Infof("Submitting transaction %s (hash %s) to node %s", transactionId, hash, nodeIds[0])
```

Alternatively, move this check into `unmarshallTransactionFromHexString` so all callers benefit from the guard uniformly.

func unmarshallTransactionFromHexString(transactionString string) (hiero.TransactionInterface, *rTypes.Error) {
	transactionBytes, err := hex.DecodeString(tools.SafeRemoveHexPrefix(transactionString))
	if err != nil {
		return nil, errors.ErrTransactionDecodeFailed
	}

	transaction, err := hiero.TransactionFromBytes(transactionBytes)
	if err != nil {
		return nil, errors.ErrTransactionUnmarshallingFailed
	}

	if rErr := isSupportedTransactionType(transaction); rErr != nil {
		return nil, rErr
	}

	return transaction, nil
}

func isSupportedTransactionType(transaction hiero.TransactionInterface) *rTypes.Error {
	switch transaction.(type) {
	case hiero.AccountCreateTransaction:
	case hiero.TransferTransaction:
	default:
		return errors.ErrTransactionInvalidType
	}

	return nil
}
