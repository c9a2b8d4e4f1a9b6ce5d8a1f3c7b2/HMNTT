### Title
Unauthenticated `/metrics` Endpoint Allows Resource Exhaustion via Prometheus Registry Serialization

### Summary
The `Routes()` function in `rosetta/app/middleware/metrics.go` registers a public `GET /metrics` route backed directly by `promhttp.Handler().ServeHTTP` with no authentication, no rate limiting, and no IP-based access control. Any external user can repeatedly invoke this endpoint, forcing a full gather-and-serialize pass over the entire `prometheus.DefaultGatherer` registry (including all Go runtime metric families) on every request, driving CPU consumption well above 30% of baseline at moderate, non-brute-force request rates.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, `Routes()`, lines 64–73:
```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,                      // "/metrics"
            promhttp.Handler().ServeHTTP,     // no auth wrapper
        },
    }
}
```

`rosetta/main.go`, lines 106–119 (online) and 148–152 (offline): `metricsController` is passed directly to `server.NewRouter(...)`. The middleware chain applied afterward (`MetricsMiddleware → TracingMiddleware → CorsMiddleware`, lines 217–219) contains no authentication or rate-limiting layer for this route.

**Root cause:** The failed assumption is that `/metrics` is an internal-only endpoint protected by network policy. In practice, the rosetta HTTP server binds on a single port (line 221: `rosettaConfig.Port`) and serves all routes — including `/metrics` — on the same public-facing listener. There is no per-route auth guard, no middleware that checks credentials before dispatching to `promhttp.Handler()`, and no connection-rate limiter anywhere in the Go server stack.

**Exploit flow:**
1. Attacker discovers the rosetta port (default 8080) is reachable.
2. Attacker issues `GET /metrics` in a loop at a moderate rate (e.g., 200–500 req/s from a single host).
3. Each request causes `prometheus.DefaultGatherer.Gather()` to lock and iterate every registered `Collector` — the four custom histograms/gauges registered in `init()` (lines 46–52) plus all default Go runtime collectors (~70+ metric families including GC, goroutine, memory, and process stats).
4. The gathered `MetricFamily` slice is then text-serialized by the `expfmt` encoder and written to the response.
5. Because `promhttp.Handler()` is stateless and creates no per-request caching, every call repeats the full gather+encode cycle.

**Why existing checks fail:**
- `MetricsMiddleware` (lines 76–83) only instruments request metrics; it does not gate access.
- `TracingMiddleware` and `server.CorsMiddleware` are pass-through for this route.
- There is no `http.Server`-level connection limit or read-rate throttle configured.
- The `rest/` service's `metricsHandler.js` (which does implement `authenticate()`) is a completely separate service and has no bearing on the Go rosetta binary.

### Impact Explanation
Beyond CPU amplification, the unauthenticated endpoint leaks internal operational data: per-route request counts, in-flight request gauges, request/response byte distributions, and all Go runtime internals (heap size, GC pause counts, goroutine counts). This constitutes both a resource-exhaustion vector and an information-disclosure vulnerability. At 200–500 req/s — achievable from a single commodity host — the continuous gather+serialize loop competes with the rosetta node's own blockchain processing goroutines for CPU time, degrading block-query and construction-API response latency for legitimate users.

### Likelihood Explanation
The endpoint requires zero credentials, zero prior knowledge beyond the port number, and is reachable from any network that can reach the rosetta port. The attack is trivially scriptable (`while true; do curl -s http://<host>:8080/metrics > /dev/null; done` in parallel). No exploit tooling is needed. The Rosetta API spec does not mandate that `/metrics` be public, so operators are unlikely to have independently firewalled it.

### Recommendation
1. **Add an authentication wrapper** around `promhttp.Handler()` in `Routes()`, checking a configurable bearer token or basic-auth credential before delegating to the Prometheus handler — mirroring the pattern already used in `rest/middleware/metricsHandler.js` (`authenticate()` at line 137).
2. **Restrict the metrics listener** to a separate, non-public port (e.g., a dedicated admin port bound to `127.0.0.1` or a cluster-internal interface), decoupled from the public Rosetta API port.
3. **Apply a per-IP rate limiter** (e.g., `golang.org/x/time/rate`) as middleware specifically for the `/metrics` route.
4. **Add a `MaxConcurrentRequests` semaphore** to the metrics handler to bound simultaneous gather operations.

### Proof of Concept
```bash
# Confirm unauthenticated access
curl -v http://<rosetta-host>:8080/metrics
# HTTP 200 with full Prometheus text output — no credentials required

# Sustained load from a single host (non-brute-force rate)
seq 1 8 | xargs -P8 -I{} bash -c \
  'while true; do curl -s http://<rosetta-host>:8080/metrics -o /dev/null; done'

# Observe CPU on the rosetta process
watch -n1 'ps -p $(pgrep rosetta) -o %cpu,rss'
# %CPU climbs >30% of baseline within seconds; legitimate API latency increases
```
