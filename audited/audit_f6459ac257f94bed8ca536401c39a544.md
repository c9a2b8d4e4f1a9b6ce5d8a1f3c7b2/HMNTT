### Title
Log Injection via Unsanitized `X-Forwarded-For` Header in `TracingMiddleware` Poisons `/construction/submit` Audit Trail

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` returns the raw, unsanitized value of the `X-Forwarded-For` header — including comma-separated IP lists and arbitrary text — which `TracingMiddleware` formats directly into a `log.Info` call with no escaping. Any unauthenticated external user can inject arbitrary text (including newline characters, depending on the HTTP stack) into the structured log line produced for every `/construction/submit` request, poisoning the audit trail for transaction submissions.

### Finding Description

**Exact code path:**

`getClientIpAddress()` ( [1](#0-0) ) performs a plain `r.Header.Get(xForwardedForHeader)` at line 67 with zero post-processing: no comma-split to extract the leftmost/rightmost IP, no newline stripping, no length cap, no character allowlist. The raw string is returned as-is.

`TracingMiddleware` ( [2](#0-1) ) then interpolates that raw string directly into a `fmt.Sprintf` format and passes it to `log.Info`. The format is:

```
<clientIpAddress> <METHOD> <path> (<status>) in <duration>
```

Because `clientIpAddress` is the verbatim header value, an attacker-supplied value like:

```
10.0.0.1, 192.168.1.1, FAKE_ENTRY\ntime="..." level=info msg="127.0.0.1 POST /construction/submit (200) in 1ms"
```

is written directly into the log stream.

`TracingMiddleware` is wired globally in `main.go` ( [3](#0-2) ) and therefore covers every endpoint, including `/construction/submit`.

**Why existing checks fail:**

The test suite ( [4](#0-3) ) only exercises a clean single IP in `X-Forwarded-For`; it never tests a comma-separated list, embedded newlines, or other injection payloads. There is no runtime sanitization anywhere in the middleware chain.

### Impact Explanation

For `/construction/submit` — the endpoint that broadcasts signed Hedera transactions — the log line is the primary audit record of which client submitted which transaction. An attacker can:

1. **IP spoofing in logs**: Make every submission appear to originate from an arbitrary IP (e.g., an internal trusted address), breaking forensic attribution.
2. **Log injection / forged entries**: If the HTTP server passes `\r\n` or `\n` through the header (possible with certain reverse-proxy configurations or direct connections), the attacker can synthesize entirely fake log lines, including fake successful or failed submission records, corrupting the audit trail used to detect tampered transaction submissions.
3. **Log aggregator confusion**: Injected commas and spaces break log-parsing pipelines (Loki, Splunk, etc.) that rely on the structured `logfmt` format configured in `main.go` ( [5](#0-4) ).

Severity: **Medium–High** for audit-trail integrity; the transaction itself is not altered, but the ability to attribute or detect tampered submissions is undermined.

### Likelihood Explanation

- **No authentication required**: The Rosetta API is unauthenticated by design; any network-reachable client can send arbitrary headers.
- **Trivial to trigger**: A single `curl` with a crafted `X-Forwarded-For` header is sufficient.
- **Repeatable**: Every request goes through `TracingMiddleware`, so every submission can be poisoned.
- **Standard header format**: Comma-separated `X-Forwarded-For` values are RFC-standard and will not be rejected by any HTTP layer.

### Recommendation

In `getClientIpAddress()`, after reading the header, extract only the **first** (or last, depending on trust model) token from the comma-separated list and strip all non-IP characters before returning:

```go
ipAddress = r.Header.Get(xForwardedForHeader)
if idx := strings.IndexByte(ipAddress, ','); idx != -1 {
    ipAddress = ipAddress[:idx]
}
ipAddress = strings.TrimSpace(ipAddress)
// Optionally validate with net.ParseIP and reject if nil
```

Additionally, before logging, replace or escape any newline characters in the final `message` string to prevent log-line injection regardless of the source field.

### Proof of Concept

```bash
# 1. Inject a comma-separated list — logged verbatim as the client IP
curl -s -X POST http://<rosetta-host>:<port>/construction/submit \
  -H 'Content-Type: application/json' \
  -H 'X-Forwarded-For: 10.0.0.1, 192.168.1.1, AUDIT_TAMPERED' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},...}'

# Expected log output (observe the full unsanitized string as the IP field):
# time="..." level=info msg="10.0.0.1, 192.168.1.1, AUDIT_TAMPERED POST /construction/submit (200) in 5ms"

# 2. Newline injection (direct TCP or permissive proxy):
curl -s -X POST http://<rosetta-host>:<port>/construction/submit \
  -H $'X-Forwarded-For: 1.2.3.4\ntime="2024-01-01" level=info msg="127.0.0.1 POST /construction/submit (200) in 1ms"' \
  -H 'Content-Type: application/json' \
  -d '{...}'

# Result: two log lines emitted — one real, one fully attacker-controlled —
# making it impossible to distinguish legitimate from forged submission records.
```
