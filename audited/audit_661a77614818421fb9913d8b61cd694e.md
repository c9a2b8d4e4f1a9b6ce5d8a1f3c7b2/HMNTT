### Title
Silent Override of Explicit `valid_duration=0` Causes Unintended 180-Second Transaction Validity Window

### Summary
An unprivileged external user who explicitly supplies `valid_duration: "0"` in a `ConstructionPayloadsRequest` metadata map will have that value silently replaced with `defaultValidDurationSeconds` (180) in the resulting transaction, with no error or warning returned. The root cause is that `0` is used as both the "key absent / unset" sentinel and a legitimate parseable integer, making the two cases indistinguishable across the call chain.

### Finding Description
**Exact code path:**

1. `ConstructionPayloads` calls `getTransactionTimestampProperty(request.Metadata)`. [1](#0-0) 

2. Inside `getTransactionTimestampProperty`, `getIntMetadataValue(metadata, "valid_duration")` is called. [2](#0-1) 

3. `getIntMetadataValue` checks `metadata[metadataKey] != nil` — for `"0"` this is **not nil** — then calls `tools.ToInt64("0")` which returns `(0, nil)`, and `0 < 0` is false, so it returns `(0, nil)` with no error. [3](#0-2) 

4. Back in `getTransactionTimestampProperty`, `isValidTransactionValidDuration(0)` evaluates `0 >= 0 && 0 <= 180` → `true`. The comment explicitly states *"A value of 0 indicates validDuration is unset"*, so the guard condition `err != nil || !isValidTransactionValidDuration(validDurationSeconds)` is `false`, no error is raised, and `validDurationSeconds = 0` is returned. [4](#0-3) 

5. `transactionSetValidDuration(0)` is called. It checks `if validDurationSeconds == 0` and silently replaces it with `defaultValidDurationSeconds` (180), then sets that on the transaction — no error, no log warning to the caller. [5](#0-4) 

**Root cause:** `getIntMetadataValue` returns `(0, nil)` for both an absent key and an explicitly supplied `"0"` value. The sentinel value `0` is therefore ambiguous: it means "unset" in `isValidTransactionValidDuration` and "use default" in `transactionSetValidDuration`, but it is also the valid parsed result of a user-supplied `"0"` string. There is no code path that distinguishes the two cases or rejects the explicit `"0"`.

### Impact Explanation
The user's explicitly supplied `valid_duration` is silently discarded and replaced with 180 seconds. The transaction is submitted with a validity window the user did not intend and was not informed about. In a Rosetta workflow, the unsigned transaction bytes returned to the client encode this 180-second window; the client signs and submits a transaction whose validity period differs from what was requested. This constitutes silent mutation of user-controlled transaction parameters, which can lead to griefing (e.g., a user who wants a very short or zero-duration window to limit replay exposure gets a full 180-second window instead). No economic damage occurs, but the user loses control over a security-relevant transaction property.

### Likelihood Explanation
Any unprivileged caller of `/construction/payloads` can trigger this by including `"valid_duration": "0"` in the metadata object. No authentication, special role, or network access beyond the public Rosetta API endpoint is required. The trigger is a single, trivially constructed JSON field. It is fully repeatable.

### Recommendation
Distinguish "key absent" from "key present with value 0" in `getIntMetadataValue`. One approach: return a sentinel such as `-1` (or use a pointer/optional) when the key is absent, and treat only that sentinel as "unset" in `isValidTransactionValidDuration` and `transactionSetValidDuration`. Alternatively, reject `valid_duration = 0` explicitly with `ErrInvalidArgument` in `isValidTransactionValidDuration` (change `validDuration >= 0` to `validDuration > 0`), and handle the "absent" case separately before calling `getIntMetadataValue`.

### Proof of Concept
```
POST /construction/payloads
{
  "network_identifier": { ... },
  "operations": [ ... ],
  "metadata": {
    "node_account_id": "0.0.3",
    "valid_duration": "0",          // explicitly supplied as zero
    "valid_start_nanos": "1700000000000000000"
  }
}
```
**Expected:** Either an error (`ErrInvalidArgument`) or a transaction with `transactionValidDuration = 0s`.

**Actual:** No error is returned. The unsigned transaction bytes encode `transactionValidDuration = 180s`. The user's `"0"` value is silently overridden.

Trace:
- `getIntMetadataValue(..., "valid_duration")` → `(0, nil)` ✓ (no error)
- `isValidTransactionValidDuration(0)` → `true` ✓ (0 accepted as "unset")
- `transactionSetValidDuration(0)` → sets 180s silently ✓

```go
func (c *constructionAPIService) getIntMetadataValue(metadata map[string]any, metadataKey string) (int64, *rTypes.Error) {
	var metadataValue int64
	if metadata != nil && metadata[metadataKey] != nil {
		value, ok := metadata[metadataKey].(string)
		if !ok {
			return metadataValue, errors.ErrInvalidArgument
		}

		var err error
		if metadataValue, err = tools.ToInt64(value); err != nil || metadataValue < 0 {
			return metadataValue, errors.ErrInvalidArgument
		}
	}

	return metadataValue, nil
}
```

```go
func transactionSetValidDuration(validDurationSeconds int64) updater {
	return func(transaction hiero.TransactionInterface) *rTypes.Error {
		if validDurationSeconds == 0 {
			// Default to 180 seconds
			validDurationSeconds = defaultValidDurationSeconds
		}

		if _, err := hiero.TransactionSetTransactionValidDuration(transaction, time.Second*time.Duration(validDurationSeconds)); err != nil {
			log.Errorf("Failed to set transaction valid duration: %s", err)
			return errors.ErrInternalServerError
		}
		return nil
	}
```
