### Title
Reference Equality Bypass in `findByBlockType()` Allows Numeric Zero Block to Skip `findEarliest()` and Trigger Incorrect Behavior

### Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect the `EARLIEST` block sentinel, but `BlockType.of("0x0000000000000000")` (or any numeric-zero input like `"0"` or `"0x0"`) constructs a **new** `BlockType` instance with `number=0` that is not the `EARLIEST` singleton. This causes the code to fall through to `findByIndex(0)` instead of `findEarliest()`. If the earliest record file in the database has an index other than 0, `findByIndex(0)` returns `Optional.empty()` while `findEarliest()` would have returned the genesis block, producing an error response instead of valid genesis block data.

### Finding Description

**Root cause — reference equality on a record type:**

In `RecordFileServiceImpl.findByBlockType()`:

```java
if (block == BlockType.EARLIEST) {          // line 20 — reference equality only
    return recordFileRepository.findEarliest();
} else if (block == BlockType.LATEST) {     // line 22
    return recordFileRepository.findLatest();
} else if (block.isHash()) {
    return recordFileRepository.findByHash(block.name());
}
return recordFileRepository.findByIndex(block.number());  // line 28 — fallthrough
``` [1](#0-0) 

`BlockType.EARLIEST` is a static singleton:

```java
public static final BlockType EARLIEST = new BlockType("earliest", 0L);
``` [2](#0-1) 

**Exploit path through `BlockType.of()`:**

When a caller passes `"0x0000000000000000"` (16 hex chars after `0x`), the regex GROUP_HEX_NUM `0x([0-9a-f]{1,16})` matches:

```java
final var hexNum = matcher.group(GROUP_HEX_NUM);   // "0000000000000000"
return new BlockType(hexNum, Long.parseLong(hexNum, 16));  // new BlockType("0000000000000000", 0L)
``` [3](#0-2) 

This produces a **new heap object** with `number=0`, not the `EARLIEST` singleton. Java `==` checks reference identity, so `block == BlockType.EARLIEST` is `false`. The same bypass applies to decimal `"0"` and `"0x0"`.

**Behavioral divergence between `findByIndex(0)` and `findEarliest()`:**

```java
// findEarliest: returns the record with the LOWEST index, whatever it is
@Query(value = "select * from record_file order by index asc limit 1", nativeQuery = true)
Optional<RecordFile> findEarliest();

// findByIndex: returns the record WHERE index = 0 exactly — empty if none exists
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
``` [4](#0-3) 

If the earliest record file has `index ≠ 0` (e.g., the chain started at index 1, or a migration left a gap), `findByIndex(0)` returns `Optional.empty()` while `findEarliest()` would have returned the genesis block.

### Impact Explanation
An unprivileged caller can send `eth_call` or `eth_getBlockByNumber` with block tag `0x0000000000000000` (or `"0"`) and receive a `BlockNumberNotFoundException` / error response instead of the genesis block. This is incorrect Ethereum JSON-RPC behavior and can break client applications that rely on querying block 0 to discover chain genesis state. Severity is Medium: no direct fund loss, but it causes unintended smart contract call behavior (calls against block 0 fail or return wrong state) and breaks standard Ethereum tooling assumptions.

### Likelihood Explanation
Any unauthenticated user can trigger this with a single JSON-RPC call. The input `"0x0000000000000000"` is a valid Ethereum block parameter per the spec. The precondition (earliest record file index ≠ 0) is realistic in production deployments that have undergone data migrations or pruning. The bypass is deterministic and repeatable.

### Recommendation
Replace reference equality with value-based comparison in `findByBlockType()`:

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (BlockType.EARLIEST.equals(block) || block.number() == 0 && !block.isHash()) {
        return recordFileRepository.findEarliest();
    } else if (BlockType.LATEST.equals(block)) {
        return recordFileRepository.findLatest();
    } else if (block.isHash()) {
        return recordFileRepository.findByHash(block.name());
    }
    return recordFileRepository.findByIndex(block.number());
}
```

Since `BlockType` is a Java `record`, it auto-generates `.equals()` based on field values, so `BlockType.EARLIEST.equals(new BlockType("0000000000000000", 0L))` would be `true` only if both `name` and `number` match — which they don't here. The safer fix is to compare `block.number() == 0 && !block.isHash()` as the EARLIEST condition, or normalize numeric-zero inputs to the `EARLIEST` singleton inside `BlockType.of()`.

### Proof of Concept

```bash
# Trigger the bypass with 8-byte hex zero
curl -X POST http://<mirror-node-web3>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "block": "0x0000000000000000",
    "data": "0x",
    "to": "0x0000000000000000000000000000000000000001"
  }'

# Expected (correct): returns genesis block state
# Actual: BlockNumberNotFoundException if record_file table has no row with index=0

# Same bypass with decimal zero
curl ... -d '{"block": "0", ...}'

# Same bypass with short hex zero  
curl ... -d '{"block": "0x0", ...}'
```

All three inputs create a new `BlockType` instance with `number=0` that is not `== BlockType.EARLIEST`, causing `findByIndex(0)` to be called instead of `findEarliest()`.
