### Title
Unbounded O(N×M) CPU Exhaustion via Shared Flux Fan-out in `SharedPollingTopicListener` (SHARED_POLL mode)

### Summary
In `SHARED_POLL` mode, `SharedPollingTopicListener.getSharedListener()` unconditionally returns a single shared `topicMessages` Flux for every subscriber regardless of `topicId` or `startTime`. `CompositeTopicListener.listen()` then attaches a per-subscriber `.filter(t -> filterMessage(t, filter))` operator to that shared Flux, meaning every emitted message is evaluated against every active subscriber's filter. With no connection limit, rate limit, or per-IP subscription cap anywhere in the stack, an unprivileged attacker can open an arbitrary number of subscriptions and drive CPU usage to O(N × M) per poll cycle.

### Finding Description

**Code path:**

`CompositeTopicListener.listen()` (line 40–43) delegates to `sharedPollingTopicListener.listen(filter)` when the type is `SHARED_POLL`, then chains `.filter(t -> filterMessage(t, filter))` on the result. [1](#0-0) 

`SharedTopicListener.listen()` (line 22–25) calls `getSharedListener(filter)` and wraps it with backpressure buffering and `publishOn`. [2](#0-1) 

`SharedPollingTopicListener.getSharedListener()` (line 56–58) **ignores the filter entirely** and returns the single pre-built `topicMessages` Flux for every caller. [3](#0-2) 

That shared Flux is constructed once at bean creation time (line 41–52) and polls `topicMessageRepository.findLatest()` — which fetches messages across **all topics** — up to `maxPageSize` (default 5,000) messages every `interval` (default 500 ms). [4](#0-3) 

The per-subscriber filter applied in `CompositeTopicListener`: [5](#0-4) 

**Root cause:** `getSharedListener()` was designed to share one Flux across subscribers of the *same* topic (reducing DB load), but it returns the identical Flux for *all* topics. Every subscriber — regardless of `topicId` — is wired as a downstream operator on the same hot publisher. Reactor's `.share()` fan-out delivers each emitted item to every subscriber's operator chain synchronously before moving to the next item, so N subscribers each paying O(1) per message yields O(N×M) total work per poll cycle.

**Why `startTime=0` maximises impact:** `TopicMessageFilter` accepts `startTime=0` (passes `@Min(0)` and `isValidStartTime()` since `0 <= DomainUtils.now()`). [6](#0-5) 

With `startTime=0` the timestamp half of `filterMessage` (`message.getConsensusTimestamp() >= 0`) is always `true`, so the only work done is the `topicId` equality check — but that check still executes for every message × every subscriber.

**No connection or rate limiting exists.** There is no gRPC interceptor, no per-IP cap, and no maximum subscriber count enforced anywhere. `subscriberCount` in `TopicMessageServiceImpl` is a metrics gauge only, not a guard. [7](#0-6) 

### Impact Explanation
With default settings (500 ms poll interval, 5,000 messages/poll), an attacker holding N=10,000 open subscriptions forces 50,000,000 `filterMessage` evaluations every 500 ms on the gRPC process's bounded-elastic thread pool. Each evaluation involves two object comparisons and a long comparison — cheap individually but collectively sufficient to saturate CPU cores, stall the shared Flux's emission loop, and cause backpressure buffer overflows (`maxBufferSize` default 16,384) for legitimate subscribers, effectively denying service to all users of the mirror node's topic subscription API. This is a non-network-based DoS (pure CPU/memory exhaustion within the process).

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. Opening thousands of long-lived gRPC streaming connections is trivial with any standard gRPC client library. The attack is repeatable, requires no special knowledge beyond the public API schema, and is effective only when the deployment uses `SHARED_POLL` mode (not the default `REDIS`). Operators who switch to `SHARED_POLL` to reduce Redis dependency are fully exposed.

### Recommendation
1. **Per-subscriber topic isolation in `SharedPollingTopicListener`:** Key the shared Flux by `topicId` (as `RedisTopicListener` already does with its `topicMessages` `ConcurrentHashMap`) so that only subscribers of the same topic share a Flux.
2. **Connection/subscription rate limiting:** Add a gRPC `ServerInterceptor` that enforces a maximum number of concurrent subscriptions per IP or per authenticated identity, and a global cap on total active subscriptions.
3. **Enforce a maximum subscriber count:** Promote `subscriberCount` from a metric to an enforced limit; reject new subscriptions when the count exceeds a configurable threshold.
4. **Bound `startTime` from below:** Consider rejecting `startTime` values older than a configurable horizon (e.g., 24 hours) to prevent degenerate filter behaviour.

### Proof of Concept

**Preconditions:** Mirror node deployed with `hiero.mirror.grpc.listener.type=SHARED_POLL` (or attacker can influence config). No authentication required.

**Steps:**

```
# Using grpcurl or any gRPC client, open N parallel streaming calls:
for i in $(seq 1 10000); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":'$i'},"consensusStartTime":{"seconds":0,"nanos":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

**Trigger:** Any topic on the network receives messages. The shared poller fetches up to 5,000 messages per 500 ms cycle and fans them out to all 10,000 subscriber filter chains simultaneously.

**Result:** CPU on the gRPC process spikes to 100% across all cores; legitimate subscribers experience buffer overflow errors (`BufferOverflowStrategy.ERROR`) and are disconnected; new subscriptions time out.
