### Title
Unauthenticated Distributed Circuit-Breaker Tripping via Retry-Amplified inFlightReq 503s on GraphQL Service

### Summary
The GraphQL service's Traefik middleware chain combines a global circuit breaker (`ResponseCodeRatio(500,600,0,600) > 0.25`), a per-IP in-flight request cap (`inFlightReq: amount: 5`), and a retry middleware (`attempts: 3`) with no per-IP rate limiting. An unprivileged attacker using multiple source IPs can deliberately exceed the per-IP inFlightReq cap to generate HTTP 503 responses, which the retry middleware amplifies 3Ã— before the global circuit breaker counts them, causing the circuit breaker to trip and partition the GraphQL service from all legitimate consumers.

### Finding Description

**Root cause and failed assumption:**

The design assumes the per-IP `inFlightReq` limit (5 concurrent requests) is sufficient to prevent abuse. This assumption fails because:

1. **No `rateLimit` middleware exists on GraphQL.** Rosetta has `rateLimit: average: 10` per host; GraphQL has no equivalent. [1](#0-0) 

2. **Middleware ordering creates retry amplification.** In Traefik's chain, `circuitBreaker` is outermost, `inFlightReq` is middle, and `retry` is innermost (closest to backend). When `inFlightReq` rejects a request with HTTP 503, the `retry` middleware retries it up to 3 times. Each retry also hits `inFlightReq` and receives another 503. The circuit breaker, being outermost, counts all 3 Ã— 503 responses toward `ResponseCodeRatio`. One attacker request exceeding the inFlightReq cap produces 3 counted 5xx responses.

3. **The circuit breaker is global, not per-IP.** It trips based on the aggregate ratio across all sources. An attacker generating enough 5xx volume relative to legitimate traffic trips it for everyone.

4. **Normal `account(evmAddress: ...)` queries return HTTP 200** even for non-existent addresses (GraphQL embeds errors in the response body, not HTTP status). The only reliable way to generate 5xx from this endpoint without backend failure is to exceed `inFlightReq`. [4](#0-3) 

**Exploit flow:**

1. Attacker deploys N source IPs (cloud instances, botnet, or proxies).
2. Each IP sends 6+ concurrent POST requests to `/graphql/alpha` with `account(input: {evmAddress: "0x<valid_hex>"})`.
3. The 6th+ concurrent request per IP is rejected by `inFlightReq` with HTTP 503.
4. The `retry` middleware retries each 503 three times (all also rejected by `inFlightReq`).
5. The `circuitBreaker` counts 3 Ã— 503 per overflowing request per IP.
6. With N IPs each generating K overflow requests, total 5xx count = N Ã— K Ã— 3.
7. When `5xx_count / total_count > 0.25`, the circuit breaker opens and returns 503 to ALL consumers.

### Impact Explanation

When the circuit breaker trips, Traefik stops forwarding any requests to the GraphQL backend and returns 503 to all clients â€” including legitimate downstream consumers â€” until the error ratio drops below the threshold. This is a complete service partition. The GraphQL service is the only API surface for the `account` query via GraphQL; there is no fallback. The circuit breaker recovery requires the 5xx ratio to drop, which only happens if the attacker stops or legitimate traffic overwhelms the attack volume. Since there is no rate limiting, the attacker can sustain the attack indefinitely at minimal cost.

### Likelihood Explanation

The attack requires no credentials, no special knowledge of the system, and no exploitation of application logic â€” only the ability to send concurrent HTTP POST requests from multiple IPs. Cloud providers offer cheap egress IPs. The `inFlightReq` cap of 5 is very low, making it trivial to exceed. The 25% circuit breaker threshold is reachable with a modest number of attacking IPs relative to legitimate traffic. The attack is fully repeatable and automatable with standard HTTP tooling (e.g., `wrk`, `hey`, `ab`).

### Recommendation

1. **Add a `rateLimit` middleware to the GraphQL chain**, mirroring the Rosetta configuration (`rateLimit: average: 10, sourceCriterion: requestHost: true`). This caps per-IP request rate before requests reach `inFlightReq`, preventing deliberate 503 generation at scale.
2. **Reorder middlewares** so `retry` is applied before `inFlightReq` in the chain, or configure `retry` to not retry on 503 responses generated by `inFlightReq` (Traefik supports `retryOn` configuration).
3. **Raise the `inFlightReq` amount** or replace it with a per-IP rate limit, which is more appropriate for a public API.
4. **Consider a per-source circuit breaker** rather than a single global one, so one attacker's traffic cannot affect the circuit state for all consumers.
