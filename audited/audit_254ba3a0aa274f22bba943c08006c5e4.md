### Title
Unauthenticated `/metrics` Endpoint Exposes `hiero_mirror_rosetta_response_bytes` Per-Route Histogram to Any External User

### Summary
The Rosetta API's `/metrics` endpoint is registered via `promhttp.Handler()` with no authentication check, making the full Prometheus default registry — including `hiero_mirror_rosetta_response_bytes` labeled by `method` and `route` — readable by any unauthenticated external caller. An attacker can enumerate all API routes and their historical response-size distributions, then target the largest-payload routes with high-volume requests to maximize bandwidth consumption per request sent.

### Finding Description
In `rosetta/app/middleware/metrics.go`, the `Routes()` function registers a bare `promhttp.Handler().ServeHTTP` at `/metrics` with no authentication wrapper:

```go
// rosetta/app/middleware/metrics.go lines 64-73
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,
            promhttp.Handler().ServeHTTP,  // no auth, serves default registry
        },
    }
}
``` [1](#0-0) 

The `responseBytesHistogram` is registered into the default Prometheus registry with `method` and `route` labels:

```go
responseBytesHistogram = prometheus.NewHistogramVec(prometheus.HistogramOpts{
    Name:    "hiero_mirror_rosetta_response_bytes",
    Buckets: sizeBuckets,  // [512, 1024, 10240, 25600, 51200]
    Help:    "Size (in bytes) of messages sent in response.",
}, []string{"method", "route"})
``` [2](#0-1) 

In `rosetta/main.go`, `metricsController` is added directly to the router, and the full middleware chain (`MetricsMiddleware → TracingMiddleware → CorsMiddleware`) contains no authentication layer: [3](#0-2) [4](#0-3) 

**Contrast with the REST API**, which explicitly gates its metrics endpoint behind `authenticate()` checking `config.metrics.config.authentication` (defaulting to `true`): [5](#0-4) 

No equivalent protection exists anywhere in the Rosetta path.

### Impact Explanation
Any unauthenticated caller receives the full Prometheus exposition text, including:
- `hiero_mirror_rosetta_response_bytes{method,route}` — bucket counts revealing which routes return 10 KB, 25 KB, or 50 KB+ responses
- `hiero_mirror_rosetta_request_duration{method,route,status_code}` — latency profiles per route
- `hiero_mirror_rosetta_request_inflight{method,route}` — live concurrency data

An attacker reads the histogram, identifies the route(s) whose `_bucket` counts are concentrated in the highest size buckets (e.g., `/block` or `/block/transaction` endpoints returning full block data), then floods those routes. Each attacker request triggers a disproportionately large server response, amplifying outbound bandwidth consumption. The metrics also reveal operational state (inflight counts, error rates) useful for timing attacks.

### Likelihood Explanation
Preconditions are minimal: the Rosetta port must be reachable from the internet (standard for a public blockchain API node). No credentials, tokens, or prior knowledge are required. The attack is fully scriptable: one `curl http://<host>/metrics` call yields the reconnaissance data; subsequent flood requests can be automated. The Rosetta API is by design a public-facing service, making network-level restriction unlikely in default deployments.

### Recommendation
Add an authentication wrapper to the `/metrics` handler in `Routes()`, mirroring the REST API pattern. At minimum:

1. Introduce a config option (e.g., `rosetta.metrics.auth.enabled`, `username`, `password`) in the Rosetta config struct.
2. Wrap `promhttp.Handler()` in a handler that checks HTTP Basic Auth before delegating, returning `401` on failure.
3. Alternatively, move the `/metrics` endpoint to a separate, non-public port (e.g., a dedicated admin/scrape port bound to `127.0.0.1` or a cluster-internal interface only), so Prometheus scrapers can reach it but external users cannot.

### Proof of Concept
```bash
# No credentials required
curl -s http://<rosetta-host>:<port>/metrics | grep hiero_mirror_rosetta_response_bytes

# Expected output (example):
# hiero_mirror_rosetta_response_bytes_bucket{application="rosetta",method="POST",route="/block",le="10240"} 0
# hiero_mirror_rosetta_response_bytes_bucket{application="rosetta",method="POST",route="/block",le="25600"} 142
# hiero_mirror_rosetta_response_bytes_bucket{application="rosetta",method="POST",route="/block",le="51200"} 389
# hiero_mirror_rosetta_response_bytes_bucket{application="rosetta",method="POST",route="/block",le="+Inf"} 412

# Attacker identifies /block as the highest-payload route, then floods it:
for i in $(seq 1 1000); do
  curl -s -X POST http://<rosetta-host>:<port>/block \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hiero","network":"mainnet"},"block_identifier":{"index":1}}' \
    -o /dev/null &
done
```

const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};

const metricsHandler = () => {
  initMetrics();
  const serializer = new PrometheusSerializer();
  // Maintain backward-compatible path: uriPath="/swagger" → endpoint at /swagger/metrics/
  const metricsPath = `${config.metrics.config.uriPath}/metrics/`;

  return function metricsMiddleware(req, res, next) {
    const {pathname} = new URL(req.url, 'http://localhost');
    const normalizedPath = pathname.endsWith('/') ? pathname : `${pathname}/`;

    if (normalizedPath === metricsPath) {
      if (!authenticate(req)) {
        res.set('WWW-Authenticate', 'Basic realm="Metrics"');
        return res.status(401).send('Unauthorized');
      }
      return exporter.collect().then(({resourceMetrics}) => {
        res.set('Content-Type', 'text/plain; charset=utf-8');
        res.send(serializer.serialize(resourceMetrics));
      });
