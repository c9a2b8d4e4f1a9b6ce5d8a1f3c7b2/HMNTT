### Title
Long-Zero EVM Address Bypasses Deletion Check, Causing Silent Incorrect Entity Resolution in Mirror Node Importer

### Summary
`EntityIdServiceImpl.findByEvmAddress` unconditionally short-circuits the database lookup when the supplied 20-byte EVM address matches the long-zero format (first 12 bytes all zero), returning a derived `EntityId` without ever consulting the database. The database path (`EntityRepository.findByEvmAddress`) enforces `deleted <> true`, but the long-zero derivation path in `DomainUtils.fromEvmAddress` has no such guard. Any unprivileged user can submit a Hedera transaction referencing a deleted entity by its long-zero EVM address, causing the mirror node importer to silently resolve and record the transaction against the deleted entity, corrupting the mirror node's transaction history.

### Finding Description

**Exact code path:**

`EntityIdServiceImpl.findByEvmAddress` — [1](#0-0) 

```java

    var id = Optional.ofNullable(DomainUtils.fromEvmAddress(evmAddress))   // line 170
            .or(() -> entityRepository.findByEvmAddress(evmAddress).map(EntityId::of)); // line 171
    ...
}
```

The `.or(...)` lambda is only evaluated if `DomainUtils.fromEvmAddress` returns `null`. For any long-zero address, it returns a non-null `EntityId`, so the lambda — and therefore the database — is **never reached**.

`DomainUtils.fromEvmAddress` — [2](#0-1) 

```java
public static EntityId fromEvmAddress(byte[] evmAddress) {
    if (isLongZeroAddress(evmAddress)) {
        final var num = Longs.fromByteArray(Arrays.copyOfRange(evmAddress, 12, 20));
        return EntityId.of(commonProperties.getShard(), commonProperties.getRealm(), num);
    }
    return null;
}
```

`isLongZeroAddress` returns `true` for any 20-byte array whose first 12 bytes are all zero. [3](#0-2) 

The database query that is bypassed: [4](#0-3) 

```sql
select id from entity where evm_address = ?1 and deleted <> true
```

**Root cause / failed assumption:** The code assumes that if an address is in long-zero format, the entity with the derived number is always valid and active. This assumption is wrong: entity `0.0.N` can be deleted, yet the long-zero address `0x000000000000000000000000[8-byte N]` will still resolve to `EntityId(0.0.N)` without any existence or deletion check.

**Why existing checks fail:** The test `lookupContractEvmAddressDeleted` only covers a deleted contract with a *custom* (non-long-zero) EVM address — it goes through the DB path and correctly returns empty. [5](#0-4)  There is no test for a deleted entity whose EVM address is in long-zero format. The `lookupAccountEvmAddress` test confirms that a long-zero address resolves to an `EntityId` with no DB entity present at all. [6](#0-5) 

The `notify()` method correctly skips deleted entities before warming the cache, [7](#0-6)  but this does not help: the long-zero path in `findByEvmAddress` bypasses both the cache and the DB, so a cache miss still resolves to the deleted entity.

### Impact Explanation

The mirror node importer processes every confirmed Hedera transaction, including **failed** ones (e.g., `CONTRACT_DELETED`, `INVALID_CONTRACT_ID`). When a failed transaction referencing a deleted entity by long-zero EVM address is processed, `EntityIdServiceImpl.findByEvmAddress` returns the deleted entity's `EntityId`. This `EntityId` is then used by `EntityRecordItemListener.onItem` to build and persist the `Transaction` record, [8](#0-7)  associating the transaction with the deleted entity in the mirror node's database. The result is silent data corruption: the mirror node's transaction history incorrectly links activity to a deleted entity. Additionally, if `HIERO_MIRROR_IMPORTER_PARSER_HALTONERROR=true` and downstream processing throws on the stale entity, `handleRecoverableError` escalates to a `ParserException` that halts the importer entirely. [9](#0-8) 

### Likelihood Explanation

No special privileges are required. Any Hedera user can:
1. Identify a deleted entity number `N` (publicly visible via the mirror node REST API).
2. Submit any contract-call or crypto-transfer transaction to the Hedera network with a `ContractID.evm_address` set to the 20-byte long-zero address for `N`.
3. The network records the transaction as failed; the mirror node importer processes it unconditionally.

This is repeatable, cheap (only transaction fees), and requires no insider access or cryptographic capability.

### Recommendation

In `EntityIdServiceImpl.findByEvmAddress`, after deriving an `EntityId` from the long-zero address, verify the entity exists and is not deleted before returning it:

```java
private Optional<EntityId> findByEvmAddress(byte[] evmAddress, boolean throwRecoverableError) {
    var longZeroId = DomainUtils.fromEvmAddress(evmAddress);
    if (longZeroId != null) {
        // Confirm the entity actually exists and is not deleted
        boolean exists = entityRepository.findById(longZeroId.getId())
                .filter(e -> e.getDeleted() == null || !e.getDeleted())
                .isPresent();
        if (exists) {
            return Optional.of(longZeroId);
        }
        // Fall through to custom evm_address lookup
    }
    var id = entityRepository.findByEvmAddress(evmAddress).map(EntityId::of);
    if (id.isEmpty() && throwRecoverableError) {
        Utility.handleRecoverableError("Entity not found for EVM address {}", Hex.encodeHexString(evmAddress));
    }
    return id;
}
```

Alternatively, add a dedicated repository method `existsByIdAndDeletedIsFalse(long id)` to avoid loading the full entity. Also add a unit test covering the case where a deleted entity's long-zero EVM address is looked up.

### Proof of Concept

**Preconditions:**
- Entity `0.0.12345` (contract) exists and is then deleted on the Hedera network.
- Mirror node has processed the delete transaction; `entity` table row for `id=12345` has `deleted=true`.

**Steps:**
1. Compute the long-zero EVM address for `0.0.12345`:
   `bytes = [0x00]*12 + [0x00,0x00,0x00,0x00,0x00,0x00,0x30,0x39]` (hex: `0x0000000000000000000000000000000000003039`)
2. Submit a `ContractCallTransaction` to the Hedera network with:
   ```
   ContractID { evm_address: 0x0000000000000000000000000000000000003039 }
   ```
3. The network confirms the transaction as `CONTRACT_DELETED` (failed).
4. The mirror node importer receives the record stream entry and calls `entityIdService.lookup(contractId)`.
5. `findByEvmAddress` is invoked with the long-zero bytes.
6. `DomainUtils.fromEvmAddress` returns `EntityId(0.0.12345)` — no DB query is made.
7. The importer persists a `Transaction` row with `entity_id=12345` (the deleted entity), bypassing the `deleted <> true` guard that the DB path would have enforced.
8. **Result:** Mirror node database contains a transaction incorrectly linked to the deleted entity `0.0.12345`, with no error logged and no importer halt (unless `haltOnError=true`).
