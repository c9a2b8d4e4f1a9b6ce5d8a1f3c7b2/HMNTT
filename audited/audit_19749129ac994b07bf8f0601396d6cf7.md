### Title
Unbounded Initcode Size in `buildContractCreateTransactionBody()` Bypasses Gas-Based Throttle, Enabling Resource Exhaustion

### Summary
Any unprivileged caller can submit an `eth_call` or `eth_estimateGas` request with `to=null`/`value=0` and a maximally-sized `data` field (up to 524,288 bytes). `buildContractCreateTransactionBody()` wraps the entire payload as initcode via `Bytes.wrap(params.getCallData())` with no size guard. The existing throttle is keyed solely on `request.getGas()`, not on data size, so the attacker's per-request CPU and memory cost is orders of magnitude higher than what the throttle accounts for, enabling a sustained >30% resource increase with no special privileges.

### Finding Description

**Trigger condition** — `TransactionExecutionService.execute()` line 74:
```java
final var isContractCreate = params.getReceiver().isZero();
```
`receiver` is set to `Address.ZERO` whenever `to` is null or empty (`ContractController.java` lines 60-63). With `value=0` the `hasTo()` bean-validation method (`ContractCallRequest.java` lines 53-57) returns `true` unconditionally (`isValidToField = value <= 0 || …`), so the request passes `@Valid` with `to=null`.

**Unchecked wrapping** — `TransactionExecutionService.java` lines 173-183:
```java
private TransactionBody buildContractCreateTransactionBody(
        final CallServiceParameters params, final long estimatedGas, final long maxLifetime) {
    return defaultTransactionBodyBuilder(params)
            .contractCreateInstance(ContractCreateTransactionBody.newBuilder()
                    .initcode(Bytes.wrap(params.getCallData()))   // ← entire callData, no size check
                    ...
```
`params.getCallData()` is the raw byte array decoded from `request.getData()`. The `@Hex` annotation on `data` (`ContractCallRequest.java` line 28-29) uses the default `maxLength = 1_048_576` hex characters (`Hex.java` line 27), permitting up to **524,288 bytes** of binary initcode per request.

**Throttle gap** — `ThrottleManagerImpl.throttle()` (lines 37-48) consumes tokens from two buckets:
- `rateLimitBucket`: 1 token per request (500 RPS default)
- `gasLimitBucket`: `scaleGas(request.getGas())` tokens — keyed on the *declared gas field*, not on data size

A request with `gas=15_000_000` (the maximum) and `data=<524KB>` consumes exactly the same throttle tokens as a request with `gas=15_000_000` and `data=4 bytes`. The 524KB initcode causes:
1. O(n) heap allocation for the byte array
2. O(n) protobuf serialization inside `ContractCreateTransactionBody`
3. O(n) jump-destination analysis in the Besu EVM executor before any gas is charged
4. Even under EIP-3860 (49,152-byte initcode cap), the EVM still processes up to 49,152 bytes fully within the 15M gas budget (200 gas/byte × 49,152 = 9.8M gas < 15M)

### Impact Explanation
At the default 500 RPS limit, an attacker sending 524KB initcode per request forces the node to process **~256 MB/second** of initcode data, versus ~2 KB/second for normal 4-byte calls — a factor of >100,000×. Even bounded by EIP-3860 to 49,152 bytes, the ratio is ~6,000×. Both scenarios far exceed the 30% resource-increase threshold. The impact is CPU exhaustion (jump-destination analysis, protobuf serialization) and heap pressure, degrading or denying service to legitimate callers.

### Likelihood Explanation
The exploit requires zero privileges: no API key, no account, no HBAR. The endpoint is public (`POST /api/v1/contracts/call`). The payload is trivially constructed (null `to`, zero `value`, maximum-length hex string). It is fully repeatable and automatable. The attacker is rate-limited to 500 RPS by the RPS bucket, but that is sufficient to sustain the attack.

### Recommendation
1. **Add an explicit callData size cap in the contract-create path.** Before calling `buildContractCreateTransactionBody()`, reject requests where `params.getCallData().length` exceeds a configured maximum (e.g., 49,152 bytes to align with EIP-3860, or a lower operator-chosen limit).
2. **Include data size in the throttle cost.** Extend `ThrottleManagerImpl.throttle()` to consume additional tokens proportional to `request.getData().length()`, so large-initcode requests are throttled more aggressively.
3. **Add a `@Size` or `@Hex(maxLength=…)` constraint** on `ContractCallRequest.data` that reflects the actual intended limit (the OpenAPI spec documents 262,146 hex chars = 131,073 bytes, but the Java annotation allows 1,048,576 chars = 524,288 bytes — these are inconsistent).

### Proof of Concept
```bash
# Build a 524,288-byte (1,048,576 hex-char) payload of valid-looking initcode
PAYLOAD="0x60806040$(python3 -c "print('00' * 524284)")39aabbf3"

curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d "{
    \"to\": null,
    \"value\": 0,
    \"gas\": 15000000,
    \"data\": \"$PAYLOAD\"
  }"
```
Repeat at 500 RPS (the RPS throttle limit). Each request causes the node to allocate and serialize 524KB as initcode and pass it to the EVM executor. Monitor CPU and heap on the mirror-node process; resource consumption will be substantially elevated compared to the same RPS with normal-sized calldata.
