### Title
Missing Connection Idle/Lifetime Limits in `grpcServerConfigurer()` Enables Indefinite Resource Retention After Network Partition

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only a thread executor and a per-connection call limit, omitting all connection lifetime controls (`maxConnectionIdle`, `maxConnectionAge`, keepalive). An unprivileged attacker can open connections, initiate long-lived streaming calls (e.g., `subscribeTopic`), then simulate a network partition by silently dropping packets. The server's TCP stack retains the connection as "established" indefinitely, holding all associated server-side resources with no mechanism to reclaim them.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35, `grpcServerConfigurer()`:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`NettyProperties` exposes only `maxConcurrentCallsPerConnection` (default: 5). No other Netty/gRPC server parameters are configured anywhere in the grpc module — confirmed by exhaustive search across all YAML configs and Java sources.

**Root cause:** `NettyServerBuilder` defaults for connection lifetime are effectively infinite:
- `maxConnectionIdle`: `Long.MAX_VALUE` (never evict idle connections)
- `maxConnectionAge`: `Long.MAX_VALUE` (connections never forcibly closed)
- `keepAliveTime`: `Long.MAX_VALUE` (no HTTP/2 PING probes sent)

No `IdleStateHandler` is added to the Netty pipeline, and no gRPC-level keepalive is configured. OS-level TCP keepalive defaults to ~2 hours on Linux and is not relied upon here.

**Failed assumption:** The design assumes that dead connections (post-partition) will be detected and cleaned up. They are not — the server's TCP stack sees the connection as `ESTABLISHED` with no data flowing, and gRPC never probes it.

**Why existing checks fail:** `maxConcurrentCallsPerConnection = 5` limits active RPC calls per connection but does not bound the number of simultaneous connections, nor does it detect or close connections where the remote peer has silently disappeared. A connection with 0 active calls (or with a stalled streaming call) is invisible to this check.

### Impact Explanation
Each retained dead connection holds:
- A TCP socket (file descriptor)
- Netty channel state and pipeline handlers
- A Reactor subscription chain (for `subscribeTopic` streaming calls)
- Thread pool slots in `applicationTaskExecutor` for in-progress stream handlers
- Potential database cursor/connection resources held by the retriever

An attacker opening thousands of connections and then partitioning them causes gradual file-descriptor exhaustion, thread pool saturation, and heap pressure. The gRPC server on port 5600 is publicly accessible with no authentication, making this a denial-of-service vector against a critical infrastructure component.

### Likelihood Explanation
**Preconditions:** Network access to port 5600. No credentials, no special privileges required.

**Feasibility:** The attacker connects, calls `subscribeTopic` (a legitimate, publicly documented streaming endpoint), then uses `iptables -A OUTPUT -p tcp --dport 5600 -j DROP` or equivalent to silently drop outgoing packets — the server receives no FIN/RST and retains the connection. This is repeatable in a loop from a single machine or botnet. The nginx proxy's `grpc_read_timeout 600s` only applies when nginx is deployed in front; direct access to port 5600 bypasses it entirely.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
serverBuilder.maxConnectionIdle(30, TimeUnit.SECONDS);
serverBuilder.maxConnectionAge(300, TimeUnit.SECONDS);
serverBuilder.maxConnectionAgeGrace(10, TimeUnit.SECONDS);
serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Expose these as configurable properties in `NettyProperties` (mirroring the existing `maxConcurrentCallsPerConnection` pattern) so operators can tune them per deployment. Additionally, consider adding a connection-count limit at the OS/load-balancer level.

### Proof of Concept
```bash
# Step 1: Open N streaming connections to the gRPC server
for i in $(seq 1 500); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <server>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 2: Simulate network partition — drop all outgoing packets to server
iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# Step 3: Observe server-side resource retention
# On server: watch -n1 'ss -tnp | grep 5600 | wc -l'
# Connections remain ESTABLISHED indefinitely; no cleanup occurs.
# Repeat until file-descriptor limit or thread pool is exhausted.
```
