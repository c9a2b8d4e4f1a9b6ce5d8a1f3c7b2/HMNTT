### Title
Unauthenticated Full Transaction Table Scan via Unbounded Timestamp Filter in `getOneAccount()`

### Summary
`getOneAccount()` in `rest/accounts.js` calls `parseTimestampFilters()` with `validateRange = false` and `allowOpenRange = true`, deliberately bypassing the configured `maxTimestampRange` (7-day) limit. Any unauthenticated user can supply a single open-ended `timestamp=gte:0` parameter, causing `doGetTransactions()` to issue a full transaction table scan. With the default `bindTimestampRange = false` configuration, no secondary bounding occurs, and concurrent exploitation exhausts the 10-connection DB pool, starving all other users of query capacity.

### Finding Description

**Exact code path:**

`getOneAccount()` at `rest/accounts.js` line 413 calls:
```javascript
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
``` [1](#0-0) 

The sixth argument `validateRange = false` disables the only range-size guard inside `parseTimestampFilters()`:
```javascript
if (validateRange) {                          // ŌåÉ skipped entirely
    if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
        throw new InvalidArgumentError(...);
    }
}
``` [2](#0-1) 

The fifth argument `allowOpenRange = true` suppresses the requirement for both a lower and upper bound:
```javascript
if (!allowOpenRange && eqValues.size === 0 && (lowerBoundFilterCount === 0 || upperBoundFilterCount === 0)) {
    throw new InvalidArgumentError('Timestamp range must have gt (or gte) and lt (or lte), or eq operator');
}
``` [3](#0-2) 

When only one bound is supplied, `difference` is `null` (both bounds must be non-null for it to be computed), so even if `validateRange` were `true`, the size check would still be skipped for open ranges:
```javascript
const difference = latest !== null && earliest !== null ? latest - earliest + 1n : null;
``` [4](#0-3) 

The resulting open range `Range(0n, null, '[]')` is passed directly to `doGetTransactions()`:
```javascript
const transactionsPromise = includeTransactions
    ? transactions.doGetTransactions(filters, req, timestampRange)
    : emptyTransactionsPromise;
``` [5](#0-4) 

Inside `doGetTransactions()` ŌåÆ `getTransactionTimestamps()`, `bindTimestampRange()` is called as the only remaining guard:
```javascript
const {range, next} = await bindTimestampRange(timestampRange.range, order);
``` [6](#0-5) 

But `bindTimestampRange` is a no-op when `bindTimestampRange = false` (the documented default):
```javascript
const bindTimestampRange = async (range, order) => {
  if (!queryConfig.bindTimestampRange) {
    return {range};   // ŌåÉ open range returned unchanged
  }
``` [7](#0-6) 

The default value is confirmed in documentation: [8](#0-7) 

The unbounded range is then embedded into the SQL query against the `transaction` table with no upper timestamp bound, causing a full sequential scan.

**Root cause:** The intentional `validateRange = false` flag in `getOneAccount()` was designed to allow historical point-in-time queries, but it also disables the only enforcement of `maxTimestampRange`. Combined with `allowOpenRange = true`, a single one-sided filter produces an unbounded range that propagates unchecked into the transaction query.

**Failed assumption:** The code assumes `bindTimestampRange = true` will be enabled in production to compensate, but the default is `false` and the documentation does not mark it as required.

### Impact Explanation

With the default configuration (`bindTimestampRange = false`, `maxConnections = 10`, `statementTimeout = 20000ms`):

- Each crafted request holds a DB connection for up to 20 seconds executing a full table scan.
- 10 concurrent requests from a single attacker exhaust the entire connection pool.
- All legitimate API calls ŌĆö including those used to observe gossip transaction propagation ŌĆö receive connection timeout errors (`connectionTimeout = 20000ms`).
- The attack is a complete, unauthenticated denial-of-service against the mirror node REST API's transaction visibility layer.

Severity: **High** ŌĆö full availability loss of the public REST API with no authentication required.

### Likelihood Explanation

- No authentication, API key, or rate-limiting is required; the endpoint `/api/v1/accounts/{id}` is fully public.
- The payload is a single HTTP query parameter: `?timestamp=gte:0`.
- The attack is trivially scriptable with `curl` or any HTTP client in a loop.
- The default configuration is vulnerable; operators must affirmatively enable `bindTimestampRange = true` to partially mitigate it.
- Repeatability is unlimited; the attacker can sustain the DoS indefinitely.

### Recommendation

1. **Remove `validateRange = false` from the `getOneAccount()` call** or replace it with a dedicated open-range size cap. The `maxTimestampRange` guard must apply even to open ranges.
2. **Enforce an upper bound on open ranges inside `parseTimestampFilters`**: when `allowOpenRange = true` and only one bound is provided, clamp the missing bound to `now` or `firstTimestamp + maxTimestampRangeNs`.
3. **Change the default of `bindTimestampRange` to `true`** so that the secondary guard in `getTransactionTimestamps()` is active by default.
4. **Add per-IP rate limiting** at the API gateway layer for the `/api/v1/accounts/{id}` endpoint.

### Proof of Concept

**Preconditions:** Default mirror node REST API deployment with `bindTimestampRange = false` (default). No credentials needed.

**Step 1 ŌĆö Single request demonstrating full table scan:**
```bash
curl "http://<mirror-node>:5551/api/v1/accounts/0.0.2?timestamp=gte:0"
```
This produces a query equivalent to:
```sql
SELECT ... FROM transaction t WHERE t.consensus_timestamp >= 0
ORDER BY t.consensus_timestamp DESC LIMIT 25
```
with no upper bound, scanning the entire `transaction` table.

**Step 2 ŌĆö Pool exhaustion (DoS):**
```bash
for i in $(seq 1 15); do
  curl "http://<mirror-node>:5551/api/v1/accounts/0.0.2?timestamp=gte:0" &
done
wait
```
15 concurrent requests saturate the 10-connection pool. All other API consumers receive connection timeout errors for up to 20 seconds per wave. Repeating the loop sustains the outage indefinitely.

**Step 3 ŌĆö Verify impact:**
```bash
curl "http://<mirror-node>:5551/api/v1/transactions"
# Returns: 503 / connection pool exhausted / timeout
```
