### Title
Unbounded `node.id` EQ Parameter Count Enables Memory and Database DoS in `getNetworkNodes()`

### Summary
The `getNetworkNodes()` method in `NetworkServiceImpl` iterates over all caller-supplied `node.id` parameters and adds each EQ-operator value into a `HashSet<Long>` with no upper bound on the list size. Because `NetworkNodeRequest.nodeIds` carries no `@Size` constraint (unlike other parameters such as `timestamp` which are capped at `@Size(max = 2)`), an unauthenticated attacker can submit thousands of distinct EQ values per request, growing the in-process `HashSet` and the resulting `Long[]` array that is forwarded verbatim to the database query, degrading service for all concurrent users.

### Finding Description

**Root cause — missing `@Size` on `nodeIds`:**

`NetworkNodeRequest.nodeIds` is declared as an unbounded `List<NumberRangeParameter>` with no size constraint: [1](#0-0) 

Compare this to the `timestamp` parameter in the same controller, which is correctly bounded: [2](#0-1) 

**Unbounded accumulation in `getNetworkNodes()`:**

Every EQ-operator entry is unconditionally added to a `HashSet<Long>`: [3](#0-2) 

The resulting set is then converted to a `Long[]` array and passed directly to the repository query: [4](#0-3) 

**Why existing checks are insufficient:**

- `getEffectiveLimit()` caps the *output* at 25 rows but places no restriction on the number of *input* filter parameters.
- The `@Min(1)` on `limit` is irrelevant to input parameter count.
- Spring Boot's embedded Tomcat defaults to `maxParameterCount = 10000`, so an attacker can legally supply up to 10 000 distinct `node.id=eq:N` values in a single GET request.

### Impact Explanation

Each request with N EQ values allocates an N-element `HashSet<Long>` on the heap and then an N-element `Long[]` that is embedded in a SQL `IN (…)` clause sent to the database. Under concurrent load from multiple attackers, heap pressure accumulates rapidly, GC pauses increase, and the database is burdened with large `IN` lists — degrading or denying service for all legitimate users. This matches the stated scope of medium-severity griefing with no economic damage to any individual user.

### Likelihood Explanation

No authentication or API key is required to call `GET /api/v1/network/nodes`. The attack requires only an HTTP client capable of repeating a query parameter thousands of times, which is trivially scriptable. It is fully repeatable and can be sustained indefinitely.

### Recommendation

Add a `@Size(max = N)` constraint on the `nodeIds` field in `NetworkNodeRequest`, consistent with how `timestamp` is bounded elsewhere:

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 25)   // or a similarly small constant
private List<NumberRangeParameter> nodeIds = List.of();
```

Additionally, enforce the constraint at the controller layer via `@Validated` on the controller class and ensure Bean Validation is triggered for `@RequestParameter`-bound objects.

### Proof of Concept

```bash
# Generate a request with 5000 distinct EQ node.id parameters
python3 -c "
params = '&'.join(f'node.id={i}' for i in range(5000))
print(f'GET /api/v1/network/nodes?{params}')
" | xargs -I{} curl -s "http://<mirror-node-host>/{}"
```

Repeat concurrently from multiple clients (e.g., with `ab -n 500 -c 50`) and observe heap growth via JVM metrics and increased query latency for other users.
