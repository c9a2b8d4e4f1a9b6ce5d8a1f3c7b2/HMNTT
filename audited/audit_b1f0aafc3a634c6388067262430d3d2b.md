### Title
Unauthenticated `/metrics` Endpoint with Silent Audit-Log Suppression in Rosetta Middleware

### Summary
The Rosetta service exposes Prometheus metrics at `/metrics` via `promhttp.Handler()` with no authentication or IP restriction at the application layer. Simultaneously, `TracingMiddleware` in `trace.go` classifies `/metrics` as an `internalPath` and logs all accesses only at `log.Debug` level — which is suppressed entirely under the production default `info` log level — leaving no audit trail of enumeration. An unprivileged external attacker who can reach the service port receives a full metrics dump silently, with per-route counters for `/construction/submit` and other routes revealing transaction submission throughput patterns.

### Finding Description

**Code path 1 — No authentication on `/metrics`:**

`rosetta/app/middleware/metrics.go` lines 64–73 register the endpoint using the bare `promhttp.Handler()` with no auth wrapper:

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,           // "/metrics"
            promhttp.Handler().ServeHTTP,   // no auth
        },
    }
}
``` [1](#0-0) 

This contrasts with the REST API's `metricsHandler.js`, which has an explicit `authenticate(req)` guard before serving metrics. No equivalent guard exists in the Rosetta path.

**Code path 2 — Silent logging:**

`rosetta/app/middleware/trace.go` line 19 places `metricsPath` in `internalPaths`:

```go
var internalPaths = map[string]bool{livenessPath: true, metricsPath: true, readinessPath: true}
``` [2](#0-1) 

Lines 55–59 then emit the access log entry at `log.Debug` for any path in that map:

```go
if internalPaths[path] {
    log.Debug(message)
} else {
    log.Info(message)
}
``` [3](#0-2) 

`main.go` initialises the logger at `"info"` and then reconfigures it from `rosettaConfig.Log.Level`: [4](#0-3) 

The default configured level is `info`, so `log.Debug` calls are discarded. Every `/metrics` request is served and returns 200 with the full metrics payload, but **zero log lines are written** in a standard production deployment.

**Code path 3 — Middleware chain confirms single-port exposure:**

`main.go` lines 217–219 wrap the router in `MetricsMiddleware → TracingMiddleware → CorsMiddleware` and bind everything to a single `rosettaConfig.Port`: [5](#0-4) 

**Code path 4 — GCP gateway routes `/rosetta/*` prefix without path exclusion:**

`charts/hedera-mirror-rosetta/values.yaml` lines 68–83 configure the GCP gateway to match path prefix `/rosetta` and rewrite to `/`, with no exclusion for `/rosetta/metrics`: [6](#0-5) 

A request to `https://<host>/rosetta/metrics` is rewritten to `/metrics` and forwarded to the service, bypassing the Traefik ingress path allowlist (which does not list `/rosetta/metrics`). [7](#0-6) 

**Root cause / failed assumption:** The name `internalPaths` implies these paths are "internal-only," but the map is used exclusively to select a log level. No enforcement of that assumption — no IP allowlist, no auth middleware, no network policy scoped to `/metrics` — exists at the application layer.

### Impact Explanation

The metrics payload includes `hiero_mirror_rosetta_request_duration`, `hiero_mirror_rosetta_request_bytes`, `hiero_mirror_rosetta_response_bytes`, and `hiero_mirror_rosetta_request_inflight`, all labelled by `route`. [8](#0-7) 

The `/construction/submit` route counter directly reveals transaction submission throughput over time. An attacker polling `/metrics` at regular intervals can build a precise time-series of submission rates, identify low-activity windows or high-load periods, and time a targeted suppression or resource-exhaustion attack against upstream consensus nodes when the mirror node is already under stress. Because no `log.Info` entry is ever written, the enumeration activity is invisible to operators and SIEM systems monitoring the production log stream.

### Likelihood Explanation

**Preconditions:** The attacker needs network reachability to the rosetta service port. This is satisfied in any of three realistic scenarios: (1) the GCP gateway is enabled (`global.gateway.enabled: true`) and the attacker sends `GET /rosetta/metrics` — the gateway rewrites it to `/metrics` and forwards it; (2) the service is exposed via a `NodePort` or `LoadBalancer` type (common in non-GKE deployments); (3) the attacker has any foothold inside the cluster (e.g., a compromised pod). No credentials, tokens, or special headers are required. The attack is trivially repeatable with a simple `curl` or a Prometheus scrape config pointed at the public endpoint.

### Recommendation

1. **Add authentication to the `/metrics` handler** in `rosetta/app/middleware/metrics.go`. Wrap `promhttp.Handler()` with a middleware that checks a configurable bearer token or basic-auth credential before serving the response, mirroring the `authenticate()` pattern already present in `rest/middleware/metricsHandler.js`.

2. **Elevate the log level for `/metrics` accesses to `log.Info`**, or introduce a separate audit-log entry at `Info` level that records the client IP and timestamp regardless of the path classification. The current `internalPaths` design silently hides all scrape activity from production log pipelines.

3. **Exclude `/rosetta/metrics` from the GCP gateway path-prefix rule**, or add an explicit deny rule for that path at the gateway layer, so that even if the application-level auth is misconfigured the endpoint is not reachable from the public internet.

### Proof of Concept

```bash
# 1. Confirm the GCP gateway rewrites /rosetta/* to /*
#    (or access the service port directly if exposed)
curl -s https://<rosetta-host>/rosetta/metrics

# Expected: HTTP 200 with full Prometheus text exposition, e.g.:
# hiero_mirror_rosetta_request_duration_count{application="rosetta",
#   method="POST",route="/construction/submit",status_code="200",...} 4712

# 2. Poll repeatedly to build a throughput time-series
for i in $(seq 1 60); do
  curl -s https://<rosetta-host>/rosetta/metrics \
    | grep 'construction/submit.*_count'
  sleep 60
done

# 3. Verify no log.Info entry is produced on the server
#    (check production log stream — no access record for /metrics appears)
```

No credentials are required at any step. The server returns 200 and the full metrics payload on every request.
