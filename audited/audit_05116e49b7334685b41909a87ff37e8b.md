### Title
Unbounded Key Count and Timestamp Range in `getHookStorageChange()` Enables Unauthenticated DoS via Expensive DB Queries

### Summary
An unprivileged external user can craft a historical hook storage request with an arbitrarily large number of keys all within the key range bounds and a maximum-width timestamp range. This bypasses the `keysInRange.isEmpty() && requestHasKeys` early-return guard and causes `findByKeyInAndTimestampBetween()` to execute a `DISTINCT ON (key)` native SQL query with a large `IN` clause over an unbounded timestamp window, consuming significant database resources. Repeated requests can degrade mirror node processing capacity.

### Finding Description

**Exact code path:**

In `HookServiceImpl.java`, `getHookStorageChange()` (lines 84–118):

```
keys          = request.getKeys();           // user-controlled, no size limit
requestHasKeys = !keys.isEmpty();
keysInRange   = request.getKeysInRange();    // filtered subset of keys

if (keysInRange.isEmpty() && requestHasKeys) {   // line 93 — the guard
    return new HookStorageResult(ownerId, List.of());
}
``` [1](#0-0) 

The guard fires **only** when `keysInRange` is empty AND keys were provided. An attacker who supplies keys that all fall within `[keyLowerBound, keyUpperBound]` makes `keysInRange` non-empty, so the guard is never triggered.

After the guard, the code passes the full `keys` collection (not `keysInRange`) and user-controlled timestamp bounds directly to the repository:

```java
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
``` [2](#0-1) 

The underlying native SQL query is:

```sql
select distinct on (key) ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id = :hookId
  and key in (:keys)
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
``` [3](#0-2) 

**Root cause — three missing constraints:**

1. **No limit on the number of keys.** `HookStorageRequest.keys` is a plain `Collection<byte[]>` with no `@Size` or any other validation constraint. [4](#0-3) 

2. **No limit on the timestamp range width.** `timestampLowerBound` and `timestampUpperBound` come directly from `request.getTimestamp()` with no maximum-span enforcement. [5](#0-4) 

3. **`DISTINCT ON` forces a full pre-limit scan.** PostgreSQL must evaluate all rows matching the `WHERE` clause before applying `LIMIT` from the `Pageable`, so the pagination does not bound the DB work. [6](#0-5) 

### Impact Explanation
A single request with, e.g., 10,000 keys all within range and a timestamp span of `[0, Long.MAX_VALUE]` forces PostgreSQL to scan the entire `hook_storage_change` table for the given `owner_id`/`hook_id`, compute `DISTINCT ON (key)` across all historical rows, and evaluate a large `IN` list. Concurrent such requests from one or more attackers can saturate DB CPU and I/O, degrading or halting query processing for legitimate mirror node consumers — consistent with the ≥30% processing degradation threshold stated in scope.

### Likelihood Explanation
The endpoint is reachable by any unprivileged external user (no authentication is visible in the service layer). The exploit requires only an HTTP client capable of sending a well-formed request with many keys and a wide timestamp range — no special knowledge, credentials, or brute force is needed. It is trivially repeatable and scriptable, making sustained DoS straightforward.

### Recommendation
Apply all three fixes:

1. **Limit key count:** Add `@Size(max = N)` (e.g., `max = 100`) to `HookStorageRequest.keys`.
2. **Limit timestamp range width:** In `getHookStorageChange()`, reject or clamp requests where `timestampUpperBound - timestampLowerBound` exceeds a configured maximum (e.g., 24 hours in nanoseconds).
3. **Pass `keysInRange` (not `keys`) to the repository:** The guard already computes the filtered set; use it to avoid passing unbounded user input to the DB.
4. **Add a DB-level statement timeout** for this query class to bound worst-case execution time.

### Proof of Concept

```
GET /api/v1/hooks/{ownerId}/{hookId}/storage?timestamp=gte:0&timestamp=lte:9999999999999999999
    &key=<key1_within_range>&key=<key2_within_range>...&key=<key10000_within_range>
    &limit=25
```

Preconditions:
- Any valid `ownerId` and `hookId` (can be discovered via public hook listing endpoints).
- Keys chosen so that all fall between the default or explicitly supplied `keyLowerBound` and `keyUpperBound`.

Trigger: Send the request. `keysInRange` will be non-empty (all keys are in range), so the guard at line 93 is skipped. `findByKeyInAndTimestampBetween()` executes with 10,000 keys and a timestamp span covering the full history.

Result: PostgreSQL executes a full-table `DISTINCT ON` scan with a 10,000-element `IN` list. Repeat the request concurrently (e.g., 20 threads) to saturate DB resources and degrade mirror node processing capacity.
