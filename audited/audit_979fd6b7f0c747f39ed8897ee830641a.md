### Title
Unauthenticated Repeated DB Exhaustion via All-Zero EVM Address in GraphQL `getByEvmAddressAndType()`

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl` routes any all-zero EVM address (`0x0000000000000000000000000000000000000000`) unconditionally to `entityRepository.findById(0)`. Entity ID 0 never exists in Hedera, so every such request produces a guaranteed DB miss. The GraphQL module has no rate limiting and no result caching, allowing any unauthenticated external user to flood the DB connection pool with repeated identical queries at zero cost.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 51-54) accepts an `evmAddress` argument with no authentication and calls `entityService.getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)`. [1](#0-0) 

Inside `EntityServiceImpl.getByEvmAddressAndType()`: [2](#0-1) 

1. `decodeEvmAddress("0x0000000000000000000000000000000000000000")` strips the `0x` prefix and hex-decodes to 20 zero bytes. [3](#0-2) 

2. `ByteBuffer.wrap(evmAddressBytes)` wraps the 20 bytes. `buffer.getInt()` reads bytes 0–3 → `0`. `buffer.getLong()` reads bytes 4–11 → `0`. The condition `0 == 0 && 0 == 0` is always true.

3. The second `buffer.getLong()` reads bytes 12–19 → `0`, so `entityRepository.findById(0)` is called. [4](#0-3) 

4. `EntityRepository` extends `CrudRepository<Entity, Long>` with no `@Cacheable` annotation — every call issues a live DB query. [5](#0-4) 

**Why existing checks are insufficient:**

- `GraphQlConfiguration` applies `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which limit the shape of a single query but do not limit the rate of repeated requests. [6](#0-5) 

- `CachedPreparsedDocumentProvider` caches only the **parsed GraphQL document** (the query string AST), not query results. The DB call is still made on every execution. [7](#0-6) 

- The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists exclusively in the `web3` module and is not wired into the `graphql` module at all. [8](#0-7) 

### Impact Explanation
An attacker can saturate the DB connection pool by sending a high volume of identical, trivially simple GraphQL queries. Each query is guaranteed to miss (entity ID 0 does not exist), so no caching ever absorbs the load. This degrades or denies service to legitimate users of the GraphQL API and indirectly stresses the shared database used by other mirror-node services. Severity matches the stated scope: griefing / resource exhaustion with no direct economic damage.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and a single fixed payload. The query is syntactically valid, passes all parser and complexity checks, and the attacker needs no knowledge of the system beyond the public GraphQL schema. It is trivially scriptable and repeatable indefinitely.

### Recommendation
Apply at least one of the following mitigations in the `graphql` module:

1. **Rate limiting per IP/client** — add a Bucket4j or similar token-bucket filter analogous to `ThrottleConfiguration` in the `web3` module.
2. **Negative-result caching** — cache `Optional.empty()` results for a short TTL (e.g., 30 s) keyed on the decoded EVM address bytes, so repeated identical lookups do not reach the DB.
3. **Reject the zero address early** — validate that the decoded EVM address is not the all-zero address before issuing any DB query, returning an appropriate error or empty result immediately.

### Proof of Concept
```
# Send 10 000 rapid GraphQL queries with the all-zero EVM address
for i in $(seq 1 10000); do
  curl -s -X POST http://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input:{evmAddress:\"0000000000000000000000000000000000000000\"}){ id } }"}' \
    &
done
```

Each request reaches `EntityServiceImpl.getByEvmAddressAndType()`, passes the `buffer.getInt()==0 && buffer.getLong()==0` check, and issues `SELECT * FROM entity WHERE id = 0 AND deleted IS NOT TRUE` against the database. With no rate limiting or negative caching, all requests hit the DB concurrently, exhausting the connection pool.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }
