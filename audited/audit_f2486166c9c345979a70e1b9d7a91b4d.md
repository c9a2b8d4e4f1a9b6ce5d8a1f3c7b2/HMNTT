### Title
Unbounded `spender.id` IN-clause Amplification via UNION ALL Enables Connection Pool Exhaustion in `getNfts()`

### Summary
An unprivileged external user can craft a request to `/api/v1/accounts/{id}/nfts` that triggers all three subqueries (lower, inner, upper) in `getQuery()`, each embedding up to 100 `spender.id` equality values in an `IN(...)` clause, producing a three-way `UNION ALL` query of maximum complexity. Flooding the endpoint with concurrent such requests holds database connections for the full `statementTimeout` duration, exhausting the connection pool and degrading all concurrent request processing by more than 30%.

### Finding Description

**Exact code path:**

`accountController.js` `extractNftMultiUnionQuery()` collects every `spender.id` filter with operator `eq` into `spenderIdInFilters` with no dedicated count cap: [1](#0-0) 

`validateFilters()` only rejects multiple range (`lt/lte/gt/gte`) operators on `spenderIdFilters`; it never limits the size of `spenderIdInFilters`: [2](#0-1) 

The global `maxRepeatedQueryParameters` (default **100**) is the only cap, applied in `buildFilters()`: [3](#0-2) 

In `getQuery()`, all three subqueries are generated when `lower`, `inner`, and `upper` are all non-empty, and each subquery receives the **full** `spenderIdInFilters` list: [4](#0-3) 

Each subquery embeds the entire IN clause: [5](#0-4) 

The three subqueries are joined with `UNION ALL` and an outer `ORDER BY … LIMIT`: [6](#0-5) 

`getNfts()` executes this via `super.getRows()` → `pool.queryQuietly()`, holding a pool connection for the full query duration: [7](#0-6) [8](#0-7) 

The pool has a finite `maxConnections` and a `statementTimeout` that bounds — but does not eliminate — the hold time: [9](#0-8) 

**Root cause:** `spenderIdInFilters` (eq-operator spender IDs) has no dedicated count limit; the only guard is the global `maxRepeatedQueryParameters` (100). Combined with the 3-subquery UNION ALL structure, a single request can inject a 100-element IN clause into each of three subqueries, producing a query of maximum planner complexity that holds a connection for the full `statementTimeout`.

**Failed assumption:** The design assumes that the global repeated-parameter cap (100) is sufficient to bound per-request DB cost. It is not, because the IN clause is replicated across all three UNION ALL branches, multiplying the scan work 3×, and no per-client rate limiting exists in the application layer.

### Impact Explanation

The connection pool (`max: config.db.pool.maxConnections`) is a shared, finite resource. If an attacker sends `N` concurrent maximum-complexity requests where `N ≥ maxConnections`, every pool slot is occupied for up to `statementTimeout` milliseconds. All other concurrent API requests (transactions, balances, contracts, etc.) queue for a connection and time out or return errors. With a default pool of tens of connections and a `statementTimeout` of several seconds, a sustained flood of ~20–50 concurrent requests is sufficient to degrade overall throughput by well over 30%, satisfying the stated impact threshold.

### Likelihood Explanation

The endpoint is public and requires no authentication. The attacker needs only an HTTP client capable of sending concurrent requests. The crafted URL is trivial to construct and repeat in a loop. No CAPTCHA, API key, or per-IP rate limit is enforced at the application layer. The attack is fully repeatable and scriptable.

### Recommendation

1. **Add a dedicated cap on `spenderIdInFilters` count** in `validateFilters()` (e.g., max 10 eq values), independent of the global `maxRepeatedQueryParameters`.
2. **Enforce per-IP or per-client rate limiting** at the application or reverse-proxy layer for this endpoint.
3. **Set a conservative `statementTimeout`** (e.g., ≤ 5 000 ms) and ensure `maxConnections` is sized relative to expected concurrency.
4. Consider **query cost estimation** or a dedicated complexity guard before executing the UNION ALL path.

### Proof of Concept

```bash
# Build a URL with:
# - token.id range → activates lower + upper subqueries
# - serialnumber range → activates inner subquery (all three branches non-empty)
# - 100 spender.id eq values → 100-element IN clause in each of the 3 subqueries
# - limit=100 → maximum rows per subquery

BASE="http://<mirror-node-host>/api/v1/accounts/0.0.1001/nfts"
PARAMS="token.id=gte:0.0.1&token.id=lte:0.0.9999999&serialnumber=gte:1&serialnumber=lte:9999&limit=100"
for i in $(seq 1 100); do PARAMS="$PARAMS&spender.id=$i"; done

# Send 50 concurrent requests to exhaust the connection pool
for i in $(seq 1 50); do
  curl -s "$BASE?$PARAMS" &
done
wait
# Observe: legitimate requests to other endpoints begin timing out or returning 503
```
