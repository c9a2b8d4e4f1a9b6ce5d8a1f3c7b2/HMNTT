### Title
Unbounded Repeated Batch DB Queries via Non-Existent Contract Storage Slot Requests (Missing Negative Cache)

### Summary
In `ContractStateServiceImpl.findStorage()`, the `contractStateCache` is only populated with values that are returned by the database. For non-existent storage slots, the DB returns nothing, so `contractStateCache` is never populated and `cachedValue` is always `null`. The guard `cachedValue != null && cachedValue != EMPTY_VALUE` (line 65) therefore always falls through to `findStorageBatch()` on every call for a non-existent slot, with no caching protection. An unprivileged attacker can exploit this to cause repeated, amplified batch DB queries by continuously requesting non-existent slots.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 58â€“122.

**Root cause â€” missing negative caching:**

```java
// Line 63-69
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);

if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
return findStorageBatch(contractId, key);
```

`contractStateCache` is only written in `findStorageBatch()` at line 109:
```java
contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);
```
This write only executes inside the loop `for (final var contractSlotValue : contractSlotValues)` â€” i.e., only for slots the DB actually returns. For a non-existent slot, `contractSlotValues` is empty, the loop body never executes, and nothing is written to `contractStateCache`. On every subsequent call for the same non-existent slot, `contractStateCache.get()` returns `null`, the guard at line 65 fails, and `findStorageBatch()` is called again unconditionally.

**Amplification via `contractSlotsCache`:**

Inside `findStorageBatch()`, line 90:
```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
```
Each new non-existent slot key is accumulated in the per-contract `contractSlotsCache` (max size 1500, `expireAfterAccess=5m`). Line 103 then issues a batch DB query for **all** accumulated slot keys:
```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
After an attacker has seeded 1500 non-existent slot keys into `contractSlotsCache`, every subsequent request for any of those slots triggers a batch DB query scanning 1500 rows â€” a 1500Ă— amplification factor per `findStorage()` invocation.

**Why existing checks are insufficient:**

- The `EMPTY_VALUE` sentinel is stored only in `contractSlotsCache` (line 90), never in `contractStateCache`. The reference-equality check `cachedValue != EMPTY_VALUE` at line 65 is therefore dead code for this path and provides no protection.
- The global HTTP-level rate limiter (`requestsPerSecond=500`) throttles incoming HTTP requests, not individual `findStorage()` calls. A single contract call can trigger many `findStorage()` invocations (one per storage slot accessed by the EVM), so the effective DB query rate is `500 req/s Ă— slots_per_call Ă— batch_size`.
- The `contractState` cache TTL (`expireAfterWrite=2s`) is irrelevant here because the cache is never written for non-existent slots.

### Impact Explanation
An attacker can drive sustained, amplified database load without any privilege. After seeding 1500 non-existent slot keys per contract (up to 3000 contracts in `contractSlotsCache`), every subsequent request generates batch queries of up to 1500 rows each. At the default 500 req/s rate limit, with even a modest number of slots per call, this can increase DB query volume by well over 30% compared to baseline, degrading service for all users of the mirror node web3 API. The `contract_state` table is a hot, large table; repeated large batch scans degrade index performance and increase I/O.

### Likelihood Explanation
No authentication or special privilege is required. The attacker only needs to know a valid contract address (publicly discoverable on-chain) and send standard `eth_call` requests that read non-existent storage slots. The attack is trivially repeatable, scriptable, and can be sustained indefinitely as long as the attacker keeps accessing the slots within the 5-minute `expireAfterAccess` window to prevent eviction from `contractSlotsCache`.

### Recommendation
Implement **negative caching**: when `findStorageBatch()` determines that a requested slot does not exist in the DB (i.e., the slot key is present in `contractSlotsCache` but absent from `contractSlotValues`), explicitly store a sentinel value (e.g., `EMPTY_VALUE`) in `contractStateCache` for that slot. The existing check `cachedValue != null && cachedValue != EMPTY_VALUE` at line 65 is already structured to handle this â€” it just needs the write side to be completed:

```java
// In findStorageBatch(), after the DB result loop:
// For slots in cachedSlots that were NOT returned by the DB, store EMPTY_VALUE as a negative cache entry
for (byte[] slot : cachedSlots) {
    SimpleKey cacheKey = generateCacheKey(contractId, slot);
    if (contractStateCache.get(cacheKey, byte[].class) == null) {
        contractStateCache.put(cacheKey, EMPTY_VALUE);
    }
}
```

This ensures that repeated requests for non-existent slots hit the `contractStateCache` (with its 2-second TTL) instead of triggering a new batch DB query each time.
