### Title
Unauthenticated EVM Address Resolution Triggers Unbounded Blocking DB Lookup, Enabling Thread Pool Exhaustion via Network Partition

### Summary
The `getHooks()` endpoint in `HooksController.java` accepts EVM address format `ownerId` from any unauthenticated caller. When an EVM address is supplied, `EntityServiceImpl.lookup()` issues a synchronous, blocking database query (`findByEvmAddress`) with no JDBC socket-level timeout. During a network partition between the REST Java service and the database, each such request holds a server thread indefinitely until the OS TCP timeout fires, enabling an attacker to exhaust the thread pool with a modest number of concurrent requests.

### Finding Description

**Exact code path:**

1. `HooksController.getHooks()` receives `@PathVariable EntityIdParameter ownerId` — no authentication required. [1](#0-0) 

2. The `ownerId` string `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef` matches `EntityIdEvmAddressParameter.EVM_ADDRESS_PATTERN` and is parsed as `EntityIdEvmAddressParameter`. [2](#0-1) 

3. `HookServiceImpl.getHooks()` calls `entityService.lookup(request.getOwnerId())` before any repository query. [3](#0-2) 

4. `EntityServiceImpl.lookup()` dispatches to `entityRepository.findByEvmAddress(p.evmAddress())` — a synchronous, blocking JDBC call — for the `EntityIdEvmAddressParameter` case. Numeric IDs (`EntityIdNumParameter`) bypass this entirely. [4](#0-3) 

**Root cause and failed assumption:**

The design assumes the database is always reachable within a bounded time. The only documented timeout for the REST Java API is `hiero.mirror.restJava.db.statementTimeout` (default 10 000 ms), which is a PostgreSQL server-side `statement_timeout`. [5](#0-4) 

This timeout is enforced by the PostgreSQL server and communicated back over the TCP connection. During a **network partition**, the TCP socket between the JDBC driver and the database is established but packets are silently dropped. The PostgreSQL server cannot send the timeout error back; the JDBC driver's `socket.read()` blocks indefinitely. No JDBC-level `socketTimeout` parameter is configured for the REST Java API's HikariCP pool at runtime (the `socketTimeout` in `CommonProperties.DatabaseStartupProperties` applies only to the startup `DatabaseWaiter` probe, not to live query sockets). [6](#0-5) [7](#0-6) 

Spring MVC uses a thread-per-request model. Each blocked thread holds a Tomcat worker thread for the duration of the OS TCP timeout (default: minutes to hours depending on kernel `tcp_keepalive` settings).

### Impact Explanation

An attacker sending ~200 concurrent `GET /api/v1/accounts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/hooks` requests during (or to induce) a DB network partition will exhaust the Tomcat thread pool (default: 200 threads). Once exhausted, **all** REST Java API endpoints become unavailable — not just the hooks endpoint. This is a complete service-level denial of service affecting every consumer of the mirror node REST Java API, including downstream applications relying on hook, account, and contract data.

### Likelihood Explanation

- **No authentication required**: the endpoint is fully public.
- **EVM address format is explicitly documented and tested** as a valid input format. [8](#0-7) 
- **Network partition is not required to be attacker-induced**: a transient DB connectivity issue (cloud provider blip, maintenance window, misconfiguration) is sufficient. The attacker only needs to time requests to coincide with any DB degradation.
- **Repeatability**: the attack is trivially scriptable with `curl` or any HTTP load tool; no special privileges, tokens, or knowledge of valid account IDs is needed.

### Recommendation

1. **Add a JDBC `socketTimeout`** to the HikariCP data source for the REST Java API (e.g., `spring.datasource.hikari.data-source-properties.socketTimeout=30`). This bounds the maximum time any JDBC socket read can block, regardless of network state.
2. **Apply a JPA/Spring Data query timeout** via `spring.jpa.properties.jakarta.persistence.query.timeout=10000` so the ORM layer also enforces a deadline.
3. **Add rate limiting** on the `/api/v1/accounts/{ownerId}/hooks` endpoint (e.g., via a servlet filter or API gateway) to cap concurrent in-flight requests per source IP.
4. **Consider a circuit breaker** (e.g., Resilience4j) around `entityService.lookup()` to fast-fail when the DB is degraded, rather than queuing threads.

### Proof of Concept

```bash
# Step 1: Induce or wait for a DB network partition
#         (e.g., iptables -A OUTPUT -p tcp --dport 5432 -j DROP on the mirror node host)

# Step 2: Flood the endpoint with EVM address ownerId (200 = default Tomcat thread pool size)
for i in $(seq 1 250); do
  curl -s "http://<mirror-node-host>:8080/api/v1/accounts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/hooks" &
done
wait

# Step 3: Observe that all subsequent requests to ANY REST Java endpoint hang or return 503
curl -v "http://<mirror-node-host>:8080/api/v1/accounts/0.0.1/hooks"
# Expected: connection hangs / no response (thread pool exhausted)
```

Each of the 250 requests triggers `entityRepository.findByEvmAddress()`, which blocks on the partitioned socket. With no JDBC socket timeout, threads are held until the OS TCP timeout, starving all other requests.
