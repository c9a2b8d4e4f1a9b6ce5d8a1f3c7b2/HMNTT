### Title
Unauthenticated Cache Exhaustion via Unique Query Document Flooding in GraphQL Preparsed Document Cache

### Summary
The `CachedPreparsedDocumentProvider` uses the raw query string as the cache key against a Caffeine cache with `maximumSize=1000`. Because the GraphQL module has no rate limiting or authentication, an unprivileged attacker can continuously submit syntactically valid but textually unique query documents (e.g., varying whitespace, aliases, or inline argument values) to fill and continuously evict the 1000-entry cache, forcing repeated parse-and-validate cycles for every subsequent legitimate request.

### Finding Description

**Exact code path:**

`CachedPreparsedDocumentProvider.java` line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```
The cache key is the **raw query string** with no normalization. Any textual variation — extra whitespace, a different alias, an inline literal value instead of a variable — produces a distinct cache key and a distinct parse+validate invocation.

`CacheProperties.java` line 15:
```java
private String query = "expireAfterWrite=1h,maximumSize=1000,recordStats";
```
The cache is capped at 1000 entries.

**Root cause / failed assumption:** The design assumes clients will reuse the same query document text (the standard GraphQL client pattern of persisted/static queries). It does not account for an adversary who deliberately varies the query text to produce unique cache keys. There is no query normalization (e.g., stripping whitespace, sorting fields, converting inline literals to variables) before the cache lookup.

**No rate limiting or authentication on the GraphQL endpoint:** A search across all configuration files in the `graphql` module reveals no `SecurityFilterChain`, no `ThrottleConfiguration`, and no IP-based rate limiter. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not wired into the `graphql` module.

**Exploit flow:**
1. Attacker crafts 1001+ syntactically valid `account` queries that differ only in whitespace or field aliases, e.g.:
   - `{ account(input:{entityId:{shard:0,realm:0,num:1}}){balance} }`
   - `{  account(input:{entityId:{shard:0,realm:0,num:1}}){balance} }` (one extra space)
   - `{ a1:account(input:{entityId:{shard:0,realm:0,num:1}}){balance} }`
   - … up to 1000+ variants
2. Each request misses the cache (unique key), triggers `parseAndValidateFunction.apply(executionInput)` — full lex, parse, AST build, schema validation.
3. Once the cache is full, Caffeine's W-TinyLFU evicts the least-frequency entries. Because the attacker's queries are each sent once, they have low frequency and are evicted. However, if the attacker **continuously** streams new unique queries, the cache is perpetually full of attacker-controlled entries, and legitimate repeated queries are continuously evicted before accumulating frequency.
4. Legitimate clients whose queries are evicted must re-parse on every request.

**Why existing checks are insufficient:**
- `maxCharacters(10000)`, `maxTokens(1000)`, `maxRuleDepth(100)` — these bound the cost of a **single** parse but do not prevent the attacker from issuing thousands of distinct parseable queries.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — these run **after** parsing and do not prevent cache exhaustion.
- `StreamReadConstraints` (JSON limits) — these reject oversized JSON bodies but do not prevent 1000 small, unique, valid query strings.
- W-TinyLFU eviction — partially mitigates a one-time burst (legitimate high-frequency queries eventually reclaim slots) but is ineffective against a sustained stream of novel queries.

### Impact Explanation
Every cache miss forces a full GraphQL parse-and-validate cycle. While the schema is small (`query.graphqls` has a single `account` query), the parse cycle still involves lexing, AST construction, and schema validation. Under a sustained attack, the server CPU is consumed by repeated parsing for every incoming request from both the attacker and legitimate users. This constitutes a non-network-based DoS: the service remains reachable but degrades in throughput and latency proportional to the attack rate. Because the GraphQL module is publicly exposed (port 8083, no auth), the attacker needs no credentials.

### Likelihood Explanation
The attack requires only an HTTP client and the ability to generate slightly varied query strings — trivially scriptable in a few lines of Python or curl. No credentials, no special network position, and no knowledge of internal state are required. The endpoint is documented as publicly accessible (`http://localhost:8083/graphql/alpha`). The attack is repeatable and can be sustained indefinitely.

### Recommendation
1. **Normalize query strings before cache lookup** — strip insignificant whitespace, sort field selections, and require variables for all argument values. This collapses textual variants to a canonical form.
2. **Add rate limiting to the GraphQL module** — apply a per-IP or global request-per-second limit analogous to the `ThrottleConfiguration` in the `web3` module.
3. **Increase cache size or use a bounded-size + TTL eviction policy** that is harder to exhaust (e.g., `maximumSize=10000`).
4. **Consider persisted queries** — only accept a whitelist of pre-registered query hashes, rejecting arbitrary query documents entirely.

### Proof of Concept
```bash
#!/bin/bash
# Fill the 1000-entry cache with unique query documents
for i in $(seq 1 1001); do
  ALIAS="a${i}"
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ ${ALIAS}: account(input: {entityId: {shard: 0, realm: 0, num: ${i}}}) { balance } }\"}" \
    > /dev/null &
done
wait

# Now send a legitimate repeated query — it will miss the cache and be re-parsed
for i in $(seq 1 100); do
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query": "{ account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance } }"}' \
    > /dev/null
done
# Observe via /actuator/metrics?name=cache.gets that cache.miss rate is 100%
