### Title
Connection Pool Exhaustion DoS via Unauthenticated Burst Requests During DB Network Partition

### Summary
`getScheduleById()` in `rest/schedules.js` calls `pool.queryQuietly()` without a `preQueryHint`, which routes through `pool.query()` — a path that acquires a pg pool connection for the full duration of the query. During a network partition between the REST API and PostgreSQL, the server-side `statement_timeout` becomes unreachable and connections hang indefinitely (until TCP keepalive fires, potentially minutes later). With no rate limiting, no circuit breaker, and a default pool cap of 10 connections, an unauthenticated attacker sending 10+ concurrent requests exhausts the pool, causing all subsequent requests to fail with a `DbError` wrapping the pg `connectionTimeoutMillis` error.

### Finding Description

**Exact code path:**

`rest/schedules.js:115-128` — `getScheduleById` calls `pool.queryQuietly(getScheduleByIdQuery, scheduleId)` with no third argument (`preQueryHint` is `undefined`).

```js
const getScheduleById = async (req, res) => {
  utils.validateReq(req);
  const scheduleId = EntityId.parseString(req.params.scheduleId, parseOptions).getEncodedId();
  const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId); // no preQueryHint
  ...
};
```

`rest/utils.js:1518-1520` — Inside `queryQuietly`, the absence of `preQueryHint` routes to `this.query()` (the pg Pool's built-in method), NOT `this.connect()`:

```js
if (!preQueryHint) {
  result = await this.query(query, params);   // pool.query() — holds connection for query duration
} else {
  client = await this.connect();              // explicit client — only this path has manual release
}
```

`pool.query()` internally acquires a client from the pool, holds it for the entire query execution, and releases it only upon completion or error. The `finally` block at lines 1540-1544 only releases `client` when `preQueryHint` was set — meaning the implicit client acquired by `pool.query()` is managed entirely by pg internals and is not released until the query resolves.

`rest/dbpool.js:7-16` — Pool is configured with:
- `max: config.db.pool.maxConnections` → default **10** connections
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` → default **20,000 ms**
- `statement_timeout: config.db.pool.statementTimeout` → default **20,000 ms**

**Root cause — failed assumption:**

The design assumes `statement_timeout` (20 s) will bound how long a connection is held. This assumption fails during a network partition. `statement_timeout` is enforced by the PostgreSQL server; when the network is partitioned, the server's timeout signal cannot traverse the broken link back to the Node.js client. The TCP connection appears alive (TCP keepalive has not yet fired; Linux default `tcp_keepalive_time` is 7200 s). The pg client therefore blocks indefinitely on the awaited `pool.query()` promise, holding the pool slot.

**No mitigating controls exist in the REST layer:**
- `rest/server.js` has no HTTP-level request timeout (`server.setTimeout` is absent; grep for `timeout|keepAlive|idleTimeout` returns zero matches).
- `rest/*.js` has zero rate-limiting, throttling, or circuit-breaker logic (grep for `rateLimit|throttle|concurrency|circuit` returns zero matches). The throttle configuration found in `web3/` is for the Java REST API only and does not apply here.

### Impact Explanation

With 10 concurrent hanging requests (one per pool slot), `pool.query()` on the 11th request blocks for `connectionTimeoutMillis` (20 s) waiting for a free slot, then throws. `queryQuietly` catches this at line 1539 and re-throws as `DbError`. The Express error handler returns a 500 to the client. Every user hitting `GET /api/v1/schedules/:scheduleId` (or any other endpoint sharing the same global `pool`) receives errors for the entire duration of the attack. Because the pool is global (`global.pool` in `dbpool.js:36`), exhaustion of the 10-slot pool affects all REST endpoints simultaneously, not just schedules.

Severity: **High** — complete availability loss for all DB-backed endpoints during the attack window, with no authentication required.

### Likelihood Explanation

Network partitions between application pods and database replicas are a routine occurrence in cloud-native deployments (node evictions, rolling restarts, AZ connectivity blips). An attacker monitoring for elevated latency (e.g., via response-time probing) can detect the partition window and immediately fire a burst of 10 HTTP GET requests — trivially achievable with `curl`, `ab`, or any HTTP client. No credentials, tokens, or special knowledge are required. The attack is repeatable: as soon as the partition clears and connections drain, the attacker can re-trigger on the next partition event.

### Recommendation

1. **Add a Node.js-side query timeout** independent of `statement_timeout`. Wrap `pool.query()` in a `Promise.race` against a `setTimeout` rejection, or use pg's `query_timeout` client option, so connections are forcibly released even when the server-side signal is unreachable.
2. **Implement a circuit breaker** (e.g., `opossum`) around `queryQuietly` that opens after a configurable number of consecutive DB failures, returning fast errors without acquiring pool connections.
3. **Add HTTP-level request timeouts** via `server.setTimeout()` or a middleware like `connect-timeout` so hung requests are aborted and their pool connections released.
4. **Add per-IP or global concurrency limiting** in the Node.js REST layer (e.g., `express-rate-limit`) to cap the number of in-flight requests that can reach the DB layer.
5. **Increase `maxConnections`** or deploy PgBouncer in front of PostgreSQL to decouple application-level connection pressure from the database.

### Proof of Concept

**Preconditions:**
- REST API is running and connected to PostgreSQL.
- A network partition is induced (e.g., `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the API host, or a cloud security-group rule blocking DB egress).

**Steps:**
```bash
# 1. Induce partition (on API host or via network policy)
iptables -A OUTPUT -p tcp --dport 5432 -j DROP

# 2. Send 15 concurrent requests (exceeds default maxConnections=10)
for i in $(seq 1 15); do
  curl -s "http://<api-host>/api/v1/schedules/0.0.1" &
done
wait

# 3. Immediately send a probe request
curl -v "http://<api-host>/api/v1/schedules/0.0.1"
```

**Expected result:**
- Requests 1–10 hang for up to 20 s (statement_timeout unreachable; TCP alive).
- Requests 11–15 hang for up to 20 s waiting for a pool slot, then return HTTP 500 with a `DbError` (connection timeout).
- The probe request in step 3 also returns HTTP 500 immediately (pool full).
- All other DB-backed endpoints (`/api/v1/accounts`, `/api/v1/transactions`, etc.) are simultaneously unavailable.
