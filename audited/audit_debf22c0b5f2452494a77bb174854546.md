### Title
Unbounded Historical Timestamp Scan in `getContractStateById` Enables Unauthenticated DB I/O Exhaustion

### Summary
The `GET /contracts/:contractId/state` endpoint accepts a `timestamp=lte:<value>` filter with no lower-bound requirement and no timestamp-range validation. When a timestamp is present, the handler switches to `contractStateTimestampQuery`, which uses `SELECT DISTINCT ON (slot) ... ORDER BY slot ASC, consensus_timestamp DESC LIMIT N` against the `contract_state_change` table. Because `DISTINCT ON` requires PostgreSQL to process all matching rows before applying `LIMIT`, a far-future `lte` timestamp causes a full historical scan of every state-change row for the target contract, regardless of the row limit.

### Finding Description

**Route registration** — `rest/routes/contractRoute.js:15`:
```js
router.getExt('/:contractId/state', ContractController.getContractStateById);
```
No authentication middleware is applied.

**Timestamp filter processing** — `rest/controllers/contractController.js:909-918`: [1](#0-0) 

The `extractContractStateByIdQuery` method accepts any `lte`, `lt`, `gte`, `gt`, or `eq` timestamp operator and pushes it directly into the SQL `WHERE` conditions. It does **not** call `utils.parseTimestampFilters` (which enforces `maxTimestampRangeNs`) and does **not** call `bindTimestampRange`. Both of those guards are used by other endpoints via `optimizeTimestampFilters` (`rest/controllers/contractController.js:384-404`), but are entirely absent here.

A single `lte` filter with no lower bound is accepted without error, setting `timestampPresent = true` and routing the query to `contractStateTimestampQuery`.

**The expensive SQL query** — `rest/service/contractService.js:111-122`: [2](#0-1) 

```sql
SELECT DISTINCT ON (slot)
  contract_id, slot, evm_address,
  coalesce(value_written, value_read) AS value,
  consensus_timestamp AS modified_timestamp
FROM contract_state_change
LEFT JOIN entity ...
WHERE contract_id = $1
  AND consensus_timestamp <= $2   -- $2 = 99999999999999999999 (far future)
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT 100
```

PostgreSQL's `DISTINCT ON` semantics require the engine to **sort and deduplicate the entire result set** before `LIMIT` is applied. There is no index that can satisfy both `ORDER BY slot ASC, consensus_timestamp DESC` in a single pass; the planner must materialize all rows matching the `WHERE` clause, sort them, and then emit the first 100 distinct-slot rows. For a heavily-used contract with millions of state-change rows, this is a full sequential scan of the per-contract partition.

**Query dispatch** — `rest/service/contractService.js:256-263`: [3](#0-2) 

**Limit cap** — `rest/utils.js:544-552`: [4](#0-3) 

The response limit is capped at `responseLimit.max` (default 100). This does **not** prevent the scan; it only limits the rows returned to the caller after PostgreSQL has already done the work.

**Contrast with other endpoints** — `rest/controllers/contractController.js:384-404`: [5](#0-4) 

Contract-results and contract-logs endpoints call `optimizeTimestampFilters`, which invokes `parseTimestampFilters` with `validateRange=true` (enforcing `config.query.maxTimestampRangeNs`) and `bindTimestampRange` (capping the window to a configured maximum). The contract-state endpoint has no equivalent protection.

### Impact Explanation
An attacker can repeatedly issue `GET /contracts/0.0.<popular_contract>/state?timestamp=lte:99999999999.999999999` with no authentication. Each request forces a full historical scan of `contract_state_change` for that contract. On a production mirror node tracking a high-activity contract (e.g., a popular DeFi or token contract on Hedera mainnet), this table can contain tens of millions of rows. Concurrent requests from a single IP or a small botnet can saturate DB I/O, increase query latency for all other API consumers, and potentially cause the database to OOM or the REST service to time out, constituting a non-network-based DoS.

### Likelihood Explanation
The exploit requires zero privileges, zero tokens, and zero knowledge beyond a valid contract ID (which is public on-chain). The request is a single standard HTTP GET. The attacker does not need to maintain state or perform any handshake. The REST service has no per-IP rate limiting visible in the Node.js codebase (the `ThrottleConfiguration` found is in the separate `web3` Java service for `eth_call`, not for the REST API). Repeatability is trivial; a single attacker with a modest connection pool can sustain the attack indefinitely.

### Recommendation
Apply the same timestamp-range guard used by other endpoints:

1. In `extractContractStateByIdQuery`, collect timestamp filters and pass them through `utils.parseTimestampFilters` with `validateRange: true` and `allowOpenRange: false`. This enforces `config.query.maxTimestampRangeNs` and rejects unbounded `lte`-only queries.
2. Alternatively, call `optimizeTimestampFilters` (already used for contract results/logs) before building the WHERE conditions, which also invokes `bindTimestampRange` to cap the window.
3. Require that any timestamp filter include both a lower bound (`gte`/`gt`) and an upper bound (`lte`/`lt`), or reject open-ended range queries for this endpoint.
4. Consider adding a DB-level statement timeout for this query class as a defense-in-depth measure.

### Proof of Concept

**Prerequisites**: A valid contract ID on the target mirror node (any public contract ID works).

**Step 1 — Single exhausting request:**
```
GET /api/v1/contracts/0.0.1234/state?timestamp=lte:99999999999.999999999
```
No API key, no authentication header required.

**Step 2 — Observe the generated SQL** (from service logs or `pg_stat_activity`):
```sql
SELECT DISTINCT ON (csc.slot)
  csc.contract_id, csc.slot, e.evm_address,
  coalesce(csc.value_written, csc.value_read) AS value,
  csc.consensus_timestamp AS modified_timestamp
FROM contract_state_change csc
LEFT JOIN entity e ON e.id = csc.contract_id
WHERE csc.contract_id = 1234
  AND csc.consensus_timestamp <= 99999999999999999999
ORDER BY csc.slot ASC, csc.consensus_timestamp DESC
LIMIT 100;
```

**Step 3 — Amplify with concurrency:**
```bash
seq 1 50 | xargs -P50 -I{} curl -s \
  "https://<mirror-node>/api/v1/contracts/0.0.1234/state?timestamp=lte:99999999999.999999999" \
  -o /dev/null
```

**Expected result**: DB CPU and I/O spike proportional to the number of historical state-change rows for the target contract; other API endpoints experience elevated latency or timeouts.
