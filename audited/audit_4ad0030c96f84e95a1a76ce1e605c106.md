### Title
Unauthenticated Double DB Query Amplification via `block.hash` Filter on `/contracts/:contractId/results`

### Summary
Any unauthenticated user can supply a `block.hash` query parameter to `/api/v1/contracts/:contractId/results`, which unconditionally triggers a separate `record_file` table lookup (`getRecordFileBlockDetailsFromHash`) before the main `contract_result` query (`getContractResultsByIdAndFilters`). The REST-layer `RecordFileService.getRecordFileBlockDetailsFromHash()` has no in-process caching, and the HTTP response cache is bypassed by rotating through distinct valid block hashes, yielding a sustained 2× DB query amplification factor achievable by any anonymous caller.

### Finding Description

**Code path:**

1. `GET /api/v1/contracts/:contractId/results?block.hash=<hash>` is handled by `getContractResultsById` in `rest/controllers/contractController.js` (line 856). No authentication is required.

2. It calls `extractContractResultsByIdQuery(filters, contractId)` (line 871). Inside that function, when `filterKeys.BLOCK_HASH` is detected (lines 464–466), the code unconditionally executes:

```js
blockData = await RecordFileService.getRecordFileBlockDetailsFromHash(blockFilter.value);
``` [1](#0-0) 

3. `getRecordFileBlockDetailsFromHash` issues a raw SQL query against `record_file` with a `LIKE $1` predicate (the hash is appended with `%` to support prefix matching) — **DB Query #1**:

```js
async getRecordFileBlockDetailsFromHash(hash) {
  const row = await super.getSingleRow(RecordFileService.recordFileBlockDetailsFromHashQuery, [`${hash}%`]);
  return row === null ? null : new RecordFile(row);
}
``` [2](#0-1) [3](#0-2) 

There is **no `@Cacheable` annotation, no in-process cache, and no memoization** on this method in the REST service layer.

4. If `blockData` is non-null (valid hash), timestamp bounds are pushed into `timestampFilters` and control returns to `getContractResultsById`, which then calls:

```js
const rows = await ContractService.getContractResultsByIdAndFilters(conditions, params, order, limit);
```
— **DB Query #2**. [4](#0-3) [5](#0-4) 

**Why existing mitigations are insufficient:**

- The HTTP response cache (`responseCacheHandler.js`) caches by full URL. An attacker who rotates through the finite set of valid block hashes (one per block in the chain) generates a unique cache key each time, so every rotation cycle is a full cache miss and triggers both queries. [6](#0-5) 

- No rate-limiting middleware is present in `rest/middleware/index.js` or `rest/server.js`. [7](#0-6) 

- Using a non-existent hash causes `skip: true` (only 1 query), so the amplification is specific to **valid** hashes — but valid hashes are publicly enumerable via `/api/v1/blocks`.

### Impact Explanation
Each request with a valid `block.hash` consumes two DB connections/round-trips instead of one. An attacker cycling through N known block hashes at rate R generates 2×R DB queries, effectively halving the request rate needed to saturate the DB connection pool or query throughput. This degrades response latency for all users of the mirror node REST API. The impact is bounded by the number of distinct valid block hashes (finite but large on mainnet), and is classified as a griefing/availability issue with no direct economic damage.

### Likelihood Explanation
The attack requires no credentials, no special tooling, and no on-chain interaction. Valid block hashes are freely enumerable via the public `/api/v1/blocks` endpoint. A single attacker with a modest HTTP client can sustain the amplification indefinitely. The pattern is trivially scriptable and repeatable.

### Recommendation
1. **Add an in-process LRU/TTL cache** in `RecordFileService.getRecordFileBlockDetailsFromHash()` (e.g., using a small in-memory map with a short TTL such as 10 s). Block hashes are immutable once written, so caching is safe.
2. **Apply rate limiting** per IP (or API key) at the middleware layer for all `/contracts` endpoints.
3. Consider **joining `record_file` directly** in the `contract_result` query when a block hash filter is present, eliminating the separate round-trip entirely.

### Proof of Concept

```bash
# 1. Enumerate valid block hashes
HASHES=$(curl -s "https://<mirror-node>/api/v1/blocks?limit=100&order=desc" \
  | jq -r '.blocks[].hash')

# 2. Rotate through them, each request causes 2 DB queries
for HASH in $HASHES; do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.5001/results?block.hash=${HASH}" &
done
wait

# Repeat in a tight loop. Each unique hash is a cache miss → 2 DB queries.
# At N concurrent requests/sec, DB load = 2N queries/sec instead of N.
```
