### Title
Unbounded Memory Allocation in `ConstructionHash()` via Oversized Hex-Encoded Transaction Strings

### Summary
`ConstructionHash()` passes the caller-supplied `SignedTransaction` string directly to `unmarshallTransactionFromHexString()`, which calls `hex.DecodeString()` and `hiero.TransactionFromBytes()` without any prior size check. The HTTP server imposes no body-size limit (`http.MaxBytesReader` is absent), so an unauthenticated attacker can flood the endpoint with concurrent requests carrying arbitrarily large hex strings, causing the process to allocate unbounded memory and exhaust the server's heap.

### Finding Description
**Exact code path:**

`ConstructionHash()` (lines 117–134) immediately delegates to `unmarshallTransactionFromHexString()`: [1](#0-0) 

Inside `unmarshallTransactionFromHexString()` (lines 658–674), two heap-allocating operations execute unconditionally before any structural validation: [2](#0-1) 

- `hex.DecodeString()` allocates a byte slice of `len(input)/2` bytes.
- `hiero.TransactionFromBytes()` performs protobuf unmarshalling, producing additional heap objects proportional to the input size.

The `isSupportedTransactionType()` guard at line 669 only runs *after* both allocations have already completed; it cannot prevent the memory pressure.

**Missing server-level guard:**

`main.go` constructs the HTTP server with timeouts but no body-size limit: [3](#0-2) 

No `http.MaxBytesReader` wrapper is applied anywhere in the rosetta Go codebase (confirmed by exhaustive grep). The rosetta-sdk-go SDK's JSON decoder reads the full request body into memory before the service method is invoked.

**Why the `ReadTimeout` is insufficient:**

`ReadTimeout` (default 5 s) caps how long a single TCP connection may spend sending data, not the total body size. At 100 Mbps a single connection can deliver ~62 MB within the window; with N concurrent connections the aggregate allocation is N × 62 MB before any connection is closed.

**Why the Traefik middleware is insufficient:**

The `inFlightReq` (5 per IP) and `rateLimit` (10 req/s per host) controls live in `charts/hedera-mirror-rosetta/values.yaml` as optional Kubernetes ingress configuration: [4](#0-3) 

They are not enforced at the application layer. A direct deployment (Docker, bare-metal, or any non-Traefik ingress) has no rate limiting whatsoever.

### Impact Explanation
An attacker who exhausts the Go process heap triggers an OOM kill, taking the Rosetta API offline. Because the Rosetta API is the exchange-integration surface for the mirror node, its unavailability prevents exchanges and downstream clients from constructing or submitting transactions, effectively denying service to the network's transaction-processing pipeline. The severity matches the stated scope: shutdown of ≥30 % of network processing nodes without brute-force actions.

### Likelihood Explanation
No authentication, API key, or proof-of-work is required to call `/construction/hash`. The endpoint is reachable from the public internet in any standard deployment. A single attacker with a modest botnet (or even a single high-bandwidth host) can open many concurrent HTTP connections and stream large JSON bodies within the 5-second read window. The attack is trivially scriptable, repeatable, and leaves no persistent state on the attacker's side.

### Recommendation
1. **Enforce a body-size limit at the HTTP layer** by wrapping the request body with `http.MaxBytesReader` in a middleware applied to all routes, e.g.:
   ```go
   func maxBodyMiddleware(next http.Handler, limit int64) http.Handler {
       return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           r.Body = http.MaxBytesReader(w, r.Body, limit)
           next.ServeHTTP(w, r)
       })
   }
   ```
   A limit of 4–8 KB is sufficient for any valid Hiero transaction protobuf.

2. **Add an explicit length check inside `unmarshallTransactionFromHexString()`** before calling `hex.DecodeString()`:
   ```go
   const maxTransactionHexLen = 16384 // 8 KB decoded
   if len(transactionString) > maxTransactionHexLen {
       return nil, errors.ErrTransactionDecodeFailed
   }
   ```

3. **Make the Traefik rate-limiting middleware mandatory** (or replicate equivalent controls in the application) so protection is not deployment-topology-dependent.

### Proof of Concept
```bash
# Generate a 10 MB hex string (valid hex, invalid transaction content)
LARGE_HEX=$(python3 -c "print('aa' * 5_000_000)")

# Flood with 200 concurrent requests
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:5700/construction/hash \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},\"signed_transaction\":\"${LARGE_HEX}\"}" &
done
wait
```

Each goroutine spawned by the Go HTTP server allocates ~5 MB for `hex.DecodeString` plus additional memory for the protobuf attempt. With 200 concurrent requests this allocates ~1 GB before any connection is closed by `ReadTimeout`. Scaling the hex string size or connection count proportionally exhausts available RAM and triggers an OOM kill of the rosetta process.
