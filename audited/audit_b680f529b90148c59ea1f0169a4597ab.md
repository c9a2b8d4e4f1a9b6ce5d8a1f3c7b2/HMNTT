### Title
Unbounded Timestamp Range on `/api/v1/blocks` Enables DB Connection Pool Exhaustion DoS

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts a `timestamp=gte:0` filter with no upper bound and passes it directly to `RecordFileService.getBlocks()` without invoking `parseTimestampFilters()` or any range-span validation. With no rate limiting on the REST API and a default DB connection pool of only 10 connections, an unauthenticated attacker sending concurrent requests can exhaust the pool and deny service to all users.

### Finding Description

**Code path:**

`blockController.js` `getBlocks()` (lines 101–112): [1](#0-0) 

calls `utils.buildAndValidateFilters(req.query, acceptedBlockParameters)` with no custom validator and no call to `parseTimestampFilters`. The timestamp filter is passed directly into `extractSqlFromBlockFilters`: [2](#0-1) 

which maps it verbatim to a `WHERE consensus_end >= $1` clause. `RecordFileService.getBlocks()` then executes: [3](#0-2) 

```sql
SELECT ... FROM record_file WHERE consensus_end >= 0 ORDER BY consensus_end ASC LIMIT 100
```

**Root cause — failed assumption:** The codebase has `parseTimestampFilters()` which can enforce both a lower and upper bound and cap the span to `maxTimestampRange` (default 7 days): [4](#0-3) 

This function is called in `contractController.js` for contract-result queries but is **never called** in the blocks path. The `filterValidityChecks` used by `buildAndValidateFilters` only validates timestamp format, not range completeness: [5](#0-4) 

**No rate limiting:** A grep across `rest/**/*.js` for `rateLimit`, `rateLimiter`, or `throttle` returns no middleware. The web3 module has Bucket4j throttling, but the Node.js REST API has none.

**Small connection pool:** The default pool is 10 connections with a 20-second statement timeout and 20-second connection timeout: [6](#0-5) [7](#0-6) 

### Impact Explanation
With 10 concurrent connections held for up to 20 seconds each (statement timeout), the pool is fully saturated. All subsequent requests queue waiting for a free connection; after `connectionTimeoutMillis` (20 s) they fail. This denies service to all legitimate users of every REST endpoint, not just `/api/v1/blocks`, because all endpoints share the same `primaryPool`. [8](#0-7) 

### Likelihood Explanation
No authentication, no rate limiting, and no IP-based throttling are required. A single attacker with a modest number of concurrent HTTP connections (≥10) from one host can trigger this. The request is valid, passes all input validation, and is indistinguishable from legitimate traffic. It is trivially repeatable and scriptable.

### Recommendation
1. **Require an upper timestamp bound** in `getBlocks()` by calling `parseTimestampFilters()` on the extracted timestamp filters (with `allowOpenRange = false`) before building the SQL, mirroring the pattern used in `contractController.js`.
2. **Add rate limiting middleware** to the Express app (e.g., `express-rate-limit`) applied globally or per-route.
3. **Increase the default pool size** or add a per-IP concurrency limit at the load-balancer/ingress layer.
4. **Add a query timeout** shorter than 20 s for the blocks endpoint specifically to reduce connection hold time.

### Proof of Concept

```bash
# Send 15 concurrent requests with open-ended timestamp range
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>:5551/api/v1/blocks?timestamp=gte:0&limit=100&order=asc" &
done
wait

# All subsequent legitimate requests will time out or receive connection errors
curl -v "http://<mirror-node-host>:5551/api/v1/blocks"
# Expected: hangs for 20 s then fails with connection timeout / 503
```

The 15 concurrent requests exceed the default pool of 10; each holds a connection for up to 20 s (statement timeout), starving all other API consumers.

const parseTimestampFilters = (
  filters,
  filterRequired = true,
  allowNe = false,
  allowOpenRange = false,
  strictCheckOverride = true,
  validateRange = true
) => {
  const forceStrictChecks = strictCheckOverride || config.strictTimestampParam;

  if (filters.length === 0) {
    if (filterRequired) {
      throw new InvalidArgumentError('No timestamp range or eq operator provided');
    }

    return {range: null, neValues: [], eqValues: []};
  }

  let earliest = null;
  let latest = null;
  let range = null;
  const eqValues = new Set();
  const neValues = new Set();
  let lowerBoundFilterCount = 0;
  let upperBoundFilterCount = 0;

  for (const filter of filters) {
    let value = BigInt(filter.value);
    switch (filter.operator) {
      case opsMap.eq:
        eqValues.add(value);
        break;
      case opsMap.ne:
        neValues.add(value);
        break;
      case opsMap.gt:
        value += 1n;
      case opsMap.gte:
        earliest = bigIntMax(earliest ?? 0n, value);
        lowerBoundFilterCount += 1;
        break;
      case opsMap.lt:
        value -= 1n;
      case opsMap.lte:
        latest = bigIntMin(latest ?? constants.MAX_LONG, value);
        upperBoundFilterCount += 1;
        break;
    }
  }

  if (forceStrictChecks) {
    if (!allowNe && neValues.size > 0) {
      throw new InvalidArgumentError('Not equals operator not supported for timestamp param');
    }

    if (lowerBoundFilterCount > 1) {
      throw new InvalidArgumentError('Multiple gt or gte operators not permitted for timestamp param');
    }

    if (upperBoundFilterCount > 1) {
      throw new InvalidArgumentError('Multiple lt or lte operators not permitted for timestamp param');
    }

    if (eqValues.size > 0 && (lowerBoundFilterCount > 0 || upperBoundFilterCount > 0 || neValues.size > 0)) {
      throw new InvalidArgumentError('Cannot combine eq with ne, gt, gte, lt, or lte for timestamp param');
    }
  }

  if (!allowOpenRange && eqValues.size === 0 && (lowerBoundFilterCount === 0 || upperBoundFilterCount === 0)) {
    throw new InvalidArgumentError('Timestamp range must have gt (or gte) and lt (or lte), or eq operator');
  }

  const difference = latest !== null && earliest !== null ? latest - earliest + 1n : null;

  if (validateRange) {
    const {maxTimestampRange, maxTimestampRangeNs} = config.query;

    // If difference is null, we want to ignore because we allow open ranges and that is known to be true at this point
    if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
      throw new InvalidArgumentError(
        `Timestamp range by the lower and upper bounds must be positive and within ${maxTimestampRange}`
      );
    }
  }
