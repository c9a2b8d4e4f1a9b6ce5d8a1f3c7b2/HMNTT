### Title
Shared `boundedElastic` Scheduler Thread Exhaustion via Blocking JDBC Polls During Network Partition

### Summary
`PollingTopicListener` is a `@Named` singleton bean whose single `Schedulers.boundedElastic()` instance is shared across every `listen()` invocation. Each poll cycle executes a synchronous, blocking JPA `getResultList()` call on a scheduler thread. During a network partition, these threads block indefinitely (the configured `statementTimeout` is a PostgreSQL server-side setting that never fires when the TCP socket itself is hung), and an unprivileged attacker who has pre-opened enough gRPC subscriptions can exhaust the bounded thread pool, starving all other subscribers' scheduled poll tasks.

### Finding Description

**Exact code path:**

`PollingTopicListener` is instantiated once as a Spring singleton: [1](#0-0) 

The single `Schedulers.boundedElastic()` instance is used both for the initial subscription delay and for every repeat cycle: [2](#0-1) 

Each repeat fires `poll()`, which calls `topicMessageRepository.findByFilter()` — a fully synchronous, blocking JPA call: [3](#0-2) 

The repository implementation calls `typedQuery.getResultList()`, which blocks the calling thread until the database responds: [4](#0-3) 

**Root cause and failed assumption:**

The design assumes that `findByFilter` will return promptly. During a network partition the TCP socket to PostgreSQL hangs at the OS level. The only configured mitigation is `statementTimeout = 10000` (10 s): [5](#0-4) 

This is a PostgreSQL server-side `SET statement_timeout` hint. It is sent as part of the query. When the TCP connection is partitioned, the query bytes never reach the server, so the server-side timer never starts. No JDBC-level `socketTimeout` is configured, so the JDBC driver's socket read blocks indefinitely.

**Exploit flow:**

The only per-connection limit is `maxConcurrentCallsPerConnection = 5`: [6](#0-5) [7](#0-6) 

There is no global subscriber cap or per-IP connection limit. `TopicMessageServiceImpl` only tracks subscriber count as a metric gauge: [8](#0-7) 

`Schedulers.boundedElastic()` caps threads at `10 × availableProcessors()` (e.g., 40 threads on a 4-core host). An attacker opens `ceil(cap / 5)` gRPC connections (e.g., 8 connections × 5 streams = 40 subscriptions). When a network partition occurs, each active poll occupies one scheduler thread indefinitely. The thread pool is exhausted; all subsequently scheduled repeat tasks queue behind the blocked threads and never execute, stalling every active gRPC stream on the server.

### Impact Explanation

All active `subscribeTopic` gRPC streams that use the `POLL` listener type stop receiving messages for the duration of the partition. Because the scheduler is shared across the entire JVM process, this is a process-wide denial of service for the gRPC polling path, not limited to the attacker's own streams. Severity is **High**: complete loss of liveness for all HCS topic subscribers.

### Likelihood Explanation

The `subscribeTopic` endpoint requires no authentication (any client can call it with a valid topic ID). The attacker needs only a small number of TCP connections (proportional to CPU count) established before the partition. Network partitions between the gRPC pod and the database are a realistic operational event (pod rescheduling, DB failover, network policy change). The attack is repeatable and requires no privileged access.

### Recommendation

1. **Add a JDBC socket timeout** to the HikariCP datasource configuration (e.g., `socketTimeout=15` seconds in the JDBC URL or `spring.datasource.hikari.connection-timeout`). This bounds how long a scheduler thread can block on a hung TCP socket.
2. **Wrap blocking DB calls with `subscribeOn(Schedulers.boundedElastic())` and an explicit `timeout()` operator** in the reactive chain so Reactor can cancel the subscription and release the thread.
3. **Enforce a global subscriber limit** in `TopicMessageServiceImpl` by rejecting new subscriptions when `subscriberCount` exceeds a configurable threshold.
4. **Consider using a dedicated, isolated scheduler per subscriber** (or a separate bounded elastic instance with a smaller cap) so one subscriber's blocking cannot affect others.

### Proof of Concept

```
# Precondition: server configured with listener.type=POLL, 4-CPU host (40 boundedElastic threads)

# Step 1: Open 8 gRPC connections, 5 streams each (40 total subscriptions)
for i in $(seq 1 8); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{"topicID":{"topicNum":1},"limit":0}' \
      <mirror-node>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Step 2: Induce network partition between gRPC pod and PostgreSQL
#   (e.g., iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the gRPC host)

# Step 3: Wait for the next poll cycle (default interval: 500ms)
#   Each of the 40 scheduler threads is now blocked in getResultList() with no socket timeout.

# Step 4: Open a new legitimate subscription from a different client.
#   The new listen() call schedules its first poll via delaySubscription(interval, scheduler).
#   The scheduler has no free threads; the task is queued and never executes.
#   The legitimate subscriber receives no messages for the duration of the partition.

# Result: All 40 bounded elastic threads are blocked; all gRPC topic streams are frozen.
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
