### Title
Missing Timestamp Range Validation in `getBlocks()` Enables Unauthenticated DB Index Exhaustion DoS

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts arbitrary `timestamp` range filters and passes them directly to the database without invoking `parseTimestampFilters()`, which is the only function that enforces a maximum range width (7-day cap). Combined with the absence of per-IP rate limiting or in-flight request caps on the REST API Helm middleware, an unauthenticated attacker can flood the endpoint with thousands of 1-nanosecond timestamp windows, each forcing a highly selective index scan on `consensus_end`, exhausting DB index read I/O.

### Finding Description

**Code path:**

`getBlocks()` at [1](#0-0)  calls `utils.buildAndValidateFilters()` then `extractSqlFromBlockFilters()`. Inside `extractSqlFromBlockFilters()`, timestamp filters are mapped directly to raw SQL comparator conditions on `RecordFile.CONSENSUS_END`: [2](#0-1) 

These conditions are then assembled into a bare SQL query in `RecordFileService.getBlocks()`: [3](#0-2) 

**Root cause — missing `parseTimestampFilters()` call:**

The codebase has `parseTimestampFilters()` in `rest/utils.js`, which enforces that a timestamp range must be positive and within the configured maximum (7 days, `maxTimestampRangeNs`): [4](#0-3) 

This function is called for `contractController.js` and `transactions.js`, but **never** for the blocks endpoint. The test snapshots confirm the 7-day cap exists: [5](#0-4) 

**Root cause — no per-IP rate limiting on REST API:**

The `hedera-mirror-rest` Helm chart middleware only configures `circuitBreaker` and `retry` — no `rateLimit` or `inFlightReq`: [6](#0-5) 

Compare to `hedera-mirror-rosetta`, which has both `rateLimit` and `inFlightReq`: [7](#0-6) 

The application-level server also has no rate-limiting middleware for the blocks route: [8](#0-7) 

**What `buildAndValidateFilters` does NOT check:**

`buildAndValidateFilters` validates individual filter format (e.g., timestamp string is parseable) but performs no range-width validation: [9](#0-8) 

### Impact Explanation
An attacker can force the PostgreSQL database to execute repeated highly selective index range scans on the `record_file.consensus_end` B-tree index. Each 1-nanosecond window (`gte:N&lte:N+1`) is a valid, accepted query that hits the index but returns 0 or 1 rows — maximizing index traversal cost per row returned. Sending thousands of such requests with different windows saturates DB index read I/O, degrading or denying service to all users of the mirror node. This is a non-network-based DoS (DB resource exhaustion), not a bandwidth flood.

### Likelihood Explanation
No authentication, API key, or special privilege is required. The attack is trivially scriptable: a single `for` loop varying the nanosecond window value. The absence of per-IP rate limiting at both the application and infrastructure layers means there is no throttle to overcome. The attack is repeatable indefinitely from a single IP.

### Recommendation
1. **Enforce range validation in `getBlocks()`**: Call `parseTimestampFilters()` on the timestamp filters extracted in `extractSqlFromBlockFilters()`, the same way `contractController.js` and `transactions.js` do, to reject ranges narrower than 1 ns or wider than the configured maximum.
2. **Add `inFlightReq` and `rateLimit` middleware** to the `hedera-mirror-rest` Helm chart values, mirroring the configuration already present in `hedera-mirror-rosetta/values.yaml`.
3. **Add a minimum range width check**: Reject timestamp ranges below a practical minimum (e.g., 1 second) to prevent single-nanosecond window abuse.

### Proof of Concept
```bash
# Send 10,000 requests with different 1-nanosecond windows, no auth required
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node>/api/v1/blocks?timestamp=gte:${i}&timestamp=lte:$((i+1))" &
done
wait
```
Each request is accepted (HTTP 200, empty blocks array), executes a selective index scan on `record_file.consensus_end`, and returns. DB CPU and I/O climb monotonically. Legitimate queries begin timing out as the DB connection pool and I/O bandwidth are saturated.

const buildAndValidateFilters = (
  query,
  acceptedParameters,
  filterValidator = filterValidityChecks,
  filterDependencyChecker = filterDependencyCheck
) => {
  const {badParams, filters} = buildFilters(query);
  const {invalidParams, unknownParams} = validateAndParseFilters(filters, filterValidator, acceptedParameters);
  badParams.push(...invalidParams);
  badParams.push(...unknownParams);
  if (badParams.length > 0) {
    throw InvalidArgumentError.forRequestValidation(badParams);
  }

  if (filterDependencyChecker) {
    filterDependencyChecker(query);
  }

  return filters;
};
