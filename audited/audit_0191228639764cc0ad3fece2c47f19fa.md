### Title
Global Single-Token Opcode Rate Limit Bucket Enables Unauthenticated DoS via Token Starvation

### Summary
The `opcodeRateLimitBucket` is a single application-wide Bucket4j bucket with a default capacity of 1 token, shared across all users and requests. Any unauthenticated caller can consume the sole token once per second, starving every other user of access to the opcode tracing endpoint for the remainder of that second. Because no per-IP or per-identity isolation exists, a single low-rate attacker (1 req/s) achieves a complete, sustained denial-of-service against all legitimate opcode tracing consumers.

### Finding Description
**Code path:**

- `ThrottleProperties.java` line 29: default `opcodeRequestsPerSecond = 1` [1](#0-0) 

- `ThrottleConfiguration.java` lines 47–55: a single Spring singleton `Bucket` is created with `capacity(rateLimit)` / `refillGreedy(rateLimit, Duration.ofSeconds(1))` where `rateLimit = 1` by default. No per-IP partitioning, no distributed key, no user identity. [2](#0-1) 

- `ThrottleManagerImpl.java` lines 52–56: every call to `throttleOpcodeRequest()` does `opcodeRateLimitBucket.tryConsume(1)` against that same global singleton. [3](#0-2) 

- `OpcodesController.java` lines 59–65: the controller calls `throttleManager.throttleOpcodeRequest()` before any authentication or identity check. [4](#0-3) 

**Root cause:** The failed assumption is that a global 1-token-per-second budget is sufficient to protect the endpoint while still serving multiple users. In reality, with capacity = 1, the first request in any 1-second window exhausts the entire global budget. All subsequent requests from any user receive HTTP 429 until the bucket refills. There is no per-source-IP bucket, no per-authenticated-user bucket, and no allowlist that would prevent an anonymous caller from monopolizing the budget.

**Why existing checks fail:** The only guard is `tryConsume(1)` on the shared singleton. The `gasLimitBucket` at least uses `SynchronizationStrategy.SYNCHRONIZED`, but `opcodeRateLimitBucket` does not even have that. [5](#0-4) 

### Impact Explanation
An attacker sending exactly 1 well-formed GET request per second to `/api/v1/contracts/results/{txHash}/opcodes` with `Accept-Encoding: gzip` consumes the single global token continuously. All legitimate users receive HTTP 429 for the entire service lifetime of the attack. Opcode-dependent transaction analysis (debugging, forensics, compliance tooling) is completely unavailable. Because opcode replay is the only mechanism for detailed EVM execution tracing, this blocks block-level transaction processing analysis indefinitely at negligible attacker cost.

### Likelihood Explanation
No privileges, no account, no API key, and no special knowledge are required. The attacker needs only a valid (or even invalid — the throttle fires before the transaction lookup) transaction hash and the `Accept-Encoding: gzip` header. The attack is trivially scriptable with `curl` or any HTTP client at 1 req/s, well below any network-level rate-limiting threshold. It is fully repeatable and persistent as long as the attacker maintains the 1 req/s cadence.

### Recommendation
Replace the single global bucket with per-source-IP buckets (e.g., using Bucket4j's `ProxyManager` keyed on `HttpServletRequest.getRemoteAddr()` or the `X-Forwarded-For` header behind a trusted proxy). Each IP should receive its own independent token budget. Additionally:
- Raise the global ceiling to a value that can serve multiple concurrent legitimate users.
- Add `SynchronizationStrategy.SYNCHRONIZED` (or `LOCK_FREE`) to the opcode bucket consistent with `gasLimitBucket`.
- Consider requiring an API key or authentication for the opcode endpoint given its computational cost.

### Proof of Concept
```bash
# Terminal 1 – attacker: consume the single global token every second
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/opcodes"
  sleep 0.9
done

# Terminal 2 – victim: every request returns 429 Too Many Requests
curl -v -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/<valid_tx_hash>/opcodes"
# Expected: HTTP/1.1 429 Too Many Requests
# Body: {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```

The attacker's requests may themselves return 404 (transaction not found) or 200, but the throttle token is consumed before the transaction lookup, so even requests to nonexistent transactions exhaust the budget. [4](#0-3)
