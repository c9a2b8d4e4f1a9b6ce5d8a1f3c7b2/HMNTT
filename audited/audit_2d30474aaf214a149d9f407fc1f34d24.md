### Title
Unbounded `json_agg()` Aggregation of Signatures per Schedule in `getSchedules()` Enables DB Exhaustion

### Summary
In `rest/schedules.js`, `getSchedules()` issues a secondary `signatureQuery` that calls `json_agg()` over **all** rows in `transaction_signature` for each returned `schedule_id` with no per-schedule row cap. The user-controlled `limit` parameter only bounds the number of schedules in the outer query (max 100), not the number of signatures aggregated per schedule. Because the `transaction_signature` table carries no unique constraint by design, a schedule can accumulate an unbounded number of signature rows, and any unauthenticated caller can repeatedly trigger the full aggregation work against the shared database.

### Finding Description

**Exact code path:**

`rest/schedules.js` lines 39–45 define `transactionSignatureJsonAgg` with no row limit:

```js
const transactionSignatureJsonAgg = `
  json_agg(json_build_object(
    'consensus_timestamp', ts.consensus_timestamp,
    'public_key_prefix', encode(ts.public_key_prefix, 'base64'),
    'signature', encode(ts.signature, 'base64'),
    'type', ts.type
  ) order by ts.consensus_timestamp)`;
```

`getSchedules()` (lines 255–259) uses this expression in a secondary query with no LIMIT on signatures:

```js
const signatureQuery = `select entity_id, ${transactionSignatureJsonAgg} as signatures
    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;
```

**Root cause:** The `limit` parameter (validated and capped at max 100 by `getLimitParamValue`) controls only the number of schedules returned by the outer `schedulesMainQuery`. The `signatureQuery` is then issued for those schedule IDs with no `LIMIT` clause on the number of signatures aggregated per `entity_id`. PostgreSQL must scan and `json_agg()` every matching row in `transaction_signature` for each schedule ID.

**Why the existing check fails:** `extractSqlFromScheduleFilters` (lines 154–204) applies `filterQuery.limit` only to the schedule-level query via `scheduleLimitQuery`. There is no analogous cap applied inside `signatureQuery`. The `transaction_signature` table schema (V2.0.0 migration, lines 701–708) has no unique constraint — the design document explicitly states: *"There's no unique constraint/primary key since the client can potentially sign multiple times with the same key"* — so signature rows per schedule are unbounded at the storage level.

**Exploit flow:**
1. Attacker submits many `ScheduleSign` transactions on the Hedera network for one or more schedule IDs (each costs a small HBAR fee). Each accepted transaction causes the importer to insert one or more rows into `transaction_signature` for that `entity_id`.
2. Attacker (or any unauthenticated caller) sends repeated `GET /api/v1/schedules` requests. No authentication is required.
3. For each request, `getSchedules()` first fetches up to 100 schedule rows, then fires `signatureQuery` which forces PostgreSQL to scan and `json_agg()` every signature row for those schedule IDs — potentially millions of rows — before returning.
4. Concurrent requests multiply the DB CPU and I/O load. Because the mirror node REST service shares a single PostgreSQL instance across all API replicas, saturation of the DB degrades every replica simultaneously.

### Impact Explanation

The shared PostgreSQL database is the single point of aggregation for all mirror node REST API instances. Forcing repeated, unbounded `json_agg()` scans over a large `transaction_signature` table saturates DB CPU, memory (for accumulating the JSON array), and I/O. This degrades or blocks all concurrent queries across every REST API replica that shares the database, including health checks and other endpoints. The impact is proportional to the number of accumulated signature rows and the request rate, both of which an attacker can control independently of each other.

### Likelihood Explanation

The attack requires no authentication and no special privilege — only the ability to call `GET /api/v1/schedules`. The precondition (many signatures per schedule) can be established with modest HBAR funds via repeated `ScheduleSign` transactions, or may already exist naturally in a production environment with active multi-sig schedules. Once the signature rows exist, the read-side amplification is permanent and repeatable at zero additional cost. The endpoint is publicly documented in the OpenAPI spec and requires no credentials.

### Recommendation

1. **Add a per-schedule signature LIMIT in `signatureQuery`**: Use a lateral subquery or `FETCH FIRST N ROWS ONLY` inside the aggregation to cap the number of signatures fetched per `entity_id` (e.g., 25 or a configurable maximum).
2. **Add a DB-level statement timeout** for the REST Node.js service (analogous to the `statementTimeout` already present for REST Java) to bound the maximum wall-clock time any single query can consume.
3. **Consider a unique constraint** on `(entity_id, consensus_timestamp, public_key_prefix)` in `transaction_signature` to prevent unbounded row accumulation per schedule at the storage level.

  const {filterQuery, params, order, limit} = extractSqlFromScheduleFilters(filters);

  const schedulesResponse = {schedules: [], links: {next: null}};
  res.locals[constants.responseDataLabel] = schedulesResponse;

  if (schedules.length === 0) {
    return;

  const entityIds = schedules.map((s) => s.schedule_id);
  const positions = range(1, entityIds.length + 1)
    .map((i) => `$${i}`)

    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;

  const [{rows: entities}, {rows: signatures}] = await Promise.all([
    pool.queryQuietly(entityQuery, entityIds),
