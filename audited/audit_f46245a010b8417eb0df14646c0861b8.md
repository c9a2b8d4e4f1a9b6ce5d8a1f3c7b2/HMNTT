### Title
Unauthenticated Full `token_account` Table Scan via Unfiltered Balance Query in `getAccounts()`

### Summary
An unauthenticated attacker can send repeated concurrent `GET /api/v1/accounts?account.balance=gte:0&order=asc&limit=100` requests with no `account.id` filter. Each request causes `getEntityBalanceQuery()` to build and execute a query containing a `latest_token_balance` CTE that performs a full sequential scan of the `token_account` table, combined with a full scan of the `entity` table. Because the REST API has no per-IP rate limiting, an attacker can flood the service with concurrent requests, exhausting database CPU and connection pool resources.

### Finding Description

**Code path:**

`getAccounts()` at [1](#0-0)  calls `getAccountQuery()` → `getEntityBalanceQuery()`.

**Step 1 – No `account.id` filter check:**
At [2](#0-1) , `entityAccountQuery` is built from `parseAccountIdQueryParam`. When no `account.id` is supplied, `entityAccountQuery.query` is an empty string. The conditional at line 347 is therefore false, so `tokenBalanceQuery` retains only `account_id = e.id` with no account restriction.

**Step 2 – Balance filter matches all rows:**
`parseBalanceQueryParam` at [3](#0-2)  translates `account.balance=gte:0` into `e.balance >= 0`, which matches every account and contract row in the `entity` table.

**Step 3 – Full `token_account` table scan via CTE:**
Inside `getEntityBalanceQuery()`, when `needHistoricalBalanceInfo` is false (no timestamp filter), the code at [4](#0-3)  prepends:
```sql
with latest_token_balance as (
   select account_id, balance, token_id
   from token_account
   where associated is true)
```
This CTE performs an unconditional full sequential scan of the entire `token_account` table on every request. On a production mirror node this table contains tens of millions of rows.

**Step 4 – `entity_stake` join with no account condition:**
`getEntityStakeAccountCondition()` at [5](#0-4)  returns an empty string when `entityAccountQuery.query` is empty, so the LEFT JOIN to `entity_stake` has no narrowing predicate beyond `es.id = e.id`.

**Step 5 – Limit is capped but does not prevent the CTE scan:**
`getLimitParamValue()` at [6](#0-5)  caps the limit at `responseLimit.max` (default **100**). The `LIMIT 100` is applied to the outer query, but the `latest_token_balance` CTE is evaluated before the limit is applied, causing the full `token_account` scan regardless.

**Step 6 – No per-IP rate limiting on the REST API:**
The throttling infrastructure found in the codebase ( [7](#0-6) ) applies only to the web3 Java service. The Node.js REST API has no equivalent per-IP or per-endpoint rate limiter. The only protection is a DB-level `statement_timeout` configured in [8](#0-7) , which limits individual query duration but does not prevent concurrent query flooding.

### Impact Explanation
Each request triggers a full sequential scan of `token_account` (potentially tens of millions of rows) plus a full scan of `entity`. Under concurrent attack, the database CPU is saturated, the connection pool (`max: config.db.pool.maxConnections`) is exhausted, and legitimate requests begin timing out or failing. Because the mirror node REST API is stateless and horizontally scaled, all replicas share the same PostgreSQL backend; saturating the DB effectively degrades or halts all REST API processing nodes simultaneously, meeting the ≥30% processing node impact threshold.

### Likelihood Explanation
No authentication, no special knowledge, and no privileged access are required. The request is a single valid HTTP GET with standard query parameters. The attack is trivially scriptable (e.g., `ab -n 10000 -c 200 "http://mirror-node/api/v1/accounts?account.balance=gte:0&order=asc&limit=100"`). It is repeatable indefinitely and requires no state. Any external actor who can reach the public API endpoint can execute it.

### Recommendation
1. **Require `account.id` filter when `account.balance` filter is used without a timestamp**, or enforce a mandatory lower-bound `account.id` pagination cursor to prevent unbounded entity table scans.
2. **Add per-IP rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit`) for the `/api/v1/accounts` endpoint.
3. **Avoid unconditional CTE full scans**: rewrite the `latest_token_balance` CTE to be correlated or deferred until after the outer `entity` query resolves its result set, so `token_account` is only scanned for the accounts actually returned.
4. **Set a conservative `statement_timeout`** and **connection pool limit** as a backstop, and monitor for query queue depth.

### Proof of Concept
```bash
# Single request demonstrating full-scan query path (no account.id filter):
curl "http://<mirror-node>/api/v1/accounts?account.balance=gte:0&order=asc&limit=100"

# Concurrent flood to exhaust DB resources:
ab -n 5000 -c 100 \
  "http://<mirror-node>/api/v1/accounts?account.balance=gte:0&order=asc&limit=100"

# Expected result: DB CPU spikes to 100%, connection pool exhausted,
# subsequent requests return 500 errors or time out, degrading ≥30% of
# mirror node REST processing capacity.
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
