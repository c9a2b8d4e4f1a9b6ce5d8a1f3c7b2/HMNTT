### Title
NFT Account Query: Unprivileged 3× Row Amplification via UNION ALL Subquery Limit Reuse

### Summary
In `rest/service/nftService.js`, `getQuery()` constructs up to three independent subqueries (lower, inner, upper) each bounded by the same user-supplied `LIMIT $2`. When all three subqueries are active, the database must fetch, sort, and merge up to `3 × limit` rows before the outer re-limit is applied. Any unauthenticated caller can trigger this by supplying a closed token.id range with serial number bounds on both ends, combined with `limit` set to the configured maximum, causing database CPU and memory consumption well above 30% compared to a default single-subquery request.

### Finding Description

**Exact code path:**

In `getQuery()`, a single `limitClause` is built once and reused for every subquery: [1](#0-0) 

```js
const params = [ownerAccountId, limit];          // limit at $2
const limitClause = super.getLimitQuery(2);       // always "LIMIT $2"
```

Each non-empty filter group (lower, inner, upper) is mapped to a full subquery that independently applies `limitClause`: [2](#0-1) 

When two or more subqueries exist, they are joined with `UNION ALL` and the outer query also applies the same `limitClause`: [3](#0-2) 

The test suite confirms the three-subquery shape with all three `LIMIT $2` clauses: [4](#0-3) 

**Root cause:** The limit parameter is placed in `params[1]` once and referenced as `$2` in every subquery. There is no mechanism to reduce the per-subquery limit when multiple subqueries are combined.

**How the limit reaches `getQuery()`:**

In `accountController.js`, `extractNftMultiUnionQuery()` reads the limit directly from the validated filter value: [5](#0-4) 

`buildAndValidateFilters` caps the value at the configured maximum (default 100), but that cap is the ceiling each subquery independently uses. [6](#0-5) 

### Impact Explanation

With `limit=100` (the configured max) and a request that activates all three subqueries, PostgreSQL must:
- Execute three independent index scans, each returning up to 100 rows (300 rows total fetched).
- Sort and merge those 300 rows in the outer `ORDER BY … LIMIT 100`.

Compared to a default single-subquery request (25 rows, no bounds), this is a **12× row amplification**. Even compared to a max-limit single-subquery request (100 rows), it is a **3× amplification** — well above the 30% threshold. This is repeatable, stateless, and requires no authentication.

### Likelihood Explanation

The endpoint `/api/v1/accounts/{id}/nfts` is public and unauthenticated. The required query parameters (`token.id=gte:X&token.id=lte:Y&serialnumber=gt:A&serialnumber=lt:B&limit=100`) are documented in the OpenAPI spec: [7](#0-6) 

Any external user can craft this request. It requires no special knowledge beyond reading the public API documentation. The attack is trivially repeatable in a loop without brute force.

### Recommendation

Divide the per-subquery limit by the number of active subqueries (minimum 1), so the total rows fetched across all subqueries never exceeds `limit`. For example:

```js
const subQueryCount = [lower, inner, upper].filter(f => f.length !== 0).length || 1;
const effectiveSubLimit = Math.ceil(limit / subQueryCount);
// use effectiveSubLimit in each subquery's LIMIT clause
// keep the outer LIMIT as the original limit
```

Alternatively, enforce a lower absolute maximum for the NFT multi-union endpoint specifically, or add rate limiting per IP/account on this endpoint.

### Proof of Concept

**Precondition:** A Hedera account (e.g., `0.0.1001`) holds NFTs across multiple token IDs.

**Trigger request (activates all three subqueries — lower, inner, upper):**

```
GET /api/v1/accounts/0.0.1001/nfts?token.id=gte:1000&token.id=lte:2500&serialnumber=gt:1&serialnumber=lt:50&limit=100
```

**Generated SQL (confirmed by test at lines 267–295 of nftService.test.js):**

```sql
(SELECT ... FROM nft LEFT JOIN entity e ...
 WHERE account_id = $1 AND token_id = $3 AND serial_number >= $4
 ORDER BY token_id DESC, serial_number DESC LIMIT $2)   -- up to 100 rows
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity e ...
 WHERE account_id = $1 AND token_id > $5 AND token_id < $6
 ORDER BY token_id DESC, serial_number DESC LIMIT $2)   -- up to 100 rows
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity e ...
 WHERE account_id = $1 AND token_id = $7 AND serial_number <= $8
 ORDER BY token_id DESC, serial_number DESC LIMIT $2)   -- up to 100 rows
ORDER BY token_id DESC, serial_number DESC LIMIT $2     -- re-sort 300 rows, return 100
```

**Result:** The database fetches, sorts, and merges up to 300 rows per request. Sending this request in a sustained loop (e.g., 10 req/s) multiplies the DB load by 3× compared to a normal max-limit request, achieving >30% resource increase without brute force.
