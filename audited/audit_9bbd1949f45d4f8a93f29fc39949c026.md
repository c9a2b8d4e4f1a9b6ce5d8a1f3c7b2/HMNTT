### Title
Unbounded `limit` Parameter in `getHooks()` Enables Resource Exhaustion via Oversized DB Result Set

### Summary
The `getHooks(HooksRequest)` method in `HookServiceImpl` passes `request.getLimit()` directly to `PageRequest.of()` with no upper-bound cap or validation. An unprivileged external user can supply an arbitrarily large `limit` value, causing the database to return a massive result set that is then fully materialized in heap memory and serialized to JSON, consuming significant CPU and memory resources.

### Finding Description
**Exact code path:**

In `HooksRequest.java`, the `limit` field is a plain `int` with a default of 25 and no `@Max`, `@Min`, or any JSR-380 constraint annotation: [1](#0-0) 

In `HookServiceImpl.java`, `getHooks()` constructs a `PageRequest` directly from the uncapped `request.getLimit()`: [2](#0-1) 

This `page` object is then passed verbatim to the repository: [3](#0-2) 

`HookRepository.findByOwnerIdAndHookIdBetween` is a Spring Data derived query that translates the `Pageable` size directly into a SQL `LIMIT` clause: [4](#0-3) 

**Root cause:** No maximum limit is enforced anywhere in the call chain — not in the DTO, not in the service, and not (based on available evidence) in the controller. The `lowerBound=0` and `upperBound=Long.MAX_VALUE` are the compiled-in defaults, so the attacker does not even need to supply them. [5](#0-4) 

### Impact Explanation
A single HTTP request with `limit=2147483647` (or any large integer) causes:
1. The database to execute `SELECT … WHERE owner_id=? AND hook_id BETWEEN 0 AND 9223372036854775807 LIMIT 2147483647`, potentially returning millions of rows.
2. All returned `Hook` objects to be fully materialized as a `List<Hook>` in JVM heap.
3. The entire list to be serialized to JSON by the HTTP response layer, consuming sustained CPU.

This can raise CPU and heap usage well above 30% of baseline from a single unauthenticated (or low-privilege) request, and repeated requests can cause heap exhaustion / OOM, degrading or crashing the node.

### Likelihood Explanation
- No authentication or rate-limiting is visible in the service layer.
- The exploit requires only a valid `ownerId` (an entity ID resolvable via `entityService.lookup`) and a crafted `limit` query parameter — both trivially obtainable by any external user.
- The attack is repeatable and requires no special tooling beyond a standard HTTP client.
- The `lowerBound`/`upperBound` defaults already cover the full ID space, so only `limit` needs to be manipulated.

### Recommendation
1. Add a `@Max` constraint (e.g., `@Max(100)`) on the `limit` field in `HooksRequest` and enable `@Validated` on the controller method.
2. Enforce a hard cap inside `HookServiceImpl.getHooks()`:
   ```java
   private static final int MAX_LIMIT = 100;
   final var page = PageRequest.of(0, Math.min(request.getLimit(), MAX_LIMIT), sort);
   ```
3. Apply rate-limiting at the API gateway or controller level to prevent repeated large queries from the same source.

### Proof of Concept
```
# Assuming ownerId resolves to a valid entity (e.g., 0.0.1234)
GET /api/v1/hooks?ownerId=0.0.1234&limit=2147483647

# Resulting SQL (Spring Data):
# SELECT * FROM hook
#   WHERE owner_id = 1234
#     AND hook_id BETWEEN 0 AND 9223372036854775807
#   ORDER BY hook_id DESC
#   LIMIT 2147483647;
```
Sending this request repeatedly (or even once against a large dataset) will spike JVM heap allocation and CPU serialization load, measurably increasing node resource consumption beyond 30% of the 24-hour baseline.
