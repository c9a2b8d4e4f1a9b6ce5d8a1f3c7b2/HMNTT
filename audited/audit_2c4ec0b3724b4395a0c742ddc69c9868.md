### Title
Unauthenticated DB Round-Trip Waste via Nonexistent `transaction.hash` on `GET /contracts/results/logs`

### Summary
Any unauthenticated caller can supply a syntactically valid but nonexistent `transaction.hash` value to `GET /contracts/results/logs`. The handler unconditionally issues a database stored-procedure call (`get_transaction_info_by_hash`) before discovering the hash does not exist, then returns an empty 200 response. Because the response-cache key is the full URL, an attacker rotating unique hash values bypasses caching entirely, forcing one DB round-trip per request with no preconditions.

### Finding Description
**Exact code path:**

`rest/routes/contractRoute.js:20` registers `GET /results/logs` → `ContractController.getContractLogs`.

`rest/controllers/contractController.js:820-848` — `getContractLogs()`:
- Line 822: `buildAndValidateFilters` performs **format-only** regex validation via `isValidTransactionHash` (accepts any 32-byte hex or 48-byte base64url string).
- Line 835: calls `extractContractLogsMultiUnionQuery(filters)` — `await`ed, so the DB call blocks the response.

`rest/controllers/contractController.js:582-699` — `extractContractLogsMultiUnionQuery()`:
- Lines 645-651: parses `TRANSACTION_HASH` filter into `transactionHash` buffer.
- Lines 669-674: **unconditionally** calls `getTransactionHash(transactionHash, …)` whenever `transactionHash !== undefined`, then returns `null` if `rows.length === 0`.

`rest/transactionHash.js:21-37` — `getTransactionHash()`:
- Line 35: `pool.queryQuietly(query, params)` executes `select * from get_transaction_info_by_hash($1)` — a full DB stored-procedure call.
- No negative-result cache, no existence pre-check.

Back in `getContractLogs()` lines 836-838: `if (query === null) { return; }` — silently returns the pre-set empty body `{logs: [], links: {next: null}}` with HTTP 200.

**Root cause:** The system assumes a `transaction.hash` filter will usually match; there is no short-circuit before the DB call, and the response cache (keyed on the full URL) is trivially defeated by rotating the hash value.

### Impact Explanation
Each unique nonexistent hash forces one execution of the `get_transaction_info_by_hash` stored procedure. In the v2 schema this procedure performs a two-phase timestamp-range scan of the `transaction_hash` table; in v1 it routes to a shard table. A sustained flood of requests with unique valid-format hashes exhausts DB connection pool slots and increases query latency for all other API consumers. No data is leaked and no state is modified, placing this squarely in the griefing / resource-exhaustion category with no direct economic damage to network participants.

### Likelihood Explanation
Zero preconditions: no account, no authentication, no special knowledge required. A valid 32-byte hex hash (64 hex chars) is trivially generated (`0x` + 64 random hex digits). The attack is fully scriptable, repeatable at any rate the network allows, and the response cache provides no protection because each unique hash is a distinct cache key.

### Recommendation
1. **Negative-result cache**: after `getTransactionHash` returns empty, cache the `(hash → empty)` result for a short TTL (e.g., 5 s) so repeated identical hashes do not re-hit the DB.
2. **Rate limiting**: apply a per-IP or global rate limit specifically on `transaction.hash`-bearing requests at the middleware layer, since this parameter is the only one that forces an extra DB round-trip before the main query.
3. **Fail-fast on obviously invalid hashes**: while format validation is already present, consider a lightweight bloom-filter or in-memory LRU of recently-missed hashes to short-circuit repeated misses without a DB call.

### Proof of Concept
```bash
# Generate 1000 unique valid-format nonexistent hashes and fire them concurrently.
# Each triggers one get_transaction_info_by_hash() DB call.
for i in $(seq 1 1000); do
  HASH=$(openssl rand -hex 32)   # 64-char hex = valid format
  curl -s "https://<mirror-node>/api/v1/contracts/results/logs?transaction.hash=${HASH}" &
done
wait
# All return HTTP 200 {"logs":[],"links":{"next":null}}
# DB shows 1000 executions of get_transaction_info_by_hash with 0 rows returned.
```

Observed: HTTP 200 `{"logs":[],"links":{"next":null}}` for every request; one DB stored-procedure execution per unique hash; response cache never reused across distinct hashes.
