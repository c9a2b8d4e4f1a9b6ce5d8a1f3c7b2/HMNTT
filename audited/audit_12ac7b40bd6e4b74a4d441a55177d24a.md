Based on my investigation of the code, I have all the information needed to evaluate this claim.

**Key findings:**

1. `getEncodedId()` triggers real DB queries only for alias or EVM address inputs (lines 124-126 of `entityService.js`). Plain numeric IDs return immediately without a DB call. [1](#0-0) 

2. The DB connection pool defaults to **10 connections** (`hiero.mirror.rest.db.pool.maxConnections` = 10), with a `connectionTimeoutMillis` of 20,000ms and `statement_timeout` of 20,000ms. [2](#0-1) 

3. At least 5 endpoints in `accountRoute.js` call `getEncodedId()` (nfts, rewards, allowances/crypto, allowances/tokens, tokens), plus contract results endpoints. [3](#0-2) 

4. `server.js` has **no rate limiting middleware** for the REST API. The only throttling in the codebase is in the separate `web3` Java service. [4](#0-3) 

5. The `statement_timeout` of 20 seconds acts as a partial mitigation — connections are released after timeout — but with only 10 connections and no rate limiting, an attacker sending ≥10 concurrent alias/EVM-address requests holds the entire pool for up to 20 seconds per wave.

**Assessment:**

The scenario described is a standard **unauthenticated DB connection pool exhaustion DoS**. It is real and exploitable. However, the specific claim about "30%+ of mirror node instances" requires the attacker to simultaneously target multiple deployed replicas, which is feasible but adds operational complexity. The impact is on the mirror node REST API availability, not on Hedera consensus nodes — the mirror node is a read-only service and its unavailability does not affect network consensus or transaction processing.

The `connectionTimeoutMillis` (20s) and `statement_timeout` (20s) are the only mitigations, and they are insufficient against a sustained flood.

### Title
Unauthenticated DB Connection Pool Exhaustion via `getEncodedId()` Across Multiple REST Endpoints

### Summary
The REST API's `getEncodedId()` function issues synchronous DB queries when called with alias or EVM address inputs. With a default pool of only 10 connections, no rate limiting on the REST API, and at least 6 endpoints that invoke this function, an unprivileged attacker can exhaust the connection pool with ≥10 concurrent requests, rendering the mirror node REST API unresponsive until connections time out.

### Finding Description
**Code path:** `rest/service/entityService.js`, `getEncodedId()`, lines 118–137.

When `entityIdString` is an alias (passes `AccountAlias.isValid()`), `getAccountIdFromAlias()` → `getAccountFromAlias()` → `super.getRows(entityFromAliasQuery, ...)` is called, acquiring a DB connection from the global pool. When it is an EVM address, `getEntityIdFromEvmAddress()` → `this.getRows(entityFromEvmAddressQuery, ...)` is called similarly. [1](#0-0) 

`BaseService.getRows()` calls `this.pool().queryQuietly(query, params)`, which acquires a connection from the `pg` pool configured in `rest/dbpool.js` with `max: config.db.pool.maxConnections` (default: **10**) and `statement_timeout: 20000` ms. [5](#0-4) [2](#0-1) 

The following endpoints all call `EntityService.getEncodedId()` as their first async operation:
- `GET /api/v1/accounts/:id/nfts` (`AccountController.getNftsByAccountId`, line 91)
- `GET /api/v1/accounts/:id/rewards` (`AccountController.listStakingRewardsByAccountId`)
- `GET /api/v1/accounts/:id/allowances/crypto` (`CryptoAllowanceController`)
- `GET /api/v1/accounts/:id/allowances/tokens` (`TokenAllowanceController`)
- `GET /api/v1/accounts/:id/tokens` (`TokenController`)
- Contract results `from` parameter (`ContractController`) [3](#0-2) [6](#0-5) 

`server.js` applies no rate limiting middleware to any of these routes. [7](#0-6) 

**Root cause:** The combination of (a) no rate limiting, (b) a tiny default pool (10 connections), (c) a long statement timeout (20 s), and (d) multiple public endpoints that each consume a pool connection for alias/EVM-address lookups.

**Failed assumption:** The design assumes that DB queries complete quickly and connections are returned promptly. Under adversarial load with alias inputs (which may scan the `entity` table), this assumption fails.

### Impact Explanation
With 10 connections and a 20-second statement timeout, an attacker sending 10 concurrent alias-based requests holds the entire pool for up to 20 seconds. All other API requests (health checks, legitimate user queries) queue up waiting for a connection. After `connectionTimeoutMillis` (20 s), queued requests fail with a connection timeout error. The REST API instance becomes effectively unresponsive. If multiple replicas are targeted simultaneously (trivial with a botnet or even a single machine with parallel HTTP clients), 30%+ of mirror node REST instances can be taken offline. The mirror node's read API is unavailable, breaking downstream applications and block explorers.

### Likelihood Explanation
No authentication or rate limiting is required. Any internet-accessible mirror node is vulnerable. The attack requires only an HTTP client capable of sending ≥10 concurrent GET requests with a valid-looking alias string (e.g., a base32-encoded value) as the path parameter. This is trivially scriptable with `curl`, `ab`, `wrk`, or any load-testing tool. The attack is repeatable and sustainable indefinitely.

### Recommendation
1. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit`) before route handlers, limiting requests per IP per second.
2. **Increase the default pool size** or configure it relative to expected concurrency.
3. **Add a concurrency limiter** (e.g., `p-limit`) around DB-bound operations in `getEncodedId()` to cap simultaneous in-flight DB queries.
4. **Cache alias/EVM-address → encoded ID lookups** (the configuration already defines `hiero.mirror.rest.cache.entityId` with `maxSize=100000` and `maxAge=1800` — verify this cache is actually applied to `getEncodedId()` results).
5. **Reduce `statement_timeout`** for alias/EVM-address lookups to limit how long a connection is held.
