### Title
Missing `@Validated` on `AllowancesController` Renders `@Max(MAX_LIMIT)` Unenforced, Allowing Unbounded NFT Allowance Queries

### Summary
`AllowancesController` declares `@Max(MAX_LIMIT)` on the `limit` request parameter, but the class lacks the Spring `@Validated` annotation required to activate Bean Validation on method parameters. Without it, the constraint is silently ignored at runtime, allowing any unauthenticated caller to supply an arbitrarily large `limit` value (up to `Integer.MAX_VALUE`). That value flows unmodified into the JOOQ query's `.limit()` clause, causing the database to attempt returning billions of rows.

### Finding Description

**Controller — constraint declared but never enforced**

`AllowancesController` (line 61) annotates the `limit` parameter with `@Positive @Max(MAX_LIMIT)`:

```java
@RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
``` [1](#0-0) 

The class-level annotations are only `@CustomLog`, `@RequestMapping`, `@RequiredArgsConstructor`, and `@RestController` — **`@Validated` is absent**:

```java
@CustomLog
@RequestMapping(value = "/api/v1/accounts/{id}/allowances", produces = APPLICATION_JSON)
@RequiredArgsConstructor
@RestController
public class AllowancesController {
``` [2](#0-1) 

In Spring MVC, `jakarta.validation` constraints on `@RequestParam` / `@PathVariable` parameters are only activated when the controller (or a parent class/interface) carries `@org.springframework.validation.annotation.Validated`. A search across the entire `restjava` module confirms `@Validated` appears only on `RestJavaProperties`, `NetworkProperties`, and `QueryProperties` — never on any controller.

**Unchecked value flows directly into the repository**

The raw `limit` is placed into `NftAllowanceRequest` with no further clamping:

```java
var request = NftAllowanceRequest.builder()
        ...
        .limit(limit)   // attacker-controlled, no cap applied
        ...
        .build();
var serviceResponse = service.getNftAllowances(request);
``` [3](#0-2) 

`NftAllowanceServiceImpl.getNftAllowances()` performs only parameter-relationship validation (operator checks, range checks) and never inspects the `limit` field: [4](#0-3) 

The value reaches the JOOQ query verbatim:

```java
.limit(request.getLimit())
.fetchInto(NftAllowance.class);
``` [5](#0-4) 

`NftAllowanceRequest.limit` has no validation annotation of its own and defaults to `25` only when built via the builder default — the controller overrides it with the attacker-supplied value: [6](#0-5) 

### Impact Explanation
An unauthenticated attacker can issue repeated requests with `limit=2147483647`. Each request causes the mirror node to execute a full-table-scan-equivalent SQL query with `LIMIT 2147483647`, materializing every matching NFT allowance row into a Java `Collection<NftAllowance>`. This exhausts database I/O, JDBC connection pool threads, and JVM heap simultaneously. A single sustained burst can render the mirror node unresponsive for all users. Because NFT allowance data is public read-only state, no account credentials are required.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication. The exploit requires only a single HTTP GET request with a crafted query string parameter — no special tooling, no prior knowledge of internal state, and no rate-limit bypass needed (unless a separate rate-limiter is deployed in front, which is not visible in this codebase). It is trivially repeatable and scriptable.

### Recommendation
Add `@Validated` to `AllowancesController` so Spring activates the existing `@Max(MAX_LIMIT)` constraint:

```java
import org.springframework.validation.annotation.Validated;

@CustomLog
@RequestMapping(value = "/api/v1/accounts/{id}/allowances", produces = APPLICATION_JSON)
@RequiredArgsConstructor
@RestController
@Validated   // ← add this
public class AllowancesController {
```

As a defense-in-depth measure, also add a hard clamp inside `NftAllowanceServiceImpl.getNftAllowances()` or inside `NftAllowanceRepositoryCustomImpl.findAll()` so that even if the controller validation is bypassed (e.g., via a future internal caller), the limit is capped before reaching the database. Verify that all other controllers in the `restjava` module that declare `@Max` / `@Positive` on parameters are similarly annotated with `@Validated`.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1/allowances/nfts?limit=2147483647
Host: <mirror-node-host>
```

1. Send the above request to any publicly reachable mirror node REST-Java endpoint.
2. Observe that no `400 Bad Request` is returned (the `@Max` constraint is not enforced).
3. The mirror node issues `SELECT * FROM nft_allowance WHERE owner = 1 AND approved_for_all = true ORDER BY spender ASC, token_id ASC LIMIT 2147483647` to the database.
4. Monitor database CPU/I/O and mirror node heap usage — both spike to exhaustion proportional to the number of NFT allowance rows in the database.
5. Repeat in a loop to sustain denial of service.
