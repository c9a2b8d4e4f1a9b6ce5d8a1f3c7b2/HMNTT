### Title
One-Way Latch in `shouldDownload()` Enables Permanent DoS of Balance Downloads via Synthetic DB Row

### Summary
`AccountBalancesDownloader.shouldDownload()` uses a one-way `AtomicBoolean` latch (`accountBalanceFileExists`) that is set permanently to `true` the first time `accountBalanceFileRepository.findLatest()` returns a non-empty result. Because `findLatest()` blindly trusts any row in the `account_balance_file` table without authenticity validation, an attacker who can write a single row to that table (e.g., via SQL injection in any upstream component sharing the same DB user) can permanently halt all balance file downloads for the lifetime of the process.

### Finding Description

**Exact code path:**

`AccountBalancesDownloader.java`, `shouldDownload()`, lines 69–84:

```java
protected boolean shouldDownload() {
    if (downloaderProperties.isEnabled()) {   // line 70 — bypassed when enabled=false (default)
        return true;
    }

    if (accountBalanceFileExists.get()) {     // line 74 — one-way latch, never reset to false
        return false;
    }

    if (accountBalanceFileRepository.findLatest().isPresent()) {  // line 78 — trusts any DB row
        accountBalanceFileExists.set(true);   // line 79 — latch permanently set
        return false;
    }

    return true;
}
```

`BalanceDownloaderProperties.enabled` defaults to `false` (line 25 of `BalanceDownloaderProperties.java`), so the `isEnabled()` guard at line 70 does **not** short-circuit to `true` in the default deployment. Execution falls through to the latch logic every scheduled invocation.

`findLatest()` executes:
```sql
select * from account_balance_file order by consensus_timestamp desc limit 1
```
(line 12 of `AccountBalanceFileRepository.java`) — no filter on authenticity, source, or the `synthetic` column.

**Root cause:** The `AtomicBoolean accountBalanceFileExists` is initialized to `false` and only ever set to `true` — it is a one-way latch with no reset path. Once set, every subsequent call to `shouldDownload()` returns `false` at line 74–76 without re-querying the DB, making the condition irrecoverable for the lifetime of the process.

**Exploit flow:**
1. Attacker exploits SQL injection in any upstream component that shares the importer's DB credentials (which hold `GRANT ALL ON TABLES` per the migration scripts).
2. Attacker executes: `INSERT INTO account_balance_file (consensus_timestamp, count, name, ...) VALUES (...)`
3. On the next scheduled `download()` call, `downloadNextBatch()` calls `shouldDownload()`.
4. `isEnabled()` returns `false` (default). `accountBalanceFileExists.get()` returns `false` (first call).
5. `findLatest()` returns the injected row as a present `Optional`.
6. `accountBalanceFileExists.set(true)` — latch permanently engaged.
7. All subsequent `shouldDownload()` calls return `false` at line 74 without any DB query.
8. Balance downloads are permanently halted for the process lifetime. Even after restart, the malicious row persists in the DB, re-triggering the latch on the very first invocation.

**Why existing checks are insufficient:**
- The `isEnabled()` guard (line 70) only bypasses the latch when the operator has explicitly set `enabled=true`; the default is `false`.
- There is no validation of the inserted row's `synthetic` flag, `file_hash`, or any other integrity field in `shouldDownload()`.
- The `AtomicBoolean` has no reset mechanism; there is no code path that calls `accountBalanceFileExists.set(false)`.

### Impact Explanation
Balance file downloads are the mechanism by which the mirror node tracks account balances. Permanently halting them means the mirror node stops reflecting current account states, breaking any downstream consumer relying on balance data. Recovery requires both manual deletion of the injected DB row **and** a service restart — an operational intervention that may not be detected quickly in automated deployments. This is a Denial-of-Service with persistent state corruption.

### Likelihood Explanation
The precondition is SQL injection in any upstream component sharing the importer's DB user. The importer DB user is granted `ALL` privileges on all tables (per `V1.10.0__account_balances.sql`), so a single injection point anywhere in the stack is sufficient. The `account_balance_file` table has no row-level security or write audit triggers. The attack is repeatable: if the row is deleted and the service restarted, re-inserting a row re-triggers the latch. The attacker needs no knowledge of the application internals beyond the table name and a minimal valid row schema.

### Recommendation
1. **Remove the one-way latch pattern.** Re-query the DB on each `shouldDownload()` invocation, or add a reset path (e.g., `accountBalanceFileExists.set(false)` when the DB row is deleted).
2. **Filter `findLatest()` to exclude synthetic or unverified rows**, e.g., add `WHERE synthetic = false AND file_hash IS NOT NULL`.
3. **Apply least-privilege DB permissions**: the importer user should not need `INSERT` rights on `account_balance_file` from external-facing components; separate DB roles per component.
4. **Add integrity validation** before trusting a DB row as evidence that a legitimate balance file was processed (e.g., verify `file_hash` is non-null and matches expected format).

### Proof of Concept

**Preconditions:** `hiero.mirror.importer.downloader.balance.enabled=false` (default). Attacker has SQL write access to the importer DB (e.g., via SQL injection in a REST endpoint sharing the DB connection).

**Steps:**
```sql
-- 1. Inject a minimal valid row into account_balance_file
INSERT INTO account_balance_file
  (consensus_timestamp, count, name, load_start, load_end, synthetic)
VALUES
  (9999999999999999999, 0, 'fake_2099-01-01T00_00_00Z_Balances.csv', 0, 0, false);
```

```
-- 2. Wait for the next scheduled download() invocation (default: every 30 seconds)
-- 3. Observe in logs: no balance files are downloaded
-- 4. Restart the service
-- 5. Observe: latch is re-set on first invocation; downloads remain halted
-- 6. Only recovery: DELETE the injected row AND restart the service
```
