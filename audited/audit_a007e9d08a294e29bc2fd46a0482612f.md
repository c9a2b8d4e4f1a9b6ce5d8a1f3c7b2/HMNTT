### Title
Unbounded Timestamp Range in `extractContractStateByIdQuery()` Enables Unauthenticated DoS via Full `contract_state_change` Scan

### Summary
The `extractContractStateByIdQuery()` function in `rest/controllers/contractController.js` accepts a user-supplied `timestamp` filter and injects it directly into a `DISTINCT ON` SQL query against the `contract_state_change` table without enforcing any upper-bound or range limit on the timestamp value. Any unauthenticated caller can supply `timestamp=lte:9999999999.999999999`, causing the database to scan and sort all historical state-change rows for the targeted contract before applying the result `LIMIT`, degrading performance for all concurrent users.

### Finding Description

**Exact code path:**

`getContractStateById` (line 964) calls `extractContractStateByIdQuery` (line 970), which iterates over filters:

```js
// rest/controllers/contractController.js lines 909-918
case filterKeys.TIMESTAMP:
  if (utils.opsMap.ne === filter.operator) {
    throw new InvalidArgumentError(...);
  }
  if (utils.opsMap.eq === filter.operator) {
    filter.operator = utils.opsMap.lte;
  }
  conditions.push(this.getFilterWhereCondition(ContractStateChange.CONSENSUS_TIMESTAMP, filter));
  timestampPresent = true;
  break;
```

The only rejection is `ne`; any `lte`, `gte`, `lt`, `gt`, or `eq` value is accepted verbatim and appended to the SQL conditions. No call to `parseTimestampFilters()` (which enforces `maxTimestampRange`) is made anywhere in this path.

When `timestampPresent = true`, `getContractStateByIdAndFilters` (line 971 → `contractService.js` line 263) executes `contractStateTimestampQuery`:

```js
// rest/service/contractService.js lines 111-122
static contractStateTimestampQuery = `
  with ${ContractService.entityCTE}
  select DISTINCT on (${ContractStateChange.SLOT})
        ${ContractStateChange.CONTRACT_ID},
        ${ContractStateChange.SLOT},
        ...
        ${ContractStateChange.CONSENSUS_TIMESTAMP} as ${ContractState.MODIFIED_TIMESTAMP}
  from ${ContractStateChange.tableName} ${ContractStateChange.tableAlias}
  left join ...
`;
```

The resulting query becomes:
```sql
SELECT DISTINCT ON (slot) ...
FROM contract_state_change csc
WHERE contract_id = $1
  AND consensus_timestamp <= 9999999999999999999   -- attacker-controlled
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT $2
```

PostgreSQL's `DISTINCT ON` with `ORDER BY` requires the engine to **fully sort all matching rows** before it can apply `LIMIT`. For a heavily-used contract, this means scanning and sorting every row across all `consensus_timestamp` partitions of `contract_state_change`.

**Why existing checks fail:**

- `buildAndValidateFilters` / `filterValidityChecks` only validates the timestamp *format* (e.g., `\d+\.\d+`), not its magnitude or range.
- `parseTimestampFilters()` (lines 1583–1666 in `rest/utils.js`) contains the `maxTimestampRange` enforcement but is **never called** in the contract-state endpoint flow.
- Other endpoints (e.g., contract results) do call `parseTimestampFilters` or `optimizeTimestampFilters` with range validation; the state endpoint does not.
- The `LIMIT` clause does not prevent the expensive sort — it only caps the rows returned after the full `DISTINCT ON` computation.

### Impact Explanation
An attacker can force the database to perform an unbounded sequential scan and sort of `contract_state_change` for any contract, consuming CPU and I/O. Because `contract_state_change` is a partitioned table (partitioned by `consensus_timestamp`), a `lte:MAX_VALUE` predicate causes the planner to include every partition. Repeated requests against a popular contract (e.g., a DEX or stablecoin) can saturate database resources and degrade response times for all API users. No data is leaked or modified; the impact is availability degradation (griefing).

### Likelihood Explanation
The endpoint is unauthenticated and publicly documented. No API key, token, or special privilege is required. The exploit requires a single HTTP GET request with one crafted query parameter. It is trivially scriptable and repeatable at high frequency. Any motivated actor — including a competitor or vandal — can execute it without any on-chain cost.

### Recommendation
1. **Apply `parseTimestampFilters` with `validateRange: true`** in `extractContractStateByIdQuery` before building conditions, consistent with how other endpoints enforce `maxTimestampRange`.
2. **Require a bounded timestamp range** (both lower and upper bound) when the timestamp path is taken, or reject open-ended `lte`/`gte`-only filters for this endpoint.
3. **Add a hard cap** on the maximum allowed timestamp value (e.g., reject values beyond `now() + epsilon`).
4. Consider **rate-limiting** the `/contracts/:contractId/state` endpoint per source IP.

### Proof of Concept

**Precondition:** A contract with a non-trivial number of state changes exists (e.g., contract `0.0.1234`).

**Trigger:**
```
GET /api/v1/contracts/0.0.1234/state?timestamp=lte:9999999999.999999999
```

No authentication header required.

**Result:** The mirror node executes:
```sql
SELECT DISTINCT ON (slot) contract_id, slot, evm_address,
       coalesce(value_written, value_read) AS value,
       consensus_timestamp AS modified_timestamp
FROM contract_state_change csc
LEFT JOIN entity e ON e.id = csc.contract_id
WHERE csc.contract_id = 1234
  AND csc.consensus_timestamp <= 9999999999999999999
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT 25;
```
