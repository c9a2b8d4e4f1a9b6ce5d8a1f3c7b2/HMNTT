### Title
Unauthenticated DoS via Unbounded DB Query Flooding on `/contracts/results/{transactionIdOrHash}` Due to Absent Application-Level Rate Limiting

### Summary
The `getContractResultsByTransactionIdOrHash` handler in `rest/controllers/contractController.js` accepts syntactically valid transaction IDs from any unauthenticated caller and unconditionally issues a parameterized SQL query against the `transaction` table for each request. No application-level rate limiting exists in the REST service. An attacker can flood this endpoint with valid-format but non-existent payer IDs, exhausting the shared DB connection pool (default max 10 connections) and degrading all REST API processing.

### Finding Description

**Code path:**

`GET /api/v1/contracts/results/:transactionIdOrHash` → `contractController.js:getContractResultsByTransactionIdOrHash` (line 1120) → `TransactionService.getTransactionDetailsFromTransactionId` (`transactionService.js` line 64) → SQL query (lines 15–26).

The SQL issued is:

```sql
select consensus_timestamp, nonce, scheduled, type, payer_account_id
from transaction
where payer_account_id = $1
  and consensus_timestamp >= $2 and consensus_timestamp <= $3   -- 35-minute range
  and valid_start_ns = $2
  and nonce = (select coalesce($4, 0))
order by (result = <success>) desc, consensus_timestamp desc
limit 1
```

`$1` is the payer entity ID extracted directly from the URL path parameter. `$3` is `validStartNs + maxTransactionConsensusTimestampRangeNs` (default **35 minutes**, `docs/configuration.md` line 585).

**Root cause — failed assumption:** The code assumes that the cost of a no-match query is negligible and that upstream infrastructure (Traefik, Kubernetes) will absorb flood traffic. Neither assumption holds in the application layer:

1. **No application-level rate limiting.** A `grep` across `rest/**/*.js` for `rateLimit`, `throttle`, `express-rate`, or `helmet` returns only a single test-utility match. The REST Node.js server has no per-IP or global request-rate guard.
2. **DB connection pool is tiny and shared.** `hiero.mirror.rest.db.pool.maxConnections` defaults to **10** (`docs/configuration.md` line 556). Every request — including those for non-existent payers — acquires one of these 10 connections for the duration of the query.
3. **The URL pattern is fully open.** The OpenAPI path parameter regex (`^(0x)?[A-Fa-f0-9]{64}|(\d{1,10})\.(\d{1,10})\.(\d{1,10})-(\d{1,19})-(\d{1,9})$`, `openapi.yml` line 5872) accepts any shard/realm/num combination, including fabricated payer IDs that will never exist in the DB.

**Why the index does not fully mitigate this:** The composite index `transaction__payer_account_id_consensus_timestamp (payer_account_id, consensus_timestamp)` (`V2.5.1__update_indexes.sql` lines 5–6) means each individual query is fast (index seek → 0 rows). However, each query still:
- Acquires a connection from the 10-connection pool
- Executes a subquery (`select coalesce($4, 0)`)
- Traverses partition metadata (table is `PARTITION BY RANGE (consensus_timestamp)`, `V2.0.0__create_tables.sql` line 688)
- Returns the connection only after full round-trip

At high request rates the pool saturates, queuing or rejecting all subsequent queries across every endpoint.

### Impact Explanation

With the pool capped at 10 connections and `statementTimeout` at 20 seconds (`docs/configuration.md` line 557), an attacker sustaining ~50–100 concurrent HTTP requests can hold all 10 DB connections busy. Every other REST API endpoint (accounts, transactions, tokens, etc.) that needs a DB connection will queue or time out. Because the REST service is stateless and all replicas share the same DB pool configuration, the attack degrades the entire REST tier proportionally to the number of replicas targeted. This satisfies the ≥30% processing degradation threshold without requiring any privileged access or brute-force credential guessing.

### Likelihood Explanation

Preconditions: none. The endpoint is public, requires no authentication, and the URL format is documented in the OpenAPI spec. A single attacker with a modest botnet or even a single high-bandwidth host running an async HTTP client (e.g., `wrk`, `hey`, `ab`) can sustain the required request rate. The attack is repeatable, stateless, and leaves no persistent side effects, making it trivially scriptable.

### Recommendation

1. **Add application-level rate limiting** to the REST Express server (e.g., `express-rate-limit` or equivalent) scoped per source IP, applied globally before route handlers.
2. **Increase or dynamically scale `db.pool.maxConnections`** and add a per-endpoint connection-acquisition timeout shorter than `statementTimeout` so pool exhaustion fails fast rather than queuing indefinitely.
3. **Add a pre-query entity existence check** (or cache of known-valid payer IDs) before issuing the `transaction` table query, so fabricated payer IDs are rejected in-process without touching the DB.
4. **Deploy Traefik `rateLimit` middleware** (already present in `charts/hedera-mirror-rosetta/values.yaml` lines 157–161 for Rosetta) consistently for the REST API Ingress.

    if (utils.conflictingPathParam(req, 'transactionIdOrHash', 'logs')) {
      return;
    }

    utils.validateReq(req, acceptedSingleContractResultsParameters);

    // Extract hbar parameter (default: true)
    const convertToHbar = utils.parseHbarParam(req.query.hbar);

    let transactionDetails;

    const {transactionIdOrHash} = req.params;
    if (utils.isValidEthHash(transactionIdOrHash)) {
      const detailsByHash = await ContractService.getContractTransactionDetailsByHash(
        utils.parseHexStr(transactionIdOrHash)
      );
      transactionDetails = detailsByHash[0];
    } else {
      const transactionId = TransactionId.fromString(transactionIdOrHash);
      const nonce = getLastNonceParamValue(req.query);
      // Map the transactions id to a consensus timestamp
      const transactions = await TransactionService.getTransactionDetailsFromTransactionId(transactionId, nonce);

      if (transactions.length === 0) {
        throw new NotFoundError();
      }
      transactionDetails = transactions[0];
      // want to look up involved contract parties using the payer account id
      transactionDetails.entityId = transactionDetails.payerAccountId;
    }
