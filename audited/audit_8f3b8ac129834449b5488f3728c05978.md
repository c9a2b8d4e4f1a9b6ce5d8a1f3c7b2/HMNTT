### Title
Unauthenticated Retry Amplification via `fallbackRetry()` Causes Up to 10 Sequential Complex CTE Queries Per Request

### Summary
The `fallbackRetry()` function in `rest/service/fileDataService.js` executes up to `config.query.maxFileAttempts` (10) sequential complex CTE database queries per single API request when protobuf parsing of `file_data` rows fails. The public, unauthenticated `GET /api/v1/network/exchangerate?timestamp=<value>` endpoint allows any external user to supply a timestamp that resolves to a known-bad file_data row, triggering the full retry loop and multiplying DB load by up to 10x per request with no per-request query cap or rate limiting in the retry path.

### Finding Description

**Exact code path:**

`rest/service/fileDataService.js`, `fallbackRetry()`, lines 94–119:

```js
fallbackRetry = async (fileEntityId, filterQueries, resultConstructor) => {
  const whereQuery = filterQueries.whereQuery ?? [];
  const filters = {whereQuery};

  let attempts = 0;
  while (++attempts <= config.query.maxFileAttempts) {          // up to 10 iterations
    const row = await this.getLatestFileDataContents(fileEntityId, filters);  // CTE query each time
    try {
      return row === null ? null : new resultConstructor(row);  // ExchangeRate constructor
    } catch (error) {
      // on FileDecodeError: tighten timestamp bound and loop again
      filters.whereQuery = [
        ...whereQuery,
        { query: FileData.CONSENSUS_TIMESTAMP + utils.opsMap.lt,
          param: row.first_consensus_timestamp },
      ];
    }
  }
  return null;
};
```

Each call to `getLatestFileDataContents()` (line 100) executes `latestFileContentsQuery` — a multi-CTE query with `string_agg`, `max`, `min`, `JOIN`, `GROUP BY`, and `ORDER BY` (lines 19–42).

**Root cause:** The `ExchangeRate` constructor (lines 15–19 of `rest/model/exchangeRate.js`) calls `fromBinary(ExchangeRateSetSchema, exchangeRate.file_data)` and throws `FileDecodeError` on any parse failure. `fallbackRetry()` catches this and unconditionally issues another full CTE query, up to 10 times, with no per-request DB query budget, no circuit breaker, and no rate limit on the retry path itself.

**Failed assumption:** The design assumes bad protobuf rows are rare and transient. It does not account for an attacker who has identified a stable timestamp pointing to a persistently invalid row and can replay that request indefinitely.

**Exploit flow:**

1. Attacker observes (or waits for) a period when the exchange rate file has an invalid/partial protobuf entry — this legitimately occurs during `FileUpdate`/`FileAppend` operations, network upgrades, or partial writes. The retry mechanism itself exists because this happens in practice.
2. Attacker identifies a `consensus_timestamp` value `T` at which the `file_data` row for the exchange rate file contains non-parseable bytes.
3. Attacker sends: `GET /api/v1/network/exchangerate?timestamp=lte:T`
4. `fallbackRetry()` fires: each of the up to 10 iterations issues the full CTE query. If all 10 prior file versions at timestamps ≤ T are also invalid (e.g., during a botched multi-chunk file update), all 10 queries execute sequentially before returning `null`.
5. Attacker repeats this request in a tight loop from multiple clients.

### Impact Explanation

Each single HTTP request can cause up to 10 sequential complex CTE queries against the PostgreSQL database. With concurrent attackers or a high request rate, this multiplies DB CPU and I/O load by up to 10x compared to normal operation. The `latestFileContentsQuery` uses `string_agg` with ordering, self-joins, and aggregation — it is not a trivial query. Sustained amplification can degrade or exhaust DB connection pools and CPU, impacting all mirror node API consumers. The endpoint returns no authentication challenge and imposes no per-request query cap.

### Likelihood Explanation

The precondition — a timestamp resolving to an invalid protobuf row — is not hypothetical: the retry mechanism was built specifically because this occurs during normal Hedera network file operations. An attacker monitoring the mirror node during a file update window can identify such a timestamp. Once found, the timestamp remains exploitable for as long as the bad row exists in the DB (it is never deleted). The attack requires only standard HTTP tooling (`curl`, `ab`, `wrk`) and zero privileges.

### Recommendation

1. **Cap DB queries per request at the service layer:** Add a hard limit (e.g., 3) to `maxFileAttempts` and enforce it as a per-request budget, not just a loop bound.
2. **Rate-limit the `/network/exchangerate` endpoint** at the API gateway or middleware layer, independent of the retry logic.
3. **Short-circuit on `row === null`:** If `getLatestFileDataContents` returns `null`, exit immediately rather than continuing to loop (already done, but verify the CTE itself doesn't do extra work on empty results).
4. **Add a per-IP or global request rate limit** specifically for endpoints that invoke `fallbackRetry`.
