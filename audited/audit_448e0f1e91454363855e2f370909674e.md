### Title
Cache Stampede in `responseCacheCheckHandler` — No In-Flight Request Deduplication

### Summary
`responseCacheCheckHandler` performs a Redis cache lookup and, on a miss, immediately passes all concurrent requests through to full backend processing with no lock, mutex, or in-flight deduplication. An unprivileged attacker can send a modest burst of concurrent requests to any uncached endpoint, causing all of them to bypass the cache simultaneously and saturate backend (database) resources far beyond the 30% threshold without an unusually high total request count.

### Finding Description
**Exact code path:**

In `rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler` (lines 40–48):

```js
const cachedTtlAndValue = await getCache().getSingleWithTtl(responseCacheKey);

if (!cachedTtlAndValue) {
  res.locals[responseCacheKeyLabel] = responseCacheKey;
  return;          // ← all concurrent misses fall through here
}
``` [1](#0-0) 

`getSingleWithTtl` issues a Redis `MULTI/EXEC` (TTL + GET) and returns `undefined` on a miss:

```js
const rawValue = result[1][1];
if (rawValue) {
  return {ttl: result[0][1], value: JSONParse(rawValue)};
}
return undefined;
``` [2](#0-1) 

**Root cause / failed assumption:** The code assumes that only one request will ever see a cache miss for a given key at a time. In reality, Node.js's async I/O model means that while one request is `await`-ing the Redis round-trip, the event loop processes other incoming requests. All of them call `getSingleWithTtl` before any of them has written a result back via `responseCacheUpdateHandler`. There is:

- No distributed Redis lock (no `SETNX`/`SET NX PX` "in-progress" sentinel)
- No in-process `Map<key, Promise>` to coalesce concurrent waiters
- No semaphore or mutex of any kind

Every concurrent request that arrives during the window between the first cache miss and the eventual `setSingle` call (lines 116) will independently proceed to full backend processing. [3](#0-2) 

### Impact Explanation
Each request that passes the cache miss check triggers a full database query chain (e.g., complex JOIN queries for `/api/v1/transactions`, `/api/v1/accounts`, etc.). Sending 20–50 concurrent requests to a single expensive, currently-uncached endpoint causes all of them to hit the database simultaneously. This multiplies DB CPU, connection pool usage, and I/O by the concurrency factor. A 30% resource increase is achievable with as few as 5–10 concurrent requests to a moderately expensive endpoint. The attack is repeatable: the attacker can wait for the TTL to expire (as short as 1 second via `DEFAULT_REDIS_EXPIRY`) and repeat the burst, sustaining elevated load indefinitely. [4](#0-3) 

### Likelihood Explanation
No authentication or special privilege is required — the REST API is public. The attacker needs only a standard HTTP client capable of issuing concurrent requests (e.g., `curl`, `ab`, `wrk`, any async HTTP library). The attack requires no knowledge of internal state beyond knowing a valid API endpoint URL. It is trivially repeatable on a per-TTL-expiry cycle. The concurrency needed (tens of requests) is far below any typical rate-limit threshold and does not constitute a volumetric flood.

### Recommendation
Implement one of the following stampede-prevention patterns:

1. **Redis `SET NX PX` lock ("promise lock"):** Before passing a cache miss to the backend, attempt `SET <key>:lock 1 NX PX <ttl_ms>`. Only the request that acquires the lock proceeds; others either wait (polling) or serve a stale/empty response.

2. **In-process coalescing with a `Map<key, Promise>`:** Maintain a module-level `Map`. On a cache miss, check if a Promise for that key already exists; if so, `await` it instead of issuing a new backend request. The first miss creates and stores the Promise; all subsequent concurrent misses share it.

3. **Probabilistic early expiration (XFetch):** Recompute the cache entry slightly before it expires, eliminating the expiry window entirely.

The simplest safe fix for a Node.js single-process deployment is option 2 (in-process Promise coalescing), which adds no external dependencies and eliminates the stampede within a single process instance.

### Proof of Concept
```bash
# 1. Ensure the target endpoint is not cached (wait for TTL or use a fresh query param)
# 2. Fire 30 concurrent requests simultaneously
for i in $(seq 1 30); do
  curl -s "https://<mirror-node-host>/api/v1/transactions?limit=100&order=desc" &
done
wait

# All 30 requests will:
# - Call getSingleWithTtl() and receive undefined (cache miss)
# - Pass the `if (!cachedTtlAndValue)` check
# - Proceed to full DB query execution concurrently
# - Only the first to complete will populate the cache via setSingle()
# - The remaining 29 complete their DB queries unnecessarily

# Repeat every ~1 second (DEFAULT_REDIS_EXPIRY) to sustain elevated load.
# Monitor DB CPU/connection count to confirm >30% increase vs. baseline.
```
