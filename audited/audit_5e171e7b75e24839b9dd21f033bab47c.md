### Title
Cache Stampede and Negative-Result Cache Pollution via Non-Existent ContractId Flooding in `findRuntimeBytecode()`

### Summary
`ContractRepository.findRuntimeBytecode()` uses Spring's `@Cacheable` with a `CaffeineCacheManager` (not a `LoadingCache`), which provides no per-key locking. Multiple concurrent threads requesting the same non-existent `contractId` simultaneously all miss the cache and all issue the SQL query `select runtime_bytecode from contract where id = :contractId` in parallel. Additionally, the `unless = "#result == null"` condition is evaluated against the `Optional<byte[]>` object itself (not the unwrapped value), meaning `Optional.empty()` is not null and null entries ARE stored in the bounded cache (`maximumSize=1000`), enabling an attacker to pollute the entire cache with null entries for non-existent IDs, evicting legitimate contract bytecode entries.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18: [1](#0-0) 

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause 1 — No per-key locking (stampede):**

Spring's `@Cacheable` with `CaffeineCacheManager` performs a non-atomic check-then-act: `get(key)` → miss → `invokeMethod()` → `put(key, value)`. There is no per-key mutex. When N concurrent threads all request the same non-existent `contractId` before any thread has completed the DB round-trip, all N threads independently miss the cache and all issue the SQL query simultaneously. [2](#0-1) 

**Root cause 2 — `unless` condition does not exclude empty Optionals:**

Spring's `CacheAspectSupport` evaluates the `unless` SpEL expression with `#result` bound to the actual method return value — the `Optional<byte[]>` object itself, not the unwrapped value. `Optional.empty() == null` is `false`, so the condition `unless = "#result == null"` evaluates to `false` for every non-existent contractId lookup. Spring then calls `unwrapReturnValue()` → `ObjectUtils.unwrapOptional(Optional.empty())` → `null`, and stores `null` (as a `NullValue` sentinel) in the cache, since `CaffeineCacheManager.allowNullValues` defaults to `true`. [3](#0-2) 

**Root cause 3 — Bounded cache with small capacity:**

The contract cache is configured as `expireAfterAccess=1h,maximumSize=1000`. An attacker using ≥1001 unique non-existent contractIds fills the entire cache with null entries, evicting all legitimate contract bytecode entries. Every subsequent legitimate contract call then misses the cache and hits the DB. [4](#0-3) 

**Call path from public endpoint:**

`POST /api/v1/contracts/call` → `ContractController.call()` → `ContractExecutionService.processCall()` → EVM execution → `ContractBytecodeReadableKVState.readFromDataSource()` → `contractRepository.findRuntimeBytecode(entityId.getId())`. [5](#0-4) [6](#0-5) 

### Impact Explanation

**Cache pollution / eviction:** An attacker using 1001+ unique non-existent contractIds fills the 1000-entry cache entirely with null entries. All legitimate contract bytecode entries are evicted. Every subsequent EVM execution for a real contract must re-fetch bytecode from the DB, increasing per-request latency and DB load. With `expireAfterAccess=1h`, evicted entries stay evicted for the duration of the attack.

**Cache stampede:** For each unique non-existent contractId, the first burst of concurrent requests all miss the cache simultaneously and all issue the SQL query in parallel. Under the 500 RPS global rate limit, an attacker can sustain 500 concurrent DB queries per second for non-existent IDs, each consuming a DB connection from the pool for the duration of the query.

**Gossip path impact:** `findRuntimeBytecode()` is in the critical path for EVM bytecode resolution during contract call simulation. Sustained DB pressure delays or fails bytecode lookups needed for transaction processing.

### Likelihood Explanation

No authentication or account creation is required. The `/api/v1/contracts/call` endpoint is publicly accessible. The attacker only needs to send HTTP POST requests with arbitrary `to` addresses that do not correspond to real contracts. The global rate limit of 500 RPS is a single shared bucket — a single attacker can consume the entire budget. [7](#0-6) 

The attack is trivially repeatable with any HTTP client capable of concurrent requests (e.g., `wrk`, `ab`, `hey`).

### Recommendation

1. **Fix the `unless` condition** to correctly exclude empty Optionals:
   ```java
   unless = "#result == null || !#result.isPresent()"
   ```
   This ensures that lookups for non-existent contractIds are never stored in the cache, preventing cache pollution.

2. **Use a `LoadingCache` or per-key locking** to prevent stampedes. Replace `CaffeineCacheManager` with a `Caffeine.newBuilder().build(key -> ...)` `LoadingCache` for this repository, which serializes concurrent loads for the same key.

3. **Increase cache capacity** or add a separate negative-result cache with a short TTL (e.g., 30s) and a larger `maximumSize` to absorb flood traffic without evicting legitimate entries.

4. **Add per-IP rate limiting** in addition to the global 500 RPS bucket to prevent a single client from consuming the entire request budget.

### Proof of Concept

```bash
# Step 1: Fill the cache with 1001 unique non-existent contractIds
for i in $(seq 1 1001); do
  ADDR=$(printf "0x%040x" $i)
  curl -s -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$ADDR\",\"data\":\"0x\",\"gas\":50000}" &
done
wait

# Step 2: Verify legitimate contract bytecode is no longer cached
# (subsequent calls to real contracts now hit the DB instead of cache)
curl -X POST http://<host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"<real_contract_address>","data":"0x","gas":50000}'

# Step 3: Sustain the attack at 500 RPS with rotating non-existent IDs
# to keep the cache permanently polluted and maintain DB pressure
wrk -t10 -c500 -d60s -s rotate_nonexistent_ids.lua http://<host>/api/v1/contracts/call
```
