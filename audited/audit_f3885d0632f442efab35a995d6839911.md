### Title
`Optional.empty()` Cached Permanently Due to Broken `unless` Condition in `findRuntimeBytecode`, Enabling Cache Poisoning by Unprivileged Users

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"` to prevent caching of absent results. However, the method returns `Optional<byte[]>`, and `#result` in Spring SpEL refers to the `Optional` wrapper object itself — not its contents. Since `Optional.empty()` is a non-null object, the `unless` condition always evaluates to `false` for empty results, causing them to be cached. An unprivileged attacker can query a not-yet-imported contract ID, poison the cache with `Optional.empty()`, and prevent the mirror node from ever serving the correct bytecode for that contract for up to 1 hour (or indefinitely via repeated access).

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:**

Spring's `@Cacheable` `unless` SpEL expression evaluates `#result` against the actual return value of the method. Since the return type is `Optional<byte[]>`, `#result` is the `Optional` object. `Optional.empty()` is a singleton non-null object, so `#result == null` is always `false` for the empty case. The intended guard — "don't cache absent results" — is never triggered. The correct expression would be `unless = "#result == null || !#result.isPresent()"`.

**Cache configuration** (`CacheProperties.java`, line 22):

```
expireAfterAccess=1h,maximumSize=1000,recordStats
```

`expireAfterAccess` means the TTL resets on every read. An attacker who keeps polling the endpoint can hold the poisoned entry alive indefinitely.

**Exploit flow:**

1. Attacker identifies a contract ID `X` that is on-chain but not yet imported by the mirror node importer (there is always a propagation delay).
2. Attacker sends any unauthenticated web3 request (e.g., `eth_getCode`, `eth_call`) that routes through `ContractBytecodeReadableKVState.readFromDataSource()` → `contractRepository.findRuntimeBytecode(X)`.
3. DB has no row yet → returns `Optional.empty()`.
4. Spring evaluates `unless = "#result == null"` → `Optional.empty() != null` → `false` → **caches `Optional.empty()` for key `X`**.
5. Importer writes the contract's `runtime_bytecode` to the DB.
6. All subsequent calls to `findRuntimeBytecode(X)` return the cached `Optional.empty()` without hitting the DB.
7. `ContractBytecodeReadableKVState.readFromDataSource()` (line 42–46) calls `.orElse(null)` on the empty Optional and returns `null` to the EVM.
8. The EVM treats the contract as having no bytecode; all calls to it revert or fail.

**Why existing checks fail:**

The `unless` guard is the sole mechanism to prevent negative caching. It is structurally broken for `Optional`-returning methods. The existing test `findRuntimeBytecodeFailCall` (line 34–38 of `ContractRepositoryTest.java`) only asserts that an empty Optional is returned for a non-existent ID; it does not verify that a subsequent DB insert becomes visible through the cache. The test `findRuntimeBytecodeSuccessfulCall` (lines 17–31) only tests cache retention for a previously-found value, not the negative-cache poisoning scenario.

### Impact Explanation

Any contract deployed on the Hedera network can be rendered permanently inaccessible via the mirror node's web3 API for the duration of the cache entry's lifetime. Because the cache uses `expireAfterAccess`, an attacker who polls the endpoint every ~59 minutes can keep the poisoned entry alive indefinitely. All `eth_call`, `eth_getCode`, and contract execution requests for the targeted contract ID will fail or return empty bytecode, constituting a targeted denial-of-service against specific smart contracts. This directly maps to the "missing record export" impact class: the mirror node fails to reflect the actual on-chain state.

### Likelihood Explanation

The attack requires no credentials, no special permissions, and no on-chain capability. The only precondition is knowledge of a contract ID that is about to be (or has just been) imported — which is publicly observable by monitoring the Hedera network. The timing window (between on-chain finality and mirror node import) is well-known and typically seconds to minutes wide, making the race condition reliably exploitable. The attack is repeatable for any new contract deployment.

### Recommendation

Fix the `unless` condition to correctly exclude empty Optionals from caching:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

Additionally, audit all other `@Cacheable` annotations in the repository layer that return `Optional<T>` for the same pattern (e.g., `ContractStateRepository.findStorage` at line 20–21 has no `unless` condition at all, which also caches empty Optionals).

### Proof of Concept

1. Start the mirror node web3 service with a clean DB.
2. Note a contract ID `X` that exists on-chain but has not yet been imported.
3. Send: `curl -X POST http://<mirror-node>/api/v1/contracts/call -d '{"to":"<address-of-X>","data":"0x"}'` (or any `eth_getCode` call for `X`).
4. Observe: the call returns empty/zero bytecode (DB miss → `Optional.empty()` cached).
5. Wait for the importer to process the contract creation record and write `runtime_bytecode` to the DB.
6. Verify directly in the DB: `SELECT runtime_bytecode FROM contract WHERE id = X;` — bytecode is present.
7. Repeat the same web3 call from step 3.
8. Observe: the call still returns empty bytecode — the cached `Optional.empty()` is served, not the DB value.
