### Title
Unauthenticated HTTP Mirror-Node Bootstrap Allows MITM Injection of Attacker-Controlled Consensus Endpoints

### Summary
`buildNetworkFromMirrorNodes()` fetches the consensus-node topology from `cfg.mirrorRest` over a plain, unauthenticated HTTP connection with no TLS and no response-body validation. An attacker who can intercept that HTTP exchange can return a crafted `nodesEnvelope` whose `service_endpoints` point to attacker-controlled hosts while keeping legitimate `node_account_id` values, causing `hiero.ClientForNetwork(netmap)` to route every subsequent transaction to the attacker. The attacker can then selectively forward or silently drop transactions, manipulating the pinger's visible transaction history without stealing funds.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–132. [1](#0-0) 

The function issues a plain `http.Get` with no TLS, no certificate pinning, and no HMAC/signature verification on the response body: [1](#0-0) 

The decoded payload is consumed without any allowlist check on the returned `DomainName`/`IPAddressV4` fields: [2](#0-1) 

The only guards are:
- `node_account_id` must parse as a valid `hiero.AccountID` (trivially satisfied with `0.0.3`)
- `host` and `port` must be non-empty (trivially satisfied by the attacker)

The resulting `netmap` is passed directly to `hiero.ClientForNetwork`: [3](#0-2) 

**Root cause:** The bootstrap URL defaults to `http://mirror-rest:5551` (plain HTTP), and the response is trusted unconditionally. [4](#0-3) 

### Impact Explanation
Once the attacker's hosts are installed as the consensus-node map, every `CryptoTransfer` the pinger submits is sent to the attacker. The attacker can:
- **Drop** transactions silently → they never appear on-chain, creating gaps in the pinger's transfer history.
- **Delay and replay** transactions out of order → the visible sequence of confirmed transfers is reordered.
- **Observe** full transaction content (amounts, account IDs, memo fields) before forwarding.

Funds are not directly stolen because the operator's private key signs each transaction, but the integrity of the pinger's observable transaction history is fully compromised. Severity: **High** (integrity + confidentiality of monitoring data; the pinger is explicitly a health/liveness signal for the mirror-node infrastructure).

### Likelihood Explanation
**Preconditions:** The attacker must be on the network path between the pinger pod and the mirror-REST service. In typical Kubernetes or cloud deployments this is achievable via:
- ARP/NDP spoofing on the same L2 segment
- DNS poisoning of the `mirror-rest` hostname (no DNSSEC required since the connection is plain HTTP anyway)
- Compromised sidecar or ingress proxy in the same namespace
- BGP route injection for cloud-hosted deployments

No credentials or privileged API access are required. The attack is repeatable on every pinger restart because `buildNetworkFromMirrorNodes` is called once at startup and the result is never re-validated. [5](#0-4) 

### Recommendation
1. **Enforce TLS for `cfg.mirrorRest`**: reject any URL that does not use `https://` at config-load time.
2. **Pin or validate the mirror-node TLS certificate** (e.g., via a configurable CA bundle or certificate fingerprint).
3. **Allowlist consensus-node endpoints**: after fetching the topology, cross-check returned `IPAddressV4`/`DomainName` values against a statically configured set of known-good consensus-node addresses or CIDR ranges.
4. **Periodically re-fetch and diff** the topology; alert on unexpected endpoint changes rather than silently adopting them.

### Proof of Concept
1. Start a rogue HTTP server on `attacker.example.com:8080` that responds to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "attacker.example.com", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```
2. Intercept (DNS spoof or ARP spoof) the pinger's DNS resolution of `mirror-rest` to resolve to `attacker.example.com`.
3. Start the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
4. `buildNetworkFromMirrorNodes` fetches the injected response; `fetchMirrorNodeNetwork` builds `{"attacker.example.com:50211": AccountID{0,0,3}}`.
5. `hiero.ClientForNetwork(netmap)` routes all subsequent `CryptoTransfer` calls to `attacker.example.com:50211`.
6. The attacker's gRPC stub receives every signed transaction and can forward, drop, or delay them at will, producing an arbitrary visible transaction history on the pinger side.
