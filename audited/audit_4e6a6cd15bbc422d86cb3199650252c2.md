### Title
Unconditional Extra Database Query Per Page via Staking Reward Transfer Detection in GET /transactions

### Summary
Every `GET /transactions` page request that returns at least one transaction containing a transfer from the staking reward account (0.0.800) unconditionally triggers an additional SQL query to the `staking_reward_transfer` table via `getStakingRewardTransferList()`. Any unauthenticated user can reliably trigger this extra query by targeting timestamp ranges known to contain staking reward transfers, increasing effective database query load per page from 2 to 3 queries (a 50% increase in DB round-trips for the detail phase, or effectively doubling the secondary query count).

### Finding Description

**Exact code path:**

In `rest/transactions.js`, `doGetTransactions` (line 695) calls `cache.get(payerAndTimestamps, loader, keyMapper)` where the loader is:

```js
const loader = (keys) => getTransactionsDetails(keys, order).then((result) => formatTransactionRows(result.rows));
``` [1](#0-0) 

`formatTransactionRows` unconditionally calls `createStakingRewardTransferList(rows)`: [2](#0-1) 

`createStakingRewardTransferList` calls `getStakingRewardTimestamps` to filter transactions whose `crypto_transfer_list` contains `entity_id === stakingRewardAccount.getEncodedId()`, then passes those timestamps to `getStakingRewardTransferList`: [3](#0-2) 

`getStakingRewardTransferList` has only one guard — an early return if the timestamp list is empty — and otherwise fires a SQL query unconditionally: [4](#0-3) 

**Root cause:** The staking reward lookup is performed as a separate, unbatched-across-pages SQL query inside the `loader` callback. The `loader` is invoked for every cache-miss batch. The Redis cache is keyed per-transaction by `transaction:${timestamp}`, so an attacker using unique timestamp ranges (e.g., sliding `timestamp=lt:X` pagination) will always produce cache misses, causing the loader — and therefore the extra staking reward query — to fire on every page. [5](#0-4) 

**Failed assumption:** The design assumes the Redis cache will absorb repeated requests for the same transactions. However, the staking reward query result is not independently cached; it is embedded inside the loader result. An attacker paginating forward with unique `timestamp` cursors bypasses the cache entirely on every page. [6](#0-5) 

### Impact Explanation

For each `GET /transactions` page request that (a) has cache misses and (b) contains at least one transaction with a staking reward account transfer, the database receives 3 queries instead of 2:
1. `getTransactionTimestamps` — always fires
2. `getTransactionsDetails` — fires on cache miss
3. `getStakingRewardTransferList` — fires on cache miss when staking reward transfers are present

On Hedera mainnet, staking reward transfers (from account 0.0.800) are extremely common — they occur in virtually every epoch for any account with staked HBAR. An attacker can trivially identify time ranges dense with staking reward transfers and paginate through them with unique `timestamp=lt:X` cursors, ensuring cache misses and triggering the extra query on every page. This increases DB query throughput by 50% per affected page request and can be sustained indefinitely by any unauthenticated client.

### Likelihood Explanation

- **No authentication required.** `GET /transactions` is a public, unauthenticated endpoint.
- **Staking reward transfers are ubiquitous** on mainnet; no special knowledge is needed to find them.
- **Cache bypass is trivial**: using the `links.next` pagination cursor returned in each response naturally advances the timestamp window, guaranteeing cache misses for each new page.
- **Repeatable and automatable**: a simple script that follows `links.next` indefinitely will continuously trigger the extra query.

### Recommendation

1. **Batch the staking reward lookup into the main query** using a `LEFT JOIN` or CTE against `staking_reward_transfer` in `getTransactionsDetails`, eliminating the second round-trip entirely.
2. Alternatively, **cache the staking reward query result independently** in Redis, keyed by the set of consensus timestamps, so repeated requests for the same page do not re-query the DB.
3. As a short-term mitigation, apply per-IP or per-client rate limiting specifically to the `GET /transactions` endpoint in the REST layer (currently no rate limiting exists for this endpoint in the Node.js REST service, unlike the web3 service which has `ThrottleConfiguration`).

### Proof of Concept

```bash
# 1. Fetch first page of transactions
RESP=$(curl -s "https://<mirror-node>/api/v1/transactions?limit=25&order=desc")

# 2. Extract the next cursor
NEXT=$(echo $RESP | jq -r '.links.next')

# 3. Loop, following pagination — each page with staking reward transfers
#    fires an extra SQL query to staking_reward_transfer table
while [ "$NEXT" != "null" ]; do
  curl -s "https://<mirror-node>$NEXT" > /dev/null
  NEXT=$(curl -s "https://<mirror-node>$NEXT" | jq -r '.links.next')
done
```

Each iteration that returns transactions containing a transfer from `entity_id = stakingRewardAccount` causes `getStakingRewardTransferList()` to execute one additional `SELECT ... FROM staking_reward_transfer WHERE consensus_timestamp IN (...)` query against the database, observable via DB query logs or metrics.
