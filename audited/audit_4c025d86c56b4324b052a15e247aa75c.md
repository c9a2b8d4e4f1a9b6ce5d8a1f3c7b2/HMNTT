### Title
Silent HTTP 200 with Empty Logs on Nonexistent `transaction_hash` in `getContractLogs()`

### Summary
When a caller supplies a `transaction.hash` query parameter to `GET /api/v1/contracts/results/logs` that does not exist in the database, `extractContractLogsMultiUnionQuery()` returns `null`, and the `getContractLogs()` handler silently returns HTTP 200 with `{"logs": [], "links": {"next": null}}`. This response is byte-for-byte identical to the response for a valid transaction that emitted no logs, making it impossible for clients to distinguish the two cases.

### Finding Description
**Code path:**

`rest/controllers/contractController.js`, `getContractLogs()` (lines 820–848):

```js
// Lines 828–833: default response pre-set BEFORE hash lookup
res.locals[responseDataLabel] = {
  logs: [],
  links: { next: null },
};

// Line 835: hash lookup delegated to extractContractLogsMultiUnionQuery
const query = await this.extractContractLogsMultiUnionQuery(filters);
if (query === null) {
  return;   // ← silent early return, HTTP 200 with empty logs
}
```

`extractContractLogsMultiUnionQuery()` (lines 669–674):

```js
if (transactionHash !== undefined) {
  const rows = await getTransactionHash(transactionHash, {order, timestampFilters});
  if (rows.length === 0) {
    return null;   // ← hash not found → null propagated to handler
  }
  ...
}
```

`getTransactionHash()` in `rest/transactionHash.js` (lines 21–37) performs a DB lookup via `get_transaction_info_by_hash($1)` and returns an empty array when no row matches.

**Root cause:** The handler pre-populates `res.locals[responseDataLabel]` with an empty-logs payload before the hash lookup occurs. When the lookup finds nothing and `extractContractLogsMultiUnionQuery` returns `null`, the handler exits via `return` at line 837 without overwriting that pre-set payload and without setting any error status. The HTTP response is 200 OK with `{"logs":[],"links":{"next":null}}`.

**Failed assumption:** The design assumes callers can infer "hash not found" from an empty log list, but the same empty list is the correct response for a real transaction that emitted no logs.

**Contrast with the correct pattern** in `rest/transactions.js` (lines 772–775):

```js
const rows = await getTransactionHash(Buffer.from(transactionIdOrHash, encoding));
if (rows.length === 0) {
  throw new NotFoundError();   // ← HTTP 404
}
```

The transactions endpoint throws `NotFoundError` (HTTP 404) for the same condition; the contract-logs endpoint does not.

### Impact Explanation
Any client that is event-driven — an EVM-compatible bridge, a dApp polling for Transfer/Approval events, a monitoring system asserting that a specific transaction triggered a specific log — will receive a semantically incorrect "no logs emitted" answer for a hash that never existed. A bridge that gates fund release on observing a log event could be tricked into believing the triggering transaction produced no events, stalling or misdirecting its state machine. Because the HTTP status is 200 and the JSON structure is valid, automated clients have no signal to retry or escalate. Severity is medium: no direct fund extraction from the mirror node itself, but the false negative can corrupt downstream protocol logic that relies on log presence as a precondition.

### Likelihood Explanation
No authentication or special privilege is required. Any external caller can craft a syntactically valid but nonexistent 32-byte hex hash (e.g., `0x` + 64 hex chars not in the DB) and issue a single HTTP GET. The condition is trivially reproducible and fully deterministic. Any automated client that does not separately verify the transaction's existence before trusting an empty-log response is affected.

### Recommendation
In `getContractLogs()`, after `extractContractLogsMultiUnionQuery` returns `null` specifically because a `transaction_hash` filter was present and unmatched, throw a `NotFoundError` (HTTP 404) instead of returning silently:

```js
const query = await this.extractContractLogsMultiUnionQuery(filters);
if (query === null) {
  // A transaction_hash filter was supplied but matched no record
  throw new NotFoundError();
}
```

Alternatively, have `extractContractLogsMultiUnionQuery` distinguish "hash not found" (throw) from other early-exit conditions (return null), mirroring the pattern already used in `rest/transactions.js` lines 773–775.

```javascript
const getTransactionHash = async (hash, {order = orderFilterValues.ASC, timestampFilters = []} = {}) => {
  const normalized = normalizeTransactionHash(hash);
  const params = [normalized];

  const timestampConditions = [];
  for (const filter of timestampFilters) {
    timestampConditions.push(`${TransactionHash.CONSENSUS_TIMESTAMP} ${filter.operator} $${params.push(filter.value)}`);
  }

  const query = `${mainQuery}
    ${timestampConditions.length !== 0 ? `where ${timestampConditions.join(' and ')}` : ''}
    ${orderClause} ${order}
    ${limitClause}`;

  const {rows} = await pool.queryQuietly(query, params);
  return normalized !== hash ? rows.filter((row) => row.hash.equals(hash)) : rows;
```
