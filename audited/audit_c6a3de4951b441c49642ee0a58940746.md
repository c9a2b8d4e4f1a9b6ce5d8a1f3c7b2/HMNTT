### Title
Unauthenticated 3x Database Query Amplification in `getTopic()` Enables Resource Exhaustion and Stale Transaction Data

### Summary
The `getTopic()` handler in `TopicController` issues three independent, sequential database queries per single unauthenticated HTTP GET request, with no rate limiting present anywhere in the `rest-java` module. An attacker can flood `GET /api/v1/topics/{id}` to generate database load at 3× the request rate, exhausting the shared PostgreSQL connection pool and starving the mirror node importer of write capacity, causing it to serve stale transaction data.

### Finding Description

**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 32–37:

```java
@GetMapping(value = "/{id}")
Topic getTopic(@PathVariable EntityIdNumParameter id) {
    var topic = topicService.findById(id.id());      // DB query 1
    var entity = entityService.findById(id.id());    // DB query 2
    var customFee = customFeeService.findById(id.id()); // DB query 3
    return topicMapper.map(customFee, entity, topic);
}
```

Each service delegates directly to a JPA/JOOQ repository `findById()` with no caching:
- `TopicServiceImpl.findById()` → `topicRepository.findById()`
- `EntityServiceImpl.findById()` → `entityRepository.findById()`
- `CustomFeeServiceImpl.findById()` → `customFeeRepository.findById()`

The three calls are sequential (not parallel), so each HTTP request occupies a server thread and holds database connection resources for the full duration of all three round-trips.

**No rate limiting in `rest-java`:**

The only servlet filters registered in the `rest-java` module are `LoggingFilter` (request logging) and `MetricsFilter` (byte-count metrics). `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter. `WebMvcConfiguration` registers only argument resolvers and formatters. There is no `ThrottleFilter`, no `Bucket4j` bean, and no Spring Security rate-limiting interceptor anywhere in the `rest-java` module. The `ThrottleConfiguration` / `ThrottleManagerImpl` that does exist lives exclusively in the `web3` module and is not applied here.

**Root cause / failed assumption:**

The design assumes that an external layer (load balancer, API gateway, reverse proxy) will enforce per-IP or global rate limits before requests reach the application. No such enforcement is implemented or enforced at the application layer, so the assumption fails whenever the service is exposed directly or the upstream proxy is misconfigured.

### Impact Explanation

PostgreSQL uses a bounded connection pool (typically 10–100 connections by default in Spring Boot / HikariCP). Each `getTopic()` request holds a connection for three sequential queries. At high concurrency, the pool saturates. The mirror node importer (a separate Spring Boot process) writes ingested Hedera transaction records to the same PostgreSQL instance. When the connection pool is exhausted by read traffic from the REST API, the importer's `INSERT`/`UPDATE` operations queue behind the read load, increasing ingestion latency. During this window, the REST API continues serving data from the last committed state, which is increasingly stale. Clients querying transaction status receive outdated results — a direct violation of the mirror node's core correctness guarantee of reflecting the current ledger state.

Severity: **Medium–High**. The 3× amplification factor means an attacker achieves the same DB saturation with one-third the request volume compared to a single-query endpoint, lowering the attack cost significantly.

### Likelihood Explanation

- **Preconditions**: None. The endpoint is unauthenticated and publicly reachable.
- **Attacker capability**: Any user with network access and a basic HTTP flood tool (e.g., `wrk`, `ab`, `hey`). No credentials, tokens, or special knowledge required.
- **Repeatability**: Fully repeatable and automatable. The attacker can target any valid or invalid topic ID (invalid IDs still consume two DB queries before the first `EntityNotFoundException` short-circuits).
- **Detection evasion**: Because each individual request is legitimate and returns HTTP 200, WAF signature-based rules will not trigger. Only volumetric anomaly detection would catch it.

### Recommendation

1. **Immediate**: Add application-layer rate limiting to the `rest-java` module. Port the existing `Bucket4j`-based `ThrottleConfiguration` from `web3` into `rest-java`, or add a `HandlerInterceptor` / servlet `Filter` that enforces a per-IP and global requests-per-second cap on all `/api/v1/**` endpoints.
2. **Short-term**: Parallelize the three service calls in `getTopic()` using `CompletableFuture.allOf()` or virtual threads so that all three DB queries execute concurrently, reducing wall-clock connection hold time from `3×query_latency` to `max(query_latency)`.
3. **Short-term**: Add a read-replica datasource for REST API queries so that read traffic cannot exhaust the write-path connection pool used by the importer.
4. **Long-term**: Introduce a response cache (e.g., Caffeine with a short TTL) at the service layer for `findById()` calls, since topic metadata changes infrequently.

### Proof of Concept

```bash
# Flood the endpoint with 500 concurrent connections, 10,000 total requests
# Each request triggers 3 DB queries = 30,000 DB round-trips
wrk -t 50 -c 500 -d 30s \
  http://<mirror-node-host>/api/v1/topics/0.0.1234

# Simultaneously monitor importer lag:
# If ingestion falls behind, the following two calls will return different
# consensus_timestamp values separated by a growing delta:
curl http://<mirror-node-host>/api/v1/transactions?limit=1&order=desc
# Compare returned consensus_timestamp against the actual Hedera network time.
# A growing gap (>5s) confirms the importer is being starved.
```

No authentication, no special headers, and no privileged access are required.
