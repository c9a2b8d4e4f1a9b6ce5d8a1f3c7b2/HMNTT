### Title
Cache Stampede in `findRuntimeBytecode()` Due to Missing `sync=true` on `@Cacheable`

### Summary
`ContractRepository.findRuntimeBytecode()` uses Spring's `@Cacheable` without `sync=true`, meaning concurrent threads that all arrive before the first DB result populates the Caffeine cache will each independently execute the native SQL query against the database. An unprivileged external user can trivially trigger this by sending many concurrent requests for the same uncached `contractId`, multiplying database load proportionally to the number of concurrent callers.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

The cache manager is configured in `EvmConfiguration.java` lines 67–73:

```java
@Bean(CACHE_MANAGER_CONTRACT)
CacheManager cacheManagerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getContract());
    return caffeineCacheManager;
}
```

**Root cause:** Spring's `@Cacheable` without `sync=true` performs a non-atomic check-then-act: each thread independently checks the cache, finds a miss, and proceeds to invoke the underlying method. There is no coalescing or locking. Caffeine supports single-flight loading natively (via `Cache.get(key, mappingFunction)` or `AsyncLoadingCache`), but Spring only activates this path when `sync=true` is set on the annotation. Without it, N concurrent threads all missing the cache for the same key will all issue N independent SQL queries.

**Exploit flow:**
1. Attacker identifies any valid (or even invalid) `contractId` not yet in the cache (e.g., a newly deployed contract, or any contract after a cache eviction).
2. Attacker sends N concurrent HTTP requests to the web3 JSON-RPC endpoint (e.g., `eth_getCode` or `eth_call`) that internally invoke `findRuntimeBytecode(contractId)`.
3. All N threads simultaneously check the Caffeine cache, all find a miss, and all execute: `select runtime_bytecode from contract where id = :contractId`.
4. The database receives N identical queries for the same row simultaneously.
5. The first thread to complete populates the cache; all subsequent requests are served from cache — but the damage (N DB hits) has already occurred.

**Why existing checks fail:** The `@Cacheable` annotation is present and correctly configured for normal single-threaded use. The `unless = "#result == null"` guard only prevents caching of Java `null` (not `Optional.empty()`), so it does not worsen the stampede for missing contracts. However, the absence of `sync=true` means the cache provides zero protection against concurrent cache misses for the same key.

### Impact Explanation
A single attacker with no credentials can open N parallel HTTP connections (e.g., N=100–500, trivially achievable with `curl --parallel` or any HTTP load tool) and target the same uncached `contractId`. Each connection causes an independent `SELECT runtime_bytecode FROM contract WHERE id = ?` query. Since `runtime_bytecode` can be tens of kilobytes of contract bytecode, each query is non-trivial in I/O terms. With N=100 concurrent requests, the database receives 100× the expected query load for that key in a single burst. Repeated across multiple contractIds or after cache evictions, this can sustain database CPU and I/O above the 30% threshold defined in scope. No authentication, no special role, no internal access is required.

### Likelihood Explanation
The attack requires only the ability to send HTTP requests to the public web3 JSON-RPC endpoint, which is the intended public interface. Any external user can do this. The attacker needs no knowledge of internal state beyond a valid (or recently-deployed) contract address, which is publicly observable on-chain. The attack is repeatable: cache entries expire (per `cacheProperties.getContract()` TTL), so the stampede window recurs on every eviction cycle. Tooling to send parallel HTTP requests is universally available.

### Recommendation
Add `sync=true` to the `@Cacheable` annotation:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null",
    sync = true
)
```

With `sync=true`, Spring delegates to Caffeine's `Cache.get(key, mappingFunction)`, which guarantees that only one thread executes the loader function per key while all other threads block and wait for the result. This eliminates the stampede entirely. Verify that `CaffeineCacheManager` is configured to support synchronous loading (it does by default when `sync=true` is used with Spring Cache).

### Proof of Concept
```bash
# Step 1: Identify a contractId not yet in cache (e.g., after service restart or TTL expiry)
CONTRACT_ID="0x<valid_contract_address>"

# Step 2: Send 200 concurrent eth_getCode requests for the same contract
for i in $(seq 1 200); do
  curl -s -X POST http://<mirror-node-web3-host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$CONTRACT_ID\",\"data\":\"0x\"}" &
done
wait

# Step 3: Observe database query logs / pg_stat_activity showing 200 simultaneous
# identical queries: SELECT runtime_bytecode FROM contract WHERE id = <id>
# Expected: 200 DB hits instead of 1, confirming the stampede.
```

Database-side verification: query `pg_stat_activity` or enable `log_min_duration_statement=0` in PostgreSQL during the test and count duplicate identical queries arriving within the same millisecond window.
