### Title
Silent Upper-Bound Overwrite in `Bound` Constructor Allows Unprivileged Users to Expand Query Result Sets via Duplicate `lte` Operators

### Summary
`TokenAirdropsController.processRequest()` constructs `Bound` objects from user-supplied query parameters but never calls `verifySingleOccurrence()` on them. When a user submits two upper-bound operators (e.g., `receiver.id=lte:100&receiver.id=lte:200`), the `Bound` constructor silently overwrites `upper` with the last value, causing the query to use the wider bound (200) while the user's visible parameters suggest the narrower bound (100). The `NftAllowanceServiceImpl` path correctly calls `verifySingleOccurrence()`, confirming the fix is known but was not applied to the token airdrop path.

### Finding Description

**Root cause — `Bound.java` constructor, lines 47–54:**

```java
for (var param : params) {
    if (param.hasLowerBound()) {
        lower = param;
    } else if (param.hasUpperBound()) {
        upper = param;   // last-write-wins; no duplicate check
    }
    cardinality.merge(param.operator(), 1, Math::addExact);
}
```

Both `lte:100` and `lte:200` satisfy `hasUpperBound()`. The loop processes them in array order: `upper` is first set to `lte:100`, then silently overwritten with `lte:200`. `cardinality[LTE]` becomes 2, but nothing reads that value in this path.

**Missing guard — `Bound.java`, lines 155–159:**

```java
public void verifySingleOccurrence() {
    verifySingleOccurrence(RangeOperator.EQ);
    verifySingleOccurrence(RangeOperator.GT, RangeOperator.GTE);
    verifySingleOccurrence(RangeOperator.LT, RangeOperator.LTE);  // would throw here
}
```

This method exists and would throw `IllegalArgumentException` when `cardinality[LT+LTE] > 1`, but it is **never called** in the token airdrop path.

**Contrast — `NftAllowanceServiceImpl.java`, lines 56–60:**

```java
private void verifyRangeId(Bound ids) {
    ids.verifyUnsupported(RangeOperator.NE);
    ids.verifySingleOccurrence();   // called here
    ids.verifyEqualOrRange();
}
```

`NftAllowanceServiceImpl` calls `verifySingleOccurrence()` before querying. `TokenAirdropServiceImpl` does not — it goes directly to `repository.findAll(request, id)`.

**Controller path — `TokenAirdropsController.java`, lines 97–108:**

```java
var entityIdsBound = new Bound(entityIds, true, ACCOUNT_ID, type.getPrimaryField());
var request = TokenAirdropRequest.builder()
        .accountId(id)
        .entityIds(entityIdsBound)
        ...
        .build();
var response = service.getAirdrops(request);
```

No validation is performed on `entityIdsBound` before it is passed to the service and then to the repository query builder.

**`@Size(max = 2)` annotation** (line 71) only limits the array to 2 elements; it does not prevent two elements from carrying the same operator class.

### Impact Explanation
An unprivileged external user can expand the upper bound of a `receiver.id` (or `sender.id`) filter beyond what their visible query parameters declare. For example, submitting `lte:100` followed by `lte:200` causes the server to return all airdrop records with receiver IDs up to 200, while the user's first parameter suggests the ceiling is 100. This is unauthorized data over-exposure: records belonging to accounts 101–200 are returned without the user explicitly requesting them, violating the intended access boundary of the query filter model.

### Likelihood Explanation
No authentication or privilege is required. The endpoint is publicly accessible. The exploit requires only crafting a standard HTTP GET request with two query parameters of the same name — a trivial operation with any HTTP client, browser, or `curl`. The attacker controls parameter ordering, so placing the wider bound second is deterministic. The attack is silent (no error, no log anomaly beyond a slightly larger result set) and fully repeatable.

### Recommendation
Call `verifySingleOccurrence()` on every `Bound` object constructed from user input in `TokenAirdropsController.processRequest()`, mirroring the pattern already used in `NftAllowanceServiceImpl.verifyRangeId()`:

```java
var entityIdsBound = new Bound(entityIds, true, ACCOUNT_ID, type.getPrimaryField());
entityIdsBound.verifySingleOccurrence();
entityIdsBound.verifyEqualOrRange();

var serialBound = new Bound(serialNumbers, false, SERIAL_NUMBER, TOKEN_AIRDROP.SERIAL_NUMBER);
serialBound.verifySingleOccurrence();

var tokenBound = new Bound(tokenIds, false, TOKEN_ID, TOKEN_AIRDROP.TOKEN_ID);
tokenBound.verifySingleOccurrence();
```

Alternatively, centralise this validation in `TokenAirdropServiceImpl.getAirdrops()` so it cannot be bypassed by any future controller.

### Proof of Concept

**Precondition:** The mirror node has airdrop records for receiver accounts 1–200 for a given sender account `0.0.500`.

**Step 1 — Baseline (correct behaviour):**
```
GET /api/v1/accounts/0.0.500/airdrops/outstanding?receiver.id=lte:100
```
Returns records for receiver IDs 1–100 only.

**Step 2 — Exploit (wider bound silently applied):**
```
GET /api/v1/accounts/0.0.500/airdrops/outstanding?receiver.id=lte:100&receiver.id=lte:200
```
Spring binds `["lte:100","lte:200"]` into `EntityIdRangeParameter[]` (allowed by `@Size(max=2)`). The `Bound` constructor sets `upper = lte:100`, then overwrites it with `upper = lte:200`. No exception is thrown. The generated SQL uses `receiver_account_id <= 200`, returning records for IDs 1–200.

**Observed result:** 200 records returned instead of 100, with no error or indication to the caller that the second `lte` parameter overrode the first.
