### Title
Retry Loop Re-executes Committed Transfers When `GetReceipt` Is Delayed or Dropped

### Summary
In `pinger/transfer.go`, `submitWithRetry()` creates a **new** `TransferTransaction` on every loop iteration. When `Execute(client)` succeeds (the transaction is accepted and committed by the network) but `resp.GetReceipt(client)` fails due to a delayed or dropped response, the code treats the attempt as failed and retries with a fresh transaction. This causes up to `maxRetries+1` independent, committed transfers per single tick — each draining `cfg.amountTinybar` from the operator account.

### Finding Description
**Exact location**: `pinger/transfer.go`, `submitWithRetry()`, lines 23–57.

```go
for i := 1; i <= attempts; i++ {          // attempts = maxRetries + 1
    // NEW transaction created every iteration — new tx ID, no idempotency
    cryptoTransfer := hiero.NewTransferTransaction().
        AddHbarTransfer(..., -cfg.amountTinybar).
        AddHbarTransfer(toID, cfg.amountTinybar)

    resp, err := cryptoTransfer.Execute(client)   // funds committed here
    if err == nil {
        receipt, rerr := resp.GetReceipt(client)  // attacker drops this response
        if rerr == nil {
            return nil                            // only exit on receipt success
        }
        err = rerr                                // receipt failure → retry
    }
    // loop continues → new Execute() → new committed transfer
}
```

**Root cause**: The code conflates two distinct failure modes:
1. `Execute` fails → transaction was never submitted; safe to retry.
2. `Execute` succeeds but `GetReceipt` fails → transaction **was** committed to the ledger; retrying creates a second independent transaction.

Because a new `hiero.NewTransferTransaction()` is constructed each iteration, each retry carries a unique transaction ID. The Hedera network has no way to deduplicate them — they are genuinely separate transactions.

**Failed assumption**: The code assumes that a `GetReceipt` error implies the transfer did not occur. In reality, `GetReceipt` is a polling call against the mirror/consensus node; its failure is entirely independent of whether the transaction reached finality.

**Existing checks and why they are insufficient**:
- `ctx.Err()` (line 24): only guards against context cancellation, not against a previously committed transaction.
- No idempotency key, no transaction-ID reuse, no "did the last tx succeed on-chain?" check before retrying.

### Impact Explanation
Per tick, instead of transferring `cfg.amountTinybar` once, the operator account can be drained of `cfg.amountTinybar × (maxRetries + 1)` tinybar. With default or large `maxRetries` values and a high-frequency tick interval, this multiplies fund loss proportionally. The operator has no in-band signal that extra transfers occurred until they inspect the ledger directly.

### Likelihood Explanation
The attacker needs only the ability to delay or drop TCP/HTTP responses on the path between the pinger process and the Hedera receipt endpoint (mirror node or consensus node gRPC). This is achievable by:
- A network-level MITM (same LAN, compromised router, BGP hijack).
- Firewall/iptables rules on a shared host.
- Targeted packet-drop via a compromised upstream ISP.

No Hedera account, no private key, and no application-layer authentication is required. The attack is repeatable every tick for as long as the attacker can sustain the receipt-drop condition.

### Recommendation
1. **Reuse the same transaction ID across receipt-poll retries** rather than creating a new `TransferTransaction` each iteration. Only create a new transaction (new ID) if `Execute` itself fails.
2. **Separate retry scopes**: retry `GetReceipt` in an inner loop without re-calling `Execute`; only fall back to a new `Execute` if the receipt definitively indicates the transaction was not found (e.g., `UNKNOWN` / `TRANSACTION_EXPIRED` status after the valid window).
3. **Check receipt status before retrying**: if `GetReceipt` returns a receipt with a non-success status code (rather than a network error), that is the correct signal to retry with a new transaction.

### Proof of Concept
**Preconditions**:
- Pinger is running with `maxRetries ≥ 1`.
- Attacker can drop/delay responses to the receipt-polling endpoint (mirror node REST or consensus node gRPC) from the pinger host.

**Steps**:
1. Attacker installs a firewall rule (or acts as MITM) that drops all inbound packets from the receipt endpoint to the pinger process.
2. Pinger tick fires → `submitWithRetry` is called.
3. Iteration 1: `Execute(client)` succeeds → Transfer #1 committed on-chain. `GetReceipt` blocks and times out → `rerr != nil` → loop continues.
4. Iteration 2: new `TransferTransaction` created → `Execute(client)` succeeds → Transfer #2 committed on-chain. `GetReceipt` times out again → loop continues.
5. … repeated until `i == attempts`.
6. Result: `maxRetries + 1` separate committed transfers, each moving `cfg.amountTinybar` out of the operator account, for a total loss of `cfg.amountTinybar × (maxRetries + 1)` per tick.
7. Attacker removes the firewall rule; normal operation resumes, but funds are already gone.
