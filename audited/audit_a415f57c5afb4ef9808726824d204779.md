### Title
Unauthenticated Plain-HTTP Mirror Node Response Enables Cache Poisoning to Permanently Redirect Fund Transfers

### Summary
`buildNetworkFromMirrorNodes()` fetches consensus node addresses over plain HTTP with no integrity verification (no HMAC, no signature, no certificate pinning) and sets no `Cache-Control: no-cache` request headers. An attacker who can poison an intermediate caching proxy or CDN can inject a malicious `nodesEnvelope` that is served on every pinger startup, permanently routing all fund transfers to attacker-controlled gRPC endpoints for the lifetime of the process.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 79–98:

```go
req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
// No Cache-Control: no-cache header added
resp, err := httpClient.Do(req)
// Only check: HTTP status 200-299
if resp.StatusCode < 200 || resp.StatusCode >= 300 { ... }
// Response body decoded directly, no integrity check
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }
```

`pinger/sdk_client.go`, lines 18–22 — called **once at startup, never refreshed**:

```go
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
client = hiero.ClientForNetwork(netmap)
```

**Root cause:** The `http.Client` is constructed with no TLS configuration and no certificate pinning (`pinger/mirror_node_client.go` line 46). The documented example URL is `http://mirror-rest:5551` (plain HTTP, `config.go` line 37). No `Cache-Control: no-cache` or `Pragma: no-cache` headers are added to the outbound request, so any intermediate caching layer is free to store and replay the response. The decoded `nodesEnvelope` is accepted without any cryptographic integrity check.

**Failed assumption:** The code assumes the network path between the pinger and the mirror node REST endpoint is trusted and that no caching layer exists between them.

**Exploit flow:**

1. Attacker identifies or controls an intermediate HTTP caching proxy or CDN in front of the mirror node REST endpoint (or exploits a web cache poisoning vector against an existing CDN).
2. Attacker injects a crafted HTTP response for `GET /api/v1/network/nodes` containing a `nodesEnvelope` with `service_endpoints` pointing to attacker-controlled gRPC servers.
3. The cache stores this response (no `Cache-Control: no-store` or `no-cache` on the request prevents this).
4. On every pinger startup, `buildNetworkFromMirrorNodes()` fetches the poisoned cached response.
5. `hiero.ClientForNetwork(netmap)` is initialized with attacker-controlled node addresses.
6. All subsequent `submitWithRetry()` calls in `transfer.go` (lines 29–33) route `CryptoTransfer` transactions to attacker-controlled nodes.

**Why existing checks fail:**

- Lines 90–93 only validate HTTP status code — a cached poisoned response returns `200 OK`.
- Lines 103–124 only validate that `NodeAccountID` parses as a valid Hedera account ID format and that `Port != 0` — attacker-controlled values trivially satisfy both.
- There is no allowlist of known-good node addresses or public keys to compare against.

### Impact Explanation

All fund transfers executed by the pinger are routed to attacker-controlled gRPC endpoints. The attacker can drop transactions (causing silent transfer failures), replay them, or collect transaction metadata. Because the network map is built once at startup and never refreshed (`sdk_client.go` line 18), the poisoned state persists for the entire process lifetime — every tick of the transfer loop (`main.go` lines 57–70) is affected. This constitutes permanent misdirection of funds until the pinger is manually restarted and the cache is purged.

### Likelihood Explanation

The `network=other` path is the only one that calls `buildNetworkFromMirrorNodes()`; the named networks (`testnet`, `mainnet`, `previewnet`) use hardcoded SDK addresses. For `network=other` deployments that route the mirror REST URL through a CDN or reverse-proxy cache (a common production pattern), an attacker with web cache poisoning capability — which does not require network-level access — can execute this attack. The plain-HTTP example URL (`http://mirror-rest:5551`) makes MITM-based cache injection trivially feasible on any shared or cloud network segment without requiring privileged access.

### Recommendation

1. **Enforce HTTPS with certificate verification** — reject any `mirrorRest` URL that does not use `https://` and do not disable TLS verification in the `http.Client`.
2. **Set `Cache-Control: no-store` on every request** — add the header in `fetchMirrorNodeNetwork()` before `httpClient.Do(req)` to prevent intermediate caches from storing the response.
3. **Validate response integrity** — compare returned node account IDs and endpoints against a locally configured allowlist or a signed manifest; reject any response that introduces unknown node IDs.
4. **Periodically refresh the network map** — instead of building it once at startup, refresh it on a schedule so a poisoned cache entry does not persist indefinitely.
5. **Pin the mirror node TLS certificate or CA** — use a custom `tls.Config` with a pinned certificate pool in the `http.Transport`.

# port 50211 != 0 → address "attacker.example.com:50211" is added to netmap.

# Step 3: All CryptoTransfer transactions are now submitted to

# The attacker drops or inspects all transactions.
