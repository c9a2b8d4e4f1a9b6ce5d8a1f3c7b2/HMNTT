### Title
Unbounded `inner_transactions` Expansion in `getTransactionsByIdOrHash` Causes Memory and DB Resource Exhaustion

### Summary
`getTransactionsByIdOrHash` in `rest/transactions.js` collects all entries from the `inner_transactions` bigint[] column of every returned row, issues a second unbounded SQL query for all of them, and then spreads the results into the in-memory `rows` array with no cap. `formatTransactionRows` subsequently processes the entire expanded array, and `getStakingRewardTransferList` issues a single `IN (…)` query whose parameter list grows linearly with the number of inner transactions. No limit, pagination, or guard exists anywhere in this path.

### Finding Description
**Code path** (`rest/transactions.js`):

1. Lines 944–947 – all `inner_transactions` bigint pairs are collected from every row with no size check:
   ```js
   const innerTransactions = rows
     .map((row) => row.inner_transactions)
     .filter((innerTransactions) => innerTransactions)
     .flat();
   ```
2. Lines 949–958 – `getTransactionsByTransactionIdsSql` builds one SQL `OR` condition per pair (no limit), the result set is fetched, and every row is spread into the existing `rows` array:
   ```js
   const {rows: innerTransactionRows} = await pool.queryQuietly(innerTransactionQuery, params);
   rows.push(...innerTransactionRows);
   ```
3. Line 960 – the entire expanded array is processed in memory:
   ```js
   const transactions = await formatTransactionRows(rows);
   ```
4. Lines 253–282 – `createStakingRewardTransferList` → `getStakingRewardTransferList` issues **one** `IN (…)` query whose parameter list equals the number of rows that contain a staking-reward transfer. This is not per-row, but the parameter list still grows linearly and unboundedly with the expanded row count.

**Root cause / failed assumption**: The implementation assumes `inner_transactions` will always be small. The `inner_transactions` column is `bigint[] DEFAULT NULL` with no database-level size constraint. The REST layer imposes no cap on how many inner-transaction pairs it will fetch, expand, or format.

**Why existing checks fail**: The only guard is `if (!isTransactionHash && innerTransactions.length > 0)` (line 949), which is a presence check, not a size check. `acceptedSingleTransactionParameters` (line 1014) only allows `nonce` and `scheduled` filters; neither limits the inner-transaction fan-out. No rate limiting was found in the REST layer for this endpoint.

### Impact Explanation
Each request for a batch transaction ID with N inner transactions causes: (a) one extra full-table-range SQL query with N OR-conditions, (b) up to N additional full transaction rows loaded into Node.js heap, (c) one `IN (N params)` staking-reward query, and (d) synchronous in-memory formatting of all N+original rows. Repeated requests against the same large batch transaction ID can exhaust Node.js heap and saturate the database connection pool, degrading or denying service to all other mirror-node API consumers. No economic damage falls on other network participants; only the mirror-node operator is affected.

### Likelihood Explanation
An attacker submits one atomic-batch transaction to the Hedera network (paying their own fees once), permanently recording a large `inner_transactions` array in the mirror node database. They then replay the cheap read-only GET request `/api/v1/transactions/<batchTxId>` at high frequency from any number of unauthenticated clients. The one-time on-chain cost is fixed and small relative to the sustained off-chain amplification achievable against the mirror node.

### Recommendation
1. **Cap inner-transaction fan-out**: After line 947, add a hard limit (e.g., matching the configured response limit or a dedicated `maxInnerTransactions` config value) and return a 400 or truncate silently:
   ```js
   if (innerTransactions.length > MAX_INNER_TRANSACTIONS) {
     throw new InvalidArgumentError('batch inner transaction count exceeds limit');
   }
   ```
2. **Paginate inner transactions**: Apply the same `limit` logic used by `doGetTransactions` to the inner-transaction secondary query.
3. **Add rate limiting** on the `/transactions/:id` endpoint to bound repeated expansion of the same large batch transaction.

### Proof of Concept
1. Submit an `ATOMIC_BATCH` transaction to the Hedera network containing the maximum allowed number of inner transactions (each a minimal `CryptoTransfer`). Record the outer transaction ID, e.g. `0.0.X-NNNNNNNNNN-NNNNNNNNN`.
2. Wait for the mirror node to ingest the record. Confirm `inner_transactions` is populated:
   ```sql
   SELECT array_length(inner_transactions,1) FROM transaction WHERE type=74 LIMIT 5;
   ```
3. From any unauthenticated client, send repeated requests:
   ```
   GET /api/v1/transactions/0.0.X-NNNNNNNNNN-NNNNNNNNN
   ```
4. Observe: Node.js heap grows with each request; database shows large `IN (…)` queries; response latency for all other API endpoints increases; under sustained load the process OOMs or the DB connection pool is exhausted.
