### Title
Global Rate-Limit Exhaustion via Minimal-Payload No-Op EVM Calls

### Summary
Any unauthenticated caller can POST to `/api/v1/contracts/call` with `data = "0x"` (or omit `data` entirely, triggering the `HEX_PREFIX` default) and `gas = 21000` (the enforced minimum). Each such request consumes one global rate-limit token and only 2 scaled gas tokens, yet still triggers a full EVM execution path. Because the rate-limit bucket is a single shared `Bucket` bean with no per-IP partitioning, an attacker can saturate the 500 req/s global cap while barely touching the gas bucket, starving legitimate users.

### Finding Description

**Code path:**

`ContractController.call()` (line 40) calls `throttleManager.throttle(request)` **before** any parameter construction. Inside `ThrottleManagerImpl.throttle()` (lines 38–42):

```java
if (!rateLimitBucket.tryConsume(1)) { throw ... }
else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) { throw ... }
```

`ThrottleProperties.scaleGas(21_000)` returns `Math.floorDiv(21_000, 10_000) = 2` scaled tokens. The gas bucket holds `scaleGas(7_500_000_000) = 750_000` tokens/s, so 2 tokens is negligible.

Then `constructServiceParameters()` (line 68):

```java
data = request.getData() != null ? request.getData() : HEX_PREFIX;
```

`"0x"` passes `HexValidator` (pattern `^(0x)?[0-9a-fA-F]{0,N}$` with zero hex chars allowed), no exception is thrown, so the `catch (InvalidParametersException)` block in `call()` (lines 46–50) — the **only** place tokens are restored — is never reached. The call proceeds to `contractExecutionService.processCall(params)` with empty calldata.

`restoreGasToBucket()` (lines 140–151) restores `min(gasRemaining, gasLimitToRestoreBaseline)`. For a no-op call, `gasUsed ≈ gasLimit = 21_000`, so `gasRemaining = 0` and 0 gas tokens are restored. The rate-limit token (1 per request) is **never** restored under any code path.

**Root cause:** The rate-limit bucket is a single global `Bucket` bean (no per-source partitioning), and the minimum-gas + empty-data combination reduces per-request cost to 1 rate-limit token + 2 gas tokens while still triggering the full EVM stack.

### Impact Explanation

The global `rateLimitBucket` refills at 500 tokens/s. An attacker sending 500 req/s with `gas=21_000, data="0x"` exhausts the rate-limit bucket entirely, causing every subsequent legitimate request to receive `ThrottleException("Requests per second rate limit exceeded")`. The gas bucket is consumed at only 1_000/750_000 = 0.13% per second, so the gas-based throttle provides no meaningful protection. Server-side, each request still traverses the full EVM setup, DB state lookup, and metrics pipeline.

### Likelihood Explanation

No credentials, API keys, or on-chain assets are required. The endpoint is publicly reachable. A single attacker with a modest HTTP client (e.g., `wrk`, `ab`, or a simple loop) can sustain 500 req/s from one host. The payload is 2 bytes (`"0x"`), minimizing bandwidth cost. The attack is trivially repeatable and requires no special knowledge beyond the public OpenAPI spec.

### Recommendation

1. **Per-source rate limiting:** Partition the `rateLimitBucket` by client IP (or authenticated identity) so one source cannot exhaust the global cap.
2. **Minimum calldata enforcement:** Reject requests where `data` resolves to `"0x"` (empty calldata) and `to` is the zero address or a non-contract address, as these are semantically no-ops.
3. **Gas floor for throttle accounting:** Treat any request with `scaleGas(gas) < some_minimum` as consuming a higher token count to prevent the 2-token loophole.
4. **Connection-level limiting:** Apply an upstream rate limit (e.g., nginx `limit_req`) per IP before requests reach the Spring layer.

```java
    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
```
