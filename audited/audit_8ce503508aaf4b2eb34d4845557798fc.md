### Title
Unauthenticated DB Amplification DoS via `findStorageBatch()` Cache Asymmetry in `ContractStateServiceImpl`

### Summary
The `findStorage()` method in `ContractStateServiceImpl` uses a two-tier cache design where slot keys accumulate in `contractSlotsCache` (5-minute TTL, up to 1500 entries per contract) while slot values expire from `contractStateCache` every 2 seconds. An unprivileged attacker sending 500 RPS (the global limit) with unique uncached slots causes up to 500 `findStorageBatch()` DB queries per second, each issuing a `WHERE slot IN (N)` query that grows to include all 1500 accumulated slot keys — a 3000× amplification of DB row lookups compared to the baseline single-slot query path.

### Finding Description

**Code path:**
- `ContractStateServiceImpl.findStorage()` — lines 58–70
- `ContractStateServiceImpl.findStorageBatch()` — lines 85–122
- `ContractStateRepository.findStorageBatch()` — line 28
- `ThrottleProperties.requestsPerSecond` default — line 35 of `ThrottleProperties.java`
- `CacheProperties.contractState` — line 28 of `CacheProperties.java` (`expireAfterWrite=2s`)
- `CacheProperties.slotsPerContract` — line 39 of `CacheProperties.java` (`expireAfterAccess=5m,maximumSize=1500`)

**Root cause — cache TTL asymmetry:**

`findStorage()` checks `contractStateCache` (2-second write TTL) for the slot value. On a miss, it calls `findStorageBatch()`:

```java
// Line 63-69: contractStateCache miss → always falls through to findStorageBatch()
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);
if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
return findStorageBatch(contractId, key);
```

Inside `findStorageBatch()`, the new slot key is added to `contractSlotsCache` (5-minute TTL), then **all currently accumulated slot keys** for that contract are passed to the DB:

```java
// Lines 90-103: accumulate key, then query ALL accumulated keys
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
// ... builds cachedSlots list from ALL keys ...
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

The DB query issued is:
```sql
SELECT slot, value FROM contract_state WHERE contract_id = :contractId AND slot IN (:slots)
```

**The asymmetry:** Slot keys live in `contractSlotsCache` for 5 minutes; slot values live in `contractStateCache` for only 2 seconds. After 2 seconds, all values expire, but the keys remain. Every subsequent request for any of those slots triggers `findStorageBatch()` with the full accumulated key set.

**The global throttle does not prevent this.** The `rateLimitBucket` in `ThrottleManagerImpl` is a single shared bucket4j token bucket with no per-IP partitioning:

```java
// ThrottleConfiguration.java lines 25-32: single global bucket
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    ...
    return Bucket.builder().addLimit(limit).build();
}
```

A single attacker can consume all 500 tokens/second. Each consumed token corresponds to one EVM execution that calls `findStorage()`.

**Attack amplification math:**
- Phase 1 (0–3s): Attacker sends 500 RPS with unique slots targeting one contract. After 1500 requests, `contractSlotsCache` is full (maximumSize=1500). Each `findStorageBatch()` call grows from 1-slot to 1500-slot IN clause.
- Phase 2 (every 2s thereafter): `contractStateCache` expires. All 1500 slot values are gone. The next 500 requests each trigger `findStorageBatch()` with 1500 slots → 500 × 1500 = **750,000 DB row lookups per second** vs. the baseline 500.

### Impact Explanation
The DB is subjected to 500 concurrent `WHERE slot IN (1500 slots)` queries per second, each requiring a full index scan over 1500 slot keys. This saturates the PostgreSQL connection pool and I/O bandwidth of the mirror node's database. Since the mirror node DB is shared infrastructure for all mirror node services (importer, REST API, web3), saturation degrades all dependent services. The `statementTimeout` of 3000ms means queries pile up rather than fail fast, exhausting the connection pool. This constitutes degradation of 30%+ of network processing nodes without requiring any privileged access or brute force.

### Likelihood Explanation
The attack requires only:
1. Network access to the web3 JSON-RPC endpoint (publicly exposed)
2. Knowledge of any valid contract address with storage slots
3. A script generating 500 unique `eth_getStorageAt` or `eth_call` requests per second

No authentication, no tokens, no special knowledge. The 500 RPS limit is the attacker's budget, not a defense. The attack is repeatable indefinitely and can be sustained from a single machine or a small botnet. The 2-second `contractStateCache` TTL ensures the amplification resets and repeats automatically.

### Recommendation
1. **Add a per-contract batch query rate limit** inside `findStorageBatch()` to prevent more than N batch DB queries per second per contract.
2. **Cap the batch size** passed to `findStorageBatch()` — do not pass all accumulated slot keys; limit to a configurable maximum (e.g., 50) to bound the IN clause size.
3. **Align cache TTLs**: Either increase `contractStateCache` TTL to match `slotsPerContract` TTL (5 minutes), or reduce `slotsPerContract` TTL to match `contractStateCache` (2 seconds) so keys and values expire together.
4. **Add per-IP rate limiting** at the ingress layer (e.g., nginx, Kubernetes ingress) so a single source cannot consume the full 500 RPS global budget.
5. **Bound the `contractSlotsCache` per-contract size** more aggressively (e.g., 50–100 slots) to limit the IN clause growth.

### Proof of Concept

```python
import asyncio, aiohttp, itertools, random

TARGET = "http://<mirror-node>:8545"
CONTRACT = "0x<valid_contract_with_storage>"

async def attack():
    async with aiohttp.ClientSession() as session:
        slot_counter = itertools.count(0)
        async def send_one():
            slot = hex(next(slot_counter))  # unique slot each time
            payload = {
                "jsonrpc": "2.0", "method": "eth_getStorageAt",
                "params": [CONTRACT, slot, "latest"], "id": 1
            }
            await session.post(TARGET, json=payload)

        # Sustain 500 RPS with unique slots
        while True:
            tasks = [send_one() for _ in range(500)]
            await asyncio.gather(*tasks)
            await asyncio.sleep(1)

asyncio.run(attack())
```
