### Title
Unbounded Historical Scan via startTime=0 in PollingTopicListener Enables Resource Exhaustion

### Summary
Any unauthenticated gRPC client can subscribe to a topic with `consensusStartTime` set to epoch (0), which passes all validation in `TopicMessageFilter` and causes `PollingTopicListener.poll()` to repeatedly query `topic_message` from timestamp 0 every 500ms. There is no lower-bound enforcement on `startTime` beyond `@Min(0)`, no rate limiting on subscriptions, and no authentication on the gRPC endpoint. Multiple concurrent such subscriptions amplify DB load, degrading query latency for all subscribers.

### Finding Description

**Code path:**

`ConsensusController.toFilter()` [1](#0-0)  converts the protobuf `Timestamp{seconds:0, nanos:0}` to `long 0` via `DomainUtils.timestampInNanosMax`, then calls `filter.startTime(0)`.

`TopicMessageFilter` validation: [2](#0-1) 
- `@Min(0)` on `startTime` explicitly permits 0.
- `isValidStartTime()` checks `startTime <= DomainUtils.now()` — 0 always satisfies this.

`PollingTopicListener.poll()`: [3](#0-2) 
On the first poll, `last == null`, so `startTime = filter.getStartTime() = 0`. The query is issued with `startTime=0` and `pageSize=min(Integer.MAX_VALUE, 5000)=5000`. The poll repeats every 500ms: [4](#0-3) 

**Root cause:** The failed assumption is that `startTime=0` is an invalid or unreachable input. The `@Min(0)` constraint and `isValidStartTime()` check together make 0 a fully valid value, with no floor enforced closer to the present. [5](#0-4) 

**Exploit flow:**
1. Attacker opens N concurrent gRPC streams to `subscribeTopic` with `consensusStartTime={seconds:0, nanos:0}` and no `consensusEndTime`.
2. Each stream triggers `PollingTopicListener.listen()` → `poll()` every 500ms.
3. Each poll issues a DB query scanning `topic_message` from timestamp 0 for the given topic, returning up to 5000 rows.
4. For a topic with millions of historical messages, each poll is a large index range scan; N concurrent attackers multiply this load linearly.
5. Legitimate subscribers' polls are queued behind attacker queries, increasing latency for real-time message delivery.

**Why existing checks fail:**
- `@Min(0)` is the only constraint — it was intended to prevent negative values, not to prevent epoch-origin scans. [6](#0-5) 
- `isValidStartTime()` only checks `<= now()`, not `>= (now - some_max_lookback)`. [7](#0-6) 
- `maxPageSize=5000` limits rows returned per query but does not limit the number of concurrent subscriptions or the scan depth. [8](#0-7) 
- No authentication, rate limiting, or per-client subscription cap is present in `ConsensusController`. [9](#0-8) 

### Impact Explanation
DB query latency increases proportionally with the number of attacker-controlled subscriptions. For deployments with large `topic_message` tables, each epoch-origin poll is a costly range scan. Legitimate subscribers sharing the same DB connection pool experience increased latency for message delivery. In a settlement context where timely message delivery is required for confirmation, this latency increase is the mechanism for delayed confirmation. The impact is bounded per-query by `maxPageSize=5000` but is unbounded in terms of concurrent subscriptions. **Note:** This vulnerability is only active when `ListenerProperties.type` is configured to `POLL` or `SHARED_POLL`; the default is `REDIS`. [10](#0-9) 

### Likelihood Explanation
Any network-reachable client with no credentials can open gRPC streams. The exploit requires only a standard gRPC client library and knowledge of a valid `topicId`. It is trivially repeatable and scriptable. The constraint that POLL mode must be configured reduces the default attack surface, but operators who enable POLL mode for performance or Redis-unavailability reasons are fully exposed.

### Recommendation
1. **Enforce a maximum lookback window**: Add a `@Min` lower bound or a custom validator in `TopicMessageFilter.isValidStartTime()` that rejects `startTime` older than a configurable maximum (e.g., `now - maxLookback`).
2. **Rate-limit subscriptions per client IP/identity**: Add a per-client subscription count cap in `ConsensusController`.
3. **Add a minimum startTime floor in `PollingTopicListener.poll()`**: If `filter.getStartTime() < (now - maxLookback)`, clamp it to `now - maxLookback`.
4. **Require authentication** on the gRPC endpoint to prevent anonymous abuse.

### Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_topic_query_pb2
from google.protobuf.timestamp_pb2 import Timestamp
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# epoch startTime = 0
query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=<valid_topic_id>),
    consensusStartTime=Timestamp(seconds=0, nanos=0),
    # no limit, no endTime
)

# Open N concurrent streams
import threading
def subscribe():
    for msg in stub.subscribeTopic(query):
        pass  # consume silently

threads = [threading.Thread(target=subscribe) for _ in range(100)]
for t in threads: t.start()
# Each thread triggers a poll from timestamp 0 every 500ms
# DB load scales linearly with N
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
