### Title
Redirect-Following HTTP Client in `buildNetworkFromMirrorNodes()` Enables MITM-Triggered Startup Freeze via Multiplied Timeout Exhaustion

### Summary
The `http.Client` constructed in `buildNetworkFromMirrorNodes()` has no `CheckRedirect` policy, so Go's default behavior follows up to 10 redirects per attempt. An attacker who can intercept the plain-HTTP mirror REST traffic can return a 301/302 to a slow/black-hole server, consuming the full `mirrorNodeClientTimeout` (default 10 s) per attempt. With the default 11 attempts and exponential backoff, total blocking time reaches ~621 s — far exceeding 500% of Hedera's ~3–5 s average block time — and prevents the pinger from ever submitting transactions.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, lines 46–69; `fetchMirrorNodeNetwork()`, lines 84–86.

```
46:  httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
...
84:  resp, err := httpClient.Do(req)
85:  if err != nil {
86:      return nil, true, fmt.Errorf("GET %s failed: %w", url, err)
```

**Root cause — three compounding failures:**

1. **No `CheckRedirect` function.** `http.Client` at line 46 sets only `Timeout`; `CheckRedirect` is `nil`. Go's default policy follows up to 10 redirects per `Do()` call. A 301/302 response pointing at a black-hole host is silently followed, and the client blocks until the full `Timeout` expires.

2. **Per-attempt timeout, not cumulative.** `http.Client.Timeout` (default 10 s, `config.go` line 90/131) is reset on every call to `httpClient.Do()`. Each of the 11 attempts (`max(10+1,1)`, line 48) can independently consume the full 10 s.

3. **Unconditional `retry=true` on transport error.** Line 86 returns `retry=true` for *any* `httpClient.Do()` error, including a timeout caused by a redirect to a slow server. The retry loop at lines 52–69 therefore re-issues the request after exponential backoff (`500ms × 2^(attempt-1)`, line 63) without any cumulative deadline.

**Exploit flow:**

- Attacker intercepts the plain-HTTP GET to `http://mirror-rest:5551/api/v1/network/nodes`.
- Returns `HTTP 301 Location: http://attacker-slow-host/` (a host that accepts TCP but never sends a response body).
- `httpClient.Do()` follows the redirect and blocks for 10 s before timing out.
- Error propagates to line 86 as `retry=true`.
- Loop retries 10 more times with backoffs of 500 ms, 1 s, 2 s, 4 s, 8 s, 16 s, 32 s, 64 s, 128 s, 256 s.
- Total blocking: 11 × 10 s (timeouts) + ~511.5 s (backoffs) ≈ **621 s**.
- `buildNetworkFromMirrorNodes()` is called from `sdk_client.go` line 18 with `context.Background()` — no deadline — so the entire pinger startup is frozen for this duration.

**Why existing checks are insufficient:**

- The `ctx.Done()` check in the backoff `select` (line 64–67) only fires if the *parent* context is cancelled (e.g., SIGTERM). There is no startup deadline on the `context.Background()` passed at `sdk_client.go:18`.
- The 301/302 response itself never reaches the `resp.StatusCode` check at line 90 because `http.Client` transparently follows the redirect before returning; the final response from the slow server is what times out.
- TLS is not enforced; the default `mirrorRest` URL is `http://` (config.go line 37), making passive MITM straightforward on any shared network segment.

### Impact Explanation

`buildNetworkFromMirrorNodes()` is the sole path for `network=other` deployments to discover consensus nodes. Blocking it for ~621 s prevents `newClient()` from returning, which prevents `main()` from writing `/tmp/ready`, which causes Kubernetes readiness probes to fail and the pod to be restarted — restarting the same vulnerable initialization loop. The pinger never submits transactions, constituting a complete, sustained freeze of monitored network activity. Severity: **High** (availability impact on the monitoring/pinger layer; no confidentiality or direct ledger-state impact).

### Likelihood Explanation

Precondition is network-level MITM on a plain-HTTP channel. This is achievable by: (a) an attacker on the same Kubernetes cluster namespace via DNS spoofing of the `mirror-rest` service name, (b) a compromised sidecar or init container, or (c) ARP/BGP poisoning on a flat network. No credentials or privileged API access are required. The attack is repeatable: every pod restart re-enters the vulnerable loop. Likelihood: **Medium** (requires network position, but no application-layer privilege).

### Recommendation

1. **Disable redirect following** by setting `CheckRedirect` to return `http.ErrUseLastResponse` (or an explicit error), so any 3xx from the mirror endpoint is treated as a non-retryable error rather than silently followed:
   ```go
   httpClient := &http.Client{
       Timeout: cfg.mirrorNodeClientTimeout,
       CheckRedirect: func(_ *http.Request, _ []*http.Request) error {
           return http.ErrUseLastResponse
       },
   }
   ```
2. **Enforce a cumulative deadline** by passing a context with a total timeout into `buildNetworkFromMirrorNodes()` rather than `context.Background()` (e.g., `context.WithTimeout(ctx, cfg.mirrorNodeClientTimeout * time.Duration(attempts))`).
3. **Require HTTPS** for the mirror REST endpoint and validate the server certificate to eliminate the MITM precondition entirely.
