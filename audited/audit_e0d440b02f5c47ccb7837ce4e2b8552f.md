### Title
Unbounded Per-Request DB Query Amplification via N+1 `GetAccountAlias()` Calls in `/block` Endpoint

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` calls `updateOperationAccountAlias()`, which issues one synchronous `GetAccountAlias()` database query per operation per transaction for every account ID not present in the in-memory LRU cache. An unauthenticated attacker can repeatedly request a block containing many transactions with many unique-account operations, causing O(transactions × operations) sequential DB queries per request. The LRU cache provides no protection against concurrent requests or cache-cold conditions, enabling a database connection pool exhaustion attack.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 104–129:

```go
for _, transaction := range transactions {
    operations := transaction.Operations
    for index := range operations {
        accountId := operations[index].AccountId
        if cached, found = s.entityCache.Get(accountId.GetId()); !found {
            result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // DB query per miss
            ...
            s.entityCache.Set(result.GetId(), result)
        }
        operations[index].AccountId = cached
    }
}
```

Each cache miss fires the query `select alias, id from entity where id = @id` against the database (`rosetta/app/persistence/account.go`, line 117).

**Root cause — failed assumption:** The code assumes the LRU cache will absorb repeated lookups. It does not account for:
1. **Cache stampede**: There is no mutex or singleflight guard around the `Get → DB query → Set` sequence. Concurrent requests for the same cold block each independently fire T×O queries before any of them can populate the cache.
2. **Cache eviction**: The LRU cache (default `maxSize = 524288`) can be evicted by requests referencing other account IDs, forcing re-queries on the next `/block` call.
3. **No per-request operation count cap**: `maxTransactionsInBlock` limits the number of transactions but places no bound on the number of operations per transaction.

**Exploit flow:**
1. Attacker identifies (or waits for) a block on the network with `T` transactions each containing `O` operations referencing distinct account IDs (e.g., a block with 500 transactions × 20 operations = 10,000 unique IDs).
2. Attacker sends N concurrent `POST /block` requests for that block identifier.
3. All N requests simultaneously enter `updateOperationAccountAlias()` with a cold cache.
4. Each request independently fires up to T×O DB queries before any can populate the cache.
5. Total DB queries = N × T × O, exhausting the connection pool (`maxOpenConnections = 100` by default).

### Impact Explanation

The database connection pool for the Rosetta service defaults to 100 open connections. A single attacker sending ~10 concurrent requests against a block with 10,000 unique account IDs can saturate the pool with 100,000 simultaneous `SELECT` queries. This causes:
- DB connection pool exhaustion, making all subsequent Rosetta API requests fail with `ErrDatabaseError`
- Increased DB CPU/IO load, potentially impacting other mirror node components sharing the same database
- Sustained impact as long as the attacker keeps sending requests (the cache is re-cold after eviction)

Severity: **High** — complete availability loss of the Rosetta API with no authentication required.

### Likelihood Explanation

The `/block` endpoint requires no credentials. The Rosetta API is a public-facing interface by design. On Hedera mainnet, blocks routinely contain hundreds of transactions with multiple crypto transfers each. The Traefik rate-limit middleware (`average: 10` per host, `inFlightReq: 5` per IP) is an optional Kubernetes deployment artifact — it is not enforced at the application layer and is absent in non-Helm deployments. The attack is trivially repeatable with a simple HTTP client loop and requires no special knowledge beyond a valid block index.

### Recommendation

1. **Batch the alias lookups**: Replace the per-operation `GetAccountAlias()` loop with a single batched query that fetches all aliases for all unique account IDs in the block in one SQL `WHERE id IN (...)` call.
2. **Add a singleflight/coalescing layer**: Use `golang.org/x/sync/singleflight` around the cache-miss DB call so concurrent requests for the same account ID share a single in-flight query.
3. **Cap total operations per request**: Enforce a hard limit on the total number of operations processed per `/block` response (not just transactions).
4. **Application-level rate limiting**: Add per-IP rate limiting inside the Go HTTP handler, independent of infrastructure middleware.

func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

	return nil

func (ar *accountRepository) GetAccountAlias(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return accountId, nil
		}

		return zero, hErrors.ErrDatabaseError
	}

	if len(entity.Alias) == 0 {
		return accountId, nil
	}

	accountAlias, err := types.NewAccountIdFromEntity(entity)
	if err != nil {
		log.Warnf("Failed to create AccountId from alias '0x%s': %v", hex.EncodeToString(entity.Alias), err)
		return accountId, nil
	}

	return accountAlias, nil
}
