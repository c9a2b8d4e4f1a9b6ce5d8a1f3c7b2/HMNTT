### Title
Negative Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode()` Allows Unprivileged DoS of Newly Deployed Contracts

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"` to suppress caching of absent results, but this guard is ineffective because `Optional.empty()` is a non-null object. Any request for a contractId that arrives before the importer writes the bytecode to the DB will cache `Optional.empty()`, and the 1-hour `expireAfterAccess` TTL means the correct bytecode is blocked from being served for up to an hour after deployment. No authentication or privilege is required to trigger this.

### Finding Description
**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause — failed assumption in `unless` condition:**
Spring's `unless` SpEL expression is evaluated against the actual return value. The method returns `Optional<byte[]>`. When the contract does not exist in the DB, the JPA query returns `Optional.empty()`. `Optional.empty()` is a singleton non-null object, so `#result == null` evaluates to `false`, and the `unless` guard does **not** suppress caching. The empty Optional is stored in the Caffeine cache under the contractId key.

**Cache TTL:**
`web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 22:
```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```
The cache uses `expireAfterAccess=1h`. Every subsequent read of the poisoned key resets the 1-hour window, so an attacker who keeps polling can extend the denial indefinitely.

**No `sync = true`:**
The `@Cacheable` annotation does not set `sync = true`. Without it, Spring does not serialize concurrent cache-miss executions. Multiple threads that simultaneously miss the cache all proceed to execute the DB query, all receive `Optional.empty()`, and all attempt to write it to the cache. This is not strictly required for the exploit (a single pre-deployment request suffices), but it amplifies the race window.

**Exploit flow:**
1. Attacker observes or predicts an upcoming contractId (Hedera entity IDs are sequential and publicly visible on-chain before the importer processes them).
2. Attacker sends one or more `eth_call` / `eth_getCode` requests to the web3 API for that contractId before the importer writes the `runtime_bytecode` row to the `contract` table.
3. `findRuntimeBytecode()` queries the DB, gets `Optional.empty()`, and caches it (the `unless` guard does not fire).
4. The importer subsequently writes the correct bytecode to the DB.
5. All future calls to `findRuntimeBytecode()` for that contractId return the cached `Optional.empty()` without hitting the DB.
6. `ContractBytecodeReadableKVState.readFromDataSource()` maps `Optional.empty()` to `null` / `Bytecode` absent, so every EVM execution that needs this contract's code sees it as non-existent.

### Impact Explanation
Any contract deployed on the Hedera network can be rendered inaccessible via the mirror node's web3 API for up to 1 hour (or indefinitely if the attacker keeps the cache entry alive by polling). Callers receive results as if the contract does not exist: `eth_getCode` returns empty, `eth_call` against the contract fails or returns as if calling an EOA. This is a targeted, repeatable denial-of-service against specific contracts with no authentication required.

### Likelihood Explanation
Hedera entity IDs are monotonically increasing and publicly observable. An attacker watching the network can predict the next contractId with high confidence. The importer introduces a processing lag (typically seconds to tens of seconds) between on-chain finality and DB write, giving a reliable window to send the poisoning request. No special privileges, tokens, or network access beyond the public web3 API endpoint are needed. The attack is trivially scriptable and repeatable for every new contract deployment.

### Recommendation
Fix the `unless` condition to also exclude empty Optionals:
```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```
Additionally, add `sync = true` to prevent concurrent cache-miss stampedes:
```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    sync = true,
    unless = "#result == null || !#result.isPresent()"
)
```
This ensures that absent results are never persisted in the cache, so the next request after the importer writes the bytecode will fetch the correct value from the DB.

### Proof of Concept
1. Deploy a Hedera smart contract on testnet; observe the assigned entity ID `N` (e.g., via the consensus node receipt before the mirror node processes it).
2. Immediately send to the mirror node web3 API:
   ```
   POST /api/v1/contracts/call
   { "to": "0x<long-zero-address-of-entity-N>", "data": "0x" }
   ```
   or equivalently `eth_getCode` for the long-zero EVM address of entity `N`.
3. Confirm the mirror node returns empty bytecode (contract not yet indexed).
4. Wait for the importer to process the deployment (observe the contract appearing in the REST API).
5. Repeat the same `eth_getCode` call — the mirror node still returns empty bytecode, served from the poisoned cache.
6. Confirm the DB contains the correct `runtime_bytecode` row (`SELECT runtime_bytecode FROM contract WHERE id = N`), proving the cache is the source of the incorrect response.
7. The condition persists for 1 hour from the last access; polling every few minutes extends it indefinitely.
