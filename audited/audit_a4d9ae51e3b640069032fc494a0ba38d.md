### Title
Unbounded Per-IP Connection Acceptance in gRPC Server Enables Resource Exhaustion DoS

### Summary
`grpcServerConfigurer()` in `GrpcConfiguration.java` configures only `maxConcurrentCallsPerConnection` (default: 5), which limits concurrent RPC calls per individual connection but places no bound on the total number of connections a single IP may open. An unprivileged attacker can open an unbounded number of TCP connections to port 5600, exhausting file descriptors, thread-pool slots, and memory, degrading or denying service to all legitimate users of the gRPC API (HCS topic subscriptions and network address-book queries).

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35 (`grpcServerConfigurer()`):

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:

```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `maxConcurrentCallsPerConnection` is a *per-connection* call cap. It does not restrict how many connections a single IP may establish, nor does it configure `maxConnectionAge`, `maxConnectionIdle`, or any `ServerTransportFilter` for per-IP accounting. `NettyServerBuilder` exposes `maxConnectionAge()` and `maxConnectionIdle()` but neither is called. No `ServerTransportFilter` or interceptor performing per-IP connection counting exists anywhere in the `grpc/` module (confirmed by grep across all `grpc/**/*.java`).

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 in a tight loop (no authentication required).
2. Each connection is accepted by Netty and allocated OS file descriptors, Netty channel objects, and JVM heap.
3. Attacker optionally issues one long-lived streaming call per connection (`subscribeToTopic`) to pin thread-pool slots.
4. With enough connections the JVM hits file-descriptor limits, the shared `applicationTaskExecutor` thread pool saturates, or heap is exhausted.
5. Legitimate subscribers receive `UNAVAILABLE` or connection timeouts.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection=5` only prevents >5 *calls* on one connection; it does not reject the 10,001st *connection* from the same IP.
- The GCP backend policy `maxRatePerEndpoint: 250` in `charts/hedera-mirror-grpc/values.yaml` (line 69) is a request-rate hint that requires HPA changes to take effect (noted inline: `# Requires a change to HPA to take effect`) and is only active when `gcp.enabled: true`; it is not a code-level protection.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting (bucket4j) lives exclusively in the `web3` module and is not applied to the gRPC module at all.

### Impact Explanation
Exhausting file descriptors or the shared executor causes the gRPC server to stop accepting new connections, making HCS topic-message subscriptions and network address-book queries unavailable to all users. Because the gRPC streaming subscriptions are long-lived, even a modest number of connections (e.g., matching the OS `ulimit -n` or the executor's queue depth) can sustain the outage indefinitely. No funds are at direct risk, but service availability is fully compromised.

### Likelihood Explanation
No authentication or special privilege is required. A single machine with a standard TCP stack can open tens of thousands of connections. The attack is trivially scriptable (`for i in $(seq 10000); do grpc_client connect &; done`), repeatable, and requires no knowledge of the application beyond the open port. The absence of any code-level connection-rate guard makes this a low-effort, high-impact denial-of-service.

### Recommendation
Inside `grpcServerConfigurer()`, add connection-lifetime and idle bounds to force connection recycling and limit resource hold time:

```java
serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
```

Additionally, implement a custom `io.grpc.ServerTransportFilter` that tracks active connections per remote IP and rejects new transports once a configurable per-IP ceiling is reached. Expose the ceiling as a new `NettyProperties` field (e.g., `maxConnectionsPerIp`). This provides a code-level guard independent of infrastructure configuration.

### Proof of Concept
```bash
# Requires grpcurl or any gRPC client that can hold open streams
# Open 5000 simultaneous streaming subscriptions from one IP:

  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <grpc-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
