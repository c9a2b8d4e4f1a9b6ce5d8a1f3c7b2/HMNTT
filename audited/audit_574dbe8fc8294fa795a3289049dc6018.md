### Title
Silent Order Parameter Fallback: Numeric String Values Bypass Validation in `parseLimitAndOrderParams`

### Summary
When an unprivileged user sends `?order=0` or `?order=1`, `canonicalizeValue()` applies `lowerCaseQueryValue` to the numeric string (no change), and `parseLimitAndOrderParams()` silently falls back to the default sort order without returning any error to the caller. Endpoints such as `/accounts` and `/balances` that call `validateReq` (not `buildAndValidateFilters`) before `parseLimitAndOrderParams` do not validate the order value, leaving the silent fallback as the only behavior.

### Finding Description

**Code path:**

1. `requestQueryParser()` (`rest/middleware/requestHandler.js:38-69`) parses the query string and calls `canonicalizeValue('order', '0')`.
2. `canonicalizeValue()` (`rest/middleware/requestHandler.js:71-78`) looks up `queryCanonicalizationMap['order']` which is `lowerCaseQueryValue`. Calling `'0'.toLowerCase()` returns `'0'` — unchanged.
3. The parsed query object now contains `{ order: '0' }`.
4. `parseLimitAndOrderParams()` (`rest/utils.js:662-675`) reads `req.query['order']` and checks:
   ```js
   if (value === constants.orderFilterValues.ASC || value === constants.orderFilterValues.DESC) {
     order = value;
   }
   ```
   `'0'` matches neither `'asc'` nor `'desc'`, so `order` stays as `defaultOrder`. **No exception is thrown, no error is returned.**

**Why existing checks are insufficient:**

- `filterValidityChecks()` (`rest/utils.js:336-339`) does correctly reject `'0'` via `isValidValueIgnoreCase`, but it is only invoked through `buildAndValidateFilters` / `validateAndParseFilters`.
- `getAccounts` (`rest/accounts.js:337`) calls `utils.validateReq(req, acceptedAccountsParameters)` — not `buildAndValidateFilters` — before calling `parseLimitAndOrderParams`. `validateReq` checks that parameter *keys* are in the accepted set, but does not validate parameter *values*.
- `getBalances` (`rest/balances.js:84`) follows the same pattern: `validateReq` then `parseLimitAndOrderParams`.
- `parseLimitAndOrderParams` itself contains no error-throwing branch for unrecognized order values.

### Impact Explanation
The user receives results sorted by the server's default order (e.g., `ASC` for `/accounts`, `DESC` for `/balances`) with no indication that their `order` parameter was ignored. This breaks the API contract: the caller believes they are controlling sort order but is silently overridden. In paginated traversal scenarios, a client relying on a specific order to implement cursor-based pagination will silently receive data in the wrong order, potentially causing missed or duplicated records in application logic. Severity is **Low** — no data is corrupted, no authentication is bypassed, and there is no economic damage.

### Likelihood Explanation
Any unauthenticated HTTP client can trigger this with a single request (`GET /api/v1/accounts?order=0`). No special privileges, tokens, or knowledge of internals are required. The behavior is deterministic and 100% reproducible.

### Recommendation
Add explicit validation inside `parseLimitAndOrderParams` to throw an `InvalidArgumentError` when the supplied order value is not `null`/`undefined` and is not one of the accepted values:

```js
const value = req.query[constants.filterKeys.ORDER];
if (value !== undefined) {
  if (value !== constants.orderFilterValues.ASC && value !== constants.orderFilterValues.DESC) {
    throw InvalidArgumentError.forParams(constants.filterKeys.ORDER);
  }
  order = value;
}
```

Alternatively, ensure all endpoints that call `parseLimitAndOrderParams` first pass through `buildAndValidateFilters`, which already performs this check via `filterValidityChecks`.

### Proof of Concept

```
GET /api/v1/accounts?order=0
GET /api/v1/balances?order=1
```

**Expected behavior:** HTTP 400 with an error message indicating `order` is invalid.

**Actual behavior:** HTTP 200, results returned in the server's default sort order (`asc` for accounts, `desc` for balances) with no error or warning in the response. The `order` parameter is silently discarded.

  const canonicalizationFunc = queryCanonicalizationMap[key];
  if (canonicalizationFunc === undefined) {
    return value;
  }

  return Array.isArray(value) ? value.map((v) => canonicalizationFunc(v)) : canonicalizationFunc(value);
};

const parseLimitAndOrderParams = (req, defaultOrder = constants.orderFilterValues.DESC) => {
  // Parse the limit parameter
  const limitQuery = `${constants.filterKeys.LIMIT} ? `;
  const limitValue = getLimitParamValue(req.query[constants.filterKeys.LIMIT]);

  // Parse the order parameters (default: descending)
  let order = defaultOrder;
  const value = req.query[constants.filterKeys.ORDER];
  if (value === constants.orderFilterValues.ASC || value === constants.orderFilterValues.DESC) {
    order = value;
  }

  return buildPgSqlObject(limitQuery, [limitValue], order, limitValue);
};
