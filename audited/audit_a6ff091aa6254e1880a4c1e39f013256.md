### Title
Missing Account Existence/Deletion Check in `getAccountCryptoAllowances` Allows Querying Allowances for Non-Existent or Deleted Accounts

### Summary
`CryptoAllowanceController.getAccountCryptoAllowances()` calls `EntityService.getEncodedId()` with a plain numeric entity ID, which resolves the ID purely in-memory with no database lookup and no existence or deletion check. Unlike `tokenController.getTokenRelationships()` and `accountController.listStakingRewardsByAccountId()`, which both call `EntityService.isValidAccount()` and throw `NotFoundError` for invalid accounts, the crypto allowance endpoint skips this guard entirely, allowing any caller to enumerate allowance records for deleted or never-existed account IDs without receiving a 404.

### Finding Description
**Exact code path:**

In `rest/controllers/cryptoAllowanceController.js` lines 76–80:
```js
getAccountCryptoAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  // No isValidAccount() call here
  const {conditions, params, order, limit} = this.extractCryptoAllowancesQuery(filters, accountId);
  const allowances = await CryptoAllowanceService.getAccountCryptoAllowances(conditions, params, order, limit);
```

In `rest/service/entityService.js` lines 118–124, `getEncodedId()` for a plain numeric/shard.realm.num ID takes the branch:
```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()   // pure in-memory, zero DB contact
    : await this.getEntityIdFromEvmAddress(entityId, requireResult);
}
```
`entityId.getEncodedId()` is a pure arithmetic encoding — no database query, no existence check, no deletion check.

**Contrast with sibling controllers:**

`rest/controllers/tokenController.js` lines 67–71:
```js
const accountId = await EntityService.getEncodedId(...);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) { throw new NotFoundError(); }
```

`rest/controllers/accountController.js` lines 171–175 does the same pattern.

**`CryptoAllowanceService.getAccountCryptoAllowances()`** (`rest/service/cryptoAllowanceService.js` lines 13–17) issues a plain `SELECT * FROM crypto_allowance WHERE owner = $1 AND amount > 0` — no join to `entity`, no deletion filter.

**Root cause:** The controller omits the `isValidAccount()` guard that every comparable controller applies, and `getEncodedId()` for numeric IDs intentionally skips the DB entirely.

### Impact Explanation
An unauthenticated caller can supply any numeric entity ID (e.g., `0.0.99999`) and receive the full allowance list for that owner without a 404. For deleted accounts whose `crypto_allowance` rows were not purged (amount still > 0 at deletion time), this discloses spender identities and approved amounts. The inconsistency also breaks API contract: clients relying on 404 to detect invalid accounts will silently receive empty-or-populated responses instead.

Severity: **Low–Medium**. The mirror node REST API is a public read-only service and Hedera ledger data is inherently public; however, the missing guard violates the established pattern, leaks historical state for deleted accounts, and prevents clients from distinguishing "account never existed" from "account has no allowances."

### Likelihood Explanation
Exploitation requires zero privileges — any HTTP client can call `GET /api/v1/accounts/{id}/allowances/crypto` with an arbitrary numeric ID. The attack is trivially repeatable and requires no special knowledge beyond knowing (or brute-forcing) a deleted account's numeric ID, which is sequential and publicly observable on-chain.

### Recommendation
Add the `isValidAccount()` guard immediately after `getEncodedId()`, mirroring the pattern in `tokenController.js` and `accountController.js`:

```js
getAccountCryptoAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  if (!(await EntityService.isValidAccount(accountId))) {
    throw new NotFoundError();
  }
  // ... rest of handler
};
```

Additionally, consider updating `entityExistenceQuery` in `EntityService` to filter `deleted <> true` so that `isValidAccount()` also rejects deleted accounts (currently it only checks existence, not deletion status).

### Proof of Concept
1. Identify a deleted Hedera account (e.g., `0.0.12345`) from on-chain history.
2. Send: `GET /api/v1/accounts/0.0.12345/allowances/crypto`
3. **Expected (correct):** HTTP 404 Not Found
4. **Actual:** HTTP 200 with `{"allowances": [...], "links": {"next": null}}` — returns any remaining allowance rows for that owner without any account validity check.
