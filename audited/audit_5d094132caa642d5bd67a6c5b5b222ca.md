### Title
Unauthenticated Pagination Griefing via Unbounded Sequential DB Queries on `/api/v1/balances`

### Summary
The `getBalances()` handler in `rest/balances.js` issues a full PostgreSQL query on every request, and `formatBalancesResult()` unconditionally emits a `links.next` cursor URL that any anonymous caller can follow indefinitely. No rate-limiting middleware exists at the application layer for unauthenticated requests, so an attacker can drive sustained, high-frequency DB load by rapidly following pagination links, degrading response times for all other users.

### Finding Description
**Code path:**

- `rest/server.js` line 106: `app.getExt(`${apiPrefix}/balances`, balances.getBalances)` — the route is registered with no per-IP or per-session throttle middleware before it.
- The middleware chain (lines 68–144 of `server.js`) is: `urlencoded → json → cors → compression → httpContext → requestLogger → authHandler → [optional metricsHandler] → [optional responseCacheCheckHandler] → getBalances → responseHandler → [optional responseCacheUpdateHandler] → handleError`. There is no rate-limiting step anywhere in this chain.
- `rest/middleware/authHandler.js` lines 15–36: `authHandler` only sets a custom response-size limit (`userLimitLabel`) for requests that supply valid Basic-Auth credentials. Unauthenticated requests fall through with `return` at line 19 — no blocking, no counting, no throttling.
- `rest/balances.js` lines 83–156 (`getBalances`): every request executes at least one and up to three sequential DB queries (timestamp range lookup → public-key lookup → main balance query). The main query (lines 139–149 or 207–216) includes a correlated subquery per row for token balances.
- `rest/balances.js` lines 44–53 (`formatBalancesResult`): `links.next` is always populated when `ret.balances.length === limit`, encoding the last `account_id` as the next cursor. An attacker simply follows this URL to trigger the next page.
- `rest/middleware/responseCacheHandler.js` line 152: the cache key is `MD5(req.originalUrl)`. Each paginated request carries a different `account.id=lt:X` cursor, producing a unique cache key and a guaranteed DB miss on first traversal.

**Root cause:** The application assumes rate limiting is handled entirely by external infrastructure (Traefik). The Helm chart for the REST service (`charts/hedera-mirror-rest/values.yaml`) contains no `rateLimit` or `inFlightReq` middleware entries — unlike the Rosetta chart which explicitly configures both. There is therefore no fallback protection when the service is deployed without Traefik or with a misconfigured ingress.

### Impact Explanation
The `account_balance` and `entity` tables on a production mirror node contain millions of rows. The correlated token-balance subquery (`getTokenAccountBalanceSubQuery` / `getTokenBalanceSubQuery`) executes once per row in the result page. An attacker requesting `limit=100` per page and following `links.next` at maximum speed forces the database to execute O(100) correlated subqueries per request, continuously. Because each page URL is unique, the Redis response cache provides zero protection during the traversal. Sustained load of this kind raises DB CPU and I/O, increasing query latency for all concurrent legitimate users. No economic loss occurs, but availability and response-time SLAs are violated — consistent with the "griefing" classification.

### Likelihood Explanation
The attack requires zero privileges, zero credentials, and zero knowledge beyond the public API documentation (the `links.next` field is documented and returned in every paginated response). It is fully scriptable with a single `while links.next` loop in any HTTP client. The attacker can run multiple parallel pagination streams from different source IPs to multiply the effect. Repeatability is unlimited.

### Recommendation
1. **Application-layer rate limiting**: Add a per-IP rate-limit middleware (e.g., `express-rate-limit` or `rate-limiter-flexible` backed by Redis) in `rest/server.js` before the route handlers, specifically covering the `/api/v1/balances` path. This must not rely solely on Traefik being present.
2. **Concurrency cap**: Add an in-flight request counter (e.g., via `express-slow-down` or a semaphore) so that a single IP cannot hold more than N simultaneous DB connections.
3. **DB statement timeout**: Enforce a short `statement_timeout` on the DB pool used by the REST service so runaway queries are killed automatically (the REST Java service already does this at `restJava.db.statementTimeout = 10000 ms`; the Node REST service should do the same).
4. **Ensure Traefik middleware is mandatory**: If infrastructure-level rate limiting is the intended control, make the Helm chart's `middleware.rateLimit` a required value with a safe default (mirroring the Rosetta chart's `average: 10` + `inFlightReq: 5` pattern).

### Proof of Concept
```bash
#!/usr/bin/env bash
# No credentials required. Runs against any public mirror node.
BASE="https://<mirror-node-host>/api/v1"
URL="${BASE}/balances?limit=100"

while [ "$URL" != "null" ] && [ -n "$URL" ]; do
  RESP=$(curl -s "$URL")
  NEXT=$(echo "$RESP" | jq -r '.links.next // "null"')
  echo "Fetched page, next: $NEXT"
  # No sleep — maximum DB pressure
  if [ "$NEXT" = "null" ]; then
    break
  fi
  URL="${BASE}${NEXT}"
done
```

const getBalances = async (req, res) => {
  utils.validateReq(req, acceptedBalancesParameters, balanceFilterValidator);

  // Parse the filter parameters for credit/debit, account-numbers, timestamp and pagination
  const [accountQuery, accountParamsPromise] = parseAccountIdQueryParam(req.query, 'ab.account_id');
  const accountParams = await Promise.all(accountParamsPromise);
  // transform the timestamp=xxxx or timestamp=eq:xxxx query in url to 'timestamp <= xxxx' SQL query condition
  let [tsQuery, tsParams] = utils.parseTimestampQueryParam(req.query, 'consensus_timestamp', {
    [utils.opsMap.eq]: utils.opsMap.lte,
  });
  const [balanceQuery, balanceParams] = utils.parseBalanceQueryParam(req.query, 'ab.balance');
  const [pubKeyQuery, pubKeyParams] = utils.parsePublicKeyQueryParam(req.query, 'public_key');
  const {
    query: limitQuery,
    params,
    order,
    limit,
  } = utils.parseLimitAndOrderParams(req, constants.orderFilterValues.DESC);

  res.locals[constants.responseDataLabel] = {
    timestamp: null,
    balances: [],
    links: {
      next: null,
    },
  };

  let sqlQuery;
  let sqlParams;

  if (tsQuery) {
    const tsQueryResult = await getTsQuery(tsQuery, tsParams);
    if (!tsQueryResult.query) {
      return;
    }

    const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit);
    if (pubKeyQuery && !accountIdsQuery) {
      return;
    }

    [sqlQuery, tsParams] = await getBalancesQuery(
      accountQuery,
      balanceQuery,
      accountIdsQuery,
      limitQuery,
      order,
      tsQueryResult
    );
    sqlParams = utils.mergeParams(tsParams, accountParams, balanceParams, params);
  } else {
    // use current balance from entity table when there's no timestamp query filter
    const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
    const whereClause = conditions && `where ${conditions}`;
    const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);
    sqlParams = utils.mergeParams(tsParams, accountParams, pubKeyParams, balanceParams, params);
    sqlQuery = `
      with entity_balance as (
        select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
        from entity
        where type in ('ACCOUNT', 'CONTRACT')
      )
      select ab.*, (${tokenBalanceSubQuery}) as token_balances
      from entity_balance ab
      ${whereClause}
      order by ab.account_id ${order}
      ${limitQuery}`;
  }

  const pgSqlQuery = utils.convertMySqlStyleQueryToPostgres(sqlQuery);
  const result = await pool.queryQuietly(pgSqlQuery, sqlParams);
  res.locals[constants.responseDataLabel] = formatBalancesResult(req, result, limit, order);
  logger.debug(`getBalances returning ${result.rows.length} entries`);
};

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
