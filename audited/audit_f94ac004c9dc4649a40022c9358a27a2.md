### Title
Missing Entity-Type Validation in `listStakingRewardsByAccountId` Allows Querying Staking Reward History for Non-ACCOUNT Entities

### Summary
The `isValidAccount` function in `rest/service/entityService.js` only checks whether an entity exists in the database, but never validates that the entity's `type` field equals `'ACCOUNT'`. Because Hedera CONTRACT entities can also stake to nodes and accumulate staking reward transfers, an unprivileged attacker can call `GET /api/v1/accounts/{contractId}/rewards` with any CONTRACT (or other non-ACCOUNT entity) ID and receive a 200 response with that entity's full historical staking reward transfer records.

### Finding Description
**Root cause — `rest/service/entityService.js` lines 27–63:**

```javascript
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;

async isValidAccount(accountId) {
    const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
    return !isNil(entity);   // ← only checks existence, never inspects entity.type
}
```

The SQL query selects `entity.type`, but the returned row is never inspected. `!isNil(entity)` returns `true` for any entity type: `ACCOUNT`, `CONTRACT`, `TOKEN`, `TOPIC`, `FILE`, `SCHEDULE`.

**Call site — `rest/controllers/accountController.js` lines 170–185:**

```javascript
listStakingRewardsByAccountId = async (req, res) => {
    const accountId = await EntityService.getEncodedId(req.params[...]);
    const isValidAccount = await EntityService.isValidAccount(accountId);
    if (!isValidAccount) {
        throw new NotFoundError();   // ← never reached for a CONTRACT entity
    }
    // proceeds to query staking_reward_transfer for any entity type
    const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(...);
```

**Why CONTRACT entities have reward records:** The importer explicitly handles staking rewards for both `ACCOUNT` and `CONTRACT` types. The `entity_state_start` materialized view (`V1.65.6__pending_reward.sql`) and `RecalculatePendingRewardMigration` both include `type in ('ACCOUNT', 'CONTRACT')`. The `staking_reward_transfer` table has no type constraint on `account_id` — it stores reward payouts for any staking entity.

**Exploit flow:**
1. Attacker identifies a CONTRACT entity ID (e.g., `0.0.12345`) that has staked to a node.
2. Attacker sends `GET /api/v1/accounts/0.0.12345/rewards`.
3. `getEncodedId` resolves the numeric ID successfully (no type check here either).
4. `isValidAccount(encodedId)` queries the entity table, finds the CONTRACT row, returns `true`.
5. `StakingRewardTransferService.getRewards` executes `SELECT ... FROM staking_reward_transfer WHERE account_id = $1`, returning all historical reward payouts for that contract.
6. Server responds HTTP 200 with the full reward history.

### Impact Explanation
An unprivileged external user can enumerate the complete historical staking reward payout history (timestamps and tinybar amounts) for any CONTRACT entity through an endpoint explicitly documented and designed for ACCOUNT entities only. This reveals the staking activity cadence and cumulative reward amounts for smart contracts, which may be considered sensitive operational or financial data about those contracts. The endpoint is not designed to serve this data for contracts, and no access control or type gate prevents it.

### Likelihood Explanation
The attack requires zero privileges — only a valid numeric entity ID for a staking contract, which is publicly discoverable via the `/api/v1/contracts` endpoint. The exploit is a single unauthenticated HTTP GET request, fully repeatable, and requires no special tooling. Any entity ID that resolves to a CONTRACT with staking history will trigger the issue.

### Recommendation
Fix `isValidAccount` to actually validate the entity type, not just existence:

```javascript
async isValidAccount(accountId) {
    const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
    return !isNil(entity) && entity.type === 'ACCOUNT';
}
```

Alternatively, add a `type = 'ACCOUNT'` predicate directly to `entityExistenceQuery`. This is consistent with how the GraphQL layer enforces type checks via `getByIdAndType(entityId, EntityType.ACCOUNT)`.
