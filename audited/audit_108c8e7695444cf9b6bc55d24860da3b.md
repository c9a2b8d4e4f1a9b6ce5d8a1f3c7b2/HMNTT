### Title
Unvalidated User-Controlled `file.id` Parameter Allows Arbitrary Address Book File ID Injection in `getNetworkNodes()`

### Summary
An unprivileged external user can supply any arbitrary `file.id` query parameter (e.g., `file.id=eq:999999`) to the network nodes endpoint. The value is parsed, accepted without allowlist validation, and passed directly to the SQL query's `WHERE file_id = :fileId` clause. This causes the query to return an empty or incorrect node list, misrepresenting the network topology to any downstream consumer of the API.

### Finding Description

**Code path:**

1. `NetworkNodeRequest.fileId` is an optional, user-supplied `EntityIdRangeParameter` with no value-domain constraint: [1](#0-0) 

2. `getAddressBookFileId()` returns the raw user-supplied value with no allowlist check: [2](#0-1) 

3. That value is passed directly to `findNetworkNodes()`: [3](#0-2) 

4. The SQL CTE filters the entire address book on the attacker-controlled `fileId`: [4](#0-3) 

5. `EntityIdRangeParameter.valueOf()` only validates format (non-negative integer parts), not semantic validity against the set of known address book file IDs: [5](#0-4) 

**Root cause:** `getAddressBookFileId()` has a missing allowlist guard. The only two semantically valid file IDs are `101` and `102` (the two Hedera address books). Any other value should be rejected, but the code unconditionally trusts the caller-supplied value.

### Impact Explanation
Any client or automated system consuming `/api/v1/network/nodes?file.id=eq:999999` receives an empty node list. A downstream service that relies on this endpoint for peer/node discovery (e.g., SDK bootstrapping, load balancers, monitoring) would see zero nodes and could fail to connect to the network, misroute traffic, or raise false alerts. The attacker does not need any credentials.

### Likelihood Explanation
The endpoint is public and unauthenticated. The `file.id` parameter is documented (constant `FILE_ID` is referenced in the DTO). Any attacker who reads the API docs or fuzzes query parameters can trivially reproduce this. It is repeatable and requires no special tooling.

### Recommendation
In `getAddressBookFileId()`, validate the caller-supplied value against the set of permitted address book file IDs before using it:

```java
private static final Set<Long> VALID_ADDRESS_BOOK_FILE_IDS = Set.of(101L, 102L); // adjust to actual shard/realm IDs

private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    if (!VALID_ADDRESS_BOOK_FILE_IDS.contains(id)) {
        throw new IllegalArgumentException("Invalid file.id: only address book file IDs are permitted");
    }
    return id;
}
```

Alternatively, derive the valid set dynamically from `systemEntity.addressBookFile101()` and `systemEntity.addressBookFile102()`.

### Proof of Concept

```
GET /api/v1/network/nodes?file.id=eq:999999
```

**Preconditions:** None. No authentication required.

**Steps:**
1. Send the above HTTP GET request to the mirror node REST-Java API.
2. `EntityIdRangeParameter.valueOf("eq:999999")` succeeds (valid format).
3. `getAddressBookFileId()` returns `999999`.
4. `findNetworkNodes(999999, ...)` executes with `WHERE file_id = 999999` in the `latest_address_book` CTE.
5. CTE returns zero rows; the join with `address_book_entry` produces no results.

**Result:** HTTP 200 with `{ "nodes": [], ... }` — an empty node list that falsely implies no network nodes exist.
