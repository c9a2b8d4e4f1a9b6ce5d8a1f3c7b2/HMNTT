### Title
Thread Pool Exhaustion via Unbounded Concurrent Requests to `TopicServiceImpl.findById()`

### Summary
The `rest-java` module exposes `GET /api/v1/topics/{id}` with no rate limiting, no bulkhead, and no semaphore. Each request occupies a Tomcat worker thread while making three sequential synchronous blocking DB calls. An unprivileged attacker flooding this endpoint can exhaust all Tomcat worker threads, causing the server to stop accepting new connections and griefing every other API user.

### Finding Description
**Exact code path:**

`TopicController.getTopic()` (line 32–37) calls three sequential blocking repository lookups per request:
- `topicService.findById(id.id())` → `topicRepository.findById(id.getId())` (line 20)
- `entityService.findById(id.id())`
- `customFeeService.findById(id.id())`

All three calls are synchronous JPA/JOOQ operations that hold the Tomcat worker thread until the DB responds.

**Root cause — failed assumption:**
The `rest-java` module has **zero rate limiting or concurrency control** on this path. The only filters registered are `LoggingFilter` (logging only) and `ShallowEtagHeaderFilter` (ETag only). `WebMvcConfiguration` adds only argument resolvers. There is no `ThrottleManager`, no Resilience4j `Bulkhead`, no `Semaphore`, and no bucket4j rate limiter — in stark contrast to the `web3` module, which has an explicit `ThrottleConfiguration` with per-second request and gas limits.

**Exploit flow:**
1. Attacker opens N concurrent HTTP connections to `GET /api/v1/topics/0.0.1` (valid or invalid ID — both paths hit the DB).
2. Each request occupies one Tomcat worker thread (default max: 200) while waiting for a HikariCP connection (default pool: 10) and DB response.
3. With 200 concurrent requests in flight, all Tomcat threads are occupied. New connections queue up to the `accept-count` limit (default: 100), then are refused.
4. Legitimate users receive connection refused or timeout errors until the attacker stops.

**Why existing checks fail:**
- `LoggingFilter` only logs; it does not gate or reject requests.
- `ShallowEtagHeaderFilter` only computes ETags.
- HikariCP's connection pool (default 10) acts as a natural bottleneck but causes threads to *pile up waiting*, not to be released — worsening the exhaustion.
- No infrastructure-level rate limiting is configured in the `rest-java` Helm chart (`charts/hedera-mirror-rest-java/values.yaml`) or application properties (no `application.yml` exists in `rest-java/src/main/resources/`).

### Impact Explanation
All Tomcat worker threads become occupied with pending DB calls. The server stops accepting new TCP connections. Every other API endpoint served by the same `rest-java` process (topics, entities, custom fees, network info, etc.) becomes unavailable to all users. This is a complete denial-of-service of the `rest-java` service with no authentication or special privileges required. Severity is Medium (griefing, no economic damage to the network itself).

### Likelihood Explanation
Any external user with network access to the API can trigger this. No credentials, tokens, or special knowledge are required — only the ability to open many concurrent HTTP connections. Standard HTTP benchmarking tools (`wrk`, `ab`, `hey`) suffice. The attack is repeatable and can be sustained indefinitely. The endpoint is publicly documented.

### Recommendation
1. **Add a global rate-limiting filter** to `rest-java` using bucket4j (already a dependency in the sibling `web3` module), mirroring `ThrottleConfiguration` and `ThrottleManagerImpl`.
2. **Add a Resilience4j `Bulkhead`** (semaphore-based) on `TopicServiceImpl.findById()` to cap concurrent in-flight DB calls, e.g. `@Bulkhead(name = "topicFindById")`.
3. **Configure Tomcat `max-threads` and `accept-count`** explicitly in `application.yml` to prevent unbounded queuing.
4. **Set HikariCP `connectionTimeout`** to a low value (e.g. 2s) so threads waiting for a DB connection fail fast rather than pile up.
5. Deploy an ingress-level rate limiter (e.g. NGINX `limit_req`) as a defense-in-depth layer.
