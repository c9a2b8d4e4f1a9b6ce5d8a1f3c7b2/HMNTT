### Title
Unbounded LRU Cache Thrashing via Unauthenticated Token ID Enumeration Enables Non-Network DoS

### Summary
The `getTokenInfoRequest` handler in `rest/tokens.js` unconditionally calls `TokenService.putTokenCache(token)` on every response when no timestamp filter is present. Because the underlying `quickLru` token cache has a finite `maxSize` (default 100,000) with no TTL and no per-IP rate limiting, an unauthenticated attacker who enumerates more than `maxSize` distinct valid token IDs continuously evicts legitimate cache entries, forcing repeated DB queries through the 10-connection pool for all callers of `getCachedTokens`.

### Finding Description

**Exact code path:**

`rest/tokens.js` `getTokenInfoRequest` (lines 548–563): [1](#0-0) 

On every successful response with no timestamp filter, it calls: [2](#0-1) 

`rest/service/tokenService.js` `putTokenCache` (lines 121–129): [3](#0-2) 

The guard at line 123 (`if (tokenCache.has(tokenId)) { return; }`) only prevents re-writing an already-present key. It does **not** prevent LRU eviction: when `tokenCache.set()` is called for a new key and the cache is at `maxSize`, `quickLru` silently evicts the least-recently-used entry.

The cache is initialized with no TTL: [4](#0-3) 

Default `maxSize` is 100,000 (documented): [5](#0-4) 

`getCachedTokens` (lines 136–161) — called by `getTokenAccounts` for account/token-relationship queries — falls back to a DB query for every token ID not found in cache: [6](#0-5) 

The DB pool is capped at 10 connections by default: [7](#0-6) 

**No rate limiting** is applied to the token endpoint in `server.js`: [8](#0-7) 

**Root cause:** The combination of (a) unconditional `putTokenCache` on every no-filter `GET /api/v1/tokens/:tokenId` response, (b) a finite LRU cache with no TTL, and (c) no per-IP or global rate limiting on the REST API allows an attacker to continuously insert new entries and evict legitimate ones.

**Exploit flow:**
1. Attacker enumerates valid token IDs (Hedera token IDs are sequential integers: `0.0.1`, `0.0.2`, …; mainnet has millions).
2. Attacker sends `GET /api/v1/tokens/0.0.N` (no `?timestamp=`) for N = 1 … 100,001+.
3. Each response triggers `putTokenCache` → `tokenCache.set()`. Once the cache is full, each new insertion evicts the LRU entry.
4. Attacker cycles through a rotating window of token IDs larger than `maxSize`, keeping the cache in a permanent thrashing state.
5. Every call to `getCachedTokens` (from account token-relationship queries) finds its tokens evicted and issues a new DB query.
6. With only 10 DB connections and a 20-second statement timeout, sustained thrashing + concurrent legitimate traffic exhausts the pool, causing query timeouts across all DB-backed endpoints.

### Impact Explanation

Cache thrashing degrades the effectiveness of the token cache for all consumers of `getCachedTokens` (primarily the account token-relationships endpoint). Combined with the 10-connection DB pool limit, sustained thrashing amplifies DB load and can cause connection exhaustion, leading to 500-class errors or severe latency for all users of the mirror node REST API. The `getTokenInfoRequest` endpoint itself always hits the DB directly, so concurrent attacker requests also directly consume DB connections, compounding the effect.

### Likelihood Explanation

The attack requires no credentials, no special network position, and no prior knowledge beyond the fact that Hedera token IDs are sequential integers. Valid token IDs can be discovered trivially by iterating `GET /api/v1/tokens` (the list endpoint). The attack is fully automatable with a simple HTTP client loop and is repeatable indefinitely. No existing server-side control (authentication is optional, no rate limiter is registered in `server.js`) prevents it.

### Recommendation

1. **Add rate limiting** to the REST API at the application layer (e.g., `express-rate-limit`) for all `/api/v1/tokens/:tokenId` requests, keyed by IP.
2. **Add a TTL** to the `quickLru` cache (use `maxAge` option) so stale entries expire naturally and the cache self-heals after an attack.
3. **Consider a per-IP request budget** enforced in middleware before the route handler, independent of the optional Traefik middleware.
4. **Increase the DB connection pool** or add a circuit breaker so that cache-miss storms do not fully exhaust available connections.

### Proof of Concept

```bash
# Step 1: Discover valid token IDs (sequential, public endpoint)
# Token IDs on mainnet: 0.0.1 through 0.0.N (millions exist)

# Step 2: Fill and thrash the cache (requires >100,000 unique IDs)
# Run with high concurrency to also exhaust the 10-connection DB pool

for i in $(seq 1 200000); do
  curl -s "https://<mirror-node-host>/api/v1/tokens/0.0.$i" &
  # throttle parallelism to ~50 concurrent to avoid local limits
  if (( i % 50 == 0 )); then wait; fi
done

# Step 3: Observe that subsequent legitimate requests to account
# token-relationship endpoints (/api/v1/accounts/:id/tokens) now
# incur DB queries for every token (cache miss), and under load
# return 500 errors or timeouts due to DB pool exhaustion.
```

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
