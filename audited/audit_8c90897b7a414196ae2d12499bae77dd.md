### Title
Missing `return` After Block Number Mismatch Check Allows Attacker-Controlled Block Node to Corrupt Block Identity or Halt Assembly

### Summary
In `BlockAssembler.onEndOfBlock()` within `BlockNode.java`, after detecting a mismatch between the `BlockHeader` number and the `EndOfBlock` block number, `Utility.handleRecoverableError()` is called but execution is **not halted** (no `return` statement follows). In the default configuration (`haltOnError=false`), the attacker-supplied `blockNumber` from `EndOfBlock` is silently used as the block's filename, causing the block to be stored under the wrong number. When `haltOnError=true`, a `ParserException` is thrown, halting block assembly entirely and preventing all transactions in that block from being processed.

### Finding Description

**Exact code path:**

`importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, `BlockAssembler.onEndOfBlock()`, lines 239–272:

```java
void onEndOfBlock(final BlockEnd blockEnd) {
    final long blockNumber = blockEnd.getBlockNumber();   // attacker-controlled
    if (pending.isEmpty()) {
        Utility.handleRecoverableError(...);
        return;   // ← return present here
    }

    final var blockHeader = pending.getFirst().getFirst().getBlockHeader();
    if (blockHeader.getNumber() != blockNumber) {
        Utility.handleRecoverableError(
                "Block number mismatch in BlockHeader({}) and EndOfBlock({})",
                blockHeader.getNumber(),
                blockNumber);
        // ← NO return here — falls through unconditionally
    }

    // ... assembles block from pending ...

    final var filename = BlockFile.getFilename(blockNumber, false);  // uses attacker-controlled blockNumber
    blockStreamConsumer.accept(new BlockStream(block, null, filename, loadStart));
}
```

**Root cause:** The mismatch branch (lines 248–253) calls `handleRecoverableError` but omits a `return`, so execution always falls through to line 270 where `blockNumber` — sourced directly from the attacker's `EndOfBlock` message — is used to construct the block's filename and identity.

**`handleRecoverableError` behavior** (`Utility.java`, lines 220–231):
- `haltOnError=false` (default): logs the error and returns normally → execution continues, block stored under wrong number.
- `haltOnError=true`: throws `ParserException` → caught as generic `Exception` in `streamBlocks` (lines 162–164) → `onError()` called → `BlockStreamException` thrown → block assembly halted.

**Exploit flow:**
1. Attacker operates a block node endpoint configured in the mirror node (or performs MITM on an unencrypted gRPC channel).
2. Attacker sends a valid `BlockItemSet` containing a `BlockHeader` with `number = N`.
3. Attacker sends `EndOfBlock` with `blockNumber = M` where `M ≠ N`.
4. **Default mode:** Mismatch is logged; block content (correct) is assembled and dispatched as block `M`. Block `N` is never stored. The mirror node's `getNextBlockNumber()` (via `cutoverService.getLastRecordFile()`) will next request block `M+1`, permanently skipping block `N` and all its transactions.
5. **`haltOnError=true` mode:** `ParserException` propagates through `streamBlocks`, `onError()` increments the error counter; after `maxSubscribeAttempts` (default 3) consecutive triggers, the node is marked inactive. Block assembly for block `N` is halted; all transactions in it are not processed.

**Why existing checks are insufficient:** The check at lines 248–253 correctly detects the mismatch but the absence of a `return` after `handleRecoverableError` in the non-halt path means the corrupted `blockNumber` is unconditionally used on line 270. The `pending.isEmpty()` guard (lines 241–245) does have a `return`, making the asymmetry a clear oversight.

### Impact Explanation

**Default mode (primary impact):** Silent, persistent data corruption. The block is stored under an attacker-chosen number `M` instead of the true number `N`. Block `N`'s transactions are permanently skipped — the mirror node advances its sequence pointer past `N` without ever processing it. This breaks chain continuity and causes all downstream consumers (REST API, gRPC API, event streams) to serve incorrect or missing data for block `N`.

**`haltOnError=true` mode:** Complete denial of service for block `N`. All transactions in that block are never ingested. Repeated triggering across consecutive blocks can stall the entire importer pipeline.

Severity: **High** — silent data integrity corruption in the default configuration; DoS in the hardened configuration.

### Likelihood Explanation

**Preconditions:**
- Attacker must control a block node endpoint the mirror node subscribes to. This is achievable by: (a) registering a malicious block node on-chain (HIP-1081 allows permissionless registration), (b) MITM on a gRPC channel without TLS (`streamingApiRequireTls` defaults may not enforce TLS), or (c) compromising a legitimate block node operator.
- No cryptographic authentication of `EndOfBlock.blockNumber` against the `BlockHeader` is performed by the mirror node.

**Repeatability:** The attack is trivially repeatable — one malformed `EndOfBlock` per block is sufficient. An attacker can corrupt or skip every block indefinitely.

**Likelihood: Medium-High** — block node registration is permissionless per HIP-1081; TLS is optional per configuration.

### Recommendation

1. **Add a `return` after the mismatch `handleRecoverableError` call** (line 253) to abort processing when the block numbers disagree, mirroring the pattern used in the `pending.isEmpty()` branch:
   ```java
   if (blockHeader.getNumber() != blockNumber) {
       Utility.handleRecoverableError(
               "Block number mismatch in BlockHeader({}) and EndOfBlock({})",
               blockHeader.getNumber(),
               blockNumber);
       return;  // ← add this
   }
   ```
2. **Use `blockHeader.getNumber()` (not `blockEnd.getBlockNumber()`) for the filename** on line 270, so the authoritative source is always the `BlockHeader`, not the untrusted `EndOfBlock` field.
3. **Enforce TLS** for block node streaming connections to prevent MITM injection of malformed messages.
4. **Consider treating block number mismatch as a non-recoverable `BlockStreamException`** (like the `BLOCK_HEADER`-while-pending check on line 285) rather than a recoverable error, since it indicates a fundamentally malformed or malicious stream.

### Proof of Concept

**Precondition:** Mirror node is configured with a block node endpoint under attacker control (or attacker can MITM the gRPC stream).

**Steps:**

1. Mirror node requests block `N` from the attacker-controlled block node via `SubscribeStreamRequest(startBlockNumber=N)`.
2. Attacker's block node sends a valid `SubscribeStreamResponse` containing a `BlockItemSet` with a `BlockHeader(number=N)` followed by normal block items.
3. Attacker's block node sends `SubscribeStreamResponse` with `EndOfBlock(blockNumber=N+1000)` (or any `M ≠ N`).
4. `BlockAssembler.onEndOfBlock()` is invoked:
   - Line 248: `blockHeader.getNumber()` (= N) `!= blockNumber` (= N+1000) → true.
   - Line 249–252: `handleRecoverableError` logs the mismatch (default mode) and returns normally.
   - Line 253: **No `return`** — falls through.
   - Line 270: `BlockFile.getFilename(N+1000, false)` — filename uses attacker-supplied number.
   - Line 271: `blockStreamConsumer.accept(new BlockStream(block, null, "N+1000.blk", loadStart))`.
5. `AbstractBlockSource.onBlockStream()` processes and persists the block as block `N+1000`.
6. `cutoverService.getLastRecordFile()` now returns index `N+1000`; next requested block is `N+1001`.
7. Block `N` (and blocks `N+1` through `N+999`) are permanently skipped — their transactions are never ingested.

**Verification:** Query the mirror node REST API for transactions in block `N` — they will be absent. The `record_file` table will show a gap from the last legitimate block to `N+1000`.
