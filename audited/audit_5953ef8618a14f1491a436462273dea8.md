### Title
Unauthenticated Health Endpoint Triggers Unbounded Outbound HTTP Calls, Enabling Connection Pool Exhaustion

### Summary
`SubscriberHealthIndicator.health()` creates a fresh reactive chain on every invocation, unconditionally issuing a live `WebClient` HTTP call to `/network/stake` with no caching or deduplication. The `/actuator/health/cluster` endpoint that invokes this indicator is publicly exposed via the Kubernetes ingress with no authentication or rate limiting. An unprivileged attacker can flood this endpoint to exhaust the shared `WebClient` connection pool, degrading or halting all outbound HTTP operations in the monitor service.

### Finding Description

**Exact code path:**

`SubscriberHealthIndicator.health()` (lines 59–63) unconditionally calls `restNetworkStakeHealth()` on every invocation:

```java
// SubscriberHealthIndicator.java lines 59-63
public Mono<Health> health() {
    return restNetworkStakeHealth()
            .flatMap(health ->
                    health.getStatus() == Status.UP ? publishing().switchIfEmpty(subscribing()) : Mono.just(health))
            .doOnNext(this::recordHealthMetric);
}
```

`restNetworkStakeHealth()` (lines 93–121) issues a live `WebClient` call with a 5-second timeout on every invocation — no caching, no deduplication:

```java
// lines 93-107
private Mono<Health> restNetworkStakeHealth() {
    return restApiClient
            .getNetworkStakeStatusCode()   // live HTTP GET /network/stake every time
            ...
            .timeout(Duration.ofSeconds(5))
            ...
}
```

`RestApiClient.getNetworkStakeStatusCode()` (line 68) makes a raw `WebClient` exchange with no result sharing:

```java
public Mono<HttpStatusCode> getNetworkStakeStatusCode() {
    return webClientRestJava.get().uri("/network/stake").exchangeToMono(r -> Mono.just(r.statusCode()));
}
```

**Contrast with `ReleaseHealthIndicator`** (line 82), which explicitly caches its result:

```java
.cache(properties.getCacheExpiry(), Schedulers.newSingle("helmrelease-health-cache"));
```

**Root cause:** The `SubscriberHealthIndicator` was never given a `.cache()` operator or any equivalent memoization. Every subscriber to the `Mono<Health>` returned by `health()` triggers a new network round-trip.

**Exposure:** The Helm chart ingress for the monitor service explicitly routes `/actuator/health/cluster` to the public internet with no authentication and no rate-limiting middleware configured:

```yaml
# charts/hedera-mirror-monitor/values.yaml lines 91-99
ingress:
  enabled: true
  hosts:
    - host: ""
      paths: ["/actuator/health/cluster"]
```

No `inFlightReq`, `circuitBreaker`, or `retry` middleware is defined for the monitor ingress (unlike, e.g., the graphql chart which has `inFlightReq`).

**Exploit flow:**
1. Attacker sends N concurrent `GET /actuator/health/cluster` requests to the public ingress.
2. Spring Boot Actuator invokes `SubscriberHealthIndicator.health()` for each request.
3. Each invocation calls `restApiClient.getNetworkStakeStatusCode()`, opening a new connection from the shared Reactor Netty connection pool.
4. During a network partition (or even under normal load), each connection is held open for up to 5 seconds (the configured timeout).
5. With N concurrent requests, N connections are consumed simultaneously. Reactor Netty's default pool has a bounded `maxConnections` (typically 500 per remote host, but configurable and shared). Once exhausted, all subsequent `WebClient` calls — including those used for actual monitoring subscriptions — queue or fail.
6. The monitor service's core function (publishing/subscribing health checks) is disrupted.

**Why existing checks are insufficient:**
- The 5-second `.timeout()` limits per-connection duration but does not prevent pool exhaustion when the flood rate exceeds `poolSize / 5 req/s`.
- Spring Boot Actuator health endpoints are unauthenticated by default and no security configuration overriding this was found in the monitor module.
- No rate-limiting middleware is configured on the monitor ingress.
- No `flatMap` concurrency cap or `onBackpressureDrop` is applied to the health chain.

### Impact Explanation
The shared `WebClient` (and its underlying Reactor Netty connection pool) is used for all outbound REST calls in the monitor service, including the subscription health checks that are the service's primary purpose. Pool exhaustion causes `PoolAcquireTimeoutException` or indefinite queuing for all callers, effectively making the monitor service blind to the state of the Hedera network it is supposed to observe. This is a Denial-of-Service against the monitoring plane. Severity is **Medium–High** for the monitoring service; it does not directly affect the Hedera consensus network but can suppress alerting during an actual incident.

### Likelihood Explanation
The attack requires only the ability to send HTTP GET requests to the public ingress — no credentials, no special protocol knowledge, no prior access. It is trivially scriptable (`ab`, `wrk`, `hey`, or a simple shell loop). The 5-second timeout window means even a modest flood of ~100 req/s sustains ~500 concurrent open connections, which is at or above typical default pool limits. The attack is repeatable and stateless. Likelihood is **High** for any attacker who discovers the exposed endpoint.

### Recommendation
1. **Add `.cache()` to `SubscriberHealthIndicator`**: Wrap the result with a short TTL cache (e.g., 10–30 seconds), mirroring the pattern in `ReleaseHealthIndicator` (line 82). This ensures concurrent health-check requests share a single in-flight upstream call.
2. **Add rate limiting at the ingress**: Apply an `inFlightReq` or equivalent middleware to the monitor ingress for `/actuator/health/cluster`, similar to the graphql chart's `inFlightReq: amount: 5` configuration.
3. **Bound WebClient concurrency**: Apply `.flatMap(..., concurrency)` or use a dedicated `WebClient` with a smaller, isolated connection pool for health checks, preventing health-check traffic from starving subscription traffic.
4. **Consider authentication**: Restrict `/actuator/health/cluster` to internal cluster traffic only if external exposure is not required.

### Proof of Concept

**Preconditions:** Monitor service deployed with the default Helm chart; `/actuator/health/cluster` reachable at `$MONITOR_HOST`; network partition or high latency to the REST API backend (to maximize connection hold time).

```bash
# Step 1: Confirm the endpoint is unauthenticated and triggers an outbound call
curl -v http://$MONITOR_HOST/actuator/health/cluster

# Step 2: Flood the endpoint with concurrent requests
# Each request holds a /network/stake connection for up to 5 seconds
hey -n 10000 -c 600 http://$MONITOR_HOST/actuator/health/cluster

# Step 3: Observe pool exhaustion in monitor logs:
# reactor.netty.internal.shaded.reactor.pool.PoolAcquireTimeoutException
# or WebClientRequestException: Connection pool exhausted

# Step 4: Confirm monitoring is degraded — subscription health checks
# now fail or time out because the shared WebClient pool is saturated

    private Mono<Health> restNetworkStakeHealth() {
        return restApiClient
                .getNetworkStakeStatusCode()
                .flatMap(statusCode -> {
                    if (statusCode.is2xxSuccessful()) {
                        return UP;
                    }

                    var status = statusCode.is5xxServerError() ? Status.DOWN : Status.UNKNOWN;
                    var statusMessage =
                            String.format("Network stake status is %s with status code %s", status, statusCode.value());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                })
                .timeout(Duration.ofSeconds(5))
                .onErrorResume(e -> {
                    var status = Status.UNKNOWN;
                    // Connection issue can be caused by database being down, since the rest API service will become
                    // unavailable eventually
                    var rootCause = ExceptionUtils.getRootCause(e);
                    if (rootCause instanceof ConnectException || rootCause instanceof TimeoutException) {
                        status = Status.DOWN;
                    }

                    var statusMessage =
                            String.format("Network stake status is %s with error: %s", status, e.getMessage());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                });
```
