### Title
Raw `IllegalArgumentException` Message Leaked to Unprivileged Users via `doProcessCall` Exception Handler

### Summary
In `ContractCallService.doProcessCall`, any `IllegalArgumentException` thrown during EVM execution is caught and its raw `.getMessage()` string is placed directly into the `detail` field of a `MirrorEvmTransactionException`. The `GenericControllerAdvice` then unconditionally serializes `e.getDetail()` into the HTTP response body, exposing internal implementation details — including state IDs, class names, and potentially malformed entity data — to any unprivileged caller.

### Finding Description

**Exact code path:**

`ContractCallService.java`, lines 120–121: [1](#0-0) 

```java
} catch (IllegalStateException | IllegalArgumentException e) {
    throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
}
```

The two-argument constructor used here is: [2](#0-1) 

```java
public MirrorEvmTransactionException(final String detail, final String hexData) {
    this(null, detail, hexData, null);   // message=null, detail=e.getMessage()
}
```

`e.getMessage()` is stored as the `detail` field; `super(null)` means `getMessage()` returns `null` and `getResponseCode()` returns `null`.

In `GenericControllerAdvice.mirrorEvmTransactionError`: [3](#0-2) 

Because `getResponseCode()` is `null`, `SERVER_RESPONSE_CODES.contains(null)` is `false`, so the `BAD_REQUEST` branch is always taken and `e.getDetail()` — the raw exception message — is written into the JSON response body:

```json
{ "_status": { "messages": [{ "message": null, "detail": "<raw IAE message>", "data": "" }] } }
```

**Sources of `IllegalArgumentException` in the EVM execution path:**

`MapReadableStates` throws messages like `"Unknown k/v state id: " + stateId` and `"State is not an instance of ReadableKVState: " + stateId`: [4](#0-3) 

`MapWritableStates` throws analogous messages: [5](#0-4) 

`Account.java` (generated protobuf) and `ScheduleReadableKVState` also throw `IllegalArgumentException` with field-level detail during entity deserialization. [6](#0-5) 

**Root cause:** No sanitization or redaction is applied to the exception message before it is placed in `detail` and returned to the caller.

**Why existing checks fail:** The `handleExceptionInternal` override in `GenericControllerAdvice` does suppress 5xx details (`!statusCode.is5xxServerError() ? ex.getMessage() : StringUtils.EMPTY`), but `MirrorEvmTransactionException` is handled by its own dedicated `@ExceptionHandler` (`mirrorEvmTransactionError`) which has no such suppression and always returns `e.getDetail()` regardless of content. [7](#0-6) 

### Impact Explanation
An attacker receives internal implementation strings such as internal state integer IDs, Java class names (`ReadableKVState`, `WritableSingletonState`), and — in the malformed-entity scenario — protobuf field names and raw byte representations of corrupted DB records. This constitutes an information-disclosure vulnerability that aids reconnaissance (confirming internal architecture, state key numbering, and data layout) and can expose partial entity data from the mirror node database. Severity: **Medium** (information disclosure, no direct data manipulation).

### Likelihood Explanation
Any unauthenticated user can issue a `POST /api/v1/contracts/call` with a historical block parameter. The trigger requires either: (a) a historical block whose state map is missing a state key the EVM tries to access (possible with edge-case block numbers near chain genesis or during state migrations), or (b) a malformed entity row in the DB (possible due to importer bugs or data corruption). Condition (a) is user-triggerable without any DB precondition by probing early block numbers. Condition (b) is opportunistic. Both are realistic in a production deployment.

### Recommendation
1. In the `catch` block at line 120–121, replace the raw message with a sanitized constant or a mapped `ResponseCodeEnum` value:
   ```java
   } catch (IllegalStateException | IllegalArgumentException e) {
       log.warn("Internal error during EVM execution", e);
       throw new MirrorEvmTransactionException(
           ResponseCodeEnum.CONTRACT_EXECUTION_EXCEPTION, StringUtils.EMPTY, EMPTY);
   }
   ```
2. Alternatively, add a sanitization step in `mirrorEvmTransactionError` that clears `detail` when `responseCode` is `null` (i.e., when the exception originated from this catch block).
3. Audit all other `MirrorEvmTransactionException(String detail, String hexData)` call sites for the same pattern.

### Proof of Concept
1. Identify a historical block number near genesis (e.g., block 1) where the EVM state map may be missing entries.
2. Send:
   ```http
   POST /api/v1/contracts/call
   Content-Type: application/json

   {
     "block": "1",
     "to": "0x0000000000000000000000000000000000000167",
     "data": "0x<any precompile selector>",
     "gas": 300000
   }
   ```
3. Observe the `400 Bad Request` response body:
   ```json
   { "_status": { "messages": [{ "message": null, "detail": "Unknown k/v state id: 42", "data": "" }] } }
   ```
4. The `detail` field exposes the internal state integer ID, confirming the leakage path. For a malformed entity, the message would contain protobuf field-level details from the deserialization failure.
