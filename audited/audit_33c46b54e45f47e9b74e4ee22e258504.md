### Title
Unbounded Synchronous ASN.1 DER Parsing in `parsePublicKey()` Without Rate Limiting Enables CPU Griefing on Node.js Event Loop

### Summary
The `getBalances()` handler in `rest/balances.js` calls `utils.parsePublicKeyQueryParam()` on every request, which unconditionally invokes `ed25519.derToEd25519()` via `parsePublicKey()`. That function performs synchronous `asn1js.fromBER()` + `toJSON()` ASN.1 parsing on every inbound DER-encoded public key with no rate limiting, no result caching, and no pre-check to skip parsing for non-DER inputs. Because Node.js runs on a single-threaded event loop, accumulated synchronous CPU work across high-frequency requests delays all concurrent I/O callbacks, degrading throughput for every user of the service.

### Finding Description

**Exact code path:**

`rest/balances.js:94` — `getBalances()` calls `utils.parsePublicKeyQueryParam(req.query, 'public_key')` on every request before any async work. [1](#0-0) 

`rest/utils.js:683-691` — `parsePublicKeyQueryParam()` calls `parsePublicKey(value)` for each supplied value. [2](#0-1) 

`rest/utils.js:677-681` — `parsePublicKey()` calls `ed25519.derToEd25519(publicKeyNoPrefix)` unconditionally, with no length pre-check, no memoization, and no short-circuit for non-DER inputs. [3](#0-2) 

`rest/ed25519.js:8-31` — `derToEd25519()` performs: (1) `Buffer.from(der, 'hex')` hex decode, (2) `asn1js.fromBER(buf)` — a pure-JavaScript BER/DER parser that walks the entire TLV tree, (3) `asn.result.toJSON()` — a full recursive JSON serialization of the parsed ASN.1 tree. All three steps are synchronous and block the event loop. [4](#0-3) 

**Root cause:** The assumption is that DER parsing is cheap enough to run inline on every request. There is no caching of the parsed result, no pre-filter to skip parsing for keys that are already the correct 64-char raw hex length, and no rate limiting on the REST API.

**Failed check — no rate limiting:** `rest/server.js` registers no rate-limiting middleware (no `express-rate-limit` or equivalent). The only throttling in the repository is in the separate Java `web3` service and is not applied to the Node.js REST API. [5](#0-4) 

**Validation does not prevent the attack:** `filterValidityChecks` for `ACCOUNT_PUBLICKEY` calls `isValidPublicKeyQuery`, which accepts 88-character hex strings (valid DER-encoded Ed25519 keys). A request carrying a well-formed 88-char DER key passes validation and then reaches `derToEd25519` — the attacker does not need to send malformed input. [6](#0-5) 

### Impact Explanation
Node.js processes all JavaScript synchronously on one thread. Each call to `asn1js.fromBER()` + `toJSON()` on a 44-byte buffer takes on the order of 10–50 µs of CPU time. At sustained 5,000–10,000 requests/second (easily achievable from a single machine with keep-alive connections), this accumulates to 50–500 ms of blocked event-loop time per second. During those blocked intervals, all pending I/O callbacks — including responses to legitimate users — are queued and delayed. The result is increased latency and reduced throughput for all concurrent users of the `/api/v1/balances` (and any other endpoint that calls `parsePublicKeyQueryParam`) without any economic cost to the attacker.

### Likelihood Explanation
The attack requires no authentication, no special account, and no on-chain action. Any internet-accessible deployment is reachable. A single attacker with a modest machine and a simple HTTP flood tool (e.g., `wrk`, `ab`, or a short script) can sustain the required request rate. The endpoint is publicly documented and the `account.publickey` parameter is explicitly listed in the OpenAPI spec. The attack is repeatable indefinitely.

### Recommendation
1. **Short-circuit non-DER inputs in `parsePublicKey`:** Before calling `derToEd25519`, check whether the key length equals 64 hex chars (32 bytes, raw Ed25519). If so, skip DER parsing entirely.
2. **Memoize/cache parsed keys:** Use a small bounded LRU cache keyed on the raw hex string so repeated identical keys are not re-parsed.
3. **Add rate limiting to the REST API:** Apply `express-rate-limit` (or equivalent) globally or at least on parameter-heavy endpoints such as `/balances`, `/accounts`, and `/tokens`.
4. **Move DER parsing off the hot path:** Validate that the input matches the exact DER prefix bytes (`302a300506032b6570032100`) before invoking the full `asn1js` parser.

### Proof of Concept
```bash
# Valid 88-char Ed25519 DER key (passes isValidPublicKeyQuery)
DER="302a300506032b65700321007a3c5477bdf4a63742647d7cfc4544acc1899d07141caf4cd9fea2f75b28a5cc"

# Flood /api/v1/balances with DER-encoded public key at high rate
# Using wrk (single machine, 10 threads, 100 connections, 60 seconds)
wrk -t10 -c100 -d60s \
  "http://<mirror-node-host>/api/v1/balances?account.publickey=${DER}"

# Expected result: event loop latency increases for all concurrent requests;
# p99 latency of unrelated endpoints (/api/v1/transactions, etc.) degrades
# proportionally to request rate, with no authentication required.
```

```javascript
const derToEd25519 = function (der) {
  try {
    const buf = new Uint8Array(Buffer.from(der, 'hex')).buffer;
    const asn = asn1js.fromBER(buf);
    if (asn.offset === -1) {
      return null; // Not a valid DER/BER format
    }

    const asn1Result = asn.result.toJSON();

    // Check if it is a ED25519 key
    if (asn1Result.valueBlock.value.length < 1 || asn1Result.valueBlock.value[0].valueBlock.value.length < 1) {
      return null;
    }
    const {valueBlock} = asn1Result.valueBlock.value[0].valueBlock.value[0];
    if (valueBlock.blockName === BLOCK_NAME && valueBlock.value === ID) {
      const ed25519Key = asn1Result.valueBlock.value[1].valueBlock.valueHex;
      return ed25519Key.toLowerCase();
    }
    return null;
  } catch (err) {
    return null;
  }
};
```
