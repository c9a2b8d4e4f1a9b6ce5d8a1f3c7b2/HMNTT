### Title
Unbounded gRPC Subscriptions Trigger Synchronized Database Query Flood via `safetyCheck` Flux

### Summary
`TopicMessageServiceImpl.subscribeTopic()` creates a `safetyCheck` Flux that fires an unthrottled database query exactly 1 second after each subscription starts if no data has arrived. Because there is no global limit on the total number of concurrent subscriptions (only a per-connection cap of 5), an unprivileged attacker can open arbitrarily many TCP connections and saturate the mirror node's database connection pool with a synchronized burst of gap-recovery queries, starving the importer's write path and degrading mirror node availability.

### Finding Description

`isComplete()` always returns `false` when no `endTime` is set — lines 203–205: [3](#0-2) 

**Root cause — failed assumption:** The design assumes the number of concurrent subscriptions is bounded by infrastructure (load balancer, connection limits). The only server-side cap is `maxConcurrentCallsPerConnection = 5` (per TCP connection), enforced in `NettyProperties` and applied in `GrpcConfiguration`: [4](#0-3) [5](#0-4) 

There is **no global subscription cap** and **no per-IP connection limit** in the gRPC service code. The `subscriberCount` AtomicLong is a metrics gauge only — it never rejects new subscriptions: [6](#0-5) 

**Exploit flow:**

1. Attacker opens *N* TCP connections to port 5600.
2. On each connection, attacker opens 5 `subscribeTopic` streams targeting any existing, quiet topic (no `endTime` set, so `isComplete()` always returns `false`).
3. Total active subscriptions = 5N, each with its own `safetyCheck` timer.
4. At t = 1 s, all 5N timers fire simultaneously. Each calls `topicMessageRetriever.retrieve(gapFilter, false)` — the **unthrottled** path (`throttled=false`), which polls up to `unthrottled.maxPolls = 12` times at `unthrottled.pollingFrequency = 20 ms` with `unthrottled.maxPageSize = 5000` rows per poll. [7](#0-6) 
5. This creates a burst of up to 5N × 12 = 60N database queries in a ~240 ms window.
6. Attacker disconnects and reconnects to repeat the burst indefinitely.

### Impact Explanation

The mirror node gRPC service and the importer share the same PostgreSQL database. A flood of 60N simultaneous unthrottled queries exhausts the HikariCP connection pool (monitored via `GrpcHighDBConnections` alert at 75% utilization): [8](#0-7) 

When the pool is saturated, the importer's write transactions queue behind the gRPC read queries, causing the mirror node to fall behind in ingesting confirmed transactions. Clients relying on the mirror node for transaction confirmation status (via gRPC subscriptions or REST) receive stale or no data. The mirror node itself becomes unavailable for all subscribers. This is a denial-of-service against the mirror node service layer; it does not directly affect Hedera consensus nodes, but it does degrade the observable confirmation pipeline for applications depending on the mirror node.

### Likelihood Explanation

**Preconditions:** None beyond network access to port 5600. No authentication is required for `subscribeTopic`. Any existing topic ID (obtainable from the public REST API) suffices.

**Feasibility:** Opening thousands of HTTP/2 connections is trivial from a single machine or a small botnet. Each connection costs minimal attacker resources (gRPC over HTTP/2 multiplexes streams). The 1-second synchronization window is deterministic and reliable.

**Repeatability:** The attacker can cycle connections continuously, producing a new burst every ~1 second with no cooldown enforced by the server.

### Recommendation

1. **Add a global subscription limit:** Reject new subscriptions when `subscriberCount` exceeds a configurable threshold (e.g., 10 000), returning `RESOURCE_EXHAUSTED`.
2. **Add per-IP connection/subscription limits** at the Netty or load-balancer layer.
3. **Make `safetyCheck` throttled:** Pass `throttled=true` to `topicMessageRetriever.retrieve()` in the safety-check branch so it uses the slower polling path (`maxPageSize=1000`, `pollingFrequency=2s`) instead of the unthrottled path.
4. **Introduce a global DB query concurrency semaphore** in `TopicMessageRetriever` to cap simultaneous in-flight queries regardless of subscription count.
5. **Enforce the GCP `maxRatePerEndpoint: 250`** gateway policy in all deployment environments, not just GCP.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
TOPIC_ID = 0  # any existing topic shard=0, realm=0, num=<known>
N_CONNECTIONS = 500  # 500 connections × 5 streams = 2500 subscriptions

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_ID)
        # no consensusEndTime → isComplete() always False
    )
    streams = []
    for _ in range(5):  # maxConcurrentCallsPerConnection
        streams.append(stub.subscribeTopic(query))
    # Hold streams open; safetyCheck fires at t=1s for all 2500 subscriptions
    import time; time.sleep(10)

threads = [threading.Thread(target=flood, args=(i,)) for i in range(N_CONNECTIONS)]
for t in threads: t.start()
# At t≈1s: 2500 × 12 = 30,000 unthrottled DB queries fire simultaneously
for t in threads: t.join()
```
