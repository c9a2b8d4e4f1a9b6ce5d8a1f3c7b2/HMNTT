### Title
Deleted Account Numeric ID Bypasses Deletion Check in `getEncodedId()`, Returning Stale Token Allowance Data

### Summary
In `rest/service/entityService.js`, `getEncodedId()` handles numeric account IDs via a pure mathematical encoding path that never queries the database, while alias and EVM address paths both explicitly filter out deleted entities via SQL `WHERE deleted <> true`. An unprivileged attacker can supply a valid numeric ID of a deleted account to `/accounts/:id/allowances/tokens`, causing the API to return stale `token_allowance` records that off-chain systems or smart contract oracles may act upon incorrectly.

### Finding Description

**Exact code path:**

In `rest/service/entityService.js`, `getEncodedId()` at lines 120–124:

```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()                                          // ← numeric path: pure math, NO DB query
    : await this.getEntityIdFromEvmAddress(entityId, requireResult);  // ← EVM path: checks deleted <> true
}
``` [1](#0-0) 

For alias inputs, `entityFromAliasQuery` enforces:
```sql
WHERE coalesce(deleted, false) <> true AND alias = $1
``` [2](#0-1) 

For EVM address inputs, `entityFromEvmAddressQuery` enforces:
```sql
WHERE deleted <> true AND evm_address = $1
``` [3](#0-2) 

For numeric IDs, `entityId.getEncodedId()` is a pure bitwise/arithmetic encoding — **no database round-trip, no deletion check whatsoever**.

This asymmetry is confirmed by the test suite itself, where the test named `"EntityService.getEncodedId - No match"` asserts that `getEncodedId` resolves successfully even when no entity exists in the database: [4](#0-3) 

In `tokenAllowanceController.js`, `getAccountTokenAllowances` calls `getEncodedId` and immediately passes the result to `TokenAllowanceService.getAccountTokenAllowances` with no intervening deletion check: [5](#0-4) 

The `token_allowance` table is not automatically purged on account deletion — it is only updated by explicit `CryptoDeleteAllowance` or zero-amount `CryptoApproveAllowance` transactions. A deleted account's allowance rows persist in the table. [6](#0-5) 

**Root cause:** The numeric ID branch of `getEncodedId()` was designed for performance (avoiding a DB lookup), but this skips the deletion guard that all other identifier types enforce, creating an inconsistent trust boundary.

### Impact Explanation
Any off-chain system (DeFi backend, smart contract oracle, automated market maker relay) that queries `/accounts/{numericId}/allowances/tokens` to verify spending authority before submitting an on-chain transaction will receive a `200 OK` with non-empty allowance data for a deleted account. This can cause the off-chain component to incorrectly believe a spending approval is still valid, leading to wasted gas on doomed transactions, incorrect UI state, or — in oracle-driven architectures — triggering contract logic branches that should not execute. The question's own severity framing ("no concrete funds at direct risk") is accurate: consensus nodes will reject transactions referencing a deleted owner, so direct fund loss is not the primary risk. The risk is incorrect protocol-level behavior and misleading state.

### Likelihood Explanation
No privileges are required. Any user who knows or can enumerate a numeric account ID (trivially discoverable via the mirror node's own `/accounts` endpoint) can trigger this. Account deletion is a normal network operation. The attack is fully repeatable and requires only a single HTTP GET request with a numeric ID parameter.

### Recommendation
In the numeric ID branch of `getEncodedId()`, add a database existence-and-deletion check analogous to the alias/EVM address paths. The existing `entityExistenceQuery` (`SELECT type FROM entity WHERE id = $1`) should be extended to also check `deleted`:

```sql
SELECT type FROM entity WHERE id = $1 AND coalesce(deleted, false) <> true
```

Replace the bare `entityId.getEncodedId()` return with a call that validates against this query and throws `NotFoundError` when the entity is absent or deleted, consistent with the behavior of the alias and EVM address paths. [7](#0-6) 

### Proof of Concept

**Preconditions:**
1. Account `0.0.12345` exists on the network and has granted a token allowance to spender `0.0.99999` for token `0.0.55555`.
2. The mirror node has ingested the allowance into the `token_allowance` table.
3. Account `0.0.12345` is subsequently deleted via a `CryptoDelete` transaction (sets `entity.deleted = true`).
4. No explicit `CryptoDeleteAllowance` or zero-amount `CryptoApproveAllowance` was submitted, so the row in `token_allowance` remains.

**Trigger:**
```
GET /api/v1/accounts/12345/allowances/tokens
```

**Expected (correct) behavior:** `404 Not Found` — consistent with alias/EVM address behavior for deleted accounts.

**Actual behavior:** `200 OK` with:
```json
{
  "allowances": [
    {
      "owner": "0.0.12345",
      "spender": "0.0.99999",
      "token_id": "0.0.55555",
      "amount": 1000,
      "amount_granted": 1000
    }
  ]
}
```

The same request using the account's alias or EVM address would correctly return `404`, demonstrating the inconsistency.
