### Title
Unauthenticated High-Rate Requests with Degenerate `token.id=lt:1` Filter Cause Guaranteed Double DB Load in `getTokenRelationships()`

### Summary
Any unauthenticated external user can send requests to `GET /api/v1/accounts/{validId}/tokens?token.id=lt:1` at an unlimited rate. Each request unconditionally triggers two sequential database queries — one for account existence validation and one for the token relationship scan — with the second query guaranteed to return zero rows due to the semantically degenerate filter. Because the REST server has no application-level rate limiting, this can be repeated at arbitrary frequency to impose sustained, wasted DB load.

### Finding Description
**Code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 66–92:

```js
const accountId = await EntityService.getEncodedId(...);   // line 67 – parses numeric ID, no DB hit for numeric form
const isValidAccount = await EntityService.isValidAccount(accountId); // line 68 – ALWAYS hits DB
if (!isValidAccount) { throw new NotFoundError(); }
const filters = utils.buildAndValidateFilters(req.query, acceptedTokenParameters); // line 72
const query = this.extractTokensRelationshipQuery(filters, accountId);             // line 73
const tokenRelationships = await TokenService.getTokenAccounts(query);             // line 74 – ALWAYS hits DB
```

**DB query 1 — `isValidAccount`** (`rest/service/entityService.js`, lines 60–62):
```sql
SELECT type FROM entity WHERE id = $1
```
Executes unconditionally for every request that reaches line 68.

**DB query 2 — `getTokenAccounts`** (`rest/service/tokenService.js`, lines 46–70):
```sql
SELECT ta.automatic_association, ta.balance, ta.created_timestamp,
       ta.freeze_status, ta.kyc_status, ta.token_id
FROM token_account ta
WHERE ta.account_id = $1
  AND ta.associated = true
  AND ta.token_id < $3          -- injected by token.id=lt:1
ORDER BY ta.token_id asc
LIMIT $2
```
With `$3 = 1`, this predicate (`token_id < 1`) can never be satisfied because Hedera token IDs are ≥ 1. The DB engine still plans and executes the full query against `token_account`.

**Why validation is insufficient:**

- `buildAndValidateFilters` calls `EntityId.isValidEntityId(1)` for the value `1` — this returns `true` because `1` is a syntactically valid entity ID. The `lt` operator is also accepted (`ne` is the only rejected operator per line 31–33 of `tokenController.js`). No semantic check exists to reject filters that are provably empty.
- `isValidAccount` has no caching layer; it issues a live DB query on every call.
- `rest/server.js` registers no rate-limiting middleware for the REST API. A `grep` across all `rest/**/*.js` files for `rateLimit`, `rate-limit`, `rateLimiter`, or `express-rate` returns zero matches. The `authHandler` middleware only sets a custom response-size limit for authenticated users; it does not throttle unauthenticated requests.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate limiter exists only in the `web3` module and is not wired into the REST server.

### Impact Explanation
Every request to this endpoint with a valid account ID and `token.id=lt:1` wastes two DB round-trips with zero useful output. At high request rates this degrades the shared PostgreSQL connection pool and increases query planner overhead on the `entity` and `token_account` tables, potentially slowing legitimate queries. No data is leaked or modified; the impact is purely resource exhaustion / service degradation (griefing).

### Likelihood Explanation
Preconditions are trivial: the attacker needs only a publicly known valid account ID (e.g., `0.0.98`, the treasury account, which is always present) and the ability to send HTTP GET requests. No credentials, tokens, or on-chain activity are required. The attack is fully scriptable with a single `curl` loop or any HTTP load tool. There is no application-level control to detect or block it.

### Recommendation
1. **Add application-level rate limiting** to the REST server (e.g., `express-rate-limit` keyed on IP) before the route handlers in `rest/server.js`, covering all `/api/v1/accounts/:id/tokens` requests.
2. **Cache `isValidAccount` results** with a short TTL (e.g., 30 s) so repeated requests for the same account ID do not each issue a live DB query.
3. **Reject semantically degenerate range filters** — if `token.id=lt:X` where `X ≤ 1` (the minimum possible token ID), return HTTP 400 immediately without touching the DB.
4. Consider moving the `isValidAccount` check after filter validation so that invalid filters short-circuit before any DB access.

### Proof of Concept
```bash
# 0.0.98 is the Hedera treasury account — always valid
ACCOUNT="0.0.98"
BASE="https://<mirror-node-host>/api/v1"

# Single request — observe two DB queries in server logs
curl -s "$BASE/accounts/$ACCOUNT/tokens?token.id=lt:1"
# Response: {"tokens":[],"links":{"next":null}}  (200 OK, empty, but DB was hit twice)

# High-rate loop — no authentication, no rate limit enforced
while true; do
  curl -s "$BASE/accounts/$ACCOUNT/tokens?token.id=lt:1" -o /dev/null &
done
```
Each iteration causes:
1. `SELECT type FROM entity WHERE id = <encoded 98>` — always returns a row (account exists).
2. `SELECT … FROM token_account ta WHERE ta.account_id = <encoded 98> AND ta.associated = true AND ta.token_id < 1 ORDER BY ta.token_id asc LIMIT 25` — always returns 0 rows, full scan wasted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
