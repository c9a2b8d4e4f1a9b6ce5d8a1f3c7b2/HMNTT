### Title
Missing `@Cacheable` on `findActiveByEvmAddressAndTimestamp` Enables Unauthenticated DB Exhaustion via Historical Contract Calls

### Summary
`findActiveByEvmAddressAndTimestamp()` in `EntityRepository.java` lacks a `@Cacheable` annotation, unlike every other current-state lookup method in the same interface. An unauthenticated attacker can flood the public `/api/v1/contracts/call` endpoint with historical block requests targeting a non-existent EVM address, forcing a multi-table CTE+UNION SQL query to execute on every single call with no result caching, up to the global 500 RPS throttle limit. Because the throttle is global (not per-IP), a single attacker saturating it degrades service for all legitimate users.

### Finding Description
**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85, method `findActiveByEvmAddressAndTimestamp(byte[] evmAddress, long blockTimestamp)`.

**Root cause:** Every non-historical entity lookup in the same interface carries a `@Cacheable` annotation:
- `findByIdAndDeletedIsFalse` — `@Caching` with two `@Cacheable` entries (lines 20–30)
- `findByEvmAddressAndDeletedIsFalse` — `@Cacheable(cacheNames = CACHE_NAME_EVM_ADDRESS, ...)` (lines 32–37)
- `findByEvmAddressOrAliasAndDeletedIsFalse` — `@Cacheable(cacheNames = CACHE_NAME_ALIAS, ...)` (lines 39–49)

`findActiveByEvmAddressAndTimestamp` (lines 59–85) has only `@Query` — no `@Cacheable`. The entity cache is configured with `expireAfterWrite=1s,maximumSize=10000` (docs/configuration.md line 691), so even a 1-second cache would absorb repeated identical calls. Without it, every invocation hits the database.

**Query execution path for a non-existent address:**
The CTE at lines 60–66 runs:
```sql
select id from entity where evm_address = ?1 and created_timestamp <= ?2 order by created_timestamp desc limit 1
```
For a non-existent address this returns empty. Both UNION branches (lines 67–81) then evaluate `e.id = (select id from entity_cte)` and `eh.id = (select id from entity_cte)`, where the subquery returns NULL. While PostgreSQL can short-circuit the NULL comparison, the CTE itself still executes an index seek on `entity.evm_address` on every call, and the query plan is re-evaluated each time.

**Call path from public API to the vulnerable method:**
1. `POST /api/v1/contracts/call` with `block` set to any historical block number → `ContractController.call()` (ContractController.java line 38)
2. `ContractCallContext.useHistorical()` returns `true` (ContractCallContext.java line 103–105)
3. `CommonEntityAccessor.getEntityByEvmAddressTimestamp()` (CommonEntityAccessor.java line 80–83) calls `entityRepository.findActiveByEvmAddressAndTimestamp(addressBytes, t)`
4. No cache intercepts the call; the SQL executes unconditionally.

**Why existing checks are insufficient:**
The throttle (`ThrottleProperties.java` line 35) sets `requestsPerSecond = 500` as a **global** bucket shared across all clients (`ThrottleManagerImpl.java` lines 38–42). A single attacker consuming all 500 tokens/second starves legitimate users. The gas bucket (`gasPerSecond = 7_500_000_000L`, line 26) is bypassed entirely for requests with `gas ≤ 10_000` because `scaleGas()` returns 0 for values ≤ `GAS_SCALE_FACTOR` (ThrottleProperties.java lines 42–46), meaning such requests consume zero gas tokens and only the RPS bucket limits them.

### Impact Explanation
Up to 500 DB queries per second can be forced by a single unauthenticated client, each executing a CTE + UNION query across the `entity` and `entity_history` tables. Under sustained load this increases DB CPU and I/O, potentially causing query latency to rise for all other operations sharing the same database connection pool (`db.statementTimeout = 3000ms`, docs line 702). Because the RPS throttle is global, the attacker simultaneously denies capacity to legitimate callers. No funds are at risk; the impact is service degradation (griefing).

### Likelihood Explanation
The attack requires no credentials, no on-chain assets, and no special knowledge beyond the public OpenAPI spec (openapi.yml line 461–480). Any attacker with a script can sustain 500 RPS indefinitely. The varying `blockTimestamp` parameter (one per historical block) ensures no future caching addition with a composite key would help for the non-existent-address case, since `unless = "#result == null"` (the pattern used on all other methods) would still skip caching empty results.

### Recommendation
1. **Short-term:** Add per-IP rate limiting in addition to the global RPS bucket so a single client cannot consume the entire global allowance.
2. **Medium-term:** Add `@Cacheable` with a short TTL and a composite cache key (e.g., `@spelHelper.hashCode(#evmAddress) + '_' + #blockTimestamp`) to absorb repeated identical calls. Use `unless = "#result == null"` consistent with the rest of the interface, and accept that non-existent-address misses will still hit the DB — but repeated calls with the same `(address, timestamp)` pair will be served from cache.
3. **Long-term:** Consider an early-exit guard: if the `entity_cte` returns no rows, skip executing both UNION branches entirely (e.g., restructure the query to check CTE emptiness before the UNION, or add an application-layer check before invoking the repository).

### Proof of Concept
```bash
# Identify any valid historical block number N from the network
# Send 500 requests/second with a non-existent EVM address and varying block numbers
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef\",
         \"data\":\"0x\",
         \"gas\":10000,
         \"block\":\"$i\"}" &
done
wait
```

Each request with `gas=10000` consumes 0 gas-bucket tokens (due to `scaleGas` returning 0 for values ≤ 10,000) and only 1 RPS token. Each triggers `findActiveByEvmAddressAndTimestamp` with a different `blockTimestamp` derived from block `$i`, executing the CTE+UNION query unconditionally with no cache hit.
