### Title
Unauthenticated, Unbounded `getNodes()` gRPC Endpoint Enables Botnet-Driven DB Connection Pool Exhaustion

### Summary
The `getNodes()` handler in `NetworkController.java` accepts unlimited concurrent connections from any unauthenticated caller with no per-IP rate limiting, no total connection cap, and no gRPC-level throttle. Each call drives repeated paginated database queries through `NetworkServiceImpl`, and because the Netty server is configured only with a per-connection call limit (not a total connection limit), a botnet can collectively saturate the shared HikariCP database connection pool, starving legitimate `subscribeTopic` (gossip monitoring) calls that share the same pool.

### Finding Description
**Code path:**

`NetworkController.getNodes()` — `grpc/src/main/java/org/hiero/mirror/grpc/controller/NetworkController.java`, lines 33–43: [1](#0-0) 

No authentication check, no rate-limit guard, no per-IP connection check is present before the call is dispatched.

`NetworkServiceImpl.getNodes()` — `grpc/src/main/java/org/hiero/mirror/grpc/service/NetworkServiceImpl.java`, lines 55–77: [2](#0-1) 

Each call pages through the address book using `transactionOperations.execute()` (line 80), acquiring a HikariCP DB connection per page, with a 250 ms `pageDelay` between pages. With `pageSize=10` and a typical address book of ~30 nodes, each `getNodes()` call makes ≥3 sequential DB queries while holding a connection.

**Root cause — failed assumption in server configuration:**

`GrpcConfiguration.java` lines 28–35 configures only `maxConcurrentCallsPerConnection` (default 5): [3](#0-2) 

`NettyProperties.java` line 14: [4](#0-3) 

`maxConcurrentCallsPerConnection` limits streams *per TCP connection*, not the total number of accepted connections. Netty's default `maxConnections` is `Integer.MAX_VALUE`. A botnet with N distinct IPs each opening one connection can therefore drive 5N concurrent `getNodes()` calls simultaneously.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection=5`: scoped per-connection; N connections = 5N concurrent calls.
- GCP `maxRatePerEndpoint: 250` (`charts/hedera-mirror-grpc/values.yaml` line 69): a global throughput cap per backend pod, not a per-source-IP limit; only active when `gateway.gcp.enabled: true`. [5](#0-4) 
- Traefik middleware for gRPC (`values.yaml` lines 156–161) contains only `circuitBreaker` and `retry` — no `inFlightReq` per-IP limiter. Compare with the graphql chart which explicitly adds `inFlightReq` with `ipStrategy`: [6](#0-5) 
- No gRPC-level `ThrottleManager` exists; the bucket4j throttle is implemented only in the `web3` module, not in `grpc`.

### Impact Explanation
Each concurrent `getNodes()` call holds a HikariCP connection for the duration of its paged DB queries. When the pool is exhausted, all gRPC calls sharing it — including `ConsensusService.subscribeTopic` — block waiting for a connection or fail with a timeout. This directly disrupts gossip transaction monitoring. The Prometheus alert `GrpcHighDBConnections` (fires at 75% pool utilization) confirms the DB connection pool is a recognized finite resource: [7](#0-6) 

Severity: **High** — unauthenticated, no-privilege attacker can cause sustained service unavailability for all gRPC consumers.

### Likelihood Explanation
Preconditions: none beyond network access to port 5600. The endpoint is publicly documented and reachable via the Traefik ingress path `/com.hedera.mirror.api.proto.NetworkService`. A botnet of a few hundred IPs (each opening one connection and looping `getNodes()` calls) is a realistic, low-cost capability. The attack is repeatable and self-sustaining: because `getNodes()` is finite, the attacker simply reconnects and re-calls in a tight loop.

### Recommendation
1. **Add a total connection limit** to the Netty server in `GrpcConfiguration.java` via `serverBuilder.maxConnectionsTotal(N)` or equivalent.
2. **Add per-IP `inFlightReq` middleware** to the gRPC Traefik middleware block (mirroring the graphql chart's `inFlightReq` + `ipStrategy` pattern).
3. **Implement a gRPC-level rate limiter** (e.g., a `ServerInterceptor` using bucket4j keyed by remote IP) applied to `NetworkService` calls, analogous to the `ThrottleManagerImpl` in the web3 module.
4. **Set `maxConnectionAge` and `maxConnectionIdle`** on the Netty server to reclaim connections from idle/abusive clients.

### Proof of Concept
```bash
# From each of N botnet IPs, run in a loop:
while true; do
  grpcurl -plaintext <mirror-node>:5600 \
    com.hedera.mirror.api.proto.NetworkService/getNodes \
    <<< '{"file_id": {"fileNum": 102}}'
done
```
With N ≥ (HikariCP pool size / 3) IPs running this concurrently (each connection making 5 parallel calls via HTTP/2 multiplexing), the DB connection pool reaches 100% utilization. Subsequent `subscribeTopic` calls return `UNAVAILABLE` or time out, halting gossip transaction monitoring.

    public Flux<AddressBookEntry> getNodes(AddressBookFilter filter) {
        var fileId = filter.getFileId();
        if (!getValidFileIds().contains(fileId)) {
            throw new IllegalArgumentException(INVALID_FILE_ID);
        }

        long addressBookTimestamp = addressBookRepository
                .findLatestTimestamp(fileId.getId())
                .orElseThrow(() -> new EntityNotFoundException(fileId));
        long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp().orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
        var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
        var context = new AddressBookContext(addressBookTimestamp, nodeStakeMap);

        return Flux.defer(() -> page(context))
                .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                        .jitter(0.5)
                        .withFixedDelay(addressBookProperties.getPageDelay())
                        .withScheduler(Schedulers.boundedElastic()))
                .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Querying for address book: {}", filter))
                .doOnComplete(() -> log.info("Retrieved {} nodes from the address book", context.getCount()));
    }
