### Title
Unbounded gRPC Connections Exhaust Private `boundedElastic()` Scheduler in `PollingTopicMessageRetriever`, Causing DoS of Historical Message Retrieval

### Summary
`PollingTopicMessageRetriever` allocates a private `Schedulers.boundedElastic()` instance whose thread pool is bounded at `10 × availableProcessors()` threads. Because there is no global inbound-connection limit and the only per-connection guard (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening additional connections, an unprivileged attacker can drive enough concurrent `retrieve()` subscriptions to exhaust all scheduler threads with blocking JDBC calls, causing subsequent subscriptions to queue (up to the 100,000-task limit) and ultimately fail, denying historical-message retrieval to all legitimate clients.

### Finding Description

**Exact code path** [1](#0-0) 

A single, private `BoundedElasticScheduler` is created at construction time. Its default ceiling is `10 × Runtime.getRuntime().availableProcessors()` worker threads and a task queue of 100,000 entries. [2](#0-1) 

The `RepeatSpec.withScheduler(scheduler)` (line 55) fires each repeat on a thread from that private pool. When the repeat fires it calls `poll()`: [3](#0-2) 

`Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` is a **synchronous, blocking JDBC call**. There is no `subscribeOn` wrapping it, so it executes on — and blocks — the `boundedElastic` scheduler thread for the full duration of the database query. Every active subscription that is in its polling phase holds one thread captive.

**Why the existing guard is insufficient** [4](#0-3) [5](#0-4) 

`maxConcurrentCallsPerConnection = 5` limits calls **per TCP connection**, but there is no global connection count limit, no per-IP connection limit, and no rate-limiting on new gRPC connections anywhere in the server configuration. An attacker simply opens `N` connections to obtain `5N` concurrent `subscribeTopic` calls, each of which invokes `retrieve(filter, true)`.

**Retry amplification** [6](#0-5) 

`Retry.backoff(Long.MAX_VALUE, ...)` retries indefinitely. If the scheduler queue fills and throws `RejectedExecutionException`, each existing subscription retries, generating additional scheduling pressure and creating a positive-feedback loop.

### Impact Explanation

Once all `boundedElastic` threads are occupied with blocking DB queries, new `retrieve()` calls cannot schedule their repeat tasks. The task queue fills (100,000 entries), after which `RejectedExecutionException` propagates up through `retryWhen` in an amplifying loop. Every new `subscribeTopic` call in `TopicMessageServiceImpl` (line 63) invokes `retrieve(filter, true)` as its first step; if that step cannot schedule work, the subscription fails immediately. The gRPC server process remains alive, but **no new historical-message subscription can succeed**, effectively denying the service to all clients. Note: this is a mirror-node gRPC DoS — it does not affect Hedera consensus-node transaction confirmation, so the "total network shutdown" label overstates the scope, but it does constitute a complete denial of the mirror node's topic-subscription API.

### Likelihood Explanation

No authentication or API key is required to open a gRPC connection. The attacker needs only a standard gRPC client (e.g., `grpcurl`, the Hedera SDK, or a raw HTTP/2 client). On a 4-core host the thread ceiling is 40; with 5 calls per connection the attacker needs only **8 connections** to saturate threads if all subscriptions poll simultaneously. Even on a 32-core host (320 threads), 64 connections suffice. Both figures are trivially achievable from a single machine. The attack is repeatable and stateless — connections can be recycled as the 60-second timeout expires.

### Recommendation

1. **Add a global connection limit** in `GrpcConfiguration`: call `serverBuilder.maxConnectionAge(...)`, `serverBuilder.maxConnectionIdle(...)`, and consider a custom `ServerInterceptor` that tracks and caps total active connections or subscriptions.
2. **Decouple blocking DB calls from the scheduler thread**: wrap `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` with `.subscribeOn(scheduler)` inside `poll()`, or switch to a non-blocking R2DBC repository so the scheduler thread is not held during I/O.
3. **Cap the retry count**: replace `Retry.backoff(Long.MAX_VALUE, ...)` with a finite retry limit to prevent amplification when the scheduler is under pressure.
4. **Add per-IP or global subscription rate limiting** at the gRPC interceptor layer (e.g., using Bucket4j or a Netty `ChannelHandler`).

    @Override
    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
