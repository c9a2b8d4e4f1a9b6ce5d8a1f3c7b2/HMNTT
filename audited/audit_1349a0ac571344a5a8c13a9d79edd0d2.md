### Title
Unauthenticated Opcode Replay Endpoint Exposes Full Contract Storage State to Any Caller

### Summary
The `getContractOpcodes()` endpoint at `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes` has zero authentication or authorization controls. Any unprivileged external caller can submit any publicly-known transaction ID or hash with `storage=true` and receive a complete dump of every storage slot key and value accessed during that transaction's EVM replay, including sensitive contract state such as vesting schedule parameters (cliff, duration, amount, beneficiary).

### Finding Description
**Exact code path:**

`OpcodesController.java` lines 52–68 — `getContractOpcodes()`:
```
@GetMapping(value = "/{transactionIdOrHash}/opcodes")
OpcodesResponse getContractOpcodes(
        @PathVariable TransactionIdOrHashParameter transactionIdOrHash,
        @RequestParam(required = false, defaultValue = "true") boolean stack,
        @RequestParam(required = false, defaultValue = "false") boolean memory,
        @RequestParam(required = false, defaultValue = "false") boolean storage,
        @RequestHeader(value = HttpHeaders.ACCEPT_ENCODING) String acceptEncoding) {
    if (properties.isEnabled()) {
        validateAcceptEncodingHeader(acceptEncoding);
        throttleManager.throttleOpcodeRequest();
        final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
        return opcodeService.processOpcodeCall(request);
    }
    throw new ResponseStatusException(HttpStatus.NOT_FOUND);
}
```

A grep across all of `web3/src/main/java/**/*.java` for `@PreAuthorize`, `@Secured`, `SecurityConfig`, `WebSecurityConfigurerAdapter`, `HttpSecurity`, `permitAll`, and `authenticated` returns **zero matches**. There is no Spring Security configuration anywhere in the web3 module.

**Root cause:** The endpoint is a full EVM debug tracer (analogous to Ethereum's `debug_traceTransaction`) that was designed as a privileged debugging tool but was deployed with no access control. The failed assumption is that the feature flag (`OpcodesProperties.enabled`, defaulting to `false`) is a sufficient security gate — it is not, because once enabled in any environment, the endpoint is fully public.

**Storage capture path:** When `storage=true`, `AbstractOpcodeTracer.captureStorage()` (lines 75–129) walks the `RootProxyWorldUpdater`'s `TxStorageUsage`, iterates every `StorageAccesses` entry, and for each `StorageAccess` emits the slot key and either the written value or the read value into a `TreeMap<String,String>`. This map is returned verbatim in the `OpcodesResponse`. There is no filtering, redaction, or ownership check on which slots are returned.

**CORS:** `GenericControllerAdvice` registers `allowedOrigins("*")` for all `/api/v1/contracts/**`, so the endpoint is reachable from any browser or script.

### Impact Explanation
Any caller who can reach the mirror node's web3 API can:
1. Obtain the full set of storage slot keys and their values for any contract, for any historical transaction, without needing to know the contract's storage layout in advance.
2. Directly read structured application data — vesting cliff timestamps, duration, total amount, beneficiary address — that the contract author may have assumed was opaque without ABI knowledge.
3. Correlate multiple replays across different transactions to reconstruct the full historical state of a contract's sensitive fields.

This is a **High severity information disclosure**. On Hedera, contract storage is technically on-chain, but the opcode tracer removes the practical barrier of needing to know storage slot indices: it hands the attacker a labeled map of every slot touched in a transaction, making extraction of structured sensitive data trivial.

### Likelihood Explanation
- **Precondition:** The feature flag `hiero.mirror.web3.opcode.tracer.enabled` must be `true`. This is `false` by default, but any deployment that enables it for debugging or developer tooling is immediately exposed.
- **Attacker capability:** Zero privilege required. Transaction hashes are publicly enumerable via the mirror node's own `/api/v1/contracts/results` REST endpoint.
- **Repeatability:** Fully repeatable and automatable. An attacker can script bulk replay of all transactions to a target contract and extract its complete storage history.
- **Detection difficulty:** The endpoint only enforces a throttle, which slows but does not prevent systematic extraction.

### Recommendation
1. **Add authentication/authorization** to the endpoint. At minimum, require an API key or operator-level credential. Annotate with `@PreAuthorize` or configure Spring Security to restrict `/api/v1/contracts/results/*/opcodes` to authenticated roles only.
2. **Treat the feature flag as insufficient**: document that enabling the tracer without authentication is a security risk and enforce auth as a hard dependency when `enabled=true` (e.g., fail-fast on startup if auth is not configured).
3. **Consider rate-limiting per authenticated identity** rather than globally, to prevent bulk extraction even by authorized users.
4. **Audit all other web3 controller endpoints** for the same pattern — the grep confirms no security annotations exist anywhere in the module.

### Proof of Concept
**Preconditions:** Mirror node web3 service running with `hiero.mirror.web3.opcode.tracer.enabled=true`.

**Step 1 — Discover a transaction hash for the target vesting contract:**
```
GET https://<mirror-node>/api/v1/contracts/results?contract.id=<vesting_contract_id>&limit=25
```
Extract any `hash` value from the response.

**Step 2 — Replay the transaction with storage capture:**
```
GET https://<mirror-node>/api/v1/contracts/results/<tx_hash>/opcodes?storage=true&stack=false&memory=false
Accept-Encoding: gzip
```

**Step 3 — Parse the response:**
The `opcodes[*].storage` field in the JSON response contains a map of hex storage slot keys to hex values. Each entry directly corresponds to a Solidity storage variable accessed during the transaction. For a vesting contract, slots encoding `cliff`, `duration`, `amount`, and `beneficiary` are returned in plaintext hex, decodable with standard ABI decoding tools (e.g., `cast --to-dec <hex_value>`).

No credentials, no special tooling, and no knowledge of the contract's storage layout are required beyond what the response itself reveals.
