### Title
Unbounded Concurrent Subscriptions Enable DB and Scheduler Exhaustion via Griefing

### Summary
The `PollingTopicMessageRetriever.retrieve()` method spawns a periodic DB-polling loop on a shared `boundedElastic()` scheduler for every subscription, with no per-IP or global subscription cap. The only guard, `maxConcurrentCallsPerConnection = 5`, is scoped to a single TCP connection; an attacker opening many connections multiplies DB query load and scheduler thread consumption without bound, degrading service for all legitimate subscribers.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (line 43–53) calls `topicMessageService.subscribeTopic(filter)` with no authentication or rate-limiting check.

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) calls `topicMessageRetriever.retrieve(filter, true)` for historical messages and `topicMessageRetriever.retrieve(gapFilter, false)` for safety-check gap recovery. The `subscriberCount` AtomicLong (line 48) is incremented on each subscription but is only exposed as a Micrometer gauge metric — it is never compared against any threshold to reject new subscriptions.

`PollingTopicMessageRetriever.retrieve()` (lines 45–63) creates a new `PollingContext` per call and schedules a `repeatWhen` loop on a single shared `Schedulers.boundedElastic()` instance (line 41). With `throttled=true` (the default for new subscriptions), `numRepeats = Long.MAX_VALUE` and `frequency = 2s`, meaning each subscription issues a DB query every 2 seconds indefinitely until the client disconnects or a 60-second inter-message timeout fires.

**Root cause / failed assumption:**

`NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14 of `NettyProperties.java`) limits concurrent gRPC calls per TCP connection, not per source IP or globally. An attacker opens `N` TCP connections and issues 5 `subscribeTopic` calls on each, yielding `5N` concurrent polling loops with no server-side enforcement to stop them.

**Exploit flow:**

1. Attacker opens `N` TCP connections to port 5600 (no per-IP connection limit exists).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` RPCs with `limit=0` (infinite) and no `consensusEndTime`, targeting any valid topic ID.
3. Each RPC triggers `PollingTopicMessageRetriever.retrieve(filter, true)` → a `repeatWhen(Long.MAX_VALUE)` loop on the shared `boundedElastic()` scheduler, issuing a `SELECT` against `topic_message` every 2 seconds.
4. Additionally, each subscription's safety-check path (line 67–70 of `TopicMessageServiceImpl`) schedules another `boundedElastic()` task after 1 second.
5. Result: `5N` concurrent DB polling loops + `5N` safety-check tasks saturate the shared scheduler thread pool (default cap: `10 × CPU cores`) and flood the database connection pool.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5`: Per-connection only; bypassed by opening multiple connections.
- `subscriberCount` gauge: Observability only; no enforcement logic reads it.
- `timeout = Duration.ofSeconds(60)` in `RetrieverProperties`: This is a `reactor.core.publisher.Flux.timeout()` applied between message emissions. For a live topic with active messages, it never fires. For a topic with no messages, it fires after 60s — but the attacker simply reconnects immediately.
- No authentication, API key, or IP-based access control exists in the gRPC layer.

### Impact Explanation
Each attacker connection contributes 5 long-lived DB polling loops. With `N=100` connections (trivially achievable from a single host), 500 concurrent queries hit the database every 2 seconds. The shared `Schedulers.boundedElastic()` thread pool becomes saturated, causing legitimate subscribers' polling tasks to queue and starve. Legitimate subscribers experience message delivery delays, timeouts, and eventual disconnection. The database connection pool exhausts, causing errors across all gRPC and potentially REST API paths that share the same DB. No economic cost to the attacker; no privileged access required.

### Likelihood Explanation
Any unauthenticated user with network access to port 5600 can execute this attack using `grpcurl` or any gRPC client library in a loop. The protobuf schema is publicly documented. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the topic ID (which can be any valid topic, including well-known public ones). The absence of any IP-level connection throttling or subscription cap makes this a low-effort, high-impact griefing vector.

### Recommendation
1. **Global subscription cap**: Read `subscriberCount` before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()` and reject with `RESOURCE_EXHAUSTED` if it exceeds a configurable threshold.
2. **Per-IP connection limit**: Configure a Netty `ServerBuilder` `maxConnectionsPerIp` or use an ingress-level policy (the GCP backend policy already has `maxRatePerEndpoint = 250` for requests/s but no connection count cap).
3. **Per-IP subscription tracking**: Maintain a `ConcurrentHashMap<String, AtomicInteger>` keyed by peer IP (available from `Context` in the gRPC interceptor) and enforce a per-IP subscription ceiling.
4. **Minimum subscription duration / reconnect backoff**: Enforce a server-side minimum lifetime before a subscription can be re-established from the same IP, preventing rapid reconnect loops that bypass the 60s timeout.

### Proof of Concept
```bash
# Open 20 connections, each with 5 concurrent subscriptions = 100 polling loops
for i in $(seq 1 20); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
# Each background process holds an infinite subscription.
# Monitor DB connections and boundedElastic thread pool to observe saturation.
# Legitimate subscribers will experience increasing latency and eventual timeout errors.
```
