### Title
Unbounded DB Connection Pool Exhaustion via `getTokenRelationships()` During Network Partition

### Summary
The REST API's pg connection pool (`max=10`, default) has no client-side query timeout and no queue size limit. During a network partition, all pool connections are held indefinitely at the TCP level (server-side `statement_timeout` cannot be delivered back to the client), and an unauthenticated attacker can flood `/accounts/:id/tokens` to fill the pool and its unbounded internal queue, starving every other API endpoint of database connections for the duration of the partition.

### Finding Description

**Code path:**

`rest/controllers/tokenController.js` `getTokenRelationships()` (lines 66–74) issues up to three sequential DB queries per request — `EntityService.getEncodedId()`, `EntityService.isValidAccount()`, and `TokenService.getTokenAccounts()` — all routed through `BaseService.getRows()` / `getSingleRow()` which call `pool.queryQuietly()` on the global `pg` Pool. [1](#0-0) 

The pool is configured in `rest/dbpool.js` with:
- `max: config.db.pool.maxConnections` → default **10**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` → default **20 000 ms** (queue-wait timeout only)
- `statement_timeout: config.db.pool.statementTimeout` → default **20 000 ms** (PostgreSQL server-side)
- **No** `query_timeout` (client-side), **no** `idleTimeoutMillis`, **no** socket-level timeout, **no** queue-size cap. [2](#0-1) 

**Root cause — failed assumption:**

`statement_timeout` is enforced by the PostgreSQL server. During a network partition the server cannot deliver the timeout signal to the client; the TCP connection appears alive (no RST) but carries no data. The pg client therefore holds the connection open indefinitely — potentially for hours until OS-level TCP keepalive fires. `connectionTimeoutMillis` only governs how long a *new* request waits in the pool's internal queue before receiving an error; it does **not** reclaim connections already checked out and hanging.

The pg `Pool` internal queue is an unbounded JavaScript array. No `maxQueue` parameter is set. [3](#0-2) 

**No rate limiting on the REST API:**

`rest/server.js` registers no rate-limiting or concurrency-limiting middleware for the Node.js REST service. The throttle configuration found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the web3 Java service and does not apply here. [4](#0-3) 

**Exploit flow:**

1. Network partition isolates the REST API pod from PostgreSQL.
2. The first 10 concurrent requests to `/api/v1/accounts/<any-valid-id>/tokens` each acquire one of the 10 pool connections and issue queries that hang at the TCP layer indefinitely.
3. All subsequent requests queue in the pg Pool's unbounded internal array.
4. After 20 s each queued request receives a `timeout exceeded when trying to connect` error — but the 10 checked-out connections are **not** returned; they remain held.
5. The attacker sustains a request rate of ≥ 1 req/s (trivially achievable with `curl` or `ab`). At any moment the queue holds ≈ RPS × 20 pending requests.
6. Every other endpoint (`/transactions`, `/balances`, `/contracts`, etc.) that calls the same global pool also queues and times out, rendering the entire REST API non-functional for the duration of the partition.

### Impact Explanation

Complete denial-of-service for all database-backed REST API endpoints. Because the Node.js REST service uses a single shared global `pool` object (set on `global.pool` in `initializePool()`), exhausting it affects every route simultaneously. Severity is **High**: no authentication is required, the attack is trivially scriptable, and the impact is total API unavailability rather than degraded performance. [5](#0-4) 

### Likelihood Explanation

Any unprivileged external user can reach `/api/v1/accounts/:id/tokens` — it is a public, unauthenticated endpoint. Network partitions between application and database tiers occur in cloud environments (AZ failures, misconfigured security groups, rolling DB maintenance). The attacker does not need to know about the partition in advance; they can simply flood the endpoint continuously and the partition amplifies the effect. The attack is repeatable and requires no special tooling beyond standard HTTP clients.

### Recommendation

1. **Add a client-side query timeout** (`query_timeout` option in pg v8+, or wrap every `pool.query()` call with `Promise.race` against a timeout) so connections are forcibly released even when the server is unreachable.
2. **Set `idleTimeoutMillis`** and configure OS-level TCP keepalive (`keepAlive: true`, `keepAliveInitialDelayMillis`) on the pool so dead connections are detected and recycled promptly.
3. **Limit the pg Pool queue size** (e.g., via a custom pool wrapper that rejects requests when pending queue length exceeds a threshold) to prevent unbounded memory growth and to return 503 quickly rather than queuing.
4. **Add API-level concurrency / rate limiting** (e.g., `express-rate-limit` or an ingress-level rate limiter) to cap the number of simultaneous in-flight requests per client IP.
5. Consider raising `max` connections modestly and pairing it with a circuit-breaker pattern that opens when the pool is saturated.

### Proof of Concept

```bash
# Step 1: Simulate or wait for a network partition between REST pod and PostgreSQL
# (e.g., iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the REST host)

# Step 2: Flood the endpoint from an external client (no auth required)
for i in $(seq 1 200); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.1234/tokens" &
done
wait

# Step 3: Immediately attempt any other endpoint
curl -v "http://<mirror-node-host>/api/v1/transactions"
# Expected result: hangs for 20 s then returns 500 "timeout exceeded when trying to connect"
# All endpoints are now starved of DB connections for the duration of the partition.
```
