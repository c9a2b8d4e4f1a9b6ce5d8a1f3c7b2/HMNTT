### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent Requests to `/api/v1/contracts/results/{transactionId}`

### Summary
The `/api/v1/contracts/results/{transactionId}` endpoint in the REST Node.js API has no rate limiting and triggers an expensive two-table JOIN query (`ethereum_transaction JOIN transaction`) via `getEthTransactionByTimestampAndPayerId()` for every request that resolves to an Ethereum transaction. The default DB connection pool is capped at 10 connections with a 20-second connection timeout. An unprivileged attacker sending concurrent requests can exhaust the pool, causing all subsequent legitimate requests to queue and timeout, effectively rendering the REST API unavailable.

### Finding Description

**Exact code path:**

`getContractResultsByTransactionIdOrHash()` in `rest/controllers/contractController.js` (lines 1120–1203) calls `getDetailedContractResults()` (lines 1205–1220), which unconditionally calls:

```js
TransactionService.getEthTransactionByTimestampAndPayerId(
  contractDetails.consensusTimestamp,
  contractDetails.payerAccountId
)
``` [1](#0-0) 

`getEthTransactionByTimestampAndPayerId()` in `rest/service/transactionService.js` (lines 74–85) executes a JOIN query between `ethereum_transaction` and `transaction` tables with no caching, no deduplication, and no circuit breaker:

```js
async getEthTransactionByTimestampAndPayerId(timestamp, payerId) {
  const params = [timestamp, payerId];
  const query = [
    TransactionService.ethereumTransactionDetailsQuery,
    `where ... = $1 and ... = $2`,
  ].join('\n');
  const rows = await super.getRows(query, params);
  return rows.map((row) => new EthereumTransaction(row));
}
``` [2](#0-1) 

The static `ethereumTransactionDetailsQuery` is a full JOIN: [3](#0-2) 

`super.getRows()` calls `this.pool().queryQuietly(query, params)` directly against the global `pg` pool: [4](#0-3) 

**Root cause — the pool:**

The DB pool is initialized with `max: config.db.pool.maxConnections`, defaulting to **10 connections**, `connectionTimeoutMillis: 20000` ms, and `statement_timeout: 20000` ms: [5](#0-4) [6](#0-5) 

**Root cause — no rate limiting on the REST API:**

The throttle/rate-limiting infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` Java module, not in the REST Node.js API. A search for any rate-limit middleware in `rest/` returns nothing. The `users` config only controls response result-count limits, not request rates: [7](#0-6) 

**Exploit flow:**

1. Attacker identifies any valid Hedera transaction ID that maps to an Ethereum transaction (publicly observable on-chain).
2. Attacker sends ≥10 concurrent HTTP GET requests to `/api/v1/contracts/results/{transactionId}`.
3. Each request acquires a DB connection and holds it for up to 20 seconds (statementTimeout) while the JOIN executes.
4. The pool is exhausted; all subsequent requests queue.
5. Queued requests timeout after 20 seconds (connectionTimeout), returning errors to legitimate users.
6. By sustaining the flood, the attacker keeps the pool permanently exhausted.

**Why existing checks are insufficient:**

- `statementTimeout` (20 s) bounds individual query duration but does not prevent pool exhaustion — 10 concurrent slow queries saturate the pool entirely.
- `connectionTimeout` (20 s) causes queued requests to fail with errors, which is the degradation itself, not a mitigation.
- No IP-based, token-based, or global request-rate limiting exists in the REST Node.js API for this endpoint.

### Impact Explanation

The REST API becomes unresponsive to all users for the duration of the attack. Since the pool has only 10 connections by default and the JOIN query can hold each connection for up to 20 seconds, a sustained flood of ≥10 concurrent requests is sufficient to deny service to all legitimate callers. This degrades the mirror node REST service — the primary read interface for the network — without requiring any privileged access or credentials.

### Likelihood Explanation

The attack requires no authentication, no special knowledge beyond a valid transaction ID (trivially obtained from any block explorer), and no sophisticated tooling — a simple `ab` or `curl` loop suffices. The endpoint is publicly documented in the OpenAPI spec: [8](#0-7) 

The attack is repeatable and sustainable indefinitely. The attacker does not need to maintain state between requests.

### Recommendation

1. **Add request-rate limiting middleware** to the REST Node.js API (e.g., `express-rate-limit`) applied globally or specifically to `/contracts/results/:transactionIdOrHash`, limiting requests per IP per second.
2. **Increase the default pool size** (`hiero.mirror.rest.db.pool.maxConnections`) and document a minimum recommended value for production deployments.
3. **Add response caching** (the Redis cache infrastructure already exists — `hiero.mirror.rest.cache.response.enabled`) for this endpoint, so repeated lookups for the same transaction ID do not hit the DB.
4. **Implement a concurrency limiter** at the application layer (e.g., `p-limit` or a semaphore) to cap the number of simultaneous in-flight DB queries for this handler.

### Proof of Concept

```bash
# Step 1: Obtain any valid Ethereum transaction ID from a public explorer
TX_ID="0.0.8001-167654-000123400"

# Step 2: Flood the endpoint with concurrent requests (no auth required)
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/results/${TX_ID}" &
done
wait

# Result: After the first 10 concurrent requests saturate the pool,
# subsequent requests return connection timeout errors (after 20s).
# Legitimate users receive errors for the duration of the flood.
```
