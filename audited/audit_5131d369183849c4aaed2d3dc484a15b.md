### Title
Unbounded Per-Subscriber DB Polling in `PollingTopicListener` Enables Unauthenticated DoS via Subscription Flood

### Summary
`PollingTopicListener.listen()` creates a fully independent `PollingContext` and DB polling loop for every subscriber, with no global subscription count limit and no rate limiting on gRPC subscription creation. An unprivileged attacker can open an arbitrary number of TCP connections (each allowing up to 5 concurrent calls via `maxConcurrentCallsPerConnection`) and flood the database with O(N) concurrent queries every polling interval, exhausting the DB connection pool and causing query timeouts that drop messages for all legitimate subscribers.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, lines 34–48:

```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);          // new context per subscriber
    Duration interval = listenerProperties.getInterval();

    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)          // repeats forever
                    .jitter(0.1)
                    .withFixedDelay(interval)                     // default 500ms
                    .withScheduler(scheduler))
            ...
```

Each call to `listen()` produces a cold `Flux` that independently calls `topicMessageRepository.findByFilter(newFilter)` (line 61) every `interval` milliseconds. There is no shared state, no deduplication, and no cap on how many such loops can run concurrently.

**Root cause — failed assumption:** The design assumes the number of concurrent subscribers will be small and operationally bounded. There is no enforcement of this assumption anywhere in the gRPC stack:

- `TopicMessageServiceImpl.subscriberCount` (lines 48–55) is a **metric gauge only** — it is never checked against any limit before accepting a new subscription.
- `maxConcurrentCallsPerConnection = 5` (`NettyProperties` line 14, applied in `GrpcConfiguration` line 33) limits calls **per TCP connection**, not globally. An attacker opens C connections → 5C independent subscriptions.
- The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) lives entirely in the `web3` module and is **never applied** to gRPC subscription handling.
- `SharedPollingTopicListener` avoids this by using `.share()` (a single hot Flux for all subscribers), but `PollingTopicListener` (POLL mode) does not.

**Precondition:** `hiero.mirror.grpc.listener.type=POLL` must be configured. This is non-default (default is `REDIS`), but it is a documented, supported mode with no security warnings.

### Impact Explanation

With default settings (`interval=500ms`, `maxPageSize=5000`, `statementTimeout=10000ms`):

- Attacker opens C connections × 5 subscriptions = 5C independent DB queries per 500ms = **10C queries/second**
- At C=200 connections: 2,000 DB queries/second, each potentially scanning up to 5,000 rows
- The HikariCP connection pool (monitored via `hikaricp_connections_active`) is exhausted; legitimate queries queue and time out
- All active subscribers — including those monitoring financial topics — receive no messages during the outage
- The `db.statementTimeout=10000ms` means each blocked query holds a DB connection for up to 10 seconds, compounding pool exhaustion

The mirror node is read-only and does not hold funds, so "direct loss of funds" is not the correct severity label. The correct impact is **availability DoS**: complete loss of topic message delivery for all subscribers for the duration of the attack.

### Likelihood Explanation

- Requires no authentication — the gRPC `subscribeTopic` endpoint is publicly accessible
- Requires only a standard gRPC client (e.g., `grpcurl`, any HCS SDK)
- Requires `POLL` mode to be configured — reduces likelihood but does not eliminate it; operators may choose POLL when Redis is unavailable
- Attack is trivially repeatable and scriptable; a single machine with 200 TCP connections suffices
- No existing control (IP blocking, rate limiting, connection limits) prevents this at the application layer

### Recommendation

1. **Enforce a global subscription limit**: Check `subscriberCount` against a configurable maximum before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()`. Return `RESOURCE_EXHAUSTED` gRPC status when exceeded.
2. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer, analogous to the `ThrottleManagerImpl` used in the web3 module.
3. **Add a `maxConnections` limit** to `NettyServerBuilder` in `GrpcConfiguration` alongside `maxConcurrentCallsPerConnection`.
4. **Document the security risk** of `POLL` mode in public-facing deployments and recommend `SHARED_POLL` or `REDIS` instead, since `SharedPollingTopicListener` uses a single shared Flux (`.share()`) and is immune to this class of attack.

### Proof of Concept

```bash
# Install grpcurl or use any HCS gRPC client
# Repeat the following in parallel across many processes/threads:

for i in $(seq 1 200); do
  # Each connection gets 5 concurrent subscriptions (maxConcurrentCallsPerConnection=5)
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1234}, "consensusStartTime": {"seconds": 0}}' \
      mirror-node-host:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Result: 1000 independent DB queries fire every 500ms.
# Monitor DB connection pool exhaustion:
# hikaricp_connections_active / hikaricp_connections_max → approaches 1.0
# Legitimate subscribers stop receiving messages; query timeouts appear in logs.
```
