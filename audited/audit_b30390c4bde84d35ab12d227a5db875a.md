### Title
`BufferUnderflowException` from Short EVM Address in `getByEvmAddressAndType()` — Unhandled Exception Bypasses Custom Error Resolver

### Summary
`decodeEvmAddress()` in `GraphQlUtils.java` performs no length validation on the decoded byte array, allowing any even-length hex string shorter than 40 hex chars (20 bytes) to pass through. In `getByEvmAddressAndType()`, the resulting short byte array is wrapped in a `ByteBuffer` and immediately read with `getInt()` (4 bytes) then `getLong()` (8 bytes); a buffer shorter than 12 bytes causes `BufferUnderflowException`. The `CustomExceptionResolver` only handles `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`, so `BufferUnderflowException` falls through to Spring GraphQL's default handler, which returns an `INTERNAL_ERROR` response and logs the full stack trace server-side — distinguishable from normal validation errors by any unauthenticated caller.

### Finding Description

**Exact code path:**

`decodeEvmAddress()` strips the `0x` prefix and hex-decodes the string with no length check: [1](#0-0) 

`getByEvmAddressAndType()` wraps the result in a `ByteBuffer` and unconditionally reads 4 + 8 + 8 = 20 bytes: [2](#0-1) 

The test suite explicitly confirms that `0x0000AaAaAa` decodes to **5 bytes** without error: [3](#0-2) 

With a 5-byte buffer: `getInt()` consumes bytes 0–3 (succeeds, 1 byte left), then `getLong()` requests 8 bytes → `BufferUnderflowException` is thrown.

**Why the existing exception handler is insufficient:**

`CustomExceptionResolver.resolveToSingleError()` only intercepts `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`. `BufferUnderflowException` (a `RuntimeException`) is not in this list, so the method returns `null`: [4](#0-3) 

When `null` is returned, Spring GraphQL's `SimpleDataFetcherExceptionHandler` takes over: it logs the full exception with stack trace at ERROR level and returns a generic `INTERNAL_ERROR` GraphQL error to the caller — a different error classification than the `ValidationError` returned for truly invalid inputs.

**Entry point:** `AccountController.account()` passes the raw `evmAddress` string directly to `getByEvmAddressAndType()` with no prior length validation: [5](#0-4) 

### Impact Explanation

1. **Error oracle**: Any unauthenticated caller can distinguish "short address" from "invalid hex" by observing `INTERNAL_ERROR` vs. `ValidationError` in the GraphQL response, enabling probing of internal branching logic.
2. **Log pollution / server-side information leakage**: Every such request writes a full Java stack trace to server logs, exposing internal class names, package structure, and call chain — useful for targeted exploit development.
3. **Reliability degradation**: Repeated triggering forces the server to handle uncaught exceptions through the fallback path on every such request, adding overhead and noise to monitoring.

Severity: **Medium** — no authentication required, reliably reproducible, but does not directly exfiltrate data or bypass authorization.

### Likelihood Explanation

The GraphQL endpoint is publicly accessible (no privilege required per `AccountController`). The exploit requires only a valid GraphQL client and knowledge of the `account(evmAddress: ...)` query shape, both of which are trivially discoverable via GraphQL introspection. The payload is a single short hex string. Fully repeatable with zero side effects on the attacker's side.

### Recommendation

Add a length check in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
// In GraphQlUtils.decodeEvmAddress() or EntityServiceImpl.getByEvmAddressAndType()
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException(
        "EVM address must be exactly 20 bytes, got: " + evmAddressBytes.length);
}
```

`IllegalArgumentException` is already handled by `CustomExceptionResolver` and will produce a clean `ValidationError` response instead of an unhandled `INTERNAL_ERROR`. Alternatively, enforce the 40-hex-char constraint at the GraphQL schema/input-validation layer before the service is ever called.

### Proof of Concept

```graphql

  account(input: { evmAddress: "0x0000AaAaAa" }) {
    id
  }
```

**Expected (correct) response:** `{"errors":[{"message":"...","extensions":{"classification":"ValidationError"}}]}`

**Actual response:** `{"errors":[{"message":"INTERNAL_ERROR","extensions":{"classification":"INTERNAL_ERROR"}}]}`

The differing `classification` field in the error response is observable by any unauthenticated caller and confirms the internal exception path was triggered.

```java
    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }
```

```java
    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
```

```java
            return GraphqlErrorBuilder.newError()
                    .errorType(ErrorType.ValidationError)
                    .message(ex.getMessage())
                    .path(env.getExecutionStepInfo().getPath())
```
