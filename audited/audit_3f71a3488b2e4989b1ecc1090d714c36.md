### Title
Unauthenticated BLOCK_NUMBER Filter Triggers Uncached DB Lookup Per Request, Enabling Connection Pool Exhaustion on `/contracts/results`

### Summary
Every GET request to `/api/v1/contracts/results?block.number=<N>` (and `/api/v1/contracts/{id}/results?block.number=<N>`) unconditionally executes a database query via `RecordFileService.getRecordFileBlockDetailsFromIndex()` with no caching, no rate limiting, and no authentication requirement. The REST API's default connection pool is only 10 connections. A flood of concurrent requests from an unprivileged attacker can exhaust the pool, causing all subsequent DB-dependent requests across the mirror node to queue or fail.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js`, `extractContractResultsByIdQuery()` (lines 486–502):

```js
if (blockFilter) {
  let blockData;
  if (blockFilter.key === filterKeys.BLOCK_NUMBER) {
    blockData = await RecordFileService.getRecordFileBlockDetailsFromIndex(blockFilter.value);
  } else {
    blockData = await RecordFileService.getRecordFileBlockDetailsFromHash(blockFilter.value);
  }
  ...
}
``` [1](#0-0) 

This is called on every invocation of `getContractResults` (line 1067) and `getContractResultsById` (line 871), both of which are publicly routed: [2](#0-1) 

`getRecordFileBlockDetailsFromIndex()` executes this SQL on every call with no in-process or Redis caching:

```sql
select consensus_start, consensus_end, hash, index
from record_file
where index = $1
limit 1
``` [3](#0-2) [4](#0-3) 

**Root cause:** The REST Node.js `RecordFileService` has no `@Cacheable` or equivalent caching (contrast with the web3 Java `RecordFileRepository.findByIndex()` which is `@Cacheable`). Every request with a `block.number` filter consumes a DB connection from the shared pool. [5](#0-4) 

The default pool size is **10 connections**: [6](#0-5) 

**No rate limiting exists** in the REST API middleware stack (`authHandler`, `httpErrorHandler`, `metricsHandler`, `openapiHandler`, `requestHandler`, `requestNormalizer`, `responseCacheHandler`, `responseHandler`) — none implement per-IP or global request throttling. The `ThrottleManagerImpl` is web3-only.

**Filter validation** only restricts the operator to `eq`, not the request rate or caller identity: [7](#0-6) 

`BLOCK_NUMBER` is in `acceptedContractResultsParameters` with no privilege gate: [8](#0-7) 

### Impact Explanation
With a pool of 10 connections and no rate limiting, an attacker sending ~50–100 concurrent requests/second with `?block.number=1` will hold all pool connections busy (each query takes a few ms but concurrent saturation is achievable). Once the pool is exhausted, all other API endpoints that require DB access (transactions, accounts, tokens, etc.) will queue behind `connectionTimeoutMillis` (default 20 s) and then fail. This degrades or takes down the entire mirror node REST API, not just the contracts endpoint. Across a deployment with multiple mirror node replicas, the same attack vector applies to each independently, making it feasible to affect 30%+ of nodes with moderate parallelism. [9](#0-8) 

### Likelihood Explanation
No authentication, no rate limiting, no CAPTCHA, and no IP-based throttling are required. The attacker needs only HTTP access to the public API. The attack is trivially scriptable (`ab`, `wrk`, `hey`, or a simple loop). The `block.number` parameter is documented in the OpenAPI spec as a valid public filter: [10](#0-9) 

The attack is repeatable, stateless, and requires no prior knowledge beyond the public API documentation.

### Recommendation
1. **Add in-process caching** to `RecordFileService.getRecordFileBlockDetailsFromIndex()` in the REST service (e.g., a short-lived LRU cache keyed on block index, TTL ~5–10 s), mirroring the pattern used in the web3 Java service.
2. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit`) scoped to endpoints that trigger secondary DB lookups, or globally per IP.
3. **Increase the default connection pool size** or add a queue depth limit with fast-fail behavior to prevent cascading starvation across unrelated endpoints.
4. Consider moving the block-to-timestamp resolution to a cached lookup table or materialized view.

### Proof of Concept

```bash
# Flood /contracts/results with block.number filter (no auth required)
# Requires: wrk or similar HTTP benchmarking tool

wrk -t 10 -c 100 -d 30s \
  "https://<mirror-node-host>/api/v1/contracts/results?block.number=1"

# Simultaneously verify other endpoints degrade:
curl -w "%{time_total}\n" \
  "https://<mirror-node-host>/api/v1/transactions?limit=1"
# Expected: response time increases to connectionTimeoutMillis (20s) then errors
```

Preconditions: Public network access to the mirror node REST API. No credentials needed.
Trigger: Concurrent requests with `?block.number=<any valid or invalid number>`.
Result: DB connection pool exhausted → all REST API endpoints return 500/timeout errors.
