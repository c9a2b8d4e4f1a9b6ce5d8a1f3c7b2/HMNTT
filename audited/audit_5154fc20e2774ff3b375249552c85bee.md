### Title
Unbounded Goroutine Spawning via Unauthenticated `/network/status` Endpoint Leads to Memory Exhaustion DoS

### Summary
The `NetworkStatus()` handler in `rosetta/app/services/network_service.go` executes three sequential database queries per request with no server-side concurrency or rate limit. Because Go's `net/http` server spawns one goroutine per request unconditionally, an unauthenticated attacker flooding this endpoint can accumulate thousands of live goroutines—each holding stack memory and a pending DB-pool slot—until the process exhausts available memory and crashes or becomes unresponsive.

### Finding Description
**Code path:**
`rosetta/app/services/network_service.go`, `NetworkStatus()`, lines 59–88:
```
genesisBlock, err := n.RetrieveGenesis(ctx)   // DB query 1
currentBlock, err := n.RetrieveLatest(ctx)    // DB query 2
peers, err := n.addressBookEntryRepo.Entries(ctx) // DB query 3
```
Each call blocks the goroutine until the DB responds.

**Server setup** (`rosetta/main.go`, lines 217–227): The `http.Server` is configured with only timeout values (`IdleTimeout`, `ReadHeaderTimeout`, `ReadTimeout`, `WriteTimeout`). No `MaxConcurrentConnections`, no semaphore, no token-bucket rate limiter is applied anywhere.

**Middleware chain** (`main.go` lines 217–219): `MetricsMiddleware → TracingMiddleware → CorsMiddleware`. Inspecting all three middleware files (`middleware/metrics.go`, `middleware/trace.go`, `middleware/health.go`) confirms they only instrument, log, and add CORS headers—none enforce a concurrency or rate limit.

**Search result**: A codebase-wide grep for `rate|throttle|semaphore|MaxConc|RateLimit` across all `rosetta/**/*.go` files returned zero hits in any HTTP handler or middleware file.

**Root cause / failed assumption:** The design assumes the endpoint will only be called at a moderate rate (e.g., by Rosetta clients polling for chain status). There is no enforcement of that assumption. Go's scheduler will happily create goroutine N+1 before goroutine N has received its first DB row, so goroutines pile up linearly with request rate.

**Why DB connection pool does not save it:** Even if the pool caps active DB connections at (say) 10, the remaining 9,990 goroutines in a 10,000-request flood are not terminated—they block on `pool.Acquire()`, each consuming ≥8 KB of initial stack plus request-context allocations. The pool throttles DB load but does not bound goroutine count or total memory.

### Impact Explanation
A single attacker machine sending a sustained HTTP flood to `POST /network/status` (a standard Rosetta endpoint, no credentials required) can drive goroutine count into the tens of thousands. At Go's minimum 8 KB stack per goroutine plus per-request heap allocations (context, DB result buffers, `rTypes.NetworkStatusResponse` with a full peer list), memory consumption grows without bound. The process either OOMs and is killed by the OS, or the Go runtime spends all time in GC, making the node unresponsive to legitimate Rosetta clients. Because this is a Rosetta mirror node for Hedera—a network with ≥25% market-cap relevance per the scope—loss of this node degrades the ability of exchanges and custodians to construct and submit transactions.

### Likelihood Explanation
- **No authentication required**: Rosetta `/network/status` is a public, unauthenticated POST endpoint by specification.
- **No special knowledge needed**: The endpoint path and request schema are publicly documented in the Coinbase Rosetta API spec.
- **Trivially scriptable**: A single `ab -n 100000 -c 500` or equivalent tool is sufficient.
- **No existing mitigation in code**: Confirmed by full middleware and config review above.
- **Repeatability**: The attack can be relaunched immediately after a process restart.

### Recommendation
1. **Add a concurrency-limiting middleware** using a buffered channel semaphore or `golang.org/x/sync/semaphore` wrapping the entire router, rejecting requests with HTTP 503 when the limit is reached.
2. **Add per-IP rate limiting** (e.g., `golang.org/x/time/rate` with a `sync.Map` of per-IP limiters) in a new middleware inserted before the router in `main.go`.
3. **Set `http.Server.MaxHeaderBytes`** and consider using `nethttp.LimitListener` to cap total open connections at the TCP level.
4. **Cache `NetworkStatus` responses** for a short TTL (e.g., 1–2 seconds) so concurrent requests share a single DB round-trip rather than each issuing three independent queries.

### Proof of Concept
**Preconditions:** Network access to the Rosetta node's port (default 5700); node running in online mode.

```bash
# Step 1 – confirm endpoint is reachable
curl -s -X POST http://<target>:5700/network/list \
  -H 'Content-Type: application/json' -d '{}'

# Step 2 – flood /network/status with high concurrency
# Replace <blockchain> and <network> with values from step 1
ab -n 200000 -c 1000 -p /tmp/body.json -T application/json \
   http://<target>:5700/network/status
# /tmp/body.json:
# {"network_identifier":{"blockchain":"Hiero","network":"mainnet"}}

# Step 3 – observe on the server
watch -n1 'ps -o pid,rss,vsz,comm -p $(pgrep rosetta)'
# RSS will grow continuously; process eventually OOMs or becomes
# unresponsive (all goroutines blocked waiting for DB pool slots).
```

Expected result: server memory climbs monotonically during the flood; legitimate requests time out; process is killed by OOM killer or stops responding within minutes on a standard cloud VM.
