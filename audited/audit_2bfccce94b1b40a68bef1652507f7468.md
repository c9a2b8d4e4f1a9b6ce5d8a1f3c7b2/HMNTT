### Title
Silent Data Loss via TCP Half-Open State in `RedisTopicListener.subscribe()` — No Heartbeat or Idle-Timeout Detection

### Summary
The `subscribe()` method in `RedisTopicListener` builds a reactive pipeline that relies exclusively on error signals from the Lettuce driver to detect Redis connection failures. In a TCP half-open state (e.g., caused by a NAT timeout or stateful firewall dropping connection tracking), the SUBSCRIBE command was already sent before the partition, so no new write is ever attempted, no error is emitted, and the `retryWhen` operator never fires. An unprivileged subscriber's gRPC stream remains open and appears healthy while silently receiving zero messages, with no mechanism to distinguish this from a legitimately quiet topic.

### Finding Description

**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()`, lines 68–79.

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    Duration interval = listenerProperties.getInterval();
    return container
            .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
            .map(Message::getMessage)
            .doOnCancel(() -> unsubscribe(topic))
            .doOnComplete(() -> unsubscribe(topic))
            .doOnError(t -> log.error("Error listening for messages", t))   // only logs
            .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
            .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
            .share();
}
```

**Root cause — failed assumption:** The pipeline assumes that any loss of connectivity to Redis will surface as a Reactor `onError` signal. This is true for active command/response operations (GET, SET, etc.) because a write attempt will fail when the socket is dead. However, a Redis `SUBSCRIBE` connection is purely read-after-write: the `SUBSCRIBE` command is sent once at setup, and thereafter the driver only reads from the socket waiting for pushed messages. If the TCP connection enters a half-open state after the initial `SUBSCRIBE` succeeds, the driver never writes again, the OS TCP stack never sends a probe (or probes are disabled / have multi-hour defaults), and the socket read simply blocks indefinitely — emitting neither items nor errors.

**Why existing checks fail:**
- `retryWhen(Retry.backoff(...))` at line 78 — only activates on `onError`; silence is not an error.
- `doOnError` at line 76 — only logs; does not help detect half-open.
- No `timeout()` operator anywhere in the pipeline.
- No periodic PING or heartbeat injected into the subscription Flux.
- `grpc/src/main/java/org/hiero/mirror/grpc/config/RedisConfiguration.java` does not configure a `LettuceClientConfiguration` with `socketOptions` (TCP keepalive / connect timeout), so the connection uses OS defaults — typically `tcp_keepalive_time` of 7200 seconds (2 hours) on Linux.
- The `container` Mono is `.cache()`-d (line 55 of the constructor), so even if a new subscriber arrives, the same dead container is reused.

**Exploit flow:**
1. Unprivileged user opens a gRPC `subscribeTopic` stream; `getSharedListener` calls `subscribe(topic)` and the Lettuce connection sends `SUBSCRIBE topic.X` to Redis successfully.
2. A network partition occurs (NAT table entry expires, cloud security-group rule change, Kubernetes network policy update, or a transient link failure). The TCP connection is now half-open: the gRPC service's OS still considers it `ESTABLISHED`; Redis has closed or forgotten it.
3. Redis publishes messages to `topic.X`. The gRPC service's Lettuce connection never receives them (socket read blocks forever). No `onError` is emitted.
4. The subscriber's gRPC stream remains open. No messages arrive. No error is sent to the client.
5. The subscriber cannot distinguish this from "no messages have been published." Data loss is silent and indefinite. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Every subscriber sharing the affected `Flux` (all subscribers to the same topic, because of `.share()`) silently stops receiving messages for the duration of the half-open state — which, without OS-level TCP keepalive tuning, can last hours. Topic messages published during this window are permanently lost to those subscribers; there is no replay mechanism. The subscriber has no signal (no error, no timeout, no heartbeat gap) to trigger a reconnect or alert an operator. This constitutes a silent, undetectable data-integrity failure for any application relying on topic message delivery guarantees.

### Likelihood Explanation
TCP half-open states are routine in cloud and Kubernetes environments:
- NAT/conntrack entries expire (default 300–900 s for established TCP in many cloud providers, well under the OS keepalive default of 7200 s).
- Kubernetes network policy or CNI plugin restarts silently drop connection state.
- Load-balancer or proxy restarts between the gRPC pod and Redis.
- Redis Sentinel failover: the old master closes connections; Lettuce may not detect this on a pure-read SUBSCRIBE socket.

No attacker capability is required — any unprivileged subscriber is automatically affected whenever any of the above infrastructure events occur. The condition is repeatable and persistent until the gRPC service is restarted.

### Recommendation
1. **Add a `timeout()` operator** on the subscription `Flux` so that a configurable period of silence triggers an `onError`, which the existing `retryWhen` will then handle:
   ```java
   .flatMapMany(r -> r.receive(...))
   .timeout(listenerProperties.getMaxSilenceDuration())   // e.g. 30s
   ```
2. **Configure Lettuce TCP keepalive** in `RedisConfiguration` via `LettuceClientConfiguration.builder().socketOptions(SocketOptions.builder().keepAlive(KeepAliveOptions.builder().enable().build()).build())` so the OS sends TCP keepalive probes at a short interval (e.g., 15 s idle, 5 s interval, 3 probes).
3. **Inject a periodic PING** into the subscription pipeline using `Flux.interval(heartbeatInterval).flatMap(i -> ping())` merged with the message stream, so any write failure surfaces as an error.
4. **Do not `.cache()` the container `Mono`** unconditionally, or invalidate it on detected connection failure so a fresh `ReactiveRedisMessageListenerContainer` is created after a partition heals.

### Proof of Concept
**Environment:** gRPC service + Redis, with a stateful firewall or `iptables` between them.

1. Start the mirror-node gRPC service with `hiero.mirror.grpc.listener.type=REDIS`.
2. Open a `subscribeTopic` gRPC stream for any topic (e.g., topic ID 1). Confirm messages flow.
3. On the host running the gRPC service, drop the established TCP connection to Redis without sending a RST/FIN:
   ```bash
   # Drop packets in both directions to simulate half-open (no RST sent)
   iptables -I INPUT  -s <redis-ip> -p tcp --sport 6379 -j DROP
   iptables -I OUTPUT -d <redis-ip> -p tcp --dport 6379 -j DROP
   ```
4. Publish messages to the topic via the importer/Redis publisher.
5. **Observe:** The subscriber's gRPC stream remains open. No messages are delivered. No error is returned to the client. Server logs show no error from `doOnError`. The `retryWhen` never fires.
6. Remove the `iptables` rules. The connection remains half-open; messages continue to be lost until the gRPC service is restarted.
7. **Confirm silent loss:** Compare the sequence numbers of messages received before step 3 and after service restart — the gap represents silently lost messages.

class RedisConfiguration {

    @Bean
    RedisSerializer<TopicMessage> redisSerializer() {
        var module = new SimpleModule();
        module.addDeserializer(EntityId.class, EntityIdDeserializer.INSTANCE);
        module.addSerializer(EntityIdSerializer.INSTANCE);

        var objectMapper = new ObjectMapper(new MessagePackFactory());
        objectMapper.registerModule(module);
        return new Jackson2JsonRedisSerializer<>(objectMapper, TopicMessage.class);
    }

    @Bean
    ReactiveRedisOperations<String, TopicMessage> reactiveRedisOperations(
            ReactiveRedisConnectionFactory connectionFactory) {
        var serializationContext = RedisSerializationContext.<String, TopicMessage>newSerializationContext()
                .key(StringRedisSerializer.UTF_8)
                .value(redisSerializer())
                .hashKey(StringRedisSerializer.UTF_8)
                .hashValue(redisSerializer())
                .build();
        return new ReactiveRedisTemplate<>(connectionFactory, serializationContext);
    }
