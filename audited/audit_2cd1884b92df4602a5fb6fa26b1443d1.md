### Title
Unbounded Redundant Work in `FindBetween()` Enables Resource Exhaustion via Repeated `/block` Requests

### Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` performs a full multi-batch database query and in-memory processing pipeline on every invocation with no result cache. The `/block` HTTP endpoint calls `FindBetween()` unconditionally on every request, and the application-level middleware stack contains no rate limiting. The Traefik-based rate limiting that exists in the Helm chart is disabled by default (`global.middleware: false`), leaving any unprivileged caller free to drive repeated full-cost re-executions for the same block.

### Finding Description
**Code path:**

`POST /block` → `blockAPIService.Block()` (`block_service.go:47-74`) → `s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` (`base_service.go:54-60`) → `transactionRepository.FindBetween()` (`transaction.go:112-171`).

**Root cause — no result cache in `FindBetween()`:**

Every call to `FindBetween()` unconditionally:
1. Opens a DB connection and executes `selectTransactionsInTimestampRangeOrdered` in a loop with `LIMIT 2000` per iteration until all rows are consumed (`transaction.go:124-147`). Each iteration issues a correlated subquery against `crypto_transfer` and `staking_reward_transfer` for every row in the batch.
2. Builds an in-memory `sameHashMap` and `hashes` slice over all returned rows (`transaction.go:149-159`).
3. Calls `constructTransaction()` for every unique hash, which JSON-unmarshals two transfer arrays per sub-transaction and runs `categorizeHbarTransfers()` (`transaction.go:161-170`).

There is no memoization, no TTL cache, and no deduplication of identical `(start, end)` pairs across concurrent or sequential requests.

**The only existing guard** is the trivial `start > end` check at line 116-118, which does not limit repeated identical calls.

**`entityCache` does not help:** The LRU cache in `blockAPIService` (`block_service.go:22, 34-37`) caches only entity-ID-to-alias mappings used in `updateOperationAccountAlias()`, not block transaction results.

**Rate limiting is disabled by default:** The Traefik middleware chain (including `rateLimit: average: 10` and `inFlightReq: amount: 5`) is rendered only when `global.middleware: true` (`middleware.yaml:3`). The default value is `global.middleware: false` (`values.yaml:95`). The application's own middleware stack (`main.go:217-219`) contains only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no rate limiting.

### Impact Explanation
An attacker targeting a block with a large transaction count (requiring multiple 2000-row DB batches) can force the node to re-execute the full query-and-processing pipeline on every request. Because the Rosetta `/block` endpoint is a public, unauthenticated POST endpoint, there is no credential barrier. Sustained parallel requests for the same high-transaction block will saturate DB connection pool slots, increase PostgreSQL CPU and I/O (the correlated subqueries against `crypto_transfer` are not free even with buffer cache), and consume Go goroutine/memory resources for JSON unmarshaling and map construction. This can raise node resource consumption well above 30% of baseline without any brute-force volume — a modest rate of concurrent requests against a large block is sufficient.

### Likelihood Explanation
The exploit requires no credentials, no special knowledge beyond a valid block index or hash (publicly observable on-chain), and no tooling beyond a standard HTTP client. The attacker can identify a historically large block (many transactions) and issue concurrent POST requests to `/block` with that block's identifier. The attack is repeatable indefinitely. Default deployments without the Traefik middleware enabled are fully exposed.

### Recommendation
1. **Add a result cache at the service layer**: Cache `FindBetween()` results keyed on `(consensusStart, consensusEnd)` with a short TTL (e.g., 5–30 seconds). Since historical blocks are immutable, cache entries never become stale.
2. **Enable application-level rate limiting**: Add per-IP or global in-flight request limiting directly in the Go HTTP server (e.g., `golang.org/x/time/rate` or a semaphore), independent of the optional Traefik middleware.
3. **Change the default**: Set `global.middleware: true` in `values.yaml` so the Traefik rate-limit and in-flight-request middleware is active in default deployments.

```bash
# 2. Send 50 concurrent POST /block requests for the same block
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-node>/block \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":1000000}}' &
done
wait
# Each request independently re-executes the full FindBetween() DB batching
# loop and in-memory processing pipeline with no deduplication or caching.
# Monitor node CPU/DB connection usage to observe the resource spike.
```

```go
func (tr *transactionRepository) FindBetween(ctx context.Context, start, end int64) (
	[]*types.Transaction,
	*rTypes.Error,
) {
	if start > end {
		return nil, hErrors.ErrStartMustNotBeAfterEnd
	}

	db, cancel := tr.dbClient.GetDbWithContext(ctx)
	defer cancel()

	transactions := make([]*transaction, 0)
	for start <= end {
		transactionsBatch := make([]*transaction, 0)
		err := db.
			Raw(
				selectTransactionsInTimestampRangeOrdered,
				sql.Named("start", start),
				sql.Named("end", end),
			).
			Limit(batchSize).
			Find(&transactionsBatch).
			Error
		if err != nil {
			log.Errorf(databaseErrorFormat, hErrors.ErrDatabaseError.Message, err)
			return nil, hErrors.ErrDatabaseError
		}

		transactions = append(transactions, transactionsBatch...)

		if len(transactionsBatch) < batchSize {
			break
		}

		start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
	}
```
