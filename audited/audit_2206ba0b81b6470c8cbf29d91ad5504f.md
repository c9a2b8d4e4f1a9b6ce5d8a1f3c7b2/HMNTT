### Title
Unauthenticated Volumetric DoS via Unbounded `account()` Queries on `/graphql/alpha` with No Application-Level Rate Limiting

### Summary
The GraphQL service exposes `account(input: AccountInput!)` at `/graphql/alpha` with no authentication and no application-level request rate limiting. Unlike the `web3` module which uses bucket4j throttling, the `graphql` module has no equivalent per-IP or global rate control. An unprivileged attacker can flood the endpoint with valid `account()` queries from multiple source IPs, exhausting the database connection pool and CPU, causing service degradation or complete unavailability for legitimate users.

### Finding Description

**Exact code path:**

`graphql/src/main/resources/graphql/query.graphqls` line 5 declares the unauthenticated entry point:
```graphql
account(input: AccountInput!): Account
```

`graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java` lines 33–58: every call to `account()` unconditionally invokes `entityService.getByIdAndType()` / `getByAliasAndType()` / `getByEvmAddressAndType()`, each of which issues a live database query via `EntityRepository`. There is no result-level caching, no request coalescing, and no per-caller throttle.

`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java` lines 42–48 configures the only query-level guards:
```java
var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
```
These protect against a single oversized query but impose **zero limit on the number of requests per second per client**.

**Root cause — failed assumption:** The design assumes that infrastructure-level controls (Traefik middleware, GCP gateway) will always be present and active. In reality:

- `charts/hedera-mirror-graphql/values.yaml` line 89: `global.middleware: false` — the Traefik `inFlightReq` middleware (which would cap 5 concurrent requests per IP) is **disabled by default**.
- `charts/hedera-mirror-graphql/values.yaml` line 95: `hpa.enabled: false` — autoscaling is off; the deployment runs a single replica by default.
- `charts/hedera-mirror-graphql/values.yaml` line 56: `maxRatePerEndpoint: 250` applies only when the GCP gateway is enabled (`gateway.gcp.enabled: true`) and is a per-pod cap, not a per-client cap.

**Contrast with web3:** `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java` implements bucket4j rate limiting (`requestsPerSecond`, `gasPerSecond`) at the application layer. The `graphql` module has no equivalent bean, filter, or interceptor.

### Impact Explanation

Each `account()` call triggers at least one synchronous SQL query against the PostgreSQL mirror node database. With HPA disabled and a single pod, the HikariCP connection pool is finite (default Spring Boot pool size: 10). A sustained flood of requests from multiple source IPs exhausts the pool, causing all subsequent queries to queue and time out. The `db.statementTimeout` of 10,000 ms means each blocked thread holds a connection slot for up to 10 seconds, compounding the exhaustion. The result is complete service unavailability — all legitimate `account()` queries return errors or hang — for the duration of the attack. Because the endpoint is public and unauthenticated, there is no mechanism to identify or block the attacker at the application layer.

### Likelihood Explanation

Preconditions: none. The attacker needs only network access to the public `/graphql/alpha` endpoint and a tool capable of sending HTTP POST requests (curl, wrk, ab, or a botnet). The query payload is trivially small and well-documented in the project's own Postman collection (`charts/hedera-mirror-graphql/postman.json` lines 39, 99). Distributing requests across multiple geographic source IPs bypasses any IP-based block at the load balancer. The attack is repeatable indefinitely and requires no authentication, no tokens, and no prior knowledge beyond the public endpoint path.

### Recommendation

1. **Add application-level rate limiting to the graphql module** using bucket4j (mirroring `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) — implement a per-IP token bucket filter on the `/graphql/alpha` path.
2. **Enable the Traefik middleware by default** (`global.middleware: true`) so `inFlightReq: amount: 5` per IP is enforced even without the GCP gateway.
3. **Enable HPA by default** (`hpa.enabled: true`) so the deployment can scale under load rather than saturating a single pod.
4. **Add a result cache** for `account()` lookups (Caffeine, similar to the existing query parser cache) to reduce DB pressure for repeated identical queries.
5. **Document the dependency on infrastructure-level rate limiting** explicitly, so operators deploying without GCP/Traefik know they must supply an equivalent control.

### Proof of Concept

```bash
# Single-source flood (replace HOST with the target)
HOST="https://target.example.com"
QUERY='{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}'

# Launch 500 concurrent workers, unlimited requests
wrk -t 10 -c 500 -d 60s \
  -s <(echo 'wrk.method="POST"
              wrk.headers["Content-Type"]="application/json"
              wrk.body='"'"''"$QUERY"''"'"'') \
  "$HOST/graphql/alpha"

# Distributed variant: repeat from N geographic VPS nodes simultaneously
# No credentials, tokens, or special headers required.
# Observable impact: HikariCP active connections saturate → 
#   legitimate requests receive HTTP 500 / timeout after ~10s
#   (hiero.mirror.graphql.db.statementTimeout = 10000ms)
```

Expected result: within seconds, `hikaricp_connections_active` reaches `hikaricp_connections_max`; all subsequent `account()` queries from legitimate users fail until the flood stops.
