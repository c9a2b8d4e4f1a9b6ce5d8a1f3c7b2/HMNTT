### Title
Unbounded Redis Channel Subscription Growth via Distinct Topic ID Flooding in `RedisTopicListener.subscribe()`

### Summary
`RedisTopicListener.getSharedListener()` uses a `ConcurrentHashMap` keyed by topic ID with no upper bound on entries. Each distinct topic ID causes a new `r.receive()` call on the shared `ReactiveRedisMessageListenerContainer`, creating a new Redis pub/sub channel subscription that persists as long as any subscriber remains connected. An unprivileged attacker can open many gRPC connections — each subscribing to a different valid topic ID — causing unbounded growth of both the in-process subscription map and the Redis channel subscription set, exhausting JVM heap and Redis server memory and degrading the pub/sub pipeline for all mirror node instances sharing that Redis connection.

### Finding Description

**Exact code path:**

`getSharedListener()` at line 61 calls `computeIfAbsent` on the unbounded `topicMessages` map:
```java
// RedisTopicListener.java line 59-62
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

The key is `"topic.<numericId>"` (line 65), so every distinct topic ID produces a distinct map entry. `subscribe()` at line 72 issues a new `r.receive()` call per entry:
```java
// RedisTopicListener.java line 71-72
return container
    .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
```

The `.share()` at line 79 means multiple subscribers to the *same* topic share one Redis subscription, but N *distinct* topics produce N independent `r.receive()` calls and N active Redis channel subscriptions. Cleanup only fires via `doOnCancel`/`doOnComplete` (lines 74-75) when the *last* subscriber to a given topic disconnects — so keeping connections open prevents any cleanup.

**Root cause:** There is no cap on the size of `topicMessages`, no global limit on the number of concurrent distinct-topic subscriptions, and no rate limiting on the gRPC `subscribeTopic` endpoint.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14, GrpcConfiguration line 33) limits calls *per TCP connection*, not globally. An attacker opens thousands of TCP connections, each carrying 5 subscriptions to distinct topics.
- `checkTopicExists = true` (GrpcProperties line 19) requires a valid topic entity, but valid topic IDs are fully enumerable via the public mirror node REST API (`/api/v1/topics`). On mainnet/testnet there are millions of valid topic IDs.
- No throttle/rate-limit bean exists in the gRPC module (the `ThrottleConfiguration` is web3-only).
- `TopicMessageServiceImpl` tracks a `subscriberCount` gauge (line 48) but enforces no ceiling on it.

### Impact Explanation

With N TCP connections × 5 calls each, an attacker creates 5N entries in the `topicMessages` map and 5N active Redis channel subscriptions. Each entry holds a live Reactor `Flux` chain backed by a Redis subscription. At scale this causes:

1. **JVM heap exhaustion** on the mirror node — the map and associated Flux objects are never evicted while connections are held open.
2. **Redis server memory pressure** — the Redis server must maintain a subscription record and routing entry for each channel; at tens of thousands of channels this degrades Redis pub/sub throughput for all legitimate subscribers.
3. **Cross-instance blast radius** — all mirror node replicas sharing the same `ReactiveRedisMessageListenerContainer` / Redis connection are affected simultaneously, matching the "≥30% of network processing nodes" severity tier.

### Likelihood Explanation

The attack requires no credentials, no special protocol knowledge, and no brute force. The attacker needs only:
- A standard gRPC client (e.g., `grpcurl`, the Hedera SDK, or a raw HTTP/2 client).
- A list of valid topic IDs, freely available from the public mirror node REST API.

Opening thousands of long-lived gRPC streams is trivially scriptable. The `maxConcurrentCallsPerConnection = 5` limit is easily bypassed by opening more connections. The attack is repeatable and can be sustained indefinitely.

### Recommendation

1. **Cap the `topicMessages` map size.** Reject or evict entries beyond a configurable maximum (e.g., 1 000 distinct topics). Use a size-bounded cache (Caffeine/Guava) instead of a raw `ConcurrentHashMap`.
2. **Add a global concurrent-subscription limit.** In `TopicMessageServiceImpl`, compare `subscriberCount` against a configurable ceiling and return `RESOURCE_EXHAUSTED` when exceeded.
3. **Add per-IP / per-connection rate limiting** on the gRPC `subscribeTopic` endpoint, analogous to the `ThrottleConfiguration` already present in the web3 module.
4. **Enforce a maximum number of distinct active topics** in `RedisTopicListener` independently of subscriber count, since the Redis cost is per-topic, not per-subscriber.

### Proof of Concept

```bash
# 1. Enumerate valid topic IDs from the public REST API
curl "https://<mirror-node>/api/v1/topics?limit=1000" | jq '.[].topic_id'

# 2. Open one long-lived gRPC subscription per distinct topic ID
#    (repeat across many parallel processes to bypass maxConcurrentCallsPerConnection=5)
for TOPIC_ID in $(cat topic_ids.txt); do
  grpcurl -d "{\"topicID\":{\"topicNum\":${TOPIC_ID##*.}}}" \
    -plaintext <mirror-node>:5600 \
    proto.ConsensusService/subscribeTopic &
done

# 3. Keep all connections open (processes stay alive).
#    Observe: topicMessages map grows to N entries,
#             Redis PUBSUB CHANNELS count grows to N,
#             mirror node heap usage climbs until OOM or severe GC pressure.
wait
```

Each background `grpcurl` process holds its stream open, preventing `doOnCancel`/`unsubscribe` from firing. The `topicMessages` map and Redis channel subscription set grow monotonically until the mirror node runs out of heap or Redis degrades.
