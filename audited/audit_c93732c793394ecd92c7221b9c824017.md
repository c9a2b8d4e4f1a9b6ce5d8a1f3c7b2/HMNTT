### Title
Unauthenticated GraphQL evmAddress All-Zeros Input Causes Uncached DB Miss Loop, Enabling DB CPU Exhaustion

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl.java` routes an all-zero 20-byte EVM address (40 hex `0` characters) to `entityRepository.findById(0L)`. Entity ID 0 does not exist in Hedera, so every such request produces a guaranteed DB miss. The GraphQL module's `EntityRepository` has no `@Cacheable` annotation on `findById`, and the GraphQL module has no rate limiting, meaning an unprivileged attacker can flood the database with primary-key lookups at arbitrary throughput.

### Finding Description
**Exact code path:**

`AccountController.account()` (line 51-54) → `EntityServiceImpl.getByEvmAddressAndType()` (lines 34-41) → `EntityRepository.findById(0L)` (inherited from `CrudRepository`).

**Root cause — ByteBuffer branch:**
```
// EntityServiceImpl.java lines 35-38
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);   // 20 zero bytes
var buffer = ByteBuffer.wrap(evmAddressBytes);
if (buffer.getInt() == 0 && buffer.getLong() == 0) {     // bytes 0-11 all zero → TRUE
    return entityRepository.findById(buffer.getLong());  // bytes 12-19 → 0L
}
```
`decodeEvmAddress("0000000000000000000000000000000000000000")` hex-decodes to 20 zero bytes. The `ByteBuffer` reads 4 + 8 = 12 bytes (all zero), the condition is true, and the final `getLong()` over bytes 12-19 (also zero) yields `0L`. `findById(0L)` executes `SELECT * FROM entity WHERE id = 0`, which always returns empty.

**Why checks fail:**

- **Schema validation** (`@Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")`) explicitly accepts 40 hex zeros — it is a syntactically valid address.
- **GraphQL `EntityRepository`** (graphql module) extends bare `CrudRepository<Entity, Long>` with no `@Cacheable` on `findById`. Contrast with the grpc module's `EntityRepository` which has `@Cacheable(cacheNames = CACHE_NAME, unless = "#result == null")`. Every call hits the database.
- **No rate limiting** exists anywhere in the `graphql` module. The `ThrottleConfiguration`/`ThrottleManagerImpl` infrastructure lives exclusively in the `web3` module and is not applied to GraphQL endpoints.
- **No authentication** — no Spring Security configuration was found in the graphql module; the endpoint is publicly accessible.
- The only caching in the graphql module is `CachedPreparsedDocumentProvider`, which caches the *parsed query document*, not the query result.

### Impact Explanation
An attacker can drive an unbounded number of `SELECT * FROM entity WHERE id = 0` primary-key lookups against the mirror node database. While each individual lookup is cheap, at high concurrency (thousands of requests/second from a single client or a small botnet) the aggregate I/O and CPU load on the database server can exceed the 30% threshold described in the scope. Because the result is always a miss, no application-level or HTTP-level cache can absorb the load. Other legitimate queries degrade as DB connection pool slots and CPU cycles are consumed.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and zero knowledge of the system beyond the public GraphQL schema. The input is a single, static, trivially constructed string. It is fully repeatable and scriptable with any HTTP client (e.g., `curl` in a loop, `wrk`, `ab`). The attacker does not need to vary the input — the same payload produces the same DB miss every time, making it trivial to automate at high throughput.

### Recommendation
1. **Add result caching** to the graphql `EntityRepository.findById` with a short TTL and a negative-result cache entry (or use a sentinel), mirroring the grpc module's `@Cacheable` pattern.
2. **Reject entity ID 0 explicitly** in `getByEvmAddressAndType()` before issuing the DB call, since ID 0 is never a valid Hedera entity.
3. **Implement rate limiting** on the GraphQL endpoint (e.g., via a Spring `HandlerInterceptor` or a gateway-level policy), analogous to the `ThrottleConfiguration` in the web3 module.
4. **Validate the decoded entity number** is positive before querying: `if (entityNum <= 0) return Optional.empty();`

### Proof of Concept
```bash
# Send 10,000 GraphQL account queries with all-zeros evmAddress
for i in $(seq 1 10000); do
  curl -s -X POST http://<mirror-node-graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input:{evmAddress:\"0000000000000000000000000000000000000000\"}){type} }"}' \
    &
done
wait
```
Each request passes schema validation, enters `getByEvmAddressAndType()`, satisfies the `getInt()==0 && getLong()==0` branch condition, and executes `SELECT * FROM entity WHERE id = 0` against the database with no caching or rate-limiting protection.     public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }
