### Title
Unauthenticated Content-Length Header Injection Permanently Inflates `api_request_size_bytes` Histogram Sum

### Summary
The `metricsMiddleware()` function in `rest/middleware/metricsHandler.js` reads the raw `content-length` header and passes it directly to `requestSizeHistogram.record()` with no upper-bound validation. Any unauthenticated user can send a GET (or other bodyless) request with `Content-Length: 9007199254740991`, causing the OpenTelemetry histogram's cumulative `_sum` counter to be inflated by `Number.MAX_SAFE_INTEGER` in a single request. Because Prometheus histogram sums are monotonically cumulative and never reset until process restart, this permanently destroys the usefulness of the `api_request_size_bytes_sum` metric for capacity planning.

### Finding Description

**Exact code path:**

`rest/middleware/metricsHandler.js`, line 209, inside the `res.on('finish', ...)` callback within `metricsMiddleware()`:

```javascript
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
``` [1](#0-0) 

**Root cause:** The only sanitization is `parseInt(...) || 0`, which guards against `NaN` and `0` but places no upper bound on the recorded value. `parseInt('9007199254740991', 10)` returns `9007199254740991` (truthy), so the `|| 0` fallback never fires and the raw attacker-controlled value is passed directly to `record()`.

**Why the body-parser limit does not protect this path:**

In `rest/server.js`, `express.json()` is registered at lines 68–73, and `metricsHandler()` is registered at lines 89–92 — *after* the body parsers. [2](#0-1) 

`express.json()` only enforces its 100 KB body-size limit when the request has `Content-Type: application/json` and an actual body to parse. For a **GET request** (or any request without a matching content-type), `express.json()` skips parsing entirely, the request flows through to `metricsMiddleware`, and the `res.on('finish')` callback is registered and fires normally — reading the attacker-supplied header verbatim.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/accounts HTTP/1.1\r\nContent-Length: 9007199254740991\r\n\r\n`
2. `express.json()` skips the request (no body / no matching content-type).
3. `metricsMiddleware` runs, registers `res.on('finish')`.
4. Response completes; callback fires at line 209.
5. `parseInt('9007199254740991', 10)` → `9007199254740991`; `|| 0` does not trigger.
6. `requestSizeHistogram.record(9007199254740991, labels)` executes.
7. The OpenTelemetry SDK adds `9007199254740991` to the cumulative `api_request_size_bytes_sum` counter.
8. `api_request_size_bytes_sum / api_request_size_bytes_count` now reports ~9 petabytes average request size for that route/method/code label combination — permanently, until process restart.

### Impact Explanation

Prometheus histogram `_sum` counters are strictly monotonically increasing and never reset during process lifetime. A single injected value of `9007199254740991` bytes (~9 PB) dwarfs any realistic traffic volume, making the computed average (`_sum / _count`) meaningless for the affected label set. Capacity-planning dashboards and alerting rules that rely on `api_request_size_bytes` (e.g., rate-of-change, percentile approximations derived from sum/count) are rendered permanently unreliable until the process is restarted. The attacker can repeat the injection immediately after each restart with zero cost.

### Likelihood Explanation

The attack requires no authentication, no special privileges, and no knowledge beyond the public API path structure. A single HTTP GET request is sufficient. The exploit is trivially scriptable and repeatable. Any internet-accessible deployment of the mirror-node REST API is exposed.

### Recommendation

Add an explicit upper-bound clamp before recording. For example:

```javascript
const MAX_RECORDED_BYTES = 100 * 1024 * 1024; // 100 MB — adjust to your body-size limit
const contentLength = Math.min(
  Math.max(0, parseInt(req.headers['content-length'] ?? '0', 10) || 0),
  MAX_RECORDED_BYTES
);
requestSizeHistogram.record(contentLength, labels);
```

Alternatively, instead of trusting the `content-length` header (which is attacker-controlled and may not reflect actual bytes received), measure the actual bytes consumed from the request stream, consistent with how `responseSize` is measured via the intercepted `res.write`/`res.end` calls. [3](#0-2) 

### Proof of Concept

```bash
# Single request sufficient to corrupt the histogram sum permanently (until restart)
curl -s -o /dev/null \
  -H "Content-Length: 9007199254740991" \
  "http://<mirror-node-rest-host>/api/v1/accounts"

# Verify corruption by scraping the metrics endpoint
curl -s "http://<mirror-node-rest-host>/swagger/metrics/" \
  | grep api_request_size_bytes_sum
# Expected output: api_request_size_bytes_sum{...} 9.007199254740992e+15
# (or accumulated sum dominated by the injected value)
```
