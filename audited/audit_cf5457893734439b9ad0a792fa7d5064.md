### Title
Deleted Entity Bypass via Long-Zero EVM Address in GraphQL `getByEvmAddressAndType()`

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, when the input EVM address matches the long-zero pattern (`0x000000000000000000000000<8-byte-entity-id>`), the code routes to `entityRepository.findById()` (Spring Data `CrudRepository`) which has no `deleted is not true` SQL filter. The `findByEvmAddress()` branch used for all other addresses does enforce this filter. An unprivileged attacker can craft a long-zero address for any known deleted entity ID to retrieve it as if it were active.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // line 37: long-zero check
        return entityRepository.findById(buffer.getLong())         // line 38: NO deleted filter
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)      // line 40: has deleted filter
                           .filter(e -> e.getType() == type);
}
```

**Root cause — asymmetric deletion filtering:**

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`:
- `findByEvmAddress`: `select * from entity where evm_address = ?1 and deleted is not true` ✅
- `findById`: inherited from `CrudRepository<Entity, Long>` — plain `SELECT * FROM entity WHERE id = ?` with **no** `deleted` predicate ❌

The only post-query check is `.filter(e -> e.getType() == type)`, which tests entity type but never tests `e.getDeleted()`.

**Exploit flow:**
1. A smart contract with Hedera entity ID `N` is deployed and later deleted (`deleted = true` in the `entity` table).
2. Attacker encodes the long-zero EVM address: 4 zero bytes (shard) + 8 zero bytes (realm) + 8-byte big-endian encoding of `N` = `0x0000000000000000000000000000000000000000` + `hex(N)` (20 bytes total).
3. Attacker sends a GraphQL query to the public mirror-node endpoint:
   ```graphql
   { contract(input: { evmAddress: "0x000000000000000000000000<hex(N)>" }) { deleted ... } }
   ```
4. `buffer.getInt() == 0 && buffer.getLong() == 0` evaluates to `true` → `findById(N)` is called.
5. Spring Data returns the deleted entity row without any deletion filter.
6. The response returns the deleted contract as a valid, non-null result, with `deleted: true` exposed in the payload but the entity itself resolved — callers checking for a non-null response treat the contract as resolvable/active.

### Impact Explanation
Any caller using the GraphQL API to resolve a contract by EVM address receives a non-empty result for a destroyed contract when using the long-zero form of its address. dApps or indexers that check "does this contract exist?" by testing for a non-null GraphQL response will incorrectly conclude the contract is live. This constitutes unintended smart contract behavior at the mirror-node layer: a destroyed contract appears resolvable, which can mislead downstream protocol logic. No funds are directly at risk, consistent with the Medium scope classification.

### Likelihood Explanation
The attack requires zero privileges — only knowledge of a deleted contract's numeric entity ID, which is public information on Hedera (visible in transaction records, mirror REST API, etc.). The long-zero address encoding is a well-known Hedera convention documented in HIPs. The exploit is trivially repeatable with a single GraphQL HTTP request and requires no special tooling.

### Recommendation
Apply the same deletion guard in the `findById` branch. Either:

1. **Add a deletion check after `findById`:**
   ```java
   return entityRepository.findById(buffer.getLong())
       .filter(e -> e.getDeleted() == null || !e.getDeleted())
       .filter(e -> e.getType() == type);
   ```
2. **Add a custom repository method with the filter baked in:**
   ```java
   @Query("select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByIdAndNotDeleted(long id);
   ```
   and use it in the long-zero branch, mirroring the pattern already used by `findByAlias` and `findByEvmAddress`.

### Proof of Concept

**Precondition:** A smart contract with Hedera entity ID `1234` (`0x00000000000004D2`) exists and has been deleted.

**Steps:**
1. Encode the long-zero EVM address: `0x0000000000000000000000000000000000000004D2` (padded to 20 bytes).
2. Send the following GraphQL request to the mirror-node GraphQL endpoint:
   ```http
   POST /graphql/alpha
   Content-Type: application/json

   {"query":"{ contract(input:{evmAddress:\"0x0000000000000000000000000000000000000004D2\"}) { deleted entityId { num } } }"}
   ```
3. **Expected (correct) result:** `null` — deleted entity should not be resolved.
4. **Actual result:** The deleted contract entity is returned (non-null), with `deleted: true` in the payload, demonstrating the bypass. [1](#0-0) [2](#0-1)
