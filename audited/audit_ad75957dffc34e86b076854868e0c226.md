### Title
Unauthenticated Three-Way FULL OUTER JOIN DoS via `account.id` + `transactiontype` Filter Combination

### Summary
Any unauthenticated user can supply both `account.id` and `transactiontype` query parameters to `/api/v1/transactions`, unconditionally routing execution into the three-way FULL OUTER JOIN branch of `getTransactionTimestampsQuery()`. The `transactiontype` filter additionally forces each of the two transfer subqueries to JOIN against the `transaction` table, resulting in the `transaction` table being scanned four times per request. The REST API has no per-IP rate limiting, making this trivially repeatable for sustained database resource exhaustion.

### Finding Description

**Exact code path:**

`GET /api/v1/transactions?account.id=0.0.98&transactiontype=CRYPTOTRANSFER`

1. `getTransactions()` → `doGetTransactions()` → `getTransactionTimestamps()` → `getTransactionTimestampsQuery()` in `rest/transactions.js`.

2. In `extractSqlFromTransactionsRequest()` (lines 375–444), `account.id` populates `accountQuery` (non-empty string) and `transactiontype` populates `transactionTypeQuery` (non-empty string). `creditDebitQuery` remains empty.

3. In `getTransactionTimestampsQuery()` (lines 500–607), the branch condition at line 555 is:
   ```js
   if (creditDebitQuery || accountQuery) {
   ```
   Since `accountQuery` is non-empty and `creditDebitQuery` is empty, execution falls through the inner `if (creditDebitQuery)` check at line 580 and reaches the three-way FULL OUTER JOIN return at lines 595–604:
   ```js
   from (${transactionOnlyQuery}) as t
       full outer join (${cryptoTransferQuery}) as ctl
                       on t.consensus_timestamp = ctl.consensus_timestamp
       full outer join (${tokenTransferQuery}) as ttl
                       on coalesce(t.consensus_timestamp, ctl.consensus_timestamp) = ttl.consensus_timestamp
   ```

4. `transactionOnlyQuery` (lines 537–553) itself contains a UNION ALL with `entity_transaction` (the `nftTransfersUnion`, lines 527–535), so it scans both `transaction` and `entity_transaction`.

5. In `getTransferDistinctTimestampsQuery()` (lines 325–356), the `joinClause` at lines 339–342 is activated whenever `transactionTypeQuery` is non-empty:
   ```js
   const joinClause =
     (resultTypeQuery || transactionTypeQuery) &&
     `join ${Transaction.tableName} as ${Transaction.tableAlias}
       using (${Transaction.CONSENSUS_TIMESTAMP}, ${Transaction.PAYER_ACCOUNT_ID})`;
   ```
   This JOIN is added to **both** `cryptoTransferQuery` (line 556) and `tokenTransferQuery` (line 568). Without the `type` filter, these subqueries scan only their respective transfer tables; with it, each also JOINs the full `transaction` table.

**Net query structure per request with both filters:**
- `transactionOnlyQuery`: scans `transaction` + `entity_transaction` (UNION ALL)
- `cryptoTransferQuery`: scans `crypto_transfer` JOIN `transaction`
- `tokenTransferQuery`: scans `token_transfer` JOIN `transaction`
- Outer: FULL OUTER JOIN of all three subqueries

The `transaction` table is accessed **three times** (once directly, twice via JOIN in transfer subqueries). Each subquery applies `limitQuery` internally, but the scans and sorts still execute against large production tables before the limit is applied.

**Root cause:** The branch condition at line 555 is `creditDebitQuery || accountQuery` — the `type` filter has no effect on which JOIN path is selected, but it silently upgrades each transfer subquery from a single-table scan to a two-table JOIN. There is no guard that detects the combined cost of `accountQuery` + `transactionTypeQuery` and rejects or simplifies the query.

**Why existing checks are insufficient:**
- `limitQuery` (default 25, max 100 per `getResponseLimit()`) bounds the *output* rows but does not prevent the underlying multi-table scans and sorts from executing in full.
- `maxTimestampRange` / `bindTimestampRange` constrain the timestamp window but do not reduce the number of JOINs.
- The REST API server (`rest/server.js` line 132) registers the route with no rate-limiting middleware. The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` module for contract calls, not for the REST transactions endpoint.

### Impact Explanation
An attacker can hold open many concurrent DB connections, each executing a query that scans `transaction`, `entity_transaction`, `crypto_transfer`, and `token_transfer` — four of the largest tables in the mirror node database — with two FULL OUTER JOINs and multiple `DISTINCT ON … ORDER BY` sorts. On a production node with hundreds of millions of rows in these tables, each such query consumes significant CPU, I/O, and a DB connection slot. Flooding with concurrent requests causes connection pool exhaustion and query queue saturation, degrading or denying service to all API consumers. Severity: **High** (availability impact, no authentication required, no per-IP throttle).

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero knowledge beyond the public API documentation (both `account.id` and `transactiontype` are documented query parameters). The request is a single HTTP GET. It is trivially scriptable with `curl` or any HTTP load tool. The attacker does not need to know valid account IDs — any syntactically valid entity ID (e.g., `0.0.1`) will route into the expensive path regardless of whether matching rows exist, because the JOIN and sort still execute. Repeatability is unlimited.

### Recommendation
1. **Add per-IP rate limiting to the REST API** — apply a token-bucket or sliding-window limiter (analogous to the existing `ThrottleManagerImpl` in the `web3` module) as Express middleware before the transactions route in `rest/server.js`.
2. **Add a DB-level statement timeout** for REST API queries (e.g., PostgreSQL `statement_timeout`) so runaway queries are killed rather than held open.
3. **Detect the expensive filter combination** in `getTransactionTimestampsQuery()`: when both `accountQuery` and `transactionTypeQuery` are non-empty, consider pushing the `transactionTypeQuery` condition into the `transactionOnlyQuery` branch only (where the `transaction` table is already being scanned) rather than also forcing a JOIN in each transfer subquery, reducing the number of `transaction` table accesses from three to one.
4. **Enforce a tighter default timestamp range** when `accountQuery` is present, to limit the scan window on large tables.

### Proof of Concept

```bash
# Single expensive request (verify it hits the three-way FULL OUTER JOIN path)
curl "https://<mirror-node-host>/api/v1/transactions?account.id=0.0.98&transactiontype=CRYPTOTRANSFER&limit=100"

# Sustained flood (no authentication required)
for i in $(seq 1 200); do
  curl -s "https://<mirror-node-host>/api/v1/transactions?account.id=0.0.98&transactiontype=CRYPTOTRANSFER&limit=100" &
done
wait
```

Preconditions: none beyond network access to the mirror node REST API.
Trigger: the presence of both `account.id` (any valid entity ID) and `transactiontype` (any valid type string) in the query string.
Result: each request executes a query touching `transaction` (×3), `entity_transaction` (×1), `crypto_transfer` (×1), `token_transfer` (×1) with two FULL OUTER JOINs; concurrent flooding exhausts the DB connection pool.
