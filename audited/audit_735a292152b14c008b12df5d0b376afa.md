### Title
Unauthenticated Resource Exhaustion via Unbounded Concurrent Historical Hook Storage Queries with 100-Element IN-Clause and Full Timestamp Range

### Summary
The `getHookStorage()` endpoint in `HooksController.java` accepts up to 100 `key=` EQ parameters and a timestamp range spanning the full table history, with no authentication, no rate limiting, and no per-request cost cap. An unprivileged attacker can flood the endpoint with concurrent requests, each triggering an expensive `DISTINCT ON (key)` SQL query with a 100-element `IN` clause over an unbounded timestamp range, saturating the database connection pool and driving CPU/memory well above 30% of baseline.

### Finding Description

**Code path:**

- `HooksController.java` lines 104–130: `getHookStorage()` accepts `List<SlotRangeParameter> keys` annotated `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` (= 100) and `TimestampParameter[] timestamps` annotated `@Size(max = 2)`. No authentication annotation, no rate-limit annotation.
- `Constants.java` line 36: `MAX_REPEATED_QUERY_PARAMETERS = 100`.
- `HookServiceImpl.java` lines 56–59: `isHistorical()` is true whenever any timestamp parameter is present; control flows to `getHookStorageChange()`.
- `HookServiceImpl.java` lines 84–118: `getHookStorageChange()` calls `hookStorageChangeRepository.findByKeyInAndTimestampBetween(ownerId, hookId, keys, timestampLowerBound, timestampUpperBound, page)` with the full unfiltered `keys` list (line 104).
- `HookStorageChangeRepository.java` lines 41–63: the native query is `SELECT DISTINCT ON (key) … FROM hook_storage_change WHERE … AND key IN (:keys) AND consensus_timestamp BETWEEN :lower AND :upper`. `DISTINCT ON` requires a sort over all matching rows.
- `Bound.java` lines 86–97 / 63–74: if only a lower-bound timestamp is supplied (e.g. `timestamp=gte:0`), `getAdjustedLowerRangeValue()` returns 0 and `adjustUpperBound()` returns `Long.MAX_VALUE`, producing a full-history scan.

**Root cause:** The controller enforces only a syntactic upper bound on parameter count (100 keys, 2 timestamps) but imposes no semantic cost cap, no concurrency limit, no authentication, and no rate limiting. A single request with 100 EQ keys and `timestamp=gte:0` forces the database to evaluate `DISTINCT ON (key)` across every row in `hook_storage_change` for the given owner/hook, for 100 distinct keys. Concurrent identical requests multiply this cost linearly.

**Why existing checks fail:**
- `@Size(max = 100)` on `keys` is the *maximum allowed*, not a restriction — it explicitly permits the worst case.
- `@Size(max = 2)` on `timestamps` prevents more than two timestamp parameters but does not prevent a single `gte:0` that opens the full range.
- `@Max(MAX_LIMIT)` on `limit` caps result rows returned to the caller but does not cap the number of rows the database must scan before applying `DISTINCT ON` and the page limit.
- No `@PreAuthorize`, Spring Security filter, or throttle annotation is present on this endpoint.

### Impact Explanation
Each request with 100 EQ keys and `timestamp=gte:0` issues a full-history `DISTINCT ON` scan with a 100-element `IN` clause. On a table with significant history, this is a multi-second, high-I/O, high-CPU query. Sending N concurrent such requests occupies N database connections simultaneously. With a typical HikariCP pool of 10–20 connections, as few as 20 concurrent attackers can exhaust the pool, causing all other API calls to queue or time out. Database CPU and application-server thread usage rise proportionally. This satisfies the ≥30% resource increase threshold without brute force (each individual request is a legitimate, syntactically valid API call).

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. The attack requires only an HTTP client capable of sending concurrent GET requests — no credentials, no tokens, no special knowledge beyond the public API schema. The parameter structure (`key=0x…&key=0x…×tamp=gte:0`) is trivially constructed. The attack is repeatable indefinitely and can be automated with a single shell script or tool like `ab` or `wrk`. Any motivated external party can execute this.

### Recommendation
1. **Rate limiting**: Apply per-IP and per-account-ID rate limiting on this endpoint (e.g., via Spring's `RateLimiter`, Bucket4j, or an API gateway policy). Limit to a small number of requests per second per client.
2. **Cap effective query cost**: Reject or rewrite requests where `keys.size() > N` (e.g., N = 10) combined with an open-ended timestamp range. Require a bounded timestamp window (e.g., max span of 1 hour) when more than a threshold number of keys are requested.
3. **Use `keysInRange` in the query**: `HookServiceImpl.java` line 104 passes `keys` (unfiltered) to the repository instead of `keysInRange` (the already-computed filtered list). Change this to pass `keysInRange` to avoid unnecessary DB work.
4. **DB query timeout**: Set a statement timeout on the database connection (e.g., `statement_timeout = 5s` in PostgreSQL) so runaway queries are killed automatically.
5. **Authentication**: Require authentication for historical storage queries, which are inherently more expensive than current-state queries.

### Proof of Concept

```bash
# Build a request with 100 EQ key parameters and a full-history timestamp range
KEYS=""
for i in $(seq 1 100); do
  HEX=$(printf '%064x' $i)
  KEYS="${KEYS}&key=eq:${HEX}"
done

# Send 50 concurrent requests
for i in $(seq 1 50); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0${KEYS}" &
done
wait
```

Each concurrent request triggers `findByKeyInAndTimestampBetween()` with a 100-element `IN` clause and `consensus_timestamp BETWEEN 0 AND 9223372036854775807`, forcing a full-history `DISTINCT ON (key)` scan. With 50 concurrent requests and a typical pool of 10–20 DB connections, the pool is exhausted within seconds, all other API requests stall, and database CPU/memory spike well above 30% of baseline.
