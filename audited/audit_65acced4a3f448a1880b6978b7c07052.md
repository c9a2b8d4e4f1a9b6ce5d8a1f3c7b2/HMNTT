### Title
Unbounded URL Reflection in TracingMiddleware Enables Memory Exhaustion DoS

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` captures the full, user-controlled `request.URL.RequestURI()` (path + raw query string) and passes it verbatim into `fmt.Sprintf` to build a log message string, with no length cap. Because Go's `net/http` server's `MaxHeaderBytes` (default 1 MiB) bounds the request line rather than the query string alone, an unauthenticated attacker can craft requests whose URLs approach 1 MiB, causing the middleware to allocate an additional ~1 MiB string per in-flight request. With no rate limiting or connection cap configured, flooding the server with thousands of concurrent such requests can exhaust process memory and crash the node.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 47 and 52–53:

```go
path := request.URL.RequestURI()          // line 47 – full attacker-controlled URI
// ...
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, // line 52-53 – path copied into new string
    tracingResponseWriter.statusCode, time.Since(start))
```

**Root cause:** `request.URL.RequestURI()` returns the raw path plus the raw query string with no truncation. `fmt.Sprintf` then allocates a brand-new `string` whose size is proportional to `len(path)`. There is no `maxLen` guard, no truncation, and no sanitisation before this allocation.

**Why existing checks are insufficient:**

- `ReadHeaderTimeout: 3 s` and `ReadTimeout: 5 s` (configured in `rosetta/main.go` lines 224–226) limit *time*, not *bytes per request*.
- Go's `MaxHeaderBytes` (not explicitly set, so it defaults to 1 MiB) caps the total header block including the request line, meaning a URL of up to ~1 MiB is accepted and fully materialised in memory before `TracingMiddleware` ever runs.
- No rate-limiting, no per-IP connection cap, and no concurrency limit exist anywhere in the middleware chain (`rosetta/main.go` lines 217–219).
- The `tracingResponseWriter` also stores the full response body (`data []byte`, line 38) for every request, adding further per-request heap pressure.

**Exploit flow:**

1. Attacker opens N concurrent TCP connections to port 5700.
2. Each sends `GET /?<~1 MiB of junk query params> HTTP/1.1\r\n\r\n`.
3. Go's HTTP server accepts and parses each request (URL fits within `MaxHeaderBytes`).
4. `TracingMiddleware` allocates a new `~1 MiB` string for `message` per goroutine.
5. Because all N goroutines are in-flight simultaneously, N × ~2 MiB (original URL + message) is live on the heap.
6. Go's GC cannot reclaim this memory fast enough under sustained load; the process OOMs and is killed.

### Impact Explanation
A successful attack crashes the Rosetta API process on the targeted node. Because the Rosetta API is the mirror-node's Coinbase Rosetta interface (port 5700, publicly reachable by default), an attacker who repeats this against multiple nodes can take down ≥ 30 % of the network's Rosetta processing capacity without any authentication or privileged access. Node operators would need to restart the process manually; during the outage, block data and construction endpoints are unavailable.

### Likelihood Explanation
The attack requires only the ability to open TCP connections to port 5700 and send valid-looking HTTP GET requests with large query strings — no credentials, no special protocol knowledge, and no brute force. A single machine with a modest upload link (~100 Mbit/s) can sustain thousands of concurrent 1 MiB requests. The attack is trivially scriptable (e.g., `wrk`, `ab`, or a short Go/Python script) and fully repeatable. No patch or WAF is deployed in the default configuration.

### Recommendation
1. **Truncate `path` before logging** — cap it to a safe maximum (e.g., 512 bytes) inside `TracingMiddleware`:
   ```go
   path := request.URL.RequestURI()
   const maxPathLog = 512
   if len(path) > maxPathLog {
       path = path[:maxPathLog] + "…"
   }
   ```
2. **Set `MaxHeaderBytes` explicitly** on the `http.Server` in `rosetta/main.go` to a small value (e.g., 8 KiB) appropriate for a JSON-RPC API that never needs long URLs.
3. **Add a connection/rate limit** middleware (e.g., `golang.org/x/time/rate`) before `TracingMiddleware` to bound concurrent requests per IP.
4. **Remove the response-body buffer** in `tracingResponseWriter.Write` if the stored `data` field is not actually used downstream (it is assigned but never read in the current code).

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg/wrk) or equivalent
# Generate a ~900 KB query string
QSTRING=$(python3 -c "print('a=b&' * 112500)")   # ~900 KB

# Flood the Rosetta node with 500 concurrent connections for 30 seconds
wrk -t8 -c500 -d30s \
    "http://<rosetta-node>:5700/network/status?${QSTRING}"

# Expected result: process RSS grows rapidly; node OOMs or becomes
# unresponsive within seconds; subsequent legitimate requests time out.
```
