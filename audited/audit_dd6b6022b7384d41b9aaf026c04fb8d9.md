### Title
Integer Overflow in `limit` Cast Bypasses Page-Size Cap, Enabling Unbounded `getResultList()` OOM via Unauthenticated gRPC Subscription

### Summary
An unauthenticated caller can send a `ConsensusTopicQuery` with `limit = Long.MAX_VALUE` (a valid `uint64` value) and `consensusStartTime = 0`. An unsafe `(int)` cast in `PollingTopicMessageRetriever.poll()` overflows to `-1`, which causes `hasLimit()` to return `false` on the internally-constructed filter, bypassing the `setMaxResults` guard in `TopicMessageRepositoryCustomImpl.findByFilter()`. The resulting `getResultList()` call loads every topic message from epoch 0 into the JVM heap in a single query, enabling a denial-of-service against the entire gRPC service.

### Finding Description

**Entry point — proto and controller:**

`limit` is declared as `uint64` in the proto. [1](#0-0) 

`ConsensusController.toFilter()` maps it directly to `TopicMessageFilter.limit` with no upper-bound check. [2](#0-1) 

**Validation — no upper bound:**

`TopicMessageFilter.limit` carries only `@Min(0)`. `Long.MAX_VALUE` satisfies `>= 0` and passes Spring's `@Validated` check on `subscribeTopic()`. There is no `@Max` constraint. [3](#0-2) 

`startTime = 0` also passes `isValidStartTime()` because `0 <= DomainUtils.now()`. [4](#0-3) 

**Integer overflow — root cause:**

In `PollingTopicMessageRetriever.poll()`, the `long` limit is narrowed to `int`:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [5](#0-4) 

With `filter.getLimit() = Long.MAX_VALUE` and `total = 0`:
- `(int)(Long.MAX_VALUE - 0)` → **`-1`** (two's-complement overflow)
- `Math.min(-1, 1000)` → **`-1`**
- `newFilter` is built with `limit = -1`

**`hasLimit()` returns `false` for `-1`:** [6](#0-5) 

**`setMaxResults` is skipped — unbounded `getResultList()`:**

```java
if (filter.hasLimit()) {          // false → skipped
    typedQuery.setMaxResults((int) filter.getLimit());
}
return typedQuery.getResultList().stream(); // loads ALL rows
``` [7](#0-6) 

The comment on line 60 confirms `getResultList()` is used intentionally (cursor-based streaming was abandoned), meaning the entire result set is materialised into a `List<TopicMessage>` in heap memory before any element is emitted.

The identical overflow pattern exists in `PollingTopicListener.poll()` (the live-message path): [8](#0-7) 

**`subscribeTopic()` wires the historical retriever directly:** [9](#0-8) 

### Impact Explanation
A single malicious gRPC connection causes the JVM to attempt loading the complete message history of a topic (potentially millions of rows, each carrying up to 6 KiB of payload) into heap in one `getResultList()` call. This triggers an `OutOfMemoryError` on the gRPC service process, crashing it and terminating all active subscriber streams. Because the gRPC service is stateless and shared, every legitimate subscriber is evicted simultaneously. The attack is repeatable: each reconnect attempt by the attacker re-triggers the same query before the service can recover.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. The attacker needs only a standard gRPC client (e.g., `grpcurl`) and knowledge of one valid topic ID. Setting `limit = 9223372036854775807` and `consensusStartTime.seconds = 0` is a single-line change to any existing client. No credentials, tokens, or special network access are required. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add an `@Max` constraint** on `TopicMessageFilter.limit` (e.g., `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate cap such as `10_000_000`).
2. **Eliminate the unsafe `(int)` cast** in `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`. Use `Math.toIntExact()` (throws on overflow) or clamp explicitly: `int limit = (int) Math.min(filter.getLimit() - total, (long) context.getMaxPageSize())`.
3. **Never bypass `setMaxResults`** in `findByFilter`. Apply a hard server-side cap regardless of whether `hasLimit()` is true.
4. Consider rate-limiting or authenticating the `subscribeTopic` endpoint to reduce abuse surface.

### Proof of Concept

```bash
# Using grpcurl against a running mirror-node gRPC service
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0, "nanos": 0},
    "limit": 9223372036854775807
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Step-by-step:**
1. `limit = 9223372036854775807` (`Long.MAX_VALUE`) passes `@Min(0)` validation.
2. `startTime = 0` passes `isValidStartTime()`.
3. `PollingTopicMessageRetriever.poll()` computes `(int)(Long.MAX_VALUE - 0) = -1`.
4. `Math.min(-1, 1000) = -1`; new filter has `limit = -1`.
5. `hasLimit()` returns `false`; `setMaxResults` is not called.
6. `getResultList()` fetches every `topic_message` row with `consensus_timestamp >= 0` for the topic into heap.
7. On a topic with millions of messages, the JVM throws `OutOfMemoryError`, crashing the service.
