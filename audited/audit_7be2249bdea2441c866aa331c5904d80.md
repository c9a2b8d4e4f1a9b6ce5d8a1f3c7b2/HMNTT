### Title
Gas Throttle Fully Bypassed for Failed EVM Calls via 100% Default Refund

### Summary
`ThrottleProperties.gasLimitRefundPercent` defaults to `100`, meaning every EVM exceptional halt (e.g., call to a non-existent contract address) causes `restoreGasToBucket` to return the full consumed gas back to the `gasLimitBucket`. An unprivileged attacker can craft a loop of failing `eth_call` requests that never deplete the gas bucket, rendering the gas-per-second throttle entirely inoperative. The only remaining protection is the separate `requestsPerSecond` rate limit.

**Root cause:** The failed-call branch of `restoreGasToBucket` is designed to refund a *configurable* fraction of gas, but the default is 100 %. For any EVM exceptional halt (`!isSuccessful() && gasUsed == gasLimit`), the net token change in `gasLimitBucket` is zero: `tryConsume(N)` followed immediately by `addTokens(N)`. The bucket is never depleted.

**Failed assumption:** The design assumes operators will lower `gasLimitRefundPercent` to a value that imposes a real cost on failing calls. With the shipped default of 100, the gas throttle imposes zero cost on any request that triggers an exceptional halt.

### Impact Explanation

The `gasLimitBucket` is the only mechanism that limits total EVM computation per second. With the bypass active, an attacker can issue up to `requestsPerSecond` (default 500) requests per second, each carrying the maximum gas limit (15 000 000), and the gas bucket is never exhausted. The intended ceiling of `gasPerSecond` (default 7.5 ├Ś 10Ōü╣ gas/s) is completely unenforced for the failing-call pattern. Any operator who raises `requestsPerSecond` above the natural `gasPerSecond / maxGasLimit` ratio (currently 500) will expose the node to unbounded EVM load from unauthenticated callers.

### Likelihood Explanation

No privileges are required. Any HTTP client can issue `eth_call` to a random 20-byte address that has no deployed code; the EVM will halt exceptionally and consume all supplied gas. The attack is trivially scriptable, requires no on-chain state, and is repeatable at the full request-rate-limit speed. The default configuration ships with the vulnerability active.

### Recommendation

1. **Lower the default** of `gasLimitRefundPercent` to a value that imposes a real cost on failing calls (e.g., 10ŌĆō25 %). The current default of 100 makes the gas throttle a no-op for the most common DoS pattern.
2. **Decouple the refund logic** for exceptional halts vs. normal reverts: a revert that consumed real computation should receive a smaller refund than a trivial halt.
3. **Add a bucket floor check** in `restore()`: do not allow `addTokens` to restore more than the tokens actually consumed in the same request lifecycle (track consumed tokens per request).
4. **Document the interaction** between `gasLimitRefundPercent` and `requestsPerSecond` so operators understand that setting `gasLimitRefundPercent = 100` disables the gas throttle for failing calls.

```java
    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
```
