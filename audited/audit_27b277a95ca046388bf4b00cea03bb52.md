### Title
Unbounded URL Path Logged in Full by `TracingMiddleware`, Enabling Log Exhaustion via Unauthenticated Requests

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` captures `request.URL.RequestURI()` without any length cap and formats it into a log message that is written at `Info` level for every non-internal request. Because the `http.Server` in `main.go` sets no `MaxHeaderBytes`, Go's default 1 MB header budget applies, meaning any unauthenticated caller can send a ~1 MB URL and force the server to allocate and persist a ~1 MB log line per request. Under sustained fire this exhausts log storage and creates continuous GC pressure with no authentication barrier.

### Finding Description
**Exact code path:**

- `rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 47–58:
  ```go
  path := request.URL.RequestURI()          // line 47 – full, unbounded URI
  ...
  message := fmt.Sprintf("%s %s %s (%d) in %s",
      clientIpAddress, request.Method, path, ...) // line 52-53 – new string = path copy
  ...
  log.Info(message)                          // line 58 – written to stdout unconditionally
  ```
- `rosetta/main.go`, lines 220–227: `http.Server` is constructed with `Addr`, `Handler`, and timeout fields only — `MaxHeaderBytes` is **never set**, so Go's default `http.DefaultMaxHeaderBytes` (1 048 576 bytes, ~1 MB) governs the maximum request-line + header size.

**Root cause / failed assumption:** The middleware assumes the path is short (a normal Rosetta API path like `/network/list`). It makes no attempt to truncate or bound `path` before embedding it in the log message. There is no application-level URL-length guard anywhere in the rosetta Go code.

**Exploit flow:**
1. Attacker sends `POST /AAAA…(~900 KB of 'A')… HTTP/1.1` directly to port 5700.
2. Go's HTTP parser accepts it (within the 1 MB header budget).
3. `TracingMiddleware` stores the ~900 KB string in `path`, then `fmt.Sprintf` allocates a second ~900 KB string for `message`.
4. `log.Info(message)` writes ~900 KB to stdout (or whatever log sink is configured).
5. Repeat at the maximum connection rate the server allows.

**Why existing checks fail:**
- `ReadHeaderTimeout` (3 s default) slows individual connections but does not prevent many parallel connections.
- The Traefik `rateLimit`/`inFlightReq` middleware in the Helm chart (`charts/hedera-mirror-rosetta/values.yaml`, lines 149–166) is **opt-in** (`{{ if and .Values.global.middleware .Values.middleware }}`); it is absent in bare-metal / docker-compose deployments and is not enforced by the Go application itself.
- No `MaxHeaderBytes` is set, so Go does not reject oversized request lines before the handler chain runs.

### Impact Explanation
Every oversized request forces two large heap allocations (the path string already held by the parser, plus the `fmt.Sprintf` copy) and a large synchronous write to the log sink. Under sustained load this produces: (a) continuous log-storage growth that can fill the container's disk or exhaust a managed logging service quota; (b) elevated GC pressure that degrades response latency for legitimate users; (c) potential OOM if many concurrent large requests are in flight. The impact is griefing / availability degradation with no economic damage to the network — consistent with the stated scope.

### Likelihood Explanation
No authentication is required to reach the Rosetta API. The attack requires only a standard HTTP client capable of sending large request lines (e.g., `curl`, `python-requests`). It is trivially scriptable, repeatable, and requires no special knowledge of the protocol. The only natural throttle is network bandwidth and the server's `ReadHeaderTimeout`, neither of which prevents a sustained low-rate attack from a single host.

### Recommendation
1. **Truncate the path before logging** — add a constant cap (e.g., 512 bytes) in `TracingMiddleware`:
   ```go
   path := request.URL.RequestURI()
   const maxLoggedPathLen = 512
   if len(path) > maxLoggedPathLen {
       path = path[:maxLoggedPathLen] + "…"
   }
   ```
2. **Set `MaxHeaderBytes`** on the `http.Server` in `main.go` to a value appropriate for the Rosetta API (e.g., 8 KB), rejecting oversized request lines before they reach any handler.
3. **Enforce rate limiting at the application layer** (not only as an optional Helm/Traefik add-on) so that deployments without Kubernetes are also protected.

### Proof of Concept
```bash
# Generate a ~900 KB path segment
LONG_PATH=$(python3 -c "print('A' * 900000)")

# Send to the Rosetta server (no credentials needed)
curl -v "http://<rosetta-host>:5700/${LONG_PATH}" -X POST \
     -H "Content-Type: application/json" -d '{}'

# Observe in server logs: a ~900 KB line is written for every request.
# Repeat in a loop to exhaust log storage:
for i in $(seq 1 1000); do
  curl -s "http://<rosetta-host>:5700/${LONG_PATH}" -X POST \
       -H "Content-Type: application/json" -d '{}' &
done
wait
```
Each iteration writes approximately 900 KB to the log sink. 1 000 concurrent requests produce ~900 MB of log output with no authentication required.
