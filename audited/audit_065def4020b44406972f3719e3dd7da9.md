### Title
Unenforced HTTP Transport Allows MITM Injection of Malicious gRPC Node Endpoints

### Summary
The `buildNetworkFromMirrorNodes` function constructs an `http.Client` with no TLS enforcement and accepts any URL scheme supplied via configuration, including plain `http://`. An attacker with a network-level position (DNS poisoning, ARP spoofing, or BGP hijacking) can intercept the unencrypted HTTP response and inject crafted `nodeEntry` records, causing the Hiero SDK client to route all subsequently signed transactions to attacker-controlled gRPC endpoints without any owner approval or integrity check.

### Finding Description
**Code path:**

- `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, line 46:
  ```go
  httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
  ```
  No custom `Transport` is set. Go's default transport accepts both `http://` and `https://` schemes; it does **not** enforce TLS.

- Lines 37–44: the request URL is built directly from `cfg.mirrorRest`, which is populated from the `HIERO_MIRROR_PINGER_REST` environment variable. The documented default example in `config.go` line 37 is `http://mirror-rest:5551` — a plain HTTP URL.

- `fetchMirrorNodeNetwork()`, lines 79–98: the HTTP response body is decoded directly into `nodesEnvelope` with no signature, HMAC, or any integrity check.

- Lines 100–124: every `serviceEndpoint` from the decoded payload is inserted verbatim into the `network` map as `host:port → AccountID`.

- `sdk_client.go` line 22: `hiero.ClientForNetwork(netmap)` consumes this map, and line 45 attaches the operator private key to the resulting client.

- `transfer.go` lines 29–33: `TransferTransaction.Execute(client)` signs and submits transactions to whatever gRPC addresses are in that map.

**Root cause:** The code assumes the mirror node REST endpoint is trusted and reachable only over a secure channel. Neither assumption is enforced in code. There is no scheme check, no TLS requirement, no certificate pinning, and no response integrity verification.

**Why existing checks fail:** The only validation applied to the response is structural JSON decoding and a check that `NodeAccountID` is non-empty and parseable (lines 103–109). Neither check detects attacker-supplied host/port values.

### Impact Explanation
An attacker who successfully injects malicious `nodeEntry` data gains the ability to receive every signed `TransferTransaction` the pinger submits. The operator private key signs the transaction body before it is sent, so the attacker receives a fully signed, valid transaction. Consequences include: silent transaction suppression (DoS of the liveness probe), transaction replay to the real network, and full visibility into transaction metadata (amounts, account IDs, timing). Because the network map is built once at startup (`newClient` → `buildNetworkFromMirrorNodes`), a single successful injection persists for the entire lifetime of the process.

### Likelihood Explanation
The precondition is a network-level MITM position between the pinger pod and the mirror REST service. In Kubernetes deployments this is achievable via: DNS record poisoning of the `mirror-rest` service name (no cluster-DNS authentication by default), ARP spoofing on a flat pod network, or a compromised sidecar/init container. The default example URL is `http://` (not `https://`), making it likely that many deployments run over plain HTTP. The attack is repeatable on every process restart and requires no credentials.

### Recommendation
1. **Enforce HTTPS at startup:** Reject any `mirrorRest` URL whose scheme is not `https` in `loadConfig()`.
2. **Configure TLS verification explicitly:** Set a custom `http.Transport` with `TLSClientConfig` that pins the expected CA or certificate.
3. **Validate the URL scheme before the request:** In `buildNetworkFromMirrorNodes`, check `strings.HasPrefix(url, "https://")` and return a fatal error otherwise.
4. **Consider response integrity:** If the mirror node can sign its `/network/nodes` responses (e.g., with a known public key), verify the signature before consuming the payload.

### Proof of Concept
1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
2. On the same network segment, poison DNS so `mirror-rest` resolves to an attacker-controlled host, or ARP-spoof the real mirror-rest IP.
3. On the attacker host, serve the following JSON at `GET /api/v1/network/nodes`:
   ```json
   {
     "nodes": [{
       "node_account_id": "0.0.3",
       "service_endpoints": [{
         "domain_name": "attacker.example.com",
         "ip_address_v4": "",
         "port": 50211
       }]
     }],
     "links": {"next": null}
   }
   ```
4. Start (or restart) the pinger process. `buildNetworkFromMirrorNodes` fetches the poisoned response over plain HTTP with no integrity check.
5. `hiero.ClientForNetwork` is initialized with `attacker.example.com:50211 → 0.0.3`.
