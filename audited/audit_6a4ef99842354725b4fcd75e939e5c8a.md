### Title
Unauthenticated Repeated Queries to `getAirdrops()` Cause Unbounded DB Re-execution Due to Missing Caching and Rate Limiting

### Summary
The `rest-java` Spring Boot service exposes `GET /api/v1/accounts/{id}/airdrops/outstanding` and `/pending` endpoints backed by `TokenAirdropServiceImpl.getAirdrops()`, which has no response caching, no `@Cacheable` annotation, and no rate limiting. Any unprivileged user can flood these endpoints with identical parameters, forcing the database to re-execute the same expensive query on every request with no mitigation.

### Finding Description
**Code path:**
- Controller: `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TokenAirdropsController.java`, lines 66–114 — calls `service.getAirdrops(request)` on every request with no cache check.
- Service: `rest-java/src/main/java/org/hiero/mirror/restjava/service/TokenAirdropServiceImpl.java`, lines 19–22 — calls `entityService.lookup()` then `repository.findAll(request, id)` unconditionally, no `@Cacheable` annotation.
- Repository: `rest-java/src/main/java/org/hiero/mirror/restjava/repository/TokenAirdropRepository.java` — extends `CrudRepository` and `TokenAirdropRepositoryCustom` with no `@Cacheable` annotation on `findAll`.

**Root cause:** The `rest-java` module has no caching infrastructure. The `rest` (Node.js) service has a Redis-based response cache middleware (`rest/middleware/responseCacheHandler.js`, lines 40–87) and a `responseCacheUpdateHandler`, but this is a completely separate process from `rest-java`. The `rest-java` Spring Boot application (`RestJavaApplication.java`) has no `@EnableCaching`, no cache configuration, and no rate limiting (unlike the `web3` module which has `ThrottleConfiguration` with `bucket4j`). Every HTTP GET to the airdrop endpoints unconditionally executes a DB query.

**Exploit flow:**
1. Attacker identifies a high-volume `accountId` with many pending airdrops (publicly observable on-chain data).
2. Attacker sends a high-frequency loop of identical `GET /api/v1/accounts/{id}/airdrops/outstanding?limit=100` requests.
3. Each request traverses the full stack: controller → `getAirdrops()` → `repository.findAll()` → DB query against `token_airdrop` table.
4. No cache intercepts the request; no rate limiter throttles the attacker; the DB executes the same scan repeatedly.

**Why existing checks fail:** The `rest-java` config directory (`rest-java/src/main/java/org/hiero/mirror/restjava/config/`) contains only `JacksonConfiguration`, `LoggingFilter`, `MetricsConfiguration`, `MetricsFilter`, `NetworkProperties`, `RestJavaConfiguration`, and `WebMvcConfiguration` — no throttle or cache configuration. The `web3` module's `ThrottleConfiguration`/`ThrottleManagerImpl` are scoped to that module only and do not apply to `rest-java`.

### Impact Explanation
An attacker can sustain a high request rate against the airdrop endpoints, causing the DB to repeatedly execute range-scan queries on the `token_airdrop` table. For accounts with large numbers of pending airdrops, these queries are non-trivial. With no caching and no rate limiting, the DB CPU load scales linearly with attacker request rate, making a sustained >30% CPU increase realistic with modest tooling (e.g., `ab`, `wrk`, or a simple script). This degrades service for all users of the mirror node.

### Likelihood Explanation
No authentication is required. No API key, no login, no privileged role. The endpoint is publicly accessible. The attacker needs only to know a valid `accountId` (trivially discoverable from public blockchain data) and a tool capable of sending repeated HTTP GET requests. This is repeatable indefinitely and requires no special knowledge beyond the public API documentation.

### Recommendation
1. Add a `@Cacheable` annotation on `TokenAirdropServiceImpl.getAirdrops()` or on the `TokenAirdropRepositoryCustom.findAll()` implementation using a short TTL (e.g., 2–5 seconds) to absorb burst traffic for identical queries.
2. Introduce rate limiting in `rest-java` analogous to the `web3` module's `ThrottleConfiguration` (bucket4j), applying per-IP or global request-per-second limits to the airdrop endpoints.
3. Alternatively, route `rest-java` endpoints through the same Redis response cache layer used by the Node.js `rest` service, or add Spring's `spring-boot-starter-cache` with Caffeine/Redis to `rest-java`.

### Proof of Concept
```bash
# Identify a high-volume accountId (e.g., 0.0.12345) with many pending airdrops
# Then flood the endpoint:
while true; do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.12345/airdrops/outstanding?limit=100" > /dev/null &
done
# Or with wrk:
wrk -t8 -c200 -d60s "http://<mirror-node-host>/api/v1/accounts/0.0.12345/airdrops/outstanding?limit=100"
# Monitor DB CPU: each request triggers a full token_airdrop table scan for the account.
# No cache hit, no rate limit rejection — DB CPU rises proportionally to request rate.
```
