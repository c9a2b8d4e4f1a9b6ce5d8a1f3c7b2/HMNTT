### Title
Missing File ID Whitelist Validation in REST `/api/v1/network/nodes` Allows Arbitrary File ID Injection Returning Empty Node List

### Summary
The REST Java `getNodes()` endpoint in `NetworkController` only validates that the `file.id` parameter uses the equality operator, but never validates that the supplied file ID is one of the two legitimate address book file IDs (`0.0.101` or `0.0.102`). An unauthenticated attacker can supply `file.id=eq:0` (or any non-existent file ID), causing the SQL query to find no matching address book and return an empty node list. Clients consuming this response may interpret zero nodes as a complete network partition.

### Finding Description

**Exact code path:**

In `NetworkController.java`, `getNodes()` performs only an operator check: [1](#0-0) 

The check at line 155–157 only rejects non-`EQ` operators. Any numeric entity ID value — including `0` — passes through.

`getAddressBookFileId()` in `NetworkServiceImpl.java` then blindly uses whatever value was supplied: [2](#0-1) 

That raw `fileId` is forwarded directly to the repository: [3](#0-2) 

The SQL CTE in `NetworkNodeRepository` filters `address_book` by `file_id = :fileId`: [4](#0-3) 

When no address book row exists for the supplied file ID, the CTE is empty, the join produces zero rows, and the response is `{"nodes": [], "links": {}}`.

**Root cause / failed assumption:** The code assumes callers will only supply valid address book file IDs. The gRPC path correctly enforces a whitelist: [5](#0-4) 

The REST Java path has no equivalent guard.

**Parser allows `0`:** `EntityIdRangeParameter.valueOf("eq:0")` parses successfully — the `filter(n -> n >= 0)` predicate accepts `0`, producing entity ID `0.0.0` (encoded as `0`): [6](#0-5) 

### Impact Explanation
Any unauthenticated client can force the endpoint to return an empty node list. SDKs or monitoring tools that call `GET /api/v1/network/nodes` to bootstrap their peer list will receive zero nodes and may halt all transaction submission, treating the result as a complete network partition. The impact is a denial-of-service at the application layer for any consumer that trusts this endpoint without independent verification.

### Likelihood Explanation
No authentication or special privilege is required. The parameter is documented in the OpenAPI spec as a standard query parameter. The attack is a single HTTP GET request, trivially repeatable, and requires no prior knowledge beyond the public API documentation. Any external party can reproduce it.

### Recommendation
Add a file ID whitelist check in `NetworkServiceImpl.getAddressBookFileId()` (or in the controller before calling the service), mirroring the gRPC implementation:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    long fileId = request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();

    var validIds = Set.of(
            systemEntity.addressBookFile101().getId(),
            systemEntity.addressBookFile102().getId());
    if (!validIds.contains(fileId)) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return fileId;
}
```

This matches the guard already present in the gRPC `NetworkServiceImpl` at lines 50–59.

### Proof of Concept

```
# Returns empty node list — no authentication required
curl -s "https://<mirror-node-host>/api/v1/network/nodes?file.id=eq:0"

# Expected (vulnerable) response:
# {"nodes":[],"links":{"next":null}}

# Legitimate request for comparison:
curl -s "https://<mirror-node-host>/api/v1/network/nodes?file.id=eq:102"
# Returns actual network nodes
```

Preconditions: none — public endpoint, no credentials needed.
Trigger: supply any file ID that has no corresponding row in the `address_book` table (e.g., `0`, `1`, `999`).
Result: empty `nodes` array with HTTP 200, indistinguishable from a real "no nodes available" state.
