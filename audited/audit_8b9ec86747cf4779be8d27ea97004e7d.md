### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent GET /blocks Requests

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` requires no authentication and has no application-level rate limiting. With a default PostgreSQL connection pool of only 10 connections and a `statementTimeout` of 20 seconds, a flood of concurrent unauthenticated `GET /api/v1/blocks` requests can saturate the entire pool, causing all REST API endpoints on the affected mirror node instance to queue or time out for the duration of the attack.

### Finding Description

**Code path:**

`rest/controllers/blockController.js` lines 101–112 (`getBlocks`):
```js
getBlocks = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedBlockParameters);
  const formattedFilters = this.extractSqlFromBlockFilters(filters);
  const blocks = await RecordFileService.getBlocks(formattedFilters);
  ...
};
```
With no query parameters, `extractSqlFromBlockFilters` returns `{order:'desc', orderBy:'consensus_end', limit:25, whereQuery:[]}` (confirmed by test at line 37 of `blockController.test.js`).

`rest/service/recordFileService.js` lines 149–162 (`getBlocks`):
```js
const query = RecordFileService.blocksQuery +
  `${where} order by ${filters.orderBy} ${filters.order} limit ${filters.limit}`;
const rows = await super.getRows(query, params);
```
With no filters, this executes: `SELECT count, hash, name, ... FROM record_file ORDER BY consensus_end DESC LIMIT 25` — a full-table scan with no WHERE clause, holding a DB connection for the full query duration.

**Pool configuration** (`rest/dbpool.js` lines 7–16):
```js
const poolConfig = {
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default: 20000ms
  max: config.db.pool.maxConnections,                         // default: 10
  statement_timeout: config.db.pool.statementTimeout,         // default: 20000ms
};
```
Default `maxConnections` is **10** (documented in `docs/configuration.md` line 556).

**No rate limiting on the REST API blocks route.** `rest/server.js` shows the middleware stack for the blocks route is: `requestLogger` → `authHandler` (auth is optional; unauthenticated requests pass through) → optional metrics → optional Redis cache check → route handler. There is no `express-rate-limit` or equivalent middleware. The throttle mechanisms in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java` and `ThrottleManagerImpl.java` apply exclusively to the `web3` module (`/contracts/call`), not to the REST API. The Traefik `rateLimit` in `charts/hedera-mirror-rosetta/values.yaml` applies only to the Rosetta service. The Redis response cache (`cache.response.enabled`) defaults to `false` (`docs/configuration.md` line 549), so cache bypass is the default.

**Root cause:** The REST API `/blocks` endpoint is publicly accessible with no per-IP or global request rate limit at the application layer, and the shared pg connection pool has a small default ceiling (10). Each concurrent request holds a connection for up to 20 seconds (statement timeout). Once all 10 connections are occupied, every subsequent request across all REST API endpoints queues waiting for a free connection, and times out after 20 seconds (`connectionTimeout`).

### Impact Explanation

Saturating the 10-connection pool blocks **all** REST API endpoints (accounts, transactions, tokens, etc.) on the targeted mirror node instance, not just `/blocks`, because all share the same `global.pool` / `global.primaryPool` initialized in `rest/dbpool.js`. The REST API becomes effectively unavailable for the duration of the attack. In a multi-instance deployment, an attacker targeting all instances simultaneously degrades ≥30% of REST API processing capacity with no special privileges. The `statementTimeout` of 20 seconds means each wave of 10 requests holds the pool for up to 20 seconds, requiring only ~10 concurrent persistent connections to maintain the saturation continuously.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only the ability to send HTTP GET requests. No exploit code is needed — standard tools (`ab`, `wrk`, `curl` in parallel) suffice. The endpoint is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml` lines 420–440). The default pool size of 10 is very small, making saturation trivial. The attack is repeatable and can be sustained indefinitely from a single machine or distributed across IPs to bypass any upstream IP-based rate limiting.

### Recommendation

1. **Add application-level rate limiting** to the REST API (e.g., `express-rate-limit`) applied globally or specifically to high-cost endpoints like `/blocks`, `/transactions`, `/accounts`.
2. **Increase the default `maxConnections`** pool size or document a minimum recommended value for production deployments.
3. **Enable the Redis response cache** by default for read-heavy, low-volatility endpoints like `/blocks` to serve repeated identical requests without hitting the DB.
4. **Add a concurrency limiter** (e.g., `express-slow-down` or an in-flight request counter) to shed load before the pool is exhausted.
5. **Enforce Traefik `inFlightReq` and `rateLimit` middleware** on the REST API ingress, analogous to what is already configured for the Rosetta service in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166.

### Proof of Concept

```bash
# Send 50 concurrent unauthenticated GET /blocks requests (no query params)
# Requires: wrk or ab; target: a mirror node REST API instance

wrk -t10 -c50 -d60s http://<mirror-node-host>:5551/api/v1/blocks

# Simultaneously, observe that other endpoints time out:
for i in $(seq 1 20); do
  curl -o /dev/null -s -w "%{http_code} %{time_total}\n" \
    http://<mirror-node-host>:5551/api/v1/transactions &
done
wait
# Expected: HTTP 500 or connection timeout errors on /transactions
# while /blocks requests are in-flight, confirming pool exhaustion
```

With 50 concurrent connections against a pool of 10, 40 requests immediately queue. Each queued request waits up to 20 seconds for a connection. During this window, all other REST API queries on the same instance are starved of connections, producing errors or extreme latency for legitimate users.
