### Title
Unbounded Log Growth via Unauthenticated Request Flooding in `TracingMiddleware`

### Summary
`TracingMiddleware` unconditionally logs every HTTP request at `Info` level, including the full user-controlled URL path and a spoofable client IP header, with no rate limiting or log throttling at the application layer. The Traefik-level rate limiting middleware that would otherwise cap request volume is disabled by default (`global.middleware: false`). An unprivileged attacker can flood the server with high-volume requests carrying long paths and arbitrary `X-Forwarded-For` values to exhaust log storage.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 43–61:

```go
func TracingMiddleware(inner http.Handler) http.Handler {
    return http.HandlerFunc(func(responseWriter http.ResponseWriter, request *http.Request) {
        start := time.Now()
        clientIpAddress := getClientIpAddress(request)   // user-controlled header
        path := request.URL.RequestURI()                  // user-controlled path
        ...
        message := fmt.Sprintf("%s %s %s (%d) in %s",
            clientIpAddress, request.Method, path, ...)

        if internalPaths[path] {
            log.Debug(message)
        } else {
            log.Info(message)   // every non-internal path → Info log, unconditionally
        }
    })
}
```

`getClientIpAddress()` (lines 63–74) trusts `X-Real-IP` then `X-Forwarded-For` without any length validation or sanitization, so an attacker controls both the IP field and the path field in every log line.

**Root cause:** There is no rate limiting, log throttling, deduplication, or path/header length cap inside `TracingMiddleware`. Every request — including 404s to arbitrary paths — produces one `Info`-level log entry whose size is proportional to the attacker-supplied path and header values.

**Why existing checks fail:**

The Traefik middleware chain (`inFlightReq: amount: 5`, `rateLimit: average: 10`) is defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166, but the guard condition in `charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3 is:

```yaml
{{ if and .Values.global.middleware .Values.middleware -}}
```

`global.middleware` defaults to `false` (values.yaml line 95), so **the Traefik middleware CRD is never created in a default deployment**. No rate limiting exists at any layer.

**Log destination amplifies impact:** Both supervisord configs (`rosetta/container/supervisord.conf` and `supervisord-offline.conf`) set `stdout_logfile_maxbytes=0`, meaning supervisord imposes no cap. Logs flow to the container's stdout, which in Docker is stored in `/var/lib/docker/containers/<id>/<id>-json.log` and in Kubernetes on the node's ephemeral storage — both finite and shared with other workloads.

### Impact Explanation
A sustained flood of HTTP requests with long paths (e.g., 8 KB paths, the Go `net/http` default max URL length) and long spoofed `X-Forwarded-For` values produces log lines of ~8+ KB each. At 1,000 req/s this generates ~8 MB/s of log data. Within minutes to hours, the node's log partition or Docker daemon storage is exhausted, causing the Rosetta pod and potentially co-located pods to crash or become unresponsive. This is a full availability DoS against the Rosetta API service.

### Likelihood Explanation
No authentication is required. The Rosetta API is publicly exposed via ingress (values.yaml lines 119–136). The attack requires only a standard HTTP client capable of sending concurrent requests — trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The default `global.middleware: false` means the rate-limiting protection is absent in any out-of-the-box deployment. The attack is repeatable and requires no special knowledge of the application.

### Recommendation
1. **Enable the Traefik middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml` line 95, or document it as a required production setting.
2. **Add application-level log rate limiting**: In `TracingMiddleware`, use a token-bucket or leaky-bucket limiter (e.g., `golang.org/x/time/rate`) to cap log output per time window.
3. **Cap path and header length before logging**: Truncate `path` and `clientIpAddress` to a safe maximum (e.g., 512 bytes) before constructing the log message.
4. **Configure Docker/Kubernetes log rotation**: Set `--log-opt max-size` and `--log-opt max-file` on Docker, or configure `containerLogMaxSize`/`containerLogMaxFiles` in the kubelet, as a defense-in-depth measure.

### Proof of Concept

```bash
# Generate a long path (~8000 chars)
LONG_PATH=$(python3 -c "print('/rosetta/' + 'A'*7990)")

# Flood the server with 1000 concurrent connections, no auth required
wrk -t 10 -c 1000 -d 60s \
    -H "X-Forwarded-For: $(python3 -c "print('1.2.3.4, ' * 200)")" \
    "http://<rosetta-host>:5700${LONG_PATH}"

# On the host, observe log growth:
# Docker: watch -n1 'du -sh /var/lib/docker/containers/*/*-json.log'
# K8s node: watch -n1 'df -h /var/log'
```

Each request produces one `Info` log line containing the full path and spoofed IP chain. With `global.middleware: false` (the default), no Traefik rate limiting intercepts the flood.
