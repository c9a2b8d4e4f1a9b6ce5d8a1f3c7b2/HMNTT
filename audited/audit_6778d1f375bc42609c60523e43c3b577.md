### Title
Cache Stampede in `getCachedTokens` Enables DB Connection Pool Exhaustion via Unauthenticated Concurrent Requests

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` performs no deduplication or locking for concurrent in-flight DB queries on the same uncached token ID. An unauthenticated attacker can send many concurrent requests to `/api/v1/accounts/:id/tokens?token.id=eq:X` for a valid token ID not yet in the in-process LRU cache, causing every concurrent request to independently issue `tokenCacheQuery` against the database. Combined with the default DB connection pool cap of 10 and the complete absence of rate limiting on the REST API, this exhausts the pool and queues or times out all legitimate requests.

### Finding Description

**Code path:**

`getTokenRelationships` (tokenController.js:66-92) → `TokenService.getTokenAccounts` (tokenService.js:96-115) → `getCachedTokens` (tokenService.js:136-161).

**Root cause — no cache-miss coalescing:**

```
// tokenService.js lines 136-161
async getCachedTokens(tokenIds) {
  const uncachedTokenIds = [];
  tokenIds.forEach((tokenId) => {
    const cachedToken = tokenCache.get(tokenId);   // ← pure synchronous LRU read
    if (cachedToken) { ... } else { uncachedTokenIds.push(tokenId); }
  });
  if (uncachedTokenIds.length === 0) return cachedTokens;
  const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]); // ← async DB call
  ...
}
```

Because the cache check and the DB call are not atomic and there is no in-flight deduplication (no `Promise` coalescing, no mutex, no "pending" sentinel), N concurrent requests that all arrive before any one of them completes the DB round-trip will each independently find the token absent from the LRU and each independently issue `tokenCacheQuery`.

**`tokenCache` has no TTL.** It is a `quick-lru` initialised with only `maxSize`:

```
// tokenService.js lines 12-14
const tokenCache = new quickLru({
  maxSize: config.cache.token.maxSize,   // default 100 000, no expiry
});
```

Entries never expire; they are only evicted when the cache is full. This means the attack window per token ID is only the first burst, but with millions of tokens on Hedera mainnet the attacker can rotate through fresh IDs indefinitely.

**DB pool is tiny and unguarded.** Default `maxConnections = 10` (docs line 556, `dbpool.js` line 14). Each `getTokenRelationships` call consumes up to three sequential pool connections: `isValidAccount`, `getTokenRelationshipsQuery`, and `tokenCacheQuery`. With 10 connections, only 3–4 full requests can be in-flight simultaneously.

**No rate limiting on the REST API.** `server.js` (lines 67-144) registers: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler` — no rate-limit or concurrency-limit middleware. The throttling found in the codebase (`ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) applies exclusively to the web3 Java API, not to this Node.js REST service.

**Exploit flow:**

1. Attacker identifies any valid `accountId` and any valid `tokenId` associated with that account (all public on-chain data, queryable via the same API).
2. Attacker sends N (e.g. 50–100) concurrent HTTP GET requests:
   `GET /api/v1/accounts/{accountId}/tokens?token.id=eq:{tokenId}`
3. All N requests simultaneously call `getCachedTokens`; all find `tokenId` absent from the LRU (synchronous check, no lock).
4. All N requests issue `tokenCacheQuery` (`SELECT decimals, freeze_status, kyc_status, token_id FROM token WHERE token_id = any($1)`) against the DB.
5. The pool (max 10 connections) is immediately saturated; remaining requests queue with a 20-second `connectionTimeout`.
6. Legitimate user requests queue behind the attacker's flood and either wait 20 s or time out.
7. Attacker rotates to the next uncached `tokenId` before the cache is populated, sustaining the exhaustion.

### Impact Explanation

The DB connection pool is the single shared resource for all REST API queries on a node. Exhausting it blocks every endpoint — not just `/accounts/:id/tokens` — because all handlers share the same `global.pool`. With a default cap of 10 connections and no rate limiting, an attacker with a modest HTTP client can hold the pool saturated indefinitely, causing ≥30% (in practice near-100%) of REST API requests to queue and time out, effectively taking the mirror-node REST service offline for legitimate users.

### Likelihood Explanation

- **No authentication required.** The endpoint is fully public.
- **No rate limiting.** Confirmed absent from `server.js` middleware chain.
- **Trivially scriptable.** A single `curl`/`ab`/`wrk` invocation with 50 concurrent workers is sufficient.
- **Precondition is trivially met.** Any valid `(accountId, tokenId)` pair is discoverable via the same public API.
- **Sustainable.** Hedera mainnet has millions of token IDs; the attacker can cycle through fresh IDs faster than the 100 000-entry LRU fills.

### Recommendation

1. **Coalesce in-flight cache misses.** Maintain a `Map<tokenId, Promise>` of pending DB fetches; if a second request arrives for the same uncached ID while a fetch is in-flight, return the existing `Promise` instead of issuing a new query.
2. **Add rate limiting to the REST API.** Apply a per-IP or global concurrency/rate limit middleware (e.g. `express-rate-limit`) in `server.js` before route handlers.
3. **Increase the DB connection pool** or use a connection-queue depth limit that rejects (HTTP 429/503) rather than indefinitely queuing requests.
4. **Consider a TTL on the token cache** to bound stale data, but this is secondary to the stampede fix.

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
