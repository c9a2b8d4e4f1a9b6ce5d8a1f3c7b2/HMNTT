### Title
Unbounded Historical Message Replay via Epoch-Zero `consensusStartTime` Enables Bandwidth Exhaustion DoS

### Summary
Any unauthenticated caller can invoke `subscribeTopic()` with `consensusStartTime` set to epoch zero (seconds=0, nanos=0) and `limit=0`, causing the server to stream the entire historical message archive of a topic with no upper bound. Because no per-client rate limiting, no mandatory limit, and no minimum start-time floor exist in the gRPC module, a single or coordinated set of connections can saturate server bandwidth and database resources, starving legitimate real-time subscribers of fund-critical messages.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any `ConsensusTopicQuery` from any caller with no authentication or rate-limiting check. It delegates directly to `toFilter()` (lines 55–73):

```java
// ConsensusController.java:56
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
// ...
if (query.hasConsensusStartTime()) {
    long startTime = convertTimestamp(query.getConsensusStartTime());
    filter.startTime(startTime);   // epoch zero → 0 nanoseconds
}
```

`convertTimestamp()` (lines 76–81) converts `{seconds:0, nanos:0}` to `DomainUtils.timestampInNanosMax({0,0}) = 0`. This value is passed directly to `TopicMessageFilter`.

**Failed validation in `TopicMessageFilter`** (lines 48–51):

```java
@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();   // 0 <= now() → always true
}
```

`@Min(0)` on `startTime` also passes for 0. There is no lower-bound floor (e.g., "must be within the last N days").

**No limit enforcement:** `limit=0` causes `hasLimit()` to return `false` (line 39 of `TopicMessageFilter`), so `TopicMessageServiceImpl` never calls `flux.take()` (lines 83–85).

**Unbounded retrieval:** `topicMessageRetriever.retrieve(filter, true)` (line 63 of `TopicMessageServiceImpl`) enters `PollingTopicMessageRetriever` in throttled mode where `numRepeats = Long.MAX_VALUE` (line 99 of `PollingTopicMessageRetriever`). It pages through all historical rows in chunks of `maxPageSize=1000`, polling every 2 seconds, until the last page returns fewer than 1000 rows — i.e., until the entire history is exhausted.

**No gRPC-level rate limiting:** Unlike the web3 module which has `ThrottleManagerImpl` with `Bucket4j`, the gRPC module has no equivalent. `NettyProperties.maxConcurrentCallsPerConnection = 5` limits calls per single TCP connection but does not cap total connections or total concurrent subscriptions globally.

### Impact Explanation

A high-volume HCS topic (e.g., a DeFi price-feed topic with millions of messages, each up to 6 KB per the proto spec) can produce terabytes of historical data. A single subscription from epoch zero with no limit forces the server to:
1. Issue continuous paginated DB queries (1000 rows/page, every 2 s) for the full history.
2. Serialize and stream every `ConsensusTopicResponse` over the gRPC connection.

Multiple concurrent connections (each with up to 5 calls per `maxConcurrentCallsPerConnection`) from different source IPs multiply this effect. Legitimate real-time subscribers — including those monitoring fund-critical HCS topics — experience stream starvation, increased latency, or connection refusal as bandwidth and DB connection pool are exhausted.

### Likelihood Explanation

Preconditions are minimal: the attacker needs only a valid `topicID` (publicly observable on-chain) and a standard gRPC client. No credentials, tokens, or privileged access are required. The attack is trivially scriptable with `grpcurl` or any protobuf client, is repeatable from any IP, and can be amplified by opening multiple TCP connections. The acceptance test suite itself demonstrates the exact attack pattern (`"1970-01-01T00:00:00.00Z"` as `startTimestamp`) confirming the path is exercised in production scenarios.

### Recommendation

1. **Enforce a minimum start-time floor:** In `TopicMessageFilter.isValidStartTime()` or in `ConsensusController.toFilter()`, reject or clamp `startTime` values older than a configurable maximum lookback window (e.g., 7 days):
   ```java
   return startTime >= DomainUtils.now() - maxLookbackNanos && startTime <= DomainUtils.now();
   ```
2. **Enforce a mandatory maximum limit:** If `query.getLimit() == 0`, apply a server-side default cap (e.g., 10,000 messages) rather than streaming indefinitely.
3. **Add per-connection/per-IP rate limiting** to the gRPC layer analogous to the `ThrottleManagerImpl` in the web3 module.
4. **Cap total concurrent historical-retrieval subscriptions** globally via a semaphore or bounded scheduler to prevent DB pool exhaustion.

    private TopicMessageFilter toFilter(ConsensusTopicQuery query) {
        final var filter = TopicMessageFilter.builder().limit(query.getLimit());

        if (query.hasTopicID()) {
            filter.topicId(EntityId.of(query.getTopicID()));
        }

        if (query.hasConsensusStartTime()) {
            long startTime = convertTimestamp(query.getConsensusStartTime());
            filter.startTime(startTime);
        }

        if (query.hasConsensusEndTime()) {
            long endTime = convertTimestamp(query.getConsensusEndTime());
            filter.endTime(endTime);
        }

        return filter.build();
    }
