### Title
Global Rate-Limit Bucket DoS via Minimal-Gas Flooding of `/api/v1/contracts/call`

### Summary
The `rateLimitBucket` in `ThrottleManagerImpl` is a single global token bucket shared across all callers with no per-IP or per-user partitioning. An unauthenticated attacker can send requests at the full 500 RPS default limit with the minimum allowed gas (`21000`), which `scaleGas` maps to only 2 scaled tokens against a `gasLimitBucket` of 750,000-token capacity. This exhausts the `rateLimitBucket` entirely while barely touching the `gasLimitBucket`, denying service to all legitimate users.

### Finding Description

**Code path:**

`ContractController.call()` (line 40) calls `throttleManager.throttle(request)` unconditionally before any processing: [1](#0-0) 

`ThrottleManagerImpl.throttle()` checks two global buckets in sequence: [2](#0-1) 

The `rateLimitBucket` is configured with capacity = `requestsPerSecond` = **500** (default), refilling 500 tokens/second: [3](#0-2) 

The `gasLimitBucket` capacity = `scaleGas(7_500_000_000)` = **750,000** tokens: [4](#0-3) 

`scaleGas` returns `0` for gas ≤ 10,000 and `Math.floorDiv(gas, 10_000)` otherwise: [5](#0-4) 

`ContractCallRequest.gas` has `@Min(21_000)` — the minimum a caller can submit: [6](#0-5) 

**Root cause:** The `rateLimitBucket` is a single global shared resource. There is no per-IP, per-session, or per-user partitioning anywhere in the throttle stack. The `requestFilter` mechanism can only match on request-body fields (`FROM`, `TO`, `DATA`, `GAS`, `BLOCK`, `VALUE`) — not on source IP — and is disabled by default: [7](#0-6) 

**Exploit arithmetic:**
- Attacker sends `gas=21000` → `scaleGas(21000) = floor(21000/10000) = 2` gas tokens consumed per request
- At 500 RPS: `500 × 1 = 500` rateLimitBucket tokens consumed/second → bucket fully exhausted
- At 500 RPS: `500 × 2 = 1,000` gasLimitBucket tokens consumed/second out of 750,000 capacity → **0.13% utilization** — gas throttle is effectively bypassed

### Impact Explanation

All legitimate users of `/api/v1/contracts/call` receive HTTP 429 (`ThrottleException: Requests per second rate limit exceeded`) for the entire duration of the attack. The attacker sustains this with trivially cheap requests (gas=21000, minimal calldata). The gas throttle — the intended secondary defense against expensive abuse — provides no meaningful protection at minimum gas values. The endpoint is completely unavailable to all other users as long as the attacker maintains 500 RPS.

### Likelihood Explanation

No authentication, no API key, no IP-based rate limiting is required. Any unprivileged external actor can open 500 concurrent HTTP connections (or use a single async client) and sustain 500 RPS indefinitely. The attack is repeatable every second as the bucket refills. The cost to the attacker is negligible: no on-chain gas is spent, and the requests require minimal computation on the server side (they are rejected at the throttle layer before EVM execution). This is a low-skill, low-cost, high-impact attack.

### Recommendation

1. **Per-IP rate limiting**: Introduce a per-source-IP token bucket (e.g., using bucket4j's distributed or local per-key buckets) so a single IP cannot consume the entire global budget.
2. **Minimum gas floor for gas-bucket accounting**: Enforce a minimum scaled-gas consumption per request (e.g., at least 1 token regardless of `scaleGas` result) so the gas bucket also acts as a meaningful per-request cost.
3. **Upstream rate limiting**: Deploy an ingress-level rate limiter (e.g., nginx `limit_req`, Envoy, or a WAF) with per-IP limits as a defense-in-depth layer independent of application logic.

```bash
# Requires: curl, GNU parallel or similar

for i in $(seq 1 500); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<mirror-node-host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "0x0000000000000000000000000000000000000167",
      "gas": 21000,
      "data": "0x",
      "block": "latest",
      "estimate": false
    }' &
done
wait

# Expected result:
# First ~500 requests: 200 OK (or EVM error, but throttle passes)
# All subsequent requests within the same second from ANY source: 429 Too Many Requests
# Legitimate users are denied service for the duration of the flood
```
