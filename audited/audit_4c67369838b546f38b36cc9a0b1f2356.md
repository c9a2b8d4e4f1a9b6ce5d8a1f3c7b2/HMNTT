### Title
Cache-Miss DoS via Unbounded Block Number in `BlockType.of()` / `findByIndex()`

### Summary
`BlockType.of()` accepts any decimal or hex string that fits in a Java `long` without validating the value against the actual chain height. Because `findByIndex()` is annotated with `unless = "#result == null"`, Spring Cache does not cache empty `Optional` results (Spring unwraps `Optional.empty()` as null for caching). An unauthenticated attacker can flood the web3 API with requests referencing non-existent block indices, each of which bypasses the cache and executes a live database query, enabling a sustained database-amplification DoS.

### Finding Description

**Code path:**

`BlockType.java` — `extractNumericBlock()` (lines 37–56): [1](#0-0) 

The only guards are:
- A check for a literal `-` character (rejects negative numbers).
- `Long.parseLong()` overflow (throws for values > `Long.MAX_VALUE`).

There is **no upper-bound check** against the current chain height. Any 1–19 digit decimal string in the range `[0, 9223372036854775807]` is accepted and wrapped in a fresh `BlockType` instance.

`RecordFileServiceImpl.java` — `findByBlockType()` (lines 19–27): [2](#0-1) 

The `EARLIEST`/`LATEST` guards use **reference equality** (`==`). A numeric `BlockType` whose `.number()` happens to equal `Long.MAX_VALUE` is still a distinct object and falls through to `findByIndex()`.

`RecordFileRepository.java` — `findByIndex()` (lines 27–29): [3](#0-2) 

The annotation `unless = "#result == null"` means: *do not cache when the result is null*. Spring Cache treats `Optional.empty()` as null for this evaluation. Therefore, a query for a non-existent block index returns `Optional.empty()`, which is **never stored in the cache**. Every subsequent request for the same (or any other) non-existent index re-executes the SQL query against the database.

**Exploit flow:**

1. Attacker sends a JSON-RPC `eth_getBlockByNumber` (or `contracts/call` with a `block` field) with a large decimal block number, e.g. `"9223372036854775000"`.
2. `BlockType.of("9223372036854775000")` succeeds; `Long.parseLong` returns `9223372036854775000L`.
3. `findByBlockType()` calls `recordFileRepository.findByIndex(9223372036854775000L)`.
4. Cache lookup misses (key never seen before).
5. JPA executes `SELECT * FROM record_file WHERE index = 9223372036854775000` — no row found.
6. `Optional.empty()` is returned; Spring does **not** cache it (`unless = "#result == null"` fires).
7. Attacker repeats with the same or different large values; each iteration hits the database.

### Impact Explanation

Every crafted request causes a full database round-trip with no caching benefit. Under sustained fire (e.g., hundreds of requests per second from a single client or a small botnet), the database connection pool and query executor become saturated. Legitimate `eth_call`, `eth_getBalance`, and contract-simulation requests that depend on the same database are starved, degrading or halting the web3 API service. While the Hedera consensus network itself is unaffected, the mirror node's web3 endpoint — which is the primary interface for EVM-compatible tooling — becomes unavailable, blocking dApp users and developers from submitting or confirming transactions through this path.

### Likelihood Explanation

The attack requires zero authentication, zero on-chain funds, and no special knowledge beyond the standard Ethereum JSON-RPC interface. The payload is a single JSON field (`"block": "9223372036854775000"`). It is trivially scriptable with `curl` or any HTTP library. The attacker can rotate block numbers across the full `long` range to prevent any future caching benefit even if the `unless` condition were changed. No existing rate-limiting or input-size guard is present in the in-scope code path.

### Recommendation

1. **Validate block number against chain height**: After parsing, compare the block number against the latest known index (e.g., from `findLatest()`). Reject or clamp values that exceed it with an `IllegalArgumentException` / HTTP 400 before reaching `findByIndex()`.
2. **Cache negative results**: Change the `unless` condition to `unless = "#result != null && !#result.isPresent()"` (or use a sentinel value) so that lookups for non-existent indices are cached for a short TTL, preventing repeated DB hits for the same key.
3. **Apply rate limiting** at the API gateway or servlet filter level on the web3 endpoint, keyed by source IP.

### Proof of Concept

```bash
# Flood with distinct large block numbers, each causing a fresh DB query
for i in $(seq 9223372036854775000 9223372036854775100); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"block\":\"$i\",\"data\":\"0x\",\"to\":\"0x0000000000000000000000000000000000000001\"}" &
done
wait
```

Each request reaches `BlockType.of(String.valueOf(i))` → `extractNumericBlock()` → `Long.parseLong()` succeeds → `findByIndex(i)` → cache miss → SQL query → `Optional.empty()` → not cached → next iteration repeats the DB hit. Database connection exhaustion is observable within seconds at moderate request rates.

    private static BlockType extractNumericBlock(String value) {
        int radix = 10;
        var cleanedValue = value;

        if (value.startsWith(HEX_PREFIX)) {
            radix = 16;
            cleanedValue = Strings.CS.removeStart(value, HEX_PREFIX);
        }

        if (cleanedValue.contains(NEGATIVE_NUMBER_PREFIX)) {
            throw new IllegalArgumentException("Invalid block value: " + value);
        }

        try {
            long blockNumber = Long.parseLong(cleanedValue, radix);
            return new BlockType(value, blockNumber);
        } catch (Exception e) {
            throw new IllegalArgumentException("Invalid block value: " + value, e);
        }
    }
