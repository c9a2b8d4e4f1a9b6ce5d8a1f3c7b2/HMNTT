### Title
Uncached Historical Entity UNION Query Enables Unauthenticated Griefing via Global Rate-Limit Saturation

### Summary
`findActiveByIdAndTimestamp()` in `EntityRepository.java` carries no `@Cacheable` annotation, so every unique `(id, blockTimestamp)` pair unconditionally executes a two-branch UNION query across the `entity` and `entity_history` tables. The only protection is a global (not per-IP) token-bucket throttle of 500 requests/second, which a single unauthenticated attacker can fully consume, simultaneously saturating the shared rate-limit budget and driving sustained uncached database load that degrades performance for all users.

### Finding Description
**Exact code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` (global bucket, 500 req/s) → `ContractCallService.callContract()` → `CommonEntityAccessor.get(EntityId, Optional<Long>)` (line 62) → `entityRepository.findActiveByIdAndTimestamp(entityId.getId(), t)` (line 62).

**Root cause – missing cache annotation:**

`findByIdAndDeletedIsFalse` (lines 20-30) is decorated with `@Caching/@Cacheable` backed by `CACHE_MANAGER_ENTITY` (`expireAfterWrite=1s, maximumSize=10000`). `findActiveByIdAndTimestamp` (lines 136-155) has no such annotation. Every invocation with a distinct `blockTimestamp` hits the database unconditionally with:

```sql
(select * from entity        where id=?1 and lower(timestamp_range)<=?2 and deleted is not true)
union all
(select * from entity_history where id=?1 and lower(timestamp_range)<=?2 and deleted is not true
 order by lower(timestamp_range) desc limit 1)
order by timestamp_range desc limit 1
```

**Why the throttle is insufficient:**

`ThrottleManagerImpl` uses a single shared `rateLimitBucket` (default 500 req/s) and `gasLimitBucket` (default 7.5 × 10⁹ gas/s scaled). Both buckets are process-global singletons with no per-IP or per-user partitioning. A single attacker can legally consume the entire 500 req/s budget. Within each EVM execution, `findActiveByIdAndTimestamp` may be called multiple times (once per entity lookup during contract simulation), multiplying the DB query count per HTTP request. ### Impact Explanation
An attacker saturating the 500 req/s global budget with unique `(id, blockTimestamp)` pairs causes:
1. **DB load amplification**: Each HTTP request triggers ≥1 uncached UNION query; a contract that touches N entities triggers N such queries per request, yielding up to 500 × N uncached queries/second against `entity_history`.
2. **Denial of service for legitimate users**: Because the rate-limit bucket is global, the attacker's 500 req/s leaves zero capacity for other callers, returning HTTP 429 to all of them.
3. **No economic cost to attacker**: Historical `eth_call` is a read-only, free operation requiring no on-chain funds or credentials. [5](#0-4) [6](#0-5) 

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of any valid entity ID (trivially enumerable from public Hedera mirror node APIs). No wallet, token, or privileged access is needed. The `block` field in `ContractCallRequest` accepts arbitrary historical block numbers, making it trivial to generate an unbounded stream of unique `blockTimestamp` values. The attack is fully repeatable and automatable. [7](#0-6) 

### Recommendation
1. **Add per-IP rate limiting** upstream (reverse proxy or Spring `HandlerInterceptor`) so a single source cannot consume the entire global budget.
2. **Cache historical entity lookups** with a composite key `(id, blockTimestamp)` and a short TTL (e.g., 10 s), or use a bounded LRU cache, so repeated identical historical queries are served from memory. Apply `@Cacheable` to `findActiveByIdAndTimestamp` analogously to `findByIdAndDeletedIsFalse`.
3. **Bound the historical query range**: reject `blockTimestamp` values older than a configurable horizon to limit `entity_history` scan depth. [1](#0-0) 

### Proof of Concept
```bash
# Enumerate a valid entity ID (e.g., 1001) from the public mirror REST API.
# Then flood historical contract calls with incrementing block numbers:

for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{
      \"to\": \"0x0000000000000000000000000000000000000001\",
      \"data\": \"0x\",
      \"gas\": 21000,
      \"block\": \"$((1000000 + i))\"
    }" &
done
wait
```

Each iteration supplies a distinct `block` value → distinct `blockTimestamp` → cache miss → uncached UNION query on `entity`/`entity_history`. Running this in a tight loop at 500 req/s saturates the global `rateLimitBucket`, returning HTTP 429 to all concurrent legitimate callers while continuously loading the database.
