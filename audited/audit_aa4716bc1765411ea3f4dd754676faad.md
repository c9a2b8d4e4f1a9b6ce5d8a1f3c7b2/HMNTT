### Title
Unauthenticated Impossible-Range Query Causes Unnecessary Database Execution in `GET /blocks`

### Summary
Any unauthenticated user can supply contradictory `block.number` range filters (e.g., `block.number=gt:0&block.number=lt:0`) to `GET /blocks`. The request passes all existing validation, `extractSqlFromBlockFilters()` blindly maps both filters into `whereQuery`, and `RecordFileService.getBlocks()` executes a logically impossible `WHERE index > $1 AND index < $2` query against the database on every request. The codebase already contains an `isEmptyRange()` utility that detects exactly this condition but is never invoked in this code path.

### Finding Description
**Code path:**

1. `getBlocks()` — `rest/controllers/blockController.js` lines 101–104 — calls `utils.buildAndValidateFilters(req.query, acceptedBlockParameters)` then immediately passes the result to `extractSqlFromBlockFilters()`. [1](#0-0) 

2. `buildAndValidateFilters()` — `rest/utils.js` lines 1208–1226 — validates each filter individually (format, operator, value type) and checks `filterDependencyCheck` (only guards against `block.hash + block.number` co-use and `transaction.index` without a block anchor). It performs **no cross-filter range consistency check**. [2](#0-1) 

3. `extractSqlFromBlockFilters()` — `rest/controllers/blockController.js` lines 63–88 — iterates every filter that matches `blockWhereFilters` and maps each one to a `{query, param}` pair with no range-sanity check. For `block.number=gt:0&block.number=lt:0` this produces `whereQuery: [{query:'index >', param:0}, {query:'index <', param:0}]`. [3](#0-2) 

4. `buildWhereSqlStatement()` — `rest/service/recordFileService.js` lines 8–17 — mechanically assembles `WHERE index > $1 AND index < $2` with `params = [0, 0]`. [4](#0-3) 

5. `RecordFileService.getBlocks()` — `rest/service/recordFileService.js` lines 149–162 — executes the assembled query unconditionally via `super.getRows(query, params)`. [5](#0-4) 

**Root cause:** `isEmptyRange()` — `rest/utils.js` lines 901–947 — already exists in the codebase and correctly detects that `gt:0` + `lt:0` on the same key produces an empty range (`upper < lower`). However, it is only called inside `getNextParamQueries()` for pagination link generation and is **never called** during request validation for the `/blocks` endpoint. [6](#0-5) [7](#0-6) 

### Impact Explanation
Every impossible-range request causes a full round-trip to the PostgreSQL backend. While the planner may short-circuit the scan, the connection is consumed, the query is parsed and planned, and the result is serialized. An attacker sending a high volume of such requests amplifies database connection and CPU load with zero authentication cost. Because the response is always a valid `200 OK` with an empty `blocks` array, the attacker receives no error signal to stop.

### Likelihood Explanation
The exploit requires no credentials, no special knowledge, and no tooling beyond a standard HTTP client. The parameter format (`gt:0`, `lt:0`) is documented in the public API. The attack is trivially scriptable and repeatable at any rate the network allows, making it a realistic low-effort denial-of-service amplification vector.

### Recommendation
Before executing the query, call `isEmptyRange(filterKeys.BLOCK_NUMBER, req.query[filterKeys.BLOCK_NUMBER])` (and the equivalent for `timestamp`) inside `getBlocks()` or at the end of `buildAndValidateFilters()`. If the function returns `true`, return an empty result immediately (or throw an `InvalidArgumentError`) without touching the database. The utility is already present and correct — it only needs to be wired into the blocks request path.

### Proof of Concept
```
# Single impossible-range request — always returns HTTP 200 with empty blocks array
# but executes: SELECT ... FROM record_file WHERE index > $1 AND index < $2  [0, 0]
curl "http://<mirror-node-rest>/api/v1/blocks?block.number=gt:0&block.number=lt:0"

# Flood variant (no auth required)
while true; do
  curl -s "http://<mirror-node-rest>/api/v1/blocks?block.number=gt:0&block.number=lt:0" > /dev/null &
done
```

Preconditions: none — the endpoint is public.  
Trigger: any `gt:N` + `lt:N` (or `gt:N` + `lt:M` where `M ≤ N`) combination on `block.number`.  
Result: database query executed, zero rows returned, HTTP 200 sent; repeatable indefinitely.

const isEmptyRange = (key, value) => {
  const values = Array.isArray(value) ? value : [value];
  let lower = null;
  let upper = null;

  for (const v of values) {
    if (!gtLtPattern.test(v)) {
      continue;
    }

    const filter = buildComparatorFilter(key, v);
    formatComparator(filter);

    // formatComparator doesn't handle CONTRACT_ID and SLOT
    if (key === constants.filterKeys.CONTRACT_ID) {
      filter.value = EntityId.parse(filter.value).getEncodedId();
    } else if (key === constants.filterKeys.SLOT) {
      filter.value = addHexPrefix(filter.value);
    }

    if (filter.value == null) {
      continue;
    }

    let parsed = BigInt(filter.value);
    switch (filter.operator) {
      case opsMap.gt:
        parsed += 1n;
      case opsMap.gte:
        lower = lower === null ? parsed : bigIntMax(lower, parsed);
        break;
      case opsMap.lt:
        parsed -= 1n;
      case opsMap.lte:
        upper = upper === null ? parsed : bigIntMin(upper, parsed);
        break;
      default:
        break;
    }
  }

  if (lower === null || upper === null) {
    return false;
  }

  return upper < lower;
};

const buildWhereSqlStatement = (whereQuery) => {
  let where = '';
  const params = [];
  for (let i = 1; i <= whereQuery.length; i++) {
    where += `${i === 1 ? 'where' : 'and'} ${whereQuery[i - 1].query} $${i} `;
    params.push(whereQuery[i - 1].param);
  }

  return {where, params};
};
