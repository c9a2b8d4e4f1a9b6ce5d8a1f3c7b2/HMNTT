### Title
Unbounded `limit=Long.MAX_VALUE` Causes Integer Overflow, Indefinite Subscription, and Repeated Full Table Scans via `subscribeTopic()`

### Summary
An unprivileged external caller can send a `ConsensusTopicQuery` with `limit` set to `Long.MAX_VALUE` (the maximum `uint64` value, valid as a positive Java `long`). This bypasses the only validation guard (`@Min(0)`), triggers an integer overflow in the polling layer that silently removes the SQL `LIMIT` clause, and holds the gRPC stream open indefinitely. Repeated full table scans on every poll interval exhaust database and server resources, starving legitimate subscribers from receiving messages.

### Finding Description

**Code path:**

1. `ConsensusController.toFilter()` (`ConsensusController.java:56`) copies `query.getLimit()` verbatim into `TopicMessageFilter.limit`. No upper-bound check exists.

2. `TopicMessageFilter` (`TopicMessageFilter.java:25-26`) only enforces `@Min(0)`. `Long.MAX_VALUE` satisfies this. `hasLimit()` (`line 39`) returns `limit > 0` → `true`.

3. `TopicMessageServiceImpl.subscribeTopic()` (`TopicMessageServiceImpl.java:83-84`) calls `flux.take(Long.MAX_VALUE)`. In Project Reactor, `Flux.take(Long.MAX_VALUE)` never completes by count — the subscription is held open indefinitely.

4. **Integer overflow — root cause of the table scan:** In `PollingTopicMessageRetriever.poll()` (`PollingTopicMessageRetriever.java:68-69`):
   ```java
   int limit = filter.hasLimit()
       ? (int) (filter.getLimit() - context.getTotal().get())
       : Integer.MAX_VALUE;
   ```
   `(int)(Long.MAX_VALUE - 0L)` truncates to `-1`. `Math.min(-1, maxPageSize)` = `-1`. The rebuilt filter is stored with `limit = -1L`.

5. The same overflow occurs in `PollingTopicListener.poll()` (`PollingTopicListener.java:54-55`) for the live-message path, which polls every 500 ms by default.

6. `TopicMessageRepositoryCustomImpl.findByFilter()` (`TopicMessageRepositoryCustomImpl.java:51-53`):
   ```java
   if (filter.hasLimit()) {          // hasLimit() = (-1 > 0) = false
       typedQuery.setMaxResults(...); // NEVER CALLED
   }
   ```
   Because `hasLimit()` returns `false` for `-1`, `setMaxResults` is skipped entirely. The JPA query runs with no `LIMIT` clause — a full table scan of `topic_message`.

7. `PollingContext.isComplete()` (`PollingTopicMessageRetriever.java:121-128`) only sets `limitHit=true` when `total.get() == Long.MAX_VALUE`, which is unreachable. The retriever loops indefinitely.

8. `maxConcurrentCallsPerConnection=5` (`NettyProperties.java:14`) is per-connection only. No global cap on total active subscriptions exists.

**Why existing checks fail:**
- `@Min(0)` rejects negative values but has no `@Max` — `Long.MAX_VALUE` is accepted.
- The `(int)` cast of a `long` difference is a silent narrowing conversion; no overflow check exists.
- `hasLimit()` uses `> 0` semantics, so the overflowed `-1` is treated as "no limit" at the DB layer while the outer `flux.take(Long.MAX_VALUE)` treats it as "has limit" — a split-brain that bypasses both guards simultaneously.

### Impact Explanation
Each malicious subscription causes:
- A full table scan of `topic_message` on every retriever poll (default: every 2 s) and every listener poll (default: every 500 ms).
- The subscription never self-terminates (no end time, no count limit reached).
- With N connections × 5 concurrent calls, an attacker can saturate the PostgreSQL connection pool and I/O bandwidth, causing legitimate HCS subscribers to time out or receive no messages. For applications using HCS as a fund-release or escrow signaling channel, this constitutes a denial-of-service against message delivery.

### Likelihood Explanation
No authentication is required. The gRPC endpoint is publicly exposed (port 5600). The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera Java SDK) and knowledge of a valid topic ID. The attack is trivially repeatable and scriptable. A single attacker with modest bandwidth can open dozens of connections.

### Recommendation
1. **Add an upper-bound validation** on `TopicMessageFilter.limit`, e.g., `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate cap (e.g., 10,000,000).
2. **Fix the integer overflow** in `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()` by using `long` arithmetic and capping at `Integer.MAX_VALUE` safely:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
3. **Enforce a global maximum subscriber count** in `TopicMessageServiceImpl` by rejecting new subscriptions when `subscriberCount` exceeds a configurable threshold.
4. **Add a per-IP or per-connection subscription rate limit** at the gRPC layer.

**Expected result:**
- The stream never closes.
- Every 500 ms, `PollingTopicListener` issues an unbounded `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with no `LIMIT` clause.
- Every 2 s, `PollingTopicMessageRetriever` issues the same unbounded query.
- Repeating this across multiple connections exhausts DB query capacity and degrades or blocks message delivery for all other subscribers.
