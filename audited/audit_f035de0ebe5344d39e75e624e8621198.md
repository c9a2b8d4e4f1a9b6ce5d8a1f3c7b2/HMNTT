### Title
Unbounded `keys` Collection Passed to Native SQL `IN` Clause in `getHookStorageChange()` Enables DoS via Query Amplification

### Summary
In `HookServiceImpl.getHookStorageChange()`, the full user-supplied `keys` collection is passed without any size limit to `HookStorageChangeRepository.findByKeyInAndTimestampBetween()`, which expands it into a native SQL `IN (...)` clause. An unprivileged attacker can supply an arbitrarily large `keys` collection with at least one entry within the timestamp range, bypassing the only early-exit guard, and force the database to parse and execute a massive query, exhausting DB connections and application threads.

### Finding Description

**Code path:**

In `HookServiceImpl.java` lines 84–118, `getHookStorageChange()` computes `keysInRange` (line 91) and checks only whether it is empty (line 93). If `keysInRange` is non-empty (i.e., at least one supplied key falls within the key bounds), the method proceeds to line 104 and passes the **full, unfiltered `keys` collection** — not `keysInRange` — to the repository:

```java
// HookServiceImpl.java line 103-105
if (requestHasKeys) {
    changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
            ownerId.getId(), hookId, keys, ...);   // ← full `keys`, not keysInRange
``` [1](#0-0) 

The repository uses a native query with `key in (:keys)`:

```sql
and key in (:keys)
``` [2](#0-1) 

Spring Data JPA expands `:keys` into individual positional parameters at query-build time, producing `key IN (?, ?, ?, ... N times)`. There is no size cap on the `keys` field in `HookStorageRequest`: [3](#0-2) 

The `limit` field (line 33) controls only the result page size, not the input `keys` size. The `getKeysInRange()` filter (lines 43–52) is computed but the result is only used for the empty-check guard; the raw `keys` is what reaches the DB. [4](#0-3) 

**Root cause:** The guard at line 93 only prevents the query when *zero* keys are in range. As soon as one key is in range, the entire unbounded `keys` collection is forwarded to the SQL engine. [5](#0-4) 

### Impact Explanation
PostgreSQL has no hard limit on `IN` clause size, but parsing and planning a query with tens of thousands of bind parameters is CPU- and memory-intensive on the DB server. Each such request ties up a DB connection and a mirror-node worker thread for the duration of the slow parse/plan/execute cycle. Concurrent flood of such requests exhausts the DB connection pool and the application thread pool, causing cascading failures across mirror-node processing threads and rendering the service unavailable — consistent with ≥30% node processing degradation.

### Likelihood Explanation
The endpoint is unauthenticated (no privileged access required). An attacker only needs to know the API surface for hook storage queries and supply a timestamp range with one valid key in bounds plus an arbitrarily large `keys` array. This is trivially repeatable with a simple HTTP client loop and requires no special knowledge beyond the public API schema.

### Recommendation
1. **Enforce a maximum `keys` collection size** in `HookStorageRequest` (e.g., cap at the `limit` value or a fixed maximum such as 1000) and reject requests exceeding it with HTTP 400.
2. **Pass `keysInRange` instead of `keys`** to `findByKeyInAndTimestampBetween()` at line 104, so only the already-filtered subset reaches the SQL engine.
3. **Add a DB-level statement timeout** for this query class to bound worst-case execution time.

### Proof of Concept
1. Identify a valid `ownerId`, `hookId`, and a timestamp range `[T1, T2]` that contains at least one known key `K0` within the key bounds.
2. Craft an HTTP request to the hook storage endpoint with `timestamp=gte:T1&timestamp=lte:T2`, `key[0]=K0` (in range), and `key[1]` through `key[N]` set to arbitrary byte values (N = 50,000+).
3. Send the request. The service calls `getHookStorageChange()`, `keysInRange` = `[K0]` (non-empty), so the guard at line 93 is bypassed, and all N+1 keys are forwarded to `findByKeyInAndTimestampBetween()`.
4. Spring Data JPA generates `key IN (?, ?, ... 50001 times)` and submits it to PostgreSQL.
5. Repeat concurrently from multiple clients; DB connection pool and application thread pool are exhausted, causing service-wide query failures.
