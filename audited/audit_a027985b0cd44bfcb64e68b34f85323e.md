### Title
Unauthenticated Forced Routing to Expensive Historical Query Path via Trivial Timestamp Parameter in `getHookStorage()`

### Summary
Any unauthenticated external user can supply a single timestamp query parameter (e.g., `timestamp=gte:0`) to force `getHookStorage()` to always route to `getHookStorageChange()`, which executes a `DISTINCT ON (key)` full-range scan across the unbounded, time-partitioned `hook_storage_change` table. Because no rate limiting exists on the rest-java hooks endpoint and the result-set `limit` cap does not bound the underlying DB scan, an attacker can repeatedly trigger this expensive code path to materially increase database and node CPU/IO resource consumption.

### Finding Description

**Routing decision — `HookServiceImpl.java` lines 56–58:** [1](#0-0) 

`request.isHistorical()` delegates to `HookStorageRequest.isHistorical()`: [2](#0-1) 

Which delegates to `Bound.isEmpty()`: [3](#0-2) 

`Bound.isEmpty()` returns `true` only when both `lower` and `upper` are `null`. Providing **any** timestamp operator (e.g., `timestamp=gte:0`) sets `lower` to a non-null value, making `isHistorical()` return `true` and unconditionally routing to `getHookStorageChange()`.

**Default bounds when no key filter is supplied — `HooksController.java` lines 167–168:** [4](#0-3) 

`keyLowerBound = MIN_KEY_BYTES` (32 × `0x00`) and `keyUpperBound = MAX_KEY_BYTES` (32 × `0xFF`). With `timestamp=gte:0`, `timestampLowerBound = 0` and `timestampUpperBound = Long.MAX_VALUE`.

**The expensive query — `HookStorageChangeRepository.java` lines 15–39:** [5](#0-4) 

`DISTINCT ON (key)` with `key BETWEEN 0x00…00 AND 0xFF…FF` and `consensus_timestamp BETWEEN 0 AND Long.MAX_VALUE` forces the database to scan **every row** in `hook_storage_change` for the given `owner_id`/`hook_id` across all time partitions, sort them, and deduplicate — regardless of the `limit` cap on the result set.

**Contrast with the cheap path** — `HookStorageRepository` uses a simple primary-key range scan on `hook_storage` (current state only): [6](#0-5) 

**No rate limiting on the rest-java hooks endpoint.** The throttling infrastructure found (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` module for contract-call requests: [7](#0-6) 

The rest-java `HooksController` has no equivalent guard.

**Failed assumption:** The design assumes historical queries are rare and user-initiated for legitimate audit purposes. There is no enforcement of this assumption — any caller can always supply a timestamp.

### Impact Explanation

The `hook_storage_change` table is partitioned by `consensus_timestamp` and grows without bound as hooks are updated over time. A wide-range `DISTINCT ON` query must cross partition boundaries, sort all matching rows, and deduplicate before applying the `LIMIT`. This is orders of magnitude more expensive than the indexed primary-key lookup on `hook_storage`. An attacker sending a sustained stream of requests with `timestamp=gte:0` and no key filter forces repeated full-history scans, consuming database CPU, I/O, and connection pool resources. Because the endpoint is unauthenticated and publicly reachable, this is a low-barrier resource exhaustion vector that can degrade service for all users of the mirror node.

### Likelihood Explanation

No special privileges, credentials, or knowledge of internal state are required. The exploit requires only knowledge of any valid `ownerId` and `hookId` (both discoverable via the public `/api/v1/accounts/{id}/hooks` endpoint). The attack is trivially scriptable, repeatable, and requires no brute force. The attacker controls the exact parameters that maximize scan cost (full key range, full timestamp range) and can sustain the attack indefinitely.

### Recommendation

1. **Add a timestamp range width cap**: Reject or clamp requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 7 days), consistent with how other historical endpoints are guarded.
2. **Add rate limiting to the rest-java hooks storage endpoint**: Apply a per-IP or global request-rate limit analogous to the web3 throttle, specifically for the `/{hookId}/storage` endpoint when `timestamp` is present.
3. **Require a minimum timestamp precision for historical queries**: Require `timestamp=eq:X` or a narrow range (e.g., `gte:X&lte:Y` where `Y-X` is bounded) rather than allowing open-ended ranges like `gte:0`.
4. **Add a DB query timeout** specific to historical hook storage queries to prevent runaway scans from holding connections.

### Proof of Concept

```
# Step 1: Discover a valid ownerId and hookId (public endpoint, no auth)
GET /api/v1/accounts/0.0.123/hooks

# Step 2: Force the expensive historical path with maximum scan range
# timestamp=gte:0  → isHistorical()=true, timestampLowerBound=0, timestampUpperBound=Long.MAX_VALUE
# No key filter    → keyLowerBound=0x00…00, keyUpperBound=0xFF…FF (full range)
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0&limit=100

# Step 3: Repeat in a tight loop from a single client (no auth required)
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0&limit=100" > /dev/null
done
```

Each iteration executes `SELECT DISTINCT ON (key) … FROM hook_storage_change WHERE owner_id=? AND hook_id=? AND key BETWEEN '\x00…' AND '\xff…' AND consensus_timestamp BETWEEN 0 AND 9223372036854775807` — a full historical scan — with no server-side rate limit to stop it.
