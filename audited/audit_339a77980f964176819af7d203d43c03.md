### Title
Unauthenticated EVM Address DB Lookup in `getNftsByAccountId` Enables Connection Pool Exhaustion

### Summary
Any unauthenticated caller can supply a `shard.realm.<40-hex-chars>` path parameter to `GET /api/v1/accounts/:idOrAliasOrEvmAddress/nfts`. This format passes format validation and unconditionally triggers a live database query in `EntityService.getEntityIdFromEvmAddress()` with no application-level caching and no rate limiting on the REST API. At high concurrency with many distinct EVM addresses, the finite `pg` connection pool is exhausted, blocking all other database-dependent endpoints and degrading 30%+ of mirror-node processing capacity.

### Finding Description

**Exact code path:**

`accountController.js` line 91 passes the raw path parameter directly to `EntityService.getEncodedId()` with no prior rate-limiting gate: [1](#0-0) 

Inside `getEncodedId()`, when the input matches `EntityId.isValidEntityId()` and the parsed `entityId.evmAddress` is non-null (i.e., the 40-hex-char suffix is present), execution falls into the DB-hitting branch: [2](#0-1) 

`getEntityIdFromEvmAddress()` executes a raw SQL query against the `entity` table on every call: [3](#0-2) 

The query used is: [4](#0-3) 

**No caching exists** in the REST (Node.js) service for EVM address lookups. Contrast this with the `web3` Java module which uses `@Cacheable` on its `findByEvmAddress` repository method — the REST service has no equivalent.

**Root cause:** The `getEntityIdFromEvmAddress()` path has no memoization, no in-process cache, and no request-level deduplication. Every distinct EVM address supplied by an attacker results in a fresh connection checkout from the pool and a synchronous DB round-trip.

**Connection pool:** The pool is bounded by `config.db.pool.maxConnections`: [5](#0-4) 

**No rate limiting on the REST API:** The `server.js` middleware stack registers only `requestLogger`, `authHandler` (response-limit only, not request throttle), optional metrics, and optional response cache — no IP-based or endpoint-level rate limiter: [6](#0-5) 

The throttle mechanisms (`ThrottleManagerImpl`, `ThrottleConfiguration`) exist only in the `web3` Java module and do not apply to the Node.js REST service.

### Impact Explanation

When the connection pool is saturated, every other endpoint that requires a DB connection (transactions, balances, tokens, etc.) queues behind the attacker's requests. Once `connectionTimeoutMillis` is exceeded, those requests fail with connection-timeout errors. Because the REST mirror node is the primary read path for network state queries, exhausting its DB pool degrades the ability of downstream consumers (wallets, explorers, dApps) to read ledger state, meeting the ≥30% network processing degradation threshold. The attack is amplified because each request is cheap for the attacker (a single HTTP GET) but expensive for the server (a DB connection checkout + index scan on `entity.evm_address`).

### Likelihood Explanation

- **No authentication required** — any anonymous internet user can trigger this.
- **No rate limiting** — the attacker can sustain thousands of requests per second from a single host or a small botnet.
- **Trivially automatable** — generating 40-character random hex strings is trivial; each unique address bypasses any hypothetical response cache.
- **Repeatable** — the attack can be sustained indefinitely; there is no circuit-breaker or back-pressure mechanism in the REST service.

### Recommendation

1. **Add application-level rate limiting** to the REST Node.js service (e.g., `express-rate-limit` or `bottleneck`) scoped per IP, applied before any DB-touching middleware.
2. **Cache EVM address lookups** in-process (e.g., an LRU cache keyed on the hex address) with a short TTL (e.g., 5–30 s) to collapse repeated or near-repeated lookups into a single DB query.
3. **Limit concurrency of EVM address DB queries** using a semaphore or connection-pool sub-limit dedicated to this lookup type, preventing it from monopolizing the shared pool.
4. **Add a `statement_timeout` guard** at the query level (already partially present via `config.db.pool.statementTimeout`) and ensure the pool's `connectionTimeout` is short enough to shed load quickly rather than queue indefinitely.

### Proof of Concept

```bash
# Generate N distinct valid EVM-address-format path parameters and fire them concurrently
# Each triggers a live DB query in getEntityIdFromEvmAddress()

for i in $(seq 1 500); do
  HEX=$(openssl rand -hex 20)  # 40 hex chars = valid EVM address
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.${HEX}/nfts" &
done
wait

# Expected result:
# - DB connection pool exhausted (pool.maxConnections slots consumed)
# - Concurrent legitimate requests to /api/v1/transactions, /api/v1/balances, etc.
#   begin returning 500 errors or timing out after connectionTimeoutMillis
# - Mirror node REST API effectively degraded for all users
```
