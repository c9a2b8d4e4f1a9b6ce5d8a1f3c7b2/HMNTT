### Title
Information Disclosure via Unsanitized Exception Messages in Spring Boot Actuator Health Endpoint

### Summary
`SubscriberHealthIndicator.health()` places raw exception messages (`e.getMessage()`) and internal REST API status codes directly into the Spring Boot health response as `reason` detail fields. If the actuator is configured with `management.endpoint.health.show-details=always`, any unauthenticated external user can retrieve these internal error descriptions by querying the `/actuator/health` endpoint, enabling reconnaissance for targeted attacks against the REST API backend.

### Finding Description
In `monitor/src/main/java/org/hiero/mirror/monitor/health/SubscriberHealthIndicator.java`, the `restNetworkStakeHealth()` method constructs status messages from raw internal data and passes them to `health()`, which stores them as the `"reason"` detail key in the Spring Boot `Health` object:

- **Lines 102–105**: When the REST API returns a non-2xx response, the raw HTTP status code integer is embedded in the message and stored as a health detail:
  ```java
  var statusMessage = String.format("Network stake status is %s with status code %s", status, statusCode.value());
  return health(status, statusMessage);
  ```

- **Lines 117–120**: On connection/timeout errors, the raw exception message (`e.getMessage()`) is embedded directly:
  ```java
  var statusMessage = String.format("Network stake status is %s with error: %s", status, e.getMessage());
  return health(status, statusMessage);
  ```

- **Lines 50–55**: The `health()` helper unconditionally stores the reason string as a health detail:
  ```java
  health.withDetail("reason", reason);
  ```

`e.getMessage()` from a `ConnectException` or `TimeoutException` typically contains internal hostnames, IP addresses, port numbers, and service URLs (e.g., `"Connection refused: rest-api.internal:443"`). These are surfaced verbatim in the health response JSON under `components.subscriberHealthIndicator.details.reason`.

Spring Boot's default for `show-details` is `never` for unauthenticated users, but when `management.endpoint.health.show-details=always` is set (a common operational choice for monitoring dashboards), no authentication is required to read full health details. There is no sanitization, filtering, or role check applied to the `reason` field before it is written into the health response.

### Impact Explanation
An attacker querying `GET /actuator/health` receives a JSON response containing internal infrastructure details: REST API hostnames, ports, and error strings. This directly enables:
- Mapping of internal service topology (hostnames, ports)
- Identification of which backend REST API endpoint the monitor is probing (`/api/v1/network/stake`)
- Confirmation of whether the REST API is down, returning specific HTTP error codes, or unreachable — enabling timing of attacks to coincide with degraded states

Severity: **Medium** (information disclosure enabling targeted follow-on attacks; no direct code execution or data exfiltration, but meaningful reconnaissance value).

### Likelihood Explanation
`show-details=always` is a common configuration in internal/ops deployments where the health endpoint feeds monitoring dashboards. If the actuator port is not separately firewalled (also common in Kubernetes sidecar/service-mesh setups where the app port and management port are the same), any network-reachable client — including an unprivileged external user — can exploit this. The exploit requires only a single unauthenticated HTTP GET request and is trivially repeatable.

### Recommendation
1. **Sanitize error messages** before embedding them in health details. Replace `e.getMessage()` with a generic, non-revealing string (e.g., `"Connection error"`) and log the full exception internally only.
2. **Replace raw status code embedding** with a fixed, non-revealing message (e.g., `"REST API returned an unexpected response"`).
3. **Restrict health detail visibility** by setting `management.endpoint.health.show-details=when-authorized` (Spring Boot default-safe option) and ensuring only authenticated/privileged roles can access full details.
4. Optionally, expose the actuator on a separate management port (`management.server.port`) that is firewalled from external access.

### Proof of Concept
**Preconditions:**
- Monitor service is running with `management.endpoint.health.show-details=always`
- The actuator health endpoint is reachable (default: same port as app, or `management.server.port`)
- The REST API backend is down or returning errors

**Steps:**
1. Trigger an error state (or wait for one): the REST API at the configured network stake endpoint is unreachable or returns 5xx.
2. Send an unauthenticated request:
   ```
   GET http://<monitor-host>:<port>/actuator/health
   ```
3. Observe the response:
   ```json
   {
     "status": "DOWN",
     "components": {
       "subscriberHealthIndicator": {
         "status": "DOWN",
         "details": {
           "reason": "Network stake status is DOWN with error: Connection refused: rest-api.internal:8080"
         }
       }
     }
   }
   ```
4. The internal hostname (`rest-api.internal`) and port (`8080`) are now disclosed to the unauthenticated attacker. [1](#0-0) [2](#0-1) [3](#0-2)
