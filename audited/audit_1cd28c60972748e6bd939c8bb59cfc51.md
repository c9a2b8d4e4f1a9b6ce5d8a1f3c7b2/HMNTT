### Title
Unauthenticated DoS via Unbounded Concurrent Requests to `/api/v1/network/nodes` with Complex Unprotected Multi-CTE Query

### Summary
The `findNetworkNodes()` method in `NetworkNodeRepository.java` executes a complex native SQL query containing multiple CTEs and per-row correlated subqueries on every request. The `/api/v1/network/nodes` endpoint has no rate limiting, no response caching, and is fully accessible to unauthenticated users, allowing any external attacker to saturate PostgreSQL CPU by flooding the endpoint with concurrent requests.

### Finding Description

**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/repository/NetworkNodeRepository.java`, lines 25–103, `findNetworkNodes()`.

The query executes three CTEs on every invocation:

1. `latest_address_book` (line 26–32): Scans `address_book` filtered by `:fileId`, ordered DESC, LIMIT 1. Varying `:fileId` values across concurrent requests prevent PostgreSQL from reusing cached generic plans efficiently.

2. `latest_node_stake` (lines 33–39): Contains a correlated subquery `(select max(consensus_timestamp) from node_stake)` — a full-table aggregation executed on every request. As `node_stake` accumulates daily historical staking data for all nodes, this becomes increasingly expensive.

3. `node_info` (lines 40–43): Unconditional full scan of the `node` table with no predicate.

4. Main query (lines 72–83): A **per-row correlated subquery** against `address_book_service_endpoint` runs for each returned `address_book_entry` row.

5. The `ORDER BY` uses a CASE expression (lines 99–101) that prevents index-based sort optimization.

**Failed assumption:** The Javadoc comment at line 14–15 states: *"Performance is maintained through conditional SQL that PostgreSQL optimizes efficiently."* This is incorrect — the `max(consensus_timestamp)` subquery and per-row service endpoint subquery are re-evaluated on every request with no memoization.

**No rate limiting exists in the rest-java module.** The `ThrottleConfiguration` and `ThrottleProperties` classes are in the `web3` module and apply only to contract call endpoints. `RestJavaConfiguration` (the rest-java config) registers only an ETag filter and CORS — no rate limiter. `NetworkServiceImpl.getNetworkNodes()` (lines 99–137) has no `@Cacheable` annotation and no throttle guard.

**Exploit flow:**
1. Attacker sends N concurrent `GET /api/v1/network/nodes?file.id=<varying>&node.id=<varying>` requests with no authentication.
2. Each request reaches `NetworkController.getNodes()` (line 152–171), calls `networkService.getNetworkNodes()`, which calls `networkNodeRepository.findNetworkNodes()`.
3. Each DB connection executes the full multi-CTE query including the `max(consensus_timestamp)` aggregation and per-row correlated subqueries.
4. With no rate limiting, N concurrent queries run simultaneously, saturating PostgreSQL CPU and connection pool.

### Impact Explanation

Sustained high CPU on the PostgreSQL backend causes query latency to spike across all mirror node endpoints sharing the same database. The `/api/v1/network/nodes` endpoint becomes unavailable or severely degraded. The `statementTimeout` of 10,000 ms (documented in `docs/configuration.md`, line 630) limits individual query duration but does not prevent concurrent saturation — 100 simultaneous 9-second queries fully occupy the connection pool. This is a Denial of Service against a public infrastructure endpoint.

### Likelihood Explanation

No authentication, no API key, no CAPTCHA, and no rate limit are required. Any internet user with a basic HTTP client (curl, Python requests, k6) can reproduce this. The attack is stateless and trivially parallelizable. The varying `:fileId` and `:nodeIds` parameters prevent HTTP-layer caching (CDN/proxy) from absorbing the load. Realistic likelihood: **High**.

### Recommendation

1. **Add rate limiting** to the rest-java module for the `/api/v1/network/nodes` endpoint (e.g., via a servlet filter using Bucket4j, mirroring the `web3` module's `ThrottleConfiguration`).
2. **Cache the query result** with a short TTL (e.g., 30–60 seconds) using Spring's `@Cacheable` on `NetworkServiceImpl.getNetworkNodes()`, since the address book changes infrequently.
3. **Materialize the `max(consensus_timestamp)` subquery** outside the CTE or use a dedicated indexed view/materialized query to avoid the full-table aggregation per request.
4. **Limit the `nodeIds` array size** at the service layer to prevent unbounded array parameters.

### Proof of Concept

```bash
# Flood the endpoint with 200 concurrent requests using varying fileId values
seq 1 200 | xargs -P 200 -I{} curl -s \
  "https://<mirror-node-host>/api/v1/network/nodes?file.id=0.0.{}&limit=25" \
  -o /dev/null

# Alternatively with varying node.id ranges
for i in $(seq 1 200); do
  curl -s "https://<mirror-node-host>/api/v1/network/nodes?node.id=gte:$i&limit=25" \
    -o /dev/null &
done
wait
```

With no rate limiting, all 200 queries execute concurrently against PostgreSQL, saturating CPU and the connection pool, causing 503 responses for all mirror node API consumers.     @Override
    public List<NetworkNodeDto> getNetworkNodes(NetworkNodeRequest request) {
        final long fileId = getAddressBookFileId(request);
        final var limit = request.getEffectiveLimit();
        final var nodeIdParams = request.getNodeIds();
        final var orderDirection = request.getOrder().name();

        final Set<Long> nodeIds = new HashSet<>();
        long lowerBound = 0L;
        long upperBound = Long.MAX_VALUE;

        for (final var nodeIdParam : nodeIdParams) {
            if (nodeIdParam.operator() == RangeOperator.EQ) {
                nodeIds.add(nodeIdParam.value());
            } else if (nodeIdParam.hasLowerBound()) {
                lowerBound = Math.max(lowerBound, nodeIdParam.getInclusiveValue());
            } else if (nodeIdParam.hasUpperBound()) {
                upperBound = Math.min(upperBound, nodeIdParam.getInclusiveValue());
            }
        }

        if (lowerBound > upperBound) {
            throw new IllegalArgumentException("Invalid range provided for node.id");
        }

        final Long[] nodeIdArray;
        if (!nodeIds.isEmpty()) {
            final var range = Range.closed(lowerBound, upperBound);
            nodeIdArray = nodeIds.stream().filter(range::contains).toArray(Long[]::new);
            if (nodeIdArray.length == 0) {
                return List.of();
            }
        } else {
            nodeIdArray = EMPTY_NODE_ID_ARRAY;
        }

        return networkNodeRepository.findNetworkNodes(
                fileId, nodeIdArray, lowerBound, upperBound, orderDirection, limit);
    }

    @GetMapping("/nodes")
    ResponseEntity<NetworkNodesResponse> getNodes(@RequestParameter NetworkNodeRequest request) {
        final var fileId = request.getFileId();
        if (fileId != null && fileId.operator() != RangeOperator.EQ) {
            throw new IllegalArgumentException("Only equality operator is supported for file.id");
        }
        final var networkNodeRows = networkService.getNetworkNodes(request);
        final var limit = request.getEffectiveLimit();

        final var networkNodes = networkNodeMapper.map(networkNodeRows);

        final var sort = Sort.by(request.getOrder(), Constants.NODE_ID);
        final var pageable = PageRequest.of(0, limit, sort);
        final var links = linkFactory.create(networkNodes, pageable, NETWORK_NODE_EXTRACTOR);

        var response = new NetworkNodesResponse();
        response.setNodes(networkNodes);
        response.setLinks(links);
        return ResponseEntity.ok(response);
    }
