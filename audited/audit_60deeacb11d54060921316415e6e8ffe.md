### Title
Non-JSON HTTP 200 Response Silently Kills Retry Loop, Preventing Pinger Startup

### Summary
In `fetchMirrorNodeNetwork()`, a JSON decode failure on an HTTP 200 response unconditionally returns `retry=false`, which causes `buildNetworkFromMirrorNodes()` to break out of its retry loop immediately. An attacker who can influence the mirror REST response (e.g., via MITM on a plaintext HTTP connection, DNS poisoning, or a misconfigured/compromised mirror node) can return HTTP 200 with a non-JSON body, causing the pinger to fail to build its network map and exit entirely.

### Finding Description

**Exact code path:**

In `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`:

```
Line 90-93: Status code check passes for HTTP 200 — no Content-Type validation.
Line 96-98: json.NewDecoder(resp.Body).Decode(&payload) is called blindly.
            On failure (e.g., body is "<html>..."), returns (nil, false, err).
            The `false` = retry=false.
``` [1](#0-0) 

In `buildNetworkFromMirrorNodes()`:

```
Line 59: if !retry || attempt == attempts { break }
         retry=false → breaks immediately, no further attempts made.
``` [2](#0-1) 

In `sdk_client.go`, `newClient()`:

```
Line 18-21: buildNetworkFromMirrorNodes error → newClient returns error.
``` [3](#0-2) 

In `main.go`:

```
Line 41-44: log.Fatalf("client error: %v", err) → pinger process exits.
``` [4](#0-3) 

**Root cause:** The code conflates a transient/attacker-induced decode error with a permanent, non-retryable failure. There is no `Content-Type: application/json` validation before decoding, and the `retry=false` return on decode failure bypasses all configured retry attempts (`mirrorNodeClientMaxRetries`, default 10).

**Why existing checks are insufficient:**
- The HTTP status code check (lines 90–93) only gates on non-2xx codes. A 200 with `text/html` body passes straight through.
- The retry logic in `buildNetworkFromMirrorNodes` (line 59) correctly implements the retry gate, but `fetchMirrorNodeNetwork` never sets `retry=true` for decode errors, so the gate always closes immediately. [5](#0-4) 

### Impact Explanation

When `network=other` is configured, the pinger depends entirely on `buildNetworkFromMirrorNodes` to construct the Hiero node map at startup. A single poisoned HTTP 200 response with a non-JSON body (e.g., an HTML error page, a WAF interstitial, or an attacker-injected response) causes `newClient` to return an error, which triggers `log.Fatalf` in `main()`, killing the pinger process. The result is **complete cessation of all transfer submissions** — no records are exported to mirror nodes for the duration of the outage. In a monitoring/liveness context, this is a silent availability failure.

### Likelihood Explanation

The default mirror REST URL example in `config.go` is `http://mirror-rest:5551` — plaintext HTTP. [6](#0-5) 

In a Kubernetes deployment (evidenced by the liveness/readiness probe files at `/tmp/alive` and `/tmp/ready`), an attacker with any foothold in the cluster network can perform ARP spoofing or DNS poisoning to redirect `mirror-rest` to a server returning HTTP 200 with a non-JSON body. Additionally, if the mirror node is a shared public service, a temporary misconfiguration (e.g., a CDN returning an HTML error page with status 200) triggers the same code path with no retry. The attack is repeatable: every pinger restart hits the same code path.

### Recommendation

1. **Validate `Content-Type` before decoding:** After the status code check, assert `strings.Contains(resp.Header.Get("Content-Type"), "application/json")` and return `retry=false` only if the content type is definitively wrong (or `retry=true` if it may be transient).
2. **Make decode errors retryable:** Change line 97 to return `retry=true` so that transient or injected bad responses are retried up to `mirrorNodeClientMaxRetries` times:
   ```go
   return nil, true, fmt.Errorf("decode mirror nodes: %w", err)
   ```
3. **Use HTTPS for the mirror REST URL** to prevent MITM injection of malicious responses.

### Proof of Concept

1. Configure pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>/`.
2. Have the attacker host respond to `GET /api/v1/network/nodes` with:
   ```
   HTTP/1.1 200 OK
   Content-Type: text/html
   
   <html><body>Service Unavailable</body></html>
   ```
3. Start the pinger. `fetchMirrorNodeNetwork` receives HTTP 200, passes the status check, calls `json.NewDecoder(resp.Body).Decode(&payload)`, which returns `invalid character '<' looking for beginning of value`.
4. Returns `(nil, false, err)` → `buildNetworkFromMirrorNodes` breaks immediately (no retries despite `mirrorNodeClientMaxRetries=10`).
5. `newClient` returns error → `main()` calls `log.Fatalf` → pinger exits.
6. In a real MITM scenario: attacker intercepts the plaintext HTTP connection to `mirror-rest:5551` and injects the above response. Same outcome, no privileged access required beyond network adjacency.
