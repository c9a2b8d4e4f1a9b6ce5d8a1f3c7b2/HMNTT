### Title
Connection Pool Exhaustion via Unbounded Query Hang During Network Partition in `getScheduleById()`

### Summary
`getScheduleById()` in `rest/schedules.js` calls `pool.queryQuietly()` with no application-level query timeout. The only timeout defense is a PostgreSQL server-side `statement_timeout`, which is rendered ineffective during a network partition because the kill signal cannot traverse the broken network path. An unprivileged attacker sending concurrent requests to `/schedules/:scheduleId` during or coinciding with a network partition can exhaust the pg pool's 10 default connections, causing a cascading denial-of-service across all endpoints sharing the same pool.

### Finding Description

**Exact code path:**

`rest/schedules.js`, `getScheduleById()`, line 120:
```js
const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId);
``` [1](#0-0) 

`queryQuietly` is defined in `rest/utils.js` lines 1481–1546. When called without a `preQueryHint` (which is always the case for `getScheduleById`), it delegates directly to `this.query(query, params)` — node-postgres's pool-managed query — with no `Promise.race`, `AbortController`, or any application-level deadline: [2](#0-1) 

The pool is initialized in `rest/dbpool.js` with:
- `max: config.db.pool.maxConnections` — default **10**
- `connectionTimeoutMillis: config.db.pool.connectionTimeout` — default **20000 ms** (time to wait for a free slot, not a query timeout)
- `statement_timeout: config.db.pool.statementTimeout` — default **20000 ms** (PostgreSQL server-side) [3](#0-2) 

**Root cause and failed assumption:**

The design assumes `statement_timeout` will bound query duration. This assumption fails during a network partition: `statement_timeout` is enforced by the PostgreSQL server, which will kill the backend process after 20 s — but the resulting error notification travels over the same broken network path and never reaches the Node.js client. The pg client's TCP socket remains open and blocking, held by the pool, until the OS TCP keepalive fires (Linux default: ~2 hours). `connectionTimeoutMillis` only governs how long a caller waits to *acquire* a slot from the pool, not how long a query can run.

There is no application-level timeout (e.g., `Promise.race(pool.query(...), timeout(20000))`) anywhere in `queryQuietly`. [4](#0-3) 

**No rate limiting on the REST API schedules endpoint:**

The throttling infrastructure found in the codebase (bucket4j `ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` Java service. The Node.js REST API has no per-endpoint concurrency limit or rate limiter protecting `/schedules/:scheduleId`. [5](#0-4) 

### Impact Explanation

With `max: 10` pool connections (documented default), an attacker needs only 10 concurrent in-flight requests to `/schedules/:scheduleId` that coincide with a network partition to hold every pool slot indefinitely. Once the pool is exhausted:

- All subsequent requests to **any** endpoint using `pool.queryQuietly()` — accounts, tokens, transactions, contracts, topic messages, etc. — will block waiting for a free connection for up to `connectionTimeoutMillis` (20 s) and then fail with a pool timeout error, returning 500 to all clients.
- The service is effectively down for the duration of the partition plus OS TCP keepalive timeout.
- Recovery requires either the network partition to heal (allowing the server-side kill to propagate) or a process restart.

Severity: **High** — complete REST API denial-of-service with a shared, small-capacity pool. [6](#0-5) [7](#0-6) 

### Likelihood Explanation

**Attacker capability required:**
- Sending 10 concurrent HTTP requests: zero privilege, trivially scriptable.
- Causing or timing requests to a network partition: the attacker does not need to cause the partition themselves. Transient network events (cloud provider blips, BGP flaps, firewall rule changes, maintenance windows) are common in production environments. An attacker who monitors for elevated latency to the DB (e.g., via timing side-channels on response times) can flood the endpoint precisely when a partition begins.

**Repeatability:** Once the pool is exhausted, the condition is self-sustaining until the partition heals or the process restarts. The attacker does not need to maintain the flood — 10 requests sent at the right moment are sufficient.

### Recommendation

1. **Add an application-level query timeout inside `queryQuietly`** using `Promise.race` or a `pg` query with a `query_timeout` option, so that even if the server-side signal is lost, the Node.js client abandons the query and releases the pool slot:
   ```js
   const queryPromise = this.query(query, params);
   const timeoutPromise = new Promise((_, reject) =>
     setTimeout(() => reject(new Error('query timeout')), APP_QUERY_TIMEOUT_MS)
   );
   result = await Promise.race([queryPromise, timeoutPromise]);
   ```
2. **Increase `maxConnections`** or deploy a connection pooler (PgBouncer) in front of PostgreSQL to absorb bursts.
3. **Add per-IP or global concurrency limiting** to the Node.js REST middleware (e.g., using `express-rate-limit` or an ingress-level `inFlightReq` limit similar to what the Rosetta chart already configures).
4. **Enable TCP keepalive with short intervals** on the pg pool connections so that partitioned sockets are detected and closed within seconds rather than hours.

  Pool.prototype.queryQuietly = async function (query, params = [], preQueryHint = undefined) {
    let client;
    let result;
    let startTime;

    params = Array.isArray(params) ? params : [params];
    const clientErrorCallback = (error) => {
      logger.error(`error event emitted on pg pool. ${error.stack}`);
    };

    const traceEnabled = logger.isTraceEnabled();
    if (traceEnabled || isTestEnv()) {
      const callerInfo = new Error().stack
        .split('\n')
        .splice(1)
        .filter((s) => !(s.includes('utils.js') || s.includes('baseService.js')))
        .map((entry) => {
          const result = entry.match(/^\s*at\s+(\S+)[^(]+\((.*\/(.*\.js)):(\d+):.*\)$/);
          return result?.length === 5 && {function: result[1], file: result[3], line: result[4], path: result[2]};
        })[0];

      if (isTestEnv()) {
        await recordQuery(callerInfo, query);
      }

      if (traceEnabled) {
        startTime = Date.now();
        const {format} = await import('sql-formatter');
        const prettyQuery = format(query, {language: 'postgresql'});
        logger.trace(
          `${callerInfo.function} (${callerInfo.file}:${
            callerInfo.line
          })\nquery: ${prettyQuery}\nparams: ${JSONStringify(params)}`
        );
      }
    }

    try {
      if (!preQueryHint) {
        result = await this.query(query, params);
      } else {
        client = await this.connect();
        client.on('error', clientErrorCallback);
        await client.query(`begin; ${preQueryHint}`);
        result = await client.query(query, params);
        await client.query('commit');
      }

      if (traceEnabled) {
        const elapsed = Date.now() - startTime;
        logger.trace(`Query took ${elapsed} ms and returned ${result.rows.length} entries`);
      }

      return result;
    } catch (err) {
      if (client !== undefined) {
        await client.query('rollback');
      }
      throw new DbError(err.message);
    } finally {
      if (client !== undefined) {
        client.off('error', clientErrorCallback);
        client.release();
      }
    }
  };
```
