### Title
Cache Poisoning via Non-Existent Storage Slots Causes Unbounded DB Query Amplification in `findStorageBatch()`

### Summary
When an unprivileged user requests storage slots that do not exist in the database, those slot keys are permanently tracked in the per-contract `slotsPerContract` cache but no entry is ever written to `contractStateCache` for them. Because `contractStateCache` has no sentinel for "slot confirmed absent," every subsequent request for any non-existent slot always misses `contractStateCache` and re-invokes `findStorageBatch()`, which issues a DB query containing the full accumulated list of all previously requested (non-existent) slot keys. An attacker can grow this list up to the cache size limit, causing every future storage lookup for that contract to execute an increasingly expensive batch DB query.

### Finding Description

**Code path:**

`findStorage()` (lines 58–70) checks `contractStateCache` first:

```java
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);
if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
return findStorageBatch(contractId, key);
``` [1](#0-0) 

`findStorageBatch()` (lines 85–122) adds the requested key to the per-contract slot-tracking cache unconditionally:

```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
``` [2](#0-1) 

It then collects **all** accumulated slot keys and fires a batch DB query:

```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
``` [3](#0-2) 

Only slots **returned by the DB** are written into `contractStateCache`:

```java
for (final var contractSlotValue : contractSlotValues) {
    contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);
    ...
}
``` [4](#0-3) 

**Root cause:** Non-existent slots are never written to `contractStateCache`. The guard at line 65 only short-circuits on `cachedValue != null && cachedValue != EMPTY_VALUE`. For a non-existent slot, `contractStateCache.get(...)` returns `null` (not `EMPTY_VALUE`), so the guard never fires and `findStorageBatch()` is called on every single request for that slot. [5](#0-4) 

**Accumulation:** The per-contract `slotsPerContract` cache is configured with `maximumSize=1500` and `expireAfterAccess=5m`. The outer `contractSlots` cache is `maximumSize=3000`, `expireAfterAccess=5m`. [6](#0-5) 

An attacker who requests 1500 unique non-existent slot keys for one contract fills the per-contract cache. Every subsequent call to `findStorageBatch()` for that contract (triggered by any slot miss) will include all 1500 non-existent keys in the SQL `IN (...)` clause sent to the database. [7](#0-6) 

### Impact Explanation

Each `eth_call` or `eth_estimateGas` that touches any storage slot of the targeted contract triggers a DB query with up to 1500 32-byte slot keys in the `IN` clause. Because `contractStateCache` has `expireAfterWrite=2s`, even legitimate callers continuously re-trigger `findStorageBatch()`, each time paying the cost of the inflated query. [8](#0-7) 

This constitutes a sustained, amplified DB load increase achievable without any privileged access, directly matching the "30%+ resource consumption increase" threat model. The attack is persistent for the 5-minute `expireAfterAccess` window and can be trivially renewed.

### Likelihood Explanation

Any user who can submit `eth_call` or `eth_estimateGas` requests (i.e., any public API user) can trigger this. No authentication, no tokens, no special role. The attacker only needs to know a valid contract address and send requests reading from 1500 distinct storage slots. This is trivially scriptable. The attack is repeatable every 5 minutes to keep the poisoned cache alive indefinitely.

### Recommendation

Write a sentinel value into `contractStateCache` for slots confirmed absent by the DB query. After the batch query returns, for every slot key in `cachedSlots` that was **not** present in `contractSlotValues`, explicitly store a sentinel (e.g., `EMPTY_VALUE`) in `contractStateCache`. Then update the guard in `findStorage()` to also short-circuit on `EMPTY_VALUE` (returning `Optional.empty()`), preventing repeated `findStorageBatch()` calls for confirmed-absent slots:

```java
// After the batch query loop, mark absent slots:
for (ByteBuffer slotKey : cachedSlotKeys) {
    SimpleKey ck = generateCacheKey(contractId, slotKey.array());
    if (contractStateCache.get(ck, byte[].class) == null) {
        contractStateCache.put(ck, EMPTY_VALUE);
    }
}
```

And in `findStorage()`:
```java
if (cachedValue != null) {  // treat EMPTY_VALUE as "confirmed absent"
    return cachedValue == EMPTY_VALUE ? Optional.empty() : Optional.of(cachedValue);
}
``` [1](#0-0) 

### Proof of Concept

1. Identify any deployed contract address `C` on the network.
2. Script 1500 `eth_call` requests, each reading a distinct storage slot `S_1 ... S_1500` that do not exist in the DB (e.g., random 32-byte keys). Each call causes `findStorageBatch()` to add `S_i` to the per-contract `slotsPerContract` cache.
3. After step 2, send any `eth_call` that reads any storage slot of contract `C` (even a real one). Observe that `findStorageBatch()` is invoked with all 1500 keys in the SQL `IN` clause.
4. Repeat step 3 in a tight loop. Each iteration fires a DB query with 1500 slot keys. Measure DB CPU/IO increase vs. baseline.
5. Renew the attack every ~5 minutes (before `expireAfterAccess` eviction) to maintain the amplified load indefinitely. [9](#0-8)

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
```
