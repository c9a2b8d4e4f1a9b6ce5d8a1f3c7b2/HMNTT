### Title
Broken Pagination Link Due to Invalid `DEFAULT_SERIAL_NUMBER = "0L"` String in EXTRACTOR

### Summary
In `TokenAirdropsController`, the constant `DEFAULT_SERIAL_NUMBER` is set to the Java string `"0L"` (two characters: `'0'` and `'L'`). When a fungible token airdrop (with `null` serial number) is the last item on a page, the EXTRACTOR places `serial_number=0L` into the generated "next" pagination link. When any user follows that link, `NumberRangeParameter.valueOf("0L")` calls `Long.parseLong("0L")`, which throws a `NumberFormatException` because Java's `Long.parseLong` does not accept the `L` suffix. This breaks pagination entirely for any result set containing fungible token airdrops.

### Finding Description

**Root cause — wrong string constant:** [1](#0-0) 

`DEFAULT_SERIAL_NUMBER = "0L"` is the two-character string `"0L"`, not the numeric string `"0"`. `DEFAULT_SERIAL_VALUE = 0L` is the correct long zero, but it is never used in the EXTRACTOR.

**EXTRACTOR emits the broken string:** [2](#0-1) 

When `serialNumber == null` (all fungible token airdrops), the map entry `SERIAL_NUMBER → "0L"` is placed into the pagination link, producing a URL like:
```
/api/v1/accounts/0.0.X/airdrops/outstanding?serial_number=0L&...
```

**Parser cannot handle `"0L"`:** [3](#0-2) 

`"0L".split(":")` → `["0L"]` (length 1, no colon), so `getNumberValue("0L")` is called, which calls `Long.parseLong("0L")`. This throws `NumberFormatException` because the `L` suffix is a Java source-code literal notation, not accepted by `Long.parseLong`.

**`getSort()` does not fix the EXTRACTOR:** [4](#0-3) 

`getSort()` correctly detects the null/zero serial case and omits `SERIAL_NUMBER` from the sort order, but this has no effect on the EXTRACTOR, which still unconditionally emits `serial_number=0L` in the link.

**Exploit flow:**
1. Attacker (or any user) calls `GET /api/v1/accounts/{id}/airdrops/outstanding` (no special privileges needed).
2. The result set contains at least one fungible token airdrop (null serial number) as the last item on the page, and the total count exceeds the page limit so a "next" link is generated.
3. The server returns a "next" link containing `serial_number=0L`.
4. The user follows the link; Spring binds `"0L"` to `NumberRangeParameter` via `valueOf("0L")`.
5. `Long.parseLong("0L")` throws `NumberFormatException` → HTTP 400/500 error.
6. Pagination is permanently broken for that result set.

### Impact Explanation
Any unprivileged user who receives a paginated response containing fungible token airdrops cannot advance to the next page. The server itself generates the broken link, so no special crafting is required. This is a functional denial-of-service on the pagination feature for all fungible airdrop queries. No funds are directly at risk, but the feature is rendered unusable.

### Likelihood Explanation
Fungible token airdrops (null serial number) are a normal, expected case. Any deployment with more fungible airdrops than the page limit will expose this bug to every user who tries to paginate. No authentication, elevated privileges, or special knowledge is required — the broken link is served directly in the API response.

### Recommendation
Change `DEFAULT_SERIAL_NUMBER` to the string `"0"` instead of `"0L"`:

```java
// Before (broken):
private static final String DEFAULT_SERIAL_NUMBER = "0L";

// After (correct):
private static final String DEFAULT_SERIAL_NUMBER = "0";
```

Alternatively, use `Long.toString(DEFAULT_SERIAL_VALUE)` to derive the string from the numeric constant, eliminating the mismatch entirely:

```java
private static final String DEFAULT_SERIAL_NUMBER = Long.toString(DEFAULT_SERIAL_VALUE);
```

Also add a unit test asserting that `NumberRangeParameter.valueOf(DEFAULT_SERIAL_NUMBER)` parses without exception.

### Proof of Concept
1. Ensure the account has more fungible token airdrops than the default page limit (25).
2. Send: `GET /api/v1/accounts/0.0.1234/airdrops/outstanding`
3. Observe the `links.next` field in the JSON response contains `serial_number=0L`.
4. Follow the `next` link exactly as returned.
5. Observe HTTP 400 or 500 error caused by `NumberFormatException` from `Long.parseLong("0L")`.
