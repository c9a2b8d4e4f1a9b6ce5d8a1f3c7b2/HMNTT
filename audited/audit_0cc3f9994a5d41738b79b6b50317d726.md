### Title
Unauthenticated X-Real-IP Header Spoofing Allows Audit Trail Falsification in Rosetta Tracing Middleware

### Summary
The `getClientIpAddress()` function in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` (and `X-Forwarded-For`) HTTP header without verifying it originates from a trusted proxy. Any unprivileged external client can set an arbitrary value in this header, causing `TracingMiddleware` to log a falsified source IP for every Rosetta API call — including transaction construction and submission requests — permanently corrupting the audit trail of who initiated which operations.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64 — blindly trusted

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67 — blindly trusted
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only reached if both absent
    }

    return ipAddress
}
```

The function checks `X-Real-IP` first, then `X-Forwarded-For`, and only falls back to `r.RemoteAddr` (the actual TCP peer address) if both headers are absent. There is no allowlist of trusted upstream proxy IPs, no validation that the header value is a legitimate IP, and no stripping of client-supplied proxy headers before this function runs.

`TracingMiddleware` (lines 43–61) calls `getClientIpAddress()` and embeds the result directly into the log line:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
log.Info(message)
```

`main.go` (lines 217–219) shows the server is exposed directly via `httpServer.ListenAndServe()` with no mandatory trusted-proxy layer enforced at the application level:

```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```

**Root cause:** The code assumes `X-Real-IP` is always set by a trusted reverse proxy, but HTTP headers are fully attacker-controlled. No check distinguishes a proxy-set header from a client-set one.

**Why existing checks fail:** The test in `trace_test.go` lines 52–54 explicitly *validates* that a client-supplied `X-Real-IP` header is accepted and logged — confirming the behavior is unguarded by design. There is no middleware, firewall rule, or configuration option in the codebase that strips or validates these headers before `getClientIpAddress()` reads them.

### Impact Explanation
Every Rosetta API call — including `/construction/submit` (transaction broadcast), `/construction/payloads`, and `/block` queries — is logged with the attacker-controlled IP. An attacker who submits fraudulent or malicious transactions via the Rosetta API can attribute those submissions to any IP address (e.g., a known-good internal service IP, a competitor's IP, or a non-existent address), making forensic attribution impossible. The audit trail, which is the primary mechanism for post-incident investigation of who initiated which blockchain operations, is permanently falsified for every spoofed request.

### Likelihood Explanation
Exploitation requires zero privileges — only the ability to send an HTTP request with a custom header, which any HTTP client supports natively (`curl -H "X-Real-IP: 1.2.3.4" ...`). The attack is trivially repeatable, leaves no trace of the real source IP in logs, and requires no authentication, credentials, or special network position. If the Rosetta port is reachable from the internet (common for public blockchain infrastructure), every external user can exploit this.

### Recommendation
Implement trusted-proxy validation before accepting forwarded IP headers:

1. **Allowlist trusted proxy CIDRs** in configuration. Only accept `X-Real-IP` / `X-Forwarded-For` if `r.RemoteAddr` falls within the trusted proxy range.
2. **Default to `r.RemoteAddr`** when no trusted proxy is configured, ignoring all forwarded headers.
3. Use a well-tested library (e.g., `github.com/realclientip/realclientip-go`) that implements RFC 7239-compliant trusted-proxy-aware IP extraction.

Example fix sketch for `getClientIpAddress()`:
```go
func getClientIpAddress(r *http.Request, trustedProxies []*net.IPNet) string {
    remoteIP, _, _ := net.SplitHostPort(r.RemoteAddr)
    if isTrustedProxy(remoteIP, trustedProxies) {
        if ip := r.Header.Get(xRealIpHeader); ip != "" {
            return ip
        }
        if ip := r.Header.Get(xForwardedForHeader); ip != "" {
            return strings.TrimSpace(strings.Split(ip, ",")[0])
        }
    }
    return remoteIP
}
```

### Proof of Concept
**Preconditions:** Rosetta server is reachable (default port 5700). No special privileges required.

**Steps:**
```bash
# Step 1: Normal request — real IP logged
curl -s http://<rosetta-host>:5700/network/list \
  -H "Content-Type: application/json" \
  -d '{"metadata":{}}'
# Log output: "1.2.3.4 POST /network/list (200) in 5ms"  ← real attacker IP

# Step 2: Spoofed request — falsified IP logged
curl -s http://<rosetta-host>:5700/network/list \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 10.0.0.1" \
  -d '{"metadata":{}}'
# Log output: "10.0.0.1 POST /network/list (200) in 5ms"  ← FALSIFIED IP

# Step 3: Submit a transaction with spoofed identity
curl -s http://<rosetta-host>:5700/construction/submit \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 192.168.100.50" \
  -d '{"network_identifier":{...},"signed_transaction":"<hex>"}'
# Audit log permanently records 192.168.100.50 as the submitter, not the real attacker IP
```

**Result:** The server logs record the spoofed IP for all requests. The real attacker IP never appears in any log entry. Forensic investigation of transaction submissions is rendered unreliable.
