### Title
Unauthenticated DoS via Unrate-Limited `IllegalArgumentException` on `GET /api/v1/network/nodes`

### Summary
The `getNodes()` handler in `NetworkController.java` throws an `IllegalArgumentException` when `file.id` is supplied with any non-equality operator (e.g., `gt:`, `lt:`). No rate limiting exists anywhere in the `rest-java` module — unlike the `web3` module which has `ThrottleManager`/Bucket4j — so any unauthenticated caller can flood this endpoint with malformed `file.id` operators at arbitrary request rates, exhausting the servlet thread pool and degrading or denying service to legitimate users.

### Finding Description

**Exact code path:**

`NetworkController.java` lines 152–157 — the operator check fires before any DB access:

```java
@GetMapping("/nodes")
ResponseEntity<NetworkNodesResponse> getNodes(@RequestParameter NetworkNodeRequest request) {
    final var fileId = request.getFileId();
    if (fileId != null && fileId.operator() != RangeOperator.EQ) {
        throw new IllegalArgumentException("Only equality operator is supported for file.id");
    }
``` [1](#0-0) 

The thrown `IllegalArgumentException` is caught by `GenericControllerAdvice.badRequest()` and returns HTTP 400:

```java
@ExceptionHandler({
    HttpMessageConversionException.class,
    IllegalArgumentException.class,
    InvalidEntityException.class
})
private ResponseEntity<Object> badRequest(final Exception e, final WebRequest request) {
    return handleExceptionInternal(e, null, null, BAD_REQUEST, request);
}
``` [2](#0-1) 

**Root cause — no rate limiting in rest-java:**

The `rest-java` module contains only `LoggingFilter` and `MetricsFilter`:


There is no `ThrottleFilter`, no Bucket4j bean, and no per-IP connection limit. By contrast, the `web3` module has a full `ThrottleConfiguration` with `rateLimitBucket` (default 500 req/s) and `gasLimitBucket`: [3](#0-2) 

**Failed assumption:** The developer assumed that returning HTTP 400 quickly (before any DB query) is sufficient protection. While the error path is cheap per-request, it still occupies a Tomcat worker thread for the full request lifecycle (parameter binding, exception dispatch, response serialization). At high concurrency, this exhausts the thread pool.

### Impact Explanation

An attacker flooding `GET /api/v1/network/nodes?file.id=gt:101` at high rate will:
1. Saturate the Tomcat thread pool (default 200 threads), blocking legitimate requests to **all** `rest-java` endpoints — not just `/network/nodes`.
2. Cause connection queue overflow, resulting in TCP connection refusals for all clients.
3. Potentially trigger cascading failures in upstream load balancers or health checks, causing the instance to be marked unhealthy.

The `/api/v1/network/nodes` endpoint is specifically sensitive because it exposes the network's address book — disrupting it impairs clients' ability to discover active consensus nodes.

### Likelihood Explanation

- **Zero privilege required**: no API key, no authentication, no session.
- **Trivially scriptable**: a single `while true; do curl ...; done` loop or any HTTP benchmarking tool suffices.
- **Amplifiable**: multiple source IPs (botnet/VPS) bypass any IP-level firewall rules that might exist at the infrastructure layer.
- **Repeatable indefinitely**: no backoff, no lockout, no CAPTCHA.

### Recommendation

1. **Add rate limiting to rest-java**: Port the Bucket4j `ThrottleConfiguration` pattern from the `web3` module into `rest-java`, applying a per-IP or global request-per-second limit via a `OncePerRequestFilter`.
2. **Return 400 before parameter binding where possible**: Reject syntactically invalid operator prefixes at the `EntityIdRangeParameter` converter level so the request never reaches the controller.
3. **Deploy infrastructure-level rate limiting**: Configure nginx/ingress rate limiting (e.g., `limit_req_zone`) as a defense-in-depth layer independent of application code.
4. **Consider converting the operator check to a binding-time validation** (e.g., a `@Constraint` on `NetworkNodeRequest.fileId`) so it is rejected during Spring's `@RequestParameter` binding and counted as a bind error, enabling uniform bad-request metrics and easier rate-limit targeting.

### Proof of Concept

```bash

curl -v "https://<mirror-node-host>/api/v1/network/nodes?file.id=gt:101"

# Flood attack — no authentication needed
ab -n 100000 -c 500 \
  "https://<mirror-node-host>/api/v1/network/nodes?file.id=gt:101"
