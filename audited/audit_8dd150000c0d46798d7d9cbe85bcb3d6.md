### Title
Premature DB Query Issued for Alias with Non-Equality Operator in `parseAccountIdQueryParam`

### Summary
In `rest/balances.js`, `parseAccountIdQueryParam` uses `utils.parseParams`, which calls the value-processing callback before the operator-checking callback. When a valid alias is supplied with a range operator (e.g., `gte:`), `EntityService.getAccountIdFromAlias` is invoked and a real DB query is dispatched before the operator check can reject the request. The resulting Promise is discarded unhandled, but the DB query runs to completion, wasting a connection-pool slot per malformed request.

### Finding Description

**Exact code path:**

`rest/utils.js` `parseParams` (lines 565–606): the value callback fires at line 584, the operator callback fires at line 589. [1](#0-0) 

`rest/balances.js` `parseAccountIdQueryParam` value callback (lines 331–332): when `AccountAlias.isValid(value, true)` is true and `evmAliasAddressCount` becomes 1, `EntityService.getAccountIdFromAlias` is called immediately — it is `async`, so it starts executing synchronously until its first internal `await super.getRows(...)`, which dispatches the SQL query to the connection pool and returns a Promise. [2](#0-1) 

The operator callback (lines 343–349) then throws `InvalidArgumentError` because `op !== opsMap.eq`. [3](#0-2) 

The Promise from `getAccountIdFromAlias` is never awaited; it floats unhandled. The DB query (`SELECT id FROM entity WHERE coalesce(deleted, false) <> true AND alias = $1`) runs to completion and the result is silently discarded. [4](#0-3) 

**Root cause:** `parseParams` is synchronous and does not validate the operator before invoking the value callback. The value callback unconditionally initiates an async DB lookup for aliases, with no guard on the operator.

**Failed assumption:** The design assumes the operator check in `processQuery` acts as a gate before any expensive work. In reality, `processValue` runs first and the async DB call escapes before the gate fires.

**Pre-validation is insufficient:** `balanceFilterValidator` → `utils.validateOpAndValue(op, val)` runs at the raw string level and does not know the value is an alias, so `gte:VALIDBASE32` passes and `parseAccountIdQueryParam` is reached. [5](#0-4) [6](#0-5) 

### Impact Explanation
An unauthenticated attacker can force one real DB query per HTTP request against the `entity` table. At high request rates this exhausts the DB connection pool, degrading or denying service to legitimate users. The query is not trivially cheap: it involves a `coalesce` predicate and an alias byte-array lookup. Connection pool exhaustion causes all subsequent queries (including legitimate ones) to queue or fail.

### Likelihood Explanation
No authentication is required. The endpoint (`GET /api/v1/balances?account.id=gte:<base32>`) is publicly reachable. Any valid base32 string (e.g., `AABBCC22`) satisfies `AccountAlias.isValid(value, true)`. The attack is trivially scriptable, repeatable at high frequency, and requires no knowledge of actual account state.

### Recommendation
1. **Check the operator before initiating the DB lookup.** In the value callback, extract the operator context and reject non-`eq` operators for aliases before calling `EntityService.getAccountIdFromAlias`.
2. **Alternatively**, restructure `parseParams` to pass the operator to `processValue` so it can gate expensive async work.
3. **Short-term mitigation**: add a pre-check in `balanceFilterValidator` / `validateOpAndValue` that rejects range operators when the value matches the alias regex, before `parseAccountIdQueryParam` is ever called.

### Proof of Concept

```
# Send a flood of requests with a valid base32 alias and a range operator
for i in $(seq 1 1000); do
  curl -s "http://<mirror-node-host>/api/v1/balances?account.id=gte:AABBCC22" &
done
```

Each request:
1. Passes `validateReq` (raw string `gte:AABBCC22` is not rejected by `validateOpAndValue`)
2. Enters `parseAccountIdQueryParam` → `parseParams`
3. `processValue("AABBCC22")`: `AccountAlias.isValid("AABBCC22", true)` → true; `EntityService.getAccountIdFromAlias` fires a `SELECT` against `entity` table
4. `processQuery(">=", Promise)`: throws `InvalidArgumentError` → HTTP 400 returned
5. DB query runs to completion in background, connection slot held until query finishes

Result: 1000 unneeded DB queries dispatched; connection pool saturated; legitimate queries queue or time out.
