### Title
Unbounded Concurrent Subscriptions in `PollingTopicListener.listen()` Exhaust Shared `boundedElastic()` Scheduler, Delaying Fee-Schedule Topic Delivery

### Summary
`PollingTopicListener.listen()` creates a new independent DB-polling loop per subscription, all sharing the process-global `Schedulers.boundedElastic()` thread pool. No global subscription cap is enforced — the only per-connection limit (`maxConcurrentCallsPerConnection = 5`) is trivially bypassed by opening many connections. An unprivileged attacker can saturate the `boundedElastic()` pool, causing legitimate fee-schedule topic messages to be delayed or dropped, freezing clients at stale fee values.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (line 44–48) → `TopicMessageServiceImpl.subscribeTopic()` (line 59) → `incomingMessages()` (line 120) → `topicListener.listen(newFilter)` → `PollingTopicListener.listen()` (line 34).

**Root cause — `PollingTopicListener.java` lines 31–49:**

```java
private final Scheduler scheduler = Schedulers.boundedElastic();   // global singleton

public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);
    Duration interval = listenerProperties.getInterval();

    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)          // schedules on boundedElastic
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                    .jitter(0.1)
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))               // every repeat on boundedElastic
            ...
}
```

`poll()` calls `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` — a **blocking JDBC stream** — which holds a `boundedElastic()` thread for the full duration of each DB query. Every independent subscription occupies a thread for ~T ms every 500 ms (default interval). With N subscriptions and query time T:

```
threads_needed ≈ N × T / 500
```

On a 4-core host the pool cap is 40 threads. With T = 50 ms (moderate DB load), only **400 concurrent subscriptions** saturate the pool. Once saturated, new tasks are queued (cap 100 000) or rejected with `RejectedExecutionException`, stalling all downstream subscribers including those watching the fee-schedule topic.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (`NettyProperties` line 14, applied in `GrpcConfiguration` line 33): limits calls *per TCP connection*, not globally. An attacker opens many connections.
- `subscriberCount` (`TopicMessageServiceImpl` line 48–55): registered as a Micrometer `Gauge` for observability only; never read back to reject new subscriptions.
- No authentication or authorization on `ConsensusController.subscribeTopic()`.
- No IP-level or token-level rate limit on the gRPC port.

### Impact Explanation
When `boundedElastic()` is saturated, the periodic polling tasks for all active subscriptions — including those watching the Hedera fee-schedule topic (e.g., `0.0.111`) — are queued behind attacker tasks. Fee-update messages are delivered late or not at all. Clients that rely on the mirror node's HCS stream to refresh their local fee schedule continue using stale values, causing transactions to be under- or over-charged outside design parameters. This is a Denial-of-Service against a protocol-critical data feed, not merely a service outage.

### Likelihood Explanation
- Requires zero privileges; the gRPC port is publicly reachable by design.
- A single attacker machine can open hundreds of TCP connections and 5 streams each with a trivial gRPC client loop.
- The attack is repeatable and self-sustaining: each subscription auto-repeats via `RepeatSpec.times(Long.MAX_VALUE)`.
- Effective only when `hiero.mirror.grpc.listener.type = POLL` is configured (non-default, but documented and used in environments without Redis). Operators who switch to POLL mode for simplicity are fully exposed.

### Recommendation
1. **Enforce a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: read `subscriberCount` before incrementing and return `Mono.error(new StatusRuntimeException(RESOURCE_EXHAUSTED))` when the configured maximum is exceeded.
2. **Isolate the polling scheduler**: replace `Schedulers.boundedElastic()` in `PollingTopicListener` with a dedicated `Schedulers.newBoundedElastic(...)` instance sized to the expected subscription cap, preventing listener starvation from affecting other reactor chains (e.g., the safety-check `subscribeOn(Schedulers.boundedElastic())` in `TopicMessageServiceImpl` line 70).
3. **Add per-IP or per-connection subscription rate limiting** at the Netty layer or via a gRPC interceptor.
4. **Make `poll()` non-blocking**: replace `Flux.fromStream(...)` with a reactive R2DBC query so threads are not held during DB I/O.

### Proof of Concept
```python
import grpc, threading
from hedera import mirror_pb2, mirror_pb2_grpc

TARGET = "mirror-node-host:5600"
# Fee-schedule topic on mainnet
TOPIC = mirror_pb2.TopicID(shardNum=0, realmNum=0, topicNum=111)

def flood():
    ch = grpc.insecure_channel(TARGET)
    stub = mirror_pb2_grpc.ConsensusServiceStub(ch)
    query = mirror_pb2.ConsensusTopicQuery(topicID=TOPIC)
    # Open 5 streams per connection (maxConcurrentCallsPerConnection default)
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    for s in streams:
        for _ in s:   # consume indefinitely
            pass

# Open 200 connections → 1000 concurrent subscriptions
threads = [threading.Thread(target=flood) for _ in range(200)]
for t in threads: t.start()
# boundedElastic() saturates; legitimate fee-schedule subscribers stall
```
