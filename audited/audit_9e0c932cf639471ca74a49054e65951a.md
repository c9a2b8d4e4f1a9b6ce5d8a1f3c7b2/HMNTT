### Title
Missing Port Range Validation in `fetchMirrorNodeNetwork` Enables Mirror-Server-Controlled Network Partition

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` deserializes the `Port` field of each `serviceEndpoint` into a Go `int` with no upper-bound check. The only guard is `ep.Port == 0`, which skips port-zero entries. An attacker who can influence the mirror REST response (e.g., via MITM on the plain-HTTP default transport, or a compromised mirror node) can return port values above 65535, which are accepted, formatted into host:port strings, and handed to the Hiero SDK. Because the resulting network map is non-empty, no error is returned, and the SDK silently attempts connections on invalid ports, causing a complete network partition.

### Finding Description
**Exact location:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–124.

```go
// line 30-34: Port is plain int — no range constraint
type serviceEndpoint struct {
    DomainName  string `json:"domain_name"`
    IPAddressV4 string `json:"ip_address_v4"`
    Port        int    `json:"port"`
}

// lines 113-124: only guard is == 0
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← only check; no upper-bound
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId    // ← invalid port enters the map
}
```

**Root cause:** `Port int` can hold any value a 64-bit Go integer can represent. `json.Unmarshal` will happily decode `"port": 99999` (or `2147483647`) into it. The sole filter `ep.Port == 0` does not reject out-of-range values.

**Why the downstream guard also fails:** After the loop, the only check is:
```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found …")
}
```
If the attacker returns even one node with a valid-looking host but an invalid port, `len(network) > 0` is true, no error is raised, and `buildNetworkFromMirrorNodes` returns the poisoned map to `newClient` → `hiero.ClientForNetwork(netmap)`.

**Transport exposure:** `config.go` line 37 shows the documented default URL scheme is `http://` (plain HTTP). The HTTP client is constructed with no TLS configuration and no certificate pinning:
```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```
Any on-path network attacker can intercept and rewrite the JSON response.

### Impact Explanation
The pinger's entire node network is derived exclusively from this mirror REST call when `network=other`. A poisoned response with all ports > 65535 causes every gRPC dial attempt by the Hiero SDK to fail (the OS rejects port numbers outside 0–65535 at the socket layer). The pinger cannot submit any transactions, producing a sustained, silent network partition. No error is surfaced to the caller because the network map is non-empty. Severity: **High** — complete loss of pinger functionality with no self-healing until the mirror REST response is corrected.

### Likelihood Explanation
The default mirror REST URL uses plain HTTP (`http://mirror-rest:5551`). Any attacker with network adjacency (same cluster, compromised sidecar, ARP/BGP poisoning) can perform a MITM and inject a crafted `nodesEnvelope`. No credentials or special privileges on the target system are required — only the ability to intercept one unauthenticated HTTP response. The attack is repeatable on every retry cycle.

### Recommendation
Add an explicit port range check immediately after the `ep.Port == 0` guard:

```go
if host == "" || ep.Port <= 0 || ep.Port > 65535 {
    continue
}
```

Additionally:
- Enforce HTTPS for the mirror REST URL (reject `http://` schemes at config load time, or enforce TLS in the HTTP client).
- Log and count skipped endpoints with out-of-range ports so operators are alerted to anomalous mirror responses.

### Proof of Concept
1. Stand up a mock HTTP server that returns:
```json
{
  "nodes": [{
    "node_account_id": "0.0.3",
    "service_endpoints": [{"ip_address_v4": "1.2.3.4", "port": 99999}]
  }],
  "links": {"next": null}
}
```
2. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<mock-server>`.
3. Start the pinger. `buildNetworkFromMirrorNodes` returns `{"1.2.3.4:99999": AccountID{…}}` with no error.
4. The Hiero SDK attempts to dial `1.2.3.4:99999`; the OS rejects the connection (invalid port).
