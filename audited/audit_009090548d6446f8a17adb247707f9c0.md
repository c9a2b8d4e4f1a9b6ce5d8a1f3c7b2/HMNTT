### Title
Unauthenticated Correlated Subquery DoS via `/api/v1/balances` No-Timestamp Path

### Summary
The `/api/v1/balances` endpoint in `rest/balances.js` executes a correlated subquery against `token_account` for every result row when no `timestamp` filter is provided. Because the REST API has no rate-limiting middleware, any unauthenticated external user can flood this endpoint with concurrent requests, each triggering multiple correlated subqueries, exhausting database CPU and denying service.

### Finding Description
**Exact code path:**

In `rest/balances.js` lines 133–149, when `tsQuery` is falsy (no timestamp parameter supplied), the handler builds:

```js
const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order); // line 137
sqlQuery = `
  with entity_balance as (
    select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
    from entity
    where type in ('ACCOUNT', 'CONTRACT')
  )
  select ab.*, (${tokenBalanceSubQuery}) as token_balances  // correlated subquery in SELECT list
  from entity_balance ab
  ${whereClause}
  order by ab.account_id ${order}
  ${limitQuery}`;
```

`getTokenAccountBalanceSubQuery` (lines 308–318) returns:

```js
`select json_agg(json_build_object('token_id', token_id, 'balance', balance))
 from (
   select token_id, balance
   from token_account ta
   where ta.account_id = ab.account_id and ta.associated is true  // correlated on ab.account_id
   order by token_id ${order}
   limit ${tokenBalanceLimit.multipleAccounts}
 ) as account_token_balance`
```

This is a scalar correlated subquery in the SELECT list. PostgreSQL executes it once per row emitted by the outer query — up to `limit` times per request (default limit is configurable, up to the max, e.g. 25–1000).

**No rate limiting exists for the REST API.** Examining `rest/server.js` lines 68–144, the full middleware stack is: `urlencoded`, `json`, `cors`, optional `compression`, `httpContext`, `requestLogger`, `authHandler` (optional basic auth, not enforced), optional `metricsHandler`, optional `responseCacheCheckHandler`, then route handlers. There is no rate-limiting or connection-throttling middleware. The `ThrottleConfiguration` and `ThrottleManagerImpl` found in `web3/` apply exclusively to the Java web3 service.

**Root cause / failed assumption:** The design assumes that the `LIMIT` clause sufficiently bounds per-request cost. While it does bound the number of correlated subquery executions per request, it does not prevent an attacker from issuing unlimited concurrent requests, each independently consuming database CPU through repeated `token_account` index lookups.

### Impact Explanation
On a production Hedera network with millions of accounts and token associations, each unauthenticated request to `GET /api/v1/balances` (no timestamp) causes up to `limit` correlated subqueries against `token_account`. With thousands of concurrent requests and no server-side throttle, the database connection pool saturates and CPU is exhausted, causing all queries (including those from legitimate users and internal services) to queue or time out. This is a non-network-based DoS: no bandwidth amplification is needed, only HTTP request volume.

### Likelihood Explanation
The attack requires zero privileges — no API key, no authentication, no special knowledge beyond the public OpenAPI spec (`rest/api/v1/openapi.yml` lines 391–419 documents the endpoint and its parameters). The trigger condition (omitting `timestamp`) is the default usage pattern. The attack is trivially repeatable with any HTTP load-testing tool (e.g., `ab`, `wrk`, `hey`). The absence of any rate-limiting middleware in the REST service makes sustained exploitation straightforward.

### Recommendation
1. **Add rate limiting middleware** to the Node.js REST service (e.g., `express-rate-limit` or an upstream API gateway rule) applied globally before route handlers in `rest/server.js`.
2. **Rewrite the correlated subquery as a JOIN or lateral join** so the database can batch the `token_account` lookups rather than executing one per row:
   ```sql
   left join lateral (
     select json_agg(json_build_object('token_id', token_id, 'balance', balance)) as token_balances
     from (
       select token_id, balance from token_account ta
       where ta.account_id = ab.account_id and ta.associated is true
       order by token_id ${order} limit ${tokenBalanceLimit.multipleAccounts}
     ) t
   ) tb on true
   ```
3. **Enforce a hard cap on concurrent DB queries** via the connection pool configuration to prevent pool exhaustion under load.
