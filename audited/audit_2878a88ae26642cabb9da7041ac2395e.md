### Title
Unbounded Growth of `cacheManagerSlotsPerContract` Internal Cache Map via Dynamic Cache Creation Leads to Memory Leak

### Summary
The `cacheManagerSlotsPerContract` bean is a `CaffeineCacheManager` configured **without** `setCacheNames()`, enabling dynamic cache creation. Each distinct `contractId` passed to `findStorageBatch()` causes `cacheManagerSlotsPerContract.getCache(contractId.toString())` to register a new named `CaffeineCache` in the manager's internal `ConcurrentHashMap<String, Cache>`. This map is never pruned — even when the outer `contractSlotsCache` evicts entries — resulting in a persistent, unbounded memory leak that can exhaust heap on web3 service nodes.

### Finding Description

**Root cause — no fixed cache names on `cacheManagerSlotsPerContract`:**

In `EvmConfiguration.java` every other `CaffeineCacheManager` bean calls `setCacheNames(Set.of(...))`, which locks the manager to a fixed set of named caches. The `slotsPerContract` bean does not:

```java
// EvmConfiguration.java lines 107-112
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    // ← NO setCacheNames() call
    return caffeineCacheManager;
}
``` [1](#0-0) 

Compare with every other bean, e.g.:
```java
// EvmConfiguration.java line 78
cacheManager.setCacheNames(Set.of(CACHE_NAME));
``` [2](#0-1) 

Without `setCacheNames()`, Spring's `CaffeineCacheManager` operates in *dynamic mode*: any call to `getCache(name)` with a previously unseen name creates a new `CaffeineCache` and stores it permanently in the manager's internal `ConcurrentHashMap<String, Cache> cacheMap`.

**Trigger site — `findStorageBatch()`:**

```java
// ContractStateServiceImpl.java lines 85-87
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [3](#0-2) 

The loader `() -> cacheManagerSlotsPerContract.getCache(contractId.toString())` is invoked whenever `contractId` is absent from the outer `contractSlotsCache`. Each invocation registers a new named cache in `cacheManagerSlotsPerContract.cacheMap` for that `contractId` string.

**Why the outer cache does not protect the inner map:**

`contractSlotsCache` (CACHE_MANAGER_CONTRACT_SLOTS) has `maximumSize=3000, expireAfterAccess=5m`: [4](#0-3) 

When an entry is evicted from `contractSlotsCache`, the eviction removes the `CaffeineCache` *value* from the outer cache, but the named cache entry in `cacheManagerSlotsPerContract.cacheMap` is **never removed**. Spring's `CaffeineCacheManager` has no eviction listener or cleanup hook tied to the outer cache's eviction events. The `cacheMap` is a plain `ConcurrentHashMap` that only grows.

**Exploit flow:**

1. Attacker sends `eth_call` / `eth_estimateGas` requests targeting many distinct deployed contracts that contain `SLOAD` opcodes.
2. Each call causes the EVM to invoke `findStorage(contractId, key)`.
3. On cache miss in `contractStateCache`, `findStorageBatch()` is called.
4. On cache miss in `contractSlotsCache`, the loader fires: `cacheManagerSlotsPerContract.getCache(contractId.toString())` → new named cache registered in `cacheMap`.
5. After 3000 distinct contractIds, `contractSlotsCache` starts evicting old entries. Each eviction causes the *next* request for that contractId to re-invoke the loader — but since the name already exists in `cacheMap`, no new entry is created. However, for each *new* contractId beyond 3000, a new entry is permanently added to `cacheMap`.
6. `cacheMap` grows monotonically: one entry per distinct contractId ever seen, with no upper bound and no cleanup.

**Existing checks and why they fail:**

- `contractSlotsCache` `maximumSize=3000`: limits active per-contract caches in the outer cache, but does **not** bound `cacheManagerSlotsPerContract.cacheMap`.
- `slotsPerContract` spec `maximumSize=1500`: limits entries *within* each per-contract cache, not the number of named caches in the manager.
- `expireAfterAccess=5m` on `slotsPerContract`: expires entries inside each per-contract Caffeine cache, but the `CaffeineCache` wrapper object and its internal Caffeine data structures remain allocated in `cacheMap`. [5](#0-4) 

### Impact Explanation
Each `CaffeineCache` wrapping a Caffeine `BoundedLocalCache` with `maximumSize=1500` allocates internal data structures (ring buffers, eviction policy arrays, segment locks) on the order of tens of kilobytes even when empty. With tens of thousands of distinct contractIds accumulated over time, `cacheManagerSlotsPerContract.cacheMap` can consume gigabytes of heap. This leads to `OutOfMemoryError` on the web3 service JVM, crashing the mirror node's JSON-RPC layer. If multiple mirror node instances share the same traffic pattern (which they do in a load-balanced deployment), all instances are equally affected, potentially taking down 30%+ of the web3 processing capacity.

### Likelihood Explanation
No authentication is required for `eth_call` or `eth_estimateGas`. The global rate limit is 500 req/s. An attacker with access to a list of deployed contracts (freely available via the mirror node REST API `/api/v1/contracts`) can iterate through them, sending one `eth_call` per contract. On Hedera mainnet there are thousands of deployed contracts with storage reads. The attack is repeatable, requires no special privileges, and accumulates damage persistently across restarts (the leak resets on restart but resumes immediately on traffic). A coordinated multi-IP campaign can exceed the global rate limit.

### Recommendation
1. **Fix the root cause**: Call `setCacheNames()` on `cacheManagerSlotsPerContract` with a fixed set of names, or redesign so that per-contract caches are not registered as named caches in a shared `CaffeineCacheManager`. Instead, store the per-contract `Cache` objects directly in the outer `contractSlotsCache` value without registering them in any `CacheManager`.
2. **Add an eviction listener** to `contractSlotsCache` that explicitly removes the corresponding named cache from `cacheManagerSlotsPerContract` when an entry is evicted.
3. **Bound the manager**: If dynamic creation is kept, wrap `CaffeineCacheManager` with a size-bounded map that evicts the least-recently-used named cache when the limit is reached.
4. **Monitor**: Add a metric or health check that alerts when `cacheManagerSlotsPerContract.getCacheNames().size()` exceeds a threshold.

### Proof of Concept
```
# 1. Fetch list of deployed contract IDs from mirror node REST API
curl "https://<mirror-node>/api/v1/contracts?limit=100&order=asc" | jq '.contracts[].contract_id'

# 2. For each contractId (0.0.X), send an eth_call targeting that contract
#    The call triggers SLOAD → findStorage → findStorageBatch → cacheManagerSlotsPerContract.getCache()
for CONTRACT_ADDR in $(cat contract_addresses.txt); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"data\":\"0x\",\"to\":\"$CONTRACT_ADDR\",\"gas\":100000}" &
done

# 3. Monitor JVM heap on the web3 service node:
#    jmap -histo <pid> | grep CaffeineCache
#    Observe count growing proportionally to distinct contractIds called.

# 4. After exhausting known contracts, repeat with incremented numeric IDs
#    (0.0.1000, 0.0.1001, ...) — non-existent contracts return empty without
#    executing bytecode, so only existing contracts with SLOAD trigger the leak.
#    On a network with 50,000+ deployed contracts, the cacheMap reaches
#    50,000 entries × ~50 KB/entry ≈ 2.5 GB heap consumed by the map alone.
```
