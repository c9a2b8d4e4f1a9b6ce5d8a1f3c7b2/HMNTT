### Title
Unauthenticated Thread Pool Exhaustion via Unbounded Blocking on `NetworkStakeRepository.findLatest()` Leading to Service DoS

### Summary
The public `GET /api/v1/network/stake` endpoint invokes `networkStakeRepository.findLatest()` on every request with no caching, no application-level circuit breaker, and no bulkhead. Under database slowness or a concurrent request flood, all Tomcat servlet threads block waiting for either a HikariCP connection or the 20-second PostgreSQL `statement_timeout` to expire, exhausting the thread pool, starving the Kubernetes liveness probe of threads, and triggering pod restarts that further disrupt availability.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (line 127–130) calls `networkService.getLatestNetworkStake()` with no guard: [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56) calls the repository directly — no `@Cacheable`, no circuit breaker, no timeout override: [2](#0-1) 

`NetworkStakeRepository.findLatest()` issues a native correlated-subquery SQL on every invocation: [3](#0-2) 

**Root cause — three compounding absences:**

1. **No caching.** A search across all `rest-java` source files finds zero `@Cacheable`, `resilience4j`, `CircuitBreaker`, `Bulkhead`, or `RateLimiter` annotations. Every HTTP request issues a live DB query.

2. **PostgreSQL statement timeout is 20 s, not 10 s.** The Helm secret template sets `statement_timeout` to `20000` ms for `mirror_rest_java`: [4](#0-3) 
Each blocked thread holds a Tomcat worker for up to 20 seconds.

3. **HikariCP default pool (10 connections) amplifies the blockage.** Threads that cannot acquire a connection within HikariCP's default 30-second `connectionTimeout` also block, meaning up to 200 Tomcat threads (Spring Boot default) can pile up waiting for either a connection or a query result.

**Why the Traefik circuit breaker does not help:**

A Traefik middleware circuit breaker is configured at the ingress level: [5](#0-4) 

This trips only after `NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25` — i.e., after significant damage is already done. It operates outside the pod and does not prevent internal thread pool saturation or liveness probe failure.

**Liveness probe interaction:**

The liveness probe hits `/actuator/health/liveness` every 10 seconds with a 2-second timeout: [6](#0-5) 

Spring Boot's actuator shares the same Tomcat thread pool. When all threads are blocked on DB calls, the probe HTTP request cannot be served within 2 seconds → Kubernetes marks the pod unhealthy → pod restart → service disruption repeats.

### Impact Explanation

Complete unavailability of the `rest-java` service for all endpoints routed through it (`/api/v1/network/*`, `/api/v1/accounts/*/allowances/nfts`, `/api/v1/topics/*`, etc.). Pod restarts under continuous attack create a crash-loop pattern. Because `minReplicas: 1` is the default and `podDisruptionBudget.enabled: false`, a single-pod deployment has no redundancy to absorb the attack. [7](#0-6) 

### Likelihood Explanation

The endpoint requires **zero authentication** — it is a plain `GET /api/v1/network/stake` with no API key, token, or IP restriction. Any internet-accessible attacker can send ~200 concurrent HTTP requests using standard tools (`curl`, `ab`, `wrk`). The attack is trivially repeatable: after each pod restart the attacker simply resumes flooding. No exploit code, no credentials, and no knowledge of internal state are required.

### Recommendation

1. **Add response caching** on `getLatestNetworkStake()` with a short TTL (e.g., 10–30 s) using `@Cacheable` — network stake data changes only once per staking period (~24 h).
2. **Add a Resilience4j `Bulkhead`** (thread-pool or semaphore) around the `findLatest()` call to cap concurrent in-flight DB queries and return `503` immediately when the limit is reached.
3. **Add a Resilience4j `CircuitBreaker`** that opens after repeated DB timeouts, returning a cached or degraded response instead of blocking.
4. **Configure an explicit HikariCP `connectionTimeout`** shorter than the statement timeout (e.g., 2–3 s) so threads fail fast rather than waiting 30 s for a pool slot.
5. **Enable rate limiting** at the Traefik ingress level (e.g., `rateLimit` middleware) for the `/api/v1/network/stake` path.

### Proof of Concept

**Preconditions:** The `rest-java` service is reachable. No credentials needed.

**Steps:**

```bash
# 1. Simulate DB slowness (or rely on natural contention under load)
# 2. Flood the endpoint with 250 concurrent requests (exceeds default 200-thread pool)
wrk -t25 -c250 -d60s https://<mirror-node-host>/api/v1/network/stake

# 3. Observe: after ~10-20s, liveness probe begins failing
kubectl get events -n <namespace> --field-selector reason=Unhealthy

# 4. Observe pod restart
kubectl get pods -n <namespace> -w

# 5. Resume flood immediately after restart to maintain crash-loop
```

**Expected result:** Pod enters a restart loop. All `/api/v1/network/*` and other rest-java-served endpoints return connection refused or 503 during restarts. The Traefik circuit breaker may eventually open (after >25% 5xx ratio), but the pod-level damage and restart cycle persist.
