### Title
`RestSubscriber.shouldRetry()` Only Retries on HTTP 404, Silently Dropping All Other Transient Errors and Permanently Failing Smart Contract Transaction Monitoring

### Summary
The `shouldRetry()` method in `RestSubscriber` only permits retries for HTTP 404 responses. Any other transient HTTP error (e.g., 429 Too Many Requests, 503 Service Unavailable) causes the retry spec to immediately exhaust, and the subsequent `onErrorResume(e -> Mono.empty())` silently swallows the error. An unprivileged external attacker who can cause the mirror node REST API to return non-404 HTTP errors (e.g., by flooding the API to trigger rate limiting) can permanently drop smart contract transaction monitoring for targeted transactions without any retry attempt.

### Finding Description
**Exact code path:**

- `RestSubscriber.java`, lines 144–147: `shouldRetry()` returns `true` only for `HttpStatus.NOT_FOUND` (404).
- `RestSubscriber.java`, line 95: the retry spec uses `.filter(this::shouldRetry)`, meaning any non-404 error is treated as non-retryable.
- `RestSubscriber.java`, lines 113–115: `.retryWhen(retrySpec)` is followed by `.doOnError(t -> subscription.onError(t))` (logs a warning) and `.onErrorResume(e -> Mono.empty())` (silently swallows the error).

**Root cause:** The `shouldRetry()` predicate has an overly narrow condition. HTTP 429 (rate-limited), 503 (service unavailable), 500 (internal server error), and connection-level errors are all transient and retryable in practice, but the filter rejects them all. The failed assumption is that only 404 is a transient condition worth retrying.

**Exploit flow:**
1. Attacker submits malicious smart contract transactions to the Hiero network.
2. Attacker simultaneously floods the mirror node REST API with high-volume requests, exhausting any rate limit and causing the API to return HTTP 429 responses to legitimate callers.
3. The monitor's `clientSubscribe()` calls `restApiClient.retrieve(...)` for the malicious transactions and receives 429 responses.
4. `shouldRetry(t)` evaluates `webClientResponseException.getStatusCode() == HttpStatus.NOT_FOUND` → `false` for 429.
5. The retry spec does not retry; `onErrorResume(e -> Mono.empty())` silently drops the error.
6. Monitoring of the malicious transactions is permanently lost for that polling cycle.

**Why existing checks are insufficient:**
- `doOnError(t -> subscription.onError(t))` only logs a warning and increments an error counter; it does not trigger any retry or compensating action.
- `onErrorResume(e -> Mono.empty())` converts the error to an empty signal, meaning the `flatMap` simply produces no output for that transaction — no alert, no retry, no dead-letter queue.
- `RestSubscription.onError()` records the HTTP status code string in an error set but does not re-queue the transaction for monitoring.

### Impact Explanation
Smart contract transaction monitoring is silently dropped for any transaction whose REST API lookup encounters a non-404 HTTP error. An attacker can use this to suppress monitoring of their own malicious smart contract activity (e.g., reentrancy exploits, unauthorized token mints) by ensuring the mirror node REST API returns 429 or 503 during the monitoring window. While no funds are directly at risk from the monitor itself, the loss of observability undermines the integrity of the monitoring layer and can delay or prevent detection of on-chain attacks. Severity is Medium per the stated classification (layer 0/1/2 network monitoring code, no direct fund loss).

### Likelihood Explanation
The mirror node REST API is a public-facing service. Any unprivileged external user can send requests to it. Triggering rate limiting (HTTP 429) requires only a moderate volume of requests — well within reach of a single attacker using commodity tooling. The attack is repeatable: the attacker can sustain the flood for the duration of the monitoring window (bounded by `properties.getTimeout()` = 5 seconds default), ensuring the monitor's retry window is never reached. No authentication or special privilege is required.

### Recommendation
Expand `shouldRetry()` to include all transient HTTP status codes and connection-level errors:

```java
protected boolean shouldRetry(Throwable t) {
    if (t instanceof WebClientResponseException ex) {
        HttpStatusCode status = ex.getStatusCode();
        return status == HttpStatus.NOT_FOUND          // 404: transaction not yet indexed
            || status == HttpStatus.TOO_MANY_REQUESTS  // 429: rate limited
            || status == HttpStatus.SERVICE_UNAVAILABLE // 503: transient outage
            || status.is5xxServerError();              // 5xx: general server errors
    }
    // Retry on connection-level errors (timeout, reset, etc.)
    return t instanceof java.io.IOException
        || t instanceof reactor.core.Exceptions.RetryExhaustedException;
}
```

Additionally, replace `onErrorResume(e -> Mono.empty())` with explicit error handling that records the dropped transaction for alerting or re-queuing, rather than silently discarding it.

### Proof of Concept
1. Deploy the Hiero mirror node monitor against a mirror node REST API endpoint.
2. Configure a REST subscriber to monitor smart contract transactions (`ContractCall` or `ContractCreate`).
3. Submit a smart contract transaction to the network; the monitor's `onPublish()` queues it for REST polling.
4. Before the monitor's `clientSubscribe()` polls `/transactions/{transactionId}`, flood the mirror node REST API with requests from a separate client to trigger HTTP 429 responses.
5. Observe that the monitor's `shouldRetry()` returns `false` for the 429 response.
6. Observe that `onErrorResume(e -> Mono.empty())` silently drops the transaction.
7. Confirm in the monitor's metrics/logs that the transaction was never successfully monitored and no retry was attempted — only a single warning log entry from `subscription.onError(t)`. [1](#0-0) [2](#0-1) [3](#0-2)
