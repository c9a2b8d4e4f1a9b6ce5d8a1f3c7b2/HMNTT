### Title
Unbounded Indefinite Subscriptions via Missing Server-Side Timeout in `subscribeTopic()` Leading to Memory Exhaustion DoS

### Summary
When a client subscribes with no `endTime` and no `limit`, `pastEndTime()` returns `Flux.never()`, causing the subscription pipeline to run indefinitely with no server-side termination. There is no global cap on total concurrent subscriptions and no per-IP connection limit, so an unauthenticated attacker opening many TCP connections (each carrying up to 5 calls) can accumulate arbitrarily many live subscription pipelines until the server exhausts memory or thread/file-descriptor resources.

### Finding Description

**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()` (line 59) builds the flux pipeline at line 73:

```java
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
```

`pastEndTime()` (lines 123–131) returns `Flux.never()` when `endTime` is null:

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();          // <-- subscription never terminates from this path
    }
    ...
}
```

`takeUntilOther(Flux.never())` means the live merge segment never receives a termination signal from the end-time guard. The `limit` branch (lines 83–85) is also skipped when `limit == 0` (`hasLimit()` returns false). The `subscriberCount` at line 48 is a **metric gauge only** — it is never checked against any ceiling before accepting a new subscription.

**No global connection cap:** `GrpcConfiguration.java` (lines 31–34) configures only `maxConcurrentCallsPerConnection = 5` (default). No `maxConnections`, `maxConnectionAge`, `maxConnectionIdle`, or per-IP limit is set. An attacker opens N TCP connections × 5 calls each = 5N indefinitely-live subscription pipelines.

**No authentication:** `ConsensusController.subscribeTopic()` (line 43) requires no credentials; any network-reachable client can call it.

**Per-subscription resource cost:** Each open subscription retains a `TopicContext` object, a multi-stage Reactor `Flux` pipeline, a `topicListener.listen()` subscription (which may hold a Redis pub/sub channel or a shared-poll slot), and a `Schedulers.boundedElastic()` task for the safety-check path.

### Impact Explanation
An attacker can exhaust JVM heap by accumulating tens of thousands of live subscription pipelines, causing OOM and crashing the gRPC service. Even before OOM, the bounded-elastic scheduler thread pool and Redis/DB listener connections can be saturated, degrading or denying service to all legitimate subscribers. Because the mirror node gRPC endpoint is a public, unauthenticated API, this affects all consumers of the HCS topic subscription service.

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK), no credentials, and a single machine with enough sockets. Opening 10,000 TCP connections each with 5 calls is trivially achievable with async gRPC clients. The attack is repeatable and persistent — subscriptions remain open until the server crashes or the attacker disconnects.

### Recommendation
1. **Enforce a maximum subscription duration:** Add a `maxSubscriptionDuration` property to `GrpcProperties` and apply `.timeout(maxSubscriptionDuration)` to the flux in `subscribeTopic()`, regardless of whether `endTime` is set.
2. **Cap total concurrent subscriptions:** Check `subscriberCount` against a configurable ceiling before accepting a new subscription; return `RESOURCE_EXHAUSTED` if exceeded.
3. **Cap connections globally:** In `GrpcConfiguration`, call `serverBuilder.maxConnectionAge(...)` and `serverBuilder.maxConnectionIdle(...)` on the `NettyServerBuilder` to bound connection lifetime and idle connections.
4. **Rate-limit new subscription requests per IP** at the load-balancer or via a gRPC interceptor.

### Proof of Concept
```bash
# Open 2000 TCP connections, each with 5 indefinite subscriptions (10,000 total)
# using grpcurl or a simple async gRPC client loop:

for i in $(seq 1 2000); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Each call omits consensusEndTime and limit=0, triggering Flux.never() in pastEndTime().
# Monitor server heap: jcmd <pid> VM.native_memory or watch /proc/<pid>/status VmRSS
# Server OOM or severe degradation observed as subscription count grows unbounded.
```
