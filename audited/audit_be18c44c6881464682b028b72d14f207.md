### Title
Unbounded Protobuf Request Body in `estimateFees()` Enables Unauthenticated DoS

### Summary
The `estimateFees()` endpoint in `NetworkController.java` accepts a raw `byte[]` request body with no size constraint, allowing any unauthenticated caller to POST an arbitrarily large protobuf payload. Spring Boot's embedded Tomcat buffers the entire body into a heap-allocated `byte[]` before the controller is invoked, and the subsequent `Transaction.PROTOBUF.parse(Bytes.wrap(body))` call may allocate additional heap memory during parsing. Repeated large requests can exhaust JVM heap, causing OOM errors or severe GC pressure that prevents legitimate fee queries from being served.

### Finding Description

**Exact code location:**
`rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 110–124:

```java
@PostMapping(
        consumes = {"application/protobuf", "application/x-protobuf"},
        value = "/fees")
FeeEstimateResponse estimateFees(
        @RequestBody @NotNull byte[] body,          // ← no @Size or length cap
        ...
) {
    try {
        final var transaction = Transaction.PROTOBUF.parse(Bytes.wrap(body));  // ← unbounded parse
```

**Root cause:** The `@RequestBody byte[] body` parameter carries only `@NotNull`; there is no `@Size(max=…)` or equivalent constraint. Spring MVC resolves `byte[]` bodies via `ByteArrayHttpMessageConverter`, which reads the entire stream into a heap array before the controller method is called. This path is completely independent of the Jackson `StreamReadConstraints` configured in `RestJavaConfiguration.java` (lines 20–27), which only limits JSON parsing.

**Why existing checks fail:**

1. `@NotNull` — only rejects a null body, not a large one. [1](#0-0) 

2. `JacksonConfiguration` (`maxDocumentLength(1000)`, `maxStringLength(1000)`) — applies exclusively to JSON deserialization via Jackson; the protobuf binary path never touches Jackson. [2](#0-1) 

3. No `server.tomcat.max-swallow-size`, `server.tomcat.max-http-form-post-size`, or any custom body-size filter was found in the rest-java module. Tomcat's default `maxPostSize` (2 MB) applies only to `application/x-www-form-urlencoded` content, not to raw binary `application/protobuf` bodies. [3](#0-2) 

4. `MetricsFilter` records `Content-Length` for observability but enforces nothing. [4](#0-3) 

**Exploit flow:**
- Attacker sends `POST /api/v1/network/fees` with `Content-Type: application/protobuf` and a body of, e.g., 200 MB of crafted/random bytes.
- Tomcat reads the full stream into a `byte[]` on the heap.
- `Transaction.PROTOBUF.parse(Bytes.wrap(body))` attempts to parse the blob, potentially allocating further heap objects for nested protobuf fields.
- With a few concurrent requests the pod's 2 GiB memory limit is breached, triggering OOM or prolonged GC pauses.

### Impact Explanation
The Kubernetes pod for `rest-java` is configured with a 2048 MiB memory limit. [5](#0-4)  A single 500 MB request body, or a handful of 100 MB concurrent requests, can exhaust available heap. This causes the JVM to throw `OutOfMemoryError` or enter continuous GC, making the `/api/v1/network/fees` (GET and POST) and all other endpoints on the same pod unavailable — effectively erasing accessible fee history for the duration of the outage.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication requirement. No rate-limiting or body-size enforcement exists at the application layer. The attack requires only a single HTTP client capable of streaming a large body; it is trivially scriptable, repeatable, and requires zero protocol knowledge beyond knowing the endpoint accepts `application/protobuf`.

### Recommendation
1. **Enforce a body size limit at the controller level** — add a `@Size(max = MAX_TX_BYTES)` or a custom `@MaxSize` constraint on the `byte[] body` parameter, where `MAX_TX_BYTES` reflects the Hedera network's maximum transaction size (6 KB).
2. **Configure Tomcat's connector limit** — set `server.tomcat.max-swallow-size` and a custom `server.tomcat.connection-timeout` in `application.yml` to reject oversized bodies before they reach the servlet layer.
3. **Add a servlet filter** — register a `ContentSizeLimitingFilter` (or equivalent) that reads `Content-Length` and immediately returns HTTP 413 for bodies exceeding the threshold, before any body bytes are buffered.

### Proof of Concept
```bash
# Generate a 50 MB payload of random bytes
dd if=/dev/urandom bs=1M count=50 of=/tmp/large_payload.bin

# Send it to the estimateFees endpoint (repeat concurrently for faster OOM)
for i in $(seq 1 10); do
  curl -s -o /dev/null \
    -X POST "https://<mirror-node-host>/api/v1/network/fees" \
    -H "Content-Type: application/protobuf" \
    --data-binary @/tmp/large_payload.bin &
done
wait
```
Expected result: JVM heap exhaustion on the `rest-java` pod, causing HTTP 500 or connection refusals for all subsequent requests including legitimate GET `/api/v1/network/fees` calls.
