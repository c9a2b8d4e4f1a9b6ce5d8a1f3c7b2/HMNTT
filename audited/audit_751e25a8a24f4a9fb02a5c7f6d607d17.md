### Title
Unbounded Connection Accumulation in `grpcServerConfigurer()` Enables Indefinite Griefing via Thread Pool Exhaustion

### Summary
`grpcServerConfigurer()` configures only `maxConcurrentCallsPerConnection` (5 calls per connection) but omits any limit on the total number of accepted connections, connection lifetime (`maxConnectionAge`), or idle connection cleanup (`maxConnectionIdle`). An unprivileged attacker can open an unbounded number of TCP connections, each holding up to 5 long-lived streaming calls, exhausting the shared `applicationTaskExecutor` thread pool and preventing the server from self-recovering as long as the attacker maintains the connections.

### Finding Description
**Code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // = 5
};
```

`NettyProperties` exposes only `maxConcurrentCallsPerConnection = 5` with no `maxConnectionAge`, `maxConnectionIdle`, or total connection cap configured anywhere in the gRPC module.

**Root cause / failed assumption:** The configuration assumes that limiting concurrent calls *per connection* to 5 bounds total server load. It does not. An attacker opens N connections; each holds 5 streaming `subscribeTopic` calls. Total concurrent calls = 5 × N, bounded only by the attacker's socket budget. Because no `maxConnectionAge` or `maxConnectionIdle` is set on the `NettyServerBuilder`, these connections and their associated streaming calls persist indefinitely — the server has no mechanism to reclaim resources without operator intervention.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` (NettyProperties line 14) limits calls per connection, not the number of connections.
- The GCP backend policy `maxRatePerEndpoint: 250` (charts/hedera-mirror-grpc/values.yaml line 69) throttles the *rate* of new requests but does not cap the count of already-established, long-lived HTTP/2 connections or their streaming calls.
- The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting exists only in the `web3` module; there is no equivalent in the `grpc` module.
- HPA is disabled by default (`enabled: false`, charts/hedera-mirror-grpc/values.yaml line 113), so horizontal scaling does not auto-mitigate.

### Impact Explanation
The `applicationTaskExecutor` thread pool is shared across all gRPC calls. With enough attacker-controlled connections (each holding 5 streaming calls), the pool is saturated. Legitimate `subscribeTopic` and `getNodes` calls queue indefinitely or are rejected. The server remains in a degraded state for as long as the attacker holds connections open — which is indefinitely given the absence of `maxConnectionAge`. No economic damage occurs to the network; the impact is pure availability griefing for all users of the mirror node gRPC API.

### Likelihood Explanation
The attack requires only a standard gRPC client library and the ability to open TCP connections to port 5600 (publicly exposed). No authentication, special privileges, or cryptographic material are needed. The attack is trivially scriptable, repeatable, and can be sustained from a single host with modest resources (thousands of open sockets are well within OS limits). The absence of per-IP connection rate limiting at the application layer means the attacker does not need to distribute the attack across multiple IPs.

### Recommendation
Add the following calls inside `grpcServerConfigurer()` on the `NettyServerBuilder`:

```java
serverBuilder.maxConnectionAge(30, TimeUnit.MINUTES);          // force periodic connection recycling
serverBuilder.maxConnectionAgeGrace(5, TimeUnit.SECONDS);      // grace period for in-flight calls
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);          // evict idle connections
```

Expose these as configurable properties in `NettyProperties` (alongside `maxConcurrentCallsPerConnection`). Additionally, consider adding a gRPC server interceptor that tracks active connections per remote IP and rejects new connections above a per-IP threshold, and enable the HPA by default so the cluster can scale under legitimate load spikes.

### Proof of Concept
```python
import grpc
import threading
from concurrent import futures

# Target: grpc server on port 5600
TARGET = "mirror-node-grpc:5600"
NUM_CONNECTIONS = 500   # each holds 5 streaming calls = 2500 total concurrent calls

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = ConsensusServiceStub(channel)   # com.hedera.mirror.api.proto.ConsensusService
    # Open 5 streaming calls per connection (maxConcurrentCallsPerConnection limit)
    streams = []
    for _ in range(5):
        req = ConsensusTopicQuery(topic_id=TopicID(topic_num=1))
        streams.append(stub.subscribeTopic(req))  # long-lived server-streaming RPC
    # Hold connections open indefinitely — no server-side timeout will close them
    threading.Event().wait()

with futures.ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as ex:
    list(ex.map(flood_connection, range(NUM_CONNECTIONS)))

# Result: 2500 concurrent streaming calls saturate applicationTaskExecutor;
# legitimate subscribeTopic calls from real users time out or are queued indefinitely.
# Server does not self-recover because no maxConnectionAge/maxConnectionIdle is configured.
```
