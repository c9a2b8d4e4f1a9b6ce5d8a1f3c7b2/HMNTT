### Title
Global Shared `opcodeRateLimitBucket` Enables Unauthenticated Single-User DoS on Opcodes Endpoint

### Summary
The `opcodeRateLimitBucket` is a single application-wide token bucket with a default capacity of 1 token per second, shared across all clients with no per-IP or per-user isolation. Any unauthenticated external user can continuously exhaust this single token, causing all other users' opcode replay requests to be rejected with HTTP 429 for the entire 1-second refill window. Repeating this at ≥1 req/s sustains a complete denial-of-service against the opcodes endpoint indefinitely.

### Finding Description
**Code path:**

- `ThrottleProperties.java` line 29: `private long opcodeRequestsPerSecond = 1;` — default capacity is 1.
- `ThrottleConfiguration.java` lines 47–55: A single `Bucket` bean named `opcodeRateLimitBucket` is created with `capacity(rateLimit)` / `refillGreedy(rateLimit, Duration.ofSeconds(1))` — this is one global JVM-singleton bucket, not per-client.
- `ThrottleManagerImpl.java` lines 52–56: `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` — no IP check, no session check, no authentication.
- `OpcodesController.java` line 61: `throttleManager.throttleOpcodeRequest()` is called before any processing for every request to `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes`.

**Root cause:** The rate-limit bucket is a global singleton. The failed assumption is that 1 req/s is a protective limit; in reality it is a shared resource that any single caller can monopolize. There is no per-source-IP bucket, no authentication gate, and no back-pressure mechanism. One attacker consuming the single token every second leaves zero capacity for all legitimate callers.

**Why existing checks fail:** The only guard is `tryConsume(1)` on the global bucket. There is no IP-based sub-bucket, no API key requirement, no CAPTCHA, and no connection-level throttle upstream of this code path.

### Impact Explanation
The `/opcodes` endpoint re-executes EVM transactions to produce full opcode traces (stack, memory, storage). With the global bucket exhausted, every legitimate opcode-trace request returns HTTP 429 for the duration of the attack. Because the bucket refills to exactly 1 token per second, an attacker sending ≥1 request/second sustains 100% denial of the endpoint. Operators and developers relying on opcode traces for debugging, auditing, or incident response are completely blocked. The documentation explicitly notes this endpoint is intentionally rate-limited to 1 RPS because it is "heavy" — making the global-bucket design especially impactful.

### Likelihood Explanation
Preconditions: none. The endpoint requires only a valid `Accept-Encoding: gzip` header and any transaction ID or hash (publicly observable on-chain). No account, API key, or privilege is needed. The attack is a simple HTTP GET loop executable with `curl` or any scripting tool. It is trivially repeatable and requires no special knowledge of the system internals.

### Recommendation
Replace the single global bucket with per-client rate limiting:
1. Introduce a `LoadingCache<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` or `HttpServletRequest.getRemoteAddr()`), creating a per-IP bucket with the configured `opcodeRequestsPerSecond` capacity.
2. In `throttleOpcodeRequest()`, accept the client IP as a parameter (or resolve it from the request context) and call `tryConsume(1)` on the per-IP bucket.
3. Optionally add an upstream network-level rate limiter (e.g., nginx `limit_req`) as a defense-in-depth layer.
4. Consider requiring an API key or authentication token for this endpoint given its computational cost.

### Proof of Concept
```bash
# Terminal 1 – attacker: exhaust the global 1-token bucket continuously
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/0xABCD.../opcodes"
done

# Terminal 2 – victim: every request is rejected with 429
curl -v -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/0x1234.../opcodes"
# Response: HTTP/1.1 429 Too Many Requests
# {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```
The attacker's first request in each second succeeds (HTTP 200); all concurrent or subsequent requests from any other client within that second receive HTTP 429. Sustained at ≥1 req/s, the endpoint is permanently unavailable to all other users.
