### Title
Unauthenticated `/balances` Endpoint Exhausts Default 10-Connection DB Pool via Distributed Requests, Causing Full REST API Outage

### Summary
The `getBalances()` handler in `rest/balances.js` is publicly accessible without authentication and issues up to three sequential database queries per request against a connection pool that defaults to only 10 connections. The Traefik middleware layer — which is the only place rate/in-flight limiting could be enforced — is disabled by default (`global.middleware: false`) and, even when enabled, the REST API chart defines no `inFlightReq` or `rateLimit` middleware. A distributed attacker using many source IPs can saturate the pool, causing every subsequent REST API query to queue or time out and producing a full service outage.

### Finding Description

**Code path — DB pool size:**
`rest/dbpool.js` line 14 sets `max: config.db.pool.maxConnections`, and the documented default is **10**:

> `hiero.mirror.rest.db.pool.maxConnections | 10 | The maximum number of clients the database pool can contain` [1](#0-0) [2](#0-1) 

**Code path — multiple queries per request in `getBalances()`:**
When a `timestamp` filter is supplied, `getBalances()` calls three separate `pool.queryQuietly()` invocations sequentially:
1. `getAccountBalanceTimestampRange()` → `pool.queryQuietly()` (line 189)
2. `getAccountIdsByPublicKey()` → `pool.queryQuietly()` (line 66)
3. Main balance query → `pool.queryQuietly()` (line 153) [3](#0-2) [4](#0-3) [5](#0-4) 

**Root cause — middleware disabled by default and missing controls:**
`charts/hedera-mirror-rest/values.yaml` sets `global.middleware: false`, meaning the Traefik middleware chain is never rendered. Even when enabled, the REST API middleware block contains only `circuitBreaker` and `retry` (10 attempts) — **no `inFlightReq` and no `rateLimit`**: [6](#0-5) [7](#0-6) 

Compare with the Rosetta chart, which explicitly adds both `inFlightReq: amount: 5` per IP and `rateLimit: average: 10` per host: [8](#0-7) 

The middleware template itself is guarded by `{{ if and .Values.global.middleware .Values.middleware }}`, so with the default `false` flag nothing is applied: [9](#0-8) 

**Amplification via retry middleware:**
The REST API middleware configures `retry: attempts: 10`. When the pool is exhausted and requests begin timing out (20 s `statementTimeout`), Traefik retries each failing request up to 10 times, multiplying effective load by up to 10×. [10](#0-9) 

### Impact Explanation
With a 10-connection pool and up to 3 sequential queries per `/balances` request, as few as 4 concurrent long-running requests can hold all connections. Once the pool is saturated, every other REST API endpoint (`/transactions`, `/accounts`, etc.) also queues or times out because they share the same `global.pool`. The mirror node REST API becomes completely unresponsive, preventing clients and downstream services from querying network state. This matches the stated critical scope: network inability to confirm new transactions due to mirror node outage. [11](#0-10) 

### Likelihood Explanation
No special privileges, credentials, or prior knowledge are required — the endpoint is documented as public. A distributed attack across many IPs bypasses any hypothetical per-IP controls. The attacker only needs to send concurrent `GET /api/v1/balances?timestamp=lte:<value>` requests with a valid timestamp to trigger all three DB queries per request. This is trivially scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, `hey`). The attack is repeatable and stateless.

### Recommendation
1. **Enable `inFlightReq` and `rateLimit` middleware for the REST API chart**, mirroring the Rosetta chart's configuration. Set `global.middleware: true` in production and add `inFlightReq` (e.g., `amount: 10`, `sourceCriterion.ipStrategy.depth: 1`) and `rateLimit` to `charts/hedera-mirror-rest/values.yaml`.
2. **Increase the default DB pool size** (`hiero.mirror.rest.db.pool.maxConnections`) to a value commensurate with expected concurrency, or enforce a hard cap on concurrent in-flight requests at the application layer before any DB connection is acquired.
3. **Reduce the retry count** from 10 to 2–3 to prevent retry amplification under pool exhaustion.
4. **Add application-level concurrency limiting** (e.g., a semaphore before `pool.queryQuietly()` calls in `getBalances()`) so pool exhaustion cannot be triggered regardless of infrastructure configuration.

### Proof of Concept

**Preconditions:** Mirror node REST API reachable at `$HOST`; default deployment (middleware disabled, pool size 10).

```bash
# Step 1: Send 15 concurrent requests each triggering 3 DB queries
# (timestamp filter forces all three pool.queryQuietly() calls)
for i in $(seq 1 15); do
  curl -s "$HOST/api/v1/balances?timestamp=lte:9999999999.000000000&limit=100" &
done
wait

# Step 2: Immediately probe another endpoint sharing the same pool
curl -v "$HOST/api/v1/transactions?limit=1"
# Expected result: connection timeout or 503/504 — pool exhausted
```

**Expected result:** After step 1 saturates all 10 pool connections (each held for up to 20 s by the statement timeout), step 2 and all other REST API calls queue indefinitely or return errors, constituting a full service outage. The Traefik retry middleware (10 attempts) will further amplify the load from any requests that do time out.

const getAccountBalanceTimestampRange = async (tsQuery, tsParams) => {
  const {lowerBound, upperBound, neParams} = getOptimizedTimestampRange(tsQuery, tsParams);
  if (lowerBound === undefined) {
    return {};
  }

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first
  // column of the primary key
  let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
  const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
  if (neParams.length) {
    condition += ' and not consensus_timestamp = any ($4)';
    params.push(neParams);
  }

  const query = `
    select consensus_timestamp
    from account_balance
    where ${condition}
    order by consensus_timestamp desc
    limit 1`;

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    return {};
  }

  const upper = rows[0].consensus_timestamp;
  const lower = utils.getFirstDayOfMonth(upper);
  return {lower, upper};
};
