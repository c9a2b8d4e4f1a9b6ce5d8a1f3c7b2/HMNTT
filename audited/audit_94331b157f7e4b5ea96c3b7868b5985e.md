### Title
Unauthenticated DoS via Database Connection Pool Exhaustion on `/accounts/:idOrAliasOrEvmAddress/allowances/tokens`

### Summary
The `getAccountTokenAllowances()` handler in `rest/controllers/tokenAllowanceController.js` is reachable by any unauthenticated user with no rate limiting enforced at the application layer. Every request unconditionally acquires a connection from the finite PostgreSQL connection pool and executes a database query. A flood of concurrent requests from an unprivileged attacker will exhaust the pool, causing all subsequent requests across the entire REST service to fail or queue indefinitely.

### Finding Description
**Exact code path:**

`rest/controllers/tokenAllowanceController.js` lines 68–81 — `getAccountTokenAllowances()` calls `EntityService.getEncodedId()` (a DB lookup) and then `TokenAllowanceService.getAccountTokenAllowances(query)` (another DB query via `BaseService.getRows()` → `pool.queryQuietly()`). Both calls consume a connection from the global `pool` initialized in `rest/dbpool.js` line 14 (`max: config.db.pool.maxConnections`).

**Root cause — no rate limiting in the REST service:**

`rest/server.js` lines 67–144 define the full middleware stack. The registered middleware is:
- `express.urlencoded` / `express.json` / `cors()`
- `httpContext.middleware` + `requestLogger`
- `authHandler` — only sets a custom response *limit* for authenticated users; unauthenticated requests pass through unconditionally (lines 15–19 of `rest/middleware/authHandler.js`: `if (!credentials) { return; }`)
- Optional metrics handler
- Optional `responseCacheCheckHandler` (requires both `config.cache.response.enabled` AND `config.redis.enabled`; disabled by default; cache misses still hit the DB; different account IDs bypass the cache)
- Route handlers

There is no `express-rate-limit`, token bucket, or any other per-IP or global request throttle. The `grep_search` across all of `rest/` for `rate.?limit|rateLimit|throttle|express-rate-limit` returned only matches in `openapi.yml` (documentation) and `integrationUtils.js` (test helpers) — confirming zero runtime enforcement.

The `ThrottleConfiguration` / `ThrottleManagerImpl` with Bucket4j that exists in `web3/src/main/java/...` applies exclusively to the Java/Spring Web3 service and has no effect on this Node.js REST service.

**Failed assumption:** The design assumes an external reverse proxy (e.g., nginx, cloud WAF) will enforce rate limits before traffic reaches the application. No such control is implemented or enforced within the application itself, and deployments without a properly configured proxy are fully exposed.

### Impact Explanation
The PostgreSQL connection pool is a shared, finite resource (`max: config.db.pool.maxConnections`, `rest/dbpool.js` line 14). When the pool is saturated:
- New requests across **all** REST endpoints (not just this one) block waiting for a free connection until `connectionTimeoutMillis` expires, returning errors to legitimate users.
- The `statement_timeout` (`rest/dbpool.js` line 15) limits individual query duration but does not prevent pool exhaustion from many concurrent short-lived queries.
- The entire mirror node REST API becomes unavailable, affecting all consumers (wallets, explorers, dApps).

Severity: **High** — complete service unavailability, no authentication required, no per-user isolation.

### Likelihood Explanation
- **Precondition:** None. The endpoint is public; no credentials, API key, or special network access is needed.
- **Attacker capability:** Any script kiddie with `curl` or `ab` (Apache Bench) can execute this. No exploit code required.
- **Repeatability:** Continuous; the attacker simply keeps the flood running. Stopping requires operator intervention (firewall rule, proxy config, or application restart).
- **Detectability:** The `responseHandler` logs requests at a sampled rate (`Math.random() <= sampleRate`, `rest/middleware/responseHandler.js` line 56), meaning high-volume attacks may not be fully logged.

### Recommendation
1. **Add application-level rate limiting** using `express-rate-limit` (or equivalent) as a middleware in `rest/server.js`, applied globally before route handlers, keyed on `req.ip`. Example:
   ```js
   import rateLimit from 'express-rate-limit';
   app.use(rateLimit({ windowMs: 1000, max: 100 })); // tune values
   ```
2. **Enforce at the infrastructure layer** with an nginx `limit_req_zone` or cloud WAF rule as a defense-in-depth measure.
3. **Increase pool connection timeout aggressiveness** and add a queue depth limit so pool exhaustion fails fast rather than queuing indefinitely.
4. **Enable response caching** (`config.cache.response.enabled` + Redis) to serve repeated identical queries from cache, reducing DB pressure.

### Proof of Concept
```bash
# Flood the endpoint with 5000 concurrent requests, 10 at a time
# No credentials required
ab -n 5000 -c 200 \
  "https://<mirror-node-host>/api/v1/accounts/0.0.1234/allowances/tokens"

# Simultaneously verify that other endpoints also fail (pool exhaustion is global):
curl "https://<mirror-node-host>/api/v1/transactions"
# Expected: connection timeout or 503 once pool is exhausted
```

The attack succeeds because:
1. Each `ab` request hits `getAccountTokenAllowances()` (`rest/controllers/tokenAllowanceController.js:68`)
2. Which calls `TokenAllowanceService.getAccountTokenAllowances()` (`rest/service/tokenAllowanceService.js:86`)
3. Which calls `BaseService.getRows()` (`rest/service/baseService.js:55`) acquiring a pool connection
4. With 200 concurrent requests and a default pool size (typically 10–20), the pool saturates immediately
5. All subsequent requests across the service queue or timeout
