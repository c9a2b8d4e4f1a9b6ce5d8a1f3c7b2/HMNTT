### Title
Cache-Busting DoS via Unbounded Parse-and-Validate Cycles on GraphQL Endpoint (No Application-Level Rate Limiting)

### Summary
The `CachedPreparsedDocumentProvider` caches parsed GraphQL documents keyed by the raw query string with a maximum of 1000 entries. Because there is no application-level rate limiting on the `/graphql/alpha` endpoint (unlike the `web3` module which has a full `ThrottleManager`), an unprivileged attacker can continuously submit syntactically valid but unique queries (e.g., varying whitespace, aliases, or field ordering) to perpetually saturate and evict the cache, forcing a full parse-and-validate cycle on every request. The parser limits bound per-query work but do not limit request throughput.

### Finding Description

**Cache key is the raw query string — trivially bypassed:** [1](#0-0) 

The cache lookup uses `executionInput.getQuery()` as the key. Any single-character difference (whitespace, alias name, comment) produces a cache miss and triggers `parseAndValidateFunction.apply(executionInput)`.

**Cache is capped at 1000 entries with LRU eviction:** [2](#0-1) 

Once 1001 unique queries are submitted, Caffeine evicts the oldest entry. An attacker cycling through >1000 unique query strings ensures every request is a cache miss.

**Parser limits bound per-query work but do not limit request rate:** [3](#0-2) 

`maxCharacters(10000)` and `maxTokens(1000)` cap the size of each query but impose no constraint on how many queries per second can be submitted.

**`MaxQueryComplexityInstrumentation` and `MaxQueryDepthInstrumentation` run post-parse:** [4](#0-3) 

These instrumentations execute only after the parse-and-validate cycle completes, providing no protection against the CPU cost of parsing itself.

**No application-level rate limiting exists for the GraphQL module.** The `ThrottleManager` / `ThrottleConfiguration` / `ThrottleProperties` stack exists exclusively in the `web3` module: [5](#0-4) 

There is no equivalent in the `graphql` module. No `@RateLimiter`, no `Bucket4j` bean, no servlet filter enforcing per-IP or global request-per-second limits on `/graphql/alpha`.

**Infrastructure-level rate limiting is optional and not default:** [6](#0-5) [7](#0-6) 

The GCP backend policy (`maxRatePerEndpoint: 250`) and Traefik `inFlightReq` middleware are infrastructure-level controls that are deployment-specific and not enforced by the application itself. `global.middleware` defaults to `false`.

### Impact Explanation
An attacker with no credentials can force the server to execute a full GraphQL parse-and-validate cycle on every request by rotating through >1000 unique query strings. Each cycle involves lexing, parsing (AST construction), and schema validation — all CPU-bound operations. With no application-level rate limiting, the attacker can sustain this at the maximum rate the network and server threads allow, driving CPU consumption well above baseline. Memory pressure also increases as the Caffeine cache continuously allocates and GC-collects `PreparsedDocumentEntry` objects. This constitutes a resource exhaustion / denial-of-service condition achievable by any unauthenticated external user.

### Likelihood Explanation
The attack requires only an HTTP client and the ability to generate unique query strings — trivially automated. No authentication, no special knowledge, and no brute-force pattern (each request is a valid, well-formed GraphQL query) is needed. The attack is repeatable indefinitely and is not detectable as brute force since every query is semantically distinct and syntactically valid. Deployments without the optional GCP or Traefik infrastructure controls are fully exposed at the application layer.

### Recommendation
1. **Add application-level rate limiting to the GraphQL module** — implement a `Bucket4j`-based filter (mirroring `ThrottleConfiguration` in `web3`) that enforces a per-IP and/or global requests-per-second limit on `/graphql/alpha`.
2. **Normalize the cache key** before storage — strip insignificant whitespace and comments from the query string before using it as the Caffeine cache key, so semantically equivalent queries share a cache entry.
3. **Make infrastructure rate limiting mandatory** — document and enforce that the GCP backend policy or Traefik `inFlightReq` middleware must be enabled in all production deployments; do not leave it as an opt-in default.
4. **Consider a query allowlist or persisted queries** — only accept pre-registered query hashes, eliminating the ability to submit arbitrary unique query strings entirely.

### Proof of Concept

```python
import requests, itertools, string, threading

URL = "http://<graphql-host>/graphql/alpha"
HEADERS = {"Content-Type": "application/json"}

BASE = 'query Q{{ account(input:{{id:"0.0.1"}}){{ balance }} # {} }}'

def flood():
    for i in itertools.count():
        # Each query differs by one character in the comment — unique cache key
        query = BASE.format(i)
        requests.post(URL, json={"query": query}, headers=HEADERS, timeout=5)

# Launch N concurrent threads to maximize throughput
threads = [threading.Thread(target=flood) for _ in range(50)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

Each iteration produces a query string that differs by the comment suffix, guaranteeing a cache miss in `CachedPreparsedDocumentProvider`. With 50 concurrent threads and no application-level rate limiting, every request triggers a full parse-and-validate cycle, sustaining elevated CPU consumption on the mirror node.
