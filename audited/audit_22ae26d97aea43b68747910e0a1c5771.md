### Title
Unauthenticated Double DB Query Per Request via Timestamp Filter on Token Balances Endpoint

### Summary
Every call to `GET /api/v1/tokens/:tokenId/balances?timestamp=<any>` by an unprivileged user unconditionally triggers two sequential database queries: one inside `balances.getAccountBalanceTimestampRange()` against the `account_balance` table (filtered on the treasury account), and a second for the actual token balance data. There is no caching, no application-level rate limiting in the REST layer, and no authentication requirement, making this a straightforward DB-amplification DoS vector.

### Finding Description
**Exact code path:**

In `rest/tokens.js`, `extractSqlFromTokenBalancesRequest()` (lines 619–640): [1](#0-0) 

When any `timestamp` filter is present, `transformTimestampFilterOp()` converts `eq` → `lte` (line 470–473): [2](#0-1) 

Then `balances.getAccountBalanceTimestampRange()` is awaited unconditionally (line 640), which executes **DB Query #1** against `account_balance` for the treasury account: [3](#0-2) 

Back in `getTokenBalances()`, **DB Query #2** is then executed with the result (line 724): [4](#0-3) 

**Root cause:** `getAccountBalanceTimestampRange()` has no result cache and always issues a live SQL query. The REST API has no application-level rate limiting (the throttle infrastructure found in the codebase is scoped exclusively to the `web3` Java module): [5](#0-4) 

**Failed assumption:** The design assumes the treasury-account lookup is cheap enough to absorb at full public request volume. It is not protected by any per-IP or per-endpoint throttle in the Node.js REST service.

### Impact Explanation
An attacker can double the effective DB query rate on the `account_balance` table (which is large and monthly-partitioned) relative to unauthenticated baseline traffic, with zero marginal cost to the attacker. At sustained high request rates this degrades DB connection pool availability and query latency for all consumers of the mirror node REST API. Because `account_balance` is partitioned by month and the query scans up to two partitions per call, the per-query cost is non-trivial under load.

### Likelihood Explanation
The endpoint is public, requires no credentials, and accepts any syntactically valid timestamp value. The attack is trivially scriptable (`curl` in a loop or any HTTP load tool), repeatable indefinitely, and requires no knowledge of valid token IDs (any numeric ID causes the timestamp branch to execute before the token is validated). A single attacker with modest bandwidth can sustain thousands of double-query requests per second.

### Recommendation
1. **Cache the treasury-account timestamp range result** with a short TTL (e.g., 1–5 seconds) keyed on the resolved `(lowerBound, upperBound)` pair. Balance snapshots are written on a fixed schedule, so stale results are acceptable for a few seconds.
2. **Add application-level rate limiting** to the Node.js REST service (e.g., `express-rate-limit`) for the `/tokens/:tokenId/balances` endpoint, mirroring the throttle already present in the web3 module.
3. **Validate the token ID exists before entering the timestamp branch** so that requests with non-existent token IDs short-circuit before issuing any DB query.

### Proof of Concept
```bash
# No authentication required. Any valid timestamp value triggers two DB queries.
TOKEN_ID=0.0.1234
TIMESTAMP=1700000000.000000000

# Single request → 2 DB queries (1 in getAccountBalanceTimestampRange + 1 for token_balance)
curl "https://<mirror-node>/api/v1/tokens/$TOKEN_ID/balances?timestamp=eq:$TIMESTAMP"

# Sustained flood → DB query rate = 2× request rate, no rate limit enforced
while true; do
  curl -s "https://<mirror-node>/api/v1/tokens/$TOKEN_ID/balances?timestamp=eq:$TIMESTAMP" &
done
```

Observe in DB metrics that `account_balance` query count equals the request count (one extra query per request vs. the no-timestamp baseline), confirming the amplification.

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first

    from account_balance
