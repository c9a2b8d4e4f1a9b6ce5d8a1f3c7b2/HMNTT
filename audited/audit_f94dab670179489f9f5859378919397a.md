### Title
GraphQL `findByEvmAddress` Exposes Entities with `deleted = NULL` via `IS NOT TRUE` Predicate Inconsistency

### Summary
The `findByEvmAddress()` query in `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java` uses `deleted is not true`, which in SQL returns rows where `deleted = false` **and** where `deleted IS NULL`. Entities with `deleted = NULL` represent records whose deletion status was never explicitly confirmed by the importer (e.g., inserted via balance migration upserts or partial transaction processing). Every other module (`rest-java`, `importer`) uses `deleted <> true`, which correctly excludes NULL rows. An unprivileged external user can query the GraphQL API with the EVM address of such an entity and receive data that all other API surfaces would refuse to return.

### Finding Description
**Exact location:** `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 16–17:
```java
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
``` [1](#0-0) 

**Root cause:** In SQL, `deleted IS NOT TRUE` evaluates to `TRUE` for both `deleted = false` and `deleted IS NULL`. The `deleted` column was intentionally made nullable by migration `V1.39.1__upsert_support.sql` with the comment *"allow nullable on entity deleted as transaction cannot make this assumption on updates"*, meaning `NULL` is a valid, persistent database state representing an entity whose active/deleted status is indeterminate. [2](#0-1) 

**Contrast with correct implementations:**
- `rest-java/EntityRepository.findByEvmAddress()` uses `deleted <> true` → excludes NULL
- `importer/EntityRepository.findByEvmAddress()` uses `deleted <> true` → excludes NULL [3](#0-2) [4](#0-3) 

The rest-java integration test explicitly asserts that `deleted = NULL` entities must **not** be returned: [5](#0-4) 

**Exploit flow:**
1. An entity with `deleted = NULL` exists in the DB (produced by `InitializeEntityBalanceMigration` upserts, or any importer path that does not set the `deleted` field). [6](#0-5) 
2. An external user sends a GraphQL query: `{ account(input: { evmAddress: "0x<target>" }) { id balance ... } }`.
3. `AccountController.account()` calls `EntityServiceImpl.getByEvmAddressAndType()`, which calls `entityRepository.findByEvmAddress(evmAddressBytes)`. [7](#0-6) 
4. The query returns the entity because `deleted IS NOT TRUE` matches `NULL`. The only downstream filter is a type check (`.filter(e -> e.getType() == type)`), which does not re-check deletion status.
5. The user receives full entity data for a record that is in an unconfirmed/indeterminate state.

**Why existing checks are insufficient:** `EntityServiceImpl.getByEvmAddressAndType()` applies only an `EntityType` filter after the repository call. There is no post-query guard on the `deleted` field value. [8](#0-7) 

### Impact Explanation
The GraphQL mirror node API returns entity records that are in an indeterminate deletion state — records that the importer has not confirmed as active. This creates an inconsistency: the same EVM address queried via the REST API returns "not found", while the GraphQL API returns a populated account object. Downstream consumers relying on the GraphQL API may treat unconfirmed entities as live accounts, leading to incorrect balance reporting, incorrect token ownership resolution, or incorrect contract existence checks. Severity is **medium**: data integrity violation with no authentication bypass, but directly observable by any external caller.

### Likelihood Explanation
Entities with `deleted = NULL` are a normal, documented database state (the nullable migration comment and the balance migration upsert both confirm this). No special privileges are required — only knowledge of an EVM address, which is public on-chain data. The attack is trivially repeatable by any user of the public GraphQL endpoint. The only friction is identifying a specific EVM address whose entity has `deleted = NULL`, which is feasible by scanning known contract/account addresses.

### Recommendation
Replace `deleted is not true` with `deleted <> true` (or equivalently `deleted = false`) in both queries in `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`:

```java
@Query(value = "select * from entity where alias = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

This aligns the GraphQL repository with the behavior of `rest-java` and `importer` repositories and ensures entities with `deleted = NULL` are never surfaced to external callers. [9](#0-8) 

### Proof of Concept
**Precondition:** An entity row exists in the `entity` table with a known `evm_address` and `deleted = NULL`. This can be produced by running `InitializeEntityBalanceMigration` when a `crypto_transfer` references an account that has no prior `account_balance` snapshot entry (the `CASE WHEN balance IS NOT NULL THEN false END` expression yields NULL for the `deleted` column).

**Trigger:**
```graphql
query {
  account(input: { evmAddress: "0x<evm_address_of_null_deleted_entity>" }) {
    id
    balance
    type
  }
}
```

**Expected (correct) result:** `{ "data": { "account": null } }` — matching REST API behavior.

**Actual result:** The account object is returned with full field data, because `deleted IS NOT TRUE` matches the `NULL` row.

**Verification:** Insert a test entity with `deleted = NULL` and a known `evm_address` directly into the DB, then issue the GraphQL query above. Confirm the REST-Java endpoint for the same address returns 404, while GraphQL returns the entity.
