### Title
Rate-Limit Token Consumed Without Restore on Gas-Check Failure Enables Rate-Bucket Exhaustion DoS

### Summary
In `ThrottleManagerImpl.throttle()`, a rate-limit token is consumed from `rateLimitBucket` before the gas check against `gasLimitBucket` is performed. When the gas check fails, a `ThrottleException` is thrown but the already-consumed rate-limit token is never restored. An unprivileged attacker who can arrange for the gas bucket to be empty can then flood the endpoint with requests that each fail the gas check but permanently drain rate tokens, exhausting the rate bucket and causing all users to receive `429`-equivalent errors.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, lines 38–42:

```java
if (!rateLimitBucket.tryConsume(1)) {                                    // line 38 – rate token consumed
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // line 40
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);          // line 41 – rate token NOT restored
}
``` [1](#0-0) 

**Root cause:** The ordering is rate-check → gas-check. Bucket4j's `tryConsume` is atomic: if it returns `false`, no tokens are consumed from the gas bucket. However, the rate token consumed at line 38 is never returned when the gas check at line 40 fails.

**Why existing restore logic is insufficient:**

`restore()` only adds tokens back to `gasLimitBucket`, never to `rateLimitBucket`:

```java
@Override
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);   // only gas bucket
    }
}
``` [2](#0-1) 

In `ContractController`, `restore()` is only called for `InvalidParametersException`, not for `ThrottleException`:

```java
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
``` [3](#0-2) 

A `ThrottleException` thrown from the gas check propagates out of the `try` block entirely uncaught by this handler, so no restore occurs.

**Bucket parameters (defaults):**

| Bucket | Capacity | Refill |
|---|---|---|
| `rateLimitBucket` | 500 tokens | 500/sec |
| `gasLimitBucket` | `scaleGas(7_500_000_000)` = 750,000 tokens | 750,000/sec | [4](#0-3) 

`maxGasLimit` = 15,000,000 → `scaleGas(15_000_000)` = 1,500 gas tokens per max-gas request. [5](#0-4) 

### Impact Explanation

Once the gas bucket is empty (see exploit path below), every subsequent request consumes one rate token and is rejected with `GAS_PER_SECOND_LIMIT_EXCEEDED`. After 500 such requests the rate bucket is exhausted and every user — including legitimate ones — receives `REQUEST_PER_SECOND_LIMIT_EXCEEDED` for up to one full second. The attacker can repeat this every second with a trivial script, creating a sustained denial-of-service against the `/api/v1/contracts/call` endpoint. No authentication, API key, or special privilege is required.

### Likelihood Explanation

The attack requires no credentials and is executable from a single machine with a basic HTTP client. The gas bucket can be drained by legitimate traffic (high-gas calls from real users) or by the attacker's own first wave of requests. The 500-request threshold to exhaust the rate bucket is reachable in well under one second with concurrent HTTP connections. The attack is fully repeatable every second.

### Recommendation

Restore the rate-limit token when the gas check fails. The simplest fix is to check the gas bucket first (without consuming) and only consume the rate token if the gas check will succeed:

```java
long scaledGas = throttleProperties.scaleGas(request.getGas());
if (!gasLimitBucket.tryConsume(scaledGas)) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
if (!rateLimitBucket.tryConsume(1)) {
    gasLimitBucket.addTokens(scaledGas);   // restore gas if rate check fails
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, use bucket4j's `tryConsumeAndReturnRemaining` on both buckets atomically, or add a `rateLimitBucket.addTokens(1)` call in the `else if` branch before throwing.

### Proof of Concept

**Preconditions:** Default configuration (`requestsPerSecond=500`, `gasPerSecond=7_500_000_000`, `maxGasLimit=15_000_000`).

**Step 1 – Drain the gas bucket (500 requests, gas=15_000_000 each):**
```bash
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x000...001","gas":15000000}' &
done
wait
```
Each request consumes 1,500 gas tokens; 500 × 1,500 = 750,000 = full gas bucket capacity. Gas bucket is now empty.

**Step 2 – Drain the rate bucket using gas-rejected requests (500 more requests):**
```bash
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x000...001","gas":15000000}' &
done
wait
```
Each request: rate token consumed at line 38, gas check fails at line 40 (bucket empty), `ThrottleException` thrown, rate token never restored. After 500 requests the rate bucket is empty.

**Step 3 – Observe impact:**
Any request from any user now returns `Requests per second rate limit exceeded` for up to 1 second.

**Step 4 – Repeat steps 1–3 every second** to maintain continuous DoS.
