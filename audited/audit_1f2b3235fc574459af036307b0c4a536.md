### Title
Unsynchronized Read of `genesisBlock` Before `sync.Once` Write Causes Data Race in `blockRepository`

### Summary
`blockRepository.initGenesisRecordFile` reads `br.genesisBlock.ConsensusStart` at line 244 without any lock or synchronization primitive, while a concurrent goroutine may be writing the entire `br.genesisBlock` struct inside `br.once.Do` at line 258. `sync.Once` only serializes the write; it provides no happens-before guarantee for the unsynchronized pre-check read at line 244. Under Go's memory model this is a data race, producing undefined behavior. Any unauthenticated HTTP client can trigger it by sending concurrent requests during server startup.

### Finding Description
**Exact code path:**

`rosetta/app/persistence/block.go`, `blockRepository`, `initGenesisRecordFile` (lines 243–263).

```
243: func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
244:     if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {  // ← unsynchronized READ
245:         return nil
246:     }
...
257:     br.once.Do(func() {
258:         br.genesisBlock = rb   // ← unsynchronized WRITE of entire struct
259:     })
```

**Root cause:**

The early-exit guard at line 244 reads `br.genesisBlock.ConsensusStart` with no mutex, no atomic, and no other synchronization. `sync.Once` only guarantees that the closure at line 257–259 runs exactly once and that *subsequent calls to `once.Do`* observe the write. It does **not** establish a happens-before relationship between the write at line 258 and the bare read at line 244 in a racing goroutine. Go's memory model therefore classifies this as a data race, and the behavior is undefined.

**Exploit flow:**

1. Server starts; `br.genesisBlock.ConsensusStart` is `-1` (unset).
2. Goroutine A (request 1) enters `initGenesisRecordFile`, passes the check at line 244 (sees `-1`), queries the DB, receives `rb`, and begins executing `once.Do` → writes `br.genesisBlock = rb` at line 258.
3. Goroutine B (request 2), racing concurrently, reads `br.genesisBlock.ConsensusStart` at line 244 **while Goroutine A is mid-write** of the multi-word `recordBlock` struct.
4. `recordBlock` contains two `string` fields (`Hash`, `PrevHash`). A Go string is a two-word header (pointer + length). A torn read can yield a pointer from the new value paired with a length from the old value (or vice versa), causing an out-of-bounds memory access and a **panic** in the server process.

**Why existing checks are insufficient:**

`sync.Once` is the only synchronization used. It serializes the write but has no effect on the unsynchronized read at line 244 that precedes the `once.Do` call. There is no mutex, `atomic.Value`, or channel protecting `br.genesisBlock` at the read site.

### Impact Explanation
- **Crash / DoS:** A torn read of the `Hash` or `PrevHash` string header causes an invalid memory dereference, panicking the goroutine. In a Go HTTP server each request runs in its own goroutine; a panic that is not recovered crashes the entire server process, taking down the Rosetta API.
- **Data integrity:** Even without a crash, a goroutine may observe a partially written `genesisBlock` and use a corrupted genesis index or hash for all subsequent block lookups in that request, returning wrong block data to the caller.
- **Severity:** High — server crash is achievable with no authentication.

### Likelihood Explanation
- The Rosetta API is an unauthenticated HTTP service. Any external client can issue concurrent requests.
- The race window is the initialization phase (first successful DB query). On a loaded node this window is easily hit by sending a burst of requests at startup or after a restart.
- The Go race detector confirms this pattern as a race; it is reproducible with `go test -race` or `wrk`/`hey` against a live instance.
- After `once.Do` completes the write is stable and subsequent reads are safe, but the window before that point is reliably exploitable.

### Recommendation
Move the entire initialization — including the sentinel check — inside `sync.Once`, so the read and write are always serialized:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()
        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })
    return initErr
}
```

Note: if retry-on-DB-failure is required, replace `sync.Once` with a `sync.Mutex` guarding both the sentinel check and the write, or use `atomic.Value` to store the `recordBlock` pointer.
