### Title
Cache TTL Reset Amplification in `findStorageBatch()` Enables Sustained DB Overload by Unprivileged Users

### Summary
`ContractStateServiceImpl.findStorageBatch()` issues a batch DB query for **all** slot keys currently held in the per-contract `slotsPerContract` cache (up to 1,500 keys) every time the short-lived `contractStateCache` (TTL=2 s) misses. Because each call to `findStorageBatch()` invokes `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)`, it resets the `expireAfterAccess=5m` TTL for the slot key being requested. An unprivileged attacker who cycles through all cached slot keys at a low, sustained rate (≈5 req/s for 1,500 slots) keeps every key alive in the `slotsPerContract` cache indefinitely while forcing a 1,500-slot batch DB query on every single request, achieving a ≥1,500× amplification of database load with no authentication required.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (line 63) checks `contractStateCache` (TTL=2 s, `expireAfterWrite`). On a miss it calls `findStorageBatch()` (line 69). [1](#0-0) 

Inside `findStorageBatch()`:
- Line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — inserts the requested slot key into the Caffeine `slotsPerContract` cache **and resets its `expireAfterAccess` TTL to 5 minutes**.
- Line 91: `contractSlotsCache.getNativeCache().asMap().keySet()` — snapshots **all** currently cached slot keys. In Caffeine, `asMap()` iteration does **not** reset the access time of other entries; only the explicit `putIfAbsent` on line 90 resets the TTL of the one key being requested.
- Line 103: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — issues a single SQL `IN (...)` query against `contract_state` for **every** key in the snapshot. [2](#0-1) 

**Cache configuration:**
- `contractState`: `expireAfterWrite=2s` — values expire 2 seconds after being written.
- `slotsPerContract`: `expireAfterAccess=5m,maximumSize=1500` — slot keys survive 5 minutes after their last explicit access.
- `contractSlots` (outer map, contractId → per-contract cache): `expireAfterAccess=5m,maximumSize=3000`. [3](#0-2) 

**Root cause / failed assumption:**
The design assumes that the `slotsPerContract` cache is a passive accumulator of "recently seen" keys and that the batch query is a cheap pre-warming optimisation. The failed assumption is that the 2-second value cache will absorb repeated reads. In reality, the 2-second TTL guarantees that every slot key that is re-requested after 2 seconds triggers a full batch query over **all** keys accumulated in the 5-minute window, not just the one being requested. There is no cap on how many keys can be batched per query beyond the cache's `maximumSize`.

**Exploit flow:**
1. Attacker enumerates up to 1,500 slot keys for a high-value contract (slot keys are deterministic for ERC-20/ERC-721 contracts or discoverable via `eth_getStorageAt`).
2. Attacker sends one `eth_call` per slot key, populating the `slotsPerContract` cache with all 1,500 keys.
3. Attacker then cycles through all 1,500 slot keys at a rate of 1,500 / 300 s = **5 req/s** (one full cycle every 5 minutes), ensuring each key's TTL is reset before it expires.
4. Every single one of those 5 requests/s misses `contractStateCache` (2 s TTL) and triggers `findStorageBatch()`, which issues a DB query for all 1,500 cached keys.
5. Net DB load: **5 req/s × 1,500 slots = 7,500 slot lookups/s**, sustained indefinitely.
6. The attacker can scale to multiple contracts (up to 3,000 in the outer `contractSlots` cache), multiplying the load further.

### Impact Explanation

Each attacker request is amplified by up to 1,500× at the database layer. With only 5 req/s (well below the 500 req/s global rate limit), the attacker sustains 7,500 DB row lookups per second for a single contract. Targeting multiple contracts simultaneously (up to 3,000 in the outer cache) and using the full 500 req/s budget yields up to **750,000 slot lookups/s** — a sustained, unbounded increase in DB CPU, I/O, and connection pool pressure. This directly satisfies the ≥30% resource consumption increase threshold without brute-force volume. The `statementTimeout=3000ms` does not help because each individual batch query completes within the timeout; the damage is from the sustained repetition. [4](#0-3) 

### Likelihood Explanation

- **No authentication required**: any caller of `/api/v1/contracts/call` can trigger this.
- **Slot key discovery is trivial**: for standard token contracts, storage layout is public; `eth_getStorageAt` can enumerate slots.
- **Request rate required is minimal**: 5 req/s is far below the 500 req/s global throttle and generates no per-IP alarm.
- **Fully repeatable and automatable**: a simple script cycling through known slot keys every 2 seconds per slot is sufficient.
- **No gas cost**: `eth_call` is a read-only simulation; the gas throttle (`gasPerSecond`) applies to simulated execution gas, not to the number of DB rows fetched. [5](#0-4) 

### Recommendation

1. **Cap the batch size**: Before calling `findStorageBatch()`, limit `cachedSlots` to a configurable maximum (e.g., 100 keys) regardless of how many keys are in the `slotsPerContract` cache.
2. **Switch `contractStateCache` to `expireAfterAccess`**: Aligning its eviction policy with `slotsPerContract` removes the guaranteed 2-second miss cycle that drives repeated batch queries.
3. **Add per-contract-per-IP rate limiting** on `findStorageBatch()` invocations to prevent a single caller from sustaining the amplification loop.
4. **Replace `expireAfterAccess` with `expireAfterWrite` on `slotsPerContract`**: This prevents indefinite TTL extension via periodic re-access and bounds the maximum lifetime of accumulated slot keys. [6](#0-5)

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
