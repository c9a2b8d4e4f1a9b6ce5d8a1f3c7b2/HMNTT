### Title
Unauthenticated Cache-Miss DoS on `/api/v1/network/stake` via Uncached Correlated Subquery

### Summary
`getNetworkStake()` in `NetworkController` is a publicly accessible, unauthenticated endpoint with no caching and no rate limiting in the `rest-java` module. Every request triggers a correlated subquery (`SELECT max(consensus_timestamp)` + full row fetch) directly against the database. A flood of concurrent requests from any unprivileged external user will saturate the database I/O and connection pool, degrading or blocking all other API endpoints that share the same database.

### Finding Description
**Exact code path:**

`NetworkController.getNetworkStake()` (lines 126–130) calls `networkService.getLatestNetworkStake()` unconditionally on every HTTP request with no caching layer: [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56) directly delegates to the repository with no in-memory result retention: [2](#0-1) 

`NetworkStakeRepository.findLatest()` executes a correlated subquery on every invocation — a `SELECT max(consensus_timestamp)` aggregate scan followed by a full row fetch — with **no `@Cacheable` annotation**: [3](#0-2) 

**Root cause / failed assumption:** The `rest-java` module has no rate limiting infrastructure. The throttle/bucket4j machinery (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) exists exclusively in the `web3` module: [4](#0-3) 

The only filters registered in `rest-java` are `ShallowEtagHeaderFilter`, `LoggingFilter`, and `MetricsFilter` — none enforce a request rate limit: [5](#0-4) 

By contrast, the `grpc` module's analogous `NodeStakeRepository` correctly applies `@Cacheable` to its stake lookup: [6](#0-5) 

The `rest-java` `NetworkStakeRepository` has no equivalent protection.

### Impact Explanation
Each request to `GET /api/v1/network/stake` issues two SQL operations against the shared PostgreSQL instance (a `MAX()` aggregate scan and a row fetch). Under concurrent flood conditions, the database connection pool is exhausted and I/O is saturated. Because all `rest-java` endpoints share the same datasource, this causes cascading latency and timeout failures across the entire REST API — including transaction submission endpoints — constituting a practical network-wide API outage. No data is modified, but availability is fully compromised.

### Likelihood Explanation
The endpoint requires zero authentication, zero parameters, and zero privileges. Any attacker with network access can issue thousands of concurrent `GET /api/v1/network/stake` requests using standard tools (`curl`, `ab`, `wrk`, `k6`). The attack is trivially repeatable, requires no special knowledge, and is not mitigated by any in-process control in the `rest-java` module. The `network_stake` table grows over time (one row per staking period), making the `MAX()` scan progressively more expensive.

### Recommendation
1. **Add `@Cacheable`** to `NetworkStakeRepository.findLatest()` with a TTL aligned to the staking period (e.g., 1 hour), mirroring the pattern already used in `grpc/NodeStakeRepository`.
2. **Add rate limiting** to the `rest-java` module (port the bucket4j `ThrottleConfiguration` pattern from `web3`, or add a servlet filter) to enforce a per-IP or global requests-per-second cap on all endpoints.
3. **Add HTTP cache headers** (`Cache-Control: max-age`) on the `/network/stake` response so upstream proxies/CDNs can absorb repeated requests.

### Proof of Concept
```bash
# Flood the endpoint with 500 concurrent connections, 10000 total requests
ab -n 10000 -c 500 http://<mirror-node-host>/api/v1/network/stake

# Observe: database CPU/IO spikes to 100%, connection pool exhausted,
# other endpoints (e.g., /api/v1/transactions) begin returning 500/503.
# No authentication, API key, or special header required.
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
