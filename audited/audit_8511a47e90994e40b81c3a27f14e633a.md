### Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Distinct Contract IDs Causes Heap Exhaustion

### Summary
In `ContractStateServiceImpl.findStorageBatch()`, every unique `contractId` triggers a call to `cacheManagerSlotsPerContract.getCache(contractId.toString())`, which stores a new `CaffeineCache` instance in the `CaffeineCacheManager`'s internal `ConcurrentHashMap` permanently. The outer `contractSlotsCache` is bounded at 3000 entries, but the `cacheManagerSlotsPerContract` bean's internal map is never evicted and grows without bound. An unprivileged user can exhaust JVM heap by issuing requests for arbitrarily many distinct contract IDs.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.java`, `findStorageBatch()`, lines 85–87:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause — two-level cache with asymmetric eviction:**

The outer `contractSlotsCache` is configured via `CACHE_MANAGER_CONTRACT_SLOTS` with `"expireAfterAccess=5m,maximumSize=3000,recordStats"` — bounded at 3000 entries.

The inner `cacheManagerSlotsPerContract` bean (`EvmConfiguration.java` lines 107–112) is a `CaffeineCacheManager` created **without** calling `setCacheNames()`:

```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    return caffeineCacheManager;  // no setCacheNames() — dynamic mode
}
```

In Spring's `CaffeineCacheManager`, when `setCacheNames()` is not called, the manager operates in **dynamic mode**: every call to `getCache(name)` for an unseen name creates a new `CaffeineCache` and stores it in an internal `ConcurrentHashMap<String, Cache>`. **This internal map has no eviction mechanism.**

**Exploit flow:**

1. Outer cache misses on `contractId` → loader fires → `cacheManagerSlotsPerContract.getCache(contractId.toString())` is called.
2. A new `CaffeineCache` (with `maximumSize=1500`) is created and inserted into `cacheManagerSlotsPerContract`'s internal `ConcurrentHashMap` under the key `contractId.toString()`.
3. The same `CaffeineCache` is stored as a value in the outer `contractSlotsCache`.
4. When the outer cache evicts the entry (due to `maximumSize=3000` or TTL), the `CaffeineCache` is removed from the outer cache — but `cacheManagerSlotsPerContract` still holds a strong reference to it in its internal map. The object is **not garbage collected**.
5. On the next request for the same evicted `contractId`, the loader fires again, `cacheManagerSlotsPerContract.getCache()` returns the **existing** (still-live) `CaffeineCache` — confirming the internal map retains all entries ever created.
6. For each new unique `contractId`, a new permanent entry is added to `cacheManagerSlotsPerContract`'s internal map.

**Why existing checks fail:**

The `maximumSize=3000` on `contractSlotsCache` (`CacheProperties.java` line 25) only bounds the outer cache. It does not limit `cacheManagerSlotsPerContract`'s internal `ConcurrentHashMap`, which is the actual memory sink. The `slotsPerContract` spec (`"expireAfterAccess=5m,maximumSize=1500"`) bounds slots *within* each per-contract cache, not the number of per-contract caches. There is no rate limiting, authentication, or contract-existence check before the cache entry is created.

### Impact Explanation
Each `CaffeineCache` instance with `maximumSize=1500` carries Caffeine's internal data structures (write buffer, read buffer, frequency sketch, node arrays). Even for an empty cache, this overhead is on the order of tens of KB per instance. With 100,000 unique contract IDs: ~100,000 × ~50 KB = ~5 GB of heap consumed in `cacheManagerSlotsPerContract`'s internal map alone, causing `OutOfMemoryError` or severe GC pressure. This degrades or crashes the web3 service, directly impacting 30%+ of network processing nodes that run this service.

### Likelihood Explanation
The attack requires no authentication, no privileged access, and no valid on-chain contracts — the code creates the cache entry before any DB lookup confirms the contract exists. A single attacker with a script can generate millions of distinct `contractId` values (they are just `long` integers) and issue `eth_getStorageAt`-equivalent calls. The attack is repeatable, automatable, and requires only HTTP access to the public web3 endpoint.

### Recommendation
1. **Remove `cacheManagerSlotsPerContract` as a `CaffeineCacheManager` value store.** Instead of storing per-contract `CaffeineCache` instances as values inside another Caffeine cache (which leaks them into the `CaffeineCacheManager`'s internal map), use a single `Cache<EntityId, Cache<ByteBuffer, byte[]>>` backed by a plain `Caffeine` cache with a removal listener that explicitly invalidates the inner cache on eviction.
2. **Add a removal listener** on the outer `contractSlotsCache` so that when a per-contract cache is evicted, it is also removed from `cacheManagerSlotsPerContract` (via `cacheManagerSlotsPerContract.getCache(name).invalidate()` or equivalent).
3. **Alternatively**, replace the two-level structure with a single flat cache keyed by `(contractId, slotKey)` with a global `maximumSize` bound, eliminating the per-contract `CaffeineCacheManager` entirely.
4. **Add rate limiting** on the web3 endpoint to limit the number of distinct contract IDs a single client can query per time window.

### Proof of Concept

```python
import requests, threading

NODE_URL = "http://<mirror-node-web3>:8545"

def spray_contract_id(start, count):
    for i in range(start, start + count):
        # eth_getStorageAt with a synthetic contract address derived from entity ID i
        contract_addr = "0x" + hex(i)[2:].zfill(40)
        payload = {
            "jsonrpc": "2.0", "method": "eth_getStorageAt",
            "params": [contract_addr, "0x0", "latest"], "id": i
        }
        requests.post(NODE_URL, json=payload, timeout=5)

# Launch 50 threads, each hitting 10,000 distinct contract IDs
threads = [threading.Thread(target=spray_contract_id, args=(i*10000, 10000))
           for i in range(50)]
for t in threads: t.start()
for t in threads: t.join()
# After 500,000 unique contract IDs, cacheManagerSlotsPerContract holds
# 500,000 CaffeineCache instances — never evicted — causing heap exhaustion.
```

**Expected result:** JVM heap fills with `CaffeineCache` instances held by `cacheManagerSlotsPerContract`'s internal `ConcurrentHashMap`. GC overhead increases, eventually triggering `OutOfMemoryError` or sustained GC pauses degrading the node.
