### Title
Unvalidated Future Timestamp in `getSupply` Causes Guaranteed `EntityNotFoundException` for Any Unprivileged User

### Summary
The `getSupply(Bound timestamp)` method in `NetworkServiceImpl` accepts user-supplied timestamps without validating that they do not exceed the current time. Supplying a far-future timestamp (e.g., year 2100) causes `getFirstDayOfMonth` to compute an `optimalLowerBound` entirely in the future, which is then passed to `accountBalanceRepository.getSupplyHistory()`. Because no balance records exist for future timestamps, the query returns `consensusTimestamp=0`, unconditionally triggering `EntityNotFoundException`. Any unauthenticated HTTP client can reproduce this deterministically.

### Finding Description
**Code path:**

- `NetworkController.java` line 133–136: The `getSupply` endpoint accepts `TimestampParameter[] timestamp` with only `@Size(max = 2)` — no constraint on the actual timestamp value.
- `NetworkServiceImpl.java` line 69–81: When a timestamp bound is provided, `maxTimestamp` is derived from user input with no upper-bound cap against the current time.
- Line 77: `optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1)` — for a year-2100 input this produces a nanosecond value ~75 years in the future.
- Line 78: `minTimestamp = Math.max(minTimestamp, optimalLowerBound)` — the lower bound is now also in the future.
- Line 81: `accountBalanceRepository.getSupplyHistory(lowerBounds, upperBounds, minTimestamp, maxTimestamp)` — the SQL `WHERE ab.consensus_timestamp BETWEEN :lowerTimestamp AND :upperTimestamp` matches zero rows; `COALESCE(MAX(consensus_timestamp), 0)` returns `0`.
- Lines 84–86: `if (networkSupply.consensusTimestamp() == 0L) { throw new EntityNotFoundException("Network supply not found"); }` — exception is thrown unconditionally.

**Root cause:** The only existing range check (line 73–75) verifies `minTimestamp > maxTimestamp` (inverted range), but never checks whether `maxTimestamp` exceeds `Instant.now()`. The implicit assumption that callers supply past-or-present timestamps is never enforced.

### Impact Explanation
Every request to `GET /api/v1/network/supply?timestamp=lte:<future_ns>` returns an HTTP 404/500 `EntityNotFoundException`. This makes the supply monitoring endpoint unreliable for any consumer that passes a timestamp parameter. Because the endpoint is public and unauthenticated, the attack surface is the entire internet. Repeated or automated requests with future timestamps can sustain a persistent error state for this endpoint, degrading observability of network supply data.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and a single HTTP GET request. The parameter format (`timestamp=lte:4102444800.000000000`) is documented in the public API. The attack is trivially scriptable and fully repeatable. Any external party — including automated scanners — can trigger it.

### Recommendation
In `NetworkServiceImpl.getSupply`, cap `maxTimestamp` to the current time before using it:

```java
final long now = DomainUtils.convertToNanosMax(Instant.now());
final var maxTimestamp = Math.min(timestamp.adjustUpperBound(), now);
```

Apply the same cap to `minTimestamp`. Alternatively, add a Jakarta `@Max` or custom validator on `TimestampParameter` at the controller layer to reject any timestamp value exceeding `Instant.now()` at parse time.

### Proof of Concept
```
# Year 2100 in seconds: 4102444800
# As a nanosecond timestamp string: 4102444800.000000000

curl "https://<mirror-node-host>/api/v1/network/supply?timestamp=lte:4102444800.000000000"

# Expected result: HTTP 404 with EntityNotFoundException "Network supply not found"
# Actual DB query runs: WHERE consensus_timestamp BETWEEN <future_ns> AND <future_ns>
# → zero rows → consensusTimestamp=0 → exception thrown
```

    private long getFirstDayOfMonth(long timestamp, int monthOffset) {
        final var instant = Instant.ofEpochSecond(0, timestamp);
        final var dateTime = instant.atZone(ZoneOffset.UTC);
        final var firstDay = dateTime.plusMonths(monthOffset).withDayOfMonth(1);

        return firstDay.toLocalDate().atStartOfDay(ZoneOffset.UTC).toEpochSecond() * DomainUtils.NANOS_PER_SECOND;
    }
