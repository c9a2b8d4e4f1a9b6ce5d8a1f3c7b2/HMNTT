### Title
Unauthenticated DoS via Repeated Timestamp-Filtered Historical UNION ALL Queries in `getOneAccount()`

### Summary
The `getOneAccount()` handler in `rest/accounts.js` unconditionally triggers multiple expensive historical `UNION ALL` queries against `entity`/`entity_history` and `entity_stake`/`entity_stake_history` tables whenever a `timestamp` query parameter is supplied. The REST API has no per-IP or global rate limiting middleware, and the default DB connection pool is only 10 connections with a 20-second statement timeout. An unauthenticated attacker can saturate the pool with long-running historical queries, starving all other DB operations including those needed for transaction confirmation monitoring.

### Finding Description

**Exact code path:**

In `rest/accounts.js`, `getOneAccount()` (lines 396–512):

1. **Lines 428–465** — When `timestampFilters.length > 0`, the function first issues a preliminary DB query via `balances.getAccountBalanceTimestampRange()`. If no balance snapshot is found for the supplied timestamp (e.g., `timestamp=lte:1`), `accountBalanceQuery.forceUnionEntityHistory = true` is set (line 463).

2. **Lines 164 and 204–219** — In `getAccountQuery()`, `needHistoricalBalanceInfo` is true whenever `accountBalanceQuery.query` is non-empty **or** `accountBalanceQuery.forceUnionEntityHistory` is true. This causes `entityTable` to be built as:
   ```sql
   (select * from entity as e where <condition>
    union all
    select * from entity_history as e where <condition>
    order by timestamp_range desc limit 1)
   ```

3. **Lines 18–31** — `getEntityStakeQuery()` also emits a `UNION ALL` between `entity_stake` and `entity_stake_history` when `isHistorical=true` (passed at line 484).

4. **Lines 487–495** — Both the entity query and a full `doGetTransactions()` call are fired in parallel via `Promise.all()`.

**Root cause — no rate limiting on the REST API:**

`rest/server.js` (lines 67–98) shows the complete middleware stack: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`. There is **no rate-limiting middleware**. A `grep` across all `rest/**/*.js` for `rateLimit`, `throttle`, `express-rate`, or `limiter` returns only a single hit in test utilities — confirming zero production rate limiting.

`rest/middleware/authHandler.js` (lines 15–36) only sets a custom response-size `limit` for authenticated users; it does not throttle request rates.

The `ThrottleConfiguration` / `ThrottleManagerImpl` classes exist only in the `web3` module (contract calls) and are entirely absent from the REST Node.js service.

**DB pool constraint:**

`rest/dbpool.js` (line 14) sets `max: config.db.pool.maxConnections`, which defaults to **10** connections (documented in `docs/configuration.md` line 556) with a `statementTimeout` of **20,000 ms** (line 557). With 10 connections each capable of holding a 20-second query, an attacker needs only 10 concurrent requests to fully saturate the pool.

### Impact Explanation

With the DB connection pool exhausted (10 connections × up to 20 s each), all other REST API queries queue or time out. The mirror node's ability to serve transaction-confirmation queries is blocked. Because the mirror node is the authoritative read layer for Hedera network state, sustained pool saturation prevents clients and network tooling from confirming new transactions, constituting a practical network-level denial of service for the read path. Severity is **Critical** given the zero-privilege requirement and the small default pool size.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond the public API spec, and no prior account enumeration — `0.0.1` (or `0.0.2`, the treasury) is always a valid account. The trigger is a single query parameter (`?timestamp=lte:1`) that reliably forces the `forceUnionEntityHistory` path. The attack is trivially scriptable with `curl` or any HTTP load tool (e.g., `ab`, `wrk`, `hey`) and is fully repeatable. No CAPTCHA, no IP blocking, no backpressure mechanism exists in the REST service.

### Recommendation

1. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit` with a per-IP sliding window) specifically for endpoints that accept `timestamp` filters, before any DB query is issued.
2. **Add a global concurrency limiter** (e.g., `express-slow-down` or a semaphore) to cap the number of simultaneous in-flight historical queries.
3. **Increase the DB pool's `statement_timeout`** for historical queries or use a separate, smaller pool for historical/expensive queries to prevent starvation of the primary pool.
4. **Validate and bound the timestamp range** before executing the historical path — reject or short-circuit requests whose timestamp is so far in the past that no useful data can exist (e.g., before network genesis).
5. Consider **caching** responses for identical `(accountId, timestamp)` pairs in Redis (the infrastructure already exists per `server.js` lines 94–98) to avoid re-executing the same expensive query.

### Proof of Concept

```bash
# Step 1: Confirm the historical path is triggered (single request)
curl "https://<mirror-node>/api/v1/accounts/0.0.1?timestamp=lte:1"
# Returns account data (or 404), confirming the UNION ALL path executed.

# Step 2: Saturate the DB pool (10 concurrent, sustained)
# Using GNU parallel or wrk:
wrk -t10 -c10 -d60s \
  "https://<mirror-node>/api/v1/accounts/0.0.1?timestamp=lte:1"

# Step 3: In a separate terminal, observe that legitimate queries now time out:
curl "https://<mirror-node>/api/v1/transactions?limit=1"
# Expected: connection timeout or 503 after pool exhaustion.
```

Each of the 10 concurrent requests forces: (1) a `getAccountBalanceTimestampRange` DB query, (2) a `UNION ALL` entity/entity_history query, (3) a `UNION ALL` entity_stake/entity_stake_history query, and (4) a `doGetTransactions` query — all holding a DB connection for up to 20 seconds, fully blocking the pool.
