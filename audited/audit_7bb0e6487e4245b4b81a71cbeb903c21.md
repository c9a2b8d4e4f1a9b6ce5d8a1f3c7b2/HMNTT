### Title
Unauthenticated DoS via Missing Rate Limiting on `/blocks` Endpoint Exhausts DB Connection Pool

### Summary
The `/api/v1/blocks` endpoint in `rest/controllers/blockController.js` has no application-level or infrastructure-level rate limiting. Any unauthenticated external user can flood the endpoint with concurrent requests, each triggering a synchronous `RecordFileService.getBlocks()` database query, exhausting the PostgreSQL connection pool and degrading service availability across all mirror node REST API consumers. The Traefik middleware configured for the REST service explicitly omits the `rateLimit` and `inFlightReq` controls that are present in other services (e.g., the Rosetta service).

### Finding Description
**Code path:**

In `rest/controllers/blockController.js` lines 101–112, `getBlocks()` unconditionally calls `RecordFileService.getBlocks(formattedFilters)` for every incoming request with no concurrency guard:

```js
getBlocks = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedBlockParameters);
  const formattedFilters = this.extractSqlFromBlockFilters(filters);
  const blocks = await RecordFileService.getBlocks(formattedFilters);
  ...
};
```

**Middleware stack (rest/server.js lines 68–144):**
The full middleware chain is: `urlencoded` → `json` → `cors` → optional `compression` → `httpContext` → `requestLogger` → `authHandler` → optional `metricsHandler` → optional `responseCacheCheckHandler` → routes → `responseHandler` → optional `responseCacheUpdateHandler` → `handleError`. There is no rate-limiting or in-flight request cap middleware anywhere in this chain.

**`authHandler` is insufficient (rest/middleware/authHandler.js lines 15–20):**
Unauthenticated requests (no `Authorization` header) immediately `return` and proceed to the route handler. The handler only sets a per-user response *result-set size* limit for authenticated users — it does not throttle request frequency for anyone.

**Infrastructure-level controls absent (charts/hedera-mirror-rest/values.yaml lines 134–139):**
The Traefik middleware for the REST service is:
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 10
      initialInterval: 100ms
```
There is no `rateLimit` or `inFlightReq` entry. By contrast, the Rosetta chart (`charts/hedera-mirror-rosetta/values.yaml` lines 149–166) explicitly configures both:
```yaml
  - inFlightReq:
      amount: 5
  - rateLimit:
      average: 10
```

**The `retry: attempts: 10` entry is an amplifier:** Traefik will retry each failing request up to 10 times, multiplying attacker-originated DB load by up to 10×.

**Response cache is not a reliable mitigation (rest/server.js lines 54, 95–98):**
```js
const applicationCacheEnabled = config.cache.response.enabled && config.redis.enabled;
```
The cache requires both Redis and the response cache feature to be explicitly enabled. In default or Redis-less deployments it is off, meaning every request hits the DB.

### Impact Explanation
With no rate limiting, an attacker can open thousands of concurrent HTTP connections to `GET /api/v1/blocks` (no authentication required). Each connection acquires a PostgreSQL connection from the pool (`rest/dbpool.js`). Once the pool is saturated, all subsequent queries — including those from other endpoints (transactions, accounts, contracts) — queue or fail. This degrades the mirror node REST API for all consumers. Because mirror node REST APIs are used by network tooling, explorers, and dApps to observe network state, sustained pool exhaustion constitutes degradation of ≥30% of network processing visibility without requiring any brute-force credential attack.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a standard HTTP client capable of sending concurrent requests (e.g., `wrk`, `ab`, `hey`, or a simple async script). The endpoint is publicly reachable via the ingress. The absence of `inFlightReq` and `rateLimit` in the Traefik middleware — while those controls exist in the Rosetta chart — indicates this is a configuration gap, not a deliberate design choice. The attack is trivially repeatable and automatable.

### Recommendation
1. **Add `inFlightReq` and `rateLimit` to `charts/hedera-mirror-rest/values.yaml`** mirroring the Rosetta chart configuration (lines 149–166 of `charts/hedera-mirror-rosetta/values.yaml`).
2. **Add application-level rate limiting** in `rest/server.js` using `express-rate-limit` (or equivalent) applied globally before route handlers, so it is effective even without Traefik.
3. **Reduce `retry.attempts`** from 10 to 2–3 to prevent Traefik from amplifying load during degradation events.
4. **Ensure response caching is enabled** in production deployments to serve repeated identical queries from Redis rather than the DB.

### Proof of Concept
