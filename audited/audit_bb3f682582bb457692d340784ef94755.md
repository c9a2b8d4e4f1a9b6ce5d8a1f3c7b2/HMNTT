### Title
Unbounded Connection Multiplier: No Global Connection Limit Allows Per-Connection Stream Cap Bypass via Connection Flooding

### Summary
`grpcServerConfigurer()` configures only `maxConcurrentCallsPerConnection=5` on the Netty server, which is a per-connection limit with no corresponding global connection cap. Because no `maxConnections()` is set on the `NettyServerBuilder`, an unauthenticated attacker can open an unbounded number of TCP connections, each carrying 5 active gRPC streams, multiplying total concurrent calls without bound and exhausting the server's thread pool, HikariCP database connection pool, and heap memory.

### Finding Description

**Exact code path:**

`GrpcConfiguration.java` lines 28–35 configure the Netty gRPC server:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`NettyProperties.java` line 14 defines the only tunable:

```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `NettyServerBuilder.maxConcurrentCallsPerConnection(5)` enforces a per-HTTP/2-connection stream limit. Netty's gRPC server has no default global connection cap. The `grpcServerConfigurer` bean never calls `serverBuilder.maxConnections(...)`, `serverBuilder.maxConnectionAge(...)`, or any equivalent. The failed assumption is that limiting streams per connection limits total server load — it does not, because the number of connections is unbounded.

**No authentication barrier:** The only registered `ServerInterceptor` is `GrpcInterceptor` (test-only, sets endpoint context). No authentication, TLS client certificate, or IP-based admission control is applied to the gRPC port (5600). Any anonymous client on the network can open connections.

**No rate limiting:** `ThrottleConfiguration` (bucket4j rate limiter) exists only in the `web3` module. The `grpc` module has no equivalent.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 (no limit enforced).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streaming calls (the per-connection maximum).
3. Total concurrent active gRPC calls = N × 5, unbounded by any server-side control.
4. Each `subscribeTopic` call allocates a `TopicContext` object, acquires a HikariCP DB connection for historical retrieval, and occupies a thread from `applicationTaskExecutor` for reactive processing.
5. At sufficient N, the shared `applicationTaskExecutor` thread pool saturates, the HikariCP pool is exhausted (blocking all legitimate DB access), and heap fills with `TopicContext` / Reactor subscription state.

### Impact Explanation

The mirror node's gRPC subscription pipeline — `TopicMessageServiceImpl.subscribeTopic()` → `TopicMessageRetriever` → HikariCP → PostgreSQL — is fully shared across all connections. Exhausting the thread pool or DB connection pool causes all legitimate `subscribeTopic` and `getNodes` calls to queue indefinitely or fail. This constitutes a complete denial of service of the gRPC API with no ability for legitimate clients to obtain topic message data. Because the gRPC mirror node is a read-only service, this does not directly tamper with consensus transactions, but it denies all downstream consumers (wallets, dApps, SDKs) access to HCS topic message streams, which is the primary function of this service.

### Likelihood Explanation

Preconditions are minimal: network reachability to port 5600 and a standard gRPC client library. No credentials, tokens, or special protocol knowledge are required. The attack is trivially scriptable using any gRPC client (e.g., `grpcurl`, the Java/Go/Python gRPC SDK). A single attacker machine with modest resources (thousands of TCP connections is well within OS limits) can sustain the attack indefinitely. The attack is repeatable and requires no prior reconnaissance beyond discovering the open port.

### Recommendation

1. **Add a global connection limit** in `grpcServerConfigurer()`:
   ```java
   serverBuilder.maxConnections(nettyProperties.getMaxConnections()); // e.g., 1000
   ```
   Add `maxConnections` to `NettyProperties` with a safe default.

2. **Add connection age / idle limits** to reclaim resources from long-lived idle connections:
   ```java
   serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
   ```

3. **Add per-IP connection rate limiting** at the infrastructure layer (Kubernetes `NetworkPolicy`, Traefik/Envoy rate limiting, or a Netty `ChannelHandler`).

4. **Enforce a global concurrent-call semaphore** via a gRPC `ServerInterceptor` that rejects calls when a configurable global ceiling is reached.

```java
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
```
