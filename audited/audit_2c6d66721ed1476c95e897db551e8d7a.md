### Title
Unbounded Timestamp Range Bypass in `optimizeTimestampFilters` Enables Full-Table-Scan DoS on `/contracts/results`

### Summary
`optimizeTimestampFilters` in `contractController.js` calls `parseTimestampFilters` with `validateRange=false`, explicitly disabling the 7-day (`maxTimestampRange`) range-size guard. The only remaining protection, `bindTimestampRange`, defaults to `false` in production config. An unauthenticated user can therefore supply a `gte:0 / lte:max` timestamp pair that passes all validation and drives an unbounded sequential scan of the `contract_result` table, exhausting DB connections and making the mirror-node REST API unavailable.

### Finding Description

**Code path:**

`getContractResults` (line 1050) → `extractContractResultsByIdQuery` (line 1067) → `optimizeTimestampFilters` (line 504-505, only when `contractId === undefined`).

`optimizeTimestampFilters` (lines 384-405, `rest/controllers/contractController.js`):

```js
const {range, eqValues, neValues} =
  utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
//                                               ^req  ^ne  ^open ^strict ^validateRange
```

The sixth argument `validateRange=false` is the critical bypass. Inside `parseTimestampFilters` (`rest/utils.js`, lines 1657-1665):

```js
if (validateRange) {                          // ← never entered from optimizeTimestampFilters
  if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
    throw new InvalidArgumentError(`...within ${maxTimestampRange}`);
  }
}
```

The 7-day (`maxTimestampRange`) guard is therefore dead code for this call site.

After `parseTimestampFilters` returns, `bindTimestampRange` is the only remaining cap:

```js
const {range: optimizedRange, next} =
  eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
```

`bindTimestampRange` (`rest/timestampRange.js`, line 20):

```js
if (!queryConfig.bindTimestampRange) {
  return {range};          // ← returns the attacker-supplied range unchanged
}
```

The documented default is `bindTimestampRange: false` (`docs/configuration.md`, line 579). With the default config both guards are inactive.

**`strictTimestampParam` does not help here.** The call passes `strictCheckOverride=false`, so `forceStrictChecks = false || config.strictTimestampParam`. Even with `strictTimestampParam=true` (default), strict checks only reject *multiple* lower-bound or upper-bound operators. A single `gte` + single `lte` pair passes strict checks while still spanning an arbitrary range.

The resulting SQL (built in `ContractService.getContractResultsByIdAndFiltersQuery`) becomes:

```sql
SELECT ... FROM contract_result cr
LEFT JOIN entity e ON e.id = cr.contract_id
WHERE cr.consensus_timestamp >= 0
  AND cr.consensus_timestamp <= 9999999999999999999
  AND cr.transaction_nonce = 0
ORDER BY cr.consensus_timestamp DESC
LIMIT 25;
```

Without a tight index range, PostgreSQL performs a sequential scan or a very wide index range scan across the entire partitioned `contract_result` table.

### Impact Explanation

Each such request holds a DB connection for up to `statementTimeout` (default 20 s) while scanning the full table. With a pool of only 10 connections (default `maxConnections`), five concurrent attacker requests saturate the pool. All legitimate API calls queue or time out, rendering the mirror-node REST API unavailable. Because the mirror node is the primary read interface for dApps and wallets querying contract execution data, sustained unavailability degrades the usability of the network for end users and operators.

### Likelihood Explanation

The endpoint `/api/v1/contracts/results` requires no authentication. The exploit is a single HTTP GET request with two well-formed timestamp query parameters. No special knowledge, credentials, or tooling is required. The attack is trivially repeatable and scriptable, and the default `bindTimestampRange=false` means most deployments that have not explicitly opted in to the mitigation are vulnerable.

### Recommendation

1. **Enable `bindTimestampRange` by default** or change the default to `true` in `application.yml` / the shipped config.
2. **Pass `validateRange=true`** in the `optimizeTimestampFilters` call to `parseTimestampFilters`, or add an explicit range-size check before calling `bindTimestampRange`:
   ```js
   // rest/controllers/contractController.js, line 387
   const {range, eqValues, neValues} =
     utils.parseTimestampFilters(timestampFilters, false, true, true, false, true); // validateRange=true
   ```
3. Add rate-limiting middleware on `/contracts/results` to bound concurrent expensive queries per source IP.

### Proof of Concept

```bash
# Single request – full-table scan, holds DB connection for up to statementTimeout
curl "https://<mirror-node>/api/v1/contracts/results?timestamp=gte:0.000000000&timestamp=lte:9999999999.999999999"

# Five concurrent requests saturate the default 10-connection pool
for i in $(seq 1 5); do
  curl -s "https://<mirror-node>/api/v1/contracts/results?timestamp=gte:0.000000000&timestamp=lte:9999999999.999999999" &
done
wait
# Subsequent legitimate requests will queue or return 503/timeout
```

**Preconditions:** No authentication. Default config (`bindTimestampRange=false`).
**Trigger:** Two timestamp query parameters with a maximally wide range.
**Result:** DB connection pool exhaustion; mirror-node REST API unavailable for the duration of the attack.
