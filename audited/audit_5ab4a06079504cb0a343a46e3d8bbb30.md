### Title
Unauthenticated Connection Pool Exhaustion via `account.publickey` + `timestamp` Parameter Combination in `/api/v1/balances`

### Summary
An unprivileged external user can trigger multiple sequential PostgreSQL queries per HTTP request by supplying both a valid `timestamp` and `account.publickey` query parameter to the `/api/v1/balances` endpoint. With no application-level rate limiting on the REST API, flooding this endpoint with concurrent requests exhausts the finite `pg` connection pool, denying service to all other API consumers.

### Finding Description

**Exact code path:**

In `rest/balances.js`, `getBalances()` (line 83) calls `getAccountIdsByPublicKey()` (line 119) only when `tsQuery` is truthy — i.e., when a `timestamp` filter is present:

```js
// rest/balances.js line 113-122
if (tsQuery) {
  const tsQueryResult = await getTsQuery(tsQuery, tsParams);   // → pool.queryQuietly #1
  if (!tsQueryResult.query) { return; }

  const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit); // → pool.queryQuietly #2
  ...
}
// rest/balances.js line 153
const result = await pool.queryQuietly(pgSqlQuery, sqlParams); // → pool.queryQuietly #3
```

`getAccountIdsByPublicKey()` at lines 60–76:

```js
const entityPublicKeyQuery = `select id from entity where type in ('ACCOUNT', 'CONTRACT') and public_key = $1 limit $2`;

const getAccountIdsByPublicKey = async (publicKey, limit) => {
  if (isEmpty(publicKey)) { return null; }
  const params = [...publicKey, limit];
  const result = await pool.queryQuietly(entityPublicKeyQuery, params);
  ...
};
```

Each request with both `timestamp` and `account.publickey` parameters issues **three sequential** `pool.queryQuietly` calls. Because they are sequential `await`s, each request holds one pool connection for the entire duration of all three queries.

**Root cause:** No application-level rate limiting exists in the REST API. `rest/server.js` (lines 68–98) registers only `cors`, `compression`, `requestLogger`, `authHandler`, and optional `metricsHandler` — no rate-limit middleware. The `rateLimit`/`throttle` search across `rest/**` returns zero hits in application code. The only throttling found is in the `web3` Java service (`ThrottleConfiguration.java`), which is a separate component.

**Why validation checks are insufficient:**

- `utils.validateReq(req, acceptedBalancesParameters, balanceFilterValidator)` (line 84) validates parameter format only — it does not limit request rate.
- `isValidPublicKeyQuery` enforces a regex (`/^(0x)?([0-9a-fA-F]{64}|[0-9a-fA-F]{66}|[0-9a-fA-F]{88})$/`) but any attacker can trivially supply a valid 64-hex-char string.
- `statement_timeout` (configured in `rest/dbpool.js` line 15) limits individual query duration but does not prevent N concurrent requests each holding a connection for the timeout window.
- The pool `max` is bounded by `config.db.pool.maxConnections` (`rest/dbpool.js` line 14); the Helm chart sets `mirror_rest` to `max_user_connections: 250` at the pgbouncer layer — a finite ceiling.

### Impact Explanation

When the pool is saturated, `pool.queryQuietly` calls from all other endpoints (accounts, transactions, tokens, etc.) queue behind `connectionTimeoutMillis`. Once that timeout expires, requests fail with a connection acquisition error, returning 500 errors to all API consumers. This is a complete, non-network-based denial of service affecting the entire REST API surface, not just the `/balances` endpoint.

### Likelihood Explanation

- **No authentication required** — the `/api/v1/balances` endpoint is fully public.
- **Trivial preconditions** — attacker needs only a valid 64-char hex string (e.g., `0000...0000`) and any valid timestamp value.
- **Repeatable** — the attack can be sustained indefinitely with a simple HTTP flood tool (e.g., `ab`, `wrk`, `hey`).
- **Low cost** — each request is a GET with two query parameters; no body, no session, no token.
- Infrastructure-level Traefik middleware (`charts/hedera-mirror-rest/templates/middleware.yaml`) may provide circuit-breaking in production deployments, but this is optional (`{{ if and .Values.global.middleware .Values.middleware }}`) and not guaranteed to be enabled.

### Recommendation

1. **Add application-level rate limiting** to the REST API server (e.g., `express-rate-limit` keyed by IP) before the route handlers in `rest/server.js`.
2. **Decouple the secondary query** — when `account.publickey` is provided alongside `timestamp`, consider joining the `entity` table inline in the main balance query rather than issuing a separate pre-query, eliminating the extra pool acquisition.
3. **Enforce a per-IP connection/request concurrency limit** at the application layer, independent of infrastructure middleware.
4. **Ensure Traefik rate-limit middleware is mandatory** (not optional) in all deployment configurations.

### Proof of Concept

```bash
