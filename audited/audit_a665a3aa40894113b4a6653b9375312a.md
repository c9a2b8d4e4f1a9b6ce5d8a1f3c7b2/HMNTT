### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent `getSupply()` Queries on `/api/v1/network/supply`

### Summary
The `/api/v1/network/supply` endpoint in the `rest-java` module has no rate limiting, no response caching, and no authentication. Every request unconditionally executes a native SQL aggregation query (`unnest`+join+`sum`/`max` over the `entity` table) that holds a HikariCP connection for its full duration. With a default pool of 10 connections and a 10-second statement timeout, an unprivileged attacker sending sustained concurrent requests can exhaust the pool and cause all subsequent database-backed requests across the mirror node instance to queue-timeout and fail.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (lines 132–150) receives every unauthenticated GET to `/api/v1/network/supply` and unconditionally calls `networkService.getSupply(bound)`. [1](#0-0) 

`NetworkServiceImpl.getSupply()` (lines 59–89), when no timestamp is supplied (the common case), calls `entityRepository.getSupply(lowerBounds, upperBounds)` with no caching or short-circuit. [2](#0-1) 

`EntityRepository.getSupply()` (lines 19–29) executes a native SQL query that performs a full `entity` table scan with `unnest`+join, `sum(balance)`, and `max(balance_timestamp)` — a non-trivial aggregation that holds a database connection for its entire execution. [3](#0-2) 

**Root cause — three absent controls:**

1. **No rate limiting in `rest-java`**: The bucket4j-based `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` exist exclusively in the `web3` module. No equivalent filter, interceptor, or annotation exists anywhere in `rest-java`. [4](#0-3) 

2. **No response caching**: There is no `@Cacheable` or any other caching annotation on `NetworkServiceImpl.getSupply()` or `EntityRepository.getSupply()`. Every HTTP request triggers a fresh database round-trip. [5](#0-4) 

3. **Small HikariCP pool with long statement timeout**: The `rest-java` module inherits the shared `CommonConfiguration` HikariCP setup with no explicit `maximumPoolSize` override (HikariCP default: **10 connections**). The documented `statementTimeout` default is **10,000 ms**, meaning each connection can be held for up to 10 seconds per query. [6](#0-5) [7](#0-6) 

**Why the only apparent safeguard fails**: The `@Size(max = 2)` on the `timestamp` parameter only limits the number of timestamp query parameters; it does not limit request concurrency or query cost. [8](#0-7) 

### Impact Explanation
With 10 pool connections and a 10-second statement timeout, an attacker sustaining ≥10 concurrent requests keeps the pool fully occupied. All other database-backed endpoints (accounts, transactions, tokens, etc.) on the same instance queue for a HikariCP connection and, after the `connectionTimeout` elapses, throw `SQLTimeoutException`, returning 500 errors to all users. If multiple mirror node instances are targeted simultaneously (trivially achievable from a single machine with many concurrent HTTP connections), ≥30% of instances can be rendered non-functional for the duration of the attack, satisfying the stated severity threshold.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero special knowledge beyond the public API documentation. The endpoint is listed in the OpenAPI spec and exercised by public monitoring scripts. A single attacker machine with a tool like `wrk`, `hey`, or `ab` can sustain thousands of concurrent connections. The attack is repeatable indefinitely and requires no brute force. [9](#0-8) 

### Recommendation
1. **Add rate limiting to `rest-java`**: Port the existing bucket4j `ThrottleConfiguration` pattern from `web3` to `rest-java`, applying a per-IP or global request-per-second cap on `/api/v1/network/supply`.
2. **Cache the supply result**: Annotate `NetworkServiceImpl.getSupply()` (no-timestamp path) with `@Cacheable` with a short TTL (e.g., 5–15 seconds). The supply value changes at most once per consensus round.
3. **Increase pool size or add a semaphore**: Raise `spring.datasource.hikari.maximumPoolSize` and add a `@Bulkhead` or `Semaphore` to cap concurrent in-flight supply queries independently of the shared pool.
4. **Deploy an API gateway / WAF** in front of `rest-java` with per-IP rate limiting as a defense-in-depth layer.

    public NetworkSupply getSupply(Bound timestamp) {
        final NetworkSupply networkSupply;

        final var bounds = networkProperties.getUnreleasedSupplyRangeBounds();
        final var lowerBounds = bounds.lowerBounds();
        final var upperBounds = bounds.upperBounds();

        if (timestamp.isEmpty()) {
            networkSupply = entityRepository.getSupply(lowerBounds, upperBounds);
        } else {
            var minTimestamp = timestamp.getAdjustedLowerRangeValue();
            final var maxTimestamp = timestamp.adjustUpperBound();

            // Validate timestamp range
            if (minTimestamp > maxTimestamp) {
                throw new IllegalArgumentException("Invalid range provided for timestamp");
            }

            final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1);
            minTimestamp = Math.max(minTimestamp, optimalLowerBound);

            networkSupply =
                    accountBalanceRepository.getSupplyHistory(lowerBounds, upperBounds, minTimestamp, maxTimestamp);
        }

        if (networkSupply.consensusTimestamp() == 0L) {
            throw new EntityNotFoundException("Network supply not found");
        }

        return networkSupply;
    }
