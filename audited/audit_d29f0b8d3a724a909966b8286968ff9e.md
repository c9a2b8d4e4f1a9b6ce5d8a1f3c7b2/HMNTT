### Title
Unauthenticated Timestamp Filter Triggers Expensive UNION ALL Entity History Query with No Rate Limiting

### Summary
Any unauthenticated external user can supply a `?timestamp=` parameter to `GET /api/v1/accounts/{id}` that causes `getOneAccount()` to set `accountBalanceQuery.forceUnionEntityHistory = true`, which forces `getEntityBalanceQuery()` to build an expensive `UNION ALL` of `entity` and `entity_history` tables for every request. Because the REST API layer has no rate limiting, an attacker can flood this endpoint with concurrent requests, exhausting the DB connection pool and significantly degrading both the REST API and the importer's DB access.

### Finding Description

**Exact code path:**

In `rest/accounts.js`, `getOneAccount()` (lines 428–465): when `timestampFilters.length > 0`, it calls `balances.getAccountBalanceTimestampRange(...)`. If no balance snapshot is found for the supplied timestamp (e.g., a timestamp far in the past like `0.000000001`), `upper` is `undefined` and the code falls into the `else` branch:

```js
// rest/accounts.js line 460-464
} else {
  accountBalanceQuery.forceUnionEntityHistory = true;
  tokenBalanceQuery.query = '';
}
```

This propagates to `getEntityBalanceQuery()` at line 164:

```js
const needHistoricalBalanceInfo = accountBalanceQuery.query || accountBalanceQuery.forceUnionEntityHistory;
```

When truthy, lines 210–219 build the expensive entity table subquery:

```js
entityTable = `(
    select *
    from ${Entity.tableName} as e
    where ${whereCondition}
    union all
    select *
    from ${Entity.historyTableName} as e
    where ${whereCondition}
    order by ${Entity.TIMESTAMP_RANGE} desc limit 1
  )`;
```

**Root cause:** The `forceUnionEntityHistory` flag is set whenever `getAccountBalanceTimestampRange` returns no result, which is trivially triggered by any timestamp predating the first balance snapshot. There is no authentication requirement, no rate limiting on the REST API layer (confirmed: `grep` for `rateLimit|throttle` in `rest/**/*.js` returns zero matches), and no per-IP or per-endpoint request throttle.

**Each such request issues two DB queries:**
1. A `SELECT` against `account_balance` to find a snapshot timestamp (`balances.getAccountBalanceTimestampRange`)
2. The `UNION ALL` of `entity` + `entity_history` filtered by account ID and timestamp range

**Existing checks reviewed:**
- `statement_timeout` is configured in `rest/dbpool.js` line 15 — this kills individual long-running queries but does not prevent connection pool exhaustion from many concurrent short queries.
- `maxConnections` pool cap exists — this limits concurrent DB connections but does not prevent the attacker from holding all slots with concurrent requests.
- No HTTP-layer rate limiting exists in the REST module (the `ThrottleConfiguration`/`ThrottleManagerImpl` found is exclusively in the `web3` EVM module, not the REST API).

### Impact Explanation
An attacker flooding `GET /api/v1/accounts/0.0.2?timestamp=0.000000001` with concurrent requests forces two DB queries per request through the expensive UNION ALL path. This exhausts the `pg` connection pool, causing all REST API queries (including those from the importer's read path) to queue or fail. If the DB is sufficiently saturated, the importer's write operations are also delayed, degrading the mirror node's ability to ingest and confirm new transactions. The impact is severe REST API DoS with secondary risk to importer throughput.

### Likelihood Explanation
Exploitation requires zero privileges — the endpoint is public, the `timestamp` parameter is documented in the OpenAPI spec, and the trigger condition (timestamp before any balance snapshot) is trivially discoverable. A single attacker with a modest HTTP flood tool can sustain the attack indefinitely. The attack is repeatable and requires no special knowledge beyond the public API documentation.

### Recommendation
1. **Add rate limiting to the REST API layer** (e.g., `express-rate-limit` per IP) for all `/api/v1/accounts/:id` requests, especially those with `timestamp` filters.
2. **Short-circuit the expensive path early**: before executing the UNION ALL, validate that the requested timestamp is within a reasonable range (e.g., not older than the oldest known balance snapshot).
3. **Reduce DB query cost**: ensure composite indexes on `(id, timestamp_range)` exist for both `entity` and `entity_history` so the UNION ALL is index-only.
4. **Cap concurrent historical queries** with a semaphore or queue to prevent pool exhaustion from this specific code path.

### Proof of Concept

```bash
# Trigger forceUnionEntityHistory by supplying a timestamp before any balance snapshot exists
# Run N concurrent requests to exhaust the DB connection pool

for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.2?timestamp=0.000000001" &
done
wait
```

Each request will:
1. Call `getAccountBalanceTimestampRange` → returns `{}` (no snapshot found) → `upper === undefined`
2. Set `accountBalanceQuery.forceUnionEntityHistory = true`
3. Execute `UNION ALL` of `entity` + `entity_history` for account `0.0.2`
4. Hold a DB connection for the duration

With 200 concurrent requests and a default `maxConnections` pool, the pool is exhausted, subsequent requests queue indefinitely, and the REST API becomes unresponsive. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
