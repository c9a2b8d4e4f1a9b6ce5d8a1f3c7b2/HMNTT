### Title
Global-Only Rate Limiting Allows Unauthenticated Thread-Pool Exhaustion via Concurrent Long-Running Contract Calls

### Summary
The web3 JSON-RPC endpoint at `/api/v1/contracts/call` enforces only a **global** (server-wide) rate limit with no per-IP or per-client concurrency cap. Because each admitted request occupies a synchronous Tomcat thread for up to the full `requestTimeout` of 10 seconds, an unprivileged attacker can send a burst of requests that simultaneously saturate all available server threads, rendering the service unavailable to legitimate callers for the duration of the attack.

### Finding Description

**Code path:**

`ContractController.call()` is a standard blocking Spring MVC handler: [1](#0-0) 

Before executing, it calls `throttleManager.throttle(request)`, which performs two non-blocking `tryConsume()` checks against **shared global buckets**: [2](#0-1) 

Both buckets are single, server-wide singletons with no per-source-IP dimension: [3](#0-2) 

Default capacity is 500 requests/second globally: [4](#0-3) 

The `requestTimeout` (10 s) is enforced only at SQL statement boundaries via a `StatementInspector`, not as a hard wall-clock interrupt on the thread: [5](#0-4) 

The timeout value itself: [6](#0-5) 

**Root cause and failed assumption:** The design assumes the global RPS cap (500/s) is sufficient to bound resource consumption. It is not, because the cap limits *admission rate*, not *concurrency*. Once admitted, each request holds a Tomcat thread for up to 10 seconds. Spring Boot's embedded Tomcat defaults to 200 worker threads. An attacker who sends 200 requests within a single 1-second window consumes all 200 tokens from the rate-limit bucket and simultaneously occupies all 200 threads for the full timeout window.

**Why the gas throttle does not help:** The gas bucket is also global and checked at admission time only. After `tryConsume()` succeeds, the EVM execution runs to completion (or timeout) on the held thread regardless of gas consumed.

### Impact Explanation
With all 200 Tomcat threads occupied for 10 seconds, Tomcat's accept queue (default 100) fills and new TCP connections are refused. The web3 API becomes completely unavailable to legitimate users for the duration of the attack. Because the attacker can continuously re-flood at the start of each new 1-second refill window, the outage can be sustained indefinitely. This prevents any user from submitting or simulating contract calls through the web3 interface.

### Likelihood Explanation
No authentication, API key, or account is required. The attacker needs only the ability to open ~200 concurrent HTTP connections — trivially achievable from a single machine or a small botnet. The attack is repeatable every 10 seconds with no cost to the attacker. The GCP gateway `maxRatePerEndpoint: 250` is an optional, infrastructure-level control that is disabled by default (`gateway.gcp.enabled: true` but `maxRatePerEndpoint` is advisory and requires HPA changes per the comment in `values.yaml`), and it is not per-IP. [7](#0-6) 

### Recommendation
1. **Add per-IP concurrency limiting** at the application layer (e.g., a `ConcurrentHashMap<String, AtomicInteger>` keyed on client IP, checked in a servlet filter before the throttle manager).
2. **Add per-IP rate limiting** in addition to the global bucket — bucket4j supports distributed per-key buckets.
3. **Enforce a hard thread-level timeout** using an `AsyncTaskExecutor` with a `Future.get(timeout)` pattern, so the thread is released even if the EVM loop does not hit a SQL boundary.
4. **Configure Tomcat's `max-threads` and `accept-count`** explicitly and document them as security-relevant parameters.
5. **Enable and enforce the GCP gateway `maxRatePerEndpoint` per source IP** as a defence-in-depth layer.

### Proof of Concept
```bash
# Send 200 concurrent max-gas contract-call requests
# Each will hold a Tomcat thread for up to 10 seconds
for i in $(seq 1 200); do
  curl -s -X POST http://<web3-host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to":   "0x0000000000000000000000000000000000000001",
      "from": "0x0000000000000000000000000000000000000002",
      "gas":  15000000,
      "data": "0x<computationally-expensive-calldata>",
      "block":"latest"
    }' &
done
wait

# Immediately attempt a legitimate request — expect connection refused or 503
curl -v -X POST http://<web3-host>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000,"block":"latest"}'
```

Preconditions: network access to the web3 endpoint, no credentials required.
Trigger: 200 concurrent requests within one second (within the 500 RPS global budget).
Result: all Tomcat threads occupied for ~10 s; legitimate request receives HTTP 503 or TCP reset.

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
