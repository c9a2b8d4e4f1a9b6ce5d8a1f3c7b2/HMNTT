### Title
Historical Bytecode Ignored During EVM Replay in Opcodes Endpoint — Fabricated Execution Trace

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint re-executes a historical transaction on the EVM but loads the **current** contract bytecode instead of the bytecode that existed at the time of the original transaction. Because `ContractBytecodeReadableKVState.readFromDataSource()` issues a timestamp-free SQL query against the `contract` table, any contract whose `runtime_bytecode` has been updated since the original execution will produce an opcode trace that never actually occurred. Any unprivileged user can trigger this by simply calling the endpoint.

### Finding Description

**Call chain:**

`OpcodesController.getContractOpcodes()` → `OpcodeService.processOpcodeCall()` → `ContractDebugService.processOpcodeCall()` → `callContract(params, ctx)`

In `ContractDebugService.processOpcodeCall()` the historical timestamp is correctly set:

```java
ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1));
``` [1](#0-0) 

The block type is set to a specific block number (not `LATEST`) in `OpcodeServiceImpl.buildCallServiceParameters()`:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);
``` [2](#0-1) 

This means `ContractCallContext.useHistorical()` returns `true` and `ContractCallContext.getTimestamp()` returns the historical timestamp. [3](#0-2) 

**The flaw:** `ContractBytecodeReadableKVState.readFromDataSource()` never reads `ContractCallContext.getTimestamp()`. It calls:

```java
return contractRepository
        .findRuntimeBytecode(entityId.getId())
        ...
``` [4](#0-3) 

which executes the timestamp-free SQL:

```sql
select runtime_bytecode from contract where id = :contractId
``` [5](#0-4) 

This always returns the **latest** bytecode. By contrast, `ContractStorageReadableKVState.readFromDataSource()` correctly branches on the timestamp:

```java
final var timestamp = ContractCallContext.get().getTimestamp();
return timestamp
        .map(t -> contractStateService.findStorageByBlockTimestamp(...))
        .orElse(contractStateService.findStorage(...))
``` [6](#0-5) 

**Historical bytecode data exists but is never queried.** The database schema includes a `contract_history` table with a `runtime_bytecode` column and a `timestamp_range` index, populated whenever a contract is updated: [7](#0-6) 

No repository method queries `contract_history` for bytecode at a given timestamp, so the historical data is silently ignored during replay.

### Impact Explanation

An unprivileged caller receives an opcode trace that was never executed on-chain. The EVM runs the **current** bytecode against the historical storage/account state, producing a chimeric execution that mixes two different points in time. This:

- Presents a fabricated execution history as authoritative debug output.
- Can be used to deceive auditors, incident responders, or counterparties about what a contract actually did at a given moment.
- Undermines the integrity of the debug/trace infrastructure for any contract that has had its bytecode updated (via `ContractUpdateTransaction` sidecar records or migration).

Severity is **High** within the stated scope (reorganizing transaction history without direct fund theft).

### Likelihood Explanation

- **Precondition:** A contract's `runtime_bytecode` in the `contract` table must differ from what it was at the time of the target transaction. This occurs after any `ContractUpdateTransaction` that carries a sidecar bytecode record, or after a `SidecarContractMigration` run.
- **Trigger:** Any unauthenticated HTTP GET to `/api/v1/contracts/results/{txHash}/opcodes` with `Accept-Encoding: gzip`.
- **No privilege required.** The endpoint has no authentication gate beyond the throttle and the gzip header check.
- **Repeatability:** Deterministic and repeatable for any affected contract.

### Recommendation

1. Add a timestamp-aware query to `ContractRepository` (or a new `ContractHistoryRepository`) that fetches `runtime_bytecode` from `contract_history` where `timestamp_range @> :timestamp`, falling back to `contract` when no history row matches.
2. Update `ContractBytecodeReadableKVState.readFromDataSource()` to read `ContractCallContext.get().getTimestamp()` and branch identically to `ContractStorageReadableKVState` — use the historical query when a timestamp is present, the current query otherwise.
3. Add a regression test that updates a contract's bytecode, then calls the opcodes endpoint for a pre-update transaction and asserts the returned opcodes match the original bytecode's execution, not the updated one.

### Proof of Concept

1. Deploy contract `C` with bytecode `V1` (e.g., a function that returns `1`). Record the deployment transaction hash.
2. Execute transaction `T` against `C` (calling the function). Record `T`'s hash and the returned opcode trace (should show `V1` opcodes).
3. Submit a `ContractUpdateTransaction` for `C` with a sidecar record containing bytecode `V2` (e.g., a function that returns `2`). The importer writes `V2` into `contract.runtime_bytecode`.
4. Call:
   ```
   GET /api/v1/contracts/results/{T_hash}/opcodes
   Accept-Encoding: gzip
   ```
5. **Observed:** The response contains opcodes from `V2`'s execution — a trace that never occurred on-chain.
6. **Expected:** The response should contain opcodes from `V1`'s execution, matching the original transaction.
