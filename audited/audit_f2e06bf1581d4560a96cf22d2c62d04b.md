### Title
Unbounded Log-Flooding via Unauthenticated HTTP Requests Enables Disk Exhaustion DoS in `TracingMiddleware`

### Summary
`TracingMiddleware()` in `rosetta/app/middleware/trace.go` unconditionally emits a `log.Info` entry for every HTTP request whose path is not in `internalPaths`, with no application-level rate limiting. The only rate-limiting defense (Traefik middleware) is disabled by default (`global.middleware: false`) and is bypassable via direct access to the application port. An unprivileged attacker can flood the server with arbitrary HTTP requests, causing unbounded log growth that exhausts disk space and renders the node unavailable.

### Finding Description
**Exact code path:**

In `rosetta/app/middleware/trace.go`, `TracingMiddleware()` (lines 43‚Äì61):

```go
if internalPaths[path] {
    log.Debug(message)
} else {
    log.Info(message)   // ‚Üê fires for every non-internal path, unconditionally
}
```

`internalPaths` contains only three entries: `/health/liveness`, `/health/readiness`, and the metrics path (line 19). Every other path ‚Äî including invalid/garbage paths ‚Äî triggers `log.Info`.

**Root cause:** There is no rate limiting, deduplication, sampling, or log-level guard at the application layer. The middleware logs every single request at `Info` level with no throttle.

**Why existing checks fail:**

The only rate-limiting mechanism is the Traefik ingress middleware chain defined in `charts/hedera-mirror-rosetta/values.yaml` (lines 149‚Äì166), which includes `rateLimit: average: 10`. However:

1. `global.middleware` defaults to `false` (line 95 of `values.yaml`), so the entire Traefik middleware chain ‚Äî including rate limiting ‚Äî is **not applied in the default deployment**.
2. Even when enabled, the Traefik middleware only applies to traffic routed through the ingress controller. The Rosetta server binds directly on port 5700 (`rosettaConfig.Port`, `main.go` line 221). An attacker with network access to that port bypasses all Traefik controls entirely.
3. The `inFlightReq: amount: 5` Traefik setting limits concurrent in-flight requests per IP but does not prevent sequential high-rate flooding.

### Impact Explanation
An attacker who can reach port 5700 (directly or through the ingress when middleware is disabled) can send a continuous stream of HTTP requests to any arbitrary path (e.g., `GET /aaaa`). Each request produces one `log.Info` line. At high request rates (easily tens of thousands per second from a single machine), log files grow without bound. When disk space is exhausted, the operating system can no longer write logs or temporary files, causing the Rosetta node process to malfunction or crash. This takes the mirror node's Rosetta API offline, disrupting blockchain data access for the Hedera/Hiero network ‚Äî a network with market capitalization meeting the stated threshold.

### Likelihood Explanation
No authentication, credentials, or special network position are required. Any machine that can reach port 5700 (common in cloud deployments where the port is exposed, or in the default Kubernetes setup without the Traefik middleware enabled) can trigger this. The attack is trivially scriptable with `curl`, `ab`, `wrk`, or any HTTP load tool. It is repeatable and requires no prior knowledge of the application beyond the open port.

### Recommendation
1. **Application-level rate limiting**: Add a per-IP token-bucket rate limiter directly in the Go middleware stack (e.g., using `golang.org/x/time/rate`), independent of any external infrastructure.
2. **Log sampling / level guard**: For high-frequency paths, use `log.Debug` instead of `log.Info`, or implement log sampling (e.g., log 1-in-N requests at Info, rest at Debug).
3. **Log rotation with size cap**: Enforce log rotation with a maximum total disk size (e.g., via `lumberjack`) so runaway logging cannot exhaust disk.
4. **Enable middleware by default**: Change `global.middleware` default to `true` in `values.yaml` so the Traefik rate-limiting chain is active in standard deployments.
5. **Restrict direct port access**: Firewall or network-policy-restrict port 5700 so it is only reachable through the ingress controller.

### Proof of Concept
**Preconditions:** Network access to the Rosetta server on port 5700 (default deployment with `global.middleware: false`).

**Steps:**
```bash
# Flood with arbitrary-path requests; each generates one log.Info line
wrk -t 10 -c 100 -d 60s http://<rosetta-host>:5700/aaaaaaaaaaaaaaaa

# Or with curl in a tight loop:
while true; do
  curl -s http://<rosetta-host>:5700/$(cat /dev/urandom | tr -dc 'a-z' | head -c 16) &
done
```

**Result:** The log file (or stdout captured by the container runtime) grows at the rate of requests. On a typical deployment with no log rotation cap, disk space is exhausted within minutes to hours depending on available disk and request rate, causing the Rosetta node to become unavailable.
