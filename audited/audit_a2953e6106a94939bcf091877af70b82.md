### Title
Historical Hook Storage Change Endpoint Leaks Deleted/Cleared Storage Entries to Unprivileged Users

### Summary
The `getHookStorageChange()` method in `HookServiceImpl` queries the `hook_storage_change` table without filtering out deleted entries (where `value_written` is null or empty), unlike the non-historical path which explicitly uses `DeletedIsFalse`. Any unauthenticated or unprivileged caller can supply a timestamp range to the historical endpoint and receive storage change records where `deleted=true`, revealing which storage keys were cleared by the hook owner — without the owner's consent or knowledge.

### Finding Description
In `HookServiceImpl.java` lines 84–118, `getHookStorageChange()` is invoked whenever `request.isHistorical()` is true (i.e., a timestamp bound is provided). It delegates to either `hookStorageChangeRepository.findByKeyInAndTimestampBetween(...)` or `findByKeyBetweenAndTimestampBetween(...)`.

Both SQL queries in `HookStorageChangeRepository.java` (lines 15–63) explicitly compute:
```sql
(value_written is null or length(value_written) = 0) as "deleted"
```
but apply **no WHERE clause** to exclude these deleted rows. All rows — including those representing storage deletions — are returned to the caller.

Contrast this with the non-historical path in `getHookStorage()` (lines 66–79 of `HookServiceImpl.java`), which calls:
- `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse(...)`
- `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse(...)`

Both of these explicitly filter out deleted entries. The historical path has no equivalent guard.

There is no authentication or authorization check in the service layer — `entityService.lookup(request.getOwnerId())` only resolves the entity ID and does not verify the caller is the owner.

**Exploit flow:**
1. Attacker identifies a target `ownerId` and `hookId` (both are public on-chain identifiers).
2. Attacker sends a GET request to the historical hook storage endpoint with any valid timestamp range.
3. The service routes to `getHookStorageChange()`, which queries `hook_storage_change` with no deleted-entry filter.
4. The response includes entries where `deleted=true` (i.e., `value_written` is null/empty), revealing the exact storage keys that were cleared by the owner.

### Impact Explanation
An unprivileged external user can enumerate all storage keys that a hook owner has ever deleted within any queried timestamp window. This leaks operational and business-logic information (e.g., which configuration keys were revoked, which secrets were rotated, which access entries were removed) without the owner's approval. The non-historical path correctly hides this data, making the inconsistency a clear unintended disclosure.

### Likelihood Explanation
Exploitation requires no special privileges, credentials, or on-chain transactions. Any user who can reach the REST API endpoint can perform this attack. The only precondition is knowing a valid `ownerId`/`hookId` pair, which is trivially discoverable from public ledger data. The attack is fully repeatable and automatable.

### Recommendation
Add a filter to exclude deleted entries in both historical queries in `HookStorageChangeRepository.java`. Specifically, add the following condition to both native SQL queries:

```sql
AND (value_written IS NOT NULL AND length(value_written) > 0)
```

This mirrors the `DeletedIsFalse` semantics used in the non-historical `HookStorageRepository` queries and ensures consistent behavior across both code paths.

### Proof of Concept
1. Identify any `ownerId` and `hookId` with historical storage changes (e.g., from public ledger explorer).
2. Send a request to the historical hook storage endpoint:
   ```
   GET /api/v1/hooks/{ownerId}/{hookId}/storage?timestamp=gte:0&timestamp=lte:9999999999999999999
   ```
3. Observe the response: entries where `"deleted": true` are returned, revealing storage keys that were cleared by the owner.
4. Compare with the non-historical endpoint (no timestamp param) — deleted entries are absent, confirming the inconsistency. [1](#0-0) [2](#0-1) [3](#0-2)

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
