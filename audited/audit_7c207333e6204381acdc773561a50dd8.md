### Title
Infrastructure Detail Leakage via Unhandled `CannotGetJdbcConnectionException` in GraphQL Error Response

### Summary
`CustomExceptionResolver.resolveToSingleError` returns `null` for any exception that is not `IllegalStateException`, `IllegalArgumentException`, or `MirrorNodeException`. When a network partition causes Spring Data JPA to throw `CannotGetJdbcConnectionException` during `EntityRepository.findByEvmAddress`, no resolver handles it, causing Spring for GraphQL's fallback handler to surface the raw exception message — including DB hostname, port, and driver-level connection details — directly in the GraphQL `errors` array. Any unauthenticated user can read this response.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 51–54) calls `entityService.getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)`. [1](#0-0) 

`EntityServiceImpl.getByEvmAddressAndType()` (line 40) calls `entityRepository.findByEvmAddress(evmAddressBytes)`. [2](#0-1) 

`EntityRepository.findByEvmAddress` executes a native JDBC query against the `entity` table. [3](#0-2) 

When the database is unreachable, Spring Data JPA throws `org.springframework.jdbc.CannotGetJdbcConnectionException`. This exception propagates up to the GraphQL data fetcher layer, where `CustomExceptionResolver.resolveToSingleError` is invoked.

**Root cause — the `else { return null; }` branch:**

```java
} else {
    return null;   // ← CannotGetJdbcConnectionException reaches here
}
``` [4](#0-3) 

`CannotGetJdbcConnectionException` is a subclass of `org.springframework.dao.DataAccessResourceFailureException`. It is not `IllegalStateException`, `IllegalArgumentException`, or `MirrorNodeException`, so it falls through to `null`. [5](#0-4) 

**Failed assumption:** The developer assumed that returning `null` would produce a safe, generic error. It does not. When `DataFetcherExceptionResolverAdapter.resolveToSingleError` returns `null`, the adapter emits `Mono.empty()`. Spring for GraphQL's `ExceptionResolversExceptionHandler` — finding no resolver that handled the exception — falls back to graphql-java's `SimpleDataFetcherExceptionHandler`, which constructs an `ExceptionWhileDataFetching` error whose `message` field is:

```
Exception while fetching data (/account) : Unable to acquire JDBC Connection;
nested exception is org.postgresql.util.PSQLException:
Connection to 127.0.0.1:5432 refused. Check that the hostname and port are
correct and that the postmaster is accepting TCP/IP connections.
```

This string is serialized verbatim into the GraphQL response `errors[].message` field.

**No sanitization exists:** `GraphQlConfiguration` configures only query complexity/depth limits and JSON size limits — no error message sanitization. [6](#0-5) 

There are no `spring.graphql.*` properties in any application YAML that suppress or mask error messages. The other modules (`rest-java`, `web3`) explicitly guard against this with `!statusCode.is5xxServerError() ? ex.getMessage() : StringUtils.EMPTY`, but the GraphQL module has no equivalent guard.

### Impact Explanation

An attacker receives the internal database hostname/IP, port number, and JDBC driver error text in a standard HTTP 200 GraphQL response body. This constitutes **infrastructure reconnaissance** enabling targeted follow-on attacks (port scanning confirmation, connection pool exhaustion, credential-stuffing against the exposed DB endpoint). Severity: **Medium–High** (CWE-209: Information Exposure Through an Error Message). The GraphQL endpoint is unauthenticated per the README (`curl` without credentials).

### Likelihood Explanation

The precondition (DB unreachable) is not attacker-controlled in the general case, but it is realistic:
- Natural transient DB failures, rolling restarts, or maintenance windows
- An attacker can exhaust the HikariCP connection pool by flooding the endpoint with concurrent queries, inducing `CannotGetJdbcConnectionException` on demand
- The trigger query requires zero privileges and is a single valid GraphQL request

Repeatability is high once the condition is met; the response is deterministic.

### Recommendation

In `CustomExceptionResolver.resolveToSingleError`, replace the `else { return null; }` branch with an explicit catch-all that returns a sanitized, generic error:

```java
} else {
    return GraphqlErrorBuilder.newError()
            .errorType(ErrorType.INTERNAL_ERROR)
            .message("INTERNAL_ERROR")
            .path(env.getExecutionStepInfo().getPath())
            .location(env.getField().getSourceLocation())
            .build();
}
```

Additionally, add an explicit handler for `DataAccessException` (parent of `CannotGetJdbcConnectionException`) that maps it to a safe "Service Unavailable" message, mirroring the pattern used in `rest-java`'s `GenericControllerAdvice.queryTimeout`.

### Proof of Concept

1. Start the GraphQL service with a reachable DB, confirm it responds normally.
2. Simulate a network partition: `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the mirror node host, or exhaust the connection pool by sending ~50 concurrent queries.
3. Send a valid GraphQL query targeting `findByEvmAddress`:

```bash
curl -X POST http://localhost:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{evmAddress:\"0x000000000000000000000000000000000000abcd\"}){balance}}"}'
```

4. Observe the response:

```json
{
  "errors": [{
    "message": "Exception while fetching data (/account) : Unable to acquire JDBC Connection; nested exception is org.postgresql.util.PSQLException: Connection to 127.0.0.1:5432 refused.",
    "locations": [{"line":1,"column":2}],
    "path": ["account"],
    "extensions": {"classification": "INTERNAL_ERROR"}
  }],
  "data": {"account": null}
}
```

The DB host and port are exposed to the unauthenticated caller.

                    .errorType(ErrorType.ValidationError)
                    .message(ex.getMessage())
