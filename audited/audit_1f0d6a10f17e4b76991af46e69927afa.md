### Title
Non-Atomic Sequential Bucket Consumption Causes Permanent Rate Token Drain Without Gas Consumption

### Summary
In `ThrottleManagerImpl.throttle()`, the `rateLimitBucket` is consumed before the `gasLimitBucket` is checked. When the gas check fails, the already-consumed rate token is never restored — there is no rollback path. An unprivileged attacker can deliberately deplete the gas bucket and then flood the endpoint with requests that each permanently drain a rate token while consuming zero gas, exhausting the shared rate bucket for all users.

### Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, lines 37–49:

```java
if (!rateLimitBucket.tryConsume(1)) {           // line 38: rate token consumed here
    throw new ThrottleException(...);
} else if (!gasLimitBucket.tryConsume(          // line 40: gas check — can fail
        throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(...);            // line 41: rate token already gone, no rollback
}
```

**Root cause:** `rateLimitBucket.tryConsume(1)` is a destructive, non-atomic operation. Once it returns `true`, the token is gone. If the subsequent `gasLimitBucket.tryConsume(...)` returns `false`, a `ThrottleException` is thrown and the rate token is permanently lost. The `restore()` method (lines 59–64) only calls `gasLimitBucket.addTokens(tokens)` — there is no corresponding mechanism to restore rate tokens.

**Exploit flow:**

1. **Deplete the gas bucket.** Default `gasPerSecond` = 7,500,000,000; `scaleGas` divides by 10,000 → 750,000 scaled tokens. Max gas per request = 15,000,000 → 1,500 scaled tokens. The attacker needs ⌈750,000 / 1,500⌉ = 500 requests, which exactly matches the default rate limit of 500 req/s. The gas bucket can be fully drained in one second.

2. **Flood with gas-failing requests.** With the gas bucket empty, every subsequent request:
   - Passes `rateLimitBucket.tryConsume(1)` → 1 rate token permanently consumed.
   - Fails `gasLimitBucket.tryConsume(...)` → `ThrottleException` thrown.
   - No EVM execution occurs; no gas is consumed; the rate token is never returned.

3. **Result.** The shared `rateLimitBucket` (500 tokens, refills at 500/s) is continuously drained by the attacker's no-op requests. Legitimate users receive `429 Too Many Requests` even though no real work is being done.

**Why existing checks are insufficient:**
- `ContractCallService.restoreGasToBucket()` (lines 140–151) is only called after EVM execution completes; it never runs when `throttle()` itself throws.
- `ThrottleManager.restore(long gas)` only touches `gasLimitBucket.addTokens()`; there is no `restoreRate()` equivalent.
- The `rateLimitBucket` bean (lines 24–32 of `ThrottleConfiguration.java`) has no external token-addition path at all.

### Impact Explanation
The `rateLimitBucket` is a global, shared resource (default 500 req/s). By keeping the gas bucket depleted and continuously sending requests, the attacker monopolises the rate bucket, causing HTTP 429 responses for every other client. No economic damage occurs, but service availability is fully disrupted for the duration of the attack — a classic griefing/DoS scenario with no authentication barrier.

### Likelihood Explanation
The attack requires no credentials, no special headers, and no privileged access — only the ability to send HTTP requests to the `/api/v1/contracts/call` endpoint. The gas bucket can be depleted in a single second at default settings. The attack is trivially repeatable and scriptable with any HTTP client. A single attacker with a modest connection can sustain it indefinitely.

### Recommendation
Perform both bucket checks atomically before consuming either token, or restore the rate token on gas-check failure:

**Option A (preferred) — check before consuming:**
```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
long scaledGas = throttleProperties.scaleGas(request.getGas());
if (!gasLimitBucket.tryConsume(scaledGas)) {
    rateLimitBucket.addTokens(1);   // restore rate token on gas failure
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

**Option B** — check gas availability with `getAvailableTokens()` before consuming the rate token, and only consume both if both checks would pass (requires careful concurrency handling given `gasLimitBucket` uses `SYNCHRONIZED` strategy).

Also add a `restoreRate(long tokens)` method to the `ThrottleManager` interface so callers can restore rate tokens in exceptional paths.

# Step 1: Deplete the gas bucket (500 max-gas requests)
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0x","gas":15000000,"to":"0x0000000000000000000000000000000000000001"}' &

wait

# Step 2: Flood with requests — each consumes a rate token, gas check fails
# Legitimate users now receive 429 even though no EVM work is happening
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0x","gas":15000000,"to":"0x0000000000000000000000000000000000000001"}'
done
# Expected output: mix of 429 (GAS_PER_SECOND_LIMIT_EXCEEDED) and
# 429 (REQUEST_PER_SECOND_LIMIT_EXCEEDED) for all concurrent legitimate clients
