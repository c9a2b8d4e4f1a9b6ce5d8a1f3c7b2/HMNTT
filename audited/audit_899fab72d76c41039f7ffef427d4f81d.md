### Title
Missing Entity Existence Validation for Numeric Account IDs in NFT Allowance Lookup

### Summary
`EntityServiceImpl.lookup()` skips database validation when the account ID is a numeric parameter (`EntityIdNumParameter`), directly returning the `EntityId` without confirming the account exists in the `entity` table. This allows any unprivileged user to query `nft_allowance` records for arbitrary or non-existent numeric account IDs, potentially surfacing orphaned allowance records for deleted accounts.

### Finding Description
**Code path:**

In `NftAllowanceServiceImpl.getNftAllowances()` (line 28), `entityService.lookup(request.getAccountId())` is called before querying the repository: [1](#0-0) 

Inside `EntityServiceImpl.lookup()`, the switch statement handles three parameter types: [2](#0-1) 

- `EntityIdAliasParameter` → calls `entityRepository.findByAlias()` (DB lookup, throws if not found)
- `EntityIdEvmAddressParameter` → calls `entityRepository.findByEvmAddress()` (DB lookup, throws if not found)
- **`EntityIdNumParameter` → returns `Optional.of(p.id())` directly — NO database lookup, NO existence check**

The returned `EntityId` is then passed directly to `NftAllowanceRepositoryCustomImpl.findAll()`: [3](#0-2) 

Which builds a SQL condition using the unvalidated numeric ID: [4](#0-3) 

**Root cause:** The `EntityIdNumParameter` branch in the switch at line 32 of `EntityServiceImpl.java` assumes numeric IDs are always valid without verifying their existence in the `entity` table, unlike alias and EVM address paths which enforce this check. [5](#0-4) 

### Impact Explanation
An attacker can query `GET /api/v1/accounts/{numericId}/allowances/nfts` with any arbitrary numeric ID. In the common case this returns an empty list. However, if an account was previously deleted on-chain but its `nft_allowance` rows were not purged (orphaned records), those records are returned to the caller. This constitutes an information disclosure of historical/orphaned allowance state that the API is not supposed to expose for non-existent accounts. The inconsistency also undermines the API contract: alias and EVM address lookups enforce account existence, but numeric lookups do not.

### Likelihood Explanation
Exploitation requires no authentication or privileges — any HTTP client can send a crafted numeric account ID. The orphaned-record scenario depends on whether the mirror node's importer cleans up `nft_allowance` rows on account deletion, which is an operational condition. The bypass itself is unconditional and repeatable for every numeric ID request.

### Recommendation
Add a database existence check for `EntityIdNumParameter` in `EntityServiceImpl.lookup()`, consistent with the alias and EVM address branches:

```java
case EntityIdNumParameter p -> entityRepository.findById(p.id().getId()).map(e -> EntityId.of(e));
```

Alternatively, call `entityRepository.findById()` and throw `EntityNotFoundException` if the entity is absent, mirroring the behavior of the other two branches. [6](#0-5) 

### Proof of Concept
1. Identify (or guess) a numeric account ID that does not exist in the mirror node's `entity` table, e.g., `0.0.9999999`.
2. Send: `GET /api/v1/accounts/0.0.9999999/allowances/nfts`
3. Observe: the request succeeds (HTTP 200) instead of returning a 404/`EntityNotFoundException`.
4. For the orphaned-record variant: use a numeric ID of a previously deleted account that had NFT allowances. The response will include those allowance records even though the account no longer exists in the `entity` table.

```java
    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
```
