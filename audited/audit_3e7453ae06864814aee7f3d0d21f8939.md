### Title
Unauthenticated High-Frequency Requests for Non-Existent Schedule IDs Cause Uncached Full JOIN Query Execution, Exhausting DB Resources

### Summary
The `getScheduleById()` handler in `rest/schedules.js` executes a three-table JOIN query for every request, including those for non-existent schedule IDs. Because 404 (NotFoundError) responses are never cached and no rate limiting exists on this endpoint, an unprivileged attacker can flood the service with valid-format but non-existent schedule IDs, forcing repeated full DB queries that return zero rows, exhausting the database connection pool and degrading the mirror node REST service.

### Finding Description
**Code path:** `rest/schedules.js`, `getScheduleById()`, lines 115–128.

```

  utils.validateReq(req);
  const parseOptions = {allowEvmAddress: false, paramName: constants.filterKeys.SCHEDULEID};
  const scheduleId = EntityId.parseString(req.params.scheduleId, parseOptions).getEncodedId();

  const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId);  // line 120
```

The query at lines 46–64 is a three-table JOIN with GROUP BY:

```sql
select ... from schedule s
left join entity e on e.id = s.schedule_id
left join transaction_signature ts on ts.entity_id = s.schedule_id
where s.schedule_id = $1
group by s.schedule_id, e.id
```

**Root cause — three compounding failures:**

1. **No rate limiting.** A grep for `rate.?limit|throttl|rateLimit` across all `rest/**/*.js` files returns zero production middleware matches. There is no per-IP or global request throttle on this route.

2. **404 responses are never cached.** `responseCacheUpdateHandler` in `rest/middleware/responseCacheHandler.js` (line 95) only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` is true. A `NotFoundError` maps to HTTP 404 in `httpErrorHandler.js` (line 17–18), which is not a success code, so it is never written to Redis. Every subsequent request for the same non-existent ID re-executes the full DB query.

3. **No authentication required.** `authHandler.js` (lines 18–20) returns immediately when no credentials are supplied, allowing fully anonymous access.

**Exploit flow:**
- `EntityId.parseString()` validates only the *format* of the ID (e.g., `0.0.<number>`). A syntactically valid but non-existent ID such as `0.0.999999999` passes format validation and reaches `pool.queryQuietly()`.
- The three-table JOIN is executed, returns 0 rows, and `NotFoundError` is thrown.
- The 404 response is not cached.
- The attacker repeats with the same or different non-existent IDs at arbitrary frequency. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
Each attack request forces a three-table JOIN with GROUP BY against the live PostgreSQL database. The DB connection pool has a finite `maxConnections` (parsed in `config.js` lines 138–147). Under sustained flood, the pool saturates, causing legitimate requests to queue or time out. The mirror node REST API becomes unavailable or severely degraded. Because the mirror node is the primary read interface for dApps, wallets, and explorers querying the Hedera network state, its degradation constitutes a meaningful service disruption. [5](#0-4) 

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only knowledge of the public REST API path (`GET /api/v1/schedules/{scheduleId}`). The schedule ID format (`0.0.<integer>`) is publicly documented. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`, or a simple async script) can sustain thousands of requests per second. The attack is trivially repeatable and requires no special infrastructure.

### Recommendation
Apply all three mitigations:

1. **Rate limiting:** Add a per-IP rate-limiting middleware (e.g., `express-rate-limit` backed by Redis) to the `/schedules/:scheduleId` route, or globally to all REST routes.

2. **Cache negative results:** Extend `responseCacheUpdateHandler` to also cache 404 responses for a short TTL (e.g., 10–30 seconds), so repeated lookups for the same non-existent ID do not hit the database.

3. **DB-level protection:** Ensure `db.pool.statementTimeout` is set to a low value (already parsed in config) and `maxConnections` is tuned, so a flood cannot hold connections indefinitely.

### Proof of Concept
```bash
# Flood with valid-format but non-existent schedule IDs
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/schedules/0.0.$((RANDOM + 9000000))" &
done
wait
```
Each request executes the full three-table JOIN, returns 0 rows, throws `NotFoundError`, and the 404 is not cached. Sustained at high concurrency, this exhausts the DB connection pool and degrades or denies service to legitimate users.
