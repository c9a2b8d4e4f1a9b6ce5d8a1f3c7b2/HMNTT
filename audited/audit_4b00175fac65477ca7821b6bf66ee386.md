### Title
Unauthenticated `/block` Endpoint Exhausts DB Connection Pool via Concurrent Requests, Causing Sustained Memory Pressure

### Summary
The Rosetta `/block` endpoint is publicly accessible with no application-level rate limiting or concurrency control. `ConnectToDb()` configures a pool of up to 100 open connections each held for up to 20 seconds per request, and the Go server's middleware stack contains no throttling layer. An unprivileged attacker sending concurrent block-index requests can saturate the pool, forcing PostgreSQL to allocate `work_mem` per active query and driving node memory consumption well above 30% of baseline.

### Finding Description

**Code path:**

`ConnectToDb()` in `rosetta/app/db/db.go` (lines 31–33) configures the shared pool:

```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)   // default 20
sqlDb.SetConnMaxLifetime(...)                              // default 30 min
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)   // default 100
``` [1](#0-0) 

`GetDbWithContext()` in `rosetta/app/db/client.go` (lines 36–37) wraps every query in a 20-second context timeout, meaning each acquired connection can be held for up to 20 seconds:

```go
childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
return d.db.WithContext(childCtx), cancel
``` [2](#0-1) 

The `Block()` handler in `rosetta/app/services/block_service.go` (lines 47–73) is unauthenticated and issues multiple sequential DB calls per request (`RetrieveBlock`, `FindBetween`, `updateOperationAccountAlias`), each acquiring a connection from the shared pool: [3](#0-2) 

The middleware stack assembled in `rosetta/main.go` (lines 217–219) contains only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — **no rate limiting or concurrency control**: [4](#0-3) 

**Root cause:** `ConnectToDb()` sets `MaxOpenConnections = 100` with a 20-second per-query timeout and no application-level concurrency guard. The failed assumption is that infrastructure-level Traefik middleware will always be present; it is not enforced at the application layer.

**Exploit flow:**
1. Attacker sends N concurrent POST requests to `/block` (or `/block/transaction`) with sequential valid block indices.
2. Each request acquires one or more connections from the 100-connection pool and holds them for up to 20 seconds.
3. PostgreSQL allocates `work_mem` (default **512 MB** per sort/hash operation) for each active backend: [5](#0-4) 
4. With 20–30 concurrent connections each running non-trivial queries, PostgreSQL memory usage spikes far above 30% of baseline.
5. The Go process also buffers all returned transaction rows in heap memory before responding.

**Why existing checks are insufficient:**

The Traefik middleware (`inFlightReq: amount: 5` per source IP, `rateLimit: average: 10` per `requestHost`) is defined only in the Helm chart values: [6](#0-5) 

- It is **optional infrastructure**, absent in bare-metal or direct-Docker deployments.
- The `rateLimit` is scoped to `requestHost` (global across all clients), not per source IP — multiple coordinated IPs each get 5 in-flight slots, bypassing the per-IP cap.
- The `retry: attempts: 3` directive causes Traefik to re-issue failed requests, amplifying load when the server is already stressed.
- Even when Traefik is present, there is no circuit-breaker at the DB pool level; the pool itself has no per-client or per-endpoint quota.

### Impact Explanation
With `MaxOpenConnections = 100` and `StatementTimeout = 20s`, an attacker can keep dozens of connections busy simultaneously. PostgreSQL's `work_mem = 512MB` means even 10 concurrent sort-heavy queries can add 5 GB of memory pressure. On a node with 16 GB RAM this trivially exceeds 30% additional allocation. Pool exhaustion also causes legitimate requests to queue or fail, degrading service availability for all users.

### Likelihood Explanation
The `/block` endpoint is part of the public Rosetta API surface, documented and reachable without credentials. No exploit tooling is required — standard HTTP clients (`curl`, `ab`, `wrk`) suffice. Deployments without Traefik (standalone Docker, bare-metal) have zero rate limiting. Even Traefik-protected deployments are vulnerable to coordinated multi-IP attacks since the per-host rate limit is global. The attack is repeatable and requires no prior knowledge beyond valid block indices (which are sequential integers starting from 0).

### Recommendation
1. **Application-level concurrency control**: Add a semaphore or token-bucket middleware in the Go server (before the Rosetta router) that limits total in-flight DB-bound requests, independent of infrastructure.
2. **Reduce `MaxOpenConnections`**: Lower the default from 100 to a value consistent with expected legitimate concurrency (e.g., 20–30), reducing the blast radius.
3. **Reduce `StatementTimeout`**: The default 20 seconds is too long for simple block-lookup queries; 3–5 seconds is sufficient and limits how long a connection is held.
4. **Per-IP rate limiting at the application layer**: Do not rely solely on Traefik; embed rate limiting (e.g., `golang.org/x/time/rate`) directly in the Go HTTP middleware chain.
5. **Enforce Traefik (or equivalent) as a hard deployment requirement**, not an optional Helm chart feature.

### Proof of Concept
```bash
# Requires: Rosetta node running in online mode, no Traefik, valid network_identifier
ROSETTA_URL="http://<host>:5700"
NETWORK='{"blockchain":"Hedera","network":"mainnet"}'

# Send 100 concurrent /block requests with sequential indices
for i in $(seq 1 100); do
  curl -s -X POST "$ROSETTA_URL/block" \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":$NETWORK,\"block_identifier\":{\"index\":$i}}" &
done
wait

# Observe PostgreSQL memory via:
# SELECT sum(work_mem) FROM pg_stat_activity WHERE state = 'active';
# Or monitor OS-level RSS of the postgres process before/after.
```

Each of the 100 concurrent requests acquires a DB connection held for up to 20 seconds, saturating the pool configured in `ConnectToDb()` and driving PostgreSQL memory allocation above the 30% threshold.

func (s *blockAPIService) Block(
	ctx context.Context,
	request *rTypes.BlockRequest,
) (*rTypes.BlockResponse, *rTypes.Error) {
	block, err := s.RetrieveBlock(ctx, request.BlockIdentifier)
	if err != nil {
		return nil, err
	}

	if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
		return nil, err
	}

	var otherTransactions []*rTypes.TransactionIdentifier
	if len(block.Transactions) > s.maxTransactionsInBlock {
		otherTransactions = make([]*rTypes.TransactionIdentifier, 0, len(block.Transactions)-s.maxTransactionsInBlock)
		for _, transaction := range block.Transactions[s.maxTransactionsInBlock:] {
			otherTransactions = append(otherTransactions, &rTypes.TransactionIdentifier{Hash: transaction.Hash})
		}
		block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
	}

	if err = s.updateOperationAccountAlias(ctx, block.Transactions...); err != nil {
		return nil, err
	}

	return &rTypes.BlockResponse{Block: block.ToRosetta(), OtherTransactions: otherTransactions}, nil
```
