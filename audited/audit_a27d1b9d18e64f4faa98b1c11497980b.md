### Title
Unauthenticated HTTP Network Bootstrap Allows MitM Injection of Forged Node Map in `buildNetworkFromMirrorNodes()`

### Summary
When `cfg.network == "other"`, `newClient()` calls `buildNetworkFromMirrorNodes()`, which issues a plain, unauthenticated HTTP GET to the mirror node REST API and unconditionally trusts the JSON response to build the entire Hiero network map. A network-positioned attacker who can intercept this HTTP connection can return a forged `nodesEnvelope` payload, causing `hiero.ClientForNetwork(netmap)` to construct a client that routes all subsequent `TransferTransaction`s to attacker-controlled gRPC endpoints. No application credentials are required; only network-level access to the path between the pinger and the mirror node is needed.

### Finding Description

**Code path:**

`pinger/sdk_client.go` `newClient()` lines 17–22 — when `cfg.network == "other"`, calls `buildNetworkFromMirrorNodes()` and passes its return value directly to `hiero.ClientForNetwork(netmap)`. [1](#0-0) 

`pinger/mirror_node_client.go` `buildNetworkFromMirrorNodes()` lines 46–53 — constructs a bare `http.Client` (no TLS config, no certificate pinning) and calls `fetchMirrorNodeNetwork()`. [2](#0-1) 

`pinger/mirror_node_client.go` `fetchMirrorNodeNetwork()` lines 79–98 — issues a plain `http.MethodGet` request, checks only the HTTP status code (200–299), then decodes the body into `nodesEnvelope` with no integrity or authenticity check. [3](#0-2) 

Lines 100–125 — every `(host:port, accountID)` pair from the attacker-controlled JSON is inserted verbatim into the `network` map returned to the caller. [4](#0-3) 

**Root cause:** The `http.Client` is created with zero TLS configuration and the URL is taken directly from `cfg.mirrorRest`. The documented default example is `http://mirror-rest:5551` (plain HTTP). [5](#0-4) 

There is no enforcement of HTTPS, no certificate pinning, no HMAC/signature on the response, and no allowlist of valid node addresses. The only validation applied to the response is:
- HTTP status in 200–299 range
- `NodeAccountID` non-empty and parseable as a Hiero account ID

Both checks are trivially satisfied by a forged response. [6](#0-5) 

### Impact Explanation

An attacker who successfully injects a forged `nodesEnvelope` causes `hiero.ClientForNetwork(netmap)` to build a client whose entire node roster points to attacker-controlled gRPC servers. Every `TransferTransaction` submitted by the pinger is then sent to those servers. Concrete consequences:

- **Full transaction observation:** The attacker sees every transaction in plaintext before it reaches the real network (account IDs, amounts, memo fields, operator identity).
- **Selective or total denial of service:** The attacker can drop transactions, causing the pinger to report failures or silently lose transfers.
- **Transaction replay / selective forwarding:** The attacker can forward transactions to the real network at will, or withhold them to manipulate timing.

Transaction signatures cannot be forged (the operator private key is never exposed), so the attacker cannot create fraudulent transfers on behalf of the operator. Severity: **High** for confidentiality and availability; **Medium** for integrity (signatures protect against forgery but not suppression).

### Likelihood Explanation

**Preconditions:**
1. `HIERO_MIRROR_PINGER_NETWORK=other` — a non-default but explicitly supported and documented deployment mode.
2. `HIERO_MIRROR_PINGER_REST` set to an `http://` URL — the only documented example uses plain HTTP.
3. Attacker has network-level access between the pinger container and the mirror node REST service.

Condition 3 is achievable without application credentials via: ARP/NDP poisoning on a shared LAN, a compromised sidecar or adjacent pod in Kubernetes (where pod-to-pod traffic is often unencrypted), DNS poisoning of the mirror node hostname, or BGP route hijacking for internet-facing deployments. In containerized/cloud environments this is a realistic attacker capability. The attack is repeatable on every pinger restart (the network map is fetched once at startup).

### Recommendation

1. **Enforce HTTPS:** Validate that `cfg.mirrorRest` begins with `https://` when `cfg.network == "other"` and reject plain `http://` URLs at config load time (`loadConfig()`).
2. **TLS with certificate verification:** Do not override `http.Client.Transport`; rely on Go's default TLS stack, which verifies the server certificate against the system trust store. Optionally add certificate pinning for the mirror node's CA.
3. **Response integrity:** If the mirror node API supports it, verify a response signature or use a known-good node allowlist to cross-check returned endpoints.
4. **Startup re-validation:** Re-fetch and re-validate the node list periodically rather than trusting a single bootstrap response for the entire process lifetime.

### Proof of Concept

**Environment:** Attacker controls a host on the same network segment as the pinger (or controls DNS for the mirror node hostname).

1. Configure the pinger:
   ```
   HIERO_MIRROR_PINGER_NETWORK=other
   HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551
   HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.1234
   HIERO_MIRROR_PINGER_OPERATOR_KEY=<valid key>
   ```

2. Attacker intercepts the HTTP GET to `http://mirror-rest:5551/api/v1/network/nodes` (via ARP poisoning, DNS hijack, or rogue proxy) and returns:
   ```json
   {
     "nodes": [
       {
         "node_account_id": "0.0.3",
         "service_endpoints": [
           { "ip_address_v4": "attacker.example.com", "port": 50211 }
         ]
       }
     ],
     "links": { "next": null }
   }
   ```

3. `fetchMirrorNodeNetwork()` decodes this payload; all validation passes (status 200, valid account ID `0.0.3`, non-empty host and port).

4. `hiero.ClientForNetwork({"attacker.example.com:50211": AccountID{0,0,3}})` is called.

5. Every subsequent `TransferTransaction` is submitted to `attacker.example.com:50211`. The attacker's gRPC server logs the full transaction details and can selectively forward or drop them.
