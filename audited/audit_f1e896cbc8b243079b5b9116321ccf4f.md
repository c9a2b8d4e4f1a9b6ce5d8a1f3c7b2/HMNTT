### Title
Silent Node Exclusion via Malformed `node_account_id` in Mirror REST Response (MITM)

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` silently skips any `nodeEntry` whose `node_account_id` fails to parse, with no minimum-node-count guard beyond an empty-map check. An attacker who can intercept and modify the mirror REST response can inject malformed account ID strings for ≥30% of nodes, causing those nodes to be silently dropped from the constructed network map while the function still returns success.

### Finding Description
**Exact code path:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 102–129.

```go
// line 107-110
nodeAccountId, err := hiero.AccountIDFromString(n.NodeAccountID)
if err != nil {
    continue   // ← silent drop, no log, no counter
}
```

After iterating all nodes, the only guard is:
```go
// line 127-129
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

**Root cause:** The `continue` on parse failure silently discards nodes without any accounting of how many were dropped. The sole safety net (`len(network) == 0`) only fires when *all* nodes are eliminated, not when a significant fraction (e.g., ≥30%) are removed.

**Exploit flow:**
1. Attacker positions themselves as a MITM between the pinger process and the mirror REST endpoint (feasible over plain HTTP, or via DNS poisoning/rogue proxy if TLS is not enforced).
2. Attacker intercepts the `GET /api/v1/network/nodes` response.
3. Attacker replaces `node_account_id` values for ≥30% of `nodeEntry` records with syntactically invalid strings (e.g., `"INVALID"`, `"0.0.XXXX"`, empty-but-non-empty garbage).
4. `hiero.AccountIDFromString()` returns an error for each poisoned entry; the `continue` silently skips them.
5. `fetchMirrorNodeNetwork()` returns a `map[string]hiero.AccountID` containing only the un-poisoned nodes — no error, no warning.
6. `hiero.ClientForNetwork(netmap)` is called with this reduced map; the pinger now only communicates with <70% of consensus nodes.

**Why existing checks fail:**
- The empty-string guard at line 103 (`if n.NodeAccountID == ""`) does not catch non-empty malformed strings.
- The `len(network) == 0` guard at line 127 does not fire as long as at least one valid node with a usable endpoint survives.
- There is no TLS pinning or response-integrity check visible in the HTTP client construction (line 46: `&http.Client{Timeout: cfg.mirrorNodeClientTimeout}`).

### Impact Explanation
The pinger's network map is built exclusively from this mirror REST response. If ≥30% of consensus nodes are excluded, the pinger will never attempt gRPC connections to those nodes. For a monitoring tool this means systematic false negatives (excluded nodes appear unreachable). If the pinger's results feed alerting or routing decisions, an attacker can selectively hide nodes from monitoring, masking outages or manipulating load distribution — matching the stated "≥30% node shutdown" impact without touching the nodes themselves.

### Likelihood Explanation
The precondition is MITM of the mirror REST call. This is realistic when:
- The `cfg.mirrorRest` URL uses `http://` (no TLS), making interception trivial on any shared network path.
- DNS for the mirror endpoint is spoofable (no DNSSEC).
- A rogue proxy or compromised network appliance sits between the pinger host and the mirror.

No credentials or privileged access to the Hedera network are required. The attack is repeatable on every startup or network-refresh cycle of the pinger.

### Recommendation
1. **Add a minimum-node threshold check** after the loop: if the number of successfully parsed nodes is less than some configurable fraction (e.g., 70%) of the total entries returned, treat it as a fatal error and do not return the partial map.
2. **Enforce HTTPS with certificate validation** for `cfg.mirrorRest`; reject `http://` URLs or at minimum log a loud warning.
3. **Log and count parse failures** rather than silently continuing, so operators can detect anomalous drops.
4. **Consider response signing or a trusted CA pin** for the mirror REST endpoint to prevent response tampering.
