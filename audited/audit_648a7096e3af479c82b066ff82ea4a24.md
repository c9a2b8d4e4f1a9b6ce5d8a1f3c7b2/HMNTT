### Title
Global Rate Limit Exhaustion via Single-Source Request Flooding (No Per-IP Isolation)

### Summary
`rateLimitBucket()` in `ThrottleConfiguration.java` creates a single, process-wide Bucket4j token bucket shared across all clients. Because `ThrottleManagerImpl.throttle()` consumes from this one global bucket without any per-IP or per-connection partitioning, a single unprivileged attacker can saturate the entire 500 RPS budget, causing every legitimate user to receive HTTP 429 `ThrottleException` responses for the duration of the attack.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` (lines 24–32) constructs one `Bucket` instance with capacity = `requestsPerSecond` (default 500) and a greedy refill of 500 tokens/second:

```java
// ThrottleConfiguration.java lines 25-32
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

This single Spring `@Bean` is injected into `ThrottleManagerImpl`, which calls `rateLimitBucket.tryConsume(1)` for every incoming request regardless of source:

```java
// ThrottleManagerImpl.java lines 37-42
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
    ...
}
```

**Root cause:** The bucket is a singleton with no key/partition by IP, user, or connection. All requests from all sources compete for the same 500 tokens/second pool.

**Why existing checks fail:**
- `LoggingFilter` (`web3/src/main/java/org/hiero/mirror/web3/config/LoggingFilter.java`) reads `request.getRemoteAddr()` only for logging — it is never used for rate limiting.
- `RequestProperties` filters (`ThrottleProperties.getRequest()`) can be configured to REJECT or THROTTLE specific request patterns, but they are **empty by default** (`List.of()`), are not per-IP, and are evaluated only *after* the global bucket check passes.
- No Spring Security filter, servlet filter, or reverse-proxy configuration in the web3 config directory enforces per-source limits.

### Impact Explanation
An attacker sending ≥500 HTTP POST requests per second to any web3 endpoint (e.g., `/api/v1/contracts/call`) drains all tokens in the global bucket. Every subsequent request from any legitimate user within that second fails with HTTP 429. Because the bucket refills greedily at 500 tokens/second and the attacker can sustain the flood continuously, legitimate traffic is effectively blocked for the entire attack duration. This directly degrades or eliminates processing throughput on the targeted node, meeting the ≥30% node throughput degradation threshold described in the scope.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and no special knowledge beyond the public API endpoint. It is executable with standard HTTP load tools (e.g., `wrk`, `ab`, `hey`) from a single machine with a modest network connection. The attack is fully repeatable and can be sustained indefinitely. No CAPTCHA, API key, or connection-level enforcement exists in the application layer to prevent it.

### Recommendation
Replace the single global bucket with a per-IP (or per-authenticated-client) bucket map, for example using Bucket4j's `BucketProxyManager` keyed by `request.getRemoteAddr()` (or the `X-Forwarded-For` header when behind a trusted proxy). Additionally:
- Enforce a per-IP connection limit at the servlet/server level (e.g., Tomcat `maxConnections` per remote address).
- Retain the global bucket as a secondary ceiling to prevent aggregate abuse.
- Consider requiring an API key so that abusive keys can be revoked independently.

### Proof of Concept
**Preconditions:** Network access to the web3 service; no credentials required.

**Steps:**
1. Identify the web3 endpoint, e.g., `http://<host>:<port>/api/v1/contracts/call`.
2. Run a sustained flood from a single machine:
   ```bash
   # Using 'hey' HTTP load generator
   hey -z 60s -c 100 -q 10 \
     -m POST \
     -H "Content-Type: application/json" \
     -d '{"data":"0xdeadbeef","to":"0x00000000000000000000000000000000000004e2","gas":21000,"block":"latest"}' \
     http://<host>:<port>/api/v1/contracts/call
   ```
   Adjust `-c` (concurrency) and `-q` (rate per worker) so total RPS ≥ 500.
3. Simultaneously send a single legitimate request from a different client:
   ```bash
   curl -X POST http://<host>:<port>/api/v1/contracts/call \
     -H "Content-Type: application/json" \
     -d '{"data":"0xdeadbeef","to":"0x00000000000000000000000000000000000004e2","gas":21000,"block":"latest"}'
   ```
4. **Expected result:** The legitimate request receives HTTP 429 with body containing `"Requests per second rate limit exceeded"` for the duration of the flood, confirming full global budget exhaustion by the attacker.
