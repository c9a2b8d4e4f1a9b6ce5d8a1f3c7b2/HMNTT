### Title
Unprivileged 3x Database Query Amplification via `getQuery()` Multi-Union Subquery Construction in Token Allowance REST API

### Summary
The `getQuery()` function in `rest/service/tokenAllowanceService.js` unconditionally generates up to three independent `SELECT` subqueries against the `token_allowance` table per HTTP request, combined via `UNION ALL`. Any unauthenticated external user can deterministically trigger all three subqueries with a single crafted request. The REST API server has no rate-limiting middleware, making this a repeatable, zero-privilege database load amplification primitive.

### Finding Description

**Exact code path:**

In `rest/service/tokenAllowanceService.js` lines 63–65:

```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));
``` [1](#0-0) 

Each non-empty array in `[lower, inner, upper]` produces a full independent `SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND <conditions> ORDER BY ... LIMIT $2` subquery. When all three are non-empty, the final SQL is three subqueries joined by `UNION ALL` with an outer `ORDER BY ... LIMIT`. [2](#0-1) 

**How all three are populated:** The `extractTokenMultiUnionQuery()` controller method calls `getLowerFilters()`, `getInnerFilters()`, and `getUpperFilters()` on the parsed bounds. When a user supplies a closed range on both `spender.id` (gte+lte) and `token.id` (gte+lte), all three filter arrays are non-empty, as confirmed by the test case "spender closed range and token closed range": [3](#0-2) 

**Why existing checks fail:**

`validateBounds()` in `baseController.js` only checks logical validity (e.g., no simultaneous range+equal, secondary requires primary). It does not cap the number of subqueries produced. [4](#0-3) 

`buildAndValidateFilters()` enforces `maxRepeatedQueryParameters` per key, but triggering all three subqueries requires only 2 values for `spender.id` and 2 for `token.id` — well within the default limit of 4. [5](#0-4) 

**No rate limiting on the REST API:** `server.js` registers no rate-limiting middleware. The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` Java service. The REST Node.js server only applies `authHandler` (response-size limits), `requestLogger`, optional metrics, and optional response caching — none of which limit request rate or subquery count. [6](#0-5) 

### Impact Explanation

Every request to `GET /api/v1/accounts/:id/allowances/tokens?spender.id=gte:X&spender.id=lte:Y&token.id=gte:A&token.id=lte:B` causes the database to execute three independent indexed scans on `token_allowance` instead of one. For accounts with large allowance sets (e.g., popular DeFi contracts on Hedera mainnet), each scan can be non-trivial. An attacker sending N concurrent requests of this form generates 3N database queries. With no server-side rate limiting, this is a straightforward database exhaustion vector. The `token_allowance` table is shared infrastructure; degrading it affects all mirror node consumers.

### Likelihood Explanation

The exploit requires zero privileges, zero authentication, and only publicly documented query parameters (`spender.id`, `token.id` with range operators). The endpoint is publicly accessible at `GET /api/v1/accounts/:idOrAliasOrEvmAddress/allowances/tokens`. The four required parameters are within the `maxRepeatedQueryParameters` default. Any attacker who reads the API documentation can reproduce this in seconds with a single `curl` command. It is trivially scriptable for sustained amplification.

### Recommendation

1. **Add a subquery count cap in `getQuery()`**: After computing `subQueries`, assert `subQueries.length <= 2` (or a configurable maximum) and throw an `InvalidArgumentError` if exceeded.
2. **Add rate limiting to the REST API**: Apply a per-IP or global request-rate middleware (e.g., `express-rate-limit`) to the Node.js REST server, mirroring the `ThrottleManagerImpl` pattern used in the web3 service.
3. **Consider collapsing the 3-subquery case**: For the closed-range-on-both-dimensions case, evaluate whether a single query with combined `OR` conditions or a different pagination strategy can replace the three-way `UNION ALL`.

### Proof of Concept

```bash
# Trigger all three subqueries (lower + inner + upper) in a single unauthenticated request:
curl "https://<mirror-node-host>/api/v1/accounts/0.0.1000/allowances/tokens?\
spender.id=gte:0.0.2&spender.id=lte:0.0.8&token.id=gte:0.0.100&token.id=lte:0.0.200"

# This generates the following SQL (3 independent DB scans):
# (SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ...)
# UNION ALL
# (SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ...)
# UNION ALL
# (SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ...)
# ORDER BY spender ASC, token_id ASC LIMIT $2

# Amplification loop (no auth required):
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.1000/allowances/tokens?\
spender.id=gte:0.0.2&spender.id=lte:0.0.8&token.id=gte:0.0.100&token.id=lte:0.0.200" &
done
# Results in 3000 DB queries from 1000 HTTP requests with zero server-side mitigation.
```
