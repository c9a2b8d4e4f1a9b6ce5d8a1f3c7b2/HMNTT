### Title
Unbounded Label Cardinality via User-Controlled Path in `toOpenApiPath()` Metrics Middleware

### Summary
The `toOpenApiPath()` function in `rest/middleware/metricsHandler.js` falls back to the raw, user-controlled `req.path` when a request does not match any registered Express route. This raw path is then used as the `path` label in four per-route OpenTelemetry instruments. An unprivileged attacker can send requests to an unbounded number of unique paths (combined with any supported HTTP method), creating an unbounded number of unique `{method, path, code}` label combinations and causing label cardinality explosion in the OTel SDK.

### Finding Description

**Exact code path:**

In `toOpenApiPath()` ( [1](#0-0) ), the function resolves the metric `path` label with this priority:

1. `res.locals[requestPathLabel]` — set only by `recordRequestPath` for matched routes ( [2](#0-1) )
2. If that is unset and `req.route` is falsy (i.e., no Express route matched), it falls back to `path = req.path` — the raw URL path from the HTTP request ( [3](#0-2) )

This raw path is then used directly as the `path` label value in all four per-route instruments:

```js
const labels = {method, path, code};
requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
``` [4](#0-3) 

**Root cause:** The assumption that `req.path` is bounded (i.e., always a known route pattern) is false for unmatched requests. Express only sets `req.route` when a handler matches; for all other requests it is `undefined`, so the fallback to `req.path` is always taken for 404 responses.

**Why existing checks fail:**
- `recordRequestPath` only sets `res.locals[requestPathLabel]` when `req.route?.path` exists — which is never true for unmatched routes. [5](#0-4) 
- `responseHandler` throws `NotFoundError` for unmatched routes, but the `res.on('finish', ...)` callback in the metrics middleware fires *after* the error handler sends the 404 response — so metrics are still recorded with the raw path. [6](#0-5) 
- There is no cardinality cap, path normalization, or rate limiting applied to the `path` label anywhere in the metrics pipeline.

### Impact Explanation
Each unique `{method, path, code}` triple causes the OTel SDK (`@opentelemetry/sdk-metrics`) to allocate a new internal accumulator and time series for each of the four instruments (`api_request`, `api_request_duration_milliseconds`, `api_request_size_bytes`, `api_response_size_bytes`). With N unique paths and M HTTP methods, the SDK holds 4 × N × M active accumulators in memory. At scale this causes: unbounded heap growth leading to OOM crashes, CPU exhaustion during metric collection/serialization (the Prometheus scrape endpoint must iterate all time series), and degraded or unavailable observability. This is a denial-of-service against the metrics subsystem with potential process-level impact.

### Likelihood Explanation
No authentication, authorization, or rate limiting is required. Any external client can send HTTP requests to arbitrary paths. The attack is trivially scriptable: a simple loop generating unique path segments (e.g., UUIDs or sequential integers) is sufficient. The attack is repeatable and persistent — once a label combination is registered in the OTel SDK it is not evicted. A single attacker with a basic HTTP client can execute this attack continuously.

### Recommendation
1. **Normalize unmatched paths to a fixed sentinel value** in `toOpenApiPath()`: when `req.route` is undefined and `res.locals[requestPathLabel]` is unset, return a fixed string such as `"unknown"` or `apiPrefix + "/unknown"` instead of `req.path`.
2. **Alternatively**, skip recording per-route metrics entirely for unmatched routes (detect via `res.statusCode === 404` and absence of `req.route`).
3. **Do not** use any user-supplied input (path, query string, headers) as an OTel label value without first mapping it to a finite, known set of values.

### Proof of Concept

```bash
# Send requests with unique paths to exhaust label cardinality
for i in $(seq 1 100000); do
  curl -s -o /dev/null "http://<mirror-node-host>/api/v1/nonexistent_path_$i" &
done

# Optionally multiply by HTTP method
for method in GET POST PUT DELETE PATCH OPTIONS HEAD; do
  for i in $(seq 1 10000); do
    curl -s -o /dev/null -X $method "http://<mirror-node-host>/api/v1/attack_$method_$i" &
  done
done

# Observe: scraping /swagger/metrics/ becomes progressively slower,
# and the mirror-node process heap grows without bound.
```

Each request to an unmatched path causes `toOpenApiPath()` to return the raw `req.path` (e.g., `/api/v1/nonexistent_path_42`), which is stored as a new label combination in all four OTel instruments, permanently consuming memory in the SDK accumulator maps.

### Citations

**File:** rest/middleware/metricsHandler.js (L13-31)
```javascript
const toOpenApiPath = (req, res) => {
  let path = res.locals[requestPathLabel];

  if (!path) {
    if (!req.route) {
      path = req.path;
    } else {
      path = (req.baseUrl ?? '') + req.route?.path;
    }
  }

  path = path.replace(/:([^/]+)/g, '{$1}');

  if (!path.startsWith(apiPrefix)) {
    return apiPrefix + '/' + path;
  }

  return path;
};
```
