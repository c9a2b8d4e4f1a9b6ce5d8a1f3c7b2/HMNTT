### Title
Unauthenticated Full-Table-Scan DoS via Unbounded Timestamp Range in `getHookStorageChange()`

### Summary
The public `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts a `timestamp` query parameter with no lower-bound enforcement. When a timestamp is supplied but no `key` filter is given, `getHookStorageChange()` routes to `findByKeyBetweenAndTimestampBetween()`, which executes a native `SELECT DISTINCT ON (key)` query over the full key range (`0x00…00` to `0xFF…FF`) and the full timestamp range (`0` to `Long.MAX_VALUE`). PostgreSQL must sort and deduplicate every matching row before applying the `LIMIT`, making the `Pageable` limit ineffective as a cost-control mechanism. No authentication or rate limiting is present, so any unauthenticated attacker can flood the endpoint with such requests to exhaust database CPU.

### Finding Description

**Endpoint and routing:**

`HooksController.getHookStorage()` at [1](#0-0)  is publicly accessible with no authentication. When `timestamp` is non-empty, `HookStorageRequest.isHistorical()` returns `true` [2](#0-1) , routing execution to `getHookStorageChange()`.

**Default bounds — maximum range:**

In `hookStorageChangeRequest()`, `keyLowerBound` defaults to 32 bytes of `0x00` and `keyUpperBound` to 32 bytes of `0xFF` [3](#0-2) . In `Bound`, when only a lower timestamp is supplied (e.g., `gte:0`), `adjustUpperBound()` returns `Long.MAX_VALUE` because `upper == null` [4](#0-3) , and `getAdjustedLowerRangeValue()` returns `0` when `lower == null` [5](#0-4) .

**No-key path in `getHookStorageChange()`:**

When `keys` is empty (the default), the code falls through to `findByKeyBetweenAndTimestampBetween()` with the full key and timestamp ranges [6](#0-5) .

**The expensive native query:**

```sql
select distinct on (key)
     owner_id, hook_id, key, ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id = :hookId
  and key >= :keyLowerBound        -- 0x00…00
  and key <= :keyUpperBound        -- 0xFF…FF
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound  -- 0 to MAX_LONG
``` [7](#0-6) 

PostgreSQL's `DISTINCT ON (key)` requires sorting all qualifying rows by `(key, consensus_timestamp DESC)` before deduplication. The `Pageable` LIMIT is appended after this sort+dedup phase, so the database performs the full scan and sort regardless of the requested limit.

**Why existing checks fail:**

- `@Max(MAX_LIMIT)` on `limit` (capped at 100) [8](#0-7)  limits only the number of rows returned, not the rows scanned or sorted by the DB engine.
- `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on `timestamps` [9](#0-8)  limits the number of timestamp parameters, not their range width.
- No rate limiting, no authentication, and no minimum timestamp range width is enforced anywhere in the controller or service layer.

### Impact Explanation
A large `hook_storage_change` table (which grows naturally as hooks update storage over time) means each such request forces a full-table scan and sort for the targeted `(owner_id, hook_id)` pair. Concurrent requests from one or more attackers can saturate database CPU, degrading query throughput for all mirror node operations that share the same database cluster. Because the mirror node's importer and REST services share the DB, sustained CPU exhaustion can stall block ingestion and degrade ≥30% of processing capacity on a shared-DB deployment.

### Likelihood Explanation
The attack requires only: (1) a valid `ownerId` and `hookId` (discoverable via the public hooks listing endpoint), and (2) the ability to send HTTP GET requests. No credentials, tokens, or privileged access are needed. The request is trivially repeatable and scriptable. Any attacker with network access to the mirror node REST API can execute this.

### Recommendation
1. **Enforce a maximum timestamp range width**: Reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 1 hour in nanoseconds).
2. **Require a timestamp upper bound**: Reject requests that supply only a lower-bound timestamp with no upper bound, preventing the `Long.MAX_VALUE` default.
3. **Add rate limiting**: Apply per-IP or per-account rate limiting on the storage endpoint.
4. **Rewrite the query**: Use a correlated subquery or `LATERAL JOIN` pattern that leverages an index on `(owner_id, hook_id, key, consensus_timestamp DESC)` to retrieve the latest row per key without a full sort, and apply LIMIT before the dedup phase.

### Proof of Concept

```bash
# Step 1: Discover a valid ownerId and hookId
curl "https://<mirror-node>/api/v1/accounts/0.0.1/hooks"

# Step 2: Send a maximum-range historical storage query (no keys, full timestamp range)
# Repeat concurrently to exhaust DB CPU
for i in $(seq 1 50); do
  curl "https://<mirror-node>/api/v1/accounts/0.0.1/hooks/1/storage?timestamp=gte:0&limit=100" &
done
wait
```

Each request triggers `findByKeyBetweenAndTimestampBetween()` with `keyLowerBound=0x00…00`, `keyUpperBound=0xFF…FF`, `timestampLowerBound=0`, `timestampUpperBound=9223372036854775807`, forcing a full scan and sort of all `hook_storage_change` rows for that hook. Fifty concurrent requests will saturate database CPU on a typical deployment.
