### Title
Unbounded Metrics Label Cardinality via Unmatched-Route Path Injection Causes Synchronous Event-Loop Block (DoS)

### Summary
The `metricsMiddleware` in `rest/middleware/metricsHandler.js` uses the raw `req.path` as the `path` label value for any request that does not match an Express route (i.e., 404 responses). Because there is no cardinality cap on the OpenTelemetry in-memory store, an unauthenticated attacker can pre-populate millions of unique label combinations by sending requests to arbitrary paths. A subsequent call to the metrics endpoint causes `serializer.serialize(resourceMetrics)` — a synchronous, CPU-bound operation — to block the Node.js event loop for seconds, stalling block record ingestion.

### Finding Description

**Root cause — unbounded `path` label from raw `req.path`:**

In `toOpenApiPath` (lines 13–31), when no Express route matches (`!req.route`), the function falls back to `req.path` verbatim:

```js
// rest/middleware/metricsHandler.js lines 16-18
if (!path) {
  if (!req.route) {
    path = req.path;   // ← raw attacker-controlled string
```

This raw path is then used as the `path` dimension of the label set (lines 189–192):

```js
const path = toOpenApiPath(req, res);
const code = String(res.statusCode);
const method = req.method;
const labels = {method, path, code};
```

All four per-route instruments record with these labels (lines 207–210):

```js
requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
```

Each unique `{method, path, code}` triple creates a new time series in the OpenTelemetry SDK's in-memory accumulation store. There is **no cardinality limit, no label sanitization, and no rate limiting** anywhere in the file.

**Trigger — synchronous serialization on the metrics endpoint:**

```js
// lines 161-164
return exporter.collect().then(({resourceMetrics}) => {
  res.set('Content-Type', 'text/plain; charset=utf-8');
  res.send(serializer.serialize(resourceMetrics));   // ← synchronous, CPU-bound
});
```

`PrometheusSerializer.serialize()` iterates over every stored data point synchronously on the Node.js event loop. With millions of accumulated series (each histogram contributing 5+ bucket lines), this call can block the loop for multiple seconds.

**Why existing checks are insufficient:**

- The authentication guard (lines 157–160) protects only the metrics endpoint itself; it does not prevent label pollution, which happens on every non-metrics request with no authentication.
- If `authentication` is disabled (`config.metrics.config.authentication === false`, line 139), the attacker can also directly trigger the serialization.
- Even with authentication enabled, the legitimate Prometheus scraper (which runs on a fixed interval) will trigger the DoS after the store has been pre-populated.

### Impact Explanation
Node.js is single-threaded. When `serializer.serialize()` blocks the event loop for seconds, no other I/O callbacks — including block record ingestion from the stream — can execute. This directly delays processing of block records by the mirror node REST service, satisfying the 500%+ average block time threshold described in the finding scope. The attack is persistent: the accumulated label series remain in memory across scrape cycles, so a single poisoning campaign causes repeated DoS on every subsequent metrics scrape.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no knowledge of internal state. Any internet-accessible mirror-node REST instance is vulnerable. The attacker needs only to send a large volume of HTTP GET requests to unique paths (trivially automated with a script or tool like `ab`, `wrk`, or a simple loop). The 404 response path is always reachable because Express passes unmatched routes through the middleware chain before returning 404. The attack is repeatable and the poisoned state persists until process restart.

### Recommendation

1. **Normalize unmatched-route paths to a fixed sentinel value** instead of using `req.path`:
   ```js
   if (!req.route) {
     path = '/unknown';  // or apiPrefix + '/unknown'
   }
   ```
2. **Enforce a cardinality limit** on the metrics store (e.g., using OpenTelemetry's `ViewCardinalityLimit` or a custom wrapper that drops new label sets beyond a threshold).
3. **Offload serialization** to a `Worker` thread or use a streaming serializer so the event loop is not blocked regardless of store size.
4. **Rate-limit or reject requests** with excessively long or structurally invalid paths before they reach the metrics middleware.

### Proof of Concept

```bash
# Step 1: Pre-populate the metrics store with 500,000 unique label combinations
# (no credentials required — these are ordinary 404 requests)
for i in $(seq 1 500000); do
  curl -s "http://<mirror-node-rest>:5551/poison-path-${i}" > /dev/null &
done
wait

# Step 2: Trigger metrics serialization
# (works without credentials if authentication is disabled;
#  otherwise wait for the Prometheus scraper interval)
time curl -s "http://<mirror-node-rest>:5551/swagger/metrics/"
# Expected: response takes 5–30+ seconds; event loop is blocked for that duration
# Block record ingestion is stalled for the entire serialization window
```
