### Title
Synchronous `unzipSync()` on Every Cache Hit Without Rate Limiting Enables CPU Exhaustion by Unprivileged Users

### Summary
In `responseCacheCheckHandler()`, when a cached response is stored compressed and a client requests without gzip support, `cachedResponse.getUncompressedBody()` calls Node.js's synchronous `unzipSync()` on every single cache hit. Because the REST API has no rate limiting whatsoever (unlike the `web3` Java module), an unprivileged attacker can flood any large-response endpoint with non-gzip requests, causing repeated synchronous event-loop-blocking decompression that drives CPU consumption well above 30%.

### Finding Description

**Exact code path:**

In `rest/model/cachedApiResponse.js` line 30:
```js
getUncompressedBody() {
  return this.compressed ? unzipSync(Buffer.from(this.body, 'base64url')) : this.body;
}
```
`unzipSync` is Node.js's **synchronous** zlib decompression — it blocks the entire event loop until complete. [1](#0-0) 

This is called from `responseCacheCheckHandler()` at line 72:
```js
const useCompressed = negotiate(cachedResponse, req, res);
body = useCompressed ? cachedResponse.getBody() : cachedResponse.getUncompressedBody();
``` [2](#0-1) 

`negotiate()` returns `false` (triggering `getUncompressedBody()`) whenever the client's `Accept-Encoding` header does not include `gzip`:
```js
if (cachedResponse.compressed) {
  const negotiator = new Negotiator(req);
  if (negotiator.encoding([GZIP_ENCODING]) === GZIP_ENCODING) { ... return true; }
}
// falls through: return false → getUncompressedBody() is called
``` [3](#0-2) 

**Root cause — failed assumption:** The design assumes that the cost of serving from cache is negligible. This is true when the client accepts gzip (compressed bytes are returned directly). But when the client sends `Accept-Encoding: identity`, the full synchronous decompression runs on every request, on the main event loop thread, with no bound on frequency.

**No rate limiting in the REST API:** The entire REST middleware stack in `rest/server.js` is: cors → compression → httpContext → requestLogger → authHandler → metricsHandler → `responseCacheCheckHandler`. There is no rate-limiting middleware anywhere. [4](#0-3) 

The rate limiting that does exist (`ThrottleConfiguration`, `ThrottleManagerImpl`) is exclusively in the `web3` Java module and applies only to `/api/v1/contracts/call` and opcode endpoints — it has zero effect on the REST Node.js service. [5](#0-4) 

### Impact Explanation

Node.js is single-threaded for JavaScript execution. Each `unzipSync()` call on a large payload (e.g., a `limit=100` transactions response, which can be hundreds of KB of JSON) blocks the event loop for the entire duration of decompression. While the event loop is blocked, no other requests can be processed. An attacker sending a sustained flood of concurrent requests to a single cached large-response endpoint forces the server into a near-continuous decompression loop, causing CPU to spike and all other clients to experience severe latency or timeouts. This directly satisfies the >30% resource consumption increase threshold without any brute-force database queries — the attacker is only hitting the cache path.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero special knowledge beyond knowing a valid API endpoint (all are publicly documented). The attacker needs only to:
1. Confirm a large-response endpoint is cached (one warm-up request suffices, or natural traffic does it).
2. Send concurrent HTTP GET requests with `Accept-Encoding: identity` (or simply omit the header).

This is trivially scriptable with `curl`, `ab`, `wrk`, or any HTTP load tool. The attack is repeatable indefinitely and is not self-limiting. The cache TTL keeps the compressed payload available in Redis, so the attacker never needs to re-populate it.

### Recommendation

1. **Replace `unzipSync` with async decompression** (`zlib.gunzip` with a Promise wrapper or `pipeline`) so the event loop is not blocked during decompression.
2. **Add rate limiting to the REST API** — introduce a middleware such as `express-rate-limit` before `responseCacheCheckHandler` in `rest/server.js`, mirroring the bucket4j throttle that already exists in the `web3` module.
3. **Cap decompressed payload size** before calling `unzipSync` by checking `cachedResponse.length` (already stored on the object) and refusing to decompress payloads above a configurable threshold, returning a 503 instead.

### Proof of Concept

**Precondition:** Response caching and compression are enabled (`config.cache.response.enabled = true`, `config.cache.response.compress = true`).

```bash
# Step 1: Warm up the cache with a large response (one request)
curl -s "https://<mirror-node>/api/v1/transactions?limit=100" \
     -H "Accept-Encoding: gzip" -o /dev/null

# Step 2: Flood with non-gzip requests to force repeated synchronous unzipSync()
# Use 50 concurrent workers, 10000 total requests
ab -n 10000 -c 50 \
   -H "Accept-Encoding: identity" \
   "https://<mirror-node>/api/v1/transactions?limit=100"

# Observe: CPU on the mirror-node REST process spikes >30% above baseline.
# Other API endpoints become slow or unresponsive during the flood.
# No authentication, no special headers, no database load required.
```

Each of the 10,000 requests hits the Redis cache (fast), then calls `unzipSync()` synchronously on the compressed transactions payload, blocking the Node.js event loop on every iteration.

const negotiate = (cachedResponse, req, res) => {
  res.setHeader(VARY_HEADER, 'accept-encoding');

  if (cachedResponse.compressed) {
    const negotiator = new Negotiator(req);
    if (negotiator.encoding([GZIP_ENCODING]) === GZIP_ENCODING) {
      res.setHeader(CONTENT_ENCODING_HEADER, GZIP_ENCODING);
      res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getLength());
      return true;
    }
  }

  res.setHeader(CONTENT_LENGTH_HEADER, cachedResponse.getUncompressedLength());
  return false;
};
```

public class ThrottleConfiguration {
