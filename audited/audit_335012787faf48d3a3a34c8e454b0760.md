### Title
Cache-Amplified SQL IN-Clause DoS via Batch Contract Slot Poisoning

### Summary
When `enableBatchContractSlotCaching` is `true` (the default), `findStorageBatch()` in `ContractStateServiceImpl` unconditionally issues a SQL `IN`-clause query containing every slot key currently held in the per-contract `slotsPerContract` Caffeine cache (up to 1500 entries). Because the `contractStateCache` expires every 2 seconds while the `slotsPerContract` cache persists for 5 minutes, an unprivileged attacker who pre-fills the slot key cache for a target contract can force a 1500-slot SQL query on every subsequent single-slot read, achieving a sustained ~1500× amplification in database query cost with minimal ongoing effort.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` — [1](#0-0) 

checks `contractStateCache` (TTL: `expireAfterWrite=2s`) and, on any miss, delegates to `findStorageBatch()`.

`findStorageBatch()` — [2](#0-1) 

unconditionally collects **all** keys currently in the per-contract `slotsPerContract` cache and passes them as the `cachedSlots` list to `contractStateRepository.findStorageBatch()`. There is no cap on `cachedSlots.size()`.

The `slotsPerContract` cache is configured as `expireAfterAccess=5m,maximumSize=1500`: [3](#0-2) 

The `contractStateCache` expires every 2 seconds: [4](#0-3) 

The resulting SQL query: [5](#0-4) 

expands to `slot IN (<1500 byte-array literals>)` — a query that is orders of magnitude more expensive than the single-slot `findStorage` query it replaces.

**Root cause / failed assumption:** The design assumes that all 1500 cached slot keys will be read together within the 2-second `contractStateCache` window, making the batch query amortized-efficient. An attacker violates this assumption by reading only one slot per 2-second window, forcing the full 1500-slot query every cycle with a single API call.

**Additional amplification for non-existent slots:** When a slot does not exist in the database, `findStorageBatch()` returns no row for it, so it is **never written** to `contractStateCache`. [6](#0-5)  This means reads of non-existent slots bypass the 2-second cooldown entirely — every single read triggers the full batch query with no caching relief.

### Impact Explanation

At the default throttle of 500 requests/second, [7](#0-6)  an attacker can sustain 500 × 1500 = 750,000 effective slot-DB-reads per second against the database, compared to 500 without the attack — a 1500× amplification. The gas-based throttle does not help because a single-slot SLOAD opcode consumes negligible gas relative to the 15M gas/request ceiling. [8](#0-7)  The database `statementTimeout` of 3000 ms provides a last-resort guard but does not prevent the load spike. The `contractSlotsCache` outer cache holds up to 3000 per-contract caches simultaneously, [9](#0-8)  so an attacker can poison multiple contracts in parallel, multiplying the effect further.

### Likelihood Explanation

The attack requires no credentials, no on-chain transactions, and no special knowledge beyond a target contract address with many storage slots (e.g., any large DeFi contract). The setup phase (1500 `eth_call` or `eth_getStorageAt` reads to fill the cache) is a one-time cost well within the public rate limit. Thereafter, the attacker needs only one request every 2 seconds per poisoned contract to sustain the amplification. The `slotsPerContract` cache TTL of 5 minutes means the attacker can keep the cache warm with one request every ~4 minutes per contract. The attack is fully repeatable, requires no privileged access, and is trivially scriptable.

### Recommendation

1. **Cap the batch query size**: In `findStorageBatch()`, limit `cachedSlots` to a configurable maximum (e.g., 50–100 slots) before calling `findStorageBatch()`. Slots beyond the cap should fall back to the single-slot `findStorage()` query.
2. **Cache negative results**: After `findStorageBatch()` returns, store a sentinel value in `contractStateCache` for any requested slot that was not found in the DB. This prevents the non-existent-slot bypass of the 2-second cooldown. The existing `EMPTY_VALUE` sentinel and the `cachedValue != EMPTY_VALUE` guard at line 65 are already in place for this purpose but are never populated for missing slots.
3. **Account for slot amplification in throttling**: Consider throttling based on the number of unique storage slots accessed per request, not just gas or request count.

### Proof of Concept

```
# Step 1: Fill the slotsPerContract cache for target contract C with 1500 unique slot keys.
# Each call reads a different slot (slots 0x0000...0001 through 0x0000...05DC).
for i in $(seq 1 1500); do
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -d "{\"to\":\"<C>\",\"data\":\"<SLOAD slot_$i>\",\"gas\":50000}"
done

# Step 2: Wait 3 seconds for contractStateCache (TTL=2s) to expire.
sleep 3

# Step 3: Send a single read of any slot. This triggers findStorageBatch() with
# all 1500 cached slot keys, issuing a SQL query:
#   SELECT slot, value FROM contract_state WHERE contract_id=C AND slot IN (<1500 values>)
curl -X POST http://mirror-node/api/v1/contracts/call \
  -d "{\"to\":\"<C>\",\"data\":\"<SLOAD slot_1>\",\"gas\":50000}"

# Step 4: Repeat Step 3 every 2+ seconds to sustain the amplification.
# Each single API call forces a 1500-slot DB query indefinitely (for 5 minutes
# without re-seeding, or indefinitely with periodic re-seeding).
```

    public Optional<byte[]> findStorage(final EntityId contractId, final byte[] key) {
        if (!cacheProperties.isEnableBatchContractSlotCaching()) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }

        final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);

        if (cachedValue != null && cachedValue != EMPTY_VALUE) {
            return Optional.of(cachedValue);
        }

        return findStorageBatch(contractId, key);
    }

        final var wrappedKey = ByteBuffer.wrap(key);

                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
