### Title
Unauthenticated EVM Address Lookup Triggers Extra Sequential DB Round-Trip, Amplifying Connection Pool Exhaustion Under Load

### Summary
`getOneAccount()` in `rest/accounts.js` unconditionally `await`s `EntityService.getEncodedId()` before issuing the main entity query. When the path parameter is an EVM address (`0x…`), `getEncodedId()` executes a synchronous DB lookup (`entityFromEvmAddressQuery`) to resolve the address to an internal ID, adding a full sequential round-trip before the main query pair. No authentication, no rate limiting, and a default pool ceiling of 10 connections make this endpoint a practical amplification point for connection-pool exhaustion.

### Finding Description

**Code path:**

`rest/accounts.js` line 399:
```js
const encodedId = await EntityService.getEncodedId(
  req.params[constants.filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]
);
``` [1](#0-0) 

`EntityService.getEncodedId()` at `rest/service/entityService.js` line 118–124 branches on whether the parsed `EntityId` carries a non-null `evmAddress`. For a `0x…` input it does, so it calls `getEntityIdFromEvmAddress()`: [2](#0-1) 

`getEntityIdFromEvmAddress()` issues a synchronous pool query and `await`s it:
```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
  [Buffer.from(entityId.evmAddress, 'hex')]);
``` [3](#0-2) 

`BaseService.getRows()` acquires a connection from the global pool, executes, and releases it: [4](#0-3) 

Only after that `await` resolves does `getOneAccount()` fire the main entity query and the transactions query concurrently via `Promise.all`: [5](#0-4) 

**Round-trip comparison per request:**

| Input type | DB calls | Concurrency |
|---|---|---|
| Numeric ID (`0.0.X`) | 2 (entity + txns) | Concurrent |
| EVM address (`0x…`) | 3 (EVM lookup → entity + txns) | 1 sequential then 2 concurrent |

The EVM lookup is not cached at the query level. `entityService.js` calls `this.getRows()` directly with no cache check. The `entityId` cache referenced in configuration (`maxAge: 1800`, `maxSize: 100000`) lives in `entityId.js` and covers `EntityId` object parsing, not the DB resolution of an EVM address to an internal numeric ID. [6](#0-5) 

**Root cause / failed assumption:** The design assumes EVM address resolution is cheap or infrequent. It is neither: the lookup is a full indexed DB query, it is sequential (blocking the request), and it is triggered by any caller who supplies a hex address — no credentials required.

### Impact Explanation

The default REST API connection pool ceiling is **10 connections**: [7](#0-6) 

With `connectionTimeoutMillis` defaulting to 20 000 ms, once the pool is saturated, new requests queue for up to 20 seconds before failing. Because EVM address requests hold a connection for three sequential query slots instead of two, they keep connections busy ~50 % longer per request. Under concurrent load this raises the steady-state connection occupancy, pushing the pool to exhaustion faster than numeric-ID requests would. The result is denial of service for all `/api/v1/accounts/*` callers sharing the same pool.

PgBouncer is present in the production Helm chart in `transaction` pool mode for `mirror_rest`, which returns server connections to PgBouncer between queries and partially absorbs the pressure. However, the Node.js-side pool (max 10) is the first bottleneck and is not protected by PgBouncer. [8](#0-7) 

### Likelihood Explanation

- **No authentication or authorization** is required to call `GET /api/v1/accounts/0x<address>`.
- **No rate limiting** exists in the REST layer (`rest/**/*.js` contains no rate-limit or throttle logic).
- Any valid 20-byte hex string triggers the extra round-trip. An attacker can enumerate known EVM addresses from public chain data or simply use any address that resolves (a 404 still consumes the first DB call).
- The attack is trivially repeatable with standard HTTP load tools (`wrk`, `hey`, `k6`).
- The default pool of 10 connections means only ~10 concurrent EVM-address requests are needed to saturate the pool.

### Recommendation

1. **Cache the EVM-address → entity-ID resolution** at the service layer (an in-process LRU or the existing Redis cache) so repeated lookups for the same address skip the DB round-trip entirely.
2. **Merge the EVM address resolution into the main entity query** using a single SQL expression (`WHERE evm_address = $1 OR id = $1`) to eliminate the sequential pre-query entirely.
3. **Add request-level rate limiting** (e.g., via the existing Traefik middleware or an Express rate-limiter) on `/api/v1/accounts/:id` to bound the number of concurrent DB-touching requests per source IP.
4. **Raise the default pool size** or document that operators must tune it for production traffic.

### Proof of Concept

```bash
# Obtain any valid EVM address from the mirror node (or use a known one)
EVM_ADDR="0xabcdef1234567890abcdef1234567890abcdef12"
BASE="https://<mirror-node-host>/api/v1"

# Flood with concurrent EVM-address lookups (no auth needed)
# Each request triggers: EVM lookup DB call → entity query → transactions query
hey -n 500 -c 50 "${BASE}/accounts/${EVM_ADDR}"

# Expected result: after pool (default 10) is saturated, subsequent requests
# return 500 / timeout errors; numeric-ID requests on the same node are also
# affected because they share the same connection pool.
hey -n 100 -c 10 "${BASE}/accounts/0.0.1234"
# → most requests fail or time out while EVM flood is active
```

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
