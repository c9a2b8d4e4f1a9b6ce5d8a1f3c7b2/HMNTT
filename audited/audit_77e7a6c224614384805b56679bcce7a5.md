### Title
Missing jOOQ DSLContext Query Timeout Enables Connection Pool Exhaustion via Unbounded Query Execution

### Summary
The `findAll` method in `NftAllowanceRepositoryCustomImpl` executes jOOQ queries through a `DSLContext` that has no query timeout configured. The `RestJavaConfiguration` customizer only sets a record mapper and schema rendering flag, leaving queries free to run indefinitely. Because the `GenericControllerAdvice` contains a `QueryTimeoutException` handler that can never fire without a configured timeout, the protection is illusory, and an attacker can hold database connections open until the pool is exhausted.

### Finding Description
**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/repository/NftAllowanceRepositoryCustomImpl.java`, lines 37–46:
```java
public Collection<NftAllowance> findAll(NftAllowanceRequest request, EntityId accountId) {
    boolean byOwner = request.isOwner();
    var bounds = request.getBounds();
    var condition = getBaseCondition(accountId, byOwner).and(getBoundConditions(bounds));
    return dslContext
            .selectFrom(NFT_ALLOWANCE)
            .where(condition)
            .orderBy(SORT_ORDERS.get(new OrderSpec(byOwner, request.getOrder())))
            .limit(request.getLimit())
            .fetchInto(NftAllowance.class);
}
```

`rest-java/src/main/java/org/hiero/mirror/restjava/config/RestJavaConfiguration.java`, line 38:
```java
return c -> c.set(domainRecordMapperProvider).settings().withRenderSchema(false);
```

No `withQueryTimeout(...)` is set on the jOOQ `Settings`, and no per-query `.queryTimeout()` is applied. The `importer` module's `JdbcTemplateConfiguration` does configure `queryTimeout` for `JdbcTemplate`, but that bean is entirely separate from the jOOQ `DSLContext` used in `rest-java`. The `web3` module has a `StatementInspector`-based timeout, also absent here.

**Root cause:** The `DefaultConfigurationCustomizer` bean in `RestJavaConfiguration` does not call `settings().withQueryTimeout(n)`, so every jOOQ statement issued through `DSLContext` in the `rest-java` module runs with no JDBC statement timeout.

**Failed assumption:** The developers assumed either that queries would always complete quickly (due to the `LIMIT` clause) or that the `QueryTimeoutException` handler in `GenericControllerAdvice` (line 121–124) would catch runaway queries. Neither assumption holds: the `LIMIT` clause bounds result-set size, not execution time, and the exception handler can only fire if a timeout is actually configured and thrown.

**Exploit flow:**
1. Attacker identifies the public NFT allowances REST endpoint (no authentication required).
2. Attacker crafts requests with range parameters (`account.id`, `token.id`) that produce complex multi-OR conditions via `getBoundConditions`, targeting an `accountId` with a large number of NFT allowances and an `APPROVED_FOR_ALL = true` filter that forces additional row filtering beyond the primary-key index scan.
3. Attacker sends a high volume of concurrent requests. Each request acquires a JDBC connection from HikariCP and holds it for the full duration of the query.
4. Because there is no statement timeout, slow queries (caused by lock contention, table bloat, or planner regressions) never get cancelled server-side.
5. The connection pool is exhausted; subsequent requests queue or fail, partitioning the application from the database.

### Impact Explanation
Connection pool exhaustion causes all database-dependent endpoints in the `rest-java` service to return errors or hang, effectively taking the mirror node REST API offline. Because the `nft_allowance` table can grow large on a production network and the `APPROVED_FOR_ALL` filter is not part of the primary-key index, queries can be made to scan many rows before satisfying the `LIMIT`. A sustained flood of such requests from a single unprivileged client is sufficient to exhaust a typical pool (10–50 connections).

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. No rate limiting is configured in the `rest-java` application properties (no `yml`/`yaml`/`properties` files exist under `rest-java/src/main/resources/` with any rate-limit or timeout settings). The attacker needs only an HTTP client capable of issuing concurrent requests; no credentials, tokens, or special knowledge are required. The attack is trivially repeatable and automatable.

### Recommendation
1. Set a global jOOQ query timeout in `RestJavaConfiguration`:
   ```java
   return c -> c.set(domainRecordMapperProvider)
                .settings()
                .withRenderSchema(false)
                .withQueryTimeout(5); // seconds
   ```
2. Alternatively, apply a per-query timeout in `NftAllowanceRepositoryCustomImpl.findAll` using jOOQ's `.queryTimeout(seconds)` on the select statement.
3. Mirror the pattern already used in the `importer` module (`JdbcTemplateConfiguration`, lines 33–35) by wiring `spring.jooq.sql-dialect` and `spring.jdbc.template.query-timeout` through `JdbcProperties` for the `rest-java` datasource.
4. Add connection-pool-level `connectionTimeout` and `maxLifetime` limits in HikariCP configuration to bound how long a connection can be held.
5. Add API-level rate limiting (e.g., via Spring's `RateLimiter` or an ingress-layer policy) to prevent connection-exhaustion floods regardless of query duration.

done
wait

# Observe: subsequent legitimate requests to any DB-backed endpoint hang or
# return 503/connection errors as the HikariCP pool is exhausted.
