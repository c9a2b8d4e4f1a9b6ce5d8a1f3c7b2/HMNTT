### Title
Deleted/Expired Contract Bytecode Returned via `findRuntimeBytecode()` — Missing Entity Status Check

### Summary
`ContractRepository.findRuntimeBytecode()` queries only the `contract` table with no join to the `entity` table, so it returns stored `runtime_bytecode` regardless of whether the corresponding entity is marked `deleted = true` or has expired. `ContractBytecodeReadableKVState.readFromDataSource()` calls this method directly for the `contractNum` and long-zero-address paths without any prior entity-status check, causing the mirror node's EVM state to present bytecode for contracts that no longer legally exist on the network.

### Finding Description

**Exact code path:**

`ContractRepository.findRuntimeBytecode()` (line 17):
```sql
select runtime_bytecode from contract where id = :contractId
```
The `contract` table has no `deleted` or `expiration_timestamp` column. Those fields live exclusively in the `entity` table. The query never joins to `entity`. [1](#0-0) 

`ContractBytecodeReadableKVState.readFromDataSource()` (lines 39–47) calls `findRuntimeBytecode()` unconditionally after resolving the entity ID: [2](#0-1) 

**Two distinct vulnerable paths inside `toEntityId()`:**

1. **`contractID.hasContractNum()` path (lines 50–51):** resolves the entity ID purely from the contract number with zero entity-table lookup — no deleted/expired check at all.
2. **Long-zero EVM address path (lines 54–55):** derives the entity ID from the address bytes directly, again with no entity-table lookup. [3](#0-2) 

**Contrast with the non-long-zero EVM address path (lines 57–60):** here `commonEntityAccessor.getEntityByEvmAddressAndTimestamp()` is called, which ultimately delegates to `findByEvmAddressAndDeletedIsFalse()` — so deleted contracts are correctly filtered out for that path only. [4](#0-3) 

**Root cause:** The `Contract` domain object has no lifecycle fields: [5](#0-4) 

The `entity` table's `deleted` and `expiration_timestamp` fields are never consulted for the `contractNum` / long-zero-address resolution paths before bytecode is fetched and returned.

### Impact Explanation

The mirror node's EVM state layer (`ContractBytecodeReadableKVState`) is the authoritative source for bytecode during `eth_getCode` queries and `eth_call` / `eth_estimateGas` execution. When a deleted or expired contract's bytecode is returned:

- **`eth_getCode`** returns non-empty bytecode for a deleted contract, violating EVM semantics (deleted contracts must return `0x`).
- **`eth_call` / `eth_estimateGas`** targeting a deleted contract's long-zero address will find bytecode present and attempt to execute the contract's logic, producing results that diverge from the canonical network state.
- Downstream tooling (DeFi protocols, wallets, indexers) that uses `eth_getCode != 0x` as a liveness check will incorrectly treat deleted contracts as live, potentially routing calls or funds to addresses that are no longer valid on the network.

This matches the stated scope: a bug in the mirror-node layer that results in unintended smart contract behavior with no direct fund loss from the mirror node itself, but with concrete potential for misbehavior in dependent systems.

### Likelihood Explanation

Any unprivileged user can trigger this with a single JSON-RPC call:
```
eth_getCode(<deleted_contract_long_zero_address>, "latest")
```
No authentication, no special role, no on-chain transaction required. The attacker only needs to know (or enumerate) the numeric ID of a previously deleted contract. Contract IDs are sequential and publicly visible on the network. The result is also cached (`@Cacheable`) so a single successful call poisons the cache for all subsequent callers until eviction. [1](#0-0) 

### Recommendation

In `ContractBytecodeReadableKVState.readFromDataSource()`, before calling `findRuntimeBytecode()`, verify the entity is neither deleted nor expired for all resolution paths. Concretely:

1. After resolving `entityId` (regardless of which branch of `toEntityId()` was taken), call `commonEntityAccessor.get(entityId, timestamp)` (which already uses `findByIdAndDeletedIsFalse` / `findActiveByIdAndTimestamp`) and return `null` if the entity is absent or deleted.
2. Alternatively, rewrite `findRuntimeBytecode` to join the `entity` table:
   ```sql
   select c.runtime_bytecode
   from contract c
   join entity e on e.id = c.id
   where c.id = :contractId
     and (e.deleted is not true)
     and (e.expiration_timestamp is null or e.expiration_timestamp > extract(epoch from now()) * 1e9)
   ```

### Proof of Concept

**Preconditions:**
- A contract with numeric ID `N` was deployed and later deleted via `ContractDeleteTransaction` on the Hiero network.
- The mirror node has ingested the deletion (entity row has `deleted = true`).
- The `contract` table still contains the `runtime_bytecode` row for ID `N`.

**Steps:**
1. Compute the long-zero EVM address: `0x000000000000000000000000000000000000000N` (20-byte, left-zero-padded).
2. Send:
   ```bash
   curl -X POST http://<mirror-node>/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     -d '{"jsonrpc":"2.0","method":"eth_getCode",
          "params":["0x000000000000000000000000000000000000000N","latest"],
          "id":1}'
   ```
3. **Expected (correct) result:** `"result": "0x"` — deleted contract has no code.
4. **Actual result:** `"result": "0x<non-empty bytecode>"` — `findRuntimeBytecode` returns the stored bytes without checking `entity.deleted`.

        if (contractID.hasContractNum()) {
            return entityIdFromContractId(contractID);

            final var evmAddress = contractID.evmAddress().toByteArray();
            if (isLongZeroAddress(evmAddress)) {
                return DomainUtils.fromEvmAddress(evmAddress);
            } else {

                        .map(Entity::toEntityId)
                        .orElse(EntityId.EMPTY);
            }
        }
        return EntityId.EMPTY;
    }
