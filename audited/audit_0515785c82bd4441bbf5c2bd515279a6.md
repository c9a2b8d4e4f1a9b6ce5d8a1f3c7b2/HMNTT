### Title
Unbounded OTel Label Cardinality via Unsanitized `req.path` in `toOpenApiPath()`

### Summary
For any request that does not match a registered Express route, `toOpenApiPath()` falls back to `req.path` — the raw, user-controlled URL path — and uses it verbatim as an OpenTelemetry metric label. Because the OpenTelemetry SDK stores one in-memory time-series per unique label combination, an unprivileged attacker can create an unbounded number of unique time-series (or inflate per-series memory with arbitrarily long path strings) by sending requests to distinct, never-matched paths, causing unbounded heap growth.

### Finding Description

**Exact code path:**

`rest/middleware/metricsHandler.js`, `toOpenApiPath()`, lines 13–31:

```js
const toOpenApiPath = (req, res) => {
  let path = res.locals[requestPathLabel];   // not set for 404 routes

  if (!path) {
    if (!req.route) {                        // null for unmatched routes
      path = req.path;                       // ← raw attacker-controlled value
    } else {
      path = (req.baseUrl ?? '') + req.route?.path;
    }
  }

  path = path.replace(/:([^/]+)/g, '{$1}');
  ...
  return path;
};
```

The returned value is immediately used as a label on four OTel instruments (lines 192, 207–210):

```js
const labels = {method, path, code};
requestTotalCounter.add(1, labels);
durationHistogram.record(duration, labels);
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(responseSize, labels);
```

**Root cause:** The function has two branches for resolving the path label. For matched routes it uses the static Express route pattern (e.g. `/api/v1/accounts/:id`), which has bounded cardinality. For *unmatched* routes (`req.route === null`) it uses `req.path` with no length cap, no normalization, and no cardinality guard. Every distinct path string produces a new time-series entry in the OTel SDK's in-memory `MetricStorage` map, and those entries are never evicted.

**Why existing checks fail:**

- `res.locals[requestPathLabel]` is only populated by `responseHandler` / `contractController` for successfully matched routes; it is `undefined` for 404s.
- `requestNormalizer.js` only normalizes query parameters for known OpenAPI routes; it does not touch the path for unmatched requests.
- The `metricsHandler` is registered globally before any route handler (`server.js` line 91), so it fires for every request including 404s.
- There is no rate-limit, path-length limit, or OTel cardinality cap configured anywhere in the codebase.

### Impact Explanation

Each unique `{method, path, code}` triple allocates a new `MetricData` record in the SDK's in-memory store for each of the four instruments. With deeply nested paths (e.g. `/a/b/c/.../z` repeated with varying segments), both the *number* of time-series and the *size* of each label string grow without bound. Sustained attack leads to heap exhaustion and eventual OOM-kill of the Node.js process, causing a full denial of service of the REST API. Even below OOM, the Prometheus `/metrics` scrape endpoint must serialize all stored series, causing CPU spikes and slow scrapes.

### Likelihood Explanation

The precondition is zero: no authentication, no special headers, no prior knowledge of the API. Any HTTP client can send `GET /aaaa/bbbb/cccc/...` in a tight loop. The attack is trivially scriptable, repeatable across restarts (each restart resets the store, but the attacker simply resumes), and requires no coordination. The metrics feature is opt-in via `config.metrics.enabled`, but when enabled (the intended production configuration) the surface is fully exposed.

### Recommendation

1. **Normalize unmatched paths to a sentinel**: In `toOpenApiPath()`, when `req.route` is `null` and `res.locals[requestPathLabel]` is unset, return a fixed string such as `"unknown"` or `apiPrefix + "/unknown"` instead of `req.path`. This collapses all 404 traffic into a single time-series.
2. **Enforce a path-length guard**: Before using any path as a label, truncate or reject strings exceeding a safe bound (e.g. 256 characters).
3. **Enable OTel SDK cardinality limits**: Configure `MetricReader` with an explicit `cardinalitySelector` or use the SDK's built-in overflow attribute (`otel.metric.overflow`) to cap per-instrument series count.

### Proof of Concept

```bash
# Send 10 000 requests each with a unique deeply-nested path
for i in $(seq 1 10000); do
  curl -s "http://<mirror-node-host>/seg$i/a/b/c/d/e/f/g/h/i/j/k/l/m/n/o/p" &
done
wait

# Observe heap growth via the metrics endpoint
curl http://<mirror-node-host>/swagger/metrics/ | grep nodejs_process_memory_heap_used_bytes
```

Each request lands on an unmatched route → `req.route` is `null` → `path = req.path` (e.g. `/seg1/a/b/c/…`) → four new OTel time-series created per unique path. After 10 000 iterations, 40 000 new time-series exist in the heap. Repeating with longer path strings (up to the HTTP parser's URL limit, ~8 KB) further inflates per-series memory.
