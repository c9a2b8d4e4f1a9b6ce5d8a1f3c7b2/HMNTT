### Title
Unbounded JSON Deserialization of `Metadata` Field in POST /mempool Enables CPU Exhaustion by Unauthenticated Attackers

### Summary
The `/mempool` HTTP endpoint accepts a `NetworkRequest` body whose `Metadata` field is typed as `map[string]interface{}`, imposing no depth or size constraint on JSON deserialization. Because no `http.MaxBytesReader` is applied anywhere in the middleware chain, an unauthenticated attacker can POST arbitrarily large or deeply nested JSON that forces Go's `encoding/json` decoder to consume unbounded CPU building recursive map/interface trees — all before the `mempoolAPIService.Mempool()` function is even invoked to return `ErrNotImplemented`. The existing `ReadTimeout` only bounds network I/O, not the CPU time spent parsing bytes already in memory.

### Finding Description

**Code path:**

`rosetta/main.go` wires the controller with no body-size guard: [1](#0-0) 

The HTTP server is configured with timeouts but no `MaxBytesReader`: [2](#0-1) 

The middleware chain (MetricsMiddleware → TracingMiddleware → CorsMiddleware → Router) adds no body-size limit: [3](#0-2) [4](#0-3) [5](#0-4) 

The service itself discards the parsed request entirely: [6](#0-5) 

**Root cause:**

The rosetta-sdk-go `MempoolAPIController` handler (vendor code, not in this repo) performs the following sequence for every POST /mempool:
1. Reads the full request body from the network (bounded by `ReadTimeout` = 5 s default)
2. Calls `json.NewDecoder(r.Body).Decode(&networkRequest)` — **unbounded CPU**, no depth limit
3. Calls `asserter.MempoolRequest(&networkRequest)` — validates only `NetworkIdentifier`, ignores `Metadata`
4. Calls `mempoolAPIService.Mempool()` — returns `ErrNotImplemented`

Step 2 is the attack surface. `types.NetworkRequest.Metadata` is `map[string]interface{}`. Go's `encoding/json` value decoder builds this tree with recursive calls proportional to nesting depth. There is no depth limit in the standard library decoder. A payload with N levels of nesting causes O(N) recursive decoder frames and O(N) allocations before any application logic runs.

**Why existing checks fail:**

- `ReadTimeout` (5 s default) only sets a deadline on the TCP read syscall. Once bytes are buffered in memory, the JSON decoder runs in the handler goroutine with no deadline.
- `ReadHeaderTimeout` (3 s) only covers headers, not the body.
- No `http.MaxBytesReader` is applied anywhere (confirmed: zero matches in `rosetta/**/*.go`).
- No rate-limiting middleware exists in the rosetta server.
- The asserter does not inspect or bound `Metadata`. [7](#0-6) 

### Impact Explanation
An unauthenticated attacker can exhaust CPU on the rosetta server by flooding POST /mempool with deeply nested `Metadata` payloads. Because the endpoint is publicly reachable (no authentication), each connection can deliver a body that fits within 5 seconds of network I/O (potentially tens of megabytes on a LAN or fast WAN) and then forces the server to spend unbounded CPU parsing it. With concurrent connections, this degrades or denies service to all rosetta endpoints sharing the same process, including construction and block endpoints used by exchanges. Sufficiently deep nesting (tens of thousands of levels) can also trigger goroutine stack growth to the point of OOM or a runtime crash.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond the public Rosetta API specification, and no prior access. The payload is trivially generated (`{"a":{"a":{"a":...}}}` repeated N times). It is repeatable at will and can be parallelized across many connections. The endpoint is intentionally exposed in online mode and is reachable by any network peer.

### Recommendation
1. Apply `http.MaxBytesReader` in the request handler or as a middleware wrapping the mux, limiting the body to a small fixed size (e.g., 64 KB) appropriate for a `NetworkRequest`.
2. Add a JSON depth limit by using a custom decoder or a library such as `github.com/bytedance/sonic` or a wrapper that counts nesting depth and returns an error beyond a threshold (e.g., 32 levels).
3. Add rate limiting (e.g., `golang.org/x/time/rate` token bucket) per source IP on all POST endpoints.
4. Consider returning 501 at the HTTP routing layer for `/mempool` without deserializing the body at all, since the endpoint is permanently unimplemented.

### Proof of Concept
```bash
# Generate a deeply nested JSON payload (~50,000 levels)
python3 -c "
depth = 50000
payload = '{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"testnet\"},\"metadata\":{'
payload += '\"a\":' * depth
payload += '1'
payload += '}' * depth
payload += '}}'
print(payload)
" > /tmp/deep.json

# Send to the mempool endpoint (repeat concurrently for amplification)
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/mempool \
    -H 'Content-Type: application/json' \
    --data-binary @/tmp/deep.json &
done
wait
```

Expected result: server CPU spikes to 100% per core for the duration of JSON parsing on each goroutine; legitimate requests time out or are queued indefinitely. The server eventually returns `{"code":501,...}` for each request, but only after exhausting CPU resources.
