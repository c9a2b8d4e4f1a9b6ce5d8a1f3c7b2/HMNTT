### Title
Historical Call Data Spoofing via Post-Transaction File Update in `getContractResultsByTransactionIdOrHash`

### Summary
The `getContractResultsByTransactionIdOrHash` function retrieves call data for Ethereum transactions that offloaded their payload to a Hedera file (`callDataId`), but fetches the **current latest** file contents with no timestamp bound. Because any Hedera user can create a file, submit an Ethereum transaction referencing it, and then update that file afterward, the mirror node API will permanently report tampered call data for that historical transaction, deceiving any third party querying the endpoint.

### Finding Description

**Exact code path:**

`rest/controllers/contractController.js`, `getContractResultsByTransactionIdOrHash()`, line 1183ŌĆō1184: [1](#0-0) 

```js
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
```

`FileDataService.getLatestFileDataContents` in `rest/service/fileDataService.js` lines 85ŌĆō88: [2](#0-1) 

With an empty `whereQuery`, the inner SQL (`latestFileContentsQuery`) has no `consensus_timestamp` upper bound: [3](#0-2) 

The CTE `latest_create` selects `max(consensus_timestamp)` with **no upper bound**, so it always resolves to the most recently written version of the file, regardless of when the referencing Ethereum transaction was executed.

**Root cause / failed assumption:**

The code assumes the `callDataId` file is immutable after the transaction executes. In Hedera, files are mutable: the file's admin key holder (the creator) can issue `FileUpdate` or `FileAppend` transactions at any time. The correct timestamp-bounded query already exists in the same service: [4](#0-3) 

`getFileData(fileId, timestamp)` constrains results to `consensus_timestamp <= $2`. The importer itself explicitly distinguishes between current-state and historical-state retrieval: [5](#0-4) 

The REST layer ignores this distinction entirely.

**The same bug exists in `getContractResultsByTimestamp` at line 1023:** [6](#0-5) 

**Exploit flow:**

1. Attacker (any regular Hedera user, no special privileges) issues a `FileCreate` transaction, storing arbitrary call data bytes in file `0.0.X`.
2. Attacker submits an `EthereumTransaction` with `callData` field empty and `callDataId = 0.0.X`. The consensus node executes the transaction using the original file contents.
3. After the transaction reaches consensus, the attacker issues a `FileUpdate` on `0.0.X` with replacement bytes of their choosing (e.g., an innocuous-looking ABI call).
4. Any subsequent call to `GET /api/v1/contracts/results/{txHash}` or `GET /api/v1/contracts/{id}/results/{timestamp}` returns the **replacement** bytes in the `function_parameters` field, not the bytes that were actually executed.

**Existing check shown insufficient:**

`utils.isValidUserFileId(ethTransaction?.callDataId)` only validates that the file ID is in the user-file range (not a system file). It does not prevent the stale-data read. [1](#0-0) 

### Impact Explanation

The mirror node is the canonical read API for Hedera historical data. Block explorers, auditors, DeFi front-ends, and compliance tools rely on `function_parameters` to reconstruct what a contract call did. By replacing the file contents after execution, an attacker can make a malicious transaction (e.g., a large token drain, a governance attack payload) appear to have invoked a benign function. The actual on-chain state is unaffected, but the mirror node's representation of the transaction is permanently falsified for all downstream consumers. Severity: **MediumŌĆōHigh** (data integrity / fraud enablement against third-party observers).

### Likelihood Explanation

The preconditions are trivially met by any Hedera account holder: file creation costs a small HBAR fee, and `FileUpdate` is a standard, permissionless operation for the file's key holder. No privileged access, no exploit of a protocol bug, and no race condition is required ŌĆö the attacker simply waits for their transaction to finalize before issuing the update. The attack is repeatable and undetectable after the fact via the mirror node API alone.

### Recommendation

Replace the unbounded `getLatestFileDataContents` call with the timestamp-bounded `getFileData` call, using the transaction's own consensus timestamp as the upper bound:

```js
// In getContractResultsByTransactionIdOrHash (line 1184) and
// getContractResultsByTimestamp (line 1023):
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getFileData(
    ethTransaction.callDataId,
    ethTransaction.consensusTimestamp   // pin to transaction time
  );
}
```

`getFileData` uses `getFileDataQuery` which enforces `consensus_timestamp <= $2`, returning the file state as it existed at execution time, consistent with how the importer's `getFileAtTimestamp` works.
