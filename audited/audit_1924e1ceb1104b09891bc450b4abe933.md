### Title
Slow-Drip Block Node Attack: Negative Timeout Causes Silent Stream Abort and Block Data Loss

### Summary
`BlockAssembler.timeout()` computes a sliding deadline as `timeout.toMillis() - stopwatch.elapsed(MILLISECONDS)` with no floor at zero. When an attacker-controlled block node sends the first `BlockItemSet` (starting the stopwatch) and then withholds subsequent items until the deadline is exhausted, `timeout()` returns zero or a negative value. `grpcCall.read(0_or_negative, MILLISECONDS)` returns `null` immediately (Java's `LinkedBlockingQueue.poll` with `timeout ≤ 0` is a non-blocking poll), the while-loop exits silently, and the partially-assembled block is discarded without any error, penalty, or retry signal.

### Finding Description

**Exact code path — `BlockNode.java`:**

The stopwatch starts on the very first call to `timeout()`, before the first `read()`:

```java
// lines 274-281
long timeout() {
    if (!stopwatch.isRunning()) {
        stopwatch.start();          // starts once, never reset until onEndOfBlock
        return timeout.toMillis();  // full budget on first call
    }
    return timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS); // shrinks every call
}
``` [1](#0-0) 

The loop passes this value directly to `grpcCall.read()`:

```java
// line 138
while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
``` [2](#0-1) 

**Root cause:** There is no `Math.max(0, …)` guard and no floor on the return value of `timeout()`. Once `stopwatch.elapsed() ≥ timeout.toMillis()`, the method returns `0` or a negative `long`. Java's `LinkedBlockingQueue.poll(long timeout, TimeUnit unit)` (which backs gRPC's `BlockingClientCall.read`) returns `null` immediately when `timeout ≤ 0` and no item is already buffered. The while-loop condition becomes `null != null → false`, so the loop exits cleanly.

**Why existing checks are insufficient:**

- `onError()` is only invoked inside the two `catch` blocks (lines 159–164). A `null` return from `read()` is not an exception — it is a normal loop-exit path. `onError()` is never called. [3](#0-2) 

- `errors.set(0)` is only executed inside the loop body (line 157). A timeout-exit never increments `errors`, so `maxSubscribeAttempts` is never reached and the node is never marked inactive. [4](#0-3) 

- After the loop, `assembler.pending` is never inspected. There is no check like `if (!assembler.pending.isEmpty()) throw …`. The partially-assembled block is silently discarded. [5](#0-4) 

- `stopwatch.reset()` (which stops and zeroes the stopwatch) is only called inside `onEndOfBlock()` (line 268), which is never reached in this attack path. [6](#0-5) 

**Exploit flow:**

1. Attacker operates (or compromises) a block node that the mirror node is configured to subscribe to.
2. Mirror node calls `streamBlocks()`. `assembler.timeout()` is evaluated: stopwatch starts, returns `T` ms (full budget).
3. `grpcCall.read(T, MILLISECONDS)` blocks.
4. Attacker sends exactly one `BlockItemSet` containing a `BLOCK_HEADER` item at time `T - ε` (just before the deadline). The read returns; `onBlockItemSet()` appends to `pending`.
5. `assembler.timeout()` is called again. Elapsed ≈ `T - ε`, so it returns `≈ ε` ms.
6. Attacker withholds the next `BlockItemSet` for `ε + 1` ms.
7. `grpcCall.read(≤0, MILLISECONDS)` returns `null` immediately.
8. While-loop exits. `streamBlocks()` returns normally. No exception, no `onError()`, no error counter increment.
9. The block (and all its transactions) is silently dropped. `blockStreamConsumer` is never called.

### Impact Explanation

Every transaction in the targeted block is permanently suppressed from the mirror node's perspective for that streaming session. Because no exception is thrown and `onError()` is never called, the node's error counter stays at zero and the node remains `active = true`. The attacker can repeat the attack on every subsequent block indefinitely without ever triggering the cooldown/readmit mechanism. This constitutes a targeted, repeatable, undetected denial-of-service against block ingestion — effectively allowing selective censorship of any block the attacker chooses.

### Likelihood Explanation

The precondition is control over one registered block node. Block nodes are external infrastructure components; a compromised or malicious operator satisfies this requirement without any privileged access to the mirror node itself. The timing manipulation required (send one item near the deadline, then stall) is trivially implementable in any gRPC server implementation. The attack is repeatable with zero cost per attempt and leaves no error trace in the mirror node's metrics or logs beyond a silent loop exit.

### Recommendation

1. **Clamp the return value of `timeout()` to a minimum of `1` ms** (or a configurable minimum) so `grpcCall.read()` always blocks for at least one millisecond:
   ```java
   return Math.max(1L, timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS));
   ```
2. **Detect incomplete blocks on loop exit**: after the while-loop, check `assembler.pending.isEmpty()`. If it is not empty, throw a `BlockStreamException` (which will invoke `onError()` and increment the error counter):
   ```java
   if (!assembler.pending.isEmpty()) {
       throw new BlockStreamException("Stream ended with a partially assembled block");
   }
   ```
3. **Use a per-item deadline reset** rather than a single sliding window, or use gRPC's built-in `withDeadlineAfter` on the call itself, which enforces the timeout at the transport layer and raises a `StatusRuntimeException` on expiry rather than returning `null`.

### Proof of Concept

**Reproducible steps:**

1. Stand up a malicious gRPC server implementing `BlockStreamSubscribeService`.
2. Configure the mirror node to use this server as its sole block node.
3. On receiving a `SubscribeBlockStream` request, record the current time `t0`.
4. After `(timeout - 50)` ms, send one `SubscribeStreamResponse` containing a `BlockItemSet` with a single `BLOCK_HEADER` item.
5. Sleep for `100` ms (ensuring `stopwatch.elapsed() > timeout.toMillis()`).
6. Observe: `streamBlocks()` returns without calling the `onBlockStream` consumer. No exception is logged. The block node's error counter remains 0. The block is silently dropped.
7. Repeat for every block number to achieve persistent block suppression.
