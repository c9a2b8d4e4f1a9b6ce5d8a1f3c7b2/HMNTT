### Title
Unauthenticated `/metrics` Endpoint Exposes Inflight Construction Request Gauge to Any External User

### Summary
The `Routes()` function in `rosetta/app/middleware/metrics.go` registers the `/metrics` endpoint using `promhttp.Handler()` with no authentication or access control. Any unprivileged external user can poll this endpoint to read `hiero_mirror_rosetta_request_inflight`, which is labeled by `{method, route}` and directly reveals the real-time count of concurrent in-flight HTTP requests per route — including Rosetta construction endpoints used for smart contract creation transactions. No credentials, tokens, or network-level restrictions are enforced anywhere in the middleware chain.

### Finding Description
**Code path:**

- `rosetta/app/middleware/metrics.go`, lines 34–37: `requestInflightGauge` is a `GaugeVec` with labels `{method, route}`, registered into `prometheus.DefaultRegisterer`.
- Lines 64–73 (`Routes()`): The `/metrics` path is served by `promhttp.Handler().ServeHTTP` — the default Prometheus HTTP handler — with no wrapping auth middleware.
- `rosetta/main.go`, lines 217–219: The middleware chain is `MetricsMiddleware → TracingMiddleware → CorsMiddleware`. None of these enforce authentication.
- `rosetta/app/middleware/trace.go`, line 19: `/metrics` is listed in `internalPaths` — but this only suppresses INFO-level logging; it applies **zero access restriction**.

**Root cause:** The failed assumption is that `/metrics` is an "internal" path. The code marks it as internal for logging purposes only (`internalPaths` map in `trace.go`), but the HTTP server binds all routes — including `/metrics` — on the same public port (`rosettaConfig.Port`) with no IP allowlist, bearer token check, or any other gatekeeping.

**Exploit flow:**
1. Attacker discovers the Rosetta node's public port (default behavior, no firewall assumed).
2. Attacker sends `GET /metrics` — receives HTTP 200 with full Prometheus text exposition.
3. Response contains `hiero_mirror_rosetta_request_inflight{application="rosetta",method="POST",route="/construction/submit"}` (and other construction sub-routes).
4. By polling repeatedly (e.g., every 100 ms), the attacker observes the gauge rising and falling, directly inferring when smart contract `ContractCreate` transactions are being constructed and submitted.
5. This timing signal can be correlated with on-chain activity to de-anonymize transaction originators or front-run pending construction flows.

**Why existing checks fail:** `server.CorsMiddleware` only sets CORS headers. `TracingMiddleware` only logs. `MetricsMiddleware` only instruments. None gate access. There is no authentication middleware registered anywhere in `rosetta/`.

### Impact Explanation
An attacker gains a real-time side-channel into the node's transaction processing pipeline. The `hiero_mirror_rosetta_request_inflight` gauge, broken down by route, reveals exactly how many smart contract construction requests are concurrently in-flight. Combined with `hiero_mirror_rosetta_request_duration` histograms and `hiero_mirror_rosetta_request_bytes` histograms (also exposed), an attacker can profile processing latency and payload sizes per construction step. While no funds are directly drained, this constitutes unintended information disclosure about network transaction state, consistent with the Medium scope defined for this finding.

### Likelihood Explanation
Exploitation requires only an HTTP client and network reachability to the Rosetta port — no credentials, no special tooling, no prior knowledge beyond the port number. The attack is trivially repeatable, stateless, and leaves minimal forensic trace (requests to `/metrics` are logged only at DEBUG level per `trace.go` line 55–58, meaning they are invisible in default INFO logging). Any internet-facing deployment is immediately exploitable.

### Recommendation
1. **Separate the metrics port**: Serve `/metrics` on a dedicated internal-only port (e.g., via a second `http.Server`) that is firewalled from external access.
2. **Add IP allowlist middleware**: If a single port must be used, wrap the `promhttp.Handler()` in middleware that restricts access to trusted CIDR ranges (e.g., Prometheus scrape targets only).
3. **Remove `/metrics` from the public router**: Do not register `metricsController` in `server.NewRouter(...)` alongside public API controllers; instead, mount it on a separate mux bound to a loopback or internal interface.

### Proof of Concept
```bash
# No credentials required. Replace HOST:PORT with the Rosetta node's address.
curl -s http://HOST:PORT/metrics | grep hiero_mirror_rosetta_request_inflight

# Poll every 100ms to observe real-time construction request concurrency:
while true; do
  curl -s http://HOST:PORT/metrics \
    | grep 'hiero_mirror_rosetta_request_inflight.*construction'
  sleep 0.1
done
```
Expected output (when construction requests are active):
```
hiero_mirror_rosetta_request_inflight{application="rosetta",method="POST",route="/construction/submit"} 3
```
This confirms 3 concurrent smart contract construction submissions are in-flight, with no authentication required.
