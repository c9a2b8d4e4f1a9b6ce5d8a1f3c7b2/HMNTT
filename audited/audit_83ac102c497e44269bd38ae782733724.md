### Title
Unauthenticated GraphQL Variable-Driven DB Connection Pool Exhaustion via Uncached `findByEvmAddress` Calls

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl` issues a raw database query on every invocation with no result caching. Because `CachedPreparsedDocumentProvider` only caches the parsed GraphQL document (keyed on the static query string), an attacker using GraphQL variables sends a single constant query string — hitting the parse cache every time — while still triggering a fresh `entityRepository.findByEvmAddress()` DB call on each execution. With no authentication, no rate limiting, and no result cache anywhere on this path, an unprivileged attacker can sustain a high-frequency request flood that exhausts the database connection pool well above the 30% baseline threshold.

### Finding Description
**Code path:**

- `CachedPreparsedDocumentProvider.java:26` — cache key is `executionInput.getQuery()`, the raw query string. This caches only parsing/validation, not execution results.
- `EntityServiceImpl.java:34–41` — `getByEvmAddressAndType()` has no `@Cacheable` or any result-caching annotation; every call proceeds unconditionally to the repository.
- `EntityRepository.java:17` — `findByEvmAddress()` is a native SQL query with no second-level cache.
- `GraphQlConfiguration.java:43–45` — only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` are installed; no rate-limiting instrumentation exists.
- No security configuration exists anywhere under `graphql/src/main/java/` (confirmed by search).

**Root cause / failed assumption:** The design assumes the preparsed document cache is a meaningful defense against repeated execution. It is not — it only deduplicates parsing overhead. The actual DB call path has no caching, no throttling, and no authentication gate.

**Exploit flow:**
1. Attacker crafts one constant query string using a variable: `query Q($a: String!) { account(input: {evmAddress: $a}) { id } }`.
2. Attacker sends this query in rapid parallel HTTP POST requests to `/graphql`, varying `$a` (or even reusing the same value).
3. Each request: parse cache is hit (no parsing cost) → `AccountController.account()` → `EntityServiceImpl.getByEvmAddressAndType()` → `entityRepository.findByEvmAddress()` → native SQL against the DB.
4. The DB connection pool is saturated; legitimate queries queue or fail.

**Why existing checks fail:**
- `MaxQueryComplexityInstrumentation(200)`: a single-field `account` query scores far below 200.
- `MaxQueryDepthInstrumentation(10)`: depth of 1–2 is trivially within limit.
- `CachedPreparsedDocumentProvider`: caches parsing only; execution is unconstrained.
- No IP-based or token-based rate limiting exists in the codebase.

### Impact Explanation
Every concurrent attacker request holds a JDBC connection for the duration of the `SELECT * FROM entity WHERE evm_address = ?` query. With a default HikariCP pool (typically 10 connections) and no back-pressure, even a modest flood (e.g., 50 concurrent requests/second from a single host) saturates the pool, causing `Connection is not available, request timed out` errors for all other consumers of the same datasource. This directly maps to the stated threat: >30% increase in DB resource consumption without brute-force query complexity.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication (confirmed: zero security annotations or filters in the graphql module). The attack requires only a standard HTTP client and knowledge of the GraphQL schema (which is introspectable by default in Spring GraphQL). It is trivially repeatable, scriptable, and requires no credentials or special privileges.

### Recommendation
1. **Add result caching** on `EntityServiceImpl.getByEvmAddressAndType()` using `@Cacheable` with a short TTL (e.g., 5–30 seconds), keyed on `(evmAddress, type)`.
2. **Add rate limiting** at the HTTP layer (e.g., Spring Cloud Gateway, Bucket4j, or a servlet filter) scoped per IP or API key, applied before GraphQL execution.
3. **Disable or restrict GraphQL introspection** in production to slow schema reconnaissance.
4. **Configure a DB connection pool timeout and queue limit** so pool exhaustion fails fast rather than cascading.

### Proof of Concept
```bash
# Constant query string; only the variable value changes (or stays the same)
QUERY='{"query":"query Q($a:String!){account(input:{evmAddress:$a}){id}}","variables":{"a":"0x0000000000000000000000000000000000000001"}}'

# Flood with 100 parallel requests (repeat indefinitely)
for i in $(seq 1 100); do
  curl -s -X POST http://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d "$QUERY" &
done
wait
# Repeat in a loop; DB connection pool exhausts within seconds.
# Each request bypasses no parse-cache overhead (same query string)
# but issues a fresh SELECT against the entity table.
