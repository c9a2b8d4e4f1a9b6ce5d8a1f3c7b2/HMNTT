### Title
Liveness Probe Spoofing via Unprivileged Creation of `/tmp/alive`

### Summary
The liveness healthcheck in `pinger/cmd/healthcheck/main.go` determines the pinger is alive solely by checking the existence and modification time of `/tmp/alive`. Because `/tmp` is world-writable on Linux (mode `1777`) and no file ownership validation is performed, any local user can create and periodically touch `/tmp/alive` to make the liveness probe permanently pass — even if the pinger heartbeat goroutine has never executed.

### Finding Description
**Code location:** `pinger/cmd/healthcheck/main.go`, `main()`, lines 27–38. [1](#0-0) 

The check calls `os.Stat("/tmp/alive")` and then inspects only `fi.ModTime()`. It never inspects `fi.Sys().(*syscall.Stat_t).Uid` to confirm the file is owned by the pinger process's UID.

The legitimate writer is the heartbeat goroutine in `pinger/main.go` lines 28–39: [2](#0-1) 

It writes to `/tmp/alive` every 15 seconds. The healthcheck accepts the file as valid if its `ModTime` is within 2 minutes — with no ownership or content integrity check.

**Root cause / failed assumption:** The design assumes only the pinger process can create or update `/tmp/alive`. This assumption fails because `/tmp` is universally writable (`drwxrwxrwt`) on Linux; the sticky bit only prevents *deletion* of other users' files, not *creation* of new ones.

**Exploit flow:**
1. Attacker gains local shell access (or `kubectl exec` into the pod if RBAC is misconfigured).
2. Before or after the pinger starts, attacker runs: `echo spoof > /tmp/alive`
3. Attacker keeps the file fresh: `while true; do touch /tmp/alive; sleep 60; done &`
4. Healthcheck is invoked by Kubernetes: `os.Stat` succeeds, `ModTime` is <2 min old → exits 0 → probe passes.

**Why existing checks are insufficient:**
- The age check (`age > 2*time.Minute`) only validates recency, not authenticity.
- There is no HMAC, no ownership check, no inode-pinning, and no use of a private directory.

### Impact Explanation
Kubernetes relies on the liveness probe to detect a dead or hung pinger and restart the pod. If an attacker can spoof `/tmp/alive`, a completely non-functional pinger (crashed, deadlocked, or never started) will never be restarted. The pinger's job is to submit periodic Hedera network transactions; a silently dead pinger means those transactions stop while the orchestration layer believes the service is healthy. This constitutes a silent availability failure with no automatic remediation.

### Likelihood Explanation
In a standard Kubernetes deployment the attacker needs shell access inside the container (e.g., via `kubectl exec`, a compromised sidecar, or a container escape). In a bare-metal or VM deployment any unprivileged OS user suffices. The technique requires only POSIX shell primitives (`touch`, `echo`, `sleep`) — no special tools or privileges. It is trivially repeatable and leaves no distinctive log entries.

### Recommendation
1. **Use a private, mode-0700 directory** owned by the pinger's UID (e.g., `/run/pinger/`) instead of `/tmp`. The heartbeat file should be written there so no other UID can create it.
2. **Validate file ownership at probe time**: read `fi.Sys().(*syscall.Stat_t).Uid` and compare it to `os.Getuid()`.
3. Alternatively, replace the file-based IPC with a Unix domain socket or an in-process HTTP liveness endpoint, both of which are not susceptible to world-writable directory races.

### Proof of Concept
```bash
# On the host or inside the container (no root required):

# 1. Create the spoofed heartbeat file
echo "spoof" > /tmp/alive

# 2. Keep it fresh in the background (refresh every 60 s, well within the 2-min window)
while true; do touch /tmp/alive; sleep 60; done &

# 3. Simulate what Kubernetes calls:
#    (assuming the healthcheck binary is built and available)
./healthcheck live
# Expected (and observed) exit code: 0  — probe passes
# Actual pinger heartbeat goroutine: never ran
```
