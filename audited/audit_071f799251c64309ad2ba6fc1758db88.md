### Title
Connection Pool Exhaustion via Unauthenticated `preQueryHint` Path in `getTopicMessages`

### Summary
Any unauthenticated request to `/api/v1/topics/:topicId/messages` with `limit` not equal to `1` (including the default limit) triggers a dedicated PostgreSQL client checkout via `pool.connect()` that holds the connection for the full duration of `BEGIN; SET LOCAL random_page_cost = 0` + query + `COMMIT`. Because the pool size is finite and no rate limiting protects this endpoint, a flood of concurrent requests exhausts the connection pool, causing all subsequent pool-dependent requests across the service to fail or block until `connectionTimeoutMillis` expires.

### Finding Description

**Code path:**

`rest/topicmessage.js` line 142–143:
```js
const queryHint = limit !== 1 ? constants.zeroRandomPageCostQueryHint : undefined;
const messages = await getMessages(query, params, queryHint);
``` [1](#0-0) 

`getMessages` passes `preQueryHint` directly to `pool.queryQuietly`: [2](#0-1) 

`queryQuietly` in `rest/utils.js` lines 1518–1527:
```js
if (!preQueryHint) {
  result = await this.query(query, params);   // pool-managed, connection returned immediately
} else {
  client = await this.connect();              // dedicated checkout from pool
  client.on('error', clientErrorCallback);
  await client.query(`begin; ${preQueryHint}`);
  result = await client.query(query, params); // connection held for full query
  await client.query('commit');
}
``` [3](#0-2) 

The connection is released only in the `finally` block after all three round-trips complete: [4](#0-3) 

**Root cause:** The `preQueryHint` branch unconditionally calls `this.connect()` (a dedicated checkout) instead of `this.query()` (pool-managed). The condition `limit !== 1` is always true for the default limit and for any user-supplied limit other than `1`, meaning virtually every real-world request to this endpoint takes the expensive path.

**Failed assumption:** The design assumes query latency is low enough that concurrent dedicated checkouts never saturate the pool. Under adversarial load this assumption breaks.

### Impact Explanation

The pg `Pool` is configured with `max: config.db.pool.maxConnections`: [5](#0-4) 

Once all slots are occupied, every new call to `pool.connect()` or `pool.query()` anywhere in the service blocks until `connectionTimeoutMillis` elapses, then throws. This affects not only topic-message queries but every other endpoint sharing the same pool (`pool.queryQuietly` is used service-wide). A sustained flood of concurrent requests to a single public endpoint can therefore degrade or deny service across 30%+ of processing capacity, matching the stated severity threshold.

### Likelihood Explanation

- **No authentication required**: the `/api/v1/topics/:topicId/messages` endpoint is publicly accessible.
- **No rate limiting visible** in the middleware stack for this route.
- **Trivially triggerable**: the default limit is not `1`, so a plain `GET /api/v1/topics/0.0.1/messages` with no parameters is sufficient.
- **Repeatable**: an attacker needs only a script sending concurrent HTTP requests; no special knowledge, credentials, or protocol access is required.
- **Amplifiable**: slow queries (large topics, wide timestamp ranges) extend connection hold time, reducing the number of concurrent requests needed to exhaust the pool.

### Recommendation

1. **Avoid dedicated checkout for hint-only sessions**: use `SET LOCAL` inside a single `pool.query` call wrapped in an explicit transaction, or use `pool.query('BEGIN')` / `pool.query('SET LOCAL ...')` / `pool.query(mainQuery)` / `pool.query('COMMIT')` with `pool.query` (not `pool.connect`) so the connection is returned to the pool between statements. Alternatively, set `random_page_cost` as a session-level default for the REST DB role rather than per-query.
2. **Apply rate limiting** on the `/api/v1/topics/:topicId/messages` endpoint to bound concurrent in-flight requests.
3. **Cap `connectionTimeoutMillis`** aggressively and add a semaphore/queue in front of `pool.connect()` calls to prevent unbounded blocking.

### Proof of Concept

```bash
# Send N concurrent requests (N >= pool.maxConnections) to exhaust the pool.
# Each request uses the default limit (not 1), triggering pool.connect().
for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>/api/v1/topics/0.0.1/messages" &
done
wait

# Subsequent requests to ANY endpoint will fail with connection timeout errors
# until the in-flight queries complete and release their dedicated connections.
curl -v "http://<mirror-node-host>/api/v1/transactions"
# Expected: 500 Internal Error / connection timeout from exhausted pool
```
