### Title
Unbounded Log Flooding via Unauthenticated Health Endpoint Polling in `restNetworkStakeHealth()`

### Summary
The `restNetworkStakeHealth()` method in `SubscriberHealthIndicator` calls `log.error()` unconditionally on every health check invocation when the upstream REST API returns a non-2xx status or is unreachable. The `/actuator/health/cluster` endpoint is publicly exposed via Kubernetes ingress with no authentication, no rate limiting, and no result caching, allowing any unprivileged external user to trigger unbounded log writes by flooding the endpoint while the REST API is degraded.

### Finding Description
**Code path:**

`monitor/src/main/java/org/hiero/mirror/monitor/health/SubscriberHealthIndicator.java`, `restNetworkStakeHealth()`, lines 93–122.

Two `log.error()` calls exist with no guard:

- Line 104: triggered on every non-2xx HTTP status from `/network/stake`
- Line 119: triggered on every `ConnectException`, `TimeoutException`, or any other error from the upstream call

Neither call is debounced, rate-limited, or cached. The public entry point is `health()` (line 59), which calls `restNetworkStakeHealth()` on every invocation with no memoization.

**Exposure:** `charts/hedera-mirror-monitor/values.yaml` lines 91–98 configure an ingress that routes `/actuator/health/cluster` to the monitor pod with no authentication annotation. The service is `ClusterIP` (line 291) but the ingress makes the path reachable from the internet.

**Root cause:** The assumption that health checks are called infrequently (by Kubernetes probes) is violated when the endpoint is internet-reachable. There is no caching layer between the HTTP request and the `log.error()` call, and no rate limiter on the actuator endpoint.

**Why existing checks fail:** The 5-second `.timeout(Duration.ofSeconds(5))` (line 107) limits per-connection throughput but does not prevent many concurrent connections from each independently triggering `log.error()`. Spring Boot Actuator's default configuration does not add authentication or rate limiting to health endpoints. No `@Cacheable` or reactive `.cache()` is applied to `restNetworkStakeHealth()`.

### Impact Explanation
When the upstream REST API is degraded (non-2xx or unreachable — a realistic condition during incidents, deployments, or database outages), every concurrent HTTP request to `/actuator/health/cluster` produces one `log.error()` write. At high concurrency (thousands of simultaneous connections), this generates continuous disk I/O proportional to the attacker's request rate. Log files fill rapidly, potentially exhausting the pod's writable filesystem (the `readOnlyRootFilesystem: true` securityContext still allows writes to the `/tmp` emptyDir volume and log sinks). This can cause the monitor process itself to stall or crash, eliminating observability of the production network during the very incident the attacker may be exploiting.

### Likelihood Explanation
The precondition — REST API returning non-2xx — occurs naturally during any backend degradation and is not under the attacker's control, but an attacker can simply wait or probe until it occurs. The endpoint requires no credentials, no API key, and no network-level restriction beyond reaching the ingress. A single attacker with a modest botnet or HTTP flood tool (e.g., `wrk`, `hey`, `ab`) can sustain thousands of concurrent connections. The attack is fully repeatable and stateless.

### Recommendation
1. **Cache the health result**: Apply `.cache(Duration.ofSeconds(X))` on the `restNetworkStakeHealth()` reactive chain (similar to how `ReleaseHealthIndicator` uses `.cache(properties.getCacheExpiry(), ...)` at line 82 of `ReleaseHealthIndicator.java`) so that repeated calls within the cache window reuse the last result and do not re-invoke `log.error()`.
2. **Throttle log output**: Replace bare `log.error(statusMessage)` with a rate-limited logger (e.g., Logback's `DuplicateMessageFilter` or a manual `AtomicLong` timestamp guard) so the same error is not written more than once per interval.
3. **Restrict the health endpoint**: Add ingress-level authentication (e.g., a Traefik `BasicAuth` middleware) or move the health endpoint to a non-public port, consistent with how other services in this chart restrict actuator access.

### Proof of Concept
**Preconditions:** The monitor's upstream REST API (`/network/stake`) is returning 5xx or is unreachable (e.g., during a deployment or database outage).

**Steps:**
```bash
# 1. Identify the public ingress hostname for the monitor cluster health path
HOST="<monitor-ingress-host>"

# 2. Flood the unauthenticated health endpoint with concurrent requests
wrk -t 50 -c 500 -d 60s "https://$HOST/actuator/health/cluster"

# 3. Observe on the monitor pod:
kubectl logs -f <monitor-pod> | grep "Network stake status"
# → Thousands of "Network stake status is DOWN with status code 503" lines per second

# 4. Check disk/log volume fill rate:
kubectl exec <monitor-pod> -- df -h /tmp
# → /tmp emptyDir fills rapidly; log appender stalls; pod becomes unresponsive
```

    private Mono<Health> restNetworkStakeHealth() {
        return restApiClient
                .getNetworkStakeStatusCode()
                .flatMap(statusCode -> {
                    if (statusCode.is2xxSuccessful()) {
                        return UP;
                    }

                    var status = statusCode.is5xxServerError() ? Status.DOWN : Status.UNKNOWN;
                    var statusMessage =
                            String.format("Network stake status is %s with status code %s", status, statusCode.value());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                })
                .timeout(Duration.ofSeconds(5))
                .onErrorResume(e -> {
                    var status = Status.UNKNOWN;
                    // Connection issue can be caused by database being down, since the rest API service will become
                    // unavailable eventually
                    var rootCause = ExceptionUtils.getRootCause(e);
                    if (rootCause instanceof ConnectException || rootCause instanceof TimeoutException) {
                        status = Status.DOWN;
                    }

                    var statusMessage =
                            String.format("Network stake status is %s with error: %s", status, e.getMessage());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                });
    }
