### Title
`NumberFormatException` on Zero-Length `value` Byte Array Crashes Opcode Replay in `OpcodeServiceImpl.getValue()`

### Summary
`OpcodeServiceImpl.getValue()` unconditionally calls `new BigInteger(ethereumTransaction.getValue())` without guarding against a zero-length byte array. The `BigInteger(byte[])` constructor mandates at least one byte and throws `NumberFormatException` on an empty array. Because zero-length `value` byte arrays are a documented, persisted state in the `ethereum_transaction` table, any unprivileged caller who supplies the hash of such a transaction will cause the opcode replay to abort and return no trace.

### Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`, lines 211–219:

```java
private BigInteger getValue(EthereumTransaction ethereumTransaction, ContractResult contractResult) {
    if (ethereumTransaction != null) {
        return new BigInteger(ethereumTransaction.getValue());   // line 213 — throws if value is byte[0]
    }
    ...
}
``` [1](#0-0) 

This is called unconditionally from `buildCallServiceParameters()` at line 173:
```java
.value(getValue(ethTransaction, contractResult).longValue())
``` [2](#0-1) 

Which is reached from `OpcodesController.getContractOpcodes()` → `opcodeService.processOpcodeCall()`: [3](#0-2) 

**Root cause — failed assumption:**
The code assumes `EthereumTransaction.value` always contains at least one byte. This assumption is false. The `value` field is a plain `byte[]` with no length constraint: [4](#0-3) 

**Evidence that zero-length values are persisted:**

1. `ConvertEthereumTransactionValueMigration` explicitly filters `length(value) > 0` in its SELECT, proving zero-length rows exist in the DB: [5](#0-4) 

2. `EthereumTransactionHandlerTest.updateTransactionEmptyWeiBars` explicitly sets `value = new byte[]{}` and asserts it is stored as-is: [6](#0-5) 

**Exception handling — why it is insufficient:**
`NumberFormatException` extends `IllegalArgumentException`. The web3 `GenericControllerAdvice` catches `IllegalArgumentException` and returns HTTP 400: [7](#0-6) 

The framework-level catch prevents a full server crash, but the opcode replay is still aborted — no trace is produced, and the caller receives a misleading `400 Bad Request` (implying bad user input) for a perfectly valid historical transaction hash. The `handleExceptionInternal` path for 4xx also leaks the raw exception message (`"Zero length BigInteger"`) to the caller: [8](#0-7) 

There is no null/empty guard anywhere between the DB read and the `new BigInteger(...)` call in the opcode service path.

### Impact Explanation
Any historical Ethereum transaction whose `value` column was stored as a zero-length byte array permanently cannot be replayed through the `/api/v1/contracts/results/{hash}/opcodes` endpoint. The endpoint returns HTTP 400 with an internal error string instead of the expected opcode trace. This is a targeted, repeatable denial-of-opcode-trace for specific transactions, with no funds at direct risk but with concrete loss of auditability/debuggability for those executions.

### Likelihood Explanation
The precondition (a zero-length `value` in the DB) is explicitly acknowledged by the codebase itself (migration SQL, test fixture). An unprivileged attacker needs only to know the Ethereum transaction hash of any such transaction — all transaction hashes are public on-chain data. No authentication, no special role, no rate-limit bypass is required beyond the standard `Accept-Encoding: gzip` header check. The trigger is deterministic and repeatable.

### Recommendation
Guard `getValue()` against null and zero-length arrays before constructing `BigInteger`:

```java
private BigInteger getValue(EthereumTransaction ethereumTransaction, ContractResult contractResult) {
    if (ethereumTransaction != null) {
        byte[] val = ethereumTransaction.getValue();
        if (ArrayUtils.isEmpty(val)) {
            return ZERO;
        }
        return new BigInteger(val);
    }
    if (contractResult.getAmount() != null) {
        return BigInteger.valueOf(contractResult.getAmount());
    }
    return ZERO;
}
```

The same pattern should be applied to the identical unguarded call in `EthereumTransactionHandler.updateContractResult()` (line 75) and `AbstractEthereumTransactionParser.getValue()` (line 70). [9](#0-8) [10](#0-9) 

### Proof of Concept

1. Identify (or insert in a test environment) an `ethereum_transaction` row where `value = ''` (zero-length bytea).
2. Note its `hash` column value, e.g. `0xabc123...`.
3. Send:
   ```
   GET /api/v1/contracts/results/0xabc123.../opcodes
   Accept-Encoding: gzip
   ```
4. Observe HTTP 400 response body containing `"Zero length BigInteger"` instead of an opcode trace.
5. The same hash on a node without this bug returns a full opcode trace — confirming the regression is in `getValue()`.
