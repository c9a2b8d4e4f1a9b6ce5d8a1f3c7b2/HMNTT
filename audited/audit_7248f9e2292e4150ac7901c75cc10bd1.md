### Title
Unauthenticated gRPC Topic Subscription with Non-Existent Entity ID Causes Unbounded DB Query and Log Amplification

### Summary
Any unauthenticated external client can send repeated `subscribeTopic` gRPC requests referencing a non-existent topic ID. Each request triggers a synchronous database existence check and a `log.warn()` call via `ProtoUtil.clientError()`, with no rate limiting or connection throttling on the gRPC endpoint. A sustained moderate-rate flood of such requests can amplify DB I/O, log I/O, and CPU overhead well beyond 30% of baseline.

### Finding Description

**Exact code path:**

1. `ConsensusController.subscribeTopic()` receives any unauthenticated gRPC request and calls `topicMessageService.subscribeTopic(filter)`. [1](#0-0) 

2. `TopicMessageServiceImpl.topicExists()` unconditionally queries the database via `entityRepository.findById(topicId.getId())`. When the entity is absent and `checkTopicExists` is `true` (the default), it throws `new EntityNotFoundException(topicId)`. [2](#0-1) 

3. `EntityNotFoundException(EntityId entityId)` calls `super(String.format(MESSAGE, entityId))`, which resolves to `MirrorNodeException(String message)`. [3](#0-2) [4](#0-3) 

4. The exception is caught by `.onErrorMap(ProtoUtil::toStatusRuntimeException)`. `toStatusRuntimeException()` matches `t instanceof EntityNotFoundException` and calls `clientError()`. [5](#0-4) 

5. `clientError()` unconditionally emits a `log.warn()` for every single request before returning the `NOT_FOUND` status to the caller. [6](#0-5) 

**Root cause:** There is no rate limiting, authentication, or connection-rate throttling on the gRPC service. The only concurrency control is `maxConcurrentCallsPerConnection = 5`, which is a per-connection limit — an attacker opens many connections to bypass it. [7](#0-6) 

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not applied to the gRPC service at all. [8](#0-7) 

**Failed assumption:** The design assumes that `clientError()` logging is a negligible cost and that connection-level concurrency limits are sufficient to bound server load. Neither holds when an attacker opens many connections and rapidly cycles through short-lived requests that each fail immediately.

### Impact Explanation

Each malicious request causes: (a) a synchronous DB SELECT on the entity table, (b) a `log.warn()` write, and (c) gRPC connection/stream setup and teardown. Because the `NOT_FOUND` response is returned immediately, the attacker's connection is freed instantly and can be reused or a new one opened. At a sustained rate of hundreds of requests per second across tens of connections, the cumulative DB query load and log I/O can realistically exceed 30% of baseline resource consumption on a lightly-to-moderately loaded node, degrading service for legitimate subscribers.

### Likelihood Explanation

No credentials, API keys, or special network access are required — the gRPC port (default 5600) is publicly exposed. The attack requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and knowledge of any non-existent topic ID (e.g., `0.0.999999999`). The pattern is trivially scriptable, repeatable, and does not require brute-force volume — a moderate sustained rate is sufficient given the absence of any server-side throttle.

### Recommendation

1. **Add gRPC-level rate limiting**: Implement a per-IP or global request rate limiter (e.g., using a token-bucket interceptor) on the gRPC server, analogous to the `ThrottleManagerImpl` already present in the `web3` module.
2. **Cache negative entity lookups**: Cache NOT_FOUND results in the existing `entityCacheSize` cache (currently only used for positive hits) to avoid repeated DB queries for the same non-existent ID. [9](#0-8) 
3. **Reduce log verbosity for client errors**: Consider downgrading or sampling `clientError()` log output (e.g., log at DEBUG, or rate-limit warn emissions per entity ID) to prevent log amplification.
4. **Limit total concurrent connections**: Configure a global maximum connection count on the Netty server, not just per-connection call limits.

```bash
# Install grpcurl: https://github.com/fullstorydev/grpcurl
# Target: mirror node gRPC at localhost:5600

for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 999999999}}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait

# Each request triggers:
# 1. DB SELECT on entity table for topicNum=999999999 (not found)
# 2. log.warn("Client error EntityNotFoundException: 0.0.999999999 does not exist")
# 3. gRPC NOT_FOUND response returned immediately
#
# Repeat in a loop to sustain load.
# Monitor server: watch DB query rate, log output rate, CPU usage.
# Expected: sustained >30% increase in DB query throughput and log I/O
# compared to idle baseline, with no authentication required.
```

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();

                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

                .build();

    }
```
