### Title
Unauthenticated IP Spoofing via Trusted Client-Controlled Headers in `getClientIpAddress()`

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the `X-Real-IP` and `X-Forwarded-For` headers supplied by the client, with no validation that the request originated from a trusted proxy. Any unprivileged external attacker can set these headers to an arbitrary value (e.g., a victim's IP address), causing all logged transaction requests to be attributed to the forged IP in the application's access logs.

### Finding Description
**Exact code path:** [1](#0-0) 

The function checks `X-Real-IP` first, then `X-Forwarded-For`, and only falls back to `r.RemoteAddr` if both are absent. There is no allowlist of trusted upstream proxies, no CIDR check on `r.RemoteAddr` before trusting the headers, and no sanitization of the header values. [2](#0-1) 

`TracingMiddleware` calls `getClientIpAddress()` and embeds the result directly into the log line (`log.Info(message)`). The logged IP is the sole basis for attributing which client originated each request.

**Root cause:** The code assumes these headers are set only by a trusted reverse proxy, but there is no enforcement of that assumption. HTTP headers are fully attacker-controlled.

**Exploit flow:**
1. Attacker sends a POST request to any Rosetta API endpoint (e.g., `/network/list`).
2. Attacker includes `X-Real-IP: <victim_IP>` (or `X-Forwarded-For: <victim_IP>`) in the request.
3. `getClientIpAddress()` returns `<victim_IP>` without any validation.
4. `TracingMiddleware` logs: `<victim_IP> POST /network/list 200 in 3ms`.
5. All audit/access logs now record the victim's IP as the originator of the attacker's requests.

**Why existing checks fail:** There are none. The fallback to `r.RemoteAddr` only triggers when both headers are completely absent — an attacker simply includes the header.

### Impact Explanation
The application's access logs are the historical record of which IP addresses submitted which transaction-related requests (e.g., `/construction/submit`, `/block/transaction`). By forging the source IP, an attacker can:
- Frame a victim's IP as the originator of suspicious or high-volume transaction requests.
- Pollute forensic/audit trails used for incident response.
- Evade IP-based rate limiting or blocking if such controls are added downstream based on these logs.

Severity is **Medium**: no funds are directly at risk, but the integrity of the audit log — the only record of request originators — is fully compromised by any unauthenticated external user.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and a single HTTP request with a crafted header. It is trivially repeatable, automatable, and undetectable from the server side (the real attacker IP is never logged). Any external user who can reach the Rosetta API endpoint can exploit this.

### Recommendation
Before trusting `X-Real-IP` or `X-Forwarded-For`, validate that `r.RemoteAddr` belongs to a configured set of trusted proxy CIDRs. Only then should the header value be used. Example fix:

```go
func getClientIpAddress(r *http.Request, trustedProxies []*net.IPNet) string {
    remoteIP, _, _ := net.SplitHostPort(r.RemoteAddr)
    if isTrustedProxy(remoteIP, trustedProxies) {
        if ip := r.Header.Get(xRealIpHeader); ip != "" {
            return ip
        }
        if ip := r.Header.Get(xForwardedForHeader); ip != "" {
            return strings.TrimSpace(strings.Split(ip, ",")[0])
        }
    }
    return remoteIP
}
```

If no reverse proxy is deployed, remove the header-based logic entirely and always use `r.RemoteAddr`.

### Proof of Concept
```bash
# Forge victim's IP (192.0.2.1) as the apparent originator
curl -X POST https://<rosetta-host>/network/list \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 192.0.2.1" \
  -d '{"network_identifier": {"blockchain": "Hedera", "network": "testnet"}}'

# Server log will show:
# 192.0.2.1 POST /network/list 200 in Xms
# Attacker's real IP is never recorded.
```
