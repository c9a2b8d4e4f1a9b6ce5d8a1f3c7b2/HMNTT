### Title
Missing Minimum Limit Enforcement in `extractLimitFromFilters` Enables Amplified DB Round-Trip Griefing

### Summary
`extractLimitFromFilters()` in `rest/controllers/blockController.js` enforces only an upper bound on the `limit` parameter, accepting any value down to `1`. An unprivileged attacker can supply `limit=1` and mechanically follow every generated `next` link, causing one separate database query per block in the chain. Because the REST module has no rate limiting, this maximizes database round-trips and degrades query throughput for all concurrent users.

### Finding Description

**Root cause — `extractLimitFromFilters` (lines 57–61):**

```js
extractLimitFromFilters = (filters) => {
  const limit = findLast(filters, {key: filterKeys.LIMIT});
  const maxLimit = getEffectiveMaxLimit();
  return limit ? (limit.value > maxLimit ? defaultLimit : limit.value) : defaultLimit;
};
```

The function only rejects values **above** `maxLimit`; any positive integer including `1` passes through unchanged. [1](#0-0) 

**Validation layer — `filterValidityChecks` (line 330–332):**

```js
case constants.filterKeys.LIMIT:
  ret = isPositiveLong(val);
  break;
```

`isPositiveLong` accepts any positive long, so `limit=1` is valid. [2](#0-1) 

**Pagination link generation — `generateNextLink` (lines 90–99):**

```js
generateNextLink = (req, blocks, filters) => {
  return blocks.length
    ? utils.getPaginationLink(req, blocks.length !== filters.limit, ...)
    : null;
};
```

When `limit=1` and one block is returned, `blocks.length (1) !== filters.limit (1)` evaluates to `false`, meaning `isEnd=false`, so a `next` link is **always** emitted until the table is exhausted. [3](#0-2) 

**One DB query per request — `RecordFileService.getBlocks` (lines 149–162):**

```js
async getBlocks(filters) {
  const query = RecordFileService.blocksQuery + `
    ${where}
    order by ${filters.orderBy} ${filters.order}
    limit ${filters.limit}
  `;
  const rows = await super.getRows(query, params);
  ...
}
```

Each HTTP request issues exactly one database query. With `limit=1`, N blocks require N separate queries. [4](#0-3) 

**No REST-layer rate limiting:** A search across all REST middleware and route files confirms there is no rate limiter applied to the `/api/v1/blocks` endpoint. The throttling infrastructure found in the codebase is scoped exclusively to the `web3` module (contract calls).


### Impact Explanation
On a mainnet mirror node with millions of blocks, an attacker issuing `limit=1` requests in a tight loop forces the database to serve one indexed lookup per request. While each individual query is cheap (index seek on `record_file.index`), the aggregate effect is a sustained high-frequency query stream that consumes database connection pool slots, increases query latency for legitimate users, and can saturate the Node.js event loop with response serialization overhead. Because there is no per-IP or per-user rate limit on this endpoint, a single client can monopolize available DB connections.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero tooling beyond a standard HTTP client. The `next` link in every response provides the exact URL for the next request, making automation trivial. Any motivated actor can script this in under ten lines of code. The attack is repeatable indefinitely and is not self-limiting.

### Recommendation
1. **Enforce a minimum limit**: In `extractLimitFromFilters`, reject or clamp values below a configurable minimum (e.g., `1` → `MIN_LIMIT = 1` is fine, but enforce `MIN_LIMIT = 1` explicitly and document it; more practically, set `MIN_LIMIT` to a value like `10` to reduce amplification).
2. **Add a REST-layer rate limiter**: Apply a per-IP request rate limit (e.g., via `express-rate-limit`) to all `/api/v1/*` routes, consistent with the throttling already present in the `web3` module.
3. **Clamp `limit=1` to a sensible floor**: Change the guard in `extractLimitFromFilters` to:
   ```js
   const MIN_LIMIT = 1; // or higher
   return limit
     ? (limit.value > maxLimit || limit.value < MIN_LIMIT ? defaultLimit : limit.value)
     : defaultLimit;
   ```

### Proof of Concept
```bash
# Step 1: fetch first page with limit=1
curl "https://<mirror-node>/api/v1/blocks?limit=1&order=asc"
# Response includes: "links": { "next": "/api/v1/blocks?limit=1&order=asc&block.number=gt:1" }

# Step 2: script to follow every next link
NEXT="/api/v1/blocks?limit=1&order=asc"
while [ -n "$NEXT" ]; do
  NEXT=$(curl -s "https://<mirror-node>$NEXT" | jq -r '.links.next // empty')
done
# Each iteration issues one DB query; for 60 million blocks this is 60 million queries.
```
