### Title
Unbounded Subscription Duration Enables Resource Exhaustion DoS in `subscribeTopic()`

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no maximum subscription duration. When a client omits `endTime` and `limit`, the server-side reactive pipeline never terminates, holding a connection slot, a `boundedElastic` scheduler thread, and associated memory indefinitely. Any unauthenticated caller can open many such subscriptions to exhaust server capacity.

### Finding Description

**Exact code path:**

`TopicMessageFilter` declares `endTime` as a plain nullable `Long` with no upper-bound constraint and `limit` with only `@Min(0)` (so `0` = no limit is valid): [1](#0-0) 

In `subscribeTopic()`, when `endTime == null` and `limit == 0`, neither termination branch is entered: [2](#0-1) 

`pastEndTime()` explicitly returns `Flux.never()` when `endTime` is null, so `takeUntilOther` never fires: [3](#0-2) 

`TopicContext.isComplete()` also hard-returns `false` when `endTime == null`: [4](#0-3) 

The live branch subscribes to `topicListener.listen()` on a `boundedElastic` scheduler and keeps polling indefinitely: [5](#0-4) 

**Root cause:** The design assumes clients will self-terminate via `endTime` or `limit`, but neither field is required, and no server-side maximum duration is enforced anywhere in `GrpcProperties` or `TopicMessageFilter`. [6](#0-5) 

### Impact Explanation

Each open subscription consumes: one gRPC HTTP/2 stream, one reactive Flux pipeline with a `boundedElastic` thread for the safety-check poller, and heap for `TopicContext`. The only per-connection cap is `maxConcurrentCallsPerConnection = 5`: [7](#0-6) 

This limits calls *per connection*, not total connections. There is no global subscriber cap — `subscriberCount` is a metric gauge only, never checked against a ceiling: [8](#0-7) 

An attacker opening N connections × 5 calls each = 5N permanent subscriptions, exhausting the `boundedElastic` thread pool and Netty connection table, denying service to legitimate users.

### Likelihood Explanation

No authentication is required to call `subscribeTopic()`. The gRPC endpoint is publicly reachable. The only precondition is knowing a valid topic ID (or exploiting `checkTopicExists = false` to skip even that check): [9](#0-8) 

A single attacker with a standard gRPC client library can script this in minutes. The attack is repeatable and requires no special privileges.

### Recommendation

1. Add a `maxSubscriptionDuration` field (e.g., default `Duration.ofHours(1)`) to `GrpcProperties`.
2. In `subscribeTopic()`, apply `.timeout(grpcProperties.getMaxSubscriptionDuration())` to the returned `Flux`.
3. Add a `@Max` constraint on `endTime` in `TopicMessageFilter` to reject far-future end times.
4. Enforce a global subscriber ceiling: check `subscriberCount` against a configurable `maxSubscribers` and reject new subscriptions when the limit is reached.
5. Consider adding per-IP connection rate limiting at the Netty/load-balancer layer.

from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

def open_permanent_subscription(channel, topic_id):

    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=topic_id,
        # No limit, no end_time -> permanent subscription
    )
    for _ in stub.subscribeTopic(request):
        pass  # consume silently, keep stream open
