### Title
Unauthenticated Wide-Range Account ID + Timestamp Filter Enables Repeated Expensive DB Scans in `getBalances()`

### Summary
The `/api/v1/balances` endpoint in `rest/balances.js` accepts `account.id=lt:<large_value>`, `timestamp=lte:<recent>`, and `limit=<max>` from any unauthenticated caller. Each such request triggers `getBalancesQuery()` to execute a `DISTINCT ON (account_id)` scan across a wide account ID range within a monthly partition, plus a correlated `token_balance` subquery per returned row. Because the REST API has no per-IP rate limiting and the response cache is trivially bypassed by varying the timestamp value, an attacker can sustain maximum-cost DB load continuously, degrading throughput for all users.

### Finding Description

**Code path:**

`getBalances()` → `getTsQuery()` → `getBalancesQuery()` → SQL execution

**Step 1 – Parameter acceptance (no auth required):**

`acceptedBalancesParameters` includes `ACCOUNT_ID`, `TIMESTAMP`, and `LIMIT`. [1](#0-0) 

`balanceFilterValidator` routes `account.id` to `validateOpAndValue`, which permits the `lt` operator. [2](#0-1) 

`parseAccountIdQueryParam` in `balances.js` (not `utils.js`) converts `lt:0.0.999999999` to `ab.account_id < 999999999`, matching nearly every account. [3](#0-2) 

**Step 2 – Timestamp optimization is a partial mitigation only:**

`getOptimizedTimestampRange()` limits the scan to at most two monthly partitions. This reduces but does not eliminate the attack surface — within those two partitions, the query still scans all rows where `account_id < 999999999`. [4](#0-3) 

**Step 3 – Expensive query structure:**

`getBalancesQuery()` builds a `DISTINCT ON (account_id)` query ordered by `account_id, consensus_timestamp DESC`, requiring PostgreSQL to process all matching rows before applying the limit. For each of the `limit` rows returned, a correlated `token_balance` subquery is executed. [5](#0-4) [6](#0-5) 

**Step 4 – Cache bypass:**

The response cache key is an MD5 of `req.originalUrl`. Incrementing the timestamp nanosecond value by 1 on each request produces a unique cache key, guaranteeing a cache miss every time. [7](#0-6) 

**Step 5 – No per-IP rate limiting in the REST API:**

The middleware stack (`requestHandler.js`, `responseCacheHandler.js`, etc.) contains no per-IP or global request-rate throttle for the REST API. The throttle infrastructure present in the codebase is scoped exclusively to the `web3` Java module. [8](#0-7) 

The only backstop is the DB pool's `statementTimeout` and `maxConnections`, which are application-wide and shared across all users. [9](#0-8) 

### Impact Explanation

Each crafted request forces a wide partition scan plus N correlated subqueries (N = max limit). Sustained at scale from one or more IPs, this exhausts DB connection slots and CPU, increasing query latency for all legitimate users. No funds are at risk; the impact is availability degradation (griefing), consistent with the Medium severity classification.

### Likelihood Explanation

No credentials, tokens, or special network access are required. The attack is fully scriptable with a simple loop that increments the timestamp by 1 nanosecond per request to defeat caching. Any public internet user can execute it. Repeatability is unlimited until infrastructure-level controls (e.g., a WAF or reverse proxy rate limiter) intervene — none of which are enforced at the application layer.

### Recommendation

1. **Add per-IP rate limiting** in the REST API middleware (e.g., `express-rate-limit`) for the `/api/v1/balances` endpoint, independent of the web3 throttle.
2. **Restrict range operators on `account.id`** (i.e., `lt`, `lte`, `gt`, `gte`) or require a minimum selectivity (e.g., enforce a lower bound alongside any upper bound) to prevent unbounded range scans.
3. **Add a DB-level query cost guard** (e.g., `SET statement_timeout` per request class) specifically for wide-range balance queries.
4. **Normalize cache keys** for timestamp filters to the resolved snapshot timestamp (returned by `getTsQuery`) rather than the raw URL, so requests with slightly different input timestamps that resolve to the same snapshot share a cache entry.

### Proof of Concept

```bash
# Baseline: single max-cost request
curl "https://<mirror-node>/api/v1/balances?account.id=lt:0.0.999999999&timestamp=lte:9999999999.000000000&limit=100"

# Cache-bypass loop: increment nanoseconds to guarantee cache miss each iteration
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/balances?account.id=lt:0.0.999999999&timestamp=lte:9999999999.$(printf '%09d' $i)&limit=100" &
done
wait
# Observe: DB CPU spikes, legitimate query latency increases, connection pool saturation
```
