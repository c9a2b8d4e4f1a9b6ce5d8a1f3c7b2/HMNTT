### Title
Unauthenticated 3-Phase Sequential DB Load via `block_hash` Filter on `GET /contracts/results`

### Summary
An unauthenticated external user can send `GET /api/v1/contracts/results?block.hash=<hash>` to trigger a deterministic 3-phase, 4-query sequential database load per request with no rate limiting in place. The `block_hash` filter adds an extra synchronous DB lookup before the main query, after which two more parallel DB queries are always issued against the result set. Flooding this endpoint with concurrent requests exhausts the DB connection pool and degrades the mirror node REST service.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js:19` registers the route with no authentication guard: [1](#0-0) 

`getContractResults()` in `rest/controllers/contractController.js:1050–1112` is the handler: [2](#0-1) 

**Phase 1 — extra sequential DB query triggered by `block_hash`:**
Inside `extractContractResultsByIdQuery()`, when `blockFilter.key === filterKeys.BLOCK_HASH`, the code awaits `RecordFileService.getRecordFileBlockDetailsFromHash()` before proceeding: [3](#0-2) 

The underlying query is a `LIKE`-prefix scan with no index guarantee: [4](#0-3) 

**Phase 2 — main contract results query:** [5](#0-4) 

**Phase 3 — two parallel DB queries always issued if rows are returned:** [6](#0-5) 

`getRecordFileBlockDetailsFromTimestampArray()` executes a correlated subquery with `unnest` over the full result-set timestamp array: [7](#0-6) 

**No rate limiting exists.** The full middleware stack in `rest/server.js` is: `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler` — no rate-limiter: [8](#0-7) 

**`authHandler` does not block unauthenticated requests** — it only sets a higher result-set limit for known users. Unauthenticated requests proceed with the default (capped) limit but still execute all DB phases: [9](#0-8) 

**Root cause / failed assumption:** The design assumes that the result-set `limit` cap is sufficient to bound per-request DB cost. It is not: the `block_hash` path adds a mandatory extra sequential query (Phase 1) that is not present for other filters, and Phases 2–3 always fire regardless of limit. There is no request-rate control to prevent concurrent amplification.

### Impact Explanation

Each request with a valid `block_hash` value forces 4 DB round-trips in 3 sequential phases against the PostgreSQL backend. Because the Phase 3 `getRecordFileBlockDetailsFromTimestampArray` query uses a correlated subquery with `unnest` over the timestamp array, its cost scales with the result-set size. Concurrent flooding of this endpoint exhausts the DB connection pool (configured via `initializePool()` in `rest/server.js:49`), causing all mirror node REST endpoints to queue or time out. The mirror node is the primary read interface for wallets, explorers, and dApps interacting with the Hedera network; its unavailability constitutes a significant service degradation.

### Likelihood Explanation

The endpoint is publicly documented in `rest/api/v1/openapi.yml:640–665` and requires zero credentials. A valid `block_hash` value is trivially obtained from any public block explorer or a prior call to `GET /api/v1/blocks`. The attack is fully repeatable, requires no special tooling beyond `curl` or `ab`, and can be sustained indefinitely. No CAPTCHA, IP throttle, or connection-rate limit is present in the codebase.

### Recommendation

1. **Add a rate-limiting middleware** (e.g., `express-rate-limit`) applied globally before route handlers in `rest/server.js`, with a per-IP request budget.
2. **Cache the `block_hash → timestamp-range` lookup** (Phase 1) at the application layer; block hashes are immutable once finalized, making them ideal cache keys.
3. **Require authentication or an API key** for the `block_hash` and `block.number` filter paths, consistent with the existing `authHandler` pattern.
4. **Add a DB connection pool guard** (pool size limit + queue timeout) so that a flood of requests fails fast with HTTP 429 rather than blocking all connections.

### Proof of Concept

```bash
# Obtain a valid block hash (unauthenticated)
HASH=$(curl -s https://<mirror-node>/api/v1/blocks?limit=1 | jq -r '.blocks[0].hash')

# Flood the endpoint — each request triggers 4 DB queries in 3 sequential phases
ab -n 5000 -c 200 \
  "https://<mirror-node>/api/v1/contracts/results?block.hash=${HASH}"
```

Expected result: DB connection pool exhaustion within seconds; all mirror node REST endpoints begin returning 500/503 errors or timing out. No authentication, no special privileges, and no brute-force enumeration are required — a single known block hash is sufficient to sustain the attack indefinitely.
