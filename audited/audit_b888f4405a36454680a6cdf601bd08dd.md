### Title
Pagination `next` Link Leaks Broad Timestamp Range When `transaction_hash` Narrows Query Scope in `getContractLogs`

### Summary
When a caller supplies both a broad `timestamp` range and a `transaction_hash` filter to `/api/v1/contracts/results/logs`, the API correctly narrows the DB query to the single `consensus_timestamp` resolved from the hash. However, the `next` pagination link is built from a stale shallow copy of the original broad timestamp bounds, and the `transaction_hash` parameter is never forwarded into the link. Following the `next` link therefore executes a fundamentally different query ŌĆö the broad range without the hash filter ŌĆö silently returning logs from unintended historical windows.

### Finding Description

**Exact code path:**

In `extractContractLogsMultiUnionQuery` (`rest/controllers/contractController.js`, lines 659ŌĆō686):

```js
// Line 659-667: shallow copy taken while bounds.primary still holds user range
const query = {
  bounds: {...bounds},   // query.bounds.primary = gte:T1, lte:T2
  ...
};

// Line 669-677: bounds.primary is replaced with the resolved consensus_timestamp
if (transactionHash !== undefined) {
  bounds.primary = new Bound(filterKeys.TIMESTAMP);
  bounds.primary.parse({..., value: rows[0].consensus_timestamp});
  // query.bounds.primary is NOT updated ŌĆö still gte:T1, lte:T2
}
```

The DB query is built from the updated `bounds` (narrow, correct), but `query.bounds` retains the original broad range.

In `getContractLogs` (`rest/controllers/contractController.js`, lines 840ŌĆō847):

```js
const rows = await ContractService.getContractLogs(query);
const logs = rows.map((row) => new ContractLogViewModel(row));
res.locals[responseDataLabel] = {
  logs,
  links: {
    next: this.getPaginationLink(req, logs, query.bounds, query.limit, query.order),
    //                                       ^^^^^^^^^^^^ stale broad range
  },
};
```

`transaction_hash` is also never added to the `next` link because it is not tracked in `query.bounds` or any pagination-aware structure.

**Root cause:** The `{...bounds}` shallow copy at line 662 was intended to preserve the original user-supplied bounds for pagination purposes, but it creates a permanent divergence: the DB query uses the narrowed `bounds.primary`, while the pagination link uses the stale `query.bounds.primary`.

**Why existing checks fail:** The comment at line 660 explicitly acknowledges the intent ŌĆö *"Save a copy of the bounds parsed from user input since the primary (timestamp) bound will get changed when there is a transaction hash filter"* ŌĆö but the implementation then uses this stale copy to generate the `next` link, which is incorrect. No check re-synchronises `query.bounds` after the hash resolution, and no mechanism forwards `transaction_hash` into the pagination link. [1](#0-0) [2](#0-1) 

### Impact Explanation
An unprivileged caller can craft a request that returns logs scoped to a single transaction, then follow the `next` link to receive logs from the entire original timestamp range ŌĆö a different historical window ŌĆö without ever supplying the `transaction_hash` again. This breaks the integrity of paginated result sets: a consumer iterating pages believes they are paginating over a hash-filtered result set, but subsequent pages are actually unfiltered broad-range queries. In a monitoring or audit context this means an attacker can silently shift the query window across record files by controlling the initial timestamp range, causing log consumers to process data outside the intended scope.

### Likelihood Explanation
No authentication or privilege is required. The only precondition is knowing any valid transaction hash (publicly observable on-chain) and its approximate timestamp. The exploit is fully deterministic and repeatable with a single HTTP request pair. Any public mirror node deployment is affected.

### Recommendation
1. After hash resolution, update `query.bounds` to reflect the narrowed timestamp so the pagination link is consistent with the actual query:
   ```js
   bounds.primary = new Bound(filterKeys.TIMESTAMP);
   bounds.primary.parse({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value: rows[0].consensus_timestamp});
   query.bounds.primary = bounds.primary;  // keep query.bounds in sync
   ```
2. When `transaction_hash` is present, set `next: null` in the response (a hash-filtered query is inherently a point lookup; pagination beyond it is semantically undefined).
3. If pagination is intentionally supported alongside `transaction_hash`, include `transaction_hash` as a parameter in the generated `next` link so subsequent pages remain scoped to the same hash. [3](#0-2) 

### Proof of Concept

**Preconditions:** A mirror node with at least two record files containing contract logs. Know a valid `transaction_hash` (e.g., `0xAAA...`) whose `consensus_timestamp` is `T_mid`, where `T_mid` is between `T1` and `T2` (spanning multiple record files).

**Step 1 ŌĆö Initial request:**
```
GET /api/v1/contracts/results/logs?timestamp=gte:T1&timestamp=lte:T2&transaction_hash=0xAAA...&limit=1
```
**Observed response:**
- `logs`: contains only the log(s) at `T_mid` (correct, hash-filtered)
- `links.next`: `...?timestamp=gte:T1&timestamp=lte:T2&limit=1&order=desc&index=gt:N`
  ŌĆö references the **broad** range `[T1, T2]`, no `transaction_hash`

**Step 2 ŌĆö Follow the `next` link verbatim:**
```
GET /api/v1/contracts/results/logs?timestamp=gte:T1&timestamp=lte:T2&limit=1&order=desc&index=gt:N
```
**Observed response:**
- `logs`: contains logs from timestamps **other than** `T_mid` within `[T1, T2]`
- The caller is now receiving data from a different historical window than the original hash-filtered query, with no indication that the scope changed.

**Result:** The attacker has used the `next` link from a hash-filtered response to traverse logs across the entire `[T1, T2]` range, bypassing the hash filter entirely on all subsequent pages.
