### Title
Single Client Can Monopolize Global Gas Budget via Absence of Per-Client Gas Throttling

### Summary
The `gasLimitBucket` in `ThrottleConfiguration.gasLimitBucket()` is a single, globally-shared Bucket4j token bucket with no per-client or per-IP partitioning. Because there is no minimum inter-request gas interval enforced per client, an unprivileged attacker can send requests at the maximum allowed rate with maximum gas, draining the entire global gas budget within one second and causing all other clients to receive "Gas per second rate limit exceeded" errors for the remainder of that window.

### Finding Description
**Code path:**

- `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 34–45: `gasLimitBucket()` creates one global `Bucket` with capacity = `scaleGas(gasPerSecond)` (default 7,500,000,000 → 750,000 scaled tokens) and a greedy refill of the full capacity every 1 second. There is no per-client, per-IP, or per-session bucket.

- `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–48: `throttle()` checks `rateLimitBucket.tryConsume(1)` (global 500 req/s) then `gasLimitBucket.tryConsume(scaleGas(request.getGas()))` against the same single global bucket for every caller.

**Root cause:** The failed assumption is that the global gas budget will be shared fairly across clients. In reality, the first client to saturate the bucket wins; there is no fairness mechanism, no per-source tracking, and no minimum interval between consecutive requests from the same source.

**Exploit flow:**
1. Attacker sends requests with `gas = maxGasLimit` (15,000,000). Each request consumes `15,000,000 / 10,000 = 1,500` scaled tokens.
2. The global bucket holds `750,000` scaled tokens (default config).
3. `750,000 / 1,500 = 500` requests exhaust the entire gas budget.
4. The global `rateLimitBucket` allows 500 req/s, so the attacker can drain the gas bucket in a single burst within one second.
5. All subsequent requests from any other client hit `gasLimitBucket.tryConsume(...)` returning `false` → `ThrottleException("Gas per second rate limit exceeded.")` until the bucket refills after ~1 second.
6. Attacker repeats every second indefinitely.

**Why existing checks are insufficient:**
- `rateLimitBucket` is also global (500 req/s total), not per-client — it does not prevent one client from consuming all 500 slots.
- `SynchronizationStrategy.SYNCHRONIZED` on the gas bucket only ensures thread safety; it does not add fairness or per-client isolation.
- The optional `request[]` filter rules (`LOG`, `REJECT`, `THROTTLE`) are opt-in, not configured by default (`List.of()`), and even when configured they match on request content fields (DATA, BLOCK, GAS, TO, VALUE), not on client identity/IP.

### Impact Explanation
Any unauthenticated client can continuously starve all other clients of gas capacity, causing sustained denial-of-service for legitimate contract call and gas estimation requests. The impact is bounded to griefing (no economic loss to the attacker, no on-chain state change), consistent with the stated medium scope. However, the service becomes effectively unavailable for all other users for as long as the attacker sustains the flood.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no on-chain funds. Any HTTP client capable of sending ~500 requests/second (trivially achievable with standard tooling such as `wrk`, `hey`, or a simple async script) can execute this. The attack is repeatable every second and is not self-limiting. The only external mitigation would be network-layer IP rate limiting (e.g., a WAF or load balancer), which is not part of the application and is not guaranteed to be present.

### Recommendation
Introduce per-client gas rate limiting keyed on a client identifier (e.g., remote IP address, or an authenticated API key). Concretely:

1. Replace the single global `gasLimitBucket` bean with a `LoadingCache<String, Bucket>` (e.g., Caffeine) that creates an individual per-IP bucket on first access, with the same per-second capacity divided by an expected fair-share factor.
2. In `ThrottleManagerImpl.throttle()`, extract the client IP from the `HttpServletRequest` (injectable via `RequestContextHolder`) and consume from the per-client bucket rather than the global one.
3. Retain the global bucket as a secondary hard cap to prevent the per-client cache from being bypassed via IP spoofing or a large botnet.
4. Alternatively, enforce per-IP rate limiting at the infrastructure layer (reverse proxy / API gateway) before requests reach the application.

### Proof of Concept
```bash
# Default config: gasPerSecond=7_500_000_000, requestsPerSecond=500, maxGasLimit=15_000_000
# Attacker sends 500 concurrent requests/second each with gas=15000000

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0xdeadbeef","gas":15000000,"to":"0x0000000000000000000000000000000000000001","estimate":false}' \
    &
done
wait

# Immediately after, a legitimate client's request:
curl -X POST http://<mirror-node-web3>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"data":"0x","gas":21000,"to":"0x0000000000000000000000000000000000000001","estimate":false}'
# Expected response: HTTP 429 {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
# Repeat every ~1 second to sustain the denial of service.
```
