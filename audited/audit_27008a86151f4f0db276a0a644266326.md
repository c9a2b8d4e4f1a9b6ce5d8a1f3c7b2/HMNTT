### Title
Unbounded Unknown Query Parameter Accumulation in `getBlocks()` Enables Amplified Error Response DoS

### Summary
`BlockController.getBlocks()` passes `req.query` directly to `buildAndValidateFilters()` with no limit on the number of distinct query parameter keys. Each unknown key is individually accumulated into a `badParams` array and serialized into the error response, with no cap on the total count. An unauthenticated attacker can craft a single request with hundreds of distinct unknown parameters to force disproportionate server-side work and an oversized error response.

### Finding Description

**Code path:**

`blockController.js` line 102 calls `buildAndValidateFilters` with the raw query object: [1](#0-0) 

`buildFilters` in `utils.js` iterates every key in `query` with no cap on the number of distinct keys: [2](#0-1) 

`validateFilters` pushes one entry into `unknownParams` for every key not in `acceptedParameters`, again unbounded: [3](#0-2) 

`buildAndValidateFilters` merges all results and throws with the full unsized array: [4](#0-3) 

`InvalidArgumentError.forRequestValidation` maps every entry to a formatted string with no truncation: [5](#0-4) 

**Root cause:** The only per-key guard is `isRepeatedQueryParameterValidLength`, which limits the number of *values* for a single repeated key. The `qs` parser in `requestHandler.js` uses `arrayLimit` with `throwOnLimitExceeded: true`, which also only restricts repeated values for one key — it places no limit on the number of *distinct* keys: [6](#0-5) 

`acceptedBlockParameters` accepts only 4 keys (`block.number`, `limit`, `order`, `timestamp`), so every other key becomes an `unknownParam` entry: [7](#0-6) 

**Failed assumption:** The code assumes the number of distinct query parameter keys is naturally bounded, but neither Express, `qs`, nor any application-layer middleware enforces such a limit.

### Impact Explanation
An attacker sending a single GET request with N distinct unknown parameter names forces the server to: (1) parse N keys, (2) build N filter objects, (3) validate each against the accepted set, (4) allocate an N-element `badParams` array, (5) map it to N formatted strings, and (6) serialize the entire array into a JSON error response. With URL lengths up to ~16 KB (Node.js default), an attacker can embed ~500–800 short distinct keys per request, producing an error response several times larger than the request and consuming CPU proportional to N on every request. Without rate limiting visible in the codebase, this can be repeated at high frequency to exhaust server resources.

### Likelihood Explanation
No authentication or special privilege is required — the `/api/v1/blocks` endpoint is public. The exploit requires only a standard HTTP client. The attack is trivially scriptable and repeatable. The constraint is URL length, but even within typical limits the amplification and CPU overhead are meaningful at scale.

### Recommendation
1. **Cap distinct query parameter count** in `requestQueryParser` or as early middleware: reject requests whose parsed query object has more than a configurable maximum number of keys (e.g., 20–30).
2. **Cap `badParams` before throwing**: in `buildAndValidateFilters`, truncate `badParams` to a small maximum (e.g., 10) before passing to `forRequestValidation`.
3. **Apply rate limiting** on error responses (HTTP 400) at the reverse-proxy or middleware layer.

### Proof of Concept
```bash
# Generate a request with 500 distinct unknown query parameters
PARAMS=$(python3 -c "print('&'.join(f'unknown{i}=val' for i in range(500)))")
curl -v "http://<host>/api/v1/blocks?$PARAMS"
```
Expected result: HTTP 400 response containing 500 `"Unknown query parameter: unknownN"` entries in the `_status.messages` array, with response body significantly larger than the request, and measurable CPU spike per request. Repeating this in a loop with concurrent connections degrades server responsiveness.

```javascript
const validateFilters = (filters, filterValidator, acceptedParameters) => {
  const invalidParams = [];
  const unknownParams = [];
  for (const filter of filters) {
    if (!acceptedParameters.has(filter.key)) {
      unknownParams.push({key: filter.key, code: InvalidArgumentError.UNKNOWN_PARAM_USAGE});
      continue;
    }
    if (!filterValidator(filter.key, filter.operator, filter.value)) {
      invalidParams.push(filter.key);
    }
  }

  return {invalidParams, unknownParams};
```
