### Title
Gas Bucket Starvation DoS via Minimum `gasPerSecond` Configuration Yielding 2-Token Capacity

### Summary
When `gasPerSecond` is set to its minimum allowed value of `21_000`, the `scaleGas()` division by `GAS_SCALE_FACTOR` (10,000) yields a bucket capacity of only 2 tokens. Any unprivileged user sending a single request with the minimum valid gas (`21_000`) consumes all 2 tokens via `tryConsume(2)`, completely blocking all other users from the gas throttle for up to 1 second. The attacker can repeat this continuously to maintain a sustained DoS.

### Finding Description

**Exact code path:**

`ThrottleProperties.java` enforces `@Min(21_000)` on `gasPerSecond` and defines `GAS_SCALE_FACTOR = 10_000`: [1](#0-0) 

`getGasPerSecond()` calls `scaleGas(gasPerSecond)`: [2](#0-1) 

At minimum config: `scaleGas(21_000)` → `21_000 > 10_000` so returns `Math.floorDiv(21_000, 10_000)` = **2**.

`gasLimitBucket()` uses this value as both capacity and refill rate: [3](#0-2) 

Bucket capacity = **2 tokens**, refilling 2 tokens/second.

`ContractCallRequest` enforces `@Min(21_000)` on the `gas` field — meaning the minimum valid request gas is exactly 21,000: [4](#0-3) 

In `ThrottleManagerImpl.throttle()`, the gas bucket is consumed with `scaleGas(request.getGas())`: [5](#0-4) 

`scaleGas(21_000)` = 2. A single request with `gas=21_000` calls `tryConsume(2)` against a bucket with capacity 2 — consuming 100% of available tokens.

**Root cause:** The minimum allowed `gasPerSecond` (21,000) is only 2.1× the `GAS_SCALE_FACTOR` (10,000), producing a bucket with 2 tokens. The minimum valid request gas (also 21,000) maps to exactly 2 scaled tokens — equal to the entire bucket capacity. There is no floor on bucket capacity relative to per-request token cost.

**Why `restore()` does not save it:**

`restoreGasToBucket()` only returns tokens *after* execution completes: [6](#0-5) 

During execution the bucket is empty. If the attacker's call uses all its gas (e.g., a failing call), `gasRemaining = 0` and 0 tokens are restored. Even when tokens are restored, the attacker immediately re-submits to drain them again.

### Impact Explanation

With `gasPerSecond=21_000`, a single attacker request with `gas=21_000` exhausts the entire gas bucket. Every concurrent or subsequent request from any other user receives HTTP 429 `"Gas per second rate limit exceeded."` for the duration of the attacker's execution window (or up to 1 second for the bucket refill). The attacker can loop this indefinitely, creating a sustained, complete denial of service of the `/api/v1/contracts/call` endpoint for all other users. No authentication, API key, or special privilege is required.

### Likelihood Explanation

The attack requires `gasPerSecond` to be configured at or near its minimum (21,000). While the default is 7.5 billion, the `@Min(21_000)` annotation explicitly permits this value, and operators tuning for restrictive environments may set low values. The attacker needs only to send standard HTTP POST requests with `gas=21_000` — no credentials, no special knowledge, no on-chain access. The attack is trivially scriptable and repeatable at high frequency.

### Recommendation

1. **Raise the minimum `gasPerSecond`** to a value that, after `scaleGas()`, yields a bucket capacity meaningfully larger than any single request's scaled gas cost. For example, enforce `@Min(1_000_000)` so the minimum bucket capacity is 100 tokens.
2. **Add a bucket capacity floor check** at startup: assert `scaleGas(gasPerSecond) >= some_minimum_tokens` (e.g., ≥ 100) and fail fast with a clear configuration error.
3. **Alternatively**, enforce that `gasPerSecond` must be at least `N × GAS_SCALE_FACTOR` where N is a safe minimum token count (e.g., N=100), making the relationship between the two constants explicit.

### Proof of Concept

**Preconditions:**
- Mirror node configured with `hiero.mirror.web3.throttle.gasPerSecond=21000`
- Resulting bucket capacity: `scaleGas(21000)` = `Math.floorDiv(21000, 10000)` = **2 tokens**

**Steps:**

1. Attacker sends:
```http
POST /api/v1/contracts/call
Content-Type: application/json

{
  "data": "0x",
  "to": "0x0000000000000000000000000000000000000001",
  "gas": 21000,
  "estimate": false
}
```
2. `ThrottleManagerImpl.throttle()` calls `gasLimitBucket.tryConsume(scaleGas(21000))` = `tryConsume(2)`. Bucket had 2 tokens → succeeds, bucket now at 0.
3. Any concurrent or immediately subsequent request from any other user hits `tryConsume(2)` on an empty bucket → `tryConsume` returns `false` → `ThrottleException("Gas per second rate limit exceeded.")` → HTTP 429.
4. Attacker loops step 1 continuously. Each iteration either: (a) succeeds and drains the bucket again, or (b) is itself throttled — but the bucket remains empty for all other users.
5. All legitimate users are blocked for the duration of the attack with no recourse.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
