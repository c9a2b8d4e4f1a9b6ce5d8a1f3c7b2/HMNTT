### Title
PollingContext Reset on Subscription Reconnect Enables Repeated Historical DB Query Amplification

### Summary
Every call to `PollingTopicListener.listen()` allocates a fresh `PollingContext` with `last = null` and `count = 0`. When `last` is null, `poll()` unconditionally resets `startTime` to the caller-supplied `filter.getStartTime()`. An unprivileged attacker who repeatedly cancels and reopens subscriptions with `startTime = 0` forces the server to re-issue bounded but expensive historical queries from the beginning of the topic's history on every reconnect cycle, with no per-client rate limit or connection cap to stop amplification.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` — [1](#0-0) 

A brand-new `PollingContext` is constructed on every invocation. Its `last` field is an `AtomicReference` initialized to `null` and its `count` is `AtomicLong(0)`.

`poll()` derives `startTime` as: [2](#0-1) 

```
startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime()
```

Because `last` is always `null` on a fresh context, the very first poll (and every poll after a reconnect) queries from `filter.getStartTime()`. The `TopicMessageFilter` validator only requires `startTime <= now()`, so `startTime = 0` (nanosecond epoch) is accepted: [3](#0-2) 

The repository executes a JPA criteria query `WHERE topic_id = ? AND consensus_timestamp >= startTime ORDER BY consensus_timestamp` with `setMaxResults(pageSize)`: [4](#0-3) 

**Root cause / failed assumption:** The design assumes a single long-lived `PollingContext` per logical subscription. It does not account for an attacker who deliberately tears down and recreates subscriptions to keep `last` perpetually null, forcing every poll to anchor at the user-supplied (potentially ancient) `startTime`.

**Why existing checks are insufficient:**

- `maxPageSize = 5000` caps rows *per query*, not the *rate* of queries or the number of concurrent subscriptions. [5](#0-4) 
- `maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, but there is no cap on the number of connections an attacker may open. [6](#0-5) 
- `GrpcConfiguration` applies only `maxConcurrentCallsPerConnection`; no global subscriber cap, no IP-level rate limit, no reconnect throttle. [7](#0-6) 
- `subscriberCount` is a metric gauge only — it is never checked against a maximum before accepting a new subscription. [8](#0-7) 

### Impact Explanation
Each reconnect cycle issues a fresh `SELECT … WHERE consensus_timestamp >= 0 … LIMIT 5000` against the `topic_message` table. Even with an index on `(topic_id, consensus_timestamp)`, the DB must perform an index seek, read up to 5000 rows, and transfer them to the application on every cycle. With *C* connections × 5 calls/connection × 2 reconnects/second, the attacker drives `C × 10 × 5000 = 50 000 C` row reads per second. At modest scale (e.g., 20 connections, C=20) this is 1 000 000 row reads/second sustained, degrading query latency for all legitimate users. No authentication or payment is required.

### Likelihood Explanation
Any anonymous gRPC client can open connections and issue `subscribeTopic` RPCs. The attack requires only a standard gRPC client library, a valid topic ID (publicly enumerable), and a loop that cancels and reopens the stream. The `ConsensusController` sets a cancel handler (`disposable::dispose`) but imposes no reconnect delay or backoff on the server side. [9](#0-8)  The attack is trivially scriptable and repeatable.

### Recommendation
1. **Persist cursor across reconnects** — store the last-seen `consensusTimestamp` server-side (keyed by a stable subscriber token) so a reconnecting client resumes from where it left off rather than from `startTime`.
2. **Enforce a global subscriber cap** — check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and reject excess subscriptions with `RESOURCE_EXHAUSTED`.
3. **Add per-IP / per-connection reconnect rate limiting** — track reconnect frequency in `ConsensusController` and apply exponential backoff or rejection after N reconnects within a window.
4. **Enforce a minimum `startTime` age** — reject or clamp `startTime` values older than a configurable retention window to bound the historical scan range.

### Proof of Concept
```python
import grpc, time
from com.hedera.hashgraph.sdk.proto import consensus_service_pb2_grpc, consensus_topic_query_pb2
from google.protobuf.timestamp_pb2 import Timestamp

channel = grpc.insecure_channel("mirror-node:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_topic_query_pb2.ConsensusTopicQuery()
query.topicID.topicNum = 1234          # any valid topic
query.consensusStartTime.CopyFrom(Timestamp(seconds=0, nanos=0))  # epoch = full history

while True:                            # tight reconnect loop
    stream = stub.subscribeTopic(query)
    try:
        next(stream)                   # receive one message to advance past initial delay
    except StopIteration:
        pass
    stream.cancel()                    # cancel → server disposes PollingContext
    # immediately reopen → new PollingContext, last=null, startTime=0 again
    # each cycle issues: SELECT … WHERE consensus_timestamp >= 0 LIMIT 5000
```
Run this loop from multiple concurrent processes/connections. Each reconnect resets `PollingContext.last` to `null`, forcing `poll()` to re-query from `startTime = 0` on the next interval tick, continuously loading the DB with historical scans bounded only by `maxPageSize`.
