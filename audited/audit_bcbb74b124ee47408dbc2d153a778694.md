### Title
NULL-Deleted Entity Invisible to EVM Address Lookup Due to Missing COALESCE in `entityFromEvmAddressQuery`

### Summary
`getEntityIdFromEvmAddress()` uses the SQL predicate `deleted <> true` without a `COALESCE` guard. In SQL three-valued logic, `NULL <> true` evaluates to `NULL` (falsy), so any entity whose `deleted` column is `NULL` is silently excluded from the result set. The identical alias-lookup query in the same file was already fixed with `coalesce(deleted, false) <> true`, but the EVM-address query was not, creating an inconsistency that makes a class of live entities permanently unreachable via EVM address.

### Finding Description
**File:** `rest/service/entityService.js`
**Lines:** 22ŌĆō25 (`entityFromEvmAddressQuery`) vs. 17ŌĆō20 (`entityFromAliasQuery`)

```js
// ALIAS query ŌĆö correctly handles NULL
static entityFromAliasQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where coalesce(${Entity.DELETED}, false) <> true   // ŌåÉ NULL treated as false (not-deleted)
      and ${Entity.ALIAS} = $1`;

// EVM ADDRESS query ŌĆö broken NULL handling
static entityFromEvmAddressQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where ${Entity.DELETED} <> true                    // ŌåÉ NULL <> true = NULL ŌåÆ row excluded
      and ${Entity.EVM_ADDRESS} = $1`;
```

`getEntityIdFromEvmAddress()` (lines 90ŌĆō104) executes `entityFromEvmAddressQuery`. When the target entity has `deleted = NULL` (the default state for any entity that has never been explicitly deleted or un-deleted), the WHERE clause evaluates to `NULL` for that row, the row is filtered out, `rows.length === 0`, and a `NotFoundError` is thrown (line 94).

**Root cause:** The developer applied `coalesce` to the alias query but omitted it from the EVM-address query. The failed assumption is that `deleted <> true` is equivalent to "not deleted", when in fact it silently rejects `NULL`.

### Impact Explanation
Any live entity whose `deleted` column is `NULL` ŌĆö a normal, expected state for entities that have never undergone a deletion transaction ŌĆö is completely unreachable through the EVM-address code path. Every API endpoint that resolves an account/contract by EVM address (e.g., `GET /api/v1/accounts/{evmAddress}`, `GET /api/v1/contracts/{evmAddress}`) will return HTTP 404 for those entities. The same entity remains reachable by numeric ID or alias, creating an inconsistent and misleading API surface. Because `deleted = NULL` is the default for newly created entities, this affects a potentially large fraction of the entity table.

### Likelihood Explanation
No privilege is required. Any unauthenticated caller who knows (or enumerates) an EVM address belonging to an entity with `deleted = NULL` can reliably reproduce the 404. The condition is not transient ŌĆö it persists until the entity is explicitly deleted and un-deleted (toggling the column away from NULL), which may never happen. The bug is deterministic and repeatable.

### Recommendation
Replace the predicate in `entityFromEvmAddressQuery` to match the already-correct alias query:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where coalesce(${Entity.DELETED}, false) <> true
      and ${Entity.EVM_ADDRESS} = $1`;
```

This mirrors the fix already applied to `entityFromAliasQuery` and aligns with the `deleted is not true` pattern used in the Java repositories (`web3/src/main/java/.../EntityRepository.java` line 47).

### Proof of Concept
1. Insert an entity row with a known `evm_address` and `deleted = NULL` (the default; no explicit `deleted` value needed).
2. Call `GET /api/v1/accounts/<evm_address>` (or any endpoint that internally calls `getEntityIdFromEvmAddress`).
3. Observe HTTP 404 / `NotFoundError("Not found")`.
4. Call `GET /api/v1/accounts/<shard.realm.num>` for the same entity ŌĆö observe HTTP 200, confirming the entity exists and is not deleted.
5. Apply the `coalesce` fix; repeat step 2 ŌĆö observe HTTP 200.
