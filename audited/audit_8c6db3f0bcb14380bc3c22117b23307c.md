### Title
Unauthenticated DB Connection Exhaustion via Timestamp-Filtered Balance Requests

### Summary
The `getBalances()` function in `rest/balances.js` unconditionally executes a database query inside `getAccountBalanceTimestampRange()` for every request that includes a timestamp filter, regardless of whether a matching balance snapshot exists. The REST API has no rate-limiting middleware, meaning an unauthenticated attacker can flood the `/api/v1/balances?timestamp=` endpoint with requests bearing non-existent timestamps (e.g., far-future values), each consuming one DB connection from the finite pool, potentially exhausting it and degrading service.

`getTsQuery()` delegates to `getAccountBalanceTimestampRange()`: [2](#0-1) 

Inside `getAccountBalanceTimestampRange()`, a real DB query is issued unconditionally (after `getOptimizedTimestampRange` returns valid bounds): [3](#0-2) 

If no rows are returned (no snapshot at the requested timestamp), the function returns `{}`, `getTsQuery()` returns `{}`, and `getBalances()` returns early at line 116 — but the DB connection was already consumed and held for the duration of the query.

**Root cause / failed assumption:** The code assumes that the "early return" path is cheap and harmless. It is cheaper than a full query, but it is not free — it still acquires a connection from the pool, executes a query against `account_balance`, and releases the connection only after the query completes. There is no guard (rate limit, connection semaphore, or request queue) before this path is entered.

**No rate limiting on the REST API:** A grep across all `rest/**/*.js` files for `rateLimit`, `throttle`, or `requestsPerSecond` returns zero hits in production code. The throttle mechanisms that exist (`ThrottleManagerImpl`, `ThrottleConfiguration`) are exclusively in the `web3` Java module and do not apply to the Node.js REST service. [4](#0-3) 

The pool is finite (`max: config.db.pool.maxConnections`). Default and configurable, but bounded.

### Impact Explanation

If the DB connection pool is exhausted, all subsequent requests to any REST endpoint that requires a DB query will queue or fail with a connection-timeout error. This renders the mirror node REST API unavailable. Because mirror nodes are the primary read interface for the Hedera network (wallets, explorers, dApps), degrading multiple mirror nodes simultaneously disrupts network observability and usability. The severity aligns with the stated scope: shutdown/degradation of ≥30% of mirror node processing capacity without requiring any privileged access.

### Likelihood Explanation

**Preconditions:** None. The endpoint is public, unauthenticated, and accepts arbitrary timestamp values.

**Feasibility:** A single attacker with a modest botnet or even a single high-throughput HTTP client (e.g., `wrk`, `ab`, `hey`) can issue thousands of concurrent requests. Each request with a non-existent timestamp (e.g., `timestamp=lte:99999999999.999999999`) triggers one DB query. If the pool size is, say, 10–20 connections and each query takes even 5–50 ms, sustaining ~200–4000 req/s is sufficient to keep the pool saturated.

**Repeatability:** Fully repeatable, requires no state, no authentication, and no special knowledge beyond the public API spec.

### Recommendation

1. **Add rate limiting to the REST API** — implement a per-IP or global request-rate limiter (e.g., `express-rate-limit`) as middleware before any route handler, specifically for timestamp-filtered endpoints.
2. **Short-circuit before the DB query for obviously invalid timestamps** — in `getOptimizedTimestampRange`, if the computed upper bound exceeds `NOW()` by more than a configurable threshold, return `{lowerBound: undefined}` immediately without issuing a DB query.
3. **Bound concurrent DB queries** — use a semaphore or connection-aware queue so that requests beyond `maxConnections` receive an immediate `503` rather than queuing indefinitely.
4. **Set a low `statementTimeout`** for the `getAccountBalanceTimestampRange` probe query specifically, to bound connection hold time.

### Proof of Concept

```bash
# Flood /api/v1/balances with a far-future timestamp (no snapshot will ever exist)
# Requires: wrk (https://github.com/wg/wrk)

wrk -t 10 -c 200 -d 60s \
  "https://<mirror-node-host>/api/v1/balances?timestamp=lte:99999999999.999999999"

# Expected result after pool exhaustion:
# - All subsequent REST API calls return 503 / connection timeout errors
# - DB pool metrics show 100% utilization
# - Other endpoints (transactions, accounts, etc.) also become unresponsive
```

Each of the 200 concurrent connections triggers `getAccountBalanceTimestampRange()` → `pool.queryQuietly()` → holds a DB connection → returns empty → releases. At sufficient concurrency the pool saturates and legitimate requests are denied. [5](#0-4) [6](#0-5)

```javascript
const getAccountBalanceTimestampRange = async (tsQuery, tsParams) => {
  const {lowerBound, upperBound, neParams} = getOptimizedTimestampRange(tsQuery, tsParams);
  if (lowerBound === undefined) {
    return {};
  }

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first
  // column of the primary key
  let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
  const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
  if (neParams.length) {
    condition += ' and not consensus_timestamp = any ($4)';
    params.push(neParams);
  }

  const query = `
    select consensus_timestamp
    from account_balance
    where ${condition}
    order by consensus_timestamp desc
    limit 1`;

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    return {};
  }

  const upper = rows[0].consensus_timestamp;
  const lower = utils.getFirstDayOfMonth(upper);
  return {lower, upper};
};
```
