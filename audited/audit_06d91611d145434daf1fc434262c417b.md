### Title
Unauthenticated `X-Real-IP` Header Spoofing Allows Audit Trail Falsification in Rosetta Tracing Middleware

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` header (and falls back to `X-Forwarded-For`) without any trusted-proxy validation. Because the Rosetta container exposes port 5700 directly, any unprivileged external client can inject an arbitrary IP value into the trace log, causing all Rosetta API calls to be attributed to a falsified source address in the audit trail.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)        // line 64 – blindly trusted

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67 – blindly trusted
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only fallback
    }

    return ipAddress
}
```

`TracingMiddleware` (lines 43–61) calls `getClientIpAddress` and embeds the result directly into the log message with no further sanitisation.

**Root cause:** The function applies a strict priority order — `X-Real-IP` → `X-Forwarded-For` → `RemoteAddr` — but never verifies that the request arrived from a trusted reverse proxy before accepting the header value. Any HTTP client that can reach the service can supply these headers.

**Deployment context that enables direct access:** `docker-compose.yml` line 151 maps `5700:5700`, exposing the Rosetta Go service directly on the host. The nginx proxy (lines 204–205) does set `X-Real-IP $remote_addr`, but that protection only applies when traffic flows through nginx. Direct connections to port 5700 bypass nginx entirely, so the client's own `X-Real-IP` value is accepted verbatim.

**Exploit flow:**
1. Attacker connects directly to `<host>:5700` (exposed port).
2. Attacker sends any Rosetta API request with header `X-Real-IP: 192.168.1.1` (or any target IP).
3. `getClientIpAddress` returns `"192.168.1.1"`.
4. `TracingMiddleware` logs: `192.168.1.1 POST /construction/submit (200) in 3ms`.
5. The audit trail now attributes the API call to the falsified address.

**Why existing checks are insufficient:** There are no checks. No trusted-proxy allowlist, no IP format/range validation, no stripping of the header before it reaches the logger. The test suite (`trace_test.go` lines 52–58) explicitly confirms that a client-supplied `X-Real-IP` or `X-Forwarded-For` value is accepted and logged as-is.

### Impact Explanation
The Rosetta API supports transaction construction and submission endpoints (`/construction/*`). The trace log is the primary record of which IP address initiated each API call. By spoofing `X-Real-IP`, an attacker can:
- Make their own transaction-submission calls appear to originate from a legitimate or trusted IP (e.g., an internal service, an administrator's address).
- Frame a third-party IP address for activity the attacker performed.
- Evade IP-based anomaly detection or rate-limit monitoring that relies on logged source addresses.

This does not alter on-chain state, but it fully corrupts the off-chain audit trail for Rosetta API activity, which is the only forensic record of who called which endpoint.

### Likelihood Explanation
Exploitation requires only the ability to send an HTTP request with a custom header — no credentials, no special tooling, no prior access. Port 5700 is directly exposed in the reference deployment. Any internet-reachable deployment where the Rosetta port is not firewalled is immediately exploitable. The attack is trivially repeatable and leaves no reliable forensic trace of the real source IP.

### Recommendation
1. **Restrict direct access:** Firewall port 5700 so only the nginx proxy (or a known trusted proxy) can reach the Rosetta service.
2. **Validate trusted proxy before trusting headers:** Maintain a configurable allowlist of trusted proxy CIDRs. In `getClientIpAddress`, only accept `X-Real-IP` / `X-Forwarded-For` if `r.RemoteAddr` falls within that allowlist; otherwise fall back to `r.RemoteAddr` directly.
3. **Validate IP format:** After extracting the value, call `net.ParseIP()` and discard the header if it does not parse as a valid IP address.

### Proof of Concept
```bash
# Direct connection to Rosetta port, bypassing nginx
curl -s \
  -H "X-Real-IP: 10.0.0.1" \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"}}' \
  http://<host>:5700/network/list

# Rosetta trace log will now contain:
# time="..." level=info msg="10.0.0.1 POST /network/list (200) in Xms"
