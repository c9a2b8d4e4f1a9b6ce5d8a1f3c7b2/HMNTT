### Title
External Users Can Suppress Access Logs to Info-Level by Targeting `internalPaths` Endpoints

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` logs requests to `/metrics`, `/health/liveness`, and `/health/readiness` at `Debug` level instead of `Info` level. Because these endpoints are served on the same public port as the API with no authentication or network-level restriction, any unprivileged external user can send requests to these paths and have their access completely invisible in production logs (where the default log level is `info`).

### Finding Description
**Code path:** `rosetta/app/middleware/trace.go`, `TracingMiddleware()`, lines 55–59:

```go
if internalPaths[path] {
    log.Debug(message)
} else {
    log.Info(message)
}
```

`internalPaths` is defined at line 19 as `{"/health/liveness": true, "/metrics": true, "/health/readiness": true}`.

The path is derived from `request.URL.RequestURI()` (line 47) — a plain string lookup with no normalization. Any request whose URI exactly matches one of these three strings is silently downgraded to `Debug`.

In `rosetta/main.go` lines 155–166, the server starts with `configLogger("info")` and then applies the operator-configured level. In a standard production deployment the effective level is `info`, meaning `Debug` messages are never emitted. All three `internalPaths` endpoints are registered in the same router (`server.NewRouter(...)`, lines 111–119 and 152) and bound to the single public port (`fmt.Sprintf(":%d", rosettaConfig.Port)`, line 221). There is no authentication middleware, no IP allowlist, and no separate internal-only listener.

**Root cause / failed assumption:** The code assumes only internal systems (Kubernetes probes, Prometheus scrapers) will call these paths. No mechanism enforces that assumption, so any external caller that sends a request with a URI of exactly `/metrics`, `/health/liveness`, or `/health/readiness` triggers the Debug branch and disappears from production logs.

### Impact Explanation
An attacker can repeatedly `GET /metrics` to harvest Prometheus histograms (request counts, error rates, latency distributions, inflight request gauges) and health status without a single `Info`-level log entry being produced. This constitutes:
- **Undetected reconnaissance**: operational telemetry is exfiltrated with no audit trail visible to standard SIEM/log-monitoring tooling.
- **Security monitoring bypass**: any alerting rule keyed on `Info`-level access logs will never fire for these paths, regardless of request volume or source IP.

Severity is moderate: the data exposed is observability metadata rather than user data, but the log-evasion property undermines the integrity of the audit trail for those endpoints.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and only knowledge of the three well-known path strings (all are standard Prometheus/Kubernetes conventions). It is trivially repeatable with a single `curl` command. Any attacker who has network access to the Rosetta port can exploit this deterministically and indefinitely.

### Recommendation
1. **Bind internal endpoints to a separate, non-public port** (e.g., a dedicated management listener on `localhost` or a private network interface). This is the strongest fix and eliminates the exposure entirely.
2. If a single port is required, **add an IP-allowlist middleware** that restricts `/metrics` and `/health/*` to known internal CIDR ranges before `TracingMiddleware` runs.
3. As a defence-in-depth measure, **always log at `Info` level regardless of path**, and rely on log-aggregation filters (not the application) to suppress high-frequency health-check noise. This ensures the audit trail is complete.

### Proof of Concept
```bash
# Default production deployment, log level = info
# Terminal 1: observe server logs (no debug output expected)
# Terminal 2: send requests to internal paths

curl http://<rosetta-host>:<port>/metrics
# → Returns full Prometheus metrics page
# → Zero Info-level log entries produced on the server

curl http://<rosetta-host>:<port>/health/liveness
# → Returns {"status":"OK"}
# → Zero Info-level log entries produced on the server

# Repeat indefinitely; server logs remain silent at Info level.
# To confirm: set server log level to debug and observe the suppressed entries appear.
