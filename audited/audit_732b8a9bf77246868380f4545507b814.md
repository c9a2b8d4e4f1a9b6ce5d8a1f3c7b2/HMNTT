### Title
Connection Pool Exhaustion via Synchronous jOOQ Blocking in Unauthenticated NFT Allowance Endpoint

### Summary
The `findAll` method in `NftAllowanceRepositoryCustomImpl` uses jOOQ's synchronous `fetchInto()` which holds a HikariCP JDBC connection for the entire duration of query execution and result-set deserialization. The `/api/v1/accounts/{id}/allowances/nfts` endpoint is publicly accessible with no authentication and no rate limiting in the `rest-java` module, allowing an unprivileged attacker to exhaust the connection pool by flooding it with concurrent requests, denying service to all other consumers of the database pool.

### Finding Description
**Code path:**

`AllowancesController.getNftAllowances()` → `NftAllowanceServiceImpl.getNftAllowances()` → `NftAllowanceRepositoryCustomImpl.findAll()` → `.fetchInto(NftAllowance.class)` (line 46). [1](#0-0) 

The `fetchInto()` call is a fully synchronous, blocking JDBC operation. It acquires a connection from the HikariCP pool at the start of the query and does not release it until the entire result set has been fetched and mapped into `NftAllowance` objects. HikariCP's default `maximumPoolSize` is 10; no override is present in the `rest-java` module resources. [2](#0-1) 

The `AllowancesController` applies only `@Max(MAX_LIMIT)` and `@Positive` on the `limit` parameter — no rate limiting, no concurrency cap, no authentication. [3](#0-2) 

The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists exclusively in the `web3` module and is not wired into `rest-java`. [4](#0-3) 

**Root cause / failed assumption:** The design assumes that upstream infrastructure (load balancer, reverse proxy) will absorb burst traffic before it reaches the application. No in-process guard prevents more concurrent requests from reaching `fetchInto()` than there are pool connections.

### Impact Explanation
Once all HikariCP connections are held by in-flight `fetchInto()` calls, every subsequent request that needs a database connection — including requests to unrelated endpoints served by the same datasource — will block until `connectionTimeout` expires and then throw `SQLTransientConnectionException`. This renders the entire `rest-java` service effectively unavailable for the duration of the attack. Because the pool is shared, the blast radius extends beyond the NFT allowance endpoint to all endpoints backed by the same `DataSource` bean. [5](#0-4) 

### Likelihood Explanation
The endpoint requires zero authentication and zero privileges. An attacker needs only a network path to the service and the ability to issue concurrent HTTP GET requests — trivially achievable with any HTTP load-testing tool (`wrk`, `ab`, `hey`) or a small botnet. The attack is repeatable and cheap: each request costs the attacker nothing beyond a TCP connection, while each request costs the server a scarce pool connection for the full query round-trip. With a default pool of 10 connections and no rate limiting, saturation requires only 10 simultaneous slow queries.

### Recommendation
1. **Add per-endpoint rate limiting** in `rest-java` analogous to the `ThrottleConfiguration` already present in `web3`, applied at the servlet filter or Spring MVC interceptor level before any database work begins.
2. **Set an explicit `maximumPoolSize`** via `spring.datasource.hikari.maximum-pool-size` tuned to the expected concurrency, and configure `connectionTimeout` to fail fast rather than queue indefinitely.
3. **Set a query-level statement timeout** (e.g., via jOOQ's `Settings.queryTimeout` or a datasource-level `statement_timeout`) so that slow or adversarially expensive queries release connections promptly.
4. **Consider async/reactive execution** (e.g., Spring WebFlux + R2DBC) for this endpoint to decouple HTTP threads from database connections entirely.

### Proof of Concept
```bash
# Saturate the 10-connection default pool with concurrent blocking queries
# Replace <HOST> and <VALID_ACCOUNT_ID> with real values
wrk -t10 -c50 -d30s \
  "http://<HOST>/api/v1/accounts/<VALID_ACCOUNT_ID>/allowances/nfts?limit=100"
```
**Expected result:** After the first 10 concurrent requests acquire all pool connections, subsequent requests receive HTTP 500 responses with `SQLTransientConnectionException: HikariPool-1 - Connection is not available, request timed out after Nms`. Legitimate traffic to other endpoints on the same service is simultaneously denied.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
