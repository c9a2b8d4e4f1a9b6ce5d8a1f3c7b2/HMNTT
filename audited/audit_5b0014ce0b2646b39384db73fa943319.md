### Title
Unbounded Concurrent Slow-Consumer Subscriptions Enable Memory Exhaustion via Per-Subscriber Backpressure Buffers

### Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no limit on the number of concurrent subscribers. Each subscription flows through `SharedTopicListener.listen()`, which allocates a dedicated per-subscriber in-memory buffer of up to 16,384 `TopicMessage` objects via `onBackpressureBuffer`. An unprivileged attacker can open arbitrarily many concurrent subscriptions to a high-volume topic while consuming messages slowly, causing the server to hold N × 16,384 messages in heap simultaneously, leading to memory exhaustion and denial of service.

### Finding Description

**Code path 1 — no subscriber cap in `subscribeTopic()`:**

`TopicMessageServiceImpl.java` lines 88–91 track `subscriberCount` only as a Micrometer `Gauge` metric. There is no guard that rejects or throttles new subscriptions when the count exceeds any threshold.

```java
return topicExists(filter)
    .thenMany(flux.doOnNext(topicContext::onNext)
        .doOnSubscribe(s -> subscriberCount.incrementAndGet())   // metric only
        .doFinally(s -> subscriberCount.decrementAndGet())
        .doFinally(topicContext::finished));
```

`GrpcProperties.java` contains no `maxSubscribers` or per-IP connection limit field. `TopicMessageFilter.java` contains no rate-limit or concurrency field.

**Code path 2 — per-subscriber heap buffer in `SharedTopicListener.listen()`:**

`SharedTopicListener.java` lines 22–25:

```java
return getSharedListener(filter)
    .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
    .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
```

`ListenerProperties.java` line 23: `maxBufferSize = 16384` (default, minimum 8192, maximum 65536).

Each call to `listen()` — one per subscriber — creates an **independent** `onBackpressureBuffer` operator. The shared hot publisher (`RedisTopicListener` / `SharedPollingTopicListener`, both using `.share()`) fans out to all subscribers; each subscriber's buffer is allocated separately on the JVM heap.

**Why `BufferOverflowStrategy.ERROR` is insufficient:**
`ERROR` terminates a single slow subscriber only after its buffer is already full (16,384 items). It does not prevent:
- Many concurrent slow subscribers each holding up to 16,384 items simultaneously before any one of them overflows.
- An attacker immediately reconnecting after being terminated, continuously re-filling the buffer.

**Root cause:** The service layer conflates "subscriber will eventually be terminated" with "subscriber cannot cause resource exhaustion." The termination is per-subscriber and deferred; the damage (heap allocation) is immediate and cumulative across all concurrent subscribers.

### Impact Explanation

With `maxBufferSize = 16384` and a conservatively estimated `TopicMessage` size of 500 bytes (consensus timestamp, topic ID, sequence number, message bytes, running hash):

- 100 concurrent slow subscribers × 16,384 × 500 B ≈ **800 MB**
- 500 concurrent slow subscribers × 16,384 × 500 B ≈ **4 GB**

Real topic messages on Hedera can be larger (up to 1 KB+), amplifying the impact. The `boundedElastic` scheduler used by `publishOn` has a thread-pool cap but does not bound heap. The result is JVM `OutOfMemoryError`, crashing the gRPC service for all legitimate users. Severity: **High**.

### Likelihood Explanation

- **No authentication required.** The gRPC `subscribeTopic` endpoint is publicly accessible; `TopicMessageFilter` requires only a valid `topicId` and a `startTime` in the past.
- **Trivially scriptable.** An attacker opens N gRPC streaming connections in parallel, sends a `subscribeTopic` request on each, and simply never reads responses (or reads them at a rate slower than message production).
- **Repeatable.** After being disconnected by `BufferOverflowStrategy.ERROR`, the attacker reconnects immediately. The reconnect loop can be automated.
- **Low cost.** A single machine with many TCP connections suffices; no special privileges, tokens, or knowledge of internal state are needed.

### Recommendation

1. **Enforce a global and per-IP subscriber limit.** Check `subscriberCount` (or a per-IP counter) before accepting a new subscription in `subscribeTopic()` and return `RESOURCE_EXHAUSTED` if the limit is exceeded.
2. **Apply `onBackpressureLatest` or `onBackpressureDrop` instead of `onBackpressureBuffer` for live streams**, or reduce `maxBufferSize` significantly and combine with a per-subscriber timeout that terminates connections that have not drained their buffer within a configurable window.
3. **Add a per-subscriber message-rate floor.** If a subscriber's drain rate falls below a threshold for more than N seconds, terminate the subscription proactively rather than waiting for buffer overflow.
4. **Limit concurrent subscriptions per topic** to prevent a single topic from being used as an amplification vector.

### Proof of Concept

```python
import grpc
import threading
# Assume generated stubs from consensus_service.proto
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = "<high-volume-topic-id>"
NUM_CONNECTIONS = 300

def slow_subscribe():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        consensusStartTime={"seconds": 0}
    )
    # Open stream but never read — or read with artificial delay
    stream = stub.subscribeTopic(request)
    import time
    time.sleep(3600)  # hold connection open without consuming

threads = [threading.Thread(target=slow_subscribe) for _ in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# Server heap fills with NUM_CONNECTIONS × 16384 × message_size bytes
# Monitor server JVM heap; OOM or GC thrashing observed within seconds on high-volume topics
```

**Preconditions:** Network access to the gRPC port; knowledge of any active high-volume topic ID (publicly observable on-chain).
**Trigger:** 300+ concurrent connections each holding a slow/stalled stream.
**Result:** JVM heap exhaustion → `OutOfMemoryError` → gRPC service crash for all users.
