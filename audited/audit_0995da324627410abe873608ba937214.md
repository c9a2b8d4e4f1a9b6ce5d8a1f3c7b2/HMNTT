### Title
Uncached Alias Resolution in `getAccountCryptoAllowances` Doubles DB Load Per Request, Enabling Amplified DoS by Unprivileged Users

### Summary
The `GET /api/v1/accounts/{idOrAliasOrEvmAddress}/allowances/crypto` endpoint unconditionally calls `EntityService.getEncodedId` before every `CryptoAllowanceService.getAccountCryptoAllowances` invocation. When the path parameter is a public-key alias, `getEncodedId` issues a synchronous DB query against the `entity` table with no caching, then a second DB query against `crypto_allowance`. No rate limiting exists on this endpoint, and the response cache only helps for byte-identical repeated URLs. An unprivileged attacker can therefore force 2× the DB queries per request compared to numeric-ID requests, amplifying DoS efficiency against the mirror node's database tier.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js:17` routes `GET /:idOrAliasOrEvmAddress/allowances/crypto` to `CryptoAllowanceController.getAccountCryptoAllowances`.

`rest/controllers/cryptoAllowanceController.js:77`:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```
This is called unconditionally on every request.

`rest/service/entityService.js:125-126`:
```js
} else if (AccountAlias.isValid(entityIdString)) {
  return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
```
When the path parameter matches the alias regex (`/^(\d{1,5}\.){0,2}[A-Z2-7]+$/` — `rest/accountAlias.js:10`), `getAccountIdFromAlias` is called.

`rest/service/entityService.js:42-43`:
```js
async getAccountFromAlias(accountAlias) {
  const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
```
This executes:
```sql
select id from entity
where coalesce(deleted, false) <> true
  and alias = $1
```
against the `entity` table — a full DB round-trip with no result caching.

After this, `cryptoAllowanceController.js:80` calls:
```js
const allowances = await CryptoAllowanceService.getAccountCryptoAllowances(conditions, params, order, limit);
```
which executes a second DB query against `crypto_allowance`.

**Root cause:** `EntityService` has no in-process or Redis-backed cache for alias→ID resolution. `rest/cache.js` implements a Redis cache, but it is never wired into `getAccountFromAlias` or `getEncodedId`. The response-level cache (`rest/middleware/responseCacheHandler.js:151-153`) keys on `MD5(req.originalUrl)`, so it only helps for byte-identical repeated URLs — an attacker using distinct valid aliases or rotating through aliases bypasses it entirely.

**Failed assumption:** The design assumes alias-based lookups are infrequent or that upstream caching absorbs repeated alias queries. Neither is enforced in code.

### Impact Explanation
Every alias-based request to `/api/v1/accounts/{alias}/allowances/crypto` causes 2 DB queries instead of 1. With no per-IP or global rate limiting on this public endpoint (`rest/middleware/requestHandler.js` contains no throttle logic; the `config.users` authentication system only caps result-set size, not request rate), an attacker can sustain a request flood where each request generates 2× the DB load of a numeric-ID request. At sufficient concurrency, this saturates the PostgreSQL connection pool and query executor, degrading all mirror node REST processing — including consensus ingestion pipelines that share the same DB — consistent with the ≥30% processing degradation threshold.

### Likelihood Explanation
The attack requires zero privileges: the endpoint is publicly accessible, no API key or authentication is needed. Valid alias strings are trivially constructable (base32 uppercase, e.g., `KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ`). An attacker needs only a standard HTTP client and a list of syntactically valid alias strings (real or fabricated — even non-existent aliases still hit the `entity` table). The attack is fully repeatable and scriptable with tools like `wrk`, `ab`, or `k6`.

### Recommendation
1. **Cache alias→ID resolution** in `EntityService.getAccountFromAlias` using the existing Redis `Cache` infrastructure (`rest/cache.js`) with a short TTL (e.g., 30–60 s), keyed on the base32 alias string.
2. **Add request-rate limiting** (e.g., via `express-rate-limit`) on the `/accounts/:id/allowances/*` route family, applied per source IP without requiring authentication.
3. **Short-circuit on 404**: when `getAccountIdFromAlias` returns null (alias not found), return 404 immediately without proceeding to `getAccountCryptoAllowances` — this is already done, but caching the negative result avoids repeated entity-table hits for the same non-existent alias.

### Proof of Concept

```bash
# Generate a syntactically valid alias (real or fabricated)
ALIAS="KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ"

# Flood with alias-based requests (each causes 2 DB queries)
wrk -t 10 -c 200 -d 60s \
  "http://<mirror-node-host>/api/v1/accounts/${ALIAS}/allowances/crypto"

# Compare DB query rate vs numeric-ID baseline (1 DB query each)
wrk -t 10 -c 200 -d 60s \
  "http://<mirror-node-host>/api/v1/accounts/0.0.1001/allowances/crypto"
```

Observe via DB monitoring (e.g., `pg_stat_activity`) that alias-based flood generates ~2× the active queries compared to the numeric-ID baseline at identical concurrency, with corresponding increase in DB CPU and connection saturation.
