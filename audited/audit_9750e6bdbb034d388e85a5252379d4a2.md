### Title
Shared Poll Flux Re-subscription Resets `lastConsensusTimestamp` to `now()`, Causing Live Message Loss

### Summary
In `SharedPollingTopicListener`, the single `PollingContext.onStart` callback unconditionally resets `lastConsensusTimestamp` to `DomainUtils.now()` every time the upstream Flux is re-subscribed. Because the upstream is wrapped with `.share()` (i.e., `publish().refCount()`), it is cancelled when the subscriber count drops to zero and re-subscribed when the next subscriber arrives. Any unprivileged external user can be the sole subscriber, disconnect, and force this reset, causing the shared poll loop to skip all messages that arrived during the gap.

### Finding Description

**Exact code path:**

`SharedPollingTopicListener` constructor builds a single `topicMessages` Flux:

```
grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedPollingTopicListener.java
lines 41–52
```

```java
topicMessages = Flux.defer(() -> poll(context).subscribeOn(scheduler))
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE).withFixedDelay(interval)...)
        ...
        .doOnSubscribe(context::onStart)   // ← fires on every upstream subscription
        .retryWhen(Retry.backoff(...))
        .share();                           // ← publish().refCount()
```

`getSharedListener` (line 56–58) returns this single `topicMessages` Flux for every caller, regardless of filter.

`PollingContext.onStart` (lines 101–106):

```java
void onStart(Subscription subscription) {
    lastConsensusTimestamp.set(DomainUtils.now());   // ← unconditional reset
    ...
}
```

`poll` uses `lastConsensusTimestamp` as the lower bound for every DB query (line 66):

```java
topicMessageRepository.findLatest(
    context.getLastConsensusTimestamp().get(), listenerProperties.getMaxPageSize())
```

**Root cause:** `.share()` cancels the upstream when the last subscriber unsubscribes and re-subscribes when the first new subscriber arrives. Re-subscription triggers `doOnSubscribe` → `onStart` → unconditional `lastConsensusTimestamp = now()`. There is no "already initialized" guard.

**Access control:** `ConsensusController.subscribeTopic` (lines 43–53) has zero authentication. The gRPC endpoint is plaintext-accessible to any network client (`grpcurl -plaintext ...`).

**Exploit flow:**

1. Attacker opens a gRPC `subscribeTopic` stream to any valid topic (no credentials needed).
2. Attacker is the sole subscriber → `.share()` subscribes upstream → `onStart` sets `lastConsensusTimestamp = T0`.
3. Poll loop runs normally; `lastConsensusTimestamp` advances to `T_last` as messages arrive.
4. Attacker closes the stream → subscriber count drops to 0 → `.share()` cancels upstream.
5. Real messages with consensus timestamps `T_last < Tx < T_now` are written to the DB during the gap.
6. A legitimate subscriber connects → `.share()` re-subscribes → `onStart` fires → `lastConsensusTimestamp = T_now`.
7. `poll` now queries `findLatest(T_now, ...)`, permanently skipping all messages in `(T_last, T_now)`.

**Why existing mitigations are insufficient:**

- The `safetyCheck` in `TopicMessageServiceImpl` (line 67–70) fires **once** after 1 second and only covers the gap between historical retrieval and the first live message. It does not continuously recover from a reset shared-listener cursor.
- The `historical` retriever covers messages up to the subscriber's `startTime`, not up to `T_now`. If the subscriber's `startTime` predates the gap, historical retrieval covers it; but if the subscriber sets `startTime ≈ now()` (the default per the proto spec), the gap messages fall between historical's end and the shared listener's new cursor and are never delivered.
- The `missingMessages` gap-fill in `incomingMessages` (line 120) is triggered only when a live message is received; if the shared listener skips the gap entirely, no trigger fires.

### Impact Explanation

Any messages written to the DB during the window between the last subscriber's disconnect and the next subscriber's connect are permanently skipped by the shared poll loop. Subscribers relying on `SHARED_POLL` mode for live delivery will silently miss those messages with no error or retry. This is a data-integrity failure for HCS consumers: sequence numbers will appear contiguous from the client's perspective only if the per-subscriber gap-fill happens to fire in time, which is not guaranteed.

### Likelihood Explanation

The gRPC endpoint requires no authentication. Any external client can open and immediately close a `subscribeTopic` stream. The attack is trivially repeatable: the attacker can cycle connect/disconnect to continuously reset the cursor whenever they choose, maximizing the gap window. No special knowledge of the system internals is required beyond knowing the topic ID (which is public on-chain data).

### Recommendation

Guard `onStart` so it only initializes `lastConsensusTimestamp` on the very first subscription, not on re-subscriptions:

```java
void onStart(Subscription subscription) {
    if (lastConsensusTimestamp.compareAndSet(0L, DomainUtils.now())) {
        log.info("Starting to poll every {}ms", listenerProperties.getInterval().toMillis());
    }
}
```

Alternatively, move the timestamp initialization outside `onStart` into the `PollingContext` constructor (initializing to `DomainUtils.now()` at construction time), so re-subscription never resets it. The `PollingContext` already survives across re-subscriptions (it is created once in the constructor), so this is safe.

### Proof of Concept

**Preconditions:** Mirror node running with `hiero.mirror.grpc.listener.type=SHARED_POLL`. Topic with known ID exists and is receiving messages.

**Steps:**

1. Open a `subscribeTopic` gRPC stream to the topic (no auth needed):
   ```
   grpcurl -plaintext -d '{"topicID":{"topicNum":41110}}' <host>:5600 \
     com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
   ```
2. Confirm the shared poll loop starts (log: `"Starting to poll every 500ms"`).
3. Wait for at least one poll cycle to confirm `lastConsensusTimestamp` has advanced past `T0`.
4. Close the gRPC stream (Ctrl-C / disconnect). Log shows `"Cancelled polling"`.
5. Wait for new messages to be written to the DB (e.g., submit HCS messages to the topic).
6. Open a new `subscribeTopic` stream with `consensusStartTime` set to the current time.
7. Observe log: `"Starting to poll every 500ms"` fires again — `lastConsensusTimestamp` reset to `now()`.
8. The messages written in step 5 are never delivered to the new subscriber, confirming message loss. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
