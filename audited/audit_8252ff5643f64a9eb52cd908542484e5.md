### Title
Unauthenticated Cache-Thrashing via Block() Causes O(N×ops) DB Queries and Connection Pool Exhaustion

### Summary
The `updateOperationAccountAlias()` function in `rosetta/app/services/block_service.go` issues one `GetAccountAlias()` database query per operation whose account ID is absent from the shared LRU `entityCache`. Because the Rosetta `/block` endpoint requires no authentication and has no application-level rate limiting (the Traefik middleware chain is disabled by default), an unprivileged attacker can flood the endpoint with requests for blocks containing many distinct account IDs, continuously evicting warm cache entries and forcing sustained DB queries that exhaust the connection pool for legitimate users.

### Finding Description

**Code path:**

`rosetta/app/services/block_service.go:104-130` — `updateOperationAccountAlias()`:
```go
for _, transaction := range transactions {
    operations := transaction.Operations
    for index := range operations {
        accountId := operations[index].AccountId
        if cached, found = s.entityCache.Get(accountId.GetId()); !found {
            result, err := s.accountRepo.GetAccountAlias(ctx, accountId)  // DB hit
            ...
            s.entityCache.Set(result.GetId(), result)
        }
    }
}
```

Each cache miss calls `GetAccountAlias()` in `rosetta/app/persistence/account.go:109-136`, which executes:
```sql
select alias, id from entity where id = @id
```
— one synchronous DB round-trip per unique uncached account ID.

**Root cause — failed assumption:** The design assumes the LRU cache will absorb repeated lookups. The failed assumption is that the cache will remain warm. Because the cache is a plain LRU with no TTL-based refresh and no per-IP or global request rate limit at the application layer, an attacker can continuously introduce new account IDs (sourced from real on-chain blocks) to evict warm entries, keeping the cache in a perpetually cold state for the IDs that legitimate users need.

**Exploit flow:**
1. Attacker enumerates blocks on mainnet/testnet that contain many transactions with many distinct account IDs (easily found via the public mirror node REST API).
2. Attacker sends a high volume of concurrent `POST /block` requests cycling through these blocks.
3. Each request triggers up to `maxTransactionsInBlock × ops_per_tx` calls to `GetAccountAlias()` for cache misses.
4. With the LRU capacity at 524,288 entries and mainnet having millions of accounts, the attacker can keep rotating through fresh account IDs, continuously evicting previously cached entries.
5. The DB connection pool (`maxOpenConnections=100`) is saturated; legitimate requests queue or fail.

**Why existing checks are insufficient:**

- **Traefik middleware** (`charts/hedera-mirror-rosetta/values.yaml:95`, `global.middleware: false`) — the rate-limit middleware (`average: 10` req/s per host) is **disabled by default**. Operators who deploy without enabling it have no protection.
- **`maxTransactionsInBlock`** — limits the number of transactions returned per response but does not bound the number of distinct account IDs across those transactions' operations. A single block with 100 transactions each having 10 operations with unique accounts yields 1,000 DB calls per request.
- **`statementTimeout`** (default 20 s) — only limits individual query duration, not the total number of concurrent queries.
- **No application-level concurrency or rate limit** exists in `main.go` or any Go middleware for the `/block` route.

### Impact Explanation
The DB connection pool (`maxOpenConnections=100`) is a finite shared resource. Sustained flooding with cache-miss-heavy Block() requests can hold all 100 connections busy with `GetAccountAlias()` queries, causing legitimate `/block`, `/account/balance`, and `/block/transaction` requests to queue indefinitely or time out. This degrades service availability for all users of the Rosetta API without any economic cost to the attacker. Severity is Medium (griefing/DoS with no direct financial loss).

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond public block data, and no on-chain activity. Any attacker with a script and network access to the Rosetta port can execute it. Block data needed to identify high-account-count blocks is freely available from the public mirror node REST API. The attack is repeatable and can be sustained indefinitely. Likelihood is **High** for deployments where `global.middleware` is left at its default `false`.

### Recommendation
1. **Add application-level rate limiting** in the Go server (e.g., `golang.org/x/time/rate` token bucket per IP) for the `/block` and `/block/transaction` endpoints, independent of the optional Traefik middleware.
2. **Enable the Traefik middleware by default** (`global.middleware: true`) or document it as a required security control.
3. **Cap the number of DB calls per request** by batching all uncached account IDs from a single Block() call into a single `SELECT alias, id FROM entity WHERE id = ANY(@ids)` query instead of N individual queries.
4. **Add a per-request concurrency semaphore** to bound the number of simultaneous `updateOperationAccountAlias()` executions sharing the DB pool.
