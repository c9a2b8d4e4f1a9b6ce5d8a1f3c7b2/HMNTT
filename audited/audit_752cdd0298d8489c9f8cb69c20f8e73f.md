### Title
Unauthenticated Timestamp-Triggered DISTINCT ON Full Scan on `contract_state_change` via `GET /contracts/:contractId/state`

### Summary
Any unauthenticated user can supply a `timestamp` query parameter to `GET /contracts/:contractId/state`, causing the server to switch from the efficient `contract_state` table query to `contractStateTimestampQuery`, which executes a `DISTINCT ON (slot)` scan over the entire `contract_state_change` table up to the given timestamp. Combined with a wide slot range (`slot=gte:0x00&slot=lte:0xff`), this forces the database to read and sort all historical state-change rows for the targeted contract before applying the output `LIMIT`, with no rate limiting on the REST endpoint to prevent repeated abuse.

### Finding Description

**Route registration** — `rest/routes/contractRoute.js:15` registers the endpoint with no authentication middleware:
```
router.getExt('/:contractId/state', ContractController.getContractStateById);
```

**Controller dispatch** — `getContractStateById` (`rest/controllers/contractController.js:964-994`) calls `extractContractStateByIdQuery`, then passes the `timestamp` boolean flag to `ContractService.getContractStateByIdAndFilters`.

**Query branching** — `extractContractStateByIdQuery` (`contractController.js:894-956`) sets `timestampPresent = true` whenever any `timestamp` filter is present (lines 909-918). Non-equality slot operators (`gte`, `lte`) are appended as range conditions on `slot` (lines 936-943). The function returns `{conditions, order, limit, timestamp: timestampPresent}`.

**Expensive query path** — `getContractStateByIdAndFilters` (`contractService.js:256-263`) switches to `contractStateTimestampQuery` when `timestamp === true`:

**The expensive query** (`contractService.js:111-122`):
```sql
with entity as (...)
SELECT DISTINCT ON (slot)
      contract_id, slot, evm_address,
      coalesce(value_written, value_read) as value,
      consensus_timestamp as modified_timestamp
FROM contract_state_change csc
LEFT JOIN entity e ON e.id = csc.contract_id
WHERE contract_id = $1
  AND consensus_timestamp <= $2   -- from timestamp param
  AND slot >= $3                  -- from slot=gte:0x00
  AND slot <= $4                  -- from slot=lte:0xff
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT $5
```

PostgreSQL's `DISTINCT ON` requires reading and sorting **all rows** matching the `WHERE` clause before it can identify the latest `consensus_timestamp` per slot and apply `LIMIT`. For a contract with millions of state-change rows, this is a full or near-full index scan of `contract_state_change` for that contract up to the given timestamp.

The index `contract_state_change__id_slot_timestamp (contract_id, slot, consensus_timestamp)` (`V1.77.1__contract_state_change__id_slot_timestamp.sql:2-3`) helps narrow the scan but does not eliminate it: for each distinct slot within the range, the database must still scan backwards through all `consensus_timestamp` values to find the latest one satisfying `<= $2`. For frequently-written slots this is many rows per slot.

**No rate limiting** — a grep across `rest/**/*.js` for `rateLimit`, `throttle`, and `express-rate-limit` returns only a single test-utility match. The throttle infrastructure found (`web3/src/main/java/...ThrottleConfiguration.java`) applies only to the separate `web3` Java service (contract call/eth RPC), not to the Node.js REST API serving this endpoint.

**Root cause**: The failed assumption is that a `LIMIT` clause on a `DISTINCT ON` query prevents expensive full scans. It does not — PostgreSQL must resolve all distinct groups before truncating output, and no per-IP or per-endpoint rate limiting exists to bound request frequency.

### Impact Explanation
An attacker targeting a high-activity contract (e.g., a popular DeFi contract with millions of state-change rows) can issue repeated requests that each force a large index scan plus sort on `contract_state_change`. With no rate limiting, concurrent requests from a single client can saturate database CPU and I/O, degrading or denying service to all other API consumers. The 30%+ resource consumption threshold is achievable by sending a modest number of concurrent requests (e.g., 10–20 parallel connections) against a contract with a large `contract_state_change` history.

### Likelihood Explanation
The attack requires zero privileges — only knowledge of a valid `contractId` for a contract with significant history, which is publicly discoverable via the same API. The timestamp value can be set to the current time (maximizing the scan range). The slot range `gte:0x00&lte:0xff` is a valid, accepted parameter combination as confirmed by the integration test spec (`rest/__tests__/specs/contracts/{id}/state/timestamp-slot.json:196-225`). The attack is trivially scriptable and repeatable.

### Recommendation
1. **Add rate limiting** to the Node.js REST API layer (e.g., `express-rate-limit`) scoped to the `/contracts/:contractId/state` endpoint, limiting requests per IP per time window.
2. **Restrict timestamp query to the `contract_state` table** when a slot range is provided without a specific slot equality filter, or require an equality `slot` filter when `timestamp` is used.
3. **Set a short `statement_timeout`** in the database connection pool specifically for this query path to bound worst-case execution time.
4. **Rewrite the timestamp path** to use a correlated subquery or lateral join per slot (leveraging the existing index on `(contract_id, slot, consensus_timestamp)`) rather than a global `DISTINCT ON` scan, so the database can stop after finding `LIMIT` slots without reading all rows.

```javascript
  async extractContractStateByIdQuery(filters, contractId) {
    let limit = defaultLimit;
    let order = orderFilterValues.ASC;
    let timestampPresent = false;
    const conditions = [this.getFilterWhereCondition(ContractState.CONTRACT_ID, {operator: '=', value: contractId})];
    const slotFilters = [];

    for (const filter of filters) {
      switch (filter.key) {
        case filterKeys.LIMIT:
          limit = filter.value;
          break;
        case filterKeys.ORDER:
          order = filter.value;
          break;
        case filterKeys.TIMESTAMP:
          if (utils.opsMap.ne === filter.operator) {
            throw new InvalidArgumentError(`Not equals (ne) operator is not supported for ${filterKeys.TIMESTAMP}`);
          }

          if (utils.opsMap.eq === filter.operator) {
            filter.operator = utils.opsMap.lte;
          }
          conditions.push(this.getFilterWhereCondition(ContractStateChange.CONSENSUS_TIMESTAMP, filter));
          timestampPresent = true;
          break;
        case filterKeys.SLOT:
          slotFilters.push(filter);
          break;
        default:
          break;
      }
    }

    const slotInValues = [];
    for (const slotFilter of slotFilters) {
      // Left pad the slot value if no timestamp filter is present
      // If a timestamp filter is present the slot value needs additional conversion
      // because there is an inconsistency between the column slot in contract_state and contract_state_change.
      const slot = utils.formatSlot(slotFilter.value, !timestampPresent);
      if (slotFilter.operator === utils.opsMap.eq) {
        slotInValues.push(slot);
      } else {
        conditions.push(
          this.getFilterWhereCondition(ContractState.SLOT, {
            operator: slotFilter.operator,
            value: slot,
          })
        );
      }
    }

    if (slotInValues.length !== 0) {
      conditions.push(this.getFilterWhereCondition(ContractState.SLOT, {operator: 'in', value: slotInValues}));
    }

    return {
      conditions,
      order,
      limit,
      timestamp: timestampPresent,
    };
  }
```
