### Title
Cache Stampede in `getCachedTokens()` Allows Unauthenticated Non-Network DoS via DB Connection Pool Exhaustion

### Summary
`TokenService.getCachedTokens()` in `rest/service/tokenService.js` contains no in-flight request deduplication. Because the cache is only populated after `await super.getRows()` resolves, any number of concurrent requests that arrive during the DB round-trip will each independently observe a cache miss and each issue their own DB query. An unprivileged attacker can exploit this to saturate the database connection pool and cause a non-network DoS.

### Finding Description
**Exact code path:** `rest/service/tokenService.js`, `getCachedTokens()`, lines 136–161, called from `getTokenAccounts()` (line 104), which is called from the unauthenticated public endpoint `GET /api/v1/accounts/{id}/tokens` via `tokenController.getTokenRelationships()` (line 74 of `tokenController.js`).

**Root cause:** Node.js is single-threaded but cooperative. The synchronous cache-check loop (lines 139–146) and the `await super.getRows(...)` call (line 152) create a window:

```
Request A: checks cache → miss → hits `await getRows(...)` → yields event loop
Request B: checks cache → still miss (A hasn't written yet) → hits `await getRows(...)` → yields
...
Request N: same
```

The cache is only written at line 156 (`tokenCache.set(tokenId, cachedToken)`) after the DB response arrives. All N concurrent requests that entered `getCachedTokens()` before any response returns will each independently issue `super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds])` against the database.

**Failed assumption:** The code assumes that the first in-flight request will populate the cache before subsequent requests check it. This is false under concurrent async load.

**Why existing checks are insufficient:** The only guard is `if (uncachedTokenIds.length === 0) return cachedTokens;` at line 148. This only short-circuits when the cache is already warm. There is no mutex, no promise coalescing, and no "pending" sentinel to prevent duplicate in-flight queries for the same token IDs. [1](#0-0) [2](#0-1) 

### Impact Explanation
Each concurrent request issues a full `SELECT decimals, freeze_status, kyc_status, token_id FROM token WHERE token_id = any ($1)` query. With N concurrent attackers and a DB connection pool of fixed size (typical default: 10–20 connections), N queries will queue or fail. Once the pool is exhausted, all other API users receive connection errors or timeouts, constituting a complete non-network DoS of the mirror node REST API. The attack is amplified by choosing accounts with large token association sets (maximizing per-query cost) and is repeatable: after the cache warms, the attacker can wait for cache eviction (the `quick-lru` cache has a finite `maxSize` configured at line 13) and repeat. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
The endpoint `GET /api/v1/accounts/{id}/tokens` requires no authentication. Any external user can send concurrent HTTP requests. The attack requires only a standard HTTP client capable of concurrent requests (e.g., `curl`, `ab`, `wrk`, or a simple script). The attacker does not need any tokens, keys, or privileged access. The attack is trivially repeatable and can be scripted to fire continuously, keeping the DB pool saturated indefinitely. [5](#0-4) 

### Recommendation
Introduce in-flight deduplication by storing a `Map<tokenId, Promise>` of pending DB queries. Before issuing a new query, check whether a promise for the same set of token IDs is already in-flight and await it instead of issuing a duplicate query. Alternatively, use a well-known pattern such as a "single-flight" / promise-coalescing wrapper:

```js
// Pseudocode
const inFlight = new Map(); // tokenId -> Promise<CachedToken>

async getCachedTokens(tokenIds) {
  // For each uncached tokenId, reuse existing in-flight promise or create new one
  const promises = uncachedTokenIds.map(id => {
    if (!inFlight.has(id)) {
      const p = fetchFromDb(id).finally(() => inFlight.delete(id));
      inFlight.set(id, p);
    }
    return inFlight.get(id);
  });
  await Promise.all(promises);
  // ...
}
```

This ensures that N concurrent requests for the same uncached token ID result in exactly one DB query.

### Proof of Concept
**Preconditions:**
- Mirror node REST API is running and accessible
- At least one account ID with token associations exists (e.g., account `0.0.98`)
- Token cache is cold (freshly started, or `maxSize` exceeded)

**Steps:**
```bash
# Send 100 concurrent requests for the same account's tokens
# before any DB response can populate the cache
for i in $(seq 1 100); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.98/tokens" &
done
wait
```

**Result:** All 100 requests enter `getCachedTokens()` concurrently, each observes a cache miss (the cache is not yet populated), and each independently issues `SELECT ... FROM token WHERE token_id = any ($1)` against the database. With a typical pool size of 10–20, 80–90 queries queue or time out. Repeating this in a loop exhausts the connection pool continuously, causing all other API requests to fail with DB connection errors — a sustained non-network DoS. [6](#0-5)

```javascript
  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
