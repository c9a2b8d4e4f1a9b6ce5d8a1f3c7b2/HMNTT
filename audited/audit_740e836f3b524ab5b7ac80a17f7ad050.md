### Title
Rate Limit Token Consumed Without Rollback When Gas Bucket Check Fails

### Summary
In `ThrottleManagerImpl.throttle()`, the `rateLimitBucket` token is consumed before the `gasLimitBucket` is checked. When the gas check fails, the rate limit token is silently discarded — there is no rollback. The `restore()` method only returns gas tokens, never rate limit tokens, making this asymmetry permanent within each refill window.

### Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–42

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);  // rate token already gone
}
```

**Root cause:** The two `tryConsume` calls are not atomic. When line 38 succeeds and line 40 fails, 1 rate limit token is permanently consumed for that second. The `restore()` method (lines 58–64) only calls `gasLimitBucket.addTokens(...)` — it has no corresponding path to return the rate limit token.

**Exploit flow:**
1. Attacker sends one request with `gas = maxGasLimit` (e.g., 15,000,000). This consumes 1 rate limit token and 1,500 scaled gas tokens (15,000,000 / 10,000).
2. Attacker repeats until the gas bucket (capacity ≈ 750,000 scaled tokens at default `gasPerSecond = 7,500,000,000`) is exhausted — roughly 500 requests, which also exhausts the rate limit bucket (capacity = 500 at default `requestsPerSecond = 500`).
3. After the gas bucket is exhausted but before the rate limit bucket refills, any additional request consumes 1 rate limit token and 0 gas tokens (gas check fails immediately). This drains the rate limit bucket faster than the gas bucket refills.
4. Legitimate users receive `REQUEST_PER_SECOND_LIMIT_EXCEEDED` even though no gas was consumed on their behalf.

**Why existing checks fail:** `ContractController.restore()` is only called on `InvalidParametersException` (line 48 of `ContractController.java`), not on `ThrottleException`. Even if it were called, `restore()` only adds back gas tokens — the rate limit token has no corresponding return path anywhere in the codebase.

### Impact Explanation
An unprivileged attacker with no credentials can exhaust the shared `rateLimitBucket` within a single refill window (1 second) by sending requests that pass the rate check but fail the gas check. Legitimate API users are then rejected with a 429-equivalent `ThrottleException` for the remainder of that second. Because both buckets use greedy refill with a 1-second period, the attack is repeatable every second, creating a sustained denial-of-service against all users sharing the same bucket instance. The default `requestsPerSecond = 500` means the attacker needs only 500 cheap HTTP requests per second to hold the service in a throttled state continuously.

### Likelihood Explanation
No authentication, API key, or IP restriction is required. Any external caller can POST to `/api/v1/contracts/call` with a high `gas` value. The attack is trivially scriptable with any HTTP client (e.g., `curl` in a loop or a simple load-testing tool). The attacker does not need to know contract addresses or valid calldata — the throttle check runs before any EVM execution. The attack is repeatable indefinitely at 500 req/s, which is well within the capability of a single machine.

### Recommendation
Replace the sequential non-atomic check with one of the following:

1. **Check-then-consume atomically:** Use Bucket4j's `Bucket.asVerbose().tryConsume(...)` or probe both buckets before consuming either, then consume both only if both would succeed.
2. **Rollback on failure:** If `gasLimitBucket.tryConsume(...)` fails, immediately call `rateLimitBucket.addTokens(1)` to return the consumed rate limit token before throwing.
3. **Reverse check order:** Check `gasLimitBucket` first (without consuming), then consume both. Bucket4j supports `tryConsumeAsMuchAsPossible` and `estimateAbilityToConsume` for non-destructive probing.

Example minimal fix for option 2:
```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);  // rollback
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

### Proof of Concept
**Preconditions:** Default config (`requestsPerSecond=500`, `gasPerSecond=7_500_000_000`). No authentication required.

**Steps:**
1. Send 500 POST requests to `/api/v1/contracts/call` with `gas=15000000` (max gas) in rapid succession within 1 second. The first ~500 requests exhaust the gas bucket; each also consumes 1 rate limit token.
2. Immediately send 1 more POST request with `gas=15000000`. The rate limit bucket still has tokens (or has partially refilled); the gas bucket is empty. This request consumes 1 rate limit token and returns `GAS_PER_SECOND_LIMIT_EXCEEDED`.
3. Repeat step 2 until the rate limit bucket is also exhausted.
4. Send a legitimate request from a different client. It receives `REQUEST_PER_SECOND_LIMIT_EXCEEDED` even though no gas was consumed on its behalf.
5. Wait 1 second; both buckets refill. Repeat from step 1 to sustain the DoS.

**Observable result:** Legitimate users are continuously rejected with rate-limit errors while the attacker's requests never successfully execute any EVM computation.
