### Title
Contract Bytecode Cache Saturation via Unprivileged eth_call Requests Causing Sustained DB I/O Amplification

### Summary
The `cacheManagerContract()` bean in `EvmConfiguration.java` configures a Caffeine cache with `maximumSize=1000` and `expireAfterAccess=1h` for contract runtime bytecode. An unprivileged attacker can cycle `eth_call` requests across 1,001+ distinct contract addresses, continuously evicting cached entries and forcing a DB query for every request. The global rate limiter (500 req/s, not per-IP) does not prevent a single attacker from consuming the full request budget while keeping the cache in permanent churn, causing a sustained 30%+ increase in DB I/O and memory pressure from bytecode loading.

**Rate limiting is global, not per-IP:**

**Exploit flow:**

1. Attacker enumerates or generates 1,001+ valid contract IDs (Hedera has tens of thousands of deployed contracts; IDs are sequential and public).
2. Attacker sends `POST /api/v1/contracts/call` with minimum gas (21,000) targeting each address in rotation. At minimum gas, `scaleGas(21000) = 2` tokens consumed from the gas bucket (capacity ~750,000 scaled tokens/s), so the binding constraint is the 500 req/s request rate limit.
3. After the first 1,000 unique addresses fill the cache, every subsequent request to address N+1 evicts the LRU entry and forces `findRuntimeBytecode` to execute `SELECT runtime_bytecode FROM contract WHERE id = ?` against the DB.
4. With 1,001 addresses cycling at 500 req/s, the cache never warms: every single request is a DB hit, loading potentially large bytecode blobs into memory.
5. No authentication is required; `ContractController.call()` only calls `throttleManager.throttle(request)` which enforces the shared global bucket. [7](#0-6) 

### Impact Explanation

Under normal operation the contract cache absorbs repeated lookups for the same contracts (the vast majority of traffic targets a small set of popular contracts). With the cache saturated, every `eth_call` request generates at least one `SELECT runtime_bytecode` query. `runtime_bytecode` columns can be tens to hundreds of KB per contract. At 500 req/s sustained, this represents a dramatic increase in DB read I/O and JVM heap pressure from bytecode byte-array allocations compared to the near-zero DB load when the cache is warm. Legitimate users sharing the global 500 req/s bucket are simultaneously starved of capacity, compounding the denial-of-service effect.

### Likelihood Explanation

The attack requires no credentials, no special network position, and no on-chain transactions. Contract IDs on Hedera are sequential integers starting from low numbers; an attacker can trivially enumerate 1,001 valid contract addresses from the public mirror node REST API (`/api/v1/contracts`). The attack is fully scriptable with a simple HTTP client loop and is repeatable indefinitely because the cache TTL (`expireAfterAccess=1h`) resets on each access, meaning evicted entries never re-warm while the attacker is cycling. The gas throttle does not provide meaningful protection at minimum gas values.

### Recommendation

1. **Add per-IP (or per-connection) rate limiting** at the ingress layer (reverse proxy or Spring filter) so a single source cannot consume the full global request budget.
2. **Increase `maximumSize`** for the contract cache to a value that exceeds the realistic number of distinct contracts an attacker can cycle through within the throttle window, or switch to a size-weighted cache that accounts for bytecode size.
3. **Cache negative results explicitly** with a short TTL (the current `unless = "#result == null"` already caches `Optional.empty()`, but ensure this is intentional and the TTL is short enough to not mask legitimate contract deployments).
4. **Separate the contract bytecode cache from the request rate limit**: consider a dedicated DB connection pool limit or query-level rate limit for `findRuntimeBytecode` to bound the maximum DB query rate regardless of cache state.

### Proof of Concept

```python
import requests, itertools

# Enumerate 1001 contract IDs from the public mirror REST API
contracts = [f"0x{i:040x}" for i in range(1, 1002)]  # or fetch from /api/v1/contracts

url = "http://<mirror-node>/api/v1/contracts/call"
for addr in itertools.cycle(contracts):
    requests.post(url, json={
        "to": addr,
        "data": "0x",
        "gas": 21000,
        "block": "latest"
    })
    # No sleep: send at maximum rate to consume the 500 req/s global bucket
```

**Expected result:** DB query rate for `SELECT runtime_bytecode FROM contract WHERE id = ?` rises from near-zero (warm cache) to ~500 queries/second sustained, with corresponding increases in DB I/O wait, JVM heap allocation rate, and response latency for all users sharing the node.
