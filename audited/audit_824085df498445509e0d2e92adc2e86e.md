### Title
Unauthenticated DB Resource Exhaustion via `coalesce(size, length(bytes))` in Blocks Query

### Summary
The `blocksQuery` in `RecordFileService` unconditionally includes `coalesce(size, length(bytes)) as size` in its SELECT list. When the `size` column is NULL for a row, PostgreSQL must dereference and read the TOAST-stored `bytes` bytea column to compute its length. An unprivileged external user can repeatedly call `GET /api/v1/blocks?limit=100` to force the database to perform this I/O-intensive computation on up to 100 rows per request, with no authentication required.

### Finding Description

**Exact code location:**

`rest/service/recordFileService.js`, line 68 — the static `blocksQuery`: [1](#0-0) 

```sql
coalesce(size, length(bytes)) as size
```

This expression is used in two public-facing paths:
- `getBlocks()` (line 149–162) — called by `BlockController.getBlocks` for `GET /api/v1/blocks`
- `getByHashOrNumber()` (line 164–179) — called by `GET /api/v1/blocks/:hashOrNumber` [2](#0-1) 

**Root cause:**

PostgreSQL's `COALESCE` short-circuits left-to-right, so `length(bytes)` is only evaluated when `size IS NULL`. The `bytes` column is of type `bytea` (confirmed in the DB schema): [3](#0-2) 

The `size` column is nullable (`size int null`). The test spec explicitly documents the case where `bytes` has a value and `size` is null: [4](#0-3) 

When `bytes` is large (record files can be several MB), PostgreSQL must fetch and process the TOAST-stored binary data to compute `length(bytes)` — even though the raw `bytes` value is never returned to the caller.

**Limit enforcement flaw:**

`extractLimitFromFilters` in `blockController.js` caps the limit at `maxLimit` (default 100): [5](#0-4) 

However, there is a logic bug: when `limit.value > maxLimit`, it falls back to `defaultLimit` (25) instead of `maxLimit`. This means a request with `limit=100` (at the boundary) returns 100 rows, each potentially triggering `length(bytes)` on large TOAST data. No authentication is required to reach this endpoint: [6](#0-5) 

### Impact Explanation

For each row where `size IS NULL` and `bytes` is non-null and large, PostgreSQL must:
1. Dereference the TOAST pointer for the `bytes` column
2. Read (and potentially decompress) the stored binary data
3. Compute its length

With up to 100 rows per request and record files potentially being several MB each, a single request can cause hundreds of MB of TOAST I/O. The DB connection pool is limited to 10 connections with a 20-second statement timeout: [7](#0-6) 

An attacker sending concurrent requests can saturate the pool and exhaust DB I/O bandwidth, degrading availability for all users. The `bytes` column data is never returned to the caller — the I/O cost is entirely wasted overhead.

### Likelihood Explanation

- No authentication or special privileges required
- The endpoint is publicly documented in the OpenAPI spec
- The exploit requires only standard HTTP GET requests with `limit=100`
- Rows with `size=NULL` and large `bytes` exist in practice (the `size` column was added in a later migration; older records may lack it)
- The attack is trivially repeatable and scriptable

### Recommendation

1. **Eliminate the TOAST read**: Replace `coalesce(size, length(bytes))` with a query that only reads `size` and never touches `bytes` in the blocks listing query. If `size` is NULL, return NULL rather than computing `length(bytes)`.
2. **Backfill `size`**: Run a one-time migration to populate `size = length(bytes)` for all rows where `size IS NULL AND bytes IS NOT NULL`, then make `size` non-nullable.
3. **Exclude `bytes` from the SELECT list entirely**: The `bytes` column should never be selected in the blocks listing query since it is not part of the API response.

### Proof of Concept

```bash
# No authentication required
# Sends 50 concurrent requests, each forcing up to 100 length(bytes) evaluations
for i in $(seq 1 50); do
  curl -s "http://<mirror-node>/api/v1/blocks?limit=100" &
done
wait
```

Each request executes:
```sql
SELECT count, hash, name, prev_hash, ...,
       coalesce(size, length(bytes)) as size   -- reads TOAST for rows where size IS NULL
FROM record_file
ORDER BY consensus_end DESC
LIMIT 100;
```

With 50 concurrent requests × 100 rows × (e.g.) 5 MB per `bytes` value = up to 25 GB of TOAST I/O triggered by a single unauthenticated attacker, saturating the 10-connection DB pool within seconds.
