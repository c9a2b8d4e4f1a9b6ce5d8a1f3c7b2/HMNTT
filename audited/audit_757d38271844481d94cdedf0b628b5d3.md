### Title
Duplicate/Malformed sigPair Entries Inflate `numTxnSignatures`, Causing Fee Estimate to Exceed True On-Chain Fee

### Summary
The `TransactionFeeContext` constructor in `FeeEstimationService.java` derives `numTxnSignatures` by calling `.sigPair().size()` directly on the raw protobuf list, with no deduplication, no pubKeyPrefix uniqueness check, and no signature content validation. Because each sigPair entry linearly increases the computed fee, any unauthenticated caller can submit a crafted transaction containing arbitrarily many duplicate or zero-byte sigPair entries to obtain a fee estimate that is a multiple of the true on-chain fee.

### Finding Description
**Exact code path:**

`NetworkController.estimateFees()` (line 119–120) parses the raw request body and delegates to `FeeEstimationService.estimateFees()`. Inside that method a `TransactionFeeContext` is constructed (line 100). The constructor at lines 140–146 reads:

```java
if (transaction.signedTransactionBytes().length() > 0) {
    final var signedTransaction = SignedTransaction.PROTOBUF.parse(transaction.signedTransactionBytes());
    this.body = TransactionBody.PROTOBUF.parse(signedTransaction.bodyBytes());
    this.numTxnSignatures = signedTransaction
            .sigMapOrElse(SignatureMap.DEFAULT)
            .sigPair()
            .size();          // ← raw list size, no dedup
```

The identical pattern appears for the legacy `bodyBytes` path at lines 149–150.

`numTxnSignatures()` (line 172–174) is then consumed by `calculator.calculateTxFee(context.body(), context)` (line 105), which uses it as a linear multiplier in the fee schedule (confirmed by the test: 0 sigs → 1,000,000 tinycents; 2 sigs → 2,000,000 tinycents — each extra sigPair adds ~500,000 tinycents for a CryptoTransfer).

**Root cause / failed assumption:** The code assumes the caller supplies a well-formed sigMap whose entry count faithfully represents the number of distinct, valid signatures the network will verify. The Hedera network itself deduplicates sigPairs (confirmed by `scheduleSignDuplicateEd25519Signatures` test, lines 308–318, which shows the network stores only unique entries). The mirror-node estimation service performs no equivalent deduplication, so the estimate diverges upward from the true on-chain fee.

**Why existing checks are insufficient:**
- `@NotNull` on the request body only prevents a null body.
- `@Min(0) @Max(10000)` on `highVolumeThrottle` is unrelated.
- Spring Boot's default max request size (~1 MB) limits the absolute number of sigPairs but still allows thousands of duplicates within that budget.
- No authentication or rate-limiting is applied to `POST /api/v1/network/fees`.

### Impact Explanation
An attacker can make the estimation endpoint return a fee estimate that is arbitrarily larger than the fee the network will actually charge. Any client application that uses this estimate to set the `transactionFee` field in a real transaction will cause end-users to overpay. Applications that enforce a fee budget ceiling may reject valid transactions as "too expensive." The discrepancy is unbounded within the request-size limit (thousands of duplicate sigPairs → millions of tinycents of artificial inflation per request).

### Likelihood Explanation
No privileges, credentials, or on-chain state are required. The endpoint is publicly reachable. The attack requires only the ability to construct a protobuf `Transaction` with a repeated `sigPair` list — a trivial operation with any protobuf library. It is fully repeatable and stateless.

### Recommendation
Before computing `numTxnSignatures`, deduplicate the sigPair list by pubKeyPrefix (and optionally by signature type). A minimal fix:

```java
this.numTxnSignatures = (int) signedTransaction
        .sigMapOrElse(SignatureMap.DEFAULT)
        .sigPair()
        .stream()
        .map(sp -> sp.pubKeyPrefix())   // or a (prefix, type) pair
        .distinct()
        .count();
```

Additionally, enforce an upper bound on the number of accepted sigPairs (e.g., reject requests where `sigPair().size() > MAX_SIGNATURES`) to prevent resource exhaustion from oversized payloads.

### Proof of Concept
1. Build a valid `CryptoTransfer` `TransactionBody` and wrap it in a `SignedTransaction`.
2. Populate the `sigMap` with, e.g., 1,000 identical `SignaturePair` entries (same `pubKeyPrefix`, same zero-byte `ed25519` value).
3. Serialize to protobuf and POST to `POST /api/v1/network/fees` with `Content-Type: application/protobuf`.
4. Observe the returned `total` is ~1,000× the baseline fee (≈1,000,000,000 tinycents instead of ~1,000,000 tinycents for a zero-signature CryptoTransfer).
5. Submit the same transaction to the actual Hedera network; the charged fee reflects only the single unique signature, confirming the estimate is inflated. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
