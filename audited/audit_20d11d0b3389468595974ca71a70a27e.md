### Title
Unauthenticated Access to Full EVM Opcode Replay Exposes Sensitive Transaction Execution State

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` performs no authentication or authorization check before replaying any historical transaction and returning its complete EVM execution trace — including raw stack values, memory contents, and storage slots. Any anonymous external user who knows (or can enumerate) a transaction hash can retrieve the full internal execution state of that transaction, including any sensitive data that was present in EVM memory during processing.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, `getContractOpcodes()`, lines 52–68.

```java
@GetMapping(value = "/{transactionIdOrHash}/opcodes")
OpcodesResponse getContractOpcodes(
        @PathVariable TransactionIdOrHashParameter transactionIdOrHash,
        @RequestParam(required = false, defaultValue = "true") boolean stack,
        @RequestParam(required = false, defaultValue = "false") boolean memory,
        @RequestParam(required = false, defaultValue = "false") boolean storage,
        @RequestHeader(value = HttpHeaders.ACCEPT_ENCODING) String acceptEncoding) {
    if (properties.isEnabled()) {
        validateAcceptEncodingHeader(acceptEncoding);
        throttleManager.throttleOpcodeRequest();

        final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
        return opcodeService.processOpcodeCall(request);   // ← no auth, no ownership check
    }
    throw new ResponseStatusException(HttpStatus.NOT_FOUND);
}
```

**Root cause / failed assumption:**

The design assumes that enabling the feature flag (`hiero.mirror.web3.opcode.tracer.enabled=true`) is sufficient access control. There is no Spring Security filter, no `@PreAuthorize`, no API-key check, and no ownership/relationship check between the caller and the transaction being replayed. The only guards are:

1. **Feature flag** (`OpcodesProperties.enabled`, default `false`) — a deployment-time toggle, not a per-request access control. Once enabled for any legitimate use, it is open to everyone.
2. **gzip header** (`validateAcceptEncodingHeader`) — trivially satisfied with `Accept-Encoding: gzip`.
3. **Global rate limit** (`throttleOpcodeRequest`, default `opcodeRequestsPerSecond = 1`) — a single server-wide token bucket, not per-IP or per-user. An attacker simply waits one second between requests; distributed attackers are not throttled at all.

**Exploit flow:**

1. Operator enables the opcode tracer feature (`hiero.mirror.web3.opcode.tracer.enabled=true`) for debugging or tooling purposes.
2. Attacker observes (or enumerates from the public ledger) the hash of a transaction that called a healthcare contract.
3. Attacker issues:
   ```
   GET /api/v1/contracts/results/0x<txhash>/opcodes?stack=true&memory=true&storage=true
   Accept-Encoding: gzip
   ```
4. The mirror node re-executes the transaction on the EVM via `ContractDebugService.processOpcodeCall()` and returns the full `OpcodesResponse` containing every opcode, the EVM stack at each step, the full memory buffer (which holds ABI-decoded function arguments and intermediate values), and every storage slot read or written.
5. Patient data (e.g., record IDs, diagnosis codes, encrypted blobs passed as calldata) that was present in EVM memory during the original execution is now fully visible in the response.

### Impact Explanation

When `stack=true&memory=true&storage=true` is requested, the response contains the raw EVM memory buffer at every opcode step. ABI-encoded function arguments — including any sensitive fields passed to the contract — are present verbatim in memory. Storage slots reveal the contract's persistent state at the time of the transaction. This goes beyond what is visible in the raw transaction record: it exposes the full intermediate computation state. For a healthcare contract this means patient identifiers, record contents, or any other data passed as calldata or computed in-memory is fully readable by an unauthenticated third party. Severity is **High** when the feature is enabled.

### Likelihood Explanation

Transaction hashes are public on the Hedera ledger and trivially discoverable via the mirror node's own `/api/v1/contracts/results` endpoint. The only precondition is that the operator has enabled the opcode tracer, which is the intended production use case for debugging and tooling (as evidenced by the k6 load-test script at `tools/k6/src/web3/test/opcodes/contractResultsOpcodesAllPropertiesEnabled.js` and the e2e acceptance test in `EthereumFeature.java`). The attack requires no credentials, no special network position, and no prior knowledge beyond the transaction hash. The global rate limit of 1 req/sec is not a meaningful barrier — an attacker can exfiltrate data from thousands of transactions per hour simply by waiting between requests.

### Recommendation

1. **Add authentication/authorization** to the endpoint. At minimum, require an API key or operator-level credential via a Spring Security filter or `@PreAuthorize` annotation before serving opcode traces.
2. **Restrict by caller identity**: verify that the requester has a relationship to the transaction (e.g., is the payer account) before replaying it.
3. **Per-IP or per-authenticated-user rate limiting** rather than a single global bucket.
4. **Network-level restriction**: expose the `/opcodes` endpoint only on an internal/admin network interface, not on the public-facing API port.

### Proof of Concept

**Precondition:** `hiero.mirror.web3.opcode.tracer.enabled=true` in the web3 service configuration.

```bash
# Step 1: Discover a transaction hash for a contract call (public information)
TX_HASH=$(curl -s "https://<mirror-node>/api/v1/contracts/results?limit=1" \
  | jq -r '.results[0].hash')

# Step 2: Replay the transaction with full memory, stack, and storage tracing
# No credentials required
curl -s \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" \
  | gunzip \
  | jq '.opcodes[] | select(.memory != null) | .memory'
```
