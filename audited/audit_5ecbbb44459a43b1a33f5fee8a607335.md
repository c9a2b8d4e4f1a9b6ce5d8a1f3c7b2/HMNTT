### Title
Unauthenticated EVM Address Resolution Forces Unconditional Extra DB Round-Trip in `getContractResultsById`

### Summary
Any unauthenticated caller can supply a valid 40-hex-character EVM address as the `:contractId` path parameter to `GET /contracts/:contractId/results`. This causes `ContractService.computeContractIdFromString()` to unconditionally execute an additional database query (`getContractIdByEvmAddress`) before the main contract-results query. No caching, no rate limiting, and no authentication guard exist on this code path, making it trivially repeatable by any external actor.

### Finding Description
**Route registration** (`rest/routes/contractRoute.js:14`):
```
router.getExt('/:contractId/results', ContractController.getContractResultsById);
```
No authentication middleware is applied.

**Handler** (`rest/controllers/contractController.js:856-862`):
```js
getContractResultsById = async (req, res) => {
  const {contractId: contractIdParam, filters} = extractContractIdAndFiltersFromValidatedRequest(req, ...);
  const contractId = await ContractService.computeContractIdFromString(contractIdParam);
  ...
```
`computeContractIdFromString` is called unconditionally on every request.

**Resolution logic** (`rest/service/contractService.js:468-476`):
```js
async computeContractIdFromString(contractIdValue) {
  const contractIdParts = EntityId.computeContractIdPartsFromContractIdValue(contractIdValue);
  if (contractIdParts.hasOwnProperty('create2_evm_address')) {
    return this.getContractIdByEvmAddress(contractIdParts);   // ← DB query #1
  }
  return EntityId.parseString(contractIdValue, ...).getEncodedId(); // ← no DB
}
```

**Branch condition** (`rest/entityId.js:284-298`): `computeContractIdPartsFromContractIdValue` calls `isEvmAddressAlias(evmAddressOrNum)`. Any 40-hex-character string (with or without `0x` prefix) satisfies `evmAddressRegex` (`/^(0x)?[A-Fa-f0-9]{40}$/`, line 40) and causes `create2_evm_address` to be set, routing execution into the DB branch.

**DB lookup** (`rest/service/contractService.js:452-466`):
```js
async getContractIdByEvmAddress(evmAddressFilter) {
  const rows = await super.getRows(ContractService.contractIdByEvmAddressQuery, [
    Buffer.from(create2EvmAddress, 'hex'),
  ]);
  ...
}
```
This executes `contractIdByEvmAddressQuery` against the `entity` table. There is **no result cache** in `computeContractIdFromString` or `getContractIdByEvmAddress`. The `quickLru` cache in `entityId.js` (line 301) is only used for `EntityId.parseString`, not for this EVM-address DB path.

**Root cause**: The code correctly distinguishes numeric IDs (no DB) from EVM addresses (DB lookup), but provides no caching layer and no rate limiting for the EVM-address branch, making the extra DB round-trip freely triggerable by any caller.

### Impact Explanation
Each request with an EVM address causes two sequential DB queries instead of one. An attacker sending a high volume of requests with distinct valid EVM addresses (e.g., `0x0000000000000000000000000000000000000001` through `0xffffffffffffffffffffffffffffffffffffffff`) forces the database to process a lookup against the `entity` table on every single request. This increases DB CPU, I/O, and connection pool pressure proportionally to request rate. Under sustained load this degrades response latency for all users of the mirror node REST API. No economic damage to on-chain users occurs, but service availability and quality are impacted.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and zero on-chain interaction. Any HTTP client can issue `GET /contracts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/results` in a tight loop. The 40-hex address space is enormous, preventing any practical cache from absorbing the load. The attack is trivially scriptable and repeatable from a single machine or botnet.

### Recommendation
1. **Add a result cache** in `computeContractIdFromString` (or inside `getContractIdByEvmAddress`) keyed on the EVM address string, with a short TTL (e.g., 30–60 s), so repeated lookups for the same address hit memory rather than the DB.
2. **Apply rate limiting** at the HTTP layer (e.g., express-rate-limit or an upstream API gateway) on all `/contracts/:contractId/*` endpoints, keyed by source IP.
3. **Validate EVM addresses against a long-zero prefix** before issuing the DB query: addresses of the form `0x000000000000000000000000<12-byte-num>` can be resolved purely in-memory (as the Java `DomainUtils.fromEvmAddress` already does), avoiding the DB entirely for that common case.

### Proof of Concept
```bash
# Single extra DB round-trip (observable via DB query logs)
curl "https://<mirror-node>/api/v1/contracts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/results"

# Sustained griefing loop with rotating addresses (no auth required)
for i in $(seq 1 10000); do
  ADDR=$(printf '%040x' $i)
  curl -s "https://<mirror-node>/api/v1/contracts/0x${ADDR}/results" &
done
wait
```
Each iteration triggers `getContractIdByEvmAddress` → one extra `SELECT` against the `entity` table → DB load scales linearly with request rate.
