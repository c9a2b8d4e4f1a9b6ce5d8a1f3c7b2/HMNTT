### Title
Throttle Function Blind to Historical Block Requests Enables Disproportionate DB I/O Amplification

### Summary
`ThrottleManagerImpl.throttle()` enforces only a global requests-per-second and gas-per-second limit, with no awareness of whether a `ContractCallRequest` targets a historical block. Historical block requests trigger significantly more expensive database operations (UNION queries across `entity_history`, `contract_state_change`, `account_balance` tables) than latest-block requests, yet consume identical throttle tokens. An unprivileged attacker can saturate the shared rate limit with minimum-gas historical requests, causing disproportionate DB I/O that degrades response quality for all other users.

### Finding Description
**Code path:**

`ContractController.call()` â†’ `throttleManager.throttle(request)` â†’ `ThrottleManagerImpl.throttle()` (lines 37â€“48):

```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
        throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
    }
    for (var requestFilter : throttleProperties.getRequest()) {
        if (requestFilter.test(request)) { action(requestFilter, request); }
    }
}
```

The function inspects only `request.getGas()` and a request count. It never reads `request.getBlock()`. After throttle passes, `ContractController` calls `contractExecutionService.processCall(params)`, which eventually calls `ContractCallService.callContract()`:

```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)));
```

For any non-`LATEST` block, `RecordFileServiceImpl.findByBlockType()` calls `recordFileRepository.findByIndex(block.number())`, and the subsequent EVM execution queries historical tables. For example, `EntityRepository.findActiveByIdAndTimestamp()` issues a UNION across `entity` and `entity_history` with a timestamp filter. Similar historical queries exist for `contract_state_change`, `account_balance`, token tables, etc.

**Root cause:** The throttle's failed assumption is that all requests within the gas/rate budget impose equal server cost. Historical requests impose additional, unbounded-by-gas DB I/O (historical table scans with timestamp predicates) that the throttle does not account for.

**Why existing checks fail:**
- `rateLimitBucket` (default 500 req/sec) is global and shared; it does not weight historical vs. latest requests differently.
- `gasLimitBucket` bounds EVM computation, but historical state loading (DB queries) happens outside the EVM gas model â€” `scaleGas(21_000)` = 2 tokens consumed regardless of how many historical DB rows are scanned.
- The optional `RequestProperties` filter system (`FilterField.BLOCK` exists) can be configured to reject/throttle by block field, but **no such filter is active by default** â€” `throttleProperties.getRequest()` returns an empty list by default.
- The `restore()` call in `ContractController` only fires on `InvalidParametersException`, not on `BlockNumberNotFoundException` or successful historical execution, so rate-limit tokens are never returned.

### Impact Explanation
An attacker sending 500 req/sec (the default rate limit) with `block=0` (`earliest`) and `gas=21000` (minimum) will:
1. Consume the entire shared `rateLimitBucket`, starving all other users of the 500 req/sec budget.
2. Each request triggers historical DB queries (UNION across `entity_history`, `account_balance`, `contract_state_change`) that are more expensive than latest-block queries, amplifying DB I/O beyond what the throttle was calibrated for.
3. The gas bucket is largely unaffected (`scaleGas(21000)` = 2 tokens out of 750,000 capacity), so the gas throttle provides no meaningful protection.

The result is degraded availability of the historical record endpoint for all legitimate users, and potential DB saturation on a production mirror node with years of history.

### Likelihood Explanation
No authentication or API key is required â€” the `/api/v1/contracts/call` endpoint is publicly accessible. The attack requires only an HTTP client capable of sustaining 500 POST requests/second, which is trivially achievable from a single machine. The `block` field accepts any non-negative integer (`BlockType.of()` parses decimal and hex values), so targeting block 0 or any early block requires no special knowledge. The attack is repeatable and persistent as long as the attacker maintains the request rate.

### Recommendation
1. **Add a historical-request-specific sub-limit**: Introduce a separate `historicalRequestsPerSecond` bucket in `ThrottleProperties` and check it in `throttle()` when `request.getBlock() != BlockType.LATEST`.
2. **Enable default BLOCK-field request filtering**: Configure a default `RequestProperties` entry with `field=BLOCK, action=THROTTLE` and a conservative rate for non-latest blocks.
3. **Weight gas tokens by block type**: Apply a configurable multiplier to `scaleGas()` for historical requests to reflect their higher DB cost.
4. **Restore rate-limit tokens on fast-fail paths**: When `BlockNumberNotFoundException` is thrown (non-existent block), restore the rate-limit token to prevent trivial exhaustion via invalid block numbers.
