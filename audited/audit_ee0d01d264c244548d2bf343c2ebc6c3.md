### Title
Unbounded Alias Count in `resolveAccountAliases()` Causes DB Connection Pool Exhaustion via Single Unauthenticated Request

### Summary
`ConstructionMetadata()` passes the caller-controlled `account_aliases` option string directly to `resolveAccountAliases()`, which splits it on commas and issues one synchronous database query per alias with no upper-bound check. An unauthenticated attacker can craft a single `/construction/metadata` request whose `account_aliases` field contains thousands of valid alias strings, exhausting the database connection pool and blocking all other mirror-node operations that depend on the database, including transaction confirmation.

### Finding Description

**Exact code path:**

`ConstructionMetadata()` at [1](#0-0)  calls `resolveAccountAliases()` unconditionally after only verifying that `operation_type` is a valid string.

Inside `resolveAccountAliases()`:

```go
// construction_service.go lines 575-586
for accountAlias := range strings.SplitSeq(accountAliases, ",") {
    accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
    if err != nil {
        return emptyResult, errors.ErrInvalidAccount
    }

    found, rErr := c.accountRepo.GetAccountId(ctx, accountId)   // ← one DB query per alias
    if rErr != nil {
        return emptyResult, rErr
    }
    accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
}
``` [2](#0-1) 

There is **no limit** on the number of comma-separated aliases. Each iteration that passes `NewAccountIdFromString` and has `HasAlias() == true` reaches `GetAccountId()`:

```go
// account.go lines 146-158
db, cancel := ar.dbClient.GetDbWithContext(ctx)
defer cancel()

var entity domain.Entity
if err := db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity).Error; err != nil {
``` [3](#0-2) 

Each call acquires a connection from the shared GORM/database connection pool and executes `selectCurrentCryptoEntityByAlias` against the `entity` table. [4](#0-3) 

**Why existing checks are insufficient:**

- The only pre-check in `ConstructionMetadata()` is that `operation_type` is a non-nil string and resolves to a known fee type. [5](#0-4) 
- There is no HTTP body-size limit in the rosetta server (`MaxBytesReader` / `ReadLimit` search returns zero matches in `rosetta/**/*.go`).
- The Traefik middleware (`inFlightReq: amount: 5`, `rateLimit: average: 10`) is **optional** — it is only rendered when `global.middleware: true`. [6](#0-5)  Even when enabled, `inFlightReq` limits concurrent requests per IP but does nothing to bound the alias count inside a single request; `rateLimit` is keyed on `requestHost` (the server's hostname), not per-client IP, so it limits total throughput, not per-attacker abuse.
- `GetAccountId()` short-circuits only for non-alias (numeric) account IDs. [7](#0-6)  Alias-format IDs always hit the database.

### Impact Explanation

Exhausting the database connection pool prevents every other service component that shares the same pool (block queries, transaction queries, balance queries) from obtaining a connection. This maps directly to the stated critical impact: **total inability to confirm new transactions** for the duration of the attack. Because the queries are sequential within a single goroutine and the loop is unbounded, even a single attacker request with ~10 000 aliases can hold the pool saturated for seconds to minutes depending on pool size and query latency.

### Likelihood Explanation

- **No authentication required.** The `/construction/metadata` endpoint is a public Rosetta API endpoint. [8](#0-7) 
- **No special knowledge required.** The attacker only needs to supply a valid `operation_type` (e.g., `"CRYPTO_TRANSFER"`) and a comma-separated list of syntactically valid alias strings (e.g., thousands of generated Ed25519 public-key hex strings).
- **Repeatable.** The attack can be re-sent immediately after each request completes, maintaining continuous pressure.
- **Low cost.** A single HTTP request is sufficient to trigger thousands of DB queries.

### Recommendation

1. **Enforce an alias count limit** inside `resolveAccountAliases()` before the loop — reject requests where the number of comma-separated aliases exceeds a small constant (e.g., 10, matching the maximum number of signers a Hedera transaction can have).
2. **Batch the lookup** into a single `WHERE alias = ANY(...)` query instead of N individual queries.
3. **Add an application-level HTTP body-size cap** (e.g., `http.MaxBytesReader`) in the rosetta server setup so oversized payloads are rejected before parsing.
4. **Make the Traefik middleware mandatory** (not opt-in) for production deployments, and key `rateLimit` on `ipStrategy` rather than `requestHost`.

### Proof of Concept

```bash
# 1. Generate N valid Ed25519 alias strings (hex-encoded 32-byte public keys prefixed
#    with the protobuf key encoding used by Hedera, e.g. 302a300506032b6570032100<32 bytes>)
# 2. Build the aliases string
ALIASES=$(python3 -c "
import os, binascii
aliases = []
for _ in range(10000):
    pub = os.urandom(32)
    # DER-encoded Ed25519 SubjectPublicKeyInfo prefix + raw key
    der = bytes.fromhex('302a300506032b6570032100') + pub
    aliases.append('0.0.' + binascii.hexlify(der).decode())
print(','.join(aliases))
")

# 3. Send a single unauthenticated POST to /construction/metadata
curl -s -X POST http://<rosetta-host>/construction/metadata \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
    \"options\": {
      \"operation_type\": \"CRYPTO_TRANSFER\",
      \"account_aliases\": \"$ALIASES\"
    }
  }"

# Result: resolveAccountAliases() issues 10 000 sequential DB queries,
# exhausting the connection pool and causing all concurrent DB-dependent
# operations to time out or fail for the duration of the request.
```
