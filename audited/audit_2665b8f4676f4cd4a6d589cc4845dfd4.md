### Title
Unvalidated Client-Supplied `node_account_id` in `/construction/payloads` Bypasses Network Membership Check, Causing Unretried Submission Failure

### Summary
`getTransactionNodeAccountId` in `construction_service.go` accepts any syntactically valid `hiero.AccountID` string from client-supplied metadata without verifying it exists in the current SDK network map. Because `sdkClient.SetMaxAttempts(1)` explicitly disables all SDK-level retries, a transaction frozen with a removed or non-existent node account ID will fail at submission with no fallback. Any unprivileged caller of the public Rosetta Construction API can trigger this condition.

### Finding Description
**Code path:**

`rosetta/app/services/construction_service.go`

- `ConstructionMetadata` (line 166) correctly calls `getRandomNodeAccountId()`, which iterates `c.sdkClient.GetNetwork()` (line 517) — the live, SDK-maintained network map — to pick a valid node.
- `ConstructionPayloads` (line 222) calls `getTransactionNodeAccountId(request.Metadata)` (lines 370–391), which reads the `node_account_id` key from the **client-supplied** metadata and only validates it with `hiero.AccountIDFromString(str)` (line 383) — a pure syntactic parse. There is **no membership check** against `c.sdkClient.GetNetwork()`.
- The parsed account ID is embedded in the frozen transaction via `transactionSetNodeAccountId` (line 251 / lines 717–725).
- `ConstructionSubmit` (lines 331–368) calls `hiero.TransactionExecute(transaction, c.sdkClient)` (line 355). The SDK routes gossip to the node account ID baked into the transaction.
- `sdkClient.SetMaxAttempts(1)` (line 636) disables all SDK retry logic.

**Root cause:** The server assumes the client will echo back the `node_account_id` it received from `/construction/metadata`, but there is no binding or re-validation of that value against the live network. The two code paths that select a node (`getRandomNodeAccountId` — network-aware) and that accept a node (`getTransactionNodeAccountId` — format-only) are asymmetric.

**Failed assumption:** The design assumes the `node_account_id` in `/construction/payloads` metadata is always one previously returned by `/construction/metadata`. Nothing enforces this.

### Impact Explanation
An attacker submits `/construction/payloads` with `node_account_id` set to a syntactically valid but removed/non-existent node account ID (e.g., `0.0.999`). The server freezes the transaction with that ID and returns it to the attacker. When `/construction/submit` is called, the SDK attempts to gossip to the non-existent node, fails, and — because `SetMaxAttempts(1)` is set — returns an error with no retry to a healthy node. The transaction is never submitted. This allows reliable, repeatable forced submission failures for any transaction the attacker constructs. At scale, repeated connection attempts to non-existent endpoints also consume server-side socket/timeout resources.

### Likelihood Explanation
No authentication or privilege is required. The Rosetta Construction API is a public HTTP interface. The attacker only needs to know any syntactically valid `shard.realm.num` string (e.g., `0.0.999`). Historical node account IDs are publicly documented. The attack is trivially repeatable and requires no cryptographic material beyond what is needed to construct a normal transaction.

### Recommendation
In `getTransactionNodeAccountId`, after parsing the account ID, validate it against the live network:

```go
func (c *constructionAPIService) getTransactionNodeAccountId(metadata map[string]any) (
    emptyAccountId hiero.AccountID, nilErr *rTypes.Error,
) {
    // ... existing parse logic ...

    // Validate the node account ID is in the current network
    network := c.sdkClient.GetNetwork()
    found := false
    for _, id := range network {
        if id == nodeAccountId {
            found = true
            break
        }
    }
    if !found {
        log.Errorf("Node account id %s is not in the current network", str)
        return emptyAccountId, errors.ErrInvalidAccount
    }

    return nodeAccountId, nilErr
}
```

This mirrors the logic already used in `getRandomNodeAccountId` and closes the asymmetry between the two paths.

### Proof of Concept
1. Start the Rosetta mirror node in online mode.
2. Call `POST /construction/preprocess` and `POST /construction/metadata` normally to obtain valid metadata.
3. Call `POST /construction/payloads` with the metadata, but **replace** `node_account_id` with a syntactically valid but non-existent value, e.g. `"0.0.9999"`:
   ```json
   {
     "metadata": {
       "node_account_id": "0.0.9999",
       "valid_until_nanos": "<value from metadata response>"
     },
     "operations": [ ... ]
   }
   ```
4. The server returns a signed unsigned transaction with `0.0.9999` baked in as the node.
5. Sign the payload and call `POST /construction/combine`, then `POST /construction/submit`.
6. Observe: the SDK attempts to gossip to node `0.0.9999`, fails to find it in the network, and returns an error. With `SetMaxAttempts(1)`, no retry occurs and the submission fails permanently. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

func (c *constructionAPIService) getTransactionNodeAccountId(metadata map[string]any) (
	emptyAccountId hiero.AccountID, nilErr *rTypes.Error,
) {
	value, ok := metadata[metadataKeyNodeAccountId]
	if !ok {
		return emptyAccountId, errors.ErrMissingNodeAccountIdMetadata
	}

	str, ok := value.(string)
	if !ok {
		return emptyAccountId, errors.ErrInvalidArgument
	}

	nodeAccountId, err := hiero.AccountIDFromString(str)
	if err != nil {
		log.Errorf("Invalid node account id provided in metadata: %s", str)
		return emptyAccountId, errors.ErrInvalidAccount
	}

	log.Infof("Use node account id %s from metadata", str)
	return nodeAccountId, nilErr
}

func (c *constructionAPIService) getRandomNodeAccountId() (hiero.AccountID, *rTypes.Error) {
	nodeAccountIds := make([]hiero.AccountID, 0)
	seen := map[hiero.AccountID]struct{}{}
	// network returned from sdkClient is a map[string]AccountID, the key is the address of a node and the value is
	// its node account id. Since a node can have multiple addresses, we need the seen map to get a unique node account
	// id array
	for _, nodeAccountId := range c.sdkClient.GetNetwork() {
		if _, ok := seen[nodeAccountId]; ok {
			continue
		}

		seen[nodeAccountId] = struct{}{}
		nodeAccountIds = append(nodeAccountIds, nodeAccountId)
	}

	if len(nodeAccountIds) == 0 {
		return hiero.AccountID{}, errors.ErrNodeAccountIdsEmpty
	}

	maxValue := big.NewInt(int64(len(nodeAccountIds)))
	index, err := rand.Int(rand.Reader, maxValue)
	if err != nil {
		log.Errorf("Failed to get a random number, use 0 instead: %s", err)
		return nodeAccountIds[0], nil
	}

	return nodeAccountIds[index.Int64()], nil
