### Title
Unauthenticated Repeated Block Requests Trigger Unbounded Correlated `json_agg` Subqueries Causing DB Resource Exhaustion

### Summary
The `selectTransactionsInTimestampRange` SQL constant in `rosetta/app/persistence/transaction.go` uses two correlated subqueries per transaction row, each executing `json_agg(...ORDER BY ...)` against `crypto_transfer` and `staking_reward_transfer`. The Rosetta `/block` endpoint is unauthenticated, has no rate limiting, and no result caching. An unprivileged attacker can repeatedly request a block with many transactions to force the database to execute thousands of sort-bearing aggregation subqueries per request, with no per-request guard preventing this.

### Finding Description

**Exact code path:**

`rosetta/app/persistence/transaction.go`, lines 49–55 — the second correlated subquery:

```sql
coalesce((
  select json_agg(json_build_object(
    'account_id', account_id,
    'amount', amount) order by account_id)
  from staking_reward_transfer
  where consensus_timestamp = t.consensus_timestamp
), '[]') as staking_reward_payouts
``` [1](#0-0) 

This subquery executes once per row returned by the outer `transaction` query. The outer query is batched at `batchSize = 2000` per iteration: [2](#0-1) [3](#0-2) 

The loop continues fetching batches until all transactions in the block's timestamp range are retrieved — there is no cap on total transactions fetched. For each batch of up to 2000 rows, the database executes 2000 correlated `json_agg(... ORDER BY account_id)` subqueries against `staking_reward_transfer`.

**Root cause — failed assumption:** The design assumes the number of transactions per block and the number of `staking_reward_transfer` rows per transaction are small enough that correlated subqueries are acceptable. There is no per-request limit on total transactions fetched, no result cache, and no rate limit.

**`maxTransactionsInBlock` does not protect the DB:** In `block_service.go` lines 61–66, the truncation to `maxTransactionsInBlock` happens *after* `FindBetween` has already fetched all transactions from the database: [4](#0-3) 

The full DB cost is paid regardless of how many transactions are ultimately returned to the client.

**No rate limiting or block-result caching:** The `blockAPIService` only caches entity/account aliases (LRU `entityCache`), not block query results: [5](#0-4) 

The `Config` struct exposes only `statementTimeout` at the DB level and `maxTransactionsInBlock` at the response level — no HTTP rate limiting or query result caching: [6](#0-5) [7](#0-6) 

### Impact Explanation
An attacker who identifies a block containing many transactions (e.g., a staking-reward-distribution block, which can contain thousands of transactions each with multiple `staking_reward_transfer` rows) can repeatedly POST to the unauthenticated Rosetta `/block` endpoint. Each request forces the database to execute O(N × M) `json_agg(ORDER BY)` sort operations, where N is the number of transactions in the block and M is the average number of staking reward transfer rows per transaction. With N=2000 and M=10, a single request triggers 20,000 sort-bearing aggregations. Sustained parallel requests can saturate database CPU and I/O, increasing processing node resource consumption well above 30% compared to baseline, degrading or denying service to legitimate users.

### Likelihood Explanation
The Rosetta API requires no authentication or API key. The attacker needs only to identify a historically expensive block (trivially done by scanning block transaction counts via the public `/block` endpoint) and issue repeated concurrent requests. No special privileges, tokens, or network access beyond HTTP connectivity are required. The attack is fully repeatable and automatable with a simple script.

### Recommendation
1. **Add result caching for block responses**: Cache the result of `FindBetween` keyed on `(consensusStart, consensusEnd)` with a short TTL (e.g., 30 seconds). Finalized blocks are immutable, so caching is safe.
2. **Apply `LIMIT` inside the correlated subqueries** if the number of staking reward transfers per transaction is bounded by protocol, or rewrite as a single `LEFT JOIN ... GROUP BY` to avoid per-row subquery execution.
3. **Enforce HTTP-level rate limiting** on the `/block` and `/block/transaction` endpoints (e.g., per-IP request rate cap).
4. **Cap total transactions fetched** in `FindBetween` to `maxTransactionsInBlock` before executing the DB query, not after, so the database never processes more rows than will be returned.

### Proof of Concept
1. Identify a block with many transactions (e.g., a staking reward period block) by querying `/block` with known block indices.
2. Note the block identifier (index + hash).
3. Run the following in parallel (e.g., 20 concurrent workers):
```
POST /block
{
  "network_identifier": { ... },
  "block_identifier": { "index": <TARGET_BLOCK_INDEX>, "hash": "<TARGET_BLOCK_HASH>" }
}
```
4. Monitor PostgreSQL `pg_stat_activity` and CPU metrics. Each request triggers up to `batchSize × 2` correlated `json_agg(ORDER BY)` subqueries. With 20 concurrent requests against a block with 2000 transactions and 10 staking reward transfers each, the database executes 400,000 sort operations per second, measurably increasing CPU utilization beyond the 30% threshold relative to idle baseline.
