### Title
Unbounded Timestamp Range in Historical Hook Storage Query Causes Full Table Scan via DISTINCT ON

### Summary
An unprivileged external user can supply a single one-sided timestamp parameter (e.g., `timestamp=gte:0`) to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, causing `HookServiceImpl.getHookStorageChange()` to invoke `hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween()` with a timestamp range of `[0, Long.MAX_VALUE]`. The native SQL query uses `SELECT DISTINCT ON (key)` which forces PostgreSQL to scan every matching row in the `hook_storage_change` table before applying the LIMIT, making the pagination limit of 25 ineffective as a scan-scope guard. Repeated requests can drive DB I/O and CPU well above 30%.

### Finding Description

**Code path:**

`HookServiceImpl.getHookStorage()` (line 57–58) delegates to `getHookStorageChange()` whenever `request.isHistorical()` is true — which is true whenever any timestamp parameter is present. [1](#0-0) 

Inside `getHookStorageChange()`, the timestamp bounds are derived from `Bound`: [2](#0-1) 

`Bound.getAdjustedLowerRangeValue()` returns `0` when no lower bound is set, and `Bound.adjustUpperBound()` returns `Long.MAX_VALUE` when no upper bound is set: [3](#0-2) [4](#0-3) 

No maximum range width validation exists anywhere in `Bound` for non-primary-sort fields (timestamp is passed with `primarySortField = false`): [5](#0-4) 

The resulting native query uses `SELECT DISTINCT ON (key)` with `consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound`: [6](#0-5) 

**Root cause:** PostgreSQL's `DISTINCT ON (key)` requires a full scan of all rows matching the WHERE clause (owner_id, hook_id, key range, timestamp range) to identify the latest `consensus_timestamp` per key before the LIMIT is applied. The pagination `Pageable` (default limit=25) only restricts the number of rows *returned*, not the number of rows *scanned*. With `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`, the DB must read the entire history for that hook. [7](#0-6) 

### Impact Explanation
Each crafted request forces a full sequential scan of `hook_storage_change` rows for the targeted `owner_id`/`hook_id`. For hooks with large storage histories (common in active DeFi-style contracts), a single request can generate significant I/O. Sending tens of such requests per second against different or the same hook IDs can sustain DB CPU and I/O above 30% of baseline, degrading query performance for all mirror node users. No data is exfiltrated, but availability is impacted.

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge beyond a valid `ownerId`/`hookId` pair (both discoverable via public API). The trigger is a single HTTP GET with one query parameter (`timestamp=gte:0`). It is trivially scriptable and repeatable. The only partial mitigation is the rate limiter referenced in `HooksController`, but a single request already causes a large scan, and rate limits on REST endpoints are typically permissive enough (e.g., 100 req/s) to sustain the attack. [8](#0-7) 

### Recommendation
1. **Enforce a maximum timestamp range width**: In `HookStorageRequest` or the controller, reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configured maximum (e.g., 7 days in nanoseconds).
2. **Require both bounds**: Validate that when `isHistorical()` is true, both a lower and upper timestamp bound must be explicitly provided.
3. **Rewrite the query**: Replace `DISTINCT ON` with a keyset-paginated query or a lateral join that uses an index seek per key, so the DB does not need to scan the full timestamp range.
4. **Add a DB-level query timeout** for this endpoint to cap worst-case scan duration.

### Proof of Concept

**Preconditions:** Any valid `ownerId` and `hookId` (obtainable from `GET /api/v1/accounts/{ownerId}/hooks`).

**Trigger:**
```
# Single one-sided timestamp — lower defaults to 0, upper defaults to Long.MAX_VALUE
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0

# Or equivalently, upper-only:
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=lte:9999999999999999999
```

**Exploit loop (bash):**
```bash
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0" &
  sleep 0.1
done
```

**Result:** PostgreSQL executes `SELECT DISTINCT ON (key) ... FROM hook_storage_change WHERE owner_id=1234 AND hook_id=1 AND consensus_timestamp BETWEEN 0 AND 9223372036854775807` — a full historical scan — for every request. DB CPU and I/O rise proportionally to the number of concurrent requests and the size of the `hook_storage_change` table for that hook.

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
