### Title
Cache Key Collision via 32-bit `Arrays.hashCode` in `findByEvmAddressAndDeletedIsFalse` Allows Wrong Entity Retrieval

### Summary
`findByEvmAddressAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` — which resolves to `Arrays.hashCode(byte[])`, a 32-bit signed integer — as the sole cache key. Because the cache stores entries keyed by this `Integer` value rather than the actual byte array, any two distinct byte arrays that produce the same `Arrays.hashCode` value will collide in the cache, causing the wrong entity to be returned. An unprivileged attacker who can control the EVM address argument (e.g., via a CALL opcode in a contract call) can precompute a colliding address offline and trigger retrieval of a different entity's data from cache.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–37:
```java
@Cacheable(
    cacheNames = CACHE_NAME_EVM_ADDRESS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```

`common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```

**Root cause:** `Arrays.hashCode(byte[])` computes a 32-bit signed integer using:
```
result = 1; for each byte b: result = 31 * result + b
```
The SpEL expression evaluates to this `int`, which is autoboxed to `Integer` and used as the Caffeine cache key. The cache therefore stores and retrieves entries keyed by `Integer`, not by the actual byte array. Two distinct byte arrays `A` and `B` where `Arrays.hashCode(A) == Arrays.hashCode(B)` will share the same cache slot. Whichever is cached first will be returned for both.

**Failed assumption:** The code assumes `Arrays.hashCode` is collision-resistant enough to serve as a unique cache key. It is not — its 32-bit output space guarantees collisions are findable with modest computation.

**Why existing checks are insufficient:**
- `unless = "#result == null"` only prevents caching of Java `null`; `Optional.of(entity)` and `Optional.empty()` are both non-null and are both cached, so a cached entity result under key `k` will be returned for any colliding address.
- `ConversionUtils.isLongZeroAddress` in `CommonEntityAccessor` only redirects Hedera long-zero addresses (first 12 bytes zero); it does not prevent arbitrary 20-byte EVM addresses from reaching `findByEvmAddressAndDeletedIsFalse`.
- The same collision pattern exists identically in `findByEvmAddressOrAliasAndDeletedIsFalse` (lines 39–49), doubling the attack surface.

**Exploit flow:**

1. Attacker identifies target entity with EVM address `T` (e.g., a high-value contract or account).
2. Attacker precomputes offline a 20-byte address `C ≠ T` such that `Arrays.hashCode(C) == Arrays.hashCode(T)`. With a 32-bit output space this is feasible via birthday attack or algebraic inversion of the linear recurrence.
3. Attacker (or any user) triggers a legitimate lookup of `T` (e.g., a `eth_call` to address `T`), causing `Optional.of(entityT)` to be stored in the `evmAddress` cache under key `Arrays.hashCode(T)`.
4. Within the cache TTL window (`expireAfterWrite=1s`), attacker submits an EVM call whose execution causes a lookup of address `C` (e.g., via a CALL/STATICCALL opcode to `C`, or by passing `C` as an account alias).
5. `findByEvmAddressAndDeletedIsFalse(C)` computes `Arrays.hashCode(C)` == `Arrays.hashCode(T)`, hits the cache, and returns `entityT` — the wrong entity.
6. The EVM execution context now treats address `C` as if it were entity `T`: balance, code, storage, and account metadata of `T` are used for `C`.

### Impact Explanation
The wrong entity's data (balance, bytecode, account type, keys) is returned for a crafted address. In EVM execution this can: (a) make a non-existent or low-value address appear to have the balance/code of a high-value entity, enabling incorrect balance reads or impersonation within a single EVM frame; (b) cause fund-transfer logic that resolves recipient addresses through this path to credit or debit the wrong on-chain entity. The `CACHE_NAME_EVM_ADDRESS` cache is shared across all concurrent requests, so a poisoned entry affects every caller until expiry.

### Likelihood Explanation
The attacker requires no privileges — only the ability to submit contract calls (a standard, permissionless operation on any EVM-compatible network). Collision precomputation against `Arrays.hashCode` is offline work requiring no special access; the 32-bit output space means expected collision cost is ~2^16 trials (birthday bound). The 1-second TTL narrows the timing window but does not eliminate it: the attacker can automate repeated attempts, and the window resets every time the target entity is re-cached by normal traffic. The attack is repeatable and scriptable.

### Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with the actual byte array wrapped in a collision-resistant key type:

```java
// Option 1: use the hex-encoded string as key (unambiguous, no collision)
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"

// Option 2: wrap in ByteBuffer (correct equals/hashCode based on content)
key = "T(java.nio.ByteBuffer).wrap(#alias)"
```

`ByteBuffer.wrap(byte[])` implements `equals` and `hashCode` based on array content, so the Caffeine cache will correctly distinguish distinct byte arrays even if their `Arrays.hashCode` values collide. Alternatively, store the hex string. Apply the same fix to `findByEvmAddressOrAliasAndDeletedIsFalse` (line 42).

### Proof of Concept

```java
// Demonstrate collision
byte[] address1 = new byte[20];
address1[0] = 1; // some legitimate entity address

// Find collision: solve 31^20 * 1 + ... ≡ Arrays.hashCode(address1) (mod 2^32)
// Brute-force example (runs in seconds on commodity hardware):
int targetHash = Arrays.hashCode(address1);
byte[] collision = new byte[20];
for (int i = Integer.MIN_VALUE; i <= Integer.MAX_VALUE; i++) {
    collision[0] = (byte)(i >> 8);
    collision[1] = (byte)(i);
    // adjust remaining bytes to match hash...
    if (Arrays.hashCode(collision) == targetHash && !Arrays.equals(collision, address1)) {
        System.out.println("Collision found: " + Arrays.toString(collision));
        break;
    }
}

// Attack sequence:
// 1. eth_call to address1 → entity cached under key=targetHash
// 2. Within 1s: eth_call using collision address → cache returns entity for address1
```
