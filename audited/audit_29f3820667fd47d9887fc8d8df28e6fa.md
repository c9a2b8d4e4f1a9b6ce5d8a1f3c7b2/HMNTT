### Title
Unbounded Prometheus Query Fan-Out via Uncached `ImporterLagHealthIndicator.health()`

### Summary
`ImporterLagHealthIndicator.health()` performs a synchronous, blocking outbound HTTP call to Prometheus on every invocation with no caching, debouncing, or rate limiting. Any caller who can reach `/actuator/health` can issue N concurrent requests, causing N simultaneous outbound TCP connections to Prometheus, each held open for up to the configured 5-second timeout. This produces a linear N-fold amplification of outbound network and CPU load on the monitor node and load on the Prometheus backend, easily exceeding the 30% resource-consumption threshold.

### Finding Description

**Exact code path:**

`monitor/src/main/java/org/hiero/mirror/monitor/health/ImporterLagHealthIndicator.java`, lines 48–62:

```java

        return up();
    }
    final var query = buildLagQuery();
    try {
        final var resp = prometheusClient.query(query);   // ← blocking HTTP call, no cache
        return evaluate(resp);
    } catch (final Exception e) { ... }
}
```

`monitor/src/main/java/org/hiero/mirror/monitor/health/PrometheusApiClient.java`, lines 46–53:

```java
PrometheusQueryResponse query(final String query) {
    final var encodedQuery = UriUtils.encodeQuery(query, StandardCharsets.UTF_8);
    return prometheusClient          // SimpleClientHttpRequestFactory — blocking
            .get()
            .uri(builder -> builder.queryParam("query", encodedQuery).build())
            .retrieve()
            .body(PrometheusQueryResponse.class);   // blocks until response or timeout
}
```

`PrometheusApiClient` is constructed with `SimpleClientHttpRequestFactory` with a `connectTimeout` and `readTimeout` both defaulting to **5 seconds** (`ImporterLagHealthProperties.timeout = Duration.ofSeconds(5)`). Every call to `health()` opens a new blocking HTTP connection to Prometheus and holds it for up to 5 seconds.

**Root cause:** The class implements the synchronous `HealthIndicator` interface and contains no `AtomicReference` cache, no `@Cacheable`, no Reactor `.cache()`, and no debounce guard. This is in direct contrast to `ReleaseHealthIndicator`, which explicitly uses `.cache(properties.getCacheExpiry(), ...)` to avoid repeated backend calls.

**Why existing checks fail:**
- `properties.isEnabled()` only short-circuits when the feature is disabled entirely; it provides no per-request throttling.
- Spring Boot's actuator health endpoint has no built-in per-indicator rate limiting.
- No Spring Security configuration restricting `/actuator/health` access was found in the monitor module (no `application.yaml` with `management.endpoint.health` access controls, no `SecurityWebFilterChain` bean).
- The `LoggingFilter` only logs requests; it does not gate them.

### Impact Explanation

Each concurrent GET to `/actuator/health` (or `/actuator/health/liveness`, `/actuator/health/readiness`) that reaches the `importerLagHealthIndicator` component causes one new blocking TCP connection to Prometheus. With N=100 concurrent requests:
- 100 simultaneous outbound connections to Prometheus, each held for up to 5 seconds
- 100 threads (or virtual threads) blocked in `SimpleClientHttpRequestFactory` I/O
- Prometheus receives 100× the normal query rate for the expensive PromQL aggregation query (multi-metric `rate()` + `sum()` + `max()` across clusters)
- Monitor JVM thread/connection pool exhaustion is possible at high N

The Prometheus query itself is non-trivial (multi-step aggregation over `[3m]` windows across multiple clusters), so the CPU amplification on the Prometheus side is significant. This directly satisfies the ">30% resource consumption increase" criterion.

### Likelihood Explanation

The monitor service exposes port 8082 (Dockerfile `EXPOSE 8082`). If the actuator port is reachable — either because the service is exposed via a LoadBalancer/NodePort, an ingress, or from within the cluster (any pod can reach ClusterIP services by default in a standard Kubernetes cluster without NetworkPolicy) — an unprivileged attacker or compromised workload can trivially issue concurrent HTTP GETs. No authentication, no token, no special header is required. The attack is fully repeatable and scriptable with a single `curl` loop or `ab`/`wrk` invocation.

### Recommendation

1. **Add result caching** in `ImporterLagHealthIndicator`, mirroring the pattern already used in `ReleaseHealthIndicator`:
   - Convert to `ReactiveHealthIndicator` and apply `.cache(Duration)` on the result `Mono`, or
   - Store the last result in an `AtomicReference<Health>` refreshed by a scheduled task (e.g., every 15–30 seconds), and return the cached value from `health()` without calling Prometheus inline.
2. **Restrict actuator endpoint access** via Spring Security or `management.endpoint.health.show-details=when-authorized` and network policy.
3. **Apply a connection pool limit** on `PrometheusApiClient` so that even without caching, concurrent Prometheus connections are bounded.

### Proof of Concept

```bash

# importer-lag health indicator enabled (prometheusBaseUrl, localCluster, clusters configured)
