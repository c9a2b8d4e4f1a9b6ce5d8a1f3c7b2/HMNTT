### Title
Unauthenticated IP Spoofing via Unvalidated `X-Real-IP` / `X-Forwarded-For` Headers Enables Audit Log Poisoning

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the `X-Real-IP` and `X-Forwarded-For` HTTP headers supplied by any client, with no trusted-proxy allowlist and no format validation. Any unprivileged external caller can inject an arbitrary IP string into every request log entry, permanently misattributing their API activity to a fabricated address and destroying the forensic integrity of the audit trail.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64 – blindly trusted

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67 – blindly trusted
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only real fallback
    }

    return ipAddress
}
```

The returned value is immediately embedded in a structured log line by `TracingMiddleware` (line 52–53) and emitted at `log.Info` level for every non-internal path (line 58). There is:

- **No trusted-proxy IP allowlist** — the header is accepted regardless of where the TCP connection originates.
- **No format/content validation** — the value is used verbatim; it can be any string (e.g., `"185.220.101.1"`, `"<script>"`, `"INTERNAL_ADMIN"`).
- **No secondary correlation** — `r.RemoteAddr` (the real peer address) is never logged alongside the header value, so the true source is silently discarded.

`main.go` line 218 wires `TracingMiddleware` as the outermost application layer with no proxy-trust configuration:

```go
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
```

The test suite (`trace_test.go` lines 52–58) explicitly asserts that a client-supplied `X-Real-IP` value is recorded verbatim, confirming the behavior is load-bearing and untested for abuse.

**Root cause / failed assumption:** The design assumes `X-Real-IP` is only ever set by a trusted reverse proxy sitting in front of the service. No deployment guard enforces this assumption at the code level.

### Impact Explanation
Every API call made by the attacker is permanently logged under a fabricated IP address. Concrete consequences:

- **Attacker anonymity:** The attacker's real `RemoteAddr` is never recorded; investigators see only the spoofed address.
- **False attribution / framing:** The attacker can supply another legitimate user's IP, causing all suspicious queries to appear to originate from that user.
- **Audit trail integrity loss:** Post-incident forensics (rate-abuse analysis, data-exfiltration investigations, regulatory audits) rely on these log entries; poisoned entries produce false conclusions.
- **No authentication barrier:** The Rosetta API requires no credentials, so the attack surface is the entire public internet.

Severity: **High** (forensic record reorganization, no direct fund theft, but complete loss of accountability for API-layer activity).

### Likelihood Explanation
- **Precondition:** Network reachability to the Rosetta port — no account, token, or special privilege required.
- **Trigger:** A single HTTP header added to any request (`X-Real-IP: <target_ip>`).
- **Repeatability:** 100% — every request with the header is mislogged; the attacker can sustain this indefinitely.
- **Tooling:** `curl -H "X-Real-IP: 185.220.101.1" http://<host>:<port>/network/list` — no exploit framework needed.
- **Detection difficulty:** Because the real `RemoteAddr` is discarded and never co-logged, there is no in-band signal to distinguish a spoofed entry from a legitimate proxy-forwarded one.

### Recommendation
1. **Maintain a trusted-proxy allowlist.** Only accept `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` matches a configured set of trusted proxy CIDRs. Reject or ignore the header otherwise.
2. **Always log `r.RemoteAddr` alongside the resolved IP** so the true TCP peer is never silently discarded.
3. **Validate the extracted value** with `net.ParseIP()` before use; reject non-IP strings entirely.
4. Consider adopting a hardened library (e.g., `realip` middleware with explicit proxy trust configuration) rather than hand-rolling header parsing.

### Proof of Concept

```bash
# Step 1 – baseline: confirm your real IP appears in logs
curl http://<rosetta-host>:<port>/network/list
# Log entry: "1.2.3.4 GET /network/list (200) in 5ms"

# Step 2 – spoof a Tor exit node IP
curl -H "X-Real-IP: 185.220.101.1" http://<rosetta-host>:<port>/network/list
# Log entry: "185.220.101.1 GET /network/list (200) in 5ms"
# Attacker's real IP (1.2.3.4) is never recorded.

# Step 3 – frame another user
curl -H "X-Real-IP: <victim-ip>" http://<rosetta-host>:<port>/block
# Log entry: "<victim-ip> GET /block (200) in 4ms"
# All subsequent forensic analysis attributes the activity to the victim.
