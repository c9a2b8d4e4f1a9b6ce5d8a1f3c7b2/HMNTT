### Title
Cache-Bypass DB Amplification DoS via Unauthenticated Token Relationships Endpoint

### Summary
The public `/api/v1/accounts/{id}/tokens` endpoint allows any unauthenticated user to trigger a secondary `WHERE token_id = any($1)` database query for up to `limit` (default max 100) token IDs per request. Because the in-process LRU cache (`quickLru`) is finite and evicts entries, an attacker supplying always-novel token IDs can continuously bypass the cache and hammer the database with large array queries, causing non-network-based DoS through DB resource exhaustion.

### Finding Description

**Code path:**

`rest/controllers/tokenController.js` → `getTokenRelationships()` (line 74) calls `TokenService.getTokenAccounts(query)`.

In `rest/service/tokenService.js`:

- `getTokenAccounts()` (lines 96–115): executes the primary `token_account` query, then at line 103 collects all returned `token_id` values into a `Set`, and at line 104 passes that Set to `getCachedTokens(tokenIds)`.
- `getCachedTokens()` (lines 136–161): iterates the Set, separates cached from uncached IDs (lines 139–146), and if any are uncached, issues a second DB query at line 152:
  ```js
  const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
  ```
  using the static query (lines 31–38):
  ```sql
  SELECT decimals, freeze_status, kyc_status, token_id
  FROM token
  WHERE token_id = any ($1)
  ```

**Root cause:** The only protection against repeated DB lookups is the in-process `quickLru` cache (line 12–14), which has a finite `maxSize`. When the attacker's requests reference token IDs not present in the cache (either because the token universe exceeds `maxSize`, or because LRU eviction has removed them), every request causes a full secondary DB query. There is no application-level rate limiting (confirmed absent from the codebase — only a test-file reference exists), and the endpoint requires no authentication.

**Failed assumption:** The design assumes the LRU cache will absorb repeated lookups for the same token IDs. This fails when the attacker rotates through a large set of token IDs (all publicly visible on-chain), keeping the cache cold.

### Impact Explanation
Each unauthenticated request to `/api/v1/accounts/{id}/tokens?limit=100` can force two DB queries: the primary `token_account` query and a secondary `WHERE token_id = any($1)` query with up to 100 IDs. At high request rates with rotating token IDs, this doubles effective DB query load. The `any($1)` array query, while indexed, is more expensive than a point lookup and scales with array size. Sustained attack degrades DB throughput for all users of the mirror node, potentially causing service unavailability — a non-network-based DoS.

### Likelihood Explanation
Preconditions are minimal: no credentials, no special access. Token IDs and account IDs are public on Hedera's ledger. An attacker can enumerate thousands of valid token IDs trivially. The attack is fully repeatable and scriptable with standard HTTP tooling. The absence of application-level rate limiting means the only throttle is infrastructure-level (e.g., a reverse proxy), which may not be present in all deployments.

### Recommendation
1. **Rate limiting**: Add per-IP or per-client request rate limiting at the application layer for this endpoint.
2. **Cache size / TTL hardening**: Ensure `config.cache.token.maxSize` is large enough to cover the active token universe; add a TTL to prevent stale entries from being evicted prematurely.
3. **Cap secondary query size**: In `getCachedTokens()`, if `uncachedTokenIds.length` exceeds a threshold (e.g., 50), split into batches or reject with a server-side guard.
4. **Consider a shared/distributed cache** (e.g., Redis) so the in-process LRU is not the sole protection layer.

  async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return [];
    }

    const tokenIds = rows.reduce((result, row) => result.add(row.token_id), new Set());
    const cachedTokens = await this.getCachedTokens(tokenIds);
    return rows.map((row) => {
      const cachedToken = cachedTokens.get(row.token_id);
      if (cachedToken) {
        row.decimals = cachedToken.decimals;
        row.freeze_status = row.freeze_status ?? cachedToken.freezeStatus;
        row.kyc_status = row.kyc_status ?? cachedToken.kycStatus;
      }

      return new TokenAccount(row);
    });
  }
```

**File:** rest/service/tokenService.js (L136-161)
```javascript
  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
