### Title
Unauthenticated Concurrent Topic Subscriptions Exhaust JVM Heap via Unbounded `getResultList()` Materialization in `PollingTopicMessageRetriever.poll()`

### Summary
Any unauthenticated external caller can open an arbitrary number of gRPC connections to `subscribeTopic`, each triggering repeated calls to `poll()` which materializes up to `maxPageSize` (1 000–5 000) fully-hydrated `TopicMessage` rows into JVM heap per cycle via `typedQuery.getResultList()`. Because there is no global concurrent-subscription cap and the only per-connection limit (5 calls/connection) is trivially bypassed by opening many connections, an attacker can exhaust JVM heap and crash the mirror-node gRPC service without sending any network-flood traffic.

### Finding Description

**Exact materialization point** — `TopicMessageRepositoryCustomImpl.findByFilter()` line 60:

```java
return typedQuery.getResultList().stream();
```

`getResultList()` executes the SQL query and loads the **entire result set** into a `java.util.List<TopicMessage>` in heap before returning. This is not a lazy cursor. The list is then wrapped as a `Stream` and handed to `Flux.fromStream(...)` at `PollingTopicMessageRetriever.poll()` line 78.

**Page-size caps** (`RetrieverProperties.java` lines 22, 39):
- Throttled path (used for historical retrieval): `maxPageSize = 1000`
- Unthrottled path: `maxPageSize = 5000`

`TopicMessageServiceImpl.subscribeTopic()` line 63 calls `topicMessageRetriever.retrieve(filter, true)` (throttled), so 1 000 rows per poll cycle is the baseline. Each `TopicMessage` carries a `message` field that the HCS proto spec allows up to 6 KiB, plus `runningHash` (48 bytes), timestamps, IDs, etc.

**Polling loop** (`PollingTopicMessageRetriever.retrieve()` lines 51–62): The `repeatWhen` loop fires every `pollingFrequency = 2 s` and continues as long as `isComplete()` returns `false`. For a topic with millions of historical messages, `isComplete()` (line 124–125) returns `false` whenever the last page was full — so polling continues indefinitely until the 60 s `timeout`.

**No global subscription limit**: `TopicMessageServiceImpl` tracks `subscriberCount` (line 48) only as a Micrometer gauge metric; it is never checked to reject new subscriptions.

**Per-connection limit is insufficient**: `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14) limits calls per TCP connection, but an attacker opens many independent connections — each contributing 5 concurrent subscriptions.

**Retry amplification** (`PollingTopicMessageRetriever.retrieve()` line 58):
```java
.retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
```
Errors are retried `Long.MAX_VALUE` times, meaning a subscription that hits an OOM-induced error will immediately restart and re-issue the heap-allocating query.

**Exploit flow**:
1. Attacker identifies a topic with a large historical message backlog (publicly visible on Hedera mainnet/testnet).
2. Attacker opens N TCP connections to port 5600 (default gRPC port).
3. Each connection issues 5 concurrent `subscribeTopic` RPCs with `consensusStartTime` set to epoch 0 (or any early timestamp) and no `limit` field.
4. Each subscription's `poll()` call materializes 1 000 `TopicMessage` objects into heap every 2 s.
5. With N = 200 connections → 1 000 concurrent subscriptions → 1 000 × 1 000 rows × ~6 KiB/row = ~6 GB heap pressure per poll cycle, far exceeding typical JVM heap allocations.
6. JVM throws `OutOfMemoryError`; gRPC service crashes or becomes unresponsive.

### Impact Explanation
The gRPC mirror-node service becomes unavailable to all legitimate subscribers. Because `subscribeTopic` is the primary API for HCS (Hedera Consensus Service) clients to receive topic messages, a crash disrupts all downstream applications relying on real-time or historical HCS data. Recovery requires a JVM restart; the attacker can immediately re-trigger the condition. No privileged access, API key, or account is required.

### Likelihood Explanation
The attack requires only a gRPC client library (e.g., `grpcurl`, any HCS SDK) and knowledge of a topic ID with historical messages — both are publicly available. Opening hundreds of TCP connections is a standard OS-level operation. The attack is fully repeatable and can be automated in under 50 lines of code. Topics with large message histories exist on mainnet today.

### Recommendation

1. **Enforce a global concurrent-subscription cap**: In `TopicMessageServiceImpl.subscribeTopic()`, check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP / per-identity connection limiting** at the Netty/gRPC layer or via an ingress proxy.
3. **Replace `getResultList()` with a true streaming cursor**: In `TopicMessageRepositoryCustomImpl.findByFilter()` line 60, use `typedQuery.getResultStream()` with a Hibernate `ScrollableResults` or a `@QueryHints({@QueryHint(name = HINT_FETCH_SIZE, value = "...")})` approach so rows are not all materialized at once. The comment on line 60 (`// getResultStream()'s cursor doesn't work with reactive streams`) should be revisited with a proper JDBC fetch-size hint instead.
4. **Cap the `timeout` and `numRepeats`** more aggressively for unauthenticated callers, or require authentication for historical backfill subscriptions.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
