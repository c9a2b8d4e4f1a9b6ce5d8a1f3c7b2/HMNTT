### Title
Unauthenticated Historical Hook Storage Disclosure via `timestamp` Parameter

### Summary
Any unauthenticated external user can supply a `timestamp` query parameter to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, causing `isHistorical()` to return `true` and routing the request to `getHookStorageChange()`, which queries the `hook_storage_change` audit table. Unlike the non-historical path, the historical path does not filter out deleted storage entries, exposing storage slots that have been overwritten or deleted — state that is intentionally hidden from the current-state view. There is no authentication or authorization gate anywhere on this endpoint.

### Finding Description

**Code path:**

1. `HooksController.java` lines 104–130: The endpoint `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` accepts a `timestamps` array parameter with no authentication annotation (`@PreAuthorize`, `@Secured`, `@RolesAllowed`) and no Spring Security configuration exists in the repository. [1](#0-0) 

2. `HookStorageRequest.java` lines 66–68: `isHistorical()` returns `true` whenever `timestamp` is non-empty — i.e., any single timestamp query parameter is sufficient. [2](#0-1) 

3. `HookServiceImpl.java` lines 56–58: When `isHistorical()` is `true`, execution is unconditionally routed to `getHookStorageChange()`. [3](#0-2) 

4. `HookStorageChangeRepository.java` lines 15–39 and 41–63: Both historical queries select from `hook_storage_change` and explicitly compute `(value_written is null or length(value_written) = 0) as "deleted"` — returning deleted entries — with **no** `AND deleted = false` filter. Contrast with the non-historical `HookStorageRepository` which uses `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse`. [4](#0-3) 

**Root cause:** The failed assumption is that the `timestamp` parameter is an optional convenience feature with no security boundary. In reality it silently switches the data source from the current-state `hook_storage` table (which filters deleted entries) to the full audit log `hook_storage_change` table (which does not). No privilege check guards this switch.

**Why existing checks fail:** The only constraints on `timestamps` are `@Size(max = 2)` (limits count to 2 parameters) and format validation in `TimestampParameter.valueOf()`. Neither constitutes an access control check. `Bound.isEmpty()` returns `false` for any valid timestamp, including `gte:0`, which covers the entire history. [5](#0-4) [6](#0-5) 

### Impact Explanation

The `hook_storage_change` table is the complete audit log of EVM storage mutations for gossip-prevention hooks. It records every `value_written` (including zero-length writes that represent deletions) for every storage slot ever touched by a hook. The non-historical path deliberately hides deleted slots via `DeletedIsFalse`; the historical path returns them with `deleted=true`. An attacker can:

- Enumerate all storage keys ever used by a hook, including those since deleted.
- Reconstruct the full mutation history of any slot by issuing requests with progressively advancing `timestamp=lte:T` values.
- Observe the exact consensus timestamps at which gossip-prevention state changed, correlating them with on-chain transactions.

This constitutes a complete bypass of the deletion/overwrite privacy model and leaks the internal state machine of gossip-prevention hooks to any internet-accessible caller. [7](#0-6) 

### Likelihood Explanation

Exploitation requires zero privileges, zero credentials, and zero special network position. The attacker only needs to know (or enumerate) a valid `ownerId` and `hookId` — both of which are public on-chain identifiers — and append `?timestamp=gte:0` to the standard storage endpoint. The attack is fully repeatable, automatable, and leaves no distinguishable server-side trace beyond normal API access logs. Any public mirror node deployment is affected.

### Recommendation

1. **Add an authorization gate**: Require that the caller is the `ownerId` (or an authorized delegate) before serving historical data. Implement a Spring Security `SecurityFilterChain` or `@PreAuthorize` annotation on `getHookStorage` that enforces this for requests where `isHistorical()` would be `true`.

2. **Filter deleted entries in the historical path**: Add `AND (value_written IS NOT NULL AND length(value_written) > 0)` to both queries in `HookStorageChangeRepository` to match the non-historical behavior, unless the design intent is to expose deletions (in which case access control is mandatory).

3. **Restrict timestamp range**: Enforce a maximum lookback window (e.g., reject `timestamp=gte:0`) to limit bulk history extraction even if some historical access is permitted.

### Proof of Concept

```
# Step 1: Identify a valid ownerId and hookId (public on-chain data)
# e.g., ownerId = 0.0.123, hookId = 1

# Step 2: Query current state (no deleted entries returned)
GET /api/v1/accounts/0.0.123/hooks/1/storage
# → returns only non-deleted current storage slots

# Step 3: Query historical state as unauthenticated user
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0
# → isHistorical() = true
# → routes to getHookStorageChange()
# → queries hook_storage_change with timestampLowerBound=0, timestampUpperBound=Long.MAX_VALUE
# → returns all storage slots including deleted ones (deleted=true in response)

# Step 4: Reconstruct full mutation history by iterating timestamp upper bound
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=lte:1726874345000000000
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=lte:1726874300000000000
# → each response reveals the most-recent value_written per key at that point in time,
#    allowing full reconstruction of the storage change history
```
