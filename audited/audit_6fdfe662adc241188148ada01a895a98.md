### Title
Cache-Control: public, max-age=0 Served on Near-Expired Cache Hits Due to Redis TTL Floor Truncation

### Summary
Redis's `TTL` command returns whole-second integers (floor), so a key with less than one second of remaining lifetime returns `0` while `GET` still returns the value within the same `MULTI` pipeline. `responseCacheCheckHandler` blindly interpolates this `0` into `Cache-Control: public, max-age=0`, instructing downstream caches to treat the response as immediately stale. Any unprivileged user can trigger this by timing a request to arrive within the last second of a cached entry's lifetime.

### Finding Description

**Code path:**

`rest/cache.js`, `getSingleWithTtl()`, lines 69–79:
```js
const result = await this.redis
  .multi()
  .ttl(key)
  .get(key)
  .exec()
  ...
const rawValue = result[1][1];
if (rawValue) {
  return {ttl: result[0][1], value: JSONParse(rawValue)};  // ttl can be 0
}
```

`rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler()`, lines 50–60:
```js
const {ttl: redisTtl, value: redisValue} = cachedTtlAndValue;
...
const headers = {
  ...cachedResponse.headers,
  ...{[CACHE_CONTROL_HEADER]: `public, max-age=${redisTtl}`},  // max-age=0 when redisTtl=0
};
```

**Root cause:** Redis `TTL` returns the remaining lifetime in whole seconds (floor). A key with 0.001–0.999 seconds remaining returns `0` from `TTL` but still returns its value from `GET` within the same atomic `MULTI/EXEC` block (the key has not yet expired). The guard at line 78 (`if (rawValue)`) only checks value existence, not whether `ttl > 0`. The returned `{ttl: 0, value: ...}` object propagates to `responseCacheCheckHandler`, which unconditionally formats `public, max-age=${redisTtl}` without any lower-bound check.

**Failed assumption:** The code assumes `redisTtl` will always be a positive integer when a cache hit occurs. This is false for the sub-second window before expiry.

**Exploit flow:**
1. Attacker makes a GET request to any cacheable endpoint (e.g., `/api/v1/accounts/0.0.12345/rewards`).
2. Server returns a cache hit with `Cache-Control: public, max-age=N` (e.g., `max-age=5`).
3. Attacker waits N−1 seconds (e.g., 4 seconds), then sends another GET request within the final second of the TTL.
4. `getSingleWithTtl` executes `MULTI/TTL/GET`: `TTL` returns `0`, `GET` returns the cached body.
5. `responseCacheCheckHandler` sets `Cache-Control: public, max-age=0` on the response.
6. Any CDN, reverse proxy, or shared cache between the attacker and the origin receives this directive and immediately marks the resource stale.

**Existing checks and why they are insufficient:**
- `if (rawValue)` (line 78, `cache.js`): only guards against a null value (cache miss), not a zero TTL.
- `if (ttl > 0)` in `responseCacheUpdateHandler` (line 97, `responseCacheHandler.js`): only prevents *writing* zero-TTL entries to Redis; it does not prevent *reading* an entry whose TTL has decayed to 0 after it was written.
- No floor/clamp on `redisTtl` before header construction.

### Impact Explanation
Any downstream HTTP cache (CDN layer, reverse proxy, shared proxy) that receives `Cache-Control: public, max-age=0` will immediately evict or bypass its cached copy of the resource. Subsequent requests from all users routed through that cache will be forwarded to the origin, defeating the purpose of the response cache and increasing origin load. Because the header says `public`, shared caches are explicitly targeted. The attacker can repeat this on a per-endpoint basis to continuously suppress caching for high-traffic endpoints, constituting a griefing attack on caching infrastructure with no authentication required.

### Likelihood Explanation
The attack requires no credentials or special access — only the ability to send HTTP GET requests. The timing window is the sub-second interval before TTL expiry. An attacker can observe `Cache-Control: max-age=X` from any cache-hit response, compute the expiry moment, and automate requests to land within the last second. This is trivially scriptable. The window repeats every cache cycle, so the attack is continuously repeatable. The only constraint is network timing precision, which is easily achievable from a co-located or low-latency vantage point.

### Recommendation
Clamp `redisTtl` to a minimum of `1` before using it in the `Cache-Control` header, and return `undefined` (cache miss) from `getSingleWithTtl` when `ttl <= 0`:

In `rest/cache.js`, `getSingleWithTtl`:
```js
const ttl = result[0][1];
const rawValue = result[1][1];
if (rawValue && ttl > 0) {          // reject zero/negative TTL
  return {ttl, value: JSONParse(rawValue)};
}
return undefined;
```

This causes the handler to treat a near-expired entry as a cache miss, falling through to a fresh origin fetch, which will produce a correct `Cache-Control` header.

### Proof of Concept
```bash
# 1. Warm the cache and capture max-age
RESPONSE=$(curl -si "http://<host>/api/v1/accounts/0.0.12345/rewards")
MAX_AGE=$(echo "$RESPONSE" | grep -i cache-control | grep -oP 'max-age=\K\d+')

# 2. Wait until the last sub-second window
sleep $((MAX_AGE - 1))

# 3. Fire request within the final second of TTL
curl -si "http://<host>/api/v1/accounts/0.0.12345/rewards" | grep -i cache-control
# Expected output: Cache-Control: public, max-age=0
```
