### Title
Gas Estimation Binary Search Amplifies EVM Execution Work Per Rate-Limit Token

### Summary
The `ContractController.call()` method consumes exactly one `rateLimitBucket` token and one `gasLimitBucket` deduction per HTTP request regardless of whether `estimate=true`. When `estimate=true`, `ContractExecutionService.estimateGas()` triggers one initial EVM execution followed by up to 20 additional `doProcessCall()` iterations inside `BinaryGasEstimator.search()`, yielding up to 21 EVM executions per single throttle check. The binary search iterations pass `estimate=true` to `doProcessCall()`, which explicitly skips `restoreGasToBucket()` and any additional throttle accounting, so all 20 search iterations run entirely outside the throttle's view.

### Finding Description

**Throttle entry point — one check, one token:**

`ContractController.call()` line 40:
```java
throttleManager.throttle(request);
```
`ThrottleManagerImpl.throttle()` lines 38–42 consumes 1 `rateLimitBucket` token and `scaleGas(request.getGas())` `gasLimitBucket` tokens — once, unconditionally, before any branching on `estimate`. [1](#0-0) 

**Estimate path triggers up to 21 EVM executions:**

`ContractExecutionService.processCall()` line 53–54 routes to `estimateGas()`. Inside `estimateGas()`:
- Line 82: `callContract(params, context)` → `doProcessCall(params, params.getGas(), false)` — 1 execution, gas partially restored.
- Lines 91–95: `binaryGasEstimator.search(gas -> doProcessCall(params, gas, true), ...)` — up to `maxGasEstimateRetriesCount = 20` additional executions. [2](#0-1) 

**Binary search loop cap is 20:**

`BinaryGasEstimator.search()` line 35: `while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount())` where `maxGasEstimateRetriesCount = 20`. [3](#0-2) 

**Binary search iterations are invisible to the throttle:**

`ContractCallService.doProcessCall()` lines 128–135: the `finally` block only calls `restoreGasToBucket()` and records metrics when `!estimate`. When called with `estimate=true` (all 20 binary search iterations), no throttle interaction occurs at all — no consumption, no restoration. [4](#0-3) 

**Root cause:** The throttle is applied at the HTTP boundary on the raw request object, before the service layer decides how many EVM executions to perform. The `estimate` flag is not factored into the throttle budget.

### Impact Explanation

An attacker can send `estimate=true` requests at the full `requestsPerSecond` rate (default 500 req/s) and trigger up to 21 EVM executions per request — an effective 21× amplification of CPU, memory, and I/O load on the mirror node compared to `estimate=false` requests at the same rate. At 500 req/s × 21 executions = 10,500 EVM invocations per second, each potentially executing up to 15M gas of EVM bytecode. This can exhaust mirror node CPU and cause denial of service for all users. The `gasLimitBucket` does not compensate because it is consumed once for the declared gas limit, not multiplied by the number of binary search iterations.

Note: the mirror node is a read-only simulation layer and does not submit transactions to the Hedera gossip network; the impact is resource exhaustion on the mirror node itself, not gossip amplification.

### Likelihood Explanation

No authentication or special privilege is required. Any HTTP client can set `"estimate": true` in the POST body to `/api/v1/contracts/call`. The attack is trivially repeatable, requires no contract deployment, and can be scripted with a single curl command in a loop. The amplification factor is deterministic and bounded only by the `maxGasEstimateRetriesCount` configuration value.

### Recommendation

1. **Account for estimate multiplier in the throttle:** Before entering `estimateGas()`, consume additional `gasLimitBucket` tokens proportional to the expected number of binary search iterations (e.g., multiply the gas deduction by `maxGasEstimateRetriesCount`), or restore the correct amount after the search completes.
2. **Separate rate limit for estimate requests:** Add a dedicated `estimateRateLimitBucket` with a lower capacity than the general `rateLimitBucket`, applied when `request.isEstimate() == true`.
3. **Restore gas for each binary search iteration:** Pass the actual gas used per iteration back to `restoreGasToBucket()` inside the search loop, so the gas bucket accurately reflects total EVM work performed.

### Proof of Concept

```bash
# Send 500 estimate=true requests per second (at the default rate limit)
# Each triggers up to 21 EVM executions internally

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "0x0000000000000000000000000000000000000001",
      "data": "0x<complex_calldata>",
      "gas": 15000000,
      "estimate": true
    }' &
done
wait
# Result: up to 10,500 EVM executions triggered while only 500 rateLimitBucket
# tokens and 500 gasLimitBucket deductions were consumed.
```

Each request passes the single `throttleManager.throttle(request)` check at `ContractController.java:40`, then internally fans out to 21 EVM executions via `estimateGas()` → `callContract()` + `binaryGasEstimator.search()` with no further throttle interaction.

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
