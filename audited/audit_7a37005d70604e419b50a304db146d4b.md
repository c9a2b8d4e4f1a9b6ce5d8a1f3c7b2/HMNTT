### Title
TOCTOU Race in `initGenesisRecordFile` Allows Concurrent Goroutines to Flood DB with Expensive `selectGenesis` CTE Queries

### Summary
In `rosetta/app/persistence/block.go`, the `initGenesisRecordFile()` function uses an unsynchronized fast-path check at line 244 before executing the expensive `selectGenesis` CTE query. The `sync.Once` guard is placed only around the *assignment* at line 257, not around the database query itself. Any number of concurrent goroutines can simultaneously pass the line 244 check and each independently execute the full `selectGenesis` CTE query against the database, with only one goroutine's result being kept. An unprivileged external user can trigger this by sending a burst of concurrent requests to any public Rosetta API endpoint during the startup window.

### Finding Description
**Exact code path:** `rosetta/app/persistence/block.go`, function `initGenesisRecordFile()`, lines 243‚Äì263.

**Root cause:** The `sync.Once` is misplaced. It guards only the struct assignment (`br.genesisBlock = rb`, line 258), not the expensive database query at lines 252‚Äì255. The fast-path guard at line 244 reads `br.genesisBlock.ConsensusStart` without any mutex or atomic operation, making it a data race under Go's memory model ‚Äî a concurrent write to `br.genesisBlock` (a value-type struct, not a pointer) at line 258 races with the read at line 244.

**Exploit flow:**
1. The Rosetta node starts up; `br.genesisBlock.ConsensusStart` is initialized to `genesisConsensusStartUnset` (`-1`).
2. An attacker sends N concurrent HTTP requests to any block-related Rosetta endpoint (`/block`, `/block/transaction`, `/network/status`, etc.), all of which call `initGenesisRecordFile`.
3. All N goroutines read `br.genesisBlock.ConsensusStart == -1` at line 244 simultaneously (no lock), pass the check, and proceed.
4. All N goroutines independently execute the full `selectGenesis` CTE query (lines 252‚Äì255) against the database.
5. All N goroutines reach `once.Do` at line 257; only one executes the closure. The other N-1 queries were wasted and their DB connections held for the full query duration.
6. Additionally, the unsynchronized read of `br.genesisBlock.ConsensusStart` at line 244 while another goroutine writes the entire `br.genesisBlock` struct at line 258 is a Go data race, which can cause memory corruption or torn reads of the struct fields.

**Why existing checks fail:** `sync.Once` is the correct primitive, but it is applied only to the assignment, not to the query. The fast-path check at line 244 is entirely outside any synchronization boundary. The correct pattern is to place the entire block ‚Äî check, query, and assignment ‚Äî inside `once.Do`. [1](#0-0) 

### Impact Explanation
During the startup window (before genesis is first successfully initialized), every concurrent request causes a full `selectGenesis` CTE execution. The query joins `account_balance` (potentially millions of rows on a production mirror node) with `record_file` and performs an ordered scan. Under a burst of N concurrent requests, N copies of this query run simultaneously, consuming N database connections and proportional CPU/IO on the PostgreSQL backend. If the database connection pool is exhausted or the DB host is CPU-bound, all Rosetta API responses stall, effectively taking the Rosetta node offline during startup. Because the startup window can be extended (e.g., if the DB is slow to return the genesis result, all goroutines remain blocked in the query), the attack window is not strictly instantaneous. Additionally, the data race on `br.genesisBlock` (a value struct) can produce torn reads of `ConsensusStart`, `ConsensusEnd`, `Hash`, `Index`, or `PrevHash`, leading to incorrect block data being served. [2](#0-1) [3](#0-2) 

### Likelihood Explanation
The Rosetta API is a public-facing HTTP service with no authentication required for block-query endpoints. Any unprivileged external user can send concurrent HTTP requests. The attack requires only a standard HTTP client capable of sending parallel requests (e.g., `curl`, `ab`, `wrk`). The startup window exists every time the service restarts (e.g., after a deployment, crash, or rolling update). In a horizontally scaled deployment, each instance has its own `blockRepository` with its own uninitialized `sync.Once`, so the attack can be repeated against each instance independently on every restart. [4](#0-3) [5](#0-4) 

### Recommendation
Move the entire initialization logic ‚Äî including the database query ‚Äî inside `once.Do`, eliminating both the TOCTOU race and the data race:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    var initErr *rTypes.Error
    br.once.Do(func() {
        db, cancel := br.dbClient.GetDbWithContext(ctx)
        defer cancel()

        var rb recordBlock
        if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
            First(&rb).Error; err != nil {
            initErr = handleDatabaseError(err, hErrors.ErrNodeIsStarting)
            // Reset once so the next call retries (use a mutex+bool instead of sync.Once if retry is needed)
            return
        }
        br.genesisBlock = rb
        log.Infof("Fetched genesis record file, index - %d", br.genesisBlock.Index)
    })
    return initErr
}
```

Note: `sync.Once` does not retry on failure. If the DB query can fail transiently (e.g., `ErrNodeIsStarting`), replace `sync.Once` with a `sync.Mutex` + `bool` initialized flag so that failed initializations are retried on the next request, while successful initialization is cached permanently. ### Proof of Concept
**Preconditions:** Rosetta node is starting up (genesis not yet initialized). The `account_balance` table is populated (normal production state).

**Steps:**
```bash
# Send 500 concurrent requests to the /network/status endpoint (calls RetrieveGenesis -> initGenesisRecordFile)
# before the node has completed its first genesis initialization
seq 500 | xargs -P 500 -I{} curl -s -X POST http://<rosetta-host>:5700/network/status \
  -H "Content-Type: application/json" \
  -d '{"network_identifier": {"blockchain": "Hedera", "network": "mainnet"}}'
```

**Expected result:** 500 concurrent `selectGenesis` CTE queries are issued to the PostgreSQL backend simultaneously. Database CPU/IO spikes; connection pool may be exhausted; all 500 requests block until the DB responds to each. The Go race detector (`go test -race`) will flag the unsynchronized read/write of `br.genesisBlock` between lines 244 and 258. [7](#0-6)
