### Title
Unbounded In-Memory Opcode Trace Accumulation Leads to Heap Exhaustion via EVM Replay

### Summary
The `getContractOpcodes()` endpoint replays historical EVM transactions and records every executed opcode into an unbounded `ArrayList<Opcode>` in `OpcodeContext`. No per-request cap on opcode count or memory consumption exists. An unprivileged attacker who previously submitted a high-gas tight-loop contract transaction to the Hedera network can trigger its replay, causing the JVM heap to be exhausted by a single request.

### Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (line 61–64) calls `throttleManager.throttleOpcodeRequest()` then `opcodeService.processOpcodeCall(request)` with no per-request memory guard. [1](#0-0) 

`OpcodeServiceImpl.processOpcodeCall()` (line 67) pre-allocates the `ArrayList` with `gas / 3` as an *initial capacity hint*, not a cap. For a 15 M-gas transaction this pre-allocates 5 M slots, but the list grows without bound as opcodes are appended. [2](#0-1) 

`OpcodeContext.addOpcodes()` (line 54–56) performs an unconditional `opcodes.add(opcode)` with no size check. [3](#0-2) 

`OpcodeActionTracer.tracePostExecution()` (lines 44–62) is called once per executed EVM opcode and calls `addOpcodes()` unconditionally. With `stack=true` (the default), each `Opcode` object carries a full copy of the EVM stack as a `List<String>`. [4](#0-3) 

**Root cause:** The failed assumption is that gas-bounded EVM execution implies memory-bounded tracing. In reality, a tight `JUMPDEST` loop (1 gas/iteration) can execute up to `gasLimit` iterations before running out of gas. With Hedera's maximum gas limit (~15 M), this produces up to 15 M `Opcode` objects, each carrying stack data, in a single unbounded `ArrayList`.

**Why the throttle is insufficient:**

`ThrottleManagerImpl.throttleOpcodeRequest()` enforces a global rate of `opcodeRequestsPerSecond = 1` (default). This limits *frequency* but does nothing to bound the memory consumed by a single request. One request is sufficient to trigger OOM. [5](#0-4) [6](#0-5) 

There is no authentication, no per-IP limit, no opcode-count cap, and no request timeout enforced at the application layer.

### Impact Explanation
A single replayed high-gas tight-loop transaction can exhaust the JVM heap of the web3 service process. This causes an `OutOfMemoryError`, crashing or severely degrading the service for all users — not just the opcodes endpoint. Because the throttle is global (not per-source), a single attacker occupying the 1 req/sec slot keeps the service under continuous memory pressure. This maps directly to the stated scope: shutdown/severe degradation of network processing nodes without brute-force volume.

### Likelihood Explanation
The endpoint is disabled by default (`enabled = false`), which reduces exposure. However, when enabled (required for any opcode debugging use case), the attack requires only:
1. A one-time HBAR expenditure to submit a high-gas tight-loop transaction to the Hedera mainnet/testnet.
2. A single unauthenticated HTTP GET with `Accept-Encoding: gzip` — no credentials, no special role.

The transaction hash is publicly visible on-chain. The attack is fully repeatable at 1 req/sec. No brute force is needed.

### Recommendation
1. **Add a hard cap on opcode count** in `OpcodeContext.addOpcodes()`: reject or truncate once a configurable maximum (e.g., 100 000 opcodes) is reached.
2. **Add a configurable gas-limit ceiling** for opcode replay requests, independent of the original transaction's gas limit.
3. **Add a per-source-IP rate limit** in addition to the global bucket, so one attacker cannot monopolize the global slot.
4. **Set a request-level timeout** on the EVM replay execution to bound wall-clock and memory exposure.

### Proof of Concept

1. Deploy a Solidity contract on Hedera testnet containing a tight loop:
   ```solidity
   function burn(uint gas) external {
       assembly { for {} gt(gas(), gas) {} { } }
   }
   ```
2. Call `burn(0)` with `gasLimit = 15_000_000`. The transaction is recorded in the mirror node DB.
3. Obtain the transaction hash `TX_HASH` from the Hedera explorer.
4. With the opcodes endpoint enabled, send:
   ```
   GET /api/v1/contracts/results/TX_HASH/opcodes?stack=true
   Accept-Encoding: gzip
   ```
5. The mirror node replays the transaction, calling `OpcodeActionTracer.tracePostExecution()` up to ~15 M times, each appending an `Opcode` object (with full stack) to the unbounded `ArrayList` in `OpcodeContext`.
6. JVM heap is exhausted; the web3 service throws `OutOfMemoryError` and crashes or becomes unresponsive.

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
