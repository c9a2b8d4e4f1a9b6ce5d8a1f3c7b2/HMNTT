### Title
Plaintext Operator Private Key in Environment Variable Enables Unauthorized Transaction Signing

### Summary
`HIERO_MIRROR_PINGER_OPERATOR_KEY` is loaded from the process environment as a raw string and stored in `cfg.operatorKey` for the entire process lifetime with no encryption or secrets-management layer. Any party who reads `/proc/<pid>/environ` in a shared-PID-namespace pod, or who simply reads the source code (a hardcoded default key is committed), obtains a fully functional Ed25519 private key. That key can be used directly with the Hiero SDK to sign and submit arbitrary `TransferTransaction`s from the operator account — identical to what `submitWithRetry()` does — with no further authorization required.

### Finding Description
**Code path:**

- `pinger/config.go:40` — key is read from the environment via `os.Getenv` and stored as a plain `string` in `cfg.operatorKey`. A real Ed25519 private key (`302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137`) is hardcoded as the default fallback, making it publicly known to anyone who reads the repository.
- `pinger/sdk_client.go:40-45` — `hiero.PrivateKeyFromString(cfg.operatorKey)` parses the plaintext key and `client.SetOperator(opID, opKey)` binds it to the client for the process lifetime.
- `pinger/transfer.go:33` — `cryptoTransfer.Execute(client)` auto-signs every transfer using that bound operator key. No additional approval, multi-sig check, or per-call authorization exists.

**Root cause:** The private key is never encrypted, never fetched from a secrets store, and never zeroed from memory. It lives verbatim in the process environment block (`/proc/<pid>/environ`) for as long as the container runs. The Hiero SDK's `Execute(client)` requires only possession of the private key object — there is no challenge/response, no nonce, and no server-side approval step that would distinguish the legitimate pinger from an attacker replaying the same key.

**Why existing checks fail:** The only "check" is `cfg.operatorKey == ""` (config.go:105-107), which merely ensures the field is non-empty. There is no integrity protection, no usage-bound token, and no rate-limit or allowlist on the Hiero network side that would block a second client using the same key.

### Impact Explanation
An attacker who extracts the key can instantiate their own `hiero.Client`, call `SetOperator` with the stolen credentials, and submit `TransferTransaction`s that debit the operator account to any destination. The operator account's entire HBAR balance is at risk. Because the Hiero network validates only the cryptographic signature — not the originating IP or process — the attacker's transactions are indistinguishable from legitimate pinger traffic. Additionally, deployments that never set `HIERO_MIRROR_PINGER_OPERATOR_KEY` use the hardcoded default key, which is already public, requiring zero runtime access.

### Likelihood Explanation
Two independent paths exist:

1. **Zero-privilege path (hardcoded default):** Any developer or CI system that clones the repository and deploys without overriding the environment variable exposes a key that is already public. No container access is needed.
2. **Low-privilege shared-namespace path:** Kubernetes pods with `shareProcessNamespace: true` (a common observability/sidecar pattern) allow any container in the pod running as UID 1000 to read `/proc/<pid>/environ` of the pinger process. The Dockerfile sets `USER 1000:1000`, so a co-located sidecar running as the same UID has direct read access. This is a realistic, repeatable attack in multi-tenant or sidecar-heavy deployments.

### Recommendation
1. **Remove the hardcoded default key** from `config.go:40` immediately; fail fast if the key is not explicitly provided.
2. **Load the key from a secrets manager** (e.g., Kubernetes `Secret` mounted as a file, HashiCorp Vault agent, or a KMS-backed secret) rather than a plain environment variable.
3. **Zero the key string from memory** after `client.SetOperator` is called (use `[]byte` and explicit zeroing, or a `memguard`-style protected buffer).
4. **Avoid `shareProcessNamespace: true`** in the pod spec, or enforce strict UID isolation between the pinger and any sidecars.
5. Consider wrapping signing in a dedicated sidecar/process that holds the key and exposes only a signing RPC, so the key never appears in the main process environment.

### Proof of Concept
```
# Step 1 – extract key from a co-located container sharing the PID namespace
$ cat /proc/$(pgrep pinger)/environ | tr '\0' '\n' | grep OPERATOR_KEY
HIERO_MIRROR_PINGER_OPERATOR_KEY=<extracted_hex_key>

# Step 2 – replay the key with the Hiero Go SDK (attacker-controlled binary)
client, _ := hiero.ClientForName("mainnet")
opID, _  := hiero.AccountIDFromString("<operator_account_id>")
opKey, _ := hiero.PrivateKeyFromString("<extracted_hex_key>")
client.SetOperator(opID, opKey)

tx := hiero.NewTransferTransaction().
    AddHbarTransfer(opID, hiero.HbarFromTinybar(-1_000_000_000)).
    AddHbarTransfer(attackerID, hiero.HbarFromTinybar(1_000_000_000))
tx.Execute(client)   // succeeds — network sees a valid signature
```
