### Title
Uncached DB Lookup per EVM Address Request in `getAccountCryptoAllowances` Enables Sustained DB Resource Exhaustion

### Summary
Every call to `GET /api/v1/accounts/:idOrAliasOrEvmAddress/allowances/crypto` with a valid EVM address path parameter unconditionally triggers a database query against the `entity` table with no application-level caching or rate limiting in the REST module. An unprivileged external attacker can flood this endpoint with repeated requests using a valid EVM address, forcing a DB lookup on every single request and sustaining elevated DB CPU and I/O consumption well above baseline.

**Root cause and failed assumption:**

The `parseCached` function in `entityId.js` (lines 314–333) caches only the *parsing* of the string into an `EntityId` object — it caches the object with `evmAddress` set to the hex string, not the DB lookup result: [5](#0-4) 

Because `entityId.evmAddress` is non-null after parsing, the branch at line 122–124 of `entityService.js` always falls through to `getEntityIdFromEvmAddress()`, which always hits the database. There is no result cache anywhere in the REST service layer for EVM address → entity ID resolution.

**No rate limiting in the REST module:**

A grep across all `rest/**/*.js` files for `rateLimit`, `rate_limit`, `rateLimiter`, or `throttle` returns zero matches. The throttle/rate-limit infrastructure found in the codebase belongs exclusively to the `web3` module: [6](#0-5) 

The REST API route registration confirms no middleware is applied to this endpoint: [7](#0-6) 

### Impact Explanation

Each request with an EVM address path param issues `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` against the shared PostgreSQL database. With no caching and no application-level rate limiting, an attacker sending a sustained flood of requests (e.g., hundreds per second from a single machine or a small botnet) forces proportional DB query load. Because the `entity` table is shared across all mirror node REST endpoints, elevated I/O and CPU on this table degrades all other API consumers. The 30% above-baseline threshold is achievable without brute force: a single attacker with a modest connection pool can sustain hundreds of concurrent DB queries continuously.

### Likelihood Explanation

The attack requires zero privileges — no authentication, no API key, no special account. Any internet-accessible mirror node REST API is exposed. The attacker needs only a syntactically valid 40-hex-character EVM address (e.g., `0xdead000000000000000000000000000000000001`). The address does not need to exist in the database; a 404 response is returned after the DB lookup completes, meaning the DB is still hit on every miss. The attack is trivially scriptable with `curl`, `ab`, `wrk`, or any HTTP load tool and is fully repeatable.

### Recommendation

1. **Add a result cache in `getEntityIdFromEvmAddress()`**: Cache the `evmAddress → entityId` mapping (including negative/null results) using a short-lived in-process LRU cache (e.g., `quick-lru` already used in `entityId.js`) keyed on the hex EVM address string. A TTL of 30–60 seconds is sufficient to absorb burst traffic while remaining consistent with chain state.

2. **Add application-level rate limiting to the REST module**: Introduce a per-IP or global request rate limiter (e.g., `express-rate-limit`) on the REST Express app, mirroring the `ThrottleManagerImpl` pattern used in the `web3` module.

3. **Optionally cache negative results**: The current `entityFromEvmAddressQuery` returns nothing for unknown addresses. Caching `null` results with a short TTL prevents repeated DB hits for non-existent EVM addresses, which is the worst-case attacker scenario.

### Proof of Concept

**Preconditions:** A running mirror node REST API accessible over the network. No credentials required.

**Steps:**

```bash
# Use any syntactically valid EVM address (need not exist in DB)
EVM_ADDR="0xdead000000000000000000000000000000000001"
BASE_URL="https://<mirror-node-host>/api/v1"

# Flood the endpoint with concurrent requests
wrk -t 10 -c 200 -d 60s \
  "${BASE_URL}/accounts/${EVM_ADDR}/allowances/crypto"
```

**What happens:**
- Each of the 200 concurrent connections issues repeated HTTP GET requests.
- Each request triggers `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` with no cache.
- DB CPU and I/O rise proportionally to request rate.
- After ~60 seconds of sustained load, DB metrics show >30% above the 24-hour baseline.
- All other REST API endpoints sharing the same DB connection pool experience increased latency.
- The attacker receives HTTP 404 responses (entity not found) but the DB is hit on every request regardless.
