### Title
Unbounded Polling DoS via `limit=1` + Empty-Result Topic in `PollingTopicListener`

### Summary
When `PollingTopicListener` is active (listener type `POLL`), an unauthenticated attacker can subscribe to a topic with `limit=1` and a `startTime` in the distant past for a topic that has no matching messages. Because `PollingContext.count` never increments (no messages are emitted), `poll()` recomputes `pageSize=1` on every cycle and issues a DB query every 500 ms indefinitely. There is no idle-timeout, no rate limit, and no connection cap visible in the code, so many concurrent such subscriptions create sustained, unbounded database load.

### Finding Description

**Entry point** — `ConsensusController.subscribeTopic()` accepts any unauthenticated gRPC call and maps it to a `TopicMessageFilter`: [1](#0-0) 

No authentication or per-client connection cap is enforced here.

**Filter validation** — `TopicMessageFilter` only requires `startTime <= now` and `limit >= 0`: [2](#0-1) [3](#0-2) 

`limit=1` and `startTime=0` (epoch) both pass validation.

**Service layer** — `TopicMessageServiceImpl.subscribeTopic()` applies `.take(filter.getLimit())` on the outer flux: [4](#0-3) 

This `.take(1)` only fires when a message is actually emitted. If no messages exist, it never fires and the subscription stays open forever.

**Listener invocation** — `incomingMessages()` passes a derived filter with `limit=1` to `topicListener.listen()`: [5](#0-4) 

**Infinite poll loop** — `PollingTopicListener.listen()` repeats `Long.MAX_VALUE` times with no termination on empty results: [6](#0-5) 

**Per-cycle DB query** — `poll()` recomputes `limit` from `filter.getLimit() - context.getCount().get()`. With no messages emitted, `count` stays 0, so `limit` stays 1 and `pageSize` stays 1 every cycle: [7](#0-6) 

The default poll interval is 500 ms: [8](#0-7) 

Each open subscription therefore issues 2 DB queries per second, indefinitely.

### Impact Explanation
With N concurrent attacker connections, the database receives 2N queries/second with no upper bound. Even though each query uses `LIMIT 1`, the cumulative load from hundreds of connections is significant. The gRPC endpoint is publicly reachable with no authentication, so this is a straightforward resource-exhaustion DoS against the database tier. Legitimate subscribers may experience degraded query latency or connection pool exhaustion.

### Likelihood Explanation
The attack requires only a standard gRPC client (e.g., `grpcurl`), no credentials, and a single valid topic ID (or even a non-existent one if `checkTopicExists=false`). The attacker simply opens many long-lived streaming connections and keeps them alive. The attack is trivially scriptable and repeatable. [9](#0-8) 

When `checkTopicExists=false` (the fallback path), even a fabricated topic ID is accepted, removing the only prerequisite.

### Recommendation
1. **Add an idle/no-progress timeout**: terminate subscriptions that have received zero messages after a configurable window (e.g., 60 s).
2. **Cap concurrent subscriptions per IP or globally**: enforce a hard limit in `ConsensusController` or via a gRPC interceptor.
3. **Terminate the poll loop when remaining limit reaches zero**: in `PollingTopicListener.listen()`, add `.takeUntil` or check `context.getCount() >= originalLimit` and return `Flux.empty()` to stop repeating.
4. **Enforce a minimum `startTime` floor**: reject `startTime` values older than a configurable threshold (e.g., 30 days) to prevent full-history scans.

### Proof of Concept
```bash
# Open 500 concurrent subscriptions, each polling forever with limit=1
# and startTime at epoch (no messages will ever match epoch on a fresh topic)
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":99999999},"consensusStartTime":{"seconds":0},"limit":1}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Each connection issues SELECT ... FROM topic_message WHERE topic_id=? 
# AND consensus_timestamp >= 0 LIMIT 1  every 500ms, indefinitely.
# 500 connections = 1000 DB queries/second sustained with zero messages returned.
```

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
