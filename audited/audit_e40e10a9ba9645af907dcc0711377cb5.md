### Title
Unbounded DB Aggregation via Unauthenticated `/account/balance` Endpoint Enables Resource Exhaustion

### Summary
The `AccountBalance()` handler in `rosetta/app/services/account_service.go` accepts unauthenticated requests and passes attacker-controlled account IDs and block identifiers directly to `RetrieveBalanceAtBlock()`. This triggers an unbounded `SUM()` aggregation over the `crypto_transfer` table with no row limit, no query timeout specific to this path, and no application-level rate limiting. An attacker targeting high-activity accounts (e.g., treasury `0.0.2` or known exchange accounts) with historical block identifiers can force the database to scan and aggregate millions of rows per request, causing sustained resource exhaustion exceeding 30% above baseline.

### Finding Description

**Exact code path:**

`AccountBalance()` at `rosetta/app/services/account_service.go:41-79` accepts any `AccountIdentifier.Address` and optional `BlockIdentifier` without authentication or input restriction. [1](#0-0) 

It calls `RetrieveBalanceAtBlock()` at `rosetta/app/persistence/account.go:161-223`, which in turn calls `getBalanceChange()` at lines 313-338. [2](#0-1) 

`getBalanceChange()` executes the `balanceChangeBetween` SQL query:

```sql
select coalesce((
  select sum(amount) from crypto_transfer
  where
    consensus_timestamp > @start and
    consensus_timestamp <= @end and
    entity_id = @account_id and
    (errata is null or errata <> 'DELETE')
), 0) as value
``` [3](#0-2) 

**Root cause:** The `SUM()` subquery has no `LIMIT`, no row cap, and no query-level timeout. The range `(@start, @end]` spans from the latest balance snapshot timestamp to the attacker-supplied `consensusEnd`. On a live network, high-activity accounts accumulate hundreds of thousands to millions of `crypto_transfer` rows between snapshots. The attacker fully controls both the account ID and the block (and thus `consensusEnd`), allowing them to maximize the scan range.

**Why checks fail:**

1. The Traefik middleware chain (rate limiting, in-flight cap) is **disabled by default** — `global.middleware: false` in `charts/hedera-mirror-rosetta/values.yaml:95`, and the template guard `{{ if and .Values.global.middleware .Values.middleware }}` means no middleware objects are created unless explicitly opted in. [4](#0-3) [5](#0-4) 

2. Even when the middleware is enabled, the `retry: attempts: 3` entry in the chain causes each logical request to trigger up to **3 database queries**, amplifying load. [6](#0-5) 

3. The Rosetta Go application itself (`rosetta/main.go`) applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which throttle or limit requests. [7](#0-6) 

4. No authentication is required to call `/account/balance`; the Rosetta spec intentionally makes data endpoints public.

### Impact Explanation

Each request to `/account/balance` targeting a high-activity account with a historical block forces a full index-range scan and aggregation over `crypto_transfer` for that account. On Hedera mainnet, treasury account `0.0.2` and major exchange accounts have millions of transfers. A single such query can consume significant DB CPU and I/O for seconds. With no application-level concurrency cap, an attacker sending even a modest stream of concurrent requests (e.g., 10–20 parallel connections) can sustain DB CPU utilization well above 30% above baseline, degrading query performance for all other mirror node consumers (REST API, web3 API, other Rosetta endpoints).

### Likelihood Explanation

The attack requires zero privileges — only a valid account ID (publicly discoverable via the mirror node REST API or Hedera explorer) and a valid historical block hash or index. The attacker needs no special tooling beyond `curl` or any HTTP client. High-activity account IDs are well-known public information. The attack is trivially repeatable and scriptable. The default deployment disables the only external rate-limiting defense, making the majority of real-world deployments fully exposed.

### Recommendation

1. **Add a DB statement timeout** for the Rosetta DB connection pool, so that any single query exceeding a threshold (e.g., 5 seconds) is cancelled rather than allowed to run to completion.
2. **Add application-level concurrency limiting** (e.g., a semaphore or token bucket) inside the Go server, independent of the optional Traefik middleware.
3. **Enable the Traefik middleware by default** by changing `global.middleware` default to `true`, and tighten `inFlightReq.amount` and `rateLimit.average` to values appropriate for a public read-only API.
4. **Remove the `retry` middleware** from the chain, or scope it only to network-level errors, not application errors — retrying expensive queries multiplies DB load.
5. Consider **caching** balance results for a short TTL (e.g., 1–5 seconds) keyed on `(accountId, blockIdentifier)` to collapse repeated identical queries.

### Proof of Concept

```bash
# Step 1: Identify a high-activity account (e.g., treasury 0.0.2 or a known exchange)
ACCOUNT="0.0.2"

# Step 2: Get a historical block index far from the latest snapshot
# (maximizes the crypto_transfer scan range)
BLOCK_INDEX=1000000

# Step 3: Send concurrent requests with no authentication
for i in $(seq 1 20); do
  curl -s -X POST https://<rosetta-host>/account/balance \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"account_identifier\": {\"address\": \"$ACCOUNT\"},
      \"block_identifier\": {\"index\": $BLOCK_INDEX}
    }" &
done
wait
```

Each concurrent request forces the DB to execute `SELECT SUM(amount) FROM crypto_transfer WHERE entity_id = 2 AND consensus_timestamp > <snapshot_ts> AND consensus_timestamp <= <block_ts>` — a scan over potentially millions of rows — with no server-side defence to abort or throttle it.
