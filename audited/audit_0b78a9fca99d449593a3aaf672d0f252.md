### Title
Unbounded Base32 Alias Input Enables DoS via Unconstrained Memory Allocation and DB Parameter Binding

### Summary
The GraphQL `AccountInput.alias` field is validated only by `@Pattern(regexp: "^[A-Z2-7]+$")`, which enforces character set but imposes no upper bound on string length. An unauthenticated attacker can submit an arbitrarily large valid Base32 string that passes this check, causing `decodeBase32` to allocate a proportionally large byte array, which is then serialized and passed to the native JDBC query in `findByAlias`. Repeated concurrent requests can exhaust JVM heap memory.

### Finding Description

**Exact code path:**

1. `account.graphqls` line 90 — the only validation on `alias` is a character-set regex with no length bound: [1](#0-0) 

2. `AccountController.java` line 48 — the raw alias string is forwarded to the service with no additional check: [2](#0-1) 

3. `EntityServiceImpl.java` line 30 — `decodeBase32(alias)` is called unconditionally: [3](#0-2) 

4. `GraphQlUtils.java` lines 83–85 — `BASE32.decode(base32)` has no length guard; it allocates `ceil(inputLen * 5 / 8)` bytes: [4](#0-3) 

5. `EntityRepository.java` line 14 — the resulting `byte[]` is bound as a native query parameter with no size check: [5](#0-4) 

**Root cause:** The `@Pattern` directive validates only that every character belongs to the Base32 alphabet. It does not constrain string length. No `@Size`, `@Length`, or equivalent annotation exists anywhere in the call chain. No HTTP-level `max-request-size` is configured in the application resources.

**Contrast with sibling module:** `rest-java`'s `EntityIdAliasParameter` enforces `{40,70}` characters on the alias via its regex, bounding the decoded byte array to at most 44 bytes: [6](#0-5) 

The GraphQL module has no equivalent bound.

### Impact Explanation

Each request with an N-character alias allocates approximately `N * 5/8` bytes on the JVM heap. A 10 MB Base32 string (all valid characters, passes the regex) decodes to ~6.25 MB. With tens of concurrent requests, heap exhaustion is achievable on a default-configured JVM. Additionally, the JDBC driver must serialize the full byte array into the PostgreSQL wire protocol for every request, adding CPU and network overhead. The service is publicly accessible with no authentication required for GraphQL queries.

### Likelihood Explanation

The exploit requires no credentials, no special knowledge, and no race condition. Any HTTP client can POST a GraphQL query with a megabyte-scale alias string. The attack is trivially repeatable and scriptable. The only natural mitigating factor is a reverse-proxy or load-balancer request-body size limit, which is not configured in the application itself and may not be present in all deployments.

### Recommendation

1. Add a maximum-length constraint to the `alias` pattern in the schema, mirroring the `rest-java` module's `{40,70}` bound:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]{1,70}$")
   ```
2. Add a byte-length guard in `decodeBase32` (or in `getByAliasAndType`) that rejects decoded arrays exceeding the maximum valid alias size (e.g., 44 bytes for a 256-bit key).
3. Configure Spring Boot's `server.tomcat.max-http-form-post-size` / `spring.codec.max-in-memory-size` (or equivalent for the embedded server) to cap request body size at the application level.

### Proof of Concept

```bash
# Generate a 2 MB valid Base32 string (all 'A's pass ^[A-Z2-7]+$)
ALIAS=$(python3 -c "print('A' * 2_000_000)")

# Send to the public GraphQL endpoint
curl -s -X POST https://<host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\":\"{ account(input: { alias: \\\"$ALIAS\\\" }) { id } }\"}"
```

Repeat this request ~20–50 times concurrently (e.g., with `parallel` or `ab`) to drive heap usage toward exhaustion. Each request allocates ~1.25 MB on the heap before the DB call returns. No authentication is required.
