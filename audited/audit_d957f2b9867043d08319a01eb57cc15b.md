### Title
Cache Slot-Key Seeding Amplification: Unprivileged DB Batch Query Inflation in `findStorageBatch()`

### Summary
An unprivileged external caller can seed the per-contract `slotsPerContract` Caffeine cache (up to its `maximumSize=1500`) by issuing 1500 unique `eth_call`/`eth_getStorageAt` requests for a single contract. Because `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` (line 90) permanently registers every queried slot key, and line 91–103 unconditionally collects and queries **all** cached keys on every subsequent cache miss, any future request for that contract triggers a 1500-slot DB batch query instead of a single-slot query. The attacker sustains this state indefinitely by re-querying at ~6 req/s, well within the 500 RPS rate limit.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageBatch()` — `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 85–122.

**Root cause:**

At line 90, every call to `findStorageBatch()` unconditionally inserts the requested slot key into the per-contract `slotsPerContract` Caffeine cache:

```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);  // line 90
```

At line 91, the full key-set of that cache is immediately read:

```java
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();  // line 91
```

At line 103, **all** cached slot keys are passed to the DB in a single batch query:

```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);  // line 103
```

The `slotsPerContract` cache is configured `expireAfterAccess=5m,maximumSize=1500` (CacheProperties line 39). The outer `contractSlots` cache is `expireAfterAccess=5m,maximumSize=3000` (CacheProperties line 25). The `contractState` cache that would short-circuit the call is `expireAfterWrite=2s` (CacheProperties line 28), meaning it expires every 2 seconds, forcing `findStorageBatch()` to be called again.

**Failed assumption:** The design assumes the set of cached slot keys for a contract grows organically from legitimate traffic and stays small. There is no validation that the requesting caller has any relationship to the contract, no per-caller slot-key quota, and no cap on how many distinct slot keys a single caller can inject into the shared per-contract cache.

**Exploit flow:**

1. Attacker identifies any deployed contract (e.g., a popular DeFi contract).
2. Attacker sends 1500 `eth_call` or `eth_getStorageAt` requests, each with a distinct, attacker-chosen slot key (e.g., `keccak256(i)` for i in 0..1499). No authentication required.
3. Each request reaches `findStorageBatch()`, calls `putIfAbsent`, and registers the slot key in the per-contract cache. After 1500 requests the cache is at `maximumSize`.
4. The `contractState` cache (`expireAfterWrite=2s`) expires every 2 seconds. After expiry, any legitimate user querying **any** slot of that contract bypasses the `contractStateCache` check (lines 63–67) and enters `findStorageBatch()`.
5. `findStorageBatch()` collects all 1500 cached keys (line 91) and issues a 1500-slot `IN (...)` query to PostgreSQL (line 103) — a 1500× amplification of the DB work per request.
6. To maintain the seeded state, the attacker re-queries each of the 1500 slots once every ~4 minutes (~6 req/s), resetting the `expireAfterAccess` timer. This is far below the 500 RPS global limit.

**Why existing checks fail:**

- **Rate limiting (500 RPS global):** The seeding phase requires only 1500 requests (~3 seconds at 500 RPS). Maintenance requires ~6 req/s. Neither triggers the rate limiter.
- **`maximumSize=1500`:** Bounds the amplification factor but does not prevent the attack; 1500 slots per batch is still a severe amplification.
- **`expireAfterAccess=5m`:** Intended to naturally expire stale keys, but the attacker trivially resets the timer by periodically re-querying.
- **`enableBatchContractSlotCaching` flag:** Defaults to `true` (CacheProperties line 30); disabling it is a global kill-switch that removes the caching benefit for all users.
- **No per-contract or per-caller slot-key injection limit** exists anywhere in the code path.

### Impact Explanation

Every legitimate request for the targeted contract, after the 2-second `contractState` cache expiry, triggers a PostgreSQL `IN` query over 1500 slots instead of a single-slot lookup. On a busy node serving hundreds of requests per second for popular contracts, this translates to a sustained, multiplicative increase in DB CPU, I/O, and connection pressure. A single attacker targeting one popular contract can inflate DB query cost by up to 1500× per affected request. Targeting multiple contracts simultaneously (up to 3000 contracts fit in the outer `contractSlots` cache) multiplies the impact further. This directly satisfies the ≥30% resource consumption increase threshold without any brute-force volume.

### Likelihood Explanation

The attack requires zero privileges — only the ability to call the public `eth_call` or `eth_getStorageAt` JSON-RPC endpoints. The seeding phase is a one-time cost of 1500 requests completable in under 3 seconds. Maintenance is trivial (~6 req/s). The attacker needs no knowledge of actual contract storage layout; arbitrary slot keys (including non-existent ones) are accepted and cached. The attack is fully repeatable and can be automated with a simple script. Any public mirror node deployment is exposed.

### Recommendation

1. **Cap per-caller slot-key injection rate** using a per-IP or per-session counter before `putIfAbsent` is called.
2. **Separate attacker-supplied keys from organically-observed keys**: only register a slot key in the per-contract cache if it was returned by a prior DB query (i.e., it actually exists in `contract_state`). Keys for non-existent slots should not be cached.
3. **Bound batch size independently of cache size**: even if the cache holds 1500 keys, limit `findStorageBatch` to query at most N slots (e.g., 50) per invocation, prioritizing recently-accessed keys.
4. **Increase `contractState` TTL** or use `expireAfterAccess` instead of `expireAfterWrite` to reduce the frequency at which `findStorageBatch` is re-entered for already-seen slots.

### Proof of Concept

```python
import requests, json, hashlib

NODE_URL = "http://<mirror-node>:8545"
TARGET_CONTRACT = "0x<contract_address>"

def eth_get_storage(contract, slot_hex):
    payload = {
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [contract, slot_hex, "latest"], "id": 1
    }
    requests.post(NODE_URL, json=payload)

# Phase 1: Seed 1500 unique slot keys (completes in ~3s at 500 RPS)
for i in range(1500):
    slot = "0x" + hashlib.sha256(i.to_bytes(4, 'big')).hexdigest()
    eth_get_storage(TARGET_CONTRACT, slot)

# Phase 2: Maintain (run every 4 minutes to reset expireAfterAccess)
# Any legitimate user querying TARGET_CONTRACT now triggers a 1500-slot DB batch query
# instead of a single-slot query, on every request after the 2s contractState cache expires.
```

```java
    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
