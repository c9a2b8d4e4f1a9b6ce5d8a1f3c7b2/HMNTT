### Title
Cache Eviction DoS via Unbounded Unique Query Flooding in `CachedPreparsedDocumentProvider`

### Summary
The GraphQL preparsed-document cache uses the raw query string as its key with no rate limiting at the application layer, allowing an unauthenticated attacker to flood the fixed-size Caffeine cache (`maximumSize=1000`) with unique near-limit-length queries. Once the cache is saturated and continuously rotated, every legitimate query is evicted and must be fully re-parsed and re-validated on every request, causing sustained CPU overhead and degraded service availability.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```
The cache key is the verbatim query string. There is no normalization, canonicalization, or structural fingerprinting.

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15:
```java
private String query = "expireAfterWrite=1h,maximumSize=1000,recordStats";
```
The cache is bounded to exactly 1000 entries, expiring after 1 hour.

`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 33–38:
```java
b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000)
```
Queries up to 9999 characters and 1000 tokens are accepted without rejection.

**Root cause and failed assumption:**

The design assumes that the set of distinct query strings submitted to the service is small and stable (i.e., clients reuse the same queries). This assumption fails against an adversary who deliberately generates unique query strings. Because the cache key is the raw string and the cache size is only 1000 entries, an attacker needs to send just 1001 distinct queries to begin evicting legitimate entries. Caffeine's Window TinyLFU eviction policy will preferentially retain frequently-accessed entries, but an attacker who continuously rotates through fresh unique queries can keep the cache filled entirely with attacker-controlled entries, starving legitimate queries of cache hits indefinitely.

**Why existing checks fail:**

- The Helm chart (`charts/hedera-mirror-graphql/values.yaml`, lines 138–142) configures only `inFlightReq: amount: 5` per IP — this limits *concurrent* in-flight requests, not the *total request rate*. An attacker can send thousands of sequential requests from a single IP, each completing before the next begins, and never trigger this limit.
- Unlike the Rosetta component (which adds `rateLimit: average: 10`), the GraphQL Helm chart has **no `rateLimit` middleware**.
- The `web3` module has a `ThrottleConfiguration` / `ThrottleManagerImpl` with bucket4j rate limiting, but **no equivalent exists in the `graphql` module**.
- The `retry: attempts: 3` middleware in the GraphQL Helm chart amplifies the attack: each attacker request may be retried up to 3 times, tripling cache-fill throughput.
- Parser limits (`maxTokens=1000`, `maxCharacters=10000`) bound the cost of each individual parse but do not prevent the cache-eviction pattern — an attacker only needs 1000 structurally distinct queries, each within limits.

### Impact Explanation

Every legitimate GraphQL query that is evicted from the cache must undergo full lexing, parsing, AST construction, and validation on every subsequent request. Under a sustained attack, the effective cache hit rate for legitimate traffic drops to zero. The parse-and-validate path for a 9999-character / 1000-token query is non-trivial CPU work executed synchronously in the request path. At scale, this degrades throughput and increases latency for all users of the GraphQL API. The `expireAfterWrite=1h` TTL means evicted entries do not naturally re-enter the cache for an hour after the attack stops, prolonging recovery.

### Likelihood Explanation

No authentication or privilege is required. The GraphQL endpoint (`/graphql/alpha`) is publicly accessible. Generating 1000+ unique syntactically valid queries is trivial — an attacker can use a script that varies a single alias name or numeric literal across requests. The `inFlightReq: amount: 5` per-IP limit means the attacker needs only 5 concurrent connections (or sequential requests from a single connection) to sustain the attack. A single commodity machine can easily maintain this. The attack is repeatable and stateless — the attacker does not need to maintain session state or credentials.

### Recommendation

1. **Add a `rateLimit` Traefik middleware** to the GraphQL Helm chart (mirroring the Rosetta configuration: `rateLimit: average: 10, sourceCriterion: requestHost: true`).
2. **Add application-level rate limiting** in the `graphql` module analogous to the `ThrottleConfiguration` in `web3`, using bucket4j to cap requests per second per IP before the cache is consulted.
3. **Increase `maximumSize`** or switch to a **weight-based eviction** policy (Caffeine supports `maximumWeight`) that accounts for query string length, making large-query flooding more expensive.
4. **Normalize/hash the cache key** before storage (e.g., strip insignificant whitespace, or store by a structural hash) to reduce the attacker's ability to generate trivially unique keys from semantically equivalent queries.
5. **Remove the `retry` middleware** for the GraphQL component, or scope it only to network-level errors, to avoid amplifying attacker request volume.

### Proof of Concept

```bash
#!/bin/bash
# Fill the 1000-entry cache with unique attacker queries, then keep rotating
ENDPOINT="http://<graphql-host>/graphql/alpha"
COUNTER=0
while true; do
  # Each query is unique via a distinct alias; padded to ~9999 chars with a long alias
  ALIAS=$(printf 'a%.0s' $(seq 1 9950))_${COUNTER}
  QUERY="{${ALIAS}: account(input:{entityId:{shard:0,realm:0,num:${COUNTER}}}){balance}}"
  curl -s -X POST "$ENDPOINT" \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"$QUERY\"}" &
  COUNTER=$((COUNTER + 1))
  # Stay within inFlightReq=5 per IP
  if (( COUNTER % 5 == 0 )); then wait; fi
done
```

After ~1000 iterations the cache is full of attacker entries. Legitimate queries (e.g., `{ account(input:{entityId:{shard:0,realm:0,num:2}}){balance} }`) will be evicted on each new attacker batch and must be re-parsed on every legitimate request for the duration of the attack.
