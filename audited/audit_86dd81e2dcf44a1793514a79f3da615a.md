### Title
Unbounded Per-Operation DB Amplification in `updateOperationAccountAlias()` via Unauthenticated `/block` Endpoint

### Summary
The `updateOperationAccountAlias()` function in `rosetta/app/services/block_service.go` issues one `GetAccountAlias()` database query per operation whose `accountId` is absent from the LRU entity cache. The existing `maxTransactionsInBlock` guard caps only the number of transactions returned, not the number of operations within each transaction. Because the `/block` endpoint requires no authentication and has no application-level rate limiting, any external caller can repeatedly request blocks containing transactions with large numbers of unique account IDs, driving proportional DB query amplification.

### Finding Description

**Exact code path:**

`Block()` (line 47) fetches all transactions for a block, truncates to `maxTransactionsInBlock` transactions (line 61–66), then passes the remaining slice to `updateOperationAccountAlias()` (line 69):

```go
// block_service.go:61-69
if len(block.Transactions) > s.maxTransactionsInBlock {
    // truncates transaction count only
    block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
}
if err = s.updateOperationAccountAlias(ctx, block.Transactions...); err != nil {
```

Inside `updateOperationAccountAlias()` (lines 104–130), every operation in every transaction is iterated unconditionally. For each operation whose `accountId.GetId()` is not in the LRU cache, a synchronous DB call is issued:

```go
// block_service.go:110-123
for index := range operations {
    accountId := operations[index].AccountId
    if cached, found = s.entityCache.Get(accountId.GetId()); !found {
        result, err := s.accountRepo.GetAccountAlias(ctx, accountId)  // DB call
        ...
        s.entityCache.Set(result.GetId(), result)
    }
    operations[index].AccountId = cached
}
```

`GetAccountAlias()` executes `selectCryptoEntityWithAliasById` (`"select alias, id from entity where id = @id"`) against the mirror node DB for every cache miss.

**Root cause:** The failed assumption is that `maxTransactionsInBlock` bounds the total work per request. It does not — it bounds only the transaction count. A single transaction can carry an arbitrarily large number of operations (crypto transfers, staking reward payouts, etc.), each generating an independent DB query on cache miss.

**Why the cache is insufficient:** The LRU cache (default `maxSize = 524288`) is shared across all requests. An attacker can thrash it by cycling through blocks with many distinct account IDs, causing continuous evictions and forcing repeated DB queries for the same accounts on subsequent requests.

**No authentication:** The Rosetta router (`rosetta/main.go:111–119`) registers the `blockAPIController` with no auth middleware. The Traefik rate-limit middleware (`average: 10`, `inFlightReq: 5`) is optional infrastructure (`middleware: false` by default in the Helm chart) and is not enforced at the application layer.

### Impact Explanation
An attacker can saturate the DB connection pool (default `maxOpenConnections: 100`) by sending concurrent `/block` requests targeting blocks with many unique account IDs. Each request with N uncached unique account IDs generates N synchronous DB queries. With `maxTransactionsInBlock` transactions each having M operations, a single request can produce up to `maxTransactionsInBlock × M` DB queries. This degrades DB response times for all mirror node consumers (REST API, gRPC, etc.), constituting a griefing/DoS with no economic damage to network participants — matching the stated Medium severity classification.

### Likelihood Explanation
On mainnet or testnet, blocks containing transactions with hundreds of operations (large token airdrops, batch HBAR distributions) naturally exist and are publicly queryable. An attacker needs only a network connection and knowledge of a block index or hash containing a high-operation transaction. No credentials, tokens, or privileged access are required. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add a per-request operation cap** inside `updateOperationAccountAlias()` or before calling it, rejecting or truncating requests where the total operation count across all transactions exceeds a configurable threshold.
2. **Deduplicate before querying**: collect all unique `accountId` values across all operations first, batch-resolve aliases (ideally with a single `IN`-clause query), then apply results — eliminating the N-queries-per-request pattern.
3. **Enforce application-level rate limiting** within the Go server (e.g., `golang.org/x/time/rate`) rather than relying solely on optional infrastructure middleware.
4. **Add a `maxOperationsInBlock` configuration knob** analogous to `maxTransactionsInBlock`.

### Proof of Concept
1. Identify a block on the target network containing a transaction with a large number of crypto transfers (e.g., a token airdrop to 500+ unique accounts). This is observable via the mirror node REST API (`/api/v1/transactions`).
2. Note the block index (`block_identifier.index`).
3. Send concurrent POST requests to `/block`:
   ```json
   { "network_identifier": {...}, "block_identifier": { "index": <TARGET_BLOCK> } }
   ```
4. On the first request (cold cache), `updateOperationAccountAlias()` issues one `SELECT alias, id FROM entity WHERE id = @id` query per unique account ID in the block's operations.
5. To force repeated cache misses, interleave requests for many other blocks to evict the LRU entries, then re-request the target block.
6. Observe DB query rate and connection pool utilization spike proportionally to the number of operations in the targeted block, degrading overall mirror node DB performance.
