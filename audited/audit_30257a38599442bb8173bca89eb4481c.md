### Title
Unbounded `findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc` Query Enables Global Opcode Endpoint Monopolization via Child Transaction Amplification

### Summary
`TransactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc()` issues an unbounded SQL query returning every transaction row sharing a `(payerAccountId, validStartNs)` pair — which includes all child transactions (nonce > 0) of a parent contract call — yet the caller immediately discards all but the first result via `getFirst()`. The opcode endpoint's rate limiter is a single global token bucket (default: 1 req/sec), not per-IP, so an unprivileged attacker can permanently monopolize the entire endpoint while forcing the database to materialize arbitrarily large result sets on every allowed request.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/TransactionRepository.java`, lines 14–15:
```java
List<Transaction> findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
        EntityId payerAccountId, long validStartNs);
```
Spring Data JPA derives a query with no `LIMIT` clause. Every transaction row with the given `(payer_account_id, valid_start_ns)` is fetched and hydrated into `Transaction` objects.

`web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`, lines 100–109:
```java
final var transactionList =
        transactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
                payerAccountId, validStartNs);
if (transactionList.isEmpty()) {
    throw new EntityNotFoundException(...);
}
final var parentTransaction = transactionList.getFirst();   // only first is ever used
```
The entire list is loaded into heap; only index 0 is consumed.

**Root cause:** In Hedera's data model, a parent contract call and all its child/internal transactions share the same `(payer_account_id, valid_start_ns)` pair, differentiated only by `nonce`. A complex contract that spawns N internal transactions produces N rows for the same key. The query fetches all N rows; the code needs exactly 1.

**Rate-limiter insufficiency:**

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleProperties.java`, line 29:
```java
private long opcodeRequestsPerSecond = 1;
```
`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 52–55:
```java
public void throttleOpcodeRequest() {
    if (!opcodeRateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```
`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 47–55 — the `opcodeRateLimitBucket` is a single application-wide `Bucket` instance. There is no per-IP, per-account, or per-session partitioning. One attacker consuming 1 token/sec exhausts 100% of the allowed opcode capacity, blocking all other users. No caching exists on the repository method.

### Impact Explanation

- **DB amplification:** A single allowed request targeting a transaction with N child rows forces the database to execute a full index scan on `(payer_account_id, valid_start_ns)`, transfer N rows over the wire, and allocate N `Transaction` objects on the JVM heap — when a `LIMIT 1` query would suffice. For contracts with hundreds of internal calls (common in DeFi-style contracts on Hedera), N can be large.
- **Endpoint monopolization:** Because the bucket is global and defaults to 1 req/sec, a single attacker sending one request per second permanently occupies the entire opcode debug service, denying it to all legitimate users. The attacker pays no cost beyond sending one HTTP request per second.
- **Severity:** Medium — no direct fund loss, but complete denial of the smart contract debug service to all users and measurable unnecessary DB load on each request.

### Likelihood Explanation

The attack requires zero privileges: the opcode endpoint (`GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes`) is publicly reachable. The attacker only needs to know one valid `transactionId` for a contract that produced child transactions (trivially discoverable from the public mirror node REST API). Sending one request per second is trivially automatable with `curl` or any HTTP client. The attack is indefinitely repeatable and requires no special knowledge of the system internals.

### Recommendation

1. **Add `LIMIT 1` to the query** — replace the derived query with a named query or `@Query` annotation:
   ```java
   @Query("SELECT t FROM Transaction t WHERE t.payerAccountId = :payerAccountId " +
          "AND t.validStartNs = :validStartNs ORDER BY t.consensusTimestamp ASC LIMIT 1")
   Optional<Transaction> findFirstByPayerAccountIdAndValidStartNs(
           @Param("payerAccountId") EntityId payerAccountId,
           @Param("validStartNs") long validStartNs);
   ```
   This eliminates the unbounded fetch entirely.

2. **Make the rate limiter per-IP** — partition the `opcodeRateLimitBucket` by client IP (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed on `X-Forwarded-For` / `REMOTE_ADDR`) so one attacker cannot exhaust the global quota.

3. **Add response caching** — cache the result of the repository call (e.g., with Spring's `@Cacheable`) keyed on `(payerAccountId, validStartNs)` to avoid repeated DB hits for the same input.

### Proof of Concept

**Preconditions:**
- Mirror node web3 service is running with default config (`opcodeRequestsPerSecond=1`).
- A contract transaction exists with many child transactions, e.g., transaction ID `0.0.1234-1700000000-000000000`.

**Steps:**
```bash
# Step 1: Identify a transaction with many child transactions via the public REST API
curl "https://<mirror-node>/api/v1/transactions/0.0.1234-1700000000-000000000"
# Observe multiple nonce entries sharing the same valid_start_timestamp

# Step 2: Continuously send opcode debug requests at 1 req/sec (saturating the global bucket)
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node-web3>/api/v1/contracts/results/0.0.1234-1700000000-000000000/opcodes" \
    -o /dev/null
  sleep 1
done

# Result: The attacker consumes 100% of the global opcode rate limit.
# All other users receive HTTP 429 Too Many Requests.
# Each request causes the DB to load all N child transaction rows unnecessarily.
```

**Observed result:** Legitimate users are denied access to the opcode endpoint; DB query load is amplified by N (number of child transactions) per request compared to what a `LIMIT 1` query would require.
