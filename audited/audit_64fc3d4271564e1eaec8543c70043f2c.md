### Title
Unauthenticated Access to EVM Opcode Trace Endpoint Exposes Full Execution State of Any Historical Transaction

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` has zero authentication or authorization controls. When the feature flag is enabled, any unauthenticated external user can submit any historical `transactionIdOrHash` — including a governance vote transaction — and receive a full per-opcode EVM trace containing stack values, memory contents, and all storage slot reads/writes. This exposes the exact storage keys and values manipulated during the vote, including vote weight and voter-derived data that the governance contract stores.

### Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, lines 52–68 — `getContractOpcodes()`:

```java
@GetMapping(value = "/{transactionIdOrHash}/opcodes")
OpcodesResponse getContractOpcodes(
        @PathVariable TransactionIdOrHashParameter transactionIdOrHash,
        @RequestParam(required = false, defaultValue = "true") boolean stack,
        @RequestParam(required = false, defaultValue = "false") boolean memory,
        @RequestParam(required = false, defaultValue = "false") boolean storage,
        @RequestHeader(value = HttpHeaders.ACCEPT_ENCODING) String acceptEncoding) {
    if (properties.isEnabled()) {
        validateAcceptEncodingHeader(acceptEncoding);
        throttleManager.throttleOpcodeRequest();
        final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
        return opcodeService.processOpcodeCall(request);
    }
    throw new ResponseStatusException(HttpStatus.NOT_FOUND);
}
```

The only guards present are:
1. `properties.isEnabled()` — a feature flag (`hiero.mirror.web3.opcode.tracer.enabled`, default `false`). When an operator enables this for debugging/tooling, the gate opens entirely.
2. `validateAcceptEncodingHeader()` — requires `Accept-Encoding: gzip`. Trivially satisfied.
3. `throttleManager.throttleOpcodeRequest()` — a **global** bucket of 1 req/sec (default), not per-IP or per-user.

A grep across the entire `web3/src/main/java/` tree confirms **zero** Spring Security annotations (`@PreAuthorize`, `@Secured`, `SecurityFilterChain`, `HttpSecurity`) exist anywhere in the web3 module.

**Root cause:** The failed assumption is that the feature flag alone acts as an access control boundary. It does not — it is a deployment toggle, not an authorization check. Once enabled, the endpoint is fully public.

**What the trace exposes:**

`AbstractOpcodeTracer.captureStorage()` (lines 75–128) iterates every `StorageAccess` from `getTxStorageUsage(true).accesses()` and returns a map of `{slotKey → writtenValue}` for every storage write during the replayed transaction. `captureStack()` (lines 60–73) returns every EVM stack item at every opcode step.

For a governance vote transaction (e.g., a `castVote(proposalId, support)` call):
- The voter's token balance (vote weight) is read via `SLOAD` from the governance token's storage and pushed onto the stack — captured verbatim in the stack trace.
- The vote record (`proposalVotes[proposalId][voterAddress] = weight`) is written via `SSTORE` — captured verbatim in the storage map with the exact slot key (which encodes the voter address) and value (vote weight).
- The voter's address appears on the stack as `CALLER` is pushed and used to compute the storage slot key.

**Exploit flow:**
1. Attacker observes a governance vote transaction on-chain (transaction hashes are public).
2. Attacker sends: `GET /api/v1/contracts/results/{txHash}/opcodes?stack=true&storage=true` with `Accept-Encoding: gzip`.
3. The service re-executes the transaction against historical state and returns the full opcode trace.
4. From the `storage` map in the response, the attacker reads the exact slot key (which encodes the voter address via Solidity's `keccak256(abi.encode(voter, mappingSlot))`) and the written value (vote weight).
5. From the `stack` entries, the attacker directly reads intermediate values including the voter address and weight as they appear on the EVM stack during `SLOAD`/`SSTORE` sequences.

### Impact Explanation
Any historical governance vote transaction can be replayed with full EVM tracing by an unauthenticated attacker. The response reveals:
- **Exact vote weight** (token balance at vote time) per voter, at per-opcode granularity.
- **Storage slot keys** that encode voter addresses, allowing reverse-mapping of voter identity to vote weight even for contracts that do not emit events.
- **Intermediate computation values** on the stack that are not exposed by any other public endpoint (e.g., `contractStateChanges` in other REST endpoints only show final state deltas, not per-opcode stack state).

This breaks the confidentiality assumption of governance protocols that rely on the absence of a convenient per-voter weight oracle. It also applies to any contract storing sensitive state (e.g., sealed bids, private balances before reveal).

### Likelihood Explanation
- **Precondition:** The operator has set `hiero.mirror.web3.opcode.tracer.enabled=true`. This is the intended production configuration for any deployment that exposes the endpoint.
- **Attacker capability:** Zero — no account, no token, no credentials required. Any HTTP client suffices.
- **Repeatability:** The global rate limit of 1 req/sec is per-server, not per-IP. An attacker can enumerate all governance vote transactions at 1 tx/sec continuously. The rate limit is also trivially bypassed by targeting multiple mirror node instances behind a load balancer.
- **Transaction discovery:** Governance vote transaction hashes are publicly visible via `/api/v1/contracts/results` and `/api/v1/transactions`.

### Recommendation
1. **Add authentication to the endpoint.** Require an operator-configured API key or IP allowlist before the opcode trace is served. The feature flag is not a substitute for access control.
2. **Add per-IP rate limiting** in addition to the global bucket, so the 1 req/sec limit cannot be trivially bypassed across distributed clients.
3. **Consider redacting sensitive fields** from the trace response (e.g., omit raw stack values containing addresses when `storage=true`) or require explicit elevated authorization for `storage=true` requests.
4. Concretely: add a Spring Security `SecurityFilterChain` bean in the web3 module that requires authentication for `GET /api/v1/contracts/results/*/opcodes`, or add an `@PreAuthorize` annotation on `getContractOpcodes()`.

### Proof of Concept
**Preconditions:** Mirror node running with `hiero.mirror.web3.opcode.tracer.enabled=true` (any deployment that exposes the endpoint).

**Step 1 — Discover a governance vote transaction hash:**
```
GET /api/v1/contracts/results?limit=25
```
Identify a transaction hash for a `castVote` call to a known governance contract.

**Step 2 — Replay with full trace (no credentials required):**
```
GET /api/v1/contracts/results/0x<txHash>/opcodes?stack=true&storage=true
Accept-Encoding: gzip
```

**Step 3 — Parse the response:**
```json
{
  "opcodes": [
    {
      "op": "SSTORE",
      "stack": ["0x<voterAddress>", "0x<voteWeight>", ...],
      "storage": {
        "0x<keccak256(voterAddress||mappingSlot)>": "0x<voteWeight>"
      }
    }
  ]
}
```

The `storage` map directly yields the vote weight written for the voter's slot. The `stack` entries yield the voter address and weight as raw hex values at the moment of the `SSTORE` opcode. No authentication, no special tooling, no privileges required.
