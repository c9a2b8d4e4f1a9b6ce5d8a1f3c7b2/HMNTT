### Title
Future mtime on `/tmp/alive` Permanently Spoofs Liveness Healthcheck

### Summary
The liveness healthcheck in `pinger/cmd/healthcheck/main.go` computes heartbeat age as `time.Since(fi.ModTime())`, which returns a **negative** `time.Duration` when the file's mtime is set to a future timestamp. Because the only staleness guard is `age > 2*time.Minute`, a negative age always passes that check, causing the healthcheck to unconditionally report the process as alive. Any unprivileged local user can exploit this by creating `/tmp/alive` with a far-future mtime, permanently suppressing liveness failure regardless of actual pinger state.

### Finding Description
**File:** `pinger/cmd/healthcheck/main.go`
**Function:** `main()`
**Lines:** 27–39

```go
fi, err := os.Stat("/tmp/alive")   // line 27
if err != nil {
    fmt.Fprintln(os.Stderr, err)
    os.Exit(1)
}

age := time.Since(fi.ModTime())    // line 34  — can be negative
if age > 2*time.Minute {           // line 35  — false when age < 0
    fmt.Fprintln(os.Stderr, "heartbeat stale")
    os.Exit(1)
}
os.Exit(0)                         // line 39  — always reached
```

**Root cause:** `time.Since(t)` is defined as `time.Now().Sub(t)`. When `t` is in the future, the result is a negative `time.Duration` (an `int64` nanosecond count). The guard `age > 2*time.Minute` compares a large negative integer against a positive constant — the condition is always `false`, so the process exits 0 (healthy).

**Failed assumption:** The code assumes `fi.ModTime()` is always ≤ `time.Now()`. It never validates that the age is non-negative.

**Exploit flow:**
1. Attacker (unprivileged) creates `/tmp/alive` with a far-future mtime:
   ```
   touch -d "2099-01-01" /tmp/alive
   ```
   `/tmp` is world-writable on all standard Linux systems; no privileges required.
2. The real pinger stops writing heartbeats (or is killed).
3. Every subsequent healthcheck invocation computes a large negative `age`, skips the staleness branch, and exits 0.
4. The container/pod orchestrator (e.g., Kubernetes liveness probe) never sees a failure and never restarts the dead pinger.

**Existing checks reviewed:**
- The `os.Stat` error check (line 28) only catches a missing or inaccessible file — it does not validate mtime direction.
- There is no lower-bound check (`age < 0`) anywhere in the function. [1](#0-0) 

### Impact Explanation
A dead pinger is never restarted by the orchestrator because the liveness probe permanently returns success. Any downstream service relying on the pinger (e.g., network reachability monitoring, alerting) silently fails. The attacker achieves a persistent denial-of-monitoring condition with a single, non-privileged shell command.

### Likelihood Explanation
Preconditions are minimal: local shell access to the container/host and write permission to `/tmp` (world-writable by default). No elevated privileges, no special tools, and no race conditions are required. The attack is trivially repeatable — even if the file is deleted, the attacker can recreate it instantly. In containerized environments where multiple processes share a pod, any compromised sidecar or init container can perform this attack.

### Recommendation
Add an explicit non-negative check before the staleness comparison:

```go
age := time.Since(fi.ModTime())
if age < 0 || age > 2*time.Minute {
    fmt.Fprintln(os.Stderr, "heartbeat stale or invalid mtime")
    os.Exit(1)
}
```

Additionally, consider writing the heartbeat file from a process running as a dedicated user and restricting `/tmp/alive` ownership/permissions so unprivileged users cannot create or modify it (e.g., use a private directory owned by the pinger user instead of `/tmp`).

### Proof of Concept
```bash
# 1. Set future mtime (no root required)
touch -d "2099-01-01 00:00:00" /tmp/alive

# 2. Kill the real pinger
pkill pinger

# 3. Run the healthcheck repeatedly — always exits 0
while true; do
    ./healthcheck live
    echo "exit code: $?"
    sleep 5
done
# Output: exit code: 0  (forever, even though pinger is dead)
```
