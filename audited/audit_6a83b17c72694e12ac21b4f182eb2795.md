### Title
Global Rate Limit Bucket Starvation — Unprivileged Single-Source DoS

### Summary
`rateLimitBucket` in `ThrottleConfiguration.java` is a single application-wide Bucket4j token bucket shared across all callers with no per-IP or per-user partitioning. An unprivileged attacker who continuously sends requests at the configured ceiling (default 500 req/s) exhausts the entire global token supply, causing every concurrent legitimate user to receive HTTP 429 responses for the duration of the attack. No existing check in the throttle stack identifies or isolates the source of requests.

### Finding Description
**Code path:**

- `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, `rateLimitBucket()`, lines 24–32: a single `Bucket` is constructed with `capacity = requestsPerSecond` (default 500) and a greedy refill of the same rate per second. This bean is a Spring singleton — one instance for the entire JVM.
- `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, line 38: every inbound request calls `rateLimitBucket.tryConsume(1)` against this shared singleton. If the bucket is empty, a `ThrottleException` is thrown immediately (line 39), returning HTTP 429 to the caller.

**Root cause:** The design assumes the 500-token budget will be shared fairly across all callers. There is no per-source-IP bucket, no per-authenticated-user bucket, and no connection-level admission control in the application layer. The `RequestFilter` / `RequestProperties` subsystem (`RequestFilter.java`, `FilterField` enum, lines 39–46) can filter on `BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, and `VALUE` — none of which is a network-layer source address — so it cannot be configured to isolate a flooding source.

**Exploit flow:**
1. Attacker opens persistent HTTP connections (or uses keep-alive) to the web3 endpoint (e.g., `POST /api/v1/contracts/call`).
2. Attacker dispatches exactly 500 minimal-gas requests per second (e.g., `gas=21000`, minimal payload). Each call consumes one token.
3. The global bucket is continuously drained to zero; the greedy refill at 500 tokens/s is immediately re-consumed by the attacker.
4. Any request from a legitimate user that arrives while the bucket is empty receives HTTP 429 `REQUEST_PER_SECOND_LIMIT_EXCEEDED`.
5. The attacker sustains this indefinitely with no authentication, no API key, and no special network position.

**Why existing checks fail:** The `gasLimitBucket` check (line 40) is only reached after the rate-limit check passes; it provides no relief. The `RequestProperties` filter list defaults to `List.of()` (empty), and even when configured, it cannot match on source IP. There is no Spring Security filter, servlet filter, or reverse-proxy rule visible in the application configuration that enforces per-IP limits.

### Impact Explanation
A sustained 500 req/s flood from a single unauthenticated source renders the entire web3 JSON-RPC surface (contract calls, gas estimates, opcode traces) unavailable to all other users. The impact is a complete, sustained denial of service at the application layer. Severity is **High**: the service's primary function is blocked for all legitimate users for as long as the attacker maintains the flood rate, with no self-healing unless the attacker stops or the operator manually reconfigures the limit.

### Likelihood Explanation
Sending 500 HTTP requests per second from a single commodity machine is trivially achievable with standard tools (`wrk`, `hey`, `ab`, or a simple multi-threaded script). No credentials, API keys, or special network access are required — only reachability to the public endpoint. The attack is repeatable, automatable, and cheap to sustain. An attacker motivated to disrupt the Hiero mirror node's EVM query service (e.g., to prevent contract state reads during a time-sensitive operation) has a straightforward, low-cost path.

### Recommendation
1. **Per-IP rate limiting at the application layer:** Replace (or supplement) the single global bucket with a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `HttpServletRequest.getRemoteAddr()` or a trusted `X-Forwarded-For` header). Each IP gets its own sub-budget (e.g., 50 req/s), and the global bucket acts as a secondary ceiling.
2. **Bucket4j with a proxy cache or Caffeine cache:** Use Bucket4j's `ProxyManager` API with a local Caffeine cache to manage per-key buckets efficiently without unbounded memory growth.
3. **Add `IP` to `FilterField`:** Extend `RequestFilter.FilterField` to include the source IP so operators can configure `REJECT` or `THROTTLE` rules targeting specific addresses via configuration without a code deploy.
4. **Upstream enforcement:** Deploy a WAF or ingress-level rate limiter (e.g., nginx `limit_req`, Envoy rate-limit filter) as a defense-in-depth layer, but do not rely on it as the sole control since the application-layer gap remains.

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg/wrk) and network access to the web3 endpoint
# Adjust HOST and PORT as appropriate

cat > payload.lua << 'EOF'
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"to":"0x0000000000000000000000000000000000000167","gas":21000,"estimate":false}'
EOF

# Flood at 500 req/s from a single source — exhausts the global rateLimitBucket
wrk -t4 -c50 -d60s -R500 -s payload.lua http://HOST:PORT/api/v1/contracts/call

# In a separate terminal, observe legitimate requests receiving HTTP 429:
curl -s -o /dev/null -w "%{http_code}" \
  -X POST http://HOST:PORT/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000167","gas":21000,"estimate":false}'
# Expected output while flood is active: 429
```
