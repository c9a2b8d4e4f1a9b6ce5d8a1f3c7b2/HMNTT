### Title
DB Connection Pool Exhaustion via Unauthenticated High-Volume Requests to `/accounts/:idOrAliasOrEvmAddress/allowances/tokens`

### Summary
The `getAccountTokenAllowances()` handler in `rest/controllers/tokenAllowanceController.js` issues database queries through a shared, finite-size connection pool (`BaseService.pool()`) with no per-IP or global rate limiting in place. An unprivileged attacker can flood the endpoint with concurrent requests using varied query parameters to bypass the response cache, exhausting the pool and degrading all mirror node REST API processing capacity.

### Finding Description

**Exact code path:**

`getAccountTokenAllowances()` (lines 68–81 of `rest/controllers/tokenAllowanceController.js`) calls:
1. `EntityService.getEncodedId()` — acquires a pool connection
2. `TokenAllowanceService.getAccountTokenAllowances(query)` (line 72) → `BaseService.getRows()` (line 55–57 of `rest/service/baseService.js`) → `this.pool().queryQuietly(query, params)` (line 56)

`BaseService.pool()` (lines 96–98 of `rest/service/baseService.js`) returns a single global `pool` object shared across all services and all concurrent requests.

**Root cause:**

The pool is configured with a finite `maxConnections` (parsed in `parseDbPoolConfig()`, lines 137–148 of `rest/config.js`). There is no rate-limiting middleware anywhere in the REST middleware stack. The middleware index (`rest/middleware/index.js`) exports only: `authHandler`, `handleError`, `openApiValidator`, `serveSwaggerDocs`, `requestLogger`, `requestQueryParser`, and cache handlers — no rate limiter. A grep across all `rest/**/*.js` for `rateLimit`, `throttle`, or `concurren` returns zero production hits.

**Cache bypass:**

The response cache key is `MD5(req.originalUrl)` (line 152 of `rest/middleware/responseCacheHandler.js`). An attacker trivially bypasses it by varying any accepted query parameter (`spender.id`, `token.id`, `limit`, `order`) across requests, generating unique cache keys and forcing a fresh DB query for each.

**Exploit flow:**
1. Attacker picks any valid account ID (publicly visible on-chain).
2. Sends N concurrent HTTP GET requests to `/api/v1/accounts/<id>/allowances/tokens?limit=<1..25>&spender.id=gt:<incrementing_value>` — each request has a unique URL, bypassing the cache.
3. Each request acquires one or two pool connections for the duration of the query.
4. When in-flight requests ≥ `maxConnections`, the pool is exhausted; all subsequent requests (across all endpoints) stall until `connectionTimeout` expires or a slot frees.
5. Sustained flooding keeps the pool perpetually exhausted.

### Impact Explanation

The global pool is shared by every REST API endpoint. Pool exhaustion does not isolate to the token-allowance endpoint — it blocks all concurrent DB-backed requests across the mirror node REST service. With a typical `maxConnections` of 10–20 (common pg-pool defaults), a modest flood of ~20–50 concurrent requests is sufficient to saturate the pool. This constitutes degradation of ≥30% of mirror node processing capacity, matching the stated severity threshold. No data is modified; the impact is availability (DoS).

### Likelihood Explanation

- **No privileges required**: The endpoint is a public read API; no authentication is needed.
- **Trivial to execute**: A single `ab -n 10000 -c 50` or equivalent tool with incrementing query parameters is sufficient.
- **Repeatable**: The attack can be sustained indefinitely from a single IP or distributed across IPs.
- **No special knowledge**: The attacker only needs a valid account ID, which is publicly observable.

### Recommendation

1. **Add rate limiting middleware** (e.g., `express-rate-limit`) applied globally or per-route, limiting requests per IP per time window before they reach the controller.
2. **Bound pool connection acquisition**: Use a semaphore or queue at the application layer to cap the number of concurrent in-flight DB queries, independent of the pool size.
3. **Deploy an API gateway or reverse proxy** (e.g., Traefik, nginx) with connection/request rate limits in front of the REST service — the Traefik `middleware.yaml` charts already exist in the repo but must be configured with rate-limit rules.
4. **Increase `statementTimeout`** awareness: ensure it is set low enough that long-running queries release pool connections quickly, reducing the window for exhaustion.

```javascript
function parseDbPoolConfig() {
  const {pool} = getConfig().db;
  const configKeys = ['connectionTimeout', 'maxConnections', 'statementTimeout'];
  configKeys.forEach((configKey) => {
    const value = pool[configKey];
    const parsed = parseInt(value, 10);
    if (Number.isNaN(parsed) || parsed <= 0) {
      throw new InvalidConfigError(`invalid value set for db.pool.${configKey}: ${value}`);
    }
    pool[configKey] = parsed;
  });
}
```
