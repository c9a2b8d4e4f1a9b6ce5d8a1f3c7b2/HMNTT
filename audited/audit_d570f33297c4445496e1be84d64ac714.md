### Title
Silent Partial-Node Exclusion via Crafted Mirror REST Response in `buildNetworkFromMirrorNodes`

### Summary
`fetchMirrorNodeNetwork` in `pinger/mirror_node_client.go` silently skips any `nodeEntry` whose every `serviceEndpoint` has both `DomainName` and `IPAddressV4` empty. The only guard against a manipulated response is a zero-length check on the final map; there is no minimum-fraction threshold. An attacker who can control the mirror REST response (e.g., via MITM on a plain-HTTP deployment, DNS poisoning, or a compromised mirror node) can therefore cause an arbitrary subset of nodes—including ≥30%—to be silently dropped from the pinger's network map.

### Finding Description

**Exact code path** — `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork`, lines 113–128:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← silent skip
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
// ...
if len(network) == 0 {               // ← only catches total wipeout
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

**Root cause** — The code trusts the mirror REST response entirely. When every endpoint for a node has `domain_name: ""` and `ip_address_v4: ""`, the inner loop exhausts without adding any entry for that node. The node is silently absent from the returned map. The only safety net (`len(network) == 0`) fires only when *all* nodes are wiped; a partial wipeout of any size—including ≥30%—passes through undetected.

**Transport exposure** — The HTTP client is created with no TLS configuration and no certificate pinning:

```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

The canonical example URL in `config.go` is `http://mirror-rest:5551` (plain HTTP). On a plain-HTTP deployment, any on-path network actor can intercept and rewrite the response without any credentials. Even on HTTPS, DNS poisoning (absent DNSSEC) redirects the request to an attacker-controlled server that presents a CA-signed certificate.

**Exploit flow**

1. Attacker positions themselves to intercept or replace the mirror REST response (MITM on HTTP, DNS poisoning, or mirror-node compromise).
2. Attacker crafts a `nodesEnvelope` JSON where the target nodes (≥30% of the real node set) have `service_endpoints` entries with `"domain_name": ""`, `"ip_address_v4": ""`, and any port value.
3. `fetchMirrorNodeNetwork` iterates those entries, hits `if host == "" || ep.Port == 0 { continue }` for every endpoint of each target node, and never inserts those nodes into `network`.
4. The remaining nodes (< 70%) pass the `len(network) == 0` guard.
5. `hiero.ClientForNetwork(netmap)` is called with the truncated map; the pinger only routes transactions to the surviving nodes.

**Why existing checks are insufficient**

| Check | Location | What it catches | What it misses |
|---|---|---|---|
| `if host == "" \|\| ep.Port == 0 { continue }` | line 118–120 | Malformed individual endpoints | Nodes whose *all* endpoints are malformed |
| `if len(network) == 0` | line 127 | Total wipeout | Any partial wipeout (1–99%) |

There is no check of the form "at least N% of the nodes returned by the mirror must have been successfully mapped."

### Impact Explanation

The pinger's network map is the sole source of truth for which consensus nodes it contacts when `network=other`. Excluding ≥30% of nodes from that map means those nodes receive zero pinger transactions. From the pinger's perspective they are invisible—equivalent to being offline. This satisfies the stated severity tier: ≥30% of network processing nodes are effectively shut out of monitoring without any brute-force action against the nodes themselves. If the pinger's liveness data feeds alerting or SLA dashboards, the attacker can selectively hide node failures or create false-positive outage signals.

### Likelihood Explanation

- **HTTP deployments** (the documented default example): any on-path attacker (same LAN, rogue router, BGP hijack) can exploit this with a single crafted HTTP response. No credentials required.
- **HTTPS + no DNSSEC**: DNS poisoning redirects the client to an attacker-controlled server; Go's default TLS validation is satisfied by any CA-signed certificate for the spoofed hostname.
- **Repeatability**: The attack is stateless and repeatable on every pinger restart or network-map refresh cycle.
- **Unprivileged requirement**: met for HTTP deployments; network-layer access is not a "privilege" in the application-security sense.

### Recommendation

1. **Enforce HTTPS and validate the server certificate** — reject `http://` mirror URLs at startup, or at minimum log a loud warning. Consider certificate pinning for known deployments.
2. **Add a minimum-node threshold** — after building the map, assert that the number of successfully mapped nodes is at least some configurable fraction (e.g., ≥50%) of the total `nodeEntry` records returned. Fail hard if the threshold is not met.
3. **Log skipped nodes** — emit a warning for every `nodeEntry` that produces zero usable endpoints so operators can detect manipulation.
4. **Integrity verification** — consider signing the mirror REST response or cross-validating the node list against a secondary source (e.g., the SDK's built-in bootstrap list).

      ]
    },
    {
      "node_account_id": "0.0.4",
      "service_endpoints": [
        {"domain_name": "", "ip_address_v4": "", "port": 50211}

      "node_account_id": "0.0.5",

      ]
    }
  ],
  "links": {"next": null}

./pinger
