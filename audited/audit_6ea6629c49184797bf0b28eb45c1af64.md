### Title
Uncached Range-Scan DB Query on Non-Existent Transaction IDs Enables Sustained DB Read I/O Amplification

### Summary
`getTransactionDetailsFromTransactionId()` in `rest/service/transactionService.js` executes a timestamp-range scan against the `transaction` table on every invocation with no result caching for negative (not-found) outcomes. The `/contracts/results/:transactionIdOrHash` endpoint that drives this function has no rate limiting, allowing an unprivileged attacker to sustain elevated database read I/O by repeatedly submitting valid-format but non-existent transaction IDs at a moderate request rate.

### Finding Description

**Exact code path:**

`transactionDetailsFromTransactionIdQuery` is defined at lines 15–26 of `rest/service/transactionService.js`:

```sql
select consensus_timestamp, nonce, scheduled, type, payer_account_id
from transaction
where payer_account_id = $1
  and consensus_timestamp >= $2 and consensus_timestamp <= $3
  and valid_start_ns = $2
  and nonce = (select coalesce($4, 0))
order by (result = <SUCCESS>) desc, consensus_timestamp desc
limit 1
``` [1](#0-0) 

`getTransactionDetailsFromTransactionId()` at lines 64–72 computes the upper bound as `BigInt(transactionId.getValidStartNs()) + maxTransactionConsensusTimestampRangeNs` and fires this query unconditionally on every call: [2](#0-1) 

**Root cause — two failed assumptions:**

1. The code assumes that a "not found" result is cheap to produce. In reality, the DB must still perform a bounded range scan (`consensus_timestamp >= $2 AND consensus_timestamp <= $3`) across the `transaction` table for the given `payer_account_id`, reading index pages even when zero rows match.
2. The code assumes that callers will not abuse the endpoint at scale. There is no result cache wrapping `getTransactionDetailsFromTransactionId()` — the Redis `Cache` class exists in `rest/cache.js` but is not wired to this function. [3](#0-2) 

**No rate limiting:** A grep across all REST service JS files for `rateLimit`, `rate_limit`, and `throttle` returns zero matches. The endpoint is fully open to unauthenticated callers.

**Call site in `contractController.js`:** `getTransactionDetailsFromTransactionId` is called twice in `rest/controllers/contractController.js` (confirmed by grep), inside the handler for `/contracts/results/:transactionIdOrHash`. Each HTTP request that carries a transaction-ID-shaped parameter triggers a fresh DB query with no short-circuit for previously-seen non-existent IDs. [4](#0-3) 

### Impact Explanation
Every request with a valid-format but non-existent transaction ID causes the database to execute a range scan bounded by `maxTransactionConsensusTimestampRangeNs` on the `transaction` table. On a busy network the `transaction` table is large; even an index-assisted range scan reads buffer-pool pages and generates measurable I/O. Because results are never cached, each unique (or repeated) non-existent transaction ID produces an identical, independently-executed DB query. A sustained moderate request stream (e.g., tens of requests per second from a single host) can keep the DB continuously executing these scans, raising read I/O well above the 24-hour baseline without triggering volumetric brute-force thresholds. The impact is degraded query latency for all other API consumers sharing the same database.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is public. A valid transaction ID is trivially constructable: `0.0.<any_account_num>-<unix_seconds>-<nanos>`. An attacker needs no knowledge of real accounts; any syntactically valid ID that passes format validation will reach the DB query. The attack is repeatable indefinitely, is stateless (no session or token needed), and is indistinguishable from legitimate traffic at the HTTP layer because each request is individually well-formed and returns a normal 404 response.

### Recommendation

1. **Cache negative results:** Wrap `getTransactionDetailsFromTransactionId()` with a short-TTL (e.g., 5–30 s) Redis cache keyed on `(payerAccountId, validStartNs, nonce)`. A cache miss that returns an empty result set should be stored as a negative-cache entry so subsequent identical requests are served from Redis without touching the DB.
2. **Apply per-IP or per-key rate limiting** on `/contracts/results/:transactionIdOrHash` at the middleware layer (e.g., express-rate-limit or an API gateway policy).
3. **Validate `maxTransactionConsensusTimestampRangeNs`** to ensure the scan window is as narrow as operationally required, reducing per-query I/O cost.
