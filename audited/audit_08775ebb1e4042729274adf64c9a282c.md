### Title
Insufficient Jitter in PollingTopicMessageRetriever Enables Synchronized DB Query Storms via Mass Concurrent Subscriptions

### Summary
The `retrieve()` method in `PollingTopicMessageRetriever` uses `RepeatSpec.create().jitter(0.1)` which applies only ±10% randomization to the polling interval. An unprivileged attacker can open an unbounded number of gRPC connections (no per-IP or global connection cap enforced in the gRPC layer), each carrying up to 5 concurrent subscriptions, causing all polling cycles to remain tightly synchronized within a narrow time window. When multiplied across hundreds of concurrent subscriptions, this produces periodic DB query bursts that saturate the connection pool and delay message delivery to all subscribers.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `retrieve()`, lines 51–55:

```java
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                .jitter(0.1)
                .withFixedDelay(context.getFrequency())
                .withScheduler(scheduler))
``` [1](#0-0) 

**Root cause:** `jitter(0.1)` applies a ±10% randomization to the base polling delay. For the throttled path (`pollingFrequency = 2s`, the default for live subscriptions), this yields a jitter window of only ±0.2 s — meaning all polls from subscriptions started at approximately the same time will fire within a 0.4 s window every 2 s. [2](#0-1) 

**Failed assumption:** The design assumes jitter(0.1) is sufficient to desynchronize concurrent subscribers. It is not — 10% jitter on a 2 s interval produces a 0.4 s spread, which is negligible when N subscriptions all start within the same second (as an attacker can arrange).

**Each poll issues a DB query:**

```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [3](#0-2) 

With N concurrent subscriptions, N DB queries fire within a 0.4 s window every 2 s. Each query can return up to `maxPageSize = 1000` rows. [4](#0-3) 

**Existing checks and why they fail:**

The only per-connection limit is `maxConcurrentCallsPerConnection = 5`: [5](#0-4) 

This is enforced per TCP connection, not globally. There is no per-IP connection limit, no global subscription cap, and no rate limiting on the gRPC `subscribeTopic` endpoint. The `web3` throttle (`ThrottleConfiguration`, `ThrottleManagerImpl`) applies only to the web3/EVM path, not to gRPC subscriptions. [6](#0-5) 

An attacker opens K TCP connections × 5 calls each = 5K simultaneous subscriptions, all started at time T=0. Every ~2 s, 5K DB queries fire within a 0.4 s window.

### Impact Explanation

The DB connection pool (HikariCP) becomes saturated during each burst. Legitimate subscribers' polling queries queue behind the attacker's queries, delaying retrieval and delivery of topic messages. Since `PollingTopicMessageRetriever` is the mechanism for delivering historical and live-catch-up messages to all gRPC subscribers, saturation directly delays the gossip of transactions to all connected clients — not just the attacker's. The Grafana alert threshold for DB connection saturation is 75%, confirming this is a recognized operational risk. [7](#0-6) 

### Likelihood Explanation

No authentication or authorization is required to call `subscribeTopic`. Any external client with network access to port 5600 can open arbitrary numbers of TCP connections. Opening 200 connections × 5 calls = 1000 subscriptions requires only a basic gRPC client script. The attack is repeatable, stateless, and requires no special knowledge beyond the public gRPC API. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` on each subscription means subscriptions self-heal after errors, making the attack self-sustaining. [8](#0-7) 

### Recommendation

1. **Increase jitter**: Change `jitter(0.1)` to `jitter(0.5)` or higher to spread polls across a wider window, reducing burst concentration.
2. **Global subscription cap**: Add a configurable maximum total concurrent subscriptions (e.g., via a `Semaphore` or counter checked in `TopicMessageServiceImpl.subscribeTopic`).
3. **Per-IP connection limit**: Enforce a per-source-IP connection limit at the Netty/load-balancer layer.
4. **Shared polling**: Replace per-subscription polling with a shared polling loop that fans out results to all subscribers on the same topic, eliminating the N×1 DB query multiplication.
5. **DB query rate limiting**: Add a global rate limiter (e.g., Bucket4j, already used in web3) gating DB poll execution across all active retrievers.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, mirror_network_service_pb2 as pb

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 200  # 200 connections × 5 calls = 1000 subscriptions

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    request = pb.ConsensusTopicQuery()
    request.topicID.topicNum = 1  # any valid topic
    request.consensusStartTime.seconds = 0
    # Open 5 concurrent streaming calls on this connection
    streams = [stub.subscribeTopic(request) for _ in range(5)]
    for s in streams:
        try:
            for _ in s:  # consume to keep alive
                pass
        except:
            pass

threads = [threading.Thread(target=flood) for _ in range(CONNECTIONS)]
for t in threads: t.start()
# Result: ~1000 DB queries fire every ~2s within a 0.4s window,
# saturating the HikariCP pool and delaying message delivery to all subscribers.
```
