### Title
Unauthenticated Content-Length Header Spoofing Corrupts `api_request_size_bytes` Histogram

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware()` function records the value of the client-supplied `Content-Length` header directly into the `requestSizeHistogram` (`api_request_size_bytes`) without verifying it against the actual bytes received. Any unprivileged external user can send a request with an arbitrarily large `Content-Length` value (e.g., `2147483647`) while transmitting only a few bytes, causing the histogram to record a falsely inflated size and permanently corrupting the metric.

### Finding Description
**Exact code location:** `rest/middleware/metricsHandler.js`, line 209, inside the `res.on('finish', ...)` callback within `metricsMiddleware()`:

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** `req.headers['content-length']` is a raw, client-controlled HTTP header. The code parses it with `parseInt(..., 10)` and records it verbatim. There is no comparison against the actual number of bytes read from the socket, no upper-bound clamp, and no rejection of implausible values.

**Failed assumption:** The code assumes the `Content-Length` header accurately reflects the body actually transmitted. HTTP does not enforce this at the header-parsing layer; a client can declare any value.

**Exploit flow:**
1. Attacker sends a GET request (no body expected, so no body-size enforcement applies) to any valid API route, e.g.:
   ```
   GET /api/v1/transactions HTTP/1.1
   Host: <mirror-node>
   Content-Length: 2147483647
   ```
2. Express routes the request normally. `express.json()` / `express.urlencoded()` (lines 68–73 of `server.js`) do not reject GET requests based on `Content-Length`.
3. The response completes; `res.on('finish')` fires.
4. Line 209 evaluates `parseInt("2147483647", 10)` → `2147483647` and calls `requestSizeHistogram.record(2147483647, labels)`.
5. The OpenTelemetry histogram now contains a ~2 GB observation for that route/method/code label combination.
6. Repeating this across multiple routes and status codes fully corrupts all per-route request-size buckets.

**Why existing checks are insufficient:**
- `express.json()` enforces a body-read limit (default 100 kB) only when it actually reads the body stream. For GET requests it is a no-op.
- Even for POST requests that are rejected with HTTP 413, the `res.on('finish')` callback still fires and the header value is still recorded.
- The `|| 0` fallback on line 209 only guards against `NaN`/`0`; it does not cap large positive integers.
- No rate limiting is visible in `server.js` that would prevent rapid repetition.

### Impact Explanation
The `api_request_size_bytes` Prometheus histogram is permanently skewed: all bucket counts and sum/count aggregates become meaningless. Operators relying on this metric for capacity planning, anomaly detection, or SLO alerting will receive false signals (apparent gigabyte-sized requests). Because histogram observations are cumulative and irreversible within a process lifetime, a single burst of spoofed requests permanently degrades the metric until the process restarts. This is a monitoring/observability integrity issue with no direct data-plane impact, consistent with the "griefing / no economic damage" severity classification.

### Likelihood Explanation
The attack requires zero authentication, zero special tooling, and zero knowledge beyond the public API path structure. A single `curl` command is sufficient. It is trivially repeatable in a loop. Any internet-accessible deployment of the mirror-node REST service is exposed.

### Recommendation
Replace the header-based measurement with the actual bytes received. Node.js / Express tracks this on `req.socket.bytesRead` or via a counting stream wrapper. If the header value must be used as a fallback, clamp it to a sane maximum (e.g., the configured body-size limit) before recording:

```js
const MAX_RECORDED_REQUEST_BYTES = 10 * 1024 * 1024; // 10 MB or match express.json limit
const declaredSize = parseInt(req.headers['content-length'] ?? '0', 10) || 0;
requestSizeHistogram.record(Math.min(declaredSize, MAX_RECORDED_REQUEST_BYTES), labels);
```

Ideally, instrument actual bytes read from the socket rather than trusting the client-declared header.

### Proof of Concept
```bash
# Single spoofed request inflating the histogram by ~2 GB
curl -s -o /dev/null -w "%{http_code}" \
  -H "Content-Length: 2147483647" \
  "http://<mirror-node-host>/api/v1/transactions"

# Verify corruption by scraping the metrics endpoint
curl -s "http://<mirror-node-host>/swagger/metrics/" \
  | grep api_request_size_bytes

# Automated loop to sustain corruption
for i in $(seq 1 1000); do
  curl -s -o /dev/null \
    -H "Content-Length: 2147483647" \
    "http://<mirror-node-host>/api/v1/transactions" &
done
wait
```

After the loop, the `api_request_size_bytes_sum` counter for the `transactions` route will reflect trillions of bytes, making the histogram permanently unusable.
