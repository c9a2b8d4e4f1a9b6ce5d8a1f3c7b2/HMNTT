### Title
Key Range Filter Bypass in Historical Hook Storage Queries

### Summary
`getHookStorageChange()` computes `keysInRange` to enforce the declared key range but then passes the raw `keys` collection to `findByKeyInAndTimestampBetween()` at line 104. The guard at line 93 only short-circuits when every EQ key is out of range; when at least one EQ key is in range, out-of-range EQ keys are silently included in the database query, bypassing the declared key range filter entirely.

### Finding Description
**Code path:**
- `HookServiceImpl.java`, `getHookStorageChange()`, lines 89–105
- `HookStorageRequest.java`, `getKeysInRange()`, lines 43–52

**Root cause:** Line 104 passes `keys` (the raw, unfiltered EQ key collection) instead of `keysInRange` (the range-filtered subset) to `findByKeyInAndTimestampBetween()`.

```java
// Line 89-95: keysInRange computed, guard only catches all-out-of-range case
final var keys = request.getKeys();
final boolean requestHasKeys = !keys.isEmpty();
final var keysInRange = request.getKeysInRange();

if (keysInRange.isEmpty() && requestHasKeys) {   // guard: only fires if ALL keys are out of range
    return new HookStorageResult(ownerId, List.of());
}

// Line 103-105: BUG — passes `keys`, not `keysInRange`
if (requestHasKeys) {
    changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
            ownerId.getId(), hookId, keys, ...);   // <-- should be keysInRange
```

**Exploit flow:**
1. Attacker knows a target key `K_secret` (outside the declared range) and a decoy key `K_decoy` (inside the range).
2. Attacker sends: `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage?key=K_decoy&key=K_secret&key=gt:K_decoy_minus_1&key=lt:K_decoy_plus_1&timestamp=lte:now`
3. `keyLowerBound = K_decoy_minus_1`, `keyUpperBound = K_decoy_plus_1`
4. `keys = [K_decoy, K_secret]`
5. `keysInRange = [K_decoy]` → non-empty, guard does not fire
6. Repository receives `keys = [K_decoy, K_secret]` → returns historical changes for both keys, including `K_secret`

**Why the existing check fails:** The guard (`keysInRange.isEmpty() && requestHasKeys`) is a necessary but insufficient condition. It only prevents the case where zero keys survive the range filter. It does not prevent the case where some keys survive and others do not — in that case, the unfiltered set is used.

### Impact Explanation
Any unprivileged caller can retrieve the full historical write/read value log (`valueRead`, `valueWritten`) for arbitrary storage keys of any hook, regardless of the key range bounds they declare. This exposes storage slot history that the API contract intends to restrict. In a hook system where storage slots encode sensitive state (balances, authorization flags, nonces), this constitutes unauthorized information disclosure of on-chain private state history.

### Likelihood Explanation
No authentication or elevated privilege is required. The exploit requires only knowledge of one valid in-range key (discoverable via a prior range scan) and the target out-of-range key. The attack is fully repeatable, requires a single crafted HTTP GET request, and leaves no distinguishable server-side trace beyond normal query logs.

### Recommendation
Replace `keys` with `keysInRange` at line 104 in `HookServiceImpl.getHookStorageChange()`:

```java
// Fix: pass keysInRange, not keys
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keysInRange, timestampLowerBound, timestampUpperBound, page);
```

This mirrors the correct pattern already used in the non-historical path at line 78–79, where `keysInRange` is correctly passed to `findByOwnerIdAndHookIdAndKeyInAndDeletedIsFalse()`.
