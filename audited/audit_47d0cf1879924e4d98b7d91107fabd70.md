### Title
Unauthenticated Three-Part UNION Query Amplification DoS on `/accounts/:id/nfts` Endpoint

### Summary
An unprivileged external user can craft a single HTTP request to `GET /api/v1/accounts/:id/nfts?token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B` that forces `extractNftMultiUnionQuery()` to populate all three filter sets — `getLowerFilters()`, `getInnerFilters()`, and `getUpperFilters()` — causing `NftService.getQuery()` to emit a three-part `UNION ALL` query, each branch performing a `LEFT JOIN entity`. With no application-level rate limiting on the Node.js REST API and a default DB connection pool of only 10 connections, an attacker can flood this endpoint to exhaust the pool and render the REST API unavailable.

### Finding Description

**Exact code path:**

In `rest/controllers/accountController.js` lines 34–82, `extractNftMultiUnionQuery()` parses filters and calls: [1](#0-0) 

These three helpers in `rest/controllers/baseController.js` are all populated when both `token.id` has a `gte` lower bound and an `lte` upper bound AND `serialnumber` has a `gte` lower bound and an `lte` upper bound: [2](#0-1) 

`getInnerFilters()` returns non-empty only when both primary and secondary bounds exist: [3](#0-2) 

`getUpperFilters()` returns non-empty only when both primary upper and secondary upper exist: [4](#0-3) 

In `rest/service/nftService.js`, `getQuery()` then emits a three-branch `UNION ALL` SQL statement, each branch containing a `LEFT JOIN entity`: [5](#0-4) 

The test suite explicitly confirms this three-part output for the `token gte and lte, serial gte and lte` combination: [6](#0-5) 

**Root cause:** The `acceptedNftAccountParameters` set accepts `TOKEN_ID` and `SERIAL_NUMBER` with range operators, and the validation in `validateBounds()` explicitly permits the `gte`+`lte` combination for both simultaneously. No application-level rate limiting exists in the REST API middleware stack. [7](#0-6) 

**Exploit flow:**
1. Attacker sends concurrent requests: `GET /api/v1/accounts/0.0.1/nfts?token.id=gte:0.0.1&token.id=lte:0.0.9999999999&serialnumber=gte:1&serialnumber=lte:9999999999`
2. Each request passes `buildAndValidateFilters()` without error.
3. `extractNftMultiUnionQuery()` populates all three filter arrays.
4. `NftService.getQuery()` emits a three-branch `UNION ALL` with three `LEFT JOIN entity` subqueries.
5. Each request holds a DB connection for the duration of the query (up to the 20-second `statementTimeout`).
6. With `maxConnections: 10`, 10 concurrent such requests exhaust the pool; all subsequent requests queue or fail.

### Impact Explanation

The REST API's database connection pool defaults to 10 connections with a 20-second statement timeout: [8](#0-7) 

Ten concurrent three-part UNION queries (each 3× more expensive than a simple query due to three subqueries with `LEFT JOIN entity`) can saturate all available connections. Legitimate users receive errors or timeouts. The mirror node REST API becomes unavailable. This does not affect Hedera consensus directly, but it denies all mirror node REST API consumers (wallets, explorers, dApps) access to account NFT data and other endpoints sharing the same pool.

### Likelihood Explanation

No authentication or application-level rate limiting is present in the REST middleware: [9](#0-8) 

The GCP gateway `maxRatePerEndpoint: 250` is infrastructure-level and deployment-specific: [10](#0-9) 

The response cache is disabled by default (`cache.response.enabled: false`), so repeated unique account IDs bypass any caching. Any unauthenticated attacker with a script and ~10 concurrent HTTP connections can trigger this. The parameter combination is documented in the OpenAPI spec, making it trivially discoverable: [11](#0-10) 

### Recommendation

1. **Add application-level rate limiting** to the Node.js REST API (e.g., `express-rate-limit` per IP) for all endpoints, not just the web3 service.
2. **Increase the DB connection pool** or add a query queue with a maximum wait time to fail fast under load.
3. **Enable the Redis response cache** by default for read-only endpoints like `/accounts/:id/nfts` to absorb repeated identical queries.
4. **Add a query complexity cost** check: when all three filter arrays are non-empty, optionally require authentication or apply a stricter rate limit.
5. **Reduce `statementTimeout`** for this endpoint specifically to limit how long each connection is held.

### Proof of Concept

```bash
# Trigger three-part UNION ALL query (single request, verify with DB logs)

# Flood to exhaust the 10-connection pool
for i in $(seq 1 50); do
  curl -s "http://mirror-node/api/v1/accounts/0.0.$((RANDOM % 1000 + 1))/nfts?token.id=gte:0.0.1&token.id=lte:0.0.9999999&serialnumber=gte:1&serialnumber=lte:9999999" &
done
wait

# Observe: legitimate requests to any REST endpoint now time out or return 503
curl "http://mirror-node/api/v1/accounts/0.0.1001/nfts"
```

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
