### Title
Unbounded Hash String in `/block` Causes Memory Exhaustion DoS via Missing Body Size Limit

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` accepts a `block_identifier.hash` field of arbitrary length from any unauthenticated caller. `SafeRemoveHexPrefix()` performs no length check and returns the full string, which is then allocated in heap memory and forwarded as a raw parameter to a PostgreSQL query. No HTTP body size limit exists anywhere in the rosetta middleware stack, allowing an attacker to force large allocations per request and exhaust server memory with concurrent requests.

### Finding Description

**Exact code path:**

1. `Block()` at `rosetta/app/services/block_service.go:51` calls `s.RetrieveBlock(ctx, request.BlockIdentifier)`.
2. `RetrieveBlock()` at `rosetta/app/services/base_service.go:80,85` calls `tools.SafeRemoveHexPrefix(*bIdentifier.Hash)` with no length guard.
3. `SafeRemoveHexPrefix()` at `rosetta/app/tools/hex.go:18-22` only checks for a `"0x"` prefix and returns the full string unchanged for non-hex input.
4. The resulting string is passed to `blockRepo.FindByIdentifier()` / `FindByHash()` at `rosetta/app/persistence/block.go:159,231`, which executes `db.Raw(selectByHashWithIndex, sql.Named("hash", hash))` — sending the full string to PostgreSQL as a query parameter.

**Root cause:** No maximum length is enforced on the hash field at any layer — not in the HTTP handler, not in `SafeRemoveHexPrefix()`, not in `FindByIdentifier()` (which only checks `hash == ""`), and not in the rosetta-sdk-go asserter (which validates network identifiers and operation types, not hash string length).

**Why existing checks fail:**
- `ReadTimeout: 5s` (`rosetta/main.go:225`) limits read *time*, not body *size*. At 100 Mbps a 10 MB payload is read in ~0.8 s, well within the window.
- `ReadHeaderTimeout: 3s` applies only to headers, not the body.
- No `http.MaxBytesReader` or equivalent is applied anywhere in the rosetta middleware chain (confirmed: zero matches for `MaxBytesReader`, `LimitReader`, `maxBodySize` in `rosetta/**/*.go`).
- The SQL query is parameterized (no injection), but the full oversized string is still allocated in Go heap and transmitted to the database.

### Impact Explanation
An attacker sending concurrent POST `/block` requests each carrying a multi-megabyte `hash` string forces the server to allocate proportional heap memory per request. With up to 100 open DB connections (`maxOpenConnections: 100`), 100 simultaneous requests with 10 MB hashes each impose ~1 GB of heap pressure, leading to OOM kills or severe GC pauses that degrade or crash the rosetta service for all users. The database also receives oversized query parameters on every request, consuming DB network bandwidth and connection memory.

### Likelihood Explanation
No authentication is required to call `/block`. The Rosetta API is designed to be publicly accessible. The attack requires only a fast network connection (trivially available via cloud VMs) and a simple HTTP client loop. It is fully repeatable and automatable with a single script. The absence of any rate limiting or body size cap makes sustained amplification straightforward.

### Recommendation
1. Apply `http.MaxBytesReader` to the request body in the HTTP handler or a middleware wrapper, capping it at a few kilobytes (a valid Rosetta `/block` request body is never more than ~1 KB).
2. Add an explicit hash length check in `SafeRemoveHexPrefix()` or at the entry of `RetrieveBlock()` / `FindByIdentifier()` / `FindByHash()`, rejecting any hash longer than the expected digest size (e.g., 96 hex characters for SHA-384).
3. Consider adding per-IP rate limiting middleware to the rosetta router.

### Proof of Concept
```bash
# Generate a 5 MB non-hex hash payload
LARGE_HASH=$(python3 -c "print('A' * 5_000_000)")

# Send 50 concurrent requests
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"block_identifier\": {\"index\": 0, \"hash\": \"$LARGE_HASH\"}
    }" &
done
wait
# Observe: rosetta process RSS grows by ~250 MB+; repeated bursts cause OOM or severe latency
```
