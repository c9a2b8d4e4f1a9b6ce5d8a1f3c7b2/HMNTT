### Title
Connection Pool Exhaustion DoS via Unauthenticated Concurrent Requests to `/accounts/:idOrAliasOrEvmAddress/tokens`

### Summary
The `getTokenRelationships` handler in `rest/controllers/tokenController.js` issues two sequential, independently connection-acquiring DB queries per request (`getEncodedId` + `isValidAccount`). The REST API has no rate-limiting middleware, and the default connection pool is only 10 connections. An unauthenticated attacker sending sustained concurrent requests with alias or EVM-address inputs can hold all pool connections busy for up to the `statement_timeout` window (20 s), causing legitimate requests to queue and eventually fail with connection-timeout errors, effectively partitioning the application from the database for the duration of the attack.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 19 registers the handler:
```js
router.getExt(getPath('tokens'), TokenController.getTokenRelationships);
```

`rest/controllers/tokenController.js` lines 67â€“68 issue two sequential awaited DB calls:
```js
const accountId = await EntityService.getEncodedId(req.params[...]);   // DB hit for alias/evm
const isValidAccount = await EntityService.isValidAccount(accountId);  // always DB hit
```

`rest/service/entityService.js`:
- `getEncodedId` (line 118): for alias inputs calls `getAccountIdFromAlias` â†’ `getRows(entityFromAliasQuery, ...)` â€” one DB round-trip.
- `isValidAccount` (line 60): always calls `getSingleRow(entityExistenceQuery, ...)` â€” one DB round-trip.

Each `getRows` / `getSingleRow` call goes through `BaseService.pool().queryQuietly()` (`rest/service/baseService.js` line 56), which uses `pg.Pool.query()`. `pg.Pool.query()` acquires a physical connection, sends the query, waits for the result, then releases the connection. While the query is in-flight the connection is held.

**Pool configuration** (`rest/dbpool.js` lines 13â€“15):
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20 000 ms
max: config.db.pool.maxConnections,                         // default 10
statement_timeout: config.db.pool.statementTimeout,         // default 20 000 ms
```

**No rate limiting on the REST API.** `rest/server.js` registers only: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, `handleError`. There is no per-IP or global request-rate or concurrency-limiting middleware. The throttling found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) applies exclusively to the Java web3 service, not to this Node.js REST service.

**Root cause / failed assumption:** The design assumes that the small pool (10) is sufficient because queries are fast. It does not account for an attacker who can keep queries slow by flooding the DB, and it provides no mechanism to shed excess load before connections are acquired.

**Exploit flow:**
1. Attacker identifies that alias-format inputs (e.g., `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ`) cause `getEncodedId` to issue a real DB query.
2. Attacker opens â‰Ą10 concurrent HTTP connections and sends `GET /api/v1/accounts/<alias>/tokens` on each.
3. Each request's first `await getEncodedId(alias)` acquires a pool connection and holds it while the DB processes the query.
4. With all 10 connections busy, the 11th request's `pool.query()` call blocks waiting for a free connection.
5. The attacker sustains the flood; the DB, now under load from 10 simultaneous queries, responds slowly (up to `statement_timeout` = 20 s).
6. The connection queue grows; new legitimate requests wait up to `connectionTimeoutMillis` (20 s) and then receive a `Error: timeout exceeded when trying to connect` error, which the error handler returns as HTTP 500.
7. The application is effectively partitioned from the database for the duration of the attack.

**Why existing checks are insufficient:**
- `statement_timeout` (20 s) bounds individual query duration but does not prevent pool exhaustion â€” it only sets the maximum hold time per connection.
- `connectionTimeoutMillis` (20 s) causes queued requests to fail, but this is the symptom of exhaustion, not a prevention.
- No authentication is required; the endpoint is fully public.
- No concurrency cap or rate limit exists in the REST middleware stack.

### Impact Explanation
All API endpoints sharing the same `pg.Pool` instance (`global.pool`) are affected, not just `/tokens`. When the pool is exhausted, every endpoint that issues a DB query returns HTTP 500 to clients. This constitutes a full denial-of-service of the mirror node REST API for the attack duration. The impact is high: the mirror node becomes unavailable to all consumers (wallets, explorers, dApps) without any privileged access required.

### Likelihood Explanation
The attack requires only an internet connection and the ability to open ~10â€“20 concurrent HTTP connections â€” trivially achievable from a single machine or a small botnet. No authentication, API key, or special knowledge is needed. The endpoint is publicly documented in the OpenAPI spec. The attack is repeatable and can be sustained indefinitely. The only natural friction is the 20 s `statement_timeout`, which means the attacker must maintain ~10 concurrent in-flight requests continuously; this is a very low bar.

### Recommendation
1. **Add a concurrency/rate-limit middleware** to the Node.js REST server (e.g., `express-rate-limit` or a semaphore on DB-bound routes) before any DB query is issued.
2. **Increase the default pool size** or, better, deploy pgbouncer in transaction-pooling mode in front of the Node.js pool (the Helm chart already supports this for production; enforce it).
3. **Merge the two sequential queries into one** in `getTokenRelationships`: a single SQL query can both resolve the alias and validate account existence, halving the per-request connection demand.
4. **Add a global in-flight request cap** (e.g., via `express-slow-down` or a custom semaphore) so that when the pool is near exhaustion, new requests are rejected with HTTP 429 rather than queued indefinitely.

### Proof of Concept
```bash
# Requires: curl, GNU parallel or xargs
# Replace <host> with the mirror node REST API host

ALIAS="AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ"
URL="http://<host>/api/v1/accounts/${ALIAS}/tokens"

# Send 20 concurrent requests in a tight loop
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "%{http_code}\n" "$URL" &
done
wait

# Repeat in a loop to sustain pool exhaustion:
while true; do
  for i in $(seq 1 20); do
    curl -s -o /dev/null "$URL" &
  done
  wait
done
```

**Expected result:** After a few seconds, responses from the mirror node switch from `200 OK` (or `404 Not Found`) to `500 Internal Server Error` for all endpoints, including unrelated ones (e.g., `/api/v1/transactions`), confirming full pool exhaustion. The attack stops being effective within ~20 s of the attacker stopping requests (one `statement_timeout` cycle).
