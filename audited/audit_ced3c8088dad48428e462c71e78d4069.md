### Title
Unbounded Alias Count in `resolveAccountAliases()` Enables Resource Exhaustion via Amplified Database Queries

### Summary
The `resolveAccountAliases()` function in `rosetta/app/services/construction_service.go` accepts a comma-separated `account_aliases` string from the public `/construction/metadata` endpoint with no upper bound on the number of aliases. Each alias triggers a synchronous database query, allowing any unauthenticated caller to amplify a single HTTP request into an arbitrary number of database round-trips, exhausting the shared PostgreSQL connection pool and degrading or blocking the importer's block-processing pipeline.

### Finding Description
**Exact code path:**

`ConstructionMetadata()` (line 159) calls `resolveAccountAliases(ctx, options)` unconditionally when `options["account_aliases"]` is non-nil. [1](#0-0) 

Inside `resolveAccountAliases()`, the string is split with `strings.SplitSeq(accountAliases, ",")` and for every element a blocking DB call is issued: [2](#0-1) 

`GetAccountId()` executes a raw SQL query against the `entity` table for every alias: [3](#0-2) 

**Root cause:** The function contains zero validation of the alias count or total string length before entering the loop. The implicit assumption is that callers supply only the one or two aliases produced by `ConstructionPreprocess()`, but the `Options` map in a `ConstructionMetadataRequest` is free-form JSON accepted directly from the network. [4](#0-3) 

**Why existing checks are insufficient:**

The only defences are infrastructure-level Traefik middlewares defined in the Helm chart: [5](#0-4) 

- `inFlightReq: amount: 5` limits concurrent requests per source IP to 5, but each of those 5 requests can carry an unbounded alias list.
- `rateLimit: average: 10` is keyed on `requestHost` (the server's hostname), not the client IP, so it does not throttle individual callers.
- Neither control is enforced at the application layer; deployments that bypass Traefik (direct NodePort, port-forward, misconfigured ingress) have no protection at all.
- The HTTP server's `ReadTimeout`/`WriteTimeout` bound the connection lifetime but not the number of DB queries issued within a single request. [6](#0-5) 

### Impact Explanation
The Rosetta service and the Hiero importer share the same PostgreSQL instance. Exhausting the DB connection pool (or saturating I/O bandwidth with sequential `SELECT` queries) directly starves the importer of database capacity, causing it to stall between record-file ingestion cycles. Hedera produces a new record file roughly every 2 seconds; sustained DB saturation can delay consensus-record ingestion well beyond 500 % of that baseline, effectively halting the mirror node's view of the network. Additionally, legitimate Rosetta API consumers are denied service for the duration of the attack.

### Likelihood Explanation
The `/construction/metadata` endpoint requires no authentication and is reachable by any network client. Crafting the exploit requires only a standard HTTP POST with a JSON body; no cryptographic material, account ownership, or privileged access is needed. The attack is trivially repeatable and scriptable. Even under the Traefik `inFlightReq: 5` constraint, five simultaneous requests each carrying 50 000 aliases produce 250 000 sequential DB queries per second-class of requests, which is sufficient to saturate a typical PostgreSQL instance.

### Recommendation
1. **Enforce an alias count cap** inside `resolveAccountAliases()` before entering the loop — e.g., reject requests where `strings.Count(accountAliases, ",") >= maxAliasCount` (suggested maximum: 10, matching the realistic number of signers in a Hedera transaction).
2. **Enforce a string-length cap** on `accountAliases` before splitting, to prevent memory amplification from a single enormous token.
3. **Batch the DB lookups** into a single `IN (...)` query rather than one query per alias, reducing per-request DB load even for legitimate callers.
4. **Add application-level rate limiting** (e.g., `golang.org/x/time/rate`) so the protection is not solely dependent on infrastructure configuration.

python3 -c "
import json, sys
aliases = ','.join(f'0.0.{i}' for i in range(1, 10001))
payload = {
  'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
  'options': {
    'operation_type': 'CRYPTO_TRANSFER',
    'account_aliases': aliases
  }
}
print(json.dumps(payload))
" > payload.json

# Send to the public /construction/metadata endpoint (no auth required)
curl -s -X POST https://<rosetta-host>/construction/metadata \
     -H 'Content-Type: application/json' \
     -d @payload.json
```

**Expected result:** The server issues 10 000 sequential `SELECT` queries against the `entity` table before returning. Repeating this request concurrently (e.g., 5 parallel `curl` processes to stay within the `inFlightReq` limit) produces 50 000 simultaneous DB queries, exhausting the connection pool and stalling the importer's record-file ingestion pipeline.

func (c *constructionAPIService) resolveAccountAliases(
	ctx context.Context,
	options map[string]any,
) (emptyResult string, nilErr *rTypes.Error) {
	if options[optionKeyAccountAliases] == nil {
		return
	}

	if !c.BaseService.IsOnline() {
		return emptyResult, errors.ErrEndpointNotSupportedInOfflineMode
	}

	accountAliases, ok := options[optionKeyAccountAliases].(string)
	if !ok {
		return emptyResult, errors.ErrInvalidOptions
	}

	var accountMap []string
	for accountAlias := range strings.SplitSeq(accountAliases, ",") {
		accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
		if err != nil {
			return emptyResult, errors.ErrInvalidAccount
		}

		found, rErr := c.accountRepo.GetAccountId(ctx, accountId)
		if rErr != nil {
			return emptyResult, rErr
		}
		accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
	}

	return strings.Join(accountMap, ","), nilErr
}
```

func (ar *accountRepository) GetAccountId(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	if !accountId.HasAlias() {
		return accountId, nil

	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return zero, hErrors.ErrAccountNotFound
		}

		return zero, hErrors.ErrDatabaseError
	}

	return types.NewAccountIdFromEntityId(entity.Id), nil
