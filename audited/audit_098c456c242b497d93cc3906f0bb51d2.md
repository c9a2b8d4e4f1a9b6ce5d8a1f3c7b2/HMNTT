### Title
Unbounded Timestamp Range in `getHookStorage()` Enables Full Historical Table Scan DoS

### Summary
The `getHookStorage()` endpoint in `HooksController` accepts user-supplied `timestamp` parameters with no range-width validation. `Bound.of()` is called with `primarySortField=false`, bypassing the only range sanity check, and the resulting unbounded range is passed directly to a `DISTINCT ON (key)` SQL query that scans all time-partitioned rows for the given owner/hook. No rate limiting exists in the `rest-java` module, so any unauthenticated caller can repeatedly trigger full cross-partition scans.

### Finding Description

**Exact code path:**

`HooksController.getHookStorage()` (line 104–130) accepts `@Size(max = 2) TimestampParameter[] timestamps` — the only constraint is that at most 2 timestamp parameters are supplied. [1](#0-0) 

`hookStorageChangeRequest()` (line 186) calls `Bound.of(timestamps, TIMESTAMP, ...)` with no range-width check: [2](#0-1) 

`Bound.of()` (line 169–182) constructs the bound with `primarySortField=false`, which means the only guard — `if (primarySortField && adjustedLower > adjustedUpper)` — is **never evaluated**: [3](#0-2) 

When only a lower bound is supplied (e.g., `timestamp=gte:0`), `adjustUpperBound()` returns `Long.MAX_VALUE`: [4](#0-3) 

`HookServiceImpl.getHookStorageChange()` (line 98–99) extracts these values and passes them directly to the repository: [5](#0-4) 

The repository executes a `DISTINCT ON (key)` native query with `consensus_timestamp between :timestampLowerBound and :timestampUpperBound`: [6](#0-5) 

`DISTINCT ON (key)` requires PostgreSQL to scan **all** rows matching the timestamp range before applying the `LIMIT`, because it must find the latest `consensus_timestamp` per key. With a range of `[0, Long.MAX_VALUE]`, every time-partition of `hook_storage_change` is touched.

The table is partitioned by `consensus_timestamp`: [7](#0-6) 

**No rate limiting exists in `rest-java`:** The `ThrottleConfiguration` / `ThrottleManager` infrastructure is confined to the `web3` module. No equivalent exists for `HooksController`.


**Contrast with the Node.js REST API**, which enforces `maxTimestampRange` / `maxTimestampRangeNs` and throws `InvalidArgumentError` when the range is too wide: [8](#0-7) 

No such check exists anywhere in the `rest-java` path for this endpoint.

### Impact Explanation
Each request with `timestamp=gte:0` forces a full cross-partition sequential scan of `hook_storage_change` for the targeted `(owner_id, hook_id)`. As the table grows with network history, each such query consumes proportionally more DB CPU, I/O, and memory. Concurrent flooding of this endpoint from multiple source IPs can saturate DB connection pools and query workers, degrading or halting consensus-timestamp-indexed query processing across nodes that share the same DB cluster — consistent with the ≥30% node processing degradation threshold described in the scope.

### Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. The exploit requires only a valid `ownerId` and `hookId` (both discoverable via the `GET /api/v1/accounts/{ownerId}/hooks` endpoint). No special privileges, tokens, or cryptographic material are needed. The attack is trivially scriptable and repeatable with standard HTTP tooling.

### Recommendation
1. **Enforce a maximum timestamp range width** in `Bound.of()` or in `hookStorageChangeRequest()`, mirroring the Node.js `maxTimestampRangeNs` check. Reject requests where `adjustUpperBound() - getAdjustedLowerRangeValue()` exceeds a configured maximum (e.g., 7 days in nanoseconds).
2. **Require both a lower and upper timestamp bound** when the historical path is taken (i.e., disallow open-ended ranges for `hook_storage_change` queries).
3. **Add rate limiting** to the `rest-java` module for `HooksController`, analogous to the `ThrottleManager` in `web3`.
4. **Add a DB-level statement timeout** for queries on `hook_storage_change` to bound worst-case execution time.

### Proof of Concept
```
# Step 1: Discover a valid hookId for a known ownerId
GET /api/v1/accounts/0.0.1234/hooks

# Step 2: Flood the storage endpoint with a max-range timestamp scan
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&timestamp=lte:9999999999999999999&limit=100" &
done
```

Each request causes PostgreSQL to execute:
```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = 1234
  AND hook_id = 1
  AND key >= '\x0000...00'
  AND key <= '\xffff...ff'
  AND consensus_timestamp BETWEEN 0 AND 9999999999999999999
```
scanning every partition of `hook_storage_change` for that owner/hook pair. Concurrent requests exhaust DB worker threads and I/O bandwidth.
