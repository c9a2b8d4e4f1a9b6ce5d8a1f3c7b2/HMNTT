### Title
Cache-Miss Amplification DoS via Non-Existent Contract IDs in `findRuntimeBytecode()`

### Summary
`ContractRepository.findRuntimeBytecode()` uses `unless = "#result == null"` on an `Optional<byte[]>` return type. Spring Cache unwraps `Optional` before evaluating SpEL `#result`, so `Optional.empty()` (non-existent contract) yields `#result == null` → true → the miss is **never cached**. An unprivileged attacker can flood `/api/v1/contracts/call` with requests targeting distinct non-existent contract addresses, forcing a live DB query on every single request, limited only by the global 500 req/s rate bucket which is shared across all users.

### Finding Description

**Exact location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [1](#0-0) 

**Root cause — Spring Optional unwrapping in SpEL:**
Per Spring Framework documentation, for `@Cacheable` methods returning `Optional`, `#result` in `unless`/`condition` SpEL expressions refers to the **unwrapped content**, not the `Optional` wrapper itself. Therefore:
- `Optional.of(bytes)` → `#result` = `bytes` (non-null) → `unless` is false → **cached** ✓
- `Optional.empty()` → `#result` = `null` → `unless` is true → **not cached** ✗

Every call for a non-existent `contractId` bypasses the cache entirely and issues a live `SELECT runtime_bytecode FROM contract WHERE id = ?` query against the database.

**Cache configuration** (`CacheProperties.java` line 22):
```
expireAfterAccess=1h, maximumSize=1000
``` [2](#0-1) 

Even if the same non-existent ID is queried repeatedly, it is never stored, so the cache provides zero protection.

**Exploit path through the throttle:**

The gas throttle uses `scaleGas()`:
```java
private static final long GAS_SCALE_FACTOR = 10_000L;
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) { return 0L; }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
``` [3](#0-2) 

The minimum allowed gas in `ContractCallRequest` is `@Min(21_000)`: [4](#0-3) 

`scaleGas(21_000)` = `Math.floorDiv(21_000, 10_000)` = **2 tokens** consumed per request. The gas bucket capacity is `scaleGas(7_500_000_000)` = **750,000 tokens**, refilling at 750,000/second. At minimum gas, the gas throttle allows ~375,000 req/s — far above the binding rate limit.

The **only** effective constraint is the global rate limit:
```java
@Min(1)
private long requestsPerSecond = 500;
``` [5](#0-4) 

This bucket is **global** (not per-IP), so a single attacker can monopolize the full 500 req/s, causing 500 uncached DB queries per second indefinitely.

The throttle check in `ThrottleManagerImpl`: [6](#0-5) 

### Impact Explanation

500 uncached `SELECT` queries/second against the `contract` table, sustained indefinitely by a single attacker with no credentials. This exhausts DB connection pool capacity, increases query latency for all other operations (entity lookups, token lookups, state reads), and degrades or denies service to legitimate users. Because the rate bucket is global, the attacker simultaneously blocks all legitimate traffic from being processed at full throughput.

### Likelihood Explanation

No authentication or API key is required. The attacker only needs to generate valid 40-character hex addresses (trivially scripted) that do not correspond to existing contracts. The attack is fully repeatable, requires no special knowledge, and can be sustained from a single machine at the 500 req/s ceiling. The minimum gas constraint (`@Min(21_000)`) does not meaningfully limit throughput given the gas bucket math above.

### Recommendation

1. **Cache negative results explicitly.** Change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` — but note this caches "not found" for up to 1 hour. Alternatively, use a short TTL negative cache (e.g., 5–10 seconds) to absorb burst attacks while not permanently hiding newly-deployed contracts.

2. **Add per-IP rate limiting** at the ingress/load-balancer layer or within `ThrottleManagerImpl` so a single source cannot consume the entire global 500 req/s budget.

3. **Increase the cache `maximumSize`** beyond 1,000 entries so that even if negative results are cached, the cache is not trivially evicted by a small number of distinct IDs.

### Proof of Concept

```bash
# Generate 10,000 distinct non-existent contract addresses and flood the endpoint
for i in $(seq 1 10000); do
  ADDR=$(printf "0x%040x" $i)  # valid 40-hex address, no contract deployed
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$ADDR\",\"gas\":21000}" &
done
```

Each request reaches `ContractBytecodeReadableKVState.readFromDataSource()` → `contractRepository.findRuntimeBytecode(entityId)` → cache miss (because `Optional.empty()` is never stored) → live DB query. At 500 req/s sustained, the database receives 500 `SELECT runtime_bytecode FROM contract WHERE id = ?` queries per second with no cache relief, degrading all concurrent legitimate contract call processing.
