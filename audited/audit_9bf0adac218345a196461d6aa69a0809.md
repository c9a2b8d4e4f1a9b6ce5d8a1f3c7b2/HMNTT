### Title
Thundering Herd Cache Miss Race in `findRuntimeBytecode()` Exhausts DB Connection Pool During Network Partition

### Summary
`ContractRepository.findRuntimeBytecode()` uses `@Cacheable` without `sync=true`, meaning Spring's cache abstraction performs a non-atomic check-then-act against Caffeine's standard `Cache`. Under concurrent load, all threads simultaneously miss the cache and each independently acquires a HikariCP connection to execute the DB query. During a network partition where DB queries hang until the 3-second `statementTimeout`, a burst of concurrent requests exhausts the connection pool and causes cascading failures across all repository operations.

### Finding Description

**Code path:**

`ContractRepository.java` line 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

`EvmConfiguration.java` lines 67–73 — the cache manager is a plain `CaffeineCacheManager` built via `setCacheSpecification()`, which creates a standard Caffeine `Cache<K,V>`, not a `LoadingCache`:
```java
@Bean(CACHE_MANAGER_CONTRACT)
CacheManager cacheManagerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getContract());
    return caffeineCacheManager;
}
```

**Root cause — failed assumption:** Spring's `@Cacheable` without `sync=true` uses a non-atomic read-compute-write sequence:
1. `cache.get(key)` → miss
2. invoke actual method (DB query)
3. `cache.put(key, result)`

Steps 1–3 are not synchronized per-key. Caffeine's per-key locking (`Cache.get(key, mappingFunction)`) is only engaged when Spring uses `sync=true`. Without it, N concurrent threads all observe a cache miss at step 1 and all proceed to step 2 simultaneously.

**Compounding factor — `unless = "#result == null"`:** If the DB is unreachable (network partition), the query throws an exception rather than returning null. Spring never caches exceptions. Every subsequent request for the same `contractId` also misses the cache and re-attempts the DB query. This is not a one-time race window; it is a persistent amplification for the entire duration of the partition.

**Connection pool exhaustion math:** The web3 DB `statementTimeout` is 3000 ms. HikariCP's default `maximumPoolSize` is 10 (no override found in the repository). The global throttle is 500 RPS. At 500 RPS with each connection held for 3 seconds, the steady-state concurrent connection demand is 500 × 3 = 1500, against a pool of 10. The pool is exhausted within the first second of the burst. All other repository operations (entity lookups, token queries, contract state reads) that also need a DB connection begin queuing or failing immediately.

### Impact Explanation

Connection pool exhaustion is a total-service denial: every JPA repository call in the web3 module blocks waiting for a pool slot. The `requestTimeout` of 10 seconds means all in-flight requests time out. The service becomes unresponsive to all callers, not just those targeting the attacked contract. Severity is **High** — complete availability loss of the web3 API for the duration of the network partition plus pool-drain recovery time.

### Likelihood Explanation

No authentication or special privilege is required. Any caller of `eth_call`, `eth_estimateGas`, or any endpoint that triggers EVM execution against a contract can cause `findRuntimeBytecode()` to be invoked. The attacker needs only to:
- Know or guess any valid (or invalid) `contractId`
- Send concurrent HTTP requests at a rate within the 500 RPS throttle

The network partition is an external precondition, but it is a realistic operational scenario (DB failover, cloud AZ split, misconfigured firewall). The attack is repeatable and requires no exploit tooling beyond a standard HTTP load generator.

### Recommendation

1. **Add `sync=true`** to the `@Cacheable` annotation so Spring delegates to `Cache.get(key, Callable)`, engaging Caffeine's per-key locking and ensuring only one thread executes the DB query per key:
   ```java
   @Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT,
              unless = "#result == null", sync = true)
   ```
2. **Set an explicit `maximumPoolSize`** in HikariCP configuration appropriate to the expected concurrency, and configure `connectionTimeout` to fail fast rather than queue indefinitely.
3. **Add a circuit breaker** (e.g., Resilience4j) around the repository call so that repeated DB failures open the circuit and return a fast error instead of holding connections.
4. **Cache negative results** (empty `Optional`) with a short TTL to prevent the persistent re-query amplification caused by `unless = "#result == null"` during outages.

### Proof of Concept

**Preconditions:**
- Web3 service is running
- Network partition is induced between the web3 service and its PostgreSQL DB (e.g., `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the DB host, or firewall rule)

**Steps:**
```bash
# 1. Induce network partition (DB unreachable, queries hang until 3s statementTimeout)

# 2. Send 200 concurrent requests for the same contractId
#    (well within the 500 RPS throttle)
seq 1 200 | xargs -P200 -I{} curl -s -X POST http://web3-host:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_call","params":[{"to":"0x0000000000000000000000000000000000000001","data":"0x"},"latest"],"id":{}}'

# 3. Observe: all 200 threads simultaneously attempt DB queries (no sync=true)
#    Each holds a HikariCP connection for 3000ms (statementTimeout)
#    Pool (10 connections) is exhausted within milliseconds

# 4. Send any other web3 request (e.g., eth_blockNumber) — it will hang or
#    fail with HikariCP pool timeout, demonstrating cascading failure:
curl -X POST http://web3-host:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":999}'
# Expected: timeout or connection pool exhaustion error
```

**Expected result:** Step 4 fails even though `eth_blockNumber` does not involve `findRuntimeBytecode`, demonstrating the cascading pool exhaustion across all repository operations.
