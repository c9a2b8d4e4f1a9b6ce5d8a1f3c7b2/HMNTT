### Title
Cache-Pressure-Induced `isKeyEvictedFromCache` Fallback Causing Double DB Query Amplification in `findStorageBatch`

### Summary
An unprivileged user can flood the per-contract `slotsPerContract` Caffeine cache (default `maximumSize=1500`) with attacker-controlled slot keys for a target contract, causing the victim's slot key to be continuously rejected by Caffeine's W-TinyLFU admission filter (frequency=0 vs. attacker's high-frequency entries). This triggers the `isKeyEvictedFromCache=true` fallback path on every subsequent `findStorage()` call for the victim's key, resulting in two DB queries per call: a batch query over up to 1500 attacker-injected keys plus an additional single-key fallback query, amplifying DB load and degrading transaction throughput.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, method `findStorageBatch`, lines 85–122.

```
Line 90:  contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
Line 91:  final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
Lines 94–101: boolean isKeyEvictedFromCache = true;
              for (var slot : cachedSlotKeys) {
                  if (wrappedKey.equals(slot)) { isKeyEvictedFromCache = false; }
              }
Line 103: contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots); // always runs
Lines 118–120: if (isKeyEvictedFromCache) {
                   return contractStateRepository.findStorage(contractId.getId(), key); // fallback
               }
```

**Root cause and failed assumption:**

The code assumes that after `putIfAbsent(wrappedKey, EMPTY_VALUE)` at line 90, the key will always be visible in `asMap().keySet()` at line 91. This assumption fails under two conditions:

1. **Caffeine W-TinyLFU admission filter rejection**: When the per-contract cache is at `maximumSize=1500` and all existing entries have high access frequency, Caffeine's admission filter can reject a brand-new entry (frequency=0) during the maintenance task that runs as part of the `putIfAbsent` write operation. The key is added to the write buffer but evicted before it becomes visible in `asMap()`.

2. **Concurrent eviction race**: In a multi-threaded environment, a concurrent `putIfAbsent` from an attacker thread on the same per-contract cache (between line 90 and line 91) triggers Caffeine's maintenance task, which can evict the victim's just-inserted key (frequency=0) in favor of the attacker's high-frequency entries.

In both cases, `isKeyEvictedFromCache` remains `true`, and the code executes **both** the batch query at line 103 (with up to 1500 attacker-controlled keys in the SQL `IN` clause) **and** the fallback single-key query at line 119.

**Cache configuration:**

- `slotsPerContract`: `expireAfterAccess=5m,maximumSize=1500` (default, `CacheProperties.java` line 39)
- `contractState`: `expireAfterWrite=2s,maximumSize=25000` — values expire every 2 seconds, forcing `findStorageBatch` to be called on nearly every request
- `contractSlots`: `expireAfterAccess=5m,maximumSize=3000` — attacker's per-contract cache entry persists for 5 minutes

**Why existing checks fail:**

The global rate limiter (`requestsPerSecond=500`, `ThrottleProperties.java` line 35) is a shared bucket across all callers. An attacker using 500 RPS can fill a 1500-entry cache in 3 seconds. There is no per-contract, per-caller, or per-slot-key rate limiting. The throttle is also gas-based (`gasPerSecond`), and `eth_call` requests with minimal gas (≤10,000 units, scaled to 0 tokens per `scaleGas()` at line 43–46 of `ThrottleProperties.java`) consume **zero gas tokens**, bypassing the gas throttle entirely.

### Impact Explanation

Every `findStorage()` call for the victim's key while the attack is sustained results in:
1. A `findStorageBatch` SQL query with up to 1500 keys in the `IN` clause (line 103) — a query the DB must evaluate even though none of the 1500 attacker keys exist in `contract_state`
2. An additional single-key `findStorage` fallback query (line 119)

This is a 2× query amplification per victim call, with each batch query carrying up to 1500 attacker-injected keys. Under sustained attack against a popular contract, this can exhaust the DB connection pool (`db.statementTimeout=3000ms`), increase query latency for all users, and degrade EVM transaction simulation throughput across the node.

### Likelihood Explanation

The attack requires no credentials, no on-chain funds, and no special knowledge beyond the target contract address. The attacker needs only to:
- Send `eth_call` requests with arbitrary `to` (target contract) and arbitrary `data` (to trigger different storage slot reads via SLOAD opcodes)
- Stay at or below 500 RPS (or use multiple source IPs)
- Periodically re-access their 1500 injected keys to reset the `expireAfterAccess=5m` TTL

The `contractState` cache's 2-second TTL means the victim's slot value is never durably cached, so the attack is self-sustaining: the victim's key will always miss `contractStateCache` and re-enter `findStorageBatch`, where it is again rejected by the admission filter.

### Recommendation

1. **Snapshot the key set before `putIfAbsent`**: Check for key presence before inserting, or snapshot `asMap().keySet()` before calling `putIfAbsent`, so the presence check is consistent with the insertion.
2. **Re-check after insertion**: After `putIfAbsent`, explicitly verify the key is present via `contractSlotsCache.getNativeCache().getIfPresent(wrappedKey)` before setting `isKeyEvictedFromCache = false`, rather than relying on the live `asMap()` view.
3. **Bound per-contract slot injection**: Enforce a per-contract maximum on how many unique slot keys a single caller can inject into the `slotsPerContract` cache within a time window.
4. **Increase `contractState` TTL or use write-through**: A 2-second TTL forces nearly every request through `findStorageBatch`, maximizing exposure to the race condition.
5. **Cap batch query size**: Limit `cachedSlots` to a configurable maximum (e.g., 100) before passing to `findStorageBatch` to bound the SQL `IN` clause size regardless of cache state.

### Proof of Concept

**Preconditions:**
- Target contract address: `0xTARGET`
- Attacker has access to the public `POST /api/v1/contracts/call` endpoint
- Default cache configuration (`slotsPerContract` maximumSize=1500, `contractState` expireAfterWrite=2s)

**Steps:**

1. **Fill the per-contract slots cache** — Send 1500 concurrent `eth_call` requests to `0xTARGET`, each with a different calldata that causes the EVM to execute `SLOAD` on a unique storage slot (e.g., slots `0x0000...0001` through `0x0000...05DC`). This populates the per-contract `slotsPerContract` cache to capacity with attacker-controlled keys.

2. **Build frequency** — Re-access the same 1500 slots repeatedly (e.g., 10× each) to raise their Caffeine frequency counters, making them high-priority in the W-TinyLFU admission filter.

3. **Trigger victim's slot lookup** — Send a legitimate `eth_call` that reads the victim's slot key `0xVICTIM_SLOT` on `0xTARGET`. This calls `findStorageBatch`:
   - `putIfAbsent(0xVICTIM_SLOT, EMPTY_VALUE)` — Caffeine's admission filter rejects the new entry (frequency=0) in favor of the 1500 high-frequency attacker entries
   - `asMap().keySet()` does not contain `0xVICTIM_SLOT`
   - `isKeyEvictedFromCache = true`
   - **Two DB queries execute**: `findStorageBatch` with 1500 attacker keys + `findStorage` for `0xVICTIM_SLOT`

4. **Sustain the attack** — Maintain concurrent attacker requests to keep the cache full and the attacker keys' frequency high. Every 2 seconds (when `contractState` TTL expires), the victim's slot re-enters `findStorageBatch` and the double-query pattern repeats.

5. **Observe impact** — Monitor DB query latency and connection pool utilization; both increase proportionally to the number of victim slots targeted and the sustained attack rate.
