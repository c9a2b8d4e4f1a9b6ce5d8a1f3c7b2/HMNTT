### Title
Missing Entity-Type Check in EVM Address Resolution Allows Contract Accounts to Be Queried via Accounts Crypto-Allowances Endpoint

### Summary
`getAccountCryptoAllowances` resolves the path parameter through `EntityService.getEncodedId`, which — when given an EVM address — executes `entityFromEvmAddressQuery` with no filter on entity type. This allows an unprivileged caller to supply a contract's EVM address and receive that contract's crypto allowances through an endpoint semantically scoped to regular accounts. No authentication or special privilege is required.

### Finding Description
**Exact code path:**

`rest/routes/accountRoute.js` line 17 routes `GET /accounts/:idOrAliasOrEvmAddress/allowances/crypto` to `CryptoAllowanceController.getAccountCryptoAllowances`.

`rest/controllers/cryptoAllowanceController.js` line 77:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```

`rest/service/entityService.js` lines 118–124 — when the input is an EVM address, `getEncodedId` calls `getEntityIdFromEvmAddress`:
```js
return entityId.evmAddress === null
  ? entityId.getEncodedId()
  : await this.getEntityIdFromEvmAddress(entityId, requireResult);
```

`rest/service/entityService.js` lines 22–25 — the backing SQL query:
```sql
select ${Entity.ID}
from ${Entity.tableName}
where ${Entity.DELETED} <> true
  and ${Entity.EVM_ADDRESS} = $1
```

**Root cause:** `entityFromEvmAddressQuery` contains no `type = 'ACCOUNT'` predicate. It matches any non-deleted entity — account or contract — whose `evm_address` column equals the supplied bytes.

**Exploit flow:**
1. Attacker identifies a contract that has a custom (non-long-zero) EVM address stored in the `entity` table and has rows in `crypto_allowance` (contracts can hold HBAR and be granted allowances on Hedera).
2. Attacker sends `GET /api/v1/accounts/<contract_evm_address>/allowances/crypto`.
3. `getEncodedId` detects a 40-hex-char EVM address, calls `getEntityIdFromEvmAddress`, which returns the contract's encoded entity ID with no type validation.
4. `extractCryptoAllowancesQuery` builds `owner = $1` with that ID; `CryptoAllowanceService.getAccountCryptoAllowances` executes the query and returns the contract's allowance rows.
5. The response is a well-formed 200 JSON payload containing the contract's crypto allowances, indistinguishable in structure from a regular account response.

**Why existing checks fail:**
- `isValidAccount` (`entityService.js` lines 60–63) exists but is **never called** in this handler.
- Even if it were called, it only checks row existence (`!isNil(entity)`) — it does not inspect the `type` column, so it would return `true` for contracts as well.
- `entityExistenceQuery` (line 28) selects `Entity.TYPE` but the result is discarded by `isValidAccount`.

### Impact Explanation
Mirror-node consumers that call `/accounts/{id}/allowances/crypto` and assume the response contains only regular-account allowances will silently receive contract-account allowances. Downstream systems that enforce business rules (e.g., "only EOA owners can spend these allowances") may make incorrect authorization or accounting decisions. The endpoint's own OpenAPI description (`openapi.yml` line 269: "Returns information for all crypto allowances for an account") creates a false contract with consumers. Severity is medium: data integrity / incorrect record exposure, not direct fund loss, but the mirror node is explicitly described as an authoritative data source.

### Likelihood Explanation
Exploitation requires zero privileges — it is a plain unauthenticated GET request. The only precondition is knowing (or discovering) the EVM address of a contract that has crypto allowances, which is publicly observable on-chain. Any automated client or integration that iterates known EVM addresses will trigger this inadvertently. Repeatability is unlimited.

### Recommendation
Add a `type = 'ACCOUNT'` predicate to `entityFromEvmAddressQuery` in `rest/service/entityService.js`:
```sql
select ${Entity.ID}
from ${Entity.tableName}
where ${Entity.DELETED} <> true
  and ${Entity.EVM_ADDRESS} = $1
  and ${Entity.TYPE} = 'ACCOUNT'
```
Alternatively, after resolving the encoded ID, call a corrected `isValidAccount` that checks `type = 'ACCOUNT'` (not just existence) and throw a `NotFoundError` if the resolved entity is not an account. The same fix should be applied to the token-allowances handler (`tokenAllowanceController.js` line 69) which has the identical pattern.

### Proof of Concept
**Preconditions:**
- Mirror node is running and accessible.
- A contract entity exists in the `entity` table with a non-null `evm_address` (e.g., `0xaabbcc...`) and `type = 'CONTRACT'`.
- At least one row exists in `crypto_allowance` with `owner` equal to that contract's encoded entity ID and `amount > 0`.

**Steps:**
```
GET /api/v1/accounts/0xaabbcc.../allowances/crypto
```

**Expected (correct) behavior:** `404 Not Found` — no account with that EVM address.

**Actual behavior:** `200 OK` with a JSON body:
```json
{
  "allowances": [
    {
      "amount": <N>,
      "owner": "0.0.<contract_num>",
      "spender": "0.0.<spender_num>",
      ...
    }
  ],
  "links": { "next": null }
}
```
The response exposes the contract's crypto allowances through an endpoint semantically reserved for regular accounts, with no error or warning to the consumer.
