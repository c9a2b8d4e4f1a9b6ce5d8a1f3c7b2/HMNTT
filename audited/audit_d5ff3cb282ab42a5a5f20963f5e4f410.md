### Title
Global Single-Bucket Opcode Rate Limiter Allows Unauthenticated DoS of Entire Opcode Endpoint

### Summary
`ThrottleManagerImpl.throttleOpcodeRequest()` enforces rate limiting via a single application-wide `opcodeRateLimitBucket` with a default capacity of 1 token per second. Because the bucket is a singleton Spring bean with no per-client or per-IP partitioning, a single unauthenticated attacker sending one request per second can permanently monopolize the entire opcode endpoint capacity, causing every legitimate caller to receive HTTP 429 indefinitely.

### Finding Description
**Code path:**

- `ThrottleProperties.java` line 29: `opcodeRequestsPerSecond` defaults to `1`.
- `ThrottleConfiguration.java` lines 47–55: a single `@Bean` named `opcodeRateLimitBucket` is created with `capacity=1` and `refillGreedy(1, Duration.ofSeconds(1))`. No `SynchronizationStrategy` is set (unlike `gasLimitBucket`), and crucially no per-client keying exists.
- `ThrottleManagerImpl.java` lines 52–56: `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` on that single shared bucket. If the token is gone, a `ThrottleException` is thrown immediately.
- `OpcodesController.java` line 61: `throttleManager.throttleOpcodeRequest()` is called before any processing for every `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes` request.

**Root cause:** The bucket is a Spring singleton — one instance for the entire JVM, shared across all concurrent HTTP threads and all remote clients. There is no per-IP, per-session, or per-identity bucket. The failed assumption is that a global 1 RPS limit is sufficient to protect the endpoint; in reality it means the first client to arrive each second wins the single token and all others are rejected.

**Exploit flow:**
1. Attacker discovers or guesses any valid (or even invalid) transaction hash.
2. Attacker sends `GET /api/v1/contracts/results/<hash>/opcodes` with `Accept-Encoding: gzip` at exactly 1 request per second from a single IP.
3. Each request consumes the one available token the instant it refills.
4. Every concurrent or subsequent legitimate request within that second hits `tryConsume(1) == false` and receives HTTP 429.
5. The attacker sustains this indefinitely at negligible cost (1 req/s, no authentication required).

**Why existing checks are insufficient:**
- The `Accept-Encoding: gzip` check (line 60 of `OpcodesController.java`) is trivially satisfied by adding the header.
- The `properties.isEnabled()` guard only prevents exposure when the feature flag is off; once enabled, no auth or IP check exists.
- There is no infrastructure-level per-IP rate limit visible in the web3 gateway/Helm values for this endpoint.

### Impact Explanation
When `hiero.mirror.web3.opcode.tracer.enabled=true`, the opcode tracing endpoint becomes completely unavailable to all legitimate users for as long as the attacker maintains 1 req/s. Developers and tooling relying on `/opcodes` for transaction debugging, audit, or compliance replay are fully denied service. Because the endpoint re-executes transactions on the EVM, operators may have deployed it for critical post-incident analysis; its unavailability during an incident is high-impact.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only 1 HTTP request per second — well within reach of any script or `curl` loop. The attacker does not need a valid transaction hash (a 404 from the service layer still consumes the token after `throttleOpcodeRequest()` succeeds). The attack is trivially repeatable, automatable, and undetectable as malicious traffic since it stays within the documented rate limit.

### Recommendation
Replace the single global bucket with a per-client bucket keyed on the remote IP (or an authenticated identity). Bucket4j supports this via `BucketProxyManager` / `ProxyManager` with a `ConcurrentHashMap` or distributed cache backend. Alternatively, add a secondary per-IP limit at the Spring filter or gateway layer (e.g., a `HandlerInterceptor` that maintains a `ConcurrentHashMap<String, Bucket>` keyed on `request.getRemoteAddr()`). The global bucket can remain as a hard ceiling, but a per-client bucket must prevent one client from consuming the entire global allowance.
