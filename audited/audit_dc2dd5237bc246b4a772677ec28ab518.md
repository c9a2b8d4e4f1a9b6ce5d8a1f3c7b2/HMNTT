### Title
Missing Length Validation on EVM Address Enables Heap Exhaustion via Unbounded `Hex.decodeHex()` Allocation

### Summary
`GraphQlUtils.decodeEvmAddress()` performs no length check before passing the caller-supplied string to `Hex.decodeHex()`, which allocates a byte array of `input.length / 2`. Because `AccountController` routes any `evmAddress` argument directly to `EntityServiceImpl.getByEvmAddressAndType()` → `decodeEvmAddress()` without a size constraint, an unauthenticated attacker can force arbitrarily large heap allocations per request, causing GC pressure or `OutOfMemoryError` that disrupts service for all users.

### Finding Description
**Exact code path:**

`AccountController.account()` [1](#0-0) 
→ `EntityServiceImpl.getByEvmAddressAndType()` [2](#0-1) 
→ `GraphQlUtils.decodeEvmAddress()` [3](#0-2) 

**Root cause:** `decodeEvmAddress()` strips the optional `0x` prefix and immediately calls `Hex.decodeHex(evmAddress)` with zero length validation:

```java
evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
return Hex.decodeHex(evmAddress);   // allocates new byte[input.length / 2]
``` [4](#0-3) 

Apache Commons Codec's `Hex.decodeHex()` allocates `new char[len]` then `new byte[len/2]` synchronously on the calling thread. A 10 MB hex string produces a 5 MB byte array per request with no reuse.

**Why existing checks fail:** The `@Valid` annotation on `AccountInput` triggers Bean Validation, but no `@Size` or `@Pattern` constraint limits the `evmAddress` field length. [5](#0-4)  The `invalidEvmAddress` unit tests only cover odd-length strings, invalid characters, and null — never an oversized input. [6](#0-5)  No GraphQL schema-level `maxLength` directive is applied to the `evmAddress` scalar.

### Impact Explanation
Each malicious request forces a heap allocation proportional to the supplied string length. Concurrent requests from a single attacker (or a small botnet) can saturate the JVM heap, triggering full GC pauses or `OutOfMemoryError`. This disrupts entity resolution for all legitimate users querying accounts or contracts by EVM address — a read path that is critical for transaction routing and contract interaction. Severity: **High (DoS)**.

### Likelihood Explanation
The GraphQL endpoint is unauthenticated and publicly reachable. No rate limiting or request-size guard is visible in the codebase. An attacker needs only a standard HTTP client to send a single POST with a multi-megabyte `evmAddress` string. The attack is trivially repeatable and requires no special knowledge of the system.

### Recommendation
Add an explicit length guard in `decodeEvmAddress()` before decoding:

```java
public static byte[] decodeEvmAddress(String evmAddress) {
    if (evmAddress == null) return ArrayUtils.EMPTY_BYTE_ARRAY;
    evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
    // EVM addresses are exactly 20 bytes = 40 hex chars
    if (evmAddress.length() > 40) {
        throw new IllegalArgumentException("evmAddress exceeds maximum length");
    }
    try {
        return Hex.decodeHex(evmAddress);
    } catch (DecoderException e) {
        throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
    }
}
``` [3](#0-2) 

Additionally, add `@Size(max = 42)` (40 hex chars + optional `0x`) to the `evmAddress` field in `AccountInput`, and configure a global GraphQL max-query-size limit.

### Proof of Concept
```bash
# Generate a 10 MB hex string (20 million 'a' characters = valid hex)
PAYLOAD=$(python3 -c "print('a' * 20_000_000)")

curl -s -X POST http://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\":\"{ account(input: { evmAddress: \\\"$PAYLOAD\\\" }) { balance } }\"}"
```

Each such request forces `Hex.decodeHex()` to allocate a ~10 MB byte array on the heap. Sending this in a tight loop from a single machine will rapidly exhaust the JVM heap, causing GC storms or `OutOfMemoryError`, making the service unavailable to all users.
