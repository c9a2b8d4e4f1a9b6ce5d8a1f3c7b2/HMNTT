### Title
Unauthenticated Metrics Endpoint Exposes Real-Time In-Flight Request Counter as Load-State Oracle When Authentication Is Disabled

### Summary
When `hiero.mirror.rest.metrics.config.authentication` is set to `false`, the `authenticate()` function in `metricsHandler.js` unconditionally returns `true`, making the `/swagger/metrics/` endpoint fully accessible to any unauthenticated external user. This exposes `api_all_request_in_processing_total` (the `inFlightCounter`) — a live gauge of all requests currently being processed — along with CPU, memory, and per-route histograms. An attacker can poll this endpoint at high frequency to observe the exact moment the server's DB connection pool approaches saturation, using it as a real-time oracle to coordinate timing-sensitive attacks such as a network partition at peak load.

### Finding Description

**Exact code path:**

`rest/middleware/metricsHandler.js`, `authenticate()`, lines 137–144:

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {   // ← returns true immediately when authentication is false/falsy
    return true;
  }
  ...
};
```

`metricsMiddleware`, lines 156–164:

```js
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) {          // ← guard bypassed when authentication=false
    res.set('WWW-Authenticate', 'Basic realm="Metrics"');
    return res.status(401).send('Unauthorized');
  }
  return exporter.collect().then(({resourceMetrics}) => {
    res.set('Content-Type', 'text/plain; charset=utf-8');
    res.send(serializer.serialize(resourceMetrics));   // ← full metrics dump served
  });
}
```

`inFlightCounter` lifecycle, lines 169 and 186:

```js
inFlightCounter.add(1);          // incremented on every non-metrics request entry
...
res.on('finish', () => {
  inFlightCounter.add(-1);       // decremented only after response is fully sent
```

**Root cause:** The `authenticate()` function treats `authentication: false` as a valid operator opt-out and returns `true` with no further checks. There is no secondary guard, no IP allowlist, no rate limit, and no scrape-token mechanism on the metrics path. The `inFlightCounter` is an `UpDownCounter` that reflects the exact number of requests currently held open inside the Node.js process — including those queued waiting for a free DB pool connection (default pool size: 10, per `docs/configuration.md` line 556). When the pool is saturated, all new requests block inside the DB layer while still holding an incremented slot in `inFlightCounter`, making the gauge a reliable proxy for pool saturation state.

**Failed assumption:** The design assumes that operators who set `authentication: false` are doing so in a network-isolated environment (e.g., internal Kubernetes scrape). The code provides no enforcement of that assumption.

### Impact Explanation

An external attacker with HTTP access to the REST port gains a real-time, sub-second-resolution view of:

- `api_all_request_in_processing_total` — exact count of in-flight requests (DB pool saturation proxy)
- `nodejs_process_cpu_usage_percentage` — CPU load
- `nodejs_process_memory_*` — heap and RSS
- Per-route duration histograms — latency distribution per endpoint

The `inFlightCounter` value crossing the configured `maxConnections` threshold (default 10) is a reliable signal that the DB pool is fully occupied. An attacker can use this to:

1. Confirm that a flood of their own requests has saturated the pool.
2. Identify the precise window when legitimate requests are queuing and the system is most vulnerable to cascading failures.
3. Coordinate a network partition (e.g., BGP hijack, firewall rule injection, or cloud security-group manipulation) at the exact moment of saturation, maximizing the probability that in-flight DB transactions are interrupted mid-query, causing data inconsistency or prolonged recovery.

Severity: **Medium–High** (information disclosure enabling precise attack timing; direct data corruption requires a separate network-level primitive).

### Likelihood Explanation

**Preconditions:**
- `authentication` must be set to `false` (non-default; default is `true` per `docs/configuration.md` line 569).
- The REST port must be reachable from the attacker (common in misconfigured cloud deployments where the metrics path is on the same port as the API).

**Feasibility:** Polling `/swagger/metrics/` with `curl` in a loop requires zero privileges and zero tooling beyond a basic HTTP client. The attacker does not need to know the DB pool size in advance — they can infer it empirically by observing the counter plateau under self-generated load. The attack is fully repeatable and automatable.

**Realistic scenario:** A DevOps operator disables metrics authentication to simplify a Prometheus scrape configuration in a staging or production cluster that is inadvertently internet-exposed. This is a documented, supported configuration path with no warning emitted by the server.

### Recommendation

1. **Remove the unauthenticated bypass entirely.** If scrape-token-free access is needed, restrict it to a separate internal-only port or a dedicated `/metrics` listener bound to `127.0.0.1`.
2. **Emit a startup warning** (or refuse to start) when `authentication: false` and the server is bound to `0.0.0.0`.
3. **Add a scrape IP allowlist** as an additional guard independent of the `authentication` flag.
4. **Rate-limit the metrics endpoint** regardless of authentication state to prevent high-frequency polling.
5. **Consider separating the metrics port** from the API port so network policy can enforce access control at the infrastructure layer.

### Proof of Concept

```bash
# 1. Confirm authentication is disabled in config:
#    hiero.mirror.rest.metrics.config.authentication: false

# 2. Poll the metrics endpoint as an unauthenticated user:
while true; do
  curl -s http://<REST_HOST>:5551/swagger/metrics/ \
    | grep 'api_all_request_in_processing_total'
  sleep 0.5
done

# 3. In a separate terminal, flood the API to saturate the DB pool (default: 10 connections):
for i in $(seq 1 50); do
  curl -s "http://<REST_HOST>:5551/api/v1/transactions?limit=100" &
done

# 4. Observe the counter in the polling terminal spike above 10,
#    confirming pool saturation. Use this moment to trigger the partition:
#    e.g., inject a firewall rule dropping traffic to the DB host.

# Expected output when pool is saturated:
