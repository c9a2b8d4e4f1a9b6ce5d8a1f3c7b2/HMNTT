### Title
Cache-Bypass DoS via Unbounded Block Number in `findByBlockType()` / `findByIndex()`

### Summary
`RecordFileServiceImpl.findByBlockType()` passes any attacker-supplied numeric block number directly to `recordFileRepository.findByIndex()` without checking whether the number is within the range of blocks that actually exist. Because `RecordFileRepository.findByIndex()` is annotated `@Cacheable(unless = "#result == null")` and Spring unwraps `Optional.empty()` to `null` before evaluating the `unless` condition, cache misses are never stored. An unprivileged attacker can therefore force an unbounded number of distinct database queries by cycling through the ~9.2 × 10¹⁸ valid `long` values that will never match a real block.

### Finding Description

**Exact code path:**

`RecordFileServiceImpl.findByBlockType()` — `web3/src/main/java/org/hiero/mirror/web3/service/RecordFileServiceImpl.java`, lines 19–28:

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (block == BlockType.EARLIEST) { ... }
    else if (block == BlockType.LATEST) { ... }
    else if (block.isHash()) { ... }

    return recordFileRepository.findByIndex(block.number());   // line 28 — no range check
}
```

`BlockType.of()` — `web3/src/main/java/org/hiero/mirror/web3/viewmodel/BlockType.java`, lines 59–66 / 73–79 — only validates that the value fits in a `long` (decimal up to 20 digits, hex up to 16 hex digits). Any value in `[0, Long.MAX_VALUE]` is accepted.

`RecordFileRepository.findByIndex()` — `web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 27–29:

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
           unless = "#result == null")
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
```

**Root cause / failed assumption:**

Spring Cache evaluates `#result` for `Optional`-returning methods against the *unwrapped* value. When the block does not exist the method returns `Optional.empty()`, Spring unwraps it to `null`, the condition `#result == null` is `true`, and the result is **not** stored in the cache. Every subsequent request for the same (or any other) non-existent index therefore bypasses the cache and issues a fresh `SELECT … WHERE index = ?` against the database.

There is no guard that compares `block.number()` against the current latest block index before issuing the query.

### Impact Explanation

An attacker can drive an arbitrarily high rate of database queries with zero authentication. Each request with a distinct large block number (e.g., `Long.MAX_VALUE`, `Long.MAX_VALUE - 1`, …) executes `SELECT * FROM record_file WHERE index = <value>` and the result is never cached. Sustained at scale this exhausts the database connection pool, increases query latency for legitimate users, and can render the web3 API unavailable — a classic cache-bypass denial-of-service. Severity: **Medium–High** (availability impact, no authentication required, trivially repeatable).

### Likelihood Explanation

The attack requires only the ability to send HTTP requests to the public JSON-RPC endpoint (e.g., `eth_getBlockByNumber` with a large hex block number). No credentials, tokens, or special knowledge of the system are needed. The attacker has ~9.2 × 10¹⁸ distinct non-existent values to cycle through, making IP-based blocking ineffective without rate limiting. The existing test `testFindByBlockTypeIndexOutOfRange` confirms the code path reaches the database for out-of-range values and returns empty without error.

### Recommendation

1. **Add an upper-bound guard in `findByBlockType()`**: fetch the latest block index once (already cached via `findLatest()`) and reject numeric block numbers that exceed it before calling `findByIndex()`.
2. **Cache negative results**: change the `unless` condition on `findByIndex` to also cache empty Optionals (e.g., use a dedicated negative-result cache with a short TTL), so repeated queries for the same non-existent index do not hit the database.
3. **Apply rate limiting** at the API gateway or filter layer for block-lookup endpoints.

### Proof of Concept

```
# Send N requests with distinct large block numbers (no auth required)
for i in $(seq 1 10000); do
  curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"block\": \"0x$(printf '%016x' $((9223372036854775807 - i)))\"}" &
done
wait
```

Each request resolves `BlockType.of("0x7fffffffffff...")` → `block.number()` = a unique large `long` → `findByIndex(<unique_large_value>)` → cache miss (empty Optional, not stored) → `SELECT * FROM record_file WHERE index = <unique_large_value>` executed against the database → `Optional.empty()` returned. With sufficient concurrency the database connection pool is saturated. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
