### Title
Cache Key Collision via `Arrays.hashCode()` Enables Cache Poisoning in `findByEvmAddressOrAliasAndDeletedIsFalse`

### Summary
`findByEvmAddressOrAliasAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` as its cache key, which internally calls `Arrays.hashCode(byte[])` — a 32-bit non-cryptographic hash. Two distinct byte arrays that produce the same `Arrays.hashCode()` value share the same cache key in `CACHE_NAME_ALIAS`. Because `Optional.empty()` is not `null`, the `unless = "#result == null"` guard does not prevent caching of empty results, allowing an attacker to poison the cache entry for a legitimate alias by first querying a crafted colliding alias.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 39–49:
```java
@Cacheable(
    cacheNames = CACHE_NAME_ALIAS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

`common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```

**Root cause:** `Arrays.hashCode()` is a 32-bit polynomial hash (`result = 31 * result + b` for each byte). It has a trivially small output space (2³² values) and is not collision-resistant. Any two byte arrays `A` and `B` where `Arrays.hashCode(A) == Arrays.hashCode(B)` map to the same Caffeine cache key.

**Why the guard fails:** The annotation specifies `unless = "#result == null"`. The method return type is `Optional<Entity>`. When no entity is found, the method returns `Optional.empty()`, which is a non-null Java object. Spring evaluates `#result == null` as `false`, so `Optional.empty()` **is cached**. This means a lookup for a non-existent alias B will store `Optional.empty()` at key `hashCode(B)`. If `hashCode(B) == hashCode(A)` for a valid alias A, the cache entry for A is now poisoned with an empty result.

**Exploit flow:**
1. Attacker identifies a valid alias `A` (e.g., a known account's EVM address or public key alias).
2. Attacker computes a byte array `B ≠ A` such that `Arrays.hashCode(B) == Arrays.hashCode(A)`. Finding such a collision is computationally trivial for a 32-bit hash — a birthday attack over random 20-byte inputs finds a collision in ~65,536 attempts on average.
3. Attacker sends a contract call referencing alias `B`. The DB returns no entity; `Optional.empty()` is cached at key `hashCode(A)`.
4. Within the 1-second TTL window (`expireAfterWrite=1s`), any legitimate request for alias `A` hits the cache and receives `Optional.empty()` — the entity appears not to exist.
5. EVM execution proceeds as if the account/contract at alias `A` does not exist, causing the call to fail or behave incorrectly.

### Impact Explanation
Any entity lookup routed through `findByEvmAddressOrAliasAndDeletedIsFalse` (called from `CommonEntityAccessor.get(Bytes alias, ...)`) can be transiently denied. During the poisoned window, contracts or accounts at the targeted alias are invisible to the EVM, causing contract calls that reference them to revert or return incorrect state. This is a griefing/DoS primitive with no direct economic gain for the attacker but potential disruption to users interacting with the targeted alias within the TTL window.

### Likelihood Explanation
The attack requires no privileges — any caller of the public JSON-RPC `eth_call` / `eth_estimateGas` endpoint can supply an arbitrary `to` or calldata address. Finding a 32-bit `Arrays.hashCode` collision is trivial offline (brute-forceable in milliseconds on commodity hardware). The 1-second TTL (`expireAfterWrite=1s,maximumSize=10000`) limits the damage window per poisoning attempt, but the attacker can repeat the attack continuously at low cost, maintaining a persistent denial-of-service against a specific alias.

### Recommendation
Replace `Arrays.hashCode(byte[])` with a collision-resistant key. The simplest correct fix is to use the byte array itself (wrapped in a type with proper `equals`/`hashCode`) or a hex-encoded string as the cache key:

```java
key = "T(org.bouncycastle.util.encoders.Hex).toHexString(#alias)"
// or
key = "T(java.util.Arrays).toString(#alias)"
```

Alternatively, use `java.util.HexFormat.of().formatHex(#alias)` (Java 17+). This produces a unique string key per distinct byte array with no collision risk. Additionally, consider changing `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` to avoid caching empty Optionals, which eliminates the poisoning vector entirely regardless of key scheme.
