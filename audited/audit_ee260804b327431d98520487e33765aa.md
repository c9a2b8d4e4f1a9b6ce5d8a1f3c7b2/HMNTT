### Title
Unauthenticated Direct HTS Precompile Calls Bypass Gas-Proportional DB Load Protection in `constructServiceParameters()`

### Summary
`constructServiceParameters()` in `ContractController.java` accepts any `to` address — including the HTS precompile at `0x0000000000000000000000000000000000000167` — without restriction, passing it directly to the EVM executor. The gas-based throttle consumes tokens proportional to declared/used EVM gas, but HTS precompile operations (e.g., `getTokenInfo`, `getFungibleTokenInfo`, `getNonFungibleTokenInfo`) trigger multiple database queries per call whose cost is not proportional to EVM gas consumed. Any unauthenticated user can exploit this to generate disproportionate database load per request.

### Finding Description

**Exact code path:**

In `ContractController.java`, `constructServiceParameters()` (lines 53–90), the `to` field is parsed unconditionally:

```java
// lines 60-64
if (request.getTo() == null || request.getTo().isEmpty()) {
    receiver = Address.ZERO;
} else {
    receiver = Address.fromHexString(request.getTo());  // no precompile filtering
}
```

There is no check that `receiver` is not a system precompile address. The resulting `ContractExecutionParameters` is passed directly to `contractExecutionService.processCall(params)`.

**Throttle mechanism (ThrottleManagerImpl.java, lines 37–48):**

```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) { ... }
    else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) { ... }
    // optional configurable request filters (default: LOG only)
}
```

Gas is consumed upfront based on `request.getGas()` (default 15,000,000), then unused gas is restored after execution. The effective throttle is thus based on actual EVM gas used. HTS precompile calls like `getTokenInfo` have a fixed, relatively low EVM gas cost, but each call triggers multiple database queries: token entity lookup, token details, custom fees (fixed/fractional/royalty), token keys (up to 7 types), treasury account, auto-renew account — 6–10+ queries per call.

**Root cause:** The gas throttle is calibrated to EVM computation cost, not to database I/O cost. HTS precompile calls have a low EVM gas cost but high DB query count, creating a disproportionate ratio. No precompile address blocklist exists in `constructServiceParameters()`.

**Confirmed by test evidence:** `ContractCallServiceTest.java` (lines 917–932) confirms that calling `0x167` with empty data throws `MirrorEvmTransactionException`, but with a valid function selector and token address the call executes and triggers the full DB query chain (confirmed by `ContractCallServicePrecompileReadonlyTest.java` lines 947–1015 showing `getTokenInfo` executing multiple DB lookups including custom fees and token keys).

### Impact Explanation

An attacker can send repeated POST requests to `/api/v1/contracts/call` with `to=0x0000000000000000000000000000000000000167` and `data` encoding `getTokenInfo(address)` for known token addresses. Each request triggers 6–10+ database queries while consuming minimal EVM gas. At the default gas-per-second limit, hundreds to thousands of such calls per second are possible, each multiplying into thousands of DB queries per second. This degrades database performance for all users of the mirror node (REST API, other web3 calls) without any economic cost to the attacker. No authentication or privileged access is required.

### Likelihood Explanation

The exploit requires only: (1) knowledge of any valid token address on the network (publicly discoverable via `/api/v1/tokens`), and (2) the ability to send HTTP POST requests. No wallet, credentials, or on-chain funds are needed. The attack is trivially scriptable, repeatable, and stateless. The default throttle configuration (requests per second + gas per second) does not prevent the DB amplification because the gas cost of HTS precompile reads is low relative to their DB footprint.

### Recommendation

1. **Add precompile address filtering in `constructServiceParameters()`**: Reject or specially handle requests where `receiver` is a known Hedera system precompile address (`0x167`, `0x168`, `0x169`) when called directly (not via a deployed contract intermediary).
2. **Introduce a DB-query-aware throttle**: Track and limit the number of database queries triggered per time window, independent of EVM gas.
3. **Apply stricter per-IP rate limiting** at the API gateway layer for `/api/v1/contracts/call`.
4. **Require a minimum gas cost floor for HTS precompile operations** that reflects actual DB I/O cost, so the gas throttle remains effective.
