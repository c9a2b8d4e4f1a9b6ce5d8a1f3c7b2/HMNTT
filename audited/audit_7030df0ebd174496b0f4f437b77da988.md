### Title
Missing Per-IP Rate Limit on `/graphql/alpha` Enables Sustained DB Connection Exhaustion via `account()` Queries

### Summary
The Traefik middleware chain for the `/graphql/alpha` endpoint enforces only a concurrency cap (`inFlightReq: amount: 5` per source IP) with no temporal `rateLimit` middleware. An unprivileged attacker can continuously cycle 5 concurrent `account()` queries from a single IP at maximum throughput, sustaining unthrottled DB load. The sister Rosetta service explicitly adds a `rateLimit` middleware that GraphQL omits, confirming this is an unintentional gap.

### Finding Description

**Exact code path:**

The GraphQL Traefik middleware is defined in `charts/hedera-mirror-graphql/values.yaml` lines 135–145:

```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - inFlightReq:
      amount: 5
      sourceCriterion:
        ipStrategy:
          depth: 1
  - retry:
      attempts: 3
      initialInterval: 100ms
```

No `rateLimit` entry is present. Compare to the Rosetta service (`charts/hedera-mirror-rosetta/values.yaml` lines 149–163) which adds:

```yaml
  - rateLimit:
      average: 10
      sourceCriterion:
        requestHost: true
```

The `account()` query resolver at `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java` lines 33–58 calls `entityService.getByIdAndType(...)` / `getByAliasAndType(...)` / `getByEvmAddressAndType(...)` synchronously, each acquiring a HikariCP connection for the duration of the DB query.

**Root cause:** `inFlightReq` is a *concurrency* cap — it limits how many requests from one IP are simultaneously in-flight. It does not limit the *rate* at which requests arrive over time. As soon as one of the 5 slots completes, the attacker immediately sends the next request. There is no cooldown, no token bucket, and no per-IP request-per-second ceiling.

**Application-level controls also insufficient:** `GraphQlConfiguration.java` lines 42–48 installs `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, but these are parse-time structural checks. They do not throttle request throughput. No Spring Security filter, no Bucket4j bean, and no custom rate-limiting interceptor exists in the `graphql` module's config package (`GraphQlConfiguration.java`, `LoggingFilter.java`, `MetricsConfiguration.java`, `CustomExceptionResolver.java`).

### Impact Explanation

Each `account()` query holds a HikariCP connection for its execution duration. With `inFlightReq=5` per IP and no rate limit, a single attacker IP can sustain 5 concurrent DB connections at 100% duty cycle indefinitely. With a small number of attacker IPs (or a small HikariCP pool), the pool is exhausted, causing legitimate requests to queue and time out. The Prometheus alert `GraphQLHighDBConnections` fires only after 5 minutes above 75% utilization — well after service degradation has begun. The `circuitBreaker` only trips on error ratios, not on connection saturation, so it provides no protection here.

### Likelihood Explanation

No authentication or API key is required to reach `/graphql/alpha`. The exploit requires only an HTTP client capable of sending parallel POST requests. A single machine with a standard HTTP library can sustain 5 concurrent connections indefinitely. The attack is trivially scriptable, repeatable, and requires zero privileges.

### Recommendation

Add a `rateLimit` middleware to the GraphQL Traefik middleware chain in `charts/hedera-mirror-graphql/values.yaml`, mirroring the Rosetta configuration:

```yaml
  - rateLimit:
      average: 10
      burst: 20
      sourceCriterion:
        ipStrategy:
          depth: 1
```

Additionally, consider adding a HikariCP `connectionTimeout` and a DB-level `statement_timeout` in the GraphQL Spring datasource configuration to bound the maximum duration any single query can hold a connection.

### Proof of Concept

```bash
# Send 5 concurrent account() queries in a tight loop from one IP
for i in $(seq 1 5); do
  curl -s -X POST https://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input:{entityId:{num:1}}) { id balance } }"}' &
done
wait
# Repeat immediately in a loop — no rate limit prevents re-submission
# Monitor: hikaricp_connections_active{application="graphql"} will saturate
```

With no `rateLimit`, each completed slot is immediately refilled. Scaling to 3–4 attacker IPs exhausts a default-sized HikariCP pool, causing `connectionTimeoutMillis` errors for all other users. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    Optional<Account> account(@Argument @Valid AccountInput input) {
        final var alias = input.getAlias();
        final var evmAddress = input.getEvmAddress();
        final var entityId = input.getEntityId();
        final var id = input.getId();

        validateOneOf(alias, entityId, evmAddress, id);

        if (entityId != null) {
            return entityService
                    .getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        if (alias != null) {
            return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
        }

        if (evmAddress != null) {
            return entityService
                    .getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        throw new IllegalStateException("Not implemented");
    }
