### Title
Unauthenticated IP Spoofing in `TracingMiddleware` Enables Complete Erasure of Attacker Identity from Info-Level Transaction Logs

### Summary
`TracingMiddleware` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` and `X-Forwarded-For` headers to determine the client IP address, with no validation that the request transited a trusted proxy. Separately, any request whose path exactly matches an entry in `internalPaths` (including `/health/liveness`) is logged only at `Debug` level. An unprivileged external attacker can combine both behaviors to ensure their real IP address never appears in Info-level logs: health-check probes are silently downgraded to Debug, and all transaction queries are attributed to an arbitrary spoofed IP.

### Finding Description
**Code path:**

`rosetta/app/middleware/trace.go`, `TracingMiddleware()` (lines 43–61) and `getClientIpAddress()` (lines 63–74).

```
// line 46 – IP is resolved before the handler runs
clientIpAddress := getClientIpAddress(request)

// line 55-59 – log level decided purely by path string
if internalPaths[path] {
    log.Debug(message)   // /health/liveness → Debug only
} else {
    log.Info(message)    // all other paths → Info
}
```

```
// lines 63-74 – no trusted-proxy check whatsoever
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // attacker-controlled
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // attacker-controlled
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }
    return ipAddress
}
```

**Root cause:** `getClientIpAddress` applies a priority chain that places two fully attacker-controlled HTTP headers (`X-Real-IP`, `X-Forwarded-For`) above the kernel-verified `RemoteAddr`. There is no allowlist of trusted upstream proxies, no CIDR check, and no stripping of these headers at an ingress layer enforced by the middleware itself.

**Exploit flow:**

1. Attacker sends `GET /health/liveness` (no special headers needed). Path matches `internalPaths` exactly → `log.Debug`. The attacker's real `RemoteAddr` is recorded only at Debug level, invisible in a default Info deployment.
2. Attacker sends `POST /block/transaction` (or any Rosetta transaction endpoint) with header `X-Real-IP: 203.0.113.1` (or any chosen decoy IP). `getClientIpAddress` returns `"203.0.113.1"`. The Info log entry reads: `203.0.113.1 POST /block/transaction (200) in 4ms`.
3. Repeat step 2 with rotating decoy IPs across multiple transaction queries.

**Why existing checks fail:** There are no existing checks. The middleware has no concept of a trusted proxy list. The `internalPaths` map is a compile-time constant that any external client can match by sending the exact path string. No authentication or network-layer restriction gates access to `/health/liveness` in this middleware.

### Impact Explanation
The attacker's real IP address is completely absent from Info-level logs for the entire session: health-check requests are downgraded to Debug, and transaction queries carry a fabricated IP. Forensic reconstruction of "who queried which transactions and when" becomes impossible from Info logs alone. In a Rosetta deployment used for blockchain analytics or compliance monitoring, this breaks the audit trail for transaction query patterns — an attacker can enumerate sensitive transaction data (balances, counterparties, timing) without leaving a traceable identity in operational logs.

### Likelihood Explanation
Preconditions are zero: no account, no token, no network position is required. Any internet-reachable Rosetta node is vulnerable. The headers `X-Real-IP` and `X-Forwarded-For` are standard HTTP headers settable by any HTTP client (`curl`, Python `requests`, etc.). The attack is trivially repeatable and scriptable. The only constraint is that the deployment must be running at Info log level (the default, as set in `main.go` line 156: `configLogger("info")`).

### Recommendation
1. **Trusted-proxy allowlist for IP resolution:** Only accept `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` falls within a configured set of trusted proxy CIDRs (e.g., the cluster's ingress CIDR). Otherwise fall back directly to `r.RemoteAddr`.
2. **Log both real and forwarded IPs:** Always record `r.RemoteAddr` in the log message alongside any forwarded IP, so the kernel-verified address is always present.
3. **Consider rate-limiting or auth on health endpoints** if the deployment is internet-facing, to prevent the Debug-suppression path from being used as a reconnaissance channel.

### Proof of Concept
```bash
# Step 1: probe the server without leaving Info-level traces
curl -s http://<rosetta-host>:<port>/health/liveness
# → logged only at Debug: "x.x.x.x GET /health/liveness (200) in Xms"
# → invisible in default Info log output

# Step 2: query transactions with a spoofed IP
curl -s -X POST http://<rosetta-host>:<port>/block/transaction \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 203.0.113.99" \
  -d '{"network_identifier":{...},"block_identifier":{...},"transaction_identifier":{...}}'
# → Info log records: "203.0.113.99 POST /block/transaction (200) in Xms"
# → attacker's real IP never appears in Info logs

# Repeat step 2 with different X-Real-IP values to scatter attribution
curl ... -H "X-Real-IP: 198.51.100.1" ...
curl ... -H "X-Real-IP: 192.0.2.55" ...
```

The attacker's actual `RemoteAddr` appears nowhere in Info-level output for the entire session.
