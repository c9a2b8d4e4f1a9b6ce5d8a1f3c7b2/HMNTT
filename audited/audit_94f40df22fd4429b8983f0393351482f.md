### Title
Unbounded `req.originalUrl` Logged Verbatim in `handleError()` Enables Log Flooding / Storage Exhaustion

### Summary
In `rest/middleware/httpErrorHandler.js`, the `handleError()` function interpolates `req.originalUrl` directly into both `logger.warn()` and `logger.error()` calls with no length check or truncation. The custom logger in `rest/logger.js` imposes no maximum line size and writes lines of arbitrary length directly to stdout. An unprivileged attacker can repeatedly send requests with maximally-long URLs to any error-triggering endpoint, flooding log storage and crowding out legitimate gossip-event log entries.

### Finding Description

**Exact code path:**

`rest/middleware/httpErrorHandler.js`, `handleError()`, lines 33–35 and 39:
```js
logger.warn(
  `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode} ${err.constructor.name} ${errorMessage}`
);
// ...
logger.error(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode}`, detailedMessage);
```
`req.originalUrl` is the raw, unmodified URL string as received by Express. It is interpolated into the log string with no `.substring()`, `.slice()`, or any length guard.

**Logger behavior (`rest/logger.js`, lines 73–88):**
```js
#write(line) {
  const bytes = Buffer.byteLength(line);
  if (this.#position + bytes > this.#bufferSize) {
    this.#flush();
  }
  // Line is larger than entire buffer so write directly
  if (bytes > this.#bufferSize) {
    process.stdout.write(line);   // ← no truncation, writes full line
    return;
  }
  this.#buffer.write(line, this.#position);
  this.#position += bytes;
}
```
When a log line exceeds the 4096-byte buffer, it is written in full to stdout with no cap. There is no maximum-line-size enforcement anywhere in the logger.

**Root cause:** The failed assumption is that `req.originalUrl` is bounded to a safe length before being embedded in a log string. No such bound exists in the application code.

**Why existing checks are insufficient:**

- Node.js's HTTP parser enforces `--max-http-header-size` (default 16 384 bytes) on the combined request-line + headers. This limits a single URL to roughly 16 KB under default settings, but this flag is trivially overridden at process startup (e.g., `node --max-http-header-size=104857600 server.js` raises it to 100 MB). There is no application-level enforcement.
- The `qs` query parser (`rest/middleware/requestHandler.js`, line 16–19) limits the *number* of repeated query parameters (`arrayLimit`, `throwOnLimitExceeded: true`) but does not limit the *total byte length* of the URL string.
- The OpenAPI validator (`rest/server.js`, line 63–65) is conditional (`openApiValidatorEnabled || isTestEnv()`) and is not guaranteed to be active in production.
- Neither `handleError()` nor the logger performs any truncation of the message string.

**Exploit flow:**

1. Attacker crafts a GET request to any valid API path (e.g., `/api/v1/transactions`) with a query string padded to the maximum permitted URL length.
2. The request triggers any error path (e.g., an invalid parameter causes `InvalidArgumentError`, or an unknown route causes `NotFoundError`), routing execution to `handleError()`.
3. `handleError()` calls `logger.warn(... req.originalUrl ...)` or `logger.error(... req.originalUrl ...)`.
4. The logger's `#write()` detects the line exceeds the 4096-byte buffer and calls `process.stdout.write(line)` with the full, untruncated string.
5. The process's stdout (typically piped to a log aggregator or file) receives a log line as large as the URL.
6. Repeated at high request rate, log storage fills rapidly.

### Impact Explanation

Log storage exhaustion prevents new log entries from being written, including gossip-event and consensus-critical log lines. On disk-backed log systems, a full disk can crash the mirror-node process or its log aggregator. Even short of full exhaustion, the signal-to-noise ratio in logs is destroyed, making incident detection and forensic analysis impossible. Because `handleError()` is the catch-all error handler for every API route, every error-triggering request (including trivially-induced 400/404 responses) passes through this code path.

### Likelihood Explanation

No authentication is required. Any internet-accessible mirror-node REST API endpoint is reachable by an anonymous attacker. The attack requires only standard HTTP tooling (`curl`, `ab`, `wrk`). The trigger condition (any error response) is trivially satisfied by sending a request with an invalid parameter. The attack is repeatable at the rate the server accepts connections, and a single attacker with modest bandwidth can sustain it indefinitely.

### Recommendation

Truncate `req.originalUrl` before embedding it in log strings. Apply a safe maximum (e.g., 512 characters) at the point of interpolation:

```js
const safeUrl = req.originalUrl?.length > 512
  ? req.originalUrl.substring(0, 512) + '…'
  : req.originalUrl;

logger.warn(`${req.ip} ${req.method} ${safeUrl} in ${elapsed} ms: ...`);
logger.error(`${req.ip} ${req.method} ${safeUrl} in ${elapsed} ms: ...`, detailedMessage);
```

Additionally, add a URL-length enforcement middleware early in the stack (before route handlers) that rejects requests whose `req.originalUrl` exceeds a configured maximum, returning HTTP 414 URI Too Long. This prevents the long URL from ever reaching `handleError()`.

### Proof of Concept

```bash
# Generate a URL with a ~15 KB query string (within Node.js default limits)
LONG_PARAM=$(python3 -c "print('x' * 15000)")

# Send to any endpoint that will produce an error (invalid param triggers InvalidArgumentError → handleError)
curl -v "http://<mirror-node-host>/api/v1/transactions?timestamp=INVALID&pad=${LONG_PARAM}"

# Observe the log output: a single warn line of ~15 KB is written
# Repeat at high rate to exhaust log storage:
for i in $(seq 1 10000); do
  curl -s "http://<mirror-node-host>/api/v1/transactions?timestamp=INVALID&pad=${LONG_PARAM}" &
done
wait
```

Each iteration writes a ~15 KB log line. At 10 000 requests, ~150 MB of log data is generated. With `--max-http-header-size` raised or a permissive reverse proxy, the per-line size scales proportionally.
