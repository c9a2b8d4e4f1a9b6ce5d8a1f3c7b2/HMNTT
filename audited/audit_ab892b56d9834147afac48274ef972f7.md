### Title
Cache Key Injection via Raw Query String Enables Unbounded Parse/Validate CPU Amplification

### Summary
`CachedPreparsedDocumentProvider` uses the raw, un-normalized query string as the Caffeine cache key. An unauthenticated attacker can generate an unbounded stream of syntactically distinct but semantically identical queries (varying aliases, operation names, or field order) that each produce a unique cache key, causing every request to bypass the cache and trigger a full GraphQL parse-and-validate cycle. No rate limiting exists on the GraphQL module, making this trivially repeatable.

### Finding Description

**Exact code location:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:

```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```

`executionInput.getQuery()` returns the raw, verbatim query string submitted by the client. No normalization (whitespace stripping, AST-based canonical form, etc.) is applied before it is used as the Caffeine cache key.

**Root cause:** The cache assumes that semantically equivalent queries will share the same string representation. This assumption is false: GraphQL allows aliases, arbitrary whitespace, different operation names, and reordered fields that all produce the same execution result but different string hashes.

**Exploit flow:**

1. Attacker sends a flood of POST requests to `/graphql/alpha` (no authentication required — no Spring Security config exists in the `graphql` module).
2. Each request carries a query that is semantically identical to `{ account(input: {entityId: {shard:0,realm:0,num:2}}) { balance } }` but with a unique alias or operation name, e.g.:
   - `query A1 { account(...) { balance } }`
   - `query A2 { account(...) { balance } }`
   - `{ x1: account(...) { balance } }`
   - `{ x2: account(...) { balance } }`
3. Each unique string misses the Caffeine cache (`maximumSize=1000`, `expireAfterWrite=1h`).
4. The cache miss triggers `parseAndValidateFunction.apply(executionInput)` — a full lexer + parser + validation pass — on every single request.
5. Once the cache fills to 1000 entries, Caffeine evicts old entries; the attacker's new unique strings continue to thrash the cache, preventing any legitimate query from being served from cache.

**Why existing checks are insufficient:**

| Check | Location | Why it fails |
|---|---|---|
| `maxCharacters(10000)` | `GraphQlConfiguration.java:34` | Limits per-query size; does not limit request rate or number of unique queries |
| `maxTokens(1000)` | `GraphQlConfiguration.java:34` | Same — per-query limit only |
| `maxWhitespaceTokens(1000)` | `GraphQlConfiguration.java:34` | Same — per-query limit only |
| `MaxQueryComplexityInstrumentation(200)` | `GraphQlConfiguration.java:43` | Runs **after** parse+validate; does not prevent the CPU cost of parsing |
| `MaxQueryDepthInstrumentation(10)` | `GraphQlConfiguration.java:44` | Same |
| `maximumSize=1000` cache cap | `CacheProperties.java:15` | Limits memory, but enables cache thrashing — attacker keeps the cache full of one-time-use entries |
| No rate limiter | `graphql/` module | `ThrottleConfiguration` / `ThrottleManagerImpl` exist only in the `web3` module; the `graphql` module has no equivalent |

### Impact Explanation
Every cache miss forces a full parse-and-validate cycle on the JVM. With no rate limiting and no authentication, an attacker can saturate the GraphQL worker thread pool with parse/validate work. The `GraphQLHighCPU` alert threshold is 80% (`values.yaml:197`), but the question threshold is 30% above baseline — achievable with a modest request rate from a single client. Legitimate users experience increased latency or timeouts as CPU is consumed by attacker-controlled parse work.

### Likelihood Explanation
The endpoint is publicly accessible (`/graphql/alpha`), requires no credentials, and accepts arbitrary POST bodies. Generating unique query strings requires only a loop counter appended to an operation name — no special knowledge, tooling, or privilege is needed. The attack is repeatable indefinitely and can be automated with a single `curl` loop or any HTTP benchmarking tool.

### Recommendation
1. **Normalize the cache key** before caching: parse the query into an AST and serialize it back to a canonical string (or hash the AST), so all semantically equivalent queries share one cache entry.
2. **Add per-IP or global rate limiting** to the GraphQL module, mirroring the `ThrottleConfiguration` pattern already used in the `web3` module.
3. **Limit operation name length and alias count** via `ParserOptions` to reduce the attacker's variation space.
4. Alternatively, use **persisted queries** (allowlist of pre-registered query hashes) so arbitrary query strings are rejected outright.

### Proof of Concept

```bash
# Flood the GraphQL endpoint with unique operation names, each causing a cache miss
i=0
while true; do
  curl -s -o /dev/null -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"query Op${i} { account(input: {entityId: {shard:0,realm:0,num:2}}) { balance } }\"}"
  i=$((i+1))
done
```

Each iteration increments `i`, producing a unique operation name (`Op0`, `Op1`, `Op2`, …). Every request misses the Caffeine cache and triggers a full `parseAndValidateFunction` call. Monitor CPU via the existing `process_cpu_usage` metric — it will climb above the 30% baseline threshold within seconds on a single-core deployment.
