### Title
Missing Address Book File ID Allowlist Validation in REST Java `getAddressBookFileId()` Allows Unprivileged Users to Query Gossip-Only Address Book (File 101)

### Summary
The `getAddressBookFileId()` method in `rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java` (lines 178–182) blindly passes any caller-supplied `file.id` value directly to `networkNodeRepository.findNetworkNodes()` without validating it against the only two legitimate address book file IDs (`0.0.101` and `0.0.102`). Any unauthenticated HTTP client can supply `file.id=0.0.101` to the `/api/v1/network/nodes` endpoint and receive node records sourced from the gossip-only address book (file 101) instead of the client-facing address book (file 102), obtaining a different set of service endpoints. The GRPC module and the REST JavaScript module both enforce an explicit allowlist for this parameter, but the REST Java module does not.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 154–157) performs only an operator check:
```java
if (fileId != null && fileId.operator() != RangeOperator.EQ) {
    throw new IllegalArgumentException("Only equality operator is supported for file.id");
}
```
It does **not** validate that the supplied value is one of `{addressBookFile101, addressBookFile102}`.

Control then flows to `NetworkServiceImpl.getNetworkNodes()` → `getAddressBookFileId()` (lines 178–182):
```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    return request.getFileId() != null
            ? request.getFileId().value()          // ← raw caller value, no allowlist check
            : systemEntity.addressBookFile102().getId();
}
```
The raw caller-supplied ID is forwarded verbatim to `networkNodeRepository.findNetworkNodes(fileId, ...)`, which executes:
```sql
where file_id = :fileId
order by start_consensus_timestamp desc
limit 1
```
against the `address_book` table.

**Root cause / failed assumption:** The code assumes that only `0.0.102` will ever be queried in practice (it is the default), but the parameter is fully user-controlled and no allowlist is enforced.

**Contrast with sibling modules that do enforce the allowlist:**

- GRPC `NetworkServiceImpl` (lines 50–59) builds `validFileIds = Set.of(addressBookFile101(), addressBookFile102())` and throws `INVALID_FILE_ID` for anything outside that set.
- REST JS `SystemEntity.isValidAddressBookFileId()` (lines 393–400) validates against the same two IDs before processing.

**Exploit flow:**
1. Attacker (no credentials required) sends: `GET /api/v1/network/nodes?file.id=0.0.101`
2. `NetworkController` passes the operator check (it is `EQ`).
3. `getAddressBookFileId()` returns the encoded ID for `0.0.101`.
4. `findNetworkNodes` queries the `address_book` table for `file_id = <101-encoded-id>`, returning the latest file-101 address book snapshot.
5. The response contains node entries whose `serviceEndpoints` are sourced from the gossip address book — IP addresses and ports intended for node-to-node gossip, not for client transaction submission.

### Impact Explanation
File `0.0.101` is the gossip address book used for internal node-to-node communication; file `0.0.102` is the client-facing address book whose service endpoints are the correct targets for transaction submission. A client that consumes the `/api/v1/network/nodes` response and uses the returned endpoints to submit transactions will be directed to gossip ports, causing submission failures or silent misdirection. An attacker can craft and distribute a URL (`?file.id=0.0.101`) that causes downstream SDK integrations or monitoring tools to silently use the wrong node topology. Because the mirror node is a read-only service the attacker cannot alter ledger state directly, but they can reliably mislead any consumer of this API about which endpoints to use, degrading availability of transaction submission for those consumers.

### Likelihood Explanation
No authentication or special privilege is required. The `file.id` query parameter is documented in the OpenAPI spec and is trivially discoverable. Any attacker who reads the API documentation or inspects a single request can reproduce this in one HTTP call. The attack is stateless and repeatable at zero cost.

### Recommendation
Add an allowlist check in `getAddressBookFileId()` (or in `NetworkController.getNodes()` immediately after the operator check) mirroring the pattern already used in the GRPC module:

```java
private static final Set<Long> VALID_FILE_IDS = Set.of(
    systemEntity.addressBookFile101().getId(),
    systemEntity.addressBookFile102().getId()
);

private long getAddressBookFileId(final NetworkNodeRequest request) {
    if (request.getFileId() == null) {
        return systemEntity.addressBookFile102().getId();
    }
    long id = request.getFileId().value();
    if (!VALID_FILE_IDS.contains(id)) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return id;
}
```
