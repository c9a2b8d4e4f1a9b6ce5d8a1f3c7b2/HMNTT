### Title
Unbounded Protobuf Body in `estimateFees()` Enables Unauthenticated DoS via Heap Exhaustion

### Summary
The `estimateFees()` endpoint in `NetworkController.java` accepts a raw `byte[]` request body with no size constraint at the controller level, no server-level limit applicable to `application/protobuf` content, and no custom filter enforcing a cap. An unauthenticated attacker can POST an arbitrarily large protobuf body, causing Spring MVC to buffer the entire payload into heap memory and then pass it to `Transaction.PROTOBUF.parse(Bytes.wrap(body))`, which allocates additional heap for deserialized objects, leading to OOM or severe GC pressure that degrades or kills the service.

### Finding Description

**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/NetworkController.java`, lines 113–124:

```java
@PostMapping(
        consumes = {"application/protobuf", "application/x-protobuf"},
        value = "/fees")
FeeEstimateResponse estimateFees(
        @RequestBody @NotNull byte[] body,   // ← no @Size, no length cap
        ...
) {
    try {
        final var transaction = Transaction.PROTOBUF.parse(Bytes.wrap(body));  // ← unbounded parse
        ...
    }
}
```

**Root cause:** The `@RequestBody byte[] body` parameter carries only `@NotNull`. There is no `@Size(max = ...)` annotation, no `Content-Length` enforcement filter, and no Spring Boot server property (`server.tomcat.max-post-size` / `server.tomcat.max-swallow-size`) configured in the module's resources (only `banner.txt` exists under `rest-java/src/main/resources/`).

**Why existing checks fail:**

- `JacksonConfiguration` (`rest-java/src/main/java/org/hiero/mirror/restjava/config/JacksonConfiguration.java`, lines 20–27) sets strict JSON stream-read constraints (`maxDocumentLength=1000`, etc.), but these apply only to JSON deserialization. This endpoint consumes `application/protobuf` / `application/x-protobuf`, so Jackson is never invoked.
- `MetricsFilter` (`rest-java/src/main/java/org/hiero/mirror/restjava/config/MetricsFilter.java`, lines 64–67) reads the `Content-Length` header only to record a metric — it performs no enforcement and does not block the request.
- Tomcat's default `maxPostSize` (2 MB) applies **only** to `application/x-www-form-urlencoded` POST bodies per the Tomcat specification. It does not apply to raw binary content types such as `application/protobuf`.
- No `OncePerRequestFilter` or `HandlerInterceptor` in the `rest-java` config package enforces a body-size limit for this endpoint.

**Exploit flow:**
1. Spring MVC's `ByteArrayHttpMessageConverter` reads the full request stream into `byte[] body` on the heap.
2. `Bytes.wrap(body)` wraps the array (zero-copy), but `Transaction.PROTOBUF.parse(...)` then allocates a full object graph for the deserialized protobuf message.
3. Repeated or concurrent large requests exhaust the JVM heap.

### Impact Explanation
A successful attack causes the mirror-node REST-Java service to become unresponsive or crash with `OutOfMemoryError`, preventing all legitimate clients from querying the current Hashgraph fee schedule. Because fee estimation is used by wallets and dApps to construct valid transactions, sustained unavailability of this endpoint disrupts the ability of ecosystem participants to correctly price and submit transactions against the live Hashgraph network. Severity is **High** (availability impact, no authentication required, no rate-limit or size-limit barrier).

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. An attacker needs only a single HTTP client capable of streaming a large body (e.g., `curl --data-binary @/dev/urandom`). The attack is trivially repeatable and scriptable. A single connection sending a multi-hundred-megabyte body is sufficient to trigger heap pressure; a handful of concurrent connections can reliably cause OOM on a default JVM heap. No special knowledge of the Hashgraph protocol is required — any binary payload is accepted as the body.

### Recommendation
1. **Enforce a server-level limit** in `application.yml` (or equivalent):
   ```yaml
   server:
     tomcat:
       max-swallow-size: 65536   # 64 KB — well above any valid HAPI transaction
   ```
2. **Add a `@Size` constraint** on the controller parameter (requires a custom `ConstraintValidator` for `byte[]`):
   ```java
   @RequestBody @NotNull @Size(max = 65536) byte[] body
   ```
3. **Add a dedicated size-guard filter** that checks `request.getContentLengthLong()` and rejects requests exceeding the threshold before the body is read, covering chunked-encoding cases where `Content-Length` is absent.
4. The threshold should be set to the maximum valid serialized HAPI `Transaction` size (the Hedera network enforces a 6 KB transaction limit; a generous server-side cap of 64 KB is more than sufficient).
