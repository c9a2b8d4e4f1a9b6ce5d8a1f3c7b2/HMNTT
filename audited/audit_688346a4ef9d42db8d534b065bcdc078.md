### Title
GraphQL `findByEvmAddress` Misses Entities Whose EVM Address Is Stored in the `alias` Column

### Summary
The GraphQL `EntityServiceImpl.getByEvmAddressAndType()` exclusively calls `findByEvmAddress()`, which queries only the `evm_address` column. On Hedera, an account created with a 20-byte EVM-address alias stores that address in the `alias` column; if the `evm_address` column holds a different value (or is null), the GraphQL API silently returns "not found" for a query that supplies the alias bytes as the EVM address. The `web3` module already handles this correctly with `findByEvmAddressOrAliasAndDeletedIsFalse`, but the GraphQL module has no equivalent fallback.

### Finding Description

**Code path:**

`EntityServiceImpl.getByEvmAddressAndType()` (lines 34–41): [1](#0-0) 

When the decoded bytes are not a long-zero address, the method calls only: [2](#0-1) 

That SQL touches only `evm_address`. There is no subsequent call to `findByAlias`.

**Root cause / failed assumption:** The code assumes every entity's EVM address is stored in the `evm_address` column. This is false for accounts whose `alias` column holds a 20-byte EVM address while `evm_address` is null or holds a different value.

**How such entities arise:**

1. `CryptoCreateTransactionHandler.doUpdateEntity()` sets `evm_address` from `transactionRecord.getEvmAddress()` first; if that field is non-empty and differs from the alias, `alias ≠ evm_address`. [3](#0-2) 

2. `AccountEvmAddressMigration` only back-fills `evm_address` for 35-byte ECDSA proto-encoded aliases; it skips 20-byte raw EVM-address aliases entirely. [4](#0-3) 

3. `CommonEntityAccessor.evmAddressFromId()` in the `web3` module explicitly acknowledges that a 20-byte alias is a valid EVM address, confirming the dual-column reality. [5](#0-4) 

**Contrast with the correct implementation** in the `web3` module: [6](#0-5) 

### Impact Explanation
Any client querying the GraphQL API with an EVM address that is stored as an entity's `alias` (not `evm_address`) receives an empty result. Applications that rely on this endpoint to verify account existence, resolve addresses, or gate downstream logic will silently treat a live account as non-existent. This is a data-correctness / information-integrity issue: the mirror node exports an incomplete view of the ledger state through its GraphQL surface.

### Likelihood Explanation
No special privilege is required. Any unauthenticated user can submit a GraphQL query with an arbitrary `evmAddress` string. The precondition (entity with `alias = X`, `evm_address ≠ X`) is a documented, real-world state: the migration explicitly targets accounts where `evm_address is null` and the alias is an ECDSA key, and the importer can write `evm_address` from the transaction record independently of the alias. The gap is therefore reproducible against any mirror node whose database contains such accounts.

### Recommendation
Replace the single-column lookup in `EntityServiceImpl.getByEvmAddressAndType()` with a query that checks both columns, mirroring the `web3` module pattern:

```sql
select * from entity
where (evm_address = ?1 or alias = ?1)
  and deleted is not true
```

Add a corresponding `findByEvmAddressOrAlias(byte[])` method to the GraphQL `EntityRepository` and call it from `getByEvmAddressAndType()` instead of `findByEvmAddress()`. [2](#0-1) [7](#0-6) 

### Proof of Concept

1. Insert (or identify) an entity row where `alias` is a 20-byte value `0xAABB…` and `evm_address` is null or a different 20-byte value.
2. Send the following GraphQL query to the mirror node:
   ```graphql
   query {
     account(input: { evmAddress: "0xAABB…" }) {
       entityId { shard realm num }
     }
   }
   ```
3. Observe that the response is `null` / "not found", even though the account exists.
4. Confirm the account exists by querying with its numeric ID (`entityId` input) — it is returned successfully.
5. Confirm that `findByAlias(0xAABB…)` would return the entity by running the SQL `select * from entity where alias = '\xAABB…'` directly against the database.
