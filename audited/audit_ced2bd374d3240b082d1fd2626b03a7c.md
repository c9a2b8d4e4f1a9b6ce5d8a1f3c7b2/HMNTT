### Title
Unauthenticated Full Entity-Table Scan with Per-Row Token Balance Subqueries Enables Connection-Pool Exhaustion DoS

### Summary
The `getAccounts()` handler in `rest/accounts.js` accepts an unauthenticated `GET /accounts?account.balance=lte:<large_value>&limit=<max>` request that forces a sequential scan of the entire `entity` table (no selective index on `e.balance` for range predicates), then executes a correlated `token_account` subquery for every row in the result set. Although a `statement_timeout` exists at the pool level, an attacker can flood the endpoint with concurrent requests, exhausting the finite DB connection pool and degrading or denying service to all other API consumers.

### Finding Description

**Exact code path:**

`getAccounts()` at [1](#0-0)  parses `account.balance` via `utils.parseBalanceQueryParam(req.query, 'e.balance')` and passes the result as `balanceQuery` (the `entityBalanceQuery` argument) to `getAccountQuery()`. No account-ID or other selective filter is required.

`getAccountQuery()` at [2](#0-1)  calls `getEntityBalanceQuery()` when `includeBalance` is `true` (the default).

`getEntityBalanceQuery()` at [3](#0-2)  takes two distinct paths. When no timestamp filter is present (the normal case for `getAccounts`), `needHistoricalBalanceInfo` is `false`, so:

- `entityTable = 'entity'` — a direct scan of the live entity table. [4](#0-3) 
- A `with latest_token_balance as (select account_id, balance, token_id from token_account where associated is true)` CTE is prepended. [5](#0-4) 
- A **correlated subquery** is injected into the SELECT list for every result row: `(select json_agg(...) from (select token_id, balance from latest_token_balance where account_id = e.id order by token_id limit N) ...)` [6](#0-5) 

The WHERE clause becomes `e.type in ('ACCOUNT','CONTRACT') AND e.balance <= $N`, which matches virtually every account when `N` is large (e.g., `999999999999`). The `entity` table has no partial or covering index on `balance` for range predicates, so PostgreSQL performs a sequential scan.

`parseBalanceQueryParam` at [7](#0-6)  accepts any numeric value with any comparator (`lte`, `gte`, `lt`, `gt`, `eq`, `ne`) and emits a raw `e.balance <= ?` clause with no additional guard.

`getLimitParamValue` at [8](#0-7)  caps the limit at `responseLimit.max` (the configured maximum, typically 1000) but does not reduce it when a non-selective filter is used.

The only DB-level protection is `statement_timeout` configured in the pool: [9](#0-8)  This kills a single long-running query but does **not** prevent an attacker from holding `maxConnections` connections simultaneously, each blocked for up to `statementTimeout` milliseconds.

**Root cause / failed assumption:** The design assumes that the `LIMIT` clause and `statement_timeout` together bound resource consumption per request. This is true in isolation, but the assumption fails under concurrent load: `maxConnections` connections can be saturated simultaneously, each executing a full entity-table scan plus up to `limit × tokenBalanceResponseLimit.multipleAccounts` correlated subquery executions, blocking all other legitimate queries.

### Impact Explanation

- **Database I/O:** Each request triggers a sequential scan of the entire `entity` table (potentially millions of rows on mainnet) plus up to `responseLimit.max` correlated subqueries against `token_account`. Concurrent requests multiply this I/O linearly.
- **Connection pool exhaustion:** With `maxConnections` connections held for up to `statementTimeout` ms each, the pool is fully occupied, causing all other API endpoints to queue or time out.
- **Node.js memory:** Each completed query returns up to `limit × tokenBalanceResponseLimit.multipleAccounts` token-balance objects that are JSON-serialized in the Node.js process. Many concurrent large responses inflate heap usage.
- **Severity:** High — complete API unavailability for all users, no authentication required, trivially repeatable.

### Likelihood Explanation

- **No authentication or API key required** — any internet-accessible mirror node is a valid target.
- **Single HTTP client suffices** — a single attacker with a modest connection rate (e.g., 50 concurrent requests) can saturate a default pool.
- **Fully repeatable** — the attack is stateless and can be scripted in seconds.
- **Publicly documented API** — the `account.balance` parameter is described in the OpenAPI spec. [10](#0-9) 

### Recommendation

1. **Require a selective anchor filter:** Reject or warn when `account.balance` range filters are supplied without a corresponding `account.id` equality filter, or enforce a tighter maximum limit when only a balance range is provided.
2. **Add rate limiting per IP/client** at the reverse-proxy or middleware layer for the `/accounts` endpoint.
3. **Add a DB-level index** on `entity(balance)` (or a partial index for `type IN ('ACCOUNT','CONTRACT')`) so range scans use an index rather than a sequential scan, reducing per-query I/O cost.
4. **Reduce `responseLimit.max`** when no selective filter is present, or introduce a separate, lower cap for balance-range-only queries.
5. **Enforce a connection-acquisition timeout** and expose a circuit-breaker so pool exhaustion degrades gracefully rather than blocking all consumers.

### Proof of Concept

```bash
# Step 1: Confirm the endpoint is unauthenticated and accepts the filter
curl -s "https://<mirror-node>/api/v1/accounts?account.balance=lte:999999999999&limit=1" | jq .

# Step 2: Flood with concurrent maximum-result requests
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/accounts?account.balance=lte:999999999999&limit=1000" \
    -o /dev/null &
done
wait

# Step 3: Observe that legitimate requests now time out or return 503
curl -v "https://<mirror-node>/api/v1/accounts?limit=1"
# Expected: connection hang or HTTP 503 / pool-exhaustion error
```

The attack requires no credentials, no special tooling, and is repeatable indefinitely. Each wave of 200 concurrent requests holds DB connections for up to `statementTimeout` ms, preventing any other query from executing during that window.

const getAccountQuery = (
  entityAccountQuery,
  tokenBalanceQuery,
  accountBalanceQuery = {query: '', params: []},
  entityBalanceQuery = {query: '', params: []},
  limitAndOrderQuery = {query: '', params: [], order: constants.orderFilterValues.ASC},
  pubKeyQuery = {query: '', params: []},
  includeBalance = true,
  isHistorical = false
) => {
  // Keep track of Postgres positional param index when converting MySQL style value placeholder ? to Postgres style
  // $1, $2, .... The query fragments are ordered in the same order as their params
  let paramIndex = 1;

  if (!includeBalance) {
    [entityAccountQuery, pubKeyQuery, limitAndOrderQuery]
      .filter((query) => !!query.query)
      .forEach((query) => {
        query.query = query.query.replace(/\?/g, (s) => `$${paramIndex++}`);
      });
    const entityCondition = [`e.type in ('ACCOUNT', 'CONTRACT')`, entityAccountQuery.query, pubKeyQuery.query]
      .filter((x) => !!x)
      .join(' and ');
    const entityOnlyQuery = `
            select ${entityFields}
            from entity e
              left join entity_stake es on es.id = e.id ${getEntityStakeAccountCondition(entityAccountQuery)}
            where ${entityCondition}
            order by id ${limitAndOrderQuery.order} ${limitAndOrderQuery.query}`;
    return {
      query: entityOnlyQuery,
      params: utils.mergeParams(entityAccountQuery.params, pubKeyQuery.params, limitAndOrderQuery.params),
    };
  }

  [tokenBalanceQuery, entityBalanceQuery, entityAccountQuery, pubKeyQuery, accountBalanceQuery, limitAndOrderQuery]
    .filter((query) => !!query.query)
    .forEach((query) => {
      query.query = query.query.replace(/\?/g, (s) => `$${paramIndex++}`);
    });

  return getEntityBalanceQuery(
    entityBalanceQuery,
    entityAccountQuery,
    limitAndOrderQuery,
    pubKeyQuery,
    tokenBalanceQuery,
    accountBalanceQuery,
    isHistorical
  );

const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;
  }
  return ret;
};
