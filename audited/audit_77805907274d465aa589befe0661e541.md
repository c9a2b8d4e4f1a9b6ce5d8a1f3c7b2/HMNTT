### Title
Unbounded `json_agg` Correlated Subquery in `selectTransactionsInTimestampRange` Enables Unauthenticated Database Resource Exhaustion

### Summary
The `selectTransactionsInTimestampRange` SQL constant in `rosetta/app/persistence/transaction.go` executes two correlated subqueries — each using `json_agg(... ORDER BY ...)` — once per transaction row returned by the outer query. There is no row-count limit on the `crypto_transfer` or `staking_reward_transfer` subqueries, and the only post-query guard (`maxTransactionsInBlock`) is applied after the full database work completes. An unauthenticated attacker can repeatedly request a block containing many transactions with many `crypto_transfer` rows, forcing the database to perform thousands of independent sort-and-aggregate operations per request.

### Finding Description

**Exact code path:**

`rosetta/app/persistence/transaction.go`, lines 32–59 — the `selectTransactionsInTimestampRange` constant:

```sql
select
  t.consensus_timestamp, ...
  coalesce((
    select json_agg(json_build_object('account_id', entity_id, 'amount', amount)
                    order by entity_id)          -- ← sort per transaction row
    from crypto_transfer
    where consensus_timestamp = t.consensus_timestamp
      and (errata is null or errata <> 'DELETE') -- ← no LIMIT
  ), '[]') as crypto_transfers,
  coalesce((
    select json_agg(json_build_object('account_id', account_id, 'amount', amount)
                    order by account_id)          -- ← sort per transaction row
    from staking_reward_transfer
    where consensus_timestamp = t.consensus_timestamp -- ← no LIMIT
  ), '[]') as staking_reward_payouts
from transaction t
where consensus_timestamp >= @start and consensus_timestamp <= @end
```

This query is executed in `FindBetween` (lines 112–171) with `Limit(batchSize)` where `batchSize = 2000`. The `Limit(2000)` applies only to the outer `transaction` table scan. For each of the up to 2000 transaction rows, PostgreSQL independently executes both correlated subqueries, each requiring a full scan of matching `crypto_transfer` rows followed by an in-memory sort (`ORDER BY entity_id`).

**Post-query guard is too late:**

In `rosetta/app/services/block_service.go`, lines 56–67:
```go
if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
    return nil, err
}
// maxTransactionsInBlock check happens HERE — after FindBetween returns
if len(block.Transactions) > s.maxTransactionsInBlock {
    ...
    block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
}
```

`FindBetween` completes the full expensive query before `maxTransactionsInBlock` is ever checked. The guard only trims the Go-side result slice; it does not prevent the database from doing the work.

**Root cause / failed assumption:** The design assumes that `batchSize = 2000` bounds total work, but it only bounds the number of outer transaction rows. The actual work is `O(N_transactions × M_transfers_per_transaction)` sort operations, which is unbounded for `M`.

### Impact Explanation

On a live Hedera network, a single record file (block) can contain thousands of transactions. High-throughput blocks (e.g., those containing batch HBAR airdrops or token distribution events) can have transactions with dozens to hundreds of `crypto_transfer` rows each. A single `/block` request against such a block forces the database to execute up to 2000 × (number of transfers per transaction) row reads and up to 2000 independent sort operations. Repeated requests from a single unauthenticated client can sustain elevated CPU and memory pressure on the PostgreSQL instance, exceeding the 30% resource increase threshold without any brute-force volume of distinct requests. Memory pressure is compounded because all `json_agg` results are materialized in memory before being returned.

### Likelihood Explanation

The Rosetta `/block` endpoint requires no authentication. The attacker only needs to:
1. Enumerate block indices (sequential integers) or hashes to find a high-transfer-count block.
2. Repeatedly POST to `/block` with that block identifier.

The Traefik rate-limit middleware (`average: 10` per host) is defined only in the Helm chart values and is not enforced at the application layer; deployments without Traefik have no rate limiting at all. The attack is repeatable, requires no credentials, and the target block is permanent (on-chain data does not change).

### Recommendation

1. **Add a `LIMIT` to the correlated subqueries** or rewrite them as a single `LEFT JOIN ... GROUP BY` with an explicit cap on rows aggregated per transaction.
2. **Move the `maxTransactionsInBlock` guard before the expensive query** — look up the transaction count for the block first and reject or paginate if it exceeds the threshold.
3. **Apply a statement-level timeout** via `db.WithContext` / `statement_timeout` so runaway aggregations are killed automatically.
4. **Enforce rate limiting at the application layer** (not only in optional infrastructure middleware) for the `/block` endpoint.

### Proof of Concept

**Preconditions:**
- A Rosetta node is running and accessible.
- A block exists in the database with ≥500 transactions, each having ≥50 `crypto_transfer` rows (realistic on mainnet).

**Steps:**
```bash
# 1. Find a high-transfer-count block index (e.g., by scanning recent blocks)
BLOCK_INDEX=<target_block_index>
BLOCK_HASH=<target_block_hash>
NETWORK=<network_identifier>

# 2. Repeatedly request the block (e.g., 20 concurrent clients, 50 requests each)
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>/block \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": $NETWORK,
      \"block_identifier\": {
        \"index\": $BLOCK_INDEX,
        \"hash\": \"$BLOCK_HASH\"
      }
    }" &
done
wait

# 3. Observe PostgreSQL CPU/memory via pg_stat_activity or system metrics.
#    Each request triggers up to 2000 json_agg+ORDER BY operations against
#    crypto_transfer with no per-subquery row limit.
```

**Expected result:** PostgreSQL CPU utilization rises sharply and remains elevated for the duration of the request flood, exceeding 30% above baseline without requiring a high request rate.
