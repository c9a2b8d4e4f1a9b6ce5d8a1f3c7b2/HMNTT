### Title
Unauthenticated Request Flood Exhausts Database Connection Pool via `GET /api/v1/blocks`

### Summary
The `getBlocks()` handler in `rest/routes/blockRoute.js` is registered with no rate-limiting middleware at the application layer. The Node.js REST service uses a shared `pg` connection pool with a default maximum of 10 connections. An unprivileged attacker sending a sustained flood of concurrent requests can hold all pool slots occupied, causing every subsequent legitimate request to wait up to 20 seconds before receiving a connection-timeout error.

### Finding Description

**Route registration — no throttle middleware:**

`rest/routes/blockRoute.js` lines 12–13 register the handler directly on the router with no intervening middleware:
```js
router.getExt('/', BlockController.getBlocks);
```

**Server middleware stack — no rate limiter present:**

`rest/server.js` lines 68–144 define the full middleware chain: `urlencoded`, `json`, `cors`, optional `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`, then routes, then `responseHandler`/`handleError`. A `grep` across all `rest/**/*.js` for `rateLimit`, `throttle`, or `rateLimiter` returns **zero matches**. No rate-limiting package appears in `rest/package.json` dependencies.

**Connection pool — 10 connections by default:**

`rest/dbpool.js` lines 13–14 configure the pool directly from config:
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20 000 ms
max: config.db.pool.maxConnections,                         // default 10
```
`docs/configuration.md` line 556 confirms `hiero.mirror.rest.db.pool.maxConnections` defaults to **10**.

**Exploit path:**
An attacker opens ≥10 concurrent HTTP connections and issues `GET /api/v1/blocks` on each. Each request acquires one pool slot and holds it for up to `statement_timeout` (default 20 s). With all 10 slots occupied, every new request from legitimate users enters the `pg` queue and waits up to `connectionTimeoutMillis` (20 s) before receiving a timeout error.

**Why existing checks fail:**
- `authHandler` (`rest/middleware/authHandler.js` lines 15–36) only enforces a custom response-size limit for *authenticated* users; unauthenticated requests pass through unconditionally.
- The Redis response cache (`responseCacheCheckHandler`) is **disabled by default** (`hiero.mirror.rest.cache.response.enabled` defaults to `false`, `docs/configuration.md` line 549), so cache hits do not reduce DB load in a default deployment.
- `statement_timeout` (20 s) limits query duration but does not prevent pool exhaustion — it only bounds how long each slot is held.
- pgbouncer's `max_user_connections: 250` for `mirror_rest` (`charts/hedera-mirror/values.yaml` line 373) operates at the DB-proxy layer and does not protect the Node.js pool from being saturated.

### Impact Explanation
With the default pool size of 10, a single attacker machine with 10 persistent concurrent connections fully saturates the pool. All legitimate `GET /api/v1/blocks` (and any other endpoint sharing the same pool) requests queue and time out after 20 seconds, producing HTTP 500 / connection-timeout responses. The service is effectively unavailable for the duration of the attack with no economic cost to the attacker.

### Likelihood Explanation
No authentication, API key, or IP-based throttle is required. The attack is reproducible with a trivial shell script (`ab`, `wrk`, or `curl` in a loop). The attacker needs only network access to the public endpoint. The attack is stateless and can be sustained indefinitely from a single host.

### Recommendation
1. Add an application-level rate-limiting middleware (e.g., `express-rate-limit`) in `rest/server.js` before the route handlers, scoped per source IP.
2. Increase `hiero.mirror.rest.db.pool.maxConnections` from the default of 10 to a value that reflects realistic concurrency, reducing the blast radius of any single flood.
3. Enable the Redis response cache (`hiero.mirror.rest.cache.response.enabled: true`) so repeated identical `getBlocks` queries are served from cache without touching the DB pool.
4. Add an infrastructure-level `inFlightReq` / `rateLimit` Traefik middleware for the REST service (analogous to what already exists for the Rosetta service in `charts/hedera-mirror-rosetta/values.yaml` lines 152–161).

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
