### Title
Unbounded Subscription Lifetime via Far-Future `endTime` Causing `isComplete()` to Never Return True

### Summary
The `convertTimestamp()` method in `ConsensusController` explicitly maps any gRPC timestamp with `seconds >= 9223372035` to `Long.MAX_VALUE`, which is then stored as `endTime`. Because `isComplete()` evaluates `Instant.ofEpochSecond(0, Long.MAX_VALUE).plus(endTimeInterval).isBefore(Instant.now())` — a condition that resolves to year ~2262 and is never true in practice — the `RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)` loop in `pastEndTime()` runs indefinitely. An unprivileged attacker can open many such subscriptions to exhaust server-side scheduler and connection resources.

### Finding Description

**Entry point — `ConsensusController.convertTimestamp()`:** [1](#0-0) 

Any client sending `consensusEndTime.seconds >= 9223372035` receives `endTime = Long.MAX_VALUE` in the resulting `TopicMessageFilter`. This is not a defensive clamp — it is the intended overflow guard, but it produces a semantically broken value downstream.

**Validation gap — `TopicMessageFilter.isValidEndTime()`:** [2](#0-1) 

The only constraint is `endTime > startTime`. `Long.MAX_VALUE > startTime` is always true, so the far-future value passes validation without error.

**`isComplete()` never returns true:** [3](#0-2) 

With `endTime = Long.MAX_VALUE` nanoseconds, `Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to approximately year 2262. Adding `endTimeInterval` (default 30 s) still yields year 2262. `Instant.now()` never reaches that value, so the method permanently returns `false`.

**Infinite repeat loop in `pastEndTime()`:** [4](#0-3) 

`RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)` fires every `endTimeInterval` (30 s) and never stops. The `takeUntilOther(pastEndTime(topicContext))` in the main pipeline therefore never terminates the subscription. [5](#0-4) 

**Per-connection limit is insufficient:** [6](#0-5) 

`maxConcurrentCallsPerConnection` (default 5) limits streams per TCP connection, but there is no global cap on total concurrent subscriptions. An attacker opens many connections, each with 5 streams, accumulating arbitrarily many zombie subscriptions.

### Impact Explanation
Each zombie subscription holds a live Reactor timer entry (firing every 30 s), a gRPC server stream, and in-memory `TopicContext` state. With no global subscription ceiling, an attacker can accumulate thousands of these, exhausting Reactor scheduler timer queues, JVM heap, and Netty connection resources, leading to denial of service for legitimate subscribers. The `subscriberCount` gauge will reflect the inflation but there is no enforcement action tied to it.

### Likelihood Explanation
The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and the ability to open TCP connections to port 5600. No authentication, no special privileges, and no knowledge of internal state is needed. The trigger value (`seconds >= 9223372035`) is trivially discoverable from the source or by fuzzing. The attack is repeatable and automatable.

### Recommendation
1. **Reject or clamp `endTime` to a reasonable maximum** (e.g., `now + 30 days`) in `ConsensusController.convertTimestamp()` or in `TopicMessageFilter` validation, rather than silently substituting `Long.MAX_VALUE`.
2. **Add an upper-bound `@Max` or `@AssertTrue` constraint** on `endTime` in `TopicMessageFilter` so that values beyond a configurable horizon are rejected with a clear gRPC `INVALID_ARGUMENT` status.
3. **Enforce a global concurrent-subscription limit** (e.g., via an `AtomicLong` gate in `subscribeTopic`) and return `RESOURCE_EXHAUSTED` when exceeded.
4. **Add a wall-clock timeout** to `pastEndTime()` independent of `isComplete()`, so subscriptions are unconditionally terminated after a configurable maximum duration.

### Proof of Concept
```bash
# Using grpcurl against the mirror node gRPC endpoint (port 5600)
# Send consensusEndTime.seconds = 9223372036 (>= threshold), triggering Long.MAX_VALUE endTime

grpcurl -plaintext -d '{
  "topicID": {"topicNum": 1},
  "consensusStartTime": {"seconds": 1},
  "consensusEndTime": {"seconds": 9223372036}
}' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# Repeat in a loop across multiple connections:
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 1},
    "consensusEndTime": {"seconds": 9223372036}
  }' <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Each subscription never terminates; subscriberCount metric climbs without bound.
# Monitor: GET /actuator/metrics/hiero.mirror.grpc.subscribers
# Scheduler timer queue and heap grow until OOM or legitimate subscribers are starved.
```

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)

    }
