### Title
Unauthenticated Timestamp Parameter Forces Full `hook_storage_change` Table Scan via Historical Code Path

### Summary
Any unprivileged external user can supply `timestamp=gte:0` to the `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint. This single parameter causes `isHistorical()` to return `true`, routing execution to `getHookStorageChange()`, which issues a `DISTINCT ON (key)` query against the append-only `hook_storage_change` table with an unbounded timestamp range (`0` to `Long.MAX_VALUE`). Because `hook_storage_change` accumulates every historical write and the query must scan all matching rows before applying the result `limit`, repeated requests can drive DB CPU and I/O well above 30% compared to the normal `hook_storage` path.

### Finding Description

**Exact code path:**

1. `HooksController.java` line 110–111: `timestamps` is a `@RequestParam(required = false, defaultValue = "")` with only `@Size(max = 2)` — no authentication guard, no restriction on range width.
2. `HooksController.java` line 186: `Bound.of(timestamps, ...)` is called. With `timestamp=gte:0`, the array is non-empty, so a `Bound` with `lower = GTE:0` and `upper = null` is constructed.
3. `HookStorageRequest.java` line 66–68: `isHistorical()` returns `!timestamp.isEmpty()`. `Bound.isEmpty()` is `lower == null && upper == null` (`Bound.java` line 126–128). With `lower` set, `isEmpty()` is `false`, so `isHistorical()` is `true`.
4. `HookServiceImpl.java` lines 57–58: The branch `if (request.isHistorical()) return getHookStorageChange(request)` is taken.
5. `HookServiceImpl.java` lines 98–99: `timestampLowerBound = 0` (GTE:0, no increment), `timestampUpperBound = Long.MAX_VALUE` (no upper supplied).
6. `HookStorageChangeRepository.java` lines 26–31: The executed SQL is:
   ```sql
   SELECT DISTINCT ON (key) ...
   FROM hook_storage_change
   WHERE owner_id = ? AND hook_id = ?
     AND key BETWEEN ? AND ?
     AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
   ```
   `DISTINCT ON (key)` requires the DB to read **every row** matching the predicate before it can deduplicate and apply the `LIMIT`. The `hook_storage_change` table is append-only and grows without bound.

**Root cause:** The `isHistorical()` predicate is purely presence-based — any non-empty `Bound` (even `gte:0`) triggers the expensive path. There is no minimum timestamp, no rate limit, and no authentication requirement on the endpoint.

**Why existing checks fail:**
- `@Size(max = 2)` only caps the number of timestamp parameters to 2; it does not restrict range width.
- `@Max(MAX_LIMIT)` on `limit` caps returned rows but does **not** reduce the DB scan — `DISTINCT ON` must process all qualifying rows first.
- No authentication annotation is present on `getHookStorage()` or the class.

### Impact Explanation
The `hook_storage_change` table is an append-only audit log that grows proportionally to all historical writes. A single request with `timestamp=gte:0` forces a full sequential scan of this table for the targeted `(owner_id, hook_id)` pair. An attacker targeting a high-activity hook can repeatedly issue this request (or fan out across multiple `hookId` values) to saturate DB I/O and CPU. Because the normal path queries the much smaller `hook_storage` (current-state) table, the resource differential easily exceeds 30% under realistic data volumes. This constitutes a denial-of-service against the database layer without requiring any credentials.

### Likelihood Explanation
The endpoint is publicly reachable (no authentication visible), the exploit requires a single HTTP GET with one query parameter, and it is trivially repeatable in a loop or via concurrent requests. No special knowledge of the system internals is needed — the `timestamp` parameter is a documented API feature. Any motivated external actor can execute this.

### Recommendation
1. **Require a bounded timestamp range:** Reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., 30 days in nanoseconds).
2. **Require an upper bound when a lower bound is provided:** If only `gte:X` is supplied with no upper bound, return HTTP 400.
3. **Add authentication/authorization** to the storage endpoint so only authorized callers can trigger the historical path.
4. **Add rate limiting** per client IP or account on this endpoint.
5. **Consider a covering index** on `(owner_id, hook_id, consensus_timestamp, key)` to reduce scan cost even when the historical path is legitimately used.

### Proof of Concept

```
# Step 1: No timestamp — fast path, queries hook_storage (current state)
GET /api/v1/accounts/0.0.1234/hooks/1/storage

# Step 2: Exploit — forces full hook_storage_change scan
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0

# Step 3: Repeat in a loop to sustain DB pressure
for i in $(seq 1 100); do
  curl -s "https://<host>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0" &
done
wait
```

Preconditions: network access to the API; no credentials required.
Trigger: `timestamp=gte:0` sets `lower=GTE:0`, `upper=null` → `isHistorical()=true` → `DISTINCT ON` full scan with `consensus_timestamp BETWEEN 0 AND MAX_VALUE`.
Result: DB CPU/IO spikes proportionally to the size of `hook_storage_change` for the targeted hook; sustained requests cause resource exhaustion exceeding the 30% threshold.
