### Title
Unbounded Account Alias List in `resolveAccountAliases()` Enables Connection Pool Exhaustion DoS During Network Partition

### Summary
`resolveAccountAliases()` in `rosetta/app/services/construction_service.go` iterates over a caller-controlled, comma-separated alias list with no upper-bound check, issuing one synchronous `GetAccountId()` database call per alias. During a network partition, each call blocks for up to the configured `statementTimeout` (default 20 seconds), and because the Rosetta DB pool has a hard cap of 100 open connections, a small number of concurrent requests carrying large alias lists can exhaust the pool and deny service to all other users. No authentication is required to reach this endpoint.

### Finding Description

**Exact code path:**

`ConstructionMetadata` (line 159) calls `resolveAccountAliases` unconditionally whenever `optionKeyAccountAliases` is present in the request options. [1](#0-0) 

Inside `resolveAccountAliases`, the alias string is split on commas and iterated with **no count limit**: [2](#0-1) 

Each iteration calls `c.accountRepo.GetAccountId(ctx, accountId)` (line 581), which in the persistence layer acquires a DB connection from the shared pool and executes a raw SQL query: [3](#0-2) 

**Root cause and failed assumption:** The design assumes the alias list is short (e.g., a handful of accounts in a transaction). There is no validation of list length anywhere in the call chain. The code also assumes DB calls complete quickly; during a network partition they block until `statementTimeout` (default 20 s). [4](#0-3) 

**Why existing checks fail:**

The only infrastructure-level controls are Traefik middlewares defined in the Helm chart: [5](#0-4) 

These are disabled by default (`global.middleware: false`) and are not part of the application binary. Deployments without Kubernetes/Traefik have zero protection. Even when enabled, `inFlightReq: amount: 5` is per source IP — a distributed attacker or one using multiple IPs bypasses it entirely. There is no application-level alias count cap, no per-request DB call budget, and no circuit breaker inside the Go service itself.

The DB pool configuration confirms the finite resource: [6](#0-5) 

Default pool limits (100 open connections, 20 s statement timeout): [7](#0-6) 

### Impact Explanation
During a network partition, each `GetAccountId()` call holds a DB connection for up to 20 seconds before timing out. With 100 max open connections, an attacker sending ~10 concurrent requests each containing 100+ aliases can saturate the entire pool. All legitimate requests that require a DB connection (balance queries, block queries, etc.) will block indefinitely waiting for a free connection, resulting in a complete service outage for all concurrent users. The `/construction/metadata` endpoint is publicly accessible with no authentication requirement.

### Likelihood Explanation
The attack requires no credentials, no special knowledge of the system, and no prior access. The attacker only needs to know the Rosetta API schema (publicly documented by Coinbase's Rosetta specification). Sending a POST request with a long comma-separated `account_aliases` string is trivial. The attack is repeatable and can be sustained indefinitely. The network partition condition is the amplifier — without it, calls return quickly — but partitions occur naturally (DB maintenance, cloud networking events) and can also be induced by the attacker if they have any influence over network routing between the Rosetta service and its database.

### Recommendation
1. **Enforce an alias count limit** inside `resolveAccountAliases()` before the loop — reject requests where `len(strings.Split(accountAliases, ",")) > N` for a small constant N (e.g., 10–20, matching the maximum number of accounts in a single Hedera transaction).
2. **Apply a per-request DB call budget** or use a semaphore to cap total concurrent DB connections consumed by a single HTTP request.
3. **Enable the Traefik middleware by default** and add per-IP `inFlightReq` limits at the application layer (e.g., via a Go HTTP middleware) so protection is not deployment-dependent.
4. **Honour the request context deadline** — ensure `GetDbWithContext` propagates the HTTP request context so that if the HTTP server's `writeTimeout` fires, in-flight DB calls are cancelled immediately rather than running to `statementTimeout`.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, reachable at `http://<host>:5700`. DB is partitioned (or attacker triggers a slow-query condition).

**Step 1 — Build a large alias list:**
```
ALIASES=$(python3 -c "print(','.join(['0.0.' + str(i) for i in range(1,501)]))")
```

**Step 2 — Send concurrent requests from multiple IPs (or use a single IP if Traefik middleware is absent):**
```bash
for i in $(seq 1 20); do
  curl -s -X POST http://<host>:5700/construction/metadata \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"testnet\"},
      \"options\": {
        \"operation_type\": \"CRYPTO_TRANSFER\",
        \"account_aliases\": \"$ALIASES\"
      }
    }" &
done
wait
```

**Step 3 — Observe:** While the 20 concurrent requests are in-flight (each triggering up to 500 sequential `GetAccountId` DB calls blocked by the partition), issue a legitimate request from a separate client. The legitimate request will block indefinitely on DB connection acquisition, returning an error or timing out — confirming pool exhaustion and denial of service.

```go
func (c *constructionAPIService) resolveAccountAliases(
	ctx context.Context,
	options map[string]any,
) (emptyResult string, nilErr *rTypes.Error) {
	if options[optionKeyAccountAliases] == nil {
		return
	}

	if !c.BaseService.IsOnline() {
		return emptyResult, errors.ErrEndpointNotSupportedInOfflineMode
	}

	accountAliases, ok := options[optionKeyAccountAliases].(string)
	if !ok {
		return emptyResult, errors.ErrInvalidOptions
	}

	var accountMap []string
	for accountAlias := range strings.SplitSeq(accountAliases, ",") {
		accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
		if err != nil {
			return emptyResult, errors.ErrInvalidAccount
		}

		found, rErr := c.accountRepo.GetAccountId(ctx, accountId)
		if rErr != nil {
			return emptyResult, rErr
		}
		accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
	}

	return strings.Join(accountMap, ","), nilErr
}
```
