### Title
Unbounded Subscription Growth Exhausts Global `boundedElastic` Scheduler in `PollingTopicListener` (POLL Mode DoS)

### Summary
`PollingTopicListener.listen()` creates a per-subscriber, indefinitely-repeating polling loop that schedules tasks on the global `Schedulers.boundedElastic()` instance with no subscription count limit. An unprivileged attacker opening many concurrent gRPC connections can queue enough delayed and repeat tasks to exhaust the scheduler's bounded task queue (default 100,000), causing `RejectedExecutionException` for all reactive operations sharing that global scheduler across the entire gRPC application.

### Finding Description

**Exact code path:**

`PollingTopicListener.java` line 31 allocates the scheduler as the global shared instance:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
```
`Schedulers.boundedElastic()` returns the **global** application-wide `BoundedElasticScheduler`, not a fresh instance. Its defaults are: thread count = `10 × CPU cores`, task queue = 100,000 entries.

`listen()` at lines 38–43 creates two scheduler task sources per subscription:
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // 1 task queued immediately per sub
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))               // 1 task re-queued every 500ms per sub
```

**Root cause:** There is no check on the number of active subscriptions before scheduling work. Every call to `listen()` unconditionally enqueues tasks on the shared global scheduler.

**Exploit flow:**
1. Attacker opens N gRPC connections to the public `subscribeTopic` endpoint.
2. Each connection carries up to 5 concurrent streams (`maxConcurrentCallsPerConnection = 5`), each calling `subscribeTopic`.
3. `ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicListener.listen()` → `PollingTopicListener.listen()` — no subscription gate anywhere in this chain.
4. Each subscription immediately enqueues a `delaySubscription` task and then re-enqueues a `repeatWhen` task every 500 ms.
5. With N×5 subscriptions, the scheduler receives N×5 tasks at t=0, then N×5 more every 500 ms.
6. When the 100,000-task queue fills, `BoundedElasticScheduler` throws `RejectedExecutionException`.
7. Because this is the **global** scheduler, the rejection also breaks the safety-check `subscribeOn(Schedulers.boundedElastic())` in `TopicMessageServiceImpl` line 70, the `PollingTopicMessageRetriever`, `NetworkServiceImpl`, and `SharedTopicListener` — all of which call `Schedulers.boundedElastic()`.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` limits streams per TCP connection but places no cap on total connections or total subscriptions server-wide.
- `subscriberCount` in `TopicMessageServiceImpl` (line 48) is a metric gauge only — it is never checked against a threshold to reject new subscriptions.
- There is no IP-based rate limit, no global subscription ceiling, and no back-pressure that would reject a new `listen()` call.

### Impact Explanation
When the global `boundedElastic` queue overflows, every reactive pipeline in the gRPC service that relies on that scheduler fails with `RejectedExecutionException`. This causes all active legitimate subscriptions to receive errors and terminate, and new subscriptions to fail immediately. The service becomes effectively unavailable for topic message streaming. Because the scheduler is global, the blast radius extends beyond `PollingTopicListener` to the retriever and network service. This is a non-network-based DoS (resource exhaustion via scheduler queue) affecting service availability.

**Condition:** Only active when `hiero.mirror.grpc.listener.type = POLL` (non-default; default is `REDIS`). Operators who deploy in POLL mode for direct-DB polling scenarios are exposed.

### Likelihood Explanation
An unprivileged attacker needs no credentials — the gRPC `subscribeTopic` endpoint is publicly accessible. Opening 20,000 TCP connections (each with 5 streams = 100,000 subscriptions) is within reach of a single machine with a high file-descriptor limit or a small botnet. The attack is repeatable: after connections are dropped and the queue drains, the attacker can re-flood. The 500 ms polling interval means the queue refills rapidly once connections are re-established. The only practical barrier is the non-default `POLL` configuration.

### Recommendation
1. **Add a global subscription limit:** In `TopicMessageServiceImpl.subscribeTopic()`, check `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Use a dedicated, isolated scheduler:** Replace `Schedulers.boundedElastic()` in `PollingTopicListener` with `Schedulers.newBoundedElastic(...)` so queue exhaustion is scoped to the polling listener and cannot cascade to other services.
3. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer.
4. **Configure Netty `maxConnectionAge` / `maxConnections`** to bound total simultaneous connections at the transport layer.

### Proof of Concept
```
# Precondition: server configured with type: POLL
# Step 1: open 20,000 connections, 5 streams each = 100,000 subscriptions
for i in $(seq 1 20000); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Step 2: observe server logs for RejectedExecutionException
# Step 3: attempt a legitimate subscription — it will fail with INTERNAL error
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
  <host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Expected: ERROR Code=Internal, RejectedExecutionException from boundedElastic scheduler
```
