### Title
Global Rate Limit Bucket Exhaustion via Single-IP Flooding (Application-Layer DoS)

### Summary
`ThrottleConfiguration.rateLimitBucket()` creates a single, process-wide `Bucket` with a capacity of `requestsPerSecond` (default 500) shared across all callers regardless of source IP or identity. An unprivileged attacker from a single IP can saturate this bucket by sending concurrent requests, causing every subsequent request from all other users on the same pod to receive HTTP 429 until the bucket refills. No per-IP isolation exists at the application layer.

### Finding Description

**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` (lines 24–32) constructs one `Bucket` instance as a Spring singleton:

```java
// web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java:24-32
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build();
}
```

`ThrottleManagerImpl.throttle()` (line 38) calls `tryConsume(1)` on this single shared bucket for every incoming request, with no reference to the caller's IP or identity:

```java
// web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java:38-39
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

**Root cause:** The bucket is a single in-memory token store. `Bucket.builder().addLimit(limit).build()` (no `SynchronizationStrategy` specified, unlike `gasLimitBucket`) uses the default lock-free strategy, meaning concurrent threads all drain the same pool. There is no keying by IP, session, or user.

**Why existing checks fail:**

1. **`LoggingFilter`** (web3) reads `request.getRemoteAddr()` only for logging — it performs no rate-limiting action on it.
2. **`ThrottleProperties.request`** (the `RequestProperties` filter list) defaults to `List.of()` — empty — so the optional per-pattern throttling is inactive by default.
3. **GCP gateway `maxRatePerEndpoint: 250`** limits requests *per backend pod*, not per source IP. An attacker can still send up to 250 req/s to a single pod.
4. **`sessionAffinity: CLIENT_IP`** at the GCP gateway level pins the attacker's IP to one pod consistently, making exhaustion of that pod's bucket more reliable, not less.
5. **Traefik middleware** is configured only as a `retry` middleware (3 attempts, 100ms interval) — no rate-limiting middleware is present.
6. **HPA is disabled by default** (`hpa.enabled: false`, `minReplicas: 1`), so no automatic scale-out mitigates the attack.

### Impact Explanation

When the 500-token bucket is drained, every `throttle()` call throws `ThrottleException`, which the controller maps to HTTP 429. All users routed to the affected pod — including those executing financial `eth_call` / `eth_estimateGas` transactions — are denied service for up to one full second per burst. Because the bucket refills greedily at 500 tokens/second, a sustained flood of exactly 500 req/s from one IP keeps the bucket perpetually empty, producing a continuous, complete denial of service for all other users on that pod. The `gasLimitBucket` (the secondary guard) is never reached because the rate-limit check fires first.

### Likelihood Explanation

No authentication or API key is required to call `/api/v1/contracts/call`. Sending 500 HTTP requests per second from a single host is trivially achievable with tools like `wrk`, `hey`, or a simple multi-threaded script. The attack is repeatable indefinitely, requires no special knowledge of the system, and produces an immediate, observable effect. With `sessionAffinity: CLIENT_IP` enabled, the attacker is deterministically routed to the same pod on every request, making the exhaustion stable rather than probabilistic.

### Recommendation

1. **Per-IP rate limiting at the application layer:** Replace the single global `Bucket` with a `ConcurrentHashMap<String, Bucket>` keyed by client IP (extracted from `X-Forwarded-For` or `RemoteAddr`), or use Bucket4j's built-in `BucketProxyManager` with a distributed cache (Redis/Hazelcast) for multi-pod deployments.
2. **Traefik `rateLimit` middleware:** Add a Traefik `RateLimit` middleware entry alongside the existing `retry` entry in `charts/hedera-mirror-web3/values.yaml` to enforce per-source-IP limits at the ingress layer before requests reach the JVM.
3. **Reduce the global burst capacity:** Lower `requestsPerSecond` to a value that reflects realistic per-user needs, and add a separate per-IP sub-limit.
4. **Enable HPA:** Set `hpa.enabled: true` so the cluster can scale out under load, reducing the blast radius of a single-pod exhaustion attack.

### Proof of Concept

**Preconditions:** Network access to the web3 endpoint; no credentials required.

**Steps:**

```bash
# Install wrk (HTTP benchmarking tool)
# Target: POST /api/v1/contracts/call with a minimal valid body

wrk -t 50 -c 500 -d 60s \
  -s post.lua \
  https://<mirror-node-host>/api/v1/contracts/call
```

Where `post.lua` contains:
```lua
wrk.method = "POST"
wrk.body   = '{"data":"0x","estimate":false,"gas":50000,"to":"0x0000000000000000000000000000000000000167"}'
wrk.headers["Content-Type"] = "application/json"
```

**Trigger:** The 500 concurrent connections saturate the `rateLimitBucket` within the first request window.

**Result:** All requests from other clients to the same pod immediately receive:
```
HTTP/1.1 429 Too Many Requests
{"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```
