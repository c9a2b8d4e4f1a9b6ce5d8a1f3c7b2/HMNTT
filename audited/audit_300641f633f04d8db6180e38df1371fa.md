### Title
Unauthenticated `/block` Hash Lookup Triggers Unbounded Correlated Subquery Execution, Enabling DB Resource Exhaustion

### Summary
The Rosetta `/block` endpoint accepts hash-only `block_identifier` requests from any unauthenticated caller and routes them directly to `findBlockByHash()`, which executes `selectByHashWithIndex` — a SQL query containing a correlated subquery against `record_file` — for every request. There is no application-level rate limiting, no result caching for block-by-hash lookups, and the Traefik-based rate limiting is disabled by default (`global.middleware: false`). An attacker with knowledge of valid block hashes (publicly available on-chain) can flood the endpoint with concurrent requests, driving DB CPU and I/O well above the 24-hour baseline.

### Finding Description

**Exact code path:**

`rosetta/main.go` wires the router with no rate-limiting middleware — only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` are applied: [1](#0-0) 

The `/block` request reaches `blockAPIService.Block()`, which calls `RetrieveBlock()`: [2](#0-1) 

`RetrieveBlock()` calls `blockRepo.FindByHash()` when only a hash is supplied: [3](#0-2) 

`FindByHash()` calls `findBlockByHash()`, which executes `selectByHashWithIndex` unconditionally for every non-empty hash: [4](#0-3) 

The SQL query contains a correlated subquery that re-enters `record_file` for every matched row: [5](#0-4) 

**Root cause:** No application-level rate limiting or result caching exists for the block-by-hash path. The `entityCache` in `blockAPIService` caches only account aliases, not block lookups: [6](#0-5) 

**Why existing checks fail:**

The only rate-limiting mechanism is the optional Traefik middleware chain (`inFlightReq: amount: 5`, `rateLimit: average: 10`), but it is gated behind `global.middleware: false` by default and is therefore not deployed in a standard installation: [7](#0-6) [8](#0-7) 

The only input validation in `FindByHash()` is an empty-string check; any non-empty string proceeds to the DB: [9](#0-8) 

The `record_file__hash` index uses `collate "C"` matching the query, so the outer lookup is efficient, but the correlated subquery (`where c.index = p.index + 1`) still executes a second index probe per matched row, doubling the index I/O per request: [10](#0-9) 

### Impact Explanation
Each concurrent `/block` request with a valid hash causes two indexed reads against `record_file` (outer hash lookup + inner index probe for `index + 1`). With no concurrency cap at the application layer, an attacker can sustain hundreds of simultaneous DB queries. On a production mirror node where `record_file` grows continuously and the DB baseline is already non-trivial, this can push DB CPU and I/O 30%+ above the 24-hour average, degrading service for all legitimate consumers of the Rosetta API and the shared mirror-node database.

### Likelihood Explanation
The preconditions are trivially met: block hashes are public (enumerable from the Hedera mirror REST API or any block explorer), no account or API key is required, and the endpoint is exposed via standard ingress. The attack is repeatable and scriptable with any HTTP client. A single attacker with a modest connection pool can sustain the load indefinitely. The default deployment (`global.middleware: false`) provides zero infrastructure-level protection.

### Recommendation
1. **Application-level rate limiting:** Add a per-IP or global in-flight request limiter directly in the Go HTTP handler chain in `main.go`, independent of optional Traefik configuration.
2. **Result caching:** Cache block-by-hash results (keyed on the normalized hash) with a short TTL (e.g., 5–10 seconds) using the existing `go-generics-cache` dependency already present in `block_service.go`.
3. **Enable middleware by default:** Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the `inFlightReq` and `rateLimit` Traefik middlewares are active in all standard deployments.
4. **Query optimization:** Replace the correlated subquery in `selectByHashWithIndex` with a window function (`LEAD`) or a self-join to reduce per-request DB round-trips.

### Proof of Concept
```bash
# 1. Enumerate valid block hashes from the public mirror REST API
HASHES=$(curl -s "https://mainnet-public.mirrornode.hedera.com/api/v1/blocks?limit=100" \
  | jq -r '.blocks[].hash')

# 2. Flood the Rosetta /block endpoint with valid, varying hashes (no auth required)
for HASH in $HASHES; do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"block_identifier\": {\"hash\": \"$HASH\"}
    }" &
done
wait

# 3. Repeat in a tight loop from multiple clients.
# Each iteration triggers findBlockByHash() -> selectByHashWithIndex with correlated subquery.
# Monitor DB CPU/IO: expect >30% increase above 24h baseline with ~50+ concurrent clients.
```

func (br *blockRepository) findBlockByHash(ctx context.Context, hash string) (*types.Block, *rTypes.Error) {
	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectByHashWithIndex, sql.Named("hash", hash)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}
