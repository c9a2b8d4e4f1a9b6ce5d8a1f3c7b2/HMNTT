### Title
Unauthenticated DoS via Expensive `DISTINCT ON` + Entity JOIN Query on `GET /api/v1/tokens/{tokenId}/balances`

### Summary
The `extractSqlFromTokenBalancesRequest()` function in `rest/tokens.js` generates a `DISTINCT ON (ti.account_id)` query against the large historical `token_balance` table with a cross-table JOIN to `entity` when both `account.publickey` and `timestamp` filters are supplied. The REST API has no rate-limiting middleware for unauthenticated users, so an attacker can flood this endpoint with concurrent requests, exhausting the PostgreSQL connection pool and degrading mirror node availability.

### Finding Description

**Exact code location:** `rest/tokens.js`, `extractSqlFromTokenBalancesRequest()`, lines 589–690.

When `timestamp=lte:X` is present, `tsConditions.length > 0` is true and the code enters the branch at line 638. It first calls `balances.getAccountBalanceTimestampRange()` to resolve a `[lower, upper]` range, then builds:

```sql
select distinct on (ti.account_id)
  ti.account_id,
  ti.balance,
  $N::bigint as snapshot_timestamp
from token_balance as ti
join entity as e
  on e.type = 'ACCOUNT'
  and e.id = ti.account_id
  and e.public_key = $M          -- injected when account.publickey is supplied
where ti.token_id = $1
  and ti.consensus_timestamp >= $P
  and ti.consensus_timestamp <= $Q
order by ti.account_id asc, ti.consensus_timestamp desc
limit $R
``` [1](#0-0) 

The `token_balance` table is a large, time-partitioned historical table. `DISTINCT ON` forces a sort over all matching rows before the `LIMIT` is applied, and the JOIN to `entity` adds a cross-table lookup for every candidate row. The timestamp range filter does not eliminate the sort cost.

**No rate limiting on the REST API.** The middleware stack in `rest/server.js` is: `requestLogger` → `authHandler` → optional `metricsHandler` → optional `responseCacheCheckHandler` → route handlers. There is no rate-limiting or throttling middleware. [2](#0-1) 

The `authHandler` only sets a custom response limit for authenticated users; it does not block or throttle unauthenticated requests. [3](#0-2) 

A `grep` across all `rest/**/*.js` for `rateLimit|rate.limit|express-rate|throttle` returns only a single match in a test utility file — confirming no production rate-limiting exists for the REST API. The throttle configuration found (`web3/src/main/java/.../ThrottleConfiguration.java`) applies exclusively to the Java web3 service, not the Node.js REST API. [4](#0-3) 

### Impact Explanation

An attacker can exhaust the PostgreSQL connection pool (`max` connections configured in `rest/dbpool.js` via `config.db.pool.maxConnections`) by sending many concurrent requests. Each connection is held for up to the `statement_timeout` of 20 000 ms (20 seconds, set for the REST DB user). [5](#0-4) [6](#0-5) 

Once the pool is exhausted, all REST API endpoints become unavailable, not just the token-balances endpoint. This matches the stated severity: shutdown of ≥30% of network processing nodes without brute force.

### Likelihood Explanation

- **No authentication required.** The endpoint is publicly documented and accessible.
- **No rate limiting.** An attacker with a single machine and a script can open hundreds of concurrent HTTP connections.
- **Repeatable.** Any valid `tokenId` and any syntactically valid public key (64-hex-char string) suffice; the key does not need to match any real account — the JOIN simply returns no rows, but the full scan still occurs.
- **Low skill required.** A simple `curl` loop or `ab`/`wrk` benchmark tool is sufficient.

### Recommendation

1. **Add rate-limiting middleware** to the REST API (e.g., `express-rate-limit`) applied globally before route handlers, with a per-IP limit appropriate for public API use.
2. **Reduce `statement_timeout`** for the REST DB user to a lower value (e.g., 5 000 ms) to free connections faster under attack.
3. **Consider query cost controls**: for the timestamp + publickey combination, require an `account.id` filter to narrow the scan before the `DISTINCT ON`, or rewrite the query to use an index-only lookup on `entity.public_key` first, then join to `token_balance`.
4. **Limit pool connections per endpoint** or use a connection queue with a short wait timeout so pool exhaustion returns HTTP 503 quickly rather than blocking indefinitely.

### Proof of Concept

```bash
# Requires: a running mirror node REST API and any valid tokenId
TOKEN_ID="0.0.1234"
PUBKEY="3c3d546321ff6f63d701d2ec5c277095874e19f4a235bee1e6bb19258bf362be"
TIMESTAMP="lte:9999999999.999999999"
URL="http://<mirror-node-host>/api/v1/tokens/${TOKEN_ID}/balances?account.publickey=${PUBKEY}&timestamp=${TIMESTAMP}"

# Send 200 concurrent requests (adjust -c to match or exceed pool maxConnections)
ab -n 10000 -c 200 "$URL"
```

**Expected result:** After the connection pool is exhausted (within seconds), all REST API endpoints return connection errors or HTTP 500/503, and legitimate clients cannot reach the mirror node REST API until the attack stops and connections drain.

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {

                .capacity(rateLimit)

                .build();
