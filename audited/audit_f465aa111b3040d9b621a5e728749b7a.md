### Title
Unbounded Redis Channel Subscription Growth via Distinct Topic ID Exhaustion in `RedisTopicListener`

### Summary
`RedisTopicListener.getSharedListener()` stores one Redis channel subscription per distinct topic ID in an unbounded `ConcurrentHashMap`. There is no cap on the number of distinct active Redis subscriptions, and no per-user or global limit on how many distinct topics can be subscribed to concurrently. An attacker who knows valid topic IDs (publicly enumerable via the REST API) can open many gRPC connections and subscribe to thousands of distinct topics, growing the map and the Redis subscription list without bound until server resources are exhausted.

### Finding Description

**Exact code path:**

`RedisTopicListener.getSharedListener()` at line 61: [1](#0-0) 

calls `computeIfAbsent` on an unbounded `ConcurrentHashMap`: [2](#0-1) 

Each miss invokes `subscribe(topic)` at lines 68–79, which calls `r.receive(...)` on the shared `ReactiveRedisMessageListenerContainer`, issuing a new Redis `SUBSCRIBE` command for that channel: [3](#0-2) 

The topic channel name is derived directly from the user-supplied topic ID: [4](#0-3) 

**Root cause:** The `topicMessages` map has no maximum size. `ListenerProperties` exposes `maxBufferSize`, `maxPageSize`, and `prefetch`, but no `maxTopics` or `maxSubscriptions` bound: [5](#0-4) 

**Why the `topicExists` check is insufficient:** `TopicMessageServiceImpl.topicExists()` (lines 94–106) rejects subscriptions to non-existent topics when `checkTopicExists = true` (the default): [6](#0-5) 

However, Hedera mainnet has tens of thousands of real, publicly enumerable topic IDs (queryable via the mirror node REST API at `/api/v1/topics`). An attacker does not need to fabricate topic IDs — they can enumerate real ones. If `checkTopicExists` is set to `false` (a documented operator option), the check is entirely bypassed and arbitrary numeric IDs work.

**Why `maxConcurrentCallsPerConnection = 5` is insufficient:** This Netty property limits concurrent gRPC calls per single TCP connection: [7](#0-6) 

An attacker opens many connections (e.g., 200 connections × 5 calls = 1,000 concurrent subscriptions to distinct topics). There is no global cap on total active subscriptions or on the size of `topicMessages`.

**Why `subscriberCount` is insufficient:** The `AtomicLong subscriberCount` in `TopicMessageServiceImpl` is only a Micrometer gauge metric — it is never checked against a maximum to reject new subscriptions: [8](#0-7) 

**Cleanup race:** `unsubscribe` fires on `doOnCancel`/`doOnComplete`, but with `.share()` the upstream Redis subscription is only cancelled when the last downstream subscriber for that topic cancels. During the window between subscription creation and cancellation, all Redis `SUBSCRIBE` commands are live. An attacker who subscribes and immediately disconnects still causes a burst of Redis subscriptions before cleanup.

### Impact Explanation

Each entry in `topicMessages` holds a live Redis pub/sub channel subscription on the `ReactiveRedisMessageListenerContainer`. With N distinct topics subscribed:
- The Redis server must maintain N channel subscriptions on its pub/sub connection, consuming memory proportional to N.
- The JVM heap holds N `ConcurrentHashMap` entries, each wrapping a `Flux` with a `.share()` operator and associated state.
- Redis message routing overhead grows with the number of subscribed channels.

At scale (e.g., 50,000–100,000 distinct topics), this can exhaust JVM heap, cause Redis OOM, or degrade Redis pub/sub throughput enough that legitimate gossip transaction messages are delayed or dropped for all subscribers — a denial-of-service against the mirror node's live topic message delivery.

### Likelihood Explanation

The gRPC endpoint is publicly accessible with no authentication. Valid topic IDs are freely enumerable via the public REST API. The attacker needs only a gRPC client (e.g., `grpcurl` or a simple Java/Go client), a list of real topic IDs, and the ability to open multiple TCP connections. The attack is fully scriptable, repeatable, and requires no special privileges. The default configuration (`checkTopicExists = true`, `maxConcurrentCallsPerConnection = 5`) provides only weak friction, not a hard bound.

### Recommendation

1. **Add a hard cap on `topicMessages` map size.** Reject `computeIfAbsent` when the map exceeds a configurable `maxTopicSubscriptions` limit (e.g., 1,000), returning a gRPC `RESOURCE_EXHAUSTED` error to the caller.
2. **Add a global active-subscription limit** in `TopicMessageServiceImpl`, checked before calling `topicListener.listen()`, analogous to the `maxActiveSubscriptions` pattern already designed for the GraphQL contract log listener.
3. **Add per-IP or per-connection subscription rate limiting** at the Netty/gRPC interceptor layer.
4. **Add `maxTopicSubscriptions` to `ListenerProperties`** and enforce it in `getSharedListener()`.

### Proof of Concept

```
# 1. Enumerate real topic IDs from the public REST API
curl "https://<mirror-node>/api/v1/topics?limit=1000" | jq '.[].topic_id'

# 2. Open many gRPC connections, each subscribing to 5 distinct topics
# (pseudocode — repeat with a gRPC client library)
for batch in batches_of_5(topic_ids[0:100000]):
    conn = grpc.insecure_channel("<mirror-node>:5600")
    stub = ConsensusServiceStub(conn)
    for topic_id in batch:
        stub.subscribeTopic(ConsensusTopicQuery(topicID=topic_id, consensusStartTime=now()))
    # hold connections open; each distinct topic_id causes one Redis SUBSCRIBE

# 3. Observe: topicMessages map grows to N entries; Redis SUBSCRIBE count grows to N;
#    JVM heap and Redis memory climb; legitimate subscribers experience message delivery failures.
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```
