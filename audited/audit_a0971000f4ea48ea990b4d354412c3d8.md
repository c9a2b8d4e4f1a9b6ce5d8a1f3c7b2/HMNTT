### Title
Missing Cache-Control Response Header on OpenAPI Spec Endpoint Enables Bandwidth Exhaustion

### Summary
The `serveSpec()` function in `rest/middleware/openapiHandler.js` serves the full OpenAPI YAML file on every request without setting any `Cache-Control` response header, preventing client-side caching. The spec routes are registered before the Redis response cache middleware, so they also bypass server-side caching entirely. An unprivileged attacker can repeatedly request the spec endpoint with no `If-None-Match` header, forcing the server to transmit the full YAML body on every request with no rate limiting in place.

### Finding Description
**Exact code location:** `rest/middleware/openapiHandler.js`, line 126:
```js
const serveSpec = (req, res) => res.type('text/yaml').send(getV1OpenApiFile());
```
No `Cache-Control`, `Expires`, or `ETag` response headers are explicitly set. Express may auto-set an `ETag`, but without `Cache-Control: max-age`, clients have no directive to cache the response and will revalidate or re-fetch on every request.

**Registration order in `rest/server.js`:**
- Line 62: `serveSwaggerDocs(app)` — registers the spec routes
- Line 97: `app.useExt(responseCacheCheckHandler)` — registers Redis cache middleware

Because the spec routes are registered at line 62, before the `responseCacheCheckHandler` at line 97, all requests to `/api/v1/docs/openapi.yml` and `/api/v1/docs/openapi.yaml` bypass the Redis response cache entirely.

**Why existing checks fail:**
- `getV1OpenApiFile()` (lines 54–61) does cache the file content in memory after the first read, so disk I/O is not repeated — but the full YAML string is serialized and transmitted over the network on every request that lacks a matching `If-None-Match` header.
- No rate limiting middleware exists anywhere in the chain for this endpoint.
- The Redis response cache (`responseCacheCheckHandler`) is never reached for these routes.

### Impact Explanation
An attacker can drive sustained bandwidth consumption by flooding the spec endpoint. Each request without `If-None-Match` receives the full YAML body. The OpenAPI spec for a mirror node is typically tens to hundreds of kilobytes. At high request rates, this translates to significant outbound bandwidth and CPU overhead for response serialization, with no server-side or client-side caching absorbing any of the load. Impact is classified as griefing/availability degradation with no economic damage to network users, consistent with the stated scope.

### Likelihood Explanation
No authentication or authorization is required. The endpoint is publicly accessible. No rate limiting exists. Any external user with network access can script repeated requests trivially (e.g., `while true; do curl -s .../openapi.yml > /dev/null; done`). The attack is repeatable, requires no special knowledge, and is not logged in a way that would trigger automated blocking.

### Recommendation
1. Add an explicit `Cache-Control: public, max-age=<seconds>` response header in `serveSpec()`, e.g.:
   ```js
   const serveSpec = (req, res) =>
     res.set('Cache-Control', 'public, max-age=3600')
        .type('text/yaml')
        .send(getV1OpenApiFile());
   ```
2. Move `serveSwaggerDocs(app)` to after the `responseCacheCheckHandler` registration in `server.js`, or explicitly add the spec routes to the Redis cache scope.
3. Add rate limiting middleware (e.g., `express-rate-limit`) to the spec and docs routes.

### Proof of Concept
```bash
# No authentication required, no If-None-Match sent, full YAML body returned every time
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{size_download}\n" \
    -H "Cache-Control: no-cache" \
    https://<mirror-node-host>/api/v1/docs/openapi.yml
done
# Each iteration returns the full YAML body (no 304), consuming full bandwidth per request
```
