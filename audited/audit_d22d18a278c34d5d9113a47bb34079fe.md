### Title
Unauthenticated Slow-Connection DoS via Unbounded Concurrent `/health/readiness` PostgreSQL Health Checks

### Summary
The `NewHealthController()` function in `rosetta/app/middleware/health.go` registers an unauthenticated `GET /health/readiness` endpoint that, on every invocation, opens a **new** PostgreSQL connection (via `hellofresh/health-go/v4/checks/postgres`) with a 10-second timeout. No rate limiting or concurrency cap is applied to this endpoint at the application layer, and the Traefik middleware that defines `inFlightReq` and `rateLimit` is both disabled by default (`global.middleware: false`) and scoped only to `/rosetta/*` ingress paths — not to `/health/*`. An attacker with network access to the service port can flood this endpoint with concurrent requests, holding each connection open for up to 10 seconds, exhausting PostgreSQL's `max_connections = 200` and denying service to legitimate application queries.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/health.go`, lines 37–43:
```go
readinessChecks := []health.Config{
    {
        Name:      "postgresql",
        Timeout:   time.Second * 10,
        SkipOnErr: false,
        Check:     postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
    },
```

The `postgres.New()` call from `github.com/hellofresh/health-go/v4/checks/postgres` returns a check function that, on each invocation, calls `sql.Open()` followed by `db.PingContext(ctx)` — opening a **fresh** database connection independent of the application's GORM pool. The connection is held until the ping completes or the 10-second context deadline fires.

**Root cause:** The check function creates a new `sql.DB` per invocation rather than reusing a persistent pool. Combined with no concurrency limit on the handler, N concurrent HTTP requests produce N concurrent PostgreSQL connection attempts, each lasting up to 10 seconds.

**Why existing checks fail:**

1. **Traefik middleware** (`charts/hedera-mirror-rosetta/templates/middleware.yaml`, line 3): rendered only when `global.middleware = true`; the default is `global.middleware: false` (`charts/hedera-mirror-rosetta/values.yaml`, line 95). Even when enabled, the ingress only routes `/rosetta/account`, `/rosetta/block`, etc. — `/health/readiness` is never behind the ingress and therefore never behind `inFlightReq` or `rateLimit`.

2. **Application GORM pool** (`rosetta/app/db/db.go`, line 33, `SetMaxOpenConns`): governs only the application's own pool. The health check's `sql.DB` is a separate handle; its connections are additive and uncapped.

3. **No application-level rate limiting or semaphore** exists anywhere in the middleware chain for health routes (`rosetta/app/middleware/`).

**PostgreSQL connection ceiling:** `rosetta/container/postgresql.conf`, line 4: `max_connections = 200`.

### Impact Explanation
When PostgreSQL's `max_connections = 200` is exhausted, every new connection attempt — including those from the application's GORM pool serving real API requests — receives `FATAL: sorry, too many clients already`. All Rosetta API endpoints that touch the database fail with 500 errors. The service becomes effectively unavailable for the duration of the attack (sustained with ~20 requests/second to keep 200 slots occupied across the 10-second window). This is a complete availability loss for the Rosetta API, rated **High** severity.

### Likelihood Explanation
The service is deployed as `ClusterIP` (`charts/hedera-mirror-rosetta/values.yaml`, line 266), so direct internet access is not the default. However, any pod co-located in the same Kubernetes namespace or cluster — including a compromised sidecar, a misconfigured workload, or an attacker who has achieved initial foothold inside the cluster — can reach the service IP directly. No credentials, tokens, or special headers are required. The attack is trivially scriptable, repeatable, and requires no knowledge of the application beyond the port number and path. In environments where the service port is exposed via NodePort, LoadBalancer, or port-forward, the attack surface extends to the network perimeter.

### Recommendation

1. **Reuse a persistent `sql.DB`**: Initialize the postgres health check's `sql.DB` once at startup and reuse it across checks. This bounds health-check connections to the pool's `MaxOpenConns`.

2. **Add a concurrency semaphore to the health handler**: Wrap `c.readinessHealth.HandlerFunc` with a semaphore (e.g., `golang.org/x/sync/semaphore`) limiting simultaneous in-flight health checks to a small number (e.g., 3–5).

3. **Enable and extend Traefik middleware to health paths**: Set `global.middleware: true` and add `/health/readiness` to the ingress paths so `inFlightReq` and `rateLimit` apply.

4. **Reduce the health check timeout**: 10 seconds is excessive for a liveness/readiness probe; 2–3 seconds limits the connection-hold window and aligns with the Kubernetes probe `timeoutSeconds: 2` already configured.

### Proof of Concept

```bash
# From any pod inside the cluster with access to the rosetta service IP:
ROSETTA_SVC="<rosetta-service-clusterip>:5700"

# Send 250 concurrent requests, each holding a DB connection for up to 10s
for i in $(seq 1 250); do
  curl -s "http://${ROSETTA_SVC}/health/readiness" &
done
wait

# Simultaneously, observe legitimate API calls failing:
curl -s -X POST "http://${ROSETTA_SVC}/network/list" \
  -H "Content-Type: application/json" \
  -d '{"metadata":{}}' 
# Returns 500 / DB error: "sorry, too many clients already"
```

While the 250 concurrent health requests are in-flight, each holds a PostgreSQL connection for up to 10 seconds. With `max_connections = 200`, the database rejects all new connection attempts from the application pool, causing complete API unavailability.
