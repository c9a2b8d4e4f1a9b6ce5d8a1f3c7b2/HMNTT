### Title
Unauthenticated Cache-Exhaustion DoS via Unbounded `subscribeTopic` Calls with Unique Non-Existent Topic IDs

### Summary
The `topicExists()` method in `TopicMessageServiceImpl` performs an uncached-miss-safe but volume-unlimited `entityRepository.findById()` call on every `subscribeTopic` invocation. Because the gRPC endpoint is unauthenticated, has no per-IP or global connection rate limit, and the entity cache is bounded at 50,000 entries with a 24-hour TTL, an attacker can flood the service with calls using distinct non-existent topic IDs, fill the cache with empty-Optional entries, and cause continuous LRU eviction of legitimate cached entities — forcing repeated DB hits that degrade performance for all users.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (no auth check) → `TopicMessageServiceImpl.subscribeTopic()` → `topicExists(filter)` at line 87 → `entityRepository.findById(topicId.getId())` at line 96. [1](#0-0) 

**Cache configuration:**

The entity cache is configured with `maximumSize=50000` and `expireAfterWrite=24h`: [2](#0-1) 

**Cache annotation — the critical detail:**

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = ENTITY_CACHE, unless = "#result == null")
Optional<Entity> findById(long entityId);
``` [3](#0-2) 

The `unless = "#result == null"` condition evaluates the `Optional<Entity>` return value. An `Optional.empty()` is **never null**, so empty Optionals (non-existent entities) **are cached**. This is correct for repeated lookups of the same ID, but it means every unique non-existent topic ID consumes one cache slot permanently for 24 hours.

**Root cause:** No rate limiting, no per-IP connection cap, and no global subscriber cap on the gRPC endpoint. The only server-side limit is `maxConcurrentCallsPerConnection = 5`: [4](#0-3) [5](#0-4) 

This limits concurrency **per connection**, not the total number of connections or the total call rate. An attacker opens N connections and issues 5 calls each, achieving 5N concurrent DB lookups with no server-side ceiling.

**Failed assumption:** The design assumes that the 50,000-entry cache absorbs repeated lookups for the same topic IDs. It does not account for an attacker deliberately cycling through >50,000 distinct IDs to saturate and continuously evict the cache.

### Impact Explanation

Once the attacker has submitted >50,000 unique non-existent topic IDs, Caffeine begins evicting entries. If the attacker sustains the flood with new IDs, every evicted slot is a legitimate topic entity that legitimate subscribers must re-fetch from the database. With enough concurrent attacker connections, the DB connection pool is saturated with `SELECT` queries for non-existent entities, causing:

- Increased query latency for all users
- DB connection pool exhaustion (blocking legitimate historical/live message retrieval)
- Potential cascading failure of the mirror node's gRPC service

The severity is high because the mirror node is a public-facing, unauthenticated service and the Hedera network's market capitalization places it in the ≥25% tier referenced in the scope.

### Likelihood Explanation

The attack requires zero privileges — the gRPC endpoint is plaintext and unauthenticated as shown in the public documentation: [6](#0-5) 

Topic IDs are sequential integers (`0.0.N`), so an attacker trivially generates 50,000+ unique IDs. Opening many TCP connections is a standard capability. The attack is repeatable and can be sustained indefinitely since the 24-hour TTL means evicted legitimate entries stay evicted for the duration of the attack window.

### Recommendation

1. **Add a global concurrent-subscriber cap** in `TopicMessageServiceImpl` using the existing `subscriberCount` `AtomicLong` — reject new subscriptions when the count exceeds a configurable threshold.
2. **Add per-IP rate limiting** at the gRPC server level (e.g., via a `ServerInterceptor` using a token-bucket per remote address).
3. **Fix the `unless` condition** to also exclude empty Optionals: `unless = "#result == null || !#result.isPresent()"` — this prevents non-existent entity IDs from consuming cache slots at all, eliminating the cache-filling vector entirely.
4. **Add a `maxConnections` limit** to `NettyProperties` and apply it via `serverBuilder.maxConnectionAge(...)` or an equivalent Netty server option.

### Proof of Concept

```bash
# Generate 60,000 unique non-existent topic IDs and flood subscribeTopic
# Each grpcurl call terminates immediately with EntityNotFoundException
# but not before hitting the DB and caching an empty Optional

for i in $(seq 1 60000); do
  grpcurl -plaintext \
    -d "{\"topicID\": {\"topicNum\": $((9000000 + i))}, \"limit\": 1}" \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  # Open a new connection every 5 calls to bypass maxConcurrentCallsPerConnection=5
  if (( i % 5 == 0 )); then wait; fi
done
```

**Preconditions:** Network access to port 5600; no credentials required.
**Trigger:** >50,000 unique topic IDs submitted; cache fills and begins evicting legitimate entries.
**Result:** Legitimate `subscribeTopic` calls for real topics miss the entity cache, each generating a DB `SELECT`; sustained flood exhausts the DB connection pool and degrades or halts message delivery for all subscribers.

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
