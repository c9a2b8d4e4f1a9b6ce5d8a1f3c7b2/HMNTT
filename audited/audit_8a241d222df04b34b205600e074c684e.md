### Title
Cache-Busting via Unique Query Parameters Bypasses Response Cache, Forcing Full Middleware and DB Execution on Every Request

### Summary
The response cache key is derived directly from `req.originalUrl` (the raw URL including query string) without any normalization. An unprivileged attacker can append an incrementing or random query parameter to any valid API endpoint, guaranteeing a cache miss on every request. Because `requestLogger()` and `requestQueryParser()` execute unconditionally before the cache check, and no rate limiting exists in the REST service, the attacker can sustain a high-volume stream of full-cost requests — each triggering crypto operations, query parsing, and a live database query — raising node CPU well above 30%.

### Finding Description

**Middleware execution order** (`rest/server.js`):

```
httpContext.middleware
→ requestLogger          (line 83)  ← always runs, no short-circuit
→ authHandler            (line 86)
→ responseCacheCheckHandler (line 97) ← cache check happens HERE
→ route handler (DB query)
→ responseHandler
→ responseCacheUpdateHandler
```

`requestQueryParser` is registered as Express's built-in query parser at line 60:
```js
app.set('query parser', requestQueryParser);
```
This means it runs on every incoming request as part of Express's own request-parsing phase, before any middleware is reached.

**`requestLogger()` always executes** (`rest/middleware/requestHandler.js` lines 22–29):

```js
const requestLogger = async (req, res) => {
  const requestId = await randomString(8);   // async crypto syscall
  httpContext.set(requestIdLabel, requestId);
  res.locals.statusCode = httpStatusCodes.OK.code;
  res.locals[requestStartTime] = Date.now();
};
```

There is no guard, no early return, and no cache awareness. It always calls `randomString(8)` (a `crypto.randomBytes` syscall) and sets context values.

**Cache key is the raw URL** (`rest/middleware/responseCacheHandler.js` line 151–153):

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

`req.originalUrl` includes the full query string verbatim. The code comment at line 148 explicitly acknowledges the absence of normalization: *"In the future, this will utilize Edwin's request normalizer (9113)."* Any unique query parameter produces a unique MD5 key, guaranteeing a cache miss.

**No rate limiting in the REST service**: A search across all `rest/**/*.js` files finds zero rate-limiting or throttle middleware. The `ThrottleConfiguration`/`ThrottleManagerImpl` found in the codebase belongs exclusively to the `web3` Java service and does not apply here.

**Exploit flow**:
1. Attacker sends `GET /api/v1/transactions?timestamp=gte:0&_=1`
2. `requestQueryParser` parses the query string (CPU: string iteration, object allocation).
3. `requestLogger` generates a random 8-byte ID via `crypto.randomBytes` (CPU: syscall).
4. `responseCacheCheckHandler` computes `md5("/api/v1/transactions?timestamp=gte:0&_=1")` → cache miss.
5. Route handler executes a live PostgreSQL query.
6. Attacker increments `_` to `2`, `3`, … — every request is a guaranteed cache miss.

### Impact Explanation
Every cache-busted request forces: (a) a `crypto.randomBytes` call in `requestLogger`, (b) full `qs.parse` + key-lowercasing loop in `requestQueryParser`, (c) an MD5 hash in `cacheKeyGenerator`, (d) a Redis lookup that always misses, and (e) a live database query. At sustained request rates (easily achievable from a single host with async HTTP clients), this exhausts CPU on both the Node.js process and the database backend. The 30% CPU threshold cited in the question is realistic at moderate request rates (hundreds of RPS) against endpoints with non-trivial DB queries (e.g., `/api/v1/transactions`).

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero tooling beyond a standard HTTP client. The parameter name and value are arbitrary — `?_=<epoch_ms>` or `?x=<counter>` both work. It is trivially scriptable, repeatable indefinitely, and undetectable as malicious by any existing middleware. Any public-facing deployment of this REST service is exposed.

### Recommendation
1. **Normalize the cache key**: Strip or ignore unknown/cache-busting query parameters before computing the cache key. Implement the query normalizer referenced in the existing TODO comment (issue 9113) — sort known parameters, drop unrecognized ones.
2. **Add rate limiting to the REST service**: Introduce a per-IP rate-limiting middleware (e.g., `express-rate-limit`) before `requestLogger` or at minimum before `responseCacheCheckHandler`.
3. **Move `requestLogger` after the cache check**: If a cached response can be served without a request ID, skip the `randomString` crypto call entirely for cache hits.
4. **Reject unknown query parameters**: The OpenAPI validator (`openApiValidatorEnabled`) can be configured to reject requests with undeclared query parameters, which would block the cache-busting vector entirely.

### Proof of Concept

```bash
# Single-host cache-busting loop (bash + curl)
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/transactions?timestamp=gte:0&_=${i}" -o /dev/null &
done
wait
```

Or with a higher-throughput tool:
```
wrk -t8 -c200 -d60s \
  --script <(echo 'math.randomseed(os.time()); \
    request=function() return wrk.format("GET", "/api/v1/transactions?timestamp=gte:0&_="..math.random(1,1e9)) end') \
  http://<mirror-node-host>
```

Every request hits a unique URL → unique MD5 cache key → guaranteed cache miss → full DB query. Monitor node CPU with `top` or Prometheus; it will climb proportionally to request rate with no cache relief.
