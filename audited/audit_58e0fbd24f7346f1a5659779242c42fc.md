### Title
Connection Pool Exhaustion DoS via Hanging Queries During Network Partition in `getScheduleById`

### Summary
`getScheduleById` in `rest/schedules.js` calls `pool.queryQuietly()` which internally acquires a pg pool connection for each request. During a network partition that causes TCP connections to hang (packets dropped without RST), an unprivileged attacker can send as few as 10 concurrent requests to exhaust all pool slots (`maxConnections` defaults to 10), causing every subsequent request to wait up to `connectionTimeoutMillis` (20 seconds) before receiving a `DbError`, effectively DoS-ing the endpoint for all users for the duration of the partition.

### Finding Description

**Code path:**

`rest/schedules.js` line 120:
```js
const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId);
```

`rest/utils.js` lines 1518–1520 — when `preQueryHint` is absent (always the case here):
```js
if (!preQueryHint) {
  result = await this.query(query, params);   // Pool.query() — holds a pool slot until resolved
```

`rest/dbpool.js` lines 13–15 — pool configuration:
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20000 ms
max: config.db.pool.maxConnections,                         // default 10
statement_timeout: config.db.pool.statementTimeout,         // default 20000 ms (server-side)
```

**Root cause and failed assumptions:**

1. `Pool.query()` (node-postgres) internally calls `connect()`, runs the query, then releases the client. While the query is in-flight, the pool slot is occupied. If the TCP connection hangs (network partition with packet-drop, not RST), the slot is held indefinitely — until the OS-level TCP keepalive timeout fires (default: hours on Linux).

2. `statement_timeout` is a **PostgreSQL server-side** setting. During a network partition where the DB is unreachable, the server cannot send the timeout signal back to the client. It provides zero protection against network-level hangs.

3. The `finally` block in `queryQuietly` only releases `client` when `preQueryHint` is set. For the `getScheduleById` path, `client` is always `undefined`, so the `finally` block is a no-op — connection lifecycle is entirely delegated to `Pool.query()` with no application-level abort.

4. The REST API (Node.js) has **no per-IP rate limiting, no concurrency limiting, and no application-level query timeout**. The middleware stack (`server.js` lines 68–97) includes only: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler` (response-limit only for authenticated users), `metricsHandler`, and optional response cache — none of which limit concurrent in-flight DB connections.

5. The throttling that exists (`ThrottleConfiguration`, `ThrottleManagerImpl`) is exclusively in the `web3` Java module and does not apply to the Node.js REST API.

**Exploit flow:**

1. Network partition occurs (or attacker arranges one) between the REST API pod and the PostgreSQL instance — specifically one where the firewall silently drops packets (no TCP RST), causing existing and new TCP connections to hang.
2. Attacker sends 10 concurrent HTTP requests to `GET /api/v1/schedules/<any_valid_id>`.
3. Each request reaches `pool.queryQuietly()` → `Pool.query()` → acquires one of the 10 pool slots → sends the SQL query → hangs waiting for a response that never arrives.
4. All 10 pool slots are now occupied by hanging connections.
5. Any subsequent request to any endpoint that uses `pool.queryQuietly()` (schedules, transactions, topics, accounts, etc.) queues for a free slot.
6. After `connectionTimeoutMillis` (20 seconds), the queued request receives a pg `Error: timeout exceeded when trying to connect` which is caught and re-thrown as `DbError` (line 1539), returning a 500 to the client.
7. The endpoint (and all other DB-backed endpoints sharing the same pool) is DoS'd for the duration of the partition plus the TCP keepalive timeout.

### Impact Explanation

All DB-backed REST API endpoints share the same global `pool` object (initialized once in `dbpool.js`). Exhausting the pool's 10 slots (the default) blocks every endpoint — not just `/schedules/:scheduleId`. Legitimate users receive 500 errors for the entire duration. The attack requires only 10 concurrent HTTP requests with no authentication, no special headers, and no payload — trivially achievable with `curl` or any HTTP client. Severity: **High** (availability impact, no authentication required, affects all users).

### Likelihood Explanation

Network partitions with silent packet drops are realistic in cloud environments (misconfigured security groups, network policy changes, transient routing failures). The attacker does not need to cause the partition — they only need to detect it (e.g., by observing slow responses) and immediately send 10 concurrent requests. The pool size of 10 is the documented default and is commonly deployed as-is. The attack is repeatable: each time the partition recurs, 10 requests re-exhaust the pool. No privileges, no account, no API key required.

### Recommendation

1. **Add a client-side query timeout**: Use `pg`'s `query_timeout` pool option (node-postgres ≥ 7.x) or wrap `Pool.query()` with `Promise.race()` against a `setTimeout` to abort hanging queries at the application layer, independent of the server-side `statement_timeout`.
2. **Increase `maxConnections`** or, more importantly, add a **concurrency semaphore** (e.g., `p-limit`) to cap the number of simultaneous in-flight DB queries, preventing a single burst from exhausting the pool.
3. **Add per-IP or global request rate limiting** to the Node.js REST API (e.g., `express-rate-limit`) to prevent a single client from consuming all pool slots.
4. **Enable TCP keepalive** on the pg pool connections with aggressive intervals so that dead connections are detected and released promptly rather than hanging for hours.
5. **Configure `idleTimeoutMillis`** on the pool so that idle connections that have gone stale (due to a partition) are evicted and re-established.
