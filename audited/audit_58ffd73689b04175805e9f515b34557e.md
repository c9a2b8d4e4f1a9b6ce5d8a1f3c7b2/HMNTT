### Title
Unbounded Concurrent Subscriptions Trigger Unthrottled DB Polling (20ms / 5000 rows) via Safety-Check Path, Enabling DoS

### Summary
Any unauthenticated gRPC client can open an unlimited number of `subscribeTopic` streams. Each stream without an `endTime` unconditionally fires a one-shot "safety check" after one second that calls `topicMessageRetriever.retrieve(gapFilter, false)` — the unthrottled path — which polls the database up to 12 times at 20 ms intervals with a page size of 5 000 rows. Because there is no cap on concurrent subscribers, an attacker multiplies this load linearly, exhausting database connections and CPU.

### Finding Description

**Unthrottled path parameters** (`RetrieverProperties.UnthrottledProperties`, defaults): [1](#0-0) 

- `maxPageSize = 5000`, `pollingFrequency = 20 ms`, `maxPolls = 12`

**How the unthrottled path is reached from a user subscription:**

`TopicMessageServiceImpl.subscribeTopic()` always starts historical retrieval with `throttled=true`, but unconditionally schedules a safety-check `Mono`: [2](#0-1) 

After one second it calls `missingMessages(topicContext, null)`, which calls: [3](#0-2) 

This invokes `retrieve(gapFilter, false)` — the unthrottled path.

**The safety check fires for every subscription without an `endTime`:**

`TopicContext.isComplete()` returns `false` whenever `endTime == null`: [4](#0-3) 

A client that omits `consensusEndTime` (the common case) guarantees the safety check fires.

**The unthrottled `PollingContext` never self-terminates without a limit:**

`isComplete()` for `throttled=false` only returns `true` on `limitHit`: [5](#0-4) 

Without a client-supplied `limit`, it polls all `maxPolls` (12) iterations at 20 ms with 5 000-row pages.

**No subscriber limit exists:**

`subscriberCount` is a metric gauge only — there is no enforcement of a maximum: [6](#0-5) 

The only per-connection cap is `maxConcurrentCallsPerConnection = 5`: [7](#0-6) 

An attacker opens many TCP connections, each carrying 5 streams, bypassing this limit entirely.

### Impact Explanation
Each subscription without an `endTime` generates up to 13 database queries (1 initial + 12 repeats) at 20 ms intervals, each fetching up to 5 000 rows. With N concurrent connections × 5 streams each, the attacker produces `N × 5 × 13 = 65N` database queries in approximately 260 ms. At N=100 connections (trivially achievable), that is 6 500 queries in a quarter-second burst, saturating the HikariCP connection pool and driving CPU to 100 %, denying service to legitimate users. The `timeout` of 60 s means each subscription holds resources for up to a minute before expiring.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly documented and requires no authentication: [8](#0-7) 

Any attacker with network access can script hundreds of concurrent `grpcurl` or SDK calls. No special knowledge, credentials, or chain state is required. The attack is repeatable and can be sustained indefinitely by reconnecting after the 60 s timeout.

### Recommendation
1. **Enforce a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: reject new subscriptions when `subscriberCount` exceeds a configurable maximum (e.g., 500), returning `RESOURCE_EXHAUSTED`.
2. **Rate-limit subscriptions per source IP** at the Netty/gRPC interceptor layer.
3. **Require `endTime` or `limit`** for subscriptions that would otherwise run indefinitely, or at minimum require a `limit` before the safety-check path can invoke the unthrottled retriever.
4. **Increase `unthrottled.maxPolls` minimum validation** or add a per-subscription unthrottled-call counter so a single subscription cannot trigger the unthrottled path more than once.

### Proof of Concept
```bash
# Open 200 concurrent subscriptions (40 connections × 5 streams each),
# each without endTime or limit, to any valid topic.
for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}}' \
    mirror-node:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
```

**Expected result:** After ~1 second, 200 safety checks fire simultaneously. Each triggers the unthrottled retriever: 200 × 13 = 2 600 DB queries at 20 ms intervals with 5 000-row pages. HikariCP connection pool is exhausted, CPU spikes, and legitimate subscribers receive errors or timeouts. The attack sustains itself by reconnecting every 60 s when the timeout fires.
