### Title
Unbounded Multi-Connection Subscription Fan-Out Enables Resource Exhaustion on gRPC Topic Listener

### Summary
`grpcServerConfigurer()` enforces only a per-connection call limit (`maxConcurrentCallsPerConnection=5`) with no cap on the total number of connections or subscriptions from a single IP or unauthenticated client. An unprivileged attacker can open an arbitrary number of TCP connections, each carrying 5 concurrent `subscribeTopic` streaming calls to the same high-traffic topic, forcing the server to fan out every incoming message to all active subscriber pipelines and driving CPU and memory well above the 30% threshold.

### Finding Description
**Code path:**

`GrpcConfiguration.java` lines 28–35 configures the Netty server with only two constraints: a shared executor and `maxConcurrentCallsPerConnection` sourced from `NettyProperties`: [1](#0-0) 

`NettyProperties.java` sets the default to 5 and enforces a minimum of 1, but places no ceiling and no per-IP or global connection limit: [2](#0-1) 

`SharedTopicListener.listen()` allocates a per-subscriber backpressure buffer (up to `maxBufferSize=16384` messages) and a `boundedElastic` scheduler slot for every subscriber independently: [3](#0-2) 

`RedisTopicListener.getSharedListener()` uses a single shared Redis subscription per topic (via `.share()`), but every gRPC subscriber attaches its own downstream pipeline to that shared `Flux`: [4](#0-3) 

There is no server interceptor, no connection-count guard, and no subscription-count guard anywhere in the `grpc` module. The only rate-limiting code (`ThrottleConfiguration`) lives exclusively in the `web3` module and is not applied here.

**Root cause:** `maxConcurrentCallsPerConnection` is a *per-connection* ceiling, not a global or per-source-IP ceiling. Nothing prevents an attacker from opening an unbounded number of connections, each saturating its 5-call quota.

**Exploit flow:**
1. Attacker identifies a high-traffic topic (e.g., the public HCS topic used by the network monitor).
2. Attacker opens *N* TCP connections to port 5600 (no authentication, no IP-level connection limit at the application layer).
3. On each connection, attacker issues 5 concurrent `subscribeTopic` streaming RPCs to the same topic.
4. The server now maintains *N × 5* active subscriber pipelines. Each message arriving from Redis is dispatched to all *N × 5* pipelines: deserialized, filtered, buffered (up to 16 384 messages each), and scheduled on `boundedElastic` threads.
5. CPU rises proportionally to subscriber count (serialization, reactive scheduling, buffer management); memory rises proportionally to the number of allocated per-subscriber buffers.

### Impact Explanation
With a high-traffic topic and a moderate number of connections (e.g., 50 connections × 5 calls = 250 simultaneous subscriptions), the server must perform 250× the work per message compared to a single subscriber. Each subscriber holds up to 16 384 buffered messages in memory. On a busy topic this easily produces >30% CPU and memory increase over baseline, degrading or denying service to legitimate subscribers. The `GrpcHighCPU` alert threshold is 80% CPU; the attack can push the node past that threshold without triggering any earlier circuit-breaker. [5](#0-4) 

### Likelihood Explanation
No authentication is required. The gRPC port (5600) is publicly exposed. The attack requires only a standard gRPC client library and a loop to open connections. It is repeatable, scriptable, and requires no special knowledge beyond the publicly documented topic ID. The GCP gateway `maxRatePerEndpoint: 250` limits new *request initiations per second*, not the number of simultaneously open streaming subscriptions, so it does not mitigate sustained open connections. [6](#0-5) 

### Recommendation
1. **Add a global active-subscription counter** in the topic service layer (analogous to the `AtomicLong subscriberCount` pattern already described in the GraphQL design doc) and reject new subscriptions above a configurable `maxActiveSubscriptions` limit.
2. **Add a per-source-IP connection limit** via a Netty `ServerBuilder` customizer (e.g., `maxConnectionsPerIp`) or a gRPC `ServerInterceptor` that tracks and rejects connections exceeding a per-IP threshold.
3. **Expose `maxConcurrentCallsPerConnection` as insufficient alone** — document that operators must also configure infrastructure-level connection limits (e.g., GCP BackendPolicy `maxConnectionsPerEndpoint`) to complement the application-level setting.
4. Consider adding a gRPC-level rate-limiting interceptor (analogous to the `ThrottleConfiguration` Bucket4j pattern in `web3`) that limits subscription initiations per IP per second.

### Proof of Concept
```bash
# Install grpcurl; replace TOPIC_NUM with a known high-traffic topic
TOPIC_NUM=41110
SERVER=<mirror-node-host>:5600

# Open 50 connections, each with 5 concurrent subscribeTopic streams (250 total)
for conn in $(seq 1 50); do
  for call in $(seq 1 5); do
    grpcurl -plaintext \
      -d "{\"topicID\": {\"topicNum\": $TOPIC_NUM}, \"limit\": 0}" \
      $SERVER \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Monitor server CPU and memory over the next 60 seconds
# Compare against baseline (0 attacker subscriptions) from the preceding 24h
# Expected: >30% increase in CPU utilization and heap usage
```
All 250 streams remain open indefinitely (no server-side timeout for active streaming subscriptions with `limit: 0`), sustaining the resource pressure without any further attacker action.
