### Title
Unprivileged Users Trigger Unthrottled DB Polling via Automatic Safety Check

### Summary
`TopicMessageServiceImpl.subscribeTopic()` unconditionally schedules a safety-check after 1 second for every user-initiated subscription, and that safety-check calls `topicMessageRetriever.retrieve(gapFilter, false)` — the unthrottled code path. Because the unthrottled `PollingContext` uses a 5× larger page size (5,000 vs 1,000), a 100× faster polling interval (20 ms vs 2 s), and its `isComplete()` never returns `true` unless a limit is hit, any unprivileged user who subscribes without a limit reliably exhausts all `maxPolls` (12) unthrottled DB polls per subscription. Multiplied across many concurrent connections this creates disproportionate database load.

### Finding Description

**Code path 1 — Safety check (primary):**

`TopicMessageServiceImpl.java` lines 66–70:
```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```
`missingMessages(topicContext, null)` (line 149):
```java
return topicMessageRetriever.retrieve(gapFilter, false);   // throttled=false
```
The `gapFilter` is built from the user's original filter without adding a limit (lines 146–147). If the user's subscription has no `endTime` and no `limit`, `topicContext.isComplete()` is always `false`, so the safety check fires for every subscription.

**Code path 2 — Gap fill (secondary):**

Line 177:
```java
return topicMessageRetriever.retrieve(newFilter, false).concatWithValues(current);
```
Here `numMissingMessages` is used as a limit, so the damage is bounded by the actual gap size.

**Root cause — unthrottled `PollingContext` behavior (`PollingTopicMessageRetriever.java` lines 94–128):**

| Property | `throttled=true` | `throttled=false` |
|---|---|---|
| `maxPageSize` | 1,000 | **5,000** |
| `pollingFrequency` | 2 s | **20 ms** |
| `numRepeats` | `Long.MAX_VALUE` | `maxPolls` (12) |
| `isComplete()` stops on partial page | YES | **NO** |

Because `isComplete()` for the unthrottled path only returns `true` on `limitHit`, and the safety-check filter carries no limit, the retriever always exhausts all 12 polls — even when the DB returns zero rows.

**Failed assumption:** The design treats the safety check as a rare exceptional event. In reality, any user subscription without `endTime`/`limit` guarantees it fires.

### Impact Explanation

Each limitless subscription triggers 12 DB queries at 20 ms intervals with page size 5,000 — up to 60,000 rows fetched per subscription. Under throttled mode the same subscription would issue at most one query per 2 s with page size 1,000. An attacker opening N concurrent gRPC streams multiplies this by N, causing database CPU/IO saturation and degraded service for all users. No authentication or rate-limiting gate exists between the gRPC endpoint and the safety-check trigger.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly reachable by any client. The attacker needs only to open many long-lived subscriptions (no `endTime`, no `limit`) to a valid topic and wait 1 second. The attack is trivially scriptable, repeatable, and requires zero privileges. The 1-second delay before the safety check fires is not a meaningful barrier.

### Recommendation

1. **Add a limit to the safety-check filter.** Before calling `retrieve(gapFilter, false)`, set a reasonable upper bound (e.g., `unthrottled.getMaxPageSize()`) on `gapFilter` so `isComplete()` can terminate early.
2. **Rate-limit or cap concurrent subscriptions per client/IP** at the gRPC interceptor layer before the safety check can be scheduled.
3. **Consider passing `throttled=true` for the safety check** unless there is a concrete operational reason to use unthrottled settings for gap recovery; the bounded `maxPolls` of the unthrottled path is not a sufficient safeguard when `isComplete()` never fires.
4. **Enforce a maximum subscription count** via the existing `subscriberCount` gauge — currently it is only a metric, not an enforced limit.

### Proof of Concept

```
# 1. Open N concurrent gRPC streams to the mirror node subscribeTopic endpoint
#    with no endTime and no limit, targeting any valid topic.
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 2. After ~1 second each subscription's safety check fires, calling
#    retrieve(gapFilter, false) — 12 polls × 5,000 rows × 200 streams
#    = up to 12,000,000 row fetches in ~240 ms.

# 3. Observe DB CPU spike and increased latency for all other clients.
```
