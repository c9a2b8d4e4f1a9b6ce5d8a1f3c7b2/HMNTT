### Title
Unauthenticated Repeated `getNfts()` Calls with Max Limit Exhaust DB Connection Pool via Multi-Union Query Amplification

### Summary
The `getNfts()` function in `rest/service/nftService.js` builds a SQL query that, when token.id and serialnumber range filters are supplied, generates up to three `UNION ALL` subqueries each carrying `limit $2` independently. Because the REST API has no rate-limiting middleware and the endpoint requires no authentication, an unprivileged attacker can flood the endpoint with max-limit requests, each causing up to 3× the declared limit in DB rows to be fetched and held in memory simultaneously, exhausting the DB connection pool and degrading REST API availability.

### Finding Description

**Code path:**

`rest/controllers/accountController.js` `getNftsByAccountId` (line 90–103) calls `NftService.getNfts(query)` after building the query via `extractNftMultiUnionQuery` (lines 34–82).

`rest/service/nftService.js` `getQuery()` (lines 84–126): when `lower`, `inner`, and `upper` filter arrays are all non-empty (triggered by providing both `token.id=gte:X&token.id=lte:Y` and `serialnumber=gt:A&serialnumber=lt:B`), three independent subqueries are assembled with `union all`, each containing `limit $2`:

```js
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
```

The `params` array is `[ownerAccountId, limit]` and `limitClause = super.getLimitQuery(2)` = `limit $2`. Every subquery shares the same `$2` binding. With `limit=100` (the default max), the DB executes three scans each returning up to 100 rows = **300 rows fetched per request** before the outer `limit $2` trims the final result to 100.

`super.getRows()` in `rest/service/baseService.js` (lines 55–57) materialises the full result set in Node.js heap memory before returning.

**Limit cap:** `getLimitParamValue()` in `rest/utils.js` (lines 544–553) caps the user-supplied limit at `responseLimit.max` (default **100**). This is the only per-request bound.

**No rate limiting on the REST API:** `rest/server.js` (lines 68–98) registers only `httpContext.middleware`, `requestLogger`, `authHandler`, optional `metricsHandler`, and optional `responseCacheCheckHandler`. There is no IP-based or global request-rate-limiting middleware. The `ThrottleManagerImpl` / `ThrottleConfiguration` classes exist exclusively in the `web3` Java module and do not apply here.

**No authentication required:** `authHandler` (lines 15–36) simply returns without blocking when no `Authorization` header is present.

### Impact Explanation

Each request with full range filters causes the PostgreSQL driver to fetch up to 300 rows and hold them in the Node.js process heap until the response is serialised. With no rate limiting, an attacker can open many concurrent connections, each consuming a DB pool slot (`maxConnections` is the only backstop) and a proportional heap allocation. Once the pool is saturated, all subsequent legitimate queries queue or time out. Because the REST API is the primary consumer-facing interface of the mirror node, sustained pool exhaustion degrades or halts query serving capacity, meeting the ≥30% processing-capacity impact threshold for the stated severity class.

### Likelihood Explanation

The attack requires zero privileges, zero credentials, and only a standard HTTP client. The endpoint is publicly documented in `openapi.yml`. The attacker needs only to know a valid account ID (trivially obtained from any prior API call) and supply two token.id range parameters plus two serialnumber range parameters to trigger the three-subquery path. The attack is trivially scriptable and repeatable with tools like `ab`, `wrk`, or a simple loop. No brute force or cryptographic work is required.

### Recommendation

1. **Add rate limiting middleware** to the Express REST API (e.g., `express-rate-limit` keyed on IP) before route handlers, mirroring the token-bucket approach already used in the web3 module.
2. **Reduce the multi-union per-subquery limit**: in `nftService.js` `getQuery()`, when multiple subqueries are generated, pass `Math.ceil(limit / subQueries.length)` as the per-subquery limit rather than the full `limit`, so total DB rows fetched stays bounded at `limit` rather than `subQueries.length × limit`.
3. **Set a DB statement timeout** (`statementTimeout`) in the REST pool configuration to bound the maximum wall-clock time any single query can hold a connection.

### Proof of Concept

```bash
# Trigger the 3-subquery union path with max limit on a known account
# Replace ACCOUNT_ID with any valid account (e.g. 0.0.1001)
for i in $(seq 1 500); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1001/nfts?\
token.id=gte:0.0.1&token.id=lte:0.0.9999999&\
serialnumber=gt:1&serialnumber=lt:9999999&\
limit=100" &
done
wait
# Observe: DB connection pool exhausted, subsequent legitimate requests
# return 500 or hang until statementTimeout fires.
```

Each concurrent request executes:
```sql
(SELECT ... FROM nft LEFT JOIN entity ... WHERE account_id=$1 AND serial_number>$3 ORDER BY token_id DESC,serial_number DESC LIMIT $2)
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity ... WHERE account_id=$1 AND token_id>$4 ORDER BY token_id DESC,serial_number DESC LIMIT $2)
UNION ALL
(SELECT ... FROM nft LEFT JOIN entity ... WHERE account_id=$1 AND token_id=$5 AND serial_number<$6 ORDER BY token_id DESC,serial_number DESC LIMIT $2)
ORDER BY token_id DESC,serial_number DESC LIMIT $2
```
with `$2 = 100`, fetching up to 300 rows per request while holding one DB connection per in-flight query.
