### Title
Thread Pool Exhaustion via Synchronous JDBC Blocking in `getByEvmAddressAndType()` During Network Partition

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` issues a synchronous, blocking JDBC call to `entityRepository.findByEvmAddress()` with no JDBC socket timeout configured on the runtime HikariCP connection pool. During a network partition between the GraphQL service and the database, the PostgreSQL `statement_timeout` is never triggered (it requires the query to reach the server), so each in-flight request blocks its servlet thread indefinitely. An unauthenticated attacker who sends concurrent requests during such a partition exhausts Tomcat's thread pool, rendering the entire GraphQL service unresponsive.

### Finding Description

**Exact code path:**

`EntityServiceImpl.getByEvmAddressAndType()` at line 40 issues a synchronous blocking call:

```java
return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
``` [1](#0-0) 

This resolves to the native SQL query in the GraphQL module's `EntityRepository`:

```java
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
``` [2](#0-1) 

**Root cause — no JDBC socket timeout on the runtime pool:**

The `socketTimeout` of 2 seconds defined in `CommonProperties.DatabaseStartupProperties` is used exclusively by `DatabaseWaiter.waitForDatabase()` during application startup: [3](#0-2) 

The runtime `DataSource` is built from `spring.datasource.hikari` properties via `CommonConfiguration`: [4](#0-3) 

The GraphQL module has no `application.yml` and its documented configuration exposes no HikariCP `socketTimeout` or per-connection read timeout: [5](#0-4) 

This means runtime JDBC connections have no socket-level read deadline.

**Why `statement_timeout` does not protect here:**

The PostgreSQL `statement_timeout` of 10 000 ms is set server-side for the `mirror_graphql` role: [6](#0-5) 

During a true network partition (packets dropped between the application and the database), the query bytes are buffered by the OS and the JDBC driver returns control to the application layer believing the send succeeded. The query never reaches PostgreSQL, so the server-side timer never starts. The JDBC driver then blocks indefinitely on the socket read waiting for a response that will never arrive.

**No rate limiting in the GraphQL module:**

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module: [7](#0-6) 

The GraphQL module's only protections are query complexity and depth limits: [8](#0-7) 

These do not bound the number of concurrent in-flight requests.

### Impact Explanation

Tomcat's default servlet thread pool is 200 threads. With no rate limiting and no socket timeout, an attacker sending 200+ concurrent GraphQL queries that reach `getByEvmAddressAndType()` during a network partition will hold all threads blocked on JDBC reads. New incoming requests queue and then fail with connection-refused or timeout errors. The entire GraphQL service becomes unresponsive for the duration of the partition. Because the GraphQL API is documented as publicly accessible with no authentication, this is a complete, unauthenticated denial-of-service against the service.

### Likelihood Explanation

The attacker requires two conditions: (1) a network partition between the GraphQL pod and the database, and (2) the ability to send HTTP POST requests to `/graphql/alpha`. Condition (2) requires no privilege whatsoever — the endpoint is public. Condition (1) can arise from infrastructure failure (making this exploitable opportunistically) or can be induced by an attacker with network-layer access (BGP manipulation, ARP poisoning in shared environments, cloud security-group misconfiguration). The GraphQL API is explicitly documented as alpha and not recommended for production, but it is deployed and reachable. The attack is trivially repeatable with a simple HTTP flood script.

### Recommendation

1. **Set a JDBC socket timeout on the runtime HikariCP pool** by adding `socketTimeout` to the JDBC URL or as a HikariCP `dataSourceProperty` in the GraphQL module's configuration (e.g., `spring.datasource.hikari.data-source-properties.socketTimeout=15`). This ensures the JDBC driver abandons a hung socket read after a bounded interval regardless of whether the query reached the server.
2. **Add rate limiting to the GraphQL module** analogous to the `ThrottleConfiguration` in the `web3` module, bounding concurrent in-flight requests per IP or globally.
3. **Configure a HikariCP `connectionTimeout`** (time to acquire a connection from the pool) and `maxLifetime` to prevent stale connections from accumulating after a partition heals.
4. Consider making `getByEvmAddressAndType()` (and the other repository calls) reactive/non-blocking, or offloading them to a bounded executor, so that blocked DB calls do not consume servlet threads.

### Proof of Concept

**Preconditions:**
- GraphQL service is running and reachable at `http://<host>:8083/graphql/alpha`
- A network partition is induced between the GraphQL pod and the PostgreSQL database (e.g., via `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the GraphQL host, or by dropping the route)

**Steps:**
```bash
# 1. Induce partition (on GraphQL host or via network policy)
iptables -A OUTPUT -p tcp --dport 5432 -j DROP

# 2. Flood with concurrent account-by-evmAddress queries (no auth required)
for i in $(seq 1 250); do
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{account(input:{evmAddress:\"0x000000000000000000000000000000000000abcd\"}){balance}}"}' &
done
wait

# 3. Attempt a new request — service is unresponsive
curl -v -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}"}'
# Expected: connection hangs or is refused — thread pool exhausted
```

**Result:** All 200+ Tomcat threads are blocked in `EntityServiceImpl.getByEvmAddressAndType()` → `EntityRepository.findByEvmAddress()` → JDBC socket read. The service stops accepting new requests until the partition is resolved and the blocked threads eventually time out (which, without a socket timeout, may never happen automatically).

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
