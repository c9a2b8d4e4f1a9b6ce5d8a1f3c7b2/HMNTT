### Title
Unbounded `token.id=eq` Filter Accumulation Causes Non-Parameterized IN Clause + Dual DB Query Compounding in `getTokenRelationships`

### Summary
`getTokenRelationships` in `rest/controllers/tokenController.js` accepts up to 100 `token.id=eq:X` query parameters (the global `maxRepeatedQueryParameters` ceiling), accumulates them all into `inConditions`, and passes them to `getInClauseSubQuery` in `rest/service/tokenService.js`, which **directly interpolates the values into the SQL string** rather than using parameterized placeholders. Combined with a `limit` of up to 100 and a mandatory second DB query for token-cache population, every maximally-crafted request forces two non-trivial database round-trips with a unique, non-cacheable query plan, enabling a griefing amplification loop by any unauthenticated caller.

### Finding Description

**Code path:**

`getTokenRelationships` (tokenController.js:66) →
`extractTokensRelationshipQuery` (tokenController.js:22-58) →
`TokenService.getTokenAccounts` (tokenService.js:96) →
`getTokenRelationshipsQuery` (tokenService.js:46) →
`getInClauseSubQuery` (tokenService.js:78-87) →
second call `getCachedTokens` (tokenService.js:136)

**Root cause — non-parameterized IN clause:**

`getInClauseSubQuery` pushes every `eq` value into a plain JS array and interpolates it directly into the SQL string:

```js
// tokenService.js:84-86
if (!isEmpty(tokenIdInParams)) {
  return ` and ${TokenAccount.getFullName(TokenAccount.TOKEN_ID)} in (${tokenIdInParams})`;
}
```

The values are **never appended to `params`**. The resulting SQL is:

```sql
… AND ta.token_id in (1000,2000,3000,…,100000) ORDER BY … LIMIT $2
```

Each unique combination of token IDs produces a structurally distinct SQL string. PostgreSQL cannot reuse a cached plan; it must parse, analyze, and plan every request from scratch.

**Root cause — no per-endpoint cap on `eq` repetitions:**

`extractTokensRelationshipQuery` (tokenController.js:34-35) pushes every `eq` filter unconditionally into `inConditions`. The only guard is the global `maxRepeatedQueryParameters` (default **100**, docs line 582). There is no tighter per-endpoint limit for this parameter.

**Compounding second query:**

After the primary query returns up to 100 rows, `getTokenAccounts` (tokenService.js:103-104) collects all distinct `token_id` values and calls `getCachedTokens`, which issues a second DB query (`tokenCacheQuery`, tokenService.js:31-38) for every token ID not already in the in-process LRU cache. A cold-cache attacker (rotating token IDs) forces this second query to touch up to 100 additional rows on every request.

**Exploit flow:**

1. Attacker identifies (or creates) any valid account ID — no privilege required.
2. Sends: `GET /api/v1/accounts/{id}/tokens?token.id=1&token.id=2&…&token.id=100&limit=100`
3. `buildAndValidateFilters` (utils.js:1208) accepts all 100 values (≤ `maxRepeatedQueryParameters`).
4. `extractTokensRelationshipQuery` accumulates 100 `inConditions`.
5. `getInClauseSubQuery` emits a 100-element raw IN clause; PostgreSQL must plan a unique query.
6. DB returns up to 100 rows; application iterates all of them.
7. `getCachedTokens` fires a second DB query for up to 100 uncached token IDs.
8. Attacker repeats with a different permutation of 100 token IDs → new plan, cache miss, two DB hits again.

**Why existing checks fail:**

- `maxRepeatedQueryParameters = 100` is the ceiling, not a mitigation — it *permits* 100 IN values.
- `limit` max of 100 bounds the result set but does not reduce the IN clause size or prevent the second query.
- No per-IP or per-account rate limiting is present in the REST layer code.
- The LRU token cache (`quickLru`, tokenService.js:12-14) is bypassed whenever the attacker rotates token IDs across requests.

### Impact Explanation

Each maximally-crafted request forces PostgreSQL to: (a) parse and plan a unique 100-element IN query, (b) execute it, and (c) execute a second `token_id = any($1)` query for up to 100 IDs. Query-plan-cache pollution degrades planning performance globally for all concurrent users. Sustained at modest concurrency (e.g., 50 req/s from a single host), this can saturate DB CPU and connection pool slots, causing latency spikes or timeouts for legitimate API consumers. No account balance or on-chain state is affected; impact is availability/performance only.

### Likelihood Explanation

The endpoint is public and unauthenticated. Constructing the maximally-amplifying request requires only knowledge of valid token IDs (publicly enumerable via `/api/v1/tokens`). No special tooling is needed — a simple shell loop or `ab`/`wrk` invocation suffices. The attack is repeatable indefinitely and stateless (no session required). Rotating the 100 token IDs per request defeats the LRU cache, ensuring the second DB query fires every time.

### Recommendation

1. **Parameterize the IN clause.** Append each value to `params` and emit `$N` placeholders, matching the pattern already used for `conditions` (tokenService.js:54-57). This allows PostgreSQL to cache and reuse the query plan.
2. **Add a hard per-endpoint cap** on the number of `token.id=eq` filters (e.g., 10), independent of the global `maxRepeatedQueryParameters`.
3. **Apply rate limiting** at the middleware layer (e.g., express-rate-limit) keyed on IP or account ID for this endpoint.
4. **Consider merging the two DB queries** into a single JOIN against the `token` table to eliminate the second round-trip entirely.

### Proof of Concept

```bash
# Enumerate 100 valid token IDs first (public endpoint)
TOKEN_IDS=$(curl -s "https://<mirror>/api/v1/tokens?limit=100&order=asc" \
  | jq -r '.tokens[].token_id' | tr '\n' '&' \
  | sed 's/&/\&token.id=/g; s/^/token.id=/')

# Target any valid account
ACCOUNT="0.0.12345"

# Single maximally-amplifying request
curl -s "https://<mirror>/api/v1/accounts/${ACCOUNT}/tokens?${TOKEN_IDS}&limit=100"

# Sustained griefing loop (rotate token IDs each iteration to defeat LRU cache)
while true; do
  OFFSET=$((RANDOM % 10000))
  TOKEN_IDS=$(curl -s "https://<mirror>/api/v1/tokens?limit=100&order=asc&token.id=gt:${OFFSET}" \
    | jq -r '.tokens[].token_id' | tr '\n' '&' \
    | sed 's/&/\&token.id=/g; s/^/token.id=/')
  curl -s "https://<mirror>/api/v1/accounts/${ACCOUNT}/tokens?${TOKEN_IDS}&limit=100" -o /dev/null &
done
```

Each iteration produces a structurally unique SQL string (new plan) and a cold-cache second query, compounding DB CPU and connection load with no authentication required.
