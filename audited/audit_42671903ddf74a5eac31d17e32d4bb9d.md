### Title
Unbounded Multi-Connection Subscription Amplification Enables DB Polling Resource Exhaustion via `grpcServerConfigurer()`

### Summary
`grpcServerConfigurer()` enforces only a per-connection call limit (`maxConcurrentCallsPerConnection = 5`) with no restriction on the number of connections a single client may open. When the `POLL` listener type is active, each `subscribeTopic` call instantiates an independent `PollingContext` with its own database polling loop executing every 500 ms. An unprivileged attacker opening many connections can therefore multiply DB query frequency and CPU consumption linearly, easily exceeding 30% above baseline.

### Finding Description
**Exact code path:**

`GrpcConfiguration.grpcServerConfigurer()` (lines 28–35) configures the Netty server with a single constraint:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

`NettyProperties` hard-codes this to `5` (line 14). There is no `maxConnectionsPerIp`, no global connection cap, and no per-topic subscription ceiling anywhere in the gRPC stack.

**Root cause:** The only guard is scoped to a single TCP connection. An attacker opens *C* connections and issues 5 `subscribeTopic` RPCs on each, yielding `5C` concurrent subscriptions with zero authentication or rate-limiting.

**Polling amplification path (POLL listener type):**

`PollingTopicListener.listen()` (lines 34–49) creates a fresh `PollingContext` per call and schedules an independent `Flux.defer(() -> poll(context)).repeatWhen(...)` loop:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .withFixedDelay(interval)   // default 500 ms
                .withScheduler(scheduler))
```

Each loop calls `topicMessageRepository.findByFilter(newFilter)` independently. With `5C` subscriptions, the DB receives `5C` queries every 500 ms = `10C` queries/second.

**Why existing checks fail:**

`TopicMessageServiceImpl` tracks `subscriberCount` (line 48) only as a Micrometer gauge — it enforces nothing. `topicExists()` (lines 94–106) only validates topic type, not subscription cardinality. The `SHARED_POLL` and `REDIS` listener types mitigate the DB amplification (shared flux / Redis pub-sub), but `POLL` type — which is a documented, supported configuration — has no such sharing.

### Impact Explanation
With `POLL` listener type, each additional connection contributes 5 independent DB polling loops at 500 ms intervals. Opening 100 connections (trivially achievable from a single host) produces 500 concurrent polling loops = 1,000 DB queries/second against a baseline of near-zero. This saturates the database connection pool, starves legitimate queries, and spikes CPU on both the mirror node JVM (Reactor scheduler threads) and the PostgreSQL backend. The 30% resource increase threshold is crossed with a small number of connections relative to what a single client can sustain.

### Likelihood Explanation
No authentication is required to call `subscribeTopic`. The gRPC port (default 5600) is publicly reachable. A single attacker machine can open hundreds of TCP connections and issue 5 streaming RPCs on each using any standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or a custom script). The attack is repeatable, requires no special knowledge beyond the public proto definition, and is not rate-limited at any layer in the gRPC module.

### Recommendation
1. **Add a per-IP (or global) connection limit** in `grpcServerConfigurer()` via `NettyServerBuilder.maxConnectionAge` / an interceptor that tracks connections per remote address and rejects beyond a threshold.
2. **Enforce a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount.get()` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Add a per-topic subscription limit** so a single topic cannot be subscribed to more than N times concurrently across all connections.
4. **Prefer `SHARED_POLL` or `REDIS`** as the enforced default and document `POLL` as unsafe for public-facing deployments.

```java
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
```
