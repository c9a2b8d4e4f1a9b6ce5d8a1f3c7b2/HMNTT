### Title
Request Timeout Bypass via Pre-Execution-Only `statementInspector` Check Allowing Multi-Query Overshoot

### Summary
The `statementInspector` in `HibernateConfiguration.java` enforces `requestTimeout` by checking elapsed time **before** each SQL statement starts, but never interrupts a query already in flight. An unprivileged caller can craft a contract call (especially `eth_estimateGas`) that triggers a sequence of DB queries where each query begins just before the timeout threshold, allowing the last query to run for an additional `statementTimeout` window beyond the intended deadline. For `LATEST`-block calls this also means successive queries execute against different PostgreSQL MVCC snapshots, producing reads across inconsistent database states.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/config/HibernateConfiguration.java`, lines 31–46:

```java
StatementInspector statementInspector() {
    long timeout = web3Properties.getRequestTimeout().toMillis(); // default 10 000 ms
    return sql -> {
        if (!ContractCallContext.isInitialized()) return sql;
        var startTime = ContractCallContext.get().getStartTime();
        long elapsed = System.currentTimeMillis() - startTime;
        if (elapsed >= timeout) {                          // ← only fires BEFORE a query
            throw new QueryTimeoutException(...);
        }
        return sql;                                        // ← query is now free to run
    };
}
```

`startTime` is set once at context creation (`ContractCallContext.java` line 31: `private final long startTime = System.currentTimeMillis()`).

**Root cause / failed assumption:** The guard is a *pre-flight* check. Once `return sql` is reached, the query executes with no further wall-clock enforcement from the application layer. The assumption that this lambda enforces a hard per-request ceiling is incorrect: it only prevents a *new* query from starting after the deadline; it cannot cancel one that is already running.

**Exploit flow:**

1. Attacker submits `eth_estimateGas` with a gas-heavy contract call. `ContractExecutionService.estimateGas()` first calls `callContract()` then enters `BinaryGasEstimator.search()`, which loops up to `maxGasEstimateRetriesCount = 20` iterations (`EvmProperties.java` line 69), each invoking `doProcessCall()` → `transactionExecutionService.execute()` → multiple Hibernate/JDBC queries.

2. Each iteration triggers several DB queries (account lookup, storage reads, record-file fetch, etc.). The inspector fires before each one.

3. The attacker crafts the contract to maximise per-query latency (e.g., large SLOAD fan-out, many token/account lookups). Queries run for duration approaching the DB-level `statement_timeout` (Helm chart sets `statement_timeout = 10000` ms for the `web3` user; application default `hiero.mirror.web3.db.statementTimeout = 3000` ms).

4. Concrete timeline with `requestTimeout = 10 000 ms`, effective DB `statement_timeout = 10 000 ms`:
   - t = 0 ms: inspector fires (elapsed = 0 < 10 000), Query-1 starts, runs 9 900 ms.
   - t = 9 900 ms: Query-1 completes. Inspector fires for Query-2 (elapsed = 9 900 < 10 000), Query-2 starts, runs 9 900 ms.
   - t = 19 800 ms: Query-2 completes. Inspector fires (elapsed = 19 800 ≥ 10 000) → exception thrown.
   - **Total wall time ≈ 19.8 s — nearly double the intended 10 s ceiling.**

5. For `LATEST`-block calls, Query-1 and Query-2 execute against different PostgreSQL MVCC snapshots (no explicit `BEGIN`/`SET TRANSACTION SNAPSHOT` wraps the full request). New importer-committed rows visible to Query-2 but not Query-1 produce a mixed-state view of transaction history.

**Why existing checks are insufficient:**

- The `statementInspector` is the *only* application-level wall-clock guard; there is no `Thread.interrupt()`, reactive timeout, or Servlet async timeout configured for the web3 service.
- The DB-level `statement_timeout` caps individual queries, not the aggregate request.
- The `BinaryGasEstimator` catches all exceptions from `safeCall()` silently (`catch (Exception ignored)`), so a timed-out iteration is swallowed and the loop continues, further extending total execution time.

### Impact Explanation
A single malicious `eth_estimateGas` request can hold a server thread for up to `requestTimeout + statementTimeout` (≈ 20 s with default Helm values) instead of the intended 10 s. Repeated at the allowed 500 RPS throttle, this exhausts the thread pool and causes denial of service. For `LATEST` calls the EVM simulation reads account balances, storage slots, and token state from different database snapshots within one logical call, producing results that do not correspond to any consistent ledger state — enabling an observer to reconstruct a view of transaction history that never existed atomically.

### Likelihood Explanation
No authentication is required; `eth_call` and `eth_estimateGas` are public endpoints. The attacker needs only a deployed contract with many SLOAD operations or token precompile calls to slow individual queries. The binary-search loop (up to 20 iterations) is a built-in amplifier. The attack is repeatable and scriptable.

### Recommendation
1. **Enforce a hard deadline on the running query**, not just at its start. Pass a `java.sql.Statement.setQueryTimeout()` or use a JDBC `socketTimeout` derived from the remaining budget (`timeout - elapsed`) before each statement executes, rather than aborting only at the next pre-flight check.
2. **Wrap the entire request in a single PostgreSQL transaction with `SET LOCAL statement_timeout`** equal to the remaining budget, so the DB itself cancels any overrunning query.
3. **Align `hiero.mirror.web3.db.statementTimeout` with `requestTimeout`** so no single query can consume more than the full request budget.
4. **Propagate a `Thread.interrupt()` or reactive cancellation signal** from a scheduled task that fires at `startTime + requestTimeout`, ensuring in-flight JDBC calls are interrupted.
5. In `BinaryGasEstimator.safeCall()`, re-throw `QueryTimeoutException` instead of swallowing it, so a timed-out iteration terminates the estimation loop immediately.

#    contract SlowRead { mapping(uint=>uint) s; function read() external view returns(uint r) {
#      for(uint i=0;i<1000;i++) r += s[i]; } }

# 2. Send eth_estimateGas targeting that function:

  }'
