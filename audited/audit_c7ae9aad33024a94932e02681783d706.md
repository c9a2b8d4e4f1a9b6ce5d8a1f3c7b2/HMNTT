### Title
Cache Amplification: Per-Contract Slot Accumulation Enables Large IN-Clause DB Queries via `findStorageBatch()`

### Summary
`ContractStateServiceImpl.findStorageBatch()` accumulates every distinct slot key seen for a given contract into a per-contract Caffeine cache (up to 1500 entries, `expireAfterAccess=5m`). On every invocation it issues a single `SELECT … WHERE slot IN (…)` query containing **all** currently cached slot keys for that contract. Because the `contractState` value cache expires after only 2 seconds (`expireAfterWrite=2s`), an unprivileged attacker can pre-fill the per-contract slot cache and then repeatedly trigger 1500-slot IN-clause queries with a single request every 2 seconds, per contract, across as many contracts as the outer `contractSlots` cache allows (up to 3000).

**Root cause and failed assumption:**

The design assumes that the `contractState` value cache (`expireAfterWrite=2s, maximumSize=25000`) will absorb repeated lookups, making the large IN-clause a one-time cost. The failed assumption is that the value cache's 2-second TTL is long enough to amortize the cost. In practice, the 2-second expiry guarantees that every 2 seconds **all** cached values for a contract become stale, and the very next `findStorage()` call for any slot of that contract bypasses the value cache (`contractStateCache.get(…)` returns null) and falls through to `findStorageBatch()`, which re-issues the full 1500-slot IN-clause.

**Exploit flow:**

1. **Phase 1 – cache warm-up (done once per contract, persists 5 minutes):**  
   Attacker calls `eth_call` (or `eth_estimateGas`) targeting contract `C_i` with 1500 distinct, previously-unseen slot keys. Each call misses the value cache, enters `findStorageBatch()`, appends the new key to the per-contract slot cache, and issues a growing IN-clause. After 1500 calls the per-contract slot cache is saturated at its `maximumSize=1500`.

2. **Phase 2 – sustained amplification (repeatable every 2 seconds):**  
   After the 2-second `contractState` TTL elapses, the attacker sends **one** request for any slot of `C_i`. `findStorage()` misses the value cache, calls `findStorageBatch()`, which reads all 1500 cached slot keys and issues:  
   `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (?, ?, … ×1500)`  
   This single HTTP request produces a 1500-row IN-clause query against the database.

3. **Scaling across contracts:**  
   The outer `contractSlots` cache (`expireAfterAccess=5m, maximumSize=3000`) allows up to 3000 contracts to maintain full slot caches simultaneously. The attacker can warm up N contracts and then, every 2 seconds, send N requests (one per contract), each triggering a 1500-slot IN-clause query. At the global 500 RPS throttle, the attacker can sustain up to 500 such amplified queries per second.

**Why existing checks are insufficient:**

- **Global RPS throttle (500/s, `ThrottleManagerImpl` line 38):** Counts HTTP requests, not DB query cost. One request → one token consumed, regardless of whether the resulting DB query has 1 or 1500 IN-clause parameters.
- **Gas throttle (`gasPerSecond`):** Measures EVM gas consumed during execution, not database I/O. Storage slot reads consume a fixed EVM gas amount regardless of how many slots are batched in the underlying SQL query.
- **`db.statementTimeout=3000ms`:** A 1500-slot IN-clause on an indexed table completes well within 3 seconds; this timeout provides no protection.
- **No per-IP or per-caller rate limiting:** The throttle is a single global bucket shared across all callers.

### Impact Explanation

A single attacker request in Phase 2 causes the database to evaluate a 1500-element IN-clause instead of a single point lookup — a query-cost amplification of up to **1500×**. With the 500 RPS global limit, the attacker can sustain 500 such queries per second, generating up to 750,000 effective slot lookups per second in the database. Even a modest attack (10 warmed-up contracts, 10 requests/2 s) produces 10 × 1500 = 15,000 slot lookups per burst, repeated every 2 seconds, with no authentication or special privilege required. This easily exceeds a 30% increase over a baseline where normal callers issue single-slot queries.

### Likelihood Explanation

The attack requires no credentials, no special network position, and no cryptographic material. The only prerequisite is sending ~1500 standard `eth_call` JSON-RPC requests per target contract to warm the cache — a one-time cost that persists for 5 minutes. Automation is trivial (a simple loop over 1500 distinct slot values). The 2-second re-trigger window is generous. The attack is fully repeatable and stateless from the attacker's perspective.

### Recommendation

1. **Cap the IN-clause size:** In `findStorageBatch()`, limit `cachedSlots` to a configurable maximum (e.g., 50–100) before passing it to `findStorageBatch`. Excess keys should be queried individually or dropped from the batch.
2. **Per-IP / per-caller rate limiting:** Introduce a per-source-IP token bucket in the throttle filter so a single caller cannot consume the entire global RPS budget with amplified queries.
3. **Increase `contractState` TTL or make it proportional to batch size:** A 2-second TTL guarantees high re-query frequency for large caches. Raising it (e.g., to 30–60 seconds) reduces the amplification window.
4. **Bound the per-contract slot cache growth:** Consider not accumulating slot keys from requests that miss the database (i.e., only cache keys that actually returned a value), preventing the cache from filling with attacker-supplied phantom keys.

### Proof of Concept

```python
import requests, itertools, time, threading

NODE_URL = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT_ADDR = "0x<target_contract>"   # any deployed contract

def slot_key(i):
    return "0x" + hex(i)[2:].zfill(64)

# Phase 1: warm up per-contract slot cache with 1500 distinct keys
for i in range(1500):
    requests.post(NODE_URL, json={
        "to": CONTRACT_ADDR,
        "data": "0x" + "00"*32,   # any calldata; EVM will call findStorage internally
        "block": "latest"
    })

# Phase 2: every 2 seconds, one request triggers a 1500-slot IN-clause query
while True:
    requests.post(NODE_URL, json={
        "to": CONTRACT_ADDR,
        "data": "0x" + "00"*32,
        "block": "latest"
    })
    time.sleep(2)   # contractState TTL = 2s; cache is stale, batch query fires again
```

Scale by repeating Phase 1 for N contracts and firing N requests every 2 seconds in Phase 2 to multiply database load proportionally.
