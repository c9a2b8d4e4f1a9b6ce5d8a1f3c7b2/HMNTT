### Title
Unauthenticated DB Amplification via Cold-Cache Block Queries in `updateOperationAccountAlias()`

### Summary
The `Block()` endpoint in `rosetta/app/services/block_service.go` issues one synchronous `GetAccountAlias` DB call per operation per transaction for every account ID not present in the LRU cache. An unprivileged attacker can repeatedly request a block containing `maxTransactionsInBlock` transactions each with many operations bearing unique account IDs, triggering O(maxTransactionsInBlock × maxOperationsPerTransaction) sequential DB round-trips per HTTP request. No authentication, payment, or special privilege is required to call the endpoint.

### Finding Description

**Exact code path:**

`Block()` (lines 47–74) caps transactions at `maxTransactionsInBlock` then calls `updateOperationAccountAlias()` (lines 104–129):

```go
// block_service.go:108-123
for _, transaction := range transactions {
    operations := transaction.Operations
    for index := range operations {
        accountId := operations[index].AccountId
        if cached, found = s.entityCache.Get(accountId.GetId()); !found {
            result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // DB call
            ...
            s.entityCache.Set(result.GetId(), result)
        }
        operations[index].AccountId = cached
    }
}
```

**Root cause:** Every operation whose account ID is absent from the LRU cache triggers a synchronous, unbatched `SELECT` against the database. There is no bulk-fetch path; the inner loop issues calls one at a time.

**Cache ineffectiveness under attack:** The LRU cache (`entityCache`) is bounded by `entityCacheConfig.MaxSize` (line 36). An attacker who alternates requests between two or more blocks whose combined unique account-ID sets exceed `MaxSize` continuously evicts entries, keeping the cache cold for the target block. Because the Rosetta `/block` endpoint requires no authentication (confirmed in `main.go` lines 80–87 — no auth middleware is registered for the block controller), this is trivially repeatable.

**Existing cap is insufficient:** The `maxTransactionsInBlock` guard (lines 61–67) limits the outer loop but does nothing to reduce the per-operation DB fan-out within those transactions.

### Impact Explanation

Each unauthenticated HTTP request to `/block` can consume up to `maxTransactionsInBlock × maxOperationsPerTransaction` database connections simultaneously held open. The database pool is finite (`Pool.MaxOpenConnections` in `config/types.go` line 79). Sustained requests exhaust the pool, causing all subsequent DB-dependent endpoints (block, account, construction) to queue or fail. The mirror node becomes unavailable to legitimate users. No funds are at risk; impact is availability degradation of the Rosetta service.

### Likelihood Explanation

The Hedera mainnet regularly produces blocks with hundreds of transactions, each containing multiple transfer operations across distinct accounts. An attacker needs only an HTTP client and knowledge of one such block hash/index — both publicly available from the Hedera mirror node REST API. The attack requires zero HBAR, zero credentials, and is fully repeatable. The only external mitigation is the Traefik ingress rate limit (`average: 10` req/s per host, `inFlightReq: 5`) present in the Helm chart (`charts/hedera-mirror-rosetta/values.yaml` lines 152–160), which is an optional deployment artifact and absent in bare deployments.

### Recommendation

1. **Batch the alias lookup**: Replace the per-operation loop with a single bulk query that fetches all aliases for all unique account IDs in the block in one DB round-trip.
2. **Pre-warm / increase cache size**: Ensure `entityCacheConfig.MaxSize` is large enough to hold all accounts seen in a typical block, preventing thrashing.
3. **Add application-level rate limiting**: Enforce per-IP or global concurrency limits inside the Go service itself, independent of the ingress layer.
4. **Deduplicate within a request**: Before entering the loop, collect the set of unique account IDs and resolve only those, avoiding redundant lookups for the same account appearing in multiple operations within one block.

func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

	return nil
```
