### Title
Future-Timestamp Mtime Bypass Silently Disables Pinger Liveness Probe

### Summary
`pinger/cmd/healthcheck/main.go` computes heartbeat age as `time.Since(fi.ModTime())` and checks `age > 2*time.Minute`. Because `time.Duration` is a signed `int64`, a future-dated mtime produces a large negative duration that always satisfies `negative < 2*time.Minute`, causing the liveness probe to unconditionally return exit 0 (healthy) regardless of whether the pinger process is actually running. Any principal with write access to `/tmp/alive` inside the container can trigger this permanently.

### Finding Description
**Exact location:** `pinger/cmd/healthcheck/main.go`, lines 34–38.

```go
age := time.Since(fi.ModTime())   // line 34 — returns negative if ModTime is in the future
if age > 2*time.Minute {          // line 35 — negative duration is NEVER > positive threshold
    fmt.Fprintln(os.Stderr, "heartbeat stale")
    os.Exit(1)
}
os.Exit(0)                        // always reached when mtime is future-dated
```

`time.Since(t)` is `time.Now().Sub(t)`. `time.Duration` is `int64` nanoseconds. If `fi.ModTime()` is set to, e.g., `2099-01-01`, the result is approximately `-2.3 × 10¹⁸` ns — a large negative value. The comparison `negativeValue > 2*time.Minute` is always `false`, so `os.Exit(0)` is always reached.

**Root cause:** The staleness check assumes `age` is non-negative (i.e., the file was written in the past). There is no guard for `age < 0`.

**Exploit flow:**
1. Attacker gains code execution inside the pinger container (e.g., via RCE in the pinger binary or its Hedera SDK dependency).
2. The pinger runs as UID 1000 and writes `/tmp/alive` with mode `0644` (`pinger/main.go` line 36). As the file owner, the attacker (same UID) can call `utimes("/tmp/alive", {tv_sec: 4070908800, tv_usec: 0})` to set mtime to year 2099.
3. The attacker then kills the pinger goroutine or the entire process.
4. Every subsequent Kubernetes liveness probe invocation (`/healthcheck live`) calls `os.Stat("/tmp/alive")` — the file still exists — computes a large negative `age`, and exits 0.
5. Kubernetes never triggers a container restart; the dead pinger is masked indefinitely.

**Why existing checks are insufficient:**
- The `os.Stat` error check (lines 28–32) only catches a missing or inaccessible file; it does not validate the mtime range.
- The `age > 2*time.Minute` check (line 35) has no lower bound; it is trivially bypassed by any negative value.
- The Kubernetes `securityContext` (`readOnlyRootFilesystem: true`, `runAsNonRoot: true`) does not prevent the file owner (UID 1000) from calling `utimes()` on a file they own in a writable tmpfs `/tmp`.

### Impact Explanation
The pinger's sole operational purpose is to submit periodic Hedera transactions and prove the network is reachable. Its liveness probe is the only automated mechanism that causes Kubernetes to restart a dead pinger. With the probe permanently bypassed, a crashed or hung pinger is never restarted, silently eliminating network health monitoring. No economic damage occurs directly, but the monitoring blind spot can mask real network degradation. Severity: Medium (operational integrity / griefing).

### Likelihood Explanation
The precondition is code execution inside the container as UID 1000, which is the same UID the pinger runs as. This is achievable via RCE in the pinger binary or its dependencies (Hedera Go SDK). Once achieved, the `utimes()` call is a single syscall requiring no elevated privileges — the attacker owns the file. The manipulation is persistent across probe cycles and survives until the container is manually restarted or the file is deleted. Repeatability is trivial.

### Recommendation
Replace the unsigned comparison with an explicit bounds check:

```go
age := time.Since(fi.ModTime())
if age < 0 || age > 2*time.Minute {
    fmt.Fprintln(os.Stderr, "heartbeat stale or clock skew detected")
    os.Exit(1)
}
```

Additionally, consider having the pinger write a monotonic sequence number or HMAC-protected token into `/tmp/alive` so the healthcheck can verify the content was produced by the legitimate pinger process, not an attacker-controlled write.

### Proof of Concept
```bash
# Inside the pinger container (UID 1000), after gaining code execution:

# 1. Set /tmp/alive mtime to year 2099
python3 -c "
import os, time, calendar
future = calendar.timegm(time.strptime('2099-01-01', '%Y-%m-%d'))
os.utime('/tmp/alive', (future, future))
print('mtime set to', future)
"

# 2. Kill the pinger heartbeat goroutine (or the whole process)
kill -9 $(pgrep pinger)

# 3. Verify the liveness probe now always exits 0
/healthcheck live; echo "exit code: $?"
# Output: exit code: 0   <-- probe passes even though pinger is dead

# 4. Confirm Kubernetes never restarts the container:
kubectl get pod -l app=hedera-mirror-pinger -w
# STATUS remains "Running" indefinitely despite dead pinger
```
