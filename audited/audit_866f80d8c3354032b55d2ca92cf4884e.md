### Title
MITM Injection of Crafted `nodesEnvelope` via Unauthenticated HTTP Client in `fetchMirrorNodeNetwork()`

### Summary
The `http.Client` constructed in `buildNetworkFromMirrorNodes()` carries no TLS certificate pinning and no transport-layer integrity guarantee. When `HIERO_MIRROR_PINGER_NETWORK=other`, the entire Hiero SDK node map is bootstrapped from a single unauthenticated HTTP/HTTPS response. A network-level attacker who can intercept that response can inject a crafted `nodesEnvelope` that routes every subsequent pinger transaction through a single attacker-controlled gRPC endpoint, distorting the apparent transaction ordering recorded by the mirror node.

### Finding Description
**Exact code path**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, line 46:
```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
``` [1](#0-0) 

This client is passed directly to `fetchMirrorNodeNetwork()` (line 53), which issues a plain `GET` with no custom `Transport`, no `tls.Config`, and no certificate pinning: [2](#0-1) 

The JSON body is decoded into `nodesEnvelope` and every `(host:port) → nodeAccountID` pair is inserted into the `network` map without any signature, HMAC, or out-of-band verification: [3](#0-2) 

That map is returned to `newClient()` and handed directly to `hiero.ClientForNetwork(netmap)`: [4](#0-3) 

Every subsequent call to `submitWithRetry()` uses this poisoned client for the lifetime of the process: [5](#0-4) 

**Root cause / failed assumption**

The code assumes the mirror node REST endpoint is trustworthy and reachable only over an authenticated channel. Neither assumption is enforced in code. The default example URL in `config.go` is `http://mirror-rest:5551` (plain HTTP), meaning TLS is not even present in the default deployment path: [6](#0-5) 

Even when HTTPS is used, Go's default `http.Client` validates only against the system CA store; without certificate pinning a trusted-CA MITM certificate is sufficient to intercept.

**Why existing checks are insufficient**

The only guards inside `fetchMirrorNodeNetwork()` are:
- Skip entries with empty `NodeAccountID` or unparseable account IDs (lines 103–109)
- Skip endpoints with empty host or zero port (lines 118–119)
- Reject an entirely empty result map (lines 127–129)

None of these checks verify *who* produced the response or whether the endpoints are legitimate Hedera/Hiero consensus nodes. [7](#0-6) 

### Impact Explanation
An attacker who successfully injects a crafted `nodesEnvelope` can map every node account ID (e.g., `0.0.3`, `0.0.4`, …) to a single endpoint they control. The Hiero SDK will then submit all `CryptoTransfer` transactions exclusively to that endpoint. The attacker-controlled node can:
- Selectively delay or reorder transaction forwarding to the real network, altering the consensus timestamp sequence visible in the mirror node history.
- Cause all pinger transactions to appear to originate from one node, breaking any monitoring logic that relies on distributed node participation.
- Silently drop transactions, making the pinger appear to succeed locally while no on-chain record is created.

Direct fund theft is not possible because transactions are signed with the operator key, but the integrity of the mirror node's transaction history as observed by the pinger is fully compromised.

### Likelihood Explanation
The attack requires a network-level MITM position between the pinger pod and the mirror REST endpoint. This is achievable without privileges in several realistic scenarios:
- **Plain HTTP deployment** (the documented default `http://mirror-rest:5551`): ARP spoofing or DNS poisoning on the same Kubernetes cluster network or LAN is sufficient — no TLS to break.
- **HTTPS deployment without pinning**: An attacker who can install or compromise a CA trusted by the container's system store (e.g., a rogue internal CA, a compromised Let's Encrypt intermediate) can present a valid certificate and intercept transparently.
- The attack is repeatable: the network map is fetched once at startup; a single successful injection persists for the entire process lifetime.

### Recommendation
1. **Enforce HTTPS and pin the certificate or public key**: supply a custom `tls.Config` with `VerifyPeerCertificate` or use a restricted CA pool (`x509.CertPool`) containing only the expected mirror node CA.
2. **Never use plain HTTP** for the mirror REST URL; reject `http://` schemes at config validation time.
3. **Cross-validate the returned node list** against a hard-coded or separately-fetched list of known-good node account IDs and IP ranges before calling `hiero.ClientForNetwork`.
4. **Re-fetch and re-validate** the node map periodically rather than trusting a single bootstrap response for the entire process lifetime.

### Proof of Concept
**Preconditions**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551` (plain HTTP, default example).
- Attacker has network access to the same segment as the pinger (e.g., same Kubernetes namespace or LAN).

**Steps**
1. Stand up a rogue HTTP server that responds to `GET /api/v1/network/nodes` with:
   ```json
   {
     "nodes": [
       {"node_account_id": "0.0.3", "service_endpoints": [{"ip_address_v4": "ATTACKER_IP", "port": 50211}]},
       {"node_account_id": "0.0.4", "service_endpoints": [{"ip_address_v4": "ATTACKER_IP", "port": 50211}]},
       {"node_account_id": "0.0.5", "service_endpoints": [{"ip_address_v4": "ATTACKER_IP", "port": 50211}]}
     ],
     "links": {"next": null}
   }
   ```
2. Perform ARP spoofing or DNS poisoning to redirect `mirror-rest:5551` to `ATTACKER_IP`.
3. Start (or restart) the pinger. `buildNetworkFromMirrorNodes()` fetches the crafted response; `fetchMirrorNodeNetwork()` builds a map where every node account ID resolves to `ATTACKER_IP:50211`.
4. `hiero.ClientForNetwork(netmap)` configures the SDK with this map. All `submitWithRetry()` calls now send `CryptoTransfer` transactions exclusively to the attacker's gRPC listener.
5. The attacker's listener can forward transactions out-of-order, drop them, or replay them at will, producing an altered transaction sequence in the mirror node history.

func fetchMirrorNodeNetwork(
	ctx context.Context,
	httpClient *http.Client,
	url string,
) (map[string]hiero.AccountID, bool, error) {
	req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
	if err != nil {
		return nil, false, err
	}

	resp, err := httpClient.Do(req)
	if err != nil {
		return nil, true, fmt.Errorf("GET %s failed: %w", url, err)
	}
	defer resp.Body.Close()

	if resp.StatusCode < 200 || resp.StatusCode >= 300 {
		retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
		return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
	}

	var payload nodesEnvelope
	if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
		return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
	}
```
