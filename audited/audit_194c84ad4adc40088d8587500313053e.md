### Title
Preparsed Document Cache Pollution via Case-Variant EVM Address Literals

### Summary
`decodeEvmAddress()` in `GraphQlUtils.java` does not normalize hex case before calling `Hex.decodeHex()`. While `Hex.decodeHex()` correctly decodes both cases to identical bytes (so entity lookups are functionally correct), the `CachedPreparsedDocumentProvider` uses the raw, un-normalized query string as its cache key. When an evmAddress is embedded as an inline literal in the GraphQL query string, case variations produce distinct cache keys, allowing any unprivileged caller to pollute the bounded 1,000-entry preparsed document cache and evict legitimate entries.

### Finding Description

**Code path:**

`decodeEvmAddress()` — `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 87–98:
```java
public static byte[] decodeEvmAddress(String evmAddress) {
    ...
    evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);  // strips "0x", no toLower()
    return Hex.decodeHex(evmAddress);                              // accepts A-F and a-f equally
}
```
No `.toLowerCase()` or `.toUpperCase()` is applied before or after stripping the prefix. `Hex.decodeHex()` (Apache Commons Codec) is case-insensitive at the byte-decoding level, so `0xABCDEF…` and `0xabcdef…` produce identical `byte[]` results and the downstream `getByEvmAddressAndType()` call in `EntityServiceImpl.java` (lines 34–41) behaves identically for both.

**Cache key root cause:**

`CachedPreparsedDocumentProvider` — `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```
The cache key is `executionInput.getQuery()` — the **raw, verbatim query string**. When a caller embeds the evmAddress as an inline literal (not a GraphQL variable), the full query string differs character-for-character for each case variant, producing a distinct cache entry per variant.

**Cache configuration:**

`CacheProperties` — `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15:
```
expireAfterWrite=1h, maximumSize=1000
```
The cache holds at most 1,000 entries for 1 hour. Caffeine evicts the least-recently-used entry when the limit is reached.

**Exploit flow:**
1. Attacker constructs a valid GraphQL query with the evmAddress as an inline literal, e.g.:
   `{ account(input: { evmAddress: "0xAABBCCDDEEFF..." }) { balance } }`
2. Attacker sends 1,000 requests, each varying the case of one or more hex characters in the address (e.g., `0xAABBCC…`, `0xaaBBCC…`, `0xAAbbCC…`, …). A 40-character hex address has up to 2^40 possible case permutations.
3. Each request is a cache miss; the GraphQL engine parses and validates the document, then stores it under the new key.
4. After 1,000 such requests the cache is full of attacker-controlled variants; any legitimate query that was previously cached is evicted and must be re-parsed on next access.

**Why existing checks are insufficient:**
- The `maximumSize=1000` cap bounds memory usage but does not prevent the attacker from continuously cycling through variants to keep legitimate entries evicted.
- There is no query-string normalization (e.g., lowercasing argument literals) before the cache lookup.
- The GraphQL parser limits (`maxCharacters=10000`, `maxTokens=1000`) do not prevent this because each individual query is well within limits.

### Impact Explanation
The preparsed document cache exists to avoid repeated parsing and validation of identical queries. Polluting it forces the server to re-parse every evicted legitimate query on its next execution, increasing CPU consumption proportionally to the eviction rate. The impact is bounded by the cache size (1,000 entries) and the 1-hour TTL, making this a low-throughput griefing primitive with no data leakage, no authentication bypass, and no financial loss to network participants. Severity is Low within the stated "griefing / no economic damage" scope.

### Likelihood Explanation
No authentication or special privilege is required. The attacker needs only HTTP access to the `/graphql/alpha` endpoint and the ability to generate case-variant query strings, which is trivially scriptable. The attack is repeatable indefinitely and requires no knowledge of valid on-chain addresses (any syntactically valid 40-hex-char string suffices to occupy a cache slot).

### Recommendation
1. **Normalize the query string before caching.** In `CachedPreparsedDocumentProvider`, apply a canonical form to the query key (e.g., lowercase all string literals, or use GraphQL's own AST printer to produce a canonical string) before the `cache.get()` call.
2. **Alternatively, normalize in `decodeEvmAddress()`.** Add `.toLowerCase(Locale.ROOT)` after stripping the `0x` prefix. This does not fix the cache key directly but is good hygiene and removes the semantic divergence.
3. **Prefer GraphQL variables over inline literals.** Document and enforce that callers pass evmAddress as a `$variable`, not an inline literal. Variable values are not part of the query string used as the cache key, so the same parsed document is reused regardless of the address value.

### Proof of Concept
```bash
# Fill the cache with 1000 case-variant entries for the same logical address
BASE="0xaabbccddeeff001122334455667788990011aabb"
for i in $(seq 1 1000); do
  # Flip one hex nibble per iteration to produce a unique cache key
  ADDR=$(echo $BASE | sed "s/a/A/$(( (i % 12) + 1 ))")
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: { evmAddress: \\\"${ADDR}\\\" }) { balance } }\"}" \
    > /dev/null
done
# Legitimate queries that were previously cached are now evicted and must be re-parsed
```
