### Title
Counter Exhaustion via Non-Matching Requests Disables REJECT Filter in `RequestProperties.test()`

### Summary
In `RequestProperties.test()`, the `AtomicLong counter` is unconditionally incremented via `counter.getAndIncrement()` **before** any filter matching occurs. This means every request — including those that do not match the configured filters — consumes the `limit` budget. An unprivileged attacker can flood the endpoint with non-matching requests to exhaust the counter, after which `test()` permanently returns `false` for all requests, silently disabling any `REJECT` filter and allowing previously-blocked transaction patterns to be gossiped freely.

### Finding Description

**Exact code path:**

`RequestProperties.java`, `test()`, line 46:
```java
if (rate == 0 || counter.getAndIncrement() >= limit) {
    return false;
}
```
`counter.getAndIncrement()` is called unconditionally. The filter-matching loop (lines 54–58) is only reached if the counter has not yet hit `limit`. Because the counter is incremented on every invocation of `test()` — regardless of whether the request matches any `RequestFilter` — non-matching requests silently consume the limit budget.

**Root cause / failed assumption:** The design assumes `limit` counts matched (and acted-upon) requests. In reality it counts all evaluated requests, including those that would have returned `false` at the filter-matching stage.

**Exploit flow:**
1. Admin configures a `RequestProperties` rule: `action=REJECT`, `limit=L` (any finite value), with filters targeting a specific malicious pattern (e.g., `field=TO`, `type=EQUALS`, `expression=0xDeadBeef...`).
2. Attacker sends `L` requests with a **different** `to` address. Each call to `throttle()` → `requestFilter.test()` increments the counter even though the filter never matches and no rejection occurs.
3. Once `counter >= L`, `test()` returns `false` for every subsequent request — including those targeting `0xDeadBeef...`.
4. The REJECT action in `ThrottleManagerImpl.action()` (line 69) is never reached for the now-unblocked pattern.

**Why existing checks are insufficient:**
- The global `rateLimitBucket` (default 500 req/s) and `gasLimitBucket` in `ThrottleManagerImpl.throttle()` (lines 38–42) throttle overall throughput but do not prevent counter exhaustion — they merely slow the attacker.
- The `rate` field (sampling probability) is irrelevant when `action=REJECT` because the sampling branch (line 50) is skipped for `THROTTLE` only; for `REJECT` the counter is still consumed on every call.
- There is no reset or decay mechanism on `counter`; once exhausted it stays exhausted for the lifetime of the bean.

### Impact Explanation
A `REJECT` filter is the operator's primary tool for blocking specific transaction patterns from being gossiped into the Hedera network. Exhausting its counter permanently disables it without any alert or log entry. Transactions matching the blocked pattern are then accepted and gossiped freely, defeating the security control entirely. Severity is **High** when `limit` is finite and the protected pattern is security-critical.

### Likelihood Explanation
No authentication or special privilege is required — any caller of the JSON-RPC endpoint can send requests. The attacker only needs to know (or guess) that a finite `limit` is configured, which is a reasonable assumption for any operator using `limit` as a circuit-breaker. With the default `requestsPerSecond=500`, a limit of 1000 can be exhausted in ~2 seconds. The attack is repeatable across restarts only if the bean is re-initialized, but within a single process lifetime the damage is permanent.

### Recommendation
Move `counter.getAndIncrement()` to **after** the filter-matching logic, so only requests that actually match the filters consume the limit:

```java
@Override
public boolean test(ContractCallRequest contractCallRequest) {
    if (rate == 0) {
        return false;
    }
    if (action != ActionType.THROTTLE && RandomUtils.secure().randomLong(0L, 100L) >= rate) {
        return false;
    }
    for (var filter : filters) {
        if (filter.test(contractCallRequest)) {
            return counter.getAndIncrement() < limit;  // only consume on match
        }
    }
    if (filters.isEmpty()) {
        return counter.getAndIncrement() < limit;
    }
    return false;
}
```
This ensures `limit` counts only matched-and-acted-upon requests, as the operator intends.

### Proof of Concept

**Preconditions:**
- `RequestProperties` configured with `action=REJECT`, `limit=100`, filter: `field=TO, type=EQUALS, expression=0x000...DEAD` (the blocked address).
- Global `requestsPerSecond` ≥ 100.

**Steps:**
1. Send 100 POST requests to `/api/v1/contracts/call` with `"to": "0x000...BEEF"` (non-matching address). Each request passes through (no rejection), but `counter` increments from 0 to 100.
2. Verify `counter` is now ≥ `limit` (100).
3. Send a POST request with `"to": "0x000...DEAD"` (the blocked address).
4. **Expected (secure):** `ThrottleException("Invalid request")` thrown.
5. **Actual (vulnerable):** Request passes — `test()` returns `false` at line 46 because `counter(100) >= limit(100)`, so `action()` is never called, and the transaction proceeds to gossip. [1](#0-0) [2](#0-1) [3](#0-2)
