### Title
Base32 Decoding Leniency in `decodeBase32()` Allows Multiple Alias Representations to Resolve to the Same Account, Corrupting Audit Trails

### Summary
`GraphQlUtils.decodeBase32()` uses Apache Commons Codec's `new Base32()` with its default case-insensitive, padding-tolerant decoder and performs no normalization of the input string before decoding. As a result, an unprivileged caller can supply any of dozens of syntactically distinct alias strings — differing only in letter case, presence/absence of `=` padding, or mixed-case combinations — that all decode to the identical byte array, causing `getByAliasAndType()` to return the same account under each variant. Because the raw (un-normalized) alias string is what appears in GraphQL query logs and any upstream audit infrastructure, the same account's transaction history becomes retrievable under an unbounded set of alias identities, making it impossible to reconstruct a reliable audit trail of which alias was actually used.

### Finding Description

**Exact code path:**

`GraphQlUtils.java` line 24 instantiates the decoder with no strictness options:
```java
private static final Base32 BASE32 = new Base32();   // case-insensitive, padding-tolerant
``` [1](#0-0) 

`decodeBase32()` at lines 83–85 passes the raw caller-supplied string directly to this decoder with zero pre-processing:
```java
public static byte[] decodeBase32(String base32) {
    return BASE32.decode(base32);   // no toUpperCase(), no padding normalization
}
``` [2](#0-1) 

`EntityServiceImpl.getByAliasAndType()` at line 30 feeds the decoded bytes directly to the repository lookup:
```java
return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
``` [3](#0-2) 

**Root cause — failed assumption:**
The code assumes that each account alias has exactly one valid string representation. This assumption is false. Apache Commons Codec's `Base32()`  constructor builds a decoder whose internal lookup table maps both `A`/`a`, `B`/`b`, … `Z`/`z` to the same 5-bit values, and whose `decode()` method silently accepts or strips `=` padding characters. Consequently:

| Alias string supplied by caller | Decoded bytes |
|---|---|
| `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ` | `[0x00 0x00 0x10 …]` |
| `aaaqeayeaudaocajcaireeyuculbogaz` | identical |
| `AaAqEaYeAuDaOcAjCaIrEeYuCuLbOgAz` | identical |
| `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ=` | identical |

The project's own test at `GraphQlUtilsTest.java` lines 76–77 explicitly confirms that both the padded and unpadded forms are accepted and produce the same output: [4](#0-3) 

**Why existing checks are insufficient:**
There is no `alias.toUpperCase()`, no `alias.replace("=","")`, no regex that restricts input to canonical uppercase-only Base32, and no re-encoding of the decoded bytes back to a canonical string for comparison or logging. The `AccountController` passes the raw `alias` string from the GraphQL input directly to `getByAliasAndType()` without any sanitization: [5](#0-4) 

### Impact Explanation
An attacker who knows (or can observe) a valid canonical alias for any account can construct an arbitrary number of syntactically distinct alias strings that all resolve to the same account. Each variant appears as a different identifier in GraphQL access logs, Spring Boot actuator traces, and any downstream SIEM or audit pipeline that records the raw query argument. This makes it impossible to determine from logs alone whether a single actor accessed the account repeatedly under different alias disguises, or whether multiple distinct aliases were legitimately used. The impact is specifically audit-trail integrity: transaction history for an account becomes attributable to an unbounded set of alias identities, undermining forensic investigations and compliance reporting. No privilege is required; any unauthenticated caller of the public GraphQL endpoint can exploit this.

### Likelihood Explanation
Exploitation requires only knowledge of one valid alias (obtainable from any prior legitimate query or from on-chain data) and the ability to send a GraphQL query — both trivially available to any external user. The transformation (lowercase the alias, or append `=`) is trivial to automate. The behavior is deterministic and repeatable across every deployment because it is a property of the library's default constructor, not of configuration or environment. No race condition, timing window, or special account state is required.

### Recommendation
Normalize the alias to its canonical form before decoding and before any logging occurs. The simplest correct fix is to uppercase and strip padding before passing to the decoder:

```java
public static byte[] decodeBase32(String base32) {
    if (base32 == null) return null;
    // Canonicalize: uppercase + strip padding so only one string maps to each byte array
    String canonical = base32.toUpperCase(Locale.ROOT).replace("=", "");
    return BASE32.decode(canonical);
}
```

Alternatively, replace `new Base32()` with a strict decoder (e.g., Guava's `BaseEncoding.base32().omitPadding()` as already used in `EntityIdAliasParameter.java` in the `rest-java` module) and add an input-validation regex that rejects lowercase or padded input at the GraphQL schema/controller layer before it reaches the service. [6](#0-5) 

### Proof of Concept

**Precondition:** Account with alias bytes `[0x00, 0x04, 0x10, …]` exists; its canonical Base32 alias is `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ`.

**Step 1 — canonical query (establishes baseline in audit log):**
```graphql
query { account(input: { alias: "AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ" }) { id } }
```
Audit log records: `alias=AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ`

**Step 2 — lowercase variant (same decoded bytes, different log entry):**
```graphql
query { account(input: { alias: "aaaqeayeaudaocajcaireeyuculbogaz" }) { id } }
```
Returns identical account. Audit log records: `alias=aaaqeayeaudaocajcaireeyuculbogaz` — appears as a different alias.

**Step 3 — padded variant:**
```graphql
query { account(input: { alias: "AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ=" }) { id } }
```
Returns identical account. Audit log records yet another distinct alias string.

**Result:** Three log entries, three apparently different aliases, one account — audit trail is fragmented and cannot be reliably correlated.
