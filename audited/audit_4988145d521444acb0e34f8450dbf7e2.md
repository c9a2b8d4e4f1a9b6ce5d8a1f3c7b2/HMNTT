### Title
Zombie gRPC Connection Resource Exhaustion via Missing maxConnectionAge and keepAlive in grpcServerConfigurer()

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only `maxConcurrentCallsPerConnection`, omitting `maxConnectionAge`, `keepAliveTime`, and `keepAliveTimeout`. An unprivileged attacker can open streaming gRPC connections (e.g., `subscribeTopic`), then silently drop TCP-level acknowledgments (via NAT timeout, firewall rule, or simply halting reads), causing the server to hold zombie connections indefinitely with no mechanism to detect or reclaim them. Multiplied across many connections, this exhausts server file descriptors, Netty channel memory, and thread-pool resources.

### Finding Description
**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35.

```java
// Lines 28-35
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // NO maxConnectionAge / maxConnectionAgeGrace
    // NO keepAliveTime / keepAliveTimeout
    // NO maxConnectionIdle
};
```

`NettyProperties` exposes only `maxConcurrentCallsPerConnection = 5` and nothing else:

```java
// NettyProperties.java line 14
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause:** `NettyServerBuilder` defaults for `maxConnectionAge`, `keepAliveTime`, and `keepAliveTimeout` are all `Long.MAX_VALUE` (effectively infinite). Without these, the server:
1. Never sends HTTP/2 PING frames to probe liveness (`keepAliveTime` not set).
2. Never closes a connection after a maximum age (`maxConnectionAge` not set).
3. Never detects that a peer has silently disappeared.

**Exploit flow:**
1. Attacker opens a long-lived streaming call (e.g., `ConsensusService/subscribeTopic`).
2. Attacker installs a firewall rule or lets a NAT mapping expire, silently dropping all TCP ACKs from the client side.
3. The server's TCP send buffer fills; the OS stops delivering data. HTTP/2 flow control (initial window ~65 KB) is exhausted per stream, so gRPC stops writing new frames — but the connection object, Netty `Channel`, associated `ChannelHandlerContext`, and all per-stream state remain allocated in the JVM heap indefinitely.
4. No PING is ever sent, so the server never detects the dead peer.
5. The attacker repeats from many source IPs or ports. Each zombie connection holds a file descriptor, Netty pipeline memory, and up to 5 call slots (per `maxConcurrentCallsPerConnection`).

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` limits streams *per connection*, not the number of connections. An attacker opens N connections, each with 5 zombie streams.
- `retriever.timeout = 60s` governs time between *message emissions from the database*, not write-side liveness to the client. It does not close a connection whose TCP window is full.
- There is no configured connection-level limit (`maxInboundConnections` is also absent from `grpcServerConfigurer()`).

### Impact Explanation
Each zombie connection holds: one OS file descriptor, one Netty `NioSocketChannel` (~several KB heap), per-stream gRPC call state, and a blocked slot in the shared `applicationTaskExecutor` thread pool if a handler thread is waiting for writability. With enough zombie connections, the server exhausts its file descriptor limit (default 1024–65536 depending on OS config), JVM heap, or executor threads, causing denial of service to all legitimate clients. The `subscribeTopic` streaming RPC is the highest-value target because it is designed to be long-lived, maximizing the window for NAT/firewall timeout to silently kill the TCP path.

### Likelihood Explanation
This requires no authentication or privilege. Any client that can reach port 5600 can open a gRPC stream. Silently killing the TCP path is trivially achievable: NAT gateways commonly time out idle TCP connections in 30–300 seconds; an attacker behind NAT simply opens the stream and does nothing. Alternatively, a `tc` (traffic control) or `iptables DROP` rule on the attacker's egress achieves the same effect. The attack is repeatable and automatable with a simple loop. The only practical rate-limiting factor is the server's accept backlog, which is not configured here either.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
import java.util.concurrent.TimeUnit;

return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // Detect dead peers: send PING after 30s idle, close if no response in 10s
    serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
    serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
    // Force-recycle connections after 1 hour (prevents indefinite zombie accumulation)
    serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);
    serverBuilder.maxConnectionAgeGrace(30, TimeUnit.SECONDS);
    // Close idle connections with no active calls after 5 minutes
    serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);
};
```

Expose these as configurable fields in `NettyProperties` (parallel to `maxConcurrentCallsPerConnection`) so operators can tune them per deployment. Also consider adding `maxInboundConnections()` to bound the total number of accepted connections.

### Proof of Concept
**Preconditions:** Network access to the gRPC port (5600). A valid topic ID (or `checkTopicExists = false`).

```bash
# Step 1: Open a long-lived streaming subscription
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
  <server>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
CLIENT_PID=$!

# Step 2: Immediately install a firewall rule to drop all ACKs from this client
# (simulates NAT timeout / silent TCP partition)
sudo iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# Step 3: Observe the server retains the connection indefinitely
# On the server, watch file descriptor count grow:
watch -n5 'ls /proc/$(pgrep -f grpc)/fd | wc -l'

# Step 4: Repeat steps 1-3 in a loop from multiple source ports
for i in $(seq 1 500); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <server>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
sudo iptables -A OUTPUT -p tcp --dport 5600 -j DROP
# Server accumulates 500+ zombie connections with no reclamation mechanism
```

Expected result: server FD count climbs monotonically; after sufficient connections, new legitimate clients receive connection refused or timeout errors. [1](#0-0) [2](#0-1) [3](#0-2)
