### Title
Unauthenticated EVM Address Lookup Causes Uncached DB Query Per Request, Enabling Resource Exhaustion

### Summary
The `getAccountCryptoAllowances()` handler unconditionally calls `EntityService.getEncodedId()` with the raw path parameter before any authentication or rate-limiting check. When the supplied value is a valid EVM address format that resolves to a non-existent entity, a database query is executed on every request because 404 responses are explicitly excluded from the response cache. With no rate limiting in the REST layer, an unprivileged attacker can flood the endpoint with random EVM addresses, forcing a DB round-trip per request and exhausting database connection pool capacity.

### Finding Description

**Exact code path:**

`rest/controllers/cryptoAllowanceController.js` line 77 — the very first operation in the handler is an unconditional `EntityService.getEncodedId()` call:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```

`rest/service/entityService.js` lines 120–124 — when the input matches `EntityId.isValidEntityId()` and the parsed `evmAddress` field is non-null, a DB query is issued:
```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()
    : await this.getEntityIdFromEvmAddress(entityId, requireResult);
```

`rest/service/entityService.js` lines 90–94 — the DB query runs unconditionally; if no row is found, `NotFoundError` (HTTP 404) is thrown:
```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
if (rows.length === 0) {
  if (requireResult) {
    throw new NotFoundError();
  }
```

**Root cause — failed assumption in the caching layer:**

`rest/middleware/responseCacheHandler.js` line 95 only stores responses that are 2xx or 304:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```
A 404 thrown by `NotFoundError` is never cached. The `EntityService` in the REST module (`rest/service/entityService.js`) has no application-level cache of its own (unlike the `importer` and `web3` modules which use `@Cacheable`). Therefore every request for a non-existent EVM address unconditionally executes `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` against the database.

**No rate limiting in the REST layer:**
A `grep` across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, and `throttle` returns zero matches. The `ThrottleManagerImpl` exists only in the `web3` module. The Traefik `rateLimit` middleware visible in `charts/hedera-mirror-rosetta/values.yaml` is for the Rosetta service, not the REST API.

### Impact Explanation
An attacker can send a continuous stream of GET requests to `/api/v1/accounts/<random-40-hex-chars>/allowances/crypto`. Each request:
1. Passes format validation (valid EVM address regex).
2. Executes a full DB query against the `entity` table.
3. Returns 404, which is never cached.
4. Forces the next identical or different request to repeat the DB query.

At sufficient request volume this saturates the database connection pool, increases query latency for all other mirror-node consumers, and can render the REST API unresponsive. Because the mirror node is the authoritative read path for Hedera state (balances, allowances, transactions), degrading it impacts any downstream system relying on it.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. Any internet-accessible deployment is reachable. The exploit is trivially scriptable with a single `curl` loop or any HTTP load tool. The attacker does not need to know valid EVM addresses — random 40-hex-char strings are sufficient and will always miss the DB, guaranteeing the worst-case (uncached) code path on every request.

### Recommendation
1. **Cache negative (404) results** for EVM address lookups at the `EntityService` level with a short TTL (e.g., 5–30 s), so repeated lookups for the same non-existent address do not hit the DB.
2. **Add rate limiting** to the REST API at the application or ingress level (analogous to the Traefik `rateLimit` middleware already used for the Rosetta service), keyed by source IP.
3. **Validate the EVM address against a known-format short-circuit** before issuing a DB query: if the address matches the long-zero form (`0x` + 24 zero bytes + 8-byte num), resolve it arithmetically without a DB round-trip (as already done for numeric IDs on line 123).

### Proof of Concept
```bash
# Flood with random non-existent EVM addresses — each triggers an uncached DB query
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-host>/api/v1/accounts/0x${ADDR}/allowances/crypto" &
done
wait
```
Each request hits `getEntityIdFromEvmAddress()` → `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` → 0 rows → 404 (not cached) → next request repeats the DB query. Monitor DB connection pool utilization and query latency to confirm exhaustion.
