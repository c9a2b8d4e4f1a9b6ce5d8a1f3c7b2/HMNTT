### Title
Unauthenticated High-Rate GraphQL Flooding via `getByEvmAddressAndType()` Causes Database Exhaustion

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl` issues an uncached, direct SQL query (`findByEvmAddress`) against the database for every unique EVM address submitted. No authentication, no rate limiting, and no result-level caching exist on this path, allowing any unauthenticated external user to flood the endpoint with unique EVM addresses and saturate the database connection pool, degrading availability for all users.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 51–54) routes any request carrying an `evmAddress` argument directly to `EntityServiceImpl.getByEvmAddressAndType()`: [1](#0-0) 

`getByEvmAddressAndType()` (lines 34–41) decodes the address and, for any non-zero-prefix address, calls `entityRepository.findByEvmAddress()`: [2](#0-1) 

`findByEvmAddress()` executes a raw, synchronous SQL query with no caching: [3](#0-2) 

**Root cause:** No rate limiting exists at any layer of the graphql module, and no `@Cacheable` annotation or result cache is applied to `findByEvmAddress()` or the service method. The only cache present is `CachedPreparsedDocumentProvider`, which caches the *parsed GraphQL document* (keyed on the query string), not query results: [4](#0-3) 

Since the GraphQL query string is the same across all requests (only the variable value changes), document parsing is cached — but every request still triggers a fresh DB round-trip.

**Existing checks reviewed and shown insufficient:**

- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` limit per-query structural complexity, not request rate: [5](#0-4) 
- Parser limits (`maxCharacters`, `maxTokens`) constrain individual query size, not volume: [6](#0-5) 
- No Spring Security configuration exists in the graphql module (only `LoggingFilter` is present): [7](#0-6) 
- A grep across the entire `graphql/` module for `rate.limit`, `RateLimit`, `throttle`, `bucket4j`, `resilience4j`, and `RateLimiter` returns zero matches.

### Impact Explanation

An attacker can continuously submit GraphQL `account` queries with unique, randomly generated 20-byte EVM addresses. Each request bypasses the document cache (same query string, different variable) and issues a full table scan or index lookup against the `entity` table. At sufficient volume this exhausts the database connection pool and/or overwhelms the DB server, causing query timeouts and degraded or unavailable service for all legitimate users. This is a pure availability impact with no authentication prerequisite.

### Likelihood Explanation

The attack requires zero privileges, zero knowledge of the system beyond the public GraphQL endpoint, and is trivially scriptable with any HTTP client. A single machine with a modest network connection can sustain thousands of requests per second. The attack is repeatable indefinitely and leaves no persistent state that would block re-execution.

### Recommendation

1. **Add application-level rate limiting** (e.g., Bucket4j or Spring Cloud Gateway rate filter) keyed on client IP, limiting requests to the `/graphql` endpoint.
2. **Cache `findByEvmAddress()` results** using Spring's `@Cacheable` on `EntityServiceImpl.getByEvmAddressAndType()` with a short TTL (e.g., 30–60 seconds), so repeated lookups for the same address hit the cache rather than the DB.
3. **Require authentication** for the GraphQL endpoint if the data is not intended to be fully public, using Spring Security to enforce it.
4. **Set a DB connection pool cap** and configure query timeouts as a last-resort backstop.

### Proof of Concept

```bash
# Generate and send 5000 unique-evmAddress queries in parallel
for i in $(seq 1 5000); do
  ADDR=$(openssl rand -hex 20)
  curl -s -X POST https://<mirror-node-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: {evmAddress: \\\"0x${ADDR}\\\"}) { balance } }\"}" &
done
wait
```

Each request carries a unique `evmAddress`, bypassing any document-level cache and forcing a fresh `SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE` execution. At scale, DB response times for all concurrent users will degrade measurably.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
