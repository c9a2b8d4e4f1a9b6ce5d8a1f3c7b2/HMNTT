### Title
Unbounded Concurrent DB Pollers via Unauthenticated `listen()` Subscriptions in `PollingTopicListener` (POLL Mode)

### Summary
When the listener type is configured as `POLL`, each call to `PollingTopicListener.listen()` creates a fully independent, perpetually-repeating database poller with no subscriber count limit, no per-IP connection cap, and no authentication requirement. An unauthenticated attacker can open an arbitrary number of TCP connections to the gRPC port and issue up to 5 `subscribeTopic` calls per connection (the only enforced limit), causing the database to serve the same large result set independently to every concurrent poller, multiplying DB load linearly with the number of subscriptions.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, lines 34–49:

```java
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);          // new independent state per call
    Duration interval = listenerProperties.getInterval();        // default 500 ms

    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)          // infinite repeat
                    .withFixedDelay(interval)
                    ...);
}
```

Every invocation allocates a fresh `PollingContext` and schedules an infinite polling loop that calls `topicMessageRepository.findByFilter(newFilter)` (line 61) every 500 ms. There is no shared state, no deduplication, and no cap on how many such loops may run simultaneously.

**Root cause:** The method is a cold Reactor publisher — each subscriber gets its own independent DB query loop. No global or per-topic subscriber counter is checked before creating the loop.

**Why existing checks are insufficient:**

| Check | Value | Why it fails |
|---|---|---|
| `maxConcurrentCallsPerConnection` | 5 | Per-connection only; no limit on number of connections from one IP |
| No per-IP connection limit | — | `GrpcConfiguration` sets only `maxConcurrentCallsPerConnection`, nothing else |
| No authentication | — | `ConsensusController.subscribeTopic()` requires no credentials |
| No gRPC-layer throttle | — | `ThrottleConfiguration`/`ThrottleManagerImpl` exist only in the `web3` module, not `grpc` |
| `maxPageSize = 5000` | 5000 rows | Caps rows per poll, but does not limit the number of concurrent pollers |

**Exploit flow:**

1. Attacker opens `C` TCP connections to port 5600 (no IP-level connection limit in code).
2. On each connection, attacker issues 5 `subscribeTopic` RPCs for the same high-traffic `topicId` with `startTime` set to a historical timestamp (maximising result set size).
3. Each RPC triggers `PollingTopicListener.listen()` → `poll()` → `topicMessageRepository.findByFilter()` every 500 ms.
4. Total concurrent DB queries = `C × 5`, each returning up to 5 000 rows, every 500 ms.
5. With `C = 200` connections: 1 000 independent DB queries/500 ms = 2 000 queries/second, each scanning the same large result set.

### Impact Explanation

With `POLL` listener type active, the database connection pool (`hikaricp`) is exhausted first (alerting threshold is 75% utilisation per the Helm chart alert rule). Beyond that, the PostgreSQL server itself is overwhelmed with identical sequential scans on `topic_message`, causing query latency to spike for all users of the mirror node — including legitimate subscribers, the REST API, and the importer. This constitutes a non-network-based DoS against the database tier, which is a shared resource across all mirror node components.

### Likelihood Explanation

No credentials, API keys, or special network access are required. The gRPC port (5600) is publicly exposed. The attack is trivially scriptable: a single machine with a standard gRPC client library can open hundreds of connections and issue thousands of subscriptions in seconds. The attack is repeatable and persistent (subscriptions run indefinitely via `RepeatSpec.times(Long.MAX_VALUE)`). The only practical friction is the `maxConcurrentCallsPerConnection = 5` limit, which is bypassed by opening more connections.

### Recommendation

1. **Enforce a global concurrent-subscription limit** in `TopicMessageServiceImpl` (which already tracks `subscriberCount` via an `AtomicLong`) — reject new subscriptions when the count exceeds a configurable threshold.
2. **Add per-IP subscription rate limiting** via a gRPC `ServerInterceptor` (analogous to the `ThrottleManagerImpl` in the `web3` module) that caps new `subscribeTopic` calls per source IP per second.
3. **Prefer `SHARED_POLL` or `REDIS` listener types** in production — both share a single DB polling loop regardless of subscriber count, eliminating the linear DB load multiplication.
4. **Set `maxInboundConnections`** on the `NettyServerBuilder` in `GrpcConfiguration` to cap total TCP connections server-wide.

### Proof of Concept

```bash
# Open 200 connections, 5 subscriptions each = 1000 concurrent DB pollers
# Requires grpcurl (https://github.com/fullstorydev/grpcurl)

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "consensusStartTime": {"seconds": 0}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Monitor DB connection pool exhaustion:
# hikaricp_connections_active{application="grpc"} / hikaricp_connections_max{application="grpc"} → approaches 1.0
# PostgreSQL: SELECT count(*) FROM pg_stat_activity WHERE query LIKE '%topic_message%';
```

Each background process holds an open streaming RPC that independently queries the DB every 500 ms. With a high-traffic topic, each query returns up to 5 000 rows, saturating the connection pool and CPU on the database server.
