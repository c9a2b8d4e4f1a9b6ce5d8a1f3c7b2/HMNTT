### Title
Unbounded In-Memory Response Buffering via `ShallowEtagHeaderFilter` with No Rate Limiting on `/api/*`

### Summary
`etagFilter()` in `RestJavaConfiguration.java` registers a `ShallowEtagHeaderFilter` over all `/api/*` endpoints. This filter unconditionally buffers the entire serialized response body in JVM heap memory to compute an MD5 ETag hash. The rest-java module has no rate limiting (unlike the `web3` module), so any unauthenticated attacker can flood concurrent requests — particularly to unbounded endpoints like `/api/v1/network/fees` — causing simultaneous heap accumulation, GC pressure, and latency degradation for all users.

### Finding Description

**Exact code path:**

`RestJavaConfiguration.java`, `etagFilter()`, lines 42–46:
```java
@Bean
FilterRegistrationBean<ShallowEtagHeaderFilter> etagFilter() {
    final var filterRegistrationBean = new FilterRegistrationBean<>(new ShallowEtagHeaderFilter());
    filterRegistrationBean.addUrlPatterns("/api/*");
    return filterRegistrationBean;
}
``` [1](#0-0) 

Spring's `ShallowEtagHeaderFilter` wraps the downstream `HttpServletResponse` in a `ContentCachingResponseWrapper`. Every byte written by the controller is intercepted and accumulated in a `ByteArrayOutputStream` in heap memory. Only after the entire response is written does the filter compute the MD5 hash, set the `ETag` header, and flush the buffer to the real response. There is no `writeLimit` or streaming path — the full response must be buffered before any byte is sent to the client.

**Root cause — failed assumption:** The filter assumes that either (a) response sizes are small enough that concurrent buffering is safe, or (b) an upstream rate limiter bounds concurrency. Neither holds. The rest-java module has **no rate limiting** at all — the `ThrottleManagerImpl`/`ThrottleProperties` throttling exists exclusively in the `web3` module: [2](#0-1) 

The `RestJavaProperties` configuration only manages response headers, not request rates: [3](#0-2) 

The documentation confirms no rate-limiting property exists for `hiero.mirror.restJava.*`: [4](#0-3) 

**Unbounded endpoint — `/api/v1/network/fees`:** This endpoint returns the full Hedera fee schedule (a protobuf-derived structure covering all transaction types) with no pagination parameter. On mainnet the serialized JSON fee schedule is hundreds of kilobytes. Each concurrent request causes the filter to hold one full copy in heap: [5](#0-4) 

Paginated endpoints are bounded at `MAX_LIMIT = 100` items, but even those responses (e.g., 100 allowances or hook storage entries with binary data) are non-trivial, and the multiplier is the number of concurrent requests: [6](#0-5) 

### Impact Explanation

Each concurrent request to `/api/v1/network/fees` holds a full fee-schedule response buffer in heap until the filter finishes. With Tomcat's default thread pool (200 threads), an attacker saturating the pool forces 200 simultaneous heap buffers. At ~300 KB per fee-schedule response, that is ~60 MB of live heap objects that cannot be GC'd until each request completes. Repeated waves of requests cause sustained GC pressure, stop-the-world pauses, and latency spikes visible to all concurrent legitimate users. In a memory-constrained deployment this can escalate to `OutOfMemoryError`. The impact is service degradation (griefing) with no economic damage to the network — matching the stated scope classification.

### Likelihood Explanation

No authentication, API key, or rate limit is required. Any external actor with HTTP access can open many concurrent connections. The attack is trivially scriptable (`ab`, `wrk`, `curl --parallel`), requires no special knowledge of the protocol, and is repeatable indefinitely. The `/api/v1/network/fees` endpoint is publicly documented and returns a deterministically large response.

### Recommendation

1. **Add a rate limiter to rest-java** analogous to the `web3` module's `ThrottleManagerImpl` — a `OncePerRequestFilter` using Bucket4j that limits requests per second per IP or globally before the ETag filter runs.
2. **Set a `writeLimit` on `ShallowEtagHeaderFilter`** (Spring 5.3+ supports `ShallowEtagHeaderFilter#setWriteWeakETag` and the underlying `ContentCachingResponseWrapper` accepts a `contentCacheLimit`). Configure a hard cap (e.g., 512 KB) so that responses exceeding the limit bypass ETag computation and are streamed directly.
3. **Exempt large/unbounded endpoints** (e.g., `/api/v1/network/fees`) from the ETag filter by narrowing `addUrlPatterns` or adding a `filterRegistrationBean.addInitParameter("excludedUrlPatterns", ...)`.

### Proof of Concept

```bash
# Flood /api/v1/network/fees with 200 concurrent requests, 1000 total
ab -n 1000 -c 200 http://<host>/api/v1/network/fees

# Or with wrk for sustained load
wrk -t 10 -c 200 -d 30s http://<host>/api/v1/network/fees
```

Expected result: JVM heap usage climbs sharply during the burst (observable via `/actuator/metrics/jvm.memory.used`), GC pause frequency increases, and p99 latency for concurrent legitimate requests degrades measurably. With a small heap (`-Xmx256m`) the process may throw `OutOfMemoryError`.

    /*
     * Post process the configured response headers. All header names are treated case insensitively, and, for each path,
     * the default headers are first inherited and their values possibly overridden.
     */
    @PostConstruct
    void mergeHeaders() {
        for (var pathHeaders : response.headers.path.entrySet()) {
            var mergedHeaders = Stream.concat(
                            response.headers.defaults.entrySet().stream(), pathHeaders.getValue().entrySet().stream())
                    .collect(Collectors.toMap(
                            Entry::getKey,
                            Entry::getValue,
                            (v1, v2) -> v2,
                            () -> new TreeMap<>(String.CASE_INSENSITIVE_ORDER)));

            pathHeaders.setValue(mergedHeaders);
        }
    }
