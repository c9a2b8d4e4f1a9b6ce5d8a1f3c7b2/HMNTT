### Title
Unauthenticated Thundering Herd via Unbounded `getByAliasAndType()` Flooding During DB Partition Recovery

### Summary
The GraphQL module's `getByAliasAndType()` path issues a direct, uncached native SQL query to the database on every invocation with no rate limiting, no circuit breaker, and no per-IP throttling. Any unauthenticated external user can flood the `/graphql/alpha` endpoint with alias-lookup queries. During a network partition, these requests accumulate in the application's thread/connection queue; when the DB comes back online, all queued requests simultaneously attempt to execute against the recovering database, overwhelming it before it can stabilize.

### Finding Description

**Exact code path:**

`AccountController.account()` (line 48) calls `entityService.getByAliasAndType(alias, EntityType.ACCOUNT)`. [1](#0-0) 

`EntityServiceImpl.getByAliasAndType()` (line 30) directly calls `entityRepository.findByAlias(decodeBase32(alias))` — no cache, no circuit breaker, no retry backoff. [2](#0-1) 

`EntityRepository.findByAlias()` executes a native SQL query on every call with no result caching. [3](#0-2) 

**Root cause — failed assumptions:**

1. **No rate limiting in the GraphQL module.** The `ThrottleConfiguration` / `ThrottleManagerImpl` (bucket4j-based) exist exclusively in the `web3` module. The `graphql` module has no equivalent. [4](#0-3) 

2. **`GraphQlConfiguration` only enforces structural query limits** (max complexity 200, max depth 10, parser token limits). None of these prevent high-frequency repeated requests. [5](#0-4) 

3. **`LoggingFilter` is the only servlet filter** in the graphql module — it only logs, applies no throttle. [6](#0-5) 

4. **HikariCP pool is the only back-pressure mechanism.** During a partition all pool connections are invalidated. Pending application threads queue against the pool. On recovery, HikariCP re-establishes up to `maximumPoolSize` connections simultaneously and immediately drains the accumulated queue, producing a burst of concurrent queries against a DB that has just restarted. [7](#0-6) 

**Exploit flow:**

1. Attacker opens many concurrent HTTP connections (or uses a single async client) and continuously POSTs `{ account(input: { alias: "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA" }) { balance } }` to `/graphql/alpha`.
2. During normal operation, each request hits the DB. During a partition, requests begin queuing in the Tomcat thread pool and HikariCP's connection-wait queue.
3. The attacker sustains the flood throughout the partition window (seconds to minutes).
4. When the DB recovers, HikariCP immediately opens `maximumPoolSize` connections and begins executing the backlog of queries in rapid succession. The DB must simultaneously handle TCP reconnection overhead, authentication, and a burst of `SELECT * FROM entity WHERE alias = ?` queries — all before it has warmed its buffer pool or stabilized replication.

### Impact Explanation

The DB server is hit with a coordinated burst of queries at the exact moment it is most vulnerable (cold caches, replication catch-up, connection setup overhead). This can cause the DB to fail again immediately after recovery (secondary outage), dramatically extending total downtime. Because the GraphQL endpoint is public and unauthenticated, the attacker requires no credentials and can sustain the attack indefinitely at negligible cost.

### Likelihood Explanation

The endpoint is publicly documented and reachable without authentication. A single attacker with a basic HTTP flood tool (e.g., `wrk`, `hey`, or a simple async script) can generate thousands of requests per second. Network partitions in cloud-hosted PostgreSQL deployments (failover, maintenance, AZ outage) are routine operational events, making the recovery window a predictable and repeatable attack surface.

### Recommendation

1. **Add rate limiting to the GraphQL module** — port the existing bucket4j `ThrottleConfiguration` pattern from `web3` to `graphql`, enforcing a global requests-per-second cap and ideally a per-IP cap via a servlet filter or `WebGraphQlInterceptor`.
2. **Add a circuit breaker** (e.g., Resilience4j `@CircuitBreaker`) on `EntityServiceImpl.getByAliasAndType()` so that when the DB is unreachable, requests fail-fast instead of queuing.
3. **Add result caching** on `getByAliasAndType()` (a short TTL Caffeine cache, similar to the existing `hiero.mirror.graphql.cache.query` pattern) to reduce DB load during recovery.
4. **Tune HikariCP `connectionTimeout` and `initializationFailTimeout`** to shed load quickly during partition rather than accumulating a large queue.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {

    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
