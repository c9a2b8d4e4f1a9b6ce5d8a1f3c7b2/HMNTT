### Title
Oversized `from`/`to` String Allocation Before Bean Validation Bypasses Rate Limiting — Memory Exhaustion via `/api/v1/contracts/call`

### Summary
In `web3/src/main/java/org/hiero/mirror/web3/config/JacksonConfiguration.java`, `jacksonCustomizer()` sets `maxStringLength` to `maxSize` (~257 KiB) globally for all JSON string fields. The `from` and `to` fields in `ContractCallRequest` are semantically constrained to 42 characters (Ethereum address), but Jackson allocates the full `maxSize` string in memory before Bean Validation (`@Hex(maxLength=40)`) rejects it. Because rate limiting runs inside the controller method body — after `@RequestBody @Valid` processing — requests with oversized `from`/`to` values bypass the rate limiter entirely, allowing an unauthenticated attacker to flood the server with large allocations.

### Finding Description

**Exact code path:**

`JacksonConfiguration.java` line 22–29:
```java
final int maxSize = jumboConfig.ethereumMaxCallDataSize() * 2 + 1024;
// ethereumMaxCallDataSize() default = 131072 (128 KiB)
// maxSize = 131072 * 2 + 1024 = 263168 bytes (~257 KiB)
...
.maxStringLength(maxSize)   // applied to ALL string fields
```

`ContractCallRequest.java` lines 33–43:
```java
@Hex(minLength = ADDRESS_LENGTH, maxLength = ADDRESS_LENGTH)  // maxLength = 40
private String from;

@Hex(minLength = ADDRESS_LENGTH, maxLength = ADDRESS_LENGTH, allowEmpty = true)
private String to;
```

`ContractController.java` line 38–40:
```java
ContractCallResponse call(@RequestBody @Valid ContractCallRequest request, ...) {
    try {
        throttleManager.throttle(request);   // ← runs AFTER @RequestBody @Valid
```

**Root cause:** `maxStringLength(maxSize)` is a single global parser-level limit applied to every JSON string token. It is sized for the `data` field (call data up to 128 KiB hex-encoded), but it also governs `from`, `to`, `block`, and every other string field. The application-level `@Hex(maxLength=40)` constraint on `from`/`to` runs only after Jackson has fully materialized the string into a Java heap object.

**Exploit flow:**
1. Jackson reads the request body and encounters `"from": "<N chars>"` where N ≤ 263168.
2. Jackson allocates a `String` of N characters on the heap.
3. Spring's `@Valid` runs `HexValidator`, which rejects the value (regex `^(0x)?[0-9a-fA-F]{40,40}$` fails).
4. Spring returns HTTP 400 — the controller method body never executes.
5. `throttleManager.throttle()` is never called; the rate-limit bucket is never consumed.
6. The attacker repeats at maximum HTTP throughput.

**Why `maxDocumentLength` does not save it:**
A document `{"from":"<262900 chars>","to":"0x..."}` is ≈ 262960 bytes, which is below `maxDocumentLength(263168)`. The document-length check passes; the full string is allocated.

**Why `maxTokenCount(100)` does not save it:**
The attack uses a single oversized string token — well within the 100-token budget.

### Impact Explanation
Each malicious request causes Jackson to allocate up to ~257 KiB on the JVM heap before the request is rejected. With no rate-limit gate on invalid requests, an attacker can sustain a high request rate. The Helm chart sets a memory limit of 2048 MiB for the web3 pod. At ~257 KiB per concurrent in-flight request, ~8000 simultaneous requests would exhaust the heap, triggering OOM or severe GC pauses and degrading or denying service to legitimate users. The severity matches the stated scope: griefing/DoS with no direct economic damage to network participants.

### Likelihood Explanation
The endpoint `POST /api/v1/contracts/call` is publicly accessible with no authentication. The attack requires only a standard HTTP client and a trivially constructed JSON body. It is easily scripted and repeatable. The attacker does not need any on-chain funds, keys, or privileged access. The only practical barrier is the bandwidth needed to send ~257 KiB per request, but even at 10% of `maxSize` (~26 KiB per request) the allocation-to-rejection ratio remains highly asymmetric.

### Recommendation
Apply a per-field string length cap at the Jackson layer for address fields, or — more practically — add a servlet filter / Spring `HandlerInterceptor` that enforces the rate limit **before** `@RequestBody` deserialization occurs (e.g., using a `ContentCachingRequestWrapper` size check or a dedicated `OncePerRequestFilter` that consumes a rate-limit token before the body is read).

A targeted fix is to add a separate, small `maxStringLength` for the address-only endpoints, or to validate string length at the HTTP body level (e.g., `server.tomcat.max-http-form-post-size` set to a few KiB for this endpoint) so that oversized bodies are rejected before Jackson allocates anything.

### Proof of Concept
```bash
# Generate a ~257 KiB string for the "from" field (within maxStringLength)
LARGE=$(python3 -c "print('0' * 263100)")

# Send the request — no authentication required
curl -s -o /dev/null -w "%{http_code}" \
  -X POST http://<web3-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d "{\"from\":\"${LARGE}\",\"to\":\"0x00000000000000000000000000000000000004e2\"}"
# Returns 400 (Bean Validation rejects), but ~257 KiB was allocated on the heap

# Flood with parallel requests — rate limiter is never triggered
for i in $(seq 1 500); do
  curl -s -o /dev/null \
    -X POST http://<web3-host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"from\":\"${LARGE}\",\"to\":\"0x00000000000000000000000000000000000004e2\"}" &
done
wait
# Observe heap exhaustion / GC pressure / OOM in web3 pod logs
```
