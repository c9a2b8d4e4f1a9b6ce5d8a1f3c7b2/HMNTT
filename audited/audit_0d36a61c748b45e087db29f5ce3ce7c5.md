### Title
Unauthenticated `/health/readiness` Endpoint Triggers Unthrottled 2x Internal HTTP Request Amplification

### Summary
The `checkNetworkStatus()` function registered as a readiness health check makes two sequential internal HTTP calls (`/network/list` then `/network/status`) back to the same server on every invocation of `GET /health/readiness`. The `/health/readiness` endpoint is excluded from the Traefik rate-limiting and in-flight-request middleware that protects `/rosetta/*` paths, meaning any unprivileged external caller can flood this endpoint with no throttling, tripling the effective server load (1 external + 2 internal requests per call).

### Finding Description
**Code location:** `rosetta/app/middleware/health.go`, `checkNetworkStatus()`, lines 80–101.

```
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
    serverUrl := fmt.Sprintf("http://localhost:%d", port)          // line 81
    cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
    rosettaClient := client.NewAPIClient(cfg)

    return func(ctx context.Context) (checkErr error) {
        networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, ...)   // line 86 → POST /network/list
        ...
        _, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, ...)            // line 93 → POST /network/status
        ...
    }
}
```

Each call to `GET /health/readiness` executes this closure synchronously, issuing two full HTTP round-trips to `http://localhost:{port}` — the same process and port that is serving the external request. Those internal requests are handled by the same `net/http` goroutine pool and consume the same DB connection pool (via `NetworkStatus`).

**Why existing checks fail:**

The Traefik middleware chain (`inFlightReq: amount: 5`, `rateLimit: average: 10`) is attached only to the ingress paths explicitly listed in `charts/hedera-mirror-rosetta/values.yaml` lines 125–133:
```
/rosetta/account, /rosetta/block, /rosetta/call, /rosetta/construction,
/rosetta/events, /rosetta/mempool, /rosetta/network, /rosetta/search
```
`/health/readiness` is absent from this list. The GCP gateway health-check policy also hits port 5700 directly (bypassing Traefik entirely). There is no application-level rate limiting, concurrency cap, or authentication on the health endpoint.

The `Timeout: time.Second * 10` on the health check config (line 46) only bounds a single check's duration; it does not limit concurrency or call rate.

### Impact Explanation
An attacker sending N concurrent requests to `/health/readiness` causes 3N concurrent requests to be processed by the server (N external + N to `/network/list` + N to `/network/status`). The internal requests contend for:
- Go HTTP server goroutines (unbounded by default)
- Database connection pool slots (used by `NetworkStatus` → block/transaction queries)
- File descriptors

Sustained flooding degrades or denies service to legitimate Rosetta API consumers. Because the Rosetta node is part of the Hedera network infrastructure, availability impact extends to any client relying on this node for transaction submission or balance queries.

### Likelihood Explanation
No privileges, credentials, or special network position are required. The endpoint is reachable wherever the application port is exposed (Kubernetes `Service`, GCP load balancer health-check path, or any direct pod access). The attack is trivially scriptable with `curl` or `ab` and is fully repeatable. The only practical barrier is network reachability to port 5700.

### Recommendation
1. **Add application-level concurrency control** on the health handler, e.g. a semaphore limiting simultaneous `checkNetworkStatus` executions to 1–2.
2. **Cache the last health-check result** for a short TTL (e.g. 5 seconds) so that burst requests do not each trigger fresh internal calls.
3. **Add `/health/readiness` to the Traefik ingress paths** (or a separate ingress rule) so the existing `inFlightReq` and `rateLimit` middleware applies to it.
4. **Alternatively**, replace the self-HTTP-call pattern with a direct in-process function call to the `NetworkAPIService` instead of making outbound HTTP requests to `localhost`.

### Proof of Concept
```bash
# Flood /health/readiness with 100 concurrent requests, 1000 total
ab -n 1000 -c 100 http://<rosetta-host>:<port>/health/readiness

# Each of the 100 concurrent external requests spawns 2 internal requests,
# resulting in up to 300 concurrent in-flight requests on the server.
# Monitor with: watch -n1 'ss -s | grep estab'
```
