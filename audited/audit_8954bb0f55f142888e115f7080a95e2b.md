### Title
Unauthenticated Historical Timestamp Query Triggers Unbounded Sort+Dedup on `token_balance` Table, Enabling DB Sort Memory Exhaustion via Concurrent Requests

### Summary
The `getEntityBalanceQuery()` function in `rest/accounts.js` generates a `DISTINCT ON (token_id) ... ORDER BY token_id ASC, consensus_timestamp DESC` subquery against the `token_balance` table when a historical timestamp is supplied. Any unauthenticated user can trigger this expensive path by calling `GET /api/v1/accounts/:id?timestamp=lte:X`. Because the REST API has no rate limiting and the `LIMIT` clause is applied only after the full sort+dedup, flooding with concurrent requests can exhaust PostgreSQL `work_mem` and degrade or crash the database.

### Finding Description
**Exact code location**: `rest/accounts.js`, `getEntityBalanceQuery()`, lines 170–180; triggered from `getOneAccount()`, lines 428–465.

**Root cause**: When `getOneAccount()` receives a `timestamp` filter and `getAccountBalanceTimestampRange()` returns a valid snapshot (`upper !== undefined`), it sets `accountBalanceQuery.query` to a non-empty string (line 446). This causes `needHistoricalBalanceInfo = true` (line 164), which selects the branch at line 170–180:

```javascript
select distinct on (token_id) token_id, balance
from token_balance
where ${tokenBalanceQuery.query}          // account_id = $1 AND consensus_timestamp >= $2 AND <= $3
order by token_id ${order}, consensus_timestamp desc
limit ${tokenBalanceQuery.limit}          // default 1000
```

PostgreSQL's `DISTINCT ON` requires the result set to be sorted by the `DISTINCT ON` key first. Because the `ORDER BY` is `(token_id ASC, consensus_timestamp DESC)` — a mixed-direction sort — it cannot be satisfied by a plain index scan on the natural `(account_id, token_id, consensus_timestamp ASC)` primary key. PostgreSQL must materialize and sort **all** rows matching `account_id = $1 AND consensus_timestamp IN [lower, upper]` before it can apply the `LIMIT 1000`. The `LIMIT` provides no early-termination benefit.

**Exploit flow**:
1. Attacker identifies a high-token-count account (e.g., a DEX or exchange account with thousands of associated tokens — publicly visible on-chain).
2. Attacker issues `GET /api/v1/accounts/0.0.X?timestamp=lte:<recent_valid_timestamp>` with no authentication.
3. `getOneAccount()` calls `getAccountBalanceTimestampRange()`, finds a valid snapshot, sets `accountBalanceQuery.query` non-empty.
4. `getEntityBalanceQuery()` emits the `DISTINCT ON` subquery scoped to `account_id = X` and a one-month timestamp window.
5. For an account with N tokens and ~2880 balance snapshots/month, the sort operates on up to N × 2880 rows.
6. Attacker floods with concurrent requests; each request forces PostgreSQL to allocate `work_mem` for the sort. When aggregate sort memory exceeds available RAM, PostgreSQL spills to disk or OOM-kills the backend.

**Why existing checks fail**:
- The `LIMIT 1000` (line 178, `tokenBalanceResponseLimit.singleAccount`) is applied **after** the sort+dedup, not before. It does not reduce sort cost.
- The timestamp range is bounded to one month (from `getAccountBalanceTimestampRange()`), but one month of snapshots for a high-token account is still very large.
- The throttle/rate-limit infrastructure found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`) applies only to the `web3` contract-call service, **not** to the Node.js REST API (`rest/`). No rate limiting exists for unauthenticated callers of `/api/v1/accounts/:id`.

### Impact Explanation
Repeated concurrent requests each force a full in-memory sort of potentially millions of rows in the `token_balance` table. PostgreSQL allocates `work_mem` per sort operation per backend. Exhausting sort memory causes query spill-to-disk (severe latency) or OOM conditions, degrading or crashing the database. Because the mirror node REST API is the primary read interface for the Hedera network, database unavailability prevents clients and nodes from confirming transaction state, constituting a network-wide denial of service.

### Likelihood Explanation
The attack requires zero privileges — only a valid account ID (publicly enumerable) and a valid historical timestamp (any recent block timestamp works). The endpoint is publicly accessible. The attacker needs only a single high-token-count account, which is trivially discoverable. The attack is fully repeatable and scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, `curl` in a loop). No authentication, CAPTCHA, or API key is required.

### Recommendation
1. **Apply rate limiting to the REST API**: Add a per-IP or global request-rate limiter (e.g., `express-rate-limit`) to the Node.js REST server, specifically for endpoints that trigger historical balance queries.
2. **Add a DB-level statement timeout**: Set `statement_timeout` on the PostgreSQL connection pool used by the REST API to abort long-running sort queries before they exhaust memory.
3. **Rewrite the subquery to avoid unbounded sort**: Replace `DISTINCT ON (token_id) ORDER BY token_id, consensus_timestamp DESC` with a lateral join or a CTE that uses the index efficiently, e.g., using `LATERAL` with `ORDER BY consensus_timestamp DESC LIMIT 1` per token_id, which allows index-only lookups.
4. **Enforce a hard cap on the timestamp range**: Reject or narrow requests where the computed `[lower, upper]` window would span more than a configurable maximum number of snapshots.
