### Title
Unauthenticated `/metrics` Endpoint Exposes Per-Route Timing and Inflight Telemetry, Enabling Precision-Timed Resource Exhaustion Against Transaction Submission

### Summary
The `Routes()` function in `rosetta/app/middleware/metrics.go` registers the `/metrics` endpoint using the raw `promhttp.Handler()` with no authentication, IP restriction, or rate limiting at the application layer. This exposes `hiero_mirror_rosetta_request_inflight` (real-time in-flight count per route) and `hiero_mirror_rosetta_request_duration_sum`/`_count` (cumulative latency per route) to any caller who can reach the service. An unprivileged attacker can poll this endpoint to observe the server's current load on the `/construction/submit` route and time a flood of submit requests to coincide with peak load, maximizing the probability that legitimate transaction submissions are dropped or timeout.

### Finding Description
**Exact code location:** `rosetta/app/middleware/metrics.go`, `Routes()`, lines 64–73.

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,           // "/metrics"
            promhttp.Handler().ServeHTTP,  // no auth wrapper
        },
    }
}
```

`promhttp.Handler()` is the default Prometheus exposition handler. It serves the full default registry, which includes all four registered histograms/gauges:

- `hiero_mirror_rosetta_request_inflight{method,route}` — real-time in-flight count
- `hiero_mirror_rosetta_request_duration{method,route,status_code,ws}` — latency histogram (`_sum`, `_count`, `_bucket`)
- `hiero_mirror_rosetta_request_bytes` and `hiero_mirror_rosetta_response_bytes`

These are registered in `init()` (lines 46–52) and instrumented by `MetricsMiddleware` (lines 76–83) which wraps every route including `/construction/submit`.

**Root cause / failed assumption:** The code assumes the `/metrics` endpoint is protected by an external layer (ingress, network policy, or firewall). No such protection exists at the application layer. Contrast with the REST API's `metricsHandler.js` which has an explicit `authenticate()` guard returning HTTP 401 when `config.metrics.config.authentication` is true. The Rosetta metrics controller has no equivalent.

**No NetworkPolicy for the rosetta service:** A glob search for `charts/hedera-mirror-rosetta/templates/networkpolicy*` returns no results. The only NetworkPolicy in the chart set restricts pgpool (port 5432), not the rosetta HTTP port.

**Ingress does not block `/metrics`:** The ingress only routes `/rosetta/account`, `/rosetta/block`, `/rosetta/construction`, etc. — not `/rosetta/metrics`. However, the Traefik `stripPrefix: /rosetta` middleware means a request to `/rosetta/metrics` would be stripped to `/metrics` and forwarded to the pod. More critically, the service is `ClusterIP` but any pod in the cluster (or any external path that reaches the pod port directly) can call `GET /metrics` with no credential.

**Exploit flow:**
1. Attacker polls `GET http://<rosetta-pod>:5700/metrics` at a low rate (e.g., every 5 seconds).
2. Reads `hiero_mirror_rosetta_request_inflight{route="/construction/submit"}` to observe real-time load.
3. Derives average latency: `rate(hiero_mirror_rosetta_request_duration_sum{route="/construction/submit"}[1m]) / rate(hiero_mirror_rosetta_request_duration_count{route="/construction/submit"}[1m])`.
4. Detects when the server is near capacity (inflight gauge approaching the Traefik `inFlightReq: amount: 5` limit, or latency spiking into the 2.5–5 s histogram buckets).
5. At that moment, floods `/construction/submit` with crafted requests, pushing the server over its concurrency limit.
6. Legitimate transaction submissions are rejected or timeout; `hiero.TransactionExecute` calls fail, and the transactions are not forwarded to the Hedera consensus node.

### Impact Explanation
The `/construction/submit` endpoint calls `hiero.TransactionExecute(transaction, c.sdkClient)` to forward signed transactions to a Hedera consensus node. If the rosetta service is saturated, this call fails and the transaction is never submitted to the network — effectively preventing gossip of that transaction. The metrics endpoint gives an attacker a free oracle to determine the optimal moment to launch a saturation attack, increasing its effectiveness compared to blind flooding. The Traefik `inFlightReq: amount: 5` limit is a very low ceiling; knowing when the server is already at 4/5 inflight allows the attacker to add just one more request to tip it over.

### Likelihood Explanation
**Preconditions:** The attacker must be able to reach the rosetta pod's HTTP port. In a default Kubernetes deployment with ClusterIP, this requires cluster-internal access (a compromised pod, a CI/CD runner, or a misconfigured ingress). In non-Kubernetes deployments or deployments where the service is exposed via LoadBalancer/NodePort (common in development and staging), the endpoint is directly internet-accessible. The absence of a NetworkPolicy for the rosetta service means there is no cluster-level enforcement even in Kubernetes. The attack is repeatable and requires no credentials.

### Recommendation
1. **Application-layer authentication:** Wrap `promhttp.Handler()` with an authentication middleware before registering it in `Routes()`. Use HTTP Basic Auth (as the REST API does) or restrict to a separate internal-only port/mux.
2. **Separate metrics port:** Serve `/metrics` on a different port (e.g., 9090) that is not exposed via the ingress or service, and add a NetworkPolicy allowing only the Prometheus scraper pod to reach that port.
3. **Add a NetworkPolicy for the rosetta service:** Create `charts/hedera-mirror-rosetta/templates/networkpolicy.yaml` restricting ingress to the rosetta pod's HTTP port to only the ingress controller and Prometheus pods.
4. **Remove `/metrics` from the Traefik-routed paths** explicitly (add a deny rule for `/rosetta/metrics` at the ingress level as a defense-in-depth measure).

### Proof of Concept
```bash
# Step 1: Confirm unauthenticated access (no credentials required)
curl -s http://<rosetta-host>:5700/metrics | grep hiero_mirror_rosetta_request_inflight

# Step 2: Poll to observe real-time load on the submit route
watch -n 5 'curl -s http://<rosetta-host>:5700/metrics | grep -E "request_inflight|request_duration_sum|request_duration_count"'

# Step 3: When inflight gauge for /construction/submit is near the inFlightReq limit,
# flood the submit endpoint:
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/construction/submit \
    -H 'Content-Type: application/json' \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"signed_transaction":"0x..."}' &
done

# Step 4: Observe that legitimate submit requests now fail (503 or timeout)
# while the attacker's timing was informed by the unauthenticated metrics oracle.
```
