### Title
`RequestFilter` BLOCK Field Filter Bypass via Numeric Block Value — Operator REJECT/THROTTLE Controls Circumvented

### Summary
`RequestFilter` converts the `BlockType` field to a string using `String.valueOf()`, which calls `BlockType.toString()`. For any numeric block value, `toString()` returns the decimal number string (e.g., `"1"`), not `"latest"`. An unprivileged user can therefore bypass any `RequestFilter` configured with `FilterField.BLOCK`, `FilterType.EQUALS`, `expression = "latest"` by supplying any numeric block value in the request, completely evading operator-configured `REJECT` or `THROTTLE` actions targeting latest-block requests.

### Finding Description

**Exact code path:**

`RequestFilter.test()` at lines 31–35: [1](#0-0) 

```java
var value = field.getExtractor().apply(request);
var stringValue = value instanceof String s ? s : String.valueOf(value);
return type.getPredicate().test(stringValue, expression);
```

`FilterField.BLOCK` extracts `ContractCallRequest::getBlock`, which returns a `BlockType` object — not a `String`. The branch `value instanceof String s` is false, so `String.valueOf(value)` is called, which invokes `BlockType.toString()`.

`BlockType.toString()` at lines 58–64: [2](#0-1) 

```java
public String toString() {
    if (this == EARLIEST || this == LATEST) {   // reference equality
        return name;
    }
    return String.valueOf(number);
}
```

`BlockType.LATEST` is a singleton (`new BlockType("latest", Long.MAX_VALUE)`). Any block created via `BlockType.of("1")` or `BlockType.of("0x1")` is a **different object instance**. The reference equality check `this == LATEST` is `false`, so `toString()` returns the numeric string `"1"`, not `"latest"`.

The `FilterType.EQUALS` predicate is `String::equalsIgnoreCase`: [3](#0-2) 

`"1".equalsIgnoreCase("latest")` → `false` → the filter does **not** match → the configured `REJECT` or `THROTTLE` action is **not applied**.

**`RecordFileServiceImpl.findByBlockType()` also uses reference equality:** [4](#0-3) 

```java
if (block == BlockType.EARLIEST) { ... }
else if (block == BlockType.LATEST) { ... }
return recordFileRepository.findByIndex(block.number());
```

A numeric block that happens to equal `Long.MAX_VALUE` is NOT treated as "latest" — it falls through to `findByIndex(Long.MAX_VALUE)`, returning `Optional.empty()` and throwing `BlockNumberNotFoundException`. For any valid historical block number, EVM execution proceeds normally at that historical state.

**Exploit flow:**

1. Operator configures: `FilterField.BLOCK`, `FilterType.EQUALS`, `expression = "latest"`, `action = REJECT` (or `THROTTLE`) — intending to block/throttle all current-state queries.
2. Attacker sends `POST /api/v1/contracts/call` with `{"block": "0x1", ...}`.
3. `BlockType.of("0x1")` creates a new `BlockType("0x1", 1L)` — not the `LATEST` singleton.
4. `RequestFilter.test()` computes `"1".equalsIgnoreCase("latest")` → `false` → filter skips.
5. `ThrottleManagerImpl.throttle()` proceeds past the filter loop without triggering the action. [5](#0-4) 
6. Request reaches `ContractCallService`, executes against block 1 state (or throws `BlockNumberNotFoundException` for non-existent blocks — a controlled error, not unexpected behavior).

### Impact Explanation

An operator who configures a `REJECT` filter on `block = "latest"` to block current-state queries (e.g., during maintenance, or to enforce historical-only access) receives **zero protection** — any user can bypass it by specifying any numeric block value. For `THROTTLE` actions, the per-filter rate limit is entirely bypassed, allowing unlimited requests that should have been throttled, while only the coarse global `gasLimitBucket` and `rateLimitBucket` remain as backstops. This undermines the entire operator-configurable request filtering system for the `BLOCK` field.

### Likelihood Explanation

No privileges are required. The bypass is trivially reproducible by any external user who reads the API documentation, which explicitly states the `block` field accepts hex block numbers or named tags. The attacker needs only to substitute `"latest"` with any numeric block value (e.g., `"0x1"`). The bypass is 100% reliable and repeatable.

### Recommendation

Fix `RequestFilter.test()` to perform semantic comparison for `BlockType`, not string comparison. Specifically, when `FilterField.BLOCK` is used with `FilterType.EQUALS` and `expression = "latest"`, the check should compare against the `BlockType.LATEST` singleton or check `block.number() == Long.MAX_VALUE && block == BlockType.LATEST`. The cleanest fix is to override the string conversion in `RequestFilter` for `BlockType` values:

```java
var stringValue = (value instanceof BlockType bt)
    ? bt.name()          // always returns "latest", "earliest", or the raw input name
    : (value instanceof String s ? s : String.valueOf(value));
```

`BlockType.name()` returns the canonical name field — for `LATEST` it is `"latest"`, for `EARLIEST` it is `"earliest"`, and for numeric blocks it is the original input string (e.g., `"0x1"`). This preserves the intended semantics of the filter expression.

### Proof of Concept

**Setup:** Configure the following in `application.yml`:
```yaml
hiero.mirror.web3.throttle.request:
  - action: REJECT
    filters:
      - field: BLOCK
        type: EQUALS
        expression: "latest"
```

**Step 1 — Confirm filter works for "latest":**
```bash
curl -X POST http://localhost:8545/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"block":"latest","to":"0x0000000000000000000000000000000000000001","gas":21000}'
# Expected: 429 / ThrottleException "Invalid request"
```

**Step 2 — Bypass with numeric block:**
```bash
curl -X POST http://localhost:8545/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"block":"0x1","to":"0x0000000000000000000000000000000000000001","gas":21000}'
# Expected: 200 OK (or 404 BlockNumberNotFoundException) — REJECT filter was NOT triggered
```

The filter is bypassed because `"1".equalsIgnoreCase("latest")` is `false` in `RequestFilter.test()` at line 34. [1](#0-0)
