### Title
Thundering Herd / Cache Stampede on `getAccountTokenAllowances()` Enables Unauthenticated DB Amplification DoS

### Summary
The `/api/v1/accounts/{id}/allowances/tokens` endpoint has no request deduplication. When the Redis response cache is cold or expired, any number of concurrent identical requests all bypass the cache simultaneously and each independently execute the same database query. With no rate limiting present anywhere in the REST middleware stack, an unauthenticated attacker can trivially amplify database load by a factor equal to the number of concurrent requests, exhausting the connection pool and degrading the mirror node REST API for all users.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js` line 18 registers the route: [1](#0-0) 

This calls `TokenAllowanceController.getAccountTokenAllowances` (`rest/controllers/tokenAllowanceController.js` lines 68–81), which calls `TokenAllowanceService.getAccountTokenAllowances`: [2](#0-1) 

`TokenAllowanceService.getAccountTokenAllowances` (lines 86–90) unconditionally executes a database query on every invocation with no deduplication or in-service caching: [3](#0-2) 

The query goes through `BaseService.getRows` → `pool.queryQuietly` → `pg.Pool.query`, hitting the database directly: [4](#0-3) 

**The cache does not prevent this.** `responseCacheCheckHandler` checks Redis before the handler runs. However, it performs no locking or deduplication — it simply checks for a cached value and, if absent, sets a local key and proceeds: [5](#0-4) 

The default Redis TTL is **1 second**: [6](#0-5) 

When the cache entry expires (every second) or Redis is unavailable (`this.ready = false`), all concurrent in-flight requests for the same URL miss the cache simultaneously and each independently execute the DB query. There is no single-flight / promise-coalescing mechanism. The `responseCacheUpdateHandler` only stores the result *after* the query completes, so all N concurrent requests that arrived before the first response was stored all hit the database.

When Redis is disabled (`lazyConnect: !redisConfig.enabled`, `this.ready = false`), `getSingleWithTtl` returns `undefined` immediately and every single request hits the database: [7](#0-6) 

**No rate limiting exists.** A search of the entire `rest/` directory found no `express-rate-limit`, no IP-based throttle, and no per-account request throttle anywhere in the middleware stack: [8](#0-7) 

The DB connection pool has a finite `maxConnections`: [9](#0-8) 

Once the pool is exhausted, new requests queue with `connectionTimeoutMillis`. Under sustained flood, the queue grows unboundedly until timeouts fire, causing errors for all users.

### Impact Explanation

An attacker sending N concurrent identical GET requests to `/api/v1/accounts/0.0.X/allowances/tokens` causes N simultaneous `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 ORDER BY spender, token_id LIMIT $2` queries. For accounts with large numbers of token allowances, each query is non-trivial. At N = 1000 concurrent requests (trivially achievable with `ab`, `wrk`, or async HTTP clients), the database receives 1000 identical queries per cache-miss window (every ≥1 second). This exhausts the connection pool, causes query queuing/timeouts, and degrades or denies service to all legitimate users of the mirror node REST API. No authentication or special privilege is required.

### Likelihood Explanation

The endpoint is fully public (no authentication required, confirmed by the absence of any auth check in the route or controller). Any attacker with internet access and a basic HTTP flood tool can exploit this. The attack is repeatable indefinitely, requires no prior knowledge beyond the public API spec, and is not mitigated by any existing control in the codebase.

### Recommendation

1. **Request deduplication (single-flight):** Implement a promise-coalescing map keyed on the cache key. If a request for key K is already in-flight, subsequent identical requests await the same promise rather than issuing a new DB query.
2. **Rate limiting:** Add per-IP (and optionally per-account-ID) rate limiting middleware (e.g., `express-rate-limit` with a Redis store) before the route handlers.
3. **Increase cache TTL:** The 1-second default TTL means the thundering herd window recurs every second. Increase it to a value appropriate for the data's staleness tolerance (e.g., 5–30 seconds).
4. **DB connection pool protection:** Set a hard cap on queued connections and return `503` immediately when the pool is saturated rather than queuing indefinitely.

### Proof of Concept

```bash
# Send 500 concurrent requests to the same account's token allowances endpoint
# All arrive before the first response is cached (within the 1-second TTL window)
wrk -t10 -c500 -d30s \
  "http://<mirror-node-host>/api/v1/accounts/0.0.1001/allowances/tokens"

# Observe: DB CPU spikes, connection pool exhaustion errors in mirror node logs,
# increased latency / 503s for all other API users.
# With Redis disabled, every single request hits the DB regardless of concurrency timing.
```
