### Title
Cache Key Bypass via Query Parameter Ordering in `cacheKeyGenerator()`

### Summary
`cacheKeyGenerator()` in `rest/middleware/responseCacheHandler.js` hashes `req.originalUrl` verbatim, meaning semantically identical requests with differently ordered query parameters (e.g., `?limit=25&order=asc` vs `?order=asc&limit=25`) produce distinct MD5 cache keys and always result in cache misses. The `requestNormalizer.js` module that would fix this exists but is explicitly not yet wired into the cache key path. Any unprivileged user can exploit this to force repeated database queries, degrading performance for all users.

### Finding Description
**Exact code location:**

`rest/middleware/responseCacheHandler.js`, lines 151–153:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
``` [1](#0-0) 

`req.originalUrl` is the raw URL string as received from the client, preserving exact query parameter order. Two requests with identical parameters in different order produce different MD5 digests and therefore different Redis keys.

**Root cause:** No normalization is applied before hashing. The comment at line 149 explicitly acknowledges the gap:
> `path?query - In the future, this will utilize Edwin's request normalizer (9113).`

**The normalizer exists but is not integrated:** `rest/middleware/requestNormalizer.js` exports `normalizeRequestQueryParams()`, which sorts parameters and adds defaults: [2](#0-1) 

However, a grep of `server.js` and all production `rest/**/*.js` files confirms `normalizeRequestQueryParams` is imported and called **only** in its own test file — it is never registered in the middleware pipeline and never called from `cacheKeyGenerator`. [3](#0-2) 

**Middleware pipeline:** `responseCacheCheckHandler` is registered at line 97 of `server.js` with no prior normalization step. The cache key is generated from the raw URL at the very start of `responseCacheCheckHandler`: [4](#0-3) 

### Impact Explanation
Every unique query parameter ordering is treated as a distinct cache entry. An attacker (or even organic client variation) can enumerate N! orderings of N parameters, each causing a full database query. For a public mirror node API with no authentication, this is trivially exploitable at scale. The backend PostgreSQL pool absorbs all these redundant queries, increasing latency and resource consumption for all concurrent users. The impact is performance degradation / griefing — no funds or data are at risk, consistent with the "Medium: griefing" scope classification.

### Likelihood Explanation
- **No privileges required:** The REST API is public; any HTTP client suffices.
- **Trivially automatable:** A script cycling through parameter orderings requires no special knowledge.
- **Repeatable indefinitely:** Each new ordering is a fresh cache miss; the attacker can sustain pressure continuously.
- **Amplified by short default TTL:** `DEFAULT_REDIS_EXPIRY = 1` second means even legitimately cached entries expire quickly, compounding the miss rate. [5](#0-4) 

### Recommendation
Integrate `normalizeRequestQueryParams` into `cacheKeyGenerator` before hashing:

```js
import {normalizeRequestQueryParams} from './requestNormalizer.js';

const cacheKeyGenerator = (req) => {
  const normalizedUrl = normalizeRequestQueryParams(
    req.openapi?.openApiRoute,
    req.path,
    req.query
  );
  return crypto.createHash('md5').update(normalizedUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

This is exactly what the existing TODO comment (issue 9113) tracks. The normalizer already handles parameter sorting, default injection, and collapsible parameters correctly, as evidenced by its test suite. [6](#0-5) 

### Proof of Concept
**Precondition:** Mirror node running with Redis cache enabled (`config.cache.response.enabled = true`, `config.redis.enabled = true`).

**Steps:**

1. Send the first request (populates cache under key A):
   ```
   GET /api/v1/accounts?limit=25&order=asc
   ```
   → Cache miss → DB query executed → response cached under `md5("/api/v1/accounts?limit=25&order=asc")-v1`.

2. Send the semantically identical request with reordered parameters:
   ```
   GET /api/v1/accounts?order=asc&limit=25
   ```
   → `cacheKeyGenerator` computes `md5("/api/v1/accounts?order=asc&limit=25")-v1` — a **different key** → cache miss → another DB query executed.

3. Automate with a script cycling all permutations of common parameters (`limit`, `order`, `account.id`, `balance`, etc.) in a tight loop. Each permutation bypasses the cache, forcing a fresh DB query every time.

**Observed result:** Redis cache hit rate approaches 0% for the targeted endpoint; database connection pool saturation and increased query latency for all users.

const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
};
```

const getNormalizedArrayValue = (name, valueArray) => {
  if (isEmpty(valueArray)) {
    return;
  }

  if (!NON_SORTED_PARAMS.includes(name)) {
    // Sort the order of the parameters within the array
    valueArray.sort();
  } else if (COLLAPSABLE_PARAMS.includes(name)) {
    // Only add the last item in the array to the query parameter
    valueArray = valueArray.slice(valueArray.length - 1);
  }

  return valueArray.join('&' + name + '=');
};
```
