### Title
Cache-Bypass Token-Saturation Griefing via Unique Near-Limit GraphQL Queries

### Summary
The GraphQL parser is configured with `maxTokens(1000)`, which bounds but does not eliminate per-request parsing work. The preparsed-document cache is keyed on the raw query string, so an attacker who sends a stream of syntactically distinct but structurally equivalent queries (e.g., varying a comment or alias) bypasses the cache on every request. With no rate limiting present in the GraphQL layer, an unprivileged attacker can sustain maximum-token parsing CPU load across all server threads, degrading throughput for legitimate concurrent users.

### Finding Description
**Code locations:**

- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, `static{}` block, line 34:
  ```java
  b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);
  ```
  This sets the upper bound for parsing work per request to 1000 tokens. The parser processes every token up to this limit before rejecting or accepting the query.

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
  ```java
  return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
  ```
  The cache key is the **raw query string**. Any byte-level difference in the query string (a comment, a renamed alias, extra whitespace) produces a cache miss and triggers a full parse+validate cycle.

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15:
  ```
  expireAfterWrite=1h,maximumSize=1000
  ```
  The cache holds at most 1000 entries. An attacker who continuously rotates unique query strings keeps the cache in a perpetual miss state.

**Root cause:** The assumed protection model is "repeated identical queries are cached; novel queries are rare." This assumption fails when an attacker deliberately generates a high-volume stream of syntactically unique queries. The `maxTokens(1000)` limit bounds the *worst-case cost per query* but does not bound the *rate* at which that cost can be imposed. No rate-limiting, connection throttling, or per-IP query budget exists in the GraphQL application layer.

**Exploit flow:**
1. Attacker constructs a query template with ~1000 tokens (e.g., a deeply aliased selection set within the `maxRuleDepth(100)` and `maxComplexity(200)` limits).
2. For each request, the attacker appends a unique comment (`# r=<counter>`) or renames one alias, producing a new raw string.
3. Each request is a cache miss → `parseAndValidateFunction.apply(executionInput)` is called → parser tokenizes up to 1000 tokens → validation runs.
4. Attacker opens many concurrent connections and repeats at maximum throughput.
5. Server threads are saturated with parse+validate work; legitimate requests queue or time out.

**Why existing checks are insufficient:**
- `maxTokens(1000)` caps per-query cost but is not a rate control.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` run *after* parsing; they do not reduce parsing overhead.
- The Caffeine cache (`maximumSize=1000, expireAfterWrite=1h`) only deduplicates identical strings; trivial query mutation defeats it entirely.
- No rate limiting, authentication, or per-source query budget is present in the GraphQL module.

### Impact Explanation
An unprivileged attacker can force the server to perform maximum-allowed GraphQL parsing work on every request indefinitely. Under sustained load this degrades parse throughput for all concurrent users — a griefing scenario with no economic damage to the network but measurable latency and throughput degradation for the mirror-node GraphQL API. Severity is Medium, consistent with the stated classification.

### Likelihood Explanation
The attack requires no credentials, no special knowledge of the schema beyond what is publicly introspectable, and no sophisticated tooling — a simple script that increments a comment counter and sends HTTP POST requests suffices. It is trivially repeatable and automatable from a single host or a small botnet.

### Recommendation
1. **Add per-IP or per-connection rate limiting** at the Spring/servlet filter layer (e.g., Bucket4j, Spring Cloud Gateway rate filter, or a servlet `Filter` counting requests per source IP per second).
2. **Normalize query strings before caching** (strip comments and normalize whitespace) so that trivially mutated queries hit the cache. The graphql-java `QueryTransformer` or a pre-processing step can canonicalize the query string before it is used as the cache key.
3. **Enforce a global concurrent-parse budget** (e.g., a `Semaphore` around the parse path) to bound total simultaneous parsing threads regardless of query uniqueness.

### Proof of Concept
```python
import requests, threading, itertools

URL = "http://<mirror-node>/graphql"
# Build a ~1000-token query (aliases + fields within complexity/depth limits)
BASE = "query { " + " ".join(f"a{i}: account(input:{{entityId:{{shard:0,realm:0,num:1}}}}){{balance}}" for i in range(40)) + " }"

def flood(seed):
    for i in itertools.count():
        q = f"# {seed}-{i}\n" + BASE   # unique raw string → cache miss every time
        requests.post(URL, json={"query": q}, timeout=5)

threads = [threading.Thread(target=flood, args=(t,)) for t in range(50)]
for t in threads: t.start()
for t in threads: t.join()
```
Each request carries a unique comment, guaranteeing a cache miss and a full 1000-token parse cycle on every call. Under 50 concurrent threads this sustains maximum parsing CPU load with no credentials required. [1](#0-0) [2](#0-1) [3](#0-2)
