### Title
Small-Subgroup / Weak-Key Attack in `ConstructionCombine` Ed25519 Signature Verification

### Summary
`ConstructionCombine` in `rosetta/app/services/construction_service.go` calls Go's `crypto/ed25519.Verify` with no pre-validation of the public key or signature bytes for low-order (small-subgroup) points. An attacker can supply a 32-byte all-zero public key — which encodes the valid Ed25519 low-order point (±√−1, 0) of order 4 — together with a 64-byte all-zero signature, and for a crafted-but-valid unsigned transaction body M where `k = SHA-512(R‖A‖M) mod order ≡ 3 (mod 4)`, `ed25519.Verify` returns `true` without the attacker possessing any private key. This bypasses `ErrInvalidSignatureVerification` and causes the endpoint to return a "signed" transaction.

### Finding Description

**Exact location:** `rosetta/app/services/construction_service.go`, function `ConstructionCombine`, lines 72–87.

```go
for _, signature := range request.Signatures {
    if signature.SignatureType != rTypes.Ed25519 {          // only type check
        return nil, errors.ErrInvalidSignatureType
    }

    pubKey, err := hiero.PublicKeyFromBytes(signature.PublicKey.Bytes)  // no curve-point validation
    if err != nil {
        return nil, errors.ErrInvalidPublicKey
    }

    if !ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes) {  // line 82
        return nil, errors.ErrInvalidSignatureVerification
    }

    _, _ = hiero.TransactionAddSignature(transaction, pubKey, signature.Bytes)
}
```

**Root cause — failed assumption:** The code assumes `ed25519.Verify` is a sufficient guard against forged signatures. It is not when the public key is a low-order point, because the Ed25519 group has cofactor 8 and contains eight low-order points whose order divides 8.

**Cryptographic mechanics:**

The 32-byte all-zero encoding has `y = 0`. Solving the Ed25519 curve equation `−x² + 0 = 1 + d·x²·0` gives `x² = −1 mod p`, which has a solution because `p ≡ 5 (mod 8)`. The resulting point `P = (±√−1, 0)` has order 4:

```
2·P = (0, −1)   [order-2 point]
4·P = (0,  1)   = identity
```

`filippo.io/edwards25519` (used by Go ≥ 1.17, and this module declares `go 1.26`) does **not** reject low-order points in `SetBytes`; it only checks canonical encoding. `y = 0` is canonical (`0 < p`), so the all-zero encoding is accepted as a valid curve point.

Go's `ed25519.Verify` (non-cofactor variant, RFC 8032) checks:

```
[S]B == R + [k]A
```

With `S = 0`, `R = P`, `A = P` (both encoded as 32 zero bytes):

```
identity == P + [k]·P == [k+1]·P
```

Since `ord(P) = 4`, this holds when `k + 1 ≡ 0 (mod 4)`, i.e., `k ≡ 3 (mod 4)`.

`k = SHA-512(zeros₃₂ ‖ zeros₃₂ ‖ M) mod order`

Because `order` is an odd prime, `k mod 4` is uniformly distributed in `{0,1,2,3}`. The attacker controls `M` (the `UnsignedTransaction` body) and can vary any mutable field (e.g., `valid_start_nanos`, memo) to find an `M` satisfying `k ≡ 3 (mod 4)` in an expected **4 attempts**.

**Existing checks and why they fail:**

| Check | Outcome |
|---|---|
| `len(request.Signatures) == 0` | Passes — attacker provides one signature |
| `signature.SignatureType != rTypes.Ed25519` | Passes — attacker sets type to Ed25519 |
| `hiero.PublicKeyFromBytes(zeros₃₂)` | Passes — SDK stores raw bytes without curve-point validation |
| `sig[63]&224 != 0` (inside `ed25519.Verify`) | Passes — `0 & 224 = 0` |
| `S < order` (inside `ed25519.Verify`) | Passes — `S = 0 < order` |
| `SetBytes` on R (inside `ed25519.Verify`) | Passes — `y = 0` is a valid canonical encoding |

No check rejects a low-order public key or a low-order R value.

### Impact Explanation

An unprivileged external user can obtain a `ConstructionCombineResponse` containing a `SignedTransaction` without possessing the legitimate private key. The Rosetta API's signature-verification gate — the only cryptographic check before `hiero.TransactionAddSignature` embeds the forged bytes — is defeated. Any downstream system or monitoring tool that trusts the Rosetta API's combine step to certify signature validity is deceived. When the forged transaction is forwarded to `ConstructionSubmit`, the Hedera network performs its own independent signature check and will reject it (unless the on-chain account key happens to be the same low-order point, which is negligible in practice). The primary impact is therefore **integrity violation at the Rosetta API layer**: forged "signed" transactions are produced and accepted by the API, undermining the security model of the offline signing flow.

### Likelihood Explanation

The attack requires no privileges, no secret material, and no brute-force beyond an expected 4 HTTP requests to find a suitable `M`. The attacker only needs to vary a freely-chosen field (e.g., `valid_start_nanos`) in a valid unsigned transaction. The all-zero public key and signature are trivially constructed. The attack is fully scriptable and repeatable by any external caller of the `/construction/combine` endpoint.

### Recommendation

1. **Reject low-order public keys before calling `ed25519.Verify`.** Maintain a deny-list of the eight known low-order Ed25519 point encodings and return `ErrInvalidPublicKey` if the supplied key matches any of them.

2. **Reject low-order R values.** Before calling `ed25519.Verify`, check that `signature.Bytes[:32]` is not one of the eight low-order point encodings and return `ErrInvalidSignatureVerification` if it is.

3. **Enforce minimum signature entropy.** Reject any `signature.Bytes` that is all-zero or matches a known-weak pattern before passing it to `ed25519.Verify`.

4. **Use a cofactor-aware verification.** Switch to a verification that multiplies both sides by the cofactor 8 (`[8][S]B == [8]R + [8][k]A`), which maps all low-order points to the identity and eliminates the small-subgroup class of attacks entirely.

### Proof of Concept

```python
import hashlib, struct

# Known low-order Ed25519 point encodings (order 4, y=0)
LOW_ORDER_POINT_32 = bytes(32)          # all-zero encoding of (±√-1, 0)
ZERO_SIG_64       = bytes(64)           # R = LOW_ORDER_POINT_32, S = 0

def k_mod4(transaction_body_bytes: bytes) -> int:
    h = hashlib.sha512(LOW_ORDER_POINT_32 + LOW_ORDER_POINT_32 + transaction_body_bytes).digest()
    # k = h mod order; for mod-4 check, only the low 2 bits of h matter
    return int.from_bytes(h[:4], 'little') % 4

# Step 1: obtain a valid unsigned transaction hex from /construction/payloads
# Step 2: iterate valid_start_nanos until k ≡ 3 (mod 4)
for nanos_delta in range(100):
    body = build_transaction_body(valid_start_nanos=BASE_NANOS + nanos_delta)
    if k_mod4(body) == 3:
        break   # found suitable M in expected 4 iterations

# Step 3: POST to /construction/combine
payload = {
    "unsigned_transaction": hex_encode(unsigned_tx),
    "signatures": [{
        "public_key": {"bytes": LOW_ORDER_POINT_32.hex(), "curve_type": "edwards25519"},
        "signature_type": "ed25519",
        "hex_bytes": ZERO_SIG_64.hex(),
        "signing_payload": {"hex_bytes": body.hex(), "signature_type": "ed25519"}
    }]
}
# Response: HTTP 200 with a signed_transaction — no private key used.
```
