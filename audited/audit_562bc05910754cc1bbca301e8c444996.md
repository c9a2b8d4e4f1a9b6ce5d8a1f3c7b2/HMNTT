### Title
Account Existence Oracle via Base32 Alias 404 Response in `getNftAllowances`

### Summary
The `getNftAllowances` endpoint at `/api/v1/accounts/{id}/allowances/nfts` performs a database lookup when the path parameter is a base32 alias, and throws a `404 EntityNotFoundException` if no matching entity exists. An unauthenticated attacker can distinguish between existing and non-existing accounts by observing the HTTP response code (404 vs 200), enabling systematic alias enumeration with no privilege required.

### Finding Description
**Exact code path:**

`AllowancesController.getNftAllowances()` (line 58–83) accepts `@PathVariable EntityIdParameter id`. The `EntityIdParameter.valueOf()` factory ( [1](#0-0) ) parses the path variable and, when it matches the base32 alias regex (`[A-Z2-7]{40,70}`), produces an `EntityIdAliasParameter`. [2](#0-1) 

The controller passes the request to `NftAllowanceServiceImpl.getNftAllowances()`, which calls `entityService.lookup(request.getAccountId())` at line 28: [3](#0-2) 

`EntityServiceImpl.lookup()` branches on the parameter type. For `EntityIdAliasParameter`, it queries the database and, if no row is found, throws `EntityNotFoundException("No account found for the given ID")`: [4](#0-3) 

**Root cause:** The alias branch unconditionally performs a DB lookup and surfaces a distinct 404 error for non-existent aliases, while a valid alias returns 200 (even with an empty allowances list). This creates a binary oracle. By contrast, numeric IDs (`EntityIdNumParameter`) are returned directly without any DB lookup (line 32: `Optional.of(p.id())`), so they never trigger this 404 path.

**Why existing checks are insufficient:** There is no authentication, rate-limiting at the application layer, or response normalization (e.g., returning 200 with empty results for non-existent aliases). The `EntityNotFoundException` propagates directly to the HTTP layer as a 404.

### Impact Explanation
An attacker can enumerate which base32 aliases correspond to live accounts on the network. This leaks account existence information that is not otherwise publicly guaranteed to be discoverable. For privacy-sensitive use cases (e.g., accounts using aliases to avoid linkability), this oracle defeats that protection. The impact is information disclosure of account existence; it does not allow fund theft or state mutation, but it is a meaningful privacy violation and reconnaissance primitive.

### Likelihood Explanation
Exploitation requires zero privileges — the endpoint is fully public. The attacker only needs to generate or collect candidate base32 aliases (e.g., from observed transactions, known public keys, or brute-force of the 40–70 character base32 space) and issue GET requests. The response code difference (404 vs 200) is unambiguous and machine-readable. The attack is trivially scriptable and repeatable at scale.

### Recommendation
Normalize the response for non-existent aliases: instead of propagating `EntityNotFoundException` as a 404, return HTTP 200 with an empty `allowances` list (matching the behavior for numeric IDs with no allowances). This removes the oracle. If a 404 is intentionally desired for aliases, apply consistent rate-limiting and consider whether the distinction between "alias not found" and "alias found but no allowances" needs to be exposed at all.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
