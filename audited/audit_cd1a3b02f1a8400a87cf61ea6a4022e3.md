### Title
Global GraphQL Preparsed Document Cache Thrashing via Unbounded Unique Query Flooding

### Summary
The `CachedPreparsedDocumentProvider` maintains a single global Caffeine `AsyncCache` keyed on raw query strings with `maximumSize=1000` and no per-user or per-IP quota. An unauthenticated attacker can continuously submit syntactically valid but unique queries to saturate and continuously evict the cache, forcing every legitimate user's request to incur a full re-parse and re-validate cycle on every hit. No rate limiting exists in the GraphQL module itself to prevent this.

### Finding Description

**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, lines 19–27:
```java
CachedPreparsedDocumentProvider(CacheProperties properties) {
    cache = Caffeine.from(properties.getQuery()).buildAsync();  // single global cache
}

public CompletableFuture<PreparsedDocumentEntry> getDocumentAsync(
        ExecutionInput executionInput, Function<ExecutionInput, PreparsedDocumentEntry> parseAndValidateFunction) {
    return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
}
```

`graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15:
```java
private String query = "expireAfterWrite=1h,maximumSize=1000,recordStats";
```

**Root cause:** The cache is a single global instance with a hard cap of 1000 entries. The cache key is the raw query string. Any syntactically distinct query string (e.g., varying aliases, argument names, or whitespace) occupies a unique slot. There is no per-IP, per-user, or per-session quota on cache entries. Once the cache is full, Caffeine evicts entries by its W-TinyLFU policy; a sustained flood of novel queries continuously displaces legitimate cached entries.

**Failed assumption:** The design assumes that the set of distinct query strings in production is small and stable (i.e., a fixed set of client queries). It does not account for an adversary who deliberately generates an unbounded stream of unique but valid query strings.

**Why existing checks are insufficient:**

- `inFlightReq: amount: 5` (Traefik middleware, `charts/hedera-mirror-graphql/values.yaml` line 138–142): limits only *concurrent* in-flight requests per source IP. It does not cap the *sequential* request rate. An attacker can send 5 concurrent requests, wait for responses, and immediately send 5 more — cycling through thousands of unique queries per minute from a single IP.
- `ParserOptions` limits (`GraphQlConfiguration.java` lines 33–38): `maxCharacters=10000`, `maxTokens=1000`. These constrain individual query size but do not prevent generating 1000+ distinct small queries.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` (`GraphQlConfiguration.java` lines 43–45): these are execution-phase instrumentations that run *after* the cache lookup and parse step. They do not prevent cache population or eviction.
- **Zero rate-limiting code exists in the GraphQL module** (`graphql/src/**/*.java` contains no throttle/rateLimit logic), unlike the web3 module which uses bucket4j.

### Impact Explanation

Every cache miss triggers `parseAndValidateFunction.apply(executionInput)` — a synchronous parse + schema validation pass. Under continuous cache thrashing, all legitimate requests become cache misses. This increases per-request CPU time and latency for all users. Under concurrent load, the parse/validate work compounds with normal DB query latency, degrading overall throughput. The cache's `expireAfterWrite=1h` TTL means evicted entries do not self-heal until the attacker stops. The attack degrades service for all users of the shared instance.

### Likelihood Explanation

The endpoint is publicly accessible (`/graphql/alpha`, no authentication). Generating unique valid queries requires only trivial variation: changing a field alias (`{ account(input:{...}) { balance } }` vs `{ a: account(input:{...}) { balance } }`), or varying whitespace. A single attacker with one IP can sustain the attack using sequential batches of 5 concurrent requests (the Traefik `inFlightReq` limit). No special privileges, credentials, or insider knowledge are required. The attack is fully repeatable and automatable with a simple script.

### Recommendation

1. **Add per-IP request rate limiting to the GraphQL module** using bucket4j (already present in the web3 module) — apply it as a `WebGraphQlInterceptor` or servlet filter before the preparsed document cache is consulted.
2. **Bound cache entry creation per source IP**: track how many distinct query strings a given IP has contributed to the cache within a time window and reject or shadow-drop requests that exceed the threshold.
3. **Reduce `maximumSize`** to a value reflecting the actual number of distinct legitimate client queries, or switch to a time-bounded sliding window eviction policy.
4. **Add a Traefik `rateLimit` middleware** (requests per second per IP) in addition to the existing `inFlightReq` middleware, to cap sequential request throughput.

### Proof of Concept

```bash
# Fill the cache with 1000 unique queries (varying alias name)
for i in $(seq 1 1000); do
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ a$i: account(input: {entityId: {shard:0,realm:0,num:2}}) { balance } }\"}" &
  # Traefik allows 5 concurrent; batch in groups of 5
  if (( i % 5 == 0 )); then wait; fi
done

# Now continuously send new unique queries to maintain eviction pressure
i=1001
while true; do
  for j in $(seq $i $((i+4))); do
    curl -s -X POST http://<host>/graphql/alpha \
      -H 'Content-Type: application/json' \
      -d "{\"query\": \"{ a$j: account(input: {entityId: {shard:0,realm:0,num:2}}) { balance } }\"}" &
  done
  wait
  i=$((i+5))
done
# Legitimate users now experience cache misses on every request,
# forcing full re-parse+validate on each query.
```
