### Title
Unauthenticated, Rate-Limit-Free `getNftAllowances` Endpoint Enables Sustained DB Connection Pool and CPU Exhaustion

### Summary
The `GET /api/v1/accounts/{id}/allowances/nfts` endpoint is publicly accessible with no authentication, no application-level rate limiting, and no per-IP throttling. Each request triggers at minimum two database operations — an entity lookup and a full `nft_allowance` table query — allowing any unprivileged attacker to flood the endpoint with high-frequency calls using distinct valid `accountId` path values, exhausting the DB connection pool and degrading response times for all legitimate users.

### Finding Description
**Code path:**
- Controller: `rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, lines 57–83
- Service: `rest-java/src/main/java/org/hiero/mirror/restjava/service/NftAllowanceServiceImpl.java`, lines 21–31
- Repository: `rest-java/src/main/java/org/hiero/mirror/restjava/repository/NftAllowanceRepositoryCustomImpl.java`, lines 37–47

**Root cause:** The controller method `getNftAllowances()` carries no `@RateLimiter`, no `@Throttle`, no Bucket4j/Resilience4j annotation, and no authentication guard. A grep across all of `rest-java/src/main/java/**/*.java` for `RateLimit`, `Bucket4j`, `resilience4j`, and `@Throttle` returns zero matches for this endpoint.

**Exploit flow:**
1. `AllowancesController.getNftAllowances()` accepts any `{id}` path variable as `EntityIdParameter` with no credential check.
2. `NftAllowanceServiceImpl.getNftAllowances()` calls `entityService.lookup(request.getAccountId())` — a DB round-trip to resolve the entity — then calls `repository.findAll(request, id)`.
3. `NftAllowanceRepositoryCustomImpl.findAll()` executes `dslContext.selectFrom(NFT_ALLOWANCE).where(condition).orderBy(...).limit(request.getLimit()).fetchInto(NftAllowance.class)` — a second DB round-trip.
4. By rotating through distinct valid `accountId` values, the attacker bypasses any hypothetical per-account caching and forces two DB operations per request, every request.

**Why existing checks fail:**
- `@Max(MAX_LIMIT)` on the `limit` parameter bounds result-set size per query but does not bound request frequency.
- `@Size(max = 2)` on `accountIds` and `tokenIds` arrays limits filter complexity but not call rate.
- `checkOwnerSpenderParamValidity()` is a logical guard, not a rate guard.
- No infrastructure-level rate limiting is configured in `rest-java/src/main/resources/`.

### Impact Explanation
Each inbound request consumes one DB connection for entity resolution and one for the allowance query. With a sufficiently high request rate (achievable from a single machine with async HTTP clients), the JDBC/R2DBC connection pool saturates. Once the pool is exhausted, all other endpoints sharing the same pool — including token, topic, and network endpoints — begin queuing or rejecting requests. This constitutes a full service degradation for all legitimate users with no economic cost to the attacker.

### Likelihood Explanation
The attacker requires no account, no API key, and no on-chain funds. Any HTTP client capable of sending concurrent GET requests suffices. Rotating through distinct `accountId` values (e.g., sequential numeric IDs `0.0.1`, `0.0.2`, …) is trivial and defeats any per-key caching. The attack is fully repeatable and automatable.

### Recommendation
1. Apply a per-IP rate limit at the application layer using Resilience4j `@RateLimiter` or Bucket4j on `AllowancesController.getNftAllowances()`.
2. Configure a global request-rate limit in the Spring WebFlux/MVC filter chain for the `/api/v1/accounts/*/allowances/nfts` path.
3. Enforce a minimum DB connection pool reservation for non-allowance endpoints so pool exhaustion on this endpoint cannot starve others.
4. Consider adding an API gateway (e.g., nginx `limit_req`) in front of the service as a defense-in-depth layer.
