### Title
Unauthenticated Account Hook Enumeration via Sequential ID Iteration in `HooksController.getHooks()`

### Summary
The `GET /api/v1/accounts/{ownerId}/hooks` endpoint in `HooksController` requires no authentication and has no rate limiting. Because `EntityServiceImpl.lookup()` returns a numeric `EntityId` directly for `EntityIdNumParameter` inputs without verifying the account exists in the database, any caller can iterate over all numeric account IDs and distinguish accounts that have gossip-prevention hooks registered (non-empty response) from those that do not (empty list), fully enumerating the hook-registered account set.

### Finding Description
**Code path:**

- `HooksController.java` lines 80–102: `@GetMapping` with no `@PreAuthorize` or any security annotation. The `ownerId` path variable is resolved as `EntityIdParameter`.
- `EntityServiceImpl.java` lines 30–38: `lookup()` switches on the parameter type; for `EntityIdNumParameter` it executes `Optional.of(p.id())` — a pure in-memory construction with **no database existence check**.
- `HookServiceImpl.java` lines 32–52: `getHooks()` calls `entityService.lookup(request.getOwnerId())` to obtain the numeric ID, then immediately queries `hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), ...)`. For an account with hooks the repository returns rows; for any other numeric ID it returns an empty list.

**Root cause:** The `EntityIdNumParameter` branch in `lookup()` intentionally skips the repository call that the alias and EVM-address branches use, so a fabricated numeric ID is treated as valid. Combined with the absence of any authentication guard or rate-limit on the controller, the response body alone (populated vs. empty) leaks whether a given account has hooks.

**Why existing checks fail:** No Spring Security configuration was found for the `rest-java` module. The `authHandler.js` and `ThrottleManagerImpl` belong to the separate `rest` (Node.js) and `web3` modules respectively and do not protect this endpoint. The `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` and `@Positive @Max(MAX_LIMIT)` annotations on query parameters constrain filter values, not access.

### Impact Explanation
An attacker learns the complete set of account IDs that have gossip-prevention hooks registered. Because hooks are specifically designed to suppress gossip of certain transactions, this list is operationally sensitive: it reveals which accounts are actively trying to hide transaction activity, enabling targeted surveillance, front-running, or social-engineering attacks against those account holders. Severity is **High** given the sensitivity of the disclosed information and the zero-barrier access.

### Likelihood Explanation
Exploitation requires only an HTTP client and knowledge of the public API path (documented in the mirror-node spec). The numeric account ID space is bounded and sequential (shard.realm.num), making a full sweep feasible with a simple loop. No credentials, tokens, or special network position are needed. The attack is fully repeatable and automatable.

### Recommendation
1. **Authentication gate:** Add `@PreAuthorize("isAuthenticated()")` (or an equivalent Spring Security rule) to `HooksController.getHooks()` and `getHookStorage()`, or configure a security filter chain in `rest-java` that requires authentication for `/api/v1/accounts/*/hooks`.
2. **Rate limiting:** Apply a per-IP or per-client token-bucket rate limiter (analogous to `ThrottleManagerImpl` in `web3`) to all `rest-java` endpoints, especially hooks.
3. **Existence validation:** In the `EntityIdNumParameter` branch of `EntityServiceImpl.lookup()`, perform a repository existence check (as the alias/EVM-address branches already do) so that requests for non-existent accounts return 404 rather than an empty list, eliminating the oracle even if authentication is misconfigured.

### Proof of Concept
```bash
# Enumerate accounts 1–10000 for registered hooks (no credentials required)
for id in $(seq 1 10000); do
  RESPONSE=$(curl -s "https://<mirror-node>/api/v1/accounts/0.0.${id}/hooks")
  # Non-empty hooks array means this account has gossip-prevention hooks
  echo "$RESPONSE" | python3 -c "
import sys, json
data = json.load(sys.stdin)
if data.get('hooks'):
    print(f'Account 0.0.${id} has hooks: {data[\"hooks\"]}')
"
done
```
Accounts returning a non-empty `hooks` array are confirmed to have gossip-prevention hooks registered. The scan completes with no authentication, no special headers, and no server-side blocking.
