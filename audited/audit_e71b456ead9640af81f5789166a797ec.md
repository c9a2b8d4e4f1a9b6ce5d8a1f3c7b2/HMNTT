### Title
Unauthenticated `internal=true` Query Parameter Bypasses Internal Transaction Filter

### Summary
The `filterKeys.INTERNAL` query parameter (`internal`) is listed as an accepted parameter on the public `/api/v1/contracts/{id}/results` and `/api/v1/contracts/results` endpoints with no authentication or authorization gate. When any unauthenticated caller passes `?internal=true`, the server omits the `transaction_nonce = 0` SQL condition that normally hides internal (child) contract transactions, exposing the full set of internal gossip-generated transactions to any external user.

### Finding Description
**Exact code path:**

`rest/constants.js` line 38 defines the key: [1](#0-0) 

`rest/controllers/contractController.js` lines 1336–1346 include `filterKeys.INTERNAL` in the publicly-accepted parameter set with no privilege annotation: [2](#0-1) 

`rest/controllers/contractController.js` lines 478–480 unconditionally assign the user-supplied boolean to the `internal` variable: [3](#0-2) 

Lines 531–533 are the only guard: the `transaction_nonce = 0` SQL condition is appended **only when `internal` is falsy**: [4](#0-3) 

**Root cause / failed assumption:** The design assumes `internal=true` will only be supplied by privileged callers, but no such restriction is enforced anywhere. The `authHandler` middleware (`rest/middleware/authHandler.js` lines 15–36) only sets a custom response-size limit for authenticated users; it does not block or inspect the `internal` parameter: [5](#0-4) 

The `filterValidityChecks` in `rest/utils.js` line 327–329 only validates that the value is a syntactically valid boolean — it performs no caller-identity check: [6](#0-5) 

The route registration in `rest/routes/contractRoute.js` attaches no auth middleware to the contract results routes: [7](#0-6) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results?internal=true` (or `GET /api/v1/contracts/{id}/results?internal=true`) with no credentials.
2. `buildAndValidateFilters` accepts `internal` because it is in `acceptedContractResultsParameters` and passes the boolean validity check.
3. `extractContractResultsByIdQuery` sets `internal = true` at line 479.
4. The `if (!internal)` branch at line 531 is skipped — the `transaction_nonce = 0` condition is never added to the SQL query.
5. The database returns all contract results including those with `transaction_nonce > 0` (internal/child transactions), which are normally invisible to public callers.

### Impact Explanation
Internal transactions (nonce > 0) represent child contract calls generated by the network during EVM execution — they are not directly submitted by users and are intentionally hidden from the default public view. Exposing them leaks the full internal execution graph of every contract interaction: intermediate call chains, internal value transfers, and child contract invocations. This constitutes an unintended information disclosure of protocol-internal state to any unauthenticated internet user, with no rate-limiting specific to this parameter.

### Likelihood Explanation
The exploit requires zero privileges, zero credentials, and a single HTTP GET request with one extra query parameter (`?internal=true`). It is trivially discoverable by reading the open-source code or by fuzzing the query parameters of the public API. It is fully repeatable and automatable. Any external party — including competitors, researchers, or malicious actors — can exploit it continuously.

### Recommendation
1. **Gate the parameter behind authentication:** In `extractContractResultsByIdQuery` (or in a dedicated middleware), check whether the caller is authenticated before honoring `internal=true`. If the caller is unauthenticated, silently force `internal = false` regardless of the query parameter.
2. **Remove from public accepted-parameter set:** Remove `filterKeys.INTERNAL` from `acceptedContractResultsParameters` and expose it only through a separate, authenticated endpoint or a privileged parameter set.
3. **OpenAPI spec enforcement:** Add the `internal` parameter to the OpenAPI spec with a security requirement so that `openApiValidator` rejects unauthenticated requests that supply it.

### Proof of Concept
```
# No credentials required — any unauthenticated user can run this
curl "https://<mirror-node-host>/api/v1/contracts/results?internal=true&limit=25"

# Or scoped to a specific contract
curl "https://<mirror-node-host>/api/v1/contracts/0.0.1234/results?internal=true&limit=25"
```

**Expected (default) behavior:** Only top-level transactions (`transaction_nonce = 0`) are returned.

**Observed behavior with exploit:** All contract results including internal child transactions (`transaction_nonce > 0`) are returned, exposing the full internal execution graph to the unauthenticated caller.
