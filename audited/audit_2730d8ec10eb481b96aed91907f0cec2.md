### Title
Pre-Execution-Only Timeout Check in StatementInspector Allows In-Flight Queries to Hold DB Connections Beyond requestTimeout, Enabling Connection Pool Exhaustion DoS

### Summary
The `StatementInspector` in `HibernateConfiguration.java` checks elapsed time only **before** a SQL statement is submitted to the database. It does not interrupt a query already in flight. A query that begins just under the 10-second `requestTimeout` can run for an additional period bounded only by the PostgreSQL user-level `statement_timeout` (10 seconds per the Helm chart), holding a HikariCP connection for up to ~20 seconds per request. An unprivileged attacker sending a modest number of concurrent contract-call requests that each trigger one slow query can exhaust the connection pool and deny service to all other users.

### Finding Description

**Exact code path:**

`Web3Properties.getRequestTimeout()` returns `Duration.ofSeconds(10L)` (10 000 ms). [1](#0-0) 

`ContractCallContext.startTime` is set once at context creation via `System.currentTimeMillis()`. [2](#0-1) 

`HibernateConfiguration.statementInspector()` reads `startTime`, computes `elapsed`, and throws `QueryTimeoutException` only when `elapsed >= timeout` — **before** the SQL string is handed to JDBC. [3](#0-2) 

**Root cause and failed assumption:**

`StatementInspector` is a Hibernate pre-execution hook. It is invoked before the driver sends the SQL to PostgreSQL. Once the driver sends the query, `StatementInspector` is never called again for that statement. There is no mechanism in this code path to cancel a query that is already executing. The design assumes that the check before each query is sufficient to bound total request duration, but this is false: a query that passes the check at `elapsed = 9 999 ms` can then run for an additional period equal to the PostgreSQL `statement_timeout`.

**Effective PostgreSQL statement_timeout for the web3 user:**

The Helm chart sets `statement_timeout` to `10000` ms (10 seconds) at the PostgreSQL role level for `mirror_web3`. [4](#0-3) 

The `hiero.mirror.web3.db.statementTimeout = 3000` property documented in configuration docs is never read by any Java class in the web3 module and is not wired into the HikariCP `connectionInitSql` or any JDBC property — there is no web3-specific `DBProperties` class that applies it. The `CommonConfiguration` only binds `spring.datasource.hikari.*`. [5](#0-4) 

**Exploit flow:**

1. Attacker sends a contract-call request (`POST /api/v1/contracts/call`) targeting a contract whose EVM execution triggers a single expensive Hibernate query (e.g., a historical-block lookup or a large contract-state scan).
2. The request processes quickly for the first ~9.9 seconds (many small queries pass the `StatementInspector` check).
3. At `elapsed ≈ 9 999 ms`, one more query is submitted. `elapsed < 10 000`, so `StatementInspector` allows it.
4. That query runs for up to 10 more seconds (PostgreSQL `statement_timeout = 10 000 ms` per-statement).
5. The HikariCP connection is held for up to ~20 seconds total for this single request.
6. HikariCP's default maximum pool size is 10 connections (no web3-specific override found in the codebase).
7. With 10 concurrent such requests, all connections are occupied. New requests block waiting for a connection until `connectionTimeout` (HikariCP default: 30 seconds), then fail with a pool-exhaustion error.
8. The attacker sustains the DoS by continuously sending new requests as old ones complete.

**Why existing checks are insufficient:**

- The `StatementInspector` is a pre-execution gate only; it cannot cancel in-flight JDBC statements.
- The gas throttle (`gasPerSecond = 1 500 000 000`) limits compute throughput but does not limit the number of concurrent slow requests.
- The RPS throttle (`requestsPerSecond = 500`) allows far more than 10 concurrent requests.
- There is no concurrency limiter (semaphore, bulkhead) on the number of simultaneous contract-call requests.
- The `QueryTimeoutException` handler returns HTTP 503, but only after the connection is already released — it does not prevent pool exhaustion during the query's execution. [6](#0-5) 

### Impact Explanation
Complete unavailability of the web3 JSON-RPC endpoint for all users. Any request requiring a database connection (every contract call, every `eth_estimateGas`) will fail with a connection-pool-exhaustion error. Because the web3 API is the EVM-compatibility layer for the Hiero mirror node, this affects all dApps and tooling relying on `eth_call` and `eth_estimateGas`. The attacker needs no credentials, no tokens, and no special network position.

### Likelihood Explanation
The attack requires only HTTP access to the public web3 endpoint and knowledge of a contract whose execution path triggers a slow query (discoverable by timing responses). The attacker needs to maintain only ~10 concurrent connections — trivially achievable from a single machine. The attack is repeatable and self-sustaining. No authentication, no gas payment, and no privileged role is required.

### Recommendation
1. **Cancel in-flight queries**: Use `Statement.setQueryTimeout()` or pass a `java.sql.Connection`-level timeout so PostgreSQL receives a `SET LOCAL statement_timeout` before each query, bounding the per-statement wall time to `requestTimeout - elapsed`.
2. **Apply `connectionInitSql`**: Wire `hiero.mirror.web3.db.statementTimeout` into HikariCP's `connectionInitSql` as `SET statement_timeout = 3000` so the 3-second limit documented in configuration is actually enforced at the JDBC level.
3. **Add a concurrency bulkhead**: Use a `Semaphore` or Resilience4j `Bulkhead` to cap the number of simultaneously executing contract-call requests to a value safely below the connection pool size.
4. **Reduce the HikariCP pool size mismatch**: Ensure the pool size is explicitly configured and that the concurrency limit is set to a value strictly less than the pool size, leaving headroom for non-contract-call queries.

### Proof of Concept

```bash
# 1. Identify a contract whose eth_call triggers a slow historical-state query
#    (e.g., a contract with large storage at a historical block).

# 2. Send 15 concurrent requests (exceeds default pool of 10):
for i in $(seq 1 15); do
  curl -s -X POST http://<mirror-node>:8545 \
    -H 'Content-Type: application/json' \
    -d '{
      "jsonrpc":"2.0","method":"eth_call",
      "params":[{
        "to":"0x<slow_contract>",
        "data":"0x<selector_triggering_large_scan>",
        "gas":"0xE4E1C0"
      },"0x<historical_block>"],
      "id":'"$i"'
    }' &
done
wait

# 3. While the above are in flight, send a normal request:
curl -s -X POST http://<mirror-node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":99}'

# Expected result: the normal request hangs or returns a 503/connection-pool-exhaustion
# error while the slow requests hold all connections.
```

```java
    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
```
