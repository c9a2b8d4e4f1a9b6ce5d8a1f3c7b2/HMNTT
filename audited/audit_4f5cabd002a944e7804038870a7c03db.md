### Title
Cache Pollution via Dual `evmAddress` Format in `CachedPreparsedDocumentProvider`

### Summary
`CachedPreparsedDocumentProvider` keys its parse cache on the raw query string (`executionInput.getQuery()`), with no normalization. Because the `AccountInput.evmAddress` schema field accepts both `0x`-prefixed and bare hex forms, an unauthenticated attacker can submit semantically identical queries that differ only in the `0x` prefix, producing two distinct cache entries per address. Combined with the ability to embed the address as an inline literal (rather than a variable), an attacker can continuously cycle through addresses to keep the bounded 1,000-entry cache filled with useless entries, evicting legitimate cached documents and forcing re-parsing on every real request.

### Finding Description

**Cache key — no normalization:**
`CachedPreparsedDocumentProvider.getDocumentAsync` uses `executionInput.getQuery()` as the cache key verbatim. [1](#0-0) 

**Schema accepts both forms:**
The `evmAddress` field in `AccountInput` is declared with `@Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")`, so both `abcdef...` and `0xabcdef...` pass validation. [2](#0-1) 

**Normalization happens too late:**
`GraphQlUtils.decodeEvmAddress` strips the `0x` prefix before decoding, but this runs at *execution* time, after the cache lookup has already occurred with the raw string key. [3](#0-2) 

**Cache is small and bounded:**
The default configuration is `maximumSize=1000,expireAfterWrite=1h`. With only 1,000 slots and a 1-hour TTL, filling the cache with attacker-controlled entries is trivial. [4](#0-3) 

**Exploit flow:**
1. Attacker sends inline-argument queries (not variables) with 500 unique addresses, each in both forms:
   - `{"query":"{ account(input:{evmAddress:\"aabbcc...\"}) { id } }"}`
   - `{"query":"{ account(input:{evmAddress:\"0xaabbcc...\"}) { id } }"}`
2. This fills all 1,000 cache slots with attacker-controlled entries.
3. Caffeine's eviction policy removes legitimate entries.
4. Every subsequent legitimate query (using variables or a fixed query string) misses the cache and triggers a full re-parse + re-validate cycle.
5. Attacker repeats continuously to sustain cache thrashing.

No authentication or special privilege is required. There is no rate limiting on the GraphQL endpoint — only complexity (`MaxQueryComplexityInstrumentation(200)`) and depth (`MaxQueryDepthInstrumentation(10)`) limits, neither of which prevents this attack. [5](#0-4) 

### Impact Explanation
The `CachedPreparsedDocumentProvider` exists specifically to avoid repeated parsing and validation of the same query structure. Cache thrashing forces every legitimate request to re-parse and re-validate its document, increasing CPU consumption proportionally to request rate. Under sustained attack, this degrades throughput and increases latency for all users of the GraphQL API. The impact is a partial DoS (performance degradation), not a complete service outage, because parsing is bounded by the `maxCharacters=10000`/`maxTokens=1000` limits. [6](#0-5) 

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public GraphQL schema, and only ~1,000 HTTP POST requests to saturate the cache. The `0x` prefix doubles the attacker's cache-slot yield per address. The attack is trivially scriptable, repeatable, and can be sustained indefinitely. The GraphQL endpoint is publicly documented. [7](#0-6) 

### Recommendation
1. **Normalize the cache key**: Strip the `0x` prefix and lowercase the hex string before using it as the cache key in `CachedPreparsedDocumentProvider`, or apply a canonical query-string normalization step.
2. **Prefer variables over inline literals**: Document and enforce (via instrumentation or middleware) that clients should use GraphQL variables for dynamic values. Inline literals produce a unique cache key per value, defeating the cache entirely.
3. **Add rate limiting**: Introduce per-IP or per-connection request rate limiting at the HTTP layer or via a Spring `HandlerInterceptor` to limit the rate at which an attacker can cycle through unique query strings.
4. **Increase cache size or reduce TTL**: A larger cache (e.g., `maximumSize=10000`) raises the cost of full cache eviction; a shorter TTL reduces the window of pollution.

### Proof of Concept
```bash
# Fill the cache with 500 address pairs (1000 entries total)
for i in $(seq 1 500); do
  ADDR=$(printf '%040x' $i)
  # Without prefix
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{evmAddress:\\\"${ADDR}\\\"}) { id } }\"}" &
  # With 0x prefix — semantically identical, different cache key
  curl -s -X POST http://localhost:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{evmAddress:\\\"0x${ADDR}\\\"}) { id } }\"}" &
done
wait

# Now send a legitimate query — it will miss the cache and be re-parsed
curl -X POST http://localhost:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input:{entityId:{shard:0,realm:0,num:2}}) { balance } }"}'
```
Repeat the fill loop continuously to sustain cache thrashing. Monitor re-parse rate via the `recordStats` metrics exposed by Caffeine (`hitRate` will drop toward 0).
