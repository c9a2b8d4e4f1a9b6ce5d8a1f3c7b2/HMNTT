### Title
Unbounded Concurrent gRPC Subscription Exhaustion via `subscribeTopic` with `limit=1` and No End Time

### Summary
The `subscribeTopic` RPC in `ConsensusController` accepts unauthenticated `ConsensusTopicQuery` requests with no global cap on concurrent subscriptions. A `limit=1` query with `consensusStartTime=now` and no `consensusEndTime` creates a reactive stream that waits indefinitely for a single message, holding server-side resources (scheduler threads, DB polling loops, memory) for each open subscription. Because the only per-connection guard (`maxConcurrentCallsPerConnection=5`) does not limit the number of TCP connections, an attacker can trivially multiply this across thousands of connections to exhaust the server.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` — `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, lines 43–53 — accepts any `ConsensusTopicQuery` with no authentication and no subscription-count gate:

```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
    if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
        serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
    }
}
```

**Root cause 1 — no global subscription limit:**

`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge metric (`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 48–55, 89–90). It is never checked before accepting a new subscription. Any number of concurrent subscriptions is accepted.

**Root cause 2 — indefinite stream lifetime with `limit=1` and no `endTime`:**

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92):
- `pastEndTime()` returns `Flux.never()` when `endTime == null` (lines 123–125), so the stream never terminates due to time.
- `flux.take(filter.getLimit())` (line 84) only completes after receiving 1 message; if no message arrives, the stream stays open forever.
- The `safetyCheck` flux (lines 67–70) schedules work on `Schedulers.boundedElastic()` for every open subscription.
- `PollingTopicListener.listen()` polls the database every 500 ms per subscription via `RepeatSpec.times(Long.MAX_VALUE)` (lines 38–48 of `PollingTopicListener.java`).

**Root cause 3 — per-connection limit is trivially bypassed:**

`GrpcConfiguration` (lines 28–35) only sets `maxConcurrentCallsPerConnection` on the Netty builder. `NettyProperties` defaults this to `5` (line 14). No `maxConnections`, `maxConnectionAge`, or global stream limit is configured. An attacker opens N TCP connections × 5 streams each = 5N concurrent indefinite subscriptions.

**`TopicMessageFilter` validation does not block the attack:**

`isValidStartTime()` (line 49–51 of `TopicMessageFilter.java`) only requires `startTime <= DomainUtils.now()`. Setting `startTime` to the current time is valid and causes the historical retriever to return zero messages, immediately transitioning to the live (indefinitely-waiting) phase.

**Nginx partial mitigation is insufficient:**

`docker-compose.yml` line 226 sets `grpc_read_timeout 600s`. This only applies when nginx is deployed in front, and an attacker can open new connections faster than old ones expire (600 s × N connections still drains resources), or can send a single message to the topic every ~599 s to reset the timeout on all subscriptions simultaneously.

### Impact Explanation

Each open subscription consumes:
- A `TopicContext` object and reactive Flux chain in heap memory
- A `boundedElastic` scheduler thread slot (safety check)
- A DB polling loop firing every 500 ms (PollingTopicListener), consuming R2DBC connection pool slots
- A gRPC server thread for stream management

With the default `boundedElastic` scheduler cap (10 × CPU cores + 100,000 task queue), and the R2DBC connection pool being finite, an attacker with enough TCP connections can exhaust the thread pool, starve the DB connection pool, or cause OOM — preventing legitimate subscribers from receiving topic messages and degrading or halting the mirror node's HCS streaming service entirely.

### Likelihood Explanation

No authentication or API key is required. The gRPC port (5600) is publicly exposed. Opening thousands of TCP connections with 5 streams each is achievable from a single machine with standard gRPC client libraries (e.g., `grpc-java`, `grpcurl` in a loop). The attack is repeatable and requires no special knowledge beyond the public proto definition. The `limit=1` + `startTime=now` pattern is a valid, well-formed request that passes all server-side validation.

### Recommendation

1. **Enforce a global concurrent-subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection rate limiting** at the gRPC layer (e.g., via a gRPC `ServerInterceptor` or at the load balancer).
3. **Set `maxConnectionAge` and `maxConnections`** on the Netty server builder in `GrpcConfiguration` to bound the total number of TCP connections the server will accept.
4. **Require a non-null `endTime`** or enforce a maximum subscription duration (e.g., 10 minutes) for any subscription with `limit > 0`, so streams cannot be held open indefinitely.
5. **Add `maxConcurrentCallsPerConnection` to the global stream count**, not just per-connection.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hapi.proto import basic_types_pb2, timestamp_pb2
import time

TARGET = "mirror-node-grpc:5600"
TOPIC_SHARD, TOPIC_REALM, TOPIC_NUM = 0, 0, 1  # any valid topic
NUM_CONNECTIONS = 2000
STREAMS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection default

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now = int(time.time())
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(shardNum=TOPIC_SHARD, realmNum=TOPIC_REALM, topicNum=TOPIC_NUM),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=now, nanos=0),
        limit=1  # waits indefinitely for 1 message that never arrives
        # no consensusEndTime set
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        # iter() keeps the stream open; never consumed
        streams.append(stub.subscribeTopic(query))
    # Hold streams open
    time.sleep(3600)

threads = [threading.Thread(target=open_subscriptions, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# 2000 connections × 5 streams = 10,000 indefinite subscriptions
# Each polls DB every 500ms → 20,000 DB queries/second
# boundedElastic scheduler and R2DBC pool exhausted; legitimate subscribers rejected
```
