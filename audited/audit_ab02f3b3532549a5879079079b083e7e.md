### Title
Unbounded `entity` Table Scan with Correlated Token Subqueries in `getBalances()` Enables Unauthenticated DB Resource Exhaustion

### Summary
In `rest/balances.js`, the `getBalances()` function's no-timestamp code path builds an `entity_balance` CTE that always performs a full scan of the `entity` table before applying any account.id range filter. Combined with up to 100 correlated `token_account` subqueries per response and the complete absence of rate limiting on the REST API, any unauthenticated caller can repeatedly issue maximally expensive queries, exhausting database CPU and I/O.

### Finding Description

**Exact code path — `rest/balances.js` lines 133–149:**

When no `timestamp` query parameter is present, `getBalances()` falls into the `else` branch:

```js
} else {
  const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
  const whereClause = conditions && `where ${conditions}`;
  const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);
  sqlParams = utils.mergeParams(tsParams, accountParams, pubKeyParams, balanceParams, params);
  sqlQuery = `
    with entity_balance as (
      select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
      from entity
      where type in ('ACCOUNT', 'CONTRACT')   -- ← no account.id filter here
    )
    select ab.*, (${tokenBalanceSubQuery}) as token_balances
    from entity_balance ab
    ${whereClause}                             -- ← account.id range applied AFTER CTE
    order by ab.account_id ${order}
    ${limitQuery}`;
}
```

The `entity_balance` CTE contains only `WHERE type IN ('ACCOUNT', 'CONTRACT')`. The caller-supplied `account.id=gte:X&account.id=lte:Y` range is placed in the outer `${whereClause}`, not inside the CTE. Whether PostgreSQL inlines the CTE (v12+ default) or materialises it, a range spanning `0.0.1`–`0.0.999999999` covers virtually every encoded entity ID and forces a near-full index/table scan of the `entity` table.

**Correlated subquery — lines 308–318:**

```js
const getTokenAccountBalanceSubQuery = (order) => {
  return `
    select json_agg(...)
    from (
      select token_id, balance
      from token_account ta
      where ta.account_id = ab.account_id and ta.associated is true
      order by token_id ${order}
      limit ${tokenBalanceLimit.multipleAccounts}   -- 50 rows per account
    ) as account_token_balance`;
};
```

This subquery executes once per row returned by the outer query. With `limit=100` (the configured maximum), up to **100 × 50 = 5 000** `token_account` index lookups are issued per single HTTP request.

**No rate limiting — `rest/server.js` lines 67–106:**

The Express middleware stack contains logging, auth, metrics, and optional response caching — but **no rate-limiting middleware** for the `/balances` route or any other REST endpoint. The `ThrottleManager` found in the codebase applies only to the web3/contract-call service.

**Root cause and failed assumption:** The code assumes that the `LIMIT` clause is sufficient to bound per-request database cost. It is not: the CTE scan cost is proportional to the size of the `entity` table, not to the number of rows ultimately returned, and the correlated subquery multiplies that cost. No compensating control (rate limiting, query timeout surfaced to the caller, mandatory timestamp filter) exists.

### Impact Explanation

An attacker sends repeated requests of the form:

```
GET /api/v1/balances?account.id=gte:0.0.1&account.id=lte:0.0.999999999&limit=100
```

Each request forces:
1. A near-full sequential or index scan of the `entity` table (potentially millions of rows on a mainnet mirror node).
2. Up to 5 000 correlated index lookups against `token_account`.

With no rate limiting, a single attacker with modest bandwidth can saturate the PostgreSQL connection pool and CPU, degrading or denying service to all other API consumers. There is no economic damage to the Hedera network itself, but the mirror node operator's infrastructure is impacted — consistent with the stated Medium / griefing scope.

### Likelihood Explanation

The exploit requires zero authentication, zero special knowledge, and a single HTTP GET request. The parameters are valid, accepted values that pass all existing input validation. The attack is trivially scriptable (`while true; do curl ...; done`) and repeatable at any rate the attacker's network allows. Any public mirror node deployment is exposed.

### Recommendation

1. **Move the account.id filter inside the CTE** (or use a plain `SELECT` without a CTE) so the database can use the `entity(id)` index to satisfy the range predicate before scanning:
   ```sql
   select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
   from entity
   where type in ('ACCOUNT', 'CONTRACT')
     and id >= $lower and id <= $upper   -- push filter inside
   ```
2. **Add rate limiting** to the REST API (e.g., `express-rate-limit`) keyed on IP address, applied globally or specifically to expensive endpoints such as `/balances`.
3. **Enforce a DB statement timeout** for the REST pool (the config supports `statementTimeout`; verify it is set to a low value such as 5–10 s) so runaway queries are killed server-side.
4. **Consider requiring a timestamp filter** for the `/balances` endpoint, or at minimum returning a 400 when no narrowing filter is provided, consistent with the documented recommendation to use `/api/v1/accounts/{id}/tokens` instead.

### Proof of Concept

**Preconditions:** Network access to a mirror node REST API. No credentials required.

**Trigger:**
```bash
# Single maximally expensive request
curl "https://<mirror-node>/api/v1/balances?account.id=gte:0.0.1&account.id=lte:0.0.999999999&limit=100"

# Sustained attack (no rate limiting stops this)
while true; do
  curl -s -o /dev/null \
    "https://<mirror-node>/api/v1/balances?account.id=gte:0.0.1&account.id=lte:0.0.999999999&limit=100" &
done
```

**Result:** Each request executes a near-full `entity` table scan plus up to 5 000 `token_account` index lookups. Concurrent requests exhaust the PostgreSQL connection pool and CPU, causing elevated latency or 503 errors for all other API consumers.
