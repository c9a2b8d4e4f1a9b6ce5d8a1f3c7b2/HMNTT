### Title
Unauthenticated High-Complexity Query Replay via Preparsed Document Cache Enables DB Connection Pool Exhaustion

### Summary
The `graphQlCustomizer()` method in `GraphQlConfiguration.java` registers a `CachedPreparsedDocumentProvider` that caches parsed and validated query documents for up to one hour, keyed solely by query string. Because `MaxQueryComplexityInstrumentation` runs as an execution-phase instrumentation (after the cache lookup), it does not prevent cached queries from executing — it only enforces the limit per execution. With no rate limiting and no authentication on the GraphQL endpoint, an unprivileged attacker can pre-populate the cache with a complexity=199 query and then replay it at maximum throughput, bypassing all per-request parsing/validation CPU cost and driving sustained DB connection acquisition at the highest possible rate.

### Finding Description

**Code path:**

- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 42–48: `graphQlCustomizer()` wires both the `ChainedInstrumentation` (containing `MaxQueryComplexityInstrumentation(200)`) and the `preparsedDocumentProvider(provider)` onto the same `GraphQL` instance.
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, lines 24–27: `getDocumentAsync()` returns a cached `PreparsedDocumentEntry` on cache hit, skipping the `parseAndValidateFunction` entirely.

**Root cause — failed assumption:**

The design assumes that `MaxQueryComplexityInstrumentation` acts as a gate that prevents expensive queries from reaching the database. However, in graphql-java the `PreparsedDocumentProvider` is consulted *before* instrumentation runs. On a cache hit, parsing and validation (including any validation-phase complexity analysis) are skipped entirely; the instrumentation only fires during the execution phase, after the document is already accepted. A complexity=199 query (one below the hard limit of 200) is therefore always allowed to proceed to execution, both on first submission and on every subsequent cache-hit replay.

**Exploit flow:**

1. Attacker crafts a query with complexity exactly 199 (e.g., deeply nested account/token/transaction fields up to depth 10, maximising resolver fan-out).
2. First request: query is parsed, validated, complexity-checked (passes at 199 < 200), and stored in the Caffeine `AsyncCache` with a 1-hour TTL.
3. Subsequent requests with the identical query string: `CachedPreparsedDocumentProvider.getDocumentAsync()` returns the cached entry immediately; parsing and validation are skipped entirely.
4. The execution-phase `MaxQueryComplexityInstrumentation` still fires and still passes (199 < 200), so execution proceeds to the data-fetching layer and the DB connection pool.
5. Because there is no rate limiting (`requestsPerSecond`, Bucket4j, Resilience4j — none present in the `graphql/` module) and no authentication, the attacker can sustain this at the maximum rate the HTTP stack will accept.

**Why existing checks are insufficient:**

- `MaxQueryComplexityInstrumentation(200)`: Does not block complexity=199; does not run before DB access; does not limit request rate.
- `MaxQueryDepthInstrumentation(10)`: Limits structural depth but does not limit the number of concurrent or sequential executions.
- Parser limits (`maxCharacters=10000`, `maxTokens=1000`): Only apply during the parse phase, which is bypassed on cache hits.
- Jackson limits: Apply to JSON deserialization of the HTTP body, not to GraphQL execution.
- Cache `maximumSize=1000`: Limits cache entries, not request rate; a single cached entry is sufficient for the attack.

### Impact Explanation

Each replayed complexity=199 execution drives multiple DB queries through the connection pool. With no rate limiting, an attacker from a single host can saturate the GraphQL service's DB connection pool, causing connection timeouts and query failures for all legitimate users. The 1-hour cache TTL means the attacker needs only one successful initial request to sustain the attack for an extended window. During a network partition where the DB is degraded, connection acquisition attempts pile up in the pool queue, accelerating resource exhaustion and potentially causing OOM or thread starvation in the application tier.

### Likelihood Explanation

The endpoint is unauthenticated and publicly reachable (confirmed: no `SecurityConfig`, no `@PreAuthorize`, no authentication filter in the `graphql/` module). The attack requires zero privileges, no special tooling beyond `curl` or any HTTP load tool, and only one successful warm-up request. The query structure needed to reach complexity=199 is straightforward to derive from the public GraphQL schema. The attack is fully repeatable and automatable.

### Recommendation

1. **Add per-IP or per-connection rate limiting** at the GraphQL endpoint (e.g., Bucket4j, Spring Cloud Gateway rate filter, or a servlet filter) to cap requests per second regardless of cache state.
2. **Lower the complexity ceiling** from 200 to a value that reflects the minimum necessary for legitimate use cases, reducing the maximum DB fan-out per request.
3. **Consider moving complexity enforcement into the validation phase** (via a custom `ValidationRule`) so that it runs inside `parseAndValidateFunction` and is therefore cached alongside the document — meaning a query that passes complexity is cached, but the check is not re-run on every execution. This does not fix rate limiting but removes the execution-phase re-check overhead.
4. **Add authentication** to the GraphQL endpoint to restrict access to known clients, raising the bar for unauthenticated DoS.
