### Title
Unbounded Historical Scan via `timestamp=lte:X` on `/contracts/:contractId/state` Causes Full Cross-Partition Table Scan

### Summary
The `getContractStateByIdAndFilters()` function in `rest/service/contractService.js` switches to `contractStateTimestampQuery` when a timestamp filter is present. This query uses `DISTINCT ON (slot)` over the `contract_state_change` table — which is partitioned by `consensus_timestamp` — with no lower-bound constraint and no `bindTimestampRange` guard. Any unauthenticated caller can supply `timestamp=lte:9999999999` to force PostgreSQL to scan every partition of `contract_state_change` for the target contract, merge and sort all historical rows, and apply `DISTINCT ON` before the `LIMIT` clause takes effect, causing sustained DB I/O and CPU spikes.

### Finding Description

**Exact code path:**

`contractStateTimestampQuery` is defined at: [1](#0-0) 

It executes:
```sql
SELECT DISTINCT ON (slot)
  contract_id, slot, evm_address,
  coalesce(value_written, value_read) as value,
  consensus_timestamp as modified_timestamp
FROM contract_state_change csc
LEFT JOIN entity e ON e.id = csc.contract_id
WHERE contract_id = $1 AND consensus_timestamp <= $2
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT $3
```

This query is chosen in `getContractStateByIdAndFilters()` whenever `timestamp = true`: [2](#0-1) 

The `timestamp` flag is set in `extractContractStateByIdQuery()` for any `lte`, `lt`, `gte`, `gt`, or `eq` (rewritten to `lte`) operator on the timestamp filter key, with no upper-bound enforcement: [3](#0-2) 

The handler `getContractStateById` calls these two functions directly with no `bindTimestampRange` guard: [4](#0-3) 

Compare this to transaction-related endpoints that import and apply `bindTimestampRange`: [5](#0-4) 
(imported but never called in the state handler path).

**Root cause — partitioned table + DISTINCT ON + no range bound:**

`contract_state_change` is declared `PARTITION BY RANGE (consensus_timestamp)`: [6](#0-5) 

The only relevant index is `(contract_id, slot, consensus_timestamp)`: [7](#0-6) 

When `consensus_timestamp <= 9999999999000000000` (the maximum the pattern allows), PostgreSQL must:
1. Scan **all** partitions whose range overlaps `[0, 9999999999000000000]` — effectively every partition ever created.
2. Merge results across partitions ordered by `(slot ASC, consensus_timestamp DESC)`.
3. Apply `DISTINCT ON (slot)` to the merged stream — this requires the full sort to complete before any row is emitted.
4. Only then apply `LIMIT`.

The `LIMIT` clause does **not** short-circuit the cross-partition merge and sort required by `DISTINCT ON`.

**Timestamp pattern allows the maximum value:**

The `stateTimestampQueryParam` schema pattern is `\d{1,10}(\.\d{1,9})?`, permitting seconds up to `9999999999` (year 2286): [8](#0-7) 

The JS-side validation mirrors this: [9](#0-8) 

### Impact Explanation

A single request with `timestamp=lte:9999999999` against a heavily-used contract (e.g., a DEX or stablecoin with thousands of storage slots and years of history) forces a full cross-partition sequential scan of `contract_state_change`, a cross-partition sort, and a `DISTINCT ON` aggregation — all before the `LIMIT 25` (default) takes effect. Repeated requests from one or more unauthenticated clients can saturate DB I/O and CPU, degrading or denying service to all other API consumers. The impact scales with the age and activity of the targeted contract.

### Likelihood Explanation

No authentication is required. The endpoint is publicly documented in the OpenAPI spec. The exploit requires only knowledge of a high-activity contract ID (trivially discoverable from block explorers) and a single HTTP GET request. It is trivially scriptable and repeatable at high frequency. There is no per-IP rate limiting visible in the REST layer for this endpoint.

### Recommendation

1. **Apply `bindTimestampRange` (or an equivalent upper-bound cap) to the state endpoint.** When a timestamp filter is present, enforce a maximum window (e.g., cap `lte` values at `now()` or require a lower bound within a configurable range).
2. **Require a lower-bound timestamp when an upper-bound is supplied**, mirroring the `allowOpenRange = false` enforcement used on other endpoints.
3. **Add a DB-level statement timeout** for queries on `contract_state_change` with open-ended timestamp ranges.
4. **Consider rewriting the query** to use a correlated subquery or lateral join per slot rather than a global `DISTINCT ON` across all partitions, so `LIMIT` can prune work early.

### Proof of Concept

```
# Identify a high-activity contract (e.g., from a public block explorer)
CONTRACT_ID="0.0.1234567"

# Single request — forces full historical scan across all partitions
curl "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}/state?timestamp=lte:9999999999"

# Sustained load — repeat in a loop from a single unauthenticated client
while true; do
  curl -s "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}/state?timestamp=lte:9999999999" &
done
```

Preconditions: target contract must exist and have a non-trivial history in `contract_state_change`. No credentials required. Observable result: DB CPU and I/O rise sharply; query latency for all other endpoints increases.
