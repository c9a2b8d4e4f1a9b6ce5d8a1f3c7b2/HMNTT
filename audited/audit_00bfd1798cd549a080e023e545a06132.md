### Title
Indefinite Bytecode Cache Staleness via Unprivileged Access-Based Timer Reset in `findRuntimeBytecode()`

### Summary
The `findRuntimeBytecode()` method in `ContractRepository` uses a Caffeine cache configured with `expireAfterAccess=1h` and no write-based expiry or eviction mechanism. Because `expireAfterAccess` resets the TTL on every read, any unprivileged caller can prevent the cache from ever expiring by repeatedly invoking `eth_call` or `eth_estimateGas` against the mirror node's JSON-RPC endpoint. After a legitimate on-chain contract upgrade updates `runtime_bytecode` in the database, the web3 module will continue serving the pre-upgrade bytecode to all EVM simulation consumers indefinitely.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

The cache manager is wired in `web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java` lines 67–73:
```java
@Bean(CACHE_MANAGER_CONTRACT)
CacheManager cacheManagerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getContract());
    return caffeineCacheManager;
}
```

The default specification is set in `web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 22:
```java
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```

**Root cause:** `expireAfterAccess` measures idle time since the last read. Every call to `findRuntimeBytecode()` — including unauthenticated `eth_call` / `eth_estimateGas` requests that flow through `ContractBytecodeReadableKVState.readFromDataSource()` — resets the 1-hour clock. There is no `expireAfterWrite` bound, no `@CacheEvict`, and no `@CachePut` anywhere in the web3 module for this cache (confirmed by exhaustive search: the only `@CacheEvict` usages in the codebase are in the importer's `AddressBookServiceImpl` and `RecordFileRepository`, neither of which touches the web3 contract cache).

**Why existing checks fail:** The `unless = "#result == null"` guard only prevents caching empty results; it does nothing to bound staleness or trigger re-reads after an upgrade. The `ContractRepositoryTest.findRuntimeBytecodeSuccessfulCall()` test (lines 25–29) explicitly demonstrates and accepts this behavior: after `deleteAll()` the cache still returns the old bytecode — confirming the design has no invalidation path.

**Exploit flow:**
1. Contract `0.0.X` is deployed; its `runtime_bytecode` is cached by the first `eth_call`.
2. The contract owner submits a `ContractUpdateTransaction` on Hedera; the importer writes the new `runtime_bytecode` to the `contract` table.
3. The attacker (no credentials required) sends `eth_call` requests targeting `0.0.X` at any interval shorter than 1 hour (e.g., every 50 minutes via a simple cron script).
4. Each request hits `ContractBytecodeReadableKVState.readFromDataSource()` → `contractRepository.findRuntimeBytecode()` → Caffeine cache hit → timer reset.
5. The cache never expires; the DB is never re-queried; all callers receive pre-upgrade bytecode indefinitely.

### Impact Explanation
All consumers of the mirror node's EVM simulation endpoints (`eth_call`, `eth_estimateGas`, opcode tracing) receive results computed against the pre-upgrade bytecode. This can cause:
- Incorrect return values from `eth_call` simulations, misleading dApps about post-upgrade contract behavior.
- Wrong gas estimates, causing submitted transactions to fail with out-of-gas errors or to over-pay.
- Security-sensitive dApps that rely on simulation to validate logic (e.g., checking access control or token balances via a proxy) may act on stale results.

The mirror node is a read-only simulation service, so on-chain state is not directly altered, but the integrity of every EVM simulation served by the node is compromised for the targeted contract for as long as the attacker sustains the polling.

### Likelihood Explanation
The attack requires zero privileges: only the ability to send HTTP requests to the publicly exposed JSON-RPC endpoint. The polling rate needed (one request per hour per contract) is trivially low and indistinguishable from normal traffic. The attacker needs only to know the contract ID of a recently upgraded contract, which is public information on Hedera. The attack is fully repeatable and can target multiple contracts simultaneously within the `maximumSize=1000` cache limit.

### Recommendation
1. **Replace `expireAfterAccess` with `expireAfterWrite`** for the contract bytecode cache. `expireAfterWrite=1h` (or a shorter interval) bounds the maximum staleness regardless of access patterns and cannot be extended by external callers.
2. **Add `@CacheEvict`** on any importer-side path that writes a new `runtime_bytecode` (e.g., `ContractCreateTransactionHandler.createContract()`, `SidecarContractMigration.migrate()`), or publish an eviction event that the web3 module consumes.
3. Consider a short `expireAfterWrite` (e.g., 60 s) combined with a larger `expireAfterAccess` window, so frequently-accessed contracts stay warm but are guaranteed to refresh after upgrades.
