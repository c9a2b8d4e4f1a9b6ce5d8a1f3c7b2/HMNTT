### Title
Silent Metrics Corruption via Non-Numeric `Content-Length` Header in `metricsMiddleware`

### Summary
In `rest/middleware/metricsHandler.js`, the `metricsMiddleware` function reads the raw `content-length` request header and parses it with `parseInt(..., 10)`. When a non-numeric string such as `'abc'` is supplied, `parseInt` returns `NaN`; the `|| 0` fallback silently coerces this to `0`, causing `requestSizeHistogram.record(0, labels)` to be called regardless of the actual body size. Any unprivileged external caller can trigger this on every request, permanently skewing the `api_request_size_bytes` histogram toward zero.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, `metricsMiddleware()` closure, line 209.

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** The `?? '0'` null-coalescing guard only handles `null`/`undefined` (i.e., a missing header). It does not guard against a header that is present but non-numeric. When `req.headers['content-length']` is `'abc'`:

1. `'abc' ?? '0'` â†’ `'abc'` (header is present, not null/undefined)
2. `parseInt('abc', 10)` â†’ `NaN`
3. `NaN || 0` â†’ `0`
4. `requestSizeHistogram.record(0, labels)` is called â€” recording zero bytes for a request that may have a real body.

**Why existing checks fail:** The only guard is `?? '0'`, which is bypassed by any non-null, non-undefined string value. There is no `isNaN()` check, no `Number.isFinite()` check, and no schema-level validation of the header value before it reaches this line.

**Upstream guards:** `server.js` registers the metrics middleware after `express.json()` and `express.urlencoded()`, but neither of those body parsers validates or normalises the raw `content-length` header string stored in `req.headers`. Node.js's llhttp parser rejects truly malformed Content-Length in direct HTTP/1.1 connections, but in common production deployments behind a reverse proxy (nginx, HAProxy, AWS ALB) that forwards or rewrites headers, a crafted non-numeric value can reach the Express layer intact. ### Impact Explanation
The corrupted metric is `api_request_size_bytes` (the `requestSizeHistogram`). Sustained injection of zero-byte observations causes the histogram's bucket distribution and computed percentiles to be permanently skewed toward zero, making the metric useless for capacity planning, anomaly detection, and SLO alerting. Because the corruption is silent (no error is logged, no exception is thrown), operators have no indication that the data is invalid. The impact is limited to observability/monitoring integrity â€” there is no effect on request processing, user data, or economic value.

### Likelihood Explanation
The precondition is trivially met: any HTTP client can set an arbitrary `Content-Length` header value. No authentication, no special role, and no rate-limit bypass is required. The attack is repeatable at will and requires a single line of curl:

```
curl -H "Content-Length: abc" https://<mirror-node-rest>/api/v1/transactions
```

In deployments behind a proxy that forwards the original header (or rewrites it to the attacker-supplied value), the attack succeeds on every request. Repeatability makes sustained metric poisoning practical.

### Recommendation
Replace the `parseInt(...) || 0` pattern with an explicit numeric validation:

```js
const rawCL = req.headers['content-length'];
const parsedCL = rawCL !== undefined ? parseInt(rawCL, 10) : 0;
const requestBytes = Number.isFinite(parsedCL) && parsedCL >= 0 ? parsedCL : 0;
requestSizeHistogram.record(requestBytes, labels);
```

Alternatively, use `Number(rawCL)` and guard with `Number.isFinite`. This mirrors the approach already used in the Java counterparts (`NumberUtils.toLong` with `Math.max(0L, ...)`) in `rest-java` and `web3` `MetricsFilter.java`. [3](#0-2) 

### Proof of Concept
**Preconditions:** Mirror Node REST service running with `config.metrics.enabled = true`; deployed behind a reverse proxy that forwards the raw `Content-Length` header (or accessed directly in a Node.js version/configuration that passes the header through).

**Steps:**

**Expected result:** The `api_request_size_bytes_bucket` counters for `le="0"` increment on every poisoned request, while higher buckets remain unchanged, regardless of the actual body size sent.
