### Title
Unauthenticated GraphQL Account Enumeration Causes Unbounded DB Load via Missing Rate Limiting and Absent `findById()` Cache

### Summary
The GraphQL module's `EntityServiceImpl.getByIdAndType()` calls `entityRepository.findById()` on every request with no caching and no rate limiting. Unlike the `grpc` and `web3` modules which both apply `@Cacheable` or bucket4j throttling respectively, the GraphQL module has neither, allowing any unauthenticated caller to drive an unbounded stream of primary-key DB lookups by cycling through incrementing `num` values.

### Finding Description

**Exact code path:**

`EntityServiceImpl.getByIdAndType()` (line 25):
```java
return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
``` [1](#0-0) 

The graphql-module `EntityRepository` extends bare `CrudRepository<Entity, Long>` with no `@Cacheable` on `findById()`: [2](#0-1) 

**Root cause — two missing controls:**

1. **No caching.** The `grpc` module's `EntityRepository` annotates `findById()` with `@Cacheable`, so repeated lookups of the same ID are served from memory: [3](#0-2) 
The graphql `EntityRepository` has no equivalent annotation, so every call — including cache-busting calls with a new `num` — hits PostgreSQL.

2. **No rate limiting.** The `web3` module has a full bucket4j `ThrottleConfiguration` with a `rateLimitBucket` enforcing `requestsPerSecond`: [4](#0-3) 
The graphql module's only filter is `LoggingFilter` — no throttle bean, no rate-limit interceptor exists anywhere under `graphql/src/main/java/`. [5](#0-4) 

**Per-query protections are insufficient for rate control:**

`GraphQlConfiguration` applies `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, plus parser/JSON size caps: [6](#0-5) 

These limits govern the *structure* of a single query, not the *rate* at which queries arrive. A minimal account query `{ account(input: {entityId: {num: N}}) { balance } }` has complexity ≈ 1–2, well under the 200 cap, so the instrumentation never fires.

**Exploit flow:**

1. Attacker opens persistent HTTP connections to `POST /graphql/alpha`.
2. Sends a pipelined stream of queries with monotonically incrementing `num`:
   ```
   {"query":"{ account(input:{entityId:{num:1}}){balance}}"}
   {"query":"{ account(input:{entityId:{num:2}}){balance}}"}
   ...
   ```
3. Each unique `num` is a guaranteed cache miss (no cache exists). `getByIdAndType()` → `entityRepository.findById(N)` → `SELECT * FROM entity WHERE id = N` executes against PostgreSQL for every request.
4. No server-side control rejects or slows the stream. HikariCP connections are acquired and held for each in-flight query; at sustained high rates the pool saturates and DB CPU climbs.
5. All queries return HTTP 200 with `data: {account: null}` — no error pattern for brute-force detection to trigger.

### Impact Explanation

Sustained high-rate enumeration forces continuous primary-key scans against the `entity` table. While each individual query is cheap, the aggregate at hundreds-to-thousands of requests per second consumes DB CPU and exhausts the HikariCP connection pool (default 10 connections in Spring Boot), degrading or blocking all other GraphQL and shared-DB consumers. The `GraphQLHighDBConnections` alert only fires after 75% pool utilization for 5 minutes — giving the attacker a wide window before any automated response. [7](#0-6) 

### Likelihood Explanation

No account, API key, or authentication is required. The GraphQL endpoint is publicly documented and reachable. The attack requires only a script that increments an integer and POSTs standard JSON — trivially automatable. Because every response is HTTP 200 and structurally valid, WAF/IDS rules tuned for error-rate anomalies will not trigger. The attack is repeatable indefinitely.

### Recommendation

1. **Add rate limiting to the graphql module.** Mirror the `web3` module's bucket4j `ThrottleConfiguration` pattern — create a `rateLimitBucket` bean and a servlet filter or `WebGraphQlInterceptor` that consumes a token per request and returns HTTP 429 when exhausted.

2. **Add `@Cacheable` to `EntityRepository.findById()` in the graphql module.** Mirror the `grpc` module's pattern so repeated lookups of the same entity ID are served from an in-process Caffeine cache rather than hitting PostgreSQL every time.

3. **Consider per-IP connection/request quotas** at the ingress/load-balancer layer as a defense-in-depth measure independent of application-level controls.

### Proof of Concept

```bash
# Requires: curl, bash
# Target: GraphQL endpoint (unauthenticated)
ENDPOINT="http://<host>:8083/graphql/alpha"

for i in $(seq 1 100000); do
  curl -s -o /dev/null -X POST "$ENDPOINT" \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{account(input:{entityId:{num:$i}}){balance}}\"}" &
  # Throttle concurrent connections to avoid local OS limits
  if (( i % 50 == 0 )); then wait; fi
done
```

Each iteration issues a structurally valid, low-complexity GraphQL query with a unique `num`, guaranteeing a cache miss and a live DB round-trip. No credentials are needed. Monitor `hikaricp_connections_active` and PostgreSQL `pg_stat_activity` to observe pool saturation and CPU rise.
