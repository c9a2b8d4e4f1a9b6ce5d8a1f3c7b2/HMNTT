### Title
Unauthenticated Subscription Flood Causes Synchronized Safety-Check DB Query Burst

### Summary
Any unauthenticated client can open an unbounded number of gRPC connections and subscriptions to `subscribeTopic`. Each subscription arms a `Mono.delay(1s)` safety-check timer that, after one second, unconditionally issues a gap-recovery database query. With no global subscription cap, an attacker opening many connections simultaneously causes all timers to fire in a synchronized burst, saturating the `boundedElastic` thread pool and the HikariCP database connection pool, degrading service for all users.

### Finding Description

**Exact code path:**

In `TopicMessageServiceImpl.subscribeTopic()` (lines 67–70), every new subscription unconditionally schedules a safety-check:

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
``` [1](#0-0) 

The `filter` guard calls `topicContext.isComplete()`, which only returns `true` when an `endTime` is configured **and** that time has already elapsed:

```java
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;
    }
    ...
}
``` [2](#0-1) 

For any open-ended subscription (no `endTime`), `isComplete()` is permanently `false`, so the guard never suppresses the query. The `flatMapMany` then calls `missingMessages(topicContext, null)`, which (lines 142–149) immediately issues a blocking database retrieval:

```java
if (current == null) {
    ...
    return topicMessageRetriever.retrieve(gapFilter, false);
}
``` [3](#0-2) 

**Root cause / failed assumption:** The design assumes the safety check fires rarely (only when the importer is down). It does not account for an attacker deliberately creating thousands of subscriptions to force thousands of simultaneous gap-recovery queries.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` (the only server-side limit) is a *per-connection* cap, not a global one. An attacker opens N connections to get N×5 concurrent subscriptions. [4](#0-3) 

- `subscriberCount` is a Micrometer gauge only — it enforces nothing. [5](#0-4) 

- `Schedulers.boundedElastic()` caps concurrent threads at `10 × CPU_count` (e.g., 40 on a 4-core host), but its task queue defaults to 100,000 entries. Excess queries queue and drain continuously, keeping the DB connection pool saturated for an extended period.
- There is no per-IP connection limit, no authentication requirement, and no global subscription quota anywhere in the gRPC module. [6](#0-5) 

### Impact Explanation
After one second, all queued safety-check queries fire simultaneously. The `boundedElastic` thread pool saturates, the HikariCP connection pool exhausts, and subsequent legitimate queries queue or time out. The mirror node's gRPC service becomes unresponsive or severely degraded for all subscribers. Because the attacker can continuously reconnect and re-arm new 1-second timers, the condition is sustainable. Severity is Medium (griefing / availability impact on the mirror node infrastructure; no on-chain economic damage).

### Likelihood Explanation
No credentials, tokens, or special network access are required — the gRPC port (5600) is publicly exposed. A single attacker machine can open hundreds of TCP connections and 5 subscriptions each with a standard gRPC client library. The attack is trivially scriptable, repeatable, and requires no knowledge of internal state.

### Recommendation
1. **Add a global (or per-IP) subscription limit** enforced atomically against `subscriberCount` before allowing a new subscription to proceed; reject with `RESOURCE_EXHAUSTED` when exceeded.
2. **Deduplicate safety-check queries per topic**: use a shared, topic-scoped `Mono` (cached with `cache(Duration)`) so N subscribers to the same topic share one safety-check query rather than issuing N independent ones.
3. **Gate the safety check on actual data absence**: only arm the timer if the historical retrieval returned zero results, eliminating the check for subscriptions that already received data.
4. **Limit connections per IP** at the Netty/load-balancer layer (e.g., `maxConnectionsPerIp` or an ingress rate-limit rule).

### Proof of Concept
```python
import grpc, threading, time
from proto import consensus_service_pb2_grpc, mirror_pb2

GRPC_HOST = "mirror-node:5600"
NUM_CONNECTIONS = 200   # 200 connections × 5 calls = 1000 subscriptions

def open_subscriptions(stub):
    for _ in range(5):   # maxConcurrentCallsPerConnection = 5
        req = mirror_pb2.ConsensusTopicQuery(
            topicID=...,
            # No consensusEndTime → isComplete() always false
        )
        try:
            for _ in stub.subscribeTopic(req):
                pass
        except Exception:
            pass

channels = [grpc.insecure_channel(GRPC_HOST) for _ in range(NUM_CONNECTIONS)]
stubs   = [consensus_service_pb2_grpc.ConsensusServiceStub(c) for c in channels]

# All 1000 subscriptions start at the same moment → all 1000 Mono.delay(1s) timers
# fire simultaneously at T+1s, issuing 1000 concurrent gap-recovery DB queries.
threads = [threading.Thread(target=open_subscriptions, args=(s,)) for s in stubs]
for t in threads: t.start()
time.sleep(2)
# Observe: DB connection pool exhausted, gRPC latency spikes for legitimate clients.
```
