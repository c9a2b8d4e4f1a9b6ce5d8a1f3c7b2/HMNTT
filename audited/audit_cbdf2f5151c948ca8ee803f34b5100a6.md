### Title
Batch DB Amplification via TTL Mismatch Between `contractStateCache` (2s) and `slotsPerContract` (5m) in `findStorageBatch()`

### Summary
`ContractStateServiceImpl.findStorageBatch()` issues a single DB query for **every slot key ever accumulated** in the per-contract `slotsPerContract` cache whenever `contractStateCache` misses. Because `contractStateCache` expires every 2 seconds while `slotsPerContract` retains keys for 5 minutes, an unprivileged user can pre-populate up to 1,500 slot keys per contract via public `eth_call` RPC, then trigger a 1,500-key batch DB query on every subsequent `findStorage()` call — repeated every 2 seconds for the full 5-minute retention window.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (lines 58–70) checks `contractStateCache` first. On any miss it calls `findStorageBatch()`. [1](#0-0) 

Inside `findStorageBatch()` (lines 85–122), the method unconditionally collects **all keys** currently in the per-contract `slotsPerContract` cache and passes them to `contractStateRepository.findStorageBatch()` in a single DB call: [2](#0-1) 

**Root cause — TTL mismatch:**

`contractStateCache` is configured with `expireAfterWrite=2s`: [3](#0-2) 

`slotsPerContract` is configured with `expireAfterAccess=5m`: [4](#0-3) 

The failed assumption is that `contractStateCache` will remain warm long enough to prevent repeated batch queries. In reality it expires every 2 seconds by design, while `slotsPerContract` accumulates keys for 5 minutes. Every time `contractStateCache` expires, the very next `findStorage()` call for any slot of a given contract re-triggers `findStorageBatch()` with the full accumulated key set.

**Exploit flow:**
1. Attacker sends `eth_call` requests for up to 1,500 distinct storage slot keys of the same contract. Each call misses `contractStateCache`, enters `findStorageBatch()`, and registers its key in `slotsPerContract` via `putIfAbsent`. [5](#0-4) 
2. After 2 seconds, all `contractStateCache` entries for that contract expire automatically (no attacker action needed).
3. A single subsequent `findStorage()` call for any slot of that contract misses `contractStateCache`, enters `findStorageBatch()`, and issues a DB query for all 1,500 accumulated keys.
4. Steps 2–3 repeat every 2 seconds for the full 5-minute `slotsPerContract` retention window — 150 amplified batch queries per contract per 5-minute window.
5. With up to 3,000 contracts in `contractSlotsCache`, the attacker can sustain this across many contracts simultaneously. [6](#0-5) 

### Impact Explanation
Each legitimate single-slot lookup is amplified into a 1,500-row batch DB query, repeated every 2 seconds per contract. With 3,000 contracts targeted, this translates to up to 4,500,000 unnecessary DB row fetches every 2 seconds. The DB becomes the bottleneck, degrading or denying service for all users of the mirror node's `eth_call` / contract-execution path. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation
The attack requires only public `eth_call` RPC access — no credentials, no special permissions, no on-chain transactions. The attacker needs to know (or enumerate) valid contract addresses and storage slot indices, both of which are publicly observable on-chain. The pattern is trivially scriptable and repeatable. Rate-limiting at the HTTP layer (if present) would slow but not prevent accumulation, since the 1,500-key population phase only needs to succeed once per 5-minute window.

### Recommendation
1. **Align TTLs**: Set `slotsPerContract` TTL ≤ `contractStateCache` TTL (e.g., both at 2s), so accumulated slot keys are discarded when their cached values expire.
2. **Cap batch size at query time**: In `findStorageBatch()`, limit `cachedSlots` to a configurable maximum (e.g., 50–100) before calling `findStorageBatch()` on the repository, discarding the oldest/least-recently-used keys beyond the cap.
3. **Evict slot keys on `contractStateCache` expiry**: Use a Caffeine removal listener on `contractStateCache` to also remove the corresponding key from `slotsPerContract`, keeping the two caches in sync.
4. **Rate-limit per-IP slot key registration**: Track how many distinct slot keys a single caller has contributed to `slotsPerContract` and reject or ignore registrations beyond a threshold.

### Proof of Concept

```
# 1. Populate 1500 slot keys for contract 0xABCD... via public eth_call
for i in $(seq 1 1500); do
  curl -s -X POST $RPC_URL -d '{
    "jsonrpc":"2.0","method":"eth_call",
    "params":[{"to":"0xABCD...","data":"0x<SLOAD slot_$i>"},"latest"],
    "id":$i}'
done

# 2. Wait for contractStateCache to expire (2 seconds)
sleep 3

# 3. Single eth_call now triggers findStorageBatch() with all 1500 keys
curl -X POST $RPC_URL -d '{
  "jsonrpc":"2.0","method":"eth_call",
  "params":[{"to":"0xABCD...","data":"0x<SLOAD slot_1>"},"latest"],
  "id":9999}'

# Observe DB query log: one SELECT ... WHERE slot IN (...1500 values...)
# Repeat step 3 every 2 seconds for 5 minutes = 150 amplified queries
```

The DB will show a single `findStorageBatch` query with 1,500 bind parameters fired on every call after the 2-second expiry, confirming the amplification. [7](#0-6)
