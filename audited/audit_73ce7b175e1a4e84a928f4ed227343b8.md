### Title
Unbounded Request Body Allocation on `/network/options` Enables Memory Exhaustion Griefing

### Summary
The `NetworkOptions()` handler in `rosetta/app/services/network_service.go` accepts a `*rTypes.NetworkRequest` body that is completely ignored after JSON decoding. No `http.MaxBytesReader` or byte-level body size limit exists anywhere in the Rosetta middleware stack, so the Go HTTP server will allocate memory proportional to the full body size for every request. An unauthenticated attacker can flood the endpoint with arbitrarily large JSON bodies to exhaust server memory.

### Finding Description
**Exact code path:**

`NetworkOptions()` discards its input entirely: [1](#0-0) 

The HTTP server is assembled in `main()` with only time-based timeouts — no byte-level body cap: [2](#0-1) 

The middleware chain is: `MetricsMiddleware → TracingMiddleware → CorsMiddleware → router`. None of these layers apply `http.MaxBytesReader`: [3](#0-2) [4](#0-3) 

A grep for `MaxBytesReader`, `LimitReader`, `maxBodySize`, or any equivalent across the entire `rosetta/` tree returns zero matches — confirmed absent.

**Root cause:** Go's `net/http` server does not impose a default body size limit. The rosetta-sdk-go controller reads and JSON-decodes the full request body before calling `NetworkOptions`. Without `http.MaxBytesReader` wrapping `r.Body`, the server allocates memory equal to the full body for every request. The `ReadTimeout` of 5 s (5 000 000 000 ns) is a *time* constraint, not a *byte* constraint: [5](#0-4) 

On a 1 Gbps link an attacker can transmit ~625 MB within the 5-second window per connection.

### Impact Explanation
An unauthenticated attacker opens many concurrent TCP connections to port 5700 and sends POST `/network/options` with a multi-megabyte JSON body on each. The server allocates heap memory for every body simultaneously. With enough concurrent connections the process OOMs or the OS kills it, taking the Rosetta API offline. No economic damage occurs to on-chain users, but the mirror-node Rosetta interface becomes unavailable — matching the "griefing, no economic damage" severity tier stated in the question.

### Likelihood Explanation
The endpoint requires zero authentication and is reachable by any network peer. The attack is trivially scriptable (`curl --data-binary @/dev/urandom`), repeatable, and requires no special knowledge. The only partial brake is the 5-second `ReadTimeout`, which limits per-connection throughput but does not bound concurrent aggregate allocation.

### Recommendation
Wrap `r.Body` with `http.MaxBytesReader` in a middleware applied before the rosetta-sdk-go router, e.g.:

```go
func BodyLimitMiddleware(maxBytes int64, next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, maxBytes)
        next.ServeHTTP(w, r)
    })
}
```

Insert it into the chain in `main.go` before `metricsMiddleware`. A reasonable limit for Rosetta requests is 1–4 MB. Additionally, expose a configurable `hiero.mirror.rosetta.http.maxBodyBytes` property consistent with the existing `Http` config struct. [5](#0-4) 

### Proof of Concept
```bash
# Generate a 200 MB JSON body with a valid outer structure but huge metadata field
python3 -c "
import sys, json
body = json.dumps({'network_identifier': {'blockchain':'Hedera','network':'mainnet'},
                   'metadata': {'x': 'A' * 200_000_000}})
sys.stdout.write(body)
" > /tmp/big_body.json

# Fire 20 concurrent requests
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/network/options \
       -H 'Content-Type: application/json' \
       --data-binary @/tmp/big_body.json &
done
wait
# Observe server RSS grow by ~4 GB; repeated rounds cause OOM
```
