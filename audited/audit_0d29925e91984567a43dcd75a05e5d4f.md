### Title
Unauthenticated Cache Stampede and Unconditionally Uncached DB Queries in `getNodes()` Enable DoS Against Address Book Availability

### Summary
The `getNodes()` gRPC endpoint in `NetworkController` is publicly accessible with no authentication. The `ADDRESS_BOOK_ENTRY_CACHE` uses Caffeine `expireAfterWrite` (2-second TTL) with Spring `@Cacheable`, which provides zero stampede protection — all concurrent threads that miss the cache simultaneously all execute the underlying DB query. Compounding this, two DB queries (`findLatestTimestamp` for address book and node stake) are completely uncached and execute on every single `getNodes` call regardless of cache state, making the DB hit rate proportional to raw request volume.

### Finding Description

**Uncached queries on every call** (`NetworkServiceImpl.java:61–65`):
```java
long addressBookTimestamp = addressBookRepository
        .findLatestTimestamp(fileId.getId())   // NEVER cached — hits DB every call
        .orElseThrow(...);
long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp()  // NEVER cached — hits DB every call
        .orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
``` [1](#0-0) 

`AddressBookRepository.findLatestTimestamp` and `NodeStakeRepository.findLatestTimestamp` carry no `@Cacheable` annotation and execute a `SELECT MAX(...)` against the DB on every invocation. [2](#0-1) [3](#0-2) 

**Cache stampede on paginated queries** (`AddressBookEntryRepository.java:16–36`):
Spring `@Cacheable` with Caffeine `expireAfterWrite` does not serialize concurrent cache misses. When the 2-second TTL expires, every thread that simultaneously calls `findByConsensusTimestampAndNodeId` with the same key proceeds to execute the DB query — there is no "single-loader" or `AsyncLoadingCache` pattern in use. [4](#0-3) 

**Cache configured with `expireAfterWrite` only** (`CacheConfiguration.java:29–32`):
```java
caffeineCacheManager.setCaffeine(Caffeine.newBuilder()
        .expireAfterWrite(addressBookProperties.getCacheExpiry())  // 2s default
        .maximumSize(addressBookProperties.getCacheSize())
        .recordStats());
```
No `refreshAfterWrite`, no `AsyncLoadingCache`, no lock-based deduplication. [5](#0-4) 

**Per-connection limit is the only guard** (`GrpcConfiguration.java:33`, `NettyProperties.java:14`):
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
```
This limits calls *per connection*, not globally. There is no `maxConnections` configured anywhere in the Netty server builder, no global call limit, and no authentication on the endpoint. [6](#0-5) [7](#0-6) 

### Impact Explanation
Each `getNodes` call unconditionally issues at least 2 `SELECT MAX(...)` queries to the DB. With N connections × 5 concurrent calls, the DB receives 10N uncached queries per request wave. At cache expiry boundaries (every 2 seconds), an additional burst of paginated queries (up to `cacheSize=50` distinct page queries) all miss simultaneously and all hit the DB. This can exhaust the DB connection pool, cause query queuing/timeouts, and make address book data unavailable to legitimate callers — directly disrupting gossip node discovery for clients relying on this endpoint.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no privileged network position. The attacker only needs to open many TCP connections to port 5600 and issue concurrent `getNodes` RPCs. The 2-second cache TTL creates a predictable, repeatable window every 2 seconds. Tooling such as `ghz` (gRPC benchmarking tool) can trivially generate thousands of concurrent calls from a single machine. The attack is fully repeatable and stateless.

### Recommendation
1. **Fix the uncached queries**: Apply `@Cacheable` (with a separate short-TTL cache) to `AddressBookRepository.findLatestTimestamp` and `NodeStakeRepository.findLatestTimestamp`, or cache the composite result at the `NetworkServiceImpl` level.
2. **Fix the stampede**: Replace the Caffeine `expireAfterWrite` cache with a `refreshAfterWrite` + `AsyncLoadingCache` pattern, or use a `LoadingCache` so only one thread reloads while others return the stale value.
3. **Add a global connection/call limit**: Configure `maxConnections` on the Netty server builder and add a global gRPC interceptor that enforces a total concurrent-call ceiling across all connections.
4. **Add rate limiting**: Introduce a per-IP or global rate limiter (e.g., a token bucket interceptor) on the `getNodes` RPC.

### Proof of Concept
```bash
# Install ghz: https://ghz.sh
# 1. Open 200 connections, 5 concurrent streams each = 1000 concurrent getNodes calls
# 2. Time the burst to repeat every 2 seconds to coincide with cache expiry

ghz --insecure \
    --proto NetworkService.proto \
    --call com.hedera.mirror.api.proto.NetworkService/getNodes \
    --data '{"fileId":{"fileNum":102}}' \
    --connections 200 \
    --concurrency 1000 \
    --duration 60s \
    <mirror-node-grpc-host>:5600

# Expected result: DB connection pool exhaustion visible in mirror node logs,
# increasing query latency/errors, and legitimate getNodes callers receiving
# errors or stale/no data.
```
