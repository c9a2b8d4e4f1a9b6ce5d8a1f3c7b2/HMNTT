### Title
Unauthenticated GET /schedules Triggers 3x DB Query Amplification with No Rate Limiting, Enabling DoS

### Summary
The `getSchedules()` handler in `rest/schedules.js` issues up to three sequential/parallel `pool.queryQuietly` calls per HTTP request (schedules, entities, signatures). The REST API's Express middleware stack contains no rate-limiting layer, and the Helm chart middleware for the REST service omits the `inFlightReq` and `rateLimit` entries present in other services. Any unauthenticated external caller can flood the endpoint and exhaust the shared database connection pool at 3× the request rate.

### Finding Description
**Exact code path:**

`rest/schedules.js`, `getSchedules()`, lines 234–282:
- Line 241: first `pool.queryQuietly(schedulesQuery, params)` — fetches schedule rows.
- Lines 261–264: `Promise.all([pool.queryQuietly(entityQuery, entityIds), pool.queryQuietly(signatureQuery, entityIds)])` — two more concurrent queries against `entity` and `transaction_signature`.

Every request that returns at least one schedule row unconditionally fires all three queries. With the default `limit` value, a single request can pull up to the configured page size of rows and still trigger all three queries.

**Root cause — no rate limiting anywhere in the REST API path:**

`rest/server.js` (lines 67–98) registers the full middleware chain: `urlencoded`, `json`, `cors`, optional `compression`, `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. There is no rate-limiter middleware at any point.

`rest/middleware/authHandler.js` (lines 15–36): when no `Authorization` header is present the handler returns immediately (`return;` at line 19) without setting any limit. Even when credentials are supplied and `userLimitLabel` is written into `httpContext` (line 33), no downstream middleware reads that label to enforce a per-request rate cap — the label is set but never consumed by a throttle.

`charts/hedera-mirror-rest/values.yaml` (lines 134–139): the Traefik middleware chain for the REST service contains only `circuitBreaker` and `retry`. Compare with `charts/hedera-mirror-rosetta/values.yaml` (lines 149–166) which adds `inFlightReq: amount: 5` and `rateLimit: average: 10`. The REST API has neither.

The `circuitBreaker` (line 135–136) only trips after `NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500,600,0,600) > 0.25`, meaning the database must already be saturated and returning errors before any protection activates.

**Exploit flow:**
1. Attacker sends a high-volume stream of `GET /api/v1/schedules` with no credentials.
2. `authHandler` passes the request through immediately.
3. `getSchedules()` fires query 1 (schedules), then queries 2+3 in parallel.
4. Each concurrent HTTP request holds 2–3 database connections from the shared pool.
5. The pool's `maxConnections` ceiling is reached; new queries queue or time out.
6. Legitimate gossip-monitoring reads against the same database are starved.

### Impact Explanation
The shared PostgreSQL connection pool (`rest/dbpool.js`) is consumed across all REST API endpoints and any other service sharing the same database. Exhausting it denies service to every consumer, including gossip-monitoring infrastructure that relies on timely schedule and transaction reads. The `statementTimeout` setting limits individual query duration but does not bound the number of concurrent connections a single attacker can hold open. Severity: **High** — complete availability loss for the database tier with no authentication barrier.

### Likelihood Explanation
The endpoint is publicly reachable, requires zero credentials, and the exploit requires only a standard HTTP flood tool (e.g., `wrk`, `ab`, `hey`). No special knowledge of the system is needed beyond the public OpenAPI spec (`rest/api/v1/openapi.yml`, line 1030) which documents the endpoint. The 3× amplification factor means the attacker needs only one-third the request rate compared to a single-query endpoint to achieve the same database load. Repeatability is unlimited.

### Recommendation
1. **Add per-IP rate limiting to the REST Helm middleware** (`charts/hedera-mirror-rest/values.yaml`): add `inFlightReq` and `rateLimit` entries matching or stricter than those in the Rosetta chart.
2. **Add application-level rate limiting** in `rest/server.js` using a middleware such as `express-rate-limit` applied globally before route handlers, so protection is not solely infrastructure-dependent.
3. **Enforce the `userLimitLabel`**: wire a rate-limiter middleware that reads `httpContext.get(userLimitLabel)` and applies it, so authenticated users with elevated limits are still bounded.
4. **Consider query result caching** for the entity and signature sub-queries to reduce the amplification factor when the same schedule IDs are requested repeatedly.

# Step 2: flood with concurrent unauthenticated requests
