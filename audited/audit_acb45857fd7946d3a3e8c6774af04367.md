### Title
Unbounded Redis Write Flooding via Unique Query Parameter Cache Misses in `responseCacheUpdateHandler`

### Summary
The `responseCacheUpdateHandler` in `rest/middleware/responseCacheHandler.js` unconditionally calls `Cache.setSingle()` (a Redis `SETEX` write) on every successful response that missed the cache. Because the cache key is derived from the full, un-normalized `req.originalUrl` (including all query parameters), an unprivileged attacker can generate an unbounded number of unique cache keys by varying query parameters, causing a continuous flood of Redis write operations with no rate limiting to stop them.

### Finding Description

**Cache key generation — no normalization:**

`rest/middleware/responseCacheHandler.js`, lines 151–153:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
`req.originalUrl` includes the full query string verbatim. The code comment at line 149 explicitly acknowledges this is not yet normalized: *"In the future, this will utilize Edwin's request normalizer (9113)."* This means `?account.id=0.0.1` and `?account.id=0.0.2` produce entirely different cache keys, as do `?a=1&b=2` vs `?b=2&a=1`.

**Cache miss path — key stored in locals:**

`responseCacheCheckHandler`, lines 45–47:
```js
if (!cachedTtlAndValue) {
  res.locals[responseCacheKeyLabel] = responseCacheKey;
  return;
}
```
On every cache miss, the key is placed in `res.locals` and the request proceeds to the route handler and DB.

**Unconditional write on every successful response:**

`responseCacheUpdateHandler`, lines 95–116:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
  const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
  if (ttl > 0) {
    ...
    await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
  }
}
```
`Cache.setSingle` calls `redis.setex(key, expiry, JSONStringify(value))` — a synchronous Redis write — for every unique URL that returns a 2xx response.

**No rate limiting in the REST API middleware chain:**

`rest/server.js` middleware stack (lines 68–144) contains: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, route handlers, `responseHandler`, `responseCacheUpdateHandler`, `handleError`. There is no rate-limiting middleware. The rate limiting found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) belongs exclusively to the `web3` Java service, not the REST Node.js API.

**Exploit flow:**
1. Attacker sends high-volume GET requests with unique query parameters to any public endpoint, e.g.:
   - `GET /api/v1/accounts?account.id=gte:0.0.1&limit=1`
   - `GET /api/v1/accounts?account.id=gte:0.0.2&limit=1`
   - `GET /api/v1/accounts?account.id=gte:0.0.3&limit=1`
   - … (millions of valid combinations using account IDs, timestamps, token IDs, etc.)
2. Each request generates a unique MD5 cache key → cache miss → DB query → 200 response.
3. `responseCacheUpdateHandler` calls `Cache.setSingle()` → `redis.setex()` for each.
4. Redis is flooded with write operations at the attacker's request rate.

### Impact Explanation
Redis write throughput becomes saturated, increasing latency for all `SETEX`/`GET` operations. Legitimate users experience degraded cache read performance. The `enableAutoPipelining: true` option in `cache.js` (line 24) batches writes efficiently, but this also means the attacker's writes are processed with minimal overhead per operation, maximizing throughput. The `maxmemory-policy` configuration (line 49 of `cache.js`) prevents memory exhaustion but does not limit write I/O or CPU load from `JSONStringify` serialization of large response bodies. Redis CPU and network I/O can be saturated, degrading the entire caching layer for all users of the mirror node.

### Likelihood Explanation
The attack requires zero authentication — the REST API is public. The valid parameter space (account IDs `0.0.1` through `0.0.50000000+`, timestamp ranges, token IDs, serial numbers) is enormous and well-documented in the public OpenAPI spec. A simple script with an incrementing counter generates millions of unique valid URLs. The attack is trivially repeatable and automatable with any HTTP client. No special knowledge of the system internals is required.

### Recommendation
1. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit`) before `responseCacheCheckHandler` in `rest/server.js`.
2. **Implement URL normalization** (the already-planned issue #9113): sort query parameters canonically before hashing in `cacheKeyGenerator` so that semantically equivalent requests share a cache key, reducing the unique key space.
3. **Add in-flight deduplication**: track keys currently being written to Redis and skip redundant `setSingle()` calls for the same key from concurrent requests (cache stampede protection).
4. **Consider a write budget**: limit the number of `setSingle()` calls per time window at the application level, independent of per-IP rate limiting.

### Proof of Concept
```bash
# Flood Redis with unique cache writes using incrementing account IDs
for i in $(seq 1 100000); do
  curl -s "https://<mirror-node-host>/api/v1/accounts?account.id=gte:0.0.$i&limit=1" &
  # Throttle to avoid local network saturation; Redis writes still accumulate
  if (( i % 100 == 0 )); then wait; fi
done
```
Each request produces a unique MD5 cache key, misses the cache, hits the DB, and triggers one `redis.setex()` call. Monitor Redis with `redis-cli INFO stats | grep total_commands_processed` and `redis-cli INFO commandstats | grep setex` to observe the write flood. Redis latency for `GET` operations (serving cached responses to legitimate users) will increase measurably as write throughput saturates the Redis instance.
