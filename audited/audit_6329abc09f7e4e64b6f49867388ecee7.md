### Title
Unauthenticated Request Flood via Unrate-Limited MetricsMiddleware Causes Prometheus Histogram Mutex Contention and CPU Saturation

### Summary
`MetricsMiddleware` is applied globally to every HTTP route — including the unauthenticated, zero-cost `/metrics` GET endpoint — with no rate limiting anywhere in the rosetta codebase. An unprivileged attacker can flood the server with concurrent requests, triggering four mutex-protected Prometheus histogram/gauge observations per request, causing lock contention in the Prometheus client library that saturates CPU and degrades response times for all legitimate users.

### Finding Description
**Code location:** `rosetta/app/middleware/metrics.go`, `MetricsMiddleware()`, lines 76–83; wired in `rosetta/main.go` line 217.

```
// main.go:217
metricsMiddleware := middleware.MetricsMiddleware(router)
```

`MetricsMiddleware` wraps the **entire** router unconditionally:

```go
// metrics.go:76-83
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,   // HistogramVec
        InflightRequests: requestInflightGauge,       // GaugeVec
        RequestBodySize:  requestBytesHistogram,      // HistogramVec
        ResponseBodySize: responseBytesHistogram,     // HistogramVec
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

Every HTTP request — including `GET /metrics` — triggers four Prometheus metric operations. Each `HistogramVec.Observe()` call acquires an internal `sync.Mutex` to update bucket counters. The `/metrics` endpoint is the optimal attack target: it requires no database queries, no authentication, and returns quickly, allowing an attacker to maximize request throughput and therefore histogram observation rate.

The `Http` config struct (`rosetta/app/config/types.go`, lines 64–69) exposes only timeout fields (`IdleTimeout`, `ReadTimeout`, `ReadHeaderTimeout`, `WriteTimeout`) — no connection limit or rate-limit field exists. A search across the entire `rosetta/` tree confirms zero rate-limiting code anywhere.

**Exploit flow:**
1. Attacker opens many concurrent TCP connections to the rosetta port (no auth required).
2. Each connection sends a stream of `GET /metrics` requests.
3. Each request passes through `MetricsMiddleware`, triggering 4 mutex-protected Prometheus operations.
4. Under high concurrency, goroutines queue on the histogram mutexes, consuming CPU in lock contention.
5. Legitimate API requests (block queries, construction calls) experience increased latency because the Go runtime scheduler is saturated with goroutines spinning on Prometheus mutexes.

### Impact Explanation
CPU saturation from Prometheus mutex contention degrades response times for all users of the rosetta API. Because the `/metrics` endpoint is the cheapest route (no DB I/O), it is the most efficient amplifier: an attacker can sustain the highest possible request rate against it, maximizing histogram observation throughput. The impact is availability degradation (griefing) with no economic cost to the attacker and no authentication barrier.

### Likelihood Explanation
Any unprivileged external user with a moderate internet connection can execute this attack using standard HTTP benchmarking tools (e.g., `wrk`, `hey`, `ab`). No credentials, tokens, or special knowledge of the API are required — only the server's IP and port. The attack is trivially repeatable and requires no persistence on the target system.

### Recommendation
1. **Add a rate-limiting middleware** (e.g., `golang.org/x/time/rate` or `github.com/ulule/limiter`) applied before `MetricsMiddleware` in the chain in `main.go`, limiting requests per source IP.
2. **Restrict `/metrics` to internal/loopback access only** (e.g., bind the metrics endpoint to `127.0.0.1` or require a separate internal port), since Prometheus scrapers are typically co-located infrastructure, not external clients.
3. **Add a `MaxConnsPerHost` or `http.Server.ConnState` hook** to cap total concurrent connections.
4. Optionally, configure a reverse proxy (nginx/Envoy) in front of rosetta with connection and request-rate limits.
