### Title
Unauthenticated Mirror-Node Response Allows Arbitrary gRPC Endpoint Injection into Hiero SDK Client

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` fetches consensus-node addresses over plain HTTP and inserts them into the SDK network map without any IP-address validation. An attacker who can intercept or spoof the mirror-node REST response (e.g., via network MITM on the unencrypted HTTP channel, DNS poisoning, or ARP spoofing) can inject `127.0.0.1` (or any other address) as a `service_endpoints` IP, causing `hiero.ClientForNetwork(netmap)` to direct all signed transactions to an attacker-controlled or loopback endpoint.

### Finding Description
**Code path:**

`pinger/sdk_client.go` `newClient()` (line 18) → `buildNetworkFromMirrorNodes()` → `fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` (lines 74–132).

**Root cause — lines 113–124 of `mirror_node_client.go`:**
```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {   // ← only emptiness check
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId    // ← injected directly into SDK map
}
```
The only guard is `host == "" || ep.Port == 0`. There is **no validation** that the IP is not loopback (`127.0.0.0/8`), link-local (`169.254.0.0/16`), RFC-1918 private ranges, or any other restricted address. The resulting `network` map is passed verbatim to `hiero.ClientForNetwork(netmap)` at `sdk_client.go` line 22, which the SDK uses as the authoritative list of consensus-node gRPC endpoints.

The HTTP client is constructed with no TLS configuration (line 46 of `mirror_node_client.go`):
```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```
The documented/example URL (`http://mirror-rest:5551`, `config.go` line 37) uses plain HTTP, making the channel trivially interceptable on any shared network segment.

**Exploit flow:**
1. Pinger is deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
2. Attacker intercepts the HTTP GET to `/api/v1/network/nodes` (MITM, DNS spoof, or ARP poisoning on the pod/node network).
3. Attacker returns a crafted JSON body:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"ip_address_v4":"127.0.0.1","port":50211}]}]}
   ```
4. `fetchMirrorNodeNetwork()` accepts this, builds `network["127.0.0.1:50211"] = AccountID{0,0,3}`, and returns it.
5. `hiero.ClientForNetwork(netmap)` configures the SDK to send all gRPC traffic to `127.0.0.1:50211`.
6. Every subsequent `submitWithRetry()` call sends a fully-signed transaction to the loopback interface.

### Impact Explanation
- **Transaction loss:** If nothing listens on `127.0.0.1:50211`, every transfer silently fails (gRPC connection refused). The pinger logs an error but continues, so the operator may not notice immediately.
- **Transaction interception/tampering:** If the attacker also runs a gRPC listener on `127.0.0.1:50211` inside the same container/pod (e.g., via a previously planted sidecar or a second exploit), they receive fully-signed, ready-to-submit transactions and can replay, delay, or selectively drop them.
- **SSRF pivot:** The injected address need not be `127.0.0.1`; any internal cluster IP (e.g., a metadata service at `169.254.169.254`) can be targeted, turning the pinger into an SSRF vector.

Severity: **High** — confidentiality and integrity of financial transactions are directly affected.

### Likelihood Explanation
- Precondition: `network=other` mode (required for private/custom deployments, which is the exact use-case this code path serves).
- The plain-HTTP default makes passive interception trivial on any shared L2 segment (cloud VPC, Kubernetes overlay network with a compromised node, etc.).
- DNS poisoning of an internal service name (`mirror-rest`) is a well-known intra-cluster attack vector.
- No authentication or TLS certificate pinning is required of the attacker.
- The attack is repeatable: the client is built once at startup, so a single poisoned response permanently redirects all transactions for the lifetime of the process.

### Recommendation
1. **Validate returned IP addresses** before inserting into the network map. Reject loopback, link-local, and RFC-1918 addresses unless an explicit allow-list is configured:
   ```go
   ip := net.ParseIP(host)
   if ip != nil && (ip.IsLoopback() || ip.IsLinkLocalUnicast() || ip.IsPrivate()) {
       continue // or return error
   }
   ```
2. **Enforce HTTPS** for the mirror-node REST URL and verify the server certificate. Reject `http://` URLs at config-load time, or at minimum warn loudly.
3. **Pin or authenticate the mirror-node response** (e.g., compare returned node IDs against a locally-configured allow-list of known account IDs).
4. **Validate port range** (e.g., only accept well-known gRPC ports like 50211/50212).

```bash

import http.server, json
class H(http.server.BaseHTTPRequestHandler):

            'links':{'next':None}}).encode()
        self.send_response(200); self.send_header('Content-Type','application/json')
```
