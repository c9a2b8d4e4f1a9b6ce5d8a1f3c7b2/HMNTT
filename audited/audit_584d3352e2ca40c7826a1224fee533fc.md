### Title
Entity Cache Cache-Busting via Distinct EVM Addresses Causing Sustained DB Overload

### Summary
The `cacheManagerEntity()` bean in `EvmConfiguration.java` configures all three entity cache partitions (`default`, `evmAddress`, `alias`) with a shared `expireAfterWrite=1s,maximumSize=10000` policy. An unprivileged attacker can send up to 500 `eth_call` requests per second (the default global RPS limit) each targeting a distinct EVM address, exhausting the 10,000-entry cache capacity and guaranteeing every request falls through to a live PostgreSQL query, causing a sustained and measurable increase in DB query load and CPU usage.

### Finding Description

**Exact code locations:**

`EvmConfiguration.java` lines 99–105 — all three cache names share one `CaffeineCacheManager` built from the same spec string: [1](#0-0) 

`CacheProperties.java` line 19 — the shared spec is hardcoded as `expireAfterWrite=1s,maximumSize=10000,recordStats`: [2](#0-1) 

`EntityRepository.java` lines 32–49 — `findByEvmAddressAndDeletedIsFalse` and `findByEvmAddressOrAliasAndDeletedIsFalse` are the two cached methods that resolve an EVM address to an entity row: [3](#0-2) 

**Root cause and failed assumption:**

The design assumes that real traffic will reuse a small, stable set of entity addresses within any 1-second window, so a 10,000-entry / 1-second cache is sufficient. That assumption fails when an attacker deliberately supplies a fresh 20-byte EVM address on every request. Because EVM addresses are arbitrary 20-byte values, the attacker has an effectively unbounded supply of distinct keys. Once more than 10,000 distinct addresses are presented within a second, every subsequent lookup is a guaranteed cache miss and hits the database.

**Exploit flow:**

1. Attacker generates a stream of distinct, syntactically valid 20-byte EVM addresses (e.g., sequential or random).
2. Each address is placed in the `to` field of a `POST /api/v1/contracts/call` request with minimal gas (e.g., `gas=21000`).
3. `ThrottleManagerImpl.throttle()` checks the global `rateLimitBucket` (500 tokens/second) and the `gasLimitBucket`. With `gas=21000`, `scaleGas()` returns `floor(21000/10000)=2` tokens, far below the 750,000-token/second gas budget, so both checks pass. [4](#0-3) 
4. `CommonEntityAccessor.get(Address, timestamp)` is called. Since the address is not a long-zero address, it routes to `findByEvmAddressAndDeletedIsFalse`. [5](#0-4) 
5. The Caffeine cache for `evmAddress` has no entry for this key (never seen before, or evicted). Spring Cache invokes the actual repository method, which executes a `SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE` against PostgreSQL.
6. The attacker sustains 500 such requests/second. After the first 10,000 distinct addresses fill the cache, every subsequent request evicts an existing entry and inserts a new one — the hit rate approaches 0%.

**Why existing checks are insufficient:**

- **Rate limiter** (`requestsPerSecond=500`): This is a *global* limit shared across all clients, not a per-IP limit. A single attacker can consume the entire budget. [6](#0-5) 
- **Gas throttle**: `scaleGas()` divides by `GAS_SCALE_FACTOR=10_000` and returns 0 for gas ≤ 10,000. A request with `gas=21,000` costs only 2 gas tokens out of 750,000/second — effectively free from the gas-budget perspective. [7](#0-6) 
- **Cache TTL**: `expireAfterWrite=1s` means even legitimately cached entries expire every second, compounding the miss rate under any sustained load pattern.
- **No per-IP or per-address rate limiting** is present anywhere in the throttle pipeline. [8](#0-7) 

### Impact Explanation

At 500 RPS of cache-miss entity lookups, the PostgreSQL instance receives at minimum 500 additional `SELECT` queries per second purely for entity resolution, on top of all other queries triggered by EVM execution (contract state, token lookups, record file lookups). On a lightly-to-moderately loaded node, this easily exceeds a 30% increase in total DB query throughput and associated CPU. Because the token cache shares the same 1-second TTL (`token = ENTITY_CACHE_CONFIG`), token-related lookups triggered by the same requests are equally affected, multiplying the DB impact per request. The attack is sustained indefinitely as long as the attacker maintains the request rate, with no automatic back-off or circuit-breaker in the entity lookup path.

### Likelihood Explanation

No authentication or API key is required to call `POST /api/v1/contracts/call`. The attacker needs only an HTTP client and the ability to generate distinct 20-byte hex strings — trivially scriptable in any language. The attack is repeatable, requires no on-chain funds, and is invisible to standard application-layer monitoring until DB saturation is observed. A single machine with a modest network connection can sustain 500 RPS of small HTTP POST requests.

### Recommendation

1. **Increase cache TTL and size**: Raise `expireAfterWrite` to at least `10s`–`30s` and `maximumSize` to `50000`–`100000` to make cache-busting economically impractical within the rate limit window.
2. **Add per-IP rate limiting**: Enforce a per-source-IP sub-limit (e.g., 10–20 RPS) in addition to the global 500 RPS bucket, using a servlet filter or Spring Security rate-limiting layer before `ThrottleManagerImpl`.
3. **Cache negative results explicitly**: The `unless = "#result == null"` condition on `findByEvmAddressAndDeletedIsFalse` correctly caches `Optional.empty()` (since it is not `null`), but confirm this behavior is tested and that no code path returns a raw `null` instead of `Optional.empty()`, which would bypass caching entirely.
4. **Validate `to` address existence early**: Reject requests whose `to` address does not correspond to a known contract before entering the EVM execution path, avoiding the entity lookup entirely for obviously invalid addresses.

### Proof of Concept

```bash
# Generate 100,000 distinct EVM addresses and send eth_call at 500 RPS
python3 - <<'EOF'
import asyncio, aiohttp, os, json

URL = "http://<mirror-node-host>:8080/api/v1/contracts/call"
HEADERS = {"Content-Type": "application/json"}

async def send(session, addr):
    body = json.dumps({"to": addr, "gas": 21000, "data": "0x", "estimate": False})
    async with session.post(URL, data=body, headers=HEADERS):
        pass

async def main():
    addrs = ["0x" + os.urandom(20).hex() for _ in range(100_000)]
    async with aiohttp.ClientSession() as session:
        # Send in batches of 500 per second, sustained
        for i in range(0, len(addrs), 500):
            batch = addrs[i:i+500]
            await asyncio.gather(*[send(session, a) for a in batch])
            await asyncio.sleep(1)

asyncio.run(main())
EOF
```

**Expected result**: PostgreSQL `pg_stat_activity` shows a sustained spike of 400–500 concurrent entity-lookup queries. DB CPU and query throughput metrics increase by ≥30% compared to the 24-hour baseline. The Caffeine cache `recordStats` for the `evmAddress` cache will show a hit rate near 0%.

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
