### Title
Connection Pool Exhaustion via Unbounded Alias DB Lookups During Network Partition

### Summary
Any unauthenticated user can send a burst of `getNftAllowances()` requests using an alias-format `accountId`. Each request synchronously blocks a JDBC connection on `entityRepository.findByAlias()`, which has no query timeout. During a network partition, these connections hang indefinitely, exhausting the connection pool and rendering the service unavailable to all users.

### Finding Description

**Code path:**

`NftAllowanceServiceImpl.getNftAllowances()` (line 28) calls `entityService.lookup(request.getAccountId())`. [1](#0-0) 

`EntityServiceImpl.lookup()` (line 33) dispatches to `entityRepository.findByAlias(p.alias())` when the parameter is an `EntityIdAliasParameter`, with no timeout, no circuit breaker, and no async wrapper — a fully synchronous blocking call. [2](#0-1) 

`EntityRepository.findByAlias()` is a plain native SQL query with no `@QueryHints` timeout: [3](#0-2) 

**Root cause:** The `findByAlias()` query carries no statement timeout hint, and the calling chain is entirely synchronous. Each in-flight request holds one JDBC connection from the pool for the full duration of the DB round-trip. Under a network partition the DB socket never returns, so the connection is held until the OS TCP timeout (minutes). HikariCP's default pool is 10 connections; once exhausted, all subsequent requests queue or fail with a pool-acquisition timeout, making the service unavailable.

**Why existing checks are insufficient:**
- `EntityIdAliasParameter.valueOf()` only validates the Base32 format of the alias — it does not prevent the DB call. [4](#0-3) 
- `checkOwnerSpenderParamValidity()` validates query parameter semantics, not request rate or DB availability. [5](#0-4) 
- No rate limiting, circuit breaker, or bulkhead annotation is present anywhere in the service or repository layer.

### Impact Explanation
Complete denial of service for the `getNftAllowances` endpoint (and any other endpoint sharing the same JDBC pool). During a network partition — even a brief one — a single attacker with a modest HTTP client can exhaust all pool connections. All legitimate users receive errors or indefinite hangs until the partition resolves and connections drain. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation
The attacker needs only an HTTP client and knowledge of the Base32 alias format (publicly documented). No credentials, tokens, or special network position are required. The attack is trivially repeatable: send N concurrent requests where N ≥ pool size. Network partitions between the application and its database are a realistic operational event (cloud AZ issues, maintenance, misconfiguration). The attack is effective even with a very short partition window.

### Recommendation
1. **Add a JDBC statement timeout** to `findByAlias()` via `@QueryHints`:
   ```java
   @QueryHints(@QueryHint(name = "jakarta.persistence.query.timeout", value = "3000"))
   ```
2. **Configure HikariCP** `connectionTimeout` and `socketTimeout` at the datasource level (e.g., `spring.datasource.hikari.connection-timeout=5000`).
3. **Apply a circuit breaker** (Resilience4j `@CircuitBreaker`) around `entityRepository.findByAlias()` to fast-fail when the DB is unreachable.
4. **Apply rate limiting** at the API gateway or via `@RateLimiter` on the controller endpoint to bound the number of concurrent alias-resolution requests per client IP.

### Proof of Concept

**Preconditions:** Application running, DB reachable initially; attacker can reach the REST API endpoint.

**Steps:**
1. Identify a valid alias-format string (any 40–70 uppercase Base32 chars, e.g., `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`).
2. Simulate or wait for a network partition between the app and DB (e.g., `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the app host).
3. Send a burst of concurrent requests:
   ```bash
   for i in $(seq 1 20); do
     curl -s "http://<host>/api/v1/accounts/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA/allowances/nfts" &
   done
   wait
   ```
4. Each request enters `entityRepository.findByAlias()` and blocks on the hung JDBC socket.
5. After 10 requests (default HikariCP pool size), all pool connections are exhausted.
6. All subsequent requests — including from legitimate users — receive connection pool timeout errors or hang indefinitely.
7. The service remains degraded until the partition ends and connections time out at the OS level.

    private void checkOwnerSpenderParamValidity(Bound ownerOrSpenderParams, Bound tokenParams) {
        if (ownerOrSpenderParams.isEmpty() && !tokenParams.isEmpty()) {
            throw new IllegalArgumentException("token.id parameter must have account.id present");
        }

        verifyRangeId(ownerOrSpenderParams);
        verifyRangeId(tokenParams);

        if (!ownerOrSpenderParams.hasLowerAndUpper()
                && tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {
            throw new IllegalArgumentException("Invalid range provided for %s".formatted(Constants.TOKEN_ID));
        }

        if (tokenParams.getCardinality(RangeOperator.LT, RangeOperator.LTE) > 0
                && ownerOrSpenderParams.getCardinality(RangeOperator.EQ, RangeOperator.LTE) == 0) {
            throw new IllegalArgumentException("Requires the presence of an lte or eq account.id parameter");
        }
        if (tokenParams.getCardinality(RangeOperator.GT, RangeOperator.GTE) > 0
                && ownerOrSpenderParams.getCardinality(RangeOperator.EQ, RangeOperator.GTE) == 0) {
            throw new IllegalArgumentException("Requires the presence of an gte or eq account.id parameter");
        }
    }

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
