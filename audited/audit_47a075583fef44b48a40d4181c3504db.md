### Title
Gas Estimation Binary Search Bypasses Throttle: Up to 21x EVM Execution Amplification per Request

### Summary
The `ThrottleManagerImpl.throttle()` method consumes gas-bucket tokens exactly once per HTTP request based on `request.getGas()`, but a request with `estimate=true` triggers up to 21 sequential EVM executions: one initial call followed by up to `maxGasEstimateRetriesCount` (default 20) binary-search iterations. The 20 binary-search iterations in `doProcessCall(..., estimate=true)` are completely invisible to the throttle â€” no tokens are consumed and no restore occurs â€” giving an unprivileged attacker a 21x EVM-execution amplification factor at the same throttle cost as a single `eth_call`.

### Finding Description

**Exact code path:**

`ContractController.call()` calls `throttleManager.throttle(request)` once: [1](#0-0) 

`ThrottleManagerImpl.throttle()` consumes `scaleGas(request.getGas())` tokens from `gasLimitBucket` and 1 token from `rateLimitBucket` â€” both consumed exactly once per HTTP request: [2](#0-1) 

For `estimate=true`, `ContractExecutionService.estimateGas()` first calls `callContract()` (EVM execution #1), then calls `binaryGasEstimator.search()` which runs up to `maxGasEstimateRetriesCount=20` additional EVM executions: [3](#0-2) 

`BinaryGasEstimator.search()` loops up to `maxGasEstimateRetriesCount` (default 20) times, each iteration calling `doProcessCall(params, mid, true)`: [4](#0-3) 

**Root cause â€” the `estimate` flag gates the entire throttle restore path:**

In `ContractCallService.doProcessCall()`, the `finally` block only calls `restoreGasToBucket()` when `estimate=false`. When `estimate=true` (all 20 binary-search iterations), neither gas restoration nor any additional throttle consumption occurs: [5](#0-4) 

**Failed assumption:** The throttle design assumes one gas-bucket deduction corresponds to one EVM execution. For `estimate=true` this assumption is violated by a factor of up to 21.

**Exploit flow:**
1. Attacker sends `POST /api/v1/contracts/call` with `estimate=true`, `gas=15_000_000`, and calldata targeting a contract that consumes near-maximum gas (e.g., a tight loop or storage-heavy precompile call).
2. `throttle()` deducts `15_000_000 / 10_000 = 1500` gas tokens and 1 request token â€” identical cost to a regular `eth_call`.
3. `estimateGas()` runs 1 initial EVM call + up to 20 binary-search EVM calls = **21 full EVM executions**.
4. The 20 binary-search iterations each invoke `transactionExecutionService.execute()` with gas values between `gasUsedByInitialCall` and `15_000_000`, consuming significant CPU, with zero additional throttle cost.

**Why existing checks are insufficient:**

- `rateLimitBucket` (default 500 req/s): limits HTTP requests, not EVM executions. 500 estimate requests/s â†’ up to 10,500 EVM executions/s.
- `gasLimitBucket` (default 7.5B gas/s, scaled to 750,000 tokens/s): deducted once per request. At 500 req/s Ã— 1500 tokens = 750,000 tokens/s consumed â€” exactly at the limit â€” but the actual EVM gas processed is 21Ã— higher.
- `validateContractMaxGasLimit()`: only enforces `gas â‰¤ 15_000_000`, which is the exact value that maximizes the binary search range and iteration count. [6](#0-5) 

### Impact Explanation

An unprivileged attacker sending estimate=true requests at the permitted rate (500 req/s) causes up to 10,500 EVM executions per second, versus 500 for a normal workload â€” a **21x amplification**. Even at a fraction of the rate limit (e.g., 25 req/s), the attacker injects 525 extra EVM executions/s on top of a baseline of 500, exceeding the 30% CPU increase threshold. Each EVM execution with 15M gas is a CPU-intensive operation (EVM bytecode interpretation, storage reads, precompile calls). This can degrade response times for all users and potentially cause the node to fall behind on its mirror-node duties.

### Likelihood Explanation

The attack requires no credentials, no on-chain assets, and no special knowledge beyond the public REST API. The endpoint is publicly documented. The attacker needs only an HTTP client and knowledge that `estimate=true` triggers binary search. The attack is trivially repeatable and automatable. A single attacker with a modest connection can sustain the attack indefinitely.

### Recommendation

1. **Account for binary-search multiplier in throttle:** Before entering `binaryGasEstimator.search()`, pre-consume additional gas tokens proportional to `maxGasEstimateRetriesCount` (e.g., consume `gas Ã— maxGasEstimateRetriesCount` tokens upfront, or consume one token per binary-search iteration inside the loop).
2. **Per-iteration throttle check:** Call `throttleManager` (or a dedicated estimate-iteration bucket) inside `BinaryGasEstimator.search()` at each iteration, so each EVM execution is individually rate-limited.
3. **Separate, lower rate limit for estimate requests:** Add a dedicated `rateLimitBucket` for `estimate=true` requests with a significantly lower cap (e.g., 25 req/s) to bound the maximum EVM execution rate from estimation traffic.
4. **Cap binary search range:** Reduce `maxGasEstimateRetriesCount` or enforce a tighter upper bound on the binary search `hi` value to limit worst-case iterations.
