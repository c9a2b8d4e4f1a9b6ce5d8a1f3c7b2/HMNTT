### Title
Hardcoded `associated = true` Filter Enables Unbounded Index Scan DoS on `token_account` Table

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` issues a query via `TokenService.getTokenRelationshipsQuery()` that always appends `AND ta.associated = true` as a hardcoded, non-indexed filter. Because the `token_account` primary key is `(account_id, token_id)` and `associated` is not part of any index, PostgreSQL must scan every row for the target `account_id` before applying the filter and the `LIMIT`. An unauthenticated attacker who identifies an account with a large number of dissociated token rows can repeatedly trigger this scan with zero cost, consuming excessive DB I/O.

### Finding Description

**Exact code path:**

`rest/service/tokenService.js` lines 20–29 define the base query:

```js
static tokenRelationshipsQuery = `
    select ...
    from token_account ta
    where ta.account_id = $1
    and ta.associated = true `;
``` [1](#0-0) 

`rest/controllers/tokenController.js` lines 66–92 handle the HTTP request, call `extractTokensRelationshipQuery`, and pass the result to `TokenService.getTokenAccounts()`: [2](#0-1) 

The `limit` parameter is parsed and capped at `max: 100` in `utils.js` line 1396:

```js
case constants.filterKeys.LIMIT:
  comparator.value = Math.min(Number(comparator.value), getEffectiveMaxLimit());
  break;
``` [3](#0-2) 

The configured max is 100: [4](#0-3) 

**Root cause:**

The `token_account` table's primary key is `(account_id, token_id)`: [5](#0-4) 

The column `associated` is not part of any index. The generated SQL is:

```sql
SELECT ... FROM token_account ta
WHERE ta.account_id = $1 AND ta.associated = true
ORDER BY ta.token_id ASC
LIMIT $2
```

PostgreSQL uses the PK index to seek to `account_id = $1` and then scans forward through all `token_id` values for that account, evaluating `associated = true` row-by-row. The `LIMIT` only stops the scan once enough **matching** rows are found. If an account has N total rows but only K rows with `associated = true`, the DB must read up to N rows to return K results. When K < limit (e.g., all rows are `associated = false`), the DB reads every row for that account and returns nothing — the most expensive case.

**Failed assumption:** The `LIMIT 100` cap prevents excessive DB work. In reality it only caps the output, not the scan depth.

### Impact Explanation

An account that has associated and subsequently dissociated from many tokens accumulates rows with `associated = false` in `token_account`. Each unauthenticated request to `GET /api/v1/accounts/{id}/tokens` forces a full per-account index scan. With no rate limiting present in the Node.js REST layer, an attacker can flood the endpoint with concurrent requests targeting such an account, driving sustained high I/O on the database. This can degrade or deny service to all mirror node consumers sharing the same DB instance.

### Likelihood Explanation

The attacker requires no credentials and no HBAR spend — they only need to identify one existing high-churn account (e.g., a DeFi protocol treasury or exchange hot wallet that has historically associated and dissociated many tokens). Such accounts exist on mainnet. The attack is trivially repeatable with standard HTTP tooling (e.g., `ab`, `wrk`). No authentication, API key, or proof-of-work is required.

### Recommendation

1. **Add a partial index** on `token_account(account_id, token_id) WHERE associated = true`. This allows PostgreSQL to satisfy the query entirely from the index without scanning non-matching rows.
2. **Alternatively**, store only `associated = true` rows in `token_account` and move dissociated rows exclusively to `token_account_history`, so the filter becomes redundant on the live table.
3. **Add rate limiting** per IP or per account-ID parameter at the API gateway or middleware layer to bound the request rate regardless of query cost.
4. **Set a statement timeout** for the Node.js REST DB pool (analogous to `hiero.mirror.restJava.db.statementTimeout`) to bound the maximum wall-clock time any single query can consume.

### Proof of Concept

1. Identify (or create) an account `0.0.X` that has a large number of rows in `token_account` with `associated = false` (e.g., an account that associated and dissociated thousands of tokens).
2. From any unauthenticated client, run:
   ```bash
   wrk -t 10 -c 100 -d 30s \
     "https://<mirror-node>/api/v1/accounts/0.0.X/tokens?limit=100"
   ```
3. Observe via `pg_stat_activity` or `pg_stat_user_tables` that `token_account` heap/index reads spike proportionally to the number of rows for account `0.0.X`, not to the `limit=100` cap.
4. Sustained load causes DB I/O saturation, increasing query latency for all other mirror node API consumers.
