### Title
Global Rate-Limit Bucket Exhaustion via Unauthenticated Concurrent Requests (No Per-IP Isolation)

### Summary
The `rateLimitBucket` in `ThrottleConfiguration` is a single, globally-shared Bucket4j token bucket with no per-IP or per-user partitioning. Any unauthenticated caller can fire `requestsPerSecond` (default: 500) concurrent `POST /api/v1/contracts/call` requests in a single burst, draining the entire bucket for the current refill window. All subsequent requests from legitimate users receive a `ThrottleException` ("Requests per second rate limit exceeded") for the remainder of that second, while the attacker's own requests have already been processed.

### Finding Description

**Code path:**

`ContractController.call()` (line 40) delegates unconditionally to `ThrottleManagerImpl.throttle()`: [1](#0-0) 

`ThrottleManagerImpl.throttle()` calls `rateLimitBucket.tryConsume(1)` on a single shared bucket: [2](#0-1) 

`rateLimitBucket` is constructed as a **global singleton** with capacity = `requestsPerSecond` (default 500) and a greedy refill of the same amount per second: [3](#0-2) 

**Root cause — failed assumption:** The design assumes the rate limit budget will be spread across many independent callers. There is no per-IP, per-session, or per-account sub-bucket. The `RequestFilter` enum covers `BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, `VALUE` — no `IP` or `REMOTE_ADDR` field exists: [4](#0-3) 

**Asymmetry note:** `gasLimitBucket` is explicitly built with `SynchronizationStrategy.SYNCHRONIZED`, while `rateLimitBucket` and `opcodeRateLimitBucket` use the default `LOCK_FREE` strategy. Both are thread-safe in Bucket4j (LOCK_FREE uses CAS), so there is no race condition allowing over-consumption — the bucket correctly enforces its capacity. The vulnerability is purely architectural: one global pool shared by all callers. [5](#0-4) 

**Exploit flow:**
1. Attacker opens 500 parallel HTTP connections (no auth required).
2. All 500 `POST /api/v1/contracts/call` requests arrive within the same 1-second window.
3. Each call atomically consumes 1 token from the shared `rateLimitBucket` (capacity 500).
4. Bucket is now at 0 tokens; refill begins but takes up to 1 second to restore.
5. Any legitimate request arriving in that window hits `tryConsume(1) == false` and receives HTTP 429.
6. Attacker repeats every second, maintaining a sustained denial of service.

### Impact Explanation
Legitimate users are completely locked out of the `/api/v1/contracts/call` endpoint for the duration of the attack. Because the endpoint is unauthenticated and the bucket is global, a single attacker with a modest number of parallel connections can sustain this indefinitely. The default capacity of 500 req/s is easily saturated from a single host with a thread pool or async HTTP client.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. The attack requires only a standard HTTP client capable of concurrent requests (e.g., `curl --parallel`, Apache Bench, Python `asyncio`). It is trivially repeatable and scriptable. The default `requestsPerSecond = 500` is a small enough number that a single commodity machine can exhaust it in one burst. [6](#0-5) 

### Recommendation
1. **Add per-IP rate limiting** as a first-class dimension. Introduce a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` or `HttpServletRequest.getRemoteAddr()`), so each IP has its own sub-bucket whose capacity is a fraction of the global limit.
2. **Alternatively**, deploy an API gateway or reverse proxy (e.g., nginx `limit_req`, Envoy rate-limit filter) in front of the service that enforces per-IP limits before requests reach the JVM.
3. **Extend `RequestFilter.FilterField`** to include `IP`/`REMOTE_ADDR` so operators can configure per-source throttle rules without code changes.
4. Consider adding `withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)` to `rateLimitBucket` for consistency with `gasLimitBucket` under high-concurrency load, though this does not fix the per-IP isolation gap.

### Proof of Concept
```bash
# Exhaust the default 500 req/s budget in a single burst from one host
# Requires: GNU parallel or xargs, curl

seq 1 500 | parallel -j 500 \
  curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":21000,"estimate":false}'

# Expected: first ~500 return 200, subsequent requests within the same second return 429
# Repeat every second to maintain sustained DoS for all other clients
```
