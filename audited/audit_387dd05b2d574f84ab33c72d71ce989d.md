### Title
Unauthenticated Unbounded Topic Subscription Enables DB Exhaustion via `PollingTopicMessageRetriever.retrieve()`

### Summary
The `retrieve()` method in `PollingTopicMessageRetriever` accepts any `TopicMessageFilter` and immediately begins repeated database polling with no authentication, authorization, or per-caller rate limiting. Any unauthenticated external client can open unlimited gRPC connections, each subscribing to any topicId (including high-traffic system topics) with no message limit and a far-past start time, causing unbounded concurrent DB polling that saturates the database and starves legitimate subscribers of timely message delivery.

### Finding Description

**Entry point** — `ConsensusController.subscribeTopic()` (`grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, lines 43–53): accepts any `ConsensusTopicQuery` from any caller with no credential check, converts it to a `TopicMessageFilter`, and passes it directly to `topicMessageService.subscribeTopic()`.

**Service layer** — `TopicMessageServiceImpl.subscribeTopic()` (`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 59–92): the only guard is `topicExists()` (lines 94–106), which checks that the entity exists and has type `TOPIC`. There is no caller identity check, no per-IP subscription cap, and no per-topic subscriber limit.

**Polling loop** — `PollingTopicMessageRetriever.retrieve()` (`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, lines 45–63): for a throttled subscription (`throttled=true`), `numRepeats = Long.MAX_VALUE` (line 99) and the loop repeats indefinitely until the page size drops below `maxPageSize` or a limit is hit. With no limit set by the attacker, this runs forever, issuing a DB query every `pollingFrequency = 2s` (default), fetching up to `maxPageSize = 1000` rows per poll.

**Connection-level limit is insufficient** — `GrpcConfiguration` (`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, line 33) sets `maxConcurrentCallsPerConnection = 5` via `NettyProperties`. This is a *per-connection* cap. There is no limit on the number of inbound connections, so an attacker opens N connections × 5 calls = 5N concurrent polling subscriptions.

**No gRPC-layer rate limiting** — The `ThrottleConfiguration` / `ThrottleManagerImpl` exist only in the `web3` module and are not wired into the gRPC module at all. The only gRPC interceptor present (`grpc/src/test/java/org/hiero/mirror/grpc/interceptor/GrpcInterceptor.java`) merely sets an endpoint context string and forwards the call.

### Impact Explanation
Each attacker-controlled subscription issues a `SELECT` against `topic_message` every 2 seconds, returning up to 1 000 rows. With 100 connections × 5 calls = 500 concurrent subscriptions, the server issues 250 DB queries per second, each potentially scanning a large partition of a high-traffic topic. This saturates the database connection pool and I/O, causing query timeouts and backpressure for legitimate subscribers. Legitimate subscribers stop receiving HCS topic messages in a timely manner — effectively preventing the mirror node from delivering (gossiping) new consensus transactions on those topics. The `retrieverProperties.timeout = 60s` only disconnects a subscription that emits *no* messages for 60 s; a high-traffic topic continuously resets this timer, keeping attacker subscriptions alive indefinitely.

### Likelihood Explanation
The gRPC port (default 5600) is publicly documented and reachable. No credentials, API keys, or tokens are required. The attack requires only a standard gRPC client (e.g., `grpcurl`) and the ability to open TCP connections. It is trivially scriptable, repeatable, and requires no special knowledge beyond the public protobuf definition. A single attacker machine with modest bandwidth can sustain hundreds of concurrent streaming RPCs.

### Recommendation
1. **Per-IP / global subscription cap**: Track active subscriptions per source IP and enforce a hard ceiling (e.g., 10 concurrent subscriptions per IP) in a `ServerInterceptor` registered via `@GlobalServerInterceptor`.
2. **Token-bucket rate limiting at the gRPC layer**: Apply a rate limiter (analogous to `ThrottleManagerImpl` in `web3`) to `subscribeTopic` calls, rejecting excess requests with `RESOURCE_EXHAUSTED`.
3. **Global concurrent-subscriber ceiling**: Use the existing `subscriberCount` gauge in `TopicMessageServiceImpl` to enforce a hard cap; return `UNAVAILABLE` when exceeded.
4. **Require a non-zero `limit`** for unauthenticated callers, preventing indefinite open-ended subscriptions.
5. **Restrict `startTime` lookback** for unauthenticated callers to prevent full-history scans on high-traffic topics.

### Proof of Concept
```bash
# Open 20 parallel shells, each running 5 concurrent subscriptions (= 100 total)
# targeting a known high-traffic system topic (e.g., topicNum 0.0.101) from epoch 0

for i in $(seq 1 5); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":101},"consensusStartTime":{"seconds":0},"limit":0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Repeat the above block in 20 terminals simultaneously.
# Observe: DB CPU/IO spikes, legitimate subscriber latency increases,
# query timeouts appear in mirror-node-grpc logs.
```
No credentials are required. The `limit:0` field means "no limit", keeping each subscription alive indefinitely and polling the DB every 2 seconds per the default `pollingFrequency`.
