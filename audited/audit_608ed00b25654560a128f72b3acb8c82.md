### Title
Historical Call Data Staleness: `getContractResultsByTransactionIdOrHash` Returns Current File Contents Instead of Point-in-Time Contents for ETHEREUMTRANSACTION `callDataId`

### Summary
When an ETHEREUMTRANSACTION references a Hashgraph file via `callDataId`, the mirror node REST API resolves the call data by fetching the **current/latest** file contents rather than the file's contents at the transaction's consensus timestamp. Any unprivileged user querying `GET /api/v1/contracts/results/{transactionHash}` for such a transaction will receive a `call_data` value that does not match what was actually executed on the Hashgraph, silently corrupting the historical record served by the mirror node.

### Finding Description

**Exact code path:**

In `rest/controllers/contractController.js`, `getContractResultsByTransactionIdOrHash` (lines 1183–1185):

```js
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
```

`getLatestFileDataContents` is called with `{whereQuery: []}` — an **empty** filter array. No timestamp constraint is passed, even though `ethTransaction.consensusTimestamp` is available in scope.

**Root cause — the SQL query has no upper-timestamp bound:**

`FileDataService.getLatestFileDataContents` in `rest/service/fileDataService.js` (lines 85–88) builds its query via `getLatestFileContentsQuery(where)`, where `where` is an empty string when `whereQuery` is `[]`. The underlying `latestFileContentsQuery` (lines 19–42) is:

```sql
with latest_create as (
    select max(consensus_timestamp) as consensus_timestamp
    from file_data
    where entity_id = $1 and transaction_type in (17, 19) <filterInnerPlaceHolder>
    ...
)
select ... string_agg(fd.file_data, '' order by fd.consensus_timestamp) as file_data
from file_data fd
join latest_create l on fd.consensus_timestamp >= l.consensus_timestamp
where fd.entity_id = $1 and fd.transaction_type in (16,17,19)
    and fd.consensus_timestamp >= l.consensus_timestamp <filterOuterPlaceHolder>
```

With both placeholders empty, `max(consensus_timestamp)` resolves to the **most recent** FILECREATE/FILEUPDATE ever recorded for that file — not the one that existed at the time of the ETHEREUMTRANSACTION. All file chunks from that latest create/update onward are concatenated and returned as `call_data`.

**The correct historical API exists but is not used:**

`FileDataService.getFileData(fileId, timestamp)` in `rest/service/fileDataService.js` (lines 71–76) uses `getFileDataQuery` which constrains with `consensus_timestamp <= $2`, returning the file state at a specific point in time. This is what `getContractById` uses (line 732), but `getContractResultsByTransactionIdOrHash` does not use it.

**Why the existing check is insufficient:**

Line 1183 only calls `utils.isValidUserFileId(ethTransaction?.callDataId)` — this validates that the ID is a non-system file entity ID. It does not add any temporal constraint. Once the check passes, the unconstrained `getLatestFileDataContents` is called unconditionally.

### Impact Explanation

The mirror node's core purpose is to serve an accurate, immutable historical record of Hashgraph activity. When a file referenced by `callDataId` is updated after the ETHEREUMTRANSACTION executes, every subsequent query for that transaction's result returns a `call_data` field that was **never** the actual input to the EVM execution. This breaks:

- **Auditability**: Verifiers replaying or auditing historical EVM executions using mirror node data will use wrong input data.
- **Forensic integrity**: Any tooling (block explorers, compliance systems, smart contract debuggers) that trusts the mirror node's `call_data` for historical transactions will silently operate on falsified data.
- **Protocol trust**: The Hashgraph history is supposed to be tamper-evident; the mirror node undermines this guarantee for this specific code path.

Severity: **Medium–High** (data integrity / historical record falsification; no direct fund loss via mirror node alone, but high impact on downstream consumers).

### Likelihood Explanation

- **No privilege required**: Any user with network access to the mirror node REST API can trigger this by querying `GET /api/v1/contracts/results/{transactionHash}`.
- **Precondition is realistic**: Ethereum-compatible transactions on Hedera/Hiero can reference files via `callDataId` for large call data. Files on Hashgraph are routinely updated (FILEUPDATE transactions). Once any such file is updated, every query for the original transaction returns stale data permanently.
- **Repeatable and persistent**: The incorrect data is returned on every query after the file update, not just once. The attacker does not need to do anything beyond issuing a standard GET request.
- **Detectable only by cross-referencing**: The discrepancy is invisible unless the caller independently fetches the file's historical state and compares — something no standard client does.

### Recommendation

Replace the unconstrained `getLatestFileDataContents` call with the timestamp-bounded `getFileData` call, using the ETHEREUMTRANSACTION's own `consensusTimestamp` as the upper bound:

```js
// In getContractResultsByTransactionIdOrHash (line 1183-1185)
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  // Use the transaction's own consensus timestamp to retrieve the
  // file contents as they existed at execution time, not the latest.
  fileData = await FileDataService.getFileData(
    ethTransaction.callDataId,
    ethTransaction.consensusTimestamp
  );
}
```

Apply the same fix to `getContractResultsByTimestamp` (line 1023), which has the identical pattern. The `getFileData(fileId, timestamp)` method already exists and correctly implements the `<= timestamp` constraint via `getFileDataQuery`.

### Proof of Concept

**Preconditions:**
1. An ETHEREUMTRANSACTION `TX_A` exists on the network with `callDataId = FILE_X` (a user file containing the original call data bytes, e.g., `0xdeadbeef...`).
2. After `TX_A` is confirmed, a FILEUPDATE transaction updates `FILE_X` with new content (e.g., `0xcafebabe...`).

**Trigger (no privileges required):**
```
GET /api/v1/contracts/results/{TX_A_hash}
```

**Observed result:**
```json
{
  "call_data": "0xcafebabe...",   // ← current file contents (post-update)
  ...
}
```

**Expected result:**
```json
{
  "call_data": "0xdeadbeef...",   // ← historical file contents at TX_A's consensus timestamp
  ...
}
```

The returned `call_data` does not match the input that was actually executed on the Hashgraph, with no error or warning to the caller.
