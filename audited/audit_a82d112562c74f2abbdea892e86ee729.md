### Title
Unauthenticated `X-Real-IP` Header Spoofing Allows Audit Log Falsification in Rosetta Tracing Middleware

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` header without verifying that the request originated from a trusted reverse proxy. Because the Rosetta service port (5700) is directly exposed to the network, any unprivileged external user can set an arbitrary `X-Real-IP` value and have it recorded verbatim in the access log, permanently falsifying the audit trail of which IP address initiated each Rosetta API call.

### Finding Description
**Code location:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74.

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)       // line 64 — blindly trusted

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67 — also blindly trusted
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only fallback
    }

    return ipAddress
}
```

The function gives `X-Real-IP` the highest priority with zero validation. The returned value is embedded directly into the log message at line 52–53 of `TracingMiddleware`.

**Why the proxy does not save you:** In the Docker Compose deployment, the nginx proxy correctly sets `X-Real-IP $remote_addr` (line 204 of `docker-compose.yml`), but the Rosetta container also binds port 5700 directly to the host (`ports: - 5700:5700`, line 151 of `docker-compose.yml`). An attacker who connects to port 5700 directly bypasses nginx entirely and can supply any `X-Real-IP` value. The Go application has no mechanism to distinguish a proxied request from a direct one, and no trusted-proxy allowlist exists anywhere in the middleware stack.

The unit tests in `trace_test.go` (lines 52–58) confirm the intended behavior: a client-supplied `X-Real-IP` header is accepted and logged without modification.

### Impact Explanation
Every Rosetta API call is logged with the spoofed IP as the apparent source. An attacker can:
- Attribute their own API activity to a victim's IP address, framing them in the audit trail.
- Mask their own IP from all access logs, making forensic attribution impossible.
- Repeatedly query sensitive endpoints (account balances, transaction history, block data) while the logs record a completely different actor.

This directly corrupts the integrity of the operational audit trail without requiring any credentials or elevated access.

### Likelihood Explanation
Exploitation requires only the ability to send an HTTP request with a custom header — a capability available to any HTTP client (`curl`, browser, script). Port 5700 is exposed by default in the reference Docker Compose configuration. No authentication, rate-limit bypass, or special network position is needed. The attack is trivially repeatable and leaves no reliable forensic trace.

### Recommendation
1. **Strip untrusted proxy headers at the application boundary.** Before calling `getClientIpAddress`, remove `X-Real-IP` and `X-Forwarded-For` from any request that did not arrive from a known trusted proxy CIDR.
2. **Maintain a trusted-proxy allowlist.** Only accept `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` falls within a configured set of trusted proxy addresses (e.g., the nginx container's IP or subnet).
3. **Do not expose the Rosetta port directly.** Remove the `5700:5700` host binding from `docker-compose.yml` so all external traffic must pass through the nginx proxy, which overwrites the header with the real remote address.

### Proof of Concept
**Precondition:** Rosetta is running with port 5700 accessible (default Docker Compose config).

```bash
# Spoof a victim's IP in the Rosetta trace log
curl -s \
  -H "X-Real-IP: 203.0.113.42" \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"}}' \
  http://<rosetta-host>:5700/network/list
```

**Result:** The Rosetta access log records:
```
level=info msg="203.0.113.42 POST /network/list (200) in 12ms"
```
The attacker's real IP never appears. The victim IP `203.0.113.42` is permanently attributed as the initiator of the API call in the audit trail.
