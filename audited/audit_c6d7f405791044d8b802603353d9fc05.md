### Title
Unbounded DB Query Amplification via Unauthenticated `/network/status` Flooding with Host-Header Rate-Limit Bypass

### Summary
Every call to `NetworkStatus()` unconditionally executes a multi-table JOIN query against the database through `addressBookEntryRepo.Entries(ctx)`, with no application-level caching or rate limiting. The only infrastructure-level rate limit (Traefik, `average: 10` keyed on `requestHost`) is trivially bypassed by rotating the HTTP `Host` header, allowing an unprivileged attacker to drive unbounded repeated expensive DB queries and exhaust database resources.

### Finding Description

**Exact code path:**

`rosetta/app/services/network_service.go`, `NetworkStatus()`, lines 59–88:
```go
peers, err := n.addressBookEntryRepo.Entries(ctx)   // line 77 — unconditional DB call
```

`rosetta/app/persistence/address_book_entry.go`, `Entries()`, lines 57–83: executes the `latestNodeServiceEndpoints` query (lines 18–30) — a correlated subquery + 3 JOINs (`address_book_entry`, `node`, `address_book_service_endpoint`) + `GROUP BY` + `string_agg` — up to **twice per request** (once for file_id 101, once for 102 if 101 returns empty).

Additionally, `RetrieveLatest()` (line 72 of `network_service.go`) always hits the DB via `selectLatestWithIndex` (block.go line 199). `RetrieveGenesis()` is cached after first call, so it is not a repeated concern.

**Root cause:** There is no TTL cache, in-memory cache, or application-level rate limiter at the `networkAPIService` layer. The `addressBookEntryRepository` struct holds no state between calls.

**Application middleware stack** (`rosetta/main.go` lines 217–219):
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```
No rate-limiting middleware exists at the application level.

**Infrastructure rate-limit bypass:** The Helm chart (`charts/hedera-mirror-rosetta/values.yaml` lines 157–160) configures Traefik with:
```yaml
- rateLimit:
    average: 10
    sourceCriterion:
      requestHost: true
```
`requestHost: true` keys the rate-limit bucket on the value of the HTTP `Host` header. An attacker rotates through arbitrary `Host` values (`Host: x1.evil.com`, `Host: x2.evil.com`, …), each getting its own fresh 10 req/s bucket — effectively unlimited aggregate throughput. The `inFlightReq: amount: 5` (lines 152–156) limits concurrent requests per source IP but does not bound total request rate, and is bypassed with multiple source IPs.

Furthermore, the Traefik middleware is optional infrastructure; direct access to the rosetta pod (bypassing the ingress) has zero rate limiting.

### Impact Explanation

Each `/network/status` request causes at minimum 2 DB round-trips (one `selectLatestWithIndex`, one or two `latestNodeServiceEndpoints` JOINs). The `latestNodeServiceEndpoints` query performs a correlated subquery against `address_book`, then joins `address_book_entry`, `node`, and `address_book_service_endpoint` with `GROUP BY` and ordered `string_agg`. At high request rates this saturates the DB connection pool and CPU, degrading or denying service to all other mirror node consumers sharing the same PostgreSQL instance (REST API, gRPC, web3). No data is corrupted and no funds are at risk, consistent with the "griefing / no economic damage" classification.

### Likelihood Explanation

Preconditions: none. The endpoint requires no authentication, no API key, and no on-chain state. A single attacker with one machine and rotating `Host` headers bypasses the only rate-limit control. The attack is trivially scriptable (`curl -H "Host: $(uuidgen).x" http://<rosetta>/network/status` in a loop) and fully repeatable. Deployments that expose the rosetta port directly (without Traefik) have zero protection.

### Recommendation

1. **Application-level in-memory cache for `Entries()`**: Cache the result in `addressBookEntryRepository` (or in `networkAPIService`) with a TTL of 60–300 seconds using `sync.Map` or a simple `sync.Mutex`-guarded struct. The address book changes at most a few times per year.
2. **Application-level rate limiter**: Add a `golang.org/x/time/rate` token-bucket middleware scoped to the rosetta HTTP server, applied before routing, independent of Traefik.
3. **Fix the Traefik rate-limit key**: Change `sourceCriterion` from `requestHost: true` to `ipStrategy` (matching the `inFlightReq` criterion) so the rate limit cannot be bypassed by header rotation.
4. **Connection pool limits**: Set a low `max_connections` cap on the DB client used by the rosetta service to bound blast radius.

### Proof of Concept

```bash
# Bypass the per-host rate limit by rotating Host headers
# No authentication, no special privileges required

for i in $(seq 1 10000); do
  curl -s -o /dev/null \
    -H "Host: host${i}.attacker.example" \
    -X POST http://<rosetta-host>/network/status \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' &
done
wait
```

Each iteration gets its own Traefik rate-limit bucket (keyed on `Host`), so all 10 000 requests are forwarded to the application. Each triggers the multi-JOIN `latestNodeServiceEndpoints` query plus `selectLatestWithIndex`, exhausting DB connections and CPU. Observable result: DB CPU spikes to 100%, connection pool exhausted, other API endpoints begin returning errors or timing out.
