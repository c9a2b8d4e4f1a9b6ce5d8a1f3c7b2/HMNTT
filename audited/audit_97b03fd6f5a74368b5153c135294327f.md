### Title
Unbounded Heap Pre-Allocation in OpcodeContext via Gas-Proportional ArrayList Capacity

### Summary
`OpcodeServiceImpl.processOpcodeCall()` initializes an `OpcodeContext` with `new ArrayList<>((int) params.getGas() / 3)`, where `params.getGas()` is read from a historical on-chain transaction. An unauthenticated attacker can identify a publicly-visible historical transaction with a gas limit of up to 15,000,000, replay it via the opcodes endpoint, and force the server to pre-allocate an `ArrayList` backing array of 5,000,000 object references (~20–40 MB) before any EVM execution begins. Combined with full EVM re-execution and per-opcode `Opcode` object accumulation with `stack=true`, `memory=true`, `storage=true`, a single request can cause severe heap pressure.

**Root cause:** The `opcodesSize` passed to the `ArrayList` constructor is derived directly from the gas limit of the replayed historical transaction — a value the attacker controls by choosing which transaction to submit. `params.getGas()` is populated from the stored `contractResult` record for that transaction, which is publicly queryable. There is no cap or sanity check on this value before it is used as an `ArrayList` initial capacity.

**Exploit flow:**
1. Attacker queries the mirror node REST API or a block explorer to find a historical contract transaction with gas limit = 15,000,000 (the Hedera EVM maximum).
2. Attacker sends: `GET /api/v1/contracts/results/{txHash}/opcodes?stack=true&memory=true&storage=true` with `Accept-Encoding: gzip`.
3. `throttleManager.throttleOpcodeRequest()` is called — but this is a rate-limiter on request count, not on per-request memory cost.
4. `buildCallServiceParameters()` reads `gas = 15_000_000` from the DB record.
5. `new OpcodeContext(opcodeRequest, 15_000_000 / 3)` → `new ArrayList<>(5_000_000)` allocates a backing `Object[]` of 5 million slots (~20–40 MB) immediately, before EVM execution.
6. The EVM then re-executes the full transaction. With `stack=true`, `memory=true`, `storage=true`, each of the potentially millions of opcodes appends an `Opcode` object containing full stack snapshots, memory words, and storage maps to that list, causing further heap growth proportional to execution complexity.

**Why checks fail:**
- The `Accept-Encoding: gzip` check only validates a header string — trivially satisfied. [3](#0-2) 
- `throttleManager.throttleOpcodeRequest()` throttles request rate but does not bound the per-request memory footprint. [4](#0-3) 
- There is no cap on `params.getGas()` before it is divided by 3 and used as `ArrayList` capacity. [1](#0-0) 
- No authentication or authorization is required on the endpoint. [5](#0-4) 

### Impact Explanation
A single request targeting a 15M-gas transaction pre-allocates ~20–40 MB of heap (the `Object[]` backing array alone), then accumulates millions of `Opcode` objects each carrying stack/memory/storage snapshots. With a modest number of concurrent requests (e.g., 5–10), heap consumption can spike by hundreds of MB, triggering GC pressure, GC pauses, or OOM on nodes with typical JVM heap configurations (512 MB–2 GB). This satisfies the "≥30% resource increase" threshold without brute force. The gzip-compressed response also means the server must hold the full uncompressed object graph in memory before streaming.

### Likelihood Explanation
The attack requires no credentials — only a valid transaction hash (publicly available on-chain) and the `Accept-Encoding: gzip` header. The endpoint must be operator-enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`), which is the primary deployment gate. When enabled, any external user can execute this. High-gas transactions are common and easily identified via the mirror node's own `/api/v1/contracts/results` listing endpoint. The attack is repeatable and scriptable.

### Recommendation
1. **Cap `opcodesSize`**: Introduce a configurable maximum (e.g., `maxOpcodes = 100_000`) and clamp: `Math.min((int) params.getGas() / 3, maxOpcodes)`.
2. **Cap gas at the service layer**: Reject or clamp requests where `params.getGas()` exceeds a configured threshold before constructing `OpcodeContext`.
3. **Lazy initialization**: Replace `new ArrayList<>(opcodesSize)` with `new ArrayList<>()` or a small default capacity, growing on demand.
4. **Per-request memory budget**: Implement a circuit-breaker that aborts tracing once the accumulated `opcodes` list exceeds a size limit.
5. **Concurrency limit**: Enforce a maximum number of simultaneous in-flight opcode replay requests, not just a rate limit.

### Proof of Concept

```bash
# Step 1: Find a high-gas historical transaction
TX_HASH=$(curl -s "https://<mirror-node>/api/v1/contracts/results?order=desc&limit=25" \
  | jq -r '.results[] | select(.gas_used > 10000000) | .hash' | head -1)

# Step 2: Replay it with full tracing enabled
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"

# Observe: server allocates new ArrayList<>(gas/3) ≈ 5,000,000 entries (~20-40 MB)
# before EVM execution begins, then accumulates Opcode objects throughout execution.
# Repeat with 5-10 concurrent requests to push heap usage >30% above baseline.
```
