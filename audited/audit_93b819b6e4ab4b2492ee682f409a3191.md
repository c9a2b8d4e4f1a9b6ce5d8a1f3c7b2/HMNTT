### Title
Unbounded `otherTransactions` Response and Unlimited DB Fetch in `/block` Endpoint Enable Unauthenticated DoS

### Summary
The `Block()` function in `rosetta/app/services/block_service.go` fetches every transaction in a block's consensus window via an unbounded loop in `FindBetween()`, then places all overflow entries into an uncapped `otherTransactions` slice that is serialized directly into the JSON response. Any unauthenticated caller who identifies a high-transaction block (naturally occurring during network slowdowns) can repeatedly trigger this path, exhausting DB connections, server memory, and network bandwidth, and delaying response delivery to downstream consumers.

### Finding Description

**Code path:**

`Block()` at [1](#0-0)  calls `FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` with no row cap. The block's time window comes from the DB record, not from the caller — but the caller can freely specify any existing block by index or hash via `RetrieveBlock`.

`FindBetween` in the persistence layer [2](#0-1)  runs an unbounded `for start <= end` loop, issuing repeated DB queries with `Limit(batchSize)` (batchSize = 2000) and appending every result to an in-memory slice until the range is exhausted. There is **no total row cap**.

Back in `Block()`, the overflow logic: [3](#0-2)  allocates `otherTransactions` with capacity `len(block.Transactions) - maxTransactionsInBlock` and appends one `TransactionIdentifier{Hash}` per overflow transaction. This slice is returned verbatim in the JSON response with no size limit.

**Root cause / failed assumption:** The design assumes `maxTransactionsInBlock` bounds the response, but it only limits the fully-detailed `block.transactions` array. The `otherTransactions` array — which can be arbitrarily large — is never capped. Additionally, all transactions are fetched from the DB before the cap is applied, so DB and memory cost is always proportional to the total transaction count, not to `maxTransactionsInBlock`.

**Why existing checks fail:**
- `maxTransactionsInBlock` [4](#0-3)  has no documented default in the Rosetta config table and defaults to Go's zero value (`0`) if unconfigured, making `len > 0` always true and routing every transaction into `otherTransactions`.
- The DB `statementTimeout` (20 s) applies per individual query, not to the total loop, so a block with N×2000 transactions issues N queries each within the timeout.
- Traefik rate-limiting (`average: 10`, `inFlightReq: 5`) [5](#0-4)  is optional Helm chart infrastructure; the application itself has zero rate limiting or authentication on `/block`.

### Impact Explanation
A single request to a high-transaction block causes: (a) N unbounded DB round-trips each executing an expensive correlated subquery against `crypto_transfer` and `staking_reward_transfer`; (b) full in-memory materialization of all transactions; (c) a JSON response whose `other_transactions` array grows linearly with transaction count — at 10,000 overflow transactions each with a 66-byte hex hash plus JSON framing, the array alone exceeds 700 KB. Concurrent requests from multiple source IPs saturate the DB connection pool (`maxOpenConnections: 100`) and server memory, delaying or dropping responses to legitimate downstream consumers (e.g., exchange integrations polling the Rosetta API for transaction finality).

### Likelihood Explanation
No privileges, credentials, or special network position are required. The attacker only needs to identify one block with an above-average transaction count — a condition that occurs naturally on Hedera mainnet during any period of network slowdown or catch-up. The block index is publicly enumerable via `/block` with an empty identifier (returns latest) and sequential index decrement. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Cap `otherTransactions` at a hard maximum** (e.g., same value as `maxTransactionsInBlock`) and return an error or truncation indicator if exceeded.
2. **Stop fetching after `maxTransactionsInBlock` rows** in `FindBetween` — pass the limit into the DB query so the loop exits early rather than materializing the entire block.
3. **Set and document a non-zero default** for `maxTransactionsInBlock` in the Rosetta configuration.
4. **Add application-level rate limiting** (e.g., per-IP token bucket) inside the Go HTTP handler, independent of Traefik.
5. **Apply a DB-level `LIMIT`** in `selectTransactionsInTimestampRangeOrdered` equal to `maxTransactionsInBlock + 1` so the database never returns more rows than needed.

### Proof of Concept
```
# 1. Enumerate a high-transaction block (repeat with decrementing index)
curl -s -X POST http://<rosetta-host>:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{}}'
# Note the returned block index I and hash H of a block with many transactions.

# 2. Repeatedly hammer that block
while true; do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":<I>,\"hash\":\"<H>\"}}" \
    -o /dev/null &
done
# Each request triggers an unbounded FindBetween loop and returns a response
# with an otherTransactions array proportional to the block's transaction count,
# exhausting DB connections and saturating server memory/network.
```

```go
func (tr *transactionRepository) FindBetween(ctx context.Context, start, end int64) (
	[]*types.Transaction,
	*rTypes.Error,
) {
	if start > end {
		return nil, hErrors.ErrStartMustNotBeAfterEnd
	}

	db, cancel := tr.dbClient.GetDbWithContext(ctx)
	defer cancel()

	transactions := make([]*transaction, 0)
	for start <= end {
		transactionsBatch := make([]*transaction, 0)
		err := db.
			Raw(
				selectTransactionsInTimestampRangeOrdered,
				sql.Named("start", start),
				sql.Named("end", end),
			).
			Limit(batchSize).
			Find(&transactionsBatch).
			Error
		if err != nil {
			log.Errorf(databaseErrorFormat, hErrors.ErrDatabaseError.Message, err)
			return nil, hErrors.ErrDatabaseError
		}

		transactions = append(transactions, transactionsBatch...)

		if len(transactionsBatch) < batchSize {
			break
		}

		start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
	}

	hashes := make([]string, 0)
	sameHashMap := make(map[string][]*transaction)
	for _, t := range transactions {
		h := t.getHashString()
		if _, ok := sameHashMap[h]; !ok {
			// save the unique hashes in chronological order
			hashes = append(hashes, h)
		}

		sameHashMap[h] = append(sameHashMap[h], t)
	}

	res := make([]*types.Transaction, 0, len(sameHashMap))
	for _, hash := range hashes {
		sameHashTransactions := sameHashMap[hash]
		transaction, err := tr.constructTransaction(sameHashTransactions)
		if err != nil {
			return nil, err
		}
		res = append(res, transaction)
	}
	return res, nil
}
```
