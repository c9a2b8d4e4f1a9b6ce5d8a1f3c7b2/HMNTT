### Title
Unauthenticated Full-Table Scan DoS via Missing Timestamp Filter in `getBalances()`

### Summary
When `GET /api/v1/balances` is called without a `timestamp` query parameter, `getBalances()` in `rest/balances.js` takes the `else` branch and constructs a query whose `entity_balance` CTE performs an unbounded full scan of the `entity` table filtered only by `type in ('ACCOUNT', 'CONTRACT')`. A correlated `getTokenAccountBalanceSubQuery` is then executed per returned row. Any unauthenticated user can trigger this path repeatedly and concurrently, holding database connections for the duration of each scan and exhausting the connection pool.

### Finding Description

**Exact code location:** `rest/balances.js`, `getBalances()`, lines 113–150.

When `tsQuery` is falsy (no `timestamp` parameter supplied), execution falls into the `else` branch:

```js
// rest/balances.js lines 133–149
} else {
  // use current balance from entity table when there's no timestamp query filter
  const conditions = [accountQuery, pubKeyQuery, balanceQuery].filter(Boolean).join(' and ');
  const whereClause = conditions && `where ${conditions}`;
  const tokenBalanceSubQuery = getTokenAccountBalanceSubQuery(order);
  sqlParams = utils.mergeParams(tsParams, accountParams, pubKeyParams, balanceParams, params);
  sqlQuery = `
    with entity_balance as (
      select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
      from entity
      where type in ('ACCOUNT', 'CONTRACT')
    )
    select ab.*, (${tokenBalanceSubQuery}) as token_balances
    from entity_balance ab
    ${whereClause}
    order by ab.account_id ${order}
    ${limitQuery}`;
}
``` [1](#0-0) 

The `entity_balance` CTE has **no LIMIT of its own**; it reads every row in `entity` matching `type in ('ACCOUNT', 'CONTRACT')`. The outer `limitQuery` is applied only after the CTE is evaluated and the result set is sorted by `account_id`. In PostgreSQL versions prior to 12, CTEs are always materialized (optimization fences), so the full scan is unconditional. Even in PG 12+ where the CTE may be inlined, the `ORDER BY ab.account_id … LIMIT N` still requires the planner to consider all matching rows before it can emit the top-N, unless a perfectly covering index is available and chosen.

The correlated subquery injected per output row is:

```js
// rest/balances.js lines 308–318
const getTokenAccountBalanceSubQuery = (order) => {
  return `
    select json_agg(json_build_object('token_id', token_id, 'balance', balance))
    from (
      select token_id, balance
      from token_account ta
      where ta.account_id = ab.account_id and ta.associated is true
      order by token_id ${order}
      limit ${tokenBalanceLimit.multipleAccounts}
    ) as account_token_balance`;
};
``` [2](#0-1) 

**Root cause / failed assumption:** The code assumes that the outer `limitQuery` is sufficient to bound query cost. It is not: the CTE materializes (or the planner must sort) the entire ACCOUNT/CONTRACT population before the limit is applied, and no server-side query timeout or rate-limiting guard is present in this code path.

**Why existing checks fail:**
- `acceptedBalancesParameters` (lines 365–372) validates parameter names but does not require a timestamp to be present.
- `balanceFilterValidator` (lines 359–363) validates operator/value correctness but imposes no mandatory-filter requirement.
- The `limitQuery` caps rows *returned*, not rows *scanned*. [3](#0-2) 

### Impact Explanation
On a production Hedera mirror node the `entity` table contains tens of millions of ACCOUNT/CONTRACT rows. Each filterless request forces a sequential scan of that table plus a sort pass, holding a PostgreSQL connection for seconds to tens of seconds. Sending a modest number of concurrent requests (e.g., 20–50) saturates the connection pool, causing all subsequent API calls — including those from legitimate users — to queue or fail. Because the endpoint is public and unauthenticated, this constitutes a non-network-based denial-of-service against the REST API layer and its shared database.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and a single HTTP client. The trigger URL is trivially constructed (`GET /api/v1/balances` with no parameters). It is repeatable at will and can be scripted with any HTTP tool. The endpoint is documented in the public OpenAPI spec (`rest/api/v1/openapi.yml`, line 391), making discovery trivial. [4](#0-3) 

### Recommendation
1. **Require a timestamp filter** when no narrowing filter (account ID, public key, balance) is present, or return a 400 error when the no-timestamp path would produce an unbounded scan.
2. **Add a LIMIT to the CTE itself** so the database never materializes more rows than the outer limit requires:
   ```sql
   with entity_balance as (
     select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
     from entity
     where type in ('ACCOUNT', 'CONTRACT')
     order by id DESC
     limit <max_limit>
   )
   ```
3. **Enforce a statement timeout** on the PostgreSQL connection used by the REST service (e.g., `SET statement_timeout = '5s'`) so runaway queries are killed automatically.
4. **Apply rate limiting** at the API gateway or middleware layer for this endpoint.

### Proof of Concept
```bash
# Single expensive request (no timestamp, no other filter)
curl "https://<mirror-node-host>/api/v1/balances"

# Concurrent flood to exhaust DB connection pool
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/balances" &
done
wait
# Subsequent legitimate requests will time out or receive 503
curl "https://<mirror-node-host>/api/v1/balances?account.id=0.0.98"
```

Each concurrent request causes the database to execute:
```sql
with entity_balance as (
  select id as account_id, balance, balance_timestamp as consensus_timestamp, public_key
  from entity
  where type in ('ACCOUNT', 'CONTRACT')   -- full table scan, no limit
)
select ab.*, (<correlated token_account subquery>) as token_balances
from entity_balance ab
order by ab.account_id DESC
limit 25;
```
