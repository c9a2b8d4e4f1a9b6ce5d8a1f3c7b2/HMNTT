### Title
Deleted-Entity Bypass via Long-Zero EVM Address in `EntityIdServiceImpl.findByEvmAddress()`

### Summary
`EntityIdServiceImpl.findByEvmAddress()` resolves long-zero EVM addresses (first 12 bytes all zero) directly from the address bytes via `DomainUtils.fromEvmAddress()`, short-circuiting the database lookup entirely. Because the database query in `EntityRepository.findByEvmAddress()` is the only place the `deleted <> true` guard is enforced, any long-zero address that encodes a deleted entity's numeric ID bypasses the deletion check and returns a valid `EntityId` for a deleted contract. Any unprivileged user can trigger this by submitting a transaction that references such an address.

### Finding Description

    if (isLongZeroAddress(evmAddress)) {
        final var num = Longs.fromByteArray(Arrays.copyOfRange(evmAddress, 12, 20));
        return EntityId.of(commonProperties.getShard(), commonProperties.getRealm(), num);

`isLongZeroAddress()` — lines 405–411:

```java
public static boolean isLongZeroAddress(byte[] evmAddress) {
    if (evmAddress == null || evmAddress.length != EVM_ADDRESS_LENGTH) { return false; }
    return Arrays.equals(MIRROR_PREFIX, 0, 12, evmAddress, 0, 12); // MIRROR_PREFIX = new byte[12]
}
```

**Root cause:** The `Optional.ofNullable(...).or(...)` chain means the `.or()` branch — which calls `entityRepository.findByEvmAddress(evmAddress)` with the SQL guard `deleted <> true` — is **never reached** when `DomainUtils.fromEvmAddress()` returns a non-null value. For any 20-byte address whose first 12 bytes are zero (a long-zero address), `fromEvmAddress()` always returns a non-null `EntityId` constructed purely from the address bytes, with no database interaction and no deletion check.

**Failed assumption:** The code assumes that `DomainUtils.fromEvmAddress()` only returns a valid `EntityId` for entities that are live. In reality it returns an `EntityId` for any numerically valid long-zero address, regardless of whether the corresponding entity has been deleted.

**Exploit flow:**
1. Contract `0.0.X` is created and later deleted (`deleted = true` in the `entity` table).
2. Its canonical long-zero EVM address is `0x0000000000000000000000000000000000000000` with the last 8 bytes encoding `X` big-endian.
3. An attacker submits any transaction (e.g., `ContractCall`, `EthereumTransaction`) referencing this EVM address in the `ContractID.evm_address` field.
4. The consensus nodes include the transaction in the record stream (even as a failed/rejected transaction — the record still exists).
5. The importer calls `lookup(ContractID)` → `findByEvmAddress(bytes, throwRecoverableError)`.
6. `DomainUtils.fromEvmAddress()` detects the long-zero pattern and returns `EntityId.of(shard, realm, X)` immediately.
7. The `.or(...)` branch with the `deleted <> true` SQL guard is never executed.
8. The importer records the transaction as associated with the deleted entity `0.0.X`.

### Impact Explanation
The mirror node importer writes incorrect entity associations into its database for transactions referencing deleted contracts via long-zero EVM addresses. Downstream mirror node API consumers (REST, GraphQL, gRPC) will observe transactions linked to deleted entities, corrupting the historical record. This can mislead block explorers, wallets, and dApps that rely on mirror node data for contract state and transaction history. The scope is limited to mirror node data integrity (not consensus-layer funds), but the mirror node is the primary source of truth for off-chain consumers.

### Likelihood Explanation
The precondition — a deleted contract with a known numeric ID — is trivially satisfied on any public Hedera network; contract deletions are public record. Constructing the long-zero EVM address requires only knowledge of the entity number. Submitting a transaction referencing that address requires no special privileges; any Hedera account can submit a `ContractCall` or `EthereumTransaction`. The attack is repeatable and cheap (only transaction fees), and requires no cryptographic capability.

### Recommendation
In `EntityIdServiceImpl.findByEvmAddress()`, do not short-circuit the database lookup for long-zero addresses. After resolving the numeric ID from the address, always verify the entity is not deleted by querying the database:

```java
private Optional<EntityId> findByEvmAddress(byte[] evmAddress, boolean throwRecoverableError) {
    EntityId fromAddress = DomainUtils.fromEvmAddress(evmAddress);
    if (fromAddress != null) {
        // Verify the resolved entity is not deleted before trusting the address-derived ID
        boolean exists = entityRepository.findById(fromAddress.getId())
            .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))
            .isPresent();
        if (!exists) {
            if (throwRecoverableError) {
                Utility.handleRecoverableError("Entity not found or deleted for EVM address {}",
                    Hex.encodeHexString(evmAddress));
            }
            return Optional.empty();
        }
        return Optional.of(fromAddress);
    }
    var id = entityRepository.findByEvmAddress(evmAddress).map(EntityId::of);
    if (id.isEmpty() && throwRecoverableError) {
        Utility.handleRecoverableError("Entity not found for EVM address {}", Hex.encodeHexString(evmAddress));
    }
    return id;
}
```

Alternatively, unify the lookup so that long-zero addresses also go through `entityRepository.findByEvmAddress()` (which already has `deleted <> true`), removing the special-case fast path entirely.

### Proof of Concept

**Preconditions:**
- Hedera network with a deleted contract at entity ID `0.0.1234` (`deleted = true` in mirror node DB).
- Long-zero EVM address: `0x00000000000000000000000000000000000004D2` (last 8 bytes = `0x00000000000004D2` = 1234 decimal).

**Steps:**
1. Submit a `ContractCall` transaction to the Hedera network with `ContractID.evm_address = 0x00000000000000000000000000000000000004D2`.
2. The transaction is included in the record stream (as a failed call — the record still exists).
3. The mirror node importer processes the record and calls `EntityIdServiceImpl.lookup(ContractID)`.
4. This calls `findByEvmAddress(bytes{0x00...04D2}, true)`.
5. `DomainUtils.fromEvmAddress()` detects `isLongZeroAddress = true`, returns `EntityId.of(0, 0, 1234)`.
6. The `.or(...)` branch (with `deleted <> true` SQL) is never reached.
7. The importer stores the transaction record linked to entity `0.0.1234` (deleted contract).
8. Query the mirror node REST API: `GET /api/v1/contracts/0.0.1234/results` — the transaction appears, incorrectly associating activity with the deleted contract.
