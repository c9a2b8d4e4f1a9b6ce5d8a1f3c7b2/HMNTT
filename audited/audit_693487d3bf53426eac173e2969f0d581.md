### Title
Unauthenticated KYC Status Disclosure via `/accounts/{accountId}/tokens`

### Summary
The `/accounts/{accountId}/tokens` endpoint returns the `kyc_status` field for every token relationship of any account without requiring authentication or ownership verification. The `authHandler` middleware only rejects requests with *invalid* credentials — requests with *no* credentials pass through freely — meaning any anonymous external user can enumerate the KYC compliance status of any account on the network.

### Finding Description

**Route registration** — no auth guard is applied: [1](#0-0) 

**`authHandler` middleware** — when no `Authorization` header is present, `credentials` is `null` and the handler simply `return`s, allowing the request to continue unauthenticated: [2](#0-1) 

**Controller** — no ownership or identity check; any `accountId` is accepted: [3](#0-2) 

**SQL query** — `kyc_status` is explicitly selected for every token relationship: [4](#0-3) 

**`getTokenAccounts`** — `kyc_status` is resolved from the DB row or the token cache and attached to every returned record: [5](#0-4) 

**View model** — `kyc_status` is serialised directly into the JSON response: [6](#0-5) 

Root cause: the `authHandler` is not an access-control gate — it is only a credential-upgrade mechanism (it sets a higher rate-limit for known users). It was never designed to block unauthenticated traffic. No separate ownership or role check exists anywhere in the call chain for this endpoint.

### Impact Explanation
`kyc_status` is a regulated compliance attribute indicating whether an account has passed Know-Your-Customer verification for a specific token. Exposing it publicly allows any attacker to:
- Determine which accounts are KYC-approved for security tokens or regulated assets.
- Build surveillance profiles linking account IDs to their compliance state.
- Target accounts that are KYC-approved (higher-value targets) for phishing or social engineering.

This constitutes a direct information-disclosure vulnerability with privacy and regulatory implications (e.g., GDPR, AML/KYC confidentiality obligations).

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and a single HTTP GET request. The account ID space on Hedera is sequential and enumerable. An attacker can trivially iterate over account IDs and harvest KYC status for the entire network. The attack is fully automated, repeatable, and leaves no distinguishing trace beyond normal API traffic.

### Recommendation
1. **Short-term**: Strip `kyc_status` from the public response, or replace it with a redacted sentinel value (e.g., `null`) for unauthenticated callers.
2. **Medium-term**: Introduce an ownership check in `getTokenRelationships` so that only the account owner (authenticated via a signed challenge or operator key) can retrieve their own token relationships including KYC status.
3. **Middleware**: If certain fields must remain restricted, add a field-level redaction layer in `TokenRelationshipViewModel` that inspects the caller's identity before including sensitive compliance fields.

### Proof of Concept
```
# No credentials required — works from any machine
curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.12345/tokens" \
  | jq '.tokens[] | {token_id, kyc_status}'
```
Expected response fragment:
```json
{
  "token_id": "0.0.9999",
  "kyc_status": "GRANTED"
}
```
Repeat for any victim account ID. No authentication header is needed; the `authHandler` passes the request through when `Authorization` is absent. [7](#0-6)
