### Title
Unbounded Timestamp Range Scan via Default-Disabled `bindTimestampRange` Allows DB Connection Pool Exhaustion

### Summary
`getTransactions()` calls `parseTimestampFilters` with `validateRange=false` and `allowOpenRange=true`, meaning no upper-bound enforcement is applied to user-supplied timestamp parameters. Because `bindTimestampRange` defaults to `false`, the range is forwarded to the database unchanged. An unauthenticated attacker can submit requests with no upper timestamp bound (or no timestamp at all), causing `getTransactionTimestamps()` to issue queries with no upper bound on `consensus_timestamp`, which — when multiplied across concurrent requests — exhausts the 10-connection pool and stalls all subsequent transaction visibility.

### Finding Description

**Exact code path:**

`getTransactions()` in `rest/transactions.js` line 674:
```js
const timestampRange = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
``` [1](#0-0) 

The 6th argument (`validateRange`) is `false` and the 4th (`allowOpenRange`) is `true`. Inside `parseTimestampFilters` (`rest/utils.js` lines 1657–1665), the `maxTimestampRange` check is gated on `validateRange`:
```js
if (validateRange) {
  if (difference !== null && (difference > maxTimestampRangeNs || difference <= 0n)) {
    throw new InvalidArgumentError(...);
  }
}
``` [2](#0-1) 

Because `validateRange=false`, this block is never reached for the `/api/v1/transactions` endpoint.

With `allowOpenRange=true`, a single-sided range like `?timestamp=gte:0` is accepted and produces `Range(0n, null, '[]')` — no upper bound. With no timestamp parameter at all, `range=null` is returned.

In `getTransactionTimestamps()` (line 464–468), `bindTimestampRange` is called:
```js
const {range, next} = await bindTimestampRange(timestampRange.range, order);
``` [3](#0-2) 

`bindTimestampRange` in `rest/timestampRange.js` line 20 immediately returns the range unchanged when the config flag is `false`:
```js
if (!queryConfig.bindTimestampRange) {
  return {range};
}
``` [4](#0-3) 

The default value of `bindTimestampRange` is `false` per the documented configuration: [5](#0-4) 

`buildTimestampQuery` (`rest/utils.js` lines 1727–1736) only adds SQL conditions for non-null bounds:
```js
if (range.begin) { conditions.push(`${column} >= ?`); }
if (range.end)   { conditions.push(`${column} <= ?`); }
``` [6](#0-5) 

With `range=null` or `Range(0n, null)`, no upper-bound condition is emitted. The resulting SQL in `getTransactionTimestampsQuery` (lines 537–606) — which can involve FULL OUTER JOINs across `transaction`, `crypto_transfer`, `token_transfer`, and `entity_transaction` — executes with no upper timestamp bound. [7](#0-6) 

### Impact Explanation

The DB pool has `maxConnections=10` and `statementTimeout=20000ms` (20 seconds) by default. [8](#0-7) 

An attacker sending 10 concurrent unbounded requests saturates all pool connections for up to 20 seconds each. Every legitimate request during that window queues or fails. With repeated waves of requests, the service is effectively unavailable. When `accountQuery` or `creditDebitQuery` filters are also supplied, the query becomes a multi-table FULL OUTER JOIN with no timestamp ceiling, dramatically increasing per-query cost and making the 20-second timeout more likely to be hit.

### Likelihood Explanation

No authentication is required. The endpoint is public. The exploit requires only a standard HTTP client. The attacker needs to send 10 concurrent GET requests — trivially automated with `curl`, `ab`, or any scripting language. The condition is fully reproducible and repeatable. The default deployment (`bindTimestampRange=false`) is the vulnerable configuration, and the Helm chart for v2 explicitly sets it to `true` only for that variant, meaning standard deployments remain exposed. [9](#0-8) 

### Recommendation

1. **Enable `bindTimestampRange: true` by default** in the base `application.yml` so that all deployments enforce the 60-day window without operator action.
2. **Move `validateRange` to `true`** in the `parseTimestampFilters` call inside `getTransactions()` (line 674), or add a dedicated upper-bound enforcement step before `buildTimestampQuery` is called.
3. As a defense-in-depth measure, enforce a mandatory upper-bound in `getTransactionTimestamps()` when `timestampRange.range?.end` is null, substituting `nowInNs()` unconditionally rather than only when `bindTimestampRange` is enabled.

### Proof of Concept

```bash
# Saturate all 10 pool connections with unbounded scans
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions?timestamp=gte:0&order=asc" &
done
wait

# All subsequent legitimate requests will queue or time out for up to 20 seconds
curl -v "http://<mirror-node-host>:5551/api/v1/transactions"
```

With `account.id` added to trigger the FULL OUTER JOIN path:
```
GET /api/v1/transactions?timestamp=gte:0&account.id=0.0.2&order=asc
```
This forces the multi-table join variant in `getTransactionTimestampsQuery`, maximizing per-query cost and connection hold time.
