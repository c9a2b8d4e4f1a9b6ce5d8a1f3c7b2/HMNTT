### Title
Cache Stampede / Thundering Herd in `getCachedTokens()` Allows Unprivileged DB DoS

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` contains no in-flight request deduplication. Because Node.js yields at every `await`, N concurrent callers that each find the same token IDs absent from the cache will each independently issue `tokenCacheQuery` against the database before any of them has had a chance to populate the cache. This multiplies DB load linearly with concurrency and can be triggered by any unprivileged external user.

### Finding Description
**Exact code path:** `rest/service/tokenService.js`, `getCachedTokens()`, lines 136–161.

**Root cause / failed assumption:** The implementation assumes that the synchronous cache check (line 140) and the subsequent DB query (line 152) are effectively atomic. They are not. Node.js yields control at every `await`. When Request 1 suspends at line 152, Requests 2…N resume from the event queue, each independently execute lines 139–151, each find the same token IDs absent from `tokenCache` (because Request 1 has not yet returned and populated it at line 156), and each independently issue `tokenCacheQuery`. There is no "pending promises" map, no mutex, no coalescing — nothing prevents this.

**Exploit flow:**
1. Attacker identifies any token IDs not yet in the LRU cache (e.g., newly minted tokens, or after natural LRU eviction).
2. Attacker sends N concurrent HTTP requests to `/api/v1/accounts/{id}/tokens` (or any endpoint that calls `getTokenAccounts` → `getCachedTokens`).
3. All N requests pass through `getTokenAccounts` (line 104), reach `getCachedTokens`, perform the synchronous cache check (lines 139–146), find the same IDs uncached, and each call `super.getRows(TokenService.tokenCacheQuery, ...)` before any of them returns.
4. The database receives N identical `SELECT decimals, freeze_status, kyc_status, token_id FROM token WHERE token_id = ANY($1)` queries simultaneously.
5. DB connection pool is saturated; legitimate queries queue or time out.

**Why existing checks are insufficient:**
- The `quickLru` cache (lines 12–14) only prevents duplicate DB queries across *sequential* calls. It provides no protection against concurrent calls that all check the cache before any one of them populates it.
- The rate-limiting found in the codebase (`web3/src/main/java/.../ThrottleConfiguration.java`, `ThrottleManagerImpl.java`) applies to the Java/Spring `web3` module, not to this Node.js REST service.
- The Traefik `inFlightReq` / `rateLimit` middleware found in `charts/hedera-mirror-rosetta/values.yaml` applies to the Rosetta service, not the REST API.
- No application-level rate limiting or concurrency guard exists in the REST service for this code path. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Every concurrent cache-miss request independently queries the database. With N=50 concurrent requests (easily achievable from a single machine with `Promise.all`), the database receives 50 identical queries. The `token` table is queried with `WHERE token_id = ANY($1)` — a full index scan per query. Under sustained attack, the DB connection pool exhausts, query latency spikes for all users, and the mirror node REST API becomes unavailable. This is a non-network-based DoS: it does not require saturating bandwidth; it only requires moderate HTTP concurrency.

### Likelihood Explanation
- **Precondition:** No authentication or privilege is required. Any public HTTP client suffices.
- **Ease:** A single attacker script using `Promise.all([...Array(100)].map(() => fetch(...)))` is sufficient.
- **Repeatability:** After the cache is populated, the attacker can force re-eviction by requesting a large number of distinct token IDs (the LRU has a finite `maxSize`), then repeat the stampede.
- **Detectability:** The requests are individually legitimate; no single request looks anomalous.

### Recommendation
Introduce a **promise coalescing / in-flight deduplication** map. Before issuing a DB query, check whether a promise for the same set of token IDs is already in-flight; if so, `await` that existing promise instead of issuing a new one. A minimal fix:

```js
// module-level
const inFlightTokenQueries = new Map(); // tokenId → Promise<CachedToken>

async getCachedTokens(tokenIds) {
  const cachedTokens = new Map();
  const uncachedTokenIds = [];
  const pendingPromises = [];

  tokenIds.forEach((tokenId) => {
    const cached = tokenCache.get(tokenId);
    if (cached) {
      cachedTokens.set(tokenId, cached);
    } else if (inFlightTokenQueries.has(tokenId)) {
      pendingPromises.push(inFlightTokenQueries.get(tokenId).then(t => cachedTokens.set(tokenId, t)));
    } else {
      uncachedTokenIds.push(tokenId);
    }
  });

  if (uncachedTokenIds.length > 0) {
    const queryPromise = super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds])
      .then(rows => { rows.forEach(row => { tokenCache.set(row.token_id, new CachedToken(row)); }); })
      .finally(() => uncachedTokenIds.forEach(id => inFlightTokenQueries.delete(id)));
    uncachedTokenIds.forEach(id => inFlightTokenQueries.set(id, queryPromise));
    await queryPromise;
    uncachedTokenIds.forEach(id => { const t = tokenCache.get(id); if (t) cachedTokens.set(id, t); });
  }

  if (pendingPromises.length > 0) await Promise.all(pendingPromises);
  return cachedTokens;
}
```

Alternatively, use a library such as `p-memoize` with a short TTL to coalesce concurrent identical lookups automatically.

### Proof of Concept
```js
// Requires: mirror node REST API running locally, token IDs 300, 400, 500 exist, cache is cold
const N = 50;
const url = 'http://localhost:5551/api/v1/accounts/0.0.98/tokens';

// Fire N concurrent requests before any one completes
const results = await Promise.all(
  Array.from({length: N}, () => fetch(url).then(r => r.json()))
);

// Observe in DB logs (pg_stat_activity or slow query log):
// N identical queries: SELECT decimals, freeze_status, kyc_status, token_id FROM token WHERE token_id = ANY($1)
// Each issued independently, DB load multiplied by N
console.log(`Issued ${N} concurrent requests; check DB for ${N} duplicate tokenCacheQuery executions`);
```

To force cache eviction before each run, request enough distinct token IDs to exceed `config.cache.token.maxSize`, then repeat.
