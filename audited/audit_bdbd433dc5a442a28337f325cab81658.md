### Title
Unbounded Re-subscription Loop via `onBackpressureBuffer` Overflow Causes Repeated GC Pressure

### Summary
`SharedTopicListener.listen()` applies a per-subscriber `onBackpressureBuffer` with `BufferOverflowStrategy.ERROR`. When a slow subscriber's buffer fills to `maxBufferSize` (default 16,384, max 65,536), the subscription terminates with an `OverflowException`. No rate limiting or cooldown exists on re-subscription, so an unprivileged attacker can immediately re-subscribe in a tight loop, repeatedly allocating and discarding large buffer queues and causing sustained GC pressure on the mirror node JVM.

### Finding Description

**Exact code location:**
`grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedTopicListener.java`, `listen()`, line 24:

```java
.onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
``` [1](#0-0) 

`maxBufferSize` is bounded between 8,192 and 65,536 (default 16,384): [2](#0-1) 

**Root cause:** The buffer is allocated fresh per-subscriber on each call to `listen()`. Reactor's `onBackpressureBuffer` creates a queue that grows up to `maxBufferSize` elements. When the queue is full and a new element arrives, `BufferOverflowStrategy.ERROR` signals `OverflowException` and terminates the subscription. The failed assumption is that a terminated subscriber will not immediately re-subscribe; there is no re-subscription rate limit, no per-IP connection throttle at the application layer, and no backoff enforced on the server side.

**Exploit flow:**
1. Attacker opens a gRPC connection and calls `subscribeTopic` targeting a high-traffic topic.
2. The call reaches `ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicListener.listen()` → `SharedTopicListener.listen()`. [3](#0-2) 

3. The attacker's subscriber reads no messages (or reads very slowly). The shared upstream (`RedisTopicListener` or `SharedPollingTopicListener`) pushes messages into the per-subscriber buffer. [4](#0-3) 

4. The buffer fills to `maxBufferSize`; `OverflowException` is thrown; the subscription is torn down and all buffered `TopicMessage` objects become garbage.
5. The attacker immediately re-subscribes. Steps 3–5 repeat in a tight loop.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection` (set in `GrpcConfiguration`) limits simultaneous in-flight calls per connection, but does not prevent sequential rapid re-subscription. [5](#0-4) 

- `subscriberCount` is a metric gauge only; it imposes no cap. [6](#0-5) 

- No authentication is required to call `subscribeTopic`.
- No server-side backoff or ban is applied after repeated `OverflowException` terminations.

### Impact Explanation
Each re-subscription cycle allocates a Reactor `SpscArrayQueue` (or equivalent) plus up to 16,384 `TopicMessage` heap objects (each containing byte arrays for `message` and `runningHash`). On a high-traffic topic producing ~1,000 msg/s, the buffer fills in ~16 seconds, producing a ~16 MB allocation burst per cycle. A single attacker looping at this rate generates ~1 MB/s of short-lived heap garbage. Multiple concurrent connections from the same attacker (each under the per-connection call limit) multiply this linearly. The result is elevated minor/major GC frequency, increased GC pause times, and potential throughput degradation for legitimate subscribers — a griefing-class denial of service with no economic damage to network participants.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond a valid topic ID (which is public), and no sophisticated tooling — a standard gRPC client suffices. Any high-traffic topic (e.g., a popular HCS application topic) provides the necessary message rate. The loop can be automated trivially. The attacker does not need to sustain a large number of concurrent connections; sequential rapid re-subscription from a single connection is sufficient to maintain continuous GC pressure.

### Recommendation
1. **Server-side re-subscription rate limiting**: Track re-subscription attempts per client IP (or per gRPC peer identity) and enforce a minimum interval (e.g., exponential backoff) before allowing a new subscription after an `OverflowException` termination.
2. **Global subscriber cap**: Enforce a hard limit on `subscriberCount` and reject new subscriptions when the cap is reached, returning `RESOURCE_EXHAUSTED`.
3. **Per-IP connection limiting**: Apply connection-rate limiting at the load balancer or Netty level to bound the number of new connections per IP per second.
4. **Overflow strategy**: Consider `BufferOverflowStrategy.DROP_LATEST` or `DROP_OLDEST` instead of `ERROR` for live subscriptions, so a slow subscriber is penalized by message loss rather than subscription termination (which enables the re-subscription loop).

### Proof of Concept
```python
import grpc
import time
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.python import basic_types_pb2

channel = grpc.insecure_channel("mirror-node:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_service_pb2.ConsensusTopicQuery()
query.topicID.topicNum = <HIGH_TRAFFIC_TOPIC_NUM>  # e.g., a busy HCS topic

while True:
    try:
        # Subscribe but never consume — iterator.next() is never called
        iterator = stub.subscribeTopic(query)
        # Hold the subscription open briefly, then abandon it
        time.sleep(0.1)
        iterator.cancel()
    except Exception:
        pass
    # Immediately re-subscribe; server allocates a new buffer each time
```

Repeat at high frequency across multiple threads/connections to amplify GC pressure proportionally.
