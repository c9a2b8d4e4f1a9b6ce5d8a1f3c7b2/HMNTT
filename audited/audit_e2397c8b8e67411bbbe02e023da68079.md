### Title
Unauthenticated Aggregate Metric Dilution via Success-Request Flooding in `metricsMiddleware`

### Summary
The `metricsMiddleware` in `rest/middleware/metricsHandler.js` maintains unlabeled aggregate counters (`api_all_success`, `api_all_errors`, `api_all_server_error`) that any unauthenticated external user can influence. Because no rate limiting exists in the REST middleware stack, an attacker can flood any public API endpoint with valid requests to inflate `allSuccessCounter`, diluting the aggregate error rate and masking 5xx failures caused by Hashgraph history tampering. The per-route breakdown exists but the aggregate counters — the most common alerting signal — are fully manipulable without any privilege.

### Finding Description
**Exact code path:**

In `rest/middleware/metricsHandler.js`, the `metricsMiddleware` function (lines 185–204) fires on every `res.on('finish')` event and unconditionally increments one of three unlabeled aggregate counters:

```js
// lines 194–204
allRequestCounter.add(1);
if (res.statusCode >= 200 && res.statusCode < 300) {
  allSuccessCounter.add(1);                  // api_all_success
} else if (res.statusCode >= 400 && res.statusCode < 500) {
  allClientErrorCounter.add(1);
  allErrorCounter.add(1);
} else if (res.statusCode >= 500) {
  allServerErrorCounter.add(1);              // api_all_server_error
  allErrorCounter.add(1);                    // api_all_errors
}
```

These counters carry **no labels** (no `path`, no `method`, no `code`). They aggregate across every route in the application.

**Root cause — failed assumption:**
The design assumes that aggregate counters provide a meaningful health signal. This assumption fails because the counters are writable by any unauthenticated HTTP client and there is no rate limiting in the middleware stack. In `rest/server.js` (lines 67–92), the full middleware chain is: `urlencoded → json → cors → compression → httpContext → requestLogger → authHandler → metricsHandler`. No rate-limiting middleware is present at any layer.

**Exploit flow:**

1. Hashgraph history tampering (e.g., corrupted record files) causes the importer or DB layer to surface errors, resulting in 5xx responses on history endpoints (e.g., `/api/v1/transactions`, `/api/v1/blocks`). Each such response increments `allServerErrorCounter` and `allErrorCounter`.
2. The attacker (zero privileges required) concurrently floods any cheap, always-200 endpoint — e.g., `GET /api/v1/accounts` or `GET /api/v1/tokens` — with high-volume requests. Each 200 response increments `allSuccessCounter`.
3. The Prometheus scrape of `api_all_server_error / api_all_request` (or `api_all_errors / api_all_request`) drops toward zero because the denominator and `api_all_success` are dominated by the flood traffic.
4. Any alert rule of the form `rate(api_all_server_error[5m]) / rate(api_all_request[5m]) > threshold` fails to fire.

**Why existing checks are insufficient:**

- The metrics *read* endpoint is protected by basic auth (lines 156–164), but this only gates who can *read* the metrics — it does nothing to prevent external requests from *writing* to the counters.
- Per-route instruments (`requestTotalCounter` with `{method, path, code}` labels, lines 207–210) do preserve per-route signal, but these are secondary instruments. Aggregate counters are the primary, commonly-alerted-on signal and are fully exposed.
- `authHandler` (lines 85–86, `rest/server.js`) only sets a custom response-size limit for authenticated users; it does not block or throttle unauthenticated requests.

### Impact Explanation
An operator relying on aggregate error-rate alerts (`api_all_server_error`, `api_all_errors`) will receive no alert while Hashgraph history endpoints are returning 5xx responses due to tampering. This directly undermines the integrity-monitoring layer for the ledger history: tampering that corrupts transaction or block records can proceed undetected for as long as the attacker sustains the flood. The impact is a silent failure of the observability plane protecting Hashgraph history integrity.

### Likelihood Explanation
The attack requires zero privileges and zero authentication. Any internet-accessible deployment of the mirror-node REST API is vulnerable. The flood traffic can be generated from a single host using standard HTTP tooling (`wrk`, `ab`, `curl` loops). The attacker does not need to know the metrics scrape interval — sustained flooding at even modest rates (hundreds of requests per minute) is sufficient to dominate the unlabeled aggregate counters. The attack is fully repeatable and can be automated.

### Recommendation
1. **Add per-route labels to aggregate counters** or deprecate the unlabeled aggregates in favour of the already-labelled `requestTotalCounter`. Alert rules should use `sum by (path, code)(rate(api_request_total[5m]))` rather than the unlabeled aggregates.
2. **Introduce rate limiting** in the REST middleware stack (e.g., `express-rate-limit`) before `metricsHandler` so that unauthenticated clients cannot sustain the volume needed to dilute counters.
3. **Add a separate, label-scoped error-rate alert** that fires on `rate(api_request_total{code=~"5.."}[5m])` per route, making it impossible to dilute with cross-route success traffic.
4. Optionally, track a **per-route error ratio gauge** server-side so that a single high-error route cannot be hidden by low-error routes.

### Proof of Concept
```bash
# Terminal 1: simulate or observe 5xx errors on a history endpoint
# (e.g., caused by DB corruption / tampering)
watch -n1 'curl -s -o /dev/null -w "%{http_code}" http://mirror-node/api/v1/transactions/0.0.1-1234567890-000000000'

# Terminal 2: flood a cheap always-200 endpoint to inflate allSuccessCounter
wrk -t4 -c100 -d300s http://mirror-node/api/v1/accounts

# Terminal 3: observe that aggregate error rate drops to near zero
# despite ongoing 5xx on /api/v1/transactions
curl -u admin:secret http://mirror-node/swagger/metrics/ | grep api_all_server_error
# Expected: api_all_server_error_total is non-zero but
# api_all_server_error_total / api_all_request_total ≈ 0.001
# — below any reasonable alert threshold
```
