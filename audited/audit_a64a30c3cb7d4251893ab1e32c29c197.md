### Title
Unvalidated `ip_address_v4` Field in `fetchMirrorNodeNetwork()` Allows Arbitrary Host Injection via Mirror REST Response

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` accepts the raw string value of `ip_address_v4` from the mirror REST JSON response and passes it directly to `net.JoinHostPort` without any format or content validation. An attacker who can influence the mirror REST response (e.g., via HTTP MITM or a compromised/rogue mirror endpoint) can supply IPv4-mapped IPv6 strings such as `::ffff:127.0.0.1` or arbitrary hostnames, causing the pinger's SDK client to be initialized with a fully attacker-controlled network map, redirecting all gRPC node connections and preventing any transaction submission.

### Finding Description
In `pinger/mirror_node_client.go` lines 113–124:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)   // ← raw, unvalidated string
    }
    if host == "" || ep.Port == 0 {
        continue
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
```

The `serviceEndpoint` struct declares `IPAddressV4 string` with no semantic enforcement:

```go
type serviceEndpoint struct {
    DomainName  string `json:"domain_name"`
    IPAddressV4 string `json:"ip_address_v4"`
    Port        int    `json:"port"`
}
```

There is **no** call to `net.ParseIP`, no check that the parsed IP is actually IPv4, no rejection of loopback/private/link-local ranges, and no rejection of IPv4-mapped IPv6 notation. `net.JoinHostPort("::ffff:127.0.0.1", "50211")` produces `[::ffff:127.0.0.1]:50211`, which Go's networking stack resolves as IPv4 loopback — indistinguishable from `127.0.0.1:50211` at the socket level. The resulting poisoned `network` map is returned directly to `newClient()` and passed to `hiero.ClientForNetwork(netmap)`, so every subsequent gRPC call uses the attacker-supplied addresses.

The mirror REST URL is operator-configured via `HIERO_MIRROR_PINGER_REST` and may use plain `http://`. The `http.Client` constructed at line 46 has no TLS certificate pinning and no response-body integrity check beyond JSON decoding.

### Impact Explanation
If all entries in the returned node list carry attacker-controlled addresses, `len(network) == 0` is never triggered (the check at line 127 only fires when the list is empty), so no error is returned and the poisoned map is used silently. Every `hiero.TransferTransaction` submitted by the pinger will fail to reach any legitimate consensus node. The pinger's sole purpose is to confirm live transaction processing; a fully poisoned network map causes a **total loss of transaction-submission capability** for the pinger, which is the "network not being able to confirm new transactions" impact described in scope.

### Likelihood Explanation
- If `HIERO_MIRROR_PINGER_REST` is an `http://` URL (no TLS), any on-path network attacker (same subnet, rogue router, DNS poisoning) can MITM the single unauthenticated GET request and inject the payload — no credentials required.
- If the mirror REST server itself is compromised or is a rogue deployment, the same payload is trivially served.
- The attack is repeatable: `buildNetworkFromMirrorNodes` is called once at startup; a single poisoned response is sufficient to permanently misconfigure the client for the lifetime of the process.
- No authentication, signing, or out-of-band verification of the mirror response exists anywhere in the call chain.

### Recommendation
1. **Validate `ip_address_v4` is a well-formed IPv4 address** before use:
   ```go
   ip := net.ParseIP(strings.TrimSpace(ep.IPAddressV4))
   if ip == nil || ip.To4() == nil {
       continue // reject IPv6, IPv4-mapped IPv6, and garbage
   }
   host = ip.String()
   ```
2. **Reject reserved/loopback/private ranges** (loopback `127.0.0.0/8`, link-local `169.254.0.0/16`, RFC-1918 ranges) unless the deployment explicitly requires them.
3. **Enforce HTTPS** for `cfg.mirrorRest` and validate the TLS certificate against a pinned CA or a configured trust anchor, so MITM injection is not possible even on hostile networks.
4. **Cross-validate** the returned node addresses against a locally configured allowlist of known consensus node IPs/account IDs when operating in `network=other` mode.

### Proof of Concept
**Preconditions:** `HIERO_MIRROR_PINGER_NETWORK=other`, `HIERO_MIRROR_PINGER_REST=http://attacker-controlled-host/`.

**Trigger:** The attacker's HTTP server responds to `GET /api/v1/network/nodes` with:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "::ffff:127.0.0.1", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```

**Result:**
- `net.JoinHostPort("::ffff:127.0.0.1", "50211")` → `[::ffff:127.0.0.1]:50211`
- `hiero.ClientForNetwork` is initialized with this single entry.
- All gRPC dials target loopback (or any other attacker-chosen address); no legitimate consensus node is ever reached.
- Every `submitWithRetry` call fails; the pinger logs continuous transfer failures and the `/tmp/ready` probe may eventually be considered stale, causing the pod to be restarted — and the cycle repeats on each restart since the network map is rebuilt from the same poisoned endpoint.
