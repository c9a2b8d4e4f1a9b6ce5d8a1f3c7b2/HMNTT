### Title
Unbounded Alias Length + No REST-Layer Rate Limiting Enables DB-Query Flooding via `getAccountFromAlias`

### Summary
The `accountAliasRegex` in `rest/accountAlias.js` imposes no upper bound on the base32 alias portion, allowing arbitrarily long strings that pass validation and are decoded to arbitrarily large `bytea` values. Each such string is forwarded directly to a parameterized PostgreSQL equality query against the `entity.alias` column with no server-side rate limiting in the REST API middleware stack. An unauthenticated attacker can flood the endpoint with many unique, long, syntactically valid aliases, saturating the database connection pool and CPU without any throttle gate.

### Finding Description

**Regex — no length cap:**
`rest/accountAlias.js` line 10 defines:
```js
const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;
```
The `+` quantifier on `[A-Z2-7]` has no upper bound. A string like `AAAA…` (thousands of characters) passes `isValid()` and proceeds to `base32.decode`, producing an arbitrarily large `Uint8Array`.

**Decoded bytes passed directly to DB:**
`rest/accountAlias.js` line 24:
```js
this.alias = base32.decode(base32Alias);
```
`rest/service/entityService.js` lines 17–20 and 43:
```js
static entityFromAliasQuery = `select ${Entity.ID}
  from ${Entity.tableName}
  where coalesce(${Entity.DELETED}, false) <> true
    and ${Entity.ALIAS} = $1`;
...
const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
```
The decoded `Uint8Array` is bound as `$1` — a `bytea` equality comparison. PostgreSQL must compare the full byte sequence; with a large value and no index hit (non-existent alias), this degrades to a sequential scan or at minimum a full index-key comparison per request.

**No rate limiting in the REST middleware stack:**
`rest/middleware/index.js` exports only: `authHandler`, `handleError`, `openApiValidator`, `serveSwaggerDocs`, `requestLogger`, `requestQueryParser`, cache handlers, and `responseHandler`. There is no rate-limiting or throttling middleware. The `ThrottleConfiguration` found in `web3/src/main/java/…/ThrottleConfiguration.java` applies only to the web3 module, not the REST API. A grep across `rest/**` for `rate.?limit|throttle|rateLimit` returns only `openapi.yml` documentation and test utilities — no enforcement code.

**No caching for misses:**
Non-existent aliases return `null` from `getAccountFromAlias` (line 45–46). Cache miss responses are not stored, so every unique alias string causes a fresh DB round-trip.

### Impact Explanation
An attacker sending a high volume of `GET /api/v1/accounts/<long-unique-alias>` requests forces one DB query per request with no server-side gate. The database connection pool becomes saturated, legitimate queries queue or time out, and the mirror node becomes unavailable to all users. Because the alias portion is unbounded, each request can also carry a large payload that increases per-query processing cost. This constitutes a non-network DoS affecting availability of the mirror node REST API.

### Likelihood Explanation
No authentication or API key is required. The attack requires only an HTTP client and knowledge of the public API (documented in `openapi.yml`). Generating unique valid base32 strings is trivial. The attack is repeatable, automatable, and requires no special privileges or insider knowledge. Any public deployment of the mirror node REST API is exposed.

### Recommendation
1. **Enforce a maximum alias length** in `accountAliasRegex` — replace `[A-Z2-7]+` with a bounded quantifier matching the maximum valid alias size (e.g., `[A-Z2-7]{8,100}` aligned with actual Hedera public key encodings).
2. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit`) applied globally or specifically to alias-lookup paths, independent of the web3 throttle.
3. **Cache negative (not-found) results** for alias lookups with a short TTL to prevent repeated DB hits for the same non-existent alias.
4. **Ensure a database index** exists on `entity.alias` so equality lookups do not degrade to sequential scans.

### Proof of Concept
```bash
# Generate and flood with unique long valid aliases (no auth required)
for i in $(seq 1 10000); do
  # Each alias: 200 uppercase base32 chars — passes accountAliasRegex, decodes to 125 bytes
  ALIAS=$(python3 -c "import random,string; print(''.join(random.choices('ABCDEFGHIJKLMNOPQRSTUVWXYZ234567',k=200)))")
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}" &
done
wait
```
Each request passes `isValid()`, constructs an `AccountAlias` with a 125-byte decoded alias, and issues `SELECT id FROM entity WHERE … AND alias = $1` with no throttle. Under sustained load, the DB connection pool exhausts and the service becomes unresponsive to legitimate traffic.
