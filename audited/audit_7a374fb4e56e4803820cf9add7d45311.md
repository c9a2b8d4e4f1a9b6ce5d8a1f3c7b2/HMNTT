### Title
Thundering Herd Retry Storm via Externally-Triggered gRPC UNAVAILABLE: Insufficient Backoff Cap with Unbounded Retry Attempts

### Summary
An unprivileged external attacker who causes a transient network partition between the monitor and the mirror node's gRPC endpoint will force all configured subscribers (up to 1024) into simultaneous infinite retry loops. The `shouldRetry()` filter passes `UNAVAILABLE` through, `maxAttempts` is set to `Long.MAX_VALUE`, and the `maxBackoff` cap of 8 seconds is too narrow to spread 1024 concurrent reconnection storms, amplifying load on the gRPC server when the partition resolves.

### Finding Description

**Code path:**

`GrpcSubscriberProperties` constructor hardcodes infinite retries and an 8-second backoff ceiling: [1](#0-0) 

`AbstractSubscriberProperties` permits up to 1024 subscriber instances per scenario: [2](#0-1) 

`SubscribeProperties` permits up to 1024 SDK clients: [3](#0-2) 

`GrpcSubscriber.shouldRetry()` only blocks `INVALID_ARGUMENT`; every other status code — including `UNAVAILABLE`, `DEADLINE_EXCEEDED`, `INTERNAL`, `RESOURCE_EXHAUSTED` — passes through and triggers a retry: [4](#0-3) 

`clientSubscribe()` wires the retry with `maxAttempts = Long.MAX_VALUE` and `maxBackoff = 8s`: [5](#0-4) 

`GrpcClientSDK` explicitly disables the SDK-level retry (`setMaxAttempts(0)`), delegating all retry responsibility to the Reactor layer above with no circuit breaker: [6](#0-5) 

**Root cause and failed assumption:**

The design assumes that Reactor's `Retry.backoff` full-jitter is sufficient to spread reconnection load. This assumption fails at scale: when all 1024 subscribers fail at the same instant (simultaneous network partition), they all begin their retry timers from the same point in time. Reactor's jitter randomizes the delay within `[0, maxBackoff]`, so with `maxBackoff = 8s`, all 1024 reconnection attempts are compressed into an 8-second window — approximately **128 connection attempts per second** — against the same gRPC endpoint. With `maxAttempts = Long.MAX_VALUE`, this repeats indefinitely for the duration of the partition.

### Impact Explanation

When the partition resolves, the mirror node's gRPC server receives a burst of up to 1024 simultaneous TLS handshakes and `subscribeTopic` stream initiations within an 8-second window. This can:
- Exhaust gRPC server thread pools or connection accept queues, causing the server to return `UNAVAILABLE` again — creating a self-reinforcing feedback loop.
- Amplify a short-lived external disruption into a prolonged service degradation of the mirror node's gRPC interface for all legitimate clients.
- Cause cascading failures if the gRPC server is shared with production traffic.

Severity: **High** — the monitor's retry storm can DoS the very service it is monitoring, turning a transient external event into a sustained internal one.

### Likelihood Explanation

Causing a network partition between the monitor and the mirror node requires no credentials or privileged access. Practical methods include:
- Volumetric DDoS against the mirror node's gRPC port (standard UDP/TCP flood).
- BGP route withdrawal or prefix hijacking affecting the path between monitor and mirror node.
- Exploiting any upstream network device on the path.

The attacker does not need to interact with the monitor application at all. The precondition (operator has configured multiple subscribers, which is the intended use case) is met in any non-trivial deployment. The attack is repeatable: each time the attacker drops and restores connectivity, a new retry storm is triggered.

### Recommendation

1. **Add a circuit breaker**: Wrap `clientSubscribe()` with a Resilience4j or Reactor circuit breaker that opens after N consecutive `UNAVAILABLE` errors, preventing further retries until a probe succeeds.
2. **Widen the backoff window**: Increase `maxBackoff` to at least 60–120 seconds so that jitter spreads 1024 subscribers across a much larger window (reducing burst rate from ~128/s to ~8–17/s).
3. **Cap `maxAttempts`**: Replace `Long.MAX_VALUE` with a finite limit (e.g., 32–64) and emit a terminal error that triggers an alert, rather than retrying silently forever.
4. **Extend `shouldRetry()` filtering**: Distinguish transient errors (`UNAVAILABLE`) from persistent ones (`INTERNAL`, `RESOURCE_EXHAUSTED`) and apply different backoff strategies or caps per status code.
5. **Add per-subscriber jitter seeding**: Seed each subscriber's retry timer with a unique offset proportional to its index so that even a simultaneous failure does not produce a synchronized retry wave.

### Proof of Concept

**Preconditions:**
- Monitor deployed with `hiero.mirror.monitor.subscribe.grpc.<name>.subscribers=1024` (within the allowed `@Max(1024)` bound).
- Monitor is actively subscribing to a topic on the mirror node's gRPC endpoint.

**Steps:**
1. Identify the network path between the monitor host and the mirror node's gRPC endpoint (default port 443 or 5600).
2. Launch a volumetric TCP SYN flood or firewall rule drop against that port for 30–60 seconds. This causes all 1024 active `subscribeTopic` streams to receive `StatusRuntimeException` with `Status.UNAVAILABLE`.
3. `shouldRetry()` returns `true` for all 1024 failures (only `INVALID_ARGUMENT` is excluded).
4. All 1024 subscribers enter `Retry.backoff(Long.MAX_VALUE, 500ms).maxBackoff(8s)` simultaneously.
5. Restore connectivity. Within the next 8-second window, all 1024 subscribers attempt simultaneous TLS reconnection and `subscribeTopic` RPC initiation.
6. Observe the mirror node's gRPC server metrics: connection accept rate spikes to ~128/s; if the server cannot handle this, it returns `UNAVAILABLE` again, restarting the cycle.
7. Repeat step 2 to re-trigger the storm on demand.
