### Title
Missing Entity Existence Check for Numeric Account IDs in Token Allowance Lookup

### Summary
In `getAccountTokenAllowances()`, when a caller supplies a numeric-only account ID (e.g., `12345` or `0.0.12345`), `EntityService.getEncodedId()` returns the encoded ID directly without any database existence check. This causes the API to respond with HTTP 200 and an empty allowances list for accounts that do not exist, instead of the specified HTTP 404. The OpenAPI spec explicitly documents a 404 response for this endpoint, making this a contract violation with concrete behavioral consequences.

### Finding Description

**Exact code path:**

In `rest/controllers/tokenAllowanceController.js`, `getAccountTokenAllowances()` at line 69: [1](#0-0) 

It calls `EntityService.getEncodedId()` with the raw path parameter. Inside `rest/service/entityService.js`, lines 120–124: [2](#0-1) 

The branching logic is:
- If `EntityId.isValidEntityId(entityIdString)` is true **and** `entityId.evmAddress === null` (i.e., it is a plain numeric/shard.realm.num ID) → returns `entityId.getEncodedId()` **immediately, with no DB lookup**.
- If it is an EVM address → calls `getEntityIdFromEvmAddress()` which **does** query the DB and throws `NotFoundError` on miss.
- If it is an alias → calls `getAccountIdFromAlias()` which **does** query the DB and throws `NotFoundError` on miss.

**Root cause:** The numeric-ID branch skips the existence check entirely. The `isValidAccount()` method exists on the same service (lines 60–63) and performs exactly the needed check: [3](#0-2) 

but it is never called in the numeric-ID path.

**Confirmed by test:** The test named `"EntityService.getEncodedId - No match"` (line 161–163) explicitly shows that `getEncodedId` resolves successfully even when the entity is **not** loaded into the database: [4](#0-3) 

**Downstream effect:** `TokenAllowanceService.getAccountTokenAllowances()` then executes a `SELECT * FROM token_allowance WHERE owner = $1` query against the non-existent account ID, returns zero rows, and the controller writes `{ allowances: [], links: { next: null } }` with HTTP 200. [5](#0-4) 

The OpenAPI spec at `rest/api/v1/openapi.yml` lines 327–337 explicitly declares a `404` response for this endpoint: [6](#0-5) 

### Impact Explanation
Any off-chain system (indexer, bridge relay, DeFi integration, or smart-contract orchestration layer) that calls `GET /api/v1/accounts/{id}/allowances/tokens` and interprets a 200+empty-list as "account exists but has no allowances" will be silently misled when the account does not exist. This is distinct from a "no allowances" state and can cause:
- Incorrect authorization decisions (e.g., a relay concluding a spender has no allowance and skipping a required approval step).
- Phantom account enumeration: an attacker can probe arbitrary numeric IDs and always receive 200, making it impossible for callers to distinguish "account exists, zero allowances" from "account does not exist."
- Inconsistency with the EVM-address and alias paths for the same endpoint, which correctly return 404, breaking any client that assumes uniform behavior across ID formats.

Severity is **Medium**: no funds are directly drained by this API call alone, but the false signal it produces can feed incorrect state into systems that do control funds.

### Likelihood Explanation
- **No authentication required.** Any unauthenticated HTTP client can trigger this.
- **Trivially repeatable.** A single GET request with any unused numeric ID (e.g., `GET /api/v1/accounts/999999999/allowances/tokens`) reproduces the issue every time.
- **No rate-limit or CAPTCHA barrier** is present on mirror-node REST endpoints by default.
- An attacker does not need any prior knowledge of the system beyond the public API spec.

### Recommendation
In `EntityService.getEncodedId()` (`rest/service/entityService.js`), after computing the encoded ID for the numeric path, add an existence check before returning:

```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  if (entityId.evmAddress === null) {
    const encodedId = entityId.getEncodedId();
    if (requireResult && !(await this.isValidAccount(encodedId))) {
      throw new NotFoundError();
    }
    return encodedId;
  }
  return await this.getEntityIdFromEvmAddress(entityId, requireResult);
}
```

This mirrors the existing behavior of the EVM-address and alias paths and makes all three input formats consistent. The `requireResult` flag is already threaded through the signature, so callers that pass `requireResult = false` retain the current no-check behavior where needed.

### Proof of Concept

**Precondition:** Account `0.0.999999999` does not exist on the network/in the mirror-node DB.

**Step 1 – Numeric ID (vulnerable path):**
```
GET /api/v1/accounts/999999999/allowances/tokens
```
**Expected (per spec):** HTTP 404  
**Actual:** HTTP 200 `{"allowances":[],"links":{"next":null}}`

**Step 2 – EVM address (correct path, for contrast):**
```
GET /api/v1/accounts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/allowances/tokens
```
**Actual:** HTTP 404 — confirms the DB check works for EVM addresses.

**Step 3 – Alias (correct path, for contrast):**
```
GET /api/v1/accounts/AAAQEAYEAUDAOCAJBAEQGK3TMNZGK4Q=/allowances/tokens
```
**Actual:** HTTP 404 — confirms the DB check works for aliases.

The discrepancy between Step 1 and Steps 2–3 is the vulnerability. The root cause is the missing `isValidAccount` call in the `entityId.evmAddress === null` branch of `getEncodedId` at `rest/service/entityService.js` lines 122–123. [7](#0-6)

  async getEncodedId(entityIdString, requireResult = true, paramName = filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS) {
    try {
      if (EntityId.isValidEntityId(entityIdString)) {
        const entityId = EntityId.parseString(entityIdString, {paramName});
        return entityId.evmAddress === null
          ? entityId.getEncodedId()
          : await this.getEntityIdFromEvmAddress(entityId, requireResult);
      } else if (AccountAlias.isValid(entityIdString)) {
        return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
      }
    } catch (ex) {
      if (ex instanceof InvalidArgumentError) {
        throw InvalidArgumentError.forParams(paramName);
      }
      // rethrow
      throw ex;
    }

    throw InvalidArgumentError.forParams(paramName);
  }
