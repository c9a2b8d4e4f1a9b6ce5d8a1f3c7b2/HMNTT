### Title
Timing Side-Channel in `getTopic()` Enables Unauthenticated Topic ID Enumeration

### Summary
The `getTopic()` method in `TopicController.java` executes three sequential, independent database queries. When a topic ID does not exist, the first query throws an exception and the remaining two queries are never executed, producing a measurably shorter response time. An unauthenticated attacker can exploit this latency difference to enumerate which topic IDs exist in the system, even if all error responses are normalized to the same HTTP status and message.

### Finding Description
**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, `getTopic()`, lines 32–37. [1](#0-0) 

The three calls are strictly sequential:
1. `topicService.findById(id.id())` — line 33
2. `entityService.findById(id.id())` — line 34
3. `customFeeService.findById(id.id())` — line 35

Each service implementation calls `orElseThrow(() -> new EntityNotFoundException(...))`: [2](#0-1) [3](#0-2) [4](#0-3) 

**Root cause / failed assumption:** The code assumes that because all three paths return the same HTTP error response, the behavior is indistinguishable to an external caller. This is false. Java exception propagation short-circuits execution: if `topicService.findById()` throws on line 33, lines 34 and 35 are never reached. The result is:

| Scenario | DB queries executed | Relative latency |
|---|---|---|
| Topic ID does **not** exist | 1 | ~1× |
| Topic ID exists (entity/fee may or may not) | 2–3 | ~2–3× |

The timing difference between one round-trip to the database and three is typically 10–100 ms — well above the noise floor for a statistical attack with repeated samples.

### Impact Explanation
An unauthenticated attacker can enumerate which topic IDs are present in the mirror node database. This constitutes unauthorized information disclosure about the state of the Hedera network as seen by this node. Enumerated topic IDs can be used for targeted follow-up queries (e.g., message harvesting, subscriber profiling) and for mapping the operational topology of the network.

### Likelihood Explanation
The endpoint is public and requires no credentials. The attacker needs only an HTTP client and the ability to measure wall-clock response times. The signal (1 query vs. 3 queries) is large enough to be detected with a modest number of repeated requests (tens of samples per ID) to average out network jitter. This is a well-understood attack class with freely available tooling. No special knowledge of the system internals is required.

### Recommendation
Normalize execution time regardless of whether the topic exists. Concrete options:

1. **Eager parallel execution:** Issue all three queries concurrently (e.g., via `CompletableFuture`) and collect results before checking for absence, so the total wall-clock time is always dominated by the slowest query regardless of which IDs exist.
2. **Existence gate with constant-time padding:** Perform a single existence check first, then add a fixed artificial delay before returning any response, so both the "found" and "not found" paths take the same time. (Less preferred — fragile under load.)
3. **Restructure to a single JOIN query:** Replace the three independent lookups with one SQL query joining `topic`, `entity`, and `custom_fee` tables, eliminating the sequential short-circuit entirely.

Option 1 or 3 is strongly preferred because they remove the structural cause rather than masking it.

### Proof of Concept
```
# Step 1 – Baseline: measure a known non-existing topic ID (e.g., 0.0.999999999)
for i in $(seq 1 50); do
  curl -o /dev/null -s -w "%{time_total}\n" \
    "https://<mirror-node>/api/v1/topics/999999999"
done | awk '{sum+=$1; n++} END {print "avg:", sum/n}'

# Step 2 – Probe: measure a candidate topic ID
for i in $(seq 1 50); do
  curl -o /dev/null -s -w "%{time_total}\n" \
    "https://<mirror-node>/api/v1/topics/<candidate_id>"
done | awk '{sum+=$1; n++} END {print "avg:", sum/n}'

# Step 3 – Compare averages.
# A statistically significant higher average for the candidate ID
# (typically >1.5× the baseline) indicates the topic exists,
# even though both return the same HTTP 404 body.
```
