### Title
Unbounded Concurrent Subscriptions with No Global Limit Enable Resource Exhaustion DoS via Hung Streams

### Summary
`TopicMessageServiceImpl.subscribeTopic()` increments `subscriberCount` on subscribe and decrements it only in `doFinally`, which is never triggered on streams that hang during a network partition (no TCP RST/FIN, no keepalive). The only server-side concurrency control is `maxConcurrentCallsPerConnection = 5`, which limits calls per HTTP/2 connection but imposes no cap on total connections or total subscriptions. An unprivileged attacker opening many connections, each with 5 indefinite (no `endTime`) subscriptions, can exhaust server-side resources and deny service to legitimate subscribers.

### Finding Description
**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()` (lines 87–91):

```java
return topicExists(filter)
        .thenMany(flux.doOnNext(topicContext::onNext)
                .doOnSubscribe(s -> subscriberCount.incrementAndGet())   // line 89
                .doFinally(s -> subscriberCount.decrementAndGet())       // line 90
                .doFinally(topicContext::finished));                     // line 91
```

`subscriberCount` is registered solely as a Micrometer gauge (lines 52–55) — it is never read back to gate or reject new subscriptions. There is no check of the form `if (subscriberCount.get() > MAX) reject`.

**Root cause — two compounding gaps:**

1. **`doFinally` is not triggered on hung streams.** In Reactor, `doFinally` fires on `onComplete`, `onError`, or `cancel`. A cancel arrives when the downstream gRPC transport cancels the stream. During a network partition where the TCP connection silently hangs (no RST, no FIN, no TCP/gRPC keepalive probe succeeds), the Netty transport never detects the dead peer, never cancels the stream, and `doFinally` is never invoked. The subscription — and all resources it holds — remains live indefinitely.

2. **No global or per-IP subscription limit.** The only concurrency control is `maxConcurrentCallsPerConnection = 5` (`NettyProperties` line 14, applied in `GrpcConfiguration` line 33). This limits calls per HTTP/2 connection but places no bound on the number of connections an attacker may open. There is no per-IP connection limit, no global subscription cap, and no idle-stream timeout anywhere in the gRPC server code.

**Why `endTime = null` makes this permanent:** When no `endTime` is set, `pastEndTime()` returns `Flux.never()` (line 125) and `isComplete()` always returns `false` (line 204), so the stream never self-terminates.

**Exploit flow:**
1. Attacker opens *N* TCP connections to the gRPC port (no connection-rate or connection-count limit enforced).
2. On each connection, attacker opens 5 `subscribeTopic` calls with a valid `topicId` and no `endTime` (5 × N total subscriptions).
3. Attacker induces a network partition (e.g., drops packets at the firewall/NIC level, or simply uses a NAT that silently drops idle flows) so the server's TCP stack never receives RST/FIN.
4. Because no keepalive probes are configured on the Netty server (no `keepAliveTime` set in `GrpcConfiguration`), the server never detects the dead connections.
5. `doFinally` is never triggered; `subscriberCount` is never decremented; each `TopicContext`, listener registration, `boundedElastic` scheduler slot, and any in-flight DB connection remains allocated.
6. With sufficient *N*, the JVM heap, thread pool, and/or DB connection pool are exhausted, causing OOM errors or connection-pool starvation for all other subscribers.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection = 5` — mitigates per-connection abuse but is trivially bypassed by opening more connections.
- `checkTopicExists` — validates the topic exists but does not limit subscription count.
- `filter.hasLimit()` / `filter.getEndTime()` — only terminate the stream if the *client* supplies these; an attacker omits them.
- Micrometer alert `GrpcNoSubscribers` — fires when count ≤ 0, not when count is too high; no upper-bound alert exists.

### Impact Explanation
Each hung subscription retains a `TopicContext` object, a registration in the `TopicListener` (Redis/poll), a slot in `Schedulers.boundedElastic()` (default 10× CPU threads), and potentially an open R2DBC/JDBC connection. Exhausting the bounded-elastic thread pool causes all reactive pipelines on the server to stall. Exhausting the HikariCP connection pool (default max visible in configuration) causes all DB-backed operations — including legitimate subscriptions and historical retrieval — to time out. The result is a complete denial of service to all other subscribers and REST API consumers sharing the same DB pool.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond the public gRPC proto, and no zero-day. Any client that can reach port 5600 can execute it. The `maxConcurrentCallsPerConnection = 5` constraint means the attacker needs only ⌈target_resource_limit / 5⌉ connections, which is trivially achievable from a single host or a small botnet. The network-partition condition (silent TCP hang) is reproducible with a simple `iptables -A OUTPUT -p tcp --dport 5600 -j DROP` rule on the attacker's side after the connections are established, or by exploiting NAT timeout behavior on cloud environments (common in AWS/GCP where idle TCP flows are silently dropped after ~350 s with no keepalive).

### Recommendation
1. **Add a global (or per-IP) subscription cap** enforced before `subscriberCount.incrementAndGet()`:
   ```java
   if (subscriberCount.get() >= grpcProperties.getMaxSubscribers()) {
       return Mono.error(new StatusRuntimeException(Status.RESOURCE_EXHAUSTED));
   }
   ```
2. **Configure gRPC/Netty keepalive** in `GrpcConfiguration` so dead connections are detected and cancelled:
   ```java
   serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
   serverBuilder.permitKeepAliveWithoutCalls(false);
   ```
3. **Add a maximum connection limit** via `serverBuilder.maxConnectionIdle(...)` and `serverBuilder.maxConnectionAge(...)` to bound the lifetime of any single connection.
4. **Add an upper-bound Prometheus alert** on `hiero_mirror_grpc_subscribers` to detect abnormal growth before resources are exhausted.

### Proof of Concept
```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "grpc-host:5600"
TOPIC_ID = "0.0.12345"   # any valid topic
CONNECTIONS = 200         # 200 connections × 5 calls = 1000 subscriptions

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,   # valid topic, no end_time, no limit
    )
    streams = [stub.subscribeTopic(req) for _ in range(5)]
    # After streams are established, drop outbound packets:
    # iptables -A OUTPUT -d <TARGET_IP> -p tcp --dport 5600 -j DROP
    # Server now has 5 hung streams per connection, doFinally never fires.
    for s in streams:
        try:
            for _ in s: pass   # consume in background; attacker drops packets
        except: pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: 1000 subscriptions permanently allocated on server,
# TopicContext objects, listener slots, and DB connections never freed.
