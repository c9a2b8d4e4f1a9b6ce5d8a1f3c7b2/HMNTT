### Title
Unbounded DB Result Set in `findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc` Enables Per-Request Resource Amplification

### Summary
The Spring Data repository method `findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc()` returns a `List<Transaction>` with no row limit, while the sole caller immediately discards all but the first element via `getFirst()`. Because the `(payer_account_id, valid_start_ns)` pair is explicitly non-unique in the database schema (child/scheduled/duplicate transactions share the same pair), an unprivileged attacker who identifies a transaction ID with many child rows can force the database to materialize and transfer the full result set on every request, amplifying DB I/O proportionally to the number of matching rows.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/TransactionRepository.java`, lines 14–15:
```java
List<Transaction> findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
        EntityId payerAccountId, long validStartNs);
```
No `@Limit`, `Top1`, or `LIMIT 1` constraint is present. Spring Data generates `SELECT … WHERE payer_account_id = ? AND valid_start_ns = ? ORDER BY consensus_timestamp ASC` with no row cap.

**Sole caller** — `web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`, lines 100–109:
```java
final var transactionList =
    transactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
            payerAccountId, validStartNs);
if (transactionList.isEmpty()) {
    throw new EntityNotFoundException("Transaction not found: " + transactionId);
}
final var parentTransaction = transactionList.getFirst();   // only first is ever used
```
All rows beyond the first are fetched, deserialized into `Transaction` objects, and immediately discarded.

**Root cause — non-unique key confirmed by schema:**

Migration `V1.9.1__transactions_indexes.sql` line 1 explicitly states:
> "transaction_ids (valid start timestamp + payer account id) are not unique; remove unique constraint."

The current V2 schema (`V2.0.3__index_init.sql`, line 262–263) creates only a non-unique index:
```sql
create index if not exists transaction__transaction_id
    on transaction (payer_account_id, valid_start_ns);
```

The repository integration test (`TransactionRepositoryTest.java`, lines 31–41) explicitly persists **10 rows** sharing the same `(payerAccountId, validStartNs)` pair and asserts all 10 are returned — confirming the unbounded behavior is real and tested.

**Exploit flow:**

1. Attacker identifies a transaction ID whose payer account has many child/scheduled/duplicate transactions sharing the same `valid_start_ns` (e.g., a complex smart-contract call that spawned N child transactions). This information is publicly visible on-chain.
2. Attacker sends `GET /api/v1/contracts/results/{payerAccountId}-{validStart}/opcodes` with `Accept-Encoding: gzip`.
3. The global opcode rate-limit bucket (`opcodeRateLimitBucket`) is consumed by one token.
4. The DB executes `SELECT … WHERE payer_account_id = ? AND valid_start_ns = ?` and returns all N rows.
5. The service calls `getFirst()` and discards the remaining N−1 rows.
6. DB CPU, I/O, network bandwidth, and JVM heap are consumed proportionally to N per request.

### Impact Explanation

Each request through the throttle causes the database to read, serialize, and transmit N full `Transaction` rows when only 1 is needed. For a transaction with a large number of child transactions (common in EVM contract calls), N can be in the hundreds. The global rate-limit bucket (`opcodeRateLimitBucket`) is shared across all callers; an attacker who saturates it with high-amplification requests both degrades DB performance and starves legitimate users of their rate-limit quota. This constitutes a griefing/resource-exhaustion attack with no economic cost to the attacker beyond network requests.

### Likelihood Explanation

- The opcodes feature is **disabled by default** (`OpcodesProperties.enabled = false`), which substantially reduces exposure in default deployments.
- When enabled, the endpoint is unauthenticated and publicly reachable; no credentials or privileged access are required.
- The attacker needs only a publicly observable transaction ID with many child rows — trivially discoverable via the mirror node's own REST API or a block explorer.
- The attack is repeatable at the rate permitted by the global throttle; the throttle does not prevent per-request amplification.
- Severity is **medium** (griefing / resource amplification; no direct economic damage to network participants).

### Recommendation

Add a `LIMIT 1` constraint directly on the repository method. The simplest fix is to rename the method to use Spring Data's `findFirst…` convention, or add Spring Data's `@Limit(1)` annotation (Spring Data 3.2+):

```java
// Option A – derived query name (no annotation needed)
Optional<Transaction> findFirstByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
        EntityId payerAccountId, long validStartNs);

// Option B – explicit annotation
@Limit(1)
List<Transaction> findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc(
        EntityId payerAccountId, long validStartNs);
```

Update the caller in `OpcodeServiceImpl` to use `Optional` and `orElseThrow` accordingly, eliminating the `getFirst()` call and the unbounded fetch entirely.

### Proof of Concept

**Precondition:** A Hedera account (e.g., `0.0.12345`) has submitted a transaction at `valid_start_ns = T` that spawned N child transactions (all stored with the same `payer_account_id` and `valid_start_ns`). The opcodes feature is enabled.

**Steps:**
```
# 1. Identify a high-child-count transaction via the public mirror REST API
GET /api/v1/transactions/0.0.12345-<seconds>-<nanos>

# 2. Repeatedly call the opcodes endpoint with that transaction ID
for i in $(seq 1 <rate_limit>); do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/0.0.12345-<seconds>-<nanos>/opcodes" &
done
```

**Result:** Each request causes the DB to return all N child-transaction rows. DB query time and I/O scale with N. The global opcode rate-limit bucket is exhausted, blocking legitimate users.
