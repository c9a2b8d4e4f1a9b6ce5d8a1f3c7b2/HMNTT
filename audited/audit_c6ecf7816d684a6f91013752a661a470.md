### Title
Unbounded Database Query Amplification via Alias Lookup with `requireResult=false` Bypasses Circuit Breaker

### Summary
`getAccountIdFromAlias()` with `requireResult=false` executes a full database query on every call via `getAccountFromAlias()` and returns `null` (HTTP 200 OK) on a miss, never generating a 5xx response. The REST API's Traefik circuit breaker is keyed exclusively on 5xx/network error ratios, so sustained alias-miss flooding produces only 200 OK responses, leaving the circuit breaker permanently closed. No per-IP rate limit or in-flight request cap is configured for the REST API, allowing an unprivileged attacker to exhaust the database connection pool.

### Finding Description

**Code path:**

`rest/service/entityService.js`, lines 71–81 (`getAccountIdFromAlias`) and lines 42–53 (`getAccountFromAlias`):

```js
async getAccountIdFromAlias(accountAlias, requireResult = true) {
  const entity = await this.getAccountFromAlias(accountAlias);   // always hits DB
  if (isNil(entity)) {
    if (requireResult) {
      throw new NotFoundError(EntityService.missingAccountAlias);
    }
    return null;   // 200 OK, no error
  }
  return entity.id;
}
```

`getAccountFromAlias` always executes:
```sql
SELECT id FROM entity
WHERE coalesce(deleted, false) <> true AND alias = $1
``` [1](#0-0) [2](#0-1) [3](#0-2) 

The `requireResult=false` call site is `rest/balances.js` line 332:
```js
return EntityService.getAccountIdFromAlias(AccountAlias.fromString(value), false);
``` [4](#0-3) 

This is triggered by any request to `/api/v1/balances?account.id=<alias>` where the alias passes `AccountAlias.isValid()`.

**Why existing checks fail:**

1. **Circuit breaker** (`charts/hedera-mirror-rest/values.yaml` lines 135–136):
   ```yaml
   circuitBreaker:
     expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
   ```
   Triggers only on 5xx or network errors. Alias misses with `requireResult=false` return HTTP 200 OK — the circuit breaker never opens. [5](#0-4) 

2. **No per-IP rate limit or in-flight cap** on the REST API. Compare with Rosetta, which has both `inFlightReq: amount: 5` and `rateLimit: average: 10`. The REST API has neither. [5](#0-4) 

3. **Redis response cache** (`responseCacheHandler.js` line 152) keys on `req.originalUrl` with a 1-second default TTL. An attacker using distinct aliases per request gets a cache miss on every request. [6](#0-5) [7](#0-6) 

4. **Retry amplification**: The REST middleware includes `retry: attempts: 10`, which can amplify load at the Traefik layer for transient failures, though this is secondary. [8](#0-7) 

### Impact Explanation
An attacker can exhaust the PostgreSQL connection pool (bounded by `db.pool.maxConnections`) by flooding `/api/v1/balances?account.id=<unique_valid_alias>` with concurrent requests. Each request consumes a DB connection for the duration of the alias query. Once the pool is exhausted, all other API endpoints that require DB access begin queuing or timing out, causing a full service-level DoS for all users. The `entity` table is the central table used by nearly every endpoint (accounts, balances, tokens, contracts), so pool exhaustion has broad blast radius.

### Likelihood Explanation
Preconditions are minimal: no authentication, no special privileges, no prior knowledge of the system. Valid alias format (base32, e.g., `AABBCC22`) is publicly documented. An attacker can generate an unbounded stream of syntactically valid but non-existent aliases programmatically. The attack is repeatable, stateless, and trivially parallelizable from a single host or botnet. The absence of per-IP rate limiting on the REST API (unlike Rosetta) makes this straightforward.

### Recommendation
1. **Add per-IP rate limiting and in-flight request cap** to the REST API Traefik middleware, mirroring the Rosetta configuration (`inFlightReq` + `rateLimit`).
2. **Cache alias-to-ID lookups** at the service layer (e.g., a short-lived in-process or Redis cache keyed on alias value), so repeated or high-volume alias misses do not each generate a DB round-trip.
3. **Adjust the circuit breaker expression** to also account for 404 response ratios or query latency, not only 5xx errors.
4. Optionally, add a **statement timeout** for alias queries at the DB level to bound per-query resource consumption.

### Proof of Concept

```bash
# Generate and flood with unique valid-format aliases (base32 strings)
# No authentication required

for i in $(seq 1 10000); do
  ALIAS=$(python3 -c "import base64, os; print(base64.b32encode(os.urandom(5)).decode().rstrip('='))")
  curl -s "https://<mirror-node>/api/v1/balances?account.id=${ALIAS}" &
done
wait
```

Each request:
1. Passes `AccountAlias.isValid()` validation (valid base32 format)
2. Reaches `getAccountIdFromAlias(..., false)` in `balances.js`
3. Executes `SELECT id FROM entity WHERE alias = $1` against the DB
4. Returns HTTP 200 OK with empty results
5. Does not trigger the circuit breaker
6. Is not served from cache (unique alias per request)

Sustained concurrent execution exhausts the DB connection pool, degrading all API endpoints.
