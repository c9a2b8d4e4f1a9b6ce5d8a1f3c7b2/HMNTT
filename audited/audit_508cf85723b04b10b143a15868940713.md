### Title
HikariCP Connection Pool Exhaustion via Unauthenticated Concurrent GraphQL `account()` Queries (DoS)

### Summary
The GraphQL service exposes `POST /graphql/alpha` with no application-level rate limiting or concurrency control. The Traefik `inFlightReq` middleware that defines a per-IP concurrent request limit is **disabled by default** (`global.middleware: false`). An unauthenticated attacker flooding the endpoint with concurrent `account(input: {entityId: ...})` queries from multiple source IPs can exhaust the HikariCP connection pool, causing all subsequent queries to queue indefinitely until the HikariCP `connectionTimeout` expires, rendering the service effectively unavailable while remaining TCP-reachable.

### Finding Description

**Code path:**

`POST /graphql/alpha` → `AccountController.account()` [1](#0-0)  → `EntityServiceImpl.getByIdAndType()` [2](#0-1)  → `entityRepository.findById()` — acquires a HikariCP connection for the duration of the DB call.

**Root cause — disabled middleware gate:**

The Traefik `inFlightReq` middleware (5 concurrent requests per source IP) is defined in `values.yaml` [3](#0-2)  but the Helm template that renders it is gated on `{{ if and .Values.global.middleware .Values.middleware }}` [4](#0-3)  and `global.middleware` defaults to `false`. [5](#0-4)  This means **no Traefik middleware is deployed in the default configuration**, leaving the endpoint completely unthrottled at the network layer.

**Root cause — no application-level rate limiting:**

The `graphql` module has no equivalent of the `web3` module's `ThrottleManagerImpl` / bucket4j rate limiter. [6](#0-5)  The `GraphQlConfiguration` only enforces query complexity (max 200) and depth (max 10) limits, [7](#0-6)  which do not bound the number of concurrent requests.

**Amplification via retry middleware:**

Even when the middleware chain is enabled, the `retry: attempts: 3` entry [8](#0-7)  causes Traefik to re-issue each failing request up to 3 times, tripling the effective load on the pool during an attack.

**Statement timeout as connection hold time:**

The documented default `statementTimeout` is 10 000 ms. [9](#0-8)  Under DB load, each connection can be held for up to 10 seconds. With Spring Boot's default HikariCP pool size of 10, only 10 simultaneous queries are needed to saturate the pool. New requests then block waiting for a free connection for up to the HikariCP `connectionTimeout` (default 30 s), producing the "reachable but hanging" symptom.

### Impact Explanation

Once the pool is exhausted, every incoming GraphQL request blocks on `HikariDataSource.getConnection()` for up to 30 seconds before throwing `SQLTransientConnectionException`. The service remains TCP-reachable and returns HTTP 200 only after the timeout, creating a network partition condition: the service is up but functionally dead. Kubernetes liveness/readiness probes check `/actuator/health/liveness` and `/actuator/health/readiness` [10](#0-9)  — these probes do not exercise the DB connection pool, so the pod is not restarted, prolonging the outage. The `GraphQLHighDBConnections` Prometheus alert fires only after 5 minutes at >75% utilization, [11](#0-10)  giving the attacker a wide window.

### Likelihood Explanation

No authentication is required. The endpoint is publicly exposed via ingress at `/graphql/alpha`. [12](#0-11)  The attack requires only valid `entityId` values (shard/realm/num are public knowledge on Hedera) and a tool capable of sending concurrent HTTP POST requests (e.g., `ab`, `wrk`, or a simple script). With the middleware disabled by default, even a single attacker from one IP with 10 concurrent connections suffices. The attack is trivially repeatable and requires no special knowledge beyond the public API schema.

### Recommendation

1. **Enable the Traefik middleware by default**: Set `global.middleware: true` in the default `values.yaml`, or move the middleware gate to not depend on `global.middleware`.
2. **Add a global `rateLimit` middleware** to the GraphQL chain (as done for Rosetta [13](#0-12) ), e.g., `rateLimit: average: 50, sourceCriterion: requestHost: true`.
3. **Add application-level concurrency control**: Implement a `Semaphore` or bucket4j-based throttle in `AccountController` or a Spring `HandlerInterceptor`, mirroring the `ThrottleManagerImpl` pattern in the `web3` module.
4. **Reduce HikariCP `connectionTimeout`** to fail fast (e.g., 5 s) so exhausted-pool errors surface quickly rather than hanging threads.
5. **Remove or scope the `retry` middleware** away from the GraphQL endpoint to prevent retry amplification.

### Proof of Concept

```bash
# Requires: curl, GNU parallel or xargs
# No authentication needed. Uses publicly known entity IDs.

ENDPOINT="https://<mirror-node-host>/graphql/alpha"
QUERY='{"query":"{ account(input: {entityId: {shard: 0, realm: 0, num: 2}}) { balance } }"}'

# Send 50 concurrent requests from a single IP (middleware disabled by default)
seq 1 50 | xargs -P 50 -I{} curl -s -X POST "$ENDPOINT" \
  -H 'Content-Type: application/json' \
  -d "$QUERY" &

# Observe: after ~10 connections are in-flight, subsequent requests hang
# for up to 30 seconds (HikariCP connectionTimeout) before returning an error.
# Legitimate users experience the same hang during the attack window.
```

Expected result: pool metrics show `hikaricp_connections_active == hikaricp_connections_max` and `hikaricp_connections_pending > 0`; all new queries stall until the attacker stops or connections time out.
