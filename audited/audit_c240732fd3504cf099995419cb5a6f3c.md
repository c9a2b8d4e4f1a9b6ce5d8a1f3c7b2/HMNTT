### Title
Unauthenticated `content-length` Header Spoofing Poisons `requestSizeHistogram` Metrics

### Summary
`metricsMiddleware()` in `rest/middleware/metricsHandler.js` unconditionally trusts the user-supplied `content-length` header to record request size into `requestSizeHistogram`. No validation is performed against the actual body size, and no upper-bound check exists. An unprivileged attacker can send a GET request to any transaction history endpoint with a fabricated `content-length: 2147483647` header and zero body, causing the histogram to record an arbitrarily large value for that route.

### Finding Description
**Exact code location:** `rest/middleware/metricsHandler.js`, `metricsMiddleware()`, line 209:

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

`req.headers['content-length']` is the raw, attacker-controlled HTTP header string. `parseInt(..., 10)` faithfully parses any integer the attacker supplies — including `2147483647` — and passes it directly to `requestSizeHistogram.record()`.

**Why upstream body-parser protection does not apply:**  
In `rest/server.js`, `express.json()` and `express.urlencoded()` are registered at lines 73–74, before the metrics middleware at line 91. However, these parsers only enforce their body-size limit (`100kb` default) when they actually attempt to read and parse a body. For **GET requests** (the standard method for transaction history endpoints like `/api/v1/transactions`), both parsers skip body processing entirely because GET requests carry no body. The spoofed `content-length` header is therefore never cross-checked against actual bytes received, and the request flows through to the metrics middleware untouched.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/transactions HTTP/1.1` with header `content-length: 2147483647` and no body.
2. `express.json()` / `express.urlencoded()` skip (GET, no body).
3. `metricsMiddleware` runs; `next()` is called; the route handler responds normally.
4. On `res.on('finish', ...)`, line 209 executes: `parseInt('2147483647', 10)` → `2147483647` is recorded into `requestSizeHistogram` with labels `{method: 'GET', path: '/api/v1/transactions', code: '200'}`.
5. Repeat at will — no authentication, no rate-limit enforced by the metrics layer itself.

### Impact Explanation
The `requestSizeHistogram` (metric name `api_request_size_bytes`) is a Prometheus histogram. Injecting values of ~2 GB per request:
- Permanently inflates the `_sum` bucket, making the mean request size appear enormous for the targeted route.
- Pushes all observations into the highest histogram bucket (`+Inf`), destroying the percentile distribution.
- Triggers false alerts on any SLO/SLA rules monitoring request payload size anomalies.
- Misleads operators into believing transaction history queries carry massive payloads, potentially causing incorrect capacity planning, misdiagnosis of performance issues, or suppression of real anomaly alerts (alert fatigue / masking).

Severity is **Medium**: no direct data exfiltration or fund movement, but persistent, unauthenticated metric poisoning with operational and observability impact.

### Likelihood Explanation
- **No authentication required** — any internet-accessible instance is vulnerable.
- **Zero body needed** — the attack is a single-header GET request; no special tooling beyond `curl`.
- **Fully repeatable** — the attacker can flood the histogram continuously.
- **No rate-limiting** in the metrics middleware itself.
- Prometheus histograms are cumulative and in-memory; poisoned `_sum` values persist until process restart.

### Recommendation
Replace the blind header read with the actual measured body size. The middleware already intercepts `res.write` / `res.end` to measure response size; apply the same pattern to the request body:

```js
// Instead of trusting the header:
// requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);

// Measure actual bytes received:
requestSizeHistogram.record(req.socket.bytesRead - requestStartBytes, labels);
// or accumulate via a 'data' listener on req before next() is called.
```

At minimum, add a sanity-cap:
```js
const MAX_REASONABLE_REQUEST_BYTES = 10 * 1024 * 1024; // 10 MB
const reportedSize = parseInt(req.headers['content-length'] ?? '0', 10) || 0;
requestSizeHistogram.record(Math.min(reportedSize, MAX_REASONABLE_REQUEST_BYTES), labels);
```

Also consider enforcing a global body-size limit that applies to all methods, not just parsed-body requests.
