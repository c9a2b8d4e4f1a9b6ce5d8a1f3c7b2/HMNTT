### Title
Rate-Limit Token Consumed Without Refund When Gas Check Fails, Enabling Unauthenticated DoS

### Summary
In `ThrottleManagerImpl.throttle()`, a rate-limit token is irrevocably consumed from `rateLimitBucket` before the gas check against `gasLimitBucket` is performed. When the gas check fails, the rate-limit token is never returned. An unprivileged attacker can exploit this to drain the entire `rateLimitBucket` using requests that are ultimately rejected, starving legitimate users of all request capacity for the remainder of each refill window.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, `throttle()`, lines 38–42:

```java
if (!rateLimitBucket.tryConsume(1)) {                                    // line 38 – token consumed
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // line 40
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);          // line 41 – token NOT refunded
}
```

`restore()` (lines 59–64) only adds tokens back to `gasLimitBucket`; there is no corresponding refund path for `rateLimitBucket`.

**Root cause:** The two-bucket check is non-atomic with no rollback. A rate-limit token is permanently consumed even when the request is ultimately rejected by the gas check.

**Clarification on synchronization:** The question's framing of a "simultaneous pass due to lack of synchronization" is not the actual mechanism. Bucket4j's default `LOCK_FREE` strategy (used by `rateLimitBucket`, built without an explicit `SynchronizationStrategy` at line 31 of `ThrottleConfiguration.java`) is CAS-based and thread-safe — concurrent `tryConsume` calls are individually atomic. The vulnerability is purely logical, not a race condition: even sequential requests can drain the rate-limit bucket through gas-failed requests.

**Exploit flow:**

1. Attacker sends 1 request with `gas = gasPerSecond` (default 7,500,000,000). After `scaleGas()` (÷10,000), this is 750,000 tokens — exactly the gas bucket's full capacity. This single request depletes `gasLimitBucket` entirely and consumes 1 rate-limit token.
2. Attacker sends 499 additional requests (any `gas > 10,000` so `scaleGas` returns > 0). Each request:
   - Passes `rateLimitBucket.tryConsume(1)` → consumes 1 rate-limit token (irrevocably)
   - Fails `gasLimitBucket.tryConsume(...)` → gas bucket is empty → `ThrottleException` thrown
3. After 500 total requests (default `requestsPerSecond = 500`), `rateLimitBucket` is empty.
4. All subsequent legitimate requests within that second are rejected with `REQUEST_PER_SECOND_LIMIT_EXCEEDED`.
5. Both buckets refill after 1 second (`refillGreedy(capacity, Duration.ofSeconds(1))`), so the attack is repeatable every second indefinitely.

**Why existing checks are insufficient:**

- `restore(long gas)` (lines 59–64) only calls `gasLimitBucket.addTokens(tokens)` — there is no `rateLimitBucket.addTokens(1)` anywhere in the codebase.
- The `rateLimitBucket` is built without `SynchronizationStrategy.SYNCHRONIZED` (line 31, `ThrottleConfiguration.java`) while `gasLimitBucket` uses it (line 42) — this asymmetry is a secondary inconsistency but does not create the core flaw.

### Impact Explanation
An unauthenticated attacker can completely block all legitimate web3 API traffic for the full duration of each 1-second refill window, repeatable indefinitely. With default settings (500 req/s rate limit, 7.5B gas/s), the attacker needs only 500 HTTP requests per second — trivially achievable from a single machine. This constitutes a sustained, low-cost denial-of-service against the mirror node's web3 endpoint, preventing any contract calls from being processed and causing missing or delayed records exported to mirror nodes.

### Likelihood Explanation
No authentication, API key, or special privilege is required. The attacker only needs network access to the web3 endpoint. The attack is fully repeatable every second, requires no state, and can be automated with a simple HTTP client loop. Any public-facing deployment is exposed.

### Recommendation
Refund the rate-limit token when the gas check fails. The fix should be applied atomically in `throttle()`:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);  // refund the rate-limit token
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, check the gas bucket first (without consuming) using `gasLimitBucket.getAvailableTokens()` before consuming the rate-limit token, or restructure to consume both atomically. Also apply `SynchronizationStrategy.SYNCHRONIZED` consistently to `rateLimitBucket` to match `gasLimitBucket`.

```java
    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
```
