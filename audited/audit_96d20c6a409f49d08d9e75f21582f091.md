### Title
Global Gas Bucket Exhaustion via Concurrent Long-Running Requests (Unauthenticated DoS)

### Summary
`ThrottleManagerImpl.throttle()` deducts gas tokens from the shared `gasLimitBucket` **before** EVM execution begins, and `ContractCallService.restoreGasToBucket()` returns unused tokens only **after** execution completes. Because both the rate-limit and gas buckets are global singletons with no per-IP or per-user partitioning, an unauthenticated attacker can keep enough concurrent requests in-flight to hold all 750,000 scaled tokens, causing every subsequent legitimate call to fail with `GAS_PER_SECOND_LIMIT_EXCEEDED` for the duration of the attack.

### Finding Description

**Exact code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` (line 40) consumes `scaleGas(request.getGas())` tokens from `gasLimitBucket` synchronously, before any EVM work starts. [1](#0-0) 

Tokens are only returned inside `ContractCallService.restoreGasToBucket()`, which is called from the `finally` block of `doProcessCall()` — i.e., after `transactionExecutionService.execute()` returns. [2](#0-1) [3](#0-2) 

**Bucket capacity math:**

- `gasPerSecond` default = 7,500,000,000; `GAS_SCALE_FACTOR` = 10,000 → bucket holds **750,000 scaled tokens**, refills at 750,000/s. [4](#0-3) 
- `maxGasLimit` = 15,000,000 → `scaleGas(15_000_000)` = **1,500 tokens per request**. [5](#0-4) 
- 750,000 / 1,500 = **500 concurrent max-gas requests exhaust the bucket**.

**Root cause — failed assumption:**

The design assumes that requests complete fast enough that restored tokens keep pace with new consumption. There is no bound on how long `transactionExecutionService.execute()` can run, and there is no per-source rate limit. An attacker who keeps ≥500 requests in-flight simultaneously holds all tokens indefinitely.

**Why the restore logic does not help:**

For a successful call, `restoreGasToBucket` returns only `min(gasLimit − gasUsed, gasLimit)` — i.e., the unused portion. For a failed call where all gas is consumed, it returns the full `gasLimitToRestoreBaseline`. Either way, the restore happens **after** execution, so tokens are unavailable to other callers during the entire execution window. [6](#0-5) 

**Why the rate-limit bucket does not help:**

`rateLimitBucket` is also a global singleton (500 req/s capacity). An attacker sending at 250 req/s stays well under the rate limit while still exhausting the gas bucket if each request takes ≥2 seconds. Legitimate users pass the rate-limit check but then fail the gas check. [7](#0-6) [8](#0-7) 

### Impact Explanation

All legitimate `POST /api/v1/contracts/call` requests that arrive while the gas bucket is empty receive HTTP 429 with `GAS_PER_SECOND_LIMIT_EXCEEDED`. Because the bucket is global, **every user on the node** is affected simultaneously. The attacker pays no on-chain cost (this is a read-only mirror-node endpoint). The bucket refills at 750,000 tokens/s, but the attacker can continuously replenish the drain by keeping new requests in-flight, sustaining the outage indefinitely.

### Likelihood Explanation

**Preconditions:** None. No authentication, no API key, no on-chain account required. [9](#0-8) 

**Feasibility:** Any caller can set `gas: 15000000` in the JSON body. Complex contract calls (e.g., calls that iterate over large storage, or calls to contracts with expensive precompiles) routinely take multiple seconds on a mirror node. The attacker needs only a single HTTP client capable of 250 concurrent connections — trivially achievable with `curl`, `ab`, or any load-testing tool.

**Repeatability:** The attack is stateless and can be restarted immediately after any interruption. No cooldown or lockout mechanism exists.

### Recommendation

1. **Per-source rate limiting:** Introduce a per-IP (or per-API-key) sub-bucket so a single source cannot monopolize the global gas budget.
2. **Execution timeout:** Enforce a hard wall-clock timeout on `transactionExecutionService.execute()` so tokens are never held for more than a bounded interval.
3. **Consume tokens proportionally after execution:** Deduct only `gasUsed` (not `gasLimit`) from the bucket at throttle time, or use a reservation/commit pattern: reserve a small deposit up-front and settle the true cost post-execution.
4. **Reject oversized gas early:** Move `validateContractMaxGasLimit` **before** `throttleManager.throttle()` so invalid-gas requests never touch the bucket. [10](#0-9) 

### Proof of Concept

```bash
# Requires: a deployed contract whose call takes ~3 seconds (e.g., a loop-heavy function)
# Target: POST /api/v1/contracts/call on an unprotected mirror-node instance

PAYLOAD='{"to":"<slow_contract_address>","gas":15000000,"data":"<expensive_function_selector>"}'

# Step 1 – flood with 250 concurrent long-running requests (stays under 500 req/s rate limit)
for i in $(seq 1 250); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
       -H 'Content-Type: application/json' \
       -d "$PAYLOAD" &
done

# Step 2 – immediately probe with a legitimate low-gas request from a different client
curl -X POST http://<mirror-node>/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     -d '{"to":"<any_contract>","gas":100000,"data":"0x"}'
# Expected response: 429 {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}

# Step 3 – repeat step 1 every second to sustain the outage
```

**Expected result:** The legitimate low-gas probe in step 2 is rejected with `GAS_PER_SECOND_LIMIT_EXCEEDED` even though it requested only 100,000 gas, because the 250 concurrent max-gas requests hold all 750,000 scaled tokens in-flight.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }

    @Min(0)
    @Max(100)
    private float gasLimitRefundPercent = 100;

    @Min(21_000)
    @Max(10_000_000_000_000L)
    private long gasPerSecond = 7_500_000_000L;

    @Min(1)
    private long opcodeRequestsPerSecond = 1;

    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
    }
