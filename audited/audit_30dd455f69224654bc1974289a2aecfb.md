### Title
Cache Miss DoS via Uncached Negative Lookups in `findByIdAndDeletedIsFalse`

### Summary
`EntityRepository.findByIdAndDeletedIsFalse` uses `unless = "#result == null"` to skip caching. Because Spring Cache unwraps `Optional<Entity>` return values before evaluating SpEL — treating `Optional.empty()` as null — lookups for non-existent entity IDs are never cached. An unauthenticated attacker can repeatedly query the public `/api/v1/contracts/call` endpoint with crafted addresses that resolve to non-existent entity IDs, forcing every call to hit the database. The only protection is a global (not per-IP, not distributed) in-memory rate limiter.

### Finding Description
**Code location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 20–30:

```java
@Caching(cacheable = {
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_ENTITY,
               unless = "#result == null"),
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_SYSTEM_ACCOUNT,
               condition = "#entityId < 1000 && ...",
               unless = "#result == null")
})
Optional<Entity> findByIdAndDeletedIsFalse(Long entityId);
```

**Root cause:** Spring's cache abstraction unwraps `Optional<T>` return values before evaluating `unless` SpEL expressions. When the entity does not exist, the method returns `Optional.empty()`, which Spring unwraps to `null`. The condition `unless = "#result == null"` then evaluates to `true`, so the result is **never stored in cache**. This is confirmed by the test at `EntityRepositoryTest.java` lines 506–520, which asserts that after cache invalidation, a non-existent entity returns empty (i.e., no negative-result caching occurs).

**Exploit flow:**
1. Attacker crafts a "long-zero" EVM address encoding a non-existent entity num, e.g. `0x000000000000000000000000FFFFFFFFFFFFFFFF`.
2. `CommonEntityAccessor.getEntityByMirrorAddressAndTimestamp` (line 74–78) calls `entityIdNumFromEvmAddress(address)` → `DomainUtils.fromEvmAddress` extracts the last 8 bytes as a `long` → passes it to `findByIdAndDeletedIsFalse`.
3. No entity exists → `Optional.empty()` → Spring unwraps to null → `unless = "#result == null"` fires → **not cached**.
4. Every subsequent request with any non-existent ID hits the database. The attacker rotates through many non-existent IDs to prevent any incidental caching.

**Why the entity cache config does not help:** `CacheProperties.java` line 19 sets `expireAfterWrite=1s,maximumSize=10000` — but this only applies to entries that are actually stored. Negative results are never stored, so the cache provides zero protection for this path.

### Impact Explanation
Every request targeting a non-existent entity ID issues a live SQL query (`SELECT * FROM entity WHERE id = ? AND deleted IS NOT TRUE`). Under sustained attack, this generates a steady stream of DB queries proportional to the request rate. In a multi-instance deployment, each instance has its own independent rate-limit bucket (`ThrottleConfiguration.java` lines 24–32 — a local `Bucket4j` instance), so the effective DB query rate scales linearly with the number of running instances. This can exhaust DB connection pools or degrade query latency for legitimate users.

### Likelihood Explanation
The `/api/v1/contracts/call` endpoint (`ContractController.java` line 37–51) requires no authentication and is publicly documented in `openapi.yml` lines 461–518. The only throttle is a global in-memory bucket of 500 req/s (`ThrottleProperties.java` line 35), shared across all callers — not per-IP. A single attacker can consume the full 500 req/s budget. The attack requires no special knowledge: any 20-byte address with a zero 12-byte prefix and a non-existent entity num in the last 8 bytes suffices. The attack is trivially repeatable and automatable.

### Recommendation
1. **Cache negative results with a short TTL:** Change `unless = "#result == null"` to `unless = "false"` (or remove it) and rely on the existing 1-second `expireAfterWrite` to expire stale negatives. This bounds DB queries to at most `maximumSize` unique IDs per TTL window.
2. **Add per-IP rate limiting:** Introduce an IP-keyed rate limiter (e.g., using Bucket4j's distributed or per-key mode) upstream of the entity lookup path, so a single attacker cannot consume the full global budget.
3. **Use a distributed rate limiter:** Replace the local `Bucket4j` bucket with a Redis-backed distributed bucket so the limit is enforced across all instances.

      "to":   "0x00000000000000000000000000000000FFFFFFFF",
      "from": "0x0000000000000000000000000000000000000001",
      "gas":  50000,
      "data": "0x"
    }' &
done
wait
# Each request triggers findByIdAndDeletedIsFalse(4294967295L),
# returns Optional.empty(), is NOT cached, and issues a live DB query.
# Rotate the last 4 bytes across requests to prevent any accidental caching.
