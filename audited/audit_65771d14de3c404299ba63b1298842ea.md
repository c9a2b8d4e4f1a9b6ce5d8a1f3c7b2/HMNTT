### Title
Unauthenticated `X-Real-IP` Header Spoofing in `TracingMiddleware` Enables Log Poisoning

### Summary
`getClientIpAddress` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` header without validating that the request originated from a trusted reverse proxy. Because the rosetta container directly exposes port 5700 alongside the nginx proxy on port 8080, an attacker who connects directly to port 5700 can inject any IP address into the access logs, including IPs belonging to known validators or network nodes, permanently falsifying the operator's audit trail of API access patterns.

### Finding Description
**Code path:**
- `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74
- `TracingMiddleware()`, lines 43–61

`getClientIpAddress` reads `X-Real-IP` first (line 64), then `X-Forwarded-For` (line 67), and only falls back to `r.RemoteAddr` (line 71) if both headers are absent. There is no check that the request arrived from a trusted upstream proxy before accepting these headers. The resolved IP is then embedded verbatim into a `log.Info` message (line 52–53) for every non-internal API path.

The nginx proxy in `docker-compose.yml` (line 204) does overwrite `X-Real-IP` with `$$remote_addr` for requests routed through port 8080. However, the rosetta container independently binds port 5700 to the host (`- 5700:5700`, line 151). Requests arriving directly on port 5700 bypass nginx entirely, so the attacker-controlled `X-Real-IP` value reaches `getClientIpAddress` unmodified.

The test suite (`trace_test.go`, lines 52–58) explicitly confirms the intended behavior: a client-supplied `X-Real-IP: 10.0.0.100` is accepted and logged without any proxy-origin check.

**Failed assumption:** The code assumes `X-Real-IP` is always set by a trusted reverse proxy that has already validated the originating address. This assumption breaks whenever the service port is reachable directly.

### Impact Explanation
Every Rosetta API call (construction, block, account, network endpoints) is logged at `INFO` level with the forged IP. An operator relying on these logs to audit which network participants are querying transaction construction or submission endpoints will see fabricated attribution. Over time this corrupts the only available record of API access patterns, making it impossible to distinguish legitimate node activity from attacker activity, and potentially implicating innocent validators in suspicious query patterns. The impact is confined to the logging/audit layer; no on-chain state or transaction data is altered.

### Likelihood Explanation
The precondition is simply network reachability to port 5700. In any deployment where the host is internet-facing and port 5700 is not firewalled (the default docker-compose configuration does not add firewall rules), any unauthenticated attacker can exploit this with a single HTTP request. No credentials, tokens, or special knowledge are required. The attack is trivially repeatable and leaves no reliable forensic trace because the forged IP overwrites the real source address in the only log record.

### Recommendation
1. **Validate proxy origin before trusting headers.** Maintain a configurable allowlist of trusted proxy CIDR ranges. In `getClientIpAddress`, only accept `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` falls within that allowlist; otherwise fall back to `r.RemoteAddr` directly.
2. **Restrict direct port exposure.** Remove the host-level port binding for rosetta (`5700:5700`) in production deployments so all traffic must pass through the nginx proxy, which already overwrites the header correctly.
3. **Use a hardened IP-extraction library** (e.g., `realip` middleware packages that enforce trusted-proxy semantics) rather than a hand-rolled header read.

### Proof of Concept
```bash
# Precondition: rosetta port 5700 is reachable directly (default docker-compose)
# Forge X-Real-IP to impersonate a known validator node (e.g., 35.237.200.180)

curl -s http://<rosetta-host>:5700/network/list \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 35.237.200.180" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"metadata":{}}'

# Expected result in rosetta application logs:
# level=info msg="35.237.200.180 POST /network/list (200) in 12ms"
#
# The real attacker IP never appears. All subsequent requests with the same
# forged header are permanently attributed to 35.237.200.180 in the audit log.
```
