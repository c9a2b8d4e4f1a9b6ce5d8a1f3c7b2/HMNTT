### Title
Unbounded Historical Timestamp Scan DoS via `getContractStateById` TIMESTAMP Filter

### Summary
The `/api/v1/contracts/{id}/state` endpoint accepts a user-supplied `timestamp` query parameter with no lower-bound or range validation. When a timestamp is present, the query switches from the lightweight `contract_state` table to a `DISTINCT ON (slot)` scan over the entire `contract_state_change` history table, bounded only by `consensus_timestamp <= <user_value>`. An unprivileged attacker can supply the maximum allowed timestamp (`9999999999`) to force a full historical table scan on every request, exhausting database CPU and I/O.

### Finding Description

**Exact code path:**

1. `getContractStateById()` — `rest/controllers/contractController.js` lines 964–994 — receives the raw HTTP request and calls `extractContractStateByIdQuery()`.

2. `extractContractStateByIdQuery()` — lines 894–956 — iterates over filters. For `filterKeys.TIMESTAMP` it only rejects the `ne` operator and converts `eq` to `lte`; it then pushes the condition directly into the WHERE clause and sets `timestampPresent = true`. **No minimum value, no range width, and no "must not exceed now" check is applied.**

3. `getContractStateByIdAndFilters()` — `rest/service/contractService.js` lines 245–267 — when `timestamp === true`, it substitutes `contractStateTimestampQuery` (lines 111–122) for the normal `contractStateQuery`. The timestamp query is:
   ```sql
   SELECT DISTINCT ON (slot)
     contract_id, slot, evm_address,
     coalesce(value_written, value_read) AS value,
     consensus_timestamp AS modified_timestamp
   FROM contract_state_change csc
   LEFT JOIN entity e ON e.id = csc.contract_id
   WHERE contract_id = $1
     AND consensus_timestamp <= $2   -- ← user-controlled, unbounded
   ORDER BY slot, consensus_timestamp DESC
   LIMIT $3
   ```
   `DISTINCT ON` with `ORDER BY slot, consensus_timestamp DESC` requires PostgreSQL to enumerate **every row** for the contract up to the given timestamp before the `LIMIT` can be applied. With `$2 = 9999999999000000000` (nanoseconds), this is the entire lifetime of the contract.

**Root cause / failed assumption:**
The code assumes the timestamp filter is already range-validated before reaching `extractContractStateByIdQuery`. In reality, the validation chain (`buildAndValidateFilters` → `validateAndParseFilters` → `filterValidityChecks`) only checks the *format* (`/^\d{1,10}(\.\d{1,9})?$/`). The `parseTimestampFilters` utility — which enforces a configurable maximum range width and rejects open-ended ranges — is **never called** for this endpoint. Other endpoints (e.g., `balances.js`) apply `getOptimizedTimestampRange` to clamp the lower bound; no equivalent guard exists here.

**Why the LIMIT does not help:**
PostgreSQL must resolve `DISTINCT ON` before applying `LIMIT`. The planner cannot push the limit inside the distinct computation when the ordering key (`slot, consensus_timestamp DESC`) differs from the partition key of the table.

### Impact Explanation
An attacker targeting a high-activity contract (many slots, many historical writes) can issue repeated requests that each force a full sequential scan of `contract_state_change` partitions. This saturates database I/O and CPU, degrading or denying service to all other API consumers. Because `contract_state_change` is partitioned by `consensus_timestamp` (v2 schema), a `<= MAX` predicate crosses every partition, amplifying the cost. The severity matches the stated classification: potential shutdown of ≥30% of network processing nodes without brute force, as a small number of concurrent requests against a busy contract is sufficient.

### Likelihood Explanation
No authentication is required. The endpoint is publicly documented in the OpenAPI spec (`/api/v1/contracts/{contractIdOrAddress}/state`). The attack requires only knowledge of a contract ID with significant historical state — easily discoverable from the public ledger. The request is trivially repeatable with a single `curl` command. No rate-limiting logic is visible in the controller or service layer for this endpoint.

### Recommendation
1. **Enforce an upper-bound cap on the timestamp value**: reject timestamps that exceed `now + small_delta` (e.g., 5 seconds), mirroring the guard in `balances.js`.
2. **Apply `parseTimestampFilters` with `validateRange: true`** in `extractContractStateByIdQuery`, enforcing the existing `config.query.maxTimestampRange` limit so that open-ended `lte:MAX` queries are rejected.
3. **Require a lower-bound companion filter** when a `lte`/`lt` timestamp is supplied (i.e., disallow open ranges), consistent with the `allowOpenRange: false` default in `parseTimestampFilters`.
4. Consider rewriting the `DISTINCT ON` query to use the index `contract_state_change__id_slot_timestamp (contract_id, slot, consensus_timestamp)` with a lateral/correlated subquery per slot, so the `LIMIT` can be pushed down.

### Proof of Concept
```
# Step 1 – identify a contract with significant historical state (public ledger)
CONTRACT_ID="0.0.1234"

# Step 2 – send a single request with maximum allowed timestamp
curl "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}/state?timestamp=lte:9999999999&limit=100"

# Step 3 – repeat concurrently to exhaust DB resources
for i in $(seq 1 50); do
  curl -s "https://<mirror-node>/api/v1/contracts/${CONTRACT_ID}/state?timestamp=lte:9999999999&limit=100" &
done
wait
```
