### Title
Unbounded `maxRetries` Starves Ticker Goroutine, Silently Disabling Transfer Submissions

### Summary
`HIERO_MIRROR_PINGER_MAX_RETRIES` is accepted from the environment and stored in `cfg.maxRetries` with no upper-bound validation — only a lower-bound clamp to 0. Because `submitWithRetry` is called **synchronously** inside the ticker's `select` case, setting this value to `2147483647` causes the retry loop to run ~2.1 billion iterations, blocking the ticker goroutine for an astronomically long time. The separate heartbeat goroutine continues writing `/tmp/alive`, so liveness probes keep passing and the process appears healthy while submitting zero transfers.

### Finding Description

**Config parsing — no upper bound:**
`pinger/config.go` lines 58–67 read the env var and parse it with `strconv.Atoi`. The only post-parse validation (lines 117–119) clamps negative values to 0; there is no maximum:

```go
// config.go:58-67
retriesStr := envOr("HIERO_MIRROR_PINGER_MAX_RETRIES", "10")
flag.Func("max-retries", "max retries per tick", func(s string) error {
    v, err := strconv.Atoi(s)
    ...
    cfg.maxRetries = v   // no upper bound
    return nil
})

// config.go:117-119
if cfg.maxRetries < 0 {
    cfg.maxRetries = 0   // only lower-bound guard
}
```

**Retry loop — synchronous, blocks ticker:**
`pinger/transfer.go` lines 21–57 compute `attempts = cfg.maxRetries + 1` and loop. With `maxRetries = 2147483647`, `attempts = 2147483648`. Each failed iteration sleeps up to 30 s (backoff capped at `30 * time.Second`, line 64). The only early-exit is `ctx.Err()` (line 24) and the backoff `select` on `ctx.Done()` (lines 50–55) — neither fires during normal operation without a shutdown signal.

**Ticker goroutine blocked:**
`pinger/main.go` line 63 calls `submitWithRetry` synchronously inside the `case <-ticker.C:` branch. While `submitWithRetry` is running, no new tick can be processed:

```go
// main.go:62-68
case <-ticker.C:
    if err := submitWithRetry(ctx, client, cfg); err != nil {
        ...
    }
```

**Heartbeat goroutine unaffected:**
`pinger/main.go` lines 28–39 run the liveness heartbeat in a **separate goroutine** that writes `/tmp/alive` every 15 s. This goroutine is never blocked, so the Kubernetes liveness exec probe keeps passing even though the transfer ticker is completely stalled.

### Impact Explanation
The pinger's sole purpose is to submit periodic HBAR transfers and confirm receipts. With the ticker goroutine blocked indefinitely, **no transfers are submitted for the entire duration of the retry storm** — effectively a complete functional denial-of-service. Because the liveness probe continues to pass (heartbeat goroutine is independent) and `/tmp/ready` was written at startup, Kubernetes does not restart the pod. Monitoring systems that rely on the pinger to verify network health receive no data, silently masking network or consensus-layer outages.

### Likelihood Explanation
The precondition is the ability to set one environment variable for the pinger process. In Kubernetes deployments this requires write access to the Deployment manifest or a referenced ConfigMap/Secret — access that may be granted to developers or CI/CD pipelines without full cluster-admin rights. In non-containerized deployments, any user who can launch or restart the process controls its environment. The attack is trivially repeatable: every restart with the malicious env var re-triggers the condition, and because the process appears healthy, automated remediation (pod restart) is not triggered.

### Recommendation
Add an explicit upper-bound check in `loadConfig()` immediately after the existing lower-bound clamp in `pinger/config.go`:

```go
const maxAllowedRetries = 100   // or any operationally reasonable ceiling

if cfg.maxRetries < 0 {
    cfg.maxRetries = 0
}
if cfg.maxRetries > maxAllowedRetries {
    return cfg, fmt.Errorf("max-retries %d exceeds maximum allowed value %d", cfg.maxRetries, maxAllowedRetries)
}
```

Additionally, consider running `submitWithRetry` in a child goroutine with a per-tick deadline context so that a runaway retry loop cannot block the ticker regardless of configuration:

```go
case <-ticker.C:
    tickCtx, cancel := context.WithTimeout(ctx, cfg.interval)
    go func() {
        defer cancel()
        if err := submitWithRetry(tickCtx, client, cfg); err != nil { ... }
    }()
```

### Proof of Concept

```bash
# 1. Build the pinger binary
cd pinger && go build -o pinger .

# 2. Launch with an absurdly large retry count and a fast interval
HIERO_MIRROR_PINGER_MAX_RETRIES=2147483647 \
HIERO_MIRROR_PINGER_INTERVAL=1s \
HIERO_MIRROR_PINGER_BASE_BACKOFF=1s \
HIERO_MIRROR_PINGER_OPERATOR_ID=0.0.2 \
HIERO_MIRROR_PINGER_OPERATOR_KEY=<valid_key> \
./pinger

# 3. Observe: after the first tick fires and the first transfer attempt fails,
#    the process logs "attempt 1/2147483648 failed: ..." and enters the retry loop.
#    No subsequent "Starting transfer ticker" tick log lines appear.
#    /tmp/alive is refreshed every 15 s (heartbeat goroutine alive).
#    /tmp/ready exists (written at startup).
#    The process is unkillable by liveness probes and submits zero transfers.
```
