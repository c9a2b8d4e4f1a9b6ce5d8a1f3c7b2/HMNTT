### Title
Unbounded Throttled Polling Loop Enables Unauthenticated DB Connection Pool Exhaustion DoS

### Summary
Any unauthenticated external user can open an unlimited number of concurrent `subscribeTopic` gRPC subscriptions with no limit set. Each subscription unconditionally invokes `retrieve(filter, true)` (throttled=true), setting `numRepeats = Long.MAX_VALUE` and polling the database every 2 seconds indefinitely. The only termination guard — `isComplete()` — fails to fire on a live mirror node with continuous message ingestion, and the per-emission timeout does not bound total subscription lifetime. Flooding the server with such subscriptions exhausts the database connection pool, causing a non-network DoS.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicMessageRetriever.retrieve(filter, true)` → `PollingContext(filter, throttled=true)`.

In `PollingContext` constructor: [1](#0-0) 

`numRepeats` is unconditionally set to `Long.MAX_VALUE` when `throttled=true`. The repeat loop in `retrieve()`: [2](#0-1) 

runs up to `Long.MAX_VALUE` iterations, terminating only when `isComplete()` returns `true`.

**Root cause — failed termination assumption:**

`isComplete()` for the throttled path: [3](#0-2) 

returns `true` only when the last DB page returned fewer rows than `maxPageSize` (default 1000), or a limit is hit. On a live Hedera mainnet mirror node with millions of historical messages and continuous ingestion, every poll returns a full page of 1000 messages, so `isComplete()` never fires. The loop runs indefinitely.

**Timeout is insufficient:**

The timeout applied: [4](#0-3) 

is Reactor's per-emission idle timeout (default 60s), not a total subscription lifetime cap. Since each poll emits up to 1000 messages every 2 seconds, items flow continuously and the timeout never fires.

**No authentication or global subscriber cap:**

`ConsensusController.subscribeTopic()` has no authentication check: [5](#0-4) 

`subscriberCount` is a metric gauge only, never enforced as a limit: [6](#0-5) 

**Per-connection limit is bypassable:**

`maxConcurrentCallsPerConnection = 5` is per-connection: [7](#0-6) 

An attacker opens N connections × 5 calls each = 5N concurrent indefinitely-polling subscriptions.

**Each poll holds a DB resource:**

Each `poll()` call opens a DB query via `topicMessageRepository.findByFilter()`: [8](#0-7) 

With many concurrent subscriptions each polling every 2 seconds, the DB connection pool is saturated.

### Impact Explanation

Exhausting the database connection pool blocks all other gRPC subscribers and internal services (importer, REST API) from acquiring DB connections, causing a complete service outage. The mirror node is critical infrastructure for Hedera ecosystem applications; its unavailability affects all downstream consumers of topic messages, account queries, and transaction data. This qualifies as a high-severity non-network DoS against a project with significant market capitalization impact.

### Likelihood Explanation

The attack requires zero privileges — the gRPC `subscribeTopic` endpoint is publicly accessible with no authentication. The attacker needs only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK), a topic ID with historical messages (publicly known on mainnet), and the ability to open multiple TCP connections. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public protobuf API. On mainnet, topics like the address book topic have millions of messages, guaranteeing full pages on every poll.

### Recommendation

1. **Enforce a total subscription lifetime cap**: Replace the per-emission `.timeout()` with a hard wall-clock deadline (e.g., `Flux.timeout(maxSubscriptionDuration)` wrapping the entire flux), independent of message flow rate.
2. **Enforce a global concurrent subscriber limit**: Check `subscriberCount` against a configurable maximum before accepting new subscriptions; reject with `RESOURCE_EXHAUSTED` when exceeded.
3. **Bound `numRepeats` for throttled mode**: Replace `Long.MAX_VALUE` with a configurable maximum poll count (analogous to `unthrottled.maxPolls`) even in throttled mode, forcing eventual termination.
4. **Add rate limiting per source IP** at the gRPC layer to limit connection/call establishment rate.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools hedera-sdk or raw grpc stubs
# Target: a mirror node gRPC endpoint (port 5600)
# Precondition: topic 0.0.X has >= 1000 historical messages (e.g., mainnet address book topic)

import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mainnet-public.mirrornode.hedera.com:443"
TOPIC_NUM = 101  # address book topic with millions of messages
NUM_CONNECTIONS = 20  # 20 connections × 5 calls = 100 concurrent subscriptions

def flood_connection(_):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        # No limit set → limit=0 → hasLimit()=false → numRepeats=Long.MAX_VALUE
        # No consensusStartTime → defaults to epoch → millions of historical messages
    )
    for _ in range(5):  # maxConcurrentCallsPerConnection=5
        threading.Thread(
            target=lambda: list(stub.subscribeTopic(query)),
            daemon=True
        ).start()
    import time; time.sleep(3600)  # hold connections open

threads = [threading.Thread(target=flood_connection, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
# Result: 100 concurrent subscriptions each polling DB every 2s → connection pool exhausted
# Legitimate subscribers and internal services begin receiving DB connection errors
```
