### Title
Malicious Block Node Stalls Ingestion Indefinitely via Continuous Empty BlockItemSet Flood

### Summary
`BlockAssembler.onBlockItemSet()` silently discards empty `BlockItemSet` messages with only a log warning and no exception, while the outer `streamBlocks()` loop resets the error counter to zero on every response. A malicious block node can flood the importer with empty `BlockItemSet`s, exhausting the per-call timeout without triggering `onError()`, keeping the node permanently active, and causing the importer to reconnect and repeat the cycle indefinitely — stalling block ingestion with no automatic recovery.

### Finding Description

**Exact code path:**

`BlockNode.streamBlocks()` — `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, lines 138–158:

```java
while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
    switch (response.getResponseCase()) {
        case BLOCK_ITEMS -> assembler.onBlockItemSet(response.getBlockItems());
        ...
    }
    errors.set(0);   // ← resets error counter on EVERY response, including empty ones
}
```

`BlockAssembler.onBlockItemSet()` — lines 224–229:

```java
void onBlockItemSet(final BlockItemSet blockItemSet) {
    var blockItems = blockItemSet.getBlockItemsList();
    if (blockItems.isEmpty()) {
        log.warn("Received empty BlockItemSet from block node");
        return;   // ← silent discard, no exception, stopwatch not reset
    }
    ...
}
```

**Root cause and failed assumption:**

The code assumes that empty `BlockItemSet`s are rare/transient anomalies. It treats them as non-errors, so:

1. No exception is thrown → `onError()` (lines 159–164) is never called → `errors` counter is never incremented.
2. `errors.set(0)` at line 157 is called after every response, including empty ones, actively preventing any error accumulation.
3. The `BlockAssembler.timeout()` stopwatch (lines 274–281) starts on the first `grpcCall.read()` call and counts down from `commonDownloaderProperties.getTimeout()` (default **30 seconds**, `CommonDownloaderProperties.java` line 83). When elapsed > timeout, `timeout()` returns a negative value, causing `grpcCall.read(negativeMs, ...)` to return `null` and exit the loop — but this is a **normal** exit with no exception.
4. After normal exit, `onError()` is not called, the node's `active` flag stays `true`, and the next scheduled `doGet()` (every 100 ms per `BlockProperties.frequency`) selects the same malicious node again.

**Why existing checks fail:**

- `maxSubscribeAttempts = 3` (StreamProperties line 27) only fires when `onError()` is called, which requires an exception. Empty `BlockItemSet`s never produce one.
- The 30-second timeout bounds each individual `streamBlocks()` call but does not prevent immediate reconnection to the same node.
- `errors.set(0)` at line 157 actively resets any error state on each empty response, making it impossible for the error counter to accumulate even if other mechanisms tried to use it.

### Impact Explanation

Block ingestion is completely stalled for as long as the malicious node is the selected node. The importer spins in a 30-second connect → flood → timeout → reconnect cycle, making zero progress on block processing. CPU is consumed by repeated gRPC reads and warning log writes. If the malicious node is the only configured or auto-discovered node, recovery is impossible without operator intervention. If multiple nodes are configured, the malicious node monopolizes the importer's single-node-at-a-time streaming model (`BlockNodeSubscriber` uses one node at a time per `doGet()`), delaying or blocking all block processing.

### Likelihood Explanation

**Precondition**: The attacker must control a node that appears in the importer's block node list. This is achievable via:
- **Auto-discovery** (`autoDiscoveryEnabled = true` by default, `BlockProperties` line 27): any node registered in the `RegisteredNodeRepository` database as a `BLOCK_NODE` with `TIER_ONE_BLOCK_NODE_APIS` (STATUS + PUBLISH + SUBSCRIBE_STREAM) is automatically included (`BlockNodeDiscoveryService.java` lines 73–92). A network participant who can register a node on-chain satisfies this.
- **Direct configuration**: an operator who misconfigures a node endpoint.

**Repeatability**: The attack is trivially repeatable — the malicious node simply never sends a non-empty `BlockItemSet`. The gRPC stream stays open, the importer loops, and the cycle repeats every 30 seconds indefinitely with no operator alert beyond repeated `WARN` log lines.

### Recommendation

1. **Count empty `BlockItemSet`s as errors**: In `onBlockItemSet()`, instead of silently returning, call `onError()` or throw a `BlockStreamException` after N consecutive empty sets (e.g., configurable threshold).
2. **Do not reset `errors` on empty responses**: Move `errors.set(0)` inside the branches that represent genuine progress (e.g., only after `onEndOfBlock()` successfully completes a block).
3. **Track no-progress timeout separately**: If the stopwatch elapses without a completed block (`onEndOfBlock()` called), treat it as a hard error that increments the error counter and marks the node inactive.
4. **Cap consecutive empty-set tolerance**: Add a counter in `BlockAssembler` for consecutive empty `BlockItemSet`s and throw `BlockStreamException` once it exceeds a small threshold (e.g., 5).

### Proof of Concept

1. Stand up a gRPC server implementing `BlockStreamSubscribeService.subscribeBlockStream()` that, upon receiving any `SubscribeStreamRequest`, responds with an infinite stream of `SubscribeStreamResponse` messages where each message has `BLOCK_ITEMS` set to an empty `BlockItemSet` (no items in `blockItemsList`).

2. Register this server as a block node (either via direct config in `hiero.mirror.importer.block.nodes` or via the `RegisteredNodeRepository` with `BLOCK_NODE` type and all three tier-1 APIs).

3. Start the importer. Observe:
   - `WARN` log lines: `"Received empty BlockItemSet from block node"` repeating at high frequency.
   - No blocks are processed (block number does not advance).
   - After ~30 seconds, the connection resets and immediately reconnects to the same node.
   - The node is never marked inactive (`node.isActive()` remains `true`).
   - `errors` counter stays at 0 throughout.
