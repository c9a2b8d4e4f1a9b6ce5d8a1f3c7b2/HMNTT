### Title
Cache Pollution via `Optional.empty()` Caching in `findRuntimeBytecode()` Enables Unauthenticated DoS

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"` as its exclusion condition. Because the method returns `Optional<byte[]>`, a miss returns `Optional.empty()` — which is a non-null object — so the `unless` guard never fires and empty results are cached. An unauthenticated attacker can flood the endpoint with unique non-existent `contractId` values, filling the bounded 1,000-entry cache entirely with `Optional.empty()` tombstones, evicting all legitimate bytecode entries and forcing every subsequent real contract lookup to hit the database.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18.

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring Cache evaluates `#result` as the actual return value. When no contract row exists, Spring Data JPA returns `Optional.empty()`. `Optional.empty() != null`, so `#result == null` is `false`, the `unless` guard does not suppress caching, and the empty `Optional` is stored in the Caffeine cache under the queried `contractId` key.

**Cache configuration** (`CacheProperties.java`, line 22):
```
expireAfterAccess=1h, maximumSize=1000
```

**Exploit flow:**
1. Attacker sends requests to any public web3 API endpoint that triggers `ContractBytecodeReadableKVState.readFromDataSource()` (e.g., `eth_getCode` with a crafted address).
2. Each request with a distinct non-existent `contractId` causes one DB query and stores `Optional.empty()` in the cache.
3. After 1,000 unique non-existent IDs, the cache is full of tombstones; Caffeine's LRU eviction removes legitimate bytecode entries.
4. Attacker cycles through fresh unique IDs to continuously evict any re-populated legitimate entries.
5. All real contract bytecode lookups now miss the cache and hit the database on every call.

**Why existing checks fail:** The `unless = "#result == null"` condition was intended to prevent caching of absent results, but it is semantically incorrect for an `Optional`-returning method. No rate limiting, no input validation on `contractId`, and no negative-result exclusion exist at any layer.

### Impact Explanation
The contract cache (`maximumSize=1000`, `expireAfterAccess=1h`) is completely neutralized. Every EVM execution that needs to load contract bytecode for a real contract must now perform a synchronous database query. Under sustained attack this translates to unbounded database load proportional to legitimate traffic, degrading or denying service for all users of the web3 module. The `expireAfterAccess=1h` TTL means evicted legitimate entries stay evicted for the duration of the attack with no self-healing.

### Likelihood Explanation
The attack requires zero privileges — only the ability to send HTTP requests to the public web3 JSON-RPC endpoint. The `contractId` parameter is a `Long`, giving an attacker 2^63 unique values to cycle through. No authentication, no CAPTCHA, and no observable rate-limiting layer is present in the reviewed code. The attack is trivially scriptable and fully repeatable.

### Recommendation
Change the `unless` condition to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This prevents negative-lookup tombstones from occupying cache slots. Additionally, consider adding rate limiting at the API gateway layer on `eth_getCode` / contract-bytecode-fetching endpoints, and consider a separate, smaller negative-result cache with a short TTL (e.g., 5–10 seconds) if repeated lookups of the same non-existent ID are a concern.

### Proof of Concept
```bash
# Generate 1001 unique non-existent contract IDs and flood the endpoint
for i in $(seq 9999999000 9999999001 9999999000000); do
  curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"data\":\"0x\",\"to\":\"0x$(printf '%040x' $i)\"}" &
done
wait

# Now query a real, previously-cached contract — it will hit the DB every time
curl -X POST http://<mirror-node-web3>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"<real_contract_evm_address>"}'
