### Title
Silent Transaction Omission in `FindBetween()` Pagination Due to Shared `consensus_timestamp` Boundary Skip

### Summary
The `FindBetween()` pagination cursor advances via `start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1`, which silently skips any additional `transaction` rows that share the same `consensus_timestamp` as the last row of a full 2000-row batch. The `transaction` table explicitly permits multiple rows per `consensus_timestamp` (differentiated by `nonce` for scheduled transactions), making this a real data-loss condition. Any unprivileged caller of the Rosetta `/block` endpoint can observe an incomplete ledger for any block where this boundary condition exists.

### Finding Description

**Exact code location**: `rosetta/app/persistence/transaction.go`, `FindBetween()`, lines 124–147. [1](#0-0) 

The pagination loop fetches up to `batchSize = 2000` rows ordered by `consensus_timestamp`. When a full batch is returned, the cursor advances:

```go
start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
```

**Root cause — failed assumption**: The code assumes `consensus_timestamp` is unique per row. It is not. The `transaction` table schema defines a composite primary key of `(consensus_timestamp, payer_account_id)`: [2](#0-1) 

Migration `V2.5.1` then dropped even that composite PK: [3](#0-2) 

The `transaction` table also carries a `nonce` column (default 0) specifically to accommodate scheduled transactions where the outer `ScheduleSign` and the inner executed transaction share the same `consensus_timestamp` but differ by `nonce`: [4](#0-3) 

**Exploit flow**:
1. A block exists on-chain with ≥ 2000 transactions.
2. The 2000th row (batch boundary) has `consensus_timestamp = T`.
3. One or more additional rows also have `consensus_timestamp = T` (e.g., a scheduled transaction execution sharing the same nanosecond timestamp as its triggering `ScheduleSign`).
4. The cursor advances to `T + 1`, permanently skipping those rows for this query.
5. The next batch fetches rows starting at `T + 1`, never revisiting timestamp `T`.

**Why existing checks fail**: The only termination check is `len(transactionsBatch) < batchSize` (line 142), which correctly detects end-of-data but does nothing to detect mid-timestamp truncation at a full-batch boundary. [5](#0-4) 

**Call path to user-facing API**: `block_service.go` line 56 calls `FindBetween` with the block's stored `ConsensusStartNanos`/`ConsensusEndNanos` in direct response to a `/block` request: [6](#0-5) 

### Impact Explanation
An incomplete transaction set is returned for the affected block. Downstream Rosetta consumers (exchange reconciliation systems, block explorers, audit tools) receive a silently truncated ledger — missing legitimate Hashgraph transactions. Because the omission is silent (no error is returned), consumers have no signal that the data is incomplete. This constitutes a tampered/incomplete view of Hashgraph history presented as authoritative, directly matching the threat model described in the question. Severity is **High** for data integrity; there is no confidentiality or availability impact, but ledger correctness is violated.

### Likelihood Explanation
- **Precondition 1**: A block must contain ≥ 2000 transactions. This is uncommon but occurs on Hedera mainnet during high-throughput periods.
- **Precondition 2**: The 2000th (or 4000th, 6000th, …) row must share its `consensus_timestamp` with at least one subsequent row. Scheduled transactions (ScheduleSign triggering execution) produce exactly this pattern.
- **Attacker capability**: Zero privilege required. Any caller of `GET /block` with a valid block identifier triggers the code path. No authentication, no special role.
- **Repeatability**: The condition is deterministic — once a block with this property exists, every `/block` request for it returns the same incomplete result.
- **Detectability**: No error is surfaced; the response appears normal.

### Recommendation
Replace the `+1` cursor with a `>` (strictly-greater-than) predicate that re-queries from the last seen timestamp using an offset or a tie-breaking secondary key. The safest fix is to track the last timestamp and, when a full batch is returned, issue a follow-up query for any additional rows at exactly that timestamp before advancing:

```go
// After collecting a full batch, fetch any remaining rows at the boundary timestamp
boundaryTs := transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp
var boundaryBatch []*transaction
db.Raw(selectTransactionsInTimestampRangeOrdered,
    sql.Named("start", boundaryTs),
    sql.Named("end", boundaryTs),
).Find(&boundaryBatch)
// merge boundaryBatch (deduplicating already-seen rows), then advance
start = boundaryTs + 1
```

Alternatively, change the query to use `(consensus_timestamp, payer_account_id, nonce)` as a composite cursor so the `LIMIT`/`OFFSET` or keyset pagination never splits a shared-timestamp group across batches.

### Proof of Concept

**Preconditions**:
- Mirror node database contains a block whose `consensus_start` to `consensus_end` range holds at least 2001 transactions.
- Transactions 2000 and 2001 (by `consensus_timestamp` order) share the same `consensus_timestamp` value (e.g., a `ScheduleSign` at timestamp `T` with `nonce=0` and its executed scheduled transaction at `T` with `nonce=1`).

**Steps**:
1. Identify the block index `B` containing the above transactions.
2. Send `POST /block` with `{"block_identifier": {"index": B}}` to the Rosetta API.
3. Observe the response: transaction at `(T, nonce=1)` is absent from `block.transactions`.
4. Confirm the missing transaction exists in the database: `SELECT * FROM transaction WHERE consensus_timestamp = T ORDER BY nonce`.
5. No error code is returned; the response HTTP status is 200 with a structurally valid but incomplete block.

**Result**: The Rosetta API presents an incomplete block, omitting one or more legitimate Hashgraph transactions, with no indication of truncation.
