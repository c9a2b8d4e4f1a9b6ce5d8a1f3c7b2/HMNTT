### Title
Inconsistent `chunkTotal` Across Chunks with Same `initialTransactionID` Stored Without Validation

### Summary
`doUpdateTransaction()` in `ConsensusSubmitMessageTransactionHandler` blindly persists the `chunkTotal` value from each individual transaction's `chunkInfo` without any cross-chunk consistency check. Because the Hedera consensus node itself does not enforce that `chunkTotal` is identical across all chunks sharing the same `initialTransactionID`, an unprivileged attacker can submit two consensus-accepted transactions with the same `initialTransactionID` but contradictory `chunkTotal` values, causing the mirror node to permanently store incoherent chunk metadata that breaks consumer message-reassembly logic.

### Finding Description
**Exact code path:**

In `importer/src/main/java/org/hiero/mirror/importer/parser/record/transactionhandler/ConsensusSubmitMessageTransactionHandler.java`, `doUpdateTransaction()` lines 76–85:

```java
if (transactionBody.hasChunkInfo()) {
    ConsensusMessageChunkInfo chunkInfo = transactionBody.getChunkInfo();
    topicMessage.setChunkNum(chunkInfo.getNumber());   // line 78
    topicMessage.setChunkTotal(chunkInfo.getTotal());  // line 79
    if (chunkInfo.hasInitialTransactionID()) {
        topicMessage.setInitialTransactionId(
                chunkInfo.getInitialTransactionID().toByteArray());
    }
}
```

Each `TopicMessage` row is inserted independently with whatever `chunkTotal` the submitter placed in that transaction. There is no lookup of previously stored rows sharing the same `initialTransactionID` to verify that `chunkTotal` is consistent.

**Root cause / failed assumption:**
The code assumes that the Hedera consensus node enforces `chunkTotal` consistency across chunks. It does not. The `INVALID_CHUNK_TRANSACTION_ID` response code comment in the codebase explicitly states: *"The entire initialTransactionID should match the transactionID of the first chunk, but this is not checked or enforced by Hedera except when the chunk number is 1."* The consensus node only validates (a) `chunkNum ∈ [1, chunkTotal]` per-transaction and (b) payer account matches the payer in `initialTransactionID`. It never cross-checks `chunkTotal` between transactions.

**Database schema:**
The `topic_message` table (migration `V2.0.0__create_tables.sql`, lines 637–650) has `chunk_num integer` and `chunk_total integer` as plain nullable columns with no `CHECK` constraint or uniqueness constraint enforcing `chunk_total` consistency across rows sharing the same `initial_transaction_id`.

**Exploit flow:**
1. Attacker submits TX_A: `chunkNum=1, chunkTotal=1, initialTransactionID=TX_A` → consensus node accepts (chunk 1 of 1, payer matches itself). Mirror node stores row: `chunk_num=1, chunk_total=1, initial_transaction_id=TX_A`.
2. Attacker submits TX_B: `chunkNum=2, chunkTotal=2, initialTransactionID=TX_A` → consensus node accepts (`chunkNum=2 ≤ chunkTotal=2` passes; payer of TX_B matches payer in TX_A; `initialTransactionID` equality to first chunk's TX is only checked when `chunkNum=1`, which is not the case here). Mirror node stores row: `chunk_num=2, chunk_total=2, initial_transaction_id=TX_A`.

**Why existing checks fail:**
The only guard in `doUpdateTransaction()` is `!recordItem.isSuccessful()` (line 53), which passes for both transactions since the consensus node accepted them. No cross-chunk state is maintained anywhere in the handler, the entity listener, or the database schema.

### Impact Explanation
Consumers (REST API clients, gRPC subscribers via `ConsensusController.toResponse()`) receive two rows with the same `initialTransactionID` but contradictory `chunk_total` fields: one row says the message is complete (total=1), another says it is chunk 2 of 2 with chunk 1 missing. Any application logic that groups chunks by `initialTransactionID` and waits until it has collected `chunk_total` chunks will either: (a) consider the message permanently incomplete and wait forever for a missing chunk 1 of the second sequence, or (b) incorrectly treat the single-chunk row as a complete message while also emitting a dangling orphan chunk. The corrupted state is permanent — there is no reconciliation mechanism in the mirror node. Severity is low-to-medium: no funds are at risk, but topic message consumers relying on fragmented message reassembly are reliably griefed.

### Likelihood Explanation
Any account holder on the Hedera network can submit `CONSENSUSSUBMITMESSAGE` transactions to any topic without a `submitKey`. No privileged access is required. The attack costs only standard transaction fees (fractions of a cent per transaction). It is trivially repeatable and scriptable, and can be applied to any public topic to continuously inject incoherent chunk sequences.

### Recommendation
In `doUpdateTransaction()`, after storing a chunk with `chunkNum > 1`, query the `topic_message` repository for existing rows sharing the same `initialTransactionId` and verify that their stored `chunkTotal` matches the current transaction's `chunkTotal`. If a mismatch is detected, log a recoverable error (consistent with the existing `Utility.handleRecoverableError` pattern) and either skip persistence or persist with a flag. Alternatively, add a database-level `CHECK` constraint or trigger that rejects inserts where `chunk_total` differs from existing rows with the same `initial_transaction_id`. At minimum, document that `chunk_total` is per-chunk and unverified, so downstream consumers know not to trust it as authoritative.

### Proof of Concept
```
// Step 1: Submit TX_A — chunk 1 of 1 (complete single-chunk message)
ConsensusMessageChunkInfo chunkInfo1 = ConsensusMessageChunkInfo.newBuilder()
    .setNumber(1)
    .setTotal(1)
    .setInitialTransactionID(TX_A_ID)  // TX_A's own TransactionID
    .build();
// Submit to any public topic → SUCCESS

// Step 2: Submit TX_B — chunk 2 of 2, referencing TX_A as initialTransactionID
ConsensusMessageChunkInfo chunkInfo2 = ConsensusMessageChunkInfo.newBuilder()
    .setNumber(2)
    .setTotal(2)
    .setInitialTransactionID(TX_A_ID)  // same initialTransactionID as step 1
    .build();
// Submit from same payer account → SUCCESS
// (consensus node only checks payer match and chunkNum ≤ chunkTotal; both pass)

// Result in mirror node topic_message table:
// Row 1: chunk_num=1, chunk_total=1, initial_transaction_id=TX_A  ← "complete"
// Row 2: chunk_num=2, chunk_total=2, initial_transaction_id=TX_A  ← "chunk 2 of 2, chunk 1 missing"
// Consumers see contradictory chunk_total for the same initialTransactionID.
```
