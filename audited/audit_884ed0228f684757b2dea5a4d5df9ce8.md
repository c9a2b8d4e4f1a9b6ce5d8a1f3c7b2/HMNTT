### Title
Deleted Entity Returned via Long-Zero EVM Address Lookup in GraphQL `getByEvmAddressAndType()`

### Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, when the input EVM address is a long-zero address (bytes 0–11 all zero), the code routes to `entityRepository.findById()` — the standard Spring Data `CrudRepository` method — which carries no `deleted` filter. The sibling path for non-long-zero addresses uses a custom `findByEvmAddress()` query that explicitly includes `deleted is not true`. Any unprivileged caller can craft a long-zero address encoding a deleted account's numeric ID and receive the deleted entity back from the GraphQL API.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0-11 == 0 → long-zero
        return entityRepository.findById(buffer.getLong())         // ← NO deleted filter
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)      // ← has "deleted is not true"
                           .filter(e -> e.getType() == type);
}
```

**Root cause:**
`entityRepository.findById()` is the inherited `CrudRepository<Entity, Long>.findById()` — a plain `SELECT * FROM entity WHERE id = ?`. The GraphQL `EntityRepository` defines no override for `findById`, so no `deleted` predicate is applied.

`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 13–17:

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
// findById — NOT overridden; no deleted filter
```

**Exploit flow:**
1. Attacker identifies a deleted account with numeric ID `N` (publicly visible on-chain or via the REST API).
2. Attacker constructs a 20-byte long-zero EVM address: `0x0000000000000000000000000000000000000000` with the last 8 bytes set to the big-endian encoding of `N`.
3. Attacker submits a GraphQL query (no authentication required on a public mirror node) using this address.
4. `buffer.getInt() == 0 && buffer.getLong() == 0` evaluates to `true` → `findById(N)` is called.
5. The deleted entity row is returned from the database and passed back to the caller.

**Why existing checks fail:**
The only post-retrieval filter is `.filter(e -> e.getType() == type)` — a type check, not a deletion check. The `deleted` field on the `Entity` object is never inspected in this code path.

### Impact Explanation
The GraphQL API returns a deleted account/contract as if it were live. Downstream consumers (wallets, dApps, smart contract tooling) that rely on the GraphQL API to verify account existence before initiating transfers or contract interactions will receive a false-positive "account exists" signal. This can lead to funds being sent to or operations being initiated against an address that no longer corresponds to a live entity on the network, resulting in irrecoverable loss of funds for the end user. The inconsistency also breaks the invariant that all three lookup paths (`findByAlias`, `findByEvmAddress`, `findById`) should agree on entity liveness.

### Likelihood Explanation
Exploitation requires zero privileges — the GraphQL endpoint is publicly accessible. The attacker needs only the numeric ID of any deleted account, which is trivially obtainable from public ledger data or the mirror node's own REST API. The crafted long-zero address is deterministic and requires no brute force. The attack is fully repeatable for any deleted entity in the database.

### Recommendation
Override `findById` in the GraphQL `EntityRepository` with a query that includes the `deleted is not true` predicate, consistent with `findByAlias` and `findByEvmAddress`:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findById(Long id);
```

Alternatively, add a `.filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` check in `EntityServiceImpl.getByEvmAddressAndType()` after the `findById` call, and apply the same fix to `getByIdAndType()` (line 25) which has the identical omission.

### Proof of Concept
1. Identify a deleted account with ID `12345` (e.g., via `GET /api/v1/accounts/0.0.12345` returning `deleted: true`).
2. Encode the long-zero EVM address: `0x000000000000000000000000000000000000000000003039` (last 8 bytes = `0x0000000000003039` = 12345 in big-endian).
3. Submit the following GraphQL query to the mirror node:
   ```graphql
   query {
     account(input: { evmAddress: { address: "0x000000000000000000000000000000000000000000003039" } }) {
       id
       deleted
     }
   }
   ```
4. Observe that the response returns the deleted account's data (including `deleted: true` in the payload), confirming the entity was fetched despite being deleted — whereas the same lookup via a non-long-zero EVM address for a deleted entity would return `null`.
