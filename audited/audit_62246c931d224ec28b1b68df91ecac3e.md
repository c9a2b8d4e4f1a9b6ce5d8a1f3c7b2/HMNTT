### Title
Unauthenticated Information Disclosure via `/api/v1/subscriber` Exposes Real-Time Monitor Health and Error Telemetry

### Summary
The `SubscriberController` exposes live operational state — including `status` (RUNNING/IDLE/COMPLETED), `rate`, `count`, `elapsed`, and internal error class names — on `GET /api/v1/subscriber` with zero authentication. No Spring Security dependency or configuration exists anywhere in the monitor module. Any unprivileged external user who can reach the port can continuously poll this endpoint to track degradation windows and error patterns in real time.

### Finding Description
**Exact code path:**

`SubscriberController` (`monitor/src/main/java/org/hiero/mirror/monitor/subscribe/controller/SubscriberController.java`, lines 27–43) is a plain `@RestController` with no `@PreAuthorize`, no `SecurityFilterChain`, and no Spring Security on the classpath (confirmed: `grep` for `spring-boot-starter-security` returns zero matches in the entire `monitor/` tree). The only filter present is `LoggingFilter` (lines 33–37), which only logs requests — it performs no authentication or authorization.

The `subscriptions()` handler at line 34–43 calls `mirrorSubscriber.getSubscriptions()`, which returns live `Scenario` objects. Each serialized response includes:
- `status` — computed in `AbstractScenario.getStatus()` (lines 72–80): `COMPLETED` if stopped, `IDLE` if rate ≤ 0, `RUNNING` otherwise
- `rate` — rolling 20-second transaction rate
- `errors` — a `Map<String, Integer>` of internal Java exception class simple names (e.g., `TimeoutException`, `StatusRuntimeException`) to occurrence counts
- `count`, `elapsed`, `name`, `id`, `protocol`

**Root cause:** The monitor module has no authentication layer whatsoever. The failed assumption is that the monitor REST API is only reachable by trusted internal operators, but the code enforces nothing — deployment misconfiguration or network exposure immediately makes this public.

**Why existing checks fail:** `LoggingFilter` only logs; it does not gate access. No `WebSecurityConfigurerAdapter`, no `SecurityWebFilterChain`, no `@Secured` annotation exists anywhere in the module. The `SubscriberControllerTest` confirms this: tests use a bare `WebTestClient.bindToController(...)` with no auth setup and all requests succeed.

### Impact Explanation
An attacker who can reach the monitor port gains:
1. **Degradation oracle**: By polling `GET /api/v1/subscriber` and watching `status` transition from `RUNNING` → `IDLE`, the attacker learns precisely when the mirror node stops processing transactions — a window during which anomalous activity is less likely to be detected or alerted on.
2. **Error type leakage**: The `errors` map exposes internal Java exception class names, revealing whether failures are network timeouts, gRPC status errors, or other internal conditions — useful for fingerprinting infrastructure state.
3. **Throughput fingerprinting**: The `rate` and `count` fields reveal transaction processing capacity and current load.
4. **No rate limiting**: There is no throttle on the endpoint, so continuous high-frequency polling imposes no cost on the attacker and can also contribute to resource exhaustion on the monitor itself.

Severity: **Medium** — direct protocol/consensus integrity is not affected, but operational security posture and attack-timing capability are materially degraded.

### Likelihood Explanation
Precondition: the attacker must be able to reach the monitor's HTTP port (default 8082). The Helm chart ingress only exposes `/actuator/health/cluster`, but misconfigured deployments, internal network access, or cloud environments with flat networking make this realistic. No credentials, tokens, or special knowledge are required — a single `curl` command suffices. The attack is fully repeatable and automatable.

### Recommendation
1. Add Spring Security (`spring-boot-starter-security`) to the monitor module and configure a `SecurityWebFilterChain` that requires authentication (at minimum HTTP Basic or a pre-shared token) for all `/api/v1/**` paths.
2. Apply rate limiting (e.g., via Spring Cloud Gateway or a servlet filter) to the monitor REST endpoints.
3. Ensure the `errors` map does not serialize raw Java exception class names; map them to generic categories before serialization.
4. At the deployment level, ensure the monitor port is not reachable from untrusted networks regardless of application-level controls.

### Proof of Concept
```bash
# Step 1: Discover the monitor is running and unauthenticated
curl -s http://<monitor-host>:8082/api/v1/subscriber | jq .

# Step 2: Poll continuously to detect degradation
while true; do
  STATUS=$(curl -s http://<monitor-host>:8082/api/v1/subscriber | jq -r '.[].status')
  ERRORS=$(curl -s http://<monitor-host>:8082/api/v1/subscriber | jq -r '.[].errors')
  echo "$(date): status=$STATUS errors=$ERRORS"
  sleep 5
done

# Step 3: When status transitions to IDLE or errors spike,
# the attacker knows the mirror node monitoring is degraded
# and can time subsequent actions (e.g., submitting anomalous
# transactions) during the blind window.
```

```java
@RequestMapping("/api/v1/subscriber")
@RequiredArgsConstructor
@RestController
class SubscriberController {

    private final MirrorSubscriber mirrorSubscriber;

    @GetMapping
    public <T extends ScenarioProperties> Flux<Scenario<T, Object>> subscriptions(
            @RequestParam("protocol") Optional<ScenarioProtocol> protocol,
            @RequestParam("status") Optional<List<ScenarioStatus>> status) {
        return mirrorSubscriber
                .<Scenario<T, Object>>getSubscriptions()
                .filter(s -> !protocol.isPresent() || protocol.get() == s.getProtocol())
                .filter(s -> !status.isPresent() || status.get().contains(s.getStatus()))
                .switchIfEmpty(Mono.error(new NoSuchElementException()));
    }
```
