### Title
Unbounded `DISTINCT ON` Scan in Historical Hook Storage Query Enables DB Resource Exhaustion

### Summary
An unprivileged caller can trigger the historical code path in `getHookStorageChange()` by supplying any timestamp parameter, then provide a maximally wide key range (`keyLowerBound=0x00…00`, `keyUpperBound=0xFF…FF`) and a maximally wide timestamp range. The native SQL query in `findByKeyBetweenAndTimestampBetween()` uses `DISTINCT ON (key)` with no row-count guard inside the SQL itself; PostgreSQL must scan, sort, and deduplicate the entire matching result set before the `LIMIT` appended by Spring Data JPA is applied, consuming disproportionate DB CPU and memory regardless of the small final page size.

### Finding Description

**Trigger path — `isHistorical()` gate:**
`getHookStorage()` delegates to `getHookStorageChange()` whenever `request.isHistorical()` is true, which is simply `!timestamp.isEmpty()`. [1](#0-0) 

Any caller that supplies a single timestamp parameter (e.g., `timestamp=gte:0`) satisfies this condition with no privilege check.

**Unbounded key range — no width validation:**
When no explicit keys are provided (`requestHasKeys == false`), the service passes `request.getKeyLowerBound()` and `request.getKeyUpperBound()` directly to the repository with no validation of the range width. [2](#0-1) 

**Unbounded timestamp range — defaults to full history:**
`Bound.getAdjustedLowerRangeValue()` returns `0` when no lower bound is set, and `adjustUpperBound()` returns `Long.MAX_VALUE` when no upper bound is set. [3](#0-2) [4](#0-3) 

A single `timestamp=gte:0` therefore sets `timestampLowerBound=0`, `timestampUpperBound=Long.MAX_VALUE`.

**The expensive SQL — `DISTINCT ON` before `LIMIT`:**
The native query performs `DISTINCT ON (key)` over the full filtered set. Spring Data JPA appends `ORDER BY key ASC, consensus_timestamp DESC LIMIT :limit` at the framework level, but PostgreSQL must first scan every matching row, sort by `(key, consensus_timestamp DESC)`, and deduplicate before the `LIMIT` is honoured. There is no `LIMIT` inside the SQL body itself. [5](#0-4) 

**No effective row cap inside the query:** The `limit` field in `HookStorageRequest` defaults to 25 and is only applied after deduplication. [6](#0-5) 

### Impact Explanation
A single crafted request forces a full-table scan of `hook_storage_change` for the targeted `owner_id`/`hook_id`, a sort of potentially millions of rows, and an in-memory deduplication pass — all to return at most 25 rows. Repeated at modest concurrency (e.g., 10–20 req/s), this can saturate DB CPU and working memory, degrading or denying service for all other mirror-node consumers. This meets the stated threshold of ≥30% increase in DB resource consumption without brute-force volume.

### Likelihood Explanation
The endpoint is reachable by any external caller with knowledge of a valid `owner_id`/`hook_id` pair (both are enumerable from other public endpoints). No authentication or elevated role is required. The attack is fully repeatable, requires only standard HTTP tooling, and a single request already imposes outsized cost relative to its payload size.

### Recommendation
1. **Add a maximum timestamp range width** — reject or cap requests where `timestampUpperBound - timestampLowerBound` exceeds a configured maximum (e.g., 7 days in nanoseconds).
2. **Add a `LIMIT` inside the native SQL** before deduplication, or rewrite the query to use a lateral/window-function approach that pushes the limit into the inner scan.
3. **Validate key range width** — reject requests where the byte-array distance between `keyLowerBound` and `keyUpperBound` exceeds a configured threshold.
4. **Rate-limit** the historical storage endpoint per caller identity/IP.

### Proof of Concept

```
# Step 1: obtain any valid owner_id and hook_id from public listing endpoints
GET /api/v1/hooks?owner.id=0.0.1234

# Step 2: trigger the historical path with a maximally wide key and timestamp range
GET /api/v1/hooks/0.0.1234/1/storage?timestamp=gte:0&key=gte:0x0000000000000000&key=lte:0xffffffffffffffff&limit=25

# Repeat at 10-20 req/s from a single host.
# Observe: DB CPU climbs proportionally to the number of rows in hook_storage_change
# for that owner/hook; the caller receives only 25 rows per response.
```

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
