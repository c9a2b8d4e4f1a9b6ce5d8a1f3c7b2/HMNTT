### Title
`BufferUnderflowException` via Under-Length EVM Address in `getByEvmAddressAndType()`

### Summary
`decodeEvmAddress()` in `GraphQlUtils.java` accepts any even-length hex string without enforcing the required 20-byte (40 hex character) minimum. When the resulting byte array is fewer than 4 bytes, the subsequent `ByteBuffer.wrap(evmAddressBytes).getInt()` call in `getByEvmAddressAndType()` throws an uncaught `BufferUnderflowException`, propagating as an unhandled server error that disrupts the GraphQL service.

### Finding Description

**Code path:**

`GraphQlUtils.decodeEvmAddress()` ( [1](#0-0) ) strips the `0x` prefix and calls `Hex.decodeHex()`. It only rejects inputs with an odd number of hex characters (via `DecoderException`). It performs **no minimum-length check** — a 2-hex-char input like `1234` decodes to a 2-byte array without error.

This is confirmed by the existing test suite, which explicitly exercises a 5-byte (10 hex char) input and expects success: [2](#0-1) 

Back in `getByEvmAddressAndType()`: [3](#0-2) 

- Line 36: `ByteBuffer.wrap(evmAddressBytes)` wraps whatever byte array was returned.
- Line 37: `buffer.getInt()` attempts to read **4 bytes**. If the buffer holds fewer than 4 bytes (e.g., 2 bytes from `0x1234`), Java throws `java.nio.BufferUnderflowException`.
- There is **no try-catch** anywhere in this method or its callers for this exception type.

**Root cause:** `decodeEvmAddress` assumes callers will validate length; `getByEvmAddressAndType` assumes `decodeEvmAddress` returns exactly 20 bytes. Neither assumption is enforced.

### Impact Explanation

An unauthenticated attacker can repeatedly submit malformed short EVM addresses to any GraphQL query that accepts an EVM address field. Each request triggers an uncaught `BufferUnderflowException`. Depending on Spring GraphQL's error handling configuration, this either surfaces as a 500 Internal Server Error or a generic GraphQL error — in either case, the legitimate query fails. Repeated at scale, this constitutes a low-cost denial-of-service against the GraphQL endpoint, preventing legitimate users from querying Hashgraph history.

### Likelihood Explanation

No authentication or authorization is required. The attacker only needs to know the GraphQL schema accepts an EVM address string field (publicly documented). The payload is trivial (`0x12` — a single byte). The attack is fully repeatable and stateless. Any automated scanner or script can trigger it continuously.

### Recommendation

1. **In `decodeEvmAddress`**: After decoding, validate that the resulting byte array is exactly 20 bytes. If not, throw `IllegalArgumentException` with a descriptive message.

```java
byte[] decoded = Hex.decodeHex(evmAddress);
if (decoded.length != 20) {
    throw new IllegalArgumentException("EVM address must be exactly 20 bytes");
}
return decoded;
```

2. **In `getByEvmAddressAndType`**: Add a defensive length check before wrapping in `ByteBuffer`:

```java
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException("Invalid EVM address length: " + evmAddressBytes.length);
}
```

3. **At the GraphQL schema level**: Enforce the EVM address format via a custom scalar or `@Pattern` constraint (40 hex chars, optional `0x` prefix) so invalid inputs are rejected before reaching service logic.

### Proof of Concept

**Precondition:** GraphQL service is running; no authentication required.

**Trigger:** Send the following GraphQL query (adjust field name to match the actual schema):

```graphql
{
  account(input: { evmAddress: "0x1234" }) {
    id
  }
}
```

**Step-by-step:**
1. `getByEvmAddressAndType("0x1234", ...)` is called.
2. `decodeEvmAddress("0x1234")` strips `0x`, calls `Hex.decodeHex("1234")` → returns `byte[]{0x12, 0x34}` (2 bytes). No exception thrown.
3. `ByteBuffer.wrap(new byte[]{0x12, 0x34})` creates a buffer with capacity 2.
4. `buffer.getInt()` attempts to read 4 bytes from a 2-byte buffer → throws `java.nio.BufferUnderflowException`.
5. Exception is uncaught → propagates as a server error response.

**Result:** The GraphQL request fails with an unhandled server error. Repeating this request at high frequency degrades or denies service to legitimate users.

```java
    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }
```
