### Title
Unauthenticated Flood of `GET /api/v1/network/stake` Triggers False Positive `DOWN` Health Status in Monitor

### Summary
The `GET /api/v1/network/stake` endpoint has no rate limiting in the `rest-java` module and its backing `NetworkStakeRepository.findLatest()` query is uncached, executing a correlated subquery on every request. The monitor's `SubscriberHealthIndicator` uses the HTTP status code of this endpoint as a hard gate: any 5xx response unconditionally sets the entire mirror node health to `DOWN`, regardless of actual publish/subscribe activity. An unprivileged attacker can flood this endpoint to exhaust the DB connection pool, induce 500 errors, and cause the monitoring system to report the mirror node as DOWN, triggering unnecessary incident response.

### Finding Description

**Code path:**

1. `NetworkController.getNetworkStake()` (no rate limiting, no auth): [1](#0-0) 

2. `NetworkServiceImpl.getLatestNetworkStake()` calls the repository with no caching: [2](#0-1) 

3. `NetworkStakeRepository.findLatest()` executes a correlated subquery on every call — no `@Cacheable`, no result reuse: [3](#0-2) 

4. `RestApiClient.getNetworkStakeStatusCode()` polls this endpoint and returns the raw HTTP status code: [4](#0-3) 

5. `SubscriberHealthIndicator.restNetworkStakeHealth()` maps any 5xx to `Status.DOWN` and short-circuits the entire health check: [5](#0-4) 

6. `health()` uses `restNetworkStakeHealth()` as the primary gate — if it returns DOWN, the result is DOWN unconditionally: [6](#0-5) 

**Root cause:** The `rest-java` module has no rate limiting on network endpoints. The only throttle configuration in the codebase is in the `web3` module and applies exclusively to contract call requests: [7](#0-6) 

This throttle is entirely absent from `rest-java`. Combined with the uncached correlated subquery in `findLatest()`, each inbound request hits the database. Under flood conditions the DB connection pool exhausts, causing 500 responses. The monitor's hard mapping of 5xx → `DOWN` then fires.

**Failed assumption:** The design assumes the endpoint is called only by the internal monitor at a low, controlled rate. There is no enforcement of this assumption.

### Impact Explanation

Any 5xx from `/api/v1/network/stake` causes `SubscriberHealthIndicator.health()` to return `Status.DOWN` for the entire mirror node, as confirmed by the test matrix: [8](#0-7) 

Even with healthy publish rate (1.0) and subscribe rate (1.0), a 500 from the stake endpoint produces `DOWN`. This causes the monitoring system to report the mirror node as completely unavailable, triggering incident response workflows for a condition that does not reflect actual node health. The `CLUSTER_UP` gauge metric is also set to 0: [9](#0-8) 

### Likelihood Explanation

The endpoint requires no authentication, no API key, and accepts no parameters. Any attacker with network access to the mirror node's REST Java API can execute this attack with a simple HTTP flood tool (e.g., `ab`, `wrk`, `hey`). The correlated subquery `SELECT MAX(consensus_timestamp) FROM network_stake` is executed on every request with no caching, making DB saturation achievable at modest request rates. The attack is repeatable and requires no special knowledge beyond the public API documentation.

### Recommendation

1. **Add response caching** on `NetworkStakeRepository.findLatest()` or `NetworkServiceImpl.getLatestNetworkStake()` using `@Cacheable` with a short TTL (e.g., 30 seconds), consistent with how `NodeStakeRepository.findAllStakeByConsensusTimestamp` is cached in the grpc module: [10](#0-9) 

2. **Add rate limiting** to the `rest-java` module's network endpoints, analogous to the `web3` throttle configuration.

3. **Decouple the health signal** in `SubscriberHealthIndicator`: instead of treating a single 5xx as an immediate hard `DOWN`, use a sliding window or consecutive-failure threshold before escalating to `DOWN` status.

### Proof of Concept

```bash
# Flood the endpoint with concurrent requests (no auth required)
wrk -t 10 -c 200 -d 30s http://<mirror-node-rest-java-host>/api/v1/network/stake

# Simultaneously observe the monitor health endpoint
watch -n 1 curl -s http://<monitor-host>/actuator/health | jq '.components.subscriber.status'
# Expected: transitions from "UP" to "DOWN" as DB connection pool exhausts
# and /api/v1/network/stake begins returning 500 errors
```

Preconditions: network access to the public REST Java API. No credentials required. The monitor health endpoint will report `DOWN` and the `hiero.mirror.monitor.health{type=cluster}` gauge will drop to 0, triggering any configured alerting rules.

    private Mono<Health> restNetworkStakeHealth() {
        return restApiClient
                .getNetworkStakeStatusCode()
                .flatMap(statusCode -> {
                    if (statusCode.is2xxSuccessful()) {
                        return UP;
                    }

                    var status = statusCode.is5xxServerError() ? Status.DOWN : Status.UNKNOWN;
                    var statusMessage =
                            String.format("Network stake status is %s with status code %s", status, statusCode.value());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                })

@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
