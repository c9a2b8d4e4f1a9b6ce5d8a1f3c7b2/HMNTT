### Title
Decoupled Heartbeat Goroutine Allows Silent Transfer Stall While Liveness Probe Passes

### Summary
The heartbeat goroutine in `pinger/main.go` writes to `/tmp/alive` independently of the transfer ticker loop. Because `Execute(client)` and `GetReceipt(client)` in `transfer.go` accept no `context.Context`, a network-level attacker who causes Hashgraph node connections to hang will permanently block the main goroutine's transfer loop while the heartbeat goroutine continues refreshing `/tmp/alive`, causing the Kubernetes liveness probe to pass indefinitely while all Hashgraph transaction submission silently stops.

### Finding Description

**Heartbeat goroutine — fully decoupled from transfer activity:** [1](#0-0) 

The goroutine above runs on its own 15-second ticker and only exits on `ctx.Done()` (SIGTERM/SIGINT). It has zero coupling to whether `submitWithRetry()` is executing or blocked.

**Transfer loop — main goroutine, sequential, no independent watchdog:** [2](#0-1) 

The main goroutine processes one tick at a time. If `submitWithRetry()` never returns, no further ticks are ever processed.

**Root cause — SDK calls accept no context, cannot be cancelled:** [3](#0-2) 

`cryptoTransfer.Execute(client)` and `resp.GetReceipt(client)` take only a `*hiero.Client`. There is no `context.Context` plumbed into these calls, so neither a deadline nor cancellation signal can interrupt them. If the underlying gRPC/TCP connection to a Hashgraph node is silently hung (peer accepts the connection but never sends a response and never sends TCP RST), these calls block forever.

**Liveness probe — checks only file mtime, not transfer activity:** [4](#0-3) 

The healthcheck binary is a separate process with no IPC to the pinger. It only checks whether `/tmp/alive` was modified within the last 2 minutes. Because the heartbeat goroutine continues writing that file every 15 seconds regardless of transfer state, the probe always exits 0.

**Exploit flow:**
1. Attacker causes all reachable Hashgraph consensus nodes to silently hang TCP connections (firewall drop-without-RST, malicious node, BGP hijack of node IP ranges, or ARP spoofing on the same L2 segment as the pinger pod).
2. `Execute(client)` in `submitWithRetry()` blocks indefinitely on the first tick.
3. The main goroutine is permanently stuck inside `submitWithRetry()`. No further ticks are ever dequeued.
4. The heartbeat goroutine continues writing `/tmp/alive` every 15 seconds.
5. `healthcheck live` reads `/tmp/alive`, sees a fresh mtime, exits 0.
6. Kubernetes never restarts the pod. The pinger emits no transactions to the Hashgraph network for the entire duration of the attack.

### Impact Explanation
The pinger's purpose is to continuously submit transfer transactions to the Hashgraph network so that the mirror node pipeline can be verified end-to-end. A silent stall means:
- The monitoring gap is invisible to Kubernetes (no pod restart, no alert from the liveness probe).
- An attacker suppressing the pinger can hide anomalies in Hashgraph history (missing transactions, forked state) that the pinger would otherwise detect, directly enabling tampering with the observable transaction record.
- The condition can be held for an arbitrary duration — as long as the attacker maintains the network hang, the pinger stays silently broken.

Severity: **High** — complete, undetected loss of the monitoring function that is the sole purpose of this component.

### Likelihood Explanation
No Kubernetes privileges are required. The attacker needs only network-level access between the pinger pod and the Hashgraph consensus node IPs:
- Cloud-hosted deployments are reachable via BGP prefix hijack or compromised upstream router.
- On-premise or co-located deployments are vulnerable to ARP spoofing or VLAN hopping.
- A single malicious or compromised Hashgraph consensus node that the SDK selects can hang the connection without any routing-level access.
- The condition is fully repeatable and stable: once the hang is established, no further attacker action is needed.

### Recommendation
1. **Add a context with deadline to SDK calls.** Wrap each `Execute` / `GetReceipt` call with a per-attempt `context.WithTimeout` derived from the parent context, e.g. `ctx, cancel := context.WithTimeout(ctx, 30*time.Second)`. Pass this context to the SDK if the version supports it, or run the call in a goroutine and select on the timer and a done channel.
2. **Couple the heartbeat to observed transfer activity.** Only write `/tmp/alive` after a successful (or at least attempted-and-returned) call to `submitWithRetry()`, not on a free-running timer. Alternatively, maintain a `lastTransferAttempt` atomic timestamp updated by the transfer loop and have the heartbeat goroutine refuse to write `/tmp/alive` if that timestamp is stale beyond a threshold.
3. **Add a separate liveness signal for transfer loop progress.** Write a second file (e.g. `/tmp/transfer_alive`) from inside the transfer loop's `select` case, and check both files in the healthcheck.

### Proof of Concept
```
# 1. On a host that can intercept traffic between the pinger pod and
#    Hashgraph node IPs (e.g., same L2 or upstream router):
iptables -I FORWARD -d <hashgraph-node-ip> -p tcp --dport 50211 \
         -j DROP   # drop without RST — connection hangs open

# 2. Wait for the pinger's current tick to call Execute(client).
#    The call blocks. No further ticks are processed.

# 3. From outside the cluster, poll the liveness endpoint:
kubectl exec <pinger-pod> -- /healthcheck live
# exits 0 indefinitely — pod is never restarted

# 4. Confirm no transactions appear on the mirror node REST API:
curl https://<mirror-rest>/api/v1/transactions?account.id=<operator-id>&limit=1
# timestamp of most recent tx stops advancing

# 5. Remove the iptables rule to restore normal operation —
#    the pinger resumes on the next tick, leaving a silent gap
#    in the transaction history of arbitrary length.
```
