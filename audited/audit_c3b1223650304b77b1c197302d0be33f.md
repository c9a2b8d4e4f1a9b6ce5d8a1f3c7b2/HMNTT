### Title
Alias-Based Query Amplification Bypasses Complexity Limit, Enabling Multiple DB Lookups and Inconsistent Account State in a Single Response

### Summary
The `graphQlCustomizer()` bean configures `MaxQueryComplexityInstrumentation` with a flat default cost of 1 per field and no custom `FieldComplexityCalculator`. Because GraphQL aliases are each counted as independent fields, an unauthenticated attacker can alias the `account` top-level query up to ~100 times within the 200-complexity budget, causing the server to execute a separate database lookup per alias. Since the mirror node continuously ingests new Hashgraph state, different aliases for the same account resolved across those sequential DB calls can return different `timestamp` (validity range) values in a single response, producing inconsistent historical snapshots.

### Finding Description
**Exact code path:**

`GraphQlConfiguration.java`, `graphQlCustomizer()`, lines 42–49:
```java
var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
// No FieldComplexityCalculator supplied → default cost = 1 per field
```
`AccountController.java`, `account()` resolver, lines 33–58:
```java
@QueryMapping
Optional<Account> account(@Argument @Valid AccountInput input) {
    ...
    return entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
            .map(accountMapper::map);  // DB lookup on every invocation
}
```
`EntityServiceImpl.java`, line 25:
```java
return entityRepository.findById(entityId.getId())...  // JPA → SQL per call
```

**Root cause:** `MaxQueryComplexityInstrumentation(200)` is constructed without a custom `FieldComplexityCalculator`. The graphql-java default assigns a uniform cost of **1 per field selection**, and each alias is treated as a distinct field selection. There is no deduplication of aliased fields before execution. The `@QueryMapping`-annotated `account()` method is invoked once per alias by the Spring GraphQL dispatcher, each time issuing a fresh `SELECT` against the database. The `CachedPreparsedDocumentProvider` caches only the parsed/validated document, not execution results.

**Exploit flow:**
1. Attacker crafts a query with 100 aliases of `account`, each selecting one scalar field (e.g., `balance`): total complexity = 100 × 2 = 200 — exactly at the limit.
2. The complexity check passes.
3. The depth check (limit 10) is irrelevant — this is horizontal, not vertical expansion.
4. Parser limits (`maxTokens: 1000`, `maxCharacters: 10000`) are not exceeded by 100 short aliases.
5. The GraphQL executor calls `AccountController.account()` 100 times, each issuing `entityRepository.findById()`.
6. Because the mirror node ingests new Hashgraph transactions continuously, the account's `timestamp` (a `TimestampRange` reflecting the validity period of the current state) can differ between the first and last DB call within the same request, yielding inconsistent snapshots in a single JSON response.

**Why checks fail:**
- Complexity limit of 200 with default cost 1/field does not reflect actual DB cost; a resolver that hits the database costs the same as a trivial scalar.
- Depth limit of 10 does not constrain horizontal alias expansion.
- No per-query or per-IP rate limiting exists in the application layer.
- No result memoization/deduplication for repeated aliased calls to the same resolver with identical arguments.

### Impact Explanation
An unauthenticated attacker can amplify a single HTTP request into up to 100 synchronous database queries. Under sustained attack this exhausts the HikariCP connection pool (monitored at 75% utilization in the alerting config), degrading or denying service to legitimate users. Additionally, a single response can contain up to 100 different `timestamp` validity ranges for the same account, misrepresenting the Hashgraph's historical record — a direct integrity violation for any consumer relying on the GraphQL API to reconstruct account history.

### Likelihood Explanation
No credentials, API keys, or special network access are required. The GraphQL endpoint (`/graphql/alpha`) is publicly documented and accessible via a simple HTTP POST. The exploit query is trivially constructable with any HTTP client. The attack is fully repeatable and can be automated. The only practical constraint is the 200-complexity ceiling, which still permits a 100× DB amplification factor per request.

### Recommendation
1. **Supply a custom `FieldComplexityCalculator`** to `MaxQueryComplexityInstrumentation` that assigns a higher cost (e.g., 10–20) to resolver-backed fields (`account`) versus trivial scalars, so that 10–20 aliased `account` calls exhaust the budget rather than 100.
2. **Deduplicate aliased fields** at the execution layer, or add a pre-execution validation rule that rejects queries containing more than N aliases of the same root field.
3. **Add result memoization** in `EntityServiceImpl` (e.g., a request-scoped cache keyed on `(entityId, type)`) so repeated calls within the same execution return the same object without additional DB round-trips.
4. **Implement per-IP or per-connection rate limiting** at the load balancer or application layer to bound the total number of DB queries attributable to a single client per time window.

```java
    Optional<Account> account(@Argument @Valid AccountInput input) {
        final var alias = input.getAlias();
        final var evmAddress = input.getEvmAddress();
        final var entityId = input.getEntityId();
        final var id = input.getId();

        validateOneOf(alias, entityId, evmAddress, id);

        if (entityId != null) {
            return entityService
                    .getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        if (alias != null) {
            return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
        }

        if (evmAddress != null) {
            return entityService
                    .getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        throw new IllegalStateException("Not implemented");
    }
```
