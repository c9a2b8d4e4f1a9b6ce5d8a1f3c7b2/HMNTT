### Title
Cache Pollution via Unbounded Slot Key Injection Causes DB Query Amplification and LRU Eviction of Legitimate Keys

### Summary
Any unprivileged caller can inject up to 1500 arbitrary, non-existent slot keys into the per-contract `slotsPerContract` Caffeine cache for a target contract by issuing 1500 unauthenticated `findStorage()` calls. Once the cache is saturated, every subsequent `findStorage()` call — by any user — triggers a batch DB query (`findStorageBatch`) for all 1500 cached slots, amplifying DB load by up to 1500×. Concurrently, the attacker's continuous injection of new unique keys causes LRU eviction of legitimate slot keys; when a legitimate key is evicted between `putIfAbsent` (line 90) and the cache snapshot (line 91), `isKeyEvictedFromCache` remains `true` and the code falls back to an individual DB query at line 119.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageBatch()` — `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 85–122.

**Root cause — no validation of injected slot keys:**

At line 90, any caller-supplied `key` is unconditionally inserted into the per-contract slot cache:

```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);   // line 90
```

There is no check that `key` corresponds to a slot that actually exists in the DB, nor any per-caller rate limit on how many distinct keys can be inserted. The cache is configured with `maximumSize=1500` (`CacheProperties.java` line 39):

```
slotsPerContract = "expireAfterAccess=5m,maximumSize=1500"
```

At line 91, the code immediately snapshots **all** keys currently in the cache:

```java
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();  // line 91
```

At line 103, a single batch DB query is issued for **every** key in that snapshot:

```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

**Exploit flow:**

1. **Phase 1 — Cache saturation:** Attacker sends 1500 `eth_call` / JSON-RPC requests targeting a specific `contractId`, each with a distinct, non-existent 32-byte slot key. Each call reaches `findStorageBatch`, inserts the junk key via `putIfAbsent`, and the per-contract cache fills to its 1500-entry maximum. The `contractState` cache (2s TTL) never caches these non-existent slots, so the `contractStateCache.get()` check at line 63 never short-circuits future calls.

2. **Phase 2 — Sustained amplification:** Attacker continues sending calls with new unique keys. Each call:
   - Inserts a new junk key (evicting the LRU entry — potentially a legitimate key)
   - Snapshots all 1500 cached keys
   - Triggers `findStorageBatch` with a 1500-element `IN` clause against the DB
   - Result: **1 API call → 1500-slot DB query** (1500× amplification)

3. **Phase 3 — Legitimate key eviction / fallback:** When a legitimate user's slot key is evicted from the per-contract cache by the attacker's continuous injection, and the legitimate user concurrently calls `findStorage()` for that key:
   - `putIfAbsent(legitimateKey)` re-inserts the key (line 90)
   - A concurrent attacker thread immediately evicts it before the snapshot at line 91
   - The loop at lines 96–101 does not find `wrappedKey` in `cachedSlotKeys`, so `isKeyEvictedFromCache` remains `true`
   - Line 118–119 falls back to the individual `findStorage` DB query:
     ```java
     if (isKeyEvictedFromCache) {
         return contractStateRepository.findStorage(contractId.getId(), key);
     }
     ```
   - The legitimate user still gets a correct answer, but via an extra individual query on top of the already-inflated batch query.

**Why existing checks are insufficient:**

- `putIfAbsent` (line 90) prevents duplicate keys per caller but does not prevent a single caller from inserting 1500 distinct junk keys.
- The `isKeyEvictedFromCache` fallback (lines 118–119) preserves correctness but does not prevent the amplification or the eviction itself.
- The global `requestsPerSecond=500` throttle (docs line 727) limits total RPS but does not bound the number of distinct slot keys a single caller can inject. At 500 RPS, the cache can be saturated in 3 seconds and maintained indefinitely with ~1 RPS (5-minute `expireAfterAccess` TTL means each injected key stays for 5 minutes without re-access).
- Gas throttling (`gasPerSecond`) applies to EVM execution, not to the cache-layer `findStorage()` path.

### Impact Explanation

**DB query amplification (primary):** Every `findStorage()` call for the target contract — from any user — triggers a `WHERE slot IN (...)` query with up to 1500 parameters. At the default 500 RPS limit, this translates to up to 750,000 slot lookups per second against the DB for a single targeted contract, compared to 500 without the attack. This can saturate DB connection pools and degrade or deny service for all contracts sharing the same DB.

**Legitimate key eviction (secondary):** In concurrent scenarios, legitimate slot keys are evicted from the per-contract cache, forcing an additional individual DB query per evicted key. While correctness is preserved, latency increases and DB load compounds.

**Severity rationale:** Medium — no funds are at risk and no on-chain state is corrupted, but the attack can cause measurable service degradation (increased DB latency, potential query timeouts given the 3000ms `statementTimeout`) for all users of the mirror node web3 service.

### Likelihood Explanation

The attack requires no credentials, no tokens, and no special network position. Any user with access to the public JSON-RPC endpoint can issue `eth_call` requests with arbitrary `data` fields that cause arbitrary slot keys to be read. The attacker needs only ~1500 initial requests to saturate the cache and then ~1 request every 5 minutes per junk key to keep them alive (or a continuous trickle of new unique keys to maintain eviction pressure). This is trivially automatable with a simple script.

### Recommendation

1. **Validate slot keys before caching:** Before calling `putIfAbsent`, check whether the key exists in the DB (or in `contractStateCache`). Only insert keys that have been confirmed to exist, eliminating the ability to pollute the cache with non-existent keys.

2. **Bound per-caller slot key injection:** Introduce a per-IP or per-request limit on how many new slot keys can be added to the per-contract cache within a time window.

3. **Cap the batch query size:** In `findStorageBatch`, limit `cachedSlots` to a configurable maximum (e.g., 100) before issuing the `IN` query, preventing unbounded DB query amplification regardless of cache state.

4. **Separate attacker-controlled input from the batch key set:** Only include in `cachedSlots` keys whose values are already confirmed absent from `contractStateCache` due to a prior legitimate hit, not keys inserted by the current (potentially adversarial) request.

### Proof of Concept

```python
import requests, random

TARGET_RPC = "http://mirror-node-web3:8545"
TARGET_CONTRACT = "0x<target_contract_address>"

def random_slot():
    return "0x" + random.randbytes(32).hex()

# Phase 1: Saturate the slotsPerContract cache with 1500 junk keys
for i in range(1500):
    requests.post(TARGET_RPC, json={
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": TARGET_CONTRACT, "data": random_slot()}, "latest"],
        "id": i
    })

# Phase 2: Continuously inject new unique keys to maintain eviction pressure
# Each call triggers a 1500-slot batch DB query
import time
while True:
    requests.post(TARGET_RPC, json={
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": TARGET_CONTRACT, "data": random_slot()}, "latest"],
        "id": 9999
    })
    # Every legitimate user's findStorage() for this contract now
    # triggers a WHERE slot IN (<1500 junk slots>) DB query
    time.sleep(0.01)  # 100 RPS sustained — well within the 500 RPS global limit
```

**Observable effect:** DB slow-query logs will show repeated `SELECT slot, value FROM contract_state WHERE contract_id = X AND slot IN (...)` queries with 1500 parameters, originating from every `findStorage()` call for the target contract, including those from legitimate users.     private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
