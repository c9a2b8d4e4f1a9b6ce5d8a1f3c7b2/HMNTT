### Title
Deleted Account Token Allowances Exposed via Numeric ID Path — Missing `deleted` Flag Check in `getEncodedId()`

### Summary
When `getAccountTokenAllowances()` resolves a path parameter that is a numeric entity ID (e.g., `0.0.18` or `18`), `EntityService.getEncodedId()` performs no database lookup and no `deleted` flag check — it returns a pure arithmetic encoding. The downstream `TokenAllowanceService` then queries the `token_allowance` table using only `owner = $1 AND amount > 0`, with no guard against deleted accounts. An unprivileged attacker can supply a deleted account's numeric ID and receive its stale token allowances with HTTP 200, while the same deleted account queried by alias or EVM address correctly returns HTTP 404.

### Finding Description

**Exact code path:**

`rest/controllers/tokenAllowanceController.js`, line 69: [1](#0-0) 

calls `EntityService.getEncodedId()` with the raw path parameter.

`rest/service/entityService.js`, lines 118–137: [2](#0-1) 

The branching logic is:
- **Numeric ID** (`EntityId.isValidEntityId()` → true, `evmAddress === null`): calls `entityId.getEncodedId()` — **pure arithmetic, zero DB interaction, no `deleted` check**.
- **EVM address**: calls `getEntityIdFromEvmAddress()` which enforces `WHERE deleted <> true`.
- **Alias**: calls `getAccountIdFromAlias()` which enforces `WHERE coalesce(deleted, false) <> true`.

`rest/entityId.js`, lines 67–82: [3](#0-2) 

`getEncodedId()` is pure bit-arithmetic — it never touches the database.

`rest/service/tokenAllowanceService.js`, lines 12–13 and 30–45: [4](#0-3) [5](#0-4) 

The SQL is `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0` — no join to `entity`, no `deleted` filter.

**Root cause:** The `getEncodedId()` function has three branches with asymmetric behavior. Only alias and EVM address branches hit the database and enforce the `deleted` guard. The numeric branch is a pure local computation, so the deleted state of the account is never consulted.

**Failed assumption:** The code assumes that a numeric ID supplied by an external caller always refers to a live account, or that the downstream query will implicitly exclude deleted accounts. Neither assumption holds.

### Impact Explanation

The `token_allowance` table stores the current (non-historical) allowance state. When an account is deleted on Hedera, its allowances are not explicitly zeroed in the mirror node's `token_allowance` table — they remain with their last recorded `amount`. Because the service filters only `amount > 0`, it returns these stale, network-invalid allowances as if they are active. Any consumer of this API (wallets, DeFi integrations, compliance tools) that trusts the response will receive a false picture of active token permissions for an account that no longer exists on the network.

The test fixture `rest/__tests__/specs/accounts/{id}/not-found.json` explicitly documents the intended behavior: [6](#0-5) 

"account 18 is deleted, when querying by its alias, the rest api should return 404" — confirming the design intent is to gate on `deleted`, but this gate is absent for numeric IDs on the allowances sub-route.

Severity: **Medium** — public data disclosure of stale state, no authentication bypass, no write access.

### Likelihood Explanation

Exploitation requires zero privilege, zero authentication, and zero special tooling. Any attacker who knows (or can enumerate) a deleted account's numeric ID — which is public on-chain data — can trigger this with a single HTTP GET. The numeric ID format (`0.0.N`) is the most common format used by developers and integrations, making accidental or deliberate triggering highly likely. The condition is permanently reproducible as long as the deleted account's rows remain in `token_allowance`.

### Recommendation

In `EntityService.getEncodedId()`, after resolving a numeric entity ID, add a database existence-and-deletion check equivalent to what alias/EVM address paths already perform. The existing `entityExistenceQuery` is close but does not check `deleted`:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;
``` [7](#0-6) 

It should be extended to:

```sql
SELECT type FROM entity
WHERE id = $1
  AND coalesce(deleted, false) <> true
```

Then call this check in the numeric branch of `getEncodedId()` and throw `NotFoundError` if no row is returned, consistent with the alias and EVM address branches. Alternatively, add a `JOIN entity ON entity.id = token_allowance.owner AND coalesce(entity.deleted, false) <> true` in `TokenAllowanceService.getSubQuery()`, though the former approach is more consistent with the existing architecture.

### Proof of Concept

**Preconditions:**
1. Account `0.0.18` exists in the mirror node DB with `deleted = true`.
2. One or more rows exist in `token_allowance` with `owner = 18` and `amount > 0`.

**Trigger:**
```
GET /api/v1/accounts/0.0.18/allowances/tokens
```
or equivalently using the encoded integer:
```
GET /api/v1/accounts/18/allowances/tokens
```

**Expected result (per design intent):** HTTP 404, consistent with alias lookup behavior.

**Actual result:** HTTP 200 with a JSON body listing the deleted account's token allowances, e.g.:
```json
{
  "allowances": [
    {
      "amount": 500,
      "owner": "0.0.18",
      "spender": "0.0.99",
      "token_id": "0.0.200",
      "timestamp": { "from": "...", "to": null }
    }
  ]
}
```

**Contrast:** `GET /api/v1/accounts/KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ/allowances/tokens` (alias for account 18) → HTTP 404, as the alias path checks `coalesce(deleted, false) <> true`. [8](#0-7)

  async getEncodedId(entityIdString, requireResult = true, paramName = filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS) {
    try {
      if (EntityId.isValidEntityId(entityIdString)) {
        const entityId = EntityId.parseString(entityIdString, {paramName});
        return entityId.evmAddress === null
          ? entityId.getEncodedId()
          : await this.getEntityIdFromEvmAddress(entityId, requireResult);
      } else if (AccountAlias.isValid(entityIdString)) {
        return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
      }
    } catch (ex) {
      if (ex instanceof InvalidArgumentError) {
        throw InvalidArgumentError.forParams(paramName);
      }
      // rethrow
      throw ex;
    }

    throw InvalidArgumentError.forParams(paramName);
  }
```
