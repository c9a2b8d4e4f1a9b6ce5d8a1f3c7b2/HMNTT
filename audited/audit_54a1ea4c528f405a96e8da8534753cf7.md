### Title
Unauthenticated DB Connection Pool Exhaustion via Parallel Queries in `getContractResultsByTimestamp()`

### Summary
An unprivileged external user can submit concurrent `GET /api/v1/contracts/:contractId/results/:consensusTimestamp` requests with valid public path parameters, triggering up to six sequential/parallel database queries per request. Because the REST API's default connection pool is only 10 connections and no rate limiting exists on this endpoint, as few as two concurrent requests can saturate the pool, denying service to all other REST API consumers.

### Finding Description

**Exact code path:**

`rest/controllers/contractController.js`, `getContractResultsByTimestamp()` (lines 1002–1042):

```
1. getAndValidateContractIdAndConsensusTimestampPathParams()  ← format-only validation
2. ContractService.getInvolvedContractsByTimestampAndContractId()  ← 1 sequential DB query (line 1011)
3. this.getDetailedContractResults()  ← 5 parallel DB queries (line 1016-1017)
4. FileDataService.getLatestFileDataContents()  ← 1 conditional DB query (line 1023)
```

`getDetailedContractResults()` (lines 1205–1220) uses `Promise.all()` to fire five queries simultaneously:

```js
return Promise.all([
  ContractService.getContractResultsByTimestamps(...),
  TransactionService.getEthTransactionByTimestampAndPayerId(...),
  RecordFileService.getRecordFileBlockDetailsFromTimestamp(...),
  ContractService.getContractLogsByTimestamps(...),
  ContractService.getContractStateChangesByTimestamps(...),
]);
```

**Root cause:** No per-endpoint rate limiting exists in the REST API (Node.js) layer. The throttle infrastructure found in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java` applies only to the separate Web3/Java service, not to this REST API. The REST API's connection pool is configured at `rest/dbpool.js` (line 14: `max: config.db.pool.maxConnections`) with a documented default of **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`, per `docs/configuration.md` line 556).

**Failed assumption:** The design assumes that the `Promise.all()` fan-out of 5 parallel queries is safe because individual requests are infrequent. There is no mechanism to bound concurrent in-flight DB connections from this endpoint.

### Impact Explanation

With a pool of 10 connections and 5 parallel queries per request, only **2 concurrent requests** fully saturate the pool (2 × 5 = 10). Additional requests queue waiting for a connection up to `connectionTimeoutMillis` (default 20,000 ms), after which they fail. During saturation, every other REST API endpoint (transactions, accounts, tokens, etc.) also fails to acquire a DB connection, causing a complete REST API outage. The `statementTimeout` of 20,000 ms means each query holds a connection for up to 20 seconds, sustaining the exhaustion with a low request rate.

### Likelihood Explanation

No authentication or API key is required. Valid `contractId` and `consensusTimestamp` values are public blockchain data trivially discoverable via other mirror node endpoints (e.g., `GET /api/v1/contracts` and `GET /api/v1/contracts/:id/results`). The attack requires no special tooling — a simple script sending ~10–20 concurrent HTTP requests is sufficient. The attack is repeatable and stateless; the attacker does not need to maintain any session.

### Recommendation

1. **Add per-endpoint rate limiting** in the REST API (Node.js) middleware layer for expensive single-result endpoints like this one, independent of the Web3 throttle.
2. **Increase the default pool size** or configure it relative to expected concurrency, with documented guidance.
3. **Serialize the query fan-out** or add a semaphore to cap the number of concurrent DB connections any single request can hold.
4. **Add a global in-flight request limit** (e.g., via `express-rate-limit` or a connection semaphore) specifically for endpoints that issue multiple parallel DB queries.

### Proof of Concept

**Preconditions:**
- Discover a valid `contractId` (e.g., `0.0.1234`) via `GET /api/v1/contracts`
- Discover a valid `consensusTimestamp` for that contract via `GET /api/v1/contracts/0.0.1234/results`

**Trigger:**
```bash
# Send 20 concurrent requests
for i in $(seq 1 20); do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1234/results/1234567890.000000000" &
done
wait
```

**Result:** The 10-connection pool is exhausted within the first 2 concurrent requests. All subsequent requests to any REST API endpoint return connection timeout errors (HTTP 500 or connection refused) for up to 20 seconds per wave, effectively taking the REST API offline for the duration of the attack.
