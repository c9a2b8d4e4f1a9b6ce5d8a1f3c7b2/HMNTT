### Title
Race Condition in `unsubscribe()` Causes Missing Messages for New Subscribers via Stale Shared Flux

### Summary
`RedisTopicListener.getSharedListener()` uses `ConcurrentHashMap.computeIfAbsent` to return a shared `.share()` Flux, but the actual subscription to that Flux is deferred (lazy). An attacker who is the sole subscriber to a topic can cancel at the exact moment a new subscriber retrieves the Flux via `computeIfAbsent`, causing `unsubscribe()` to remove the Flux from `topicMessages` before the new subscriber has subscribed. The new subscriber then holds a reference to a cancelled Flux, re-triggers an upstream Redis re-subscription, and misses all messages published during the re-subscription window.

### Finding Description

**Exact code path:**

`getSharedListener` atomically returns a Flux via `computeIfAbsent`: [1](#0-0) 

The returned Flux is a `.share()` (i.e., `publish().refCount()`) Flux. The `doOnCancel` hook on the upstream of `.share()` calls `unsubscribe`: [2](#0-1) 

`unsubscribe` performs an unconditional key-based remove: [3](#0-2) 

The actual subscription is deferred — `listen()` returns the Flux lazily, and the caller subscribes later: [4](#0-3) 

**Root cause:** The compound operation "retrieve Flux from map → subscribe to Flux" is not atomic. `ConcurrentHashMap.computeIfAbsent` is atomic for the map operation, but the subscription to the returned Flux happens outside that atomic boundary.

**Exploit flow:**

```
T1: Attacker (A) is the sole subscriber to Flux_1 stored in topicMessages["topic.X"]
T2: Victim (B) calls listen() → getSharedListener() → computeIfAbsent finds Flux_1 → returns Flux_1
    (B has not yet subscribed — subscription is lazy)
T3: Attacker (A) cancels → share() refCount drops 1→0 → upstream cancelled
    → doOnCancel fires → unsubscribe() → topicMessages.remove("topic.X")
T4: Victim (B) subscribes to the Flux chain wrapping Flux_1
T5: B's subscription reaches Flux_1 (.share()); refCount was 0, upstream was cancelled
    → Reactor's refCount re-subscribes to upstream (new Redis connection via container.flatMapMany)
T6: During re-subscription window, messages published to Redis are missed by B
```

**Why existing checks fail:**

- `ConcurrentHashMap` only makes individual map operations atomic; it does not protect the "get-then-subscribe" compound operation.
- `unsubscribe` uses `topicMessages.remove(topic.getTopic())` — an unconditional key-based remove with no value check. If a new Flux_2 was inserted for the same key after the remove, a subsequent stale `doOnCancel` from Flux_1 will remove Flux_2, corrupting the map for all future subscribers. [3](#0-2) 

**Cascading effect:** After the race, Flux_1 is no longer in the map. Subscriber B re-subscribes via Flux_1 (a separate Redis subscription). Subscriber C creates Flux_2 (another Redis subscription). When B eventually cancels, Flux_1's `doOnCancel` fires and removes `topicMessages["topic.X"]` — which now holds Flux_2, not Flux_1. This removes the wrong entry, breaking C's subscription from the shared map and causing the same problem to cascade to all future subscribers.

### Impact Explanation

New subscribers can miss topic messages during the Redis re-subscription window, directly causing "missing records exported to mirror nodes." The unconditional `remove` in `unsubscribe` also means a stale cancel from one Flux can evict a different, active Flux from the map, breaking the shared-subscription invariant for all subsequent subscribers on that topic. In a sustained attack, every subscriber ends up on a separate Redis subscription (resource exhaustion) and each misses messages during their individual re-subscription delay.

### Likelihood Explanation

The attacker needs only a standard, unprivileged gRPC subscription — no elevated access. The race window is narrow (between `computeIfAbsent` returning and the lazy subscription being activated), but it is reliably reproducible under concurrent load. An attacker can widen the window by subscribing and immediately cancelling in a tight loop while a victim is subscribing, increasing the probability of a hit. In production systems with many concurrent subscribers, this race can also occur naturally without deliberate exploitation.

### Recommendation

1. **Use `remove(key, value)` instead of `remove(key)` in `unsubscribe`** to ensure only the specific Flux that triggered the cancel is removed, preventing stale cancels from evicting a newer Flux:
   ```java
   private void unsubscribe(Topic topic, Flux<TopicMessage> flux) {
       topicMessages.remove(topic.getTopic(), flux);
   }
   ```
   Pass the Flux reference into `doOnCancel`/`doOnComplete` via closure.

2. **Use `refCount(1, Duration)` (grace period) instead of bare `.share()`** so that the upstream is not immediately cancelled when the last subscriber leaves, giving new subscribers a window to attach without triggering a re-subscription.

3. **Validate the Flux is still active after retrieval** — after `computeIfAbsent` returns, check whether the Flux has already been terminated before returning it to the caller, and retry if so.

### Proof of Concept

```
1. Client A subscribes to topic X via gRPC → Flux_1 created, stored in topicMessages["topic.X"]
2. Client B calls listen(filterForTopicX) → getSharedListener returns Flux_1 (computeIfAbsent)
   [B has not yet subscribed — the Flux is returned but subscription is deferred]
3. Client A immediately cancels its gRPC stream
   → share() refCount 1→0 → doOnCancel fires → topicMessages.remove("topic.X")
4. Client B's subscription activates (e.g., after a scheduler yield or network round-trip)
   → subscribes to Flux_1 → refCount 0→1 → Reactor re-subscribes to upstream Redis channel
5. During the re-subscription (container.flatMapMany connecting to Redis), any messages
   published to Redis topic.X are not delivered to B
6. Repeat steps 1-5 in a loop: each iteration causes B to miss messages and
   eventually creates multiple separate Redis subscriptions for the same topic
```
