### Title
X-Forwarded-For Spoofing Bypasses Per-IP In-Flight Request Limit on GraphQL `account` Endpoint

### Summary
Traefik is globally configured with `--entryPoints.web.forwardedHeaders.insecure` and `--entryPoints.websecure.forwardedHeaders.insecure`, causing it to blindly trust client-supplied `X-Forwarded-For` headers from any source. The `inFlightReq` middleware protecting the GraphQL `account` query uses `ipStrategy.depth: 1` to derive the source IP from that same untrusted header. An unauthenticated attacker can rotate the `X-Forwarded-For` value across requests to appear as unlimited distinct IPs, completely bypassing the per-IP limit of 5 concurrent requests and exhausting backend resources.

### Finding Description

**Code locations:**

- Traefik global config — `charts/hedera-mirror-common/values.yaml` lines 635–637:
  ```yaml
  globalArguments:  # Expose X-Forwarded-For header for tracing
    - --entryPoints.web.forwardedHeaders.insecure
    - --entryPoints.websecure.forwardedHeaders.insecure
  ``` [1](#0-0) 

- GraphQL middleware config — `charts/hedera-mirror-graphql/values.yaml` lines 135–145:
  ```yaml
  middleware:
    - circuitBreaker:
        expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
    - inFlightReq:
        amount: 5
        sourceCriterion:
          ipStrategy:
            depth: 1
    - retry:
        attempts: 3
        initialInterval: 100ms
  ``` [2](#0-1) 

- GraphQL schema — `graphql/src/main/resources/graphql/query.graphqls` line 5:
  ```graphql
  account(input: AccountInput!): Account
  ``` [3](#0-2) 

**Root cause:** `--forwardedHeaders.insecure` instructs Traefik to accept `X-Forwarded-For` from **any** client without validating that it came from a trusted upstream proxy. Traefik's `ipStrategy.depth: 1` then reads the source IP for rate-limiting purposes directly from that attacker-controlled header. The failed assumption is that `X-Forwarded-For` is set by a trusted intermediary; in reality any external client can set it to an arbitrary value.

**Exploit flow:**
1. Attacker sends concurrent POST requests to `/graphql/alpha` with `account(input: {entityId: {num: N}})`.
2. Each request carries a distinct, spoofed `X-Forwarded-For` value (e.g., `1.0.0.1`, `1.0.0.2`, …).
3. Traefik evaluates `inFlightReq` per-IP using the spoofed value; each fake IP has 0 prior in-flight requests.
4. All requests pass the middleware and reach the backend simultaneously.
5. The `retry` middleware (3 attempts) further multiplies load on transient failures.

**Why existing checks fail:**
- `inFlightReq` — bypassed entirely via IP spoofing as described.
- `circuitBreaker` — only trips on error ratios (>10% network errors or >25% 5xx); a flood of successful but expensive queries never triggers it.
- `retry` — actively worsens the situation by tripling backend load on any transient error.
- No `rateLimit` middleware exists in the GraphQL chart (unlike the Rosetta chart which adds one).
- No authentication or authorization is required to call `account(input: AccountInput!)`. [4](#0-3) 

### Impact Explanation
An attacker can hold an unbounded number of concurrent database-backed GraphQL queries open against the backend. Each `account` query hits the PostgreSQL database. Exhausting the DB connection pool (visible via the `GraphQLHighDBConnections` alert threshold of 75%) causes all legitimate requests to queue or fail, constituting a full denial-of-service / network partition of the GraphQL service and its downstream database. Severity: **High** (unauthenticated DoS of a public API endpoint with no compensating control). [5](#0-4) 

### Likelihood Explanation
- **No privileges required**: the `/graphql/alpha` endpoint is publicly exposed with no authentication.
- **Trivial to exploit**: any HTTP client (curl, Python requests, etc.) can set arbitrary headers.
- **Repeatable and automatable**: a simple script rotating `X-Forwarded-For` values in a loop is sufficient.
- **No detection before impact**: the circuit breaker and alerting rules only fire after degradation has already occurred. [6](#0-5) 

### Recommendation

1. **Remove `--forwardedHeaders.insecure`** from Traefik's `globalArguments`. Replace it with `forwardedHeaders.trustedIPs` listing only the known upstream load balancer CIDR ranges:
   ```yaml
   globalArguments:
     - --entryPoints.web.forwardedHeaders.trustedIPs=10.0.0.0/8,<LB-CIDR>
     - --entryPoints.websecure.forwardedHeaders.trustedIPs=10.0.0.0/8,<LB-CIDR>
   ```

2. **Change `ipStrategy` to use the real remote address** (remove `depth: 1` or set `depth: 0`) so that rate limiting is applied to the actual TCP peer IP, which cannot be spoofed.

3. **Add a `rateLimit` middleware** (as done in the Rosetta chart) to enforce a per-IP request rate in addition to the in-flight limit:
   ```yaml
   - rateLimit:
       average: 10
       sourceCriterion:
         ipStrategy:
           depth: 0
   ```

4. Consider adding authentication/authorization to the GraphQL endpoint to prevent unauthenticated abuse. [7](#0-6) 

### Proof of Concept

```bash
#!/bin/bash
# Bypass inFlightReq per-IP limit by rotating X-Forwarded-For
ENDPOINT="https://<mirror-node-host>/graphql/alpha"
QUERY='{"query":"{ account(input: {entityId: {num: 1000}}) { balance { balance } } }"}'

for i in $(seq 1 1000); do
  FAKE_IP="10.$(( RANDOM % 256 )).$(( RANDOM % 256 )).$(( RANDOM % 256 ))"
  curl -s -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: $FAKE_IP" \
    -d "$QUERY" &
done

wait
echo "All requests dispatched"
```

Each iteration presents a unique spoofed IP. Traefik's `inFlightReq` sees each as a new source with 0 in-flight requests, allowing all 1000 concurrent requests to reach the backend simultaneously, exhausting DB connections and causing service degradation or outage.
