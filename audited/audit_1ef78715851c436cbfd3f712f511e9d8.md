### Title
Unauthenticated LRU Cache-Miss Amplification via `/block` Endpoint Causes Sustained DB Query Rate Elevation

### Summary
The `blockAPIService.updateOperationAccountAlias()` function in `rosetta/app/services/block_service.go` issues one `accountRepo.GetAccountAlias()` database query per operation whose account ID is absent from the shared LRU `entityCache`. Because the `/block` and `/block/transaction` endpoints require no authentication and the Traefik rate-limiting middleware is disabled by default (`global.middleware: false`), an unprivileged external attacker can sustain a high rate of cache-miss-driven DB queries by repeatedly requesting blocks that contain many unique account IDs not yet present in the cache, raising the DB query rate well above the 24-hour baseline.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, lines 104–130 (`updateOperationAccountAlias`):

```go
accountId := operations[index].AccountId
if cached, found = s.entityCache.Get(accountId.GetId()); !found {   // line 115
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)    // line 116 – DB query
    ...
    s.entityCache.Set(result.GetId(), result)
}
```

For every operation in every transaction of a block response, the function performs a synchronous DB query (`SELECT alias, id FROM entity WHERE id = @id`) on each LRU cache miss. The LRU cache is a single shared instance created at startup with capacity `hiero.mirror.rosetta.cache.entity.maxSize` (default 524 288).

**Root cause / failed assumption:**

The design assumes that the LRU cache will absorb repeated lookups for the same accounts. This holds for steady-state traffic where the same popular accounts recur. It fails when an attacker deliberately cycles through blocks whose operations reference large sets of distinct account IDs that collectively exceed the cache capacity, or simply requests many different blocks before their accounts have been cached. There is no per-request query budget, no deduplication of account IDs across concurrent requests, and no application-level rate limiting.

**Why existing checks are insufficient:**

1. **No authentication** on `/block` or `/block/transaction` — `main.go` lines 80–87 wire `blockAPIService` directly into the public router with no auth wrapper.
2. **Rate-limiting middleware disabled by default** — `charts/hedera-mirror-rosetta/values.yaml` line 95: `global.middleware: false`. The Traefik `rateLimit` / `inFlightReq` chain (lines 152–166 of the same file) is only activated when this flag is `true`, which is not the default deployment posture.
3. **`maxTransactionsInBlock` only caps transactions per response**, not the total number of DB queries across concurrent requests from multiple clients.
4. **LRU eviction is attacker-influenceable**: by cycling through blocks whose operations reference more than 524 288 distinct accounts in aggregate, the attacker continuously evicts warm entries, ensuring future requests for the same blocks also miss.

### Impact Explanation

Each `/block` request for a block with *N* operations referencing uncached accounts triggers *N* synchronous `SELECT` queries against the `entity` table. On a live network (mainnet/testnet) with millions of accounts spread across many blocks, an attacker can sustain a stream of cache-miss-heavy requests. Even at the Traefik default of 10 req/s per host (when enabled), 10 requests × 50 operations × 1 DB query = 500 extra queries/second. Without the middleware, the rate is bounded only by the DB connection pool (`maxOpenConnections: 100`). Saturating the connection pool degrades query latency for all other mirror-node consumers (REST API, gRPC, importer), constituting a resource-exhaustion denial-of-service against the database tier.

### Likelihood Explanation

The attack requires zero privileges, zero credentials, and only knowledge of the public Rosetta block API (a documented, standard interface). The attacker needs only to enumerate block indices (sequential integers) and select blocks known to contain many crypto-transfer operations — information freely available from the mirror node's own REST API. The attack is repeatable, automatable with a simple HTTP client loop, and distributable across multiple source IPs to bypass any IP-based in-flight request limits.

### Recommendation

1. **Enable and enforce application-level rate limiting unconditionally**, or move rate limiting inside the Go HTTP server (e.g., `golang.org/x/time/rate`) so it cannot be bypassed by skipping the Traefik layer.
2. **Deduplicate account IDs before querying**: collect all unique `accountId.GetId()` values across all operations in a request, batch-fetch only the uncached ones in a single `IN (...)` query, then populate the cache and rewrite operations — replacing N queries with at most 1.
3. **Add a per-request DB query cap**: if the number of cache misses in a single request exceeds a configurable threshold, return an error or serve the response without alias resolution rather than issuing unbounded DB queries.
4. **Set `global.middleware: true` as the default** in the Helm chart, or document it as a required security configuration.

### Proof of Concept

```bash
# 1. Identify blocks with many crypto-transfer operations via the mirror REST API
curl "https://<mirror-node>/api/v1/transactions?transactiontype=CRYPTOTRANSFER&limit=100" \
  | jq '.transactions[].consensus_timestamp'

# 2. Convert consensus timestamps to block indices via Rosetta /network/status or /block
# 3. Rapidly request those blocks in a loop (no auth required):
for BLOCK_IDX in $(seq 1000 1500); do
  curl -s -X POST https://<rosetta-endpoint>/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$BLOCK_IDX}}" &
done
wait

# Each request for a block with N unique uncached account IDs fires N SELECT queries
# against the entity table. With no rate limiting (global.middleware=false, default),
# the DB connection pool is the only throttle.
```

Monitoring `pg_stat_activity` on the mirror-node database will show a spike in concurrent `SELECT alias, id FROM entity WHERE id = $1` queries proportional to the request rate, confirming sustained DB query rate elevation above the 24-hour baseline.
