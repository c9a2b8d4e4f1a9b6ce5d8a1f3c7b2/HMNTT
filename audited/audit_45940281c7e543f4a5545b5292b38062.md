### Title
Missing `@Validated` on `AllowancesController` Renders `@Max(MAX_LIMIT)` Unenforced, Allowing Unbounded DB Result Sets

### Summary
`AllowancesController` declares `@Max(MAX_LIMIT)` on the `limit` request parameter, but the controller class itself lacks `@Validated`, which is required for Spring MVC to activate method-level Bean Validation. Without it, the constraint is dead metadata. An unauthenticated attacker can supply an arbitrarily large `limit` value that flows unmodified through the service into a raw jOOQ `.limit(request.getLimit())` call, causing the database to return an unbounded result set.

### Finding Description

**Controller — constraint declared but never enforced:**

`AllowancesController.java` line 61 declares:
```java
@RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
``` [1](#0-0) 

The class-level annotations are only `@CustomLog`, `@RequestMapping`, `@RequiredArgsConstructor`, and `@RestController` — `@Validated` is absent. [2](#0-1) 

A grep across the entire controller package confirms `@Validated` appears nowhere. Without it, Spring's `MethodValidationInterceptor` is never registered for this bean, so `@Max`, `@Positive`, and `@Size` are all ignored at runtime.

**DTO — no independent guard:**

`NftAllowanceRequest.limit` carries no validation annotation of its own; it is a plain `int` with a default of 25. [3](#0-2) 

**Service — no limit check:**

`NftAllowanceServiceImpl.getNftAllowances()` performs only parameter-relationship checks (owner/spender/token range validity) and passes the request object directly to the repository. [4](#0-3) 

**Repository — raw limit propagation:**

`NftAllowanceRepositoryCustomImpl.findAll()` calls `.limit(request.getLimit())` with no capping or sanity check before issuing the SQL query. [5](#0-4) 

**Root cause:** The sole enforcement point (`@Max(MAX_LIMIT)`) is a Bean Validation annotation that requires `@Validated` on the declaring class to be activated. Its absence means the annotation is compile-time decoration only.

### Impact Explanation

An attacker can force the mirror node to execute `SELECT * FROM nft_allowance WHERE owner = ? AND approved_for_all = true LIMIT 2147483647`, returning every matching row in the table. This causes:
- **Memory exhaustion** in the JVM (all rows materialized via `fetchInto`)
- **Database CPU/IO spike** on every such request
- **Effective DoS** against the REST API with no authentication required, since the endpoint is public

The `Constants.MAX_LIMIT = 100` design parameter is completely bypassed. [6](#0-5) 

### Likelihood Explanation

- No authentication is required; the endpoint is public (`/api/v1/accounts/{id}/allowances/nfts`)
- The exploit is a single HTTP GET with `?limit=2147483647` — trivially repeatable
- No special knowledge of the system is needed
- The attacker only needs a valid account ID (enumerable from public ledger data)
- The attack can be parallelized across many accounts to amplify resource exhaustion

### Recommendation

1. **Immediate fix — add `@Validated` to the controller:**
   ```java
   @Validated
   @RestController
   public class AllowancesController { ... }
   ```
   This activates Spring's method validation interceptor and makes `@Max(MAX_LIMIT)` effective.

2. **Defense-in-depth — cap the limit in the service or DTO:**
   Add a hard cap in `NftAllowanceServiceImpl.getNftAllowances()`:
   ```java
   int safeLimit = Math.min(request.getLimit(), Constants.MAX_LIMIT);
   ```
   Or add `@Max(100)` to the `limit` field in `NftAllowanceRequest` and enable validation there.

3. **Defense-in-depth — cap in the repository:**
   ```java
   .limit(Math.min(request.getLimit(), Constants.MAX_LIMIT))
   ```

### Proof of Concept

**Preconditions:** Mirror node is running; at least one account with NFT allowances exists (account num `12345` used as example).

**Steps:**

```bash
# 1. Normal request — returns at most 100 rows (design intent)
curl "http://<mirror-node>/api/v1/accounts/12345/allowances/nfts?limit=100"

# 2. Exploit — @Max is not enforced; limit passes through unchecked
curl "http://<mirror-node>/api/v1/accounts/12345/allowances/nfts?limit=2147483647"
```

**Expected (broken) behavior:** The second request executes `SELECT ... FROM nft_allowance WHERE owner = 12345 AND approved_for_all = true LIMIT 2147483647`, returning all matching rows, consuming unbounded memory and DB resources.

**Verification:** Enable jOOQ SQL logging and observe the emitted `LIMIT` clause reflects the attacker-supplied value rather than being capped at 100.
