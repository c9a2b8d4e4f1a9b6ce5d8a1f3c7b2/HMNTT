### Title
`BufferUnderflowException` via Under-Length EVM Address in `getByEvmAddressAndType()`

### Summary
`decodeEvmAddress()` in `GraphQlUtils.java` performs no length validation on the decoded byte array, accepting any even-length hex string. When a hex string shorter than 20 bytes is passed to `getByEvmAddressAndType()`, the subsequent `ByteBuffer.getInt()` / `ByteBuffer.getLong()` calls throw an uncaught `BufferUnderflowException`. The custom exception resolver does not handle this exception type, so it propagates as an `INTERNAL_ERROR` GraphQL response rather than a `ValidationError`, leaking the existence of an internal code path and producing a different observable error classification than expected.

### Finding Description

**Exact code path:**

`decodeEvmAddress()` in `GraphQlUtils.java` lines 87–98: [1](#0-0) 

It strips the `0x` prefix and calls `Hex.decodeHex()` with no check that the result is exactly 20 bytes. The test suite explicitly confirms this — `0000AaAaAa` (10 hex chars = 5 bytes) is a valid accepted input: [2](#0-1) 

`getByEvmAddressAndType()` in `EntityServiceImpl.java` lines 34–41: [3](#0-2) 

- Line 37 calls `buffer.getInt()` (consumes 4 bytes) then `buffer.getLong()` (consumes 8 bytes) — 12 bytes minimum required for the first condition.
- Line 38 calls `buffer.getLong()` again — 20 bytes total required.
- If the input is 4–11 bytes, `getInt()` succeeds but the first `getLong()` throws `java.nio.BufferUnderflowException`.
- If the input is < 4 bytes, `getInt()` itself throws immediately.

**Why the existing check fails:**

`CustomExceptionResolver.resolveToSingleError()` only handles `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`: [4](#0-3) 

`BufferUnderflowException` is a `RuntimeException` not in this list. The method returns `null`, causing Spring GraphQL's default handler to emit an `INTERNAL_ERROR` classification instead of a `ValidationError`. This is a distinct, observable difference from the `ValidationError` returned for an odd-length hex string (which throws `IllegalArgumentException` from `Hex.decodeHex()`).

### Impact Explanation

An unprivileged attacker can:
1. Distinguish between "odd-length hex" (→ `ValidationError`) and "even-length short hex" (→ `INTERNAL_ERROR`) by observing the GraphQL error `classification` field — confirming the internal branching structure of the entity resolution path.
2. Force an unhandled Java exception on every such request, bypassing the intended validation layer.
3. In non-default or debug configurations (e.g., `spring.graphql.schema.printer.enabled`, exception detail propagation), the full exception class name and message are included in the error response, further exposing internal implementation details.

Severity: **Medium** — information leakage about internal code paths; no authentication required; no data modification possible.

### Likelihood Explanation

Exploitation requires only the ability to send a GraphQL query with an `evmAddress` field — no credentials, no special network position. The trigger is a single well-formed GraphQL request with a short even-length hex string (e.g., `0x00000000aabb`). It is trivially repeatable and requires no prior knowledge of the system beyond the public GraphQL schema.

### Recommendation

1. **Add a length check in `decodeEvmAddress()`** or at the start of `getByEvmAddressAndType()`:
   ```java
   if (evmAddressBytes.length != 20) {
       throw new IllegalArgumentException("EVM address must be exactly 20 bytes");
   }
   ```
   This throws `IllegalArgumentException`, which `CustomExceptionResolver` already handles correctly as a `ValidationError`.

2. **Alternatively**, add `BufferUnderflowException` to the handled types in `CustomExceptionResolver` and map it to a `ValidationError`.

3. The fix belongs in `GraphQlUtils.decodeEvmAddress()` since it is the canonical entry point for EVM address decoding and the length invariant should be enforced there.

### Proof of Concept

Send the following GraphQL request to the mirror node GraphQL endpoint (no authentication required):

```graphql
query {
  account(input: { evmAddress: "0x00000000aabb" }) {
    id
  }
}
```

`0x00000000aabb` decodes to 6 bytes. `buffer.getInt()` reads the first 4 bytes (value `0`), then `buffer.getLong()` attempts to read 8 bytes from a buffer with only 2 remaining → `java.nio.BufferUnderflowException` is thrown.

**Expected (correct) response:** `{"errors":[{"extensions":{"classification":"ValidationError"}, ...}]}`

**Actual response:** `{"errors":[{"message":"INTERNAL_ERROR","extensions":{"classification":"INTERNAL_ERROR"}, ...}]}`

The differing `classification` value confirms the internal exception path and that the input bypassed the intended validation layer.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {

            return GraphqlErrorBuilder.newError()
                    .errorType(ErrorType.ValidationError)
                    .message(ex.getMessage())
                    .path(env.getExecutionStepInfo().getPath())
