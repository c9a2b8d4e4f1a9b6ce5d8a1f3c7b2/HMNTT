### Title
Unauthenticated Retry Amplification via `fallbackRetry()` Causes Up to 10x DB Load Per Request

### Summary
The `fallbackRetry()` function in `rest/service/fileDataService.js` unconditionally executes the full two-level CTE query up to 10 times per single HTTP request when corrupt file versions are encountered. The `/api/v1/network/exchangerate` endpoint is publicly accessible with no authentication or per-request rate limiting, allowing any external user to trigger this amplification by targeting a `timestamp=lte:X` value that lands on a known-corrupt file version, causing each retry to issue a fresh full CTE scan against the database.

### Finding Description

**Exact code path:**

`rest/service/fileDataService.js`, `fallbackRetry()`, lines 93–118:

```js
fallbackRetry = async (fileEntityId, filterQueries, resultConstructor) => {
    const whereQuery = filterQueries.whereQuery ?? [];
    const filters = {whereQuery};

    let attempts = 0;
    while (++attempts <= 10) {                                    // line 98 — hard cap of 10
      const row = await this.getLatestFileDataContents(fileEntityId, filters);  // line 99 — full CTE each time
      try {
        return row === null ? null : new resultConstructor(row);
      } catch (error) {
        ...
        filters.whereQuery = [
          ...whereQuery,                                          // line 108 — spreads ORIGINAL, not accumulated
          {
            query: FileData.CONSENSUS_TIMESTAMP + utils.opsMap.lt,
            param: row.first_consensus_timestamp,
          },
        ];
      }
    }
    return null;
};
``` [1](#0-0) 

Each call to `getLatestFileDataContents` executes the full `latestFileContentsQuery` CTE:

```sql
with latest_create as (
  select max(consensus_timestamp) ...
  from file_data
  where entity_id = $1 and transaction_type in (17, 19) <filter>
  group by entity_id
  order by consensus_timestamp desc
)
select max(...), min(...), string_agg(file_data, '' order by consensus_timestamp)
from file_data fd
join latest_create l on fd.consensus_timestamp >= l.consensus_timestamp
where ...
group by entity_id
``` [2](#0-1) 

This is a non-trivial aggregation query (CTE + join + `string_agg`) executed once per retry attempt.

**Root cause and failed assumption:**

The design assumes corrupt file versions are rare and transient. There is no guard preventing an external caller from deliberately targeting a timestamp window that contains corrupt data. The loop cap of 10 is treated as a safety net, but it is simultaneously the amplification factor: a single HTTP request can cause up to 10 full CTE executions.

**Exploit flow:**

1. Attacker probes `GET /api/v1/network/exchangerate?timestamp=lte:T` with varying `T` values and observes response latency. Requests that trigger retries take measurably longer (each retry is a synchronous `await`).
2. Attacker identifies a `T` value that lands just after a corrupt file version (or a sequence of corrupt versions).
3. Attacker floods the endpoint with concurrent requests using that `T` value.
4. Each request causes 2–10 full CTE executions against the database, depending on how many consecutive corrupt versions exist.

**Why existing checks fail:**

- The `authHandler` middleware only sets a custom response-row *limit* for authenticated users; it does not block or throttle unauthenticated requests to the exchange rate endpoint. [3](#0-2) 

- The `/api/v1/network/exchangerate` endpoint requires no authentication per the OpenAPI spec and the controller. [4](#0-3) 

- There is no per-endpoint rate limiter, no circuit breaker, and no caching layer visible in the service path that would absorb repeated identical queries.

### Impact Explanation

A single unauthenticated request can cause up to 10 full CTE+join+aggregation queries against the `file_data` table. With a modest number of concurrent connections (e.g., 50 concurrent requests each triggering 5 retries), the database receives 250 heavy aggregation queries instead of 50. This directly maps to the stated threat model of increasing processing node resource consumption by ≥30% without brute-force volume. The `file_data` table grows continuously as the network operates, making the CTE scans progressively more expensive over time.

### Likelihood Explanation

- The endpoint is unauthenticated and publicly documented.
- Corrupt file versions can arise naturally (partial file updates, chunked appends, network-level corruption during ingestion). Their existence is not under attacker control, but their *discovery* requires only passive timing analysis of the public endpoint — no privileged access.
- Once a corrupt timestamp window is identified, the attack is trivially repeatable and scriptable.
- No exploit tooling beyond `curl` or any HTTP client is required.

### Recommendation

1. **Add a response cache** for the exchange rate (and fee schedule) at the service layer, keyed on the effective timestamp bound. A short TTL (e.g., 10–30 seconds) eliminates redundant CTE executions for repeated identical queries.
2. **Limit retry amplification**: reduce `maxRetries` from 9 (10 total attempts) to 2–3, which is sufficient for legitimate recovery from transient corruption.
3. **Add per-IP or global rate limiting** on the `/api/v1/network/exchangerate` endpoint at the API gateway or middleware layer, independent of authentication status.
4. **Pre-validate file data integrity** at ingestion time so corrupt versions are never stored, removing the precondition entirely.

### Proof of Concept

```bash
# Step 1: Discover a slow (retrying) timestamp via timing
for T in $(seq 1000000000 1000 1001000000); do
  TIME=$(curl -o /dev/null -s -w "%{time_total}" \
    "https://<mirror-node>/api/v1/network/exchangerate?timestamp=lte:$T")
  echo "$T $TIME"
done | sort -k2 -rn | head -5
# Timestamps with highest latency are candidates for corrupt file versions

# Step 2: Flood with the identified timestamp (e.g., T=1000500000)
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/network/exchangerate?timestamp=lte:1000500000" &
done
wait
# Each of the 200 concurrent requests triggers up to 10 CTE queries = up to 2000 DB queries
# vs. the expected 200, a 10x amplification
```
