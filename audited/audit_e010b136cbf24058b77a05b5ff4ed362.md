### Title
Binary Search Gas Estimation Multiplier Bypasses Throttle: Up to 21× EVM Execution Per Single Throttle-Deducted Request

### Summary
`ThrottleManagerImpl.throttle()` deducts gas tokens from the bucket exactly once per request, based on the declared `request.getGas()`. When `estimate=true`, `ContractExecutionService.estimateGas()` runs one initial EVM execution followed by up to 20 additional binary-search iterations via `BinaryGasEstimator.search()`. Each binary-search iteration calls `doProcessCall(params, gas, true)`, which skips `restoreGasToBucket` and—critically—never deducts additional tokens from the throttle bucket. A single unauthenticated request therefore causes up to 21 full EVM executions while consuming only one request's worth of throttle tokens.

### Finding Description

**Entry point** — `ContractController.call()`: [1](#0-0) 

`throttleManager.throttle(request)` is called once, consuming `scaleGas(request.getGas())` tokens from the gas bucket: [2](#0-1) 

`scaleGas` divides by `GAS_SCALE_FACTOR = 10,000`, so `gas=15,000,000` → 1,500 tokens consumed: [3](#0-2) 

**Estimate path** — `ContractExecutionService.estimateGas()`: [4](#0-3) 

Step 1: `callContract(params, context)` → `doProcessCall(params, params.getGas(), false)` — `estimate=false`, so `restoreGasToBucket` runs and returns some tokens.

Step 2: `binaryGasEstimator.search(gas -> doProcessCall(params, gas, true), ...)` — up to `maxGasEstimateRetriesCount = 20` iterations: [5](#0-4) 

**Root cause** — `doProcessCall` with `estimate=true` skips the `restoreGasToBucket` call entirely, and no additional tokens are deducted from the throttle bucket for any binary-search iteration: [6](#0-5) 

The `maxGasEstimateRetriesCount` default is 20: [7](#0-6) 

**Failed assumption**: The throttle assumes one gas deduction ≈ one EVM execution. For estimate requests, the ratio is 1 deduction : up to 21 executions.

### Impact Explanation

With `gas=15,000,000` (the default maximum) and bytecode that consumes ~14M gas per execution:

| Metric | Value |
|---|---|
| Throttle tokens deducted | 1,500 (net ~1,400 after initial-call restore) |
| Actual EVM executions | 21 (1 initial + 20 binary search) |
| Actual gas-equivalent work | ~294,000,000 |
| Throttle-equivalent work | ~14,000,000 |
| Multiplier | **~21×** |

A single request causes 21× the EVM computation that the throttle accounts for, far exceeding the 30% threshold. Repeated at the rate limit (500 req/s default), this can saturate node CPU and memory, causing denial of service for all other users.

### Likelihood Explanation

- **No authentication required**: any external user can POST to `/api/v1/contracts/call`
- **No special knowledge needed**: setting `estimate=true` and providing any gas-heavy bytecode (e.g., a loop-heavy constructor) is sufficient
- **Repeatable**: the rate-limit bucket (500 req/s) still allows 500 such amplified requests per second
- **Bytecode is not validated** for computational complexity before the binary search begins

### Recommendation

1. **Deduct gas proportionally for estimate requests**: Before entering `binaryGasEstimator.search()`, pre-deduct `maxGasEstimateRetriesCount × scaleGas(params.getGas())` tokens from the gas bucket, or deduct one additional token per binary-search iteration inside `BinaryGasEstimator.search()`.

2. **Alternatively, apply a separate lower rate limit for estimate requests**: Add a dedicated throttle bucket for `estimate=true` requests with a much lower capacity, since each such request is inherently more expensive.

3. **Cap binary search iterations based on remaining gas budget**: Track cumulative gas consumed across all iterations and abort early if it exceeds a configurable multiple of the declared gas limit.

### Proof of Concept

```bash
# Single request triggering up to 21 EVM executions while consuming 1 throttle slot
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "estimate": true,
    "gas": 15000000,
    "data": "0x<gas-exhausting-constructor-bytecode>",
    "to": null
  }'
```

**Gas-exhausting constructor bytecode** (Solidity pseudocode):
```solidity
constructor() {
    // Loop consuming ~14M gas
    uint x = 0;
    for (uint i = 0; i < 500000; i++) { x += i; }
}
```

**Steps**:
1. Deploy the above contract bytecode as `data` with `estimate=true` and `gas=15000000`
2. `ThrottleManagerImpl.throttle()` deducts 1,500 tokens once
3. `estimateGas()` runs 1 initial call + up to 20 binary-search iterations, each executing the full constructor loop
4. Node CPU spikes ~21× compared to a single `eth_call` with the same bytecode
5. Repeat at 500 req/s to sustain the amplified load

    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {
            return Bytes.ofUnsignedLong(gasUsedByInitialCall);
        }

        final var status = ResponseCodeEnum.SUCCESS.toString();
        final var estimatedGas = binaryGasEstimator.search(
                (totalGas, iterations) -> updateMetrics(params, totalGas, iterations, status),
                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
    }
