### Title
Uncached EVM Address DB Lookup in `parseAccountIdQueryParam` Enables Connection Pool Exhaustion DoS

### Summary
`parseAccountIdQueryParam()` in `rest/balances.js` unconditionally fires an uncached database query via `EntityService.getEncodedId()` for every request containing a valid EVM address as `account.id`. With no rate limiting on the Node.js REST API and a default pool of only 10 connections, an unprivileged attacker sending concurrent requests with EVM addresses can saturate the connection pool, causing all subsequent requests to queue and time out, degrading the mirror node REST service.

### Finding Description

**Exact code path:**

In `rest/balances.js` lines 320–357, `parseAccountIdQueryParam` processes the `account.id` query parameter:

```js
if (EntityId.isValidEvmAddress(value, EvmAddressType.NO_SHARD_REALM) && ++evmAliasAddressCount === 1) {
    return EntityService.getEncodedId(value, false);   // line 329 — returns a Promise
}
``` [1](#0-0) 

The returned Promise is collected and awaited in `getBalances`:

```js
const [accountQuery, accountParamsPromise] = parseAccountIdQueryParam(req.query, 'ab.account_id');
const accountParams = await Promise.all(accountParamsPromise);
``` [2](#0-1) 

`EntityService.getEncodedId` resolves the EVM address by calling `getEntityIdFromEvmAddress`, which issues a live SQL query against the `entity` table with no caching:

```js
async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
        [Buffer.from(entityId.evmAddress, 'hex')]);
``` [3](#0-2) 

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where ${Entity.DELETED} <> true
      and ${Entity.EVM_ADDRESS} = $1`;
``` [4](#0-3) 

`BaseService.getRows` calls `pool.queryQuietly`, consuming a connection from the shared pool: [5](#0-4) 

**Root cause — failed assumption:** The code assumes that one extra DB query per EVM-address request is harmless. There is no caching layer for this lookup (unlike `entityId.js`'s `parseCached` LRU which only caches parsing, not DB results), and there is no rate limiting on the Node.js REST API.

**Pool configuration:** The default `maxConnections` is **10** and `statementTimeout` is **20 000 ms**: [6](#0-5) [7](#0-6) 

**Why the existing guard is insufficient:** The `evmAliasAddressCount === 1` check only prevents more than one EVM address *within a single request*. It does not limit the number of concurrent requests, each of which independently acquires a pool connection for the EVM lookup plus a second connection for the main balance query. [8](#0-7) 

### Impact Explanation
With a pool of 10 connections and a 20-second statement timeout, an attacker holding 10 concurrent requests open (each awaiting the EVM address DB lookup) fully saturates the pool. All other REST API requests — including those from legitimate users — queue indefinitely and eventually time out. Because the Node.js REST API has no rate limiting (the throttle infrastructure exists only in the Java web3 service), there is no server-side defence to prevent this. A single attacker with modest bandwidth can degrade the mirror node REST service to the point where it cannot serve any requests. [9](#0-8) 

### Likelihood Explanation
No authentication or special privilege is required. The `/api/v1/balances` endpoint is public. The exploit requires only an HTTP client capable of sending concurrent requests with a 40-character hex string as `account.id` (e.g., `account.id=0xabcdef...`). The `isValidEvmAddress` check accepts any 40-hex-char string with optional `0x` prefix, so the attacker does not need a real on-chain address. The attack is trivially scriptable, repeatable, and requires no prior knowledge of the system. [10](#0-9) 

### Recommendation
1. **Cache EVM-address-to-entity-id resolutions** in `EntityService` using the existing `entityId` cache infrastructure (already configured for up to 100 000 entries with a 1 800-second TTL).
2. **Add rate limiting** to the Node.js REST API (e.g., via an express-rate-limit middleware or an ingress-level policy), analogous to the throttle already present in the web3 Java service.
3. **Increase the default pool size** or add a per-IP concurrency limit at the load-balancer/ingress layer.
4. Consider rejecting EVM address lookups that resolve to `null` (i.e., unknown addresses) early, before they consume a pool connection for the main query.

### Proof of Concept
```bash
# Send 20 concurrent requests, each triggering an uncached EVM-address DB lookup
for i in $(seq 1 20); do
  curl -s "http://<mirror-node>/api/v1/balances?account.id=0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef" &
done
wait
# Subsequent legitimate requests will queue/timeout while the pool is saturated
curl -v "http://<mirror-node>/api/v1/balances"
```

Each of the 20 concurrent requests enters `parseAccountIdQueryParam`, passes `isValidEvmAddress` (any 40-hex string is valid), increments `evmAliasAddressCount` to 1, and calls `EntityService.getEncodedId` → `getEntityIdFromEvmAddress` → `pool.queryQuietly`, holding a connection for up to 20 seconds. With a default pool of 10, the pool is exhausted after 10 concurrent requests; the remaining 10 queue, and all new legitimate requests are blocked.
