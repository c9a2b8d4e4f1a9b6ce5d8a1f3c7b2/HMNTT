### Title
Unauthenticated Connection Pool Exhaustion via Concurrent Maximum-Parameter Crypto Allowance Requests

### Summary
The `getAccountCryptoAllowances()` handler in `rest/controllers/cryptoAllowanceController.js` accepts up to 100 repeated `spender.id` query parameters (the default `maxRepeatedQueryParameters` limit) with no rate limiting. Because the REST API's default database pool is only 10 connections and each query can hold a connection for up to 20 seconds (the default `statementTimeout`), an unauthenticated attacker sending 10+ concurrent maximum-parameter requests exhausts the pool, causing all subsequent database-backed requests to stall for the full 20-second `connectionTimeout` before failing.

### Finding Description
**Code path:**

`rest/controllers/cryptoAllowanceController.js` → `getAccountCryptoAllowances()` (lines 76–98) calls `utils.buildAndValidateFilters(req.query, acceptedCryptoAllowanceParameters)`, then `this.extractCryptoAllowancesQuery(filters, accountId)`, then `CryptoAllowanceService.getAccountCryptoAllowances(conditions, params, order, limit)`.

**Root cause — three compounding weaknesses:**

1. **No rate limiting on the REST API.** A grep for `rateLimit`, `rateLimiter`, `express-rate`, and `throttle` across all `rest/**/*.js` files returns zero matches. The throttling that exists (`web3/src/main/java/…/ThrottleManagerImpl.java`) applies only to the web3 Java service, not to this Node.js REST service.

2. **Tiny default connection pool.** `rest/dbpool.js` lines 7–16 sets `max: config.db.pool.maxConnections`. Per `docs/configuration.md` line 556, the default is **10 connections**. Per line 555, `connectionTimeout` defaults to **20,000 ms**.

3. **Large IN-clause amplification.** `rest/utils.js` lines 1241–1247 (`buildFilters`) enforces `isRepeatedQueryParameterValidLength` (line 488: `values.length <= config.query.maxRepeatedQueryParameters`). The default `maxRepeatedQueryParameters` is **100** (docs line 581). In `extractCryptoAllowancesQuery` (lines 31–46), every `eq`-operator `spender.id` value is pushed into `spenderInValues`; `updateQueryFiltersWithInValues` (baseController.js lines 29–44) then emits `spender IN ($2, $3, …, $101)`. The resulting query is issued against the pool via `CryptoAllowanceService.getAccountCryptoAllowances` → `super.getRows(query, params)` with a `statement_timeout` of **20,000 ms** (docs line 557).

**Why existing checks fail:**
- `maxRepeatedQueryParameters=100` rejects a single request with >100 values, but does nothing to limit the *rate* or *concurrency* of requests.
- `statementTimeout=20000` bounds a single query's runtime, but 20 seconds × 10 concurrent connections = the pool is fully occupied for up to 20 seconds per wave of attack requests.
- `connectionTimeout=20000` means queued requests stall silently for 20 seconds before receiving an error, rather than failing fast.

### Impact Explanation
With the pool exhausted, every other REST endpoint that issues a database query (transactions, balances, tokens, etc.) stalls for up to 20 seconds per request. Hedera's consensus round time is approximately 3–5 seconds; a 20-second stall represents 400–667% of average round time, satisfying the ≥500% threshold for the described impact. The REST mirror node becomes effectively unavailable to all consumers during the attack window, which can be sustained continuously by recycling the 10-connection attack wave.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and only a standard HTTP client capable of sending concurrent requests. The endpoint is publicly documented in `rest/api/v1/openapi.yml` lines 267–289. The attacker needs only to discover any valid account ID (trivially obtained from the same API) and send 10 concurrent requests with 100 `spender.id` parameters each. The attack is fully repeatable with no cool-down because there is no per-IP or per-account rate limiting.

### Recommendation
1. **Add rate limiting middleware** to the Node.js REST service (e.g., `express-rate-limit`) scoped per IP, applied globally before route handlers.
2. **Reduce `maxRepeatedQueryParameters`** for the `spender.id` filter specifically, or add a per-filter cap in `extractCryptoAllowancesQuery` (e.g., max 10 `eq` values).
3. **Increase `maxConnections`** or introduce a per-request connection acquisition timeout shorter than `statementTimeout` so pool exhaustion fails fast rather than stalling callers.
4. **Add a query-level circuit breaker** or concurrency limiter (e.g., `pg-pool` queue depth limit) so that pool-wait queue depth is bounded.

### Proof of Concept
```bash
# Generate 100 spender.id params
PARAMS=$(python3 -c "print('&'.join(f'spender.id={i}' for i in range(1,101)))")

# Send 15 concurrent requests (> pool size of 10)
for i in $(seq 1 15); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1001/allowances/crypto?${PARAMS}" &
done
wait

# All subsequent requests to ANY REST endpoint now stall for ~20 seconds:
time curl "http://<mirror-node>/api/v1/transactions?limit=1"
# Expected: ~20 second delay before response or connection timeout error
```

The first 10 concurrent requests each acquire one of the 10 pool connections and hold them for up to 20 seconds. Requests 11–15 queue. Any unrelated request (transactions, balances, etc.) issued during this window also queues and stalls for the full `connectionTimeout` of 20,000 ms.
