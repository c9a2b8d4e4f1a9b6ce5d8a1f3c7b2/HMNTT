### Title
`ne`-Only Timestamp Filter Bypasses `bindTimestampRange` Enforcement in `optimizeTimestampFilters()`, Enabling Unbounded Contract Result Enumeration

### Summary
In `optimizeTimestampFilters()`, when a user supplies only `timestamp=ne:X` filters, the parsed `range` is `null` and `eqValues` is empty. The function delegates range bounding to `bindTimestampRange(null, order)`, but when `bindTimestampRange` is disabled (the default), it returns `{range: null}`, causing no range conditions to be emitted. Only the bare `ne` exclusion filter reaches the SQL query, producing an unbounded `WHERE consensus_timestamp != X` scan over the entire `contract_result` table. An unprivileged external user can exploit this to paginate through the complete contract transaction history.

### Finding Description

**Exact code path:**

`rest/controllers/contractController.js`, `optimizeTimestampFilters()`, lines 384–405:

```js
const {range, eqValues, neValues} = utils.parseTimestampFilters(
  timestampFilters, false, true, true, false, false   // allowNe=true, validateRange=false
);
// When input is only timestamp=ne:X:
//   range     = null
//   eqValues  = []
//   neValues  = [X]

const {range: optimizedRange, next} =
  eqValues.length === 0                          // true → calls bindTimestampRange
    ? await bindTimestampRange(range, order)     // bindTimestampRange(null, order)
    : {range};

// bindTimestampRange with config.query.bindTimestampRange=false (default):
//   returns {range: null}  →  optimizedRange = null

if (optimizedRange?.begin) { /* skipped */ }
if (optimizedRange?.end)   { /* skipped */ }

// Only the ne filter is pushed:
neValues.forEach((value) =>
  filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value})
);
``` [1](#0-0) 

`bindTimestampRange` in `rest/timestampRange.js` lines 19–38 immediately returns `{range}` unchanged when `queryConfig.bindTimestampRange` is `false`:

```js
const bindTimestampRange = async (range, order) => {
  if (!queryConfig.bindTimestampRange) {
    return {range};          // range is null → no bounding occurs
  }
  ...
};
``` [2](#0-1) 

The resulting filter is handed to `updateConditionsAndParamsWithInValues` in `baseController.js` (lines 12–27), which for a non-`eq` operator emits `consensus_timestamp != $N` directly into the SQL `WHERE` clause with no range guard. [3](#0-2) 

This path is reached for the global `/api/v1/contracts/results` endpoint (no `contractId`), confirmed at line 504–505:

```js
const {filters: optimizedTimestampFilters, next} =
  contractId === undefined
    ? await optimizeTimestampFilters(timestampFilters, order)
    : {filters: timestampFilters};
``` [4](#0-3) 

**Root cause:** `optimizeTimestampFilters` assumes `bindTimestampRange` will always impose a time window, but `bindTimestampRange` is a no-op when disabled. The `neValues` path is never independently bounded; it is silently passed through regardless of whether a range was established.

**Why existing checks fail:**
- `parseTimestampFilters` is called with `validateRange=false` and `allowOpenRange=true`, so no range-size or completeness check fires.
- The skip-on-empty guard at line 506 (`optimizedTimestampFilters.length === 0`) does not trigger because the `ne` filter itself is non-empty.
- `strictCheckOverride=false` means strict timestamp checks defer to `config.strictTimestampParam`, which does not block a lone `ne` filter when `allowNe=true`. [5](#0-4) [6](#0-5) 

### Impact Explanation
The generated SQL becomes effectively `SELECT … FROM contract_result WHERE consensus_timestamp != <X> ORDER BY … LIMIT <default>`. With pagination (`limit` + cursor), an attacker can retrieve every row in the `contract_result` table. This:
1. **Bypasses the intended `maxTransactionsTimestampRange` window** that protects both data confidentiality and database performance.
2. **Causes full sequential table scans** on potentially very large partitioned tables, enabling a low-cost denial-of-service against the mirror node database.
3. **Exposes the complete contract execution history** (sender, calldata, results, gas) to any unauthenticated caller.

### Likelihood Explanation
- No authentication or special privilege is required; the `/api/v1/contracts/results` endpoint is public.
- The `ne` operator is explicitly accepted by `optimizeTimestampFilters` (`allowNe=true`), so no input-validation rejection occurs.
- `bindTimestampRange` defaults to `false` in standard deployments (confirmed by test teardown resetting to `false` and the feature being opt-in).
- The attack is trivially repeatable with a single HTTP request and standard pagination.

### Recommendation
1. **Require a bounded range when `ne`-only filters are present.** Inside `optimizeTimestampFilters`, if `range` is `null` and `eqValues` is empty but `neValues` is non-empty, either reject the request or synthesize a default bounded range before calling `bindTimestampRange`.
2. **Apply `bindTimestampRange` unconditionally when `neValues` are present**, regardless of whether `range` is null, so the time window is always enforced.
3. **Enable `bindTimestampRange=true` by default** in production configuration, or enforce a mandatory range check in `parseTimestampFilters` when called from `optimizeTimestampFilters`.

### Proof of Concept

**Preconditions:** Default mirror-node deployment with `bindTimestampRange=false` (default).

**Step 1 – Send a `ne`-only timestamp filter:**
```
GET /api/v1/contracts/results?timestamp=ne:1.000000000&order=asc&limit=100
```

**Step 2 – Observe the response:** Returns up to 100 contract results spanning the entire history, with no time-range restriction.

**Step 3 – Paginate to enumerate all records:**
Use the `links.next` cursor from each response to iterate through the complete `contract_result` table.

**Expected SQL generated (simplified):**
```sql
SELECT * FROM contract_result
WHERE consensus_timestamp != 1000000000
ORDER BY consensus_timestamp ASC
LIMIT 100;
```

No lower or upper bound on `consensus_timestamp` is present, confirming the bypass.

const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
};
