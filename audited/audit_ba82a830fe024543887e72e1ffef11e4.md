### Title
Unbounded Concurrent Subscriptions Exhaust Shared `boundedElastic` Scheduler, Starving Legitimate Topic-Message Subscribers

### Summary
`TopicMessageServiceImpl.subscribeTopic()` increments `subscriberCount` solely as a Prometheus gauge metric with no enforcement gate. An unauthenticated attacker can open an arbitrary number of TCP connections (each carrying up to the per-connection limit of 5 concurrent streams) and flood the shared Reactor `boundedElastic` scheduler with safety-check and retrieval tasks, exhausting its thread pool and task queue. Legitimate subscribers then have their scheduled work rejected or indefinitely delayed, causing them to miss critical HCS topic messages.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.java` line 48 declares `subscriberCount` as a plain `AtomicLong`: [1](#0-0) 

It is wired only into a Micrometer `Gauge` at lines 52–55 and incremented/decremented at lines 89–90: [2](#0-1) [3](#0-2) 

There is **no conditional check** of `subscriberCount` anywhere in `subscribeTopic()` that would reject or throttle a new subscription.

Every accepted subscription schedules a `safetyCheck` task on the **global shared** `Schedulers.boundedElastic()` at line 70: [4](#0-3) 

`PollingTopicMessageRetriever` and `PollingTopicListener` also call `Schedulers.boundedElastic()` — all three share the same JVM-wide scheduler instance: [5](#0-4) [6](#0-5) [7](#0-6) 

**Root cause:** `subscriberCount` is a read-only metric. The only server-side concurrency control is `maxConcurrentCallsPerConnection = 5`, which is a *per-TCP-connection* limit, not a global one: [8](#0-7) [9](#0-8) 

There is no limit on the number of TCP connections an attacker may open.

**Exploit flow:**

1. Attacker opens `C` TCP connections to port 5600 (no authentication required, no IP-based connection cap at the application layer).
2. On each connection the attacker opens 5 concurrent `subscribeTopic` streams (the per-connection maximum), yielding `5C` total live subscriptions.
3. Each subscription fires a `safetyCheck` `Mono.delay(1s).subscribeOn(Schedulers.boundedElastic())` task. With `5C` subscriptions, `5C` tasks are queued on the shared `boundedElastic` pool.
4. Reactor's `boundedElastic` scheduler caps threads at `10 × availableProcessors` (e.g., 40 threads on a 4-core pod) and its task queue at 100,000 entries. Once the queue is saturated, new `subscribeOn`/`publishOn` calls throw `RejectedExecutionException`.
5. Legitimate subscribers' safety-check and retrieval tasks are rejected or starved; their reactive pipelines stall, causing missed or delayed topic messages.

### Impact Explanation
HCS topics carry financial-grade messages (token transfers, payment confirmations). A subscriber that misses messages due to scheduler starvation may fail to detect a fund transfer, double-spend guard, or settlement confirmation. The impact is denial-of-service against the subscriber tier of the mirror node, with direct downstream risk of financial message loss for any application relying on real-time HCS delivery. Severity is high because the endpoint is unauthenticated and the shared scheduler is a single point of failure for all reactive work in the gRPC process.

### Likelihood Explanation
The gRPC port (5600) is publicly exposed with no authentication. An attacker needs only a standard gRPC client library. Opening thousands of connections is trivial from a single machine or a small botnet. The `maxConcurrentCallsPerConnection = 5` default actually makes the attack *cheaper* per connection (5 streams saturate the per-connection limit immediately). The infrastructure-level `maxRatePerEndpoint: 250` in the GCP gateway config limits *new request rate*, not the number of *sustained open streaming connections*: [10](#0-9) 

The attack is repeatable and requires no special privileges.

### Recommendation
1. **Enforce a global subscriber cap** — check `subscriberCount` against a configurable `maxSubscribers` property inside `subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a per-IP or per-client connection limit** at the Netty layer via `NettyServerBuilder.maxConnectionsPerIp()` or an equivalent interceptor.
3. **Isolate the safety-check scheduler** — create a dedicated, bounded `Schedulers.newBoundedElastic(...)` instance for the safety-check path rather than using the shared global one, so attacker-induced saturation cannot spill over into the retriever and listener pipelines.
4. **Add gRPC server-side rate limiting** (e.g., a `ServerInterceptor` with a token-bucket per remote address).

### Proof of Concept
```python
# Requires: pip install grpcio
import grpc, threading, time
from concurrent.futures import ThreadPoolExecutor

# proto-generated stub for ConsensusService/subscribeTopic
# Assume stub is available as consensus_pb2_grpc / consensus_pb2

TARGET = "mirror.mainnet.hedera.com:443"
NUM_CONNECTIONS = 500   # 500 connections × 5 streams = 2500 subscriptions

def flood(i):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_pb2.ConsensusTopicQuery(topicID=..., consensusStartTime=...)
    streams = []
    for _ in range(5):   # maxConcurrentCallsPerConnection default
        streams.append(stub.subscribeTopic(req))
    # Keep streams open indefinitely
    for s in streams:
        try:
            for _ in s: pass
        except: pass

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as ex:
    for i in range(NUM_CONNECTIONS):
        ex.submit(flood, i)
        time.sleep(0.01)   # slow ramp to avoid TCP RST storms

# After ~2500 open subscriptions the boundedElastic queue fills;
# legitimate subscriber safety-check tasks are rejected and pipelines stall.
```
