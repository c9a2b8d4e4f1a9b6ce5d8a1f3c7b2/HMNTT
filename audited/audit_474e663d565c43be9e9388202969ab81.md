### Title
Sustained HTTP 500 Responses to Mirror Node REST Endpoint Exhaust Retries and Fatally Terminate the Pinger Process

### Summary
When `network=other` is configured, `newClient()` calls `buildNetworkFromMirrorNodes()`, which retries HTTP requests to the operator-configured mirror node REST endpoint. HTTP 500 responses are explicitly treated as retryable; once all retries are exhausted the error propagates to `main()`, which calls `log.Fatalf`, terminating the process. The heartbeat goroutine then stops writing `/tmp/alive`, causing the liveness probe to fail within 2 minutes.

### Finding Description
**Code path:**

`pinger/mirror_node_client.go` line 91 — HTTP 500 sets `retry = true`:
```go
retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
```

`pinger/mirror_node_client.go` lines 48–69 — finite retry loop with exponential backoff; on exhaustion returns `nil, lastErr`:
```go
attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)
for attempt := 1; attempt <= attempts; attempt++ {
    network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
    if err == nil { return network, nil }
    lastErr = ...
    if !retry || attempt == attempts { break }
    backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
    ...
}
return nil, lastErr
```

`pinger/sdk_client.go` lines 18–21 — error returned from `buildNetworkFromMirrorNodes` is returned from `newClient()`:
```go
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
if err != nil { return nil, err }
```

`pinger/main.go` lines 41–44 — error from `newClient()` causes fatal exit:
```go
client, err := newClient(cfg)
if err != nil {
    log.Fatalf("client error: %v", err)
}
```

`pinger/cmd/healthcheck/main.go` lines 34–38 — liveness probe fails if `/tmp/alive` is older than 2 minutes:
```go
age := time.Since(fi.ModTime())
if age > 2*time.Minute { ... os.Exit(1) }
```

**Root cause:** The code assumes the mirror node REST endpoint is trustworthy and available at startup. There is no fallback, no graceful degradation, and no recovery path — a finite retry budget feeds directly into `log.Fatalf`.

**Failed assumption:** The mirror node REST endpoint will eventually return a non-5xx response within `mirrorNodeClientMaxRetries` attempts. If an attacker can sustain 500 responses for the full retry window, this assumption is violated and the process dies.

### Impact Explanation
The pinger process exits via `log.Fatalf`. The heartbeat goroutine (started before `newClient()`) stops writing `/tmp/alive`. Within 2 minutes the liveness probe binary (`healthcheck live`) returns exit code 1, causing Kubernetes to restart or mark the pod unhealthy. The readiness file `/tmp/ready` is never written (it is written only after `newClient()` succeeds), so the pod also fails readiness checks. The pinger's primary function — submitting periodic HBAR transfers to verify network health — is completely unavailable for the duration of the attack.

### Likelihood Explanation
The precondition is `network=other`, which is the deployment mode for custom/private mirror node environments. In such deployments the mirror REST URL (`HIERO_MIRROR_PINGER_REST`) is often a publicly reachable HTTP endpoint. An unprivileged attacker who can reach that endpoint needs only to flood it with requests sufficient to keep it returning HTTP 500 for the full retry window. With default settings (`mirrorNodeClientMaxRetries=10`, `mirrorNodeClientBaseBackoff=500ms`, exponential backoff), the total window is approximately 511 seconds (~8.5 minutes). Sustaining a service-level disruption on a REST API for 8–9 minutes is well within the capability of a low-resource attacker. No credentials, no authentication, and no privileged network position are required — only the ability to send HTTP traffic to the mirror node REST endpoint.

### Recommendation
1. **Do not call `log.Fatalf` on `newClient()` failure at startup.** Instead, retry `newClient()` in a loop (with backoff and a context deadline) so the process stays alive and the heartbeat continues.
2. **Decouple startup from the liveness probe.** The heartbeat goroutine should be sufficient to keep the pod alive while the client is being initialized; the readiness probe (`/tmp/ready`) already gates traffic correctly.
3. **Add a context deadline to `buildNetworkFromMirrorNodes`.** Pass a context with a total deadline (e.g., 60 s) rather than `context.Background()`, so the retry loop is bounded by wall-clock time, not just attempt count.
4. **Consider jitter in backoff** to prevent thundering-herd re-connection storms after the mirror node recovers.

### Proof of Concept
1. Deploy the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-or-DDoSed-host>/api/v1`.
2. Ensure the target REST endpoint returns `HTTP 500` for every `GET /api/v1/network/nodes` request (e.g., configure a mock server or sustain a DDoS on the real mirror node REST service).
3. Start (or restart) the pinger process.
4. Observe: the pinger logs `attempt 1/11 … attempt 11/11` with 500 errors, then logs `client error: attempt 11/11: GET … returned 500 Internal Server Error` and exits.
5. Wait 2 minutes; run `healthcheck live` — it exits with code 1 (`heartbeat stale`), confirming the liveness probe is down.
6. Kubernetes restarts the pod; if the mirror node endpoint still returns 500, the cycle repeats indefinitely, keeping the pinger permanently unavailable.
