### Title
Unauthenticated Sustained Request Flood on `/api/v1/network/nodes` Causes Unbounded DB CPU/I/O Exhaustion via Unprotected Multi-CTE Correlated Subquery

### Summary
The `getNetworkNodes()` method in `NetworkServiceImpl.java` executes a multi-CTE SQL query with a per-row correlated `jsonb_agg` subquery against `address_book_service_endpoint` on every request, with no server-side rate limiting or result caching in the `rest-java` module. Any unauthenticated external user can sustain a high-rate stream of individually valid GET requests to `/api/v1/network/nodes`, causing cumulative database CPU and I/O consumption to exceed 30% above the 24-hour baseline without triggering any brute-force or authentication guard.

### Finding Description

**Exact code path:**

`NetworkController.java` line 153–158 routes `GET /api/v1/network/nodes` to `networkService.getNetworkNodes(request)` with no throttle guard: [1](#0-0) 

`NetworkServiceImpl.getNetworkNodes()` (lines 100–137) performs no caching and calls `networkNodeRepository.findNetworkNodes()` unconditionally on every invocation: [2](#0-1) 

`NetworkNodeRepository.findNetworkNodes()` (lines 25–105) executes a native multi-CTE query that includes:
- A `latest_address_book` CTE scanning `address_book` with `ORDER BY start_consensus_timestamp DESC LIMIT 1`
- A `latest_node_stake` CTE with a nested `SELECT max(consensus_timestamp) FROM node_stake`
- A `node_info` CTE scanning the full `node` table
- A **correlated subquery per result row** executing `jsonb_agg(...)` against `address_book_service_endpoint` filtered by `consensus_timestamp` and `node_id` [3](#0-2) 

**Root cause — failed assumption:** The design assumes an external rate-limiting layer (e.g., API gateway, reverse proxy) will protect this endpoint. The `rest-java` module itself has no rate limiting for this path. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) exists exclusively in the `web3` module and is not wired into `rest-java`: [4](#0-3) 

The only filters registered in `rest-java` are `MetricsFilter` (metrics recording only), `LoggingFilter` (logging only), and a `ShallowEtagHeaderFilter` (HTTP ETag — does not prevent DB query execution): [5](#0-4) [6](#0-5) 

No `@Cacheable` or equivalent annotation exists anywhere in `NetworkServiceImpl`: [7](#0-6) 

The `MAX_LIMIT = 25` cap in `NetworkNodeRequest` limits result set size but does not reduce the number of CTE scans or correlated subquery executions per request: [8](#0-7) 

The REST Java API configuration table confirms no throttle or cache properties exist for this module: [9](#0-8) 

### Impact Explanation
Each request forces the database to: (1) scan `address_book` with sort, (2) compute `max(consensus_timestamp)` over `node_stake`, (3) scan `node`, and (4) execute one `jsonb_agg` correlated subquery per returned node row against `address_book_service_endpoint`. On a production network with ~30 nodes, this is ~30 correlated subquery executions per request. At 100–500 requests/second (easily achievable from a single machine with HTTP keep-alive), this translates to thousands of correlated subquery executions per second, driving DB CPU and I/O well above 30% of the 24-hour baseline. The impact is denial-of-service to all other DB-dependent services sharing the same PostgreSQL instance (importer, other REST endpoints).

### Likelihood Explanation
The endpoint requires zero authentication, zero API keys, and zero prior knowledge beyond the public OpenAPI spec. The attack is trivially repeatable with standard HTTP load tools (`wrk`, `ab`, `hey`). A single attacker with a modest internet connection can sustain the required request rate. The attack is non-brute-force — each request is a fully valid, well-formed GET with no anomalous parameters.

### Recommendation
1. **Add server-side rate limiting in `rest-java`**: Port the `bucket4j`-based `ThrottleConfiguration`/`ThrottleManagerImpl` pattern from the `web3` module into `rest-java`, applying a per-IP or global RPS cap to `/api/v1/network/nodes`.
2. **Add result caching**: Annotate `getNetworkNodes()` with `@Cacheable` (Spring Cache) with a short TTL (e.g., 30–60 seconds). The address book changes infrequently; caching is safe and eliminates repeated DB hits for identical parameter sets.
3. **Add a DB-level statement timeout**: The `restJava.db.statementTimeout` is already configurable (default 10000ms); ensure it is set aggressively enough to bound runaway query cost per request.
4. **Flatten the correlated subquery**: Rewrite the `jsonb_agg` correlated subquery as a lateral join or a pre-aggregated CTE to reduce per-row DB work.

    public List<NetworkNodeDto> getNetworkNodes(NetworkNodeRequest request) {
        final long fileId = getAddressBookFileId(request);
        final var limit = request.getEffectiveLimit();
        final var nodeIdParams = request.getNodeIds();
        final var orderDirection = request.getOrder().name();

        final Set<Long> nodeIds = new HashSet<>();
        long lowerBound = 0L;
        long upperBound = Long.MAX_VALUE;

        for (final var nodeIdParam : nodeIdParams) {
            if (nodeIdParam.operator() == RangeOperator.EQ) {
                nodeIds.add(nodeIdParam.value());
            } else if (nodeIdParam.hasLowerBound()) {
                lowerBound = Math.max(lowerBound, nodeIdParam.getInclusiveValue());
            } else if (nodeIdParam.hasUpperBound()) {
                upperBound = Math.min(upperBound, nodeIdParam.getInclusiveValue());
            }
        }

        if (lowerBound > upperBound) {
            throw new IllegalArgumentException("Invalid range provided for node.id");
        }

        final Long[] nodeIdArray;
        if (!nodeIds.isEmpty()) {
            final var range = Range.closed(lowerBound, upperBound);
            nodeIdArray = nodeIds.stream().filter(range::contains).toArray(Long[]::new);
            if (nodeIdArray.length == 0) {
                return List.of();
            }
        } else {
            nodeIdArray = EMPTY_NODE_ID_ARRAY;
        }

        return networkNodeRepository.findNetworkNodes(
                fileId, nodeIdArray, lowerBound, upperBound, orderDirection, limit);
    }
