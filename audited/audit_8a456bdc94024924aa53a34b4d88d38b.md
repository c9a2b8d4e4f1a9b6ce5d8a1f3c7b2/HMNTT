### Title
LRU Entity Cache Thrashing via Unauthenticated Block() Requests Causes Amplified DB Query Load

### Summary
The `entityCache` LRU in `blockAPIService` has a fixed capacity (`MaxSize`, default 524288) with no TTL and no application-level rate limiting on the Rosetta API. An unauthenticated attacker can cycle `Block()` requests across blocks whose cumulative unique account IDs exceed `MaxSize`, causing LRU eviction of previously-cached entries. Subsequent re-requests for those same blocks produce repeated `GetAccountAlias` DB queries for every evicted account, amplifying database load with no economic cost to the attacker.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 104–130:

```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {   // line 115
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)     // DB hit
    ...
    s.entityCache.Set(result.GetId(), result)                        // line 121
    cached = result
}
```

The cache is constructed at lines 34–37:
```go
entityCache := cache.NewContext(
    serverContext,
    cache.AsLRU[int64, types.AccountId](lru.WithCapacity(entityCacheConfig.MaxSize)),
)
```

Default `MaxSize` is **524288** (documented as `hiero.mirror.rosetta.cache.entity.maxSize`). There is **no TTL** — entries are only removed by LRU eviction when capacity is exceeded.

**Root cause and failed assumption:**

The design assumes the working set of account IDs across concurrent requests will remain within `MaxSize`. This assumption fails when an attacker deliberately cycles requests across blocks whose union of account IDs exceeds the cache capacity, causing the LRU to continuously evict entries that are immediately needed again.

**Exploit flow:**

1. Attacker identifies two disjoint sets of blocks: set **S1** (blocks referencing accounts A₁…Aₙ) and set **S2** (blocks referencing accounts Aₙ₊₁…A₂ₙ), where n > MaxSize/2.
2. Attacker sends `Block()` for all blocks in S1 → accounts A₁…Aₙ fill the LRU cache.
3. Attacker sends `Block()` for all blocks in S2 → accounts Aₙ₊₁…A₂ₙ fill the cache, evicting A₁…Aₙ.
4. Attacker re-requests blocks in S1 → every account A₁…Aₙ is a cache miss → `GetAccountAlias` DB query fires for each one.
5. Repeat cyclically to maintain sustained DB pressure.

**Why existing checks are insufficient:**

- The Rosetta Go application has **no application-level rate limiting**. The only rate limiting present is in the optional Traefik Kubernetes middleware (`rateLimit: average: 10` per `requestHost`, `inFlightReq: amount: 5` per IP — see `charts/hedera-mirror-rosetta/values.yaml` lines 157–160). This is a deployment-level control that is absent in non-Kubernetes or direct-exposure deployments.
- The web3 API has a `ThrottleConfiguration` with `requestsPerSecond` enforcement; the Rosetta API has no equivalent.
- The `maxOpenConnections` for the Rosetta DB pool is **100** (documented default). Sustained cache-miss amplification can exhaust this pool.

### Impact Explanation

Each cache miss in `updateOperationAccountAlias()` triggers a synchronous `SELECT alias, id FROM entity WHERE id = @id` query against the mirror node database. With the DB pool capped at 100 open connections, an attacker sustaining ~100 concurrent cache-miss-heavy `Block()` requests can saturate the connection pool, causing legitimate requests to queue or time out. This degrades service availability for all users of the Rosetta API without any economic cost or privilege requirement for the attacker.

### Likelihood Explanation

The Rosetta API requires no authentication. The Hedera mainnet has tens of millions of accounts, making it straightforward to identify blocks whose union of account IDs exceeds 524288. The attack requires only HTTP access and knowledge of block indices — both publicly available via the Rosetta `/block` endpoint itself. The cycling pattern is trivially scriptable and repeatable indefinitely.

### Recommendation

1. **Add application-level rate limiting** to the Rosetta API (analogous to the web3 `ThrottleConfiguration`) — a per-IP or global request-per-second cap enforced in Go middleware, not relying solely on Traefik.
2. **Add a TTL to the entity cache** so that even without eviction, stale entries expire and the cache naturally refreshes, reducing the value of thrashing.
3. **Deduplicate account IDs within a single `updateOperationAccountAlias()` call** before issuing DB queries, to reduce per-request DB amplification.
4. **Cap the number of unique account IDs resolved per request** to bound the worst-case DB query fan-out per `Block()` call.

### Proof of Concept

```
# Step 1: Identify two groups of blocks with many unique account IDs
# (e.g., using /block endpoint to enumerate blocks on mainnet)

# Step 2: Fill cache with group S1
for block_index in S1_blocks:
    POST /block {"block_identifier": {"index": block_index}}

# Step 3: Evict S1 entries by filling with group S2
for block_index in S2_blocks:
    POST /block {"block_identifier": {"index": block_index}}

# Step 4: Re-request S1 — all account IDs are now cache misses → DB queries
for block_index in S1_blocks:
    POST /block {"block_identifier": {"index": block_index}}

# Step 5: Repeat steps 2-4 in a tight loop with concurrency=100
# Result: DB connection pool saturated, legitimate requests time out
