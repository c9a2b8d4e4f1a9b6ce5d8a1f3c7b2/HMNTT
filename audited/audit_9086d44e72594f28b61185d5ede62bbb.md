### Title
`RegisteredNodeType.UNKNOWN` Queryable via Public API Exposes Unclassified Infrastructure Nodes

### Summary
The `GET /api/v1/network/registered-nodes?type=UNKNOWN` endpoint accepts `UNKNOWN` as a valid `type` filter because `RegisteredNodeType.UNKNOWN` is a member of the Java enum and no validation excludes it. This causes the SQL query's `type @> array[0]::smallint[]` containment check to return nodes whose type array contains `0`, exposing unclassified/unintended infrastructure nodes to any unauthenticated caller. The response mapper silently re-labels these nodes as `GENERAL_SERVICE`, masking the disclosure.

### Finding Description
**Code path:**

`RegisteredNodesRequest.type` is declared as `RegisteredNodeType` with no constraint excluding `UNKNOWN`: [1](#0-0) 

`RequestParameterArgumentResolver` uses Spring's `WebDataBinder` to bind the raw string `"UNKNOWN"` to `RegisteredNodeType.UNKNOWN` via standard enum conversion — no custom converter or allowlist is applied: [2](#0-1) 

`NetworkServiceImpl.getRegisteredNodes()` calls `nodeType.getId()` unconditionally when `nodeType != null`, passing `(short) 0` to the repository: [3](#0-2) 

The native SQL query uses the PostgreSQL array containment operator `@>`. With `:type = 0`, the condition becomes `type @> array[0]::smallint[]`, which matches every row whose `type` column contains the value `0`: [4](#0-3) 

`RegisteredNodeType.UNKNOWN` has `id = 0`: [5](#0-4) 

The response mapper then silently re-labels `UNKNOWN` nodes as `GENERAL_SERVICE`, hiding the true classification from the caller: [6](#0-5) 

**Root cause / failed assumption:** The design assumes `UNKNOWN` is an internal sentinel value that will never reach the query layer. There is no `@NotNull`-equivalent constraint, no allowlist validator, and no service-layer guard that rejects `UNKNOWN` before it is converted to its numeric id and forwarded to the database.

**Why existing checks fail:** The only guard in the query is `:type is null or …`. Because `UNKNOWN.getId()` returns `0` (not `null`), the null-bypass branch is never taken and the containment filter executes normally against `UNKNOWN`-typed rows.

The test suite confirms `UNKNOWN` is intentionally excluded from the set of valid user-facing type values (it is never listed in `typeQueryParameterIsCaseInsensitive`), yet no runtime enforcement of that exclusion exists: [7](#0-6) 

### Impact Explanation
An unauthenticated caller can enumerate all non-deleted registered nodes whose `type` array contains `0`. These nodes are unclassified infrastructure — potentially nodes in a provisioning, quarantine, or error state — whose service endpoints (IP address, port, TLS flag) are fully returned in the response body. Exposing these endpoints allows an attacker to map shadow infrastructure, attempt direct connections to nodes that are not yet hardened or monitored, and potentially route protocol-level traffic (e.g., block-streaming or mirror-node gRPC calls) to unintended endpoints. The mapper's silent re-labelling of `UNKNOWN` → `GENERAL_SERVICE` means the caller cannot distinguish these nodes from legitimate `GENERAL_SERVICE` nodes, compounding the confusion.

### Likelihood Explanation
Exploitation requires zero privileges and zero special tooling — a single HTTP GET request with `?type=UNKNOWN` suffices. The parameter name and enum values are documented in the OpenAPI spec. Any developer or researcher reading the spec or the source code can discover this within minutes. The attack is fully repeatable and leaves no distinguishing trace beyond a normal API access log entry.

### Recommendation
Add an explicit allowlist validation in `RegisteredNodesRequest` or in `NetworkServiceImpl` that rejects `RegisteredNodeType.UNKNOWN` before it reaches the repository. The simplest fix is a Jakarta Validation custom constraint or an inline guard:

```java
// In NetworkServiceImpl.getRegisteredNodes()
if (nodeType == RegisteredNodeType.UNKNOWN) {
    throw new IllegalArgumentException("Invalid parameter: type");
}
```

Alternatively, remove `UNKNOWN` from the public-facing `RegisteredNodeType` enum and use a separate internal sentinel, or add a `@ValidRegisteredNodeType` constraint to the `type` field in `RegisteredNodesRequest` that enumerates only `BLOCK_NODE`, `GENERAL_SERVICE`, `MIRROR_NODE`, and `RPC_RELAY` as accepted values.

### Proof of Concept
**Precondition:** At least one `registered_node` row exists with `deleted = false` and a `type` array containing `0` (e.g., inserted by the importer when the consensus node reports an unrecognised node type).

**Trigger:**
```
GET /api/v1/network/registered-nodes?type=UNKNOWN
```

**Expected (intended) result:** HTTP 400 — `Invalid parameter: type`

**Actual result:** HTTP 200 with a JSON body listing all non-deleted `UNKNOWN`-type nodes, each including their full `serviceEndpoints` (IP, port, TLS), with `type` field shown as `GENERAL_SERVICE` in the response.

**Verification steps:**
1. Insert a test row: `INSERT INTO registered_node (registered_node_id, deleted, type, timestamp_range) VALUES (999, false, ARRAY[0]::smallint[], '[1,)'::int8range);`
2. `curl "http://<host>/api/v1/network/registered-nodes?type=UNKNOWN"`
3. Observe node `999` returned with `"type": "GENERAL_SERVICE"`.
