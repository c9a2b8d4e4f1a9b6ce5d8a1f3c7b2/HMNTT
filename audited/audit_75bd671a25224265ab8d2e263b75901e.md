### Title
Unauthenticated Double DB Query via `?scheduled=true` on Non-Existent Transaction ID

### Summary
Any unprivileged user can force `getTransactionsByIdOrHash()` to execute two database queries per request by supplying `?scheduled=true` with any syntactically valid but non-existent transaction ID. The condition `scheduled && transactions.length === 0` in `mayMissLongTermScheduledTransaction()` unconditionally returns `true` when the first query returns zero rows and `scheduled=true`, triggering a second, wider-range query. Since the transaction never exists, this double-query pattern fires on every such request with no bound.

### Finding Description

**Exact code path:**

In `rest/transactions.js`, `getTransactionsByIdOrHash()` (lines 923–938):

```javascript
const {rows} = await pool.queryQuietly(query, params).then((result) => {
  if (!mayMissLongTermScheduledTransaction(isTransactionHash, scheduled, result.rows)) {
    return result;
  }
  params[params.upperConsensusTimestampIndex] =
    params[params.lowerConsensusTimestampIndex] + maxScheduledTransactionConsensusTimestampRangeNs;
  return pool.queryQuietly(query, params);  // second query, wider range
});
``` [1](#0-0) 

`mayMissLongTermScheduledTransaction()` (lines 974–996):

```javascript
const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  if (isTransactionHash || scheduled === false) {
    return false;
  }
  if (scheduled === undefined) { /* ... */ }
  else if (scheduled && transactions.length === 0) {
    return true;   // <-- triggered unconditionally
  }
  return false;
};
``` [2](#0-1) 

**Root cause:** The branch at line 991 (`scheduled && transactions.length === 0`) was designed to handle the case where a scheduled transaction might exist beyond the initial timestamp window. However, it does not distinguish between "transaction exists but scheduled child is outside the window" and "transaction never existed at all." When the transaction ID is entirely absent from the DB, the first query returns 0 rows, the condition is `true`, and the second query fires unconditionally.

**Why existing checks fail:** The only guard is `isTransactionHash || scheduled === false` (line 976). For a transaction-ID-based request with `?scheduled=true`, `isTransactionHash` is `false` and `scheduled` is `true`, so neither guard fires. There is no pre-check that the base transaction exists before deciding to retry. [3](#0-2) 

The `acceptedSingleTransactionParameters` set explicitly allows `SCHEDULED` as a query parameter with no restrictions: [4](#0-3) 

### Impact Explanation
Every request of the form `GET /api/v1/transactions/<valid-format-but-nonexistent-id>?scheduled=true` always executes two DB queries. The second query uses `maxScheduledTransactionConsensusTimestampRangeNs` as its upper bound — a significantly larger timestamp window than the first query's `maxTransactionConsensusTimestampRangeNs` — making it more expensive. An attacker can sustain this 2× (or greater) DB amplification at will, degrading database throughput for all users. No authentication, no special account, and no existing transaction is required.

### Likelihood Explanation
The exploit requires only: (1) knowledge of the public REST API, (2) any syntactically valid transaction ID (e.g., `0.0.1-1234567890-000000000`), and (3) appending `?scheduled=true`. It is trivially repeatable in a loop with no rate-limiting visible in the reviewed code. Any external user or automated scanner can trigger it.

### Recommendation
Before executing the retry logic, verify that the first query returned at least one row for the base (non-scheduled) transaction. If `transactions.length === 0` and `scheduled === true`, the transaction simply does not exist — return `false` from `mayMissLongTermScheduledTransaction` immediately and let the `NotFoundError` path handle it. The retry should only be attempted when the first query returned rows for the parent transaction but no scheduled child was found within the initial window:

```javascript
} else if (scheduled && transactions.length === 0) {
  // Do NOT retry — no base transaction exists at all
  return false;
}
```

Alternatively, add a lightweight existence check (e.g., query only `payer_account_id` + `valid_start_ns`) before the full query, and skip the retry entirely if no row is found.

### Proof of Concept
```bash
# Step 1: Pick any syntactically valid but non-existent transaction ID
TX_ID="0.0.99999999-9999999999-999999999"

# Step 2: Send request with ?scheduled=true
curl -v "https://<mirror-node-host>/api/v1/transactions/${TX_ID}?scheduled=true"

# Result: HTTP 404 Not Found
# DB effect: Two queries executed per request instead of one.
#   Query 1: consensus_timestamp range [valid_start - drift, valid_start + maxTransactionConsensusTimestampRangeNs]
#   Query 2: consensus_timestamp range [valid_start - drift, valid_start + maxScheduledTransactionConsensusTimestampRangeNs]
#            (wider range, more expensive)

# Step 3: Repeat in a loop to amplify DB load
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node-host>/api/v1/transactions/0.0.${i}-9999999999-999999999?scheduled=true" &
done
```
