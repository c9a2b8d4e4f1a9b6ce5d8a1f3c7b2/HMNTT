### Title
Missing Lower-Bound Check in `findByTimestamp` Causes Wrong RecordFile Attribution After Pruning

### Summary
The `findByTimestamp` query in `RecordFileRepository` uses only `consensusEnd >= ?1` without a corresponding `consensusStart <= ?1` guard. After pruning removes older RecordFiles, any consensus timestamp that belonged to a pruned file will match the next surviving RecordFile, attributing the transaction to the wrong block. This is exploitable by any unprivileged user who can reference a transaction whose RecordFile has been pruned.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, line 45:

```java
@Query("select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1")
Optional<RecordFile> findByTimestamp(long timestamp);
``` [1](#0-0) 

The query finds the first RecordFile whose `consensusEnd` is **greater than or equal to** the supplied timestamp. It does **not** verify that `consensusStart <= timestamp`, i.e., that the timestamp actually falls within the returned file's range.

**Root cause / failed assumption:**

The design assumes that every valid consensus timestamp will always be contained within some surviving RecordFile (`consensusStart <= T <= consensusEnd`). After pruning, this assumption breaks: a timestamp T that was the `consensusEnd` of pruned file A now satisfies `consensusEnd >= T` for the next surviving file B, even though T was never part of file B.

Pruning is performed via:

```java
@Query("delete from RecordFile where consensusEnd <= ?1")
int prune(long consensusTimestamp);
``` [2](#0-1) 

This deletes all RecordFiles with `consensusEnd <= pruneTimestamp`, creating a gap in the timeline.

**Exploit flow:**

`RecordFileServiceImpl.findByTimestamp` is a zero-validation pass-through:

```java
@Override
public Optional<RecordFile> findByTimestamp(Long timestamp) {
    return recordFileRepository.findByTimestamp(timestamp);
}
``` [3](#0-2) 

The result is consumed in `OpcodeServiceImpl.buildCallServiceParameters` to determine the block context for EVM replay:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);
``` [4](#0-3) 

The `consensusTimestamp` here is derived from a `ContractTransactionHash` or `Transaction` lookup — both of which are user-supplied (transaction hash or transaction ID). If the transaction record still exists in the DB but its RecordFile has been pruned, `findByTimestamp` silently returns the next surviving RecordFile.

**Why existing checks are insufficient:**

There are no bounds checks anywhere in the call chain. `RecordFileServiceImpl` does not validate that the returned RecordFile's `consensusStart <= timestamp <= consensusEnd`. The test suite (`RecordFileRepositoryTest.findByTimestamp`) only tests the happy path where the timestamp equals `consensusEnd` of an existing file and never tests the post-pruning scenario. [5](#0-4) 

### Impact Explanation

When a pruned-era transaction is queried for opcode debugging, the EVM replay executes under the wrong block's state context (wrong block number, wrong block hash, wrong gas parameters). This misrepresents Hashgraph history: a transaction that occurred in block N is replayed as if it occurred in block M (M > N). For smart-contract forensics and audit tools relying on the mirror node's opcode endpoint, this produces silently incorrect results — wrong storage snapshots, wrong block-context opcodes — without any error signal to the caller.

### Likelihood Explanation

Pruning is a standard, documented operational procedure. Any user who knows (or can discover) the consensus timestamp of a transaction whose RecordFile has been pruned can trigger this by calling the opcode debug endpoint with that transaction's hash or ID. No privileges are required; the transaction hash is publicly observable on-chain. The condition is reliably reproducible on any mirror node instance that has performed pruning and retains transaction records beyond the pruning horizon.

### Recommendation

Add a lower-bound guard to the JPQL query so that the returned RecordFile is verified to actually contain the timestamp:

```java
@Query("select r from RecordFile r where r.consensusStart <= ?1 and r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1")
Optional<RecordFile> findByTimestamp(long timestamp);
```

Additionally, `RecordFileServiceImpl.findByTimestamp` should validate the result:

```java
@Override
public Optional<RecordFile> findByTimestamp(Long timestamp) {
    return recordFileRepository.findByTimestamp(timestamp)
        .filter(rf -> rf.getConsensusStart() <= timestamp && rf.getConsensusEnd() >= timestamp);
}
```

This ensures that a pruned-era timestamp returns `Optional.empty()` rather than a wrong RecordFile, allowing callers to handle the missing-history case explicitly.

### Proof of Concept

1. Insert two RecordFiles:
   - File A: `consensusStart=100`, `consensusEnd=200`, `index=5`
   - File B: `consensusStart=201`, `consensusEnd=300`, `index=6`
2. Insert a transaction with `consensusTimestamp=200` (belongs to File A).
3. Prune: call `prune(200)` — this deletes File A (`consensusEnd <= 200`).
4. Call the opcode debug endpoint with the transaction hash whose `consensusTimestamp=200`.
5. Internally, `findByTimestamp(200)` executes `consensusEnd >= 200 ORDER BY consensusEnd ASC LIMIT 1` and returns File B (`consensusEnd=300`).
6. The EVM replay runs under block index 6 instead of the correct block index 5, misattributing the transaction to the wrong block and using the wrong historical state context.
