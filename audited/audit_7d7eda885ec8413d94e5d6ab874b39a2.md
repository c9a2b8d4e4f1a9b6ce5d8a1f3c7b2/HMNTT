### Title
Cache-Poisoning via Non-Existent Slot Keys Causes Unbounded IN-Clause Amplification in `findStorageBatch()`, Enabling DB Load Amplification Against Fee-Oracle Contract

### Summary
`ContractStateServiceImpl.findStorageBatch()` unconditionally inserts every requested slot key into a per-contract Caffeine cache and then issues a single `slot IN (:slots)` query containing **all** keys accumulated in that cache. Because non-existent slot keys are never promoted to the `contractStateCache` (only found slots are stored there), they remain in the per-contract slot cache for up to 5 minutes and are re-included in every subsequent batch query for that contract. An unprivileged attacker can fill the per-contract cache with up to 1,500 phantom slot keys, causing every legitimate fee-oracle storage read to issue a 1,500-key IN-clause query instead of a single-key lookup, amplifying DB load by up to 1,500× and potentially delaying fee-oracle responses enough to trigger EVM timeouts and fallback fee values.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (line 69) delegates to `findStorageBatch()` whenever the requested key is absent from `contractStateCache`.

Inside `findStorageBatch()`:

```
Line 90: contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
Line 91: final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
Line 93-101: all cached keys are collected into cachedSlots
Line 103: contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
``` [1](#0-0) 

The repository query is:

```sql
select slot, value from contract_state
where contract_id = :contractId
and slot in (:slots)
``` [2](#0-1) 

**Root cause — failed assumption:** The design assumes that slot keys accumulate in the per-contract cache only for slots that actually exist in the DB (so the batch query is a useful prefetch). The failed assumption is that non-existent slot keys are silently discarded. In reality:

- Line 90 adds **any** key to the per-contract cache unconditionally.
- Lines 106–113 only write **found** slots into `contractStateCache`. Non-existent slots are never written there. [3](#0-2) 

- Therefore, on the next call to `findStorage()` for a non-existent slot, `contractStateCache.get()` returns `null`, `findStorageBatch()` is called again, and the phantom key is re-included in the IN clause. [4](#0-3) 

**Cache size bounds:**

The per-contract slot cache (`slotsPerContract`) is configured with `maximumSize=1500`, so the IN clause is bounded at 1,500 keys. The outer contract-slots cache (`contractSlots`) allows up to 3,000 contracts. [5](#0-4) 

The `contractState` cache (where found values live) expires after `expireAfterWrite=2s`, meaning every 2 seconds the cached fee-oracle slot value expires and `findStorageBatch()` is invoked again with the full 1,500-key IN clause. [6](#0-5) 

**Exploit flow:**

1. Attacker identifies the fee-oracle contract address (public on-chain).
2. Attacker sends 1,500 `eth_call` / `POST /api/v1/contracts/call` requests, each reading a different non-existent storage slot of the fee-oracle contract (e.g., slot keys `0x0000…0001` through `0x0000…05DC`). Each request passes the global rate limiter (500 req/s default), so the cache is filled in ~3 seconds.
3. Per-contract slot cache is now full with 1,500 phantom keys.
4. Every 2 seconds, when the `contractState` cache expires, any legitimate read of the fee-oracle's actual slot triggers `findStorageBatch()` with a 1,500-key IN clause.
5. Attacker keeps phantom keys alive by re-requesting them every <5 minutes (resetting `expireAfterAccess`), sustaining the attack indefinitely.
6. Under DB load, the 1,500-key IN query delays the fee-oracle response; EVM times out and uses a fallback fee value outside design parameters.

**Why existing checks are insufficient:**

The global rate limiter (`requestsPerSecond = 500`) is not per-IP or per-contract; it limits total throughput but does not prevent a single attacker from spending their budget to poison one contract's cache. [7](#0-6) 

There is no validation, deduplication, or cap on how many unique non-existent slot keys can accumulate per contract in the per-contract cache.

### Impact Explanation

Every legitimate fee-oracle storage read is amplified into a 1,500-key IN-clause DB query (1,500× amplification). The `contractState` cache TTL of 2 seconds means this amplified query fires repeatedly. Under sustained DB load, fee-oracle reads are delayed, causing EVM execution to time out and fall back to a default/stale fee value. This constitutes modification of transaction fees outside design parameters — a medium-severity protocol-integrity issue.

### Likelihood Explanation

The attack requires no privileges, no on-chain funds, and no special knowledge beyond the fee-oracle contract address (which is public). The attacker needs only an HTTP client capable of sending ~1,500 POST requests in ~3 seconds. The attack is repeatable and can be sustained indefinitely with periodic keep-alive requests. The only friction is the global 500 req/s rate limit, which is easily saturated by a single attacker.

### Recommendation

1. **Cap the IN-clause size**: Before calling `findStorageBatch`, limit `cachedSlots` to a small maximum (e.g., 50–100 keys). If the current key is not in the truncated list, fall back to `findStorage` directly.
2. **Evict non-existent slots**: After `findStorageBatch` returns, remove from the per-contract slot cache any key that was not found in the DB (i.e., not present in `contractSlotValues`). This prevents phantom keys from accumulating.
3. **Per-contract rate limiting**: Apply a per-contract (or per-IP) rate limit on unique slot key submissions to prevent cache-filling attacks.
4. **Store negative results**: Cache a sentinel value for non-existent slots in `contractStateCache` so repeated lookups for the same non-existent key short-circuit before reaching `findStorageBatch`.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
