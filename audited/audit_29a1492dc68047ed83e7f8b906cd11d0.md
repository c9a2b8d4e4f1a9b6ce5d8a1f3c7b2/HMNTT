### Title
Unbounded Timestamp Range in `getHookStorageChange()` Enables Unauthenticated Database DoS via Full Table Scan

### Summary
The `getHookStorageChange()` method in `HookServiceImpl` passes attacker-controlled timestamp bounds directly to a native SQL query that uses `DISTINCT ON (key)`, which requires PostgreSQL to scan every matching row in `hook_storage_change` before applying `LIMIT`. Because no timestamp range span validation exists anywhere in the call chain, an unauthenticated user can supply a single lower-bound timestamp (e.g., `timestamp=gte:1`) causing the upper bound to default to `Long.MAX_VALUE`, forcing a full table scan across all historical storage changes for the targeted hook. Repeated concurrent requests exhaust database CPU, I/O, and connection pool resources, resulting in a denial-of-service against the mirror node.

### Finding Description

**Exact code path:**

`HooksController.getHookStorage()` accepts `TimestampParameter[] timestamps` with only `@Size(max = 2)` validation — no range span check. [1](#0-0) 

`hookStorageChangeRequest()` calls `Bound.of(timestamps, ...)` which performs no range-width validation. [2](#0-1) 

In `Bound`, when only a lower bound is supplied (e.g., `gte:1`), `adjustUpperBound()` returns `Long.MAX_VALUE` because `this.upper == null`. [3](#0-2) 

`getHookStorageChange()` uses these bounds verbatim — no range-size guard exists between lines 97–114. [4](#0-3) 

The native query uses `DISTINCT ON (key)` with `consensus_timestamp between :timestampLowerBound and :timestampUpperBound`. PostgreSQL must scan **all** rows matching `owner_id`, `hook_id`, and key range within the timestamp window to compute the distinct-per-key result before `LIMIT` is applied. [5](#0-4) 

The table's primary key is `(owner_id, hook_id, key, consensus_timestamp)`. With `ORDER BY key ASC, consensus_timestamp DESC` (injected by `Pageable`), PostgreSQL must visit every row per key within the timestamp range to find the most recent value — O(N) over all historical changes for that hook. [6](#0-5) 

**Root cause:** The failed assumption is that a `LIMIT` on the result set bounds the database work. `DISTINCT ON` is evaluated before `LIMIT`; the planner cannot short-circuit without a covering index on `(owner_id, hook_id, key, consensus_timestamp DESC)` that also satisfies the timestamp filter, and no such secondary index is defined.

**Why existing checks fail:**
- `@Size(max = 2)` only limits the number of timestamp parameters, not the span between them.
- `Bound.of()` converts `EQ` to `LTE` but adds no span guard.
- `isHistorical()` only checks whether any timestamp was provided — a single `gte:1` satisfies it and routes to the expensive path. [7](#0-6) 

### Impact Explanation

An attacker can force the PostgreSQL backend to perform unbounded sequential scans of the `hook_storage_change` table (which is time-partitioned but not range-limited per query). Each such query holds a database connection and consumes CPU proportional to the total number of historical storage changes for the targeted hook. With a high-activity hook and concurrent flood of requests, this exhausts the DB connection pool and CPU, making the mirror node unresponsive to all clients. The `hook_storage_change` table grows continuously as the network processes transactions, so impact worsens over time. Severity: **High** (availability impact, no authentication required, no per-IP rate limiting visible in the codebase).

### Likelihood Explanation

The endpoint is public and unauthenticated. The exploit requires only a single HTTP GET with one query parameter (`timestamp=gte:1`). No special knowledge, credentials, or protocol access is needed. The attack is trivially repeatable and scriptable. Any external user who discovers the API (documented in `docs/design/hiero-hooks.md`) can execute it immediately.

### Recommendation

1. **Enforce a maximum timestamp range span** in `hookStorageChangeRequest()` or `getHookStorageChange()`: reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configured maximum (e.g., 7 days in nanoseconds).
2. **Require both bounds** when the historical path is used: reject requests that supply only a lower or only an upper timestamp bound without the other.
3. **Add a secondary index** on `(owner_id, hook_id, key, consensus_timestamp DESC)` to allow the planner to satisfy `DISTINCT ON (key) ORDER BY key, consensus_timestamp DESC LIMIT N` via an index-only scan, reducing the scan to O(distinct keys × average versions per key in range) rather than O(all rows in range).
4. **Add API-level rate limiting** on the historical storage endpoint.

### Proof of Concept

**Preconditions:** Mirror node is running with at least one hook that has accumulated historical storage changes.

**Step 1 — Trigger the unbounded scan:**
```
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:1
```
This sets `timestampLowerBound = 1` and `timestampUpperBound = Long.MAX_VALUE` (9223372036854775807), causing the query:
```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = 123
  AND hook_id = 1
  AND key >= '\x0000...00'
  AND key <= '\xffff...ff'
  AND consensus_timestamp BETWEEN 1 AND 9223372036854775807
ORDER BY key ASC, consensus_timestamp DESC
LIMIT 100
```

**Step 2 — Amplify with concurrency:**
```bash
for i in $(seq 1 200); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:1" &
done
wait
```

**Expected result:** Database CPU spikes to 100%, connection pool exhausts, subsequent API requests (including unrelated endpoints) time out or return 503.
