### Title
Unbounded Bytecode Fetch via Unauthenticated EVM Address Lookup Enables Non-Network DoS

### Summary
Any unauthenticated user can supply a valid EVM address to `GET /contracts/:contractId`, triggering an uncached, unbounded `string_agg` SQL query that concatenates all file-data chunks for the contract's bytecode file into a single in-memory buffer. Because there is no rate limiting, no response-size cap, and no caching on the `getFileData` path, an attacker who knows (or enumerates) the EVM address of a large-bytecode contract can flood the service with requests that each impose significant DB CPU, I/O, and Node.js heap pressure, degrading availability for all users.

### Finding Description

**Route registration** — `rest/routes/contractRoute.js` line 13:
```js
router.getExt('/:contractId', ContractController.getContractById);
```
No authentication middleware is applied.

**Input validation** — `rest/controllers/contractController.js` lines 330–338 (`validateContractIdParam`):
```js
if (EntityId.isValidEvmAddress(contractId)) {
  return;          // ← EVM address accepted, no further checks
}
```
A 40-hex-char EVM address passes validation unconditionally.

**Handler** — `rest/controllers/contractController.js` lines 707–737 (`getContractById`):
```js
const {query, params} = getContractByIdOrAddressContractEntityQuery({...contractIdParam});
const {rows} = await pool.queryQuietly(query, params);   // DB lookup by evm_address
if (rows.length !== 1) throw new NotFoundError();
const contract = rows[0];
if (utils.isValidUserFileId(contract.file_id)) {
  contract.bytecode = await FileDataService.getFileData(  // ← unbounded fetch
    contract.file_id, contract.created_timestamp
  );
}
```
`getContractByIdOrAddressContractEntityQuery` (lines 179–208) builds a SQL query with `evm_address = $N` when the input is an EVM address — no `computeContractIdFromString` indirection is needed; the lookup happens directly.

**Unbounded aggregation** — `rest/service/fileDataService.js` lines 43–75 (`getFileDataQuery` / `getFileData`):
```sql
SELECT string_agg(file_data, '' ORDER BY consensus_timestamp) AS data
FROM file_data
WHERE entity_id = $1
  AND consensus_timestamp >= (
    SELECT consensus_timestamp FROM file_data
    WHERE entity_id = $1 AND consensus_timestamp <= $2
    AND (transaction_type = 17 OR (transaction_type = 19 AND length(file_data) <> 0))
    ORDER BY consensus_timestamp DESC LIMIT 1
  )
  AND consensus_timestamp <= $2
```
`string_agg` concatenates every chunk of the bytecode file into one row with no `LIMIT`, no `MAX(length(...))` guard, and no size cap. The result is returned as a raw `Buffer` and serialised into the JSON response. There is **no caching** in this REST-layer `getFileData` path (unlike the Java importer's `FileDataServiceImpl` which has an explicit Spring cache).

**Why existing checks are insufficient:**
- `validateContractIdParam` only validates address format.
- `isValidUserFileId` only checks that `file_id` is a non-system entity ID; it does not inspect bytecode size.
- The `responseCacheHandler` middleware caches HTTP responses, but only for identical URLs; an attacker using the same EVM address repeatedly would be cached after the first hit, but rotating across multiple large-contract addresses bypasses this entirely.
- No rate limiting is applied at the application layer on this path.

### Impact Explanation
Hedera contracts can have bytecode up to ~1 MB stored across many `file_data` rows. Each request to a large-bytecode contract causes: (1) a correlated subquery + `string_agg` scan on the `file_data` table (DB CPU/I/O), (2) allocation of a large `Buffer` in the Node.js heap, and (3) JSON serialisation of a hex-encoded string of equal size. Concurrent flooding of this endpoint with a handful of known large-contract EVM addresses can exhaust DB connection pool capacity, spike heap usage, and increase GC pressure, degrading or denying service for all users of the mirror-node REST API — a public, shared infrastructure component.

### Likelihood Explanation
The precondition (a valid EVM address of a contract with a large `file_id`-backed bytecode) is trivially satisfied: large contracts are publicly visible on Hedera mainnet and their EVM addresses are enumerable from the same `/contracts` list endpoint. No credentials, tokens, or special network access are required. The attack is fully repeatable and scriptable with a single `curl` loop. A single attacker with a modest HTTP client can sustain the load.

### Recommendation
1. **Add a bytecode size cap**: In `getFileData` (or its SQL query), add a `LIMIT` on the aggregated length, e.g. reject or truncate responses where `length(string_agg(...)) > MAX_BYTECODE_BYTES`.
2. **Cache bytecode at the REST layer**: Introduce an LRU/TTL cache keyed on `(file_id, created_timestamp)` so repeated requests for the same contract do not re-execute the `string_agg` query.
3. **Apply rate limiting**: Add per-IP or global rate limiting middleware to the `/contracts/:contractId` route.
4. **Stream or paginate large responses**: Instead of loading the entire bytecode into memory, consider streaming the DB result directly to the HTTP response.

### Proof of Concept
```bash
# 1. Identify a large-bytecode contract EVM address from the public list endpoint
ADDR=$(curl -s 'https://<mirror-node>/api/v1/contracts?limit=1&order=desc' \
  | jq -r '.contracts[0].evm_address')

# 2. Flood the endpoint with concurrent requests (no auth required)
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/contracts/${ADDR}" -o /dev/null &
done
wait
```
Each concurrent request triggers a full `string_agg` scan of the `file_data` table for the target contract's `file_id` and allocates a large buffer in the Node.js process. Sustained over seconds, this exhausts DB query concurrency and Node.js heap, causing latency spikes and eventual OOM or connection-pool exhaustion for all users.                   and

        order by ${FileData.CONSENSUS_TIMESTAMP} desc
        limit 1
        ) and ${FileData.CONSENSUS_TIMESTAMP} <= $2`;

  /**
   * The function returns the data for the fileId at the provided consensus timestamp.
   * @param fileId
   * @param timestamp
   * @return {data: string}
   */
  getFileData = async (fileId, timestamp) => {
    const params = [fileId, timestamp];
    const query = FileDataService.getFileDataQuery;
    const row = await super.getSingleRow(query, params);
    return row === null ? null : row.data;
  };
