### Title
Missing Account Existence Validation in Token Allowance Endpoint Enables DB Connection Pool Exhaustion (DoS)

### Summary
`getAccountTokenAllowances()` in `rest/controllers/tokenAllowanceController.js` omits the `EntityService.isValidAccount()` guard present in `tokenController.js`, allowing any unauthenticated caller to trigger a `token_allowance` table query for arbitrary non-existent account IDs. With no rate limiting on the REST API and a default pool ceiling of 10 connections, a modest flood of concurrent requests can exhaust the connection pool and render the mirror-node REST API completely unresponsive. The claimed impact of "halting transaction confirmation / total network shutdown" is inaccurate — the mirror-node REST API is a read-only query service and plays no role in Hedera consensus; the real impact is full DoS of the mirror-node REST API.

### Finding Description

**Code path — missing guard:**

`rest/controllers/tokenAllowanceController.js` lines 68–81:
```js
getAccountTokenAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);
  // ← no isValidAccount() call here
  const query = this.extractTokenMultiUnionQuery(filters, accountId);
  const tokenAllowances = await TokenAllowanceService.getAccountTokenAllowances(query);
  ...
};
```

**Contrast with `tokenController.js` lines 66–71:**
```js
getTokenRelationships = async (req, res) => {
  const accountId = await EntityService.getEncodedId(...);
  const isValidAccount = await EntityService.isValidAccount(accountId); // ← guard
  if (!isValidAccount) throw new NotFoundError();
  ...
};
```

**Why `getEncodedId()` does not protect against this:**
For a plain numeric entity ID (e.g. `0.0.999999999`), `EntityService.getEncodedId()` (`rest/service/entityService.js` lines 118–137) takes the pure-math branch (`entityId.getEncodedId()`) and issues **zero DB queries**. The first DB touch is the `token_allowance` query itself.

**The DB query always fires:**
`TokenAllowanceService.getAccountTokenAllowances()` (`rest/service/tokenAllowanceService.js` lines 86–90) unconditionally executes:
```sql
SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 ...
```
For a non-existent owner this returns empty rows but still acquires and holds a connection for the full query duration.

**Pool size:**
`rest/dbpool.js` lines 7–16 sets `max: config.db.pool.maxConnections`. The documented default is **10 connections** with a `statementTimeout` of **20 000 ms** (`docs/configuration.md` line 556–557).

**No rate limiting on the REST API:**
The REST server middleware stack (`rest/server.js` lines 67–98) includes: urlencoded, json, cors, compression, httpContext, requestLogger, authHandler, metricsHandler, responseCacheCheckHandler — **no rate-limiter**. The bucket4j throttle found in `web3/src/main/java/.../ThrottleConfiguration.java` applies only to the separate web3 Java service, not this Node.js REST service.

### Impact Explanation

An attacker who saturates all 10 pool connections causes every subsequent REST API request that needs a DB connection to queue or time out. This makes the entire mirror-node REST API (accounts, transactions, tokens, balances, etc.) unavailable for the duration of the attack. The mirror-node REST API does **not** participate in Hedera transaction confirmation (consensus is handled by network nodes), so "total network shutdown" is not achievable; the actual impact is **complete DoS of the mirror-node REST API**.

### Likelihood Explanation

No authentication, no rate limiting, no CAPTCHA, and no IP-based throttle protect this endpoint. Any internet-accessible deployment is reachable. Sending 10–20 concurrent HTTP requests in a tight loop is trivially scriptable with `curl`, `ab`, or any HTTP client. The attacker needs only a valid-format but non-existent account ID (e.g. `0.0.9999999999`), which passes `EntityService.getEncodedId()` without a DB round-trip, making the attack cheap and repeatable.

### Recommendation

1. **Add the existence check** — mirror `tokenController.js` by calling `EntityService.isValidAccount(accountId)` immediately after `getEncodedId()` in `getAccountTokenAllowances()` and throwing `NotFoundError` for unknown accounts.
2. **Add REST-layer rate limiting** — introduce a per-IP (or global) rate-limit middleware (e.g. `express-rate-limit`) in `rest/server.js` before route handlers.
3. **Increase pool size or add connection-wait timeout** — raise `maxConnections` and ensure `connectionTimeoutMillis` is low enough to fail fast rather than queue indefinitely under load.

### Proof of Concept

```bash
# Send 20 concurrent requests for a non-existent account
for i in $(seq 1 20); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.9999999999/allowances/tokens" &
done
wait
# All 10 pool connections are held; subsequent legitimate API calls time out or return 500
curl "https://<mirror-node>/api/v1/transactions"  # → connection pool exhausted / timeout

  async getEncodedId(entityIdString, requireResult = true, paramName = filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS) {
    try {
      if (EntityId.isValidEntityId(entityIdString)) {
        const entityId = EntityId.parseString(entityIdString, {paramName});
        return entityId.evmAddress === null
          ? entityId.getEncodedId()
          : await this.getEntityIdFromEvmAddress(entityId, requireResult);
      } else if (AccountAlias.isValid(entityIdString)) {
        return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
      }
    } catch (ex) {
      if (ex instanceof InvalidArgumentError) {
        throw InvalidArgumentError.forParams(paramName);
      }
      // rethrow
      throw ex;
    }

    throw InvalidArgumentError.forParams(paramName);
  }
