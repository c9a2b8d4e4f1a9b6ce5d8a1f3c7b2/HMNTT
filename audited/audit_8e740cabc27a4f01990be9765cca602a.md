### Title
Unbounded Full-Account NFT Scan via Unrestricted `spender.id=eq` IN-Clause Without Token/Serial Index Restriction

### Summary
In `rest/service/nftService.js`, when `getQuery()` is called with empty `lower`, `inner`, and `upper` filter arrays (no `token_id` or `serial_number` bounds), it falls into the `subQueries.length === 0` branch and calls `getSubQuery()` with an empty `filters` array but passes `spenderIdInFilters` through unchanged. An unprivileged attacker can supply up to `maxRepeatedQueryParameters` (default: 100) `spender.id=eq` values with no `token_id` or `serial_number` constraints, generating a query of the form `WHERE account_id = $1 AND spender IN ($3,...,$102)` that must perform a full sequential scan of all NFTs owned by the target account. No authentication is required and the endpoint is publicly accessible.

### Finding Description

**Exact code path:**

In `rest/service/nftService.js` lines 93–118: [1](#0-0) 

When `lower`, `inner`, and `upper` are all empty arrays (no `token_id` or `serial_number` filters provided), the `.filter((filters) => filters.length !== 0)` at line 94 produces an empty `subQueries` array. The `subQueries.length === 0` branch at line 108 then calls `getSubQuery([], params, ..., spenderIdInFilters, spenderIdFilters)`.

In `getSubQuery()` (lines 55–76): [2](#0-1) 

With `filters = []` and a non-empty `spenderIdInFilters`, the generated SQL is:
```sql
SELECT ... FROM nft LEFT JOIN entity e ON e.id = nft.token_id
WHERE account_id = $1 AND spender IN ($3,$4,...,$N)
ORDER BY token_id desc, serial_number desc
LIMIT $2
```
There is no `token_id` or `serial_number` predicate, so the database must scan **all NFTs for the account** to evaluate the `spender IN (...)` filter.

**Root cause — failed assumption in `validateFilters()`:**

`validateFilters()` in `rest/controllers/accountController.js` (lines 18–32) only validates `spenderIdFilters` (range operators: lt, lte, gt, gte, ne): [3](#0-2) 

It performs **zero validation on `spenderIdInFilters`** (eq operators). There is no check requiring at least one `token_id` or `serial_number` filter when `spenderIdInFilters` is non-empty, and no cap on the count of eq-operator spender filters beyond the global `maxRepeatedQueryParameters` default of 100: [4](#0-3) [5](#0-4) 

The `maxRepeatedQueryParameters` default is 100 (documented in `docs/configuration.md` line 581), meaning an attacker can legally supply up to 100 `spender.id=eq` values in a single request.

### Impact Explanation

The generated query `WHERE account_id = $1 AND spender IN ($3,...,$102)` has no selectivity on the composite `(token_id, serial_number)` index. For any account holding a large number of NFTs (e.g., a marketplace or high-volume account), the database must scan the entire per-account NFT set before applying the spender filter and the `LIMIT`. Repeated concurrent requests from one or more attackers can saturate the database connection pool and CPU, causing degraded response times or outright unavailability for all users of the mirror node REST API. The `LIMIT $2` clause (max 100 rows returned) does **not** prevent the full scan — it only limits the output after the scan completes.

### Likelihood Explanation

The endpoint `GET /api/v1/accounts/{id}/nfts` is publicly accessible with no authentication: [6](#0-5) 

Any external user can craft the request. The attack requires no special knowledge beyond knowing a target account ID with many NFTs (publicly discoverable via the same API). The attack is trivially repeatable and scriptable. The only mitigations in place — `maxRepeatedQueryParameters` (100) and DB `statementTimeout` — reduce but do not eliminate the impact, especially under concurrent load.

### Recommendation

1. **Require at least one `token_id` or `serial_number` filter when `spenderIdInFilters` is non-empty** — add a check in `validateFilters()` (or `extractNftMultiUnionQuery()`) that throws `InvalidArgumentError` if `spenderIdInFilters.length > 0` and both `bounds.primary` and `bounds.secondary` are unbounded.
2. **Cap the number of `spender.id=eq` filters independently** — add a dedicated check (e.g., max 10) for `spenderIdInFilters.length` in `validateFilters()`, separate from the global `maxRepeatedQueryParameters`.
3. **Ensure a DB index exists on `(account_id, spender)`** to make the spender-only filter path efficient if it must be supported.
4. **Consider rate-limiting** the `/accounts/{id}/nfts` endpoint at the API gateway level.

### Proof of Concept

```
# Target: any account with a large NFT holding (e.g., 0.0.1234567)
# No authentication required

GET /api/v1/accounts/0.0.1234567/nfts?\
  spender.id=1&spender.id=2&spender.id=3&...&spender.id=100
```

This triggers the `subQueries.length === 0` path in `getQuery()` (no `token.id` or `serialnumber` params supplied), and `getSubQuery()` is called with `filters = []` and `spenderIdInFilters` containing 100 entries. The resulting SQL performs a full scan of all NFTs for account `0.0.1234567` with no index restriction on `token_id` or `serial_number`. Sending this request concurrently (e.g., 50 parallel connections) amplifies the DB load proportionally.
