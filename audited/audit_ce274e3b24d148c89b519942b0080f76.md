### Title
Uncached Negative EVM Address DB Lookups in `getEncodedId()` Enable Unauthenticated Non-Network DoS

### Summary
`EntityService.getEncodedId()` routes any input matching `evmAddressShardRealmRegex` (including bare 40-hex-char strings and realm-prefixed variants) to a live database query via `getEntityIdFromEvmAddress()`. The `parseCached` LRU only caches the parsed `EntityId` object (shard/realm/num/evmAddress fields), never the database lookup result. An unauthenticated attacker who submits a stream of unique, non-existent EVM addresses forces one unbounded DB query per request with no negative-result caching and no rate limiting in the code path.

### Finding Description

**Exact code path:**

`rest/service/entityService.js`, `getEncodedId()`, lines 118–137:

```js
async getEncodedId(entityIdString, requireResult = true, ...) {
  if (EntityId.isValidEntityId(entityIdString)) {          // line 120
    const entityId = EntityId.parseString(entityIdString, {paramName}); // line 121
    return entityId.evmAddress === null
      ? entityId.getEncodedId()
      : await this.getEntityIdFromEvmAddress(entityId, requireResult); // line 124 — DB hit
  }
  ...
}
```

`rest/service/entityService.js`, `getEntityIdFromEvmAddress()`, lines 90–104:

```js
async getEntityIdFromEvmAddress(entityId, requireResult = true) {
  const rows = await this.getRows(EntityService.entityFromEvmAddressQuery,
    [Buffer.from(entityId.evmAddress, 'hex')]);   // raw DB query, no cache
  if (rows.length === 0) {
    if (requireResult) throw new NotFoundError();
    return null;
  }
  return rows[0].id;
}
```

**Root cause — failed assumption:**

`rest/entityId.js`, `parseCached()`, lines 314–333 caches only the **parsed `EntityId` JS object** (the result of interpreting the string into shard/realm/num/evmAddress fields):

```js
const entityId = of(shard, realm, num, evmAddress);
cache.set(key, entityId);   // caches parse result only
return entityId;
```

When `evmAddress` is non-null (opaque EVM address), `getEncodedId` must still call `getEntityIdFromEvmAddress` — a live DB query — on every single request. The cache never stores "this address was not found in the DB." The failed assumption is that caching the parse step is sufficient; it is not, because the expensive operation is the DB lookup, not the string parse.

**Regex coverage:**

`rest/entityId.js` line 39:
```js
const evmAddressShardRealmRegex = /^(\d{1,4}\.)?(\d{1,5}\.)?[A-Fa-f0-9]{40}$/;
```

This matches bare 40-hex strings (`aabbcc...`), realm-prefixed (`0.aabbcc...`), and shard+realm-prefixed (`0.0.aabbcc...`). All three forms pass `isValidEntityId` → `parseString` → `parseCached` → `getEntityIdFromEvmAddress` → DB query.

**Why existing checks are insufficient:**

- `parseCached` LRU (`rest/entityId.js` lines 301–333): caches parse result, not DB result. Even repeated identical addresses still hit the DB on every call.
- No rate limiting found in the REST service layer (`rest/**/*.js` search returned only a test utility reference).
- `entityFromEvmAddressQuery` (`entityService.js` lines 22–25) runs `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` — a full DB round-trip per request.

### Impact Explanation
Every HTTP request carrying a unique (or even repeated) non-existent EVM address causes one synchronous DB query. With 2¹⁶⁰ possible EVM addresses, the attacker's input space is effectively unlimited, making cache saturation impossible. Sustained high-rate requests exhaust the DB connection pool and degrade or deny service to all legitimate users of the mirror node REST API. Because the mirror node is read-only infrastructure relied upon by wallets, explorers, and dApps, a sustained DoS degrades the entire ecosystem depending on it.

### Likelihood Explanation
No authentication or API key is required. The attacker needs only an HTTP client and the ability to generate hex strings. The attack is trivially scriptable, repeatable indefinitely, and requires no special knowledge of the target system beyond the public API surface. The cost to the attacker is negligible (CPU to generate hex strings); the cost to the defender is a DB query per request.

### Recommendation
Cache negative DB lookup results for EVM addresses. Introduce a second cache layer in `getEntityIdFromEvmAddress` (or in `getEncodedId`) that stores a sentinel value (e.g., `null` or a dedicated `NOT_FOUND` symbol) for addresses confirmed absent, with a short TTL (e.g., 30–60 seconds). This prevents repeated DB hits for the same non-existent address. Additionally, apply per-IP or global rate limiting on endpoints that invoke `getEncodedId` to bound the rate of unique-address probing.

### Proof of Concept

```bash
# Generate N unique random EVM addresses and hammer the accounts endpoint
for i in $(seq 1 10000); do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/accounts/${ADDR}" &
done
wait
```

Each request matches `evmAddressShardRealmRegex`, passes `isValidEntityId`, is parsed as an opaque EVM address by `parseCached` (cache miss for unique addresses), and triggers `getEntityIdFromEvmAddress` → `SELECT id FROM entity WHERE evm_address = $1`. With 10 000 concurrent requests, 10 000 DB queries are issued simultaneously. Repeating the loop exhausts the DB connection pool and causes query queuing/timeout for legitimate traffic. Even with repeated addresses, the DB is still queried on every call because only the parse result is cached, not the lookup result.
