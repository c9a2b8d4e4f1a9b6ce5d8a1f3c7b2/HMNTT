### Title
Unbounded Historical Replay via Uncapped `limit` and Unrestricted `startTime` in `subscribeTopic()`

### Summary
Any unauthenticated caller can submit a `subscribeTopic` gRPC request with `limit = Long.MAX_VALUE` and `consensusStartTime` at epoch zero. The throttled `PollingTopicMessageRetriever` will then page through the entire topic message history (1 000 rows every 2 seconds) before the live Flux is ever started, holding a DB polling loop open for an arbitrarily long period. Repeating this across many connections exhausts DB connection-pool slots and bounded-elastic scheduler threads with no authentication barrier.

### Finding Description

**Code path:**

1. `ConsensusController.toFilter()` copies `query.getLimit()` directly into the filter with no upper-bound check: [1](#0-0) 

2. `TopicMessageFilter.limit` is annotated only `@Min(0)` — no maximum is enforced: [2](#0-1) 

3. `isValidStartTime()` only requires `startTime <= now()`, so `startTime = 0` (Unix epoch) is accepted: [3](#0-2) 

4. `TopicMessageServiceImpl.subscribeTopic()` builds the pipeline as `historical.concatWith(live)` — the live Flux is deferred until the historical Flux completes: [4](#0-3) 

5. `PollingTopicMessageRetriever.retrieve()` in throttled mode sets `numRepeats = Long.MAX_VALUE`, polls every 2 s, and fetches up to 1 000 rows per page: [5](#0-4) 

6. `isComplete()` in throttled mode terminates only when a page returns fewer than `maxPageSize` rows **or** when `total == filter.getLimit()`. With `limit = Long.MAX_VALUE`, `total` can never equal `Long.MAX_VALUE`, so the only termination condition is reaching the natural end of historical data: [6](#0-5) 

7. The only timeout applied is a per-emission idle timeout of 60 s. While messages are flowing continuously (which they are during a large historical replay), this timeout never fires: [7](#0-6) 

**Root cause:** No upper bound on `limit`, no minimum age restriction on `startTime`, no total-stream wall-clock timeout, and no per-subscriber rate limit.

### Impact Explanation

For a topic with N historical messages, each malicious stream holds a DB polling loop open for approximately `(N / 1000) × 2` seconds. With 1 million messages that is ~33 minutes per stream. The `maxConcurrentCallsPerConnection = 5` limit applies only per TCP connection: [8](#0-7) 

An attacker opening many TCP connections multiplies this linearly. Each active stream occupies a `boundedElastic` scheduler thread and a DB connection for the full duration, leading to DB connection-pool exhaustion and degraded or denied service for legitimate subscribers. No economic damage occurs on-chain, but the mirror-node gRPC service becomes unavailable.

### Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC port (default 5600) is publicly reachable. The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and knowledge of any existing topic ID. It is trivially repeatable and scriptable. A single attacker with modest bandwidth can sustain dozens of concurrent abusive streams.

### Recommendation

1. **Cap `limit`**: Enforce a configurable maximum (e.g., 10 000) in `TopicMessageFilter` with a `@Max` constraint and reject requests that exceed it in `toFilter()`.
2. **Restrict `startTime` age**: Reject `startTime` values older than a configurable horizon (e.g., 30 days) unless the request also sets a bounded `endTime`.
3. **Add a total stream wall-clock timeout**: Apply a `timeout(maxStreamDuration)` on the entire `flux` in `TopicMessageServiceImpl.subscribeTopic()`, independent of per-emission idle time.
4. **Limit total concurrent streams per remote IP**: Track and cap active subscriptions per source address at the gRPC interceptor layer.

### Proof of Concept

```bash
# Using grpcurl against a mirror node with a high-volume topic
grpcurl -plaintext \
  -proto consensus_service.proto \
  -d '{
    "topicID": {"topicNum": 1234},
    "consensusStartTime": {"seconds": 0, "nanos": 0},
    "limit": 9223372036854775807
  }' \
  mirror-node-host:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

Repeat this command across 10+ parallel connections. Observe:
- Each stream begins paging through the full topic history at 1 000 rows / 2 s.
- DB connection-pool metrics show sustained saturation.
- Legitimate subscribers experience increased latency or `RESOURCE_EXHAUSTED` errors.
- Streams remain open for tens of minutes with no server-side termination.
