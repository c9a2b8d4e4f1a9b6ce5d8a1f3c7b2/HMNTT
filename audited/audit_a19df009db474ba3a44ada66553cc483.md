### Title
Integer Overflow in `EntityIdRangeParameter.getInclusiveValue()` Causes Filter Bypass via Crafted GT/LT Entity ID

### Summary
`EntityIdRangeParameter.getInclusiveValue()` performs unchecked Java `long` arithmetic (`value + 1` for GT, `value - 1` for LT) on the raw encoded entity ID. Because the encoding scheme maps `511.65535.274877906943` to `Long.MAX_VALUE` and `512.0.0` to `Long.MIN_VALUE`, an attacker can supply a valid entity ID that causes the arithmetic to wrap, producing a completely wrong query bound. The `Bound` constructor's range-validity check fails to catch this because the overflowed value compares in the wrong direction.

### Finding Description

**Code locations:**

`EntityIdRangeParameter.getInclusiveValue()` — no overflow guard: [1](#0-0) 

`Bound.getAdjustedLowerRangeValue()` — same unchecked `lowerBound++`: [2](#0-1) 

`Bound.getInclusiveRangeValue()` — same unchecked `value += 1L`: [3](#0-2) 

**Encoding boundary facts** (confirmed by unit tests):

- `511.65535.274877906943` encodes to `Long.MAX_VALUE` (9223372036854775807) — a fully valid entity ID per `encode()`'s own validation.
- `512.0.0` encodes to `Long.MIN_VALUE` (-9223372036854775808) — also fully valid. [4](#0-3) 

**Exploit flow (GT case):**

1. Attacker sends: `GET /api/v1/accounts/0.0.1000/allowances/nfts?account.id=gt:511.65535.274877906943`
2. `getEntityId("511.65535.274877906943")` → `EntityId.of(511, 65535, 274877906943)` → `id = Long.MAX_VALUE`. Passes all validation in `encode()`.
3. `EntityIdRangeParameter(GT, Long.MAX_VALUE)` is stored.
4. `Bound` constructor calls `getAdjustedLowerRangeValue()` → `Long.MAX_VALUE + 1 = Long.MIN_VALUE` (silent Java long overflow).
5. Range validity check: `Long.MIN_VALUE > Long.MAX_VALUE` → **false** → no exception thrown, overflow is silently accepted. [5](#0-4) 
6. The query executes with lower bound `Long.MIN_VALUE`, i.e., `WHERE spender_id >= -9223372036854775808`, which matches **every row** in the table.

**LT mirror case:** `account.id=lt:512.0.0` → `value = Long.MIN_VALUE` → `value - 1 = Long.MAX_VALUE` → `WHERE spender_id <= 9223372036854775807` → matches every row.

**Why the existing check is insufficient:**

The `Bound` constructor checks `adjustedLower > adjustedUpper` only for `primarySortField = true`. After overflow, `adjustedLower = Long.MIN_VALUE` and `adjustedUpper = Long.MAX_VALUE` (no upper bound supplied), so `Long.MIN_VALUE > Long.MAX_VALUE` is `false` and the check passes silently. [5](#0-4) 

No overflow guard exists anywhere in `getInclusiveValue()`, `getAdjustedLowerRangeValue()`, or `getInclusiveRangeValue()`. [1](#0-0) 

### Impact Explanation

The `account.id` filter on `getNftAllowances` is completely bypassed: the query returns all NFT allowances for the target account regardless of the intended spender/owner filter. Because this is a public read API, the data is already accessible without the filter, so there is no direct privilege escalation. However, the incorrect bounds can also affect pagination cursor logic (the `links.next` URL is generated from the last returned item), potentially causing infinite pagination loops or exposing unintended cursor values. Severity: **Medium** — logic correctness violation with filter bypass on a public endpoint.

### Likelihood Explanation

Any unauthenticated HTTP client can trigger this with a single crafted GET request. The entity ID `511.65535.274877906943` is a valid, parseable value that passes all input validation. No special privileges, tokens, or prior knowledge of the system state are required. The attack is trivially repeatable and requires no brute-forcing.

### Recommendation

Replace the unchecked arithmetic in all three locations with overflow-safe equivalents:

```java
// EntityIdRangeParameter.getInclusiveValue()
public long getInclusiveValue() {
    return switch (operator) {
        case GT -> Math.addExact(value, 1);   // throws ArithmeticException on overflow
        case LT -> Math.subtractExact(value, 1);
        default -> value;
    };
}
```

Apply the same fix to `Bound.getAdjustedLowerRangeValue()`, `Bound.adjustUpperBound()`, and `Bound.getInclusiveRangeValue()`. Catch `ArithmeticException` at the controller/service boundary and return HTTP 400. Alternatively, add an explicit pre-check: reject any GT input whose encoded value equals `Long.MAX_VALUE`, and any LT input whose encoded value equals `Long.MIN_VALUE`.

### Proof of Concept

```
# GT overflow — lower bound wraps to Long.MIN_VALUE, returns all allowances
GET /api/v1/accounts/0.0.1000/allowances/nfts?account.id=gt:511.65535.274877906943

# LT overflow — upper bound wraps to Long.MAX_VALUE, returns all allowances
GET /api/v1/accounts/0.0.1000/allowances/nfts?account.id=lt:512.0.0&token.id=lte:0.0.1
```

Expected: 0 results (no entity ID is greater than `511.65535.274877906943` in the GT case).
Actual: All NFT allowances for account `0.0.1000` are returned (up to `limit`), because the overflowed bound `Long.MIN_VALUE` is used as `WHERE spender_id >= -9223372036854775808`.
