### Title
Unauthenticated Connection Pool Exhaustion via Unthrottled `/api/v1/network/registered-nodes` EQ Operator Path

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module is publicly accessible with no rate limiting or per-IP throttling. Each request — including those using the `registerednode.id=eq:N` single-value EQ path — unconditionally acquires a HikariCP database connection. An unprivileged attacker flooding the endpoint with concurrent requests can exhaust the finite connection pool, causing all subsequent legitimate requests to block or fail waiting for a connection.

### Finding Description

**Exact code path:**

`NetworkController.getRegisteredNodes()` (line 174) calls `NetworkServiceImpl.getRegisteredNodes()` (line 140), which calls `resolveRegisteredNodeIdBounds()` (line 154). When the EQ operator is used (`registerednode.id=eq:N`), the method returns `Range.closed(N, N)` at line 163 without any guard or short-circuit:

```java
// NetworkServiceImpl.java lines 159-163
if (range.operator() == RangeOperator.EQ) {
    if (registeredNodeIdRanges.size() > 1) {
        throw new IllegalArgumentException("...");
    }
    return Range.closed(range.value(), range.value());
}
```

This immediately flows into `registeredNodeRepository.findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs(N, N, type, page)` (line 150), which executes:

```sql
SELECT * FROM registered_node
WHERE registered_node_id >= :lowerBound
  AND registered_node_id <= :upperBound
  AND deleted IS FALSE
  AND (:type IS NULL OR type @> array[:type]::smallint[])
```

Each invocation holds a HikariCP connection for the duration of the query.

**Root cause — no rate limiting in `rest-java`:**

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties`) exists exclusively in the `web3` module. The `rest-java` module has no equivalent. The `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter — no rate-limiting filter. `WebMvcConfiguration` adds only argument resolvers. No `Filter`, `HandlerInterceptor`, or Spring Security rule limits request rate to this endpoint.

**Connection pool:**

`CommonConfiguration` wires HikariCP via `@ConfigurationProperties("spring.datasource.hikari")`. No `maximumPoolSize` override is present in any `rest-java` resource file; the HikariCP default is **10 connections**. With 10 or more concurrent in-flight requests each waiting on a DB response, the pool is fully occupied and new requests block at `HikariDataSource.getConnection()` until the configured `connectionTimeout` (default 30 s) elapses, at which point they throw `SQLTransientConnectionException`.

### Impact Explanation

A fully exhausted connection pool causes all requests to the `rest-java` service — not just to this endpoint — to fail with connection-timeout errors, because the pool is shared. This constitutes a complete service-level denial of service for the duration of the attack. The `registered-nodes` endpoint is intended to expose block-node discovery data; unavailability of this endpoint disrupts clients that rely on it for network topology discovery.

### Likelihood Explanation

No authentication, API key, or network-level access control is required. The endpoint is documented in the public OpenAPI spec. An attacker needs only HTTP access and a tool capable of sending concurrent requests (e.g., `ab`, `wrk`, `hey`, or a simple async script). The attack is trivially repeatable and requires no special knowledge beyond the public API documentation. The EQ path is particularly convenient because it produces cheap point-lookup queries that complete quickly, requiring the attacker to sustain only moderate concurrency (≥10 simultaneous in-flight requests) to keep the pool saturated.

### Recommendation

1. **Add rate limiting to `rest-java`**: Implement a `HandlerInterceptor` or servlet `Filter` using Bucket4j (already a dependency in `web3`) to enforce a per-IP or global request-per-second cap on the `/api/v1/network/registered-nodes` endpoint, mirroring the pattern in `ThrottleConfiguration` / `ThrottleManagerImpl`.
2. **Increase pool size or add connection-acquisition timeout handling**: Configure `spring.datasource.hikari.maximumPoolSize` to a value appropriate for expected concurrency, and ensure `connectionTimeout` is set low enough to fail fast rather than queue indefinitely.
3. **Consider caching**: Since registered-node data changes infrequently, a short-lived response cache (e.g., via Spring Cache or the existing Redis infrastructure) would absorb burst traffic without hitting the DB at all.

### Proof of Concept

```bash
# Exhaust the 10-connection HikariCP pool with 20 concurrent requests,
# each targeting a different node ID via the EQ path.
for i in $(seq 1 20); do
  curl -s "http://<host>/api/v1/network/registered-nodes?registerednode.id=eq:$i" &
done
wait

# Legitimate request issued while pool is saturated will time out:
curl -v "http://<host>/api/v1/network/registered-nodes"
# Expected: connection acquisition timeout / 503 after ~30s
```

Sustained with a tool like `wrk`:
```
wrk -t 10 -c 50 -d 60s \
  "http://<host>/api/v1/network/registered-nodes?registerednode.id=eq:1"
```

    @ConfigurationProperties("spring.datasource.hikari")

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
