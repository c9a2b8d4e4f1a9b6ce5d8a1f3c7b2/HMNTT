Looking at the actual code:

**`decodeEvmAddress`** in `GraphQlUtils.java` (lines 87-98): strips `0x`, calls `Hex.decodeHex()`, and returns the raw bytes — **no length check**.

**`getByEvmAddressAndType`** in `EntityServiceImpl.java` (lines 34-41): wraps whatever bytes are returned in a `ByteBuffer`, then unconditionally calls:
- `buffer.getInt()` — reads 4 bytes (positions 0–3)
- `buffer.getLong()` — reads 8 bytes (positions 4–11)
- `buffer.getLong()` — reads 8 bytes (positions 12–19) — **only reached if the first two reads both return 0**

For a 16-byte input (32 hex chars) where the first 12 bytes are `0x00`:
- `getInt()` → 0 ✓
- `getLong()` → 0 ✓ (condition is true, enters the `if` block)
- `getLong()` → only 4 bytes remain → **`BufferUnderflowException`**

The vulnerability is real. However, the severity label "Critical: Direct loss of funds" is inaccurate. This is a **denial-of-service** issue — an unhandled `BufferUnderflowException` propagating from the GraphQL endpoint — not a funds-loss vulnerability.

### Title
`BufferUnderflowException` via Short EVM Address in `getByEvmAddressAndType`

### Summary
`decodeEvmAddress` performs no length validation on the decoded byte array. When a caller supplies a 16-byte hex-encoded address whose first 12 bytes are zero, the `ByteBuffer` in `getByEvmAddressAndType` satisfies the `getInt() == 0 && getLong() == 0` condition and then throws `BufferUnderflowException` on the third `getLong()` call because only 4 bytes remain. This is an unhandled unchecked exception that crashes the request.

### Finding Description
**File:** `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`
**Function:** `getByEvmAddressAndType`, lines 34–41

**File:** `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`
**Function:** `decodeEvmAddress`, lines 87–98

`decodeEvmAddress` only validates that the input is non-null and valid hex; it never asserts `result.length == 20`. [1](#0-0) 

`getByEvmAddressAndType` wraps the returned bytes in a `ByteBuffer` and unconditionally reads 4 + 8 + 8 = 20 bytes, assuming the buffer always holds exactly 20 bytes. [2](#0-1) 

With a 16-byte input where bytes 0–11 are `0x00`, the condition on line 37 is satisfied and execution enters the `if` block. The third `buffer.getLong()` on line 38 attempts to read 8 bytes from a position where only 4 remain, throwing `java.nio.BufferUnderflowException`.

### Impact Explanation
Any unauthenticated caller who can invoke the GraphQL query that reaches `getByEvmAddressAndType` can trigger an unhandled `BufferUnderflowException`. This causes the request to fail with a 500-level error. Repeated calls constitute a targeted denial-of-service against this endpoint. There is no direct loss of funds; the severity label in the prompt is overstated.

### Likelihood Explanation
The exploit requires no authentication, no special privileges, and no knowledge of internal state. The attacker only needs to know the GraphQL schema exposes an EVM address input field. The crafted input is a valid hex string (32 hex chars, first 24 of which are `0`), indistinguishable from a legitimate short address at the transport layer. It is trivially repeatable.

### Recommendation
Add an explicit length check in `decodeEvmAddress` (or at the call site in `getByEvmAddressAndType`) immediately after decoding:

```java
byte[] bytes = Hex.decodeHex(evmAddress);
if (bytes.length != 20) {
    throw new IllegalArgumentException(
        "evmAddress must be exactly 20 bytes, got: " + bytes.length);
}
return bytes;
```

This converts the `BufferUnderflowException` into a controlled `IllegalArgumentException` that the GraphQL layer can surface as a client error (400-equivalent) rather than a server error.

### Proof of Concept
1. Identify any GraphQL query that accepts an EVM address and routes to `getByEvmAddressAndType` (e.g., a `contract` or `account` query by `evmAddress`).
2. Send the following input as the `evmAddress` argument:
   ```
   0x000000000000000000000000AABBCCDD
   ```
   (32 hex chars = 16 bytes; first 12 bytes are `0x00`, last 4 bytes are arbitrary non-zero)
3. The server decodes 16 bytes, `getInt()` returns 0, `getLong()` returns 0, the `if` branch is taken, and `buffer.getLong()` throws `BufferUnderflowException`.
4. The GraphQL response returns a 500-level error. Repeat at will to deny service to this endpoint.
