### Title
Unauthenticated Endpoint Triggers 5 Parallel DB Queries Per Request With No Application-Level Rate Limiting

### Summary
The `GET /contracts/:contractId/results/:consensusTimestamp` endpoint is publicly accessible without authentication and, for any valid contractId+timestamp pair, unconditionally fires 5 parallel database queries via `getDetailedContractResults()`. The REST service has no application-level rate limiting or concurrency cap on this endpoint, allowing any unprivileged attacker to exhaust the database connection pool and degrade service for all users by flooding the endpoint at scale.

### Finding Description

**Exact code path:**

`getContractResultsByTimestamp` at [1](#0-0)  accepts `hbar` as a query parameter (default `true`) and, after a single gating DB call to `getInvolvedContractsByTimestampAndContractId`, immediately calls `getDetailedContractResults()`.

`getDetailedContractResults` at [2](#0-1)  issues 5 simultaneous DB queries via `Promise.all`:
1. `ContractService.getContractResultsByTimestamps`
2. `TransactionService.getEthTransactionByTimestampAndPayerId`
3. `RecordFileService.getRecordFileBlockDetailsFromTimestamp`
4. `ContractService.getContractLogsByTimestamps`
5. `ContractService.getContractStateChangesByTimestamps`

A 6th query (`FileDataService.getLatestFileDataContents`) fires conditionally if the eth transaction has a `callDataId`. [3](#0-2) 

`setContractResultsResponse` with `convertToHbar=true` is pure in-memory computation (no additional DB queries); the `hbar` parameter does not change DB load. [4](#0-3) 

**Root cause — failed assumption:** The code assumes infrastructure-level rate limiting (Traefik middleware) is always present and sufficient. The REST application itself has no per-IP request throttling or concurrency limit on this endpoint.

The `authHandler` only grants authenticated users a higher *response page size limit* — it is not a rate limiter. [5](#0-4) 

The `acceptedContractResultsByTimestampParameters` set only allows `hbar` as a query parameter, confirming no other guard exists. [6](#0-5) 

The Traefik middleware rate limiting is defined only in the Helm chart values and is explicitly optional (`if and .Values.global.middleware .Values.middleware`). [7](#0-6) 

### Impact Explanation
Each flood request against a valid contractId+timestamp pair consumes at minimum 5 simultaneous DB connections for the duration of those queries. At N concurrent flood requests, the DB connection pool is saturated with 5×N connections. Legitimate users experience query timeouts or connection-refused errors. Because the gating query (`getInvolvedContractsByTimestampAndContractId`) is cheap and returns quickly for known-valid timestamps, the attacker can sustain the 5-query burst continuously. This is a denial-of-service against the shared database, affecting all API endpoints that share the same pool.

### Likelihood Explanation
Preconditions are minimal: the attacker needs only one valid `contractId` and `consensusTimestamp`, both of which are publicly observable on-chain or via other mirror node API calls (e.g., `GET /contracts/:contractId/results`). No credentials, tokens, or special privileges are required. The attack is trivially scriptable with standard HTTP tooling (`ab`, `wrk`, `curl` loops). It is repeatable indefinitely and requires no state.

### Recommendation
1. **Application-level rate limiting**: Add per-IP (or per-connection) request rate limiting directly in the Express middleware stack for this endpoint, independent of infrastructure configuration.
2. **DB connection concurrency cap**: Enforce a maximum number of concurrent in-flight DB queries per endpoint or globally, rejecting excess requests with HTTP 429 before they reach the DB.
3. **Do not rely solely on Traefik middleware**: The Helm chart middleware is optional and may not be deployed in all environments. The application must be safe by default.
4. **Short-circuit on invalid timestamps early**: The gating query already does this for unknown timestamps, but consider caching or lightweight validation to reduce even that single query under flood conditions.

### Proof of Concept

**Preconditions:**
- Obtain any valid `contractId` (e.g., `0.0.1234`) and a known `consensusTimestamp` (e.g., `1234567890.000000001`) from the public mirror node API.

**Steps:**
```bash
# Flood the endpoint from a single unprivileged client
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1234/results/1234567890.000000001?hbar=true" &
done
wait
```

**Result:** Each concurrent request triggers 5 parallel DB queries. With 500 concurrent requests, up to 2,500 simultaneous DB queries are issued. The DB connection pool is exhausted, and all other API endpoints begin returning errors or timing out for legitimate users.
