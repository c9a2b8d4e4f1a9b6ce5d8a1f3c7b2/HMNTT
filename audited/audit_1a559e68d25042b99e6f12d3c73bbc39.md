### Title
Cache Key Based on Raw `req.originalUrl` Enables Unauthenticated Cache-Bypass DoS via URL Permutation

### Summary
`cacheKeyGenerator` in `responseCacheHandler.js` hashes the raw `req.originalUrl` string as the Redis cache key, with no normalization applied. Because `normalizeRequestQueryParams` is explicitly not yet integrated into the cache key path (acknowledged in a code comment as a future TODO), an unauthenticated attacker can generate an unbounded set of semantically identical URLs that each produce a distinct cache key, forcing a fresh database query for every variant and exhausting backend resources.

### Finding Description
**Exact code path:**

`rest/middleware/responseCacheHandler.js` line 151–153:
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
The comment immediately above (lines 147–149) reads:
> `path?query - In the future, this will utilize Edwin's request normalizer (9113).`

This confirms the normalizer (`normalizeRequestQueryParams` in `rest/middleware/requestNormalizer.js`) is **not wired into the cache key pipeline**. A `grep` across the entire codebase shows `normalizeRequestQueryParams` is only referenced in its own source file and its unit test — it is never called during live request processing.

**Root cause:** The cache key is the MD5 of the verbatim client-supplied URL. Any two URLs that differ syntactically but are semantically identical (same query result) receive different cache keys and each triggers a full database round-trip.

**Exploit dimensions — how many unique cache-busting URLs exist per endpoint:**

1. **Default-value presence/absence** — `normalizeRequestQueryParams` shows that `/api/v1/accounts` has at least three parameters with defaults (`balance=true`, `limit=25`, `order=asc`). Each can be omitted or explicitly supplied → 2³ = 8 combinations.
2. **Parameter ordering** — `req.originalUrl` is a raw string; `?balance=true&limit=25&order=asc` and `?order=asc&balance=true&limit=25` are different strings → 3! = 6 orderings per combination → 48 unique keys for the same logical query.
3. **Key-case variation** — `requestQueryParser` lowercases keys before processing, but `req.originalUrl` is captured before parsing. `ORDER=asc` and `order=asc` produce different MD5 hashes but identical query results.
4. **Repeated/equivalent values** — e.g., `limit=25` vs `limit=025` vs `limit=25&limit=25` (the last is accepted per the test suite).

Combining all dimensions, hundreds of distinct cache-miss-inducing URLs exist per endpoint, each forcing a Postgres query.

**Middleware order (server.js lines 95–141):**
```
responseCacheCheckHandler   ← cache lookup uses req.originalUrl
  → route handler           ← DB query executed on miss
responseCacheUpdateHandler  ← stores result under same req.originalUrl key
```
There is no normalization step between the client and the cache check.

### Impact Explanation
Every cache miss causes a full SQL query against the Postgres mirror database. The REST API serves financial ledger data (account balances, transaction history, token transfers). An attacker flooding the server with URL permutations of high-cost queries (e.g., `/api/v1/transactions` with timestamp ranges) can saturate the database connection pool, causing query timeouts and HTTP 500 errors for all legitimate users. The mirror node is a read-only financial data API; degraded availability directly impairs the ability of wallets, exchanges, and auditors to read on-chain financial state.

### Likelihood Explanation
- **No authentication required** — the REST API is public; `server.js` applies `authHandler` but it is not a blocking gate for unauthenticated read endpoints.
- **No rate limiting in the REST middleware stack** — `server.js` registers no rate-limit middleware for the REST API (the `rateLimit` configuration visible in the repo applies only to the Rosetta/web3 Traefik ingress, a separate service).
- **Trivially scriptable** — a simple loop generating URL permutations requires no special tooling or privileges.
- **Repeatable indefinitely** — because the cache is keyed per exact URL string, the attacker's permutations never warm each other's cache entries; the attack sustains itself as long as the attacker keeps sending requests.

### Recommendation
1. **Immediate:** Integrate `normalizeRequestQueryParams` into `cacheKeyGenerator` as the TODO comment (line 149) already anticipates. Replace `req.originalUrl` with the output of `normalizeRequestQueryParams(req.openapi?.openApiRoute, req.path, req.query)` so all semantically equivalent requests share one cache entry.
2. **Defense-in-depth:** Add per-IP rate limiting middleware to the REST API (e.g., `express-rate-limit`) independent of the ingress layer, so the application layer can shed load before it reaches the database.
3. **Short-term mitigation:** Until the normalizer is integrated, canonicalize the cache key by sorting query parameters alphabetically and lowercasing parameter names before hashing.

### Proof of Concept
```bash
# All requests below return identical data but each misses the cache
# and forces a separate DB query.

BASE="http://<mirror-node>/api/v1/accounts"

# Permutation 1 – no explicit defaults (normalizer would add them)
curl "$BASE"

# Permutation 2 – explicit defaults, one ordering
curl "$BASE?balance=true&limit=25&order=asc"

# Permutation 3 – explicit defaults, different ordering
curl "$BASE?order=asc&balance=true&limit=25"

# Permutation 4 – uppercase key (parsed identically, different raw URL)
curl "$BASE?ORDER=asc&balance=true&limit=25"

# Permutation 5 – partial defaults
curl "$BASE?balance=true"

# Repeat in a tight loop with all permutations to exhaust DB connections:
for i in $(seq 1 1000); do
  curl -s "$BASE?balance=true&limit=25&order=asc" &
  curl -s "$BASE?order=asc&limit=25&balance=true" &
  curl -s "$BASE?limit=25&order=asc&balance=true" &
  curl -s "$BASE" &
done
wait
```
Each variant produces a distinct MD5 cache key, confirmed by the `cacheKeyGenerator` logic at `rest/middleware/responseCacheHandler.js:152`, and each triggers a full Postgres query with no cache hit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```javascript
const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
};
```
