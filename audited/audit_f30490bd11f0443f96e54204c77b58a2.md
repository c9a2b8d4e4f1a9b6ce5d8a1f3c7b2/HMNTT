### Title
Unauthenticated Internal Infrastructure Disclosure via `/health/readiness` Error Propagation

### Summary
The `/health/readiness` endpoint in `rosetta/app/middleware/health.go` is publicly accessible without authentication and delegates directly to `hellofresh/health-go/v4`'s `HandlerFunc`, which serializes raw check error strings into the JSON response body. When the PostgreSQL or network checks fail, the underlying driver error messages expose internal infrastructure details — including database host, port, username, and database name — to any unauthenticated external caller. This enables targeted reconnaissance of the internal network topology without any privileged access.

### Finding Description

**Code path:**

`NewHealthController()` at [1](#0-0)  constructs two readiness checks. The PostgreSQL check is initialized with the full DSN produced by `GetDsn()`: [2](#0-1) 

This DSN contains `host`, `port`, `user`, `dbname`, and `password` in plaintext. It is passed directly to `postgres.New()`: [3](#0-2) 

`Routes()` registers the handler with no authentication middleware: [4](#0-3) 

**Root cause:** The `hellofresh/health-go/v4` library's `HandlerFunc` captures the `error.Error()` string returned by each failing check function and serializes it into the JSON response body under the `output` field of each named check component. No error sanitization or wrapping is applied before this serialization.

**What leaks:**
- **PostgreSQL check failure** → the `lib/pq` / `pgx` driver returns errors such as:
  - `dial tcp <db-host>:5432: connect: connection refused` → reveals DB host and port
  - `pq: password authentication failed for user "mirror_rosetta"` → reveals DB username
  - `pq: database "mirror_node" does not exist` → reveals DB name
- **Network check failure** → the Rosetta API client returns errors containing the internal service URL (`http://localhost:<port>`), revealing the internal port: [5](#0-4) 

**Example leaked JSON response (503):**
```json
{
  "status": "unavailable",
  "output": "",
  "checks": {
    "postgresql": {
      "status": "unavailable",
      "output": "dial tcp 10.0.1.45:5432: connect: connection refused",
      "timestamp": "..."
    },
    "network": {
      "status": "unavailable",
      "output": "Post \"http://localhost:8080/network/list\": dial tcp ...",
      "timestamp": "..."
    }
  }
}
```

**Existing checks reviewed:** There is no authentication guard, no IP allowlist, no error wrapping/sanitization, and no rate limiting on this endpoint. The `SkipOnErr: false` setting ensures failures are always reported. [6](#0-5) 

### Impact Explanation
An unauthenticated attacker learns the exact internal DB host, port, username, and database name from a single HTTP GET. This constitutes a complete internal network topology map for the database tier. Armed with this, an attacker can: (1) target the DB host directly with exploits or DoS at the network layer; (2) use the username for credential stuffing against other exposed surfaces; (3) enumerate adjacent services on the same subnet. If the DB host is reachable (misconfigured firewall, cloud security group, etc.) or if the attacker is on the same network segment, this reconnaissance directly enables a targeted attack that can take down the database backing multiple Rosetta nodes simultaneously, meeting the ≥30% node shutdown threshold without brute force.

### Likelihood Explanation
The endpoint requires zero authentication and is reachable by any external caller who can reach the Rosetta API port. The attack is a single unauthenticated GET request — trivially repeatable, requires no special tooling, and produces deterministic output. Health endpoints are commonly exposed in Kubernetes/cloud deployments for liveness probes, making external exposure a realistic deployment scenario. The attacker needs no prior knowledge beyond the service hostname and port.

### Recommendation
1. **Sanitize error output:** Wrap check errors before returning them, stripping infrastructure details. Implement a custom check wrapper that returns a generic error string (e.g., `"postgresql check failed"`) while logging the full error server-side only.
2. **Restrict endpoint access:** Place `/health/readiness` behind a network policy or middleware that restricts access to internal cluster IPs / orchestration systems only (e.g., Kubernetes probe source IPs).
3. **Do not pass raw DSN errors to the library:** Wrap `postgres.New(...)` in a closure that catches and sanitizes errors before returning them to the health framework.

### Proof of Concept
```bash
# Step 1: Trigger a readiness check against a node where DB is unreachable or misconfigured
curl -s http://<rosetta-node-host>:<port>/health/readiness | jq .

# Step 2: Observe response (HTTP 503) containing internal details:
# {
#   "status": "unavailable",
#   "checks": {
#     "postgresql": {
#       "status": "unavailable",
#       "output": "dial tcp 10.0.1.45:5432: connect: connection refused"
#     },
#     "network": {
#       "status": "unavailable",
#       "output": "Post \"http://localhost:8080/network/list\": ..."
#     }
#   }
# }

# Step 3: Extract DB host (10.0.1.45), port (5432), username from auth error:
curl -s http://<rosetta-node-host>:<port>/health/readiness | jq '.checks.postgresql.output'
# "pq: password authentication failed for user \"mirror_rosetta\""

# Step 4: Use extracted topology to target DB host directly
nmap -p 5432 10.0.1.45
# or target with known PostgreSQL CVEs / network-layer DoS
```

No authentication, no special privileges, and no brute force required. A single HTTP request is sufficient to extract the internal DB host, port, and username.
