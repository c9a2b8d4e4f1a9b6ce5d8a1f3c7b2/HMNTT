### Title
Unbounded Concurrent Subscriptions Enable Resource Exhaustion DoS via gRPC Topic Streaming

### Summary
An unprivileged external user can open arbitrarily many concurrent gRPC topic subscriptions with `limit=0` and no `endTime`, each of which runs indefinitely. There is no enforcement of a maximum subscriber count, no mandatory termination condition, and no per-client rate limiting, allowing an attacker to exhaust CPU, thread-pool, and connection resources on the mirror node.

### Finding Description

**Code path:**

`TopicMessageFilter` declares both `endTime` and `limit` as optional with no upper-bound constraint: [1](#0-0) 

`hasLimit()` returns `false` when `limit == 0` (the default): [2](#0-1) 

In `subscribeTopic()`, both termination operators are gated behind null/zero checks — if neither is supplied, neither fires: [3](#0-2) 

`pastEndTime()` returns `Flux.never()` when `endTime` is null, so the `takeUntilOther` in the main pipeline never completes: [4](#0-3) 

`isComplete()` unconditionally returns `false` when `endTime` is null, so `incomingMessages()` never self-terminates: [5](#0-4) 

`subscriberCount` is tracked only as a Micrometer gauge — it is never checked against a maximum, so no subscription is ever rejected: [6](#0-5) 

**Correction on the specific memory-accumulation claim:** `TopicContext.count` is a fixed-size `AtomicLong` (8 bytes) and `TopicContext.last` is an `AtomicReference` holding exactly one `TopicMessage` at a time — neither field grows unboundedly in memory. [7](#0-6) 

The real exhaustion mechanism is: each indefinite subscription occupies a live gRPC connection, a Reactor scheduler slot on `Schedulers.boundedElastic()`, and continuous CPU cycles processing every incoming message through `concatMap`/`missingMessages`. Under high topic throughput, the reactive pipeline's internal demand-management buffers (not TopicContext) can also grow if the downstream consumer is slower than the source.

### Impact Explanation
An attacker opening hundreds of concurrent unlimited subscriptions to a high-throughput topic will saturate the bounded elastic thread pool, exhaust available gRPC connections, and drive CPU to 100%, making the mirror node's gRPC endpoint unavailable to legitimate clients. This is a non-network-based DoS (no bandwidth flooding required — a single client with a normal internet connection suffices). Given the mirror node's role in serving Hedera network data, availability impact is high.

### Likelihood Explanation
No authentication or authorization is required to call `subscribeTopic`. The gRPC endpoint is publicly reachable. The attacker needs only a gRPC client (e.g., `grpcurl` or the Hedera SDK), knowledge of any valid topic ID, and the ability to open many concurrent streams. This is trivially scriptable and repeatable.

### Recommendation
1. **Enforce a maximum concurrent subscriber count**: check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if the limit is exceeded.
2. **Require or cap `endTime`**: either make `endTime` mandatory or impose a server-side maximum subscription duration (e.g., 1 hour).
3. **Enforce a maximum `limit`**: reject or cap requests where `limit` exceeds a configurable threshold.
4. **Per-client rate limiting**: track subscriptions per source IP/identity and reject excess connections.

### Proof of Concept
```
# Open 500 concurrent unlimited subscriptions to a high-throughput topic
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 1234}, "consensusStartTime": {}}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Result: bounded elastic thread pool saturated, CPU pegged,
# legitimate subscribers receive UNAVAILABLE or timeout errors.
```

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
