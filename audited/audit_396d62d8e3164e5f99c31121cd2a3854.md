### Title
Unauthenticated Resource Exhaustion via Maximum IN-Clause DB Query on `/api/v1/accounts/{ownerId}/hooks`

### Summary
The `getHooks()` endpoint in `HooksController` permits any unauthenticated caller to supply exactly 100 distinct `hook.id=eq:N` query parameters in a single request. This causes `hooksRequest()` to build a 100-element `TreeSet` and `HookServiceImpl.getHooks()` to issue a `WHERE hook_id IN (?, ?, …, ?)` query with 100 bind values against the database. No rate limiting exists in the `rest-java` module, so an attacker can flood the endpoint with maximally-loaded requests, driving sustained elevated DB CPU and I/O.

### Finding Description

**Code path:**

1. `HooksController.getHooks()` — `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on the `hookId` array permits 0–100 parameters inclusive. Exactly 100 is accepted; 101 is rejected. [1](#0-0) [2](#0-1) 

2. `hooksRequest()` iterates all 100 filters; each `EQ` operator adds its value to a `TreeSet<Long>`. With 100 distinct values the set has 100 elements. [3](#0-2) 

3. `HookServiceImpl.getHooks()` — when `hookIds` is non-empty, it streams all IDs through a range filter (default bounds 0 to `Long.MAX_VALUE`), so all 100 IDs survive, and `findByOwnerIdAndHookIdIn()` is called with a 100-element collection. [4](#0-3) 

4. `HookRepository.findByOwnerIdAndHookIdIn()` generates `WHERE owner_id = ? AND hook_id IN (?, ?, …, ?)` with 100 bind values — a significantly heavier query than the default range query (`BETWEEN`). [5](#0-4) 

5. The `rest-java` module has **no rate-limiting filter**. The only filters present are `LoggingFilter` and `MetricsFilter`. The Bucket4j throttle (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is not applied here. [6](#0-5) [7](#0-6) 

**Root cause:** The `@Size(max=100)` guard is a *cardinality* cap, not a *cost* cap. It was designed to prevent unbounded arrays but inadvertently defines the maximum attack payload size. The failed assumption is that 100 IN-clause parameters per unauthenticated request is safe without rate limiting.

### Impact Explanation
Each maximally-loaded request forces the DB to evaluate a 100-value `IN` predicate, which is substantially more expensive than the default `BETWEEN` range scan (which uses a simple index range). An attacker sending N concurrent such requests multiplies DB CPU and I/O by a factor proportional to N×100 compared to a baseline of normal single-parameter requests. This can push DB CPU and I/O well above the 30% threshold cited in the scope, degrading service for all users of the mirror node without any brute-force credential guessing.

### Likelihood Explanation
The endpoint is fully public — no authentication, no API key, no session. The exploit requires only an HTTP client and knowledge of the public API. The attack is trivially scriptable (e.g., `curl` in a loop or a simple load-testing tool). It is repeatable indefinitely and can be distributed across multiple source IPs to evade any upstream IP-based throttling.

### Recommendation
1. **Add per-IP or global rate limiting** to the `rest-java` module (e.g., extend the existing Bucket4j infrastructure from `web3` or add a Spring `HandlerInterceptor`).
2. **Charge cost proportional to the number of `eq` parameters**: e.g., count each `eq` filter as one token against a per-request budget, or cap the number of `eq` parameters at a lower value (e.g., 10).
3. **Alternatively**, reject requests that mix `eq` filters with the maximum allowed count, or require that the `eq` set size be bounded by `limit` rather than `MAX_REPEATED_QUERY_PARAMETERS`.

### Proof of Concept
```bash
# Build a request with 100 distinct hook.id=eq:N parameters
PARAMS=$(seq 1 100 | awk '{printf "hook.id=eq:%d&", $1}' | sed 's/&$//')

# Send it repeatedly (no auth required)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks?$PARAMS" &
done
wait
```

Each of the 500 concurrent requests causes the DB to execute:
```sql
SELECT * FROM hook
WHERE owner_id = 1234
  AND hook_id IN (1, 2, 3, ..., 100)
ORDER BY hook_id DESC
LIMIT 25;
```
Monitor DB CPU/IO before and during the attack; the increase will be proportional to the concurrency level, with no server-side throttle to stop it.

    private HooksRequest hooksRequest(
            EntityIdParameter ownerId, NumberRangeParameter[] hookIdFilters, int limit, Sort.Direction order) {
        final var hookIds = new TreeSet<Long>();
        long lowerBound = 0L; // The most restrictive lower bound (max of all gt/gte)
        long upperBound = MAX_VALUE; // The most restrictive upper bound (min of all lt/lte)

        for (final var hookIdFilter : hookIdFilters) {
            if (hookIdFilter.operator() == RangeOperator.EQ) {
                hookIds.add(hookIdFilter.value());
            } else if (hookIdFilter.hasLowerBound()) {
                lowerBound = Math.max(lowerBound, hookIdFilter.getInclusiveValue());
            } else if (hookIdFilter.hasUpperBound()) {
                upperBound = Math.min(upperBound, hookIdFilter.getInclusiveValue());
            }
        }
