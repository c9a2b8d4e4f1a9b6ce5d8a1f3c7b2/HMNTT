### Title
Unbounded Concurrent Polling Subscriptions Cause Database CPU Exhaustion in PollingTopicListener

### Summary
`PollingTopicListener.listen()` creates a per-subscription infinite polling loop using `RepeatSpec.times(Long.MAX_VALUE)` with a fixed 500ms interval, issuing one DB query per poll cycle per active subscription. No global subscription cap, no per-IP rate limit, and no authentication exist on the gRPC `subscribeTopic` endpoint, allowing any unauthenticated attacker to open N concurrent subscriptions and generate 2N DB queries/second indefinitely until the database is exhausted.

### Finding Description
**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()` (lines 38–43):

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)   // ← effectively infinite
                .jitter(0.1)
                .withFixedDelay(interval)              // ← default 500ms
                .withScheduler(scheduler))
```

Each call to `listen()` creates an independent reactive chain that calls `poll(context)` → `topicMessageRepository.findByFilter(newFilter)` (a live DB query) every ~500ms with no termination condition unless the client disconnects or a `limit` is reached. With `limit=0` (unlimited, the default when a client omits it), the loop never self-terminates.

**Root cause:** The failed assumption is that subscriptions are bounded by infrastructure (connection limits, authentication, or a global subscription cap). None of these exist:

- `ConsensusController.subscribeTopic()` (lines 43–53) performs no authentication or authorization check before calling `topicMessageService.subscribeTopic()`.
- `TopicMessageFilter` validation (lines 25–51) only enforces `limit >= 0`, `startTime <= now`, and `endTime > startTime` — no maximum subscription count.
- `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14) limits calls *per TCP connection*, but an attacker opens many connections. No global connection limit is enforced in `GrpcConfiguration`.
- `subscriberCount` in `TopicMessageServiceImpl` (line 48) is a Micrometer gauge for observability only — it is never checked against a maximum.
- The `ThrottleConfiguration` (bucket4j rate limiter) exists only in the `web3` module; the `grpc` module has no equivalent.

**Exploit flow:**
1. Attacker opens C TCP connections to port 5600 (no TLS required by default).
2. On each connection, attacker opens 5 concurrent `subscribeTopic` streams (the per-connection max), each with a valid `topicId`, `limit=0`, no `endTime`.
3. Each stream activates `PollingTopicListener.listen()` → one DB query every ~500ms.
4. Total DB query rate = C × 5 × 2 queries/second.
5. With C=100 connections: 1,000 queries/second sustained, indefinitely.

### Impact Explanation
Each `findByFilter` call executes a parameterized SQL query against the `topic_message` table. Under sustained load from hundreds of subscriptions, the database connection pool (HikariCP) saturates, CPU spikes, and query latency increases for all users including legitimate ones. The Prometheus alert `GrpcHighDatabaseConnections` fires at 75% pool utilization but does not auto-mitigate. The result is a non-network DoS: the mirror node's gRPC API becomes unresponsive or returns errors to all clients, degrading the HCS subscription service.

### Likelihood Explanation
The attack requires only a gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or any HTTP/2 client) and knowledge of one valid `topicId` (all topic IDs are public on-chain). No credentials, tokens, or privileged access are needed. The attack is trivially scriptable, repeatable, and can be sustained from a single machine with modest bandwidth. The only prerequisite is that the deployment uses `ListenerType.POLL` (configured via `hiero.mirror.grpc.listener.type=POLL`); the default is `REDIS`, but POLL mode is a documented and supported configuration used when Redis is unavailable.

### Recommendation
1. **Add a global concurrent-subscription cap**: Enforce a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` by checking `subscriberCount` against a configured threshold and returning `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP subscription rate limiting**: Intercept at the gRPC server interceptor layer (similar to `ThrottleConfiguration` in web3) using bucket4j or Resilience4j.
3. **Enforce a maximum `RepeatSpec` count tied to a configurable TTL**: Replace `Long.MAX_VALUE` with a computed repeat count based on a maximum subscription duration (e.g., 1 hour → 7,200 repeats at 500ms).
4. **Require authentication** for `subscribeTopic` via a gRPC server interceptor to prevent anonymous abuse.

### Proof of Concept
```bash
# Requires grpcurl and a running mirror node in POLL listener mode
# with a known valid topicId (e.g., 0.0.41110)

for i in $(seq 1 100); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Each background process holds an open streaming subscription.
# 100 connections × 1 stream each = ~200 DB queries/second sustained.
# Scale to 500 connections for ~1000 queries/second.
# Monitor DB CPU: watch -n1 "psql -c 'SELECT count(*) FROM pg_stat_activity;'"
```
