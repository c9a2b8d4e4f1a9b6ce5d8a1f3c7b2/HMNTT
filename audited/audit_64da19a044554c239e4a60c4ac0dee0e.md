### Title
Unauthenticated `/network/status` Endpoint Triggers Unbounded DB Queries with No Application-Level Rate Limiting

### Summary
The `networkAPIService.NetworkStatus()` handler in `rosetta/app/services/network_service.go` executes 2–3 database queries per request (one `selectLatestWithIndex` via `RetrieveLatest`, and one or two `latestNodeServiceEndpoints` JOIN queries via `Entries()`), with no authentication, no application-level rate limiting, and no response caching. The Traefik-based rate-limiting middleware defined in the Helm chart is disabled by default (`global.middleware: false`), leaving the endpoint fully exposed to unauthenticated flooding that can drive DB CPU and connection consumption well above the 30% baseline threshold.

### Finding Description

**Code path:**

`rosetta/app/services/network_service.go` lines 59–88 — `NetworkStatus()` sequentially calls:
1. `n.RetrieveGenesis(ctx)` → `blockRepo.RetrieveGenesis()` → `initGenesisRecordFile()` in `rosetta/app/persistence/block.go` lines 243–263. This uses `sync.Once`, so after the first successful call the genesis block is cached in memory and **no further DB query is issued** for this step.
2. `n.RetrieveLatest(ctx)` → `blockRepo.RetrieveLatest()` in `rosetta/app/persistence/block.go` lines 190–208. This **always** executes `selectLatestWithIndex` against `record_file`.
3. `n.addressBookEntryRepo.Entries(ctx)` in `rosetta/app/persistence/address_book_entry.go` lines 57–83. This **always** executes the `latestNodeServiceEndpoints` query — a multi-table JOIN with a correlated subquery, `LEFT JOIN node`, `LEFT JOIN address_book_service_endpoint`, and a `GROUP BY` with `string_agg` — once for file_id 101, and a second time for file_id 102 if the first returns no rows.

**Root cause:** Neither `RetrieveLatest` nor `Entries` caches results. Every HTTP request to `/network/status` unconditionally issues 2–3 DB queries. There is no authentication gate, no in-process rate limiter, and no response-level cache anywhere in the Go application.

**Why existing checks are insufficient:**
- The `sync.Once` in `initGenesisRecordFile` only eliminates the genesis query after the first call; it does not protect `RetrieveLatest` or `Entries`.
- The Traefik middleware chain (`inFlightReq: amount: 5`, `rateLimit: average: 10`) is defined in `charts/hedera-mirror-rosetta/values.yaml` lines 149–166 but is conditioned on `{{ if and .Values.global.middleware .Values.middleware }}` in `charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3. `global.middleware` defaults to `false` (line 95 of `values.yaml`), so the middleware is **not deployed by default**.
- The DB connection pool cap (`maxOpenConnections: 100`) limits connection exhaustion but does not prevent CPU saturation from concurrent complex queries.

### Impact Explanation
An attacker can saturate the PostgreSQL instance serving the mirror node by flooding `/network/status` with concurrent requests. Each request issues the `latestNodeServiceEndpoints` query — a correlated subquery with three JOINs and a `string_agg` GROUP BY — plus a full `ORDER BY index DESC LIMIT 1` scan of `record_file`. With 100 concurrent connections (the pool maximum) all executing these queries simultaneously, DB CPU and I/O can spike far above the 30% baseline threshold, degrading or blocking all other mirror node services (REST API, gRPC, Web3) that share the same database.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The endpoint is a standard Rosetta API path (`POST /network/status`) documented in the Coinbase Rosetta specification, so any attacker familiar with Rosetta knows it exists. A single machine with a modest HTTP flood tool (e.g., `wrk`, `hey`, or `ab`) can sustain hundreds of requests per second. The attack is trivially repeatable and requires no state.

### Recommendation
1. **Add application-level response caching** for `NetworkStatus`: cache the full response (or at minimum the `Entries()` result) with a short TTL (e.g., 5–30 seconds), since address book data changes infrequently.
2. **Enable the Traefik middleware by default** by changing `global.middleware` to `true` in the default `values.yaml`, or enforce rate limiting unconditionally.
3. **Add an in-process rate limiter** (e.g., `golang.org/x/time/rate`) in the HTTP handler layer so protection is not solely dependent on infrastructure configuration.
4. **Cache `RetrieveLatest` results** with a short TTL (e.g., 1–2 seconds) since the latest block changes at most every few seconds.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode, default Helm deployment (no Traefik middleware), accessible on port 5700.

**Steps:**
```bash
# Flood /network/status with 50 concurrent workers, no auth needed
wrk -t 50 -c 50 -d 60s \
  -s <(echo 'wrk.method="POST"; wrk.body="{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"}}"; wrk.headers["Content-Type"]="application/json"') \
  http://<rosetta-host>:5700/network/status
```

**Expected result:** PostgreSQL CPU and active connection count rise sharply. Each request triggers execution of `selectLatestWithIndex` and `latestNodeServiceEndpoints` (with JOINs across `address_book`, `address_book_entry`, `node`, `address_book_service_endpoint`). At 50 concurrent requests, DB CPU will exceed the 30% baseline increase threshold on any lightly-to-moderately loaded node. Monitor with `SELECT count(*), state FROM pg_stat_activity GROUP BY state` and `pg_stat_statements` to confirm query volume.
