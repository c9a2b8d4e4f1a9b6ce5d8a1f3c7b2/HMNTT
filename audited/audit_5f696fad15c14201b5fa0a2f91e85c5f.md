### Title
Mirror Node REST Endpoint Poisoning Causes Permanent Consensus Network Partition in Pinger

### Summary
When `network=other`, `buildNetworkFromMirrorNodes()` fetches consensus node endpoints over an unauthenticated HTTP connection and blindly trusts the returned JSON. An attacker who can intercept or control the mirror REST endpoint can return a crafted `nodesEnvelope` with all `serviceEndpoints` pointing to unreachable IPs. Because the resulting network map is built once at startup and never refreshed, all subsequent `submitWithRetry()` calls fail indefinitely while the process continues to appear alive.

### Finding Description

**Code path:**

`pinger/sdk_client.go` `newClient()` (line 18) calls `buildNetworkFromMirrorNodes()` exactly once at startup. The returned map is passed directly to `hiero.ClientForNetwork(netmap)` (line 22) and the client is never rebuilt. [1](#0-0) 

Inside `fetchMirrorNodeNetwork()` (`pinger/mirror_node_client.go` lines 74–132), the only validation applied to each returned endpoint is:

- `NodeAccountID` must be non-empty and parseable (lines 103–109) — does not touch the IP/host.
- `host` must be non-empty and `port` must be non-zero (lines 118–119) — purely syntactic.
- The final map must contain at least one entry (lines 127–129) — trivially satisfied by one crafted entry. [2](#0-1) 

There is no:
- TLS/certificate pinning on the mirror REST connection (the documented example URL is `http://mirror-rest:5551`, plaintext HTTP).
- Reachability probe of returned endpoints before accepting them.
- Cross-check of returned node IDs against any trusted/expected set.
- Periodic refresh of the network map after startup.

**Root cause:** The function assumes the mirror REST endpoint is trustworthy and that any syntactically valid endpoint is a reachable consensus node. Neither assumption is enforced.

### Impact Explanation

Once the poisoned client is built, every `cryptoTransfer.Execute(client)` call in `submitWithRetry()` attempts to reach only the attacker-supplied unreachable addresses. All `cfg.maxRetries + 1` attempts per tick fail. The process logs errors but continues running: the heartbeat goroutine keeps writing `/tmp/alive` (line 36 of `main.go`) and `/tmp/ready` was already created at startup (line 47), so Kubernetes liveness/readiness probes do not trigger a restart. [3](#0-2) [4](#0-3) 

The result is a silent, permanent denial-of-service: the pinger appears healthy to the orchestrator but never successfully submits a transaction for the entire lifetime of the process.

### Likelihood Explanation

The attack surface is realistic under several conditions:

1. **Plaintext HTTP**: `cfg.mirrorRest` defaults to and is documented as `http://mirror-rest:5551`. Any attacker with network-layer access (compromised pod in the same Kubernetes namespace, ARP/DNS poisoning on the same segment) can MITM the request with no TLS to overcome.
2. **DNS poisoning**: The hostname `mirror-rest` is resolved at request time; poisoning the in-cluster DNS record redirects all mirror REST calls to an attacker-controlled server.
3. **Compromised mirror node**: If the mirror node service itself is compromised (supply-chain or lateral movement), the attacker controls the response directly.
4. **One-shot window**: The attack only needs to succeed for the single HTTP request made at pinger startup. After that, the poisoned map is permanent.

The `network=other` mode is the custom/private-network deployment path, which is exactly the environment where internal network controls may be weaker.

### Recommendation

1. **Use HTTPS with certificate verification** for `cfg.mirrorRest`; reject plaintext HTTP in production configurations.
2. **Validate returned endpoints against a trusted allowlist** (e.g., expected node account IDs configured at deploy time).
3. **Probe endpoint reachability** (TCP dial with short timeout) before accepting a node into the network map; require a minimum quorum of reachable nodes.
4. **Periodically refresh the network map** (e.g., on each ticker tick or on repeated transfer failures) so a poisoned startup does not cause permanent partition.
5. **Fail closed**: if fewer than a configurable minimum number of endpoints are reachable, abort startup rather than building a client with an untested map.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
- Attacker can respond to HTTP GET requests to `http://mirror-rest:5551/api/v1/network/nodes` (DNS poisoning, ARP spoofing, or direct mirror node compromise).

**Steps:**

1. Attacker serves the following JSON at the mirror REST endpoint:
```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "ip_address_v4": "192.0.2.1", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```
(`192.0.2.1` is an RFC 5737 documentation address guaranteed unreachable.)

2. Pinger starts; `buildNetworkFromMirrorNodes()` fetches this response.
3. `fetchMirrorNodeNetwork()` passes all checks: `node_account_id` is valid, host is non-empty, port is non-zero, map length is 1.
4. `hiero.ClientForNetwork({"192.0.2.1:50211": AccountID{0,0,3}})` is called; client is stored.
5. Every subsequent `submitWithRetry()` call attempts gRPC to `192.0.2.1:50211`, times out, exhausts retries, and logs failure.
6. `/tmp/alive` is refreshed every 15 s; `/tmp/ready` exists — orchestrator sees a healthy pod.
7. No transaction is ever submitted for the lifetime of the process.
