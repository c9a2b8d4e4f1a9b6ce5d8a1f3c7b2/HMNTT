### Title
Malicious Block Node Can Trigger Repeated `BlockStreamException` via Malformed `BlockItemSet`, Disrupting Block Processing

### Summary
The `BlockAssembler.append()` method in `BlockNode.java` throws a `BlockStreamException` when the first item in a received `BlockItemSet` is not a `BLOCK_HEADER` and no block is currently pending. A block node operator (or network-level attacker on a non-TLS connection) can repeatedly send `SubscribeStreamResponse` messages with a `BLOCK_ITEMS` case whose first `BlockItem` has `ItemCase.UNSET` or any non-`BLOCK_HEADER` case, causing the mirror node to fail every streaming session against that node. With enough repetitions, the node is marked inactive, halting block ingestion until the readmit delay expires.

### Finding Description

**Exact code path:**

`BlockNode.streamBlocks()` (lines 138–155) reads `SubscribeStreamResponse` messages in a loop. When `BLOCK_ITEMS` is received, it calls `assembler.onBlockItemSet(response.getBlockItems())` (line 140).

`onBlockItemSet()` (lines 224–237) extracts the first item's `ItemCase` and immediately calls `append(blockItems, firstItemCase)` (line 232) without any prior validation of the case value.

`append()` (lines 283–298) enforces:
```java
} else if (firstItemCase != BLOCK_HEADER && pending.isEmpty()) {
    throw new BlockStreamException("Incorrect first block item case " + firstItemCase);
}
``` [1](#0-0) 

At the start of a streaming session `pending` is always empty, so any `BlockItemSet` whose first item is not `BLOCK_HEADER` (including `UNSET`, `EVENT_HEADER`, `TRANSACTION_RESULT`, etc.) unconditionally throws.

The exception is caught in `streamBlocks()` at line 159:
```java
} catch (BlockStreamException ex) {
    onError();
    throw ex;
}
``` [2](#0-1) 

`onError()` increments the error counter and, once `errors >= maxSubscribeAttempts`, marks the node `active = false` and sets a `readmitTime` cooldown: [3](#0-2) 

**Root cause / failed assumption:** The code assumes that any block node returning `BLOCK_ITEMS` will always start a new block with a `BLOCK_HEADER` item. There is no graceful skip/warn path for a malformed first item — the only code path is an exception that counts as a connection error.

**Why existing checks are insufficient:**
- The only guard before `append()` is an empty-list check (line 226–229), which logs a warning and returns safely. There is no analogous soft-failure path for an unexpected `ItemCase`.
- `onBlockItemSet()` passes the raw `firstItemCase` directly to `append()` with no pre-validation. [4](#0-3) 

### Impact Explanation
Every streaming session against the malicious node is aborted before any block is assembled. After `maxSubscribeAttempts` consecutive failures the node is placed in a cooldown. If only one block node is configured, the mirror node stops ingesting blocks entirely until the readmit delay expires, then the cycle repeats. Transactions in every affected block are never written to the mirror node database, making them invisible to downstream consumers (REST API, gRPC API, etc.) for the duration of the attack.

### Likelihood Explanation
- **With TLS disabled** (`requiresTls = false`, which is a valid configuration): any on-path network attacker can inject a single malformed gRPC message per session — no block node credentials required.
- **With TLS enabled**: any operator who can register a block node endpoint in the mirror node's configuration (e.g., via `blockNodeDiscoveryService`) can exploit this. In a permissionless or loosely-governed deployment this is a low bar.
- The attack is trivially repeatable: one malformed message per session is sufficient to trigger the exception. [5](#0-4) 

### Recommendation
Replace the hard throw in `append()` for the `firstItemCase != BLOCK_HEADER && pending.isEmpty()` branch with a recoverable warning (similar to the empty-list branch in `onBlockItemSet()`), discarding the malformed `BlockItemSet` and continuing the stream rather than aborting the entire session. Alternatively, add a pre-check in `onBlockItemSet()` that logs and returns early when the first item is not `BLOCK_HEADER` and `pending` is empty, never reaching `append()`. Additionally, enforce mutual TLS or a node-identity check so that only authenticated block nodes can feed data to the subscriber. [6](#0-5) 

### Proof of Concept
1. Stand up a gRPC server implementing `BlockStreamSubscribeService.subscribeBlockStream`.
2. Configure the mirror node to use this server as its sole block node endpoint.
3. When the mirror node sends a `SubscribeStreamRequest`, respond with:
   ```
   SubscribeStreamResponse {
     response_case: BLOCK_ITEMS,
     block_items: BlockItemSet {
       block_items: [ BlockItem { item_case: UNSET } ]
     }
   }
   ```
4. Observe: `append()` throws `BlockStreamException("Incorrect first block item case UNSET")`, `onError()` increments the error counter, and the session terminates without processing any block.
5. Repeat step 3 on each reconnect until `errors >= maxSubscribeAttempts`; the node is marked inactive and block ingestion halts for the readmit delay period.
6. Resume sending the malformed response after the readmit delay to sustain the denial of service indefinitely.
