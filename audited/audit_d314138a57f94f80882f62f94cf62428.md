### Title
Timeout Bypass via Shared Scheduler Saturation in `PollingTopicMessageRetriever.retrieve()`

### Summary
`PollingTopicMessageRetriever.retrieve()` applies `.timeout(retrieverProperties.getTimeout(), scheduler)` using the **same** `Schedulers.boundedElastic()` instance that drives the polling repeat loop. Because `Schedulers.boundedElastic()` returns the global shared scheduler, an unprivileged attacker who opens enough concurrent gRPC subscriptions can saturate all scheduler threads with polling tasks, causing the `TimeoutTask` to be queued and arbitrarily delayed. Subscriptions then run well beyond the configured 60-second timeout, consuming DB connections, threads, and memory without bound.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java`, constructor line 41:
```java
scheduler = Schedulers.boundedElastic();   // returns the GLOBAL shared instance
```

`retrieve()`, lines 51–62:
```java
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(...)
                .withScheduler(scheduler))          // polling occupies scheduler threads
        ...
        .timeout(retrieverProperties.getTimeout(), scheduler)  // timeout task queued on SAME scheduler
```

**Root cause:** `Schedulers.boundedElastic()` is a static factory that returns the **same global shared** `BoundedElasticScheduler` instance on every call (Reactor documented behavior: "Retrieves a default, shared, bounded elastic Scheduler"). The `timeout(Duration, Scheduler)` operator schedules a `TimeoutTask` runnable on the provided scheduler. If that scheduler's thread pool is fully occupied, the task is queued — the timeout fires only when a thread becomes free, not at the wall-clock deadline.

**Why existing checks fail:**
- No rate limiting, no max-subscriber enforcement, and no connection cap exist anywhere in the gRPC layer (confirmed: zero matches for `maxSubscribers`, `rateLimit`, `connectionLimit` in `grpc/**/*.java`).
- The `subscriberCount` gauge (line 52–55 of `TopicMessageServiceImpl`) is metrics-only; it enforces nothing.
- The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` at line 58 further extends subscription lifetime before the timeout even starts racing.
- The same global scheduler is also used by `PollingTopicListener`, `SharedPollingTopicListener`, `SharedTopicListener`, and `TopicMessageServiceImpl`, widening the saturation surface.

**Exploit flow:**
1. Attacker opens N concurrent gRPC `subscribeTopic` streams (no auth required).
2. Each subscription calls `retrieve(filter, true)` → schedules a polling repeat on the global `boundedElastic()` scheduler.
3. Each poll executes `topicMessageRepository.findByFilter(newFilter)` (a blocking DB stream) on a scheduler thread.
4. With N > `10 × CPU_cores` (e.g., 40 on a 4-core node), all scheduler threads are occupied.
5. The `TimeoutTask` for every active subscription is queued behind the polling tasks.
6. Subscriptions continue running indefinitely (or until the DB connection pool is exhausted), far beyond the 60-second timeout.

### Impact Explanation
The timeout is the **sole** resource-bounding mechanism for historical retrieval subscriptions. Bypassing it allows subscriptions to hold open DB connections (from `findByFilter` streams), consume heap for `PollingContext` objects, and keep scheduler threads busy — all indefinitely. On a typical deployment this translates to a full service DoS: DB connection pool exhaustion causes all new subscriptions and queries to fail, and the gRPC server becomes unresponsive. Severity: **High** (availability impact, no authentication required).

### Likelihood Explanation
The attacker needs only a gRPC client and the ability to open ~40–80 concurrent streams (one per scheduler thread). This is trivially achievable with any standard gRPC load-testing tool (e.g., `ghz`, `grpcurl` in a loop). The attack is repeatable and requires no special knowledge of the system internals — just the public gRPC endpoint. No credentials or elevated privileges are needed.

### Recommendation
1. **Use a dedicated `Schedulers.newBoundedElastic(...)` for timeout scheduling**, separate from the polling scheduler, so polling saturation cannot delay timeout delivery:
   ```java
   private final Scheduler pollingScheduler = Schedulers.newBoundedElastic(10, 1000, "retriever-poll");
   private final Scheduler timeoutScheduler = Schedulers.newBoundedElastic(10, 1000, "retriever-timeout");
   // ...
   .withScheduler(pollingScheduler)
   // ...
   .timeout(retrieverProperties.getTimeout(), timeoutScheduler)
   ```
2. **Enforce a maximum concurrent subscriber count** in `TopicMessageServiceImpl.subscribeTopic()`, rejecting new subscriptions when the limit is reached.
3. **Add gRPC-level connection/stream rate limiting** (e.g., via a `ServerInterceptor`) to prevent a single client from opening unbounded concurrent streams.
4. Consider using `Schedulers.parallel()` (fixed-size, non-blocking) for timeout scheduling, which is immune to blocking-task saturation.

### Proof of Concept

**Preconditions:** gRPC endpoint reachable, no authentication enforced, server has 4 CPU cores (40 scheduler threads).

**Steps:**
```bash
# 1. Open 50 concurrent long-running subscriptions (exceeds scheduler thread count)
for i in $(seq 1 50); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
    <host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 2. All 40 scheduler threads are now occupied with polling DB queries.
# 3. The TimeoutTask for each subscription is queued behind polling tasks.
# 4. After 60+ seconds, subscriptions are still alive (timeout not fired).
# 5. DB connection pool exhausts; new requests fail with connection timeout errors.

# Verify: observe hiero_mirror_grpc_subscribers metric stays elevated past 60s
# and DB connection pool metrics show exhaustion.
```
