### Title
N Sequential DB Queries via Multiple EVM Address `contract_id` Filters in `GET /contracts`

### Summary
`extractSqlFromContractFilters()` in `rest/controllers/contractController.js` iterates over all `contract_id` query filters and calls `ContractService.computeContractIdFromString()` for each one inside an `await`-in-loop pattern. When the value is a 20-byte EVM (CREATE2) address, `computeContractIdFromString()` issues a separate synchronous DB lookup per address. An unauthenticated attacker supplying N EVM-address `contract_id` values triggers N sequential DB queries before the main contracts query executes, enabling a non-network-based DoS against the database tier.

### Finding Description

**Exact code path:**

`GET /contracts` → `getContracts` (contractController.js line 745) → `extractSqlFromContractFilters(filters)` (line 750).

Inside `extractSqlFromContractFilters` (lines 83–144), the loop at line 102 processes every filter. For each `filterKeys.CONTRACT_ID` entry, line 105 unconditionally `await`s:

```js
const contractIdValue = await ContractService.computeContractIdFromString(filter.value);
``` [1](#0-0) 

`computeContractIdFromString` (contractService.js lines 468–476) checks whether the parsed value contains a `create2_evm_address` field. If it does, it calls `getContractIdByEvmAddress()`:

```js
if (contractIdParts.hasOwnProperty('create2_evm_address')) {
  return this.getContractIdByEvmAddress(contractIdParts);
}
``` [2](#0-1) 

`getContractIdByEvmAddress` (lines 452–466) executes a live DB query each time:

```js
const rows = await super.getRows(ContractService.contractIdByEvmAddressQuery, [
  Buffer.from(create2EvmAddress, 'hex'),
]);
``` [3](#0-2) 

The backing SQL is:

```sql
select id from entity e
where deleted <> true and type = 'CONTRACT' and evm_address = $1
``` [4](#0-3) 

Because the `await` is inside the `for` loop (not batched), each EVM address resolves sequentially: address 1 query completes → address 2 query starts → … → address N query completes → main contracts query runs. This is a classic N+1 query amplification.

**Why existing checks are insufficient:**

`buildAndValidateFilters` validates format and operator legality but imposes no hard cap on the number of repeated `contract_id` parameters that would prevent the amplification. Even if a `MAX_REPEATED_QUERY_PARAMS` constant exists in `utils.js`, the attacker can send many concurrent requests each carrying the maximum allowed number of EVM addresses, multiplying DB load linearly with concurrency. [5](#0-4) 

### Impact Explanation
Each request with N EVM-address `contract_id` values consumes N+1 DB connections/query slots sequentially. A modest number of concurrent attackers (e.g., 50 connections × 25 addresses each = 1,250 simultaneous DB queries) can saturate the database connection pool, stall legitimate queries, and degrade or halt the mirror node REST API. No authentication, API key, or special privilege is required. The endpoint is publicly accessible.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of any 20-byte hex strings (they do not need to correspond to real contracts — the DB query simply returns zero rows, but the round-trip still occurs). It is trivially scriptable, repeatable, and rate-limiting at the HTTP layer does not prevent amplification within a single request. Any external party can execute this.

### Recommendation
1. **Batch the EVM address lookups**: Collect all `create2_evm_address` values from the filter list first, then issue a single `WHERE evm_address = ANY($1)` query to resolve all of them at once before building the SQL conditions.
2. **Enforce a hard cap** on the number of repeated `contract_id` query parameters (e.g., ≤ 10) at the filter-validation layer in `buildAndValidateFilters` / `acceptedContractParameters`.
3. Consider moving the EVM-address resolution out of the filter-building loop entirely and into a dedicated pre-processing step that is aware of batching.

### Proof of Concept

```bash
# Generate 25 random 20-byte EVM addresses and send a single request
python3 -c "
import os, urllib.parse
addrs = ['0x' + os.urandom(20).hex() for _ in range(25)]
qs = '&'.join(f'contract_id={a}' for a in addrs)
print(f'GET /api/v1/contracts?{qs}')
"

# Fire 50 concurrent copies of that request
seq 50 | xargs -P50 -I{} curl -s \
  "https://<mirror-node-host>/api/v1/contracts?$(python3 -c "
import os
addrs=['0x'+os.urandom(20).hex() for _ in range(25)]
print('&'.join(f'contract_id={a}' for a in addrs))
")" -o /dev/null
```

Each of the 50 concurrent requests triggers 25 sequential `SELECT id FROM entity WHERE evm_address = $1` queries before the main contracts query, for a total of up to 1,300 DB queries per second from a single attacker machine. Database connection pool exhaustion and query queue buildup are observable within seconds.

  async getContractIdByEvmAddress(evmAddressFilter) {
    const create2EvmAddress = evmAddressFilter.create2_evm_address;
    const rows = await super.getRows(ContractService.contractIdByEvmAddressQuery, [
      Buffer.from(create2EvmAddress, 'hex'),
    ]);
    if (rows.length === 0) {
      throw new NotFoundError(`No contract with the given evm address 0x${create2EvmAddress} has been found.`);
    }
    // since evm_address is not a unique index, it is important to make this check.
    if (rows.length > 1) {
      throw new Error(`More than one contract with the evm address 0x${create2EvmAddress} have been found.`);
    }

    return rows[0].id;
  }
```
