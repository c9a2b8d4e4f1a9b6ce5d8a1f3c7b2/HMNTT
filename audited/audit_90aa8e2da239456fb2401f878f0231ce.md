### Title
Unauthenticated Connection Pool Exhaustion via Concurrent `GET /api/v1/accounts?account.publickey=` Requests

### Summary
The `getAccounts` handler in `rest/accounts.js` has no rate limiting or concurrency control. When a `public_key` filter is present, `queryQuietly` explicitly acquires a dedicated client from `global.pool` and holds it for the full query duration (up to the 20-second `statement_timeout`). Since `global.pool` defaults to only 10 connections and is shared by all API endpoints, an unauthenticated attacker sending 10+ concurrent requests can exhaust the pool and deny service to all other endpoints.

### Finding Description

**Code path:**

`rest/server.js` line 101 registers the route with no rate-limiting middleware: [1](#0-0) 

The middleware stack (`authHandler`, `requestLogger`, `metricsHandler`, `responseCacheCheckHandler`) contains no per-IP rate limiter or concurrency semaphore for unauthenticated users. [2](#0-1) 

In `getAccounts`, when `account.publickey` is supplied, `preQueryHint` is set to `constants.zeroRandomPageCostQueryHint`: [3](#0-2) 

`queryQuietly` in `rest/utils.js` takes the `preQueryHint` branch, which calls `this.connect()` â€” explicitly acquiring a **dedicated client** from the pool and holding it until `client.release()` in the `finally` block: [4](#0-3) 

`global.pool` is initialized with `max: config.db.pool.maxConnections` (default **10**) and `statement_timeout: config.db.pool.statementTimeout` (default **20000 ms**): [5](#0-4) [6](#0-5) 

**Root cause:** The failed assumption is that `statement_timeout` alone prevents resource exhaustion. It does not â€” it only bounds how long each connection is held. With 10 connections and a 20-second timeout, an attacker needs only 10 concurrent in-flight requests to hold all connections for the full 20-second window, then immediately re-issue them to maintain exhaustion indefinitely.

### Impact Explanation

All API endpoints (`/api/v1/transactions`, `/api/v1/balances`, `/api/v1/tokens`, etc.) share `global.pool`. Once all 10 connections are held, new requests queue until `connectionTimeoutMillis` (20 s) elapses, then fail with a `DbError`. The entire REST API becomes unavailable to all users. The `statement_timeout` provides no real protection because the attacker simply re-issues requests in a loop, maintaining continuous pool saturation.

### Likelihood Explanation

No authentication, API key, or IP-based rate limit is required. Any external user with a basic HTTP client (e.g., `curl`, `ab`, `wrk`) can reproduce this with a single shell loop. The attack is trivially repeatable and requires no knowledge of the system beyond the public API documentation. The default pool size of 10 makes the threshold extremely low.

### Recommendation

1. **Add a concurrency/rate-limit middleware** to the Node.js REST API (e.g., `express-rate-limit` or a semaphore) specifically for unauthenticated callers on expensive list endpoints.
2. **Increase `maxConnections`** from the default of 10 to a value appropriate for expected concurrency, so that a small number of slow queries cannot exhaust the pool.
3. **Reduce `statement_timeout`** for the `mirror_api` DB user (currently 20 s) to a lower value (e.g., 5 s) to shrink the exhaustion window.
4. **Consider a per-route query semaphore** to cap the number of simultaneous in-flight DB queries for the `/accounts` list endpoint.

### Proof of Concept

```bash
# Exhaust the pool (default max=10) with concurrent public_key queries
# Replace <HOST> with the target mirror node REST API host

for i in $(seq 1 15); do
  curl -s "http://<HOST>/api/v1/accounts?account.publickey=0x$(head -c 32 /dev/urandom | xxd -p | tr -d '\n')" &
done
wait

# Immediately after, all other endpoints return connection timeout errors:
curl -v "http://<HOST>/api/v1/transactions"
# Expected: 500 DbError (connection pool exhausted / connectionTimeoutMillis exceeded)
```
