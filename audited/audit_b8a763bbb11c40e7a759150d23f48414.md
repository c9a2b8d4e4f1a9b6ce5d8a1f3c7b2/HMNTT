### Title
Missing Server-Side gRPC Keepalive Allows Indefinite Resource Retention on Network Partition

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server without setting `keepAliveTime` or `keepAliveTimeout`, so the server never sends HTTP/2 PING frames to probe connection liveness. When a TCP connection is silently partitioned (no FIN/RST), the server cannot detect the dead client, and all associated Reactor subscription pipelines, bounded-elastic scheduler threads, and gRPC channel resources remain allocated indefinitely. An unprivileged external user can trigger this by subscribing and then silently dropping the connection.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // No keepAliveTime, keepAliveTimeout, maxConnectionAge, or maxConnectionIdle set
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java` (lines 11–15) exposes only `maxConcurrentCallsPerConnection = 5`; no keepalive fields exist.

**Root cause:** `NettyServerBuilder` defaults to `keepAliveTime = Long.MAX_VALUE` (effectively disabled). Without periodic PING frames, the server has no application-layer mechanism to detect a silently-partitioned TCP connection. The failed assumption is that the OS TCP stack will reliably signal dead peers — it does not in network partition scenarios (default OS keepalive is ~2 hours).

**Exploit flow:**

1. Attacker calls `ConsensusController.subscribeTopic()` with no `endTime` (infinite subscription). The controller registers `setOnCancelHandler(disposable::dispose)` (line 51), but this handler only fires when the gRPC framework itself detects stream cancellation — which requires connection-level detection.
2. `TopicMessageServiceImpl.subscribeTopic()` (lines 59–91) creates a live Reactor pipeline via `topicListener.listen()` → `incomingMessages()`. The pipeline increments `subscriberCount` on subscribe and decrements it only in `doFinally` — which never fires for a silently-dead connection.
3. Attacker uses `iptables -A OUTPUT -p tcp --dport 5600 -j DROP` (or equivalent) to silently partition the connection. No FIN/RST is sent.
4. The server's Reactor pipeline continues running. When the gRPC send buffer fills (flow control), the pipeline stalls but remains allocated — it does not time out or self-cancel.
5. Attacker repeats with many connections. Each connection holds up to 5 streams (`maxConcurrentCallsPerConnection = 5`), each stream holding a Reactor subscription, a bounded-elastic scheduler thread slot, and a gRPC channel object.

**Why existing checks fail:**

- `retriever.timeout = 60s` (`RetrieverProperties`, line 28): applies only to the historical retrieval phase (`PollingTopicMessageRetriever.retrieve()`, line 59 `.timeout(...)`), not to the live `topicListener.listen()` pipeline.
- `maxConcurrentCallsPerConnection = 5`: limits streams per connection but does not detect or evict dead connections.
- `listener.maxBufferSize = 16384`: the shared listener emits an error to subscribers when the buffer overflows, but delivering that error to a dead client fails silently — the Reactor `onError` path calls `responseObserver::onError`, which attempts a gRPC write that blocks or is dropped.
- `setOnCancelHandler(disposable::dispose)` (line 51): only fires on gRPC-detected cancellation; never fires for a silently-partitioned connection.
- OS TCP keepalive: default idle time is ~7200 s (2 hours), far too slow to be a practical mitigation.

### Impact Explanation
Each silently-dead connection retains: up to 5 active Reactor subscription chains (each polling the database or Redis on a `boundedElastic` scheduler thread), the gRPC Netty channel object, and an inflated `subscriberCount` gauge. With enough dead connections, the bounded-elastic thread pool (`Schedulers.boundedElastic()`, default cap 10× CPU cores) is exhausted, blocking all live subscribers from receiving messages. Secondary effects include database connection pool saturation (each live polling subscription issues periodic queries) and Redis subscription slots. This constitutes a denial-of-service against live clients requiring no authentication.

### Likelihood Explanation
Any client that can reach port 5600 can trigger this. The attacker needs only to: (a) open a `subscribeTopic` stream, and (b) silently drop outbound packets (trivially done with `iptables` on the client host, or by exploiting any middlebox). No credentials, no special protocol knowledge, and no rate limiting on connection establishment is present in the reviewed configuration. The attack is repeatable and scalable — each new TCP connection is a new resource slot. Cloud and Kubernetes environments where clients may be behind NAT or load balancers that silently time out idle connections make accidental triggering plausible as well.

### Recommendation
Add the following to `grpcServerConfigurer()` in `GrpcConfiguration.java`:

```java
serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);       // send PING every 30s of inactivity
serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);    // close if no PING ACK within 10s
serverBuilder.maxConnectionIdle(300, TimeUnit.SECONDS);  // close idle connections after 5 min
serverBuilder.maxConnectionAge(3600, TimeUnit.SECONDS);  // recycle connections after 1 hour
serverBuilder.permitKeepAliveWithoutCalls(false);        // reject client PINGs on idle streams
```

Expose these as fields in `NettyProperties` so they are operator-configurable. Additionally, apply a `timeout()` operator on the live `topicListener.listen()` pipeline in `TopicMessageServiceImpl.incomingMessages()` to bound the maximum duration of a stalled stream.

### Proof of Concept
```bash
# 1. Subscribe to a topic (infinite, no endTime)
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
  <server>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
CLIENT_PID=$!

# 2. Silently partition the connection (no FIN/RST sent to server)
iptables -A OUTPUT -p tcp --dport 5600 -j DROP

# 3. Observe server retains the subscription indefinitely
# Check subscriber gauge via actuator:
curl http://<server>:8080/actuator/metrics/hiero.mirror.grpc.subscribers
# Count remains elevated; repeat steps 1-2 with many connections
# to exhaust the boundedElastic thread pool and block live subscribers

# 4. Cleanup
kill $CLIENT_PID
iptables -D OUTPUT -p tcp --dport 5600 -j DROP
```
