### Title
Unbounded Concurrent Subscriptions in `PollingTopicListener` Saturate Shared `boundedElastic()` Scheduler, Causing Polling Delays for All Subscribers

### Summary
`PollingTopicListener.listen()` creates an independent `PollingContext` and schedules recurring DB polls on `Schedulers.boundedElastic()` for every incoming subscription, with no cap on total subscriptions or connections. Because `GrpcConfiguration` sets only `maxConcurrentCallsPerConnection = 5` with no total-connection limit, an unauthenticated attacker can open arbitrarily many TCP connections and flood the shared `boundedElastic` scheduler, degrading or stalling polling for all legitimate subscribers.

### Finding Description

**Exact code path:**

Every call allocates a fresh `PollingContext` and chains `delaySubscription` + `repeatWhen` both pinned to the single shared `Schedulers.boundedElastic()` instance declared at line 31: [2](#0-1) 

The same global `boundedElastic` scheduler is also used by `TopicMessageServiceImpl`'s safety-check path: [3](#0-2) 

**Only mitigation present â€” `maxConcurrentCallsPerConnection = 5`:** [4](#0-3) 

`GrpcConfiguration` calls only `maxConcurrentCallsPerConnection`; there is no `maxConnections`, `maxConnectionAge`, or any other server-level connection cap. An attacker opens *C* TCP connections Ă— 5 streams = 5*C* concurrent subscriptions.

**No subscription-count enforcement:** `subscriberCount` in `TopicMessageServiceImpl` is a Micrometer gauge only â€” it is never checked against a maximum: [5](#0-4) 

**No gRPC-layer throttle:** The `ThrottleConfiguration`/`ThrottleManagerImpl` rate-limiter exists only in the `web3` module; there is no equivalent for gRPC topic subscriptions.

**Root cause:** The design assumes a small, bounded subscriber population. The failed assumption is that the transport-layer `maxConcurrentCallsPerConnection` limit is sufficient to bound total scheduler load. It is not, because the number of connections is unlimited.

### Impact Explanation

Reactor's `boundedElastic` scheduler defaults to `10 Ă— availableProcessors` threads with a 100,000-task queue. With thousands of subscriptions each enqueuing a poll task every 500 ms (default `interval`), the queue fills and tasks are delayed far beyond the configured interval. This causes:

- **Polling delays across all subscribers** â€” legitimate subscribers receive messages late or not at all within a time-bounded window (`endTime` filter), constituting missing records from their perspective.
- **Safety-check starvation** â€” `TopicMessageServiceImpl`'s 1-second safety-check `Mono.delay` also runs on `boundedElastic`; under saturation it never fires, leaving gap-filling for missing messages broken for every active subscription.
- **Database connection pool exhaustion** â€” each delayed-but-eventually-executed poll issues a `findByFilter` query; a backlog of thousands of queued polls can exhaust the JDBC connection pool, cascading to other components.

Severity: **High** â€” complete denial of the mirror-node's HCS subscription service for all users.

### Likelihood Explanation

- No authentication is required to call `subscribeTopic`; the gRPC endpoint is publicly reachable.
- Opening thousands of HTTP/2 connections is trivial with standard tooling (`grpc-bench`, `ghz`, or a simple loop with any gRPC client library).
- The attack is repeatable and requires no special knowledge beyond the public protobuf API.
- The condition is that the deployment uses `ListenerType.POLL` (not the default `REDIS`), which reduces exposure but does not eliminate it â€” operators do configure POLL mode, and the `SHARED_POLL` path does not share this specific flaw only because it uses a single shared `Flux`. [6](#0-5) 

### Recommendation

1. **Enforce a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: compare `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add a total-connection limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge(...)` and a `maxConnections` equivalent (e.g., via a custom `ServerInterceptor` that counts active calls globally).
3. **Add per-IP or per-client subscription rate limiting** as a `ServerInterceptor`, analogous to the `ThrottleManagerImpl` in the `web3` module.
4. **Isolate the `PollingTopicListener` scheduler** â€” instead of relying on the global `Schedulers.boundedElastic()`, create a dedicated, capacity-bounded `Schedulers.newBoundedElastic(...)` instance with a hard cap on threads and queue depth, so saturation is contained and does not bleed into other reactor pipelines.

### Proof of Concept

**Preconditions:** Mirror node configured with `hiero.mirror.grpc.listener.type=POLL`; gRPC port reachable without authentication.

**Steps:**

```bash
# Open 2000 concurrent long-lived subscriptions (400 connections Ă— 5 streams each)
# using ghz or a simple Go/Java loop:
for i in $(seq 1 400); do
  ghz --insecure \
      --proto consensus_service.proto \
      --call com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
      --data '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
      --concurrency 5 \
      --duration 300s \
      <mirror-node-host>:5600 &
done
wait
```

**Trigger:** After ~200â€“400 connections the `boundedElastic` task queue begins to fill. Poll tasks scheduled at 500 ms intervals are delayed by seconds to tens of seconds.

**Observable result:**
- Legitimate subscribers stop receiving messages or receive them with multi-second latency.
- `hiero.mirror.grpc.subscribers` gauge spikes to thousands.
- JVM thread dump shows `boundedElastic` queue depth near 100,000.
- DB connection pool metrics show exhaustion.
- Subscriptions with an `endTime` miss messages that arrived during the polling gap, producing incomplete record exports.

```java
    private final Scheduler scheduler = Schedulers.boundedElastic();

    @Override
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
```
