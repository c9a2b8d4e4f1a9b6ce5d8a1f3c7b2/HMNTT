### Title
Unindexed `amount > 0` Filter in `getSubQuery()` Enables Read-Amplification DoS Against Database

### Summary
The `getSubQuery()` function in `rest/service/tokenAllowanceService.js` unconditionally appends `amount > 0` to every `token_allowance` query, but no index exists on the `amount` column. PostgreSQL must perform a full index scan over all rows for a given `owner` using the primary key `(owner, spender, token_id)`, then discard every row where `amount = 0` as a heap filter. An unprivileged attacker can target any account whose allowances are predominantly exhausted (amount=0), causing the database to scan thousands-to-millions of rows per request while returning almost nothing, with no rate limiting on the REST API to prevent it.

### Finding Description
**Exact code path:**

`rest/service/tokenAllowanceService.js`, lines 13 and 30–46:
```js
static amountCondition = `${TokenAllowance.AMOUNT} > 0`;   // line 13

getSubQuery(filters, params, accountIdCondition, limitClause, orderClause) {
  const conditions = [
    accountIdCondition,
    TokenAllowanceService.amountCondition,   // always injected, line 33
    ...filters.map(...)
  ];
  ...
}
```

This produces SQL of the form:
```sql
SELECT * FROM token_allowance
WHERE owner = $1 AND amount > 0
ORDER BY spender ASC, token_id ASC
LIMIT $2
```

**Root cause — missing index on `amount`:**

The `token_allowance` table's only index is the primary key `(owner, spender, token_id)` (migration `V2.0.3__index_init.sql`, line 230). There is no index on `amount`, and no partial index such as `WHERE amount > 0`. PostgreSQL satisfies `owner = $1` via the primary key index, then evaluates `amount > 0` as a post-scan heap filter against every row belonging to that owner. The `LIMIT $2` clause (capped at 100) limits *output* rows only — it does not short-circuit the scan when most rows are filtered out.

**Why existing checks fail:**

- The `LIMIT` clause (default 25, max 100) caps returned rows, not scanned rows.
- The REST API (Node.js) has **no rate limiting** for this endpoint. The throttle configuration found (`web3/ThrottleConfiguration.java`) applies exclusively to the Spring Boot web3 module (contract calls), not to the REST API.
- No authentication is required to call `GET /api/v1/accounts/{id}/allowances/tokens`.

### Impact Explanation
On a production network, high-activity accounts accumulate large numbers of exhausted allowances (amount=0) naturally over time. A single HTTP request targeting such an account forces the database to read and discard every zero-amount row for that owner before returning the small set with `amount > 0`. With an account holding, e.g., 500,000 exhausted allowances, one request generates the same DB I/O as ~500,000 normal row reads. Because the REST API is unauthenticated and unthrottled, even a modest request rate (a few requests per second) against a well-chosen target account can sustain a disproportionate database I/O load, plausibly exceeding 30% above baseline on a lightly-to-moderately loaded node.

### Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no brute-force volume. The attacker only needs to identify one high-activity account with many exhausted allowances — information that is publicly observable on-chain. The setup cost is zero if a suitable account already exists (which is expected on mainnet). The attack is repeatable indefinitely and is invisible to application-layer monitoring because each individual request is syntactically valid and returns HTTP 200.

### Recommendation
1. **Add a partial index** on `token_allowance` to make the `amount > 0` filter index-supported:
   ```sql
   CREATE INDEX token_allowance__owner_amount
     ON token_allowance (owner, spender, token_id)
     WHERE amount > 0;
   ```
   This allows PostgreSQL to satisfy both `owner = $1` and `amount > 0` from the index alone, skipping zero-amount rows entirely.

2. **Add rate limiting to the REST API** for the `/accounts/:id/allowances/tokens` endpoint (and similar allowance endpoints), mirroring the throttle mechanism already present in the web3 module.

3. **Consider a `statement_timeout`** at the database role level for the REST API user to bound the maximum wall-clock time any single query can consume.

### Proof of Concept
**Preconditions:**
- Identify (or create) an account `0.0.X` on the network that has a large number of token allowances with `amount = 0` (exhausted allowances are common on mainnet for active DeFi accounts).

**Trigger:**
```
GET /api/v1/accounts/0.0.X/allowances/tokens
```
No authentication header required.

**Observable result:**
- The REST API returns HTTP 200 with 0 or very few allowances.
- The database executes:
  ```sql
  SELECT * FROM token_allowance
  WHERE owner = <X> AND amount > 0
  ORDER BY spender ASC, token_id ASC
  LIMIT 25
  ```
  PostgreSQL's `EXPLAIN ANALYZE` will show `Rows Removed by Filter: N` where N equals the number of zero-amount rows for that owner — potentially hundreds of thousands — while `Rows` returned is near zero.
- Repeating this request at even 5–10 req/s against a sufficiently large account sustains continuous full-owner-partition scans, measurably elevating database I/O and CPU above the 24-hour baseline.
