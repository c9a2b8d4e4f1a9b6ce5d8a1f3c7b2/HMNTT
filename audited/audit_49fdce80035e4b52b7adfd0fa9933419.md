### Title
Unbounded Indefinite Subscriptions Enable Resource Exhaustion DoS via PollingTopicListener

### Summary
`PollingTopicListener.listen()` creates a per-subscriber `PollingContext` and schedules it to poll the database indefinitely using `RepeatSpec.times(Long.MAX_VALUE)` with no server-enforced maximum subscription duration or concurrent-subscription cap. An unauthenticated attacker can open arbitrarily many long-lived subscriptions, exhausting heap memory (via retained `PollingContext` objects), saturating the `boundedElastic` scheduler task queue, and flooding the database with polling queries.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 34–49:

```java
PollingContext context = new PollingContext(filter);   // line 35 – one object per subscriber
...
.repeatWhen(RepeatSpec.times(Long.MAX_VALUE)           // line 40 – effectively infinite
        .withFixedDelay(interval)                      // line 42 – polls every 500 ms by default
        .withScheduler(scheduler))                     // line 43
```

Each call to `listen()` allocates a fresh `PollingContext` (lines 65–79) containing an `AtomicLong` and an `AtomicReference<TopicMessage>`. The Flux chain captures `context` via the `Flux.defer(() -> poll(context))` lambda (line 38) and the `doOnNext(context::onNext)` hook (line 47). As long as the Flux is subscribed, the JVM cannot collect `context` or any `TopicMessage` it references.

**Root cause – failed assumptions:**

1. `ListenerProperties` (lines 17–43) defines no `maxSubscriptions`, no `maxSubscriptionDuration`, and no per-client rate limit.
2. `TopicMessageFilter` (lines 19–51) makes `endTime` and `limit` fully optional; `hasLimit()` returns `false` when `limit == 0` (line 39–41), so a client sending `limit=0` and no `endTime` gets an infinite stream.
3. `ConsensusController.toFilter()` (lines 55–73) only sets `endTime` and `limit` when the client supplies them; it never enforces a server-side cap.
4. The only teardown path is the gRPC cancel handler (`disposable::dispose`, line 51 of `ConsensusController`), which fires only on client-initiated cancellation — a cooperative mechanism an attacker simply ignores.

**Exploit flow:**

An attacker opens *N* concurrent gRPC `subscribeTopic` calls, each with `limit=0` and no `consensusEndTime`. For each call:
- A `PollingContext` is allocated and pinned in the Flux chain.
- Every `interval` ms (default 500 ms), `poll()` executes a `findByFilter` DB query.
- The `boundedElastic` scheduler queues tasks; when saturated, the internal task queue grows without bound.

### Impact Explanation
- **Memory**: *N* `PollingContext` objects (plus captured `TopicMessage` references) are permanently retained until the attacker closes connections. With large *N* this causes heap pressure and eventually OOM.
- **Database**: *N* × 2 queries/second (at default 500 ms interval) are issued continuously. 1 000 open subscriptions produce ~2 000 DB queries/second with zero legitimate traffic.
- **Scheduler starvation**: `Schedulers.boundedElastic()` has a bounded thread count but an unbounded task queue; flooding it delays or starves legitimate subscribers.
- **Severity**: High — full denial of service achievable without authentication.

### Likelihood Explanation
The gRPC endpoint is publicly reachable (documented with `grpcurl` examples, no auth layer visible in the controller). The exploit requires only a standard gRPC client, a valid topic ID (discoverable from the mirror node REST API), and the ability to hold TCP connections open. It is trivially scriptable and repeatable.

### Recommendation
1. **Enforce a server-side maximum subscription duration**: add a `maxSubscriptionDuration` property to `ListenerProperties` and apply `.timeout(maxSubscriptionDuration)` to the Flux returned by `listen()`.
2. **Cap concurrent subscriptions per IP / globally**: track `subscriberCount` (already present in `TopicMessageServiceImpl`) and reject new subscriptions above a configurable threshold.
3. **Enforce a minimum non-zero `endTime` or `limit`** for the `POLL` listener type, or default `endTime` to `now + maxSubscriptionDuration` when the client omits it.
4. **Add gRPC-level connection/stream rate limiting** via Netty or a reverse proxy in front of the service.

### Proof of Concept
```bash
# Open 500 indefinite subscriptions in parallel (requires grpcurl and a known topicNum)
for i in $(seq 1 500); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}}' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
wait
# Each connection holds a PollingContext alive and issues a DB query every 500 ms.
# Monitor heap: jmap -histo <pid> | grep PollingContext
# Monitor DB: SELECT count(*) FROM pg_stat_activity WHERE query LIKE '%topic_message%';
```
Expected result: heap shows 500 `PollingContext` instances; DB shows ~1 000 active queries/second; legitimate subscribers experience degraded or no response.
