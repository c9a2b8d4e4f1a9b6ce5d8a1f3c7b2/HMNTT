### Title
Unbounded `json_agg` on `transaction_signature` Enables Response-Size DoS via Schedule Signature Inflation

### Summary
The `transactionSignatureJsonAgg` expression in `rest/schedules.js` uses PostgreSQL `json_agg()` with no `LIMIT` clause on the number of signatures aggregated per schedule. Both the `/api/v1/schedules/{id}` and `/api/v1/schedules` endpoints load all signatures for a schedule into a single JSON array in one query, with no cap on array size. An attacker who submits many `ScheduleSign` transactions against a long-term schedule can cause the mirror node to produce arbitrarily large response payloads, exhausting Node.js heap memory and denying gossip clients access to signature data.

### Finding Description

**Exact code location:**

`rest/schedules.js` lines 39–45 — the shared aggregation expression:
```js
const transactionSignatureJsonAgg = `
  json_agg(json_build_object(
    'consensus_timestamp', ts.consensus_timestamp,
    'public_key_prefix', encode(ts.public_key_prefix, 'base64'),
    'signature', encode(ts.signature, 'base64'),
    'type', ts.type
  ) order by ts.consensus_timestamp)`;
``` [1](#0-0) 

This expression is embedded verbatim in two places with no `LIMIT`:

1. `getScheduleByIdQuery` (lines 46–64): `left join transaction_signature ts on ts.entity_id = s.schedule_id` — all rows for the schedule are aggregated. [2](#0-1) 

2. `signatureQuery` inside `getSchedules` (lines 255–259): `where entity_id in (${positions})` — all signatures for every schedule in the page are aggregated. [3](#0-2) 

**Root cause and failed assumption:**

The design assumes the number of signatures per schedule is naturally bounded by the key structure of the scheduled transaction. However, the `transaction_signature` table has **no unique constraint** (explicitly noted in the design doc: *"There's no unique constraint/primary key since the client can potentially sign multiple times with the same key"*). [4](#0-3) 

The schema confirms this: [5](#0-4) 

This means the same key can produce multiple rows across repeated `ScheduleSign` transactions, and the mirror node stores all of them. The `json_agg()` then materializes every row into a single JSON array in one DB round-trip, which is then held entirely in Node.js heap before being sent via `res.send()`. [6](#0-5) 

**Exploit flow:**

1. Attacker creates a long-term schedule (`wait_for_expiry=true`, expiry up to 62 days).
2. Attacker repeatedly submits `ScheduleSign` transactions — each transaction's `sigMap` can carry multiple sig pairs (each ~96 bytes of key+signature data), and the same key may be reused across transactions, creating new rows each time.
3. Over the schedule's lifetime, thousands of rows accumulate in `transaction_signature` for that `entity_id`.
4. Any client (including the attacker) queries `GET /api/v1/schedules/{scheduleId}`.
5. `getScheduleByIdQuery` executes `json_agg()` over all accumulated rows with no row limit.
6. PostgreSQL materializes the full array; the Node.js process receives and holds the entire result in heap before calling `res.send()`.
7. Repeated concurrent requests against the same schedule ID exhaust Node.js heap, causing OOM or severe GC pressure that degrades all API responses.

### Impact Explanation

The mirror node REST API is the primary interface for gossip clients to retrieve schedule signature data. An OOM crash or severe heap pressure in the single Node.js process takes down the entire REST API, not just the schedule endpoint. Even short of OOM, a response of tens of megabytes per request ties up the event loop and connection pool slots (`maxConnections: 10` by default), starving other legitimate requests. The `response.limit` parameter (default 25, max 100) controls only the number of *schedules* returned, not the number of *signatures per schedule* — so pagination provides no protection here. [7](#0-6) 

### Likelihood Explanation

Each `ScheduleSign` transaction costs HBAR fees, which is a real cost barrier. However, for a well-resourced attacker or one targeting a specific high-value schedule, the cost is manageable: a long-term schedule with a 62-day window allows thousands of low-cost sign transactions to be submitted over time. The attack is also amplifiable — once a schedule with many signatures exists, any unauthenticated client can trigger the large response repeatedly at zero additional cost. The `statement_timeout` of 20 seconds for the REST DB user provides no protection because `json_agg()` over an indexed `entity_id` column completes quickly even for large row counts; the damage is in response size, not query duration. [8](#0-7) 

### Recommendation

1. **Add a `LIMIT` inside the aggregation subquery** for signatures, e.g., wrap the `transaction_signature` join in a subquery with `LIMIT N` (e.g., 1000) before passing to `json_agg()`.
2. **Add a dedicated `maxSignaturesPerSchedule` config** analogous to `tokenBalance.singleAccount` (currently 1000) and enforce it at the SQL level.
3. **Add a response body size guard** in `responseHandler.js` that returns HTTP 500 or truncates before calling `res.send()` if the serialized body exceeds a configured threshold.
4. Consider adding a unique constraint or deduplication on `(entity_id, public_key_prefix, consensus_timestamp)` in `transaction_signature` to prevent unbounded row growth from repeated signing with the same key.

### Proof of Concept

```
# 1. Create a long-term schedule (wait_for_expiry=true, expiry=62 days from now)
#    via ScheduleCreate transaction on the Hedera network.
#    Record the resulting schedule_id (e.g., 0.0.12345).

# 2. Submit thousands of ScheduleSign transactions referencing 0.0.12345,
#    each carrying multiple sig pairs (same or different keys).
#    Each transaction is accepted by the network and creates rows in transaction_signature.

# 3. After accumulating N rows (e.g., N=50,000):
for i in $(seq 1 100); do
  curl -s "https://<mirror-node>/api/v1/schedules/0.0.12345" &
done
wait

# 4. Observe: each request causes the mirror node to execute json_agg() over 50,000 rows,
#    producing a ~50MB+ JSON response. 100 concurrent requests consume ~5GB of Node.js heap,
#    triggering OOM or severe GC pauses that deny service to all API clients.
```
