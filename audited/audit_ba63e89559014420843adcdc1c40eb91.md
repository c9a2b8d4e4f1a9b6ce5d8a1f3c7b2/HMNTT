### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `/account/balance` Flood

### Summary
The `AccountBalance()` handler in `rosetta/app/services/account_service.go` accepts unauthenticated requests and triggers up to 5 sequential database queries per call through `RetrieveBalanceAtBlock()`. With no application-level concurrency control and the Traefik rate-limiting middleware disabled by default, a flood of concurrent requests from an unprivileged attacker can exhaust the 100-connection pool, rendering the rosetta service unresponsive to all further requests.

### Finding Description

**Exact code path:**

`AccountBalance()` at [1](#0-0)  calls `RetrieveBalanceAtBlock()` which chains three DB-acquiring functions:

- `getCryptoEntity()` — acquires a connection, runs a UNION query across `entity` + `entity_history` [2](#0-1) 
- `getLatestBalanceSnapshot()` — acquires a connection twice: once for `selectPreviousPartitionLowerBound`, once for `latestBalanceBeforeConsensus` [3](#0-2) 
- `getBalanceChange()` — acquires a connection for `balanceChangeBetween` [4](#0-3) 

Each `GetDbWithContext()` call acquires a connection from the shared pool and holds it for the duration of that query (up to `statementTimeout` = 20 s): [5](#0-4) 

The pool is capped at `MaxOpenConnections = 100` (default): [6](#0-5) 

**Root cause — no application-level concurrency gate.** The only middleware applied in `main()` is metrics and tracing: [7](#0-6) 

The Traefik `inFlightReq` (5/IP) and `rateLimit` (10 avg/host) middleware exist only in the Helm chart values and are **disabled by default** (`global.middleware: false`): [8](#0-7) 

The middleware template is gated on that flag: [9](#0-8) 

### Impact Explanation

When 100+ concurrent requests are in-flight, every new request blocks waiting for a free connection. Because each request holds connections sequentially for up to 20 s each (statement timeout), the pool stays saturated. All subsequent requests — including health checks and block queries — queue indefinitely, making the rosetta instance fully unresponsive. With the HPA minimum of 1 replica and a typical small deployment of 2–3 replicas, taking down a single instance crosses the ≥30% threshold. The shared PostgreSQL backend also suffers elevated load, potentially cascading to other mirror-node components (importer, REST API) that share the same database host.

### Likelihood Explanation

No credentials, tokens, or special network position are required. Any internet-accessible rosetta endpoint (the ingress exposes `/rosetta/account` by default) can be targeted. The attacker needs only to send valid `shard.realm.num` account identifiers (trivially enumerable from public ledger data) in concurrent HTTP POST requests. The attack is repeatable and scriptable with standard tools (`wrk`, `hey`, `ab`). The default deployment does not enable the Traefik middleware chain, so the protection is absent in the most common configuration.

### Recommendation

1. **Add an application-level semaphore** in `AccountBalance()` (or in a shared HTTP middleware) to cap total in-flight DB-bound requests, independent of the ingress layer.
2. **Enable the Traefik middleware by default** (`global.middleware: true`) and lower `inFlightReq.amount` to a value consistent with `MaxOpenConnections / queries_per_request` (e.g., ≤20).
3. **Reduce `statementTimeout`** from 20 s to a tighter value (e.g., 5 s) to release connections faster under load.
4. **Add a `MaxOpenConnections`-aware semaphore** in `RetrieveBalanceAtBlock()` so that the number of concurrent callers never exceeds `MaxOpenConnections / 4`.
5. Consider **caching** the latest block result (already partially done for entities) to reduce per-request DB round-trips.

### Proof of Concept

```bash
# Prerequisites: rosetta endpoint reachable, no Traefik middleware enabled (default)
# Step 1: identify any valid account (e.g., 0.0.2 is always present)
# Step 2: flood with 200 concurrent requests

seq 1 200 | xargs -P200 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST http://<rosetta-host>/account/balance \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Hedera","network":"mainnet"},
    "account_identifier": {"address":"0.0.{}"}
  }'

# Expected result:
# - First ~100 requests return 200
# - Subsequent requests hang or return 500/timeout
# - Health endpoint /health/liveness also becomes unresponsive
# - DB connection pool metrics show 100/100 active connections
```

func (ar *accountRepository) getCryptoEntity(ctx context.Context, accountId types.AccountId, consensusEnd int64) (
	*domain.Entity,
	*rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var query string
	var args []any
	consensusEndArg := sql.Named("consensus_end", getInclusiveInt8Range(consensusEnd, consensusEnd))
	if accountId.HasAlias() {
		query = selectCryptoEntityByAlias
		args = []any{
			sql.Named("alias", accountId.GetAlias()),
			consensusEndArg,
		}
	} else {
		query = selectCryptoEntityById
		args = []any{
			consensusEndArg,
			sql.Named("id", accountId.GetId()),
		}
	}

	entities := make([]domain.Entity, 0)
	if err := db.Raw(query, args...).Scan(&entities).Error; err != nil {
		log.Errorf(
			databaseErrorFormat,
			hErrors.ErrDatabaseError.Message,
			fmt.Sprintf("%v looking for entity %s", err, accountId),
		)
		return nil, hErrors.ErrDatabaseError
	}

	if len(entities) == 0 {
		return nil, nil
	}

	// if it's by alias, return the first match which is the current one owns the alias, even though it may be deleted
	return &entities[0], nil
}

func (ar *accountRepository) getLatestBalanceSnapshot(ctx context.Context, accountId, timestamp int64) (
	int64,
	*types.HbarAmount,
	*rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var partitionLowerBound int64
	if err := db.Raw(
		selectPreviousPartitionLowerBound,
		sql.Named("timestamp", timestamp),
	).Scan(&partitionLowerBound).Error; err != nil {
		log.Errorf(
			databaseErrorFormat,
			hErrors.ErrDatabaseError.Message,
			fmt.Sprintf("%v looking for previous account_balance partition before %d", err, timestamp),
		)
	}

	// gets the most recent balance at or before timestamp
	ab := &accountBalance{}
	if err := db.Raw(
		latestBalanceBeforeConsensus,
		sql.Named("account_id", accountId),
		sql.Named("lower_bound", partitionLowerBound),
		sql.Named("timestamp", timestamp),
		sql.Named("treasury_entity_id", ar.treasuryEntityId.EncodedId),
	).First(ab).Error; err != nil {
		log.Errorf(
			databaseErrorFormat,
			hErrors.ErrDatabaseError.Message,
			fmt.Sprintf("%v looking for account %d's balance at or before %d", err, accountId, timestamp),
		)
		return 0, nil, hErrors.ErrDatabaseError
	}

	if ab.ConsensusTimestamp == 0 {
		return 0, nil, hErrors.ErrNodeIsStarting
	}

	hbarAmount := types.HbarAmount{Value: ab.Balance}

	return ab.ConsensusTimestamp, &hbarAmount, nil
}

func (d *client) GetDbWithContext(ctx context.Context) (*gorm.DB, context.CancelFunc) {
	if d.statementTimeout <= 0 {
		db := d.db
		if ctx != nil {
			db = db.WithContext(ctx)
		}
		return db, noop
	}

	if ctx == nil {
		ctx = context.Background()
	}

	// #nosec G118
	childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
	return d.db.WithContext(childCtx), cancel
}
