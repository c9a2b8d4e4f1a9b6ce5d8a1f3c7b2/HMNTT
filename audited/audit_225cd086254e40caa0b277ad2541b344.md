### Title
Unauthenticated Repeated Maximum-Limit Token Queries Enable DB Throughput Degradation

### Summary
The `getTokensRequest()` handler in `rest/tokens.js` accepts an unauthenticated `GET /tokens?limit=<max>` request and executes a full-table scan up to the configured maximum row count (default 100) on every call. The REST API server (`server.js`) registers no application-level rate-limiting middleware, so any external caller can flood the endpoint with maximum-limit queries, saturating the shared database connection pool and degrading query throughput for all other users.

### Finding Description

**Code path:**

1. `rest/server.js` line 119 registers the route with no rate-limit middleware:
   ```js
   app.getExt(`${apiPrefix}/tokens`, tokens.getTokensRequest);
   ```
   The middleware stack (`server.js` lines 68–144) includes `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, optional metrics, and optional cache — but **no rate limiter**.

2. `getTokensRequest()` (`rest/tokens.js` lines 360–422) calls `utils.buildAndValidateFilters` with the custom validator `validateTokenQueryFilter`.

3. `validateTokenQueryFilter` (`rest/tokens.js` lines 330–331) validates the `limit` parameter only with `utils.isPositiveLong(val)` — it checks that the value is a positive integer but does **not** enforce the configured maximum at the validation stage.

4. `extractSqlFromTokenRequest` (`rest/tokens.js` lines 171–173) assigns the limit directly from the filter value:
   ```js
   if (filter.key === filterKeys.LIMIT) {
     limit = filter.value;
     continue;
   }
   ```
   The value is then pushed into the SQL query at line 210:
   ```js
   const limitQuery = `limit $${params.push(limit)}`;
   ```

5. The configured hard cap (`responseLimit.max = 100`, `rest/config.js`) is enforced via `getLimitParamValue` (`rest/utils.js` lines 544–553), but this function is only invoked through the standard filter path — not through the custom `validateTokenQueryFilter` path used by `getTokensRequest`. Even if the cap is applied, it still allows 100-row scans per request with no frequency control.

6. No application-level rate limiting exists anywhere in `rest/server.js`. The Traefik `rateLimit` middleware visible in the Helm charts applies only to the Rosetta API, not the REST Node.js service.

**Root cause:** The REST API server has no per-IP or global request-rate throttle. The `GET /tokens` endpoint is publicly accessible, requires no authentication, and executes a database query on every call. The only protection is a row-count cap (100), which does not prevent high-frequency abuse.

### Impact Explanation
An attacker sending continuous `GET /tokens?limit=100` requests forces the database to execute sequential scans of the `token` and `entity` tables on every request. The shared `pg` connection pool (`rest/dbpool.js`) has a finite number of connections; saturating it with maximum-limit queries starves legitimate requests, increasing latency and causing timeouts for all other API consumers. No economic cost is required — a single machine with a fast network connection is sufficient. The impact is availability degradation (griefing) with no direct financial loss to any on-chain participant.

### Likelihood Explanation
The exploit requires zero privileges, zero on-chain funds, and no special knowledge beyond the public API documentation. The endpoint is reachable from the open internet. Any script that loops `curl "https://<mirror-node>/api/v1/tokens?limit=100"` at high frequency reproduces the condition. The attack is trivially repeatable and automatable.

### Recommendation
1. **Add application-level rate limiting** to the REST API server. Insert an `express-rate-limit` (or equivalent) middleware in `rest/server.js` before the route handlers, keyed on client IP, with a configurable requests-per-second ceiling.
2. **Enforce the max-limit cap inside `validateTokenQueryFilter`** for `filterKeys.LIMIT` so that values above `getEffectiveMaxLimit()` are rejected with a 400 error rather than silently clamped or passed through.
3. **Expose a configurable per-IP rate-limit setting** in `application.yml` alongside the existing `response.limit.max` so operators can tune it per deployment.

### Proof of Concept
```bash
# No authentication required. Run from any machine.
while true; do
  curl -s "https://<mirror-node-host>/api/v1/tokens?limit=100&order=asc" -o /dev/null &
done
```
Preconditions: network access to the mirror node REST API port (default 5551).  
Trigger: sustained flood of `GET /tokens?limit=100` requests.  
Result: database connection pool exhaustion; other API endpoints (`/accounts`, `/transactions`, etc.) begin returning errors or timing out as the pool is saturated by token-scan queries.
