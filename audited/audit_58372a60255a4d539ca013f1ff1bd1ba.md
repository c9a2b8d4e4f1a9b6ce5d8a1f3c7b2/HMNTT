### Title
Cache Stampede via Unsynchronized `@Cacheable` on `findRuntimeBytecode`

### Summary
`ContractRepository.findRuntimeBytecode()` uses Spring's `@Cacheable` without `sync=true`, meaning concurrent cache misses for the same `contractId` are not serialized. Any unprivileged external user can trigger N simultaneous requests for the same uncached contract, causing N concurrent identical SQL queries against the database before the first result is stored, leading to database overload.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16–18.

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

The cache manager is a plain `CaffeineCacheManager` configured at `EvmConfiguration.java` lines 67–73:

```java
@Bean(CACHE_MANAGER_CONTRACT)
CacheManager cacheManagerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME_CONTRACT));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getContract());
    return caffeineCacheManager;
}
```

with spec `"expireAfterAccess=1h,maximumSize=1000,recordStats"` (`CacheProperties.java` line 22).

**Root cause:** Spring's `@Cacheable` without `sync=true` implements a non-atomic check-then-act pattern: it calls `cache.get(key)`, and if null, invokes the underlying method, then calls `cache.put(key, value)`. There is no per-key lock. When N threads simultaneously find a cache miss for the same `contractId`, all N proceed to execute `SELECT runtime_bytecode FROM contract WHERE id = :contractId` concurrently. The `unless = "#result == null"` condition only prevents caching empty results — it does not address the race window.

With `sync=true`, Spring would instead call Caffeine's atomic `cache.get(key, callable)`, which serializes loaders per key. That flag is absent here and confirmed absent across all web3 `@Cacheable` usages (grep for `sync=true` in `**/*.java` returns no hits in the web3 module).

**Exploit flow:**
1. Attacker identifies a `contractId` not yet in the cache (e.g., a freshly deployed contract, or any of the 1000+ contracts that exceed `maximumSize` and get evicted).
2. Attacker sends a large burst of concurrent `eth_getCode` (or equivalent contract-call) requests targeting that address.
3. All requests reach `findRuntimeBytecode` simultaneously, all find a cache miss, and all issue `SELECT runtime_bytecode FROM contract WHERE id = ?` to the database.
4. The database receives a fan-out of identical queries, consuming connection pool slots and CPU.

### Impact Explanation
The database connection pool is exhausted or heavily loaded by redundant identical queries. Legitimate requests queue or time out. Because the cache TTL is `expireAfterAccess=1h` and size is capped at 1000, an attacker can continuously evict entries (by requesting 1001+ distinct contracts) and re-trigger stampedes, sustaining the attack. This constitutes a denial-of-service against the database tier with no authentication required.

### Likelihood Explanation
The web3 JSON-RPC endpoint is publicly accessible with no privilege requirement. The attacker needs only a valid (or even invalid) contract address — `unless = "#result == null"` means empty results are not cached, so non-existent IDs can be used repeatedly to guarantee perpetual cache misses. Sending concurrent HTTP requests is trivial with any load-generation tool. The attack is repeatable and requires no special knowledge beyond a contract address.

### Recommendation
Add `sync=true` to the `@Cacheable` annotation:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null",
    sync = true
)
```

With `sync=true`, Spring delegates to Caffeine's `get(key, callable)`, which holds a per-key lock so only one thread executes the DB query for a given `contractId` while others wait for the cached result. Additionally, consider adding rate limiting at the API gateway layer for `eth_getCode`-equivalent endpoints.

### Proof of Concept
1. Deploy or identify any contract on the network whose `contractId` is not yet cached (or use a non-existent ID to guarantee perpetual misses since `unless="#result==null"` skips caching empty `Optional`).
2. Send 500 concurrent HTTP POST requests to the web3 JSON-RPC endpoint:
   ```
   POST /api/v1/contracts/call
   {"data":"0x","to":"<contract_address>","estimate":false}
   ```
   (or equivalent `eth_getCode` call)
3. Observe in database metrics (pg_stat_activity / slow query log) that 500 identical `SELECT runtime_bytecode FROM contract WHERE id = ?` queries execute simultaneously.
4. Repeat continuously to sustain database pressure; no authentication token required.
