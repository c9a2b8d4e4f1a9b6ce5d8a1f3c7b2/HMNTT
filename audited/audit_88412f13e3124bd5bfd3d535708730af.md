### Title
Unauthenticated GraphQL Endpoint Lacks Rate Limiting, Enabling Connection Pool Exhaustion During DB Degradation

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl` issues a synchronous database query per request with no rate limiting or concurrency cap on the GraphQL endpoint. During a network partition or DB degradation event, each in-flight request holds a JDBC connection until the driver timeout expires. An unprivileged attacker can flood the endpoint with concurrent `evmAddress` queries, exhausting the connection pool and denying service to all other users.

### Finding Description
**Code path:**

- `EntityServiceImpl.getByEvmAddressAndType()` — [1](#0-0) 
- `EntityRepository.findByEvmAddress()` — [2](#0-1) 

**Root cause:** Every GraphQL request that resolves an `evmAddress` field calls `entityRepository.findByEvmAddress(evmAddressBytes)`, which issues a native SQL query and holds a JDBC connection for the duration of the call. [3](#0-2) 

**Existing protections reviewed and found insufficient:**

`GraphQlConfiguration` applies parser limits (`maxCharacters`, `maxTokens`) and query-level instrumentation (`MaxQueryComplexityInstrumentation(200)`, `MaxQueryDepthInstrumentation(10)`). [4](#0-3) 

These controls limit the *size and depth* of a single query but impose **no limit on the rate or concurrency of requests**. A search across the entire `graphql/` module found zero rate-limiting, throttling, or authentication components. The `LoggingFilter` only logs requests; it enforces nothing. [5](#0-4) 

**Exploit flow:**

1. Attacker (no credentials required) sends a high volume of concurrent, syntactically valid GraphQL queries each containing a unique `evmAddress` argument.
2. Each request passes all parser/complexity checks and reaches `getByEvmAddressAndType()`.
3. During a network partition (DB unreachable or high-latency), each call to `findByEvmAddress()` blocks on a JDBC connection until the driver socket/query timeout fires.
4. With N concurrent requests and a pool of size P (HikariCP default: 10), once N ≥ P all pool slots are occupied.
5. All subsequent legitimate requests fail immediately with a connection-acquisition timeout, producing a full availability outage for the GraphQL service.

### Impact Explanation
Complete denial of service for the GraphQL endpoint. All users — including legitimate ones — are unable to execute any query because the shared JDBC connection pool is exhausted. The impact is amplified during any DB degradation event (partition, slow replica, maintenance) because connection hold times grow from milliseconds to seconds or minutes, dramatically lowering the request volume needed to trigger exhaustion.

### Likelihood Explanation
The endpoint is publicly reachable with no authentication. A single attacker with a modest HTTP client (e.g., `ab`, `wrk`, or a simple async script) can sustain hundreds of concurrent requests. The attack is trivially repeatable and requires no special knowledge beyond the GraphQL schema. The precondition (network partition) is an operational reality in distributed systems and can also be partially induced by the attacker via slow-loris-style DB connection abuse.

### Recommendation
1. **Rate limiting:** Apply per-IP and global request-rate limits at the servlet or API-gateway layer (e.g., Spring's `RateLimiter`, Bucket4j, or an upstream nginx/Envoy policy).
2. **Connection pool timeout:** Set an aggressive `connectionTimeout` and `maxLifetime` in HikariCP so hung connections are reclaimed quickly during partition events.
3. **Query timeout:** Set a JDBC `queryTimeout` / `socketTimeout` on the DataSource so `findByEvmAddress` never holds a connection longer than a bounded interval.
4. **Concurrency cap:** Use a `Semaphore` or reactive back-pressure to cap the number of simultaneous in-flight DB calls from the GraphQL layer.

### Proof of Concept
```bash
# Requires: graphql endpoint reachable, DB partitioned or slow
# Tool: wrk or parallel curl

QUERY='{"query":"{ account(input:{evmAddress:\"0x000000000000000000000000000000000000DEAD\"}) { id } }"}'

# Fire 500 concurrent requests
for i in $(seq 1 500); do
  curl -s -X POST http://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d "$QUERY" &
done
wait

# Expected result during DB degradation:
# Legitimate requests begin returning HTTP 500 or connection errors
# as the HikariCP pool (default size 10) is fully occupied by hung queries.
```
