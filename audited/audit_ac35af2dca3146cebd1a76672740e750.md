### Title
Unauthenticated HTTP Mirror-Node Bootstrap Allows Network-Positioned Attacker to Inject Malicious Node Map

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` fetches the consensus-node address list over a plain, unauthenticated HTTP connection with no TLS enforcement, no certificate pinning, and no validation of the returned addresses against any known-good set. A network-positioned attacker (DNS cache poisoning, ARP spoofing on the same L2 segment, or BGP prefix hijack) can intercept that request and return a crafted `nodesEnvelope`, causing `hiero.ClientForNetwork(netmap)` to route every subsequent transaction to attacker-controlled gRPC endpoints. This enables silent transaction-history manipulation without requiring any credentials or access to the target host.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()` (lines 36–72) constructs a URL directly from `cfg.mirrorRest` — whose documented default is the plaintext scheme `http://mirror-rest:5551` (see `pinger/config.go` line 37). It then calls `fetchMirrorNodeNetwork()` (lines 74–132) using a bare `http.Client` with no TLS configuration:

```go
// mirror_node_client.go line 46
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

Inside `fetchMirrorNodeNetwork()`, the only checks performed on the response are:
1. HTTP status code in 2xx range (line 90–93)
2. Successful JSON decode into `nodesEnvelope` (line 96–98)
3. At least one usable endpoint exists (line 127–129)

The decoded `ServiceEndpoints` (domain names or raw IPv4 addresses) are accepted verbatim and inserted into `netmap` (lines 113–124), which is then passed directly to `hiero.ClientForNetwork(netmap)` in `sdk_client.go` line 22. There is:
- No HTTPS enforcement
- No TLS certificate pinning
- No comparison of returned addresses against a known-good allowlist
- No cryptographic signature on the response

**Root cause:** The bootstrap trust anchor is an unauthenticated HTTP endpoint. The code assumes the network path to `cfg.mirrorRest` is trusted, which is not guaranteed for any deployment where the mirror-node REST service is reached over a routable network.

**Exploit flow:**
1. Attacker poisons the DNS record for the hostname in `cfg.mirrorRest` (or performs ARP spoofing if on the same L2 segment, or announces a more-specific BGP prefix for the mirror node's IP).
2. The pinger process calls `buildNetworkFromMirrorNodes()` at startup.
3. The HTTP GET is redirected to the attacker's server, which returns a well-formed JSON `nodesEnvelope` containing attacker-controlled gRPC addresses.
4. `fetchMirrorNodeNetwork()` passes all validation checks and returns the poisoned `netmap`.
5. `hiero.ClientForNetwork(netmap)` is called with the poisoned map; all subsequent `CryptoTransfer` transactions are submitted to attacker-controlled nodes.
6. The attacker's nodes can selectively drop, delay, reorder, or replay transactions, reorganizing the visible transaction history without stealing funds directly.

### Impact Explanation
Every transaction submitted by the pinger after a successful bootstrap poisoning is routed to attacker-controlled infrastructure. The attacker can:
- Silently drop transactions (making them appear to fail or never land on-chain)
- Replay or reorder transactions to manipulate the apparent history seen by downstream consumers of the mirror node
- Cause the pinger's liveness/readiness probes to pass while no real transactions reach the legitimate network

Severity is **High** for integrity of the transaction stream. Fund theft is out of scope per the question framing, but history reorganization and denial-of-service to the pinger are fully achievable.

### Likelihood Explanation
The `network=other` path is explicitly designed for private/custom deployments where the mirror-node REST URL is operator-supplied. In such environments:
- The mirror-node service is often reached over an internal cluster network (Kubernetes service DNS), making DNS cache poisoning or ARP spoofing realistic for any attacker with a foothold on the same cluster or network segment.
- The default example URL uses `http://` (plaintext), meaning TLS is not even attempted unless the operator explicitly configures an HTTPS URL.
- DNS cache poisoning (Kaminsky-style) against an unvalidated resolver is a well-understood, repeatable attack requiring no special privileges on the target host.
- BGP hijacking, while requiring more infrastructure, has been demonstrated repeatedly against production networks.

The attack is repeatable: if the pinger process restarts (e.g., after a crash or rolling update), it re-bootstraps from the mirror node, giving the attacker another opportunity.

### Recommendation
1. **Enforce HTTPS**: Reject any `cfg.mirrorRest` URL that does not use the `https://` scheme at config-load time (`pinger/config.go`, `loadConfig()`).
2. **TLS certificate pinning or CA restriction**: Configure the `http.Client` with a custom `tls.Config` that pins the expected certificate or restricts the trusted CA pool to a known internal CA.
3. **Response validation**: After decoding `nodesEnvelope`, validate each returned `NodeAccountID` and endpoint against a configurable allowlist of known-good account IDs or IP ranges.
4. **Integrity check**: If the mirror node supports signed responses or a known-good node list is available out-of-band (e.g., embedded in the binary or fetched from a separate pinned source), verify the bootstrap data against it before use.

### Proof of Concept
**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`
- Attacker has network access to the same L2 segment or controls a DNS resolver used by the pinger pod

**Steps:**
1. Stand up a malicious HTTP server returning:
   ```json
   {
     "nodes": [{
       "node_account_id": "0.0.3",
       "service_endpoints": [{"ip_address_v4": "<attacker-ip>", "port": 50211}]
     }],
     "links": {"next": null}
   }
   ```
2. Poison the DNS record for `mirror-rest` to resolve to `<attacker-ip>`, or ARP-spoof the pinger's gateway.
3. Restart (or wait for the next startup of) the pinger process.
4. `buildNetworkFromMirrorNodes()` fetches from the attacker's server; all checks pass.
5. `hiero.ClientForNetwork({"<attacker-ip>:50211": AccountID{0,0,3}})` is called.
6. All subsequent `CryptoTransfer` calls in `transfer.go` are submitted to `<attacker-ip>:50211` instead of the legitimate network. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```go
func fetchMirrorNodeNetwork(
	ctx context.Context,
	httpClient *http.Client,
	url string,
) (map[string]hiero.AccountID, bool, error) {
	req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
	if err != nil {
		return nil, false, err
	}

	resp, err := httpClient.Do(req)
	if err != nil {
		return nil, true, fmt.Errorf("GET %s failed: %w", url, err)
	}
	defer resp.Body.Close()

	if resp.StatusCode < 200 || resp.StatusCode >= 300 {
		retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
		return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
	}

	var payload nodesEnvelope
	if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
		return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
	}
```
