### Title
Unauthenticated GraphQL `account()` Query Allows Forced Slow-Path DB Lookups via Crafted `evmAddress` Input

### Summary
Any unauthenticated external user can submit `account()` GraphQL queries with `evmAddress` values whose first 12 bytes are non-zero, deterministically forcing `EntityServiceImpl.getByEvmAddressAndType()` to execute the `findByEvmAddress()` secondary-index scan instead of the `findById()` primary-key lookup. Because the GraphQL endpoint has no per-IP or per-user rate limiting, an attacker can sustain a high volume of these slower queries to exhaust DB connection pool resources and degrade service availability.

### Finding Description

**Code path:**

`query.graphqls` (line 5) exposes `account(input: AccountInput!)` with no authentication requirement. [1](#0-0) 

`AccountInput.evmAddress` is validated only by a format regex `^(0x)?[a-fA-F0-9]{40}$`, which permits any valid 40-hex-char address — including those with non-zero first 12 bytes. [2](#0-1) 

`AccountController.account()` passes the `evmAddress` directly to `entityService.getByEvmAddressAndType()` with no additional checks. [3](#0-2) 

**Root cause — branch selection in `EntityServiceImpl.getByEvmAddressAndType()`:**

```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
var buffer = ByteBuffer.wrap(evmAddressBytes);
if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0-11 all zero
    return entityRepository.findById(buffer.getLong())...;     // PRIMARY KEY lookup (fast)
}
return entityRepository.findByEvmAddress(evmAddressBytes)...;  // secondary index scan (slow)
``` [4](#0-3) 

The fast path (`findById`) uses the primary key. The slow path (`findByEvmAddress`) executes:
```sql
select * from entity where evm_address = ?1 and deleted is not true
``` [5](#0-4) 

This is a secondary-index scan that holds a DB connection longer than a primary-key lookup.

**Why existing checks are insufficient:**

The `@Pattern` directive only validates hex format, not the semantic content of the first 12 bytes. The `OnValidationErrorStrategy.RETURN_NULL` strategy means a pattern mismatch silently nullifies the field rather than rejecting the request hard. [6](#0-5) 

The only GraphQL-level protections are query complexity (200) and depth (10) limits — neither of which limits request rate or restricts which DB code path is taken. [7](#0-6) 

There is no authentication, no per-IP rate limiting, and no token bucket anywhere in the GraphQL module.


### Impact Explanation
An attacker can sustain a flood of `account(input: { evmAddress: "0xdeadbeef..." })` queries (any address with non-zero bytes 0–11) that each force a secondary-index DB scan. Under sufficient load this exhausts the JDBC connection pool, causing legitimate queries to queue or time out. Because the endpoint is public and unauthenticated, the attack requires zero credentials and is trivially scriptable. Severity: **Medium** (availability impact, no data exfiltration).

### Likelihood Explanation
The attack requires only a valid HTTP client and knowledge of the GraphQL schema (publicly documented). The attacker needs no account, no token, and no prior access. The trigger condition (non-zero first 12 bytes) is the default for any real Ethereum-style EVM address, so the attacker does not even need to craft special input — any real-world EVM address will do. Repeatability is unlimited.

### Recommendation
1. **Add rate limiting** at the GraphQL endpoint layer (e.g., Spring's `HandlerInterceptor` or a gateway-level token bucket) keyed on source IP.
2. **Ensure a database index exists** on `entity.evm_address` so `findByEvmAddress()` is an indexed point-lookup, minimising the performance gap between the two branches.
3. Consider **connection pool timeouts and query timeouts** to bound the worst-case hold time per request.
4. Optionally, expose the `account()` query only after authentication if the deployment model permits it.

### Proof of Concept

```bash
# Any EVM address whose first 12 bytes are non-zero triggers findByEvmAddress()
# e.g. 0xdeadbeef0000000000000000000000000000cafe

while true; do
  curl -s -X POST http://<host>/graphql \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input: { evmAddress: \"0xdeadbeef0000000000000000000000000000cafe\" }) { entityId { num } } }"}' &
done
```

Each request bypasses the fast `findById()` path and executes `SELECT * FROM entity WHERE evm_address = $1 AND deleted IS NOT TRUE`, holding a DB connection for the duration. With no rate limiting, concurrent requests saturate the connection pool, degrading or denying service to legitimate users.
