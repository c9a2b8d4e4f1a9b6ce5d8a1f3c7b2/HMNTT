### Title
GraphQL Preparsed Document Cache Bypass via Inline Alias Arguments Enabling Unauthenticated Resource Exhaustion

### Summary
The `CachedPreparsedDocumentProvider` keys its Caffeine cache on the raw GraphQL query string. An unprivileged attacker can inline a unique Base32 alias literal per request instead of using a GraphQL variable, producing a distinct cache key on every call and forcing a full parse-validate-execute cycle plus a native SQL `findByAlias` database hit for each request. No rate limiting exists on the GraphQL module's endpoint, so this attack can be sustained indefinitely at arbitrary throughput.

### Finding Description

**Cache key is the raw query string.**

`CachedPreparsedDocumentProvider.getDocumentAsync()` at line 26 uses `executionInput.getQuery()` as the Caffeine cache key:

```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
``` [1](#0-0) 

The cache is configured with `maximumSize=1000` and `expireAfterWrite=1h`. [2](#0-1) 

**Inline literals produce unique query strings.**

When a client sends the alias as an inline literal rather than a variable:
```graphql
{ account(input: { alias: "AAAAAAAAAAAAAAAA" }) { balance } }
{ account(input: { alias: "BBBBBBBBBBBBBBBB" }) { balance } }
```
each string is a distinct cache key. The cache never hits. If the client used a variable (`query($a:String){account(input:{alias:$a}){balance}}`), the query string would be identical across all calls and the cache would serve every subsequent request. Nothing in the code enforces or normalizes to the variable form.

**Every cache miss triggers a DB query.**

`EntityServiceImpl.getByAliasAndType()` at line 30 calls `entityRepository.findByAlias()` unconditionally — there is no service-level or repository-level result cache:

```java
return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
``` [3](#0-2) 

The repository executes a native SQL full-table scan filtered by the `alias` column:

```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
``` [4](#0-3) 

**No rate limiting on the GraphQL module.**

`GraphQlConfiguration` installs only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` plus static parser size limits. There is no per-IP or global request-rate limiter: [5](#0-4) 

The `ThrottleConfiguration`/`ThrottleManagerImpl` that provides bucket4j rate limiting exists only in the `web3` module and is not wired into the `graphql` module at all. [6](#0-5) 

**The `@Pattern` directive does not prevent the attack.**

The schema constrains the alias to `^[A-Z2-7]+$` (Base32 alphabet): [7](#0-6) 

This is a format check, not a uniqueness or rate check. The Base32 alphabet of length ≥ 8 characters yields more than 10¹² distinct valid values, giving the attacker an effectively unlimited supply of unique cache-busting strings.

### Impact Explanation

Each request with a novel inline alias forces: (1) GraphQL lexing and parsing, (2) schema validation, (3) instrumentation passes (complexity + depth), (4) resolver execution, and (5) a native SQL query against the `entity` table. Steps 1–3 are CPU-bound work that the preparsed document cache is specifically designed to eliminate for repeated queries. Bypassing the cache multiplies CPU consumption proportionally to request rate. Step 5 adds sustained DB I/O. Because the endpoint is unauthenticated and has no rate limit, a single attacker with a modest HTTP client can drive CPU and DB connection utilization well above the 30% threshold, as confirmed by the Prometheus alert thresholds already defined for this service (CPU > 80%, DB connections > 75%). [8](#0-7) 

### Likelihood Explanation

No privileges, credentials, or special network position are required. The GraphQL endpoint is documented as publicly accessible at `/graphql/alpha`. The attack requires only an HTTP client capable of sending POST requests at high throughput with a trivially generated sequence of unique Base32 strings. It is fully repeatable and scriptable. The only external dependency is network bandwidth to the server.

### Recommendation

1. **Normalize query strings before caching:** Strip or reject inline literal arguments and require clients to use GraphQL variables. Alternatively, hash the AST structure (without literal values) as the cache key so that structurally identical queries with different literal values share one cache entry.
2. **Add request-rate limiting to the GraphQL module:** Port the bucket4j `ThrottleConfiguration` pattern from the `web3` module, or add a Spring `HandlerInterceptor`/`WebFilter` that enforces a per-IP or global requests-per-second cap before the GraphQL execution pipeline is entered.
3. **Add a result cache on `EntityServiceImpl.getByAliasAndType()`:** A short-lived (e.g., 30-second) Caffeine cache keyed on the decoded alias bytes would absorb repeated DB hits for the same alias regardless of how the query is structured.

### Proof of Concept

```bash
#!/usr/bin/env bash
# Requires: bash, curl, python3 (for Base32 generation)
ENDPOINT="http://<graphql-host>:8083/graphql/alpha"

python3 - <<'EOF'
import subprocess, base64, os, itertools, string

chars = "ABCDEFGHIJKLMNOPQRSTUVWXYZ234567"
for i in range(10000):
    # Generate a unique valid Base32 string of length 8
    alias = ''.join(chars[(i >> (5*j)) & 0x1F] for j in range(8))
    query = '{{"query":"{{ account(input: {{ alias: \\"{}\\" }}) {{ balance }} }}"}}'.format(alias)
    subprocess.Popen(
        ["curl", "-s", "-o", "/dev/null", "-X", "POST",
         "-H", "Content-Type: application/json",
         "-d", query,
         "http://<graphql-host>:8083/graphql/alpha"]
    )
EOF
```

Each iteration sends a structurally identical but lexically distinct query. Monitor the GraphQL node's CPU (`process_cpu_usage`) and HikariCP active connections (`hikaricp_connections_active`) via Prometheus — both will climb linearly with request rate, with no cache hits observed in the `graphql_preparsed_cache` metrics.
