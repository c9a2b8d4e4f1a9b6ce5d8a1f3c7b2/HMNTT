### Title
`BufferUnderflowException` via Under-Length Hex `evmAddress` in `getByEvmAddressAndType()`

### Summary
`decodeEvmAddress()` performs no length validation on the decoded byte array, allowing any valid hex string shorter than 40 characters to pass through. `getByEvmAddressAndType()` unconditionally wraps the result in a `ByteBuffer` and reads 4 + 8 + 8 = 20 bytes, so a crafted input whose first 4 bytes are zero but whose total length is between 5 and 11 bytes causes `buffer.getLong()` to throw an unchecked `BufferUnderflowException`. `CustomExceptionResolver` does not handle this exception type, so it propagates as an unresolved internal error.

### Finding Description
**`GraphQlUtils.decodeEvmAddress()`** (`graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 87–98):
```java
public static byte[] decodeEvmAddress(String evmAddress) {
    if (evmAddress == null) {
        return ArrayUtils.EMPTY_BYTE_ARRAY;
    }
    try {
        evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
        return Hex.decodeHex(evmAddress);   // ← no length check
    } catch (DecoderException e) {
        throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
    }
}
```
Only hex-format validity is checked. Any even-length hex string of any byte length is returned.

**`EntityServiceImpl.getByEvmAddressAndType()`** (`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41):
```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
var buffer = ByteBuffer.wrap(evmAddressBytes);
if (buffer.getInt() == 0 && buffer.getLong() == 0) {   // ← assumes ≥ 12 bytes
    return entityRepository.findById(buffer.getLong())  // ← assumes ≥ 20 bytes
        .filter(e -> e.getType() == type);
}
```
- `buffer.getInt()` consumes bytes 0–3 (needs ≥ 4 bytes).
- First `buffer.getLong()` consumes bytes 4–11 (needs ≥ 12 bytes total).
- Second `buffer.getLong()` consumes bytes 12–19 (needs ≥ 20 bytes total).

Due to Java's short-circuit `&&`, the `getLong()` at position 4 is only reached when `getInt() == 0`, i.e., when the first 4 bytes are all zero. An attacker supplies a hex string whose first 8 hex chars are `00000000` and whose total length is between 10 and 22 hex chars (5–11 bytes). `getInt()` returns 0, then `getLong()` throws `java.nio.BufferUnderflowException`.

**`CustomExceptionResolver`** (`graphql/src/main/java/org/hiero/mirror/graphql/config/CustomExceptionResolver.java`, lines 16–29) only handles `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`. `BufferUnderflowException` (which extends `RuntimeException` directly) is not in this list; `resolveToSingleError` returns `null`, causing Spring GraphQL's default handler to emit a generic `INTERNAL_ERROR` response.

### Impact Explanation
Every request carrying a crafted short evmAddress triggers an unhandled `BufferUnderflowException` that bypasses the application's own error-handling layer. The caller receives an opaque `INTERNAL_ERROR` GraphQL response, which leaks the fact that an unhandled exception occurred. At scale, flooding this endpoint with such requests produces a continuous stream of unhandled exceptions logged at ERROR level, polluting observability infrastructure and potentially masking legitimate errors. The "thread exhaustion" claim in the question is not directly supported by the code — Spring GraphQL's framework-level handler does catch the exception and completes the request thread normally — but the unhandled exception path and information leakage are real.

### Likelihood Explanation
No authentication is required. The GraphQL endpoint is publicly reachable. The crafted input is trivially constructed: any even-length hex string starting with `00000000` and shorter than 24 hex characters (e.g., `"0000000001"`) reliably triggers the bug. It is repeatable, requires no special knowledge, and can be automated.

### Recommendation
Add an explicit length check in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException("evmAddress must be exactly 20 bytes");
}
```

`IllegalArgumentException` is already handled by `CustomExceptionResolver` and will produce a clean `ValidationError` GraphQL response instead of an unhandled internal exception.

### Proof of Concept
Send the following GraphQL query to the public endpoint (no credentials needed):

```graphql
{
  account(input: { evmAddress: "0000000001" }) {
    id
  }
}
```

**Step-by-step:**
1. `evmAddress = "0000000001"` → `decodeEvmAddress` strips no prefix, calls `Hex.decodeHex("0000000001")` → 5-byte array `[0x00, 0x00, 0x00, 0x00, 0x01]`.
2. `ByteBuffer.wrap([0x00,0x00,0x00,0x00,0x01])` → 5-byte buffer.
3. `buffer.getInt()` → reads bytes 0–3 = `0`. Condition is `true`; short-circuit does NOT skip `getLong()`.
4. `buffer.getLong()` → attempts to read 8 bytes from position 4; only 1 byte remains → **`java.nio.BufferUnderflowException`** thrown.
5. `CustomExceptionResolver.resolveToSingleError` receives `BufferUnderflowException`, matches none of the three handled types, returns `null`.
6. Spring GraphQL default handler returns `{"errors":[{"message":"INTERNAL_ERROR","extensions":{"classification":"INTERNAL_ERROR"}}]}`.
