### Title
Deleted Account Fully Exposed via EntityId Lookup in GraphQL `account()` Query Due to Missing Deleted Filter and Nullable `deleted` Field

### Summary
The `account()` GraphQL query, when invoked with an `entityId` input, uses `EntityRepository.findById()` which applies no `deleted` filter, returning full `Account` responses (including `balance`, `key`, `nonce`) for accounts with `deleted = true` in the database. This is inconsistent with the `alias` and `evmAddress` lookup paths, which both enforce `deleted is not true` in their SQL queries. Additionally, the `deleted` field is declared as nullable `Boolean` (not `Boolean!`) in the schema, meaning accounts with `deleted = null` in the DB return `null` for the field â€” which off-chain consumers that do not null-check (e.g., `!account.deleted` in JavaScript) will silently misinterpret as "not deleted."

### Finding Description

**Code path 1 â€” deleted=true account returned via entityId:**

`AccountController.java` line 41â€“44:
```java

    return entityService
            .getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
```

`EntityServiceImpl.java` line 24â€“26:
```java

}
```

`findById` is the standard Spring Data `CrudRepository` method â€” it performs a plain `SELECT * FROM entity WHERE id = ?` with **no** `deleted` filter.

**Contrast with alias/evmAddress paths** (`EntityRepository.java` lines 13â€“17):
```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

Both alias and evmAddress lookups explicitly exclude deleted entities. The entityId path has no such guard.

**Code path 2 â€” long-zero evmAddress also bypasses the filter:**

```java

```

A long-zero EVM address (e.g., `0x0000000000000000000000000000000000000ABC`) is decoded to an entity ID and resolved via `findById` â€” again with no deleted filter.

**Code path 3 â€” nullable `deleted` field:**

`common.graphqls` line 122:
```graphql
deleted: Boolean
```

`AbstractEntity.java` line 64:
```java

```

The DB column allows `NULL`. When `deleted = null` in the DB, the GraphQL response returns `null` for the `deleted` field. Any consumer that evaluates `!account.deleted` (JavaScript/TypeScript) treats `null` as falsy, concluding the account is active.

### Impact Explanation
Any unauthenticated user can query `account(input: {entityId: {num: X}})` for a known deleted account and receive a complete `Account` response including `balance`, `key`, `nonce`, and `deleted: true`. Off-chain code (dApps, indexers, oracle bridges) that feeds this data to smart contracts without checking `deleted === true` (or without null-guarding `deleted`) will interact with a deleted entity. For the null case, a smart contract integration that checks `if (!account.deleted)` will treat `null` as "not deleted" and proceed with a stale/invalid entity. This matches the stated severity: unintended smart contract behavior with no direct fund loss but incorrect protocol-level state interpretation.

### Likelihood Explanation
No authentication or special privilege is required. The attacker only needs to know (or enumerate) a valid entity numeric ID â€” these are sequential integers publicly visible on any block explorer. The exploit is trivially repeatable via a single unauthenticated HTTP POST to `/graphql/alpha`. The long-zero EVM address variant is equally trivial: construct `0x000...0{num}` for any known entity num.

### Recommendation
1. Add a `deleted is not true` filter to the `findById`-based path. Either add a custom query method to `EntityRepository`:
   ```java
   @Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByIdAndNotDeleted(Long id);
   ```
   and use it in `getByIdAndType` and the long-zero evmAddress branch of `getByEvmAddressAndType`.

2. Change the `deleted` field in the GraphQL schema from `Boolean` (nullable) to `Boolean!` (non-null) with a server-side default of `false` when the DB value is `null`, eliminating the null-misinterpretation vector.
