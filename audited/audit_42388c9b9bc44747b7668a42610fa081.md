### Title
Unbounded `nodeIds` List in `getNetworkNodes()` Enables Resource Amplification via Uncapped EQ Parameter Injection

### Summary
`NetworkNodeRequest` imposes no size limit on the `nodeIds` list, allowing any unauthenticated caller to submit an arbitrarily large number of `node.id=eq:X` query parameters. `NetworkServiceImpl.getNetworkNodes()` iterates every entry unconditionally, builds a `HashSet<Long>`, and passes the resulting `Long[]` to a complex native SQL query via `any(:nodeIds)`, with no guard beyond the unrelated `limit` cap of 25.

### Finding Description
**Code locations:**

- `NetworkNodeRequest.java` line 38: `private List<NumberRangeParameter> nodeIds = List.of();` — no `@Size` or any cardinality constraint.
- `NetworkServiceImpl.java` lines 106–133: the loop at line 110 iterates every element of `nodeIdParams` without bound; EQ entries are added to `nodeIds` (a `HashSet<Long>`) at line 112; the full set is streamed and filtered at line 127 and converted to `Long[]` at line 127; that array is forwarded to the repository at line 135.
- `NetworkNodeRepository.java` line 96: `where (coalesce(array_length(:nodeIds, 1), 0) = 0 or abe.node_id = any(:nodeIds))` — PostgreSQL receives the full, unbounded array.

**Root cause:** The only input-size guard in the request DTO is `@Min(1)` on `limit` (capped to 25 at line 54 of `NetworkNodeRequest`). The `nodeIds` list has no `@Size(max=…)` annotation and no runtime check in the service layer.

**Exploit flow:**
1. Attacker crafts a GET request with thousands of `node.id=eq:N` parameters (e.g., `node.id=eq:0&node.id=eq:1&…&node.id=eq:9999`).
2. Spring binds all values into `List<NumberRangeParameter>` without rejection.
3. `getNetworkNodes()` iterates all N entries, inserting each into a `HashSet<Long>` — O(N) heap allocations.
4. The full set is streamed, filtered, and converted to `Long[]` — another O(N) allocation.
5. The `Long[]` is bound as a PostgreSQL array parameter; the planner must evaluate `array_length`, allocate the array on the DB side, and execute `= any(array[0,1,…,9999])` against every row in `address_book_entry`.
6. The complex CTE query (three CTEs, a correlated subquery per row for `service_endpoints`) runs in full for every request.

**Why existing checks fail:** The `limit` cap (25) only bounds the result set rows returned; it does not limit the size of the `nodeIds` filter array evaluated before the `LIMIT` clause, nor the application-side HashSet/array construction.

### Impact Explanation
Each malicious request forces O(N) heap allocations in the JVM (HashSet + Long[] boxing) and passes an O(N) array to a multi-CTE native query. Sending a modest number of concurrent requests (e.g., 10–20 parallel connections each with 5,000–10,000 `node.id=eq:X` parameters) can saturate JVM GC and inflate PostgreSQL query planning/execution time well beyond 30% compared to normal traffic, with no authentication required.

### Likelihood Explanation
The endpoint is public (no authentication), the exploit requires only a standard HTTP client, and the parameters are trivially generated. The attack is repeatable and scriptable. No special knowledge of the network topology is needed — arbitrary Long values are accepted and processed regardless of whether they match real node IDs.

### Recommendation
Add a `@Size(max = 25)` (or a domain-appropriate constant) constraint to the `nodeIds` field in `NetworkNodeRequest`:

```java
@Size(max = MAX_LIMIT)
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
private List<NumberRangeParameter> nodeIds = List.of();
```

Additionally, add a runtime guard at the top of `getNetworkNodes()` in `NetworkServiceImpl` to reject oversized lists before any allocation occurs, and ensure the validator is triggered (e.g., via `@Validated` on the controller).

```bash
# Generate 5000 node.id=eq:N parameters and send the request
python3 -c "
params = '&'.join(f'node.id=eq:{i}' for i in range(5000))
print(f'GET /api/v1/network/nodes?{params}')
" | xargs -I{} curl -s -o /dev/null -w "%{time_total}\n" \
  "http://<mirror-node-host>/api/v1/network/nodes?$(python3 -c \
  "print('&'.join(f'node.id=eq:{i}' for i in range(5000)))")"

# Repeat with 20 parallel connections:
seq 20 | xargs -P20 -I{} curl -s -o /dev/null \
  "http://<mirror-node-host>/api/v1/network/nodes?$(python3 -c \
  "print('&'.join(f'node.id=eq:{i}' for i in range(5000)))")"
```

Monitor JVM heap (GC pressure) and PostgreSQL `pg_stat_activity` / CPU during the burst versus baseline to confirm the ≥30% resource increase.
