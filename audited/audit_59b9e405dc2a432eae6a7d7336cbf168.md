### Title
Unauthenticated Repeated Requests to `/api/v1/network/exchangerate` Trigger Uncached Database Queries, Enabling Sustained DB Load

### Summary
The `getExchangeRate()` handler in `NetworkController` accepts up to two timestamp parameters from any unauthenticated caller and unconditionally delegates to `FileServiceImpl.getExchangeRate()`, which executes a correlated subquery with `string_agg` against the `file_data` table on every invocation. No caching is applied to the `rest-java` module's `FileDataRepository.getFileAtTimestamp()`, and no rate-limiting infrastructure exists in the `rest-java` module. A single attacker can therefore generate sustained, unbounded database load with minimal effort.

### Finding Description

**Exact code path:**

`NetworkController.getExchangeRate()` (lines 92–98) receives the request and immediately calls `fileService.getExchangeRate(bound)` with no guard:

```java
@GetMapping("/exchangerate")
NetworkExchangeRateSetResponse getExchangeRate(
        @RequestParam(required = false) @Size(max = 2) TimestampParameter[] timestamp) {
    final var bound = Bound.of(timestamp, TIMESTAMP, FileData.FILE_DATA.CONSENSUS_TIMESTAMP);
    final var exchangeRateSet = fileService.getExchangeRate(bound);
    return exchangeRateMapper.map(exchangeRateSet);
}
``` [1](#0-0) 

`FileServiceImpl.getExchangeRate()` (line 42–44) calls `getSystemFile()`, which calls `fileDataRepository.getFileAtTimestamp()` on every invocation with no caching: [2](#0-1) 

The repository method executes a correlated subquery with `string_agg` on every call: [3](#0-2) 

**Root cause — failed assumption:** The `rest-java` `FileDataRepository` has no `@Cacheable` annotation (unlike, e.g., `grpc` module repositories which use `@Cacheable`). The `FileServiceImpl` also has no caching layer. The only "protection" registered in `RestJavaConfiguration` is a `ShallowEtagHeaderFilter`, which buffers the response body to compute an ETag — but the controller and service code (including the DB query) execute in full before the filter can short-circuit with a 304. [4](#0-3) 

The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module and is not wired into `rest-java` at all. [5](#0-4) 

Additionally, `FileServiceImpl` wraps the call in a `RetryTemplate` with up to 9 retries triggered by parse failures. If the attacker's timestamp range hits a corrupt file entry, a single HTTP request can cause up to 10 sequential DB queries. [6](#0-5) 

### Impact Explanation
Each request to `/api/v1/network/exchangerate` with two timestamp parameters executes at least one correlated subquery with `string_agg` against the `file_data` table. A single attacker sending a high-frequency loop of such requests can saturate database connection pool slots and CPU, degrading query performance for all other mirror node API consumers. There is no economic damage to network participants, but availability of the mirror node REST API is impaired — consistent with the "griefing / medium" classification.

### Likelihood Explanation
No authentication, API key, or IP-based rate limit is required. The endpoint is publicly documented in the OpenAPI spec. Any attacker with a basic HTTP client and a loop can reproduce this. The attack requires zero privileges and zero knowledge beyond the public API documentation.

### Recommendation
1. **Add response-level caching** on `FileDataRepository.getFileAtTimestamp()` in the `rest-java` module using `@Cacheable` with a short TTL (e.g., 10–30 seconds), similar to how the `web3` module caches `systemFile` results (`expireAfterWrite=10m`).
2. **Add rate limiting** to the `rest-java` module for read endpoints, analogous to the `ThrottleConfiguration`/`ThrottleManagerImpl` pattern already present in `web3`.
3. **Add HTTP `Cache-Control` response headers** (e.g., `max-age=30, s-maxage=60`) so that CDN/reverse-proxy layers can absorb repeated identical requests before they reach the application.

### Proof of Concept
```bash
# No authentication required. Run from any machine.
while true; do
  curl -s "https://<mirror-node-host>/api/v1/network/exchangerate?timestamp=gt:0&timestamp=lt:9999999999999999999" > /dev/null
done
```
Each iteration triggers `FileDataRepository.getFileAtTimestamp()` with a full table scan range, executing the correlated `string_agg` subquery against `file_data`. Sustained execution degrades DB performance for all concurrent API consumers.

@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {

                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
