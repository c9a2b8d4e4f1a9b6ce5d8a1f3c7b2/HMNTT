### Title
Unbounded Per-Subscriber Database Polling in `PollingTopicListener.listen()` Enables Unauthenticated DoS via Subscription Flooding

### Summary
`PollingTopicListener.listen()` creates an independent, perpetual database polling loop for every subscriber with no cap on the number of concurrent subscriptions, no per-IP limit, and no per-client connection limit. An unauthenticated attacker can open an arbitrarily large number of gRPC streams, each triggering its own 500ms polling cycle against the database, exhausting the DB connection pool and preventing the importer from ingesting new transactions.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` (lines 34–49) creates a new `PollingContext` per call and schedules an infinite `Flux.defer(() -> poll(context))` loop with `RepeatSpec.times(Long.MAX_VALUE)` and a fixed delay of `interval` (default 500ms from `ListenerProperties`):

```
Flux.defer(() -> poll(context))
    .delaySubscription(interval, scheduler)
    .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
            .jitter(0.1)
            .withFixedDelay(interval)
            .withScheduler(scheduler))
``` [1](#0-0) 

Each `poll()` call executes a blocking JDBC query via `topicMessageRepository.findByFilter(newFilter)`, fetching up to `maxPageSize` (default 5000) rows per poll: [2](#0-1) 

**Root cause — no subscriber cap is enforced:**

`TopicMessageServiceImpl` tracks a global `subscriberCount` via an `AtomicLong`, but this is wired only to a Micrometer `Gauge` for observability. There is no enforcement gate — no maximum is checked, no subscription is rejected: [3](#0-2) [4](#0-3) 

`ListenerProperties` contains no `maxSubscriptions`, `maxConnectionsPerIp`, or any subscriber-limiting field: [5](#0-4) 

A search across the entire grpc module confirms zero occurrences of `maxSubscribers`, `maxClients`, `perIp`, `connectionPerIp`, or `subscriberLimit`.

**Exploit flow:**

1. Attacker opens N gRPC connections (or multiplexes N streams over fewer connections) to the `ConsensusController` `subscribeTopic` endpoint — no authentication required.
2. Each stream causes `TopicMessageServiceImpl.subscribeTopic()` → `incomingMessages()` → `topicListener.listen(newFilter)` to be called once per subscriber.
3. Each `listen()` call spawns an independent polling loop on `Schedulers.boundedElastic()`, issuing a JDBC query every ~500ms (±10% jitter).
4. With N=500 concurrent subscriptions: ~1000 DB queries/second, each potentially returning up to 5000 rows, consuming DB connections from the shared pool.
5. The DB connection pool saturates; the importer's write path is starved of connections and cannot persist new transaction records.

**Why existing checks fail:**

- `Schedulers.boundedElastic()` queues up to 100,000 tasks — it does not reject subscriptions.
- The gRPC Netty server's default `maxConcurrentCallsPerConnection` is effectively unbounded (Integer.MAX_VALUE unless explicitly configured in `GrpcConfiguration`).
- No IP-based throttle exists in the grpc module (throttle infrastructure is confined to the `web3` module).
- `RetrieverProperties.timeout` (60s) applies only to the historical retriever, not to the live `PollingTopicListener` loop which runs `Long.MAX_VALUE` repeats indefinitely. [6](#0-5) 

### Impact Explanation
When `ListenerType.POLL` is active, every subscriber independently hammers the database. Saturating the DB connection pool blocks the importer's write path, meaning new HCS (Hedera Consensus Service) transactions cannot be recorded. This constitutes a total network-level denial of service for transaction confirmation visibility — matching the "Critical: Network not being able to confirm new transactions" classification. The attack is amplifiable: each additional subscription linearly increases DB load with no diminishing returns until the pool is exhausted.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no exploit code — only a gRPC client library (e.g., `grpcurl`, `ghz`, or any HCS SDK). It is trivially scriptable and repeatable. The attacker does not need to maintain state between connections. The only practical barrier is network bandwidth to sustain N open gRPC streams, which is negligible for long-lived streaming RPCs. Any public mirror node deployment with `POLL` mode configured is directly exposed.

### Recommendation
1. **Enforce a global subscriber cap**: Check `subscriberCount` against a configurable `maxSubscribers` limit in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` gRPC status when exceeded.
2. **Add per-IP connection limiting**: Intercept at the gRPC `ServerInterceptor` level, tracking active streams per remote IP and rejecting new ones above a threshold.
3. **Configure Netty `maxConcurrentCallsPerConnection`**: Set an explicit bound in `GrpcConfiguration` to limit stream multiplexing abuse.
4. **Add `maxSubscriptions` to `ListenerProperties`**: Make the cap operator-configurable so it can be tuned per deployment.
5. **Consider `SharedPollingTopicListener`** as the default for POLL mode, which shares a single DB poll across all subscribers rather than one poll per subscriber. [7](#0-6) 

### Proof of Concept
```bash
# Install ghz (gRPC benchmarking tool)
# Assumes mirror node is running with POLL listener type on localhost:5600

# Open 500 concurrent long-lived subscriptions to a valid topic
ghz --insecure \
    --proto hedera/mirror/api/proto/consensus_service.proto \
    --call com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    --data '{"topicID":{"topicNum":1},"consensusStartTime":{"seconds":0}}' \
    --concurrency 500 \
    --connections 50 \
    --duration 60s \
    localhost:5600

# Expected result:
# - DB connection pool exhausted within seconds
# - Importer write latency spikes to timeout
# - New HCS transactions stop being confirmed in the mirror node
# - subscriberCount metric climbs to 500 with no rejection
```
