### Title
Unicode Bidirectional Override (U+202E) Survives `DomainUtils.sanitize()` in Token Symbol, Enabling Visual Spoofing via Mirror Node REST API

### Summary
The `setSymbol()` method in `AbstractToken.java` delegates sanitization entirely to `DomainUtils.sanitize()`, which only strips the PostgreSQL-incompatible null byte (U+0000). The right-to-left override character U+202E (and other Unicode bidirectional control characters) pass through unchanged, are persisted to the database, and are returned verbatim by the mirror node REST API. Any unprivileged account holder can create a token whose symbol visually misrepresents its identity to end users in any Unicode-aware rendering environment.

### Finding Description
**Code path:**

- `common/src/main/java/org/hiero/mirror/common/domain/token/AbstractToken.java`, lines 113–115:
  ```java
  public void setSymbol(String symbol) {
      this.symbol = DomainUtils.sanitize(symbol);
  }
  ```
- `common/src/main/java/org/hiero/mirror/common/util/DomainUtils.java`, lines 239–241:
  ```java
  public static String sanitize(String input) {
      return StringUtils.isNotEmpty(input) ? input.replace(NULL_CHARACTER, NULL_REPLACEMENT) : input;
  }
  ```
  where `NULL_CHARACTER = (char) 0` (U+0000).

**Root cause:** `sanitize()` was designed solely to prevent PostgreSQL null-byte insertion failures (as stated in its own Javadoc: *"PostgreSQL does not allow the null character (0x0000) to be inserted"*). It performs a single `String.replace` of U+0000 only. No other Unicode control characters — including the bidirectional controls U+202E (RIGHT-TO-LEFT OVERRIDE), U+202D (LEFT-TO-RIGHT OVERRIDE), U+200F (RIGHT-TO-LEFT MARK), U+2066–U+2069 (isolate controls), etc. — are touched.

**Failed assumption:** The code assumes that the only dangerous character for stored token strings is the null byte. This ignores the well-documented class of Unicode bidirectional spoofing attacks.

**Exploit flow:**
1. Attacker constructs a `TokenCreateTransaction` with `symbol = "USD\u202Egnp"`.
2. The Hedera network accepts the transaction (the HTS protocol does not restrict Unicode bidirectional characters in symbol fields).
3. The mirror node importer calls `setSymbol("USD\u202Egnp")` → `DomainUtils.sanitize("USD\u202Egnp")` → returns `"USD\u202Egnp"` unchanged.
4. The string is persisted to the `token` table in PostgreSQL.
5. The REST API (`/api/v1/tokens/{id}`) returns `"symbol": "USD\u202Egnp"` in the JSON response.
6. Any browser or UI rendering this string displays it as `"USDpng"` (characters after U+202E are rendered right-to-left), visually identical to a legitimate `"USDpng"` token.

### Impact Explanation
An attacker can create a token whose REST API–returned symbol is visually indistinguishable from a legitimate, well-known token symbol (e.g., a stablecoin ticker). Wallets, explorers, and DeFi UIs that consume the mirror node REST API and render the symbol field without their own bidirectional-control stripping will display the spoofed symbol to users. This enables phishing, fraudulent token listings, and social-engineering attacks against token holders. The same issue applies to the `name` field via `setName()` at line 109–111, which uses the identical `sanitize()` call.

### Likelihood Explanation
The precondition is trivially met: any Hedera account (obtainable for a small fee) can submit a `TokenCreateTransaction`. No elevated privileges, no key compromise, and no network-level exploit are required. The attack is repeatable, cheap, and fully automated. The Hedera network does not reject such transactions. The only barrier is that consuming UIs must render the symbol without their own sanitization — a common real-world condition for wallets and explorers that trust the mirror node as a source of truth.

### Recommendation
Extend `DomainUtils.sanitize()` to strip or replace Unicode bidirectional control characters in addition to the null byte. At minimum, remove the following codepoints before storage:

- U+202A–U+202E (directional embedding/override)
- U+2066–U+2069 (directional isolate)
- U+200E, U+200F (directional marks)
- U+061C (Arabic letter mark)

A safe implementation using Java's `Character` API or a regex:
```java
private static final Pattern BIDI_CONTROL = Pattern.compile(
    "[\u202A-\u202E\u2066-\u2069\u200E\u200F\u061C\u0000]");

public static String sanitize(String input) {
    return StringUtils.isNotEmpty(input)
        ? BIDI_CONTROL.matcher(input).replaceAll("\uFFFD")
        : input;
}
```
Apply the same fix to `setName()` (which already calls `sanitize()` and is equally affected).

### Proof of Concept
```java
// Attacker-controlled token creation
String maliciousSymbol = "USD\u202Egnp"; // visually renders as "USDpng" in browsers

// Mirror node processing
String stored = DomainUtils.sanitize(maliciousSymbol);
System.out.println(stored.equals(maliciousSymbol)); // true — U+202E is NOT stripped
System.out.println(stored.length());                // 7 — U+202E is present at index 3

// REST API response (as seen by a consuming UI):
// { "symbol": "USD\u202Egnp" }
// Browser renders: "USDpng" — visually identical to a legitimate "USDpng" token
```
