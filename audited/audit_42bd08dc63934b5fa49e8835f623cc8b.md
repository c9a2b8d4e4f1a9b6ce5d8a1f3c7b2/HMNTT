### Title
Unauthenticated Metrics Endpoint Exposes Per-Route Transaction History Access Patterns When Authentication Is Disabled

### Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` contains an unconditional bypass: when `config.metrics.config.authentication` is falsy, it returns `true` immediately without checking any credentials. The Kubernetes `ServiceMonitor` template scrapes the metrics endpoint with no auth configuration, creating a real operational incentive for deployments to set `authentication: false`. Any unprivileged external user who discovers this can read the full Prometheus metrics dump, including per-route `api_request` counters and `durationHistogram` data labeled by `{method, path, code}`, revealing which transaction history endpoints are being accessed and at what frequency.

### Finding Description
**Exact code path:**

`rest/middleware/metricsHandler.js`, `authenticate()`, lines 137–144:
```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;          // ← unconditional bypass, no credentials checked
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

`metricsMiddleware`, lines 156–164:
```js
if (normalizedPath === metricsPath) {
  if (!authenticate(req)) {          // ← only gate; passes when authentication=false
    res.set('WWW-Authenticate', 'Basic realm="Metrics"');
    return res.status(401).send('Unauthorized');
  }
  return exporter.collect().then(({resourceMetrics}) => {
    res.send(serializer.serialize(resourceMetrics));  // ← full metrics dump served
  });
}
```

**Root cause:** The authentication check is a single boolean flag with no secondary control. When `authentication` is `false`, `authenticate()` returns `true` and the only gate at line 157 is bypassed entirely.

**Failed assumption:** The design assumes `authentication` will always be `true` in production. However, the Kubernetes `ServiceMonitor` template at `charts/hedera-mirror-rest/templates/servicemonitor.yaml` (lines 11–14) scrapes `/swagger/metrics` with no `basicAuth` or `authorization` stanza, making it operationally necessary for many Kubernetes deployments to set `authentication: false` for Prometheus scraping to work.

**Exploit flow:**
1. Attacker sends `GET /swagger/metrics/` (or `/swagger/metrics`) with no `Authorization` header.
2. If the response is `200 text/plain` (Prometheus format), authentication is disabled.
3. Attacker reads the full metrics payload, which includes per-route instruments labeled `{method, path, code}` (line 192): `api_request_total`, `api_request_duration_milliseconds_bucket`, `api_request_size_bytes_bucket`, `api_response_size_bytes_bucket`.
4. The `path` label is the OpenAPI-normalized route (e.g., `/api/v1/transactions/{id}`, `/api/v1/accounts/{id}/transactions`), revealing exactly which transaction history endpoints are being queried and at what rate and latency distribution.

**Why existing checks are insufficient:** There is no IP allowlist, no network policy enforced at the application layer, no secondary authentication mechanism, and no rate limiting on the metrics endpoint itself. The metrics endpoint is served on the same public port (default 5551) as the REST API.

### Impact Explanation
An attacker gains a continuous, unauthenticated view of:
- Which transaction history endpoints (`/api/v1/transactions/{id}`, `/api/v1/accounts/{id}/transactions`, etc.) are being queried and at what frequency (`api_request` counter per route/method/code).
- Response latency distributions per route (`durationHistogram`), revealing backend processing load and indirectly which queries are expensive.
- Error rates per route (via `code` label), revealing which endpoints are under stress or returning errors.

This constitutes a meaningful information disclosure: an attacker can infer which accounts or transaction IDs are being actively monitored, identify quiet periods to time attacks, and understand the operational posture of the deployment — all without any credentials or privileges.

### Likelihood Explanation
- **Discovery is trivial**: a single unauthenticated `GET /swagger/metrics/` either returns `401` (auth enabled) or a full metrics dump (auth disabled). No brute force or special tooling required.
- **Misconfiguration is incentivized**: the official Helm `ServiceMonitor` template has no auth configuration, meaning any operator using it for Prometheus scraping must disable authentication or add out-of-band configuration not provided by the chart.
- **Repeatability**: once discovered, the attacker can poll the endpoint continuously (e.g., every 15 seconds, matching Prometheus scrape intervals) to track access pattern changes in near-real-time.
- **No privileges required**: the exploit requires only network access to the REST API port.

### Recommendation
1. **Enforce authentication by default and make it non-bypassable**: remove the `if (!authentication) return true` shortcut. Instead, require explicit credential configuration; if credentials are not set, either refuse to start or serve metrics only on a separate internal-only port.
2. **Add auth to the ServiceMonitor template**: add a `basicAuth` or `bearerTokenSecret` stanza to `charts/hedera-mirror-rest/templates/servicemonitor.yaml` so Prometheus can scrape with credentials, eliminating the operational incentive to disable auth.
3. **Bind the metrics endpoint to a separate internal port** (not the public API port), so network policy can restrict access without relying on application-level auth alone.
4. **Add IP allowlisting** as a defense-in-depth layer even when auth is enabled.

### Proof of Concept
```bash
# Step 1: Probe whether authentication is disabled
curl -s -o /dev/null -w "%{http_code}" http://<mirror-node-host>:5551/swagger/metrics/
# If output is 200, authentication is disabled. If 401, it is enabled.

# Step 2: Retrieve full metrics dump (no credentials)
curl -s http://<mirror-node-host>:5551/swagger/metrics/

# Step 3: Extract per-route transaction history access counts
curl -s http://<mirror-node-host>:5551/swagger/metrics/ | \
  grep 'api_request_total' | \
  grep 'transactions'

  grep 'api_request_duration_milliseconds' | \
