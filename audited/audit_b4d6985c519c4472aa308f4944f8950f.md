### Title
Unauthenticated Full-Range DB Scan via Default Bounds in `getHooks()` Enables Resource Exhaustion

### Summary
The `getHooks()` endpoint in `HooksController` performs a full `hookId` range scan (`lowerBound=0` to `Long.MAX_VALUE`) by default with no authentication requirement visible in the controller. An unprivileged attacker can repeatedly issue requests with `limit=100` (the enforced maximum), causing `hookRepository.findByOwnerIdAndHookIdBetween()` to execute the widest possible indexed range scan on every call, driving up cumulative DB CPU and I/O load without any per-client throttle.

### Finding Description
**Code path:**

- `HooksController.java` lines 81–102 (`getHooks()`) and lines 132–156 (`hooksRequest()`)
- `HookServiceImpl.java` lines 32–53 (`getHooks()`)
- `HookRepository.java` line 15 (`findByOwnerIdAndHookIdBetween`)

**Root cause:**

In `hooksRequest()`, `lowerBound` is initialized to `0L` and `upperBound` to `Long.MAX_VALUE` (imported as `MAX_VALUE` at line 5). When no `hook.id` filter is supplied — or when the attacker explicitly supplies `hook.id=gte:0` and `hook.id=lte:9223372036854775807` — these defaults are used unchanged:

```java
long lowerBound = 0L;
long upperBound = MAX_VALUE;
``` [1](#0-0) 

The service then unconditionally calls:

```java
return hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), lowerBound, upperBound, page);
``` [2](#0-1) 

There is no `@PreAuthorize`, no session check, and no rate-limit annotation anywhere in `HooksController`. [3](#0-2) 

**Why the explicit "crafted" parameters are not special — but the issue is still real:**
The explicit `hook.id=gte:0` / `hook.id=lte:MAX_VALUE` parameters produce exactly the same `lowerBound`/`upperBound` as the defaults. The widest possible scan is therefore the *default* behavior of every call to this endpoint, not something that requires special crafting. Any unauthenticated call with `limit=100` and no `hook.id` EQ filter triggers the same maximum-cost query.

### Impact Explanation
Each request forces the DB to execute a range scan over the entire `hookId` space for the target `ownerId`, bounded only by the `LIMIT 100` page clause. With a high-cardinality `hook` table, this translates to a large index range scan plus up to 100 row fetches and full JSON serialization per response. Because there is no per-IP or per-client throttle visible in the controller, an attacker issuing hundreds of concurrent requests can sustain elevated DB CPU/IO, potentially exceeding the 30% resource-consumption threshold cited in the scope.

### Likelihood Explanation
The endpoint is HTTP GET, requires no credentials, and accepts a publicly documented path parameter (`ownerId`). Any valid `ownerId` (obtainable from public ledger data) is sufficient. The attack is trivially scriptable with `curl` or any HTTP load tool, is repeatable indefinitely, and requires no special knowledge beyond the API path.

### Recommendation
1. **Require authentication** — add `@PreAuthorize` or equivalent Spring Security enforcement on `getHooks()`.
2. **Add rate limiting** — apply a per-IP/per-client request-rate filter (e.g., Bucket4j, Spring Cloud Gateway rate limiter) on this endpoint.
3. **Tighten default bounds** — instead of defaulting `upperBound` to `Long.MAX_VALUE`, require at least one explicit `hook.id` filter or default to a narrow recent window.
4. **Add a DB query timeout** — configure a statement timeout on the repository query so runaway scans are killed server-side.

### Proof of Concept
```bash
# Widest-range scan (explicit parameters — identical to default behavior)
curl -s "https://<mirror-node>/api/v1/accounts/0.0.12345/hooks?hook.id=gte:0&hook.id=lte:9223372036854775807&limit=100"

# Equivalent default-behavior call (no hook.id params needed)
curl -s "https://<mirror-node>/api/v1/accounts/0.0.12345/hooks?limit=100"

# Flood script to drive resource consumption
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.12345/hooks?limit=100" &
done
wait
```

Both requests trigger `findByOwnerIdAndHookIdBetween(ownerId, 0, 9223372036854775807, PageRequest(size=100))`. Repeated in parallel, this drives sustained full-range index scans on the `hook` table, increasing DB processing load well above the 30% threshold without any brute-force credential attack.
