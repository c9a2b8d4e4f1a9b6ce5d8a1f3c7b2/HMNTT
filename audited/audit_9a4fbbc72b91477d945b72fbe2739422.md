### Title
Unauthenticated Double DB Query Chain in `getContractStateById` Enables Non-Network DoS

### Summary
The public `GET /api/v1/contracts/:contractId/state` endpoint unconditionally executes two sequential database queries per request when the `:contractId` path parameter is an EVM address: one to resolve the EVM address to an internal contract ID, and a second to fetch contract state. There is no rate limiting in the REST service, no authentication requirement, and no caching guarantee, allowing any unprivileged attacker to exhaust the PostgreSQL connection pool through concurrent requests.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 15 registers the route with no authentication or rate-limiting middleware: [1](#0-0) 

`getContractStateById` in `rest/controllers/contractController.js` lines 964–971 executes the two-query chain unconditionally: [2](#0-1) 

**DB Query 1** — `computeContractIdFromString` detects an EVM address and calls `getContractIdByEvmAddress`, which issues a `SELECT` against the `entity` table: [3](#0-2) [4](#0-3) 

The SQL template used: [5](#0-4) 

**DB Query 2** — `getContractStateByIdAndFilters` issues a second `SELECT` against `contract_state` (or `contract_state_change` with timestamp): [6](#0-5) 

**Root cause:** The handler always performs both queries sequentially with no short-circuit, no per-IP rate limiting, and no mandatory caching layer. The REST server (`rest/server.js`) registers only `authHandler` (which is a no-op for unauthenticated requests), `requestLogger`, optional metrics, optional response cache, `responseHandler`, and `handleError` — no rate-limiting middleware exists for the REST service: [7](#0-6) 

The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3/` service, not in `rest/`: [8](#0-7) 

The DB pool has a configurable `maxConnections` but this only bounds the pool size — it does not prevent request queuing and eventual timeout-based service degradation: [9](#0-8) 

### Impact Explanation
An attacker sending N concurrent `GET /api/v1/contracts/0x<valid-40-hex>/state` requests forces 2×N database queries. With a default pool size, the pool is exhausted quickly, causing all other REST API endpoints (accounts, transactions, tokens, etc.) to queue or fail. This is a non-network-based DoS affecting the entire mirror node REST API surface. The `entity` table query (`WHERE evm_address = $1`) and the `contract_state` query are both non-trivial table scans under load. The impact is complete REST API unavailability for legitimate users.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. Any attacker who can reach the public REST endpoint can trigger this. The attack is trivially scriptable with standard HTTP tooling (e.g., `ab`, `wrk`, `curl` in a loop). It is repeatable indefinitely. The only partial mitigation is the optional Redis response cache (`config.cache.response.enabled && config.redis.enabled`), which is disabled by default and only helps for repeated identical requests — an attacker can trivially vary the EVM address to bypass it.

### Recommendation
1. **Add rate limiting to the REST service** — integrate a per-IP rate limiter (e.g., `express-rate-limit`) as Express middleware in `rest/server.js` before route handlers, specifically targeting high-cost endpoints.
2. **Cache EVM-address-to-contract-ID resolution** — `getContractIdByEvmAddress` results are immutable once a contract is deployed; cache the mapping in-process (e.g., LRU cache) or in Redis to eliminate DB Query 1 on repeated EVM addresses.
3. **Add a DB connection pool statement timeout and max queue depth** — already partially present via `statementTimeout`, but enforce a request queue limit to shed load early.
4. **Consider requiring authentication for state queries** or applying stricter query limits for unauthenticated callers.

### Proof of Concept

```bash
# Step 1: Identify any valid CREATE2 EVM address from the network
EVM_ADDR="0x71eaa748d5252be68c1185588beca495459fdba4"

# Step 2: Send 500 concurrent requests (no auth, no special headers)
ab -n 5000 -c 500 \
  "https://<mirror-node-host>/api/v1/contracts/${EVM_ADDR}/state"

# Expected result:
# - DB connection pool exhausted within seconds
# - All /api/v1/* endpoints begin returning 500/503 or timing out
# - Legitimate users cannot query transactions, accounts, tokens, etc.

# Step 3: Vary EVM address to bypass response cache
for i in $(seq 1 1000); do
  ADDR=$(printf "0x%040x" $i)
  curl -s "https://<mirror-node-host>/api/v1/contracts/${ADDR}/state" &
done
wait
# Each request triggers 2 DB queries regardless of 404 result from Query 1
```

      throw new Error(`More than one contract with the evm address 0x${create2EvmAddress} have been found.`);

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

                .build();

function parseDbPoolConfig() {
  const {pool} = getConfig().db;
  const configKeys = ['connectionTimeout', 'maxConnections', 'statementTimeout'];
  configKeys.forEach((configKey) => {
    const value = pool[configKey];
    const parsed = parseInt(value, 10);
    if (Number.isNaN(parsed) || parsed <= 0) {
      throw new InvalidConfigError(`invalid value set for db.pool.${configKey}: ${value}`);
    }
    pool[configKey] = parsed;
  });
}
