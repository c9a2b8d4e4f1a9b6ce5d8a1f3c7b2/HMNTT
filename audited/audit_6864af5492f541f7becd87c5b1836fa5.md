### Title
Shared Global `Schedulers.boundedElastic()` Thread Pool Exhaustion via Unauthenticated Concurrent Subscriptions in `PollingTopicMessageRetriever`

### Summary
`PollingTopicMessageRetriever` stores `Schedulers.boundedElastic()` in a field at construction time, but `Schedulers.boundedElastic()` returns the **global JVM-wide singleton** bounded elastic scheduler shared across the entire Reactor application. Each subscription's `poll()` executes a synchronous blocking database query on a thread from this shared pool. Because there is no authentication or global subscription limit on the gRPC endpoint, an unprivileged attacker can open enough concurrent subscriptions across multiple connections to exhaust the shared thread pool, causing delayed or failed delivery for all other subscribers.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever` constructor, line 41:
```java
scheduler = Schedulers.boundedElastic();
```
`Schedulers.boundedElastic()` is a static method that returns `BOUNDED_ELASTIC` — a JVM-wide static singleton (not a new scheduler instance). This same global pool is also used by `PollingTopicListener` (line 31), `SharedPollingTopicListener` (line 37), `SharedTopicListener` (line 25), and `TopicMessageServiceImpl` (line 70).

`retrieve()`, lines 51–62:
```java
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(...)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))          // delay + re-subscription on global pool
    ...
    .timeout(retrieverProperties.getTimeout(), scheduler)  // timeout also on global pool
```

`poll()`, lines 65–79:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
`Flux.fromStream()` iterates the stream synchronously on the calling thread. When `repeatWhen` fires, it re-subscribes to `Flux.defer(() -> poll(context))` on the scheduler thread, so the blocking DB query occupies that thread for its full duration.

**Root cause:** The false assumption is that `Schedulers.boundedElastic()` creates a private scheduler. It does not — it returns the global singleton. All concurrent subscriptions compete for the same bounded thread pool (`10 × availableProcessors` threads by default, e.g. 80 on an 8-core host). Slow or concurrent DB queries hold threads, starving other subscriptions' scheduled polls.

**Exploit flow:**
1. Attacker opens many gRPC connections (no authentication required — `ConsensusController.subscribeTopic()` has no auth check, only a topic-existence check).
2. Each connection allows up to 5 concurrent calls (`maxConcurrentCallsPerConnection = 5`), so N connections yield 5N concurrent subscriptions.
3. Each subscription's `poll()` runs a synchronous DB query on the global bounded elastic scheduler thread.
4. With enough concurrent subscriptions and non-trivial DB query latency, all scheduler threads are occupied.
5. Pending poll tasks for legitimate subscriptions queue up, delaying message delivery.
6. If the queue delay exceeds `timeout` (60 s), the `.timeout(retrieverProperties.getTimeout(), scheduler)` fires — but the timeout itself is also scheduled on the same exhausted pool, further delaying or preventing the timeout signal, causing subscriptions to hang rather than fail cleanly.
7. `retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))` causes failed subscriptions to retry indefinitely, sustaining the load.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5`: Per-connection only. Attacker uses multiple connections — no global connection or subscription cap exists in the code.
- `timeout = 60s`: Scheduled on the same exhausted pool; cannot fire promptly under thread starvation.
- `retryWhen(Long.MAX_VALUE)`: Amplifies attacker load by retrying indefinitely.
- No authentication, no rate limiting on the gRPC subscribe endpoint.

### Impact Explanation
Thread pool exhaustion on the global `boundedElastic` scheduler causes delayed or completely missed message delivery for all legitimate subscribers sharing the same JVM. Because the timeout signal is also scheduled on the same pool, subscriptions can hang indefinitely rather than failing with a clean error. This constitutes a full denial-of-service against the mirror node's topic subscription service — a core protocol function — without any privileged access.

### Likelihood Explanation
The gRPC endpoint is unauthenticated and publicly reachable. The only per-connection limit is 5 concurrent calls. On a typical 8-core deployment (80 bounded elastic threads), an attacker needs only 16 connections (80 threads ÷ 5 calls/connection) to saturate the pool if DB queries are slow. The unthrottled path (`pollingFrequency = 20ms`) is especially aggressive: with 100 ms DB query latency, threads are held ~83% of the time, requiring fewer than 100 concurrent subscriptions to exhaust the pool. This is trivially achievable with a standard gRPC client library.

### Recommendation
1. **Use a private scheduler per retriever** — replace `Schedulers.boundedElastic()` with `Schedulers.newBoundedElastic(threadCap, queueSize, "retriever")` so the retriever's thread pool is isolated from the rest of the application.
2. **Enforce a global subscription limit** — add an `AtomicInteger` counter in `PollingTopicMessageRetriever` (or `TopicMessageServiceImpl`) and reject new subscriptions when the cap is reached, returning `RESOURCE_EXHAUSTED`.
3. **Make DB queries non-blocking** — replace `Flux.fromStream(topicMessageRepository.findByFilter(...))` with a reactive R2DBC query so threads are not held during DB I/O.
4. **Add per-IP or global connection rate limiting** at the Netty/gRPC layer in `GrpcConfiguration`.

### Proof of Concept
```
Preconditions:
  - Mirror node gRPC port 5600 reachable, no authentication required
  - Any valid topic ID (e.g., 0.0.1000) exists in the DB
  - DB query latency >= 50ms (normal under load)

Steps:
  1. Open 20 TCP connections to grpc://mirror-node:5600
  2. On each connection, open 5 concurrent subscribeTopic streams:
       ConsensusTopicQuery { topicID: {topicNum: 1000}, consensusStartTime: {seconds: 0} }
     (100 total concurrent subscriptions)
  3. Each subscription triggers PollingTopicMessageRetriever.retrieve() ->
     poll() -> findByFilter() running synchronously on the global boundedElastic pool
  4. On an 8-core host (80 threads), all threads are occupied within seconds
  5. Observe: legitimate subscriber's poll tasks queue up; message delivery
     latency grows unboundedly; subscriptions eventually hang or timeout
  6. Confirm via metrics: hiero.mirror.grpc.subscribers gauge spikes;
     message delivery rate for pre-existing subscribers drops to zero
```
