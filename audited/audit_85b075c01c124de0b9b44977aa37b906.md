### Title
Unauthenticated Double DB Query Amplification via `?scheduled=true` Exhausts Connection Pool in `getTransactionsByIdOrHash`

### Summary
In `rest/transactions.js`, the `getTransactionsByIdOrHash()` handler unconditionally issues a second, significantly more expensive `pool.queryQuietly()` call whenever `mayMissLongTermScheduledTransaction()` returns `true`. Any unauthenticated user can reliably trigger this path by appending `?scheduled=true` to any syntactically valid transaction ID, causing two sequential DB queries per request ŌĆö the second scanning a ~62-day timestamp window ŌĆö with no rate limiting in place. Flooding this endpoint exhausts the default 10-connection pool and denies service to all other users.

### Finding Description

**Code path ŌĆö `rest/transactions.js`, lines 930ŌĆō938:**

```js
const {rows} = await pool.queryQuietly(query, params).then((result) => {
  if (!mayMissLongTermScheduledTransaction(isTransactionHash, scheduled, result.rows)) {
    return result;
  }
  params[params.upperConsensusTimestampIndex] =
    params[params.lowerConsensusTimestampIndex] + maxScheduledTransactionConsensusTimestampRangeNs;
  return pool.queryQuietly(query, params);   // ŌåÉ second query, always issued
});
```

**Root cause ŌĆö `mayMissLongTermScheduledTransaction`, lines 991ŌĆō993:**

```js
} else if (scheduled && transactions.length === 0) {
  return true;   // ŌåÉ triggered whenever scheduled=true and first query returns nothing
}
```

When `?scheduled=true` is supplied and the first query returns zero rows (which happens for any non-existent transaction ID), `mayMissLongTermScheduledTransaction(false, true, [])` unconditionally returns `true`. The handler then re-issues the same query with the upper consensus timestamp bound widened to `lowerTimestamp + maxScheduledTransactionConsensusTimestampRangeNs` ŌĆö a window of **89,285 minutes (~62 days)** versus the first query's 35-minute window.

**`pool.queryQuietly` without `preQueryHint` ŌĆö `rest/utils.js`, line 1520:**

```js
result = await this.query(query, params);
```

This uses the pool-level `pg.Pool.query()`, which borrows a connection, runs the query, and releases it. The two sequential calls each hold a connection for the full duration of their respective queries (up to the 20-second `statementTimeout`). With the default pool of 10 connections and no rate limiting, 10 concurrent flood requests saturate the pool; subsequent requests queue indefinitely.

**No rate limiting exists** in the REST API application layer ŌĆö confirmed by absence of any `rateLimit`, `express-rate-limit`, or `slowDown` middleware in `rest/server.js` or any handler file.

### Impact Explanation

The default `maxConnections` is **10** (`hiero.mirror.rest.db.pool.maxConnections: 10`). Each flood request causes two sequential queries, the second scanning a 62-day timestamp range. With `statementTimeout` at 20 seconds, a single request can hold a pool connection for up to 40 seconds total. Ten concurrent flood requests fully saturate the pool; all legitimate API traffic (balance queries, contract lookups, topic messages, etc.) queues behind them, producing effective service denial for the entire mirror node REST API. This is a non-network-based DoS requiring no special privileges.

### Likelihood Explanation

The attack requires zero authentication, zero knowledge of real on-chain data, and only a syntactically valid transaction ID string (e.g., `0.0.1-1234567890-000000000`). The trigger parameter `?scheduled=true` is documented in the public OpenAPI spec. A single attacker with a modest HTTP flood tool can sustain the condition indefinitely. The attack is repeatable, stateless, and trivially scriptable.

### Recommendation

1. **Add application-level rate limiting** (e.g., `express-rate-limit`) on the `/api/v1/transactions/:id` route, keyed by IP or API token.
2. **Guard the second query** with a short-circuit: only issue it if the first query actually returned a SCHEDULECREATE row with a successful result code, not when the result set is empty due to a non-existent transaction ID. Specifically, when `scheduled === true` and `transactions.length === 0`, the transaction simply does not exist ŌĆö return `NotFoundError` immediately rather than issuing a second expensive query.
3. **Increase the default pool size** or add a per-IP connection/concurrency limit at the infrastructure layer (pgBouncer `max_user_client_connections` is already configurable but not enforced by default for `mirror_rest`).
