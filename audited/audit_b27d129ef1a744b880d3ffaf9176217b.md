### Title
Unauthenticated High-Frequency EVM Address Lookups Cause Database Load Amplification via Uncached `findByEvmAddress()` in GraphQL Module

### Summary
The `findByEvmAddress()` method in `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java` carries no `@Cacheable` annotation and is invoked on every GraphQL `account` query that supplies an `evmAddress` input. The GraphQL module has no application-level rate limiter, meaning any unauthenticated external user can issue an unbounded stream of distinct (or repeated) EVM-address lookups, each of which executes a full table scan against the `entity` table, amplifying database load and degrading availability of all Hashgraph history query responses.

`EntityServiceImpl.getByEvmAddressAndType()` calls `entityRepository.findByEvmAddress()` directly with no caching wrapper: [1](#0-0) 

The repository method itself has no `@Cacheable` annotation: [2](#0-1) 

**Contrast with the `web3` module**, where the equivalent lookup is protected by `@Cacheable`: [3](#0-2) 

**Root cause:** The graphql-module `EntityRepository` was never annotated with `@Cacheable`, and `EntityServiceImpl` adds no in-process caching layer. The `CacheProperties` bean in the graphql module only caches parsed query documents, not entity lookups: [4](#0-3) 

**No application-level rate limiter exists in the graphql module.** The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`, `ThrottleProperties`) lives entirely in the `web3` module and is never wired into the graphql module: [5](#0-4) 

### Impact Explanation

Every unauthenticated POST to `/graphql/alpha` with `{ account(input: { evmAddress: "0x<addr>" }) { ... } }` executes `SELECT * FROM entity WHERE evm_address = ? AND deleted IS NOT TRUE` directly against the database. An attacker issuing thousands of such requests per second with rotating EVM addresses (preventing any DB-level query plan cache benefit) causes sustained, unbounded read load on the `entity` table. This degrades or denies availability of all mirror-node GraphQL queries, including Hashgraph history lookups, for legitimate users. Severity is **High** (availability impact on a public read-only API with no authentication barrier).

### Likelihood Explanation

The GraphQL endpoint is publicly reachable with no authentication requirement. A single attacker with a commodity machine and a simple loop or tool (e.g., `ab`, `wrk`, or a Python script) can sustain thousands of requests per second. No special knowledge, credentials, or on-chain assets are required. The attack is trivially repeatable and scriptable.

The only infrastructure-level mitigation is an optional GCP backend policy (`maxRatePerEndpoint: 250`) that is not enforced at the application layer and is only active when the GCP gateway is enabled: [6](#0-5) 

250 req/s still translates to 250 uncached DB queries per second per pod, which is significant under sustained attack.

### Recommendation

1. **Add `@Cacheable`** to `findByEvmAddress()` in the graphql module's `EntityRepository`, mirroring the pattern used in the `web3` module.
2. **Add an application-level rate limiter** to the graphql module (e.g., a Bucket4j filter similar to `ThrottleManagerImpl` in `web3`), enforcing a per-IP or global requests-per-second cap.
3. **Enforce the GCP `maxRatePerEndpoint`** at the infrastructure level as a defense-in-depth measure, and ensure it is always active in production deployments.

### Proof of Concept

```bash
# Send 5000 concurrent EVM-address lookups with unique addresses
for i in $(seq 1 5000); do
  curl -s -X POST https://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input: { evmAddress: \\\"$(openssl rand -hex 20)\\\" }) { balance } }\"}" &
done
wait
```

Each request bypasses any cache (unique addresses), executes a full `SELECT * FROM entity WHERE evm_address = ?` query, and returns immediately. Monitoring the database will show a linear spike in active queries and I/O, degrading response times for all other GraphQL consumers querying Hashgraph history.
