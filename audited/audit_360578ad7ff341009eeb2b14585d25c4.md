### Title
Log Injection via Unsanitized User-Controlled `err.message` in `handleError()`

### Summary
The `handleError()` middleware in `rest/middleware/httpErrorHandler.js` logs `err.message` verbatim for all client-error (4xx) responses. Because `parseHbarParam()` in `rest/utils.js` constructs an `InvalidArgumentError` whose message directly embeds the raw, URL-decoded query-parameter value, an unauthenticated attacker can inject arbitrary newlines and ANSI escape sequences into the application's structured log output. The logger performs no sanitization before writing to stdout.

### Finding Description

**Exact code path:**

1. **Trigger point** — `rest/utils.js` line 231:
   ```js
   throw new InvalidArgumentError(`Invalid hbar parameter value: ${hbarParam}. Must be 'true' or 'false'`);
   ```
   `hbarParam` is the raw, URL-decoded query-string value supplied by the caller. No stripping of `\n`, `\r`, or ANSI CSI sequences occurs before it is interpolated into the error message.

2. **Logging sink** — `rest/middleware/httpErrorHandler.js` lines 31–35:
   ```js
   if (shouldReturnMessage(statusCode)) {
     errorMessage = err.message;
     logger.warn(
       `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode} ${err.constructor.name} ${errorMessage}`
     );
   }
   ```
   `shouldReturnMessage` returns `true` for any client error (HTTP 400), so the full, attacker-influenced `err.message` is concatenated into the log string.

3. **Logger** — `rest/logger.js` lines 69–70:
   ```js
   const text = `${time} ${levelName} ${requestId} ${msg}${stack}\n`;
   this.#write(text);
   ```
   `msg` is written to stdout as-is; there is no call to strip control characters, escape sequences, or newlines.

**Root cause / failed assumption:** The code assumes `err.message` for client errors is safe to log verbatim. That assumption fails because `parseHbarParam` (and similar helpers) embed the raw query-parameter *value* — not just a static key name — into the error message, and Express's query parser URL-decodes `%0a` → `\n` before the value ever reaches application code.

### Impact Explanation
An attacker can forge arbitrary log lines in the mirror node's stdout stream. In environments where logs are ingested by a SIEM, log-aggregation pipeline, or audit system, injected lines are indistinguishable from genuine entries. This can:
- Fabricate or suppress apparent contract-query audit events.
- Poison log-based alerting rules (e.g., inject a fake `ERROR` line to trigger false alarms, or a fake `INFO` line to bury a real error).
- Corrupt ANSI-rendered terminal output for operators monitoring logs live (via `\x1b[...m` sequences).

Severity is **medium**: no direct fund loss or state corruption, but audit-log integrity is compromised for an unauthenticated, zero-privilege attacker.

### Likelihood Explanation
Exploitation requires only an HTTP GET request with a URL-encoded newline in the `hbar` query parameter on any contract-results endpoint that calls `parseHbarParam` (confirmed reachable via `rest/controllers/contractController.js`). No authentication, no special role, no rate-limit bypass needed. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Sanitize before logging** — strip or replace `\n`, `\r`, and ANSI escape sequences in the logger's `#log` method before constructing `text`, or add a dedicated sanitizer called at the `logger.warn(...)` call site in `handleError`.
2. **Do not embed raw user values in error messages** — `parseHbarParam` and similar helpers should reference only the parameter *name* (a static string) in the thrown error, not the parameter *value*. Follow the pattern already used in `GenericControllerAdvice.java` line 88–89 (`"Explicitly don't log the parameter value to avoid cross-site scripting attacks"`).
3. **Structured logging** — emit log records as JSON objects with distinct fields for `ip`, `method`, `url`, `errorMessage`, etc., so injected newlines cannot create new top-level log records.

### Proof of Concept

```
GET /api/v1/contracts/results?hbar=%0a2026-04-28T00:00:00.000Z%20WARN%20fakeReqId%20INJECTED%3A%20contract%200x1234%20drained HTTP/1.1
Host: <mirror-node-host>
```

**Expected log output (two lines instead of one):**
```
2026-04-28T12:00:00.000Z WARN realReqId 1.2.3.4 GET /api/v1/contracts/results?hbar=... in 3 ms: 400 Bad Request InvalidArgumentError Invalid hbar parameter value:
2026-04-28T00:00:00.000Z WARN fakeReqId INJECTED: contract 0x1234 drained. Must be 'true' or 'false'
```

The second line is entirely attacker-controlled and appears as a legitimate log entry to any downstream log consumer.
