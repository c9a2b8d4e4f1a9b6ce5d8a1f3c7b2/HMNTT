### Title
Unbounded Key Collection Enables DoS via Expensive SQL IN-Clause in `getHookStorageChange()`

### Summary
The `getHookStorageChange()` method in `HookServiceImpl` contains an early-return guard that only fires when all user-supplied keys fall outside the key range. An unprivileged attacker can trivially bypass this guard by supplying an arbitrarily large number of keys that are all within range, causing `findByKeyInAndTimestampBetween()` to execute a `DISTINCT ON` SQL query with an unbounded `key IN (...)` clause over a maximum timestamp range, exhausting database CPU and I/O.

### Finding Description
**Code path:**

In `getHookStorageChange()` [1](#0-0) , the guard at line 93 is:

```java
if (keysInRange.isEmpty() && requestHasKeys) {
    return new HookStorageResult(ownerId, List.of());
}
``` [2](#0-1) 

`keysInRange` is computed by `HookStorageRequest.getKeysInRange()`, which filters the user-supplied `keys` collection to those satisfying `keyLowerBound <= key <= keyUpperBound`: [3](#0-2) 

**Root cause:** There is no upper bound on the size of the `keys` collection anywhere in `HookStorageRequest` or `HookServiceImpl`. The guard's only purpose is to short-circuit when every supplied key is out of range. If an attacker supplies N keys all within range, `keysInRange` is non-empty, the guard does not fire, and execution falls through to:

```java
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
``` [4](#0-3) 

Note that `keys` (the full, unbounded user-supplied collection) is passed — not `keysInRange`. The underlying SQL query is:

```sql
select distinct on (key) ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id = :hookId
  and key in (:keys)
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
``` [5](#0-4) 

`DISTINCT ON (key)` requires a sort/group pass over all matching rows. With thousands of keys in the IN clause and a wide timestamp range, the database must evaluate a massive predicate set and perform an expensive sort, regardless of the pagination `limit` (default 25) which only caps the result set, not the work done.

**Failed assumption:** The guard assumes that filtering out-of-range keys is sufficient protection. It does not account for an attacker deliberately supplying a large number of in-range keys to maximize query cost.

### Impact Explanation
A single malicious request with, e.g., 10,000 in-range keys and a maximum timestamp range generates a query that forces the database to evaluate a 10,000-element IN clause combined with a `DISTINCT ON` sort over potentially millions of rows. Repeated at modest concurrency (e.g., 5–10 concurrent requests), this can saturate database CPU and I/O, degrading or halting mirror node processing capacity for all users. This matches the stated scope of ≥30% processing node degradation without brute force.

### Likelihood Explanation
No authentication or authorization is required — the endpoint is publicly accessible. The attacker needs only to know the valid `ownerId`/`hookId` and any key range (which can be the full byte range). The attack is trivially repeatable with a simple HTTP client loop. No special tooling, credentials, or protocol knowledge beyond the REST API schema is needed.

### Recommendation
1. **Enforce a hard cap on the number of keys per request** (e.g., ≤100) in `HookStorageRequest` validation, rejecting requests that exceed it with HTTP 400.
2. **Pass `keysInRange` instead of `keys`** to `findByKeyInAndTimestampBetween()` (line 104) to at minimum ensure the range filter is applied before the query.
3. **Enforce a maximum timestamp window** (e.g., ≤7 days) in `Bound.adjustUpperBound()` / `getAdjustedLowerRangeValue()` to limit the scan range.
4. Consider **rate limiting** on the historical storage endpoint per IP/client.

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }

    public List<byte[]> getKeysInRange() {
        if (keys.isEmpty()) {
            return List.of();
        }

        return keys.stream()
                .filter(key -> Arrays.compareUnsigned(key, keyLowerBound) >= 0
                        && Arrays.compareUnsigned(key, keyUpperBound) <= 0)
                .toList();
    }
```
