### Title
Unbounded Per-Operation Sequential DB Queries in `updateOperationAccountAlias()` Enable Resource Exhaustion via `/block` Endpoint

### Summary
`updateOperationAccountAlias()` in `rosetta/app/services/block_service.go` issues one synchronous DB query per operation per transaction on every LRU cache miss, with no batching and no application-level rate limiting on the `/block` endpoint. An unprivileged attacker can identify blocks with many transactions (each containing many unique-account operations), then rotate requests across many such blocks to continuously evict the LRU cache and force repeated O(transactions × operations) DB round-trips per request, sustaining elevated database load.

### Finding Description

**Exact code path:**

`Block()` (lines 47–74) fetches all transactions for a block, caps them at `maxTransactionsInBlock`, then calls `updateOperationAccountAlias()` (lines 104–130) with the full capped slice.

Inside `updateOperationAccountAlias()`:
```
for _, transaction := range transactions {          // up to maxTransactionsInBlock
    for index := range operations {                 // unbounded per transaction
        if cached, found = s.entityCache.Get(accountId.GetId()); !found {
            result, err := s.accountRepo.GetAccountAlias(ctx, accountId)  // DB query
            ...
            s.entityCache.Set(result.GetId(), result)
        }
    }
}
``` [1](#0-0) 

`GetAccountAlias()` executes `SELECT alias, id FROM entity WHERE id = @id` synchronously for every cache miss: [2](#0-1) 

**Root cause:** There is no batching of account lookups; each operation triggers an independent DB round-trip on a cache miss. The LRU cache key is `accountId.GetId()` (numeric entity ID), and the cache is populated with `result.GetId()`. While this is correct for repeated requests to the *same* block, the cache is a process-wide LRU with a fixed capacity (default 524,288 entries): [3](#0-2) 

An attacker who rotates requests across many different blocks—each containing many distinct account IDs—continuously evicts previously cached entries, forcing every subsequent request for a "heavy" block to re-issue all its DB queries.

**No application-level rate limiting exists** for the Rosetta Go service itself. The only rate limiting is an optional Traefik middleware (10 req/s per host, 5 in-flight) that is infrastructure-level and not enforced in the application code: [4](#0-3) 

### Impact Explanation
Each `/block` request for a block with T transactions × O operations per transaction (all unique account IDs, cache cold) issues T×O sequential DB queries. On Hedera mainnet, blocks regularly contain hundreds of transactions each with multiple crypto-transfer operations. Sustained cache-eviction cycling across many such blocks forces the DB connection pool (default 100 open connections) to service a continuous stream of single-row `entity` lookups, increasing DB CPU, I/O, and connection utilization well above baseline. The DB statement timeout is 20 seconds, meaning each query can hold a connection for up to 20 s before timing out, amplifying pool exhaustion risk. [5](#0-4) 

### Likelihood Explanation
The `/block` endpoint requires no authentication and is publicly reachable. Block identifiers (index or hash) are trivially enumerable from the chain. An attacker needs only an HTTP client and knowledge of which blocks are "heavy." The attack is repeatable indefinitely, requires no special privileges, and is not prevented by any in-process control.

### Recommendation
1. **Batch account lookups**: Collect all unique `AccountId` values across all operations before entering the loop, issue a single `SELECT alias, id FROM entity WHERE id = ANY(@ids)` query, then populate the cache and update operations in a second pass.
2. **Add application-level rate limiting** to the Rosetta HTTP server (e.g., `golang.org/x/time/rate`) for the `/block` and `/block/transaction` endpoints, independent of infrastructure middleware.
3. **Cap operations per transaction** in the response, similar to how `maxTransactionsInBlock` caps transactions per block.

### Proof of Concept
1. Identify a block on the target network with a large number of transactions (up to `maxTransactionsInBlock`) each containing many crypto-transfer operations with distinct account IDs. On mainnet this is common for airdrop or batch-payment blocks.
2. Send a burst of `/block` requests for ~600,000 distinct blocks (to fill and evict the 524,288-entry LRU cache with fresh account IDs).
3. Immediately re-request the "heavy" block identified in step 1. All account IDs are now evicted from cache; the server issues one DB query per operation.
4. Repeat steps 2–3 in a loop from a single unauthenticated HTTP client. Monitor DB CPU and query rate; they will rise proportionally to the number of unique account IDs per block × request rate.

func (ar *accountRepository) GetAccountAlias(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return accountId, nil
		}

		return zero, hErrors.ErrDatabaseError
	}

	if len(entity.Alias) == 0 {
		return accountId, nil
	}

	accountAlias, err := types.NewAccountIdFromEntity(entity)
	if err != nil {
		log.Warnf("Failed to create AccountId from alias '0x%s': %v", hex.EncodeToString(entity.Alias), err)
		return accountId, nil
	}

	return accountAlias, nil
}
```
