### Title
Unauthenticated Historical EVM Address Lookup Exhausts DB Connection Pool via Expensive UNION Query

### Summary
The `getContractById` handler in `rest/controllers/contractController.js` accepts a `timestamp=` query parameter from any unauthenticated user. When the `:contractId` path parameter is a create2 EVM address, `getContractByIdOrAddressContractEntityQuery()` builds a UNION query across both `entity` and `entity_history` tables filtered by `evm_address` and `timestamp_range`. There is no rate limiting on this REST endpoint, and the default DB connection pool is only 10 connections with a 20-second statement timeout, making it trivial to exhaust the pool and degrade service for all users.

This produces:
```sql
SELECT ... FROM entity e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND c.timestamp_range && $2 AND e.evm_address = $3
UNION
SELECT ... FROM entity_history e LEFT JOIN contract c ON e.id = c.id
WHERE e.type = 'CONTRACT' AND c.timestamp_range && $1 AND c.timestamp_range && $2 AND e.evm_address = $3
ORDER BY timestamp_range DESC
LIMIT 1
```

**Root cause:** The `entity_history` table can be very large (it accumulates all historical states). The UNION with a range-overlap condition (`&&`) on `timestamp_range` joined against `contract` is significantly more expensive than a simple primary-key lookup. No rate limiting exists on this REST endpoint — the throttle manager (`ThrottleManagerImpl`) only applies to the web3 `/contracts/call` POST endpoint, not to the REST API GET endpoints.

**Failed assumption:** The design assumes that the DB-level `statementTimeout` (20 s) and pool size (10 connections) are sufficient safeguards. They are not: an attacker sending ~10 concurrent requests keeps the pool permanently saturated, since each query can hold a connection for up to 20 seconds before timing out.

### Impact Explanation

With the default pool of 10 connections and a 20-second statement timeout, an attacker sending a sustained flood of ~10 requests/second causes every legitimate user's query to queue indefinitely, resulting in connection timeout errors (HTTP 500 or connection refused) for all concurrent users. The `entity_history` table grows unboundedly over time, making each such query progressively more expensive. No data is leaked or modified; the impact is pure service degradation (griefing DoS).

### Likelihood Explanation

No authentication, API key, or special knowledge is required. Any external user who knows the public REST API can send:
```
GET /api/v1/contracts/70f2b2914a2a4b783faefb75f459a580616fcb5e?timestamp=gte:0&timestamp=lte:9999999999
```
The attack is trivially scriptable with `curl` or any HTTP client. The attacker does not need a valid EVM address — even a non-existent address triggers the full UNION scan before returning 404.

### Recommendation

1. **Add per-endpoint rate limiting** to the REST API for `GET /contracts/:contractId` (and similar historical lookup endpoints), e.g., using an Express rate-limit middleware keyed by IP.
2. **Increase the DB pool size** or add a query queue with a maximum wait time to fail fast rather than holding connections.
3. **Add a short-circuit**: if the EVM address does not exist in `entity` (fast indexed lookup), skip the `entity_history` UNION entirely.
4. **Reduce the statement timeout** for this specific query class, or add an application-level timeout shorter than the DB-level 20 s.
5. **Enable the Redis response cache** (`hiero.mirror.rest.cache.response.enabled=true`) so repeated identical queries are served from cache rather than hitting the DB.

### Proof of Concept

```bash
# Flood the endpoint with 20 concurrent historical EVM address lookups
for i in $(seq 1 20); do
  curl -s "http://<mirror-node>/api/v1/contracts/70f2b2914a2a4b783faefb75f459a580616fcb5e?timestamp=gte:0000000001.000000000&timestamp=lte:9999999999.000000000" &
done
wait

# Legitimate request now times out or receives 500
curl -v "http://<mirror-node>/api/v1/contracts/0.0.1234"
```

**Expected result:** The legitimate request either hangs until the pool frees a connection or receives a connection-pool-exhaustion error, while the attacker's requests each hold a DB connection for up to 20 seconds executing the expensive UNION scan.

const getContractByIdOrAddressContractEntityQuery = ({timestampConditions, timestampParams, contractIdParam}) => {
  const conditions = [...timestampConditions];
  const params = [...timestampParams];
  const contractIdParamParts = EntityId.computeContractIdPartsFromContractIdValue(contractIdParam);

  if (contractIdParamParts.create2_evm_address) {
    const index = params.push(Buffer.from(contractIdParamParts.create2_evm_address, 'hex'));
    conditions.push(`${Entity.getFullName(Entity.EVM_ADDRESS)} = $${index}`);
  } else {
    const encodedId = EntityId.parseString(contractIdParam).getEncodedId();
    params.push(encodedId);
    conditions.push(`${Entity.getFullName(Entity.ID)} = $${params.length}`);
  }

  const tableUnionQueries = [getContractByIdOrAddressQueryForTable(Entity.tableName, conditions)];
  if (timestampConditions.length !== 0) {
    // if there is timestamp condition, union the result from both tables
    tableUnionQueries.push(
      'union',
      getContractByIdOrAddressQueryForTable(Entity.historyTableName, conditions),
      `order by ${Entity.TIMESTAMP_RANGE} desc`,
      `limit 1`
    );
  }

  return {
    query: tableUnionQueries.join('\n'),
    params,
  };
};
