### Title
Fee Multiplication via New Transaction Per Retry When Receipt Query Fails After Successful Submission

### Summary
`submitWithRetry()` in `pinger/transfer.go` creates a **brand-new** `TransferTransaction` on every retry iteration. If `Execute(client)` succeeds (transaction accepted and fee charged on-chain) but `resp.GetReceipt(client)` subsequently fails due to a network error, the code treats the attempt as failed and submits a fresh transaction on the next iteration. With the default `maxRetries = 10`, up to 11 distinct on-chain transactions — each incurring fees — can be submitted per tick while the operator account is drained.

### Finding Description

**Exact code path** — `pinger/transfer.go`, lines 23–57:

```go
for i := 1; i <= attempts; i++ {          // attempts = maxRetries + 1 = 11 (default)
    ...
    cryptoTransfer := hiero.NewTransferTransaction()   // NEW tx, NEW unique tx ID every iteration
        .AddHbarTransfer(...)
        .AddHbarTransfer(...)

    resp, err := cryptoTransfer.Execute(client)        // tx submitted on-chain; fee charged here
    if err == nil {
        receipt, rerr := resp.GetReceipt(client)       // separate gRPC call; can fail independently
        if rerr == nil {
            return nil                                 // only exit on receipt success
        }
        err = rerr                                     // receipt failure → treated as full failure
    }
    // loop continues → new transaction submitted next iteration
}
```

**Root cause**: The code conflates two independent operations — transaction submission (`Execute`) and receipt retrieval (`GetReceipt`). A successful `Execute` means the transaction is accepted by the network and fees are charged. `GetReceipt` is a subsequent query that can fail for reasons entirely unrelated to whether the transaction succeeded. The failed assumption is: *"if `GetReceipt` errors, the transaction did not execute."* This is false.

**Why no existing check prevents it**: There is no guard that records the transaction ID of a successfully submitted transaction and checks its status before retrying. Each iteration unconditionally calls `hiero.NewTransferTransaction()`, generating a fresh transaction ID, so Hedera's built-in duplicate-transaction deduplication does not apply across retries.

**Exploit flow**:
1. Attacker positions themselves to selectively interfere with gRPC receipt-query responses (e.g., ARP poisoning on the same LAN segment, or injecting TCP RSTs targeting the receipt-query response stream).
2. Per tick: `Execute` succeeds → transaction lands on-chain, fee charged → attacker drops/corrupts the `GetReceipt` response → `rerr != nil` → loop retries.
3. Repeat for all `maxRetries + 1` iterations.
4. Result: 11 on-chain transactions submitted per tick instead of 1.

### Impact Explanation
With default settings (`maxRetries = 10`, `interval = 1s`, `amountTinybar = 10000`), the operator account is charged fees for up to 11 transactions per second instead of 1 — an 11× fee amplification. Over time this drains the operator account. The pinger is a monitoring/liveness component; draining its operator account causes it to stop functioning, defeating its purpose. The impact is direct financial loss (fee drain) and loss of the monitoring capability the pinger provides.

### Likelihood Explanation
The precondition — network-level interference with receipt queries — is achievable by an attacker on the same LAN (ARP poisoning, TCP RST injection) or by anyone who can cause sustained network congestion on the path between the pinger and the Hedera consensus nodes. The attack is repeatable every tick (default: every second) and requires no credentials or application-level access. The attacker does not need to interfere with `Execute` at all — only with the receipt-query response, which is a separate gRPC call and can be targeted independently.

### Recommendation
1. **Reuse the transaction ID across receipt-query retries**: Build the `TransferTransaction` once before the loop, capture `resp` from the first successful `Execute`, and on subsequent iterations only retry `resp.GetReceipt(client)` rather than submitting a new transaction.
2. **Separate submission retries from receipt-query retries**: Only re-submit a new transaction if `Execute` itself fails (precheck error, node unavailable). If `Execute` succeeds, loop only on `GetReceipt` without calling `Execute` again.
3. **Idempotency guard**: Record the transaction ID of any successfully submitted transaction and, before submitting a new one, query whether that ID already has a receipt on-chain.

### Proof of Concept

**Preconditions**: Attacker has LAN-level access to the host running the pinger (or can inject TCP RSTs on the path to Hedera consensus nodes).

**Steps**:
1. Deploy the pinger with default config (`maxRetries=10`, `interval=1s`).
2. On the attacker machine, use `iptables` or `tc` to drop inbound TCP packets from Hedera consensus node IPs that carry gRPC receipt-query responses (identifiable by the gRPC method path `proto.CryptoService/getTransactionReceipts`), while leaving outbound transaction-submission packets unaffected.
3. Observe pinger logs: each tick logs 11 `attempt N/11 failed` lines.
4. Inspect the Hedera operator account transaction history: 11 distinct `CryptoTransfer` transactions appear per second, each with a unique transaction ID and each charged the standard fee.
5. After sustained attack, the operator account balance reaches zero and the pinger halts.
