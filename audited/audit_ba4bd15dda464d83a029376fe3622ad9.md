### Title
Cache Key Non-Normalization Enables Per-Format TTL Side-Channel on `/api/v1/accounts/<id>/rewards`

### Summary
`cacheKeyGenerator()` hashes `req.originalUrl` verbatim, so the four semantically-equivalent account identifier formats (`1001`, `0.0.1001`, `0x…3e9`, `KGNABD5…`) each produce a distinct Redis cache entry for the same underlying account. Because every cache-hit response carries `Cache-Control: public, max-age=<remaining_redis_ttl>`, an unprivileged attacker can probe all formats, observe which ones are cache hits and what their residual TTL is, and thereby reconstruct the exact wall-clock time at which each format was last queried by any other user.

### Finding Description

**Exact code path**

`rest/middleware/responseCacheHandler.js`, `cacheKeyGenerator()`, lines 151-153:

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

`req.originalUrl` is the raw, un-normalised URL string. The code comment at lines 148-149 explicitly acknowledges the missing normalisation: *"In the future, this will utilize Edwin's request normalizer (9113)."*

The existing `requestNormalizer.js` (`normalizeRequestQueryParams`) only sorts/collapses **query parameters**; it never touches the path segment that carries the account identifier. Consequently, the following URLs — all confirmed by the test fixture `rest/__tests__/specs/accounts/{id}/rewards/no-params.json` to resolve to the same account — hash to ten different MD5 digests and create ten independent Redis entries:

```
/api/v1/accounts/1001/rewards
/api/v1/accounts/0.1001/rewards
/api/v1/accounts/0.0.1001/rewards
/api/v1/accounts/0x00000000000000000000000000000000000003e9/rewards
/api/v1/accounts/KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ/rewards
… (plus realm-prefixed alias variants)
```

**TTL leakage path**

On a cache hit, `responseCacheCheckHandler` (lines 50, 60) sets:

```js
const {ttl: redisTtl, value: redisValue} = cachedTtlAndValue;
…
{[CACHE_CONTROL_HEADER]: `public, max-age=${redisTtl}`}
```

`redisTtl` is the **remaining** Redis TTL in seconds. If the configured cache TTL is `T` seconds and the response header shows `max-age=R`, the entry was populated `T − R` seconds ago. This is deterministic and requires no privileged access.

**Root cause / failed assumption**

The system assumes that cache entries are keyed by logical resource identity. In reality they are keyed by raw URL string, so the same logical resource has as many independent cache clocks as there are syntactic representations of its identifier.

### Impact Explanation

An unprivileged external attacker can determine, to within one-second precision, when any specific account's staking-reward data was last fetched by any client, for each identifier format independently. This constitutes a **query-pattern / behavioural side-channel**: it reveals whether an account is being actively monitored, at what cadence, and which identifier format the monitoring client uses. Additionally, the cache is fragmented (up to 10× storage overhead per account), and an attacker can force cache misses — and therefore fresh database queries — by always using a format that has not yet been cached, constituting a low-cost amplification vector against the database.

### Likelihood Explanation

No authentication or rate-limiting is required. The attacker needs only an HTTP client and knowledge of the four publicly-documented account identifier formats (all described in the Hedera/Hiero API documentation). The attack is fully repeatable, stateless, and produces a deterministic result on every probe. The `max-age` value is present in every cache-hit response by design, so no timing measurement or statistical analysis is needed — the information is read directly from a response header.

### Recommendation

Normalise the account-identifier path segment to a canonical form (e.g., `shard.realm.num`) **before** computing the cache key, so that all syntactic representations of the same account share a single Redis entry. The planned "request normalizer (9113)" referenced in the comment should be extended to cover path parameters, not only query parameters. Until that work lands, a targeted fix inside `cacheKeyGenerator` can resolve the account-ID path segment to its numeric entity ID and substitute it into the URL before hashing.

### Proof of Concept

```
# Assume cache TTL = 60 s. A legitimate user queries account 1001 using numeric format.
GET /api/v1/accounts/1001/rewards
→ cache miss; entry created at T=0.

# 20 seconds later, attacker probes all formats:
GET /api/v1/accounts/1001/rewards
→ Cache-Control: public, max-age=40   ← hit; cached 20 s ago

GET /api/v1/accounts/0.0.1001/rewards
→ Cache-Control: public, max-age=60   ← miss then fresh entry; no prior query with this format

GET /api/v1/accounts/0x00000000000000000000000000000000000003e9/rewards

→ Cache-Control: public, max-age=60   ← miss; no prior query with this format
