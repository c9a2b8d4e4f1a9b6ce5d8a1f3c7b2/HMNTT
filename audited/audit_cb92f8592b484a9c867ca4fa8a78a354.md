### Title
SQL NULL Handling Bug in `findByEvmAddress` Causes Active Entities with `deleted = NULL` to Be Silently Skipped, Triggering Recoverable Errors on Cache-Cold Lookups

### Summary
The `findByEvmAddress` query in the importer's `EntityRepository` uses `deleted <> true`, which in SQL evaluates `NULL <> true` as `NULL` (falsy), causing entities whose `deleted` column is `NULL` to be excluded from results. This directly contradicts the `notify()` method in `EntityIdServiceImpl`, which explicitly treats `deleted = NULL` as an active (non-deleted) entity and caches it. After a mirror node restart (cold cache), any transaction referencing such an entity by EVM address will fail to resolve it, triggering `handleRecoverableError` and producing incorrect transaction processing.

### Finding Description

**Exact code location:**

`importer/src/main/java/org/hiero/mirror/importer/repository/EntityRepository.java`, line 19:
```java
@Query(value = "select id from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByEvmAddress(byte[] evmAddress);
``` [1](#0-0) 

**Root cause — SQL three-valued logic:**
In PostgreSQL, `NULL <> true` evaluates to `NULL`, not `TRUE`. A `WHERE` clause only passes rows where the predicate is `TRUE`. Therefore, any entity row where `deleted IS NULL` is silently excluded. The correct predicate is `deleted IS NOT TRUE` (used correctly in the `graphql` and `web3` modules). [2](#0-1) [3](#0-2) 

**Inconsistency with `notify()`:**
`EntityIdServiceImpl.notify()` explicitly treats `deleted = NULL` as an active entity and caches it:
```java
if (entity == null || (entity.getDeleted() != null && entity.getDeleted())) {
    return;  // only skip if deleted == true
}
// ... cache the entity
``` [4](#0-3) 

So on a warm cache the entity is found correctly; on a cold cache (post-restart) the DB query is used and the entity is invisible.

**Call chain to error:**
`lookup(ContractID)` → `cacheLookup(...)` → `findByEvmAddress(bytes, throwRecoverableError=true)` → `entityRepository.findByEvmAddress(evmAddress)` returns empty → `Utility.handleRecoverableError(...)`. [5](#0-4) 

**Test gap confirmed:**
The importer's `EntityRepositoryTest.findByEvmAddress` tests `deleted=true` and `deleted=false` but never tests `deleted=NULL`. The `rest-java` module's equivalent test *does* cover `deleted=NULL` and explicitly asserts it returns empty — confirming the behavior is real and reproducible. [6](#0-5) [7](#0-6) 

### Impact Explanation

When `handleRecoverableError` is invoked with the default configuration (`HIERO_MIRROR_IMPORTER_PARSER_HALTONERROR=false`), the importer logs an ERROR and continues, processing the transaction with a missing/empty entity ID. This corrupts the mirror node's view of that transaction — contract call results, token transfers, and contract actions that reference the affected entity will be stored with a null/empty entity reference instead of the correct one.

If `HIERO_MIRROR_IMPORTER_PARSER_HALTONERROR=true`, the importer throws `ParserException` and halts entirely, causing a denial-of-service for all subsequent transaction ingestion until the node is manually restarted. [8](#0-7) 

### Likelihood Explanation

**Precondition:** At least one entity in the database has `deleted = NULL`. This is a realistic steady-state: the `rest-java` integration test explicitly constructs such entities, confirming the schema allows it. Newly ingested entities whose creation transaction has been processed but whose `deleted` column was never explicitly set to `false` satisfy this condition.

**Trigger:** The mirror node restarts (cache is cold). This is a routine operational event (deployments, crashes, rolling restarts).

**Attacker action:** Submit any transaction to the Hedera network that references the known EVM address of such an entity as a `ContractID.EVM_ADDRESS` or `AccountID.ALIAS` (20-byte form). EVM addresses are public. No special privileges are required — any Hedera account can submit a `ContractCall` or `EthereumTransaction` targeting an arbitrary EVM address.

**Repeatability:** The bug fires on every cache-miss lookup for every affected entity, on every restart, indefinitely until the SQL is fixed.

### Recommendation

Replace `deleted <> true` with `deleted is not true` in both affected queries, consistent with the `graphql` and `web3` modules:

```sql
-- importer EntityRepository.java
select id from entity where evm_address = ?1 and deleted is not true
select id from entity where alias = ?1 and deleted is not true

-- rest-java EntityRepository.java (same fix)
select id from entity where evm_address = ?1 and deleted is not true
select id from entity where alias = ?1 and deleted is not true
```

Add a test case in `importer/src/test/java/.../EntityRepositoryTest.java` for `deleted = NULL` that asserts the entity **is** returned (matching the `notify()` semantics), mirroring the coverage already present in the `rest-java` test. [9](#0-8) [10](#0-9) 

### Proof of Concept

1. Insert an entity with `deleted = NULL` and a known EVM address into the mirror node database:
   ```sql
   INSERT INTO entity (id, shard, realm, num, evm_address, type, deleted, timestamp_range)
   VALUES (9999, 0, 0, 9999, '\xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef', 'CONTRACT', NULL, '[1,)');
   ```
2. Start the mirror node importer (cache is cold — no prior `notify()` call for this entity).
3. Submit a Hedera `ContractCall` transaction to the network targeting `ContractID` with `evm_address = 0xdeadbeef...deadbeef`.
4. Observe the importer log: `ERROR Recoverable error. Entity not found for EVM address deadbeefdeadbeefdeadbeefdeadbeefdeadbeef`.
5. Confirm the stored transaction record has a null/empty `contract_id` for the called contract.
6. Now warm the cache by calling `notify()` with the same entity (simulating in-session processing), repeat step 3 — the entity is found correctly, confirming the cache/DB split behavior.

    public static void handleRecoverableError(String message, Object... args) {
        var haltOnError = Boolean.parseBoolean(System.getProperty(HALT_ON_ERROR_PROPERTY));

        if (haltOnError) {
            var formattingTuple = MessageFormatter.arrayFormat(message, args);
            var throwable = formattingTuple.getThrowable();
            var formattedMessage = formattingTuple.getMessage();
            throw new ParserException(formattedMessage, throwable);
        } else {
            log.error(RECOVERABLE_ERROR + message, args);
        }
    }
