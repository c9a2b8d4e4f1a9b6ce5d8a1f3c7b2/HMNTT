### Title
Unauthenticated Parallel Request DoS via Unbounded NFT Allowance Pagination in `getNftAllowances()`

### Summary
The `getNftAllowances()` endpoint in `AllowancesController` is publicly accessible with no authentication, no per-IP rate limiting, and allows up to 100 records per response. An attacker can identify accounts with large NFT allowance sets from public blockchain data and flood the endpoint with parallel requests, exhausting server memory, DB connection pool, and network bandwidth without any server-side throttle to stop them.

### Finding Description

**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, `getNftAllowances()`, lines 57–83.

```java
@GetMapping(value = "/nfts")
NftAllowancesResponse getNftAllowances(
        @PathVariable EntityIdParameter id,
        ...
        @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
        ...)
```

`MAX_LIMIT = 100` and `DEFAULT_LIMIT = "25"` are defined in `Constants.java` (lines 34–35). The `@Max(MAX_LIMIT)` annotation caps a single response at 100 records, but imposes no constraint on request frequency or concurrency.

**No rate limiting in rest-java:** The entire `rest-java/src/main/java/org/hiero/mirror/restjava/config/` directory contains only `LoggingFilter`, `MetricsFilter`, `WebMvcConfiguration`, `RestJavaConfiguration`, `JacksonConfiguration`, and `NetworkProperties` — none of which implement throttling. The `ThrottleConfiguration`/`ThrottleManagerImpl` that exist in the `web3` module are entirely absent from `rest-java`.

The constant `HIGH_VOLUME_THROTTLE` is declared in `Constants.java` (line 19) but is never wired to any enforcement logic for this endpoint.

**No infrastructure-level rate limiting:** The Helm chart for `hedera-mirror-rest-java` (`charts/hedera-mirror-rest-java/values.yaml`, lines 158–163) configures only two Traefik middlewares: `circuitBreaker` (triggers on >10% network errors or >25% 5xx) and `retry` (3 attempts). There is no `rateLimit` middleware. The circuit breaker only activates after errors accumulate — it does not protect against a flood of valid, successful requests.

**Exploit flow:**
1. Attacker queries the public Hedera ledger or mirror node to identify an account ID with thousands of NFT allowances (fully public data).
2. Attacker sends `N` parallel `GET /api/v1/accounts/{id}/allowances/nfts?limit=100` requests from multiple IPs or a single IP.
3. Each request causes `NftAllowanceServiceImpl.getNftAllowances()` → `NftAllowanceRepository.findAll()` to execute a DB query returning up to 100 rows, serialize them to JSON, and stream the response.
4. With no rate limit, the server processes all concurrent requests, consuming DB connections, heap memory for result sets, and outbound bandwidth simultaneously.
5. Repeated in a loop, this degrades or crashes the service for all users.

### Impact Explanation
Each response can carry up to 100 NFT allowance objects, each containing multiple string fields (owner, spender, token_id, timestamps). At scale, parallel requests create: (a) DB connection pool exhaustion, (b) heap pressure from concurrent result-set deserialization, (c) network bandwidth saturation. The `ShallowEtagHeaderFilter` registered in `RestJavaConfiguration` (line 42) buffers the entire response body in memory to compute ETags, amplifying per-request memory cost. This is a realistic denial-of-service against a public, unauthenticated read endpoint with no compensating control.

### Likelihood Explanation
No special privilege is required. The target account ID is discoverable from public blockchain explorers. A single attacker with a modest botnet or even a single machine with async HTTP clients (e.g., `curl`, `ab`, `wrk`, Python `aiohttp`) can sustain thousands of requests per second. The circuit breaker only fires on error responses — successful 200 responses from a valid account ID will never trigger it, making the attack invisible to the existing infrastructure guard.

### Recommendation
1. **Add per-IP rate limiting** in the Traefik middleware for the `rest-java` ingress (add a `rateLimit` entry alongside `circuitBreaker` in `charts/hedera-mirror-rest-java/values.yaml`).
2. **Implement application-level throttling** in `rest-java` mirroring the `ThrottleManagerImpl` pattern from the `web3` module, keyed on client IP extracted from `X-Forwarded-For`.
3. **Disable `ShallowEtagHeaderFilter`** for large-collection endpoints, or replace it with a weak ETag strategy that does not buffer the full response body.
4. **Wire `HIGH_VOLUME_THROTTLE`** (already declared in `Constants.java`) to an actual enforcement interceptor for endpoints that return collections.

### Proof of Concept

```bash
# Step 1: Identify a high-allowance account (public data)
ACCOUNT="0.0.12345"   # replace with a real account having many NFT allowances

# Step 2: Flood with parallel requests (no auth needed)
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ACCOUNT}/allowances/nfts?limit=100" \
    -o /dev/null &
done
wait

# Step 3: Observe server memory/CPU spike and degraded response times for all users.
# The circuit breaker will NOT trigger because all responses are HTTP 200.

    @GetMapping(value = "/nfts")
    NftAllowancesResponse getNftAllowances(
            @PathVariable EntityIdParameter id,
            @RequestParam(name = ACCOUNT_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] accountIds,
            @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
            @RequestParam(defaultValue = "asc") Sort.Direction order,
            @RequestParam(defaultValue = "true") boolean owner,
            @RequestParam(name = TOKEN_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] tokenIds) {
        var field = owner ? NFT_ALLOWANCE.SPENDER : NFT_ALLOWANCE.OWNER;
        var request = NftAllowanceRequest.builder()
                .accountId(id)
                .isOwner(owner)
                .limit(limit)
                .order(order)
                .ownerOrSpenderIds(new Bound(accountIds, true, ACCOUNT_ID, field))
                .tokenIds(new Bound(tokenIds, false, TOKEN_ID, NFT_ALLOWANCE.TOKEN_ID))
                .build();

        var serviceResponse = service.getNftAllowances(request);
        var allowances = nftAllowanceMapper.map(serviceResponse);

        var sort = Sort.by(order, ACCOUNT_ID, TOKEN_ID);
        var pageable = PageRequest.of(0, limit, sort);
        var links = linkFactory.create(allowances, pageable, EXTRACTORS.get(owner));

        return new NftAllowancesResponse().allowances(allowances).links(links);
    }
```
