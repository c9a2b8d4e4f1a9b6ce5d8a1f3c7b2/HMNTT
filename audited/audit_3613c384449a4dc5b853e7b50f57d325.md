### Title
Unauthenticated `getNodes()` gRPC Endpoint Enables Bounded Elastic Scheduler Exhaustion via Unbounded Connection Flooding

### Summary
The `getNodes()` endpoint in `NetworkController` accepts unlimited unauthenticated connections with no rate limiting, no per-IP connection cap, and no global connection ceiling. Each subscription drives blocking database page fetches scheduled on Reactor's JVM-global `Schedulers.boundedElastic()` singleton, which is shared with all topic-message listeners and retrievers. An attacker opening enough concurrent connections can saturate that shared thread pool, starving topic-message processing across targeted mirror node replicas.

### Finding Description

**Exact code path:**

`NetworkController.getNodes()` (lines 33–43) accepts any unauthenticated `AddressBookQuery` and immediately subscribes to `NetworkServiceImpl.getNodes()` with no throttle guard, no IP check, and no connection counter. [1](#0-0) 

`NetworkServiceImpl.getNodes()` (lines 68–76) builds a paged `Flux` that re-executes a blocking DB call (`transactionOperations.execute()`) on every page, with the inter-page delay scheduled via `Schedulers.boundedElastic()`. When the delay fires, the next `page()` call runs on the same bounded-elastic thread, holding it for the duration of the synchronous JDBC query. [2](#0-1) [3](#0-2) 

`Schedulers.boundedElastic()` is a JVM-global singleton. The same pool is used by `PollingTopicListener`, `SharedPollingTopicListener` (via `subscribeOn` and `withScheduler`), `PollingTopicMessageRetriever`, and `SharedTopicListener.publishOn`. [4](#0-3) [5](#0-4) [6](#0-5) 

**The only connection-level guard** is `maxConcurrentCallsPerConnection = 5`, which limits calls *per TCP connection* but places no ceiling on the number of connections or on total concurrent calls across all connections. [7](#0-6) [8](#0-7) 

**No rate-limiting exists in the grpc module.** The `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` stack lives exclusively in the `web3` module and is never applied to any gRPC endpoint. [9](#0-8) 

**Root cause:** The design assumes that `maxConcurrentCallsPerConnection` is sufficient to bound total scheduler load, but it only limits calls per connection. An attacker opens arbitrarily many connections, each contributing up to 5 concurrent blocking page-fetch tasks to the shared bounded-elastic pool.

### Impact Explanation

Reactor's `boundedElastic` pool defaults to `10 × availableProcessors` threads (e.g., 40 threads on a 4-core pod). Each concurrent `getNodes()` page fetch holds one thread for the duration of a synchronous JDBC query. With N connections × 5 calls each, once N exceeds ~8–10 (on a 4-core pod), the pool is saturated. All subsequent `boundedElastic` tasks — including the periodic polls in `PollingTopicListener` and `PollingTopicMessageRetriever` and the `publishOn` dispatch in `SharedTopicListener` — queue behind the flood. Topic-message delivery to all subscribers on that replica halts. Targeting ≥30% of deployed replicas simultaneously (trivially achievable from a single machine with many TCP connections) meets the stated ≥30% network-processing-node shutdown threshold.

### Likelihood Explanation

No authentication, no API key, no IP allowlist, and no rate limit are required. A single attacker machine can open thousands of TCP connections to port 5600 and issue 5 concurrent `getNodes()` calls on each. Because each call completes in a few seconds (small address book, pageSize=10, pageDelay=250ms), the attacker simply reconnects in a tight loop to maintain sustained pressure. This requires only a standard gRPC client library and a loop — no exploit tooling, no credentials, no brute force. The attack is fully repeatable and can be scripted in under 20 lines of Go or Python.

### Recommendation

1. **Add a global concurrent-call limit** to `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge` / `maxConnections` or a custom `ServerInterceptor` that tracks and rejects calls beyond a configurable global ceiling.
2. **Add per-IP rate limiting** via a gRPC `ServerInterceptor` (analogous to the `ThrottleManagerImpl` in `web3`) that rejects `getNodes()` calls exceeding N requests/second per remote address.
3. **Isolate the `getNodes()` scheduler** — replace the shared `Schedulers.boundedElastic()` in `NetworkServiceImpl` with a dedicated, capacity-bounded `Schedulers.newBoundedElastic(...)` instance so that address-book floods cannot starve topic-message processing.
4. **Add a connection-count limit** in `NettyProperties` (e.g., `maxConnections`) and wire it into `GrpcConfiguration` alongside `maxConcurrentCallsPerConnection`.

### Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor
# proto stubs for com.hedera.mirror.api.proto.NetworkService

TARGET = "mirror-node-grpc-host:5600"
NUM_CONNECTIONS = 500   # 500 connections × 5 calls = 2500 concurrent page fetches

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = NetworkServiceStub(channel)
    query = AddressBookQuery(file_id=FileID(file_num=102), limit=0)  # limit=0 = no limit
    while True:
        try:
            # Blocking streaming call; completes in ~2-3s, then reconnects
            list(stub.getNodes(query))
        except Exception:
            pass

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS * 5) as pool:
    for _ in range(NUM_CONNECTIONS * 5):
        pool.submit(flood)
# Result: bounded-elastic pool saturated; topic-message polling stalls on targeted replicas
```

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
    }

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
