### Title
Unbounded HTTP Request Body Enables OOM-Based Denial of Service on `/api/v1/contracts/call`

### Summary
The `ContractController.call()` endpoint accepts an unbounded JSON request body via `@RequestBody`. Spring MVC fully buffers the entire HTTP body into memory before any validation is applied. No server-level body size limit is configured anywhere in the web3 module, meaning an unauthenticated attacker can send a multi-gigabyte POST body to exhaust JVM heap and crash the service.

### Finding Description
**Exact code path:**

`ContractController.java` line 38:
```java
ContractCallResponse call(@RequestBody @Valid ContractCallRequest request, HttpServletResponse response)
```
Spring's `HttpMessageConverter` (Jackson) reads the full `InputStream` from the servlet container into a `ContractCallRequest` object before `@Valid` is evaluated. Only after full deserialization does the `@Hex(maxLength = 1048576L)` constraint on the `data` field fire.

**Root cause — failed assumption:** The `@Hex` annotation in `Hex.java` (line 27) sets `maxLength = 1048576L` (1 MiB in hex characters), but this is a *post-parse* Bean Validation constraint. Jackson must fully read and parse the JSON string into a Java `String` before Hibernate Validator can check its length. A 2 GB `"data"` value in JSON is fully allocated on the heap before the constraint is ever evaluated.

**No server-level limit exists:** The only Spring Boot configuration file in `web3/src/main/resources/` is `banner.txt` — there is no `application.yml` or `application.properties`. No `server.tomcat.max-swallow-size`, `server.tomcat.max-http-form-post-size`, or any equivalent Netty/Undertow limit is set. Spring Boot's default `max-http-form-post-size` (2 MB) applies only to `application/x-www-form-urlencoded` form data, **not** to `application/json` `@RequestBody` endpoints.

**`LoggingFilter` does not help:** `LoggingFilter.java` line 44 wraps the request in `ContentCachingRequestWrapper(request, web3Properties.getMaxPayloadLogSize() * 10)` — this only caps the *logging cache*, not the actual read from the socket. The full body is still consumed by Jackson upstream.

**Throttle/filter does not help:** `RequestProperties` and `RequestFilter` operate on the already-deserialized `ContractCallRequest` object — they cannot intercept an oversized raw body.

### Impact Explanation
A single malicious POST with a JSON body of several gigabytes (e.g., `{"data":"<2GB hex string>"}`) causes the JVM to allocate a `String` of that size on the heap. This exhausts available heap memory, triggering `OutOfMemoryError` and crashing the web3 service. Because the endpoint is publicly accessible with no authentication, this is a complete, unauthenticated denial-of-service against the entire node's web3 API. Recovery requires a process restart.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. Any attacker with network access to the service can trigger this with a single HTTP request using standard tools (`curl`, `python requests`). The attack is trivially repeatable and can be automated. The absence of any configuration file for the web3 module makes it highly likely this is an oversight rather than an intentional reliance on an upstream proxy, especially since no proxy-level size enforcement is documented or enforced in the codebase.

### Recommendation
Add a server-level body size limit in `web3/src/main/resources/application.yml`:
```yaml
server:
  tomcat:
    max-swallow-size: 1MB
    max-http-form-post-size: 1MB
spring:
  servlet:
    multipart:
      max-request-size: 1MB
```
For `@RequestBody` JSON specifically, register a `CommonsRequestLoggingFilter` or a custom `OncePerRequestFilter` that checks `Content-Length` before delegating to the filter chain, and configure Jackson's `StreamReadConstraints` with a max string length:
```java
StreamReadConstraints.builder().maxStringLength(1_048_576).build()
```
This enforces the limit at the parser level, before heap allocation, matching the intent of the existing `@Hex(maxLength = 1048576L)` constraint.

### Proof of Concept
```bash
# Generate a ~2GB JSON body with an oversized hex data field
python3 -c "
import sys
sys.stdout.write('{\"data\":\"0x')
sys.stdout.flush()
# Write 2GB of hex chars
for _ in range(2 * 1024 * 1024 * 1024 // 64):
    sys.stdout.write('a' * 64)
sys.stdout.write('\"}')
" | curl -X POST http://<host>:<port>/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     --data-binary @-
```
**Expected result:** JVM heap exhaustion, `OutOfMemoryError`, service crash or severe degradation. No authentication or special headers required.
