### Title
`failed` Field in Opcodes Response Derived from Re-execution Result, Not Original On-Chain Consensus Status

### Summary
`OpcodeServiceImpl.buildOpcodesResponse()` sets the `failed` field exclusively from the EVM re-execution result (`txnResult.isSuccessful()`), never consulting the original `contractResult.transactionResult` stored in the database. For transactions that failed on-chain due to Hedera-specific consensus-layer errors (e.g., `CONSENSUS_GAS_EXHAUSTED`, `INVALID_ACCOUNT_ID`) rather than EVM-level reverts, the EVM re-execution can succeed, causing the endpoint to return `failed: false` for a transaction that definitively failed in Hashgraph history. Any unprivileged user can trigger this by querying the public opcodes endpoint for such a transaction.

### Finding Description

**Exact code path:**

`OpcodeServiceImpl.buildOpcodesResponse()` at line 146:
```java
.failed(txnResult == null || !txnResult.isSuccessful())
```
`txnResult` is the `EvmTransactionResult` from `contractDebugService.processOpcodeCall()` — the re-execution result. `EvmTransactionResult.isSuccessful()` returns `true` only when `responseCodeEnum == ResponseCodeEnum.SUCCESS` (re-execution success).

The original on-chain status is stored in `ContractResult.transactionResult` (an integer proto status code, e.g., `CONSENSUS_GAS_EXHAUSTED_VALUE`). This field is fetched at line 154–156 of `OpcodeServiceImpl` to build `ContractDebugParameters`, but is **never read** when constructing the `OpcodesResponse`. There is no code anywhere in `buildOpcodesResponse()` or `buildCallServiceParameters()` that cross-checks the re-execution outcome against `contractResult.transactionResult`.

**Root cause — failed assumption:**
The design assumes the EVM re-execution faithfully reproduces the original consensus outcome. This holds for EVM-level reverts (`CONTRACT_REVERT_EXECUTED`) but breaks for Hedera-specific consensus-layer failures that are not modeled inside the EVM (e.g., `CONSENSUS_GAS_EXHAUSTED`, `INVALID_ACCOUNT_ID`, `INSUFFICIENT_GAS` at the throttle level). The EVM re-execution runs with the original `gasLimit` from `contractResult.getGasLimit()` and does not enforce Hedera network-level throttles, so it can return `SUCCESS` where the original transaction returned a non-EVM failure code.

**Why existing checks are insufficient:**
`ContractDebugService.validateResult()` (lines 62–72) catches and silently swallows `MirrorEvmTransactionException`, ensuring the re-execution result is always returned regardless of EVM-level failure. However, this does not address the inverse case: when re-execution *succeeds* but the original transaction *failed* at the consensus layer. There is no guard that reads `contractResult.transactionResult` and overrides or validates the re-execution `failed` field.

`TransactionExecutionService.handleFailedResult()` (lines 147–158) correctly returns a failed `EvmTransactionResult` in the opcode-trace path when the EVM itself fails — but this path is only reached when the EVM re-execution also fails. When the EVM re-execution succeeds (consensus-layer-failed transaction), `buildSuccessResult()` is called and returns `ResponseCodeEnum.SUCCESS`, which propagates to `failed: false`.

**Exploit flow:**
1. Attacker identifies a historical transaction that failed on-chain with a Hedera-specific code (e.g., `CONSENSUS_GAS_EXHAUSTED`). Such transactions have a `ContractResult` record (populated by `ContractResultServiceImpl.addDefaultEthereumTransactionContractResult()`) and a `ContractTransactionHash` record, making them queryable.
2. Attacker sends `GET /api/v1/contracts/results/{hash}/opcodes` with `Accept-Encoding: gzip` (the only required header).
3. `OpcodeServiceImpl.processOpcodeCall()` fetches the `ContractResult`, builds `ContractDebugParameters` using the original gas limit, and re-executes via the EVM at the historical block state.
4. The EVM re-execution succeeds (no consensus-layer throttle in the EVM path).
5. `buildOpcodesResponse()` sets `failed: false` and returns HTTP 200 with a response that contradicts the Hashgraph record.

### Impact Explanation
The opcodes endpoint is the only Mirror Node API that provides EVM execution traces for historical transactions. Consumers (block explorers, auditing tools, forensic debuggers) rely on the `failed` field to determine whether a historical transaction succeeded. A response of `failed: false` for a transaction that definitively failed on-chain constitutes a false representation of Hashgraph history. This can be used to deceive users or automated systems into believing a transaction succeeded when it did not — for example, falsely confirming a token transfer or contract state change that never occurred. The impact is bounded to read-only misrepresentation (no state mutation), but it directly undermines the integrity guarantee of the historical debug endpoint.

### Likelihood Explanation
The precondition — a transaction that failed at the Hedera consensus layer rather than the EVM layer — is common in production. `CONSENSUS_GAS_EXHAUSTED` and similar codes occur regularly under network load. The endpoint is publicly accessible with no authentication; the only barrier is the `Accept-Encoding: gzip` header check, which is trivially satisfied. The divergence is deterministic and repeatable for any qualifying transaction. No special privileges, keys, or network access beyond standard HTTP are required.

### Recommendation
In `buildOpcodesResponse()`, override the `failed` field using the original on-chain status from `contractResult.transactionResult` rather than (or in addition to) the re-execution result. Pass the fetched `ContractResult` through to `buildOpcodesResponse()` and set:
```java
.failed(isOriginalTransactionFailed(contractResult.getTransactionResult()))
```
where `isOriginalTransactionFailed` checks whether the stored proto status code is a non-success code (mirroring the logic in `ContractResultDetailsViewModel._SUCCESS_PROTO_IDS`). The re-execution result can still be used for `gas`, `returnValue`, and `opcodes`, but `failed` must reflect the immutable consensus record.

### Proof of Concept
1. Identify a transaction hash `H` in the mirror node database where `contract_result.transaction_result = 10` (`CONSENSUS_GAS_EXHAUSTED`) or any non-EVM Hedera failure code.
2. Send:
   ```
   GET /api/v1/contracts/results/H/opcodes
   Accept-Encoding: gzip
   ```
3. Observe the response contains `"failed": false` and `"opcodes": [...]` with a successful execution trace.
4. Cross-reference against `GET /api/v1/contracts/results/H` which returns `"result": "CONSENSUS_GAS_EXHAUSTED"` and `"status": "0x0"` — confirming the on-chain failure.
5. The two responses contradict each other: the opcodes endpoint reports success; the contract results endpoint reports failure. The `failed` field in the opcodes response is derived solely from the re-execution at [1](#0-0) , never from the stored `contractResult.transactionResult` at [2](#0-1) .
