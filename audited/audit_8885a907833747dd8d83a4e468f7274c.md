### Title
Per-Instance Local Throttle Bucket Enables Multi-Pod CPU Exhaustion via Opcodes Endpoint

### Summary
The `getContractOpcodes()` endpoint in `OpcodesController` enforces rate limiting via a local in-memory `Bucket4j` bucket that is scoped per JVM instance, not distributed across the cluster. In a horizontally scaled deployment, each pod independently allows `opcodeRequestsPerSecond` (default: 1) requests per second, so an unprivileged attacker can send 1 request/second to each pod simultaneously, triggering full EVM re-execution of a historically gas-heavy transaction (up to `maxGasLimit=15,000,000`) on every instance. No per-IP rate limiting exists at any layer for this endpoint.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (line 61) calls `throttleManager.throttleOpcodeRequest()`: [1](#0-0) 

`ThrottleManagerImpl.throttleOpcodeRequest()` checks a single local bucket: [2](#0-1) 

The `opcodeRateLimitBucket` bean is constructed as a **local in-memory** `Bucket4j` bucket — `Bucket.builder().addLimit(limit).build()` — with no distributed backend (no Redis, no Hazelcast): [3](#0-2) 

The default rate is 1 RPS: [4](#0-3) 

**Root cause — failed assumption:** The design assumes the `opcodeRateLimitBucket` enforces a global cluster-wide limit. It does not. Each pod has its own independent in-memory bucket. With N pods, the effective global rate is N × 1 = N requests/second, each triggering a full EVM re-execution.

**No per-IP protection at any layer:** The web3 Helm chart middleware is only `retry` (3 attempts): [5](#0-4) 

Furthermore, `global.middleware` defaults to `false`, meaning even this retry middleware is not applied: [6](#0-5) 

Compare to Rosetta, which has `inFlightReq` (5 per IP) and `rateLimit` (10 per host) at the Traefik layer — the web3 opcodes endpoint has neither.

The GCP gateway rule for the opcodes path is **commented out**, meaning it bypasses GCP-level backend policy (`maxRatePerEndpoint: 250`): [7](#0-6) 

HPA is disabled by default (`hpa.enabled: false`), so the pod count is fixed: [8](#0-7) 

### Impact Explanation

With N pods (up to `maxReplicas: 12`), an attacker can sustain N concurrent full EVM re-executions of a 15M-gas transaction. Each re-execution is CPU-bound (Besu EVM, opcode tracing with stack/memory/storage). At 12 pods × 1 RPS × up to 10 seconds per execution (`requestTimeout: 10000`), up to 12 simultaneous heavy EVM executions run continuously. Pod CPU limit is 2 cores each. This can saturate CPU across all web3 instances, degrading or halting contract call processing for all users — consistent with the ≥30% network processing node impact threshold. No authentication, no cost, no privileged access required.

### Likelihood Explanation

The attack requires: (1) the endpoint is enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`); (2) knowledge of a gas-heavy historical transaction hash (publicly visible on-chain); (3) sending HTTP GET requests with `Accept-Encoding: gzip`. All three conditions are trivially met by any external user. The attacker does not need to know individual pod IPs — a load balancer distributes requests across pods, and the per-pod local bucket means each pod processes its own 1 RPS independently. The attack is fully repeatable and requires no special tooling beyond `curl` or a simple script.

### Recommendation

1. **Replace the local `Bucket4j` bucket with a distributed implementation** (e.g., `bucket4j-redis` or `bucket4j-hazelcast`) so the `opcodeRateLimitBucket` enforces a true cluster-wide rate limit.
2. **Add per-IP rate limiting** at the Traefik middleware layer for the opcodes path (analogous to Rosetta's `inFlightReq` + `rateLimit` middleware).
3. **Uncomment and enable the GCP gateway rule** for the opcodes path so `maxRatePerEndpoint` applies.
4. **Add concurrency limiting** (e.g., a semaphore or `inFlightReq`) to cap simultaneous EVM re-executions cluster-wide, independent of RPS.

### Proof of Concept

**Preconditions:**
- `hiero.mirror.web3.opcode.tracer.enabled=true`
- Deployment has N ≥ 2 pods behind a load balancer
- Attacker identifies a gas-heavy transaction hash `TX_HASH` from public chain data

**Steps:**
```bash
# Send 1 request/second continuously; load balancer distributes across pods
# Each pod independently passes its local 1 RPS bucket check
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/<TX_HASH>/opcodes?stack=true&memory=true&storage=true" \
    -o /dev/null &
  sleep 0.1  # faster than 1/s to ensure each pod gets hit
done
```

**Result:** Each pod independently accepts 1 request/second (local bucket), triggering full EVM re-execution of the 15M-gas transaction on every pod simultaneously. CPU across all web3 instances saturates, degrading contract call processing for all users. The attacker receives no throttle rejection as long as requests are distributed across pods.
