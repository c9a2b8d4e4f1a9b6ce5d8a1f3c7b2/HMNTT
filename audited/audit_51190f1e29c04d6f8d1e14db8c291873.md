### Title
Cache Key Based on Raw `req.originalUrl` Bypasses Response Cache via Parameter Order Manipulation

### Summary
The `cacheKeyGenerator` in `responseCacheHandler.js` derives the Redis cache key exclusively from `req.originalUrl` — the raw, client-supplied URL — rather than a normalized form. Because `normalizeRequestQueryParams()` is explicitly not yet integrated into cache key generation (noted as future work in a TODO comment), an unprivileged attacker can trivially bypass the cache for any endpoint, including staking rewards, by reordering query parameters in each request, forcing every request to hit the database directly.

### Finding Description

**Cache key generation** (`rest/middleware/responseCacheHandler.js`, lines 151–153): [1](#0-0) 

The key is `MD5(req.originalUrl)`, where `req.originalUrl` is the verbatim URL string as received from the client, including the exact parameter order. The comment immediately above this function (lines 147–149) explicitly acknowledges the normalizer is not yet used: [2](#0-1) 

**Normalizer is not wired into the pipeline.** A `grep` for `normalizeRequestQueryParams` across the entire codebase shows it is only referenced in its own implementation file and its unit test — it is never called from `server.js`, any route, or any middleware in the live request path: [3](#0-2) 

**Middleware order** (`rest/server.js`, lines 94–140): `responseCacheCheckHandler` runs before route handlers and uses `req.originalUrl` to look up the cache key. `responseCacheUpdateHandler` runs after route handlers and stores the response under the same key. Both sides of the cache use the raw URL: [4](#0-3) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts/0.0.12345/rewards?limit=25&order=desc` → cache miss → DB query → response cached under `MD5("/api/v1/accounts/0.0.12345/rewards?limit=25&order=desc")`.
2. Attacker sends `GET /api/v1/accounts/0.0.12345/rewards?order=desc&limit=25` → different `req.originalUrl` → different MD5 → cache miss → another DB query → cached under a different key.
3. With N parameters, there are N! distinct orderings, each producing a unique cache key, each forcing a fresh DB query.
4. No rate limiting exists in the REST middleware layer: [5](#0-4) 

### Impact Explanation
Every unique parameter ordering is treated as a distinct cache entry. For the staking rewards endpoint (`/api/v1/accounts/{id}/rewards`), an attacker can enumerate reward data for arbitrary accounts at high frequency while guaranteeing every request bypasses the cache and hits PostgreSQL directly. This exhausts DB connection pool resources and degrades service for all users (availability impact). Additionally, the cache provides no throttling benefit for reward enumeration, enabling real-time scraping of reward data across all accounts with no caching friction.

### Likelihood Explanation
No privileges are required. The attack requires only the ability to send HTTP GET requests with reordered query parameters — trivially scriptable. There is no rate limiting in the application layer. The parameter set for the rewards endpoint is small (e.g., `limit`, `order`, `timestamp`), making the full permutation space easy to enumerate in a loop. The vulnerability is repeatable indefinitely.

### Recommendation
Integrate `normalizeRequestQueryParams()` into `cacheKeyGenerator` so the cache key is derived from the canonical, normalized URL rather than `req.originalUrl`. The normalizer already exists and produces a deterministic, parameter-order-independent URL string:

```js
// rest/middleware/responseCacheHandler.js
import {normalizeRequestQueryParams} from './requestNormalizer.js';

const cacheKeyGenerator = (req) => {
  const normalizedUrl = normalizeRequestQueryParams(
    req.openapi?.openApiRoute,
    req.path,
    req.query
  );
  return crypto.createHash('md5').update(normalizedUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

This collapses all semantically equivalent requests to a single cache key regardless of parameter order, closing the bypass. The TODO comment at line 147–149 already anticipates this fix. [6](#0-5) 

### Proof of Concept

```bash
# Warm the cache with canonical order
curl "http://localhost:5551/api/v1/accounts/0.0.12345/rewards?limit=25&order=desc"
# → DB hit, response cached

# Bypass cache with reordered parameters (different MD5 key)
curl "http://localhost:5551/api/v1/accounts/0.0.12345/rewards?order=desc&limit=25"
# → Cache MISS, DB hit again

# Script to enumerate rewards for many accounts, always bypassing cache
for id in $(seq 1 10000); do
  curl -s "http://localhost:5551/api/v1/accounts/0.0.${id}/rewards?order=desc&limit=25" &
done
# Each request hits the DB directly; cache provides zero protection
```

Confirm cache misses by observing Redis key count grows with each unique ordering, and DB query logs show a query for every request variant.

```javascript
const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
```
