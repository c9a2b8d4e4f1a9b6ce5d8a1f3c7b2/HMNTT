### Title
Unauthenticated `from` Field Allows Trivial Bypass of `FilterField.FROM` + `ActionType.REJECT` Throttle Filters

### Summary
The `throttle()` method in `ThrottleManagerImpl` evaluates `RequestProperties` filters against the `from` field of a `ContractCallRequest`, but the `from` field is a plain user-supplied string with no cryptographic authentication. Any caller can supply an arbitrary EVM address as `from`, allowing them to trivially evade any address-based `REJECT` filter by simply using a different address or omitting the field entirely.

### Finding Description
**Code path:**

- `ThrottleManagerImpl.java` lines 44–48: iterates over configured `RequestProperties` and calls `requestFilter.test(request)` then `action(requestFilter, request)`.
- `RequestFilter.java` lines 31–35: the `test()` method extracts the field value via `field.getExtractor().apply(request)`, where `FilterField.FROM` maps to `ContractCallRequest::getFrom`.
- `ContractCallRequest.java` line 34: `from` is declared as `private String from;` — a plain, user-supplied JSON field validated only for hex format (`@Hex(minLength=40, maxLength=40)`), with no ownership proof or authentication.
- `RequestFilter.java` line 43: `FROM(ContractCallRequest::getFrom)` — the extractor simply reads the raw user-supplied string.

**Root cause:** The filter assumes the `from` field identifies the actual caller. It does not. The `from` field in an EVM simulation call (`eth_call` equivalent) is a simulation parameter freely set by the client. There is no signature, session binding, IP binding, or any other mechanism tying the `from` value to the actual requester.

**Exploit flow:**
1. Operator configures a `RequestProperties` with `FilterField.FROM`, `FilterType.EQUALS`, expression = `"aabbccdd..."` (blocked address), `ActionType.REJECT`.
2. Attacker sends a request with `from = "aabbccdd..."` → filter matches → `ThrottleException("Invalid request")` thrown → blocked.
3. Attacker resends the identical request with `from = "1111111111111111111111111111111111111111"` (any other valid 40-hex-char address) → filter does not match → request proceeds normally.
4. Alternatively, attacker omits `from` entirely (valid when `value == 0`, per `hasFrom()` at line 49–51 of `ContractCallRequest.java`) → `getFrom()` returns `null` → `String.valueOf(null)` = `"null"` → does not match the blocked address expression → request proceeds.

**Why existing checks fail:** The `@Hex` annotation only validates format (valid hex, correct length). There is no authentication layer, no IP-to-address binding, and no session credential tied to the `from` value. The `RequestFilter.test()` method blindly trusts the client-supplied string.

### Impact Explanation
Any operator-configured address-based rejection policy (`FilterField.FROM` + `ActionType.REJECT`) is completely ineffective as an access control or abuse-prevention mechanism. An attacker who is "blocked" by address can immediately resume submitting requests by rotating to any other valid EVM address. This renders the entire `FROM`-based filtering feature useless for security enforcement, potentially allowing continued abuse of gas-intensive simulation calls, denial-of-service amplification, or evasion of per-address rate limits.

### Likelihood Explanation
Exploitation requires zero privilege, zero special tooling, and zero knowledge beyond the public API contract. Any HTTP client can set the `from` field to an arbitrary value. The bypass is deterministic and 100% repeatable. The only precondition is knowing (or guessing) that a `FROM`-based filter is active, but even without that knowledge, rotating addresses is a standard practice.

### Recommendation
1. **Document the limitation clearly**: If `FilterField.FROM` is intended only for simulation-context filtering (not caller identity), document that it must not be used for security enforcement.
2. **Bind filtering to authenticated identity**: For actual caller blocking, filter on a verified attribute such as the authenticated IP address (add `FilterField.IP` using the real remote address from the HTTP request, not a user-supplied field).
3. **Remove `FROM` from security-sensitive filter fields** or add a warning in configuration validation that `FROM`-based `REJECT` rules are bypassable.
4. If caller identity is required, implement API key authentication or mTLS and bind throttle rules to the authenticated principal, not to a user-supplied simulation parameter.

```
# Step 1: Send request with "blocked" address — gets rejected
POST /api/v1/contracts/call
{"from":"0xaabbccddaabbccddaabbccddaabbccddaabbccdd","to":"0x...","gas":21000}
→ HTTP 429 ThrottleException: Invalid request

# Step 2: Send identical request with any other address — succeeds
POST /api/v1/contracts/call
{"from":"0x1111111111111111111111111111111111111111","to":"0x...","gas":21000}
→ HTTP 200 OK  (filter does not match, REJECT never triggered)

# Step 3: Omit from entirely (value=0) — also succeeds
POST /api/v1/contracts/call
{"to":"0x...","gas":21000}
→ HTTP 200 OK  (from=null, String.valueOf(null)="null", no match)
```
