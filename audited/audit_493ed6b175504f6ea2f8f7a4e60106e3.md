### Title
`metricsMiddleware` Relies Solely on `'finish'` Event — Client-Disconnect Requests Permanently Leak `inFlightCounter` and Are Invisible to All Exported Metrics

### Summary
In `rest/middleware/metricsHandler.js`, `metricsMiddleware()` increments `inFlightCounter` synchronously on every incoming request but places **all** metric recording — including the corresponding decrement — exclusively inside a `res.on('finish', ...)` handler. In Node.js, the `'finish'` event is only emitted after `res.end()` successfully flushes data to the OS kernel buffer. When a client closes the TCP connection before the server sends the response, the `'close'` event fires instead and `'finish'` never fires. The result is a permanent `inFlightCounter` leak and complete invisibility of those requests across every exported Prometheus metric.

### Finding Description

**Exact code path:**

`rest/middleware/metricsHandler.js`, function `metricsMiddleware()`:

- Line 169 — `inFlightCounter.add(1)` executes synchronously for every request, with no guard.
- Lines 185–211 — the sole `res.on('finish', ...)` handler is the **only** place where `inFlightCounter.add(-1)` (line 186), `allRequestCounter` (line 195), `allClientErrorCounter` / `allErrorCounter` (lines 199–200), `allServerErrorCounter` (lines 202–203), and all per-route histograms (lines 207–210) are ever touched.
- There is **no** `res.on('close', ...)` or `req.on('close', ...)` handler anywhere in the REST middleware tree (confirmed by grep across `rest/**/*.js`).

**Root cause / failed assumption:**

The code assumes that every request that enters the middleware will eventually emit `'finish'`. This assumption is false. Node.js HTTP semantics guarantee `'finish'` only when the response is fully written. If the client drops the TCP connection before `res.end()` completes, Node.js emits `'close'` on the response object and `'finish'` is never emitted. No fallback path exists.

**Exploit flow:**

1. Attacker opens a TCP connection and sends a syntactically valid HTTP request (e.g., `GET /api/v1/transactions HTTP/1.1\r\nHost: target\r\n\r\n`).
2. Immediately after sending the request headers, the attacker resets or closes the TCP connection (RST or FIN) before the server has written the response.
3. Node.js fires `'close'` on the response; `'finish'` is never emitted.
4. `inFlightCounter` was already incremented at line 169 and is now permanently +1 per iteration.
5. `allRequestCounter`, `allClientErrorCounter`, `allErrorCounter`, and all per-route instruments remain unchanged — the request is entirely absent from exported metrics.

**Why existing checks are insufficient:**

There are none. The middleware contains zero handling for the `'close'` event. The `res.write` / `res.end` monkey-patches (lines 176–183) only accumulate `responseSize`; they do not trigger metric recording and do not compensate for a missing `'finish'`.

### Impact Explanation

1. **`inFlightCounter` unbounded inflation** — repeated connect-and-abort cycles drive `api_all_request_in_processing_total` monotonically upward. Alerting rules or dashboards that threshold on in-flight requests will fire false positives, potentially masking or drowning out real overload signals.
2. **Complete metric blindspot** — every aborted request is invisible to `api_all_request`, `api_all_errors`, `api_all_client_error`, `api_request`, `api_request_duration_milliseconds`, and `api_response_size_bytes`. Operators cannot detect, rate-limit, or investigate a flood of aborted requests through the metrics plane.
3. **Activity concealment** — an attacker performing reconnaissance or probing endpoints can abort responses before they are counted, leaving no trace in the exported Prometheus records that are forwarded to mirror-node monitoring infrastructure.

### Likelihood Explanation

No authentication, special headers, or privileged network position is required. Any HTTP client capable of sending a request and immediately closing the socket can trigger this. It is reproducible with a single `curl` invocation or a trivial script. The behavior is deterministic on all major OS TCP stacks. An attacker can repeat it at arbitrary rate.

### Recommendation

Add a `res.on('close', ...)` handler that runs the same accounting logic when `'finish'` has not already fired. A standard pattern:

```js
let finished = false;

res.on('finish', () => {
  finished = true;
  recordMetrics(res);
});

res.on('close', () => {
  if (!finished) {
    // client disconnected before finish; still record/decrement
    inFlightCounter.add(-1);
    allRequestCounter.add(1);
    // optionally record with a synthetic 499 or 0 status label
  }
});
```

This ensures `inFlightCounter` is always decremented and every request is counted regardless of whether the client stayed connected.

### Proof of Concept

```bash
# Terminal 1 – start the mirror-node REST service normally

# Terminal 2 – send a request and immediately abort before response arrives
# (the sleep gives the server time to start processing but not respond)
curl -v --max-time 0.001 http://<mirror-node-host>/api/v1/transactions 2>&1 || true

# Repeat N times
for i in $(seq 1 100); do
  curl --max-time 0.001 http://<mirror-node-host>/api/v1/transactions 2>/dev/null || true
done

# Terminal 3 – scrape Prometheus metrics endpoint
curl http://<mirror-node-host>/swagger/metrics/

# Observed: api_all_request_in_processing_total has grown by ~100
#           api_all_request_total has NOT grown (requests are invisible)
#           api_request_total counter shows 0 new entries for those paths
```
