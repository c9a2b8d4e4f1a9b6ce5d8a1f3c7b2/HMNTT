### Title
Accept-Encoding Header Bypass via Substring Match in `validateAcceptEncodingHeader()`

### Summary
The `validateAcceptEncodingHeader()` method in `OpcodesController` uses a plain `.contains("gzip")` substring check instead of proper RFC 7231 token-based parsing. Any unprivileged external user can craft a header value such as `not-gzip`, `xgzip`, or `gzip-disabled` to satisfy the check while the HTTP framework does not actually negotiate gzip compression, causing the server to emit large uncompressed opcode-trace responses.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, `validateAcceptEncodingHeader()`, line 76.

```java
// line 76
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
```

The check passes for any string that contains the four characters `gzip` anywhere, including invalid encoding tokens like `not-gzip`. The HTTP servlet layer (Spring/Tomcat) performs its own independent, RFC-compliant token parsing of `Accept-Encoding` to decide whether to compress the response body. Because `not-gzip` is not a recognized encoding token, the framework sends the response **uncompressed**. The controller's check is therefore trivially bypassed: the guard passes, `throttleManager.throttleOpcodeRequest()` is called (consuming one throttle token), and `opcodeService.processOpcodeCall(request)` performs a full EVM transaction replay and returns the full, uncompressed opcode trace.

The code comment at lines 71–73 explicitly states the motivation: *"the response from this endpoint is huge and without compression this will result in big network latency."* The check is the sole mechanism enforcing that clients accept compressed responses; there is no secondary enforcement at the framework or infrastructure layer visible in this codebase.

### Impact Explanation
An attacker can force the server to perform expensive EVM transaction replays (CPU-intensive) and return multi-megabyte uncompressed opcode traces over the wire. This amplifies both server-side CPU cost and outbound bandwidth per request compared to the intended compressed path. Because the `opcodeRateLimitBucket` is a single global bucket (not per-IP), a coordinated set of clients each staying within the throttle window can collectively saturate outbound bandwidth with uncompressed payloads, degrading service for legitimate users. The bypass does not allow transaction tampering (the endpoint is read-only), but it undermines the bandwidth-protection invariant the check was designed to enforce.

### Likelihood Explanation
No authentication or special privilege is required. The exploit requires only the ability to send an HTTP GET request with a crafted `Accept-Encoding` header value — a capability available to any internet user. The bypass is trivial to discover by reading the open-source code or by fuzzing the header. It is fully repeatable up to the throttle limit per second.

### Recommendation
Replace the substring check with proper RFC 7231 token-based parsing. Split the header value on commas and optional whitespace, strip quality-value (`q=`) parameters, and compare each token exactly (case-insensitively) against `"gzip"`:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw ...;
    }
    boolean hasGzip = Arrays.stream(acceptEncodingHeader.split(","))
        .map(token -> token.split(";")[0].trim().toLowerCase())
        .anyMatch("gzip"::equals);
    if (!hasGzip) {
        throw ...;
    }
}
```

This ensures only the exact token `gzip` (possibly with a `q` parameter) passes, matching the HTTP specification and the framework's own negotiation logic.

### Proof of Concept
**Precondition:** The opcodes endpoint is enabled (`properties.isEnabled() == true`). No credentials required.

**Request:**
```
GET /api/v1/contracts/results/<valid-tx-hash>/opcodes?stack=true&memory=true&storage=true HTTP/1.1
Host: <mirror-node-host>
Accept-Encoding: not-gzip
```

**Expected (correct) behavior:** Server returns `406 Not Acceptable` with `"Accept-Encoding: gzip header is required"`.

**Actual behavior:** The `.contains("gzip")` check passes because `"not-gzip".toLowerCase().contains("gzip")` is `true`. The throttle token is consumed, the EVM replay executes, and the server returns the full uncompressed opcode trace (potentially megabytes) with no `Content-Encoding: gzip` header, because the HTTP framework correctly does not compress for the non-standard token `not-gzip`.
