### Title
Unbounded `endTime` + `startTime=0` Enables Unauthenticated Resource Exhaustion via Unlimited Concurrent Subscriptions

### Summary
Any unauthenticated caller can open an arbitrary number of `subscribeTopic` (gRPC) sessions with `startTime=0` and `endTime=Long.MAX_VALUE`. Each session simultaneously triggers a full historical database scan from epoch and holds a live reactive pipeline open until approximately year 2262, with no per-user or global subscription cap enforced anywhere in the service layer. Stacking many such sessions exhausts the bounded-elastic scheduler threads, database connection pool, and per-subscription memory buffers.

### Finding Description

**Validation gap — no upper bound on `endTime`:**

`TopicMessageFilter.isValidEndTime()` only checks that `endTime > startTime`: [1](#0-0) 

`Long.MAX_VALUE` (9,223,372,036,854,775,807 ns from epoch ≈ year 2262) trivially satisfies `endTime > 0`, so it passes validation. `startTime=0` passes `isValidStartTime()` because `0 <= DomainUtils.now()`: [2](#0-1) 

**Subscription never terminates:**

`TopicContext.isComplete()` computes: [3](#0-2) 

With `endTime=Long.MAX_VALUE`, `Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to ~year 2262, so `isComplete()` returns `false` for the entire practical lifetime of the service. The `pastEndTime` flux therefore polls every 30 s indefinitely: [4](#0-3) 

**Full historical scan per subscription:**

`subscribeTopic` unconditionally calls `topicMessageRetriever.retrieve(filter, true)` (throttled=true): [5](#0-4) 

With `throttled=true`, `numRepeats=Long.MAX_VALUE` and the retriever pages through all messages from timestamp 0 until the page is smaller than `maxPageSize` (1000 rows), bounded only by a 60-second timeout: [6](#0-5) [7](#0-6) 

**Per-subscription polling loop (POLL mode):**

`PollingTopicListener.listen()` creates an independent polling loop with `RepeatSpec.times(Long.MAX_VALUE)` per subscription: [8](#0-7) 

**No subscription limit:**

`subscriberCount` is only a Micrometer gauge — it is never checked against any cap: [9](#0-8) 

There is no per-IP, per-user, or global subscription ceiling anywhere in `TopicMessageServiceImpl`, `CompositeTopicListener`, or `SharedTopicListener`. [10](#0-9) 

### Impact Explanation

Each malicious subscription consumes:
- A slot in the `boundedElastic` scheduler (historical retrieval + `pastEndTime` polling)
- Up to 60 seconds of repeated DB queries scanning from timestamp 0
- A 16,384-element in-memory buffer per subscription (`maxBufferSize`)
- An ongoing DB poll every 500 ms (POLL mode) or a Redis channel slot (REDIS mode)

Stacking hundreds of such sessions exhausts the scheduler thread pool, saturates the database connection pool, and causes OOM via buffer accumulation — degrading or denying service to legitimate subscribers. The mirror node's gRPC port is publicly accessible with no authentication.

### Likelihood Explanation

No credentials, tokens, or special network access are required. A single attacker with a gRPC client library can script the attack in minutes. The filter parameters (`startTime=0`, `endTime=Long.MAX_VALUE`) pass all server-side validation as shown above. The attack is trivially repeatable and can be automated. The only partial friction is the Netty connection limit (not inspectable from available code), but even a modest number of long-lived subscriptions (tens to low hundreds) is sufficient to exhaust DB connections or scheduler threads.

### Recommendation

1. **Enforce an upper bound on `endTime`**: Add a validation constraint (e.g., `endTime <= DomainUtils.now() + maxFutureWindow`) in `isValidEndTime()`.
2. **Enforce a maximum subscription count**: Check `subscriberCount` against a configurable cap in `subscribeTopic()` and reject new subscriptions when the cap is reached.
3. **Enforce a minimum `startTime` floor**: Reject `startTime` values older than a configurable retention window (e.g., 30 days) to prevent full-history scans.
4. **Add per-IP/per-client rate limiting** at the gRPC interceptor layer.
5. **Add a subscription timeout**: Terminate subscriptions that have been open longer than a configurable maximum duration regardless of `endTime`.

### Proof of Concept

```python
import grpc
# Pseudocode — adapt to actual proto definitions
stubs = []
for i in range(500):
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = ConsensusServiceStub(channel)
    req = ConsensusTopicQuery(
        topic_id=TopicID(topic_num=1),
        consensus_start_time=Timestamp(seconds=0, nanos=0),   # startTime = 0
        consensus_end_time=Timestamp(seconds=9223372036, nanos=854775807),  # Long.MAX_VALUE nanos
    )
    # Each call opens a streaming subscription; no auth required
    stubs.append(stub.subscribeTopic(req))
# 500 concurrent subscriptions each trigger full historical DB scan
# and hold open a live pipeline indefinitely
```

Each of the 500 sessions immediately begins paginating all historical topic messages from epoch (up to the 60 s retriever timeout) and then holds a live reactive pipeline open until ~year 2262, with the `pastEndTime` flux polling every 30 s and (in POLL mode) the listener polling the DB every 500 ms per session.

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
