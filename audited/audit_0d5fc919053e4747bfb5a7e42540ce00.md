### Title
Unbounded Per-Unique-EVM-Address DB Query Amplification via Importer `findByEvmAddress` with No Negative-Cache Saturation Guard

### Summary
The importer's `EntityIdServiceImpl` calls `entityRepository.findByEvmAddress()` for every non-parseable (non-long-zero) 20-byte EVM address encountered in Hedera transactions. While the alias cache stores negative results for previously seen addresses, each *new* unique address unconditionally triggers a live DB query. An unprivileged Hedera account holder can submit a sustained stream of `ContractCall` or `CryptoTransfer` transactions referencing distinct, never-before-seen non-parseable EVM addresses, causing one DB query per unique address with no cache protection, amplifying mirror-node DB load proportionally to the attacker's transaction rate.

### Finding Description

**Exact code path:**

`ContractCallTransactionHandler.getEntity()` calls `entityIdService.lookup(contractIdReceipt, contractIdBody)` for every processed `CONTRACTCALL` transaction. [1](#0-0) 

When the `ContractID` uses the `EVM_ADDRESS` case, `EntityIdServiceImpl.lookup(ContractID)` routes to `cacheLookup()` with the loader `findByEvmAddress()`: [2](#0-1) 

`cacheLookup()` uses Spring Cache's programmatic `cache.get(key, loader)` API. This caches the loader result (including `Optional.empty()`) keyed by the raw `ByteString` of the EVM address: [3](#0-2) 

Inside the loader, `DomainUtils.fromEvmAddress()` returns `null` for any address whose first 12 bytes are not all zero (i.e., any non-long-zero / non-parseable address), so execution falls through unconditionally to `entityRepository.findByEvmAddress()`: [4](#0-3) 

`findByEvmAddress()` in the repository is a plain JPA native query with **no `@Cacheable` annotation**: [5](#0-4) 

**Root cause and failed assumption:**

The cache (`CACHE_ALIAS`, Caffeine, `maximumSize=100000,expireAfterAccess=30m`) does store negative results — `Optional.empty()` is cached for addresses not found in the DB. This prevents repeated queries for the *same* address. However, the cache provides **zero protection against a stream of distinct unique addresses**: each new unique address is a cache miss, unconditionally issuing a DB query. The design assumes the set of EVM addresses seen in practice is bounded and re-used; it does not account for an attacker deliberately generating an unbounded stream of unique addresses. [6](#0-5) 

Additionally, the cache is capped at 100,000 entries. Once saturated, Caffeine's eviction policy removes entries, meaning previously cached negative results can be evicted and re-queried if the attacker cycles through more than 100,000 unique addresses.

**Why existing checks are insufficient:**

- `DomainUtils.fromEvmAddress()` only short-circuits for long-zero addresses (first 12 bytes all zero). Any address with a non-zero byte in positions 0–11 bypasses this check and hits the DB.
- The `notify()` cache-warming path only fires for entities that actually exist (accounts/contracts). Non-existent attacker-controlled addresses are never pre-warmed.
- There is no rate-limiting, deduplication window, or per-transaction EVM address lookup budget in the importer pipeline. [7](#0-6) 

### Impact Explanation

Every `ContractCall` or `CryptoTransfer` (alias path) transaction referencing a unique non-parseable EVM address causes one synchronous DB query against the `entity` table during record-stream ingestion. The importer is a single-threaded pipeline per record file; DB query latency directly increases per-file processing time. Sustained injection of unique-address transactions increases the DB query rate linearly with the attacker's transaction submission rate. If the attacker can sustain a transaction rate representing ≥30% of the mirror node's normal EVM-address lookup volume, the 30% resource consumption threshold is met. The `entity` table is large and indexed on `evm_address`; each query is individually cheap but the aggregate load at scale is significant. Secondary effects include increased DB connection pool pressure and potential query queue buildup.

### Likelihood Explanation

**Preconditions**: A funded Hedera account with no special privileges. Any mainnet/testnet participant qualifies.

**Feasibility**: Hedera charges per-transaction fees (~$0.10 USD on mainnet for a `ContractCall`). Sustaining 30 additional DB queries/second costs ~$3/second (~$10,800/hour). This is a meaningful economic barrier for mainnet but is negligible on testnet (free or near-free). On testnet mirror nodes (which operators often run with the same codebase), the attack is essentially free. On mainnet, a well-resourced attacker (e.g., a competitor or nation-state actor) could sustain this for hours.

**Repeatability**: The attack is fully repeatable. The attacker generates fresh random 20-byte addresses (with at least one non-zero byte in positions 0–11) for each transaction. No special knowledge of the system is required.

### Recommendation

1. **Introduce a negative-result cache with a bounded bloom filter or fixed-size LRU set** for EVM addresses confirmed absent from the DB, separate from the positive-result cache, to prevent re-querying the same absent address across cache evictions.
2. **Add a per-record-file deduplication set** for EVM address lookups: within a single record file's processing batch, deduplicate EVM address lookups so that N transactions referencing the same unknown address result in only one DB query.
3. **Batch EVM address lookups**: instead of one DB query per address, collect all unknown EVM addresses encountered in a record file and resolve them in a single `WHERE evm_address = ANY(?)` query.
4. **Apply `unless = "#result != null && #result.isEmpty()"` negative-cache suppression** carefully — or alternatively use a dedicated negative-cache with a shorter TTL and larger capacity — to prevent cache eviction from re-exposing the DB to repeated absent-address queries.

### Proof of Concept

1. Obtain a funded Hedera testnet account (free).
2. In a loop, generate a random 20-byte value where bytes 0–11 are not all zero (e.g., `byte[0] = 0x01`, bytes 1–19 random). This ensures `DomainUtils.fromEvmAddress()` returns `null`.
3. Submit a `ContractCall` transaction with `ContractID.evm_address` set to this unique 20-byte value. The transaction will fail at the consensus layer (no such contract), but the importer still processes the record and calls `entityIdService.lookup(contractIdReceipt, contractIdBody)`.
4. Repeat with a new unique address for each transaction.
5. Monitor the mirror node's DB query rate (e.g., via `pg_stat_activity` or Prometheus metrics). Each submitted transaction produces one `SELECT id FROM entity WHERE evm_address = $1 AND deleted <> true` query with no cache hit.
6. At 30+ unique-address transactions/second, observe DB query rate increase of ≥30% above baseline on a lightly loaded mirror node.

    private @NonNull Optional<EntityId> cacheLookup(ByteString key, Callable<Optional<EntityId>> loader) {
        try {
            return Objects.requireNonNullElse(cache.get(key, loader), Optional.empty());
        } catch (Cache.ValueRetrievalException e) {
            Utility.handleRecoverableError("Error looking up alias or EVM address {} from cache", key, e);
            return Optional.empty();
        }
    }

    private Optional<EntityId> findByEvmAddress(byte[] evmAddress, boolean throwRecoverableError) {
        var id = Optional.ofNullable(DomainUtils.fromEvmAddress(evmAddress))
                .or(() -> entityRepository.findByEvmAddress(evmAddress).map(EntityId::of));

        if (id.isEmpty() && throwRecoverableError) {
            Utility.handleRecoverableError("Entity not found for EVM address {}", Hex.encodeHexString(evmAddress));
        }

        return id;
    }

    public static EntityId fromEvmAddress(byte[] evmAddress) {
        final var commonProperties = CommonProperties.getInstance();

        try {
            if (isLongZeroAddress(evmAddress)) {
                final var num = Longs.fromByteArray(Arrays.copyOfRange(evmAddress, 12, 20));
                return EntityId.of(commonProperties.getShard(), commonProperties.getRealm(), num);
            }
        } catch (InvalidEntityException ex) {
            log.debug("Failed to parse long zero evm address into EntityId", ex);
        }
        return null;
    }
