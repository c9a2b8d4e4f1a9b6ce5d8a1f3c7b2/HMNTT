### Title
Statement-Boundary-Only Timeout Check Allows DB Connection Exhaustion via Single Long-Running Query

### Summary
The `statementInspector()` in `HibernateConfiguration.java` enforces the 10-second `requestTimeout` only at Hibernate statement boundaries — it cannot interrupt a query already in flight. The documented `hiero.mirror.web3.db.statementTimeout = 3000ms` property is never read or applied anywhere in the web3 Java codebase, leaving the only per-query backstop as the PostgreSQL-level `ALTER USER mirror_web3 SET statement_timeout TO '10000'` (10 seconds), which equals `requestTimeout`. An unprivileged attacker can craft a contract call that causes a slow SQL query to be submitted just before the 10-second mark, allowing that query to run for an additional 10 seconds, holding a DB connection for up to ~20 seconds per request and exhausting the pool of 275 `mirror_web3` connections with ~28 concurrent requests.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/config/HibernateConfiguration.java`, lines 31–47:

```java
StatementInspector statementInspector() {
    long timeout = web3Properties.getRequestTimeout().toMillis(); // 10,000 ms
    return sql -> {
        if (!ContractCallContext.isInitialized()) { return sql; }
        var startTime = ContractCallContext.get().getStartTime();
        long elapsed = System.currentTimeMillis() - startTime;
        if (elapsed >= timeout) {
            throw new QueryTimeoutException(...);
        }
        return sql;  // ← query is dispatched; inspector cannot cancel it
    };
}
```

`web3/src/main/java/org/hiero/mirror/web3/common/ContractCallContext.java`, line 31:
```java
private final long startTime = System.currentTimeMillis();
```

**Root cause — three compounding failures:**

1. **Inspector is pre-dispatch only.** The `StatementInspector` callback fires before the SQL is sent to the database. Once `return sql` executes, the query is in flight and the inspector has no further control. A query submitted at elapsed = 9,999 ms can then run for up to the PostgreSQL `statement_timeout` before being killed.

2. **`db.statementTimeout = 3000 ms` is never applied.** A grep of `web3/**/*.java` for `statementTimeout` returns zero matches. The property is documented in `docs/configuration.md` (line 702) but no `DbProperties` class exists in the web3 module and no Hikari `connectionInitSql` or JDBC URL parameter sets it. The only effective per-query timeout is the PostgreSQL role-level default set by `ALTER USER mirror_web3 SET statement_timeout TO '10000'` (10,000 ms) in `charts/hedera-mirror/templates/secret-passwords.yaml` line 127.

3. **pgbouncer silently drops `statement_timeout` startup parameters.** `charts/hedera-mirror/values.yaml` configures pgbouncer with `ignore_startup_parameters: extra_float_digits,options,statement_timeout`. Even if the application attempted to set `statement_timeout` as a connection startup parameter, pgbouncer would silently discard it.

**Effective maximum request duration:**
```
requestTimeout (10 s) + PostgreSQL statement_timeout (10 s) = up to 20 s of DB connection hold time per request
```

**Connection pool capacity:** pgbouncer limits `mirror_web3` to `max_user_connections: 275`. At 20 s per request, ~14 concurrent attackers (each holding 20 connections) can exhaust the pool.

### Impact Explanation

DB connection pool exhaustion causes all legitimate contract call requests to queue or fail with connection timeout errors. The web3 API becomes unavailable or severely degraded. The Grafana alert `Web3HighDBConnections` fires when active connections exceed 75% of max (≈206 connections), meaning the threshold for observable impact is lower than full exhaustion. This directly satisfies the "30%+ resource consumption increase" criterion: 28 concurrent slow requests consume 100% of the 275-connection pool, a ~33%+ increase over normal baseline utilization.

### Likelihood Explanation

No authentication is required to call `POST /api/v1/contracts/call`. The rate limiter allows 500 requests/second globally and the gas throttle allows up to 15,000,000 gas per request. An attacker needs only to identify or deploy a contract whose execution path triggers a slow Hibernate query (e.g., a historical block query against a large table, or a storage slot scan) and submit it repeatedly just before the 10-second mark. The attack is repeatable, requires no special privileges, and can be automated with a simple HTTP client sending ~30 concurrent long-running requests.

### Recommendation

1. **Apply `db.statementTimeout` in the web3 datasource.** Add a `connectionInitSql` to the Hikari config (e.g., `spring.datasource.hikari.connection-init-sql=SET statement_timeout=3000`) so the 3-second limit is actually enforced per query.

2. **Reduce the PostgreSQL role-level `statement_timeout` for `mirror_web3`** from 10,000 ms to 3,000 ms in `secret-passwords.yaml` line 127, consistent with the documented intent.

3. **Remove `statement_timeout` from pgbouncer's `ignore_startup_parameters`** so that per-connection timeout overrides are not silently dropped.

4. **Add a mid-query cancellation mechanism.** Use JDBC `Statement.setQueryTimeout()` via a Hibernate `StatementInspector` that also sets the timeout on the `PreparedStatement`, or use Spring's `@QueryHints` with `JDBC_TIMEOUT`, so the timeout is enforced inside the running query, not only at the next statement boundary.

### Proof of Concept

```
Preconditions:
- Mirror node web3 API accessible at https://<host>/api/v1/contracts/call
- A deployed contract (or precompile call) known to trigger a slow historical DB query

Steps:
1. Identify a contract call that causes a Hibernate query taking ~2-3 seconds
   (e.g., a historical block query: set "block": "0x1" to force historical path)

2. Send 30 concurrent requests:
   POST /api/v1/contracts/call
   {"to":"<contract>","data":"<calldata>","gas":15000000,"block":"0x1"}

3. Each request holds a DB connection for up to 20 seconds:
   - Inspector allows the last statement at T≈9.999s
   - PostgreSQL runs that statement for up to 10s more
   - Connection is held for ~20s total

4. With 30 concurrent requests × 20s hold = 600 connection-seconds consumed
   against a pool of 275 connections → pool exhausted

5. Legitimate requests receive connection timeout errors or 500 responses.

Expected result: Web3HighDBConnections alert fires; legitimate users receive errors.
