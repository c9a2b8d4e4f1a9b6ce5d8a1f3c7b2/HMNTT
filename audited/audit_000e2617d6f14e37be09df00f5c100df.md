### Title
Missing Account Existence Check in `getAccountCryptoAllowances` Returns False 200 for Non-Existent Numeric Account IDs

### Summary
`CryptoAllowanceController.getAccountCryptoAllowances` omits the `EntityService.isValidAccount()` guard that sibling handlers `listStakingRewardsByAccountId` and `getTokenRelationships` both perform. When a caller supplies a numeric account ID (e.g. `0.0.99999`) that does not exist in the entity table, the endpoint returns HTTP 200 with an empty allowances array instead of 404, making a non-existent account indistinguishable from an existing account that simply has no crypto allowances.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 17 routes `allowances/crypto` to `CryptoAllowanceController.getAccountCryptoAllowances`.

`rest/controllers/cryptoAllowanceController.js` lines 76-98:
```js
getAccountCryptoAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);
  // ← NO isValidAccount() call here
  const {conditions, params, order, limit} = this.extractCryptoAllowancesQuery(filters, accountId);
  const allowances = await CryptoAllowanceService.getAccountCryptoAllowances(...);
  // returns 200 + empty array when account does not exist
```

`rest/service/entityService.js` lines 118-137 — `getEncodedId` for a numeric ID (shard.realm.num) takes the fast path:
```js
if (EntityId.isValidEntityId(entityIdString)) {
  const entityId = EntityId.parseString(entityIdString, {paramName});
  return entityId.evmAddress === null
    ? entityId.getEncodedId()   // ← pure arithmetic, zero DB lookup
    : await this.getEntityIdFromEvmAddress(entityId, requireResult);
}
```
No database query is made; the encoded integer is returned regardless of whether the entity exists.

**Contrast with the two correctly-guarded handlers:**

`rest/controllers/accountController.js` lines 171-175 (`listStakingRewardsByAccountId`):
```js
const accountId = await EntityService.getEncodedId(...);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) { throw new NotFoundError(); }
```

`rest/controllers/tokenController.js` lines 67-71 (`getTokenRelationships`):
```js
const accountId = await EntityService.getEncodedId(...);
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) { throw new NotFoundError(); }
```

**Root cause:** The developer added `isValidAccount` guards to `listStakingRewardsByAccountId` and `getTokenRelationships` but forgot to add the same guard to `getAccountCryptoAllowances` (and identically to `getAccountTokenAllowances` in `tokenAllowanceController.js`).

**Why alias/EVM-address inputs are unaffected:** `getEncodedId` calls `getAccountIdFromAlias` / `getEntityIdFromEvmAddress`, both of which throw `NotFoundError` when the entity is absent. Only the numeric-ID branch skips the DB check.

**Test evidence:** `rest/__tests__/specs/accounts/{id}/allowances/crypto/alias-not-found.json` confirms a missing alias returns 404. No equivalent spec exists for a missing numeric ID — the gap is untested and the behavior is silently wrong.

### Impact Explanation

Any downstream system (indexer, wallet, DeFi protocol) that calls `GET /api/v1/accounts/{numericId}/allowances/crypto` and receives `200 {"allowances":[]}` cannot tell whether the account exists with zero allowances or does not exist at all. This causes:

- **False "no allowances" state** recorded for phantom accounts, enabling logic that depends on "account has no allowances" to be triggered for accounts that were never created.
- **Inconsistent API contract**: the same mirror node returns 404 for the same non-existent numeric account on `/rewards` and `/tokens`, but 200 on `/allowances/crypto`, breaking any consumer that cross-checks endpoints.
- Severity: **Medium** — no funds are directly at risk within the mirror node itself, but consumers that act on allowance state (e.g., approve-then-transfer flows) can be misled.

### Likelihood Explanation

No authentication or special privilege is required. Any unauthenticated HTTP client can trigger this by supplying any valid-format but non-existent numeric account ID. The attack is trivially repeatable, requires no prior knowledge of the system beyond the public API spec, and is stable across restarts (the entity table simply never contains the queried ID).

### Recommendation

Add the same `isValidAccount` guard to `getAccountCryptoAllowances` (and `getAccountTokenAllowances`) immediately after resolving the account ID:

```js
// rest/controllers/cryptoAllowanceController.js
getAccountCryptoAllowances = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  const isValidAccount = await EntityService.isValidAccount(accountId);   // ADD
  if (!isValidAccount) { throw new NotFoundError(); }                      // ADD
  ...
```

Add a corresponding integration-test spec (e.g. `account-not-found.json`) that asserts HTTP 404 when a numeric account ID that is absent from the entity table is queried, mirroring the existing `alias-not-found.json` spec.

### Proof of Concept

**Preconditions:** Mirror node is running; account `0.0.99999` does not exist in the entity table.

**Steps:**

1. Confirm the account does not exist:
   ```
   GET /api/v1/accounts/99999
   → 404 Not Found
   ```

2. Confirm `/rewards` correctly returns 404:
   ```
   GET /api/v1/accounts/99999/rewards
   → 404 Not Found  {"_status":{"messages":[{"message":"Not found"}]}}
   ```

3. Confirm `/tokens` correctly returns 404:
   ```
   GET /api/v1/accounts/99999/tokens
   → 404 Not Found  {"_status":{"messages":[{"message":"Not found"}]}}
   ```

4. Trigger the vulnerability:
   ```
   GET /api/v1/accounts/99999/allowances/crypto
   → 200 OK  {"allowances":[],"links":{"next":null}}
   ```

Step 4 returns a successful response with an empty allowances list for an account that does not exist, while steps 2 and 3 correctly return 404 for the same non-existent account.
