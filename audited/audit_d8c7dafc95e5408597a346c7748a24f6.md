### Title
Unauthenticated Double DB Query Per Request in `listStakingRewardsByAccountId` Enables DB Connection Pool Exhaustion

### Summary
The `listStakingRewardsByAccountId` function in `rest/controllers/accountController.js` unconditionally executes two sequential database queries per request — `EntityService.isValidAccount()` and `StakingRewardTransferService.getRewards()` — with no application-level rate limiting on the REST API. An unauthenticated attacker can enumerate valid account IDs (trivially sequential integers) and flood the endpoint with concurrent maximum-limit requests, exhausting the default DB connection pool of 10 connections and causing sustained degradation or unavailability of the mirror node REST API across all instances sharing that pool.

### Finding Description
**Exact code path:**

`rest/controllers/accountController.js`, `listStakingRewardsByAccountId`, lines 170–203:

```js
listStakingRewardsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(...);   // DB query #1 (alias/EVM path)
  const isValidAccount = await EntityService.isValidAccount(accountId); // DB query #2 always
  if (!isValidAccount) { throw new NotFoundError(); }
  ...
  const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(...); // DB query #3 always
```

`EntityService.isValidAccount` at `rest/service/entityService.js` line 61 always issues:
```sql
SELECT type FROM entity WHERE id = $1
```

`StakingRewardTransferService.getRewards` at `rest/service/stakingRewardTransferService.js` line 18–21 always issues:
```sql
SELECT ... FROM staking_reward_transfer srt WHERE srt.account_id = $1 ORDER BY ... LIMIT $2
```

**Root cause:** The function performs a mandatory existence check (`isValidAccount`) followed unconditionally by the data query (`getRewards`), resulting in 2–3 DB round-trips per request. The REST API server (`rest/server.js`) has no rate-limiting middleware — only `cors`, `compression`, `requestLogger`, `authHandler` (response-limit only), and an optional Redis response cache that is **disabled by default** (`cache.response.enabled: false`). The DB pool is capped at **10 connections** by default (`hiero.mirror.rest.db.pool.maxConnections: 10`).

**Account ID enumeration:** Hedera account IDs are sequential integers (e.g., `0.0.1`, `0.0.2`, …). No authentication or CAPTCHA is required. The `invalid-path.json` spec confirms only malformed IDs are rejected; numeric IDs pass straight through to DB.

### Impact Explanation
With a pool of 10 DB connections and 2–3 queries per request held open for up to 20 seconds (`statementTimeout: 20000`), as few as 5 concurrent attackers each sending a stream of `GET /api/v1/accounts/{id}/rewards?limit=100` requests can saturate the entire connection pool. All other REST API endpoints sharing the same pool (accounts, transactions, tokens, etc.) will queue indefinitely or time out. If multiple mirror node REST API pods share the same PostgreSQL instance (standard deployment), all pods are affected simultaneously, taking down the entire mirror node REST API tier.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero specialized knowledge. Account IDs are public and sequential. The endpoint is publicly documented in the OpenAPI spec. A single attacker with a modest script (e.g., `ab`, `wrk`, or a simple async loop) can sustain the attack indefinitely. The response cache being disabled by default means no natural protection exists in a default deployment.

### Recommendation
1. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit`) scoped per IP, applied before route handlers in `rest/server.js`.
2. **Merge the two queries**: Replace the separate `isValidAccount` + `getRewards` pattern with a single query that returns rewards only if the account exists (e.g., using a `JOIN` or `EXISTS` subquery), eliminating the doubled DB load.
3. **Enable the response cache by default** (`cache.response.enabled: true`) to serve repeated identical requests from Redis without hitting the DB.
4. **Increase the DB pool size** or add a connection queue timeout to fail fast under saturation rather than holding connections open for 20 seconds.

### Proof of Concept
```bash
# Step 1: Enumerate a valid account ID (trivial — they are sequential)
curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/rewards" | jq .rewards

# Step 2: Flood with concurrent max-limit requests (no auth required)
seq 1 200 | xargs -P 50 -I{} \
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1001/rewards?limit=100" -o /dev/null

# Step 3: Observe that all other REST API endpoints begin timing out or returning 503
curl -s "https://<mirror-node>/api/v1/transactions" 
# → connection timeout or pool exhaustion error
```

Each of the 50 concurrent requests triggers 2 DB queries held for up to 20 seconds, saturating the 10-connection pool within the first wave and sustaining degradation for the duration of the attack.
