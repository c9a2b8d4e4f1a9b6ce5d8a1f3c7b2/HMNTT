### Title
Missing COALESCE on Nullable `deleted` Column in `entityFromEvmAddressQuery` Causes Live Entities with `deleted = NULL` to Be Silently Excluded

### Summary
`entityFromEvmAddressQuery` filters with `deleted <> true` without `COALESCE`, but the `deleted` column is explicitly nullable (NOT NULL constraint was dropped in migration V1.39.1). In SQL, `NULL <> true` evaluates to `NULL` (unknown), not `TRUE`, so any entity whose `deleted` column is `NULL` is silently excluded from the result set. An unprivileged external user who submits the EVM address of such an entity will receive a `NotFoundError` as if the entity does not exist, while the parallel alias-lookup query correctly handles this case with `COALESCE`.

### Finding Description

**Exact code location:**

`rest/service/entityService.js`, lines 22–25 (`entityFromEvmAddressQuery`), consumed by `getEntityIdFromEvmAddress()` at line 90–104:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where ${Entity.DELETED} <> true   // ← no COALESCE
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

Compare with the alias query, which correctly handles NULL:

```js
static entityFromAliasQuery = `select ${Entity.ID}
                               from ${Entity.tableName}
                               where coalesce(${Entity.DELETED}, false) <> true
                                 and ${Entity.ALIAS} = $1`;
``` [2](#0-1) 

**Root cause — `deleted` is nullable by design:**

Migration `V1.39.1__upsert_support.sql` explicitly dropped the NOT NULL constraint and default value from the `deleted` column with the comment *"allow nullable on entity deleted as transaction cannot make this assumption on updates"*:

```sql
alter table entity
    alter column deleted drop default,
    alter column deleted drop not null;
``` [3](#0-2) 

This means `deleted = NULL` is a legitimate, expected database state for entities that were partially upserted. In PostgreSQL, `NULL <> true` yields `NULL`, which is falsy in a `WHERE` clause, so those rows are silently dropped from the result.

**Exploit flow:**

1. Entity `E` exists in the `entity` table with `evm_address = X` and `deleted = NULL`.
2. Attacker (no privileges required) submits any API request that resolves an entity by EVM address — e.g., `GET /api/v1/accounts/{evmAddress}`, contract calls, token operations, or transaction routing — all of which ultimately call `getEncodedId()` → `getEntityIdFromEvmAddress()`. [4](#0-3) 
3. `entityFromEvmAddressQuery` is executed with `evm_address = X`. The predicate `deleted <> true` evaluates to `NULL` for entity `E`, so zero rows are returned.
4. `getEntityIdFromEvmAddress()` sees `rows.length === 0` and throws `NotFoundError` (when `requireResult = true`, the default). [5](#0-4) 
5. The caller treats the entity as non-existent; any transaction targeting it is rejected or misdirected.

**Why existing checks are insufficient:**

The only guard is the `rows.length > 1` duplicate check. There is no fallback, no `IS NULL` branch, and no `COALESCE`. The alias path has the correct fix already applied; the EVM address path does not.

### Impact Explanation

Any live entity whose `deleted` column is `NULL` — a state the importer explicitly produces during partial upserts — becomes completely invisible to EVM-address-based lookups. API endpoints for accounts, contracts, tokens, and transaction routing that resolve identifiers via `getEncodedId()` / `getEntityIdFromEvmAddress()` will return 404 for these entities. Transactions targeting such entities by EVM address will be rejected. Because the alias path is unaffected, the same entity may be reachable by alias but not by EVM address, creating an inconsistent identity resolution that can be exploited to selectively deny service to specific entities.

### Likelihood Explanation

No privileges are required. The attacker only needs to know the EVM address of an affected entity, which is public information on a mirror node. The `deleted = NULL` state is not an edge case — it is the documented, intentional result of the upsert mechanism used during normal importer operation. Any entity that was last written via a partial update (rather than a full insert) may be in this state. The attack is trivially repeatable: submit the EVM address, observe 404, repeat.

### Recommendation

Apply `COALESCE` to the `deleted` predicate in `entityFromEvmAddressQuery`, exactly as is already done in `entityFromAliasQuery`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
```

This treats `NULL` as `false` (not deleted), matching the intended semantics and the alias query's behavior.

### Proof of Concept

```sql
-- 1. Insert a live entity with deleted = NULL (as the importer upsert path produces)
INSERT INTO entity (id, num, realm, shard, evm_address, type, deleted)
VALUES (1234, 1234, 0, 0, decode('abcdef1234567890abcdef1234567890abcdef12', 'hex'), 'ACCOUNT', NULL);

-- 2. Demonstrate the bug: query returns 0 rows
SELECT id FROM entity
WHERE deleted <> true                                          -- NULL <> true → NULL → excluded
  AND evm_address = decode('abcdef1234567890abcdef1234567890abcdef12', 'hex');
-- Result: 0 rows

-- 3. Demonstrate the fix: query returns 1 row
SELECT id FROM entity
WHERE coalesce(deleted, false) <> true                        -- coalesce(NULL, false) = false → false <> true → TRUE
  AND evm_address = decode('abcdef1234567890abcdef1234567890abcdef12', 'hex');
-- Result: 1 row (entity found correctly)
```

An API call such as `GET /api/v1/accounts/0xabcdef1234567890abcdef1234567890abcdef12` against a mirror node where this entity has `deleted = NULL` will return HTTP 404, while the entity is fully live and reachable by other identifier types.
