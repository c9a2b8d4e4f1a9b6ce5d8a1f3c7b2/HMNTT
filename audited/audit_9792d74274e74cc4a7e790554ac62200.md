### Title
Unprivileged DoS via inFlightReq-Generated 503s Tripping Traefik Circuit Breaker

### Summary
The Traefik middleware chain for the rosetta service places `circuitBreaker` as the outermost wrapper around `inFlightReq`, which returns HTTP 503 when the per-IP concurrent request limit (5) is exceeded. Because the circuit breaker's `ResponseCodeRatio(500, 600, 0, 600) > 0.25` expression measures responses from its entire inner chain — including 503s emitted by `inFlightReq` — an unprivileged attacker can deliberately flood the endpoint with concurrent requests, generating enough 503s to open the circuit breaker and deny service to all users.

### Finding Description
**Code locations:**

- `rosetta/app/middleware/metrics.go` lines 76–83: `MetricsMiddleware` instruments requests at the application layer using `weaveworks/common/middleware.Instrument`. It records `hiero_mirror_rosetta_request_duration` (with `status_code` label) and `hiero_mirror_rosetta_request_inflight`. Critically, this middleware only sees requests that actually reach the Go backend — it is blind to Traefik-layer rejections.

- `charts/hedera-mirror-rosetta/values.yaml` lines 149–166: The Traefik middleware chain is ordered as:
  ```
  circuitBreaker  →  inFlightReq (amount: 5, per-IP depth:1)
                  →  rateLimit (average: 10, per requestHost)
                  →  retry (attempts: 3)
                  →  stripPrefix
                  →  rosetta backend
  ```

**Root cause and failed assumption:**

The design assumes that `inFlightReq` acts as a protective gate that shields the circuit breaker from self-generated 503s. This assumption is wrong. In Traefik, `circuitBreaker` wraps the entire remaining middleware chain as its "next handler." When `inFlightReq` rejects a request with 503, that response propagates directly back to `circuitBreaker`, which records it in `ResponseCodeRatio(500, 600, 0, 600)`. The `retry` middleware (which could absorb some 5xx responses) is positioned *inside* `inFlightReq`'s scope and is never invoked for `inFlightReq` rejections.

**Exploit flow:**

1. Attacker opens >5 concurrent long-lived HTTP connections to any rosetta endpoint (e.g., `POST /rosetta/network/status`) from a single IP.
2. `inFlightReq` allows 5 and returns HTTP 503 for all additional concurrent requests.
3. `circuitBreaker` records each 503 in its sliding-window `ResponseCodeRatio`.
4. If the attacker's 503s constitute >25% of all requests in the window (trivially achievable when legitimate traffic is low, or by flooding), the circuit breaker opens.
5. Once open, the circuit breaker returns 503 to **every** incoming request — including legitimate ones — until the ratio recovers.

**Why existing checks are insufficient:**

- `inFlightReq` is per-IP (`depth: 1`), but the circuit breaker ratio is **global** across all IPs and all requests. A single IP generating 6 concurrent requests produces a 503 rate of ~17% from that IP alone; with minimal legitimate traffic, this easily exceeds 25% globally.
- `rateLimit` is keyed on `requestHost` (the HTTP `Host` header), not the client IP. An attacker can vary the `Host` header across requests to bypass per-host rate limiting entirely, or simply stay within 10 req/s while keeping connections open.
- No authentication or authorization is required to reach any rosetta endpoint listed in the ingress (`/rosetta/account`, `/rosetta/block`, `/rosetta/network`, etc.).
- The `retry` middleware (3 attempts) is irrelevant to `inFlightReq` rejections — it only retries backend failures.

### Impact Explanation
When the circuit breaker opens, Traefik returns HTTP 503 to **all** rosetta requests globally, not just the attacker's. This is a complete, service-wide denial of service. The rosetta API is used for Coinbase Rosetta-compatible blockchain access; an outage blocks all block data queries, account lookups, and construction operations. The `MetricsMiddleware` application-level metrics (`hiero_mirror_rosetta_request_duration`) will show a sudden drop to zero (no requests reaching the backend), masking the root cause from application-level monitoring while the Prometheus alert (`RosettaApiErrors`) fires only after 1 minute — giving the attacker a sustained window of disruption.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only the ability to open concurrent TCP connections to the rosetta ingress. Any public internet user can execute it. The `inFlightReq` limit of 5 is extremely low, making it trivial to exceed. The attack is repeatable: after the circuit breaker recovers (half-open state), the attacker can immediately re-trigger it. A single attacker with a standard HTTP client (e.g., `curl`, `ab`, `wrk`) can sustain the condition indefinitely.

### Recommendation
1. **Reorder the middleware chain** so that `inFlightReq` and `rateLimit` are placed *outside* (before) the `circuitBreaker`, or use a separate Traefik `IngressRoute` chain where rate-limiting middlewares are not wrapped by the circuit breaker. This prevents self-generated 503s from counting toward the circuit breaker ratio.
2. **Alternatively**, replace `inFlightReq` with a middleware that returns HTTP 429 (Too Many Requests) instead of 503. Since `ResponseCodeRatio(500, 600, 0, 600)` only counts 5xx responses, 429s would not trip the circuit breaker.
3. **Increase `inFlightReq` amount** or apply it globally (not per-IP) to reduce the ratio of 503s to total traffic.
4. **Add IP-based authentication or network policy** to restrict access to the rosetta ingress to known clients.
5. In `MetricsMiddleware`, consider propagating Traefik-layer rejection metrics to application-level observability so that circuit breaker trips are visible in the same dashboard as application errors.

### Proof of Concept
```bash
# Step 1: Open 20 concurrent long-lived requests to a rosetta endpoint
# (adjust HOST to the actual rosetta ingress hostname)
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://HOST/rosetta/network/status \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' &
done
wait

# Expected output: 5x "200" (or backend response), 15x "503" (inFlightReq rejections)
# These 503s inflate ResponseCodeRatio to ~75%, well above the 0.25 threshold.

# Step 2: Immediately after, send a legitimate request
curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST https://HOST/rosetta/network/status \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'

# Expected output: "503" — circuit breaker is open, all users denied service.
```
