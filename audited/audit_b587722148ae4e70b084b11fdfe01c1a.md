### Title
Timing Side-Channel in `authenticate()` Allows Username Enumeration via `&&` Short-Circuit Evaluation

### Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` uses JavaScript's `&&` short-circuit operator to chain two `tsscmp` (timing-safe string compare) calls. Because `&&` skips the second operand when the first is falsy, a wrong username causes only one `tsscmp` call to execute, while a correct username causes two. This measurable timing difference allows an unprivileged attacker to enumerate the configured metrics username, directly undermining the constant-time guarantee that `tsscmp` is meant to provide.

### Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, `authenticate()`, line 143.

```js
// line 137-144
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

**Root cause:** `tsscmp` is a constant-time comparator — it always iterates over the full length of both strings regardless of where they differ. However, the `&&` operator in JavaScript is not constant-time: if `tsscmp(credentials.name, username)` returns `false`, the engine never calls `tsscmp(credentials.pass, password)`. This creates two distinct execution paths:

| Submitted username | Calls executed | Relative wall-clock time |
|---|---|---|
| Wrong | `tsscmp(name, username)` only | T |
| Correct | `tsscmp(name, username)` + `tsscmp(pass, password)` | T + ΔT |

`ΔT` is the cost of one additional `tsscmp` invocation. Over many samples this signal is statistically distinguishable, especially on a low-latency (same-datacenter or LAN) path.

**Why existing checks fail:** `tsscmp` is the only protection. There is no rate-limiting, no account lockout, and no jitter/delay added to the authentication path. The `config.metrics.config.username` value has a well-known default (`mirror_api_metrics`, per `docs/configuration.md` line 570) but operators may change it; the timing oracle works regardless of the actual value.

### Impact Explanation
An attacker who can reach the metrics endpoint (default path `/swagger/metrics/`) can:
1. Enumerate the configured username by timing many requests with candidate values.
2. Use the confirmed username to reduce the brute-force search space for the password to a single dimension.
3. Gain access to Prometheus-format metrics, which expose internal request rates, error rates, latency histograms, CPU/memory usage, and in-flight request counts — operational intelligence useful for planning further attacks or detecting monitoring blind spots.

Severity: **Medium**. The metrics endpoint is not a data-plane API, but leaking operational telemetry to an unauthenticated attacker is a meaningful information disclosure, and the timing oracle directly weakens the authentication scheme that is supposed to protect it.

### Likelihood Explanation
- **Precondition:** The metrics endpoint must be reachable (it is enabled by default, `metrics.enabled: true`). No account or privilege is required.
- **Feasibility:** The attack requires sending O(10³–10⁴) requests per candidate username to overcome network jitter. On a co-located scraper or internal network this is straightforward. From the public internet it is harder but not impossible with statistical averaging.
- **Repeatability:** Fully repeatable; there is no lockout or rate-limit to exhaust.
- **Tooling:** Standard HTTP benchmarking tools (e.g., `wrk`, `hey`, or a simple Python script with `time.perf_counter_ns`) are sufficient.

### Recommendation
Evaluate both comparisons unconditionally before combining the results, so the number of `tsscmp` calls is always two regardless of the username value:

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  const credentials = basicAuth(req);
  if (!credentials) return false;
  // Both comparisons MUST run before the result is combined
  const nameOk = tsscmp(credentials.name, username);
  const passOk = tsscmp(credentials.pass, password);
  return nameOk && passOk;
};
```

This ensures the execution time is constant with respect to whether the username is correct, preserving the timing-safe guarantee of `tsscmp`.

### Proof of Concept

```python
import base64, statistics, time, requests

TARGET = "http://<host>/swagger/metrics/"
CANDIDATES = ["mirror_api_metrics", "admin", "prometheus", "metrics", ""]

def probe(username, password="x", n=500):
    creds = base64.b64encode(f"{username}:{password}".encode()).decode()
    headers = {"Authorization": f"Basic {creds}"}
    times = []
    for _ in range(n):
        t0 = time.perf_counter_ns()
        requests.get(TARGET, headers=headers)
        times.append(time.perf_counter_ns() - t0)
    return statistics.median(times)

baseline = probe("")          # empty username — only 1 tsscmp call
for c in CANDIDATES:
    t = probe(c)
    print(f"{c!r:30s}  median_ns={t:>10.0f}  delta={t - baseline:+.0f}")
# The candidate whose median is consistently higher than the baseline
# by ~ΔT (cost of one extra tsscmp call) is the correct username.
```

A statistically significant positive delta for one candidate identifies the configured username. The attacker then brute-forces only the password dimension.
