### Title
Unauthenticated Multiplicative Query Amplification DoS via `/api/v1/accounts?limit=100&balance=true`

### Summary
In `getAccounts()` (`rest/accounts.js`), any unauthenticated user can set `limit` to its server-enforced maximum (100) while `balance=true` is the default. This causes `getEntityBalanceQuery()` to emit a correlated subquery per returned row, each fetching up to `tokenBalanceResponseLimit.multipleAccounts` (50) token balances. With no application-level rate limiting present, an attacker can flood the service with concurrent requests, each forcing the database to execute 100 correlated subqueries against a potentially massive `latest_token_balance` CTE, exhausting DB CPU and connection pool resources.

### Finding Description

**Code path:**

`getAccounts()` at `rest/accounts.js:335-388` sets:
```js
const tokenBalanceQuery = {query: 'account_id = e.id', params: [], limit: tokenBalanceResponseLimit.multipleAccounts};
// tokenBalanceResponseLimit.multipleAccounts = 50 (server-side constant, not user-controlled)
``` [1](#0-0) 

`limitAndOrderQuery` is parsed via `parseLimitAndOrderParams`, which caps the user-supplied `limit` at `responseLimit.max` = 100: [2](#0-1) 

Default config confirms: `{default: 25, max: 100, tokenBalance: {multipleAccounts: 50, singleAccount: 1000}}`: [3](#0-2) 

`getEntityBalanceQuery()` then builds the following SQL structure (non-historical path):
```sql
with latest_token_balance as (
   select account_id, balance, token_id
   from token_account
   where associated is true)          -- full CTE, potentially millions of rows
select ...,
  (select json_agg(...)
   from (
     select token_id, balance
     from latest_token_balance
     where account_id = e.id          -- correlated: re-evaluated per outer row
     order by token_id asc
     limit 50                         -- fixed server-side
   ) as account_token_balance) as token_balances
from entity as e
...
limit 100                             -- user-controlled, capped at max
``` [4](#0-3) 

The `latest_token_balance` CTE is materialized once but the correlated subquery (`where account_id = e.id`) is re-evaluated for each of the up to 100 outer rows. On a production network with millions of token associations, each CTE scan is expensive.

**No application-level rate limiting exists** in the REST layer: [5](#0-4) 

The only protection is a `statement_timeout` at the DB pool level, which limits individual query duration but does not prevent concurrent request flooding that exhausts the connection pool (`max: config.db.pool.maxConnections`).

**Root cause / failed assumption:** The design assumes that bounding the account limit (100) and token balance limit (50) is sufficient to prevent abuse. This ignores the multiplicative cost structure (100 × 50 = 5,000 correlated subquery evaluations per request) and the absence of any per-IP or per-client rate limiting, allowing an attacker to amplify DB load linearly with concurrent requests.

### Impact Explanation

Each request with `limit=100&balance=true` (or just `limit=100`, since `balance=true` is the default) forces the database to:
1. Materialize the full `latest_token_balance` CTE (all associated token accounts)
2. Execute 100 correlated subqueries against it

Flooding with concurrent requests exhausts the DB connection pool and CPU, causing query queuing, timeouts, and ultimately service unavailability for all users — including the network's ability to serve transaction confirmation queries. This is a **Critical** severity DoS with no authentication required.

### Likelihood Explanation

The exploit requires zero privileges, zero authentication, and a single HTTP endpoint. The request `GET /api/v1/accounts?limit=100` is indistinguishable from legitimate traffic. Any attacker with a modest botnet or even a single machine with high concurrency can trigger this. The endpoint is publicly documented in the OpenAPI spec. Repeatability is unlimited. [6](#0-5) 

### Recommendation

1. **Add application-level rate limiting** (e.g., `express-rate-limit`) per IP/client on the `/api/v1/accounts` endpoint, specifically tightened when `balance=true`.
2. **Decouple token balance fetching**: Instead of a correlated subquery per row, fetch token balances in a single separate query after the account list is retrieved, joining by `account_id IN (...)`.
3. **Reduce `tokenBalanceResponseLimit.multipleAccounts`** default (currently 50) or require explicit opt-in for balance inclusion rather than defaulting `balance=true`.
4. **Add a DB-level per-role query cost limit** or connection limit per application user to bound the blast radius.

### Proof of Concept

```bash
# Single expensive request (no auth required, balance=true is default)
curl "https://<mirror-node>/api/v1/accounts?limit=100"

# Concurrent flood (e.g., 200 parallel requests)
seq 200 | xargs -P 200 -I{} curl -s -o /dev/null \
  "https://<mirror-node>/api/v1/accounts?limit=100"
```

Expected result: DB CPU spikes to 100%, connection pool exhausts (`max: maxConnections`), subsequent requests time out or return 500 errors, service becomes unavailable to all clients.
