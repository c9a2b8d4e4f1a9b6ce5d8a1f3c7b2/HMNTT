### Title
Hash-Based Transaction Lookup Silently Filtered by `nonce`/`scheduled` Parameters, Causing Valid Transactions to Return 404

### Summary
`getTransactionsByIdOrHash()` accepts `nonce` and `scheduled` query parameters for all lookups via `acceptedSingleTransactionParameters`, but `isTransactionHash` is only determined *inside* `extractSqlFromTransactionsByIdOrHashRequest()` — after filter validation. When a hash is supplied, the same `nonce`/`scheduled` filters are unconditionally appended to the SQL `WHERE` clause by `getTransactionByIdQueryParamConditions()`, even though a hash already uniquely identifies a single transaction record. If the supplied filter value does not match the stored column value, the query returns zero rows and the handler throws `NotFoundError`, making a provably valid transaction appear non-existent to any caller.

### Finding Description

**Code path:**

`rest/transactions.js` line 924 — filters are built and validated before `isTransactionHash` is known:
```js
const filters = utils.buildAndValidateFilters(req.query, acceptedSingleTransactionParameters);
```
`acceptedSingleTransactionParameters` (line 1014) is `new Set([NONCE, SCHEDULED])`, so both parameters pass validation unconditionally.

`extractSqlFromTransactionsByIdOrHashRequest()` (lines 763–795) determines `isTransactionHash` internally. When `true`, it resolves the hash to `(payer_account_id, consensus_timestamp)` pairs and immediately delegates to `getTransactionsByTransactionIdsSql(lookupKeys, filters, Transaction.CONSENSUS_TIMESTAMP)` (line 781) — passing the full, unmodified `filters` array.

`getTransactionsByTransactionIdsSql()` (line 882) calls `getTransactionByIdQueryParamConditions(filters, params)` with no `isTransactionHash` guard:
```js
const {conditions: filterConditions, scheduled} = getTransactionByIdQueryParamConditions(filters, params);
const mainConditions = [...commonConditions, `(${idConditions.join(' or ')})`, ...filterConditions].join(' and ');
```

`getTransactionByIdQueryParamConditions()` (lines 797–826) unconditionally appends `nonce = $N` and/or `scheduled = $N` to the SQL `WHERE` clause for any non-`undefined` filter value. The comment at line 799 ("only parse nonce and scheduled query filters if the path parameter is transaction id **or hash**") confirms this is the intended path, but the semantic contract is broken: a hash already uniquely identifies one record, so an additional `nonce`/`scheduled` predicate can only *exclude* that record.

**Root cause:** The decision point for "is this a hash lookup?" is buried inside an async helper called *after* filter acceptance. There is no code path that strips or ignores `nonce`/`scheduled` when `isTransactionHash === true`.

**Exploit flow:**
1. Attacker identifies a valid transaction hash `H` for a transaction with `nonce = 0`.
2. Attacker issues: `GET /api/v1/transactions/H?nonce=1`
3. `buildAndValidateFilters` accepts `nonce=1` (it is in `acceptedSingleTransactionParameters`).
4. Hash lookup resolves `H` → `(payer_account_id, consensus_timestamp)`.
5. SQL query is built with `AND nonce = $N` where `$N = 1`.
6. The stored row has `nonce = 0`; zero rows are returned.
7. Line 940–942: `if (rows.length === 0) { throw new NotFoundError(); }` → HTTP 404.

The same attack works with `scheduled=true` against a non-scheduled transaction, or `scheduled=false` against a scheduled one.

### Impact Explanation
Any caller — including automated payment processors, block explorers, or audit tools — that queries a known transaction hash and receives a 404 may incorrectly conclude the transaction does not exist. In financial workflows this can be exploited to deny proof of payment, trigger duplicate submissions, or bypass transaction-existence checks. The attacker needs only a valid hash (publicly observable on-chain) and knowledge of a mismatching `nonce` or `scheduled` value (also publicly observable). No authentication or elevated privilege is required.

### Likelihood Explanation
The attack is trivially repeatable by any unauthenticated HTTP client. Transaction hashes and their nonce/scheduled values are public. The only precondition is knowing a valid hash, which is available from any block explorer or the mirror node's own `/api/v1/transactions` list endpoint. The crafted URL can be shared or embedded in links to mislead downstream consumers.

### Recommendation
In `getTransactionsByIdOrHash()` (or inside `extractSqlFromTransactionsByIdOrHashRequest()`), detect `isTransactionHash` *before* passing filters downstream and strip `nonce`/`scheduled` filters when the lookup is hash-based:

```js
const isTransactionHash = isValidTransactionHash(req.params.transactionIdOrHash);
const effectiveFilters = isTransactionHash
  ? filters.filter(f => f.key !== constants.filterKeys.NONCE && f.key !== constants.filterKeys.SCHEDULED)
  : filters;
```

Alternatively, reject `nonce`/`scheduled` at the validation layer when a hash is detected, or document and enforce that these parameters are a no-op (ignored, not applied) for hash-based lookups. The `acceptedSingleTransactionParameters` set should not be used uniformly for both lookup modes without a guard.

### Proof of Concept

**Preconditions:** Mirror node is running; a transaction with hash `0xAABB…` exists in the DB with `nonce = 0` and `scheduled = false`.

```
# Step 1 – confirm transaction exists
curl -s "https://<mirror>/api/v1/transactions/0xAABB..."
# → HTTP 200, transaction returned

# Step 2 – same hash, mismatched nonce filter
curl -v "https://<mirror>/api/v1/transactions/0xAABB...?nonce=1"
# → HTTP 404 {"_status":{"messages":[{"message":"Not found"}]}}

# Step 3 – same hash, mismatched scheduled filter
curl -v "https://<mirror>/api/v1/transactions/0xAABB...?scheduled=true"
# → HTTP 404 {"_status":{"messages":[{"message":"Not found"}]}}
```

The valid transaction is hidden from the caller solely by the attacker-controlled query parameter, with no server-side rejection or warning.
