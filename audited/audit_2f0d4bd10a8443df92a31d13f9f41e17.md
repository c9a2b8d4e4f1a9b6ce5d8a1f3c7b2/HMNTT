### Title
Global Gas-Bucket Exhaustion DoS via Unauthenticated High-Gas Requests

### Summary
The `gasLimitBucket` and `rateLimitBucket` in `ThrottleManagerImpl` are single global (non-per-IP) Bucket4j instances. An unauthenticated attacker can send requests at the maximum allowed rate (500 req/s) each carrying the default maximum gas value (15,000,000), which exactly saturates both global buckets simultaneously. All concurrent legitimate users then receive `ThrottleException` for every request until the attacker stops.

### Finding Description
**Code path:**

`ContractController.call()` (line 40) → `ThrottleManagerImpl.throttle()` (lines 38–41) → global `rateLimitBucket` and `gasLimitBucket`.

**Root cause — global shared buckets with no per-client fairness:**

`ThrottleConfiguration` creates exactly one `rateLimitBucket` bean and one `gasLimitBucket` bean for the entire JVM:

```
// ThrottleConfiguration.java lines 24-45
@Bean(name = RATE_LIMIT_BUCKET)   // capacity = requestsPerSecond (default 500)
@Bean(name = GAS_LIMIT_BUCKET)    // capacity = scaleGas(gasPerSecond)
```

`ThrottleProperties` defaults:
- `gasPerSecond = 7_500_000_000` → `scaleGas()` → bucket capacity = **750,000 tokens**
- `requestsPerSecond = 500` → bucket capacity = **500 tokens**

`ContractCallRequest.gas` defaults to `15_000_000` (line 37 of `ContractCallRequest.java`).

`scaleGas(15_000_000)` = `15_000_000 / 10_000` = **1,500 tokens per request**.

**Saturation arithmetic:**
- Attacker sends 500 req/s (the global rate-limit ceiling).
- Each request consumes 1,500 gas tokens.
- 500 × 1,500 = **750,000 tokens/s** = exact bucket capacity.
- Result: both the `rateLimitBucket` (all 500 slots) and the `gasLimitBucket` (all 750,000 tokens) are drained within the first second and kept empty as long as the attacker sustains the rate.

**Why the restore path does not help:**
`throttleManager.restore(request.getGas())` (line 48 of `ContractController.java`) is only called when `InvalidParametersException` is thrown (gas > `maxGasLimit`). Requests with `gas = 15_000_000` (the default, within limits) never trigger a restore, so consumed tokens are permanently lost until the bucket's 1-second refill cycle.

**No per-IP or per-identity check exists anywhere in the throttle path.** The `throttle()` method receives only a `ContractCallRequest` with no caller identity, and the bucket beans are singletons.

### Impact Explanation
Any unauthenticated attacker can render `POST /api/v1/contracts/call` completely unavailable for all other users. Every legitimate request will receive `ThrottleException("Gas per second rate limit exceeded.")` or `ThrottleException("Requests per second rate limit exceeded")` for as long as the attack is sustained. Because the mirror node's `/contracts/call` endpoint is the primary EVM simulation surface, this constitutes a full availability outage for that API with zero economic cost to the attacker.

### Likelihood Explanation
No credentials, API keys, or special network position are required. A single machine with a modest HTTP client (e.g., `wrk`, `ab`, or a simple async script) can sustain 500 req/s. The attack is trivially repeatable, stateless, and requires no knowledge of contract addresses or ABI — an empty body with the default gas value is sufficient.

### Recommendation
1. **Per-IP (or per-client) rate limiting:** Replace or augment the global buckets with per-source-IP buckets (e.g., using Bucket4j's `ProxyManager` backed by a local or distributed cache). Each IP should have its own `rateLimitBucket` and `gasLimitBucket` with a fraction of the global capacity.
2. **Global cap + per-IP cap:** Keep the global bucket as a hard ceiling but add a per-IP sub-limit so no single client can consume more than `globalCapacity / N` tokens.
3. **Restore on all non-execution paths:** Extend the `restore()` call to cover any exception thrown after `throttle()` but before actual EVM execution completes, so speculative gas consumption is not permanently lost.

### Proof of Concept
```bash
# Drain both global buckets from a single unauthenticated client
# Requires: wrk or any HTTP load tool capable of 500 req/s

cat > payload.json <<'EOF'
{"gas": 15000000, "to": "0x0000000000000000000000000000000000000001"}
EOF

wrk -t4 -c50 -d30s \
    -s <(echo '
      wrk.method = "POST"
      wrk.headers["Content-Type"] = "application/json"
      wrk.body = io.open("payload.json"):read("*a")
    ') \
    http://<mirror-node-host>/api/v1/contracts/call

# Expected result after ~1 second:
# All responses from OTHER concurrent clients return HTTP 429 with:
# {"message":"Gas per second rate limit exceeded."}
# or
# {"message":"Requests per second rate limit exceeded"}
```
