### Title
Unbounded Brute-Force Attack on Basic Auth Endpoint Due to Missing Rate Limiting in `authHandler`

### Summary
The `authHandler` middleware in `rest/middleware/authHandler.js` processes HTTP Basic Auth credentials on every request with no rate limiting, failed-attempt counter, IP-based lockout, or account throttling. The Traefik ingress for the REST service also omits any `rateLimit` or `inFlightReq` middleware, unlike other services in the same codebase. An unauthenticated external attacker can therefore submit an unlimited number of credential guesses against any known username to recover the plaintext password.

### Finding Description
**Exact code path:**

`rest/middleware/authHandler.js`, lines 15–36 — the `authHandler` function:
- Parses the `Authorization: Basic <base64>` header via `basicAuth(req)`.
- Calls `findUser(credentials.name, credentials.pass)` which iterates `config.users` and compares with `tsscmp` (constant-time string compare).
- On mismatch, returns HTTP 401 with `{"_status":{"messages":[{"message":"Invalid credentials"}]}}`.
- On match, sets a per-user query-limit in `httpContext`.

There is **no counter of failed attempts**, **no per-IP or per-username delay**, **no lockout**, and **no CAPTCHA** anywhere in this function or in any middleware registered before it in `rest/server.js` (lines 68–86).

**Infrastructure layer — also absent:**

`charts/hedera-mirror-rest/values.yaml`, lines 134–139, defines the Traefik middleware chain for the REST ingress:
```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.25 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 10
      initialInterval: 100ms
```
No `rateLimit` or `inFlightReq` entry exists here. Compare with `charts/hedera-mirror-rosetta/values.yaml` lines 157–161, which **does** include:
```yaml
  - rateLimit:
      average: 10
      sourceCriterion:
        requestHost: true
  - inFlightReq:
      amount: 5
```
The REST service is the only one that omits these controls.

**Root cause:** The design assumes that credential enumeration is not a threat, or that an external WAF/proxy will handle it. Neither assumption is enforced or documented, and the default deployment has no such control.

### Impact Explanation
Successful credential recovery grants the attacker an authenticated session with an elevated API query limit (the `user.limit` value set in `httpContext`). This allows bulk data extraction from the Hedera mirror node (accounts, transactions, tokens, contracts, NFTs) at rates unavailable to anonymous users. The confidentiality of the credential itself is also compromised, and if the same password is reused elsewhere the blast radius widens. Severity: **Medium–High** (authentication bypass via offline-equivalent online attack).

### Likelihood Explanation
Preconditions are minimal: the attacker needs only network access to the public REST API endpoint (exposed via the ingress at `/api/v1`). Usernames are the only unknown; in practice they are short identifiers like `testuser`, `premium`, `txuser` (as seen in test fixtures). A dictionary or credential-stuffing attack against a small, static user list stored in a Kubernetes secret is highly feasible. The `retry: attempts: 10` Traefik middleware actually **amplifies** the attack by automatically retrying failed requests up to 10 times per attacker request. No special tooling beyond `curl` or `hydra` is required.

### Recommendation
1. **Application layer:** Add a per-IP and per-username failed-attempt counter (e.g., using `express-rate-limit` + an in-memory or Redis store) inside `authHandler` before the `findUser` call. After N failures (e.g., 5) within a window, return 429 and impose a back-off delay.
2. **Infrastructure layer:** Add `rateLimit` and `inFlightReq` Traefik middleware entries to `charts/hedera-mirror-rest/values.yaml`, mirroring the rosetta chart configuration.
3. **Remove the `retry` amplifier** for 4xx responses — the Traefik `retry` middleware should not retry client authentication errors.
4. Consider moving credentials out of static config into a proper identity provider with built-in lockout policies.

### Proof of Concept
```bash
# 1. Identify a known username (e.g., from leaked config or default fixtures)
USERNAME="testuser"

# 2. Iterate passwords from a wordlist — no delay, no lockout
while IFS= read -r password; do
  CREDS=$(echo -n "${USERNAME}:${password}" | base64)
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
    -H "Authorization: Basic ${CREDS}" \
    https://<mirror-node-host>/api/v1/transactions?limit=1)
  if [ "$STATUS" = "200" ]; then
    echo "[+] Valid credentials: ${USERNAME}:${password}"
    break
  fi
done < /usr/share/wordlists/rockyou.txt

# 3. On success, use the recovered credentials to extract bulk data
curl -H "Authorization: Basic $(echo -n "${USERNAME}:${password}" | base64)" \
  "https://<mirror-node-host>/api/v1/transactions?limit=100"
```
Each iteration receives either HTTP 200 (valid) or HTTP 401 (invalid) with no throttling, no lockout, and no observable side-effect on the server side. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```javascript
const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;
  }

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
```
