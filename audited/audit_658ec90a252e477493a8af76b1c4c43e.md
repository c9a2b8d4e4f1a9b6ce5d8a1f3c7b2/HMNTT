### Title
Unbounded Prometheus Label Cardinality via Crafted Route Paths in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates route-label extraction to the gorilla/mux `RouteMatcher`. When an incoming request does not match any registered route, the weaveworks `middleware.Instrument` falls back to using the raw `r.URL.Path` as the `route` label value. Because no cardinality cap, normalization, or rate-limiting exists, an unprivileged attacker can flood the server with requests to unique paths, creating an unbounded number of distinct label combinations in every registered `HistogramVec` and `GaugeVec`, causing monotonically growing memory consumption in the Prometheus default registry.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83 — `MetricsMiddleware()`:
```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,   // labels: method, route, status_code, ws
        InflightRequests: requestInflightGauge,       // labels: method, route
        RequestBodySize:  requestBytesHistogram,      // labels: method, route
        ResponseBodySize: responseBytesHistogram,     // labels: method, route
        RouteMatcher:     next.(middleware.RouteMatcher),
    }.Wrap(next)
}
```

**Root cause:**
The `weaveworks/common/middleware.Instrument` wrapper calls `RouteMatcher.MatchesRoute(r)` (backed by gorilla/mux `Router.Match`) before the request is dispatched. For any path that does not match a registered Rosetta route, `Match` returns `false` and `mux.CurrentRoute(r)` returns `nil`. The weaveworks library then falls back to `r.URL.Path` as the `route` label value. There is no normalization, truncation, or cardinality guard applied anywhere in the middleware chain.

**Exploit flow:**
1. Attacker sends HTTP POST (or GET) requests to paths that do not match any registered Rosetta route: `/a1`, `/a2`, `/a3`, … `/aN`.
2. For each request, `RouteMatcher.MatchesRoute(r)` returns `false`.
3. The weaveworks `Instrument` records metrics using `route = r.URL.Path` (e.g., `/a1`, `/a2`, …).
4. Each unique path causes `prometheus.HistogramVec.With(labels)` to allocate a new set of internal metric objects (one per bucket + sum + count) in the Prometheus default registry.
5. The registry never garbage-collects label combinations; memory grows monotonically.

**Why existing checks fail:**
- The gorilla/mux router (from `rosetta-sdk-go v0.11.0`) correctly returns 404 for unmatched routes, but this happens *after* the weaveworks middleware has already recorded the metric with the raw path label.
- There is no rate-limiting, path-length cap, or label-cardinality guard anywhere in the middleware stack (`rosetta/main.go` lines 217–219).
- The Rosetta API is publicly accessible with no authentication on the instrumented port.

### Impact Explanation
Each unique path creates approximately 23 new time-series objects across the four registered metric vectors (6 duration buckets + 2 counters per histogram × 3 histograms + 1 gauge = ~23 objects). Each object carries label maps and float64 bucket arrays. At 10,000 unique paths the registry holds ~230,000 time-series objects; at 100,000 unique paths this reaches ~2.3 million objects. A typical Rosetta process baseline is 50–100 MB; 100,000 unique label combinations easily adds several hundred MB, exceeding the 30% threshold. Because the registry is a global singleton (`prometheus.DefaultRegisterer`), the memory is never reclaimed for the lifetime of the process, making this a permanent resource exhaustion condition.

### Likelihood Explanation
No privileges are required. The Rosetta API is designed to be internet-facing for exchange integrations. Generating 10,000–100,000 unique paths requires only a trivial script (e.g., `for i in $(seq 1 100000); do curl -s http://target:8080/x$i & done`). The attack is repeatable, requires no authentication, no valid request body, and produces a 404 response — meaning the attacker receives immediate confirmation that each probe was processed. The attack can be sustained indefinitely since the server never rejects requests based on path novelty.

### Recommendation
1. **Normalize the route label for unmatched requests**: Override the route label to a fixed sentinel value (e.g., `"unmatched"` or `"unknown"`) when `RouteMatcher.MatchesRoute(r)` returns `false`, instead of allowing the raw URL path to propagate.
2. **Wrap the weaveworks instrument with a custom handler** that intercepts the route label before it reaches the histogram, replacing any value not in a known allowlist with `"other"`.
3. **Add a cardinality-limiting wrapper** around the `HistogramVec` registrations that panics or drops observations once the number of distinct label combinations exceeds a configured threshold.
4. **Deploy a rate-limiting / WAF layer** in front of the Rosetta port to reject requests with novel paths at the network edge.

### Proof of Concept
```bash
# Prerequisites: rosetta node running on localhost:8080, no auth required
# Step 1: Establish baseline memory
curl -s http://localhost:8080/metrics | grep process_resident_memory_bytes

# Step 2: Flood with unique paths (adjust count as needed)
for i in $(seq 1 50000); do
  curl -s -o /dev/null -X POST http://localhost:8080/unique_path_$i \
    -H "Content-Type: application/json" -d '{}'
done

# Step 3: Observe cardinality explosion in Prometheus metrics
curl -s http://localhost:8080/metrics | grep hiero_mirror_rosetta_request_duration | wc -l
# Expected: thousands of unique route label combinations

# Step 4: Observe memory growth
curl -s http://localhost:8080/metrics | grep process_resident_memory_bytes
# Expected: significantly higher than baseline (>30%)
```
