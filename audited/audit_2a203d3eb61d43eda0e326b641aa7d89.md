### Title
Missing JDBC Socket Timeout on Runtime Connection Pool Enables Thread Exhaustion via Network Partition

### Summary
The `rest-java` service's HikariCP runtime connection pool does not configure a JDBC-level `socketTimeout`. When a firewall silently drops packets on an established database TCP connection, the JDBC socket read in `TopicServiceImpl.findById()` blocks indefinitely — far beyond any application-level timeout — holding a Tomcat thread for the duration. An unprivileged external user can amplify this by flooding `GET /api/v1/topics/{id}` during a partition, exhausting the thread pool and causing a denial of service.

### Finding Description

**Exact code path:**

`TopicController.getTopic()` (line 32–36) calls three sequential blocking JPA repository operations with no timeout annotation: [1](#0-0) 

Each call flows into `TopicServiceImpl.findById()`: [2](#0-1) 

This is a plain Spring Data JPA `findById()` — no `@QueryHints`, no `@Transactional(timeout=...)`, no `Statement.setQueryTimeout()`.

**Root cause — `socketTimeout` is startup-only, never applied to the runtime pool:**

`DatabaseWaiter.waitForDatabase()` sets `socketTimeout=2` and `connectTimeout=2` only on the temporary `DriverManager.getConnection()` probe used at startup: [3](#0-2) 

These properties are passed to a one-shot `DriverManager` connection and are **never copied into the HikariCP pool** that serves live requests. The runtime `HikariDataSource` is built from `HikariConfig` populated via `@ConfigurationProperties("spring.datasource.hikari")`: [4](#0-3) [5](#0-4) 

No `socketTimeout` JDBC URL parameter is injected into this pool anywhere in the rest-java module.

**Why existing mitigations fail:**

1. **`statementTimeout = 10000` (10 s, server-side PostgreSQL):** This is enforced by the PostgreSQL server. When a firewall silently drops packets, the client's TCP socket blocks waiting for a response. If packets are dropped before reaching the server, the server never receives the query and never fires its timeout. If packets are dropped on the return path, the server times out and closes its side, but the client socket is still blocked reading — the RST/FIN packets are also dropped. Either way, the client thread remains blocked.

2. **`QueryTimeoutException` handler in `GenericControllerAdvice`:** This handler (line 121–124) only fires if the JDBC driver raises a query timeout exception, which requires `Statement.setQueryTimeout()` to have been called. Without a socket-level timeout, the JDBC driver never raises this exception during a silent partition. [6](#0-5) 

3. **HikariCP `connectionTimeout`:** This governs how long a caller waits to *acquire* a connection from the pool, not how long a query on an already-acquired connection may block. It provides no protection once the connection is checked out.

### Impact Explanation
Each `GET /api/v1/topics/{id}` request that hits a partitioned connection holds one Tomcat worker thread indefinitely (until OS TCP keepalive fires — Linux default: ~2 hours). The rest-java service uses a bounded thread pool. With enough concurrent requests during a partition, all threads are exhausted, new requests queue and time out, and the service becomes fully unavailable. This is a complete availability loss (DoS) for the public-facing API.

### Likelihood Explanation
The endpoint is unauthenticated and publicly accessible. No rate limiting or authentication is required. A network partition can occur naturally (cloud provider incident, misconfigured firewall rule, BGP flap) or be induced by an attacker with access to network infrastructure between the app and DB. Once a partition exists, any external user sending concurrent requests to `/api/v1/topics/{id}` will trigger thread exhaustion. The attack is trivially repeatable with a simple HTTP load tool (e.g., `ab`, `wrk`, `curl` in a loop).

### Recommendation
1. **Set `socketTimeout` in the HikariCP JDBC URL for the runtime pool.** For PostgreSQL (pgjdbc), add `?socketTimeout=30` (seconds) to the JDBC URL, or set it via `spring.datasource.hikari.data-source-properties.socketTimeout=30` in `application.yml`. This causes the JDBC driver to raise a `SocketTimeoutException` after the configured interval on any blocked read, regardless of server reachability.
2. **Set `connectTimeout` in the JDBC URL** for the same pool to bound new connection establishment.
3. **Add `@Transactional(timeout = N)` or `@QueryHints`** on repository methods as a defense-in-depth layer, so Hibernate raises `QueryTimeoutException` even if the socket timeout is misconfigured.
4. **Configure HikariCP `keepaliveTime`** to detect dead connections in the pool before they are handed to request threads.

### Proof of Concept
1. Deploy the rest-java service with default configuration.
2. Establish a firewall rule (e.g., `iptables -I FORWARD -p tcp --dport 5432 -j DROP`) that silently drops packets on the DB port **after** the HikariCP pool has established its connections (so existing sockets are affected, not new ones).
3. Send concurrent requests: `wrk -t10 -c100 -d120s http://<host>/api/v1/topics/0.0.1000`
4. Observe: Tomcat thread pool fills up (via `/actuator/metrics/executor.active`), new requests begin returning 503 or timing out at the HTTP layer, and the service is effectively unavailable for the duration of the partition.
5. Remove the firewall rule; threads remain blocked until OS TCP keepalive fires (default ~2 hours on Linux) unless the JVM is restarted.
