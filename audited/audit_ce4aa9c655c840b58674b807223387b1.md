### Title
`findByBlockType()` Reference Equality Bypass Causes `findByIndex(0)` Instead of `findEarliest()` on Migrated Nodes

### Summary
`findByBlockType()` uses Java reference equality (`==`) to detect `BlockType.EARLIEST`, but `BlockType.of("0")` and `BlockType.of("0x0")` produce a new `BlockType("0", 0L)` object that is never the `EARLIEST` singleton. This causes the fallthrough path `findByIndex(0)` to be called. On MAINNET and TESTNET deployments where `BlockNumberMigration.updateIndex()` has shifted all record file indices (e.g., to ~34M on mainnet), no record file has `index=0`, so the query returns empty — silently hiding the actual earliest block from any caller.

### Finding Description

**Code location:** `web3/src/main/java/org/hiero/mirror/web3/service/RecordFileServiceImpl.java`, lines 19–27.

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (block == BlockType.EARLIEST) {          // line 20 — reference equality only
        return recordFileRepository.findEarliest();
    } else if (block == BlockType.LATEST) {
        return recordFileRepository.findLatest();
    }
    return recordFileRepository.findByIndex(block.number());  // line 26
}
```

**Root cause — singleton identity vs. value semantics:**

`BlockType` is a Java `record`: [1](#0-0) 

`EARLIEST` is defined as `new BlockType("earliest", 0L)`. When a user supplies `"0"` or `"0x0"`, `BlockType.of()` hits the `default` branch: [2](#0-1) 

`extractNumericBlock("0")` returns `new BlockType("0", 0L)` — a distinct heap object with `name="0"`, not `name="earliest"`. The `==` check on line 20 therefore fails (both reference identity and record `.equals()` fail because the `name` fields differ), and execution falls through to `findByIndex(0)`. [3](#0-2) 

**Why `findByIndex(0)` diverges from `findEarliest()` after migration:**

`findEarliest()` is: [4](#0-3) 

`findByIndex(long index)` is: [5](#0-4) 

`BlockNumberMigration` runs on MAINNET and TESTNET and calls `updateIndex(offset)`: [6](#0-5) 

`updateIndex` executes: [7](#0-6) 

This shifts every record file's `index` by a large positive offset (e.g., ~34,305,852 on MAINNET, ~22,384,256 on TESTNET). After migration, no row has `index = 0`. `findByIndex(0)` returns `Optional.empty()`. `findEarliest()` still returns the correct earliest block (smallest index after shift).

The test data confirms the pre-migration state starts at index 0: [8](#0-7) 

### Impact Explanation

Any caller of `findByBlockType(BlockType.of("0"))` or `findByBlockType(BlockType.of("0x0"))` on a migrated MAINNET/TESTNET node receives `Optional.empty()` instead of the actual earliest block. This affects all JSON-RPC methods that accept a block number parameter (e.g., `eth_getBlockByNumber`, `eth_call`, `eth_getBalance` at block 0). Applications that use block 0 as the genesis/anchor block receive a silent null/not-found response, causing incorrect historical state queries and potential downstream failures in EVM simulation paths that depend on the genesis block context.

### Likelihood Explanation

No privileges are required. Any user of the public JSON-RPC endpoint can trigger this by passing `"0x0"` or `"0"` as the block parameter. MAINNET and TESTNET are confirmed affected deployments (the migration mapping is hardcoded). The condition is permanent post-migration and repeatable on every request. [9](#0-8) 

### Recommendation

Replace the reference equality checks with semantic equality. The simplest correct fix in `findByBlockType()`:

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (BlockType.EARLIEST.equals(block) || "earliest".equalsIgnoreCase(block.name())) {
        return recordFileRepository.findEarliest();
    } else if (BlockType.LATEST.equals(block)) {
        return recordFileRepository.findLatest();
    }
    return recordFileRepository.findByIndex(block.number());
}
```

Alternatively, normalize in `BlockType.of()`: when the parsed numeric value equals `0`, return the `EARLIEST` singleton directly instead of creating a new object. This ensures the `==` check in `findByBlockType()` works as intended.

### Proof of Concept

**Precondition:** MAINNET or TESTNET mirror node where `BlockNumberMigration` has run (standard production state). All record file indices are ≥ 34,305,852 (mainnet) or ≥ 22,384,248 (testnet).

**Trigger:**
```bash
# Call eth_getBlockByNumber with block number 0 (hex)
curl -X POST http://<mirror-node>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["0x0",false],"id":1}'
# Result: null / not found

# Compare with "earliest" — returns the actual earliest block
curl -X POST http://<mirror-node>:8545 \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["earliest",false],"id":2}'
# Result: actual earliest block data
```

**Code trace:**
1. `"0x0"` → `BlockType.of("0x0")` → `extractNumericBlock("0x0")` → `new BlockType("0x0", 0L)` (not EARLIEST singleton)
2. `findByBlockType(new BlockType("0x0", 0L))` → `block == BlockType.EARLIEST` is `false`
3. → `recordFileRepository.findByIndex(0)` → `SELECT r FROM RecordFile r WHERE r.index = 0` → empty result set
4. Caller receives `Optional.empty()` → JSON-RPC returns `null`
