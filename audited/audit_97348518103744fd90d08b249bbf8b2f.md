### Title
Unauthenticated Single-Request Resource Exhaustion via Uncapped Full-Mode EVM Re-Execution on Opcodes Endpoint

### Summary
The `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint allows any unauthenticated caller to simultaneously enable all three expensive tracing modes (`stack=true&memory=true&storage=true`) against any historical transaction. The sole throttle is a global count-based rate limiter (1 RPS) that consumes a flat 1 token regardless of request cost, and no gas cap is applied to the re-execution — unlike the `/call` endpoint which enforces both gas-based throttling and a `maxGasLimit` ceiling. A single well-crafted request targeting a historically complex transaction triggers a full EVM re-execution with per-opcode capture of stack, memory, and storage into an unbounded in-memory list, causing sustained CPU and heap pressure from one request.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (lines 52–65) accepts `stack`, `memory`, `storage` as optional booleans with no restriction on combining them:

```java
@RequestParam(required = false, defaultValue = "true") boolean stack,
@RequestParam(required = false, defaultValue = "false") boolean memory,
@RequestParam(required = false, defaultValue = "false") boolean storage,
```

The only guards before EVM re-execution are:
1. `validateAcceptEncodingHeader` — trivially satisfied with `Accept-Encoding: gzip`.
2. `throttleManager.throttleOpcodeRequest()` — consumes exactly 1 token from `opcodeRateLimitBucket`.

**Root cause — flat-cost throttle with no gas cap:**

`ThrottleManagerImpl.throttleOpcodeRequest()` (lines 52–56) unconditionally consumes 1 token regardless of which tracing modes are requested or how complex the target transaction is:

```java
public void throttleOpcodeRequest() {
    if (!opcodeRateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```

The bucket is configured with a default capacity of `opcodeRequestsPerSecond = 1` (a global, not per-IP, bucket). This means the attacker is guaranteed one full EVM re-execution per second.

Contrast with `ContractController.call()` (lines 38–51), which enforces **both** a count-based throttle **and** a gas-based throttle (`gasLimitBucket.tryConsume(scaleGas(request.getGas()))`) **and** a hard `validateContractMaxGasLimit` ceiling (`maxGasLimit = 15,000,000`). The opcodes endpoint has none of these additional protections.

**No gas cap on re-execution:**

`OpcodeServiceImpl.processOpcodeCall()` (line 66–67) derives the gas limit directly from the stored historical transaction record with no ceiling:

```java
final var params = buildCallServiceParameters(opcodeRequest.getTransactionIdOrHashParameter());
final var opcodeContext = new OpcodeContext(opcodeRequest, (int) params.getGas() / 3);
```

A historical transaction that originally used the network's maximum gas (e.g., 15M+ gas) is replayed at full scale.

**Per-opcode cost with all three modes:**

`OpcodeActionTracer.tracePostExecution()` (lines 44–62) is called for every single EVM opcode. With `stack=true`, `memory=true`, `storage=true`, each invocation captures:
- Stack: up to 1024 × 32-byte items serialized to hex strings
- Memory: entire EVM memory (potentially megabytes) split into 32-byte words
- Storage: all touched storage slots (key-value pairs)

All captured `Opcode` objects are accumulated in an unbounded `ArrayList` (`OpcodeContext.opcodes`). A complex transaction with hundreds of thousands of opcodes and large memory/storage footprints produces a massive in-memory structure that must be serialized to JSON before the response is sent.

### Impact Explanation

A single request with `stack=true&memory=true&storage=true` targeting a known high-gas historical transaction (e.g., a complex DeFi interaction or contract deployment near the gas limit) forces the server to:
- Execute the full EVM re-execution loop (CPU-bound, proportional to gas used)
- Capture and heap-allocate stack+memory+storage snapshots at every opcode step
- Serialize the entire result to JSON (additional CPU + memory)

The server's own documentation acknowledges this: *"the transaction needs to be re-executed on the EVM, which may take a significant amount of time to complete if stack and memory information is requested."* The `requestTimeout` of 10 seconds means a single request can hold a server thread and heap allocation for the full timeout window. This directly satisfies the >30% resource consumption threshold for a single request against a complex target transaction, with no authentication required.

### Likelihood Explanation

Preconditions are minimal: the endpoint must be enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`), and the attacker needs only a publicly known transaction hash from any block explorer. No credentials, no special network access, no brute force. The global 1-RPS throttle does not prevent the attack — it merely serializes it. The attacker reliably gets one maximally expensive request per second processed. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Apply gas-based throttle cost to opcodes requests**: Charge the `gasLimitBucket` proportionally to the gas of the replayed transaction, mirroring the `/call` endpoint's `gasLimitBucket.tryConsume(scaleGas(gas))` pattern.
2. **Enforce a gas cap on re-execution**: In `OpcodeServiceImpl.processOpcodeCall()`, cap `params.getGas()` to `evmProperties.getMaxGasLimit()` before passing to `ContractDebugService`, consistent with `ContractController.validateContractMaxGasLimit()`.
3. **Differentiate throttle cost by active tracing modes**: Charge additional tokens when `memory=true` or `storage=true` are requested, since these modes multiply per-opcode work significantly.
4. **Add a per-IP rate limit** in addition to the global bucket to prevent a single client from monopolizing the global 1-RPS allowance.

### Proof of Concept

```
# Step 1: Identify a historically complex transaction hash from a block explorer
# (e.g., a high-gas DeFi interaction near the 15M gas limit)
TX_HASH="0x<known_complex_transaction_hash>"

# Step 2: Send a single maximally expensive request
curl -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" \
  --output /dev/null

# Result: Server performs full EVM re-execution with all three tracing modes,
# capturing stack+memory+storage at every opcode step into an unbounded ArrayList,
# causing sustained CPU spike and heap allocation for the duration of the request
# (up to the 10-second requestTimeout), from a single unauthenticated HTTP GET.

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
