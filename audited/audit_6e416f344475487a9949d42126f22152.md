### Title
Hardcoded `sslmode=disable` in Rosetta DB Connection Enables Plaintext MITM Interception of Staking Reward Data

### Summary
The `GetDsn()` function in `rosetta/app/config/types.go` hardcodes `sslmode=disable` in the PostgreSQL DSN with no configurable override. `ConnectToDb()` in `rosetta/app/db/db.go` passes this DSN directly to GORM, meaning all traffic between the Rosetta service and PostgreSQL — including staking reward queries — is transmitted in plaintext. An attacker with local network access (same L2 segment, e.g., a co-tenant VM, compromised sidecar container, or cloud VPC peer) can ARP-poison the path and intercept, modify, or selectively drop DB responses without any privileges on the target hosts.

### Finding Description
**Exact code location:**

- `rosetta/app/config/types.go`, `GetDsn()`, line 51: `sslmode=disable` is hardcoded as a string literal.
- `rosetta/app/db/db.go`, `ConnectToDb()`, line 18: `dbConfig.GetDsn()` is passed directly to `postgres.Open()` with no TLS override.

**Root cause:** The `Db` struct (lines 39–47 of `types.go`) has no `SslMode` or `TlsConfig` field. There is no mechanism — neither config file nor environment variable — to enable TLS for this connection. The `sslmode=disable` is a compile-time constant embedded in the format string.

**Exploit flow:**
1. Attacker gains presence on the same L2 network segment as the Rosetta service and PostgreSQL host (e.g., another container in the same pod network, a co-tenant in a cloud VPC, or a compromised adjacent service).
2. Attacker sends gratuitous ARP replies to poison the ARP cache of both the Rosetta host and the PostgreSQL host, redirecting traffic through the attacker's machine.
3. Because `sslmode=disable` is hardcoded, the PostgreSQL wire protocol flows in plaintext (no TLS handshake, no certificate validation).
4. Attacker uses a tool (e.g., `ettercap`, `arpspoof` + `tcpflow`, or a custom proxy) to intercept the TCP stream on port 5432.
5. Attacker selectively drops or corrupts PostgreSQL response packets for staking reward queries (e.g., queries against `staking_reward_transfer` or `entity_stake` tables), causing the Rosetta API to return empty or incorrect reward data to callers.

**Why existing checks fail:** There are no checks. The `Db` struct has no SSL field, `GetDsn()` has no conditional SSL logic, and `ConnectToDb()` performs no post-connection TLS verification. The test in `types_test.go` line 24 explicitly asserts the DSN contains `sslmode=disable`, confirming this is intentional and tested behavior — meaning it will not be caught by CI.

### Impact Explanation
An attacker can silently corrupt staking reward query results returned by the Rosetta API. Consumers of the Rosetta API (exchanges, wallets, block explorers) would receive falsified or missing staking reward data, leading to incorrect balance reporting. Additionally, the plaintext stream exposes the DB password (`mirror_rosetta_pass` by default per docs line 655), enabling full database credential theft and subsequent direct DB access.

### Likelihood Explanation
In cloud-native deployments (Kubernetes, ECS, GKE), containers sharing a node or VPC subnet are on the same L2 segment. ARP poisoning within a pod network or a flat cloud VPC requires no OS-level privileges on the target — only network reachability. Tools like `arpspoof` run as a normal user process. The default host `127.0.0.1` (docs line 653) means loopback-only deployments are safe, but any deployment where Rosetta and PostgreSQL are on separate hosts (the common production case) is exposed. The hardcoded `sslmode=disable` with no override makes this permanently exploitable regardless of operator intent.

### Recommendation
1. Add an `SslMode string` field to the `Db` struct in `rosetta/app/config/types.go` with a secure default (e.g., `"require"` or `"verify-full"`).
2. Update `GetDsn()` to use `db.SslMode` instead of the hardcoded `"disable"`.
3. Add `SslRootCert string` to the struct to support certificate pinning (`sslmode=verify-full`).
4. Update the default `application.yml` to document and default to `sslmode=require`.
5. Remove the test assertion that hardcodes `sslmode=disable` as the expected value.

### Proof of Concept
```
# Precondition: attacker is on same L2 as rosetta-host (10.0.0.10) and pg-host (10.0.0.20)

# Step 1: Enable IP forwarding
echo 1 > /proc/sys/net/ipv4/ip_forward

# Step 2: ARP poison both hosts (no root needed on many Linux distros with CAP_NET_RAW)
arpspoof -i eth0 -t 10.0.0.10 10.0.0.20 &
arpspoof -i eth0 -t 10.0.0.20 10.0.0.10 &

# Step 3: Capture plaintext PostgreSQL wire protocol (port 5432)
tcpdump -i eth0 -A 'tcp port 5432' 2>/dev/null | grep -i "staking_reward"
# => Plaintext SQL queries and result rows visible, including DB password in startup packet

# Step 4: Selectively drop staking reward responses
iptables -I FORWARD -p tcp --sport 5432 -m string \
  --string "staking_reward" --algo bm -j DROP
# => Rosetta API now returns empty reward data for all staking reward queries
