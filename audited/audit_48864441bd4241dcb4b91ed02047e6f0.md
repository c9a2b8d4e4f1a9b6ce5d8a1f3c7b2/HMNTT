### Title
Unbounded `account_aliases` List in `/construction/metadata` Causes N DB Queries Per Request (Resource Exhaustion)

### Summary
The `resolveAccountAliases()` function in `rosetta/app/services/construction_service.go` iterates over a comma-separated `account_aliases` string from the unauthenticated `/construction/metadata` request with no limit on the number of aliases. Each valid alias triggers a synchronous, uncached `GetAccountId()` database query. An attacker supplying a large list of valid aliases in a single request can amplify DB load far beyond the 30% threshold with no privilege required.

### Finding Description
**Exact code path:**

`ConstructionMetadata()` (line 159) calls `resolveAccountAliases()` (lines 557–589 of `rosetta/app/services/construction_service.go`):

```go
for accountAlias := range strings.SplitSeq(accountAliases, ",") {
    accountId, err := types.NewAccountIdFromString(accountAlias, c.systemShard, c.systemRealm)
    if err != nil {
        return emptyResult, errors.ErrInvalidAccount
    }
    found, rErr := c.accountRepo.GetAccountId(ctx, accountId)
    if rErr != nil {
        return emptyResult, rErr   // stops on first DB error
    }
    accountMap = append(accountMap, fmt.Sprintf("%s:%s", accountAlias, found))
}
```

`GetAccountId()` (lines 138–159 of `rosetta/app/persistence/account.go`) executes a raw SQL query for every alias-type account:

```go
if err := db.Raw(selectCurrentCryptoEntityByAlias, sql.Named("alias", accountId.GetAlias())).First(&entity).Error; err != nil {
```

**Root cause:** No upper bound is enforced on the number of comma-separated aliases. The `constructionAPIService` uses a plain `accountRepository` with **no caching** (the LRU entity cache exists only in `blockAPIService`). Every alias-type entry in the list produces one synchronous DB round-trip.

**Failed assumption:** The design assumes `account_aliases` will contain at most one alias (the payer), as set by `ConstructionPreprocess()` (line 314 of `construction_service.go`). Nothing in the server enforces this assumption for externally crafted requests.

**Exploit flow:**
1. Attacker enumerates valid alias-format account IDs from the public Hedera ledger (mainnet/testnet mirror REST API exposes them).
2. Attacker crafts a POST to `/construction/metadata` with `account_aliases` set to a comma-separated list of N valid aliases.
3. `resolveAccountAliases()` iterates all N aliases, issuing N sequential DB queries against the `entity` table.
4. The loop only aborts on the first `ErrAccountNotFound`; with all-valid aliases it runs to completion.
5. Repeated or concurrent requests (from multiple source IPs) multiply the DB load linearly.

**Why existing checks are insufficient:**
- The only input validation is `types.NewAccountIdFromString()` (format check) and an `IsOnline()` guard — neither limits alias count.
- Rate limiting (`average: 10`, `inFlightReq: 5`) exists only as Traefik Helm chart defaults; it is **not enforced at the application layer** and is absent in bare deployments.
- HTTP `ReadTimeout` of 5 s still allows a body containing hundreds to thousands of short alias strings to be received.
- The DB `statementTimeout` of 20 s limits individual query duration but not query count per request.

### Impact Explanation
A single request with 500 valid aliases issues 500 sequential DB queries. At the Traefik default of 10 req/s (if deployed), that is 5,000 DB queries/s from one attacker IP. From multiple IPs the multiplier is unbounded. This exhausts DB connection pool slots (default `maxOpenConnections: 100`) and drives CPU on both the Go process and the PostgreSQL server well above the 30% baseline threshold, degrading or denying service to legitimate users.

### Likelihood Explanation
The endpoint requires no authentication. Valid alias strings are publicly observable on mainnet and testnet via the Hedera mirror node REST API. The attack requires only HTTP access and a list of known aliases — no special tooling or insider knowledge. It is trivially repeatable and scriptable.

### Recommendation
1. **Enforce an alias count limit** in `resolveAccountAliases()`: reject requests where `strings.Count(accountAliases, ",") + 1` exceeds a small constant (e.g., 10).
2. **Add application-level rate limiting** (e.g., `golang.org/x/time/rate`) on the `/construction/metadata` handler, independent of infrastructure.
3. **Batch the alias lookup** into a single `WHERE alias = ANY(...)` query instead of N individual queries.
4. **Cache alias→accountId results** in `constructionAPIService` analogously to the LRU cache in `blockAPIService`.

# 2. Send a single /construction/metadata request with all aliases

    \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
