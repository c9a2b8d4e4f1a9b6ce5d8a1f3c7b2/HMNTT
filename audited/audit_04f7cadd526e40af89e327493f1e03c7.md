### Title
Unbounded Per-Request DB Query in `NetworkStatus()` Enables Unauthenticated Connection-Pool Exhaustion and Stale Block Reporting

### Summary
Every call to `/network/status` unconditionally executes a multi-table JOIN query against PostgreSQL via `addressBookEntryRepo.Entries(ctx)` with no result caching and no application-level rate limiting. An unprivileged attacker can flood this endpoint to exhaust the Rosetta service's DB connection pool, starving legitimate queries and causing the service to report a stale `CurrentBlockIdentifier` while the chain tip advances.

### Finding Description

**Exact code path:**

`rosetta/app/services/network_service.go`, `NetworkStatus()`, line 77:
```go
peers, err := n.addressBookEntryRepo.Entries(ctx)
``` [1](#0-0) 

This unconditionally calls `addressBookEntryRepository.Entries()` in `rosetta/app/persistence/address_book_entry.go` lines 57–83, which issues up to two raw SQL queries with multi-table JOINs (`address_book`, `address_book_entry`, `node`, `address_book_service_endpoint`) and a `string_agg` aggregation on every invocation: [2](#0-1) 

**Root cause / failed assumption:**

There is no in-process cache for `Entries()` results (unlike the entity cache `hiero.mirror.rosetta.cache.entity.maxSize`). There is also no application-level rate limiting — the only rate-limiting mechanism is an **optional** Traefik middleware defined in the Helm chart, gated on `{{ if and .Values.global.middleware .Values.middleware }}`: [3](#0-2) [4](#0-3) 

When this middleware is not deployed (bare-metal, Docker Compose, or Helm with `global.middleware: false`), there is zero throttling between the HTTP listener and the DB.

The DB connection pool is capped at `maxOpenConnections: 100` (default): [5](#0-4) [6](#0-5) 

**Exploit flow:**

1. Attacker sends a continuous stream of concurrent POST requests to `/network/status` (no authentication required, valid `network_identifier` is publicly known).
2. Each request passes the `IsOnline()` check, calls `RetrieveGenesis`, `RetrieveLatest`, and then `Entries()` — three DB queries per request.
3. With 100 concurrent connections all blocked on the `Entries()` JOIN, the Rosetta service's connection pool is saturated.
4. Legitimate `/network/status` calls queue behind the flood; the `statementTimeout` of 20 s means each attacker connection holds a pool slot for up to 20 s.
5. While the pool is saturated, `RetrieveLatest` calls also queue, so the `CurrentBlockIdentifier` returned to legitimate callers reflects a stale snapshot even as the importer continues writing new blocks to the shared PostgreSQL instance.

### Impact Explanation

- **Direct:** Rosetta service becomes unresponsive or extremely slow for all clients; `CurrentBlockIdentifier` diverges from the actual chain tip, breaking any downstream system (e.g., Coinbase Rosetta validator, exchange integrations) that relies on this value for liveness checks.
- **Indirect:** The Rosetta service and the importer share the same PostgreSQL server. Saturating 100 Rosetta connections with long-running aggregation queries increases I/O pressure on the shared server, potentially degrading importer write throughput.
- **Severity rationale:** Classified as Medium/griefing — no funds are at risk, but chain-tip staleness can cause exchange deposit/withdrawal pipelines to halt or misreport finality.

### Likelihood Explanation

- No authentication or API key is required to call `/network/status`.
- The request body is minimal (a valid `network_identifier` JSON object, publicly documented).
- A single attacker with a modest HTTP client (e.g., `wrk`, `hey`, or a simple Go script) can sustain hundreds of concurrent requests from one IP.
- The Traefik middleware (`inFlightReq: 5`, `rateLimit: average: 10`) is only active when explicitly enabled in the Helm deployment; many self-hosted or CI deployments omit it.
- Repeatability is trivial — the attack requires no state and can be resumed immediately after any restart.

### Recommendation

1. **Cache `Entries()` results** in-process with a short TTL (e.g., 60 s), since the address book changes infrequently. The existing `Cache` config type and entity-cache pattern in the codebase provide a template.
2. **Add application-level rate limiting** (e.g., `golang.org/x/time/rate` token bucket) inside the HTTP handler or as a Go middleware, independent of the optional Traefik layer.
3. **Reduce `statementTimeout`** for the `Entries()` query specifically, or set a tighter context deadline, to release pool connections faster under load.
4. **Enforce the Traefik middleware** as a hard deployment requirement (or document it as a mandatory security control) rather than an opt-in.

func (n *networkAPIService) NetworkStatus(
	ctx context.Context,
	_ *rTypes.NetworkRequest,
) (*rTypes.NetworkStatusResponse, *rTypes.Error) {
	if !n.IsOnline() {
		return nil, errors.ErrEndpointNotSupportedInOfflineMode
	}

	genesisBlock, err := n.RetrieveGenesis(ctx)
	if err != nil {
		return nil, err
	}

	currentBlock, err := n.RetrieveLatest(ctx)
	if err != nil {
		return nil, err
	}

	peers, err := n.addressBookEntryRepo.Entries(ctx)
	if err != nil {
		return nil, err
	}

	return &rTypes.NetworkStatusResponse{
		CurrentBlockIdentifier: currentBlock.GetRosettaBlockIdentifier(),
		CurrentBlockTimestamp:  currentBlock.GetTimestampMillis(),
		GenesisBlockIdentifier: genesisBlock.GetRosettaBlockIdentifier(),
		Peers:                  peers.ToRosetta(),
	}, nil
}

func (aber *addressBookEntryRepository) Entries(ctx context.Context) (*types.AddressBookEntries, *rTypes.Error) {
	db, cancel := aber.dbClient.GetDbWithContext(ctx)
	defer cancel()

	nodes := make([]nodeServiceEndpoint, 0)
	// address book file 101 has service endpoints for nodes, resort to file 102 if 101 doesn't exist
	for _, fileId := range []int64{aber.addressBook101.EncodedId, aber.addressBook102.EncodedId} {
		if err := db.Raw(
			latestNodeServiceEndpoints,
			sql.Named("file_id", fileId),
		).Scan(&nodes).Error; err != nil {
			log.Error("Failed to get latest node service endpoints", err)
			return nil, errors.ErrDatabaseError
		}

		if len(nodes) != 0 {
			break
		}
	}

	entries := make([]types.AddressBookEntry, 0, len(nodes))
	for _, node := range nodes {
		entries = append(entries, node.toAddressBookEntry())
	}

	return &types.AddressBookEntries{Entries: entries}, nil
}
