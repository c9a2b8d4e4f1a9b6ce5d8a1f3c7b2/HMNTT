### Title
Unauthenticated Gossip Injection via `ConstructionSubmit` Without Payer Signature Pre-Validation

### Summary
`ConstructionSubmit` in `rosetta/app/services/construction_service.go` forwards any structurally valid hex-encoded transaction directly to the consensus node via `hiero.TransactionExecute` without verifying that the embedded payer signature is cryptographically valid. An unprivileged external user can bypass the `ConstructionCombine` signature-verification gate entirely and submit a transaction carrying an invalid or forged payer signature, causing the consensus node to gossip the transaction before rejecting it with `INVALID_PAYER_SIGNATURE`, while the Rosetta layer surfaces `ErrTransactionSubmissionFailed` (code 118, `retriable=false`).

### Finding Description

**Signature verification only exists in `ConstructionCombine`:**

`rosetta/app/services/construction_service.go` lines 72–84:
```go
for _, signature := range request.Signatures {
    ...
    if !ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes) {
        return nil, errors.ErrInvalidSignatureVerification
    }
    _, _ = hiero.TransactionAddSignature(transaction, pubKey, signature.Bytes)
}
``` [1](#0-0) 

This check is only reachable through the `/construction/combine` endpoint. It is not called from `ConstructionSubmit`.

**`ConstructionSubmit` performs no signature pre-validation before forwarding:**

`rosetta/app/services/construction_service.go` lines 332–368:
```go
func (c *constructionAPIService) ConstructionSubmit(...) {
    if !c.IsOnline() { return nil, errors.ErrEndpointNotSupportedInOfflineMode }

    transaction, rErr := unmarshallTransactionFromHexString(request.SignedTransaction)
    ...
    _, err = hiero.TransactionExecute(transaction, c.sdkClient)   // ← no sig check
    if err != nil {
        return nil, errors.AddErrorDetails(errors.ErrTransactionSubmissionFailed, ...)
    }
}
``` [2](#0-1) 

The only guards are: (1) online-mode check, (2) hex-decode, (3) protobuf unmarshal, (4) supported transaction type check. None of these verify the payer signature embedded in the `SignedTransaction` bytes.

**Error definition — non-retriable:** [3](#0-2) 

**No application-level authentication on the Rosetta server:**

`rosetta/main.go` wires only CORS, metrics, and tracing middleware — no auth or rate-limiting at the Go application layer. [4](#0-3) 

The Helm chart defines an optional Traefik rate-limit of 10 req/host, but this is infrastructure-level and not enforced in code. [5](#0-4) 

**Exploit flow:**
1. Attacker obtains any valid unsigned transaction bytes (e.g., from a prior `/construction/payloads` call, or by constructing a `TransferTransaction` protobuf manually).
2. Attacker injects a random 64-byte blob as the Ed25519 signature in the `SignatureMap` of the serialized `SignedTransaction`.
3. Attacker calls `POST /construction/submit` directly with the crafted hex string — skipping `/construction/combine` entirely.
4. `ConstructionSubmit` deserializes the transaction, computes its hash, and calls `hiero.TransactionExecute`.
5. The Hiero SDK submits the transaction to the consensus node, which gossips it to the network before evaluating it.
6. The consensus node rejects it with `INVALID_PAYER_SIGNATURE` (proto_id 43).
7. `ConstructionSubmit` returns `ErrTransactionSubmissionFailed` (code 118, `retriable=false`).

### Impact Explanation
Every call injects one invalid transaction into the consensus gossip layer. Because gossip propagation precedes signature validation at the consensus node, each crafted request causes network-wide gossip traffic for a transaction that will never be committed. At scale this constitutes a low-cost, unauthenticated DoS vector against the consensus network routed through the mirror node's Rosetta API. The non-retriable error code means the Rosetta client stops retrying, but the gossip damage is already done before the error is returned.

### Likelihood Explanation
No credentials, tokens, or privileged access are required. The `/construction/submit` endpoint is publicly reachable in online mode. The attacker only needs to produce a structurally valid protobuf-encoded `TransferTransaction` or `AccountCreateTransaction` (the two accepted types per `isSupportedTransactionType`) with an arbitrary signature blob. This is trivially achievable with the Hiero SDK or by hand-crafting protobuf bytes. The attack is fully repeatable and automatable.

### Recommendation
Add signature pre-validation inside `ConstructionSubmit` before calling `hiero.TransactionExecute`. Specifically:

1. Extract the `SignedTransaction` protobuf from the submitted bytes.
2. For each entry in the `sigMap`, verify the Ed25519 signature over `bodyBytes` using the corresponding public key (mirroring the logic already present in `ConstructionCombine` at lines 72–84).
3. Return `ErrInvalidSignatureVerification` (code 107) if any signature fails — before the transaction is forwarded to the consensus node.

Additionally, enforce application-level rate limiting on `/construction/submit` independent of the optional Traefik middleware.

### Proof of Concept
```bash
# 1. Obtain a valid unsigned transaction hex from /construction/payloads (or craft one manually)
UNSIGNED_TX="0x0a432a41..."   # valid protobuf-encoded TransferTransaction

# 2. Craft a "signed" transaction by appending a random 64-byte signature
#    to the SignatureMap without going through /construction/combine
FAKE_SIGNED_TX=$(python3 -c "
import binascii, sys
# Decode the unsigned tx, inject a random 64-byte sig into sigMap field
# (exact protobuf manipulation omitted for brevity but trivially done with
#  the hiero-sdk-go or protoc-generated Python bindings)
print('0x<crafted_bytes_with_random_sig>')
")

# 3. Submit directly to /construction/submit — bypassing /construction/combine
curl -s -X POST http://<rosetta-online-host>/construction/submit \
  -H 'Content-Type: application/json' \
  -d "{
    \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"testnet\"},
    \"signed_transaction\": \"$FAKE_SIGNED_TX\"
  }"

# Expected response: Rosetta error code 118 (ErrTransactionSubmissionFailed,
# retriable=false), with reason containing INVALID_PAYER_SIGNATURE.
# The transaction was already gossiped to the consensus network before rejection.
```

func (c *constructionAPIService) ConstructionSubmit(
	_ context.Context,
	request *rTypes.ConstructionSubmitRequest,
) (*rTypes.TransactionIdentifierResponse, *rTypes.Error) {
	if !c.IsOnline() {
		return nil, errors.ErrEndpointNotSupportedInOfflineMode
	}

	transaction, rErr := unmarshallTransactionFromHexString(request.SignedTransaction)
	if rErr != nil {
		return nil, rErr
	}

	hashBytes, err := hiero.TransactionGetTransactionHash(transaction)
	if err != nil {
		return nil, errors.ErrTransactionHashFailed
	}

	hash := tools.SafeAddHexPrefix(hex.EncodeToString(hashBytes))
	transactionId, _ := hiero.TransactionGetTransactionID(transaction)
	log.Infof("Submitting transaction %s (hash %s) to node %s", transactionId,
		hash, transaction.GetNodeAccountIDs()[0])

	_, err = hiero.TransactionExecute(transaction, c.sdkClient)
	if err != nil {
		log.Errorf("Failed to execute transaction %s (hash %s): %s", transactionId, hash, err)
		return nil, errors.AddErrorDetails(
			errors.ErrTransactionSubmissionFailed,
			"reason",
			fmt.Sprintf("%s", err),
		)
	}

	return &rTypes.TransactionIdentifierResponse{
		TransactionIdentifier: &rTypes.TransactionIdentifier{Hash: hash},
	}, nil
}
