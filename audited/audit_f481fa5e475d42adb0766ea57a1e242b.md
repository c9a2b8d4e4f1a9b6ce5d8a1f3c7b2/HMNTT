### Title
Unbounded Subscriber Amplification via Unthrottled Safety-Check DB Polling in `subscribeTopic()`

### Summary
Any unauthenticated client can open an arbitrary number of gRPC subscriptions to a valid topic that has no messages. After exactly one second, each subscription independently triggers a safety-check that calls `topicMessageRetriever.retrieve(gapFilter, false)` (unthrottled mode), which executes up to 13 sequential database queries per subscriber (1 initial + 12 repeats at 20 ms intervals). With no global subscriber cap or per-IP rate limit on subscription creation, an attacker with many connections can generate tens of thousands of database queries in a sub-second burst, exhausting the DB connection pool and degrading or crashing the gRPC processing tier.

### Finding Description

**Exact code path:**

1. `TopicMessageServiceImpl.subscribeTopic()` — [1](#0-0)  — constructs a one-shot `Mono.delay(1s)` safety-check that fires for every active subscription where `!topicContext.isComplete()`.

2. `TopicContext.isComplete()` — [2](#0-1)  — unconditionally returns `false` when `filter.getEndTime() == null`, which is the default for a live subscription. The safety-check filter therefore always passes.

3. `missingMessages(topicContext, null)` — [3](#0-2)  — when `current == null` (safety-check path), unconditionally calls `topicMessageRetriever.retrieve(gapFilter, false)` with `throttled=false`.

4. `PollingTopicMessageRetriever.retrieve(filter, false)` — [4](#0-3)  — when `throttled=false`, sets `numRepeats = unthrottled.getMaxPolls()` (default **12**) and `pollingFrequency = 20 ms`.

5. `PollingContext.isComplete()` for unthrottled — [5](#0-4)  — returns only `limitHit`. With no limit set and no messages in the DB, `limitHit` is always `false`, so the retriever polls all 12 repeat cycles, issuing **13 DB queries** (initial + 12 repeats) per safety-check invocation.

6. The retriever also carries `retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))` — [6](#0-5)  — meaning DB errors caused by pool exhaustion trigger indefinite retries, amplifying load further.

**Root cause:** There is no global cap on concurrent subscriptions, no per-IP rate limit on subscription creation, and the safety-check unconditionally invokes the unthrottled retriever (13 DB queries, ~240 ms window) for every open subscription that has received no messages.

**Why existing checks are insufficient:**

- `maxConcurrentCallsPerConnection = 5` — [7](#0-6)  — limits calls *per connection*, but an attacker opens many connections. There is no global connection limit configured.
- `subscriberCount` — [8](#0-7)  — is a metrics gauge only; it is never checked against a maximum.
- `checkTopicExists = true` — [9](#0-8)  — requires a valid topic, but any publicly known topic ID satisfies this.
- `Schedulers.boundedElastic()` limits concurrent threads but its default task queue holds 100,000 pending tasks, so thousands of safety-check operations queue up and execute sequentially, sustaining DB pressure over time rather than shedding it.

### Impact Explanation

With N open subscriptions on a topic with no messages, the safety-check fires simultaneously at T+1s, issuing N×13 DB queries in a ~240 ms window. At N=1,000 subscriptions (200 connections × 5 calls/connection), this produces ~13,000 queries in 240 ms (~54,000 QPS). The gRPC module's DB connection pool is exhausted; legitimate queries time out. The `retryWhen(Long.MAX_VALUE)` loop means failed queries re-enter the queue, sustaining the overload. This matches the "≥30% of network processing nodes" threshold because all gRPC mirror-node replicas share the same DB tier and would be simultaneously impacted.

### Likelihood Explanation

The attack requires only a valid topic ID (publicly observable on-chain) and the ability to open many TCP connections to port 5600 — no credentials, no privileged access, no brute force. A single attacker machine with standard tooling (e.g., a gRPC client loop) can open hundreds of connections. The attack is fully repeatable: after the 60-second retriever timeout, subscriptions can be re-opened to trigger another burst.

### Recommendation

1. **Add a global concurrent-subscriber cap** checked in `subscribeTopic()` before accepting a new subscription; reject with `RESOURCE_EXHAUSTED` when exceeded.
2. **Rate-limit subscription creation per source IP** at the Netty/gRPC interceptor layer.
3. **Make the safety-check use the throttled retriever** (`retrieve(gapFilter, true)`) or impose a hard limit on unthrottled polls triggered by the safety-check path.
4. **Cap `maxPolls` for the safety-check path** independently of the gap-fill path, or replace the unthrottled retriever call with a single bounded query.
5. **Limit the `retryWhen` attempts** in `PollingTopicMessageRetriever` to a small finite number to prevent retry amplification under DB pressure.

### Proof of Concept

```
# Preconditions:
# - Mirror node gRPC endpoint reachable at <host>:5600
# - A valid topic ID with no recent messages (e.g., 0.0.12345)

# Step 1: Open 200 TCP connections, each with 5 concurrent subscribeToTopic streams
#         (respects maxConcurrentCallsPerConnection=5 per connection)
#         Total: 1,000 concurrent subscriptions

for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":12345},"consensusStartTime":{"seconds":9999999999}}' \
    -H "content-type: application/grpc" \
    <host>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  # open 5 streams per connection via multiplexing
done

# Step 2: Wait 1 second
sleep 1

# Step 3: Observe DB connection pool exhaustion in mirror node logs:
#   "Safety check triggering gap recovery query with filter ..." × 1000
#   DB pool timeout errors / query failures for legitimate subscribers
#   Retriever retry storms due to retryWhen(Long.MAX_VALUE, ...)
```
