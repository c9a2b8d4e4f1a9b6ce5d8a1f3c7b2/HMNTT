### Title
Missing Entity-Type Validation in `getNftsByAccountId()` Allows Contract EVM Address to Return NFT Records via Accounts Endpoint

### Summary
`getNftsByAccountId()` resolves the path parameter through `EntityService.getEncodedId()`, which accepts any EVM address that matches the hex-address regex — including addresses belonging to contract entities — and returns their numeric ID without ever checking the entity type. The resolved ID is passed directly to `NftService.getNfts()`, which queries the `nft` table purely by `account_id`, so NFT records owned by a contract are returned through the `/accounts/{evmAddress}/nfts` endpoint. No privilege is required.

### Finding Description

**Exact code path:**

`GET /api/v1/accounts/{evmAddress}/nfts` → `AccountController.getNftsByAccountId()` → `EntityService.getEncodedId()` → `EntityService.getEntityIdFromEvmAddress()` → `NftService.getNfts()`

**Root cause — `getNftsByAccountId()` performs no entity-type check:**

```
// rest/controllers/accountController.js  lines 90-94
getNftsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);
  // ← no isValidAccount() call, no type check
  const query = this.extractNftMultiUnionQuery(filters, accountId);
  const nonFungibleTokens = await NftService.getNfts(query);
``` [1](#0-0) 

Compare with `listStakingRewardsByAccountId()`, which at least calls `isValidAccount()` (though that check is also type-blind): [2](#0-1) 

**`getEntityIdFromEvmAddress()` — no type filter in SQL:**

```sql
select id
from entity
where deleted <> true
  and evm_address = $1
-- ← no "and type = 'ACCOUNT'" predicate
``` [3](#0-2) 

The function returns `rows[0].id` for any entity — account or contract — that has a matching `evm_address` column: [4](#0-3) 

**`NftService.getNfts()` — purely ID-based query:**

```sql
select account_id, ...
from nft
left join entity e on e.id = nft.token_id
where account_id = $1   -- ← any entity ID accepted
``` [5](#0-4) 

**`isValidEntityId()` — accepts any 40-hex-char EVM address:**

```js
const isValidEntityId = (entityId, allowEvmAddress = true, ...) => {
  ...
  return allowEvmAddress && isValidEvmAddress(entityId, evmAddressType);
};
``` [6](#0-5) 

`isValidEvmAddress` matches `^(0x)?[A-Fa-f0-9]{40}$` — a contract's EVM address satisfies this trivially. [7](#0-6) 

**Why existing checks fail:**

- `isValidEntityId()` validates *format only*, not entity type.
- `getEntityIdFromEvmAddress()` queries without a `type` predicate.
- `isValidAccount()` (not even called here) only checks existence (`!isNil(entity)`), not type: [8](#0-7) 

### Impact Explanation

An unauthenticated attacker can enumerate NFT holdings of any smart contract by supplying the contract's EVM address to the `/accounts/{evmAddress}/nfts` endpoint. The API contract explicitly scopes this endpoint to accounts (OpenAPI summary: "Returns information for all non-fungible tokens for an account"): [9](#0-8) 

Concrete effects:
1. **API boundary violation** — contract NFT data is served through the accounts endpoint, bypassing any access-control or rate-limiting policies that differ between `/accounts` and `/contracts` routes.
2. **Information disclosure** — NFT holdings of contracts (e.g., DAO treasuries, escrow contracts) are exposed through an unintended surface.
3. **Downstream consumer confusion** — clients that assume the accounts endpoint only returns user-account data will silently process contract NFT records as if they belong to a user account.

### Likelihood Explanation

Exploitation requires zero privileges and zero authentication. The attacker only needs to know (or enumerate) a contract's EVM address, which is public on-chain data. The request is a single unauthenticated HTTP GET. The attack is trivially repeatable and scriptable against any deployed mirror-node instance.

### Recommendation

After resolving the entity ID in `getNftsByAccountId()`, verify the entity type before proceeding:

```js
getNftsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  // Add: reject non-account entities
  const entity = await EntityService.getEntityTypeById(accountId);
  if (!entity || entity.type !== EntityType.ACCOUNT) {
    throw new NotFoundError();
  }
  ...
};
```

Alternatively, add `and type = 'ACCOUNT'` to `entityFromEvmAddressQuery` in `EntityService` so the lookup itself rejects contract addresses. Also fix `isValidAccount()` to check `entity.type === 'ACCOUNT'` rather than mere existence.

### Proof of Concept

**Preconditions:** A contract with a known EVM address (e.g., `0xaabbccdd...`) exists in the mirror-node database with NFT holdings (`account_id` entries in the `nft` table).

**Steps:**

1. Identify a contract's EVM address (publicly available from any block explorer or the `/contracts` endpoint).
2. Send:
   ```
   GET /api/v1/accounts/0xaabbccddee1122334455667788990011aabbccdd/nfts
   ```
3. `EntityId.isValidEntityId('0xaabbccddee...')` returns `true` (valid hex address format).
4. `getEntityIdFromEvmAddress()` resolves the contract's `evm_address` column to its numeric entity ID (no type filter).
5. `NftService.getNfts()` executes `WHERE account_id = <contract_entity_id>` and returns the contract's NFT records.
6. The response is HTTP 200 with NFT data belonging to the contract, served through the accounts endpoint.

const isValidEvmAddress = (address, evmAddressType = constants.EvmAddressType.ANY) => {
  if (typeof address !== 'string') {
    return false;
  }

  if (evmAddressType === constants.EvmAddressType.ANY) {
    return evmAddressRegex.test(address) || evmAddressShardRealmRegex.test(address);
  }
  if (evmAddressType === constants.EvmAddressType.NO_SHARD_REALM) {
    return evmAddressRegex.test(address);
  }
  if (evmAddressType === constants.EvmAddressType.NUM_ALIAS) {
    return (evmAddressRegex.test(address) || evmAddressShardRealmRegex.test(address)) && !isEvmAddressAlias(address);
  }
  return evmAddressShardRealmRegex.test(address);
};
```
