### Title
Unauthenticated Request Flood DoS on `GET /api/v1/blocks` Due to Absent Application-Level Rate Limiting

### Summary
The `GET /api/v1/blocks` endpoint in the Node.js REST API has no per-IP or per-client rate limiting. The `authHandler` middleware only rejects requests with *invalid* credentials — unauthenticated requests pass through unconditionally. An unprivileged attacker can flood this endpoint, exhausting the PostgreSQL connection pool and queuing unbounded async work in the Node.js event loop, denying service to legitimate callers.

### Finding Description

**Route registration** — no rate-limiting middleware is applied: [1](#0-0) 

**`authHandler` passes unauthenticated requests silently** — the guard only fires when credentials are present and wrong; absent credentials return immediately with no rejection: [2](#0-1) 

**Server middleware stack** — no rate-limiter is registered anywhere between the transport and the route handler. The only optional protection is a Redis response cache (`responseCacheCheckHandler`), which is disabled unless both `config.cache.response.enabled` and `config.redis.enabled` are true: [3](#0-2) 

**`getBlocks` issues a DB query on every request** — each call to `RecordFileService.getBlocks` executes a parameterized SQL query against PostgreSQL: [4](#0-3) [5](#0-4) 

**No rate-limiting library present in the REST module** — a search across all `rest/**/*.js` files for `rateLimit`, `throttle`, or `express-rate-limit` returns zero application-code matches. The Bucket4j throttling that exists lives exclusively in the Java `web3` module and is entirely separate from this Node.js service: [6](#0-5) 

**Root cause**: The REST API's design assumes an upstream reverse proxy (e.g., Traefik with a `Middleware` chain) enforces rate limits. No such enforcement exists at the application layer, and the Traefik middleware templates are optional Helm chart values — they are not guaranteed to be deployed. [7](#0-6) 

### Impact Explanation

Each unauthenticated `GET /api/v1/blocks` request causes the Node.js process to: parse the request, build SQL filters, acquire a connection from the PostgreSQL pool, execute a query, and serialize the response. Under sustained flood conditions:

1. The PostgreSQL connection pool is exhausted; new requests queue in memory.
2. The Node.js heap grows with pending promise chains and request objects.
3. Legitimate callers receive timeouts or `ECONNREFUSED` once the listen backlog fills.
4. The mirror node becomes effectively partitioned — it cannot serve block data to wallets, explorers, or other consumers that depend on this endpoint.

The `statementTimeout` pool config limits individual query duration but does not cap the *number* of concurrent queries or incoming connections. [8](#0-7) 

### Likelihood Explanation

Preconditions: none. The attacker needs only network access to the mirror node's HTTP port. No credentials, API keys, or special knowledge are required. The attack is trivially repeatable with standard tools (`wrk`, `ab`, `hey`) from a single host or a small botnet. Public mirror nodes (e.g., mainnet/testnet) are directly reachable from the internet, making this a realistic threat.

### Recommendation

1. **Add application-level rate limiting** in the Node.js REST server using `express-rate-limit` (or equivalent), keyed on `req.ip`, applied globally before route handlers in `server.js`.
2. **Make the Traefik middleware mandatory** (not opt-in) in the Helm chart, or document that deployment without an upstream rate-limiting proxy is unsupported.
3. **Enable the Redis response cache** by default for read-only list endpoints like `/blocks`; a cache hit avoids the DB query entirely.
4. **Set an explicit `maxConnections` ceiling** in the DB pool that is low enough to leave headroom for the importer process, preventing a request flood from starving the ingestion pipeline.

### Proof of Concept

```bash
# No credentials required. Run from any host with network access.
# Replace HOST with the mirror node's address.

wrk -t8 -c400 -d60s "http://HOST/api/v1/blocks"

# Expected result under flood:
# - Legitimate requests begin timing out within seconds
# - DB pool exhausted (pool queue depth climbs)
# - Node.js process RSS grows monotonically
# - /health/readiness may begin returning 503 as DB becomes unavailable
```

Alternatively, with `curl` in a tight loop to confirm zero authentication barrier:

```bash
for i in $(seq 1 10000); do
  curl -s "http://HOST/api/v1/blocks" -o /dev/null &
done
wait
```

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();

function parseDbPoolConfig() {
  const {pool} = getConfig().db;
  const configKeys = ['connectionTimeout', 'maxConnections', 'statementTimeout'];
  configKeys.forEach((configKey) => {
    const value = pool[configKey];
    const parsed = parseInt(value, 10);
    if (Number.isNaN(parsed) || parsed <= 0) {
      throw new InvalidConfigError(`invalid value set for db.pool.${configKey}: ${value}`);
    }
    pool[configKey] = parsed;
  });
}
