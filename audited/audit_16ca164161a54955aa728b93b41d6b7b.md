### Title
Cache Bypass via Inline Literal evmAddress Enables Unbounded CPU Exhaustion on GraphQL Endpoint

### Summary
The `CachedPreparsedDocumentProvider` caches parsed GraphQL documents keyed on the raw query string. An unprivileged attacker who embeds a unique `evmAddress` literal directly in each query string (rather than using GraphQL variables) produces a distinct cache key per request, forcing full parse+validate+execute on every call. The GraphQL module has no rate limiting, making this trivially repeatable at scale.

### Finding Description
**Cache key is the raw query string:**
In `CachedPreparsedDocumentProvider.java` line 26, the cache lookup is:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
```
`executionInput.getQuery()` returns the verbatim query string. Two requests with structurally identical queries but different inline literal values (e.g., different `evmAddress` strings) are treated as entirely different cache entries.

**Cache is bounded at 1000 entries** (`CacheProperties.java` line 15: `maximumSize=1000`). An attacker cycling through more than 1000 unique addresses continuously evicts old entries, keeping the effective cache hit rate near zero.

**No rate limiting on the GraphQL module:** `ThrottleConfiguration`, `ThrottleManagerImpl`, and `ThrottleProperties` all live under `web3/src/main/java/…/web3/throttle/` and are wired only to the web3/EVM endpoint. The `graphql` module has no equivalent throttle bean, no `@RateLimiter`, and no servlet filter enforcing a request-per-second cap.

**Existing mitigations are insufficient:**
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` (`GraphQlConfiguration.java` lines 43–45) limit per-query work but do not bound request rate.
- Parser limits (`maxCharacters(10000)`, `maxTokens(1000)`, `GraphQlConfiguration.java` lines 33–38) prevent oversized individual queries but do not prevent flooding with many small unique queries.
- The `account(evmAddress:)` field is publicly reachable at `/graphql/alpha` with no authentication.

**Exploit flow:**
1. Attacker generates a stream of 40-hex-character EVM addresses (e.g., incrementing counter encoded as hex).
2. Each request embeds the address as an inline literal: `{ account(input: {evmAddress: "0x<unique>"}) { id } }`.
3. Each request is a cache miss → GraphQL parser tokenizes and validates the document → resolver calls `EntityServiceImpl.getByEvmAddressAndType()` → DB query.
4. With no rate limit, the attacker can sustain thousands of such requests per second from a single machine or a small botnet.

### Impact Explanation
Every cache-miss request incurs full GraphQL parse, schema validation, and a database round-trip. Sustained flooding drives CPU utilization on the mirror-node JVM well above the 30% baseline threshold. Under sufficient load the service becomes unresponsive to legitimate users. Because the GraphQL module is a public read API with no authentication or rate limiting, the attacker bears no cost beyond network bandwidth.

### Likelihood Explanation
The attack requires zero privileges, zero credentials, and no special knowledge beyond the publicly documented GraphQL schema. The query pattern is trivially scriptable with `curl` or any HTTP client in a loop. The 40-hex-character address space (2^160) is effectively inexhaustible, so address reuse is not a practical constraint. Any motivated external party can execute this.

### Recommendation
1. **Enforce parameterized queries at the cache layer:** Key the preparsed document cache on a normalized query string (variables stripped out) or reject requests that embed non-variable literal arguments for high-cardinality fields.
2. **Add rate limiting to the GraphQL module:** Introduce a `Bucket4j` or Spring `RateLimiter` bean in the `graphql` module analogous to `ThrottleConfiguration` in the `web3` module, enforcing a per-IP or global requests-per-second cap.
3. **Encourage/enforce GraphQL variables:** Document or enforce that callers must use `$evmAddress` variables rather than inline literals; this keeps the query string constant across calls and allows the cache to function as intended.

### Proof of Concept
```bash
# Generate and send 5000 unique-address queries with no authentication
for i in $(seq 1 5000); do
  ADDR=$(printf '%040x' $i)
  curl -s -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: {evmAddress: \\\"0x${ADDR}\\\"}) { id } }\"}" &
done
wait
# Each request carries a unique query string → 5000 cache misses → 5000 full parse+validate+DB cycles
# Monitor JVM CPU via JMX or /actuator/metrics; expect sustained elevation above 30% baseline
```
