### Title
Missing Past-Timestamp Validation in `ConstructionPayloads` Allows Crafting Pre-Expired Transactions

### Summary
The `getTransactionTimestampProperty` function in `construction_service.go` only validates that the computed `validStartNanos` is greater than zero when `valid_until_nanos` is supplied, but never checks whether the resulting validity window lies in the past relative to the server's current time. An unprivileged caller can supply a `valid_until_nanos` value that is a past timestamp (but large enough that `validUntil - maxValidDurationNanos > 0`), causing the server to build and return a well-formed unsigned transaction whose entire validity window has already elapsed. When that transaction is signed and submitted, the Hedera network rejects it as expired, triggering `ErrTransactionSubmissionFailed`, while the server reported no error at the `ConstructionPayloads` stage.

### Finding Description

**Exact code path:**

`rosetta/app/services/construction_service.go`, function `getTransactionTimestampProperty`, lines 396–414:

```go
if _, ok := metadata[metadataKeyValidUntilNanos]; ok {
    var validUntil int64
    validUntil, err = c.getIntMetadataValue(metadata, metadataKeyValidUntilNanos)
    ...
    validDurationSeconds = maxValidDurationSeconds          // 180
    validStartNanos = validUntil - maxValidDurationNanos    // line 407
    if validStartNanos <= 0 {                               // line 408 — ONLY check
        err = errors.ErrInvalidArgument
    }
    return
}
```

`maxValidDurationNanos` = 180 × 1 000 000 000 = 180 000 000 000 ns (≈ 180 seconds from the Unix epoch).

**Root cause:** The guard at line 408 only rejects `validStartNanos ≤ 0`. It does **not** compare `validStartNanos` (or `validUntil`) against `time.Now().UnixNano()`. Any `valid_until_nanos` value that is positive and greater than `maxValidDurationNanos` (i.e., any timestamp after 1970-01-01T00:03:00Z) passes the check, even if it represents a moment years in the past.

**Failed assumption:** The code assumes that a positive `validStartNanos` implies a usable transaction. The Hedera network's assumption is that the transaction's validity window `[validStartNanos, validStartNanos + 180s]` must overlap with the current wall-clock time.

**Exploit flow:**

1. Attacker (no credentials required) calls `POST /construction/payloads` with:
   ```json
   {
     "metadata": {
       "node_account_id": "0.0.3",
       "valid_until_nanos": "181000000001"
     },
     ...
   }
   ```
   `validStartNanos` = 181 000 000 001 − 180 000 000 000 = **1 000 000 001** (positive → passes line 408).  
   The validity window is [~1970-01-01T00:00:01Z, ~1970-01-01T00:03:01Z] — entirely in the past.

2. `ConstructionPayloads` returns HTTP 200 with a well-formed `unsigned_transaction` hex and `payloads` for signing. No error is surfaced.

3. The legitimate client signs the payloads and calls `ConstructionCombine`, receiving a fully signed transaction hex.

4. The client calls `ConstructionSubmit`. The Hedera node rejects the transaction as expired. The server returns `ErrTransactionSubmissionFailed` (error code 118).

**Why existing checks fail:**

The test suite (`construction_service_test.go` lines 1104–1113) only covers `valid_until_nanos` values that produce `validStartNanos ≤ 0` (e.g., `"180000000000"` and `"179999999999"`). No test covers a past-but-positive timestamp. The `getIntMetadataValue` helper (lines 540–554) only rejects negative values, not past ones.

### Impact Explanation

- **Transaction submission DoS:** Any caller can force the construction pipeline to produce a transaction that will always be rejected by the network, wasting the payer's fee budget and causing `ErrTransactionSubmissionFailed` with no actionable error at the `ConstructionPayloads` stage.
- **Signed-transaction leakage window:** The signed transaction hex (produced after `ConstructionCombine`) encodes a specific `(payer, validStartNanos)` transaction ID. If a Hedera consensus node has a clock skew of even a few seconds relative to the server, a transaction whose validity window ended very recently could be accepted by that node, enabling a narrow replay window without the payer's fresh intent.
- **Severity:** Medium. The primary impact is reliable DoS of the submission flow for any payer whose metadata is tampered with (e.g., MITM between `ConstructionMetadata` and `ConstructionPayloads`). The replay window is narrow but non-zero under realistic clock-skew conditions (Hedera nodes tolerate ±10 s skew by default).

### Likelihood Explanation

- The `/construction/payloads` endpoint requires no authentication; any network-reachable client can call it.
- The `valid_until_nanos` field is explicitly documented as a client-supplied metadata key (line 38 of `construction_service.go`), so supplying it is a normal, expected operation.
- A MITM or a malicious Rosetta proxy that intercepts the metadata map from `ConstructionMetadata` before it reaches `ConstructionPayloads` can silently downgrade `valid_until_nanos` to a past value with no detectable error until submission time.
- Repeatability is 100%: the condition is deterministic and requires no brute-force.

### Recommendation

Add a wall-clock lower-bound check immediately after computing `validStartNanos` in `getTransactionTimestampProperty`:

```go
validStartNanos = validUntil - maxValidDurationNanos
if validStartNanos <= 0 {
    err = errors.ErrInvalidArgument
    return
}
// NEW: reject validity windows that are already expired
if validUntil <= time.Now().UnixNano() {
    log.Errorf("valid_until_nanos %d is in the past", validUntil)
    err = errors.ErrInvalidArgument
    return
}
```

Checking `validUntil` (rather than `validStartNanos`) against `time.Now()` is sufficient and avoids a separate `time.Now()` call for `validStartNanos`. A small tolerance (e.g., 5 s) can be added to absorb minor clock drift between the Rosetta server and the caller.

func (c *constructionAPIService) getTransactionTimestampProperty(metadata map[string]any) (
	validDurationSeconds, validStartNanos int64, err *rTypes.Error,
) {
	if _, ok := metadata[metadataKeyValidUntilNanos]; ok {
		// ignore valid duration seconds and valid start nanos in the metadata if valid until is present
		var validUntil int64
		validUntil, err = c.getIntMetadataValue(metadata, metadataKeyValidUntilNanos)
		if err != nil {
			log.Errorf("Invalid valid until %s", metadata[metadataKeyValidUntilNanos])
			err = errors.ErrInvalidArgument
			return
		}

		validDurationSeconds = maxValidDurationSeconds
		validStartNanos = validUntil - maxValidDurationNanos
		if validStartNanos <= 0 {
			log.Errorf("Valid start nanos determined from valid until %s is not positive",
				metadata[metadataKeyValidUntilNanos])
			err = errors.ErrInvalidArgument
		}

		return
	}
