### Title
206 Partial Content Response Incorrectly Cached, Poisoning Subsequent Users with Incomplete Transaction History

### Summary
`responseCacheUpdateHandler()` uses `httpStatusCodes.isSuccess(res.statusCode)` as its cache-admission guard, and `isSuccess` is defined as `code >= 200 && code < 300`, which includes HTTP 206. The application intentionally emits 206 for endpoints such as NFT transaction history and contract results when the mirror node's data is incomplete. Once a 206 response is cached, every subsequent request for the same URL is served the partial body with status 206 for the full cache TTL, even after the mirror node has caught up and would return a complete 200 OK.

**Where 206 is emitted (server-side data conditions, no privilege required to reach):**

```js

  res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
``` [4](#0-3) 

`rest/controllers/contractController.js` lines 1026–1028 — contract results return 206 when `callResult` is nil:
```js
if (isNil(contractResults[0].callResult)) {
  res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
``` [5](#0-4) 

**Root cause:** The failed assumption is that every 2xx response is a complete, cacheable result. HTTP 206 is explicitly used by this application to signal *incomplete* data (mirror node still syncing). The `isSuccess()` predicate does not distinguish between 200 and 206, so the partial body and the 206 status code are written into Redis under the normal cache key (URL hash) with the full `max-age` TTL.

**Why existing checks are insufficient:** There is no exclusion for `httpStatusCodes.PARTIAL_CONTENT.code` (206) anywhere in the cache-update path. The `isUnmodified` branch correctly remaps 304 → 200 before storing, but there is no analogous guard that prevents 206 from being stored, nor any short-TTL override for partial responses.

### Impact Explanation

Any user who queries `/api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions` or `/api/v1/contracts/results/{transactionIdOrHash}` while the mirror node has incomplete data will cause a 206 partial response to be cached. All subsequent users querying the identical URL receive the cached partial body with status 206 for the full cache TTL — even after the mirror node has fully ingested the missing data and would return a complete 200 OK. For NFT transaction history this means the mint transaction is absent from the cached response, presenting a falsified provenance record to every consumer of that cached entry. The impact is data-integrity corruption of transaction history served at scale.

### Likelihood Explanation

Mirror nodes routinely have transient windows of incomplete data during normal sync operation. No special privileges, authentication, or crafted payload are required — a plain unauthenticated GET request is sufficient. An attacker who monitors mirror-node lag (e.g., by comparing the latest consensus timestamp in responses against the network's known head) can reliably time a request to land during an incomplete-data window, guaranteeing cache poisoning for the duration of the TTL. The attack is repeatable: after TTL expiry the attacker can re-trigger it with another GET request if the data is still incomplete, or simply wait for the next sync gap.

### Recommendation

Change the cache-admission condition to accept **only** HTTP 200 (and 304 via the existing `isUnmodified` path):

```js
// Before
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {

// After
if (responseBody && responseCacheKey && (isUnmodified || res.statusCode === httpStatusCodes.OK.code)) {
```

Alternatively, explicitly exclude `PARTIAL_CONTENT`:
```js
const isCacheable = res.statusCode === httpStatusCodes.OK.code;
if (responseBody && responseCacheKey && (isUnmodified || isCacheable)) {
```

If 206 responses must ever be cached (e.g., for short-term deduplication), they must use a separate, very short TTL and must never overwrite an existing 200-cached entry for the same key.

### Proof of Concept

**Preconditions:** Redis cache enabled; mirror node is behind on ingesting an NFT's mint transaction for token `0.0.1500`, serial `2` (i.e., `nft.createdTimestamp IS NULL` in the DB).

**Steps:**

1. Send a GET request as an unprivileged user:
   ```
   GET /api/v1/tokens/0.0.1500/nfts/2/transactions
   ```
   Server returns HTTP 206 with a partial transaction list (mint transaction absent). `responseCacheUpdateHandler` caches this 206 response under the URL's MD5 cache key with the full `max-age` TTL.

2. Mirror node catches up; the NFT's mint transaction is now ingested (`createdTimestamp` is populated). A direct DB-backed request would now return HTTP 200 with the complete history.

3. Send the same GET request again (or have any other user send it):
   ```
   GET /api/v1/tokens/0.0.1500/nfts/2/transactions
   ```
   `responseCacheCheckHandler` finds the cached entry and returns HTTP 206 with the **incomplete** body — the mint transaction is still absent — for all users until the TTL expires.

4. Repeat step 1 after TTL expiry to re-poison the cache indefinitely.
