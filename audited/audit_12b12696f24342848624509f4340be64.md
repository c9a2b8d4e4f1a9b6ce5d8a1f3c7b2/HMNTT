### Title
Thread Pool Exhaustion via Cache-Poisoned Large IN Clause Queries in `findStorageBatch()`

### Summary
An unprivileged attacker can fill the per-contract slot Caffeine cache (up to 1500 entries) by sending unique-slot `eth_call` requests, causing every subsequent `findStorageBatch()` call for that contract to issue a SQL `IN` clause with up to 1500 parameters. The `statementInspector` timeout guard is a **pre-execution** check only — it does not cancel in-progress queries — so slow batch queries hold web3 request threads for up to the full `requestTimeout` (10 s) plus the DB `statementTimeout` (3 s) overshoot. Sustained at ~20 RPS (well within the 500 RPS global limit), this exhausts the Tomcat thread pool and causes the node to stop processing new requests.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (line 58–70) delegates to `findStorageBatch()` whenever batch caching is enabled and the value is not already in `contractStateCache`.

Inside `findStorageBatch()` (lines 85–122):

- **Line 90**: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — every unique slot key a caller supplies is unconditionally inserted into the per-contract Caffeine cache.
- **Line 91**: `final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet()` — the **entire** set of accumulated slot keys is read back.
- **Line 103**: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — a native SQL query is executed with all accumulated keys in an `IN` clause:

```sql
SELECT slot, value FROM contract_state
WHERE contract_id = :contractId
AND slot IN (:slots)   -- up to 1500 parameters
```

The `slotsPerContract` cache is configured with `maximumSize=1500` (`CacheProperties.java` line 39). An attacker who sends 1500 `eth_call` requests for the same contract, each referencing a distinct storage slot, fills this cache. Every subsequent request for that contract then triggers a 1500-parameter IN clause query.

**Root cause — the `statementInspector` is a pre-execution guard only:**

`HibernateConfiguration.statementInspector()` (lines 31–47) checks elapsed time **before** submitting each SQL statement to the database:

```java
long elapsed = System.currentTimeMillis() - startTime;
if (elapsed >= timeout) {          // checked BEFORE the query runs
    throw new QueryTimeoutException(...);
}
return sql;                        // query is then submitted and runs freely
```

Once the check passes (elapsed < 10 000 ms), the query executes without any cancellation mechanism. The inspector does **not** call `PreparedStatement.setQueryTimeout()`, does not interrupt the JDBC thread, and does not cancel the in-flight database operation. A query that starts at t = 9 999 ms passes the guard and can run for an additional 3 000 ms (the DB `statementTimeout`), holding the thread until t ≈ 13 000 ms — 3 seconds beyond the intended limit. More generally, any slow batch query holds its thread for the full DB-level timeout regardless of when it started.

**Why existing checks fail:**

| Check | Why insufficient |
|---|---|
| `statementInspector` (pre-execution, 10 s) | Does not cancel in-progress queries; last query per request can overshoot by up to 3 s |
| `db.statementTimeout = 3000 ms` | Limits a single query, not total request duration; multiple queries per request still accumulate to ~10 s |
| Global rate limit (500 RPS) | Not per-IP; attacker needs only ~20 RPS (4 % of limit) to exhaust 200 threads |
| Cache `maximumSize=1500` | Acts as the attack amplifier, not a mitigation |

### Impact Explanation
Spring Boot's embedded Tomcat uses a default thread pool of 200 threads. With each poisoned request holding a thread for up to ~10 s, only 200 ÷ 10 = **20 sustained RPS** of slow requests is needed to saturate the pool. New incoming requests queue and eventually time out, rendering the web3 node unresponsive. Because mirror nodes are typically deployed in small clusters, exhausting one node's thread pool constitutes ≥ 30 % of network processing capacity. No authentication or special privilege is required — any caller of the public `eth_call` endpoint can trigger this.

### Likelihood Explanation
The attack is fully automatable with a standard HTTP client. Phase 1 (cache poisoning) requires 1 500 requests with distinct slot keys for one contract — achievable in under 3 seconds at the allowed rate. Phase 2 (thread exhaustion) requires only ~20 concurrent slow requests, a tiny fraction of the 500 RPS global limit. The attack is repeatable: the `slotsPerContract` cache expires after 5 minutes (`expireAfterAccess=5m`), but the attacker can re-poison it continuously. No on-chain funds, credentials, or insider knowledge are needed.

### Recommendation
1. **Replace the pre-execution guard with a true JDBC query timeout**: call `PreparedStatement.setQueryTimeout(remainingSeconds)` before each query so the database driver cancels in-flight queries when the deadline passes. This is the only mechanism that actually interrupts an executing query.
2. **Cap the IN clause size**: in `findStorageBatch()`, chunk `cachedSlots` into batches of a fixed maximum (e.g., 100) rather than issuing a single query with up to 1 500 parameters.
3. **Add per-IP / per-source rate limiting** at the API gateway layer so a single client cannot consume the full 500 RPS budget needed to poison the cache.
4. **Bound the per-contract slot cache growth per request**: instead of accumulating all historical slot keys globally, limit how many keys a single request contributes to the batch.

### Proof of Concept

```python
import requests, threading, time

TARGET = "http://<web3-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract_address>"

# Phase 1: poison the slotsPerContract cache with 1500 unique slot keys
def poison():
    for i in range(1500):
        slot = hex(i).zfill(66)   # unique 32-byte slot key
        requests.post(TARGET, json={
            "data": f"<SLOAD calldata for slot {slot}>",
            "to": CONTRACT,
            "estimate": False
        })

poison()
time.sleep(1)

# Phase 2: flood with concurrent requests — each triggers a 1500-entry IN clause
def flood():
    while True:
        requests.post(TARGET, json={
            "data": "<any SLOAD calldata>",
            "to": CONTRACT
        })

threads = [threading.Thread(target=flood) for _ in range(25)]
for t in threads:
    t.start()

# Observe: node thread pool saturates; new requests receive connection-refused or timeout
```

**Expected result**: within seconds, the Tomcat thread pool is exhausted. Requests from legitimate users begin timing out or receiving HTTP 503 responses. The node stops processing new web3 requests until the attack traffic ceases and in-flight queries drain.
