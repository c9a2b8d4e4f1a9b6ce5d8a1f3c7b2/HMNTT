### Title
No Rate Limiting on Basic Auth Endpoint Enables Brute-Force Credential Escalation to Elevated Response Limits

### Summary
`parseUsersConfig()` in `rest/config.js` stores user passwords as plaintext strings with no hashing. The `authHandler` middleware in `rest/middleware/authHandler.js` performs per-request credential comparison with no failed-attempt counter, no lockout, and no rate limiting specific to authentication. An unprivileged external attacker can send unlimited Basic Auth requests to any REST endpoint to brute-force a privileged user's password and obtain a higher per-user `limit`, retrieving more transaction records than the unauthenticated maximum allows.

### Finding Description

**Plaintext storage in `parseUsersConfig()`:**

`rest/config.js` lines 195–208 validate that each user entry has a `username` and `password` field of type `string`, then store them as-is in the in-memory config object. No hashing (bcrypt, argon2, etc.) is applied. [1](#0-0) 

**Plaintext comparison in `authHandler`:**

`rest/middleware/authHandler.js` line 12 uses `tsscmp` (a timing-safe string comparator) to compare the submitted password directly against the stored plaintext. `tsscmp` prevents timing-oracle attacks but provides zero protection against repeated guessing. [2](#0-1) 

**No rate limiting or lockout on auth failures:**

On a failed match, `authHandler` returns HTTP 401 and exits — no counter is incremented, no IP is tracked, no delay is introduced, no lockout is triggered. [3](#0-2) 

The full REST middleware stack registered in `rest/server.js` is: `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. None of these implement per-IP or per-username auth-failure throttling. [4](#0-3) 

The throttle infrastructure that exists (`web3/src/main/java/.../ThrottleConfiguration.java`) is scoped exclusively to the `web3` Java service and is entirely absent from the Node.js REST service. [5](#0-4) 

**Privilege escalation path:**

A successfully authenticated user has `httpContext.set(userLimitLabel, user.limit)` called, which overrides the global `response.limit.max` cap for that request. Integration tests confirm unauthenticated requests are capped at the global max while authenticated users can exceed it up to their personal `limit`. [6](#0-5) [7](#0-6) 

### Impact Explanation

An attacker who successfully brute-forces a privileged user's password can set `limit` to the privileged user's configured value (e.g., 500) instead of the global max (default 100). This allows bulk extraction of transaction records beyond the intended public cap. Because the REST API exposes ledger data (transactions, accounts, token balances), this constitutes unauthorized data over-retrieval. Severity is **Medium–High**: the data is public ledger data (not secret), but the rate-limit bypass undermines the operator's intended access-tier model and can be used to drive disproportionate database load.

### Likelihood Explanation

Basic Auth over HTTP/HTTPS with no lockout is a well-known brute-force target. The attacker needs only a valid username (which may be guessable or leaked via config exposure) and an internet connection. Passwords stored as plaintext mean any config-file exposure (misconfigured secret manager, leaked YAML in version control, environment variable dump) immediately yields working credentials without any cracking step. Online brute-force against a public REST API with no lockout is fully automatable with standard tools (hydra, ffuf, custom scripts). The attack is repeatable and leaves no persistent server-side state.

### Recommendation

1. **Hash passwords at config load time**: In `parseUsersConfig()`, replace plaintext storage with bcrypt/argon2 hashes. Store only the hash; compare using the library's async `verify` function in `authHandler`.
2. **Add per-IP / per-username auth-failure rate limiting**: Use `express-rate-limit` or equivalent, scoped to the `Authorization` header path, with a low threshold (e.g., 5 failures per minute per IP) and exponential backoff or temporary lockout.
3. **Add a delay on auth failure**: Even a fixed 500 ms delay on 401 responses significantly reduces brute-force throughput.
4. **Audit config file permissions**: Ensure `application.yml` / `application.yaml` containing user credentials are not world-readable and are not committed to version control.

### Proof of Concept

**Preconditions:** The server has a configured privileged user, e.g.:
```yaml
hiero.mirror.rest.users:
  - username: premium
    password: secret
    limit: 500
```
Global `response.limit.max` is 100.

**Steps:**

```bash
# 1. Confirm unauthenticated cap
curl -s "https://mirror-node/api/v1/transactions?limit=200" | jq '.transactions | length'
# Returns <= 100

# 2. Brute-force with a wordlist (no lockout, no delay)
for pw in $(cat rockyou.txt); do
  CODE=$(curl -s -o /dev/null -w "%{http_code}" \
    -H "Authorization: Basic $(echo -n "premium:$pw" | base64)" \
    "https://mirror-node/api/v1/transactions?limit=1")
  if [ "$CODE" = "200" ]; then echo "Found: $pw"; break; fi
done

# 3. Use cracked credential to exceed global max
curl -s \
  -H "Authorization: Basic $(echo -n 'premium:secret' | base64)" \
  "https://mirror-node/api/v1/transactions?limit=500" | jq '.transactions | length'
# Returns up to 500 — exceeds the 100-record public cap
```

**Result:** The attacker retrieves up to 5× more transaction records than the unauthenticated maximum, with no server-side mechanism to detect or block the brute-force campaign.
