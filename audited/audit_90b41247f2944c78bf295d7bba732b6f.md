### Title
`requestQueryParser()` Bypasses `qs` Array-Limit Guard via Mixed-Case Key Variants, Building Unbounded Merged Arrays

### Summary
`requestQueryParser()` in `rest/middleware/requestHandler.js` first delegates to `qs.parse()` with `arrayLimit: 100` and `throwOnLimitExceeded: true` to cap repeated parameters. Because `qs` is case-sensitive, keys that differ only in case (e.g., `Timestamp`, `TIMESTAMP`, `timestamp`) are treated as entirely separate keys — each with a single scalar value — so the `qs` guard never fires. The subsequent lowercasing loop then merges all of these distinct keys into one array with no size check, producing a merged array that can be up to 10× the intended limit before any downstream validation runs.

### Finding Description

**Exact code path:**

`rest/middleware/requestHandler.js`, `requestQueryParser()`, lines 38–69.

```
qs.parse(queryString, queryOptions)   // line 54 — case-sensitive parse
for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    if (lowerKey in caseInsensitiveQueryString) {
        caseInsensitiveQueryString[lowerKey] = merge(...);  // line 62 — no size check
    }
}
``` [1](#0-0) [2](#0-1) 

**Root cause — failed assumption:**

The `queryOptions` object sets `arrayLimit: config.query.maxRepeatedQueryParameters` (default **100**) and `throwOnLimitExceeded: true`. [1](#0-0) 

The assumption is that `qs` will reject any attempt to build an array larger than 100 elements. This holds for same-case repeated keys (`timestamp=1&timestamp=2&…&timestamp=101` → `qs` throws). It **fails** for mixed-case variants: `qs` is case-sensitive, so `Timestamp=1`, `TIMESTAMP=2`, and `timestamp=3` are parsed as **three independent scalar-valued keys**. No array is ever created inside `qs`, so `throwOnLimitExceeded` is never evaluated. The merge loop then combines all three scalars into one array with no upper-bound check.

**Maximum merged array size:**

`qs` defaults to `parameterLimit: 1000` (not overridden in `queryOptions`). The word `timestamp` has 9 letter positions → 2⁹ = 512 unique case variants. Each variant can carry up to `arrayLimit` (100) same-case repeated values without triggering `throwOnLimitExceeded`. An attacker can therefore send:

```
10 case-variants × 100 same-case repetitions = 1000 parameters (within parameterLimit)
→ merged array of 1000 elements (10× the intended cap of 100)
```

**Why downstream checks are insufficient:**

`isRepeatedQueryParameterValidLength` in `utils.js` (line 488) and `buildFilters` (lines 1241–1248) do check array length against `maxRepeatedQueryParameters` and return a 400 error. [3](#0-2) [4](#0-3) 

However, these checks run **after** `requestQueryParser` has already allocated the oversized array in memory. Under high concurrency, thousands of such requests can each hold a 1000-element array in the Node.js heap simultaneously, before any of them reach the validation stage.

### Impact Explanation

Every in-flight request that uses the mixed-case bypass holds a heap-allocated array up to 10× the intended size. Node.js is single-threaded with a shared heap; sustained concurrent flooding causes heap growth that can trigger OOM kills or severe GC pressure, degrading or halting the REST API. Because the REST API is the primary interface for querying mirror node state, this maps directly to the stated scope: network inability to confirm new transactions via the mirror node. [5](#0-4) 

### Likelihood Explanation

No authentication or special privilege is required. The attack requires only the ability to send HTTP GET requests with a crafted query string — trivially achievable from any HTTP client. The payload is deterministic and repeatable. A single attacker with a modest number of concurrent connections (e.g., 500–1000) can sustain the memory pressure continuously. No rate-limiting or request-size guard specific to this bypass path exists in the codebase. [2](#0-1) 

### Recommendation

Add an explicit size guard **inside the merge loop**, before any array is grown, so that the total merged length for a single lowercased key never exceeds `config.query.maxRepeatedQueryParameters`:

```js
if (lowerKey in caseInsensitiveQueryString) {
  const merged = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
  if (merged.length > config.query.maxRepeatedQueryParameters) {
    throw new RangeError(`Parameter '${lowerKey}' exceeds maximum repetition count`);
  }
  caseInsensitiveQueryString[lowerKey] = merged;
}
```

Additionally, pass `parameterLimit` explicitly in `queryOptions` to cap the total number of keys `qs` will parse, reducing the surface for this and similar attacks. [1](#0-0)

const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);

      // handle repeated values, merge into an array

    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;
};
