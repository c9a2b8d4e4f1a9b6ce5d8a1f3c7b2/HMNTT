### Title
Missing Connection Lifecycle Controls in `grpcServerConfigurer` Enable HTTP/2 Frame-Level Resource Exhaustion

### Summary
The `grpcServerConfigurer` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only a thread-pool executor and a per-connection concurrent-call cap. It omits all connection lifecycle controls (`maxConnectionAge`, `maxConnectionIdle`) and any connection-rate limiting, leaving the HTTP/2 transport layer fully exposed to a reconnect-and-malform loop that forces repeated GOAWAY processing and connection teardown without any throttle.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35; backed by `NettyProperties.java` line 14.

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // = 5
};
```

A grep across every Java file under `grpc/` for `maxConnectionAge`, `maxConnectionIdle`, `maxInboundMessageSize`, `keepAlive`, `connectionRate`, and `flowControl` returns **zero matches**. `NettyProperties` exposes only `maxConcurrentCallsPerConnection`.

**Root cause / failed assumption:**
The design assumes that limiting concurrent gRPC *calls* per connection (5) is sufficient to bound server work. This assumption fails because HTTP/2 frame processing occurs at the Netty codec layer, *before* any gRPC call is established. Malformed HTTP/2 frames (invalid SETTINGS, HPACK-compressed HEADERS with bad state, frames with illegal stream IDs, etc.) trigger connection-level errors and GOAWAY at the transport layer — entirely bypassing the per-call cap. Without `maxConnectionAge`/`maxConnectionIdle`, connections can be held open or torn down and re-established at will, and without a connection-rate limit, the cycle is unbounded.

**Exploit flow:**
1. Attacker opens a raw TCP connection to port 5600.
2. Sends a valid HTTP/2 client preface (`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`).
3. Sends a malformed HTTP/2 frame (e.g., a SETTINGS frame with an unknown critical flag, or a HEADERS frame referencing a server-initiated even stream ID).
4. Netty's HTTP/2 codec emits a GOAWAY frame with the appropriate error code and closes the connection — consuming I/O thread time, allocating/deallocating channel state, and logging the error.
5. Attacker immediately reconnects (no application-level connection rate limit exists) and repeats from step 2.
6. Scaled across many source IPs or using a single host with rapid reconnects, this saturates the I/O thread pool and connection-accept queue, starving legitimate clients.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection = 5` — scoped to gRPC RPC calls; irrelevant to HTTP/2 frame-level errors that occur before any call is created.
- GCP gateway `maxRatePerEndpoint: 250` — infrastructure-level, only active when `gateway.gcp.enabled: true`; not an application control; does not limit raw TCP connection establishment rate for malformed-frame attacks; bypassable if the pod is reachable directly (e.g., in non-GCP deployments, dev/test environments, or via internal cluster access).
- No OS-level TCP backlog or JVM-level Netty `SO_BACKLOG` tuning is present in the configurer to compensate.

### Impact Explanation
Each malformed-frame cycle forces the server to: parse the frame, detect the protocol violation, construct and write a GOAWAY frame, flush the channel, and release all connection state. Under sustained attack this exhausts Netty I/O threads and the `applicationTaskExecutor` thread pool, causing legitimate `subscribeToTopics` / `getNodes` gRPC calls to queue indefinitely or be rejected. The result is a network partition from the perspective of real clients — they receive connection refused or timeout errors while the server is busy processing attacker-controlled teardown cycles. Severity is **High** (availability impact, no authentication required, publicly reachable port).

### Likelihood Explanation
The attack requires only a TCP socket and knowledge of the HTTP/2 preface — no credentials, no API key, no prior knowledge of the application. HTTP/2 frame fuzzing tools (e.g., `h2spec`, `nghttp2`) are freely available and can drive thousands of malformed-frame connections per second from a single host. The absence of any application-level connection rate limit or lifecycle bound makes this trivially repeatable and scalable. Likelihood is **High**.

### Recommendation
In `grpcServerConfigurer`, add the following `NettyServerBuilder` controls (expose them as configurable properties in `NettyProperties`):

```java
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
serverBuilder.maxConnectionAgeGrace(nettyProperties.getMaxConnectionAgeGrace(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
serverBuilder.maxInboundMessageSize(nettyProperties.getMaxInboundMessageSize());
serverBuilder.maxInboundMetadataSize(nettyProperties.getMaxInboundMetadataSize());
```

Reasonable defaults: `maxConnectionAge` = 300 s, `maxConnectionIdle` = 60 s, `maxInboundMessageSize` = 4 MB, `maxInboundMetadataSize` = 8 KB. Additionally, add a Netty `ChannelHandler` or a gRPC `ServerInterceptor` that tracks per-IP connection error rates and drops repeat offenders, independent of the infrastructure gateway.

### Proof of Concept
```bash
# Prerequisites: nghttp2 installed (apt install nghttp2-client)

# 1. Confirm no connection rate limit — open 500 connections rapidly
for i in $(seq 1 500); do
  # Send HTTP/2 preface then an invalid SETTINGS frame (unknown critical flag 0xFF)
  printf 'PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n\x00\x00\x06\x04\xFF\x00\x00\x00\x00\x00\x03\x00\x00\x00\x64' \
    | nc -q1 <grpc-host> 5600 &
done
wait

# 2. Observe server-side: I/O threads spike, legitimate clients time out
# grpcurl (legitimate client) will hang or return "connection refused":
grpcurl -plaintext <grpc-host>:5600 list
# Expected under attack: deadline exceeded / connection refused

# 3. h2spec can drive systematic HTTP/2 protocol violations:
h2spec -h <grpc-host> -p 5600 --timeout 2 generic/3.5   # HTTP/2 connection preface errors
h2spec -h <grpc-host> -p 5600 --timeout 2 http2/6.9     # WINDOW_UPDATE flow control errors
```
