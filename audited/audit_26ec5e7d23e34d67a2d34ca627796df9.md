### Title
`Optional.empty()` Negative-Cache Poisoning in `findRuntimeBytecode()` Allows Indefinite Missing-Bytecode Export

### Summary
`ContractRepository.findRuntimeBytecode()` uses `@Cacheable(unless = "#result == null")`, but the method returns `Optional<byte[]>`. An `Optional.empty()` result is not `null`, so it passes the `unless` guard and is stored in the Caffeine cache with `expireAfterAccess=1h`. Any unprivileged caller who queries a contract's bytecode during the brief window between contract entity creation and `runtime_bytecode` row insertion will poison the cache with an empty result. Because the policy is `expireAfterAccess`, repeated queries reset the TTL, keeping the stale empty entry alive indefinitely and causing the mirror node to export a missing-bytecode record for that contract forever.

### Finding Description
**Exact location:**
- `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18
- Cache TTL configured in `web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 22

**Root cause:**

```java
// ContractRepository.java line 16-18
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

The `unless` SpEL expression is `#result == null`. Spring evaluates this against the actual return value of the method. When no row exists (or when `runtime_bytecode` is `NULL`), Spring Data JPA wraps the absence in `Optional.empty()`. `Optional.empty()` is a non-null Java object, so `#result == null` evaluates to `false`, and the empty Optional **is written into the cache**.

The cache specification is:
```
expireAfterAccess=1h,maximumSize=1000,recordStats
```
`expireAfterAccess` resets the TTL on every read. An attacker who polls the endpoint once per hour keeps the poisoned entry alive without bound.

**Exploit flow:**

1. Attacker monitors the network (or the Hedera consensus stream) for a `ContractCreate` transaction that has been accepted but whose sidecar record (containing `runtime_bytecode`) has not yet been committed to the `contract` table by the importer.
2. During that sub-second to multi-second window, attacker sends an `eth_getCode` (or any JSON-RPC call that resolves through `ContractBytecodeReadableKVState.readFromDataSource()`) for the new contract address.
3. `findRuntimeBytecode(contractId)` executes the SQL query, finds no `runtime_bytecode`, returns `Optional.empty()`.
4. Spring's `@Cacheable` proxy checks `#result == null` → `false` → stores `Optional.empty()` under key `contractId` in the Caffeine cache.
5. The importer subsequently inserts the real bytecode into the `contract` table, but the cache is never invalidated (no `@CacheEvict` exists anywhere for this cache).
6. All future calls to `findRuntimeBytecode(contractId)` return the cached `Optional.empty()` for up to 1 hour per access window.
7. Attacker (or any client) re-queries every ~59 minutes to reset `expireAfterAccess`, keeping the poisoned entry alive indefinitely.

**Why existing checks fail:**

The only guard is `unless = "#result == null"`. This is semantically wrong for `Optional`-returning methods. The correct expression to also exclude empty Optionals would be `unless = "#result == null || !#result.isPresent()"`. There is no `@CacheEvict` or `@CachePut` on any write path for this cache, so once poisoned, the entry cannot be corrected without a service restart or cache expiry.

### Impact Explanation
Any contract deployed on the network can be permanently rendered as "no bytecode" from the mirror node's perspective. `eth_getCode` returns `0x` (empty), making the contract indistinguishable from an EOA. DApps, block explorers, and downstream integrations that rely on the mirror node's web3 API will see the contract as non-existent. The mirror node exports this missing-bytecode record to consumers, propagating the incorrect state. Severity: **Medium–High** (data integrity / availability for all users of the affected contract address).

### Likelihood Explanation
No privileges are required. The attacker only needs to know a contract's address at deployment time, which is deterministic from the deployer's account and nonce and is visible on-chain within milliseconds of consensus. The race window (entity creation before bytecode insertion) is a normal part of the importer's two-phase write and is reliably reproducible. The attack is repeatable for every new contract deployment and requires only standard JSON-RPC calls.

### Recommendation
Fix the `unless` condition to correctly exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

Additionally, consider changing the cache policy from `expireAfterAccess` to `expireAfterWrite` so that even a legitimately cached empty result expires on a fixed schedule rather than being refreshable indefinitely.

### Proof of Concept
1. Deploy a contract on a Hedera testnet node connected to a mirror node running this code.
2. Immediately after the `ContractCreate` transaction reaches consensus (visible via the consensus stream), before the importer's sidecar processing completes, send:
   ```
   curl -X POST http://<mirror-node>:8545 \
     -H 'Content-Type: application/json' \
     -d '{"jsonrpc":"2.0","method":"eth_getCode","params":["<contract_evm_address>","latest"],"id":1}'
   ```
3. Observe the response returns `"result":"0x"` (empty bytecode).
4. Wait for the importer to finish writing `runtime_bytecode` to the `contract` table (verify directly in the DB).
5. Re-send the same `eth_getCode` request — it still returns `"result":"0x"` from cache.
6. Confirm the cache entry persists by repeating step 5 every 59 minutes; the result never updates.
7. Restart the web3 service (clears in-memory Caffeine cache) — the correct bytecode is now returned, confirming the root cause is the stale cache entry.
