### Title
Silent Discard of First Upper Bound in `Bound` Constructor Causes Missing Records in Token Airdrop Queries

### Summary
The `Bound` constructor in `Bound.java` iterates all parameters and unconditionally overwrites `upper` with each successive upper-bound parameter, silently discarding all but the last. When a caller sends `token_id=lte:X&token_id=lt:Y` (two upper bounds) to `/api/v1/accounts/{id}/airdrops/outstanding`, only the last upper bound (`lt:Y`) is stored and applied to the SQL query. The `verifySingleOccurrence()` guard that would catch this exists on `Bound` but is never invoked in the TokenAirdrop code path, unlike the NftAllowance path where it is called.

### Finding Description

**Root cause — `Bound` constructor, lines 47–54:**

```java
for (var param : params) {
    if (param.hasLowerBound()) {
        lower = param;
    } else if (param.hasUpperBound()) {
        upper = param;   // last writer wins; earlier upper bound is lost
    }
    cardinality.merge(param.operator(), 1, Math::addExact);
}
``` [1](#0-0) 

When `token_id=lte:1000&token_id=lt:500` is supplied, the loop runs twice:
1. `lte:1000` → `upper = lte:1000`, cardinality `{LTE:1}`
2. `lt:500` → `upper = lt:500` (overwrites), cardinality `{LTE:1, LT:1}`

Only `lt:500` survives. Records with `500 ≤ token_id ≤ 1000` are excluded from the query result.

**Guard that exists but is not called:**

`Bound.verifySingleOccurrence()` checks `getCardinality(LT, LTE) > 1` and throws `IllegalArgumentException`: [2](#0-1) 

This guard is called in `NftAllowanceServiceImpl.verifyRangeId()`: [3](#0-2) 

But `grep` confirms `verifySingleOccurrence` is **not** referenced anywhere in the TokenAirdrop service or controller path. The `TokenAirdropServiceImpl` has no equivalent `verifyRangeId` call.

**Other checks are insufficient:**

- The `primarySortField && adjustedLower > adjustedUpper` check at line 58 only fires when `primarySortField=true`; `token_id` is always passed as a secondary bound (`primarySortField=false`), so this never triggers. [4](#0-3) 

- `verifyEqualOrRange()` only checks for mixing EQ with range operators, not for duplicate upper bounds. [5](#0-4) 

### Impact Explanation
An unprivileged external user can cause the mirror node to silently return an incomplete set of outstanding airdrops. Any `token_id` values between the discarded upper bound and the surviving upper bound are excluded from the response with no error or indication. Downstream clients (wallets, dApps) relying on this endpoint for complete airdrop listings will make decisions on incomplete data. Severity is medium: no privilege escalation or data exfiltration, but correctness of the public API is broken on demand.

### Likelihood Explanation
No authentication or special privilege is required. The attack is a single crafted HTTP GET request. It is trivially repeatable and requires no knowledge beyond the public API documentation. Any user who can query the REST API can trigger this.

### Recommendation
Call `verifySingleOccurrence()` on the `tokenIds` (and `entityIds`, `serialNumbers`) bounds inside `TokenAirdropServiceImpl.getAirdrops()` before passing the request to the repository, mirroring the pattern already used in `NftAllowanceServiceImpl.verifyRangeId()`. This will return a `400 Bad Request` for duplicate upper (or lower) bound operators instead of silently discarding one.

### Proof of Concept

**Precondition:** An account `0.0.100` has outstanding airdrops for token IDs 400, 600, and 800.

**Request:**
```
GET /api/v1/accounts/0.0.100/airdrops/outstanding?token_id=lte:1000&token_id=lt:500
```

**Expected behavior:** Both upper bounds should either be rejected (400) or the stricter one applied (`lt:500`), returning only token 400.

**Actual behavior:**
- `Bound` constructor stores `lte:1000` first, then overwrites with `lt:500`.
- SQL query applies only `token_id < 500`.
- Response returns only token 400 — tokens 600 and 800 are silently missing.
- No error is returned; the response appears normal.

**Variant (reversed order, less restrictive bound wins):**
```
GET /api/v1/accounts/0.0.100/airdrops/outstanding?token_id=lt:500&token_id=lte:1000
```
- `lt:500` is overwritten by `lte:1000`.
- SQL applies `token_id <= 1000`, returning tokens 400, 600, and 800 — more than the first bound intended.
