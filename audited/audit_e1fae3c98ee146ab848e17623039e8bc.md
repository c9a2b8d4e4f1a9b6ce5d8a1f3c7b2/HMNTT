### Title
Global Gas Bucket Starvation via Burst Consumption — No Per-User Isolation in `gasLimitBucket`

### Summary
The `gasLimitBucket` and `rateLimitBucket` in `ThrottleConfiguration.java` are global Spring singleton beans with no per-user or per-IP partitioning. Because `refillGreedy` initializes the bucket at full capacity and the default `requestsPerSecond` (500) multiplied by the maximum scaled gas per request (1,500 tokens) exactly equals the total gas bucket capacity (750,000 tokens), a single unprivileged attacker can exhaust the entire per-second gas budget in one burst, denying service to all other users for the remainder of that second. This is repeatable every second indefinitely.

### Finding Description

**Exact code path:**

`ThrottleConfiguration.java` lines 34–45 — `gasLimitBucket()` creates a single global bucket:
```java
Bucket gasLimitBucket() {
    long gasLimit = throttleProperties.getGasPerSecond();   // 750,000 scaled tokens
    final var limit = Bandwidth.builder()
            .capacity(gasLimit)
            .refillGreedy(gasLimit, Duration.ofSeconds(1))  // starts FULL, refills continuously
            .build();
    return Bucket.builder()
            .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
            .addLimit(limit)
            .build();
}
```

`ThrottleManagerImpl.java` lines 37–42 — both checks are against the same global singletons for every caller:
```java
if (!rateLimitBucket.tryConsume(1)) { ... }
else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) { ... }
```

**Root cause:** `refillGreedy` initializes the bucket at full capacity. There is no per-IP, per-account, or per-session bucket — `gasLimitBucket` and `rateLimitBucket` are both application-scoped singletons shared across all concurrent callers.

**Failed assumption:** The design assumes the global `requestsPerSecond` cap (500 RPS) prevents any single user from exhausting the gas budget. However, since both limits are global and unenforced per-caller, a single attacker consumes all 500 request slots AND all gas tokens simultaneously.

**Exploit arithmetic:**
- `gasPerSecond` default = 7,500,000,000 → `scaleGas()` → **750,000 tokens** (`ThrottleProperties.java` line 26, 42–46)
- `maxGasLimit` = 15,000,000 → `scaleGas()` → **1,500 tokens per request** (docs line 710)
- `requestsPerSecond` = **500** (`ThrottleProperties.java` line 35)
- 500 requests × 1,500 tokens = **750,000 tokens = 100% of gas capacity**

A single attacker sending 500 concurrent requests each with `gas=15000000` drains both buckets to zero simultaneously.

### Impact Explanation
All legitimate users receive HTTP 429 (`GAS_PER_SECOND_LIMIT_EXCEEDED` or `REQUEST_PER_SECOND_LIMIT_EXCEEDED`) for the remainder of each second. Because `refillGreedy` refills continuously, the attacker can re-burst at the start of every new second, sustaining the denial of service indefinitely. The `gasLimitRefundPercent=100` restore path (`ThrottleManagerImpl.java` lines 59–64) only helps if the attacker's transactions complete and gas is returned — but the attacker can intentionally submit failing or long-running calls to delay or prevent restoration.

### Likelihood Explanation
No authentication, API key, or account is required to call the contract endpoint. The attacker needs only an HTTP client capable of sending 500 concurrent requests per second — trivially achievable with tools like `wrk`, `hey`, or a simple async script. The attack is stateless, requires no on-chain funds, and is repeatable every second from a single machine or a small botnet.

### Recommendation
1. **Add per-IP (or per-caller) rate limiting** using a `ConcurrentHashMap<String, Bucket>` keyed on the client IP (extracted from `X-Forwarded-For` or `RemoteAddr`), so each caller has its own gas and request budget.
2. **Cap the per-request gas contribution** to a fraction of the total budget (e.g., no single request may consume more than 10% of `gasPerSecond`) to prevent a small number of max-gas requests from starving others.
3. **Consider `refillIntervally`** instead of `refillGreedy` to prevent mid-second token accumulation from being immediately burst-consumed again.
4. As a short-term mitigation, lower `evm.maxGasLimit` relative to `throttle.gasPerSecond` so that no single request can consume a disproportionate share of the bucket.

### Proof of Concept
```bash
# Prerequisites: curl, GNU parallel or wrk; no auth required
# Send 500 concurrent max-gas contract call requests in one burst

seq 500 | parallel -j500 curl -s -X POST http://<mirror-node-web3>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000001","gas":15000000}'

# Immediately after (within same second), legitimate user gets:
# HTTP 429: "Gas per second rate limit exceeded."
# or
# HTTP 429: "Requests per second rate limit exceeded"

# Repeat every ~1 second to sustain DoS
```

    private static final long GAS_SCALE_FACTOR = 10_000L;

    @Min(0)
    @Max(100)
    private float gasLimitRefundPercent = 100;

    @Min(21_000)
    @Max(10_000_000_000_000L)
    private long gasPerSecond = 7_500_000_000L;

    @Min(1)
    private long opcodeRequestsPerSecond = 1;

    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
