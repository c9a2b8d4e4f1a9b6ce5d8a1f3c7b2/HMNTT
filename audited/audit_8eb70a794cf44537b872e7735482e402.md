### Title
Missing Application-Level Rate Limiting on `GET /api/v1/topics/{id}` Enables Thread Pool Exhaustion

### Summary
The `getTopic()` handler in `TopicController` accepts unlimited unauthenticated requests with no application-level rate limiting. Each request triggers three synchronous database queries, and the rest-java module contains no throttle filter or bucket-based limiter of any kind. A single unprivileged attacker can saturate the embedded Tomcat thread pool, making the endpoint unresponsive for all legitimate users at zero cost.

### Finding Description
**Exact code path:**

`rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 31–37:

```java
@GetMapping(value = "/{id}")
Topic getTopic(@PathVariable EntityIdNumParameter id) {
    var topic = topicService.findById(id.id());       // DB query 1
    var entity = entityService.findById(id.id());     // DB query 2
    var customFee = customFeeService.findById(id.id()); // DB query 3
    return topicMapper.map(customFee, entity, topic);
}
```

**Root cause:** The rest-java module has no rate-limiting infrastructure applied to this endpoint. The only registered servlet filters are `LoggingFilter` (logs requests) and `MetricsFilter` (records byte counts), and `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter. `WebMvcConfiguration` adds only argument resolvers. No `ThrottleConfiguration`, `ThrottleManager`, or Bucket4j bean exists anywhere in the rest-java module — these exist exclusively in the `web3` module (`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`), which is a separate Spring Boot application.

**Failed assumption:** The design assumes that infrastructure-level controls (ingress/load balancer) will enforce rate limits before requests reach the application. No such enforcement is visible in the application code, and the application itself provides no fallback.

**Exploit flow:**
1. Attacker sends a continuous flood of `GET /api/v1/topics/0.0.1` requests from one or more IPs.
2. Each request occupies a Tomcat worker thread for the duration of three sequential JDBC calls.
3. Spring Boot's embedded Tomcat defaults to a maximum of 200 worker threads (`server.tomcat.threads.max`). No override is configured in rest-java resources.
4. Once all 200 threads are occupied, new incoming requests queue in the accept backlog and then time out.
5. Legitimate users receive connection timeouts or HTTP 503 responses.

**Why existing checks fail:** `LoggingFilter.doFilterInternal` and `MetricsFilter.doFilterInternal` both call `filterChain.doFilter(request, response)` unconditionally — neither inspects request rate or rejects requests. There is no `OncePerRequestFilter` or `HandlerInterceptor` in the rest-java config directory that performs throttling.

### Impact Explanation
The endpoint becomes unresponsive for all legitimate users during the attack. Because `getTopic()` issues three database queries per request, each blocked thread also holds a JDBC connection from the connection pool, potentially cascading to starve other endpoints in the same application of database connections. The impact is a full denial of service for the rest-java service with no collateral damage to the Hedera network itself, consistent with the "griefing" severity classification.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no financial cost. A single machine running `ab`, `wrk`, or a simple loop of `curl` commands is sufficient. The attack is trivially repeatable and can be sustained indefinitely. The attacker does not need to maintain state between requests.

### Recommendation
1. Add a Bucket4j-based `OncePerRequestFilter` to the rest-java module (mirroring the pattern already used in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`) that enforces a global requests-per-second ceiling and, optionally, a per-IP ceiling.
2. Configure `server.tomcat.threads.max` and `server.tomcat.accept-count` explicitly in rest-java application properties to bound queue depth.
3. Ensure ingress-level rate limiting (e.g., Traefik `middleware.yaml` in `charts/hedera-mirror-rest-java/templates/middleware.yaml`) is enabled and not left as an optional operator concern.

### Proof of Concept
```bash
# Flood the endpoint with 1000 concurrent requests, 50 at a time
ab -n 100000 -c 200 http://<mirror-node-host>/api/v1/topics/0.0.1

# In a separate terminal, observe legitimate requests timing out:
curl -v http://<mirror-node-host>/api/v1/topics/0.0.2
# Expected during attack: connection timeout or HTTP 503
```

No authentication header is required. The `{id}` value `0.0.1` (or any valid/invalid topic ID) is sufficient; even 404 responses consume a thread for the duration of the three service calls.
