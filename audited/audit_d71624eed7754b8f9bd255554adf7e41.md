### Title
Missing Application-Level Rate Limiting on `GET /api/v1/topics/{id}` Enables Connection Pool Exhaustion DoS

### Summary
The `TopicController.getTopic()` handler issues three synchronous, unbounded database calls per request (`topicService.findById`, `entityService.findById`, `customFeeService.findById`) with no application-level rate limiting in the rest-java module. Any unauthenticated external user can flood this endpoint to exhaust the HikariCP connection pool, causing service unavailability for all users. The web3 module has an explicit throttling infrastructure (bucket4j-based `ThrottleConfiguration`) that rest-java entirely lacks.

### Finding Description

**Code path:**

`TopicController.java` lines 31–37 — the handler makes three sequential synchronous DB calls per request:
```java
var topic = topicService.findById(id.id());      // DB call 1
var entity = entityService.findById(id.id());    // DB call 2
var customFee = customFeeService.findById(id.id()); // DB call 3
```

`TopicServiceImpl.java` line 20 — no caching, no async, no throttle:
```java
return topicRepository.findById(id.getId()).orElseThrow(...);
```

**Root cause:** The rest-java module has no application-level rate limiting. A search across all rest-java Java sources confirms zero usage of `RateLimiter`, `Throttle`, or bucket4j. The `WebMvcConfiguration` registers only argument resolvers and formatters; `RestJavaConfiguration` registers only an ETag filter and Protobuf converter — no rate-limiting filter or interceptor exists.

**Contrast with web3:** The web3 module has `ThrottleConfiguration` (bucket4j `rateLimitBucket`, `gasLimitBucket`, `opcodeRateLimitBucket`) and `ThrottleManagerImpl` enforcing per-second limits. The rest-java module has no equivalent.

**Exploit flow:**
1. Attacker sends a high-volume stream of `GET /api/v1/topics/0.0.1` requests (no authentication required, no API key).
2. Each request acquires up to 3 HikariCP connections simultaneously for the duration of the DB queries.
3. The HikariCP pool (finite, default Spring Boot pool size of 10 connections per the rest-java docs: `hiero.mirror.restJava.db.*` has no explicit pool size override, defaulting to HikariCP's default of 10) is exhausted.
4. New requests block waiting for a connection until `connectionTimeout` expires, returning errors to all users.

### Impact Explanation
Complete service unavailability for the rest-java API for all users while the flood is sustained. Because the endpoint is unauthenticated and the path parameter accepts any numeric ID (valid or not — a 404 still consumes a DB connection before throwing `EntityNotFoundException`), the attacker does not need any valid topic IDs. The three-DB-calls-per-request multiplier means the effective connection pressure is 3× the request rate.

### Likelihood Explanation
Trivially exploitable by any external actor with network access to the service. No credentials, tokens, or special knowledge are required. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`, `hey`) can sustain enough concurrent requests to exhaust a 10-connection pool. The attack is repeatable and stateless.

### Recommendation
1. Add a bucket4j-based rate-limiting filter to rest-java, mirroring the `ThrottleConfiguration` already present in the web3 module, applied globally to `/api/v1/**`.
2. Explicitly configure HikariCP pool size (`spring.datasource.hikari.maximum-pool-size`) to a value appropriate for expected concurrency, and document it as a tunable property alongside the existing `hiero.mirror.restJava.db.*` properties.
3. Consider adding a Spring `HandlerInterceptor` or `OncePerRequestFilter` in `WebMvcConfiguration`/`RestJavaConfiguration` that enforces per-IP or global RPS limits before any DB call is made.
4. Cache results of `topicService.findById`, `entityService.findById`, and `customFeeService.findById` for frequently-requested IDs to reduce DB connection pressure per request.

### Proof of Concept
```bash
# Flood the endpoint with 500 concurrent connections, no auth required
wrk -t 10 -c 500 -d 60s http://<rest-java-host>:8080/api/v1/topics/0.0.1

# Expected result after pool exhaustion:
# - Legitimate requests receive 500 errors or hang until connectionTimeout (default varies)
# - HikariCP metrics show hikaricp_connections_pending > 0 and hikaricp_connections_active == max
# - Alert fires: RestJavaHighDBConnections (>75% utilization for 5m)
```
