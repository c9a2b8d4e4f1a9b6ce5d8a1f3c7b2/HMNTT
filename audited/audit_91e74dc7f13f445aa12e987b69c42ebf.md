### Title
Unauthenticated Connection Pool Exhaustion via Concurrent DB Queries in `getOneAccount()`

### Summary
`GET /api/v1/accounts/:id` with the default `transactions=true` parameter fires two concurrent database query chains via `Promise.all([entityPromise, transactionsPromise])`. Because `doGetTransactions()` itself issues two sequential DB queries (timestamp lookup + detail fetch), each inbound HTTP request can hold up to two pool connections simultaneously for up to 20 seconds. With the default pool ceiling of 10 connections and no application-level rate limiting on this endpoint, approximately 5–6 concurrent unauthenticated requests are sufficient to exhaust the pool and cause all subsequent requests to fail with connection-timeout errors.

### Finding Description

**Exact code path:**

`rest/accounts.js`, `getOneAccount()`, lines 487–495:
```js
const entityPromise = pool.queryQuietly(entityQuery, entityParams);          // holds 1 connection
const transactionsPromise = includeTransactions
  ? transactions.doGetTransactions(filters, req, timestampRange)             // holds 1 connection
  : emptyTransactionsPromise;
const [entityResults, transactionResults] = await Promise.all([entityPromise, transactionsPromise]);
```

`rest/transactions.js`, `doGetTransactions()` (lines 695–723) calls `getTransactionTimestamps()` (line 483: `pool.queryQuietly(...)`) and then `getTransactionsDetails()` (line 662: `pool.queryQuietly(...)`), each acquiring a pool connection sequentially. The entity query and the entire `doGetTransactions` chain run concurrently, so at peak a single HTTP request holds **2 connections** simultaneously.

**Pool configuration** (`rest/dbpool.js`, line 14):
```js
max: config.db.pool.maxConnections,   // default: 10
```
Default documented in `docs/configuration.md` line 556:
> `hiero.mirror.rest.db.pool.maxConnections` | **10**

`statementTimeout` defaults to **20 000 ms** and `connectionTimeout` to **20 000 ms**.

**Root cause:** The failed assumption is that 10 pool connections are sufficient when each request can hold 2 connections for up to 20 seconds, and there is no rate-limiting or in-flight-request cap on the endpoint.

**Why existing checks fail:**
- `authHandler` (`rest/middleware/authHandler.js`) only sets a custom response-size limit for authenticated users; it does not reject or throttle unauthenticated requests.
- `statementTimeout` (20 s) bounds individual query duration but is long enough that an attacker maintaining ~5 concurrent requests keeps the pool continuously exhausted.
- No `express-rate-limit`, no in-flight-request middleware, and no per-IP throttle exists in `rest/server.js` for this route.

### Impact Explanation
When the 10-connection pool is exhausted, every new DB call blocks for up to 20 seconds before throwing a connection-timeout error. This affects **all** REST API endpoints sharing the same pool (accounts, transactions, tokens, balances, etc.), not just the attacked route. The result is a complete, sustained denial of service for all legitimate users of the REST layer for as long as the attacker maintains the request stream.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no amplification beyond a standard HTTP client. Five to six concurrent long-running `GET /api/v1/accounts/:id` requests (targeting an account with a large transaction history to maximize query duration) are sufficient. The attack is trivially repeatable and scriptable with `curl`, `ab`, or any load-testing tool. Default deployments (pool size 10, no rate limiting) are directly vulnerable.

### Recommendation
1. **Add a per-IP rate limit** on the `/api/v1/accounts/:id` route using `express-rate-limit` or an ingress-level rate limiter.
2. **Add an in-flight request cap** (e.g., `express-slow-down` or a semaphore) to bound the number of concurrent DB-bound requests.
3. **Increase the default pool size** or document a minimum safe value relative to expected concurrency.
4. **Reduce `statementTimeout`** for the transactions sub-query to limit how long a single connection can be held.
5. **Consider making `transactions=false` the default** for `GET /accounts/:id`, requiring callers to opt in to the heavier query.

### Proof of Concept
```bash
# Target an account with a large transaction history (e.g., 0.0.98 on mainnet)
# Run 6 concurrent requests; each holds 2 pool connections for up to 20 s
for i in $(seq 1 6); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.98" &
done
wait

# While the above is running, observe that all other API calls time out:
curl -v "https://<mirror-node>/api/v1/transactions"
# Expected: connection timeout / 500 after ~20 s
```
