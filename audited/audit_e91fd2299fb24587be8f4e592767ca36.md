### Title
Silent Heartbeat Write Failure Enables Liveness-Probe DoS via /tmp Exhaustion

### Summary
The pinger's heartbeat goroutine in `pinger/main.go` writes a timestamp to `/tmp/alive` every 15 seconds but silently discards the error with `_ =`. If `/tmp` is filled to capacity — achievable by any process or user with write access to the pod's writable `/tmp` mount — the write fails without any log or recovery, the heartbeat file goes stale, and the Kubernetes liveness probe kills and restarts the pod after ~50 seconds.

### Finding Description
**Exact code path:**

`pinger/main.go`, lines 28–38 — the heartbeat goroutine:
```go
_ = os.WriteFile("/tmp/alive", []byte(time.Now().Format(time.RFC3339Nano)), 0644)
```
The return value of `os.WriteFile` is unconditionally discarded. No log, no retry, no fallback.

`pinger/cmd/healthcheck/main.go`, lines 27–38 — the liveness probe binary:
```go
fi, err := os.Stat("/tmp/alive")
...
age := time.Since(fi.ModTime())
if age > 2*time.Minute {
    fmt.Fprintln(os.Stderr, "heartbeat stale")
    os.Exit(1)
}
```
The probe exits 1 if `/tmp/alive` is missing or its mtime is older than 2 minutes.

**Root cause:** The failed assumption is that `os.WriteFile` to `/tmp` will always succeed. When `/tmp` is full, the syscall returns `ENOSPC`; the error is thrown away; the file's mtime freezes; the probe starts failing.

**Exploit flow:**
1. `/tmp` is a writable mount inside the container (required because `readOnlyRootFilesystem: true` is set in `values.yaml` line 84, yet the pinger must write to `/tmp` — so an emptyDir or tmpfs must be mounted there).
2. `values.yaml` defines **no `sizeLimit`** on any volume for `/tmp`, so the only bound is node disk/memory.
3. An attacker with write access to the pod's `/tmp` (e.g., via `kubectl exec`, a compromised sidecar, or on a bare-metal/VM deployment where `/tmp` is world-writable) creates large files until the filesystem is full.
4. The next `os.WriteFile("/tmp/alive", ...)` call returns `ENOSPC`; the error is silently dropped.
5. After 2 minutes without an mtime update, `healthcheck live` exits 1.
6. With `failureThreshold: 5` and `periodSeconds: 10` (`values.yaml` lines 38–44), Kubernetes kills the pod after 5 × 10 = 50 seconds of consecutive failures.

**Why existing checks fail:**
- `readOnlyRootFilesystem: true` protects the image layers but not the writable `/tmp` mount.
- `runAsNonRoot: true` / `runAsUser: 1000` does not prevent the pod's own UID from writing to its own `/tmp`.
- No `sizeLimit` is set on the emptyDir/tmpfs backing `/tmp`, so there is no kernel-enforced quota.
- The goroutine has no error-handling path; it cannot alert, retry with a different path, or self-terminate cleanly.

### Impact Explanation
The direct result is a forced pod restart (liveness probe kill). Because `replicas: 1` (`values.yaml` line 68), there is no redundancy: the pinger stops submitting heartbeat transactions to the Hedera network for the duration of the restart cycle. This is a repeatable availability disruption (griefing/DoS) with no economic damage to network users, matching the stated Medium severity scope.

### Likelihood Explanation
On a bare-metal or VM deployment, `/tmp` is world-writable by default (sticky bit, mode 1777); any local unprivileged user can run `dd if=/dev/zero of=/tmp/fill bs=1M` to exhaust it. In Kubernetes, the attacker needs `kubectl exec` into the pod or a co-located container sharing the `/tmp` volume — a realistic scenario in multi-tenant clusters or when a sidecar is compromised. The attack is trivially repeatable: after each pod restart, `/tmp` is cleared, but the attacker can immediately re-fill it.

### Recommendation
1. **Check and log the error** — replace `_ = os.WriteFile(...)` with explicit error handling that at minimum logs the failure, and optionally triggers a controlled shutdown so Kubernetes restarts cleanly rather than waiting for the stale-heartbeat timeout.
2. **Set a `sizeLimit` on the `/tmp` emptyDir** in the Helm chart template (e.g., `sizeLimit: 1Mi`) so the kernel enforces a quota and prevents unbounded growth.
3. **Use `os.Chtimes`** to update only the mtime of the existing file rather than rewriting it, eliminating the need for free space entirely.
4. **Add a dedicated tmpfs with a tight size limit** for `/tmp` in the pod spec, separate from any shared volumes.

### Proof of Concept
```bash
# 1. Exec into the running pinger pod
kubectl exec -it <pinger-pod> -- /bin/sh

# 2. Fill /tmp until ENOSPC
dd if=/dev/zero of=/tmp/fill bs=1M   # runs until "No space left on device"

# 3. Wait — the heartbeat goroutine silently fails every 15 s
# 4. After ~2 minutes, observe liveness probe failures in pod events:
kubectl describe pod <pinger-pod>
# Events: Liveness probe failed: heartbeat stale

# 5. After 5 failures (~50 s), Kubernetes restarts the pod
# kubectl get pod <pinger-pod>  shows RESTARTS counter increment
```
