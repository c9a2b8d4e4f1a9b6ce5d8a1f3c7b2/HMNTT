### Title
Unauthenticated Connection Pool Exhaustion via `/api/v1/balances` Causes Cascading HTTP 503 DoS

### Summary
An unprivileged external attacker can flood `/api/v1/balances` with high-concurrency requests, exhausting the bounded `node-postgres` connection pool (`max: config.db.pool.maxConnections`). When the pool is exhausted, `pool.queryQuietly()` catches the pg connection-timeout error and re-throws it as `DbError`, which the error middleware maps to HTTP 503 Service Unavailable. Because the REST API Helm chart ships with no per-IP concurrency limit (`inFlightReq`) and no rate limit, and because `global.middleware` defaults to `false` (disabling even the circuit breaker), there is no in-path control to prevent this from affecting all concurrent legitimate users, including the Kubernetes readiness probe.

### Finding Description

**Code path — pool configuration:**
`rest/dbpool.js` lines 7–16 initializes the global `pool` with `max: config.db.pool.maxConnections` and `connectionTimeoutMillis: config.db.pool.connectionTimeout`. [1](#0-0) 

**Code path — `getBalances()` multi-query fan-out:**
A single request to `/api/v1/balances?timestamp=lte:X&account.publickey=Y` causes `getBalances()` to issue up to **three sequential** `pool.queryQuietly()` calls:
1. `getAccountBalanceTimestampRange` → `pool.queryQuietly()` (line 189)
2. `getAccountIdsByPublicKey` → `pool.queryQuietly()` (line 66)
3. Main balance query → `pool.queryQuietly()` (line 153)

Each call holds a pool connection for its full execution duration. [2](#0-1) 

**Code path — error propagation:**
`queryQuietly` wraps every pg error (including `Error: timeout exceeded when trying to connect`) in `throw new DbError(err.message)`. [3](#0-2) 

**Code path — HTTP 503 mapping:**
`handleError` maps `DbError` → `httpStatusCodes.SERVICE_UNAVAILABLE` (HTTP 503). [4](#0-3) 

**Root cause — absent rate/concurrency controls on the REST API:**
The `hedera-mirror-rest` Helm chart middleware section contains only a `circuitBreaker` and a `retry: attempts: 10` entry — **no `inFlightReq` and no `rateLimit`**. [5](#0-4) 

Furthermore, `global.middleware` defaults to `false`, meaning the Traefik middleware chain (including the circuit breaker) is **not rendered at all** by default. [6](#0-5) 

Contrast with the Rosetta chart, which explicitly sets `inFlightReq: amount: 5` per IP and `rateLimit: average: 10`. [7](#0-6) 

**Secondary impact — readiness probe failure:**
`health.js` `readinessCheck` calls `pool.query()` on the same global pool. If the pool is exhausted, this call also fails, causing Kubernetes to mark the pod as not ready and stop routing traffic entirely. [8](#0-7) 

**Retry amplification:**
When `global.middleware` is enabled, the `retry: attempts: 10` middleware causes Traefik to retry each 503 response up to 10 times, amplifying the attacker's pool pressure by 10×. [9](#0-8) 

### Impact Explanation
All concurrent legitimate users of any REST API endpoint (not just `/api/v1/balances`) receive HTTP 503 for the duration of the attack, because all endpoints share the same global `pool`. If the readiness probe also fails, Kubernetes removes the pod from the load-balancer backend, causing a complete service outage for that replica. The mirror node reports unavailability (`DbError` → 503) outside its designed degradation path (which is intended for actual database failures, not attacker-induced pool starvation). Severity: **High** (availability impact, no authentication required, affects all endpoints).

### Likelihood Explanation
The attack requires only the ability to send concurrent HTTP GET requests — no credentials, no special knowledge, no exploit chain. The endpoint is publicly accessible. The attacker needs to sustain concurrent requests above `maxConnections`; with `statement_timeout` providing a natural connection-release cadence, the attacker simply needs to replenish requests at the same rate. This is trivially scriptable with `curl`, `ab`, `wrk`, or any HTTP load tool. The attack is repeatable and stateless.

### Recommendation
1. **Add `inFlightReq` per-IP concurrency limiting** to the `hedera-mirror-rest` Helm chart middleware, matching the Rosetta chart pattern (`amount: 5`, `ipStrategy.depth: 1`).
2. **Add `rateLimit`** (e.g., `average: 100`, `sourceCriterion: requestHost: true`) to the REST middleware chain.
3. **Enable `global.middleware: true`** in production deployments so the circuit breaker and other middleware are actually applied.
4. **Add application-level concurrency semaphore** in `getBalances()` (and other handlers) to cap in-flight DB queries independently of the ingress layer.
5. **Reserve pool connections** for the readiness probe (e.g., use a separate single-connection pool for health checks) so pool exhaustion cannot cause false readiness failures.

### Proof of Concept
```bash
# Precondition: mirror node running, pool.maxConnections = N (e.g., 10)
# Step 1: Flood with concurrent requests holding connections via timestamp+pubkey path
for i in $(seq 1 200); do
  curl -s "http://<mirror-node>/api/v1/balances?timestamp=lte:9999999999.000000000&account.publickey=<valid_pubkey>" &
done
wait

# Step 2: Immediately send a legitimate request
curl -v "http://<mirror-node>/api/v1/balances"
# Expected result during flood: HTTP 503 {"_status":{"messages":[{"message":"Service Unavailable"}]}}

# Step 3: Verify readiness probe also fails
curl -v "http://<mirror-node>/health/readiness"
# Expected result: HTTP 503 (pod marked NotReady by Kubernetes)
```
