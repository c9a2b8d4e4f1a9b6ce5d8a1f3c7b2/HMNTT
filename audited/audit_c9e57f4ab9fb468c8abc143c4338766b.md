### Title
Unauthenticated REST Address Book Poisoning Enables Attacker-Controlled Node Routing and Fund Drain

### Summary
The monitor's `NodeSupplier.getAddressBook()` fetches node data from the mirror node REST API without any cryptographic verification of the response. An attacker who can intercept or spoof that HTTP response (DNS poisoning, MITM) can inject `NodeProperties` containing attacker-controlled endpoints, account IDs, and TLS certificate hashes. These properties flow directly into `TransactionPublisher.toClient()`, causing all operator transactions to be routed to attacker-controlled infrastructure, and causing the node-validation transfer to deposit funds into the attacker's account.

### Finding Description

**Code path:**

`NodeSupplier.getAddressBook()` (lines 112–124) calls `restApiClient.getNodes()` and pipes the raw REST response through `toNodeProperties()` (lines 126–145) with no signature or integrity check: [1](#0-0) 

`toNodeProperties()` blindly maps every field from the REST response — including `host`, `nodeAccountId`, `nodeCertHash`, and `port` — into a `NodeProperties` object: [2](#0-1) 

The attacker-supplied `certHash` is then embedded into the protobuf `NodeAddress` used to configure TLS verification: [3](#0-2) 

`TransactionPublisher.toClient()` calls `client.setNetworkFromAddressBook()` and `client.setNetwork()` using these attacker-controlled values, then attaches the real operator credentials: [4](#0-3) 

**Why the `validateNode()` check is insufficient:**

`validateNode()` sends a 1-tinyhbar `TransferTransaction` to `nodeAccountId` and checks for a `SUCCESS` receipt: [5](#0-4) 

The attacker's server acts as a transparent proxy: it forwards the signed transaction to the real Hedera network (which processes it legitimately, crediting the attacker's account), then returns the real `SUCCESS` receipt. The SDK has no way to distinguish this from a genuine node. The node passes validation and is added to the live node list.

**Exploit flow:**

1. Attacker poisons DNS (or performs MITM) for the configured mirror node REST endpoint.
2. Attacker returns a crafted `/api/v1/network/nodes` response containing:
   - `host` / `ipAddressV4`: attacker's server
   - `nodeAccountId`: attacker-controlled Hedera account (e.g. `0.0.99999`)
   - `nodeCertHash`: SHA-384 hash of attacker's own TLS certificate
3. `getAddressBook()` → `toNodeProperties()` creates `NodeProperties` with these values.
4. `validateNode()` builds a client via `NodeSupplier.toClient()` pointing to the attacker's server, sends a `TransferTransaction` crediting `0.0.99999`.
5. Attacker proxies the transaction to the real network; real network returns `SUCCESS`; attacker's account receives 1 tinyhbar.
6. Node is added to `nodes` list.
7. `TransactionPublisher.getClients()` calls `nodeSupplier.refresh()`, which returns the poisoned nodes, and `toClient()` builds all publish clients pointing to the attacker's server with the attacker's cert hash accepted for TLS.
8. Every subsequent operator transaction is routed through the attacker's infrastructure.

### Impact Explanation

- **Direct fund drain**: Every node-validation cycle (configurable frequency) transfers 1 tinyhbar from the operator account to the attacker's Hedera account. With a short validation interval this accumulates.
- **Full transaction interception**: All publish transactions (including `TransactionRecordQuery` and `TransactionReceiptQuery` carrying fee payments) pass through the attacker's server. The attacker can log, delay, selectively drop, or replay them.
- **TLS bypass**: Because `certHash` originates from the same untrusted source, the SDK's certificate-pinning mechanism is completely neutralized — the attacker's TLS certificate is accepted as legitimate.

Severity: **High** — operator private key is not directly exposed, but funds are drained and all transaction traffic is compromised.

### Likelihood Explanation

- DNS poisoning of an internal or cloud-hosted mirror node REST endpoint is a realistic attacker capability in misconfigured environments (no DNSSEC, shared DNS infrastructure).
- MITM is feasible if the REST client does not enforce certificate pinning on the mirror node REST connection itself.
- The attack is repeatable: the `Flux.interval`-driven validation loop re-runs on every cycle, re-validating (and re-draining) on each iteration.
- No privileged access to the monitor process or Hedera credentials is required.

### Recommendation

1. **Verify the address book cryptographically**: Compare the fetched node data against the on-chain address book (file `0.0.101` / `0.0.102`) obtained via a trusted, pinned gRPC connection, not the unauthenticated REST API.
2. **Pin the mirror node REST TLS certificate** or enforce HTTPS with a known CA bundle so MITM of the REST endpoint is not possible.
3. **Do not accept `certHash` from the REST response**: derive it from the on-chain address book file, which is consensus-verified.
4. **Validate `nodeAccountId` against a static allowlist** of known network account IDs before using it as a transfer target in `validateNode()`.

### Proof of Concept

```
# 1. Stand up a fake mirror REST API server returning:
POST /api/v1/network/nodes →
{
  "nodes": [{
    "node_account_id": "0.0.99999",   // attacker-controlled account
    "node_cert_hash": "<SHA384 of attacker TLS cert>",
    "node_id": 0,
    "service_endpoints": [{"ip_address_v4": "<attacker IP>", "port": 50211}]
  }]
}

# 2. Stand up a fake Hedera node at <attacker IP>:50211 that:
#    - Accepts gRPC connections with attacker's TLS cert
#    - Proxies all transactions to a real Hedera testnet node
#    - Returns real receipts (so validateNode() passes)

# 3. Poison DNS so the monitor's configured mirror REST hostname resolves to attacker IP.

# 4. Restart (or wait for refresh interval of) the monitor.
#    NodeSupplier.refresh() → getAddressBook() fetches attacker's response.
#    validateNode() sends 1 tinyhbar to 0.0.99999 → SUCCESS (proxied).
#    Node added to live list.

# 5. TransactionPublisher.getClients() builds clients pointing to attacker node.
#    All subsequent publish() calls route through attacker infrastructure.
#    Attacker logs all signed transactions and drains 1 tinyhbar per validation cycle.
```

    private List<NodeProperties> toNodeProperties(NetworkNode networkNode) {
        var nodeValidation = monitorProperties.getNodeValidation();
        var tlsPredicate = nodeValidation.getTls().getPredicate();
        return networkNode.getServiceEndpoints().stream()
                .filter(s -> tlsPredicate.test(s.getPort()))
                .limit(nodeValidation.getMaxEndpointsPerNode())
                .map(serviceEndpoint -> {
                    var host = StringUtils.isNotBlank(serviceEndpoint.getDomainName())
                            ? serviceEndpoint.getDomainName()
                            : serviceEndpoint.getIpAddressV4();
                    var nodeProperties = new NodeProperties();
                    nodeProperties.setAccountId(networkNode.getNodeAccountId());
                    nodeProperties.setCertHash(Strings.CS.remove(networkNode.getNodeCertHash(), "0x"));
                    nodeProperties.setHost(host);
                    nodeProperties.setNodeId(networkNode.getNodeId());
                    nodeProperties.setPort(serviceEndpoint.getPort());
                    return nodeProperties;
                })
                .toList();
    }
