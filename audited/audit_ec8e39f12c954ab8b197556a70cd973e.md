### Title
Unauthenticated Multi-Query Chain in `getContractResultsByTransactionIdOrHash` Enables DB Connection Pool Exhaustion

### Summary
The `GET /contracts/results/:transactionIdOrHash` endpoint in the Mirror Node REST API requires no authentication and triggers a chain of 7+ sequential and parallel database queries per request. With no rate limiting or concurrency control in the REST API layer, a single attacker can exhaust the DB connection pool by sending a modest number of concurrent requests with valid Ethereum hashes, degrading or denying service to the mirror node.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 21 registers the route with no auth guard: [1](#0-0) 

`rest/controllers/contractController.js` `getContractResultsByTransactionIdOrHash()` (lines 1120ŌĆō1203) executes the following sequential DB calls when a valid Ethereum hash is supplied:

- **Query 1 (sequential):** `ContractService.getContractTransactionDetailsByHash()` ŌåÆ `ethereumTransactionsByHashQuery` against `ContractTransactionHash` table [2](#0-1) 

- **Query 2 (sequential):** `ContractService.getInvolvedContractsByTimestampAndContractId()` ŌåÆ `involvedContractsQuery` against `ContractTransaction` table [3](#0-2) 

- **Queries 3ŌĆō7 (parallel via `Promise.all`):** `getDetailedContractResults()` fires 5 simultaneous queries: [4](#0-3) 

  1. `ContractService.getContractResultsByTimestamps()`
  2. `TransactionService.getEthTransactionByTimestampAndPayerId()`
  3. `RecordFileService.getRecordFileBlockDetailsFromTimestamp()`
  4. `ContractService.getContractLogsByTimestamps()`
  5. `ContractService.getContractStateChangesByTimestamps()`

- **Query 8 (conditional sequential):** `FileDataService.getLatestFileDataContents()` if `callDataId` is valid [5](#0-4) 

**Root cause ŌĆö no rate limiting in the REST API layer:**

`rest/server.js` registers no IP-based rate limiting, no per-endpoint concurrency control, and no request throttle middleware for this endpoint: [6](#0-5) 

The `authHandler` middleware only enforces response-size limits for authenticated users; it does not throttle unauthenticated requests: [7](#0-6) 

The `ThrottleConfiguration`/`ThrottleManagerImpl` rate limiting exists only in the `web3` (Java) module, not in the REST (Node.js) module.

**DB connection pool:**

`rest/dbpool.js` creates a single pool bounded by `config.db.pool.maxConnections`: [8](#0-7) 

The Helm chart sets `mirror_rest` to `max_user_connections: 250` via pgbouncer: [9](#0-8) 

With 5 parallel queries per request, only **50 concurrent requests** are needed to hold all 250 connections simultaneously for the duration of the `statement_timeout` window.

### Impact Explanation

An attacker exhausting the DB connection pool causes all subsequent requests to queue or fail with connection timeout errors, effectively taking the mirror node REST API offline. The `statement_timeout` is a per-query bound, not a concurrency bound ŌĆö it limits how long each query runs but does not prevent 50 concurrent requests from each holding 5 connections simultaneously. The mirror node is a critical read path for wallets, explorers, and dApps querying Hedera state; its unavailability constitutes a denial-of-service against dependent infrastructure.

### Likelihood Explanation

No special privileges, accounts, or credentials are required. The attacker only needs a single valid Ethereum transaction hash (publicly observable on any block explorer or by submitting a real transaction). The attack is trivially scriptable with any HTTP concurrency tool (`ab`, `wrk`, `hey`). It is repeatable and stateless ŌĆö each request is independent. The 50-connection threshold is well within reach of a single machine.

### Recommendation

1. **Add per-IP rate limiting** to the REST API using middleware such as `express-rate-limit`, applied globally or specifically to expensive endpoints like `/contracts/results/:transactionIdOrHash`.
2. **Add a concurrency semaphore** for `getDetailedContractResults()` to cap the number of simultaneous in-flight multi-query chains.
3. **Reduce the `statement_timeout`** to a low value (e.g., 5ŌĆō10 seconds) to bound how long each connection is held.
4. **Consider response caching** (already partially supported via Redis) for hash-based lookups, since the same hash always returns the same result.

### Proof of Concept

**Preconditions:** Obtain any valid Ethereum transaction hash from the public Hedera network (e.g., from HashScan).

**Trigger:**
```bash
# Send 60 concurrent requests ŌĆö enough to exhaust the 250-connection pool
# (5 parallel queries ├Ś 60 concurrent requests = 300 connections > 250 limit)
hey -n 600 -c 60 \
  "https://<mirror-node-host>/api/v1/contracts/results/0x4a563af33c4871b51a8b108aa2fe1dd5280a30dfb7236170ae5e5e7957eb6392"
```

**Result:** Subsequent requests to any endpoint on the mirror node REST API return connection timeout errors or HTTP 500 responses until in-flight queries complete, degrading service for all users.
