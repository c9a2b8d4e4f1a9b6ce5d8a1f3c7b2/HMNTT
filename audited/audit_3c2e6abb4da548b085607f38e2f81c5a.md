### Title
Missing FileId Whitelist Validation in `getNetworkNodes()` Allows Arbitrary DB Queries via Unauthenticated Requests

### Summary
The REST Java `NetworkServiceImpl.getNetworkNodes()` accepts any arbitrary `file.id` value from unauthenticated callers and passes it directly to the database layer without validating it against the two known valid address book file IDs (101 and 102). The gRPC counterpart explicitly rejects invalid file IDs before touching the database, but the REST Java service has no equivalent guard. An unprivileged attacker can flood the endpoint with distinct arbitrary `file.id` values, causing the database to execute a query (including subquery and JOINs) for every request, with no rate limiting in the visible code path.

### Finding Description
**Code path:**

`NetworkController.getNodes()` (line 152–171) validates only that the operator is `EQ`, not that the value is one of the two valid file IDs: [1](#0-0) 

It then calls `networkService.getNetworkNodes(request)`, which delegates to `getAddressBookFileId()`: [2](#0-1) 

This method returns the caller-supplied value verbatim with zero validation. It is then forwarded to the repository: [3](#0-2) 

**Contrast with the gRPC service**, which maintains a whitelist and throws before any DB access: [4](#0-3) 

**The repository test confirms the DB query executes for non-existent file IDs** (returning empty results, not short-circuiting before the query): [5](#0-4) 

**Root cause:** `getAddressBookFileId()` has a failed assumption — it assumes callers will only supply valid file IDs, but there is no enforcement of this assumption anywhere in the REST Java stack.

### Impact Explanation
Every request with an arbitrary `file.id` value causes the database to execute a query involving the `address_book` table (subquery for latest timestamp), `address_book_entry`, `node_stake`, and `node` tables. For non-existent file IDs the query returns empty results, but the query is still dispatched and parsed by the DB engine. With no rate limiting visible in the code path and a `MAX_LIMIT` of 25 that only caps result size (not query cost), a sustained flood of requests with rotating arbitrary file IDs keeps the DB busy with repeated lookups. This degrades response latency for legitimate users. The severity matches the stated classification: griefing with no direct economic damage. [6](#0-5) 

### Likelihood Explanation
No authentication is required. The endpoint is publicly accessible (`GET /api/v1/network/nodes`). The only input constraint enforced is that the operator must be `EQ` — the value itself is unconstrained. Any attacker with HTTP access can automate requests with rotating numeric `file.id` values using a trivial script. The attack is repeatable indefinitely.

### Recommendation
Add a whitelist check in `getAddressBookFileId()` (or at the top of `getNetworkNodes()`) mirroring the gRPC service:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    long id = request.getFileId() != null
            ? request.getFileId().value()
            : systemEntity.addressBookFile102().getId();
    var valid = Set.of(
            systemEntity.addressBookFile101().getId(),
            systemEntity.addressBookFile102().getId());
    if (!valid.contains(id)) {
        throw new IllegalArgumentException("Not a valid address book file");
    }
    return id;
}
```

This rejects invalid file IDs before any database interaction, consistent with the gRPC service behavior. [7](#0-6) 

### Proof of Concept
```bash
# Flood the endpoint with arbitrary non-existent file IDs
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/network/nodes?file.id=0.0.$((RANDOM + 200))" &
done
wait
```

Each request reaches the database with a distinct non-existent `file_id`, causing repeated query execution. Monitor DB CPU/query throughput to observe degradation. No credentials or special privileges required.
