### Title
Unbounded Mutual Recursion in `mapKey()`/`mapKeyList()` Enables DoS via Deeply Nested Protobuf Key Structures

### Summary
The `mapKey()` and `mapKeyList()` methods in `CommonMapper.java` mutually recurse without any depth limit when processing `THRESHOLDKEY` and `KEYLIST` protobuf key structures. An unprivileged attacker can create a Hedera account with a deeply nested key structure, causing the mirror node's GraphQL `account()` query to trigger unbounded recursion, resulting in `StackOverflowError` per request. Flooding with such requests degrades service availability, though the specific "network partition" framing is overstated — the actual impact is per-request DoS, not heap exhaustion or broken DB connections.

### Finding Description
**Exact code path:**

In `graphql/src/main/java/org/hiero/mirror/graphql/mapper/CommonMapper.java`:

- `mapKey(Key key)` lines 105–108: for `THRESHOLDKEY`, calls `mapKeyList(key.getThresholdKey().getKeys())`
- `mapKey(Key key)` line 103: for `KEYLIST`, calls `mapKeyList(key.getKeyList())`
- `mapKeyList(KeyList keyList)` line 122: iterates and calls `mapKey(key)` for each child key

This is unbounded mutual recursion. There is no depth counter, no recursion guard, and no maximum nesting limit applied to the protobuf key data itself.

**Root cause / failed assumption:** The code assumes that key structures stored in the database are shallow. The Hedera protobuf `Key` type allows arbitrary nesting of `THRESHOLDKEY` → `KEYLIST` → `THRESHOLDKEY` → ... with no enforced depth limit at the network protocol level beyond the transaction byte size (~6KB). Within that budget, an attacker can construct ~200–400 levels of nesting (each wrapper adds ~10–15 bytes of protobuf overhead), sufficient to exhaust the JVM thread stack.

**Why existing checks fail:**

- `MaxQueryDepthInstrumentation(10)` in `GraphQlConfiguration.java` line 44 limits the depth of the *GraphQL query document* (e.g., nested field selections), not the depth of the *data* being processed during field resolution.
- `MaxQueryComplexityInstrumentation(200)` similarly applies to query structure, not data processing.
- GraphQL parser limits (`maxCharacters(10000)`, `maxRuleDepth(100)`, `maxTokens(1000)`) apply to the incoming query string, not to the protobuf key bytes fetched from the database.
- Jackson `maxNestingDepth(10)` applies to JSON input parsing, not to protobuf key traversal.
- The Traefik `inFlightReq: amount: 5` per-IP limit reduces but does not eliminate the attack surface — an attacker using multiple IPs or rotating proxies bypasses it entirely.

### Impact Explanation
Each `account(input: ...) { key }` query against an account with a deeply nested key triggers a `StackOverflowError` in the request handler thread. Spring GraphQL catches this and returns an error response, but the thread is consumed and the JVM may experience elevated GC pressure from partially-constructed `ArrayList` and `Map` objects. Under sustained flooding (multiple IPs, 5 concurrent requests each), the GraphQL service degrades in availability. The "network partition" and "heap exhaustion" framing in the question is overstated — the actual impact is per-request thread exhaustion and service degradation, not broken database connections.

### Likelihood Explanation
- **Precondition**: Attacker must create a Hedera account with a deeply nested key. This is possible on mainnet (costs a small amount of HBAR) or for free on testnet. The mirror node mirrors all accounts.
- **No authentication required**: The GraphQL `account()` query is publicly accessible with no authentication.
- **Repeatability**: Once the account exists in the mirror node database, the attacker can query it indefinitely. The account's `num` is discoverable by enumeration.
- **Amplification**: A single malicious account can be used to flood the service from multiple IPs, bypassing the per-IP in-flight limit.

### Recommendation
Add a depth limit to the `mapKey()` recursion:

```java
default Object mapKey(Key key) {
    return mapKey(key, 0);
}

private static final int MAX_KEY_DEPTH = 10;

default Object mapKey(Key key, int depth) {
    if (depth > MAX_KEY_DEPTH) {
        logger.warn("Key nesting depth exceeded limit of {}", MAX_KEY_DEPTH);
        return null;
    }
    var keyCase = key.getKeyCase();
    return switch (keyCase) {
        case KEYLIST -> Map.of(KEYS, mapKeyList(key.getKeyList(), depth + 1));
        case THRESHOLDKEY -> Map.of(
            THRESHOLD, key.getThresholdKey().getThreshold(),
            KEYS, mapKeyList(key.getThresholdKey().getKeys(), depth + 1));
        // ... other cases unchanged
    };
}
```

Also add a maximum key list width limit in `mapKeyList()` to bound the object graph size.

### Proof of Concept
1. Using the Hedera Java SDK, create an account on testnet with a key structured as 200 levels of nested `ThresholdKey(threshold=1, keys=[ThresholdKey(...)])`.
2. Wait for the mirror node to index the account (typically seconds to minutes).
3. Identify the account's entity number `N`.
4. Send repeated POST requests to `/graphql/alpha`:
   ```graphql
   { account(input: { entityId: { num: N } }) { key } }
   ```
5. Observe `StackOverflowError` in mirror node logs and error responses from the GraphQL endpoint.
6. From multiple source IPs (to bypass the per-IP in-flight limit of 5), flood with this query to sustain service degradation.
