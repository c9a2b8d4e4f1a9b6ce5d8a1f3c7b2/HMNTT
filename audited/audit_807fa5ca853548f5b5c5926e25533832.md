### Title
Plaintext gRPC Transport Enables MITM Injection of Fabricated ConsensusTopicResponse Messages

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the `NettyServerBuilder` with only an executor and connection limit — no TLS context is ever applied. The default Helm chart deployment (`ingress.tls.enabled: false`) compounds this by also disabling TLS at the ingress layer, leaving the entire gRPC stream unencrypted end-to-end. A network-adjacent attacker can intercept and inject fabricated `ConsensusTopicResponse` messages containing arbitrary `runningHash` and `sequenceNumber` values that clients cannot distinguish from legitimate Hashgraph history.

### Finding Description

**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35:

```java
@Bean
ServerBuilderCustomizer<NettyServerBuilder> grpcServerConfigurer(
        GrpcProperties grpcProperties, Executor applicationTaskExecutor) {
    final var nettyProperties = grpcProperties.getNetty();
    return serverBuilder -> {
        serverBuilder.executor(applicationTaskExecutor);
        serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    };
}
```

No call to `serverBuilder.useTransportSecurity(sslContext)` or any equivalent is present. `NettyProperties` (`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`) exposes only `maxConcurrentCallsPerConnection` — there is no TLS property at all.

**Default Helm chart** (`charts/hedera-mirror-grpc/values.yaml`, lines 142–144):
```yaml
tls:
  enabled: false
  secretName: ""
```
TLS at the Traefik ingress is **disabled by default**. The official documentation and GCP marketplace README both demonstrate access with `grpcurl -plaintext` on port 5600/80, confirming the default plaintext posture.

**No server-side interceptors** exist in `grpc/src/main/java/` for authentication or message integrity. The `ConsensusTopicResponse` proto (`protobuf/src/main/proto/com/hedera/mirror/api/proto/consensus_service.proto`, lines 28–41) carries `runningHash` (SHA-384) and `sequenceNumber` with no transport-level or message-level authentication.

**Root cause / failed assumption:** The design assumes TLS will be provided by the infrastructure layer (Traefik), but the default Helm values ship with `ingress.tls.enabled: false`, making the assumption false in any default or misconfigured deployment. The application layer provides zero fallback enforcement.

**Exploit flow:**
1. Attacker positions themselves on the network path (ARP poisoning on LAN, BGP hijack, compromised upstream router, or cloud VPC tap).
2. Client connects to the gRPC server on port 5600 (or 80 via ingress) — plaintext HTTP/2.
3. Attacker intercepts the `subscribeTopic` stream for `ConsensusService`.
4. Attacker replaces or injects `ConsensusTopicResponse` frames with crafted protobuf payloads containing arbitrary `runningHash` bytes and `sequenceNumber` values.
5. Client receives the fabricated messages; there is no transport MAC, no message signature, and no out-of-band verification mechanism in the gRPC API to detect tampering.

**Why existing checks fail:**
- `ConsensusController.subscribeTopic()` streams data directly from the database to the client with no signing step.
- `SubscriptionResponse.validateResponse()` (test/e2e) only checks that `sequenceNumber` increments by 1 and timestamps are ordered — both trivially satisfied by a crafted injection.
- The `runningHash` is a SHA-384 chain hash, but clients have no independent reference to verify it against; the mirror node is the sole authority.

### Impact Explanation

Clients subscribing to HCS topics receive what they believe to be authoritative, immutable Hashgraph history. A successful injection allows an attacker to:
- Forge message content, `runningHash`, and `sequenceNumber` for any topic
- Suppress real messages (drop frames) or replay old ones
- Cause downstream applications (DeFi, audit trails, supply chain) to act on fabricated consensus history

Severity is **High**: the data being tampered with is the core integrity guarantee of the Hedera Consensus Service mirror API.

### Likelihood Explanation

- **Default deployment is vulnerable**: `ingress.tls.enabled: false` is the shipped default; operators who follow the quickstart without explicitly enabling TLS are exposed.
- **Network position requirement**: The attacker must be on the network path. This is achievable via ARP poisoning (same LAN/VPC), BGP hijacking (ISP/transit level), or a compromised cloud network component — all realistic for a motivated, targeted attacker.
- **No authentication barrier**: The gRPC service requires no client credentials, so any attacker who achieves network position can immediately begin injecting without needing to bypass any auth layer.
- **Repeatability**: Once positioned, the attacker can inject on every subscription stream indefinitely.

### Recommendation

1. **Enforce TLS at the application layer** in `grpcServerConfigurer()`:
   ```java
   SslContext sslContext = GrpcSslContexts.forServer(certChain, privateKey).build();
   serverBuilder.useTransportSecurity(sslContext);
   ```
   Add TLS certificate properties to `NettyProperties` and `GrpcProperties`.

2. **Change the Helm default** to `ingress.tls.enabled: true` and require a `secretName`, failing deployment if absent.

3. **Add message-level integrity**: Sign `ConsensusTopicResponse` payloads with a server private key so clients can verify authenticity independent of transport security.

4. **Document TLS as mandatory**, not optional, for any internet-facing deployment.

### Proof of Concept

**Preconditions:** Default Helm deployment (TLS disabled); attacker has network-adjacent position (e.g., same Kubernetes node or VPC with packet capture capability).

```bash
# Step 1: Confirm plaintext service
grpcurl -plaintext <GRPC_IP>:5600 list
# Returns: com.hedera.mirror.api.proto.ConsensusService (no TLS required)

# Step 2: Intercept with mitmproxy (HTTP/2 / gRPC support)
mitmproxy --mode transparent --ssl-insecure -p 8080

# Step 3: Write a mitmproxy addon to rewrite ConsensusTopicResponse frames
# (inject forged runningHash=<48 arbitrary bytes>, sequenceNumber=<any value>)
# using protobuf encoding matching consensus_service.proto field layout

# Step 4: ARP-poison the client to route traffic through attacker host
arpspoof -i eth0 -t <CLIENT_IP> <GRPC_SERVER_IP>

# Step 5: Client subscribes normally
grpcurl -plaintext -d '{"topicID":{"topicNum":41110},"limit":5}' \
  <GRPC_IP>:5600 com.hedera.mirror.api.proto.ConsensusService/subscribeTopic

# Result: Client receives attacker-injected ConsensusTopicResponse messages
# with fabricated runningHash and sequenceNumber, indistinguishable from
# legitimate Hashgraph history at the transport and application layer.
```
