### Title
Missing `lower(timestamp_range) <= blockTimestamp` Guard in `entity` Branch of `findActiveByEvmAddressOrAliasAndTimestamp()`

### Summary
The `entity` branch of the UNION in `findActiveByEvmAddressOrAliasAndTimestamp()` omits the `lower(timestamp_range) <= ?2` predicate that is present in the equivalent `findActiveByIdAndTimestamp()` query. Because the `entity` table holds the current (live) state with an open-ended `timestamp_range = [T_last_update, ‚ąě)`, and because `‚ąě` always sorts above any closed historical range, the final `ORDER BY timestamp_range DESC LIMIT 1` will always prefer the current row over the correctly-bounded `entity_history` row. Any unprivileged caller who supplies a `blockTimestamp` that falls between the entity's creation and its most recent update will receive the post-update (future) state instead of the correct historical state.

### Finding Description

**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, `findActiveByEvmAddressOrAliasAndTimestamp()`, lines 95‚Äď121.

**Root cause ‚ÄĒ the missing predicate:**

```sql
-- entity branch (lines 103-108): NO upper-bound guard
select *
from entity e
where e.deleted is not true
  and e.id = (select id from entity_cte)

-- compare with findActiveByIdAndTimestamp (line 140): CORRECT guard present
where id = ?1 and lower(timestamp_range) <= ?2
```

**Exploit flow:**

1. Entity `E` is created at `T1` (`created_timestamp = T1`; `timestamp_range = [T1, T3)`).
2. At `T3 > T2`, entity `E` is updated (e.g., balance, key, memo). The `entity` table now holds `timestamp_range = [T3, ‚ąě)`. The old state is moved to `entity_history` with `timestamp_range = [T1, T3)`.
3. Attacker calls `eth_call` (or any historical RPC) with `blockTimestamp = T2` (where `T1 ‚Č§ T2 < T3`).
4. `entity_cte` resolves the ID because `created_timestamp (T1) ‚Č§ T2` ‚ÄĒ correct so far.
5. `entity` branch returns the current row (`timestamp_range = [T3, ‚ąě)`) ‚ÄĒ **no check that T3 ‚Č§ T2**.
6. `entity_history` branch returns the correct historical row (`timestamp_range = [T1, T3)`).
7. `ORDER BY timestamp_range DESC LIMIT 1` picks `[T3, ‚ąě)` over `[T1, T3)` because `‚ąě > T3`.
8. The caller receives the **post-T3 state** for a query that should reflect the **T2 state**.

**Why existing checks are insufficient:**

- `entity_cte`'s `created_timestamp <= ?2` only proves the entity *existed* at `T2`; it says nothing about whether the current row's `timestamp_range` lower bound is ‚Č§ `T2`.
- `deleted is not true` is orthogonal to the timestamp issue.
- The test suite (lines 384‚Äď403) only covers the case where `created_timestamp > blockTimestamp` (entity_cte returns nothing); it does **not** cover the case where `created_timestamp ‚Č§ blockTimestamp < lower(entity.timestamp_range)`.

### Impact Explanation
Any historical `eth_call`, `eth_getBalance`, contract-code lookup, or alias resolution that goes through `CommonEntityAccessor.get(Bytes alias, Optional<Long> timestamp)` (line 48 of `CommonEntityAccessor.java`) will silently return the **current** entity state instead of the state at the requested block. This allows an attacker to observe post-block data (updated balances, changed keys, new contract bytecode) while claiming to query a past block, directly misrepresenting Hashgraph history. Protocols or DApps that rely on historical state proofs (e.g., dispute resolution, snapshot-based governance, audit trails) are affected.

### Likelihood Explanation
No privilege is required. Any user who can issue a JSON-RPC call with a block number parameter (standard Ethereum API) can trigger this path. The attacker only needs to know (or guess) that an entity was updated after the target block ‚ÄĒ information that is publicly visible on the mirror node's REST API. The bug is deterministic and fully repeatable.

### Recommendation
Add `lower(e.timestamp_range) <= ?2` to the `entity` branch, mirroring the pattern already used in `findActiveByIdAndTimestamp()`:

```sql
(
    select *
    from entity e
    where e.deleted is not true
      and lower(e.timestamp_range) <= ?2          -- ADD THIS
      and e.id = (select id from entity_cte)
)
```

Apply the identical fix to `findActiveByEvmAddressAndTimestamp()` (lines 59‚Äď85), which has the same structural defect.

### Proof of Concept

**Preconditions:**
- Entity `E` with `evm_address = 0xABCD` exists.
- `E` was created at `T1 = 1000` and last updated at `T3 = 3000` (e.g., balance changed from 100 to 999).
- `entity` table: `id=42, timestamp_range=[3000,‚ąě), balance=999`.
- `entity_history` table: `id=42, timestamp_range=[1000,3000), balance=100`.

**Steps:**
1. Issue `eth_call` (or `eth_getBalance`) with `block = T2 = 2000` and address `0xABCD`.
2. Mirror node calls `findActiveByEvmAddressOrAliasAndTimestamp(0xABCD, 2000)`.
3. `entity_cte` returns `id=42` (because `created_timestamp 1000 ‚Č§ 2000`).
4. `entity` branch returns row with `timestamp_range=[3000,‚ąě), balance=999`.
5. `entity_history` branch returns row with `timestamp_range=[1000,3000), balance=100`.
6. `ORDER BY timestamp_range DESC LIMIT 1` selects `[3000,‚ąě)`.
7. **Returned balance = 999**, but the correct historical balance at block `T2=2000` is **100**. [1](#0-0) [2](#0-1) [3](#0-2)
