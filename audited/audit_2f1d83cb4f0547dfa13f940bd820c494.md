### Title
Unbounded Request Body Allows Memory Exhaustion DoS via `/mempool/transaction` Endpoint

### Summary
The `MempoolTransaction()` handler in `rosetta/app/services/mempool_service.go` immediately discards its `*types.MempoolTransactionRequest` argument and returns `ErrNotImplemented`, but the rosetta-sdk-go controller fully deserializes the JSON request body into memory before invoking the service method. No `http.MaxBytesReader` or byte-level body size cap is applied anywhere in the rosetta HTTP stack, meaning an unauthenticated attacker can send arbitrarily large request bodies that are fully allocated in heap memory before being silently discarded, enabling a memory-exhaustion denial-of-service.

### Finding Description
**Code path:**

- `rosetta/app/services/mempool_service.go` lines 30–35: `MempoolTransaction()` accepts `_ *types.MempoolTransactionRequest` (blank identifier), performs zero work, and returns `errors.ErrNotImplemented`. The request struct — including the `transaction_identifier.hash` string field — has already been fully deserialized and heap-allocated by the rosetta-sdk-go `MempoolAPIController` before this function is ever called.

- `rosetta/main.go` lines 220–227: The `http.Server` is configured with only time-based limits (`ReadTimeout`, `ReadHeaderTimeout`, `IdleTimeout`, `WriteTimeout`). No `http.MaxBytesReader` wrapper, no `http.MaxBytesHandler`, and no middleware-level byte cap is applied to the request body.

- `rosetta/app/config/types.go` lines 64–69: The `Http` config struct exposes only duration fields; there is no `MaxBodySize` or equivalent configuration knob.

- A grep across all `rosetta/**/*.go` files for `MaxBytesReader`, `MaxBytesHandler`, `ReadLimit`, `maxBodySize` returns zero matches.

**Root cause:** The codebase assumes that the time-based `ReadTimeout` (default 5 s, i.e. 5 000 000 000 ns per `docs/configuration.md` line 664) is sufficient to bound resource consumption. This assumption is false: `ReadTimeout` is a wall-clock deadline, not a byte cap. On a 1 Gbps link, 5 seconds allows ~625 MB per connection; on a 100 Mbps link, ~62 MB. Go's `encoding/json` decoder allocates the full decoded struct in heap memory before returning, so the entire body is live in memory simultaneously.

**Why existing checks fail:** `ReadTimeout` terminates the connection after the deadline but does not prevent a fast sender from delivering tens or hundreds of megabytes within that window. There is no rate limiting, no authentication, and no connection-level concurrency cap visible in the rosetta stack.

### Impact Explanation
An attacker sending concurrent POST requests to `/mempool/transaction` with megabyte-scale `transaction_identifier.hash` strings forces the Go runtime to allocate and then GC large heap objects for every request. With enough concurrent connections the heap grows faster than GC can reclaim it, causing:
- Increased GC pause times degrading all endpoint latency
- RSS growth potentially triggering OOM-kill of the rosetta process
- Effective denial of service for legitimate transaction submission and block queries sharing the same process

Severity: **Medium–High** (unauthenticated, remotely triggerable, process-level impact).

### Likelihood Explanation
The endpoint requires no credentials and is reachable by any network peer. The attack is trivially scriptable: a single `curl` loop or a tool like `wrk`/`hey` with a crafted payload suffices. Because the handler returns immediately after deserialization, the server can accept a high rate of such requests. The attack is repeatable and stateless from the attacker's perspective.

### Recommendation
1. Wrap the request body with `http.MaxBytesReader` in a middleware applied to all routes, e.g.:
   ```go
   func maxBodyMiddleware(next http.Handler, limit int64) http.Handler {
       return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
           r.Body = http.MaxBytesReader(w, r.Body, limit)
           next.ServeHTTP(w, r)
       })
   }
   ```
   Insert this before `MetricsMiddleware` in `rosetta/main.go`. A limit of 1–4 MB is appropriate for Rosetta payloads.
2. Add a `MaxBodySize` field to `rosetta/app/config/types.go`'s `Http` struct so operators can tune the limit.
3. Consider returning HTTP 501 at the router level for `/mempool` and `/mempool/transaction` before deserialization occurs, since the endpoint is permanently unimplemented.

### Proof of Concept
```bash
# Generate a ~5 MB payload with an oversized transaction_identifier.hash
python3 -c "
import json, sys
payload = {
  'network_identifier': {'blockchain': 'Hedera', 'network': 'testnet'},
  'transaction_identifier': {'hash': 'A' * 5_000_000}
}
sys.stdout.write(json.dumps(payload))
" > /tmp/big_payload.json

# Flood the endpoint with 200 concurrent connections
hey -n 2000 -c 200 -m POST \
    -H 'Content-Type: application/json' \
    -D /tmp/big_payload.json \
    http://<rosetta-host>:5700/mempool/transaction

# Observe RSS growth on the rosetta process:
watch -n1 'ps -o pid,rss,vsz -p $(pgrep rosetta)'
```
Each request forces ~5 MB of heap allocation before the handler discards it. With 200 concurrent connections the live heap footprint reaches ~1 GB before GC intervenes, causing measurable latency spikes on all other endpoints. [1](#0-0) [2](#0-1) [3](#0-2)
