### Title
Unauthenticated DoS via Unbounded `/network/list` Flooding Due to Disabled Rate-Limiting Middleware by Default

### Summary
`NetworkList()` performs no rate limiting and the Traefik middleware chain that would enforce per-IP connection limits and rate limits is disabled by default (`global.middleware: false`). An unprivileged external attacker can flood `/network/list` with concurrent connections, exhausting Go HTTP server goroutines and OS-level file descriptors, rendering all Rosetta API endpoints — including `/network/status` (peer discovery) — unavailable.

### Finding Description
**Exact code path:**

`rosetta/app/services/network_service.go` lines 27–32: `NetworkList()` returns `n.network` directly with zero I/O, no DB call, and no rate-limiting guard. [1](#0-0) 

`rosetta/main.go` lines 217–219: The application-level middleware stack is `MetricsMiddleware → TracingMiddleware → CorsMiddleware → router`. None of these implement connection throttling or rate limiting. [2](#0-1) 

`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3: The Traefik `Middleware` CRD (which contains `inFlightReq: amount: 5` and `rateLimit: average: 10`) is only rendered when `global.middleware` is truthy. [3](#0-2) 

`charts/hedera-mirror-rosetta/values.yaml` line 95: `global.middleware` defaults to `false`, so the Traefik Middleware CRD is **never created** in a default deployment. [4](#0-3) 

**Root cause / failed assumption:** The design assumes the Traefik ingress middleware chain is active and enforcing per-IP in-flight limits and rate limits. That assumption is false in the default Helm deployment because `global.middleware: false` suppresses CRD creation. The ingress annotation that references the middleware CRD (`traefik.ingress.kubernetes.io/router.middlewares`) is therefore referencing a non-existent resource, making it a no-op. [5](#0-4) 

**Exploit flow:** Attacker opens thousands of concurrent TCP connections to the Rosetta port and sends valid POST `/network/list` requests in a tight loop. Go's `net/http` server spawns one goroutine per accepted connection. Because `NetworkList()` completes in microseconds, the attacker must maintain a high concurrency level, but with no per-IP or global connection cap at the application layer, the server's goroutine count and file-descriptor table grow unboundedly until the OS kills the process or the kernel drops new connections. All endpoints share the same listener and goroutine pool, so `/network/status`, `/block`, and `/construction` become unreachable simultaneously.

### Impact Explanation
In a default Helm deployment the Traefik rate-limiting CRD does not exist, so there is no network-layer throttle. A successful flood exhausts the Go runtime's goroutine scheduler and the OS file-descriptor limit, causing a complete denial of service across all Rosetta API endpoints. Clients relying on `/network/status` for peer discovery cannot reach the node, effectively isolating it from the Rosetta network view. The HTTP server timeouts (`ReadTimeout`, `WriteTimeout`, `IdleTimeout`) mitigate slow-loris variants but do not bound the total number of concurrent accepted connections. [6](#0-5) 

### Likelihood Explanation
No authentication, no API key, and no application-level rate limit are required. Any unprivileged attacker with a standard HTTP client and sufficient bandwidth can trigger this. The attack is trivially scriptable (`ab`, `wrk`, `hey`, or a simple Go/Python loop). The HPA scales pods up to 10 replicas under CPU pressure, but scaling lag (tens of seconds) means the window of unavailability is real and repeatable. The attack is also repeatable indefinitely since no state is tracked per client. [7](#0-6) 

### Recommendation
1. **Enable the middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the Traefik `inFlightReq` (5 per IP) and `rateLimit` (10 req/s per host) CRDs are always created. [8](#0-7) 
2. **Add application-level connection limiting**: In `rosetta/main.go`, wrap the HTTP server listener with `netutil.LimitListener` (from `golang.org/x/net/netutil`) to cap total concurrent connections independently of the ingress layer.
3. **Add per-endpoint in-flight limiting in Go**: Insert a `golang.org/x/time/rate` token-bucket middleware in the application middleware chain so rate limiting is enforced even when accessed directly (bypassing ingress), e.g., via `kubectl port-forward` or a `NodePort`.

### Proof of Concept
```bash
# Prerequisites: default Helm deployment (global.middleware=false), direct access to Rosetta port
# Step 1: confirm no rate limiting
curl -s -X POST http://<rosetta-host>:<port>/network/list \
  -H 'Content-Type: application/json' -d '{}' | jq .

# Step 2: flood with 5000 concurrent connections
hey -n 500000 -c 5000 -m POST \
  -H 'Content-Type: application/json' \
  -d '{}' \
  http://<rosetta-host>:<port>/network/list

# Step 3: in a separate terminal, observe /network/status becomes unreachable
curl -s -X POST http://<rosetta-host>:<port>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' 
# Expected: connection timeout or 503 while flood is active
```
