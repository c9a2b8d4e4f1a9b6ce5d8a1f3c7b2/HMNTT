### Title
Gas Token Leak via Uncaught `IllegalArgumentException` from `hexToBytes` in `ContractController.call()`

### Summary
In `ContractController.call()`, the `catch` block only restores gas tokens when `InvalidParametersException` is thrown. However, `constructServiceParameters()` calls `hexToBytes(data)` outside its own try-catch, and `hexToBytes` throws `IllegalArgumentException` (not `InvalidParametersException`) for odd-length hex input. An unprivileged attacker can craft requests with odd-length hex `data` that pass `@Valid` bean validation, consume gas tokens via `throttleManager.throttle()`, then cause `hexToBytes` to throw — permanently draining the gas token bucket without any EVM execution.

### Finding Description

**Exact code path:**

`ContractController.java`, `call()`, lines 38–51: [1](#0-0) 

The catch block only handles `InvalidParametersException`:
```java
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
```

Any other unchecked exception escapes without calling `throttleManager.restore()`.

**Root cause — misplaced try-catch in `constructServiceParameters()`:**

`ContractController.java`, lines 66–72: [2](#0-1) 

The try-catch wraps only the dead-code assignment `data = request.getData() != null ? request.getData() : HEX_PREFIX;` — a getter call that can never throw. The actual dangerous call, `hexToBytes(data)`, is placed **outside** the try-catch, inside the builder at line 80:

```java
.callData(hexToBytes(data))   // line 80 — OUTSIDE the try-catch
```

**`hexToBytes` throws `IllegalArgumentException` for odd-length hex:**

`BytesDecoder.java`, lines 79–89: [3](#0-2) 

```java
try {
    return org.apache.commons.codec.binary.Hex.decodeHex(hex);
} catch (DecoderException e) {
    throw new IllegalArgumentException(e.getMessage());  // NOT InvalidParametersException
}
```

**`@Hex` validator allows odd-length hex strings:**

`HexValidator.java`, line 22: [4](#0-3) 

```java
pattern = Pattern.compile("^(0x)?[0-9a-fA-F]{%d,%d}$".formatted(minLength, hex.maxLength()));
```

The `data` field is annotated `@Hex` with no `minLength`/`maxLength` constraints: [5](#0-4) 

The regex `^(0x)?[0-9a-fA-F]{0,N}$` matches any count of hex chars — including odd counts like 3 (`0xabc`). So `0xabc` passes `@Valid` but causes `Hex.decodeHex("abc")` to throw `DecoderException` → `IllegalArgumentException`.

**Exploit flow:**
1. `throttleManager.throttle(request)` — succeeds, gas tokens consumed from `gasLimitBucket`
2. `validateContractMaxGasLimit(request)` — passes (gas ≤ max limit)
3. `constructServiceParameters(request)` — `hexToBytes("0xabc")` throws `IllegalArgumentException`
4. `call()` catch block — only catches `InvalidParametersException`, so `throttleManager.restore()` is **never called**
5. Gas tokens are permanently lost; no EVM execution occurred

### Impact Explanation
The `gasLimitBucket` (Bucket4j token bucket) is permanently depleted by the attacker's gas value on every such request. Since no actual processing occurs, this is pure waste. An attacker can repeatedly send minimal-gas requests (gas = 21,000, the minimum per `@Min(21_000)`) with odd-length hex data to continuously drain the gas-per-second bucket, causing legitimate requests to be rejected with `ThrottleException("Gas per second rate limit exceeded.")`. This is a denial-of-service against the gas throttle mechanism — legitimate users are throttled while the attacker consumes no real resources. [6](#0-5) 

### Likelihood Explanation
No authentication or special privileges are required — the `/api/v1/contracts/call` endpoint is public. The trigger condition (odd-length hex `data`) is trivially crafted. The attack is fully repeatable and scriptable. The only rate limiting is the `rateLimitBucket` (requests-per-second), but since each request drains gas tokens without restoring them, even a low request rate can exhaust the gas bucket over time. [7](#0-6) 

### Recommendation

**Option 1 (preferred):** Widen the catch in `call()` to restore gas for any exception that occurs before actual EVM processing:

```java
} catch (Exception e) {
    if (/* no processing has started */) {
        throttleManager.restore(request.getGas());
    }
    throw e;
}
```

Or more precisely, catch `InvalidParametersException | IllegalArgumentException` and restore in both cases.

**Option 2:** Move `hexToBytes(data)` inside the existing try-catch in `constructServiceParameters()` and wrap the `IllegalArgumentException` as `InvalidParametersException`:

```java
try {
    data = request.getData() != null ? request.getData() : HEX_PREFIX;
    // validate hex is decodable here, or move hexToBytes call inside
} catch (final Exception e) {
    throw new InvalidParametersException(
        "data field '%s' contains invalid hex".formatted(request.getData()));
}
```

**Option 3:** Add an even-length constraint to the `@Hex` validator so odd-length hex strings are rejected at the `@Valid` stage, before `throttle()` is called. [8](#0-7) 

### Proof of Concept

```bash
# Send a request with odd-length hex data (3 hex chars after 0x prefix)
# This passes @Valid but causes hexToBytes to throw IllegalArgumentException

curl -X POST http://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "data": "0xabc",
    "gas": 21000,
    "to": "0x0000000000000000000000000000000000000001"
  }'

# Expected: 500 error (IllegalArgumentException propagates as unhandled)
# Effect: 21000 gas tokens consumed from gasLimitBucket, never restored

# Repeat in a loop to exhaust the gas bucket:
for i in $(seq 1 1000); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"data":"0xabc","gas":21000,"to":"0x0000000000000000000000000000000000000001"}' &
done

# Legitimate requests now receive:
# {"message":"Gas per second rate limit exceeded."}
```
