### Title
Unauthenticated Access to Computationally Expensive EVM Opcode Tracing Endpoint

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint in `OpcodesController.java` uses `properties.isEnabled()` as its sole access gate with no authentication or authorization. When enabled by an operator, any unauthenticated internet user can trigger full EVM transaction re-execution and retrieve complete stack, memory, and storage traces of any historical transaction. The global rate limit of 1 request/second is insufficient to prevent information disclosure and can be monopolized by a single attacker.

### Finding Description
**Exact code path:**

In `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, lines 59–65:

```java
if (properties.isEnabled()) {
    validateAcceptEncodingHeader(acceptEncoding);
    throttleManager.throttleOpcodeRequest();
    final var request = new OpcodeRequest(transactionIdOrHash, stack, memory, storage);
    return opcodeService.processOpcodeCall(request);
}
```

The only access control is the boolean flag from `OpcodesProperties` (`hiero.mirror.web3.opcode.tracer.enabled`, default `false`). There is no Spring Security configuration in the web3 module (no `SecurityConfig.java` exists), no API key check, no IP allowlist, and no identity verification of any kind.

**Throttle analysis:**

`ThrottleManagerImpl.throttleOpcodeRequest()` (lines 52–55) consumes from a single global `opcodeRateLimitBucket` with a default of `opcodeRequestsPerSecond = 1`. This is a **global** bucket — not per-IP, not per-user. A single attacker sending one request per second continuously monopolizes the entire quota, denying access to all other users while simultaneously keeping the server busy re-executing EVM transactions.

**Root cause:** The design assumes that `enabled=false` (operator-level control) is sufficient access control. There is no mechanism to restrict access to specific trusted callers even when an operator intentionally enables the endpoint.

### Impact Explanation
1. **Information disclosure**: With `stack=true`, `memory=true`, `storage=true` query parameters, the response exposes the complete EVM execution trace of any historical transaction — including all storage slot values, memory contents, and stack frames. This can reveal sensitive contract internals (e.g., private variable values at execution time) for any transaction ever processed.
2. **Resource exhaustion / DoS**: The endpoint re-executes the transaction on the EVM, which is CPU-intensive. Even at 1 req/sec, a sustained attack keeps the EVM busy and can degrade overall node performance. The gzip requirement does not prevent this — it only reduces bandwidth.
3. **Rate limit monopolization**: Because the bucket is global and not per-source, one attacker at 1 req/sec starves all legitimate users.

### Likelihood Explanation
- **Precondition**: Operator must set `hiero.mirror.web3.opcode.tracer.enabled=true`. This is a non-default configuration, but it is explicitly documented and intended for production debug use.
- **Attacker capability**: Zero — requires only an HTTP client and knowledge of any valid transaction hash (publicly available on-chain).
- **Repeatability**: Fully repeatable and automatable. No credentials, tokens, or prior access needed.
- **Feasibility**: Trivially exploitable with a single `curl` command once the endpoint is enabled.

### Recommendation
1. **Add authentication/authorization**: Protect the endpoint with an API key, IP allowlist, or Spring Security configuration (e.g., `@PreAuthorize` or a `SecurityFilterChain` that restricts `/api/v1/contracts/results/*/opcodes` to specific roles or source IPs).
2. **Per-source rate limiting**: Replace or supplement the global `opcodeRateLimitBucket` with per-IP rate limiting (e.g., using the `X-Forwarded-For` header already processed by `LoggingFilter`) to prevent quota monopolization.
3. **Operator guidance**: Document that enabling this endpoint without network-level access controls (reverse proxy, firewall) exposes sensitive EVM internals to the public internet.

### Proof of Concept
**Precondition**: `hiero.mirror.web3.opcode.tracer.enabled=true` is set on the target node.

```bash
# Step 1: Obtain any valid transaction hash (publicly available from block explorers)
TX_HASH="0xabc123..."

# Step 2: Call the opcodes endpoint with no credentials whatsoever
curl -H "Accept-Encoding: gzip" \
     "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true" \
     --compressed

# Result: Full EVM execution trace returned, including all storage slot values,
# memory contents, and stack frames — with zero authentication.

# Step 3: Monopolize the rate limit (denying other users)
while true; do
  curl -s -H "Accept-Encoding: gzip" \
       "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes" \
       --compressed > /dev/null
  sleep 1
done
