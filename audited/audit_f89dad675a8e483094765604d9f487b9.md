### Title
Cache-Key Bypass via Whitespace Variation Forces Repeated Maximum-Cost GraphQL Parsing

### Summary
The preparsed-document cache uses the raw, unnormalized query string as its cache key. An unprivileged attacker can generate an unbounded number of semantically identical but syntactically distinct queries by varying whitespace counts (up to the 1000-token limit), each producing a unique cache key and a guaranteed cache miss. On every miss the ANTLR-based GraphQL lexer must tokenize up to 999 whitespace tokens in addition to the real query tokens, multiplying per-request parse CPU by a large constant factor with no rate-limiting defence in the application layer.

### Finding Description
**Code locations:**

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, line 26:
  `cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));`
  The cache key is `executionInput.getQuery()` — the literal, byte-for-byte query string. No normalization (whitespace collapsing, AST fingerprinting, etc.) is applied before the lookup.

- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, line 34:
  `b -> b.maxCharacters(10000).maxRuleDepth(100).maxTokens(1000).maxWhitespaceTokens(1000);`
  The parser accepts up to 1000 whitespace tokens per query before rejecting it. Queries with 1–999 whitespace tokens are fully parsed.

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15:
  `private String query = "expireAfterWrite=1h,maximumSize=1000,recordStats";`
  The cache holds at most 1000 entries. An attacker who generates >1000 unique whitespace variants continuously evicts cached entries, keeping the hit-rate near zero.

**Root cause / failed assumption:** The developers added the preparsed-document cache to amortize parse cost across repeated identical queries. The assumption is that real clients send the same query text repeatedly. The assumption fails because whitespace is semantically insignificant in GraphQL but is significant to a byte-equality cache key. The `maxWhitespaceTokens(1000)` guard was intended to cap the cost of a *single* parse, but it inadvertently defines the *maximum* cost an attacker can impose on *every* parse.

**Exploit flow:**
1. Attacker crafts a valid minimal account query, e.g.:
   `{account(input:{entityId:{shard:0,realm:0,num:2}}){balance}}`
2. Attacker inserts N spaces (N = 1 … 999) between each token pair, producing 999 syntactically distinct strings that are semantically identical.
3. Each string is a unique cache key → cache miss → full ANTLR lex+parse of ~(real tokens + N whitespace tokens).
4. Attacker cycles through these variants in a tight loop (or with concurrent connections). The cache fills with 1000 entries and then churns; every request is a miss.
5. No application-level rate limiting exists (grep across `graphql/src/main/java` finds no `RateLimit`, `throttle`, or per-IP counter).

### Impact Explanation
The ANTLR lexer processes every whitespace token individually. A normal account query has roughly 15–25 real tokens; adding 999 whitespace tokens increases the lexer workload by ~40–65×. Even if parsing is 5–10 % of total request CPU under normal load, forcing every request to parse at maximum whitespace cost while keeping the cache hit-rate at zero can push parse CPU well above the 30 % threshold under moderate concurrency. The attack requires no authentication, no special headers, and no knowledge beyond the public GraphQL schema.

### Likelihood Explanation
The attack is trivially scriptable: a single loop that increments a whitespace counter and POSTs to `/graphql/alpha` is sufficient. No credentials, tokens, or insider knowledge are required. The endpoint is publicly reachable. The attack is repeatable indefinitely and can be parallelized across multiple source IPs or connections. The only external constraint is network bandwidth, which is negligible given the small query size.

### Recommendation
1. **Normalize the cache key** before the cache lookup: strip or collapse all whitespace in the query string (or compute an AST-level fingerprint) so that all semantically equivalent queries share one cache entry. This is the primary fix.
2. **Add per-IP or per-connection rate limiting** at the HTTP layer (e.g., Spring's `HandlerInterceptor` or an API gateway) to bound the number of parse operations any single client can trigger per second.
3. Consider lowering `maxWhitespaceTokens` to a value closer to what legitimate queries actually need (typically < 50), reducing the per-miss cost even if normalization is not implemented immediately.

### Proof of Concept
```bash
# Bash loop: send 1000 unique whitespace-padded queries, each a cache miss
for i in $(seq 1 999); do
  SPACES=$(printf '%*s' "$i" '')
  QUERY="{${SPACES}account(input:{${SPACES}entityId:{shard:0,realm:0,num:2}}){balance}}"
  curl -s -o /dev/null -X POST http://<host>:8083/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"$QUERY\"}" &
done
wait
# Repeat the loop continuously to keep cache churn at 100% and parse CPU elevated.
```
Each iteration sends a query accepted by the parser (whitespace token count < 1000) but never seen before by the cache, guaranteeing a full parse on every request.
