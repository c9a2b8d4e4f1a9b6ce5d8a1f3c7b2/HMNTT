### Title
Unauthenticated Full Table Scan DoS via Unbounded Historical Hook Storage Query

### Summary
The `getHookStorage()` endpoint in `HooksController.java` accepts a `timestamp` parameter with no minimum range requirement and no authentication or rate limiting. When any timestamp filter is supplied (e.g., `timestamp=gte:0`), the code routes to `getHookStorageChange()`, which — with no `key=eq:` parameters — calls `findByKeyBetweenAndTimestampBetween` using the full default key range `[0x00...00, 0xff...ff]` and timestamp range `[0, Long.MAX_VALUE]`. This executes an expensive `DISTINCT ON (key)` full-partition scan on `hook_storage_change` per request, and concurrent flooding by any unauthenticated attacker exhausts database resources.

### Finding Description

**Exact code path:**

1. **`HooksController.getHookStorage()`** — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, lines 104–130 — is a public `@GetMapping` with no authentication annotation. [1](#0-0) 

2. **`hookStorageChangeRequest()`** — lines 158–198 — initializes `lowerBound = MIN_KEY_BYTES` (32×`0x00`) and `upperBound = MAX_KEY_BYTES` (32×`0xFF`) as defaults. When no `key=eq:` parameters are sent, `keyFilters` remains empty. When `key=gte:0x00...00` is sent, `Arrays.compareUnsigned(value, lowerBound) > 0` evaluates to `0 > 0 = false`, so `lowerBound` is never updated. Same for `lte:0xff...ff`. The full range is preserved. [2](#0-1) 

3. **`Bound.of()`** — `rest-java/src/main/java/org/hiero/mirror/restjava/service/Bound.java`, lines 169–182 — with `timestamp=gte:0`, creates a `Bound` with `lower=0`. `isEmpty()` returns `false`, so `HookStorageRequest.isHistorical()` returns `true`. [3](#0-2) 

4. **`HookServiceImpl.getHookStorageChange()`** — `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`, lines 84–118 — the early-exit guard `if (keysInRange.isEmpty() && requestHasKeys)` is **not triggered** because `requestHasKeys = false`. Execution falls through to `findByKeyBetweenAndTimestampBetween` with `keyLowerBound=0x00...00`, `keyUpperBound=0xff...ff`, `timestampLowerBound=0`, `timestampUpperBound=Long.MAX_VALUE`. [4](#0-3) 

5. **`findByKeyBetweenAndTimestampBetween()`** — `rest-java/src/main/java/org/hiero/mirror/restjava/repository/HookStorageChangeRepository.java`, lines 15–39 — executes:
   ```sql
   SELECT DISTINCT ON (key) ...
   FROM hook_storage_change
   WHERE owner_id = ? AND hook_id = ?
     AND key >= '\x00...00' AND key <= '\xff...ff'
     AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
   ```
   The `DISTINCT ON (key)` requires PostgreSQL to sort and deduplicate all rows for the given `(owner_id, hook_id)` before applying `LIMIT 100`. The primary key is `(owner_id, hook_id, key, consensus_timestamp)`, so the index scan covers all rows for that owner/hook pair. [5](#0-4) 

**No rate limiting exists in rest-java:** `RestJavaConfiguration` registers only `ShallowEtagHeaderFilter` and `MetricsFilter` — no throttle. The `ThrottleManagerImpl` and `ThrottleConfiguration` exist only in the `web3` module. [6](#0-5) 

**The `hook_storage_change` table** has primary key `(owner_id, hook_id, key, consensus_timestamp)` with no secondary index on `consensus_timestamp` alone, and is time-partitioned. A full-range query scans all partitions. [7](#0-6) 

### Impact Explanation

Each request with `timestamp=gte:0` and no key filters forces a full scan of all `hook_storage_change` rows for the targeted `(owner_id, hook_id)` pair, plus an in-memory sort for `DISTINCT ON`. A busy hook with millions of storage change records will cause each query to consume significant CPU and I/O on the database. Concurrent flooding exhausts the DB connection pool and CPU, causing query timeouts and failures across all mirror node REST API consumers. The `statementTimeout` default of 10 seconds (per `hiero.mirror.restJava.db.statementTimeout`) limits individual query duration but does not prevent connection pool exhaustion under concurrent load. This directly impacts the mirror node's ability to serve transaction confirmation queries, constituting a network-observable service disruption. [8](#0-7) 

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only a valid `ownerId`/`hookId` path (both publicly enumerable via the `/api/v1/accounts/{id}/hooks` endpoint). The request is a standard HTTP GET with two query parameters. Any attacker with network access to the mirror node REST Java API can execute this. The attack is trivially scriptable and repeatable with tools like `curl` or `ab`. The `routes.hooks` Helm value defaults to `false` (disabled in routing), but if the hooks feature is enabled in deployment, the endpoint is fully exposed. [9](#0-8) 

### Recommendation

1. **Require a bounded timestamp range**: Enforce a maximum timestamp window (e.g., no more than 24 hours) when `isHistorical()` is true. Reject requests where `timestampUpperBound - timestampLowerBound` exceeds a configured maximum.
2. **Require at least one non-trivial filter**: If no `key=eq:` parameters are provided and the timestamp range covers more than a configured threshold, return HTTP 400.
3. **Add rate limiting to rest-java**: Apply a per-IP or global request rate limiter (e.g., Bucket4j, as already used in `web3`) to `HooksController` endpoints.
4. **Add a secondary index**: Add an index on `(owner_id, hook_id, consensus_timestamp)` to allow the DB to efficiently filter by timestamp before scanning keys.
5. **Enforce a minimum timestamp lower bound**: Reject `timestamp=gte:0` or any lower bound below a reasonable floor (e.g., current time minus retention window).

### Proof of Concept

```bash
# Step 1: Enumerate a valid ownerId/hookId (publicly accessible)
curl "https://<mirror-node>/api/v1/accounts/0.0.1/hooks"
# Note a hookId from the response, e.g., hookId=1

# Step 2: Single full-scan request (triggers DISTINCT ON full table scan)
curl "https://<mirror-node>/api/v1/accounts/0.0.1/hooks/1/storage?timestamp=gte:0&limit=100"

# Step 3: Flood with concurrent requests to exhaust DB resources
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1/hooks/1/storage?timestamp=gte:0&limit=100" &
done
wait

# Expected result: DB connection pool exhaustion, query timeouts,
# and service degradation/unavailability for all mirror node API consumers.
```
