### Title
Unauthenticated Unbounded gRPC Stream Exhaustion via `subscribeTopic()` Enables Resource Depletion and Service Unavailability

### Summary
`ConsensusController.subscribeTopic()` accepts streaming subscriptions from any unauthenticated client with no per-IP connection limit, no global subscriber cap, and no rate limiting. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — limits streams per TCP connection, not the number of TCP connections. An attacker opening thousands of TCP connections, each carrying 5 indefinite streams (`limit=0`, no `endTime`), exhausts R2DBC database connections, file descriptors, and Reactor scheduler resources, rendering the mirror node gRPC service unavailable to legitimate consensus nodes.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) directly subscribes every incoming request into a live Reactor pipeline with no authentication check, no IP-based throttle, and no global subscriber ceiling:

```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
            .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
    ...
}
``` [1](#0-0) 

**The only server-side connection guard** is configured in `GrpcConfiguration.java`:

```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [2](#0-1) 

with a default of 5: [3](#0-2) 

This limits concurrent RPC calls **per TCP connection**, not the total number of TCP connections. No `maxConnectionAge`, no `maxConnections`, and no IP-based limit is configured.

**`subscriberCount` is a metric gauge, not a cap.** It is incremented on subscribe and decremented on completion, but is never checked against a maximum: [4](#0-3) [5](#0-4) 

**The sole `ServerInterceptor`** (`GrpcInterceptor`) only sets an endpoint context string for table-usage tracking and unconditionally forwards every call: [6](#0-5) 

**Indefinite streams are by design.** The proto specifies `limit=0` means "receive indefinitely" and omitting `consensusEndTime` means "no end": [7](#0-6) 

Each indefinite stream holds a live Reactor subscription, a Redis channel subscription (default listener type `REDIS`), and periodic R2DBC database polling for gap recovery: [8](#0-7) 

**No throttling exists in the gRPC module.** Rate limiting via bucket4j exists only in the `web3` module and is entirely absent from the `grpc` module.

### Impact Explanation

Each attacker-controlled stream consumes:
- One R2DBC connection slot (historical retrieval + gap recovery polling)
- One Redis pub/sub subscription
- One `boundedElastic` scheduler thread slot (safety check)
- One OS file descriptor (TCP socket)

With N TCP connections × 5 streams each, the attacker can saturate the R2DBC connection pool (causing all legitimate DB queries to queue/timeout), exhaust OS file descriptors (causing new connection refusals), and fill the `boundedElastic` thread pool. The result is a complete denial of service for legitimate consensus nodes and SDK clients that depend on the mirror node's HCS subscription service. This constitutes the "network partition" described in the question: legitimate nodes can no longer reach the mirror node service.

### Likelihood Explanation

Preconditions: none. The gRPC port (default 5600) is publicly exposed with no authentication. The attack requires only a standard gRPC client (e.g., `grpcurl`) and the ability to open many TCP connections — trivially achievable from a single machine or a small botnet. The attack is repeatable and persistent: streams remain open until the attacker closes them or the server runs out of resources. No special knowledge of the Hedera protocol is required.

### Recommendation

1. **Enforce a global subscriber cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add per-IP connection limiting**: Configure Netty with a `ChannelTrafficShapingHandler` or an equivalent `ServerInterceptor` that tracks and limits concurrent connections per remote address.
3. **Add a gRPC-level rate-limiting interceptor**: Implement a `ServerInterceptor` using the same bucket4j pattern already used in the `web3` module to throttle new stream creation per IP.
4. **Set `maxConnectionAge`**: Call `serverBuilder.maxConnectionAge(duration, unit)` in `GrpcConfiguration` to force periodic connection recycling, limiting the duration any single connection can hold resources.
5. **Require authentication**: Add mTLS or token-based authentication via a `ServerInterceptor` for production deployments.

### Proof of Concept

```bash
# Open 1000 TCP connections, each with 5 indefinite streams (5000 total streams)
# Requires grpcurl and a tool to parallelize (e.g., GNU parallel or a simple shell loop)

for i in $(seq 1 1000); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 41110}, "limit": 0}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# Each background process holds one indefinite gRPC stream.
# Monitor server-side:
#   - R2DBC pool exhaustion: hikaricp_connections_active approaches hikaricp_connections_max
#   - File descriptors: process_files_open_files approaches process_files_max_files
#   - Subscriber gauge: hiero_mirror_grpc_subscribers climbs without bound
# Legitimate clients will begin receiving UNAVAILABLE or connection-refused errors.
```
