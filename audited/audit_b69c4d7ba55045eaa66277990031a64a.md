### Title
Unbatched Per-Slot DB Queries in Historical `eth_call` Enable Disproportionate Database Resource Exhaustion

### Summary
`ContractStateServiceImpl.findStorageByBlockTimestamp()` issues one separate `contract_state_change` DB query per storage slot read during historical EVM execution, while the equivalent non-historical path (`findStorage()`) batches all slot reads for a contract into a single `IN`-clause query and caches results. Any unprivileged user can trigger this amplification by submitting `eth_call` requests with a historical block parameter against a contract that reads many storage slots, causing N DB queries per request instead of 1, with no per-slot caching to absorb repeated calls.

### Finding Description

**Exact code path:**

`ContractStorageReadableKVState.readFromDataSource()` dispatches to one of two paths based on whether a timestamp is present in `ContractCallContext`: [1](#0-0) 

When a historical block is requested (`timestamp` is non-empty), every single EVM `SLOAD` opcode calls `findStorageByBlockTimestamp()`. When no timestamp (latest block), it calls `findStorage()`.

**Non-historical path — batched + cached:**

`findStorage()` checks a per-contract Caffeine cache, and on a miss calls `findStorageBatch()`, which issues a single `SELECT ... WHERE slot IN (...)` query against `contract_state` for all previously-seen slots of that contract: [2](#0-1) [3](#0-2) 

**Historical path — unbatched, uncached:**

`findStorageByBlockTimestamp()` is a one-liner that directly delegates to the repository with zero caching or batching: [4](#0-3) 

The repository query is a single-slot lookup with `ORDER BY consensus_timestamp DESC LIMIT 1`: [5](#0-4) 

There is no `@Cacheable` annotation on this method, no batch equivalent, and no in-memory deduplication. Each `SLOAD` in the EVM for a historical call produces exactly one DB round-trip.

**Why the throttle is insufficient:**

The throttle consumes tokens from the gas bucket based on `request.getGas()` (the declared gas limit), scaled by dividing by 10,000: [6](#0-5) [7](#0-6) 

This throttle is applied once per HTTP request, not per DB query. It does not account for the N-fold amplification of DB queries that historical calls produce. A request with 15M gas limit consumes only 1,500 throttle tokens (out of 750,000/second capacity), yet can trigger thousands of individual DB queries.

### Impact Explanation

A contract that reads N storage slots costs 1 batch DB query when called at the latest block, but N sequential DB queries when called at any historical block. For a contract with 1,000 storage slots (easily achievable — a cold `SLOAD` costs 2,100 gas, so ~7,100 cold reads fit in a 15M gas limit), a single historical `eth_call` generates ~7,100 DB queries against `contract_state_change` vs. 1 batch query for the same call at the latest block. At the default rate limit of 500 requests/second, an attacker can sustain millions of DB queries per second, exhausting DB connection pools and CPU, degrading service for all users.

### Likelihood Explanation

No authentication or special privilege is required. The `/api/v1/contracts/call` endpoint is public. The attacker only needs to know (or deploy) a contract that reads many storage slots and supply any valid historical block number. The attack is trivially repeatable and scriptable. The asymmetry is structural and present in every deployment.

### Recommendation

1. **Add a per-request in-memory cache for historical slot reads**: Within a single `ContractCallContext` execution, cache `(contractId, slot, blockTimestamp) → value` so repeated reads of the same slot within one EVM execution do not re-query the DB.
2. **Implement a batch historical query**: Add a `findStorageBatchByBlockTimestamp(contractId, slots, blockTimestamp)` repository method using a lateral join or `DISTINCT ON` pattern, and call it from a batch-aware version of `findStorageByBlockTimestamp()` analogous to `findStorageBatch()`.
3. **Apply per-DB-query throttling or cap the number of storage reads per historical call**: Limit the number of distinct slot reads allowed per historical `eth_call` invocation.

### Proof of Concept

1. Deploy or identify a contract on the network that reads a large number of distinct storage slots (e.g., a mapping with 500+ entries read in a single view function).
2. Record a historical block number at which the contract existed.
3. Send repeated `POST /api/v1/contracts/call` requests with `"block": "<historical_block_number>"` and calldata targeting the storage-reading function.
4. Monitor the DB server: each request will generate N individual queries against `contract_state_change` (one per slot), while an equivalent call with `"block": "latest"` generates a single batched query.
5. At 500 req/s with 500 slots per call, observe ~250,000 DB queries/second vs. ~500 for the latest-block equivalent — a >30% increase in DB load achievable without brute force, using only the standard public API.

        if (!cacheProperties.isEnableBatchContractSlotCaching()) {
