### Title
Unauthenticated Contradictory Filter Combination Causes Repeated Full Index Scans on `record_file` Table (DoS)

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts both `block.number` and `timestamp` query parameters and maps them independently to SQL `WHERE` conditions without any cross-filter contradiction check. An unprivileged attacker can submit logically impossible filter combinations (e.g., `block.number=gte:1&timestamp=lte:0.000000001`) that force PostgreSQL to perform a full index scan on the `record_file` table for every request, returning empty results each time. Repeated at scale, this degrades database performance for all users.

### Finding Description

**Code path:**

`getBlocks()` in `rest/controllers/blockController.js` (lines 101–112): [1](#0-0) 

calls `utils.buildAndValidateFilters(req.query, acceptedBlockParameters)` with no custom validator, then `extractSqlFromBlockFilters(filters)` (lines 63–88): [2](#0-1) 

`extractSqlFromBlockFilters` maps each filter independently — `block.number` → `index` condition, `timestamp` → `consensus_end` condition — with no cross-filter contradiction check. The resulting `whereQuery` array is passed directly to `RecordFileService.getBlocks()` (lines 149–162): [3](#0-2) 

`buildWhereSqlStatement` (lines 8–17) simply ANDs all conditions together and executes the query: [4](#0-3) 

The generated SQL for `block.number=gte:1&timestamp=lte:0.000000001` is:
```sql
SELECT ... FROM record_file
WHERE index >= $1 AND consensus_end <= $2
ORDER BY index DESC
LIMIT 25
-- $1 = 1, $2 = 1 (nanosecond)
```

No real block has `consensus_end <= 1` nanosecond (all Hedera mainnet blocks have timestamps from 2019 onward). PostgreSQL, using the `index` index to satisfy `ORDER BY index DESC`, must scan the entire index from the highest value downward before concluding no rows match, because it cannot know in advance the result set is empty.

**Why existing checks fail:**

`filterValidityChecks` (utils.js lines 278–396) validates each filter value in isolation — `block.number=gte:1` and `timestamp=lte:0.000000001` are both individually valid: [5](#0-4) 

`filterDependencyCheck` (utils.js lines 402–435) only enforces that `transaction.index` requires `block.number`/`block.hash`, and that `block.number` and `block.hash` cannot be combined. It has no logic to detect contradictions between `block.number` and `timestamp`: [6](#0-5) 

The `parseTimestampFilters()` utility (utils.js lines 1583–1681) **can** detect empty/contradictory ranges (line 1671 returns `Range()` when `difference <= 0n`): [7](#0-6) 

`contractController.js` uses `optimizeTimestampFilters()` (lines 384–404) which calls `parseTimestampFilters()` and short-circuits on an empty range: [8](#0-7) 

This protection is entirely absent in `blockController.js`. No rate-limiting middleware was found in the REST layer. [9](#0-8) 

### Impact Explanation
The `record_file` table has one row per ~2-second Hedera consensus round. On mainnet (running since 2019), this is tens of millions of rows. Each contradictory request forces a full sequential scan of the `record_file__consensus_end` or `record_file` index before returning empty results, because the `LIMIT 25` clause only terminates early when matching rows are found — not when the scan is exhausted. Under concurrent load from multiple attackers or a single attacker with connection pooling, this saturates I/O and CPU on the database, degrading response times for all legitimate API consumers. Severity: **Medium** (availability impact, no data confidentiality/integrity loss).

### Likelihood Explanation
The endpoint is unauthenticated and publicly accessible. The exploit requires no special knowledge beyond the public API documentation listing `block.number` and `timestamp` as accepted parameters. The contradictory combination is trivially constructable, requires no credentials, and is infinitely repeatable from a single IP or distributed across many. Automated tooling (e.g., `ab`, `wrk`, or a simple loop) can sustain thousands of requests per second.

### Recommendation
1. **Short-circuit on logically empty filter combinations**: Before calling `RecordFileService.getBlocks()`, detect when the combined `block.number` range and `timestamp` range cannot overlap. The existing `parseTimestampFilters()` utility already handles empty-range detection; a similar utility should be applied to the `block.number` (index) range.
2. **Adopt the `optimizeTimestampFilters` pattern**: Mirror the approach in `contractController.js` — after parsing filters, check if the resulting range is empty and return an empty result immediately without hitting the database.
3. **Add rate limiting**: Apply per-IP or per-connection rate limiting at the API gateway or middleware layer for all `/api/v1/blocks` requests.
4. **Add a DB-level query timeout**: Ensure all queries against `record_file` have a statement timeout to bound worst-case scan duration.

### Proof of Concept
```bash

curl "https://<mirror-node-host>/api/v1/blocks?block.number=gte:1&timestamp=lte:0.000000001"
# Returns: {"blocks":[],"links":{"next":null}}

# DoS amplification: flood with contradictory requests (no auth required)
for i in $(seq 1 10000); do
  curl -s "https://<mirror-node-host>/api/v1/blocks?block.number=gte:1&timestamp=lte:0.000000001" &
done
wait
# Each request causes a full index scan on record_file; DB CPU/IO spikes,
# legitimate queries begin timing out or returning degraded latency.
```

```javascript
const buildWhereSqlStatement = (whereQuery) => {
  let where = '';
  const params = [];
  for (let i = 1; i <= whereQuery.length; i++) {
    where += `${i === 1 ? 'where' : 'and'} ${whereQuery[i - 1].query} $${i} `;
    params.push(whereQuery[i - 1].param);
  }

  return {where, params};
};
```

```javascript
const filterDependencyCheck = (query) => {
  const badParams = [];
  let containsBlockNumber = false;
  let containsBlockHash = false;
  let containsTransactionIndex = false;
  for (const key of Object.keys(query)) {
    if (key === constants.filterKeys.TRANSACTION_INDEX) {
      containsTransactionIndex = true;
    } else if (key === constants.filterKeys.BLOCK_NUMBER) {
      containsBlockNumber = true;
    } else if (key === constants.filterKeys.BLOCK_HASH) {
      containsBlockHash = true;
    }
  }

  if (containsTransactionIndex && !(containsBlockNumber || containsBlockHash)) {
    badParams.push({
      key: constants.filterKeys.TRANSACTION_INDEX,
      error: 'transaction.index requires block.number or block.hash filter to be specified',
      code: 'invalidParamUsage',
    });
  }

  if (containsBlockHash && containsBlockNumber) {
    badParams.push({
      key: constants.filterKeys.BLOCK_HASH,
      error: 'cannot combine block.number and block.hash',
      code: 'invalidParamUsage',
    });
  }

  if (badParams.length) {
    throw InvalidArgumentError.forRequestValidation(badParams);
  }
```

```javascript
const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
```
