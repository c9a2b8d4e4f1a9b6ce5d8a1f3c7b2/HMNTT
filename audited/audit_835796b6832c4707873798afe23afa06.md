### Title
Unbounded Paginated Enumeration DoS via `GET /api/v1/blocks` â€” No Rate Limiting on REST API

### Summary
The `getBlocks()` handler in `rest/controllers/blockController.js` accepts `block.number=gte:0&order=asc&limit=100` from any unauthenticated caller and returns a `next` pagination link pointing to the next page. Because the REST API (Node.js/Express) has **no per-IP or global rate-limiting middleware**, an attacker can follow these links in a tight automated loop, issuing an unbounded stream of sequential DB queries that collectively perform a full sequential scan of the entire `record_file` table, exhausting database I/O and connection pool resources.

### Finding Description

**Code path â€” controller:**
`rest/controllers/blockController.js`, `getBlocks()` (lines 101â€“112):
```js
getBlocks = async (req, res) => {
  const filters = utils.buildAndValidateFilters(req.query, acceptedBlockParameters);
  const formattedFilters = this.extractSqlFromBlockFilters(filters);
  const blocks = await RecordFileService.getBlocks(formattedFilters);
  res.locals[responseDataLabel] = {
    blocks: blocks.map((model) => new BlockViewModel(model)),
    links: { next: this.generateNextLink(req, blocks, formattedFilters) },
  };
};
```

**Limit enforcement** (`extractLimitFromFilters`, lines 57â€“61): caps the per-page limit at `getEffectiveMaxLimit()` (default `responseLimit.max = 100`). This only bounds rows per response, not request frequency.

**Next-link generation** (`generateNextLink`, lines 90â€“99): emits a `next` link whenever `blocks.length === filters.limit`, advancing the cursor via `block.number=gt:{last_index}`. This is correct pagination behavior but becomes a DoS vector without rate limiting.

**DB query** (`RecordFileService.getBlocks`, `rest/service/recordFileService.js` lines 149â€“162):
```sql
SELECT ... FROM record_file
WHERE index >= $1
ORDER BY index ASC
LIMIT 100
```
Each paginated request issues this query. With millions of blocks on mainnet, a full traversal requires tens of thousands of sequential DB queries.

**Root cause â€” no rate limiting on the REST API:**
`rest/server.js` (lines 67â€“98) registers the following middleware stack: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`. There is **no rate-limiting middleware** (e.g., `express-rate-limit`). A `grep` across all of `rest/**` for `rateLimit|rate-limit|express-rate|throttle|rateLimiter` returns **zero hits in application code** â€” only documentation references in `openapi.yml`.

The `ThrottleConfiguration.java` / `ThrottleManagerImpl.java` throttle found in the repo applies exclusively to the **web3 Java service** (`/api/v1/contracts/call`), not to the Node.js REST API.

**Failed assumption:** The per-page `LIMIT 100` cap was assumed sufficient to prevent abuse. It bounds data per response but does not prevent an attacker from issuing thousands of paginated requests per second.

### Impact Explanation
An attacker running a simple loop following `next` links from block 0 to the chain tip forces the database to execute a full sequential scan of the `record_file` table across thousands of queries. Multiple concurrent attackers amplify this linearly. Effects include: exhaustion of the DB connection pool (bounded by `pg` pool size), elevated disk I/O on the PostgreSQL host, increased query latency for all other API consumers, and potential service unavailability for legitimate users. Because the `record_file` table grows continuously (one row per ~2-second consensus round), the attack surface grows over time.

### Likelihood Explanation
No authentication, no API key, no CAPTCHA, and no rate limit are required. Any attacker with a script and internet access can execute this. The `next` link is machine-readable and self-describing, making automation trivial. The attack is repeatable from any number of IPs. It is indistinguishable from a legitimate bulk data consumer at the HTTP layer.

### Recommendation
1. **Add per-IP rate limiting** to the Express middleware stack using `express-rate-limit` (or equivalent), applied globally before route handlers in `rest/server.js`.
2. **Add a maximum timestamp/block-number range constraint** for the blocks endpoint (analogous to `maxTimestampRange` / `maxTransactionsTimestampRange` already enforced on other endpoints per `docs/configuration.md` lines 584â€“586), so a single paginated traversal cannot span the full chain history.
3. **Add a DB statement timeout** for the REST Node.js service (similar to `hiero.mirror.restJava.db.statementTimeout = 10000` already present for the Java REST service) to bound individual query execution time.
4. Consider **response caching** for the blocks list endpoint (already supported via Redis, `rest/server.js` lines 94â€“98) to serve repeated identical paginated requests from cache rather than hitting the DB.
