### Title
Unauthenticated Query Amplification via Triple-Subquery Path in `NftService.getQuery()`

### Summary
Any unprivileged user can craft a single HTTP request to `GET /api/v1/accounts/{id}/nfts` with four query parameters (`token.id=gte:X`, `token.id=lte:Y`, `serialnumber=gte:A`, `serialnumber=lte:B`) that passes all validation checks and causes `getQuery()` to generate three independent subqueries, each containing a full `LEFT JOIN` against the `entity` table, `UNION ALL`'d together with redundant `ORDER BY` and `LIMIT` clauses at both subquery and outer level. This produces at least 3× the database processing cost of a normal single-subquery request from a single API call, with no authentication or rate-limiting gate in the visible code path.

### Finding Description

**Code path:**

- `rest/controllers/accountController.js`, `getNftsByAccountId()` (line 90–103): receives raw HTTP query params, calls `extractNftMultiUnionQuery()` with no auth check.
- `rest/controllers/accountController.js`, `extractNftMultiUnionQuery()` (lines 34–82): parses user-supplied `token.id` and `serialnumber` filters into `Bound` objects, then calls `getLowerFilters()`, `getInnerFilters()`, `getUpperFilters()`.
- `rest/controllers/baseController.js`, `getLowerFilters()` (lines 131–146), `getInnerFilters()` (lines 154–168), `getUpperFilters()` (lines 176–183): all three return non-empty arrays when primary has both a lower (`gte`) and upper (`lte`) bound AND secondary has both a lower (`gte`) and upper (`lte`) bound.
- `rest/service/nftService.js`, `getQuery()` (lines 93–105): iterates `[lower, inner, upper]`, filters out empty arrays, and calls `getSubQuery()` for each non-empty one — each `getSubQuery()` emits a full `NftService.nftQuery` which contains `LEFT JOIN entity e on e.id = nft.token_id` (lines 33–34).
- `rest/service/nftService.js`, `getQuery()` (line 122): when `subQueries.length > 1`, wraps all in `UNION ALL` with an outer `ORDER BY` + `LIMIT`.

**Root cause:** The `getQuery()` function unconditionally generates one subquery per non-empty filter bucket. The maximum of three subqueries is reachable via a single, unauthenticated HTTP request. No complexity cap, no per-request cost limit, and no authentication requirement exists in this code path.

**Why validation fails to prevent it:**

`validateBoundsRange()` only rejects a bound that has *both* a range *and* an `eq` filter simultaneously — it does not reject having both a lower and upper range bound together.

`validateLowerBounds()` only throws when `primary.lower.operator === gt` (strict); using `gte` bypasses it entirely.

`validateUpperBounds()` only throws when `primary.upper.operator === lt` (strict); using `lte` bypasses it entirely.

The combination `token.id=gte:X&token.id=lte:Y&serialnumber=gte:A&serialnumber=lte:B` satisfies all four conditions to populate `lower`, `inner`, and `upper` simultaneously and clears every validation check.

This is confirmed by the existing test case `'token closed range and serial closed range'` in `rest/__tests__/service/nftService.test.js` (lines 251–296), which explicitly documents the three-subquery output as expected behavior — meaning the design accepts this path without any guard.

### Impact Explanation

Each of the three subqueries performs a `LEFT JOIN` against the `entity` table (which can be large), plus its own `ORDER BY` and `LIMIT`. The outer query then re-sorts and re-limits the merged result. A single crafted request therefore imposes at minimum 3× the I/O, join, and sort cost of a baseline single-subquery request. On a busy node, repeated such requests from one or more clients can sustain elevated CPU and I/O load well above the 30% threshold without any brute-force volume requirement — a modest concurrency of crafted requests suffices.

### Likelihood Explanation

The exploit requires zero privileges: the endpoint is public, the parameters are standard query-string fields documented in the API, and the four-parameter combination is trivially discoverable from the API spec or by reading the test suite. It is fully repeatable and automatable. No special knowledge of internals is needed beyond knowing the parameter names.

### Recommendation

1. **Cap query complexity at the controller layer**: in `extractNftMultiUnionQuery()`, after computing `lower`/`inner`/`upper`, count the non-empty buckets and reject (HTTP 400) any request that would produce more than one subquery unless the caller is authenticated/privileged.
2. **Alternatively, collapse the three-bucket path**: for the closed-range case, rewrite the query as a single subquery with compound `WHERE` conditions instead of three `UNION ALL` branches.
3. **Add rate limiting** at the API gateway or middleware layer specifically for endpoints that can trigger multi-subquery execution.
4. **Enforce a query-cost budget**: track the number of subqueries that will be generated and return an error if it exceeds a configurable threshold.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1234/nfts?token.id=gte:1&token.id=lte:9999999&serialnumber=gte:1&serialnumber=lte:9999999&limit=100
```

This single unauthenticated request triggers the following generated SQL (confirmed by the test at `rest/__tests__/service/nftService.test.js` lines 267–295):

```sql
(select ... from nft left join entity e on e.id = nft.token_id
 where account_id = $1 and token_id = $3 and serial_number >= $4
 order by token_id desc, serial_number desc limit $2)
union all
(select ... from nft left join entity e on e.id = nft.token_id
 where account_id = $1 and token_id > $5 and token_id < $6
 order by token_id desc, serial_number desc limit $2)
union all
(select ... from nft left join entity e on e.id = nft.token_id
 where account_id = $1 and token_id = $7 and serial_number <= $8
 order by token_id desc, serial_number desc limit $2)
order by token_id desc, serial_number desc limit $2
```

Three full `LEFT JOIN entity` scans, three sorts, one outer sort — all from one HTTP request, no credentials required.   getQuery(query) {
    const {lower, inner, upper, order, ownerAccountId, limit, spenderIdInFilters, spenderIdFilters} = query;
    const params = [ownerAccountId, limit];
    const accountIdCondition = `${Nft.ACCOUNT_ID} = $1`;
    const limitClause = super.getLimitQuery(2);
    const orderClause = super.getOrderByQuery(
      ...NftService.orderByColumns.map((column) => OrderSpec.from(column, order))
    );

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) =>
        this.getSubQuery(
          filters,
          params,
          accountIdCondition,
          limitClause,
          orderClause,
          spenderIdInFilters,
          spenderIdFilters
        )
      );

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = this.getSubQuery(
        [],
        params,
        accountIdCondition,
        limitClause,
        orderClause,
        spenderIdInFilters,
        spenderIdFilters
      );
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
    }

    return {sqlQuery, params};
  }

  extractNftMultiUnionQuery(filters, ownerAccountId) {
    const bounds = {
      primary: new Bound(filterKeys.TOKEN_ID, 'token_id'),
      secondary: new Bound(filterKeys.SERIAL_NUMBER, 'serial_number'),
    };
    let limit = defaultLimit;
    let order = orderFilterValues.DESC;
    const spenderIdFilters = [];
    const spenderIdInFilters = [];

    for (const filter of filters) {
      switch (filter.key) {
        case filterKeys.SERIAL_NUMBER:
          bounds.secondary.parse(filter);
          break;
        case filterKeys.TOKEN_ID:
          bounds.primary.parse(filter);
          break;
        case filterKeys.LIMIT:
          limit = filter.value;
          break;
        case filterKeys.ORDER:
          order = filter.value;
          break;
        case filterKeys.SPENDER_ID:
          filter.operator === utils.opsMap.eq ? spenderIdInFilters.push(filter) : spenderIdFilters.push(filter);
          break;
        default:
          break;
      }
    }

    this.validateFilters(bounds, spenderIdFilters);

    const lower = this.getLowerFilters(bounds);
    const inner = this.getInnerFilters(bounds);
    const upper = this.getUpperFilters(bounds);
    return {
      bounds,
      lower,
      inner,
      upper,
      order,
      ownerAccountId,
      limit,
      spenderIdInFilters,
      spenderIdFilters,
    };
  }
