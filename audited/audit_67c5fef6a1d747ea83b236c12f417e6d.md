### Title
Cache Stampede (Thundering Herd) DoS via Concurrent Cache-Miss Requests

### Summary
`responseCacheCheckHandler()` performs a Redis cache lookup and, on a miss, unconditionally passes every concurrent request through to the database with no in-flight deduplication, no mutex, and no stampede protection. An unprivileged attacker can send a large burst of concurrent requests to any cacheable endpoint at the moment the cache is cold or expired, causing all requests to simultaneously hit the database and exhaust the DB connection pool, resulting in a denial-of-service for all users.

### Finding Description
**Exact code path:**

`rest/middleware/responseCacheHandler.js`, `responseCacheCheckHandler()`, lines 40–48:
```js
const responseCacheCheckHandler = async (req, res) => {
  const startTime = res.locals[requestStartTime] || Date.now();
  const responseCacheKey = cacheKeyGenerator(req);
  const cachedTtlAndValue = await getCache().getSingleWithTtl(responseCacheKey); // (1)

  if (!cachedTtlAndValue) {          // (2) cache miss
    res.locals[responseCacheKeyLabel] = responseCacheKey;
    return;                          // (3) falls through to DB handler
  }
  ...
```

`rest/cache.js`, `getSingleWithTtl()`, lines 64–83 — performs a plain Redis `MULTI/EXEC` read with no lock, no "in-flight" marker, and no stampede guard:
```js
async getSingleWithTtl(key) {
  if (!this.ready) { return undefined; }
  const result = await this.redis.multi().ttl(key).get(key).exec()...
  const rawValue = result[1][1];
  if (rawValue) { return {ttl: result[0][1], value: JSONParse(rawValue)}; }
  return undefined;   // cache miss → no protection
}
```

**Root cause:** Node.js's async event loop allows all N concurrent requests to reach the `await getCache().getSingleWithTtl(...)` call before any of them receives a response. Because there is no in-flight deduplication (no promise sharing, no Redis `SET NX` lock, no probabilistic early expiration), every one of the N requests independently observes a cache miss and independently falls through to the database handler (`return` at line 47 passes control to the next Express middleware, which issues a DB query). The `responseCacheUpdateHandler` only writes to Redis after the DB query completes, so all N queries race to completion before any of them can warm the cache for the others.

**Why existing checks fail:** The only guard in `getSingleWithTtl` is `if (!this.ready)` (Redis connectivity check). There is no rate limiter, no request coalescing, no mutex, and no "lock key" pattern anywhere in the middleware chain (`server.js` lines 94–141 confirm no rate-limiting middleware is registered).

### Impact Explanation
A successful attack exhausts the PostgreSQL connection pool (initialized at `rest/server.js` line 49 via `initializePool()`), causing all subsequent DB-dependent requests to queue or fail. Because the cache is warmed only after the first successful DB response, the attacker can sustain the attack by continuously sending bursts timed to each cache expiry cycle (TTL is as low as `DEFAULT_REDIS_EXPIRY = 1` second for some responses). This results in full service degradation (HTTP 503 / timeouts) for all legitimate users — a complete availability impact with no data confidentiality or integrity impact.

### Likelihood Explanation
The attack requires zero privileges — the mirror node REST API is a public, unauthenticated read API. The attacker needs only an HTTP client capable of sending concurrent requests (e.g., `ab`, `wrk`, `hey`, or a simple script). The attack is trivially repeatable: the attacker can target any popular endpoint (e.g., `/api/v1/transactions/:id`) and re-trigger the stampede every TTL seconds. No special knowledge of the system internals is required beyond knowing the public API URL.

### Recommendation
Implement a **cache stampede / thundering herd guard** using one of the following patterns:

1. **Redis `SET NX` lock (recommended):** Before falling through to the DB, attempt to acquire a short-lived Redis lock for the cache key (`SET key:lock 1 NX EX <short_ttl>`). Only the first request that acquires the lock proceeds to the DB; all others either wait and retry the cache read, or return a `503 Retry-After` response.

2. **In-process promise coalescing:** Maintain a `Map<cacheKey, Promise>` in the Node.js process. If a request finds an in-flight promise for the same key, it awaits that promise instead of issuing a new DB query.

3. **Probabilistic early expiration (XFetch):** Recompute the cache entry slightly before it expires, eliminating the cold-cache window entirely.

Additionally, add a global rate limiter (e.g., `express-rate-limit`) to bound the number of concurrent requests per IP, reducing the blast radius of any stampede attempt.

### Proof of Concept
```bash
# 1. Identify a cacheable endpoint (e.g., a specific transaction)
ENDPOINT="https://<mirror-node-host>/api/v1/transactions/0.0.1234@1234567890.000000000"

# 2. Wait for (or force) cache expiry — default TTL can be as low as 1 second

# 3. Send 500 concurrent requests simultaneously (all will miss the cache)
ab -n 500 -c 500 "$ENDPOINT"
# or
hey -n 500 -c 500 "$ENDPOINT"

# 4. Observe: all 500 requests bypass the cache and hit the DB simultaneously.
#    DB connection pool exhaustion causes subsequent requests to time out or return 503.
#    Repeat every TTL seconds to sustain the DoS.
```
