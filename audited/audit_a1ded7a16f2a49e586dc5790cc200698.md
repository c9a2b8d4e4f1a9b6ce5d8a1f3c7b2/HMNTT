### Title
Unauthenticated `/health/readiness` Endpoint Opens New PostgreSQL Connection Per Request with 10-Second Timeout, Enabling DB Connection Exhaustion

### Summary
The `NewHealthController` function registers `/health/readiness` as a publicly accessible, unauthenticated endpoint that invokes `postgres.New(...)` from `hellofresh/health-go/v4`. This check opens a **new** `sql.DB` and a live PostgreSQL connection on every invocation, holding it for up to 10 seconds. With no rate limiting or concurrency cap on this path, an unprivileged attacker can flood the endpoint to exhaust PostgreSQL's connection limit and drive measurable resource overhead on the node.

### Finding Description

**Exact code location:** `rosetta/app/middleware/health.go`, `NewHealthController()`, lines 37–43.

```go
readinessChecks := []health.Config{
    {
        Name:      "postgresql",
        Timeout:   time.Second * 10,   // ← connection held up to 10 s
        SkipOnErr: false,
        Check:     postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()}),
    },
```

**Root cause – `postgres.New` opens a fresh connection per invocation.**
The `hellofresh/health-go/v4` (v4.7.0, declared in `rosetta/go.mod` line 12) postgres check implementation calls `sql.Open(dsn)` + `db.PingContext(ctx)` + `defer db.Close()` inside the returned closure. A new `*sql.DB` (and therefore a new TCP connection to PostgreSQL) is created on **every** call to the check function. The 10-second context timeout is the upper bound for how long that connection is held open before the context is cancelled.

**No rate limiting on health paths.**
The Traefik middleware chain (rate limit `average: 10`, `inFlightReq amount: 5`, circuit breaker) is attached only to the ingress paths `/rosetta/account`, `/rosetta/block`, etc. (`charts/hedera-mirror-rosetta/values.yaml` lines 149–166). The `/health/readiness` path is not listed in the ingress rules (lines 125–133) and therefore bypasses all of those controls entirely. The Go HTTP server itself applies no per-route concurrency limit or authentication to this handler.

**Failed assumption:** The design assumes health probes are issued only by the Kubernetes kubelet at a low, controlled rate. No code-level guard enforces this assumption.

### Impact Explanation

PostgreSQL is configured with `max_connections = 600` (per `docs/database/README.md` line 29). Each concurrent request to `/health/readiness` opens one additional connection outside the application's GORM pool (`rosetta/app/db/db.go` lines 31–33). Under a flood:

1. N concurrent requests → N simultaneous new PostgreSQL connections.
2. PostgreSQL connection establishment is CPU- and memory-intensive (~5–10 MB per backend process).
3. As the connection count approaches `max_connections`, new connection attempts from the application pool queue or fail, introducing wait overhead and errors for legitimate traffic.
4. If the DB is already under load (which the flood itself induces), `db.PingContext` takes longer, holding each health-check connection open for up to the full 10-second timeout, multiplying the sustained connection count.
5. Result: measurable increase in PostgreSQL CPU, memory, and connection-wait latency — consistent with the ≥30% resource consumption increase threshold described in the scope.

### Likelihood Explanation

- **No privilege required:** The endpoint is unauthenticated and, depending on deployment, reachable directly on the service port (the Go server binds `0.0.0.0:<port>` — `rosetta/main.go` line 221).
- **Trivially repeatable:** A single `ab -n 10000 -c 500 http://<host>/health/readiness` or equivalent is sufficient.
- **No exploit complexity:** Standard HTTP flood; no knowledge of the API schema needed.
- **Amplification:** Each HTTP request maps 1:1 to a PostgreSQL connection establishment, making the amplification ratio high relative to attacker bandwidth.

### Recommendation

1. **Reuse a single persistent `sql.DB` for the health check** rather than opening a new one per invocation. Instantiate `sql.Open` once at startup and pass the handle into the check closure, calling only `db.PingContext(ctx)` per request.
2. **Apply a concurrency cap** to the `/health/readiness` handler (e.g., a semaphore or `golang.org/x/net/netutil` limiter) so that at most N (e.g., 5) simultaneous health checks can be in-flight.
3. **Add the health paths to the ingress** (or a separate internal-only ingress) so the existing Traefik `inFlightReq` and `rateLimit` middleware applies, or restrict the health port to cluster-internal traffic via a `NetworkPolicy`.
4. **Reduce the timeout** from 10 seconds to 2–3 seconds to limit the maximum connection-hold duration.
