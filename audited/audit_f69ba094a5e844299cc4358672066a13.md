### Title
Unbounded Sequential DB Queries in `updateOperationAccountAlias` Enable Cache-Bypass DoS via Public Block Endpoints

### Summary
The `updateOperationAccountAlias` function in `rosetta/app/services/block_service.go` issues one sequential `GetAccountAlias()` database round-trip per operation per transaction for every `AccountId` not present in the in-process LRU cache. Because the Rosetta `/block` and `/block/transaction` endpoints require no authentication and the Go application enforces no rate limiting, an unprivileged attacker can continuously request blocks containing many unique `AccountId` values to sustain a high volume of sequential DB queries, materially increasing database CPU and connection consumption.

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 104–130:

```go
func (s *blockAPIService) updateOperationAccountAlias(
    ctx context.Context,
    transactions ...*types.Transaction,
) *rTypes.Error {
    for _, transaction := range transactions {
        operations := transaction.Operations
        for index := range operations {
            accountId := operations[index].AccountId
            if cached, found = s.entityCache.Get(accountId.GetId()); !found {
                result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // DB round-trip
                ...
                s.entityCache.Set(result.GetId(), result)
            }
        }
    }
    return nil
}
```

`GetAccountAlias()` (`rosetta/app/persistence/account.go`, lines 109–136) executes a raw SQL query against the `entity` table for every cache miss.

**Root cause:** Queries are issued one-at-a-time inside a nested loop with no batching. The only mitigation is an in-process LRU cache (default capacity 524,288 entries, no TTL) keyed by `int64` entity ID. The cache is shared across requests but is trivially exhausted or bypassed.

**Cache bypass mechanism:**
- The LRU cache is keyed by `accountId.GetId()` (the numeric entity ID). Any `AccountId` not previously seen causes a DB query.
- On a live Hedera network, blocks routinely contain hundreds of transactions involving many distinct accounts (token transfers, contract calls, etc.). A single `/block` request for a block with *M* transactions × *K* operations each = up to *M×K* sequential DB queries if none are cached.
- Once the attacker has exhausted the 524,288-entry LRU by cycling through enough distinct blocks, previously evicted entries cause cache misses again on re-request.

**No application-level rate limiting:** The Go server's middleware stack (`main.go`, lines 217–219) applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no request rate limiter. The Traefik `rateLimit` / `inFlightReq` configuration in the Helm chart (`values.yaml`, lines 152–165) is infrastructure-optional and disabled by default (`global.middleware: false`).

### Impact Explanation

Each `/block` request for a block with *N* uncached unique `AccountId` values generates *N* sequential DB queries. With concurrent requests targeting different blocks, the attacker multiplies DB load linearly. On mainnet-scale blocks (hundreds of transactions, multiple operations each), a modest flood of requests (e.g., 20 concurrent clients each requesting a different block) can produce thousands of sequential DB round-trips per second, exhausting the DB connection pool (default `maxOpenConnections: 100`) and driving CPU well above the 30% threshold. The impact is amplified because queries are sequential, not parallel, so each request holds a DB connection open for the full duration of the loop.

### Likelihood Explanation

The attack requires no credentials, no special network access, and no on-chain capability. Any internet-accessible Rosetta deployment is reachable. Block indices and hashes are public. The attacker only needs to enumerate block numbers and send HTTP POST requests to `/block`. The attack is fully repeatable and scriptable with standard HTTP tooling.

### Recommendation

1. **Batch the alias lookups**: Collect all unique `AccountId` values across all transactions/operations first, deduplicate, issue a single `WHERE id = ANY(...)` query, then populate the cache and update operations — eliminating the per-operation DB round-trip.
2. **Add application-level rate limiting** to the Rosetta Go server (e.g., `golang.org/x/time/rate` per-IP token bucket) rather than relying solely on optional infrastructure middleware.
3. **Pre-warm or persist the entity cache** across restarts to reduce cold-start exposure.
4. **Add a TTL** to cache entries so stale entries are eventually replaced, but pair this with batching to avoid re-introducing the per-operation query pattern.

         \"block_identifier\":{\"index\":$BLOCK_IDX}}" &

wait
