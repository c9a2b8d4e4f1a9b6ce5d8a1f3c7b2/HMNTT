### Title
Unauthenticated POST /api/v1/network/fees with mode=STATE Causes DB Connection Pool Exhaustion via Uncached Double-Query per Token in FeeTokenStore

### Summary
The `POST /api/v1/network/fees?mode=STATE` endpoint is publicly accessible with no authentication or rate limiting in the `rest-java` module. When `mode=STATE`, each token ID referenced in the submitted transaction body causes `FeeTokenStore.get()` to issue two sequential, uncached database queries (`tokenRepository.findById()` + `customFeeRepository.findById()`). An attacker flooding this endpoint with concurrent requests containing multi-token CryptoTransfer bodies can exhaust the HikariCP connection pool, denying database access to all mirror node services.

### Finding Description

**Exact code path:**

`NetworkController.estimateFees()` (lines 110–124) accepts any unauthenticated POST with no rate limiting: [1](#0-0) 

When `mode=STATE`, `FeeEstimationService.estimateFees()` constructs a `FeeEstimationFeeContext` that wires in the live `FeeTokenStore`: [2](#0-1) 

`FeeEstimationFeeContext.readableStore()` returns the live `feeTokenStore` for every `ReadableTokenStore` lookup the fee calculator makes: [3](#0-2) 

`FeeTokenStore.get()` issues two uncached DB queries per token ID — one to `tokenRepository` and one to `customFeeRepository` — with no in-memory cache or deduplication: [4](#0-3) [5](#0-4) 

The test comment explicitly confirms this is intentional live-DB behavior with no caching: [6](#0-5) 

**Root cause:** `FeeTokenStore` is a stateless `@Named` singleton with no cache. Every invocation of `get()` hits the database twice. The `rest-java` module's filter chain contains only `LoggingFilter` and `MetricsFilter` — no rate limiting exists for this endpoint, unlike the `web3` module which has `ThrottleConfiguration`/`ThrottleManagerImpl`: [7](#0-6) 

The shared HikariCP pool is configured via `CommonConfiguration` with no rest-java-specific pool size floor: [8](#0-7) 

### Impact Explanation
Exhausting the HikariCP connection pool blocks all database-dependent operations across the mirror node — including the importer's record ingestion pipeline. New Hedera transactions cannot be confirmed or indexed by the mirror node, effectively causing a total mirror node outage for all downstream consumers (wallets, explorers, dApps). The impact matches the stated severity: network-wide inability to confirm new transactions from the mirror node's perspective.

### Likelihood Explanation
No privileges are required. The endpoint is publicly documented in the OpenAPI spec: [9](#0-8) 

A CryptoTransfer protobuf body can reference multiple token IDs in its `tokenTransfers` list. The mirror node does not validate the transaction against network-level limits (it only estimates fees). An attacker with basic protobuf knowledge can craft a request with many distinct token IDs and flood the endpoint with concurrent HTTP connections using standard tools (e.g., `wrk`, `ab`, `curl` in parallel). The attack is repeatable, requires no account, no key, and no on-chain state.

### Recommendation
1. **Add rate limiting** to the `POST /api/v1/network/fees` endpoint in `rest-java`, mirroring the `ThrottleConfiguration` pattern from the `web3` module.
2. **Add a per-request token ID cache** inside `FeeTokenStore.get()` (e.g., a `ThreadLocal<Map<TokenID, Token>>` or a request-scoped cache) so repeated lookups of the same token within one fee estimation do not re-query the database.
3. **Limit the number of distinct token IDs** the fee estimator will process per request (e.g., reject or cap at the network's `ledger.tokenTransfers.maxLen` limit, currently 10 on mainnet).
4. **Isolate the fee estimation DB connection pool** from the main pool so exhaustion cannot cascade to the importer.

### Proof of Concept
```bash
# 1. Build a CryptoTransfer protobuf with N distinct token IDs (e.g., N=50)
#    using the Hedera Java/Python SDK or raw protobuf encoding.
#    Each TokenTransferList entry references a different tokenNum.

# 2. Flood the endpoint concurrently (e.g., 200 parallel connections):
for i in $(seq 1 200); do
  curl -s -X POST \
    "https://<mirror-node>/api/v1/network/fees?mode=STATE" \
    -H "Content-Type: application/protobuf" \
    --data-binary @multi_token_cryptotransfer.pb &
done
wait

# 3. Observe: subsequent requests to any mirror node endpoint
#    (including GET /api/v1/transactions) begin timing out or returning 500
#    as the HikariCP pool is saturated.
#    The importer's DB writes also stall, halting record ingestion.
```

    private static List<CustomFee> getCustomFees(final long tokenId, final CustomFeeRepository customFeeRepository) {
        return customFeeRepository
                .findById(tokenId)
                .filter(customFee -> !customFee.isEmptyFee())
                .map(customFee -> {
                    final var count = size(customFee.getFixedFees())
                            + size(customFee.getFractionalFees())
                            + size(customFee.getRoyaltyFees());
