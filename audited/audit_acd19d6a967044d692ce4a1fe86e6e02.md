### Title
Unbounded Synchronous `unzipSync()` on Every Non-Gzip Cache Hit Enables CPU Exhaustion DoS

### Summary
When the response cache stores a compressed entry (via `gzipSync` at write time), any subsequent cache hit from a client that does not advertise `accept-encoding: gzip` triggers a synchronous `unzipSync()` call on the Node.js event loop for every single request. The decompressed result is never memoized, and no rate limiting exists in the REST middleware cache path, allowing an unprivileged attacker to saturate CPU by flooding the server with non-gzip requests to any warm cached endpoint.

### Finding Description

**Cache key excludes `accept-encoding`** — `cacheKeyGenerator` hashes only `req.originalUrl`: [1](#0-0) 

This means a single compressed cache entry is shared by all clients regardless of their encoding preference.

**Compression at write time** — `responseCacheUpdateHandler` stores the body compressed when `shouldCompress()` is true: [2](#0-1) [3](#0-2) 

**Synchronous decompression at read time, every hit** — in `responseCacheCheckHandler`, when `negotiate()` returns `false` (client does not accept gzip), `getUncompressedBody()` is called: [4](#0-3) 

`getUncompressedBody()` calls `unzipSync()` synchronously with no result caching: [5](#0-4) 

`unzipSync` is a blocking call that occupies the Node.js event loop for the full duration of decompression. The decompressed buffer is discarded after the response is sent — it is never stored back, so the next identical request repeats the full decompression.

**No rate limiting in the REST cache path** — the throttling found in the codebase applies only to the `web3` Java service and the Rosetta Traefik ingress. The `responseCacheCheckHandler` middleware has no per-IP or global request rate limit. [6](#0-5) 

### Impact Explanation
Node.js is single-threaded. Each `unzipSync()` call blocks the event loop, preventing other requests from being processed until decompression completes. For large cached responses (above `compressThreshold`), decompression is non-trivial. A flood of concurrent non-gzip requests to a single warm cached endpoint can peg one CPU core at 100% and stall all other in-flight requests, achieving the >30% node resource consumption increase without any brute-force database or network amplification. The attack is self-sustaining as long as the cache entry remains warm (TTL-bounded but renewable by any legitimate traffic).

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge beyond knowing a valid API endpoint (all endpoints are public and documented), and zero tooling beyond `curl` or any HTTP load tool with `--no-compressed`. The attacker does not need to bypass any authentication, CSRF token, or API key. The only precondition is that the target endpoint's response is large enough to exceed `compressThreshold` and that compression is enabled in config — both are expected in production deployments serving large account/transaction list responses.

### Recommendation
1. **Memoize the decompressed body** inside `CachedApiResponse`: compute and store the uncompressed form lazily on first call so subsequent calls within the same object lifetime return the cached buffer.
2. **Use `zlib.gunzip` (async) instead of `unzipSync`** to avoid blocking the event loop.
3. **Store both compressed and uncompressed forms in Redis** (or store only uncompressed and let the Express compression middleware handle on-the-wire gzip), eliminating per-request decompression entirely.
4. **Add per-IP rate limiting** to the REST middleware stack for cache-hit paths.

### Proof of Concept

**Preconditions:**
- `config.cache.response.compress = true`
- `config.cache.response.compressThreshold` is below the size of a typical list response (e.g., `/api/v1/transactions?limit=100`)

**Steps:**

```bash
# Step 1: Warm the cache (one request with gzip so the entry is stored compressed)
curl -H "accept-encoding: gzip" https://<mirror-node>/api/v1/transactions?limit=100

# Step 2: Flood with non-gzip requests (no accept-encoding header)
# Each hit calls unzipSync() synchronously on the Node.js event loop
ab -n 10000 -c 100 -H "accept-encoding: identity" \
  https://<mirror-node>/api/v1/transactions?limit=100
```

**Expected result:** Node.js process CPU climbs sharply; event loop lag increases; other API endpoints experience latency degradation or timeouts. Server-side logs show all requests served from cache (`from cache (ttl: ...)`) confirming no DB load — the CPU cost is purely from repeated `unzipSync()` calls.

```javascript
// Response middleware that checks for and returns cached response.
const responseCacheCheckHandler = async (req, res) => {
  const startTime = res.locals[requestStartTime] || Date.now();
  const responseCacheKey = cacheKeyGenerator(req);
  const cachedTtlAndValue = await getCache().getSingleWithTtl(responseCacheKey);

  if (!cachedTtlAndValue) {
    res.locals[responseCacheKeyLabel] = responseCacheKey;
    return;
  }

  const {ttl: redisTtl, value: redisValue} = cachedTtlAndValue;
  const cachedResponse = Object.assign(new CachedApiResponse(), redisValue);
  const conditionalHeader = req.get(CONDITIONAL_HEADER);
  const clientCached = conditionalHeader && conditionalHeader === cachedResponse.headers[ETAG_HEADER]; // 304
  const statusCode = clientCached ? httpStatusCodes.UNMODIFIED.code : cachedResponse.statusCode;
  const isHead = req.method === 'HEAD';

  let body;
  const headers = {
    ...cachedResponse.headers,
    ...{[CACHE_CONTROL_HEADER]: `public, max-age=${redisTtl}`},
  };

  if (isHead || clientCached) {
    if (clientCached) {
      delete headers[contentTypeHeader];
    } else {
      // For HEAD requests when status code is not 304, negotiate the encoding and set corresponding headers
      negotiate(cachedResponse, req, res);
    }
  } else {
    const useCompressed = negotiate(cachedResponse, req, res);
    body = useCompressed ? cachedResponse.getBody() : cachedResponse.getUncompressedBody();
  }

  res.set(headers);
  res.status(statusCode);
  if (body !== undefined) {
    res.send(body);
  } else {
    res.end();
  }

  const elapsed = Date.now() - startTime;
  logger.info(
    `${req.ip} ${req.method} ${req.originalUrl} from cache (ttl: ${redisTtl}) in ${elapsed} ms: ${statusCode}`
  );
};
```
