### Title
Unauthenticated Alias-Based DoS via Unbounded DB Queries in `getAirdrops()` — No Rate Limiting in rest-java

### Summary
The `getAirdrops()` method in `TokenAirdropServiceImpl` unconditionally calls `entityService.lookup()`, which executes a live database query for every alias or EVM-address-type account ID. The rest-java module has no rate limiting (unlike the `web3` module which uses bucket4j), and there is no negative-result caching. An unprivileged attacker can flood `/api/v1/accounts/{non-existent-alias}/airdrops/pending` to exhaust the database connection pool and degrade REST API throughput.

### Finding Description

**Exact code path:**

`TokenAirdropServiceImpl.getAirdrops()` (line 20) calls `entityService.lookup(request.getAccountId())` unconditionally before any result is returned: [1](#0-0) 

`EntityServiceImpl.lookup()` (lines 33–37) executes a native SQL query against the `entity` table for alias/EVM-address parameters, and throws `EntityNotFoundException` when the result is empty: [2](#0-1) 

The repository query is a direct DB hit with no caching layer: [3](#0-2) 

The exception is caught by `GenericControllerAdvice.notFound()` and returned as a clean 404 — no log flooding, but the DB query has already been executed: [4](#0-3) 

**Root cause:** No rate limiting exists in the rest-java module. The `web3` module has a full bucket4j `ThrottleConfiguration` with per-second request caps: [5](#0-4) 

The rest-java config directory contains only `LoggingFilter` and `MetricsFilter` — no throttle equivalent:



### Impact Explanation
Every request with a non-existent alias causes one synchronous DB query. A sustained flood saturates the JDBC connection pool, causing legitimate requests to queue or fail with connection-timeout errors. This degrades the entire rest-java service, not just the airdrop endpoint, because the connection pool is shared. The mirror node is a read-only service and does not affect Hedera consensus directly, so "total network shutdown" is not accurate — but full REST API unavailability for all mirror node consumers is a realistic outcome.

### Likelihood Explanation
No authentication, API key, or IP-based throttle is required. Any attacker with HTTP access can trigger this with a single `curl` loop or a low-volume distributed script. The alias parameter space is effectively unbounded (32-byte values), making it trivial to generate non-existent aliases indefinitely. The attack is repeatable and stateless.

### Recommendation
1. Add a bucket4j (or equivalent) rate-limiting filter to the rest-java module, mirroring the `ThrottleConfiguration` already present in `web3`.
2. Add a short-lived negative-result cache (e.g., Caffeine with a 5–10 second TTL) in `EntityServiceImpl.lookup()` for alias and EVM-address lookups to avoid repeated DB hits for the same non-existent key.
3. Consider returning a 400 Bad Request (rather than 404) for structurally valid but unresolvable aliases, to allow upstream proxies/WAFs to apply stricter throttling rules to error responses.

### Proof of Concept
```bash
# Generate a random non-existent alias (base32-encoded, Hedera format)
ALIAS="0.0.AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA"

# Flood the endpoint (no auth required)
while true; do
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/airdrops/pending"
done
```

Each iteration executes `SELECT id FROM entity WHERE alias = ? AND deleted <> true` against the database. Under sustained load (hundreds of concurrent connections), the DB connection pool exhausts, and all rest-java endpoints begin returning 503 or connection-timeout errors.

```java
    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
    }
```
