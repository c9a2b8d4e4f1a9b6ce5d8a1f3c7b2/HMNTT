### Title
Unthrottled DB Query Burst via Concurrent gRPC Subscriptions Bypassing Per-Connection Limit

### Summary
The `PollingTopicMessageRetriever` unthrottled path (`frequency=20ms`, `maxPolls=12`) is triggered for every subscription via the safety-check mechanism in `TopicMessageServiceImpl`. An unprivileged attacker can open subscriptions across multiple TCP connections to bypass the per-connection call limit, causing a coordinated burst of ~5000 DB queries in ~260ms that can starve the database of capacity for transaction confirmation queries.

### Finding Description

**Code location:** `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, lines 94–107 (`PollingContext` constructor) and lines 51–62 (`retrieve`).

**Unthrottled path configuration:** [1](#0-0) 

Default values from `RetrieverProperties.UnthrottledProperties`:
- `pollingFrequency = 20ms` → 50 polls/second
- `maxPolls = 12` → 13 total polls per invocation (~260ms burst)
- `maxPageSize = 5000` [2](#0-1) 

**Trigger path — safety check fires for every subscription:**

In `TopicMessageServiceImpl.subscribeTopic()`, a safety check fires unconditionally after 1 second for any subscription where `!topicContext.isComplete()`. For subscriptions with no `endTime`, `isComplete()` always returns `false`: [3](#0-2) [4](#0-3) 

The safety check calls `missingMessages(topicContext, null)`, which calls `retrieve(gapFilter, false)` — the unthrottled path: [5](#0-4) 

**`isComplete()` for unthrottled never terminates early without a limit:** [6](#0-5) 

Without a `limit` on the filter, `limitHit = false` always, so the unthrottled retrieval runs all 13 polls (initial + `maxPolls=12`) at 20ms intervals.

**Per-connection limit is bypassable:**

The only connection-level guard is `maxConcurrentCallsPerConnection = 5`: [7](#0-6) [8](#0-7) 

There is no per-IP connection limit, no global subscription cap, and no rate limiting on the gRPC endpoint. An attacker opens 20 TCP connections × 5 calls each = 100 concurrent subscriptions.

**Exploit arithmetic:**
- 100 subscriptions × 13 polls × (1/0.020s) = **~5000 DB queries in ~260ms**
- Each query can request up to 5000 rows (`maxPageSize=5000`)
- The burst repeats each time the attacker cycles subscriptions (close + reopen)

### Impact Explanation

A 5000 QPS burst of large-page DB queries against a shared PostgreSQL instance will saturate connection pool and I/O capacity. The mirror node's importer and REST API share the same database. Transaction confirmation queries (used by consensus nodes and clients verifying finality) execute against the same DB. Under saturation, these queries time out (`statementTimeout = 10000ms`), preventing clients from confirming transaction inclusion and effectively halting observable network finality from the mirror node's perspective. This meets the "network not being able to confirm new transactions" severity threshold.

### Likelihood Explanation

- **No authentication required**: gRPC `subscribeTopic` is unauthenticated; any network-reachable client qualifies.
- **Trivially scriptable**: Opening 20 TCP connections with 5 gRPC streams each is a standard gRPC client operation achievable with `grpc-java`, `grpcurl`, or any gRPC library.
- **No special topic needed**: Any valid topic ID (or even with `checkTopicExists=false`) suffices.
- **Repeatable**: Cycling subscriptions every ~2 seconds re-arms the safety-check trigger.
- **Low cost to attacker**: A single commodity machine with a 1 Gbps link can sustain this indefinitely.

### Recommendation

1. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl` (e.g., reject new subscriptions when `subscriberCount` exceeds a configurable threshold).
2. **Add per-IP subscription rate limiting** at the gRPC interceptor layer.
3. **Bound the safety-check trigger**: Make the safety check a one-time, non-repeating operation (it already is a `Mono.delay`, but ensure it cannot be re-armed by subscription cycling).
4. **Increase `unthrottled.pollingFrequency`** minimum or reduce `maxPolls` default to limit burst magnitude.
5. **Isolate the gRPC DB user** to a separate connection pool with a hard cap, preventing it from starving the importer's pool.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 20
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    streams = []
    for _ in range(STREAMS_PER_CONN):
        req = consensus_service_pb2.ConsensusTopicQuery(
            topicID=...,   # any valid topic
            consensusStartTime=...,  # far in the past, no endTime, no limit
        )
        # open stream; safety check fires after 1s triggering unthrottled retrieval
        s = stub.subscribeTopic(req)
        streams.append(s)
    # hold open for burst window, then cycle
    import time; time.sleep(2)
    channel.close()

threads = [threading.Thread(target=open_subscriptions, args=(i,)) for i in range(CONNECTIONS)]
# Step 1: launch 20 connections × 5 streams = 100 concurrent subscriptions
for t in threads: t.start()
# Step 2: after ~1s, 100 safety checks fire simultaneously
# Step 3: each triggers retrieve(filter, false) → 13 polls at 20ms = 5000 QPS burst
# Step 4: cycle by restarting threads every 2s to sustain pressure
for t in threads: t.join()
```

**Expected result**: DB CPU/IO spikes to saturation; `statement_timeout` errors appear in importer and REST API logs; transaction confirmation queries begin failing.
