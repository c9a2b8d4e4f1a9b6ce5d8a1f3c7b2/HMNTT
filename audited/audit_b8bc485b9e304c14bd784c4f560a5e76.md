### Title
Liveness Probe Denial-of-Service via `/tmp/alive` Rename by Any User with Directory Write Access

### Summary
The liveness probe in `pinger/cmd/healthcheck/main.go` relies solely on `os.Stat("/tmp/alive")` to confirm the heartbeat file exists. Because Linux `rename(2)` requires only write permission on the *directory* (not the file), any process with write access to `/tmp` — regardless of whether it owns `/tmp/alive` — can atomically move the file away immediately before the probe fires, causing `os.Stat` to return an error and the probe to exit 1. After `failureThreshold: 5` consecutive failures Kubernetes restarts the container.

### Finding Description
**Exact code path:**

`pinger/cmd/healthcheck/main.go`, `main()`, lines 27–32: [1](#0-0) 

`pinger/main.go` writes the heartbeat file every 15 s with mode `0644`: [2](#0-1) 

The Kubernetes deployment mounts `/tmp` as an `emptyDir` volume: [3](#0-2) 

**Root cause:** The heartbeat file is written with `0644` (owner-writable), but the `/tmp` directory itself is world-writable (`0777` by default on Linux). The `rename(2)` / `os.Rename` syscall requires write permission on the *containing directory*, not on the file. Therefore any UID with access to the container's `/tmp` can rename `alive` to `alive.bak` without owning the file. The probe performs no ownership check, no HMAC/integrity check, and no inode-identity check — only `os.Stat` for existence and `fi.ModTime()` for staleness. [4](#0-3) 

**Why existing checks fail:** The `failureThreshold: 5` and `periodSeconds: 10` settings in `values.yaml` mean five consecutive probe failures (50 s window) trigger a restart. [5](#0-4) 

The `readOnlyRootFilesystem: true` and dropped capabilities only prevent writes to the root FS and privilege escalation; they do not restrict directory-level rename operations within the writable `emptyDir` mount. [6](#0-5) 

### Impact Explanation
An attacker who can execute code in the same pod (e.g., via a compromised sidecar, a second container sharing the `tmp` volume, or `kubectl exec` as any UID) can repeatedly rename `/tmp/alive` away before each probe invocation. After 5 failures the container is restarted, interrupting the pinger's Hedera transfer submissions. This is a griefing/availability impact with no economic damage to network users, matching the stated Medium scope.

### Likelihood Explanation
The precondition — a second process with `/tmp` write access in the same pod — is non-trivial in the default single-container deployment. However it becomes realistic when: (a) a sidecar (e.g., log shipper, service mesh proxy) is injected and shares the `emptyDir`; (b) an attacker achieves RCE in any container in the pod; (c) a cluster operator runs `kubectl exec` as a different UID. The attack is repeatable and requires no elevated OS privileges once the precondition is met.

### Recommendation
1. **Restrict `/tmp` directory permissions** to `0700` (or `0750`) so only UID 1000 can rename files within it. This can be done via an init container that `chmod`s the mount after creation.
2. **Use a named pipe or Unix socket** for the heartbeat signal instead of a plain file, eliminating the rename attack surface entirely.
3. **Verify file ownership** inside the healthcheck binary (`fi.Sys().(*syscall.Stat_t).Uid == expectedUID`) before trusting the file.
4. **Use `O_EXCL` + atomic replace with `os.Rename`** from a path only the writer controls, combined with a sticky-bit (`+t`) on `/tmp` so only the file owner can rename/delete it.

### Proof of Concept
```bash
# Precondition: shell in the same pod as any UID with /tmp write access
# Terminal 1 – attacker loop (rename file away just before each probe)
while true; do
  mv /tmp/alive /tmp/alive.bak 2>/dev/null
  sleep 9   # probe fires every 10 s; race window is wide
done

# Terminal 2 – observe Kubernetes restart counter increment after ~50 s
kubectl get pod <pinger-pod> -w
# RESTARTS column increments after failureThreshold (5) failures
```

The race window is generous: the heartbeat writer runs every 15 s, the probe every 10 s, and the rename is atomic (single syscall), so the attacker wins the race reliably by sleeping just under `periodSeconds`.
