### Title
Rate-Limiter Bypass via Large Invalid-Hex String Causes Unbounded Memory Allocation Per Request

### Summary
`jacksonCustomizer` sets `maxStringLength` to `ethereumMaxCallDataSize() * 2 + 1024` (≈ 263 KB with the default 128 KB call-data limit). Because `@RequestBody @Valid` deserialization and Bean Validation both complete before `throttleManager.throttle()` is ever invoked, an unauthenticated attacker can flood `/api/v1/contracts/call` with requests whose `data` field is exactly `maxSize − 1` valid hex characters followed by one invalid character. Jackson allocates the full ≈ 263 KB string, the `@Hex` regex validator scans and rejects it, and the rate-limiter bucket is never touched — all per request, with no application-level cap on the rate of such requests.

### Finding Description

**Code path:**

1. `JacksonConfiguration.java` line 22: `final int maxSize = jumboConfig.ethereumMaxCallDataSize() * 2 + 1024;` — with the Hedera default of 131,072 bytes this yields `maxSize = 263,168`.
2. Line 29: `.maxStringLength(maxSize)` — Jackson will fully materialise any string up to 263,168 characters into a heap `String` before returning control to Spring.
3. `ContractController.java` line 38: `call(@RequestBody @Valid ContractCallRequest request, …)` — Spring's `RequestResponseBodyMethodProcessor` runs Jackson deserialization **and** Bean Validation (`@Valid`) as part of argument resolution, before the method body is entered.
4. Line 40: `throttleManager.throttle(request)` — the rate-limiter is the **first statement inside the method body**. If `@Valid` throws `MethodArgumentNotValidException`, this line is never reached.
5. `HexValidator.java` line 22: `pattern = Pattern.compile("^(0x)?[0-9a-fA-F]{%d,%d}$".formatted(minLength, hex.maxLength()))` — the compiled regex is applied to the already-allocated ≈ 263 KB string; it scans the entire string before returning `false`.

**Root cause / failed assumption:** The design assumes the rate-limiter guards all incoming traffic. It does not: it only guards requests that pass Bean Validation. Invalid requests — including the crafted oversized-hex payload — are never counted against the bucket.

**Exploit flow:**
- Attacker sends `POST /api/v1/contracts/call` with body `{"data":"<262167 valid hex chars>Z","to":"0x…","from":"0x…","gas":21000}`.
- Jackson reads and allocates a 263,167-character `String` (≈ 263 KB on the heap).
- `HexValidator.isValid` runs the regex, scans all 263,167 characters, returns `false`.
- Spring returns HTTP 400; `throttleManager.throttle()` is never called.
- The attacker repeats at whatever rate the server's thread pool and network allow.

### Impact Explanation
Each malicious request forces ≈ 263 KB of heap allocation plus an O(n) regex scan of that allocation, with zero application-level rate limiting. With Tomcat's default 200-thread pool, a sustained flood of concurrent connections can generate ≈ 52 MB of live heap per thread-pool cycle, driving frequent young-generation GC pauses and, under sustained load, old-generation pressure or `OutOfMemoryError`. Because the rate-limiter bucket is never decremented, the attacker can maintain this indefinitely without triggering the 500 req/s guard. This can degrade or halt the web3 mirror-node service for all legitimate callers.

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly reachable. The payload is trivial to construct (any hex string of the right length with one non-hex character appended). The attack is fully repeatable and scriptable with standard HTTP tooling (curl, wrk, etc.). The only practical constraint is the attacker's own outbound bandwidth (≈ 263 KB per request body).

### Recommendation
1. **Move throttle check before `@Valid`**: Apply rate-limiting in a `HandlerInterceptor` or `OncePerRequestFilter` that fires before argument resolution, so every request — valid or not — is counted against the bucket.
2. **Reduce `maxStringLength`**: The `data` field carries hex-encoded call data; its maximum legitimate length is `ethereumMaxCallDataSize() * 2 + 2` (the `0x` prefix). Set `maxStringLength` to that tighter bound rather than adding an extra 1,024-byte buffer.
3. **Reject oversized bodies early**: Configure `server.tomcat.max-http-form-post-size` / `spring.servlet.multipart.max-request-size` or a `ContentCachingRequestWrapper` filter to reject bodies larger than the expected maximum before Jackson is invoked.

### Proof of Concept
```bash
# Build a payload: 263166 valid hex chars + one invalid char 'Z'
VALID=$(python3 -c "print('a' * 263166)")
PAYLOAD="{\"data\":\"${VALID}Z\",\"to\":\"0x00000000000000000000000000000000000004e4\",\"from\":\"0x00000000000000000000000000000000000004e2\",\"gas\":21000}"

# Flood the endpoint (no auth required)
for i in $(seq 1 1000); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "$PAYLOAD" &
done
wait
```
Each iteration: Jackson allocates ≈ 263 KB, regex scans it, returns HTTP 400, rate-limiter bucket untouched. Repeat indefinitely to exhaust heap and induce GC pressure.
