### Title
Unauthenticated 2x DB Query Amplification DoS via `/accounts/:idOrAliasOrEvmAddress/rewards` Endpoint

### Summary
The `listStakingRewardsByAccountId` handler unconditionally issues two sequential database queries per request when the path parameter is an EVM address or alias: one inside `getEncodedId()` (via `getEntityIdFromEvmAddress` or `getAccountIdFromAlias`) and a second inside `isValidAccount()`. Neither call is cached or rate-limited at the application layer, and the default REST API connection pool is only 10 connections. An unauthenticated attacker flooding the endpoint with unique valid EVM addresses can exhaust the pool and deny service to all other REST API consumers.

### Finding Description
**Exact code path:**

`rest/controllers/accountController.js` lines 170–175:
```js
listStakingRewardsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);  // DB query #1 for EVM/alias
  const isValidAccount = await EntityService.isValidAccount(accountId); // DB query #2, always
```

`rest/service/entityService.js` lines 60–63 (`isValidAccount`):
```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
```
`entityExistenceQuery` = `select type from entity where id = $1` — executes unconditionally, no caching.

`rest/service/entityService.js` lines 90–103 (`getEntityIdFromEvmAddress`):
```js
const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
```
Also executes unconditionally, no caching.

**Root cause:** `isValidAccount()` always issues a DB query regardless of input type, and for EVM-address or alias inputs `getEncodedId()` issues a second DB query first. There is zero application-level rate limiting in the Node.js REST service (`grep_search` for `rateLimit|throttle` in `rest/**/*.js` returns no matches). The `entityId.js` LRU cache only caches parsed `EntityId` objects (in-memory, no DB round-trip), not the results of existence checks.

**DB query count per request:**
- Plain numeric ID (`0.0.1001`): 1 query (only `isValidAccount`)
- EVM address (`0xebb9a1be...`): **2 queries** (`getEntityIdFromEvmAddress` + `isValidAccount`)
- Alias (`KGNABD5L...`): **2 queries** (`getAccountIdFromAlias` + `isValidAccount`)

**Connection pool:** Default `maxConnections = 10`, `statementTimeout = 20000 ms` (`docs/configuration.md` line 556). Each in-flight request holds a pool slot for the duration of both queries.

### Impact Explanation
With a pool of 10 connections and a 20-second statement timeout, an attacker sending ≥10 concurrent requests with unique EVM addresses saturates the pool. All subsequent requests from legitimate users receive connection-timeout errors (after up to 20 seconds of waiting). Because each mirror node instance runs its own Node.js process with its own pool, a single attacker targeting multiple instances in parallel can render 30%+ of a mirror node fleet unavailable. The REST API is the primary public data-access surface for the Hedera mirror node; its unavailability prevents dApps, wallets, and explorers from reading chain state.

### Likelihood Explanation
No authentication, API key, or credential is required. The endpoint is publicly reachable. The attacker needs only to generate unique 20-byte hex strings (valid EVM address format) — trivially scriptable. The attack is repeatable and stateless: each request is independent, so the attacker does not need to maintain sessions or track state. A single machine with a modest HTTP concurrency tool (e.g., `wrk`, `ab`, `hey`) is sufficient.

### Recommendation
1. **Add application-level rate limiting** to the Node.js REST service (e.g., `express-rate-limit`) scoped per IP, applied before any DB-touching handler.
2. **Cache `isValidAccount()` results** with a short TTL (e.g., 30–60 seconds) using the existing Redis layer (`hiero.mirror.rest.cache.response` or a dedicated in-process cache), keyed on the encoded account ID.
3. **Merge the two queries into one**: a single `SELECT type FROM entity WHERE id = $1` already serves both purposes — `getEncodedId()` for EVM addresses could be restructured to return the encoded ID and existence flag together, eliminating the redundant second query.
4. **Increase the default connection pool size** or configure pgBouncer's `max_user_client_connections` limit for `mirror_rest` to enforce back-pressure at the proxy layer.
5. **Enable Traefik `rateLimit` middleware** for the REST service chart (currently absent from the REST chart's middleware chain, unlike the Rosetta chart which explicitly configures it).

### Proof of Concept
```bash
# Generate 500 unique valid EVM addresses and flood the endpoint concurrently
python3 -c "
import os, sys
for _ in range(500):
    addr = '0x' + os.urandom(20).hex()
    print(f'GET /api/v1/accounts/{addr}/rewards HTTP/1.1')
    print('Host: <mirror-node-host>')
    print()
" | \
# Send with high concurrency using wrk or similar:
seq 1 500 | xargs -P 50 -I{} \
  curl -s -o /dev/null "https://<mirror-node-host>/api/v1/accounts/0x$(openssl rand -hex 20)/rewards"

# Expected result:
# - First ~10 concurrent requests consume all pool connections
# - Subsequent requests return 500 / connection timeout after 20s
# - Legitimate API consumers are denied service for the duration of the flood
```

**Verification:** Monitor `pg_stat_activity` on the database during the flood:
```sql
SELECT count(*), wait_event_type, state
FROM pg_stat_activity
WHERE usename = 'mirror_api'
GROUP BY wait_event_type, state;
```
All 10 pool slots will show `active` state, and new connection attempts will queue or fail.
