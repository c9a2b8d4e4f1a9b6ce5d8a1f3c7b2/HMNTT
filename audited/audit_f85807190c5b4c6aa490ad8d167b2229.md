### Title
Missing Historical EVM Version Map for TESTNET Causes All Historical Block Queries to Use Latest EVM Version

### Summary
`HederaNetwork.TESTNET` is hardcoded with `Collections.emptyNavigableMap()` for its `evmVersions` field. When no YAML override is configured, `getEvmVersions()` falls through all three tiers and returns `DEFAULT_EVM_VERSION_MAP = {0L → EVM_VERSION_0_67}`. As a result, `getEvmVersionForBlock(blockNumber)` returns `EVM_VERSION_0_67` for every historical block on TESTNET, regardless of which EVM version was actually active at that block height. Any unprivileged user can trigger this by issuing a historical `eth_call` or `eth_estimateGas` against any TESTNET block.

### Finding Description

**Exact code path:**

`HederaNetwork.TESTNET` is defined with an empty version map: [1](#0-0) 

The default `network` field is `HederaNetwork.TESTNET`: [2](#0-1) 

`getEvmVersions()` has a three-tier fallback: [3](#0-2) 

Because both `evmVersions` (YAML config, empty `TreeMap` by default) and `network.evmVersions` (empty for TESTNET) are empty, it always falls to: [4](#0-3) 

`DEFAULT_EVM_VERSION_MAP` maps only `0L → EVM_VERSION_0_67` (the latest). Then `getEvmVersionForBlock` calls `floorEntry(blockNumber)`: [5](#0-4) 

For any `blockNumber >= 0`, `floorEntry` returns `(0L, EVM_VERSION_0_67)`. This is consumed by `getSemanticEvmVersion()` for every historical call: [6](#0-5) 

And the `TransactionExecutorFactory` uses this version to select the executor: [7](#0-6) 

**Root cause:** The `HederaNetwork` enum provides a complete historical version map only for MAINNET. TESTNET, PREVIEWNET, and OTHER all receive `Collections.emptyNavigableMap()`, causing the fallback to `DEFAULT_EVM_VERSION_MAP` which unconditionally maps all blocks to the current latest version (`EVM_VERSION_0_67`).

**Failed assumption:** The code assumes that if no YAML config and no network-level map is present, using the latest EVM version is an acceptable default. This is incorrect for TESTNET, which has been running since before EVM 0_30 and has gone through the same upgrade sequence as MAINNET.

### Impact Explanation

Any historical `eth_call` or `eth_estimateGas` on TESTNET simulates using EVM 0_67 semantics regardless of the actual block height. Early TESTNET blocks (e.g., block 0 through the equivalent of MAINNET block ~87M) would have used EVM versions 0_30 through 0_66. Simulating them with 0_67 introduces incorrect opcode availability, incorrect gas cost schedules, and incorrect precompile behavior. This means the mirror node exports historically inaccurate contract execution records for all TESTNET historical queries — a direct violation of the stated correctness guarantee for the mirror node's historical data API.

### Likelihood Explanation

No privileges are required. Any user with access to the TESTNET mirror node's JSON-RPC endpoint can issue `eth_call` with a historical block tag (e.g., `"0x0"`) and trigger the incorrect EVM version selection. The condition is permanently present in the default configuration and requires no special timing or state. It is trivially repeatable.

### Recommendation

Add a historical EVM version map for TESTNET (and PREVIEWNET) in the `HederaNetwork` enum, mirroring the structure of `mainnetEvmVersionsMap()` with the actual block heights at which each EVM upgrade was activated on those networks. For example:

```java
TESTNET(unhex("01"), Bytes32.fromHexString("0x0128"), testnetEvmVersionsMap()),
```

with a `testnetEvmVersionsMap()` method populated with the correct TESTNET upgrade block boundaries. If the exact boundaries are not tracked, at minimum document this as a known limitation and consider returning an error or warning for historical queries on networks without a version map, rather than silently returning incorrect results.

### Proof of Concept

**Preconditions:** Mirror node deployed against TESTNET with default configuration (no `hiero.mirror.web3.evm.evmVersions` YAML override).

**Steps:**

1. Issue a historical `eth_call` to the mirror node's JSON-RPC endpoint targeting block 0:
   ```json
   {
     "jsonrpc": "2.0",
     "method": "eth_call",
     "params": [{"to": "<contract>", "data": "<calldata>"}, "0x0"],
     "id": 1
   }
   ```
2. Observe that `getSemanticEvmVersion()` → `getEvmVersionForBlock(0)` → `getEvmVersions().floorEntry(0)` returns `(0L, EVM_VERSION_0_67)` from `DEFAULT_EVM_VERSION_MAP`.
3. The `TransactionExecutorFactory` selects the EVM 0_67 executor.
4. The call is simulated with EVM 0_67 semantics, not the historically correct EVM 0_30 that was active at TESTNET block 0.
5. Repeat for any block number on TESTNET — the result is always EVM 0_67, regardless of actual historical version.
