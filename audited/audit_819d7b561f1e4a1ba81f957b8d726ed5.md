### Title
Unauthenticated `GET /contracts/results?block_number=` Triggers Four Sequential/Parallel DB Round-Trips Per Request, Enabling Resource Exhaustion

### Summary
The public `GET /contracts/results` endpoint, when called with a valid `block_number` parameter, unconditionally executes four database queries per single HTTP request: one to resolve the block's timestamp range, one to fetch contract results, and two more in parallel to enrich the response with Ethereum transaction and record-file data. No authentication, rate limiting, or caching exists on this path, allowing any unprivileged caller to amplify DB load by a factor of 4× per request.

### Finding Description

**Exact code path:**

`rest/routes/contractRoute.js` line 19 registers the route with no middleware guard:
```
router.getExt('/results', ContractController.getContractResults);
```

`rest/controllers/contractController.js` `getContractResults()` (lines 1050–1112):

**Step 1 – DB Query 1** (`extractContractResultsByIdQuery`, lines 486–501):
When `block_number` is supplied, `blockFilter.key === filterKeys.BLOCK_NUMBER` is true and the code unconditionally awaits:
```js
blockData = await RecordFileService.getRecordFileBlockDetailsFromIndex(blockFilter.value);
```
This is a synchronous DB round-trip against `record_file` (`recordFileBlockDetailsFromIndexQuery`, `rest/service/recordFileService.js` lines 52–56).

**Step 2 – `optimizeTimestampFilters` path** (lines 504–505):
Because `contractId === undefined` (this is the `/contracts/results` endpoint, not `/:contractId/results`), the code always calls:
```js
await optimizeTimestampFilters(timestampFilters, order)
```
The two gte/lte filters injected from `blockData` have `eqValues.length === 0`, so `bindTimestampRange` is called. When `queryConfig.bindTimestampRange` is `true` (a supported config), this may issue an additional DB query for `getFirstTransactionTimestamp` (`rest/timestampRange.js` lines 51–54) — but even with it `false`, the path still proceeds.

**Step 3 – DB Query 2** (line 1072):
```js
const rows = await ContractService.getContractResultsByIdAndFilters(conditions, params, order, limit);
```
Main contract_result table scan.

**Step 4 – DB Queries 3 & 4 in parallel** (lines 1083–1086):
If `rows.length > 0` (i.e., the block contains any contract results — normal on a live network):
```js
const [ethereumTransactionMap, recordFileMap] = await Promise.all([
  ContractService.getEthereumTransactionsByPayerAndTimestampArray(payers, timestamps),
  RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps),
]);
```
Two additional DB queries fire concurrently.

**Root cause / failed assumption:** The design assumes the four queries are cheap and that callers are well-behaved. There is no per-IP or per-endpoint rate limiting (confirmed: `grep -r "rateLimit|throttle"` in `rest/**/*.js` returns zero matches), no response caching, and no authentication on this route. The `block_number` parameter is the sole trigger; any valid integer that maps to a block containing at least one contract result is sufficient.

### Impact Explanation
Each request to `GET /contracts/results?block_number=<N>` where block N has contract results causes exactly 4 DB round-trips (potentially 5 with `bindTimestampRange=true`). A baseline request to most other read endpoints issues 1–2 queries. An attacker sending requests at even a modest rate (e.g., 50 req/s) generates 200+ DB queries/s from a single client. Because the `contract_result` and `record_file` tables are large and the timestamp-array query uses a correlated subquery (`recordFileBlockDetailsFromTimestampArrayQuery`, lines 23–43 of `recordFileService.js`), each query is non-trivial. This can sustain a ≥30% increase in DB CPU/IO compared to normal traffic without requiring brute-force volume, satisfying the stated threshold.

### Likelihood Explanation
The endpoint is fully public, documented in `rest/api/v1/openapi.yml` (lines 640–665), and requires only knowledge of any valid block number — trivially obtained by querying `/api/v1/blocks`. No credentials, tokens, or special network access are needed. The attack is repeatable, scriptable, and stateless. Any external party can execute it.

### Recommendation
1. **Rate limiting**: Apply per-IP request rate limiting on `/api/v1/contracts/results` (and similar multi-query endpoints) at the reverse-proxy or application middleware layer.
2. **Short-circuit caching**: Cache `getRecordFileBlockDetailsFromIndex` results by block number (block data is immutable once written); this eliminates DB Query 1 on repeated calls for the same block.
3. **Conditional enrichment**: Only execute the parallel DB Queries 3 & 4 when the caller explicitly requests enriched fields (e.g., via a query parameter), rather than unconditionally for every non-empty result set.
4. **Authentication/throttling tier**: Consider requiring an API key for endpoints that fan out to multiple DB queries.

### Proof of Concept
```bash
# 1. Obtain a valid block number containing contract results
BLOCK=$(curl -s "https://<mirror-node>/api/v1/blocks?limit=1&order=desc" \
  | jq '.blocks[0].number')

# 2. Single request triggers 4 DB queries
curl -s "https://<mirror-node>/api/v1/contracts/results?block.number=${BLOCK}"

# 3. Sustained amplification (no auth required)
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/contracts/results?block.number=${BLOCK}" &
done
wait
# Each iteration fires 4 DB queries; 1000 concurrent requests = up to 4000 DB queries
# from a single unauthenticated client, measurably increasing node DB resource consumption.
```
