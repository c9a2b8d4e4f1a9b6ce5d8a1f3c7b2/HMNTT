### Title
Integer Overflow in `poll()` Limit Calculation Causes Unbounded Database Query via `limit=Long.MAX_VALUE`

### Summary
In `PollingTopicMessageRetriever.poll()`, the remaining-limit calculation casts a `long` to `int` without overflow protection. An unprivileged user can submit a gRPC subscription with `limit = Long.MAX_VALUE`, causing the cast to produce `-1`, which then bypasses the `hasLimit()` guard in `TopicMessageRepositoryCustomImpl.findByFilter()`, resulting in a JPA query with no `setMaxResults` bound — returning every message in the topic per poll cycle.

### Finding Description

**Exact code path:**

`ConsensusController.toFilter()` directly maps the protobuf `uint64` limit field to `TopicMessageFilter.limit` with no upper-bound check: [1](#0-0) 

`TopicMessageFilter` only enforces `@Min(0)` — `Long.MAX_VALUE` satisfies this: [2](#0-1) 

In `poll()`, the remaining limit is computed with an unsafe narrowing cast: [3](#0-2) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getTotal().get() = 0`:
- `(int)(Long.MAX_VALUE - 0)` = `(int)(0x7FFFFFFFFFFFFFFF)` = lower 32 bits = `0xFFFFFFFF` = **`-1`**
- `Math.min(-1, maxPageSize)` = **`-1`** (maxPageSize is always positive)

The poisoned `pageSize = -1` is written into `newFilter`: [4](#0-3) 

In `TopicMessageRepositoryCustomImpl.findByFilter()`, `hasLimit()` returns `limit > 0`, which is `false` for `-1`, so `setMaxResults` is never called: [5](#0-4) 

The JPA query executes with no `LIMIT` clause, fetching **all** topic messages from the database.

**Why `isComplete()` doesn't save it:**

`isComplete()` checks `filter.getLimit() == total.get()`, i.e., `Long.MAX_VALUE == 0` — always false. For throttled mode, if the unbounded result set is ≥ `maxPageSize`, the subscription keeps polling indefinitely: [6](#0-5) 

**Why the outer `flux.take()` doesn't save it:**

`flux.take(Long.MAX_VALUE)` is effectively unlimited: [7](#0-6) 

### Impact Explanation
Each poll cycle issues a full table scan of `topic_message` for the given `topic_id` with no row limit. On a busy topic with millions of messages, this saturates database I/O, exhausts JVM heap (the result list is materialized in memory via `getResultList()`), and can degrade or crash the mirror node for all subscribers. The default timeout of 60 seconds means up to 60 seconds of unbounded queries per attacker connection, and multiple concurrent connections multiply the effect. [8](#0-7) 

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint requires no authentication. Any network-reachable client can send a `ConsensusTopicQuery` with `limit = 9223372036854775807` (Long.MAX_VALUE). The protobuf field is `uint64`, so this is a valid wire value. The attack is trivially scriptable, repeatable, and requires zero privileges.

### Recommendation
1. **Guard the cast**: Before casting, clamp the remaining limit to `Integer.MAX_VALUE`:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = filter.hasLimit()
       ? (int) Math.min(remaining, Integer.MAX_VALUE)
       : Integer.MAX_VALUE;
   ```
2. **Add an upper-bound constraint** on `TopicMessageFilter.limit`:
   ```java
   @Max(Long.MAX_VALUE / 2)   // or a domain-appropriate cap
   private long limit;
   ```
3. **Validate in the controller**: Reject or clamp `query.getLimit()` values that exceed a reasonable maximum before building the filter.

### Proof of Concept
```java
// Using the Hedera Java SDK or raw gRPC stub:
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setLimit(Long.MAX_VALUE)   // 9223372036854775807
    .build();

// Send to mirror node gRPC endpoint (no auth required).
// Result: poll() computes (int)(Long.MAX_VALUE - 0) = -1,
//         newFilter.limit = -1, hasLimit() = false,
//         findByFilter() issues SELECT ... FROM topic_message
//         WHERE topic_id = 1 ORDER BY consensus_timestamp
//         with NO LIMIT clause — full table scan returned.
```

Reproducible steps:
1. Start mirror node with a topic containing a large number of messages.
2. Connect to the gRPC port (default 5600) without credentials.
3. Send `subscribeTopic` with `limit = Long.MAX_VALUE`.
4. Observe: database CPU/IO spikes, JVM heap pressure, and the subscription never self-terminates until the 60-second timeout.
