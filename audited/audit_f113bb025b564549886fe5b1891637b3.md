Key facts established:
1. `cacheManagerRecordFileIndex` uses `maximumSize(10000)` and `expireAfterWrite(10, TimeUnit.MINUTES)` — hardcoded, not configurable.
2. `findByIndex` uses `unless = "#result == null"`. In Spring Cache with `Optional` return types, Spring unwraps the Optional, so `#result` is `null` when the Optional is empty. This means **empty Optionals (non-existent block numbers) are never cached** — every request for a non-existent block hits the DB.
3. The throttle is a **global** bucket4j bucket (500 RPS default), not per-IP.
4. No authentication is required for `/api/v1/contracts/call`.

### Title
Cache Non-Population for Non-Existent Block Numbers Enables Sustained DB I/O Amplification via Unauthenticated eth_call Requests

### Summary
The `recordFileIndex` cache in `cacheManagerRecordFileIndex()` is configured with `unless = "#result == null"` on `findByIndex()`, which causes Spring Cache to skip caching when the result is an empty `Optional` (non-existent block number). An unauthenticated attacker can repeatedly request eth_call with a non-existent block number, causing every request to bypass the cache and issue a live DB query, up to the global rate limit of 500 RPS. This creates a sustained, repeatable increase in DB I/O with no per-IP mitigation.

### Finding Description
**Exact code path:**

`ContractController.call()` → `ThrottleManager.throttle()` (global 500 RPS bucket) → `ContractCallService.callContract()` → lazy block supplier → `RecordFileServiceImpl.findByBlockType()` → `RecordFileRepository.findByIndex(nonExistentIndex)`.

**Root cause:**

In `RecordFileRepository.java` line 27–29:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
```
Spring Cache unwraps `Optional` for SpEL evaluation. When `findByIndex` returns `Optional.empty()` (block not found), `#result` evaluates to `null`, so `unless = "#result == null"` is `true` → the empty result is **never stored in the cache**. Every subsequent call with the same non-existent block number re-executes the DB query.

**Cache configuration** (`EvmConfiguration.java` lines 153–164):
```java
@Bean(CACHE_MANAGER_RECORD_FILE_INDEX)
@Primary
CacheManager cacheManagerRecordFileIndex() {
    final var caffeine = Caffeine.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .maximumSize(10000)
            .recordStats();
    ...
}
```
There is no negative-result caching (no null-value sentinel), so the cache provides zero protection against repeated lookups of non-existent indices.

**Exploit flow:**
1. Attacker sends POST `/api/v1/contracts/call` with `block: "9999999999"` (any non-existent block number).
2. Global rate limiter passes (500 RPS budget, no per-IP check).
3. `findByIndex(9999999999)` → cache miss → DB executes `SELECT r FROM RecordFile r WHERE r.index = 9999999999` → returns empty.
4. Empty Optional → `unless` condition true → result not cached.
5. `BlockNumberNotFoundException` thrown (request fails fast, low gas consumed → gas throttle not binding for small gas values due to `scaleGas` returning 0 for gas ≤ 10,000).
6. Attacker repeats at 500 RPS using the **same** block number — every single request hits the DB.

**Why checks fail:**
- The `unless = "#result == null"` guard was intended to avoid caching null, but it inadvertently prevents caching of legitimate "not found" responses.
- The global rate limiter (`rateLimitBucket`, 500 RPS) is not per-IP, so a single attacker can consume the full budget.
- The gas throttle is bypassed: `scaleGas(gas)` returns 0 for `gas ≤ 10,000`, so requests with minimal gas consume no gas tokens.

### Impact Explanation
At 500 RPS (the default global limit), an attacker generates 500 uncached DB queries per second against the `record_file` table. If baseline legitimate traffic is served mostly from cache (typical for a mirror node), this represents a near-unbounded relative increase in DB I/O. Even at a fraction of the rate limit (e.g., 100 RPS), the absolute number of additional DB queries easily exceeds a 30% increase over a 24-hour baseline. The DB query itself (`SELECT ... WHERE index = ?`) is indexed but still incurs round-trip latency, connection pool pressure, and I/O on every call. Sustained attack degrades response times for all users sharing the DB connection pool.

### Likelihood Explanation
The attack requires no credentials, no special knowledge beyond the public API, and no brute force. A single attacker with a script sending HTTP POST requests at the rate limit can sustain the attack indefinitely. The non-existent block number can be a fixed value (e.g., `Long.MAX_VALUE - 1`), requiring no enumeration. The attack is trivially repeatable and automatable.

### Recommendation
1. **Cache negative results**: Change `unless = "#result == null"` to `unless = "false"` or use a sentinel value to cache empty Optionals, preventing repeated DB queries for the same non-existent block. Alternatively, use `unless = "#result != null && !#result.isPresent()"` inverted — actually, cache empty results by removing the `unless` guard or using a short TTL for negative entries.
2. **Add per-IP rate limiting**: The current global bucket4j throttle does not prevent a single client from consuming the full RPS budget. Add per-source-IP rate limiting at the ingress/load-balancer layer or within `ThrottleManagerImpl`.
3. **Validate block range before DB query**: In `RecordFileServiceImpl.findByBlockType()`, check that the requested block number does not exceed the latest known block (cached via `findLatest()`) before calling `findByIndex()`, short-circuiting the DB query for obviously out-of-range values.

### Proof of Concept
```bash
# Send 500 requests/second with a non-existent block number
# Each request causes an uncached DB query

BLOCK="0x7FFFFFFFFFFFFFFF"  # Non-existent block number
URL="https://<mirror-node>/api/v1/contracts/call"
PAYLOAD='{"block":"'"$BLOCK"'","to":"0x0000000000000000000000000000000000000001","gas":21000,"data":"0x"}'

# Use ab or hey to send at rate limit
hey -n 10000 -c 50 -q 500 \
  -m POST \
  -H "Content-Type: application/json" \
  -d "$PAYLOAD" \
  "$URL"

# Observe DB query rate increase via pg_stat_activity or slow query log:
# SELECT r FROM RecordFile r WHERE r.index = 9223372036854775807
# will appear ~500 times/second with no cache hits
```

**Verification**: Monitor `pg_stat_activity` on the PostgreSQL instance. During the attack, the query `select ... from record_file ... where index = <nonexistent>` will appear at the full request rate with 0% cache hit ratio, confirming every request reaches the DB.
