### Title
Global Rate-Limit Bucket Exhaustion via Greedy Refill Token Starvation (DoS)

### Summary
The `rateLimitBucket` in `ThrottleManagerImpl` is a single application-wide Bucket4j instance with a greedy refill strategy and no per-client (IP/identity) isolation. Any unauthenticated external user can burst exactly `requestsPerSecond` (default: 500) requests at the start of each refill window, atomically draining all available tokens and causing every other legitimate caller to receive HTTP 429 for the remainder of that second. This is repeatable indefinitely.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.java` constructs the `rateLimitBucket` as a single Spring singleton bean: [1](#0-0) 

`refillGreedy(rateLimit, Duration.ofSeconds(1))` means all `rateLimit` tokens are added in one shot at the start of each 1-second window (as opposed to `refillIntervally` which spreads tokens over time). The bucket capacity equals the refill amount, so the bucket starts full and refills fully every second.

`ThrottleManagerImpl.throttle()` checks this single global bucket for every incoming request, with no per-IP or per-user partitioning: [2](#0-1) 

**Root cause / failed assumption:** The design assumes the 500 RPS budget will be shared fairly across all callers. There is no mechanism to enforce that assumption — a single caller can consume the entire budget. The `gasLimitBucket` at least uses `SynchronizationStrategy.SYNCHRONIZED`, but the `rateLimitBucket` has no per-client guard at all. [3](#0-2) 

**Existing checks reviewed:**
- The `request[]` filter system (`RequestProperties`, `ActionType.THROTTLE`) can apply per-pattern sub-limits, but these are opt-in, content-based, and not configured by default. [4](#0-3) 
- The Helm chart defines `maxRatePerEndpoint: 250` with `sessionAffinity: CLIENT_IP` at the GCP gateway layer, but this is infrastructure-optional, not enforced in application code, and limits per-pod endpoint rather than per-source-IP across the global token pool.

### Impact Explanation
A single unauthenticated attacker can render the entire web3 JSON-RPC API (`/api/v1/contracts/call`, etc.) unavailable to all other users for sustained periods. Every second the attacker re-drains the bucket, every legitimate request in that window receives HTTP 429 `"Requests per second rate limit exceeded"`. This is a complete service-availability denial for the web3 endpoint — contract calls, gas estimates, and opcode traces all route through `throttle()`. The impact is high availability loss with zero cost or privilege required from the attacker.

### Likelihood Explanation
Preconditions: none. The endpoint is publicly accessible and requires no authentication. Sending 500 HTTP requests per second is trivially achievable with `curl`, `ab`, `wrk`, or any scripted HTTP client from a single machine or a small botnet. The attack is perfectly repeatable every second and requires no special knowledge of the system beyond the public API URL. The greedy refill makes timing straightforward — the attacker does not need sub-millisecond precision; any burst within the 1-second window before legitimate traffic arrives suffices.

### Recommendation
1. **Per-IP rate limiting at the application layer**: Replace the single global bucket with a `ConcurrentHashMap<String, Bucket>` keyed on client IP (extracted from `X-Forwarded-For` or `RemoteAddr`), so each source IP gets its own token budget.
2. **Switch to `refillIntervally`**: Use `Bandwidth.builder().refillIntervally(rateLimit, Duration.ofSeconds(1))` instead of `refillGreedy`. This spreads token availability evenly across the window, preventing a single burst from consuming the entire second's budget at time zero.
3. **Enforce infrastructure-level per-IP limits unconditionally**: Make the GCP gateway `maxRatePerEndpoint` / `sessionAffinity: CLIENT_IP` a required deployment constraint, not an optional one.
4. **Add a per-IP hard cap** (e.g., 10–20 RPS per source IP) independent of the global budget, so even if the global bucket is large, no single IP can monopolize it.

### Proof of Concept
```bash
# Step 1: Identify the web3 endpoint (default port 8545 or via ingress)
TARGET="https://<mirror-node-host>/api/v1/contracts/call"

# Step 2: At the start of a second, fire requestsPerSecond (500) concurrent requests
# This exhausts all tokens in the global rateLimitBucket immediately
seq 1 500 | xargs -P500 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST "$TARGET" \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":50000}'

# Expected: first ~500 return 200, subsequent legitimate requests return 429

# Step 3: Verify legitimate user is throttled
curl -X POST "$TARGET" \
  -H "Content-Type: application/json" \
  -d '{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":50000}'
# Returns: HTTP 429 {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}

# Step 4: Repeat step 2 every second for sustained DoS
while true; do
  seq 1 500 | xargs -P500 -I{} curl -s -o /dev/null \
    -X POST "$TARGET" -H "Content-Type: application/json" \
    -d '{"data":"0x","to":"0x0000000000000000000000000000000000000167","gas":50000}'
  sleep 1
done
```
