### Title
Slow-Loris Body Delivery Inflates `hiero_mirror_rosetta_request_duration` Histogram, Enabling Metric Manipulation and False Alert Triggering

### Summary
The `MetricsMiddleware` in `rosetta/app/middleware/metrics.go` starts the request duration timer at handler invocation via `middleware.Instrument{}.Wrap(next)`, which includes the time spent reading the HTTP request body. Because Go's `net/http` reads the body lazily (inside the handler), an unprivileged attacker can deliberately trickle the POST body to `/construction/submit` over several seconds, artificially inflating their own duration observations into the highest histogram bucket (5 s) or beyond. No application-level body-read deadline is enforced before the metric is recorded, and the existing `ReadHeaderTimeout` only protects header delivery.

### Finding Description
**Code path:**
- `rosetta/app/middleware/metrics.go`, lines 76–83: `MetricsMiddleware` wraps every handler with `middleware.Instrument{Duration: requestDurationHistogram, …}.Wrap(next)`.
- `rosetta/main.go`, lines 220–227: the `http.Server` is configured with `ReadHeaderTimeout` and `ReadTimeout` sourced from `rosettaConfig.Http`; the default `application.yml` does not set a `ReadTimeout`, leaving it at `0` (no limit).
- `rosetta/app/config/types.go`, lines 64–69: `Http.ReadTimeout` is a plain `time.Duration` field with no enforced minimum.

**Root cause:** The weaveworks `Instrument.Wrap()` records `time.Now()` at the start of `ServeHTTP` and observes the elapsed duration after `next.ServeHTTP()` returns. The Rosetta SDK's JSON decoder reads `r.Body` inside `next.ServeHTTP()`. If the client delivers the body bytes slowly, the body-read latency is fully included in the observed duration. With `ReadTimeout = 0` (default), there is no server-side deadline to abort the slow read.

**Exploit flow:**
1. Attacker opens a TCP connection to the Rosetta server.
2. Sends HTTP headers for `POST /construction/submit` immediately (satisfying `ReadHeaderTimeout`).
3. Delivers the JSON body at ~1 byte/second, keeping the connection alive for 5–6 seconds.
4. The weaveworks `Instrument` middleware records a duration of ≥5 s and increments the `le="5"` and `+Inf` buckets of `hiero_mirror_rosetta_request_duration{route="/construction/submit"}`.
5. Repeated across multiple connections (Traefik allows 5 inflight per IP and 10 req/s per host), the attacker can continuously push observations into the highest bucket.

### Impact Explanation
- **False alert triggering:** The `RosettaSlowResponse` Prometheus rule fires when the rolling average latency exceeds 2000 ms for 1 minute. An attacker holding 5 connections open at ~2.1 s each can sustain this condition indefinitely, generating continuous critical-severity pages and causing alert fatigue.
- **Apdex score degradation:** The Grafana dashboard computes Apdex from the `le="100"` and `le="250"` buckets. Inflated high-bucket counts drive the score toward 0, making the service appear degraded to operators.
- **Histogram pollution / observability blind spot:** Legitimate slow requests (e.g., due to Hiero network congestion) become indistinguishable from attacker-induced slow observations, hampering incident response.

### Likelihood Explanation
No authentication or special privilege is required. The attacker needs only TCP access to the Rosetta port (typically exposed via Traefik ingress). The Traefik `inFlightReq` limit of 5 per IP and `rateLimit` of 10 req/host do not prevent slow body delivery — they only cap concurrency and request rate. A single attacker IP can sustain 5 concurrent slow-body connections indefinitely, which is sufficient to keep the `RosettaSlowResponse` alert firing. The attack is trivially scriptable with standard tools (`curl --limit-rate 1`, `slowhttptest`).

### Recommendation
1. **Set a `ReadTimeout`** in `application.yml` (e.g., `readTimeout: 10s`) so that the server aborts connections that do not complete body delivery within a reasonable window. This is the primary fix.
2. **Set a `MaxBytesReader` limit** inside the handler chain (e.g., `r.Body = http.MaxBytesReader(w, r.Body, maxBodyBytes)`) to reject oversized or stalled bodies early.
3. **Add a per-request body-read deadline** using `http.TimeoutHandler` or a context deadline applied before the metrics timer starts, so that aborted/timed-out requests are not recorded as legitimate slow observations.
4. Consider adding a Traefik `readTimeout` middleware to enforce body-read deadlines at the ingress layer as a defense-in-depth measure.
