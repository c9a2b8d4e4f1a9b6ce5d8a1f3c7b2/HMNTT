### Title
Thread Pool Exhaustion via Unauthenticated Concurrent Blocking Requests to NftAllowance Endpoint

### Summary
The `findAll` method in `NftAllowanceRepositoryCustomImpl` executes a synchronous, blocking jOOQ query that holds a Tomcat worker thread for the full duration of the database call (up to 20 seconds per the configured statement timeout). Unlike the `web3` module, the `rest-java` module has no rate limiting or concurrency controls on the `/api/v1/accounts/{id}/allowances/nfts` endpoint. An unauthenticated attacker can exhaust the Tomcat thread pool by sustaining a modest volume of concurrent requests, preventing the service from accepting new connections.

### Finding Description
**Exact code path:**

`NftAllowanceRepositoryCustomImpl.java`, lines 37–46:
```java
public Collection<NftAllowance> findAll(NftAllowanceRequest request, EntityId accountId) {
    ...
    return dslContext
            .selectFrom(NFT_ALLOWANCE)
            .where(condition)
            .orderBy(...)
            .limit(request.getLimit())
            .fetchInto(NftAllowance.class);  // synchronous blocking call
}
```

`AllowancesController.java`, lines 57–83: The `@GetMapping("/nfts")` handler calls `service.getNftAllowances(request)` with no rate limiting, no concurrency guard, and no authentication requirement.

**Root cause:** jOOQ's `fetchInto()` is a synchronous blocking JDBC call. The calling Tomcat thread is pinned for the entire duration of the database round-trip. Spring Boot's default Tomcat thread pool has 200 threads. The database-level `statement_timeout` for `mirror_rest_java` is set to 20,000 ms (20 seconds), meaning each thread can be held for up to 20 seconds per request.

**Why existing checks fail:**
- The `web3` module has `ThrottleManagerImpl` with bucket4j rate limiting (`requestsPerSecond = 500` default), but **no equivalent exists in `rest-java`**. `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter — no rate limiting filter.
- The HikariCP connection pool (default `maximumPoolSize = 10`) creates a secondary blocking point: once the pool is exhausted, threads additionally block waiting for a connection (up to HikariCP's default 30-second `connectionTimeout`), compounding the hold time.
- The DB-level `statement_timeout = 20000` is a partial mitigation but still allows 20-second thread holds, and it does not limit the number of concurrent requests.

### Impact Explanation
With 200 Tomcat threads and up to 20-second query hold times, an attacker needs to sustain only ~10 requests/second to saturate the thread pool. Once saturated, Tomcat queues and then rejects all new connections, effectively taking the `rest-java` service offline for all users. This is a complete denial of service against the REST Java API without affecting the database or other services directly.

### Likelihood Explanation
The endpoint requires no authentication and accepts any valid account ID. The attack requires no special knowledge, no credentials, and no exploit tooling — only a standard HTTP load generator (e.g., `ab`, `wrk`, `hey`). The attack is repeatable and sustainable. The absence of rate limiting in `rest-java` (while `web3` has it) indicates this protection gap was not intentional design but an oversight.

### Recommendation
1. Add a rate-limiting filter to `rest-java` equivalent to `web3`'s `ThrottleManagerImpl`, using bucket4j or Spring's `RateLimiter` support, applied globally or per-endpoint in `RestJavaConfiguration`.
2. Configure `server.tomcat.max-connections` and `server.tomcat.accept-count` to bound the connection queue.
3. Consider enabling Spring Boot virtual threads (`spring.threads.virtual.enabled=true`) to decouple thread count from concurrency, reducing the impact of blocking I/O.
4. Add a per-IP or global request concurrency limit at the ingress/load-balancer level as a defense-in-depth measure.

### Proof of Concept
```bash

# Expected result: after thread pool saturation (~200 in-flight),
# new requests receive connection refused or HTTP 503.
# Legitimate users are denied service for the duration of the attack.
```

```java
        var serviceResponse = service.getNftAllowances(request);
```
