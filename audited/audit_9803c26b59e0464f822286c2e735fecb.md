### Title
Silent Empty Result on Inverted Spender ID Range in `extractTokenMultiUnionQuery`

### Summary
An unprivileged external user can supply `spender.id=gte:X&spender.id=lte:Y` where X > Y to the `/accounts/{id}/allowances/tokens` endpoint. Because `validateBoundsRange()` only rejects the combination of a range filter with an equality filter — and never compares the lower bound value against the upper bound value — the inverted range passes all validation and is forwarded directly to PostgreSQL as `spender >= X AND spender <= Y`, an unsatisfiable predicate that silently returns zero rows with HTTP 200.

### Finding Description

**Code path:**

1. `getAccountTokenAllowances` (tokenAllowanceController.js:68–81) calls `utils.buildAndValidateFilters`, which validates individual filter syntax but performs no cross-filter value comparison. [1](#0-0) 

2. `extractTokenMultiUnionQuery` (tokenAllowanceController.js:22–60) parses `spender.id=gte:X` into `bounds.primary.lower` and `spender.id=lte:Y` into `bounds.primary.upper` via `Bound.parse()`. [2](#0-1) 

3. `Bound.parse()` stores each filter independently with no comparison between `lower.value` and `upper.value`. [3](#0-2) 

4. `validateBounds` calls `validateBoundsRange`, which only throws when a bound and an equal filter coexist on the same key. It never checks whether `lower.value > upper.value`. [4](#0-3) 

5. `getLowerFilters` — because no secondary bound is present — returns `[primary.lower, primary.upper]`, i.e., both the `>=X` and `<=Y` filters. [5](#0-4) 

6. `TokenAllowanceService.getSubQuery` maps each filter directly to a SQL condition `spender >= $3 AND spender <= $4` with X and Y as bound parameters. [6](#0-5) 

**Root cause:** `validateBoundsRange` is the only place where the combined state of `lower` and `upper` is examined, and it only guards against the range-plus-equal combination. The inverted-range case (`lower.value > upper.value`) is never tested anywhere in the validation chain.

### Impact Explanation
The resulting SQL predicate `spender >= X AND spender <= Y` (X > Y) is logically unsatisfiable. PostgreSQL evaluates it, finds no matching rows, and returns an empty result set. The API responds with HTTP 200 and `{"allowances": [], "links": {"next": null}}`. A caller — whether a human, a monitoring system, or an automated reconciliation tool — receives a response that is indistinguishable from "no allowances exist in this range," when in fact the range was never evaluated correctly. This constitutes silent data suppression of token allowance records for any spender range the attacker chooses to invert.

### Likelihood Explanation
No authentication or special privilege is required. The endpoint is public-facing (`/api/v1/accounts/{id}/allowances/tokens`). The exploit requires only knowledge of two valid entity IDs and the ability to send an HTTP GET request. It is trivially repeatable and requires no tooling beyond a browser or `curl`. Any entity ID pair where the numerically larger ID is supplied as `gte` and the smaller as `lte` triggers the condition.

### Recommendation
Add an explicit inverted-range check inside `validateBoundsRange` (or as a new `validateBoundsInversion` step called from `validateBounds`). After both `lower` and `upper` are set on a `Bound`, compare their numeric values and throw `InvalidArgumentError` if `lower.value > upper.value` (or `lower.value >= upper.value` for strict operators). Example guard to add in `baseController.js`:

```js
validateBoundsRange(bounds) {
  Object.keys(bounds).forEach((key) => {
    const b = bounds[key];
    if (b.hasBound() && b.hasEqual()) {
      throw new InvalidArgumentError(`Can't support both range and equal for ${b.filterKey}`);
    }
    // NEW: reject inverted ranges
    if (b.hasLower() && b.hasUpper() && b.lower.value > b.upper.value) {
      throw new InvalidArgumentError(`Lower bound must not exceed upper bound for ${b.filterKey}`);
    }
  });
}
```

### Proof of Concept
```
GET /api/v1/accounts/0.0.1001/allowances/tokens?spender.id=gte:9999&spender.id=lte:1000
```

**Preconditions:** The account `0.0.1001` has token allowances with spenders whose IDs fall between 1000 and 9999.

**Expected (correct) behavior:** HTTP 400 with an error indicating the range is invalid.

**Actual behavior:**
- HTTP 200
- `{"allowances": [], "links": {"next": null}}`
- PostgreSQL executes: `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 AND spender >= 9999 AND spender <= 1000 ORDER BY spender ASC, token_id ASC LIMIT $2`
- Zero rows returned; all existing allowances in the range are silently suppressed.
