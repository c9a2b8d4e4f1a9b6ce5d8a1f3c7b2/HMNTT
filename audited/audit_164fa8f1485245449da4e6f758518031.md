### Title
Unauthenticated DoS via Connection Pool Exhaustion on `GET /api/v1/network/nodes`

### Summary
The `GET /api/v1/network/nodes` endpoint in the `rest-java` module is publicly accessible with no rate limiting and executes a multi-table native SQL CTE query that holds a HikariCP database connection for the full query duration (up to the 10-second statement timeout). An unprivileged attacker can flood this endpoint with concurrent requests to exhaust the connection pool, causing all subsequent requests — including legitimate gossip-node discovery — to queue and time out.

### Finding Description

**Code path:**

`NetworkController.getNodes()` (line 152–171) accepts any unauthenticated `GET /api/v1/network/nodes` request and immediately delegates to `NetworkServiceImpl.getNetworkNodes()` (lines 100–137), which calls `NetworkNodeRepository.findNetworkNodes()` (lines 104–105). That method executes a native SQL CTE query (lines 25–103) that:

- Resolves the latest address book via `ORDER BY start_consensus_timestamp DESC LIMIT 1`
- Resolves the latest node stake via a full-table `SELECT MAX(consensus_timestamp) FROM node_stake`
- Joins `address_book_entry`, `node_stake`, and `node`
- Runs a correlated `jsonb_agg` subquery against `address_book_service_endpoint` **per result row**

Each invocation holds one HikariCP connection for the entire query duration.

**Root cause — failed assumption:** The code assumes infrastructure-level (load balancer / WAF) rate limiting will protect this endpoint. No application-level rate limiting exists in the `rest-java` module for `/api/v1/network/nodes`. The only guard is `getEffectiveLimit()` capping results at 25 rows (line 53–55 of `NetworkNodeRequest.java`), which limits result size but does nothing to limit concurrent DB connection consumption.

The `web3` module has a full `ThrottleManagerImpl` / `ThrottleConfiguration` with bucket4j rate limiting, but **no equivalent exists in `rest-java`**.

The `statementTimeout` is 10,000 ms (per `hiero.mirror.restJava.db.statementTimeout`). HikariCP's default pool size is 10 connections. With 10 concurrent requests each holding a connection for up to 10 seconds, the pool is fully saturated.

### Impact Explanation

When the HikariCP pool is exhausted, all new requests to any `rest-java` endpoint (not just `/network/nodes`) must wait for a connection. Requests waiting beyond HikariCP's `connectionTimeout` (default 30 s) receive a `500` error. Clients attempting to discover gossip nodes for transaction submission are denied service. The `RestJavaHighDBConnections` Grafana alert fires at 75% pool utilization, confirming this is a recognized operational risk — but there is no automated mitigation that rejects excess requests before pool exhaustion occurs.

### Likelihood Explanation

Preconditions: none — the endpoint requires no authentication, no API key, and no special headers. The attack is trivially scriptable with any HTTP client. It is repeatable indefinitely. The attacker does not need to know anything about the network topology; the endpoint is publicly documented in the OpenAPI spec (`/api/v1/network/nodes`). A single attacker with modest bandwidth can sustain the attack.

### Recommendation

1. **Add application-level rate limiting** to the `rest-java` module for `/api/v1/network/nodes`, mirroring the bucket4j `ThrottleConfiguration` already present in the `web3` module.
2. **Cache the query result** — the address book changes infrequently; a short-lived cache (e.g., 30–60 seconds) would serve the vast majority of requests without touching the database.
3. **Reduce `statementTimeout`** for this query to a lower value (e.g., 2–3 s) to limit per-request connection hold time.
4. **Increase HikariCP pool size** or configure a maximum pending queue depth with fast-fail behavior so pool exhaustion returns `429 Too Many Requests` rather than blocking all threads.

### Proof of Concept

```bash
# Exhaust the connection pool with 15 concurrent requests
# (exceeds default HikariCP pool of 10)
for i in $(seq 1 15); do
  curl -s "https://<mirror-node-host>/api/v1/network/nodes" &
done
wait

# Legitimate request now queues, waits connectionTimeout, then fails:
curl -v "https://<mirror-node-host>/api/v1/network/nodes"
# Expected: 500 Internal Server Error or connection timeout
# after HikariCP connectionTimeout (default 30s) is exceeded
```

    public List<NetworkNodeDto> getNetworkNodes(NetworkNodeRequest request) {
        final long fileId = getAddressBookFileId(request);
        final var limit = request.getEffectiveLimit();
        final var nodeIdParams = request.getNodeIds();
        final var orderDirection = request.getOrder().name();

        final Set<Long> nodeIds = new HashSet<>();
        long lowerBound = 0L;
        long upperBound = Long.MAX_VALUE;

        for (final var nodeIdParam : nodeIdParams) {
            if (nodeIdParam.operator() == RangeOperator.EQ) {
                nodeIds.add(nodeIdParam.value());
            } else if (nodeIdParam.hasLowerBound()) {
                lowerBound = Math.max(lowerBound, nodeIdParam.getInclusiveValue());
            } else if (nodeIdParam.hasUpperBound()) {
                upperBound = Math.min(upperBound, nodeIdParam.getInclusiveValue());
            }
        }

        if (lowerBound > upperBound) {
            throw new IllegalArgumentException("Invalid range provided for node.id");
        }

        final Long[] nodeIdArray;
        if (!nodeIds.isEmpty()) {
            final var range = Range.closed(lowerBound, upperBound);
            nodeIdArray = nodeIds.stream().filter(range::contains).toArray(Long[]::new);
            if (nodeIdArray.length == 0) {
                return List.of();
            }
        } else {
            nodeIdArray = EMPTY_NODE_ID_ARRAY;
        }

        return networkNodeRepository.findNetworkNodes(
                fileId, nodeIdArray, lowerBound, upperBound, orderDirection, limit);
    }

                limit 1

                       staking_period
                from node_stake
                where consensus_timestamp = (select max(consensus_timestamp) from node_stake)

            node_info as (
                select account_id, admin_key, associated_registered_nodes, decline_reward, grpc_proxy_endpoint, node_id
                from node

    public static final int DEFAULT_LIMIT = 10;
    public static final int MAX_LIMIT = 25;

    @RestJavaQueryParam(name = FILE_ID, required = false)
    private EntityIdRangeParameter fileId;

    @RestJavaQueryParam(name = NODE_ID, required = false)
    @Builder.Default
    private List<NumberRangeParameter> nodeIds = List.of();

    @RestJavaQueryParam(name = LIMIT, required = false)
    @Builder.Default
    @Min(1)
    private int limit = DEFAULT_LIMIT;

    @RestJavaQueryParam(name = ORDER, required = false)
    @Builder.Default
    private Direction order = Direction.ASC;

    /**
     * Gets the effective limit, capped at MAX_LIMIT. Matches rest module behavior where limit is capped at 25 for
     * network nodes.
     */
    public int getEffectiveLimit() {
        return Math.min(limit, MAX_LIMIT);
    }
