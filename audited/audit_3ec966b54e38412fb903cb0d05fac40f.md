### Title
Unauthenticated 4-Query DB Chain per Request in `getContractResults()` Enables Connection Pool Exhaustion DoS

### Summary
`GET /api/v1/contracts/results?from=<evm_address>` is accessible without authentication and triggers a sequential 4-database-query chain per request: an entity EVM-address lookup, a contract results query, and two parallel follow-up queries (ethereum transactions + record files). With the default connection pool capped at 10 connections and no rate limiting on the REST API, a modest flood of concurrent requests exhausts the pool, causing the mirror node REST API to become unavailable. There is no caching for the EVM address resolution step.

### Finding Description

**Exact code path:**

1. `getContractResults()` — `rest/controllers/contractController.js` line 1050 — calls `extractContractResultsByIdQuery(filters)` at line 1067.

2. Inside `extractContractResultsByIdQuery()` (line 415), for each `filterKeys.FROM` filter whose value passes `EntityId.isValidEvmAddress()`, line 444 unconditionally calls:
   ```js
   filter.value = await EntityService.getEncodedId(filter.value);
   ```
   `getEncodedId()` (`rest/service/entityService.js` line 118) delegates to `getEntityIdFromEvmAddress()` (line 90), which executes `entityFromEvmAddressQuery` — a live `SELECT id FROM entity WHERE evm_address = $1` — against the database. **DB Query #1.**

3. Back in `getContractResults()` line 1072:
   ```js
   const rows = await ContractService.getContractResultsByIdAndFilters(conditions, params, order, limit);
   ```
   **DB Query #2.**

4. Lines 1083–1086:
   ```js
   const [ethereumTransactionMap, recordFileMap] = await Promise.all([
     ContractService.getEthereumTransactionsByPayerAndTimestampArray(payers, timestamps),
     RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps),
   ]);
   ```
   **DB Queries #3 and #4** (concurrent but both consume pool connections).

**Root cause — failed assumptions:**

- *No rate limiting*: `grep` across `rest/**/*.js` for `rateLimit|throttle` returns only a test-utility match. The `ThrottleConfiguration` / `ThrottleManagerImpl` exist only in the `web3` (Java) module, not the Node.js REST API. The `requestHandler.js` middleware performs no throttling.
- *No caching*: `EntityService` in the REST module (`rest/service/entityService.js`) has no memoization or cache layer around `getEntityIdFromEvmAddress()`. Every request with a non-long-zero EVM address hits the database.
- *Tiny default pool*: `rest/dbpool.js` line 14 sets `max: config.db.pool.maxConnections`; the documented default is **10** (`hiero.mirror.rest.db.pool.maxConnections | 10`).
- *Long statement timeout*: default `statementTimeout` is **20 000 ms**, so each connection can be held for up to 20 seconds.

### Impact Explanation

With 10 pool connections and 4 queries per request held for up to 20 s each, only ~2–3 concurrent attacker requests are needed to saturate the pool. Subsequent legitimate requests queue until `connectionTimeoutMillis` (default 20 000 ms) expires and then fail. The mirror node REST API becomes entirely unavailable. The importer uses a separate DB user (`mirror_importer`) with its own pool, so confirmed-transaction ingestion is not directly blocked; the impact is full REST API denial-of-service rather than consensus-layer shutdown.

### Likelihood Explanation

No authentication, no API key, no rate limit, and no CAPTCHA protect this endpoint. The `from=` parameter accepts any 20-byte hex string (e.g. `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`). A single attacker with a standard HTTP client can issue hundreds of concurrent requests per second. The attack is trivially repeatable and requires zero privileges.

### Recommendation

1. **Add rate limiting** to the Node.js REST API (e.g. `express-rate-limit`) scoped per IP, targeting endpoints that trigger DB lookups.
2. **Cache EVM-address → encoded-id resolutions** in `EntityService.getEncodedId()` using an in-process LRU cache (the documentation already defines `hiero.mirror.rest.cache.entityId.*` properties for the Java REST module — apply the same pattern here).
3. **Increase the default pool size** or configure PgBouncer `max_user_client_connections` limits for `mirror_rest` to prevent starvation.
4. **Validate and short-circuit early**: if the EVM address resolves to `null` (entity not found), return 404 before executing the remaining three queries.

### Proof of Concept

```bash
# Flood with 50 concurrent requests using a non-existent EVM address
# (still triggers the entity lookup query before returning 404)
seq 1 50 | xargs -P50 -I{} curl -s \
  "https://<mirror-node-host>/api/v1/contracts/results?from=0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef" \
  -o /dev/null -w "%{http_code}\n"
```

After pool exhaustion, legitimate requests begin returning `500` / connection-timeout errors. The four-query chain is confirmed by the code path:

- `extractContractResultsByIdQuery` line 444 → `EntityService.getEncodedId` → `entityFromEvmAddressQuery` [1](#0-0) 
- `getContractResultsByIdAndFilters` call [2](#0-1) 
- `Promise.all` for ethereum transactions + record files [3](#0-2) 
- No rate-limiting middleware in the REST API [4](#0-3) 
- Default pool size of 10 connections [5](#0-4) 
- No cache in `EntityService.getEncodedId` [6](#0-5)

```javascript
import httpContext from 'express-http-context';
import qs from 'qs';

import config from '../config';
import {httpStatusCodes, requestIdLabel, requestStartTime} from '../constants';
import {lowerCaseQueryValue, randomString} from '../utils';

const queryCanonicalizationMap = {
  order: lowerCaseQueryValue,
  result: lowerCaseQueryValue,
};

const queryOptions = {
  arrayLimit: config.query.maxRepeatedQueryParameters,
  depth: 1,
  strictDepth: true,
  throwOnLimitExceeded: true,
};

const requestLogger = async (req, res) => {
  const requestId = await randomString(8);
  httpContext.set(requestIdLabel, requestId);

  // set default http OK code for reference
  res.locals.statusCode = httpStatusCodes.OK.code;
  res.locals[requestStartTime] = Date.now();
};

/**
 * Manage request query params to support case insensitive keys and some values (queryCanonicalizationMap above).
 * Express default query parser uses qs, other option is querystring, both are case sensitive
 * Parse using default qs logic and use to populate a new map in which all keys are lowercased
 * @param queryString
 * @returns Query string map object
 */
const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);
    if (lowerKey in caseInsensitiveQueryString) {
      // handle repeated values, merge into an array
      caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;
};

const canonicalizeValue = (key, value) => {
  const canonicalizationFunc = queryCanonicalizationMap[key];
  if (canonicalizationFunc === undefined) {
    return value;
  }

  return Array.isArray(value) ? value.map((v) => canonicalizationFunc(v)) : canonicalizationFunc(value);
};

export {requestLogger, requestQueryParser};
```
