### Title
Unauthenticated Multi-Query Chain in `getContractLogs` Enables Connection Pool Exhaustion DoS

### Summary
The `GET /api/v1/contracts/results/logs?transaction_hash=<hash>&topic0=<val>` endpoint is publicly accessible with no authentication and executes three sequential database queries per request: one in `getTransactionHash()`, one in `ContractService.getContractLogs()`, and one in `RecordFileService.getRecordFileBlockDetailsFromTimestampArray()`. Under concurrent load from an unauthenticated attacker, this multi-query chain exhausts the finite shared connection pool, denying service to all other API consumers.

### Finding Description

**Code path:**

1. **Route** — `rest/routes/contractRoute.js` line 20: `router.getExt('/results/logs', ContractController.getContractLogs)` — no auth middleware, publicly reachable. [1](#0-0) 

2. **Handler** — `rest/controllers/contractController.js` lines 820–848: `getContractLogs` calls `extractContractLogsMultiUnionQuery(filters)` then `ContractService.getContractLogs(query)`. [2](#0-1) 

3. **DB Query 1** — `extractContractLogsMultiUnionQuery` lines 669–677: when `transaction_hash` is present, calls `getTransactionHash(transactionHash, ...)` which issues `pool.queryQuietly(query, params)` against the `get_transaction_info_by_hash` stored procedure. [3](#0-2) [4](#0-3) 

4. **DB Query 2** — `ContractService.getContractLogs` line 378: `super.getRows(sqlQuery, params)` — the timestamp-bounded, topic-filtered log query. [5](#0-4) 

5. **DB Query 3** — `ContractService.getContractLogs` line 390: `RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps)` — a third sequential DB call on the same shared pool. [6](#0-5) 

**Root cause:** The three queries are sequential and each independently acquires a connection from the shared `pg` pool. The pool is bounded by `config.db.pool.maxConnections`. [7](#0-6) 

**Why `checkTimestampsForTopics` does not block this:** The check operates on raw request filters. When `transaction_hash` is supplied, the timestamp is resolved dynamically from DB Query 1 — it is never present in the original filter set. The topic check therefore cannot enforce the timestamp requirement for this code path, allowing `?transaction_hash=<hash>&topic0=<val>` (no explicit timestamp) to proceed through all three queries. [8](#0-7) 

### Impact Explanation
With a default pool of (typically) 10–20 connections and three sequential connection acquisitions per request, an attacker sending ~7–10 concurrent requests per pool slot can saturate the pool. All other API endpoints sharing the same pool (`/api/v1/transactions`, `/api/v1/accounts`, etc.) begin queuing or timing out, producing a full-service DoS. The `connectionTimeoutMillis` setting causes queued requests to fail with errors rather than wait indefinitely, surfacing the outage to legitimate users. [9](#0-8) 

### Likelihood Explanation
No credentials, API keys, or special network position are required. A valid 32-byte Ethereum transaction hash (obtainable from any public block explorer or by querying `/api/v1/contracts/results`) is the only precondition. The attack is trivially scriptable with `curl` or any HTTP load tool. Global rate limiting exists in `config.js` but operates per-IP; an attacker with even a small number of source IPs (or using distributed infrastructure) can exceed the threshold needed to exhaust the pool. The attack is repeatable and stateless. [10](#0-9) 

### Recommendation
1. **Enforce a per-request concurrency limit** (e.g., `p-limit` or a semaphore) on the `/contracts/results/logs` handler so that at most N requests execute DB work simultaneously, where N < `maxConnections`.
2. **Merge the hash-to-timestamp resolution into the log query** using a CTE or subquery, reducing three round-trips to one.
3. **Apply a tighter per-endpoint rate limit** on `/contracts/results/logs` (and specifically on requests carrying `transaction_hash`) independent of the global rate limiter.
4. **Set a short `statement_timeout`** specifically for the `get_transaction_info_by_hash` call to bound the time each connection is held during Query 1.
5. **Require an explicit timestamp range** even when `transaction_hash` is present, so `checkTimestampsForTopics` can enforce the seven-day window constraint and reject requests that would produce unbounded scans.

### Proof of Concept
```bash
# Step 1: obtain a valid transaction hash from the public API
HASH=$(curl -s 'https://<mirror-node>/api/v1/contracts/results?limit=1' \
  | jq -r '.results[0].hash')

# Step 2: flood the endpoint with concurrent requests (no auth needed)
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/contracts/results/logs?\
transaction_hash=${HASH}&topic0=0xddf252ad1be2c89b69c2b068fc378daa952ba7f163c4a11628f55a4df523b3ef" \
    -o /dev/null &
done
wait

# Expected result: legitimate requests to /api/v1/transactions or /api/v1/accounts
# begin returning 503 / connection timeout errors as the shared pool is exhausted.
```

  async getContractLogs(query) {
    const [sqlQuery, params] = this.getContractLogsQuery(query);
    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return rows;
    }

    const timestamps = [];
    // The timestamps are ordered, and may have duplicates, dedup them
    rows.forEach((row) => {
      if (row.consensus_timestamp !== timestamps[timestamps.length - 1]) {
        timestamps.push(row.consensus_timestamp);
      }
    });
    const recordFileMap = await RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps);

    return rows.map((cr) => new ContractLog(cr, recordFileMap.get(cr.consensus_timestamp)));
  }
