### Title
Per-Filter THROTTLE Bucket Starvation via Shared Global Token Bucket (No Per-User Isolation)

### Summary
The `THROTTLE` action in `ThrottleManagerImpl.action()` consumes tokens from a single shared per-filter `Bucket` (capacity = `rate`, max 100) with no per-IP or per-user partitioning. Because the global `rateLimitBucket` defaults to 500 req/s while the per-filter bucket caps at 100 tokens/s, an unprivileged attacker can continuously exhaust the per-filter bucket while remaining well within the global limit, causing all other users whose requests match that filter to receive `ThrottleException` for the duration of the attack.

### Finding Description
**Code path:**

- `ThrottleManagerImpl.java` lines 44–48: iterates `throttleProperties.getRequest()` and calls `action()` for every matching filter.
- `ThrottleManagerImpl.java` lines 70–73: `THROTTLE` branch calls `filter.getBucket().tryConsume(1)` on the shared per-filter bucket.
- `RequestProperties.java` lines 63–68: `createBucket()` builds a single `Bucket` with `capacity=rate` and `refillGreedy(rate, Duration.ofSeconds(1))`. This instance is stored as a lazy singleton on the `RequestProperties` object — one bucket shared by every thread and every caller.
- `RequestProperties.java` lines 46–51: for `ActionType.THROTTLE` the random-sampling guard (`action != ActionType.THROTTLE`) is explicitly skipped, so **every** matching request hits the bucket.

**Root cause:** The per-filter bucket is a process-wide singleton with no per-source-IP or per-session dimension. The failed assumption is that the configured `rate` (≤ 100) is low enough that no single client can monopolise it — but because the global `rateLimitBucket` allows 500 req/s (default, `ThrottleProperties.java` line 35) and the per-filter bucket allows only 100 req/s, an attacker can saturate the per-filter bucket while consuming only 20 % of the global allowance.

**Exploit flow:**
1. Attacker identifies (or guesses) a `RequestFilter` criterion — e.g., a specific `to` address, `data` prefix, or gas range — that triggers a `THROTTLE`-action filter with `rate=100`.
2. Attacker sends exactly 100 crafted matching requests per second. Each request passes the global `rateLimitBucket.tryConsume(1)` check (500-token bucket, only 100 consumed) and the `gasLimitBucket` check.
3. All 100 per-filter tokens are consumed each second. The bucket refills at 100/s, but the attacker immediately re-exhausts it.
4. Any legitimate user whose request matches the same filter hits `filter.getBucket().tryConsume(1) == false` and receives `ThrottleException("Requests per second rate limit exceeded")`.

**Why existing checks are insufficient:**
- The global `rateLimitBucket` (500/s) is checked before the per-filter bucket; it does not prevent a single client from consuming the entire per-filter quota.
- There is no IP-level, session-level, or token-based identity check anywhere in the throttle path.
- The `limit` counter (`counter.getAndIncrement() >= limit`) defaults to `Long.MAX_VALUE`, providing no practical ceiling.

### Impact Explanation
An attacker can selectively deny service to all legitimate users whose requests match a configured `THROTTLE` filter, while their own requests also get throttled only after the bucket is empty. Because the attacker controls the timing (send exactly `rate` req/s to keep the bucket at zero), the DoS is continuous and targeted. Operators who deploy `THROTTLE` filters to protect high-value or expensive call patterns (e.g., large-gas contract calls to a specific address) are most exposed: the very filter meant to protect them becomes the denial vector.

### Likelihood Explanation
No authentication or special privilege is required. The attacker needs only to discover or enumerate a matching filter criterion, which may be inferred from public API behaviour (e.g., observing which request shapes receive throttle errors vs. normal errors). Sending 100 HTTP requests per second is trivially achievable from a single host. The attack is repeatable indefinitely and requires no state beyond knowing the filter pattern.

### Recommendation
1. **Per-source-IP sub-buckets:** Replace the single shared `Bucket` with a `ConcurrentHashMap<String, Bucket>` keyed on client IP (extracted from the servlet request or a forwarded header), so each IP gets its own `rate`-token allocation.
2. **Alternatively, use Bucket4j's `ProxyManager`** with a key derived from the client identity to achieve the same effect with less boilerplate.
3. **Reduce the asymmetry:** Ensure the global `requestsPerSecond` limit is not significantly larger than the smallest per-filter `rate`; this limits how much of the global budget a single client can redirect into a per-filter bucket.
4. **Add a per-IP global pre-check** before the per-filter loop so that no single IP can consume a disproportionate share of any bucket.

  wait
  sleep 1
