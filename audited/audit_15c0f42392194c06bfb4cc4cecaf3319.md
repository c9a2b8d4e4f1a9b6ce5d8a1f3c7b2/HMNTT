### Title
Unbounded Throttled Polling Loop via Full-Page Response — Scheduler Resource Exhaustion DoS

### Summary
In `PollingTopicMessageRetriever.PollingContext.isComplete()`, the throttled termination condition `pageSize.get() < retrieverProperties.getMaxPageSize()` is never satisfied when the database returns exactly `maxPageSize` (default 1000) messages per poll. Combined with `numRepeats = Long.MAX_VALUE` and a per-element timeout that resets on every received message, an unprivileged subscriber targeting a topic with a large historical backlog (or one receiving continuous new messages) can hold a polling loop open for an arbitrarily long time, consuming `boundedElastic` scheduler resources across many concurrent connections.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java`, `PollingContext.isComplete()`, lines 121–129:

```java
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();

    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }

    return limitHit;
}
```

`pageSize` is reset to `0` at the start of every poll (line 73: `context.getPageSize().set(0L)`) and incremented once per received message (line 134: `pageSize.incrementAndGet()`). The condition `pageSize.get() < maxPageSize` is `false` whenever the DB returns a full page of exactly 1000 messages, so `isComplete()` returns `false` and the repeat loop continues.

**Repeat loop configuration (lines 51–55):**

```java
return Flux.defer(() -> poll(context))
        .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                ...
```

When `throttled=true`, `numRepeats = Long.MAX_VALUE` (line 99) — there is no cap on iterations.

**Timeout (line 59):**

```java
.timeout(retrieverProperties.getTimeout(), scheduler)
```

`retrieverProperties.getTimeout()` defaults to 60 seconds (`RetrieverProperties.java`, line 28). In Reactor, `Flux.timeout(Duration)` fires only when no item arrives within the window. When the DB returns 1000 messages per poll every 2 seconds (`pollingFrequency`), items arrive continuously and the timeout never fires.

**`startTime` advancement (line 72):**

```java
var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
```

Each poll advances the window forward. If new messages are being written to the topic at ≥1000 per 2-second polling interval, every poll returns a full page and the loop never terminates. Even with a static backlog of N messages, the loop runs for `(N / 1000) × 2` seconds — for 10 million messages, ~5.5 hours per subscription.

**Call site (`TopicMessageServiceImpl.java`, line 63):**

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
```

Every `subscribeTopic()` call from an unauthenticated gRPC client triggers this path with `throttled=true`.

### Impact Explanation

Each long-running subscription holds a slot on the shared `Schedulers.boundedElastic()` instance (line 41: `scheduler = Schedulers.boundedElastic()`). Bounded elastic schedulers have a finite thread/task ceiling. An attacker opening N concurrent subscriptions to a high-volume topic can saturate the scheduler, causing all subsequent retrieval operations — including legitimate ones — to queue indefinitely or be rejected. This constitutes a denial-of-service against the gRPC mirror node's historical message retrieval subsystem. Because the retriever is also used in the safety-check path (`missingMessages`, line 149), scheduler exhaustion can cascade to gap-recovery for live subscribers as well.

### Likelihood Explanation

**Precondition 1 — large static backlog:** Any topic that has accumulated more than ~1000 messages (trivially true for any active topic) causes the loop to run for multiple polling cycles. A topic with millions of messages (common on mainnet) causes hours-long loops per subscription. No attacker action beyond subscribing is required.

**Precondition 2 — continuous full-page returns:** An attacker who also controls message submission can sustain the loop indefinitely by writing ≥500 messages/second to the topic. On Hedera this incurs transaction fees, raising the cost of attack, but does not prevent it.

**No authentication required:** The gRPC `subscribeTopic` endpoint is publicly accessible. An attacker needs only a valid topic ID (publicly visible on-chain) and a gRPC client. Opening tens of concurrent subscriptions to a high-message-count topic is trivially scriptable.

### Recommendation

1. **Add a hard wall-clock deadline independent of message flow.** Replace or supplement `.timeout(retrieverProperties.getTimeout(), scheduler)` with a `Flux.take(Duration)` or a `takeUntil` based on `Stopwatch` elapsed time so the loop terminates regardless of whether items are flowing.

2. **Cap throttled repeats.** Set `numRepeats` to a finite value even in throttled mode (e.g., derived from `timeout / pollingFrequency`), mirroring the unthrottled path's `maxPolls` guard.

3. **Enforce a per-subscriber connection limit** at the gRPC layer to bound the number of concurrent long-running subscriptions from a single client.

4. **Alternatively, treat `pageSize == maxPageSize` as a signal to do one more poll, then terminate** rather than looping indefinitely — the current `equalPageSize` test already validates this one-extra-poll behavior, so the fix would be to count consecutive full-page polls and cap them.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
