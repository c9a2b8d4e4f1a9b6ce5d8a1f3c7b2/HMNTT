### Title
Silent `minTimestamp` Forward-Shift in `getSupply()` Causes Incomplete Historical Supply Data

### Summary
When an unprivileged user queries `/api/v1/network/supply` with only an upper-bound timestamp (e.g., `?timestamp=lte:X`), `Bound.getAdjustedLowerRangeValue()` returns `0` because `lower == null`. The service then unconditionally computes `optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1)` and applies `minTimestamp = Math.max(0, optimalLowerBound)`, silently narrowing the search window to at most ~one month before the upper bound. Any balance records older than that window are excluded from the supply calculation, producing incorrect or missing supply data without any error or warning to the caller.

### Finding Description

**Exact code path:**

`NetworkController.getSupply()` (line 133–137) accepts a `timestamp` query parameter and constructs a `Bound` via `Bound.of()`. When only an upper bound is supplied (e.g., `lte:X`), the `Bound` constructor sets `lower = null` and `upper = param`.

`Bound.isEmpty()` (line 126–128) returns `false` because `upper != null`, so execution enters the `else` branch of `NetworkServiceImpl.getSupply()`.

`Bound.getAdjustedLowerRangeValue()` (line 86–97):
```java

    if (this.lower == null) {
        return 0;          // ← returns 0 when no lower bound
    }
    ...
}
```

Back in `NetworkServiceImpl.getSupply()` (line 69–81):
```java
var minTimestamp = timestamp.getAdjustedLowerRangeValue();   // = 0
final var maxTimestamp = timestamp.adjustUpperBound();        // = user's upper bound

if (minTimestamp > maxTimestamp) { ... }                     // 0 > X → false, no exception

final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1); // first day of prev month
minTimestamp = Math.max(minTimestamp, optimalLowerBound);    // = optimalLowerBound (always > 0)

accountBalanceRepository.getSupplyHistory(..., minTimestamp, maxTimestamp);
// Query window: [optimalLowerBound, maxTimestamp] — NOT [0, maxTimestamp]
```

The only guard (`minTimestamp > maxTimestamp`) does not catch this case because `0 < maxTimestamp` is always true for any valid timestamp. The `Math.max` silently replaces the user's effective lower bound (0 = beginning of time) with `optimalLowerBound`, which is the first nanosecond of the month preceding `maxTimestamp`.

The same override also applies when the user explicitly provides a lower bound earlier than `optimalLowerBound` (e.g., `?timestamp=gte:2020-01-01&timestamp=lte:2023-12-31`): the explicit lower bound is silently discarded in favor of `optimalLowerBound`.

### Impact Explanation

The `account_balance` table stores periodic snapshots (not every-block updates). If the most recent snapshot for an unreleased-supply account falls before `optimalLowerBound` (i.e., more than ~one month before the queried upper bound), `getSupplyHistory` returns `consensusTimestamp = 0`, triggering an `EntityNotFoundException("Network supply not found")` — a false 404 — even though valid data exists in the database. Alternatively, if some but not all accounts have snapshots in the narrowed window, the returned `unreleasedSupply` is understated, causing the reported `releasedSupply` (circulating supply) to be overstated. Any downstream system (exchange, wallet, analytics) consuming this API receives silently wrong supply figures.

### Likelihood Explanation

No authentication or special privilege is required. Any HTTP client can issue:
```
GET /api/v1/network/supply?timestamp=lte:1700000000.000000000
```
The trigger is a single, valid, well-formed query parameter. The behavior is deterministic and 100% reproducible. The window of affected queries is any request where the user omits a lower bound or supplies a lower bound earlier than one month before the upper bound — a common and natural usage pattern.

### Recommendation

1. Only apply the `optimalLowerBound` optimization when the user has **not** provided an explicit lower bound, and document the narrowing behavior in the API spec.
2. When `lower == null` (no lower bound provided), either use `0` as the true lower bound (accepting the full scan cost) or explicitly document and enforce the one-month window as a hard API constraint, returning a 400 error if the implied range exceeds it.
3. When the user provides an explicit lower bound that is earlier than `optimalLowerBound`, respect the user's bound rather than silently overriding it:

```java
// Only apply optimization when no explicit lower bound was given
if (timestamp.getLower() == null) {
    minTimestamp = optimalLowerBound;
} else {
    // respect the explicit lower bound; optionally reject if range too wide
    minTimestamp = timestamp.getAdjustedLowerRangeValue();
}
```

### Proof of Concept

**Precondition:** The `account_balance` table contains a snapshot for an unreleased-supply account at timestamp `T_old`, where `T_old` is more than one calendar month before the queried upper bound `T_upper`.

**Steps:**
1. Insert a balance record at `T_old = 1_600_000_000_000_000_000` (Sept 2020).
2. Query: `GET /api/v1/network/supply?timestamp=lte:1700000000.000000000` (Nov 2023).
3. `optimalLowerBound = getFirstDayOfMonth(1700000000000000000, -1)` ≈ `1696118400000000000` (Oct 1, 2023).
4. `minTimestamp = Math.max(0, 1696118400000000000)` = `1696118400000000000`.
5. `getSupplyHistory(..., 1696118400000000000, 1700000000000000000)` finds no records (the only record is at `T_old` which is before the window).
6. `consensusTimestamp == 0` → `EntityNotFoundException("Network supply not found")` is thrown.

    private long getFirstDayOfMonth(long timestamp, int monthOffset) {

        final var dateTime = instant.atZone(ZoneOffset.UTC);
        final var firstDay = dateTime.plusMonths(monthOffset).withDayOfMonth(1);
