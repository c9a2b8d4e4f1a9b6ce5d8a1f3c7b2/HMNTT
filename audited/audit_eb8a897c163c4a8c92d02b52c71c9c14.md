### Title
`RegisteredNodeType.UNKNOWN` Queryable via Public API Exposes Unclassified Infrastructure Nodes

### Summary
The `RegisteredNodeType` Java enum includes `UNKNOWN(0)` as a valid member with id `0`, but the OpenAPI specification intentionally omits it from the public `type` query parameter. No server-side validation blocks an unprivileged caller from passing `?type=UNKNOWN`, causing the repository query `type @> array[:type]::smallint[]` to execute with `0` and return all non-deleted nodes whose type array contains `0`. These nodes are then silently re-labeled as `GENERAL_SERVICE` in the response by the mapper, masking their true unclassified status.

### Finding Description
**Code location:**
- `rest-java/src/main/java/org/hiero/mirror/restjava/repository/RegisteredNodeRepository.java`, lines 14–22, function `findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()`
- `rest-java/src/main/java/org/hiero/mirror/restjava/dto/RegisteredNodesRequest.java`, line 47
- `common/src/main/java/org/hiero/mirror/common/domain/node/RegisteredNodeType.java`, line 9

**Root cause:**
`RegisteredNodeType.UNKNOWN` is a fully valid Java enum constant with `id = 0`. [1](#0-0) 

`RegisteredNodesRequest.type` is declared as `RegisteredNodeType` with no `@Pattern`, `@AssertFalse`, or custom constraint that would reject the `UNKNOWN` member. [2](#0-1) 

The `RequestParameterArgumentResolver` delegates binding to Spring's `WebDataBinder`, which uses `StringToEnumConverterFactory`. That factory calls `Enum.valueOf()` after uppercasing the input, so `?type=UNKNOWN`, `?type=unknown`, and `?type=UnKnOwN` all resolve to `RegisteredNodeType.UNKNOWN`. [3](#0-2) 

`NetworkServiceImpl.getRegisteredNodes()` then calls `nodeType.getId()` → `(short) 0` and passes it directly to the repository. [4](#0-3) 

The native query executes `type @> array[0]::smallint[]`, returning every non-deleted row whose `type` array contains `0`. [5](#0-4) 

Finally, `RegisteredNodeMapper` silently re-maps `UNKNOWN` → `GENERAL_SERVICE` in the JSON response, hiding the true classification from the caller. [6](#0-5) 

**Why existing checks fail:**
The OpenAPI schema for `RegisteredNodeType` lists only `[BLOCK_NODE, GENERAL_SERVICE, MIRROR_NODE, RPC_RELAY]` — `UNKNOWN` is absent. [7](#0-6) 

However, this is a documentation-level constraint only; the server never enforces it. The `invalidTypeParam` integration test only verifies that a completely non-enum string (e.g., `"invalid"`) is rejected — it never tests `?type=UNKNOWN`. [8](#0-7) 

The test suite itself confirms that UNKNOWN-typed nodes can be persisted and are reachable: [9](#0-8) 

### Impact Explanation
An unprivileged external caller can enumerate all registered nodes that carry an unclassified (`UNKNOWN`) type — nodes that the system operators have not yet categorized. Because the mapper re-labels them as `GENERAL_SERVICE`, a caller acting on the response may route block-node or smart-contract RPC traffic to infrastructure that was never validated or intended for production use. This constitutes unintended smart-contract traffic routing with no direct fund loss but with concrete network-integrity risk (layer-0/1 node discovery).

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and a single HTTP GET request. The `UNKNOWN` enum name is visible in the Java source and is a natural value to probe. Any developer or researcher reading the codebase or fuzzing the `type` parameter would discover it immediately. Repeatability is unlimited.

### Recommendation
1. Add a server-side validation constraint on `RegisteredNodesRequest.type` that rejects `UNKNOWN`:
   ```java
   // Option A: custom @ValidRegisteredNodeType annotation
   // Option B: inline check in NetworkServiceImpl
   if (nodeType == RegisteredNodeType.UNKNOWN) {
       throw new IllegalArgumentException("Invalid parameter: type");
   }
   ```
2. Alternatively, remove `UNKNOWN` from the public `RegisteredNodeType` enum and use a separate internal sentinel, so it can never be bound from an HTTP parameter.
3. Add a test case asserting that `?type=UNKNOWN` returns HTTP 400.

### Proof of Concept
```
# 1. Seed a node with UNKNOWN type (id=0) in the database (as the importer does).
# 2. Send the following request as an unauthenticated user:

GET /api/v1/network/registered-nodes?type=UNKNOWN HTTP/1.1
Host: <mirror-node-host>

# 3. Observe HTTP 200 with the UNKNOWN-typed node returned,
#    its type silently shown as "GENERAL_SERVICE" in the JSON body.

# Case-insensitive variants also work:
GET /api/v1/network/registered-nodes?type=unknown
GET /api/v1/network/registered-nodes?type=Unknown
```
