### Title
Unbounded Historical Retrieval Loop via High-Volume Topic Subscription Without Limit in Throttled Mode

### Summary
In `PollingTopicMessageRetriever.retrieve()`, when called in throttled mode with no `limit` set on the filter, the `isComplete()` method never returns `true` as long as each DB poll returns a full page of results. Because `numRepeats` is set to `Long.MAX_VALUE` in throttled mode and the `.timeout()` operator only fires on per-item idle periods (not total duration), an unprivileged user can subscribe to a high-volume topic and hold a bounded elastic scheduler thread indefinitely while issuing continuous database queries.

### Finding Description

**Code path:**

In `TopicMessageServiceImpl.subscribeTopic()`, the historical retriever is always invoked with `throttled=true`: [1](#0-0) 

This calls `PollingTopicMessageRetriever.retrieve(filter, true)`. Inside `retrieve()`, the repeat predicate is `!context.isComplete()` with `numRepeats = Long.MAX_VALUE`: [2](#0-1) [3](#0-2) 

**Root cause — `isComplete()` in throttled mode:** [4](#0-3) 

- `limitHit` is `false` when no limit is set (`filter.hasLimit()` returns false when `limit == 0`).
- The only other exit condition is `pageSize.get() < retrieverProperties.getMaxPageSize()` (default 1000).
- `pageSize` is reset to 0 at the start of each `poll()` call and incremented per message via `onNext()`.
- If the topic has ≥ 1000 messages available per poll window, `pageSize` always equals `maxPageSize`, so `isComplete()` always returns `false`.

**Why the timeout does not protect:** [5](#0-4) 

Reactor's `Flux.timeout(Duration)` is a **per-item** idle timeout — it fires only if no item arrives within the window. When the topic continuously delivers a full page every poll cycle (every 2 seconds by default), items flow without interruption and the 60-second timeout never triggers.

**No subscription limit or authentication at the gRPC layer:** [6](#0-5) 

`subscriberCount` is tracked but no maximum is enforced: [7](#0-6) 

**Scheduler resource:** [8](#0-7) 

A single `Schedulers.boundedElastic()` instance is shared across all subscriptions. Each long-running repeat loop occupies a worker from this pool (default cap: `10 × CPU cores`).

### Impact Explanation
Each malicious subscription that never completes holds a bounded elastic scheduler worker and issues a DB query every 2 seconds indefinitely. With enough concurrent connections (easily opened from a single client), the attacker exhausts the bounded elastic thread pool, starving legitimate subscribers and degrading or halting the gRPC service. The DB is also subjected to sustained query load proportional to the number of attacker connections. This is a griefing/DoS with no economic damage to network participants, consistent with the stated scope.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint requires no authentication and no API key. Any network-reachable client can open arbitrarily many connections. The precondition — a topic with a large historical backlog or sustained high message rate — is realistic on a public mirror node (e.g., a popular HCS topic). The attack is trivially repeatable and scriptable with any gRPC client library.

### Recommendation
1. **Enforce a maximum subscription duration** using a total-elapsed timeout (not per-item), e.g., replace or supplement `.timeout(Duration)` with a `take(Duration)` or a wall-clock check inside `isComplete()`.
2. **Cap `numRepeats` in throttled mode** to a finite value (similar to the unthrottled path's `maxPolls`), or add a configurable `maxThrottledPolls` property.
3. **Enforce a per-IP or global maximum concurrent subscription count** in `ConsensusController` or `TopicMessageServiceImpl`, rejecting new subscriptions when the cap is reached.
4. **Require a non-zero `limit`** for historical retrieval requests, or enforce a server-side maximum limit when none is provided.

### Proof of Concept
**Preconditions:**
- A topic exists with ≥ 1,000 messages in the DB (or is actively receiving messages at that rate).
- The mirror node gRPC endpoint is reachable.

**Steps:**
1. Open N concurrent gRPC connections to `ConsensusService/subscribeTopic`.
2. In each request, set `topicID` to the high-volume topic, set `consensusStartTime` to a timestamp far in the past, and **omit `limit`** (set to 0).
3. Each connection triggers `retrieve(filter, true)` with `throttled=true`, `numRepeats=Long.MAX_VALUE`, and no limit.
4. Each poll returns 1,000 messages → `pageSize == maxPageSize` → `isComplete()` returns `false` → repeat fires again after 2 seconds.
5. Items flow continuously → per-item timeout never fires.
6. After opening `10 × CPU_CORES` such connections, the bounded elastic scheduler is saturated; new legitimate subscriptions cannot obtain a scheduler thread.
