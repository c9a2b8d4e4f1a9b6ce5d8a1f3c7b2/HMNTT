### Title
Cache Stampede in `getCachedTokens()` Allows Unauthenticated Non-Network DoS via DB Connection Pool Exhaustion

### Summary
`getCachedTokens()` in `rest/service/tokenService.js` has no in-flight request deduplication. Because the in-memory `tokenCache` is only populated after the async DB query resolves, any number of concurrent requests that arrive before the first query completes will each independently find the cache empty and each issue their own identical DB query. An unauthenticated attacker can exploit this to saturate the database connection pool and cause a denial of service.

### Finding Description
**Exact code path:**

`GET /api/v1/accounts/{id}/tokens` → `TokenController.getTokenRelationships` (line 74, `rest/controllers/tokenController.js`) → `TokenService.getTokenAccounts` (line 104, `rest/service/tokenService.js`) → `getCachedTokens()` (line 136).

Also reachable via `GET /api/v1/tokens/{tokenId}/balances` → `getTokenBalances` → `TokenService.getCachedTokens` (line 726, `rest/tokens.js`).

**Root cause — `rest/service/tokenService.js`, lines 136–161:**

```js
async getCachedTokens(tokenIds) {
  const uncachedTokenIds = [];
  tokenIds.forEach((tokenId) => {
    const cachedToken = tokenCache.get(tokenId);   // (1) synchronous cache check
    if (cachedToken) { ... } else { uncachedTokenIds.push(tokenId); }
  });

  if (uncachedTokenIds.length === 0) { return cachedTokens; }

  const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]); // (2) async DB query
  rows.forEach((row) => {
    tokenCache.set(tokenId, cachedToken);           // (3) cache populated ONLY after await resolves
  });
}
```

JavaScript's event loop means step (1) is synchronous for every concurrent request. All N concurrent requests check the cache before any of them reaches step (3). Because `tokenCache` is only written at step (3) — after the `await` resolves — every request that arrives between step (2) and step (3) of the first request will independently pass the `uncachedTokenIds.length === 0` guard and issue its own DB query.

There is no promise coalescing, no mutex, and no in-flight tracking. The `quickLru` cache (`tokenCache`, lines 12–14) is a plain synchronous key-value store with no concept of "pending" entries.

**Why the existing check is insufficient:**

The guard at line 148 (`if (uncachedTokenIds.length === 0) return`) only prevents a DB query when the cache is already warm. It provides zero protection against concurrent requests racing to populate a cold cache entry.

### Impact Explanation
An attacker who sends N concurrent requests to the unauthenticated public endpoint `GET /api/v1/accounts/{id}/tokens` (or `/api/v1/tokens/{tokenId}/balances`) for any account/token whose token IDs are not yet cached will cause N simultaneous identical SQL queries (`SELECT decimals, freeze_status, kyc_status, token_id FROM token WHERE token_id = ANY($1)`) to be issued against the database. With a sufficiently large N (easily achievable with a single script using `Promise.all`), the database connection pool is exhausted, causing all subsequent legitimate requests to queue or fail. The service is rendered unavailable without any network-layer flooding — a pure application-layer, non-network DoS.

### Likelihood Explanation
No authentication or special privilege is required. The endpoints are public. The attack requires only an HTTP client capable of issuing concurrent requests (e.g., `curl`, `ab`, `wrk`, or a 10-line Node.js script). The cold-cache condition is trivially achievable: the attacker simply targets token IDs that have not been recently queried, or waits for the LRU cache to evict entries (bounded by `config.cache.token.maxSize`). The attack is repeatable and can be sustained indefinitely.

### Recommendation
Introduce in-flight promise deduplication ("promise cache" or "single-flight" pattern). Before issuing a DB query, store the in-flight `Promise` in a secondary `Map` keyed by the sorted token ID set. Subsequent concurrent requests for the same IDs await the same promise instead of issuing new queries. Once the promise resolves, remove it from the in-flight map. Example sketch:

```js
const inFlight = new Map();

async getCachedTokens(tokenIds) {
  // ... check tokenCache as before, collect uncachedTokenIds ...
  if (uncachedTokenIds.length === 0) return cachedTokens;

  const key = uncachedTokenIds.slice().sort().join(',');
  if (!inFlight.has(key)) {
    const promise = super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds])
      .then(rows => { rows.forEach(row => tokenCache.set(row.token_id, new CachedToken(row))); })
      .finally(() => inFlight.delete(key));
    inFlight.set(key, promise);
  }
  await inFlight.get(key);
  // re-read from tokenCache
}
```

Alternatively, use a library such as `async-singleflight` or `p-memoize` with a short TTL.

### Proof of Concept
```js
// Precondition: token IDs 1001, 1002, 1003 are NOT in the tokenCache
// (fresh server start, or LRU eviction)

const fetch = require('node-fetch');
const TARGET = 'http://<mirror-node-host>/api/v1/accounts/0.0.5000/tokens';

// Fire 200 concurrent requests before any DB query can return and populate the cache
const requests = Array.from({length: 200}, () => fetch(TARGET));
Promise.all(requests).then(responses => {
  console.log('All done:', responses.map(r => r.status));
});
```

**Expected result:** 200 identical `SELECT ... FROM token WHERE token_id = ANY($1)` queries are issued simultaneously to the database. The DB connection pool (typically 10–20 connections by default) is saturated. Subsequent legitimate requests receive connection pool timeout errors or hang indefinitely, constituting a DoS.
