### Title
Unauthenticated Connection Pool Exhaustion via Unbounded Concurrent GET /blocks Requests

### Summary
The `GET /blocks` endpoint in `rest/routes/blockRoute.js` has no in-flight request limiting or per-IP rate limiting. The REST API's default database connection pool is capped at 10 connections per pod. An unprivileged attacker sustaining more than 10 concurrent requests forces legitimate requests to queue for up to 20 seconds waiting for a free connection, degrading or denying service to all users.

### Finding Description

**Code path:**

`rest/routes/blockRoute.js:12` registers `BlockController.getBlocks` with no concurrency guard: [1](#0-0) 

`rest/controllers/blockController.js:101-112` — `getBlocks()` immediately calls `RecordFileService.getBlocks()` with no semaphore, queue depth check, or per-client throttle: [2](#0-1) 

`rest/service/recordFileService.js:149-161` — every call acquires a real database connection from the shared pool and executes a `SELECT … FROM record_file WHERE index >= $1 AND index <= $2 ORDER BY index LIMIT N`: [3](#0-2) 

`rest/dbpool.js:14` — the pool hard-cap is `config.db.pool.maxConnections`, which defaults to **10** per REST pod: [4](#0-3) 

Confirmed default from documentation: [5](#0-4) 

`connectionTimeoutMillis` is 20 000 ms, meaning a request that cannot acquire a connection blocks a Node.js async context for up to 20 seconds before failing: [6](#0-5) 

**Why existing checks fail:**

The REST API Traefik middleware chain contains only a `circuitBreaker` (triggers on error-ratio or network-error-ratio thresholds) and a `retry` directive — **no `inFlightReq` and no `rateLimit`**: [7](#0-6) 

Compare with the Rosetta API, which explicitly adds both: [8](#0-7) 

The application-level middleware stack (`server.js`) adds only `authHandler`, logging, metrics, and optional Redis cache — zero concurrency controls: [9](#0-8) 

The `block.number=gte:0&lte:0` filter is accepted as valid input by `buildAndValidateFilters` and `extractSqlFromBlockFilters`, which maps it to `WHERE index >= 0 AND index <= 0`: [10](#0-9) 

The query is a fast indexed lookup, so each connection is held for only a few milliseconds — but at sustained concurrency above the pool cap, the 11th+ concurrent request queues inside `pg.Pool`, holding a Node.js Promise chain open for up to 20 seconds. Under continuous flood, the queue grows unboundedly, consuming event-loop memory and starving legitimate requests.

### Impact Explanation
All users of the public REST API share the same 10-connection pool per pod. Pool exhaustion causes `connectionTimeoutMillis` (20 s) delays or outright `Error: timeout exceeded when trying to connect` errors for every endpoint — not just `/blocks`. HPA can scale to 15 pods, but each pod has its own 10-connection pool; an attacker simply needs to direct enough load at a single pod (or scale the attack proportionally). The pgbouncer `max_user_connections: 250` for `mirror_rest` is a ceiling on the database side, not a per-client guard, and does not prevent pool exhaustion inside the Node.js process. [11](#0-10) 

### Likelihood Explanation
No authentication is required. The endpoint is publicly reachable at `/api/v1/blocks`. Standard HTTP benchmarking tools (`wrk`, `hey`, `ab`) can trivially sustain hundreds of concurrent connections from a single machine. Because the query is cheap (indexed lookup), the attacker does not need slow queries — they only need to keep more than 10 requests in-flight simultaneously per pod. This is achievable with a single laptop on a broadband connection. The attack is repeatable and stateless (no session needed).

### Recommendation
1. **Add `inFlightReq` Traefik middleware** to the REST API Helm chart, mirroring the Rosetta configuration (e.g., `amount: 50` globally, with per-IP source criterion), in `charts/hedera-mirror-rest/values.yaml`.
2. **Add `rateLimit` Traefik middleware** per source IP to cap sustained request rates.
3. **Increase `maxConnections`** or introduce an application-level semaphore (e.g., `p-limit`) in `rest/dbpool.js` / `rest/service/baseService.js` to bound the number of concurrent DB-acquiring calls.
4. **Reduce `connectionTimeoutMillis`** from 20 000 ms to a lower value (e.g., 3 000 ms) so queued requests fail fast rather than holding Node.js resources.

### Proof of Concept
```bash
# Requires: wrk (https://github.com/wg/wrk)
# Target: a single REST pod (bypass load balancer if possible, or use any pod)

wrk -t 20 -c 200 -d 60s \
  "https://<mirror-node-host>/api/v1/blocks?block.number=gte:0&block.number=lte:0"
```

**Expected result:** After the first 10 concurrent requests saturate the pool, subsequent requests from legitimate users receive HTTP 500 errors or 20-second hangs. Monitor with:
```bash
# In parallel, observe legitimate request latency
curl -w "%{time_total}\n" -o /dev/null -s \
  "https://<mirror-node-host>/api/v1/blocks?limit=1"
```
Latency will spike from <100 ms to ~20 000 ms (connection timeout) while the flood is active.
