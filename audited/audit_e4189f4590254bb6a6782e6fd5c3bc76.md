### Title
Unbounded Timestamp Query via Missing Default Enforcement in `optimizeTimestampFilters()` When `bindTimestampRange=false`

### Summary
When `bindTimestampRange` is `false` (the documented default), an unprivileged user can call `GET /api/v1/contracts/results` with no timestamp parameters, causing `optimizeTimestampFilters()` to return an empty filter set and the resulting SQL query to execute against the `contract_result` table with no timestamp `WHERE` clause. The only backstops are a `LIMIT` of at most 100 rows and a 20-second statement timeout, both of which are insufficient to prevent repeated, expensive full-table-scan queries.

### Finding Description

**Code path:**

`optimizeTimestampFilters()` — `rest/controllers/contractController.js` lines 384–405: [1](#0-0) 

Step 1 — `parseTimestampFilters` is called with `filterRequired=false` and `validateRange=false`: [2](#0-1) 

When `timestampFilters` is empty, `parseTimestampFilters` returns `{range: null, eqValues: [], neValues: []}` immediately without error: [3](#0-2) 

Step 2 — `range?.isEmpty()` on `null` evaluates to `undefined` (falsy), so the early-return guard is skipped: [4](#0-3) 

Step 3 — Because `eqValues.length === 0`, `bindTimestampRange(null, order)` is called: [5](#0-4) 

Step 4 — `bindTimestampRange` checks the config flag first. The documented default is `false`: [6](#0-5) 

With the flag `false`, the function immediately returns `{range: null}` — the null passed in, unchanged: [7](#0-6) 

Step 5 — Back in `optimizeTimestampFilters`, `optimizedRange` is `null`. Both `optimizedRange?.begin` and `optimizedRange?.end` are `undefined` (falsy), so no timestamp filters are pushed: [8](#0-7) 

The function returns `{filters: [], next: undefined}`.

Step 6 — The caller's skip-guard only fires when the *original* `timestampFilters` array was non-empty: [9](#0-8) 

Because the user supplied no timestamp filters, `timestampFilters.length === 0`, so the condition is `false` and the query is **not** skipped.

Step 7 — The SQL query is built and executed with no timestamp `WHERE` clause, performing a full scan of `contract_result`.

**Root cause / failed assumption:** The code assumes that either (a) `bindTimestampRange` will be enabled in production to enforce bounds, or (b) users will always supply at least one timestamp filter. Neither assumption is enforced. The `validateRange=false` argument to `parseTimestampFilters` explicitly disables the `maxTimestampRange` guard, and the skip-guard at line 506 is conditioned on the user having provided filters in the first place.

### Impact Explanation

An unprivileged, unauthenticated user can repeatedly issue `GET /api/v1/contracts/results` with no query parameters. Each request triggers a full sequential scan of the `contract_result` table. On a production mirror node with millions of rows this is a high-cost query. With a connection pool of 10 (default) and a statement timeout of 20 seconds, a small number of concurrent requests can saturate the pool and deny service to all other API consumers. Additionally, the response leaks up to 100 contract result records per request with no time-based access scoping, exposing the full historical dataset to any caller.

### Likelihood Explanation

No authentication or special privilege is required. The endpoint is publicly reachable. The exploit is a single HTTP GET with no parameters — trivially repeatable by any script. The default configuration (`bindTimestampRange=false`) means every standard deployment is affected unless an operator has explicitly opted in to the binding feature.

### Recommendation

1. **Change the default** of `bindTimestampRange` to `true`, or make `optimizeTimestampFilters` enforce a mandatory timestamp bound when no filters are supplied (e.g., default to `[now - maxTransactionsTimestampRangeNs, now]`) regardless of the config flag.
2. **Fix the skip-guard** at line 506 so that it also fires when `timestampFilters` was empty but `optimizedTimestampFilters` is also empty (i.e., remove the `timestampFilters.length !== 0` precondition, or add a separate guard for the no-filter case).
3. **Do not pass `validateRange=false`** to `parseTimestampFilters` from `optimizeTimestampFilters`; the range-size cap should apply here.

const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
};
