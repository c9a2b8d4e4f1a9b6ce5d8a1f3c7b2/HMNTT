### Title
Connection Pool Exhaustion DoS in `getBalances()` via Sequential Unauthenticated Timestamp-Filtered Requests During Network Partition

### Summary
When a `timestamp` query parameter is supplied, `getBalances()` in `rest/balances.js` executes up to 3 sequential `pool.queryQuietly()` calls per request. The default connection pool is capped at 10 connections with no application-level rate limiting. During a network partition, server-side `statement_timeout` never fires (it is a PostgreSQL-side setting and the server never receives the query over a partitioned TCP connection), so connections hang at the OS TCP layer until keepalive fires (minutes to hours). An unauthenticated attacker sending ~10 concurrent timestamp-filtered requests can permanently exhaust the pool, causing all subsequent `/balances` consumers to queue and eventually time out.

### Finding Description

**Exact code path — 3 sequential DB calls per timestamp-filtered request:**

`rest/balances.js`, `getBalances()`, lines 113–153:

```
if (tsQuery) {
    // DB call 1 — getAccountBalanceTimestampRange → pool.queryQuietly (line 189)
    const tsQueryResult = await getTsQuery(tsQuery, tsParams);
    ...
    // DB call 2 — getAccountIdsByPublicKey → pool.queryQuietly (line 66)
    const accountIdsQuery = await getAccountIdsByPublicKey(pubKeyParams, limit);
    ...
}
// DB call 3 — final main query
const result = await pool.queryQuietly(pgSqlQuery, sqlParams);
``` [1](#0-0) 

**Pool configuration — 10 connections, no app-level rate limiting:**

`rest/dbpool.js` configures the pool with `max: config.db.pool.maxConnections` (default **10**) and `connectionTimeoutMillis` (default **20 000 ms**) and `statement_timeout` (default **20 000 ms**). [2](#0-1) 

The `statement_timeout` is a PostgreSQL server-side setting. During a network partition the client's TCP connection is established but queries are never acknowledged by the server; the server never receives the command and therefore never fires the timeout. The connection hangs at the OS TCP layer until TCP keepalive fires (Linux default: ~2 hours).

**`queryQuietly` — no client-side query deadline:**

`rest/utils.js` lines 1518–1520: when called without `preQueryHint`, `queryQuietly` calls `this.query(query, params)` directly on the pool with no `AbortSignal`, no `Promise.race`, and no client-side socket timeout. [3](#0-2) 

**No application-level rate limiting on the REST API:**

A search of `rest/**/*.js` finds zero rate-limiting or in-flight-request-limiting middleware. The Traefik `inFlightReq`/`rateLimit` middleware in the Helm chart is optional infrastructure configuration, not enforced by the application itself. [4](#0-3) 

**Root cause / failed assumption:** The design assumes the database is always reachable within `statement_timeout`. Under a network partition this assumption fails: the server-side timeout never fires, connections are held open indefinitely, and the fixed-size pool (10) is exhausted by a small number of concurrent requests.

### Impact Explanation

With a pool of 10 connections and 3 sequential queries per timestamp-filtered request, only **4 concurrent attacker requests** are needed to hold all 10 connections (4 × 3 = 12 > 10, with overlap from sequential execution). Once the pool is saturated, every subsequent request from any consumer waits up to `connectionTimeoutMillis` (20 s) and then receives a pool-timeout error. The entire `/api/v1/balances` endpoint is effectively unavailable for all users for the duration of the partition. Because the attacker can continuously re-issue requests, the DoS is sustained, not transient.

### Likelihood Explanation

The endpoint is public and unauthenticated. No API key, token, or credential is required. The attack requires only an HTTP client capable of sending ~10 concurrent GET requests with a `?timestamp=lte:X&account.publicKey=Y` parameter. The network partition can be induced by an attacker who controls network infrastructure between the REST node and the database (e.g., a cloud security-group rule change, BGP hijack, or firewall manipulation), or it can be exploited opportunistically during any real partition event. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Add a client-side query timeout** inside `queryQuietly`: wrap `this.query()` with a `Promise.race` against a deadline (e.g., using `AbortController` or a manual timeout) so that hanging connections are released regardless of server-side timeout delivery.
2. **Reduce pool size or add per-endpoint concurrency limits** so a single endpoint cannot monopolize all connections.
3. **Add application-level rate limiting / in-flight request limiting** to the Node.js REST API (e.g., `express-rate-limit` or a semaphore per endpoint) rather than relying solely on optional Traefik middleware.
4. **Configure TCP keepalive** on the `pg` pool connections (`keepAlive: true`, `keepAliveInitialDelayMillis: <short value>`) so partitioned connections are detected and recycled quickly.

### Proof of Concept

**Preconditions:** The REST API is reachable. A network partition exists (or is induced) between the REST node and the PostgreSQL database.

**Steps:**

```bash
# 1. Induce or wait for a network partition between REST node and DB
#    (e.g., drop traffic: iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the REST host)

# 2. Send a burst of concurrent timestamp-filtered requests (no auth required)
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>:5551/api/v1/balances?timestamp=lte:9999999999.000000000&account.publicKey=3c3d546321ff6f63d701d2ec5c277095874e19f4a235bee1e6bb19258bf362be" &
done
wait

# 3. Immediately send a legitimate request from a different client
curl -v "http://<mirror-node-host>:5551/api/v1/balances"
# Result: hangs for 20 s then returns a 500/503 pool-timeout error
#         instead of a normal response — DoS confirmed
```

The 15 concurrent attacker requests each attempt 3 sequential `pool.queryQuietly()` calls. All 10 pool connections are held by hanging TCP sockets. The legitimate request in step 3 cannot acquire a connection and times out after `connectionTimeoutMillis` (20 s), demonstrating complete denial of service for all `/balances` consumers.
