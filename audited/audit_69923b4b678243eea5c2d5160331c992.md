### Title
Unauthenticated Connection Pool Exhaustion via `getDetailedContractResults()` Promise.all() Fan-Out

### Summary
The `getDetailedContractResults()` method in `rest/controllers/contractController.js` fires 5 concurrent database queries per request via `Promise.all()`. The REST API applies no rate limiting to the publicly accessible `GET /contracts/:contractId/results/:consensusTimestamp` endpoint. With a default pool of only 10 connections, as few as 2 concurrent unauthenticated requests fully exhaust the pool, causing all subsequent requests across all REST API instances sharing the same database to queue or timeout, effectively taking the REST API offline.

### Finding Description

**Exact code path:**

`getDetailedContractResults()` at `rest/controllers/contractController.js` lines 1205–1220:

```js
getDetailedContractResults = async (contractDetails, contractId = undefined) => {
  return Promise.all([
    ContractService.getContractResultsByTimestamps(...),
    TransactionService.getEthTransactionByTimestampAndPayerId(...),
    RecordFileService.getRecordFileBlockDetailsFromTimestamp(...),
    ContractService.getContractLogsByTimestamps(...),
    ContractService.getContractStateChangesByTimestamps(...),
  ]);
};
```

This is called unconditionally from `getContractResultsByTimestamp` (lines 1016–1017), which is the handler for `GET /contracts/:contractId/results/:consensusTimestamp` registered in `rest/routes/contractRoute.js` line 18.

**Root cause — three compounding failures:**

1. **No rate limiting on the REST API.** `rest/server.js` registers only `authHandler`, `requestLogger`, metrics, and cache middleware. There is no `express-rate-limit` or equivalent; `package.json` does not include any rate-limiting library. The throttling in `web3/ThrottleConfiguration.java` applies exclusively to the Web3 Java service.

2. **5 simultaneous DB connections per request.** `Promise.all()` dispatches all 5 queries at once. Each query holds a connection from the `pg` pool for its full execution duration (up to the `statementTimeout` of 20 000 ms).

3. **Default pool of 10 connections.** `rest/dbpool.js` line 14 sets `max: config.db.pool.maxConnections`, whose documented default is **10** (`hiero.mirror.rest.db.pool.maxConnections = 10`, `docs/configuration.md` line 556). Two concurrent requests consume all 10 connections.

**Why existing checks fail:**

`authHandler` only sets a per-user *response-size* limit in `httpContext` (it does not reject or throttle requests). No middleware enforces a per-IP or global request rate. The `statementTimeout` of 20 s prevents indefinite connection holding but creates a 20-second window per wave of requests — more than enough for an attacker to sustain pool exhaustion with a low request rate.

In production, pgbouncer is configured with `max_user_connections: 250` for `mirror_rest` (`charts/hedera-mirror/values.yaml` line 373). In transaction-mode pooling, connections are returned after each statement, but `Promise.all()` holds 5 server connections simultaneously for the duration of the slowest query. Exhausting 250 pgbouncer server connections requires only 50 concurrent requests — trivially achievable.

### Impact Explanation

When the pool is exhausted, every new request to any endpoint on every REST API pod that shares the same pgbouncer user (`mirror_rest`) blocks at connection acquisition until `connectionTimeoutMillis` (default 20 000 ms) elapses, then throws an error. This renders the entire REST API tier unavailable. Because all REST API replicas share the same pgbouncer pool (`max_user_connections: 250`), a single attacker targeting one endpoint can deny service to all replicas simultaneously. Mirror node REST API is the primary data-access layer for downstream integrators and wallets; its unavailability constitutes a significant availability impact on the mirror node infrastructure.

### Likelihood Explanation

The endpoint requires no credentials, no API key, and no special network access — it is publicly reachable on port 5551. The `contractId` and `consensusTimestamp` path parameters are easily obtained from any block explorer or prior API call. Sustaining 50 concurrent HTTP GET requests is achievable with a single commodity machine using `curl`, `ab`, `wrk`, or any scripting language. The attack is repeatable indefinitely because no IP-based blocking or circuit-breaker exists in the REST API layer.

### Recommendation

1. **Add rate limiting middleware** to the Express app (e.g., `express-rate-limit`) applied globally or specifically to the contracts route, before route handlers are registered in `rest/server.js`.
2. **Increase the default pool size** or document a minimum safe value relative to expected concurrency. At minimum, `maxConnections` should be sized to `(expected_concurrent_requests × 5) + headroom`.
3. **Serialize or batch the 5 queries** in `getDetailedContractResults()` where possible, or use a single joined query, to reduce per-request connection consumption from 5 to 1.
4. **Add a pgbouncer-level per-client connection limit** tied to source IP to prevent a single client from monopolizing the `mirror_rest` pool.
