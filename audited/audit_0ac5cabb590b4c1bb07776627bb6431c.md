### Title
Log Injection via Unsanitized `X-Forwarded-For` Header Enabling Tab-Character Field Poisoning

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the `X-Forwarded-For` (or `X-Real-IP`) header value verbatim and passes it directly into a `fmt.Sprintf`-constructed log message with zero sanitization. Because Go's `net/http` stack permits tab characters (0x09) in HTTP/1.1 header values, any unauthenticated client can inject a literal tab into the log stream. The application is explicitly configured to emit logfmt output destined for Loki, and any downstream tab-delimited parser or SIEM pipeline will misalign fields on every such request.

### Finding Description
**Code path:**

`rosetta/app/middleware/trace.go` — `getClientIpAddress()` (lines 63–74) returns the raw header string:

```go
ipAddress = r.Header.Get(xForwardedForHeader)   // line 67 — no validation
return ipAddress                                  // line 74
```

That value is immediately interpolated into the log message at lines 52–53:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```

and emitted via `log.Info(message)` (line 58).

**Logger configuration** (`rosetta/main.go`, lines 43–52) sets `log.TextFormatter` with `DisableColors: true` / `FullTimestamp: true` and the comment explicitly states *"Use logfmt for easy parsing by Loki"*. Logrus's `TextFormatter` does **not** escape tab characters inside the `msg` field value.

**Root cause:** The code assumes the header contains a well-formed IP address string. It performs no format validation, no character-class filtering, and no escaping before the value enters the log pipeline.

**Why Go passes tabs through:** RFC 7230 permits horizontal tab (0x09) as optional whitespace in header values. Go's `net/http`/`textproto` stack trims leading/trailing whitespace but preserves embedded tabs. A value such as `1.2.3.4\t<injected>` survives header parsing intact and reaches `getClientIpAddress()` unchanged.

**Exploit flow:**
1. Attacker sends: `X-Forwarded-For: 1.2.3.4\tFAKE_FIELD fake_value`
2. `getClientIpAddress()` returns `"1.2.3.4\tFAKE_FIELD fake_value"`.
3. `fmt.Sprintf` produces: `"1.2.3.4\tFAKE_FIELD fake_value GET /network/list (200) in 2ms"`.
4. Logrus emits: `time="..." level=info msg="1.2.3.4` **TAB** `FAKE_FIELD fake_value GET /network/list (200) in 2ms"`.
5. Any tab-splitting parser (Loki pipeline stage, Splunk field extractor, custom SIEM rule) splits at the injected tab, misaligning every subsequent field.

**Existing checks reviewed:** The test file `trace_test.go` only tests well-formed IP strings (`"10.0.0.100"`, `"192.0.2.1"`). There is no IP-format validation, no character allowlist, and no escaping anywhere in the middleware or in the logger configuration.

### Impact Explanation
Every log line for an external request is poisoned. In a tab-delimited SIEM pipeline the injected tab shifts all fields right by one column: `level`, `method`, `path`, `status`, and `latency` are all misread or dropped. An attacker can also inject plausible-looking fake log entries (e.g., a spoofed IP followed by a tab and a fabricated status code), undermining audit trails and alerting fidelity. Because the Rosetta API is a public-facing HTTP endpoint, every request is logged at `INFO` level, maximising exposure.

### Likelihood Explanation
No authentication, no special network position, and no prior knowledge of the system is required. Any HTTP client (curl, Python requests, etc.) can set an arbitrary `X-Forwarded-For` value. The attack is trivially repeatable and fully automated. The only prerequisite is that a tab-delimited log consumer exists downstream — a realistic assumption given the explicit Loki/logfmt configuration.

### Recommendation
Sanitize the IP string returned by `getClientIpAddress()` before it enters the log pipeline. The minimal fix is to validate that the value is a syntactically legal IP address using `net.ParseIP` (after splitting on `,` for multi-hop `X-Forwarded-For` lists) and fall back to `r.RemoteAddr` on failure. Additionally, strip or replace all ASCII control characters (0x00–0x1F, 0x7F) from any header-derived string before logging. Example:

```go
func sanitizeIP(raw string) string {
    // take only the first address in a comma-separated list
    if idx := strings.IndexByte(raw, ','); idx != -1 {
        raw = raw[:idx]
    }
    raw = strings.TrimSpace(raw)
    if ip := net.ParseIP(raw); ip != nil {
        return ip.String()
    }
    return "" // invalid — caller falls through to RemoteAddr
}
```

### Proof of Concept
```bash
# Inject a tab character into X-Forwarded-For
curl -s \
  -H $'X-Forwarded-For: 1.2.3.4\tINJECTED_FIELD injected_value' \
  http://<rosetta-host>:<port>/network/list \
  -X POST -d '{...}'

# Expected log line (logfmt, tabs rendered as \t):
# time="2024-01-01T00:00:00Z" level=info msg="1.2.3.4	INJECTED_FIELD injected_value POST /network/list (200) in 3ms"
#
# A tab-splitting parser sees:
#   col1: time="2024-01-01T00:00:00Z" level=info msg="1.2.3.4
#   col2: INJECTED_FIELD injected_value POST /network/list (200) in 3ms"
