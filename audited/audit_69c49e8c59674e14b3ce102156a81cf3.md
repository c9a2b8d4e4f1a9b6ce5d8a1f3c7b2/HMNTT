### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Alias-Keyed GraphQL Requests

### Summary
The `account(input: AccountInput!)` GraphQL query accepts an `alias` field validated only by a regex pattern (`^[A-Z2-7]+$`). The Caffeine preparsed-document cache is keyed on the raw query string, not on variables, so embedding distinct alias values inline in each request produces a unique cache key per request, bypassing the parse cache entirely. Because `EntityServiceImpl.getByAliasAndType()` has no result cache and the graphql module has no rate limiting, every request unconditionally acquires a HikariCP connection and issues a live SQL query, allowing an unprivileged attacker to exhaust the database connection pool.

### Finding Description

**Cache keying â€” parse cache only, no result cache:**

`CachedPreparsedDocumentProvider.getDocumentAsync()` keys the Caffeine cache on `executionInput.getQuery()` â€” the raw query string. [1](#0-0) 

When an alias value is embedded inline (not as a GraphQL variable), each distinct alias produces a distinct query string and therefore a distinct cache key. The cache holds at most 1,000 entries (`maximumSize=1000`), so an attacker rotating through >1,000 unique Base32 strings keeps the cache permanently cold. [2](#0-1) 

**No result caching in the service layer:**

`EntityServiceImpl.getByAliasAndType()` calls `entityRepository.findByAlias()` directly with no `@Cacheable` annotation, unlike the web3 module's `EntityRepository` which does cache alias lookups. Every request reaches the database. [3](#0-2) [4](#0-3) 

**No rate limiting in the graphql module:**

The `ThrottleConfiguration` / `ThrottleManagerImpl` bucket-based rate limiter exists only in the `web3` module. The graphql module's only filter is `LoggingFilter`, which performs no throttling. `GraphQlConfiguration` adds `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, but these limit per-query complexity, not request throughput. [5](#0-4) [6](#0-5) 

**Alias field accepts an unbounded valid input space:**

The schema constrains the alias to `^[A-Z2-7]+$` (RFC 4648 Base32 without padding), but imposes no maximum length at the schema level. The parser-level cap (`maxCharacters=10000`) allows alias strings up to thousands of characters, each producing a unique cache key and a unique DB query. [7](#0-6) 

**Exploit flow:**

1. Attacker generates N unique Base32 strings matching `^[A-Z2-7]+$` (e.g., `AAAA`, `AAAB`, `AAAC`, â€¦).
2. Each request is sent as: `POST /graphql/alpha` with body `{"query":"{ account(input:{alias:\"AAAA\"}) { id } }"}`.
3. Because the alias is inline, each request has a unique query string â†’ Caffeine cache miss â†’ GraphQL re-parses the document.
4. `AccountController.account()` calls `EntityServiceImpl.getByAliasAndType()` â†’ `EntityRepository.findByAlias()` â†’ acquires a HikariCP connection and executes `SELECT * FROM entity WHERE alias = ? AND deleted IS NOT TRUE`.
5. With concurrent requests exceeding the HikariCP pool size (default 10), subsequent requests block waiting for a connection. Under sustained load the pool is fully occupied, causing `SQLTransientConnectionException` / timeout for all legitimate users.

### Impact Explanation

Complete denial of service for the GraphQL API and any other service sharing the same HikariCP pool. Because the graphql module uses the shared `CommonConfiguration` datasource with no graphql-specific pool sizing, pool exhaustion affects the entire application. The attack requires no credentials, no special knowledge, and no expensive computation â€” only the ability to POST to a public HTTP endpoint.

### Likelihood Explanation

The endpoint is publicly accessible with no authentication. Generating valid Base32 strings is trivial. A single attacker with a modest script (e.g., 50 concurrent threads, each cycling through unique aliases) can sustain pool exhaustion indefinitely. The attack is repeatable, requires no prior reconnaissance, and leaves no persistent state that would block re-execution.

### Recommendation

1. **Add rate limiting to the graphql module**: Implement a `bucket4j`-based filter (mirroring `ThrottleConfiguration` in the web3 module) that limits requests per IP or globally per second on the `/graphql/alpha` endpoint.
2. **Add result caching to `EntityServiceImpl`**: Annotate `getByAliasAndType()` with `@Cacheable` (keyed on the decoded alias bytes) so repeated lookups for the same alias do not hit the database.
3. **Enforce a maximum alias length** in the schema (e.g., `@Size(max=64)`) to bound the input space and reduce parse-cache churn.
4. **Size the HikariCP pool explicitly** for the graphql module and configure a short `connectionTimeout` so pool exhaustion fails fast rather than queuing indefinitely.

### Proof of Concept

```bash
# Generate 2000 unique valid Base32 alias strings and fire concurrent requests
python3 - <<'EOF'
import itertools, string, subprocess, threading

chars = "ABCDEFGHIJKLMNOPQRSTUVWXYZ234567"
aliases = [''.join(c) for c in itertools.islice(itertools.product(chars, repeat=4), 2000)]

def send(alias):
    subprocess.run([
        "curl", "-s", "-o", "/dev/null",
        "-X", "POST", "http://<GRAPHQL_HOST>/graphql/alpha",
        "-H", "Content-Type: application/json",
        "-d", f'{{"query":"{{ account(input:{{alias:\\"{alias}\\"}}) {{ id }} }}"}}'
    ])

threads = [threading.Thread(target=send, args=(a,)) for a in aliases]
for t in threads: t.start()
for t in threads: t.join()
EOF
```

Each of the 2,000 requests carries a unique inline alias, producing a unique Caffeine cache key (cache miss) and an unconditional `SELECT` against the `entity` table. With the default HikariCP pool size of 10, the pool is exhausted within the first wave of concurrent requests, and subsequent legitimate GraphQL queries receive connection-timeout errors.
