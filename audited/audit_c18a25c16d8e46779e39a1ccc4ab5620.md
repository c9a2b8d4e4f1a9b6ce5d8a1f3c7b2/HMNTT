### Title
Unauthenticated Connection Pool Exhaustion via Multi-Query Amplification in `getSchedules()`

### Summary
`getSchedules()` in `rest/schedules.js` issues up to three sequential/parallel database queries per request against a shared pool defaulting to 10 connections, with no rate limiting on the REST API layer. During a network partition — when the database becomes slow and connections are held for up to the 20-second `statementTimeout` — an unprivileged attacker sending as few as 5 concurrent requests can exhaust the entire pool, denying service to all REST API endpoints for the duration of each burst.

### Finding Description

**Code path:**

`rest/schedules.js`, `getSchedules()`, lines 234–282:

```
Line 241:  const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);
           // Connection 1 acquired, held until query resolves, then released.

Lines 261–264:
  const [{rows: entities}, {rows: signatures}] = await Promise.all([
    pool.queryQuietly(entityQuery, entityIds),   // Connection 2 acquired
    pool.queryQuietly(signatureQuery, entityIds), // Connection 3 acquired simultaneously
  ]);
```

The first query is sequential (1 connection at a time). The second and third queries run concurrently via `Promise.all`, holding **2 connections simultaneously** per in-flight request during that phase.

**Pool configuration** (`rest/dbpool.js`, line 14):
```js
max: config.db.pool.maxConnections,  // default: 10
```
Default `maxConnections` is **10** and `statementTimeout` is **20,000 ms** (20 seconds), with `connectionTimeoutMillis` also 20,000 ms.

**No rate limiting exists on the REST API.** A grep across all `rest/**/*.js` files finds zero rate-limiting middleware. The middleware stack (`rest/middleware/index.js`) exports only: `authHandler`, `handleError`, `openApiValidator`, `requestHandler`, `responseCacheHandler`, `responseHandler` — no throttle. The `ThrottleManagerImpl` and Traefik rate-limit middleware found in the repository apply exclusively to the `web3` (Java) and `rosetta` (Go) services respectively, not to the Node.js REST API.

**Root cause:** The failed assumption is that the pool's `connectionTimeout` and `statementTimeout` are sufficient to self-protect. They are not: they define the *maximum hold time*, not a *concurrency ceiling per client*. During a network partition, every query hangs for the full 20 seconds, turning each `getSchedules()` call into a 20-second, 2-connection lease. With pool size 10, only 5 concurrent requests are needed to saturate the pool.

### Impact Explanation

Pool exhaustion blocks **all** REST API endpoints — not just `/schedules` — because they share the same global `pool` object. Any endpoint calling `pool.queryQuietly()` will queue behind the exhausted pool and fail after `connectionTimeoutMillis` (20 seconds), returning 500 errors to all users. The impact is a complete REST API denial-of-service for the duration of each attack burst (up to 20 seconds per wave, repeatable continuously). Severity is **High** given the total service impact and zero-privilege requirement.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no sophisticated tooling — only the ability to send HTTP GET requests. The `/schedules` endpoint is publicly documented in the OpenAPI spec. Sending 5–10 concurrent requests is trivially achievable with `curl --parallel`, `ab`, `wrk`, or any scripting language. The attack is repeatable: each 20-second burst can be immediately followed by another. A single attacker on a residential connection can sustain this indefinitely. Likelihood is **High**.

### Recommendation

1. **Add per-IP rate limiting to the REST API layer.** Introduce an Express rate-limiting middleware (e.g., `express-rate-limit`) applied globally or specifically to expensive endpoints. The rosetta Traefik config (`inFlightReq.amount: 5`, `rateLimit.average: 10`) provides a reference model.

2. **Reduce `statementTimeout` for list endpoints.** A 20-second timeout is excessive for paginated list queries. A 5-second timeout would reduce the window during which connections are held under partition.

3. **Increase default `maxConnections` or use a connection pooler** (e.g., PgBouncer in transaction mode, already present in the Helm chart for `mirror_rest` with `max_user_connections: 250`) to decouple the application pool size from the PostgreSQL server limit.

4. **Merge the three queries into a single SQL query** using JOINs or CTEs, as is already done in `getScheduleByIdQuery` (lines 46–64), which performs all three lookups in one round-trip and holds only one connection.

### Proof of Concept

**Preconditions:** Network partition between the REST API node and the database (simulate with `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the DB host, or by introducing artificial latency exceeding `statementTimeout`).

**Steps:**

```bash
# Send 6 concurrent requests to /schedules (exceeds pool capacity of 10 / 2 = 5)
for i in $(seq 1 6); do
  curl -s "http://<mirror-node-host>:5551/api/v1/schedules?account.id=0.0.1024" &
done
wait

# Immediately probe a different endpoint to confirm pool exhaustion
curl -v "http://<mirror-node-host>:5551/api/v1/accounts/0.0.2"
# Expected: 500 Internal Server Error or connection timeout after ~20 seconds
```

**Expected result:** The 6 concurrent `/schedules` requests each enter the `Promise.all` phase and hold 2 connections each (12 total > pool max of 10). The probe to `/api/v1/accounts` queues for a connection, waits 20 seconds, and returns an error. The attack is repeatable in a continuous loop to maintain the DoS.
