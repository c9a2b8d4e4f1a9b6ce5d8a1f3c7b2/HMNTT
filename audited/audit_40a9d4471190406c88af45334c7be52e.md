### Title
N+1 Sequential DB Lookups via Repeated EVM Address `from` Filters in `GET /contracts/results`

### Summary
The `getContractResults` handler passes all `from` query parameters to `extractContractResultsByIdQuery`, which iterates over each filter sequentially and calls `await EntityService.getEncodedId()` for every EVM-address-formatted value. Because the lookups are sequential (not batched or cached) and no authentication is required, an unauthenticated attacker can trigger N individual DB round-trips before the main query executes, simply by repeating the `from` parameter up to the `maxRepeatedQueryParameters` limit.

### Finding Description

**Exact code path:**

1. `GET /contracts/results` → `ContractController.getContractResults` [1](#0-0) 

2. `getContractResults` calls `utils.buildAndValidateFilters` then `this.extractContractResultsByIdQuery(filters)` with no `contractId`. [2](#0-1) 

3. Inside `extractContractResultsByIdQuery`, the `for...of` loop processes each `FROM` filter. For every filter value that passes `EntityId.isValidEvmAddress()`, it issues `await EntityService.getEncodedId(filter.value)` — a sequential, blocking DB call: [3](#0-2) 

4. `EntityService.getEncodedId` → `getEntityIdFromEvmAddress` executes a live DB query with no caching: [4](#0-3) 

5. The only guard is `isRepeatedQueryParameterValidLength` inside `buildFilters`, which rejects arrays longer than `config.query.maxRepeatedQueryParameters` (default > 4 per tests). This bounds N but does not prevent the amplification: [5](#0-4) 

**Root cause:** The `await` inside the `for...of` loop serialises every EVM-address lookup. There is no batching (e.g., a single `WHERE evm_address = ANY($1)` query), no in-request deduplication, and no cache. Each unique EVM address in the `from` array costs one full DB round-trip before the main `contract_result` query runs.

**Failed assumption:** The code assumes that the number of EVM-address `from` filters per request will be small enough that sequential lookups are harmless. The `maxRepeatedQueryParameters` cap limits N but still allows an attacker to force N+1 DB queries per HTTP request with zero privilege.

### Impact Explanation
An attacker sends a stream of concurrent requests, each carrying the maximum allowed number of distinct EVM-address `from` values. Every request forces N sequential DB queries (entity lookups) plus the main contract-results query. Because the entity lookups are sequential and not cached, each request holds a DB connection for the full duration of N round-trips. Under sustained concurrent load this exhausts the DB connection pool, increases CPU and I/O on the database node, and degrades response times for all users. The amplification factor is proportional to `maxRepeatedQueryParameters`, making it straightforward to exceed a 30% resource increase without brute-force volume.

### Likelihood Explanation
No authentication, API key, or special privilege is required. The endpoint is publicly reachable. The attacker only needs to know valid EVM address formats (40 hex chars), which are trivially constructable. The attack is repeatable, scriptable, and requires no prior knowledge of the system state — even addresses that resolve to `NotFoundError` still complete the DB round-trip before throwing. A single attacker with a modest HTTP client can sustain the load.

### Recommendation
1. **Batch the EVM lookups**: Collect all EVM-address `from` values first, then resolve them in a single query (`WHERE evm_address = ANY($1::bytea[])`) before building conditions.
2. **Deduplicate**: Skip repeated identical EVM addresses within the same request.
3. **Short-lived request-scoped cache**: Cache resolved EVM→encodedId mappings for the lifetime of a single request.
4. **Tighten `maxRepeatedQueryParameters`** for the `from` filter specifically, or add a separate, lower cap for filters that require DB resolution.

### Proof of Concept

```bash
# Construct N distinct EVM addresses (all syntactically valid 40-hex-char strings)
# and send them as repeated `from` parameters in a single request.

BASE="http://<mirror-node-host>/api/v1"
PARAMS=""
for i in $(seq 1 10); do
  ADDR=$(printf "deadbeef%036x" $i)   # valid 40-char hex EVM address
  PARAMS="${PARAMS}&from=${ADDR}"
done

# Single unauthenticated GET triggers 10 sequential DB lookups + 1 main query
curl -s "${BASE}/contracts/results?${PARAMS}"

# Repeat concurrently to amplify DB load
for j in $(seq 1 50); do
  curl -s "${BASE}/contracts/results?${PARAMS}" &
done
wait
```

Each request causes 10 sequential `SELECT id FROM entity WHERE evm_address = $1` queries before the main `contract_result` query, resulting in 11 DB round-trips per HTTP request with no authentication required.
