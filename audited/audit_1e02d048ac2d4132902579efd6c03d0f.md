### Title
Arbitrary Address Book File ID Injection via Unvalidated `file.id` Query Parameter in `/api/v1/network/nodes`

### Summary
The `GET /api/v1/network/nodes` endpoint accepts a user-supplied `file.id` query parameter that is passed directly to the database query without any validation that it equals the canonical address book file ID (102). Any unauthenticated caller can supply an arbitrary entity ID (e.g., `0.0.101`), causing the service to return node data from a different or stale address book, misleading clients about the current network topology.

### Finding Description

**Controller** (`NetworkController.java`, lines 154–157): The only check performed on the user-supplied `fileId` is that its operator is `EQ`. The actual numeric value is never validated: [1](#0-0) 

**Service** (`NetworkServiceImpl.java`, lines 178–182): `getAddressBookFileId()` returns the caller-supplied value verbatim when it is non-null, with no check that it equals `systemEntity.addressBookFile102().getId()`: [2](#0-1) 

**Repository** (`NetworkNodeRepository.java`, line 29): The native SQL query uses `where file_id = :fileId`, so whatever value is injected flows directly into the database filter: [3](#0-2) 

**Exploit flow:**
1. Attacker sends `GET /api/v1/network/nodes?file.id=101` (no authentication required).
2. Controller validates operator is `EQ` — passes.
3. `getAddressBookFileId()` returns `101` (the legacy address book entity ID).
4. `findNetworkNodes(101, ...)` executes `where file_id = 101` against `address_book`, returning the legacy/stale node set.
5. Response is served to the caller as if it were the current authoritative node list.

### Impact Explanation
File `0.0.101` is the legacy Hedera address book (nodes 0.0.3–0.0.27); file `0.0.102` is the current authoritative one. A client using the poisoned response to select nodes for transaction submission would route transactions to stale or incorrect endpoints, causing failures or enabling targeted denial-of-service against specific nodes. The mirror node API is a trusted source of network topology for SDK clients and third-party integrations, amplifying the blast radius.

### Likelihood Explanation
Exploitation requires zero privileges — it is a plain unauthenticated HTTP GET request with a single modified query parameter. It is trivially repeatable, requires no special tooling, and is discoverable by any attacker who reads the OpenAPI spec or observes the `file.id` parameter in normal traffic. The attack is completely passive from the mirror node's perspective (read-only query), so it leaves no anomalous write footprint.

### Recommendation
In `getAddressBookFileId()`, reject any supplied `fileId` that does not equal the canonical address book ID:

```java
private long getAddressBookFileId(final NetworkNodeRequest request) {
    long canonicalId = systemEntity.addressBookFile102().getId();
    if (request.getFileId() != null) {
        long supplied = request.getFileId().value();
        if (supplied != canonicalId) {
            throw new IllegalArgumentException(
                "file.id must be " + EntityId.of(canonicalId).toString());
        }
        return supplied;
    }
    return canonicalId;
}
```

Alternatively, remove the `file.id` parameter from the public API entirely if supporting multiple address books is not an intended feature.

### Proof of Concept

```bash
# Returns nodes from the legacy address book (file 101) instead of the current one (file 102)
curl -s "https://<mirror-node-host>/api/v1/network/nodes?file.id=101" | jq '.nodes[].node_id'

# Compare with canonical response
curl -s "https://<mirror-node-host>/api/v1/network/nodes" | jq '.nodes[].node_id'

# The two responses will differ, confirming arbitrary address book selection.
# No authentication, API key, or special headers required.
```
