### Title
Unauthenticated Full Table Scan via Unbounded `/contracts/results/logs` Endpoint When `bindTimestampRange` Is Disabled

### Summary
The public `GET /api/v1/contracts/results/logs` endpoint (`getContractLogs()`) calls `extractContractLogsMultiUnionQuery()` without a `contractId`, unconditionally entering the `contractId === undefined` optimization branch and invoking `optimizeTimestampFilters()` → `bindTimestampRange()`. When `bindTimestampRange=false` (the default), no timestamp constraint is applied and the resulting SQL query performs an unbounded full table scan of `contract_log`. Any unauthenticated user can trigger this with a single zero-filter request.

### Finding Description
**Exact code path:**

`getContractLogs()` (line 835) calls `extractContractLogsMultiUnionQuery(filters)` with no `contractId` argument (defaults to `undefined`): [1](#0-0) 

Inside `extractContractLogsMultiUnionQuery`, the branch at line 678 is taken whenever `transactionHash === undefined` AND `contractId === undefined`: [2](#0-1) 

This calls `optimizeTimestampFilters()` (line 384), which calls `bindTimestampRange(range, order)` whenever `eqValues.length === 0` (i.e., no `eq` timestamp filter was supplied): [3](#0-2) 

`bindTimestampRange` returns `{range: null}` immediately when `queryConfig.bindTimestampRange` is `false`: [4](#0-3) 

Back in `extractContractLogsMultiUnionQuery`, `bounds.primary` is reset to an empty `Bound` with no filters, so the final SQL query carries **no timestamp WHERE clause** — a full table scan of `contract_log`.

**Root cause:** The global logs endpoint has no mandatory timestamp filter requirement (unlike the per-contract endpoint). The `bindTimestampRange` guard is opt-in and defaults to `false`, leaving the query unbounded.

**Why existing checks fail:**
- `checkTimestampsForTopics` (line 823) only enforces a timestamp when topic filters are present; a bare request with no parameters passes it silently. [5](#0-4) 
- `acceptedContractLogsParameters` does not require any filter to be present. [6](#0-5) 
- The `limit` parameter caps the number of *returned* rows but does not prevent the database from scanning the entire `contract_log` table to find them.

### Impact Explanation
A single unauthenticated HTTP request causes the database to perform a full sequential scan of the `contract_log` table with no predicate. On a production network with millions of log entries, this query can consume significant I/O, CPU, and memory on the database node. Repeated requests (even at low rate) can saturate database resources, causing query timeouts and service unavailability for all mirror node consumers — consistent with the "≥30% network processing node shutdown" severity tier.

### Likelihood Explanation
The exploit requires zero privileges, zero authentication, and zero knowledge of contract IDs or timestamps. The attack surface is the public REST API. A single curl command is sufficient to trigger the condition. The attack is trivially repeatable and automatable.

### Recommendation
1. **Require at least one timestamp filter** on the global `/contracts/results/logs` endpoint (similar to how topics require a timestamp). Reject requests with no timestamp parameter with HTTP 400.
2. **Enable `bindTimestampRange=true` by default** in production configuration so that even filter-less requests are bounded to `maxTransactionsTimestampRangeNs`.
3. Add a server-side statement timeout on the database connection used by this endpoint to cap runaway query duration.

### Proof of Concept
```
# No authentication, no filters — triggers unbounded full table scan
curl -s "https://<mirror-node-host>/api/v1/contracts/results/logs"

# Also triggers the same path (no eq timestamp, no transaction_hash, no contractId):
curl -s "https://<mirror-node-host>/api/v1/contracts/results/logs?order=asc"
```

Preconditions: `config.query.bindTimestampRange = false` (default). No account or API key required.

Trigger: The request reaches `getContractLogs()` → `extractContractLogsMultiUnionQuery([], undefined)` → `optimizeTimestampFilters([], 'asc')` → `bindTimestampRange(null, 'asc')` returns `{range: null}` → SQL executes with no timestamp predicate → full `contract_log` table scan.

  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);

  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
