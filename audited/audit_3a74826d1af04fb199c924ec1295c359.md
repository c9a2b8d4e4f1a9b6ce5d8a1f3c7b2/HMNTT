### Title
Unbounded `node.id` Parameter List Enables Database Resource Exhaustion via `getNetworkNodes()`

### Summary
The `GET /api/v1/network/nodes` endpoint accepts an unlimited number of `node.id=eq:X` query parameters. These are collected into a `HashSet<Long>` with no size cap, then passed as a `Long[]` array directly into a database query, allowing any unauthenticated user to craft a request that forces the database to evaluate a `node_id = any(:nodeIds)` clause with thousands of entries, exhausting database resources.

### Finding Description
In `NetworkNodeRequest.java`, the `nodeIds` field is declared as a `List<NumberRangeParameter>` with only a `@RestJavaQueryParam` annotation and no `@Size` constraint limiting how many values can be submitted: [1](#0-0) 

In `NetworkServiceImpl.java`, `getNetworkNodes()` iterates over every submitted `nodeIdParam` and unconditionally adds EQ values to a `HashSet<Long>`: [2](#0-1) 

The entire populated set is then streamed into a `Long[]` array and forwarded to the repository: [3](#0-2) 

The `getEffectiveLimit()` method only caps the number of *returned rows* at 25, not the number of input node IDs: [4](#0-3) 

The controller applies no additional size guard before delegating to the service: [5](#0-4) 

### Impact Explanation
An attacker can force the database to parse and evaluate a `node_id = any(:nodeIds)` array with tens of thousands of elements per request. Repeated requests amplify this into sustained CPU and memory pressure on the database, causing degraded performance or denial of service for all users of the mirror node API. No authentication is required.

### Likelihood Explanation
The endpoint is public and unauthenticated. HTTP query strings support repeated parameters natively; any HTTP client can trivially generate a request with thousands of `node.id=eq:X` values. The attack is stateless, requires no special knowledge, and is trivially repeatable or scriptable. No rate limiting or input size validation exists at any layer in the observed code path.

### Recommendation
Add a `@Size(max = N)` constraint on the `nodeIds` field in `NetworkNodeRequest`, where `N` is a small reasonable bound (e.g., 25, matching `MAX_LIMIT`):

```java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 25)
private List<NumberRangeParameter> nodeIds = List.of();
```

Additionally, enforce validation in `getNetworkNodes()` as a defense-in-depth measure, and ensure Spring's `@Validated` is active on the controller or service so the `@Size` constraint is actually enforced at runtime.

### Proof of Concept
```bash
# Generate a request with 10,000 node.id=eq:X parameters
python3 -c "
params = '&'.join(f'node.id=eq:{i}' for i in range(10000))
print(f'GET /api/v1/network/nodes?{params}')
" | curl -s -o /dev/null -w "%{time_total}" \
  "http://<mirror-node-host>/api/v1/network/nodes?$(python3 -c "print('&'.join(f'node.id=eq:{i}' for i in range(10000)))")"
```

Repeating this in a loop from a single client will cause measurable database CPU and memory spikes. The attack requires no credentials, no special headers, and no prior knowledge of valid node IDs.
