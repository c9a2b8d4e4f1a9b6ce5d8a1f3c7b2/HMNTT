### Title
Unauthenticated Alias Lookup DoS via Uncached DB Queries in GraphQL `account()` Endpoint

### Summary
The GraphQL `account()` endpoint in `AccountController` accepts an `alias` input validated only by a regex pattern (`^[A-Z2-7]+$`), which confirms format but not entity existence. Every valid alias that resolves to a non-existent entity causes `EntityServiceImpl.getByAliasAndType()` to unconditionally invoke `decodeBase32()` and `entityRepository.findByAlias()` (a raw SQL query) with no result caching at the application layer. An unprivileged external user can sustain elevated database query load by repeatedly submitting valid-format, non-existent aliases.

### Finding Description

**Code path:**

`AccountController.java` line 47–48:
```java
if (alias != null) {
    return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
}
``` [1](#0-0) 

`EntityServiceImpl.java` line 29–31 — no caching, direct DB call on every invocation:
```java
public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
    return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
}
``` [2](#0-1) 

`EntityRepository.java` line 13–14 — raw SQL executed per request:
```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
``` [3](#0-2) 

**Schema validation** (`account.graphqls` line 90) only enforces character set, not existence:
```graphql
alias: String @Pattern(regexp: "^[A-Z2-7]+$")
``` [4](#0-3) 

**Root cause:** `EntityServiceImpl` has no `@Cacheable` annotation or any result cache for alias lookups. The only cache in the GraphQL module (`CachedPreparsedDocumentProvider`) caches the *parsed query document*, not query results. [5](#0-4) [6](#0-5) 

**No application-level rate limiting exists in the GraphQL module.** The `ThrottleManagerImpl` / `ThrottleConfiguration` with bucket4j rate limiting exists only in the `web3` module, not in `graphql`. [7](#0-6) 

The only infrastructure-level mitigation is a Traefik `inFlightReq: amount: 5` per source IP in the Helm chart default, which limits *concurrent* requests per IP but not request rate, and is absent at the application layer. [8](#0-7) 

### Impact Explanation
Each request with a valid-format, non-existent alias causes one full `SELECT * FROM entity WHERE alias = ? AND deleted IS NOT TRUE` query against the database. With no result caching and no application-level rate limiting, an attacker sustaining even a modest request rate (e.g., 5 concurrent requests per IP continuously, or distributed across multiple IPs) generates a proportional, unbounded increase in database query load. This can degrade response times for all users sharing the same database connection pool and push DB CPU/IO above the 30% threshold relative to baseline. The `graphql.db.statementTimeout` of 10,000 seconds (not milliseconds — see docs) means queries are not quickly aborted. [9](#0-8) 

### Likelihood Explanation
No authentication or API key is required to reach `/graphql/alpha`. The query is a standard GraphQL POST with a single valid string argument. Any external user can craft the request with a fixed valid alias string (e.g., `ABCDEFGHIJKLMNOPQABCDEFGHIJKLMNOPQ` — 34 chars, all `[A-Z2-7]`). The attack is trivially repeatable with `curl` or any HTTP client in a loop. The `inFlightReq: 5` per-IP Traefik middleware is bypassable with multiple source IPs (VPN, proxies, cloud egress IPs) and does not enforce a per-second rate cap. [10](#0-9) 

### Recommendation
1. **Add result caching** in `EntityServiceImpl.getByAliasAndType()` using Spring's `@Cacheable` with a short TTL (e.g., 1–5 seconds) for both hit and miss results, similar to the caching already present in the `web3` module's entity cache (`expireAfterWrite=1s,maximumSize=10000`).
2. **Add application-level rate limiting** to the GraphQL module, mirroring the `ThrottleManagerImpl` pattern from the `web3` module, enforcing a per-IP or global requests-per-second cap.
3. **Add a length constraint** to the `alias` `@Pattern` directive or a separate `@Size` constraint to bound the input to the expected alias byte length (e.g., max 50 characters), reducing the decode cost.

### Proof of Concept
```bash
# Single-IP sustained load (5 concurrent, continuous)
for i in $(seq 1 1000); do
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input: {alias: \"ABCDEFGHIJKLMNOPQABCDEFGHIJKLMNOPQ\"}) { id } }"}' &
  # Keep 5 in flight
  [ $((i % 5)) -eq 0 ] && wait
done
```
Each iteration triggers `SELECT * FROM entity WHERE alias = \x... AND deleted IS NOT TRUE` with no cache hit. Monitor `pg_stat_activity` or DB CPU to observe sustained query load increase. The alias `ABCDEFGHIJKLMNOPQABCDEFGHIJKLMNOPQ` passes `^[A-Z2-7]+$` and decodes to a non-existent entity, so every request returns `null` with a full DB round-trip.
