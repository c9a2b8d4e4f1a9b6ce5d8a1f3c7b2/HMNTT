### Title
Unauthenticated Double Database Query Per Request in `getMessageByTopicAndSequenceRequest()` When `topicMessageLookup` Is Enabled

### Summary
When `config.query.topicMessageLookup` is enabled, every call to `GET /api/v1/topics/:topicId/messages/:sequenceNumber` unconditionally executes two sequential PostgreSQL queries per request with no rate limiting applied to unauthenticated callers. Any external user can sustain a moderate flood of valid requests, doubling the per-request database load and exhausting mirror-node database connection/query capacity.

### Finding Description
**Exact code path:**

`rest/topicmessage.js`, `getMessageByTopicAndSequenceRequest()`, lines 90‚Äì110:

```js
if (config.query.topicMessageLookup) {
    const {begin, end} = await getTopicMessageTimestampRange(topicId, seqNum); // Query 1
    params.push(begin, end);
    query += ` and ... >= $3 and ... < $4`;
}
res.locals[...] = await getMessage(query, params); // Query 2
```

Query 1 (`getTopicMessageTimestampRange`, lines 337‚Äì352) hits `topic_message_lookup` with a GiST index range containment operator (`@>`). Query 2 (`getMessage`, lines 322‚Äì329) hits `topic_message` with the narrowed timestamp bounds. Both are awaited sequentially; neither is cached or short-circuited before the second fires.

**Root cause:** The branch at line 103 is gated only on a server-side config flag, not on any per-request credential or quota. The `authHandler` middleware (`rest/middleware/authHandler.js`, lines 15‚Äì36) only sets a custom *response-limit* for authenticated users; it does not block or throttle unauthenticated requests. No `express-rate-limit` or equivalent middleware is registered for this route in `rest/server.js` (lines 67‚Äì144). The REST API has zero rate-limiting for unauthenticated callers on this endpoint.

**Why existing checks fail:**
- `validateGetSequenceMessageParams` (lines 31‚Äì44) only rejects malformed entity IDs and non-positive sequence numbers ‚Äî any valid `topicId` + `sequenceNumber` pair passes.
- `authHandler` is optional and non-blocking for unauthenticated requests (line 18‚Äì19: `if (!credentials) { return; }`).
- No connection-pool back-pressure or per-IP throttle exists in the REST layer.

### Impact Explanation
With `topicMessageLookup` enabled, each request to this endpoint consumes two database round-trips instead of one. An attacker sending *N* requests/second forces *2N* queries/second against the PostgreSQL backend. At moderate sustained rates (e.g., hundreds of requests/second from a single host or a small botnet), the database connection pool saturates, query latency spikes across all mirror-node API endpoints, and the service becomes unavailable or severely degraded ‚Äî affecting all consumers of the mirror node, not just the topic-message endpoint.

### Likelihood Explanation
The endpoint is publicly documented in `rest/api/v1/openapi.yml` (lines 1181‚Äì1209), requires no authentication, and accepts any valid topic ID and positive integer sequence number. A valid topic ID and at least one sequence number are trivially discoverable from the public ledger. The attacker needs no credentials, no special knowledge, and no brute-force: a simple HTTP loop with a known `topicId/1` is sufficient. The attack is repeatable, scriptable, and amplified by the 2√ó query multiplier.

### Recommendation
1. **Add per-IP rate limiting** to the REST API using `express-rate-limit` (or equivalent) applied globally or specifically to the `/:topicId/messages/:sequenceNumber` route before the handler fires.
2. **Cache the `getTopicMessageTimestampRange` result** (e.g., in Redis, which is already optionally wired in `server.js`) keyed on `(topicId, seqNum)` with a short TTL, so repeated requests for the same message do not re-hit the database.
3. **Fail fast**: if `getTopicMessageTimestampRange` returns no rows (throws `NotFoundError`), the second query is already skipped ‚Äî but this only helps for non-existent messages, not for valid ones under flood.
4. Consider requiring the `topicMessageLookup` optimization only when the caller is authenticated or the request passes a rate-limit gate.

```javascript
const getTopicMessageTimestampRange = async (topicId, seqNum) => {
  // Get the timestamp range from the topic message lookup table. Note explicitly casting topic id to bigint is required
  // to utilize the btree gist index on (topic_id, sequence_number_range)
  const query = `select timestamp_range
                    from topic_message_lookup
                    where topic_id = $1::bigint and sequence_number_range @> $2::bigint
                    order by sequence_number_range`;
  const params = [topicId, seqNum];

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    throw new NotFoundError();
  }

  return rows[0].timestamp_range;
};
```

```javascript
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};
```
