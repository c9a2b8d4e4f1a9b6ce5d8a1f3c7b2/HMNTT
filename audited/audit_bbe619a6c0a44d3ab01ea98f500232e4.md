### Title
Unbounded Alias Length in `accountAliasRegex` Enables Memory Exhaustion DoS via `/accounts/:idOrAliasOrEvmAddress/allowances/crypto`

### Summary
The `accountAliasRegex` in `rest/accountAlias.js` places no upper bound on the alias portion (`[A-Z2-7]+`), allowing an arbitrarily long string to pass validation. The OpenAPI path-parameter pattern that would enforce a bounded length is never applied to incoming requests because `validateRequests: false` is hardcoded in the validator middleware, and the validator itself is disabled by default in production. An unauthenticated attacker can therefore send repeated requests with megabyte-scale alias strings, causing proportional memory allocation in `base32.decode()` and linear CPU consumption in regex matching, leading to denial-of-service.

### Finding Description

**Step 1 – Unbounded regex**

`rest/accountAlias.js` line 10 defines:
```js
const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;
``` [1](#0-0) 

The quantifier `[A-Z2-7]+` has no upper bound. Any string of valid base32 characters of any length passes `AccountAlias.isValid()`. [2](#0-1) 

**Step 2 – Call chain from the controller**

`getAccountCryptoAllowances()` passes the raw path parameter directly to `EntityService.getEncodedId()`: [3](#0-2) 

`getEncodedId()` calls `AccountAlias.isValid()` (the unbounded regex) and then `AccountAlias.fromString()`: [4](#0-3) 

`AccountAlias.fromString()` calls `new AccountAlias(...)`, which calls `base32.decode(base32Alias)`: [5](#0-4) 

`base32.decode()` calls `rfc4648`'s `base32.parse()` with `{loose: true}`, allocating a `Uint8Array` proportional to the input length (≈ `input.length × 5/8` bytes): [6](#0-5) 

**Step 3 – Both OpenAPI guards are ineffective**

The OpenAPI spec *does* define a bounded pattern for the path parameter:
```
(?:[A-Z2-7]{8})*(?:[A-Z2-7]{2}|[A-Z2-7]{4,5}|[A-Z2-7]{7,8})
``` [7](#0-6) 

However, the OpenAPI validator middleware is configured with `validateRequests: false`, so path parameters are **never** validated against this pattern even when the validator is active: [8](#0-7) 

Furthermore, the validator is **disabled by default** in production (`openapi.validation.enabled` defaults to `false`): [9](#0-8) 

And the server only activates it when explicitly enabled or in test environments: [10](#0-9) 

### Impact Explanation
Each request with a 1 MB alias string causes the server to: (a) run the regex over 1 MB of input (linear CPU), (b) allocate a ~625 KB `Uint8Array` in `base32.decode()`. With tens of concurrent requests per second, heap memory grows rapidly. Node.js's single-threaded event loop is also blocked during synchronous regex execution and buffer allocation, degrading response latency for all users. No authentication is required to reach this endpoint.

### Likelihood Explanation
The endpoint is public and unauthenticated. The exploit requires only a standard HTTP client. Express 5 imposes no built-in URL length limit; without a correctly configured reverse proxy enforcing URL size limits, the full attack surface is exposed. The default production configuration (`openapi.validation.enabled: false`) means most deployments are vulnerable out of the box. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add a maximum length check** in `AccountAlias.isValid()` before regex evaluation, e.g. reject any input longer than 100 characters (the longest valid base32-encoded public key alias is ~56 characters).
2. **Fix the regex** to enforce an upper bound: change `[A-Z2-7]+` to `[A-Z2-7]{8,56}` (or the exact valid range for supported key types).
3. **Enable request validation** in the OpenAPI middleware: change `validateRequests: false` to `validateRequests: true` (or at minimum `{allowUnknownQueryParameters: true}`).
4. **Set a URL length limit** at the reverse proxy layer (e.g., nginx `large_client_header_buffers`) as a defense-in-depth measure.

### Proof of Concept
```bash
# Generate a 500 KB alias string of valid base32 characters
ALIAS=$(python3 -c "print('A' * 500000)")

# Send repeated requests to the allowances endpoint
for i in $(seq 1 100); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts/${ALIAS}/allowances/crypto" &
done
wait
```
Each request passes `accountAliasRegex`, enters `base32.decode()`, allocates ~312 KB, and the 100 concurrent requests allocate ~30 MB in a single burst. Sustained at higher rates, this exhausts the Node.js heap and causes OOM crashes or severe latency degradation for all API consumers.
