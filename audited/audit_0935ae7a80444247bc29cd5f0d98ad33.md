### Title
Unauthenticated IP Spoofing via Unvalidated `X-Real-IP` Header in `getClientIpAddress()` Enabling Log Attribution Manipulation

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` header with no validation, normalization, or source-trust check. In the Kubernetes deployment, Traefik is explicitly configured with `--entryPoints.web.forwardedHeaders.insecure`, meaning it forwards all client-supplied headers verbatim — including `X-Real-IP` — to the rosetta backend. An external attacker can therefore inject any string (including `::ffff:192.168.1.1`) as the logged IP, making log-based attribution of Rosetta API calls unreliable.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)       // line 64 — blindly trusts client header

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }

    return ipAddress   // returned and logged as-is, no parsing or validation
}
``` [1](#0-0) 

The value is logged directly in `TracingMiddleware` at line 52–53 with no further processing:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, ...)
``` [2](#0-1) 

**Why the proxy layer does not protect:**

The Kubernetes Traefik deployment is configured with:

```yaml
globalArguments:
  - --entryPoints.web.forwardedHeaders.insecure
  - --entryPoints.websecure.forwardedHeaders.insecure
``` [3](#0-2) 

`forwardedHeaders.insecure` instructs Traefik to trust and forward all `X-Forwarded-*` and `X-Real-IP` headers from **any** source, including untrusted external clients. Traefik does not strip or override `X-Real-IP` before passing the request to rosetta.

The docker-compose nginx proxy does overwrite `X-Real-IP` with `$remote_addr`:

```nginx
proxy_set_header X-Real-IP $$remote_addr;
``` [4](#0-3) 

However, the rosetta container also directly exposes port 5700, allowing bypass of nginx entirely:

```yaml
ports:
  - 5700:5700
``` [5](#0-4) 

**Root cause / failed assumption:** The code assumes `X-Real-IP` is always set by a trusted reverse proxy. Neither the application code nor the Kubernetes ingress enforces this.

### Impact Explanation

An attacker can log any arbitrary string — including `::ffff:192.168.1.1` — as the source IP for any Rosetta API call (e.g., `/block`, `/construction/submit`, `/network/status`). This:

- Breaks IP-based correlation between Rosetta logs and network/firewall/SIEM logs (which record the real TCP source IP in IPv4 form, while Rosetta logs show the injected IPv6-mapped form).
- Allows an attacker to frame another IP address in audit logs for all Rosetta API activity.
- Prevents forensic reconstruction of which client submitted which construction or network request.
- Severity: Medium — no direct fund theft, but directly undermines the integrity of the audit trail for all Rosetta transaction history.

### Likelihood Explanation

- **Precondition:** Kubernetes deployment (default, production-grade) with Traefik `forwardedHeaders.insecure` enabled (present in the shipped chart values), OR direct access to port 5700 in docker-compose.
- **Attacker capability:** Zero privilege required. Any HTTP client can set arbitrary request headers.
- **Repeatability:** 100% — trivially scriptable with `curl -H "X-Real-IP: ::ffff:192.168.1.1"`.
- **Detection difficulty:** High — the injected value is syntactically valid and indistinguishable from a legitimate proxy-set header in the log.

### Recommendation

1. **In `getClientIpAddress()`:** Parse and validate the header value using `net.ParseIP()`. If the result is an IPv4-mapped IPv6 address (`ip.To4() != nil` after parsing an IPv6), normalize it to its IPv4 form. Reject or fall back to `RemoteAddr` if the header value is not a valid IP.

2. **In Traefik:** Replace `forwardedHeaders.insecure` with an explicit trusted IP allowlist (`forwardedHeaders.trustedIPs`) scoped to the internal cluster CIDR, so external clients cannot inject proxy headers.

3. **In docker-compose:** Remove the direct port 5700 exposure for rosetta so all traffic is forced through nginx, which already overwrites `X-Real-IP`.

### Proof of Concept

**Kubernetes (Traefik) path:**
```bash
# Attacker sends request with spoofed IPv6-mapped IPv4 X-Real-IP
curl -H "X-Real-IP: ::ffff:192.168.1.1" \
     https://<rosetta-ingress>/network/list

# Rosetta log output (observed):
# ::ffff:192.168.1.1 GET /network/list (200) in 12ms

# Actual firewall/network log shows:
# SRC=203.0.113.42  (attacker's real IP)
```

**Docker-compose direct port path:**
```bash
# Bypass nginx, connect directly to rosetta
curl -H "X-Real-IP: ::ffff:10.0.0.1" \
     http://<host>:5700/network/list

# Rosetta log: ::ffff:10.0.0.1 GET /network/list (200) in 8ms
# Nginx access log: no entry (request bypassed nginx)
```

The two log sources now show irreconcilably different IP representations for the same request, making cross-source correlation impossible.
