### Title
Unauthenticated `/block` Endpoint Exhausts DB Connection Pool via Unbounded Concurrent `FindBetween()` Queries (DoS)

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` issues at least two database queries per request — one via `RetrieveBlock()` and one (or more, for large blocks) via `FindBetween()` — with no application-level rate limiting and no caching of transaction results. Because the Traefik middleware that provides per-IP in-flight and rate limiting is **disabled by default**, any unauthenticated attacker can flood the `/block` endpoint with concurrent requests referencing different valid blocks, holding DB connections open for up to the 20-second statement timeout each, exhausting the finite connection pool and denying service to all users.

### Finding Description

**Exact code path:**

`Block()` in `rosetta/app/services/block_service.go` (lines 47–74): [1](#0-0) 

Each call unconditionally invokes `FindBetween()` via `BaseService` (lines 54–59 of `base_service.go`): [2](#0-1) 

`FindBetween()` in the persistence layer executes a complex SQL query with correlated subqueries against `transaction`, `crypto_transfer`, and `staking_reward_transfer` tables, batched in groups of 2000 rows: [3](#0-2) 

The query itself uses two correlated subqueries per row: [4](#0-3) 

Each call to `FindBetween()` acquires a DB connection from the pool for the full duration of the query. The pool is configured with a default of **100 max open connections** and a **20-second statement timeout**: [5](#0-4) 

**Root cause — failed assumption:**

The design assumes an external layer (Traefik) will enforce rate limiting and in-flight request caps. However, the Helm chart sets `middleware: false` by default: [6](#0-5) 

The rate-limiting and in-flight middleware definitions exist but are gated on this flag: [7](#0-6) 

The application itself applies only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which throttle requests: [8](#0-7) 

There is **no caching** of `FindBetween()` results. The `entityCache` in `blockAPIService` caches only account aliases, not block transaction data: [9](#0-8) 

**No authentication** is required to call `/block`. Block indices are sequential and trivially enumerable.

### Impact Explanation

With 100 concurrent requests (matching the default `maxOpenConnections`), the entire DB connection pool is saturated. All subsequent requests — including from legitimate users and other endpoints — either fail immediately or queue indefinitely. The 20-second statement timeout means each attacker connection holds a pool slot for up to 20 seconds, so the attacker needs only to sustain ~5 requests/second to keep the pool permanently exhausted. This is a complete denial of service for the Rosetta API and any other component sharing the same DB pool.

### Likelihood Explanation

The attack requires no credentials, no special knowledge beyond knowing the service is running, and no sophisticated tooling — a simple script using `curl` or any HTTP client in a loop suffices. Valid block indices are sequential integers starting from 0, so the attacker does not need to guess them. The default deployment (Helm chart with `middleware: false`) ships without any protection. This is trivially repeatable and automatable.

### Recommendation

1. **Enable the Traefik middleware by default** — change `middleware: false` to `middleware: true` in `charts/hedera-mirror-rosetta/values.yaml` so the `inFlightReq` (5 per IP) and `rateLimit` (10 req/s) policies are active out of the box.
2. **Add application-level rate limiting** — implement a Go middleware (e.g., using `golang.org/x/time/rate`) so protection is not solely dependent on infrastructure configuration.
3. **Cache `FindBetween()` results** — add a short-lived (e.g., 5–30 second) in-process or Redis cache keyed on `(consensusStartNanos, consensusEndNanos)` to avoid redundant DB queries for the same block.
4. **Add a DB connection semaphore** — limit the number of concurrent `FindBetween()` calls at the application layer to a fraction of `maxOpenConnections`, preventing a single endpoint from monopolizing the pool.

### Proof of Concept

```bash
# Enumerate 200 valid block indices and fire concurrent /block requests
for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$i}}" &
done
wait
# Result: DB connection pool exhausted; all subsequent /block requests
# return errors or hang until the 20-second statement timeout expires.
```

func (s *blockAPIService) Block(
	ctx context.Context,
	request *rTypes.BlockRequest,
) (*rTypes.BlockResponse, *rTypes.Error) {
	block, err := s.RetrieveBlock(ctx, request.BlockIdentifier)
	if err != nil {
		return nil, err
	}

	if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
		return nil, err
	}

func (tr *transactionRepository) FindBetween(ctx context.Context, start, end int64) (
	[]*types.Transaction,
	*rTypes.Error,
) {
	if start > end {
		return nil, hErrors.ErrStartMustNotBeAfterEnd
	}

	db, cancel := tr.dbClient.GetDbWithContext(ctx)
	defer cancel()

	transactions := make([]*transaction, 0)
	for start <= end {
		transactionsBatch := make([]*transaction, 0)
		err := db.
			Raw(
				selectTransactionsInTimestampRangeOrdered,
				sql.Named("start", start),
				sql.Named("end", end),
			).
			Limit(batchSize).
			Find(&transactionsBatch).
			Error
		if err != nil {
			log.Errorf(databaseErrorFormat, hErrors.ErrDatabaseError.Message, err)
			return nil, hErrors.ErrDatabaseError
		}

		transactions = append(transactions, transactionsBatch...)

		if len(transactionsBatch) < batchSize {
			break
		}

		start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
	}
