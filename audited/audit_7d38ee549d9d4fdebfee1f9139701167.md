### Title
Missing EVM Address Length Validation Causes Unhandled `BufferUnderflowException` in `getByEvmAddressAndType()`

### Summary
`decodeEvmAddress()` in `GraphQlUtils.java` validates only that the input is valid hex, but imposes no length constraint on the decoded byte array. `getByEvmAddressAndType()` in `EntityServiceImpl.java` unconditionally calls `ByteBuffer.getInt()` and `getLong()` on the result, assuming a 20-byte EVM address. An unauthenticated attacker can supply a valid hex string shorter than 20 bytes with leading zero bytes, triggering a `BufferUnderflowException` that bypasses `CustomExceptionResolver` and surfaces as an `INTERNAL_ERROR` GraphQL response rather than a `ValidationError`.

### Finding Description

**Code path:**

`AccountController.account()` (line 52–54) passes the raw `evmAddress` string directly to `entityService.getByEvmAddressAndType()` with no pre-validation of length. [1](#0-0) 

`getByEvmAddressAndType()` calls `decodeEvmAddress()` and immediately wraps the result in a `ByteBuffer`, then reads 4 + 8 + 8 = 20 bytes unconditionally: [2](#0-1) 

`decodeEvmAddress()` only strips the `0x` prefix and decodes hex — it performs **no length check**: [3](#0-2) 

**Root cause:** The implicit assumption that `decodeEvmAddress()` always returns exactly 20 bytes is never enforced. The function's own tests confirm it can return arrays of arbitrary length (e.g., 5 bytes for `"0000AaAaAa"`): [4](#0-3) 

**Exploit flow:**

1. Attacker sends `"0x00000000"` (8 hex chars → 4 bytes).
2. `decodeEvmAddress()` succeeds, returning `[0x00, 0x00, 0x00, 0x00]`.
3. `buffer.getInt()` reads all 4 bytes → returns `0` (condition passes).
4. `buffer.getLong()` attempts to read 8 bytes from an empty buffer → throws `java.nio.BufferUnderflowException`.

Alternatively, `"0x000000000000000000000000"` (12 bytes) passes both the `getInt() == 0` and first `getLong() == 0` checks, then throws on the second `getLong()` at line 38. [5](#0-4) 

**Why existing checks fail:**

`CustomExceptionResolver.resolveToSingleError()` only handles `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`. `BufferUnderflowException` (a `RuntimeException`) matches none of these, so the method returns `null`: [6](#0-5) 

When `null` is returned, Spring GraphQL's default `SimpleDataFetcherExceptionHandler` takes over, logs the full exception with stack trace server-side, and returns an `INTERNAL_ERROR` classification to the client — distinct from the `ValidationError` returned for genuinely invalid hex.

### Impact Explanation

- **Error oracle**: The response type difference (`INTERNAL_ERROR` vs. `ValidationError`) allows an attacker to distinguish between "invalid hex" and "valid hex but wrong length," leaking internal parsing behavior.
- **Server-side log flooding / log pollution**: Each crafted request writes a full stack trace to server logs, enabling log flooding or obscuring legitimate errors.
- **Unhandled exception path**: The exception bypasses the application's own error-handling contract, violating the intended security boundary of `CustomExceptionResolver`.
- **Severity**: Medium — no authentication required, reliably reproducible, but does not directly expose data or allow code execution.

### Likelihood Explanation

- **No authentication required**: The GraphQL `account` query is publicly accessible.
- **Trivial to craft**: Any valid hex string of fewer than 20 bytes with leading zero bytes triggers the bug. Example: `"0x00000000"`.
- **Fully repeatable**: The condition is deterministic and not dependent on server state or timing.
- **Low noise**: A single malformed request is sufficient; no brute-forcing needed.

### Recommendation

Add an explicit length check in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
// In GraphQlUtils.decodeEvmAddress(), after decoding:
byte[] decoded = Hex.decodeHex(evmAddress);
if (decoded.length != 20) {
    throw new IllegalArgumentException(
        "EVM address must be exactly 20 bytes, got: " + decoded.length);
}
return decoded;
```

Or equivalently, guard in `getByEvmAddressAndType()`:

```java
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException("Invalid EVM address length: " + evmAddressBytes.length);
}
```

This ensures the exception is an `IllegalArgumentException`, which `CustomExceptionResolver` already handles correctly as a `ValidationError`.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }
