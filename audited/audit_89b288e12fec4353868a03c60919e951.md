### Title
Single-Node Network Injection via Unauthenticated HTTP Mirror Node Response in `buildNetworkFromMirrorNodes()`

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` accepts any network map containing at least one node, with no minimum node count enforced. Because the mirror node REST URL is fetched over plain HTTP with no TLS enforcement or certificate pinning, a network-level attacker (DNS poisoning, ARP spoofing, or MITM on the same segment) can inject a crafted response containing exactly one node pointing to an attacker-controlled gRPC server. The Hiero SDK client is then initialized exclusively against that server, causing all gossip transactions to be silently dropped.

### Finding Description

**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 74–131.

The HTTP client is constructed with no TLS configuration: [1](#0-0) 

The only guard against a degenerate network is: [2](#0-1) 

A response containing exactly one node with one valid `service_endpoint` passes this check (`len(network) == 1 > 0`) and is returned without further validation.

The resulting map is passed directly to the SDK: [3](#0-2) 

The operator-configured `mirrorRest` URL defaults to a plain `http://` scheme (no TLS): [4](#0-3) 

**Root cause:** No minimum node count check exists. The threshold is `> 0` instead of `>= N` (where N is some operationally meaningful minimum, e.g., 2 or 3). Combined with an unauthenticated HTTP transport, the response content is fully attacker-controllable.

**Exploit flow:**

1. Attacker poisons DNS for the mirror node REST hostname (or performs ARP spoofing / BGP hijack on the path), redirecting the pinger's HTTP GET to their own server.
2. Attacker's server returns a valid JSON `nodesEnvelope` with one entry: `node_account_id` set to a real account ID (e.g., `0.0.3`) and `service_endpoints` pointing to `attacker-ip:50211`.
3. `fetchMirrorNodeNetwork()` parses this, builds `network = {"attacker-ip:50211": AccountID{0,0,3}}`, and returns it — the `len(network) == 0` check does not fire.
4. `hiero.ClientForNetwork(netmap)` initializes the SDK with only the attacker's endpoint.
5. Every `cryptoTransfer.Execute(client)` call gossips the signed transaction exclusively to the attacker's gRPC server, which drops it.

**Why existing checks fail:**

- The `len(network) == 0` guard (line 127) only rejects a completely empty map; it does not enforce a minimum meaningful size.
- There is no TLS enforcement, certificate pinning, or response signing to authenticate the mirror node's reply.
- There is no cross-validation against a hardcoded or separately-sourced node list.

### Impact Explanation

All periodic transfer transactions submitted by the pinger are silently dropped. The pinger's core function — verifying liveness of the Hiero network — is completely subverted. The operator's tinybar balance is consumed by failed transactions (fees may still be deducted depending on SDK behavior). Monitoring alerts that depend on the pinger's success signal will not fire, masking a real network outage or enabling a targeted denial-of-service against the monitoring infrastructure.

### Likelihood Explanation

The attack requires network-level access between the pinger container and the mirror node REST endpoint. In Kubernetes/cloud deployments using plain HTTP internal services (as shown by the example URL `http://mirror-node:5551`), DNS poisoning via a compromised sidecar, a rogue CoreDNS entry, or ARP spoofing on a flat container network is achievable without OS-level privileges. The attack is repeatable: the pinger only calls `buildNetworkFromMirrorNodes()` once at startup, so a single poisoned response persists for the entire process lifetime.

### Recommendation

1. **Enforce a minimum node count:** After building the network map, reject responses with fewer than a configurable minimum (e.g., 2):
   ```go
   if len(network) < cfg.mirrorNodeClientMinNodes {
       return nil, false, fmt.Errorf("mirror node returned only %d node(s), minimum is %d", len(network), cfg.mirrorNodeClientMinNodes)
   }
   ```
2. **Enforce HTTPS / TLS for the mirror REST URL:** Validate that `cfg.mirrorRest` uses `https://` and configure the HTTP client with a strict TLS policy. Reject plain `http://` URLs in production.
3. **Cross-validate returned nodes against a known-good seed list:** For `network=other`, require that at least one returned node account ID matches a pre-configured allowlist.
4. **Periodically refresh the network map** and detect sudden drops in node count as an anomaly signal.

### Proof of Concept

**Preconditions:**
- Pinger deployed with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
- Attacker can poison DNS for `mirror-rest` (e.g., via a rogue entry in CoreDNS ConfigMap, or ARP spoofing on the pod network).

**Steps:**

1. Stand up a minimal HTTP server on attacker-controlled host that responds to `GET /api/v1/network/nodes` with:
   ```json
   {
     "nodes": [
       {
         "node_account_id": "0.0.3",
         "service_endpoints": [
           { "ip_address_v4": "ATTACKER_IP", "port": 50211 }
         ]
       }
     ],
     "links": { "next": null }
   }
   ```
2. Redirect DNS for `mirror-rest` to `ATTACKER_IP`.
3. Start (or restart) the pinger. `buildNetworkFromMirrorNodes()` fetches from the attacker's server.
4. `fetchMirrorNodeNetwork()` returns `{"ATTACKER_IP:50211": 0.0.3}` — passes the `len > 0` check.
5. Stand up a gRPC server on `ATTACKER_IP:50211` that accepts connections but drops all submitted transactions.
6. Observe that all pinger transfer attempts fail silently; the real Hiero network receives no transactions.
