### Title
Connection Pool Exhaustion via Unauthenticated API Flood Causes `readinessCheck()` Failure and Load Balancer Removal

### Summary
`readinessCheck()` in `rest/health.js` calls `pool.query()` on the same shared `global.pool` used by all regular API request handlers. Because there is no rate limiting on any API endpoint and the default pool size is only 10 connections, an unauthenticated attacker can flood concurrent API requests to exhaust all pool slots, causing `readinessCheck()` to time out waiting for a connection, throw a `DbError`, and cause `createTerminus` to return HTTP 503 on `/health/readiness`, triggering load balancer removal of the node.

### Finding Description

**Code path:**

`readinessCheck()` in `rest/health.js` (lines 22–33) calls `pool.query()` directly on `global.pool`:

```js
const readinessCheck = async () => {
  return pool
    .query(readinessQuery)   // acquires a slot from global.pool
    .catch((err) => {
      throw new DbError(err.message);  // any pool error → DbError → terminus 503
    })
    ...
};
``` [1](#0-0) 

`global.pool` is initialized in `rest/dbpool.js` with a hard cap of `config.db.pool.maxConnections` (default: **10**) and a `connectionTimeoutMillis` of **20,000 ms**:

```js
const poolConfig = {
  connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20000
  max: config.db.pool.maxConnections,                         // default 10
  statement_timeout: config.db.pool.statementTimeout,         // default 20000
};
global.pool = new Pool(poolConfig);
``` [2](#0-1) 

The same `global.pool` is used by every API handler (e.g., `pool.queryQuietly(...)` in `balances.js`, `transactions.js`, etc.) and by `readinessCheck()`. There is no dedicated connection reserved for health checks.

`createTerminus` is wired directly to `health.readinessCheck` with no fallback:

```js
createTerminus(server, {
  healthChecks: {
    '/health/readiness': health.readinessCheck,
    ...
  },
  ...
});
``` [3](#0-2) 

The middleware stack in `rest/server.js` contains no rate-limiting layer — only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, and optional metrics/cache handlers. No per-IP or global concurrency limit exists. [4](#0-3) 

**Root cause:** `readinessCheck()` competes for connections from the same bounded pool as untrusted user traffic. When all 10 slots are held, `pool.query()` in `readinessCheck()` is queued. After `connectionTimeoutMillis` (20 s) elapses without a free slot, node-postgres throws `Error: timeout exceeded when trying to connect`, which is caught and re-thrown as `DbError`, causing terminus to respond 503.

**Failed assumption:** The design assumes that regular API traffic will not saturate the pool long enough to starve the health check. With no rate limiting and a 10-connection default, this assumption fails under trivial concurrent load.

### Impact Explanation

When `readinessCheck()` throws, `createTerminus` returns HTTP 503 on `/health/readiness`. Kubernetes (or any load balancer using this endpoint as a readiness probe) will remove the pod from the service endpoints, effectively partitioning the node from the cluster without any network-level action. The attack is sustained as long as the flood continues, which requires no authentication and no special privileges.

**Severity: High** — complete removal of a node from the serving cluster, achievable by any internet-accessible client.

### Likelihood Explanation

The default pool size of 10 is documented and public. [5](#0-4) 

Sending 10–20 concurrent HTTP GET requests to any public API endpoint (e.g., `/api/v1/transactions`) is trivial with `curl --parallel`, `ab`, `wrk`, or any scripting language. Each request holds a connection for up to `statement_timeout` (20 s). The attacker only needs to maintain ≥10 concurrent in-flight requests continuously. No authentication, no special headers, no exploit code — just sustained concurrency. Repeatability is unlimited.

### Recommendation

1. **Reserve a dedicated connection for health checks**: Create a separate `pg.Pool` with `max: 1` exclusively for `readinessCheck()`, isolated from `global.pool`.
2. **Add rate limiting / concurrency limiting**: Apply a middleware (e.g., `express-rate-limit` or `p-limit`) to cap concurrent in-flight DB-bound requests per IP or globally.
3. **Increase pool size** or use PgBouncer in front of the REST service (the Helm chart already configures PgBouncer with `max_user_connections: 250` for `mirror_rest`, but the Node.js pool cap of 10 is the binding constraint).
4. **Add a timeout to `readinessCheck()` shorter than `connectionTimeoutMillis`** so terminus does not block for 20 s before returning 503.

### Proof of Concept

```bash
# Step 1: Flood the pool with 15 concurrent slow-ish requests (adjust endpoint as needed)
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>:5551/api/v1/transactions?limit=100" &
done

# Step 2: While flood is running, poll the readiness endpoint
watch -n 1 curl -s -o /dev/null -w "%{http_code}" http://<mirror-node-host>:5551/health/readiness
```

**Expected result:** Within ≤20 seconds of starting the flood, `/health/readiness` returns `503`. Kubernetes readiness probe fails; pod is removed from the service endpoint slice. The 503 persists for as long as the flood is maintained.
