### Title
Reference Equality Bypass in `findByBlockType()` via Hex-Encoded `Long.MAX_VALUE` Block Parameter

### Summary
`BlockType.of("0x7fffffffffffffff")` parses the hex string to `Long.MAX_VALUE` and returns a **new, non-singleton** `BlockType` instance, while `RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect `BlockType.LATEST`. Because the new instance is not the same object reference as the `LATEST` singleton, the LATEST branch is bypassed and `findByIndex(Long.MAX_VALUE)` is called instead of `findLatest()`, yielding an empty `Optional` and throwing `BlockNumberNotFoundException` to the caller.

### Finding Description

**Root cause — `BlockType.java` lines 37–56:**

`extractNumericBlock` always constructs a fresh `new BlockType(value, blockNumber)` for any numeric input, including `0x7fffffffffffffff`:

```java
long blockNumber = Long.parseLong(cleanedValue, radix);   // == Long.MAX_VALUE
return new BlockType(value, blockNumber);                  // new instance, not LATEST singleton
``` [1](#0-0) 

`BlockType.LATEST` is defined as:
```java
public static final BlockType LATEST = new BlockType("latest", Long.MAX_VALUE);
``` [2](#0-1) 

**Failed check — `RecordFileServiceImpl.java` lines 19–27:**

```java
if (block == BlockType.EARLIEST) { ... }
else if (block == BlockType.LATEST) {          // reference equality — fails for new instance
    return recordFileRepository.findLatest();
}
return recordFileRepository.findByIndex(block.number());  // called with Long.MAX_VALUE
``` [3](#0-2) 

No record file has `index = Long.MAX_VALUE`, so `findByIndex` returns `Optional.empty()`.

**Propagation — `ContractCallService.java` line 103–104:**

```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)));
``` [4](#0-3) 

The empty `Optional` triggers `BlockNumberNotFoundException` for every call that supplies `"0x7fffffffffffffff"` as the block parameter.

**Entry point — `ContractController.java` line 76–79:**

The `block` field of `ContractCallRequest` is deserialized via `@JsonCreator BlockType.of()`, so any unauthenticated POST to `/api/v1/contracts/call` with `"block": "0x7fffffffffffffff"` reaches this path. [5](#0-4) 

### Impact Explanation
Any unprivileged caller can make every `eth_call` / `eth_estimateGas` request fail with a block-not-found error by supplying `"0x7fffffffffffffff"` as the block tag. The value is the legitimate hex encoding of `Long.MAX_VALUE` and is syntactically valid per the Ethereum JSON-RPC spec (unsigned 64-bit hex). The result is a targeted, repeatable denial-of-service against the contract-call endpoint without any authentication or special privilege. Additionally, any code path that does not throw on empty `Optional` (e.g., future callers that use `orElse(null)`) would silently execute against a null/default block context, producing incorrect EVM state.

### Likelihood Explanation
The exploit requires only a single unauthenticated HTTP POST with a crafted `block` field. No wallet, token, or account is needed. The value `0x7fffffffffffffff` is a well-known sentinel in many Ethereum toolchains and is trivially discoverable by anyone reading the `BlockType` source or fuzzing block-parameter inputs. It is repeatable with 100% reliability.

### Recommendation
Replace the reference equality checks in `findByBlockType()` with value-based checks:

```java
public Optional<RecordFile> findByBlockType(BlockType block) {
    if (block.number() == 0L && block == BlockType.EARLIEST) {
        return recordFileRepository.findEarliest();
    } else if (block.number() == Long.MAX_VALUE) {   // catches both LATEST singleton and numeric equivalent
        return recordFileRepository.findLatest();
    }
    return recordFileRepository.findByIndex(block.number());
}
```

Alternatively, add a sentinel check inside `extractNumericBlock`: if the parsed `blockNumber == Long.MAX_VALUE`, return the `LATEST` singleton directly instead of constructing a new instance. A complementary guard in `BlockType.of()` to reject block numbers ≥ some reasonable maximum (e.g., the current chain height + 1) would also limit abuse.

### Proof of Concept

```bash
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "block": "0x7fffffffffffffff",
    "to": "0x0000000000000000000000000000000000000001",
    "gas": 50000,
    "data": "0x"
  }'
```

**Expected (correct) behavior:** executes against the latest block, returns a result.

**Actual behavior:** `BlockType.of("0x7fffffffffffffff")` creates a non-singleton `BlockType` with `number = Long.MAX_VALUE`; `findByBlockType()` skips the `== LATEST` branch; `findByIndex(Long.MAX_VALUE)` returns `Optional.empty()`; `orElseThrow(BlockNumberNotFoundException::new)` fires; the API returns a block-not-found error for every such request.
