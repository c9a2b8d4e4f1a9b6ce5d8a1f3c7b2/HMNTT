### Title
Lack of Request Deduplication and Ineffective Gas Throttle for Historical Block Calls Enables DB-Exhaustion DoS

### Summary
The `call()` endpoint in `ContractController` performs no request deduplication and has no response caching for historical block queries. The gas-based throttle is structurally ineffective for minimum-gas historical calls (consuming only 2 of 750,000 available tokens), leaving only a global 500 RPS bucket as protection. An unprivileged attacker can flood the endpoint with identical minimum-gas historical calls, each triggering multiple uncached database queries against historical state tables, exhausting DB connection pools and degrading availability for all users.

### Finding Description

**Code path:**

`ContractController.call()` (lines 38–51) calls `throttleManager.throttle(request)` then unconditionally invokes `contractExecutionService.processCall(params)`. [1](#0-0) 

`ThrottleManagerImpl.throttle()` applies two checks: a global RPS bucket (`tryConsume(1)`, default 500/s) and a gas bucket (`tryConsume(scaleGas(gas))`). [2](#0-1) 

**Root cause — gas throttle bypass for minimum-gas requests:**

`ThrottleProperties.scaleGas()` divides gas by `GAS_SCALE_FACTOR = 10_000`. For the minimum allowed gas of 21,000 (`@Min(21_000)` on `ContractCallRequest.gas`), this yields `Math.floorDiv(21000, 10000) = 2` tokens consumed per request. [3](#0-2) [4](#0-3) 

The gas bucket capacity is `scaleGas(7_500_000_000L) = 750,000` tokens. At 2 tokens per request, an attacker can make **375,000 requests** before hitting the gas limit. The binding constraint is therefore only the 500 RPS global bucket — a single attacker can consume all 500 slots. [5](#0-4) 

**No caching for historical state queries:**

For historical calls, `ContractCallContext.useHistorical()` returns `true` when `block != BlockType.LATEST`, and all state reads go through timestamp-scoped DB queries. [6](#0-5) 

`ContractStateServiceImpl.findStorageByBlockTimestamp()` bypasses the contract-state cache entirely and issues a direct DB query with no `@Cacheable` annotation. [7](#0-6) 

`EntityRepository.findActiveByIdAndTimestamp()` and `findActiveByEvmAddressAndTimestamp()` are also uncached historical queries hitting `entity` and `entity_history` tables. [8](#0-7) 

`AccountBalanceRepository.findHistoricalAccountBalanceUpToTimestamp()` executes a multi-CTE query joining `account_balance` and `crypto_transfer` tables, also uncached. [9](#0-8) 

Each `ContractCallContext` is created fresh per request via `ScopedValue`, so no cross-request state is shared. [10](#0-9) 

**Why existing checks fail:**

The gas throttle was designed to limit computational (EVM gas) cost, but historical calls are primarily I/O-bound. A minimum-gas historical call (gas=21,000) consumes only 2 of 750,000 gas tokens — the throttle provides no meaningful protection against DB-intensive historical queries. The 500 RPS global bucket is the only real guard, but it is not per-IP, so a single attacker can monopolize it entirely.

### Impact Explanation

An attacker sending 500 identical historical block calls per second (each with `gas=21,000` and a fixed historical block number) triggers up to 500 × N uncached DB queries per second (where N is the number of state reads per EVM execution). This can saturate the PostgreSQL connection pool, cause query timeouts (`statementTimeout=3000ms`), and make the historical state service unavailable to all other users. Because the throttle is global and not per-IP, legitimate traffic is starved without the attacker needing any special privilege.

### Likelihood Explanation

No authentication or API key is required to call `/api/v1/contracts/call`. The attack requires only a single HTTP client capable of sending 500 POST requests per second — trivially achievable with standard tooling (`wrk`, `ab`, `curl` in a loop). The attacker needs no knowledge of contract internals; any valid historical block number and a minimal `data` payload suffice. The attack is repeatable indefinitely and requires no state.

### Recommendation

1. **Add per-IP rate limiting** in addition to the global bucket, so a single source cannot consume the entire RPS budget.
2. **Cache historical call responses** keyed on `(block, to, data, from, gas, value)` with a short TTL (e.g., 5–30 seconds). Historical state is immutable, so caching is safe and effective.
3. **Fix the gas throttle for historical calls**: either apply a separate, lower per-second limit for historical requests, or account for the I/O cost of historical state loading independently of EVM gas.
4. **Deduplicate in-flight identical requests**: use a request-keyed `CompletableFuture` map so concurrent identical historical calls share a single DB execution.

### Proof of Concept

```bash
# Send 500 identical minimum-gas historical calls per second
# Replace BLOCK_NUMBER with any valid historical block number
# Replace CONTRACT_ADDRESS with any deployed contract address

wrk -t4 -c100 -d30s -s post.lua http://<mirror-node-host>/api/v1/contracts/call
```

`post.lua`:
```lua
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"block":"0x64","to":"0x0000000000000000000000000000000000000167","gas":21000,"data":"0x"}'
```

Expected result: within seconds, DB query latency spikes, `statementTimeout` errors appear in logs, and legitimate `/api/v1/contracts/call` requests begin returning 500 errors or timing out. The attacker's own requests continue to be accepted (consuming 2 gas tokens each from a 750,000-token bucket) while the global 500 RPS slot is fully occupied.
