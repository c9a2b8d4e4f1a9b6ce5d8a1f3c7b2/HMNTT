### Title
Unbounded Concurrent Subscription Fan-Out Enables Resource Exhaustion DoS in Shared Topic Listeners

### Summary
`CompositeTopicListener.listen()` imposes no limit on the number of concurrent subscribers. Each subscription allocates a per-subscriber buffer of up to 16,384 messages and occupies a slot on the global `Schedulers.boundedElastic()` thread pool. An unprivileged attacker opening thousands of concurrent gRPC subscriptions exhausts JVM heap and the shared scheduler, causing OOM or `RejectedExecutionException` errors that propagate to all legitimate subscribers.

### Finding Description

**Code path:**

`CompositeTopicListener.listen()` [1](#0-0)  delegates unconditionally to `SharedTopicListener.listen()`:

```java
return getSharedListener(filter)
    .onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
    .publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
``` [2](#0-1) 

**Root cause — two shared global resources are unboundedly consumed per subscriber:**

1. **Per-subscriber heap buffer**: `maxBufferSize` defaults to 16,384 entries. [3](#0-2)  With N concurrent subscribers, up to `N × 16,384 × sizeof(TopicMessage)` bytes are allocated on the heap simultaneously.

2. **Global bounded-elastic scheduler**: Every `publishOn(Schedulers.boundedElastic())` call enqueues tasks on the single JVM-wide scheduler. Reactor's `boundedElastic` has a hard task-queue cap (default 100,000). Saturating it causes `RejectedExecutionException` on all threads sharing the scheduler, including those serving legitimate subscribers.

**Why the existing check is insufficient:**

`BufferOverflowStrategy.ERROR` only terminates the *individual slow subscriber* when its own buffer fills; it does not limit how many subscribers can be created, nor does it prevent the aggregate memory and scheduler pressure from N simultaneous subscribers. [4](#0-3) 

**Shared Flux does not isolate resource consumption:**

Both `SharedPollingTopicListener` and `RedisTopicListener` use `.share()` to multicast one upstream to all subscribers. [5](#0-4) [6](#0-5)  While the upstream source is shared, the per-subscriber buffer and scheduler slot are *not* shared — they are allocated fresh for every call to `listen()`.

**No subscriber count tracking anywhere in the filter or listener chain:** [7](#0-6) 

### Impact Explanation

- **Memory exhaustion (OOM)**: 5,000 concurrent subscriptions × 16,384 buffer slots × ~200 bytes/message ≈ **~16 GB heap pressure**, crashing the JVM and terminating all active subscriptions.
- **Scheduler starvation**: Saturating `Schedulers.boundedElastic()` causes `RejectedExecutionException` propagated as terminal errors to every subscriber on that scheduler — a full service outage for all users.
- **Severity: Critical** — a single unauthenticated client can trigger a complete gRPC service outage.

### Likelihood Explanation

- No authentication is required at the listener layer; any gRPC client can call `subscribeTopic`.
- The attack is trivially scriptable: open thousands of long-lived streaming RPCs to the same topic ID.
- The attack is repeatable and requires no special knowledge beyond the public gRPC API.
- Default `maxBufferSize = 16384` and no connection-count limit make the threshold easy to reach on commodity hardware.

### Recommendation

1. **Add a global and per-topic subscriber cap** in `SharedTopicListener.listen()` using an `AtomicInteger` counter; reject new subscriptions with `Status.RESOURCE_EXHAUSTED` when the cap is exceeded.
2. **Add a per-IP / per-identity rate limit** at the gRPC interceptor layer before requests reach `CompositeTopicListener`.
3. **Expose `maxSubscribers` in `ListenerProperties`** so operators can tune the limit without code changes.
4. **Use a dedicated scheduler** (not the global `Schedulers.boundedElastic()`) for topic listener fan-out to isolate blast radius.

### Proof of Concept

```python
import grpc, threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

def flood(i):
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=...,  # any valid topic
        consensusStartTime=...
    )
    for _ in stub.subscribeTopic(req):  # blocks, holding subscription open
        pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(5000)]
for t in threads: t.start()
# Expected result: mirror node OOM crash or RejectedExecutionException
# propagated to all legitimate subscribers within seconds
```

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
