### Title
Unbounded Timestamp Range in Historical Hook Storage Query Enables DoS via Full Table Scan

### Summary
The `getHookStorageChange()` method in `HookServiceImpl.java` passes user-supplied timestamp bounds directly to a native SQL query using `DISTINCT ON (key)` with no server-side cap on the timestamp range. An unauthenticated caller can supply `timestamp=gte:0&timestamp=lte:9223372036854775807`, forcing PostgreSQL to scan the entire `hook_storage_change` table for a given `ownerId`/`hookId` pair across all time. Because `DISTINCT ON` requires processing all rows matching the WHERE clause before the `LIMIT` can be applied, the result-count cap does not prevent the expensive scan.

### Finding Description
**Code path:**

- `HooksController.java` line 104–130: `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` accepts `TimestampParameter[] timestamps` with only `@Size(max = 2)` — this limits the *count* of timestamp parameters to 2, not their *values*. [1](#0-0) 

- `HooksController.java` line 186: `Bound.of(timestamps, ...)` is built from raw user input with no range clamping. [2](#0-1) 

- `HookServiceImpl.java` lines 98–99: `timestampLowerBound` and `timestampUpperBound` are taken directly from the `Bound` object and forwarded to the repository. [3](#0-2) 

- `HookStorageChangeRepository.java` lines 15–39 / 41–63: Both native queries use `consensus_timestamp between :timestampLowerBound and :timestampUpperBound` with `SELECT DISTINCT ON (key)`. PostgreSQL must evaluate all rows satisfying the WHERE clause to compute `DISTINCT ON` before the `LIMIT` from `Pageable` is applied. [4](#0-3) 

**Root cause:** No maximum timestamp window is enforced anywhere in the controller, DTO, or service layer. The `@Max(MAX_LIMIT)` guard on `limit` only caps returned rows, not the number of rows the database must scan to satisfy `DISTINCT ON`. [5](#0-4) 

### Impact Explanation
A single HTTP request with `timestamp=gte:0&timestamp=lte:9223372036854775807` forces a full historical scan of `hook_storage_change` for the targeted `(owner_id, hook_id)`. On a busy chain with many storage-change events, this translates to high CPU and I/O on the database node. Repeated concurrent requests from one or more clients can exhaust database connection pools or saturate disk I/O, causing service degradation or outage for all users of the mirror node REST API.

### Likelihood Explanation
The endpoint is a public, unauthenticated REST API — no credentials or special role are required. The attack requires only knowledge of any valid `ownerId`/`hookId` pair (discoverable via the `GET /api/v1/accounts/{ownerId}/hooks` endpoint). The exploit is trivially repeatable with a simple HTTP client and can be parallelized.

### Recommendation
1. **Enforce a maximum timestamp window** in `hookStorageChangeRequest()` or inside `Bound`: reject or clamp requests where `upperBound - lowerBound` exceeds a configurable maximum (e.g., 7 days in nanoseconds).
2. **Require at least one bound to be recent**: reject queries whose upper timestamp bound is more than N days in the past or whose range spans more than a configured window.
3. **Add a query timeout** at the JDBC/datasource level as a defense-in-depth measure.
4. Consider adding authentication/rate-limiting to historical query endpoints.

### Proof of Concept
```
# Step 1: Discover a valid hookId for a known account
GET /api/v1/accounts/0.0.1234/hooks

# Step 2: Issue the unbounded historical storage query
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&timestamp=lte:9223372036854775807&limit=100

# Repeat Step 2 in a loop (e.g., 20 concurrent threads) to exhaust DB resources.
# The DISTINCT ON query will scan every row in hook_storage_change
# for owner_id=<resolved> and hook_id=1 regardless of the LIMIT clause.
```
