### Title
GraphQL Parser Exhaustion via Cache-Bypassing Max-Token Queries with No Application-Level Rate Limiting

### Summary
The `GraphQlConfiguration` static initializer sets `maxTokens(1000)` and `maxWhitespaceTokens(1000)` as upper bounds, meaning queries just below these limits are fully processed by the lexer/parser. The `CachedPreparsedDocumentProvider` caches parsed documents keyed on the exact query string, so an attacker sending many syntactically unique but structurally equivalent queries (e.g., varying aliases) bypasses the cache entirely. The `graphql` module has no application-level rate limiting — unlike the `web3` module — allowing an unauthenticated attacker to flood the endpoint with maximum-cost parse operations.

### Finding Description

**Code path 1 — Parser limits (upper bound, not rejection before work):** [1](#0-0) 

`maxTokens(1000)` and `maxWhitespaceTokens(1000)` are enforced by the graphql-java lexer by counting tokens as they are produced. The parser throws only *after* processing up to the limit. A query with 999 non-whitespace tokens and 999 whitespace tokens is fully lexed — ~1998 total lexer steps — before being accepted. A typical query has 20–80 tokens. The per-query parsing cost ratio is therefore 12–100×.

**Code path 2 — Parse cache keyed on exact query string:** [2](#0-1) 

The cache key is `executionInput.getQuery()` — the raw query string. Any character-level difference (e.g., a different field alias `f1`, `f2`, … `fN`) produces a cache miss and forces a full re-parse. The cache holds at most 1000 entries; an attacker cycling through 1001+ unique queries keeps the cache permanently cold for their payloads.

**Code path 3 — No rate limiting in the graphql module:** [3](#0-2) 

The `graphql` module's only production configuration beans are in `GraphQlConfiguration`. There is no `ThrottleConfiguration`, no `RateLimiter`, and no per-IP request counter. The `web3` module has `ThrottleConfiguration` with bucket4j rate limiting, but this is entirely absent from the `graphql` module. [4](#0-3) 

**Code path 4 — Infrastructure-only mitigation is insufficient:** [5](#0-4) 

The Traefik `inFlightReq: amount: 5` middleware limits *concurrent* requests per source IP, not the *rate*. An attacker with 5 concurrent connections sending max-token queries continuously can sustain 5 simultaneous full-cost parses indefinitely. With multiple source IPs (distributed attack), the effective limit is 5 × N IPs. This middleware is also only present in the Helm chart deployment path and is absent in non-Kubernetes deployments.

**Root cause:** The failed assumption is that `maxTokens(1000)` *prevents* expensive parsing. It does not — it bounds the cost per query but does not limit how many such maximum-cost queries can be submitted per second. Without application-level rate limiting, the attacker controls the aggregate parsing throughput.

### Impact Explanation

A query at the token/whitespace limit costs ~12–100× more CPU to parse than a typical query. With 5 concurrent connections (the per-IP Traefik limit), an attacker can sustain 5 × 100× = 500 equivalent-query units of parsing load continuously. On a node handling a normal mix of requests, this can easily exceed a 30% CPU increase. The `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` checks run *after* parsing and do not reduce the lexer/parser cost. The attack causes resource exhaustion without executing any database queries, making it cheap for the attacker and expensive for the server.

### Likelihood Explanation

No authentication is required. The GraphQL endpoint is publicly accessible at `/graphql/alpha`. Crafting a max-token query requires only knowledge of the GraphQL schema (publicly documented) and a script to vary aliases. The attack is trivially repeatable and automatable. A single attacker with a modest connection pool can sustain the attack indefinitely.

### Recommendation

1. Add application-level rate limiting to the `graphql` module, mirroring the `ThrottleConfiguration` pattern from `web3`, with a per-IP request-per-second bucket.
2. Alternatively, add a `WebGraphQlInterceptor` that enforces a per-IP token bucket before the query reaches the parser.
3. Consider reducing `maxTokens` and `maxWhitespaceTokens` further (e.g., to 200–300), consistent with the actual schema complexity.
4. Consider a cost-based cache eviction strategy or a request-body hash check to detect cache-bypass patterns.

### Proof of Concept

```bash
# Generate N unique max-token queries by varying aliases
python3 - <<'EOF'
import requests, threading, itertools

URL = "http://<graphql-host>/graphql/alpha"

def make_query(n):
    # Build a query with ~999 tokens using repeated aliased fields
    # Each "alias: field" pair = ~3 tokens; 333 pairs ≈ 999 tokens
    # Whitespace between each pair adds whitespace tokens
    aliases = " ".join(f"f{i}_{n}: balance" for i in range(333))
    return f"{{ account(input: {{entityId: {{shard:0,realm:0,num:2}}}}) {{ {aliases} }} }}"

def flood(thread_id):
    for i in itertools.count():
        q = make_query(thread_id * 100000 + i)  # unique per iteration
        requests.post(URL,
            json={"query": q},
            headers={"Content-Type": "application/json"},
            timeout=10)

threads = [threading.Thread(target=flood, args=(t,)) for t in range(5)]
for t in threads: t.start()
for t in threads: t.join()
EOF
```

Each request is a unique string (cache miss), contains ~999 non-whitespace tokens and ~999 whitespace tokens (at the configured limits), and forces a full lexer/parser pass. Five concurrent threads sustain maximum parsing load continuously with no application-level throttle to stop them.

```java
@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
```
