### Title
Unhandled Plain `Error` on `multipleEvmAddressMatch` Causes HTTP 500 in `getAccountTokenAllowances`

### Summary
In `rest/service/entityService.js`, `getEntityIdFromEvmAddress()` throws a plain `new Error(...)` (not a `RestError` subclass) when multiple alive entities share the same EVM address. The error middleware in `rest/middleware/httpErrorHandler.js` has no handler for plain `Error`, so it falls through to the default `INTERNAL_ERROR` (500) status code. Any unprivileged caller who supplies an EVM address that matches this DB state receives an opaque 500 response instead of a defined API error, breaking smart contract integrations that depend on deterministic allowance responses.

### Finding Description

**Exact code path:**

1. `rest/controllers/tokenAllowanceController.js` line 69 calls `EntityService.getEncodedId(req.params[...])`.
2. `rest/service/entityService.js` `getEncodedId()` (line 124) calls `this.getEntityIdFromEvmAddress(entityId, requireResult)` when the input is an EVM address.
3. `getEntityIdFromEvmAddress()` lines 98–100:
   ```js
   } else if (rows.length > 1) {
     logger.error(`Incorrect db state: ...`);
     throw new Error(EntityService.multipleEvmAddressMatch);  // plain Error, NOT RestError
   }
   ```
4. The `catch` block in `getEncodedId()` (lines 128–134) only intercepts `InvalidArgumentError` to rethrow it as a param error; all other exceptions — including this plain `Error` — are re-thrown unchanged.
5. `rest/middleware/httpErrorHandler.js` `handleError()` (lines 17–25) checks only `NotFoundError`, `InvalidArgumentError`, `RangeError`, `DbError`, and `HttpError`. A plain `Error` matches none of these, so `statusCode` stays at `defaultStatusCode = httpStatusCodes.INTERNAL_ERROR` (500).
6. `shouldReturnMessage()` (line 57) returns `true` only for client errors (4xx) or `BAD_GATEWAY`. A 500 is not a client error, so the actual error message is suppressed and a generic 500 body is returned.

**Root cause:** The thrown object is `new Error(...)`, not a subclass of `RestError`. The error handler has no catch-all for unexpected DB integrity errors, so they silently become 500s.

**Failed assumption:** The code assumes the `multipleEvmAddressMatch` state is purely an ops/monitoring concern (logged via `logger.error`), but the thrown plain `Error` escapes into the HTTP response layer with no defined mapping.

### Impact Explanation
Any smart contract system or off-chain component that calls `/accounts/:evmAddress/allowances/tokens` and receives a 500 instead of a 404 or a structured error cannot distinguish "address not found" from "server broken." Systems that treat non-200 as "allowance is zero" or retry indefinitely will either under-approve or over-approve token operations. The scope is limited to the mirror node read path (no funds are directly moved), but incorrect allowance state assumptions in dependent contracts qualify as unintended smart contract behavior at the medium layer-1/2 level.

### Likelihood Explanation
The precondition — multiple alive entities sharing one EVM address — is described as "Incorrect db state" and should not occur under normal operation. However:
- A bug in the mirror node ingestion pipeline could produce this state transiently.
- A network-level actor could submit transactions crafted to create multiple entities with the same EVM address if the Hedera network permits it (e.g., during contract creation edge cases), which the mirror node would faithfully ingest.
- Once the state exists, **any** unprivileged user who knows or brute-forces the affected EVM address can reliably and repeatedly trigger the 500 with a simple GET request — no authentication, no special headers required.

### Recommendation
Replace the plain `Error` throw with a typed error that the middleware can map to a defined HTTP status. Two options:

1. **Preferred — map to 500 with a `RestError` subclass** so the process does not crash and the error is contained:
   ```js
   // in entityService.js line 100
   throw new RestError(EntityService.multipleEvmAddressMatch);
   ```
   Then add a handler in `httpErrorHandler.js` for `RestError` (or its subclass) that returns 500 with a safe generic message, preventing leakage of internal state details while still being a controlled response.

2. **Alternative — treat as 404/409** if the policy is that an ambiguous address should not resolve:
   ```js
   throw new NotFoundError('Ambiguous EVM address');
   ```
   This gives callers a deterministic 404 they can handle.

Additionally, add a database-level unique constraint on `(evm_address)` where `deleted <> true` to prevent the inconsistent state from arising.

### Proof of Concept

**Precondition:** The mirror node database contains two rows in the `entity` table with `deleted = false` and identical `evm_address` values, e.g. `0xaabbccdd...` (20 bytes).

**Steps:**
1. Identify or induce the duplicate EVM address state in the DB (e.g., via ingestion bug or crafted network transactions).
2. Send an unauthenticated HTTP GET request:
   ```
   GET /api/v1/accounts/0xaabbccddaabbccddaabbccddaabbccddaabbccdd/allowances/tokens
   ```
3. The request reaches `getAccountTokenAllowances()` → `getEncodedId()` → `getEntityIdFromEvmAddress()`.
4. The DB query returns 2 rows; `rows.length > 1` is true.
5. `throw new Error('Multiple alive entities matching evm address')` is raised.
6. `httpErrorHandler.handleError` receives a plain `Error`; no branch matches; `statusCode = INTERNAL_ERROR`.
7. Response: `HTTP 500` with body `{"_status":{"messages":[{"message":"Internal error"}]}}`.

// Error middleware which formats thrown errors and maps them to appropriate http status codes
// next param is required to ensure express maps to this middleware and can also be used to pass onto future middleware
const handleError = async (err, req, res, next) => {
  var statusCode = defaultStatusCode;

  if (err instanceof NotFoundError) {
    statusCode = httpStatusCodes.NOT_FOUND;
  } else if (err instanceof InvalidArgumentError || err instanceof RangeError) {
    statusCode = httpStatusCodes.BAD_REQUEST;
  } else if (err instanceof DbError) {
    statusCode = httpStatusCodes.SERVICE_UNAVAILABLE;
  } else if (err instanceof HttpError) {
    statusCode = new StatusCode(err.statusCode, err.msg);
  }
