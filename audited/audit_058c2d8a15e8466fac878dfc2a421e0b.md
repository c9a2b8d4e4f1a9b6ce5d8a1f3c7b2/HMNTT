### Title
Floor Division in `scaleGas()` Allows Disproportionate EVM Execution Per Throttle Token

### Summary
The `scaleGas()` method in `ThrottleProperties.java` uses `Math.floorDiv`, causing all gas values within a band of 10,000 to consume the same number of throttle tokens. An unprivileged attacker can craft requests with gas values just below a multiple of `GAS_SCALE_FACTOR` (e.g., gas=29,999) to consume the same token count as gas=21,000 (the minimum), while receiving ~43% more EVM execution capacity per token — exceeding the 30% CPU increase threshold without sending additional requests.

### Finding Description

**Exact code location:**

`ThrottleProperties.java`, `scaleGas()`, lines 42–47: [1](#0-0) 

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {
        return 0L;
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);  // floor division
}
```

`GAS_SCALE_FACTOR = 10_000L` [2](#0-1) 

This scaled value is consumed directly from the `gasLimitBucket` in `throttle()`: [3](#0-2) 

**Root cause — floor division banding:**

`Math.floorDiv` maps entire ranges of gas values to the same token cost:

| Gas value | Tokens consumed | Gas/token ratio |
|-----------|----------------|-----------------|
| 21,000    | 2              | 10,500          |
| 29,999    | 2              | 14,999          |
| 30,000    | 3              | 10,000          |
| 39,999    | 3              | 13,333          |

An attacker using gas=29,999 consumes **2 tokens** — identical to gas=21,000 — but receives **29,999 gas** of EVM execution capacity, a **~43% increase** in EVM work per token.

**Note on the question's gas=19,999 example:** `ContractCallRequest.gas` carries `@Min(21_000)`: [4](#0-3) 

The `@Valid` annotation on the controller endpoint enforces this before `throttle()` is ever called: [5](#0-4) 

So gas=19,999 is rejected. However, gas=29,999 and gas=39,999 are fully valid and still exploit the same floor-division band effect.

**Exploit flow:**

1. Attacker sends 500 req/s (the global `requestsPerSecond` limit) with gas=21,000 → 500 × 21,000 = **10.5M gas/s** of EVM work.
2. Attacker switches to gas=29,999 (same 2 tokens/request from the gas bucket) → 500 × 29,999 = **~15M gas/s** of EVM work.
3. Delta: +43% EVM execution per second with zero additional requests.

**Why existing checks are insufficient:**

- The `rateLimitBucket` (500 req/s) is a global singleton: [6](#0-5) 
  It limits request count, not gas throughput. It does not compensate for the floor-division efficiency gain.

- The `gasLimitBucket` capacity is `scaleGas(7_500_000_000)` = 750,000 tokens/s: [7](#0-6) 
  At 500 req/s × 2 tokens/req = 1,000 tokens/s consumed, the gas bucket is nowhere near its limit and provides no protection here. The rate limit is the sole binding constraint, and the floor-division gain passes through it unchecked.

- There is no per-IP rate limiting; both buckets are process-wide singletons, so a single attacker IP can consume the entire global budget.

### Impact Explanation

The gas throttle's stated purpose is to bound EVM CPU consumption per second. The floor-division band allows a single unprivileged client to extract up to ~43% more EVM execution per second than the minimum-gas baseline, without triggering any throttle. For a node already running near capacity, this translates directly to a >30% increase in EVM CPU load. The effect is repeatable and sustained as long as the attacker keeps sending requests at the rate limit.

### Likelihood Explanation

No authentication, special accounts, or privileged access is required. The `GAS_SCALE_FACTOR` value (10,000) is visible in the public source code. Any external caller can set `gas=29,999` in a standard JSON-RPC `eth_call`-style POST to `/api/v1/contracts/call`. The exploit requires only knowledge of the constant and a script that fires requests at the rate limit — well within the capability of any motivated external party.

### Recommendation

Replace `Math.floorDiv` with ceiling division so that gas values just below a band boundary consume the *higher* token count, not the lower one:

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {
        return 0L;
    }
    // ceiling division: Math.ceilDiv (Java 18+) or (gas + GAS_SCALE_FACTOR - 1) / GAS_SCALE_FACTOR
    return Math.ceilDiv(gas, GAS_SCALE_FACTOR);
}
```

With ceiling division, gas=29,999 → 3 tokens (same as gas=30,000), eliminating the band-exploitation advantage. Also consider enforcing a minimum token cost of 1 for any gas > 0 to close the `gas ≤ 10,000 → 0 tokens` free-ride path (though that path is currently blocked by `@Min(21_000)` on the request).

### Proof of Concept

```python
import requests, time

URL = "http://<mirror-node>/api/v1/contracts/call"
HEADERS = {"Content-Type": "application/json"}

# Baseline: minimum valid gas — 2 tokens each, 21,000 gas each
baseline_payload = {
    "to": "0x000000000000000000000000000000000000dead",
    "gas": 21000,
    "data": "0x" + "5b" * 20000  # JUMPDEST loop to burn gas
}

# Exploit: same 2 tokens each, but 29,999 gas each (~43% more EVM work)
exploit_payload = {
    "to": "0x000000000000000000000000000000000000dead",
    "gas": 29999,
    "data": "0x" + "5b" * 29000
}

# Fire at the 500 req/s rate limit
for payload in [baseline_payload, exploit_payload]:
    start = time.time()
    for _ in range(500):
        requests.post(URL, json=payload, headers=HEADERS)
    elapsed = time.time() - start
    print(f"gas={payload['gas']}: 500 requests in {elapsed:.2f}s")
    # Observe node CPU: exploit run will show ~43% higher EVM execution load
    time.sleep(2)
```

**Expected result:** Both runs consume 1,000 gas-bucket tokens (2 per request × 500 requests) and 500 rate-limit tokens. The exploit run delivers 29,999 × 500 = ~15M gas of EVM work vs 21,000 × 500 = ~10.5M gas for the baseline — a **~43% increase in node CPU load** with no additional requests and no throttle rejection.
