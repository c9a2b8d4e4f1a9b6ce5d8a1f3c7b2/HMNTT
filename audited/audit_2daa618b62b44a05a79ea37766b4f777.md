### Title
Cache Bypass for Address Book Files Due to Missing Timestamp Normalization Causes Unbounded DB + Protobuf Parse Per Request

### Summary
`SystemFileLoader.load()` applies `roundDownToHour()` timestamp normalization only for exchange-rate, fee-schedule, and simple-fee-schedule files, but not for `addressBookFile101` or `addressBookFile102`. Because the caller (`FileReadableKVState`) always passes `Utils.getCurrentTimestamp()` — the current wall-clock time in nanoseconds — as the cache key, every request produces a unique `CacheKey` and unconditionally misses the Caffeine cache. Each miss triggers a `fileDataRepository.getFileAtTimestamp()` DB query followed by `NodeAddressBook.PROTOBUF.parse()`, and any unprivileged user who can trigger address-book file reads can flood the service with these operations.

### Finding Description

**Exact code path:**

`FileReadableKVState.readFromDataSource` (line 54–57):
```java
final var currentTimestamp = getCurrentTimestamp();   // Instant.now() in nanos
if (systemFileLoader.isSystemFile(key)) {
    return systemFileLoader.load(key, currentTimestamp);
}
``` [1](#0-0) 

`Utils.getCurrentTimestamp()` (line 57–60):
```java
public static long getCurrentTimestamp() {
    final var now = Instant.now();
    return DomainUtils.convertToNanos(now.getEpochSecond(), now.getNano());
}
``` [2](#0-1) 

`SystemFileLoader.load()` (lines 109–118): the `roundDownToHour` branch covers only three file IDs; `addressBookFile101` and `addressBookFile102` fall through with the raw nanosecond timestamp:
```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp);  // address book NOT here
}
final var cacheKey = new CacheKey(fileId, consensusTimestamp);
``` [3](#0-2) 

Because `Instant.now()` in nanoseconds is unique for every request, `cache.get(cacheKey, File.class)` always returns `null`, and execution falls through to `loadFromDB` → `loadWithRetry`: [4](#0-3) 

Inside `loadWithRetry`, for every cache miss:
1. `fileDataRepository.getFileAtTimestamp(fileId, nanoSeconds.get())` — a live DB query.
2. `codec.parse(bytes.toReadableSequentialData())` — `NodeAddressBook.PROTOBUF.parse()` allocates a full `NodeAddressBook` object (discarded immediately after validation). [5](#0-4) 

The `systemFile` Caffeine cache is configured `maximumSize=20` — it fills instantly with 20 distinct nanosecond-keyed entries and evicts continuously, providing zero reuse benefit for address-book files: [6](#0-5) 

**Root cause:** The failed assumption is that the `defaultSystemFileCacheManager` cache will absorb repeated reads of the address book. It cannot, because the cache key embeds a nanosecond wall-clock timestamp that is unique per request, and no normalization is applied to collapse nearby timestamps into a shared key.

### Impact Explanation
Every request that causes the EVM to read `addressBookFile101` or `addressBookFile102` unconditionally issues a PostgreSQL query and allocates a `NodeAddressBook` protobuf object. Under the default throttle of 500 req/s, this translates to 500 DB queries/s and 500 protobuf parse allocations/s sustained indefinitely for address-book reads alone. Under concurrent load the short-lived `NodeAddressBook` objects accumulate in the young generation faster than minor GC can collect them, increasing GC pause frequency and, at high concurrency, promoting objects to old-gen and causing heap pressure. The DB connection pool is the harder limit: sustained address-book-triggering traffic can saturate the pool and degrade or deny service to all other queries.

### Likelihood Explanation
No privilege is required. Any caller who can submit a `eth_call` or `eth_estimateGas` request that internally reads the address book (e.g., a precompile or system-contract call that resolves node addresses) can trigger this path. The attack requires no special knowledge beyond knowing which contract call causes an address-book read. It is trivially repeatable and scriptable.

### Recommendation
Apply the same `roundDownToHour` (or a coarser granularity such as round-down-to-minute) normalization to `addressBookFile101` and `addressBookFile102` before constructing the `CacheKey`, exactly as is already done for exchange-rate and fee-schedule files:

```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)
        || fileId.equals(addressBookFile101Id)      // add
        || fileId.equals(addressBookFile102Id)) {   // add
    consensusTimestamp = roundDownToHour(consensusTimestamp);
}
```

This collapses all requests within the same hour to a single cache entry, reducing DB queries and protobuf allocations to at most one per hour per address-book file.

### Proof of Concept
1. Identify any contract call that causes the EVM to read `FileID(0,0,101)` (e.g., a precompile that resolves node addresses).
2. Send 500 such requests per second (within the default throttle limit) using a script:
   ```bash
   for i in $(seq 1 500); do
     curl -s -X POST http://mirror-node-web3:8545 \
       -H 'Content-Type: application/json' \
       -d '{"jsonrpc":"2.0","method":"eth_call","params":[{...address-book-triggering call...},"latest"],"id":1}' &
   done
   ```
3. Observe via DB metrics that `getFileAtTimestamp` is called ~500 times/s for entity ID 101.
4. Observe via JVM heap metrics that `NodeAddressBook` allocation rate is ~500 objects/s with no cache hits recorded for the `systemFile` cache.
5. Sustain for several minutes and observe increasing GC pause times and DB connection pool saturation.
