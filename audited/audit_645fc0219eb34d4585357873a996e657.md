### Title
Cache Stampede via Synchronized TTL Expiry in CachedPreparsedDocumentProvider

### Summary
The `CachedPreparsedDocumentProvider` uses a Caffeine `AsyncCache` configured with `expireAfterWrite=1h,maximumSize=1000`. Because all entries written in the same time window share the same absolute expiry instant, an unprivileged attacker can deliberately fill the cache with up to 1000 distinct queries, then replay all 1000 simultaneously just after the 1-hour mark, forcing concurrent re-parsing of every entry with no rate-limiting guard in the GraphQL module.

### Finding Description
**Exact locations:**
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CacheProperties.java`, line 15: default spec `"expireAfterWrite=1h,maximumSize=1000,recordStats"`.
- `graphql/src/main/java/org/hiero/mirror/graphql/cache/CachedPreparsedDocumentProvider.java`, lines 19–20: cache built directly from that spec via `Caffeine.from(properties.getQuery()).buildAsync()`.
- Same file, lines 24–27: `getDocumentAsync()` calls `cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput))`.

**Root cause:** Caffeine's `AsyncCache.get()` is atomic *per key* — it prevents a thundering herd for the *same* query string. However, it provides no cross-key coordination. When 1000 *different* keys all expire at the same wall-clock instant (because they were all written within the same second), 1000 independent `parseAndValidateFunction` invocations are submitted concurrently to `ForkJoinPool.commonPool()`. There is no rate limiter, no per-IP throttle, and no request-queue depth limit anywhere in the GraphQL module.

**Exploit flow:**
1. Attacker sends 1000 syntactically distinct but valid queries (trivially achieved by varying a field alias or argument name) at time T. All 1000 are parsed, validated, and stored in the cache with expiry T+3600s.
2. Attacker waits until T+3600s+ε.
3. Attacker replays all 1000 queries in a single burst (e.g., via 1000 concurrent HTTP/2 streams or rapid HTTP/1.1 keep-alive requests). Caffeine's lazy expiry means every entry is stale on first access; `cache.get()` finds no live entry for any key and submits 1000 `parseAndValidateFunction` tasks simultaneously.
4. Each task runs the full GraphQL parse + validation pipeline (lexer, parser, rule-depth check, type-check) on the ForkJoinPool, saturating available CPU threads.

**Why existing checks are insufficient:**
- `maxCharacters(10000)`, `maxTokens(1000)`, `maxRuleDepth(100)` (`GraphQlConfiguration.java:33–38`) bound the *cost per query* but do not limit *concurrency*. 1000 bounded-cost operations issued simultaneously still spike CPU.
- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` (`GraphQlConfiguration.java:43–44`) execute *after* parsing and validation, so they do not reduce the parse-phase CPU load.
- No `@RateLimiter`, no Spring Security filter, no Resilience4j bulkhead, and no infrastructure-level throttle is configured in the GraphQL module.

### Impact Explanation
Simultaneous submission of 1000 parse+validate tasks to `ForkJoinPool.commonPool()` monopolizes all available worker threads. On a typical 4–8 core node, this translates to a sustained CPU spike for the duration of the parse burst (empirically 1–5 seconds per wave). Because the attack is repeatable every hour with zero authentication, an attacker can maintain elevated CPU consumption continuously, degrading response latency for legitimate users and potentially triggering autoscaler thrashing or OOM conditions if the thread pool queue grows unboundedly.

### Likelihood Explanation
The attack requires no credentials, no special network position, and no knowledge beyond the public GraphQL endpoint URL. Generating 1000 syntactically distinct queries is trivial (vary one alias). Timing the replay to the 1-hour boundary requires only a single clock observation. The attack is fully repeatable every hour and can be automated with a simple script. No exploit tooling beyond `curl` or any HTTP client library is needed.

### Recommendation
1. **Add per-IP or global rate limiting** to the GraphQL endpoint (e.g., Spring Security's `RequestRateLimiterGatewayFilter`, Resilience4j `RateLimiter`, or an ingress-level policy) to cap the number of requests that can reach the parse pipeline per second.
2. **Replace `expireAfterWrite` with `expireAfterAccess`** in `CacheProperties.java` so that actively-used entries are not evicted synchronously, reducing the blast radius of any single expiry wave.
3. **Add TTL jitter** (e.g., randomize expiry between 50–70 minutes) so that entries written in the same second do not all expire at the same instant.
4. **Bound the ForkJoinPool or use a dedicated bounded executor** for the `AsyncCache` so that a burst of cache misses cannot monopolize the common pool.

### Proof of Concept
```bash
# Step 1: Fill the cache with 1000 distinct queries at time T
for i in $(seq 1 1000); do
  curl -s -X POST https://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ accounts(input:{entityId:{shard:0,realm:0,num:$i}}) { nodes { alias$i: alias } } }\"}" &
done
wait

# Step 2: Wait until T+1h (3600 seconds)
sleep 3600

# Step 3: Replay all 1000 queries simultaneously at T+1h+1s
for i in $(seq 1 1000); do
  curl -s -X POST https://<graphql-host>/graphql \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ accounts(input:{entityId:{shard:0,realm:0,num:$i}}) { nodes { alias$i: alias } } }\"}" &
done
wait
