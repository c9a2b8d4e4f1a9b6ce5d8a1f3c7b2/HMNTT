### Title
Missing Cross-Chunk `initialTransactionId` Consistency Validation Allows Falsely-Complete Chunk Sequence Streaming

### Summary
The `poll()` method in `PollingTopicListener` streams `TopicMessage` records directly from the database without any validation that `initialTransactionId` is consistent across chunks belonging to the same logical message. Because the importer (`ConsensusSubmitMessageTransactionHandler`) also stores `chunkNum`, `chunkTotal`, and `initialTransactionId` verbatim from the transaction body without cross-chunk verification, an unprivileged user can submit a transaction with `chunkNum == chunkTotal` but a different `initialTransactionId` than the first chunk, causing the mirror node to stream a falsely-complete chunk sequence to all subscribers.

### Finding Description
**Importer — no cross-chunk validation at ingestion:**
`ConsensusSubmitMessageTransactionHandler.doUpdateTransaction()` (lines 76–84) reads `chunkInfo.getNumber()`, `chunkInfo.getTotal()`, and `chunkInfo.getInitialTransactionID()` directly from the submitted transaction body and persists them to the `topic_message` table with no check that the `initialTransactionId` matches the one recorded for chunk 1 of the same sequence:

```java
if (transactionBody.hasChunkInfo()) {
    ConsensusMessageChunkInfo chunkInfo = transactionBody.getChunkInfo();
    topicMessage.setChunkNum(chunkInfo.getNumber());
    topicMessage.setChunkTotal(chunkInfo.getTotal());
    if (chunkInfo.hasInitialTransactionID()) {
        topicMessage.setInitialTransactionId(
                chunkInfo.getInitialTransactionID().toByteArray());
    }
}
```

**`poll()` — blind database passthrough:**
`PollingTopicListener.poll()` (lines 51–62) constructs a time-range filter and calls `topicMessageRepository.findByFilter(newFilter)`, returning every stored row as-is. There is no chunk-grouping or `initialTransactionId` consistency check at any point in this path.

**`findByFilter()` — no chunk awareness:**
`TopicMessageRepositoryCustomImpl.findByFilter()` (lines 38–46) queries only on `topicId` and `consensusTimestamp`. It has no awareness of chunk relationships.

**`ConsensusController.toResponse()` — verbatim passthrough to gRPC:**
Lines 93–113 emit whatever `chunkNum`, `chunkTotal`, and `initialTransactionId` are stored, with no cross-chunk consistency check before delivery to the subscriber.

**Confirmed by existing test:**
`ConsensusControllerTest.fragmentedMessagesGroupAcrossHistoricAndIncoming()` (lines 251–258) explicitly stores a message with `chunkNum=3, chunkTotal=3` and `initialTransactionId=new byte[]{1,2}` — a value inconsistent with the first chunk — and the test asserts it is streamed as chunk 3 of 3 without any error or filtering. This confirms the system is designed to pass through inconsistent chunk metadata.

**Exploit flow:**
1. Attacker submits chunk 1: `chunkNum=1, chunkTotal=2, initialTransactionId=TxID_A` → accepted by network, stored by importer.
2. Attacker submits a "final" chunk: `chunkNum=2, chunkTotal=2, initialTransactionId=TxID_B` (different from TxID_A) → accepted by network, stored by importer without validation.
3. `poll()` queries the DB and streams both rows to all active subscribers.
4. Subscribers using `initialTransactionId` to group chunks see two separate "complete" single-chunk messages instead of one two-chunk message, or reassemble the wrong payload.

### Impact Explanation
Any gRPC subscriber that performs chunk reassembly using `initialTransactionId` (the standard HCS client pattern) will silently reconstruct the wrong logical message. The attacker controls the content of the injected "final" chunk, so the reassembled payload at the subscriber is attacker-controlled. For topics without a `submitKey`, this is a zero-trust-required write primitive: the attacker only needs HBAR for transaction fees. For topics with a `submitKey`, the attack is limited to authorized submitters.

### Likelihood Explanation
The Hedera network does not validate `initialTransactionId` consistency across chunks — it is purely application-level metadata. Any account with sufficient HBAR can submit a `ConsensusSubmitMessage` with arbitrary `chunkInfo` fields to any open topic. The attack is trivially repeatable, requires no special tooling beyond the standard Hedera SDK, and leaves no distinguishable error in the mirror node logs.

### Recommendation
1. **At the importer level** (`ConsensusSubmitMessageTransactionHandler`): When persisting a chunk with `chunkNum > 1`, look up the stored `initialTransactionId` for `chunkNum=1` of the same logical sequence (matched by `payerAccountId` + `validStartTimestamp`) and reject/flag the row if it differs.
2. **At the streaming level** (`PollingTopicListener` / `TopicMessageServiceImpl`): Add a stateful chunk-grouping operator that verifies `initialTransactionId` is identical across all chunks of a group before emitting the sequence downstream, dropping or logging any chunk that breaks the invariant.
3. **At the gRPC response level** (`ConsensusController.toResponse()`): Emit a structured error or omit `chunkInfo` for messages whose `initialTransactionId` cannot be verified against the first chunk.

### Proof of Concept
```
// Using Hedera Java SDK
TopicId topicId = TopicId.fromString("0.0.XXXXX"); // open topic, no submitKey

TransactionId txIdA = TransactionId.generate(operatorId);

// Chunk 1 — legitimate first chunk
new TopicMessageSubmitTransaction()
    .setTopicId(topicId)
    .setMessage("chunk1-payload")
    .setChunkInfo(new ChunkInfo(txIdA, 1, 2))
    .execute(client);

// Chunk 2 — tampered: chunkNum==chunkTotal but different initialTransactionId
TransactionId txIdB = TransactionId.generate(operatorId); // different ID
new TopicMessageSubmitTransaction()
    .setTopicId(topicId)
    .setMessage("attacker-controlled-payload")
    .setChunkInfo(new ChunkInfo(txIdB, 2, 2))  // txIdB != txIdA
    .execute(client);

// Mirror node stores both rows; poll() streams both.
// A subscriber grouping by initialTransactionId sees:
//   Group TxID_A: [chunk 1/2]  → incomplete, never resolves
//   Group TxID_B: [chunk 2/2]  → falsely "complete", wrong payload delivered
```
