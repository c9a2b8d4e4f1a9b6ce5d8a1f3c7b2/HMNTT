### Title
Unauthenticated Plain-HTTP Mirror Node Response Allows Network Map Injection at Startup

### Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` fetches the consensus node list over a plain, unauthenticated HTTP connection with no response integrity verification. A network-level attacker (DNS hijacking, ARP/BGP MITM) who can intercept the single HTTP GET at pinger startup can return a crafted JSON payload that replaces every consensus node address with attacker-controlled endpoints, causing all subsequent gossip transactions to be submitted to those endpoints for the entire lifetime of the pinger process.

### Finding Description
**Code location:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 79–131; called once at startup from `pinger/sdk_client.go` line 18.

**Root cause:** The HTTP client is constructed with no TLS configuration and no transport-level security enforcement:

```go
// mirror_node_client.go line 46
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
```

The canonical example URL (`http://mirror-rest:5551`, shown in `config.go` line 37) is plain HTTP. The response body is decoded directly into `nodesEnvelope` with no HMAC, no signature, no certificate pinning, and no cross-validation against a known-good source:

```go
// lines 95–98
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```

The only checks performed are:
- HTTP status code in 200–299 range (lines 90–93)
- JSON parse success (lines 95–98)
- `NodeAccountID` non-empty (line 103)
- `AccountIDFromString` parse success (line 107)
- `host` non-empty and `port != 0` (line 118)
- At least one endpoint present (line 127)

None of these checks verify that the data originates from a legitimate mirror node. The resulting map is passed directly to `hiero.ClientForNetwork(netmap)` (`sdk_client.go` line 22), which permanently sets the gossip targets for the SDK client used by every subsequent `submitWithRetry` call.

**Exploit flow:**
1. Attacker poisons DNS for the mirror REST hostname (e.g., `mirror-rest`) or performs ARP/BGP MITM on the network path between the pinger pod and the mirror REST service.
2. Pinger starts; `newClient()` calls `buildNetworkFromMirrorNodes()` once.
3. The HTTP GET reaches the attacker's server instead of the real mirror node.
4. Attacker returns a valid-looking JSON body with `service_endpoints` pointing to attacker-controlled IPs/ports and legitimate-looking `node_account_id` values (e.g., `0.0.3`).
5. `fetchMirrorNodeNetwork()` passes all validation checks and returns the injected map.
6. `hiero.ClientForNetwork(netmap)` builds the SDK client with attacker nodes as the only gossip targets.
7. Every `cryptoTransfer.Execute(client)` for the rest of the pinger's lifetime submits transactions to attacker-controlled endpoints.

**Note on "at any point during the pinger's lifetime":** `buildNetworkFromMirrorNodes()` is called **only once at startup**, not periodically. The attack window is the single HTTP fetch at process start, but the impact persists for the entire process lifetime since the network map is never refreshed.

### Impact Explanation
All gossip transactions are permanently redirected to attacker-controlled nodes for the lifetime of the pinger process. The attacker can silently drop transactions (causing monitoring/liveness failures), replay them, or observe transaction content including amounts and account IDs. Because the pinger is a liveness/health-check component, sustained redirection causes false health signals and potential cascading operational failures. Severity: **High**.

### Likelihood Explanation
The default and documented deployment URL is plain HTTP (`http://mirror-rest:5551`). In Kubernetes, DNS poisoning within a namespace is achievable by any workload that can create a `Service` or manipulate CoreDNS records — a low bar in misconfigured clusters. ARP poisoning is feasible on flat container networks. The attack requires no application-level credentials and succeeds in a single request at a predictable moment (process startup). Repeatability is high: every pinger restart re-executes the vulnerable fetch.

### Recommendation
1. **Enforce HTTPS with TLS verification** for the mirror REST URL; reject `http://` schemes at config load time in `loadConfig()`.
2. **Add response integrity verification**: validate the returned node list against a known-good set (e.g., a configurable allowlist of expected account IDs or a signed manifest).
3. **Cross-validate endpoints**: compare fetched endpoints against a secondary source or a pinned set of known node addresses before accepting the map.
4. **Periodic refresh with change detection**: if the network map must be refreshed, alert on unexpected changes rather than silently adopting them.
