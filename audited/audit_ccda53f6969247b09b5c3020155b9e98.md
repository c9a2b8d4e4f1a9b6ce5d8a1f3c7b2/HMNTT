### Title
Cache Key Granularity in `findByTimestamp` Enables Global Rate-Budget Exhaustion and Sustained DB Query Amplification

### Summary
`RecordFileRepository.findByTimestamp(long timestamp)` uses the raw timestamp value as its Spring `@Cacheable` key. Because every distinct timestamp value is a separate cache entry, an unprivileged attacker can send requests with a continuous stream of unique timestamp values, guaranteeing a cache miss and a live DB query on every single call. The only protection is a **global** (not per-IP) rate bucket capped at 500 req/s by default, which the attacker can fully consume, simultaneously exhausting the DB query budget and denying service to all legitimate users.

### Finding Description

**Exact code path**

`RecordFileServiceImpl.findByTimestamp` delegates directly to the repository: [1](#0-0) 

The repository method is annotated with `@Cacheable` using `CACHE_MANAGER_RECORD_FILE_TIMESTAMP` and `unless = "#result == null"`: [2](#0-1) 

Spring's default cache key is the method argument — the raw `long timestamp` value. The backing query is:

```sql
select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1
```

**Cache configuration** [3](#0-2) 

`maximumSize(10000)` and `expireAfterAccess(10 minutes)`. The cache is keyed on the exact input timestamp, not on the resolved `consensusEnd` of the returned record file. Two timestamps that resolve to the **same** record file (e.g., `consensusEnd - 1` and `consensusEnd - 2`) occupy **two separate cache slots** and each triggers a separate DB query on first access.

**`unless = "#result == null"` makes it worse**

If the attacker supplies timestamps that fall beyond the last record file (future timestamps), the result is `null` and is **never cached**. Every repeated request with such a timestamp unconditionally hits the DB.

**Rate limiter is global, not per-IP** [4](#0-3) [5](#0-4) 

A single `Bucket` instance is shared across all callers. The default is 500 req/s. A single attacker can consume the entire 500 req/s budget, generating 500 uncached DB queries per second while simultaneously starving all legitimate users.

### Impact Explanation

- Up to 500 DB queries/second are forced against `record_file` (a range scan on `consensus_end`) by a single unauthenticated attacker.
- The global rate bucket is fully consumed, returning HTTP 429 to all other users — effectively a complete service denial for the `/api/v1/contracts/call` endpoint.
- The cache provides zero protection because every request uses a fresh, unique timestamp key.
- Mirror node DB connections and CPU are degraded proportionally; at 500 range-scan queries/second sustained, this is consistent with the ≥30% processing degradation threshold described in the scope.

### Likelihood Explanation

- No credentials, API keys, or special network access are required.
- The attack is trivially scriptable: enumerate block numbers, compute `consensusEnd - k` for incrementing `k`, and send at maximum rate.
- The attacker does not need to know actual `consensusEnd` values; sending any monotonically increasing sequence of nanosecond-precision timestamps achieves the same effect.
- The attack is repeatable and sustainable indefinitely.

### Recommendation

1. **Normalize the cache key to `consensusEnd`**: After the DB query resolves the record file, store the result under the returned `consensusEnd` value (not the input timestamp). Subsequent requests for any timestamp within the same block will hit the cache. This can be done by adding a `@CachePut` keyed on `#result.consensusEnd` alongside the existing `@Cacheable`.
2. **Add per-IP rate limiting**: Replace or supplement the global `rateLimitBucket` with a per-source-IP bucket (e.g., using Bucket4j with a `ProxyManager` backed by a local or distributed store) so a single attacker cannot exhaust the global budget.
3. **Cache null results with a short TTL**: Remove `unless = "#result == null"` and instead cache null/empty results with a short TTL (e.g., 5 seconds) to prevent repeated DB hits for non-existent timestamps.

### Proof of Concept

```python
import requests, time, threading

BASE = "http://<mirror-node>/api/v1/contracts/call"
HEADERS = {"Content-Type": "application/json"}

# Hedera consensus timestamps are nanoseconds since epoch.
# Use a large base value and increment to guarantee cache misses.
BASE_TS = 1_700_000_000_000_000_000  # arbitrary valid-looking nanosecond timestamp

def flood():
    ts = BASE_TS
    while True:
        ts += 1  # unique timestamp per request → guaranteed cache miss
        payload = {
            "to": "0x0000000000000000000000000000000000000167",
            "data": "0x",
            "block": hex(ts),   # passed as block parameter, resolved to timestamp internally
            "gas": 21000
        }
        requests.post(BASE, json=payload, headers=HEADERS)

# Launch threads to saturate the 500 req/s global bucket
threads = [threading.Thread(target=flood) for _ in range(20)]
for t in threads: t.start()
# Observe: DB query rate climbs to ~500/s; legitimate requests receive HTTP 429
```

Each iteration uses a fresh timestamp, guaranteeing a cache miss in `CACHE_MANAGER_RECORD_FILE_TIMESTAMP` and a live `SELECT … WHERE consensusEnd >= ? ORDER BY consensusEnd ASC LIMIT 1` query against the database.
