### Title
Unbounded Alias Resolution DB Calls Enable Connection Pool Exhaustion DoS on REST API

### Summary
`getOneAccount()` in `rest/accounts.js` unconditionally issues a synchronous DB query to resolve an alias before the main entity query, with no rate limiting on the REST API. An unauthenticated attacker sending concurrent `GET /accounts/<alias>` requests can exhaust the shared `pg.Pool` connection pool, causing all REST API endpoints to fail with connection timeout errors.

### Finding Description

**Exact code path:**

In `rest/accounts.js:399`, `getOneAccount()` unconditionally `await`s alias resolution before any other work:

```js
const encodedId = await EntityService.getEncodedId(req.params[...]);
``` [1](#0-0) 

`EntityService.getEncodedId()` at `rest/service/entityService.js:125-126` routes valid alias strings to `getAccountIdFromAlias()`:

```js
} else if (AccountAlias.isValid(entityIdString)) {
  return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
}
``` [2](#0-1) 

`getAccountFromAlias()` calls `super.getRows()` which calls `this.pool().queryQuietly()`: [3](#0-2) [4](#0-3) 

This acquires a connection from the single shared global `pool` (initialized in `rest/dbpool.js`) configured with a finite `max: config.db.pool.maxConnections`: [5](#0-4) 

**Root cause:** Each concurrent alias request holds a pool connection for the full duration of the alias resolution query. The connection is not released until the `await` resolves. With N concurrent requests where N ≥ `maxConnections`, the pool is exhausted. All subsequent requests — including non-alias requests to any endpoint sharing the pool — block until `connectionTimeoutMillis` expires and then fail.

**Why existing checks fail:** The `rest/server.js` middleware stack contains `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and `responseCacheCheckHandler` — **no rate limiting or concurrency cap**: [6](#0-5) 

The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleProperties`) exists only in the Java-based `web3` service, not in the Node.js REST API: [7](#0-6) 

The `statementTimeout` and `connectionTimeoutMillis` configs only bound individual query duration and connection wait time — they do not prevent pool exhaustion from concurrent requests.

### Impact Explanation

When the pool is exhausted, every REST API endpoint that calls `pool.queryQuietly()` — accounts, balances, contracts, tokens, transactions, topics — returns errors until connections free up. The attacker does not need to sustain the attack indefinitely; a continuous stream of alias requests at concurrency ≥ `maxConnections` keeps the pool permanently saturated. The default `pg.Pool` size is 10 connections, making this trivially achievable. Even with a larger configured pool, the attack scales linearly with pool size. This is a full REST API denial-of-service requiring zero privileges.

### Likelihood Explanation

Any unauthenticated user can send `GET /api/v1/accounts/KGNABD5L3ZGSRVUCSPDR7TONZSRY3D5OMEBKQMVTD2AC6JL72HMQ` (or any valid base32 alias pattern matching `^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$`). The alias does not need to exist in the database — a non-matching alias still issues the full DB query and holds the connection until the query returns zero rows. A single attacker with a basic HTTP load tool (e.g., `ab`, `wrk`, `hey`) can sustain the required concurrency from a single machine.

### Recommendation

1. **Add per-IP or global concurrency/rate limiting** to the Node.js REST API middleware, mirroring the `ThrottleManagerImpl` pattern used in the web3 service.
2. **Cache alias→id resolutions** in a short-lived in-memory or Redis cache (the response cache infrastructure already exists) to avoid a DB round-trip per request.
3. **Validate alias format before issuing any DB query** — invalid aliases already throw `InvalidArgumentError` without a DB call; ensure the regex check is as cheap as possible and happens before pool acquisition.
4. **Set a per-user or global `max_user_client_connections`** at the pgBouncer layer for the `mirror_rest` user (already partially configured at 1000 client / 250 server connections in the Helm chart) to bound the blast radius. [8](#0-7) 

### Proof of Concept

```bash
# Generate 50 concurrent alias requests (all non-existent, still hit DB)
# Valid base32 alias format: 40+ uppercase A-Z2-7 chars
ALIAS="AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA"  # 40 chars, valid format

# Using 'hey' HTTP load tool (or ab/wrk):
hey -n 10000 -c 50 \
  "http://<mirror-node-rest>/api/v1/accounts/${ALIAS}"

# Expected result:
# - First ~maxConnections requests succeed (slow)
# - Remaining requests return 500 / connection timeout errors
# - All other REST API endpoints (/api/v1/transactions, /api/v1/tokens, etc.)
#   also begin returning errors during the attack
# - Attack stops when requests stop; pool recovers automatically
```

final class ThrottleManagerImpl implements ThrottleManager {

    static final String REQUEST_PER_SECOND_LIMIT_EXCEEDED = "Requests per second rate limit exceeded";
    static final String GAS_PER_SECOND_LIMIT_EXCEEDED = "Gas per second rate limit exceeded.";

    @Qualifier(GAS_LIMIT_BUCKET)
    private final Bucket gasLimitBucket;

    @Qualifier(RATE_LIMIT_BUCKET)
    private final Bucket rateLimitBucket;

    @Qualifier(OPCODE_RATE_LIMIT_BUCKET)
    private final Bucket opcodeRateLimitBucket;

    private final ThrottleProperties throttleProperties;

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
