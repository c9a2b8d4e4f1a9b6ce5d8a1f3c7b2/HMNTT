### Title
Cache Poisoning via Unchecked `Optional.empty()` in `findRuntimeBytecode` Enables Unauthenticated Denial-of-Service for Contract Bytecode Lookups

### Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"`, which does not exclude `Optional.empty()` from being stored in the cache. When an unprivileged caller submits an `eth_call` targeting a non-existent contract ID, the resulting `Optional.empty()` is cached for up to 1 hour (`expireAfterAccess=1h`). Any subsequent legitimate bytecode lookup for that contract ID — including after the contract is actually deployed — will be served the stale empty result from cache, causing the EVM to treat the contract as having no bytecode.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, line 16:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [1](#0-0) 

**Root cause:**

Spring's `@Cacheable` evaluates `#result` as the actual Java object returned by the method. When the contract does not exist in the `contract` table, the JPA query returns `Optional.empty()`. This is a non-null `Optional` object, so `#result == null` evaluates to `false`. The `unless` condition is therefore not triggered, and `Optional.empty()` is written into the Caffeine cache under the key `contractId`.

**Cache TTL:**

The `contract` cache manager is configured with `expireAfterAccess=1h,maximumSize=1000`: [2](#0-1) 

Because the policy is `expireAfterAccess` (not `expireAfterWrite`), an attacker who periodically re-queries the same contract ID resets the TTL on each access, keeping the poisoned entry alive indefinitely.

**Downstream consumer:**

`ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode()` and maps the result:
```java
return contractRepository
        .findRuntimeBytecode(entityId.getId())
        .map(Bytes::wrap)
        .map(Bytecode::new)
        .orElse(null);   // Optional.empty() → null → "no bytecode"
``` [3](#0-2) 

When the cache returns `Optional.empty()`, `.orElse(null)` returns `null`, and the EVM layer treats the target address as a non-contract account with no executable bytecode.

**Exploit flow:**

1. Attacker identifies a target contract ID (e.g., a contract that is about to be deployed, or any existing contract whose cache entry has not yet been populated).
2. Attacker sends an `eth_call` (no authentication required) with `to` set to the long-zero EVM address corresponding to that contract ID.
3. `ContractBytecodeReadableKVState.toEntityId()` resolves the address to the numeric entity ID and calls `findRuntimeBytecode(entityId)`.
4. The DB query returns `Optional.empty()` (contract not yet in the `contract` table).
5. Spring caches `Optional.empty()` under key `contractId` because `#result == null` is `false`.
6. The contract is subsequently deployed and its `runtime_bytecode` is written to the DB.
7. All future `eth_call` / `eth_estimateGas` requests for that contract ID hit the cache and receive `Optional.empty()` → `null` bytecode → execution fails as if the contract does not exist.
8. Attacker re-queries every ~59 minutes to reset `expireAfterAccess`, keeping the poisoned entry alive beyond the 1-hour window.

**Why existing checks are insufficient:**

The only guard is `unless = "#result == null"`. This correctly excludes a raw `null` return, but the method's declared return type is `Optional<byte[]>`, so the repository never returns a raw `null` — it always returns either `Optional.of(bytes)` or `Optional.empty()`. The guard therefore never fires for the miss case. [4](#0-3) 

The existing test `findRuntimeBytecodeFailCall` confirms that querying a non-existent ID returns `Optional.empty()`, but does not assert that this result is excluded from the cache: [5](#0-4) 

### Impact Explanation

Any unprivileged caller can permanently deny bytecode resolution for up to 1,000 distinct contract IDs simultaneously (the cache `maximumSize`). Affected `eth_call` and `eth_estimateGas` requests against poisoned contract IDs will behave as if the contract has no code — calls return empty results or revert, and gas estimation is incorrect. Because the cache is `expireAfterAccess`, the attacker can sustain the denial indefinitely with periodic re-queries. This constitutes a targeted, persistent denial-of-service against specific contracts on the mirror node's web3 API.

### Likelihood Explanation

The attack requires no credentials, no on-chain transaction, and no special knowledge beyond the numeric entity ID of the target contract (which is publicly derivable from the Hedera entity ID sequence or the mirror node REST API). The `eth_call` endpoint is publicly exposed. The attack is trivially scriptable: a single HTTP request per target ID every hour is sufficient to maintain the poisoned cache entry. The maximum blast radius (1,000 IDs) can be reached with 1,000 requests.

### Recommendation

Change the `unless` expression to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

This ensures that only a populated `Optional.of(bytes)` is stored in the cache, and cache misses (non-existent contracts) always fall through to the database. Additionally, add a test that verifies a miss result is not cached (i.e., a subsequent call after the contract is inserted returns the real bytecode, not `Optional.empty()`).

### Proof of Concept

**Preconditions:** Mirror node web3 API is running. Target contract with numeric entity ID `N` exists in the `entity` table but has not yet had its `runtime_bytecode` written to the `contract` table (e.g., it is about to be deployed), OR the attacker queries before the contract is created.

**Steps:**

1. Send a poison request before the contract is deployed:
   ```
   POST /api/v1/contracts/call
   {
     "to": "0x000000000000000000000000000000000000000N",  // long-zero address for entity ID N
     "data": "0x",
     "gas": 100000
   }
   ```
2. The call resolves to `findRuntimeBytecode(N)`, gets `Optional.empty()` from DB, and caches it.
3. The contract is now deployed; `runtime_bytecode` is written to the `contract` table for ID `N`.
4. Send a legitimate call to the same address:
   ```
   POST /api/v1/contracts/call
   { "to": "0x000000000000000000000000000000000000000N", "data": "<valid_calldata>", "gas": 100000 }
   ```
5. **Observed:** The cache returns `Optional.empty()` → `readFromDataSource` returns `null` → EVM sees no bytecode → call fails or returns empty as if the contract does not exist.
6. **Expected:** The DB is queried, real bytecode is returned, and the call executes correctly.
7. To sustain the attack, repeat step 1 every ~59 minutes to reset the `expireAfterAccess` timer.
