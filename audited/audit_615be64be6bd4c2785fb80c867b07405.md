### Title
Unsanitized `req.originalUrl` Interpolated Into Log Strings in `handleError()` Enables Log Flooding and Log Injection

### Summary
In `rest/middleware/httpErrorHandler.js`, the `handleError()` function directly interpolates `req.originalUrl` — a fully attacker-controlled value — into `logger.warn` and `logger.error` template strings with no length truncation or character sanitization. The logger itself (`rest/monitoring/logger.js`) performs no truncation and writes the full string to `process.stdout`. An unprivileged external user can craft requests with maximally-sized URLs (up to the Node.js HTTP parser ceiling of ~8–16 KB depending on runtime version) containing embedded newlines or other control characters, generating oversized or structurally malformed log entries on every error response.

### Finding Description
**Exact code path:**

`rest/middleware/httpErrorHandler.js`, `handleError()`, lines 33–35 and 39:
```js
logger.warn(
  `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode} ${err.constructor.name} ${errorMessage}`
);
// ...
logger.error(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${statusCode}`, detailedMessage);
```

`req.originalUrl` is the raw, unmodified request URL as received by Express, set directly from the incoming HTTP request line. It is never truncated, sanitized, or escaped before being embedded in the log string.

**Logger — no truncation:**

`rest/monitoring/logger.js`, `#log()`, lines 21–22:
```js
const text = `${time} ${levelName} ${msg}${stack}\n`;
process.stdout.write(text);
```
The full string is written verbatim to stdout. There is no max-length guard.

**Root cause / failed assumption:**
The code assumes `req.originalUrl` will be a short, well-formed path. No middleware in the chain (`server.js` lines 68–98) enforces a URL length limit or strips control characters from the URL before it reaches `handleError`. The `express.urlencoded` / `express.json` body-size limits apply only to request bodies, not to the request line. The `requestQueryParser` (`requestHandler.js`) enforces `throwOnLimitExceeded: true` for repeated query parameters, but this itself triggers an error that flows into `handleError`, causing the oversized URL to be logged.

**Exploit flow:**
1. Attacker sends a GET request with a query string padded to the Node.js HTTP parser ceiling (~8 KB on Node ≤v12, ~16 KB on Node ≥v13), e.g., `GET /api/v1/accounts?x=AAAA...AAAA`.
2. The request reaches Express. Any error condition (404, 400 from query parser limit, 500) routes through `handleError`.
3. `handleError` interpolates the full `req.originalUrl` into the log string and calls `logger.warn` or `logger.error`.
4. The logger writes the full oversized string to stdout with no truncation.
5. For log injection: the attacker URL-encodes a newline (`%0a`) in the query string. Express preserves percent-encoded characters in `req.originalUrl`. The logger writes the decoded or raw string, injecting a fake log line into the output stream.

**Why existing checks are insufficient:**
- No middleware enforces a URL length cap before `handleError` is reached.
- The logger has no per-message size limit.
- The `requestQueryParser` limit only limits the *number* of repeated parameters, not total URL byte length.
- There is no `req.originalUrl` sanitization anywhere in the middleware chain.

### Impact Explanation
Every error response (404, 400, 500) causes a log entry whose size is proportional to the attacker-supplied URL. At scale (high request rate), this produces log volume that can:
- Exhaust log aggregation system per-entry size limits (e.g., AWS CloudWatch: 256 KB/event, Elasticsearch default: 100 MB/bulk but per-document limits vary), causing truncation of the event record.
- Flood log storage, crowding out legitimate partition/error events and degrading observability of real incidents.
- Inject fake log lines via `%0a`-encoded newlines in `req.originalUrl`, spoofing log entries that appear to come from the server itself, undermining audit integrity.

Severity: **Medium** (log integrity / availability impact; no direct data exfiltration or RCE).

### Likelihood Explanation
- Requires zero authentication or privileges — any HTTP client can trigger this.
- Triggering an error is trivial: request any nonexistent path (404) or exceed the query parameter repeat limit (400).
- Fully repeatable and automatable; a single attacker with a script can sustain continuous log flooding.
- The Node.js HTTP parser ceiling (~8–16 KB) limits individual entry size, but does not prevent high-volume flooding.

### Recommendation
1. **Truncate `req.originalUrl` before logging.** Define a constant (e.g., `MAX_LOG_URL_LENGTH = 512`) and apply it:
   ```js
   const safeUrl = req.originalUrl.slice(0, MAX_LOG_URL_LENGTH);
   logger.warn(`${req.ip} ${req.method} ${safeUrl} in ${elapsed} ms: ...`);
   ```
2. **Strip or encode control characters** (newlines, carriage returns) from `req.originalUrl` before interpolation to prevent log injection.
3. **Add a URL length-limiting middleware** early in the Express chain (before route handlers) that rejects requests whose URL exceeds a configured maximum, returning 414 URI Too Long.
4. **Add per-message size limiting in the logger** as a defense-in-depth measure.

### Proof of Concept
```bash
# 1. Log flooding: send a request with a ~8KB query string to any nonexistent path
python3 -c "
import socket
payload = 'A' * 8000
req = f'GET /api/v1/nonexistent?x={payload} HTTP/1.1\r\nHost: target\r\n\r\n'
s = socket.create_connection(('target-host', 5551))
s.send(req.encode())
print(s.recv(4096))
s.close()
"
# Result: server logs a ~8KB warn/error line for every such request.

# 2. Log injection: inject a fake log line via newline in query string
curl -v "http://target-host:5551/api/v1/nonexistent?x=foo%0aERROR+2026-04-28+FAKE+INJECTED+LOG+ENTRY"
# Result: the logger writes a line containing a literal newline, splitting the output
# into two lines — the second appearing as an independent log entry.
```
