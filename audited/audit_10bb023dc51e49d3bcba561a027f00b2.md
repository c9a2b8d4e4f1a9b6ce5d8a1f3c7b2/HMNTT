### Title
Unbounded Historical Poll Loop via `startTime=0` + No Limit Causes DB Connection Exhaustion and Delays Live Transaction Delivery

### Summary
An unprivileged gRPC client can subscribe to any high-volume topic with `consensusStartTime=0` and no `consensusEndTime`/`limit`, causing `PollingTopicMessageRetriever.poll()` to loop indefinitely — fetching `maxPageSize` rows per cycle, advancing `startTime` by one nanosecond, and holding a DB connection on every iteration. With enough concurrent subscriptions the DB connection pool is saturated, blocking the live-message path and delaying delivery of new consensus transactions to legitimate subscribers.

### Finding Description

**Entry point** — `ConsensusController.toFilter()` accepts the raw protobuf fields with no floor on `startTime` and no requirement for `endTime` or `limit`: [1](#0-0) 

**Filter validation** — `TopicMessageFilter` permits `startTime=0` via `@Min(0)` and `isValidStartTime()` (0 ≤ `DomainUtils.now()`). `endTime` is nullable; `limit=0` means no limit: [2](#0-1) 

**poll() — startTime advances by +1 ns per cycle** — when `last != null`, the next query starts at `last.getConsensusTimestamp() + 1`. On a topic with millions of messages this advances by one nanosecond per 1 000-row page: [3](#0-2) 

**isComplete() never returns true for a full page** — for the throttled path (`retrieve(filter, true)` called from `TopicMessageServiceImpl`), completion requires `pageSize < maxPageSize`. A full 1 000-row page keeps the loop alive. `numRepeats = Long.MAX_VALUE`: [4](#0-3) [5](#0-4) 

**Idle timeout does not help** — `.timeout(60s)` fires only when *no item is emitted* for 60 seconds. Because messages are continuously emitted (1 000 rows every 2 s), the timeout never triggers: [6](#0-5) 

**DB queries are blocking** — `findByFilter` calls `getResultList()` (blocking JPA) on the `boundedElastic()` scheduler, holding a thread and a DB connection for the full query duration: [7](#0-6) 

**Per-connection call limit is not a global guard** — `maxConcurrentCallsPerConnection=5` limits calls per TCP connection; an attacker opens many connections: [8](#0-7) [9](#0-8) 

No global subscriber count cap is enforced anywhere in the service layer. [10](#0-9) 

### Impact Explanation

The mirror node's DB connection pool is shared between the historical-retrieval path and the live-message listener path. When the pool is exhausted by attacker-controlled historical polling subscriptions, the live listener cannot acquire connections to fetch newly arrived consensus transactions, stalling delivery of those transactions to all legitimate subscribers. This is a denial-of-service against the mirror node's core function of propagating new consensus messages. Severity is **High**: the attack degrades service for all subscribers on the affected instance, not just the attacker's own stream.

### Likelihood Explanation

The attack requires zero privileges — any client that can reach port 5600 can issue a `subscribeTopic` gRPC call. The required parameters (`startTime=0`, no `endTime`, no `limit`) are explicitly documented as valid inputs. A single attacker machine can open dozens of TCP connections, each carrying 5 concurrent subscriptions, for a total of hundreds of indefinitely-looping DB queries with no special tooling beyond a standard gRPC client. The attack is fully repeatable and stateless.

### Recommendation

1. **Add a wall-clock subscription timeout** — replace or supplement the idle `.timeout()` with an absolute maximum subscription duration (e.g., 5–10 minutes) so that no single historical-retrieval stream can hold resources indefinitely.

2. **Enforce a global concurrent-subscriber cap** — add an `AtomicInteger` gate in `TopicMessageServiceImpl.subscribeTopic()` that rejects new subscriptions when a configurable ceiling is reached, similar to the existing `subscriberCount` gauge but with enforcement.

3. **Require `endTime` or `limit` for deep-history queries** — reject (or cap) subscriptions where `startTime` is more than a configurable threshold in the past and neither `endTime` nor `limit` is set, preventing unbounded scans of the full message history.

4. **Cap `startTime` minimum age** — add a validation rule that `startTime >= now - maxHistoricalWindow` (e.g., 24 h) unless a `limit` is also provided, preventing epoch-0 scans.

5. **Bound the DB connection pool separately** for the retriever scheduler so that historical polling cannot starve the live-message path.

### Proof of Concept

```

# topic 0.0.12345 has > 1 000 historical messages.

# Open 20 parallel subscriptions with startTime=0, no endTime, no limit.
for i in $(seq 1 20); do
  grpcurl -plaintext \
    -d '{
      "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 12345},
      "consensusStartTime": {"seconds": 0, "nanos": 0}
    }' \
    grpc.example.com:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic \
    > /dev/null &
done

# Each background process holds a DB connection and polls every 2 s indefinitely.
# Monitor DB connection pool exhaustion:
#   SELECT count(*) FROM pg_stat_activity WHERE datname='mirror_node';
# Observe that legitimate subscribers stop receiving new messages as the pool fills.
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```

```java
    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }
```

```java
    @Override
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
```
