### Title
Unauthenticated Alias-Based DB Query Flooding via `GET /api/v1/accounts/{id}/hooks` Causes Mirror Node DoS

### Summary
The `HooksController` endpoint `GET /api/v1/accounts/{id}/hooks` accepts alias-format path parameters with no rate limiting in the `rest-java` module. Each request with a valid-format but non-existent alias unconditionally triggers a `findByAlias()` database query through `EntityServiceImpl.lookup()`, with no negative-result caching. An unprivileged attacker can flood this endpoint with random valid-format aliases to saturate the database connection pool and deny service to all mirror node REST API consumers.

### Finding Description

**Exact code path:**

`HooksController.getHooks()` receives `@PathVariable EntityIdParameter ownerId` and immediately delegates to `hookService.getHooks(hooksRequest)`: [1](#0-0) 

`HookServiceImpl.getHooks()` calls `entityService.lookup(request.getOwnerId())` before any hook repository access: [2](#0-1) 

`EntityServiceImpl.lookup()` pattern-matches on `EntityIdAliasParameter` and unconditionally calls `entityRepository.findByAlias(p.alias())` — a live database query — with no caching: [3](#0-2) 

**Root cause:** The `rest-java` module has no rate limiting. The only throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is scoped to `ContractCallRequest` — it is never applied to `rest-java` endpoints: [4](#0-3) 

The `rest-java` configuration registers only an ETag filter and a protobuf converter — no rate limiter: [5](#0-4) 

The `LoggingFilter` only logs; `MetricsFilter` only records metrics. No `OncePerRequestFilter` enforces request rate limits: [6](#0-5) 

**Why format validation is insufficient:** `EntityIdAliasParameter` validates that the alias matches `^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$` — this only rejects malformed input. Any valid-format alias (e.g., `HIQQEXWKW53RKN4W6XXC4Q232SYNZ3SZANVZZSUME5B5PRGXL663UAQA`) that does not exist in the database still reaches the DB: [7](#0-6) 

The database connection pool is a finite Hikari pool configured via `spring.datasource.hikari`: [8](#0-7) 

### Impact Explanation
An attacker can exhaust the Hikari connection pool of the `rest-java` service, causing all subsequent requests (from any user, to any endpoint) to time out waiting for a connection. This denies service to all mirror node REST API consumers. The Hiero consensus network itself is unaffected (the mirror node is read-only), but mirror node availability — including account, transaction, and token queries — is fully disrupted for the duration of the attack.

### Likelihood Explanation
The attack requires no credentials, no special knowledge, and no prior access. Any internet-connected client can generate valid-format Base32 aliases matching the regex. The attack is trivially scriptable, parallelizable across multiple source IPs, and repeatable indefinitely. The absence of any per-IP or global rate limit in `rest-java` means there is no server-side defense to overcome.

### Recommendation
1. **Add rate limiting to `rest-java`**: Introduce a `OncePerRequestFilter` using Bucket4j (already a dependency in `web3`) applied globally to `/api/*` in `rest-java`, enforcing a per-IP and/or global requests-per-second cap.
2. **Cache negative alias lookups**: In `EntityServiceImpl.lookup()`, cache `Optional.empty()` results for alias and EVM address lookups with a short TTL (e.g., 30 seconds) using Caffeine or Spring Cache, analogous to the cache used in `importer`'s `EntityIdServiceImpl`.
3. **Deploy an API gateway or WAF** in front of `rest-java` with rate limiting per source IP.

### Proof of Concept

```bash
# Generate random valid-format Base32 aliases and flood the endpoint
for i in $(seq 1 10000); do
  curl -s -o /dev/null \
    "https://<mirror-node>/api/v1/accounts/HIQQEXWKW53RKN4W6XXC4Q232SYNZ3SZANVZZSUME5B5PRGXL663UAQA/hooks" &
done
wait
# Result: Hikari pool exhausted; subsequent legitimate requests receive 500 or timeout
```

**Preconditions:** None — public endpoint, no authentication.
**Trigger:** Each request with a valid-format non-existent alias hits `entityRepository.findByAlias()`.
**Result:** Connection pool exhaustion; mirror node REST API unavailable.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));

    public static final String ALIAS_REGEX = "^((\\d{1,5})\\.)?((\\d{1,5})\\.)?([A-Z2-7]{40,70})$";
    public static final Pattern ALIAS_PATTERN = Pattern.compile(ALIAS_REGEX);
    private static final BaseEncoding BASE32 = BaseEncoding.base32().omitPadding();

    static @Nullable EntityIdAliasParameter valueOfNullable(String id) {
        var aliasMatcher = ALIAS_PATTERN.matcher(id);

        if (!aliasMatcher.matches()) {
            return null;
        }

        var properties = CommonProperties.getInstance();
        long shard = properties.getShard();
        long realm = properties.getRealm();
        String realmString;

        if ((realmString = aliasMatcher.group(4)) != null) {
            realm = Long.parseLong(realmString);
            shard = Long.parseLong(aliasMatcher.group(2));
        } else if ((realmString = aliasMatcher.group(2)) != null) {
            realm = Long.parseLong(realmString);
        }

        var alias = BASE32.decode(aliasMatcher.group(5));
        return new EntityIdAliasParameter(shard, realm, alias);
    }
