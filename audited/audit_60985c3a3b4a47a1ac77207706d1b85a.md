### Title
Asymmetric Bound Condition Drops Secondary GTE Constraint When Primary Has Only Upper Bound

### Summary
In `TokenAirdropRepositoryCustomImpl.findAll()`, when a caller supplies `receiver_id` with operator `LTE` and `token_id` with operator `GTE`, the `getBoundConditions` logic in `JooqRepository` builds an upper outer condition of `receiver_id = X` with no `token_id` constraint. This is because `toUpper()` on a bound that has only a lower range (`GTE`) returns `Bound.EMPTY`, which collapses to `noCondition()` in the recursive call. The result is that all records where `receiver_id = X` are returned regardless of `token_id`, leaking airdrop records that should be excluded by the `token_id >= Y` filter.

### Finding Description

**Exact code path**:

`TokenAirdropRepositoryCustomImpl.findAll()` (line 58–72) calls `getBoundConditions(bounds)` from `JooqRepository`. [1](#0-0) 

`getBounds()` in `TokenAirdropRequest` places `entityIds` (receiver_id, LTE-only) as primary and `tokenIds` (token_id, GTE-only) as secondary: [2](#0-1) 

In `getBoundConditions`, `getOuterCondition` is called with `isUpper=true`: [3](#0-2) 

`removeRanges` calls `b.toUpper()` on each secondary bound: [4](#0-3) 

`toUpper()` on a bound with only a lower range (`GTE Y`, upper=null) calls `createBound(null)` which returns `Bound.EMPTY`: [5](#0-4) [6](#0-5) 

The recursive `getBoundConditions([EMPTY, EMPTY], true, true)` produces `noCondition()` for all three sub-conditions (lower, middle, upper), so the upper outer condition becomes simply `receiver_id = X` with no `token_id` constraint.

**Root cause**: `toUpper()` is semantically correct only when the secondary bound has an upper range. When the secondary bound is lower-only (e.g., `GTE`), `toUpper()` silently returns `Bound.EMPTY`, and the code has no guard to detect or compensate for this asymmetry. The design assumes that if the primary has an upper bound, the secondary will also have an upper bound to constrain it.

**Failed assumption**: The algorithm assumes that when building the upper outer condition, secondary bounds will have an upper range to apply. When they do not (lower-only), the constraint is silently dropped rather than preserved or rejected.

### Impact Explanation

An unprivileged API caller receives airdrop records that should be excluded by their own filter. Specifically, for a PENDING airdrop query with `receiver_id=lte:X&token_id=gte:Y`, all records where `receiver_id = X AND token_id < Y` are incorrectly returned. This is an information disclosure: the caller can enumerate airdrop records beyond their intended filter window. No funds are directly at risk, but the API contract is violated and pagination-based cursor logic built on top of this endpoint will produce incorrect results, potentially exposing data from other accounts' airdrop histories.

### Likelihood Explanation

Any unauthenticated or unprivileged user of the REST API can trigger this by constructing a request with `receiver_id=lte:N&token_id=gte:M` on the pending airdrops endpoint. No special privileges, tokens, or knowledge of internal state are required. The combination is a natural pagination pattern (e.g., "give me airdrops up to receiver X, starting from token Y"), making accidental or deliberate triggering highly feasible. The condition is deterministic and 100% reproducible.

### Recommendation

In `getOuterCondition` (or `removeRanges`), when building the upper outer condition (`isUpper=true`), if a secondary bound has no upper range (i.e., `toUpper()` returns `Bound.EMPTY`), the secondary bound's lower constraint must still be applied rather than silently dropped. One approach: when `isUpper=true` and a secondary bound is lower-only, include the full secondary bound (both lower and upper, i.e., no range removal) in the recursive call, since for the upper outer condition the secondary field is unconstrained from above and should still be constrained from below. Alternatively, add an explicit check: if `b.toUpper().isEmpty()` and `!b.isEmpty()`, substitute the original bound or a `noCondition()`-safe fallback that preserves the lower constraint. [7](#0-6)

    private Condition getOuterCondition(
            Bound primary, List<Bound> secondaryBounds, boolean isUpper, boolean processed) {
        var outerCondition = getPrimaryCondition(primary, isUpper);
        if (outerCondition != noCondition()) {
            var outerBounds = processed ? secondaryBounds : removeRanges(secondaryBounds, isUpper);
            outerCondition = outerCondition.and(getBoundConditions(outerBounds, true, isUpper));
        }

        return outerCondition;
    }

    // Returns a list of new bounds that have had their lower or upper ranges removed
    private List<Bound> removeRanges(List<Bound> bounds, boolean isUpper) {
        return bounds.stream().map(b -> isUpper ? b.toUpper() : b.toLower()).toList();
    }
