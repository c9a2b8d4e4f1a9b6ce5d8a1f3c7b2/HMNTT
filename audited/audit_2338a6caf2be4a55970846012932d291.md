### Title
Unauthenticated Parallel Request Flood Exhausts Database Connection Pool via Unthrottled `getHooks()` Endpoint

### Summary
The `getHooks()` method in `HookServiceImpl` issues a direct, uncached database query on every invocation with no rate limiting, request deduplication, or per-IP throttling in the `rest-java` module. An unprivileged attacker can open many concurrent HTTP connections and flood the endpoint with identical requests, causing the database connection pool to be exhausted and rendering the mirror node REST API unavailable.

### Finding Description
**Code path:**
- Controller: `rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, `getHooks()`, lines 80–102 — accepts unauthenticated GET requests with no concurrency guard.
- Service: `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`, `getHooks()`, lines 32–53 — unconditionally calls `hookRepository.findByOwnerIdAndHookIdBetween()` (line 40) or `hookRepository.findByOwnerIdAndHookIdIn()` (line 51) on every invocation.

**Root cause:** There is no caching (`@Cacheable` is absent), no request deduplication, and no rate limiting applied to this endpoint. The `rest-java` module's only throttle infrastructure is `ShallowEtagHeaderFilter` (registered in `RestJavaConfiguration.java` lines 42–46), which computes an ETag from the response body *after* the database query has already executed. It saves bandwidth on repeat requests that send `If-None-Match`, but the full DB query still runs on every request that omits that header — which an attacker trivially does.

**Why existing checks fail:**
- `ShallowEtagHeaderFilter`: only suppresses response body transmission; does not prevent the DB query.
- `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` on `hookId` (line 84): limits filter parameter count per request, not concurrent request count.
- `@Max(MAX_LIMIT)` on `limit` (line 86): caps result set size, not parallelism.
- The `ThrottleManagerImpl` / `ThrottleConfiguration` / `ThrottleProperties` rate-limiting stack exists exclusively in the `web3` module; it is not wired into `rest-java` at all.
- No authentication or authorization is required to call this endpoint.

### Impact Explanation
Each concurrent request acquires a database connection and executes a potentially expensive range or IN-list query against the `hook` table. With enough parallel connections the database connection pool is exhausted, causing all subsequent queries (including those from legitimate users and internal components) to queue or fail. This results in a full denial-of-service of the mirror node REST API. Because the mirror node is used by wallets, explorers, and dApps to query state and submit transactions, its unavailability degrades the broader ecosystem's ability to interact with the network.

### Likelihood Explanation
Exploitation requires no credentials, no special knowledge, and no on-chain assets. Any attacker with a standard HTTP client (e.g., `curl`, `ab`, `wrk`) can reproduce it. The attack is repeatable, automatable, and can be launched from a single host or distributed across many IPs to bypass any upstream network-level rate limiting. The endpoint is publicly reachable on any deployed mirror node instance.

### Recommendation
1. **Add rate limiting to `rest-java`**: Port or reuse the `ThrottleManagerImpl`/`Bucket4j` pattern already present in `web3` to apply a per-IP or global requests-per-second limit to all `rest-java` endpoints, including `/api/v1/accounts/{ownerId}/hooks`.
2. **Add response caching**: Annotate `getHooks()` with `@Cacheable` (Spring Cache + Caffeine/Redis) keyed on the normalized request parameters, with a short TTL (e.g., 1–5 seconds), so repeated identical queries are served from memory rather than hitting the database.
3. **Limit DB connection pool**: Configure HikariCP `maximumPoolSize` conservatively and set a `connectionTimeout` so that pool exhaustion fails fast rather than queuing indefinitely.
4. **Deploy an API gateway or reverse proxy** (e.g., nginx, Envoy) in front of the service with per-IP connection and request-rate limits as a defense-in-depth layer.

### Proof of Concept
```bash
# Send 500 concurrent identical requests to exhaust the DB connection pool
# No authentication, no special headers required
seq 500 | xargs -P 500 -I{} curl -s \
  "https://<mirror-node-host>/api/v1/accounts/0.0.1234/hooks?limit=100&order=desc" \
  -o /dev/null

# Verify service degradation: subsequent legitimate requests time out or return 500
curl -v "https://<mirror-node-host>/api/v1/accounts/0.0.1234/hooks"
```

    public Collection<Hook> getHooks(HooksRequest request) {
        final var sort = Sort.by(request.getOrder(), HOOK_ID);
        final var page = PageRequest.of(0, request.getLimit(), sort);
        final var id = entityService.lookup(request.getOwnerId());
        final long lowerBound = request.getLowerBound();
        final long upperBound = request.getUpperBound();

        if (request.getHookIds().isEmpty()) {
            return hookRepository.findByOwnerIdAndHookIdBetween(id.getId(), lowerBound, upperBound, page);
        } else {
            // Both equal and range filters are present.
            final var idsInRange = request.getHookIds().stream()
                    .filter(hookId -> hookId >= lowerBound && hookId <= upperBound)
                    .toList();

            if (idsInRange.isEmpty()) {
                return List.of();
            }

            return hookRepository.findByOwnerIdAndHookIdIn(id.getId(), idsInRange, page);
        }
    }
