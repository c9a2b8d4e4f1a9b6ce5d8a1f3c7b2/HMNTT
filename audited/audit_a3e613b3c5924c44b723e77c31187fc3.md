### Title
Accept-Encoding q-value Bypass Allows Uncompressed Opcode Trace Response Delivery

### Summary
The `validateAcceptEncodingHeader()` method in `OpcodesController` uses a naive `.contains("gzip")` substring check to enforce gzip compression on the large opcode trace response. An attacker can supply `Accept-Encoding: gzip;q=0` (or `identity;q=1, gzip;q=0`), which satisfies the substring check while RFC 7231 q-value semantics instruct the server not to use gzip, resulting in the full uncompressed response being transmitted. The throttle guard limits request rate but does not bound per-response payload size.

### Finding Description
**Exact code location:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, `validateAcceptEncodingHeader()`, line 76:

```java
if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
```

**Root cause:** The guard performs only a substring presence check. It does not parse the RFC 7231 `Accept-Encoding` field grammar, which allows each token to carry a quality value (`q=0` means "not acceptable"). The failed assumption is that the presence of the string `"gzip"` anywhere in the header guarantees the server will compress the response.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/{txHash}/opcodes` with `Accept-Encoding: gzip;q=0`.
2. `acceptEncodingHeader.toLowerCase().contains("gzip")` → `true`; the check passes and execution continues.
3. `throttleManager.throttleOpcodeRequest()` consumes one token from the opcode rate-limit bucket (rate-limited, not size-limited).
4. `opcodeService.processOpcodeCall(request)` executes the full EVM replay and returns the complete opcode trace (stack, memory, storage for every executed opcode).
5. Spring's embedded server (Tomcat/Jetty) evaluates the actual `Accept-Encoding` header for content negotiation. Because `q=0` explicitly marks gzip as unacceptable per RFC 7231 §5.3.4, the server either sends the response uncompressed or falls back to `identity`.
6. The full, uncompressed opcode trace payload is transmitted to the attacker.

**Why existing checks fail:**
- The substring check (line 76) is the only application-level guard; it is bypassed as shown.
- `throttleManager.throttleOpcodeRequest()` (line 61) enforces a request-per-second rate limit via a token bucket but places no constraint on the size of each individual response. A single opcode trace for a complex contract can be many megabytes.
- No server-level `server.compression.*` configuration was found in the repository, meaning compression is not guaranteed to be enabled at the infrastructure layer either.

### Impact Explanation
The opcode trace endpoint is explicitly documented as producing a "huge" response (code comment, line 38–41). Without compression, each allowed request can transfer megabytes of JSON. An attacker operating within the rate limit can sustain continuous high-bandwidth consumption, exhausting outbound bandwidth and increasing server memory pressure during serialization. Because the EVM replay itself is CPU-intensive, the combination of full replay plus large uncompressed serialization amplifies resource cost per request.

### Likelihood Explanation
No authentication or privilege is required; the endpoint is publicly accessible. The bypass requires only crafting a single HTTP header value, which any HTTP client (curl, Python requests, etc.) can do trivially. The attack is fully repeatable up to the rate limit and requires no special knowledge of the target system beyond the public API path.

### Recommendation
Replace the substring check with proper RFC 7231 `Accept-Encoding` parsing that evaluates q-values:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw ...;
    }
    boolean gzipAccepted = Arrays.stream(acceptEncodingHeader.split(","))
        .map(String::trim)
        .anyMatch(token -> {
            String[] parts = token.split(";");
            String encoding = parts[0].trim().toLowerCase();
            if (!encoding.equals("gzip") && !encoding.equals("*")) return false;
            for (int i = 1; i < parts.length; i++) {
                String param = parts[i].trim();
                if (param.startsWith("q=")) {
                    try { return Double.parseDouble(param.substring(2)) > 0; }
                    catch (NumberFormatException e) { return false; }
                }
            }
            return true; // no q= means q=1.0
        });
    if (!gzipAccepted) {
        throw HttpClientErrorException.create(...);
    }
}
```

Additionally, enable server-level compression (`server.compression.enabled=true`, `server.compression.mime-types=application/json`) so that compression is enforced at the transport layer independently of the application check.

### Proof of Concept
```bash
# Bypass: gzip present as substring but q=0 disables it
curl -v \
  -H "Accept-Encoding: identity;q=1, gzip;q=0" \
  "https://<mirror-node-host>/api/v1/contracts/results/<tx-hash>/opcodes?stack=true&memory=true&storage=true"

# Expected (vulnerable): HTTP 200 with full uncompressed JSON body, no Content-Encoding: gzip header
# Expected (patched):    HTTP 406 Not Acceptable

# Alternative bypass using wildcard with gzip exclusion
curl -v \
  -H "Accept-Encoding: *, gzip;q=0" \
  "https://<mirror-node-host>/api/v1/contracts/results/<tx-hash>/opcodes"
```

The response will lack `Content-Encoding: gzip` and deliver the full opcode trace payload uncompressed, confirming the bypass.
