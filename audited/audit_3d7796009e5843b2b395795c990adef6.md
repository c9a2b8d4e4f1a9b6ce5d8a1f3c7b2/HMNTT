### Title
Unauthenticated `/metrics` Endpoint Exposes `hiero_mirror_rosetta_response_bytes` Histogram, Enabling Inference of Mirror Node Data Gaps

### Summary
The Rosetta middleware registers `/metrics` on the same HTTP port as the public API using a bare `promhttp.Handler()` with no authentication or access control. Any unprivileged external user can retrieve the full Prometheus metrics dump, including the `hiero_mirror_rosetta_response_bytes` histogram broken down by route. By monitoring bucket counts for block-related routes over time, an attacker can detect when the mirror node returns anomalously small responses (e.g., error payloads or empty blocks), inferring gaps or anomalies in the ingested Hashgraph history.

### Finding Description
**Code location:** `rosetta/app/middleware/metrics.go`, `Routes()`, lines 64–73.

```go
func (c *metricsController) Routes() server.Routes {
    return server.Routes{
        {
            "metrics",
            "GET",
            metricsPath,           // "/metrics"
            promhttp.Handler().ServeHTTP,  // no auth wrapper
        },
    }
}
```

The `responseBytesHistogram` is declared at lines 39–43 with `sizeBuckets = []float64{512, 1024, 10*1024, 25*1024, 50*1024}` and labeled by `{method, route}`. Every HTTP response to every Rosetta route (including `/block`, `/block/transaction`) is bucketed here via `MetricsMiddleware` (lines 76–83).

**Root cause:** `promhttp.Handler()` is used directly with no authentication middleware wrapping it. The REST API's metrics endpoint explicitly defaults to `authentication: true` (documented in `docs/configuration.md` line 569), but the Rosetta metrics endpoint has no equivalent protection.

**Exploit path:**
1. The Helm chart's gateway rules (values.yaml lines 68–83) route the prefix `/rosetta` → `/` via `URLRewrite`, so `GET /rosetta/metrics` is rewritten to `GET /metrics` and served by the same `http` port (5700).
2. The `ServiceMonitor` (servicemonitor.yaml lines 13–14) confirms `/metrics` is on the `http` port — the same port exposed by the `Service` and reachable through the ingress/gateway.
3. An attacker issues `GET /rosetta/metrics` (no credentials required) and receives the full Prometheus text exposition, including:
   ```
   hiero_mirror_rosetta_response_bytes_bucket{application="rosetta",method="POST",route="/block",le="512"} N
   ```
4. By polling `/metrics` periodically while also issuing block queries, the attacker correlates increments in the `le="512"` bucket for the block route with specific block indices, identifying which blocks returned small (error/empty) responses — indicating gaps or truncation in the Hashgraph history.

**Why existing checks fail:** There is no IP allowlist, no auth middleware, no separate metrics port, and no network policy template in the chart. The ingress template (ingress.yaml) uses `pathType: ImplementationSpecific` with operator-configured paths and no path-based exclusion of `/metrics`.

### Impact Explanation
An attacker gains passive, continuous visibility into the mirror node's operational state without any credentials. The `hiero_mirror_rosetta_response_bytes` histogram with `route` labels reveals when block or transaction queries return anomalously small payloads (≤512 bytes), which correspond to error responses or empty blocks. This allows an attacker to map data gaps in the mirror node's view of the Hashgraph — information that could be used to time attacks, exploit inconsistencies in downstream systems relying on the mirror node, or undermine trust in the node's data completeness. Additionally, the full metrics dump leaks request rates, inflight counts, and duration distributions, providing a detailed operational fingerprint.

### Likelihood Explanation
The Rosetta API is a public-facing blockchain interface by design. The gateway URL-rewrite makes `/metrics` reachable via the standard public endpoint with zero preconditions — no authentication, no special headers, no network position. Any internet user can exploit this with a single HTTP GET. The attack is trivially repeatable and undetectable (it generates only normal GET traffic to a legitimate endpoint).

### Recommendation
1. **Wrap the metrics handler with authentication** in `Routes()`:
   ```go
   promhttp.InstrumentMetricHandler(
       prometheus.DefaultRegisterer,
       promhttp.HandlerFor(prometheus.DefaultGatherer, promhttp.HandlerOpts{}),
   )
   ```
   wrapped with a `BasicAuth` or token-check middleware before `ServeHTTP`.
2. **Serve metrics on a separate, non-public port** (e.g., 9090) distinct from the Rosetta API port (5700), and ensure the ingress/gateway does not route to that port.
3. **Add a NetworkPolicy** in the Helm chart restricting access to the metrics port to only the Prometheus scraper's namespace/pod selector.
4. **Block `/metrics` at the ingress/gateway level** using path-based rules to prevent external access regardless of application-level controls.

### Proof of Concept
```bash
# Step 1: Confirm unauthenticated access (returns HTTP 200 with Prometheus text)
curl -s https://<rosetta-host>/rosetta/metrics | grep hiero_mirror_rosetta_response_bytes

# Step 2: Observe bucket counts for block route
curl -s https://<rosetta-host>/rosetta/metrics \
  | grep 'hiero_mirror_rosetta_response_bytes_bucket.*route="/block"'

# Expected output (example showing responses ≤512 bytes — error/empty blocks):
# hiero_mirror_rosetta_response_bytes_bucket{application="rosetta",method="POST",route="/block",le="512"} 7

# Step 3: Poll periodically while issuing block queries to correlate increments
# in the le="512" bucket with specific block indices, mapping data gaps.
watch -n 5 'curl -s https://<rosetta-host>/rosetta/metrics \
  | grep hiero_mirror_rosetta_response_bytes_bucket'
```
