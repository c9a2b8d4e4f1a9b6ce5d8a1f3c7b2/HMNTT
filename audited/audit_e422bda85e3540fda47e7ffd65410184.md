### Title
Integer Overflow in `poll()` Allows Unauthenticated Full Table Scan via `limit=Long.MAX_VALUE`

### Summary
An unprivileged gRPC client can set `limit = Long.MAX_VALUE` in a `ConsensusTopicQuery`. This value passes all Bean Validation checks (`@Min(0)`), but inside `PollingTopicMessageRetriever.poll()` the unchecked `(int)` cast overflows to `-1`, which propagates as the DB page limit. The resulting internal filter has `limit = -1`, causing `hasLimit()` to return `false` and `setMaxResults` to never be called, producing an unbounded full table scan of the `topic_message` table on every polling cycle.

### Finding Description

**Entry point** — `ConsensusController.toFilter()` maps the raw proto `uint64` field directly to the filter with no upper-bound check: [1](#0-0) 

`Long.MAX_VALUE` is a legal `uint64` value and satisfies the only constraint on `limit`: [2](#0-1) 

`hasLimit()` returns `true` because `Long.MAX_VALUE > 0`: [3](#0-2) 

**Overflow site** — `poll()` performs an unchecked narrowing cast: [4](#0-3) 

`(int)(Long.MAX_VALUE - 0L)` = `(int)(0x7FFFFFFFFFFFFFFFL)` = `0xFFFFFFFF` = **`-1`** as a signed `int`. `Math.min(-1, maxPageSize)` = `-1` regardless of `maxPageSize`.

**Propagation** — the corrupted value is written into a new internal filter without any validation: [5](#0-4) 

**DB layer** — `findByFilter` checks `hasLimit()` on the internal filter. With `limit = -1`, `hasLimit()` returns `false`, so `setMaxResults` is never called: [6](#0-5) 

The JPA query executes with no `LIMIT` clause → full table scan. The identical overflow exists in `PollingTopicListener.poll()`: [7](#0-6) 

### Impact Explanation
Every polling cycle (default every 2 s for throttled, every 20 ms for unthrottled) issues an unbounded `SELECT` against `topic_message`. A single persistent subscription can saturate DB I/O and memory, causing denial of service for all other subscribers and the importer. The `timeout` of 60 s means each connection can sustain ~30 full-scan queries before being dropped, and the attacker can immediately reconnect.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. Setting `limit = Long.MAX_VALUE` in a `ConsensusTopicQuery` proto message requires only a standard gRPC client. The attack is trivially scriptable, repeatable, and requires zero privileges.

### Recommendation
1. Add an upper-bound constraint on `limit` in `TopicMessageFilter` (e.g., `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate cap).
2. Replace the unchecked `(int)` cast with a safe clamp before computing `pageSize`:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   int pageSize = Math.min(limit, context.getMaxPageSize());
   ```
3. Apply the same fix to `PollingTopicListener.poll()`.
4. In `TopicMessageRepositoryCustomImpl.findByFilter`, add a guard: if `(int) filter.getLimit() <= 0`, treat as no-limit or throw, rather than silently omitting `setMaxResults`.

### Proof of Concept
```python
import grpc
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc
from com.hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    limit=9223372036854775807,  # Long.MAX_VALUE — valid uint64
)

# Each next() call triggers poll() → integer overflow → full table scan
for msg in stub.subscribeTopic(query):
    pass  # DB is hammered every polling interval
```

Preconditions: valid topic ID, no authentication. Result: unbounded SQL query issued on every poll cycle, exhausting DB resources.
