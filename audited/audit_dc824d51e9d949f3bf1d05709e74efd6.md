### Title
Historical State Bypass via Missing `timestamp_range` Upper-Bound in `findActiveByEvmAddressOrAliasAndTimestamp` Entity Branch

### Summary
`findActiveByEvmAddressOrAliasAndTimestamp()` resolves an entity's ID using `created_timestamp <= ?2` in the CTE, but the `entity` branch of the UNION fetches the current live row with no `lower(timestamp_range) <= ?2` guard. When an entity has been updated after the queried `blockTimestamp`, the live row's `timestamp_range` lower-bound is later than any matching `entity_history` row, so `ORDER BY timestamp_range DESC LIMIT 1` selects the post-timestamp state, returning a future state as if it were the historical one. The sibling method `findActiveByIdAndTimestamp` correctly applies `lower(timestamp_range) <= ?2` to the entity branch, confirming this is an unintentional omission.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 95–121 (`findActiveByEvmAddressOrAliasAndTimestamp`):

```sql
with entity_cte as (
    select id
    from entity
    where created_timestamp <= ?2 and (evm_address = ?1 or alias = ?1)  -- only creation-time bound
    ...
)
(
    select *
    from entity e
    where e.deleted is not true
    and e.id = (select id from entity_cte)
    -- ← NO lower(timestamp_range) <= ?2 here
)
union all
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2          -- correctly bounded
    and eh.id = (select id from entity_cte)
    ...
)
order by timestamp_range desc
limit 1
```

Compare with `findActiveByIdAndTimestamp` (lines 136–155), which correctly adds `lower(timestamp_range) <= ?2` to the entity branch:

```sql
select *
from entity
where id = ?1 and lower(timestamp_range) <= ?2   -- ← correct guard present
and deleted is not true
```

**Root cause:** The `entity_cte` only proves the entity *existed* at `blockTimestamp` (via `created_timestamp`). It does not constrain which version of the entity is returned. The `entity` table always holds the most-recent live state; its `timestamp_range` lower-bound equals the timestamp of the last update. If that update occurred after `blockTimestamp`, the live row has a higher `lower(timestamp_range)` than any `entity_history` row valid at `blockTimestamp`, so it wins the `ORDER BY timestamp_range DESC LIMIT 1` sort and is returned.

**Failed assumption:** The query assumes the live `entity` row is always the correct answer when the entity was created before `blockTimestamp`. This is false whenever the entity was subsequently modified after `blockTimestamp`.

**Exploit flow:**

1. Entity `E` is created at `T1 = 100` with state `S1` (e.g., `balance = 1000`, `key = K1`). Live row: `timestamp_range = [100, ∞)`.
2. At `T3 = 300`, `E` is updated (e.g., `balance = 500`, `key = K2`). The old state `S1` is moved to `entity_history` with `timestamp_range = [100, 300)`. The live row now has `timestamp_range = [300, ∞)`.
3. Attacker calls `eth_call` (or equivalent historical RPC) with `blockNumber` corresponding to `T2 = 200` (i.e., `T1 < T2 < T3`), targeting `E`'s evm_address or alias.
4. `entity_cte` resolves `E`'s ID (created at 100 ≤ 200). ✓
5. `entity` branch returns `S2` (`balance=500`, `timestamp_range=[300,∞)`). No timestamp check blocks it.
6. `entity_history` branch returns `S1` (`balance=1000`, `timestamp_range=[100,300)`). `lower(100) ≤ 200`. ✓
7. `ORDER BY timestamp_range DESC LIMIT 1`: `lower(300) > lower(100)` → `S2` is selected.
8. **Result:** The API returns `balance=500, key=K2` for block `T2=200`, but the correct answer is `balance=1000, key=K1`. [1](#0-0) [2](#0-1) 

### Impact Explanation

Any caller of `eth_call`, `eth_getBalance`, `eth_getCode`, or any EVM simulation that resolves an account/contract by evm_address or alias at a historical block will receive the **current** entity state instead of the state valid at the requested block. Concretely:

- **Balance misrepresentation:** A contract or account's balance at a past block is reported as its current balance, enabling false proof-of-funds assertions.
- **Key/admin misrepresentation:** The admin key, auto-renew settings, or contract bytecode at a past block is reported as the current value, corrupting historical audit trails.
- **Smart-contract simulation integrity:** `eth_call` at a past block is the foundation for off-chain verification in bridges, oracles, and governance systems. Returning wrong state breaks these guarantees.
- **Scope:** Every entity looked up by evm_address or alias (not by numeric ID) via a historical timestamp is affected. This covers the majority of EVM-compatible address lookups.

Severity: **High** — the mirror node's historical query guarantee is broken for a broad class of lookups, and the returned data is silently wrong with no error signal to the caller.

### Likelihood Explanation

- **No authentication required.** The `blockTimestamp` parameter is derived from the user-supplied block number in standard JSON-RPC calls (`eth_call`, `eth_getBalance`, etc.). Any anonymous user can supply an arbitrary past block number.
- **Precondition is common.** Any entity that has ever been updated (key rotation, balance change, contract upgrade) satisfies the precondition. On a live network this is the majority of active accounts and contracts.
- **Trivially repeatable.** The attacker only needs to know a valid evm_address or alias and supply a block number prior to the entity's last update. No special privileges, no gas, no on-chain transaction.
- **Undetectable by the caller.** The API returns HTTP 200 with a well-formed result; there is no error or warning that the returned state is from the wrong point in time.

### Recommendation

Add `lower(e.timestamp_range) <= ?2` to the `entity` branch of both `findActiveByEvmAddressOrAliasAndTimestamp` and `findActiveByEvmAddressAndTimestamp`, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- entity branch (fix)
select *
from entity e
where e.deleted is not true
  and lower(e.timestamp_range) <= ?2          -- ← add this line
  and e.id = (select id from entity_cte)
```

This ensures the live row is only included when its effective start time is at or before the queried `blockTimestamp`, which is exactly the semantics `findActiveByIdAndTimestamp` already enforces. [3](#0-2) [4](#0-3) 

### Proof of Concept

**Setup (SQL):**
```sql
-- Entity created at T=100
INSERT INTO entity (id, evm_address, alias, balance, created_timestamp, timestamp_range, deleted)
VALUES (42, '\xDEAD...', '\xBEEF...', 1000, 100, '[100,300)', false);

-- Entity updated at T=300 (old row moved to history, new row in entity)
INSERT INTO entity_history (id, evm_address, alias, balance, created_timestamp, timestamp_range, deleted)
VALUES (42, '\xDEAD...', '\xBEEF...', 1000, 100, '[100,300)', false);

UPDATE entity
SET balance = 500, timestamp_range = '[300,)'
WHERE id = 42;
```

**Trigger (Java / JSON-RPC):**
```
eth_call { to: "0xDEAD...", blockNumber: <block corresponding to T=200> }
```

Or directly:
```java
entityRepository.findActiveByEvmAddressOrAliasAndTimestamp(
    hexToBytes("0xBEEF..."),
    200L   // blockTimestamp between 100 and 300
);
```

**Expected result:** Entity with `balance = 1000` (state at T=200).

**Actual result:** Entity with `balance = 500` (state from T=300, after the queried block).

The fix can be verified by adding `lower(e.timestamp_range) <= 200` to the entity branch and confirming the query then returns `balance = 1000`.
