### Title
Unbounded Idle gRPC Connection Accumulation Leading to File Descriptor Exhaustion DoS

### Summary
`grpcServerConfigurer()` configures only `maxConcurrentCallsPerConnection` (default: 5) and an executor, but sets no `maxConnectionIdle`, `maxConnectionAge`, or total inbound connection count limit on the Netty gRPC server. An unprivileged attacker can open an unbounded number of idle TCP connections that persist indefinitely, exhausting the process's file descriptors and preventing legitimate subscribers from connecting or the server from opening database connections, causing complete gRPC service unavailability.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
    // No maxConnectionIdle, maxConnectionAge, or maxInboundConnections
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–15 — `NettyProperties` exposes only `maxConcurrentCallsPerConnection`; no idle/age/count fields exist.

**Root cause:** Netty gRPC's default for `maxConnectionIdle` is `Long.MAX_VALUE` (effectively infinite). Because no override is applied, every accepted TCP connection lives until the remote peer closes it. `maxConcurrentCallsPerConnection=5` limits active RPC calls *per connection*, but imposes zero constraint on the number of simultaneous connections. A client that opens connections and sends no RPCs bypasses this check entirely.

**Exploit flow:**
1. Attacker opens thousands of TCP connections to port 5600 and sends the HTTP/2 client preface (making them valid gRPC connections) but issues no RPC calls.
2. The server accepts each connection, allocates a file descriptor, and keeps it open indefinitely.
3. `maxConcurrentCallsPerConnection` is never triggered because no calls are in flight.
4. The process file descriptor table fills to the OS limit (`ulimit -n`, typically 65535 in containers).
5. The JVM can no longer accept new TCP connections, open database connections via HikariCP, or perform any file I/O.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection=5` — only limits calls, not connections.
- GCP backend policy `maxRatePerEndpoint: 250` — rate-limits RPC *requests*, not idle TCP connections.
- Traefik circuit breaker (`NetworkErrorRatio > 0.10`) — triggers on error responses, not on silent idle connections.
- Prometheus alert `GrpcHighFileDescriptors` (>80%) — a reactive alert, not a preventive control; by the time it fires, the server is already degraded.

### Impact Explanation
When file descriptors are exhausted, the gRPC server cannot accept new subscriber connections, HikariCP cannot open new database connections (blocking topic message retrieval), and internal health checks may fail. All legitimate consumers of the `ConsensusService` (topic message streaming) and `NetworkService` (address book) are denied service. This is a complete availability loss for the gRPC API with no self-recovery until the attacker releases connections or the process is restarted.

### Likelihood Explanation
No authentication or rate limiting is required to open a TCP connection to port 5600. A single attacker machine can open tens of thousands of idle connections using standard tools (`hping3`, a trivial Python script, or `grpc_cli`). The attack is repeatable, cheap, and requires no special knowledge of the Hedera protocol. The absence of any server-side connection count limit or idle timeout makes this straightforwardly exploitable from the public internet if the gRPC port is reachable.

### Recommendation
Add the following to `grpcServerConfigurer()` and expose the values as configurable `NettyProperties` fields:

```java
serverBuilder.maxConnectionIdle(30, TimeUnit.SECONDS);
serverBuilder.maxConnectionAge(300, TimeUnit.SECONDS);
serverBuilder.maxConnectionAgeGrace(10, TimeUnit.SECONDS);
serverBuilder.maxInboundConnections(1000); // Netty 1.57+
```

Additionally, enforce connection-count limits at the load balancer/ingress layer and consider IP-based rate limiting on new connection establishment.

### Proof of Concept
```python
import grpc
import time

stubs = []
for i in range(50000):
    # Open connection, send HTTP/2 preface, issue no RPCs
    channel = grpc.insecure_channel("grpc-mirror-node:5600")
    stubs.append(channel)
    if i % 1000 == 0:
        print(f"Opened {i} idle connections")

# Hold connections open; server FD table exhausts
time.sleep(3600)
```

Run from any host with network access to port 5600. Monitor with:
```
kubectl exec -n <ns> <grpc-pod> -- cat /proc/self/fd | wc -l
```
