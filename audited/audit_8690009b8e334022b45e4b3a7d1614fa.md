### Title
Unauthenticated Hash-Lookup DoS via Uncached 404 on `/api/v1/blocks/{hashOrNumber}` with No Rate Limiting

### Summary
Any unauthenticated user can supply an unlimited stream of syntactically valid but non-existent 96-hex-char Hedera block hashes to `/api/v1/blocks/{hashOrNumber}`. Each request passes format validation, executes a `LIKE`-based SQL query against the partitioned `record_file` table, returns a 404 that is explicitly excluded from the response cache, and is never throttled. Under concurrent load this exhausts the database connection pool and degrades or denies service to all API consumers.

### Finding Description

**Validation gate — passes format check, reaches DB every time:**

In `rest/controllers/blockController.js` lines 28–38, `validateHashOrNumber` calls `isValidEthHashOrHederaHash` (defined in `rest/utils.js` lines 195–202 via `ethHashOrHederaHashPattern = /^(0x)?([0-9A-Fa-f]{64}|[0-9A-Fa-f]{96})$/`). A 96-hex-char string passes unconditionally and is forwarded to `RecordFileService.getByHashOrNumber(hash, null)`. [1](#0-0) [2](#0-1) 

**DB query — LIKE on a partitioned table:**

`RecordFileService.getByHashOrNumber` (lines 164–179) builds:
```sql
SELECT ... FROM record_file WHERE hash LIKE $1
```
where `$1 = hash + '%'`. [3](#0-2) 

The `record_file` table is partitioned by `consensus_end` range (V2 schema). The `record_file__hash` index exists per-partition (`collate "C"`). Because `hash` has no correlation with `consensus_end`, PostgreSQL must probe every partition's index for each hash lookup. On a production node with years of data this means dozens-to-hundreds of index scans per request. [4](#0-3) [5](#0-4) 

**404 responses are never cached:**

`responseCacheUpdateHandler` (line 95) only stores responses when `httpStatusCodes.isSuccess(res.statusCode)` (200–299) or 304. A non-existent block returns 404 via `NotFoundError`, which is explicitly excluded. Every request for a non-existent hash hits the database cold. [6](#0-5) [7](#0-6) 

**No rate limiting on the REST API blocks endpoint:**

The throttle/rate-limit infrastructure found (`ThrottleConfiguration`, `ThrottleManagerImpl`) is scoped exclusively to the `web3` Java service's `/contracts/call` POST endpoint. The Node.js REST service has no per-IP or global request-rate limiting for `GET /api/v1/blocks/:hashOrNumber`. [8](#0-7) 

### Impact Explanation
An attacker sending N concurrent requests/second with unique valid 96-char hex hashes forces N simultaneous cross-partition index scans against the database. Because 404 responses are never cached, there is no amortisation. The database connection pool is finite; once saturated, all API endpoints (not just blocks) begin queuing or rejecting connections, causing a full non-network DoS of the mirror node REST API. No authentication, account, or special privilege is required.

### Likelihood Explanation
The attack requires zero privileges and zero knowledge of the network state. Generating unique valid 96-char hex strings is trivial (e.g., `openssl rand -hex 48`). The endpoint is publicly documented in the OpenAPI spec. A single attacker with a modest HTTP client (e.g., `wrk`, `hey`, or a simple script) can sustain thousands of requests per second. The absence of any rate-limiting middleware in the REST service makes this repeatable and persistent.

### Recommendation
1. **Cache negative results**: Store 404 responses in Redis with a short TTL (e.g., 5–30 s). The cache key already uses the full URL, so unique hashes will still miss once, but repeated identical hashes are absorbed.
2. **Add rate limiting to the REST API**: Introduce per-IP (and optionally global) request-rate limiting middleware (e.g., `express-rate-limit`) for all `GET` endpoints, with tighter limits on path-parameter endpoints.
3. **Validate hash existence cheaply before full query**: Consider a lightweight existence check (e.g., a covering index scan limited to the hash prefix) before executing the full `blocksQuery` SELECT.
4. **Partition pruning**: If feasible, store a hash-to-partition mapping or use a hash-partitioned auxiliary table to avoid full cross-partition scans.

### Proof of Concept
```bash
# Generate 10 000 unique valid 96-hex-char Hedera hashes and flood the endpoint
for i in $(seq 1 10000); do
  HASH=$(openssl rand -hex 48)
  curl -s "https://<mirror-node>/api/v1/blocks/${HASH}" &
done
wait
```
Each request passes `isValidEthHashOrHederaHash`, reaches `RecordFileService.getByHashOrNumber`, executes `SELECT ... FROM record_file WHERE hash LIKE '<96chars>%'` across all partitions, returns 404, and is never cached. Under sustained concurrent load the database connection pool is exhausted and the service becomes unavailable to all clients.

const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
};
