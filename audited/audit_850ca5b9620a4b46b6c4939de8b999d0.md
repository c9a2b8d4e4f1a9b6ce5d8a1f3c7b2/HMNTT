### Title
Degenerate `gte:X&lte:X` Range Bypasses Validation and Produces Over-Broad UNION Query in `extractTokenMultiUnionQuery`

### Summary
When an unprivileged caller supplies `spender.id=gte:X&spender.id=lte:X&token.id=gte:Y&token.id=lte:Y` (identical lower and upper bounds), all four validation checks in `validateBounds()` pass silently. The resulting three-part UNION query generates an impossible inner predicate (`spender > X AND spender < X`) and two boundary sub-queries whose union covers **every** `token_id` for spender X, returning far more records than the stated filter implies.

### Finding Description

**Parsing — `bound.js` `parse()`**

`gte:X` sets `primary.lower = {operator: gte, value: X}` and `lte:X` sets `primary.upper = {operator: lte, value: X}`. Neither slot is `equal`, so `hasEqual()` returns `false`. [1](#0-0) 

**Validation — `baseController.js` `validateBoundsRange()`**

The only guard against a degenerate range checks `hasBound() && hasEqual()`. Because no `eq` operator was used, `hasEqual()` is `false` for both bounds, so the check never fires. [2](#0-1) 

`validateLowerBounds()` and `validateUpperBounds()` also pass: both require `primary.lower.operator === gt` (not `gte`) or `primary.upper.operator === lt` (not `lte`) to throw, so the `gte`/`lte` pair slips through. [3](#0-2) 

**Query generation — `baseController.js` filter builders**

With `primary.lower = gte:X`, `primary.upper = lte:X`, `secondary.lower = gte:Y`, `secondary.upper = lte:Y`:

| Part | Generated filters | SQL predicate |
|------|------------------|---------------|
| `getLowerFilters` | `spender = X`, `token_id >= Y` | `spender = X AND token_id >= Y` |
| `getInnerFilters` | `spender > X`, `spender < X` | `spender > X AND spender < X` ← impossible |
| `getUpperFilters` | `spender = X`, `token_id <= Y` | `spender = X AND token_id <= Y` | [4](#0-3) 

**SQL assembly — `tokenAllowanceService.js` `getQuery()`**

The three non-empty filter arrays are each wrapped in a sub-query and joined with `UNION ALL`: [5](#0-4) 

The resulting SQL is:
```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ...)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ...)  -- always empty
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ...)
ORDER BY spender, token_id LIMIT $2
```
params: `[owner, limit, X, Y, X, X, X, Y]`

For any `token_id T`, either `T >= Y` or `T <= Y` is always true, so the UNION returns **all** token allowances for `spender = X`, not just those where `token_id = Y`.

### Impact Explanation
A caller who intends to query a single `(spender, token)` pair instead receives every token allowance granted by the owner to that spender. The API response is presented as filtered by `token.id=gte:Y&token.id=lte:Y` but silently returns the full allowance set for that spender, misrepresenting the scope of the historical record returned. Because token allowances are public read data on this endpoint, the direct confidentiality impact is low; however, the response is factually incorrect relative to the stated filter, which undermines the integrity of the mirror-node's historical view and can mislead downstream consumers or auditors who trust the filter semantics.

### Likelihood Explanation
No authentication is required. The endpoint is `/api/v1/accounts/:id/allowances/tokens`, publicly reachable. The payload is a trivial HTTP query string requiring no special knowledge. The condition is deterministically reproducible on every request.

### Recommendation
Add a value-equality check inside `validateBoundsRange` (or a dedicated validator) that rejects a bound whose `lower.value === upper.value` when both operators are inclusive (`gte`/`lte`), or normalise such a pair into a single `eq` filter before query construction. Additionally, `getInnerFilters` should detect and skip the impossible `lower > X AND upper < X` predicate when `lower.value === upper.value`.

```js
// in validateBoundsRange or a new validateBoundValues():
if (bound.hasLower() && bound.hasUpper() && bound.lower.value === bound.upper.value) {
  throw new InvalidArgumentError(
    `Degenerate range: gte and lte values are equal for ${bound.filterKey}; use eq instead`
  );
}
```

### Proof of Concept

```
GET /api/v1/accounts/0.0.1001/allowances/tokens
    ?spender.id=gte:2001&spender.id=lte:2001
    &token.id=gte:10001&token.id=lte:10001
```

**Expected**: records where `spender = 0.0.2001 AND token_id = 0.0.10001` (at most 1 row).

**Actual**: all token allowances where `owner = 0.0.1001 AND spender = 0.0.2001`, regardless of `token_id`.

Trace through `extractTokenMultiUnionQuery`:
1. `primary.lower = {gte, 2001}`, `primary.upper = {lte, 2001}`, `secondary.lower = {gte, 10001}`, `secondary.upper = {lte, 10001}`.
2. `validateBoundsRange` → no throw (`hasEqual()` is false for both).
3. `getLowerFilters` → `[{eq, 2001}, {gte, 10001}]`.
4. `getInnerFilters` → `[{gt, 2001}, {lt, 2001}]` (impossible predicate).
5. `getUpperFilters` → `[{eq, 2001}, {lte, 10001}]`.
6. SQL UNION returns all rows for `spender = 2001`.

  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
