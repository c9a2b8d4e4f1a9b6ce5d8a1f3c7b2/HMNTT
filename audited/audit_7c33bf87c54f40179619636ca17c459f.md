### Title
Unauthenticated `account()` GraphQL Query Enables Circuit-Breaker-Evasive DB Connection Exhaustion via Missing Global Rate Limit

### Summary
The GraphQL `/graphql/alpha` endpoint exposes the `account()` query with no authentication, no global rate limit, and only a per-source-IP in-flight request cap of 5. An unprivileged attacker controlling multiple IPs can sustain enough concurrent `account()` queries to exhaust the HikariCP connection pool. Because pool exhaustion produces HTTP 500 responses (counted by `ResponseCodeRatio`, not `NetworkErrorRatio`), the attacker can keep `NetworkErrorRatio` naturally below the 0.10 circuit-breaker threshold while calibrating the 500-error rate to stay just under the 0.25 `ResponseCodeRatio` threshold, permanently preventing the circuit breaker from opening and maintaining a degraded-partition state for all other users.

**Root cause and failed assumption:**

The design assumes the `inFlightReq: amount: 5` per-IP cap is sufficient to bound aggregate backend load. This assumption fails because:

1. There is **no global rate limit** on the GraphQL endpoint. Compare with the Rosetta chart, which adds `rateLimit: average: 10` per host on top of the same `inFlightReq` cap. [6](#0-5) 

2. The `ipStrategy: depth: 1` criterion means the limit is enforced per source IP. An attacker with N distinct IPs (cloud VMs, residential proxies, or a small botnet) can sustain 5N concurrent backend requests with no global ceiling.

3. The GraphQL service has **no application-level throttle** (the `ThrottleConfiguration`/`ThrottleManagerImpl` bucket4j throttle exists only in the `web3` module). [7](#0-6) 

4. The `retry: attempts: 3` middleware re-submits each failed request up to three times, tripling the effective backend load per attacker request. [8](#0-7) 

5. HikariCP pool exhaustion causes the Spring application to return **HTTP 500**, which Traefik counts under `ResponseCodeRatio(500,600,0,600)`, **not** `NetworkErrorRatio()`. Therefore `NetworkErrorRatio` stays naturally near zero regardless of how hard the pool is hit.

**Exploit flow:**

- Attacker sends 5 concurrent `account()` queries from each of N IPs. Each query acquires a HikariCP connection and holds it for the duration of the DB round-trip.
- Once the pool is exhausted, subsequent legitimate requests queue inside HikariCP until the connection-acquisition timeout fires, at which point Spring returns HTTP 500.
- The attacker tunes N so that the 500-error rate stays at ~20–24% (below the 0.25 `ResponseCodeRatio` threshold). `NetworkErrorRatio` remains ≈0 because no TCP-level failures occur.
- Neither circuit-breaker condition (`NetworkErrorRatio() > 0.10` or `ResponseCodeRatio > 0.25`) is ever satisfied, so the breaker never opens.
- The remaining ~76% of requests complete but with severely elevated latency (waiting for a free connection), degrading all legitimate users indefinitely.

### Impact Explanation

All users of the GraphQL API experience persistent high latency or HTTP 500 errors. The circuit breaker — the primary automated recovery mechanism — is structurally prevented from opening because the error type (HTTP 500) is counted by the wrong metric relative to the threshold that would actually trip. The service enters a stable degraded-partition state that requires manual operator intervention to resolve. Severity is **High** (availability impact across all users, no authentication required, no automated recovery).

### Likelihood Explanation

The attack requires only multiple IP addresses, which are trivially obtainable via cloud providers (each costing fractions of a cent per hour) or residential proxy networks. No credentials, tokens, or prior knowledge of the system are needed. The calibration required (keep 500-error rate between ~10% and 24%) is straightforward to achieve with a simple feedback loop measuring response codes. The attack is fully repeatable and can be sustained indefinitely. Likelihood is **High**.

### Recommendation

1. **Add a global `rateLimit` middleware** to the GraphQL middleware chain, mirroring the Rosetta configuration (`rateLimit: average: <N>` per host or per IP). This caps aggregate request volume regardless of how many source IPs the attacker uses. [1](#0-0) 

2. **Lower the `NetworkErrorRatio` circuit-breaker threshold** or add a tighter `ResponseCodeRatio` threshold (e.g., `> 0.10`) so that a 10–15% 500-error rate triggers the breaker before the pool is fully saturated.

3. **Add application-level throttling** in the GraphQL service (analogous to `ThrottleManagerImpl` in `web3`) to enforce a global requests-per-second ceiling independent of the Traefik layer.

4. **Set a HikariCP `connectionTimeout`** short enough that pool-exhaustion failures are fast (reducing connection hold time) and consider a `maximumPoolSize` alarm threshold lower than the current 75% alert. [9](#0-8) 

### Proof of Concept

```bash
# Requires: N distinct source IPs (e.g., N cloud VMs or a proxy list)
# Replace <GRAPHQL_ENDPOINT> with the public /graphql/alpha URL

QUERY='{"query":"{ account(input: {entityId: {num: 1}}) { id balance } }"}'

# From each of N IPs, run 5 concurrent workers in a loop:
for i in $(seq 1 5); do
  while true; do
    curl -s -X POST \
      -H "Content-Type: application/json" \
      -d "$QUERY" \
      https://<GRAPHQL_ENDPOINT>/graphql/alpha &
  done
done

# Monitor: watch the HTTP 500 rate approach ~20-24%.
# Observe: legitimate requests experience severe latency or 500 errors.
# Observe: Traefik circuit breaker remains CLOSED (NetworkErrorRatio ≈ 0,
#          ResponseCodeRatio < 0.25).
# The degraded state persists until the attacker stops or an operator intervenes.
```

    @QueryMapping
    Optional<Account> account(@Argument @Valid AccountInput input) {
        final var alias = input.getAlias();
        final var evmAddress = input.getEvmAddress();
        final var entityId = input.getEntityId();
        final var id = input.getId();

        validateOneOf(alias, entityId, evmAddress, id);

        if (entityId != null) {
            return entityService
                    .getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        if (alias != null) {
            return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
        }

        if (evmAddress != null) {
            return entityService
                    .getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)
                    .map(accountMapper::map);
        }

        throw new IllegalStateException("Not implemented");
    }

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
