### Title
Cache-Miss Amplification via Uncached Non-Existent Contract Bytecode Lookups Enabling Sustained DB Resource Exhaustion

### Summary
`ContractRepository.findRuntimeBytecode()` uses `@Cacheable(unless = "#result == null")`, but Spring Framework unwraps `Optional` return values before evaluating the `unless` SpEL expression, meaning `Optional.empty()` (returned for non-existent contracts) is treated as null and is **never cached**. An unprivileged attacker can send a sustained stream of `eth_call` requests targeting unique non-existent contract addresses, each generating an uncached database query, with no per-IP rate limiting to prevent a single client from consuming the full global 500 RPS budget.

### Finding Description

**Exact code path:**

`ContractRepository.java` line 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

`ContractBytecodeReadableKVState.java` lines 39–46:
```java
protected Bytecode readFromDataSource(@NonNull ContractID contractID) {
    final var entityId = toEntityId(contractID);
    return contractRepository
            .findRuntimeBytecode(entityId.getId())
            .map(Bytes::wrap)
            .map(Bytecode::new)
            .orElse(null);
}
```

**Root cause — failed assumption:**

The `unless = "#result == null"` condition assumes the cached method's return value is a raw nullable object. However, the method returns `Optional<byte[]>`. Spring Framework's `CacheAspectSupport` calls `ObjectUtils.unwrapOptional()` on the return value **before** evaluating the `unless` SpEL expression. For a non-existent contract, the query returns `Optional.empty()`, which unwraps to `null`, making `#result == null` evaluate to `true`. The result is therefore **never stored in the Caffeine cache**. Every subsequent call for the same non-existent `contractId` hits the database.

**Cache configuration** (`CacheProperties.java` line 22):
```
expireAfterAccess=1h,maximumSize=1000,recordStats
```
Even for existing contracts, the cache holds only 1,000 entries. Beyond that, LRU eviction causes cache misses for previously-seen addresses.

**Throttle configuration** (`ThrottleProperties.java` lines 35, 26):
- `requestsPerSecond = 500` — **global**, not per-IP or per-client
- `gasPerSecond = 7,500,000,000` — also global; `scaleGas()` returns 0 for gas ≤ 10,000, bypassing the gas bucket entirely for minimal-gas requests

A single unauthenticated client can consume the entire 500 RPS global budget. Each request to a unique non-existent contract address produces one uncached `SELECT runtime_bytecode FROM contract WHERE id = ?` query.

**Per-request in-memory read cache** (`ReadableKVStateBase.java` lines 82–108) only deduplicates lookups **within a single EVM execution**, not across requests. It is reset between calls.

### Impact Explanation

At 500 RPS (the global rate limit), an attacker targeting unique non-existent contract addresses generates 500 uncached database queries per second, sustained indefinitely. For a node with a low baseline query rate (e.g., 50–200 DB queries/second from normal traffic), this represents a 150–900% increase in database load, far exceeding the 30% threshold. The database connection pool, query executor threads, and I/O bandwidth are all affected. Because the queries never cache, the attack does not degrade over time — it remains constant as long as the attacker cycles through new addresses.

### Likelihood Explanation

The attack requires no privileges, no on-chain deployment, and no special tooling — only the ability to send HTTP POST requests to `/api/v1/contracts/call`. The attacker needs only to vary the `to` field across requests (e.g., incrementing a numeric contract ID or generating random EVM addresses). The global rate limit of 500 RPS is reachable from a single machine with a modest HTTP client. The attack is repeatable, requires no state, and is trivially automatable.

### Recommendation

1. **Fix the `unless` condition** to correctly handle `Optional` return types. Replace:
   ```java
   unless = "#result == null"
   ```
   with:
   ```java
   unless = "!#result.isPresent()"
   ```
   This ensures `Optional.empty()` (non-existent contracts) is also cached, preventing repeated DB queries for the same missing address.

2. **Add per-IP or per-client rate limiting** in addition to the global bucket, so a single client cannot consume the full 500 RPS budget.

3. **Increase the contract cache size** or add a dedicated negative-result (null/empty) cache with a shorter TTL (e.g., 10–30 seconds) to absorb repeated lookups for non-existent addresses.

### Proof of Concept

```bash
# Generate N unique non-existent contract addresses and send eth_call requests
# Each request targets a different address, bypassing the 1000-entry cache

for i in $(seq 1 10000); do
  ADDR=$(printf "0x%040x" $i)  # unique address per request
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$ADDR\",\"data\":\"0x\",\"gas\":50000}" &
done
wait
```

**Expected result:** Each request triggers `findRuntimeBytecode()` for a unique non-existent `contractId`. Spring evaluates `unless = "#result == null"` as true (because `Optional.empty()` unwraps to null), so no result is cached. The database receives one `SELECT runtime_bytecode FROM contract WHERE id = ?` query per request. At 500 concurrent RPS, database CPU and I/O increase by >30% above baseline, sustained for the duration of the attack.
