### Title
Prometheus Label Cardinality Explosion via Unmatched URL Paths in MetricsMiddleware

### Summary
`MetricsMiddleware` in `rosetta/app/middleware/metrics.go` delegates route extraction to the weaveworks `middleware.Instrument`, which falls back to the raw `r.URL.Path` when no registered route matches an incoming request. Because there is no sanitization, truncation, or normalization of this value before it is used as a Prometheus label, an unauthenticated attacker can send requests to an unbounded number of unique URL paths, creating a new in-memory time-series entry for every unique path across all four metric vectors. This exhausts Go heap memory and crashes the Rosetta node.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/metrics.go`, lines 76–83:

```go
func MetricsMiddleware(next http.Handler) http.Handler {
    return middleware.Instrument{
        Duration:         requestDurationHistogram,
        InflightRequests: requestInflightGauge,
        RequestBodySize:  requestBytesHistogram,
        ResponseBodySize: responseBytesHistogram,
        RouteMatcher:     next.(middleware.RouteMatcher),   // gorilla/mux router
    }.Wrap(next)
}
```

The four metric vectors all carry a `"route"` label:

- `requestDurationHistogram` — `[]string{"method", "route", "status_code", "ws"}` (6 buckets + sum + count = 8 series per unique label set)
- `requestInflightGauge` — `[]string{"method", "route"}`
- `requestBytesHistogram` — `[]string{"method", "route"}` (5 buckets + 2 = 7 series)
- `responseBytesHistogram` — `[]string{"method", "route"}` (5 buckets + 2 = 7 series)

**Root cause — how the route label is resolved:**

`github.com/weaveworks/common/middleware.Instrument` calls `RouteMatcher.Match(r, &mux.RouteMatch{})` (the gorilla/mux router) to obtain the matched route template. When the request path does not match any registered route, `Match` returns `false` and the library falls back to `r.URL.Path` — the raw, attacker-controlled URL string — as the label value. There is no length cap, allow-list, or normalization applied anywhere in the application before or after this call.

**Failed assumption:**

The code assumes that `RouteMatcher` will always return a finite, pre-registered set of route names. It does not account for the fallback path where unmatched requests inject arbitrary strings into the label namespace.

**Exploit flow:**

1. Attacker sends HTTP requests to paths that do not match any registered Rosetta route (e.g., `POST /aaaa`, `POST /bbbb`, … or `POST /` + random UUID suffix).
2. `middleware.Instrument.Wrap` calls `getRouteName`, which calls `mux.Router.Match` → returns `false` → falls back to `r.URL.Path`.
3. The raw path string is passed to `.With(prometheus.Labels{"route": rawPath})` on each of the four `*Vec` metrics.
4. The Prometheus client library allocates a new `metricWithLabelValues` struct (and, for histograms, one counter per bucket) for every unique label combination and stores it in an internal `sync.Map`.
5. With N unique paths, memory grows as O(N × buckets). At ~100 k unique paths the process RSS exceeds several hundred MB; at millions it OOMs.
6. The Go runtime panics or the OOM killer terminates the process, taking the Rosetta node offline.

### Impact Explanation

A crashed Rosetta node during a Hedera staking reward distribution window prevents exchanges and custodians from querying account balances and submitting reward-claim transactions through the Rosetta API. Staking rewards that are not claimed within the distribution period are forfeited or delayed, constituting direct financial loss to token holders. The attack is a complete denial-of-service against the node with no data-integrity recovery path until the process is restarted and memory is reclaimed — by which time the distribution window may have closed.

### Likelihood Explanation

The Rosetta port is publicly reachable (it is the primary exchange integration endpoint). No authentication is required to send HTTP requests. The attacker needs only a script that generates unique URL paths and sends them at any rate the network allows. The Kubernetes-level Traefik rate-limit (`average: 10` req/s per host) is an optional, separately deployed component not enforced by the application itself; it is absent in bare-metal or Docker deployments. Even at 10 req/s the label table grows by 864 000 unique entries per day. The attack is trivially repeatable and requires no special knowledge of the protocol.

### Recommendation

1. **Normalize the route label before recording metrics.** After calling `RouteMatcher.Match`, if no route is found, replace the label value with a fixed sentinel string such as `"unmatched"` instead of `r.URL.Path`.
2. **Wrap the four `*Vec` metrics with `prometheus.Labels` validation** that rejects or truncates any label value exceeding a fixed length (e.g., 128 bytes).
3. **Apply rate-limiting at the application layer** (not only at the ingress) so that the Rosetta HTTP server itself rejects excessive requests before metrics are recorded.
4. Consider switching to a middleware that performs route-template extraction only from successfully matched routes and never falls back to the raw URL.

### Proof of Concept

```bash
#!/usr/bin/env bash
# Sends 500 000 requests with unique paths to the Rosetta node.
# Each unique path creates new Prometheus label entries in-process memory.
ROSETTA_HOST="http://<rosetta-host>:8080"
for i in $(seq 1 500000); do
  curl -s -o /dev/null -X POST \
    -H "Content-Type: application/json" \
    -d '{}' \
    "${ROSETTA_HOST}/probe-$(cat /proc/sys/kernel/random/uuid)" &
  # Throttle to avoid local socket exhaustion; still fast enough to OOM
  if (( i % 200 == 0 )); then wait; fi
done
wait
```

Monitor the Rosetta process RSS (`/proc/<pid>/status`) or watch `go_memstats_heap_inuse_bytes` in Prometheus; it will grow linearly with the number of unique paths sent. The node will become unresponsive or crash once heap is exhausted.
