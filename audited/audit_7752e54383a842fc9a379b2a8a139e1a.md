### Title
Unsynchronized Cache-Check-Then-DB-Query Race in `findStorageBatch()` Enables Concurrent DB Query Amplification

### Summary
`ContractStateServiceImpl.findStorage()` performs a non-atomic cache check followed by a DB query with no per-key locking or deduplication. An unprivileged attacker sending concurrent requests for the same `contractId`/`key` pair can cause all threads to simultaneously miss the cache and each independently issue a full `findStorageBatch()` DB query, amplifying DB load by up to the global request-rate limit (default 500 req/s) for a single logical slot lookup.

### Finding Description
**Code path:**

`findStorage()` (line 63) reads `contractStateCache` for the slot value. If absent, it calls `findStorageBatch()` (line 69). Inside `findStorageBatch()` (lines 85–122):

- Line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — registers the key in the per-contract slot-key tracking cache (thread-safe, but only prevents duplicate key insertion, not duplicate DB calls).
- Line 91: `cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet()` — snapshot of all tracked keys.
- Line 103: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — **unconditional DB query, no guard against concurrent identical calls**.
- Lines 106–114: Results are written into `contractStateCache`.

**Root cause:** There is no synchronization, per-key locking, or request-coalescing (e.g., Caffeine's `AsyncLoadingCache`) between the cache miss check at line 63 and the DB call at line 103. The check and the query are not atomic.

**Exploit flow:**

1. Attacker sends N concurrent `POST /api/v1/contracts/call` requests, all targeting the same contract address and triggering a read of the same storage slot (e.g., a simple `eth_call` to a getter function).
2. All N threads pass the global rate-limit bucket (default capacity: 500 tokens/s — all consumed simultaneously).
3. All N threads call `findStorage()` and read `contractStateCache.get(key)` → all get `null` (cache cold or expired).
4. All N threads enter `findStorageBatch()`, each calling `contractStateRepository.findStorageBatch()` with an overlapping (or identical) `cachedSlots` list.
5. N identical SQL queries hit the database: `SELECT slot, value FROM contract_state WHERE contract_id = X AND slot IN (...)`.
6. Each thread independently writes the same results back to `contractStateCache` — wasted work, but the DB has already absorbed N queries.

**Why existing checks fail:**

- The global rate-limit bucket (`rateLimitBucket`, 500 req/s) is consumed before the cache check; it limits total throughput but does not prevent concurrent threads from all being in-flight simultaneously within the same second.
- `contractSlotsCache.putIfAbsent()` (line 90) only deduplicates slot-key registration; it does not block or coalesce the subsequent DB call.
- There is no per-`(contractId, key)` lock, `CompletableFuture`-based deduplication, or Caffeine `AsyncLoadingCache` that would cause the second-through-Nth threads to wait for the first thread's DB result. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Each redundant `findStorageBatch()` issues a `SELECT … WHERE slot IN (…)` against the `contract_state` table. Under 500 concurrent requests (the default rate-limit capacity), a single cold-cache slot lookup generates up to 500 simultaneous identical DB queries. This saturates DB connection pools, increases query latency for all users, and degrades or halts transaction simulation (`eth_call`, `eth_estimateGas`) across the node. The impact is a targeted, amplified availability degradation of the DB tier with no authentication required. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
No privileges, API keys, or special knowledge are required — only a publicly accessible `POST /api/v1/contracts/call` endpoint and knowledge of any deployed contract address. The attack is trivially repeatable: the attacker simply keeps the cache cold (by targeting a slot that expires or by targeting many distinct slots) and floods concurrent requests. The default 500 req/s limit is the amplification ceiling, not a prevention mechanism. Any standard HTTP load-testing tool (e.g., `wrk`, `ab`, `hey`) can reproduce this. [6](#0-5) [7](#0-6) 

### Recommendation
Replace the manual cache-check-then-query pattern with Caffeine's `AsyncLoadingCache` or a `LoadingCache`, which natively coalesces concurrent requests for the same key into a single loader invocation. Alternatively, introduce a per-`(contractId, key)` `ConcurrentHashMap`-based in-flight request deduplicator (e.g., using `computeIfAbsent` with a `CompletableFuture`) so that only the first thread issues the DB query and all subsequent threads await its result. The fix must make the cache-miss-to-DB-query path atomic per key. [8](#0-7)

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
