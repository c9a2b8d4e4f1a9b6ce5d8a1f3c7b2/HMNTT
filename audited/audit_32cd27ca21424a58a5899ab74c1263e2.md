### Title
Unauthenticated 3x DB Query Amplification via `getSchedules()` Enables Resource Exhaustion

### Summary
The `getSchedules()` handler in `rest/schedules.js` unconditionally fires three database queries per request — one sequential `schedulesQuery` followed by two concurrent queries (`entityQuery` and `signatureQuery`) via `Promise.all()`. The `signatureQuery` uses an unbounded `json_agg` aggregation with no per-entity signature limit. Because the REST API has no rate-limiting middleware, any unauthenticated caller can sustain amplified DB load with minimal request rate.

### Finding Description

**Code path:** `rest/schedules.js`, `getSchedules()`, lines 234–264.

```
234: const getSchedules = async (req, res) => {
241:   const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);   // Query 1
...
261:   const [{rows: entities}, {rows: signatures}] = await Promise.all([
262:     pool.queryQuietly(entityQuery, entityIds),    // Query 2 (concurrent)
263:     pool.queryQuietly(signatureQuery, entityIds), // Query 3 (concurrent)
264:   ]);
```

**Root cause — three queries per request:**
Every non-empty response triggers exactly three DB queries. The `schedulesQuery` is bounded by `limit`. The `entityQuery` is an `IN(…)` lookup on up to `limit` IDs. The `signatureQuery` is:

```sql
select entity_id, json_agg(json_build_object(
    'consensus_timestamp', ts.consensus_timestamp, ...
  ) order by ts.consensus_timestamp) as signatures
from transaction_signature ts
where entity_id in ($1, …, $N)
group by entity_id
order by entity_id asc
```

There is **no `LIMIT` clause** on the number of signatures aggregated per entity. Every signature row ever recorded for each returned schedule is pulled and aggregated in memory by PostgreSQL.

**Limit is capped but not rate-limited:**
`getLimitParamValue()` in `rest/utils.js` (lines 544–553) caps the user-supplied `limit` at `responseLimit.max`. However, `rest/server.js` (lines 67–98) shows the REST API middleware stack contains only compression, CORS, logging, auth, metrics, and response-cache handlers — **no request-rate-limiting middleware**. The throttle infrastructure found in the codebase (`web3/src/main/java/…/ThrottleConfiguration.java`) applies exclusively to the web3 Java module, not the Node.js REST API.

**Failed assumption:** The design assumes that capping `limit` at `responseLimit.max` is sufficient to bound per-request DB cost. It does not account for (a) the 3× query multiplier, (b) the unbounded `json_agg` in `signatureQuery`, or (c) the absence of any request-rate control.

### Impact Explanation

An attacker sending `C` concurrent requests each with `limit=MAX` forces the DB to service `3×C` simultaneous queries. The `signatureQuery` with `json_agg` can be disproportionately expensive if any returned schedule has accumulated many signatures (e.g., multi-sig schedules with large threshold keys). Sustained load can exhaust the PostgreSQL connection pool (`db.pool.maxConnections`), cause statement timeouts, and degrade or deny service to all API consumers. This is a public, unauthenticated endpoint.

### Likelihood Explanation

No authentication or API key is required. The endpoint is publicly documented in `rest/api/v1/openapi.yml` (line 1030). A single attacker with a modest HTTP client can sustain hundreds of requests per second. The amplification factor is deterministic and repeatable. The only friction is network bandwidth, which is negligible for small HTTP GET requests.

### Recommendation

1. **Add rate limiting to the REST API** — integrate a middleware such as `express-rate-limit` in `rest/server.js`, keyed by IP or API key, applied globally or specifically to expensive list endpoints.
2. **Limit signatures per schedule in `signatureQuery`** — add a `LIMIT` inside the `json_agg` window or use a subquery with `FETCH FIRST N ROWS ONLY` to cap the number of signatures returned per entity.
3. **Consider merging the three queries** — the single-schedule `getScheduleByIdQuery` already uses `LEFT JOIN` to fetch entity and signature data in one query; the same pattern could be applied to the list endpoint using a lateral join or CTE, eliminating the 3× multiplier entirely.

### Proof of Concept

```bash
# Send 50 concurrent requests each requesting the maximum allowed schedules
# (replace MAX with the configured responseLimit.max, typically 100)
seq 50 | xargs -P50 -I{} \
  curl -s "https://<mirror-node-host>/api/v1/schedules?limit=MAX" -o /dev/null

# Observe: DB connection pool saturation, rising query latency,
# and eventual 503/timeout responses for legitimate users.
```

Each request triggers:
- 1× `SELECT … FROM schedule … LIMIT MAX`
- 1× `SELECT … FROM entity WHERE id IN (…MAX ids…)`
- 1× `SELECT entity_id, json_agg(…) FROM transaction_signature WHERE entity_id IN (…MAX ids…) GROUP BY entity_id` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

const getSchedules = async (req, res) => {
  // extract filters from query param
  const filters = utils.buildAndValidateFilters(req.query, acceptedSchedulesParameters);

  // get sql filter query, params, order and limit from query filters
  const {filterQuery, params, order, limit} = extractSqlFromScheduleFilters(filters);
  const schedulesQuery = getSchedulesQuery(filterQuery, order, params.length);
  const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);

  const schedulesResponse = {schedules: [], links: {next: null}};
  res.locals[constants.responseDataLabel] = schedulesResponse;

  if (schedules.length === 0) {
    return;
  }

  const entityIds = schedules.map((s) => s.schedule_id);
  const positions = range(1, entityIds.length + 1)
    .map((i) => `$${i}`)
    .join(',');
  const entityQuery = `select ${entityFields} from entity where id in (${positions}) order by id ${order}`;
  const signatureQuery = `select entity_id, ${transactionSignatureJsonAgg} as signatures
    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;

  const [{rows: entities}, {rows: signatures}] = await Promise.all([
    pool.queryQuietly(entityQuery, entityIds),
    pool.queryQuietly(signatureQuery, entityIds),
  ]);

const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;
  }
  return ret;
};
