### Title
GraphQL Endpoint Missing Rate Limiting Enables Thread Pool Exhaustion via Zero-Prefix EVM Address Burst During DB Unavailability

### Summary
The `getByEvmAddressAndType()` method in `EntityServiceImpl` routes zero-prefix EVM addresses (first 12 bytes all zero) to a synchronous blocking `entityRepository.findById()` call. The GraphQL module has no per-request rate limiting — unlike the `web3` module which has an explicit `ThrottleManagerImpl` — so an unprivileged attacker can flood the publicly accessible GraphQL endpoint with valid zero-prefix EVM address queries. During a network partition or sustained DB slowdown, each request holds a Tomcat thread for the full HikariCP connection-timeout duration (30 seconds by default), exhausting the thread pool and rendering the service completely unresponsive.

### Finding Description

**Exact code path:**

`AccountController.account()` (lines 51–54) receives any GraphQL `account` query with an `evmAddress` argument and calls `entityService.getByEvmAddressAndType()` with no rate check: [1](#0-0) 

`EntityServiceImpl.getByEvmAddressAndType()` (lines 37–38) detects the zero-prefix format and routes to `entityRepository.findById()`, a synchronous blocking JDBC call: [2](#0-1) 

**Root cause — missing rate limiting in the GraphQL module:**

`GraphQlConfiguration` only installs `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, plus JSON/parser size caps. There is no requests-per-second bucket, no per-IP limit, and no concurrency cap: [3](#0-2) 

By contrast, the `web3` module has an explicit `ThrottleManagerImpl` with a `rateLimitBucket` (default 500 req/s) applied before any DB call: [4](#0-3) 

**No statement/query timeout for runtime queries:**

`CommonProperties.DatabaseStartupProperties` defines `connectTimeout` and `socketTimeout`, but these are scoped exclusively to the `DatabaseWaiter` startup probe, not to the main HikariCP pool's runtime connections: [5](#0-4) 

The main pool is built from `@ConfigurationProperties("spring.datasource.hikari")` with no explicit `connectionTimeout` or `statementTimeout` set in code, leaving HikariCP at its 30-second default: [6](#0-5) 

**Exploit flow:**

1. Attacker crafts a valid zero-prefix EVM address: `0x000000000000000000000000<any-8-bytes>` — passes the `@Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")` schema validation.
2. During a network partition (or induced DB slowdown), the attacker sends N concurrent GraphQL POST requests to `/graphql`.
3. Each request passes complexity/depth checks (single-field query, complexity = 1), reaches `getByEvmAddressAndType()`, and blocks on `entityRepository.findById()` waiting for a DB connection for up to 30 seconds.
4. With N ≥ Tomcat's thread pool size (default 200), all threads are occupied; the service stops accepting new connections entirely. [7](#0-6) 

### Impact Explanation

Complete denial of service for the GraphQL endpoint. All legitimate users are blocked for the duration of the attack (sustained as long as the attacker keeps sending requests and the DB remains slow/unreachable). Because the GraphQL service shares infrastructure with other mirror-node consumers, cascading impact to dependent services is possible. Severity: **High** (unauthenticated, no privileges required, full availability loss).

### Likelihood Explanation

The attacker needs only: (a) network access to the public GraphQL endpoint, and (b) a DB slowdown condition — which can be a natural partition, a separate DB-layer DoS, or even self-induced by the same flood (connection pool saturation causes DB slowness for all callers). The zero-prefix address format is documented behavior, not an obscure edge case. Sending 200+ concurrent HTTP requests is trivially achievable with standard tools (`ab`, `wrk`, `hey`). No credentials, tokens, or prior knowledge of entity IDs are required.

### Recommendation

1. **Add rate limiting to the GraphQL module** mirroring the `web3` module's `ThrottleManagerImpl` / `ThrottleConfiguration` pattern — a per-IP or global requests-per-second bucket applied before any resolver executes.
2. **Set a JDBC statement timeout** on the HikariCP pool used by the GraphQL module (e.g., `spring.datasource.hikari.connection-timeout=5000` and a PostgreSQL `statement_timeout` via `spring.datasource.hikari.data-source-properties.socketTimeout`), so blocked DB calls fail fast rather than holding threads for 30 seconds.
3. **Cap Tomcat's max threads and add a request queue limit** so the service sheds load gracefully rather than silently queuing.

### Proof of Concept

```bash
# During a network partition or DB slowdown, run from any machine with HTTP access:
# Zero-prefix EVM address: first 4 bytes (int=0) + next 8 bytes (long=0) + 8-byte entity id
EVM="0x0000000000000000000000000000000000000001"

QUERY='{"query":"{ account(input: { evmAddress: \"'$EVM'\" }) { entityId { num } } }"}'

# Send 250 concurrent requests (exceeds default Tomcat thread pool of 200)
for i in $(seq 1 250); do
  curl -s -X POST http://<graphql-host>/graphql \
    -H "Content-Type: application/json" \
    -d "$QUERY" &
done
wait

# Expected result: service stops responding to all subsequent requests
# for up to 30 seconds per wave (HikariCP default connection timeout),
# with no 429 or throttle response ever returned.
```

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }

    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
