### Title
Unauthenticated DoS via Non-Existent Account ID Flooding Exhausts DB Connection Pool in `AccountBalance()`

### Summary
`AccountBalance()` in `rosetta/app/services/account_service.go` performs no account-existence pre-check before invoking `RetrieveBalanceAtBlock()`. For any syntactically valid but non-existent numeric account ID (e.g., `0.0.999999999`), the persistence layer unconditionally executes four sequential database queries against large partitioned tables. The Traefik rate-limiting and in-flight-request middleware chain is disabled by default (`global.middleware: false`), and even when enabled the rate limit is keyed on the `Host` request header rather than source IP, making it trivially bypassable. An unprivileged attacker can flood the endpoint to exhaust the shared PostgreSQL connection pool, degrading or halting the mirror node's gossip ingestion writer.

### Finding Description

**Code path:**

`AccountBalance()` at [1](#0-0)  parses the address string (only format-validates it), resolves a block, then unconditionally calls `RetrieveBalanceAtBlock()`.

Inside `RetrieveBalanceAtBlock()` at [2](#0-1) , the early-exit guard at line 172 is:

```go
if entity == nil && accountId.HasAlias() {
    return types.AmountSlice{&types.HbarAmount{}}, ...
}
``` [3](#0-2) 

This guard fires **only for alias-type account IDs**. For a numeric ID like `0.0.999999999`, `HasAlias()` is false, so even when `getCryptoEntity()` returns `nil` (account not found), execution falls through to `getLatestBalanceSnapshot()` and `getBalanceChange()`.

**Four DB queries execute per request for a non-existent numeric account:**

1. `selectCryptoEntityById` — UNION ALL across `entity` + `entity_history` with range operator `@>` [4](#0-3) 
2. `selectPreviousPartitionLowerBound` — CTE scan over `mirror_node_time_partitions` [5](#0-4) 
3. `latestBalanceBeforeConsensus` — correlated subquery scanning `account_balance` partitions [6](#0-5) 
4. `balanceChangeBetween` — `SUM` over `crypto_transfer` filtered by `entity_id` and timestamp range [7](#0-6) 

All four return empty/zero results but still acquire a DB connection and execute query plans against large tables.

**Existing protections are insufficient:**

The Traefik middleware chain (circuit breaker, in-flight limiter, rate limiter) is gated by:

```yaml
{{ if and .Values.global.middleware .Values.middleware -}}
``` [8](#0-7) 

with the default:

```yaml
global:
  middleware: false
``` [9](#0-8) 

So the entire middleware chain — including `inFlightReq: amount: 5` and `rateLimit: average: 10` — is **not deployed by default**. [10](#0-9) 

Even when manually enabled, the rate limit is keyed on `requestHost` (the `Host` header), not source IP. An attacker rotating the `Host` header value bypasses the per-host bucket entirely. [11](#0-10) 

There is no application-level rate limiting, caching, or account-existence short-circuit anywhere in the Go service layer.

### Impact Explanation

The Rosetta API and the mirror node's gossip importer share the same PostgreSQL instance. PostgreSQL connection pools are finite. Each flood request holds a connection for the duration of four sequential queries against `entity`, `entity_history`, `account_balance` (partitioned), and `crypto_transfer` (partitioned). At sufficient request rate, the pool is exhausted: the gossip importer's write transactions queue behind the pool limit, stalling consensus record ingestion. Even below pool exhaustion, I/O saturation from full-table scans on large partitioned tables degrades write throughput. The endpoint returns HTTP 200 with a zero balance for non-existent accounts, so no error-ratio circuit breaker triggers.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge beyond the public Rosetta API spec. The attacker needs only a valid block index (obtainable from `/network/status`) and any syntactically valid but non-existent account number. The attack is trivially scriptable, repeatable, and can be distributed across multiple source IPs to defeat even correctly-configured per-IP limits. Default deployments via the published Helm chart have no rate limiting active.

### Recommendation

1. **Add an account-existence short-circuit for numeric IDs** in `RetrieveBalanceAtBlock()`: mirror the alias path — if `getCryptoEntity()` returns `nil` for a numeric ID, return a zero balance immediately without executing the snapshot and transfer queries.
2. **Enable the middleware by default**: change `global.middleware` default to `true` in `values.yaml`, or document it as a required production setting.
3. **Key rate limiting on source IP**, not `requestHost`, to prevent Host-header bypass.
4. **Add application-level concurrency limiting** (e.g., a semaphore or Go `golang.org/x/time/rate` limiter) inside the service, independent of the ingress layer.
5. **Cache block lookups**: `RetrieveBlock()` is called on every request; a short TTL cache for recent block identifiers reduces the query count further.

### Proof of Concept

```bash
# 1. Obtain a valid block index
BLOCK=$(curl -s -X POST http://<rosetta-host>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq .current_block_identifier.index)

# 2. Flood with non-existent numeric account IDs (no auth required)
for i in $(seq 1 10000); do
  curl -s -X POST http://<rosetta-host>/account/balance \
    -H "Content-Type: application/json" \
    -H "Host: attacker-host-$i.example.com" \
    -d "{
      \"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
      \"account_identifier\":{\"address\":\"0.0.$((RANDOM + 900000000))\"},
      \"block_identifier\":{\"index\":$BLOCK}
    }" &
done
# Each request triggers 4 DB queries against large partitioned tables.
# With default Helm deployment (global.middleware: false), no rate limiting applies.
# Host header rotation defeats per-host rate limits even when middleware is enabled.
```

func (ar *accountRepository) RetrieveBalanceAtBlock(
	ctx context.Context,
	accountId types.AccountId,
	consensusEnd int64,
) (types.AmountSlice, string, []byte, *rTypes.Error) {
	var entityIdString string
	entity, err := ar.getCryptoEntity(ctx, accountId, consensusEnd)
	if err != nil {
		return nil, entityIdString, nil, err
	}

	if entity == nil && accountId.HasAlias() {
		return types.AmountSlice{&types.HbarAmount{}}, entityIdString, nil, nil
	}

	balanceChangeEndTimestamp := consensusEnd
	balanceSnapshotEndTimestamp := consensusEnd
	if entity != nil && entity.Deleted != nil && *entity.Deleted && entity.GetModifiedTimestamp() <= consensusEnd {
		// if an account / contract is deleted at t1, a balance snapshot at t1 (if exists) won't have info for the
		// entity, thus look for a balance snapshot at or before the deleted timestamp - 1
		// however, the balanceChangeEndTimestamp should be the deletion timestamp since the crypto delete transaction
		// may have a transfer which moves the remaining hbar balance to another account
		balanceChangeEndTimestamp = entity.GetModifiedTimestamp()
		balanceSnapshotEndTimestamp = balanceChangeEndTimestamp - 1
	}

	id := accountId.GetId()
	if accountId.HasAlias() {
		// entity can't be nil if accountId has alias
		id = entity.Id.EncodedId
	}
	snapshotTimestamp, hbarAmount, err := ar.getLatestBalanceSnapshot(
		ctx,
		id,
		balanceSnapshotEndTimestamp,
	)
	if err != nil {
		return nil, entityIdString, nil, err
	}

	hbarValue, err := ar.getBalanceChange(
		ctx,
		id,
		snapshotTimestamp,
		balanceChangeEndTimestamp,
	)
	if err != nil {
		return nil, entityIdString, nil, err
	}

	hbarAmount.Value += hbarValue

	amounts := make(types.AmountSlice, 0, 1)
	amounts = append(amounts, hbarAmount)

	var publicKey []byte
	if entity != nil {
		// return the entity id string in the format of 'shard.realm.num'
		entityIdString = entity.Id.String()
		publicKey = entity.Key
	}
	return amounts, entityIdString, publicKey, nil
}
