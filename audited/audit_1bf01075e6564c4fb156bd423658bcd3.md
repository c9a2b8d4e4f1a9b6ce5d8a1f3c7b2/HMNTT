### Title
Integer Overflow in `poll()` Bypasses DB Query Limit, Enabling Unlimited Message Retrieval

### Summary
In `PollingTopicMessageRetriever.poll()`, the remaining limit is computed as `(int)(filter.getLimit() - context.getTotal().get())`. When a user supplies a `limit` value between `Integer.MAX_VALUE + 1` and `Long.MAX_VALUE`, the narrowing cast produces a negative `int`. This negative value causes `hasLimit()` to return `false` on the derived filter, so `TypedQuery.setMaxResults()` is never called and the database returns all topic messages without any row limit.

### Finding Description
**Exact code path:**

`ConsensusController.toFilter()` passes `query.getLimit()` (a proto `uint64` → Java `long`) directly to `TopicMessageFilter.builder().limit(...)`. [1](#0-0) 

`TopicMessageFilter.limit` is annotated `@Min(0)` only — no upper-bound constraint — so any value in `[1, Long.MAX_VALUE]` passes validation. [2](#0-1) 

In `poll()`, the narrowing cast overflows for any `filter.getLimit() > Integer.MAX_VALUE`: [3](#0-2) 

The negative `pageSize` is written into a new filter via `toBuilder().limit(pageSize).build()` — this builder call is **not** intercepted by Spring's `@Valid` AOP proxy: [4](#0-3) 

In `TopicMessageRepositoryCustomImpl.findByFilter()`, `hasLimit()` returns `limit > 0`, which is `false` for the negative value, so `setMaxResults()` is never called: [5](#0-4) 

The JPA query executes with no row limit, returning every message for the topic into memory.

**Why existing checks fail:**
- `@Min(0)` on `TopicMessageFilter.limit` only rejects negative user input; it does not enforce `limit <= Integer.MAX_VALUE`.
- `hasLimit()` (`limit > 0`) correctly guards the original filter but is re-evaluated on the derived filter where `limit` is now negative.
- The outer `flux.take(filter.getLimit())` in `TopicMessageServiceImpl` operates at the reactive layer after the DB has already loaded all rows. [6](#0-5) 

### Impact Explanation
An attacker can force the mirror node to issue unbounded `SELECT *` queries against the `topic_message` table for any topic. For a high-volume topic with millions of messages, each such query loads the entire result set into the JVM heap, causing memory exhaustion, GC pressure, and potential OOM crashes. This degrades or denies service for all other subscribers. The `isComplete()` check also misbehaves: with `total` never reaching `2147483648`, polling continues indefinitely, compounding the load. [7](#0-6) 

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication. The proto `limit` field is `uint64`, so any gRPC client can trivially set it to `2147483648`. No special privileges, credentials, or knowledge of internal state are required. The attack is repeatable and can be automated.

### Recommendation
1. **Add an upper-bound validation** on `TopicMessageFilter.limit`: `@Max(Integer.MAX_VALUE)` or a custom constraint, so values > `Integer.MAX_VALUE` are rejected at the service boundary.
2. **Eliminate the narrowing cast**: compute the page limit as a `long` and clamp it safely before casting:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
3. **Validate the derived filter**: pass the `newFilter` through the same validation path, or add a defensive check `if (pageSize <= 0) return Flux.empty();` before issuing the query.
