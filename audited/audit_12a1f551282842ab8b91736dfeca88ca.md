### Title
Unauthenticated Connection Pool Exhaustion DoS on `/api/v1/network/registered-nodes` via Absent Rate Limiting in rest-java

### Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` service has no application-level rate limiting. Each request acquires a HikariCP database connection to execute the `findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()` native query. An unauthenticated attacker flooding this endpoint with concurrent requests exhausts the finite connection pool, causing all subsequent requests â€” including those from registered nodes performing peer discovery â€” to fail with connection timeout errors.

### Finding Description
**Code path:**

`NetworkController.getRegisteredNodes()` (lines 173â€“187) calls `NetworkServiceImpl.getRegisteredNodes()` (lines 139â€“152), which calls `RegisteredNodeRepository.findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()` (lines 21â€“22). This executes a native SQL query holding a HikariCP connection for the full query duration.

**Root cause â€” no rate limiting in rest-java:**

The `web3` module has a full throttle stack (`ThrottleConfiguration`, `ThrottleManagerImpl`, bucket4j) but the `rest-java` module has **none**. The only servlet filters registered in rest-java are `LoggingFilter` and `MetricsFilter` â€” neither enforces any request rate limit. `RestJavaConfiguration` registers only an ETag filter and a Protobuf converter.

**Connection pool:**

`CommonConfiguration` creates a `HikariDataSource` from `spring.datasource.hikari` properties. No `maximumPoolSize` is set anywhere in the codebase's YAML files, so HikariCP defaults to **10 connections**. The Helm chart's Prometheus alert fires at 75% pool utilization, confirming the pool is small and finite.

**Request parameters provide no protection:**

`RegisteredNodesRequest` enforces `@Max(MAX_LIMIT)` on the `limit` field, bounding result set size per request, but places no constraint on request concurrency or frequency. The `registeredNodeIds` range parameters accept arbitrary `lowerBound`/`upperBound` values (defaulting to `0` and `Long.MAX_VALUE`), meaning each request can trigger a full table scan.

**Why existing mitigations are insufficient:**

- The GCP gateway `maxRatePerEndpoint: 250` is a backend load-balancing hint, not a hard rate limit, and only applies when `gcp.enabled: true` is configured â€” it is not an application-level control.
- Tomcat's default thread pool (200 threads) queues requests, but threads blocked waiting for a HikariCP connection (default `connectionTimeout` = 30 s) still hold Tomcat threads, making the entire service unresponsive under sustained flood.
- No per-IP, per-endpoint, or global request-rate enforcement exists in the rest-java application code.

### Impact Explanation
With the HikariCP pool exhausted, all requests to the mirror node â€” not just `/registered-nodes` â€” begin failing with `SQLTransientConnectionException` (connection pool timeout). Registered nodes (block nodes, mirror nodes, RPC relays) that poll this endpoint for peer discovery receive errors and cannot obtain updated service endpoint information. If the attack is sustained, â‰Ą30% of registered nodes lose access to current peer data, degrading the network's ability to route and discover services. The impact is a targeted, protocol-level availability disruption without requiring any brute-force or cryptographic attack.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and zero privileged access â€” only the ability to send HTTP GET requests. A single attacker with a modest number of concurrent connections (e.g., 50â€“100) is sufficient to saturate a 10-connection HikariCP pool. The attack is trivially repeatable, scriptable with standard tools (`ab`, `wrk`, `hey`), and can be sustained indefinitely. The endpoint is publicly documented in the OpenAPI spec.

### Recommendation
1. **Add application-level rate limiting to rest-java** using the same bucket4j pattern already present in `web3`: create a `ThrottleConfiguration` bean and a servlet filter or Spring interceptor that enforces a per-IP or global requests-per-second limit on `/api/v1/network/*` endpoints.
2. **Set an explicit `spring.datasource.hikari.maximum-pool-size`** appropriate for the expected concurrency, and configure `connectionTimeout` to fail fast rather than queue indefinitely.
3. **Add a Tomcat/Spring `server.tomcat.max-connections` and `server.tomcat.accept-count` limit** to prevent thread exhaustion from queued requests.
4. **Enforce the GCP gateway `maxRatePerEndpoint` as a hard limit** and add a corresponding ingress-level rate-limit annotation as a defense-in-depth layer.
