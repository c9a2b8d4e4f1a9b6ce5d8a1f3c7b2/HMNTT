### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Concurrent `/block` Requests

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` performs multiple synchronous database queries per request (block lookup + `FindBetween()` transaction scan) with no application-level rate limiting, concurrency cap, or response cache. An unprivileged attacker sending a high volume of concurrent requests targeting distinct valid block indices can hold all available database connections simultaneously, exhausting the pool and denying service to all users.

### Finding Description

**Exact code path:**

`Block()` at `rosetta/app/services/block_service.go:51-56` unconditionally calls:
1. `s.RetrieveBlock()` → `blockRepo.FindByIndex()` → DB query
2. `s.FindBetween()` → `transactionRepo.FindBetween()` → **one or more DB queries in a loop**

`FindBetween()` at `rosetta/app/persistence/transaction.go:112-147` acquires a DB connection via `tr.dbClient.GetDbWithContext(ctx)` (line 120) and then executes a batched loop:

```go
for start <= end {
    err := db.Raw(selectTransactionsInTimestampRangeOrdered, ...).Limit(batchSize).Find(&transactionsBatch).Error
    ...
    if len(transactionsBatch) < batchSize { break }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```

For blocks with many transactions, this loop issues multiple sequential DB queries while holding the connection. The connection is held for the entire duration of the request (up to `statementTimeout: 20s` per query, per `docs/configuration.md:662`).

**DB pool configuration** (`rosetta/app/db/db.go:31-33`):
```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)   // default: 20
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)   // default: 100
```
Default `maxOpenConnections` is 100 (`docs/configuration.md:660`).

**Root cause:** No application-level rate limiting, no per-IP concurrency cap, and no block-level response cache exist within the Rosetta Go service itself. Every `Block()` call unconditionally consumes at least one DB connection for the full query duration.

**Why existing checks fail:**

The only mitigations are in the optional Traefik ingress middleware (`charts/hedera-mirror-rosetta/values.yaml:152-160`):
- `inFlightReq: amount: 5` per source IP — bypassed trivially with 20+ attacker IPs or a botnet
- `rateLimit: average: 10` per `requestHost` — at 10 req/s with a 20s statement timeout, up to 200 connection-seconds/second are consumed, exceeding the 100-connection pool

Critically, these are Kubernetes/Helm deployment artifacts — they are **not enforced by the application**. Bare-metal, Docker Compose, or non-Traefik deployments have zero protection.

### Impact Explanation
When the 100-connection pool is exhausted, all subsequent DB operations across the entire Rosetta service block or fail, including network status checks, account queries, and construction endpoints. Legitimate users receive database errors or indefinite hangs. The impact is complete denial of service for the Rosetta API, affecting any downstream exchange or indexer relying on it for block data.

### Likelihood Explanation
The `/block` endpoint requires no authentication or API key. Any external user can send POST requests to `/block` with sequential valid block index values. A single machine with 100+ concurrent HTTP connections targeting distinct block indices is sufficient to exhaust the pool. The attack is trivially repeatable and requires no special knowledge beyond knowing the endpoint exists and that block indices are sequential integers starting from 0.

### Recommendation
1. **Application-level concurrency limit:** Add a semaphore or `golang.org/x/sync/semaphore` guard inside `Block()` to cap concurrent in-flight DB-consuming requests (e.g., 20).
2. **Block-level response cache:** Cache `BlockResponse` keyed by block index/hash with a short TTL (e.g., 5s). Finalized blocks are immutable — repeated queries for the same block are pure waste.
3. **DB connection timeout on acquire:** Configure `SetConnMaxIdleTime` and ensure `GetDbWithContext` propagates a short deadline so pool-wait failures surface quickly rather than queuing indefinitely.
4. **Do not rely solely on infrastructure middleware** for application-level resource protection. The Traefik `inFlightReq`/`rateLimit` settings should be treated as defense-in-depth, not primary protection.

### Proof of Concept

```bash
# Enumerate valid block indices (blocks are sequential from genesis)
# Send 200 concurrent Block() requests targeting distinct indices

for i in $(seq 1 200); do
  curl -s -X POST http://<rosetta-host>:5700/block \
    -H "Content-Type: application/json" \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$i}}" &
done
wait
```

**Expected result:** After ~100 concurrent requests are in-flight, the DB connection pool (`maxOpenConnections=100`) is exhausted. All subsequent requests — including from legitimate users — receive `ErrDatabaseError` or hang until the 20-second statement timeout expires per connection. The service is effectively unavailable for the duration of the attack, which can be sustained indefinitely by the attacker re-issuing requests as connections free up.
