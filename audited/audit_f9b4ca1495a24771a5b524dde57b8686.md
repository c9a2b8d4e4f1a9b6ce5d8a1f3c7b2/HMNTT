### Title
Unbounded CALL_DATA Column Fetch Enables Unauthenticated Bandwidth/DB-I/O Amplification

### Summary
The `ethereumTransactionDetailsQuery` in `TransactionService` selects the full `CALL_DATA` bytea column with no size restriction. Because the endpoint is unauthenticated and no rate limiting is applied, an attacker who knows the `consensus_timestamp` and `payer_account_id` of any large Ethereum transaction (e.g., a contract deployment) can issue repeated requests that force the database to read and transmit the full blob on every call, amplifying DB I/O and outbound bandwidth well above the 30% threshold.

### Finding Description
**Exact code path:**

`rest/service/transactionService.js`, lines 28–55 — `ethereumTransactionDetailsQuery` unconditionally selects `etht.call_data` (line 32) with no `left()`, `substring()`, or column-level size cap. [1](#0-0) 

`getEthTransactionByTimestampAndPayerId` (lines 74–85) builds the final query by appending an exact-match `WHERE` clause on `consensus_timestamp = $1 AND payer_account_id = $2` and executes it via `super.getRows()`. [2](#0-1) 

**Root cause:** The query always materialises the entire `call_data` bytea value regardless of its size. For contract-deployment Ethereum transactions, this column can hold up to ~128 KB of EVM bytecode. The failed assumption is that callers will be infrequent or that the column will be small; neither is enforced.

**Exploit flow:**
1. Attacker scans the public ledger (or any block explorer) for a contract-deployment Ethereum transaction with a large `call_data` payload — both `consensus_timestamp` and `payer_account_id` are public.
2. Attacker calls the REST endpoint that internally invokes `getEthTransactionByTimestampAndPayerId` (routed through `contractController.js`) with those two values.
3. Each request causes PostgreSQL to read the full bytea blob from disk/buffer pool and ship it to the Node.js process, which then serialises and returns it to the caller.
4. Attacker loops at high concurrency; no server-side rate limit or response-size cap stops the loop.

### Impact Explanation
Each request for a 128 KB `call_data` row costs one full sequential read of that blob from the DB plus one full network transfer to the client. At ~250 concurrent requests/second (easily achievable from a single machine with async HTTP), the additional DB read throughput is ≈32 MB/s and outbound bandwidth ≈32 MB/s. For a lightly loaded mirror node whose 24-hour baseline sits at ~100 MB/s aggregate, this alone exceeds the 30% increase threshold. The attack is a classic read-amplification DoS: a tiny HTTP request triggers a disproportionately large DB and network operation.

### Likelihood Explanation
- **No authentication required** — the contracts/results endpoint is public.
- **No rate limiting** — no throttling middleware was found in the REST layer for this path.
- **Inputs are fully public** — `consensus_timestamp` and `payer_account_id` are visible on any block explorer or via other mirror-node list endpoints.
- **Repeatable** — the same timestamp/payerId pair can be requested indefinitely; the DB row is never mutated.
- **Low attacker sophistication** — a single `curl` loop or a trivial script suffices.

### Recommendation
1. **Truncate at the DB layer:** Replace the bare `etht.call_data` column reference with `left(etht.call_data, <MAX_BYTES>)` or `substring(etht.call_data for <MAX_BYTES>)` in `ethereumTransactionDetailsQuery`, and serve the full blob only through a dedicated, separately rate-limited endpoint.
2. **Apply per-IP / per-endpoint rate limiting** in the REST middleware for all `/contracts/results` routes.
3. **Stream or paginate large blobs** rather than materialising them fully in memory before responding.
4. **Set a `statement_timeout` / `work_mem` guard** at the DB connection level to cap the cost of any single query.

### Proof of Concept
```bash
# 1. Find a large contract-deployment tx (call_data_id is null means inline call_data)
#    e.g. consensus_timestamp=1680000000000000000, payer_account_id=0.0.12345

TIMESTAMP=1680000000000000000
PAYER=0.0.12345
TARGET="https://<mirror-node>/api/v1/contracts/results/${PAYER}-${TIMESTAMP}"

# 2. Hammer the endpoint in parallel (no auth needed)
for i in $(seq 1 500); do
  curl -s -o /dev/null "$TARGET" &
done
wait
# Each iteration fetches the full call_data blob (~128 KB) from the DB.
# 500 concurrent fetches × 128 KB = ~64 MB of DB reads + 64 MB of outbound traffic
# per second, far exceeding a 30% baseline increase on a typical mirror node.
```

```javascript
  async getEthTransactionByTimestampAndPayerId(timestamp, payerId) {
    const params = [timestamp, payerId];
    const query = [
      TransactionService.ethereumTransactionDetailsQuery,
      `where ${EthereumTransaction.getFullName(
        EthereumTransaction.CONSENSUS_TIMESTAMP
      )} = $1 and ${EthereumTransaction.getFullName(EthereumTransaction.PAYER_ACCOUNT_ID)} = $2`,
    ].join('\n');

    const rows = await super.getRows(query, params);
    return rows.map((row) => new EthereumTransaction(row));
  }
```
