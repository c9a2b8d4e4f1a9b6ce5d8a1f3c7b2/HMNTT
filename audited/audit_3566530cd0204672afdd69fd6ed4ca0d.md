### Title
Stale Replica Read in `findByAlias()` Bypasses Entity Deletion Filter Due to Missing pgpool Load-Balance Guard in REST-Java

### Summary
The `rest-java` module's `EntityRepository.findByAlias()` issues a plain SQL read with no mechanism to force routing to the primary PostgreSQL node. When pgpool (or equivalent middleware) is deployed and a replication lag exists, the query can be served by a replica that has not yet applied a recent `deleted = true` update, causing the method to return an entity ID for a logically deleted account. The importer module explicitly solves this identical problem with a `/* NO PGPOOL LOAD BALANCE */` statement-inspector comment, but that protection is entirely absent from `rest-java`.

### Finding Description

**Exact code location — the vulnerable query:**

`rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java`, lines 13-14:

```java
@Query(value = "select id from entity where alias = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByAlias(byte[] alias);
``` [1](#0-0) 

The `deleted <> true` predicate is correct in isolation, but it only works if the replica being queried has already applied the WAL entry that set `deleted = true` on the primary.

**The existing fix in the importer — absent in rest-java:**

`importer/src/main/java/org/hiero/mirror/importer/config/HibernateConfiguration.java`, lines 19-35:

```java
private static final String NO_LOAD_BALANCE = "/* NO PGPOOL LOAD BALANCE */\n";
// ...
// pgpool disables load balancing for SQL statements beginning with an arbitrary comment
// and sends them to the primary node. This is used to prevent the stale read-after-write issue.
private StatementInspector statementInspector() {
    return sql -> NO_LOAD_BALANCE + sql;
}
``` [2](#0-1) 

The importer's `DBProperties` exposes a `loadBalance` flag that gates this inspector. When `loadBalance = false`, every Hibernate-issued SQL is prefixed with the comment, forcing pgpool to send it to the primary. [3](#0-2) 

**rest-java has no equivalent protection:**

The `rest-java/src/main/java/org/hiero/mirror/restjava/config/` directory contains `JacksonConfiguration`, `MetricsConfiguration`, `RestJavaConfiguration`, `WebMvcConfiguration`, etc., but **no `HibernateConfiguration`**, no `StatementInspector`, and no `loadBalance` property. A grep for `NO PGPOOL LOAD BALANCE`, `loadBalance`, `StatementInspector`, and `pgpool` across the entire `rest-java/` tree returns zero matches.



**Call chain that exposes the stale ID to callers:**

`EntityServiceImpl.lookup()` calls `findByAlias()` directly and propagates the returned `EntityId` to the controller with no secondary freshness check:

```java
case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
``` [4](#0-3) 

Because rest-java also has no `HibernateConfiguration`, every subsequent repository call in the same request is equally eligible for pgpool routing to the stale replica, so the entire request is served from stale state.

### Impact Explanation

An unprivileged caller who supplies the alias of a recently deleted account receives a valid `EntityId` back from the API. All downstream REST-Java endpoints that accept an account ID (token balances, NFT holdings, allowances, etc.) will then serve data for a logically deleted account as if it were active. This constitutes:

- **Information disclosure**: data that should be inaccessible after account deletion is returned.
- **Integrity violation**: the API presents an inconsistent view of ledger state — the account is deleted on the Hedera network and on the primary DB, but the mirror node reports it as live.
- **Application-level exploitation**: any downstream system that uses the mirror node to gate access (e.g., "does this alias resolve to an active account?") can be deceived into granting access for a deleted identity.

Severity is **medium** — the mirror node is read-only and cannot submit Hedera transactions, so funds cannot be moved. However, the stale-read window can persist for seconds to minutes under normal replication lag, and the attacker controls the timing.

### Likelihood Explanation

**Preconditions:**
1. pgpool (or a compatible load-balancing proxy) is deployed in front of PostgreSQL — this is the documented and charted production topology for this project.
2. Streaming replication lag exists — this is always nonzero in practice.
3. The attacker knows the alias of a recently deleted account — account deletions are public on the Hedera network; the attacker can monitor consensus nodes or the importer's own REST API.

**Trigger:** The attacker submits a `GET /api/v1/accounts/<alias>` (or any alias-routed endpoint) within the replication lag window after the deletion is committed on the primary.

**Repeatability:** The attacker can retry rapidly; the window is bounded only by replication lag, which can be artificially widened by network pressure. No authentication is required.

### Recommendation

Apply the same `StatementInspector` / pgpool comment pattern that the importer already uses:

1. Add a `loadBalance` property to the rest-java DB configuration (mirroring `importer/src/main/java/org/hiero/mirror/importer/db/DBProperties.java`).
2. Create `rest-java/src/main/java/org/hiero/mirror/restjava/config/HibernateConfiguration.java` implementing `HibernatePropertiesCustomizer`, and when `loadBalance = false`, register a `StatementInspector` that prepends `/* NO PGPOOL LOAD BALANCE */\n` to every SQL string — exactly as the importer does.
3. Set `loadBalance = false` in the default/production Helm values for `hedera-mirror-rest-java`.

Alternatively, annotate all read-only service methods with `@Transactional(readOnly = true)` and configure Spring's `AbstractRoutingDataSource` to route read-only transactions to the primary, though the pgpool comment approach is already proven in this codebase.

### Proof of Concept

```
# 1. Observe account 0.0.12345 with alias ABCDEF... is active on the network.
GET /api/v1/accounts/ABCDEF...
→ 200 OK, returns account data

# 2. Submit a CryptoDelete transaction for 0.0.12345 on the Hedera network.
#    Wait for the importer to write deleted=true to the PRIMARY DB.
#    Do NOT wait for the replica to catch up (replication lag window).

# 3. Immediately query the REST-Java service:
GET /api/v1/accounts/ABCDEF...
→ pgpool routes the SELECT to the stale replica (deleted=false still present)
→ findByAlias() returns entity ID 12345
→ 200 OK — deleted account is returned as active

# 4. Use the returned entity ID for further queries:
GET /api/v1/accounts/12345/tokens
→ Returns token balances for the deleted account

# Repeat step 3 rapidly; success rate depends on replication lag duration.
```
