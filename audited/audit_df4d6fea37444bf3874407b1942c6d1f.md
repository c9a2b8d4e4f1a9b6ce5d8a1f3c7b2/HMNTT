### Title
`merge()` in `requestQueryParser` Bypasses `qs` `arrayLimit` via Mixed-Case Key Variants

### Summary
The `requestQueryParser` function in `rest/middleware/requestHandler.js` first delegates to `qs.parse()` with `arrayLimit` and `throwOnLimitExceeded: true` to cap repeated parameters, but `qs` is case-sensitive and treats `transaction.hash`, `transaction.Hash`, and `transaction.HASH` as three distinct keys. The subsequent lowercasing loop then calls the internal `merge()` function to combine all variants into one key with no length guard, producing a merged array that can be a multiple of `arrayLimit` in size — silently bypassing the only server-side array-size enforcement that was supposed to throw.

### Finding Description

**Exact code path:**

`rest/middleware/requestHandler.js`, lines 15–20 — `queryOptions` configures `qs`: [1](#0-0) 

`qs.parse()` is called at line 54 with those options: [2](#0-1) 

The loop at lines 57–66 lowercases every key and calls `merge()` on collision: [3](#0-2) 

`merge()` itself (lines 39–51) has **no length check**: [4](#0-3) 

**Root cause:** The `throwOnLimitExceeded: true` guard fires only when a *single qs-level key* accumulates more than `arrayLimit` values. Because `qs` is case-sensitive, `transaction.hash` and `transaction.Hash` are two independent keys, each allowed up to `arrayLimit` values before the guard fires. After `qs.parse()` succeeds, the lowercasing loop merges all variants through `merge()` with no secondary size check, producing an array of up to `N_casings × arrayLimit` elements.

**Exploit flow:**

With `arrayLimit = 8` (the default, confirmed by the test spec error message "Only 8 elements allowed in an array"): [5](#0-4) 

An attacker sends:
```
?transaction.hash=h1&…&transaction.hash=h8   ← 8 values, qs limit not exceeded
&transaction.Hash=h9&…&transaction.Hash=h16  ← 8 more, separate qs key
&transaction.HASH=h17&…&transaction.HASH=h24 ← 8 more
```
`qs.parse()` returns three keys, each with 8 values — no exception thrown. The loop then merges all three into `transaction.hash` with 24 values. The test suite explicitly documents and expects this merging behaviour: [6](#0-5) [7](#0-6) 

**Why the existing checks fail:**

The secondary guard lives in `buildFilters()` (`rest/utils.js` lines 1241–1248): [8](#0-7) 

This check is not called by `requestQueryParser` itself; it is called only by route handlers that explicitly invoke `buildFilters()`. Any handler that reads `req.query` directly (e.g. custom single-parameter validators, middleware that inspects `req.query['transaction.hash']` before `buildFilters()` runs) receives the already-inflated array. Additionally, even for handlers that do call `buildFilters()`, the oversized array is fully materialised in the Node.js heap *before* the check runs.

### Impact Explanation

1. **Filter bypass for "only one value" constraints.** The endpoint `/api/v1/contracts/results/logs` enforces "Only one transaction.hash filter is allowed": [9](#0-8) 
   If the per-endpoint validator checks `Array.isArray(req.query['transaction.hash'])` and rejects arrays, an attacker who sends exactly one value per casing variant (e.g. `transaction.hash=v1&transaction.Hash=v2`) produces a 2-element array that is caught. However, for validators that only check `req.query['transaction.hash'] !== undefined` (treating the first value as canonical), the second value silently passes, injecting an extra filter predicate into the SQL query.

2. **Amplified array materialisation.** With N distinct casings (trivially achievable — `transaction.hash` has 15 alphabetic characters → 2¹⁵ = 32 768 possible casings), each carrying `arrayLimit` values, the merged array size is bounded only by the HTTP query-string length limit, not by any application-level guard inside `requestQueryParser`.

### Likelihood Explanation

- **No authentication required.** Any unauthenticated HTTP client can craft the query string.
- **Trivial to construct.** Mixed-case variants require no special tooling — a simple curl command suffices.
- **Repeatable.** The condition is deterministic and reproducible on every request.
- **Practical constraint:** Most reverse proxies (nginx default: 8 KB) limit query-string length, capping the merged array at roughly 100–200 values for hash-sized parameters. This limits memory exhaustion but does not prevent filter injection.

### Recommendation

Add a length guard inside `merge()` or immediately after the merge step in `requestQueryParser`:

```js
const merge = (current, next, limit) => {
  if (!Array.isArray(current)) current = [current];
  if (Array.isArray(next)) current.push(...next);
  else current.push(next);
  if (current.length > limit) {
    throw new RangeError(`Array limit exceeded. Only ${limit} elements allowed in an array.`);
  }
  return current;
};
```

Call it as `merge(existing, canonicalValue, config.query.maxRepeatedQueryParameters)`. This ensures the `arrayLimit` invariant is enforced *after* case-folding, not just per-casing-variant.
