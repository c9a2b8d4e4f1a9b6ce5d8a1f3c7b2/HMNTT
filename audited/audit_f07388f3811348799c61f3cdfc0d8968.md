### Title
Unauthenticated Alias-Based DB Flooding via `GetAccountId()` Exhausts Connection Pool

### Summary
The `GetAccountId()` function in `rosetta/app/persistence/account.go` unconditionally issues a `selectCurrentCryptoEntityByAlias` SQL query for every alias-bearing account identifier, with no application-level caching or rate limiting. The Traefik middleware that provides the only rate-limiting layer is disabled by default (`global.middleware: false`), and even when enabled, its `rateLimit` is keyed per request host (global, not per-IP), allowing any unauthenticated attacker to exhaust the database connection pool and degrade service for all users.

### Finding Description

**Exact code path:**

`GetAccountId()` at `rosetta/app/persistence/account.go` lines 138–159:

```go
func (ar *accountRepository) GetAccountId(ctx context.Context, accountId types.AccountId) (...) {
    if !accountId.HasAlias() {
        return accountId, nil   // fast path: no DB hit
    }
    db, cancel := ar.dbClient.GetDbWithContext(ctx)
    defer cancel()
    var entity domain.Entity
    if err := db.Raw(selectCurrentCryptoEntityByAlias,
        sql.Named("alias", accountId.GetAlias())).First(&entity).Error; err != nil {
        ...
    }
    ...
}
```

The query executed is:
```sql
select id from entity
where alias = @alias and (deleted is null or deleted is false)
```
(lines 60–61)

**Call chain that is reachable without authentication:**

`POST /construction/metadata` → `ConstructionMetadata()` → `resolveAccountAliases()` (`construction_service.go` line 581) → `GetAccountId()` for each comma-separated alias in the `account_aliases` option field.

**Root cause and failed assumption:**

The code assumes that upstream infrastructure will throttle abusive callers before they reach the DB. That assumption fails because:

1. **Middleware is disabled by default.** `charts/hedera-mirror-rosetta/values.yaml` line 95 sets `global.middleware: false`. The middleware template (`templates/middleware.yaml` line 3) gates on `{{ if and .Values.global.middleware .Values.middleware }}`, so the entire Traefik chain — including `inFlightReq` and `rateLimit` — is not deployed unless an operator explicitly overrides this.

2. **When middleware is enabled, `rateLimit` is per request host, not per source IP** (lines 157–160 of `values.yaml`):
   ```yaml
   - rateLimit:
       average: 10
       sourceCriterion:
         requestHost: true
   ```
   `requestHost: true` keys the bucket on the HTTP `Host` header. Since all clients share the same host, this is a single global bucket of 10 req/s for all users combined. A single attacker consuming 10 req/s starves every legitimate user.

3. **`inFlightReq` is per source IP** (lines 152–156), capped at 5 concurrent requests per IP. An attacker using even a small number of IPs (e.g., 10 IPs × 5 concurrent = 50 simultaneous DB queries) can saturate a typical connection pool.

4. **No application-level caching.** There is no in-process cache for alias→entity-id mappings. Every request, including those for non-existent aliases, hits the database.

5. **Multiple aliases per request.** The `account_aliases` option is a comma-separated string (parsed in `resolveAccountAliases()` lines 575–586). A single HTTP request can carry N aliases, triggering N sequential `GetAccountId()` DB calls, multiplying the load beyond what per-request rate limits protect against.

**DB connection pool:**

`rosetta/app/db/db.go` line 33: `sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)`. Once the pool is exhausted, all DB-dependent handlers block or fail, degrading the entire service.

### Impact Explanation

An attacker can cause sustained DB connection pool exhaustion, making the Rosetta API unresponsive for all users (account balance queries, block queries, construction flows). There is no economic cost to the attacker — no tokens, no fees, no on-chain action required. The impact is a griefing/availability attack against the mirror node operator and all downstream integrators (exchanges, wallets, block explorers) relying on the Rosetta API.

### Likelihood Explanation

The attack requires only the ability to send HTTP POST requests to a public endpoint — no credentials, no privileged access, no on-chain interaction. The endpoint `/construction/metadata` is part of the standard Rosetta Construction API and is expected to be publicly reachable. The attack is trivially scriptable, repeatable, and can be amplified by embedding many aliases per request. The default-off middleware means most deployments that have not explicitly opted into Traefik middleware are fully unprotected at the application layer.

### Recommendation

1. **Add application-level caching** for alias→entity-id lookups (e.g., a short-lived TTL cache keyed on alias bytes) to avoid repeated DB hits for the same or non-existent aliases.
2. **Add application-level rate limiting** (e.g., `golang.org/x/time/rate` token bucket) per source IP inside the Go service, independent of infrastructure middleware.
3. **Cap the number of aliases per request** in `resolveAccountAliases()` to a small constant (e.g., ≤ 10) to prevent per-request amplification.
4. **Change the Traefik `rateLimit` source criterion** from `requestHost: true` to `ipStrategy` (per-IP) so that one attacker cannot consume the global budget.
5. **Enable middleware by default** (`global.middleware: true`) or document clearly that it must be enabled for production deployments.

### Proof of Concept

```bash
# Step 1: Craft a request with many random non-existent hex aliases
ALIASES=$(python3 -c "
import os, ','.join
aliases = ['0x' + os.urandom(32).hex() for _ in range(50)]
print(','.join(aliases))
")

# Step 2: Flood the /construction/metadata endpoint
while true; do
  curl -s -X POST https://<rosetta-host>/construction/metadata \
    -H 'Content-Type: application/json' \
    -d "{
      \"network_identifier\": {\"blockchain\": \"Hedera\", \"network\": \"mainnet\"},
      \"options\": {
        \"operation_type\": \"CRYPTO_TRANSFER\",
        \"account_aliases\": \"$ALIASES\"
      }
    }" &
done
# Each iteration fires 50 selectCurrentCryptoEntityByAlias queries against the DB.
# With ~10 concurrent processes, the DB connection pool is saturated within seconds,
# causing all other Rosetta API calls to time out or return errors.
```
