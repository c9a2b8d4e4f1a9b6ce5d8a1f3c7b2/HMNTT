### Title
Unauthenticated Concurrent Request Flood on `/api/v1/network/nodes` Exhausts DB CPU via Unthrottled Complex Multi-Table CTE Query

### Summary
The `GET /api/v1/network/nodes` endpoint in the `rest-java` module is publicly accessible with no rate limiting, no authentication, and no query-result caching. Each request triggers a complex multi-table join query (across `address_book`, `address_book_entry`, `node_stake`, `node`, `address_book_service_endpoint`) via `findNetworkNodes()`. An unprivileged attacker flooding this endpoint with concurrent requests can saturate database CPU, degrading or denying mirror node service to all consumers.

### Finding Description

**Code path:**

`NetworkController.java:152-171` → `NetworkServiceImpl.java:100-136` → `NetworkNodeRepository.findNetworkNodes()`

`NetworkNodeRequest` defines `MAX_LIMIT = 25` and `DEFAULT_LIMIT = 10`, enforced via `getEffectiveLimit()`:

```java
// NetworkNodeRequest.java:31-54
public static final int DEFAULT_LIMIT = 10;
public static final int MAX_LIMIT = 25;

public int getEffectiveLimit() {
    return Math.min(limit, MAX_LIMIT);
}
```

`getNetworkNodes()` unconditionally calls the repository with no caching:

```java
// NetworkServiceImpl.java:135-136
return networkNodeRepository.findNetworkNodes(
        fileId, nodeIdArray, lowerBound, upperBound, orderDirection, limit);
```

The `NetworkNodeRepository.findNetworkNodes()` executes a complex CTE query joining five tables. No `@Cacheable` annotation is present on this repository method (contrast with `grpc`'s `AddressBookEntryRepository.findByConsensusTimestampAndNodeId()` which is `@Cacheable`).

**Root cause — missing rate limiting in rest-java:**

The `web3` module has a full throttle stack: `ThrottleConfiguration`, `ThrottleManagerImpl`, `ThrottleProperties` with a default of 500 requests/second per instance. The `rest-java` module has **no equivalent**. The only filters registered in `RestJavaConfiguration` are `ShallowEtagHeaderFilter` (ETag caching) and `MetricsFilter` (metrics recording only). Neither limits request concurrency or rate.

The `statementTimeout` documented for rest-java is 10,000 ms (10 seconds). This means each concurrent query can hold a DB connection and consume CPU for up to 10 seconds before being killed.

**Why existing checks fail:**

- `MAX_LIMIT = 25` limits result rows but does not reduce query complexity; the CTE still scans and joins all five tables to find the latest address book, latest node stake, and aggregate service endpoints.
- `statementTimeout = 10000` is a backstop, not a rate control. With N concurrent requests each running for up to 10 seconds, N×10 seconds of DB CPU is consumed simultaneously.
- No per-IP, per-session, or global request rate limit exists in rest-java for this endpoint.
- No caching of `findNetworkNodes()` results exists, so every request hits the database.

### Impact Explanation

A sustained flood of concurrent requests to `GET /api/v1/network/nodes` exhausts the PostgreSQL connection pool and CPU. The `RestJavaHighDBConnections` alert fires at 75% pool utilization; beyond that, new requests queue or fail. Because the mirror node database is shared across all rest-java endpoints, DB CPU saturation degrades or denies service for all mirror node API consumers — not just the `/network/nodes` endpoint. This matches the stated scope: 30%+ of mirror node processing capacity lost without brute-force network-layer actions.

### Likelihood Explanation

The attack requires zero privileges, zero authentication, and only a standard HTTP client. The endpoint is publicly documented and reachable. A single attacker with modest bandwidth (e.g., 50–100 concurrent HTTP connections) can sustain the flood indefinitely. The attack is trivially repeatable and scriptable (e.g., `ab -n 10000 -c 100 https://host/api/v1/network/nodes?limit=25`). No exploit tooling beyond `curl` or `ab` is needed.

### Recommendation

1. **Add rate limiting to rest-java**: Port the `ThrottleConfiguration`/`ThrottleManagerImpl` pattern from the `web3` module, or add a servlet filter using Bucket4j (already a dependency) to enforce a global and/or per-IP request rate on all rest-java endpoints.
2. **Cache `findNetworkNodes()` results**: The address book changes infrequently. Apply `@Cacheable` (with a short TTL, e.g., 30–60 seconds) to `NetworkNodeRepository.findNetworkNodes()`, similar to how `AddressBookEntryRepository` in the `grpc` module caches its results.
3. **Reduce `statementTimeout`**: Lower the 10-second DB statement timeout for this query to 2–3 seconds to limit per-request DB resource consumption.
4. **Add connection pool limits per endpoint**: Consider using a separate, smaller HikariCP pool for the `/network/nodes` endpoint to prevent it from starving other endpoints.

    @Override
    public List<NetworkNodeDto> getNetworkNodes(NetworkNodeRequest request) {
        final long fileId = getAddressBookFileId(request);
        final var limit = request.getEffectiveLimit();
        final var nodeIdParams = request.getNodeIds();
        final var orderDirection = request.getOrder().name();

        final Set<Long> nodeIds = new HashSet<>();
        long lowerBound = 0L;
        long upperBound = Long.MAX_VALUE;

        for (final var nodeIdParam : nodeIdParams) {
            if (nodeIdParam.operator() == RangeOperator.EQ) {
                nodeIds.add(nodeIdParam.value());
            } else if (nodeIdParam.hasLowerBound()) {
                lowerBound = Math.max(lowerBound, nodeIdParam.getInclusiveValue());
            } else if (nodeIdParam.hasUpperBound()) {
                upperBound = Math.min(upperBound, nodeIdParam.getInclusiveValue());
            }
        }

        if (lowerBound > upperBound) {
            throw new IllegalArgumentException("Invalid range provided for node.id");
        }

        final Long[] nodeIdArray;
        if (!nodeIds.isEmpty()) {
            final var range = Range.closed(lowerBound, upperBound);
            nodeIdArray = nodeIds.stream().filter(range::contains).toArray(Long[]::new);
            if (nodeIdArray.length == 0) {
                return List.of();
            }
        } else {
            nodeIdArray = EMPTY_NODE_ID_ARRAY;
        }

        return networkNodeRepository.findNetworkNodes(
                fileId, nodeIdArray, lowerBound, upperBound, orderDirection, limit);
    }
