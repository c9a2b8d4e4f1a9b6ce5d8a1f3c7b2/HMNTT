### Title
Unvalidated ILIKE Wildcard Injection in Token Name Filter Enables Database CPU Exhaustion

### Summary
The `getTokensRequest()` function in `rest/tokens.js` accepts a `name` query parameter that is validated only for byte length (3–100 bytes) and operator type, with no restriction on ILIKE wildcard characters (`%`, `_`). The value is then wrapped with additional `%` wildcards and passed directly into a `t.name ILIKE $N` clause, forcing PostgreSQL to perform a full table scan with expensive backtracking pattern matching. Any unauthenticated external user can repeatedly trigger this to exhaust database CPU.

### Finding Description

**Exact code path:**

In `validateTokenQueryFilter()` at `rest/tokens.js` line 334:
```js
case filterKeys.NAME:
  ret = op === queryParamOperators.eq && utils.isByteRange(val, 3, 100);
  break;
``` [1](#0-0) 

The only validation applied is:
1. Operator must be `eq`
2. Value must be 3–100 bytes in length

There is **no check** that rejects or escapes ILIKE wildcard characters (`%`, `_`).

In `extractSqlFromTokenRequest()` at `rest/tokens.js` line 177:
```js
conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
``` [2](#0-1) 

The server itself prepends and appends `%` to the user-supplied value. A user-supplied value of `a%b%c%d%e%f%g%h%i%j%k%l%m%n%o%p%q%r%s%t%u%v%w%x%y%z` (51 bytes, passes validation) becomes the pattern `%a%b%c%d%e%f%g%h%i%j%k%l%m%n%o%p%q%r%s%t%u%v%w%x%y%z%` — 27 wildcard segments requiring catastrophic backtracking across every row in the `token` table.

**Root cause:** `isByteRange` validates only the byte length of the input, not its content. The failed assumption is that a length-bounded string is safe to use in an ILIKE pattern.

**Why the `limit` clause does not help:** The `LIMIT` is applied after PostgreSQL evaluates the `WHERE` clause. PostgreSQL must scan and pattern-match every row in the `token` table before it can return the first `limit` rows.

### Impact Explanation
A single request with a maximally complex 100-byte wildcard pattern (e.g., `%a%b%c%...`) causes PostgreSQL to perform a full sequential scan of the `token` table with O(n × m) backtracking per row, where n is the row count and m is the pattern complexity. Repeated concurrent requests from one or more unauthenticated clients can saturate database CPU, degrading or denying service to all users of the mirror node REST API. No index (including `pg_trgm` GIN indexes) can accelerate a pattern with a leading `%`.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication requirement. The exploit requires only a standard HTTP GET request with a crafted `name` query parameter. The attack is trivially repeatable, scriptable, and requires no special knowledge beyond reading the public API documentation. A single attacker with a modest request rate can sustain the attack indefinitely.

### Recommendation
1. **Reject wildcard characters in the `name` filter value** inside `validateTokenQueryFilter()`: add a check that the value does not contain `%` or `_` before accepting it.
2. Alternatively, **escape wildcard characters** before interpolating into the ILIKE pattern (replace `%` → `\%`, `_` → `\_`, and add `ESCAPE '\'` to the SQL clause).
3. Consider enforcing a **statement timeout** at the database connection level for REST API queries to bound worst-case query duration.
4. Consider **rate-limiting** the `/api/v1/tokens` endpoint by IP.

### Proof of Concept
```
GET /api/v1/tokens?name=%25a%25b%25c%25d%25e%25f%25g%25h%25i%25j%25k%25l%25m%25n%25o%25p%25q%25r%25s%25t%25u%25v%25w%25x%25y%25z HTTP/1.1
Host: <mirror-node-host>
```
URL-decoded `name` value: `%a%b%c%d%e%f%g%h%i%j%k%l%m%n%o%p%q%r%s%t%u%v%w%x%y%z` (51 bytes — passes `isByteRange(val, 3, 100)`)

Resulting SQL condition executed against the database:
```sql
WHERE t.name ILIKE '%%a%b%c%d%e%f%g%h%i%j%k%l%m%n%o%p%q%r%s%t%u%v%w%x%y%z%'
```

Sending this request in a tight loop (e.g., `while true; do curl "..." & done`) will drive database CPU to 100% and degrade all concurrent queries.
