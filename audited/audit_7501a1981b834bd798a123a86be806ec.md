### Title
Counter Exhaustion via Non-Matching Requests Bypasses `RequestProperties` Filter Actions (REJECT/THROTTLE)

### Summary
In `RequestProperties.test()`, the `counter.getAndIncrement()` call on line 46 unconditionally increments the counter for every request before the filter-matching loop on lines 54–58 is evaluated. When an operator configures a finite `limit`, an unprivileged attacker can exhaust the counter by flooding the endpoint with requests that do not match any configured filter. Once `counter >= limit`, `test()` permanently returns `false` for all subsequent requests — including legitimate ones that would have matched — silently disabling the configured `REJECT` or `THROTTLE` action.

### Finding Description

**Exact code path:**

`RequestProperties.java`, `test()`, lines 45–61:

```java
public boolean test(ContractCallRequest contractCallRequest) {
    if (rate == 0 || counter.getAndIncrement() >= limit) {  // line 46 — counter incremented HERE
        return false;
    }
    // ...
    for (var filter : filters) {                            // line 54 — filter check AFTER increment
        if (filter.test(contractCallRequest)) {
            return true;
        }
    }
    return filters.isEmpty();
}
```

`ThrottleManagerImpl.java`, `throttle()`, lines 44–48:

```java
for (var requestFilter : throttleProperties.getRequest()) {
    if (requestFilter.test(request)) {   // calls RequestProperties.test()
        action(requestFilter, request);  // REJECT/THROTTLE only reached if test() == true
    }
}
```

**Root cause:** `counter.getAndIncrement()` is evaluated unconditionally on every invocation of `test()`, regardless of whether the request satisfies any `RequestFilter`. The `limit` field is intended to cap how many times the filter rule fires, but because the counter is incremented before filter matching, non-matching requests consume the budget.

**Failed assumption:** The design assumes the counter tracks "how many times this filter matched and acted upon a request," but it actually tracks "how many times `test()` was called at all."

**Exploit flow:**
1. Operator configures a `RequestProperties` entry with `action=REJECT`, a non-empty `filters` list (e.g., `field=FROM, type=EQUALS, expression=0xVictimAddress`), and a finite `limit` (e.g., `limit=10000`).
2. Attacker sends `limit` requests from any address that does NOT equal `0xVictimAddress`. Each call to `test()` increments the counter but returns `false` (no filter match), so `action()` is never called — the requests pass through normally.
3. After `limit` iterations, `counter.getAndIncrement() >= limit` is permanently `true`.
4. Attacker now sends requests from `0xVictimAddress`. `test()` returns `false` immediately at line 46 — the `REJECT` action is never invoked. The previously-blocked address is now unblocked.

**Why existing checks are insufficient:**
- The `limit` default of `Long.MAX_VALUE` (line 35 of `RequestProperties.java`) prevents exploitation in default configurations, but any operator-configured finite `limit` is vulnerable.
- There is no guard that prevents non-matching requests from consuming the counter budget.
- The `rateLimitBucket` and `gasLimitBucket` in `ThrottleManagerImpl` (lines 38–42) are separate global buckets and do not protect the per-`RequestProperties` counter.

### Impact Explanation

For `action=REJECT`: A security rule blocking a specific caller, contract address, or calldata pattern is permanently disabled after counter exhaustion. The attacker's previously-rejected requests now pass through to execution.

For `action=THROTTLE`: A per-pattern rate limit is permanently disabled; the attacker can send unlimited matching requests without hitting the per-filter bucket.

For `action=LOG`: Audit/monitoring coverage is silently dropped, enabling undetected abuse.

Severity: **High** when `limit` is finite and `action` is `REJECT` or `THROTTLE`, as it constitutes a complete bypass of an operator-configured security control.

### Likelihood Explanation

- **Precondition:** Operator must configure a finite `limit`. This is a realistic operational choice (e.g., "apply this block rule for the first N requests of the day").
- **Attacker capability:** Zero privilege required. Any user who can call the web3 endpoint can send arbitrary `ContractCallRequest` payloads.
- **Detectability:** The bypass is silent — no error is thrown, no log is emitted once the counter is exhausted.
- **Repeatability:** The `AtomicLong counter` is never reset (no TTL, no refill), so exhaustion is permanent until the service restarts.
- **Cost:** The attacker only needs to send `limit` cheap, non-matching requests. With `requestsPerSecond=500` (default), exhausting a `limit=10000` counter takes ~20 seconds of sustained requests.

### Recommendation

Move the counter increment to after a successful filter match, so only matching requests consume the budget:

```java
@Override
public boolean test(ContractCallRequest contractCallRequest) {
    if (rate == 0) {
        return false;
    }

    if (action != ActionType.THROTTLE && RandomUtils.secure().randomLong(0L, 100L) >= rate) {
        return false;
    }

    boolean matched = filters.isEmpty();
    for (var filter : filters) {
        if (filter.test(contractCallRequest)) {
            matched = true;
            break;
        }
    }

    if (!matched) {
        return false;
    }

    // Only increment after confirming a match
    return counter.getAndIncrement() < limit;
}
```

This ensures the `limit` accurately reflects "how many times this rule has fired on a matching request," not "how many total requests were evaluated."

### Proof of Concept

**Setup:** Configure `application.yml`:
```yaml
hiero.mirror.web3.throttle:
  request:
    - action: REJECT
      limit: 100
      filters:
        - field: FROM
          type: EQUALS
          expression: "0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"
```

**Step 1 — Exhaust the counter with non-matching requests:**
```bash
for i in $(seq 1 100); do
  curl -s -X POST http://localhost:8545 \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","method":"eth_call","params":[{"from":"0x1111111111111111111111111111111111111111","to":"0xabc","data":"0x"},"latest"],"id":1}'
done
```
Each request increments the counter but does not match the `FROM` filter → counter reaches 100.

**Step 2 — Send a request that should be rejected:**
```bash
curl -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"eth_call","params":[{"from":"0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef","to":"0xabc","data":"0x"},"latest"],"id":2}'
```

**Expected result (without bug):** HTTP 429 / `ThrottleException("Invalid request")`

**Actual result (with bug):** Request succeeds — `test()` returns `false` at line 46 because `counter (100) >= limit (100)`, so `action()` is never called and the `REJECT` is silently skipped. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
