### Title
Unbounded Consensus Latency Recording via User-Controlled Historical `startTime` in `recordMetric()`

### Summary
`CompositeTopicListener.recordMetric()` computes latency as `System.currentTimeMillis() - (topicMessage.getConsensusTimestamp() / 1000000)` and records it unconditionally (only gated by `latency > 0`) into the shared `MeterRegistry` Timer. An unprivileged gRPC client can subscribe with `consensusStartTime` set to epoch (0) or any arbitrarily old timestamp, causing the polling listener to surface historical messages whose `consensusTimestamp` is years old, injecting astronomically large latency samples into the `hiero.mirror.grpc.consensus.latency` histogram.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` → `toFilter()` (line 63–64) converts the user-supplied `consensusStartTime` proto field into `filter.startTime(startTime)`. The only validation is `TopicMessageFilter.isValidStartTime()` (line 49–51), which only rejects future timestamps (`startTime <= DomainUtils.now()`). `startTime=0` (epoch) passes `@Min(0)` and `isValidStartTime()`.

`TopicMessageServiceImpl.incomingMessages()` (line 117) computes:
```java
long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
```
When no historical messages have been processed (`last == null`), the listener is invoked with the user-supplied `startTime=0`.

`CompositeTopicListener.listen()` (line 40–43) calls the underlying polling listener with this filter and attaches `doOnNext(this::recordMetric)`. The polling listener (`PollingTopicListener` / `SharedPollingTopicListener`) queries the database for `consensusTimestamp >= 0`, returning messages potentially years old.

`recordMetric()` (lines 66–71):
```java
private void recordMetric(TopicMessage topicMessage) {
    long latency = System.currentTimeMillis() - (topicMessage.getConsensusTimestamp() / 1000000);
    if (latency > 0) {
        getConsensusLatencyTimer().record(latency, TimeUnit.MILLISECONDS);
    }
}
```
For a message with `consensusTimestamp` from 2019 (nanoseconds), the computed latency is ~6 years in milliseconds (~1.89 × 10¹¹ ms). This value is recorded unconditionally into the shared Timer histogram.

**Root cause:** `recordMetric()` assumes messages flowing through the listener are always recent (live). It has no upper-bound guard on latency. The `isValidStartTime()` check only prevents future `startTime` values, not arbitrarily old ones, so historical messages with ancient `consensusTimestamp` values flow through the live listener path and pollute the metric.

**Why existing checks fail:**
- `@Min(0)` on `startTime` — allows epoch (0).
- `isValidStartTime()` — only rejects `startTime > now()`.
- `latency > 0` guard in `recordMetric()` — old messages always satisfy this; it only prevents negative latency (future-timestamped messages). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
The `hiero.mirror.grpc.consensus.latency` Timer is a shared, process-wide Micrometer histogram. Injecting values of ~10¹¹ ms permanently skews percentile buckets (p99, p999, max) for all subscribers, not just the attacker's. Monitoring systems and alerting rules based on this metric will fire false high-latency alerts, masking real latency regressions or triggering unnecessary incident response. Repeated subscriptions amplify the effect. This constitutes metric-based denial of observability.

### Likelihood Explanation
Any unauthenticated gRPC client (the gRPC endpoint is publicly accessible) can send a `ConsensusTopicQuery` with `consensusStartTime` set to `{seconds: 0, nanos: 0}`. No special privileges, credentials, or knowledge of internal state are required. The attack is trivially repeatable — each new subscription with `startTime=0` injects another batch of large latency samples. If the topic has any historical messages (common in production), the exploit succeeds immediately.

### Recommendation
1. **Cap recorded latency in `recordMetric()`**: Add an upper-bound guard, e.g., only record if `latency <= someReasonableMaxMs` (e.g., 60,000 ms for 1 minute):
   ```java
   private static final long MAX_RECORDABLE_LATENCY_MS = 60_000L;
   private void recordMetric(TopicMessage topicMessage) {
       long latency = System.currentTimeMillis() - (topicMessage.getConsensusTimestamp() / 1_000_000);
       if (latency > 0 && latency <= MAX_RECORDABLE_LATENCY_MS) {
           getConsensusLatencyTimer().record(latency, TimeUnit.MILLISECONDS);
       }
   }
   ```
2. **Only record metric for live messages**: Track whether a message is historical vs. live (e.g., via a flag on the filter or context) and skip `recordMetric()` for historical replay messages.
3. **Add a minimum `startTime` floor**: Optionally reject subscriptions with `startTime` older than a configurable threshold (e.g., 24 hours) to limit the scope of historical replay through the live listener path.

### Proof of Concept
1. Start the mirror node gRPC service with `POLL` or `SHARED_POLL` listener type and a topic that has messages older than a few minutes.
2. Send the following gRPC request (using `grpcurl` or any gRPC client):
   ```
   subscribeTopic({
     topicID: { shardNum: 0, realmNum: 0, topicNum: <valid_topic_id> },
     consensusStartTime: { seconds: 0, nanos: 0 }  // epoch
   })
   ```
3. Observe that `recordMetric()` is called for each historical message returned by the polling listener.
4. Query the Micrometer metrics endpoint (`/actuator/metrics/hiero.mirror.grpc.consensus.latency`) and observe that `max`, `p99`, and `mean` values are now in the range of years (10¹¹ ms), permanently skewing the histogram until the JVM restarts.
