### Title
Cache Key Collision in `findByTimestamp` `@CachePut` Poisons the Record File Index Cache

### Summary
`RecordFileRepository.findByTimestamp(long timestamp)` uses a `@CachePut` annotation that writes into the `CACHE_MANAGER_RECORD_FILE_INDEX` cache using the **input timestamp** as the default Spring cache key. `findByIndex(long index)` reads from the same cache using the **block index** as the key. Because both share the same cache namespace and an attacker controls the timestamp value, sending a timestamp equal to any block index N causes the wrong `RecordFile` to be stored under key N, permanently poisoning that index cache slot for up to 10 minutes per node.

### Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 38–46:

```java
@Caching(
        cacheable =
                @Cacheable(
                        cacheNames = CACHE_NAME,
                        cacheManager = CACHE_MANAGER_RECORD_FILE_TIMESTAMP,
                        unless = "#result == null"),
        put = @CachePut(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX))  // line 44
@Query("select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1")
Optional<RecordFile> findByTimestamp(long timestamp);
```

`findByIndex` at lines 27–29:

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
```

**Root cause:** Spring's `@CachePut` with no explicit `key` attribute defaults to using the method's parameter list as the cache key. For `findByTimestamp(long timestamp)` the key is the raw `timestamp` value. For `findByIndex(long index)` the key is the raw `index` value. Both annotations target the same cache manager (`CACHE_MANAGER_RECORD_FILE_INDEX`) and the same cache name (`CACHE_NAME = "default"`), so they share one flat key space.

**Failed assumption:** The developer intended `@CachePut` to pre-populate the index cache with the returned `RecordFile` keyed by its `.index` field (a block number). Instead, it keys by the caller-supplied timestamp — a completely different domain of values.

**Exploit flow:**

1. Block indices are small sequential integers (0, 1, 2, … millions). Consensus timestamps are nanosecond Unix values (~1.6 × 10¹⁸). An attacker supplies a timestamp value of N where N is a known block index (e.g., N = 5000).
2. The DB query `select r from RecordFile r where r.consensusEnd >= 5000 order by r.consensusEnd asc limit 1` returns the **earliest** record file in the chain (since every real `consensusEnd` is far larger than 5000).
3. `@CachePut` unconditionally writes `Optional<RecordFile(earliest)>` into the index cache under key `5000`.
4. The next call to `findByIndex(5000)` hits the cache, finds key `5000`, and returns the **earliest** record file instead of block 5000.
5. Because `@CachePut` has no `unless` guard, it also overwrites a previously correct cache entry if one existed.

**Why existing checks fail:**

- `findByIndex` has `unless = "#result == null"` — this only prevents caching a null result on reads; it does not prevent `@CachePut` on `findByTimestamp` from overwriting an existing correct entry.
- The index cache TTL is 10 minutes (`expireAfterWrite(10, TimeUnit.MINUTES)`) with `maximumSize=10000`, so poisoned entries persist and serve incorrect data for the full TTL window.
- There is no input validation on the timestamp parameter before it reaches the repository layer.

### Impact Explanation

Every EVM historical call that resolves a block by number (e.g., `eth_call` with `blockNumber=5000`) goes through `findByIndex`. With a poisoned cache entry, the node returns EVM execution results computed against the **wrong block's** historical state (wrong `consensusEnd`, wrong EVM version context, wrong fee schedule, wrong contract bytecode snapshot). This silently corrupts `eth_call`, `eth_estimateGas`, and `eth_getBalance` responses for any block whose index cache slot has been poisoned. An attacker can poison up to 10,000 distinct block index slots (the cache `maximumSize`) per node per 10-minute window, covering the entire active block range on a busy network. Targeting 30%+ of mirror nodes requires repeating the same HTTP requests against each node independently — no coordination or privileged access is needed.

### Likelihood Explanation

The attacker needs only unauthenticated HTTP access to any endpoint that calls `RecordFileService.findByTimestamp` (e.g., `eth_getTransactionByHash`, `eth_getTransactionReceipt`, or any historical EVM call that resolves a transaction timestamp). The timestamp is attacker-controlled. The attack is trivially scriptable: iterate over block indices 0–N, send one request per index with `timestamp = index`, and all index cache slots are poisoned within seconds. No brute force, no credentials, no special protocol knowledge beyond standard JSON-RPC.

### Recommendation

Add an explicit `key` expression to the `@CachePut` that uses the returned record file's index field, and add an `unless` guard to avoid caching absent results:

```java
put = @CachePut(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
    key = "#result?.orElse(null)?.index",
    unless = "#result == null || !#result.isPresent()")
```

This ensures the index cache is keyed by the actual block index of the returned `RecordFile`, matching the key space used by `findByIndex`, and prevents empty results from being written.

### Proof of Concept

**Preconditions:** Mirror node is running; block index 5000 exists in the database.

**Step 1 — Warm the correct cache entry (optional, shows overwrite):**
```
GET /api/v1/contracts/call?blockNumber=5000
```
This calls `findByIndex(5000)` → correct `RecordFile` cached under key `5000`.

**Step 2 — Poison the cache:**
Send any request that resolves a transaction timestamp and passes it to `findByTimestamp`. For example, craft a JSON-RPC call that supplies timestamp `5000` as the consensus timestamp parameter. The repository executes:
```sql
select r from RecordFile r where r.consensusEnd >= 5000 order by r.consensusEnd asc limit 1
```
Returns the genesis/earliest `RecordFile` (e.g., index=0). `@CachePut` writes `Optional(RecordFile{index=0})` into the index cache under key `5000`.

**Step 3 — Observe corruption:**
```
GET /api/v1/contracts/call?blockNumber=5000
```
`findByIndex(5000)` now returns `RecordFile{index=0}` from cache. The EVM call executes against genesis block state instead of block 5000 state. The response contains wrong balances, wrong bytecode, wrong block hash — silently, with HTTP 200.
