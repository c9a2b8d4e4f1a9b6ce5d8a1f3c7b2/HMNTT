### Title
Global Opcode Rate Limit Exhaustion by Single Unprivileged Client (DoS on `/opcodes` Endpoint)

### Summary
The `opcodeRateLimitBucket` is a single application-wide token bucket with a default capacity of 1 request per second, shared across all clients with no per-IP or per-identity isolation. Any unauthenticated user can continuously consume the entire global budget from a single IP, causing all other users to receive HTTP 429 for the duration of the attack. Because the bucket refills at only 1 token/second, even a modest request rate fully starves legitimate callers.

### Finding Description
**Exact code path:**

- `OpcodesController.java` line 61: `throttleManager.throttleOpcodeRequest()` is called unconditionally before processing.
- `ThrottleManagerImpl.java` lines 52–55: `throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` on a single shared `Bucket` instance. No caller identity (IP, session, token) is examined.
- `ThrottleConfiguration.java` lines 47–55: `opcodeRateLimitBucket` is a single Spring singleton `Bucket` with `capacity = opcodeRequestsPerSecond` (default **1**) and `refillGreedy(rateLimit, Duration.ofSeconds(1))`.
- `ThrottleProperties.java` line 29: `opcodeRequestsPerSecond` defaults to `1`.

**Root cause:** The bucket is a process-global singleton. `throttleOpcodeRequest()` does not receive the HTTP request object, so it has no access to the caller's IP or any identity. The `request[]` filter system in `ThrottleProperties` (which does support per-field filtering) is only wired into `throttle(ContractCallRequest)`, not into `throttleOpcodeRequest()`, and even that system has no IP/source-address field (`FilterField` covers only `BLOCK`, `DATA`, `ESTIMATE`, `FROM`, `GAS`, `TO`, `VALUE`).

**Failed assumption:** The design assumes the 1 req/sec global cap is low enough that no single caller can monopolize it. In reality, 1 req/sec is exactly the full budget, so a single caller sending one request per second consumes 100% of the allowance.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/{anyHash}/opcodes` with `Accept-Encoding: gzip` at 1 req/sec.
2. Each request consumes the single available token.
3. Every concurrent or subsequent legitimate request within that second hits `tryConsume(1) == false` and receives HTTP 429.
4. The bucket refills 1 token/second; the attacker immediately re-consumes it.
5. Legitimate users are permanently denied access for the duration of the attack.

**Why existing checks are insufficient:** There is no per-IP bucket, no authentication gate, no connection-level throttle in the application layer, and no IP field in the opcode throttle path. The `request[]` filter system cannot be applied here even if configured.

### Impact Explanation
When the opcode tracer is enabled, the `/contracts/results/{transactionIdOrHash}/opcodes` endpoint becomes completely inaccessible to all users except the attacker. Transactions whose opcode traces are needed for audit, debugging, or dispute resolution become effectively unreachable for the attack window. While the underlying transaction data is not modified, the inability to retrieve execution traces for specific transactions constitutes a targeted availability attack on transaction history inspection. Severity is **Medium** (availability impact on a non-default, privileged-use endpoint; no confidentiality or integrity impact).

### Likelihood Explanation
**High feasibility.** No authentication, no special privileges, no account, and no funds are required. The attacker needs only HTTP access to the endpoint and the ability to send 1 request/second — trivially achievable with `curl` in a loop or any scripting language. The attack is repeatable indefinitely and requires no prior knowledge beyond the public API path. The only precondition is that the operator has set `hiero.mirror.web3.opcode.tracer.enabled=true`; when that is the case, exploitation is trivial.

### Recommendation
1. **Per-IP rate limiting:** Replace (or supplement) the global bucket with a `ConcurrentHashMap<String, Bucket>` keyed on the client IP extracted from `HttpServletRequest.getRemoteAddr()` (or `X-Forwarded-For` behind a trusted proxy). Inject `HttpServletRequest` into `getContractOpcodes()` and pass the IP to `throttleOpcodeRequest(String ip)`.
2. **Tighten the global cap as a secondary guard:** Keep the global bucket as a hard ceiling but set it to a value that accommodates legitimate concurrent users (e.g., 10–20 req/sec globally, 1 req/sec per IP).
3. **Infrastructure-level mitigation:** Deploy a reverse proxy (nginx, Traefik, GCP Cloud Armor) with per-source-IP rate limiting on this path as a defense-in-depth measure.

### Proof of Concept
```bash
# Terminal 1 – attacker (runs indefinitely at 1 req/sec)
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "https://<host>/api/v1/contracts/results/0xDEADBEEF.../opcodes"
  sleep 0.9   # slightly under 1s to stay ahead of refill
done

# Terminal 2 – legitimate user (concurrent)
curl -v -H "Accept-Encoding: gzip" \
  "https://<host>/api/v1/contracts/results/0xLEGIT.../opcodes"
# Expected result: HTTP 429 Too Many Requests
# {"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}
```
The legitimate user in Terminal 2 will receive HTTP 429 for as long as Terminal 1 is running, because the single global token is consumed before the legitimate request arrives.
