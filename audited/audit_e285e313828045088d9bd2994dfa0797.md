### Title
Missing `deleted` Filter on `entity_history` Branch in `findActiveByEvmAddressAndTimestamp` Returns Deleted Entities as Active at Historical Timestamps

### Summary
`findActiveByEvmAddressAndTimestamp` applies `deleted is not true` only to the `entity` branch of its UNION query, but omits any deleted check on the `entity_history` branch. This is directly inconsistent with `findActiveByIdAndTimestamp`, which correctly applies `deleted is not true` to both branches. As a result, any entity_history record — including those with `deleted = true` or `deleted = NULL` from a post-deletion state — can be returned as "active" at a historical timestamp when queried by EVM address.

### Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85 (`findActiveByEvmAddressAndTimestamp`):

```sql
-- entity branch: correctly guarded
select * from entity e
where e.deleted is not true          -- line 70: guards entity table
and e.id = (select id from entity_cte)

union all

-- entity_history branch: NO deleted check
select * from entity_history eh
where lower(eh.timestamp_range) <= ?2   -- line 77: only timestamp filter
and eh.id = (select id from entity_cte) -- line 78: no deleted check at all
order by lower(eh.timestamp_range) desc
limit 1
```

Compare with `findActiveByIdAndTimestamp` (lines 136–155), which correctly has `and deleted is not true` on **both** branches (lines 141 and 148).

**Root cause:** The `entity_history` branch in `findActiveByEvmAddressAndTimestamp` (and identically in `findActiveByEvmAddressOrAliasAndTimestamp`, lines 109–116) has no `deleted` predicate. The `entity_cte` (lines 60–65) also has no deleted check — it only requires `evm_address = ?1 and created_timestamp <= ?2` — so it resolves the entity ID even for currently-deleted entities.

**Failed assumption:** The developer assumed that the entity branch's `deleted is not true` guard is sufficient to prevent deleted entities from being returned. It is not, because the outer `ORDER BY timestamp_range DESC LIMIT 1` can select the entity_history row over the entity row (or the entity row can be absent from the UNION result entirely when `deleted = true`), leaving the unguarded entity_history row as the sole result.

**Exploit flow:**

1. Entity A (EVM address `0xABCD`) is created at T1. `entity` table: `deleted = NULL`.
2. Entity A is modified at T2. Old state written to `entity_history` with `deleted = NULL`, `timestamp_range = [T1, T2)`.
3. Entity A is deleted at T3. `entity` table updated: `deleted = true`. A deletion record is written to `entity_history` with `deleted = true`, `timestamp_range = [T3, ...)`.
4. Attacker calls `eth_call` or `eth_getBalance` at historical block timestamp T3 (or T3+1) for address `0xABCD`.
5. Query execution:
   - `entity_cte`: finds entity A's ID (no deleted check, `created_timestamp T1 <= T3` ✓)
   - Entity branch: `deleted = true` → filtered out → **no row**
   - `entity_history` branch: most recent record with `lower(timestamp_range) <= T3` is the deletion record (`deleted = true`, `lower = T3`) → **returned with no deleted check**
   - `ORDER BY timestamp_range DESC LIMIT 1` → deletion record selected
6. The function returns the deleted entity as if it were active.

**Why the existing test (line 126–132) does not catch this:**
```java
void findHistoricalEntityByEvmAddressAndTimestampRangeAndDeletedTrueCall() {
    final var entityHistory = persistEntityHistoryWithDeleted(); // only entity_history, NO entity record
    assertThat(entityRepository.findActiveByEvmAddressAndTimestamp(
                    entityHistory.getEvmAddress(), entityHistory.getTimestampLower() - 1)) // timestamp BEFORE record
            .isEmpty();
}
```
`persistEntityHistoryWithDeleted()` (line 547–549) creates **only** an `EntityHistory` row — no `entity` table row. The `entity_cte` therefore returns nothing, making the entire query empty regardless of any deleted check. The test also uses `getTimestampLower() - 1`, which would exclude the record by timestamp anyway. Neither condition tests the actual deleted-check gap.

### Impact Explanation
An unprivileged user can issue historical EVM queries (`eth_call`, `eth_getBalance`, `eth_estimateGas` at a past block) targeting the EVM address of a deleted account or contract. The mirror node will resolve the entity as active, returning stale state (balance, bytecode, storage) that should be inaccessible. This enables:
- Incorrect historical balance reporting for deleted accounts, which can be used to mislead off-chain systems relying on mirror node historical data.
- Incorrect historical contract existence/bytecode resolution, enabling replays of historical calls against contracts that were deleted at the queried timestamp.
- Manipulation of transaction history reconstruction tools that rely on `findActiveByEvmAddressAndTimestamp` to determine entity liveness at a given block.

Severity: **Medium**. No direct on-chain fund movement is possible, but the integrity of historical state served by the mirror node is compromised.

### Likelihood Explanation
Any unprivileged user with knowledge of a deleted entity's EVM address can trigger this by issuing a standard JSON-RPC historical query. No special credentials, private keys, or privileged access are required. The EVM addresses of deleted entities are publicly visible on-chain and in the mirror node's REST API. The attack is trivially repeatable and requires no timing precision beyond knowing the deletion timestamp (also publicly available).

### Recommendation
Add `and eh.deleted is not true` to the `entity_history` branch in both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp` (line 148):

```sql
-- findActiveByEvmAddressAndTimestamp, entity_history branch (line 76-81)
select * from entity_history eh
where lower(eh.timestamp_range) <= ?2
and eh.id = (select id from entity_cte)
and eh.deleted is not true          -- ADD THIS
order by lower(eh.timestamp_range) desc
limit 1
```

Apply the same fix to `findActiveByEvmAddressOrAliasAndTimestamp` (lines 110–116). Add a regression test that:
1. Creates both an `entity` record (deleted=true) AND an `entity_history` record (deleted=true) with the same ID.
2. Queries `findActiveByEvmAddressAndTimestamp` at `getTimestampLower()` (not `getTimestampLower() - 1`).
3. Asserts the result is empty.
