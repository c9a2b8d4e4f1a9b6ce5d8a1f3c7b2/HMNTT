### Title
Unbounded Per-Opcode Memory Snapshot Accumulation Causes OOM Crash via Single `memory=true` Request

### Summary
When `memory=true` is passed to `getContractOpcodes()`, `captureMemory()` in `AbstractOpcodeTracer` reads and hex-encodes the entire EVM memory frame at every single opcode step, appending each snapshot to an unbounded `ArrayList<Opcode>`. For a transaction that allocated significant EVM memory and executed many cheap opcodes, a single request can force the JVM to allocate tens to hundreds of gigabytes of heap, triggering an `OutOfMemoryError` that crashes the web3 service and halts its ability to process any further requests.

### Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` accepts `memory=true` from any caller with no additional validation beyond the gzip header check and a 1-req/sec rate limit. [1](#0-0) 

This flows into `OpcodeServiceImpl.processOpcodeCall()`, which creates an `OpcodeContext` with an unbounded `ArrayList`: [2](#0-1) 

`OpcodeContext.addOpcodes()` appends every opcode with no size cap: [3](#0-2) 

On every opcode step, `OpcodeActionTracer.tracePostExecution()` calls `captureMemory()`: [4](#0-3) 

`captureMemory()` reads the **entire** EVM memory at that instant and converts every 32-byte word to a hex string, with no cap on `size`: [5](#0-4) 

**Root cause:** There is no upper bound on (a) the number of `Opcode` entries accumulated in `OpcodeContext.opcodes`, nor (b) the size of the `List<String>` memory snapshot attached to each entry.

**Exploit math (conservative):**
- Hedera max gas limit: 15,000,000 gas
- Optimal split: spend ~5.1M gas expanding EVM memory to ~1.6 MB (50,596 words); remaining ~9.9M gas executes ~9.9M cheap opcodes (e.g., `JUMPDEST` at 1 gas each)
- Per-opcode memory snapshot: 50,596 strings × ~66 bytes = ~3.2 MB
- Total heap required: 9.9M opcodes × ~3.2 MB = **~31 TB** (JVM OOM occurs far earlier, crashing the process)

**Why existing checks fail:**

1. **gzip requirement** (`validateAcceptEncodingHeader`) — only reduces network transfer size; the server-side heap allocation happens entirely before any bytes are sent to the client. [6](#0-5) 

2. **Rate limit** (`throttleOpcodeRequest`, default 1 req/sec) — prevents request flooding but does nothing to bound the resource consumption of a single request. [7](#0-6) [8](#0-7) 

3. **`requestTimeout` (10 s)** — the EVM replay and memory capture loop runs synchronously on a request thread; the timeout may not interrupt mid-loop heap growth before OOM is reached.

### Impact Explanation
An `OutOfMemoryError` in the web3 service process terminates the JVM (or causes cascading GC thrashing that makes the node unresponsive). This halts all contract call processing, including the relay of new transactions, directly matching the "Critical: Network not being able to confirm new transactions" severity tier. A single well-chosen transaction hash is sufficient; no repeated requests are needed.

### Likelihood Explanation
The endpoint is **disabled by default** (`hiero.mirror.web3.opcode.tracer.enabled = false`), which is the primary mitigating factor. [9](#0-8) 

When enabled (e.g., for debugging or developer tooling), any unauthenticated caller can trigger this. The attacker only needs to identify one historical transaction with high memory usage — such transactions are publicly visible on-chain. No special privileges, credentials, or prior relationship with the node are required. The attack is repeatable after a service restart.

### Recommendation

1. **Cap total opcodes:** Enforce a configurable maximum on `OpcodeContext.opcodes` size (e.g., 100,000 entries) and return an error or truncated result when exceeded.
2. **Cap per-opcode memory snapshot size:** In `captureMemory()`, add a guard: if `frame.memoryWordSize()` exceeds a configurable threshold (e.g., 1,024 words / 32 KB), skip capture or return a truncated/summarized snapshot.
3. **Add a gas-based pre-check:** Before replaying, estimate the worst-case output size from the stored gas used and reject requests likely to exceed a configured response size budget.
4. **Stream the response:** Instead of accumulating all opcodes in a `List` before serializing, stream `Opcode` objects directly to the HTTP response writer to avoid holding the full dataset in heap simultaneously.
5. **Require authentication/authorization** for the opcodes endpoint when enabled, to restrict access to trusted operators.

### Proof of Concept

**Preconditions:**
- `hiero.mirror.web3.opcode.tracer.enabled=true`
- A historical transaction exists that allocated significant EVM memory (e.g., a contract that `MSTORE`s into a large offset, consuming most of the 15M gas budget on memory expansion, then loops with `JUMPDEST`)

**Steps:**
```
# 1. Identify a memory-heavy transaction hash from the public ledger
TX_HASH="0x<hash_of_memory_heavy_transaction>"

# 2. Send a single request with memory=true
curl -H "Accept-Encoding: gzip" \
     "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?memory=true&stack=false&storage=false"

# Result: web3 service JVM throws OutOfMemoryError during captureMemory() loop,
# crashing the process and halting all subsequent transaction processing.
```

    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
    }
