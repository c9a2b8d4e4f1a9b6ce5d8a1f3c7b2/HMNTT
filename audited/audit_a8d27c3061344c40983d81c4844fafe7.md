### Title
Unvalidated `req.method` in `metricsMiddleware` Enables Metric Label Cardinality Pollution by Unprivileged Attackers

### Summary
The `metricsMiddleware` function in `rest/middleware/metricsHandler.js` records `req.method` directly into Prometheus/OpenTelemetry metric labels without any allowlist validation. Because Node.js's `llhttp` HTTP parser accepts ~40+ method tokens beyond GET/POST (including WebDAV verbs like `PROPFIND`, `PURGE`, `MKCOL`, `LOCK`, etc.), any unauthenticated external client can craft requests with non-standard methods against Hashgraph history endpoints, injecting arbitrary `{method, path, code}` label combinations into `requestTotalCounter` and `durationHistogram`. No existing middleware in the stack rejects these requests before the `finish` event fires and the metrics are committed.

### Finding Description

**Exact code location:**

`rest/middleware/metricsHandler.js`, lines 191–208, inside the `res.on('finish', ...)` callback of `metricsMiddleware`:

```js
// line 191
const method = req.method;          // raw, unvalidated
const labels = {method, path, code};
// ...
requestTotalCounter.add(1, labels);       // line 207
durationHistogram.record(duration, labels); // line 208
requestSizeHistogram.record(..., labels);
responseSizeHistogram.record(..., labels);
```

`req.method` is taken verbatim from the parsed HTTP request and placed into every per-route instrument label set with no allowlist check.

**Root cause / failed assumption:**
The code assumes `req.method` will always be a standard, bounded HTTP verb (GET, POST, etc.). This assumption is false: Node.js's `llhttp` parser accepts a large set of method tokens including `PATCH`, `DELETE`, `PURGE`, `PROPFIND`, `MKCOL`, `COPY`, `MOVE`, `LOCK`, `UNLOCK`, `PROPPATCH`, `SEARCH`, `REPORT`, `MKACTIVITY`, `CHECKOUT`, `MERGE`, `MKCALENDAR`, `LINK`, `UNLINK`, `SOURCE`, and others. Express sets `req.method` to whatever the parser accepted.

**Why existing checks fail:**

1. **`openApiValidator`** (`rest/middleware/openapiHandler.js`, lines 148–158) is configured with `validateRequests: false` — it performs no request-level method validation. It is also conditionally loaded only when `config.openapi.validation.enabled` is true (production default may be false).

2. **`metricsHandler` registration order** (`rest/server.js`, lines 89–92): the metrics middleware is registered as a global `app.useExt` handler *before* any route-specific handlers. The `res.on('finish', ...)` callback fires after the response is sent regardless of whether Express returned 404 or 405 — the metrics are always committed.

3. **No method allowlist** exists anywhere in the middleware chain (`authHandler`, `requestLogger`, `requestQueryParser`, `responseCacheCheckHandler`) that would normalize or reject non-standard methods before the finish event.

**Exploit flow:**
- Attacker sends `PROPFIND /api/v1/transactions HTTP/1.1` → Express finds no matching route → returns 404 → `finish` fires → metrics record `{method:"PROPFIND", path:"/api/v1/transactions", code:"404"}`.
- Attacker repeats with `PURGE`, `LOCK`, `MKCOL`, `PATCH`, `DELETE`, etc. across all API paths (`/api/v1/accounts`, `/api/v1/blocks`, `/api/v1/tokens`, `/api/v1/topics/:topicId/messages`, etc.).

### Impact Explanation

Each unique `{method, path, code}` triple creates a new time series in the OpenTelemetry/Prometheus metrics store. With ~40 accepted non-standard methods × ~20 API paths × several response codes, an attacker can inject hundreds of spurious time series into `api_request_total`, `api_request_duration_milliseconds`, `api_request_size_bytes`, and `api_response_size_bytes`. Concrete consequences:

- **Monitoring blind spots:** Dashboards and alerts keyed on method-path combinations (e.g., "alert if non-GET traffic appears on history endpoints") become unreliable or fire false positives, masking real anomalous write-like attempts.
- **Memory pressure:** The OpenTelemetry SDK accumulates all label combinations in memory; sustained flooding can cause heap growth in the mirror-node REST process.
- **Operational confusion:** SRE/security teams investigating traffic patterns on Hashgraph history endpoints see fabricated method labels, making forensic analysis of real read/write patterns harder.

Severity: **Medium** — no Hashgraph ledger data is modified, but the integrity of the observability layer for history endpoints is compromised by an unprivileged attacker.

### Likelihood Explanation

- **No authentication required:** The metrics instrumentation runs before any auth check and for all requests.
- **Trivially reproducible:** Any HTTP client (`curl`, `netcat`) can send non-standard methods; no special tooling needed.
- **Repeatable at scale:** A simple loop over the ~40 llhttp-accepted methods × all API paths takes seconds and can be scripted.
- **Persistent effect:** Each injected label combination persists in the metrics store for the lifetime of the process (or until scrape/reset).

### Recommendation

Add a method normalization/allowlist step inside `metricsMiddleware` before constructing `labels`:

```js
const ALLOWED_METHODS = new Set(['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'HEAD', 'OPTIONS']);

const method = ALLOWED_METHODS.has(req.method) ? req.method : 'UNKNOWN';
const labels = {method, path, code};
```

This caps the `method` label cardinality to a fixed, known set and prevents any non-standard verb from creating new time series. Alternatively, normalize to `'OTHER'` for any method outside the expected set for this read-only API (which only needs `GET` and `HEAD`).

### Proof of Concept

```bash
# Send WebDAV PROPFIND to a Hashgraph history endpoint
curl -X PROPFIND http://<mirror-node-rest>:5551/api/v1/transactions -v

# Send PURGE (cache-busting verb accepted by llhttp)
curl -X PURGE http://<mirror-node-rest>:5551/api/v1/transactions -v

# Loop all non-standard methods across history endpoints
for METHOD in PROPFIND MKCOL COPY MOVE LOCK UNLOCK PURGE PATCH DELETE LINK UNLINK SOURCE REPORT CHECKOUT MERGE; do
  for PATH in /api/v1/transactions /api/v1/accounts /api/v1/blocks /api/v1/tokens /api/v1/topics/1/messages; do
    curl -s -X $METHOD http://<mirror-node-rest>:5551$PATH -o /dev/null
  done
done

# Scrape metrics and observe fabricated method labels
curl http://<mirror-node-rest>:5551/api/v1/metrics/ | grep api_request_total
# Expected: lines with method="PROPFIND", method="PURGE", etc. alongside legitimate method="GET"
```
