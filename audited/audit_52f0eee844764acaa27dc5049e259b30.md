### Title
Missing JDBC Socket Timeout Causes Thread Exhaustion via Network Partition in `entityService.lookup()`

### Summary
`NftAllowanceServiceImpl.getNftAllowances()` calls `entityService.lookup()` synchronously, which issues blocking JDBC queries against PostgreSQL for alias and EVM-address account IDs. No JDBC-level `socketTimeout` is configured on the runtime HikariCP connection pool, so a network partition between the application and the database causes the JDBC socket read to block for the full OS TCP timeout (minutes), holding a Tomcat thread for the entire duration. Any unprivileged user can trigger this code path by supplying a valid alias or EVM-address account ID, and with enough concurrent requests during a partition, the thread pool is exhausted.

**Root cause — failed assumption:**

The design assumes that the PostgreSQL server-side `statement_timeout` (set to 10 000 ms for the `mirror_rest_java` user via `ALTER USER ... SET statement_timeout`) will bound query duration. This assumption fails during a network partition: the server fires the timeout and sends a cancellation packet, but the JDBC driver's socket `read()` call is blocked waiting for data that can never arrive. Without a client-side `socketTimeout` in the JDBC URL or HikariCP `dataSourceProperties`, the driver blocks indefinitely (until the OS TCP keepalive fires, typically 2–15 minutes).

The HikariCP pool is built from `spring.datasource.hikari` properties with no `socketTimeout` or `connectionTimeout` covering in-flight queries: [4](#0-3) 

The `socketTimeout` that exists in `CommonProperties.DatabaseStartupProperties` (2 s) is used only by `DatabaseWaiter` at startup, not by the runtime pool: [5](#0-4) 

Unlike the web3 module, rest-java has no application-level request timeout or `StatementInspector` to bound query time. The only documented rest-java DB timeout is the server-side `statementTimeout`: [6](#0-5) 

**Numeric IDs are safe; alias/EVM-address IDs are not.** The `EntityIdNumParameter` branch returns immediately without a DB call. Only `EntityIdAliasParameter` and `EntityIdEvmAddressParameter` issue queries, and both are reachable by any unauthenticated caller supplying a valid alias or EVM address in the `account.id` query parameter.

### Impact Explanation

During a network partition, each in-flight request that hits the alias/EVM-address branch holds a Tomcat worker thread blocked in a JDBC `read()` for up to the OS TCP timeout. The Traefik middleware configured in the Helm chart includes a **retry policy of 3 attempts**, which triples thread consumption per client request: [7](#0-6) 

Thread pool exhaustion causes the service to stop accepting new connections, resulting in a complete denial of service for all endpoints, not just the NFT allowance endpoint. Upstream proxies that time out and retry amplify the effect further.

### Likelihood Explanation

The attacker requires no credentials or special privileges — only the ability to send HTTP GET requests with an alias or EVM-address `account.id` parameter. Network partitions between application pods and the database are a realistic operational event in Kubernetes environments (network policy changes, node failures, CNI bugs). An attacker who can observe or induce elevated latency (e.g., via a volumetric attack against the DB network path) can reliably trigger the blocking condition. The attack is repeatable: each new request during the partition consumes another thread.

### Recommendation

1. **Add a JDBC `socketTimeout`** to the PostgreSQL JDBC URL for the rest-java runtime pool (e.g., `socketTimeout=15` seconds), ensuring the driver unblocks when the network is unavailable. This is the primary fix.
2. **Add a HikariCP `connectionTimeout`** that covers the full query lifecycle, not just pool acquisition.
3. **Add an application-level request timeout** analogous to the web3 module's `requestTimeout` + `StatementInspector` pattern, so that even if the JDBC layer blocks, the HTTP thread is released.
4. **Configure a Traefik request timeout** on the middleware to prevent the proxy from holding connections open indefinitely and retrying against an already-blocked backend.

### Proof of Concept

**Preconditions:** Network partition between the rest-java pod and the PostgreSQL database (e.g., via `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the app node, or a network policy change).

**Trigger:**
```
# Send concurrent requests with an EVM-address account ID (forces DB lookup)
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.000000000000000000000000000000000186Fb1b/allowances/nfts" &
done
wait
```

**Result:**
- Each request enters `EntityServiceImpl.lookup()` → `entityRepository.findByEvmAddress()` → JDBC `read()` blocks.
- With the partition active, all 200 threads block simultaneously for the OS TCP timeout duration (minutes).
- The Tomcat thread pool is exhausted; subsequent requests to any endpoint receive connection-refused or gateway-timeout errors.
- Traefik's retry policy (3 attempts) causes each client request to consume up to 3 threads, reducing the number of clients needed to exhaust the pool by a factor of 3.

    public EntityId lookup(EntityIdParameter accountId) {
        var id = switch (accountId) {
            case EntityIdNumParameter p -> Optional.of(p.id());
            case EntityIdAliasParameter p -> entityRepository.findByAlias(p.alias()).map(EntityId::of);
            case EntityIdEvmAddressParameter p -> entityRepository.findByEvmAddress(p.evmAddress()).map(EntityId::of);
        };

        return id.orElseThrow(() -> new EntityNotFoundException("No account found for the given ID"));
