### Title
Unauthenticated Repeated `/network/status` Calls Trigger Uncached Expensive SQL Query, Enabling DB Resource Exhaustion

### Summary
Every call to the Rosetta `/network/status` endpoint unconditionally executes the `latestNodeServiceEndpoints` SQL query — which includes a `MAX` subquery, two `JOIN`s, a `LEFT JOIN`, `GROUP BY`, and `string_agg` aggregation — with no application-level caching or rate limiting. Any unauthenticated external user can flood this endpoint, causing each request to hit the database with a full aggregation query, driving DB CPU and I/O well above the 30% baseline threshold.

### Finding Description

**Code path:**

`POST /network/status` → `networkAPIService.NetworkStatus()` (line 77) → `addressBookEntryRepo.Entries(ctx)` → executes `latestNodeServiceEndpoints` raw SQL. [1](#0-0) 

The `Entries()` implementation in the Go Rosetta persistence layer: [2](#0-1) 

executes this query on every invocation with no caching: [3](#0-2) 

The query involves:
- A correlated `MAX` subquery over `address_book`
- A `JOIN` on `address_book_entry`
- A `LEFT JOIN` on `node`
- A `LEFT JOIN` on `address_book_service_endpoint`
- A `GROUP BY` + `string_agg` aggregation

Critically, the loop iterates over **both** file IDs (101 and 102), potentially executing the query **twice** per request if file 101 returns no rows.

**Root cause — no caching:** The Java gRPC `AddressBookEntryRepository` uses `@Cacheable` with a Caffeine cache: [4](#0-3) 

The Rosetta Go `addressBookEntryRepository` has no equivalent caching mechanism whatsoever. [5](#0-4) 

**Root cause — no application-level rate limiting:** The middleware chain in `main.go` only wraps with metrics, tracing, and CORS: [6](#0-5) 

The Traefik-level rate limit (`average: 10` per `requestHost`) is gated behind `global.middleware`, which defaults to `false`: [7](#0-6) [8](#0-7) 

Even when enabled, `requestHost`-based limiting (not per-IP) is trivially bypassed by spoofing the `Host` header or using multiple source IPs.

### Impact Explanation

Each `/network/status` request causes a full table scan + aggregation across `address_book`, `address_book_entry`, `node`, and `address_book_service_endpoint`. On a production network with dozens of nodes and many service endpoints, this is a non-trivial I/O and CPU operation. A single attacker sending requests at even modest concurrency (e.g., 50–100 req/s from a single machine) will generate a sustained stream of these aggregation queries, driving DB CPU and I/O above the 30% baseline. The `db.statementTimeout` of 20 seconds means each query can hold a DB connection for up to 20 seconds, compounding connection pool exhaustion alongside CPU/I/O pressure. [9](#0-8) 

### Likelihood Explanation

The preconditions are minimal: the attacker needs only network access to port 5700 and a valid `network_identifier`, which is freely obtained from the unauthenticated `/network/list` endpoint. No credentials, tokens, or special privileges are required. The attack is trivially scriptable with `curl` or any HTTP client in a loop. The Rosetta API is by design a public-facing interface intended for blockchain data consumers, making this attack surface inherently exposed.

### Recommendation

1. **Add in-process caching** to `addressBookEntryRepository.Entries()`: cache the result with a TTL (e.g., 30–60 seconds) using a `sync.Map` or a library like `ristretto`/`bigcache`. The address book changes infrequently (only on network upgrades), so a short TTL is sufficient.
2. **Add application-level rate limiting** in the Go HTTP server (e.g., `golang.org/x/time/rate` per-IP token bucket) rather than relying on optional external Traefik middleware.
3. **Enable the Traefik middleware by default** and switch `sourceCriterion` from `requestHost` to `ipStrategy` to prevent trivial bypass.
4. **Add a DB connection pool limit** specifically for the Rosetta user to cap the blast radius.

### Proof of Concept

```bash
# Step 1: Discover the network identifier (unauthenticated)
NETWORK=$(curl -s -X POST http://<rosetta-host>:5700/network/list \
  -H 'Content-Type: application/json' \
  -d '{"metadata":{}}' | jq '.network_identifiers[0]')

# Step 2: Flood /network/status — each request triggers the expensive SQL query
BODY=$(jq -n --argjson n "$NETWORK" '{"network_identifier": $n}')

for i in $(seq 1 500); do
  curl -s -X POST http://<rosetta-host>:5700/network/status \
    -H 'Content-Type: application/json' \
    -d "$BODY" &
done
wait

# Observe: DB CPU and I/O spike on each wave of requests.
# Monitor with: SELECT query, calls, total_exec_time FROM pg_stat_statements
#               WHERE query LIKE '%string_agg%' ORDER BY total_exec_time DESC;
```

Each concurrent request independently executes the `latestNodeServiceEndpoints` aggregation query (up to twice per request), with no deduplication, caching, or throttling at the application layer.

const latestNodeServiceEndpoints = `select
                                    abe.node_id,
                                    coalesce(n.account_id, abe.node_account_id) as node_account_id,
                                    string_agg(ip_address_v4 || ':' || port::text, ','
                                      order by ip_address_v4,port) endpoints
                                  from (
                                    select max(start_consensus_timestamp) from address_book where file_id = @file_id
                                  ) current
                                  join address_book_entry abe on abe.consensus_timestamp = current.max
                                  left join node n on n.node_id = abe.node_id
                                  left join address_book_service_endpoint abse
                                    on abse.consensus_timestamp = current.max and abse.node_id = abe.node_id
                                  group by abe.node_id, n.account_id, abe.node_account_id`

func (aber *addressBookEntryRepository) Entries(ctx context.Context) (*types.AddressBookEntries, *rTypes.Error) {
	db, cancel := aber.dbClient.GetDbWithContext(ctx)
	defer cancel()

	nodes := make([]nodeServiceEndpoint, 0)
	// address book file 101 has service endpoints for nodes, resort to file 102 if 101 doesn't exist
	for _, fileId := range []int64{aber.addressBook101.EncodedId, aber.addressBook102.EncodedId} {
		if err := db.Raw(
			latestNodeServiceEndpoints,
			sql.Named("file_id", fileId),
		).Scan(&nodes).Error; err != nil {
			log.Error("Failed to get latest node service endpoints", err)
			return nil, errors.ErrDatabaseError
		}

		if len(nodes) != 0 {
			break
		}
	}

	entries := make([]types.AddressBookEntry, 0, len(nodes))
	for _, node := range nodes {
		entries = append(entries, node.toAddressBookEntry())
	}

	return &types.AddressBookEntries{Entries: entries}, nil
}
