### Title
Unauthenticated Full-Table Scan of `token_account` via Unguarded CTE in `getEntityBalanceQuery()`

### Summary
Any unauthenticated external user can issue a plain `GET /api/v1/accounts` request (no parameters required) and deterministically trigger a CTE that reads every row in `token_account` where `associated is true` with no account-scoped predicate. Because the CTE carries no per-account filter and the only available indexes on `token_account` are on `(account_id, token_id)` and `(token_id, account_id)`, the database must perform a sequential scan of the entire table on every such request. Repeated requests saturate DB I/O and can partition the REST service from the database.

### Finding Description

**Exact code path**

`rest/accounts.js`, function `getEntityBalanceQuery()`, lines 183–186:

```js
queries.push(`with latest_token_balance as (
   select account_id, balance, token_id
   from token_account
   where associated is true)`);
```

**When is this branch reached?**

The branch is guarded by `needHistoricalBalanceInfo` (line 164):

```js
const needHistoricalBalanceInfo = accountBalanceQuery.query || accountBalanceQuery.forceUnionEntityHistory;
```

For the `/accounts` list endpoint (`getAccounts`, line 335), `accountBalanceQuery` is never passed — it defaults to `{query: '', params: []}` (line 257). An empty string is falsy, so `needHistoricalBalanceInfo` is always `false`, and the CTE branch is **unconditionally entered** for every call to `GET /api/v1/accounts`, regardless of any query parameters the caller supplies or omits.

**Why the CTE is dangerous**

The CTE contains no `account_id` predicate. The only filter is `associated is true`. The indexes on `token_account` are:

- Primary key `(account_id, token_id)` (migration `V1.66.1`)
- Secondary index `token_account__token_id_account_id` on `(token_id, account_id)` (migration `V2.18.0`)

Neither index is usable for a bare `associated = true` filter. PostgreSQL must perform a sequential scan of the entire table. In PostgreSQL < 12 the CTE is an optimization fence and is always materialized before the outer query runs. In PostgreSQL ≥ 12 the planner *may* inline it, but even then the `associated` column has no index, so a seq-scan is the only plan available.

The inner query that consumes the CTE (lines 187–195) applies `account_id = e.id` and a `LIMIT`, but those predicates are applied **after** the CTE is fully materialized, not pushed into it.

**Authentication / rate-limiting checks**

`authHandler` (middleware/authHandler.js, lines 18–20) explicitly returns without blocking when no `Authorization` header is present:

```js
if (!credentials) {
  return;
}
```

There is no rate-limiting middleware in the server pipeline (server.js, lines 67–99). The `limit` query parameter only caps the number of account rows returned; it does not reduce the CTE scan.

**Exploit flow**

1. Attacker sends `GET /api/v1/accounts` — no credentials, no parameters.
2. `getAccounts` calls `getAccountQuery` with `accountBalanceQuery = undefined` → defaults to `{query:'', params:[]}`.
3. `getEntityBalanceQuery` evaluates `needHistoricalBalanceInfo = '' || undefined = false`.
4. The CTE `select … from token_account where associated is true` is emitted with no account filter.
5. PostgreSQL seq-scans the entire `token_account` table.
6. Attacker repeats at high frequency; each request issues a new full scan.

### Impact Explanation

On a production Hedera mirror node `token_account` contains tens of millions of rows (Hedera mainnet has >100 M token associations). Each unauthenticated request forces a full sequential scan of this table. Sustained parallel requests saturate disk I/O and shared-buffer bandwidth, causing query latency to spike across all DB-backed endpoints. If the DB connection pool is exhausted or query timeouts are hit, the REST service becomes unavailable — a complete denial of service achievable without any credentials.

### Likelihood Explanation

The endpoint is public, requires zero authentication, and accepts zero parameters. The attacker needs only a network path to the REST API. The attack is trivially scriptable (`while true; do curl /api/v1/accounts & done`), fully repeatable, and requires no knowledge of internal state. Any internet-facing deployment is exposed.

### Recommendation

1. **Add an `account_id` predicate inside the CTE** so the scan is always bounded:
   ```sql
   with latest_token_balance as (
     select account_id, balance, token_id
     from token_account
     where associated is true
       and account_id = ANY($account_ids_from_outer_query)
   )
   ```
   Or restructure the query to use a correlated subquery / lateral join so the `account_id` filter is always pushed down.

2. **Add an index on `(associated, account_id)`** or a partial index `WHERE associated IS TRUE` on `(account_id, token_id)` so that even if the CTE form is retained, the planner can use an index scan.

3. **Add rate limiting** (e.g., express-rate-limit) for unauthenticated callers on all `/api/v1/*` routes.

4. **Set a `statement_timeout`** on the DB connection pool so runaway scans are killed before they saturate I/O.

### Proof of Concept

```bash
# Single request — triggers full token_account scan
curl -s "https://<mirror-node-host>/api/v1/accounts" > /dev/null

# DoS amplification — 50 concurrent requests, repeated
for i in $(seq 1 50); do
  curl -s "https://<mirror-node-host>/api/v1/accounts" > /dev/null &
done
wait
# Repeat in a loop; DB I/O saturates within seconds on large datasets.
```

Verify by checking `pg_stat_activity` on the database during the attack:
```sql
SELECT query, state, wait_event_type, wait_event
FROM pg_stat_activity
WHERE query LIKE '%token_account%associated%';
-- Will show multiple concurrent seq-scan workers on token_account
```
