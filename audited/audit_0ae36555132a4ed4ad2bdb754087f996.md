### Title
Unbounded Concurrent gRPC Subscriptions Enable Resource Exhaustion via Unauthenticated `subscribeTopic`

### Summary
An unauthenticated external user can open an unlimited number of long-lived `subscribeTopic` streams by sending `ConsensusTopicQuery` with `limit=1` and `consensusStartTime` set to the current time. Each such subscription holds a live reactive stream context indefinitely waiting for one message that may never arrive. No global subscriber cap, no per-IP rate limit, and no idle-subscription timeout exist on the gRPC path, allowing an attacker to exhaust server memory, Reactor scheduler threads, and database listener slots.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` ( [1](#0-0) ) accepts any `ConsensusTopicQuery` without authentication or rate-limiting and delegates to `TopicMessageServiceImpl.subscribeTopic()`.

Inside `TopicMessageServiceImpl.subscribeTopic()`: [2](#0-1) 

With `limit=1` and `startTime=now`:
1. `historical = topicMessageRetriever.retrieve(filter, true)` returns empty (no messages at current time).
2. `live = Flux.defer(() -> incomingMessages(topicContext))` calls `topicListener.listen(newFilter)` and waits indefinitely for the next message.
3. `pastEndTime(topicContext)` returns `Flux.never()` because no `endTime` is set: [3](#0-2) 
4. `flux.take(1)` keeps the stream open until exactly one message arrives.

**Root cause:** No idle-subscription timeout is applied to the live listener path. The `retriever.timeout` (60 s) governs only the historical retrieval polling interval between emissions, not the live `topicListener.listen()` path. The `subscriberCount` field is a Micrometer gauge only — it is never checked against a maximum: [4](#0-3) 

**Validation constraint that does NOT block the attack:** `TopicMessageFilter.isValidStartTime()` only requires `startTime <= DomainUtils.now()`, so setting `startTime` to the current nanosecond is fully valid: [5](#0-4) 

**The only per-connection guard** is `maxConcurrentCallsPerConnection = 5`: [6](#0-5)  applied in: [7](#0-6) 

This limits calls *per TCP connection*, not globally. An attacker opens N connections and gets 5N concurrent live subscriptions. No global connection limit, no IP-based rate limit, and no authentication are enforced on the gRPC module (unlike the web3 module which has `ThrottleConfiguration`): [8](#0-7) 

### Impact Explanation
Each open subscription holds: a `TopicContext` object, a Reactor `Disposable`, a slot in the `boundedElastic` scheduler (safety-check path), and a registration in the shared `TopicListener`. Exhausting these resources causes:
- JVM heap exhaustion → OOM / GC pressure → service crash.
- `Schedulers.boundedElastic()` thread pool saturation → legitimate subscriptions queue indefinitely.
- Database listener connection pool exhaustion → new subscriptions and historical queries fail.

This directly maps to the stated critical severity: *network not being able to confirm new transactions (total network shutdown)* because the mirror node gRPC API becomes unavailable to all clients, including those monitoring consensus.

### Likelihood Explanation
No privileges, tokens, or special accounts are required. Any internet-reachable gRPC endpoint (port 5600) is the attack surface. A single attacker machine can open thousands of TCP connections and multiplex 5 streams each using standard gRPC client libraries (e.g., `grpc-java`, `grpcurl`). The attack is trivially repeatable and scriptable. The GCP gateway `maxRatePerEndpoint: 250` limits new-request rate per backend pod but does not cap *concurrent open streams*, so it does not prevent accumulation of long-lived subscriptions.

### Recommendation
1. **Add a global concurrent-subscriber cap**: check `subscriberCount` against a configurable maximum before accepting a new subscription and return `RESOURCE_EXHAUSTED` if exceeded.
2. **Add an idle-subscription timeout on the live path**: apply `.timeout(Duration)` or a `takeUntil` timer to the `live` flux so subscriptions that receive no message within a configurable window (e.g., 5 minutes) are terminated.
3. **Add per-IP or per-connection subscription rate limiting** at the gRPC interceptor layer, analogous to the `ThrottleConfiguration` used in the web3 module.
4. **Enforce a maximum `endTime`** when no `endTime` is provided, capping how long an open-ended subscription can live.

### Proof of Concept
```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2
import time

TARGET = "mirror.mainnet.hedera.com:443"
NUM_CONNECTIONS = 500
STREAMS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection

def open_streams(conn_id):
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now = int(time.time())
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),  # any valid topic
        consensusStartTime=timestamp_pb2.Timestamp(seconds=now),
        limit=1  # waits for exactly 1 future message, never completes
    )
    streams = []
    for _ in range(STREAMS_PER_CONN):
        # Non-blocking: iterator is lazy, stream stays open server-side
        streams.append(stub.subscribeTopic(query))
    time.sleep(3600)  # hold connections open for 1 hour

threads = [threading.Thread(target=open_streams, args=(i,)) for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# Result: 500 * 5 = 2500 concurrent live subscriptions held open server-side,
# each consuming a TopicContext, Reactor scheduler slot, and listener registration.
# Scale NUM_CONNECTIONS to exhaust server resources.
```

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
