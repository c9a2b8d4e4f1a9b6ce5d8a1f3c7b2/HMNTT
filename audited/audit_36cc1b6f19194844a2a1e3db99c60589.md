### Title
Unbounded DB Load via Unauthenticated Historical Entity Queries with No Per-Request Caching and Bypassable Global Throttle

### Summary
`findActiveByEvmAddressAndTimestamp()` in `EntityRepository.java` carries no `@Cacheable` annotation, meaning every historical contract call with a unique `blockTimestamp` issues a fresh multi-table CTE+UNION query against the database. The only protection is a single global token-bucket throttle (default 500 req/s) shared across all users, with no per-IP enforcement. An unprivileged attacker can consume the entire global budget with varying timestamps, saturating the DB connection pool and degrading or denying historical query service to all other users.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85:

```java
@Query(value = """
        with entity_cte as (
            select id
            from entity
            where evm_address = ?1 and created_timestamp <= ?2
            order by created_timestamp desc
            limit 1
        )
        (
            select *
            from entity e
            where e.deleted is not true
            and e.id = (select id from entity_cte)
        )
        union all
        (
            select *
            from entity_history eh
            where lower(eh.timestamp_range) <= ?2
            and eh.id = (select id from entity_cte)
            order by lower(eh.timestamp_range) desc
            limit 1
        )
        order by timestamp_range desc
        limit 1
        """, nativeQuery = true)
Optional<Entity> findActiveByEvmAddressAndTimestamp(byte[] evmAddress, long blockTimestamp);
```

**Root cause — no `@Cacheable` on the historical variant:**

The non-historical counterparts are all cached:
- `findByEvmAddressAndDeletedIsFalse()` → `@Cacheable(CACHE_NAME_EVM_ADDRESS, CACHE_MANAGER_ENTITY)` (lines 32–37)
- `findByEvmAddressOrAliasAndDeletedIsFalse()` → `@Cacheable(CACHE_NAME_ALIAS, CACHE_MANAGER_ENTITY)` (lines 39–49)
- `findByIdAndDeletedIsFalse()` → `@Caching(...)` (lines 20–30)

`findActiveByEvmAddressAndTimestamp()` has **no caching annotation at all**. Every call hits the database.

**Throttle is global and bypassable:**

`ThrottleManagerImpl.throttle()` (lines 37–48) uses a single shared `rateLimitBucket` (default 500 req/s) and `gasLimitBucket`. Neither is per-IP or per-user. A single attacker can consume the entire 500 req/s budget.

Additionally, `ThrottleProperties.scaleGas()` (lines 42–47) returns `0L` for any `gas ≤ 10_000`:

```java
public long scaleGas(long gas) {
    if (gas <= GAS_SCALE_FACTOR) {   // GAS_SCALE_FACTOR = 10_000
        return 0L;
    }
    return Math.floorDiv(gas, GAS_SCALE_FACTOR);
}
```

`gasLimitBucket.tryConsume(0)` always returns `true` in bucket4j, so the gas throttle is completely bypassed for low-gas requests. Furthermore, `gasLimitRefundPercent = 100` (default) means failed calls restore 100% of gas to the bucket, making the gas throttle ineffective for calls that revert.

**Exploit flow:**

1. Attacker calls `POST /api/v1/contracts/call` with a historical block value (any non-`latest` block tag or block number).
2. The `ContractCallContext.useHistorical()` check (line 103–105 of `ContractCallContext.java`) returns `true`, so `CommonEntityAccessor.getEntityByEvmAddressTimestamp()` calls `entityRepository.findActiveByEvmAddressAndTimestamp(addressBytes, t)` (lines 80–83 of `CommonEntityAccessor.java`).
3. Each request with a distinct `blockTimestamp` is a cache miss (no cache exists), issuing a fresh CTE+UNION query across `entity` and `entity_history`.
4. With gas ≤ 10,000, `scaleGas` returns 0, bypassing the gas bucket entirely. Only the 500 req/s global rate limit applies.
5. Attacker saturates the 500 req/s budget, exhausting DB connections and degrading all concurrent users.

### Impact Explanation

Every historical `eth_call` that resolves an EVM address triggers at least one uncached DB query. Under sustained attack at the global rate limit (500 req/s), the PostgreSQL connection pool is saturated. Legitimate users querying historical Hashgraph state receive timeouts or errors (the `statementTimeout` is 3000 ms by default). This constitutes a denial-of-service against the mirror node's historical data serving capability — directly impacting the integrity and availability of Hashgraph history access.

### Likelihood Explanation

No authentication is required. The `/api/v1/contracts/call` endpoint is publicly accessible. The attacker needs only an HTTP client and knowledge of any valid EVM address. Varying the `block` parameter across requests (e.g., sequential block numbers) trivially generates unique `blockTimestamp` values, ensuring every request is a cache miss. The attack is repeatable, scriptable, and requires no special privileges or on-chain resources.

### Recommendation

1. **Add per-IP rate limiting** at the application or reverse-proxy layer (e.g., nginx `limit_req`) so a single source cannot consume the global budget.
2. **Cache historical entity queries** with a composite key of `(evmAddress, blockTimestamp)` using a bounded, short-TTL cache (historical data is immutable once finalized, so a TTL-based cache is safe).
3. **Fix the gas throttle bypass**: `scaleGas` returning `0` for gas ≤ 10,000 means those requests consume no gas tokens. Apply a minimum token cost of 1 for any request regardless of gas value.
4. **Separate throttle buckets** for historical vs. latest queries so historical query floods cannot starve latest-block queries.

### Proof of Concept

```bash
# Send 500 historical contract calls per second with varying block numbers
# Each call resolves an EVM address at a unique historical timestamp
# gas=10000 bypasses the gas throttle (scaleGas returns 0)

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{
      \"to\": \"0x<valid_evm_address>\",
      \"data\": \"0x<any_calldata>\",
      \"gas\": 10000,
      \"block\": \"$i\"
    }" &
done
wait

# Result: 500 concurrent uncached CTE+UNION queries against entity/entity_history tables
# DB connection pool exhausted; legitimate users receive timeouts
```
