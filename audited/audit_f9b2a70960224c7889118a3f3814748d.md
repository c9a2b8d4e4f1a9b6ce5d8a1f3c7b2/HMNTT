### Title
Unauthenticated 3x DB Query Amplification DoS via `GET /transactions/:transactionIdOrHash`

### Summary
An unprivileged external user can craft a single `GET /api/v1/transactions/{transactionId}?scheduled=true` request that deterministically triggers up to three sequential `pool.queryQuietly()` calls within `getTransactionsByIdOrHash()`. With a default connection pool of only 10 connections and no per-IP rate limiting on the REST API, a small number of concurrent attacker requests can saturate the database connection pool, causing denial of service for all other users.

**Root cause and failed assumption:** The design assumes the retry path (`mayMissLongTermScheduledTransaction`) and the inner-transactions path are rare edge cases. In practice, both are fully attacker-controllable via public request parameters and publicly known transaction IDs.

**Exploit flow:**

1. **Query 1 always fires** for any valid transaction ID.
2. **Query 2 fires** when `mayMissLongTermScheduledTransaction(false, true, [])` returns `true` — which it does unconditionally when `scheduled=true` is passed and the first query returns zero rows (line 991: `else if (scheduled && transactions.length === 0) { return true; }`). The attacker passes `?scheduled=true` with a transaction ID whose scheduled counterpart falls outside the initial narrow timestamp window but inside `maxScheduledTransactionConsensusTimestampRangeNs` (up to ~62 days). This is trivially achievable with any long-term scheduled transaction, all of which are publicly visible on-chain.
3. **Query 3 fires** when the rows returned by Query 2 contain `inner_transactions` (batch transactions). Batch transaction IDs are publicly visible. The check at line 949 (`!isTransactionHash && innerTransactions.length > 0`) is fully satisfied by a transaction-ID-format request for a known batch transaction.

**Why existing checks are insufficient:**

- `acceptedSingleTransactionParameters` (line 1014) permits `SCHEDULED` — the attacker's key parameter — by design.
- `mayMissLongTermScheduledTransaction` has no guard against attacker-controlled `scheduled=true` with empty results; it unconditionally returns `true` (line 991).
- The inner-transactions branch (line 949) has no guard against repeated exploitation.
- The REST API has **no per-IP rate limiting** (the throttle configuration found in the codebase applies only to the `web3` module). The Traefik middleware rate limiting in `charts/hedera-mirror-rosetta/values.yaml` applies only to the Rosetta service. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The default DB connection pool for the REST API is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`, `statementTimeout = 20000ms`). [4](#0-3) [5](#0-4) 

Each attacker request issues up to 3 sequential DB queries. With the 20-second statement timeout, a single slow query holds a connection for up to 20 seconds. With only 10 pool connections, as few as **4 concurrent attacker requests** (each holding a connection across 3 sequential queries) can exhaust the pool, causing `connectionTimeoutMillis` (default 20s) errors for all legitimate users. This is a non-network-based DoS requiring no special privileges, no authentication, and no exploit of memory-safety issues — only knowledge of a public batch transaction ID.

### Likelihood Explanation

- **No authentication required.** The endpoint is fully public.
- **Transaction IDs are public.** Any batch (atomic) transaction ID is visible on-chain and in block explorers.
- **Trivially repeatable.** A single attacker script sending `GET /api/v1/transactions/0.0.X-NNNN-NNNN?scheduled=true` in a loop with ~4 concurrent connections is sufficient.
- **Low cost.** No tokens, no accounts, no gas — just HTTP requests.
- **Stable.** The amplification is deterministic once a suitable transaction ID is known; it does not depend on race conditions or timing.

### Recommendation

1. **Limit the retry to non-attacker-controlled conditions.** Before issuing Query 2, verify that the first query returned at least one non-scheduled result (i.e., a SCHEDULECREATE), rather than triggering the retry solely because `scheduled=true` was passed and results were empty. Specifically, change the `scheduled && transactions.length === 0` branch (line 991) to only retry when the caller did not explicitly pass `scheduled=true` as a filter, or require a prior SCHEDULECREATE result to exist.

2. **Gate the inner-transactions query.** Add a configurable cap on `innerTransactions.length` before issuing Query 3, and consider caching results for recently queried batch transaction IDs.

3. **Add per-IP rate limiting to the REST API.** Apply the same Traefik `inFlightReq` + `rateLimit` middleware used for Rosetta to the REST API ingress.

4. **Increase the default connection pool size** or add a per-request DB query count limit enforced at the middleware layer.

### Proof of Concept

**Precondition:** Identify a known atomic/batch transaction ID (e.g., `0.0.1234-1609459200-000000000`) whose scheduled execution falls more than 35 minutes after its valid start (i.e., a long-term scheduled transaction). These are publicly visible on Hedera mainnet/testnet explorers.

**Steps:**

```bash
# Step 1: Confirm 3-query path is reachable
# Query 1 fires (initial narrow-range lookup, returns empty for scheduled=true)
# Query 2 fires (mayMissLongTermScheduledTransaction returns true: scheduled=true && rows=[])
# Query 3 fires (retry returns batch tx rows with inner_transactions)
curl "https://<mirror-node>/api/v1/transactions/0.0.1234-1609459200-000000000?scheduled=true"

# Step 2: Flood with concurrent requests to exhaust the 10-connection pool
for i in $(seq 1 20); do
  curl -s "https://<mirror-node>/api/v1/transactions/0.0.1234-1609459200-000000000?scheduled=true" &
done
wait

# Step 3: Observe legitimate requests timing out with 503/504 or connection timeout errors
curl "https://<mirror-node>/api/v1/transactions"
# Expected: connection timeout or pool exhaustion error
```

**Expected result:** Legitimate API requests begin failing with connection timeout errors after the pool (default: 10 connections) is saturated by the amplified query load from the attacker's concurrent requests.
