### Title
Hardcoded `sslmode=disable` in Rosetta DB Connection Enables MITM Injection of Forged `pending_reward` Responses

### Summary
`ConnectToDb()` in `rosetta/app/db/db.go` calls `dbConfig.GetDsn()`, which unconditionally hardcodes `sslmode=disable` in the PostgreSQL DSN with no override mechanism and no TLS fields in the `Db` config struct. All wire traffic between the Rosetta service and PostgreSQL is transmitted in plaintext. An attacker with network-level access (no database credentials required) can intercept and forge PostgreSQL wire protocol responses, causing the Rosetta API to report `pending_reward = 0` for a victim account while the actual reward remains unclaimed in `entity_stake`.

### Finding Description
**Exact code path:**

`rosetta/app/config/types.go`, lines 39–57: The `Db` struct has no TLS/SSL fields whatsoever. `GetDsn()` unconditionally emits `sslmode=disable`:
```go
func (db Db) GetDsn() string {
    return fmt.Sprintf(
        "host=%s port=%d user=%s dbname=%s password=%s sslmode=disable",
        ...
    )
}
```

`rosetta/app/db/db.go`, line 18: `ConnectToDb()` passes this DSN directly to GORM:
```go
db, err := gorm.Open(postgres.Open(dbConfig.GetDsn()), ...)
```

**Root cause:** `sslmode=disable` is a compile-time constant with no configuration path to override it. The `Db` struct (lines 39–47) contains `Host`, `Name`, `Password`, `Pool`, `Port`, `StatementTimeout`, `Username` — no `TLSMode`, `SSLCert`, or equivalent. By contrast, the REST service (`rest/dbpool.js`) has a `config.db.tls.enabled` branch that conditionally attaches TLS certificates. Rosetta has no equivalent.

**Exploit flow:**
1. Attacker gains network-level adjacency (same Kubernetes namespace, same cloud VPC subnet, or compromised sidecar) — no DB credentials needed.
2. Attacker performs ARP spoofing or equivalent L2/L3 interception to position themselves between the Rosetta pod and the PostgreSQL pod on port 5432.
3. Attacker observes the unencrypted PostgreSQL wire protocol (protocol v3, plaintext). When Rosetta issues a query joining `entity` and `entity_stake` to compute `pending_reward` (the CASE expression in `rest/accounts.js` lines 114–118 mirrors the same logic used by Rosetta's persistence layer), the attacker intercepts the server's DataRow message.
4. Attacker replaces the `pending_reward` column value in the DataRow with `0` and recomputes the wire-protocol message length fields. No authentication or DB credentials are required to forge a response — only the ability to intercept and modify TCP segments.
5. Rosetta returns `pending_reward: 0` to the API caller for the victim account.

**Why existing checks fail:** There are no checks. `sslmode=disable` explicitly instructs the `lib/pq` / `pgx` driver to skip the SSL handshake entirely and never send a `SSLRequest` message. PostgreSQL will not upgrade the connection. There is no certificate pinning, no HMAC on query results, and no application-layer integrity check on the returned `pending_reward` value.

### Impact Explanation
An attacker can make the Rosetta API silently misreport staking reward balances for any account. A victim querying their account balance via the Rosetta API will see `pending_reward = 0` even though `entity_stake.pending_reward` holds a non-zero value. This corrupts financial data surfaced by the Rosetta API without any on-chain evidence of tampering, since the actual `staking_reward_transfer` table is untouched. The attack is silent, leaves no application log, and is indistinguishable from a legitimate zero-reward state.

### Likelihood Explanation
The precondition is network adjacency, not internet reachability. In Kubernetes deployments (the standard deployment model for this project, as evidenced by `charts/hedera-mirror/values.yaml`), a compromised pod in the same namespace or network policy zone satisfies this precondition without any database credentials. The `citus.node_conninfo: "sslmode=disable"` setting in `charts/hedera-mirror/values.yaml` line 337 further confirms that intra-cluster PostgreSQL traffic is expected to be unencrypted, making the attack surface persistent and not incidental. ARP spoofing within a Kubernetes overlay network (e.g., Flannel, Calico in certain modes) is a known technique requiring no elevated OS privileges on the attacker's pod.

### Recommendation
1. Add a `Sslmode string` (or `TlsMode`) field to the `Db` struct in `rosetta/app/config/types.go` with a default of `"require"` or `"verify-full"`.
2. Update `GetDsn()` to interpolate this field instead of the hardcoded literal `sslmode=disable`.
3. Add `SslCert`, `SslKey`, `SslRootCert` fields to support mutual TLS, mirroring the REST service's `config.db.tls` block.
4. Update default configuration files and Helm chart values to set `sslmode=require` at minimum for production deployments.

### Proof of Concept
```
# Precondition: attacker has a pod in the same Kubernetes namespace as rosetta

# 1. Identify rosetta pod and postgres service IPs
kubectl get pods -o wide
kubectl get svc

# 2. ARP-spoof the postgres service IP from the rosetta pod's perspective
# (using arpspoof or similar, no root required in many CNI configurations)
arpspoof -i eth0 -t <rosetta-pod-ip> <postgres-service-ip>

# 3. Run a TCP proxy that parses PostgreSQL wire protocol v3
# When a DataRow message is received containing a pending_reward column,
# replace the 8-byte big-endian int64 value with 0x0000000000000000
# Forward all other messages unmodified.

# 4. Query the Rosetta API for a victim account known to have staking rewards:
curl http://<rosetta-service>/account/balance \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "account_identifier":{"address":"0.0.VICTIM"}}'

# Result: pending_reward reported as 0 despite non-zero value in entity_stake table.
# The staking_reward_transfer table is unmodified; the actual reward remains unclaimed.
```
