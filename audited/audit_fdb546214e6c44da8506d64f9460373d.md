### Title
Connection Pool Exhaustion via Unauthenticated Requests Using DB-Triggering ID Formats During Network Partition

### Summary
`getAccountCryptoAllowances()` in `rest/controllers/cryptoAllowanceController.js` calls `EntityService.getEncodedId()`, which issues real database queries only for EVM address and alias formats, while numeric IDs return immediately without touching the database. With no rate limiting on the REST API and a default pool of only 10 connections, an unauthenticated attacker can deliberately use EVM address or alias formats to hold all pool connections open during a network partition, exhausting the shared pool and causing all endpoints to fail for the duration of the partition.

### Finding Description

**Code path:**

`cryptoAllowanceController.js` line 77:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
```

`entityService.js` lines 118–137 — three distinct branches:
- **Numeric/shard.realm.num** (`EntityId.isValidEntityId` true AND `evmAddress === null`): returns `entityId.getEncodedId()` — **pure computation, zero DB I/O**
- **EVM address** (`evmAddress !== null`): calls `getEntityIdFromEvmAddress()` → `this.getRows(EntityService.entityFromEvmAddressQuery, ...)` — **DB query issued**
- **Alias** (`AccountAlias.isValid`): calls `getAccountIdFromAlias()` → `getAccountFromAlias()` → `super.getRows(EntityService.entityFromAliasQuery, ...)` — **DB query issued**

`baseService.js` line 56:
```js
async getRows(query, params) {
  return (await this.pool().queryQuietly(query, params)).rows;
}
```

`dbpool.js` lines 7–16 — the pool is configured with:
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20000ms
max: config.db.pool.maxConnections,                         // default 10
statement_timeout: config.db.pool.statementTimeout,         // default 20000ms
```

**Root cause:** `statement_timeout` is a PostgreSQL server-side setting. During a network partition, the TCP connection is already established and the query bytes are sent, but the server's response (including any timeout error) never returns to the Node.js client. The `pg` pool has no application-level `query_timeout` or socket-level idle timeout configured. Connections hang until the OS TCP keepalive fires (Linux default: ~2 hours). `connectionTimeoutMillis` only governs how long a caller waits to *acquire* a free connection from the pool — once all 10 connections are occupied by hanging queries, new requests queue and then fail after 20 seconds, but the 10 connections remain stuck for the duration of the partition.

**No rate limiting exists** on the Node.js REST API — `grep` for `rateLimit|rateLimiter|throttle` in `rest/**/*.js` returns zero matches.

### Impact Explanation

With 10 concurrent requests using EVM address or alias formats (e.g., `GET /api/v1/accounts/0xebb9a1be370150759408cd7af48e9eda2b8ead57/allowances/crypto`), all 10 pool connections are occupied by hanging DB queries. Every subsequent request to **any endpoint** sharing the same pool (all REST endpoints) fails with a connection acquisition timeout after 20 seconds. The service is effectively down for the duration of the network partition. The asymmetry between ID formats (numeric = no DB, EVM/alias = DB) means an attacker can deliberately maximize pool exhaustion by choosing the right format.

### Likelihood Explanation

Preconditions: (1) a network partition between the REST service and its database must already exist — the attacker does not cause this but exploits it; (2) the attacker needs no credentials, no special knowledge beyond the public API spec (the supported URL formats are documented in the OpenAPI spec and test fixtures). The attack is trivially repeatable with 10 concurrent HTTP requests. The pool size of 10 is the default and is small enough that a single attacker with a basic HTTP client can exhaust it.

### Recommendation

1. Add an application-level query timeout in the `pg` pool config (`query_timeout` or socket `idleTimeoutMillis`) so Node.js-side hangs are bounded independently of PostgreSQL's `statement_timeout`.
2. Add a per-IP or global rate limiter (e.g., `express-rate-limit`) to the REST API middleware stack.
3. Consider increasing `maxConnections` or separating the entity-lookup pool from the data-query pool so a partition affecting entity lookups does not starve all other endpoints.
4. Add a circuit breaker around `EntityService.getEncodedId()` DB calls so repeated failures open the circuit and return errors immediately rather than queuing against a saturated pool.

### Proof of Concept

```bash
# During a network partition between REST service and DB:
# Send 10 concurrent requests using EVM address format (triggers DB query)
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/accounts/0xebb9a1be370150759408cd7af48e9eda2b8ead57/allowances/crypto" &
done
wait

# Now all 10 pool connections are held by hanging queries.
# Any subsequent request to any endpoint will queue and fail after connectionTimeoutMillis (20s):
curl -v "http://<mirror-node>/api/v1/accounts/0.0.1001/allowances/crypto"
# Expected: hangs ~20 seconds, then connection timeout error

# Contrast: numeric format does NOT trigger a DB query and would succeed instantly
# if a connection were available, confirming the format-dependent code path asymmetry.
```
