### Title
Unbounded Concurrent Flux Subscriptions via Unauthenticated gRPC Exhaust Bounded Elastic Thread Pool, Starving Fee-Schedule Topic Delivery

### Summary
`CompositeTopicListener.listen()` imposes no rate limit, no global subscription cap, and no authentication before creating a new Reactor `Flux` pipeline per call. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — is scoped to a single TCP connection, not to the server globally. An unprivileged attacker who opens many parallel connections can therefore accumulate an arbitrarily large number of live subscriptions, each of which continuously schedules work on the JVM-wide `Schedulers.boundedElastic()` pool, exhausting it and causing fee-schedule topic messages to queue indefinitely or be dropped.

### Finding Description

**Entry point — no guard before `listen()`**

`ConsensusController.subscribeTopic()` calls `topicMessageService::subscribeTopic` with no authentication, no IP-based throttle, and no check against any subscription ceiling. [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` increments `subscriberCount` only as a Micrometer `Gauge` — it is never compared against a limit and never rejects a new subscriber. [2](#0-1) [3](#0-2) 

`CompositeTopicListener.listen()` performs no rate-limiting or subscription-count check before delegating to the underlying listener. [4](#0-3) 

**Thread-pool consumption — POLL mode**

`PollingTopicListener` holds a reference to the global `Schedulers.boundedElastic()` instance. Every call to `listen()` creates an independent polling loop that schedules recurring work on that shared pool via `delaySubscription` and `repeatWhen(...withScheduler(scheduler))`. [5](#0-4) 

**Thread-pool consumption — REDIS / SHARED_POLL mode**

`SharedTopicListener.listen()` calls `publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch())`. Each active subscriber holds a worker slot in the bounded elastic pool for the lifetime of the streaming call. [6](#0-5) 

`TopicMessageServiceImpl` also schedules a per-subscription safety-check task on `Schedulers.boundedElastic()`. [7](#0-6) 

**Why the only existing guard is insufficient**

The Netty server is configured with `maxConcurrentCallsPerConnection = 5` (default). This is a *per-connection* ceiling; there is no `maxConnections` or global call limit configured. [8](#0-7) [9](#0-8) 

An attacker who opens *N* TCP connections can therefore hold *5N* concurrent streaming subscriptions simultaneously, with no server-side rejection.

### Impact Explanation

`Schedulers.boundedElastic()` is a JVM-wide shared pool capped at `10 × availableProcessors` threads (typically 40–160 on a production host) with a finite task queue. Once the pool is saturated, new scheduling requests for legitimate subscribers — including those monitoring the Hedera fee-schedule topic (0.0.111) — are queued indefinitely or rejected with `RejectedExecutionException`. Fee-schedule update messages therefore fail to be delivered to SDK clients and other consumers within design-time latency bounds, constituting a denial-of-service against the fee-update propagation path. The `maxBufferSize = 16384` overflow guard in `SharedTopicListener` will then trigger `BufferOverflowStrategy.ERROR`, terminating legitimate subscriber streams. [10](#0-9) 

### Likelihood Explanation

The attack requires only a standard gRPC client library and network access to port 5600. No credentials, tokens, or privileged network position are needed. The attacker opens connections in a loop (e.g., 200 connections × 5 calls = 1,000 subscriptions) and keeps them alive by never cancelling. This is trivially scriptable, repeatable, and can be sustained indefinitely from a single host. The default deployment exposes port 5600 publicly (Kubernetes `service.port: 5600`). [11](#0-10) 

### Recommendation

1. **Global connection limit**: Add `serverBuilder.maxConnectionAge(...)` and a global connection cap in `GrpcConfiguration` to bound the total number of live TCP connections.
2. **Global subscription cap**: In `TopicMessageServiceImpl.subscribeTopic()`, compare `subscriberCount` against a configurable maximum and return `RESOURCE_EXHAUSTED` when exceeded.
3. **Per-IP rate limiting**: Intercept new calls with a gRPC `ServerInterceptor` that enforces a per-source-IP subscription rate and concurrent-call ceiling.
4. **Dedicated scheduler with explicit bound**: Replace the implicit `Schedulers.boundedElastic()` references in `PollingTopicListener` and `SharedTopicListener` with an explicitly sized, named scheduler so its capacity is visible and tunable independently of other JVM subsystems.

### Proof of Concept

```python
import grpc, threading
from hedera import mirror_api_pb2, mirror_api_pb2_grpc

TARGET = "mirror-node-host:5600"
TOPIC_ID = "0.0.111"          # fee-schedule topic
CONNECTIONS = 200
CALLS_PER_CONN = 5            # matches maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = mirror_api_pb2_grpc.ConsensusServiceStub(channel)
    threads = []
    for _ in range(CALLS_PER_CONN):
        req = mirror_api_pb2.ConsensusTopicQuery()
        req.topicID.topicNum = 111   # 0.0.111
        req.consensusStartTime.seconds = 0
        t = threading.Thread(
            target=lambda: list(stub.subscribeTopic(req)),
            daemon=True)
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

workers = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for w in workers: w.start()
# 200 connections × 5 calls = 1,000 concurrent subscriptions
# Schedulers.boundedElastic() saturates; legitimate fee-schedule
# subscribers receive no messages or RejectedExecutionException.
```

    private final Scheduler scheduler = Schedulers.boundedElastic();

    @Override
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
