### Title
Unbounded Exponential Backoff in `buildNetworkFromMirrorNodes()` Enables DoS via Mirror REST Rate-Limit Exhaustion

### Summary
The `buildNetworkFromMirrorNodes()` function in `pinger/mirror_node_client.go` applies exponential backoff with no upper cap when the mirror REST endpoint returns HTTP 429 or any 5xx status. Both response codes unconditionally set `retry = true`, and the backoff formula `cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))` grows without bound. With default settings (10 retries, 500 ms base), total blocked time reaches ~511 seconds before the pinger fatally exits, far exceeding any reasonable block-time threshold.

### Finding Description
**Code path:**

- `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, line 91:
  ```go
  retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
  ```
  Both 429 and every 5xx unconditionally set `retry = true`.

- `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()`, line 63:
  ```go
  backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
  ```
  No `min(…, cap)` guard. Compare `pinger/transfer.go` line 63–64 which caps at 30 s.

- `pinger/sdk_client.go`, line 18:
  ```go
  netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
  ```
  `context.Background()` carries no deadline, so the `ctx.Done()` escape at line 65 of `mirror_node_client.go` never fires.

- `pinger/main.go`, line 41–43: `newClient()` failure calls `log.Fatalf`, killing the process.

**Root cause / failed assumption:** The code assumes the mirror REST endpoint is either healthy or transiently unavailable for a short window. It does not account for a sustained adversarially-induced error state where every attempt returns a retryable code, and it omits the backoff cap that the transfer retry path correctly applies.

**Exploit flow:**

1. Attacker continuously floods the public mirror REST endpoint (`/api/v1/network/nodes`) with cheap unauthenticated GET requests, exhausting the endpoint's global rate limit.
2. The mirror REST service returns HTTP 429 to all clients, including the pinger.
3. On each pinger attempt, `fetchMirrorNodeNetwork()` returns `retry=true`.
4. `buildNetworkFromMirrorNodes()` sleeps: 500 ms → 1 s → 2 s → 4 s → 8 s → 16 s → 32 s → 64 s → 128 s → 256 s (total ≈ 511.5 s) before returning a fatal error.
5. `log.Fatalf` terminates the pinger. Under Kubernetes, the pod restarts and the cycle repeats indefinitely as long as the flood continues.

**Why existing checks are insufficient:**

- The `ctx.Done()` branch (line 65) uses `context.Background()`, which never cancels.
- There is no per-call or aggregate timeout on `buildNetworkFromMirrorNodes()`.
- There is no backoff cap (unlike `transfer.go`).
- There is no circuit-breaker or jitter to spread load.

### Impact Explanation
The pinger is the liveness/transaction-submission component. While it cannot start (or after each crash-loop restart), no transactions are submitted. With default settings the startup delay is ~511 seconds per attempt cycle. Hedera consensus rounds are on the order of 1–3 seconds; a 511-second outage represents hundreds of missed submission windows, well beyond the 500 % block-time threshold cited in the finding scope. A sustained flood keeps the pinger in a permanent crash-loop, constituting a full denial of the transaction-submission service.

### Likelihood Explanation
The mirror REST endpoint (`/api/v1/network/nodes`) is a public, unauthenticated HTTP endpoint. No credentials, tokens, or network-level access are required. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `hey`, or a small botnet) can saturate a shared rate-limit bucket. The attack is repeatable, cheap, and requires no knowledge of the pinger's internal state. The pinger's crash-loop amplifies the impact because each restart re-enters the full 511-second backoff sequence.

### Recommendation
1. **Cap the backoff** in `buildNetworkFromMirrorNodes()` identically to `transfer.go`:
   ```go
   backoff := min(cfg.mirrorNodeClientBaseBackoff*time.Duration(1<<(attempt-1)), 30*time.Second)
   ```
2. **Pass a deadline context** from `newClient()` instead of `context.Background()`, so the entire retry sequence is bounded by an operator-configurable timeout.
3. **Differentiate 429 from 5xx**: honour the `Retry-After` header for 429 responses rather than using unbounded exponential backoff.
4. **Add jitter** to the backoff to avoid thundering-herd on restart.
