### Title
Unauthenticated 3x DB Scan Amplification via Crafted Token Allowance Range Query

### Summary
The `getQuery()` function in `rest/service/tokenAllowanceService.js` unconditionally generates up to three independent `SELECT * FROM token_allowance` subqueries joined by `UNION ALL` when a caller supplies a closed `spender.id` range (`gte` + `lte`) combined with a `token.id` bound. Any unauthenticated HTTP client can trigger this code path, causing the database to execute three separate index scans per request. Because the REST API has no per-endpoint rate limiting, an attacker can sustain a flood of such requests to amplify database read load by 3x and degrade response times for all concurrent users.

### Finding Description
**Exact code path:**

`rest/service/tokenAllowanceService.js`, `getQuery()`, lines 63–74:
```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));
// ...
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
```

Each non-empty bucket (`lower`, `inner`, `upper`) becomes a full `select * from token_allowance where owner = $1 and amount > 0 and <conditions> order by spender, token_id limit $2` subquery. When all three are populated the final SQL is:

```sql
(select * from token_allowance where owner=$1 and amount>0 and spender=$3 and token_id>=$4 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender>$5 and spender<$6 order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance where owner=$1 and amount>0 and spender=$7 and token_id<=$8 order by spender asc, token_id asc limit $2)
order by spender asc, token_id asc limit $2
```

**Root cause / failed assumption:** The design assumes the `limit` cap adequately bounds per-request DB cost. It bounds *rows returned per subquery* but not the *number of independent index scans*. Three scans are always issued regardless of how sparse the data is.

**Why validation is insufficient:**

`validateBounds()` in `rest/controllers/baseController.js` (lines 56–61) only checks structural correctness (no mixed range+equal, secondary requires primary, operator constraints). It explicitly permits and is designed to produce the three-subquery case. `Bound.parse()` enforces at most one `eq`, one `gt/gte`, and one `lt/lte` per key — exactly the combination that triggers all three buckets. No check prevents a caller from supplying the full closed-range + secondary-bound combination.

### Impact Explanation
Every request matching the trigger pattern causes the PostgreSQL query planner to execute three independent index scans on `token_allowance` plus a final merge-sort, compared to one scan for a simple query. With no REST-layer rate limiting (the `ThrottleManagerImpl` / bucket4j throttle found in the codebase applies only to the `web3` module, not the REST API), an attacker can sustain a high request rate. At N concurrent attack requests the database sees 3N scans instead of N, degrading latency for all legitimate users sharing the same DB connection pool. Severity is medium/griefing: no data is leaked or corrupted, but service quality degrades proportionally to attack volume.

### Likelihood Explanation
The endpoint `/accounts/:idOrAliasOrEvmAddress/allowances/tokens` is public and requires no credentials. The trigger parameters (`spender.id=gte:X&spender.id=lte:Y&token.id=gte:A&token.id=lte:B`) are standard, documented query parameters. Any script-kiddie with `curl` or a load-testing tool can reproduce this. The attack is repeatable, stateless, and requires no prior knowledge of the system beyond the public API spec.

### Recommendation
1. **Add per-endpoint rate limiting** to the REST API (e.g., express-rate-limit or a reverse-proxy rule) so the three-subquery path cannot be flooded.
2. **Count subqueries as a cost multiplier** in any future request-cost accounting: treat a 3-subquery request as 3 units, not 1.
3. **Consider query-timeout enforcement** at the DB layer so a slow union-all cannot hold a connection indefinitely.
4. **Document the amplification** so operators can tune connection-pool sizes and DB resource limits accordingly.

### Proof of Concept
```bash
# Triggers all three subqueries (lower + inner + upper) in one request
curl "https://<mirror-node>/api/v1/accounts/0.0.1234/allowances/tokens?\
spender.id=gte:0.0.1&spender.id=lte:0.0.9999999&\
token.id=gte:0.0.1&token.id=lte:0.0.9999999&\
limit=100"

# Flood with concurrent requests to amplify DB load 3x
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/allowances/tokens?\
spender.id=gte:0.0.1&spender.id=lte:0.0.9999999&\
token.id=gte:0.0.1&token.id=lte:0.0.9999999&limit=100" &
done
wait
```

Observe in DB logs: each HTTP request produces three `select * from token_allowance` statements instead of one, confirming the 3x scan amplification.
