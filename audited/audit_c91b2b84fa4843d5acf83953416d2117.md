### Title
Unauthenticated Alias-Driven DB Connection Pool Exhaustion via Uncached `entityFromAliasQuery` in `getEncodedId()`

### Summary
`getEncodedId()` in `rest/service/entityService.js` unconditionally issues a live database query for every alias-format input with no per-alias result caching and no rate limiting on the REST API layer. An unprivileged attacker can flood any public endpoint that calls `getEncodedId()` with syntactically valid alias strings, exhausting the REST API's DB connection pool (default: 10 connections) and generating sustained DB-level CPU/IO contention that degrades both the REST API and the shared PostgreSQL instance used by the importer.

### Finding Description

**Exact code path:**

`getEncodedId()` at [1](#0-0)  branches on two conditions. When the input passes `AccountAlias.isValid()` (line 125), it calls `getAccountIdFromAlias()` → `getAccountFromAlias()` → `super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias])`.

The query issued is: [2](#0-1) 

`getRows()` in `BaseService` calls `this.pool().queryQuietly(query, params)` directly: [3](#0-2) 

**Root cause — no caching for alias lookups:**

The `quickLru` cache in `entityId.js` only caches parsed `EntityId` objects for `shard.realm.num` / encoded-id / EVM-address strings: [4](#0-3) 

It is never consulted for the alias branch. Every alias-format call to `getEncodedId()` unconditionally hits the database.

**Root cause — no rate limiting:**

A grep across all `rest/**/*.js` files for `rateLimit`, `throttle`, or `rateLimiter` returns zero hits in production middleware. The middleware stack contains: [5](#0-4)  — no rate-limiting component is registered.

**Connection pool size:**

The default pool is capped at 10 connections with a 20-second statement timeout: [6](#0-5) 

Pool initialization: [7](#0-6) 

**Alias validation is format-only:**

`AccountAlias.isValid()` only tests a regex — it does not throttle, deduplicate, or cache: [8](#0-7) 

Any string matching `^(\d{1,5}\.){0,2}[A-Z2-7]+$` passes and proceeds to a live DB query.

**Exposed endpoints:**

Multiple public routes call `EntityService.getEncodedId()` directly from HTTP path parameters, e.g.: [9](#0-8)  and [10](#0-9) 

### Impact Explanation

With 10 concurrent connections and a 20-second statement timeout, an attacker holding all 10 pool slots with alias queries blocks every other REST API request that needs a DB connection. Because the REST API and the importer share the same PostgreSQL instance, sustained high-concurrency alias queries generate CPU and I/O contention at the DB server level. This degrades importer write throughput, increasing record-file processing latency. Under sufficient load this can push block ingestion latency well beyond normal bounds. The REST API itself becomes unavailable to legitimate users for the duration of the attack.

### Likelihood Explanation

The attack requires zero authentication, zero special knowledge, and only a valid base32 alias string (e.g., `AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ`). It is trivially scriptable with any HTTP load tool (`wrk`, `ab`, `hey`). The attacker does not need the alias to match any real entity — a miss still executes the full table scan against the `entity` table. The attack is repeatable and stateless, requiring no session or token.

### Recommendation

1. **Cache alias-to-entity-id results** in the existing `quickLru` / Redis cache layer (keyed by the decoded alias bytes), consistent with the `entityId.maxAge`/`entityId.maxSize` configuration already present.
2. **Add rate limiting middleware** to the REST API (e.g., `express-rate-limit` or a Traefik `rateLimit` middleware) scoped per source IP, targeting endpoints that accept `idOrAliasOrEvmAddress` path parameters.
3. **Increase connection pool size** or add a request-queue depth limit so pool exhaustion causes fast 429/503 responses rather than indefinite blocking.
4. **Add a DB-level query timeout** shorter than the current 20-second `statementTimeout` for alias lookups specifically.

### Proof of Concept

```bash
# Send 50 concurrent requests with a valid alias string that will miss in the DB
# (forces full entityFromAliasQuery execution each time)
hey -n 5000 -c 50 \
  "https://<mirror-node-rest>/api/v1/accounts/AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ/nfts"
```

**Preconditions:** Public REST API reachable; no rate limiting configured (default).

**Trigger:** 50 concurrent requests each pass `AccountAlias.isValid()`, each call `getAccountFromAlias()`, each acquire a DB connection and execute `entityFromAliasQuery`.

**Result:** All 10 pool connections are held; subsequent REST API requests queue until `connectionTimeoutMillis` (20 s) expires and return 500; DB CPU/IO contention degrades importer write throughput for the duration of the flood.
