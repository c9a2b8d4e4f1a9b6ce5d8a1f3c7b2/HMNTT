### Title
Unauthenticated Resource Exhaustion via Unbounded Repeated Airdrop Queries Against High-Activity Accounts

### Summary
The `/api/v1/accounts/{id}/airdrops/outstanding` and `/api/v1/accounts/{id}/airdrops/pending` endpoints are publicly accessible with no authentication or rate limiting. An unprivileged attacker can repeatedly query a known high-activity airdrop sender with `limit=MAX_LIMIT`, forcing the mirror node to execute expensive DB scans, serialize large result sets, and transmit large payloads on every call, causing disproportionate resource consumption.

### Finding Description
**Code path:**
- Controller: `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TokenAirdropsController.java`, lines 66–75 and 77–86
- Service: `rest-java/src/main/java/org/hiero/mirror/restjava/service/TokenAirdropServiceImpl.java`, lines 19–22
- Repository: `rest-java/src/main/java/org/hiero/mirror/restjava/repository/TokenAirdropRepositoryCustomImpl.java`, lines 58–72

**Root cause:**

In `TokenAirdropsController`, the `limit` parameter is validated only with `@Positive @Max(MAX_LIMIT)` — it is bounded per-request but there is no rate limiting, no authentication, and no per-IP or per-account throttle anywhere in this code path. [1](#0-0) 

In `TokenAirdropRepositoryCustomImpl.findAll()`, the query is:

```sql
SELECT * FROM token_airdrop
WHERE sender_account_id = :accountId AND state = 'PENDING'
ORDER BY receiver_account_id, token_id, serial_number
LIMIT :limit
``` [2](#0-1) 

When `limit=MAX_LIMIT` and the target account has many pending airdrops, the DB must scan and return the maximum number of rows, the application must serialize all of them into `TokenAirdrop` objects, and the full payload is transmitted over the network. There is no caching, no result deduplication guard, and no request coalescing. [3](#0-2) 

**Why existing checks fail:**

`@Max(MAX_LIMIT)` only caps the per-request result size; it does not prevent the same maximum-cost query from being issued thousands of times per second by the same or distributed clients. The `@Positive` constraint only prevents zero/negative values. [4](#0-3) 

### Impact Explanation
Each request with `limit=MAX_LIMIT` against a high-activity account forces: (1) a full index scan up to MAX_LIMIT rows in PostgreSQL, (2) JOOQ result-set materialization and Java object deserialization for every row, (3) JSON serialization of the full response, and (4) network transmission of a large payload. Sustained at even moderate request rates (tens of requests/second), this can push DB CPU, JVM CPU, and outbound bandwidth well above normal baselines — meeting the ≥30% resource increase threshold without requiring millions of requests. [5](#0-4) 

### Likelihood Explanation
The target account ID is publicly visible on-chain. No credentials, tokens, or special access are needed — any HTTP client can call the endpoint. The attacker only needs to know one high-activity sender account ID (trivially discoverable from a block explorer) and issue repeated GET requests. This is fully automatable with a single script and requires no exploit infrastructure. [6](#0-5) 

### Recommendation
1. **Rate limiting**: Apply per-IP (and optionally per-account-ID) rate limiting at the gateway or Spring filter layer for all `/api/v1/accounts/{id}/airdrops/*` endpoints.
2. **Result caching**: Cache responses for identical `(accountId, limit, order, filters)` tuples for a short TTL (e.g., 5–10 seconds) to collapse repeated identical queries.
3. **Query cost awareness**: Consider lowering `MAX_LIMIT` or requiring pagination tokens (cursor-based) so that large result sets cannot be re-fetched from the beginning on every call.
4. **Observability**: Add per-endpoint request-rate metrics and alerting to detect abnormal query patterns against specific account IDs. [7](#0-6) 

### Proof of Concept
```bash
# 1. Identify a high-activity airdrop sender from a public block explorer (e.g., account 0.0.12345)
# 2. Repeatedly query with maximum limit:

while true; do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.12345/airdrops/outstanding?limit=<MAX_LIMIT>" \
    -o /dev/null -w "%{time_total}\n"
done

# Each iteration forces a full MAX_LIMIT DB scan + serialization + network transfer.
# Monitor mirror node CPU, DB query time, and outbound bandwidth — expect sustained elevation.
``` [2](#0-1)

    public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
        var type = request.getType();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, type.getBaseField())
                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

        var order = SORT_ORDERS.getOrDefault(type, Map.of()).get(request.getOrder());
        return dslContext
                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())
                .fetchInto(TokenAirdrop.class);
    }
