### Title
Silent Overwrite of Lower Bound in `Bound` Constructor Bypasses `verifySingleOccurrence()` Validation in Token Airdrop Endpoints

### Summary
When two identical-direction range operators (e.g., `receiver.id=gt:100&receiver.id=gt:200`) are supplied to `getOutstandingAirdrops()` or `getPendingAirdrops()`, the `Bound` constructor silently overwrites the first lower bound with the second while incrementing cardinality for both. Because `TokenAirdropServiceImpl` never calls `verifySingleOccurrence()` — unlike `NftAllowanceServiceImpl` which does — no validation error is raised, and the query executes with only the second (higher) GT value, returning a narrower result set that omits valid airdrops.

### Finding Description

**Code path:**

`TokenAirdropsController.java` line 71 accepts `@Size(max = 2) EntityIdRangeParameter[] receiverIds`, allowing up to two values. Line 97 passes them directly to the `Bound` constructor:

```java
var entityIdsBound = new Bound(entityIds, true, ACCOUNT_ID, type.getPrimaryField());
```

In `Bound.java` lines 47–53, the constructor iterates over all params:

```java
for (var param : params) {
    if (param.hasLowerBound()) {
        lower = param;          // silently overwrites any previous lower bound
    } else if (param.hasUpperBound()) {
        upper = param;
    }
    cardinality.merge(param.operator(), 1, Math::addExact);  // increments for both
}
```

`RangeParameter.hasLowerBound()` returns `true` for `GT`, `GTE`, and `EQ`. So with `receiver.id=gt:100&receiver.id=gt:200`:
- Iteration 1: `lower = gt:100`, `cardinality[GT] = 1`
- Iteration 2: `lower = gt:200` (overwrites), `cardinality[GT] = 2`

The `primarySortField` guard at line 58 only checks `adjustedLower > adjustedUpper`. With no upper bound, `adjustedUpper = Long.MAX_VALUE`, so `201 > Long.MAX_VALUE` is false — no exception.

`TokenAirdropServiceImpl.getAirdrops()` (lines 19–22) then calls `repository.findAll(request, id)` with zero validation:

```java
public Collection<TokenAirdrop> getAirdrops(TokenAirdropRequest request) {
    var id = entityService.lookup(request.getAccountId());
    return repository.findAll(request, id);
}
```

`verifySingleOccurrence()` (defined in `Bound.java` lines 155–159) is **never called** in this path. By contrast, `NftAllowanceServiceImpl.verifyRangeId()` (line 58) explicitly calls it. The TokenAirdrop service has no equivalent guard.

**Result:** The query executes with `receiver_account_id > 200` instead of `receiver_account_id > 100`, silently dropping all airdrops with receiver IDs in the range (100, 200].

### Impact Explanation
Any caller can supply two same-direction operators and receive a silently truncated result set with no error signal. For the outstanding airdrops endpoint, a sender querying their own airdrops will miss entries in the dropped range. For the pending airdrops endpoint, a receiver will miss claimable airdrops in the dropped range. Because no error is returned, the caller has no indication the result is incomplete. This is a data-integrity / result-correctness issue on a read-only endpoint; it does not directly move funds, but it causes silent omission of airdrop records that the caller is entitled to see, which can lead to missed claim or cancel actions.

### Likelihood Explanation
No privileges are required. Any unauthenticated HTTP client can craft the request. The `@Size(max = 2)` annotation explicitly permits two values for the parameter, so the input is within the declared contract of the API. The bug is trivially reproducible and repeatable on every request.

### Recommendation
Add the same validation that `NftAllowanceServiceImpl` applies. In `TokenAirdropServiceImpl.getAirdrops()` (or in a shared validation helper), call `verifySingleOccurrence()` on every `Bound` before passing the request to the repository:

```java
public Collection<TokenAirdrop> getAirdrops(TokenAirdropRequest request) {
    request.getEntityIds().verifySingleOccurrence();
    request.getTokenIds().verifySingleOccurrence();
    request.getSerialNumbers().verifySingleOccurrence();
    var id = entityService.lookup(request.getAccountId());
    return repository.findAll(request, id);
}
```

Alternatively, move the validation into the `Bound` constructor itself when `primarySortField = true`, or add a shared `verifyRangeId` utility called from all service implementations.

### Proof of Concept

**Preconditions:** A sender account (e.g., `0.0.1000`) has outstanding airdrops to receivers `0.0.150` and `0.0.250`.

**Trigger:**
```
GET /api/v1/accounts/1000/airdrops/outstanding?receiver.id=gt:100&receiver.id=gt:200
```

**Expected behavior:** HTTP 400 — "Only one range operator from [GT, GTE] is allowed for the given parameter for account.id"

**Actual behavior:** HTTP 200 — result contains only the airdrop to `0.0.250`; the airdrop to `0.0.150` is silently omitted because the `gt:100` bound was overwritten by `gt:200` in the `Bound` constructor with no error raised.

**Verification:** Issue the correct single-operator query:
```
GET /api/v1/accounts/1000/airdrops/outstanding?receiver.id=gt:100
```
This returns both airdrops, confirming the two-GT variant silently narrows the result.
