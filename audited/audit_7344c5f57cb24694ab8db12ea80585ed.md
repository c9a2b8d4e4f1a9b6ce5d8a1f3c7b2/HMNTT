### Title
Connection Pool Exhaustion via Unbounded Concurrent Requests to NFT Allowances Endpoint

### Summary
The `findAll()` method in `NftAllowanceRepositoryCustomImpl` acquires a HikariCP database connection per request and holds it for the full query duration (up to the 10-second statement timeout). The `rest-java` module has no rate limiting on the `/api/v1/accounts/{id}/allowances/nfts` endpoint, allowing any unauthenticated attacker to exhaust the finite connection pool with concurrent requests, causing all subsequent legitimate requests to queue and eventually fail.

### Finding Description
**Code path:**
- `AllowancesController.getNftAllowances()` → `NftAllowanceServiceImpl.getNftAllowances()` → `NftAllowanceRepositoryCustomImpl.findAll()` (lines 37–47)

```java
// NftAllowanceRepositoryCustomImpl.java lines 41-46
return dslContext
        .selectFrom(NFT_ALLOWANCE)
        .where(condition)
        .orderBy(SORT_ORDERS.get(new OrderSpec(byOwner, request.getOrder())))
        .limit(request.getLimit())
        .fetchInto(NftAllowance.class);
```

Each call to `fetchInto()` acquires a JDBC connection from the HikariCP pool and holds it synchronously until the query completes or the statement timeout fires.

**Root cause — no rate limiting in rest-java:**
The `web3` module has a full `ThrottleConfiguration` (bucket4j, `requestsPerSecond`, gas limits). The `rest-java` module has **none**. Searching the entire `restjava` package confirms zero usage of bucket4j, `RateLimiter`, or any per-IP/global request throttle on the allowances endpoint.

**Only existing guards (all insufficient):**
- `@Max(MAX_LIMIT)` on `limit` — bounds result rows (max 100), not request concurrency
- `@Size(max = 2)` on `accountIds`/`tokenIds` — bounds query complexity, not request rate
- `statementTimeout = 10000ms` — limits per-query duration but means each connection is held for up to 10 seconds

**HikariCP pool size:** No explicit `maximumPoolSize` override exists in the rest-java resources. Spring Boot's HikariCP default is 10 connections.

**Exploit flow:**
1. Attacker sends ≥10 concurrent GET requests to `/api/v1/accounts/0.0.1/allowances/nfts`
2. Each request enters `findAll()`, acquires one HikariCP connection, and blocks on `fetchInto()`
3. Pool is exhausted; all subsequent requests block waiting for a connection (HikariCP `connectionTimeout` default: 30 seconds)
4. Legitimate users receive connection timeout errors or extreme latency
5. Attacker sustains the attack by keeping the 10 connections busy (easy with a 10-second statement timeout window per wave)

### Impact Explanation
Full service unavailability for the `rest-java` NFT allowances endpoint and, because the pool is shared across all `rest-java` endpoints via the single `DataSource` bean (`CommonConfiguration`), potentially all other `rest-java` endpoints as well. The impact is a denial-of-service equivalent to a network partition from the perspective of legitimate clients — all new DB-dependent requests fail until the attacker stops.

### Likelihood Explanation
No authentication or API key is required. The endpoint is publicly accessible. An attacker needs only a standard HTTP client capable of sending 10–20 concurrent requests. This is trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The attack is repeatable indefinitely and requires no special knowledge of the system beyond the public API documentation.

### Recommendation
1. Add a global request rate limiter to the `rest-java` module mirroring the `web3` `ThrottleConfiguration` — apply it as a servlet filter or Spring `HandlerInterceptor` before the controller layer.
2. Explicitly configure `spring.datasource.hikari.maximum-pool-size` to a documented, intentional value and set `connection-timeout` to a short value (e.g., 2–3 seconds) so exhausted-pool requests fail fast rather than queuing.
3. Consider per-IP rate limiting at the ingress/load-balancer layer as a defense-in-depth measure.

### Proof of Concept
```bash
# Send 15 concurrent requests to exhaust a default 10-connection pool
for i in $(seq 1 15); do
  curl -s "http://<host>/api/v1/accounts/0.0.1/allowances/nfts" &
done
wait

# Subsequent legitimate request will time out or receive a 500
curl -v "http://<host>/api/v1/accounts/0.0.2/allowances/nfts"

```

    public Collection<NftAllowance> findAll(NftAllowanceRequest request, EntityId accountId) {
        boolean byOwner = request.isOwner();
        var bounds = request.getBounds();
        var condition = getBaseCondition(accountId, byOwner).and(getBoundConditions(bounds));
        return dslContext
                .selectFrom(NFT_ALLOWANCE)
                .where(condition)
                .orderBy(SORT_ORDERS.get(new OrderSpec(byOwner, request.getOrder())))

    NftAllowancesResponse getNftAllowances(
            @PathVariable EntityIdParameter id,
            @RequestParam(name = ACCOUNT_ID, required = false) @Size(max = 2) EntityIdRangeParameter[] accountIds,
            @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,

                .limit(limit)
                .order(order)

                .build();

        var serviceResponse = service.getNftAllowances(request);
        var allowances = nftAllowanceMapper.map(serviceResponse);

        var sort = Sort.by(order, ACCOUNT_ID, TOKEN_ID);
        var pageable = PageRequest.of(0, limit, sort);

        return new NftAllowancesResponse().allowances(allowances).links(links);

    @Bean
    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
