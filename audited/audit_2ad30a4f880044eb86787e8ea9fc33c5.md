### Title
Unprivileged User Can Force Immediate Subscription Completion via Past `endTime`, Enabling Subscription-Flood DoS

### Summary
An unprivileged caller can supply a `TopicMessageFilter` with both `startTime` and `endTime` set to past timestamps (with `endTime > startTime`) that passes all Bean Validation constraints. Because `TopicContext.isComplete()` compares `filter.endTime` against the wall-clock time at context creation (`DomainUtils.now()`), not against `filter.startTime`, the subscription is immediately marked complete, short-circuiting live message delivery. Repeated at scale this creates a flood of short-lived subscriptions that each exercise the full subscription lifecycle.

### Finding Description
**Code path:**
- `TopicMessageFilter.isValidEndTime()` — `grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java` lines 43-46 — validates only `endTime > filter.startTime`.
- `TopicMessageFilter.isValidStartTime()` — lines 48-51 — validates only `startTime <= now`, permitting any past value.
- `TopicContext(TopicMessageFilter)` constructor — `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java` line 194 — sets `this.startTime = DomainUtils.now()` (wall-clock at subscription time).
- `TopicContext.isComplete()` — lines 203-215 — returns `true` when `filter.getEndTime() < this.startTime` (i.e., `endTime < now`).

**Root cause:** The validation layer (`isValidEndTime`) guards only the relationship between `endTime` and the user-supplied `filter.startTime`. It does not enforce `endTime > now`. The completion check (`isComplete`) uses a different reference point (`TopicContext.startTime = now`). These two reference points diverge whenever the user supplies a past `startTime`.

**Exploit flow:**
1. Attacker sends a `subscribeTopic` gRPC request with `startTime = T_past` (e.g., 60 s ago) and `endTime = T_past + 1 ns` (still in the past, but `> startTime`).
2. `isValidStartTime()` passes: `T_past <= now`. `isValidEndTime()` passes: `T_past+1 > T_past`.
3. `TopicContext` is created; `topicContext.startTime = now`.
4. `isComplete()` evaluates `filter.endTime < topicContext.startTime` → `T_past+1 < now` → **`true`**.
5. `incomingMessages()` (line 109) returns `Flux.empty()` immediately.
6. `pastEndTime()` (line 129) repeat condition `!isComplete()` is false; terminates immediately.
7. The safety-check `Mono.delay` (line 68) is filtered out because `isComplete()` is true.
8. `subscriberCount.incrementAndGet()` fires on subscribe; `subscriberCount.decrementAndGet()` fires on `doFinally` — both within milliseconds.
9. A historical DB query for `[T_past, T_past+1ns]` is still issued via `topicMessageRetriever.retrieve(filter, true)` (line 63), adding DB round-trip cost per request.

**Why existing checks are insufficient:**
- `@AssertTrue isValidEndTime` only compares `endTime` to `filter.startTime`, not to the server's current time.
- `@AssertTrue isValidStartTime` explicitly allows past `startTime` values.
- There is no rate-limiting or per-client subscription throttle in the service layer.

### Impact Explanation
Each malicious subscription: (a) opens a gRPC stream, (b) fires a database historical-retrieval query, (c) increments/decrements `subscriberCount` (Micrometer `AtomicLong` gauge), and (d) emits two log lines. At scale (thousands of requests/second from multiple clients), the primary impact is **database connection pool exhaustion and query load** from the historical retrieval, **log flooding**, and **gRPC thread-pool saturation**. The `subscriberCount` metric itself is low-cost (atomic integer), but Micrometer scrape endpoints and alerting pipelines that react to rapid oscillation of the gauge can be stressed. Legitimate subscribers may be starved of DB connections or gRPC worker threads.

### Likelihood Explanation
No authentication or authorization is required to call `subscribeTopic`. The gRPC endpoint is publicly reachable. Crafting the malicious filter requires only knowledge of the protobuf schema (publicly documented). The attack is trivially scriptable and repeatable with standard gRPC client libraries. No special privileges, tokens, or insider knowledge are needed.

### Recommendation
1. **Add a server-side lower bound on `endTime`** in `isValidEndTime()` or in `TopicMessageServiceImpl.subscribeTopic()`:
   ```java
   if (filter.getEndTime() != null && filter.getEndTime() <= DomainUtils.now()) {
       throw new IllegalArgumentException("endTime must be in the future");
   }
   ```
   Or add a Bean Validation constraint directly in `TopicMessageFilter`:
   ```java
   @AssertTrue(message = "End time must be in the future")
   public boolean isValidEndTime() {
       return endTime == null || (endTime > startTime && endTime > DomainUtils.now());
   }
   ```
2. **Add per-client rate limiting** on subscription creation (e.g., via a gRPC interceptor) to bound the subscription-creation rate regardless of filter values.
3. **Cap historical retrieval window**: enforce a maximum `endTime - startTime` range to limit DB query cost per subscription.

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
