### Title
Unauthenticated Unbounded Request Flood on `getNftAllowances()` Exhausts Shared Database Pool, Degrading Mirror Node Transaction Ingestion

### Summary
The `GET /api/v1/accounts/{id}/allowances/nfts` endpoint in `AllowancesController` delegates to `NftAllowanceServiceImpl.getNftAllowances()`, which issues a database query on every call with no rate limiting in the `rest-java` module. An unauthenticated attacker can flood this endpoint at arbitrary request rates, saturating the shared database connection pool and starving the importer service of DB connections, causing it to fall behind on ingesting gossip transactions from the Hedera network.

### Finding Description
**Code path:**
- `rest-java/src/main/java/org/hiero/mirror/restjava/controller/AllowancesController.java`, lines 57–83: `@GetMapping("/nfts")` with no authentication, no rate-limit annotation, no throttle guard.
- `rest-java/src/main/java/org/hiero/mirror/restjava/service/NftAllowanceServiceImpl.java`, lines 21–31: `getNftAllowances()` calls `entityService.lookup()` (DB read) then `repository.findAll(request, id)` (DB read) on every invocation.

**Root cause:** The `rest-java` module has no rate-limiting infrastructure. The `ThrottleConfiguration`/`ThrottleManagerImpl`/`ThrottleProperties` classes exist exclusively in the `web3` module (for EVM contract calls). The `rest-java` config package (`LoggingFilter`, `MetricsFilter`, `RestJavaConfiguration`, `WebMvcConfiguration`, etc.) contains no throttle filter or bucket. The only input guard on the endpoint is `@Max(MAX_LIMIT)` on the `limit` parameter, which caps result-set size per query but does not bound request frequency.

**Failed assumption:** The design assumes that infrastructure-level controls (API gateway, reverse proxy) will enforce rate limits before requests reach the application. No such enforcement is present in the application code itself, and no documentation or configuration enforces it as a deployment requirement.

**Exploit flow:**
1. Attacker sends `GET /api/v1/accounts/<id>/allowances/nfts` with varying `{id}` path values (defeating any hypothetical per-key caching).
2. Each request causes `NftAllowanceServiceImpl` to call `entityService.lookup()` + `repository.findAll()`, consuming one HikariCP connection for the duration of the query.
3. At high concurrency the connection pool (`hikaricp_connections_max`) is exhausted.
4. The importer service, which writes gossip-ingested transactions to the same PostgreSQL instance, blocks waiting for a connection, causing ingestion lag.

### Impact Explanation
The mirror node's importer and `rest-java` REST service share the same PostgreSQL database. Saturating the connection pool from the REST side directly degrades the importer's write throughput. This causes the mirror node to fall behind the live Hedera gossip stream — transactions are delayed or dropped from the mirror's view, breaking downstream consumers that rely on the mirror for transaction history and state. The impact is a sustained, operator-invisible denial of ingestion rather than a crash, making it harder to detect and remediate quickly.

### Likelihood Explanation
No privileges, API keys, or accounts are required. The endpoint is publicly documented in `rest/api/v1/openapi.yml` (lines 340–389). A single attacker with a commodity machine and a tool like `wrk` or `hey` can sustain thousands of requests per second. Varying the `{id}` path parameter (sequential account numbers are predictable on Hedera) defeats ETag/cache short-circuits. The attack is trivially repeatable and requires no prior knowledge of the system beyond the public API spec.

### Recommendation
1. **Add rate limiting to `rest-java`**: Introduce a `bucket4j`-based `OncePerRequestFilter` in `rest-java/src/main/java/org/hiero/mirror/restjava/config/` mirroring the pattern in `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, scoped to `/api/v1/**` endpoints.
2. **Per-IP rate limiting**: Apply per-source-IP token buckets (not a single global bucket) so a single attacker cannot consume the entire quota.
3. **Separate connection pools**: Configure a dedicated HikariCP pool for the importer with a reserved minimum, preventing REST API queries from starving importer writes.
4. **Enforce at deployment**: Document and enforce API-gateway–level rate limiting as a hard deployment requirement, not an optional recommendation.

### Proof of Concept
```bash
# Flood the endpoint with 500 concurrent connections for 30 seconds
# varying accountId to bypass any caching
wrk -t 10 -c 500 -d 30s \
  "http://<mirror-node-host>/api/v1/accounts/0.0.$(shuf -i 1-1000000 -n 1)/allowances/nfts"

# Observe importer lag via Prometheus metric:
# hiero_mirror_importer_transaction_latency_seconds increasing
# hikaricp_connections_pending{application="rest-java"} saturating
```

Expected result: `hikaricp_connections_active` reaches `hikaricp_connections_max`, importer write latency climbs, and the mirror node's latest ingested consensus timestamp diverges from the live network timestamp.
