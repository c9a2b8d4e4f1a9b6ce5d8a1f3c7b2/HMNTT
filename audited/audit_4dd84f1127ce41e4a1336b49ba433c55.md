### Title
Unauthenticated Global Rate-Limit Bypass Enables Per-Request Memory Exhaustion via Opcode Tracer

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint requires no authentication and accepts `stack=true` (default), `memory=true`, and `storage=true` simultaneously from any caller. The only rate-limiting protection is a single **global** token bucket (default: 1 req/s shared across all clients), not a per-IP or per-user limit. A single attacker can therefore monopolize the entire opcode-tracing capacity and force the server to re-execute the most gas-expensive historical transactions while capturing full stack, memory, and storage state for every opcode, allocating large heap objects on every request.

### Finding Description

**Entry point** — `OpcodesController.getContractOpcodes()`: [1](#0-0) 

`stack` defaults to `true`; `memory` and `storage` default to `false` but are freely settable by any caller. No authentication annotation is present.

**Throttle check** — `ThrottleManagerImpl.throttleOpcodeRequest()`: [2](#0-1) 

This consumes from a **single shared** `opcodeRateLimitBucket`. The default capacity is 1 token/second: [3](#0-2) 

There is no per-IP, per-session, or per-user sub-bucket. One attacker saturates the global quota entirely.

**Per-opcode memory allocation** — `AbstractOpcodeTracer`:

- `captureMemory()` allocates an `ArrayList<String>` sized to `frame.memoryWordSize()` with no upper-bound check: [4](#0-3) 

- `captureStack()` allocates an `ArrayList<String>` sized to `frame.stackSize()` with no upper-bound check: [5](#0-4) 

- `captureStorage()` builds an unbounded `TreeMap` from all cumulative storage accesses: [6](#0-5) 

All three are called for **every opcode** in `OpcodeActionTracer.tracePostExecution()`: [7](#0-6) 

The resulting `Opcode` objects accumulate in an `ArrayList` inside `OpcodeContext` whose initial capacity is `gas / 3` — directly proportional to the original transaction's gas limit: [8](#0-7) 

**Failed assumption**: The design assumes the global 1 req/s throttle is sufficient protection. It is not, because (a) it is global, not per-caller, so one attacker monopolizes it; and (b) it does not bound the heap allocated *within* a single request.

### Impact Explanation

A Hedera transaction can consume up to ~15 million gas. With all three flags enabled, the server allocates heap proportional to `(number_of_opcodes × (stack_depth + memory_words + storage_slots))` per request. For a gas-heavy contract this can reach hundreds of megabytes per request. At 1 req/s sustained, a single attacker can keep the JVM under continuous GC pressure, leading to latency spikes, OOM errors, or full service unavailability for all other API consumers. The gzip requirement only reduces *network* bandwidth; it does not reduce server-side heap allocation.

### Likelihood Explanation

- No credentials, API key, or account required.
- The endpoint is publicly documented in the OpenAPI spec.
- The attacker only needs to identify any historical high-gas transaction hash (trivially available from the public ledger) and send one request per second with `Accept-Encoding: gzip`, `memory=true`, `storage=true`.
- The attack is fully repeatable and automatable with a single `curl` loop.

### Recommendation

1. **Add per-IP (or per-authenticated-caller) rate limiting** in addition to the global bucket, so one client cannot monopolize the global quota.
2. **Cap per-request output size**: abort or truncate the opcode list when the accumulated `Opcode` count or estimated byte size exceeds a configurable threshold (e.g., 50 000 opcodes or 50 MB uncompressed).
3. **Require authentication** (e.g., an API key or signed request) for this endpoint, given its documented high cost.
4. **Bound memory captures**: add a maximum on `captureMemory()` word count and `captureStack()` depth before allocating the `ArrayList`.

### Proof of Concept

```bash
# 1. Find any high-gas historical Ethereum transaction hash from the public mirror REST API:
TX_HASH=$(curl -s "https://mainnet-public.mirrornode.hedera.com/api/v1/contracts/results?limit=1&order=desc" \
  | jq -r '.results[0].hash')

# 2. Continuously send maximum-cost opcode requests (1/s saturates the global throttle):
while true; do
  curl -s -o /dev/null \
    -H "Accept-Encoding: gzip" \
    "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"
  sleep 1
done
```

Each iteration forces a full EVM re-execution of the target transaction with all tracing enabled, allocating heap proportional to the transaction's gas usage. No credentials are required. A single attacker monopolizes the global 1 req/s opcode quota and keeps the server under sustained memory pressure.

            hexMemoryEntries.add(result);
        }

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
        final var context = ContractCallContext.get();

        // Reset the balance call flag after BALANCE opcode completes
        if (frame.getCurrentOperation() != null
                && BALANCE_OPERATION_NAME.equals(frame.getCurrentOperation().getName())) {
            context.setBalanceCall(false);
        }

        final var options = context.getOpcodeContext();
        final var memory = captureMemory(frame, options);
        final var stack = captureStack(frame, options);
        final var storage = captureStorage(frame, options, context);

        final var revertReasonBytes = frame.getRevertReason().orElse(null);
        final var reason = revertReasonBytes != null ? revertReasonBytes.toHexString() : null;
        context.getOpcodeContext()
                .addOpcodes(createOpcode(frame, operationResult.getGasCost(), reason, stack, memory, storage));
    }
