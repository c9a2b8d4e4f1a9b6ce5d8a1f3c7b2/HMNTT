### Title
Thread Pool Exhaustion via Uncached DB Query Blocking During Partial Network Partition

### Summary
The `findRuntimeBytecode()` method in `ContractRepository` executes a native SQL query against PostgreSQL. The only pre-execution guard is a `StatementInspector` that checks elapsed time **before** dispatching SQL, not during in-flight execution. During a partial network partition where the DB is reachable but slow, JDBC threads block indefinitely waiting for query responses (or delayed PostgreSQL cancellation signals), and an unauthenticated attacker sending requests at the permitted rate (500 RPS default) can exhaust the Tomcat thread pool, causing a complete service outage.

### Finding Description

**Exact code path:**

`ContractRepository.java` line 17–18 — the native query: [1](#0-0) 

The `@Cacheable` annotation uses `unless = "#result == null"`, meaning null results (non-existent contract IDs) are **never cached**: [2](#0-1) 

An attacker can therefore always bypass the cache by supplying non-existent contract IDs, forcing a DB round-trip on every request.

**The pre-execution-only guard:**

`HibernateConfiguration.statementInspector()` checks elapsed time only **before** the SQL string is handed to JDBC: [3](#0-2) 

Once the JDBC driver has dispatched the query and is blocked waiting for a response, this inspector is never called again. It cannot interrupt an in-flight query.

**The `statementTimeout` gap:**

The application-level `statementTimeout` is 3000 ms: [4](#0-3) 

This is a PostgreSQL server-side setting. In a slow-network partition, PostgreSQL fires the cancellation after 3000 ms **server-side**, but the cancellation error response must still traverse the degraded network back to the JDBC driver. The JDBC driver thread remains blocked until that response arrives. No JDBC-level socket timeout is configured for the web3 datasource to bound this wait independently of network conditions.

Additionally, PgBouncer is configured to **ignore** the `statement_timeout` startup parameter sent by clients: [5](#0-4) 

The DB-level user setting is 10000 ms (not 3000 ms): [6](#0-5) 

This means the effective server-side timeout is 10 s, not 3 s, and the application-level 3000 ms property has no enforced JDBC-level backing.

**Rate limiting is insufficient:**

The default rate limit is 500 RPS: [7](#0-6) 

The `requestTimeout` is 10 s: [8](#0-7) 

Spring Boot's default Tomcat thread pool is 200 threads. At 500 RPS with each thread blocking for ≥0.4 s (easily exceeded under a slow-network partition), the thread pool is exhausted in under one second. Subsequent requests queue and time out, producing a complete service outage.

### Impact Explanation

Complete denial of service for the web3 API. All endpoints share the same Tomcat thread pool; exhausting it with blocked DB threads prevents any request — including health checks — from being processed. The `requestTimeout` `StatementInspector` guard is ineffective once queries are in-flight, so threads remain blocked for the full network-round-trip duration, which is unbounded without a JDBC socket timeout.

### Likelihood Explanation

The attacker requires no credentials or special privileges — only the ability to send HTTP requests to the public web3 endpoint. The precondition (partial network partition / DB slowness) can arise from infrastructure events, cloud provider degradation, or an attacker who can induce network congestion between the API pod and the database. The exploit is trivially repeatable: send requests with distinct non-existent contract IDs at ≤500 RPS to stay under the rate limiter while still saturating the thread pool.

### Recommendation

1. **Configure a JDBC socket timeout** on the web3 HikariCP datasource (e.g., `socketTimeout` in the JDBC URL or `hikari.connection-timeout`) so that the JDBC driver itself aborts blocked reads independently of PostgreSQL's server-side cancellation signal.
2. **Align the PostgreSQL user-level `statement_timeout`** in the Helm chart with the intended 3000 ms application value (currently set to 10000 ms).
3. **Cache null/empty results** for non-existent contract IDs (remove or adjust `unless = "#result == null"`) or use a negative-result cache to prevent cache-bypass via non-existent IDs.
4. **Reduce the rate limit** or add per-IP rate limiting to limit the number of concurrent blocked threads a single client can create.

### Proof of Concept

**Preconditions:** Partial network partition between the web3 API pod and PostgreSQL (e.g., induced via `tc netem` adding 5000 ms latency on the DB-facing interface).

**Steps:**

```bash
# Generate 1000 requests with distinct non-existent contract IDs
# (null results are not cached due to unless="#result == null")
for i in $(seq 1 1000); do
  curl -s -X POST http://<web3-api>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"to\":\"0x$(printf '%040x' $((9999900 + i)))\",\"data\":\"0x70a08231$(printf '%064x' 1)\"}" &
done
wait
```

**Result:** With 5000 ms network latency, each request holds a Tomcat thread blocked in JDBC for ≥10 s (server-side cancellation + slow return path). After ~200 concurrent requests, the thread pool is exhausted. Subsequent legitimate requests receive connection-refused or timeout errors, constituting a complete service outage.

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
