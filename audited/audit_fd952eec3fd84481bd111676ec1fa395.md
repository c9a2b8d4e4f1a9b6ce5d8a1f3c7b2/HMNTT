### Title
Unauthenticated X-Real-IP Header Spoofing Enables Audit Log Identity Forgery in TracingMiddleware

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` (and `X-Forwarded-For`) header with no validation, allowlist, or source-IP check. Any unprivileged external attacker can set this header to an arbitrary value — including a trusted internal IP — causing all audit log entries to record the forged identity instead of the real source. This eliminates forensic traceability for any activity against the Rosetta API.

### Finding Description
**Code path:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74.

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)        // line 64 — no validation
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67 — no validation
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only fallback
    }
    return ipAddress
}
```

The returned value is embedded directly into every log line by `TracingMiddleware` (line 52–53):

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```

**Root cause:** The code assumes `X-Real-IP` is set exclusively by a trusted upstream reverse proxy. There is no enforcement of this assumption — the server binds directly via `httpServer.ListenAndServe()` (`main.go` line 231) with no mandatory proxy layer, and no check that the header originates from a trusted source IP.

**Failed assumption confirmed by tests:** `trace_test.go` lines 52–58 explicitly test that a client-supplied `X-Real-IP: 10.0.0.100` is accepted and logged verbatim — this is treated as correct behavior, meaning the spoofing surface is intentionally left open.

**Exploit flow:**
1. Attacker sends any Rosetta API request (e.g., `POST /block`) with header `X-Real-IP: 127.0.0.1` or any trusted internal IP.
2. `getClientIpAddress()` returns `"127.0.0.1"` without consulting `r.RemoteAddr`.
3. `TracingMiddleware` logs `127.0.0.1 POST /block (200) in Xms` — the real source IP is never recorded.
4. All subsequent requests from the attacker appear in logs as originating from the forged address.

**No existing mitigations:** There is no IP allowlist for header trust, no `RemoteAddr` cross-check, no stripping of the header at ingress, and no secondary logging of the real `RemoteAddr`.

### Impact Explanation
Every audit log entry for the Rosetta API can be forged by any external client. An attacker conducting reconnaissance, scraping, or abuse of the API (e.g., high-volume block/transaction queries, construction endpoint probing) can attribute all activity to `127.0.0.1` or any other trusted IP, making incident response and forensic attribution impossible. Since the mirror node is read-only, direct Hashgraph state modification is not possible, but the loss of forensic traceability is a concrete security control failure — particularly relevant in regulated or security-audited deployments.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and a single HTTP header. It is trivially repeatable with any HTTP client (`curl -H "X-Real-IP: 127.0.0.1" ...`). Any attacker who can reach the Rosetta port can exploit this unconditionally. The test suite confirms the behavior is stable and not accidentally introduced.

### Recommendation
Trust `X-Real-IP` / `X-Forwarded-For` only when the request's `r.RemoteAddr` matches a configured allowlist of trusted proxy IPs. If no trusted proxy is configured, fall back exclusively to `r.RemoteAddr`. Example fix:

```go
func getClientIpAddress(r *http.Request, trustedProxies []net.IP) string {
    remoteIP, _, _ := net.SplitHostPort(r.RemoteAddr)
    if isTrustedProxy(remoteIP, trustedProxies) {
        if ip := r.Header.Get(xRealIpHeader); ip != "" {
            return ip
        }
        if ip := r.Header.Get(xForwardedForHeader); ip != "" {
            return strings.TrimSpace(strings.Split(ip, ",")[0])
        }
    }
    return remoteIP
}
```

Additionally, always log `r.RemoteAddr` alongside the resolved client IP so the real network peer is always recorded.
