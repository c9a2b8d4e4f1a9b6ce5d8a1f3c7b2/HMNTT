### Title
Uncached EVM Address DB Lookup Amplification Across Five Parallel Account Sub-Routes

### Summary
Every one of the five account sub-routes in `rest/routes/accountRoute.js` independently calls `EntityService.getEncodedId()`, which — when given an opaque EVM address path parameter — unconditionally executes a live `entityFromEvmAddressQuery` DB query with no caching. An unauthenticated attacker can fan out a single logical account lookup into five simultaneous DB queries by hitting all five sub-routes concurrently, and can sustain this at high volume to multiply DB I/O and increase node resource consumption well beyond 30%.

### Finding Description

**Exact code locations:**

`rest/routes/accountRoute.js` lines 15–19 register five sub-routes, all accepting the same `/:idOrAliasOrEvmAddress` path parameter: [1](#0-0) 

Each handler independently calls `EntityService.getEncodedId()` as its first action:
- `AccountController.getNftsByAccountId` → `rest/controllers/accountController.js` line 91 [2](#0-1) 
- `AccountController.listStakingRewardsByAccountId` → line 171 [3](#0-2) 
- `CryptoAllowanceController.getAccountCryptoAllowances` → `rest/controllers/cryptoAllowanceController.js` line 77 [4](#0-3) 
- `TokenAllowanceController.getAccountTokenAllowances` → `rest/controllers/tokenAllowanceController.js` line 69 [5](#0-4) 
- `TokenController.getTokenRelationships` → `rest/controllers/tokenController.js` line 67 [6](#0-5) 

**Root cause — `getEncodedId()` branch for EVM addresses:**

In `rest/service/entityService.js` lines 118–137, when the input is a valid entity ID with a non-null `evmAddress` (i.e., an opaque EVM address, not a long-zero address), the code unconditionally calls `getEntityIdFromEvmAddress()`: [7](#0-6) 

`getEntityIdFromEvmAddress()` executes `entityFromEvmAddressQuery` — a raw DB query — on every invocation with no caching: [8](#0-7) [9](#0-8) 

**Failed assumption:** The design assumes each sub-route is called independently. There is no shared in-process cache, no request-scoped deduplication, and no result memoization across the five handlers. By contrast, a numeric account ID (`0.0.1234`) returns immediately via `entityId.getEncodedId()` (line 123) with zero DB queries. [10](#0-9) 

**No rate limiting on REST routes:** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module for contract calls. There is no rate-limiting middleware applied to the REST account sub-routes. [11](#0-10) 

### Impact Explanation
Each concurrent fan-out of 5 requests to the five sub-routes with an EVM address path parameter causes 5 independent `SELECT id FROM entity WHERE deleted <> true AND evm_address = $1` queries against the database. With numeric IDs the same 5 requests cause 0 DB queries for the ID resolution step. An attacker sustaining even a modest rate (e.g., 100 fan-outs/second = 500 DB queries/second purely for ID resolution) can measurably increase DB CPU, I/O, and connection pool pressure. Because the `entity` table is large and the query is not trivially indexed in all deployment configurations, this directly translates to increased node resource consumption exceeding the 30% threshold at achievable request rates. No authentication or special privilege is required.

### Likelihood Explanation
The attack requires only an HTTP client and knowledge of any valid (or even invalid) EVM address. No credentials, tokens, or privileged access are needed. The five sub-route URLs are publicly documented. The attack is trivially scriptable, repeatable, and can be parallelized across many source IPs to avoid simple IP-based blocking. The absence of any application-layer rate limiting on these endpoints makes sustained exploitation straightforward.

### Recommendation
1. **Add a request-scoped cache / memoization** in `EntityService.getEncodedId()` so that repeated lookups for the same EVM address within the same request context (or across a short TTL) are served from memory rather than hitting the DB each time.
2. **Introduce rate limiting** on the REST account sub-routes (e.g., via an Express middleware using a token-bucket algorithm), mirroring the throttle infrastructure already present in the `web3` module.
3. **Deduplicate at the route level**: consider a shared middleware that resolves and caches the `idOrAliasOrEvmAddress` path parameter once per request before dispatching to the handler, rather than re-resolving it independently in each controller.

### Proof of Concept
```bash
# Replace <EVM_ADDRESS> with any 40-hex-char EVM address (valid or not)
EVM="0xabcdef1234567890abcdef1234567890abcdef12"
BASE="https://<mirror-node-host>/api/v1/accounts/${EVM}"

# Fan out all 5 sub-routes simultaneously — no auth required
curl -s "${BASE}/nfts" &
curl -s "${BASE}/rewards" &
curl -s "${BASE}/allowances/crypto" &
curl -s "${BASE}/allowances/tokens" &
curl -s "${BASE}/tokens" &
wait

# Each of the 5 concurrent requests independently executes:
#   SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
# against the database — 5 DB queries for one logical account lookup.

# To amplify: loop at high concurrency
for i in $(seq 1 1000); do
  curl -s "${BASE}/nfts" &
  curl -s "${BASE}/rewards" &
  curl -s "${BASE}/allowances/crypto" &
  curl -s "${BASE}/allowances/tokens" &
  curl -s "${BASE}/tokens" &
done
wait
# Results in 5000 entityFromEvmAddressQuery DB executions with no rate limiting.
```

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }

public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
