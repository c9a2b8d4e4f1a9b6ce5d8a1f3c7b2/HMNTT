### Title
Unauthenticated `BufferUnderflowException` DoS via Short EVM Address in `getByEvmAddressAndType()`

### Summary
`decodeEvmAddress()` performs no length validation on the decoded byte array, allowing any caller to supply a hex-encoded EVM address shorter than 20 bytes. When the first 4 bytes are zero, `ByteBuffer.getInt()` succeeds but the subsequent `getLong()` call throws an uncaught `BufferUnderflowException`, crashing the request and potentially exposing internal stack traces.

### Finding Description
**Code path:**
- `EntityServiceImpl.java` → `getByEvmAddressAndType()` (lines 34–41)
- `GraphQlUtils.java` → `decodeEvmAddress()` (lines 87–98)

**Root cause:** `decodeEvmAddress()` only validates that the input is valid hex; it never asserts `result.length == 20`. The caller `getByEvmAddressAndType()` unconditionally calls `buffer.getInt()` (4 bytes) then `buffer.getLong()` (8 bytes) then `buffer.getLong()` (8 bytes), requiring exactly 20 bytes. A buffer shorter than 12 bytes where the first 4 bytes are zero will pass the `getInt() == 0` check and then throw `BufferUnderflowException` on the first `getLong()`.

**Why existing checks fail:** `decodeEvmAddress()` catches only `DecoderException` (odd-length or non-hex input). A valid even-length hex string of any length under 40 chars passes through. The test suite in `GraphQlUtilsTest.java` (line 93–97) explicitly treats 5-byte decoded results as valid output of `decodeEvmAddress()`.

### Impact Explanation
Any unauthenticated GraphQL client can crash individual account-lookup requests. Depending on the Spring/GraphQL error handler configuration, the raw `BufferUnderflowException` stack trace (revealing internal class names, package structure, and data model layout) may be returned in the GraphQL `errors` response field. Repeated calls constitute a low-cost DoS against the account resolution endpoint.

### Likelihood Explanation
No authentication is required. The attacker only needs to know the GraphQL endpoint and supply a short hex string. The `GraphQlUtilsTest` test suite itself demonstrates 5-byte inputs are accepted, so the precondition is trivially met. This is repeatable and scriptable.

### Recommendation
Add an explicit length check in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
// In GraphQlUtils.decodeEvmAddress(), after Hex.decodeHex():
byte[] decoded = Hex.decodeHex(evmAddress);
if (decoded.length != 20) {
    throw new IllegalArgumentException("EVM address must be exactly 20 bytes");
}
return decoded;
```

Alternatively, add the check at the start of `getByEvmAddressAndType()` before `ByteBuffer.wrap()`.

### Proof of Concept
Send the following GraphQL query to the mirror node GraphQL endpoint (no credentials needed):

```graphql

  account(input: { evmAddress: "0000000001" }) {
    id
  }
}
```

1. `decodeEvmAddress("0000000001")` → returns `byte[5]` (no exception, valid hex).
2. `ByteBuffer.wrap(byte[5])` → buffer of capacity 5.
3. `buffer.getInt()` → reads bytes 0–3 = `0x00000000` = 0 → condition `== 0` is **true**.
4. `buffer.getLong()` → attempts to read 8 bytes from position 4, only 1 byte remains → **`BufferUnderflowException`** thrown, propagates uncaught through `getByEvmAddressAndType()`. [1](#0-0) [2](#0-1) [3](#0-2)
