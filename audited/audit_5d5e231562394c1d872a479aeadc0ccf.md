### Title
Missing Connection-Level Rate Limiting in `grpcServerConfigurer` Enables Malformed-Frame Resource Exhaustion

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only a single protection: `maxConcurrentCallsPerConnection(5)`. It omits all connection-lifecycle controls (`maxConnectionAge`, `maxConnectionIdle`, `maxInboundMessageSize`, `permitKeepAliveTime`). An unprivileged attacker can repeatedly open TCP connections, send malformed HTTP/2 frames to force GOAWAY responses and connection teardown, and cycle this at high frequency, exhausting server-side file descriptors, CPU (TLS handshake cost), and connection-state memory without any gRPC-layer throttle.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`NettyProperties.java` (lines 11–15) exposes only `maxConcurrentCallsPerConnection = 5`. No other `NettyServerBuilder` options are wired.

**Root cause:** The `NettyServerBuilder` API provides `maxConnectionAge()`, `maxConnectionIdle()`, `maxInboundMessageSize()`, `permitKeepAliveTime()`, and `permitKeepAliveWithoutCalls()` — none are configured. The failed assumption is that OS-level TCP limits or upstream proxies will absorb connection-cycling attacks before they reach the gRPC server.

**Exploit flow:**
1. Attacker opens a raw TCP connection to the gRPC port (5600 direct, or via the nginx proxy on 8080 which passes gRPC through without connection-rate limiting per `docker-compose.yml` lines 226–229).
2. Sends a valid HTTP/2 connection preface (`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`) followed by a deliberately malformed frame (e.g., invalid frame type, bad stream ID 0 DATA frame, oversized HEADERS without CONTINUATION, or invalid HPACK literal).
3. Netty's HTTP/2 codec detects the protocol error, allocates error state, generates a GOAWAY frame, and tears down the connection.
4. Attacker immediately repeats from step 1 at high frequency (thousands of cycles/second from a single host).
5. No gRPC-layer counter, token bucket, or connection-age limit exists to slow or block this cycle.

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection(5)` limits simultaneous RPC streams on a live connection; it has zero effect on connection establishment rate or on connections that are torn down before any RPC is dispatched.
- The Traefik circuit breaker (`NetworkErrorRatio() > 0.10`) in `charts/hedera-mirror-grpc/values.yaml` line 158 operates at the ingress/proxy layer and only triggers after a sustained error ratio is observed — it does not prevent the attack from reaching the gRPC server, and in direct-access deployments it is absent entirely.

### Impact Explanation
Each malformed-connection cycle consumes: one TCP handshake, optionally one TLS handshake (CPU-intensive), Netty HTTP/2 connection-state allocation (~tens of KB), frame-parsing CPU, GOAWAY frame serialization, and connection teardown. At scale, this exhausts OS file descriptors (each open/closing connection holds one), JVM heap (connection state objects), and CPU. Legitimate clients experience connection refusals or severe latency degradation — a practical network partition between the server and valid subscribers. Severity is high because the server exposes a public gRPC port and the attack requires zero authentication.

### Likelihood Explanation
Any unprivileged network-reachable attacker can execute this. The tooling required is minimal: a raw TCP socket or a modified gRPC client that sends a bad frame after the HTTP/2 preface. The attack is fully repeatable, stateless on the attacker side, and amplified by the fact that each cycle forces server-side work disproportionate to attacker cost. No exploit framework or special knowledge beyond the HTTP/2 RFC is needed.

### Recommendation
In `grpcServerConfigurer()`, add the following `NettyServerBuilder` calls and expose them as configurable `NettyProperties` fields:

```java
serverBuilder.maxConnectionAge(nettyProperties.getMaxConnectionAge(), TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(nettyProperties.getMaxConnectionIdle(), TimeUnit.SECONDS);
serverBuilder.maxInboundMessageSize(nettyProperties.getMaxInboundMessageSize());
serverBuilder.permitKeepAliveTime(nettyProperties.getPermitKeepAliveTime(), TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Reasonable defaults: `maxConnectionAge=300s`, `maxConnectionIdle=60s`, `maxInboundMessageSize=4MB`. Additionally, add a connection-rate limit at the nginx/Traefik layer (`limit_conn` / `limit_req` for gRPC locations) so that connection cycling is throttled before reaching the JVM.

### Proof of Concept
```python
import socket, time

TARGET_HOST = "grpc-server-host"
TARGET_PORT = 5600

# Valid HTTP/2 client preface
H2_PREFACE = b"PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n"

# Malformed frame: type=0xFF (unknown), flags=0x00, stream_id=0, length=0
MALFORMED_FRAME = b"\x00\x00\x00\xff\x00\x00\x00\x00\x00"

for i in range(10000):
    try:
        s = socket.create_connection((TARGET_HOST, TARGET_PORT), timeout=2)
        s.sendall(H2_PREFACE + MALFORMED_FRAME)
        # Server sends GOAWAY and closes; we don't wait
        s.close()
    except Exception:
        pass
    # No sleep — maximum connection cycling rate
```

Run this from a single unprivileged host. Monitor server file-descriptor count (`/proc/<pid>/fd`) and CPU usage. Legitimate gRPC clients will begin experiencing `UNAVAILABLE` or connection-refused errors as file descriptors are exhausted and the thread pool is saturated with teardown work.
