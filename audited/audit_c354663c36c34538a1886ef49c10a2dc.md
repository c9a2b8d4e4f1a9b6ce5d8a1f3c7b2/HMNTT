### Title
Global Rate Limit Exhaustion via Single-IP Flooding (No Per-IP Isolation in `rateLimitBucket`)

### Summary
`ThrottleConfiguration.rateLimitBucket()` creates a single application-wide Bucket4j token bucket shared across all callers with no per-IP or per-user partitioning. A single unauthenticated client can legally consume all 500 tokens/second, reducing the remaining capacity for every other user to zero and producing a sustained denial-of-service against financial transaction endpoints.

### Finding Description
**Code path:**
- `ThrottleConfiguration.java` lines 24â€“32: one `Bucket` bean, capacity=500, `refillGreedy(500, Duration.ofSeconds(1))`, no `SynchronizationStrategy` override (defaults to `LOCK_FREE`, i.e. fully concurrent).
- `ThrottleManagerImpl.java` line 38: `if (!rateLimitBucket.tryConsume(1))` â€” every inbound request, regardless of origin, draws from this single shared pool.

**Root cause:** The design assumes the 500 req/s budget will be spread across many independent callers. There is no per-IP, per-session, or per-account bucket. The `ThrottleProperties` class (`requestsPerSecond = 500`) and the `RequestProperties` per-filter buckets only apply to optional, operator-configured content filters â€” they are not a substitute for caller isolation.

**Exploit flow:**
1. Attacker opens â‰Ą500 concurrent HTTP connections (no authentication required for `eth_call`/`eth_estimateGas` endpoints).
2. Each connection fires one request per second, collectively consuming all 500 tokens.
3. `rateLimitBucket.tryConsume(1)` returns `false` for every legitimate user request.
4. `ThrottleManagerImpl` throws `ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED)` for all subsequent callers.
5. Attacker sustains the flood; the bucket refills at exactly 500/s, which the attacker immediately re-consumes.

**Why existing checks are insufficient:**
- `gasLimitBucket` is a separate concern (gas accounting) and does not protect the request-rate path.
- `opcodeRateLimitBucket` only covers opcode-trace endpoints.
- The `RequestProperties` per-filter buckets (`createBucket()`, lines 63â€“68) are opt-in, operator-configured, and match on request *content*, not caller identity â€” they are not deployed by default (`request = List.of()`).
- No IP-allowlist, connection-limit, or network-layer guard is present in this code path.

### Impact Explanation
Any unauthenticated user can block 100 % of `eth_call`, `eth_estimateGas`, and related JSON-RPC endpoints for all other users. Because these endpoints are used to simulate and submit financial transactions (token transfers, DeFi interactions), a sustained attack prevents legitimate users from executing or estimating any on-chain financial operation. Severity is **High**: complete availability loss for a financial-transaction API with no authentication barrier to the attacker.

### Likelihood Explanation
The attack requires only an HTTP client capable of sustaining 500 req/s â€” trivially achievable with `wrk`, `ab`, or a small script on a single commodity machine. No credentials, tokens, or special knowledge of the application are needed. The attack is repeatable indefinitely and is not self-limiting (the attacker is never throttled harder than legitimate users). Real-world exploitation by a motivated party (competitor, extortionist, or vandal) is highly feasible.

### Recommendation
Replace the single global bucket with per-IP (or per-authenticated-identity) buckets, e.g. using a `ConcurrentHashMap<String, Bucket>` keyed on the resolved client IP, or integrate a dedicated rate-limiting layer (e.g. Spring Cloud Gateway `RequestRateLimiter`, Nginx `limit_req`, or a Redis-backed Bucket4j distributed store). At minimum, add a per-IP hard cap (e.g. 50 req/s per IP) so that no single source can consume more than a defined fraction of the global budget. The `gasLimitBucket` already demonstrates use of `SynchronizationStrategy.SYNCHRONIZED`; the same pattern should be extended to caller-scoped buckets.

### Proof of Concept
```bash
# Exhaust the global 500 req/s budget from a single machine
wrk -t10 -c500 -d60s \
  -s post.lua \          # lua script sending a valid eth_call JSON body
  http://<mirror-node-host>/api/v1/contracts/call

# Simultaneously, from a second machine, observe all requests rejected:
curl -X POST http://<mirror-node-host>/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"data":"0x...","to":"0x...","estimate":false}'
# Expected response while flood is active:
# HTTP 429 - "Requests per second rate limit exceeded"
```
The flood machine holds all 500 tokens/s; the observer machine receives HTTP 429 on every attempt for the duration of the attack.
