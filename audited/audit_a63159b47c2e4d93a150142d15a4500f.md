### Title
Missing Aggregate Request Timeout and Rate Limiting on `getTopic()` Enables Thread Pool Exhaustion via Concurrent Slow-Query Griefing

### Summary
`getTopic()` in `TopicController.java` executes three sequential, synchronous, blocking database calls with no aggregate request-level timeout and no rate limiting. While each individual query is bounded by a PostgreSQL `statement_timeout` (10–20 seconds), the absence of a per-request ceiling means a single request can hold a Tomcat thread for up to 3× that value. An unprivileged attacker flooding the endpoint with concurrent requests can exhaust the bounded thread pool, degrading availability for legitimate users.

### Finding Description

**Exact code path:**

`getTopic()` at `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TopicController.java` lines 32–37 executes three sequential blocking repository calls:

```java
var topic = topicService.findById(id.id());      // DB call 1
var entity = entityService.findById(id.id());    // DB call 2
var customFee = customFeeService.findById(id.id()); // DB call 3
```

Each call delegates to a Spring Data JPA `findById` with no timeout annotation or reactive wrapper:
- `TopicServiceImpl.findById` → `topicRepository.findById`
- `EntityServiceImpl.findById` → `entityRepository.findById`
- `CustomFeeServiceImpl.findById` → `customFeeRepository.findById`

**Root cause — failed assumption:**

The design assumes that the PostgreSQL-level `statement_timeout` (configured at `hiero.mirror.restJava.db.statementTimeout = 10000 ms` and enforced via `ALTER USER mirror_rest_java SET statement_timeout TO '20000'`) is sufficient to bound request duration. It is not, because:

1. The timeout applies **per individual statement**, not per HTTP request.
2. Three sequential calls each allowed up to 10–20 s = **30–60 s maximum thread hold time per request**.
3. There is **no aggregate request timeout** in the rest-java module. The web3 module has `HibernateConfiguration.statementInspector()` enforcing `web3Properties.getRequestTimeout()` across all queries in a request context; rest-java has no equivalent.
4. There is **no rate limiting** on the `/api/v1/topics/{id}` endpoint. The web3 module has `ThrottleConfiguration` with `requestsPerSecond` and bucket4j enforcement; rest-java has none.

The only filters registered in rest-java are `LoggingFilter` and `MetricsFilter` — neither enforces any throttle or timeout.

**Exploit flow:**

An attacker sends a high volume of concurrent `GET /api/v1/topics/{id}` requests. Under DB load (or by timing requests to coincide with natural DB slowness), each request holds a Tomcat worker thread for up to 60 s. Spring MVC uses a blocking thread-per-request model with a bounded Tomcat thread pool (default 200). With enough concurrent slow requests, the pool saturates and legitimate requests queue or are rejected.

**Why existing checks are insufficient:**

- `statement_timeout` at the DB level bounds each query individually, not the sum of three sequential queries.
- HikariCP connection pool exhaustion adds a secondary bottleneck but does not release the Tomcat thread — it causes the thread to block waiting for a connection (up to `connectionTimeout`), compounding the hold time.
- No `@Timeout`, no `CompletableFuture` with deadline, no reactive pipeline, no `QueryTimeoutException` guard at the controller or service layer.

### Impact Explanation

Thread pool exhaustion causes HTTP 503 / connection refused for legitimate users. The impact is availability degradation (griefing) with no economic damage to any on-chain participant, consistent with the Medium scope classification. The three-call sequential structure triples the worst-case thread hold time compared to a single-call endpoint, making this endpoint disproportionately expensive to serve under adversarial conditions.

### Likelihood Explanation

No authentication or API key is required to call `GET /api/v1/topics/{id}`. Any external user can issue requests. The attacker does not need to control the database — they only need to sustain enough concurrent connections to saturate the thread pool before the DB statement_timeout fires. With a 20 s per-statement timeout and three calls, even modest concurrency (e.g., 200 threads × 60 s hold = sustained saturation) is achievable from a single machine with keep-alive connections. The attack is trivially repeatable and requires no special knowledge beyond the public API path.

### Recommendation

1. **Add an aggregate request timeout** in the rest-java module, mirroring the web3 `HibernateConfiguration` pattern: inject a `StatementInspector` that reads a configurable `requestTimeout` and throws `QueryTimeoutException` if the elapsed time since request start exceeds the limit across all queries in the request.
2. **Add rate limiting** to the rest-java controllers using bucket4j or Spring's `HandlerInterceptor`, analogous to `ThrottleConfiguration` in the web3 module.
3. **Parallelize the three service calls** using `CompletableFuture.allOf()` or virtual threads, reducing the maximum thread hold time from `3 × statementTimeout` to `1 × statementTimeout`.
4. **Configure `server.tomcat.connection-timeout`** and `server.tomcat.keep-alive-timeout` to limit how long idle or slow connections hold resources.

```java
    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
    }
```
