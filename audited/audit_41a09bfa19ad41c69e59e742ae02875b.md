### Title
Unauthenticated Endpoint `GET /api/v1/network/stake` Has No Rate Limiting or Caching, Enabling Resource Exhaustion via Repeated `findLatest()` Calls

### Summary
The `GET /api/v1/network/stake` endpoint is publicly accessible with no authentication, no rate limiting, and no caching. Every request causes `NetworkStakeRepository.findLatest()` to open a Spring-managed transaction, execute a correlated subquery against the database, and close the transaction. Under high concurrency from a single unprivileged attacker, this exhausts the database connection pool and saturates transaction manager synchronization, degrading all transactional operations in the application.

### Finding Description
**Code path:**

`NetworkController.getNetworkStake()` (line 126–130) → `NetworkServiceImpl.getLatestNetworkStake()` (line 52–56) → `NetworkStakeRepository.findLatest()` (lines 12–19).

```
// NetworkStakeRepository.java lines 12-19
@Query(value = """
    select *
    from network_stake
    where consensus_timestamp = (
        select max(consensus_timestamp) from network_stake
    )
    """, nativeQuery = true)
Optional<NetworkStake> findLatest();
```

Spring Data JPA wraps every repository method invocation in a transaction by default. Each call to `findLatest()` therefore: acquires a connection from the pool, begins a transaction, executes the correlated subquery, commits/closes the transaction, and releases the connection. There is no `@Cacheable` annotation anywhere on this call path in the `rest-java` module (confirmed: `grep` for `Cacheable` in `rest-java/src/main/java/**` returns zero hits on this path).

**No rate limiting exists in the `rest-java` module.** The `ThrottleConfiguration` and `ThrottleManagerImpl` classes exist exclusively in the `web3` module and are not applied to `rest-java` endpoints. The `WebMvcConfiguration` and `RestJavaConfiguration` register only an ETag filter and argument resolvers — no rate-limiting filter. The `NetworkController` has no `@RateLimiter`, no Bucket4j guard, and no Spring Security layer protecting this endpoint.

**Root cause:** The combination of (a) a public, unauthenticated, parameter-free GET endpoint, (b) no response caching, and (c) a per-request DB transaction with a correlated subquery means every inbound HTTP request unconditionally consumes a connection-pool slot and incurs full transaction lifecycle overhead.

### Impact Explanation
Under sustained flood (e.g., thousands of concurrent requests/second from a single host), the HikariCP connection pool is exhausted. Threads waiting for a connection block, causing request queuing to grow unboundedly. The Spring `PlatformTransactionManager` synchronization registry (`TransactionSynchronizationManager`) is accessed on every transaction begin/end under thread-local locking, adding contention at scale. All other transactional endpoints in the same application (supply, exchange rate, nodes, etc.) share the same connection pool and transaction manager, so they degrade simultaneously. The result is application-wide latency increase and eventual timeout/503 responses for all users — a full denial of service with no economic cost to the attacker.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero special knowledge beyond the public OpenAPI spec (which documents `GET /api/v1/network/stake`). The endpoint accepts no parameters, so no input crafting is needed. A single attacker with a modest HTTP flood tool (e.g., `wrk`, `ab`, `hey`) can sustain thousands of requests per second. The attack is trivially repeatable and requires no state.

### Recommendation
1. **Add response caching** at the service layer using `@Cacheable` with a short TTL (e.g., 30–60 seconds) on `getLatestNetworkStake()` or directly on `findLatest()`. Network stake data changes at most once per staking period (daily), so caching is safe and eliminates nearly all DB load.
2. **Add rate limiting** to the `rest-java` module analogous to the `web3` module's `ThrottleConfiguration`/Bucket4j setup, applied globally or per-IP via a servlet filter registered in `RestJavaConfiguration`.
3. **Tune the connection pool** with a maximum pool size and acquisition timeout so that pool exhaustion fails fast rather than queuing indefinitely.

### Proof of Concept
```bash
# Flood the public endpoint with no credentials required
wrk -t 16 -c 500 -d 60s http://<mirror-node-host>/api/v1/network/stake
```
**Preconditions:** None. No authentication, no parameters.  
**Trigger:** Each of the 500 concurrent connections continuously issues `GET /api/v1/network/stake`.
