### Title
Unconditional DB Query in `getTokenAccounts()` with No REST Rate Limiting Enables Connection Pool Exhaustion DoS

### Summary
`TokenService.getTokenAccounts()` always executes a database query against the `token_account` table before checking whether the result is empty, with no rate limiting on the REST API layer. An unprivileged attacker can flood `GET /api/v1/accounts/{id}/tokens` using any valid account with no token associations, holding DB connections on every request and exhausting the connection pool, denying service to all other users.

### Finding Description
**Exact code path:**

In `rest/service/tokenService.js`, `getTokenAccounts()` at line 98 unconditionally executes the SQL query via `super.getRows(sqlQuery, params)` before the empty-result check at line 99:

```js
async getTokenAccounts(query) {
    const {sqlQuery, params} = this.getTokenRelationshipsQuery(query);
    const rows = await super.getRows(sqlQuery, params);   // DB hit always happens
    if (rows.length === 0) {
      return [];                                           // early return AFTER DB query
    }
    ...
}
``` [1](#0-0) 

`super.getRows()` in `rest/service/baseService.js` calls `this.pool().queryQuietly(query, params)`, which acquires a connection from the global `pg` pool configured in `rest/dbpool.js` with `max: config.db.pool.maxConnections`. [2](#0-1) [3](#0-2) 

**No rate limiting on the REST API:** Reviewing `rest/server.js`, the middleware stack applies only `httpContext`, `requestLogger`, `authHandler`, optional metrics, and optional response cache — there is no per-IP rate limiting, concurrency limiting, or request throttling middleware for any REST route including `/accounts/:id/tokens`. [4](#0-3) 

The throttling code (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` Java module and does not apply to the Node.js REST API.

**Controller flow:** `tokenController.js` calls `EntityService.isValidAccount(accountId)` (itself a DB query) and then unconditionally calls `TokenService.getTokenAccounts(query)`. For a valid account with no token associations, `isValidAccount` returns `true`, so the guard does not prevent the subsequent DB query in `getTokenAccounts`. [5](#0-4) 

**Root cause / failed assumption:** The code assumes the empty-result path is cheap because it returns early, but the DB connection is already acquired and held for the duration of the query before the `rows.length === 0` check is reached. There is no cache, no pre-flight check, and no concurrency guard.

### Impact Explanation
With the connection pool exhausted, all subsequent requests to any REST endpoint that requires a DB connection will queue or fail with a connection timeout error (`connectionTimeoutMillis`). This is a full-service DoS affecting all API consumers — not just the targeted endpoint — because the Node.js REST service uses a single shared global pool. The `statement_timeout` limits individual query duration but does not prevent pool exhaustion under sustained concurrent load; an attacker simply needs to maintain enough concurrent in-flight requests to keep all pool slots occupied.

### Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge beyond a valid account ID (which is publicly enumerable from the mirror node itself). The endpoint is publicly accessible. The attacker needs only a valid account with no token associations — trivially obtainable — and a tool capable of sending concurrent HTTP requests (e.g., `wrk`, `ab`, `hey`). The attack is repeatable and sustainable indefinitely. No existing application-layer control prevents it.

### Recommendation
1. **Add rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit` per IP) before DB-touching routes, specifically for the `/accounts/:id/tokens` endpoint.
2. **Add a concurrency/in-flight request limit** (e.g., via `express-slow-down` or a semaphore) to cap simultaneous DB-bound requests.
3. **Cache empty results**: after confirming `rows.length === 0`, cache the negative result for the account ID for a short TTL (e.g., using the existing Redis layer already configured in the service) so repeated requests for the same account do not hit the DB.
4. **Consider a pre-flight existence check** using a lightweight cached lookup before issuing `tokenRelationshipsQuery`.

### Proof of Concept
**Preconditions:**
- Mirror node REST API is running and accessible.
- Account `0.0.X` exists (valid, passes `isValidAccount`) but has zero token associations.

**Steps:**
```bash
# 1. Confirm account exists with no tokens (returns {"tokens":[]})
curl "http://<mirror-node>/api/v1/accounts/0.0.X/tokens"

# 2. Flood the endpoint with concurrent requests (adjust -c to exceed pool size)
wrk -t8 -c50 -d60s "http://<mirror-node>/api/v1/accounts/0.0.X/tokens"

# 3. In parallel, observe that other endpoints begin timing out or returning 500
curl "http://<mirror-node>/api/v1/transactions"
# Expected: connection timeout / pool exhaustion error
```

**Result:** With pool connections saturated, all DB-dependent REST endpoints become unavailable for the duration of the flood, constituting a non-network-based DoS requiring no privileges.
