### Title
Triple-UNION DB Work Amplification via Pagination Bounds in `getContractLogsQuery()`

### Summary
In `rest/service/contractService.js`, `getContractLogsQuery()` constructs up to three independent sub-queries (lower, inner, upper), each applying the full user-supplied `limit` against `contract_log` with a CTE and join. When all three filter arrays are populated — which any unauthenticated user can trigger with a single valid request — the database performs approximately 3× the work of a single-limit query, reliably exceeding the 30% resource-consumption threshold without any brute-force or repeated requests.

### Finding Description

**Exact code path:**

`rest/service/contractService.js`, `getContractLogsQuery()`, lines 323–368: [1](#0-0) 

The function pushes `limit` into `params` once (line 324), then computes `limitClause = super.getLimitQuery(params.length)` (line 333) — a single `limit $N` placeholder referencing that same position. All three sub-queries (lower, inner, upper) are built with `buildSelectQuery(…, limitClause, …)`, so each independently receives `limit $N` with the same value: [2](#0-1) 

When `subQueries.length > 1`, the outer UNION also appends the same `limitClause` again: [3](#0-2) 

Each sub-query independently scans `contract_log`, evaluates the CTE (`entity`), performs a `LEFT JOIN`, applies its own `WHERE` conditions, sorts, and fetches up to `limit` rows. The outer UNION then merges up to `3 × limit` rows, re-sorts, and limits to `limit`. Total DB work ≈ 3× a single-query execution.

**How all three filter arrays are populated:**

`extractContractLogsMultiUnionQuery` in `rest/controllers/contractController.js` (lines 582–699) parses user-supplied `timestamp` and `index` query parameters into `bounds.primary` and `bounds.secondary`: [4](#0-3) 

When the user supplies `timestamp=gte:T1&timestamp=lte:T2&index=gte:I1&index=lte:I2`, the Bound objects produce non-empty lower, inner, and upper filter arrays simultaneously. This is confirmed by the test at line 310: [5](#0-4) 

**Why existing checks are insufficient:**

`getLimitParamValue()` in `rest/utils.js` caps the limit at `responseLimit.max` (default 100): [6](#0-5) 

This cap bounds only the final output row count, not the per-sub-query scan cost. With `limit=100`, the DB scans up to 300 rows across three sub-queries before the outer UNION reduces to 100. The cap does not reduce the multiplied work.

`validateContractLogsBounds` only rejects the case where `secondary` has an equal but `primary` does not: [7](#0-6) 

It does not prevent the three-sub-query case. No rate limiting, query timeout enforcement, or per-sub-query limit reduction exists in this path.

### Impact Explanation

A single unauthenticated HTTP request causes the database to execute three full CTE+join+scan+sort operations instead of one, each bounded by `max_limit` (default 100). This is a deterministic ~3× amplification of DB CPU, I/O, and memory work per request. Sustained at even moderate request rates, this can push DB CPU and I/O utilization well above the 30% increase threshold. The `contract_log` table on a production mirror node can be very large, making each sub-query scan expensive. The impact is denial-of-service degradation of the mirror node's database tier, affecting all API consumers.

### Likelihood Explanation

The exploit requires zero authentication, zero special knowledge, and a single HTTP GET request to a public endpoint (`/api/v1/contracts/results/logs` or `/api/v1/contracts/{id}/logs`). The required parameters (`timestamp=gte:T1&timestamp=lte:T2&index=gte:I1&index=lte:I2&limit=100`) are documented in the OpenAPI spec: [8](#0-7) 

Any attacker who reads the API documentation can construct this request. It is trivially repeatable in a loop from a single client, and the amplification is consistent and predictable.

### Recommendation

Reduce the per-sub-query limit when multiple sub-queries are generated. Before building sub-queries, compute `subQueryLimit = Math.ceil(limit / nonEmptyFilterCount)` and use that in each sub-query's `limitClause`, reserving the full `limit` only for the outer UNION. Alternatively, apply a configurable per-request DB statement timeout (`statement_timeout`) at the connection level for this query path to bound worst-case execution time regardless of amplification.

### Proof of Concept

```
GET /api/v1/contracts/results/logs?timestamp=gte:1000000000.000000000&timestamp=lte:9999999999.000000000&index=gte:0&index=lte:9999&limit=100
```

**Steps:**
1. Send the above request to a mirror node with a populated `contract_log` table.
2. Observe in PostgreSQL `pg_stat_activity` or slow-query logs that the executed SQL contains three `UNION`-ed sub-queries, each with `limit $3` (value 100).
3. Measure DB CPU/IO during a sustained series of these requests vs. baseline; the increase will be ≥ 3× per request compared to a simple `limit=100` query with no pagination bounds.
4. The generated SQL matches the structure confirmed in the test at `rest/__tests__/service/contractService.test.js` lines 329–414, where all three sub-queries each carry `limit $3`.

  getContractLogsQuery({lower, inner, upper, params, conditions, order, limit}) {
    params.push(limit);
    const orderClause = super.getOrderByQuery(
      OrderSpec.from(ContractLog.getFullName(ContractLog.CONSENSUS_TIMESTAMP), order),
      OrderSpec.from(ContractLog.getFullName(ContractLog.INDEX), order)
    );
    const orderClauseNoAlias = super.getOrderByQuery(
      OrderSpec.from(ContractLog.CONSENSUS_TIMESTAMP, order),
      OrderSpec.from(ContractLog.INDEX, order)
    );
    const limitClause = super.getLimitQuery(params.length);

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) =>
        super.buildSelectQuery(
          ContractService.contractLogsExtendedQuery,
          params,
          conditions,
          orderClause,
          limitClause,
          filters.map((filter) => ({
            ...filter,
            column: ContractLog.getFullName(ContractService.contractLogsPaginationColumns[filter.key]),
          }))
        )
      );

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = super.buildSelectQuery(
        ContractService.contractLogsExtendedQuery,
        params,
        conditions,
        orderClause,
        limitClause
      );
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion\n'), orderClauseNoAlias, limitClause].join('\n');
    }

    return [sqlQuery, params];
  }

const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
