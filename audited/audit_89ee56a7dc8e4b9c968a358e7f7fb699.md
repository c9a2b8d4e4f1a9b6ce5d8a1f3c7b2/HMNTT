### Title
Unauthenticated Bandwidth/CPU Exhaustion via Unprotected `serveSpec` Endpoint Bypassing Response Cache

### Summary
The `serveSpec` handler in `rest/middleware/openapiHandler.js` is registered on Express **before** any middleware (including the Redis response cache, rate limiting, and metrics), meaning every request unconditionally serializes and transmits the full OpenAPI YAML payload with no caching, no rate limiting, and no connection throttle. An unprivileged external attacker can flood this endpoint with concurrent requests to exhaust server bandwidth and saturate the Node.js event loop.

### Finding Description

**Exact code path:**

`serveSpec` is a one-liner that always sends the full YAML string:

```js
// rest/middleware/openapiHandler.js, line 126
const serveSpec = (req, res) => res.type('text/yaml').send(getV1OpenApiFile());
``` [1](#0-0) 

It is registered in `serveSwaggerDocs` and called at **line 62** of `server.js`, before any middleware is attached:

```js
serveSwaggerDocs(app);          // line 62 — registers serveSpec routes
// ...
app.use(express.json());        // line 73
app.useExt(requestLogger);      // line 83
app.useExt(authHandler);        // line 86
app.useExt(responseCacheCheckHandler); // line 97
``` [2](#0-1) 

Because Express processes routes in registration order, the two `serveSpec` routes (`/api/v1/docs/openapi.yml` and `/api/v1/docs/openapi.yaml`) are matched and responded to **before** the middleware chain runs. This means:

- The **Redis response cache** (`responseCacheCheckHandler`) never fires for these routes — every request pays the full serialization cost.
- No **rate limiting** exists anywhere in the REST Node.js layer (confirmed: no `express-rate-limit` or equivalent dependency found).
- No `Cache-Control`, `ETag`, or `Expires` headers are set by `serveSpec`, so HTTP clients cannot cache the response either.

`getV1OpenApiFile()` does cache the file content in memory after the first read:

```js
const getV1OpenApiFile = (apiVersion) => {
  if (isUndefined(v1OpenApiFile)) {
    v1OpenApiFile = fs.readFileSync(openApiSpecPath, 'utf8');
  }
  return v1OpenApiFile;
};
``` [3](#0-2) 

So disk I/O is not repeated, but the full YAML string is re-serialized and transmitted over the network on **every single request** with no throttle.

**Root cause / failed assumption:** The developer assumed that serving a cached in-memory string is cheap enough to be safe without rate limiting. The failed assumption is that bandwidth and event-loop saturation from concurrent large-payload responses are negligible.

### Impact Explanation

The OpenAPI YAML for a comprehensive mirror-node REST API is typically 50–300 KB. With no rate limiting and no HTTP caching headers, an attacker sending, e.g., 1,000 concurrent requests/second forces the server to transmit 50–300 MB/s of data from a single endpoint. This can:

1. **Exhaust outbound bandwidth**, degrading or blocking all other API responses.
2. **Saturate the Node.js event loop** with response-write callbacks, increasing latency for all other endpoints.
3. **Prevent metrics collection** (the metrics middleware is also bypassed), making the attack invisible to monitoring.

Severity: **Medium** — no data is leaked and no DB is touched, but availability of the entire REST API is impacted.

### Likelihood Explanation

- **No authentication required** — any internet-accessible client can trigger this.
- **No rate limiting** at the application layer for the REST API (confirmed absent in codebase).
- **Trivially scriptable**: a simple `ab -n 100000 -c 500 https://host/api/v1/docs/openapi.yml` suffices.
- The Rosetta component has Traefik-level rate limiting configured, but the REST Node.js service does not. [4](#0-3) 

### Recommendation

1. **Add HTTP caching headers** in `serveSpec` so clients and CDNs cache the static response:
   ```js
   const serveSpec = (req, res) =>
     res.type('text/yaml')
        .set('Cache-Control', 'public, max-age=3600')
        .set('ETag', specEtag)   // pre-computed hash of v1OpenApiFile
        .send(getV1OpenApiFile());
   ```
2. **Move `serveSwaggerDocs(app)` after the middleware registration block** in `server.js` so the response cache and future rate-limiting middleware apply to these routes.
3. **Add application-level rate limiting** (e.g., `express-rate-limit`) for all routes, including the spec endpoints, consistent with the Traefik-level limits already applied to Rosetta.

### Proof of Concept

```bash
# Flood the spec endpoint with 500 concurrent connections, no auth needed
ab -n 50000 -c 500 http://<mirror-node-host>:5551/api/v1/docs/openapi.yml

# Observe: other API endpoints (e.g., /api/v1/transactions) become slow or
# time out while the flood is in progress. No 429 or throttle response is
# ever returned by the server.
```
