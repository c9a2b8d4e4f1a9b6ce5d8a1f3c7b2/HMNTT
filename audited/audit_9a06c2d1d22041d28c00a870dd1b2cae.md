### Title
Thundering Herd via Misplaced `sync.Once` in `initGenesisRecordFile` Exhausts DB Connection Pool

### Summary
In `rosetta/app/persistence/block.go`, `initGenesisRecordFile()` uses `sync.Once` only to protect the *write* to `br.genesisBlock`, not the expensive `selectGenesis` DB query that precedes it. Any number of concurrent unauthenticated HTTP requests arriving before genesis is initialized will all pass the unsynchronized early-return check and simultaneously execute the complex CTE query, exhausting the DB connection pool and causing a non-network-based DoS.

### Finding Description
**File:** `rosetta/app/persistence/block.go`, function `initGenesisRecordFile`, lines 243–263.

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    // LINE 244: unsynchronized read — N goroutines all pass this simultaneously
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil
    }

    db, cancel := br.dbClient.GetDbWithContext(ctx)   // each goroutine acquires a pool connection
    defer cancel()

    var rb recordBlock
    // LINE 252-254: ALL N goroutines execute this expensive CTE query
    if err := db.Raw(selectGenesis, sql.Named("treasury_entity_id", br.treasuryEntityId.EncodedId)).
        First(&rb).Error; err != nil {
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
    }

    // LINE 257-259: once.Do only protects the assignment, NOT the query above
    br.once.Do(func() {
        br.genesisBlock = rb
    })
    ...
}
```

**Root cause:** `sync.Once` is placed *after* the expensive DB query. It only serializes the final assignment. The guard at line 244 is a plain unsynchronized struct-field read with no mutex, so N concurrent goroutines all observe `ConsensusStart == genesisConsensusStartUnset` and all proceed to acquire a DB connection and execute `selectGenesis`.

**`selectGenesis`** is a multi-table CTE joining `account_balance` and `record_file` with ordering — a non-trivial query on a production database.

**Call chain to public HTTP surface:**
- `POST /network/status` → `networkAPIService.NetworkStatus()` → `BaseService.RetrieveGenesis()` → `blockRepository.RetrieveGenesis()` → `initGenesisRecordFile()`
- Same path via `FindByHash`, `FindByIndex`, `FindByIdentifier`, `RetrieveLatest` — all call `initGenesisRecordFile`.

The Rosetta API is unauthenticated and publicly exposed on port 5700 (default). No credentials or special privileges are required.

**DB pool configuration** (`docs/configuration.md` line 660, `rosetta/app/db/db.go` line 33):
```
hiero.mirror.rosetta.db.pool.maxOpenConnections = 100  (default)
```
With 100 concurrent requests, every pool connection is consumed executing `selectGenesis`. Subsequent legitimate requests block waiting for a free connection, and the DB server is hit with 100 simultaneous complex CTE queries.

### Impact Explanation
- DB connection pool (default 100) is fully consumed during the attack window.
- All other repository operations (`RetrieveLatest`, `FindByIndex`, etc.) that also need a DB connection are blocked or time out.
- The DB server itself may be overwhelmed by 100 concurrent CTE queries against large `account_balance` / `record_file` tables.
- Result: the Rosetta API becomes unresponsive for all users until the pool drains, which may not happen if the attacker sustains the flood.
- The Hiero Mirror Node Rosetta API is used by exchanges and infrastructure providers for Hedera (HBAR), which has significant market capitalization, making availability critical.

### Likelihood Explanation
- **No authentication required.** Any internet-accessible Rosetta node is vulnerable.
- **Startup window is predictable and repeatable.** The window exists on every node restart. An attacker monitoring the node (e.g., via `/network/status` returning `ErrNodeIsStarting`) knows exactly when to strike.
- **Trivially scriptable.** Sending 100–200 concurrent HTTP POST requests to `/network/status` requires only `curl` or `ab` (Apache Bench).
- **Repeatable if node is forced to restart** (e.g., via a separate crash-inducing bug or OOM).

### Recommendation
Move the entire DB query inside `once.Do`. Because `once.Do` prevents retrying on failure, use a separate mutex-based pattern that allows retrying when genesis is not yet available but prevents thundering herd once it succeeds:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    br.mu.Lock()
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        br.mu.Unlock()
        return nil
    }
    br.mu.Unlock()

    // Only one goroutine executes the query at a time
    br.initMu.Lock()
    defer br.initMu.Unlock()

    // Re-check under lock (double-checked locking)
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil
    }

    db, cancel := br.dbClient.GetDbWithContext(ctx)
    defer cancel()

    var rb recordBlock
    if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
    }

    br.mu.Lock()
    br.genesisBlock = rb
    br.mu.Unlock()
    return nil
}
```

Alternatively, use `sync.Once` correctly by wrapping the entire query inside it, and handle the retry-on-failure case with a separate flag protected by a mutex.

### Proof of Concept
**Preconditions:** Rosetta node is starting up (genesis not yet initialized, i.e., `account_balance` / `record_file` tables are being populated).

**Steps:**
1. Monitor `/network/status` until it returns `ErrNodeIsStarting` (confirming genesis is uninitialized).
2. Immediately send 100+ concurrent POST requests:
   ```bash
   seq 1 150 | xargs -P150 -I{} curl -s -X POST http://<rosetta-host>:5700/network/status \
     -H "Content-Type: application/json" \
     -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}'
   ```
3. **Result:** All 150 goroutines pass the unsynchronized check at line 244, each acquires a DB connection, and all execute `selectGenesis` simultaneously. The 100-connection pool is exhausted. Subsequent legitimate requests time out or receive DB errors. The DB server is hit with 150 concurrent CTE queries.
4. **Verification:** DB monitoring shows 100+ active queries all running `selectGenesis`; application logs show connection wait timeouts for other operations.
