### Title
Unbounded `data` Field Logged in Full Causes Log-Based Denial-of-Service via LOG-Action Filter

### Summary
The `ContractCallRequest.data` field has no `maxLength` constraint, allowing an attacker to submit arbitrarily large hex payloads. When a `RequestProperties` filter with `action = LOG` matches the request, `ThrottleManagerImpl.action()` calls `log.info("{}", request)`, serializing the entire request object â€” including the unbounded `data` field â€” into the log. Combined with the default rate limit of 500 req/s, this enables rapid disk exhaustion or log-buffer saturation with no privilege required.

### Finding Description

**Exact code path:**

`ThrottleManagerImpl.action()` at line 68: [1](#0-0) 

```java
case LOG -> log.info("{}", request);
```

`request` is a `ContractCallRequest` whose `data` field is annotated only with `@Hex` â€” no `maxLength`: [2](#0-1) 

Compare with `from` and `to`, which explicitly cap size: [3](#0-2) 

The `@Data` Lombok annotation generates a `toString()` that includes all fields, so `log.info("{}", request)` emits the full `data` value verbatim. [4](#0-3) 

**Root cause:** Missing `maxLength` on `@Hex` for the `data` field, combined with unconditional full-object logging in the LOG action branch.

**Exploit flow:**
1. Operator configures a `RequestProperties` with `action = LOG` (the default `ActionType`) and a broad or empty filter list. With `filters.isEmpty()` the filter matches every request. [5](#0-4) 
2. Attacker sends POST requests to the `/api/v1/contracts/call` endpoint with a `data` field containing, e.g., 2 MB of valid hex characters.
3. Each request passes `@Hex` validation (no size limit), passes the rate-limit bucket (1 token consumed), and reaches `action()`.
4. `log.info("{}", request)` writes ~2 MB per log entry.
5. At the default 500 req/s cap, this produces ~1 GB/s of log output.

### Impact Explanation
Disk space is exhausted within seconds to minutes depending on available storage, causing the JVM's log appender to block or throw, degrading or crashing the web3 service. Log-shipping pipelines (Fluentd, Logstash) and SIEM ingestion are also overwhelmed. Severity: **High** â€” full service disruption with no authentication required.

### Likelihood Explanation
The LOG action is the default `ActionType` in `RequestProperties`: [6](#0-5) 

Any operator who enables request-filter logging for auditing or debugging (a common operational practice) exposes this path. The attacker needs only a network connection and knowledge of the public API. The attack is trivially repeatable and scriptable.

### Recommendation
1. **Add `maxLength` to the `@Hex` annotation on `data`** â€” e.g., `@Hex(maxLength = 131072)` (64 KB is already generous for calldata): [2](#0-1) 
2. **Truncate before logging** â€” in the LOG branch, log a sanitized/truncated summary rather than the full object (e.g., log only `from`, `to`, `gas`, and `data.length()`).
3. **Configure a global HTTP request body size limit** (e.g., `spring.servlet.multipart.max-request-size` or Tomcat's `maxPostSize`) to reject oversized payloads at the servlet layer before they reach business logic.
