### Title
Unbounded Wide-Range Query DoS via Missing Rate Limiting in `getPendingAirdrops()`

### Summary
The `getPendingAirdrops()` endpoint in `TokenAirdropsController` accepts `serialNumbers` and `tokenIds` range parameters with no validation on range width and no rate limiting in the `rest-java` module. An unauthenticated attacker can submit requests with maximally wide `NumberRangeParameter` values (`GTE:0` + `LTE:9223372036854775807`) combined with wide `tokenIds` ranges, causing the `JooqRepository.getBoundConditions()` logic to generate complex three-part OR conditions that are expensive to execute. With no per-IP or global rate limiter on this path, repeated flooding degrades database and mirror node processing capacity.

### Finding Description

**Code path:**

`TokenAirdropsController.getPendingAirdrops()` (lines 77–86) accepts `@Size(max = 2) NumberRangeParameter[] serialNumbers` and `@Size(max = 2) EntityIdRangeParameter[] tokenIds` with no constraint on range width. [1](#0-0) 

`NumberRangeParameter.getNumberValue()` only rejects negative values; `Long.MAX_VALUE` (9223372036854775807) is accepted as a valid upper bound. [2](#0-1) 

`Bound.adjustUpperBound()` returns `Long.MAX_VALUE` when no upper is set, and accepts `LTE:9223372036854775807` as a legitimate upper bound, producing a range spanning the entire positive `long` space. [3](#0-2) 

`JooqRepository.getBoundConditions()` generates a three-part OR condition (lower outer + middle + upper outer) for multi-bound queries. With `serialNumbers GTE:0 / LTE:MAX` and wide `tokenIds`, this produces a complex nested OR that the query planner may not resolve via index seeks, forcing a scan of all rows for the target account. [4](#0-3) 

`TokenAirdropRepositoryCustomImpl.findAll()` executes this condition directly against `TOKEN_AIRDROP` with only a `LIMIT` clause; no query timeout or cost guard is present. [5](#0-4) 

**No rate limiting exists in `rest-java`:** The `ThrottleManager` / `ThrottleManagerImpl` (bucket4j-based) lives exclusively in the `web3` module and is not wired into `rest-java` controllers. [6](#0-5) 

`Constants.MAX_LIMIT = 100` caps the result set size but does not bound the cost of the underlying scan. [7](#0-6) 

### Impact Explanation
An attacker can continuously issue requests that each trigger an expensive multi-bound OR query against `TOKEN_AIRDROP`. Because the base condition fixes only `receiver_account_id = X` (the path variable), the scan is bounded per account but the complex OR structure can defeat index-only lookups. With no rate limiter, a single attacker from one IP can saturate the database connection pool and CPU, degrading mirror node query processing for all consumers. The scope matches the stated severity: sustained degradation of ≥30% of mirror node processing nodes without requiring any privileged access or brute-force credential attack.

### Likelihood Explanation
The endpoint is fully public — no authentication, no API key, no IP-based throttle. The exploit requires only knowledge of the REST API (publicly documented) and the ability to send HTTP GET requests in a loop. Any script-kiddie with `curl` or a simple Python loop can execute this. The parameters are standard query strings; no special encoding or protocol knowledge is needed. The attack is trivially repeatable and automatable.

### Recommendation
1. **Add rate limiting to `rest-java`**: Apply a bucket4j or Spring-based rate limiter (mirroring the `web3` `ThrottleManagerImpl` pattern) to all `/api/v1/accounts/{id}/airdrops/*` endpoints, enforcing per-IP and global request-per-second limits.
2. **Validate range width**: In `Bound` or `NumberRangeParameter`, reject ranges whose span exceeds a configurable maximum (e.g., reject `serialNumber` ranges wider than a few thousand).
3. **Set DB statement timeout**: Configure a PostgreSQL `statement_timeout` for the `rest-java` datasource so runaway queries are killed server-side.
4. **Add query cost guard**: Enforce a maximum on the combined cardinality of range parameters before executing the query.

### Proof of Concept
```bash
# No authentication required
# Flood with maximally wide serialNumber + tokenId ranges
while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/airdrops/pending\
?serialnumber=gte%3A0\
&serialnumber=lte%3A9223372036854775807\
&token.id=gte%3A0.0.1\
&token.id=lte%3A9223372036854775807\
&limit=100" &
done
```

Each request causes `getBoundConditions()` to emit a three-part OR SQL condition spanning the full `serial_number` and `token_id` ranges for the target account. With no rate limiter in `rest-java`, hundreds of concurrent requests exhaust the DB connection pool and degrade mirror node processing.

    private Condition getBoundConditions(List<Bound> bounds, boolean lowerProcessed, boolean upperProcessed) {
        if (bounds == null || bounds.isEmpty()) {
            return noCondition();
        }

        var primary = bounds.getFirst();
        if (bounds.size() == 1) {
            return getCondition(primary);
        }

        var secondaryBounds = bounds.subList(1, bounds.size());
        if (!lowerProcessed) {
            for (var bound : secondaryBounds) {
                // Only secondary bounds should be adjusted
                bound.adjustUpperRange();
            }
        }

        // Lower conditions need to be discovered before upper conditions because the methods involved update the
        // primary bound
        var lowerCondition = getOuterCondition(primary, secondaryBounds, false, lowerProcessed);
        var middleCondition = getMiddleCondition(primary, secondaryBounds);
        var upperCondition = getOuterCondition(primary, secondaryBounds, true, upperProcessed);

        return lowerCondition.or(middleCondition).or(upperCondition);
    }

    public Collection<TokenAirdrop> findAll(TokenAirdropRequest request, EntityId accountId) {
        var type = request.getType();
        var bounds = request.getBounds();

                .and(getBoundConditions(bounds))
                .and(TOKEN_AIRDROP.STATE.eq(AirdropState.PENDING));

                .selectFrom(TOKEN_AIRDROP)
                .where(condition)
                .orderBy(order)
                .limit(request.getLimit())

    @Qualifier(GAS_LIMIT_BUCKET)

    private final Bucket rateLimitBucket;

    @Qualifier(OPCODE_RATE_LIMIT_BUCKET)
    private final Bucket opcodeRateLimitBucket;

    private final ThrottleProperties throttleProperties;

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
