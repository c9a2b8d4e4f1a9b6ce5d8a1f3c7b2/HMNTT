### Title
Unauthenticated Hash-Mismatch Requests to `/block` Cause Unbounded DB Queries with No Negative-Result Caching

### Summary
The `Block()` handler in `rosetta/app/services/block_service.go` accepts a `PartialBlockIdentifier` with both `Index` and `Hash` set, routing to `FindByIdentifier()` which unconditionally executes a full database query against the `record_file` table for every request. Because there is no negative-result caching and no application-level rate limiting (the Traefik middleware is disabled by default), an unprivileged attacker can flood the endpoint with valid indices paired with random hashes, causing every request to hit the database and always fail, creating sustained DB overhead.

### Finding Description
**Exact code path:**

1. `Block()` (`rosetta/app/services/block_service.go:51`) calls `s.RetrieveBlock(ctx, request.BlockIdentifier)`.
2. `RetrieveBlock()` (`rosetta/app/services/base_service.go:79-81`): when both `bIdentifier.Hash != nil` and `bIdentifier.Index != nil`, it calls `b.blockRepo.FindByIdentifier(ctx, *bIdentifier.Index, h)`.
3. `FindByIdentifier()` (`rosetta/app/persistence/block.go:147-169`): the only guard is `index < 0 || hash == ""` (line 151). A valid non-negative index with any non-empty string passes this check.
4. After `initGenesisRecordFile` (cached after first call via `sync.Once`), it calls `br.findBlockByHash(ctx, hash)` (line 159).
5. `findBlockByHash()` (`rosetta/app/persistence/block.go:226-241`) executes the raw SQL query `selectByHashWithIndex` (line 231): `SELECT ... FROM record_file p WHERE hash = @hash COLLATE "C"`. A random hash never matches, so `gorm.ErrRecordNotFound` is returned → `ErrBlockNotFound`.

**Root cause:** `FindByIdentifier` always resolves the block by hash first (DB query), then validates the index. There is no cache for negative (not-found) hash lookups. Every distinct random hash string triggers a full DB round-trip.

**Why existing checks fail:**
- The `index < 0 || hash == ""` guard (line 151) only rejects obviously invalid inputs; a valid index with a random 64-hex-char string passes cleanly.
- The Traefik middleware chain (circuit breaker, `inFlightReq`, `rateLimit`) is gated on `{{ if and .Values.global.middleware .Values.middleware }}` (`charts/hedera-mirror-rosetta/templates/middleware.yaml:3`), and `global.middleware` defaults to `false` (`charts/hedera-mirror-rosetta/values.yaml:95`), so the middleware is **not deployed by default**.
- There is no application-level rate limiting in the Go service itself.

### Impact Explanation
Each malicious request causes one SQL query against `record_file` with a hash that will never match. At scale, this creates sustained read pressure on the database. Since the Rosetta API is a public-facing HTTP endpoint with no authentication, the attacker can drive arbitrarily high query rates. The impact is DB resource exhaustion and potential degradation of legitimate block-query performance. This aligns with the stated severity: griefing/DoS with no direct economic damage to network users, but service availability impact.

### Likelihood Explanation
The attack requires zero privileges — only HTTP access to the Rosetta `/block` endpoint. The payload is trivial: a JSON body with a known valid block index (obtainable from `/block` with no hash) and a random 32-byte hex string as the hash. The attack is fully repeatable, stateless, and parallelizable across many source IPs. Default deployments without the Traefik middleware enabled are fully exposed.

### Recommendation
1. **Add application-level rate limiting** in the Go service (e.g., `golang.org/x/time/rate` per-IP token bucket) independent of the optional Traefik middleware.
2. **Cache negative hash-lookup results** with a short TTL (e.g., 5–10 seconds) in `findBlockByHash` or `FindByIdentifier` to avoid repeated DB hits for the same invalid hash.
3. **Enable the Traefik middleware by default** (`global.middleware: true`) or document it as a required security configuration.
4. **Validate hash format** before querying: reject inputs that are not valid 48-byte (96 hex char) SHA-384 hashes (the format used by Hedera record file hashes) at the service layer, before any DB call.

### Proof of Concept
```bash
# Step 1: Obtain a valid block index (no hash needed)
curl -s -X POST http://<rosetta-host>/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{}}' \
  | jq '.block.block_identifier.index'
# Returns e.g. 12345

# Step 2: Flood with valid index + random hash
while true; do
  curl -s -X POST http://<rosetta-host>/block \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":12345,\"hash\":\"$(openssl rand -hex 48)\"}}" &
done
```

func (br *blockRepository) FindByIdentifier(ctx context.Context, index int64, hash string) (
	*types.Block,
	*rTypes.Error,
) {
	if index < 0 || hash == "" {
		return nil, hErrors.ErrInvalidArgument
	}

	if err := br.initGenesisRecordFile(ctx); err != nil {
		return nil, err
	}

	block, err := br.findBlockByHash(ctx, hash)
	if err != nil {
		return nil, err
	}

	if block.Index != index {
		return nil, hErrors.ErrBlockNotFound
	}

	return block, nil
}

func (br *blockRepository) findBlockByHash(ctx context.Context, hash string) (*types.Block, *rTypes.Error) {
	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectByHashWithIndex, sql.Named("hash", hash)).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	if rb.Index < br.genesisBlock.Index {
		log.Errorf("The block with hash %s is before the genesis block", hash)
		return nil, hErrors.ErrBlockNotFound
	}

	return rb.ToBlock(br.genesisBlock), nil
}
