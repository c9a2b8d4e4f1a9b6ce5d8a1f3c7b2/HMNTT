### Title
Unauthenticated EVM Address Flooding Causes Unbounded DB Queries via Missing Negative-Result Cache in `getEntityIdFromEvmAddress()`

### Summary
The `getAccountCryptoAllowances()` handler in `cryptoAllowanceController.js` accepts any valid EVM address as a path parameter and unconditionally issues a database query for each one via `EntityService.getEntityIdFromEvmAddress()`. There is no negative-result caching and no rate limiting on the REST API, so an unauthenticated attacker can flood the endpoint with distinct non-existent EVM addresses, generating one uncached DB query per request and saturating the mirror node's database I/O, causing denial of service to the mirror node REST API.

### Finding Description

**Exact code path:**

`cryptoAllowanceController.js` line 77 calls `EntityService.getEncodedId()` with the raw path parameter: [1](#0-0) 

`getEncodedId()` in `entityService.js` lines 120–124 detects a valid EVM address and routes to `getEntityIdFromEvmAddress()`: [2](#0-1) 

`getEntityIdFromEvmAddress()` lines 90–104 executes a raw DB query with no caching layer. When 0 rows are returned it throws `NotFoundError` immediately: [3](#0-2) 

The SQL query used: [4](#0-3) 

**Root cause — two missing defenses:**

1. **No negative-result caching.** The `parseCached` LRU in `entityId.js` only caches the parsed `EntityId` struct (format validation), not the DB lookup result. Every call to `getEntityIdFromEvmAddress()` always hits the database regardless of whether the same address was queried before: [5](#0-4) 

2. **No rate limiting on the REST API.** A `grep` across all `rest/**/*.js` files for `rateLimit`, `throttle`, or `maxRequests` returns zero matches. `server.js` registers only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optional response cache middleware — no request-rate limiter: [6](#0-5) 

The throttling found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3` Java service and does not apply here: [7](#0-6) 

### Impact Explanation

An attacker can generate an unbounded stream of DB queries against the `entity` table by sending requests to `/api/v1/accounts/{evmAddress}/allowances/crypto` with distinct, syntactically valid but non-existent 40-hex-char EVM addresses. Each request bypasses all caches and issues a full table scan (or index scan) on `entity.evm_address`. Under sustained flood, this saturates the PostgreSQL connection pool and disk I/O, degrading or denying service to the mirror node REST API for all consumers. **Note:** the mirror node is a read-only service and does not participate in Hedera consensus; this is a DoS against the mirror node REST API, not a shutdown of the Hedera network itself. The impact is loss of availability of the mirror node.

### Likelihood Explanation

No authentication, no rate limiting, and no CAPTCHA are required. Any internet-accessible attacker with a script generating random 40-hex-char strings prefixed with `0x` can trigger this. The attack is trivially repeatable, parallelizable across multiple source IPs, and requires no special knowledge of the system. The `evmAddressRegex` validation (`/^(0x)?[A-Fa-f0-9]{40}$/`) is the only gate, and it is trivially satisfied: [8](#0-7) 

### Recommendation

Apply at least two of the following mitigations:

1. **Negative-result caching:** Cache `null` results from `getEntityIdFromEvmAddress()` with a short TTL (e.g., 30 s) keyed on the hex EVM address. This prevents repeated DB hits for the same non-existent address.
2. **Rate limiting:** Add a per-IP rate-limiting middleware (e.g., `express-rate-limit`) to the REST API server, applied globally or specifically to EVM-address path parameters.
3. **Infrastructure-level throttling:** Apply Traefik/nginx rate limiting at the ingress layer for the REST API (similar to what is already configured for the Rosetta service). [9](#0-8) 

### Proof of Concept

```bash
# Generate and flood with distinct non-existent EVM addresses
for i in $(seq 1 10000); do
  ADDR=$(python3 -c "import secrets; print('0x' + secrets.token_hex(20))")
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ADDR}/allowances/crypto" &
done
wait
```

Each request passes format validation, reaches `getEntityIdFromEvmAddress()`, executes a DB query against the `entity` table, finds 0 rows, and throws `NotFoundError` — with no caching of the miss and no rate limiting to slow the flood. Under sustained load this exhausts the DB connection pool and degrades mirror node REST API availability for all users.
