### Title
Thundering Herd via Unjittered `delaySubscription` in `PollingTopicListener.listen()` Enables Synchronized DB Connection Pool Exhaustion

### Summary
`PollingTopicListener.listen()` applies `delaySubscription(interval, scheduler)` with a fixed, unjittered delay before the first database poll. When many unprivileged clients open subscriptions simultaneously, all first-poll DB queries fire at exactly the same instant after `interval` elapses. The `jitter(0.1)` present in `repeatWhen` only applies to subsequent polls, leaving the initial burst completely unsynchronized, which can exhaust the HikariCP connection pool and cause timeouts across all active subscribers.

### Finding Description

**Exact code location:** `grpc/src/main/java/org/hiero/mirror/grpc/listener/PollingTopicListener.java`, `listen()`, lines 38–43.

```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)          // line 39 — fixed delay, NO jitter
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)                             // line 41 — jitter only on repeat
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

**Root cause:** The `delaySubscription` operator uses the raw `interval` value (default 500 ms, minimum 50 ms per `@DurationMin`) with no randomization. The `jitter(0.1)` in `RepeatSpec` is only applied to the `repeatWhen` cycle — i.e., the second and subsequent polls — and has no effect on the initial subscription delay. The failed assumption is that the initial delay provides natural spread; it does not when clients connect concurrently.

**Exploit flow:**
1. Attacker opens N gRPC streams to `ConsensusService/subscribeTopic` simultaneously (no authentication required per the public proto definition and `ConsensusController`).
2. Each call reaches `TopicMessageServiceImpl.subscribeTopic()` → `incomingMessages()` → `topicListener.listen(newFilter)` (when `type=POLL`).
3. Each `PollingTopicListener.listen()` invocation schedules its first `poll()` call at `T₀ + interval` on the shared `boundedElastic` scheduler.
4. At `T₀ + 500 ms`, all N `poll()` calls execute concurrently, each invoking `topicMessageRepository.findByFilter(newFilter)` — a full SQL query against the `topic_message` table.
5. HikariCP's default pool (10 connections) is saturated; excess queries queue and time out, denying service to all subscribers including legitimate ones.

**Why existing checks fail:**
- `jitter(0.1)` on `repeatWhen` — does not touch `delaySubscription`; only kicks in from the second poll onward.
- `maxPageSize = 5000` — limits rows returned per query, not the number of concurrent queries.
- `Schedulers.boundedElastic()` — limits threads (default: 10 × CPU cores), but each task still issues a blocking JDBC call that holds a DB connection for its duration.
- No per-client subscription rate limit or connection cap exists in the gRPC module (`ThrottleConfiguration` is scoped to `web3`, not `grpc`).
- `subscriberCount` gauge is metrics-only; it enforces no ceiling.

### Impact Explanation
When `hiero.mirror.grpc.listener.type=POLL` is configured, a single attacker with a scripted client can trigger a synchronized burst of hundreds of DB queries in a 1–2 ms window. HikariCP's default pool of 10 connections is exhausted instantly; all waiting queries time out. This denies topic-message streaming to every subscriber on the affected mirror node, satisfying the "≥30% of network processing nodes" DoS threshold if multiple nodes share the same database or if the attacker targets multiple nodes in parallel. The attack is stateless and requires no credentials.

### Likelihood Explanation
The `subscribeTopic` endpoint is unauthenticated and publicly documented (`grpcurl` example in `docs/grpc/README.md`). Opening hundreds of concurrent gRPC streams is trivial with standard tooling (e.g., `ghz`, a simple Go/Python script). The attack is repeatable: after the initial burst, the attacker can disconnect and reconnect to trigger a new synchronized burst. The only prerequisite is that the operator has selected `type=POLL` instead of the default `REDIS`; this is a realistic configuration when Redis is unavailable or cost-prohibitive.

### Recommendation
Add jitter to the initial subscription delay so that concurrent subscribers are naturally spread across the interval window:

```java
Duration jitteredDelay = interval.plus(
    Duration.ofMillis((long)(Math.random() * interval.toMillis()))
);
return Flux.defer(() -> poll(context))
        .delaySubscription(jitteredDelay, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))
```

Additionally: (1) enforce a configurable maximum concurrent subscriber count with a server-side semaphore or `flatMap` concurrency cap; (2) apply a per-IP or global gRPC connection rate limit in the `ConsensusController`; (3) set an explicit HikariCP `maximumPoolSize` and `connectionTimeout` in the grpc module's configuration to fail fast rather than queue indefinitely.

### Proof of Concept

```python
import grpc, threading, time
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2

TARGET = "mirror-node-host:5600"
N = 200  # simultaneous subscribers

def subscribe():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=41110)
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass
    except Exception:
        pass

threads = [threading.Thread(target=subscribe) for _ in range(N)]
# Launch all threads simultaneously to align subscription start times
for t in threads:
    t.start()
# After ~500 ms (default interval), all N threads fire their first DB query
# concurrently, exhausting the connection pool and causing timeouts
for t in threads:
    t.join()
```

Expected result: DB connection pool exhaustion logged by HikariCP (`Connection is not available, request timed out`), all active subscribers receive gRPC errors, mirror node becomes unresponsive to new topic subscriptions for the duration of the attack.
