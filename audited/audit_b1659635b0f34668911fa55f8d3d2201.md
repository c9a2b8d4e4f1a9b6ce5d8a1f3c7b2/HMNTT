### Title
Unrestricted `ne` Operator on `ACCOUNT_ID` Filter Enables Sequential Scan Griefing

### Summary
The `filterValidityChecks` function in `rest/utils.js` validates the `account.id` query parameter by checking only the value format, not the operator. This allows any unprivileged user to supply `account.id=ne:0.0.1`, which generates a `account_id != ?` SQL predicate that cannot use a B-tree index, forcing PostgreSQL to perform a sequential scan on large tables. Repeated requests can degrade query performance for all other users with no financial cost to the attacker.

### Finding Description

**Code path:**

In `rest/utils.js`, `basicOperators` is explicitly defined to exclude `ne`: [1](#0-0) 

This exclusion is correctly applied to `BLOCK_NUMBER` and `SLOT`: [2](#0-1) [3](#0-2) 

However, the `ACCOUNT_ID` case in `filterValidityChecks` only validates the value, never the operator: [4](#0-3) 

The `opsMap` maps `ne` directly to the SQL `!=` operator: [5](#0-4) 

`parseAccountIdQueryParam` then injects this operator verbatim into the SQL clause: [6](#0-5) 

**Root cause:** The operator restriction pattern (`basicOperators`) was applied to `BLOCK_NUMBER` and `SLOT` but was omitted for `ACCOUNT_ID`. The `filterValidityChecks` switch case for `ACCOUNT_ID` performs no `op` check, so `ne` passes validation and reaches query construction.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/transactions?account.id=ne:0.0.1`
2. `filterValidityChecks` accepts it (value `0.0.1` is a valid entity ID; operator is not checked).
3. `parseAccountIdQueryParam` builds: `account_id != $1`.
4. PostgreSQL cannot use the B-tree index on `account_id` for a `!=` predicate and falls back to a sequential scan of the `crypto_transfer` or `transaction` table.
5. Attacker repeats at high frequency; each request saturates I/O and CPU, slowing concurrent indexed queries from legitimate users.

### Impact Explanation
Any endpoint accepting `account.id` as a filter (e.g., `/api/v1/transactions`, `/api/v1/balances`, NFT endpoints) is affected. On a production mirror node with millions of rows, a sequential scan on `crypto_transfer` or `transaction` can take seconds per query. Concurrent legitimate queries that rely on the same table will experience lock contention and I/O starvation. There is no financial loss, but service availability degrades for all users — consistent with the "griefing" severity tier.

### Likelihood Explanation
The attack requires zero authentication, zero tokens, and zero on-chain activity. The endpoint is publicly accessible. The attacker needs only a valid entity ID (e.g., `0.0.1`, which always exists) and the knowledge that `ne:` is an accepted operator prefix. The attack is trivially scriptable and repeatable at any rate the network allows.

### Recommendation
Apply the same `basicOperators` guard to `ACCOUNT_ID` that is already applied to `BLOCK_NUMBER` and `SLOT`:

```js
// rest/utils.js, filterValidityChecks
case constants.filterKeys.ACCOUNT_ID:
  ret = EntityId.isValidEntityId(val) && includes(basicOperators, op);
  break;
```

Since `basicOperators` already excludes `ne` by construction, this single-line change blocks the `!=` predicate while preserving `eq`, `lt`, `lte`, `gt`, `gte` for legitimate range queries. Apply the same fix to any other high-cardinality entity-ID fields (`SPENDER_ID`, `TOKEN_ID`, `SCHEDULE_ID`) that currently omit an operator check.

### Proof of Concept

```bash
# Step 1 – confirm the operator is accepted (HTTP 200, not 400)
curl -s -o /dev/null -w "%{http_code}" \
  "https://<mirror-node>/api/v1/transactions?account.id=ne:0.0.1"
# Expected: 200

# Step 2 – observe sequential scan in PostgreSQL
# On the mirror-node DB:
EXPLAIN ANALYZE
SELECT * FROM crypto_transfer
WHERE entity_id != 1
LIMIT 25;
# Expected plan: Seq Scan on crypto_transfer (not Index Scan)

# Step 3 – griefing loop
for i in $(seq 1 200); do
  curl -s -o /dev/null \
    "https://<mirror-node>/api/v1/transactions?account.id=ne:0.0.$(( RANDOM % 1000 + 1 ))" &
done
wait
# Result: concurrent legitimate queries slow significantly due to I/O saturation
```

const parseAccountIdQueryParam = (parsedQueryParams, columnName) => {
  return parseParams(
    parsedQueryParams[constants.filterKeys.ACCOUNT_ID],
    (value) => EntityId.parseString(value).getEncodedId(),
    (op, value) => {
      return Array.isArray(value)
        ? [`${columnName} IN (?`.concat(', ?'.repeat(value.length - 1)).concat(')'), value]
        : [`${columnName}${op}?`, [value]];
    },
    true
  );
};
