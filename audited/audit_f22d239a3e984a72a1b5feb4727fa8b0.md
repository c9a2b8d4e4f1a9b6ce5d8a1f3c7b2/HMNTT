### Title
Unauthenticated Griefing via Malformed Timestamp Scalar Causing Unhandled Exception Amplified by Retry Middleware

### Summary
`GraphQlTimestamp.parseLiteral()` and `parseValue()` call `Instant.parse()` without a try-catch, causing an uncaught `DateTimeParseException` for any malformed ISO 8601 string. The GraphQL endpoint has no rate limiting, and the Traefik `retry: attempts: 3` middleware silently re-submits each failed request three times to the backend, tripling the exception generation rate per attacker request. The circuit breaker does not trigger because GraphQL coercion errors return HTTP 200, not 5xx.

### Finding Description

**Exact code location:**

`graphql/src/main/java/org/hiero/mirror/graphql/scalar/GraphQlTimestamp.java`, lines 33–36 (`parseLiteral`) and lines 44–46 (`parseValue`):

```java
// parseLiteral — line 34
return Instant.parse(str.getValue());   // throws DateTimeParseException, uncaught

// parseValue — line 45
return Instant.parse(string);           // same issue
```

**Root cause:** Neither method wraps `Instant.parse()` in a try-catch. A malformed string (e.g., `"NOT_A_DATE"`) causes `java.time.format.DateTimeParseException` (a `RuntimeException`) to propagate uncaught out of the coercing method. The graphql-java framework catches it internally and converts it to a GraphQL error, but the exception object — including its full stack trace — is already allocated.

**Exploit flow:**

1. Attacker sends a POST to `/graphql/alpha` (no authentication required) with a query using the `Timestamp` scalar and a malformed value, e.g.:
   ```graphql
   { account(input: { entityId: { shard: 0, realm: 0, num: 1 }, timestamp: { from: "AAAA" } }) { balance } }
   ```
2. `parseLiteral` is called → `Instant.parse("AAAA")` throws `DateTimeParseException`.
3. Traefik's `retry: attempts: 3` middleware re-submits the request 3 more times (the backend returns a non-network error, but Traefik retries on any failure). Each retry triggers the same exception path.
4. The circuit breaker expression `ResponseCodeRatio(500, 600, 0, 600) > 0.25` never fires because GraphQL coercion errors produce HTTP 200 responses with an `errors` array — not 5xx.
5. The `inFlightReq: amount: 5` per-IP limit can be saturated from a single IP (5 concurrent × 3 retries = 15 backend exception invocations per IP), and bypassed entirely with multiple source IPs.
6. No application-level rate limiting exists in the GraphQL module (unlike `web3` which uses bucket4j `ThrottleConfiguration`).

**Why existing checks fail:**

| Check | Why it fails |
|---|---|
| `MaxQueryComplexityInstrumentation(200)` | Runs at query planning, before scalar coercion |
| `MaxQueryDepthInstrumentation(10)` | Same — pre-execution |
| `PreparsedDocumentProvider` cache | Caches the parsed AST; coercion runs at execution time, not parse time |
| `circuitBreaker` | Keyed on 5xx; GraphQL errors are HTTP 200 |
| `inFlightReq: 5` per IP | Bypassable with multiple IPs; retry multiplier still applies |
| `retry: attempts: 3` | **Amplifier**, not a mitigant — triples backend load per attacker request |

### Impact Explanation
An unprivileged attacker can sustain a stream of `DateTimeParseException` allocations at 3× the request rate due to the retry middleware. While the JVM's `-XX:+OmitStackTraceInFastThrow` optimization eventually suppresses stack trace filling after ~64 occurrences at the same site, the initial burst and any new JVM instances (e.g., after pod restarts or HPA scale-up) are fully exposed. Combined with no rate limiting and a publicly accessible endpoint, this degrades service for legitimate users — consistent with the "griefing, no economic damage" classification.

### Likelihood Explanation
No credentials, tokens, or special network access are required. The `/graphql/alpha` endpoint is publicly exposed via ingress. Any attacker with HTTP access and a script can trigger this. The retry amplification is automatic and requires no special knowledge. Repeatability is high.

### Recommendation

1. **Wrap `Instant.parse()` in both methods** with a proper try-catch and re-throw as the correct coercing exception:
   ```java
   // parseLiteral
   try {
       return Instant.parse(str.getValue());
   } catch (DateTimeParseException e) {
       throw new CoercingParseLiteralException("Invalid Timestamp value: " + str.getValue(), e);
   }
   ```
   Same pattern for `parseValue` → `CoercingParseValueException`.

2. **Add a `rateLimit` middleware** to the GraphQL Traefik chain (matching the Rosetta configuration: `rateLimit: average: 10, sourceCriterion: requestHost: true`).

3. **Remove or scope the `retry` middleware** — retrying on scalar coercion errors is counterproductive and amplifies load. Retries should only apply to network-level failures.

### Proof of Concept

```bash
# Single attacker IP, 5 concurrent streams (saturates inFlightReq per IP)
# Each request triggers 3 backend invocations via retry middleware

for i in $(seq 1 5); do
  while true; do
    curl -s -X POST http://<host>/graphql/alpha \
      -H 'Content-Type: application/json' \
      -d '{"query":"{ account(input:{entityId:{shard:0,realm:0,num:1},timestamp:{from:\"NOT_A_DATE\"}}) { balance } }"}' \
      > /dev/null
  done &
done
wait
```

Expected result: backend receives up to 15 exception-generating requests per second from a single IP; HTTP 200 responses with `errors` array prevent circuit breaker activation; no rate limiting terminates the flood.
