### Title
Global Single-Bucket Rate Limit Allows Unprivileged Client to Monopolize the Opcodes Endpoint (DoS)

### Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint uses a single, globally-shared `opcodeRateLimitBucket` with a default capacity of **1 request per second** across all clients. Because no per-IP or per-client isolation exists, any unauthenticated attacker sending one request per second continuously exhausts the entire global token budget, causing every other legitimate caller to receive a throttle error for the duration of the attack.

### Finding Description
**Code path:**

- `OpcodesController.java:61` calls `throttleManager.throttleOpcodeRequest()` for every incoming request.
- `ThrottleManagerImpl.java:52-56` implements this as a single `opcodeRateLimitBucket.tryConsume(1)` check with no caller identity involved.
- `ThrottleConfiguration.java:47-55` creates `opcodeRateLimitBucket` as a single Spring singleton `Bucket` bean, shared JVM-wide.
- `ThrottleProperties.java:29` sets the default capacity to `opcodeRequestsPerSecond = 1`.

**Root cause:** The bucket is a process-wide singleton. `tryConsume(1)` decrements the same counter regardless of which IP or client issued the request. The failed assumption is that a global limit is sufficient to protect the endpoint; in reality it provides no isolation between clients.

**Exploit flow:**
1. Attacker sends `GET /api/v1/contracts/results/<any_hash>/opcodes` with `Accept-Encoding: gzip` at a rate of 1 req/sec.
2. Each request consumes the single available token in `opcodeRateLimitBucket`.
3. Every concurrent or subsequent request from any other client finds the bucket empty and receives a `ThrottleException` (HTTP 429).
4. The bucket refills at 1 token/sec, which the attacker immediately re-consumes.

**Why existing checks fail:** The only guard is `opcodeRateLimitBucket.tryConsume(1)` — a binary global gate. There is no IP extraction, session tracking, or per-client sub-bucket. The `RequestProperties`-based per-request filters in `throttle()` are not applied to `throttleOpcodeRequest()` at all.

### Impact Explanation
The opcodes endpoint becomes completely unavailable to all legitimate users for as long as the attacker sustains 1 req/sec. Because the endpoint replays EVM transactions (computationally expensive), even a brief monopolization has outsized operational impact. Severity is **High** for availability; confidentiality and integrity are unaffected.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only 1 HTTP request per second — well within reach of any script or `curl` loop. The `Accept-Encoding: gzip` header requirement is trivially satisfied. The attack is fully repeatable and sustainable indefinitely with negligible attacker resources.

### Recommendation
Replace the single global bucket with a per-client (per-IP or per-authenticated-identity) bucket map, e.g. using Bucket4j's `BucketProxyManager` keyed on `request.getRemoteAddr()` or a resolved client identifier. Additionally, consider adding a separate, lower per-IP sub-limit inside `throttleOpcodeRequest()` so that even if the global cap is raised, no single client can consume a disproportionate share.

### Proof of Concept
```bash
# Terminal 1 – attacker (1 req/sec, consumes entire global budget)
while true; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Accept-Encoding: gzip" \
    "http://<host>/api/v1/contracts/results/0xdeadbeef.../opcodes"
  sleep 1
done

# Terminal 2 – legitimate user (receives 429 on every attempt)
curl -v -H "Accept-Encoding: gzip" \
  "http://<host>/api/v1/contracts/results/0xdeadbeef.../opcodes"
# Expected: HTTP 429 "Requests per second rate limit exceeded"
```

    Bucket opcodeRateLimitBucket() {

                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
