### Title
Cache Poisoning DoS via Unbounded Error Entry Caching in GraphQL PreparsedDocument Cache

### Summary
`CachedPreparsedDocumentProvider.getDocumentAsync()` caches all `PreparsedDocumentEntry` objects — including those representing parse/validation errors — without distinction, in a Caffeine cache bounded at `maximumSize=1000`. An unprivileged attacker can send 1000 unique malformed queries to fill the cache entirely with error entries, causing legitimate queries to be evicted and re-parsed on every subsequent request. No authentication or rate limiting exists on the GraphQL endpoint to prevent this.

### Finding Description
**Exact code path:**

`CachedPreparsedDocumentProvider.getDocumentAsync()` at line 26:
```java
return cache.get(executionInput.getQuery(), key -> parseAndValidateFunction.apply(executionInput));
``` [1](#0-0) 

The cache is configured as:
```
expireAfterWrite=1h,maximumSize=1000,recordStats
``` [2](#0-1) 

**Root cause:** The `cache.get(key, mappingFunction)` call unconditionally stores the result of `parseAndValidateFunction.apply(executionInput)` regardless of whether the returned `PreparsedDocumentEntry` contains errors (`hasErrors() == true`). There is no guard to skip caching of error entries.

**Failed assumption:** The design assumes the cache will be populated primarily by legitimate queries. In reality, any unique string within parser limits is a valid cache key, and error entries occupy slots identically to valid ones.

**Why existing checks fail:**

- Parser limits (`maxCharacters=10000`, `maxTokens=1000`) still permit thousands of syntactically distinct malformed queries (e.g., `{ f1 }`, `{ f2 }`, ..., `{ f1000 }` — each references a non-existent field, passes the parser, fails validation, and gets cached). [3](#0-2) 

- `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` run **after** parsing and caching, so they do not prevent cache pollution. [4](#0-3) 

- **No rate limiting exists in the GraphQL module.** The `ThrottleConfiguration` and `ThrottleManagerImpl` are scoped exclusively to the `web3` module and are not applied to GraphQL endpoints. [5](#0-4) 

**Caffeine W-TinyLFU interaction (makes attack more effective, not less):** Once the cache is full with 1000 malformed entries (each with access frequency ≥ 1), new legitimate queries arrive with frequency = 0. Under W-TinyLFU admission policy, a new item with lower estimated frequency than the eviction candidate is **rejected** from the cache. This means legitimate queries are not only evicted — they may be permanently excluded from the cache as long as the attacker refreshes the malformed entries.

### Impact Explanation
Every legitimate GraphQL query that is evicted from or rejected by the cache must go through the full parse-and-validate cycle on every request. Under sustained attack, this eliminates the performance benefit of the preparsed document cache entirely, increases CPU consumption proportionally to request volume, and degrades response latency for all users of the GraphQL API. Since the cache TTL is 1 hour, a single burst of 1000 unique malformed queries can sustain the degradation for up to 1 hour without any follow-up requests from the attacker.

### Likelihood Explanation
The attack requires zero privileges — the GraphQL endpoint is publicly accessible. Generating 1000 unique syntactically-valid but semantically-invalid queries is trivial (e.g., vary a single field name). The attacker needs to send only 1000 HTTP requests to saturate the cache, which can be done in seconds with any HTTP client. The attack is repeatable and automatable. No special knowledge of the system is required beyond knowing the GraphQL endpoint URL.

### Recommendation
In `CachedPreparsedDocumentProvider.getDocumentAsync()`, do not cache entries that contain errors. Replace the unconditional `cache.get(...)` with logic that checks `PreparsedDocumentEntry.hasErrors()`:

```java
@Override
public CompletableFuture<PreparsedDocumentEntry> getDocumentAsync(
        ExecutionInput executionInput, Function<ExecutionInput, PreparsedDocumentEntry> parseAndValidateFunction) {
    return cache.get(executionInput.getQuery(), key -> {
        PreparsedDocumentEntry entry = parseAndValidateFunction.apply(executionInput);
        if (entry.hasErrors()) {
            // Do not cache error entries; return but evict immediately
            cache.synchronous().invalidate(key);
        }
        return entry;
    });
}
```

Alternatively, use `cache.asMap().computeIfAbsent` with explicit error-entry rejection. Additionally, consider adding per-IP or global rate limiting to the GraphQL endpoint, consistent with the throttling already implemented in the `web3` module. [1](#0-0) 

### Proof of Concept
```python
import requests, string, random

GRAPHQL_URL = "http://<host>/graphql"

# Step 1: Fill cache with 1000 unique invalid queries
for i in range(1000):
    field = f"nonExistentField{i}"
    query = f"{{ {field} }}"
    requests.post(GRAPHQL_URL, json={"query": query})

# Step 2: Send a legitimate query — it will be re-parsed on every call
# because the cache is full of error entries with frequency >= 1,
# and the legitimate query (frequency=0) is rejected by W-TinyLFU
import time
for _ in range(100):
    start = time.time()
    r = requests.post(GRAPHQL_URL, json={"query": "{ account(input: {entityId: {shard:0,realm:0,num:1}}) { balance } }"})
    print(f"Latency: {time.time()-start:.3f}s")
    # Observe consistently higher latency vs. baseline (no cache pollution)
```

Baseline (warm cache): sub-millisecond parse overhead.
Under attack (cold/evicted cache): full parse+validate on every request, observable latency increase proportional to query complexity and server load.
