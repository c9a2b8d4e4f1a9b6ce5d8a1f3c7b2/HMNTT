### Title
Unbounded Historical Retrieval Stalls Live Message Delivery in `subscribeTopic()`

### Summary
An unprivileged external user can subscribe to any topic with `startTime=0` (epoch nanoseconds, i.e., January 1, 1970) and `limit=0` (no limit). Because the pipeline is constructed as `historical.concatWith(live)`, the live stream is never reached until the historical `Flux` completes. With no lower bound enforced on `startTime` and no mandatory cap on `limit`, a topic with a large message history causes the live stream to be indefinitely delayed, and multiple such connections exhaust server-side resources.

### Finding Description

**Exact code path:**

In `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, `subscribeTopic()` (lines 59–92):

```java
Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);   // line 63
Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));      // line 64
...
Flux<TopicMessage> flux = historical
    .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))  // line 72-73
```

`concatWith` is strictly sequential: `live` (and `safetyCheck`) only subscribe after `historical` terminates. If `historical` never terminates (or takes arbitrarily long), the live stream is never reached.

**Root cause — failed assumptions in `TopicMessageFilter`:**

In `grpc/src/main/java/org/hiero/mirror/grpc/domain/TopicMessageFilter.java`:

```java
@Min(0)
private long startTime = DomainUtils.now();   // @Min(0) allows epoch 0

@Min(0)
private long limit;                            // 0 means no limit

@AssertTrue(message = "Start time must be before the current time")
public boolean isValidStartTime() {
    return startTime <= DomainUtils.now();     // trivially satisfied by any past time
}
```

- `startTime` is validated only to be `<= now()` and `>= 0`. Setting it to `0` (nanoseconds since Unix epoch) is fully accepted.
- `limit = 0` means `hasLimit()` returns `false`, so no `flux.take()` is applied (line 83–85).
- There is no minimum `startTime` floor (e.g., "no more than N days in the past").

**Why the `throttled=true` parameter does not fix this:**

The `throttled` flag passed to `topicMessageRetriever.retrieve(filter, true)` controls the *rate* of polling (pacing between DB queries), not the *total* number of messages returned. Throttling actually worsens the stall: it slows down the historical retrieval, keeping the live stream blocked even longer per connection.

**Why the `safetyCheck` does not fix this:**

```java
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
    .filter(_ -> !topicContext.isComplete())
    .flatMapMany(_ -> missingMessages(topicContext, null))
    .subscribeOn(Schedulers.boundedElastic());
```

`safetyCheck` is merged with `live` inside `Flux.merge(safetyCheck, live)`, which is itself the argument to `historical.concatWith(...)`. It does not run until `historical` completes. The `safetyCheck` is designed for gap recovery *after* the historical phase, not as a bypass of it.

### Impact Explanation

1. **Live message delay / denial of service per subscriber:** Any subscriber with `startTime=0` on a topic with years of history will never receive live messages until the full historical backlog is streamed. For a topic with millions of messages, this can take minutes to hours even with throttling.
2. **Server resource exhaustion:** Each such subscription holds a DB cursor/connection, a Reactor pipeline, a `boundedElastic` thread for `safetyCheck`, and memory for buffered messages. An attacker opening many such connections simultaneously can exhaust the gRPC thread pool, DB connection pool, and heap.
3. **No authentication required:** The gRPC `subscribeTopic` endpoint is publicly accessible to any client; no privilege is needed.

### Likelihood Explanation

- Any external client with network access to the gRPC port can trigger this.
- The exploit requires only a valid `topicId` (which is publicly discoverable via the REST API or mirror node explorer) and setting `startTime=0`, `limit=0`.
- It is trivially scriptable and repeatable. A single attacker with a modest number of parallel gRPC connections can saturate the server.

### Recommendation

1. **Enforce a minimum `startTime` floor:** Reject subscriptions where `startTime` is older than a configurable maximum lookback window (e.g., 30 days):
   ```java
   @AssertTrue(message = "Start time must be within the allowed lookback window")
   public boolean isValidStartTime() {
       long minAllowed = DomainUtils.now() - maxLookbackNanos;
       return startTime >= minAllowed && startTime <= DomainUtils.now();
   }
   ```
2. **Enforce a maximum `limit` for historical-only queries:** If `endTime` is set in the past, cap `limit` to a server-configured maximum (e.g., 10,000 messages).
3. **Cap total historical messages per subscription:** In `PollingTopicMessageRetriever`, enforce a hard ceiling on total emitted items regardless of the `limit` field.
4. **Rate-limit subscriptions per client IP/identity:** Limit the number of concurrent subscriptions from a single source.

### Proof of Concept

```python
import grpc
# Assumes proto stubs are generated from the mirror node proto definitions
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from com.hedera.hashgraph.sdk.proto import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# topicId of any active, high-volume topic (publicly discoverable)
topic_id = basic_types_pb2.TopicID(topicNum=1234)

request = consensus_service_pb2.ConsensusTopicQuery(
    topicID=topic_id,
    consensusStartTime=0,   # epoch 0 = Jan 1, 1970 — accepted by isValidStartTime()
    limit=0,                # no limit — hasLimit() returns false
)

# Open many parallel connections to amplify resource exhaustion
for msg in stub.subscribeTopic(request):
    pass  # live messages never arrive; server is busy streaming historical data
```

**Expected result:** The server streams the entire historical message backlog before delivering any live messages. With many parallel connections, DB connections and gRPC threads are exhausted, causing denial of service for legitimate subscribers.
