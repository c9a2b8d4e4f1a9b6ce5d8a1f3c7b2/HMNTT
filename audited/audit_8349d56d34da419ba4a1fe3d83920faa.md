### Title
Unauthenticated EVM Address `from=` Parameter Triggers Uncached DB Lookup Per Request, Enabling DB Connection Pool Exhaustion via Request Flood

### Summary
The `GET /contracts/results` endpoint accepts a `from=` query parameter. When the value is a valid EVM address, `extractContractResultsByIdQuery()` unconditionally calls `EntityService.getEncodedId()`, which executes a live database query against the `entity` table with no result caching. Because the REST API has no application-level rate limiting, an unprivileged attacker can flood this endpoint with syntactically valid EVM addresses, causing one DB connection slot to be consumed per request and degrading service for all users.

### Finding Description

**Exact code path:**

`getContractResults()` at [1](#0-0)  calls `extractContractResultsByIdQuery(filters)` with no contractId argument.

Inside `extractContractResultsByIdQuery`, the `FROM` filter branch at [2](#0-1)  checks `EntityId.isValidEvmAddress(filter.value)` — a pure regex check — and if true, immediately calls `await EntityService.getEncodedId(filter.value)`.

`getEncodedId` at [3](#0-2)  parses the string, detects a non-parsable EVM address (`evmAddress !== null`), and calls `getEntityIdFromEvmAddress`.

`getEntityIdFromEvmAddress` at [4](#0-3)  executes the query:

```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
```

against the live database with no caching layer. [5](#0-4) 

**Root cause — failed assumption:** The code assumes EVM address resolution is cheap or infrequent. There is no result cache for `getEntityIdFromEvmAddress`. The `entityId.js` LRU cache at [6](#0-5)  only caches the *parsing* of entity ID strings, not the DB lookup result. Each unique EVM address (or even the same one, since the cache is keyed on the parsed `EntityId` object, not the DB result) causes a fresh DB round-trip.

**Why `requireResult = true` makes it worse:** The default `requireResult = true` means even a non-existent EVM address causes the full DB query to execute before a `NotFoundError` is thrown. [7](#0-6) 

**Multiple `from=` values per request:** The request parser at [8](#0-7)  allows repeated query parameters up to `config.query.maxRepeatedQueryParameters`. Each repeated `from=<evmAddress>` value iterates through the filter loop and triggers a separate `getEncodedId` call, multiplying DB queries per request.

### Impact Explanation

Each attacker request with `from=<valid_evm_address>` consumes at least one DB connection slot for the duration of the `entity` table lookup. With N concurrent requests, N connection slots are held. PostgreSQL connection pools in Node.js REST services are typically sized 10–20 connections. A sustained flood of requests exhausts the pool, causing all subsequent DB operations (including unrelated endpoints) to queue or fail, degrading service for all users. This is a classic resource exhaustion / griefing DoS with no authentication barrier.

### Likelihood Explanation

The endpoint is publicly accessible with no authentication. [9](#0-8)  The only application-level protection is the `maxRepeatedQueryParameters` parse limit, which bounds per-request amplification but does not limit request rate. The Traefik-level rate limiting visible in Helm chart values is infrastructure-optional and not enforced by the application itself. Any attacker with HTTP access and a script can sustain the flood.

### Recommendation

1. **Cache EVM address → entity ID lookups** in `EntityService.getEntityIdFromEvmAddress` using a short-lived in-process LRU cache (similar to the existing `entityId.js` parse cache), keyed on the hex EVM address bytes.
2. **Add application-level rate limiting** to the REST API (e.g., express-rate-limit) for endpoints that trigger secondary DB lookups, independent of infrastructure-level Traefik configuration.
3. **Limit `from=` to a single occurrence** for the `/contracts/results` endpoint (as is done for EVM addresses in `balances.js` at [10](#0-9)  where `evmAliasAddressCount` is checked and only one EVM address is allowed).

### Proof of Concept

```bash
# Generate a random valid EVM address (non-existent in DB to ensure full lookup)
EVM_ADDR="0x$(openssl rand -hex 20)"

# Flood the endpoint with concurrent requests
for i in $(seq 1 500); do
  curl -s "https://<mirror-node-host>/api/v1/contracts/results?from=${EVM_ADDR}" &
done
wait
```

Each request triggers `EntityService.getEncodedId(EVM_ADDR)` → `getEntityIdFromEvmAddress()` → `SELECT id FROM entity WHERE evm_address = $1`. With 500 concurrent requests and a pool of ~10–20 connections, the pool is exhausted within milliseconds, causing all other API calls to stall or return errors until the flood subsides.
