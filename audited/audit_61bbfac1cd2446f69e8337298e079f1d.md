### Title
Unauthenticated IP Spoofing via Unsanitized `X-Forwarded-For` / `X-Real-IP` Headers in `getClientIpAddress()`

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` blindly trusts client-supplied `X-Real-IP` and `X-Forwarded-For` headers with no validation, trusted-proxy allowlist, or IP parsing. Any unprivileged external attacker can set these headers to an arbitrary value — including a trusted internal IP or a multi-value string like `10.0.0.1, attacker-ip` — causing the raw spoofed value to be written directly into the application's access logs. Because `main.go` explicitly configures logfmt output for Loki ingestion, downstream log parsers will attribute the request to the forged address, making security monitoring blind to the real source of any attack.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:
```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64 — blindly trusted
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67 — blindly trusted
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // line 71 — only fallback
    }
    return ipAddress
}
```

`TracingMiddleware` (lines 43–61) logs the raw return value directly:
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
```

`main.go` lines 217–219 show `TracingMiddleware` is applied directly to the HTTP server with no upstream proxy that would strip these headers:
```go
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
```

**Root cause:** The function assumes `X-Real-IP` and `X-Forwarded-For` are set by a trusted reverse proxy. There is no trusted-proxy allowlist, no IP format validation, and no extraction of only the first (client-side) IP from a multi-value XFF chain. `r.Header.Get("X-Forwarded-For")` in Go returns the full raw string, including commas and multiple IPs, verbatim.

**Why existing checks fail:** There are none. The test suite (`trace_test.go` lines 52–58) only tests single clean IP values in these headers, never multi-value or spoofed-internal-IP cases. No middleware strips or validates these headers before `getClientIpAddress()` is called.

**Exploit flow:**
1. Attacker sends: `GET /block HTTP/1.1` with header `X-Forwarded-For: 10.0.0.1, 203.0.113.99`
2. `r.Header.Get("X-Forwarded-For")` returns the string `"10.0.0.1, 203.0.113.99"` verbatim.
3. Log entry written: `10.0.0.1, 203.0.113.99 GET /block (200) in 3ms`
4. Loki (configured in `main.go` line 43 for logfmt parsing) ingests this; a parser splitting on the first space extracts `10.0.0.1,` as the IP field, or a comma-splitting parser extracts `10.0.0.1` — both attribute the request to the internal address.
5. Alternatively, attacker sets `X-Real-IP: 10.0.0.1` alone → log shows `10.0.0.1` with no trace of the real source.

### Impact Explanation
All access log entries for requests carrying these headers are permanently falsified. An attacker probing or tampering with Hashgraph history via the Rosetta API (e.g., submitting crafted construction transactions, querying block data) will appear in logs as originating from a trusted internal address. Security monitoring, alerting rules keyed on source IP, and forensic investigation after an incident are all rendered ineffective. The logfmt/Loki pipeline explicitly configured in `main.go` amplifies the impact because structured log consumers will parse the spoofed value as the authoritative client IP field.

### Likelihood Explanation
Exploitation requires zero privileges and zero authentication — any HTTP client reachable to the server can set arbitrary headers. The attack is trivially repeatable with a single curl command. It is silent (no error, no rejection, no side-effect visible to the attacker). The only precondition is network access to the Rosetta API port, which is the same precondition as using the API at all.

### Recommendation
1. **Extract only the leftmost (client) IP from XFF** after validating the request came through a known trusted proxy. The leftmost entry in `X-Forwarded-For` is the original client; entries appended by trusted proxies follow it.
2. **Maintain a trusted-proxy CIDR allowlist.** Only honour `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` is within the trusted set; otherwise fall back to `r.RemoteAddr` directly.
3. **Validate the extracted value** with `net.ParseIP()` before logging; reject or sanitize non-IP strings.

Example fix for `getClientIpAddress`:
```go
func getClientIpAddress(r *http.Request, trustedProxies []*net.IPNet) string {
    remoteIP, _, _ := net.SplitHostPort(r.RemoteAddr)
    if isTrustedProxy(remoteIP, trustedProxies) {
        if ip := r.Header.Get(xRealIpHeader); net.ParseIP(ip) != nil {
            return ip
        }
        if xff := r.Header.Get(xForwardedForHeader); xff != "" {
            // leftmost entry is the original client
            if first := strings.SplitN(xff, ",", 2)[0]; net.ParseIP(strings.TrimSpace(first)) != nil {
                return strings.TrimSpace(first)
            }
        }
    }
    return remoteIP
}
```

### Proof of Concept
```bash
# Spoof a trusted internal IP completely

# Multi-value XFF to confuse log parsers

# Observe log output — real attacker IP (203.0.113.99 or the actual RemoteAddr) never appears.
# Log will contain: "10.0.0.1 GET /network/list (200) in Xms"
