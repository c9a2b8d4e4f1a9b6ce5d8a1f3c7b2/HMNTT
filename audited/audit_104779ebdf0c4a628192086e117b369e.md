### Title
Crafted Mirror Node Response with Empty Service Endpoints Bypasses Retry Logic and Causes Fatal Pinger Crash

### Summary
In `fetchMirrorNodeNetwork()`, when the mirror node returns an HTTP 200 response with syntactically valid nodes (parseable `node_account_id`) but all `service_endpoints` having empty `domain_name` and `ip_address_v4` fields, every endpoint is silently skipped and the resulting empty network map is returned as a **non-retryable** error (`retry = false`). This bypasses the entire retry/backoff loop in `buildNetworkFromMirrorNodes()`, propagates to `newClient()`, and terminates the pinger process via `log.Fatalf`, causing a persistent crash loop for as long as the attacker controls the mirror node response.

### Finding Description

**Code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 113–129:

```go
for _, ep := range n.ServiceEndpoints {
    host := strings.TrimSpace(ep.DomainName)
    if host == "" {
        host = strings.TrimSpace(ep.IPAddressV4)
    }
    if host == "" || ep.Port == 0 {
        continue          // silently skipped — no error, no flag
    }
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId
}
```

After the loop, if `network` is still empty:

```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
    //              ^^^^^ retry = false
}
```

The `false` retry flag is consumed in `buildNetworkFromMirrorNodes()` at line 59:

```go
if !retry || attempt == attempts {
    break   // exits immediately on first attempt, all configured retries skipped
}
```

This error propagates to `newClient()` (`sdk_client.go`, line 19):

```go
netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
if err != nil {
    return nil, err
}
```

Which propagates to `main()` (`main.go`, line 43):

```go
client, err := newClient(cfg)
if err != nil {
    log.Fatalf("client error: %v", err)   // process exits
}
```

**Root cause:** The code conflates two distinct failure modes under the same `retry = false` path: (a) a genuinely permanent misconfiguration, and (b) a transiently or maliciously crafted response. A response that is HTTP 200, valid JSON, and contains parseable `node_account_id` values but has all endpoints with empty host fields is treated as a permanent, non-retryable condition, even though it is fully attacker-controllable.

**Failed assumption:** The developer assumed that a semantically valid envelope with no usable endpoints indicates a permanent configuration error that should not be retried. An attacker who controls the mirror node response can craft exactly this payload on demand, indefinitely.

**Exploit payload:**

```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        { "domain_name": "", "ip_address_v4": "", "port": 50211 }
      ]
    }
  ],
  "links": { "next": null }
}
```

This passes every guard: HTTP 200, valid JSON decode, non-empty `node_account_id`, parseable `AccountID` — but `host == ""` causes the endpoint to be skipped, leaving `network` empty.

### Impact Explanation

The pinger process calls `log.Fatalf` and exits. In a Kubernetes deployment, the pod enters a `CrashLoopBackOff`. The liveness heartbeat (`/tmp/alive`) stops being written; the readiness file (`/tmp/ready`) is never created for new pod instances. As long as the attacker serves the crafted response, every restart attempt fails at startup before any transfer is ever submitted. The pinger — whose purpose is to continuously submit transactions to verify consensus node liveness — is completely silenced. This constitutes a full denial-of-service of the monitoring component for any `network=other` deployment.

### Likelihood Explanation

The precondition is control over the mirror node HTTP response. This is achievable via: (1) a compromised or malicious mirror node operator, (2) DNS hijacking or BGP hijacking of the mirror REST endpoint, (3) a TLS MITM if certificate pinning is absent (no pinning is present in the code — a plain `http.Client` with no custom TLS config is used). The attack requires a single crafted HTTP response, is repeatable on every pod restart, and requires no authentication or privileged access to the pinger itself. The `network=other` mode is explicitly the custom/private network path, which is the highest-value target for operators running their own Hiero networks.

### Recommendation

1. **Make the empty-network case retryable.** Change `return nil, false, fmt.Errorf(...)` to `return nil, true, fmt.Errorf(...)` at line 128 so that a response with no usable endpoints is treated as a transient failure subject to the configured retry/backoff policy.

2. **Add a minimum-node threshold.** Reject responses where the total number of nodes in the envelope is zero separately from the case where nodes are present but all endpoints are unusable, so operators get a clear signal about which failure mode occurred.

3. **Do not fatal on startup for retryable conditions.** Consider restructuring `newClient` / `main` so that mirror node fetch failures during startup are retried at the application level rather than causing `log.Fatalf`.

4. **Enforce TLS and consider certificate pinning** for the mirror REST endpoint when `network=other` is used, to prevent MITM-based response injection.

### Proof of Concept

**Preconditions:** Attacker can intercept or serve responses for the configured `HIERO_MIRROR_PINGER_REST` URL.

**Steps:**

1. Stand up an HTTP server that always responds to `GET /api/v1/network/nodes` with:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"","ip_address_v4":"","port":50211}]}],"links":{"next":null}}
   ```
2. Set `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-server>`.
3. Start the pinger.

**Result:** The pinger logs `client error: attempt 1/11: no usable service_endpoints found from http://<attacker-server>/api/v1/network/nodes` and exits immediately via `log.Fatalf`. Despite `mirrorNodeClientMaxRetries=10` being configured, only one attempt is made because `retry=false`. The pod enters `CrashLoopBackOff` and never becomes ready. Repeatable on every restart as long as the attacker controls the response.
