### Title
Inverted Timestamp Range Bypasses Validation, Causing Wasteful DB Query Execution in `getHookStorageChange`

### Summary
When a user supplies two timestamp parameters forming an inverted range (e.g., `timestamp=gt:9999999999&timestamp=lt:1`), the `Bound` constructor's inversion guard is skipped because `Bound.of()` always passes `primarySortField=false` for timestamp bounds. The computed `timestampLowerBound > timestampUpperBound` values are forwarded directly to a native SQL query using `BETWEEN`, which executes against the database and returns zero rows, consuming DB resources on every such request.

### Finding Description

**Code path:**

1. `HooksController.hookStorageChangeRequest()` calls `Bound.of(timestamps, ...)` at line 186, which constructs a `Bound` with `primarySortField = false`: [1](#0-0) 

2. Inside the `Bound` constructor, the inversion guard is gated on `primarySortField`: [2](#0-1) 

   Because `primarySortField = false` for timestamp bounds, the `IllegalArgumentException` is **never thrown** even when `adjustedLower > adjustedUpper`.

3. `HookServiceImpl.getHookStorageChange()` reads both bounds without any additional inversion check and passes them directly to the repository: [3](#0-2) 

4. Both native SQL queries use `consensus_timestamp between :timestampLowerBound and :timestampUpperBound`: [4](#0-3) [5](#0-4) 

   PostgreSQL evaluates `BETWEEN x AND y` as `value >= x AND value <= y`; when `x > y` this is always false, but the query is still planned and executed.

**Root cause:** `Bound.of()` unconditionally passes `false` for `primarySortField`, disabling the only inversion guard in the `Bound` constructor. No secondary check exists in `getHookStorageChange()`.

### Impact Explanation
Every crafted request with an inverted timestamp range causes a full round-trip to the database: connection acquisition, query parsing, planning, and execution — all returning zero rows. With no rate limiting enforced at the service layer, an attacker can flood the endpoint to exhaust DB connection pool capacity or increase query-processing overhead, degrading service for legitimate users. The impact is griefing/resource exhaustion with no direct economic damage to network participants.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication required. The attack requires only a standard HTTP GET request with two timestamp query parameters in inverted order (e.g., `?timestamp=gt:9999999999&timestamp=lt:1`). The `@Size(max = 2)` constraint on the `timestamps` array confirms two parameters are accepted. The attack is trivially scriptable and repeatable at high frequency.

### Recommendation
Add an inversion check in `getHookStorageChange()` before executing the query:

```java
if (timestampLowerBound > timestampUpperBound) {
    return new HookStorageResult(ownerId, List.of());
}
```

Alternatively, remove the `primarySortField` gate from the `Bound` constructor check so that all `Bound` instances — including timestamp bounds — reject inverted ranges at construction time: [2](#0-1) 

Change line 58 from `if (primarySortField && adjustedLower > adjustedUpper)` to `if (adjustedLower > adjustedUpper)`, which would cause `Bound.of()` to throw an `IllegalArgumentException` translated to a 400 response before any DB interaction.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1000/hooks/1/storage?timestamp=gt:9999999999&timestamp=lt:1
```

**Preconditions:** No authentication needed; endpoint is public.

**Trigger:**
- `gt:9999999999` → `lower.value() = 9999999999`, `getAdjustedLowerRangeValue()` returns `10000000000`
- `lt:1` → `upper.value() = 1`, `adjustUpperBound()` returns `0`
- `primarySortField = false` → inversion guard skipped
- `getHookStorageChange()` calls repository with `timestampLowerBound=10000000000`, `timestampUpperBound=0`

**Result:** SQL executes `consensus_timestamp BETWEEN 10000000000 AND 0` — always false, zero rows returned, DB resources consumed. Repeating this in a loop degrades DB connection pool availability.
