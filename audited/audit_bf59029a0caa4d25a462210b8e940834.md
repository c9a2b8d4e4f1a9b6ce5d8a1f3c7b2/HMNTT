### Title
Unauthenticated DoS via Connection Pool Exhaustion on `GET /api/v1/network/stake`

### Summary
The `getNetworkStake()` handler in `NetworkController` issues an uncached, unthrottled database query on every request. No rate limiting exists in the `rest-java` module for this endpoint, and no `@Cacheable` annotation is applied anywhere in the call chain. A single attacker sending thousands of concurrent requests can exhaust the HikariCP connection pool, causing all database-backed endpoints on the mirror node to fail.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (line 127–130) calls `networkService.getLatestNetworkStake()` with no guard:

```java
@GetMapping("/stake")
NetworkStakeResponse getNetworkStake() {
    final var networkStake = networkService.getLatestNetworkStake();
    return networkStakeMapper.map(networkStake);
}
``` [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–56) directly delegates to the repository with no caching layer:

```java
@Override
public NetworkStake getLatestNetworkStake() {
    return networkStakeRepository
            .findLatest()
            .orElseThrow(() -> new EntityNotFoundException("No network stake data found"));
}
``` [2](#0-1) 

`NetworkStakeRepository.findLatest()` executes a native SQL query with a correlated subquery on every invocation:

```sql
select * from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
``` [3](#0-2) 

**Root cause — no caching:** A search across all `rest-java` Java sources finds zero `@Cacheable`, `@EnableCaching`, or `CacheManager` declarations. The `grpc` module does cache its equivalent `NodeStakeRepository.findAllStakeByConsensusTimestamp()` with a dedicated `NODE_STAKE_CACHE` Caffeine manager, but that protection is absent in `rest-java`. [4](#0-3) 

**Root cause — no rate limiting:** The `rest-java` module's `WebMvcConfiguration` registers only argument resolvers and formatters; `RestJavaConfiguration` adds only an ETag filter. Neither installs any throttle interceptor. [5](#0-4) [6](#0-5) 

The `ThrottleManager` / `ThrottleConfiguration` / `ThrottleProperties` stack that protects the `web3` module's contract-call endpoints is entirely absent from `rest-java`. [7](#0-6) 

### Impact Explanation

Every concurrent `GET /api/v1/network/stake` request acquires a JDBC connection from HikariCP (default pool size: 10). Under flood conditions the pool is exhausted; subsequent requests from all endpoints queue or fail with connection-timeout errors. This denies service to the entire mirror node REST API, not just the stake endpoint. Stake data is queried by wallets, SDKs, and tooling to determine staking rewards and validator weights; unavailability disrupts those consumers. The mirror node itself does not participate in consensus, so the Hedera network's own consensus is not directly halted, but any downstream system relying on the mirror node for stake data is affected.

### Likelihood Explanation

No authentication, API key, or IP-based restriction is required. Any internet-accessible mirror node deployment is reachable. A single machine with a modest HTTP flood tool (e.g., `wrk`, `hey`, `ab`) can sustain thousands of concurrent requests. The attack is trivially repeatable and requires no special knowledge beyond the public API path.

### Recommendation

1. **Add caching** — annotate `NetworkServiceImpl.getLatestNetworkStake()` with `@Cacheable` backed by a short-TTL Caffeine cache (e.g., 30 s), mirroring the pattern already used in the `grpc` module's `NodeStakeCache`.
2. **Add rate limiting** — apply a bucket4j `RateLimiter` interceptor or filter to the `rest-java` module for all `/api/v1/network/*` endpoints, consistent with the `ThrottleManagerImpl` pattern in `web3`.
3. **Infrastructure guard** — if Traefik or another ingress is in front of `rest-java`, add `inFlightReq` and `rateLimit` middleware (as already done for the Rosetta service).

### Proof of Concept

```bash
# Flood the endpoint with 500 concurrent connections for 30 seconds
wrk -t 10 -c 500 -d 30s http://<mirror-node-host>/api/v1/network/stake

# Observe: after pool exhaustion, all mirror node endpoints return 500 or hang
curl http://<mirror-node-host>/api/v1/network/stake
# Expected: connection timeout or HTTP 500 due to pool exhaustion
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
