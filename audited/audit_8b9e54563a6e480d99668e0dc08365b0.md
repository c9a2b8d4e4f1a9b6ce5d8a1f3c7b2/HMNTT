### Title
Gas Token Leak via Unrestored `gasLimitBucket` Tokens on REJECT Filter ThrottleException

### Summary
In `ThrottleManagerImpl.throttle()`, gas tokens are consumed from `gasLimitBucket` before the configured REJECT filter is evaluated. When a REJECT filter matches, `action()` throws `ThrottleException("Invalid request")`, but neither `ThrottleManagerImpl` nor `ContractController` restores the already-consumed gas tokens. An unprivileged attacker who can craft requests matching a REJECT filter can repeatedly drain the `gasLimitBucket`, starving all legitimate requests of gas capacity.

### Finding Description

**Exact code path:**

1. `ContractController.call()` calls `throttleManager.throttle(request)` inside a `try` block that only catches `InvalidParametersException` — `ThrottleException` is not caught and no `restore()` is called on that path.

   `ContractController.java` lines 38–50:
   ```java
   throttleManager.throttle(request);   // gas consumed here; if REJECT fires, exception escapes
   ...
   } catch (InvalidParametersException e) {
       throttleManager.restore(request.getGas());  // only for InvalidParametersException
       throw e;
   }
   ```

2. Inside `ThrottleManagerImpl.throttle()`, gas is consumed from `gasLimitBucket` at line 40 **before** the filter loop at lines 44–48:
   ```java
   } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
       throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
   }
   for (var requestFilter : throttleProperties.getRequest()) {
       if (requestFilter.test(request)) {
           action(requestFilter, request);   // REJECT throws here
       }
   }
   ```

3. `action()` line 69 throws unconditionally on REJECT:
   ```java
   case REJECT -> throw new ThrottleException("Invalid request");
   ```

**Root cause:** The design assumes that a `ThrottleException` from the REJECT path means no gas was "used," but `gasLimitBucket.tryConsume()` already succeeded and deducted tokens. The `restore()` call path in `ContractController` is guarded exclusively by `catch (InvalidParametersException)`, so `ThrottleException` propagates to `GenericControllerAdvice` (HTTP 429) with zero token restoration.

**Why existing checks fail:** The `rateLimitBucket` limits the attacker to `requestsPerSecond` (default 500) requests/sec, but within that budget each request can carry up to `maxGasLimit` gas. With default settings (`gasPerSecond` = 7.5 B, scaled by 10,000 = 750,000 tokens/sec; `maxGasLimit` = 15 M → 1,500 scaled tokens/request), 500 requests/sec × 1,500 tokens = 750,000 tokens/sec — exactly the full bucket capacity — can be drained every second with zero restoration.

### Impact Explanation
Every REJECT-matched request permanently removes up to `scaleGas(maxGasLimit)` tokens from the shared `gasLimitBucket`. Because the bucket is global and shared across all callers, sustained flooding causes `gasLimitBucket.tryConsume()` to fail for every subsequent legitimate request, returning HTTP 429 ("Gas per second rate limit exceeded") to all users. This is a complete, sustained denial-of-service against the `/api/v1/contracts/call` endpoint with no self-healing until the attacker stops (the bucket refills at its configured rate, but the attacker can drain it as fast as it refills).

### Likelihood Explanation
No privileges are required. The attacker only needs to discover which request fields trigger a REJECT filter. This is trivially achievable: REJECT returns the distinct message `"Invalid request"` (HTTP 429) while normal throttling returns `"Requests per second rate limit exceeded"` or `"Gas per second rate limit exceeded."` — a simple fuzzing loop over DATA/FROM/TO/BLOCK values reveals the filter patterns within seconds. Once known, the exploit is fully repeatable and automatable with any HTTP client.

### Recommendation
Restore gas tokens whenever `throttle()` throws any `ThrottleException` that was preceded by a successful `gasLimitBucket.tryConsume()`. The cleanest fix is one of:

1. **In `ThrottleManagerImpl.throttle()`**: Move the REJECT/THROTTLE filter evaluation to **before** `gasLimitBucket.tryConsume()`, so rejected requests never consume gas tokens.

2. **In `ContractController.call()`**: Broaden the restore catch to cover `ThrottleException` from the REJECT path:
   ```java
   } catch (InvalidParametersException | ThrottleException e) {
       throttleManager.restore(request.getGas());
       throw e;
   }
   ```
   (Care must be taken not to double-restore for the gas-bucket-exceeded case where no tokens were consumed.)

3. **In `ThrottleManagerImpl.action()`**: Call `restore()` before throwing on REJECT, since the gas was already consumed.

Option 1 (reorder checks) is the safest because it avoids consuming tokens for requests that will never be processed.

### Proof of Concept

**Precondition:** The operator has configured at least one REJECT filter, e.g.:
```yaml
hiero.mirror.web3.throttle.request:
  - action: REJECT
    filters:
      - field: DATA
        type: CONTAINS
        expression: "deadbeef"
```

**Steps:**

1. Discover the REJECT pattern by sending probe requests and observing the `"Invalid request"` response body (distinct from other throttle messages).

2. Send a flood of matching requests at the rate limit:
   ```bash
   for i in $(seq 1 500); do
     curl -s -X POST http://mirror-node/api/v1/contracts/call \
       -H 'Content-Type: application/json' \
       -d '{"data":"0xdeadbeef","gas":15000000,"to":"0x000...001","block":"latest"}' &
   done
   wait
   ```

3. Immediately send a legitimate (non-matching) request:
   ```bash
   curl -X POST http://mirror-node/api/v1/contracts/call \
     -H 'Content-Type: application/json' \
     -d '{"data":"0x12345678","gas":21000,"to":"0x000...001","block":"latest"}'
   ```

4. **Observed result:** The legitimate request receives HTTP 429 `"Gas per second rate limit exceeded."` because the `gasLimitBucket` has been fully drained by the REJECT-matched flood with no restoration.

5. **Expected result (correct behavior):** The legitimate request should succeed; REJECT-matched requests should not consume gas tokens.
