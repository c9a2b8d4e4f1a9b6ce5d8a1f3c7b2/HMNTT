### Title
Missing Timestamp Guard on `entity` Branch in `findActiveByEvmAddressAndTimestamp()` Returns Future State for Historical Queries

### Summary
The `entity` branch of the `UNION ALL` in `findActiveByEvmAddressAndTimestamp()` applies no `lower(timestamp_range) <= blockTimestamp` filter. When an entity is modified after the queried block timestamp, the entity's current row (with `timestamp_range = [T_modified, ∞)`) is always ranked higher by the outer `ORDER BY timestamp_range DESC LIMIT 1` than the correct `entity_history` row (with `timestamp_range = [T_created, T_modified)`), causing the mirror node to return a post-modification state for a pre-modification block query. Any unprivileged caller of a historical EVM endpoint can trigger this deterministically.

### Finding Description

**Exact code path:**
`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85, `findActiveByEvmAddressAndTimestamp()`.

```sql
-- entity branch (lines 67-72): NO lower(timestamp_range) <= ?2 guard
(
    select *
    from entity e
    where e.deleted is not true          -- only deleted check
    and e.id = (select id from entity_cte)
)
union all
-- entity_history branch (lines 74-81): correctly guarded
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2   -- ← guard present here
    and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
order by timestamp_range desc
limit 1
```

**Root cause:** The `entity_cte` (lines 60–66) only checks `created_timestamp <= ?2`, confirming the entity existed before the queried block. It does **not** verify that the entity's *current* state (`lower(timestamp_range)`) is also ≤ `?2`. The entity branch inherits this gap: it returns the entity row regardless of when it was last modified.

**Data model (confirmed by migration `V1.46.2__entity_history.sql`):**
- `entity.timestamp_range` = `int8range(last_modified_timestamp, null)` — open-ended `[T_mod, ∞)`
- `entity_history.timestamp_range` = `int8range(T_start, T_end)` — closed `[T_start, T_end)`

**Exploit flow:**

| Step | State |
|------|-------|
| T1 | Entity with `evm_address = X` created; `entity.timestamp_range = [T1, ∞)` |
| T2 > B | Entity modified; old row moved to `entity_history` with `[T1, T2)`; `entity.timestamp_range = [T2, ∞)` |
| Query | `findActiveByEvmAddressAndTimestamp(X, B)` where `T1 ≤ B < T2` |
| entity_cte | Finds entity (created_timestamp = T1 ≤ B) ✓ |
| entity branch | Returns row with `timestamp_range = [T2, ∞)` — no filter blocks it |
| entity_history branch | Returns row with `timestamp_range = [T1, T2)` — correctly filtered |
| `ORDER BY timestamp_range DESC` | `[T2, ∞)` > `[T1, T2)` because T2 > T1 → entity row wins |
| Result | **Wrong**: post-modification state returned for pre-modification block |

**Why existing checks fail:**
- `entity_cte`'s `created_timestamp <= ?2` only proves the entity was *born* before the block, not that its current state is valid at that block.
- `deleted is not true` on the entity branch only excludes deleted entities; a modified-but-not-deleted entity passes through unconditionally.
- The outer `ORDER BY timestamp_range DESC LIMIT 1` deterministically selects the entity row because `[T2, ∞)` always sorts above `[T1, T2)` in PostgreSQL `int8range` ordering (lower bound comparison: T2 > T1).

The same structural flaw exists in `findActiveByEvmAddressOrAliasAndTimestamp()` (lines 95–121) and `findActiveByIdAndTimestamp()` (lines 136–155).

### Impact Explanation
The mirror node's web3 module serves historical EVM execution (`eth_call`, `eth_getBalance`, `eth_getCode`, contract state reads) by translating block numbers to consensus timestamps and calling these repository methods. Returning a post-modification entity state for a pre-modification block means:

- **Balance misrepresentation**: An account's balance at a historical block is reported as its current (future) balance.
- **Contract state corruption**: A contract's bytecode, admin key, or other fields at a historical block reflect a later update.
- **Historical smart contract re-execution produces wrong results**: Any `eth_call` replay at a historical block that touches a modified entity will compute with incorrect state, breaking auditability and dispute resolution.

Severity: **High** — directly violates the core guarantee of a mirror node (faithful historical state reconstruction) and affects all historical EVM query endpoints without any authentication requirement.

### Likelihood Explanation
Preconditions are trivially met on any live network:
1. Any entity (account, contract, token) that has ever been updated satisfies the condition.
2. The attacker only needs to know a valid EVM address and any block number prior to the entity's last modification — both are publicly observable on-chain.
3. No credentials, no special role, no transaction submission required — a single read-only JSON-RPC call (`eth_call` or `eth_getBalance` with a `blockNumber` parameter) is sufficient.
4. The behavior is **deterministic**, not probabilistic — every qualifying query returns the wrong result, every time.

### Recommendation
Add `lower(e.timestamp_range) <= ?2` to the entity branch, mirroring the guard already present on the `entity_history` branch:

```sql
(
    select *
    from entity e
    where e.deleted is not true
    and lower(e.timestamp_range) <= ?2      -- ← add this
    and e.id = (select id from entity_cte)
)
union all
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2
    and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
order by timestamp_range desc
limit 1
```

Apply the same fix to `findActiveByEvmAddressOrAliasAndTimestamp()` (line 103–107) and `findActiveByIdAndTimestamp()` (lines 137–142), which carry the identical structural defect.

### Proof of Concept

**Setup (PostgreSQL):**
```sql
-- Entity created at T=100, later modified at T=200
INSERT INTO entity (id, evm_address, created_timestamp, timestamp_range, deleted)
VALUES (1, '\xDEADBEEF...', 100, int8range(200, null), false);

-- Historical state valid during [100, 200)
INSERT INTO entity_history (id, evm_address, created_timestamp, timestamp_range, deleted, balance)
VALUES (1, '\xDEADBEEF...', 100, int8range(100, 200), false, 1000);

-- Current state (post-modification, balance changed to 9999)
UPDATE entity SET balance = 9999 WHERE id = 1;
```

**Trigger (Java / JSON-RPC):**
```java
// blockTimestamp = 150, which is between T=100 (creation) and T=200 (modification)
Optional<Entity> result = entityRepository
    .findActiveByEvmAddressAndTimestamp(evmAddress, 150L);

// Expected: entity_history row with balance=1000, timestamp_range=[100,200)
// Actual:   entity row with balance=9999, timestamp_range=[200,∞)
assert result.get().getBalance() == 1000;  // FAILS — returns 9999
```

**Equivalent JSON-RPC call:**
```json
{
  "method": "eth_getBalance",
  "params": ["0xDEADBEEF...", "0x64"],
  "id": 1
}
```
Where `0x64` = block number corresponding to timestamp 150. Returns the post-modification balance instead of the historically correct value.
