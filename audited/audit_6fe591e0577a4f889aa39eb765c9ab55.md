### Title
Unauthenticated Double DB Round-Trip via Non-Existent EVM Address in `getBalances()`

### Summary
Any unauthenticated caller can supply a syntactically valid but non-existent EVM address as `account.id=eq:<addr>` to `/api/v1/balances`. The handler unconditionally performs two sequential database queries per request — one entity-table lookup that returns `null`, followed by a full balance query that executes with a `null` parameter and returns empty rows — with no application-level rate limiting, no negative-result caching, and no early-exit guard. At scale this amplifies wasted DB connection and query time linearly with request volume.

### Finding Description

**Step 1 — EVM address triggers async entity lookup (DB round-trip 1)**

In `rest/balances.js` `parseAccountIdQueryParam` (lines 320–357), when the supplied `account.id` value passes `EntityId.isValidEvmAddress(value, EvmAddressType.NO_SHARD_REALM)`, the code calls:

```js
return EntityService.getEncodedId(value, false);   // line 329
```

`requireResult = false` is intentional — it suppresses the `NotFoundError`. Inside `rest/service/entityService.js`, `getEncodedId` (line 118) delegates to `getEntityIdFromEvmAddress` (line 124), which executes:

```sql
SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
```

When the address does not exist the query returns 0 rows and `getEntityIdFromEvmAddress` returns `null` (line 97). `getEncodedId` propagates that `null` back as the resolved Promise value.

**Step 2 — null propagates into balance query with no guard (DB round-trip 2)**

Back in `getBalances` (line 88):

```js
const accountParams = await Promise.all(accountParamsPromise);
```

`accountParams` is now `[null]`. There is **no null-check and no early return** after this `await`. Execution falls through to lines 152–153:

```js
const pgSqlQuery = utils.convertMySqlStyleQueryToPostgres(sqlQuery);
const result = await pool.queryQuietly(pgSqlQuery, sqlParams);
```

The balance query executes with `null` bound to the `ab.account_id = ?` placeholder. In PostgreSQL `col = NULL` is always `false`, so the query scans the CTE / `account_balance` table and returns zero rows — a wasted second round-trip.

**Step 3 — No application-level rate limiting**

`rest/server.js` (line 106) registers the route with no rate-limiting middleware:

```js
app.getExt(`${apiPrefix}/balances`, balances.getBalances);
```

The only middleware present is `authHandler` (line 86), which only sets a custom response limit for authenticated users and does **not** block or throttle unauthenticated requests. The `ThrottleManager` / Bucket4j infrastructure found in the codebase is scoped exclusively to the `web3` module and is not applied here.

**Step 4 — No negative-result caching**

`getEntityIdFromEvmAddress` performs a raw `pool.queryQuietly` call with no caching layer. Every request with a distinct (or even repeated) non-existent EVM address hits the database.

### Impact Explanation

Each crafted request wastes two DB round-trips instead of zero or one. An attacker sending N requests/second forces 2N DB queries, consuming connection-pool slots, CPU, and I/O on the PostgreSQL backend. Because the REST API is publicly accessible with no authentication requirement and no application-level throttle, the attacker can sustain this indefinitely. Under sufficient load this degrades query latency for legitimate users and can exhaust the DB connection pool, causing cascading 500 errors across all API endpoints that share the same pool.

### Likelihood Explanation

The attack requires no credentials, no special knowledge, and no on-chain assets. Any HTTP client can generate a stream of requests with random 40-hex-character strings that are syntactically valid EVM addresses but absent from the entity table. The attack is trivially scriptable (`curl` in a loop, `ab`, `wrk`, etc.) and can be distributed across multiple source IPs to bypass any upstream IP-based rate limiting that may be configured at the infrastructure layer (Traefik/ingress).

### Recommendation

1. **Add an early-exit guard**: After `await Promise.all(accountParamsPromise)`, check whether any resolved value is `null` and return immediately (with the pre-set empty response already in `res.locals`) before issuing the balance query.

```js
const accountParams = await Promise.all(accountParamsPromise);
if (accountParams.some((p) => p === null)) {
  return; // EVM address / alias not found; empty result already set
}
```

2. **Cache negative entity lookups**: In `getEntityIdFromEvmAddress`, cache `null` results for a short TTL (e.g., 30 s) keyed on the EVM address bytes to prevent repeated DB hits for the same non-existent address.

3. **Apply application-level rate limiting to the REST API**: Add a per-IP rate-limit middleware (e.g., `express-rate-limit`) to the Express app, specifically covering the `/api/v1/balances` route, independent of any infrastructure-layer controls.

### Proof of Concept

```bash
# Generate a random valid-format but non-existent EVM address
ADDR="deadbeefdeadbeefdeadbeefdeadbeefdeadbeef"

# Single request — observe two DB queries in pg_stat_activity / slow-query log
curl "https://<mirror-node>/api/v1/balances?account.id=eq:${ADDR}"
# Returns: {"timestamp":null,"balances":[],"links":{"next":null}}

# Amplification loop (no auth required)
for i in $(seq 1 10000); do
  RAND=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/balances?account.id=eq:${RAND}" &
done
# Each iteration forces: 1× entity SELECT + 1× balance SELECT = 20 000 DB queries
# DB connection pool exhaustion degrades all concurrent legitimate requests
```
