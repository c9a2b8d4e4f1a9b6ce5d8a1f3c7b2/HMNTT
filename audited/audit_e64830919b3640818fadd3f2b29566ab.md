### Title
Cache Stampede in `findRuntimeBytecode()` Due to Missing `sync = true` on `@Cacheable`

### Summary
`ContractRepository.findRuntimeBytecode()` uses Spring's `@Cacheable` without `sync = true`, meaning concurrent threads that all miss the cache for the same `contractId` will each independently execute the underlying SQL query against the database. An unprivileged external user can deliberately trigger this by sending a large burst of concurrent `eth_call` requests for the same uncached contract, multiplying database load proportionally to the number of concurrent requests.

### Finding Description
**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16â€“18.

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring's `@Cacheable` without `sync = true` performs a non-atomic check-then-act: each thread independently checks the cache, finds a miss, and proceeds to invoke the underlying method. There is no per-key locking or coalescing. Caffeine *does* support a synchronous loading mode (via `get(key, loader)`) that serializes concurrent loaders for the same key, but Spring only uses that path when `sync = true` is explicitly set.

**Exploit flow:**
1. Attacker identifies any valid `contractId` that is not yet in the cache (e.g., a freshly deployed contract, or after cache expiry).
2. Attacker sends N concurrent HTTP requests to the `eth_call` endpoint targeting that contract.
3. All N threads reach `findRuntimeBytecode()` simultaneously, all observe a cache miss, and all issue `SELECT runtime_bytecode FROM contract WHERE id = :contractId` to the database.
4. All N results are returned; the cache is populated by whichever thread finishes last (or first, depending on race), but Nâˆ’1 queries were redundant.
5. The attacker repeats this pattern across multiple contract IDs or waits for TTL expiry and repeats.

**Why existing checks are insufficient:** The `@Cacheable` annotation is present and does cache results after the first response, but it provides no protection against the thundering-herd window. No `sync = true` is set anywhere in the codebase (confirmed: zero matches for `@Cacheable.*sync\s*=\s*true`). The `CaffeineCacheManager` configured in `EvmConfiguration.cacheManagerContract()` (line 68â€“73) is capable of synchronous loading but is never instructed to use it via Spring's abstraction.

### Impact Explanation
The direct impact is database query amplification: a single logical cache miss becomes N simultaneous identical SQL queries. For a high-concurrency burst (e.g., 500 concurrent requests), the database receives 500 identical primary-key lookups instead of 1. This degrades database throughput for all users of the mirror node, not just the attacker. Sustained or repeated attacks can exhaust database connection pool capacity, increasing latency or causing connection timeouts for legitimate users. There is no data corruption or economic loss, consistent with the stated "griefing" severity classification.

### Likelihood Explanation
No authentication or special privilege is required â€” any user with network access to the JSON-RPC endpoint can issue `eth_call`. HTTP clients trivially support concurrent requests (e.g., `curl`, `ab`, `wrk`, or any async HTTP library). The attack is repeatable: the attacker can wait for cache TTL expiry and repeat. The only limiting factor is the attacker's own network bandwidth and the server's request rate limiting (if any exists).

### Recommendation
Add `sync = true` to the `@Cacheable` annotation:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null",
    sync = true
)
```

With `sync = true`, Spring delegates to Caffeine's `get(key, mappingFunction)`, which guarantees that for any given key, only one thread executes the loader while all other concurrent threads block and receive the same computed value. This eliminates the stampede window entirely. The same fix should be audited and applied to other `@Cacheable` usages in the web3 repository layer that lack `sync = true`.

### Proof of Concept
```bash

CONTRACT_ID="0x0000000000000000000000000000000000001234"
ENDPOINT="http://<mirror-node-host>:8545"

# Send 200 concurrent eth_call requests for the same contract
for i in $(seq 1 200); do
  curl -s -X POST "$ENDPOINT" \
    -H "Content-Type: application/json" \
    -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_call\",\"params\":[{\"to\":\"$CONTRACT_ID\",\"data\":\"0x\"},\"latest\"],\"id\":$i}" &
done
wait
