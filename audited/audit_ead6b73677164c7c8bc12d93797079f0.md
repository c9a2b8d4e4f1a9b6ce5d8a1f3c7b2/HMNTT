### Title
Parameter Count Mismatch DoS in `getAccountIdsByPublicKey` via Multiple `account.publickey` Values

### Summary
`getAccountIdsByPublicKey` in `rest/balances.js` spreads the entire `pubKeyParams` array (which can contain multiple entries when a user supplies multiple `account.publickey` query parameters) into the `params` array passed to `entityPublicKeyQuery`. That query is hardcoded with exactly two positional placeholders (`$1`, `$2`), so supplying two or more valid public keys causes PostgreSQL to receive more bound parameters than the query declares, triggering a driver-level error that is re-thrown as a `DbError` and produces a 500 response.

### Finding Description

**Exact code path:**

`entityPublicKeyQuery` is a module-level constant with exactly two placeholders: [1](#0-0) 

`getAccountIdsByPublicKey` blindly spreads the entire `publicKey` array (which is `pubKeyParams` from the caller) and appends `limit`: [2](#0-1) 

The caller passes `pubKeyParams` directly — the raw array returned by `parsePublicKeyQueryParam`: [3](#0-2) 

`parsePublicKeyQueryParam` uses `allowMultiple=false`, meaning each `account.publickey` occurrence becomes a separate entry in the returned values array: [4](#0-3) 

The test suite confirms multiple values are returned when multiple `account.publickey` params are supplied: [5](#0-4) 

**Why validation fails:**

`validateReq` checks that each individual `account.publickey` value matches the public key format regex (64/66/88 hex chars) and that the total count does not exceed `maxRepeatedQueryParameters`. It does **not** restrict the parameter to a single occurrence: [6](#0-5) [7](#0-6) 

`ACCOUNT_PUBLICKEY` is listed as an accepted parameter with no single-occurrence enforcement: [8](#0-7) 

**Why the error propagates to a 500:**

`queryQuietly` catches the PostgreSQL bind-parameter-count error and **re-throws** it as a `DbError` — it does not return `null` or swallow the error: [9](#0-8) 

There is no try/catch in `getAccountIdsByPublicKey` or `getBalances`, so the `DbError` propagates unhandled, resulting in a 500 response.

**Root cause:** The function assumes `publicKey` always contains exactly one element (the single parsed key value), but `parsePublicKeyQueryParam` can return N values for N repeated query parameters. The spread `[...publicKey, limit]` produces N+1 params for a query with only 2 placeholders.

### Impact Explanation
Any unauthenticated external user can repeatedly crash the `/api/v1/balances` request handler with a crafted HTTP request, producing 500 responses. Because the trigger is stateless and requires no authentication, it can be used as a reliable, low-cost denial-of-service vector against the balances endpoint. The error path also leaks a `DbError` message to the caller, which may expose internal query details.

### Likelihood Explanation
Exploitation requires zero privileges — only the ability to send HTTP GET requests. The attacker needs to supply two syntactically valid public keys (64 hex characters each, trivially constructable) and a valid timestamp. The attack is deterministic and infinitely repeatable with a single HTTP request per attempt.

### Recommendation
Enforce that `account.publickey` is a single-occurrence parameter in the `/api/v1/balances` handler, or fix `getAccountIdsByPublicKey` to only use the first element of the array:

```js
// Option A: use only the first key
const params = [publicKey[0], limit];

// Option B: reject multiple publickey values in validateReq / balanceFilterValidator
// by checking Array.isArray(req.query['account.publickey']) and throwing InvalidArgumentError
```

The safest fix is Option B — reject the request with a 400 before any DB interaction if more than one `account.publickey` value is present, consistent with how the OpenAPI spec documents it as a single-occurrence parameter. [10](#0-9) 

### Proof of Concept

```
GET /api/v1/balances?timestamp=1234567890.000000000
  &account.publickey=3c3d546321ff6f63d701d2ec5c277095874e19f4a235bee1e6bb19258bf362be
  &account.publickey=6ceecd8bb224da4914d53f292e5624f6f4cf8c134c920e1cac8d06f879df5819
```

1. Both keys pass `isValidPublicKeyQuery` (64 hex chars each).
2. `validateReq` passes — two occurrences is within `maxRepeatedQueryParameters`.
3. `pubKeyParams = ['3c3d...', '6cee...']` (length 2).
4. `tsQuery` is non-empty → enters the `if (tsQuery)` branch.
5. `getAccountIdsByPublicKey(['3c3d...', '6cee...'], 25)` is called.
6. `params = ['3c3d...', '6cee...', 25]` — 3 elements.
7. `entityPublicKeyQuery` has only `$1` and `$2` — PostgreSQL returns: *"bind message supplies 3 parameters, but prepared statement requires 2"*.
8. `queryQuietly` re-throws as `DbError` → unhandled → **HTTP 500**.

```javascript
const validateReq = (req, acceptedParameters = emptySet, filterValidator) => {
  const badParams = [];
  // Check the validity of every query parameter
  for (const key in req.query) {
    if (!acceptedParameters.has(key)) {
      badParams.push({code: InvalidArgumentError.UNKNOWN_PARAM_USAGE, key});
      continue;
    }
    if (Array.isArray(req.query[key])) {
      if (!isRepeatedQueryParameterValidLength(req.query[key])) {
        badParams.push({
          code: InvalidArgumentError.PARAM_COUNT_EXCEEDS_MAX_CODE,
          key,
          count: req.query[key].length,
          max: config.query.maxRepeatedQueryParameters,
        });
        continue;
      }
      for (const val of req.query[key]) {
        if (!paramValidityChecks(key, val, filterValidator)) {
          badParams.push({code: InvalidArgumentError.INVALID_ERROR_CODE, key});
        }
      }
    } else if (!paramValidityChecks(key, req.query[key], filterValidator)) {
      badParams.push({code: InvalidArgumentError.INVALID_ERROR_CODE, key});
    }
  }

  if (badParams.length > 0) {
    throw InvalidArgumentError.forRequestValidation(badParams);
  }
};
```
