### Title
Per-Subscription Blocking JDBC Poll Exhausts Global `boundedElastic` Thread Pool Under DB Connectivity Loss

### Summary
When `PollingTopicListener` is active (`ListenerType.POLL`), each gRPC subscription spawns its own independent polling loop that executes a synchronous, blocking JDBC call (`getResultList()`) on the global `Schedulers.boundedElastic()` thread pool. No query timeout is set on the JPA `TypedQuery`. An unprivileged attacker who opens N concurrent subscriptions before or during a network partition will cause N threads to block simultaneously for the full JDBC connection-timeout duration, exhausting the shared bounded-elastic pool and denying service to all other reactive operations in the process.

### Finding Description

**Code path:**

`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicListener.listen(newFilter)` → `PollingTopicListener.listen()` → repeated calls to `PollingTopicListener.poll()`.

In `PollingTopicListener.java`, the scheduler is the global shared pool:

```java
// line 31
private final Scheduler scheduler = Schedulers.boundedElastic();
```

`listen()` schedules every repeat on that pool:

```java
// lines 38-43
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
        .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                .jitter(0.1)
                .withFixedDelay(interval)
                .withScheduler(scheduler))   // ← each repeat fires on boundedElastic
```

`poll()` immediately executes a blocking JDBC call on whichever `boundedElastic` thread picks up the task:

```java
// lines 51-62
private Flux<TopicMessage> poll(PollingContext context) {
    ...
    return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    //                     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    //  findByFilter() calls typedQuery.getResultList() synchronously
}
```

`TopicMessageRepositoryCustomImpl.findByFilter()` line 60:

```java
return typedQuery.getResultList().stream();
// getResultList() is a blocking JDBC call; no setTimeout() is ever called on typedQuery
```

There is **no** `typedQuery.setTimeout(...)` call anywhere in `findByFilter()`. The only timeout protecting this call is the JDBC driver's socket/connection timeout, which is typically 30 seconds or more and is not configured in this codebase.

Each subscription creates its own independent `PollingContext` and its own repeat loop — unlike `SharedPollingTopicListener`, which shares a single loop across all subscribers. Therefore N active subscriptions produce N concurrent `poll()` invocations, each occupying one `boundedElastic` thread for the full timeout duration when the DB is unreachable.

`Schedulers.boundedElastic()` is a JVM-global singleton (Reactor's static factory). Its default cap is `10 × availableProcessors` threads (e.g., 40 on a 4-core host). Once all threads are blocked, every other reactive pipeline in the application that uses `boundedElastic` (including the safety-check path in `TopicMessageServiceImpl` line 70) is also starved.

**Why existing checks fail:**

- `ConsensusController.subscribeTopic()` has no rate limit, no per-IP connection cap, and no authentication requirement.
- `ListenerProperties` sets no upper bound on concurrent subscriptions.
- There is no circuit breaker or bulkhead around the JDBC call.
- `repeatWhen` has no error-handling path that would terminate the loop on a connection failure; the loop simply retries indefinitely.

### Impact Explanation
Complete denial of service for the gRPC mirror-node process. All `boundedElastic` threads are occupied waiting for JDBC timeouts; no new polls, safety checks, or other reactive tasks can execute. Existing subscribers receive no messages and new subscribers cannot be served. The outage lasts for the full JDBC timeout window (typically 30 s) per poll cycle, and repeats on every subsequent poll interval for as long as the partition persists. Because the pool is global, the blast radius extends beyond topic polling to every component that uses `Schedulers.boundedElastic()`.

### Likelihood Explanation
The attacker needs only the ability to open gRPC connections — no credentials, no special privileges. Opening tens of long-lived `subscribeTopic` streams is trivially achievable with a standard gRPC client or even `grpcurl`. The network partition itself can be a natural infrastructure event (the attacker merely pre-positions) or can be induced by the attacker if they have any network-adjacent capability (e.g., BGP manipulation, firewall rule injection, or simply saturating the DB host's network interface). The attack is repeatable and requires no ongoing effort once the subscriptions are established.

### Recommendation
1. **Set a JPA query timeout** in `TopicMessageRepositoryCustomImpl.findByFilter()`:
   ```java
   typedQuery.setTimeout(queryTimeoutSeconds); // e.g., 5 s, from config
   ```
2. **Wrap the blocking call with `subscribeOn` + a dedicated, bounded scheduler** (separate from the global `boundedElastic`) so that exhaustion is isolated.
3. **Add a per-IP or global subscription cap** in `ConsensusController` or via a gRPC server interceptor.
4. **Add a circuit breaker** (e.g., Resilience4j) around the repository call so that repeated failures open the circuit and return immediately instead of blocking.
5. **Consider always preferring `SharedPollingTopicListener`** (`ListenerType.SHARED_POLL`) over `PollingTopicListener` (`ListenerType.POLL`) for production, since the shared variant uses a single polling loop regardless of subscriber count.

### Proof of Concept
1. Deploy the mirror-node gRPC service with `hiero.mirror.grpc.listener.type=POLL`.
2. From an attacker host, open N concurrent `subscribeTopic` streams (no auth required):
   ```bash
   for i in $(seq 1 50); do
     grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
       mirror-node:5600 \
       com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
   done
   ```
3. Induce a network partition between the mirror-node and its PostgreSQL database (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the mirror-node host, or block at the network level).
4. Observe: within one poll interval (default 500 ms), all 50 subscriptions fire `poll()` concurrently. Each call blocks a `boundedElastic` thread in `getResultList()` for the JDBC timeout (e.g., 30 s).
5. On a 4-core host the pool cap is 40 threads; with 50 subscriptions the pool is fully exhausted. New gRPC requests hang indefinitely; existing subscribers receive no data.
6. Restore connectivity: the threads unblock after the timeout, but the pool remains saturated for the full timeout window, and the cycle repeats on the next poll interval.

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
```
