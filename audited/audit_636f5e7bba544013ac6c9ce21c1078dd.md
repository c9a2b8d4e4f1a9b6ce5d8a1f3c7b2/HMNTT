### Title
Per-Subquery Limit Amplification in UNION ALL Token Allowance Query Enables 3× DB Row Amplification

### Summary
`extractTokenMultiUnionQuery()` assigns the user-controlled `limit` (capped at 100) directly to the query object, and `TokenAllowanceService.getQuery()` applies that same `$2` limit parameter independently to each of up to three UNION ALL subqueries before applying it again to the outer query. An unprivileged user can trigger all three subqueries simultaneously by supplying a closed spender.id range and a closed token.id range, causing the database to scan and return up to 300 rows per request while the caller receives at most 100. No rate limiting exists in the REST layer.

### Finding Description

**Code path:**

`tokenAllowanceController.js:38-39` — the limit value from the validated filter is stored verbatim: [1](#0-0) 

`tokenAllowanceService.js:56-58` — limit is placed at position `$2` and `getLimitQuery(2)` produces `LIMIT $2`, which is reused for every subquery: [2](#0-1) 

`tokenAllowanceService.js:63-74` — each non-empty filter set (lower, inner, upper) becomes an independent subquery each carrying its own `LIMIT $2`, and the outer wrapper also appends `LIMIT $2`: [3](#0-2) 

The three-subquery case is confirmed by the test fixture: [4](#0-3) 

**Root cause:** The limit is a single shared bind parameter applied once per subquery and once for the outer query. There is no aggregate cap across subqueries.

**Trigger condition:** Supplying `spender.id=gte:X&spender.id=lte:Y&token.id=gte:A&token.id=lte:B` causes `getLowerFilters`, `getInnerFilters`, and `getUpperFilters` to all return non-empty arrays, producing three independent subqueries each with `LIMIT $2`.

**Why the existing cap is insufficient:** `formatComparator` correctly caps the limit at 100 for unauthenticated users: [5](#0-4) 

But this cap applies to the per-subquery limit, not to the total rows fetched. With limit=100 and three subqueries, the database executes three separate index scans each returning up to 100 rows (300 total) before the outer `LIMIT 100` discards 200 of them.

### Impact Explanation
Every request to `GET /api/v1/accounts/:id/allowances/tokens` with a closed spender+token range and `limit=100` causes the database to do 3× the declared work. Under concurrent load from multiple clients (no rate limiting is present in the REST layer), this multiplies DB I/O and memory pressure proportionally. The realistic impact is degraded or unavailable mirror node REST API service. The claim of "halting Hedera network transaction confirmation" is **not accurate**: the mirror node is a read-only indexing service with no role in Hedera consensus; its outage cannot prevent new transactions from being confirmed by the network.

### Likelihood Explanation
The exploit requires zero privileges — any HTTP client can send the crafted query. The parameter combination is a natural pagination pattern (range on both dimensions), so it may also be triggered unintentionally. The attack is trivially repeatable and scriptable. The absence of any rate-limiting middleware in the REST layer means concurrent flooding is unconstrained from the application side.

### Recommendation
1. **Apply a single aggregate limit** across all subqueries rather than reusing `$2` per subquery. Pass `limit * subQueryCount` as the per-subquery cap and enforce the true limit only at the outer level, or restructure the query to use a single CTE/subquery with one LIMIT.
2. **Add request-level rate limiting** (e.g., express-rate-limit) on the allowances endpoint.
3. **Add a DB-level statement timeout** so runaway queries are killed before exhausting connection pool resources.

### Proof of Concept
```
# Trigger all three subqueries with limit=100 (max for unauthenticated users)
GET /api/v1/accounts/0.0.1234/allowances/tokens?spender.id=gte:0.0.100&spender.id=lte:0.0.999&token.id=gte:0.0.1&token.id=lte:0.0.9999&limit=100

# Resulting SQL (confirmed by test fixtures):
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ORDER BY spender ASC, token_id ASC LIMIT $2)   -- up to 100 rows
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ORDER BY spender ASC, token_id ASC LIMIT $2)      -- up to 100 rows
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ORDER BY spender ASC, token_id ASC LIMIT $2)    -- up to 100 rows
ORDER BY spender ASC, token_id ASC LIMIT $2                                                                                                -- outer cap: 100

# Script concurrent requests to amplify DB load:
for i in $(seq 1 200); do
  curl -s "http://<mirror-node>/api/v1/accounts/0.0.1234/allowances/tokens?spender.id=gte:0.0.100&spender.id=lte:0.0.999&token.id=gte:0.0.1&token.id=lte:0.0.9999&limit=100" &
done
wait
```
