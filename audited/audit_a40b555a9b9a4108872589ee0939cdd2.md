### Title
Unauthenticated `Content-Length` Header Spoofing Skews `api_request_size_bytes` Histogram

### Summary
The `metricsMiddleware()` function in `rest/middleware/metricsHandler.js` records request size by blindly parsing the client-supplied `Content-Length` header with no validation against the actual body size. Any unauthenticated external user can send a request with a spoofed `Content-Length: 9999999999` and a near-empty body, causing arbitrarily large values to be recorded in the `api_request_size_bytes` histogram. This permanently corrupts the metric for the lifetime of the process.

### Finding Description
**Exact code location:** `rest/middleware/metricsHandler.js`, function `metricsMiddleware()`, line 209:

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

**Root cause:** The code assumes the `content-length` header faithfully represents the actual body size. HTTP does not enforce this — a client can set `Content-Length` to any arbitrary value. Node.js's HTTP server passes the raw header value to the application without validating it against the actual bytes received. Express does not add any such validation either.

**Exploit flow:**
1. Attacker sends `POST /api/v1/transactions` (or any valid route) with header `Content-Length: 9999999999` and a 2-byte body `{}`.
2. Node.js accepts the connection and passes `req.headers['content-length'] = '9999999999'` to the middleware.
3. `parseInt('9999999999', 10)` evaluates to `9999999999` (within JS safe integer range).
4. `requestSizeHistogram.record(9999999999, labels)` is called, injecting a 9.3 GB data point into the histogram.
5. The `|| 0` fallback only guards against `NaN`/`0`/falsy — it provides zero protection against large numeric strings.

**Why existing checks fail:** The only sanitization is `parseInt(..., 10) || 0`, which correctly handles missing or non-numeric headers but does nothing to bound or validate a large numeric value. There is no upper-bound clamp, no comparison to `req.socket.bytesRead`, and no cross-check against the actual body bytes consumed.

### Impact Explanation
The `api_request_size_bytes` OpenTelemetry histogram is exported to Prometheus and visualized in Grafana dashboards. Injecting extreme values permanently distorts histogram bucket counts, sum, and mean for the targeted route+method+code label combination. Operators relying on this metric for capacity planning, anomaly detection, or alerting will receive false data. A sustained attack (e.g., 100 req/s) renders the metric entirely unusable. The response-size metric is unaffected because it is measured from actual `res.write`/`res.end` intercepts (lines 176–183), not from a header.

### Likelihood Explanation
The attack requires zero authentication, zero special tooling, and zero knowledge of the application beyond knowing a valid API path. A single `curl` command is sufficient. The attack is fully repeatable, stateless, and can be automated trivially. Any internet-facing deployment is exposed.

### Recommendation
Replace the header-trust approach with the actual bytes received from the socket, or clamp/validate the header value:

```js
// Option 1: use actual bytes received (most accurate)
const requestSize = req.socket ? req.socket.bytesRead : 0;

// Option 2: clamp the header to a sane maximum (e.g., configured body limit)
const MAX_BODY = 1_048_576; // 1 MB
const claimedSize = parseInt(req.headers['content-length'] ?? '0', 10) || 0;
requestSizeHistogram.record(Math.min(claimedSize, MAX_BODY), labels);
```

The cleanest fix is Option 1, which measures what was actually transferred rather than what the client claimed.

### Proof of Concept
```bash
# Spoof a 9.3 GB content-length with a 2-byte body
curl -s -X POST https://<mirror-node-host>/api/v1/transactions \
  -H "Content-Length: 9999999999" \
  -H "Content-Type: application/json" \
  --data-raw "{}" \
  --max-time 5

# Verify the histogram is corrupted by scraping metrics
curl -s https://<mirror-node-host>/swagger/metrics/ | grep api_request_size_bytes
