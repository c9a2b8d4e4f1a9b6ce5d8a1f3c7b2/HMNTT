### Title
Unauthenticated GraphQL `account` Query with Zero EVM Address Causes Unbounded Database Load (No Caching, No Rate Limiting)

### Summary
The GraphQL `account(input: AccountInput!)` query accepts `evmAddress: "0x0000000000000000000000000000000000000000"` as syntactically valid input. This address routes through `EntityServiceImpl.getByEvmAddressAndType` to an uncached `entityRepository.findById(0L)` call on every request. Because the graphql module's `EntityRepository` has no `@Cacheable` annotations and the graphql service has no rate limiting, an unprivileged attacker can flood the endpoint in a tight loop, issuing a live database query per request and exhausting the DB connection pool.

### Finding Description

**Step 1 — Input passes validation.**
`evmAddress` is declared with `@Pattern(regexp: "^(0x)?[a-fA-F0-9]{40}$")`. The zero address `0x0000000000000000000000000000000000000000` is 40 valid hex characters with the `0x` prefix and passes this check unconditionally. [1](#0-0) 

**Step 2 — Zero address routes to `findById(0)`.**
In `EntityServiceImpl.getByEvmAddressAndType`, the 20-byte zero address causes `buffer.getInt() == 0` (bytes 0–3) and `buffer.getLong() == 0` (bytes 4–11) to both be true, so the code takes the `findById(buffer.getLong())` branch where the final `getLong()` (bytes 12–19) is also 0. This calls `entityRepository.findById(0L)`. [2](#0-1) 

**Step 3 — The graphql `EntityRepository` has zero caching.**
The graphql module's own `EntityRepository` declares only two native queries (`findByAlias`, `findByEvmAddress`) with no `@Cacheable` annotations anywhere. The inherited `CrudRepository.findById` is also uncached. Every invocation issues a live SQL query. [3](#0-2) 

(Contrast: the `web3` module's `EntityRepository` has `@Cacheable(unless = "#result == null")` on `findById` and `findByEvmAddress` — the graphql module has none of this.)

**Step 4 — No rate limiting exists on the graphql service.**
`GraphQlConfiguration` installs only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — structural query-shape limits, not request-rate limits. The bucket4j throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is not wired into the graphql service. No authentication is required. [4](#0-3) 

**Step 5 — Result is never cached even if caching were added.**
Entity ID 0 does not exist; `findById(0L)` always returns `Optional.empty()`. Spring Cache's standard `unless = "#result == null"` guard does not suppress caching of an empty `Optional` (it is non-null), but since no caching is configured at all in this module, every request unconditionally hits the database.

### Impact Explanation
An attacker sending the same one-line GraphQL query in a tight loop forces one `SELECT * FROM entity WHERE id = 0` database round-trip per request. With no rate limiting and no caching, the DB connection pool can be saturated, causing query timeouts and degraded or unavailable service for all legitimate users of the graphql endpoint. No funds or on-chain state are affected; the impact is availability (griefing), consistent with the Medium scope classification.

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero knowledge of the system beyond the public GraphQL schema. The payload is a single, trivially constructed HTTP POST. It is repeatable indefinitely from a single IP or distributed across many IPs to evade any network-layer IP throttling. The zero address is a well-known sentinel value that any attacker familiar with EVM would try first.

### Recommendation
1. **Add caching for negative (empty) results** in the graphql `EntityRepository`, e.g. a short-TTL cache for `findById` and `findByEvmAddress` that also caches misses (remove `unless` or use a null-object sentinel).
2. **Add rate limiting to the graphql service** — port the bucket4j `ThrottleConfiguration` pattern from the `web3` module, or add a `WebGraphQlInterceptor` that enforces per-IP or global request-per-second limits.
3. **Reject the zero address explicitly** in `EntityServiceImpl.getByEvmAddressAndType` before issuing any repository call, since entity ID 0 is guaranteed not to exist.

### Proof of Concept

```bash
# No authentication required. Run in a tight loop:
while true; do
  curl -s -X POST https://<graphql-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d '{"query":"{ account(input: { evmAddress: \"0x0000000000000000000000000000000000000000\" }) { id } }"}' \
    > /dev/null
done
```

Each iteration:
1. Passes `@Pattern` validation on `evmAddress`.
2. Enters `getByEvmAddressAndType` → `buffer.getInt()==0 && buffer.getLong()==0` → `findById(0L)`.
3. Issues `SELECT * FROM entity WHERE id = 0` with no cache hit.
4. Returns `null` / empty — result is never cached.
5. Repeat indefinitely, one DB query per HTTP request, no throttle applied.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }

package org.hiero.mirror.graphql.repository;

import java.util.Optional;
import org.hiero.mirror.common.domain.entity.Entity;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.CrudRepository;
import org.springframework.graphql.data.GraphQlRepository;

@GraphQlRepository
public interface EntityRepository extends CrudRepository<Entity, Long> {
    @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
    Optional<Entity> findByAlias(byte[] alias);

    @Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
    Optional<Entity> findByEvmAddress(byte[] evmAddress);
}
