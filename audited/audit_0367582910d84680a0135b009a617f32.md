### Title
Log Injection via Unsanitized `X-Real-IP` Header in `getClientIpAddress()`

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the `X-Real-IP` header directly from the HTTP request without any content sanitization. The raw value is embedded into a logrus log message, allowing an unprivileged attacker to inject newline characters that create forged log entries. This can blind log-based monitoring systems to real node health alerts, enabling a secondary attack to shut down 30%+ of network processing nodes undetected.

### Finding Description
**Code path:**
- `getClientIpAddress()` (lines 63–75): reads `r.Header.Get(xRealIpHeader)` at line 64 with no content validation — only an emptiness check (`len(ipAddress) == 0`) at line 66.
- `TracingMiddleware` (lines 43–61): embeds the unsanitized return value directly into a `fmt.Sprintf` format string at lines 52–53, then passes it to `log.Info(message)` at line 58.

**Root cause:** The only guard on the header value is `len() == 0`. There is no IP format validation, no character allowlist, and no stripping of control characters (newlines `\n`, carriage returns `\r`, tabs `\t`). The failed assumption is that `X-Real-IP` will always contain a valid IP address string.

**Exploit flow:**
1. Attacker sends an HTTP request directly to the Rosetta endpoint (no proxy, or proxy misconfigured to forward client-supplied `X-Real-IP`) with:
   ```
   X-Real-IP: 1.2.3.4\ntime="2026-01-01T00:00:00Z" level=info msg="all nodes healthy, no anomalies detected"
   ```
2. `getClientIpAddress()` returns the string verbatim including the embedded `\n`.
3. `fmt.Sprintf` at line 52 produces a multi-line string.
4. `log.Info(message)` at line 58 writes both lines to the log output.
5. Log aggregators (Splunk, ELK, Loki, etc.) parse the injected second line as a legitimate log record.

**Why existing checks fail:** The `len(ipAddress) == 0` check at line 66 only prevents an empty header — it does nothing to reject or escape control characters embedded within a non-empty value.

### Impact Explanation
An attacker can fabricate arbitrary log entries that appear to originate from the Rosetta service itself (same timestamp format, same log level, same structured fields). A monitoring pipeline relying on log-based alerting for node health (e.g., alerting on error rates, connection failures, or consensus anomalies) can be silenced or flooded with false-positive "healthy" entries. This directly enables a secondary coordinated attack — such as targeted RPC flooding, resource exhaustion, or network partition — to shut down 30%+ of processing nodes while the operations team sees no alerts. Severity is Medium-High given the indirect but concrete path to operational blindness.

### Likelihood Explanation
No authentication or special privilege is required. Any HTTP client can set arbitrary request headers. The attack is trivially repeatable with a single `curl` command. The only deployment-level mitigation would be a correctly configured reverse proxy that strips or overwrites `X-Real-IP` before it reaches the application — but the application itself provides no such defense, and misconfigured or absent proxies are common in staging/internal deployments.

### Recommendation
Sanitize the header value in `getClientIpAddress()` before returning it. At minimum:
1. Validate that the value parses as a valid IP address using `net.ParseIP()` or `net.SplitHostPort()`, and return an empty/fallback string if it does not.
2. Strip or replace all control characters (`\n`, `\r`, `\t`, and other non-printable bytes) before the value is used in any log statement.
3. Consider using logrus structured fields (`log.WithField("client_ip", clientIpAddress).Info(...)`) with a formatter that quotes field values, which reduces (but does not eliminate) injection risk.

### Proof of Concept
```bash
# Direct request with newline-injected X-Real-IP (URL-encoded \n as literal in header)
curl -H $'X-Real-IP: 1.2.3.4\ntime="2026-01-01T00:00:00Z" level=info msg="node health OK, no issues"' \
     http://<rosetta-host>:<port>/network/status \
     -X POST -H "Content-Type: application/json" -d '{}'
```
**Expected log output (two separate parsed entries):**
```
time="2026-01-01T00:00:00Z" level=info msg="1.2.3.4
time="2026-01-01T00:00:00Z" level=info msg="node health OK, no issues"
```
The second line is a fully forged log record indistinguishable from a real one. Repeating this at high frequency while executing a node-shutdown attack keeps monitoring dashboards clear of real alerts. [1](#0-0) [2](#0-1)
