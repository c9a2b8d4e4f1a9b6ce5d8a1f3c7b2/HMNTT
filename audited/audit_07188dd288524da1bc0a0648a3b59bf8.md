### Title
Uncached Reactive Health Pipeline Enables Unauthenticated Amplification of `/network/stake` Requests

### Summary
The `health()` method in `SubscriberHealthIndicator` returns a cold, uncached `Mono<Health>` that unconditionally issues a live HTTP GET to `/network/stake` on every subscription. Because the `/actuator/health/cluster` path is exposed via the Kubernetes ingress without authentication, any unprivileged external caller can trigger independent, full-pipeline executions — each producing a separate outbound request to the REST Java API — stacking on top of the baseline load already generated by Kubernetes probes, Prometheus scrapers, and load balancer checks.

### Finding Description

**Exact code path:**

`SubscriberHealthIndicator.health()` (lines 59–64) unconditionally delegates to `restNetworkStakeHealth()` (lines 93–122), which calls `restApiClient.getNetworkStakeStatusCode()`:

```java
// SubscriberHealthIndicator.java:59-64
@Override
public Mono<Health> health() {
    return restNetworkStakeHealth()          // always executes
            .flatMap(health -> ...)
            .doOnNext(this::recordHealthMetric);
}
```

```java
// RestApiClient.java:67-69
public Mono<HttpStatusCode> getNetworkStakeStatusCode() {
    return webClientRestJava.get().uri("/network/stake").exchangeToMono(r -> Mono.just(r.statusCode()));
}
```

Every time `health()` is subscribed to, a new cold `Mono` chain is assembled and a new HTTP GET to `/network/stake` is issued. There is no `.cache()`, no `.cache(Duration)`, and no shared result between concurrent callers.

**Public attack surface:**

The Helm chart values expose the cluster health path via ingress with no authentication:

```yaml
# charts/hedera-mirror-monitor/values.yaml:91-99
ingress:
  enabled: true
  hosts:
    - host: ""
      paths: ["/actuator/health/cluster"]
  tls:
    enabled: false
```

**Root cause:** The `Mono<Health>` returned by `health()` is cold and stateless. Spring Boot's `ReactiveHealthIndicator` contract calls `health()` and subscribes to the returned `Mono` on every health endpoint invocation. Without a shared cached result, N concurrent HTTP requests to `/actuator/health/cluster` produce exactly N outbound HTTP calls to `/network/stake` on the REST Java API node.

**Why existing checks fail:** The 5-second timeout in `restNetworkStakeHealth()` (line 107) and the `onErrorResume` handler (lines 108–121) are error-handling guards, not rate-limiting or deduplication controls. They do not prevent multiple concurrent subscriptions from each independently issuing their own request.

### Impact Explanation

The REST Java API's `/network/stake` endpoint is the network processing node target. Each health probe invocation — from Kubernetes liveness (every 10 s), readiness probes, Prometheus scraper (every 30 s per `serviceMonitor.interval`), and load balancer checks — already generates a baseline stream of `/network/stake` requests. An external attacker hitting the publicly exposed `/actuator/health/cluster` endpoint adds directly to this count with a 1:1 amplification ratio and no authentication barrier. Sustained concurrent requests can push the REST Java API's `/network/stake` handler well above the 30% resource consumption threshold relative to the 24-hour baseline, degrading response times for legitimate consumers of that endpoint across the network.

### Likelihood Explanation

No privileges, credentials, or special network position are required. The ingress is enabled by default with TLS disabled. The attacker needs only to discover the public hostname and send HTTP GET requests to `/actuator/health/cluster`. Given that the baseline `/network/stake` call rate from internal probes is low (on the order of 6–12 requests/minute), adding even a modest rate of external requests (3–5/minute) is sufficient to exceed a 30% increase — well below any threshold that would be classified as brute force. The attack is trivially repeatable and scriptable.

### Recommendation

1. **Add a shared time-bounded cache** to the `Mono` returned by `health()` or by `restNetworkStakeHealth()`:
   ```java
   private final Mono<Health> cachedHealth = restNetworkStakeHealth()
       .cache(Duration.ofSeconds(10));
   ```
   Use this cached `Mono` in `health()` instead of constructing a new pipeline per call.

2. **Restrict the ingress path** for `/actuator/health/cluster` to known IP ranges (load balancer CIDRs only) via ingress annotations, or require mTLS/token authentication.

3. **Apply rate limiting** at the ingress layer (e.g., Traefik `RateLimit` middleware) on the `/actuator/health/*` paths to prevent external flooding.

### Proof of Concept

```bash
# Baseline: observe /network/stake request rate on the REST Java API node
# (e.g., via Prometheus metric or access log)

# Trigger: send concurrent unauthenticated requests to the public health endpoint
for i in $(seq 1 20); do
  curl -s "https://<monitor-ingress-host>/actuator/health/cluster" &
done
wait

# Result: each of the 20 concurrent requests independently issues a GET /network/stake
# to the REST Java API, immediately multiplying the /network/stake request rate
# by (20 + baseline_rate) / baseline_rate — exceeding 30% increase with minimal effort.
```

**Verification:** Enable access logging on the REST Java API and observe that the count of `GET /api/v1/network/stake` entries spikes in direct proportion to the concurrent requests sent to `/actuator/health/cluster`, with no deduplication or caching occurring between them.     private Mono<Health> restNetworkStakeHealth() {
        return restApiClient
                .getNetworkStakeStatusCode()
                .flatMap(statusCode -> {
                    if (statusCode.is2xxSuccessful()) {
                        return UP;
                    }

                    var status = statusCode.is5xxServerError() ? Status.DOWN : Status.UNKNOWN;
                    var statusMessage =
                            String.format("Network stake status is %s with status code %s", status, statusCode.value());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                })
                .timeout(Duration.ofSeconds(5))
                .onErrorResume(e -> {
                    var status = Status.UNKNOWN;
                    // Connection issue can be caused by database being down, since the rest API service will become
                    // unavailable eventually
                    var rootCause = ExceptionUtils.getRootCause(e);
                    if (rootCause instanceof ConnectException || rootCause instanceof TimeoutException) {
                        status = Status.DOWN;
                    }

                    var statusMessage =
                            String.format("Network stake status is %s with error: %s", status, e.getMessage());
                    log.error(statusMessage);
                    return health(status, statusMessage);
                });
    }
