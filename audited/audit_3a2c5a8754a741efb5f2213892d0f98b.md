### Title
Unbounded Thread Exhaustion via Hung Blocking JDBC Calls in `PollingTopicListener.listen()` During Network Partition

### Summary
`PollingTopicListener.listen()` schedules each subscription's first poll on the global `Schedulers.boundedElastic()` thread pool via `delaySubscription(interval, scheduler)`. The poll executes a blocking JDBC call (`typedQuery.getResultList()`) with no Reactor-level timeout operator. During a network partition, the JDBC driver hangs indefinitely at the TCP socket level, holding `boundedElastic` threads permanently. Because the same global scheduler is shared across all subscriptions and other system components, an unprivileged attacker opening enough concurrent subscriptions can exhaust the thread pool, preventing all subsequent subscriptions from ever executing their initial poll.

### Finding Description

**Exact code path:**

`PollingTopicListener.java` line 31 allocates the scheduler:
```java
private final Scheduler scheduler = Schedulers.boundedElastic();
```
`Schedulers.boundedElastic()` returns the **global singleton** bounded elastic scheduler (capped at `10 × availableProcessors` threads, default queue 100,000).

`listen()` lines 38–39:
```java
return Flux.defer(() -> poll(context))
        .delaySubscription(interval, scheduler)
```
`delaySubscription` schedules a one-shot timer task on `scheduler`. When the timer fires (after `interval` = 500 ms default), a `boundedElastic` thread is consumed to execute the upstream subscription — i.e., `poll(context)`.

`poll()` line 61:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
`findByFilter` (`TopicMessageRepositoryCustomImpl.java` line 60) is a **synchronous blocking JDBC call**:
```java
return typedQuery.getResultList().stream();
```
This call holds the `boundedElastic` thread for its entire duration.

**No timeout operator is present** in `listen()`. Compare with `PollingTopicMessageRetriever.retrieve()` which explicitly adds `.timeout(retrieverProperties.getTimeout(), scheduler)` — `PollingTopicListener` has no equivalent.

**Root cause / failed assumption:** The design assumes the JDBC call completes in bounded time. During a network partition, the PostgreSQL JDBC driver hangs at the TCP socket layer waiting for a response that never arrives. The PostgreSQL-side `statement_timeout = 10000ms` (configured via `alter user mirror_grpc set statement_timeout`) only fires once the query is executing server-side; it cannot fire if TCP packets are silently dropped. No JDBC `socketTimeout` property is configured in the gRPC module (confirmed: no `socketTimeout` in `grpc/**`). The thread is therefore held indefinitely.

**Why existing checks fail:**
- `hiero.mirror.grpc.db.statementTimeout = 10000` is a PostgreSQL server-side setting; it is unreachable when the TCP connection is partitioned.
- `hiero.mirror.grpc.netty.maxConcurrentCallsPerConnection = 5` limits calls per single TCP connection; an attacker opens multiple connections.
- No authentication or rate-limiting on `ConsensusController.subscribeTopic()` — any caller can open unlimited subscriptions.
- The `boundedElastic` scheduler queues tasks (up to 100,000) rather than rejecting them, so new subscriptions appear to succeed but their polls never execute once threads are exhausted.

### Impact Explanation

Once the `boundedElastic` thread pool is saturated, every new subscription's initial poll task queues but never runs. Subscribers receive no messages and no error — the stream silently stalls. Because `Schedulers.boundedElastic()` is the JVM-global singleton, exhaustion also starves `TopicMessageServiceImpl`'s safety-check flux (`.subscribeOn(Schedulers.boundedElastic())`, line 70) and `SharedTopicListener`'s `publishOn` (line 25), cascading the denial of service to all listener types. The condition persists for the full duration of the network partition with no self-healing, since hung threads are never released.

**Severity: High** — complete denial of new topic subscriptions for all users, no authentication required, no automatic recovery.

### Likelihood Explanation

The attack requires only: (a) network-reachable gRPC port (publicly exposed by design), (b) ability to open concurrent gRPC streams (standard gRPC client), and (c) a network partition between the mirror node and its PostgreSQL database (achievable via firewall rule injection, cloud security-group manipulation, or a real infrastructure fault). On a 4-core host the thread pool cap is 40 threads; an attacker needs only 40 concurrent subscriptions held open across the 500 ms delay window. This is trivially scriptable with `grpcurl` or any gRPC client library. The attack is repeatable and requires no credentials.

### Recommendation

1. **Add a Reactor timeout operator** in `listen()` immediately after `delaySubscription`:
   ```java
   return Flux.defer(() -> poll(context))
           .delaySubscription(interval, scheduler)
           .timeout(Duration.ofSeconds(10), scheduler)   // add this
           .repeatWhen(...)
   ```
   This bounds the maximum thread-hold time regardless of JDBC behavior.

2. **Configure a JDBC socket timeout** for the gRPC datasource (e.g., HikariCP `connectionTimeout` + PostgreSQL JDBC `socketTimeout` property) so the driver itself terminates hung connections.

3. **Use a dedicated, bounded scheduler** (not the global `Schedulers.boundedElastic()`) for `PollingTopicListener` to isolate its thread consumption from the rest of the application.

4. **Enforce a per-IP or per-connection subscription rate limit** at the gRPC server layer to limit the number of concurrent streams an unauthenticated caller can open.

### Proof of Concept

```

#    (e.g., iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the mirror host)

# 2. Open 50 concurrent subscriptions (exceeds default 10×CPU thread cap on a 4-core host)
for i in $(seq 1 50); do
  grpcurl -plaintext \
    -d '{"topicID":{"topicNum":1},"limit":0}' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 3. Wait 500ms (default interval) for all delaySubscription timers to fire.
#    Each fires a poll() → findByFilter() → JDBC call → hangs indefinitely.
#    boundedElastic thread pool is now exhausted.

grpcurl -plaintext \
  -d '{"topicID":{"topicNum":1},"limit":0}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Result: stream opens but never emits any messages and never errors —
# the initial poll task is queued but never scheduled.
