### Title
Unbounded pg Pool Queue Exhaustion via Concurrent GET /schedules/:scheduleId During Network Partition

### Summary
`getScheduleById()` in `rest/schedules.js` calls `pool.queryQuietly()` with no rate limiting, no per-IP concurrency cap, and no queue-depth guard. The global `pool` has a default maximum of 10 connections and an unbounded internal pending-queue. During a network partition the 10 connections hang indefinitely (PostgreSQL's `statement_timeout` is server-side and cannot be delivered across a broken TCP link), so every subsequent request is appended to the unbounded queue. An unprivileged attacker who floods the endpoint fills that queue, exhausting Node.js heap and blocking every other API endpoint that shares the same pool.

### Finding Description

**Exact code path**

`rest/schedules.js` line 120:
```js
const {rows} = await pool.queryQuietly(getScheduleByIdQuery, scheduleId);
```
`queryQuietly` is defined in `rest/utils.js` lines 1518–1520. Because `getScheduleById` passes no `preQueryHint`, the fast path `this.query(query, params)` is taken — node-postgres internally calls `pool.connect()`, which enqueues the request in the pool's unbounded `_pendingQueue` when all `max` connections are occupied.

**Pool configuration** (`rest/dbpool.js` lines 13–15):
```js
connectionTimeoutMillis: config.db.pool.connectionTimeout,  // default 20 000 ms
max: config.db.pool.maxConnections,                         // default 10
statement_timeout: config.db.pool.statementTimeout,         // default 20 000 ms
```
`statement_timeout` is a PostgreSQL server-side directive. During a network partition the server is unreachable, so the timeout signal is never delivered; the 10 TCP connections hang until the OS-level TCP keepalive fires (Linux default: ~2 hours).

**No rate limiting on the REST layer** — `rest/server.js` registers the route at line 116 with no throttle middleware. The throttling code found in the repository (`web3/src/main/java/…/ThrottleConfiguration.java`) belongs exclusively to the separate web3 Java service and does not protect the Node.js REST API.

**Exploit flow**

1. Network partition isolates the REST API from PostgreSQL.
2. The first 10 concurrent `GET /api/v1/schedules/:scheduleId` requests each acquire one of the 10 pool connections; those connections block on the hanging TCP socket.
3. Every subsequent request is appended to node-postgres's internal `_pendingQueue` (no size limit).
4. `connectionTimeoutMillis = 20 000 ms` means each queued entry waits 20 s before throwing `DbError`; the attacker re-sends at a rate faster than the drain rate, keeping the queue perpetually full.
5. Because `pool` is a `global` object shared by every handler (`rest/dbpool.js` line 36), all other endpoints — accounts, tokens, transactions, topics — are simultaneously starved of connections.

**Why existing checks fail**

| Mechanism | Why it fails |
|---|---|
| `connectionTimeoutMillis = 20 000` | Limits per-request wait, but the queue itself is unbounded; memory grows O(N) with attacker request rate |
| `statement_timeout = 20 000` | Server-side; unreachable during partition; never fires |
| `authHandler` middleware | Performs no rate limiting; no IP-based concurrency cap |
| Response cache | Disabled by default (`cache.response.enabled = false`) |

### Impact Explanation
All REST API endpoints share the single global `pool`. Once the 10 connections are held by hanging queries, every endpoint that calls `pool.queryQuietly()` — accounts, balances, contracts, tokens, transactions, topics — queues behind the attacker's flood. Node.js heap grows proportionally to the queue depth. At sufficient request volume the process OOMs or becomes unresponsive, constituting a complete denial of service for all API consumers. Severity: **High** (full API unavailability, no authentication required).

### Likelihood Explanation
The precondition (network partition) is an infrastructure event outside the attacker's control, but network partitions occur in cloud environments due to misconfigured security groups, BGP flaps, or AZ failures. An attacker who monitors public mirror-node latency can detect a partition and immediately begin flooding. The attack requires only an HTTP client capable of sending concurrent GET requests — no credentials, no special knowledge. It is repeatable for the duration of the partition.

### Recommendation
1. **Bound the pool queue**: Set `node-postgres` pool option `allowExitOnIdle: true` and add a custom queue-size guard (e.g., reject with HTTP 503 when pending queue depth exceeds `maxConnections * 2`).
2. **Add application-level rate limiting**: Apply `express-rate-limit` (or equivalent) globally in `rest/server.js` before route registration, limiting requests per IP per second.
3. **Set TCP-level timeouts**: Configure `connect_timeout` and OS-level `tcp_keepalives_idle`/`tcp_keepalives_interval` on the PostgreSQL connection so partitioned sockets are detected within seconds, not hours.
4. **Circuit breaker**: Wrap `pool.queryQuietly()` in a circuit breaker (e.g., `opossum`) that opens after N consecutive `DbError` failures, returning HTTP 503 immediately instead of queuing.
