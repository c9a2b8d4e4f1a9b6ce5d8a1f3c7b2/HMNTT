### Title
Unauthenticated `/health/readiness` Endpoint Triggers Unbounded PostgreSQL Connections and Dual Internal HTTP Calls Per Request, Enabling Sustained Resource Exhaustion

### Summary
The `GET /health/readiness` endpoint in `rosetta/app/middleware/health.go` is publicly accessible with no authentication, rate limiting, or result caching. Each request synchronously executes a fresh PostgreSQL connection/ping via `hellofresh/health-go/v4/checks/postgres` and two sequential internal HTTP calls (`/network/list` + `/network/status`) that themselves trigger additional database queries. An unprivileged external attacker polling at 1 req/sec sustains ~3 DB operations and 2 HTTP round-trips per second continuously, causing resource consumption well above the 30% baseline threshold over 24 hours.

### Finding Description
**Exact code path:**

`rosetta/app/middleware/health.go`, `NewHealthController()` lines 37–54 and `checkNetworkStatus()` lines 80–101, registered via `Routes()` lines 63–78.

**Root cause — three compounding issues with no mitigations:**

1. **Fresh PostgreSQL connection per request (line 42):**
   ```go
   Check: postgres.New(postgres.Config{DSN: rosettaConfig.Db.GetDsn()})
   ```
   The `hellofresh/health-go` v4.7.0 postgres checker (`go.mod` line 12) calls `sql.Open` + `db.PingContext` + `db.Close` on every invocation. This is not the application's pooled GORM connection from `rosetta/app/db/db.go`; it is a standalone, ephemeral connection created and torn down on each health check call.

2. **Two internal HTTP calls per request (lines 86–96):**
   ```go
   networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
   // ...
   _, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{...})
   ```
   Each of these calls hits the live Rosetta API handlers on `localhost:<port>`, which execute their own database queries (block/network status lookups).

3. **Zero rate limiting or caching:** A grep across all `rosetta/**/*.go` for `rateLimit`, `throttle`, `cache`, `sync.Once`, `checkInterval`, or `maxRequests` returns no matches on the health path. The `server.NewRouter` in `main.go` (line 111) applies only CORS middleware — no request throttling.

**Failed assumption:** The endpoint is designed for Kubernetes liveness/readiness probes (called every 10–30 seconds by the cluster). The implementation assumes only trusted internal callers at low frequency. There is no network policy, ingress rule, or application-layer guard preventing external access.

**Exploit flow:**
```
Attacker → GET /health/readiness (1/sec, no auth)
  └─ postgres.New().Check()        → sql.Open + PingContext + Close  [new TCP conn to PG]
  └─ NetworkList(ctx, ...)         → HTTP POST localhost/network/list → DB query
  └─ NetworkStatus(ctx, ...)       → HTTP POST localhost/network/status → DB query
```
Per second: 1 new PG connection lifecycle + 2 internal HTTP calls + ≥2 DB queries.
Over 24 hours at 1 req/sec: 86,400 PG connection open/close cycles, 172,800 internal HTTP calls, 172,800+ DB queries.

### Impact Explanation
- **PostgreSQL:** Each health check opens a new `lib/pq` connection outside the application pool. At 1 req/sec, PostgreSQL receives 86,400 connection establishment/teardown cycles per day. This consumes PG `max_connections` slots, increases `pg_stat_activity` churn, and elevates CPU on both the application server and the database server.
- **Internal HTTP amplification:** Each external request spawns 2 internal HTTP requests to the Rosetta API server itself. These are full handler invocations including JSON marshaling, DB queries for network/block status, and response serialization. This creates a 3× request amplification factor.
- **Memory:** Goroutines for each in-flight check, HTTP client buffers, and PG connection objects accumulate under sustained load.
- **Sustained nature:** At 1 req/sec (non-brute-force, easily evading basic rate monitors), the load is continuous and additive over 24 hours, making it straightforward to exceed a 30% resource increase above the baseline where the endpoint is called only by Kubernetes probes every 10–30 seconds.

### Likelihood Explanation
- **No privileges required:** The endpoint is a plain `GET` with no authentication token, API key, or IP restriction in the application layer.
- **Non-brute-force rate:** 1 req/sec is indistinguishable from normal monitoring traffic and will not trigger standard DDoS protections.
- **Trivially scriptable:** `while true; do curl -s http://<host>/health/readiness > /dev/null; sleep 1; done`
- **Stable and repeatable:** The attack requires no state, no session, and no knowledge of the system beyond the public endpoint path, which is documented in the Rosetta API specification.

### Recommendation
1. **Add a result cache with TTL** (e.g., 10–30 seconds) to the readiness handler so repeated calls within the window return the cached result without re-executing checks. The `hellofresh/health-go` library supports `health.WithCacheDuration(30 * time.Second)` — pass this option to `health.New(health.WithChecks(...), health.WithCacheDuration(...))`.
2. **Reuse the application's existing pooled DB connection** for the postgres health check instead of opening a new connection via DSN. Pass the existing `*sql.DB` handle and call `db.PingContext(ctx)` directly.
3. **Apply network-level access control** to restrict `/health/readiness` to the cluster's internal network (Kubernetes NetworkPolicy or ingress annotation) so it is not reachable from external clients.
4. **Add application-layer rate limiting** (e.g., `golang.org/x/time/rate` token bucket) on the health routes, capping concurrent or per-IP invocations.

```go
func checkNetworkStatus(port uint16) func(ctx context.Context) error {
	serverUrl := fmt.Sprintf("http://localhost:%d", port)
	cfg := client.NewConfiguration(serverUrl, "readiness-check", nil)
	rosettaClient := client.NewAPIClient(cfg)

	return func(ctx context.Context) (checkErr error) {
		networkList, _, err := rosettaClient.NetworkAPI.NetworkList(ctx, &types.MetadataRequest{})
		if err != nil {
			log.Errorf("Readiness check, /network/list failed: %v", err)
			return err
		}

		network := networkList.NetworkIdentifiers[0]
		_, _, err = rosettaClient.NetworkAPI.NetworkStatus(ctx, &types.NetworkRequest{NetworkIdentifier: network})
		if err != nil {
			log.Errorf("Readiness check, /network/status failed: %v", err)
			return err
		}

		return
	}
}
```
