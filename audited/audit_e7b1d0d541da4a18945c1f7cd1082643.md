### Title
Cache Stampede in `updateOperationAccountAlias` Enables Unauthenticated DB Amplification via Concurrent `Block()` Requests

### Summary
The `updateOperationAccountAlias` function in `rosetta/app/services/block_service.go` performs a non-atomic cache-check-then-DB-fetch sequence with no mutex or singleflight guard. Any unauthenticated caller can send arbitrarily many concurrent `/block` requests for the same block index, causing every goroutine to race past the empty-cache check and issue redundant `GetAccountAlias()` SQL queries for the same account IDs before any result is stored. The Traefik rate-limiting middleware that could partially mitigate this is disabled by default (`global.middleware: false`).

### Finding Description

**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`, lines 115–121:

```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // DB call
    if err != nil {
        return err
    }
    s.entityCache.Set(result.GetId(), result)   // written after DB round-trip
    cached = result
}
```

**Root cause:** The `go-generics-cache` library (`github.com/Code-Hex/go-generics-cache v1.5.1`) provides individually thread-safe `Get` and `Set` operations, but there is no atomic "get-or-compute" primitive. The window between `Get` returning `found=false` and `Set` completing spans the entire DB round-trip. N concurrent goroutines all calling `Get` before any `Set` completes will all observe `found=false` and all proceed to call `GetAccountAlias()`.

**No singleflight or mutex exists** anywhere in the rosetta service layer — confirmed: zero matches for `singleflight`, `sync.Mutex`, or `sync.RWMutex` in `rosetta/**/*.go`.

**Rate-limiting middleware is disabled by default.** The Traefik middleware chain (including `inFlightReq: amount: 5` and `rateLimit: average: 10`) is gated on:

```yaml
# charts/hedera-mirror-rosetta/values.yaml line 95
global:
  middleware: false
```

and the template only renders when both `global.middleware` and `.Values.middleware` are truthy:

```yaml
# charts/hedera-mirror-rosetta/templates/middleware.yaml line 3
{{ if and .Values.global.middleware .Values.middleware -}}
```

A default deployment has no rate limiting applied at all.

**Exploit flow:**

1. Attacker identifies a block index with many operations (e.g., a high-activity block).
2. Attacker sends N concurrent `POST /block` requests with the same `block_identifier.index`.
3. All N goroutines enter `updateOperationAccountAlias` simultaneously.
4. For each of the M unique account IDs in that block's operations, all N goroutines call `s.entityCache.Get()` before any `Set()` completes, all receive `found=false`.
5. All N goroutines call `s.accountRepo.GetAccountAlias()` → `db.Raw(selectCryptoEntityWithAliasById, ...)` for each account ID.
6. Total DB queries = N × M instead of M. With N=100 concurrent requests and M=50 accounts per block, this is 5,000 queries instead of 50.

### Impact Explanation

The DB is the shared bottleneck for the entire mirror-node stack. Sustained cache-stampede traffic against a single popular block index can saturate the PostgreSQL connection pool, increase query latency for all other services (REST API, gRPC, web3), and in extreme cases cause connection exhaustion or OOM on the DB host. No funds are at risk, but service availability for all users degrades. This matches the stated "griefing with no economic damage" classification at medium severity.

### Likelihood Explanation

No authentication is required for the Rosetta `/block` endpoint. The attacker needs only an HTTP client capable of issuing concurrent requests — trivially achievable with `curl`, `ab`, `wrk`, or any scripting language. The attack is repeatable indefinitely, requires no special knowledge beyond the block index, and is amplified by choosing blocks with many operations. Default deployments (with `global.middleware: false`) have zero server-side throttling.

### Recommendation

1. **Introduce `golang.org/x/sync/singleflight`** around the cache-miss path in `updateOperationAccountAlias`. Key the singleflight group on `accountId.GetId()` so that only one in-flight DB call is made per account ID regardless of concurrency.

```go
result, err, _ := s.sfGroup.Do(strconv.FormatInt(accountId.GetId(), 10), func() (interface{}, error) {
    return s.accountRepo.GetAccountAlias(ctx, accountId)
})
```

2. **Enable the Traefik middleware by default** (`global.middleware: true`) or document that it must be enabled in production. The existing `inFlightReq: amount: 5` per IP provides a partial backstop but is insufficient alone without the singleflight fix.

3. Consider adding a **per-block-index request coalescing layer** so that concurrent requests for the same block share a single in-flight DB fetch for the entire block, not just per account.

### Proof of Concept

```bash

BLOCK_INDEX=1000
for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>/block \
```

```go
func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

	return nil
}
```
