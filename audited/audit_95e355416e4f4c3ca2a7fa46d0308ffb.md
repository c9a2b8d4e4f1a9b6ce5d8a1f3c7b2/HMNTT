### Title
Unbounded Goroutine Accumulation via Missing DB Statement Timeout in `NetworkStatus()`

### Summary
`NetworkStatus()` in `rosetta/app/services/network_service.go` passes the HTTP request context directly to database queries (`selectLatestWithIndex`, `selectGenesis`) without any per-request or per-statement timeout. When `statementTimeout` is 0 (the default for the rosetta DB config, as it is absent from the documented defaults), `GetDbWithContext` applies no deadline to the query. An unprivileged attacker can flood `/network/status` with concurrent requests, causing goroutines to block indefinitely on slow or stalled DB queries, accumulating until memory is exhausted.

### Finding Description

**Exact code path:**

1. `NetworkStatus()` receives `ctx context.Context` from the HTTP handler and calls `n.RetrieveLatest(ctx)` and `n.RetrieveGenesis(ctx)`. [1](#0-0) 

2. `RetrieveLatest(ctx)` calls `br.dbClient.GetDbWithContext(ctx)` and then executes `selectLatestWithIndex` via GORM. [2](#0-1) 

3. `GetDbWithContext` only wraps the context with a timeout when `statementTimeout > 0`. When `statementTimeout <= 0` (the zero-value default for `int` in Go, and no default is documented for `hiero.mirror.rosetta.db.statementTimeout`), it returns the context as-is with a no-op cancel: [3](#0-2) 

4. The `Db.StatementTimeout` field has type `int` with no default value set in the rosetta config struct: [4](#0-3) 

**Root cause:** When `statementTimeout` is 0, no deadline is placed on the DB query context. The HTTP server's `WriteTimeout` (default 10s) closes the TCP connection after the deadline, but Go's `net/http` does **not** cancel the request context when `WriteTimeout` fires — the handler goroutine continues blocking on the GORM query indefinitely. [5](#0-4) 

**Why existing checks fail:**
- `WriteTimeout` (10s default) only sets a deadline on the `net.Conn` write path; it does not cancel the `context.Context` passed to the handler. The goroutine remains alive.
- The Traefik `inFlightReq: amount: 5` and `rateLimit: average: 10` in the Helm chart are optional deployment-level controls, not enforced by the application itself. [6](#0-5) 
- The DB connection pool (`MaxOpenConnections`) limits concurrent DB queries, but once exhausted, goroutines block waiting for a free connection — they do not return an error — so they still accumulate. [7](#0-6) 

### Impact Explanation
An attacker sending a sustained flood of concurrent POST requests to `/network/status` causes goroutines to pile up, each holding a stack (minimum ~8 KB, growing under load) and a pending DB connection slot. Once the connection pool is saturated, subsequent goroutines block indefinitely waiting for a slot. With no upper bound on goroutine count, the process heap grows until the OS OOM-killer terminates it, resulting in a complete service outage for all Rosetta API consumers. Because this is a public, unauthenticated endpoint, the impact is a full non-network DoS of the Rosetta node.

### Likelihood Explanation
The endpoint requires no authentication and accepts any valid POST body. A single attacker with a modest HTTP client (e.g., `wrk`, `hey`, or a simple Go script) can open hundreds of concurrent connections. The condition is reliably triggered whenever the DB is under load or the connection pool is exhausted — both of which the attacker can induce by the flood itself. The attack is repeatable and requires no special privileges or knowledge of the system internals.

### Recommendation
1. **Set a non-zero default for `statementTimeout`** in the rosetta DB configuration (analogous to `hiero.mirror.web3.db.statementTimeout: 3000`). This ensures `GetDbWithContext` always wraps the context with a bounded deadline.
2. **Add a per-request context timeout** at the HTTP handler or middleware layer (e.g., `context.WithTimeout(ctx, 10*time.Second)`) so that even if `statementTimeout` is misconfigured, the goroutine is cancelled when the HTTP deadline passes.
3. **Enforce application-level concurrency limits** (e.g., a semaphore or `golang.org/x/net/netutil.LimitListener`) rather than relying solely on optional Traefik middleware.

### Proof of Concept

**Preconditions:** Rosetta node running in online mode with default config (`statementTimeout = 0`).

**Steps:**
```bash
# 1. Identify the network identifier
curl -s -X POST http://<rosetta-host>:5700/network/list \
  -H 'Content-Type: application/json' -d '{}'

# 2. Flood /network/status with high concurrency
# (replace network_identifier with actual values from step 1)
cat > payload.json <<'EOF'
{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}
EOF

# Send 1000 concurrent requests, keeping connections open
hey -n 100000 -c 500 -m POST \
  -H 'Content-Type: application/json' \
  -D payload.json \
  http://<rosetta-host>:5700/network/status
```

**Result:** Goroutine count climbs monotonically (observable via `pprof` or `/debug/pprof/goroutine`). Once the DB connection pool is exhausted, all new goroutines block indefinitely. Memory grows until the process is OOM-killed or becomes unresponsive to legitimate requests.

		CurrentBlockTimestamp:  currentBlock.GetTimestampMillis(),
		GenesisBlockIdentifier: genesisBlock.GetRosettaBlockIdentifier(),
		Peers:                  peers.ToRosetta(),
	}, nil
}

func (br *blockRepository) RetrieveLatest(ctx context.Context) (*types.Block, *rTypes.Error) {
	if err := br.initGenesisRecordFile(ctx); err != nil {
		return nil, err
	}

	db, cancel := br.dbClient.GetDbWithContext(ctx)
	defer cancel()

	rb := &recordBlock{}
	if err := db.Raw(selectLatestWithIndex).First(rb).Error; err != nil {
		return nil, handleDatabaseError(err, hErrors.ErrBlockNotFound)
	}

	if rb.Index < br.genesisBlock.Index {
		return nil, hErrors.ErrBlockNotFound
	}

	return rb.ToBlock(br.genesisBlock), nil
}

func (d *client) GetDbWithContext(ctx context.Context) (*gorm.DB, context.CancelFunc) {
	if d.statementTimeout <= 0 {
		db := d.db
		if ctx != nil {
			db = db.WithContext(ctx)
		}
		return db, noop
	}

	if ctx == nil {
		ctx = context.Background()
	}

	// #nosec G118
	childCtx, cancel := context.WithTimeout(ctx, time.Duration(d.statementTimeout)*time.Second)
	return d.db.WithContext(childCtx), cancel
}
