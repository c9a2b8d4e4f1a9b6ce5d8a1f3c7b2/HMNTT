### Title
Unauthenticated Hash-Based LIKE Query DoS via `getByHashOrNumber()` Without Application-Level Rate Limiting

### Summary
The public `/api/v1/blocks/{hashOrNumber}` endpoint accepts valid 96-character Hedera hex hashes and passes them directly to a PostgreSQL `LIKE hash+'%'` query with no application-level rate limiting. Because PostgreSQL B-tree indexes do not support `LIKE` prefix scans unless created with `text_pattern_ops`/`varchar_pattern_ops` or under a `C` locale, each request may trigger a full sequential scan of the `record_file` table. An unauthenticated attacker flooding this endpoint with valid hashes can exhaust database I/O and degrade service for all consumers.

**Root cause and failed assumption:**

`validateHashOrNumber` accepts any string matching `/^(0x)?([0-9A-Fa-f]{64}|[0-9A-Fa-f]{96})$/`. A full 96-character Hedera hash passes this check. The hash is then forwarded to `getByHashOrNumber(hash, null)`, which builds:

```sql
WHERE hash LIKE $1   -- $1 = '<96-char-hex>%'
```

The `%` wildcard is unconditionally appended regardless of whether the caller supplied a full-length hash or a short prefix. PostgreSQL's standard B-tree index on `hash` does **not** support `LIKE` prefix scans unless the index was created with `text_pattern_ops` (or the database uses `C` locale). Without that operator class, the planner falls back to a sequential scan of the entire `record_file` table for every request. The migration `V1.65.1__record_file_hash_index.sql` adds an index on the hash column, but its operator class could not be confirmed; if it omits `text_pattern_ops`, every hash lookup is a full table scan.

**No application-level rate limiting exists:**

A grep across all REST JS source files returns zero matches for `rateLimit`, `rate_limit`, `throttle`, or `rateLimiter`. [6](#0-5) 

Infrastructure-level rate limiting (Traefik middleware, GCP backend policy) is present in Helm charts but is optional and not enforced by the application itself. [7](#0-6) 

### Impact Explanation

If the `record_file` hash index lacks `text_pattern_ops`, every call to `GET /api/v1/blocks/<96-char-hex>` causes a full sequential scan of the `record_file` table. As the table grows (one row per block, tens of millions of rows on mainnet), each scan reads the entire table from disk. Flooding this endpoint with concurrent requests saturates disk I/O and shared-buffer bandwidth, causing query latency to spike across all database consumers — including ingestion pipelines and other REST endpoints — constituting a non-network-based DoS.

### Likelihood Explanation

The endpoint requires no authentication, no API key, and no session. Any external actor can craft valid 96-character hex strings (e.g., 96 `a` characters) and issue them at high concurrency using standard HTTP tooling (`wrk`, `ab`, `curl` in a loop). The attack is stateless, trivially repeatable, and requires no prior knowledge of the system beyond the public API specification, which documents the `hashOrNumber` path parameter pattern explicitly. [8](#0-7) 

### Recommendation

1. **Use equality for full-length hashes.** In `getByHashOrNumber`, detect when the supplied hash is already at full length (96 hex chars) and emit `WHERE hash = $1` instead of `WHERE hash LIKE $1`. Reserve `LIKE` only for shorter prefix inputs (e.g., 64-char ETH hashes matching against 96-char Hedera hashes).

2. **Ensure the index supports LIKE.** If prefix matching must be retained, recreate the `record_file` hash index with `text_pattern_ops`:
   ```sql
   CREATE INDEX CONCURRENTLY record_file_hash_pattern_idx
     ON record_file (hash text_pattern_ops);
   ```

3. **Add application-level rate limiting.** Enforce per-IP or global request-rate limits in the Express middleware layer, independent of infrastructure configuration.

### Proof of Concept

```bash
# Generate a valid 96-char Hedera hex hash (all 'a's passes the regex)
HASH=$(python3 -c "print('a'*96)")

# Flood the endpoint with concurrent requests
wrk -t8 -c200 -d60s "https://<mirror-node-host>/api/v1/blocks/$HASH"
```

**Expected result:** Each request triggers `WHERE hash LIKE 'aaa...96chars%'` on the `record_file` table. Without a `text_pattern_ops` index, PostgreSQL performs a sequential scan per request. Under sustained concurrency, database CPU and I/O utilization climb to saturation, causing latency degradation across all dependent services.

  async getByHashOrNumber(hash, number) {
    let whereStatement = '';
    const params = [];
    if (hash) {
      hash = hash.toLowerCase();
      whereStatement += `${RecordFile.HASH} like $1`;
      params.push(hash + '%');
    } else {
      whereStatement += `${RecordFile.INDEX} = $1`;
      params.push(number);
    }

    const query = `${RecordFileService.blocksQuery} where ${whereStatement}`;
    const row = await super.getSingleRow(query, params);
    return row ? new RecordFile(row) : null;
  }
