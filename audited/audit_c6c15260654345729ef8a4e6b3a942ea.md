### Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Concurrent 100-Element IN-Clause Queries on `getHooks()`

### Summary
The `getHooks()` endpoint in `HooksController` accepts up to 100 `hook.id=eq:N` parameters per request (enforced by `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)`), each of which is collected into a `TreeSet` and forwarded to `HookServiceImpl.getHooks()`, which issues a single `IN (...)` clause DB query with up to 100 elements. No rate limiting exists in the `rest-java` module for this endpoint. An unauthenticated attacker sending many concurrent requests each carrying 100 distinct EQ values can saturate the DB connection pool, rendering the rest-java API service unavailable.

### Finding Description

**Exact code path:**

`HooksController.getHooks()` — `rest-java/src/main/java/org/hiero/mirror/restjava/controller/HooksController.java`, lines 81–102: [1](#0-0) 

The `@Size(max = MAX_REPEATED_QUERY_PARAMETERS)` annotation permits up to 100 `hook.id` parameters per request. [2](#0-1) 

`MAX_REPEATED_QUERY_PARAMETERS = 100`.

All EQ-operator values are collected into a `TreeSet<Long>` (deduplication only, no size reduction beyond the 100-cap): [3](#0-2) 

This `TreeSet` is passed to `HookServiceImpl.getHooks()`, which — when `hookIds` is non-empty — issues a `findByOwnerIdAndHookIdIn(...)` call with up to 100 elements: [4](#0-3) 

**Root cause:** The `@Size(max = 100)` check is a per-request parameter count guard only. It does not limit the rate of concurrent requests, nor does it bound the DB work per unit time. The `rest-java` module contains no rate-limiting filter or throttle bean for this endpoint — the only throttle infrastructure in the codebase is in the `web3` module and is scoped to contract-call endpoints: [5](#0-4) 

The only filter registered in `rest-java` is a `LoggingFilter` — no rate limiter.

**Failed assumption:** The design assumes that the per-request 100-parameter cap is sufficient to prevent resource exhaustion. It is not, because the cap applies per request, not across concurrent requests. Each concurrent request legally holds a DB connection for the duration of its 100-element IN-clause query (up to the 10-second `statementTimeout`). [6](#0-5) 

### Impact Explanation
Exhausting the DB connection pool (Spring Boot's default HikariCP pool is 10 connections) causes all subsequent requests to the rest-java service to queue or fail with connection-timeout errors. This makes the entire rest-java API unavailable for the duration of the attack. Since the endpoint is a public read API with no authentication, any external actor can trigger this. The `statementTimeout` of 10 seconds means each malicious request holds a connection for up to 10 seconds, so as few as 10 concurrent requests are sufficient to saturate the default pool.

### Likelihood Explanation
The attack requires zero privileges — the endpoint is publicly documented and unauthenticated. The attacker needs only an HTTP client capable of sending concurrent requests. The payload is a simple GET request with 100 query parameters, trivially scriptable. The attack is repeatable indefinitely because there is no IP-based rate limiting, no per-client throttle, and no circuit breaker on the endpoint. The `@Size(max = 100)` guard actually defines the maximum amplification factor per request, making it a known, stable attack surface.

### Recommendation
1. **Add rate limiting to the rest-java hooks endpoint.** Introduce a bucket4j or Spring-based rate-limiting filter (analogous to the `web3` `ThrottleConfiguration`) scoped to the `rest-java` module, applied globally or specifically to `/api/v1/accounts/*/hooks`.
2. **Reduce `MAX_REPEATED_QUERY_PARAMETERS` for EQ filters specifically.** The current value of 100 is the same as `MAX_LIMIT` (response page size). EQ-filter lists should be capped at a much lower value (e.g., 10) since legitimate pagination does not require 100 point-lookups per request.
3. **Enforce a DB connection pool size that accounts for concurrent IN-clause queries**, and configure connection-acquisition timeouts to fail fast rather than queue indefinitely.

### Proof of Concept
```bash
# Send 20 concurrent requests, each with 100 distinct hook.id=eq:N parameters
# This saturates a default 10-connection HikariCP pool with 100-element IN queries

for i in $(seq 1 20); do
  PARAMS=$(python3 -c "print('&'.join(f'hook.id=eq:{j}' for j in range($((i*100)), $((i*100+100)))))")
  curl -s "http://<mirror-node-host>/api/v1/accounts/0.0.1/hooks?$PARAMS" &
done
wait

# Subsequent legitimate requests will receive connection timeout errors
curl -v "http://<mirror-node-host>/api/v1/accounts/0.0.1/hooks"
# Expected: 503 / connection pool exhausted
```
