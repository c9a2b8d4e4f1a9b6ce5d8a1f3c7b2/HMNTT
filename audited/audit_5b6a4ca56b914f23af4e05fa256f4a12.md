### Title
Unbounded DB I/O Amplification via Synthetic Token Transfer Transactions Exploiting Non-Caching of Absent EVM Addresses

### Summary
`SyntheticLogListener.getCacheMisses()` issues a `findEvmAddressesByIds()` query for every cache-miss batch of entity IDs derived from synthetic contract log topics. Because `getCacheMisses()` only returns (and thus only causes Caffeine to cache) IDs that *have* an `evm_address`, IDs belonging to ordinary accounts (no `evm_address`) are never stored as cache entries by the loader itself. An unprivileged attacker who continuously submits fungible token transfer transactions involving large sets of previously-unseen account IDs can force a large `SELECT … WHERE id IN (…) AND length(evm_address) > 0` query against the `entity` table on every record-file boundary, with the DB scanning thousands of rows that return no results, amplifying I/O well beyond 30 % of baseline.

### Finding Description

**Exact code path:**

1. `TransferEventsGenerator.generate()` calls `SyntheticContractLogService.create(new TransferContractLog(…, senderId, receiverId, …))` for every pair of senders/receivers in a token transfer list. [1](#0-0) 

2. `TransferContractLog` encodes `senderId` and `receiverId` as trimmed long-zero EVM addresses in `topic1`/`topic2`. [2](#0-1) [3](#0-2) 

3. `SyntheticLogListener.onContractLog()` calls `fromTrimmedEvmAddress()` on `topic1`/`topic2`. For any Hedera account encoded as a long-zero address, this successfully parses back to an `EntityId`, and `populateSearchIds()` adds the numeric ID to `parserContext.evmAddressLookupIds` (an unbounded `HashSet<Long>`). [4](#0-3) [5](#0-4) [6](#0-5) 

4. At `onEnd()`, `getEvmCache().getAll(keys)` is called. Caffeine invokes `loadAll()` only for cache-miss keys, which delegates to `getCacheMisses()`. [7](#0-6) 

5. `getCacheMisses()` issues `findEvmAddressesByIds()` in batches of up to 30,000 IDs. `processBatch()` only inserts entries into `loadedFromDb` for rows that actually have an `evm_address`. The returned map is what Caffeine uses to populate the cache — IDs absent from the map are **not** stored as cache entries by the loader. [8](#0-7) [9](#0-8) 

6. The underlying query filters with `length(evm_address) > 0`, so the DB must evaluate every looked-up row but returns nothing for ordinary accounts. [10](#0-9) 

**Root cause / failed assumption:** The design assumes the Caffeine `LoadingCache` will cache negative results (absent keys) when `loadAll()` returns a partial map. It does not. Caffeine only caches keys present in the returned map. IDs for accounts without `evm_address` remain perpetual cache misses from the loader's perspective. (They are eventually cached by the explicit `getEvmCache().put(key.getId(), defaultValue)` call inside `updateTopicField()`, but only *after* the DB query has already executed for that record file.) [11](#0-10) 

**Why the post-query `put` does not fully mitigate:** `updateTopicField()` caches the default value only for IDs that appear in the *current* record file's updater list. If the attacker rotates through more than `maximumSize=500,000` unique IDs, earlier entries are evicted and become cache misses again on the next encounter. [12](#0-11) 

### Impact Explanation
Each record file (produced every ~2 seconds on mainnet) triggers one or more `SELECT … WHERE id IN (…) AND length(evm_address) > 0` queries with up to 30,000 IDs per batch. On a production node with millions of entities, each such query performs thousands of index lookups that return zero rows. Sustained at scale, this adds significant read I/O and CPU to the PostgreSQL backend. Because the feature is **enabled by default** (`syntheticContractLogEvmAddressLookup = true`), all production deployments are affected. [13](#0-12) 

### Likelihood Explanation
Hedera is a permissionless network. Any account holding a fungible token can submit `CryptoTransfer` transactions that include token transfers to many different recipient accounts. The attacker does not need to own the recipient accounts — they only need to reference valid account IDs. Transaction fees are low. By submitting batches of token transfers involving hundreds of unique account IDs per transaction, and rotating through more than 500,000 distinct IDs over time to defeat the cache, an attacker can sustain elevated DB I/O indefinitely. No privileged access is required.

### Recommendation
1. **Cache negative results explicitly:** After `getCacheMisses()` returns, insert a sentinel value (e.g., `ArrayUtils.EMPTY_BYTE_ARRAY`) for every key in the input set that is absent from the returned map, before returning from `loadAll()`. This prevents Caffeine from re-invoking `loadAll()` for the same absent keys.
2. **Bound `evmAddressLookupIds`:** Add a maximum size to `ParserContext.evmAddressLookupIds` (e.g., cap at `MAX_CACHE_LOAD_ENTRIES`) and log/drop excess IDs.
3. **Consider a rate limit or per-record-file cap** on the number of synthetic log EVM lookups to prevent a single record file from triggering arbitrarily large IN-clause queries.

### Proof of Concept
**Preconditions:**
- Mirror node running with default config (`syntheticContractLogEvmAddressLookup=true`, `syntheticContractLogs=true`).
- Attacker holds any fungible HTS token.
- Attacker has a list of N > 500,000 valid Hedera account IDs (none with a custom EVM address).

**Steps:**
1. In each ~2-second window (one record file), submit as many `CryptoTransfer` transactions as possible, each transferring a token amount from the attacker's account to a batch of unique recipient account IDs not previously used.
2. The importer processes the record file: for each transfer, `TransferEventsGenerator` creates a `TransferContractLog`; `SyntheticLogListener.onContractLog()` parses the recipient IDs from `topic2` and adds them to `evmAddressLookupIds`.
3. At `onEnd()`, `getEvmCache().getAll(keys)` finds all IDs as cache misses (first encounter) and calls `findEvmAddressesByIds()` with up to 30,000 IDs per batch.
4. The DB executes `SELECT evm_address,id FROM entity WHERE id IN (...30000 ids...) AND length(evm_address) > 0` — returns 0 rows but scans thousands of index entries.
5. After 500,000 unique IDs are exhausted, rotate back to the first batch; cache eviction ensures they are cache misses again.
6. Monitor PostgreSQL `pg_stat_user_tables` for `entity`: `idx_scan`, `idx_tup_read`, and `blks_read` increase sharply relative to the 24-hour baseline, exceeding 30%.

    private Map<Long, byte[]> getCacheMisses(Set<? extends Long> keys) {
        if (keys.isEmpty()) {
            return Collections.emptyMap();
        }

        Map<Long, byte[]> loadedFromDb = new HashMap<>(keys.size());
        if (keys.size() <= MAX_CACHE_LOAD_ENTRIES) {
            processBatch(loadedFromDb, entityRepository.findEvmAddressesByIds(keys));
        } else {
            final var keyIterator = keys.iterator();
            final var batches = Iterators.partition(keyIterator, MAX_CACHE_LOAD_ENTRIES);

            while (batches.hasNext()) {
                List<? extends Long> batch = batches.next();
                processBatch(loadedFromDb, entityRepository.findEvmAddressesByIds(batch));
            }
        }

        return loadedFromDb;
    }
