### Title
Cache TTL Asymmetry in `findStorageBatch()` Enables Low-Rate DB Amplification Attack

### Summary
A structural asymmetry between the `contractStateCache` TTL (2 seconds, write-based) and the `slotsPerContract` cache TTL (5 minutes, access-based) allows an unprivileged attacker to trigger a full 1500-slot batch database query with a single request every 2 seconds. Because the slot-key accumulator cache outlives the slot-value cache by 150x, each cache miss causes `findStorageBatch()` to re-query every previously seen slot for a contract, creating a 1500x DB amplification ratio at 0.5 RPS — well below all rate-limiting thresholds.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (lines 58–70) checks `contractStateCache` (TTL: `expireAfterWrite=2s`). On any miss it calls `findStorageBatch()`.

`findStorageBatch()` (lines 85–103) unconditionally collects **all** slot keys currently in the per-contract `slotsPerContract` cache (TTL: `expireAfterAccess=5m`, max 1500 entries) and issues a single `findStorageBatch` SQL query for all of them:

```java
// line 91 — collects every accumulated slot key for this contract
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
// ...
// line 103 — one DB query for all of them
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

The SQL query (repository lines 23–28):
```sql
select slot, value from contract_state
where contract_id = :contractId
and slot in (:slots)   -- up to 1500 entries
```

**Root cause:** The design assumes that slot values cached in `contractStateCache` will still be warm when the next request arrives. That assumption fails at exactly the 2-second boundary. The slot-key accumulator (`slotsPerContract`) is kept alive by access, so it grows to its maximum (1500) and stays there. The slot-value cache (`contractStateCache`) expires every 2 seconds regardless of access. The result is that a single request arriving just after the 2-second expiry re-triggers a batch query for all 1500 accumulated keys.

**Failed assumption:** The batch-caching design assumes that `contractStateCache` misses are rare (i.e., the cache is warm). It provides no guard against the case where the slot-key accumulator is full but the slot-value cache is cold.

**Exploit flow:**

1. **Populate the slot-key accumulator.** Send 1500 `eth_call` requests each reading a different storage slot of the same contract. This fills `slotsPerContract` to its maximum (1500 keys, TTL resets on each access).
2. **Wait 2 seconds.** All 1500 `contractStateCache` entries expire.
3. **Send one request.** Any `eth_call` reading any slot of that contract misses `contractStateCache`, enters `findStorageBatch()`, collects all 1500 cached slot keys, and issues a 1500-row `IN` query to PostgreSQL.
4. **Repeat every 2 seconds.** Each access to the `slotsPerContract` cache resets its 5-minute TTL, keeping the accumulator full. The attacker sustains 0.5 RPS indefinitely.

### Impact Explanation

Each attacker request at 0.5 RPS causes the database to execute a query scanning up to 1500 rows via an `IN` clause. Multiple attackers targeting different contracts multiply this linearly. The `contract_state` table is the hot path for all EVM `SLOAD` operations; sustained large `IN` queries on it increase I/O, connection pool pressure, and query planner overhead. The `db.statementTimeout` is 3000 ms, which is long enough for a 1500-row `IN` query to complete and return, so no timeout protection applies. The attack is silent from a rate-limiting perspective.

### Likelihood Explanation

No authentication is required. The attacker only needs a publicly accessible `eth_call` endpoint and knowledge of any contract with many storage slots (e.g., any ERC-20 token contract). Slot keys for standard contracts are deterministic and enumerable. The attack requires no special tooling beyond a script that sends one HTTP request every 2 seconds. It is repeatable indefinitely and scalable by targeting multiple contracts simultaneously.

### Recommendation

1. **Align TTLs.** Set `slotsPerContract` TTL to match or be shorter than `contractStateCache` TTL, so the slot-key accumulator cannot outlive the slot-value cache.
2. **Cap batch size independently of cache size.** Add a hard limit on the number of slots passed to `findStorageBatch()` regardless of how many keys are in the accumulator.
3. **Add per-contract rate limiting.** Throttle `findStorageBatch()` invocations per `contractId` per time window, not just global RPS.
4. **Invalidate slot keys on value expiry.** When `contractStateCache` entries expire, also evict the corresponding keys from `slotsPerContract` so the accumulator does not silently grow stale.

### Proof of Concept

```python
import requests, time, json

RPC = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract_with_many_slots>"

# Phase 1: populate slotsPerContract cache with 1500 distinct slot keys
for i in range(1500):
    slot = hex(i).zfill(66)  # storage slot i
    payload = {"to": CONTRACT, "data": "0x<eth_getStorageAt_selector>" + slot, "block": "latest"}
    requests.post(RPC, json=payload)

# Phase 2: trigger 1500-slot batch DB query every 2 seconds at 0.5 RPS
while True:
    time.sleep(2)  # wait for contractStateCache to expire
    payload = {"to": CONTRACT, "data": "0x<any_slot_read>", "block": "latest"}
    r = requests.post(RPC, json=payload)
    # Each request triggers findStorageBatch() with 1500 slots
    # DB receives: SELECT slot, value FROM contract_state WHERE contract_id=X AND slot IN (1500 values)
    print(f"Sent trigger, status={r.status_code}")
```

```java
    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
