Based on my investigation of the codebase, I have confirmed the following facts:

1. `getOutstandingAirdrops()` in `TokenAirdropsController.java` (lines 67–75) accepts `limit` up to `MAX_LIMIT = 100` with no authentication requirement.
2. The rest-java module contains **zero** rate-limiting code — no `ThrottleManager`, no bucket4j, no Spring Security rate limiting (grep across all of `rest-java/**` returned no matches for any throttle/rate-limit primitives).
3. The only filters registered in rest-java are `LoggingFilter`, `MetricsFilter`, and `ShallowEtagHeaderFilter` — none enforce request frequency limits.
4. The `web3` module's `ThrottleManager`/`ThrottleConfiguration` is entirely separate and does not apply to rest-java endpoints.
5. Any account ID can be queried with no authorization check.

### Title
Missing Application-Level Rate Limiting on Unauthenticated `/api/v1/accounts/{id}/airdrops/outstanding` Endpoint

### Summary
The `getOutstandingAirdrops()` endpoint in `TokenAirdropsController` is publicly accessible with no authentication and no application-level rate limiting. Any unauthenticated caller can issue unlimited requests at `limit=100` against arbitrary account IDs, driving unbounded database query load. The rest-java module contains no throttle mechanism whatsoever, unlike the `web3` module which uses bucket4j.

### Finding Description
**Code location:** `rest-java/src/main/java/org/hiero/mirror/restjava/controller/TokenAirdropsController.java`, `getOutstandingAirdrops()`, lines 67–75.

```java
@GetMapping(value = "/outstanding")
TokenAirdropsResponse getOutstandingAirdrops(
        @PathVariable EntityIdParameter id,
        @RequestParam(defaultValue = DEFAULT_LIMIT) @Positive @Max(MAX_LIMIT) int limit,
        ...
```

`MAX_LIMIT = 100` (from `Constants.java` line 35). The `@Max(MAX_LIMIT)` annotation only caps the result page size — it does not limit request frequency.

**Root cause:** The rest-java module has no rate-limiting layer. The config classes (`RestJavaConfiguration`, `WebMvcConfiguration`, `LoggingFilter`, `MetricsFilter`) contain no throttle logic. The `ThrottleManager`/bucket4j infrastructure exists only in the `web3` module (`web3/src/main/java/org/hiero/mirror/web3/throttle/`). The failed assumption is that infrastructure-level rate limiting (e.g., Traefik middleware, as configured for the Rosetta service in `charts/hedera-mirror-rosetta/values.yaml` lines 157–160) is always present in front of rest-java deployments.

**Exploit flow:**
1. Attacker identifies any valid account ID (e.g., from public ledger data).
2. Attacker sends a tight loop of `GET /api/v1/accounts/0.0.X/airdrops/outstanding?limit=100` with no credentials.
3. Each request triggers a full database query via `TokenAirdropService.getAirdrops()`.
4. No server-side counter, bucket, or reject path exists in the application to stop this.

**Existing checks and why they fail:**
- `@Max(MAX_LIMIT)`: caps result size only, not request rate.
- `ShallowEtagHeaderFilter`: only adds ETag caching headers, no rate control.
- `LoggingFilter` / `MetricsFilter`: observability only.
- No `@PreAuthorize`, `@Secured`, or Spring Security configuration exists in rest-java.

### Impact Explanation
An attacker can exhaust database connection pool and CPU resources by issuing thousands of unauthenticated read queries per second. Each request at `limit=100` forces a full paginated scan of the `token_airdrop` table for the given account. This can degrade or deny service for all users of the mirror node REST API. The endpoint also enables complete enumeration of outstanding airdrops for any account without any cost or credential.

### Likelihood Explanation
Exploitation requires zero privileges, zero credentials, and only a basic HTTP client. The attack is trivially scriptable (`while true; do curl ...; done`). Any external actor with network access to the API can execute it. The attack is repeatable and persistent until infrastructure-level mitigations (if any) intervene.

### Recommendation
Add application-level rate limiting to the rest-java module, mirroring the pattern already used in the `web3` module:
- Introduce a `ThrottleManager` bean (or equivalent bucket4j `Bucket`) in `rest-java/src/main/java/org/hiero/mirror/restjava/config/`.
- Apply it as a `OncePerRequestFilter` or as an interceptor in `WebMvcConfiguration` for all `/api/v1/accounts/*/airdrops/*` paths.
- Alternatively, enforce Traefik/ingress rate-limit middleware for rest-java deployments (as is done for Rosetta), and document this as a required deployment prerequisite.

### Proof of Concept
```bash
# No credentials required. Run from any machine with network access.
while true; do
  curl -s "https://<mirror-node-host>/api/v1/accounts/0.0.1234/airdrops/outstanding?limit=100" \
    -o /dev/null &
done
# Spawns unlimited concurrent unauthenticated DB-hitting requests.
# No 429 or rejection will be returned from the application layer.
