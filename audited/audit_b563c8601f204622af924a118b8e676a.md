### Title
Zero-Length `initialTransactionId` Bypasses Null Guard, Streams Default `TransactionID` to gRPC Subscribers Breaking Chunk Reassembly

### Summary
An unprivileged attacker can submit a `ConsensusSubmitMessage` transaction with `chunkInfo.initialTransactionID` set to a default (all-zero) protobuf `TransactionID`. The importer stores this as a zero-length byte array. When `poll()` streams the message, `parseTransactionID` only guards against `null` — not an empty array — so `TransactionID.parseFrom(new byte[0])` silently succeeds and returns a default `TransactionID` (account `0.0.0`, timestamp `0.0`). Every subscriber using `initialTransactionID` for chunk correlation receives a structurally valid but semantically empty key, causing incorrect chunk grouping and data loss for multi-chunk messages.

### Finding Description

**Step 1 — Importer stores zero-length bytes**

In `ConsensusSubmitMessageTransactionHandler.doUpdateTransaction()`:

```java
if (chunkInfo.hasInitialTransactionID()) {                          // line 81 — true even for default instance
    topicMessage.setInitialTransactionId(
            chunkInfo.getInitialTransactionID().toByteArray());     // line 82-83 — returns [] for default proto
}
```

`TransactionID.getDefaultInstance().toByteArray()` returns `[]` in protobuf-java because all fields are at their default values and are not serialized. `hasInitialTransactionID()` returns `true` because the field was explicitly set by the attacker. The result is `initialTransactionId = new byte[0]` stored in the DB. [1](#0-0) 

**Step 2 — `poll()` streams the record without validation**

`PollingTopicListener.poll()` queries the repository and emits every `TopicMessage` as-is:

```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));  // line 61
```

No validation of `initialTransactionId` content occurs here. [2](#0-1) 

**Step 3 — `parseTransactionID` null-only guard is bypassed**

```java
private TransactionID parseTransactionID(byte[] transactionIdBytes, ...) {
    if (transactionIdBytes == null) {   // line 119 — only null check, not length check
        return null;
    }
    try {
        return TransactionID.parseFrom(transactionIdBytes);  // line 123 — succeeds on []
    } catch (InvalidProtocolBufferException e) { ... }
}
```

`TransactionID.parseFrom(new byte[0])` does not throw; it returns `TransactionID.getDefaultInstance()` — a non-null object with `accountID = 0.0.0`, `transactionValidStart = 0.0`, `nonce = 0`, `scheduled = false`. [3](#0-2) 

**Step 4 — Default `TransactionID` is set on the chunk and streamed**

```java
if (transactionID != null) {                              // line 103 — true for default instance
    chunkBuilder.setInitialTransactionID(transactionID);  // line 104 — sets empty key
}
```

The subscriber receives a `ConsensusTopicResponse` with `chunkInfo.initialTransactionID = {0.0.0, 0.0, 0, false}`. [4](#0-3) 

**Failed assumption**: The code assumes `transactionIdBytes != null` implies a valid, non-empty serialized `TransactionID`. This is false: protobuf allows a message with all-default fields to serialize to zero bytes, and `parseFrom` on zero bytes returns the default instance without error.

### Impact Explanation

SDK clients (Hedera Java SDK and compatible) use `initialTransactionID` as the correlation key to group chunks of a multi-chunk message into a single logical message. When an attacker submits multiple chunk transactions each carrying an empty `initialTransactionID`, all such chunks share the same correlation key (`0.0.0 / 0.0`). The SDK incorrectly groups them together regardless of which logical message they belong to. Legitimate multi-chunk messages whose chunks are interleaved with tampered ones will be mis-assembled or dropped entirely, causing **data corruption and data loss** for all subscribers of the affected topic. Severity is high for topics carrying critical application data over multi-chunk messages.

### Likelihood Explanation

Preconditions are minimal: the attacker needs only a funded Hedera account (a few cents of HBAR per transaction) and a target topic without a `submitKey` (the majority of public HCS topics). The attack is fully repeatable and scriptable. No privileged access, no key material beyond the attacker's own account key, and no race condition is required. The Hedera network does not validate the semantic content of `chunkInfo.initialTransactionID`.

### Recommendation

In `parseTransactionID`, add an explicit length guard before parsing:

```java
private TransactionID parseTransactionID(byte[] transactionIdBytes, long topicId, long sequenceNumber) {
    if (transactionIdBytes == null || transactionIdBytes.length == 0) {
        return null;
    }
    try {
        return TransactionID.parseFrom(transactionIdBytes);
    } catch (InvalidProtocolBufferException e) {
        log.error("Failed to parse TransactionID for topic {} sequence number {}", topicId, sequenceNumber);
        return null;
    }
}
```

Additionally, in `ConsensusSubmitMessageTransactionHandler.doUpdateTransaction()`, skip storing the bytes when the serialized form is empty:

```java
if (chunkInfo.hasInitialTransactionID()) {
    byte[] bytes = chunkInfo.getInitialTransactionID().toByteArray();
    if (bytes.length > 0) {
        topicMessage.setInitialTransactionId(bytes);
    }
}
```

### Proof of Concept

```java
// 1. Build a ConsensusSubmitMessage with an empty initialTransactionID
ConsensusMessageChunkInfo chunkInfo = ConsensusMessageChunkInfo.newBuilder()
    .setNumber(1)
    .setTotal(2)
    .setInitialTransactionID(TransactionID.getDefaultInstance())  // all-zero, serializes to []
    .build();

ConsensusSubmitMessageTransactionBody body = ConsensusSubmitMessageTransactionBody.newBuilder()
    .setTopicID(targetTopicId)
    .setMessage(ByteString.copyFromUtf8("chunk1"))
    .setChunkInfo(chunkInfo)
    .build();

// 2. Sign and submit to the Hedera network with any funded account.
//    Network accepts it — no semantic validation of initialTransactionID.

// 3. Subscribe via gRPC:
//    ConsensusTopicQuery → subscribeTopic
//    poll() streams the stored TopicMessage (initialTransactionId = [])
//    parseTransactionID(new byte[0]) returns TransactionID.getDefaultInstance() (non-null)
//    Subscriber receives chunkInfo.initialTransactionID = {0.0.0, 0.0, 0, false}

// 4. Submit a second chunk (chunkNum=2) with the same empty initialTransactionID.
//    Both chunks share the same correlation key → SDK groups them as one message.
//    Any legitimate 2-chunk message on the same topic whose chunks are interleaved
//    with these tampered chunks will be mis-assembled or silently dropped.
```
