### Title
Unprivileged User Can Force Short Cache-Control (max-age=5) on SCHEDULECREATE Transaction Queries, Amplifying CDN/Proxy Upstream Load

### Summary
Any unauthenticated user can query a recently-created, successful SCHEDULECREATE transaction by its transaction ID (without the `scheduled` query parameter) and receive a response with `cache-control: public, max-age=5`. Because this header is propagated to CDN/proxy caches and used as the Redis TTL, the resource will be re-fetched from the origin every 5 seconds instead of the default longer TTL, amplifying upstream load with no economic cost to the attacker.

### Finding Description
**Exact code location:** `rest/transactions.js`, function `getTransactionsByIdOrHashCacheControlHeader`, lines 895–916.

```js
const getTransactionsByIdOrHashCacheControlHeader = (isTransactionHash, scheduledParamExists, transactions) => {
  if (isTransactionHash || scheduledParamExists) {
    return {};
  }
  let header = {};
  for (const transaction of transactions) {
    if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
      const elapsed = utils.nowInNs() - transaction.consensus_timestamp;
      if (elapsed < maxScheduledTransactionConsensusTimestampRangeNs) {   // default: 1 day
        header = SHORTER_CACHE_CONTROL_HEADER;   // { 'cache-control': 'public, max-age=5' }
      }
    } else if (transaction.scheduled) {
      return {};
    }
  }
  return header;
};
```

**Root cause:** The function unconditionally emits `SHORTER_CACHE_CONTROL_HEADER` (`public, max-age=5`) for any successful SCHEDULECREATE transaction whose `consensus_timestamp` is within the last `maxScheduledTransactionConsensusTimestampRangeNs` (default 1 day). There is no caller-identity check, no rate limit, and no restriction on who may trigger this path.

**Exploit flow:**
1. Attacker observes any recently-created successful SCHEDULECREATE transaction on the public Hedera network (transaction IDs are public).
2. Attacker issues `GET /api/v1/transactions/{transactionId}` — no `scheduled` query param, no hash format.
3. `isTransactionHash = false`, `scheduledParamExists = false` → the two early-exit guards at line 896 are bypassed.
4. The DB row has `type = 42` (SCHEDULECREATE), `result ∈ SUCCESS_PROTO_IDS`, and `consensus_timestamp` within the last day → `header = SHORTER_CACHE_CONTROL_HEADER`.
5. `responseHandler.js` (line 37) merges `res.locals[responseHeadersLabel]` into the response headers, emitting `cache-control: public, max-age=5`.
6. `responseCacheHandler.js` (line 96) reads this header to set the Redis TTL to 5 seconds via `getCacheControlExpiryOrDefault`.
7. Any CDN or reverse proxy honouring `Cache-Control` will re-fetch the resource every 5 seconds.

**Why existing checks are insufficient:**
- The `isTransactionHash` guard (line 896) is bypassed simply by using the transaction-ID URL form.
- The `scheduledParamExists` guard (line 896) is bypassed by omitting the `?scheduled=` parameter.
- The `transaction.scheduled` early-return (line 910–911) only fires when a *scheduled* child transaction is already present in the result set; it does not fire when only the parent SCHEDULECREATE row exists (the exact scenario the attacker targets).
- No authentication, no rate limiting, and no IP-based throttling is present in this code path.

### Impact Explanation
Every CDN edge node or reverse proxy that caches the targeted URL will re-validate against the origin every 5 seconds. An attacker who publishes or repeatedly requests a handful of such URLs can sustain a continuous amplified load on the mirror-node API tier. The window of exploitability per transaction is up to 1 day (the default `maxScheduledTransactionConsensusTimestampRangeNs`). Because SCHEDULECREATE transactions are common on Hedera, an attacker can maintain a large set of "live" URLs simultaneously. No economic damage to end-users occurs, but origin infrastructure load is amplified.

### Likelihood Explanation
The attack requires zero privileges: no API key, no account, no HBAR. The attacker only needs to read publicly-available transaction IDs from any Hedera explorer or the mirror-node `/api/v1/transactions` list endpoint. The exploit is trivially repeatable and scriptable. The only cost is the HTTP request itself.

### Recommendation
1. **Bound the short-TTL window more tightly.** Replace the 1-day `maxScheduledTransactionConsensusTimestampRangeNs` window with the actual schedule expiry time stored on the schedule entity, so the short TTL is only applied when the scheduled transaction is genuinely still pending.
2. **Add a `private` or `no-store` directive** for responses that are inherently user/query-specific rather than `public`, preventing CDN caching of the short-TTL response entirely.
3. **Apply rate limiting** at the API gateway layer on the `/api/v1/transactions/:id` endpoint to limit how frequently a single IP can request the same transaction ID.
4. **Cap the minimum CDN TTL** at the infrastructure level so that even if `max-age=5` is emitted, the CDN enforces a floor (e.g., 30 seconds) before re-fetching.
