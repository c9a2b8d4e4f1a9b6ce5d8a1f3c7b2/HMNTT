### Title
Connection Pool Exhaustion via Unbounded Read Transaction During Network Partition in gRPC Service

### Summary
The `transactionOperationsReadOnly()` bean in `GrpcConfiguration.java` creates a `TransactionTemplate` with no timeout configured (`setTimeout()` is never called, leaving the default of `-1` meaning indefinite). This template is used in the publicly-accessible `getNodes()` gRPC endpoint. During a network partition between the gRPC service and the database, JDBC sockets hang indefinitely because no socket-level timeout is configured for the gRPC module's connection pool, causing HikariCP connections to be held open permanently and exhausting the pool for all subsequent callers.

### Finding Description
**Exact code location:**

`GrpcConfiguration.java`, lines 21–25:
```java
TransactionOperations transactionOperationsReadOnly(PlatformTransactionManager transactionManager) {
    var transactionTemplate = new TransactionTemplate(transactionManager);
    transactionTemplate.setReadOnly(true);
    return transactionTemplate;  // No setTimeout() call — default is TIMEOUT_DEFAULT (-1)
}
``` [1](#0-0) 

This bean is injected into `NetworkServiceImpl` and called on every page fetch of the public `getNodes()` gRPC endpoint:

```java
private Flux<AddressBookEntry> page(AddressBookContext context) {
    return transactionOperations.execute(t -> {
        ...
        var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(...);
        ...
    });
}
``` [2](#0-1) 

The `getNodes()` call is triggered by any external gRPC client with no authentication requirement. [3](#0-2) 

**Root cause — two compounding gaps:**

1. **No Spring-level transaction timeout**: `TransactionTemplate.setTimeout()` is never called. Spring's default is `TIMEOUT_DEFAULT = -1` (no timeout), so the transaction (and its held connection) is never forcibly rolled back by the framework.

2. **No JDBC socket timeout for the gRPC pool**: The gRPC module has no dedicated `DatabaseConfiguration.java`. It inherits `CommonConfiguration`'s HikariCP setup, which reads from `spring.datasource.hikari` properties. [4](#0-3) 

No `socketTimeout` JDBC property is set for the gRPC module's pool. The PostgreSQL JDBC driver defaults `socketTimeout` to `0` (infinite). During a TCP-level network partition, the OS socket blocks indefinitely — the JVM thread and the HikariCP connection are both frozen.

**Why the existing `statementTimeout` check fails:**

The documented `hiero.mirror.grpc.db.statementTimeout = 10000` is a PostgreSQL server-side `statement_timeout`. It is enforced by the database engine and requires a live TCP path to the server to fire. During a network partition, the TCP connection is hung at the socket layer before any query bytes reach PostgreSQL, so the server never receives the query and never fires `statement_timeout`. The mitigation is therefore completely ineffective for the partition scenario. [5](#0-4) 

### Impact Explanation
HikariCP's pool is finite. Each in-flight `transactionOperations.execute()` call holds one connection for the duration of the transaction. With no socket timeout and no Spring transaction timeout, every connection acquired during a partition is permanently frozen. Once the pool is exhausted, HikariCP's `connectionTimeout` (default 30 s) causes all new DB-requiring gRPC calls to throw `SQLTransientConnectionException`, making the entire gRPC service unavailable for DB-backed operations. This is a complete denial-of-service for the gRPC API's address book and topic message functionality.

### Likelihood Explanation
The `getNodes()` endpoint is unauthenticated and publicly reachable. An attacker does not need to cause the network partition themselves — they only need to:
- Maintain a modest number of concurrent `getNodes()` streaming calls (the default pool size is small), timed to coincide with any transient network disruption between the gRPC pod and the database (cloud provider blip, rolling restart, failover).
- Alternatively, an attacker with network-adjacent access (same VPC, compromised sidecar) can induce the partition deliberately via firewall rules or traffic shaping.

The `maxConcurrentCallsPerConnection` of 5 limits calls per client connection but does not bound total server-side DB connections across all clients. [6](#0-5) 

### Recommendation
Apply all three fixes together:

1. **Set a Spring transaction timeout** on the read-only template:
   ```java
   transactionTemplate.setTimeout(10); // seconds, matches statementTimeout
   ``` [7](#0-6) 

2. **Set a JDBC socket timeout** via HikariCP's `connectionInitSql` or the `socketTimeout` datasource property in the gRPC application configuration so that TCP-level hangs are bounded:
   ```yaml
   spring.datasource.hikari.data-source-properties.socketTimeout: 10
   ```

3. **Set HikariCP `connectionTimeout`** explicitly to a low value (e.g., 5 s) so pool exhaustion is detected quickly rather than after the 30 s default.

### Proof of Concept
1. Deploy the gRPC service with default configuration against a PostgreSQL backend.
2. Open `N` concurrent gRPC streams calling `getNodes()` (where `N` equals the HikariCP pool size, typically 10).
3. While those calls are in-flight (waiting for DB results), introduce a network partition between the gRPC pod and PostgreSQL (e.g., `iptables -I OUTPUT -p tcp --dport 5432 -j DROP` on the gRPC host).
4. Observe: all `N` threads block indefinitely in `transactionOperations.execute()` holding their HikariCP connections.
5. Issue any new gRPC call that requires DB access (e.g., a fresh `getNodes()` or topic subscription).
6. Observe: the call fails after HikariCP's `connectionTimeout` (30 s default) with `Unable to acquire JDBC Connection`, confirming pool exhaustion and full service denial.

    public Flux<AddressBookEntry> getNodes(AddressBookFilter filter) {
        var fileId = filter.getFileId();
        if (!getValidFileIds().contains(fileId)) {
            throw new IllegalArgumentException(INVALID_FILE_ID);
        }

        long addressBookTimestamp = addressBookRepository
                .findLatestTimestamp(fileId.getId())
                .orElseThrow(() -> new EntityNotFoundException(fileId));
        long nodeStakeTimestamp = nodeStakeRepository.findLatestTimestamp().orElse(NODE_STAKE_EMPTY_TABLE_TIMESTAMP);
        var nodeStakeMap = nodeStakeRepository.findAllStakeByConsensusTimestamp(nodeStakeTimestamp);
        var context = new AddressBookContext(addressBookTimestamp, nodeStakeMap);

        return Flux.defer(() -> page(context))
                .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                        .jitter(0.5)
                        .withFixedDelay(addressBookProperties.getPageDelay())
                        .withScheduler(Schedulers.boundedElastic()))
                .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Querying for address book: {}", filter))
                .doOnComplete(() -> log.info("Retrieved {} nodes from the address book", context.getCount()));
    }

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
    }
