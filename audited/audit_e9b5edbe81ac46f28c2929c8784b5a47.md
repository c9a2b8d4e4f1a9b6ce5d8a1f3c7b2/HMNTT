### Title
Missing EVM Address Length Validation Causes Unhandled `BufferUnderflowException` in `getByEvmAddressAndType()`

### Summary
`decodeEvmAddress()` in `GraphQlUtils.java` performs no length check on the decoded byte array, allowing any unprivileged user to supply a valid even-length hex string shorter than 8 hex characters (4 bytes). This causes `ByteBuffer.getInt()` (or subsequent `getLong()` calls) to throw a `java.nio.BufferUnderflowException` that `CustomExceptionResolver` does not handle, resulting in a Spring GraphQL `INTERNAL_ERROR` response instead of a proper `ValidationError`.

### Finding Description
**Exact code path:**

1. `AccountController.account()` passes the raw user-supplied `evmAddress` string directly to `entityService.getByEvmAddressAndType()`. [1](#0-0) 

2. `EntityServiceImpl.getByEvmAddressAndType()` calls `decodeEvmAddress(evmAddress)` and immediately wraps the result in a `ByteBuffer`, then reads 4 + 8 + 8 = 20 bytes unconditionally (the third read is conditional, but the first two are not). [2](#0-1) 

3. `decodeEvmAddress()` strips the optional `0x` prefix and calls `Hex.decodeHex()`. It catches `DecoderException` (odd-length or non-hex input) and rethrows as `IllegalArgumentException`, but **performs no check on the length of the resulting byte array**. [3](#0-2) 

4. `CustomExceptionResolver.resolveToSingleError()` only handles `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`. `BufferUnderflowException` (a `RuntimeException`) is not in this list, so the method returns `null`, and Spring GraphQL falls back to a generic `INTERNAL_ERROR` response. [4](#0-3) 

**Root cause:** The failed assumption is that `decodeEvmAddress()` will always return exactly 20 bytes. There is no guard enforcing this invariant before the `ByteBuffer` reads.

**Exploit flow:**
- Input `"aabb"` (2 hex chars → 1 byte): `buffer.getInt()` throws `BufferUnderflowException` immediately.
- Input `"aabbccdd"` (8 hex chars → 4 bytes): `buffer.getInt()` succeeds, `buffer.getLong()` throws.
- Input of any even-length hex string with fewer than 40 hex characters (< 20 bytes) will eventually underflow.

### Impact Explanation
Every such request returns an HTTP 200 with a GraphQL `INTERNAL_ERROR` classification instead of a `ValidationError`. This:
- Leaks the fact that the server encountered an unhandled internal exception (error-type oracle), distinguishable from normal validation failures.
- Can be used to repeatedly trigger unhandled exceptions, polluting server logs and metrics.
- Prevents proper client-side error handling (clients cannot distinguish a real internal failure from a bad-input failure).

Severity: **Medium**. No sensitive data is directly exposed, but the unhandled exception path is fully attacker-controlled and repeatable.

### Likelihood Explanation
No authentication or special privilege is required. Any user who can send a GraphQL query to the `account` endpoint can trigger this with a single crafted request. The input constraint (even-length hex string shorter than 40 chars) is trivially satisfied. The attack is fully repeatable and requires no prior knowledge of the system.

### Recommendation
Add an explicit length check in `decodeEvmAddress()` (or at the call site in `getByEvmAddressAndType()`) to enforce exactly 20 bytes:

```java
// In GraphQlUtils.decodeEvmAddress(), after Hex.decodeHex():
byte[] decoded = Hex.decodeHex(evmAddress);
if (decoded.length != 20) {
    throw new IllegalArgumentException(
        "evmAddress must be exactly 20 bytes (40 hex chars), got: " + decoded.length);
}
return decoded;
```

Because `IllegalArgumentException` IS handled by `CustomExceptionResolver`, this would produce a proper `ValidationError` response instead of an `INTERNAL_ERROR`.

### Proof of Concept
Send the following GraphQL request to the mirror node GraphQL endpoint (no authentication required):

```graphql
query {
  account(input: { evmAddress: "aabb" }) {
    id
  }
}
```

**Expected (correct) behavior:** GraphQL `ValidationError` with message about invalid address length.

**Actual behavior:** GraphQL `INTERNAL_ERROR` response, server log shows unhandled `java.nio.BufferUnderflowException` at `EntityServiceImpl.getByEvmAddressAndType()` line 37, not caught by `CustomExceptionResolver`.

Variants:
- `"00"` (1 byte) → `BufferUnderflowException` at `getInt()`
- `"000000000000000000000000"` (12 bytes, all zeros) → `getInt()==0`, `getLong()==0`, then `getLong()` underflows at line 38
- Any even-length hex string between 2 and 38 hex characters triggers the bug

```java
    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }
```

```java
            return GraphqlErrorBuilder.newError()
                    .errorType(ErrorType.ValidationError)
                    .message(ex.getMessage())
                    .path(env.getExecutionStepInfo().getPath())

                    .build();
```
