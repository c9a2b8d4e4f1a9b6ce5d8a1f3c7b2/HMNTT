### Title
Connection Pool Exhaustion via Unauthenticated `/api/v1/network/stake` Flooding During DB Degradation

### Summary
`NetworkStakeRepository.findLatest()` is called on every unauthenticated `GET /api/v1/network/stake` request with no rate limiting and no result caching in the rest-java service. During any DB slowdown or network partition, each in-flight request holds a HikariCP connection for up to the full `statementTimeout` (10 seconds). An unprivileged attacker sending concurrent requests can exhaust the shared connection pool, causing all other rest-java API endpoints (not just `/network/stake`) to fail with connection acquisition timeouts.

### Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (lines 126–130) calls `NetworkService.getLatestNetworkStake()` (lines 52–56 of `NetworkServiceImpl`), which calls `networkStakeRepository.findLatest()` (lines 12–19 of `NetworkStakeRepository`):

```sql
select *
from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
``` [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — three compounding gaps:**

1. **No rate limiting on rest-java endpoints.** The throttle/rate-limit infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists exclusively in the `web3` module. There is no equivalent in rest-java. [4](#0-3) 

2. **No caching of the result.** `getLatestNetworkStake()` carries no `@Cacheable` annotation. A grep across all rest-java service/repository Java files returns zero `@Cacheable` hits. Compare this to the grpc module's `NodeStakeRepository`, which caches the equivalent query result. [5](#0-4) 

3. **Shared HikariCP pool with a 10-second statement timeout window.** The rest-java service uses a single HikariCP pool (default size 10) shared across all endpoints. The documented `statementTimeout` is 10,000 ms. [6](#0-5) [7](#0-6) 

**Exploit flow:**

During a DB slowdown or network partition (TCP connections do not fail immediately — they hang until the OS/DB timeout fires), each `findLatest()` call blocks its thread and holds a HikariCP connection for up to 10 seconds. With a pool of 10 connections, an attacker sustaining just 1–2 concurrent requests per second for 10 seconds fills the pool. All subsequent requests to any endpoint (fees, supply, nodes, exchange rate) that need a DB connection receive a HikariCP `SQLTransientConnectionException` (connection acquisition timeout), producing HTTP 500 responses.

**Why existing checks are insufficient:**

- `statementTimeout = 10000ms` limits individual query duration but does not prevent pool exhaustion — it merely sets the upper bound of the hold window. During that window, N concurrent requests × 10 s = N×10 connection-seconds consumed.
- HikariCP's own `connectionTimeout` (default 30 s) causes waiting threads to fail, but by then the pool is already saturated and all other endpoints are already failing.
- There is no per-IP or global request rate limit on the rest-java `/network/stake` endpoint.

### Impact Explanation

The entire rest-java API service shares one connection pool. Exhausting it via `/network/stake` denies service to all other endpoints (`/network/fees`, `/network/supply`, `/network/nodes`, `/network/exchangerate`, etc.) for the duration of the attack. This is a full service-level DoS, not limited to the targeted endpoint. Severity: **High** (availability impact across all rest-java API consumers).

### Likelihood Explanation

- **No authentication required** — the endpoint is fully public.
- **No rate limiting** — the attacker faces no application-layer throttle.
- **Precondition (DB degradation)** is realistic: it can occur naturally (DB maintenance, replica lag, cloud network blip) or be induced by a separate DB-layer attack. The attacker does not need to cause the partition; they only need to detect or wait for it and then amplify its impact.
- **Low sophistication** — a simple HTTP flood with a handful of concurrent connections suffices.
- **Repeatable** — the attack can be sustained for as long as the DB is degraded.

### Recommendation

1. **Add result caching** on `getLatestNetworkStake()` (e.g., `@Cacheable` with a short TTL such as 30–60 seconds). The `network_stake` table is updated at most once per staking period (daily), so stale data risk is negligible and this eliminates the DB round-trip for the vast majority of requests.
2. **Add rate limiting** to the rest-java service for public endpoints, mirroring the bucket4j-based throttle already present in the web3 module.
3. **Reduce `statementTimeout`** for this specific query (it is a simple indexed lookup; 1–2 seconds is more than sufficient).
4. **Consider a dedicated or bounded connection pool** for lightweight read-only endpoints to prevent cross-endpoint pool starvation.

### Proof of Concept

**Precondition:** Introduce artificial DB latency (e.g., via `tc netem` on the DB host, or a `pg_sleep`-based lock) so that `SELECT max(consensus_timestamp) FROM network_stake` takes > 5 seconds.

**Steps:**
```bash
# Send 20 concurrent requests to /network/stake
for i in $(seq 1 20); do
  curl -s http://<rest-java-host>/api/v1/network/stake &
done
wait

# Immediately probe an unrelated endpoint
curl -v http://<rest-java-host>/api/v1/network/fees
```

**Expected result:** The `/network/fees` request (and all other DB-backed endpoints) returns HTTP 500 with a HikariCP connection pool timeout error, demonstrating cross-endpoint DoS caused solely by flooding the unauthenticated `/network/stake` endpoint.

        var jdbcUrl = dataSourceProperties.determineUrl();

            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);
