### Title
Connection Pool Exhaustion DoS via Concurrent Unauthenticated GET /api/v1/schedules Requests

### Summary
The `getSchedules()` handler in `rest/schedules.js` issues up to 3 concurrent database queries per request against a shared `pg.Pool` capped at 10 connections by default. Because the REST API has no rate limiting or concurrency cap, an unauthenticated attacker sending as few as 4 simultaneous requests can exhaust all pool connections, causing every subsequent legitimate request to block for up to `connectionTimeoutMillis` (20 s default) before failing, producing sustained service unavailability.

### Finding Description
**Code path:**

`rest/schedules.js` `getSchedules()` (lines 234–282):
- Line 241: `await pool.queryQuietly(schedulesQuery, params)` — acquires 1 pool connection, holds it for up to `statement_timeout` (20 s).
- Lines 261–264: `await Promise.all([pool.queryQuietly(entityQuery, ...), pool.queryQuietly(signatureQuery, ...)])` — acquires 2 more pool connections simultaneously.

Each invocation of `getSchedules` therefore holds **up to 3 pool connections concurrently** for the full query duration.

**Pool configuration** (`rest/dbpool.js`, lines 7–16):
```
max: config.db.pool.maxConnections          // default 10
connectionTimeoutMillis: config.db.pool.connectionTimeout  // default 20 000 ms
statement_timeout: config.db.pool.statementTimeout         // default 20 000 ms
```

**`queryQuietly` implementation** (`rest/utils.js`, lines 1518–1520): when no `preQueryHint` is supplied (the normal path for all schedules queries), it calls `this.query(query, params)` directly on the pool. The `pg` library queues the call and blocks until a connection is available or `connectionTimeoutMillis` elapses.

**No rate limiting exists on the REST API.** `rest/server.js` (lines 67–116) registers only: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and an optional `responseCacheCheckHandler`. There is no per-IP limit, no concurrency cap, and no token-bucket middleware for the `/schedules` route. The throttle subsystem (`web3/src/main/java/.../ThrottleConfiguration.java`) belongs to the separate Web3 Java service and does not apply here.

**Root cause:** The design assumes that the number of concurrent in-flight HTTP requests will remain well below `maxConnections / 3`. No mechanism enforces this assumption.

### Impact Explanation
With 4 concurrent attacker requests each holding 3 connections (4 × 3 = 12 > 10), the pool is fully exhausted. Every subsequent request — from any user — blocks for 20 seconds then receives a connection-timeout error. The attacker needs only to keep 4 long-running requests alive (trivially achieved by targeting a large dataset or relying on the 20 s `statement_timeout`) to maintain the outage indefinitely. All three queries in `getSchedules` are affected, as are `getScheduleById` (line 120) and every other endpoint sharing the same global `pool`. The impact is complete denial of service for the `/schedules` endpoint and potential degradation of all other REST endpoints sharing the pool.

### Likelihood Explanation
The attack requires zero authentication, zero special knowledge, and only 4 concurrent HTTP connections — well within reach of any script or commodity tool (`ab`, `wrk`, `curl` in parallel). The endpoint is publicly documented in the OpenAPI spec (`rest/api/v1/openapi.yml`, lines 1030–1050). The default pool size of 10 is small, and the 20 s statement timeout gives the attacker a long hold window. The attack is trivially repeatable and automatable.

### Recommendation
1. **Add a per-IP (or global) concurrency / rate limit** to the REST API middleware stack in `rest/server.js`, e.g. via `express-rate-limit` or a semaphore, before the route handlers.
2. **Increase `maxConnections`** or, better, deploy a connection pooler (PgBouncer in transaction mode) in front of PostgreSQL so the application pool is not the hard ceiling.
3. **Reduce `connectionTimeoutMillis`** to fail fast (e.g. 2–3 s) so exhausted-pool errors surface quickly rather than holding HTTP workers for 20 s.
4. **Reduce `statement_timeout`** for read-only list queries to limit how long each connection is held.
5. Consider enabling the Redis response cache (`config.cache.response.enabled`) so repeated identical requests are served without touching the pool.

### Proof of Concept
```bash
# Exhaust the 10-connection pool with 4 concurrent long-running requests
for i in 1 2 3 4; do
  curl -s "http://<mirror-node>/api/v1/schedules?limit=25" &
done
wait

# Now issue a legitimate request — it will block for ~20 s then fail
time curl -v "http://<mirror-node>/api/v1/schedules"
# Expected: connection timeout / 503 after ~20 000 ms
```
