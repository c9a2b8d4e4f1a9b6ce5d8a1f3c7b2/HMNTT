### Title
Unbounded Reactive Subscription Accumulation via Open-Ended Topic Subscriptions (No Idle Timeout, No Global Subscriber Limit)

### Summary
An unprivileged user can open an unlimited number of gRPC topic subscriptions against any valid topic with no `endTime` and no `limit`. Because `pastEndTime()` returns `Flux.never()` when `endTime` is absent, the server-side reactive chain never self-terminates, and no global subscriber cap or idle timeout exists to reclaim resources. An attacker opening many connections can accumulate thousands of permanently-open subscriptions, exhausting memory and (in POLL mode) the bounded-elastic scheduler.

### Finding Description

**Code path:**

`subscribeTopic()` (line 59) builds the combined flux and wires `takeUntilOther(pastEndTime(topicContext))` at line 73. `pastEndTime()` (lines 123–131) returns `Flux.never()` when `filter.getEndTime() == null`:

```java
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();          // ← stream never terminates server-side
    }
    ...
}
```

`incomingMessages()` (lines 108–121) is deferred via `Flux.defer(() -> incomingMessages(topicContext))` (line 64) and calls `topicListener.listen(newFilter)` (line 120). For a topic that never receives messages, this Flux never emits and never completes. Combined with `Flux.never()` from `pastEndTime()`, the entire reactive chain for that subscriber is held open indefinitely.

**Root cause — three missing controls:**

1. **No global subscriber cap.** `subscriberCount` (line 48) is a `AtomicLong` used only as a Micrometer gauge metric. It is incremented at `doOnSubscribe` (line 89) and decremented at `doFinally` (line 90), but there is no enforcement gate that rejects new subscriptions when the count exceeds a threshold.

2. **No idle/inactivity timeout.** There is no `timeout()` or `elapsed`-based operator applied to the live flux. A subscription that receives zero messages for hours or days is indistinguishable from a healthy one.

3. **Per-connection limit does not bound total connections.** `NettyProperties.maxConcurrentCallsPerConnection = 5` (line 14 of `NettyProperties.java`) limits concurrent RPCs per TCP connection, but there is no limit on the number of TCP connections an attacker may open.

**Why `topicExists()` is insufficient:** It only validates that the entity exists and has type `TOPIC` (lines 94–105). A valid, permanently-silent topic passes this check and allows the subscription to proceed.

**POLL mode amplification:** `PollingTopicListener.listen()` (lines 34–48) creates a per-subscriber `PollingContext` and schedules a `repeatWhen(RepeatSpec.times(Long.MAX_VALUE)...)` loop on `Schedulers.boundedElastic()`. Each open subscription continuously re-queues polling tasks. Exhausting the bounded-elastic pool (default: 10 × CPU cores threads + bounded queue) stalls all other async work on the server.

### Impact Explanation

- **Memory exhaustion:** Each open subscription retains a `TopicContext` object, a `TopicMessageFilter`, and the full reactive operator chain. With thousands of subscriptions, heap pressure leads to GC thrashing and eventual OOM.
- **Scheduler exhaustion (POLL mode):** Each subscriber schedules periodic DB-poll tasks on `boundedElastic`. Saturating the scheduler delays or starves legitimate subscribers and internal async operations.
- **Service degradation for legitimate users:** Legitimate subscribers experience increased latency or connection refusal as server resources are consumed.
- **No economic cost to attacker:** Opening gRPC streams is free; the attacker needs only network access to the gRPC port.

### Likelihood Explanation

The gRPC endpoint is publicly reachable by design (it serves SDK clients). No authentication is required to call `subscribeTopic`. A single attacker machine can open hundreds of TCP connections and 5 subscriptions each. The attack is trivially scriptable, repeatable, and requires no special knowledge beyond the public gRPC proto definition. The default listener type is `REDIS`, which shares one underlying Redis channel per topic, reducing per-subscriber cost slightly, but memory for each reactive chain is still allocated per subscriber. Switching to `POLL` mode (or if the operator uses it) makes the attack significantly more damaging.

### Recommendation

1. **Enforce a global maximum subscriber count.** Before incrementing `subscriberCount` in `doOnSubscribe`, check the current value and return `Mono.error(...)` if it exceeds a configurable threshold (e.g., `grpcProperties.getMaxSubscribers()`).

2. **Enforce a per-connection or per-IP subscription limit** at the gRPC interceptor layer, in addition to `maxConcurrentCallsPerConnection`.

3. **Add a server-side idle timeout.** Apply `.timeout(grpcProperties.getMaxIdleSubscriptionDuration())` (or a custom operator) to the live flux so subscriptions that receive no messages within a configurable window are terminated with a meaningful status code.

4. **Require or cap open-ended subscriptions.** Consider requiring `endTime` for subscriptions, or enforcing a maximum subscription lifetime (e.g., 24 hours) even when `endTime` is absent.

5. **Add a connection-count limit** at the Netty/gRPC server configuration level to complement the per-connection call limit.

    t.daemon = True
