### Title
Unbounded TIMESTAMP Filter on `/contracts/:contractId/state` Forces Full Cross-Partition Scan of `contract_state_change`, Enabling Unauthenticated DoS

### Summary
Any unauthenticated user can supply a `timestamp` query parameter to `GET /contracts/:contractId/state`. This causes `getContractStateById()` to switch from a cheap keyed lookup on `contract_state` to an expensive `DISTINCT ON (slot)` full scan of the partitioned `contract_state_change` table with no upper-bound range enforcement. Repeated concurrent requests can saturate DB connection pools and degrade or halt mirror-node REST processing.

### Finding Description

**Code path:**

`rest/controllers/contractController.js` → `getContractStateById()` (line 964) calls `extractContractStateByIdQuery()` (line 970). Inside `extractContractStateByIdQuery()` (lines 909–918), any `timestamp` filter with operator other than `ne` is accepted and sets `timestampPresent = true`:

`timestampPresent = true` is returned as `timestamp: true` and passed to `ContractService.getContractStateByIdAndFilters()` (line 971). Inside that function (lines 256–263), the query is switched from `contractStateQuery` (a simple `contract_state` keyed lookup) to `contractStateTimestampQuery`:

```js

`contractStateTimestampQuery` is:

```sql
SELECT DISTINCT ON (slot)
  contract_id, slot, evm_address,
  coalesce(value_written, value_read) as value,
  consensus_timestamp as modified_timestamp
FROM contract_state_change csc
LEFT JOIN entity e ON e.id = csc.contract_id
``` [3](#0-2) 

With the attacker-supplied `WHERE consensus_timestamp <= 99999999999999999`, PostgreSQL must scan **all** time-range partitions of `contract_state_change` (confirmed partitioned by `consensus_timestamp`): [4](#0-3) 

The `DISTINCT ON (slot)` with `ORDER BY slot ASC, consensus_timestamp DESC` requires reading every matching row across all partitions before the `LIMIT` can be applied. The index `contract_state_change__id_slot_timestamp (contract_id, slot, consensus_timestamp)` exists but cannot eliminate the cross-partition aggregation when the timestamp bound covers all history. [5](#0-4) 

**Root cause — failed assumption:** Other contract endpoints (e.g., contract results/logs) call `optimizeTimestampFilters()` → `parseTimestampFilters()` which enforces a maximum 7-day range window. The contract state endpoint **never calls** `parseTimestampFilters()` or any equivalent range guard. A single open-ended `lte` timestamp is accepted without restriction. [6](#0-5) 

**Accepted parameters** confirm `TIMESTAMP` is in scope for this endpoint with no additional validation layer: [6](#0-5) 

### Impact Explanation
Each request with `timestamp=lte:99999999999999999` triggers a full cross-partition sequential scan of `contract_state_change` for the targeted contract, plus a `DISTINCT ON` aggregation across all results. For a high-activity contract (e.g., a popular DeFi contract with millions of state-change records across years of partitions), a single query can hold a DB connection for seconds to tens of seconds. Concurrent flooding of this endpoint by a single attacker with no credentials can exhaust the DB connection pool, causing all mirror-node REST API queries to queue or fail — degrading or shutting down ≥30% of network processing nodes that depend on mirror-node data.

### Likelihood Explanation
- **No authentication required**: the endpoint is fully public.
- **No rate limiting visible in code**: nothing in `getContractStateById()` or its call chain throttles per-IP or per-contract.
- **Trivially scriptable**: a single `curl` loop or `ab`/`wrk` invocation suffices.
- **Targeted amplification**: the attacker can identify the highest-activity contract (e.g., USDC, HBAR-wrapped tokens) to maximize scan cost per request.
- **Repeatability**: the attack is stateless and can be sustained indefinitely.

### Recommendation
1. **Apply timestamp range validation** in `extractContractStateByIdQuery()` identical to other endpoints: call `parseTimestampFilters()` with `validateRange = true` and enforce the configured `maxTimestampRange` (default 7 days). Reject open-ended `lte`-only filters or treat them as bounded by `now - maxRange`.
2. **Require both lower and upper bounds** when a timestamp filter is present, or default the lower bound to `upper - maxRange`.
3. **Add a DB-level statement timeout** for this query class so runaway scans are killed before they exhaust the pool.
4. **Consider rate-limiting** the `/contracts/:contractId/state` endpoint per IP/contract when a timestamp filter is present.

### Proof of Concept

**Preconditions:** Mirror node REST API is publicly accessible. Target a contract with significant historical state changes (e.g., `0.0.1234`).

**Single expensive request:**
```bash
curl "https://<mirror-node>/api/v1/contracts/0.0.1234/state?timestamp=lte:99999999999999999"
```

**DoS flood (no credentials needed):**
```bash
# 50 concurrent workers, unlimited repetitions
ab -n 100000 -c 50 \
  "https://<mirror-node>/api/v1/contracts/0.0.1234/state?timestamp=lte:99999999999999999"
```

**Result:** Each request executes:
```sql
SELECT DISTINCT ON (slot) contract_id, slot, ...
FROM contract_state_change
WHERE contract_id = 1234 AND consensus_timestamp <= 99999999999999999
ORDER BY slot ASC, consensus_timestamp DESC
LIMIT 25;
```
This scans all `contract_state_change` partitions from genesis to present for the target contract. Under concurrent load, DB connections are held for the duration of each scan, exhausting the pool and causing all other API queries to time out.
