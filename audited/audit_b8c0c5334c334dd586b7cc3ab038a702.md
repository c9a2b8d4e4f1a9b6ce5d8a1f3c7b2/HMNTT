### Title
Unbounded Synchronous Prometheus HTTP Call in `health()` Enables Thread Pool Exhaustion DoS via Unauthenticated `/actuator/health`

### Summary
`ImporterLagHealthIndicator.health()` performs a fully synchronous, blocking HTTP call to Prometheus on every invocation with no caching, no rate limiting, and no concurrency guard. The monitor module contains no actuator security configuration, leaving `/actuator/health` unauthenticated by default. An attacker sending a flood of concurrent requests to that endpoint can exhaust the Tomcat thread pool, rendering the entire Spring Boot application unresponsive.

### Finding Description

**Code path:**

`ImporterLagHealthIndicator.health()` — `monitor/src/main/java/org/hiero/mirror/monitor/health/ImporterLagHealthIndicator.java`, lines 48–62:

```java
@Override
public Health health() {
    if (!properties.isEnabled()) {   // fast-exit only when fully disabled
        return up();
    }
    final var query = buildLagQuery();
    try {
        final var resp = prometheusClient.query(query);   // BLOCKING call
        return evaluate(resp);
    } catch (final Exception e) {
        log.warn("Importer lag health check failed; returning UP: {}", e.getMessage());
        return up();
    }
}
```

`PrometheusApiClient.query()` — `monitor/src/main/java/org/hiero/mirror/monitor/health/PrometheusApiClient.java`, lines 46–53:

```java
PrometheusQueryResponse query(final String query) {
    final var encodedQuery = UriUtils.encodeQuery(query, StandardCharsets.UTF_8);
    return prometheusClient
            .get()
            .uri(builder -> builder.queryParam("query", encodedQuery).build())
            .retrieve()
            .body(PrometheusQueryResponse.class);   // blocks until response or timeout
}
```

The `RestClient` is built with `SimpleClientHttpRequestFactory` (lines 25–27 of `PrometheusApiClient.java`), which is a **blocking, synchronous** HTTP client. The read/connect timeout defaults to **5 seconds** (`ImporterLagHealthProperties`, line 64). There is no result cache, no semaphore, and no debounce on `health()`.

**Root cause:** Every HTTP request that reaches `/actuator/health` causes Spring Boot's `HealthEndpoint` to invoke all registered `HealthIndicator` beans synchronously on the servlet thread. `ImporterLagHealthIndicator` holds that thread blocked for up to 5 seconds per call. No security configuration was found anywhere in the monitor module (`monitor/src/main/java/org/hiero/mirror/monitor/config/` contains no `SecurityConfig`; `monitor/src/main/resources/` contains only `banner.txt`). Spring Boot's default is to expose `/actuator/health` without authentication and with zero health-result caching (`health.cache.time-to-live = 0`).

**Why existing checks fail:**
- The `properties.isEnabled()` guard (line 49) only short-circuits when the feature is administratively disabled; in production it is enabled.
- The `catch (Exception e)` block (lines 58–61) does not prevent thread blocking — it only handles the case where the call eventually throws; the thread is still held for the full timeout duration before the exception propagates.
- No Spring Security `SecurityFilterChain` restricts the actuator endpoints.
- No `management.endpoint.health.cache.time-to-live` is configured, so every request re-invokes all health indicators.

### Impact Explanation
Tomcat's default thread pool is 200 threads. With a 5-second blocking timeout per call, an attacker sustaining ~40 concurrent requests/second saturates all 200 threads within 5 seconds. Once the pool is exhausted, every subsequent HTTP request — including legitimate API calls, liveness/readiness probes used by Kubernetes, and operator dashboards — queues indefinitely or is rejected. Kubernetes will eventually mark the pod as unhealthy and restart it, but the attacker can immediately re-trigger the condition, causing a persistent, cyclic outage of the monitor service.

### Likelihood Explanation
The monitor application's actuator port is typically reachable within a cluster network and, depending on ingress configuration, may be reachable externally. No credentials are required. The attack requires only a standard HTTP client capable of issuing concurrent GET requests — no exploit code, no authentication bypass, and no special knowledge beyond the well-known `/actuator/health` path. It is fully repeatable and automatable.

### Recommendation
1. **Cache health results**: Set `management.endpoint.health.cache.time-to-live=10s` (or similar) so repeated calls within the window return the cached value without re-invoking Prometheus.
2. **Secure the actuator endpoint**: Add a `SecurityFilterChain` bean in the monitor module that requires authentication (e.g., IP allowlist or basic auth) for all `/actuator/**` paths except the paths explicitly needed by Kubernetes probes.
3. **Use a non-blocking HTTP client**: Replace `SimpleClientHttpRequestFactory` with a reactive/non-blocking client (e.g., `WebClient` backed by Reactor Netty) so Prometheus calls do not hold platform threads.
4. **Add a concurrency guard**: Use a `Semaphore` or `AtomicBoolean` inside `health()` to allow at most one in-flight Prometheus call at a time, returning the last known result for concurrent callers.

### Proof of Concept

**Preconditions:**
- Monitor application is running with `hiero.mirror.monitor.health.importer-lag.enabled=true` (production default when Prometheus URL and cluster are configured).
- `/actuator/health` is reachable (default: no authentication).

**Steps:**
```bash
# Send 300 concurrent requests to /actuator/health
# Each will block a Tomcat thread for up to 5 seconds waiting for Prometheus
seq 300 | xargs -P300 -I{} curl -s -o /dev/null http://<monitor-host>:8080/actuator/health &

# Immediately probe a normal endpoint — it will hang or be refused
curl --max-time 3 http://<monitor-host>:8080/actuator/health/liveness
# Expected: connection timeout or HTTP 503 — thread pool exhausted
```

**Result:** The liveness probe times out, Kubernetes marks the pod `NotReady`, and the monitor service is effectively down for the duration of the attack.
