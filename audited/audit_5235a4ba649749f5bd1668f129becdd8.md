### Title
Unauthenticated Concurrent `/block` Requests Exhaust Database Connection Pool via Unbounded `GetAccountAlias()` Calls on LRU Cache Misses

### Summary
The `Block()` function in `rosetta/app/services/block_service.go` calls `updateOperationAccountAlias()` for every operation in every returned transaction. On an LRU cache miss, a synchronous `GetAccountAlias()` database query is issued with no application-level concurrency control. An unauthenticated attacker sending concurrent requests targeting blocks with many unique account IDs can simultaneously saturate the database connection pool (default: 100 open connections), denying service to all other database-dependent endpoints.

### Finding Description
**Code path:**

`Block()` (lines 47–74) → `updateOperationAccountAlias()` (lines 104–130) → `accountRepo.GetAccountAlias()` (persistence/account.go line 117)

In `updateOperationAccountAlias()`:
```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)   // synchronous DB query
    ...
    s.entityCache.Set(result.GetId(), result)
}
``` [1](#0-0) 

Each cache miss issues a raw SQL query against the `entity` table:
```go
db.Raw(selectCryptoEntityWithAliasById, sql.Named("id", accountId.GetId())).First(&entity)
``` [2](#0-1) 

**Root cause — three compounding weaknesses:**

1. **No application-level concurrency control.** The Go application has no semaphore, worker pool, or rate limiter. The only protection is optional Traefik middleware (`inFlightReq: amount: 5` per source IP; `rateLimit: average: 10` per request host) defined in the Helm chart values. [3](#0-2) 

2. **Finite DB connection pool.** `SetMaxOpenConns` is set to `dbConfig.Pool.MaxOpenConnections`, defaulting to **100** open connections. [4](#0-3) [5](#0-4) 

3. **LRU cache has no TTL.** The `entityCache` is a pure LRU with capacity 524,288 and no expiry. On a cold start, or when targeting blocks with account IDs not yet cached, every operation triggers a DB round-trip. [6](#0-5) 

**Failed assumption:** The design assumes the LRU cache will absorb the majority of alias lookups. This fails when: (a) the service is freshly started (empty cache), (b) an attacker deliberately targets blocks with diverse account IDs not present in the cache, or (c) the cache is thrashed by rotating through more than 524,288 unique IDs across concurrent requests.

### Impact Explanation
When the 100-connection pool is exhausted, `gorm`/`database/sql` blocks all new queries waiting for a free connection (up to `statementTimeout = 20s`). This affects every DB-backed endpoint — `/block`, `/block/transaction`, `/account/balance`, `/construction/metadata` — causing a full service denial for legitimate users. The attack is amplified because each `/block` request can trigger up to `maxTransactionsInBlock × operations_per_tx` simultaneous DB queries from a single HTTP request.

### Likelihood Explanation
The `/block` endpoint requires no authentication. The Traefik middleware (`inFlightReq: amount: 5` per IP) is:
- **Optional** — not enforced at the application layer; deployments without Traefik have zero protection.
- **Bypassable** — an attacker using 20 source IPs (trivially achievable with cloud VMs or a small botnet) can sustain 100 concurrent in-flight requests, saturating the pool. Even from a single IP, if each request has ≥20 operations with cache misses, 5 concurrent requests already generate 100 simultaneous DB queries.

The `rateLimit: average: 10` per request host is a global 10 req/s cap, but at 10 req/s with many operations per block, the cumulative DB query rate can still overwhelm the pool during the `statementTimeout` window.

### Recommendation
1. **Application-level concurrency limit:** Add a semaphore or `golang.org/x/sync/semaphore` in `updateOperationAccountAlias()` or at the HTTP handler level to cap simultaneous DB-bound goroutines.
2. **Per-request DB query cap:** Enforce a hard limit on the number of `GetAccountAlias()` calls per single request invocation (e.g., bail out or batch after N misses).
3. **Add TTL to the entity cache:** Use `cache.WithExpiration` so stale entries are refreshed gradually rather than all at once after a restart.
4. **Enforce rate limiting in-process:** Do not rely solely on external Traefik middleware; add a token-bucket or `golang.org/x/time/rate` limiter inside the Go service.
5. **Increase pool or add connection timeout:** Set a short `WaitTimeout` on the pool so exhausted-pool errors fail fast rather than holding goroutines for 20 seconds.

### Proof of Concept
**Preconditions:** Rosetta service running in online mode, either without Traefik or attacker controls ≥20 source IPs.

**Steps:**
1. Identify several blocks on the network that contain the maximum number of transactions (`maxTransactionsInBlock`), each involving many distinct account IDs not yet in the LRU cache (e.g., immediately after service restart).
2. From 20 source IPs (or a single IP if Traefik is absent), send 5 concurrent POST requests per IP to `POST /block` with each request targeting a different block:
   ```json
   {"network_identifier": {...}, "block_identifier": {"index": <N>}}
   ```
3. Each request triggers `updateOperationAccountAlias()` for up to `maxTransactionsInBlock` transactions. With, e.g., 20 operations per transaction and 5 operations per transaction average, 100 concurrent requests × 20 DB queries = 2,000 simultaneous `GetAccountAlias()` calls against a 100-connection pool.
4. **Observed result:** The DB connection pool is exhausted. Subsequent legitimate requests to any DB-backed endpoint receive errors or hang for up to 20 seconds (`statementTimeout`), constituting a denial of service.
