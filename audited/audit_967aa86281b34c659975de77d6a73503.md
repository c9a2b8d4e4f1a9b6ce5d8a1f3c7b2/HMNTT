### Title
Surrogate-Pair Split in `LoggingFilter` Produces Invalid UTF-8 Log Entries via Crafted POST Body

### Summary
`LoggingFilter.getContent()` decodes the raw request body into a Java `String` (UTF-16), then truncates it with `StringUtils.substring(content, 0, maxPayloadLogSize)`, which cuts on UTF-16 `char` indices. An unprivileged attacker can position a supplementary Unicode character (e.g., an emoji, code point > U+FFFF) so that the cut falls between its high and low surrogate, producing a Java `String` with an unpaired surrogate. When the logger serializes this to UTF-8, it emits invalid UTF-8 bytes, corrupting the log line and potentially causing downstream log-parsing tools to drop or misparse it.

### Finding Description

**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/config/LoggingFilter.java`, `getContent()`, lines 86–107:

```
content = StringUtils.deleteWhitespace(wrapper.getContentAsString()); // line 86
...
content = StringUtils.substring(content, 0, maxPayloadLogSize);       // line 107
``` [1](#0-0) [2](#0-1) 

**Root cause:**

`ContentCachingRequestWrapper.getContentAsString()` decodes the raw request bytes into a Java `String` using the charset from the `Content-Type` header (e.g., `charset=UTF-8`). Java `String` is UTF-16 internally. Supplementary Unicode characters (code points > U+FFFF — emoji, certain CJK extension blocks, etc.) are stored as two `char` values: a high surrogate followed by a low surrogate.

`StringUtils.substring` from Apache Commons Lang 3 delegates directly to `String.substring(int, int)`, which operates on UTF-16 `char` indices, not Unicode code points. [3](#0-2) 

If `maxPayloadLogSize` (default 300) falls at the index of a low surrogate, the cut produces a `String` whose last `char` is an unpaired high surrogate — invalid Unicode. When Logback (or any UTF-8 encoder) serializes this `String`, it encodes the lone surrogate as a 3-byte CESU-8/MUTF-8 sequence (`0xED 0xA0 0x??`), which is explicitly forbidden by the UTF-8 standard (RFC 3629 §3).

**Why existing checks are insufficient:**

1. **Compression branch (lines 89–102):** Only replaces `content` with the Base64-compressed form if `compressed.length() <= maxPayloadLogSize`. If the compressed form is still > 300 chars (easily arranged with high-entropy or already-compressed data), the original decoded `String` is kept and the surrogate-splitting truncation at line 107 still applies. [4](#0-3) 

2. **5xx bypass (line 105):** Truncation is skipped only for HTTP 5xx responses. For any 2xx or 4xx response, truncation always occurs. [2](#0-1) 

3. **`maxPayloadLogSize` is operator-controlled** (default 300), but the attacker only needs to know the approximate value (or simply try multiple payload lengths). [5](#0-4) 

### Impact Explanation

- **Log corruption / log evasion:** A log line containing invalid UTF-8 is silently dropped or causes a parse error in strict UTF-8 log consumers (Logstash JSON codec, Fluentd, Splunk heavy forwarder with UTF-8 validation). The attacker's request — including their IP, method, URI, and payload — disappears from the SIEM, defeating audit trails.
- **Repeated, targeted log poisoning:** The attacker can send a stream of such requests to corrupt a specific time window of logs, hiding concurrent malicious activity.
- Severity: **Medium** (no direct data exfiltration or RCE, but meaningful security-monitoring bypass for an unprivileged attacker with zero authentication required).

### Likelihood Explanation

- **No authentication required.** Any HTTP client can POST to the web3 endpoint.
- **Trivially reproducible.** The attacker needs only to: (a) know or probe `maxPayloadLogSize` (default 300, publicly documented), (b) craft a body of exactly `N-1` ASCII bytes followed by one emoji.
- **Deterministic.** The surrogate split is 100% reliable once the payload length is calibrated.
- **No rate-limiting or input validation** on the body charset or content prevents this.

### Recommendation

Replace the char-index truncation with a code-point–aware truncation that never cuts inside a surrogate pair:

```java
// Instead of:
content = StringUtils.substring(content, 0, maxPayloadLogSize);

// Use:
content = content.codePoints()
    .limit(maxPayloadLogSize)
    .collect(StringBuilder::new,
             StringBuilder::appendCodePoint,
             StringBuilder::append)
    .toString();
```

Alternatively, truncate on the **byte** representation after encoding to UTF-8, then re-decode, ensuring the result is always valid UTF-8. Also consider explicitly passing `StandardCharsets.UTF_8` to `getContentAsString()` (or a wrapper that enforces it) so the decoding step is deterministic regardless of the `Content-Type` header.

### Proof of Concept

```python
import requests

# maxPayloadLogSize default = 300
# Place a 4-byte emoji (surrogate pair in Java) at char index 299-300
# 299 ASCII 'A's + emoji U+1F600 (😀) + filler to prevent compression
payload = "A" * 299 + "\U0001F600" + "B" * 400

headers = {
    "Content-Type": "application/json; charset=UTF-8"
}

# Send to any non-5xx-returning endpoint (e.g., eth_call)
r = requests.post(
    "http://<mirror-node-web3>/api/v1/contracts/call",
    data=payload.encode("utf-8"),
    headers=headers
)
print(r.status_code)
# Expected: the logged line for this request contains an unpaired high surrogate
# (0xED 0xA0 0x?? in the UTF-8 log file), causing log parsers to reject/drop the line.
```

Verify by inspecting the raw log file bytes:
```bash
grep -P '\xed[\xa0-\xbf]' /var/log/mirror-node/web3.log
```
A match confirms invalid CESU-8 surrogate bytes in the log output.
