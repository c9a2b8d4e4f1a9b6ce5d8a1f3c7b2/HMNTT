### Title
Global Rate-Limit Bucket Starvation via Single Unprivileged Client

### Summary
`ThrottleManagerImpl.throttle()` checks a single process-wide `rateLimitBucket` shared across all callers with no per-IP or per-user partitioning. A single unauthenticated client can exhaust the entire `requestsPerSecond` quota (default 500 tokens/s) within one refill window, causing every other concurrent user to receive HTTP 429 for the remainder of that second. The attack requires no credentials, no special knowledge, and is trivially repeatable.

### Finding Description
**Exact code path:**

`ThrottleConfiguration.rateLimitBucket()` (file `web3/src/main/java/org/hiero/mirror/web3/config/ThrottleConfiguration.java`, lines 24–32) creates a single application-scoped `Bucket` with capacity = `requestsPerSecond` (default 500) and a greedy 1-second refill:

```java
Bucket.builder().addLimit(limit).build();   // one global bucket, no per-caller key
```

`ThrottleManagerImpl.throttle()` (file `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, line 38) unconditionally deducts one token from this shared bucket for every incoming request regardless of source:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

No caller identity (IP address, API key, session) is ever read or used as a partitioning key. The `ContractController.call()` handler (file `web3/src/main/java/org/hiero/mirror/web3/controller/ContractController.java`, line 40) passes only the `ContractCallRequest` body to `throttle()`; the `HttpServletRequest` (which carries the remote address) is present in the method signature but is never forwarded to the throttle layer.

**Why existing checks fail:**

1. The GCP gateway `maxRatePerEndpoint: 250` (file `charts/hedera-mirror-web3/values.yaml`, line 56) limits requests per *backend pod*, not per *client IP*. A single attacker can send 250 RPS to one pod and exhaust its bucket.
2. `sessionAffinity: CLIENT_IP` (line 57–58) routes the same client to the same pod, which actually *worsens* the problem: the attacker's flood is concentrated on one pod rather than spread across replicas.
3. The `request[].action = THROTTLE` filter (file `web3/src/main/java/org/hiero/mirror/web3/throttle/RequestProperties.java`, lines 63–69) is an opt-in, operator-configured feature that is empty by default (`request = List.of()`), and even when configured it still uses a single global sub-bucket, not a per-caller one.
4. `restore(gas)` (line 59–64 of `ThrottleManagerImpl.java`) only returns gas tokens to `gasLimitBucket`; it never restores `rateLimitBucket` tokens, so even failed/invalid requests permanently consume rate-limit quota.

### Impact Explanation
A single attacker sending ≥500 HTTP POST requests per second to `/api/v1/contracts/call` drains the global `rateLimitBucket` to zero. All other users receive `HTTP 429 Too Many Requests` for the rest of that 1-second window. The bucket refills greedily at the start of the next second, but the attacker can immediately re-flood. In practice this produces continuous denial of service for all legitimate contract-call users as long as the attacker sustains the flood. No on-chain state is corrupted and no funds are at risk, placing this squarely in the griefing / service-availability category (Medium per the stated scope).

### Likelihood Explanation
Preconditions: none. The endpoint is public, unauthenticated, and reachable from any internet host. Sending 500 HTTP POST requests per second is well within the capability of a single commodity machine or a small botnet. The attack is stateless, requires no prior knowledge of the system, and is trivially scriptable with standard tools (e.g., `wrk`, `hey`, `ab`). It is repeatable indefinitely at negligible cost.

### Recommendation
Replace the single global `Bucket` with a per-IP (or per-API-key) bucket map, for example using Bucket4j's `ProxyManager` backed by a `ConcurrentHashMap` keyed on `HttpServletRequest.getRemoteAddr()` (or the `X-Forwarded-For` header when behind a trusted proxy). Each caller then has its own token budget, so one client exhausting its own quota does not affect others. Additionally:
- Pass the caller identity into `ThrottleManager.throttle()` so the interface supports per-caller enforcement.
- Restore rate-limit tokens (not just gas tokens) when a request is rejected before processing.
- Consider a two-tier limit: a per-IP limit (e.g., 50 RPS) nested inside the existing global cap (500 RPS).

### Proof of Concept
```bash
# Flood the endpoint from a single host at 600 RPS for 5 seconds
# Requires: wrk (https://github.com/wg/wrk)

wrk -t4 -c100 -d5s -s post.lua http://<mirror-node-host>/api/v1/contracts/call

# post.lua:
# wrk.method = "POST"
# wrk.headers["Content-Type"] = "application/json"
# wrk.body = '{"to":"0x0000000000000000000000000000000000000167","data":"0x","gas":21000}'
```

Expected result during the flood: all requests from *other* concurrent clients receive `HTTP 429` with body `{"_status":{"messages":[{"message":"Requests per second rate limit exceeded"}]}}`. After the flood stops, service resumes normally within 1 second (next refill window), confirming the root cause is the shared global bucket.
