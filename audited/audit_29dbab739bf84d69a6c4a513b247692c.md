### Title
Unauthenticated Pagination Flood in `getTokenRelationships()` Exhausts DB Connection Pool

### Summary
The `getTokenRelationships()` handler in `rest/controllers/tokenController.js` issues at least two DB queries per request (entity lookup + account validation + token fetch) with no per-IP or per-session rate limiting at the application layer. An unprivileged attacker can follow the cursor-based `next` link produced by `getPaginationLink()` in rapid succession with `limit=1`, generating a sustained flood of DB queries that saturates the finite connection pool and causes legitimate requests to time out.

### Finding Description

**Exact code path:**

`rest/controllers/tokenController.js`, `getTokenRelationships()`, lines 66–92:

```js
getTokenRelationships = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);   // DB query 1
  const isValidAccount = await EntityService.isValidAccount(accountId);  // DB query 2
  ...
  const tokenRelationships = await TokenService.getTokenAccounts(query); // DB query 3
  ...
  if (tokens.length === query.limit) {
    nextLink = utils.getPaginationLink(req, false, lastValue, query.order); // cursor URL
  }
```

Each call to this handler issues **3 sequential DB queries** before returning. `getPaginationLink()` (`rest/utils.js` lines 805–819) simply constructs a URL with a `token.id=gt:<last_id>` cursor — it performs no tracking of how many pages have been fetched or at what rate.

**Route registration** (`rest/routes/accountRoute.js` line 19):
```js
router.getExt(getPath('tokens'), TokenController.getTokenRelationships);
```
No rate-limiting middleware is attached to this route.

**Server middleware stack** (`rest/server.js` lines 68–144): The middleware chain is `cors → httpContext → requestLogger → authHandler → metricsHandler → responseCacheCheckHandler → routes → responseHandler → handleError`. There is no IP-based throttle or concurrency limiter for the REST Node.js service. The `ThrottleManagerImpl` (bucket4j) exists only in the `web3` Java module, not here.

**DB pool** (`rest/dbpool.js` lines 7–16):
```js
const poolConfig = {
  max: config.db.pool.maxConnections,
  connectionTimeoutMillis: config.db.pool.connectionTimeout,
  statement_timeout: config.db.pool.statementTimeout,
};
```
The pool is finite. When all slots are occupied, new requests queue until `connectionTimeoutMillis` expires and then fail.

**Root cause / failed assumption:** The design assumes that `statement_timeout` and `connectionTimeoutMillis` are sufficient back-pressure. They are not: `statement_timeout` limits individual query duration but does not prevent a sustained flood from keeping the pool continuously saturated. `responseCacheCheckHandler` does not help because each paginated request carries a unique cursor (`token.id=gt:<X>`), producing cache misses on every page.

### Impact Explanation

An attacker maintaining N concurrent pagination streams, each with `limit=1`, holds up to 3N DB connections at any moment. With the default pgbouncer `max_user_connections: 250` for `mirror_rest` and a pool of finite size, a modest number of concurrent streams (e.g., 100 connections × 3 queries in flight) can saturate the pool. Legitimate API users then receive connection-timeout errors for all endpoints that share the same pool, including transactions, balances, and account lookups. No economic damage occurs to any on-chain account; the impact is pure service degradation (griefing).

### Likelihood Explanation

No authentication or API key is required. Any internet user can:
- Identify a large account (e.g., a DEX or exchange account with thousands of token associations) via the public API.
- Script a loop that follows the `next` link with `limit=1`, issuing one request per token association.
- Run dozens of such loops in parallel from a single machine or a small botnet.

The attack is trivially repeatable, requires no special knowledge, and produces no on-chain footprint.

### Recommendation

1. **Add per-IP rate limiting** at the Express middleware layer (e.g., `express-rate-limit`) applied globally or specifically to paginated endpoints, before route handlers.
2. **Enforce a minimum `limit`** (e.g., ≥ 10) to reduce the number of pages — and thus DB round-trips — needed to traverse a large list.
3. **Add a concurrency semaphore** on DB pool acquisition so that a single IP cannot hold more than K connections simultaneously.
4. **Enable and enforce infrastructure-level rate limiting** (Traefik `rateLimit` / `inFlightReq` middleware) for the REST service, mirroring what is already configured for the Rosetta service.

### Proof of Concept

```bash
# Step 1: identify a large account
ACCOUNT="0.0.98"   # treasury or exchange account with many token associations

# Step 2: fetch first page
NEXT="/api/v1/accounts/${ACCOUNT}/tokens?limit=1&order=asc"

# Step 3: loop rapidly, following next links (run N copies in parallel)
while true; do
  RESP=$(curl -s "https://<mirror-node-host>${NEXT}")
  NEXT=$(echo "$RESP" | jq -r '.links.next // empty')
  [ -z "$NEXT" ] && NEXT="/api/v1/accounts/${ACCOUNT}/tokens?limit=1&order=asc"
done &

# Repeat the above block 80–100 times concurrently.
# Observe: legitimate requests to /api/v1/transactions or /api/v1/accounts
# begin returning 503 / connection-timeout errors as the DB pool is exhausted.
```
