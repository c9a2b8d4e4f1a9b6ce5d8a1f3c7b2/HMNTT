### Title
`ContractDebugService.validateResult()` Swallows Exception, Causing Reverted Transactions to Be Metrically Recorded as SUCCESS

### Summary
`ContractDebugService` overrides `validateResult()` and silently swallows the `MirrorEvmTransactionException` thrown by the parent for reverted transactions. Because `doProcessCall()` only updates `status` away from `"SUCCESS"` inside the `catch (MirrorEvmTransactionException e)` block â€” which is never reached when the exception is swallowed inside `validateResult()` â€” `updateMetrics()` is called with `status = "SUCCESS"` for every reverted transaction processed through the debug path.

### Finding Description
**Root cause â€” `ContractDebugService.validateResult()` (lines 62â€“72):**

```java
@Override
protected void validateResult(final EvmTransactionResult txnResult, final CallServiceParameters params) {
    try {
        super.validateResult(txnResult, params);   // throws MirrorEvmTransactionException for reverts
    } catch (MirrorEvmTransactionException e) {
        log.warn(...);   // exception swallowed here â€” never re-thrown
    }
}
``` [1](#0-0) 

**Execution flow in `doProcessCall()` (lines 109â€“138):**

```
status = "SUCCESS"                          // line 112
result = execute(params, estimatedGas)      // returns failed EvmTransactionResult (not thrown)
validateResult(result, params)              // called, but exception is swallowed inside override
// catch(MirrorEvmTransactionException) block NEVER reached â†’ status stays "SUCCESS"
// finally: result != null && !estimate â†’ updateMetrics(..., "SUCCESS")
``` [2](#0-1) 

The companion enabler is in `TransactionExecutionService.handleFailedResult()`: when an opcode context is active (i.e., the debug path), it **returns** the failed `EvmTransactionResult` instead of throwing, so `execute()` itself does not throw and `result` is non-null with a non-SUCCESS `responseCodeEnum`. [3](#0-2) 

`EvmTransactionResult.isSuccessful()` correctly returns `false` for any non-SUCCESS code, confirming the result is genuinely reverted. [4](#0-3) 

### Impact Explanation
Every call through `ContractDebugService.processOpcodeCall()` that results in a reverted EVM transaction increments the `hiero.mirror.web3.evm.invocation` counter and the `hiero.mirror.web3.evm.gas.used` counter with `TAG_STATUS = "SUCCESS"`. Operators and automated alerting systems relying on these Micrometer metrics to detect failure rates will see an artificially inflated success rate, masking real revert activity. This constitutes incorrect status records exported via the metrics pipeline (Prometheus/Micrometer scrape endpoints). [5](#0-4) 

### Likelihood Explanation
The `processOpcodeCall()` endpoint is a read-only historical replay operation requiring no privileged credentials â€” any user who can reach the mirror node's web3 API can submit a `ContractDebugParameters` request referencing a known historically-reverted transaction. The trigger is fully deterministic and repeatable: every such request will produce the incorrect SUCCESS metric. No special account, token, or on-chain state is required. [6](#0-5) 

### Recommendation
`ContractDebugService.validateResult()` must re-throw the exception after logging, or the `status` variable must be derived from `txnResult.responseCodeEnum()` directly in the `finally` block rather than relying solely on the catch path. The simplest fix:

```java
@Override
protected void validateResult(final EvmTransactionResult txnResult, final CallServiceParameters params) {
    try {
        super.validateResult(txnResult, params);
    } catch (MirrorEvmTransactionException e) {
        log.warn(...);
        throw e;   // re-throw so doProcessCall's catch block updates status correctly
    }
}
```

Alternatively, derive `status` in `doProcessCall`'s `finally` block from `result.isSuccessful()` rather than from whether a `MirrorEvmTransactionException` was caught.

### Proof of Concept
1. Identify any historically-reverted contract call transaction on the mirror node (any transaction with a non-SUCCESS EVM receipt).
2. Send a `processOpcodeCall` request to `ContractDebugService` referencing that transaction's consensus timestamp.
3. Observe that `hiero.mirror.web3.evm.invocation` counter increments with `status="SUCCESS"` despite the transaction being reverted.
4. Confirm via `EvmTransactionResult.isSuccessful()` returning `false` for the returned result, while the metric tag reads `SUCCESS`.
