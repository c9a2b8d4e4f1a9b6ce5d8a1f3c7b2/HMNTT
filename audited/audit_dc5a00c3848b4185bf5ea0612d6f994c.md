### Title
Unauthenticated POST /api/v1/network/fees with mode=STATE Enables DB Connection Pool Exhaustion via Concurrent Requests

### Summary
The `estimateFees()` endpoint in `NetworkController` is publicly accessible with no authentication and no rate limiting in the `rest-java` module. When called with `mode=STATE`, each request triggers live database queries through `FeeTopicStore` and `FeeTokenStore`. An attacker flooding this endpoint with concurrent requests can exhaust the shared HikariCP connection pool, denying service to all other API endpoints sharing that pool.

### Finding Description

**Exact code path:**

`NetworkController.estimateFees()` (lines 113–124) accepts any unauthenticated POST with no rate-limiting guard: [1](#0-0) 

When `mode=STATE`, `FeeEstimationService.estimateFees()` (lines 101–103) constructs a `FeeEstimationFeeContext` backed by the real `feeTopicStore` and `feeTokenStore` singletons: [2](#0-1) [3](#0-2) 

`FeeEstimationFeeContext.readableStore()` (lines 196–201) returns the live DB-backed stores for `ReadableTopicStore` and `ReadableTokenStore`: [4](#0-3) 

`FeeTopicStore.getTopic()` (lines 28–33) issues two DB queries per call — `topicRepository.findById()` and `customFeeRepository.findById()`: [5](#0-4) 

`FeeTokenStore.get()` (lines 30–35) similarly issues `tokenRepository.findById()` and `customFeeRepository.findById()`: [6](#0-5) 

The comments in both stores confirm the fee calculator actively calls these methods: [7](#0-6) [8](#0-7) 

**Root cause:** The `estimateFees` endpoint has no per-IP or global rate limit in the `rest-java` module. The `ThrottleConfiguration`/`ThrottleManagerImpl` with bucket4j rate limiting exists only in the `web3` module and is not applied here: [9](#0-8) 

The shared HikariCP pool is configured via `CommonConfiguration`: [10](#0-9) 

**Failed assumption:** The design assumes callers will use this endpoint at a reasonable rate. No enforcement mechanism exists to prevent concurrent flooding.

### Impact Explanation
With thousands of concurrent `POST /api/v1/network/fees?mode=STATE` requests carrying a `ConsensusSubmitMessage` or token-transfer transaction body, each request acquires one or more HikariCP connections for the duration of its DB queries. The pool (finite, typically 10–50 connections in default/production configs) becomes saturated. All other `rest-java` endpoints (`/api/v1/transactions`, `/api/v1/accounts`, etc.) that share the same pool begin timing out on connection acquisition, resulting in 500 errors for legitimate users. No economic damage occurs to the Hedera network itself, but the mirror node REST API becomes unavailable — consistent with the Medium/griefing severity classification.

### Likelihood Explanation
No privileges, API keys, or special knowledge are required. The attacker only needs to know the public endpoint URL and the protobuf format for a valid `Transaction` (publicly documented). A single machine with a modest HTTP concurrency tool (e.g., `wrk`, `ab`, `hey`) can sustain thousands of concurrent requests. The attack is trivially repeatable and requires no on-chain activity or funds.

### Recommendation
1. **Add rate limiting to `estimateFees`**: Apply a bucket4j or similar per-IP rate limit (e.g., 10–50 req/s) to `POST /api/v1/network/fees` in the `rest-java` filter chain, mirroring the pattern in `web3/ThrottleConfiguration`.
2. **Limit server-side concurrency**: Use a semaphore or bounded executor for `mode=STATE` requests to cap simultaneous DB-touching fee estimations.
3. **Separate connection pools**: Optionally isolate the fee estimation DB queries to a dedicated, size-limited pool so exhaustion cannot cascade to other endpoints.
4. **Request body size limit**: Enforce a maximum body size to prevent amplification via large protobuf payloads.

### Proof of Concept
```bash
# 1. Encode a minimal ConsensusSubmitMessage transaction in protobuf (body only)
#    (standard HAPI protobuf, publicly documented)
TXBODY=$(python3 -c "
from hedera.proto import transaction_pb2, transaction_body_pb2, consensus_submit_message_pb2
body = transaction_body_pb2.TransactionBody()
body.consensusSubmitMessage.CopyFrom(consensus_submit_message_pb2.ConsensusSubmitMessageTransactionBody())
tx = transaction_pb2.Transaction()
tx.body_bytes = body.SerializeToString()
import sys; sys.stdout.buffer.write(tx.SerializeToString())
" | base64)

# 2. Flood the endpoint with thousands of concurrent requests
hey -n 10000 -c 500 \
  -m POST \
  -T "application/protobuf" \
  -D <(echo "$TXBODY" | base64 -d) \
  "https://<mirror-node-host>/api/v1/network/fees?mode=STATE"

# 3. Simultaneously observe other endpoints returning 500 / connection timeout errors
curl "https://<mirror-node-host>/api/v1/transactions"
# Expected: HTTP 500 or connection timeout due to pool exhaustion
```

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @ConfigurationProperties("spring.datasource.hikari")
    HikariConfig hikariConfig() {
        return new HikariConfig();
    }

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
