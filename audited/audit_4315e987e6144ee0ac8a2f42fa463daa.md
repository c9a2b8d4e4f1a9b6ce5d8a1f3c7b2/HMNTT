### Title
Unauthenticated IP Spoofing via `X-Real-IP` Header in `getClientIpAddress()` Due to Traefik `forwardedHeaders.insecure` Configuration

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally trusts the `X-Real-IP` header without verifying it originates from a trusted proxy. In the production Kubernetes deployment, Traefik is configured with `--entryPoints.web.forwardedHeaders.insecure`, which causes it to pass all client-supplied forwarded headers through unmodified. Any unprivileged external client can therefore inject an arbitrary `X-Real-IP` value that will be recorded in access logs as the true client IP, effectively hiding the attacker's real address from security monitoring.

### Finding Description
**Code path:** `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74.

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)       // line 64 — X-Real-IP checked first, no trust validation

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)  // line 71
    }

    return ipAddress
}
```

**Root cause:** The function applies a simple priority order (`X-Real-IP` → `X-Forwarded-For` → `RemoteAddr`) with no check that the header was set by a trusted upstream proxy. The failed assumption is that only a trusted reverse proxy will ever set `X-Real-IP`.

**Why the existing proxy check is insufficient:**

- *Docker-compose (nginx):* `proxy_set_header X-Real-IP $$remote_addr` **overwrites** any client-supplied `X-Real-IP` with the actual TCP peer address (`docker-compose.yml` line 204). This deployment is not vulnerable.
- *Kubernetes (Traefik):* `--entryPoints.web.forwardedHeaders.insecure` and `--entryPoints.websecure.forwardedHeaders.insecure` (`charts/hedera-mirror-common/values.yaml` lines 636–637) instruct Traefik to **trust and forward all headers from any source**, including external clients. Traefik does not set `X-Real-IP` itself, so a client-supplied value is forwarded verbatim to the rosetta service.

**Exploit flow:**
1. Attacker sends a request to the Traefik ingress with `X-Real-IP: 192.0.2.1` (any desired decoy IP).
2. Traefik, operating in `forwardedHeaders.insecure` mode, passes the header unchanged.
3. `getClientIpAddress()` reads `X-Real-IP` at line 64, finds it non-empty, and returns `192.0.2.1`.
4. `TracingMiddleware` logs `192.0.2.1 POST /construction/... (200) in Xms` (line 52–53), recording the spoofed IP instead of the attacker's real address.

### Impact Explanation
All access-log entries for the rosetta API can be attributed to an arbitrary IP chosen by the attacker. Security monitoring, rate-limiting, and IP-based blocking that rely on these logs are rendered ineffective. An attacker probing fee schedules or construction endpoints repeatedly can rotate spoofed IPs to avoid detection thresholds, and forensic investigation after an incident will point to non-existent or innocent addresses. Severity is **Medium** (log integrity / monitoring bypass; no direct on-chain or financial state change, but directly undermines the security-monitoring layer cited in the question scope).

### Likelihood Explanation
Exploitation requires only the ability to send an HTTP request with a custom header — no credentials, no special network position, no prior knowledge beyond the public API endpoint. The Traefik `forwardedHeaders.insecure` flag is present in the committed production Helm values. Any external user can repeat this on every request with zero cost, making it trivially repeatable and undetectable through standard IP-based alerting.

### Recommendation
1. **Remove `forwardedHeaders.insecure` from Traefik** and replace it with an explicit trusted-IP allowlist using `--entryPoints.web.forwardedHeaders.trustedIPs=<pod-CIDR>` so only headers from known internal proxies are trusted.
2. **Harden `getClientIpAddress()`** to accept an optional list of trusted CIDR ranges and only honour `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` falls within that list; otherwise fall back directly to `r.RemoteAddr`.
3. As a defence-in-depth measure, validate that the value extracted from `X-Real-IP` is a syntactically valid IP address (using `net.ParseIP`) before using it, rejecting malformed or obviously spoofed values.
