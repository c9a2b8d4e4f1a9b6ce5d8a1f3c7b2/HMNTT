### Title
Unauthenticated Resource Exhaustion via Uncached, Unthrottled CTE Query in `getExchangeRate()`

### Summary
`getExchangeRate()` in `rest/service/fileDataService.js` executes a multi-CTE PostgreSQL query on every invocation with no result caching and no rate limiting on the endpoint. The Redis response cache is disabled by default, and the REST API (Node.js) has no per-IP or global rate limiter. Any unauthenticated attacker can sustain a moderate flood of requests to `/api/v1/network/exchangerate` to saturate the 10-connection database pool and drive measurable database CPU/memory increases well above 30%.

### Finding Description

**Exact code path:**

`getExchangeRate()` (line 90–92) delegates unconditionally to `fallbackRetry()` (lines 94–119), which loops up to `config.query.maxFileAttempts` (default **12**) times:

```
rest/service/fileDataService.js  lines 90–119
```

Each loop iteration calls `getLatestFileDataContents()` (line 100), which executes `latestFileContentsQuery` — a two-stage CTE query:

```sql
with latest_create as (
    select max(consensus_timestamp) ...
    from file_data
    where entity_id = $1 and transaction_type in (17, 19) ...
    group by entity_id
    order by consensus_timestamp desc
)
select max(...), min(...), string_agg(fd.file_data, '' order by ...) ...
from file_data fd
join latest_create l on fd.consensus_timestamp >= l.consensus_timestamp
where fd.entity_id = $1 ...
group by fd.entity_id
```

This query performs a CTE, a self-join on `file_data`, `string_agg` over potentially large binary blobs, and a `group by` — all on every single HTTP request.

**Root cause — three compounding absent controls:**

1. **No in-process cache**: `fileDataService.js` contains zero cache/memoization logic. [1](#0-0) 

2. **Redis response cache disabled by default**: `hiero.mirror.rest.cache.response.enabled = false` per documentation. The `responseCacheHandler` only activates when Redis is explicitly enabled and configured. [2](#0-1) 

3. **No rate limiting on the REST API**: A grep across all `rest/**/*.js` files for `rateLimit`, `rateLimiter`, and `throttle` returns zero matches. The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) exists only in the `web3` module for contract calls, not the Node.js REST API. [3](#0-2) 

**DB pool constraint**: The default pool is capped at **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`). [4](#0-3) 

### Impact Explanation

With 10 concurrent connections and no rate limiting, an attacker sending ~10–20 sustained concurrent requests to `GET /api/v1/network/exchangerate` will:

- Saturate the entire database connection pool, blocking all other REST API queries.
- Force the database to execute the CTE + `string_agg` query continuously, driving CPU and I/O well above baseline.
- In edge cases where file data is transiently unparseable (e.g., during a file update), each request triggers up to **12 sequential CTE queries** per the `fallbackRetry` loop, multiplying the load by up to 12×. [5](#0-4) 

This constitutes a realistic denial-of-service against the mirror node's REST API and its backing database, achievable without authentication and without sending thousands of requests per second.

### Likelihood Explanation

The endpoint is public, unauthenticated, and documented in the OpenAPI spec. [6](#0-5)  No special knowledge is required. A single attacker with a basic HTTP client can sustain 10–20 concurrent connections indefinitely. The default deployment (Redis cache disabled, no rate limiter) is the vulnerable configuration.

### Recommendation

1. **Add an in-process TTL cache** (e.g., node-cache or lru-cache) inside `FileDataService` for the exchange rate result, keyed by the timestamp filter, with a TTL of 30–60 seconds. Exchange rates update at most once per hour.
2. **Enable the Redis response cache** by default for the `/api/v1/network/exchangerate` endpoint, or at minimum document it as a required hardening step.
3. **Add a global rate limiter** (e.g., express-rate-limit) to the Node.js REST API, applied at the server middleware level, to cap requests per IP per second.
4. **Increase the DB pool size** or add a query queue with a maximum depth to prevent pool exhaustion from cascading to other endpoints.

### Proof of Concept

```bash
# Saturate the DB pool with 12 concurrent persistent connections
# No authentication, no special headers required
for i in $(seq 1 12); do
  while true; do
    curl -s "https://<mirror-node-host>/api/v1/network/exchangerate" > /dev/null
  done &
done
wait
```

With the default 10-connection pool and no rate limiting, all 10 DB connections will be held by this loop. Monitor `pg_stat_activity` on the database — all slots will show the `latestFileContentsQuery` CTE running continuously. Other REST API endpoints requiring DB access will queue or time out. Database CPU will spike proportionally to the `string_agg` workload on the `file_data` table.

  fallbackRetry = async (fileEntityId, filterQueries, resultConstructor) => {
    const whereQuery = filterQueries.whereQuery ?? [];
    const filters = {whereQuery};

    let attempts = 0;
    while (++attempts <= config.query.maxFileAttempts) {
      const row = await this.getLatestFileDataContents(fileEntityId, filters);
      try {
        return row === null ? null : new resultConstructor(row);
      } catch (error) {
        logger.warn(
          `Failed to load file data for file id ${fileEntityId} at ${row.consensus_timestamp}, failing back to previous file. Attempt ${attempts}. Error: ${error}`
        );

        filters.whereQuery = [
          ...whereQuery,
          {
            query: FileData.CONSENSUS_TIMESTAMP + utils.opsMap.lt,
            param: row.first_consensus_timestamp,
          },
        ];
      }
    }

    return null;
  };
