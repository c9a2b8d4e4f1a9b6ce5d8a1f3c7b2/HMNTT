### Title
Unbounded DB Connection Queue in `getAccountFromAlias()` Enables Unauthenticated DoS via Pool Exhaustion

### Summary
The REST API exposes alias-lookup endpoints with no rate limiting. `getAccountFromAlias()` unconditionally calls into the global `pg` pool, which has no `maxQueueSize` configured. During a network partition (or sustained DB latency), all pool connections stall, and an unauthenticated attacker can queue an unbounded number of pending requests in memory, causing OOM or event-loop saturation that degrades the service for all users.

### Finding Description

**Exact code path:**

`getAccountFromAlias()` at [1](#0-0)  calls `super.getRows(...)`, which at [2](#0-1)  calls `this.pool().queryQuietly(query, params)`.

`queryQuietly` at [3](#0-2)  calls `this.query(query, params)` — the standard `pg.Pool.query()` path. When all pool connections are occupied, `node-postgres` queues the request internally with **no size bound**.

**Pool configuration:**

`rest/dbpool.js` configures the pool with only three parameters: [4](#0-3) 

- `max` (default **10** connections per docs)
- `connectionTimeoutMillis` (default **20 000 ms**)
- `statement_timeout` (default **20 000 ms**)

No `maxQueueSize` is set. `node-postgres` defaults this to `Infinity`.

**Default values confirmed:** [5](#0-4) 

**No rate limiting on the REST API alias path:** The throttling infrastructure (bucket4j `ThrottleManagerImpl`) exists only in the web3 Java service. The Node.js REST API has no equivalent middleware protecting alias-lookup routes. [6](#0-5) 

**Root cause / failed assumption:** The design assumes the OS or an upstream proxy will bound concurrent requests before the pool queue grows large. No such control exists in the codebase for the REST API. During a network partition the 10 pool connections are all blocked waiting up to 20 s for a DB response; every new inbound request appends a Promise to the unbounded internal queue, consuming heap memory and event-loop slots.

### Impact Explanation

- **Memory exhaustion / OOM**: Each queued `pg` request holds a Promise, closure, and query parameters. At tens of thousands of concurrent requests the Node.js process runs out of heap and crashes or triggers the GC continuously, making the process unresponsive.
- **Event-loop saturation**: Even without OOM, the sheer number of pending microtasks degrades response latency for all other endpoints sharing the same process.
- **Scope**: The global `pool` object is shared across all service calls, so exhausting it via alias lookups also blocks every other DB-backed endpoint (transactions, tokens, contracts, etc.).
- **Severity**: High — complete service unavailability for all users, triggered by an unauthenticated attacker.

### Likelihood Explanation

- **No privileges required**: Alias-lookup is a public, unauthenticated endpoint.
- **Trivially scriptable**: A single machine running `ab`, `wrk`, or a simple async loop can saturate the 10-connection pool in milliseconds.
- **Network partition is not required**: Even without a partition, sustained DB slowness (e.g., a long-running vacuum, lock contention) is sufficient to fill the queue. A partition simply makes the window deterministic and controllable by the attacker.
- **Repeatable**: After the queue drains (20 s timeout per request), the attacker simply re-floods. The service never recovers to normal throughput while the attack continues.

### Recommendation

1. **Set `maxQueueSize`** in the `pg` pool configuration in `rest/dbpool.js` (e.g., `maxQueueSize: 100`). Requests beyond this limit are immediately rejected with a 503, preventing unbounded memory growth. [4](#0-3) 

2. **Add rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit` or a Traefik/nginx upstream rule) scoped to alias-lookup and evm-address-lookup routes, mirroring the pattern used in the Rosetta Helm chart. [7](#0-6) 

3. **Reduce `connectionTimeoutMillis`** from 20 000 ms to a lower value (e.g., 3 000 ms) so queued requests fail fast and free memory sooner.

4. **Add a per-request HTTP timeout** at the framework layer so long-stalled requests are aborted before they accumulate.

### Proof of Concept

```bash
# 1. Induce or simulate a network partition to the DB host
#    (e.g., iptables -I OUTPUT -d <db_host> -j DROP on the mirror-node host)

# 2. Flood the alias-lookup endpoint from an unauthenticated client:
wrk -t 8 -c 500 -d 60s \
  "http://<mirror-node>:5551/api/v1/accounts/0.0.AABBCCDD"

# 3. Observe:
#    - Node.js heap grows unboundedly (monitor with `process.memoryUsage()` or Prometheus)
#    - All other REST endpoints (e.g., /api/v1/transactions) also begin timing out
#    - After 20 s the queued requests start failing with "timeout exceeded when trying to connect"
#      but new requests immediately re-fill the queue as long as the flood continues

# 4. Restore the network partition:
#    iptables -D OUTPUT -d <db_host> -j DROP
#    The queue drains, but if the attacker continues flooding, the service remains degraded.
```
