### Title
Missing `COALESCE` in `entityFromEvmAddressQuery` Causes `NotFoundError` for Entities with `deleted = NULL`

### Summary
`EntityService.entityFromEvmAddressQuery` uses `deleted <> true` without `COALESCE`, so any entity whose `deleted` column is `NULL` (the normal, never-deleted state) is silently excluded from results. When an unprivileged user submits a valid EVM address for such an entity, `getEntityIdFromEvmAddress()` receives zero rows and throws `NotFoundError`, making the entity unreachable via EVM address on every API endpoint that calls `getEncodedId()`. The parallel alias query correctly uses `coalesce(deleted, false) <> true`, confirming the omission in the EVM address path is an oversight.

### Finding Description

**Exact code locations:**

`entityFromEvmAddressQuery` — [1](#0-0) 

```sql
where deleted <> true          -- NULL <> true → NULL (falsy) → row excluded
  and evm_address = $1
```

`entityFromAliasQuery` (correct reference) — [2](#0-1) 

```sql
where coalesce(deleted, false) <> true   -- NULL → false → false <> true → true → row included
  and alias = $1
```

**Call chain:**

`getEncodedId()` (line 118–137) receives any EVM-address-shaped string, calls `EntityId.isValidEntityId()` (passes), then `EntityId.parseString()` which returns an `EntityId` with non-null `evmAddress` for opaque EVM addresses (confirmed in `parseFromString`, line 252–253 of `entityId.js`). [3](#0-2) 

This routes to `getEntityIdFromEvmAddress()`, which executes the broken query. [4](#0-3) 

When `rows.length === 0` and `requireResult` is `true` (the default), `NotFoundError` is thrown unconditionally. [5](#0-4) 

**Root cause:** In SQL, `NULL <> true` evaluates to `NULL`, which is falsy in a `WHERE` clause. Entities that have never been explicitly deleted carry `deleted = NULL` (not `false`). The `COALESCE` present in the alias query proves the developers knew `NULL` was a valid state; it was simply omitted from the EVM address query.

**Exploit flow:**
1. Entity `E` exists in the DB with `evm_address = 0xABCD…` and `deleted = NULL` (standard never-deleted state).
2. Attacker (or any user) sends `GET /api/v1/accounts/0xABCD…/nfts` or any endpoint that calls `EntityService.getEncodedId('0xABCD…')`.
3. `getEncodedId` → `getEntityIdFromEvmAddress` → SQL returns 0 rows → `NotFoundError` thrown.
4. The API returns 404 for a fully live entity, permanently, for every request using its EVM address.

**Affected endpoints** (all call `EntityService.getEncodedId` with user-supplied EVM address): [6](#0-5) [7](#0-6) 

### Impact Explanation
Any entity reachable by EVM address whose `deleted` column is `NULL` is permanently inaccessible through the EVM-address lookup path. All API operations that resolve an account/contract by EVM address — NFT queries, staking reward queries, allowance queries, token queries — return 404. Because `deleted = NULL` is the normal state for entities that have never been deleted, this affects the majority of EVM-addressed entities in a production mirror node. The alias path is unaffected because it has the correct `COALESCE`.

### Likelihood Explanation
No privileges are required. Any external user who knows (or discovers) the EVM address of an entity can trigger the error deterministically and repeatedly. EVM addresses are public on-chain data. The condition (`deleted = NULL`) is the default state, not a rare edge case, making this trivially reproducible against real production data.

### Recommendation
Apply `COALESCE` consistently in `entityFromEvmAddressQuery`, matching the alias query:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

### Proof of Concept
1. Insert an entity row: `evm_address = '\xdeadbeef…'`, `deleted = NULL`.
2. `GET /api/v1/accounts/0xdeadbeef…/nfts`
3. Server calls `EntityService.getEncodedId('0xdeadbeef…')` → `getEntityIdFromEvmAddress` → query returns 0 rows → `NotFoundError` → HTTP 404.
4. Repeat with `deleted = false` on the same row → query returns 1 row → HTTP 200. Confirms the `NULL` handling is the sole cause.

  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
```
