### Title
Unauthenticated Resource Exhaustion via Unbounded Concurrent Three-Way UNION ALL NFT Queries

### Summary
The `getNftsByAccountId` handler in `rest/controllers/accountController.js` generates up to three parallel DB subqueries (UNION ALL) per request when a closed token.id range with serial number bounds is supplied. The REST API has no rate-limiting middleware, no per-IP throttle, and no account-existence pre-check, allowing any unauthenticated caller to sustain high database load across many concurrent connections using only valid, distinct requests.

### Finding Description

**Code path:**

`rest/routes/accountRoute.js:15` registers the route with no rate-limiting middleware: [1](#0-0) 

`rest/server.js` applies only `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and optional cache middleware — no rate limiter exists for the REST API: [2](#0-1) 

`AccountController.getNftsByAccountId` performs no account-existence check (unlike `listStakingRewardsByAccountId` which calls `EntityService.isValidAccount`) before issuing the DB query: [3](#0-2) 

`NftService.getQuery` builds up to three independent subqueries joined with `UNION ALL`, each carrying its own `ORDER BY … LIMIT $2` clause, then wraps them in an outer `ORDER BY … LIMIT $2`: [4](#0-3) 

Each subquery performs a `LEFT JOIN entity` on the `nft` table: [5](#0-4) 

**Root cause:** The system assumes the `limit` cap (default max = 100) is sufficient to bound per-request DB work. It does not account for the multiplicative effect: a single request with a closed token.id range (`token.id=gte:A&token.id=lte:B`) combined with serial number bounds (`serialnumber=gt:X&serialnumber=lt:Y`) triggers three subqueries, each scanning up to 100 rows with a JOIN, before the outer merge and re-sort. The default max limit is confirmed at 100 for unauthenticated users: [6](#0-5) 

**Why checks fail:**
- The `limit` cap applies per subquery, not to the total work. Three subqueries × 100 rows × LEFT JOIN = up to 300 row-fetches + sort per request.
- `maxRepeatedQueryParameters` guards against repeated identical params, not against the valid combination of one `gte` + one `lte` for `token.id` and one `gt` + one `lt` for `serialnumber`.
- No brute-force detection exists because each request targets a different account ID and uses valid parameters.
- The throttling infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module, not in the `rest` module. [7](#0-6) 

### Impact Explanation
An attacker can open many concurrent HTTP connections and issue requests such as:
```
GET /api/v1/accounts/0.0.<N>/nfts?token.id=gte:0.0.1&serialnumber=gt:1&token.id=lte:0.0.999999&serialnumber=lt:999999&limit=100
```
Each request causes the DB to execute three independent indexed scans with LEFT JOINs and per-subquery sorts. With a large NFT dataset, each request is a non-trivial DB operation. Across many concurrent connections this saturates the DB connection pool and CPU, degrading or denying service to legitimate users. The impact is increased DB resource consumption (CPU, I/O, connection slots) proportional to concurrency, with no server-side mechanism to stop it.

### Likelihood Explanation
No privileges, credentials, or special knowledge are required. The attacker needs only to know valid account IDs (publicly enumerable via `/api/v1/accounts`) and the documented query parameter syntax. The attack is repeatable indefinitely, distributable across IPs, and each individual request is indistinguishable from legitimate pagination use. The absence of any rate-limiting layer in the REST API makes this trivially sustainable.

### Recommendation
1. **Add a rate-limiting middleware** (e.g., `express-rate-limit`) to the REST API, keyed by IP or authenticated identity, applied globally before route handlers in `rest/server.js`.
2. **Add an account-existence pre-check** in `getNftsByAccountId` (matching the pattern in `listStakingRewardsByAccountId`) to short-circuit DB queries for non-existent accounts.
3. **Cap total DB work per request** by enforcing an outer row limit before the UNION ALL is executed, or by limiting the number of subqueries that can be generated.
4. **Apply a DB statement timeout** (`statementTimeout` in pool config) to bound the maximum wall-clock time any single query can consume.

### Proof of Concept

**Preconditions:** Mirror node REST API is publicly accessible. At least some accounts with NFTs exist (or even without — the DB query still executes).

**Steps:**

```bash
# Enumerate a few valid account IDs
curl "https://<mirror-node>/api/v1/accounts?limit=100" | jq '.accounts[].account'

# Launch concurrent three-way UNION ALL requests across different account IDs
for i in $(seq 1 500); do
  ACCOUNT="0.0.$((RANDOM % 10000 + 1000))"
  curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT}/nfts?\
token.id=gte:0.0.1&serialnumber=gt:1&\
token.id=lte:0.0.9999999&serialnumber=lt:9999999&\
limit=100" &
done
wait
```

**Result:** Each request generates a three-subquery UNION ALL with LEFT JOINs. 500 concurrent requests × 3 subqueries × up to 100 rows each = up to 150,000 row-fetches simultaneously. DB CPU and I/O spike; connection pool exhausts; legitimate requests time out or receive errors. No single request is flagged as brute force because each targets a different account with valid parameters.
