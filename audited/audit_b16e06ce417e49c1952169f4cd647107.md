### Title
Unauthenticated Side-Channel Oracle: DbError → HTTP 503 Leaks DB Availability State to Any External User

### Summary
The `handleError` middleware unconditionally maps every `DbError` to HTTP 503 with no authentication gate on any REST endpoint. An unprivileged external attacker can send arbitrary valid requests and observe the binary 200/503 transition to determine precisely when the database connection pool is exhausted or a network partition is in effect. No credentials, no special access, and no rate-limiting barrier exist to prevent this.

### Finding Description

**Exact code path:**

`rest/middleware/httpErrorHandler.js`, lines 21–22:
```js
} else if (err instanceof DbError) {
  statusCode = httpStatusCodes.SERVICE_UNAVAILABLE;   // → HTTP 503
``` [1](#0-0) 

`DbError` is a catch-all for every database-layer failure. It is thrown in `rest/utils.js` and `rest/health.js` whenever the pg pool cannot acquire a connection (pool exhaustion, TCP timeout, network partition):

```js
// rest/health.js
.catch((err) => {
  throw new DbError(err.message);
})
``` [2](#0-1) 

The error handler is registered globally with no preceding authentication requirement:

```js
// rest/server.js line 144
app.useExt(handleError);
``` [3](#0-2) 

**Why the existing suppression check is insufficient:**

`shouldReturnMessage` correctly suppresses the raw DB error *message body* for 503:
```js
const shouldReturnMessage = (statusCode) => {
  return statusCode.isClientError() || statusCode === httpStatusCodes.BAD_GATEWAY;
};
``` [4](#0-3) 

This means the response body is a generic `"Service Unavailable"` string — the specific pg error is not echoed. **However, the HTTP status code itself (503) is the oracle.** The attacker does not need the body; the status code alone is sufficient to distinguish "DB reachable" (200/404/400) from "DB unreachable" (503).

**Authentication is optional and unenforced:**

`authHandler` only sets a custom rate limit for authenticated users; it does not block unauthenticated requests:
```js
const authHandler = async (req, res) => {
  const credentials = basicAuth(req);
  if (!credentials) {
    return;   // ← unauthenticated requests pass through silently
  }
  ...
};
``` [5](#0-4) 

No rate-limiting middleware is unconditionally applied; `metricsHandler` and `responseCacheCheckHandler` are conditional on config flags. [6](#0-5) 

### Impact Explanation

Any external attacker can poll any public REST endpoint (e.g., `GET /api/v1/transactions`) at will. During normal operation they receive 200/404. The moment the pg connection pool is exhausted or a network partition isolates the DB, every request returns 503. The attacker has a real-time, zero-privilege oracle for internal infrastructure state. This can be used to:

- Detect and timestamp DB outages or maintenance windows with second-level precision.
- Correlate 503 windows with other attack activity (e.g., confirm a DoS against the DB tier is effective).
- Infer deployment topology (primary vs. replica failover timing) by observing 503 duration and recovery patterns.

Severity: **Medium** — information disclosure of internal infrastructure state with no authentication barrier.

### Likelihood Explanation

Preconditions: none. The attacker needs only HTTP access to the public REST API, which is the intended exposure surface. The exploit is a single HTTP GET in a loop. It is repeatable, automatable, and leaves no distinguishing fingerprint beyond normal API traffic. Any script-kiddie or automated scanner can perform this.

### Recommendation

1. **Do not distinguish DB errors from other 5xx errors at the HTTP status code level for external consumers.** Map `DbError` to 500 (Internal Server Error) rather than 503, or return 503 only after an internal retry/circuit-breaker has confirmed sustained unavailability — not on the first pool acquisition failure.
2. **Apply unconditional rate limiting** (e.g., express-rate-limit) before the route handlers so that polling the oracle is throttled regardless of authentication state.
3. If 503 must be retained for operational reasons (load-balancer health signaling), consider returning it only on the `/health/readiness` endpoint (which is already purpose-built for this) and mapping `DbError` on data endpoints to 500.

### Proof of Concept

```bash
# Step 1: Confirm normal operation
curl -o /dev/null -s -w "%{http_code}" https://<mirror-node>/api/v1/transactions
# → 200

# Step 2: Trigger or wait for DB pool exhaustion / network partition

# Step 3: Poll — observe 503 with no credentials required
while true; do
  code=$(curl -o /dev/null -s -w "%{http_code}" https://<mirror-node>/api/v1/transactions)
  echo "$(date -u +%T) HTTP $code"
  sleep 1
done
# Output transitions from 200 → 503 exactly when DB becomes unreachable,
# and back to 200 when it recovers — a precise, unauthenticated availability oracle.
```
