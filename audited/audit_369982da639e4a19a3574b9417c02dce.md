### Title
Prototype-Safe Object Injection via `qs` Depth-1 Parsing Causes Unhandled TypeError in `buildComparatorFilter`

### Summary
An unprivileged attacker can send `ORDER[x]=asc` in any query string, causing `qs.parse()` with `depth: 1` to produce `{ORDER: {x: 'asc'}}`. The object passes through `lowerCaseQueryValue` unchanged (it is not a string), and when downstream `buildComparatorFilter` calls `.split(':')` on the object value, a `TypeError` is thrown. This is reliably reproducible on any endpoint that routes through `buildFilters`/`buildAndValidateFilters`.

### Finding Description

**Step 1 — `qs.parse` with `depth: 1` and `strictDepth: true`** [1](#0-0) 

`strictDepth: true` only throws when depth **exceeds** the limit. `ORDER[x]=asc` produces exactly depth-1 nesting `{ORDER: {x: 'asc'}}`, which is at the limit — no throw occurs.

**Step 2 — `canonicalizeValue` dispatches to `lowerCaseQueryValue` with an object** [2](#0-1) 

Since `value` is not an array, `lowerCaseQueryValue({x: 'asc'})` is called directly.

**Step 3 — `lowerCaseQueryValue` silently passes the object through** [3](#0-2) 

`typeof {x:'asc'} === 'string'` is `false`, so the object is returned unchanged. `req.query.order` is now `{x: 'asc'}`.

**Step 4 — `buildComparatorFilter` calls `.split(':')` on the object** [4](#0-3) 

`{x: 'asc'}.split(':')` → `TypeError: filter.split is not a function`. This propagates up through `buildFilters` → `buildAndValidateFilters` with no type guard. [5](#0-4) 

### Impact Explanation

Every API endpoint that processes the `order` query parameter via `buildAndValidateFilters` (transactions, contracts, tokens, topics, etc.) will throw an unhandled `TypeError` for any request containing `ORDER[x]=<any>`. Depending on Express error-handler configuration this surfaces as an unhandled 500 or, in the absence of a catch-all handler, a process-level uncaught exception. At minimum it is a reliable, unauthenticated DoS against all such endpoints. The `parseLimitAndOrderParams` path is safe (object comparison silently falls through to default), but `buildAndValidateFilters` is not.

### Likelihood Explanation

No authentication is required. The payload is a single, trivially crafted HTTP query parameter (`?ORDER[x]=asc`). It is repeatable at will, requires no special knowledge, and affects all REST API endpoints that accept an `order` parameter via `buildAndValidateFilters`. Any automated scanner or curious user could discover and exploit this.

### Recommendation

1. **In `requestQueryParser`**: After `qs.parse`, reject or flatten any value that is a plain object (not string, number, or array) before it enters the canonicalization map. Add a guard:
   ```js
   if (value !== null && typeof value === 'object' && !Array.isArray(value)) {
     // treat as invalid — throw or skip
   }
   ```
2. **In `buildComparatorFilter`**: Add a type guard before calling `.split`:
   ```js
   if (typeof filter !== 'string') throw new InvalidArgumentError(...);
   ```
3. **In `queryOptions`**: Consider setting `depth: 0` to prevent any nested object parsing entirely, since the API does not use nested query parameters.

### Proof of Concept

```
GET /api/v1/transactions?ORDER[x]=asc HTTP/1.1
Host: <mirror-node-host>
```

1. `qs.parse("ORDER[x]=asc", {depth:1, strictDepth:true})` → `{ORDER: {x:'asc'}}` (no throw, depth == limit).
2. `canonicalizeValue('order', {x:'asc'})` → calls `lowerCaseQueryValue({x:'asc'})` → returns `{x:'asc'}`.
3. `req.query = {order: {x:'asc'}}`.
4. Route handler calls `buildAndValidateFilters(req.query, ...)` → `buildFilters` → `buildComparatorFilter('order', {x:'asc'})`.
5. `{x:'asc'}.split(':')` → `TypeError: filter.split is not a function`.
6. Server returns 500 (or crashes if no error middleware catches it).
