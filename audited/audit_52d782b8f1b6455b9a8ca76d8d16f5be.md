### Title
Prototype Pollution via Case-Bypass in `requestQueryParser()` Enables Persistent Filter Injection and Silent Record Exclusion

### Summary
`requestQueryParser()` lowercases query parameter keys **after** `qs.parse()` runs, meaning `qs`'s case-sensitive `__proto__` block is bypassed by sending `__PROTO__[key]=value`. The resulting object value is then assigned via `caseInsensitiveQueryString['__proto__'] = {key: value}`, which triggers JavaScript's `__proto__` setter and pollutes `Object.prototype`. Because the duplicate-key check uses the `in` operator (which walks the prototype chain), all subsequent requests in the same Node.js process that include the polluted key will have the attacker's value silently merged into their filter array, creating impossible query ranges and returning empty results.

### Finding Description

**Exact code path:**

`rest/middleware/requestHandler.js`, `requestQueryParser()`, lines 54–66:

```js
const parsedQueryString = qs.parse(queryString, queryOptions);   // line 54
const caseInsensitiveQueryString = {};                            // line 56
for (const [key, value] of Object.entries(parsedQueryString)) {
  const lowerKey = key.toLowerCase();                            // line 58
  const canonicalValue = canonicalizeValue(lowerKey, value);
  if (lowerKey in caseInsensitiveQueryString) {                  // line 60 — uses `in`
    caseInsensitiveQueryString[lowerKey] = merge(...);
  } else {
    caseInsensitiveQueryString[lowerKey] = canonicalValue;       // line 64
  }
}
``` [1](#0-0) 

**Root cause — two compounding flaws:**

1. `qs` 6.15.1 blocks `__proto__` (exact, case-sensitive string) but not `__PROTO__`. With `depth: 1` enabled, `qs.parse('__PROTO__[timestamp]=lt:1000000000.000000000')` returns `{ '__PROTO__': { timestamp: 'lt:1000000000.000000000' } }` — a plain object value, not a string. [2](#0-1) 

2. Line 58 lowercases the key to `__proto__` **after** qs has already allowed it. Line 64 then executes `caseInsensitiveQueryString['__proto__'] = { timestamp: 'lt:1000000000.000000000' }`. Because the value is an object, JavaScript's `__proto__` setter fires and sets `Object.prototype.timestamp = 'lt:1000000000.000000000'` — a persistent, process-wide mutation. [3](#0-2) 

3. The `in` operator on line 60 checks the prototype chain. For every subsequent request that includes `timestamp`, `'timestamp' in caseInsensitiveQueryString` returns `true` (from the polluted prototype), `caseInsensitiveQueryString['timestamp']` reads the attacker's value from the prototype, and `merge()` prepends it to the user's own value, producing e.g. `['lt:1000000000.000000000', 'gt:1234567890.000000000']`. [4](#0-3) 

**Why downstream checks fail:**

`buildFilters` uses `Object.entries(query)` which returns only own enumerable properties, so the prototype-polluted key is not directly visible there. However, the merge already happened inside `requestQueryParser` and the merged array is stored as an **own property** on the returned object. `validateAndParseFilters` validates each individual filter value (e.g. `lt:1000000000.000000000` is a syntactically valid timestamp), so it passes. The impossible range (`timestamp < epoch_A AND timestamp > epoch_B` where A < B) is never detected and reaches the SQL layer, returning zero rows. [5](#0-4) 

OpenAPI validation is **disabled by default** (`hiero.mirror.rest.openapi.validation.enabled = false`), so no schema-level guard fires. [6](#0-5) 

### Impact Explanation

A single unauthenticated HTTP request permanently corrupts `Object.prototype` for the lifetime of the Node.js process. Every subsequent API call from any user that includes the polluted parameter key (e.g. `timestamp`) will silently receive an empty result set. This constitutes a persistent, cross-tenant integrity failure: mirror node data exists in the database but is never returned, with no error surfaced to the caller. Severity is **High** — it requires no credentials, is triggered by one request, affects all concurrent users, and persists until process restart.

### Likelihood Explanation

The attack requires only an HTTP GET request with a crafted query string — zero privileges, zero authentication. The bypass (`__PROTO__` uppercase) is a well-known prototype pollution technique. The `depth: 1` setting in `queryOptions` is required to produce an object value (rather than a string), and it is explicitly configured. The attack is fully repeatable and deterministic. Any public-facing mirror node deployment is reachable.

### Recommendation

1. **Immediate fix:** Replace `caseInsensitiveQueryString = {}` with `caseInsensitiveQueryString = Object.create(null)`. A null-prototype object has no `__proto__` setter and no prototype chain, making both the pollution assignment and the `in` false-positive impossible.

2. **Secondary fix:** Replace `if (lowerKey in caseInsensitiveQueryString)` with `if (Object.prototype.hasOwnProperty.call(caseInsensitiveQueryString, lowerKey))` to restrict the lookup to own properties regardless of prototype state.

3. **Defense-in-depth:** Explicitly block `__proto__`, `constructor`, and `prototype` (case-insensitively) after lowercasing, before any assignment.

### Proof of Concept

**Step 1 — Pollute `Object.prototype.timestamp`:**
```
GET /api/v1/transactions?__PROTO__[timestamp]=lt:1000000000.000000000
```
`qs.parse` returns `{ '__PROTO__': { timestamp: 'lt:1000000000.000000000' } }`. `requestQueryParser` lowercases to `__proto__` and assigns the object value, setting `Object.prototype.timestamp = 'lt:1000000000.000000000'`.

**Step 2 — Observe silent record exclusion for all subsequent requests:**
```
GET /api/v1/transactions?timestamp=gt:1234567890.000000000
```
Inside `requestQueryParser`, `'timestamp' in {}` is now `true` (prototype chain). The merge produces `['lt:1000000000.000000000', 'gt:1234567890.000000000']`. Both values pass `isValidTimestampParam`. The SQL WHERE clause becomes `consensus_timestamp < 1000000000000000000 AND consensus_timestamp > 1234567890000000000` — an impossible range. Response: `{ "transactions": [], "links": { "next": null } }` with HTTP 200, silently hiding all records.

**Step 3 — Confirm persistence:**
Any subsequent request to any endpoint that processes `timestamp` (transactions, contracts/results, topic messages, etc.) will exhibit the same empty-result behavior until the Node.js process is restarted.
