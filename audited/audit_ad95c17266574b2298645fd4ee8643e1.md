### Title
Unbounded In-Memory Accumulation of Up to 800,000 Block Items Before Synchronous Consumer Invocation Enables Streaming-Thread DoS

### Summary
The `BlockAssembler.append()` method in `BlockNode.java` adds incoming `BlockItem` lists to the `pending` accumulator **before** evaluating the `maxBlockItems` guard, and the guard uses a strict `>` comparison, meaning exactly 800,000 items (the configured limit) are silently accepted into heap memory without triggering the exception. When `END_OF_BLOCK` arrives, `onEndOfBlock()` assembles the full list and calls `blockStreamConsumer.accept()` **synchronously on the streaming thread**, holding the entire 800,000-item structure in memory for the duration of downstream processing and preventing any further block ingestion from that thread.

### Finding Description

**Exact code path:**

`StreamProperties.java` line 20 sets the default: [1](#0-0) 

`BlockNode.java` `append()` (lines 291–297) adds items to `pending` first, then checks: [2](#0-1) 

The guard is `pendingCount > maxBlockItems` (strictly greater-than), so a `pendingCount` of exactly 800,000 never triggers the exception. All 800,000 items are already resident in the `pending` `List<List<BlockItem>>` at that point.

`onEndOfBlock()` then assembles a flat `ArrayList` from all pending sub-lists and calls the consumer **synchronously**: [3](#0-2) 

The `maxStreamResponseSize` (36 MB) is enforced only as a per-gRPC-message `maxInboundMessageSize` on the channel: [4](#0-3) 

This limits each individual `SubscribeStreamResponse` message but places **no bound on the total accumulated size** across the many messages that compose a single block. An attacker spreads 800,000 items across N messages (each ≤ 36 MB) and the importer accumulates them all.

**Root cause / failed assumption:** The design assumes that `maxBlockItems` acts as a hard ceiling on heap usage, but because the check fires after insertion and uses `>` rather than `>=`, the ceiling is never enforced for the exact limit value. Additionally, the per-message size cap is incorrectly assumed to bound total in-flight memory.

**Exploit flow:**
1. Malicious block node sends `BlockItemSet` containing a valid `BLOCK_HEADER` as the first item.
2. It then streams 799,999 additional `BlockItemSet` messages (each ≤ 36 MB), each containing items with maximally large payloads (e.g., large `EventHeader` or `StateChanges` blobs).
3. `append()` is called for each message; `pendingCount` reaches 800,000 — the check `800000 > 800000` is **false**, so no exception is thrown.
4. The malicious node sends `END_OF_BLOCK`.
5. `onEndOfBlock()` allocates a new `ArrayList`, copies all 800,000 items into it, and calls `blockStreamConsumer.accept()` synchronously.
6. The streaming thread is blocked for the entire duration of `blockStreamReader.read()` + `blockStreamVerifier.verify()` downstream processing while holding the oversized list.

### Impact Explanation
The streaming thread in `BlockNodeSubscriber` is single-threaded (`Executors.newSingleThreadExecutor()`): [5](#0-4) 

While the thread is blocked processing the oversized block, no subsequent blocks can be ingested from any block node. Combined with the heap pressure of holding 800,000 large items, this can cause GC stalls or OOM, effectively halting the mirror node importer. The `maxSubscribeAttempts` / `readmitDelay` error-recovery path is never reached because no exception is thrown — the consumer simply runs to completion (or OOM-crashes).

### Likelihood Explanation
Block nodes are configured with no mutual authentication (`requiresTls = false` by default): [6](#0-5) 

Auto-discovery is enabled by default and merges database-discovered nodes with static config: [7](#0-6) 

Any entity that can register a block node endpoint in the network address book (or perform DNS/IP spoofing against a plaintext gRPC connection) can act as a block node without any credential. The attack is repeatable: after the importer recovers, the same malicious node can be re-selected and the attack repeated indefinitely.

### Recommendation
1. **Change the guard to `>=`** in `append()` so that the 800,000th item triggers the exception before it is added to `pending`:
   ```java
   if (pendingCount >= streamProperties.getMaxBlockItems()) { throw ... }
   pending.add(blockItems);
   pendingCount += blockItems.size();
   ```
2. **Add a total-bytes accumulator** alongside `pendingCount` and enforce a separate `maxPendingBytes` limit that accounts for cross-message accumulation, independent of `maxStreamResponseSize`.
3. **Make `blockStreamConsumer.accept()` non-blocking** (hand off to a bounded queue) so the streaming thread is never stalled by downstream processing.
4. **Enforce mutual TLS** for block node connections to prevent unauthenticated actors from acting as block nodes.

### Proof of Concept
```
1. Stand up a gRPC server implementing BlockStreamSubscribeService on any host/port.
2. Register that host:port as a block node in the importer config
   (hiero.mirror.importer.block.nodes[0].host=attacker.host).
3. On subscribeBlockStream(), respond with:
   a. SubscribeStreamResponse { block_items: { block_items: [BLOCK_HEADER{number:1}] } }
   b. For i in 1..799999:
        SubscribeStreamResponse { block_items: { block_items: [EventHeader with ~35MB payload] } }
   c. SubscribeStreamResponse { end_of_block: { block_number: 1 } }
4. Observe: BlockAssembler.pendingCount reaches 800,000; the guard
   (800000 > 800000) == false is never triggered; onEndOfBlock() assembles
   a 800,000-element ArrayList and calls blockStreamConsumer.accept()
   synchronously, blocking the streaming thread and exhausting heap.
5. The importer stops processing new blocks for the duration of the call
   (or crashes with OutOfMemoryError).
```
