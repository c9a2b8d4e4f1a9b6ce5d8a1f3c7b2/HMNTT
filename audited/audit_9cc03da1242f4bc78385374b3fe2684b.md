### Title
Shared Global `boundedElastic` Scheduler Between `repeatWhen` and `timeout` Enables Timeout Bypass via Scheduler Thread Starvation

### Summary
`PollingTopicMessageRetriever.retrieve()` assigns both the `repeatWhen` polling loop and the `timeout` operator to the same global `Schedulers.boundedElastic()` instance. Because `poll()` executes a synchronous, blocking JDBC call (`Flux.fromStream(topicMessageRepository.findByFilter(...))`) on that same scheduler thread, an attacker who opens enough concurrent gRPC subscriptions can exhaust the scheduler's thread pool. When all threads are blocked on stalled JDBC calls (e.g., during a network partition where TCP connections hang until OS timeout), the `timeout` operator's scheduled task is queued but cannot execute, allowing subscriptions to hold gRPC streams, DB connection pool slots, and heap memory indefinitely beyond the configured 60-second timeout.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java`, constructor line 41:
```java
scheduler = Schedulers.boundedElastic();
```
`Schedulers.boundedElastic()` returns the **global** singleton scheduler shared across the entire application. The same reference is used at lines 55 and 59:

```java
.repeatWhen(RepeatSpec.create(...)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))          // line 55 â€“ repeat delay fires on scheduler thread
...
.timeout(retrieverProperties.getTimeout(), scheduler)  // line 59 â€“ timeout timer also on same scheduler
```

When `repeatWhen` fires after its delay, it re-subscribes to `Flux.defer(() -> poll(context))` on the scheduler thread. `poll()` at line 78 calls:
```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
`Flux.fromStream` is synchronous â€” it iterates the JDBC `Stream` on the calling (scheduler) thread. During a network partition, the underlying TCP connection to PostgreSQL can hang silently. The PostgreSQL `statementTimeout` (default 10 s per docs) is a server-side mechanism that requires a live connection to deliver the cancellation; a silently-dropped TCP link means the timeout signal never arrives, and the JDBC call blocks for the full OS TCP keepalive/retransmit window (minutes).

**Root cause / failed assumption:** The design assumes the `timeout` operator's timer fires independently of scheduler load. In Reactor's `BoundedElasticScheduler`, delayed tasks (including the timeout task) are submitted to the same `ScheduledExecutorService` worker pool. If all worker threads are occupied with blocked JDBC calls, the timeout `Runnable` is enqueued but never dispatched until a thread becomes free â€” which cannot happen while the partition persists and new subscriptions keep arriving.

**Why existing checks fail:**
- `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` at line 58 only triggers on terminal errors; a hanging JDBC call produces no error signal.
- `timeout(retrieverProperties.getTimeout(), scheduler)` at line 59 is the sole resource-bounding mechanism, but it is self-defeating when the scheduler it depends on is starved.
- `maxConcurrentCallsPerConnection = 5` (per docs) limits calls per connection, not total connections; an attacker opens many connections.
- The global `boundedElastic` scheduler is also consumed by `SharedPollingTopicListener`, `PollingTopicListener`, `SharedTopicListener`, `NetworkServiceImpl`, and `TopicMessageServiceImpl` â€” all competing for the same thread pool, lowering the starvation threshold further.

**Exploit flow:**
1. Attacker opens `N` concurrent gRPC connections, each issuing a `subscribeTopic` request with `startTime=0` (historical query, no `endTime`, no `limit`).
2. Each subscription calls `retrieve(filter, true)` â†’ `poll()` â†’ `findByFilter()` on a scheduler thread.
3. A network partition (or attacker-induced DB overload) causes JDBC calls to hang.
4. After `10 * availableProcessors` subscriptions (e.g., 80 on an 8-core host), the scheduler thread pool is exhausted.
5. The `timeout` task for every active subscription is queued but never executed.
6. Subscriptions hold gRPC streams, `PollingContext` heap objects, and DB connection pool slots indefinitely.
7. DB connection pool exhaustion causes all subsequent legitimate queries to fail.

### Impact Explanation
- **DB connection pool exhaustion**: Each stalled `findByFilter` call holds a JDBC connection. With the pool exhausted, all other gRPC and REST queries fail.
- **gRPC stream exhaustion**: Netty gRPC stream slots are held open, eventually hitting server-level limits and rejecting new connections.
- **Heap pressure**: Each `PollingContext` and its associated `Flux` chain accumulates in memory for the duration of the stall.
- **Cascading service unavailability**: Because the global `boundedElastic` scheduler is shared, starvation also blocks `SharedPollingTopicListener`'s polling loop and `TopicMessageServiceImpl`'s safety-check path, halting message delivery for all subscribers.
- Severity: **High** â€” full service DoS achievable by an unprivileged external user with no credentials.

### Likelihood Explanation
- The gRPC port (5600) is publicly exposed with no authentication required to call `subscribeTopic`.
- No per-IP or per-user subscription rate limit is present in the code.
- Opening 80â€“200 concurrent gRPC streams is trivially achievable with standard gRPC client libraries.
- Network partitions between the gRPC pod and the database are a realistic operational event (pod restarts, network policy changes, cloud provider blips).
- The attacker does not need to cause the partition; they only need to have subscriptions open when one occurs. Alternatively, they can induce DB slowness by flooding it with queries through their own subscriptions, achieving the same effect without an external partition.
- The attack is repeatable and can be sustained for as long as the attacker maintains connections.

### Recommendation
1. **Use a dedicated, independent scheduler for the `timeout` operator** â€” never the same scheduler that executes blocking upstream work:
   ```java
   // Replace line 59:
   .timeout(retrieverProperties.getTimeout(), Schedulers.parallel())
   // or a dedicated single-thread timer scheduler:
   .timeout(retrieverProperties.getTimeout(), Schedulers.newSingle("retriever-timeout"))
   ```
2. **Offload blocking JDBC calls off the scheduler** by wrapping `poll()` with `subscribeOn(Schedulers.boundedElastic())` and using a *separate* scheduler for `repeatWhen` delays, or switch to a non-blocking R2DBC driver.
3. **Add a JDBC socket/connect timeout** at the datasource level (e.g., `socketTimeout=15` in the JDBC URL) so that silently-dropped TCP connections are detected within a bounded window regardless of server-side `statementTimeout`.
4. **Enforce a per-IP or global subscription concurrency limit** at the gRPC interceptor layer to bound the number of concurrent `subscribeTopic` streams.

### Proof of Concept
**Preconditions:** Mirror node gRPC service running; database reachable initially.

**Steps:**
1. Write a gRPC client that opens 200 concurrent bidirectional `subscribeTopic` streams, each with `startTime=0`, no `endTime`, no `limit`, targeting a valid topic ID.
2. Start the client â€” all 200 subscriptions begin polling the DB.
3. Induce a network partition between the gRPC pod and PostgreSQL (e.g., `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` on the gRPC host, or use a network policy).
4. Observe: JDBC calls in `findByFilter` hang on all scheduler threads.
5. Wait 120 seconds (2Ă— the configured 60 s `retrieverProperties.getTimeout()`).
6. **Expected (correct) behavior:** All 200 subscriptions should have received a `TimeoutException` and terminated, releasing resources.
7. **Actual behavior:** No `TimeoutException` is delivered. All 200 subscriptions remain open. DB connection pool is fully held. New `subscribeTopic` calls from legitimate clients hang or fail immediately. The `subscriberCount` gauge remains at 200+.
8. Restore network connectivity â€” only then do the JDBC calls unblock, the scheduler threads free up, and the timeout tasks finally execute (60 s after the threads become free, not 60 s after the partition began). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
