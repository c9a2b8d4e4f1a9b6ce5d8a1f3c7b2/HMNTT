### Title
Unauthenticated EVM Address Parameter Triggers Mandatory DB Lookup That Fails Independently, Causing Full Request Failure

### Summary
Any unauthenticated user can supply `account.id=<evm_address>` to `GET /api/v1/balances`, which causes `parseAccountIdQueryParam()` in `rest/balances.js` to call `EntityService.getEncodedId(value, false)`, injecting a live `Promise` into the params array that is later resolved via `Promise.all()`. If this entity-lookup DB call throws a `DbError` (e.g., statement timeout, connection failure), the entire request fails with HTTP 503 — even though the main balance query was never attempted and would have succeeded. The `requireResult = false` flag only suppresses "not found" errors, not database errors.

### Finding Description

**Code path:**

`rest/balances.js` `getBalances()` (lines 83–88):
```js
const [accountQuery, accountParamsPromise] = parseAccountIdQueryParam(req.query, 'ab.account_id');
const accountParams = await Promise.all(accountParamsPromise);
``` [1](#0-0)

**Root cause / failed assumption:** The `requireResult = false` argument passed at line 329 of `balances.js` was designed to make the entity lookup *optional* — returning `null` when no entity is found. However, this flag only suppresses `NotFoundError`; it has no effect on `DbError`. When the entity-lookup query fails at the DB level, the error propagates through `Promise.all()` and aborts `getBalances()` before the main balance query (line 153) is ever issued. [7](#0-6) 

### Impact Explanation

Any unauthenticated caller can force the `/api/v1/balances` endpoint to return HTTP 503 by including a valid EVM address in `account.id` during any period of DB stress (statement timeouts, connection pool exhaustion, transient network errors). The main balance query — which reads from `entity_balance`, a different code path — is never reached. The endpoint becomes selectively unavailable for all requests that include an EVM address filter, while requests using numeric account IDs continue to succeed. This is an availability (DoS) vulnerability with no authentication requirement.

### Likelihood Explanation

The precondition is a DB under stress or a transient network hiccup — a realistic operational condition. The attacker needs no credentials, no special knowledge beyond knowing that EVM addresses are accepted as `account.id` values (documented in the API), and no rate-limit bypass. The trigger is a single HTTP GET request. The attack is repeatable and can be automated to amplify the impact of any DB degradation event.

### Recommendation

In `parseAccountIdQueryParam()` in `rest/balances.js`, wrap the `EntityService.getEncodedId()` call so that `DbError` is caught and treated as a non-fatal condition (e.g., skip the filter or return `null`), consistent with the intent of `requireResult = false`. Alternatively, resolve the Promise inside `parseAccountIdQueryParam` with a try/catch that converts `DbError` to `null`, and then filter out `null` values before building the SQL clause. The `requireResult = false` contract should be extended to cover DB-level failures, not just "not found" results.

### Proof of Concept

1. Start the mirror-node REST service connected to a PostgreSQL instance.
2. Introduce a statement timeout or simulate a transient DB error on the `entity` table (e.g., `ALTER TABLE entity SET (autovacuum_enabled = false)` and lock the table, or use a proxy to drop connections mid-query).
3. Send:
   ```
   GET /api/v1/balances?account.id=0x000000000000000000000000000000000000abcd
   ```
4. Observe HTTP 503 response.
5. Send the same request without the EVM address:
   ```
   GET /api/v1/balances
   ```
6. Observe HTTP 200 with balance data — confirming the main query path is unaffected and the failure is caused solely by the entity-lookup triggered by the EVM address parameter.
