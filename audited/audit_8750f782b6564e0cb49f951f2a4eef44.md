### Title
Unbounded Cache Thrashing via Token Relationship Endpoint Enables Non-Network DoS

### Summary
The `tokenCache` in `rest/service/tokenService.js` is a module-level `quickLru` instance with a fixed `maxSize` and no `maxAge`. An unprivileged attacker can continuously request `/accounts/:id/tokens` for accounts whose associated token IDs are not present in the cache (or exceed the cache capacity), forcing `getCachedTokens()` to always execute `tokenCacheQuery` against the database for every request, doubling DB load and degrading service availability.

### Finding Description

**Exact code location:**

`rest/service/tokenService.js`, lines 12–14 — cache initialization: [1](#0-0) 

`rest/service/tokenService.js`, lines 136–161 — `getCachedTokens()`: [2](#0-1) 

`rest/controllers/tokenController.js`, lines 66–92 — `getTokenRelationships()` (the public endpoint handler): [3](#0-2) 

**Root cause:**

The `tokenCache` is a `quickLru` with only `maxSize` set — no `maxAge`, no rate limiting, no per-IP throttling: [1](#0-0) 

`getCachedTokens()` separates token IDs into cached vs. uncached, and for any uncached IDs it unconditionally fires `tokenCacheQuery` against the database: [4](#0-3) 

Because `quickLru` uses LRU eviction when `maxSize` is reached, an attacker who cycles through requests referencing more unique token IDs than `maxSize` will continuously evict previously cached entries, ensuring every subsequent request for those token IDs results in a DB query.

**Failed assumption:** The design assumes the working set of token IDs accessed in practice will fit within `maxSize`. On a large network (Hedera mainnet has millions of tokens), this assumption fails trivially.

**Exploit flow:**

1. Attacker enumerates accounts with large, diverse token association sets (publicly available via the same API).
2. Attacker cycles through these accounts in a pattern where each batch of requests references token IDs not currently in the cache (or exceeds `maxSize` in aggregate).
3. Each request triggers `getCachedTokens()` → `uncachedTokenIds` is always non-empty → `tokenCacheQuery` fires against the DB.
4. Newly cached entries are immediately evicted by subsequent requests referencing other token IDs, keeping the cache perpetually cold.

**Why existing checks are insufficient:**

- `getTokenRelationships()` requires no authentication or API key for read access. [5](#0-4) 
- There is no rate limiting applied at the controller or middleware level for this endpoint path.
- `putTokenCache()` (called from the `/tokens` list endpoint) uses a `has()` guard, but `getCachedTokens()` has no such guard against repeated DB queries for the same uncached IDs across concurrent requests. [6](#0-5) 

### Impact Explanation
Every cache miss results in an additional `SELECT` against the `token` table. Under sustained cache thrashing, every call to `getTokenRelationships()` generates two DB queries instead of one (the `token_account` join query plus the `tokenCacheQuery`). This doubles effective DB load, exhausts DB connection pool slots, and degrades response times for all users of the mirror node — constituting a non-network-based DoS against a high-market-cap project.

### Likelihood Explanation
No privileges are required. The attacker only needs knowledge of account IDs with diverse token associations, which is freely discoverable via the same public API. The attack is trivially scriptable, repeatable, and requires no special infrastructure — a single machine with a modest HTTP request rate is sufficient to keep the cache thrashed given a large enough token universe.

### Recommendation
1. **Add `maxAge` to `tokenCache`** so entries expire and the cache self-heals, reducing the window of thrashing impact.
2. **Implement per-IP or global rate limiting** on `/accounts/:id/tokens` at the middleware layer.
3. **Deduplicate in-flight DB requests** (request coalescing / promise memoization): if multiple concurrent requests miss the cache for the same token ID, only one DB query should be issued and the result shared.
4. **Increase `maxSize`** based on the actual token universe size of the target network, or make it configurable per deployment.
5. Consider **negative caching**: cache a sentinel for token IDs that returned no rows, preventing repeated DB hits for non-existent tokens.

```javascript
  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
