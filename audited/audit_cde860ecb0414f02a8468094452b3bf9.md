### Title
Unbounded In-Flight Request Accumulation Leading to Memory Exhaustion During Network Partition

### Summary
`getAccountCryptoAllowances()` in `rest/controllers/cryptoAllowanceController.js` has no controller-level or server-level request timeout. During a network partition between the Node.js process and PostgreSQL, the `pg` pool's internal queue grows without bound because `statement_timeout` is a server-side PostgreSQL signal that is never delivered over a severed TCP connection, and the Traefik ingress for the REST API lacks the `inFlightReq` middleware present in other services. An unprivileged attacker can exploit this to exhaust the 600 MiB memory limit and crash the Node.js process.

### Finding Description

**Exact code path:**

`rest/routes/accountRoute.js` line 17 routes `GET /:id/allowances/crypto` to `CryptoAllowanceController.getAccountCryptoAllowances`. [1](#0-0) 

`getAccountCryptoAllowances` (lines 76–98) is a plain `async` function with no timeout guard, no `AbortController`, and no `Promise.race`. It awaits two sequential DB calls: [2](#0-1) 

`rest/dbpool.js` configures the `pg.Pool` with `statement_timeout` and `connectionTimeoutMillis` but sets no `idleTimeoutMillis` or queue-depth cap: [3](#0-2) 

`rest/server.js` registers no `server.timeout`, no `connect-timeout` middleware, and no in-flight-request limiter anywhere in the middleware chain: [4](#0-3) 

**Root cause and failed assumption:**

The design assumes `statement_timeout: 20000` (20 s, default per docs) will bound query duration: [5](#0-4) 

This assumption fails during a network partition. `statement_timeout` is enforced by the PostgreSQL backend process, which sends a cancellation over the same TCP connection. When that connection is severed, the cancellation is never delivered. The Node.js `pg` client's `query()` promise hangs indefinitely — until the OS TCP keepalive fires (default ~2 hours on Linux).

`connectionTimeoutMillis: 20000` only governs acquiring a free slot from the pool, not the duration of an already-executing query. Once all 10 pool connections (`max: 10`) are stuck, new requests queue inside `pg.Pool`'s internal queue, which has **no depth limit**.

**Exploit flow:**

1. Attacker triggers or waits for a network partition between the REST pod and PostgreSQL.
2. Attacker sends a sustained flood of `GET /api/v1/accounts/0.0.1/allowances/crypto` requests.
3. The Traefik ingress for the REST service has `retry: attempts: 10`, multiplying each client request into up to 10 backend requests: [6](#0-5) 
4. The 10 pool connections are immediately occupied. Subsequent requests queue in `pg.Pool`. Each queued request holds an Express `res` object and associated heap allocations open.
5. After 20 s, `connectionTimeoutMillis` fires for queued requests, but with 10× retry amplification and no rate limit, the attacker can sustain the queue faster than it drains.
6. Heap grows until the 600 MiB container memory limit is hit, triggering OOM kill.

**Why existing checks are insufficient:**

- `statement_timeout` (DB-level, 20 s): not delivered over a severed TCP connection.
- `connectionTimeoutMillis` (20 s): only bounds pool-slot acquisition, not query execution; with 10× retry amplification the queue refills faster than it drains.
- Traefik `circuitBreaker` (`NetworkErrorRatio() > 0.25`): requests are *queuing*, not immediately erroring, so the ratio stays low and the breaker does not trip promptly. [7](#0-6) 
- **No `inFlightReq` middleware** in the REST chart. Compare with the Rosetta chart, which has `inFlightReq: amount: 5` and `rateLimit: average: 10` — both absent from the REST chart. [6](#0-5) 

### Impact Explanation

Memory exhaustion crashes the single-threaded Node.js process, taking the entire REST API pod offline. Kubernetes will restart it, but the attacker can repeat the attack immediately. The pod memory limit is 600 MiB: [8](#0-7) 
With ~50–100 KB per in-flight request object and 10× Traefik retry amplification, ~5,000–10,000 accumulated requests suffice to exhaust available heap — achievable at modest request rates sustained over the 20 s `connectionTimeoutMillis` window.

### Likelihood Explanation

- **No privilege required**: the endpoint is publicly accessible.
- **Network partition**: achievable by an attacker who can disrupt the pod-to-database network path (e.g., network policy manipulation, cloud infrastructure disruption, or simply targeting a degraded environment).
- **Amplification**: the 10-retry Traefik policy means the attacker needs only ~50 client requests/second to generate ~500 backend requests/second.
- **Repeatability**: after OOM kill and pod restart, the attack can be immediately re-launched.

### Recommendation

1. **Add a server-level request timeout** in `rest/server.js`:
   ```js
   server.setTimeout(30_000); // 30 s hard deadline per request
   ```
2. **Add a controller-level timeout** in `getAccountCryptoAllowances` using `Promise.race` or `AbortController` to abort the DB query if no response arrives within a configurable deadline.
3. **Add `inFlightReq` middleware** to the REST Traefik middleware chain (as already done for Rosetta/GraphQL) to cap concurrent backend requests per source IP.
4. **Reduce Traefik `retry.attempts`** from 10 to 2–3 to limit amplification.
5. **Set `idleTimeoutMillis`** on the `pg.Pool` and consider a finite pool queue depth via a wrapper that rejects immediately when the queue exceeds a threshold.
