### Title
Connection Pool Exhaustion via Unauthenticated Concurrent Requests to `/contracts/:contractId/results/:consensusTimestamp`

### Summary
The `getContractResultsByTimestamp()` handler in `rest/controllers/contractController.js` calls `getDetailedContractResults()`, which unconditionally fires 5 concurrent database queries per request via `Promise.all()`. Because the Node.js REST API has no rate-limiting middleware and the endpoint requires no authentication, an unprivileged attacker can saturate the shared `pg` connection pool by issuing many concurrent requests with valid, publicly-known consensus timestamps, denying service to all other API consumers.

### Finding Description

**Exact code path:**

`getContractResultsByTimestamp()` (lines 1002–1042) calls `getDetailedContractResults()` (lines 1205–1220) unconditionally after a single guard query succeeds:

```js
// contractController.js:1016-1017
const [contractResults, ethTransactions, recordFile, contractLogs, contractStateChanges] =
  await this.getDetailedContractResults(contractDetails, contractId);
```

`getDetailedContractResults()` issues exactly 5 queries in parallel:

```js
// contractController.js:1205-1219
getDetailedContractResults = async (contractDetails, contractId = undefined) => {
  return Promise.all([
    ContractService.getContractResultsByTimestamps(...),
    TransactionService.getEthTransactionByTimestampAndPayerId(...),
    RecordFileService.getRecordFileBlockDetailsFromTimestamp(...),
    ContractService.getContractLogsByTimestamps(...),
    ContractService.getContractStateChangesByTimestamps(...),
  ]);
};
```

A 6th query may follow at line 1023 if `ethTransaction?.callDataId` is valid.

**Connection pool:**

`rest/dbpool.js` (lines 7–16) creates a single `pg.Pool` with `max: config.db.pool.maxConnections`. The pool is shared across all concurrent requests. With a typical default of 10 connections, just 2 simultaneous requests to this endpoint consume all 10 pool slots.

**Missing control:**

`rest/server.js` registers: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, `handleError` — no rate-limiting or concurrency-limiting middleware exists for the Node.js REST API. The `ThrottleConfiguration.java` found in the repo applies only to the separate Spring Boot web3 service, not this API.

**Root cause:** `Promise.all()` with 5 queries per request + no rate limiting + shared bounded connection pool + unauthenticated public endpoint.

### Impact Explanation

When the pool is exhausted, all subsequent queries across the entire REST API block waiting for `connectionTimeoutMillis`. Once that timeout fires, callers receive errors. This degrades or completely denies service to all REST API consumers — not just the contract endpoint — because the pool is global. In a production deployment where `mirror_rest` is limited to 250 server-side connections via PgBouncer (`max_user_connections: 250`), an attacker sending ~50 concurrent requests holds all 250 PgBouncer slots (5 queries × 50 requests), blocking every other REST API user. This matches the stated severity: shutdown of ≥30% of network processing nodes without brute force.

### Likelihood Explanation

- **No authentication required.** Any internet user can call the endpoint.
- **Valid timestamps are public.** Consensus timestamps appear in block explorers, transaction receipts, and other mirror node API responses. An attacker trivially collects hundreds of valid `(contractId, consensusTimestamp)` pairs.
- **Trivially scriptable.** A simple loop with `curl -s ... &` or any async HTTP client reproduces the condition.
- **Sustained attack is cheap.** The attacker need not brute-force anything; they replay known-good timestamps indefinitely.
- **`statement_timeout`** (the only DB-side guard) limits individual query duration but does not prevent the pool from being continuously re-saturated as long as the attacker's request rate exceeds the drain rate.

### Recommendation

1. **Add per-IP rate limiting** to the Node.js REST API (e.g., `express-rate-limit`) before the contract routes, targeting expensive endpoints.
2. **Limit per-request DB concurrency** — replace the bare `Promise.all([5 queries])` with a semaphore or sequential execution for this endpoint, or use a dedicated sub-pool with a lower `max`.
3. **Set a hard HTTP concurrency limit** at the reverse-proxy/ingress layer (e.g., Nginx `limit_req`, HAProxy `maxconn`) so a single client cannot hold more than N in-flight requests.
4. **Increase `statement_timeout`** awareness: ensure it is set aggressively enough (e.g., ≤5 s) so held connections are released quickly under attack.

### Proof of Concept

```bash
# Step 1: Obtain a valid (contractId, consensusTimestamp) pair from any public mirror node
CONTRACT="0.0.5001"
TIMESTAMP="987654.000123456"
TARGET="https://<mirror-node-host>/api/v1"

# Step 2: Fire N concurrent requests (adjust N to match pool size / 5)
for i in $(seq 1 60); do
  curl -s "$TARGET/contracts/$CONTRACT/results/$TIMESTAMP" &
done
wait

# Step 3: Observe that legitimate requests to any REST endpoint now time out
# (connection pool exhausted; connectionTimeoutMillis exceeded)
curl -v "$TARGET/accounts/0.0.1"
# Expected: connection timeout error or HTTP 500/503
```

Each of the 60 concurrent requests acquires 5 pool connections simultaneously via `Promise.all()`, consuming 300 logical connection slots. With a pool capped at 10–250 connections, the pool is saturated and all other API traffic is blocked for the duration of the queries.
