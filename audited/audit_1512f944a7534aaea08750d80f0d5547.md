### Title
Cache Stampede in `doGetTransactions()` Enables Unauthenticated Connection Pool Exhaustion via Concurrent Cache-Miss DB Queries

### Summary
The `doGetTransactions()` function in `rest/transactions.js` delegates to `cache.get()` which has no stampede/thundering-herd protection: concurrent requests that arrive before Redis is populated all independently invoke the `loader` callback, each firing a full `getTransactionsDetails()` multi-CTE SQL query against the shared `pg` connection pool. Because the Redis write-back (`mset`) is fire-and-forget and not awaited, the race window persists across the entire round-trip of the first DB query. An unprivileged attacker can exploit this to exhaust the finite connection pool, causing all subsequent DB queries — including those serving new-transaction lookups — to queue or time out.

### Finding Description
**Code path:**

`rest/transactions.js:695–705` — `doGetTransactions()` calls `cache.get(payerAndTimestamps, loader, keyMapper)` where `loader = (keys) => getTransactionsDetails(keys, order).then(...)`. [1](#0-0) 

`rest/cache.js:95–132` — `cache.get()` performs `mgetBuffer` to check Redis, identifies missing keys, then calls `await loader(missingKeys)`. After the loader returns, it calls `this.redis.mset(newValues)` **without `await`** — fire-and-forget. [2](#0-1) 

`rest/cache.js:99–101` — When Redis is not ready (`!this.ready`), the method falls through directly to `return loader(keys)` with zero caching, making every request a cache miss unconditionally. [3](#0-2) 

`rest/transactions.js:616–663` — `getTransactionsDetails()` executes a three-CTE query joining `crypto_transfer`, `token_transfer`, and `transaction` tables via `pool.queryQuietly()`, consuming one connection from the shared pool per call. [4](#0-3) 

`rest/dbpool.js:7–16` — The pool is bounded by `config.db.pool.maxConnections`. [5](#0-4) 

**Root cause / failed assumption:** The `cache.get()` implementation assumes that the first request to populate a key will do so before any concurrent request checks the same key. This assumption fails because: (a) `mset` is not awaited, so the cache is not populated until after the current event-loop tick completes, and (b) there is no in-flight deduplication (no promise coalescing, no mutex, no "already loading" sentinel). N concurrent requests for overlapping or distinct uncached timestamp ranges each independently call `getTransactionsDetails()`, each consuming a pool connection.

### Impact Explanation
The shared `pg` pool (`global.pool`) has a hard `max` connection limit. With enough concurrent requests (easily achievable from a single client using `Promise.all` or `async` HTTP libraries), all pool slots are occupied by in-flight `getTransactionsDetails()` queries. New queries — including those from legitimate users polling for the latest transactions — block on `pool.connect()` until `connectionTimeoutMillis` is reached, at which point they error. This directly delays the visibility of newly ingested transactions to all API consumers for the duration of the attack, which can far exceed any reasonable "block time" threshold when the pool is fully saturated and connection timeout is long.

### Likelihood Explanation
No authentication is required. The `/api/v1/transactions` endpoint is publicly accessible. An attacker needs only an HTTP client capable of issuing concurrent requests with varied `timestamp` query parameters (to generate distinct cache keys and guarantee misses). The attack is trivially repeatable and scriptable. Application-level rate limiting does not exist in the REST service itself — the only potential mitigation is an optional Traefik `inFlightReq` middleware that is conditionally rendered only when `global.middleware` and `middleware` values are both set in the Helm chart, meaning it is absent in many deployments. [6](#0-5) 

### Recommendation
1. **Add stampede protection to `cache.get()`**: Use a promise-coalescing map (keyed by cache key) so that concurrent requests for the same uncached key share a single in-flight `loader` call rather than each spawning their own.
2. **Await the `mset` write-back** (or at minimum complete it synchronously before returning) to close the race window.
3. **Add application-level concurrency limiting** (e.g., a semaphore or token bucket) specifically for the `getTransactionsDetails()` DB call, independent of infrastructure-level Traefik middleware.
4. **Set a low `connectionTimeoutMillis`** on the pool so that pool exhaustion fails fast rather than queuing indefinitely.

### Proof of Concept
```bash
# Generate 200 concurrent requests with distinct timestamp ranges to guarantee cache misses
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/transactions?timestamp=gt:$(date -d "-${i} seconds" +%s).000000000&limit=100" &
done
wait

# Observe: subsequent legitimate requests to /api/v1/transactions hang or return 500/503
# as the pg connection pool is exhausted
curl -v "https://<mirror-node>/api/v1/transactions"
```

Each background request triggers `getTransactionTimestamps()` (one pool connection) followed by `getTransactionsDetails()` (one pool connection) with no cache hit, saturating the pool and delaying all new-transaction queries.
