### Title
Unauthenticated Database Connection Pool Exhaustion via Stateless `name` ILIKE Queries on `GET /api/v1/tokens`

### Summary
The `getTokensRequest()` function in `rest/tokens.js` accepts a `name` query parameter that generates an `ILIKE '%X%'` SQL query against the token table. When `hasNameParam` is true, pagination links are suppressed entirely, making every request stateless and equally expensive with no cursor optimization. Because the REST API has no rate-limiting middleware and the database connection pool defaults to only 10 connections with a 20-second statement timeout, an unprivileged attacker can exhaust the entire connection pool with a small number of concurrent requests, causing all REST API endpoints to fail.

### Finding Description

**Exact code path:**

`rest/tokens.js`, `extractSqlFromTokenRequest()`, line 177 constructs the query:
```js
conditions.push(`t.name ILIKE $${params.push('%' + filter.value + '%')}`);
``` [1](#0-0) 

`getTokensRequest()`, lines 405–406 unconditionally nulls the pagination link when `name` is present:
```js
const nextLink = hasNameParam
  ? null
  : utils.getPaginationLink(...)
``` [2](#0-1) 

**Root cause — three compounding weaknesses:**

1. **No rate limiting on the REST API.** The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) exists only in the `web3` module for contract calls. The `rest/` server has no per-IP or global rate-limiting middleware for any endpoint. [3](#0-2) 

2. **Tiny connection pool with a long statement timeout.** `rest/dbpool.js` configures the pool with `max: config.db.pool.maxConnections` (default **10**) and `statement_timeout: config.db.pool.statementTimeout` (default **20 000 ms**). Each `name` query can hold a connection for up to 20 seconds. [4](#0-3) 

3. **Stateless, cursor-free queries.** Because `hasNameParam` forces `nextLink = null`, there is no cursor state. Every request re-executes the full `ILIKE '%X%'` scan from scratch, with no way for the server to amortize cost across requests. [5](#0-4) 

**Why the GIN trigram index does not neutralize the attack:**
A GIN trigram index (`token__name on token using gin (name gin_trgm_ops)`) was added in migrations `V1.98.2` and `V2.3.2`. [6](#0-5) 

The index reduces per-query latency for selective patterns, but it does **not** prevent pool exhaustion: (a) for high-frequency 3-character trigrams (e.g., `"USD"`, `"BTC"`, `"tok"`), the index still returns large result sets that must be sorted and limited; (b) even a 1–2 second query repeated across 10 concurrent connections saturates the pool; (c) the index is irrelevant to the absence of rate limiting.

**Name validation** enforces only `op === eq` and `isByteRange(val, 3, 100)` — no restriction on how many requests per second a client may issue. [7](#0-6) 

### Impact Explanation

With the pool capped at 10 connections and each `name` query able to hold a connection for up to 20 seconds, an attacker needs only **10 concurrent HTTP requests** to exhaust the pool. All subsequent REST API requests — including unrelated endpoints like `/api/v1/transactions`, `/api/v1/accounts`, etc. — queue waiting for a free connection and eventually fail with a connection-timeout error (`connectionTimeoutMillis` default 20 000 ms). This constitutes a full, non-network-based application-layer DoS of the mirror node REST API, affecting every consumer of the API without requiring any privileged access or special knowledge.

### Likelihood Explanation

The attack requires zero authentication, zero special tooling, and zero prior knowledge beyond the public API documentation. A single attacker with a standard HTTP client (e.g., `curl`, `ab`, `wrk`) can sustain the attack indefinitely by sending 10 concurrent requests in a loop. The minimum valid `name` value is 3 bytes, so the request is trivially constructed: `GET /api/v1/tokens?name=abc`. The attack is fully repeatable and stateless on the attacker's side.

### Recommendation

1. **Add rate limiting to the REST API** — apply a per-IP (or global) rate limiter (e.g., `express-rate-limit`) specifically to the `GET /api/v1/tokens` route, with a tighter limit when the `name` parameter is present.
2. **Reduce the statement timeout for `name` queries** — set a shorter per-query timeout (e.g., 2–3 seconds) for ILIKE searches, separate from the general 20-second timeout.
3. **Increase the connection pool size** or use a connection-queue depth limit so that pool exhaustion does not cascade to unrelated endpoints.
4. **Enforce a minimum trigram selectivity threshold** — reject `name` values whose trigram set is known to be non-selective (e.g., single-character repeated patterns), or require a minimum of 4–5 characters to improve index selectivity.
5. **Consider caching** name-search results in Redis (the infrastructure already exists) with a short TTL to absorb repeated identical queries.

### Proof of Concept

```bash
# Exhaust the 10-connection pool with 10 concurrent, long-running name queries.
# Uses a common 3-char trigram to maximize result set size and query duration.
for i in $(seq 1 10); do
  curl -s "http://<mirror-node-host>:5551/api/v1/tokens?name=tok" &
done
wait

# Now issue a normal request — it will time out or return a 503/connection error
# because all pool connections are occupied.
curl -v "http://<mirror-node-host>:5551/api/v1/transactions?limit=1"
```

Expected result: the final `transactions` request hangs for up to 20 seconds and then fails, demonstrating full REST API unavailability for the duration of the attack. The attack loop can be sustained indefinitely with no authentication.
