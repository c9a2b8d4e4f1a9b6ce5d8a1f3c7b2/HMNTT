### Title
Global Rate-Limit Bucket Exhaustion Allows Single Unprivileged Client to DoS All Other Users

### Summary
`ThrottleManagerImpl` enforces request-rate throttling using a single application-wide `rateLimitBucket` with no per-IP or per-client partitioning. Any unauthenticated client can saturate the entire `requestsPerSecond` budget (default 500 req/s) in a single burst, causing every subsequent request from every other client to receive HTTP 429 for the remainder of that second. This repeats every second, producing a sustained denial-of-service against all legitimate users.

### Finding Description
**Code path:**

`ThrottleConfiguration.rateLimitBucket()` (lines 24–32) creates one singleton `Bucket` for the entire JVM process:

```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    final var limit = Bandwidth.builder()
            .capacity(rateLimit)
            .refillGreedy(rateLimit, Duration.ofSeconds(1))
            .build();
    return Bucket.builder().addLimit(limit).build(); // no SynchronizationStrategy, no per-IP key
}
```

`ThrottleManagerImpl.throttle()` (lines 37–42) deducts one token from this shared bucket for every incoming request, regardless of source IP or identity:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

**Root cause:** The bucket is a single global counter. There is no per-client, per-IP, or per-session bucket. The failed assumption is that 500 req/s is too high for any single client to monopolize; in practice it is trivially achievable with a single HTTP client using concurrent connections.

**Why existing checks fail:**
- The `gasLimitBucket` has `SynchronizationStrategy.SYNCHRONIZED` but the `rateLimitBucket` does not — this is irrelevant to the DoS but shows the two buckets were treated differently.
- The `RequestProperties` per-filter `THROTTLE` sub-bucket is only evaluated *after* the global `rateLimitBucket` check passes, so it provides no protection against global exhaustion.
- There is no authentication, API key, or IP-allowlist check anywhere in the throttle path.

### Impact Explanation
A single attacker with no credentials can send 500 concurrent HTTP requests per second to any `/api/v1/contracts/call` endpoint. Once the global bucket is empty, every other user's request throws `ThrottleException` → HTTP 429 for the rest of that second. Because the bucket refills greedily every second, the attacker simply repeats the burst each second, maintaining a continuous blackout for all legitimate users. In the gossip-prevention context, this means no legitimate client can submit contract-call transactions through the mirror node's web3 API, effectively silencing all legitimate traffic while the attacker's own requests succeed.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and no special knowledge beyond the public API endpoint. A single `ab -n 500 -c 500` or equivalent tool is sufficient. The attack is repeatable every second indefinitely. Any motivated actor — competitor, vandal, or automated bot — can sustain it with minimal resources.

### Recommendation
Replace the single global bucket with a per-IP (or per-API-key) bucket map, e.g. using a `ConcurrentHashMap<String, Bucket>` keyed on the client IP extracted from `HttpServletRequest.getRemoteAddr()` (or `X-Forwarded-For` behind a trusted proxy). Alternatively, enforce per-IP limits at the ingress layer (nginx `limit_req`, cloud WAF, or an API gateway) before requests reach the application. A two-tier approach — per-IP limit + global limit — is the most robust solution.
