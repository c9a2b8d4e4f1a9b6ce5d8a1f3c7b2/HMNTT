### Title
Unauthenticated Resource Exhaustion via Triple Sequential DB Query Pattern in `/tokens/:tokenId/nfts/:serialNumber/transactions`

### Summary
The `getNftTransferHistoryRequest()` handler in `rest/tokens.js` issues three sequential database queries per request — two unconditional and one conditional at line 1047 via `NftService.getNft()`. Because the endpoint is publicly accessible with no rate limiting, an unprivileged attacker can flood it with concurrent requests to valid `tokenId`/`serialNumber` pairs, exhausting the connection pool and degrading service for all users.

### Finding Description
The code path in `rest/tokens.js` `getNftTransferHistoryRequest()` executes:

1. **Query 1** (line 1022): `pool.queryQuietly(timestampQuery, timestampParams)` — fetches consensus timestamps.
2. **Query 2** (line 1032): `pool.queryQuietly(query, params)` — fetches transaction details.
3. **Query 3** (line 1047): `NftService.getNft(tokenId, serialNumber)` — fetches the NFT row to check `createdTimestamp`. [1](#0-0) [2](#0-1) 

`NftService.getNft()` issues its own `pool.queryQuietly` call: [3](#0-2) 

Because all three calls use `await` sequentially, a single request holds a pool connection for the full round-trip duration of all three queries before releasing it. Under high concurrency this means the pool is saturated for 3× longer per request compared to a single-query endpoint. The pool is bounded by `config.db.pool.maxConnections`: [4](#0-3) 

The route is registered with no authentication or rate-limiting middleware: [5](#0-4) 

No per-IP rate limiting, concurrency cap, or request queue is applied before the handler runs. The `authHandler` middleware present in `server.js` is for authentication, not throttling. [6](#0-5) 

The third query fires whenever `response.transactions.length > 0` (line 1045), which is the common case for any NFT that has ever been transferred. An attacker can reliably trigger all three queries by targeting any valid, active NFT.

**Root cause / failed assumption:** The design assumes the third query is a cheap, infrequent check. In reality it is triggered on every request with results, adds a full DB round-trip to request latency, and there is no mechanism to prevent an unauthenticated user from issuing arbitrarily many concurrent requests.

### Impact Explanation
Under concurrent load, the connection pool is exhausted faster because each request occupies a connection slot for the combined duration of three sequential queries rather than one. Once the pool is full, new requests queue or time out (`connectionTimeoutMillis`), causing HTTP 503 / gateway-timeout responses for all users of the mirror node REST API — not just the attacker's requests. This is a non-network-based DoS affecting availability of the public API.

### Likelihood Explanation
No special privileges, tokens, or accounts are required. Any attacker who can enumerate a handful of valid `tokenId`/`serialNumber` pairs (trivially obtained from the same public API, e.g., `/tokens/:tokenId/nfts`) can script concurrent HTTP GET requests. The attack is stateless, cheap to sustain, and requires no authentication. Standard HTTP flood tooling (e.g., `ab`, `wrk`, `hey`) is sufficient.

### Recommendation
1. **Eliminate the third query where possible.** The `createdTimestamp` check at line 1047 can often be inferred from data already present in the transaction history rows (e.g., a `TOKENMINT` transaction implies creation). Avoid the extra round-trip when the information is derivable from existing results.
2. **Add per-IP rate limiting** on this endpoint (and ideally all public endpoints) using middleware such as `express-rate-limit` before the handler is invoked.
3. **Add a concurrency/queue limit** at the application level so that the number of in-flight DB-bound requests is bounded independently of the connection pool size.
4. **Increase observability:** expose a metric for connection pool wait time so pool exhaustion is detectable before it causes user-visible failures.

### Proof of Concept
```bash
# 1. Discover a valid tokenId and serialNumber (public endpoint, no auth)
curl "https://<mirror-node>/api/v1/tokens?type=NON_FUNGIBLE_UNIQUE&limit=1"
# -> extract tokenId

curl "https://<mirror-node>/api/v1/tokens/<tokenId>/nfts?limit=1"
# -> extract serialNumber

# 2. Flood the three-query endpoint concurrently
hey -n 10000 -c 200 \
  "https://<mirror-node>/api/v1/tokens/<tokenId>/nfts/<serialNumber>/transactions"

# Expected result: connection pool exhausted; other API endpoints begin returning
# 503 / connection timeout errors for legitimate users.
```
