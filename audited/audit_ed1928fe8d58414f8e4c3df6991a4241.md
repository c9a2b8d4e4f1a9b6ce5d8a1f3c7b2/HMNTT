### Title
Unauthenticated, Unbounded gRPC Subscription Flood Causes Heap Exhaustion via Per-Subscription Context Object Accumulation

### Summary
The gRPC `subscribeTopic` endpoint accepts connections from any unprivileged client with no authentication and no application-level rate limiting. The only concurrency guard — `maxConcurrentCallsPerConnection = 5` — is scoped per-connection, not globally, so an attacker opening thousands of TCP connections can drive tens of thousands of simultaneous subscriptions. Each subscription unconditionally allocates a `TopicContext` (and, when `listenerType=POLL`, a `PollingContext`), both holding `AtomicLong`, `AtomicReference`, and associated Reactor Flux chains, leading to heap exhaustion and JVM crash.

### Finding Description

**Entry point — no auth, no throttle:**
`ConsensusController.subscribeTopic()` (line 43) calls `topicMessageService::subscribeTopic` directly with no credential check and no rate-limiting guard. The only registered `ServerInterceptor` is `GrpcInterceptor` (test-scoped), which only sets an `EndpointContext` thread-local and immediately forwards the call. [1](#0-0) 

**Per-connection limit is not a global limit:**
`GrpcConfiguration` configures `maxConcurrentCallsPerConnection = 5` on the Netty server. No `maxConnections` or global concurrent-call ceiling is set anywhere in the gRPC module. An attacker opening `N` TCP connections can sustain `5N` simultaneous streaming RPCs. [2](#0-1) [3](#0-2) 

**Unconditional `TopicContext` allocation per subscription:**
`TopicMessageServiceImpl.subscribeTopic()` (line 61) allocates a `TopicContext` for every accepted subscription. Each `TopicContext` holds an `AtomicLong count`, an `AtomicReference<TopicMessage> last`, a `Stopwatch`, and a full Reactor Flux pipeline including a `safetyCheck` Mono scheduled on `Schedulers.boundedElastic()`. [4](#0-3) [5](#0-4) 

**Additional `PollingContext` allocation when `listenerType=POLL`:**
`CompositeTopicListener.listen()` delegates to `PollingTopicListener` when the configured type is `POLL`. `PollingTopicListener.listen()` (line 35) allocates a new `PollingContext` per call, holding its own `AtomicLong count` and `AtomicReference<TopicMessage> last`, plus a `RepeatSpec` Flux that polls the database on `Schedulers.boundedElastic()` every 500 ms indefinitely. [6](#0-5) [7](#0-6) 

**`topicExists` check does not prevent the attack:**
`topicExists()` only rejects subscriptions for non-existent topic IDs. An attacker using any single valid topic ID bypasses this check entirely. [8](#0-7) 

**No throttle in the gRPC module:**
The `ThrottleConfiguration` / `ThrottleManagerImpl` rate-limiting infrastructure exists only in the `web3` module and is never wired into the gRPC service path. [9](#0-8) 

### Impact Explanation
With 10,000 attacker-controlled TCP connections × 5 calls each = 50,000 concurrent subscriptions. Each subscription holds at minimum one `TopicContext` (~200–400 bytes of object headers + fields + Reactor chain references). In POLL mode each also holds a `PollingContext` plus a live `boundedElastic` scheduler task issuing DB queries every 500 ms. At scale this exhausts JVM heap (OOM / crash) and saturates the bounded-elastic thread pool, halting all message delivery. This maps directly to the stated scope: shutdown of ≥30% of network processing nodes without brute-force, since a single mirror node crash disrupts all clients relying on it for topic subscription.

### Likelihood Explanation
Preconditions are minimal: the attacker needs network access to port 5600 (publicly exposed per the Helm ingress config) and one valid topic ID (trivially discoverable from the public ledger). No credentials, no API key, no prior account. The attack is fully scriptable with any gRPC client library (e.g., `grpcurl`, a custom Go/Python client). It is repeatable and can be sustained indefinitely since connections are never rejected at the application layer. [10](#0-9) 

### Recommendation
1. **Global connection/subscription cap**: Set `maxConnections` on `NettyServerBuilder` and add a global `AtomicInteger` subscription counter in `TopicMessageServiceImpl` that rejects new subscriptions above a configurable ceiling.
2. **Per-IP rate limiting**: Add a gRPC `ServerInterceptor` that tracks subscription open-rate per remote IP using a token-bucket (analogous to `ThrottleManagerImpl` in the web3 module) and returns `RESOURCE_EXHAUSTED` when exceeded.
3. **Authentication**: Require a bearer token or mTLS for `subscribeTopic` so anonymous flooding is impossible.
4. **Infrastructure backstop**: The GCP backend policy `maxRatePerEndpoint: 250` is optional and only active when the GCP gateway is enabled; it should be enforced unconditionally and complemented by application-level controls. [11](#0-10) 

### Proof of Concept
```bash
# 1. Identify any valid topic ID from the public mirror REST API
TOPIC_ID=0.0.1234

# 2. Open 2000 parallel gRPC streaming connections (each allowed 5 calls = 10,000 subscriptions)
for i in $(seq 1 2000); do
  grpcurl -plaintext \
    -d "{\"topicID\":{\"topicNum\":1234}, \"consensusStartTime\":{\"seconds\":0}}" \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# 3. Observe JVM heap via actuator
curl http://<mirror-node-host>:8081/actuator/metrics/jvm.memory.used

# Expected result: heap climbs monotonically; JVM throws OutOfMemoryError
# or boundedElastic thread pool saturates, halting all topic delivery.
```

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }

        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)

                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }
