### Title
Integer Overflow in `poll()` Removes DB Query Limit, Enabling Unlimited Result Set DoS

### Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int) (filter.getLimit() - context.getTotal().get())` performs an unchecked narrowing cast from `long` to `int`. When a user supplies `limit = Long.MAX_VALUE` (permitted by the `@Min(0)`-only constraint), the cast produces `-1`. This negative value causes `hasLimit()` to return `false` on the internally-built `newFilter`, so `TopicMessageRepositoryCustomImpl.findByFilter` calls `typedQuery` with **no `setMaxResults()` call at all**, returning every matching row in the database. The same bug exists identically in `PollingTopicListener.poll()`.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.java`, `poll()`, lines 68–75:
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())  // line 69 — overflow here
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());       // line 71 — min(-1, N) = -1
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build(); // line 75 — limit=-1L
```

**Root cause:** No bounds check before the narrowing cast. `Long.MAX_VALUE` (0x7FFFFFFFFFFFFFFF) truncated to 32 bits yields `0xFFFFFFFF` = `-1`.

**Why existing checks fail:**
- `TopicMessageFilter.limit` carries only `@Min(0)` — no `@Max`. `Long.MAX_VALUE` passes validation.
- `hasLimit()` (`limit > 0`) is `true` for the original filter, so the cast branch is taken.
- The internally-built `newFilter` has `limit = -1L`. `hasLimit()` on that returns `false` (`-1 > 0` is false).
- In `TopicMessageRepositoryCustomImpl.findByFilter` (lines 51–53):
  ```java
  if (filter.hasLimit()) {          // false for newFilter
      typedQuery.setMaxResults(...); // never called
  }
  ```
  The JPA query executes with **no row limit**, streaming the entire `topic_message` table for that topic.

**`isComplete()` loop behavior (lines 121–128):** Uses the *original* filter's `limit = Long.MAX_VALUE`. `Long.MAX_VALUE == total.get()` is never true in practice. In throttled mode the loop continues until the unlimited result page happens to be smaller than `maxPageSize`; in unthrottled mode it runs for `maxPolls` (default 12) iterations, each fetching all rows.

**Same bug in `PollingTopicListener.java`, lines 54–59** — identical pattern.

### Impact Explanation

Each malicious subscription triggers one or more full-table scans of `topic_message` with no `LIMIT` clause. A busy topic (e.g., the Hedera fee-schedule topic 0.0.101 or any high-volume topic) can have millions of rows. The JPA layer materialises the entire result into a `List` in heap memory (`getResultList()`, line 60 of `TopicMessageRepositoryCustomImpl`). Multiple concurrent malicious subscriptions can exhaust JVM heap, cause GC pressure, and crash or severely degrade the mirror-node gRPC service — a denial-of-service against all subscribers. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` wrapper (line 58 of `PollingTopicMessageRetriever`) means any transient OOM-induced error is automatically retried, amplifying the load.

### Likelihood Explanation

The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable. The `ConsensusTopicQuery` proto field `limit` is `uint64`, so any gRPC client can set it to `2^63-1`. No rate-limiting or per-connection limit cap is visible in the retriever or service layer. A single attacker with a gRPC client (e.g., `grpcurl`) can trigger this repeatedly. The attack is trivially scriptable and repeatable.

### Recommendation

1. **Clamp before casting:** Replace the unchecked cast with a safe narrowing:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
2. **Add an upper bound to `TopicMessageFilter.limit`:** Add `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate cap (e.g., `@Max(1_000_000)`) so the field is validated at ingress.
3. **Guard `setMaxResults` against non-positive values** in `TopicMessageRepositoryCustomImpl.findByFilter` as a defence-in-depth check.
4. Apply the same fix to `PollingTopicListener.poll()` (line 54–55).

### Proof of Concept

```bash
# Using grpcurl against a running mirror-node gRPC endpoint
grpcurl -plaintext \
  -d '{
    "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 101},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected result:** The mirror-node gRPC process issues a full-table-scan SQL query against `topic_message` with no `LIMIT` clause, materialises all rows into heap, and — under repeated concurrent connections — exhausts JVM memory, causing service degradation or crash.

**Verification:** Enable SQL logging (`spring.jpa.show-sql=true`); the emitted query will have no `LIMIT` clause when `limit = Long.MAX_VALUE` is supplied.
