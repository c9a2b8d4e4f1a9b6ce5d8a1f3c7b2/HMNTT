### Title
Connection Pool Exhaustion via Unbounded Topic Endpoint Flooding During Network Partition

### Summary
`TopicServiceImpl.findById()` issues a blocking JDBC call through a shared HikariCP pool with no bulkhead, no per-endpoint concurrency limit, and no JDBC socket timeout. During a network partition between the rest-java service and PostgreSQL, TCP-level hangs cause pool connections to be held indefinitely. An unprivileged attacker flooding `GET /api/v1/topics/{id}` during such a partition can exhaust all pool connections, blocking every other endpoint (accounts, network, etc.) that shares the same pool.

### Finding Description

**Exact code path:**

`TopicServiceImpl.findById()` at line 19–21:
```java
public Topic findById(EntityId id) {
    return topicRepository.findById(id.getId())
        .orElseThrow(() -> new EntityNotFoundException("Topic not found"));
}
``` [1](#0-0) 

`TopicRepository` is a plain Spring Data `CrudRepository` with no query timeout override: [2](#0-1) 

The shared `HikariDataSource` is constructed in `CommonConfiguration` from `spring.datasource.hikari` properties with no per-service pool isolation: [3](#0-2) 

**Root cause — three compounding gaps:**

1. **No JDBC socket timeout at runtime.** The `socketTimeout` in `DatabaseStartupProperties` is used only by `DatabaseWaiter` during startup, not for runtime queries. No `socketTimeout` JDBC URL parameter is set for the operational pool. During a TCP-level network partition, the PostgreSQL JDBC driver socket hangs indefinitely; the PostgreSQL-level `statementTimeout` (10 s default) never fires because the TCP layer never delivers the command. [4](#0-3) 

2. **No bulkhead / semaphore on the topic endpoint.** There is no Resilience4j bulkhead, no `@Bulkhead`, and no `inFlightReq` Traefik middleware for rest-java (compare: graphql and rosetta both configure `inFlightReq`). Any number of concurrent topic requests can each hold a pool connection. [5](#0-4) 

3. **No per-IP rate limiting on the topic route.** `TopicController.getTopic()` has no throttle guard; the Traefik middleware for rest-java omits `rateLimit` and `inFlightReq`. [6](#0-5) 

**Exploit flow:**

During a network partition, each `topicRepository.findById()` call acquires a HikariCP connection and blocks on a hung TCP socket. With the default pool size of 10 connections, 10 concurrent requests saturate the pool. All subsequent requests — including account lookups, network queries, and fee estimations — queue for `connectionTimeout` (HikariCP default 30 s) before failing with `SQLTransientConnectionException`. The Traefik circuit breaker only trips after 10 % network error ratio or 25 % 5xx ratio, meaning the pool is already exhausted before it activates. [7](#0-6) 

### Impact Explanation

All endpoints served by the rest-java process share one HikariCP pool. Pool exhaustion caused by hung topic connections produces a complete denial-of-service for every other API path (`/api/v1/accounts/`, `/api/v1/network/`, etc.) for the duration of the partition plus the HikariCP `connectionTimeout` window per queued request. The Prometheus alert for high DB connections fires only after 5 minutes at >75 % utilization, providing no real-time protection. [8](#0-7) 

### Likelihood Explanation

The attacker requires no credentials — `GET /api/v1/topics/{id}` is a public, unauthenticated endpoint. The network partition is an environmental precondition (infrastructure failure, cloud AZ split, or deliberate network-layer disruption). Once a partition exists, the attacker needs only to sustain ≥10 concurrent HTTP requests to the topic endpoint, which is trivially achievable with any HTTP load tool. The attack is repeatable for the entire duration of the partition and requires no knowledge of internal topology.

### Recommendation

1. **Add a JDBC socket timeout** to the HikariCP data source configuration (e.g., `spring.datasource.hikari.data-source-properties.socketTimeout=15` for PostgreSQL JDBC) so that hung TCP connections are released within a bounded time regardless of partition state.
2. **Add a Resilience4j `@Bulkhead`** (semaphore type) to `TopicServiceImpl.findById()` to cap the number of concurrent DB calls from this endpoint, preventing it from monopolising the shared pool.
3. **Add `inFlightReq` Traefik middleware** to the rest-java chart (as already done for graphql and rosetta) to limit concurrent requests per source IP at the ingress layer.
4. Consider a **dedicated, smaller connection pool** for read-only topic lookups to provide hard isolation from other endpoint pools.

### Proof of Concept

```
# 1. Induce or wait for a network partition between rest-java and PostgreSQL
#    (e.g., iptables -I OUTPUT -p tcp --dport 5432 -j DROP on the DB host)

# 2. Send 15 concurrent requests to the topic endpoint (exceeds default pool of 10)
seq 1 15 | xargs -P15 -I{} \
  curl -s -o /dev/null -w "%{http_code}\n" \
  "https://<mirror-node>/api/v1/topics/0.0.1234"

# 3. Immediately probe an unrelated endpoint
curl -v "https://<mirror-node>/api/v1/network/supply"
# Expected result: 500 or connection hang (SQLTransientConnectionException:
# HikariPool-1 - Connection is not available, request timed out after 30000ms)
```
