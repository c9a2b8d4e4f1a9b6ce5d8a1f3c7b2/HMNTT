### Title
Unbounded `inner_transactions` Fetch in `getTransactionsByIdOrHash` Enables Amplified DB Load Griefing

### Summary
`getTransactionsByIdOrHash()` in `rest/transactions.js` unconditionally issues a second `getTransactionsByTransactionIdsSql()` database query whose WHERE clause grows linearly with the number of entries in `inner_transactions`, and then appends all returned rows via `rows.push(...innerTransactionRows)` with no cap. Because the REST API for this endpoint accepts no `limit` parameter and has no rate limiter, an attacker who submits one atomic-batch transaction can repeatedly query the mirror node for free, forcing two unbounded DB round-trips and full response serialization on every request.

### Finding Description
**Exact code path** — `rest/transactions.js`, `getTransactionsByIdOrHash()`, lines 944–958:

```js
const innerTransactions = rows          // line 944
  .map((row) => row.inner_transactions) // line 945
  .filter((innerTransactions) => innerTransactions) // line 946
  .flat();                              // line 947

if (!isTransactionHash && innerTransactions.length > 0) {  // line 949 — only hash guard, no size cap
  const {query: innerTransactionQuery, params} = getTransactionsByTransactionIdsSql(
    innerTransactions,   // passed in full, no slice/limit
    filters,
    Transaction.VALID_START_NS
  );
  const {rows: innerTransactionRows} = await pool.queryQuietly(innerTransactionQuery, params);
  rows.push(...innerTransactionRows);  // line 957 — unbounded append
}
```

**Root cause** — `innerTransactions` is the raw `bigint[]` array from the DB column (stored as `[payer1, ts1, payer2, ts2, …]`). It is passed directly to `getTransactionsByTransactionIdsSql()`, which builds one `(payer_account_id = $N and valid_start_ns = $M)` OR-condition per pair. There is no check on `innerTransactions.length` before the second query, no `LIMIT` clause on the inner-transaction query, and no cap on `innerTransactionRows` before the spread into `rows`.

**Why existing checks fail:**
- `acceptedSingleTransactionParameters` (line 1014) only allows `nonce` and `scheduled`; no `limit` parameter is accepted for this endpoint, so the `response.limit.max` (default 100) never applies here.
- The guard `!isTransactionHash` (line 949) only skips the second query for hash-based lookups; transaction-ID lookups always proceed.
- There is no rate limiter on the REST API (confirmed: zero matches for `rateLimit`/`throttle` in `rest/**/*.js`).
- The DB `statementTimeout` (configured in `rest/dbpool.js`) provides a last-resort backstop but does not prevent repeated requests from exhausting connection pool slots.

### Impact Explanation
Each request to `GET /api/v1/transactions/{batchTxId}` for a batch transaction with N inner transactions causes:
1. One primary DB query.
2. One secondary DB query with N OR-conditions and N correlated subqueries (crypto_transfer, token_transfer, assessed_custom_fee per row).
3. `formatTransactionRows()` serializing N+1 full transaction objects.

An attacker who submits a single atomic-batch transaction with the maximum number of inner transactions (bounded by the Hedera 6 KB transaction-size limit, practically ~30–60 inner transactions) can then issue unlimited GET requests at no further cost, each one forcing the mirror node to execute two heavy DB queries and serialize a large response. Sustained at even modest request rates this degrades DB connection pool availability and increases CPU/memory pressure on the REST service for all other users.

### Likelihood Explanation
The precondition (submitting one batch transaction) requires paying HBAR fees once, which is a trivial cost. After that, the endpoint is publicly accessible with no authentication, no per-IP rate limit, and no request cost. The attack is trivially repeatable from a single machine or a small botnet. Any existing batch transaction already on-chain (submitted by anyone) can be used as the trigger, meaning the attacker does not even need to pay the on-chain fee themselves.

### Recommendation
1. **Cap inner-transaction fetch size**: Before line 949, add a guard such as `if (innerTransactions.length > MAX_INNER_TRANSACTIONS)` and either reject or truncate.
2. **Add a `LIMIT` clause** to the inner-transaction SQL query so the DB never returns more than a configured maximum.
3. **Apply rate limiting** to the `/api/v1/transactions/:id` endpoint consistent with the web3 API's `requestsPerSecond` throttle.
4. **Enforce a response-row cap** after `rows.push(...innerTransactionRows)` analogous to the `response.limit.max` used by list endpoints.

### Proof of Concept
1. Submit an `ATOMIC_BATCH` transaction to the Hedera network containing the maximum allowed number of inner transactions (e.g., 50). Record the outer transaction ID, e.g., `0.0.1234-1700000000-000000000`.
2. Wait for the mirror node to ingest the transaction.
3. In a loop, issue:
   ```
   GET /api/v1/transactions/0.0.1234-1700000000-000000000
   ```
4. Observe via DB monitoring that each request triggers two queries — the primary lookup and a secondary query with 50 OR-conditions — and that the response contains 51 fully serialized transaction objects.
5. At ~100 req/s (easily achievable from one machine), the mirror node DB receives 200 queries/s for this single endpoint, each with 50 OR-conditions and 3 correlated subqueries per row, amplifying load by a factor of ~150× compared to a normal single-transaction lookup. const getTransactionsByTransactionIdsSql = (transactionKeys, filters, timestampField) => {
  let payerAccountParams = [];
  let params = [];

  let minTimestamp = constants.MAX_LONG;
  let maxTimestamp = -1;
  let idConditions = [];

  if (transactionKeys.length % 2 !== 0) {
    throw new InvalidArgumentError('transaction keys must be in sequenced pairs of [payer,timestamp]');
  }

  for (let index = 0; index < transactionKeys.length; index += 2) {
    let paramIndex = params.length;
    const payer = transactionKeys[index];
    const timestamp = BigInt(transactionKeys[index + 1]);

    if (payer) {
      payerAccountParams.push(`$${++paramIndex}`);
      idConditions.push(`(${Transaction.PAYER_ACCOUNT_ID} = $${paramIndex} and 
                               ${timestampField} = $${++paramIndex})`);
      params.push(payer);
    } else if (timestampField === Transaction.VALID_START_NS) {
      throw new InvalidArgumentError('payer is required when timestamp is valid_start_ns');
    } else {
      idConditions.push(`${timestampField} = $${++paramIndex}`);
    }
    params.push(timestamp);

    if (timestamp < minTimestamp) {
      minTimestamp = timestamp;
    }
    if (timestamp > maxTimestamp) {
      maxTimestamp = timestamp;
    }
  }

  const maxConsensusTimestamp =
    timestampField === Transaction.CONSENSUS_TIMESTAMP
      ? transactionKeys[transactionKeys.length - 1]
      : maxTimestamp + maxTransactionConsensusTimestampRangeNs;
  const commonConditions = [
    `${Transaction.CONSENSUS_TIMESTAMP} >= $${params.length + 1}`,
    `${Transaction.CONSENSUS_TIMESTAMP} <= $${params.length + 2}`,
  ];
  params.lowerConsensusTimestampIndex = params.length;
  params.push(
    timestampField === Transaction.VALID_START_NS
      ? minTimestamp - config.query.maxValidStartTimestampDriftNs
      : minTimestamp
  );
  params.upperConsensusTimestampIndex = params.length;
  params.push(maxConsensusTimestamp);

  const {conditions: filterConditions, scheduled} = getTransactionByIdQueryParamConditions(filters, params);
  const mainConditions = [...commonConditions, `(${idConditions.join(' or ')})`, ...filterConditions].join(' and ');
  const subqueryConditions = payerAccountParams.length
    ? [...commonConditions, `${Transaction.PAYER_ACCOUNT_ID} in (${payerAccountParams.join(',')})`].join(' and ')
    : commonConditions.join(' and ');

  return {
    query: getTransactionQuery(mainConditions, subqueryConditions),
    params,
    scheduled,
  };
};
