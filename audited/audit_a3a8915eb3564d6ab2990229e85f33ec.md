### Title
Gas Token Drain via REJECT Filter: `gasLimitBucket` Consumed Without Restoration on `ThrottleException`

### Summary
In `ThrottleManagerImpl.throttle()`, `gasLimitBucket` tokens are consumed unconditionally before the request filter loop is evaluated. When a configured `REJECT` filter matches, a `ThrottleException` is thrown, but `ContractController.call()` only restores gas tokens on `InvalidParametersException` — never on `ThrottleException`. An unprivileged attacker who knows (or can enumerate) the REJECT filter patterns can repeatedly send matching requests with maximum gas, draining the shared `gasLimitBucket` and causing legitimate users to receive "Gas per second rate limit exceeded" errors with no EVM execution ever occurring.

### Finding Description
**Exact code path:**

`ThrottleManagerImpl.throttle()` — `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–49:
```java
public void throttle(ContractCallRequest request) {
    if (!rateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // line 40: tokens consumed here
        throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
    }

    for (var requestFilter : throttleProperties.getRequest()) {   // line 44
        if (requestFilter.test(request)) {
            action(requestFilter, request);   // line 46: may throw ThrottleException for REJECT
        }
    }
}
```

`action()` — lines 66–76:
```java
case REJECT -> throw new ThrottleException("Invalid request");  // line 69
```

`ContractController.call()` — lines 38–51:
```java
try {
    throttleManager.throttle(request);   // ThrottleException escapes here
    ...
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());  // ONLY restores for InvalidParametersException
    throw e;
}
// ThrottleException is never caught → restore() is never called
```

**Root cause:** The `gasLimitBucket.tryConsume()` call at line 40 is unconditional and precedes filter evaluation. The `restore()` mechanism exists and is used correctly for `InvalidParametersException`, but the developer failed to apply the same pattern to `ThrottleException` thrown by REJECT filters. The failed assumption is that a rejected request should still "pay" gas — but since no EVM execution occurs, the gas cost is artificial and permanently deducted from the shared bucket.

**Exploit flow:**
1. Attacker identifies (or guesses) a configured REJECT filter pattern (e.g., a specific `data` prefix, `from` address, or `to` address).
2. Attacker sends POST `/api/v1/contracts/call` with `gas` set to the maximum allowed (default `maxGasLimit` = 15,000,000) and request fields crafted to match the REJECT filter.
3. `gasLimitBucket.tryConsume(scaleGas(15_000_000))` = `tryConsume(1500)` succeeds and permanently deducts 1500 scaled tokens.
4. REJECT filter matches → `ThrottleException("Invalid request")` thrown.
5. `ContractController.call()` does not catch `ThrottleException` → `restore()` is never called.
6. Attacker repeats up to the `requestsPerSecond` limit (default 500 req/s).
7. At 500 req/s × 1500 tokens = 750,000 tokens/s, the entire default `gasPerSecond` bucket (`scaleGas(7_500_000_000)` = 750,000) is drained every second.
8. All legitimate requests receive `GAS_PER_SECOND_LIMIT_EXCEEDED`.

### Impact Explanation
The `gasLimitBucket` is a shared, global resource protecting EVM execution capacity. Draining it completely blocks all legitimate `/api/v1/contracts/call` requests with "Gas per second rate limit exceeded." No EVM execution is required by the attacker — each rejected request costs the attacker only an HTTP round-trip. The bucket refills at 750,000 tokens/second, but the attacker can sustain the drain indefinitely at the rate limit. This constitutes a complete, sustained DoS of the contract call endpoint for all users, with zero computational cost to the attacker.

### Likelihood Explanation
The precondition is that at least one `REJECT` filter is configured by the operator. This is an explicitly documented, production-intended feature (`hiero.mirror.web3.throttle.request[].action = REJECT`). Any operator using REJECT filters to block abusive patterns inadvertently enables this attack. The attacker needs no credentials, no special network access, and no knowledge of internal state beyond the filter pattern — which may be discoverable by probing (sending varied requests and observing whether the response is `ThrottleException("Invalid request")` vs. a normal error). The attack is fully repeatable and automatable with a simple HTTP client.

### Recommendation
Restore `gasLimitBucket` tokens whenever a `ThrottleException` is thrown due to a REJECT (or THROTTLE) filter action, since no EVM execution occurred. The fix should mirror the existing `InvalidParametersException` handling:

```java
// ContractController.call()
try {
    throttleManager.throttle(request);
    validateContractMaxGasLimit(request);
    ...
} catch (InvalidParametersException | ThrottleException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
```

Alternatively, restructure `throttle()` to evaluate filters **before** consuming `gasLimitBucket` tokens, so gas is only consumed when the request is confirmed to proceed. A third option is to have `ThrottleManagerImpl` itself call `restore()` internally before throwing from a REJECT action.

### Proof of Concept
**Precondition:** Operator has configured a REJECT filter matching `data` field containing `"deadbeef"`:
```yaml
hiero.mirror.web3.throttle.request:
  - action: REJECT
    filters:
      - field: DATA
        type: CONTAINS
        expression: "deadbeef"
```

**Steps:**
```bash
# Drain the gasLimitBucket (run in a loop up to 500 req/s)
for i in $(seq 1 500); do
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"data":"0xdeadbeef","gas":15000000,"to":"0x0000000000000000000000000000000000000001"}' &
done
wait

# Now send a legitimate request (no deadbeef in data)
curl -X POST http://mirror-node/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{"data":"0x12345678","gas":21000,"to":"0x0000000000000000000000000000000000000001"}'
# Expected result: {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

Each attacker request consumes `scaleGas(15_000_000)` = 1500 tokens from `gasLimitBucket` with no restoration. After 500 such requests, the 750,000-token bucket is exhausted, and the legitimate request is denied.
