### Title
Unbounded Alias Length Allows Oversized `bytea` Parameter Injection into `entityFromAliasQuery`, Enabling DB Resource Exhaustion

### Summary
The `accountAliasRegex` in `rest/accountAlias.js` imposes no upper bound on the alias portion (`[A-Z2-7]+`), allowing an unauthenticated attacker to supply an arbitrarily long base32 string that passes `AccountAlias.isValid()`. The string is decoded into a proportionally large `Uint8Array` by `base32.decode()` and passed directly as the `$1` parameter to `entityFromAliasQuery`, forcing PostgreSQL to allocate memory and process an oversized `bytea` equality comparison on every such request. Under sustained attack this degrades shared database resources, including throughput for transaction gossip queries.

### Finding Description

**Exact code path:**

`rest/accountAlias.js` line 10 — the validation regex:
```js
const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;
```
The `[A-Z2-7]+` quantifier has **no upper bound**. A string of 8 000 valid base32 characters passes this regex.

`rest/accountAlias.js` line 41-44 — `isValid()` only tests the regex, no length check:
```js
static isValid(accountAlias, noShardRealm = false) {
  const regex = noShardRealm ? noShardRealmAccountAliasRegex : accountAliasRegex;
  return typeof accountAlias === 'string' && regex.test(accountAlias);
}
```

`rest/accountAlias.js` line 24 — constructor decodes without length guard:
```js
this.alias = base32.decode(base32Alias);   // Uint8Array ∝ input length
```

`rest/base32.js` line 13 — decode is unbounded:
```js
const decode = (str) => str && base32.parse(str, decodeOpts);
```

`rest/service/entityService.js` lines 17-20 — the decoded bytes become `$1`:
```sql
select id from entity
where coalesce(deleted, false) <> true
  and alias = $1
```

`rest/service/entityService.js` lines 125-126 — call path from `getEncodedId()`:
```js
} else if (AccountAlias.isValid(entityIdString)) {
  return await this.getAccountIdFromAlias(AccountAlias.fromString(entityIdString), requireResult);
```

**Root cause / failed assumption:** The code assumes the regex is sufficient to bound input size. It is not — `[A-Z2-7]+` matches 1 to ∞ characters. The Java counterpart (`EntityIdAliasParameter.java` line 13) correctly constrains the alias to `[A-Z2-7]{40,70}`, but the Node.js REST API has no equivalent bound.

**Why existing checks fail:**
- `AccountAlias.isValid()` — only a regex test, no length cap.
- `AccountAlias.fromString()` — calls `isValid()` then immediately constructs the object and decodes.
- No Express middleware enforces an alias-specific length limit; the server only applies generic body-size limits, which do not apply to URL path/query parameters.
- The OpenAPI schema pattern (`(?:[A-Z2-7]{8})*(?:[A-Z2-7]{2}|...)`) does constrain length, but it is documentation only — not enforced in runtime code.

### Impact Explanation
Each malicious request causes: (1) Node.js allocates a `Uint8Array` of up to ~5 000 bytes (8 000-char URL → ~5 000 decoded bytes) per request; (2) the pg driver serialises and transmits this oversized `bytea` to PostgreSQL over the wire; (3) PostgreSQL allocates working memory and performs a `bytea` equality comparison against every row (or index entry) in the `entity` table. Under concurrent flood of such requests the shared database connection pool and working memory are exhausted, degrading query throughput for all consumers of the database — including internal transaction gossip queries that rely on the same PostgreSQL instance.

### Likelihood Explanation
No authentication or rate-limiting specific to alias length is present. Any external user can issue HTTP GET requests to `/api/v1/accounts/<alias>` or use `account.id=<alias>` query parameters. The attack is trivially scriptable: a single `curl` loop with an 8 000-character alias string is sufficient to trigger the condition. It is repeatable at high frequency from a single host or distributed across multiple IPs.

### Recommendation
1. **Add an explicit length cap in `accountAliasRegex`** — change `[A-Z2-7]+` to `[A-Z2-7]{2,128}` (or the exact maximum for a valid protobuf-prefixed public key, which is ≤ 70 characters as the Java API enforces).
2. **Mirror the Java constraint** — align with `EntityIdAliasParameter.ALIAS_REGEX` which uses `[A-Z2-7]{40,70}`.
3. **Add a pre-decode length guard** in `AccountAlias.isValid()` or the constructor before calling `base32.decode()`.
4. **Apply per-parameter length validation** in the Express middleware layer for all alias-bearing parameters.

### Proof of Concept
```bash
# Generate an 8000-character valid base32 alias (all A's)
ALIAS=$(python3 -c "print('A' * 8000)")

# Send to the Node.js REST API — no authentication required
curl -v "http://<mirror-node-host>:5551/api/v1/accounts/${ALIAS}"

# Expected: request passes AccountAlias.isValid(), base32.decode() produces
# a ~5000-byte Uint8Array, entityFromAliasQuery executes with a 5000-byte
# bytea $1 parameter against the entity table.
# Repeat at high concurrency to exhaust DB working memory / connection pool:
for i in $(seq 1 500); do
  curl -s "http://<mirror-node-host>:5551/api/v1/accounts/${ALIAS}" &
done
wait
```
