### Title
Cache Stampede (Thundering Herd) in `findStorageBatch()` Due to Mismatched TTLs Between `contractStateCache` and `slotsPerContract` Cache

### Summary
The `contractStateCache` uses `expireAfterWrite=2s` while the `slotsPerContract` per-contract slot-key cache uses `expireAfterAccess=5m`. When `contractStateCache` entries expire simultaneously, all concurrent threads that miss the cache call `findStorageBatch()` with no locking or stampede protection, each independently issuing the same large batch DB query. The specific claim of 25,000 slots per contract is incorrect — the per-contract slot cache is capped at 1,500 entries — but the thundering herd is real and exploitable within that bound.

### Finding Description

**Cache configuration** (`web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, lines 28 and 39):
- `contractState`: `expireAfterWrite=2s, maximumSize=25000` — global cache of `(contractId, slotKey) → slotValue`
- `slotsPerContract`: `expireAfterAccess=5m, maximumSize=1500` — per-contract cache of slot keys

**Code path** (`ContractStateServiceImpl.java`):

1. `findStorage()` (lines 58–70): checks `contractStateCache`; on miss, calls `findStorageBatch()`.
2. `findStorageBatch()` (lines 85–122):
   - Line 91: reads **all** slot keys from the per-contract `slotsPerContract` cache (up to 1,500 keys).
   - Line 103: issues a single `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (...)` with all 1,500 keys — **no locking, no "loading" sentinel, no stampede guard**.
   - Lines 106–114: writes results back to `contractStateCache`.

**Root cause**: There is no cache-stampede protection (e.g., Caffeine's `AsyncLoadingCache`, a per-key lock, or a "loading" flag). When `contractStateCache` entries expire after 2 seconds, every concurrent thread that calls `findStorage()` for any slot of that contract independently enters `findStorageBatch()` and issues the same large batch query against the database.

**Why the 25,000-entry claim is wrong but the vulnerability is real**: The global `contractStateCache` holds up to 25,000 entries across all contracts, but `findStorageBatch()` reads slot keys from the per-contract `slotsPerContract` cache (max 1,500 per contract). A single thundering herd event therefore issues up to 1,500-slot batch queries, not 25,000-slot queries. However, with up to 500 concurrent requests (the default RPS throttle), all hitting the same contract after the 2-second expiry, the DB receives up to 500 simultaneous `IN (1500 slots)` queries.

### Impact Explanation

Each thundering herd event can produce hundreds of simultaneous large batch queries against the database. The DB connection pool (typically 10–50 connections) becomes saturated, causing:
- Request queuing and latency spikes across all users of the mirror node web3 service.
- Potential DB thread exhaustion, leading to query timeouts (3,000 ms default) and HTTP 500 errors for all concurrent callers.
- The effect repeats every 2 seconds as long as the attacker sustains traffic, making it a continuous, repeatable DoS.

### Likelihood Explanation

An unprivileged user needs only:
1. A contract with ≥1,500 storage slots (common for large DeFi/NFT contracts on Hedera mainnet).
2. To issue `eth_call` or `eth_estimateGas` requests that read those slots, seeding the `slotsPerContract` cache (5-minute TTL).
3. To wait 2 seconds for `contractStateCache` to expire.
4. To send concurrent requests (up to 500 RPS allowed) immediately after expiry.

No authentication, no privileged access, and no on-chain transactions are required. The attack is repeatable every 2 seconds indefinitely.

### Recommendation

1. **Use Caffeine's `AsyncLoadingCache` or a per-key loading lock** in `findStorageBatch()` so that only one thread issues the batch DB query per contract while others wait for the result.
2. **Switch `contractStateCache` to `expireAfterAccess`** instead of `expireAfterWrite` to prevent simultaneous mass expiry.
3. **Add a "loading" sentinel** (e.g., a `ConcurrentHashMap<EntityId, CompletableFuture>`) so that concurrent threads for the same contract share a single in-flight DB query.
4. **Reduce the TTL asymmetry**: align `contractStateCache` and `slotsPerContract` TTLs, or use a refresh-ahead strategy.

### Proof of Concept

```
1. Identify a contract on the mirror node with ≥1,500 storage slots (e.g., a large ERC-20/NFT contract).

2. Seed the slotsPerContract cache:
   for slot_key in slot_keys[0..1499]:
       POST /api/v1/contracts/call  { "to": "<contract>", "data": "SLOAD(<slot_key>)" }
   # slotsPerContract cache now holds 1,500 keys; contractStateCache holds 1,500 values.

3. Wait 2 seconds (contractStateCache expires; slotsPerContract cache still live for ~5 min).

4. Trigger the thundering herd — send 500 concurrent requests immediately:
   parallel for i in 0..499:
       POST /api/v1/contracts/call  { "to": "<contract>", "data": "SLOAD(<any_of_the_1500_slots>)" }

5. Observe:
   - All 500 threads miss contractStateCache.
   - All 500 threads enter findStorageBatch() simultaneously (ContractStateServiceImpl.java:85).
   - All 500 threads read the same 1,500 slot keys (line 91) and issue the same
     SELECT ... WHERE slot IN (<1500 keys>) query (line 103).
   - DB connection pool saturates; latency spikes; HTTP 500/504 errors appear for all users.
   - Repeat every 2 seconds to sustain the DoS.

    public Optional<byte[]> findStorage(final EntityId contractId, final byte[] key) {
        if (!cacheProperties.isEnableBatchContractSlotCaching()) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }

        final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);

        if (cachedValue != null && cachedValue != EMPTY_VALUE) {
            return Optional.of(cachedValue);
        }

        return findStorageBatch(contractId, key);
    }

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
