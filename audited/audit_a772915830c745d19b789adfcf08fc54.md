### Title
Unauthenticated SLOAD-Heavy Historical Contract Calls Exhaust Shared Database Connection Pool via Uncached Per-Slot Queries

### Summary
The `call()` endpoint in `ContractController` accepts unauthenticated requests with a historical block parameter. Each EVM SLOAD opcode executed against a historical block triggers a separate, completely uncached SQL query against the `contract_state_change` table via `findStorageByBlockTimestamp`. The gas-based throttle is per-node in-memory (not distributed), allowing up to 500 max-gas requests per second per node, each capable of generating ~7,142 uncached DB queries, overwhelming the shared PostgreSQL connection pool across all mirror nodes.

### Finding Description

**Exact code path:**

`ContractController.call()` (line 38) accepts any unauthenticated POST to `/api/v1/contracts/call`. No authentication is required. [1](#0-0) 

The only guards are `throttleManager.throttle(request)` and `validateContractMaxGasLimit`. The throttle is a JVM-local Bucket4j instance with `SynchronizationStrategy.SYNCHRONIZED` — not distributed across nodes. [2](#0-1) 

Default limits: `gasPerSecond = 7,500,000,000` scaled by `/10,000 = 750,000` tokens/second; max gas per request = `15,000,000` → 1,500 tokens consumed → **500 max-gas requests/second per node** before throttling. [3](#0-2) 

When the request includes a historical block, `ContractStorageReadableKVState.readFromDataSource` routes every SLOAD to `contractStateService.findStorageByBlockTimestamp(...)`. [4](#0-3) 

`ContractStateServiceImpl.findStorageByBlockTimestamp` has **no caching whatsoever** — it directly calls the repository on every invocation. [5](#0-4) 

The repository method `findStorageByBlockTimestamp` issues a full `ORDER BY consensus_timestamp DESC LIMIT 1` query against `contract_state_change` for each slot, with no `@Cacheable` annotation. [6](#0-5) 

Contrast with `findStorage` (latest block path), which carries `@Cacheable`: [7](#0-6) 

**Root cause and failed assumption:** The gas throttle assumes gas consumption is a reliable proxy for resource cost. For historical SLOAD operations, a cold SLOAD costs 2,100 gas but triggers one uncached `ORDER BY … DESC LIMIT 1` query on a large, append-only `contract_state_change` table — a cost orders of magnitude higher than the gas model accounts for. The throttle is also per-node (in-memory), so N mirror nodes each independently allow the full rate against the single shared PostgreSQL instance.

**Amplification math:**
- Max gas per request: 15,000,000 / 2,100 gas (cold SLOAD) ≈ **7,142 DB queries per call**
- Throttle allows: **500 calls/second per node**
- DB queries per second per node: **~3.57 million**
- With multiple mirror nodes sharing one PostgreSQL: multiply by node count

The project itself documents this attack surface in its own k6 load test tooling, confirming the `SlotContractCaller.heavyReadBothHalves()` function produces 3,000 SLOADs per call: [8](#0-7) 

### Impact Explanation
A single attacker with no credentials can sustain millions of uncached `contract_state_change` queries per second against the shared PostgreSQL database. This exhausts the DB connection pool, causing query timeouts and failures across all mirror nodes that share the database. Because the throttle is per-node and not distributed, adding more mirror nodes worsens the problem — each additional node independently allows the full attack rate. This directly maps to degradation of ≥30% of network processing nodes without any brute-force requirement.

### Likelihood Explanation
No authentication, no API key, no per-IP rate limit, and no per-call DB query budget are required. Any external user can craft a single `curl` command targeting a known SLOAD-heavy contract (or any contract with many storage slots) with a historical block number. The attack is trivially automatable, repeatable, and requires no special knowledge beyond the public API specification. The project's own performance test suite (`contractCallEstimateReadStorage.js`) demonstrates the exact attack pattern.

### Recommendation
1. **Cache historical slot reads**: Apply a short-lived (e.g., 30-second) cache keyed on `(contractId, slot, blockTimestamp)` in `findStorageByBlockTimestamp` to prevent repeated identical queries.
2. **Distribute the throttle**: Replace the JVM-local Bucket4j bucket with a distributed implementation (e.g., Bucket4j + Redis/Hazelcast) so the gas budget is shared across all nodes against the same DB.
3. **Add a per-call DB query budget**: Track the number of storage reads within a single EVM execution and abort with an error if it exceeds a configurable threshold (e.g., 1,000 slots per call).
4. **Add per-IP rate limiting**: Enforce a per-source-IP request limit at the ingress/filter layer to prevent a single client from consuming the full node-level quota.

### Proof of Concept

**Preconditions:**
- A contract exists on the network with ≥1,000 populated storage slots (e.g., the `SlotContractCaller` contract referenced in the k6 tests, or any contract with a large mapping).
- The attacker knows the contract address and a function selector that reads many slots.

**Steps:**

```bash
# Replace CONTRACT_ADDR with a SLOAD-heavy contract address
# Replace CALLDATA with the function selector (e.g., heavyReadBothHalves = 0x523adad6)
# Replace HISTORICAL_BLOCK with any past block number (forces uncached findStorageByBlockTimestamp)

while true; do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "to": "CONTRACT_ADDR",
      "data": "CALLDATA",
      "gas": 15000000,
      "block": "HISTORICAL_BLOCK"
    }' &
done
```

**Result:** Each concurrent request triggers ~7,142 uncached SQL queries against `contract_state_change`. At the allowed 500 req/s per node, the PostgreSQL connection pool is saturated within seconds, causing all mirror node API responses to time out or return 500 errors, degrading ≥30% of nodes sharing the database.

    ContractCallResponse call(@RequestBody @Valid ContractCallRequest request, HttpServletResponse response) {
        try {
            throttleManager.throttle(request);

            final var params = constructServiceParameters(request);
            final var result = contractExecutionService.processCall(params);
            return new ContractCallResponse(result);
        } catch (InvalidParametersException e) {
            // The validation failed, but no processing occurred so restore the consumed tokens.
            throttleManager.restore(request.getGas());
            throw e;
        }
    }
