### Title
Cache Stampede in `findStorage()` Allows Unprivileged DoS via Concurrent DB Query Flood

### Summary
`ContractStateServiceImpl.findStorage()` performs a non-atomic cache check followed by an unconditional database call with no synchronization. When many concurrent requests arrive for the same uncached `(contractId, key)` pair, all threads simultaneously miss `contractStateCache` and all independently invoke `findStorageBatch()`, issuing concurrent `findStorageBatch` SQL queries to the database. Because `contractStateCache` expires every 2 seconds, this stampede is repeatable and can be sustained indefinitely by any unprivileged caller.

### Finding Description

**Exact code path:**

`ContractStateServiceImpl.java`, `findStorage()`, lines 58–70:

```java
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class); // non-atomic read
if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
return findStorageBatch(contractId, key); // all N threads reach here simultaneously
```

`contractStateCache.get(key, byte[].class)` is Spring's `Cache.get(Object key, Class<T> type)` — a plain read that returns `null` on a miss. It does **not** use Caffeine's atomic `get(key, loader)` / `computeIfAbsent` variant, which would serialize concurrent loaders for the same key. There is no lock, no `synchronized` block, and no "single-flight" guard between the cache check and the DB call.

**Root cause:** The cache check and the subsequent DB call are not an atomic unit. N concurrent threads can all observe a cache miss and all proceed to `findStorageBatch()`.

**Inside `findStorageBatch()` (lines 85–122):**

```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

Each of the N threads issues this query independently. The `cachedSlots` list includes **all** slot keys previously seen for the contract (up to `maximumSize=3000` in `contractSlotsCache`), so each query can be a large `IN (...)` clause. Results are written back to `contractStateCache` only after the query completes — too late to prevent the other N−1 threads from also querying.

**Cache expiry amplification:** `contractStateCache` is configured `expireAfterWrite=2s`. Every 2 seconds the entire cache expires, reopening the stampede window. An attacker can sustain the attack indefinitely by sending bursts every 2 seconds.

**Why existing checks fail:**

The `ThrottleManagerImpl` applies a global `requestsPerSecond` limit (default 500 req/s) using `rateLimitBucket.tryConsume(1)`. This is a token-bucket that allows a burst of up to 500 simultaneous requests to pass in a single second. All 500 can arrive concurrently, all pass the rate limiter, all call `findStorage()`, all miss the cache, and all issue DB queries simultaneously. The rate limiter is applied at the HTTP controller layer (`ContractController.call()`), not at the `findStorage()` level, and it does not prevent concurrent cache misses for the same key.

### Impact Explanation

Up to 500 concurrent DB queries (the default rate limit burst) can be issued simultaneously per stampede event, each potentially carrying a large `IN` clause of up to 3,000 slot keys. This can:
- Exhaust the database connection pool, causing all subsequent queries (across all services sharing the DB) to queue or fail.
- Exhaust the web3 node's HTTP thread pool, causing request timeouts.
- Repeat every 2 seconds (the `contractStateCache` TTL), sustaining the attack indefinitely.

Because the database is typically shared infrastructure, a stampede on one node degrades all nodes that share it, meeting the ≥30% network processing node impact threshold.

### Likelihood Explanation

No authentication or special privilege is required. Any caller with access to the JSON-RPC endpoint (e.g., `eth_call` or `eth_getStorageAt`) can trigger `findStorage()`. The attacker needs only to:
1. Identify a contract address (publicly visible on-chain).
2. Send 500 concurrent HTTP requests within the same second targeting the same storage slot.
3. Repeat every 2 seconds.

This is trivially scriptable with standard HTTP load tools (`wrk`, `ab`, `hey`). No wallet, no gas, no on-chain transaction required.

### Recommendation

Replace the non-atomic cache-check + DB-call pattern with Caffeine's atomic loading API. Specifically, use `Cache.get(key, valueLoader)` (Spring's `Callable`-based variant) or directly use the native Caffeine `Cache.get(key, mappingFunction)`, which guarantees that for any given key only one thread executes the loader while all other threads block and receive the same result:

```java
// Replace:
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);
if (cachedValue != null && cachedValue != EMPTY_VALUE) { return Optional.of(cachedValue); }
return findStorageBatch(contractId, key);

// With (atomic, stampede-safe):
final var result = contractStateCache.get(generateCacheKey(contractId, key),
    () -> findStorageBatch(contractId, key).orElse(EMPTY_VALUE));
return (result == EMPTY_VALUE) ? Optional.empty() : Optional.ofNullable(result);
```

Additionally, consider reducing the `contractState` cache TTL or adding a per-key lock (e.g., Guava's `Striped<Lock>`) if the atomic loader approach is not feasible.

### Proof of Concept

**Preconditions:**
- Mirror node web3 service is running and accessible.
- `enableBatchContractSlotCaching=true` (the default).
- A known contract address `CONTRACT` with at least one storage slot `SLOT`.

**Steps:**

1. Wait for `contractStateCache` to be cold (or wait 2 seconds after any prior access).
2. Send 500 concurrent `eth_getStorageAt` requests for `(CONTRACT, SLOT)`:
   ```bash
   seq 500 | xargs -P500 -I{} curl -s -X POST http://mirror-node:8545 \
     -H 'Content-Type: application/json' \
     -d '{"jsonrpc":"2.0","method":"eth_getStorageAt","params":["CONTRACT","SLOT","latest"],"id":{}}'
   ```
3. Observe on the database server: 500 simultaneous queries against `contract_state` with large `IN` clauses.
4. Observe thread pool saturation and elevated response latency / timeouts on the mirror node.
5. Repeat step 2 every 2 seconds to sustain the attack.

**Expected result:** Database connection pool exhaustion, query queue buildup, and degraded or failed responses from the mirror node and any co-located services sharing the database.
