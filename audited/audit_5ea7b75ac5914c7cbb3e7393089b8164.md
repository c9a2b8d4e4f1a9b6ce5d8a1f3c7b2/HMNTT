### Title
Unbounded `spenderIdInFilters` IN Clause Replicated Across All Three UNION ALL Subqueries in `NftService.getQuery()`

### Summary
In `rest/service/nftService.js`, the `getQuery()` method passes the full `spenderIdInFilters` array to every subquery generated from `lower`, `inner`, and `upper`. When all three bound arrays are non-empty (triggering the three-subquery UNION ALL path), the `spender IN (...)` clause is independently appended to each branch, causing the database to evaluate the full IN clause three times per request. An unprivileged user can craft a single HTTP request that forces this worst-case path, multiplying per-request database CPU cost by a factor of three compared to a baseline query without spender filters.

### Finding Description

**Exact code path:**

In `rest/service/nftService.js`, `getQuery()` (lines 93–105) iterates over `[lower, inner, upper]`, filtering out empty arrays, and calls `getSubQuery()` for each non-empty one, passing the same `spenderIdInFilters` reference each time:

```js
// rest/service/nftService.js lines 93-105
const subQueries = [lower, inner, upper]
  .filter((filters) => filters.length !== 0)
  .map((filters) =>
    this.getSubQuery(
      filters,
      params,
      accountIdCondition,
      limitClause,
      orderClause,
      spenderIdInFilters,   // ← same array passed to every subquery
      spenderIdFilters
    )
  );
```

Inside `getSubQuery()` (lines 66–73), each call independently appends all `spenderIdInFilters` values to the shared `params` array and emits a fresh `spender IN (...)` condition:

```js
// rest/service/nftService.js lines 66-73
if (!isEmpty(spenderIdInFilters)) {
  const paramsForCondition = spenderIdInFilters.map((filter) => {
    params.push(filter.value);          // ← appended again for every subquery
    return `$${params.length}`;
  });
  conditions.push(`${Nft.SPENDER} in (${paramsForCondition})`);
}
```

When all three of `lower`, `inner`, and `upper` are non-empty, the result is a three-branch UNION ALL where each branch carries its own `spender IN ($N, $N+1, ...)` clause. The database planner must evaluate the IN predicate independently for each branch.

**Root cause:** `spenderIdInFilters` is not deduplicated or hoisted outside the UNION ALL; it is blindly replicated into every subquery branch.

**How the three-subquery path is triggered:** `lower`, `inner`, and `upper` are populated by `extractNftMultiUnionQuery()` in `rest/controllers/accountController.js` (lines 68–70) via `getLowerFilters`, `getInnerFilters`, and `getUpperFilters` on the `bounds` object. A closed range on `token.id` combined with serial-number bounds on both ends produces all three non-empty arrays. For example:

```
token.id=gte:1000&serialnumber=gt:1   → lower (token eq lower-bound + serial gt)
token.id=gt:1000&token.id=lt:2000     → inner (token open range)
token.id=lte:2000&serialnumber=lt:5   → upper (token eq upper-bound + serial lt)
```

**Existing checks and why they are insufficient:**

`validateFilters()` in `accountController.js` (lines 18–32) only rejects multiple `lt`/`lte` or `gt`/`gte` operators on `spenderIdFilters` (non-eq spender filters) and rejects `ne`. It places **no limit** on the count of `spenderIdInFilters` (eq-operator spender filters) beyond the global `maxRepeatedQueryParameters` cap enforced by `buildFilters()` in `utils.js` (lines 1241–1247). That cap is a per-parameter-key ceiling (e.g., 25), not a per-subquery or per-request cost ceiling. The multiplication factor of 3 (one IN clause per UNION ALL branch) is entirely unmitigated.

### Impact Explanation
Each request with the maximum number of `spender.id=eq:X` values and a three-branch UNION ALL forces the database to evaluate the IN predicate three times instead of once. Because the IN clause is the most selective filter in each branch, the planner cannot hoist it; it must re-evaluate it per branch. This directly multiplies index-scan and filter CPU cost by ~3× compared to a single-subquery baseline. Sustained at moderate request rates, this can push database CPU utilization well above the 30% threshold relative to normal traffic, constituting a denial-of-service amplification vector against the database tier without any brute-force volume requirement.

### Likelihood Explanation
No authentication or authorization is required to call `/api/v1/accounts/{id}/nfts`. The parameter combination needed (a closed `token.id` range, serial-number bounds, and multiple `spender.id=eq:X` values) is entirely within the documented API surface. The attack is repeatable, deterministic, and requires only a single HTTP client. Any external user who reads the OpenAPI spec can construct the exploit request.

### Recommendation
1. **Hoist the spender IN clause outside the UNION ALL** by wrapping the entire UNION ALL in a CTE or subquery and applying the spender filter once at the outer level.
2. **Alternatively**, detect when `spenderIdInFilters` is non-empty and the query would produce multiple subqueries, and refuse to combine them into a UNION ALL (fall back to a single-subquery form or reject the combination with a 400 error).
3. **Lower `maxRepeatedQueryParameters`** for the `spender.id` key specifically, or add a dedicated cap on `spenderIdInFilters.length` inside `extractNftMultiUnionQuery()`.

### Proof of Concept

```
GET /api/v1/accounts/0.0.1001/nfts?\
  token.id=gte:1000&serialnumber=gt:1\
  &token.id=gt:1000&token.id=lt:2000\
  &token.id=lte:2000&serialnumber=lt:5\
  &spender.id=eq:100&spender.id=eq:200&spender.id=eq:300\
  &spender.id=eq:400&spender.id=eq:500&...  (up to maxRepeatedQueryParameters)
```

This produces a SQL query of the form:

```sql
(SELECT ... FROM nft ... WHERE account_id=$1 AND token_id=$3 AND serial_number>$4
   AND spender IN ($5,$6,...,$N) ORDER BY ... LIMIT $2)
UNION ALL
(SELECT ... FROM nft ... WHERE account_id=$1 AND token_id>$N+1 AND token_id<$N+2
   AND spender IN ($N+3,...,$2N) ORDER BY ... LIMIT $2)
UNION ALL
(SELECT ... FROM nft ... WHERE account_id=$1 AND token_id=$2N+1 AND serial_number<$2N+2
   AND spender IN ($2N+3,...,$3N) ORDER BY ... LIMIT $2)
ORDER BY ... LIMIT $2
```
