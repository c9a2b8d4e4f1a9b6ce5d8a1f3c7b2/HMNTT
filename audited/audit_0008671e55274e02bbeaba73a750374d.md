### Title
Missing Range-Filter Combination Validation in `getAccountCryptoAllowances` Enables Maximum-Cost DB Queries

### Summary
`CryptoAllowanceController.extractCryptoAllowancesQuery()` accepts any combination of `spender.id` equality values (up to 100 via `maxRepeatedQueryParameters`) and range filters simultaneously, with no guard preventing their co-occurrence. Unlike the NFT endpoint's `AccountController.validateFilters()`, which explicitly rejects multiple range params for `spender.id`, the crypto allowance endpoint has no equivalent check. An unprivileged attacker can craft a single request that generates a SQL query with both a large `IN (...)` list and a range condition, at maximum `limit=100` and `order=asc`, maximizing per-request DB work.

### Finding Description

**Code path:**

`getAccountCryptoAllowances()` (line 76) calls `extractCryptoAllowancesQuery()` (line 79).

Inside `extractCryptoAllowancesQuery()` (lines 33–57), every `SPENDER_ID` filter is routed through `updateConditionsAndParamsWithInValues()`: [1](#0-0) 

- `eq` operators accumulate into `spenderInValues[]`
- Range operators (`gt`, `gte`, `lt`, `lte`) are pushed directly into `conditions[]` as individual SQL predicates

After the loop, `updateQueryFiltersWithInValues()` appends the `IN (...)` clause: [2](#0-1) 

`updateQueryFiltersWithInValues()` in `BaseController` builds the IN list from all accumulated eq values: [3](#0-2) 

There is **no check** preventing both from being present simultaneously. The resulting SQL for a crafted request becomes:

```sql
SELECT * FROM crypto_allowance
WHERE owner = $1
  AND spender >= $2          -- range filter
  AND spender IN ($3,...,$101) -- 99 eq values
  AND amount > 0
ORDER BY spender ASC
LIMIT $102
```

**Contrast with the NFT endpoint**, which explicitly rejects this combination: [4](#0-3) 

**Root cause:** `CryptoAllowanceController` only blocks the `ne` operator (line 36–38) but has no equivalent of `AccountController.validateFilters()` to reject multiple range params or the range+IN combination. [5](#0-4) 

### Impact Explanation

Each crafted request forces PostgreSQL to evaluate both a range scan and a 99-element IN membership test on the `crypto_allowance` table, ordered ascending (forward scan), returning up to 100 rows. The DB connection pool defaults to 10 connections and `statementTimeout` to 20 seconds: [6](#0-5) 

A small number of concurrent attackers (≤10) can saturate the connection pool with maximally expensive queries, degrading response times for all other users. No authentication or economic cost is required. The impact is service degradation (griefing), not data exfiltration.

### Likelihood Explanation

The endpoint is public and unauthenticated. The exploit requires only a valid account ID (trivially enumerable from the public ledger) and knowledge of the query parameter format. The attack is trivially scriptable and repeatable. `maxRepeatedQueryParameters=100` is the only bound, and it is exactly what the attacker uses to maximize the IN list size. [7](#0-6) 

### Recommendation

Add the same validation present in `AccountController.validateFilters()` to `extractCryptoAllowancesQuery()`:

1. Reject requests that supply more than one `lt`/`lte` or more than one `gt`/`gte` operator for `spender.id`.
2. Reject requests that combine any range operator with `eq` values for `spender.id` (or explicitly document and limit this combination).
3. Alternatively, refactor `CryptoAllowanceController` to use the `Bound`-based pattern used by `TokenAllowanceController`, which enforces mutual exclusivity of range and equality filters via `validateBoundsRange()`. [8](#0-7) 

### Proof of Concept

```
GET /api/v1/accounts/0.0.1000/allowances/crypto?\
  spender.id=gte:1\
  &spender.id=eq:100&spender.id=eq:200&...  (×99 eq values)\
  &limit=100\
  &order=asc
```

**Steps:**
1. Identify any valid account ID (e.g., `0.0.1000`) from the public mirror node.
2. Construct a request with 1 `spender.id=gte:1` range filter and 99 `spender.id=eq:N` equality filters (total = 100, within `maxRepeatedQueryParameters`).
3. Add `limit=100&order=asc`.
4. Send this request repeatedly in parallel (up to 10 concurrent, matching the DB pool size).
5. Observe increased DB query latency for all other users of the mirror node.

The generated SQL will contain both `spender >= $2` and `spender IN ($3,...,$101)` with `ORDER BY spender ASC LIMIT 100`, forcing the DB to evaluate the maximum-cost query path on every request. [9](#0-8) [10](#0-9)
