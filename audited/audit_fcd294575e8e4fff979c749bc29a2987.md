### Title
Goroutine Accumulation via DB Connection Pool Exhaustion in Rosetta API (No Default Rate Limiting)

### Summary
The Rosetta API's `ConnectToDb()` sets `MaxOpenConnections=100` with a 20-second `statementTimeout`, but applies no application-level concurrency limit. The Traefik middleware chain (which includes `inFlightReq: amount: 5` per IP and `rateLimit: average: 10`) is gated behind `global.middleware: false` by default and is therefore never applied. An unprivileged attacker can send more than 100 concurrent requests targeting slow DB-backed endpoints, exhausting the connection pool and causing an unbounded number of goroutines to accumulate while blocked waiting for a connection, driving memory well above 30% of baseline.

### Finding Description

**Code path:**

`rosetta/app/db/db.go`, `ConnectToDb()`, lines 31–33:
```go
sqlDb.SetMaxIdleConns(dbConfig.Pool.MaxIdleConnections)   // default 20
sqlDb.SetConnMaxLifetime(...)                              // default 30 min
sqlDb.SetMaxOpenConns(dbConfig.Pool.MaxOpenConnections)   // default 100
```
No `SetConnMaxIdleTime` and no connection-wait timeout are set. Go's `database/sql` queues callers indefinitely when `MaxOpenConns` is reached.

**Root cause — no application-level concurrency gate:**

`rosetta/main.go` lines 217–227 build the middleware chain as:
```
MetricsMiddleware → TracingMiddleware → CorsMiddleware → http.Server
```
None of these layers enforce a concurrency limit. Go's `net/http` spawns one goroutine per accepted connection with no cap.

**Why the Traefik protection is inert by default:**

`charts/hedera-mirror-rosetta/values.yaml` line 95:
```yaml
global:
  middleware: false   # ← gate is OFF
```
The middleware template (`charts/hedera-mirror-rosetta/templates/middleware.yaml` line 3) renders only when `global.middleware` is true:
```
{{ if and .Values.global.middleware .Values.middleware -}}
```
Because `global.middleware` defaults to `false`, the entire Traefik chain — including `inFlightReq: amount: 5` (per-IP) and `rateLimit: average: 10` — is never instantiated.

**Exploit flow:**

1. Attacker opens 150 concurrent HTTP connections to port 5700 and sends POST requests to a DB-heavy endpoint (e.g., `/block` or `/account/balance`).
2. The first 100 goroutines acquire DB connections and begin executing queries; each query runs for up to 20 seconds (`statementTimeout = 20`).
3. The remaining 50+ goroutines block inside `database/sql`'s internal `db.conn()` waiting for a free slot.
4. Go's `http.Server.WriteTimeout` (10 s) closes the TCP socket but does **not** cancel the handler goroutine; the goroutine continues blocking on the pool semaphore.
5. The attacker repeats the wave every 5–10 seconds. Goroutines accumulate faster than they drain (each lives up to 20 s), growing the goroutine count and heap.
6. Each blocked goroutine holds a minimum 2–8 KB stack that grows under pressure; 500 blocked goroutines ≈ 4–40 MB additional heap on top of baseline, easily exceeding 30%.

### Impact Explanation

Memory and goroutine count grow proportionally to the sustained request rate. At ~150 req/s the pool saturates within one second; goroutines accumulate at the rate of incoming requests minus the drain rate (100 connections / 20 s = 5/s drain). Within 30–60 seconds the process can hold thousands of blocked goroutines, causing OOM or severe GC pressure that degrades all users of the node. Because the Rosetta API is the exchange-integration surface, degradation here directly affects exchange deposit/withdrawal pipelines.

### Likelihood Explanation

No authentication is required; the Rosetta API is publicly reachable on port 5700. A single attacker machine with a modest HTTP load tool (e.g., `wrk`, `hey`, `ab`) can sustain 150+ concurrent connections. The attack is repeatable, requires no special knowledge beyond knowing the endpoint paths (documented in the Helm chart ingress rules), and leaves no persistent state to clean up between waves.

### Recommendation

1. **Add a DB connection-wait timeout** in `ConnectToDb()`:
   ```go
   // rosetta/app/db/db.go
   sqlDb.SetConnMaxIdleTime(time.Duration(dbConfig.Pool.MaxIdleTime) * time.Second)
   ```
   And pass a context with a deadline into every repository query so waiting goroutines are cancelled promptly.

2. **Add an application-level semaphore** in `main.go` before the router, e.g. using `golang.org/x/net/netutil.LimitListener` or a counting semaphore middleware, to cap concurrent in-flight handlers independently of Traefik.

3. **Change the default** of `global.middleware` to `true` in `charts/hedera-mirror-rosetta/values.yaml` so the `inFlightReq` and `rateLimit` Traefik middlewares are active out of the box.

4. **Reduce `statementTimeout`** from 20 s to a value consistent with expected query latency (e.g., 5 s) to shorten the window during which connections are held.

### Proof of Concept

```bash
# Prerequisites: Rosetta running on localhost:5700 with default config,
# global.middleware=false (the default), wrk installed.

# Step 1 – establish baseline memory
curl -s http://localhost:5700/metrics | grep go_memstats_heap_inuse_bytes

# Step 2 – flood with concurrent slow requests (block endpoint triggers DB scan)
wrk -t 10 -c 200 -d 30s \
  -s post.lua \          # lua script POSTing {"network_identifier":...,"block_identifier":{"index":1}}
  http://localhost:5700/block

# Step 3 – observe goroutine and memory growth
curl -s http://localhost:5700/metrics | grep -E 'go_goroutines|go_memstats_heap_inuse_bytes'
# Expected: go_goroutines >> baseline; heap_inuse_bytes > 1.3 × baseline
```
