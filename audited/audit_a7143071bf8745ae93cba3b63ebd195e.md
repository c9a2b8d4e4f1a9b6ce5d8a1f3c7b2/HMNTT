### Title
Unauthenticated Full-Range Network Nodes Query Enables DB Connection Pool Exhaustion via High-Frequency Requests

### Summary
The `/api/v1/network/nodes` endpoint is publicly accessible with no authentication or rate limiting. When called with no `node.id` filter, `NetworkServiceImpl.getNetworkNodes()` passes `lowerBound=0` and `upperBound=Long.MAX_VALUE` with an empty node ID array to a complex multi-CTE SQL query containing a correlated subquery per result row. An attacker flooding this endpoint with concurrent requests can hold database connections open for the duration of each expensive query, exhausting the connection pool and denying service to all other clients.

### Finding Description

**Code path:**

In `NetworkServiceImpl.java` lines 107–108, when no `node.id` parameters are supplied, the bounds default to the widest possible range:

```java
long lowerBound = 0L;
long upperBound = Long.MAX_VALUE;
``` [1](#0-0) 

At lines 131–132, with no EQ node IDs, `nodeIdArray` is set to `EMPTY_NODE_ID_ARRAY`:

```java
} else {
    nodeIdArray = EMPTY_NODE_ID_ARRAY;
}
``` [2](#0-1) 

This is passed directly to `networkNodeRepository.findNetworkNodes()` at lines 135–136: [3](#0-2) 

The SQL query in `NetworkNodeRepository.java` is a multi-CTE query. The `latest_node_stake` CTE executes `select max(consensus_timestamp) from node_stake` on every invocation: [4](#0-3) 

Additionally, a correlated subquery runs once per returned row to aggregate service endpoints: [5](#0-4) 

The WHERE clause with `lowerBound=0` / `upperBound=Long.MAX_VALUE` and an empty `nodeIds` array (the `coalesce(array_length(:nodeIds, 1), 0) = 0` branch) performs a full scan of `address_book_entry` joined to the latest address book: [6](#0-5) 

**Why existing checks fail:**

`getEffectiveLimit()` caps returned rows at 25, but this only limits the output — it does not reduce the cost of the CTEs, the `max(consensus_timestamp)` aggregate scan, or the correlated subquery execution. Each request holds a database connection open for the full query duration. [7](#0-6) 

The controller endpoint `GET /api/v1/network/nodes` has no authentication, no rate-limiting annotation, and no throttling mechanism: [8](#0-7) 

No `RateLimitFilter`, no per-IP throttle, and no connection-pool configuration was found anywhere in the repository.

### Impact Explanation
Each concurrent request to this endpoint acquires a database connection and holds it for the duration of the multi-CTE query. With no rate limiting, an attacker sending hundreds of concurrent requests exhausts the JDBC connection pool. Once the pool is exhausted, all other API clients — including legitimate users querying account data, transaction history, or token info — receive connection timeout errors. This constitutes a full denial-of-service against the mirror node's REST API layer, effectively partitioning all clients from node data. Severity: **High** (unauthenticated, no preconditions, complete service disruption).

### Likelihood Explanation
The attack requires zero privileges, zero authentication, and only a standard HTTP client. The endpoint is publicly documented. A single attacker with modest bandwidth can sustain hundreds of concurrent requests using tools like `ab`, `wrk`, or `hey`. The attack is trivially repeatable and automatable. The absence of any rate-limiting or IP-based throttling in the codebase means there is no server-side defense to overcome.

### Recommendation
1. **Rate limiting:** Apply per-IP rate limiting at the reverse proxy (e.g., nginx `limit_req`) or in the application using a filter (e.g., Bucket4j or Spring's `HandlerInterceptor`) specifically for `/api/v1/network/nodes`.
2. **Query timeout:** Set a statement timeout on the database connection used by this query (e.g., `SET LOCAL statement_timeout = '5s'`) to bound the maximum connection hold time.
3. **Connection pool isolation:** Use a separate, size-limited connection pool for this endpoint so exhaustion cannot affect other API paths.
4. **CTE optimization:** Cache the `latest_node_stake` and `latest_address_book` results (e.g., via a short-lived application-level cache) to avoid repeated full-table scans on every request.

### Proof of Concept

```bash
# Send 200 concurrent requests with no filters (full-range, max limit)
# Requires: wrk or ab installed, mirror node running on localhost:8080

wrk -t 20 -c 200 -d 30s \
  "http://localhost:8080/api/v1/network/nodes?limit=25"

# Observe: after connection pool exhaustion, all other API endpoints
# begin returning 500/503 errors:
curl "http://localhost:8080/api/v1/accounts/0.0.1"
# Expected: {"_status":{"messages":[{"message":"Unable to acquire JDBC Connection"}]}}
```

Preconditions: None. No account, no API key, no special headers required.
Trigger: Sustained concurrent GET requests to `/api/v1/network/nodes` with no parameters.
Result: Database connection pool exhausted; all mirror node REST API clients denied service.
