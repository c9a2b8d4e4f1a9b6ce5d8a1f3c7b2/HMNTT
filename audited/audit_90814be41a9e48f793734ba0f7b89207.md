### Title
IP Spoofing via Unvalidated `X-Forwarded-For` Enables Log Poisoning in `handleError()`

### Summary
`rest/server.js` sets `app.set('trust proxy', true)` unconditionally, causing Express to resolve `req.ip` from the attacker-controlled `X-Forwarded-For` header without any validation. `handleError()` logs `req.ip` directly on every error path, so any unprivileged external user can inject an arbitrary IP address into all authorization-failure and error log entries, permanently destroying the forensic trail for unauthorized transaction attempts.

### Finding Description
**Root cause — unconditional proxy trust:**
`rest/server.js` line 58 sets `app.set('trust proxy', true)`. With this setting, Express resolves `req.ip` to the **leftmost** value in the `X-Forwarded-For` request header, regardless of whether the request actually arrived through a trusted proxy. Any direct client connection can freely set this header.

**Sink — unvalidated `req.ip` in `handleError()`:**
`rest/middleware/httpErrorHandler.js` lines 33–34 and 39 log `req.ip` verbatim on every error:

```js
logger.warn(
  `${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ...`
);
// and
logger.error(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ...`, ...);
```

There is no call to `req.socket.remoteAddress`, no allowlist of trusted upstream proxies, and no sanitization of the header value before it reaches the log sink.

**Exploit flow:**
1. Attacker sends any HTTP request with header `X-Forwarded-For: 8.8.8.8` (or any chosen decoy IP).
2. Express resolves `req.ip` → `"8.8.8.8"`.
3. The request triggers any error (e.g., `InvalidArgumentError`, `HttpError 401`, `NotFoundError`).
4. `handleError()` fires and writes `"8.8.8.8 POST /api/v1/transactions ..."` to the warn/error log.
5. The real attacker IP (`req.socket.remoteAddress`) is never recorded anywhere in the error log.

**Why existing checks fail:**
`trust proxy: true` is the broadest possible setting — it trusts every hop unconditionally. Express documentation explicitly warns that this setting should only be used when the server sits behind a known, controlled reverse proxy, and that using `true` makes `req.ip` fully attacker-controlled from the internet.

### Impact Explanation
Every security-relevant error log entry — including authorization failures, bad-request probes, and internal server errors triggered during transaction processing — can be permanently attributed to a false IP. Incident responders and automated IP-based blocking/alerting systems will act on fabricated data. The real attacker remains untraced across all repeated unauthorized transaction attempts. Severity is **Medium-High**: it does not bypass authentication itself, but it completely undermines the auditability and non-repudiation of the authorization layer.

### Likelihood Explanation
Exploitation requires zero privileges, zero authentication, and zero special tooling — a single HTTP header field. It is repeatable on every request, works against every error code path in `handleError()`, and is invisible to the application (no error, no side-effect). Any attacker who can reach the REST API endpoint can exploit this.

### Recommendation
1. Replace `app.set('trust proxy', true)` in `rest/server.js` with an explicit allowlist of trusted upstream proxy IP ranges (e.g., `app.set('trust proxy', 'loopback, 10.0.0.0/8')`). This causes Express to only honor `X-Forwarded-For` when the immediately-connecting socket IP is in the trusted set.
2. In `handleError()`, log `req.socket.remoteAddress` (the real TCP peer) alongside or instead of `req.ip` for security-sensitive log lines, so the ground-truth IP is always preserved even when proxy headers are present.
3. Validate and sanitize `X-Forwarded-For` values against an IP regex before logging, to prevent log-injection of non-IP strings as well.

### Proof of Concept
```bash
# Trigger a 400 Bad Request with a spoofed source IP
curl -s \
  -H "X-Forwarded-For: 203.0.113.99" \
  "http://<mirror-node-rest>:5551/api/v1/transactions?timestamp=INVALID"

# Observe the warn log entry — it will show 203.0.113.99, not the real client IP:
# WARN: 203.0.113.99 GET /api/v1/transactions?timestamp=INVALID in 3 ms: 400 ...

# Repeat with any decoy IP to permanently poison the audit trail for all error events.
```
