### Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent `transaction.hash` Queries in `getContractLogs()`

### Summary
The `GET /api/v1/contracts/results/logs?transaction.hash=<hash>` endpoint, handled by `getContractLogs()` in `rest/controllers/contractController.js`, triggers two to three sequential database queries per request with no rate limiting or concurrency control on the REST API layer. An unprivileged attacker with knowledge of valid transaction hashes (publicly available on Hedera) can flood this endpoint with concurrent requests, exhausting the shared `pg` connection pool and causing a non-network denial of service for all REST API consumers.

### Finding Description

**Code path:**

`rest/routes/contractRoute.js` line 20 registers the endpoint with no middleware beyond basic Express routing: [1](#0-0) 

`getContractLogs()` in `rest/controllers/contractController.js` (lines 820–848) calls `extractContractLogsMultiUnionQuery(filters)` unconditionally: [2](#0-1) 

Inside `extractContractLogsMultiUnionQuery`, when `transactionHash !== undefined`, **DB Query 1** is issued synchronously via `await getTransactionHash(...)`: [3](#0-2) 

`getTransactionHash()` in `rest/transactionHash.js` calls `pool.queryQuietly()` directly — no caching, no deduplication: [4](#0-3) 

If the hash resolves (rows.length > 0), control returns to `getContractLogs()` which then issues **DB Query 2** via `ContractService.getContractLogs(query)` (line 840), which itself issues **DB Query 3** via `RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps)`: [5](#0-4) 

The connection pool is configured with a hard `max` ceiling: [6](#0-5) 

`queryQuietly` checks out a real connection from the pool for the duration of each query: [7](#0-6) 

**Root cause:** Each request holds pool connections across 2–3 sequential awaited DB calls. There is no per-IP rate limiting, no concurrency cap, and no request queuing with backpressure on the REST API layer. The throttle/rate-limit infrastructure that exists lives entirely in the `web3` Java module and does not apply here: [8](#0-7) 

### Impact Explanation
When the pool is saturated, `pg` queues pending `query()` calls. Once `connectionTimeoutMillis` elapses, those calls throw, causing HTTP 500 responses for all concurrent users of the REST API — not just the attacker's requests. This is a full service-level DoS affecting every endpoint that shares the same pool, including account, transaction, and block APIs.

### Likelihood Explanation
Hedera is a public network; valid transaction hashes are trivially enumerable from any block explorer or by replaying recent transactions. No authentication, API key, or privileged access is required. A single attacker with a modest number of concurrent HTTP connections (matching or exceeding `maxConnections`) can sustain the condition indefinitely. The attack is repeatable and scriptable with standard tools (`ab`, `wrk`, `curl --parallel`).

### Recommendation
1. **Add a per-IP rate limiter** (e.g., `express-rate-limit`) specifically on the `/results/logs` route before the handler is invoked.
2. **Add a global concurrency semaphore** for DB-bound handlers so that in-flight DB requests are capped below `maxConnections`.
3. **Cache `getTransactionHash` results** (short TTL, keyed by hash) to collapse repeated lookups for the same hash under flood conditions.
4. **Increase `connectionTimeoutMillis`** to fail fast and free queued waiters sooner, limiting cascade damage.

### Proof of Concept
```bash
# Step 1: collect a valid 32-byte ETH transaction hash from any Hedera block explorer
HASH="9db63968721fc76815d11b0b97734669bf88ae3810e07c199924553f0548eb99"

# Step 2: flood the endpoint with concurrent requests (adjust -c to exceed pool max)
ab -n 10000 -c 200 \
  "https://<mirror-node-host>/api/v1/contracts/results/logs?transaction.hash=${HASH}"

# Expected result: after pool exhaustion, legitimate requests begin returning HTTP 500
# or timing out; the condition persists as long as the flood continues.
```

    const rows = await super.getRows(sqlQuery, params);
    if (rows.length === 0) {
      return rows;

    const timestamps = [];
    // The timestamps are ordered, and may have duplicates, dedup them

    });
    const recordFileMap = await RecordFileService.getRecordFileBlockDetailsFromTimestampArray(timestamps);

    return rows.map((cr) => new ContractLog(cr, recordFileMap.get(cr.consensus_timestamp)));
