### Title
Unauthenticated Thread Pool Exhaustion via Zero-Prefix EVM Address Lookup During DB Network Partition

### Summary
`EntityServiceImpl.getByEvmAddressAndType()` routes zero-prefix EVM addresses directly to `entityRepository.findById()`, which is an uncached, unthrottled Spring Data JPA call with no application-level timeout. The GraphQL module has no rate limiting (unlike the web3 module) and no per-request timeout enforcement (unlike the web3 module's `HibernateConfiguration`). During a database network partition, each request blocks a Tomcat thread for the full Hikari connection-acquisition timeout, allowing an unprivileged attacker to exhaust the JVM thread pool.

### Finding Description

**Exact code path:**

`EntityServiceImpl.java` lines 37–38: [1](#0-0) 

When the first 4 bytes and next 8 bytes of the decoded EVM address are zero (the "long-zero" / zero-prefix format, e.g. `0x000000000000000000000000000000000000NNNN`), the code calls `entityRepository.findById(buffer.getLong())`. This is the standard `CrudRepository.findById()` inherited by: [2](#0-1) 

**Root cause — three missing defenses, all present in other modules but absent here:**

1. **No caching on `findById()`**: The graphql `EntityRepository` has no `@Cacheable` annotation on `findById()`. Compare the grpc module which caches it: [3](#0-2) 

2. **No application-level rate limiting**: The web3 module has a full `ThrottleManager` with bucket4j: [4](#0-3) 
No equivalent exists anywhere in the graphql module.

3. **No per-request timeout enforcement**: The web3 module enforces a `requestTimeout` via a Hibernate `StatementInspector` that throws `QueryTimeoutException` if elapsed time exceeds the limit. The graphql module has no such mechanism.

**During a network partition**, `entityRepository.findById()` cannot acquire a JDBC connection. Hikari's default `connectionTimeout` is 30 seconds. Each blocked request holds a Tomcat worker thread for the full 30 seconds. Spring Boot's default Tomcat thread pool is 200 threads. An attacker sending 200+ concurrent requests exhausts the pool, making the service completely unresponsive to all users.

**The only infrastructure-level guard** is the Traefik `inFlightReq: amount: 5` per source IP in the Helm chart: [5](#0-4) 

This is insufficient because: (a) it is only active in Kubernetes deployments using this specific chart, not enforced at the application layer; (b) it is per-IP, trivially bypassed with multiple source IPs or a botnet; (c) even 5 concurrent blocking requests per IP from 40 IPs fills the 200-thread pool.

### Impact Explanation

Complete denial of service for the GraphQL API. All 200 Tomcat threads become occupied waiting for DB connection timeouts. No legitimate requests can be processed. The service remains down for the duration of the partition plus the time to drain blocked threads. Since the GraphQL API is publicly accessible (no authentication required per the README and chart configuration), this is exploitable by any external actor.

### Likelihood Explanation

The precondition is a DB network partition, which can occur due to infrastructure failure, cloud provider issues, or a coordinated attack on the DB network. Once the partition exists, the exploit requires only a standard HTTP client sending concurrent POST requests to `/graphql/alpha` — no credentials, no special knowledge, no exploit code. The zero-prefix EVM address format is documented behavior. The attacker needs only ~200 concurrent connections from distributed IPs to exhaust the thread pool.

### Recommendation

1. **Add application-level rate limiting** to the graphql module, mirroring the web3 `ThrottleManager` pattern with bucket4j.
2. **Add `@Cacheable`** to `EntityRepository.findById()` in the graphql module to avoid repeated DB hits for the same entity ID.
3. **Enforce a per-request timeout** at the application layer (e.g., a `HandlerInterceptor` or Hibernate `StatementInspector` similar to `web3/HibernateConfiguration`) so that blocked DB calls are cancelled after a bounded time rather than holding threads for the full Hikari connection timeout.
4. **Configure Hikari `connectionTimeout`** explicitly to a low value (e.g., 2–5 seconds) for the graphql datasource to bound the maximum thread-hold time per request.

### Proof of Concept

```bash
# Zero-prefix EVM address: first 12 bytes zero, last 8 bytes = entity ID
# e.g., 0x0000000000000000000000000000000000000001

GRAPHQL_URL="http://<target>/graphql/alpha"
QUERY='{"query":"{account(input:{evmAddress:\"0x0000000000000000000000000000000000000001\"}){balance}}"}'

# During a DB network partition, fire 200+ concurrent requests:
for i in $(seq 1 250); do
  curl -s -X POST "$GRAPHQL_URL" \
    -H "Content-Type: application/json" \
    -d "$QUERY" &
done
wait

# Each request blocks a Tomcat thread for ~30s (Hikari connectionTimeout).
# After ~200 concurrent requests, the thread pool is exhausted.
# Subsequent legitimate requests receive connection refused or timeout.
```
