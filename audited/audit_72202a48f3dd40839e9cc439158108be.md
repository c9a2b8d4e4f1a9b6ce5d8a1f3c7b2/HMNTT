### Title
Unauthenticated Timestamp Parameter Doubles DB Round-Trips in `/api/v1/balances`, Enabling Connection-Pool Exhaustion via Griefing

### Summary
Any unauthenticated caller can supply `?timestamp=eq:<value>` to `GET /api/v1/balances`. The `opOverride` in `parseTimestampQueryParam()` silently converts `eq` to `lte`, which causes `getBalances()` to execute two sequential database queries per request instead of one. With no rate-limiting middleware in the REST service and a default pool of only 10 connections, a sustained flood of such requests can exhaust the connection pool and deny service to legitimate users.

### Finding Description

**Exact code path:**

In `rest/balances.js`, `getBalances()` (lines 83–156):

```
Line 90-92: timestamp=eq:X → opOverride converts eq → lte → tsQuery is non-empty
Line 113:   if (tsQuery) {
Line 114:     const tsQueryResult = await getTsQuery(tsQuery, tsParams);   // DB query #1
...
Line 153:   const result = await pool.queryQuietly(pgSqlQuery, sqlParams); // DB query #2
```

DB query #1 is issued inside `getTsQuery()` → `getAccountBalanceTimestampRange()` (lines 278–292, 167–197):

```js
// rest/balances.js line 189
const {rows} = await pool.queryQuietly(query, params);
```

This query hits `account_balance` to find the nearest snapshot timestamp for the treasury account. Only after it returns does DB query #2 execute (line 153) to fetch the actual balances.

The no-timestamp path (lines 133–150) issues only DB query #2.

**Root cause:** The `eq`→`lte` override is applied unconditionally with no guard that would short-circuit or cache the first lookup. Every request carrying any `timestamp` parameter (including the trivially valid `timestamp=eq:1`) unconditionally serialises two pool checkouts.

**Why existing checks fail:**

- `utils.validateReq()` (line 84) validates parameter syntax only; it does not reject or throttle timestamp-bearing requests.
- `getOptimizedTimestampRange()` (lines 229–276) may return early if `lowerBound > upperBound`, but a well-formed `timestamp=eq:<recent_ns>` always produces a valid range and proceeds to the DB.
- `grep` across `rest/**/*.js` for `rateLimit`, `rateLimiter`, `inFlightReq`, `throttle` returns **zero matches** — there is no application-level rate limiter.
- The REST Helm chart (`charts/hedera-mirror-rest/values.yaml`) contains no Traefik `rateLimit` or `inFlightReq` middleware, unlike `hedera-mirror-rosetta` and `hedera-mirror-graphql` which both define per-IP rate limits.
- The default DB pool is **10 connections** (`hiero.mirror.rest.db.pool.maxConnections = 10`, docs/configuration.md line 556) with a `statementTimeout` of 20 s, meaning each in-flight timestamped request can hold two pool slots for up to 20 s each.

### Impact Explanation

With a pool of 10 connections and each timestamped request consuming two sequential connections (each up to 20 s), as few as **5 concurrent attacker requests** can saturate the pool. All subsequent legitimate requests queue or time out. The impact is full denial of the `/api/v1/balances` endpoint (and potentially other endpoints sharing the same pool) for the duration of the flood. No authentication, API key, or on-chain action is required.

### Likelihood Explanation

The attack requires only an HTTP client and knowledge of the public API. The `timestamp` parameter is documented in the OpenAPI spec (`rest/api/v1/openapi.yml` lines 391–419). The attacker needs no credentials, no tokens, and no economic stake. The attack is trivially scriptable (`curl` in a loop), repeatable, and can be sustained indefinitely. The absence of any rate-limiting layer in the REST service (confirmed by code and Helm chart inspection) means there is no automatic mitigation.

### Recommendation

1. **Add a rate-limiting middleware** to the REST service (e.g., Traefik `rateLimit` + `inFlightReq` per source IP, matching the pattern already used for `hedera-mirror-rosetta`).
2. **Cache the `getAccountBalanceTimestampRange` result** (e.g., in Redis, which is already deployed) keyed on the effective `upperBound` timestamp, so repeated requests for the same or nearby timestamps do not each issue a separate DB query.
3. **Increase the default pool size** or add a per-endpoint concurrency cap so a single endpoint cannot monopolise all connections.
4. Consider **merging the two queries** into a single CTE so the snapshot lookup and balance fetch are one round-trip.

### Proof of Concept

```bash
# Flood with timestamp=eq:<current_ns> — no auth required
TS=$(date +%s)000000000   # current time in nanoseconds

for i in $(seq 1 50); do
  curl -s "http://<mirror-node-host>/api/v1/balances?timestamp=eq:${TS}" &
done
wait

# Legitimate request now queues/times out:
curl -v "http://<mirror-node-host>/api/v1/balances"
# Expected: connection timeout or 503 once pool is exhausted
```

Each background `curl` causes `getAccountBalanceTimestampRange` (DB query #1, `rest/balances.js` line 189) followed by the main balance query (DB query #2, line 153), holding two pool connections per request. const getAccountBalanceTimestampRange = async (tsQuery, tsParams) => {
  const {lowerBound, upperBound, neParams} = getOptimizedTimestampRange(tsQuery, tsParams);
  if (lowerBound === undefined) {
    return {};
  }

  // Add the treasury account to the query as it will always be in the balance snapshot and account_id is the first
  // column of the primary key
  let condition = 'account_id = $1 and consensus_timestamp >= $2 and consensus_timestamp <= $3';
  const params = [EntityId.systemEntity.treasuryAccount.getEncodedId(), lowerBound, upperBound];
  if (neParams.length) {
    condition += ' and not consensus_timestamp = any ($4)';
    params.push(neParams);
  }

  const query = `
    select consensus_timestamp
    from account_balance
    where ${condition}
    order by consensus_timestamp desc
    limit 1`;

  const {rows} = await pool.queryQuietly(query, params);
  if (rows.length === 0) {
    return {};
  }

  const upper = rows[0].consensus_timestamp;
  const lower = utils.getFirstDayOfMonth(upper);
  return {lower, upper};
};
