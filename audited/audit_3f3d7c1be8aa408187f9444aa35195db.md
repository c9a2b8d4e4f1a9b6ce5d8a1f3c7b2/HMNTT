### Title
Missing Impossible Range Validation for Non-Primary Sort Fields Allows Contradictory DB Queries

### Summary
The `Bound` constructor in `Bound.java` only validates that `adjustedLower <= adjustedUpper` when `primarySortField=true`. Since serial numbers are always constructed with `primarySortField=false`, an attacker can supply `serialnumber=gte:1&serialnumber=lte:0` to create an impossible range that bypasses all validation and propagates a contradictory `WHERE serial_number >= 1 AND serial_number <= 0` condition to the database. The query executes, consumes server resources, and always returns zero results.

### Finding Description

**Code path:**

In `TokenAirdropsController.java` line 103, the serial number `Bound` is always constructed with `primarySortField=false`:

```java
.serialNumbers(new Bound(serialNumbers, false, SERIAL_NUMBER, TOKEN_AIRDROP.SERIAL_NUMBER))
``` [1](#0-0) 

In `Bound.java` lines 56–60, the range inversion check is gated on `primarySortField`:

```java
long adjustedLower = getAdjustedLowerRangeValue();
long adjustedUpper = adjustUpperBound();
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException("Invalid range provided for %s".formatted(parameterName));
}
``` [2](#0-1) 

**Root cause:** The condition `primarySortField && adjustedLower > adjustedUpper` short-circuits to `false` for serial numbers because `primarySortField=false`. The impossible range is accepted silently.

**Propagation to DB:** In `JooqRepository.java` lines 32–35, the `getCondition(Bound)` method blindly applies both lower and upper bounds:

```java
private Condition getCondition(Bound bound) {
    var field = bound.getField();
    return getCondition(bound.getLower(), field).and(getCondition(bound.getUpper(), field));
}
``` [3](#0-2) 

This generates `serial_number >= 1 AND serial_number <= 0` — a contradiction that always returns zero rows but still executes a full DB query cycle.

### Impact Explanation
Any unauthenticated caller can force the server to execute DB queries that are guaranteed to return no results. Each such request consumes: an HTTP thread, an entity lookup via `entityService.lookup`, a DB connection, and query planning/execution overhead. At scale this constitutes a low-cost resource exhaustion vector. There is no data exposure or privilege escalation; the mirror node is read-only.

### Likelihood Explanation
Exploitation requires zero privileges — the endpoints `/api/v1/accounts/{id}/airdrops/outstanding` and `/api/v1/accounts/{id}/airdrops/pending` are public. The payload is trivial (`serialnumber=gte:1&serialnumber=lte:0`). It is repeatable at will with no rate-limiting visible in the reviewed code.

### Recommendation
Remove the `primarySortField` gate from the inversion check in `Bound.java`. The validation should apply unconditionally whenever both a lower and upper bound are present:

```java
if (adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException("Invalid range provided for %s".formatted(parameterName));
}
```

This ensures impossible ranges are rejected at input validation time regardless of whether the field is a primary or secondary sort field.

### Proof of Concept
```
GET /api/v1/accounts/0.0.1234/airdrops/outstanding?serialnumber=gte:1&serialnumber=lte:0
```
1. Controller calls `new Bound(serialNumbers, false, SERIAL_NUMBER, TOKEN_AIRDROP.SERIAL_NUMBER)`.
2. `adjustedLower=1`, `adjustedUpper=0`; check skipped because `primarySortField=false`.
3. `Bound` is created with lower=GTE:1, upper=LTE:0.
4. `TokenAirdropRepositoryCustomImpl.findAll` executes: `WHERE ... AND serial_number >= 1 AND serial_number <= 0 AND state = 'PENDING'`.
5. DB returns zero rows; server consumed resources for a provably empty query.
6. Repeat indefinitely with no authentication required.
