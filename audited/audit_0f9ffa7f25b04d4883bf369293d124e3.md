### Title
GraphQL `findByEvmAddress` Connection Pool Exhaustion via Unauthenticated Concurrent Requests (No Application-Level Rate Limiting)

### Summary
The GraphQL module exposes `findByEvmAddress()` through an unauthenticated endpoint with no application-level rate limiting, no result caching, and a HikariCP connection pool that defaults to 10 connections. Unlike the `web3` module which has a `ThrottleManagerImpl` with bucket-based rate limiting, the GraphQL module has no equivalent. A distributed attacker using multiple source IPs can bypass the per-IP Traefik `inFlightReq` middleware and exhaust the connection pool across mirror node instances, causing legitimate requests to queue and time out.

### Finding Description

**Exact code path:**

`AccountController.account()` → `EntityServiceImpl.getByEvmAddressAndType()` → `EntityRepository.findByEvmAddress()` executes:
```sql
select * from entity where evm_address = ?1 and deleted is not true
``` [1](#0-0) [2](#0-1) [3](#0-2) 

**Root cause — no application-level rate limiting in the GraphQL module:**

The `web3` module has a full `ThrottleManagerImpl` with bucket4j rate limiting applied before any DB access. The `graphql` module has no equivalent — `GraphQlConfiguration` only configures query complexity/depth limits and parser token limits, none of which throttle concurrent DB-bound requests. [4](#0-3) [5](#0-4) 

**No caching on `findByEvmAddress()`:**

The `web3` module's `EntityRepository` annotates its EVM address lookup with `@Cacheable`, absorbing repeated lookups without hitting the DB. The GraphQL module's `EntityRepository` has no such annotation — every request hits the database. [6](#0-5) [7](#0-6) 

**Connection pool:**

The `DataSource` bean is built from `spring.datasource.hikari` properties with no graphql-specific `application.yml` found. HikariCP's default `maximumPoolSize` is 10. The statement timeout is 10 seconds (`hiero.mirror.graphql.db.statementTimeout = 10000`), meaning each connection can be held for up to 10 seconds per query under load. [8](#0-7) [9](#0-8) 

**The only infrastructure-level mitigation — bypassable:**

The Traefik middleware limits in-flight requests to 5 per source IP. However: (1) a distributed attacker using ≥2 source IPs can hold all 10 default pool connections; (2) the `retry: attempts: 3` middleware multiplies each user-visible request into up to 3 actual backend requests, amplifying pool pressure; (3) the GCP gateway path (`gateway.gcp.enabled: true`) does not apply Traefik middleware. [10](#0-9) [11](#0-10) 

### Impact Explanation

When the HikariCP pool is exhausted, new requests block waiting for a connection up to HikariCP's `connectionTimeout` (default 30 seconds), then throw `SQLTransientConnectionException`. This renders the GraphQL endpoint unresponsive for all users. Because each mirror node instance has an independent pool, an attacker targeting multiple instances simultaneously (feasible with a botnet or cloud VMs) can degrade ≥30% of the network's GraphQL processing capacity. The `GraphQLHighDBConnections` Prometheus alert only fires after 5 minutes at >75% utilization, providing no fast-path defense. [12](#0-11) 

### Likelihood Explanation

No authentication is required to reach the GraphQL endpoint. The query accepts any 20-byte hex string as `evmAddress`. An attacker needs only 2 source IPs to saturate the default 10-connection pool (5 concurrent per IP × 2 IPs). The attack is trivially scriptable with `curl` or any HTTP client, is repeatable indefinitely, and requires no knowledge of valid addresses (distinct random addresses each bypass any hypothetical per-address deduplication). The `retry: attempts: 3` middleware means the attacker's effective request rate is tripled at the backend. [13](#0-12) 

### Recommendation

1. **Add application-level rate limiting to the GraphQL module** — port the `ThrottleManagerImpl`/bucket4j pattern from `web3` to `graphql`, applying a per-IP or global requests-per-second cap before any DB access.
2. **Add `@Cacheable` to `EntityRepository.findByEvmAddress()`** — mirror the caching strategy used in the `web3` module to absorb repeated lookups.
3. **Explicitly configure `spring.datasource.hikari.maximum-pool-size`** in the GraphQL application configuration to a value appropriate for expected concurrency, and set `connection-timeout` to fail fast rather than queue indefinitely.
4. **Reduce or remove the `retry: attempts: 3` Traefik middleware** for the GraphQL route, or ensure it only retries on network errors, not on application-level timeouts.
5. **Enforce the `inFlightReq` limit globally** (not just per-IP) as a secondary backstop.

### Proof of Concept

```bash
# From 2 source IPs (or using 2 proxies), run simultaneously:
# IP-1: 5 concurrent requests
for i in $(seq 1 5); do
  curl -s -X POST http://<mirror-node>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{account(input: {evmAddress: \\\"$(openssl rand -hex 20)\\\"}){balance}}\"}" &
done

# IP-2: 5 concurrent requests (same pattern)
# Result: all 10 HikariCP connections held; subsequent legitimate requests
# receive SQLTransientConnectionException / HTTP 500 until connections are released.
```

    Optional<Account> account(@Argument @Valid AccountInput input) {
