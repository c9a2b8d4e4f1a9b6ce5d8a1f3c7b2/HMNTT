### Title
Unauthenticated Cache-Bypass DoS via Unique Timestamp Enumeration in `doGetTransactions()`

### Summary
The `doGetTransactions()` function in `rest/transactions.js` performs two sequential database queries per request: one unconditional call to `getTransactionTimestamps()` and one conditional call to `getTransactionsDetails()` via the Redis cache loader. Because the cache key is derived from individual nanosecond-precision consensus timestamps, an unprivileged attacker can trivially guarantee perpetual cache misses by querying timestamp ranges containing unseen transactions, forcing both DB queries on every request. No rate limiting exists in the REST Node.js service, making this a straightforward connection-pool exhaustion attack.

### Finding Description

**Code path:**

`rest/transactions.js`, `doGetTransactions()`, lines 695–723:

```js
const doGetTransactions = async (filters, req, timestampRange) => {
  const { limit, order, nextTimestamp, rows: payerAndTimestamps } =
    await getTransactionTimestamps(filters, timestampRange);   // ← DB query 1, ALWAYS

  const loader = (keys) =>
    getTransactionsDetails(keys, order).then((result) => formatTransactionRows(result.rows));

  const transactions = await cache.get(payerAndTimestamps, loader, keyMapper); // ← DB query 2, on cache miss
  ...
};
``` [1](#0-0) 

**DB query 1 is unconditional.** `getTransactionTimestamps()` calls `pool.queryQuietly()` on every request with no caching layer protecting it. [2](#0-1) 

**Cache key construction** (`keyMapper`, lines 679–685) maps each result row to `transaction:<consensus_timestamp_digits>`. Consensus timestamps are nanosecond-precision integers unique per transaction. [3](#0-2) 

**`cache.get()` in `rest/cache.js`** (lines 95–140): on any cache miss, `loader(missingKeys)` is awaited synchronously, executing `getTransactionsDetails()` — a multi-table CTE JOIN — against the DB. [4](#0-3) 

**`mset` stores without TTL** (line 130): cached entries never expire on their own, so the cache fills with historical transaction data. Any timestamp range not previously queried is guaranteed to miss. [5](#0-4) 

**No rate limiting** exists in the REST Node.js service for the `/transactions` endpoint. The throttling found in the codebase (`ThrottleConfiguration`, `ThrottleManagerImpl`) belongs exclusively to the `web3/` Java service and does not apply here. [6](#0-5) 

**Root cause / failed assumption:** The design assumes the Redis cache will absorb repeated queries for the same transactions. This assumption fails because: (a) DB query 1 is never cached, (b) the blockchain continuously produces new transactions with new nanosecond timestamps that are never in cache, and (c) there is no mechanism to prevent an attacker from querying only fresh or previously-unseen timestamp ranges.

### Impact Explanation

Each attacker request causes at minimum one complex DB query (`getTransactionTimestamps`) and, on cache miss, a second heavy multi-table CTE query (`getTransactionsDetails`). With no rate limiting, a single attacker with modest concurrency (e.g., 50–200 concurrent HTTP connections) can hold the DB connection pool saturated. PostgreSQL connection pools in typical mirror-node deployments are bounded; exhaustion causes query queuing and timeouts across all API endpoints, degrading or halting service for all users. This meets the stated threshold of degrading ≥30% of network processing nodes because the mirror node REST API is a shared infrastructure component.

### Likelihood Explanation

The attack requires zero authentication, zero special knowledge, and zero brute force. The attacker only needs to:
- Know the public API endpoint (`/api/v1/transactions`)
- Use any standard HTTP load tool (e.g., `wrk`, `ab`, `hey`)
- Supply a `timestamp=gte:X&timestamp=lte:Y` parameter with a sliding or random window

Since the Hedera network produces thousands of transactions per second, there is an effectively infinite supply of timestamp ranges that will always miss the cache. The attack is repeatable, stateless, and requires no prior access.

### Recommendation

1. **Add rate limiting to the REST Node.js service** for the `/transactions` endpoint (e.g., via `express-rate-limit` or an API gateway policy), keyed by IP or API key.
2. **Cache the first DB query** (`getTransactionTimestamps`) results, not just the second. A short TTL (e.g., 1–5 seconds) keyed on the normalized filter parameters would dramatically reduce DB load for repeated or similar queries.
3. **Set a TTL on `mset`** in `cache.js` (replace `mset` with `mset` + `expire` or use `SET EX` per key) so stale entries are evicted and the cache does not fill with unbounded historical data.
4. **Enforce a DB connection pool limit with a request queue cap** and return HTTP 503 early when the pool is exhausted, preventing cascading timeouts.
