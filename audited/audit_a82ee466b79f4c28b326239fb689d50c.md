### Title
Unauthenticated Transaction Enumeration via `from` Filter in `/contracts/:contractId/results`

### Summary
The public endpoint `GET /api/v1/contracts/:contractId/results` accepts a `from` query parameter that is directly translated into a `cr.sender_id = $N` SQL condition with no authentication or authorization check. Any unprivileged external caller can supply an arbitrary victim address as `from` and enumerate every transaction that victim submitted to the target contract, including timestamps, function parameters (calldata), transaction results, and gas usage.

### Finding Description
**Code path:**

In `rest/controllers/contractController.js`, `getContractResultsById()` (lines 856–892) calls `extractContractIdAndFiltersFromValidatedRequest()` which only validates the format of the `from` parameter (entity ID or EVM address format), then passes the filters to `extractContractResultsByIdQuery()` (lines 415–542).

Inside `extractContractResultsByIdQuery()`, the `filterKeys.FROM` case (lines 441–453) converts an EVM address to an encoded entity ID if needed, then calls `updateConditionsAndParamsWithInValues()` to append `cr.sender_id = $N` to the SQL WHERE conditions. No ownership check, session check, or any form of authorization is performed at any point.

The resulting query executed by `ContractService.getContractResultsByIdAndFilters()` (lines 214–227 of `rest/service/contractService.js`) returns rows containing: `consensus_timestamp`, `transaction_result`, `gas_used`, `function_parameters`, `call_result`, `bloom`, `error_message`, `created_contract_ids`, `transaction_hash`, `transaction_nonce`, and `payer_account_id`.

**Root cause:** The API is a read-only mirror node endpoint with no authentication layer. The `from` filter is treated purely as a data-filtering parameter with no check that the requester has any relationship to the address being queried.

**Failed assumption:** The design assumes that knowing a sender address plus a contract address is not sensitive, but the combination reveals the full interaction history of that address with the contract.

### Impact Explanation
An attacker who knows (or can guess/enumerate) a victim's account address can:
- Reconstruct the complete history of every transaction the victim sent to a specific contract, including exact timestamps
- Read `function_parameters` (raw calldata), revealing which contract functions were called and with what arguments — this can expose DeFi trading strategies, voting behavior, private protocol interactions, etc.
- Observe `transaction_result` and `error_message`, revealing failed attempts and their reasons
- Correlate `transaction_nonce` values to infer account activity patterns

This is a **privacy/information-disclosure** vulnerability. On a public blockchain the data is technically on-chain, but the mirror node API makes bulk enumeration trivially easy with no rate-limiting barrier beyond standard HTTP, and the `function_parameters` field exposes decoded calldata that would otherwise require parsing raw transactions.

### Likelihood Explanation
- **Precondition:** Attacker only needs to know the victim's account address (publicly derivable from any prior on-chain interaction) and the contract address of interest.
- **No credentials required:** The endpoint is fully public.
- **Trivially repeatable:** A single HTTP GET request suffices; pagination via `timestamp` allows full history extraction.
- **Automated enumeration:** Attackers can script bulk queries across many addresses and contracts.

### Recommendation
1. **Rate-limit** the `from` filter at the API gateway level to prevent bulk enumeration.
2. **Consider removing or gating** the `from` filter behind an authenticated/permissioned tier if the mirror node deployment is intended to serve sensitive data.
3. At minimum, **document** that the `from` filter exposes full sender transaction history so operators can make an informed access-control decision.
4. If privacy is a requirement, **redact** `function_parameters` and `call_result` from unauthenticated responses when a `from` filter is applied.

### Proof of Concept
```
# Step 1: Identify victim address (any known account, e.g. 0.0.9001)
# Step 2: Identify target contract (e.g. 0.0.5001)

# Step 3: Enumerate all transactions victim sent to the contract
GET /api/v1/contracts/0.0.5001/results?from=0.0.9001

# Step 4: Paginate through full history
GET /api/v1/contracts/0.0.5001/results?from=0.0.9001&order=asc&limit=100&timestamp=lt:987654000123456

# Response includes for each transaction:
# - consensus_timestamp (exact time)
# - function_parameters (calldata — what function, what args)
# - call_result (return value)
# - transaction_result (success/failure)
# - error_message (failure reason)
# - gas_used, transaction_nonce, transaction_hash
```
