### Title
Unbounded Concurrent Query DoS via Missing Timeout on `getSupplyHistory()` in `/api/v1/network/supply`

### Summary
`NetworkServiceImpl.getSupply()` invokes `accountBalanceRepository.getSupplyHistory()` with no query timeout, no rate limiting, and no DB-level statement timeout. Although a one-month window clamp exists, an unprivileged attacker can flood the endpoint with concurrent requests, each triggering an expensive `DISTINCT ON` + `unnest` join scan over up to ~31 days of `account_balance` rows, exhausting the DB connection pool and starving all other queries.

### Finding Description

**Code path:**

`NetworkController.getSupply()` (line 133–150) → `NetworkServiceImpl.getSupply()` (lines 59–89) → `AccountBalanceRepository.getSupplyHistory()` (lines 12–28).

**Root cause:**

In `NetworkServiceImpl.getSupply()`, when a timestamp bound is provided, the only mitigation is:

```java
// NetworkServiceImpl.java lines 77-78
final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1);
minTimestamp = Math.max(minTimestamp, optimalLowerBound);
```

This clamps `minTimestamp` to the first day of the prior calendar month relative to `maxTimestamp`, bounding the scan to at most ~31 days. However, no `statement_timeout`, `@QueryHints`, Spring `@Transactional(timeout=...)`, or JDBC socket timeout is applied anywhere in the call chain (confirmed: zero matches for `timeout`, `QueryHint`, `lock_timeout` in `rest-java/**/*.java`).

The underlying SQL in `AccountBalanceRepository.getSupplyHistory()` (lines 12–28) performs:
- A `WHERE ab.consensus_timestamp BETWEEN :lowerTimestamp AND :upperTimestamp` range scan
- A `JOIN unnest(...)` on account-ID ranges
- A `DISTINCT ON (ab.account_id) ... ORDER BY ab.account_id ASC, ab.consensus_timestamp DESC`

On a production mirror node with millions of balance records per month, this query can run for many seconds per request. The `/api/v1/network/supply` endpoint is publicly accessible with no authentication and no rate limiting (confirmed: no `RateLimiter` or throttle annotation on this endpoint).

**Why the existing check is insufficient:**

The `getFirstDayOfMonth` clamp only limits the *time window*, not the *number of rows* within that window, the *number of concurrent requests*, or the *query execution duration*. A 31-day window over a high-throughput network can still scan tens of millions of rows per query.

### Impact Explanation

An attacker sending N concurrent `GET /api/v1/network/supply?timestamp=lte:<now>` requests causes N simultaneous long-running DB queries. The HikariCP connection pool (typically 10–20 connections by default in Spring Boot) is exhausted within seconds. All other DB-dependent endpoints (block ingestion queries, other REST endpoints) queue or fail with connection timeout errors, effectively halting the mirror node's ability to serve data. This is a complete, unauthenticated denial-of-service against the mirror node's database tier.

### Likelihood Explanation

Preconditions: none — the endpoint is public, unauthenticated, and requires only a valid timestamp parameter. The attack is trivially scriptable with `curl` or any HTTP load tool. It is repeatable indefinitely. A single attacker with a modest internet connection can sustain the attack. No special knowledge of the system is required beyond the public API documentation.

### Recommendation

1. **Add a DB-level query timeout** via `@QueryHints` on `getSupplyHistory`:
   ```java
   @QueryHints(@QueryHint(name = "jakarta.persistence.query.timeout", value = "5000"))
   ```
2. **Add a Spring `@Transactional(timeout = 5)`** on `NetworkServiceImpl.getSupply()`.
3. **Apply rate limiting** to `GET /api/v1/network/supply` (e.g., Bucket4j or Spring's `RateLimiter` integration), separate from the high-volume throttle already used for fee estimation.
4. **Set PostgreSQL `statement_timeout`** at the connection/role level as a defense-in-depth measure.

### Proof of Concept

```bash

        final NetworkSupply networkSupply;

        final var bounds = networkProperties.getUnreleasedSupplyRangeBounds();
        final var lowerBounds = bounds.lowerBounds();
        final var upperBounds = bounds.upperBounds();

        if (timestamp.isEmpty()) {
            networkSupply = entityRepository.getSupply(lowerBounds, upperBounds);
        } else {
            var minTimestamp = timestamp.getAdjustedLowerRangeValue();
            final var maxTimestamp = timestamp.adjustUpperBound();

            // Validate timestamp range
            if (minTimestamp > maxTimestamp) {
                throw new IllegalArgumentException("Invalid range provided for timestamp");
            }

            final var optimalLowerBound = getFirstDayOfMonth(maxTimestamp, -1);
            minTimestamp = Math.max(minTimestamp, optimalLowerBound);

            networkSupply =
                    accountBalanceRepository.getSupplyHistory(lowerBounds, upperBounds, minTimestamp, maxTimestamp);
        }

        if (networkSupply.consensusTimestamp() == 0L) {
            throw new EntityNotFoundException("Network supply not found");
        }

        return networkSupply;
    }
