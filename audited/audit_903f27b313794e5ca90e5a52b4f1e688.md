### Title
Unprivileged DB Statement Timeout Flood Silences Loki Alerting and Suppresses Stack Traces in REST Error Handler

### Summary
Any unprivileged external user can send requests that trigger PostgreSQL statement timeouts on the REST API. When this occurs, `shouldPrintStacktrace()` suppresses the full stack trace from logs, and the Loki `RestLogErrors` alert rule explicitly filters out "canceling statement due to statement timeout" log lines — meaning a sustained DB overload attack produces no operator alert and reduced diagnostic logging. The two suppressions compound: the code-level suppression reduces log detail, and the monitoring-level suppression eliminates alerting entirely.

### Finding Description

**Code path 1 — stack trace suppression:**

In `rest/middleware/httpErrorHandler.js`, the regex `simpleErrors = /statement timeout/` is defined at line 10. [1](#0-0) 

`shouldPrintStacktrace()` returns `false` whenever the error message matches this pattern: [2](#0-1) 

In `handleError()`, when the status code is `SERVICE_UNAVAILABLE` (the code path for `DbError`), `shouldReturnMessage()` returns `false`, so execution falls into the `else` branch where `shouldPrintStacktrace()` controls whether the full error object or only `err.message` is logged: [3](#0-2) 

**Code path 2 — how user requests reach this path:**

All DB queries go through `queryQuietly()` in `rest/utils.js`. Any PostgreSQL error — including "canceling statement due to statement timeout" — is caught and re-thrown as `new DbError(err.message)`, preserving the PostgreSQL error string verbatim: [4](#0-3) 

The DB pool has a default `statement_timeout` of 20,000 ms and only 10 max connections: [5](#0-4) 

**Code path 3 — Loki alert suppression:**

The `RestLogErrors` Prometheus/Loki alert rule explicitly excludes log lines containing "canceling statement due to statement timeout" from the alert expression: [6](#0-5) 

**Root cause:** The design assumes statement timeouts are routine/expected noise and intentionally suppresses both stack traces and alerting for them. This assumption fails when an attacker deliberately and continuously triggers timeouts to exhaust the 10-connection pool, causing all legitimate queries to also time out — with no alert firing.

### Impact Explanation

With the 10-connection pool exhausted by attacker-controlled slow queries, all legitimate REST API requests for transaction/block data return 503 `Service Unavailable`. The mirror node becomes effectively unavailable for querying Hedera transaction data. Because the Loki alert rule filters out exactly these errors, operators receive no automated notification. The only signal is raw log inspection, which is not a real-time detection mechanism. The reduced log detail (message only, no stack trace) also makes post-incident forensics harder.

### Likelihood Explanation

No authentication or rate limiting is required. The attacker only needs to identify REST API endpoints that generate expensive DB queries (e.g., wide timestamp range scans, which the configuration explicitly allows up to 7 days: `maxTimestampRange: 7d`). Sending 10 concurrent requests to such endpoints is trivial and can be sustained indefinitely. The attack is repeatable, cheap, and requires no special knowledge beyond the public API documentation.

### Recommendation

1. **Remove the Loki filter exclusion** for "canceling statement due to statement timeout" in `charts/hedera-mirror-common/values.yaml` line 149, or replace it with a rate-based threshold alert that fires when statement timeouts exceed a sustained rate (e.g., >N per minute for >M minutes).
2. **Do not conflate "suppress stack trace" with "suppress alerting"** — the `shouldPrintStacktrace()` logic is a log verbosity optimization, but it should not be the basis for the monitoring exclusion. These are separate concerns.
3. **Add rate limiting** on expensive REST API endpoints to prevent a single client from exhausting the connection pool.
4. **Add a dedicated alert** for sustained 503 response rates from the REST API, independent of the log-content-based alert.

### Proof of Concept

```
# 1. Identify an expensive endpoint (e.g., transactions with wide timestamp range)
# 2. Send 10 concurrent requests that will each run for ~20s before timing out:

for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/transactions?timestamp=gte:2020-01-01T00:00:00Z&timestamp=lte:2027-01-01T00:00:00Z&limit=100" &
done
wait

# 3. Observe: all requests return HTTP 503
# 4. Check Loki/alerting: no RestLogErrors alert fires (filtered by != "canceling statement due to statement timeout")
# 5. Check logs: only err.message logged, no stack trace, no alert
# 6. Repeat continuously — the mirror node remains unavailable for legitimate users
#    with zero automated operator notification
```
