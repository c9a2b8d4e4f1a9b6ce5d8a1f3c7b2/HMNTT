### Title
Half-Open TCP Connection Resource Exhaustion via Missing Keepalive and Idle Timeout on gRPC Netty Server

### Summary
`grpcServerConfigurer()` in `GrpcConfiguration.java` configures the `NettyServerBuilder` with only an executor and `maxConcurrentCallsPerConnection`, omitting all connection liveness controls (`keepAliveTime`, `keepAliveTimeout`, `maxConnectionIdle`, `maxConnectionAge`). An unprivileged external attacker can open large numbers of TCP connections, then silently drop all outgoing packets, leaving the server holding half-open connections indefinitely with no mechanism to detect or reclaim them.

### Finding Description
**Exact code path:**
`grpc/src/main/java/org/hiero/mirror/grpc/config/GrpcConfiguration.java`, `grpcServerConfigurer()`, lines 28–35:

```java
return serverBuilder -> {
    serverBuilder.executor(applicationTaskExecutor);
    serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
};
```

`NettyProperties` (lines 11–15) exposes only `maxConcurrentCallsPerConnection = 5`. A grep across the entire `grpc/` module for `keepAlive`, `maxConnectionIdle`, `maxConnectionAge`, `connectionTimeout`, and `idleTimeout` returns **zero matches** — none of these are set anywhere.

**Root cause:** `NettyServerBuilder` has no default application-level keepalive or idle-connection eviction. Without `keepAliveTime`/`keepAliveTimeout`, the server never sends HTTP/2 PING frames to probe liveness. Without `maxConnectionIdle`, connections that carry no active calls are never reaped. The OS TCP keepalive fires only after ~7200 s (Linux default), so dead connections persist for up to two hours per cycle.

**Exploit flow:**
1. Attacker opens N TCP connections to port 5600 and completes the three-way handshake (server accepts each).
2. Attacker installs a local `iptables -A OUTPUT -d <server> -p tcp --dport 5600 -j DROP` rule, silencing all outgoing packets — simulating a one-sided network partition without any privileged network access.
3. Attacker never sends the HTTP/2 client preface; no RPC call is initiated, so `maxConcurrentCallsPerConnection = 5` is never reached and provides zero protection.
4. Server holds every connection open: each consumes a file descriptor, a Netty `Channel` object, and a slot in the `applicationTaskExecutor` thread pool.
5. Repeat until the server's file-descriptor limit (`ulimit -n`, typically 65535) or thread pool is exhausted.

**Why existing checks fail:** `maxConcurrentCallsPerConnection` caps *calls per connection*, not the number of connections. There is no per-IP connection rate limit, no total connection cap, and no idle-connection reaper anywhere in the configuration.

### Impact Explanation
Exhausting file descriptors causes `accept()` to fail for all new connections, making the gRPC service completely unavailable to legitimate clients. Thread-pool exhaustion causes in-flight subscriptions to stall. Because the gRPC module is the sole interface for topic subscriptions (the primary consumer-facing API), a sustained attack produces a full denial of service. Severity: **High**.

### Likelihood Explanation
The attack requires only the ability to reach TCP port 5600 — no credentials, no authentication, no special network position. The `iptables DROP` trick is a standard technique requiring only local root on the attacker's own machine (not on the server). The attack is trivially scriptable, repeatable, and can be sustained indefinitely. A single commodity host can open tens of thousands of connections before exhausting the server.

### Recommendation
Add the following calls inside `grpcServerConfigurer()`:

```java
serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
serverBuilder.maxConnectionIdle(60, TimeUnit.SECONDS);
serverBuilder.maxConnectionAge(300, TimeUnit.SECONDS);
serverBuilder.maxConnectionAgeGrace(10, TimeUnit.SECONDS);
serverBuilder.permitKeepAliveWithoutCalls(false);
```

Expose `keepAliveTime`, `keepAliveTimeout`, `maxConnectionIdle`, and `maxConnectionAge` as fields in `NettyProperties` so operators can tune them. Additionally, consider adding a per-IP connection rate limit at the infrastructure layer (nginx `limit_conn`, or a Kubernetes `NetworkPolicy`).

python3 - <<'EOF'
import socket, time, threading

socks = []
for i in range(5000):
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect(("<grpc-server>", 5600))
    socks.append(s)
    if i % 100 == 0:
        print(f"Opened {i} connections")

# Simulate network partition: drop all outgoing packets to the server
import subprocess
subprocess.run(["iptables", "-A", "OUTPUT", "-d", "<grpc-server>",
                "-p", "tcp", "--dport", "5600", "-j", "DROP"])

print("Partition simulated. Server now holds half-open connections.")
time.sleep(7300)   # wait past OS keepalive default (~2h)
EOF
