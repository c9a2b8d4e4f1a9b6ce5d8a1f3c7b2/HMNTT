### Title
Unbounded Reactive Subscription via `Long.MAX_VALUE` endTime in `convertTimestamp()`

### Summary
An unprivileged gRPC client can supply a `Timestamp` with `seconds >= 9223372035L` to `subscribeTopic`, causing `convertTimestamp()` to return `Long.MAX_VALUE` as `endTime`. This value passes all validation checks and causes the reactive subscription to never terminate, because `isComplete()` evaluates the corresponding `Instant` as year ~2262, which is never in the past. An attacker can open arbitrarily many such subscriptions to exhaust server thread/connection resources.

### Finding Description

**Exact code path:**

`ConsensusController.convertTimestamp()` ŌĆö [1](#0-0) 

```java
private long convertTimestamp(Timestamp timestamp) {
    if (timestamp.getSeconds() >= 9223372035L) {
        return Long.MAX_VALUE;
    }
    return DomainUtils.timestampInNanosMax(timestamp);
}
```

Any `Timestamp` with `seconds >= 9223372035L` unconditionally returns `Long.MAX_VALUE` (9223372036854775807).

**Validation bypass:**

`TopicMessageFilter.isValidEndTime()` ŌĆö [2](#0-1) 

```java
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

The default `startTime` is `DomainUtils.now()` (current time in nanoseconds, ~1.7├Ś10┬╣ŌüĖ). Since `Long.MAX_VALUE` (9.2├Ś10┬╣ŌüĖ) > any current nanosecond timestamp, this check always passes. No privileged startTime is needed.

`isValidStartTime()` ŌĆö [3](#0-2)  ŌĆö only requires `startTime <= DomainUtils.now()`, which the default satisfies.

**Non-terminating subscription:**

`TopicMessageServiceImpl.isComplete()` ŌĆö [4](#0-3) 

```java
return Instant.ofEpochSecond(0, filter.getEndTime())
        .plus(grpcProperties.getEndTimeInterval())
        .isBefore(Instant.now());
```

`Instant.ofEpochSecond(0, Long.MAX_VALUE)` normalizes to approximately year 2262. Adding `endTimeInterval` (default 30 s) keeps it in year 2262. `isBefore(Instant.now())` is always `false` in 2026, so `isComplete()` never returns `true`.

`pastEndTime()` ŌĆö [5](#0-4)  ŌĆö polls `isComplete()` indefinitely and never emits a terminal signal, so `takeUntilOther(pastEndTime(topicContext))` never fires.

`takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime())` ŌĆö [6](#0-5)  ŌĆö since `Long.MAX_VALUE` is larger than any real consensus timestamp, this predicate is always `true` and never terminates the stream.

No subscriber count cap or rate limit exists in `GrpcProperties`. [7](#0-6) 

### Impact Explanation
Each malicious subscription holds a Reactor subscription, a `TopicContext` object, a live gRPC stream, and a bounded-elastic scheduler slot. With no per-client limit or max-subscriber guard, an attacker can open thousands of such subscriptions from a single client or a small botnet, exhausting the server's thread pool and connection table. Legitimate subscribers ŌĆö including those waiting for fund-release or settlement messages ŌĆö are starved or rejected. This is a server-side resource exhaustion / denial-of-service with no authentication requirement.

### Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. The trigger requires only setting `consensusEndTime.seconds` to any value `>= 9223372035` in a standard protobuf message ŌĆö trivially achievable with any gRPC client or `grpcurl`. The attack is repeatable, scriptable, and requires no knowledge of internal state. No special privileges, tokens, or on-chain funds are needed.

### Recommendation
1. **Cap `convertTimestamp` output**: Instead of returning `Long.MAX_VALUE`, reject or clamp to a reasonable maximum (e.g., `now + configurable_max_window`):
   ```java
   private long convertTimestamp(Timestamp timestamp) {
       if (timestamp.getSeconds() >= 9223372035L) {
           throw new IllegalArgumentException("Timestamp exceeds maximum allowed value");
       }
       return DomainUtils.timestampInNanosMax(timestamp);
   }
   ```
2. **Add a maximum subscription duration**: Enforce a configurable `maxEndTime = now + maxWindow` in `toFilter()`, rejecting requests where `endTime` exceeds it.
3. **Add a maximum subscriber count**: Track `subscriberCount` in `TopicMessageServiceImpl` and reject new subscriptions above a threshold.
4. **Add per-IP rate limiting** at the gRPC interceptor layer.

### Proof of Concept
```bash
# Using grpcurl against a running mirror-node gRPC endpoint
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusEndTime": {"seconds": 9223372035, "nanos": 0}
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Connection stays open indefinitely; repeat in a loop to exhaust server resources.
```

**Step-by-step:**
1. Send `ConsensusTopicQuery` with `consensusEndTime.seconds = 9223372035` (no auth required).
2. `convertTimestamp()` returns `Long.MAX_VALUE` (line 77ŌĆō78 of `ConsensusController.java`).
3. `isValidEndTime()` passes: `Long.MAX_VALUE > DomainUtils.now()`.
4. `isComplete()` always returns `false`: endTime Instant is year 2262.
5. `pastEndTime()` never emits; `takeWhile` never triggers.
6. Subscription remains open indefinitely, consuming server resources.
7. Repeat from multiple connections to exhaust the thread pool.
