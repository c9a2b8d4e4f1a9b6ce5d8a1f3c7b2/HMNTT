### Title
Unbounded Historical Scan via `startTime=0` Enables Resource Exhaustion DoS in `subscribeTopic()`

### Summary
An unprivileged gRPC client can set `consensusStartTime` to epoch zero (nanosecond value `0`) in a `ConsensusTopicQuery`, which passes all validation checks and causes `PollingTopicMessageRetriever` to paginate through every historical message for the given topic with no upper bound on total pages or duration. With no per-client rate limiting or maximum subscriber enforcement, multiple concurrent such subscriptions exhaust database connections, bounded-elastic scheduler threads, and heap, degrading or denying service to legitimate subscribers.

### Finding Description

**Validation allows `startTime = 0`:**

`TopicMessageFilter.startTime` carries `@Min(0)`, making `0` the explicit minimum accepted value. [1](#0-0) 

The only semantic guard is `isValidStartTime()`, which checks `startTime <= DomainUtils.now()`. Because `0` is always ≤ the current nanosecond timestamp, this check always passes for `startTime = 0`. [2](#0-1) 

**Repository query with `startTime = 0`:**

`TopicMessageRepositoryCustomImpl.findByFilter()` builds a JPA criteria query with `consensusTimestamp >= filter.getStartTime()`. With `startTime = 0` this predicate matches every row ever written for the topic. [3](#0-2) 

`setMaxResults` is only applied when `filter.hasLimit()` is true (i.e., `limit > 0`). A client that omits the limit field sends `limit = 0`, so `hasLimit()` returns false and no SQL `LIMIT` is applied to the underlying query. [4](#0-3) 

**Pagination loop in `PollingTopicMessageRetriever`:**

`poll()` caps each individual page at `maxPageSize`, but `isComplete()` in the throttled path only returns `true` when the last page returned *fewer* rows than `maxPageSize`. For a topic with millions of messages, every page is full, so `isComplete()` never returns `true` and the `repeatWhen` loop runs indefinitely (up to `Long.MAX_VALUE` repeats). [5](#0-4) [6](#0-5) 

**No subscriber cap or rate limiting:**

`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge metric; there is no enforcement of a maximum value. Any number of concurrent zero-startTime, no-limit subscriptions are accepted. [7](#0-6) 

**`subscribeTopic()` wires historical retrieval directly:**

The historical `Flux` is created unconditionally from `topicMessageRetriever.retrieve(filter, true)` before any back-pressure or concurrency guard. [8](#0-7) 

### Impact Explanation
Each malicious subscription with `startTime=0` and no limit issues a continuous stream of paginated database queries for the full history of a topic. Concurrent such subscriptions multiply database load linearly. The `boundedElastic` scheduler has a finite thread pool; saturating it blocks legitimate subscribers from receiving live consensus messages, directly disrupting the gossip/notification path described in the HCS architecture. In the worst case the service becomes unresponsive to all clients.

### Likelihood Explanation
The gRPC `ConsensusTopicQuery` protobuf exposes `consensusStartTime` as an optional field; omitting it or setting it to `{seconds:0, nanos:0}` is trivially achievable by any unauthenticated client with network access to the mirror node's gRPC port. No credentials, special roles, or knowledge of internal state are required. The attack is fully repeatable and scriptable (open N connections in a loop). The own test suite confirms `startTime(0)` is a supported and exercised code path. [9](#0-8) 

### Recommendation
1. **Enforce a minimum `startTime` floor**: Reject or clamp `startTime` values older than a configurable retention window (e.g., 30 days before `now()`). Replace `@Min(0)` with a custom validator that checks `startTime >= DomainUtils.now() - maxHistoricalLookback`.
2. **Enforce a maximum subscriber count**: Add a configurable `maxSubscribers` property and reject new subscriptions when the limit is reached.
3. **Require a non-zero `limit` for open-ended historical queries**: If `startTime` is significantly in the past and no `endTime` is set, require `limit > 0`.
4. **Add per-IP / per-client rate limiting** at the gRPC interceptor layer.

### Proof of Concept
```python
# Requires grpcio and the mirror node proto stubs
import grpc
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# Craft filter: startTime = epoch 0, no limit, valid topic
query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1234),
    consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
    # limit intentionally omitted (defaults to 0 = unlimited)
)

# Open N concurrent subscriptions to exhaust resources
import threading
def subscribe():
    for _ in stub.subscribeTopic(query):
        pass  # drain indefinitely

threads = [threading.Thread(target=subscribe) for _ in range(50)]
for t in threads: t.start()
# Observe: database CPU spikes, legitimate subscribers stop receiving messages
```
