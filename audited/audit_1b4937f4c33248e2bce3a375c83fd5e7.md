### Title
LRU Entity Cache Poisoning via Unauthenticated `/block` Requests Causes Sustained DB Query Load Amplification

### Summary
The Rosetta `blockAPIService` uses a pure LRU entity cache (no TTL, no expiry) to memoize account alias lookups. Any unauthenticated caller can request blocks referencing valid but rarely-accessed entity IDs, continuously displacing hot cache entries with cold ones. Each displaced hot entry becomes a guaranteed DB query on the next legitimate access, enabling a sustained increase in database query load well above the 30% threshold.

### Finding Description

**Exact code locations:**

`rosetta/app/config/types.go` lines 35–37 — the `Cache` struct exposes only `MaxSize`; there is no TTL, no expiry-after-access, and no expiry-after-write field: [1](#0-0) 

`rosetta/app/services/block_service.go` lines 34–37 — the cache is instantiated as a pure LRU with capacity `entityCacheConfig.MaxSize` (default 524 288 per docs) and no time-based eviction: [2](#0-1) 

`rosetta/app/services/block_service.go` lines 104–130 — `updateOperationAccountAlias` is called for every operation in every transaction returned by `/block` and `/block/transaction`. On a cache miss it calls `accountRepo.GetAccountAlias`, which executes `SELECT alias, id FROM entity WHERE id = @id` against the database: [3](#0-2) 

`rosetta/app/persistence/account.go` line 51 — the raw SQL issued on every cache miss: [4](#0-3) 

**Root cause:** The LRU policy evicts the least-recently-used entry when capacity is reached. An attacker who continuously injects new, unique entity IDs into the cache forces eviction of the most-frequently-used (hot) entries. Because there is no TTL, evicted hot entries are not re-warmed until legitimate traffic re-requests them — at which point each one costs a DB round-trip. The `Cache` type has no mechanism to express or enforce any time-based protection. [1](#0-0) 

**Failed assumption:** The design assumes the working set of entity IDs accessed by legitimate traffic is small enough relative to `MaxSize` that the LRU will naturally retain hot entries. This assumption fails when an adversary can freely inject an unbounded stream of distinct cold IDs.

### Impact Explanation

Every `/block` response can contain up to `maxTransactionsInBlock` transactions, each with multiple operations, each referencing a distinct `AccountId`. A single block request can therefore trigger O(transactions × operations) cache lookups. When those lookups miss, each one issues a synchronous DB query before the response is returned. Sustained cache-poisoning at even a modest request rate (e.g., 2–5 req/s, each touching 20–50 unique cold IDs) can multiply the DB query rate by a factor that easily exceeds 30% above the 24-hour baseline, degrading response latency for all legitimate callers sharing the same database connection pool (default `maxOpenConnections = 100`). [5](#0-4) 

### Likelihood Explanation

**Preconditions:** None. The Rosetta `/block` endpoint requires no authentication. Entity IDs on Hedera are sequential integers and are publicly enumerable from the ledger or mirror node REST API.

**Feasibility:** The optional Traefik middleware in the Helm chart defines `rateLimit: average: 10, sourceCriterion: requestHost: true` — a global 10 req/s cap shared across all clients, not a per-IP cap — and `inFlightReq: amount: 5` per IP depth-1. [6](#0-5) 

These controls are: (a) only present in Kubernetes/Traefik deployments, absent from bare-metal or Docker deployments; (b) not part of the application itself; (c) bypassable by distributing requests across multiple source IPs. The Rosetta Go application contains **no built-in rate limiting or per-client throttling** — unlike the web3 API which has a `ThrottleManager`.

**Repeatability:** The attack is self-sustaining. As long as the attacker keeps sending requests with fresh entity IDs at a rate that exceeds the natural re-warming rate of legitimate traffic, the hot-entry eviction cycle continues indefinitely.

### Recommendation

1. **Add TTL to the entity cache.** Replace the pure LRU with an LRU+TTL policy (the `go-generics-cache` library supports `cache.WithTTL`). A TTL of minutes ensures hot entries are periodically re-warmed from DB regardless of attacker activity, and cold attacker-injected entries expire quickly.

2. **Add application-level rate limiting.** Implement per-IP request rate limiting inside the Rosetta Go service (e.g., using `golang.org/x/time/rate` or a token-bucket middleware) so the protection is not deployment-topology-dependent.

3. **Extend the `Cache` config type** to include a `TTL` field so operators can tune expiry alongside capacity:
   ```go
   type Cache struct {
       MaxSize int           `yaml:"maxSize"`
       TTL     time.Duration `yaml:"ttl"`
   }
   ```

4. **Consider a frequency-based eviction policy** (LFU instead of LRU) so that genuinely hot entries are not displaced by a burst of cold ones.

### Proof of Concept

```
# 1. Enumerate valid entity IDs (publicly available via mirror node REST API)
#    e.g., accounts 0.0.1000000 through 0.0.2000000

# 2. For each block index B from 1 to N, send:
POST /block
{
  "network_identifier": { "blockchain": "Hedera", "network": "mainnet" },
  "block_identifier": { "index": B }
}

# Each response triggers updateOperationAccountAlias for all operations in block B.
# Operations referencing entity IDs not yet in the cache cause GetAccountAlias DB queries.
# After ~524288 unique IDs are injected, the LRU cache is fully saturated with cold entries.
# Subsequent legitimate /block requests for recently-active accounts all miss the cache
# and each issue a SELECT against the entity table.

# 3. Measure DB query rate before and after the injection phase.
#    Expected: >30% increase in SELECT count on the entity table within the attack window.
```

```go
func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

	return nil
}
```
