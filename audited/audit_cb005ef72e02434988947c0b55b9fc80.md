### Title
Unauthenticated Cache-Bypass DoS via Timestamp-Parameterized Token Info Queries Triggering Expensive 3-Way UNION ALL History Queries

### Summary
The `GET /tokens/:tokenId?timestamp=lte:<ts>` endpoint invokes `buildHistoryQuery()` three times per request, each producing a UNION ALL across a live table and its `_history` counterpart. Because the response cache key is derived from the full URL (including the timestamp value), an unprivileged attacker can trivially bypass the cache by varying the timestamp or tokenId across requests, forcing every request to execute the expensive 6-table query against the database with no application-level rate limiting in the REST service.

### Finding Description

**Code path:**

`getTokenInfoRequest()` (lines 548–565) is the handler for `GET /tokens/:tokenId`: [1](#0-0) 

When any timestamp filter is present, `extractSqlFromTokenInfoRequest()` (lines 482–528) calls `buildHistoryQuery()` three separate times — once each for the `token`/`token_history`, `entity`/`entity_history`, and `custom_fee`/`custom_fee_history` table pairs: [2](#0-1) 

Each `buildHistoryQuery()` call (lines 530–546) emits a subquery of the form:

```sql
(SELECT ... FROM <table> WHERE <conditions>)
UNION ALL
(SELECT ... FROM <table>_history WHERE <conditions>
 ORDER BY lower(timestamp_range) DESC LIMIT 1)
ORDER BY modified_timestamp DESC LIMIT 1
``` [3](#0-2) 

The three subqueries are then JOINed together, meaning a single request with a timestamp filter touches **six tables** (three live + three history).

**Cache bypass:**

The response cache key is the MD5 of `req.originalUrl`: [4](#0-3) 

The cache TTL for `GET /tokens/{id}` is only 5 seconds: [5](#0-4) 

Because the timestamp value is part of the URL, every distinct `?timestamp=lte:<ts>` value produces a unique cache key. An attacker cycling through different timestamps (e.g., `lte:1000000000`, `lte:1000000001`, …) or different valid tokenIds generates a cache miss on every request, forcing each one to execute the full 6-table query.

**No application-level rate limiting:**

The REST Node.js service has no per-IP or global rate-limiting middleware. The `authHandler` only grants elevated limits to authenticated users; unauthenticated requests are not throttled: [6](#0-5) 

The throttle mechanisms found in the codebase (`ThrottleManagerImpl`, `ThrottleConfiguration`) belong exclusively to the `web3` Java service and do not apply to the REST service: [7](#0-6) 

The Traefik middleware template for the REST chart exists but is conditional on `global.middleware` being enabled in the Helm values — it is not a guaranteed control: [8](#0-7) 

Additionally, the token-level in-memory cache (`TokenService.putTokenCache`) is only populated when **no** filters are present, so timestamp-filtered requests never benefit from it: [9](#0-8) 

### Impact Explanation
An attacker can sustain a high rate of DB-expensive queries against the mirror node's PostgreSQL backend with no authentication, no rate limiting, and no effective caching. Each request scans up to six tables (three of which are unbounded history tables). Under sustained load with many unique (tokenId, timestamp) pairs, DB connection pool exhaustion or query queue saturation will degrade or deny service for all legitimate API consumers. The impact is availability degradation (no funds at risk), consistent with the "griefing" severity classification.

### Likelihood Explanation
The attack requires zero privileges, zero on-chain interaction, and only knowledge of valid tokenIds (publicly enumerable via `GET /tokens`). The timestamp parameter accepts any nanosecond-precision value, giving an attacker an effectively unbounded keyspace to defeat caching. The attack is trivially scriptable with a single HTTP client in a loop and is repeatable indefinitely.

### Recommendation
1. **Add application-level rate limiting** to the REST Node.js service (e.g., `express-rate-limit`) scoped per IP for the `GET /tokens/:tokenId` route, independent of Traefik deployment configuration.
2. **Extend cache coverage to timestamp-filtered requests**: use a coarser cache key (e.g., round the timestamp to the nearest consensus interval) so nearby timestamp values share a cache entry.
3. **Add a DB-level statement timeout** for history queries to bound worst-case query duration.
4. **Ensure Traefik `rateLimit` middleware is enabled by default** in the REST chart's `values.yaml` rather than being opt-in.

### Proof of Concept
```bash
# Enumerate a valid tokenId first
TOKEN_ID="0.0.1234"
BASE_URL="https://<mirror-node>/api/v1/tokens/${TOKEN_ID}"

# Flood with unique timestamps to bypass cache, each triggering the 3x UNION ALL query
for ts in $(seq 1000000000 1 1000010000); do
  curl -s "${BASE_URL}?timestamp=lte:${ts}" &
done
wait
```

Each request hits a unique cache key, executes the 6-table UNION ALL query, and returns before the 5-second TTL can provide any protection. Running this from multiple IPs or with multiple tokenIds amplifies the DB load proportionally.

const buildHistoryQuery = (selectColumns, conditions, tableName, tableAlias) => {
  return `
   (select ${selectColumns}
    from
    (
      (select ${selectColumns}, lower(${tableAlias}.timestamp_range) as modified_timestamp
        from ${tableName} ${tableAlias}
        where ${conditions})
      union all
      (select ${selectColumns}, lower(${tableAlias}.timestamp_range) as modified_timestamp
      from ${tableName}_history ${tableAlias}
        where ${conditions} 
        order by lower(${tableAlias}.timestamp_range) desc limit 1)
      order by modified_timestamp desc limit 1
    ) as ${tableAlias})
    `;
};
```
