### Title
Silent Overwrite of Lower Bound Allows Duplicate GTE Parameters to Omit Pending Airdrop Records

### Summary
In `getPendingAirdrops()`, the `Bound` constructor iterates over all supplied `senderIds` parameters and unconditionally overwrites `lower` on each iteration for any lower-bound operator. When an unprivileged user supplies two `GTE` parameters (e.g., `sender.id=gte:0.0.1&sender.id=gte:0.0.1000`), the first value is silently discarded and the query executes with only the second (higher) bound, causing all pending airdrop records for senders 0.0.1–0.0.999 to be omitted from the response. The `verifySingleOccurrence()` guard that would catch this is never invoked for airdrop requests.

### Finding Description

**Exact code path:**

In `Bound.java`, the constructor loop at lines 47–54:

```java
for (var param : params) {
    if (param.hasLowerBound()) {
        lower = param;          // unconditional overwrite
    } else if (param.hasUpperBound()) {
        upper = param;
    }
    cardinality.merge(param.operator(), 1, Math::addExact);
}
``` [1](#0-0) 

When `params = [GTE:1, GTE:1000]`, the loop sets `lower = GTE:1` then immediately overwrites it with `lower = GTE:1000`. The `cardinality` map correctly records `{GTE: 2}`, but `lower` retains only the last value.

**The guard that should prevent this — `verifySingleOccurrence()` — is never called for airdrop requests.** It exists in `Bound.java`:

```java
public void verifySingleOccurrence() {
    verifySingleOccurrence(RangeOperator.EQ);
    verifySingleOccurrence(RangeOperator.GT, RangeOperator.GTE);
    verifySingleOccurrence(RangeOperator.LT, RangeOperator.LTE);
}
``` [2](#0-1) 

`NftAllowanceServiceImpl` correctly calls it via `verifyRangeId()`: [3](#0-2) 

But `TokenAirdropServiceImpl.getAirdrops()` performs no such validation — it goes directly to the repository: [4](#0-3) 

**Why the existing range check doesn't catch it:**

The only guard in the `Bound` constructor is `adjustedLower > adjustedUpper` (line 58), which checks for an inverted range. With two GTE params and no upper bound, `adjustedUpper = Long.MAX_VALUE`, so the check never fires regardless of the GTE values supplied. [5](#0-4) 

**Entry point in the controller:**

`@Size(max = 2)` on `senderIds` explicitly permits two parameters, so Spring's binding accepts both GTE values without error: [6](#0-5) 

The `Bound` is then constructed with `primarySortField = true` and the two-element array: [7](#0-6) 

### Impact Explanation
An unprivileged external caller can craft a request that silently narrows the effective lower bound of the sender filter. Records for all senders between the discarded (lower) GTE value and the surviving (higher) GTE value are omitted from the API response. This constitutes a data-integrity / information-omission issue: clients relying on this endpoint for auditing or reconciliation of pending airdrops will receive an incomplete, misleading result set without any error signal. The severity is moderate — no data is written or deleted, but the mirror node's read-path correctness guarantee is violated for an unauthenticated caller.

### Likelihood Explanation
No authentication or privilege is required. The endpoint is public. The `@Size(max = 2)` annotation is documented behavior that explicitly invites two-parameter usage (it is the intended mechanism for supplying both a lower and upper bound). An attacker or a misconfigured client can trigger this with a single crafted HTTP GET request, repeatably and at will.

### Recommendation
Add a validation step in `TokenAirdropServiceImpl.getAirdrops()` (or in a shared helper) equivalent to what `NftAllowanceServiceImpl` does — call `verifySingleOccurrence()` (and `verifyEqualOrRange()`) on each `Bound` before passing the request to the repository. Specifically:

```java
public Collection<TokenAirdrop> getAirdrops(TokenAirdropRequest request) {
    verifyBound(request.getEntityIds());
    verifyBound(request.getTokenIds());
    verifyBound(request.getSerialNumbers());
    var id = entityService.lookup(request.getAccountId());
    return repository.findAll(request, id);
}

private void verifyBound(Bound bound) {
    bound.verifyUnsupported(RangeOperator.NE);
    bound.verifySingleOccurrence();
    bound.verifyEqualOrRange();
}
```

Alternatively, move the duplicate-operator check into the `Bound` constructor itself so it is enforced universally, rather than relying on each service to remember to call it.

### Proof of Concept

**Precondition:** Mirror node is running; pending airdrops exist for receiver account `0.0.500` from senders including `0.0.1` through `0.0.1500`.

**Request:**
```
GET /api/v1/accounts/0.0.500/airdrops/pending?sender.id=gte:0.0.1&sender.id=gte:0.0.1000
```

**Expected (correct) behavior:** Returns pending airdrops from all senders `>= 0.0.1`.

**Actual behavior:**
1. Spring binds `senderIds = [GTE:1, GTE:1000]` (both accepted by `@Size(max=2)`).
2. `Bound` constructor iterates: sets `lower = GTE:1`, then overwrites `lower = GTE:1000`.
3. `verifySingleOccurrence()` is never called; no exception is raised.
4. Repository query executes with `sender_account_id >= 1000`.
5. Response omits all pending airdrops from senders `0.0.1` through `0.0.999`.

**Verification:** Compare response to `?sender.id=gte:0.0.1` (single param) — the two-GTE request returns a strict subset, silently dropping records.
