Audit Report

## Title
Cache Key Collision via `Arrays.hashCode()` Truncation Enables Wrong Entity Return in `CACHE_NAME_EVM_ADDRESS` and `CACHE_NAME_ALIAS`

## Summary
`findByEvmAddressAndDeletedIsFalse()` and `findByEvmAddressOrAliasAndDeletedIsFalse()` use `@spelHelper.hashCode(#alias)` as their Spring `@Cacheable` key. This resolves to `Arrays.hashCode(byte[])` — a 32-bit polynomial hash — applied to a 160-bit (20-byte) EVM address. Because the key space is only 2³² while the input space is 2¹⁶⁰, an attacker can precompute a crafted address `B` that collides with a real entity address `A`, poison the cache, and cause the mirror node to return Entity A's data for all subsequent queries for address `B`.

## Finding Description

**`SpelHelper.hashCode`** is defined as:

```java
// common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);   // 32-bit int
}
```

This integer is used verbatim as the Spring cache key for both affected methods:

```java
// web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java
@Cacheable(
    cacheNames = CACHE_NAME_EVM_ADDRESS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",   // 32-bit key for 160-bit input
    unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);

@Cacheable(
    cacheNames = CACHE_NAME_ALIAS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

The cache is a Caffeine instance configured via `cacheProperties.getEntity()`:

```java
// web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java
@Bean(CACHE_MANAGER_ENTITY)
CacheManager cacheManagerEntity() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheNames(Set.of(CACHE_NAME, CACHE_NAME_EVM_ADDRESS, CACHE_NAME_ALIAS));
    caffeineCacheManager.setCacheSpecification(cacheProperties.getEntity());
    return caffeineCacheManager;
}
```

**Root cause:** The cache key is a 32-bit integer derived from a 160-bit input. The pigeonhole principle guarantees collisions exist. The `unless = "#result == null"` guard only prevents caching of empty results — it does not prevent a cached non-null result from being returned for a colliding key. There is no post-cache verification that the returned entity's actual EVM address matches the queried address.

## Impact Explanation
Any caller querying a crafted address `B` (which collides with real entity `A`'s hash) receives Entity A's data: wrong bytecode, wrong balance, wrong storage. In the mirror node's `eth_call` / `eth_estimateGas` context, the EVM executes with Entity A's state for address B. This produces silently incorrect results for all users querying `B` for the full TTL of the cache entry. Contracts or off-chain systems relying on mirror node query results for verification receive wrong data with no error signal. The same flaw exists in `CACHE_NAME_ALIAS` via `findByEvmAddressOrAliasAndDeletedIsFalse`.

## Likelihood Explanation
The attack requires no credentials, no on-chain transactions, and no special privileges — only the ability to send JSON-RPC or REST requests to the public mirror node API (`/api/v1/contracts/call`). The `to` field in `eth_call` is fully attacker-controlled, so any crafted 20-byte address can be submitted. Collision search is offline: given a target hash `h = Arrays.hashCode(A)`, an attacker iterates candidate 20-byte arrays until one matches — expected ~2³² attempts in the worst case, but the birthday paradox means ~65,536 candidates yield ~50% probability of a collision among a set of known addresses. On commodity hardware this completes in milliseconds. The poisoned cache entry persists for the full configured TTL, serving wrong data to all users during that window. The attack is repeatable and deterministic.

## Recommendation
Replace the 32-bit `Arrays.hashCode()` cache key with the full byte array itself (or a collision-resistant representation). Spring's default key generation for a single `byte[]` argument uses `Arrays.equals`-compatible `SimpleKey` wrapping, which preserves the full value. The simplest fix is to remove the custom `key = "@spelHelper.hashCode(#alias)"` expression entirely and let Spring use its default key generator, which wraps the `byte[]` in a `SimpleKey` and uses `Arrays.equals` for equality — eliminating the truncation entirely. Alternatively, encode the byte array as a hex string key: `key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"`.

## Proof of Concept

```java
// Find two 20-byte arrays with the same Arrays.hashCode
byte[] addressA = /* known real entity EVM address */;
int targetHash = Arrays.hashCode(addressA);

byte[] addressB = new byte[20];
Random rng = new Random();
while (true) {
    rng.nextBytes(addressB);
    if (Arrays.hashCode(addressB) == targetHash) break;
    // Typically found in < 2^32 iterations; birthday attack among
    // known addresses finds collision in ~65K candidates
}

// Step 1: Seed the cache with Entity A
// POST /api/v1/contracts/call  { "to": "<addressA hex>", ... }
// -> Entity A fetched from DB, stored in cache under key targetHash

// Step 2: Query the colliding address B
// POST /api/v1/contracts/call  { "to": "<addressB hex>", ... }
// -> Spring cache intercepts, finds key targetHash already populated,
//    returns Entity A — without querying the DB.
// -> EVM executes with Entity A's bytecode/state for address B.
```
