# Audit Report

## Title
Gas Throttle Bypass via `eth_estimateGas` Binary Search Loop — Up to 21x EVM Execution Budget Multiplication

## Summary
`ContractExecutionService.estimateGas()` performs one initial EVM execution with `estimate=false` (which triggers `restoreGasToBucket()` to account for unused gas), then invokes `BinaryGasEstimator.search()` with `estimate=true` for up to 20 additional EVM executions. Because `doProcessCall()` skips `restoreGasToBucket()` entirely when `estimate=true`, those 20 executions consume no throttle tokens. Any unprivileged caller of `eth_estimateGas` therefore obtains up to 21 full EVM executions for the throttle cost of a single execution's actual gas usage.

## Finding Description

**Throttle deduction flow:**

`ThrottleManagerImpl.throttle()` pre-debits the full `request.getGas()` (up to 15,000,000) from the `gasLimitBucket` once per HTTP request at the controller layer:

```java
// ContractController.java:40
throttleManager.throttle(request);
// ThrottleManagerImpl.java:40
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
``` [9](#0-8) [10](#0-9) 

**Initial call — throttle accounting fires normally:**

`estimateGas()` calls `callContract(params, context)` → `doProcessCall(params, params.getGas(), false)`. With `estimate=false`, the `finally` block calls `restoreGasToBucket()`, returning unused gas to the bucket. Net throttle cost = actual `gasUsed` from the initial execution. [5](#0-4) 

**Binary search loop — throttle accounting is skipped entirely:**

`binaryGasEstimator.search()` is called with `gas -> doProcessCall(params, gas, true)`. The loop runs up to `maxGasEstimateRetriesCount` (default 20) iterations:

```java
// BinaryGasEstimator.java:35
while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount()) {
    ...
    final var transactionResult = safeCall(mid, call); // calls doProcessCall(..., true)
``` [11](#0-10) 

Each iteration calls `doProcessCall` with `estimate=true`. The `finally` block is gated on `!estimate`:

```java
// ContractCallService.java:127-135
} finally {
    if (!estimate) {
        restoreGasToBucket(result, params.getGas());
        ...
    }
}
``` [5](#0-4) 

`restoreGasToBucket()` is never called for any of the 20 binary search EVM executions. No tokens are deducted or restored for them. [12](#0-11) 

**Root cause:** The throttle bucket is pre-debited once per request. `restoreGasToBucket()` is the sole mechanism for accounting for actual gas consumed during execution. By passing `estimate=true`, the binary search loop completely bypasses this accounting for all 20 iterations.

**Failed assumption:** The design assumes the binary search iterations are negligible overhead. In reality, each iteration is a full EVM execution at a gas value between `gasUsedByInitialCall` and `params.getGas()` (up to 15M), and none are charged to the throttle.

## Impact Explanation

An attacker submitting a single `eth_estimateGas` request with the maximum gas limit (15,000,000) triggers up to 21 full EVM executions (1 initial + 20 binary search) while the throttle bucket is debited only for the gas used in the initial call.

- **Throttle bypass / resource exhaustion**: Sustained `eth_estimateGas` spam with a computationally expensive contract payload saturates the EVM execution thread pool at a fraction of the throttle cost of equivalent `eth_call` spam. With `gasLimitRefundPercent = 100` (default), the net throttle cost per `eth_estimateGas` request equals only the gas used by the initial call, while 20 additional executions run uncharged.
- **Amplification factor**: Up to 20x confirmed by `maxGasEstimateRetriesCount = 20`. [7](#0-6) [13](#0-12) 

## Likelihood Explanation

- **No privileges required**: Any caller of the public `eth_estimateGas` JSON-RPC endpoint can trigger this.
- **Trivially repeatable**: A loop of `eth_estimateGas` calls with a gas-heavy contract payload is sufficient.
- **No special knowledge needed**: The attacker only needs to supply a gas-heavy contract call to `eth_estimateGas`.

## Recommendation

Apply throttle accounting to binary search iterations. Two options:

1. **Charge each binary search iteration proportionally**: In `doProcessCall`, when `estimate=true`, still call `restoreGasToBucket()` using the `estimatedGas` parameter (the mid-point gas value passed to each iteration) rather than `params.getGas()`. This ensures each iteration's gas consumption is reflected in the throttle bucket.

2. **Pre-deduct a multiplier for estimate requests**: In `ContractController.call()`, when `request.isEstimate()` is true, multiply the gas deduction by a factor (e.g., `maxGasEstimateRetriesCount + 1`) to account for the full binary search budget upfront.

Option 1 is more precise. The key change is removing the `!estimate` guard from the `restoreGasToBucket()` call, or adding a separate throttle-restore path for estimate iterations that uses `estimatedGas` as the gas limit parameter. [14](#0-13)

    protected final EvmTransactionResult doProcessCall(
            CallServiceParameters params, long estimatedGas, boolean estimate) throws MirrorEvmTransactionException {
        EvmTransactionResult result = null;
        var status = ResponseCodeEnum.SUCCESS.toString();

        try {
            result = transactionExecutionService.execute(params, estimatedGas);

            if (!estimate) {
                validateResult(result, params);
            }
        } catch (IllegalStateException | IllegalArgumentException e) {
            throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
        } catch (MirrorEvmTransactionException e) {
            // This result is needed in case of exception to be still able to call restoreGasToBucket method
            result = e.getResult();
            status = e.getMessage();
            throw e;
        } finally {
            if (!estimate) {
                restoreGasToBucket(result, params.getGas());

                // Only record metric if EVM is invoked and not inside estimate loop
                if (result != null) {
                    updateMetrics(params, result.gasUsed(), 1, status);
                }
            }
        }
        return result;
    }

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
