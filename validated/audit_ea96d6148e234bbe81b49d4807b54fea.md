Audit Report

## Title
Unbounded In-Memory Accumulation of EVM Memory Snapshots in `getContractOpcodes()` Leads to Node OOM

## Summary
When `memory=true` is passed to `getContractOpcodes()`, the tracer captures a full snapshot of the entire EVM memory at every single opcode step and accumulates all snapshots in an unbounded `ArrayList` before serialization. No cap exists on the number of opcodes stored, the size of each memory snapshot, or the total response payload. An unprivileged user can trigger this against any existing high-gas historical transaction, causing the mirror node to exhaust its heap.

## Finding Description

**Verified code path:**

`OpcodesController.getContractOpcodes()` accepts `memory=true` from any caller with no authentication: [1](#0-0) 

The only guards are: (1) a gzip header check (reduces network bytes, not server-side heap), and (2) a token-bucket rate limiter: [2](#0-1) 

The rate limiter uses `tryConsume(1)` — non-blocking. After 1 second the bucket refills, so if a request takes 60 s to process, up to ~60 concurrent memory-exhausting requests can be in flight simultaneously: [3](#0-2) 

`OpcodeServiceImpl.processOpcodeCall()` initialises the opcode list with an initial capacity hint of `gas/3` (up to 5 million for a 15 M-gas transaction) but imposes **no hard upper bound**: [4](#0-3) 

`OpcodeContext.addOpcodes()` appends to an unbounded `ArrayList` with no size check: [5](#0-4) 

`AbstractOpcodeTracer.captureMemory()` reads the **entire** EVM memory at every opcode step with no size limit: [6](#0-5) 

**Why existing checks fail:**

- `validateAcceptEncodingHeader`: enforces gzip on the response wire, not on the in-process heap.
- `throttleOpcodeRequest()`: token-bucket rate limit of 1/s; does not bound concurrency or per-request heap.
- `maxGasLimit = 15_000_000`: caps gas for *new* contract calls via `ContractController.validateContractMaxGasLimit()`, but the opcode endpoint reads gas directly from the stored historical `ContractResult`/`EthereumTransaction` record with no cap applied: [7](#0-6) [8](#0-7) 

- `requestTimeout = 10s` (`Web3Properties`): enforced only via a Hibernate `StatementInspector` that fires before SQL queries — **not** during pure EVM computation. A 15 M-gas replay with memory snapshots runs entirely in-JVM with no SQL calls, so the timeout never fires: [9](#0-8) 

**Root cause:** Gas cost bounds total EVM *work*, but once memory is expanded it stays expanded for the rest of the call frame. Cheap opcodes (3 gas each) executed after a large memory expansion each trigger a full memory snapshot. With 15 M gas, ~1 M cheap opcodes after a ~1,000-word expansion produces hundreds of gigabytes of `String` objects before any serialisation.

## Impact Explanation
A single well-chosen request can allocate tens of gigabytes of heap inside the JVM before the response is serialised. This causes an `OutOfMemoryError`, crashing the web3 mirror-node process. Because the rate limiter refills every second and does not track in-flight requests, a sustained stream of such requests keeps the node permanently OOM-cycling, disrupting all contract-call and opcode-trace services hosted by the node.

## Likelihood Explanation
The endpoint requires only that `hiero.mirror.web3.opcode.tracer.enabled=true` (operators enable it for debugging). [10](#0-9) 

No account, API key, or privileged role is needed. The attacker only needs to identify one historical transaction with high gas usage and significant memory expansion — such transactions are publicly visible on-chain. The exploit is fully repeatable at 1 req/s per attacker IP, and multiple IPs can be used simultaneously since the bucket is global but non-concurrent.

## Recommendation

1. **Cap total opcode count**: Enforce a hard limit (e.g., 500,000 opcodes) in `OpcodeContext.addOpcodes()` and throw a bounded exception when exceeded. [5](#0-4) 

2. **Cap memory snapshot size**: In `AbstractOpcodeTracer.captureMemory()`, limit `frame.memoryWordSize()` to a configurable maximum (e.g., 1,024 words / 32 KB) and truncate or skip beyond that. [11](#0-10) 

3. **Apply `maxGasLimit` to historical replay**: Reject opcode requests where the stored gas limit exceeds `evmProperties.getMaxGasLimit()`. [7](#0-6) 

4. **Enforce a concurrency limit**: Replace the token-bucket rate limiter with a semaphore-based concurrency limiter (e.g., max 1–2 in-flight opcode requests at a time). [3](#0-2) 

5. **Enforce a wall-clock timeout on EVM execution**: The existing `requestTimeout` only fires on SQL queries. Add an interrupt-based or `Future`-based timeout that also bounds pure EVM computation time. [9](#0-8) 

## Proof of Concept

```
# 1. Find a historical high-gas transaction with large memory expansion
#    (e.g., a complex DeFi deployment visible on-chain with gasUsed ~15M)
TX_HASH="0x<high_gas_tx_hash>"

# 2. Send a single request with memory=true
curl -H "Accept-Encoding: gzip" \
     "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?memory=true" \
     --output /dev/null

# The mirror node JVM will begin allocating:
#   N_opcodes × memory_words × 66 bytes of String objects
# For a 15M-gas tx with 1000-word memory expansion and ~1M cheap opcodes:
#   ~1,000,000 × 1,000 × 66 bytes ≈ 66 GB heap → OutOfMemoryError → JVM crash

# 3. Repeat at 1 req/s (rate limit refills) to keep the node OOM-cycling
while true; do
  curl -H "Accept-Encoding: gzip" \
       "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?memory=true" \
       --output /dev/null &
  sleep 1
done
```

    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
    }
