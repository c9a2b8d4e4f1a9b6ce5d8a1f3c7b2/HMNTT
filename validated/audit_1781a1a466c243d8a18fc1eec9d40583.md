Audit Report

## Title
Missing Alias Resolution for `autoRenewAccount` in `ConsensusCreateTopicTransactionHandler.doUpdateEntity()` Causes Incorrect Mirror Node Records

## Summary
`ConsensusCreateTopicTransactionHandler.doUpdateEntity()` resolves `autoRenewAccount` via `EntityId.of(transactionBody.getAutoRenewAccount())`, which only reads `getAccountNum()` from the protobuf `AccountID`. When the account is specified as an alias-based `AccountID`, `getAccountNum()` returns `0`, causing `EntityId.EMPTY` (id=0) to be stored as `auto_renew_account_id`. Every analogous sibling handler uses `entityIdService.lookup()` for this field, which correctly resolves aliases. `ConsensusCreateTopicTransactionHandler` is the sole outlier.

## Finding Description

**Exact code path:**

`ConsensusCreateTopicTransactionHandler.doUpdateEntity()`, lines 40–44:

```java
if (transactionBody.hasAutoRenewAccount()) {
    var autoRenewAccountId = EntityId.of(transactionBody.getAutoRenewAccount()); // BUG
    entity.setAutoRenewAccountId(autoRenewAccountId.getId());
    recordItem.addEntityId(autoRenewAccountId);
}
``` [1](#0-0) 

`EntityId.of(AccountID)` is defined as:

```java
public static EntityId of(AccountID accountID) {
    return of(accountID.getShardNum(), accountID.getRealmNum(), accountID.getAccountNum());
}
``` [2](#0-1) 

When `AccountID` carries an alias (i.e., `accountID.getAccountCase() == ALIAS`), `getAccountNum()` returns `0`. The `encode()` path then produces `id = 0`, which maps to `EntityId.EMPTY`: [3](#0-2) [4](#0-3) 

**Root cause:** The handler bypasses `entityIdService.lookup()`, the only code path that performs alias-to-numeric-ID resolution via cache and DB lookup. The failed assumption is that `autoRenewAccount` will always be a numeric `AccountID`; the HAPI protobuf `AccountID` supports both `accountNum` and `alias` forms.

**Contrast with all sibling handlers** — every other handler that processes `autoRenewAccount` uses `entityIdService.lookup()`:

- `ConsensusUpdateTopicTransactionHandler.doUpdateEntity()` lines 43–54: [5](#0-4) 

- `ContractCreateTransactionHandler.doUpdateEntity()` lines 75–85: [6](#0-5) 

- `TokenCreateTransactionHandler.doUpdateEntity()` lines 61–71: [7](#0-6) 

`ConsensusCreateTopicTransactionHandler` is the sole outlier among all create/update handlers that handle `autoRenewAccount`.

**Why existing checks fail:** `transactionBody.hasAutoRenewAccount()` only tests field presence (protobuf `has*` semantics), not whether the `AccountID` is in numeric or alias form. There is no guard against alias-based input anywhere in this handler.

## Impact Explanation

The mirror node's `entity` table stores `auto_renew_account_id = 0` for any topic created with an alias-based `autoRenewAccount`. REST API consumers querying topic details receive a wrong or null `auto_renew_account` field. The entity transaction record for the actual `autoRenewAccount` is absent from the mirror node export, breaking downstream analytics and audit trails that rely on entity transaction tracking. The corruption is silent (no error is thrown, no log is emitted) and permanent for the affected topic row.

## Likelihood Explanation

Any unprivileged user with a Hedera account that has an alias (standard for ECDSA-key accounts, which are common) can trigger this with a single `ConsensusCreateTopic` transaction. No special permissions, admin keys, or coordination are required. The HAPI SDK allows specifying `AccountID` by alias. The attack is trivially repeatable for every new topic creation.

## Recommendation

Replace the direct `EntityId.of()` call with `entityIdService.lookup()`, consistent with all sibling handlers. The fix should mirror the pattern used in `ConsensusUpdateTopicTransactionHandler`:

```java
if (transactionBody.hasAutoRenewAccount()) {
    entityIdService
        .lookup(transactionBody.getAutoRenewAccount())
        .ifPresentOrElse(
            entityId -> {
                entity.setAutoRenewAccountId(entityId.getId());
                recordItem.addEntityId(entityId);
            },
            () -> Utility.handleRecoverableError(
                "Invalid autoRenewAccountId at {}", recordItem.getConsensusTimestamp()));
}
``` [5](#0-4) 

## Proof of Concept

1. Hold account `0.0.X` with an ECDSA alias.
2. Submit `ConsensusCreateTopic` with `autoRenewAccount` set as `AccountID{alias: <alias_bytes>}` (alias form, not numeric form).
3. Consensus node validates and accepts the transaction; the record stream preserves the alias form in the transaction body.
4. Mirror node importer calls `doUpdateEntity()`. `EntityId.of(aliasAccountID)` calls `getAccountNum()` → returns `0` → returns `EntityId.EMPTY`.
5. `entity.setAutoRenewAccountId(0)` — the topic entity row records `auto_renew_account_id = 0`.
6. `recordItem.addEntityId(EntityId.EMPTY)` — the entity transaction record for `0.0.X` is never emitted.
7. Query the mirror node REST API for the topic: `auto_renew_account` is `null` or `0.0.0` instead of `0.0.X`.
