**Audit Report**

## Title
Throttle Token Leak via Odd-Length Hex `data` Field Bypassing `@Hex` Validator

## Summary
The `@Hex` validator on `ContractCallRequest.data` does not enforce even-length hex strings. Odd-length values (e.g., `"0x1"`, `"1aa"`) pass validation, reach `hexToBytes()` in `BytesDecoder`, and cause an `IllegalArgumentException`. Because `ContractController.call()` only restores throttle tokens on `InvalidParametersException`, the gas tokens consumed by `throttleManager.throttle()` are never returned, allowing an attacker to drain the throttle bucket without performing any EVM work.

## Finding Description

**1. `@Hex` validator does not enforce even-length.**
The compiled pattern is `^(0x)?[0-9a-fA-F]{0,1048576}$`. It counts individual hex characters with no parity constraint, so `"0x1"`, `"1aa"`, `"abc"` all pass. [1](#0-0) 

**2. `data` field uses this validator with defaults.** [2](#0-1) 

**3. The try/catch in `constructServiceParameters()` is dead code for its stated purpose.**
The `try` block wraps only `request.getData()`, a plain Lombok getter that never throws. The catch block — which would convert the exception to `InvalidParametersException` — can never be reached by the odd-length path. [3](#0-2) 

**4. The actual dangerous call, `hexToBytes(data)`, is at line 80, outside the try/catch.** [4](#0-3) 

**5. `hexToBytes()` throws `IllegalArgumentException` for odd-length input.**
`org.apache.commons.codec.binary.Hex.decodeHex` rejects odd-length strings with `DecoderException("Odd number of characters")`, which is re-wrapped as `IllegalArgumentException`. [5](#0-4) 

**6. `call()` only restores tokens for `InvalidParametersException`.**
`throttleManager.throttle(request)` runs at line 40, consuming both a rate-limit token and gas tokens. The `IllegalArgumentException` from `hexToBytes()` propagates past the catch at line 46, so `throttleManager.restore(request.getGas())` is never called. [6](#0-5) 

**7. The existing test confirms the 400 response but does not assert token restoration.** [7](#0-6) 

## Impact Explanation
Each request with an odd-length `data` value permanently consumes gas tokens from `gasLimitBucket` (defaulting to 15,000,000 gas per request) without performing any EVM execution. Once the bucket is exhausted, all legitimate callers to `/api/v1/contracts/call` are rate-limited or rejected, constituting a denial-of-service. The `restore()` method only adds back gas tokens and is never invoked on this path. [8](#0-7) 

## Likelihood Explanation
The attack requires only an unauthenticated HTTP POST with a crafted JSON body (e.g., `{"data":"0x1"}`). No account, no on-chain state, and no special privilege is needed. The odd-length bypass is trivially discoverable by reading the validator regex or by fuzzing the `data` field. It is fully automatable in a tight loop.

## Recommendation
1. **Fix the root cause in `HexValidator`**: Enforce even-length by changing the pattern to require an even number of hex characters, e.g., `^(0x)?([0-9a-fA-F]{2}){0,524288}$`.
2. **Fix the misplaced try/catch in `constructServiceParameters()`**: Move `hexToBytes(data)` inside the try block so that `DecoderException`/`IllegalArgumentException` is caught and re-thrown as `InvalidParametersException`, which the `call()` method already handles correctly for token restoration.
3. **Add a test asserting token restoration**: Extend `callBadRequestWithInvalidHexData` to verify that `throttleManager.restore()` is called when odd-length data is submitted.

## Proof of Concept
```
POST /api/v1/contracts/call
Content-Type: application/json

{"data":"0x1","gas":15000000}
```
Repeat in a tight loop. Each request consumes 15,000,000 gas tokens from the bucket without restoration. Once the bucket is drained, subsequent legitimate requests receive throttle rejections.
