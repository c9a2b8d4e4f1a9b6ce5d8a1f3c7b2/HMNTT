# Audit Report

## Title
Permanent Cache Poisoning of `recordFileEarliest` via `Optional.empty()` Bypass of `unless` Guard

## Summary
`RecordFileRepository.findEarliest()` is annotated with `unless = "#result == null"`, but the method returns `Optional<RecordFile>`. When the database contains no record files, the return value is `Optional.empty()` — which is not `null` — so the guard is bypassed and the empty result is stored permanently in a no-TTL, single-entry Caffeine cache. Any request that triggers this path before the first record file is ingested permanently poisons the cache, causing all subsequent `BlockType.EARLIEST` queries to return empty until the service is restarted.

## Finding Description
**Cache configuration** — `EvmConfiguration.java`, lines 190–197:
```java
@Bean(CACHE_MANAGER_RECORD_FILE_EARLIEST)
CacheManager cacheManagerRecordFileEarliest() {
    final var caffeine = Caffeine.newBuilder().maximumSize(1).recordStats();
    // No expireAfterWrite / expireAfterAccess — entries never expire
    ...
}
``` [1](#0-0) 

**Flawed cache guard** — `RecordFileRepository.java`, line 23:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_EARLIEST,
           unless = "#result == null")
Optional<RecordFile> findEarliest();
``` [2](#0-1) 

Spring's `@Cacheable` `unless` SpEL expression evaluates against the actual return value. Since the method signature is `Optional<RecordFile>`, an empty-table query returns `Optional.empty()`. `Optional.empty() != null`, so `unless = "#result == null"` evaluates to `false`, and the empty Optional **is written into the cache**.

The cache has `maximumSize(1)` and **no TTL**. Once `Optional.empty()` occupies the single slot, it stays there indefinitely. Every subsequent call to `findEarliest()` hits the cache and returns `Optional.empty()` without ever querying the database again.

**Call path:**
```
HTTP request (block = "earliest")
  → RecordFileServiceImpl.findByBlockType(BlockType.EARLIEST)   [lines 20–21]
  → RecordFileRepository.findEarliest()                          [line 23–25]
  → DB returns empty → Optional.empty() cached permanently
``` [3](#0-2) 

**Contrast with `findLatest()`**: that cache uses `expireAfterWrite(500, TimeUnit.MILLISECONDS)`, so any poisoned entry self-heals in 500 ms. The `findEarliest()` cache has no such safety valve. [5](#0-4) 

## Impact Explanation
After the cache is poisoned, every call to `eth_getBlockByNumber("earliest", ...)`, `eth_call` with block tag `earliest`, or any other API path that resolves `BlockType.EARLIEST` returns an empty/not-found response for all users. The service must be restarted to recover. This is a targeted, permanent denial-of-service for the earliest-block feature.

## Likelihood Explanation
The exploit requires the attacker to send one HTTP request during the window between service startup and the first record file being ingested into the database. This window is predictable (mirror nodes are publicly observable; restart events are visible via monitoring gaps or version-upgrade announcements). No credentials, special permissions, or on-chain funds are required — a single unauthenticated HTTP request suffices. The attack is repeatable after every service restart.

## Recommendation
Change the `unless` guard on `findEarliest()` to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_EARLIEST,
    unless = "#result == null || !#result.isPresent()")
Optional<RecordFile> findEarliest();
```

Apply the same fix to `findByIndex`, `findLatest`, `findByTimestamp`, and any other `Optional`-returning methods that use `unless = "#result == null"`. [6](#0-5) 

Additionally, consider adding a short TTL (e.g., `expireAfterWrite(60, TimeUnit.SECONDS)`) to `cacheManagerRecordFileEarliest` as a defense-in-depth measure, consistent with how `cacheManagerRecordFileLatest` is configured. [1](#0-0) 

## Proof of Concept
1. Start the mirror node web3 service against an empty database (no record files ingested yet).
2. Send any HTTP request that resolves `BlockType.EARLIEST`, e.g.:
   ```
   POST /api/v1/contracts/call
   { "block": "earliest", ... }
   ```
3. `RecordFileServiceImpl.findByBlockType(BlockType.EARLIEST)` calls `RecordFileRepository.findEarliest()`.
4. The DB returns no rows; Spring Data wraps this as `Optional.empty()`.
5. Spring evaluates `unless = "#result == null"` → `false` (because `Optional.empty() != null`).
6. `Optional.empty()` is stored in the single-slot, no-TTL Caffeine cache.
7. Ingest a record file into the database.
8. Send any subsequent `earliest`-block request — the cache is hit, `Optional.empty()` is returned, the database is never queried. The service must be restarted to recover.
