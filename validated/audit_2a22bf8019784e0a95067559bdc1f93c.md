Audit Report

## Title
Cache Poisoning via NULL `runtime_bytecode` Causes Stale Empty Bytecode Served for Up to 1 Hour

## Summary
`ContractRepository.findRuntimeBytecode()` is annotated with `@Cacheable(unless = "#result == null")` but returns `Optional<byte[]>`. When a contract row exists with `runtime_bytecode = NULL`, Spring Data JPA returns `Optional.empty()` — a non-null object — so the `unless` guard never fires and the empty result is written into the Caffeine cache. Any caller who queries such a contract ID before its bytecode is populated permanently poisons the cache entry for the full TTL (default `expireAfterAccess=1h`), causing the web3 API to serve empty bytecode even after the DB is correctly populated.

## Finding Description

**Exact location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [1](#0-0) 

**Root cause:** The `unless` SpEL expression `#result == null` tests whether the `Optional` wrapper object itself is `null`. Spring Data JPA never returns a literal `null` for an `Optional`-typed repository method — it always returns either `Optional.of(bytes)` or `Optional.empty()`. When the DB row exists but `runtime_bytecode IS NULL`, the native query returns a SQL NULL, which Spring Data JPA maps to `Optional.empty()`. Since `Optional.empty() != null`, the condition `#result == null` is `false`, the `unless` clause does not fire, and `Optional.empty()` is written into the Caffeine cache under the contract's ID.

The cache manager for contracts is configured with `expireAfterAccess=1h,maximumSize=1000` as the default: [2](#0-1) 

The cache bean is wired via `setCacheSpecification(cacheProperties.getContract())`: [3](#0-2) 

**Downstream impact path:** `ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode()` and chains `.orElse(null)`, returning `null` to the EVM when the cached `Optional.empty()` is served: [4](#0-3) 

**Why the per-request cache does not mitigate this:** `ReadableKVStateBase.get()` maintains a per-request read cache in `ContractCallContext` (a `ScopedValue` scoped to a single request). This cache is created fresh for every request via `ScopedValue.where(SCOPED_VALUE, new ContractCallContext())`. It does not prevent the cross-request Caffeine cache on `findRuntimeBytecode` from being poisoned. [5](#0-4) [6](#0-5) 

## Impact Explanation

For the duration of the cache TTL (up to 1 hour by default, `expireAfterAccess=1h`), the web3 API incorrectly reports the contract as having no bytecode (`0x` for `eth_getCode`). Any `eth_call` or `eth_estimateGas` targeting that contract will fail or behave as if calling an EOA. This constitutes incorrect data served by the mirror node's web3 layer — a denial-of-service on contract interaction for any contract that passes through the NULL-bytecode transient state. The impact is bounded by the cache TTL and the cache size (up to 1,000 entries by default).

## Likelihood Explanation

The precondition — a contract row with `runtime_bytecode = NULL` — is a normal transient state that occurs between contract creation and sidecar/importer processing. No privileges are required; any caller with network access to the web3 API endpoint can trigger this by issuing a standard JSON-RPC call (`eth_getCode`, `eth_call`) for a known contract ID. The attack is trivially repeatable and requires no authentication.

## Recommendation

Change the `unless` condition to also exclude `Optional.empty()` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

This mirrors the pattern already used in `AddressBookEntryRepository` in the grpc module, which uses a SpEL helper `@spelHelper.isNullOrEmpty(#result)` to guard against empty collections being cached: [7](#0-6) 

## Proof of Concept

1. Insert a contract row with `runtime_bytecode = NULL`:
   ```sql
   INSERT INTO contract (id, ...) VALUES (12345, ...);
   -- runtime_bytecode column left NULL
   ```
2. Issue `eth_getCode` for contract ID `12345` via the web3 API. The native query returns SQL NULL → `Optional.empty()` → cached in Caffeine under key `12345`.
3. Update the DB to populate bytecode:
   ```sql
   UPDATE contract SET runtime_bytecode = '\x6080...' WHERE id = 12345;
   ```
4. Re-issue `eth_getCode` for `12345`. The Caffeine cache returns the poisoned `Optional.empty()` entry; `readFromDataSource` returns `null`; the API responds with `0x` (no code) — despite the DB now having valid bytecode.
5. This persists for up to 1 hour until the `expireAfterAccess` TTL expires.
