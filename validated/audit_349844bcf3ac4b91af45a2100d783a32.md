## Audit Report

## Title
Unauthenticated GraphQL `account()` Query Enables DB Connection Pool Exhaustion via Non-Existent EntityId Flood

## Summary
The `account(input: AccountInput!)` GraphQL endpoint accepts queries from any unauthenticated user and performs a synchronous, uncached database lookup for every request. The GraphQL module has no application-level rate limiting and no negative-result caching. A flood of valid-format queries for non-existent `entityId` values will each consume a HikariCP connection for the full query duration, enabling pool exhaustion and denial of service.

## Finding Description

**Entry point** — `query.graphqls` line 5 defines `account(input: AccountInput!): Account` with no authentication directive of any kind. [1](#0-0) 

**Controller** — `AccountController.account()` at lines 33–44 calls `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)` unconditionally for any syntactically valid `entityId` input, with no guard before the service call. [2](#0-1) 

**Service** — `EntityServiceImpl.getByIdAndType()` at line 25 calls `entityRepository.findById(entityId.getId())` directly with no caching layer. [3](#0-2) 

**Repository — no caching** — The GraphQL module's `EntityRepository` has zero `@Cacheable` annotations. Every call, including misses for non-existent IDs, hits the database. [4](#0-3) 

Contrast with the `web3` module's `EntityRepository`, which uses `@Cacheable(unless = "#result == null")` on its primary lookup methods, caching both hits and suppressing repeated misses. [5](#0-4) 

**No application-level rate limiting** — `GraphQlConfiguration` installs only `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`, which limit per-query structure, not request rate. There is no `ThrottleConfiguration`, `ThrottleManager`, or Bucket4j bean anywhere in the `graphql` module (all three exist in the `web3` module). [6](#0-5) 

**Infrastructure guard is limited** — The Helm chart configures a Traefik `inFlightReq: amount: 5` per source IP. This only caps concurrent in-flight requests per single IP; a distributed attacker using N IPs can sustain N×5 concurrent DB-holding requests. The guard is absent in non-Helm deployments entirely. [7](#0-6) 

The GCP gateway also configures `maxRatePerEndpoint: 250` (line 56), which provides an additional rate limit, but this is only active in GCP-gateway-enabled deployments (`gateway.gcp.enabled: true`), not in all deployment configurations. [8](#0-7) 

**Recognized risk surface** — The team's own Prometheus alert `GraphQLHighDBConnections` fires when HikariCP active connections exceed 75% of the pool maximum for 5 minutes, confirming DB connection exhaustion is a known operational concern. [9](#0-8) 

**Root cause and failed assumption:** The design assumes infrastructure-level guards (Traefik `inFlightReq`, GCP gateway rate limit) are always present and sufficient. They are not: they are absent in non-Helm/non-GCP deployments, and even when present, a distributed attacker trivially multiplies the per-IP cap across many source IPs. The application itself has no rate limiting or negative-result caching to absorb the load.

## Impact Explanation
Full denial of service for the GraphQL API. When the HikariCP pool is saturated, all queries — including legitimate ones — queue and time out. Because the `entity` table primary-key lookup is fast, each connection is held only briefly, but at high concurrency the pool stays saturated continuously. No data is leaked or mutated; availability is completely lost. Severity: **High** (availability impact, no privilege required, publicly reachable endpoint).

## Likelihood Explanation
Any external attacker with a script and a handful of IP addresses (or a single IP with HTTP pipelining) can trigger this. The attack requires zero knowledge of the system beyond the public GraphQL schema, which is discoverable via introspection or the public `graphiql` UI at `/graphiql`. It is trivially repeatable and automatable. The `inFlightReq: 5` Traefik guard is the only per-IP barrier and is easily bypassed with distributed sources; the GCP `maxRatePerEndpoint: 250` guard is only present in GCP-enabled deployments.

## Recommendation

1. **Add negative-result caching** in `EntityServiceImpl` or `EntityRepository` (graphql module) using `@Cacheable(unless = "#result != null")` — mirroring the pattern already used in the `web3` module's `EntityRepository` — so repeated lookups for non-existent IDs do not reach the database.
2. **Add application-level rate limiting** in the `graphql` module, analogous to the `ThrottleConfiguration`/`ThrottleManager`/Bucket4j pattern already implemented in the `web3` module.
3. **Do not rely solely on infrastructure guards** for DoS protection; application-level controls must be present regardless of deployment topology.

## Proof of Concept

```
# No credentials required.
# Send concurrent POST requests to /graphql/alpha with valid-format but non-existent entityId values.

POST /graphql/alpha HTTP/1.1
Content-Type: application/json

{"query":"{ account(input: {entityId: {shard:0, realm:0, num:999999999}}) { id } }"}
```

Repeat at high concurrency from multiple IPs. Each request:
1. Passes all parser limits, complexity/depth checks, and `@Valid` bean validation.
2. Reaches `entityRepository.findById(999999999L)` — acquires a HikariCP connection.
3. Executes `SELECT * FROM entity WHERE id = ?` — gets an empty result.
4. Releases the connection.

At sufficient concurrency (N IPs × 5 `inFlightReq` each), the HikariCP pool is saturated. The `GraphQLHighDBConnections` Prometheus alert fires after 5 minutes at >75% utilization — by which point service is already degraded.
