Audit Report

## Title
Gas Estimation Binary Search Amplifies EVM Execution Work Per Rate-Limit Token

## Summary
`ContractController.call()` consumes exactly one `rateLimitBucket` token and one `gasLimitBucket` deduction per HTTP request regardless of the `estimate` flag. When `estimate=true`, `ContractExecutionService.estimateGas()` triggers one initial EVM execution followed by up to 20 additional `doProcessCall()` iterations inside `BinaryGasEstimator.search()`, yielding up to 21 EVM executions per single throttle check. All 20 binary search iterations pass `estimate=true` to `doProcessCall()`, which explicitly skips `restoreGasToBucket()` and any additional throttle accounting.

## Finding Description

**Throttle entry point — one check, one token:**

`ContractController.call()` at line 40 calls `throttleManager.throttle(request)` unconditionally before any branching on `estimate`. [1](#0-0) 

`ThrottleManagerImpl.throttle()` consumes 1 `rateLimitBucket` token and `scaleGas(request.getGas())` `gasLimitBucket` tokens — once, before the service layer decides how many EVM executions to perform. [2](#0-1) 

**Estimate path triggers up to 21 EVM executions:**

`ContractExecutionService.processCall()` routes to `estimateGas()` when `params.isEstimate()` is true. [3](#0-2) 

Inside `estimateGas()`: line 82 calls `callContract(params, context)` → `doProcessCall(params, params.getGas(), false)` for 1 initial execution, then lines 91–95 call `binaryGasEstimator.search(gas -> doProcessCall(params, gas, true), ...)` for up to 20 additional executions. [4](#0-3) 

**Binary search loop cap is 20:**

`BinaryGasEstimator.search()` line 35: `while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount())` where `maxGasEstimateRetriesCount = 20` (default). [5](#0-4) [6](#0-5) 

**Binary search iterations are invisible to the throttle:**

`ContractCallService.doProcessCall()` lines 128–135: the `finally` block only calls `restoreGasToBucket()` and records metrics when `!estimate`. When called with `estimate=true` (all 20 binary search iterations), no throttle interaction occurs at all. [7](#0-6) 

**Root cause:** The throttle is applied at the HTTP boundary on the raw request object, before the service layer decides how many EVM executions to perform. The `estimate` flag is not factored into the throttle budget.

## Impact Explanation

An attacker can send `estimate=true` requests at the full `requestsPerSecond` rate (default 500 req/s) and trigger up to 21 EVM executions per request — an effective 21× amplification of CPU, memory, and I/O load on the mirror node. At 500 req/s × 21 executions = 10,500 EVM invocations per second, each potentially executing up to 15M gas of EVM bytecode (`maxGasLimit = 15_000_000`). This can exhaust mirror node CPU and cause denial of service for all users. [8](#0-7) 

The `gasLimitBucket` does not compensate because it is consumed once for the declared gas limit, not multiplied by the number of binary search iterations. Additionally, `scaleGas()` returns 0 for gas values ≤ 10,000, meaning an attacker can bypass the `gasLimitBucket` entirely by using a low gas value while still triggering the binary search loop. [9](#0-8) 

## Likelihood Explanation

No authentication or special privilege is required. Any HTTP client can set `"estimate": true` in the POST body to `/api/v1/contracts/call`. The attack is trivially repeatable, requires no contract deployment, and can be scripted with a single `curl` command in a loop. The amplification factor is deterministic and bounded only by the `maxGasEstimateRetriesCount` configuration value (default 20). [10](#0-9) 

## Recommendation

1. **Multiply the throttle cost for estimate requests**: Before calling `estimateGas()`, consume additional `rateLimitBucket` and/or `gasLimitBucket` tokens proportional to `maxGasEstimateRetriesCount` (e.g., consume `maxGasEstimateRetriesCount + 1` rate tokens for estimate requests).
2. **Separate rate limit for estimate requests**: Introduce a dedicated `estimateRateLimitBucket` with a lower capacity than the standard `rateLimitBucket`, checked when `request.isEstimate() == true` in `ThrottleManagerImpl.throttle()`.
3. **Account for gas in binary search iterations**: Extend `restoreGasToBucket()` logic to also apply during estimate iterations, or charge the `gasLimitBucket` once per binary search iteration inside `BinaryGasEstimator.search()`.

## Proof of Concept

```bash
# Send estimate=true requests at maximum rate to trigger 21x EVM amplification
while true; do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{
      "to": "0x0000000000000000000000000000000000000167",
      "data": "0x<complex_bytecode_selector>",
      "gas": 15000000,
      "estimate": true
    }' &
done
```

Each request consumes 1 `rateLimitBucket` token but triggers up to 21 EVM executions (1 initial + up to 20 binary search iterations in `BinaryGasEstimator.search()`), none of which interact with the throttle. At the default limit of 500 req/s, this yields up to 10,500 EVM invocations per second, each executing up to 15M gas of bytecode, exhausting CPU on the mirror node. [11](#0-10)

```java
    private Bytes estimateGas(final ContractExecutionParameters params, final ContractCallContext context) {
        final var processingResult = callContract(params, context);
        final var gasUsedByInitialCall = processingResult.gasUsed();

        // sanity check ensuring gasUsed is always lower than the inputted one
        if (gasUsedByInitialCall >= params.getGas()) {

                gas -> doProcessCall(params, gas, true),
                gasUsedByInitialCall,
                params.getGas());

        return Bytes.ofUnsignedLong(estimatedGas);
```

```java
    public long search(
            final ObjIntConsumer<Long> metricUpdater, final LongFunction<EvmTransactionResult> call, long lo, long hi) {
        long prevGasLimit = lo;
        int iterationsMade = 0;
        long totalGasUsed = 0;

        // Now that we also support gas estimates for precompile calls, the default threshold is too low, since
        // it does not take into account the minimum threshold of 5% higher estimate than the actual gas used.
        // The default value is working with some calls but that is not the case for precompile calls which have higher
        // gas consumption.
        // Configurable tolerance of 10% over 5% is used, since the algorithm fails when using 5%, producing too narrow
        // threshold. Adjust via estimateGasIterationThresholdPercent value.
        final long estimateIterationThreshold = (long) (lo * properties.getEstimateGasIterationThresholdPercent());

        ContractCallContext contractCallContext = ContractCallContext.get();
        while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount()) {
            contractCallContext.reset();

            long mid = (hi + lo) / 2;

            // If modularizedServices is true - we call the safeCall function that handles if an exception is thrown
            final var transactionResult = safeCall(mid, call);

            iterationsMade++;

            boolean err =
                    transactionResult == null || !transactionResult.isSuccessful() || transactionResult.gasUsed() < 0;
            long gasUsed = err ? prevGasLimit : transactionResult.gasUsed();
            totalGasUsed += gasUsed;
            if (err || gasUsed == 0) {
                lo = mid;
            } else {
                hi = mid;
                if (Math.abs(prevGasLimit - mid) < estimateIterationThreshold) {
                    lo = hi;
                }
            }
            prevGasLimit = mid;
        }

        metricUpdater.accept(totalGasUsed, iterationsMade);
        return hi;
    }
```
