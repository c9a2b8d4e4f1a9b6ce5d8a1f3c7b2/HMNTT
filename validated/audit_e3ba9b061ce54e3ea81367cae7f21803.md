Audit Report

## Title
Cache Key Hash Collision in `findByEvmAddressOrAliasAndDeletedIsFalse` Enables Cross-User Entity Impersonation via 32-bit `Arrays.hashCode`

## Summary
`EntityRepository.findByEvmAddressOrAliasAndDeletedIsFalse` and `findByEvmAddressAndDeletedIsFalse` use `@spelHelper.hashCode(#alias)` — which resolves to `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole Caffeine cache key. Because no secondary equality check on the originating byte array is performed after a cache hit, two distinct byte arrays that produce the same 32-bit hash value are indistinguishable to the cache. An attacker who can craft such a collision receives the victim's `Entity` object for their own lookup.

## Finding Description

**Exact code path:**

`SpelHelper.hashCode` delegates directly to `Arrays.hashCode(byte[])`: [1](#0-0) 

This 32-bit integer is used as the sole cache key on both alias-lookup methods: [2](#0-1) 

The `CACHE_MANAGER_ENTITY` bean registers `CACHE_NAME_ALIAS` and `CACHE_NAME_EVM_ADDRESS` as shared, process-wide Caffeine caches backed by the same `cacheProperties.getEntity()` spec (`expireAfterWrite=1s,maximumSize=10000`): [3](#0-2) 

The caller `CommonEntityAccessor.get(Bytes alias, ...)` passes arbitrary user-controlled bytes directly to the cached repository method: [4](#0-3) 

**Root cause:** The cache key is a 32-bit `int`. Spring's `@Cacheable` uses this integer as the map key with no further validation. After a cache hit, the framework returns the stored `Optional<Entity>` without verifying that the stored entry's originating byte array equals the current query's byte array.

**Why `unless = "#result == null"` fails:** The return type is `Optional<Entity>`. Spring Data JPA never returns a raw `null` for `Optional`-typed repository methods — it returns `Optional.empty()` for no result. Therefore `#result == null` is always `false`, and both `Optional.empty()` and `Optional.of(entity)` are cached unconditionally under the hash key. [5](#0-4) 

## Impact Explanation
An attacker who successfully collides with a victim's cache entry receives the victim's `Entity` object — including account ID, balance, alias, EVM address, and key material — for their own address lookup. In the web3 module's EVM simulation context (`eth_call`, `eth_getBalance`, smart contract execution), this causes:

- **Incorrect balance reads:** the attacker's address reports the victim's balance in simulated calls.
- **Incorrect entity resolution in EVM simulation:** token transfer simulations, allowance checks, and contract interactions that resolve addresses via this path operate on the wrong entity.
- **Cache poisoning (DoS variant):** the attacker can pre-populate the cache with `Optional.empty()` under hash `H` before the victim's entity is ever cached, causing all subsequent legitimate lookups for the victim's address to return empty for the duration of the cache TTL (`expireAfterWrite=1s` default, but operator-configurable), effectively making the victim's account invisible to the EVM layer. [6](#0-5) 

The mirror node is read-only and does not execute on-chain fund transfers directly; however, incorrect entity resolution in `eth_call` simulations can mislead dApps and wallets into displaying wrong balances or approving transactions based on false state.

## Likelihood Explanation
- **No privilege required:** Any user who can submit an `eth_call` or account query can trigger this.
- **Collision is computationally feasible:** `Arrays.hashCode` over a byte array has a 32-bit output. For a targeted preimage attack (finding byte array `A` such that `Arrays.hashCode(A) == Arrays.hashCode(victimAddress)`), an analytical approach fixes a prefix and solves for a suffix byte: `b_last = (H - 31 * h_prefix) mod 2^32`. Since a Java byte is in `[-128, 127]`, the probability of a valid solution for a random prefix is `256/2^32`. Iterating over ~16 million prefixes offline yields a valid collision — entirely feasible without any network interaction.
- **Cache TTL is short by default (1s) but configurable:** operators may increase TTL, extending the poisoning window.
- **Repeatability:** the attacker can re-poison after each eviction with no rate limiting on the cache key computation path.

## Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with the actual byte array wrapped in a type that implements correct `equals`/`hashCode` semantics (e.g., `ByteBuffer.wrap(alias)` or a dedicated value object). This ensures the cache performs an equality check on the full byte content, not just a 32-bit hash. Alternatively, use `java.util.Arrays.toString(alias)` or a hex-encoded string as the key, which is collision-free within the domain of valid byte arrays.

Additionally, fix the `unless` condition: change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` (or `unless = "#result?.isEmpty() == true"`) to avoid caching empty results if that is the intended behavior.

//   If not, vary the prefix and retry (~16M iterations worst case)
byte[] attackerAlias = findCollision(H); // Arrays.hashCode(attackerAlias) == H

// Attacker submits eth_call with alias = attackerAlias
// -> findByEvmAddressOrAliasAndDeletedIsFalse(attackerAlias) is called
// -> cache key = H -> cache HIT -> returns victim's Entity
// -> EVM simulation resolves attacker's address to victim's entity
