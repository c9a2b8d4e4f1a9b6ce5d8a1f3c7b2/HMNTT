# Audit Report

## Title
Unauthenticated Multiplicative Query Amplification DoS via `/api/v1/accounts?limit=100&balance=true`

## Summary
Any unauthenticated client can issue `GET /api/v1/accounts?limit=100` (with `balance=true` as the default) to force the database to materialize a full `latest_token_balance` CTE and execute up to 100 correlated subqueries against it per request. With no application-level rate limiting in the REST layer, concurrent flooding exhausts DB CPU and connection pool resources.

## Finding Description

**Verified code path:**

`getAccounts()` in `rest/accounts.js` at line 345 sets:
```js
const tokenBalanceQuery = {query: 'account_id = e.id', params: [], limit: tokenBalanceResponseLimit.multipleAccounts};
``` [1](#0-0) 

`tokenBalanceResponseLimit.multipleAccounts` is the server-side constant `50`, confirmed by the config test:
```js
expect(func()).toEqual({default: 25, max: 100, tokenBalance: {multipleAccounts: 50, singleAccount: 1000}});
``` [2](#0-1) 

`balance=true` is the default when the parameter is absent:
```js
const values = query[constants.filterKeys.BALANCE] || 'true';
``` [3](#0-2) 

`parseLimitAndOrderParams` calls `getLimitParamValue`, which caps the user-supplied `limit` at `responseLimit.max` (100) for unauthenticated requests:
```js
const maxLimit = getEffectiveMaxLimit();
ret = parsed > maxLimit ? maxLimit : parsed;
``` [4](#0-3) 

`getEntityBalanceQuery()` (non-historical path) builds the following SQL structure:
```sql
with latest_token_balance as (
   select account_id, balance, token_id
   from token_account
   where associated is true)   -- full CTE, no account filter
select ...,
  (select json_agg(...)
   from (
     select token_id, balance
     from latest_token_balance
     where account_id = e.id   -- correlated: re-evaluated per outer row
     order by token_id asc
     limit 50
   ) as account_token_balance) as token_balances
from entity as e
...
limit 100
``` [5](#0-4) 

The CTE at lines 183–186 selects from `token_account` with no account-level predicate — it materializes all associated token accounts. The correlated subquery at line 191 (`where account_id = e.id`) is then re-evaluated for each of the up to 100 outer rows. [6](#0-5) 

**No application-level rate limiting exists in the REST layer.** `server.js` registers the following middleware chain: `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`, route handlers, `responseHandler`, optional `responseCacheUpdateHandler`, and `handleError`. There is no rate-limiting middleware. [7](#0-6) 

The `authHandler` only sets a custom per-user response `limit` for authenticated users via `httpContext`; it does not throttle or reject unauthenticated requests. [8](#0-7) 

The throttling infrastructure that does exist (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3/` Java service, not the Node.js REST API. [9](#0-8) 

## Impact Explanation

Each request with `limit=100` (and `balance=true` default) causes the database to:
1. Materialize the full `latest_token_balance` CTE (all rows in `token_account where associated is true` — potentially millions on mainnet).
2. Execute 100 correlated subquery evaluations against that materialized result.

Flooding with concurrent requests exhausts the DB connection pool (`max: config.db.pool.maxConnections`) and CPU. The only protection is a `statement_timeout` at the pool level, which limits individual query duration but does not prevent concurrent request flooding. This can cause query queuing, cascading timeouts, and service unavailability for all users.

## Likelihood Explanation

The exploit requires zero privileges and zero authentication. The request `GET /api/v1/accounts?limit=100` is syntactically indistinguishable from legitimate traffic. The endpoint is publicly documented in the OpenAPI spec. A single machine with high concurrency (e.g., `ab -c 200 -n 10000`) is sufficient to trigger the condition. Repeatability is unlimited.

## Recommendation

1. **Add rate limiting middleware** to the Node.js REST layer (e.g., `express-rate-limit` or a Redis-backed limiter) scoped per IP, applied before route handlers in `server.js`.
2. **Add a per-request concurrency limit** (e.g., `express-slow-down` or an in-flight request counter) to bound simultaneous DB queries.
3. **Restrict the `latest_token_balance` CTE scope**: add a predicate to bound the CTE to only the accounts being queried (e.g., using the `account.id` filter range), rather than materializing all associated token accounts unconditionally.
4. Consider adding a `pg_advisory_lock` or query-level cost guard for this specific query shape.

## Proof of Concept

```bash
# Flood with 200 concurrent unauthenticated requests, each forcing max-cost query
ab -c 200 -n 5000 \
  'https://<mirror-node-host>/api/v1/accounts?limit=100'
```

Each request triggers the full CTE materialization + 100 correlated subqueries. With 200 concurrent connections and a pool of (default) limited size, the DB connection pool saturates, subsequent requests queue or time out, and the service becomes unavailable to all users.

const authHandler = async (req, res) => {
  const credentials = basicAuth(req);

  if (!credentials) {
    return;
  }

  const user = findUser(credentials.name, credentials.pass);
  if (!user) {
    res.status(httpStatusCodes.UNAUTHORIZED.code).json({
      _status: {
        messages: [{message: 'Invalid credentials'}],
      },
    });
    return;

  if (user.limit !== undefined && user.limit > 0) {
    httpContext.set(userLimitLabel, user.limit);
    logger.debug(`Authenticated user ${user.username} with custom limit ${user.limit}`);
  }
};

    @Bean(name = RATE_LIMIT_BUCKET)
