Audit Report

## Title
Permanent Processing Stall via Malformed Signature File Injection into Cloud Storage Bucket

## Summary
An attacker with write access to the cloud storage bucket can inject malformed `.rcd_sig` files for a target timestamp, causing `ConsensusValidatorImpl.validate()` to throw `SignatureVerificationException` on every poll cycle. Because `Downloader.downloadNextBatch()` catches and swallows this exception without advancing the `lastStreamFile` cursor, the downloader permanently re-attempts the same failing files on every scheduled tick, halting all record stream processing indefinitely.

## Finding Description

**Verified code path:**

**Step 1 — Scheduled entry point.** `RecordFileDownloader.download()` is annotated `@Scheduled` and calls `downloadNextBatch()` on every tick: [1](#0-0) 

**Step 2 — Exception swallowed without cursor advancement.** Inside `downloadNextBatch()`, `SignatureVerificationException` is caught and only logged — no cursor advancement, no skip, no circuit-breaker: [2](#0-1) 

**Step 3 — Cursor only advances on success.** `lastStreamFile` is updated exclusively inside `onVerified()`, which is only reached after a successful verification and `onVerified()` call: [3](#0-2) 

**Step 4 — Same cursor returned on next tick.** `getStartAfterFilename()` reads the unchanged `lastStreamFile` reference, causing the downloader to re-fetch the same malformed files: [4](#0-3) 

**Step 5 — Exception re-thrown from last failing group.** In `verifySigsAndDownloadDataFiles()`, when the last (or only) signature group fails verification, the exception is re-thrown up to `downloadNextBatch()`: [5](#0-4) 

**Step 6 — Consensus threshold never met for malformed files.** `NodeSignatureVerifier.verifySignature()` returns `false` for all malformed files (RSA verification fails), so no signature is marked `VERIFIED`, `signatureHashMap` remains empty, and `ConsensusValidatorImpl.validate()` throws: [6](#0-5) [7](#0-6) 

**Root cause:** The failed assumption is that all files present in the bucket are legitimate. There is no mechanism to skip past a permanently-failing signature file — the cursor only advances on success, and the exception is silently swallowed on failure.

## Impact Explanation
All record stream processing halts permanently at the injected timestamp. No subsequent record files are downloaded or parsed. The mirror node stops reflecting new network state, breaking all downstream consumers (REST API, gRPC streaming, etc.). Recovery requires manual operator intervention to either remove the injected file or reconfigure the start position.

## Likelihood Explanation
The precondition is write access to the cloud storage bucket. This is realistic in:
- Self-hosted mirror node deployments where operators configure their own S3/GCS buckets with overly permissive ACLs
- Deployments using a public bucket with write permissions inadvertently enabled
- Any environment where the bucket policy has not been hardened

The attack requires uploading one malformed `.rcd_sig` file per node folder for the target timestamp. No cryptographic material is needed — any byte sequence that fails RSA signature verification suffices. The stall is permanent until an operator manually intervenes.

## Recommendation

1. **Implement a skip-ahead mechanism:** Track consecutive failures per filename. After a configurable threshold (e.g., 3 consecutive poll cycles failing on the same filename), advance the cursor past the offending file and emit a high-severity alert.

2. **Add a circuit-breaker / dead-letter queue:** Files that fail signature verification repeatedly should be quarantined and logged with full detail, allowing processing to continue with subsequent files.

3. **Bucket write hardening:** Enforce strict write-only access for consensus nodes and read-only access for the mirror node importer. Bucket ACLs should be audited as part of deployment checklists.

4. **Alerting:** The current behavior only emits a `log.warn`. A permanently stalled downloader should trigger an operational alert (metric counter, health check failure, etc.) so operators are notified promptly.

    --content-length 512

# - Every subsequent poll repeats identically → permanent stall
