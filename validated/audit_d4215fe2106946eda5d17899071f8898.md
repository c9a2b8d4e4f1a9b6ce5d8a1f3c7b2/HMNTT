Audit Report

## Title
Unsynchronized Shared `LinkedList` in Singleton `MirrorNodeState` Enables `ConcurrentModificationException` via Concurrent Queue Read/Write

## Summary
`MirrorNodeState` is a singleton Spring bean (`@Named`) that initializes a single `new LinkedList<>()` as the backing store for the `RecordCacheService` transaction-receipts queue. Both `ListReadableQueueState` and `ListWritableQueueState` hold a direct reference to this same `LinkedList` instance with no synchronization. When two concurrent requests race — one iterating the queue via `iterateOnDataSource()` and another modifying it via `addToDataSource()` or `removeFromDataSource()` — Java's fail-fast `LinkedList` iterator throws `ConcurrentModificationException`, corrupting the transaction receipt processing pipeline.

## Finding Description

**Shared mutable state (root cause):**

`MirrorNodeState.initQueueStates()` (line 171) creates one `LinkedList<>` and stores it in the `states` map:
```java
states.put(RecordCacheService.NAME, new HashMap<>(Map.of(TRANSACTION_RECEIPTS_STATE_ID, new LinkedList<>())));
```

`getReadableStates()` (lines 77–78) wraps this same instance in `ListReadableQueueState`:
```java
if (state instanceof Queue<?> queue) {
    data.put(stateId, new ListReadableQueueState<>(serviceName, stateId, queue));
```

`getWritableStates()` (lines 101–102) wraps the same instance in `ListWritableQueueState`:
```java
if (state instanceof Queue<?> queue) {
    data.put(stateId, new ListWritableQueueState<>(serviceName, stateId, queue));
```

Both `readableStates` and `writableStates` are cached in `ConcurrentHashMap` fields (lines 52–53), so the same wrapper objects — and therefore the same `LinkedList` — are reused across all concurrent requests.

**Unsynchronized iterator (trigger):**

`ListReadableQueueState.iterateOnDataSource()` (lines 40–42) returns a raw fail-fast iterator:
```java
protected Iterator<E> iterateOnDataSource() {
    return backingStore.iterator();
}
```

`ListWritableQueueState.addToDataSource()` (lines 31–33) and `removeFromDataSource()` (lines 36–38) mutate the same `LinkedList` directly:
```java
protected void addToDataSource(@NonNull E element) { backingStore.add(element); }
protected void removeFromDataSource() { backingStore.remove(); }
```

**Why existing checks are insufficient:**

The `ConcurrentHashMap` at lines 52–53 only protects the map of state wrappers, not the `LinkedList` contents. There is no `synchronized` block, `ReentrantLock`, `CopyOnWriteArrayList`, or `Collections.synchronizedList()` wrapping the `LinkedList`. The `WritableQueueStateBase` buffers adds locally, but `commit()` (called from `MapWritableStates.commit()` line 102) flushes them via `addToDataSource()` directly onto the shared `LinkedList` with no lock held.

## Impact Explanation
A `ConcurrentModificationException` thrown inside `iterateOnDataSource()` propagates up through the receipt-lookup or receipt-iteration path of the transaction processing pipeline. This causes the affected request to fail with an unhandled exception. Under sustained concurrent load, this is reliably triggerable, resulting in denial-of-service for transaction receipt queries and potential corruption of in-flight receipt processing state. Severity: **Medium–High** (availability impact on the web3 transaction pipeline; no authentication required).

## Likelihood Explanation
The mirror node's web3 module is publicly accessible. Any unprivileged user can submit concurrent `eth_sendRawTransaction` (or equivalent) calls. Two concurrent requests are sufficient: one that causes a write commit to the `RecordCacheService` queue and one that iterates it. This is trivially achievable with a simple HTTP client sending parallel requests. The race window is narrow but repeatable under load, making it a practical denial-of-service vector.

## Recommendation
Replace the bare `new LinkedList<>()` in `initQueueStates()` with a thread-safe alternative. The simplest fix is:

```java
// Option 1: ConcurrentLinkedDeque (lock-free, weakly consistent iterator — no CME)
states.put(RecordCacheService.NAME,
    new HashMap<>(Map.of(TRANSACTION_RECEIPTS_STATE_ID, new ConcurrentLinkedDeque<>())));

// Option 2: synchronized wrapper
states.put(RecordCacheService.NAME,
    new HashMap<>(Map.of(TRANSACTION_RECEIPTS_STATE_ID,
        Collections.synchronizedList(new LinkedList<>()))));
```

`ConcurrentLinkedDeque` is preferred because its iterator is weakly consistent and never throws `ConcurrentModificationException`, matching the expected semantics of a concurrent queue. No changes to `ListReadableQueueState` or `ListWritableQueueState` are required.

## Proof of Concept

```java
// Pseudocode — two threads sharing the singleton MirrorNodeState bean

// Thread A: iterates the RecordCacheService queue (e.g., receipt lookup)
ReadableStates rs = mirrorNodeState.getReadableStates(RecordCacheService.NAME);
ReadableQueueState<TransactionReceiptEntries> queue =
    rs.getQueue(TRANSACTION_RECEIPTS_STATE_ID);
Iterator<?> it = queue.iterator(); // calls iterateOnDataSource() → LinkedList.iterator()
// --- context switch to Thread B here ---

// Thread B: commits a write to the same queue (e.g., eth_sendRawTransaction processing)
WritableStates ws = mirrorNodeState.getWritableStates(RecordCacheService.NAME);
WritableQueueState<TransactionReceiptEntries> wq =
    ws.getQueue(TRANSACTION_RECEIPTS_STATE_ID);
wq.add(newReceipt);
((CommittableWritableStates) ws).commit(); // → addToDataSource() → LinkedList.add()
// modCount of the shared LinkedList is now incremented

// Thread A resumes:
it.hasNext(); // ← throws ConcurrentModificationException
```

HTTP-level trigger: send two parallel POST requests to the web3 JSON-RPC endpoint — one `eth_getTransactionReceipt` (triggers iteration) and one `eth_sendRawTransaction` (triggers a write commit) — targeting the same node under load.
