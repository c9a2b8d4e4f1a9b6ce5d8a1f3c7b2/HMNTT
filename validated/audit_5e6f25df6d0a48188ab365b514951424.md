Audit Report

## Title
Cache-Amplified DoS via Unbounded Slot Key Accumulation in `findStorageBatch()`

## Summary
An unauthenticated attacker can fill the per-contract `CaffeineCache` (bounded at `maximumSize=1500`) with 1500 unique, non-existent slot keys for a single `contractId`. Because non-existent slots are never written to `contractStateCache`, every subsequent request for any of those keys unconditionally triggers a SQL `IN` query carrying all 1500 accumulated keys. This creates a sustained, amplified DB load from a minimal request rate, with no authentication or rate-limiting guard in the affected code path.

## Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` (lines 58–70) checks `contractStateCache` for the requested `(contractId, key)` pair. [1](#0-0) 

On a miss it calls `findStorageBatch()` (lines 85–122). Inside `findStorageBatch()`:

- **Line 90**: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — unconditionally registers the caller-supplied slot key into the per-contract cache *before* any DB query. [2](#0-1) 
- **Line 91**: `cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet()` — collects **all** currently cached slot keys for that contract.
- **Lines 93–101**: copies every cached key into `cachedSlots` with no size cap. [3](#0-2) 
- **Line 103**: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — issues one SQL `slot IN (…)` query containing every accumulated key. [4](#0-3) 

**Root cause:** The per-contract cache (`slotsPerContract`) is configured with `expireAfterAccess=5m,maximumSize=1500`. [5](#0-4)  There is no upper bound enforced on `cachedSlots` before it is passed to the repository. The design assumes the cache accumulates only keys that were legitimately accessed together, but an attacker can deliberately inject up to 1500 arbitrary keys.

**Critical amplification factor for non-existent slots:** `contractStateCache` only stores values for slots that *exist* in the DB (lines 106–113). [6](#0-5)  For non-existent (attacker-injected) slots, `contractStateCache.get()` always returns `null`, so the condition at line 65 (`cachedValue != null && cachedValue != EMPTY_VALUE`) is always false. [7](#0-6)  This means the attacker does **not** need to wait for `contractStateCache` to expire — every single request for any non-existent slot immediately calls `findStorageBatch()` and triggers the full 1500-key batch query.

**Why existing checks are insufficient:**
- `enableBatchContractSlotCaching` (line 59) is `true` by default and is a global kill-switch, not a per-request guard. [8](#0-7) 
- `maximumSize=1500` is the attacker's target ceiling, not a mitigation.
- `expireAfterAccess=5m` resets on every access, so continuous attacker traffic keeps the poisoned cache alive indefinitely.
- No authentication, rate limiting, or slot-key validation exists in the `findStorageBatch()` call chain.

The outer `contractSlots` cache holds up to 3000 per-contract caches (`maximumSize=3000`). [9](#0-8)  Scaling the attack across 3000 contracts multiplies the effect proportionally.

## Impact Explanation
The web3 JSON-RPC service and the mirror node importer share the same PostgreSQL instance. Sustained large `IN` queries (1500 bind parameters per query, potentially against a large `contract_state` table) consume DB connection pool slots, CPU, and I/O. The `statementTimeout` for web3 is 3000 ms. [10](#0-9)  Under sufficient load this causes query timeouts in the importer, which can stall consensus record processing and result in gaps in data exported to downstream mirror node consumers (REST API, gRPC streaming). Severity: **High** — availability impact on a public, unauthenticated endpoint with a direct path to shared infrastructure.

## Likelihood Explanation
The attack requires zero privileges: the web3 JSON-RPC endpoint is publicly accessible. The attacker needs only 1500 HTTP requests to prime the cache (achievable within the 500 RPS global throttle in ~3 seconds). [11](#0-10)  After priming, a single request per cycle sustains the amplified DB load indefinitely, since non-existent slots are never cached in `contractStateCache` and always re-trigger the full batch query. No knowledge of the contract's actual storage layout is needed — random 32-byte slot keys suffice because line 90 registers the key before the DB query, and the DB simply returns no row for non-existent slots. The attack is fully repeatable and trivially automatable.

## Recommendation
1. **Cap `cachedSlots` before the DB call**: enforce a hard maximum (e.g., equal to `maximumSize` of `slotsPerContract`) on the size of `cachedSlots` before passing it to `contractStateRepository.findStorageBatch()`.
2. **Cache negative results**: after a batch query, store `EMPTY_VALUE` in `contractStateCache` for every key in `cachedSlots` that was *not* returned by the DB. This prevents non-existent slots from bypassing `contractStateCache` on every request.
3. **Per-contract rate limiting**: limit the number of distinct slot keys that can be registered in the per-contract cache from a single source IP within a time window.
4. **Separate DB users/pools**: isolate the web3 DB connection pool from the importer's pool to prevent cross-component availability impact.

## Proof of Concept

```python
import requests, os

TARGET = "http://<mirror-node-web3>/api/v1/contracts/call"  # or eth_getStorageAt endpoint
CONTRACT_ID = "0.0.12345"
MAX_SLOTS = 1500

# Step 1: Prime the per-contract cache with 1500 unique non-existent slot keys
for i in range(MAX_SLOTS):
    slot = hex(i).zfill(66)  # unique 32-byte slot key
    requests.post(TARGET, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [CONTRACT_ID, slot, "latest"], "id": i
    })

# Step 2: Sustain — one request every 2s triggers a 1500-key IN query
import time
while True:
    requests.post(TARGET, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [CONTRACT_ID, "0x" + "aa" * 32, "latest"], "id": 9999
    })
    time.sleep(2)
```

After step 1, every iteration of step 2 causes `findStorageBatch()` to issue `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (<1500 values>)` against the shared DB. Since none of the 1500 injected slots exist, no results are cached in `contractStateCache`, and the full batch query fires on every request indefinitely.
