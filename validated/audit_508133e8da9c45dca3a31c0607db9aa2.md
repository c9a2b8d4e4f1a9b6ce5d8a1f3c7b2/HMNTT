Audit Report

## Title
Race Condition in `RedisTopicListener.getSharedListener()` Causes Permanent Message Loss for New Subscribers

## Summary
A non-atomic sequence between `topicMessages.remove()` (triggered via `doOnCancel` when the last subscriber cancels) and `computeIfAbsent()` (called by the next subscriber) creates a race window during which messages published to the Redis channel are silently dropped. The higher-level recovery mechanisms in `TopicMessageServiceImpl` do not recover these messages because gap detection is bypassed when `last == null`, and the safety check queries forward from the last-received message's timestamp if any message arrives before the 1-second timer fires.

## Finding Description

**Verified code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`

- `getSharedListener()` at line 61 calls `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))`. [1](#0-0) 

- `subscribe()` at line 74 registers `.doOnCancel(() -> unsubscribe(topic))` on the upstream chain, before `.share()`. [2](#0-1) 

- `unsubscribe()` at lines 82–84 calls `topicMessages.remove(topic.getTopic())`. [3](#0-2) 

**Root cause:**

`subscribe()` terminates with `.share()` (Reactor's `publish().refCount()`). When the last subscriber cancels, `.share()` propagates a cancel signal upstream through the operator chain. The `doOnCancel` callback fires synchronously as part of that cancel propagation — calling `unsubscribe()` → `topicMessages.remove()` — before the upstream Redis subscription is fully torn down. These two steps (map key removal and new-subscription setup via `computeIfAbsent`) are not atomic with respect to each other.

**Race window:**

```
Thread A (last subscriber cancels):
  1. .share() refCount → 0, cancel propagated upstream
  2. doOnCancel fires → topicMessages.remove("topic.X")  ← key gone from map

Thread B (new subscriber, concurrent):
  3. computeIfAbsent("topic.X", ...) → key absent → subscribe() called
  4. New Redis SUBSCRIBE command sent (async, not yet acknowledged/active)

Redis broker (between steps 2 and 4):
  → Messages M1, M2, M3 published to "topic.X"
  → No active subscriber on the mirror node → messages silently dropped
```

**Why existing recovery mechanisms fail:**

1. **Per-message gap detection** (`missingMessages`, line 152 of `TopicMessageServiceImpl`):
   ```java
   if (last == null || topicContext.isNext(current)) {
       return Flux.just(current);
   }
   ```
   When Thread B's new subscriber receives its first message M4, `last == null`, so `isNext()` short-circuits to `true` and no gap query is issued for M1–M3. [4](#0-3) 

2. **Safety check** (lines 67–70):
   ```java
   Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
       .filter(_ -> !topicContext.isComplete())
       .flatMapMany(_ -> missingMessages(topicContext, null))
       ...
   ```
   If M4 arrives within 1 second (typical on any active topic), `topicContext.getLast()` is already M4 when the timer fires. `missingMessages(ctx, null)` then sets `startTime = M4.consensusTimestamp + 1`, querying only forward — M1–M3 are permanently skipped. [5](#0-4) [6](#0-5) 

## Impact Explanation
Messages published to a Redis topic channel during the unsubscribe/resubscribe gap are permanently lost for any subscriber that joins during that window and receives at least one subsequent message before the 1-second safety-check timer expires. This directly violates the mirror node's correctness guarantee of delivering all consensus-ordered `TopicMessage` records to gRPC subscribers. Affected subscribers receive a silently incomplete stream with no error signal, no gap indicator, and no retry opportunity.

## Likelihood Explanation
Any unauthenticated gRPC client can open and immediately cancel a `subscribeTopic` stream (zero privilege required). On a topic with moderate traffic, an attacker who is the sole subscriber can reliably open the race window by cancelling and immediately re-subscribing. The window is narrow (milliseconds) but the attack is repeatable at will and requires no special knowledge beyond the topic ID, which is public on-chain data. The combination of a narrow but repeatable window and a complete absence of any error signal to the victim subscriber makes this a realistic correctness threat.

## Recommendation

Replace the non-atomic remove-then-reinsert pattern with a conditional removal that ties the map entry's lifetime to the specific `Flux` instance:

1. **Use `ConcurrentHashMap.remove(key, value)`** in `unsubscribe()` so the entry is only removed if it still holds the exact `Flux` instance being torn down. This prevents a newly inserted entry from being accidentally evicted by a delayed cancel callback from an older subscription.

2. **Alternatively**, move the `doOnCancel`/`doOnComplete` cleanup inside the `.share()` lifecycle by using a `Sinks`-based approach or by wrapping the shared flux in a `refCounted` publisher that atomically removes the map entry only after the new upstream subscription is confirmed active.

3. **Strengthen gap detection** so that when `last == null` and the subscriber's `startTime` is in the past, a DB query is issued to verify no messages were missed before the first live message arrived — regardless of whether `isNext()` returns `true`.

## Proof of Concept

```
1. Client A subscribes to topic.42 (becomes the sole subscriber).
2. Client A cancels the subscription.
   → .share() refCount → 0 → doOnCancel → topicMessages.remove("topic.42")
3. Client B immediately calls subscribeTopic for topic.42.
   → computeIfAbsent sees key absent → subscribe() called → async Redis SUBSCRIBE sent
4. While the new Redis SUBSCRIBE is in-flight (not yet acknowledged):
   → Messages M1, M2, M3 are published to topic.42 on Redis
   → No active subscriber on the mirror node → M1–M3 dropped
5. Redis SUBSCRIBE acknowledged; Client B starts receiving from M4 onward.
6. In missingMessages(), last == null → isNext(M4) == true → no gap query.
7. Safety check fires at T+1s: last == M4 → queries from M4.timestamp+1 → M1–M3 never retrieved.
8. Client B's stream completes with M4, M5, … — M1–M3 permanently absent, no error raised.
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }
```
