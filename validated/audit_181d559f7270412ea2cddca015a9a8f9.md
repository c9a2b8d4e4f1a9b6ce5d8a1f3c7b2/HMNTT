## Audit Report

## Title
Unbounded Growth of `cacheManagerSlotsPerContract` Internal Cache Map via Unique `contractId` Flooding

## Summary
In `ContractStateServiceImpl.findStorageBatch()`, each unique `contractId` that misses the outer bounded `contractSlotsCache` triggers `cacheManagerSlotsPerContract.getCache(contractId.toString())`. Because `cacheManagerSlotsPerContract` is a `CaffeineCacheManager` configured without `setCacheNames()` (dynamic mode), Spring permanently registers every new cache name in its internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no eviction mechanism. The outer `contractSlotsCache` is bounded and evicts entries, but those evictions never propagate to `cacheManagerSlotsPerContract.cacheMap`, causing it to grow without bound.

## Finding Description

**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 85–87:

```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [2](#0-1) 

**Root cause — two-layer cache mismatch:**

- `cacheManagerContractSlots` (`EvmConfiguration.java` lines 75–81) calls `setCacheNames(Set.of(CACHE_NAME))` and sets a `maximumSize` via `cacheProperties.getContractSlots()`. It is a **bounded** Caffeine cache mapping `contractId → CaffeineCache`. [3](#0-2) 

- `cacheManagerSlotsPerContract` (`EvmConfiguration.java` lines 107–112) does **not** call `setCacheNames()`, placing it in Spring's dynamic mode. In dynamic mode, `CaffeineCacheManager.getCache(name)` creates a new `CaffeineCache` and inserts it permanently into an internal `ConcurrentHashMap<String, Cache> cacheMap` with no size limit and no eviction. [1](#0-0) 

**Exploit flow:**

1. For each new unique `contractId` not in `this.contractSlotsCache`, the value loader `() -> cacheManagerSlotsPerContract.getCache(contractId.toString())` is invoked.
2. `CaffeineCacheManager.getCache()` creates a new `CaffeineCache` and inserts it into `cacheMap` under `contractId.toString()`.
3. The resulting `CaffeineCache` is stored in `this.contractSlotsCache` (bounded).
4. When `this.contractSlotsCache` reaches `maximumSize` and evicts an old `contractId` entry, the corresponding entry in `cacheManagerSlotsPerContract.cacheMap` is **never removed** — no removal listener or cleanup path exists.
5. Repeating with N unique `contractId` values causes `cacheManagerSlotsPerContract.cacheMap` to accumulate N entries indefinitely.

**Why existing checks are insufficient:**

`ContractStorageReadableKVState.readFromDataSource()` performs no validation that the `contractId` corresponds to an existing on-chain contract before calling `contractStateService.findStorage(entityId, keyBytes)`. [4](#0-3) 

`EntityIdUtils.entityIdFromContractId` only extracts the `contractNum` field (shard.realm.num), so the effective key space is the 38-bit entity number (~274 billion values) — more than sufficient for a sustained attack. [5](#0-4) 

## Impact Explanation
Each entry permanently retained in `cacheManagerSlotsPerContract.cacheMap` holds a `String` key and a `CaffeineCache` wrapping a Caffeine cache instance. Even an empty Caffeine cache carries internal data structures (node arrays, internal queues, scheduler references). With a sufficiently large number of unique `contractId` values, the JVM heap is exhausted, causing `OutOfMemoryError` and crashing the mirror node web3 service. This denies EVM query service (`eth_call`, `eth_estimateGas`, `eth_getStorageAt`, contract storage reads) to all users.

## Likelihood Explanation
An attacker needs only to send repeated JSON-RPC requests (e.g., `eth_getStorageAt` or `eth_call`) referencing different contract addresses that map to distinct Hedera entity numbers (long-zero EVM addresses). No privileged access, no HBAR spend, and no special tooling is required — a simple HTTP client loop incrementing the entity number suffices. The attack is persistent across JVM restarts as long as the attacker continues sending requests.

## Recommendation

1. **Remove `cacheManagerSlotsPerContract` as a `CacheManager` factory.** Instead, create `CaffeineCache` instances directly using a shared `Caffeine` spec, bypassing `CaffeineCacheManager`'s internal registry entirely.

2. **Alternatively**, register a Caffeine removal listener on `contractSlotsCache` that calls a cleanup method to remove the corresponding entry from `cacheManagerSlotsPerContract` when an entry is evicted.

3. **Add contract existence validation** in `ContractStorageReadableKVState.readFromDataSource()` before invoking `findStorage`, rejecting requests for non-existent contracts early.

## Proof of Concept

```python
import requests, itertools

url = "http://<mirror-node-web3>:8545"
for i in itertools.count(1_000_000):  # start beyond real contracts
    payload = {
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [
            # long-zero address encoding entity 0.0.i
            "0x" + "00" * 12 + i.to_bytes(8, "big").hex(),
            "0x0",
            "latest"
        ],
        "id": i
    }
    requests.post(url, json=payload)
    # Each unique address causes a permanent entry in cacheManagerSlotsPerContract.cacheMap
```

Each iteration adds one permanent entry to `cacheManagerSlotsPerContract.cacheMap`. The outer `contractSlotsCache` evicts old entries but never cleans up the inner map, causing monotonically increasing heap consumption until `OutOfMemoryError`.
