Audit Report

## Title
`isValidAccount()` Accepts Deleted Entities — Staking Rewards Exposed for Deleted Accounts

## Summary
`EntityService.isValidAccount()` queries the `entity` table without filtering on the `deleted` column. As a result, `listStakingRewardsByAccountId` in `AccountController` returns HTTP 200 with full staking reward transfer history for deleted accounts instead of HTTP 404, contrary to the design intent demonstrated by the alias and EVM-address lookup paths.

## Finding Description

**Root cause — missing `deleted` filter in `entityExistenceQuery`:**

`rest/service/entityService.js` defines three queries. The alias and EVM-address lookups both explicitly exclude deleted entities:

- Alias lookup: `where coalesce(${Entity.DELETED}, false) <> true and ${Entity.ALIAS} = $1` [1](#0-0) 

- EVM-address lookup: `where ${Entity.DELETED} <> true and ${Entity.EVM_ADDRESS} = $1` [2](#0-1) 

However, `entityExistenceQuery` — used exclusively by `isValidAccount()` — has no such guard:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;
``` [3](#0-2) 

`isValidAccount()` returns `true` for any row returned, including rows where `deleted = true`:

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
``` [4](#0-3) 

**Exploit path in `listStakingRewardsByAccountId`:**

The controller calls `isValidAccount()` as its only guard before querying staking rewards:

```js
listStakingRewardsByAccountId = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[...]);
  const isValidAccount = await EntityService.isValidAccount(accountId);
  if (!isValidAccount) {
    throw new NotFoundError();   // never reached for deleted accounts
  }
  ...
  const stakingRewardsTransfers = await StakingRewardTransferService.getRewards(...);
``` [5](#0-4) 

Because `entityExistenceQuery` returns a row for a deleted entity (the row still exists in the `entity` table with `deleted = true`), `isValidAccount()` returns `true`, the `NotFoundError` is never thrown, and the full staking reward transfer history is served.

## Impact Explanation
Any unauthenticated caller can retrieve the complete staking reward transfer history for accounts that have been deleted from the network. The REST API is supposed to return HTTP 404 for deleted accounts — as confirmed by the explicit `deleted` guards on the alias and EVM-address lookup paths — but instead returns HTTP 200 with financial history. This constitutes unauthorized disclosure of financial transaction records that the protocol explicitly marks as inaccessible post-deletion.

## Likelihood Explanation
No privileges are required. The attacker only needs a numeric account ID (`0.0.<num>`) of a deleted account. Account IDs are sequential integers, making enumeration trivial. The endpoint `/api/v1/accounts/<id>/rewards` is publicly reachable. The attack is fully repeatable with a standard HTTP client.

## Recommendation
Add a `deleted` filter to `entityExistenceQuery` in `rest/service/entityService.js`, consistent with the pattern already used by the alias and EVM-address queries:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where coalesce(${Entity.DELETED}, false) <> true
                                 and ${Entity.ID} = $1`;
``` [3](#0-2) 

## Proof of Concept
1. Identify a deleted account (e.g., `0.0.12345`) — any previously valid account ID works since IDs are sequential.
2. Send: `GET /api/v1/accounts/0.0.12345/rewards`
3. Expected: HTTP 404 (account is deleted).
4. Actual: HTTP 200 with the full `rewards` array from `staking_reward_transfer` for that account.

The `entityExistenceQuery` at `rest/service/entityService.js:28-30` returns the row (since `deleted = true` rows are not filtered), `isValidAccount()` at line 62 returns `true`, the `NotFoundError` at `accountController.js:174` is never thrown, and `StakingRewardTransferService.getRewards()` at line 180 executes and returns the data. [6](#0-5)
