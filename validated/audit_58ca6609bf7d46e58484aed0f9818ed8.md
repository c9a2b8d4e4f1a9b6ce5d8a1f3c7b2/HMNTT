Audit Report

## Title
206 Partial Content Response Incorrectly Cached, Poisoning Subsequent Users with Incomplete Transaction History

## Summary
`responseCacheUpdateHandler()` admits HTTP 206 responses into the Redis cache because `httpStatusCodes.isSuccess()` returns `true` for any code in `[200, 300)`, including 206. The application deliberately emits 206 to signal that the mirror node has not yet fully ingested data. Once a 206 response is cached under the normal URL-hash key, every subsequent request for that URL is served the stale partial body with status 206 for the full cache TTL, even after the mirror node has caught up.

## Finding Description

**`isSuccess` predicate — `rest/constants.js` line 159:** [1](#0-0) 

`code >= 200 && code < 300` includes 206.

**Cache-admission guard — `rest/middleware/responseCacheHandler.js` line 95:** [2](#0-1) 

The condition `isUnmodified || httpStatusCodes.isSuccess(res.statusCode)` has no exclusion for 206.

**Status code stored verbatim — line 114:** [3](#0-2) 

For non-304 responses, `res.statusCode` (which is 206 at this point) is written directly into the `CachedApiResponse` object and persisted to Redis.

**206 emitted for NFT transaction history — `rest/tokens.js` lines 1048–1050:** [4](#0-3) 

When `nft?.createdTimestamp == null` (mint transaction not yet ingested), `res.locals.statusCode` is set to `PARTIAL_CONTENT.code` (206).

**206 emitted for contract results — `rest/controllers/contractController.js` lines 1026–1028:** [5](#0-4) 

When `contractResults[0].callResult` is nil, `res.locals.statusCode` is set to 206.

The `responseHandler.js` middleware (confirmed to reference `res.locals.statusCode`) applies this value to the actual HTTP response before `responseCacheUpdateHandler` runs, so `res.statusCode` is 206 at cache-write time.

The 304 → 200 remapping on line 114 shows the developers were aware that raw status codes need normalisation before caching, but no analogous guard exists for 206. [6](#0-5) 

## Impact Explanation
Any client that queries `/api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions` or `/api/v1/contracts/results/{transactionIdOrHash}` while the mirror node has incomplete data causes a 206 partial response to be written to Redis under the URL-hash key with the full `max-age` TTL. All subsequent clients querying the same URL receive the cached partial body and status 206 for the entire TTL, even after the mirror node has fully ingested the missing data and would return a complete 200 OK. For NFT transaction history this means the mint transaction is absent from every cached response, presenting a falsified provenance record to all consumers of that cache entry.

## Likelihood Explanation
Mirror nodes routinely have transient windows of incomplete data during normal sync operation. No authentication, special privileges, or crafted payload are required — a plain unauthenticated GET request is sufficient. An attacker who monitors mirror-node lag (e.g., by comparing the latest consensus timestamp in responses against the network's known head) can reliably time a request to land during an incomplete-data window, guaranteeing cache poisoning for the full TTL. The attack is repeatable: after TTL expiry the attacker can re-trigger it with another GET if the data is still incomplete, or simply wait for the next sync gap.

## Recommendation
Exclude HTTP 206 from the cache-admission condition in `responseCacheUpdateHandler`. The minimal fix is:

```js
// rest/middleware/responseCacheHandler.js line 95
if (
  responseBody &&
  responseCacheKey &&
  (isUnmodified || (httpStatusCodes.isSuccess(res.statusCode) &&
    res.statusCode !== httpStatusCodes.PARTIAL_CONTENT.code))
) {
```

Alternatively, define a separate `isCacheable` predicate in `constants.js` that explicitly excludes 206, mirroring the existing special-case handling of 304.

## Proof of Concept
1. Identify a token/NFT whose mint transaction has not yet been ingested by the mirror node (observable via a 206 response on the endpoint).
2. Send `GET /api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions` — the mirror node returns 206 with a partial body; `responseCacheUpdateHandler` writes this to Redis.
3. Wait for the mirror node to fully ingest the mint transaction (it would now return 200 with complete data).
4. Send the same GET request again — the response is served from Redis with status 206 and the incomplete body, missing the mint transaction, for the remainder of the TTL.
5. The cache entry can be refreshed by the attacker after TTL expiry by repeating step 2 during any subsequent sync gap.
