Audit Report

## Title
Missing `lower(timestamp_range)` Guard in Entity Branch Causes Historical Query to Return Post-Modification State

## Summary
`findActiveByEvmAddressAndTimestamp()` and `findActiveByEvmAddressOrAliasAndTimestamp()` in `EntityRepository.java` omit a `lower(timestamp_range) <= blockTimestamp` filter on the live `entity` table branch of their `UNION ALL`. As a result, any historical query whose `blockTimestamp` falls in the window `[created_timestamp, lower(entity.timestamp_range))` — i.e., after entity creation but before a subsequent modification — unconditionally returns the post-modification (current) entity state instead of the state valid at the queried block.

## Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`

Both affected methods share the same structural defect. Shown for `findActiveByEvmAddressOrAliasAndTimestamp` (lines 95–121):

```sql
with entity_cte as (
    select id
    from entity
    where created_timestamp <= ?2          -- (A) anchors on immutable birth time
      and (evm_address = ?1 or alias = ?1)
    order by created_timestamp desc
    limit 1
)
(
    select *
    from entity e
    where e.deleted is not true
      and e.id = (select id from entity_cte)
    -- (B) *** NO lower(timestamp_range) <= ?2 guard ***
)
union all
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2  -- (C) correctly guarded
      and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
order by timestamp_range desc
limit 1
``` [1](#0-0) 

**Root cause:** Guard (A) uses `created_timestamp`, which is the entity's immutable birth timestamp and never changes on modification. After a modification at T2, the live `entity` row retains `created_timestamp = T1` but its `timestamp_range` advances to `[T2, ∞)`. Because guard (B) is absent, the live row is always included in the `UNION ALL` regardless of `blockTimestamp`. The final `ORDER BY timestamp_range DESC LIMIT 1` then selects `[T2, ∞)` over the history row `[T1, T2)` for any query with `blockTimestamp ∈ [T1, T2)`.

**Contrast with the correct pattern** in `findActiveByIdAndTimestamp` (lines 136–155), which correctly applies `lower(timestamp_range) <= ?2` on the live `entity` branch: [2](#0-1) 

The same defect is present in `findActiveByEvmAddressAndTimestamp` (lines 59–85): [3](#0-2) 

## Impact Explanation
Any historical EVM simulation that resolves an account or contract by EVM address or alias receives incorrect entity state for blocks in the window `[T1, T2)`. Concretely: balance, nonce, contract bytecode pointer, key material, and deletion status as of the queried block may all be wrong. A contract deleted and re-created between T1 and T2 could appear with its post-recreation state; a contract that did not yet exist at the queried block could appear to exist. This breaks the correctness guarantee of `eth_call` at historical blocks, which is relied upon by block explorers, auditing tools, and DeFi protocols that replay historical transactions.

Under replication lag, the `entity_history` branch (C) returns nothing at all, making the incorrect current-state result the sole candidate — amplifying the defect under normal high-load or failover conditions.

## Likelihood Explanation
No privilege is required. Any caller who can issue a JSON-RPC `eth_call` (or `eth_getBalance`, etc.) with a `blockNumber` parameter can trigger this. The attacker only needs to know the EVM address or alias of any entity that has ever been modified — a trivially public fact on a transparent ledger — and supply a block number that predates the modification. The base logic bug fires unconditionally on a fully-synced primary; the replication-lag variant requires only a transient infrastructure condition that is normal under high write load or failover.

## Recommendation
Add `lower(e.timestamp_range) <= ?2` to the live `entity` branch in both affected methods, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
(
    select *
    from entity e
    where e.deleted is not true
      and lower(e.timestamp_range) <= ?2   -- add this guard
      and e.id = (select id from entity_cte)
)
```

Apply the same fix to both `findActiveByEvmAddressAndTimestamp` (line 70) and `findActiveByEvmAddressOrAliasAndTimestamp` (line 106).

## Proof of Concept
1. Entity E is created at consensus timestamp T1; `entity.created_timestamp = T1`, `entity.timestamp_range = [T1, ∞)`.
2. E is modified at T2 > T1; `entity.timestamp_range` becomes `[T2, ∞)`, old state moves to `entity_history` with `timestamp_range = [T1, T2)`.
3. Issue a historical `eth_call` specifying a block whose consensus timestamp is T1 (any value in `[T1, T2)`).
4. The web3 service calls `findActiveByEvmAddressOrAliasAndTimestamp(alias, T1)`.
5. `entity_cte` resolves the ID (`created_timestamp T1 ≤ T1` ✓). Branch (B) returns the current entity row with `timestamp_range = [T2, ∞)` (no guard). Branch (C) returns the history row with `timestamp_range = [T1, T2)`.
6. `ORDER BY timestamp_range DESC LIMIT 1` selects `[T2, ∞)` because `T2 > T1`.
7. The caller receives the post-modification entity state for a block that predates the modification — confirming the bug.
