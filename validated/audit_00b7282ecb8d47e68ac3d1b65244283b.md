# Audit Report

## Title
Indefinite gRPC Subscription via `Long.MAX_VALUE` End Time Causes Unbounded Resource Exhaustion

## Summary
An unauthenticated caller can send a `ConsensusTopicQuery` with `consensusEndTime.seconds >= 9223372035L`, causing the server to hold a gRPC subscription open indefinitely. The completion check `TopicContext.isComplete()` resolves `Long.MAX_VALUE` nanoseconds to the year ~2262 and never returns `true`, so the stream, its DB polling loop, and its scheduler threads are never released. An attacker opening many such connections can exhaust server-side resources.

## Finding Description

**Step 1 — `convertTimestamp()` produces `Long.MAX_VALUE`**

`ConsensusController.convertTimestamp()` explicitly returns `Long.MAX_VALUE` for any timestamp with `seconds >= 9223372035L`: [1](#0-0) 

`toFilter()` passes this value directly into the filter's `endTime` field with no further clamping: [2](#0-1) 

**Step 2 — Validation passes**

`TopicMessageFilter.isValidEndTime()` only checks `endTime > startTime`. Since `Long.MAX_VALUE` is always greater than any current nanosecond timestamp, the `@AssertTrue` constraint is satisfied and the filter is accepted: [3](#0-2) 

**Step 3 — `isComplete()` never returns `true`**

`TopicContext.isComplete()` converts `endTime` to an `Instant` via `Instant.ofEpochSecond(0, Long.MAX_VALUE)`, which resolves to approximately the year 2262. Adding the 30-second `endTimeInterval` still yields a far-future instant, so `.isBefore(Instant.now())` is permanently `false`: [4](#0-3) 

**Step 4 — `pastEndTime()` never signals termination**

`pastEndTime()` is the only mechanism that can terminate the live-message flux via `takeUntilOther`. It emits a completion signal only when `isComplete()` returns `true`. Since that never happens, the `RepeatSpec` predicate `!topicContext.isComplete()` is always `true`, and `takeUntilOther` never fires: [5](#0-4) 

**Step 5 — `takeWhile` on `endTime` also never terminates**

The secondary guard `takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime())` is permanently `true` because real consensus timestamps (~`1.7 × 10¹⁸`) are always less than `Long.MAX_VALUE` (`9.2 × 10¹⁸`): [6](#0-5) 

## Impact Explanation

Each malicious subscription holds a live gRPC stream open indefinitely. The stream drives a DB polling loop on a `boundedElastic` scheduler thread and a `safetyCheck` flux on another `boundedElastic` thread, and increments the `subscriberCount` gauge with no enforcement cap. [7](#0-6) 

The only per-connection limit is `maxConcurrentCallsPerConnection = 5`: [8](#0-7) 

An attacker can open arbitrarily many TCP connections, each carrying 5 such subscriptions. There is no global subscriber cap, no per-IP rate limit, and no authentication requirement. Thread pool exhaustion, memory pressure, and database connection saturation are all realistic outcomes.

## Likelihood Explanation

The exploit requires zero privileges — the gRPC `subscribeTopic` endpoint is publicly accessible. The trigger is a single protobuf field set to a value `>= 9223372035` seconds. The attacker needs a valid topic ID (since `checkTopicExists` defaults to `true`): [9](#0-8) 

Valid topic IDs are publicly observable on-chain. The attack is trivially scriptable and parallelizable across many connections from a single host.

## Recommendation

1. **Clamp `endTime` to a reasonable maximum** in `convertTimestamp()` or `toFilter()`. Instead of returning `Long.MAX_VALUE`, cap the value to `currentTimeNanos + some_max_window` (e.g., 1 hour). This prevents the far-future sentinel from bypassing `isComplete()`.
2. **Add a global subscriber cap** enforced in `subscribeTopic()` — reject new subscriptions when `subscriberCount` exceeds a configurable threshold.
3. **Add a maximum `endTime` validation constraint** in `TopicMessageFilter.isValidEndTime()` — reject any `endTime` beyond a reasonable future bound relative to the current time.
4. **Consider per-IP connection rate limiting** at the infrastructure or gRPC interceptor level.

## Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import timestamp_pb2, basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# Set consensusEndTime.seconds >= 9223372035 to trigger Long.MAX_VALUE path
query = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),  # any valid topic
    consensusEndTime=timestamp_pb2.Timestamp(seconds=9223372036)
)

# This stream never completes; repeat across many connections to exhaust resources
for msg in stub.subscribeTopic(query):
    pass
```

Each such call holds a gRPC stream, a `boundedElastic` DB polling thread, and a `safetyCheck` thread open indefinitely. Opening hundreds of connections from a single host saturates the thread pool and database connection pool.

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
