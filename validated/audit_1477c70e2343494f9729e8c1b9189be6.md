# Audit Report

## Title
Gas Throttle Bucket Fully Bypassed via Deliberate Out-of-Gas Exceptional Halt

## Summary
`ContractCallService.restoreGasToBucket()` unconditionally restores `gasLimitRefundPercent`% (default 100%) of the submitted `gasLimit` back to the `gasLimitBucket` whenever a transaction result is failed **and** `gasUsed == gasLimit`. An out-of-gas exceptional halt in the EVM satisfies this condition by design: the EVM sets `gasUsed = gasLimit` and marks the result as failed. An unauthenticated attacker can deliberately engineer this condition by submitting requests with a gas value that passes the bucket check but is too low to complete execution, resulting in zero net token consumption from the gas throttle bucket.

**Root cause:** The condition `!result.isSuccessful() && gasLimit == result.gasUsed()` was designed to handle infrastructure pre-check failures (where the EVM was never invoked and `gasUsed` is reported as `gasLimit` by convention). However, an EVM out-of-gas exceptional halt also satisfies this condition: the EVM executes until gas is exhausted, sets `gasUsed = gasLimit`, and returns a failed `EvmTransactionResult`. The code's own comment ŌĆö *"If the transaction fails, gasUsed is equal to gasLimit"* ŌĆö conflates these two distinct cases. [5](#0-4) 

`EvmTransactionResult.isSuccessful()` returns `false` for any non-`SUCCESS` response code, and `gasUsed()` reads directly from `ContractFunctionResult.gasUsed()`, which the EVM sets to `gasLimit` on an out-of-gas halt. [6](#0-5) 

**Net token accounting per request:**
- Consumed: `scaleGas(gasLimit)` tokens
- Restored: `scaleGas(gasLimit)` tokens (with 100% refund default)
- **Net: 0**

## Impact Explanation

The `gasLimitBucket` is the primary defense against high-gas-cost request floods on the mirror node's EVM simulation endpoint (`/api/v1/contracts/call`). With this bypass, an attacker can submit requests at the full request-rate limit (default 500 req/s, confirmed in `ThrottleProperties.java:35`) with a gas value up to `evmProperties.maxGasLimit` (default 15,000,000 per `docs/configuration.md:710`) without ever depleting the gas bucket. The EVM still executes until it runs out of gas on each call, consuming real CPU and memory on the server. The gas throttle provides zero protection; only the weaker request-rate bucket remains as a constraint. [7](#0-6) 

## Likelihood Explanation

No privileges, authentication, or special contract deployment is required. Any caller can send a standard `eth_call` JSON-RPC request to `POST /api/v1/contracts/call` with a deliberately low gas value. The condition is trivially reproducible: set `gas` to any value `> 10,000` (to pass `scaleGas`) but insufficient to complete the target contract's execution. The attack is stateless, repeatable, and requires no on-chain state. [8](#0-7) 

## Recommendation

The fix must distinguish between infrastructure pre-check failures (EVM never invoked) and EVM-level exceptional halts (EVM ran and consumed real resources). Two complementary approaches:

1. **Do not restore gas on out-of-gas halts.** Inspect the failure reason from `EvmTransactionResult` to determine whether the failure was a pre-check (no EVM execution occurred) or an EVM-level halt. Only apply the full refund for pre-check failures. For out-of-gas halts, treat `gasUsed = gasLimit` as fully consumed and restore nothing (or only the standard `gasRemaining = 0` path).

2. **Reduce `gasLimitRefundPercent` default.** Setting `hiero.mirror.web3.throttle.gasLimitRefundPercent` to a value significantly below 100 (e.g., 0ŌĆō25) limits the maximum refund even in the worst case, reducing the bypass effectiveness. This is a mitigation, not a fix. [2](#0-1)
