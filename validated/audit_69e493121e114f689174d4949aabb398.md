# Audit Report

## Title
Unbounded Per-Subscription Heap Allocation in `PollingTopicListener.poll()` Enables Memory Exhaustion via Concurrent Subscriptions

## Summary
When the gRPC listener is configured in `POLL` mode, each subscription independently calls `PollingTopicListener.poll()` every 500 ms, which fully materializes up to 5,000 `TopicMessage` objects into heap via `getResultList()`. There is no global cap on concurrent subscriptions, and the per-connection call limit is trivially bypassed by opening many connections. An unauthenticated attacker can exhaust JVM heap and crash the mirror node process.

## Finding Description

**Code path:**

`PollingTopicListener.poll()` computes `pageSize = Math.min(Integer.MAX_VALUE, 5000)` when no limit is set, then calls: [1](#0-0) 

```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`TopicMessageRepositoryCustomImpl.findByFilter()` calls `getResultList()`, which fully materializes all matching rows into a `java.util.List<TopicMessage>` before wrapping as a `Stream`: [2](#0-1) 

```java
return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
```

**Root cause:** `Flux.fromStream()` does not provide lazy/streaming semantics here because the underlying `Stream` is backed by an already-fully-loaded `List`. Every poll cycle per subscription allocates a full page of up to 5,000 objects on the heap simultaneously.

**Why existing checks fail:**

1. Any per-connection call limit restricts calls *per TCP connection*, not globally. An attacker opens N connections × calls/connection = many total subscriptions with no server-side total cap.

2. `TopicMessageFilter.isValidStartTime()` only requires `startTime <= DomainUtils.now()`. Epoch 0 (year 1970) is fully valid, guaranteeing the maximum number of historical messages is returned on every poll. [4](#0-3) 

3. `TopicMessageServiceImpl.subscriberCount` is only a metrics gauge — it is never used to reject new subscriptions. [3](#0-2) 

4. No authentication is required to call the gRPC `subscribeTopic` endpoint.

**Precondition:** The listener type must be set to `POLL` (`hiero.mirror.grpc.listener.type=POLL`). The default is `REDIS`, but `POLL` is a documented and supported configuration. [7](#0-6) 

## Impact Explanation

Each subscription in `POLL` mode allocates up to 5,000 `TopicMessage` objects (~200–500 bytes each) per 500 ms poll cycle. With a large number of concurrent subscriptions, steady-state heap pressure can reach multiple gigabytes allocated every 500 ms. This triggers `OutOfMemoryError`, crashing the JVM process and taking the mirror node gRPC API offline entirely — a complete denial of service for all consumers of HCS topic data. [6](#0-5) 

## Likelihood Explanation

The attack requires no credentials, no special protocol knowledge beyond standard gRPC, and no prior access. Opening many HTTP/2 connections and issuing streaming RPCs per connection is trivially scriptable with any gRPC client library. The attack is repeatable and can be sustained indefinitely. The only barrier is that the operator must have chosen `POLL` mode over the default `REDIS` mode, which reduces the exposed population but does not eliminate it.

## Recommendation

1. **Enforce a global subscription cap:** Use `subscriberCount` (already tracked) to reject new subscriptions above a configurable threshold before allocating any resources. [8](#0-7) 

2. **Prefer `SHARED_POLL` over `POLL`:** `SHARED_POLL` shares a single poll across all subscribers for the same topic, eliminating per-subscription heap allocation. Document that `POLL` mode is unsafe for public-facing deployments. [9](#0-8) 

3. **Bound `startTime` more aggressively:** Reject or warn on `startTime` values that would return very large historical result sets (e.g., require `startTime` to be within a configurable recent window).

4. **Add rate limiting per client IP** at the gRPC server level to limit connection/subscription creation rate.

## Proof of Concept

```python
import grpc
import concurrent.futures
from mirror_node_pb2_grpc import ConsensusServiceStub
from mirror_node_pb2 import ConsensusTopicQuery

def flood_subscribe(conn_id):
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(channel)
    # startTime=0 (epoch) maximizes historical rows returned per poll
    query = ConsensusTopicQuery(topic_id=..., consensus_start_time=0)
    futures = []
    for _ in range(5):  # 5 subscriptions per connection
        futures.append(stub.subscribeTopic(query))
    # Hold subscriptions open indefinitely
    for f in futures:
        for _ in f:
            pass

# Open 200 connections × 5 subscriptions = 1,000 concurrent POLL cycles
# Each cycle: 5,000 TopicMessage objects × ~400 bytes × 1,000 = ~2 GB/500ms
with concurrent.futures.ThreadPoolExecutor(max_workers=200) as ex:
    list(ex.map(flood_subscribe, range(200)))
```

Each active subscription triggers `getResultList()` every 500 ms, materializing up to 5,000 objects. With 1,000 concurrent subscriptions, heap exhaustion and `OutOfMemoryError` follow rapidly. [10](#0-9)
