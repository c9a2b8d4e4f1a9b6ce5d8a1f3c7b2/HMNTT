# Audit Report

## Title
Cache Key Collision via `Arrays.hashCode` in `SpelHelper.hashCode(byte[])` Allows Wrong Entity Served for EVM Address Lookups

## Summary
`SpelHelper.hashCode(byte[])` uses `Arrays.hashCode`, which returns a 32-bit `int`, as the sole cache key for byte-array EVM address and alias lookups in the web3 `EntityRepository`. Because the key space is only 2^32 while EVM addresses occupy 2^160 possible values, hash collisions are computationally trivial to find. Two distinct on-chain EVM addresses that collide under `Arrays.hashCode` will share a cache slot, causing the mirror node's web3 layer to serve the wrong entity record.

## Finding Description

**Verified code path:**

`SpelHelper.hashCode(byte[])` at `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22: [1](#0-0) 

```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```

This method is registered as a Spring bean named `spelHelper` in `CommonConfiguration.spelHelper()` at `common/src/main/java/org/hiero/mirror/common/CommonConfiguration.java`, lines 55–58: [2](#0-1) 

It is used as the cache key expression in two `@Cacheable` annotations in `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`: [3](#0-2) 

```java
@Cacheable(
    cacheNames = CACHE_NAME_EVM_ADDRESS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);

@Cacheable(
    cacheNames = CACHE_NAME_ALIAS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
```

Both caches (`"evmAddress"` and `"alias"`) are backed by the same `CACHE_MANAGER_ENTITY` Caffeine cache manager, configured at `web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java`, lines 99–105: [4](#0-3) 

**Root cause:** `Arrays.hashCode(byte[])` computes a polynomial rolling hash (`result = 31 * result + (b & 0xff)`) and returns a 32-bit `int`. The output space is 2^32 ≈ 4 billion values. For 20-byte EVM addresses (2^160 possible inputs), the birthday bound gives a 50% collision probability after only ~65,536 distinct addresses are hashed. The cache stores and retrieves entries using only this 32-bit integer as the key — the original byte array is never stored or compared at retrieval time.

**Failed assumption:** The design assumes `Arrays.hashCode` produces unique keys for distinct byte arrays. It does not. Two distinct byte arrays `A` and `B` where `Arrays.hashCode(A) == Arrays.hashCode(B)` map to the same cache slot, and whichever is cached first is returned for both.

**Why existing guards are insufficient:** The `unless = "#result == null"` condition only prevents caching null results. It performs no equality check between the stored byte array and the incoming byte array at retrieval time.

**Exploit flow:**
1. Attacker enumerates existing EVM addresses via the public REST API.
2. Attacker computes `Arrays.hashCode` for each address offline and finds two addresses `A` and `B` with the same hash value (statistically guaranteed on mainnet with millions of accounts).
3. Attacker sends a web3 call (e.g., `eth_call`) triggering `findByEvmAddressAndDeletedIsFalse(A)` — entity A is fetched from DB and stored in cache under key `H`.
4. Attacker sends a second call triggering `findByEvmAddressAndDeletedIsFalse(B)` — cache returns entity A's record instead of querying the DB for entity B.
5. The web3 execution layer proceeds with entity A's data (wrong account ID, wrong balance, wrong keys) for address B.

## Impact Explanation
The web3 module uses these cached entity lookups to resolve EVM addresses to Hedera entities during contract execution (`eth_call`, `eth_estimateGas`). Serving the wrong entity for an EVM address means incorrect account balance, type, or key material is used in EVM execution, producing incorrect contract call results returned to callers. An attacker can make a contract call appear to succeed or fail incorrectly by poisoning the cache with a colliding address they control.

## Likelihood Explanation
The attack requires no privileges — only access to the public web3 API and the public REST API. Finding a collision in `Arrays.hashCode` over 20-byte inputs is trivial: the birthday bound gives ~50% probability after ~65,536 addresses. On Hedera mainnet with millions of accounts, natural collisions among existing addresses are virtually certain to exist. The attacker can pre-compute the collision offline and execute the cache-poisoning in two sequential API calls with no special tooling.

## Recommendation
Replace the 32-bit `Arrays.hashCode` key with the full byte array as the cache key. Spring's Caffeine-backed `ConcurrentHashMap` uses `equals()` for key comparison, so using the raw `byte[]` directly will not work (arrays use reference equality). Instead:

- Wrap the byte array in a `ByteBuffer` or `HexString` (e.g., `HexFormat.of().formatHex(#alias)`) as the cache key, which provides correct value-based equality.
- Alternatively, use `java.util.Arrays.toString(#alias)` or encode to hex in SpEL.
- Remove `SpelHelper.hashCode(byte[])` entirely or document that it must never be used as a cache key.

## Proof of Concept
Two byte arrays that collide under `Arrays.hashCode` can be found by brute force in milliseconds. For example, any two 20-byte arrays `A` and `B` satisfying the polynomial recurrence `31^20*a[0] + ... + a[19] ≡ 31^20*b[0] + ... + b[19] (mod 2^32)` are a valid collision pair. With ~65,536 random 20-byte inputs, the birthday bound guarantees a collision with ~50% probability. On a network with millions of accounts, natural collisions among real on-chain EVM addresses are statistically near-certain and can be identified by downloading the account list from the public REST API and computing `Arrays.hashCode` for each address.
