Audit Report

## Title
Unbounded `limit=Long.MAX_VALUE` Causes Integer Overflow, Indefinite Subscription, and Repeated Unbound DB Queries via `subscribeTopic()`

## Summary
An unprivileged caller can send a `ConsensusTopicQuery` with `limit` set to `Long.MAX_VALUE`. This value passes the only validation guard (`@Min(0)`), triggers a silent integer overflow in the polling layer that removes the SQL `LIMIT` clause entirely, and holds the gRPC stream open indefinitely. Every poll interval issues an unbounded query against the `topic_message` table, exhausting database and server resources.

## Finding Description

**Step 1 — No upper-bound check on ingress.**
`ConsensusController.toFilter()` copies `query.getLimit()` verbatim into `TopicMessageFilter.limit` with no ceiling check. [1](#0-0) 

**Step 2 — `@Min(0)` is the only constraint; `Long.MAX_VALUE` satisfies it.**
`TopicMessageFilter.limit` carries only `@Min(0)`. `Long.MAX_VALUE` is positive, so it passes. `hasLimit()` returns `limit > 0`, which is `true` for `Long.MAX_VALUE`. [2](#0-1) [3](#0-2) 

**Step 3 — `flux.take(Long.MAX_VALUE)` never terminates by count.**
`TopicMessageServiceImpl.subscribeTopic()` calls `flux.take(filter.getLimit())`. In Project Reactor, `Flux.take(Long.MAX_VALUE)` is documented to never complete by count — the subscription is held open indefinitely. [4](#0-3) 

**Step 4 — Integer overflow in `PollingTopicMessageRetriever.poll()` (root cause of missing LIMIT).**
```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```
`(int)(Long.MAX_VALUE - 0L)` silently truncates to `-1`. `Math.min(-1, maxPageSize)` = `-1`. The rebuilt filter is stored with `limit = -1L`. [5](#0-4) 

**Step 5 — Same overflow in `PollingTopicListener.poll()`.**
The live-message path performs the identical `(int)(filter.getLimit() - context.getCount().get())` cast, producing `pageSize = -1`, and polls on every interval (default 500 ms) via `RepeatSpec.times(Long.MAX_VALUE)` — no termination condition. [6](#0-5) [7](#0-6) 

**Step 6 — `setMaxResults` is never called; query runs without a row limit.**
`TopicMessageRepositoryCustomImpl.findByFilter()` only calls `typedQuery.setMaxResults(...)` when `filter.hasLimit()` is true. Because the rebuilt filter has `limit = -1`, `hasLimit()` returns `(-1 > 0)` = `false`, so `setMaxResults` is skipped. The JPA query executes with no `LIMIT` clause. [8](#0-7) 

**Step 7 — `PollingContext.isComplete()` never fires the limit-hit branch.**
`limitHit` is `filter.getLimit() == total.get()`, i.e., `Long.MAX_VALUE == total`, which is practically unreachable. For the throttled retriever the `pageSize < maxPageSize` branch can eventually fire (when no new messages exist), but the live `PollingTopicListener` has no such guard and polls indefinitely. [9](#0-8) 

**Step 8 — No global subscriber cap.**
`maxConcurrentCallsPerConnection = 5` is per-TCP-connection only. An attacker opens N connections to multiply concurrent subscriptions without bound. [10](#0-9) 

**Why existing checks fail — the split-brain:**
- `@Min(0)` has no `@Max`; `Long.MAX_VALUE` is accepted.
- The `(int)` narrowing cast is silent; no overflow guard exists.
- `hasLimit()` uses `> 0` semantics. After overflow the rebuilt filter has `limit = -1`, so `hasLimit()` returns `false` at the DB layer (no `LIMIT` clause) while the outer `flux.take(Long.MAX_VALUE)` treats the original filter as "has limit" — bypassing both guards simultaneously.

## Impact Explanation
Each malicious subscription causes the live `PollingTopicListener` to issue an unbounded query against `topic_message` on every poll interval (default 500 ms) for the lifetime of the connection. With N connections × 5 concurrent calls, an attacker saturates the PostgreSQL connection pool and I/O bandwidth. Legitimate HCS subscribers time out or receive no messages. For applications using HCS as a fund-release or escrow signaling channel, this constitutes a denial-of-service against message delivery.

## Likelihood Explanation
No authentication is required. The gRPC endpoint is publicly exposed (port 5600). The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera Java SDK) and a valid topic ID. It is trivially scriptable. A single attacker with modest bandwidth can open dozens of connections.

## Recommendation

1. **Add an upper-bound constraint on `limit`** in `TopicMessageFilter`:
   ```java
   @Min(0)
   @Max(Long.MAX_VALUE / 2)   // or a domain-appropriate ceiling
   private long limit;
   ```
   Or reject values above a configured maximum in `ConsensusController.toFilter()`.

2. **Guard the narrowing cast** in `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```

3. **Add a global subscriber cap** in `TopicMessageServiceImpl` (an `AtomicLong subscriberCount` already exists — enforce a configurable ceiling and reject new subscriptions when exceeded). [11](#0-10) 

## Proof of Concept

```bash
# Requires grpcurl and a valid topic ID on the target network
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1234},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

This sends `limit = Long.MAX_VALUE` (valid `uint64`). The server accepts it, overflows to `-1` in the polling layer, omits `setMaxResults`, and issues unbounded queries on every 500 ms poll interval for the lifetime of the connection. Repeat across multiple connections to amplify the effect.
