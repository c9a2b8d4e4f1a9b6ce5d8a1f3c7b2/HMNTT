Audit Report

## Title
Unauthenticated Unbounded Request Flood on `getSchedules()` Enables Database Resource Exhaustion

## Summary
The `getSchedules()` handler in `rest/schedules.js` is accessible to any unauthenticated caller with no rate limiting at either the application or infrastructure (Traefik) layer. Each request with a non-empty result set triggers three database queries. The absence of rate limiting — confirmed both in the Node.js middleware stack and in the Helm chart — leaves the database connection pool exposed to exhaustion under sustained parallel load.

## Finding Description

**Route registration (confirmed):**
`getSchedules` is registered at `rest/server.js` line 115 with no per-route middleware: [1](#0-0) 

**Middleware stack (confirmed — no rate limiting):**
The global middleware chain registered before all routes is: `httpContext`, `requestLogger`, `authHandler`, optional `metricsHandler`, optional `responseCacheCheckHandler`. None impose per-IP or global request rate limits. [2](#0-1) 

**`authHandler` does not throttle unauthenticated requests (confirmed):**
It only sets a custom row limit in `httpContext` for authenticated users with a configured `limit` field; unauthenticated requests pass through unconditionally. [3](#0-2) 

**No rate-limiting module in middleware index (confirmed):**
`rest/middleware/index.js` exports `authHandler`, `handleError`, `openApiValidator`, `serveSwaggerDocs`, request/response handlers, and cache handlers — no rate limiter. [4](#0-3) 

**Three database queries per non-empty request (confirmed):**
`getSchedules()` issues one query for schedules (line 241), then two parallel queries for entities and signatures via `Promise.all` (lines 261–264): [5](#0-4) 

**`account.id` filter maps directly to `creator_account_id` (confirmed):** [6](#0-5) 

**Infrastructure-level rate limiting absent for REST, present for Rosetta (confirmed):**
The REST Helm chart defines only `circuitBreaker` and `retry`: [7](#0-6) 

The Rosetta chart explicitly adds `inFlightReq` (5 concurrent per IP) and `rateLimit` (10 req/s per host): [8](#0-7) 

**Response includes `transaction_body` and signatures (confirmed):** [9](#0-8) 

## Impact Explanation
At N concurrent requests/second, the database receives up to 3N queries/second. The `circuitBreaker` only trips at >25% network error ratio or >25% 5xx ratio — well after the connection pool is saturated. The `retry` middleware (10 attempts) actively amplifies load during degradation. This can render the mirror node unavailable for all users. The "account enumeration" framing in the original report is weaker: scheduled transaction data is public blockchain data by design, so its exposure is not a confidentiality violation.

## Likelihood Explanation
- Zero preconditions: no account, API key, or credential required.
- Trivially scriptable with any HTTP client.
- The three-query amplification means even moderate request rates translate to disproportionate DB load.
- No proactive throttle exists; the circuit breaker only reacts after errors accumulate.

## Recommendation
1. Add a Traefik `inFlightReq` and `rateLimit` middleware to `charts/hedera-mirror-rest/values.yaml`, mirroring the configuration already present in the Rosetta chart.
2. Optionally add application-layer rate limiting (e.g., `express-rate-limit`) in `rest/server.js` as a defense-in-depth measure, applied globally before route handlers.
3. Consider reducing the default `retry.attempts` from 10 to a lower value (e.g., 3, matching Rosetta) to avoid load amplification during degradation.

## Proof of Concept
```bash
# Flood getSchedules with parallel requests, each triggering 3 DB queries
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/schedules?account.id=0.0.$i" &
done
wait
```
Each request with results fires queries against `schedule`, `entity`, and `transaction_signature` tables in parallel. At 1000 concurrent requests, up to 3000 simultaneous DB queries are issued, exhausting the connection pool before the circuit breaker threshold is reached.
