Audit Report

## Title
Unauthenticated Unbounded Historical Subscription Causes Indefinite DB Polling DoS via `subscribeTopic()`

## Summary
Any unauthenticated caller can open a `subscribeTopic` gRPC stream with `startTime=0`, no limit, and no end time against a topic with millions of historical messages. The `PollingTopicMessageRetriever` enters throttled mode with `numRepeats=Long.MAX_VALUE`, and its `isComplete()` check never returns `true` because each page fills to `maxPageSize=1000`. This causes a `findByFilter()` DB query every 2 seconds indefinitely per subscription. Since `maxConcurrentCallsPerConnection` is per-connection only and there is no global subscription cap, an attacker multiplies this load across many connections.

## Finding Description

**Entry point — no authentication:**
`ConsensusController.subscribeTopic()` at line 43 accepts any `ConsensusTopicQuery` with no authentication or rate-limiting check. [1](#0-0) 

**Filter construction — `startTime=0` passes all validation:**
`toFilter()` at line 55 converts `consensusStartTime{seconds=0}` to `startTime=0`. `TopicMessageFilter` validates this with `@Min(0)` (passes) and `isValidStartTime()` which checks `startTime <= DomainUtils.now()` (0 ≤ now, passes). `limit=0` is also valid per `@Min(0)`. [2](#0-1) 

**Throttled retrieval — `numRepeats=Long.MAX_VALUE`:**
`TopicMessageServiceImpl.subscribeTopic()` at line 63 calls `topicMessageRetriever.retrieve(filter, true)` (throttled=`true`). In `PollingTopicMessageRetriever.PollingContext`, throttled mode sets `numRepeats = Long.MAX_VALUE`, `frequency = pollingFrequency` (default 2s), and `maxPageSize = 1000` (default). [3](#0-2) [4](#0-3) 

**`isComplete()` never returns `true`:**
The completion check is:
```java
boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
if (throttled) {
    return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
}
```
With `limit=0`, `hasLimit()` returns `false` (since `limit > 0` is the check), so `limitHit` is always `false`. With millions of messages, every poll returns exactly 1000 rows, so `pageSize.get() < 1000` is also `false`. `isComplete()` never returns `true`. [5](#0-4) [6](#0-5) 

**Indefinite polling loop:**
`RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())` repeats `poll()` indefinitely (up to `Long.MAX_VALUE` times) since the predicate never becomes false. Each `poll()` resets `pageSize` to 0, constructs a new filter, and calls `topicMessageRepository.findByFilter()`. [7](#0-6) 

**Why the 60s timeout does not protect:**
`.timeout(retrieverProperties.getTimeout(), scheduler)` (default 60s) is Reactor's idle timeout — it fires only if no item is emitted within 60 seconds. Since 1000 messages are emitted every 2 seconds, the timer resets continuously and never fires. [8](#0-7) 

**Why `maxConcurrentCallsPerConnection` does not protect:**
`maxConcurrentCallsPerConnection=5` is enforced per TCP connection only. There is no global subscription limit. An attacker opens N connections × 5 calls each. [9](#0-8) [10](#0-9) 

## Impact Explanation
Each open subscription with `startTime=0`, `limit=0`, and no `endTime` against a topic with ≥1000 historical messages issues one `findByFilter()` DB query every 2 seconds, each reading 1000 rows from `topic_message`. With N connections × 5 calls each, the attacker drives N×5 concurrent polling loops. The database connection pool is exhausted, query latency rises for all users, and legitimate subscribers are starved. The gRPC port (5600) is publicly exposed and requires no credentials. [11](#0-10) 

## Likelihood Explanation
The attack requires only a gRPC client (e.g., `grpcurl`), knowledge of any topic ID with historical messages (all public on mainnet/testnet), and the ability to open TCP connections to port 5600. No credentials, tokens, or privileged access are needed. The attack is trivially repeatable and scriptable. The `grpcurl` invocation pattern is even documented in the project's own README. [12](#0-11) 

## Recommendation
1. **Add a global subscription cap** in `TopicMessageServiceImpl` — reject new subscriptions when `subscriberCount` exceeds a configurable maximum.
2. **Fix `isComplete()` for unlimited throttled subscriptions** — after historical catch-up is complete (i.e., the retriever has reached messages with `consensusTimestamp` near `DomainUtils.now()`), transition to the live listener rather than continuing to poll indefinitely.
3. **Enforce a per-IP or per-connection subscription rate limit** at the gRPC server layer.
4. **Require an explicit `endTime` or `limit` for historical-only queries** (where `startTime` is significantly in the past), or cap the maximum duration of a throttled retrieval session.
5. **Reduce the default `timeout`** for the throttled retriever to a wall-clock duration rather than an idle timeout, so long-running historical retrievals are bounded.

## Proof of Concept
```bash
# Open 5 indefinitely-polling subscriptions on a single connection
grpcurl -plaintext \
  -d '{"topicID": {"topicNum": 41110}, "consensusStartTime": {"seconds": 0}, "limit": 0}' \
  mainnet-public.mirrornode.hedera.com:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# Repeat across N connections to multiply DB load
for i in $(seq 1 100); do
  grpcurl -plaintext \
    -d '{"topicID": {"topicNum": 41110}, "consensusStartTime": {"seconds": 0}, "limit": 0}' \
    mainnet-public.mirrornode.hedera.com:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
```
Each background process holds an open stream. The `PollingTopicMessageRetriever` for each stream issues `SELECT ... FROM topic_message WHERE ... LIMIT 1000` every 2 seconds indefinitely, as confirmed by the `isComplete()` logic above. [13](#0-12)

    @Override
    public Flux<TopicMessage> retrieve(TopicMessageFilter filter, boolean throttled) {
        if (!retrieverProperties.isEnabled()) {
            return Flux.empty();
        }

        PollingContext context = new PollingContext(filter, throttled);
        return Flux.defer(() -> poll(context))
                .repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
                        .jitter(0.1)
                        .withFixedDelay(context.getFrequency())
                        .withScheduler(scheduler))
                .name(METRIC)
                .tap(Micrometer.observation(observationRegistry))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
                .timeout(retrieverProperties.getTimeout(), scheduler)
                .doOnCancel(context::onComplete)
                .doOnComplete(context::onComplete)
                .doOnNext(context::onNext);
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
