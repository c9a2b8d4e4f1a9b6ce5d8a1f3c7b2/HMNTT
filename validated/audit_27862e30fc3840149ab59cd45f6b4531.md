Audit Report

## Title
Permanent Caching of Empty Optional in `findEarliest()` Causes Persistent Denial of "earliest" Block Lookups

## Summary
The `CACHE_MANAGER_RECORD_FILE_EARLIEST` Caffeine cache is configured with no TTL and `maximumSize=1`. The `@Cacheable` guard `unless = "#result == null"` does not exclude `Optional.empty()` — which is a non-null Java object — so if `findEarliest()` is called before the genesis record file is ingested, the empty result is permanently cached. Every subsequent call returns the stale empty `Optional` for the lifetime of the process.

## Finding Description

**Cache configuration — no TTL:**

`EvmConfiguration.java` lines 190–197 builds the earliest-record-file cache with only `maximumSize(1)` and no `expireAfterWrite` / `expireAfterAccess`:

```java
@Bean(CACHE_MANAGER_RECORD_FILE_EARLIEST)
CacheManager cacheManagerRecordFileEarliest() {
    final var caffeine = Caffeine.newBuilder().maximumSize(1).recordStats();
    ...
}
``` [1](#0-0) 

Compare this to the `CACHE_MANAGER_RECORD_FILE_LATEST` cache immediately above it, which correctly sets `.expireAfterWrite(500, TimeUnit.MILLISECONDS)`: [2](#0-1) 

**Flawed cache guard:**

`RecordFileRepository.java` line 23 annotates `findEarliest()` with:

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_EARLIEST, unless = "#result == null")
Optional<RecordFile> findEarliest();
``` [3](#0-2) 

**Root cause:** Spring evaluates the `unless` SpEL expression against the raw return value — the `Optional` object itself. `Optional.empty()` is a non-null singleton (`Optional.empty() != null` is always `true`), so `#result == null` evaluates to `false`. Spring therefore proceeds to cache the result. Spring's cache abstraction then stores a `NullValue` sentinel (unwrapped from the empty Optional) in Caffeine. On every subsequent call the cache hits, Spring re-wraps the stored null back to `Optional.empty()`, and the database is never queried again. With no TTL configured, this entry is never evicted.

**Call path:**

`RecordFileServiceImpl.findByBlockType(BlockType.EARLIEST)` → `recordFileRepository.findEarliest()`: [4](#0-3) 

## Impact Explanation
Any API call that resolves `block: "earliest"` (e.g., `eth_call`, `eth_estimateGas`, `eth_getBalance` at the earliest block) will permanently receive a "block not found" / empty response for the entire lifetime of the process after the cache is poisoned. Recovery requires a full application restart. The scope is limited to the web3 JSON-RPC service; no direct fund theft is possible, but it constitutes a persistent, targeted denial-of-service against all historical query functionality that uses `"earliest"` as a block parameter.

## Likelihood Explanation
The attack requires no credentials — only HTTP access to the web3 API endpoint. The vulnerable window exists on every fresh deployment and every restart, for as long as the importer has not yet ingested the genesis record file. In cloud/Kubernetes environments where the web3 pod starts independently of the importer, this window can span seconds to minutes. A single well-timed HTTP request is sufficient to permanently poison the cache. The attack is repeatable on every restart.

## Recommendation
Fix the `unless` guard to explicitly check for an empty Optional:

```java
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_EARLIEST,
    unless = "#result == null || !#result.isPresent()"
)
Optional<RecordFile> findEarliest();
```

Additionally, add a TTL to `cacheManagerRecordFileEarliest()` (e.g., matching the 500 ms used for the latest-record-file cache) so that even if an empty result is cached, it expires quickly:

```java
final var caffeine = Caffeine.newBuilder()
    .expireAfterWrite(500, TimeUnit.MILLISECONDS)
    .maximumSize(1)
    .recordStats();
```

The same `unless` flaw exists on `findByIndex` and `findByTimestamp` and should be reviewed for the same fix. [5](#0-4) 

## Proof of Concept
1. Start the web3 service before the importer has ingested any record files (empty `record_file` table).
2. Send any request that resolves `BlockType.EARLIEST`, e.g.:
   ```
   POST /api/v1/contracts/call
   {"block": "earliest", "data": "0x...", "to": "0x..."}
   ```
3. `RecordFileServiceImpl.findByBlockType(EARLIEST)` calls `recordFileRepository.findEarliest()`.
4. The DB returns no rows → Spring Data JPA returns `Optional.empty()`.
5. `unless = "#result == null"` → `false` (empty Optional is non-null) → Spring caches the empty result.
6. The importer subsequently ingests the genesis record file.
7. All future calls to `findEarliest()` hit the no-TTL cache and return `Optional.empty()` — the DB is never queried again, and all `"earliest"` block queries are permanently broken until restart.
