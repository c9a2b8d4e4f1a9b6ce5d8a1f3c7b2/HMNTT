# Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Causes Unbounded Database Query

## Summary
An unchecked narrowing cast from `long` to `int` in `PollingTopicListener.poll()` allows a gRPC client to supply a `limit` value of `2147483648L` (one above `Integer.MAX_VALUE`), causing the computed `pageSize` to overflow to `Integer.MIN_VALUE`. The resulting negative value bypasses the `hasLimit()` guard in `TopicMessageRepositoryCustomImpl.findByFilter()`, causing `setMaxResults` to never be called and every poll cycle to execute an unbounded full-table query that loads all matching rows into JVM heap memory.

## Finding Description

**Entry point** — `ConsensusController.toFilter()` reads `query.getLimit()` (a protobuf `uint64`, stored as Java `long`) and passes it directly to `TopicMessageFilter.builder().limit(...)` with no upper-bound check. [1](#0-0) 

**Overflow site** — In `PollingTopicListener.poll()`, the expression `(int)(filter.getLimit() - context.getCount().get())` performs an unchecked narrowing cast. With `filter.getLimit() = 2147483648L` and `count = 0`, `(int) 2147483648L` evaluates to `-2147483648` (Java narrowing-cast overflow). `Math.min(-2147483648, maxPageSize)` then returns `-2147483648` because any positive `maxPageSize` is larger. [2](#0-1) 

**Validation bypass** — The negative `pageSize` (`int`) is widened to `long -2147483648L` and stored in a new `TopicMessageFilter` via the internal builder. This builder call is not a Spring-managed method invocation, so the `@Min(0)` Bean Validation constraint on `TopicMessageFilter.limit` is never evaluated. [3](#0-2) [4](#0-3) 

**`hasLimit()` semantics** — `TopicMessageFilter.hasLimit()` returns `limit > 0`. With `limit = -2147483648L`, this returns `false`. [5](#0-4) 

**Unbounded query** — In `TopicMessageRepositoryCustomImpl.findByFilter()`, `setMaxResults` is only called when `filter.hasLimit()` is `true`. Since it returns `false`, no row limit is applied to the JPA query. The call to `getResultList()` then loads every matching row into heap memory before returning a stream. [6](#0-5) [7](#0-6) 

**Why `flux.take()` does not mitigate** — `TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` only when `filter.hasLimit()` is true. The outer filter's `limit` is `2147483648L` (positive), so `hasLimit()` is `true` there and `take` is applied. However, `take` only limits downstream *emission*; it cannot prevent `getResultList()` from having already loaded all rows into memory before the Flux layer can cancel. [8](#0-7) 

## Impact Explanation
On every poll cycle (default interval: 500 ms per `ListenerProperties`), the database executes a full sequential scan for the topic with no `LIMIT` clause. On a topic with millions of messages, this causes heap exhaustion / OOM in the mirror node JVM (all rows loaded via `getResultList()`), excessive database load, and denial of service for all other subscribers sharing the same JVM and database connection pool. A single malicious subscription is sufficient to trigger this continuously until the connection is dropped. [9](#0-8) 

## Likelihood Explanation
The precondition is trivial: any unauthenticated gRPC client can send a `ConsensusTopicQuery` with `limit = 2147483648`. No credentials or special knowledge beyond the public proto schema is required. **However**, this vulnerability only affects deployments where `hiero.mirror.grpc.listener.type` is set to `POLL` or `SHARED_POLL`. The default listener type is `REDIS`, which uses a different code path and is not affected. [10](#0-9) 

## Recommendation
1. **Clamp the cast** — Replace the unchecked narrowing cast with a safe clamp in `PollingTopicListener.poll()`:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Validate at the service boundary** — Add an explicit upper-bound check (e.g., `@Max(Long.MAX_VALUE)` or a configurable cap) on `TopicMessageFilter.limit` in `ConsensusController.toFilter()` before the filter is passed downstream.
3. **Guard `setMaxResults`** — In `TopicMessageRepositoryCustomImpl.findByFilter()`, add a defensive check: only call `setMaxResults` when `filter.getLimit() > 0` (which `hasLimit()` already does), but also ensure the internal builder path cannot produce a negative limit by validating before building.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
