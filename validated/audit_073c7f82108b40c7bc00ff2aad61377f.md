# Audit Report

## Title
Unthrottled Unlimited DB Polling via Unauthenticated gRPC Topic Subscription Safety Check

## Summary
Any unauthenticated client subscribing to a valid topic via the gRPC API triggers a one-shot safety check that invokes `topicMessageRetriever.retrieve(gapFilter, false)` in unthrottled mode with no query limit. Due to a logic flaw in `PollingContext.isComplete()`, the retriever unconditionally executes all 12 configured polls (up to 5,000 rows each) per subscription per reconnect cycle. With no enforced cap on concurrent connections or subscribers, an attacker can sustain arbitrarily high database load.

## Finding Description

**Root cause — `PollingContext.isComplete()` never terminates the unthrottled loop when no limit is set:**

In `PollingTopicMessageRetriever.java`, the unthrottled branch of `isComplete()` is:

```java
// line 122-128
boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
if (throttled) {
    return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
}
return limitHit;
```

The throttled branch terminates early when a page returns fewer rows than `maxPageSize` (i.e., the DB has no more data). The **unthrottled branch does not** — it only returns `true` when `limitHit`. [4](#0-3) 

**Failed assumption:** The design assumes the safety check fires only when the importer is down and there are genuinely missing messages. In practice, `topicContext.isComplete()` returns `false` for every subscription without an `endTime` (the common case for live subscribers), so the safety check fires for every such subscription after 1 second. [1](#0-0) 

**Exact code path:**

1. `subscribeTopic()` schedules the safety check for every subscription: [2](#0-1) 

2. `missingMessages(topicContext, null)` builds `gapFilter` with no limit and calls the unthrottled retriever: [3](#0-2) 

3. The unthrottled `PollingContext` is configured with `maxPageSize=5000`, `maxPolls=12`, `pollingFrequency=20ms`: [5](#0-4) 

4. Because `gapFilter` carries no limit, `filter.hasLimit()` is `false`, `limitHit` is always `false`, and `isComplete()` never returns `true`. All 12 polls execute unconditionally, each fetching up to 5,000 rows — **60,000 rows and 12 DB round-trips per safety-check invocation**. [8](#0-7) 

**No enforced subscriber limit:** `subscriberCount` is a Micrometer gauge only and is never checked against a maximum. [6](#0-5)  `maxConcurrentCallsPerConnection = 5` limits calls per TCP connection but places no cap on the number of TCP connections. [7](#0-6) 

## Impact Explanation
Each attacker-controlled subscription triggers 12 sequential unthrottled database queries of up to 5,000 rows each within approximately 240 ms. With N open connections × 5 concurrent subscriptions per connection, the attacker generates 60N unthrottled DB queries per reconnect cycle. The database connection pool becomes saturated, causing legitimate subscriber queries and importer writes to queue or time out. This constitutes a denial-of-service against the mirror node's gRPC topic subscription service and any other component sharing the same database pool, with no ability for the server to distinguish attacker traffic from legitimate traffic.

## Likelihood Explanation
The gRPC port (default 5600) is publicly reachable with no authentication. A single attacker machine can open hundreds of TCP connections. The attack requires only a standard gRPC client library and a known topic ID (topic IDs are public on-chain). The reconnect loop (subscribe → wait 1 s → safety check fires → disconnect → repeat) is trivially scriptable. The attack is fully repeatable, requires no special privileges, and produces a measurable, sustained DB load increase proportional to the number of connections maintained.

## Recommendation

1. **Fix `PollingContext.isComplete()` for unthrottled mode** to also terminate early when the last page returned fewer rows than `maxPageSize`, mirroring the throttled branch:
   ```java
   return pageSize.get() < maxPageSize || limitHit;
   ```
   This ensures the retriever stops polling as soon as the DB has no more data, regardless of whether a limit was set. [4](#0-3) 

2. **Enforce a maximum subscriber count** by checking `subscriberCount` against a configurable threshold in `subscribeTopic()` and rejecting new subscriptions with `RESOURCE_EXHAUSTED` when the threshold is exceeded. [6](#0-5) 

3. **Enforce a maximum connection count** at the Netty level in addition to `maxConcurrentCallsPerConnection`. [7](#0-6) 

4. **Rate-limit the safety check** so it does not re-fire on rapid reconnects from the same client.

## Proof of Concept

```python
import grpc
import threading
from mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from google.protobuf.timestamp_pb2 import Timestamp

def attack(target, topic_shard, topic_realm, topic_num):
    while True:
        try:
            channel = grpc.insecure_channel(target)
            stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
            req = consensus_service_pb2.ConsensusTopicQuery(
                topic_id=...,  # known valid topic
                # no end_time → isComplete() always False → safety check always fires
            )
            for _ in stub.subscribeTopic(req, timeout=2):
                pass
        except Exception:
            pass  # reconnect immediately

threads = [threading.Thread(target=attack, args=("mirror-node:5600", 0, 0, 1))
           for _ in range(200)]
for t in threads:
    t.start()
```

Each of the 200 threads opens a subscription, waits 1 second for the safety check to fire (triggering 12 × 5,000-row unthrottled DB queries), then reconnects. This generates up to **12,000 unthrottled DB queries per second** sustained, saturating the connection pool and denying service to legitimate users.
