Audit Report

## Title
`BinaryGasEstimator.safeCall()` Silently Swallows `QueryTimeoutException`, Allowing Gas Estimation to Exceed `requestTimeout`

## Summary
The `safeCall()` method in `BinaryGasEstimator` uses an overly broad `catch (Exception ignored)` that absorbs the `QueryTimeoutException` thrown by the Hibernate `statementInspector` when the per-request wall-clock timeout is reached. As a result, the binary search loop continues executing EVM calls and DB queries for all remaining iterations after the timeout fires, allowing a single `eth_estimateGas` request to consume server resources well beyond the configured bound.

## Finding Description

**Verified code path:**

1. `ContractExecutionService.processCall()` calls `ContractCallContext.run(ctx -> ...)`, which creates a new `ContractCallContext`. Its `startTime` is set once as a `final` field and is never reset — `reset()` only clears `writeCache`. [1](#0-0) [2](#0-1) 

2. Because `params.isEstimate()` is true, `estimateGas(params, ctx)` is called. It first calls `callContract(params, context)` — the initial EVM execution — which is **not** wrapped in `safeCall`, so a timeout here propagates normally. [3](#0-2) 

3. If `gasUsedByInitialCall < params.getGas()`, `binaryGasEstimator.search()` is invoked with up to `maxGasEstimateRetriesCount = 20` iterations. [4](#0-3) [5](#0-4) 

4. Each iteration calls `safeCall(mid, call)`: [6](#0-5) 

5. The Hibernate `statementInspector` fires before every SQL statement, computing elapsed time from the original `startTime`. Once `elapsed >= timeout` (default 10 s), it throws `QueryTimeoutException`: [7](#0-6) 

6. `QueryTimeoutException` is a `RuntimeException` (Spring's `org.springframework.dao.QueryTimeoutException`). `safeCall`'s `catch (Exception ignored)` catches it, logs an info message, returns `null`, and the loop continues for all remaining iterations.

**Root cause:** `safeCall` was introduced to handle modularized-service failures gracefully (contract execution failures that throw instead of returning a failed result), but its `catch (Exception ignored)` is too broad and neutralises the timeout enforcement mechanism entirely.

**Failed assumption:** The design assumes that once `QueryTimeoutException` is thrown, the request terminates. Instead, the exception is absorbed and the binary search continues.

## Impact Explanation

Each post-timeout iteration still executes EVM bytecode (CPU-bound) before hitting the first SQL statement that would trigger the inspector. For contracts with expensive precompile operations or computation loops before their first storage read, this pre-DB phase can take hundreds of milliseconds per iteration. With up to 20 iterations and a 10-second timeout, actual wall-clock time per request can reach `10 s + (20 × per-iteration EVM time)`. At 500 ms per iteration, that is 20 s — a 100% overshoot. Concurrent exploitation ties up HTTP worker threads and DB connection-pool slots for twice the intended duration, directly increasing node resource consumption beyond the configured bound.

## Likelihood Explanation

No authentication or special privilege is required — `eth_estimateGas` is a standard, publicly accessible JSON-RPC endpoint. An attacker only needs to reference a contract that:
- Succeeds with the full gas limit (so the binary search is entered)
- Performs expensive computation before its first storage read in each call

Both conditions are trivially achievable with a contract containing a tight arithmetic or hashing loop followed by a single `SLOAD`. The attack is repeatable and can be parallelised across many concurrent connections.

## Recommendation

In `safeCall`, narrow the caught exception type to exclude `QueryTimeoutException` (and other infrastructure-level exceptions that signal intentional termination). For example:

```java
private EvmTransactionResult safeCall(long mid, LongFunction<EvmTransactionResult> call) {
    try {
        return call.apply(mid);
    } catch (QueryTimeoutException e) {
        throw e; // re-throw timeout signals — do not swallow
    } catch (Exception ignored) {
        log.info("Exception while calling contract for gas estimation");
        return null;
    }
}
```

Alternatively, check `ContractCallContext.get().getStartTime()` at the top of the `search()` loop and break early if the timeout has already been exceeded before attempting the next iteration. [8](#0-7) 

## Proof of Concept

1. Deploy a contract with a tight computation loop (e.g., 10 000 iterations of `keccak256`) followed by a single `SLOAD`.
2. Submit `eth_estimateGas` with `gas = 15000000` targeting that contract.
3. The initial `callContract` call succeeds; the binary search is entered.
4. After ~10 s, the `statementInspector` fires and throws `QueryTimeoutException` inside an iteration.
5. `safeCall` catches and discards it; the loop continues for the remaining iterations.
6. Observe via server-side timing that the request completes significantly after the 10-second `requestTimeout`, with all 20 iterations attempted. [6](#0-5) [7](#0-6)
