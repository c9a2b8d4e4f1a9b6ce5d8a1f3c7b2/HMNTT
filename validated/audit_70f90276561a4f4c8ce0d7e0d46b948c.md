# Audit Report

## Title
Missing Null-Byte Sanitization on `ContractResult.errorMessage` Causes COPY-Based Batch Insert Failure (DoS)

## Summary
`ContractResultServiceImpl.processContractResult()` assigns `functionResult.getErrorMessage()` directly to `contractResult.errorMessage` without calling `DomainUtils.sanitize()`. If the protobuf string contains a null byte (0x00), the downstream `BatchInserter` will fail when streaming the row via PostgreSQL's `COPY … FROM STDIN WITH CSV`, aborting the entire batch and throwing a `ParserException`.

## Finding Description

**Exact location — unsanitized assignment:**

`ContractResultServiceImpl.java` line 251:
```java
contractResult.setErrorMessage(functionResult.getErrorMessage());
``` [1](#0-0) 

No call to `DomainUtils.sanitize()` wraps this assignment.

**`DomainUtils.sanitize()` exists and is documented for exactly this purpose:**

`DomainUtils.java` lines 239–241:
```java
public static String sanitize(String input) {
    return StringUtils.isNotEmpty(input) ? input.replace(NULL_CHARACTER, NULL_REPLACEMENT) : input;
}
``` [2](#0-1) 

The Javadoc explicitly states: *"PostgreSQL does not allow the null character (0x0000) to be inserted."*

**`sanitize()` is applied to other user-controlled string fields but not `errorMessage`:**

It is called in `AbstractEntity` (entity memo) and `AbstractToken` (name/symbol/memo), but `ContractResult.errorMessage` is a plain `String` field with no custom setter — unlike `setBloom`, which does have custom logic. [3](#0-2) [4](#0-3) 

**`BatchInserter` uses `COPY … FROM STDIN WITH CSV` with no string sanitization:**

`BatchInserter.java` line 85:
```java
sql = String.format("COPY %s(%s) FROM STDIN WITH CSV", this.tableName, columnsCsv);
``` [5](#0-4) 

The `CsvMapper` only registers serializers for `byte[][]`, `byte[]`, `EntityId`, `List`, and `Range` — no string-sanitizing serializer is present. [6](#0-5) 

**Exception propagation aborts the entire batch:**

`BatchInserter.java` lines 110–111:
```java
} catch (Exception e) {
    throw new ParserException(String.format("Error copying %d items to table %s", items.size(), tableName), e);
``` [7](#0-6) 

Because all items are streamed in a single `COPY` command, one bad row aborts every row in the batch.

## Impact Explanation
A single contract transaction whose `ContractFunctionResult.errorMessage` contains a null byte causes the entire `contract_result` batch write to fail with a `ParserException`. All contract results in that batch are lost. Downstream REST API consumers lose visibility into those contract results. The importer must recover or skip the offending record file before writes resume. The attack is repeatable: the same transaction can be resubmitted each time the importer recovers.

## Likelihood Explanation
Any account on the network can submit a `ContractCall` or `ContractCreate`. When the EVM reverts, the consensus node populates `errorMessage` from the revert reason. ABI-encoded revert reasons contain 0x00 padding bytes. The Java protobuf runtime does not validate UTF-8 on deserialization of `string` fields, so a consensus node that copies raw revert-reason bytes verbatim into this field produces a Java `String` containing null characters. The mirror node ingests this without any sanitization check.

## Recommendation
Wrap the assignment in `ContractResultServiceImpl.processContractResult()` with `DomainUtils.sanitize()`, consistent with how other user-controlled string fields are handled:

```java
contractResult.setErrorMessage(DomainUtils.sanitize(functionResult.getErrorMessage()));
```

Additionally, consider adding a custom `setErrorMessage(String errorMessage)` setter in `ContractResult` that calls `DomainUtils.sanitize()` defensively, so any future call sites are automatically protected.

## Proof of Concept
1. Deploy a contract that reverts with a reason string containing a null byte (e.g., via `revert("\x00bad")` in Solidity, or by crafting a raw ABI-encoded revert payload with 0x00 padding).
2. Submit a `ContractCall` transaction targeting that contract.
3. The consensus node records `ContractFunctionResult.errorMessage` containing 0x00.
4. The mirror node importer calls `ContractResultServiceImpl.processContractResult()`, which sets `contractResult.errorMessage` to the unsanitized string.
5. `BatchInserter.persistItems()` serializes the collection to CSV and streams it via `COPY contract_result(...) FROM STDIN WITH CSV`.
6. PostgreSQL raises `ERROR: invalid byte sequence for encoding "UTF8": 0x00`.
7. `BatchInserter.persist()` catches the exception and throws `ParserException`, aborting the entire batch.
8. All contract results from that batch are not persisted; the importer stalls until the record file is skipped or the importer is restarted.
