# Audit Report

## Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` Bypasses Pagination, Enabling Unbounded DB Query via `limit=Long.MAX_VALUE`

## Summary
An unprivileged gRPC client can supply `limit=Long.MAX_VALUE` in a `subscribeTopic` request. This value passes Bean Validation (only `@Min(0)` is present, no `@Max`). Inside `PollingTopicMessageRetriever.poll()`, the narrowing cast `(int)(Long.MAX_VALUE - 0L)` overflows to `-1`. The resulting page-level filter has `limit=-1`, causing `hasLimit()` to return `false`. `TopicMessageRepositoryCustomImpl.findByFilter()` then calls `getResultList()` with no `setMaxResults()` bound, loading every historical row for the topic into JVM heap in a single query.

## Finding Description

**No upper bound on `limit` in `TopicMessageFilter`:**

`TopicMessageFilter.java` carries only `@Min(0)` on `limit` with no `@Max`. [1](#0-0) 

`hasLimit()` returns `limit > 0`, so `Long.MAX_VALUE` evaluates to `true` and passes validation. [2](#0-1) 

**Integer overflow in `PollingTopicMessageRetriever.poll()`:**

```
(int)(Long.MAX_VALUE - 0L)
= (int) 0x7FFFFFFFFFFFFFFF
= 0xFFFFFFFF
= -1
```

`Math.min(-1, maxPageSize)` = `-1`. [3](#0-2) 

**Page-level filter loses its limit:**

`pageSize = -1` is stored into `newFilter.limit`. Because `hasLimit()` checks `limit > 0`, `-1 > 0 = false`. [4](#0-3) 

**`findByFilter` omits `setMaxResults` when `hasLimit()` is false:**

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
```

When `hasLimit()` is `false`, `setMaxResults()` is never called. The query then calls `getResultList()` — which eagerly loads **all matching rows into JVM heap** — with no SQL `LIMIT` clause. [5](#0-4) 

Note: the comment on line 60 explicitly explains why `getResultList()` (eager, full load) is used instead of `getResultStream()`. This means the entire result set is materialized in memory before being returned as a `Stream`, making the memory impact worse than the report's cursor-hold framing.

**The same overflow exists in `PollingTopicListener.poll()`:** [6](#0-5) 

**No subscriber cap:**

`subscriberCount` is a Micrometer gauge only; it is never checked against a maximum before accepting a new subscription. [7](#0-6) 

**Subscriptions without `endTime` never terminate:**

`pastEndTime()` returns `Flux.never()` when `endTime` is null, keeping each subscription alive indefinitely. [8](#0-7) 

The top-level `flux.take(filter.getLimit())` at line 84 does not mitigate this: `take(Long.MAX_VALUE)` is effectively a no-op. [9](#0-8) 

## Impact Explanation
Each malicious subscription with `limit=Long.MAX_VALUE` and `startTime=0` triggers a full table scan of `topic_message` for the targeted topic, loading all rows into JVM heap. With a large topic (millions of messages), a single request can cause an `OutOfMemoryError` or severe GC pressure. Multiple concurrent attackers can exhaust the database connection pool and JVM heap simultaneously, causing a complete service outage for all legitimate subscribers.

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible to any unprivileged client. The `ConsensusTopicQuery` protobuf `limit` field is a `uint64`, so `Long.MAX_VALUE` (or any large value) is trivially expressible. No authentication or rate-limiting is visible in the code path. The exploit requires only a single well-formed gRPC call and is fully reproducible.

## Recommendation

1. **Add `@Max` to `TopicMessageFilter.limit`** — enforce a reasonable upper bound (e.g., matching the configured `maxPageSize`) at the validation layer. [1](#0-0) 

2. **Fix the narrowing cast in `poll()`** — compute the remaining limit as a `long` and clamp it to `Integer.MAX_VALUE` before casting, or keep it as a `long` throughout. [3](#0-2) 

3. **Apply the same fix to `PollingTopicListener.poll()`** — the identical overflow exists there. [10](#0-9) 

4. **Enforce a maximum concurrent subscriber count** — check `subscriberCount` against a configurable cap before accepting new subscriptions.

## Proof of Concept

```java
// gRPC client (any unauthenticated caller)
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).setNanos(0).build())
    .setLimit(Long.MAX_VALUE)   // passes @Min(0); no @Max exists
    .build();

stub.subscribeTopic(query, responseObserver);
// Server-side:
// filter.getLimit()          = Long.MAX_VALUE  → hasLimit() = true
// (int)(Long.MAX_VALUE - 0L) = -1              ← integer overflow
// Math.min(-1, maxPageSize)  = -1
// newFilter.limit            = -1  → hasLimit() = false
// setMaxResults() never called → getResultList() returns ALL rows
// All rows loaded into JVM heap; DB connection held for full duration
```
