# Audit Report

## Title
Unauthenticated `content-length` Header Spoofing Poisons `requestSizeHistogram` Metrics

## Summary
`metricsMiddleware()` in `rest/middleware/metricsHandler.js` unconditionally trusts the attacker-controlled `content-length` HTTP header to record request size into the `api_request_size_bytes` Prometheus histogram. No validation against actual body bytes is performed. A single unauthenticated GET request with a fabricated `content-length: 2147483647` header permanently inflates the histogram's `_sum` for the targeted route until process restart.

## Finding Description
**Exact location:** `rest/middleware/metricsHandler.js`, `metricsMiddleware()`, line 209:

```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```

`req.headers['content-length']` is the raw, attacker-supplied string. `parseInt(..., 10)` faithfully parses any integer value — including `2147483647` — and passes it directly to `requestSizeHistogram.record()` with no upper-bound check and no comparison against actual bytes received.

**Why body-parser protection does not apply:**
In `rest/server.js`, `express.urlencoded()` and `express.json()` are registered at lines 68–73, before `metricsHandler()` at lines 89–92. However, both parsers only enforce their body-size limit when they attempt to read and parse a body. For GET requests (the standard method for all transaction/account/token history endpoints), both parsers skip body processing entirely because GET requests carry no body. The spoofed `content-length` header is never cross-checked against actual bytes received, and the request flows through to the metrics middleware untouched.

**Exploit flow:**
1. Attacker sends: `GET /api/v1/transactions HTTP/1.1` with header `content-length: 2147483647` and no body.
2. `express.json()` / `express.urlencoded()` skip (GET, no body to parse).
3. `metricsMiddleware` runs; `next()` is called; the route handler responds normally.
4. On `res.on('finish', ...)`, line 209 executes: `parseInt('2147483647', 10)` → `2147483647` is recorded into `requestSizeHistogram` with labels `{method: 'GET', path: '/api/v1/transactions', code: '200'}`.
5. A single such request permanently inflates `_sum`; repeat at will with no authentication or rate-limiting enforced by the metrics layer.

## Impact Explanation
`requestSizeHistogram` (`api_request_size_bytes`) is a Prometheus histogram. Injecting values of ~2 GB per request:
- Permanently inflates the `_sum` bucket, making the mean request size appear enormous for the targeted route.
- Pushes all observations into the highest histogram bucket (`+Inf`), destroying the percentile distribution.
- Triggers false alerts on any SLO/SLA rules monitoring request payload size anomalies.
- Misleads operators into believing transaction history queries carry massive payloads, potentially causing incorrect capacity planning, misdiagnosis of performance issues, or suppression of real anomaly alerts (alert fatigue / masking).
- Poisoned `_sum` values persist in-memory until process restart; no self-healing occurs.

**Severity: Medium** — no direct data exfiltration or fund movement, but persistent, unauthenticated metric poisoning with concrete operational and observability impact.

## Likelihood Explanation
- **No authentication required** — any internet-accessible instance is vulnerable.
- **Zero body needed** — the attack is a single-header GET request; no special tooling beyond `curl`.
- **Fully repeatable** — the attacker can flood the histogram continuously.
- **No rate-limiting** in the metrics middleware itself.
- A single request is sufficient to cause lasting damage to the histogram's statistical validity.

## Recommendation
Replace the blind trust of the `content-length` header with the actual measured request body size. The response size is already measured correctly via `res.write`/`res.end` interception (lines 176–183). Apply the same pattern to the request side:

```js
// In metricsMiddleware, before next():
let requestSize = 0;
req.on('data', (chunk) => { requestSize += getChunkSize(chunk); });
req.on('end', () => { /* requestSize is now accurate */ });
```

Then record `requestSize` (the actual byte count) instead of `parseInt(req.headers['content-length'] ?? '0', 10)`. If retaining the header-based approach for performance reasons, add an upper-bound clamp (e.g., `Math.min(parsed, MAX_EXPECTED_REQUEST_BYTES)`) and validate that the value is a non-negative finite integer.

## Proof of Concept
```bash
curl -s -o /dev/null -w "%{http_code}" \
  -H "content-length: 2147483647" \
  "https://<mirror-node-host>/api/v1/transactions"
# Returns 200; requestSizeHistogram _sum for GET /api/v1/transactions
# is now inflated by 2,147,483,647 bytes.
# Verify via the metrics endpoint:
curl "https://<mirror-node-host>/swagger/metrics/" | grep api_request_size_bytes
```
