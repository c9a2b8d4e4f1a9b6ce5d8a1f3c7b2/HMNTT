Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses `maxPageSize` Guard, Causing Unlimited Database Queries

## Summary
When `hiero.mirror.grpc.listener.type = POLL`, a gRPC client supplying a `limit` value in the range `(Integer.MAX_VALUE, Long.MAX_VALUE]` triggers a silent integer overflow in `PollingTopicListener.poll()`. The resulting negative `pageSize` is stored in `newFilter`, causing `TopicMessageFilter.hasLimit()` to return `false`, which prevents `setMaxResults()` from ever being called in `TopicMessageRepositoryCustomImpl.findByFilter()`. Every poll cycle (default: 500 ms) then issues a fully unbounded SQL query against the database.

## Finding Description

**Step 1 — No upper-bound validation on `limit`.**

`ConsensusController.toFilter()` maps the raw proto `uint64 limit` field directly to `TopicMessageFilter.limit` (a Java `long`) with no upper-bound check: [1](#0-0) 

`TopicMessageFilter.limit` is annotated only with `@Min(0)`, permitting any non-negative `long` including values > `Integer.MAX_VALUE`: [2](#0-1) 

**Step 2 — Remaining limit passed as `long` to the listener.**

`TopicMessageServiceImpl.incomingMessages()` computes the remaining limit correctly as a `long` (no overflow here) and builds a new filter with `limit = 3_000_000_000L`: [3](#0-2) 

**Step 3 — Integer overflow in `PollingTopicListener.poll()`.**

The remaining `long` is **narrowed to `int`** before `Math.min`: [4](#0-3) 

With `filter.getLimit() = 3_000_000_000L` and `count = 0`:
- `(int)(3_000_000_000L - 0L)` → `-1_294_967_296` (int overflow)
- `Math.min(-1_294_967_296, 5000)` → `-1_294_967_296`
- `newFilter.limit = -1_294_967_296L` (widened back to `long`)

**Step 4 — `hasLimit()` returns `false` for the negative value.**

`TopicMessageFilter.hasLimit()` checks `limit > 0`, which is `false` for `-1_294_967_296L`: [5](#0-4) 

**Step 5 — `setMaxResults()` is never called; unbounded query is issued.**

In `TopicMessageRepositoryCustomImpl.findByFilter()`, the `if (filter.hasLimit())` guard is not entered, so `typedQuery.setMaxResults()` is never called: [6](#0-5) 

The JPA query has no `LIMIT` clause and returns every matching row for the topic.

**Step 6 — Outer `flux.take()` does not prevent repeated DB queries.**

The `flux.take(filter.getLimit())` in `subscribeTopic()` limits records delivered to the client but does not cancel the repeated poll cycles that each issue an unbounded query: [7](#0-6) 

**Note:** The identical overflow pattern also exists in `PollingTopicMessageRetriever.poll()`: [8](#0-7) 

## Impact Explanation

Every poll cycle (default interval: 500 ms, configurable via `hiero.mirror.grpc.listener.interval`) issues a full-table scan for the topic with no row limit. The `maxPageSize = 5000` guard is completely bypassed. A topic with millions of messages causes the server to load all of them into memory on each tick, exhausting heap and database connection resources. A single persistent gRPC subscription can degrade or crash the mirror node for all other subscribers (server-side Denial-of-Service). The database-level `statementTimeout` (default 10 s) provides partial mitigation per query, but the repeated 500 ms polling means the database is continuously hammered with expensive queries. [9](#0-8) 

## Likelihood Explanation

**Precondition:** The server must be configured with `hiero.mirror.grpc.listener.type = POLL`. This is not the default (`REDIS`) but is a documented and supported mode: [10](#0-9) 

**Trigger:** Any unauthenticated gRPC client sends one `subscribeTopic` request with `limit` set to any value in `(Integer.MAX_VALUE, Long.MAX_VALUE]` — e.g., `limit = 3_000_000_000`. The proto field is `uint64`, so this is a fully legal value: [11](#0-10) 

No authentication, no special role, and no prior knowledge of the system is required beyond knowing the topic ID.

## Recommendation

1. **Add an upper-bound cap in `PollingTopicListener.poll()`**: Replace the unchecked `(int)` cast with a safe clamp:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Add a `@Max(Integer.MAX_VALUE)` constraint** on `TopicMessageFilter.limit` to reject oversized values at the validation layer: [2](#0-1) 
3. **Apply the same fix to `PollingTopicMessageRetriever.poll()`**, which contains the identical overflow pattern: [8](#0-7) 

## Proof of Concept

```java
// gRPC client (any language) — sends a legal uint64 limit > Integer.MAX_VALUE
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1234).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
    .setLimit(3_000_000_000L)   // valid uint64; > Integer.MAX_VALUE (2_147_483_647)
    .build();
stub.subscribeTopic(query);     // single unauthenticated call

// Server-side arithmetic in PollingTopicListener.poll():
// (int)(3_000_000_000L - 0L) == -1_294_967_296   ← overflow
// Math.min(-1_294_967_296, 5000) == -1_294_967_296
// newFilter.limit = -1_294_967_296L
// hasLimit() → (-1_294_967_296L > 0) == false
// setMaxResults() never called → SELECT ... FROM topic_message WHERE topic_id=1234 (no LIMIT)
// Repeated every 500 ms indefinitely
```
