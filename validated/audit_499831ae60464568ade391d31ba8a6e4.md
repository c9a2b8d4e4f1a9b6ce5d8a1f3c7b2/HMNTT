Audit Report

## Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Concurrent EVM Address Lookups in `getEncodedId()`

## Summary
The REST API's `getEncodedId()` function unconditionally issues a live database query for every EVM address input with no caching of DB lookup results and no application-level rate limiting. The default connection pool is 10 connections per instance. An unauthenticated attacker sending ≥10 concurrent requests with unique valid EVM addresses can saturate the pool, causing all subsequent legitimate requests to queue and time out, rendering the REST API unresponsive.

## Finding Description

**Code path — `getEncodedId()` unconditionally queries DB for EVM addresses:**

In `rest/service/entityService.js`, `getEncodedId()` at lines 118–137, when `entityIdString` is a valid EVM address, `EntityId.parseString()` produces an `EntityId` with `evmAddress !== null`, and the code unconditionally calls `getEntityIdFromEvmAddress()`: [1](#0-0) 

`getEntityIdFromEvmAddress()` at lines 90–104 executes `entityFromEvmAddressQuery` directly against the DB via `this.getRows()`: [2](#0-1) 

**Root cause — no caching of DB lookup results:**

The `parseCached()` LRU cache in `rest/entityId.js` (lines 301–333) caches only the parsed `EntityId` struct (string-to-struct conversion), not the DB lookup result. Even for a repeated EVM address, `getEntityIdFromEvmAddress()` is still called and still queries the DB. For unique addresses (the attack case), there is zero cache benefit: [3](#0-2) 

**Root cause — no rate limiting in `rest/server.js`:**

`rest/server.js` registers no rate-limiting middleware. The middleware stack includes only `cors`, `compression`, `requestLogger`, `authHandler`, optional metrics, and optional response cache — no in-flight request limiting or per-IP rate limiting: [4](#0-3) 

**Root cause — Traefik middleware for REST chart has no rate limiting and is disabled by default:**

The REST chart's Traefik middleware contains only `circuitBreaker` and `retry` — no `inFlightReq` or `rateLimit`: [5](#0-4) 

Furthermore, `global.middleware` defaults to `false`, meaning even this minimal chain is disabled by default: [6](#0-5) 

Contrast with the Rosetta chart, which explicitly configures both `inFlightReq` (5 per IP) and `rateLimit` (10/host): [7](#0-6) 

**Root cause — small default connection pool:**

The default `maxConnections` is **10** per REST instance, as documented: [8](#0-7) 

Configured in `rest/dbpool.js` at line 14: [9](#0-8) 

With `statementTimeout` of 20,000 ms and `connectionTimeoutMillis` of 20,000 ms, each saturating request holds a connection for up to 20 seconds before timing out.

## Impact Explanation

With a pool of 10 connections and no rate limiting, an attacker maintaining ≥10 concurrent in-flight requests with unique EVM addresses fully saturates the pool. All subsequent legitimate requests queue waiting for a free connection; after `connectionTimeout` (20 s default) they fail with a connection timeout error. The REST API becomes effectively unresponsive for the duration of the attack. With up to 15 HPA replicas (each holding 10 connections = 150 total), and pgbouncer capping `mirror_rest` at 250 server-side connections, a sustained flood can exhaust the DB-side pool as well, affecting all REST instances simultaneously. [10](#0-9) [11](#0-10) 

## Likelihood Explanation

The attack requires only the ability to send HTTP GET requests to any public REST endpoint that resolves an account or contract by EVM address (e.g., `/api/v1/accounts/0x<40 hex chars>`, `/api/v1/contracts/0x<40 hex chars>`). No credentials, tokens, or special knowledge are needed. Generating unique 40-character hex strings is trivial. The attack is repeatable, low-cost, and requires no brute force — a single attacker with modest bandwidth can sustain it indefinitely. [12](#0-11) 

## Recommendation

1. **Cache DB lookup results**: Add an application-level cache (e.g., using the existing Redis infrastructure or an in-process LRU) in `getEntityIdFromEvmAddress()` keyed on the EVM address bytes, with a TTL aligned to `hiero.mirror.rest.cache.entityId.maxAge`. This eliminates repeated DB hits for the same address.
2. **Add in-flight request limiting**: Add Traefik `inFlightReq` middleware to the REST chart (mirroring the Rosetta chart's configuration) and set `global.middleware: true` by default, or add application-level concurrency limiting in `rest/server.js`.
3. **Add per-IP rate limiting**: Add a `rateLimit` middleware entry to the REST chart's `middleware` list, as already done for Rosetta.
4. **Increase pool size or add connection queuing limits**: Consider raising `maxConnections` or adding a maximum queue depth so that pool exhaustion fails fast rather than holding requests for 20 seconds.

## Proof of Concept

```bash
# Generate 20 unique EVM addresses and fire concurrent requests
for i in $(seq 1 20); do
  addr=$(openssl rand -hex 20)
  curl -s "https://<mirror-node-rest>/api/v1/accounts/0x${addr}" &
done
wait
# All 10 pool connections are now occupied by in-flight DB queries.
# Any legitimate request issued during this window will queue and
# time out after connectionTimeout (20 s), returning a 500/503 error.
```

Each request triggers `getEncodedId()` → `getEntityIdFromEvmAddress()` → `entityFromEvmAddressQuery` against the DB, holding a pool connection for up to `statementTimeout` (20 s). With 10 concurrent unique-address requests, the pool is fully saturated. [13](#0-12) [9](#0-8)
