Audit Report

## Title
Unauthenticated Hash-Lookup DoS via Uncached 404 on `/api/v1/blocks/{hashOrNumber}` with No Rate Limiting

## Summary
Any unauthenticated caller can send an unbounded stream of syntactically valid but non-existent 96-hex-char block hashes to `GET /api/v1/blocks/{hashOrNumber}`. Each request passes format validation, executes a `LIKE`-prefixed SQL query against the `record_file` table, receives a 404 that is explicitly excluded from the response cache, and is never throttled. Under concurrent load this exhausts the database connection pool and degrades or denies service to all REST API consumers.

## Finding Description

**1. Format validation passes, DB is always reached**

`validateHashOrNumber` in `rest/controllers/blockController.js` calls `utils.isValidEthHashOrHederaHash`, which tests against `ethHashOrHederaHashPattern = /^(0x)?([0-9A-Fa-f]{64}|[0-9A-Fa-f]{96})$/`. Any 96-hex-char string passes unconditionally and is forwarded to `RecordFileService.getByHashOrNumber(hash, null)`. [1](#0-0) [2](#0-1) 

**2. LIKE-prefixed query executed on every miss**

`RecordFileService.getByHashOrNumber` builds:
```sql
SELECT ... FROM record_file WHERE hash LIKE $1
```
where `$1 = hash + '%'`. There is no equality shortcut; every hash lookup is a prefix scan. [3](#0-2) 

**3. 404 responses are never cached**

`responseCacheUpdateHandler` only stores a response when `httpStatusCodes.isSuccess(res.statusCode)` (2xx) or the status is 304. A non-existent block throws `NotFoundError` → HTTP 404, which is explicitly excluded. Every request for a non-existent hash hits the database cold with no amortisation. [4](#0-3) [5](#0-4) 

**4. No rate limiting on the REST service**

A search across all `rest/**/*.js` files and `rest/package.json` found zero references to any rate-limiting or throttling middleware (`express-rate-limit`, `rate-limiter-flexible`, etc.). The `ThrottleConfiguration` / `ThrottleManagerImpl` infrastructure that does exist is scoped exclusively to the `web3` Java service's `/contracts/call` endpoint and has no effect on the Node.js REST service.


## Impact Explanation
An attacker sending N concurrent requests per second with unique valid 96-char hex hashes forces N simultaneous `LIKE`-prefix scans against the database. Because 404 responses are never cached there is no amortisation across requests. The database connection pool is finite; once saturated, all API endpoints (not just `/blocks`) begin queuing or rejecting connections, causing a full non-network denial of service of the mirror node REST API. No authentication, account, or special privilege is required.

## Likelihood Explanation
The attack requires zero privileges and zero knowledge of network state. Generating unique valid 96-char hex strings is trivial (`openssl rand -hex 48`). The endpoint is publicly documented in the OpenAPI spec. A single attacker with a modest HTTP client (`wrk`, `hey`, or a simple script) can sustain thousands of requests per second. The absence of any rate-limiting middleware in the REST service makes the attack repeatable and persistent.

## Recommendation
1. **Rate limiting**: Add per-IP and global request-rate limiting middleware (e.g., `express-rate-limit` backed by Redis) to the Node.js REST service, applied at least to the `/api/v1/blocks/:hashOrNumber` route.
2. **Cache negative results with a short TTL**: Cache 404 responses for a few seconds (e.g., 5 s) so repeated lookups for the same non-existent hash do not reach the database.
3. **Exact-match before LIKE**: If the full 96-char hash is supplied, use an equality predicate (`hash = $1`) rather than a prefix `LIKE`, eliminating the need for a range scan entirely.
4. **Connection pool protection**: Apply a per-endpoint database query timeout and consider a semaphore/concurrency limit on the block-hash lookup path.

## Proof of Concept
```bash
# Generate 10 000 unique valid 96-hex-char hashes and fire them concurrently
for i in $(seq 1 10000); do
  echo "GET /api/v1/blocks/$(openssl rand -hex 48)"
done | wrk -t 10 -c 200 -d 60s --script /dev/stdin http://<mirror-node-host>
```
Each request passes format validation, executes a `LIKE`-prefix scan against `record_file`, returns 404 (never cached), and is never throttled. Under sustained load the database connection pool is exhausted and all REST API endpoints begin returning errors.

```javascript
const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
};
```
