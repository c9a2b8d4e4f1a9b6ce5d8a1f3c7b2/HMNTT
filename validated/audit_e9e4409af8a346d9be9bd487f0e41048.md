Audit Report

## Title
Unauthenticated DB Connection Pool Exhaustion via Alias/EVM Address Lookups on `/accounts/:idOrAliasOrEvmAddress/nfts`

## Summary
Any unauthenticated attacker can exhaust the application's database connection pool by flooding `/api/v1/accounts/{alias_or_evm_address}/nfts` with concurrent requests using syntactically valid but non-existent alias or EVM address values. Each request unconditionally issues a live database query with no result caching and no application-level rate limiting, allowing an attacker to hold all available DB connections simultaneously and render the entire REST API unavailable.

## Finding Description

**Route registration:**
`rest/routes/accountRoute.js` registers the route at lines 12 and 15:
```js
const getPath = (path) => `/:${filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS}/${path}`;
router.getExt(getPath('nfts'), AccountController.getNftsByAccountId);
``` [1](#0-0) 

**Handler immediately calls `getEncodedId` with no prior guard:**
`accountController.js:91` passes the raw path parameter directly to `EntityService.getEncodedId()` before any other validation:
```js
const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
``` [2](#0-1) 

**Two DB-hitting branches inside `getEncodedId`:**

1. **EVM address branch** — when `EntityId.isValidEntityId` returns true for a 40-hex-char string and the parsed `EntityId` has a non-null `evmAddress`, `getEntityIdFromEvmAddress` is called unconditionally, executing `entityFromEvmAddressQuery`: [11](#0-10) [3](#0-2) 

2. **Alias branch** — when `AccountAlias.isValid` returns true for any `[A-Z2-7]+` string, `getAccountIdFromAlias` → `getAccountFromAlias` is called, executing `entityFromAliasQuery`: [12](#0-11) [5](#0-4) 

**No caching of DB results:**
The `quickLru` cache in `entityId.js` caches only the parsed `EntityId` struct (shard/realm/num/evmAddress). Even with a cache hit, if `entityId.evmAddress !== null`, `getEntityIdFromEvmAddress` is still invoked and hits the database. The alias-to-ID mapping is never cached. [7](#0-6) 

**Small connection pool, no rate limiting:**
The pool is initialized with `max: config.db.pool.maxConnections`, `connectionTimeoutMillis`, and `statement_timeout` directly from config: [8](#0-7) 

`rest/server.js` registers no rate-limit, concurrency-limit, or per-IP throttle middleware anywhere in the middleware chain: [13](#0-12) 

**Unbounded alias length:**
The alias regex imposes no upper length limit on the base32 portion, meaning arbitrarily long strings pass `isValid` and proceed to a DB query:
```js
const accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/;
``` [10](#0-9) 

## Impact Explanation
With a bounded DB connection pool and no application-level rate limiting, an attacker holding N concurrent requests (where N equals the pool size) fully exhausts all available connections. All other REST API endpoints — transactions, balances, tokens, contracts, etc. — that require DB access will queue waiting for a free connection and fail with connection timeout errors. This constitutes complete REST API unavailability for the duration of the attack. The attack is self-sustaining: as long as the attacker maintains N concurrent open HTTP connections, the pool remains exhausted.

## Likelihood Explanation
No authentication is required. No rate limiting, no per-IP concurrency limit, and no application-level guard exist. Any internet-accessible mirror node deployment is vulnerable. The attacker needs only a basic HTTP client capable of concurrent requests (e.g., `curl`, `ab`, `wrk`). No knowledge of valid account IDs is needed — any syntactically valid EVM address (e.g., `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`) or uppercase base32 string triggers the DB lookup. The attack is repeatable and requires no special knowledge of the target system.

## Recommendation

1. **Add application-level rate limiting** — integrate a middleware such as `express-rate-limit` in `rest/server.js` to limit requests per IP per time window, applied globally before route handlers.
2. **Add per-endpoint concurrency limiting** — use a semaphore or `express-async-semaphore` to cap the number of concurrent in-flight DB-bound requests.
3. **Cache alias and EVM address DB lookups** — introduce a short-lived (e.g., 5–30 second) in-memory or Redis cache keyed on the alias/EVM address string in `EntityService.getAccountIdFromAlias` and `EntityService.getEntityIdFromEvmAddress` to avoid redundant DB hits for repeated inputs.
4. **Enforce alias length limit** — update `accountAliasRegex` to cap the base32 portion (e.g., `[A-Z2-7]{1,100}`) to prevent excessively large query parameters.
5. **Enable infrastructure-level rate limiting** — ensure Traefik `inFlightReq` and `rateLimit` middlewares are enabled by default in the Helm chart for `hedera-mirror-rest`.

## Proof of Concept

```bash
# Exhaust the DB connection pool using 15 concurrent requests with a non-existent EVM address
for i in $(seq 1 15); do
  curl -s "http://<mirror-node-host>/api/v1/accounts/0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef/nfts" &
done
wait

# Verify pool exhaustion — this legitimate request will now time out or return a 503
curl -v "http://<mirror-node-host>/api/v1/transactions"
```

Each of the 15 concurrent requests enters `getEntityIdFromEvmAddress`, acquires a DB connection, and waits up to `statementTimeout` milliseconds for a response. Once the pool is exhausted, the transactions request cannot acquire a connection and fails.
