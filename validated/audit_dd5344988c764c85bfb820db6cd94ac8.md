Audit Report

## Title
Missing `lower(timestamp_range) <= blockTimestamp` Guard on `entity` Branch Causes Historical State Corruption in `findActiveByEvmAddressAndTimestamp()`

## Summary
The `entity` branch of the `UNION ALL` in `findActiveByEvmAddressAndTimestamp()` omits the `lower(timestamp_range) <= ?2` predicate that the sibling method `findActiveByIdAndTimestamp()` correctly applies. As a result, an entity updated after `blockTimestamp` passes the CTE filter (which only checks the immutable `created_timestamp`) and its post-update row — carrying `timestamp_range = [T_update, ∞)` — is ranked above the correct `entity_history` row by the final `ORDER BY timestamp_range DESC LIMIT 1`, returning the wrong (future) state for any historical EVM query.

## Finding Description

**Affected method:** `findActiveByEvmAddressAndTimestamp()` in `EntityRepository.java`

The CTE resolves the entity ID using only `created_timestamp <= ?2`: [1](#0-0) 

The entity branch emits the current row with no upper-bound timestamp check: [2](#0-1) 

The `entity_history` branch correctly applies `lower(eh.timestamp_range) <= ?2`: [3](#0-2) 

The final sort selects the row with the highest `timestamp_range`, which is always the open-ended `[T_update, ∞)` from the entity table: [4](#0-3) 

**Asymmetry with `findActiveByIdAndTimestamp()`:** That method correctly guards its entity branch with `lower(timestamp_range) <= ?2` at line 140, making the omission in the evm-address variant explicit: [5](#0-4) 

The same missing guard also affects `findActiveByEvmAddressOrAliasAndTimestamp()`: [6](#0-5) 

**Root cause:** `created_timestamp` is immutable after entity creation. `lower(timestamp_range)` in the `entity` table advances to the timestamp of each subsequent update. After an update at `T_update > T_created`, the entity row carries `timestamp_range = [T_update, ∞)` while `created_timestamp` remains `T_created`. The CTE's `created_timestamp <= blockTimestamp` check passes whenever `blockTimestamp >= T_created`, even when `blockTimestamp < T_update`. The entity branch then emits the post-update row unconditionally.

## Impact Explanation

Any historical `eth_call`, `eth_getBalance`, `eth_getCode`, or contract-state read routed through `CommonEntityAccessor.getEntityByEvmAddressTimestamp()` or `getEntityByEvmAddressAndTimestamp()` will silently return the entity's *current* attributes for a past block: [7](#0-6) 

No post-query re-validation of the returned row's `timestamp_range` against `blockTimestamp` exists in `CommonEntityAccessor`. This corrupts the historical view of the ledger: balances appear inflated or deflated, contracts appear to have bytecode they did not yet possess, and deleted-then-recreated accounts may appear with wrong state. Because the mirror node is the authoritative source for historical EVM replay, this undermines the integrity guarantee of all historical queries resolved via EVM address.

## Likelihood Explanation

No privilege is required. Any caller that can issue a standard JSON-RPC call with a historical block number — a universally supported feature of Ethereum-compatible APIs — can trigger the path. The precondition (an entity that has been updated at least once) is satisfied by virtually every active account or contract on the network. The bug is deterministic and repeatable for any entity whose `lower(timestamp_range)` in the entity table exceeds the queried `blockTimestamp`.

## Recommendation

Add `lower(e.timestamp_range) <= ?2` to the entity branch in both `findActiveByEvmAddressAndTimestamp()` and `findActiveByEvmAddressOrAliasAndTimestamp()`, mirroring the guard already present in `findActiveByIdAndTimestamp()` at line 140:

```sql
(
    select *
    from entity e
    where e.deleted is not true
      and e.id = (select id from entity_cte)
      and lower(e.timestamp_range) <= ?2   -- ADD THIS
)
```

This ensures the entity branch only emits the current row when its validity window has already started at or before `blockTimestamp`, which is the invariant `findActiveByIdAndTimestamp()` already enforces.

## Proof of Concept

1. Entity E with `evm_address = A` is created at `T1` → entity table: `timestamp_range = [T1, ∞)`, `created_timestamp = T1`.
2. E is updated at `T2 > T1` → entity_history: `timestamp_range = [T1, T2)`, entity table: `timestamp_range = [T2, ∞)`, `created_timestamp = T1` (unchanged).
3. Caller issues a historical JSON-RPC call (e.g., `eth_getBalance`) with a block number mapping to `T_mid` where `T1 ≤ T_mid < T2`.
4. `findActiveByEvmAddressAndTimestamp(A, T_mid)` executes:
   - CTE: `created_timestamp (T1) <= T_mid` → entity ID resolved ✓
   - Entity branch: row `[T2, ∞)` emitted — no upper-bound check ✗
   - entity_history branch: row `[T1, T2)` emitted — `lower(T1) <= T_mid` ✓
   - `ORDER BY timestamp_range DESC LIMIT 1`: `[T2, ∞)` sorts above `[T1, T2)` → entity row wins.
5. Caller receives the post-`T2` state (e.g., updated balance, new bytecode) instead of the correct `T_mid` state.
