Audit Report

## Title
Unbounded SQL OR Expansion via `inner_transactions` in `getTransactionsByIdOrHash()` Enables DB Connection Pool Exhaustion

## Summary
`getTransactionsByIdOrHash()` in `rest/transactions.js` reads the `inner_transactions` bigint array from a batch transaction row and passes it without any size cap to `getTransactionsByTransactionIdsSql()`, which generates one parameterized OR condition per inner-transaction pair. An unauthenticated attacker who knows the ID of a large ATOMIC_BATCH transaction can repeatedly trigger this path, holding DB connections for up to `statementTimeout` milliseconds per request and exhausting the bounded connection pool.

## Finding Description
In `rest/transactions.js`, `getTransactionsByIdOrHash()` at lines 944–958:

```js
const innerTransactions = rows
  .map((row) => row.inner_transactions)
  .filter((innerTransactions) => innerTransactions)
  .flat();                          // no length cap

if (!isTransactionHash && innerTransactions.length > 0) {
  const {query: innerTransactionQuery, params} = getTransactionsByTransactionIdsSql(
    innerTransactions,              // passed verbatim
    filters,
    Transaction.VALID_START_NS
  );
  ...
}
```

`getTransactionsByTransactionIdsSql()` at lines 840–883 iterates over every pair in `transactionKeys` and appends one OR condition, then joins them all at line 883:

```js
const mainConditions = [...commonConditions, `(${idConditions.join(' or ')})`, ...filterConditions].join(' and ');
```

The `inner_transactions` column is declared as `bigint[] default null` with no size constraint in the schema migration. The code imposes no cap between reading the array and passing it to the SQL builder.

**Partial mitigation:** The correlated subqueries (crypto_transfer, token_transfer, assessed_custom_fee) use a simpler `payer_account_id IN (...)` clause rather than the full OR expansion, reducing their cost relative to what the report claims.

**Protocol-level bound:** The Hedera network protocol limits the number of inner transactions per ATOMIC_BATCH, so the OR clause is not unbounded in absolute terms — but the code itself enforces no such limit, and the maximum protocol-allowed batch size is still sufficient to produce a costly query.

## Impact Explanation
Each request against a large batch transaction ID forces PostgreSQL to evaluate a wide OR predicate on the `transaction` table. The DB connection pool is capped at 10 connections by default. With `statement_timeout` set (default 20 s per `dbpool.js`), each request holds a connection for up to 20 s. A small number of concurrent requests (≥10) against the same batch transaction ID saturates the pool, causing all other API endpoints to queue or time out — a full denial-of-service against the mirror node REST API.

## Likelihood Explanation
- The REST endpoint is public and unauthenticated.
- The attacker pays Hedera network fees once to submit the batch transaction; subsequent queries are free.
- No per-IP or per-endpoint rate limiting exists anywhere in the `rest/` layer.
- Pool exhaustion requires only ~10 concurrent requests, which is trivially achievable.

## Recommendation
1. **Add a size cap before calling `getTransactionsByTransactionIdsSql()`**: enforce a maximum on `innerTransactions.length` (e.g., reject or truncate if it exceeds a configured limit) in `getTransactionsByIdOrHash()` at line 949.
2. **Add rate limiting** on the `/transactions/{id}` endpoint (and ideally all REST endpoints) using a middleware such as `express-rate-limit`.
3. **Consider a DB-level constraint** on the `inner_transactions` array length using a `CHECK` constraint or trigger, so the invariant is enforced at the storage layer as well.

## Proof of Concept
1. Submit an ATOMIC_BATCH transaction to the Hedera network containing the maximum allowed number of inner transactions.
2. Wait for the mirror node importer to ingest the batch and populate `inner_transactions` in the `transaction` table.
3. Send 10+ concurrent `GET /api/v1/transactions/<batch-tx-id>` requests in a tight loop.
4. Each request triggers a second DB query with N OR conditions (N = 2 × number of inner transactions).
5. Each query holds a connection for up to `statementTimeout` ms; with 10 concurrent requests, the pool is saturated and all other API calls queue or time out.
