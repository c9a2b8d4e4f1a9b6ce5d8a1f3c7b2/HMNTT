Audit Report

## Title
Malformed Msgpack Injection via Unauthenticated Redis Causes Sustained Transaction Gossip DoS via Retry Loop

## Summary
An attacker with write access to an unauthenticated Redis instance can publish a crafted malformed msgpack payload to a `topic.<id>` channel. The `Jackson2JsonRedisSerializer` throws a `SerializationException` on deserialization, which terminates the reactive Flux and triggers the `retryWhen` backoff loop in `subscribe()`. During each backoff window the Redis subscription is torn down, causing all legitimate `TopicMessage` gossip published in that window to be silently dropped. Continuous publication of malformed messages maintains a near-permanent subscription gap for any targeted topic.

## Finding Description
**Exact code path:**

`grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`, `subscribe()`, lines 68–79: [1](#0-0) 

The `messageSerializer` is a `Jackson2JsonRedisSerializer` backed by a `MessagePackFactory` `ObjectMapper` with no `StreamReadConstraints` and no per-message error recovery operator (`onErrorContinue`/`onErrorResume`). [2](#0-1) 

When `r.receive(...)` deserializes a message, any malformed msgpack byte sequence causes Jackson to throw a `JsonParseException`, which Spring wraps in a `SerializationException`. This propagates as a Reactor error signal through the Flux.

The `doOnError` at line 76 only logs the error — it does not recover. The `retryWhen` at line 78 then re-subscribes after an exponential backoff (up to `interval × 4`). During this backoff window the `ReactiveRedisMessageListenerContainer` subscription is not active; any legitimate `TopicMessage` published by the importer's `RedisPublisher` during this window is permanently lost. [3](#0-2) 

The importer publishes to the same `topic.<id>` channels via `RedisPublisher.publish()` using Redis pipelining with no deduplication or replay mechanism: [4](#0-3) 

**Why existing checks fail:** The `doOnError` callback is a side-effect only; it does not swallow or skip the error. The `retryWhen` is designed for transient connection failures, not per-message deserialization errors, so every single malformed message causes a full subscription teardown and backoff cycle. The grpc `RedisConfiguration` does not apply `StreamReadConstraints`, unlike `rest-java` and `web3` which explicitly configure `maxDocumentLength`, `maxStringLength`, etc.: [5](#0-4) [6](#0-5) 

## Impact Explanation
Every crafted publish causes a subscription gap of at least `interval` milliseconds (up to `interval × 4` with backoff). All legitimate `TopicMessage` gossip for the targeted topic published during that window is dropped and never delivered to gRPC subscribers. An attacker who continuously publishes malformed messages can keep the subscription in a perpetual retry loop, effectively silencing all real-time transaction gossip for any chosen topic. This directly undermines the mirror node's core function of delivering consensus topic messages to clients. There is no deduplication or replay mechanism — dropped pub/sub messages are gone.

## Likelihood Explanation
Redis ships with authentication disabled by default. Exposed Redis instances (port 6379) are actively scanned by attackers. The attacker requires only TCP connectivity to the Redis port and the ability to issue a single `PUBLISH` command — no credentials, no application-layer access, no knowledge of internal state beyond the predictable channel name format `topic.<numeric_id>`. The attack is trivially repeatable with a simple loop and requires no special tooling beyond `redis-cli` or a raw TCP socket.

## Recommendation
1. **Add per-message error recovery** in `subscribe()` using `.onErrorContinue(...)` or `.onErrorResume(...)` before the `retryWhen`, so that a single deserialization failure skips the bad message rather than tearing down the entire subscription.
2. **Apply `StreamReadConstraints`** to the `MessagePackFactory`-backed `ObjectMapper` in `grpc/src/main/java/org/hiero/mirror/grpc/config/RedisConfiguration.java`, consistent with the approach already used in `rest-java` and `web3`.
3. **Enforce Redis authentication** (`requirepass`) and network-level access controls (firewall rules, VPC isolation) so that only the importer service can publish to Redis channels.

## Proof of Concept
```bash
# Publish a malformed msgpack payload to a targeted topic channel
# This triggers a SerializationException in Jackson2JsonRedisSerializer,
# terminating the Flux and initiating the retryWhen backoff cycle.
redis-cli -h <redis-host> -p 6379 PUBLISH topic.1234 "\xff\xfe\xfd\x00\x01"

# Repeat in a loop to maintain a perpetual subscription gap:
while true; do
  redis-cli -h <redis-host> -p 6379 PUBLISH topic.1234 "\xff\xfe\xfd\x00\x01"
  sleep 0.5
done
# All legitimate TopicMessages published during each backoff window are silently dropped.
```

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
