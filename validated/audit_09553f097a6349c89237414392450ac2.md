Audit Report

## Title
Retry Loop Creates Fresh Transaction IDs on Each Attempt, Enabling Multiple On-Chain Transfers Per Tick via gRPC Error Injection

## Summary
`submitWithRetry()` in `pinger/transfer.go` constructs a new `TransferTransaction` — and therefore a new unique transaction ID — on every loop iteration. An attacker with a MITM position on the gRPC channel can inject a transient error after a transaction has already been accepted by the Hedera network, causing the function to discard the response and resubmit a distinct transaction. Because each retry carries a different transaction ID, the network cannot deduplicate them, resulting in up to 11 independent on-chain fund transfers per tick invocation under default configuration.

## Finding Description
The loop body in `submitWithRetry()` unconditionally calls `hiero.NewTransferTransaction()` on every iteration: [1](#0-0) 

`hiero.NewTransferTransaction()` generates a fresh transaction ID (operator account + current nanosecond timestamp) on each call. `Execute(client)` signs and submits that transaction to a Hedera consensus node over gRPC. The gRPC call is two-phase: the node receives and processes the transaction, then sends back a response. If the attacker intercepts the response leg and substitutes a transient gRPC error (`UNAVAILABLE`, `DEADLINE_EXCEEDED`), `Execute()` returns a non-nil error even though the transaction was already finalized on-chain.

The error path logs the failure, sleeps for a backoff period, and continues to the next iteration: [2](#0-1) 

On the next iteration, a new `NewTransferTransaction()` call produces a new transaction ID. The Hedera network has no basis to deduplicate this against the prior accepted transaction and processes it as an independent transfer.

**No deduplication guard exists.** The only early-exit paths are `ctx.Err()` (line 24) and a successful receipt (lines 36–39). There is no idempotency key, no transaction ID reuse across retries, and no pre-retry check for a prior successful submission.

Default `maxRetries = 10`: [3](#0-2) 

So `attempts = cfg.maxRetries + 1 = 11`. Each of the 11 iterations can independently land on-chain.

**Note on timing:** The backoff is exponential (`base * 2^(attempt-1)`, capped at 30 s): [4](#0-3) 

With the default `baseBackoff = 2s`, exhausting all 11 attempts requires approximately 210 seconds of cumulative sleep. The 11 duplicate transfers occur within a single tick invocation, not within a single 1-second interval. The report's "11× per second" rate framing is overstated; the correct framing is up to 11 transfers per tick invocation, spread over several minutes of backoff.

## Impact Explanation
Each successful on-chain transfer debits `cfg.amountTinybar` (default 10,000 tinybar = 0.0001 HBAR) from the operator account. Under a sustained MITM attack, every tick invocation can produce up to 11 accepted transfers instead of 1, multiplying the financial drain by up to 11×. This is a direct, repeated loss of funds from the operator account with no on-chain safeguard. The operator private key is never exposed; the attacker only needs to manipulate the transport layer.

## Likelihood Explanation
The precondition is a MITM position on the gRPC channel between the pinger pod and the Hedera consensus nodes. Realistic vectors include:

- **BGP hijacking / rogue DNS**: Redirecting the consensus node address to an attacker-controlled endpoint that forwards the transaction to the real node and returns a synthetic error response. This does not require local network access.
- **Compromised container network / service mesh**: An attacker who has compromised another pod in the same Kubernetes namespace can intercept pod-to-pod or pod-to-external gRPC traffic.
- **Compromised load balancer or ingress**: Infrastructure-level interception without requiring application credentials.

The attack is repeatable on every tick invocation, is fully automated, and requires no knowledge of the operator private key or any Hedera account.

## Recommendation
1. **Reuse the transaction ID across retries.** Generate the `TransferTransaction` (and its transaction ID) once before the loop and reuse the same object on each retry attempt. The Hedera network will deduplicate submissions sharing the same transaction ID within its validity window.
2. **Query receipt before retrying.** Before constructing a new transaction, query the network for the status of the previous transaction ID. If it was accepted (`SUCCESS`), return immediately.
3. **Limit retries for financial transactions.** Consider reducing `maxRetries` to 0 or 1 for operations that transfer funds, accepting a missed tick rather than risking duplicate debits.

## Proof of Concept
1. Operator runs the pinger with default config (`maxRetries=10`, `amountTinybar=10000`, `interval=1s`).
2. Attacker establishes a MITM proxy on the gRPC channel (e.g., via BGP hijack or compromised network path).
3. On each tick, the proxy forwards the `CryptoTransfer` gRPC request to the real Hedera consensus node and receives the acceptance response.
4. The proxy discards the real response and returns a synthetic `UNAVAILABLE` status to the pinger.
5. `Execute()` returns a non-nil error; the loop sleeps `backoff(2s, 1) = 2s` and iterates.
6. A new `NewTransferTransaction()` call at line 29 produces a new transaction ID; the proxy again forwards and drops the response.
7. Steps 5–6 repeat up to 10 more times.
8. Result: up to 11 distinct `CryptoTransfer` transactions, each with a unique ID, are accepted on-chain per tick invocation. The operator account is debited up to 110,000 tinybar (11 × 10,000) instead of 10,000 per invocation.
