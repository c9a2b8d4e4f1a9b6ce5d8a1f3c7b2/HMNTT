Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses `maxPageSize` Guard, Causing Unlimited Database Queries

## Summary
When `hiero.mirror.grpc.listener.type = POLL`, a gRPC client supplying a `limit` value greater than `Integer.MAX_VALUE` (e.g., `3_000_000_000`) causes a silent narrowing cast overflow in `PollingTopicListener.poll()`. The resulting negative `pageSize` is stored in a rebuilt `TopicMessageFilter`, causing `hasLimit()` to return `false` and `findByFilter()` to issue an unbounded SQL query on every poll cycle (default: every 500 ms).

## Finding Description

**Step 1 — No upper-bound check on ingress.**
`ConsensusController.toFilter()` maps the proto `uint64 limit` field directly to `TopicMessageFilter.limit` (a Java `long`) with no cap: [1](#0-0) 

**Step 2 — Validation only enforces `>= 0`.**
`TopicMessageFilter.limit` carries only `@Min(0)`, so any value in `[0, Long.MAX_VALUE]` passes Bean Validation, including values far above `Integer.MAX_VALUE`: [2](#0-1) 

**Step 3 — Service layer passes the large limit unchanged.**
`TopicMessageServiceImpl.incomingMessages()` computes the remaining limit correctly as a `long` and passes it to `topicListener.listen(newFilter)`: [3](#0-2) 

**Step 4 — Narrowing cast to `int` overflows.**
Inside `PollingTopicListener.poll()`, the remaining `long` is cast to `int` before `Math.min`: [4](#0-3) 

With `filter.getLimit() = 3_000_000_000L` and `count = 0`:
- `(int)(3_000_000_000L - 0L)` → `-1_294_967_296` (wraps past `Integer.MAX_VALUE`)
- `Math.min(-1_294_967_296, 5000)` → `-1_294_967_296`
- `newFilter.limit = -1_294_967_296L` (widened back to `long` when stored)

**Step 5 — Negative limit bypasses `setMaxResults()`.**
`TopicMessageRepositoryCustomImpl.findByFilter()` gates `setMaxResults()` on `filter.hasLimit()`, which returns `limit > 0`. A negative value returns `false`, so no SQL `LIMIT` clause is ever added: [5](#0-4) [6](#0-5) 

## Impact Explanation
Every poll cycle (default 500 ms) issues a full table scan for the topic with no row limit. The `maxPageSize = 5000` guard is completely bypassed. [7](#0-6) 

A topic with millions of messages causes the server to load all rows into memory on each tick, exhausting heap and database connection resources. The outer `flux.take(filter.getLimit())` in `TopicMessageServiceImpl` limits records delivered to the client but does not prevent the repeated unbounded DB queries: [8](#0-7) 

This is a server-side Denial-of-Service: a single persistent gRPC subscription can degrade or crash the mirror node for all other subscribers.

## Likelihood Explanation
**Precondition:** The server must be configured with `hiero.mirror.grpc.listener.type = POLL`. The default is `REDIS`, but `POLL` is a documented and supported mode: [9](#0-8) 

**Trigger:** Any unauthenticated gRPC client sends one `subscribeTopic` request with `limit` set to any value in `(Integer.MAX_VALUE, Long.MAX_VALUE]`. No authentication, no special role, and no prior knowledge beyond a valid topic ID is required.

## Recommendation

1. **Cap the limit at ingress.** In `ConsensusController.toFilter()`, clamp `query.getLimit()` to `Integer.MAX_VALUE` (or a configurable maximum) before storing it in `TopicMessageFilter`.

2. **Fix the narrowing cast.** In `PollingTopicListener.poll()`, replace:
   ```java
   int limit = filter.hasLimit()
           ? (int) (filter.getLimit() - context.getCount().get())
           : Integer.MAX_VALUE;
   ```
   with a safe clamp:
   ```java
   long remaining = filter.hasLimit()
           ? filter.getLimit() - context.getCount().get()
           : listenerProperties.getMaxPageSize();
   int pageSize = (int) Math.min(remaining, listenerProperties.getMaxPageSize());
   ```
   This eliminates the narrowing cast entirely and makes `maxPageSize` the hard ceiling.

3. **Add a `@Max` constraint.** Add `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) to `TopicMessageFilter.limit` to enforce the bound at the validation layer.

## Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

request = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1234),
    limit=3_000_000_000,   # > Integer.MAX_VALUE; triggers int overflow in poll()
)

# Single subscription; every 500ms the server issues an unbounded
# SELECT * FROM topic_message WHERE topic_id = 1234 AND consensus_timestamp >= ?
# with no LIMIT clause, loading all rows into memory on each tick.
for response in stub.subscribeTopic(request):
    pass  # attacker receives at most 3B messages; server OOMs long before that
```
