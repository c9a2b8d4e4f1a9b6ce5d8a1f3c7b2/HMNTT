## Audit Report

## Title
Cache Poisoning via Incorrect `unless` Guard on `Optional`-Returning `findRuntimeBytecode()`

## Summary
`ContractRepository.findRuntimeBytecode()` is annotated with `@Cacheable(unless = "#result == null")` but returns `Optional<byte[]>`. When no contract row exists, the method returns `Optional.empty()`, which is a non-null object. The `unless` guard evaluates to `false`, so Spring proceeds to cache the unwrapped `null` value. This means negative lookups (non-existent contracts) are permanently cached with a 1-hour access-based TTL, causing the EVM to return incorrect results for any contract ID that was queried before the contract was deployed and indexed.

**Root cause:** Spring's `CacheAspectSupport` evaluates the `unless` SpEL expression against the actual method return value — the `Optional` object itself, not the unwrapped value. `Optional.empty() != null`, so `#result == null` is always `false` for empty results. Spring then unwraps the `Optional` to `null` and stores it in Caffeine (which permits null values by default since `allowNullValues` defaults to `true` in `CaffeineCacheManager`). The developer's intent — to skip caching of absent results — is never realized. The correct guard would be `unless = "#result == null || !#result.isPresent()"`.

**Why existing checks are insufficient:** The `unless = "#result == null"` guard is the sole protection against caching empty results. It is structurally incorrect for an `Optional`-returning method. No secondary guard, no separate negative-result cache with a shorter TTL, and no cache invalidation on contract deployment exist.

## Impact Explanation
There are two distinct impacts:

1. **Stale negative caching (correctness):** With `expireAfterAccess=1h`, a cached null entry for a given `contractId` persists for one hour after the last access. If a contract ID is queried before the contract is deployed and indexed, the cache stores a null entry. After deployment, `ContractBytecodeReadableKVState.readFromDataSource()` returns `null` (via `.orElse(null)`) for up to one hour, causing the EVM to treat the contract as non-existent. Any `eth_call` or `eth_estimateGas` targeting that contract returns incorrect results — as if the contract has no bytecode — for the duration of the TTL window.

2. **Cache eviction / thrashing (availability):** The cache has `maximumSize=1000`. An attacker querying 1,000+ distinct non-existent contract IDs fills the cache with null entries, evicting legitimate bytecode entries and forcing every subsequent EVM call to a real contract to re-query the database, increasing latency and database load.

## Likelihood Explanation
No authentication is required. The web3 JSON-RPC endpoint is publicly accessible. Hedera contract IDs are sequential integers, making enumeration of non-existent IDs trivial. The attack requires only standard `eth_call` or `eth_estimateGas` requests. The stale negative caching scenario also occurs organically (without any attacker) whenever a user queries a contract ID before the importer has indexed the deployment transaction.

## Recommendation
Fix the `unless` guard to correctly handle the `Optional` return type:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

The same incorrect guard pattern appears in `EntityRepository.java` (both `web3` and `grpc` modules) and should be audited for the same issue. [5](#0-4) [6](#0-5) 

Additionally, consider using `expireAfterWrite` instead of `expireAfterAccess` for the contract cache to bound the maximum staleness window regardless of access patterns.

## Proof of Concept
1. Deploy a fresh mirror node instance with default configuration.
2. Send an `eth_call` targeting a contract ID (e.g., `0.0.99999`) that does not yet exist in the database. The repository returns `Optional.empty()`; the `unless` guard does not fire; Caffeine stores `null` for key `99999L`.
3. Deploy a contract to the network and wait for the importer to index it at ID `0.0.99999`.
4. Send another `eth_call` to `0.0.99999`. The cache returns the stored `null`; `readFromDataSource()` returns `null`; the EVM treats the contract as having no bytecode and returns an incorrect result.
5. This incorrect behavior persists until the cache entry expires (up to 1 hour after the last access to that key).
