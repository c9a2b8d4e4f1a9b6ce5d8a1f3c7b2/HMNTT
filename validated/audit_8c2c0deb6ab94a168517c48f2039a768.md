## Audit Report

## Title
Unauthenticated Alias-Triggered DB Query Flood Exhausts REST Connection Pool (DoS)

## Summary
The `/api/v1/balances` endpoint unconditionally triggers a live database query for every request containing a syntactically valid base32 alias as the `account.id` parameter. Because the REST API has no application-level rate limiting and the response cache is keyed per unique URL, an attacker flooding the endpoint with distinct alias values generates a sustained stream of concurrent DB queries that can exhaust the finite connection pool, rendering the mirror node REST API unavailable.

## Finding Description

**Code path:**

`rest/balances.js` `getBalances()` (line 87) calls `parseAccountIdQueryParam(req.query, 'ab.account_id')`:

```js
const [accountQuery, accountParamsPromise] = parseAccountIdQueryParam(req.query, 'ab.account_id');
``` [1](#0-0) 

Inside `parseAccountIdQueryParam` (lines 320–357), when `AccountAlias.isValid(value, true)` passes and `evmAliasAddressCount === 1` (first alias in this request), it calls:

```js
return EntityService.getAccountIdFromAlias(AccountAlias.fromString(value), false);
``` [2](#0-1) 

`AccountAlias.isValid(value, true)` uses the regex `/^[A-Z2-7]+$/` — any uppercase base32 string passes with no authentication: [3](#0-2) 

`getAccountIdFromAlias` chains to `getAccountFromAlias`, which executes a live DB query:

```js
const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
``` [4](#0-3) 

The query is:
```sql
select id from entity where coalesce(deleted, false) <> true and alias = $1
``` [10](#0-9) 

The DB pool is bounded by `config.db.pool.maxConnections`: [5](#0-4) 

The response cache key is MD5 of `req.originalUrl`:

```js
return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
``` [6](#0-5) 

Each distinct alias value produces a unique URL → unique cache key → cache miss → DB query.

`server.js` registers the balances route with no throttle middleware: [7](#0-6) 

The throttle infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`) is scoped exclusively to the `web3` Spring Boot module and operates on `ContractCallRequest` objects — it has no bearing on the Node.js REST service: [11](#0-10) 

**Why the `evmAliasAddressCount` guard is insufficient:**

The counter is declared locally inside each call to `parseAccountIdQueryParam`, so it only prevents more than one alias *within a single request*. It does nothing to limit alias lookups *across* concurrent requests: [9](#0-8) 

## Impact Explanation

An attacker can exhaust the `mirror_rest` DB connection pool. Once exhausted, all REST API endpoints that require DB access begin queuing or timing out (`connectionTimeoutMillis`). This causes complete unavailability of the mirror node REST API for all downstream consumers (wallets, explorers, dApps). The `statement_timeout` config partially bounds per-query duration but does not prevent pool exhaustion under sustained concurrent load.

## Likelihood Explanation

- **No privileges required:** The `/api/v1/balances` endpoint is fully public.
- **Trivial alias generation:** Valid aliases are any uppercase base32 strings (`[A-Z2-7]+`). An attacker can generate millions of distinct values programmatically (e.g., `AAAAAAA`, `AAAAAAB`, …).
- **Amplification:** A single HTTP client with async concurrency (`wrk`, `ab`, `curl` with `&`) can saturate the pool. Each request holds a DB connection for the full query round-trip.
- **Repeatability:** The attack is stateless and trivially repeatable. No session, token, or account is needed.

## Recommendation

1. **Add application-level rate limiting** to the REST API (e.g., `express-rate-limit`) for all endpoints, particularly those that trigger DB lookups based on user-supplied parameters.
2. **Cache alias-to-ID resolutions** in-process (e.g., an LRU cache with a short TTL) so repeated lookups for the same alias do not each consume a DB connection.
3. **Enforce infrastructure-level rate limiting** via the existing Traefik middleware support in the Helm chart (`charts/hedera-mirror-rest/templates/middleware.yaml`) with `inFlightReq` and `rateLimit` entries, similar to what is already configured for the Rosetta service.
4. **Consider rejecting alias lookups that resolve to null** earlier in the pipeline, or short-circuiting with a negative cache entry to avoid repeated DB hits for non-existent aliases.

## Proof of Concept

```bash
# Generate 500 unique base32 aliases and flood the endpoint concurrently
python3 -c "
import itertools, string
chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'
aliases = [''.join(p) for p in itertools.product(chars[:8], repeat=3)]
for a in aliases[:500]:
    print(f'http://mirror-node/api/v1/balances?account.id={a}')
" | xargs -P 100 -I{} curl -s {} > /dev/null
```

Each request with a unique alias value (`AAA`, `AAB`, `AAC`, …) bypasses the response cache (unique URL → unique MD5 key) and triggers a live `SELECT id FROM entity WHERE alias = $1` query. With 100 concurrent workers and a pool capped at the configured `maxConnections`, the pool is exhausted and subsequent requests to any REST endpoint begin timing out with `connectionTimeoutMillis` errors.
