Audit Report

## Title
Unbounded Heap Allocation via Oversized Hex String in Construction Endpoints Leading to Memory Exhaustion DoS

## Summary
The `unmarshallTransactionFromHexString` function in `rosetta/app/services/construction_service.go` passes attacker-controlled input directly to `hex.DecodeString` with no prior length check. No HTTP body size limit (`MaxBytesReader` or equivalent) exists anywhere in the rosetta codebase. An unauthenticated attacker can POST arbitrarily large hex strings to four public construction endpoints, causing proportional heap allocations and potential OOM under concurrent load.

## Finding Description
The function `unmarshallTransactionFromHexString` at lines 658–662 calls `hex.DecodeString` on unsanitized input: [1](#0-0) 

This function is called unconditionally from four public endpoints:

- `ConstructionCombine` → line 62 [2](#0-1) 
- `ConstructionHash` → line 121 [3](#0-2) 
- `ConstructionParse` → line 187 [4](#0-3) 
- `ConstructionSubmit` → line 340 [5](#0-4) 

Go's `hex.DecodeString` internally converts the input string to `[]byte` (`src := []byte(s)`) before decoding, allocating a full copy of the input. For a 500 MB hex string this means ~500 MB for the copy plus ~250 MB for the decoded output, all on the heap per request.

The `Http` config struct contains only timeout fields — no body size limit field exists: [6](#0-5) 

The HTTP server in `main.go` applies only those timeout values: [7](#0-6) 

The middleware chain consists solely of `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware`: [8](#0-7) 

A full-codebase grep for `MaxBytesReader`, `LimitReader`, and related patterns returns **zero matches** across the entire rosetta subtree. 

## Impact Explanation
An attacker sending concurrent requests each containing a multi-hundred-megabyte hex string can exhaust the process heap, triggering an OOM kill or severe GC pressure that renders the Rosetta API unresponsive. `ConstructionSubmit` is online-only, but `/construction/combine`, `/construction/hash`, and `/construction/parse` are available in offline mode as well, widening the attack surface. All four endpoints are unauthenticated per the Rosetta specification.

## Likelihood Explanation
The attack requires no credentials, no prior system knowledge, and no special tooling. A single `curl -d @large_payload.json` suffices. The endpoints are part of the standard public Rosetta API surface. Concurrent requests amplify impact multiplicatively. No authentication, rate limiting, or body size cap exists anywhere in the middleware chain.

## Recommendation
1. **Wrap the request body with `http.MaxBytesReader`** in a middleware applied before the rosetta-sdk-go router, limiting bodies to a reasonable maximum (e.g., 1 MB).
2. **Add an explicit length guard** in `unmarshallTransactionFromHexString` before calling `hex.DecodeString`, rejecting strings exceeding a defined maximum (e.g., matching the maximum valid Hiero transaction size, which is well under 10 KB).
3. **Add the `MaxBodySize` field** to the `Http` config struct in `rosetta/app/config/types.go` and wire it into a `MaxBytesReader` middleware so the limit is operator-configurable.

## Proof of Concept
```bash
# Generate a ~200 MB hex string payload
python3 -c "
import json, sys
payload = {'signed_transaction': 'aa' * 100_000_000}
json.dump(payload, sys.stdout)
" > large_payload.json

# Send to /construction/hash (no auth required)
curl -s -X POST http://<rosetta-host>:5700/construction/hash \
  -H 'Content-Type: application/json' \
  -d @large_payload.json &

# Repeat concurrently to exhaust heap
for i in $(seq 1 10); do
  curl -s -X POST http://<rosetta-host>:5700/construction/hash \
    -H 'Content-Type: application/json' \
    -d @large_payload.json &
done
wait
```
Each concurrent request allocates ~300–400 MB on the heap before returning `ErrTransactionDecodeFailed`. Ten concurrent requests can exhaust ~3–4 GB of heap, causing OOM or severe GC stalls.
