Audit Report

## Title
`incrementByteArray()` Integer Overflow on Max-Value `gt` Parameter Exposes Full Hook Storage

## Summary
Supplying `key=gt:0xFFFF...FFFF` (64 hex `F`s) to the hook storage endpoint causes `BytesUtil.incrementByteArray()` to silently overflow to `0x00...00`. The resulting value equals the default lower bound, so the "more restrictive" guard in `hookStorageChangeRequest()` never updates `lowerBound`, and the repository query returns every non-deleted storage slot for the targeted hook.

## Finding Description

**Step 1 — Parsing (`SlotRangeParameter.java:41-49`)**

When `operator == GT`, `getInclusiveValue` calls `incrementByteArray(bytes)`: [1](#0-0) 

**Step 2 — Silent overflow (`BytesUtil.java:13-14`)**

`MutableBytes32.wrap(bytes).increment()` treats the 32-byte array as a big-endian integer. `0xFF…FF + 1` wraps to `0x00…00` with no exception: [2](#0-1) 

**Step 3 — Operator normalisation (`SlotRangeParameter.java:38`, `RangeOperator.java:53-55`)**

`operator.toInclusive()` converts `GT → GTE`, so the stored parameter is `(GTE, [0x00×32])`: [3](#0-2) [4](#0-3) 

**Step 4 — Guard bypass (`HooksController.java:167,176-178`)**

`lowerBound` is initialised to `MIN_KEY_BYTES` (`[0x00×32]`). The guard only updates it when `value > lowerBound`. Because the overflowed value equals `lowerBound`, `compareUnsigned` returns `0`, the condition `> 0` is false, and `lowerBound` is never changed: [5](#0-4) [6](#0-5) 

**Step 5 — Full-table scan (`HookServiceImpl.java:65-68`)**

The repository is called with `keyLowerBound = [0x00×32]` and `keyUpperBound = [0xFF×32]` (the unchanged default), returning every non-deleted slot: [7](#0-6) 

## Impact Explanation
An unauthenticated caller can enumerate the complete current storage of any smart contract hook with a single paginated GET request. Storage slots may contain balances, access-control mappings, or other sensitive application state. No authentication is required on the endpoint.

## Likelihood Explanation
The exploit requires zero privileges and zero on-chain interaction. `ownerId` and `hookId` are enumerable via the `/hooks` endpoint. The payload is one well-formed query parameter (`key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF`). It is trivially repeatable and automatable.

## Recommendation
Add an explicit overflow check in `BytesUtil.incrementByteArray`: if the input is `[0xFF×32]`, throw an `IllegalArgumentException` (or return a sentinel that the caller treats as "no valid key exists above this value", resulting in an empty response). Alternatively, add a post-increment validation in `getInclusiveValue` — if the returned byte array is lexicographically less than the original input, the increment wrapped and the parameter should be rejected.

Additionally, the guard in `hookStorageChangeRequest` should use `>= 0` (or `== 0` treated as a no-op with a separate overflow check) to be robust against future edge cases.

## Proof of Concept
```
GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage?key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF
```
1. `SlotRangeParameter.valueOf(...)` parses `GT` + 64 `F`s.
2. `incrementByteArray([0xFF×32])` → `[0x00×32]` (overflow).
3. Stored as `(GTE, [0x00×32])`.
4. `Arrays.compareUnsigned([0x00×32], [0x00×32]) == 0` → guard skipped, `lowerBound` stays `[0x00×32]`.
5. Repository called with range `[0x00×32, 0xFF×32]` → all storage slots returned.
