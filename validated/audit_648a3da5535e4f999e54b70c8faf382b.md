Audit Report

## Title
Integer Narrowing Cast Overflow in `poll()` Enables Unbounded Query via `limit = Long.MAX_VALUE`

## Summary
In `PollingTopicMessageRetriever.poll()`, an unchecked narrowing cast from `long` to `int` overflows when a caller supplies `limit = Long.MAX_VALUE`. The resulting `-1` value propagates as the page size into the downstream JPA repository, which — because `hasLimit()` returns `false` for `-1` — omits the `LIMIT` clause entirely, causing an unbounded full-table scan on every poll interval for the lifetime of the subscription.

## Finding Description

**Location:** `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `poll()`, lines 68–71.

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())   // line 69
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());        // line 71
``` [1](#0-0) 

**Arithmetic overflow (confirmed):**
- `filter.getLimit()` = `Long.MAX_VALUE` = `0x7FFFFFFFFFFFFFFF`
- `context.getTotal().get()` = `0L` (first poll)
- `(int) 0x7FFFFFFFFFFFFFFF` truncates to low 32 bits: `0xFFFFFFFF` = **`-1`** (signed int)
- `Math.min(-1, maxPageSize)` = **`-1`** (maxPageSize is always positive)

**Propagation into the repository (confirmed):**

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
// newFilter.limit == -1L
``` [2](#0-1) 

`hasLimit()` is defined as `return limit > 0;`, so `-1 > 0` = `false`. [3](#0-2) 

In `TopicMessageRepositoryCustomImpl.findByFilter()`, the LIMIT clause is only applied when `hasLimit()` is `true`:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [4](#0-3) 

Since `hasLimit()` returns `false`, `setMaxResults()` is **never called**, and the JPA query executes with no row cap — fetching every matching row in `topic_message` for the given topic.

**Why existing guards fail:**

- `@Min(0)` on `TopicMessageFilter.limit` only enforces a floor of 0; `Long.MAX_VALUE` satisfies it. [5](#0-4) 
- Lombok's `toBuilder()` constructs the derived `newFilter` without triggering Spring Bean Validation, so the `@Min(0)` constraint is never re-evaluated on the `-1` value. [6](#0-5) 
- `isComplete()` checks `filter.getLimit() == total.get()` against the **original** filter's `Long.MAX_VALUE`, which `total` can never realistically reach, so the subscription never terminates. [7](#0-6) 
- The retry operator `.retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))` ensures any exception path also loops indefinitely. [8](#0-7) 

## Impact Explanation

On every poll interval, `findByFilter(newFilter)` executes a full scan of `topic_message` for the given topic with no row limit. The result set is loaded entirely into memory via `getResultList().stream()`. [9](#0-8) 

This causes:
1. **Memory exhaustion** in the gRPC service JVM (entire topic history loaded per poll).
2. **Extreme database load** — repeated full scans on a high-volume topic, degrading service for all subscribers.
3. **Subscription never terminates** — `isComplete()` never returns `true`, so polling continues indefinitely at the configured frequency, compounding the load.

A single malicious `subscribeTopic` call is sufficient to trigger this condition.

## Likelihood Explanation

The `subscribeTopic` gRPC endpoint is publicly accessible with no authentication required. The `limit` field maps to a `uint64`/`long` in the protobuf definition, so any gRPC client can trivially set it to `9223372036854775807`. No internal knowledge, credentials, or special state are required. The attack is independently repeatable per subscription.

## Recommendation

1. **Add an `@Max` constraint** on `TopicMessageFilter.limit` to cap the maximum accepted value (e.g., `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate business limit).
2. **Replace the narrowing cast** with a bounds-safe calculation:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
3. **Add a defensive guard** in `findByFilter` to enforce a minimum page size of 1 if `hasLimit()` is `true` but `getLimit() <= 0`, or assert that `limit > 0` before calling `setMaxResults`.
4. **Validate the derived filter** — either run Bean Validation on the `toBuilder()`-constructed object or add an explicit check before passing it to the repository.

## Proof of Concept

```java
// gRPC client pseudocode
ConsensusTopicQuery query = new ConsensusTopicQuery()
    .setTopicId(TopicId.fromString("0.0.12345"))
    .setLimit(Long.MAX_VALUE);  // 9223372036854775807

// This triggers:
// poll() -> (int)(Long.MAX_VALUE - 0) = -1
// pageSize = Math.min(-1, 1000) = -1
// newFilter.limit = -1L, hasLimit() = false
// findByFilter: setMaxResults() skipped -> full table scan
// isComplete() never true -> unbounded polling loop
```
