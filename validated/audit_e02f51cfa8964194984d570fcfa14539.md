Audit Report

## Title
Malicious Block Node Can Permanently Evade Selection via Crafted `ServerStatus` Response Without Triggering Circuit Breaker

## Summary
`BlockNode.getBlockRange()` silently catches `IllegalArgumentException` thrown by Guava's `Range.closed()` when a block node returns `firstAvailableBlock > lastAvailableBlock`. Because `onError()` is never called in this path, the node's `active` flag and `errors` counter are never updated. The node is skipped on every selection cycle but is never marked inactive and never enters the readmit cooldown, causing permanent block ingestion failure when it is the sole or last available node.

## Finding Description
In `BlockNode.getBlockRange()` (lines 99–112), the call `Range.closed(firstBlockNumber, response.getLastAvailableBlock())` at line 106 throws `IllegalArgumentException` if `firstBlockNumber > lastAvailableBlock`, per Guava's documented contract. [8](#0-7) 

The broad `catch (Exception ex)` at line 108 swallows this exception, logs an error, and returns `EMPTY_BLOCK_RANGE` — without calling `onError()`. [2](#0-1) 

By contrast, `streamBlocks()` calls `onError()` on every exception path (lines 159–164), which increments the error counter and eventually marks the node inactive. [3](#0-2) 

`onError()` is the sole mechanism that sets `active = false` and schedules a `readmitTime` cooldown. [4](#0-3) 

In `BlockNodeSubscriber.getNode()`, active nodes are checked via `hasBlock()` → `getBlockRange()`. An empty range causes `hasBlock()` to return `false` and the node is skipped. [9](#0-8) 

The fallback loop (lines 122–128) force-readmits nodes from `inactiveNodes`, but since `getBlockRange()` never calls `onError()`, the node remains `active = true` and is never added to `inactiveNodes`. This fallback is unreachable for this failure mode. [6](#0-5) 

If no node passes `hasBlock()`, `getNode()` throws `BlockStreamException("No block node can provide block X")` on every `doGet()` invocation. [10](#0-9) 

## Impact Explanation
If an attacker controls all configured block nodes (or the single configured node), they can halt block ingestion permanently. The mirror node will repeatedly query each node's status, receive `EMPTY_BLOCK_RANGE` every time, and throw `BlockStreamException` on every `doGet()` call. Because `onError()` is never invoked from `getBlockRange()`, the circuit breaker never fires, the node is never marked inactive, and the system loops indefinitely without self-healing. This constitutes a complete denial of block data ingestion with no automatic recovery path.

## Likelihood Explanation
Block node operators are semi-trusted external parties. An attacker only needs to control the gRPC endpoint the mirror node is configured to contact — no special privilege within the mirror node itself is required. The malicious `ServerStatus` response (e.g., `firstAvailableBlock=100`, `lastAvailableBlock=50`) can be returned on every status poll with zero additional effort, making the attack trivially repeatable and persistent.

## Recommendation
1. **Validate the range before constructing it**: In `getBlockRange()`, check `firstBlockNumber <= lastAvailableBlock` before calling `Range.closed()`. If the invariant is violated, treat it as a node error and call `onError()` before returning `EMPTY_BLOCK_RANGE`.
2. **Distinguish malformed responses from transient errors**: Narrow the `catch` clause or add explicit pre-validation so that a structurally invalid `ServerStatus` response is counted as a node fault, triggering the same circuit-breaker logic as `streamBlocks()`.
3. **Optionally unify error accounting**: Consider refactoring `getBlockRange()` to call `onError()` on any exception (mirroring `streamBlocks()`), so all failure modes contribute to the node's error budget and eventual deactivation.

## Proof of Concept
1. Configure the mirror node with a single block node endpoint controlled by the attacker.
2. Have the endpoint return a `ServerStatus` gRPC response with `firstAvailableBlock = 100`, `lastAvailableBlock = 50`.
3. `getBlockRange()` calls `Range.closed(100, 50)`, which throws `IllegalArgumentException`.
4. The exception is caught; `EMPTY_BLOCK_RANGE` is returned; `onError()` is never called; `active` remains `true`.
5. `hasBlock()` returns `false`; `getNode()` throws `BlockStreamException("No block node can provide block X")`.
6. `doGet()` fails on every invocation. The node is never deactivated. Block ingestion halts permanently with no self-healing.

    private BlockNode getNode(final AtomicLong nextBlockNumber) {
        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
                inactiveNodes.add(node);
                continue;
            }

            if (hasBlock(nextBlockNumber, node)) {
                return node;
            }
        }

        // find the first inactive node with the block and force activating it
        for (final var node : inactiveNodes) {
            if (hasBlock(nextBlockNumber, node)) {
                node.tryReadmit(true);
                return node;
            }
        }

        throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
    }
