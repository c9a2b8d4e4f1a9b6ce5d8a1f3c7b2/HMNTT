## Audit Report

## Title
Cache Griefing via Unbounded `expireAfterAccess` Refresh in `findStorageBatch()` Slot Key Cache

## Summary
An unprivileged external caller can permanently fill a target contract's per-contract slot key cache with up to 1500 attacker-controlled bogus (non-existent) slot keys, then keep every entry alive indefinitely by re-querying each key before the 5-minute `expireAfterAccess` window closes. Because `findStorageBatch()` unconditionally issues a batch DB query for every key currently in the per-contract cache, every subsequent legitimate query for that contract triggers a database `IN`-clause lookup for up to 1500 rows instead of one, permanently degrading batch query performance for all legitimate callers of the targeted contract.

## Finding Description

**Exact code path:**

`ContractStateServiceImpl.java`, `findStorageBatch()`, lines 85–122.

The root cause is at line 90: any queried slot key is unconditionally inserted into the per-contract cache via `putIfAbsent` with no validation that the key actually exists in the contract's on-chain storage:

```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
```

Immediately after, line 91 snapshots the entire per-contract cache key set:

```java
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
```

And line 103 issues a single batch DB query for **all** of those keys:

```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

The underlying SQL (in `ContractStateRepository.java`, lines 23–28) is:

```sql
select slot, value from contract_state
where contract_id = :contractId
and slot in (:slots)
```

**Cache configuration (confirmed in `CacheProperties.java`):**

- `contractSlots` (`CACHE_MANAGER_CONTRACT_SLOTS`): `expireAfterAccess=5m, maximumSize=3000` — maps `contractId → CaffeineCache` (per-contract slot key cache).
- `slotsPerContract` (`CACHE_MANAGER_SLOTS_PER_CONTRACT`): `expireAfterAccess=5m, maximumSize=1500` — each per-contract cache holds up to 1500 slot keys.
- `contractState` (`CACHE_MANAGER_CONTRACT_STATE`): `expireAfterWrite=2s` — slot values expire after 2 seconds, forcing frequent re-entry into `findStorageBatch()`.

**Exploit flow:**

1. Attacker sends 1500 `eth_call`/`eth_getStorageAt` requests for contract X, each with a distinct, non-existent slot key. Each call reaches `findStorageBatch()`, inserts the bogus key via `putIfAbsent` (line 90), and fills the per-contract cache to its `maximumSize=1500` limit.
2. Because the per-contract cache is now full, any new legitimate slot key queried by a real user causes Caffeine to evict the LRU entry (a legitimate key, since the attacker refreshes their bogus keys more frequently).
3. Every ~4 minutes, the attacker re-queries each of the 1500 bogus keys (~6.25 req/s, well under the 500 req/s global throttle). Each re-query resets the `expireAfterAccess` timer for that slot key.
4. For every legitimate user query on contract X (after the 2-second `contractStateCache` expiry), `findStorageBatch()` reads all 1500 cached slot keys and issues a single batch DB query for all of them — all 1500 returning nothing.
5. If the legitimate key was evicted by the attacker's keys, `isKeyEvictedFromCache` is `true` (line 118) and the code falls back to a single-row query — but the 1500-row batch query was already issued and wasted.

**Why existing checks are insufficient:**

- `maximumSize=1500` bounds the number of slot keys per contract but does not prevent the attacker from occupying all 1500 slots with bogus keys.
- `expireAfterAccess=5m` is intended to evict stale entries, but the attacker trivially resets it before expiry at only ~6.25 req/s.
- Iterating over `getNativeCache().asMap().keySet()` (line 91) does **not** reset per-entry access timers in Caffeine — only explicit `get`/`putIfAbsent` calls do. The attacker must re-query each bogus key individually, which is trivially achievable.
- The global rate limiter (500 req/s) is not a meaningful barrier: maintaining 1500 bogus keys requires only ~6.25 req/s.
- There is no check that a queried slot key exists in the contract's on-chain storage before inserting it into the cache.

## Impact Explanation

Every legitimate query for the targeted contract (after the 2-second `contractStateCache` expiry) triggers a DB `IN`-clause query for up to 1500 rows, all of which are bogus and return nothing. This amplifies DB load by up to 1500× per legitimate query for the targeted contract. Under sustained legitimate traffic, this can cause significant DB CPU/IO pressure, increased query latency for all users of the targeted contract, and potential cascading degradation of the shared DB connection pool. The attack is persistent: the attacker can maintain it indefinitely with minimal resources.

## Likelihood Explanation

The attack requires no special privileges, no credentials, and no on-chain assets. The attacker needs only HTTP access to the mirror node's web3 API. The setup phase requires 1500 requests (a one-time cost), and the maintenance phase requires only ~6.25 req/s — well within the 500 req/s global rate limit. The attack is targeted (per-contract), stealthy (no errors are returned to the attacker or legitimate users), and persistent. Any public-facing mirror node deployment is exposed.

## Recommendation

1. **Validate before caching**: After the batch DB query at line 103, only retain in the per-contract cache slot keys that were actually returned by the DB (i.e., keys that exist in `contract_state`). Remove or do not insert keys that returned no result. This ensures the cache only contains keys with confirmed on-chain storage.
2. **Alternatively, use `expireAfterWrite` instead of `expireAfterAccess`** for `slotsPerContract`: write-based expiry cannot be reset by attacker re-queries, so bogus keys will naturally expire within the configured window regardless of access patterns.
3. **Evict non-existent keys post-query**: After `findStorageBatch()` returns, explicitly evict from `contractSlotsCache` any key that was not present in the DB result set. This limits the cache to keys with confirmed storage values.
