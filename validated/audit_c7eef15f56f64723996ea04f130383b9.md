Audit Report

## Title
NULL-Deleted Entity Invisible to EVM Address Lookup Due to Missing COALESCE in `entityFromEvmAddressQuery`

## Summary
`entityFromEvmAddressQuery` in `rest/service/entityService.js` uses `deleted <> true` without a `COALESCE` guard. In SQL three-valued logic, `NULL <> true` evaluates to `NULL` (falsy), silently excluding any entity whose `deleted` column is `NULL` from the result set. The parallel alias-lookup query in the same file was already fixed with `coalesce(deleted, false) <> true`, but the EVM-address query was not, creating a persistent inconsistency.

## Finding Description
**File:** `rest/service/entityService.js`

```js
// ALIAS query — correctly handles NULL (line 17–20)
static entityFromAliasQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where coalesce(${Entity.DELETED}, false) <> true
      and ${Entity.ALIAS} = $1`;

// EVM ADDRESS query — broken NULL handling (line 22–25)
static entityFromEvmAddressQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where ${Entity.DELETED} <> true          // NULL <> true = NULL → row excluded
      and ${Entity.EVM_ADDRESS} = $1`;
```

Migration `V1.39.1__upsert_support.sql` explicitly drops the NOT NULL constraint and the `false` default from the `deleted` column with the comment *"allow nullable on entity deleted as transaction cannot make this assumption on updates"*, confirming that `NULL` is a valid and expected production value for this column.

When `getEntityIdFromEvmAddress()` (lines 90–104) executes `entityFromEvmAddressQuery` against an entity with `deleted = NULL`, the WHERE clause evaluates to `NULL` for that row, the row is filtered out, `rows.length === 0`, and a `NotFoundError` is thrown (line 94).

## Impact Explanation
Any live entity whose `deleted` column is `NULL` — a normal, expected state after migration `V1.39.1` — is completely unreachable through the EVM-address code path. Every API endpoint that resolves an account or contract by EVM address (e.g., `GET /api/v1/accounts/{evmAddress}`, `GET /api/v1/contracts/{evmAddress}`) returns HTTP 404 for those entities. The same entity remains reachable by numeric ID or alias, creating an inconsistent API surface. Because `deleted = NULL` is a valid default for entities that have never undergone a deletion transaction, this can affect a large fraction of the entity table.

## Likelihood Explanation
No privilege is required. Any unauthenticated caller who knows an EVM address belonging to an entity with `deleted = NULL` will reliably receive a 404. The condition is not transient — it persists indefinitely unless the entity is explicitly deleted and then un-deleted (toggling the column away from NULL), which may never happen. The bug is deterministic and repeatable.

## Recommendation
Apply the same `COALESCE` guard already used in `entityFromAliasQuery`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
    from ${Entity.tableName}
    where coalesce(${Entity.DELETED}, false) <> true
      and ${Entity.EVM_ADDRESS} = $1`;
```

This mirrors the fix already applied to the alias query and correctly treats `NULL` as "not deleted".

## Proof of Concept
1. Insert an entity row into the `entity` table with a valid `evm_address` and `deleted = NULL` (the state produced by migration `V1.39.1`).
2. Call `GET /api/v1/accounts/{evmAddress}` or `GET /api/v1/contracts/{evmAddress}` using that EVM address.
3. Observe HTTP 404, because `entityFromEvmAddressQuery` evaluates `NULL <> true` → `NULL` → row excluded → `rows.length === 0` → `NotFoundError` thrown.
4. Call the same endpoint using the entity's numeric ID — observe HTTP 200, confirming the entity exists and is live.
5. Swap `deleted <> true` for `coalesce(deleted, false) <> true` in `entityFromEvmAddressQuery` and repeat step 2 — observe HTTP 200.
