## Audit Report

## Title
Unbounded Timestamp Range in Historical Hook Storage Query Enables Database Resource Exhaustion

## Summary
An unauthenticated caller can supply a single open-ended `timestamp` parameter (e.g., `timestamp=gte:0`) to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, causing the server to execute a `DISTINCT ON (key)` query over the entire `hook_storage_change` partition for the given `(owner_id, hook_id)` pair with no server-side range-width guard. The operation is repeatable with no credentials required.

## Finding Description

**Controller layer — no range-width validation:**

`HooksController.getHookStorage()` accepts `timestamps` constrained only by `@Size(max = 2)` — no minimum range width, no requirement for both bounds to be present. [1](#0-0) 

`hookStorageChangeRequest()` passes the raw array directly to `Bound.of()` with no further validation. [2](#0-1) 

**Bound.java — open-ended defaults:**

`adjustUpperBound()` returns `Long.MAX_VALUE` when `upper == null`. [3](#0-2) 

`getAdjustedLowerRangeValue()` returns `0` when `lower == null`. [4](#0-3) 

So `timestamp=gte:0` produces `timestampLowerBound = 1`, `timestampUpperBound = Long.MAX_VALUE`; `timestamp=lte:99999999999` produces `timestampLowerBound = 0`, `timestampUpperBound = 99999999999`.

**Service layer — full key-space fallback:**

When no `key` filters are provided, `getHookStorageChange()` calls `findByKeyBetweenAndTimestampBetween()` with `MIN_KEY_BYTES` (32×`0x00`) and `MAX_KEY_BYTES` (32×`0xFF`) as key bounds. [5](#0-4) 

**Repository layer — expensive SQL:**

The resulting query is:
```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id  = :hookId
  AND key >= '\x0000...00'   -- matches all keys
  AND key <= '\xffff...ff'   -- matches all keys
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
``` [6](#0-5) 

**Index mismatch — no efficient plan possible:**

The table's only index is the primary key `(owner_id, hook_id, key, consensus_timestamp ASC)`. [7](#0-6) 

The `Pageable` appended by `getPageRequest()` orders by `(key ASC/DESC, consensus_timestamp DESC)`. [8](#0-7) 

Because the primary key stores `consensus_timestamp ASC` but the query requires `consensus_timestamp DESC` within each key group for `DISTINCT ON` to return the most-recent value, PostgreSQL cannot satisfy the sort from the index and must perform a full partition scan + in-memory sort of every row for the given `(owner_id, hook_id)` before applying `LIMIT`.

In the V2 schema the table is range-partitioned by `consensus_timestamp`, so an unbounded range forces cross-partition scans. [9](#0-8) 

**No authentication anywhere in the codebase:**

A search for `@PreAuthorize`, `SecurityConfig`, `permitAll`, and `authenticated` across all Java sources returns zero matches. The controller carries no security annotation. [10](#0-9) 

## Impact Explanation

For any hook with a large history of storage changes, each request with an open-ended timestamp forces PostgreSQL to scan and sort the entire `(owner_id, hook_id)` partition before returning up to `LIMIT` rows. Because the endpoint requires no credentials, any external actor can issue these requests. Repeated or concurrent requests against a high-activity hook can saturate database I/O and CPU, degrading query performance for all users of the mirror node.

## Likelihood Explanation

The attack requires no credentials. Both `ownerId` and `hookId` are publicly discoverable via `GET /api/v1/accounts/{ownerId}/hooks`. The exploit is a single HTTP request with one query parameter (`timestamp=gte:0`). No rate limiting is present in the controller or visible anywhere in the codebase. The attack is trivially scriptable.

## Recommendation

1. **Enforce a maximum timestamp range width** server-side (e.g., reject requests where `upper - lower > MAX_ALLOWED_RANGE`).
2. **Require both bounds** when a timestamp filter is present for the historical path, or default the upper bound to the current consensus timestamp rather than `Long.MAX_VALUE`.
3. **Add a covering index** ordered `(owner_id, hook_id, key, consensus_timestamp DESC)` so `DISTINCT ON (key)` can be resolved via an index scan rather than a full sort.
4. **Add authentication/authorization** to the hook storage endpoint if the data is not intended to be fully public.
5. **Add rate limiting** at the API gateway or controller level for historical query endpoints.

## Proof of Concept

```
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0
```

This single unauthenticated request sets `timestampLowerBound = 1` and `timestampUpperBound = Long.MAX_VALUE` with key bounds spanning the full 32-byte space, forcing PostgreSQL to evaluate `DISTINCT ON (key)` over every row in `hook_storage_change` for `owner_id=123, hook_id=1`. Repeating this request concurrently against multiple hooks exhausts database resources.
