Audit Report

## Title
Integer Narrowing Cast Overflow in `PollingTopicListener.poll()` Bypasses `maxPageSize` Cap, Enabling Unlimited DB Queries

## Summary
In `PollingTopicListener.poll()`, a narrowing cast `(int)(filter.getLimit() - context.getCount().get())` overflows when `limit = Long.MAX_VALUE`, producing `-1`. `Math.min(-1, maxPageSize)` returns `-1`, and the resulting filter with `limit = -1` causes `hasLimit()` to return `false`, so `setMaxResults()` is never called and the database returns all matching rows on every poll cycle.

## Finding Description

**`PollingTopicListener.java`, lines 54–57** — the narrowing cast and the broken cap:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())  // overflow
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // -1 wins
``` [1](#0-0) 

`filter.getLimit()` and `context.getCount().get()` are both `long`. The subtraction is a `long` operation, but the result is immediately narrowed to `int`. For `Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF`, the lower 32 bits are `0xFFFFFFFF`, so `(int) Long.MAX_VALUE = -1`. `Math.min(-1, maxPageSize)` returns `-1` regardless of `maxPageSize`. [2](#0-1) 

**`TopicMessageFilter.java`, lines 39–41** — `hasLimit()` returns `false` for `-1`:

```java
public boolean hasLimit() {
    return limit > 0;
}
``` [3](#0-2) 

`-1 > 0` is `false`, so the filter built with `limit = -1` is treated as "no limit."

**`TopicMessageRepositoryCustomImpl.java`, lines 51–53** — `setMaxResults()` is never called:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [4](#0-3) 

The JPA query has no row limit; `getResultList()` materialises every matching row. [5](#0-4) 

**Why `@Min(0)` does not block the input:**

```java
@Min(0)
private long limit;
``` [6](#0-5) 

`Long.MAX_VALUE > 0`, so it passes validation. The overflow occurs *after* validation, inside `poll()`.

**Why the Lombok builder does not re-validate:**

`filter.toBuilder().limit(pageSize).startTime(startTime).build()` constructs a new `TopicMessageFilter` with `limit = -1`. Lombok's builder does not invoke Bean Validation; `@Min(0)` is never re-checked on the internally-constructed filter. [7](#0-6) 

**Call chain from user input to unlimited query:**

1. User sends gRPC `subscribeTopic` with `limit = Long.MAX_VALUE` — passes `@Min(0)`.
2. `TopicMessageServiceImpl.incomingMessages()` computes `limit = Long.MAX_VALUE - 0` as a `long` and calls `topicListener.listen(newFilter)`. [8](#0-7) 
3. `PollingTopicListener.listen()` schedules repeated calls to `poll()` every 500 ms. [9](#0-8) 
4. `poll()` computes `(int)(Long.MAX_VALUE - 0) = -1`; `pageSize = -1`; `findByFilter()` executes a full table scan per poll cycle.

The overflow persists for an astronomically large count range: the cast only produces a positive value once `count` exceeds approximately `Long.MAX_VALUE - Integer.MAX_VALUE ≈ 9.2 × 10¹⁸`, which is never reached in practice.

## Impact Explanation

Every 500 ms, the attacker's subscription triggers a full, unbounded `SELECT` against the `topic_message` table for the chosen topic. On a production node with millions of messages, each query can return gigabytes of data, consuming database CPU and I/O, JPA/Hibernate heap (entire result list materialised via `getResultList()`), and network bandwidth between DB and mirror node. Multiple concurrent subscriptions multiply the effect linearly. Because the subscription persists indefinitely (`RepeatSpec.times(Long.MAX_VALUE)`), a single connection sustains the attack without re-authentication.

## Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication requirement beyond a valid topic ID (public on-chain data). The attacker needs only to set the `limit` field in a `ConsensusTopicQuery` protobuf message to `Long.MAX_VALUE`. The exploit is trivially reproducible with any gRPC client, requires no elevated privileges, no special knowledge of internals, and no race condition.

## Recommendation

**Primary fix — add an upper-bound check before the cast in `PollingTopicListener.poll()`:**

```java
long remaining = filter.getLimit() - context.getCount().get();
int limit = (remaining > Integer.MAX_VALUE || remaining < 0)
        ? Integer.MAX_VALUE
        : (int) remaining;
```

Or use `Math.toIntExact` with a try/catch, or clamp with `(int) Math.min(remaining, Integer.MAX_VALUE)`.

**Secondary fix — add `@Max` to `TopicMessageFilter.limit`:**

```java
@Min(0)
@Max(Long.MAX_VALUE / 2)  // or a domain-appropriate cap
private long limit;
``` [6](#0-5) 

**Tertiary fix — validate the internally-built filter** by invoking Bean Validation explicitly in the builder, or by adding a defensive check in `findByFilter()`:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
} else if (filter.getLimit() < 0) {
    throw new IllegalArgumentException("Negative limit: " + filter.getLimit());
}
``` [4](#0-3) 

## Proof of Concept

```java
// Pseudocode using any gRPC client
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setLimit(Long.MAX_VALUE)   // passes @Min(0), triggers overflow in poll()
    .build();

stub.subscribeTopic(query, responseObserver);
// Every 500ms, a full table scan executes against topic_message with no LIMIT clause.
```

Verification: attach a query logger to the JPA layer and observe that `SELECT ... FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` is issued without any `LIMIT` clause on every poll interval.
