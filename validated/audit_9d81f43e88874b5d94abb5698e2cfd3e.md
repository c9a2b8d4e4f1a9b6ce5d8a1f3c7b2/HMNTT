Audit Report

## Title
Gas Token Drain via REJECT Filter: `gasLimitBucket` Consumed Without Restoration on `ThrottleException`

## Summary
In `ThrottleManagerImpl.throttle()`, `gasLimitBucket` tokens are consumed before the request filter loop is evaluated. When a `REJECT` filter matches, a `ThrottleException` is thrown, but `ContractController.call()` only restores gas tokens on `InvalidParametersException` — never on `ThrottleException`. This allows an attacker who can match a configured `REJECT` filter to permanently drain the shared `gasLimitBucket` without any EVM execution occurring.

## Finding Description

**`ThrottleManagerImpl.throttle()`** — gas tokens are consumed at line 40, unconditionally before the filter loop: [1](#0-0) 

**`action()`** — the `REJECT` case throws `ThrottleException` at line 69, after gas has already been consumed: [2](#0-1) 

**`ContractController.call()`** — `restore()` is only called in the `catch (InvalidParametersException e)` block. `ThrottleException` is never caught here, so `restore()` is never invoked when a `REJECT` filter fires: [3](#0-2) 

The `restore()` method itself is correctly implemented and does add tokens back to `gasLimitBucket`: [4](#0-3) 

**Root cause:** The developer correctly applied the restore pattern for `InvalidParametersException` (with the comment "no processing occurred so restore the consumed tokens"), but failed to apply the same logic to `ThrottleException` thrown by `REJECT` filters. A `REJECT`-filtered request also involves no EVM processing, yet its gas cost is permanently deducted from the shared bucket.

## Impact Explanation

The `gasLimitBucket` is a shared global resource. Draining it causes all concurrent `/api/v1/contracts/call` requests to receive `GAS_PER_SECOND_LIMIT_EXCEEDED` errors. No EVM execution is required by the attacker — each rejected request costs only an HTTP round-trip. The default configuration confirms the math:

- `gasPerSecond` = 7,500,000,000 → scaled to 750,000 tokens/s [5](#0-4) 
- `requestsPerSecond` = 500 req/s [6](#0-5) 
- `scaleGas(15_000_000)` = `floor(15_000_000 / 10_000)` = 1,500 tokens per request [7](#0-6) 

At 500 req/s × 1,500 tokens = 750,000 tokens/s, the attacker can sustain a complete drain of the gas bucket indefinitely, blocking all legitimate users.

## Likelihood Explanation

The precondition is that at least one `REJECT` filter is configured — an explicitly documented, production-intended operator feature (`hiero.mirror.web3.throttle.request[].action = REJECT`). Any operator using `REJECT` filters to block abusive patterns inadvertently enables this attack. The attacker requires no credentials or privileged access. The filter pattern may be discoverable by probing: a matched request returns `ThrottleException("Invalid request")` which is distinguishable from normal validation errors. The attack is fully automatable with a simple HTTP client.

## Recommendation

Apply the same `restore()` pattern to `ThrottleException` when it originates from a `REJECT` filter. The simplest fix is to catch `ThrottleException` in `ContractController.call()` and call `throttleManager.restore(request.getGas())` before re-throwing, mirroring the existing `InvalidParametersException` handler:

```java
} catch (ThrottleException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
```

Alternatively, move the `gasLimitBucket.tryConsume()` call to after the filter loop in `ThrottleManagerImpl.throttle()`, so gas is only consumed for requests that pass all filters.

## Proof of Concept

1. Operator has configured a `REJECT` filter matching, e.g., a specific `to` address `0xdeadbeef...`.
2. Attacker sends `POST /api/v1/contracts/call` with `gas=15000000` and `to=0xdeadbeef...` in a tight loop.
3. Each request: `gasLimitBucket.tryConsume(1500)` succeeds → REJECT filter matches → `ThrottleException("Invalid request")` thrown → `restore()` never called → 1500 tokens permanently lost.
4. At 500 req/s, the 750,000-token bucket is fully drained every second.
5. All legitimate users receive `"Gas per second rate limit exceeded."` with no EVM execution ever occurring.
6. The attacker sustains this indefinitely; the bucket refills at exactly the rate the attacker drains it.
