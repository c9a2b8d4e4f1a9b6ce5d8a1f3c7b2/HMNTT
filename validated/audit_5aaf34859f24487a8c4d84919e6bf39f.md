Audit Report

## Title
Duplicate `TransferTransaction` Execution via `GetReceipt()` Failure in `submitWithRetry()`

## Summary
`submitWithRetry()` in `pinger/transfer.go` constructs a new `TransferTransaction` (with a new `TransactionID`) on every retry iteration. If `Execute()` succeeds but the subsequent `GetReceipt()` call fails, the function treats the attempt as failed and retries with a fresh transaction. Because Hedera deduplicates only by `TransactionID`, each retry is an independent on-chain transfer, causing the operator account to be debited once per retry.

## Finding Description
In `pinger/transfer.go`, `submitWithRetry()` loops up to `cfg.maxRetries + 1` times. On every iteration, a brand-new `TransferTransaction` is constructed: [1](#0-0) 

`Execute(client)` submits the transaction to the Hedera consensus network. A `nil` error from `Execute` means the transaction has been accepted and finalized on-chain — the debit has already occurred: [2](#0-1) 

`GetReceipt(client)` is a **separate, subsequent** network call. If it fails (timeout, TCP reset, packet drop), `rerr` is non-nil and is assigned to `err`: [3](#0-2) 

The loop then continues. On the next iteration, `hiero.NewTransferTransaction()` produces a transaction with a **new** `TransactionID` (new `validStart` timestamp auto-assigned by the SDK). Hedera's replay-protection deduplicates only within the same `TransactionID`; a new ID bypasses it entirely. The second `Execute(client)` call submits and executes a completely independent transfer.

**Root cause**: The failed assumption is that a `GetReceipt()` error implies the transaction did not execute. In reality, `Execute()` success and `GetReceipt()` success are two independent network round-trips. The code conflates receipt-query failure with transaction failure.

## Impact Explanation
Each successful `Execute()` followed by a dropped `GetReceipt()` response causes one additional debit of `cfg.amountTinybar` from the operator account. The default `maxRetries` is **10**: [4](#0-3) 

This means up to **11** independent transfers can execute in a single pinger interval, multiplying the intended transfer amount by up to 11. Since the pinger runs continuously on a ticker: [5](#0-4) 

Sustained interference across multiple intervals can progressively drain the operator account. The impact is direct, repeatable financial loss from the operator account with no recovery mechanism in the code.

## Likelihood Explanation
The vulnerability can be triggered in two ways:

1. **Without any attacker**: Natural packet loss (>5% on a flaky link) can cause `GetReceipt()` to time out while `Execute()` already succeeded, triggering unintentional duplicate debits. No adversary is required.

2. **With a network-path attacker**: An attacker who can selectively drop TCP segments carrying the receipt response (e.g., via a cloud NAT gateway, shared network segment, or BGP-hijacked route) while allowing the submission to pass can reliably trigger this on every pinger interval. No privileged access to the pinger process, its keys, or the Hedera network is required.

## Recommendation
Once `Execute()` returns successfully, the code must not retry with a new transaction. The fix is to break out of the retry loop immediately after a successful `Execute()`, regardless of whether `GetReceipt()` fails. If receipt confirmation is needed, retry `GetReceipt()` on the **same** `resp` object (same `TransactionID`) rather than constructing a new transaction:

```go
resp, err := cryptoTransfer.Execute(client)
if err != nil {
    lastErr = err
    // safe to retry with a new transaction — Execute failed, nothing was submitted
    continue
}
// Execute succeeded — do NOT retry with a new transaction from here on
receipt, rerr := resp.GetReceipt(client)
if rerr == nil {
    log.Printf(...)
    return nil
}
// Optionally retry GetReceipt on the same resp, but never re-Execute
return fmt.Errorf("transaction submitted (txID=%s) but receipt unavailable: %w",
    resp.TransactionID, rerr)
```

Additionally, consider reducing `maxRetries` to a small value (e.g., 2–3) and adding an explicit guard that exits the retry loop after any successful `Execute()`.

## Proof of Concept
1. Deploy the pinger with default config (`maxRetries=10`, `amountTinybar=10000`).
2. Intercept the network path between the pinger and the Hedera mirror/consensus node (or simulate packet loss with `tc netem loss 100%` on the receipt-response port only).
3. Observe that `Execute()` succeeds on the first attempt (transaction appears on-chain), but `GetReceipt()` times out.
4. The loop continues; on iteration 2, a new `TransferTransaction` with a new `TransactionID` is submitted and executed.
5. Repeat for all 11 attempts.
6. Query the operator account balance on-chain: it has been debited `10000 * 11 = 110000` tinybar instead of the intended `10000`. [6](#0-5)
