# Audit Report

## Title
Integer Overflow in `poll()` Removes DB Query Limit, Enabling Unlimited Result Set DoS

## Summary
In `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` performs an unchecked narrowing cast from `long` to `int`. When a caller supplies `limit = Long.MAX_VALUE` — permitted by the `@Min(0)`-only constraint on `TopicMessageFilter.limit` — the cast produces `-1`. This propagates as `limit = -1L` into an internally-built `newFilter`, causing `hasLimit()` to return `false`, which in turn causes `TopicMessageRepositoryCustomImpl.findByFilter` to omit `setMaxResults()` entirely, executing an unbounded full-table scan of `topic_message`.

## Finding Description

**Entry point — `ConsensusController.toFilter()`:**

`query.getLimit()` (a proto `uint64`, so any 64-bit value is valid) is passed directly to `TopicMessageFilter.builder().limit(...)` with no cap. [1](#0-0) 

**Constraint gap — `TopicMessageFilter.limit`:**

Only `@Min(0)` is present; there is no `@Max`. `Long.MAX_VALUE` passes validation and `hasLimit()` (`limit > 0`) returns `true`. [2](#0-1) 

**Overflow — `PollingTopicMessageRetriever.poll()` lines 68–75:**

```
(int)(Long.MAX_VALUE - 0L)
= (int)(0x7FFFFFFFFFFFFFFF)
= 0xFFFFFFFF (lower 32 bits)
= -1 (signed int)
```

`Math.min(-1, maxPageSize)` = `-1` for any positive `maxPageSize`. The builder call `filter.toBuilder().limit(pageSize)` widens `-1` back to `long`, producing `newFilter.limit = -1L`. `hasLimit()` on `newFilter` evaluates `-1 > 0` → `false`. [3](#0-2) 

**Identical bug — `PollingTopicListener.poll()` lines 54–59:** [4](#0-3) 

**`setMaxResults()` skipped — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51–53:**

Because `newFilter.hasLimit()` is `false`, `typedQuery.setMaxResults(...)` is never called. The JPA query executes with no `LIMIT` clause and materialises the entire matching result set into a `List` via `getResultList()`. [5](#0-4) 

**Loop termination failure — `PollingContext.isComplete()` lines 121–128:**

`limitHit` compares `filter.getLimit() == total.get()`, i.e. `Long.MAX_VALUE == total.get()`. This is never true in practice. In throttled mode the loop continues until the returned page is smaller than `maxPageSize`; since the query is now unlimited, a topic with ≥ `maxPageSize` rows causes the loop to repeat indefinitely. [6](#0-5) 

**Retry amplification:**

Any transient failure (e.g. OOM) is automatically retried up to `Long.MAX_VALUE` times with exponential backoff. [7](#0-6) 

## Impact Explanation

Each malicious subscription triggers repeated full-table scans of `topic_message` for the targeted topic with no `LIMIT` clause. The JPA layer materialises the entire result into a heap-resident `List`. A high-volume topic (e.g. fee-schedule topic 0.0.101) can have millions of rows. Multiple concurrent subscriptions can exhaust JVM heap, cause GC pressure, and crash or severely degrade the mirror-node gRPC service — a denial-of-service against all legitimate subscribers. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` wrapper means any OOM-induced error is automatically retried, amplifying load.

## Likelihood Explanation

The `subscribeTopic` gRPC endpoint is unauthenticated and publicly reachable. The proto `limit` field is `uint64`, so any gRPC client can set it to `2^63-1` (`Long.MAX_VALUE`). No rate-limiting or per-connection limit cap exists in the controller or service layer. A single attacker with `grpcurl` or any gRPC client can trigger this repeatedly. The attack is trivially scriptable. [8](#0-7) 

## Recommendation

1. **Cap the limit at the filter level.** Add `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) to `TopicMessageFilter.limit` so values that would overflow `int` are rejected at validation time.
2. **Use safe narrowing.** Replace the bare `(int)` cast with `Math.toIntExact(...)` or `(int) Math.min(longValue, Integer.MAX_VALUE)` so overflow is either detected or clamped.
3. **Validate in the controller.** In `ConsensusController.toFilter()`, clamp `query.getLimit()` to a configured maximum before building the filter.
4. **Add a negative-limit guard in `findByFilter`.** Treat any `limit <= 0` as "no limit requested" or throw, rather than silently omitting `setMaxResults`.

## Proof of Concept

```bash
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 101},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Trace through the code:**
1. `ConsensusController.toFilter()` sets `filter.limit = 9223372036854775807L` (Long.MAX_VALUE). [9](#0-8) 
2. `filter.hasLimit()` → `true`; cast: `(int)(Long.MAX_VALUE - 0) = -1`. [10](#0-9) 
3. `pageSize = Math.min(-1, maxPageSize) = -1`; `newFilter.limit = -1L`. [11](#0-10) 
4. `newFilter.hasLimit()` → `false`; `setMaxResults()` is never called. [12](#0-11) 
5. `getResultList()` returns every row for topic 0.0.101 into heap memory. [13](#0-12) 
6. `isComplete()` never returns `true`; the poll loop repeats, re-fetching all rows on each iteration. [6](#0-5)
