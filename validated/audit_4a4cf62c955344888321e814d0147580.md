# Audit Report

## Title
Missing Timestamp Bound in `getLatestFileDataContents` Returns Current File Data Instead of Historical Call Data

## Summary
`getContractResultsByTimestamp()` and `getContractResultsByTransactionIdOrHash()` both call `FileDataService.getLatestFileDataContents()` with an empty `whereQuery`, causing the underlying SQL CTE to return the **most recent** file state unconditionally. The consensus timestamp already available in scope is never used as an upper bound, so the `callData` field in historical contract result responses reflects current file contents rather than what was submitted at transaction time.

## Finding Description

In `rest/controllers/contractController.js`, `getContractResultsByTimestamp()` extracts the request timestamp at line 1010 but does not pass it to the file lookup at line 1023:

```js
// line 1010 — timestamp IS in scope
const {contractId, timestamp} = await getAndValidateContractIdAndConsensusTimestampPathParams(req);
...
// line 1023 — empty whereQuery, no upper bound
fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
``` [1](#0-0) 

The same defect is present in `getContractResultsByTransactionIdOrHash()` at line 1184, where `transactionDetails.consensusTimestamp` is available but also not forwarded: [5](#0-4) 

**Root cause — `latestFileContentsQuery` in `rest/service/fileDataService.js` lines 19–42:**

The CTE selects `max(consensus_timestamp)` for create/update operations. The `filterInnerPlaceholder` and `filterOuterPlaceholder` tokens are replaced with empty strings when `whereQuery` is `[]`, so the query has no timestamp ceiling and unconditionally returns the latest file state: [2](#0-1) 

The correct bounded alternative, `getFileDataQuery`, already exists and uses `<= $2` to cap results at a given timestamp: [3](#0-2) 

The timestamp-bounding capability of `getLatestFileDataContents` itself is also confirmed by the existing integration test at lines 106–118, which passes a `<= timestamp` filter and receives the correct historical state: [4](#0-3) 

The fix is not implemented — the available `timestamp` / `consensusTimestamp` is simply never wired into the `whereQuery`.

## Impact Explanation
`GET /api/v1/contracts/{contractId}/results/{timestamp}` is a historical audit endpoint whose purpose is to faithfully reproduce what occurred at a specific consensus timestamp. The `callData` field is supposed to reflect the exact bytes submitted in the Ethereum transaction. Because the file lookup has no timestamp ceiling, any file update after the original transaction causes every subsequent query to that historical endpoint to return post-update file contents as `callData`. The API thereby asserts that a transaction contained call data it never actually contained, breaking the integrity of the historical record for any downstream auditor, verifier, or system reconstructing transaction inputs.

## Likelihood Explanation
No authentication or special privileges are required — the endpoint is publicly accessible via a standard GET request. The precondition for a meaningful discrepancy (the referenced file being updated after the transaction) is a routine operational event on the Hedera network. Once a file update occurs, the incorrect behavior is deterministic and repeatable on every subsequent call to the affected endpoint.

## Recommendation
Pass the already-available consensus timestamp as a `whereQuery` filter in both call sites:

```js
// getContractResultsByTimestamp — line 1023
fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {
  whereQuery: [{query: `${FileData.CONSENSUS_TIMESTAMP} <= `, param: timestamp}]
});

// getContractResultsByTransactionIdOrHash — line 1184
fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {
  whereQuery: [{query: `${FileData.CONSENSUS_TIMESTAMP} <= `, param: ethTransaction.consensusTimestamp}]
});
```

This mirrors the pattern already demonstrated in the integration test and the existing `getFileDataQuery` implementation.

## Proof of Concept

1. Submit an Ethereum transaction whose `callData` is stored in a Hedera file (e.g., file `0.0.X`) at consensus timestamp `T1`.
2. Query `GET /api/v1/contracts/{contractId}/results/T1` — observe `callData` = original file contents (correct).
3. Submit a file update to `0.0.X` at timestamp `T2 > T1`, changing its contents.
4. Re-query `GET /api/v1/contracts/{contractId}/results/T1` — observe `callData` = **updated** file contents (incorrect). The response now misrepresents the historical transaction.

The same steps apply to `GET /api/v1/contracts/results/{transactionId}` via `getContractResultsByTransactionIdOrHash`.
