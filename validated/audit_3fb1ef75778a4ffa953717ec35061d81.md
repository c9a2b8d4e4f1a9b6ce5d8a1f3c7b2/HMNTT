## Audit Report

## Title
Cache-Eviction Griefing via Batch Amplification in `findStorageBatch()` â€” Unbounded Per-User Writes to Shared `contractStateCache`

## Summary
`ContractStateServiceImpl.findStorageBatch()` unconditionally writes all per-contract cached slot keys back into the global `contractStateCache` (`maximumSize=25000`, `expireAfterWrite=2s`) on every cache miss, with no per-caller quota on cache entries. An attacker who pre-populates the per-contract `slotsPerContract` caches (one per contract, `maximumSize=1500`) for ~17 contracts can trigger batch writes of up to 1500 entries per request into the shared cache, filling all 25,000 slots with only 17 requests per 2-second TTL window â€” well under the 500 req/s global rate limit â€” continuously evicting legitimate users' cached slot values and forcing every subsequent legitimate `findStorage()` call to re-execute batch DB queries.

## Finding Description

**Exact code path (all references verified against source):**

`ContractStateServiceImpl.findStorage()` checks `contractStateCache` at line 63. On a miss (or `EMPTY_VALUE`), it calls `findStorageBatch()` at line 69. [1](#0-0) 

Inside `findStorageBatch()` (lines 85â€“122):

- **Line 90**: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` â€” unconditionally registers the new key in the per-contract slots cache.
- **Line 91**: `contractSlotsCache.getNativeCache().asMap().keySet()` â€” reads **all** currently cached slot keys for this contract.
- **Line 103**: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` â€” batch-queries **all** of them from the DB.
- **Line 109**: `contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue)` â€” writes **all** results into the global shared `contractStateCache`. [2](#0-1) 

**Cache configuration (verified):**

| Cache | Config |
|---|---|
| `contractState` (global shared) | `expireAfterWrite=2s, maximumSize=25000` |
| `slotsPerContract` (per-contract amplifier) | `expireAfterAccess=5m, maximumSize=1500` |
| `contractSlots` (outer map of per-contract caches) | `expireAfterAccess=5m, maximumSize=3000` | [3](#0-2) 

**Root cause:** There is no bound on how many entries a single caller can inject into `contractStateCache` per request. Each call to `findStorageBatch()` for a contract with N slots in its per-contract `slotsPerContract` cache writes N entries into the global shared cache. The `slotsPerContract` cache (`maximumSize=1500`) acts as an amplifier: one HTTP request â†’ up to 1500 `contractStateCache.put()` calls.

**Exploit flow:**

1. **Setup phase** â€” Attacker sends 1500 requests for each of 17 distinct contracts (25,500 total), each with a different existing slot key. This populates the `slotsPerContract` cache for each contract to its maximum of 1500 entries. The `contractSlots` cache (`maximumSize=3000`) holds the per-contract cache references. Slot keys are publicly enumerable from the mirror node's own REST API.

2. **Attack phase** â€” After the 2-second `contractStateCache` TTL expires, the attacker sends exactly 17 requests (one per pre-loaded contract). Each request triggers `findStorageBatch()`, which reads all 1500 cached slot keys and writes up to 1500 entries into `contractStateCache`. After 17 requests, all 25,000 slots in `contractStateCache` are filled with attacker-controlled entries (17 Ă— 1500 = 25,500 > 25,000).

3. **Sustain phase** â€” The attacker repeats step 2 every 2 seconds. Only 17 requests per 2-second window (8.5 req/s) are needed to maintain full cache saturation.

**Why existing checks fail:**

- **Rate limiting** (`ThrottleManagerImpl`, `rateLimitBucket`, 500 req/s global): The maintenance phase requires only 17 requests per 2 seconds (8.5 req/s), well under the 500 req/s global limit. The rate limit is a single shared bucket, not per-IP. [4](#0-3) 

- **`maximumSize=25000`**: This is the attack surface, not a defense. Caffeine's W-TinyLFU eviction policy evicts the least-recently-used entries when the cache is full, which are legitimate users' entries written before the attacker's flood.

- **`expireAfterWrite=2s`**: The attacker re-floods every 2 seconds, keeping the cache perpetually saturated with attacker entries.

- **No per-user/per-IP cache quota**: `contractStateCache` is a single global `CaffeineCache` instance with no partitioning by caller identity. [5](#0-4) 

## Impact Explanation

Every legitimate `findStorage()` call that misses the cache (because its entry was evicted) falls through to `findStorageBatch()`, which issues a DB query (`findStorageBatch` or `findStorage` on `ContractStateRepository`). Under sustained attack, all 25,000 cache slots are occupied by attacker entries, so every legitimate request incurs a DB round-trip. This increases database load proportionally to legitimate traffic, degrades response latency for all `eth_call`/`eth_estimateGas` users, and can cascade into DB connection pool exhaustion under high legitimate load. No funds are at risk; the impact is availability degradation (griefing/DoS).

## Likelihood Explanation

The attack requires no credentials, no on-chain transactions, and no special tooling â€” only the ability to send HTTP requests to the public web3 API. Existing slot keys for popular contracts are trivially enumerable from the mirror node's own REST API. The maintenance cost (17 req/2s) is negligible and well below the global rate limit. The `slotsPerContract` cache uses `expireAfterAccess=5m`, so the attacker only needs to refresh each slot key once every 5 minutes to keep the amplifier loaded. The attack is fully repeatable and can be sustained indefinitely from a single IP without triggering the global rate limit.

## Recommendation

1. **Limit batch write size per request**: Cap the number of entries written to `contractStateCache` in a single `findStorageBatch()` invocation (e.g., limit `cachedSlots` to a configurable maximum, such as 50â€“100 entries per call).

2. **Partition `contractStateCache` by contract**: Instead of a single flat global cache, use per-contract sub-caches (similar to how `slotsPerContract` is structured), so one contract's entries cannot evict another contract's entries.

3. **Add per-IP or per-contract rate limiting on cache writes**: Track the number of `contractStateCache.put()` calls per source IP or per contract per time window and reject or throttle requests that exceed the threshold.

4. **Reduce `slotsPerContract` maximumSize**: Lowering the per-contract slot key cache size reduces the amplification factor per request.

5. **Consider disabling `enableBatchContractSlotCaching`** as a short-term mitigation if the attack is observed in production, falling back to single-slot `findStorage()` queries. [6](#0-5)

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
