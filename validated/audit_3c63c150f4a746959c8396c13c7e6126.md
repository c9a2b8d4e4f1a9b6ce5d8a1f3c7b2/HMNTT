# Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses Query Limit, Enabling Unbounded Database Reads

## Summary
In `PollingTopicListener.poll()`, the remaining limit is computed as `(int)(filter.getLimit() - context.getCount().get())`. A user-supplied `limit` of `2147483648L` (one above `Integer.MAX_VALUE`) causes this narrowing cast to produce `Integer.MIN_VALUE`. Because `TopicMessageFilter.hasLimit()` returns `false` for any negative value, `TopicMessageRepositoryCustomImpl.findByFilter()` never calls `setMaxResults()`, executing an unbounded SQL query on every poll cycle. The service-level guard uses the original large limit, allowing ~2 billion messages through.

## Finding Description

**Step 1 ‚Äî Overflow in `PollingTopicListener.poll()`**

```java
// PollingTopicListener.java lines 54‚Äì57
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
``` [1](#0-0) 

With `filter.getLimit() = 2147483648L` and `context.getCount().get() = 0`:
- `(int)(2147483648L) = Integer.MIN_VALUE = -2147483648`
- `Math.min(Integer.MIN_VALUE, maxPageSize) = Integer.MIN_VALUE`
- `newFilter` is built with `limit = -2147483648L` (int widened to long)

**Step 2 ‚Äî Derived filter bypasses Spring validation**

```java
// PollingTopicListener.java line 59
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [7](#0-6) 

This is a direct Lombok builder call, not a Spring-proxied method invocation. The `@Min(0)` constraint on `limit` is never re-evaluated, so `-2147483648L` is accepted silently.

**Step 3 ‚Äî `hasLimit()` returns false, `setMaxResults()` skipped**

```java
// TopicMessageFilter.java lines 39‚Äì41
public boolean hasLimit() {
    return limit > 0;
}
``` [2](#0-1) 

```java
// TopicMessageRepositoryCustomImpl.java lines 51‚Äì53
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [3](#0-2) 

`-2147483648L > 0` is false ‚Üí `setMaxResults()` is never called ‚Üí the JPA query executes with no SQL `LIMIT`, materializing the entire result set via `getResultList()`. [8](#0-7) 

**Step 4 ‚Äî Service-level guard is ineffective**

```java
// TopicMessageServiceImpl.java lines 83‚Äì85
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
``` [4](#0-3) 

This uses the **original** filter where `limit = 2147483648L`. `hasLimit()` returns true, so `flux.take(2147483648L)` is applied ‚Äî effectively no bound in practice.

**Root cause:** No `@Max` upper-bound constraint on `TopicMessageFilter.limit`, combined with a silent narrowing `(int)` cast that wraps large positive longs to negative ints, which then defeats the `hasLimit()` guard downstream. [5](#0-4) 

## Impact Explanation
Every poll cycle (default every 500 ms) issues a full table scan against `topic_message` for the targeted topic with no row limit. `getResultList()` materializes the entire result set in JVM heap before streaming. For a high-volume topic with years of history this can:

1. **Exhaust database I/O and CPU** ‚Äî repeated unbounded scans degrade service for all users (DoS).
2. **Expose complete message history** ‚Äî all records for the targeted topic are streamed to the attacker, up to ~2 billion messages per subscription.
3. **Exhaust JVM heap** ‚Äî full result set materialization per poll cycle can trigger OOM errors.

Severity: **High** (DoS + information disclosure).

## Likelihood Explanation
- Any gRPC client can call `subscribeTopic` with an arbitrary `limit` value; no authentication is required beyond a valid gRPC connection.
- The trigger value (`2147483648L = Integer.MAX_VALUE + 1`) is trivially derivable from basic knowledge of Java integer overflow.
- The attack is repeatable: each new subscription with the crafted limit re-triggers the unbounded query on every poll interval.
- Requires `listenerProperties.type = POLL` (non-default; default is `REDIS`), which reduces the attack surface but does not eliminate it ‚Äî `PollingTopicListener` is a first-class supported deployment mode.

## Recommendation
1. **Add an `@Max` constraint** on `TopicMessageFilter.limit` capping it at `Integer.MAX_VALUE` (or a lower operational maximum):
   ```java
   @Min(0)
   @Max(Integer.MAX_VALUE)
   private long limit;
   ```
2. **Replace the unchecked `(int)` cast** in `PollingTopicListener.poll()` with an explicit clamp:
   ```java
   long remaining = filter.hasLimit()
           ? filter.getLimit() - context.getCount().get()
           : Integer.MAX_VALUE;
   int pageSize = (int) Math.min(Math.max(remaining, 0), listenerProperties.getMaxPageSize());
   ```
3. **Validate derived filters**: pass the derived `newFilter` through the validator before use, or assert `pageSize >= 0` before building the new filter.
4. **Treat `hasLimit() == false` in `findByFilter` as a hard error** if the incoming limit is negative, rather than silently dropping the constraint.
