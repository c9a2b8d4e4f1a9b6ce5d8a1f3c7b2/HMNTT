# Audit Report

## Title
Gas Token Drain via REJECT Filter: Unrestored `gasLimitBucket` Tokens on `ThrottleException`

## Summary
In `ThrottleManagerImpl.throttle()`, gas tokens are consumed from `gasLimitBucket` before request filters are evaluated. When a REJECT-action filter matches and throws `ThrottleException`, those tokens are never restored because `ContractController.call()` only restores gas on `InvalidParametersException`, not on `ThrottleException`. An unprivileged attacker who discovers a REJECT filter pattern can repeatedly send matching high-gas requests to drain the shared `gasLimitBucket`, starving legitimate users of gas capacity.

## Finding Description

**`ThrottleManagerImpl.throttle()` (lines 37–49):**

Gas tokens are consumed at line 40, unconditionally before the filter loop:

```java
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // line 40 — tokens consumed
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}

for (var requestFilter : throttleProperties.getRequest()) {   // line 44 — filters evaluated AFTER
    if (requestFilter.test(request)) {
        action(requestFilter, request);                        // line 46 — may throw ThrottleException
    }
}
``` [1](#0-0) 

The `action()` method throws `ThrottleException("Invalid request")` for REJECT filters:

```java
case REJECT -> throw new ThrottleException("Invalid request");  // line 69
``` [2](#0-1) 

**`ContractController.call()` (lines 38–51):**

The only `restore()` call is guarded by `catch (InvalidParametersException e)`. `ThrottleException` is not caught, so it propagates out without restoring gas:

```java
try {
    throttleManager.throttle(request);          // ThrottleException escapes here — line 40
    ...
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas()); // never reached for ThrottleException — line 48
    throw e;
}
``` [3](#0-2) 

**Root cause:** The code assumes gas tokens only need to be restored when `InvalidParametersException` is thrown (post-throttle validation failure). It does not account for `throttle()` itself throwing after consuming gas — specifically when a REJECT filter fires.

**Failed assumption:** The comment on line 47 reads *"The validation failed, but no processing occurred so restore the consumed tokens"* — but this same condition (no processing occurred, tokens were consumed) also holds when `throttle()` throws `ThrottleException` via a REJECT filter. The catch block is too narrow. [5](#0-4) 

**Exploit flow:**
1. Attacker probes `POST /api/v1/contracts/call` with varying `to`, `from`, `data`, or `block` values.
2. Responses returning HTTP 429 with body `"Invalid request"` (distinct from `"Requests per second rate limit exceeded"`) identify a matching REJECT filter pattern.
3. Attacker sends requests at the `requestsPerSecond` rate (default 500/s) with the maximum allowed `gas` value, each matching the REJECT filter.
4. Each request: `rateLimitBucket` consumes 1 token (passes), `gasLimitBucket` consumes `scaleGas(gas)` tokens (passes), REJECT filter fires → `ThrottleException` thrown → gas tokens NOT restored.
5. `gasLimitBucket` drains to zero. Subsequent legitimate requests fail at line 40–42 with `"Gas per second rate limit exceeded."`.

**Numerical illustration (using defaults from `ThrottleProperties`):**
- `gasPerSecond` = 7,500,000,000; `GAS_SCALE_FACTOR` = 10,000 → scaled capacity = 750,000 tokens
- `requestsPerSecond` = 500; max gas per request (e.g., 15,000,000) → `scaleGas(15_000_000)` = 1,500 tokens each
- Attacker drain rate: 500 × 1,500 = 750,000 tokens/sec = full bucket drained per second
- Bucket refills at 750,000 tokens/sec; attacker matches the refill rate exactly, keeping the bucket perpetually empty [6](#0-5) 

## Impact Explanation
The `gasLimitBucket` is a shared, server-wide resource protecting `POST /api/v1/contracts/call`. Draining it prevents all legitimate `eth_call` / `eth_estimateGas` requests from executing EVM calls — those requests return `"Gas per second rate limit exceeded."` instead of results. This is a targeted denial-of-service against all gas-consuming operations with no cost to the attacker beyond network bandwidth; rejected requests consume no server-side EVM execution resources.

## Likelihood Explanation
No privileged access is required. Any external user can send HTTP POST requests to the public `/api/v1/contracts/call` endpoint. REJECT filter patterns are discoverable by probing (the distinct `"Invalid request"` error message differentiates REJECT from other throttle responses). The attack is fully repeatable and automatable with a simple HTTP client loop. The only friction is the `requestsPerSecond` rate limit, which the attacker can saturate legitimately. The attack is sustainable indefinitely as long as the attacker keeps sending requests.

## Recommendation
Restore gas tokens whenever `throttle()` throws for any reason other than the gas limit itself being exceeded. The fix should be applied in `ContractController.call()`:

```java
try {
    throttleManager.throttle(request);
    validateContractMaxGasLimit(request);
    final var params = constructServiceParameters(request);
    final var result = contractExecutionService.processCall(params);
    return new ContractCallResponse(result);
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());
    throw e;
} catch (ThrottleException e) {
    // If throttle() consumed gas tokens before throwing (e.g., REJECT filter),
    // restore them so the bucket is not drained by rejected requests.
    throttleManager.restore(request.getGas());
    throw e;
}
```

Alternatively, restructure `ThrottleManagerImpl.throttle()` so that gas tokens are consumed **after** all filters have been evaluated, eliminating the window where tokens are held but then thrown away.
