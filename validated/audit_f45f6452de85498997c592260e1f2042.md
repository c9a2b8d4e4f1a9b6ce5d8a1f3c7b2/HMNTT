Audit Report

## Title
Unauthenticated EVM Address DB Lookup Exhausts Connection Pool via `getNftsByAccountId`

## Summary
Every `GET /api/v1/accounts/:idOrAliasOrEvmAddress/nfts` request that supplies a non-long-zero EVM address unconditionally issues an uncached SQL query against the `entity` table. With no rate limiting in the REST layer and a bounded database connection pool, an attacker can saturate all pool slots with concurrent requests, denying service to all other API consumers.

## Finding Description

**Verified code path:**

`rest/routes/accountRoute.js` line 15 routes `GET /:id/nfts` to `AccountController.getNftsByAccountId`: [1](#0-0) 

`AccountController.getNftsByAccountId` at line 91 calls `EntityService.getEncodedId` with the raw path parameter on every request, with no pre-check or cache guard: [2](#0-1) 

`EntityService.getEncodedId` lines 120–124: when the input is a valid EVM address, `entityId.evmAddress` is non-null, so execution always falls into `getEntityIdFromEvmAddress`: [3](#0-2) 

`getEntityIdFromEvmAddress` lines 90–91 executes a raw SQL query against the `entity` table on every invocation — **no result caching**: [4](#0-3) 

The query used: [5](#0-4) 

**Why `parseCached` does not help:**
`parseCached` in `entityId.js` lines 301–332 caches only the parsed `EntityId` struct (shard/realm/num/evmAddress fields). It does not cache the DB lookup result. Furthermore, `getEncodedId` calls `EntityId.parseString` (not `parseCached`), so even the parse-level cache is bypassed entirely. After parsing, `getEntityIdFromEvmAddress` still hits the database unconditionally: [6](#0-5) 

**No rate limiting in the REST API:**
A search across `rest/**/*.js` for `rateLimit`, `throttle`, or `rateLimiter` returns zero matches. The REST layer has no request-rate throttling.

**DB pool is bounded:**
`dbpool.js` configures the pool with `max: config.db.pool.maxConnections`, `connectionTimeoutMillis`, and `statement_timeout` — all finite values: [7](#0-6) 

## Impact Explanation
With a bounded connection pool and no rate limiting, an attacker sustaining ~`maxConnections` concurrent requests with distinct valid EVM addresses holds all pool slots for the duration of each query (up to `statementTimeout`). All other API consumers — including gossip-transaction-record polling — receive connection-timeout errors. This is a full, unauthenticated denial-of-service against the mirror node REST API. The attacker does not need the EVM addresses to resolve to real entities; a 404 is returned only *after* the DB query completes, meaning the pool slot is consumed regardless.

## Likelihood Explanation
The attack requires no credentials, no on-chain state, and no special knowledge. Any 20-byte hex string that passes `EntityId.isValidEntityId` is sufficient. The attack is trivially scriptable with standard HTTP tooling (`ab`, `wrk`, `curl` loop) and is repeatable indefinitely.

## Recommendation
1. **Add a result cache in `getEntityIdFromEvmAddress`** (e.g., a short-TTL in-memory LRU cache keyed on the EVM address bytes) so repeated lookups for the same address do not hit the database.
2. **Apply rate limiting at the REST API layer** (e.g., via an express middleware such as `express-rate-limit`) to bound the number of requests per IP per time window.
3. **Increase the default pool size** or add a request queue with a maximum depth so that pool exhaustion results in fast 429/503 responses rather than hanging connections.
4. **Use `parseCached`** instead of `parseString` inside `getEncodedId` to at least avoid redundant parse work, and consider extending the cache to store resolved entity IDs for EVM addresses.

## Proof of Concept
```bash
# Send 50 concurrent requests with distinct random EVM addresses
for i in $(seq 1 50); do
  addr=$(openssl rand -hex 20)
  curl -s "http://<mirror-node>/api/v1/accounts/${addr}/nfts" &
done
wait
# All subsequent legitimate requests will receive connection timeout errors
# until the pool slots are released (up to statementTimeout ms each)
```
