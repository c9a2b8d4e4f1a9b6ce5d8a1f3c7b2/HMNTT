# Audit Report

## Title
Unauthenticated Gas Estimation Requests Can Exhaust DB Connection Pool via 10-Second Timeout Window

## Summary
An unprivileged attacker can submit crafted `estimate=true` contract call requests that trigger the `BinaryGasEstimator` binary search loop. The loop holds database connections for up to the full 10-second `requestTimeout` per request because: (1) `startTime` is never reset across iterations, (2) `safeCall` silently swallows `QueryTimeoutException`, and (3) gas throttling is explicitly bypassed for estimate calls. With enough concurrent requests, the HikariCP connection pool is exhausted, denying service to all legitimate callers.

## Finding Description

**Root cause — three independent design flaws that combine:**

**Flaw 1: `startTime` is immutable across the entire binary search.**
`ContractCallContext` sets `startTime` as a `final` field at construction time. [2](#0-1) 

`ContractCallContext.reset()`, called at the top of every binary search iteration in `BinaryGasEstimator.search()`, only clears `writeCache` — `startTime` is never touched. [3](#0-2) [8](#0-7) 

**Flaw 2: `statementInspector` throws `QueryTimeoutException`, but `safeCall` catches it silently.**
`HibernateConfiguration.statementInspector()` computes elapsed time against the shared `startTime` before every SQL statement and throws `QueryTimeoutException` once `elapsed >= timeout` (10,000 ms). [9](#0-8) 

`BinaryGasEstimator.safeCall()` uses a broad `catch (Exception ignored)` that silently absorbs this exception, logs a generic info message, and returns `null` — the loop is never terminated. [5](#0-4) 

The loop then treats `null` as a failed call (`err = true`) and continues up to `maxGasEstimateRetriesCount = 20` total iterations. [6](#0-5) 

**Flaw 3: Gas throttling is bypassed for estimate calls.**
`doProcessCall` wraps `restoreGasToBucket` in `if (!estimate)`, so estimate calls never consume from or restore to the token bucket. [7](#0-6) 

**Exploit flow:**
1. Attacker crafts a contract call that triggers many Hibernate SQL reads per EVM execution (e.g., reads many storage slots or calls precompiles that hit the DB).
2. Submits it as `estimate=true` with no authentication.
3. `ContractCallContext.run()` creates a context; `startTime` is set once.
4. `BinaryGasEstimator.search()` runs up to 20 iterations; each calls `doProcessCall()` → `transactionExecutionService.execute()` → multiple Hibernate SQL statements → `statementInspector()` checks elapsed time.
5. For the first ~9.9 seconds, all SQL statements pass the check; the Hibernate session (and its DB connection) is held open throughout.
6. When elapsed ≥ 10,000 ms, `statementInspector` throws `QueryTimeoutException`; `safeCall` catches it silently, returns `null`, and the loop continues.
7. Remaining iterations immediately fail at the first SQL check (fast-fail), but the connection was already held for ~10 seconds.
8. Attacker sends N concurrent such requests; with a typical HikariCP pool of 10–20 connections, N ≈ 10–20 concurrent requests saturates the pool.

**Why existing checks fail:**
- `statementInspector` is the only timeout enforcement, but it only prevents *new* SQL statements after the deadline; it cannot interrupt in-progress EVM execution or release already-held connections.
- `safeCall`'s broad `catch (Exception ignored)` neutralizes `QueryTimeoutException` as a loop-termination signal.
- Gas throttling is bypassed for estimate calls, so the attacker pays no gas cost and faces no rate limiting.

## Impact Explanation
An attacker can hold each DB connection for up to 10 seconds per request. With a standard HikariCP pool of 10 connections and 10 concurrent crafted estimate requests, the pool is fully saturated. All legitimate requests (including non-estimate calls) that need a DB connection will queue or fail with connection timeout errors. This constitutes a denial-of-service against the entire web3 module with no authentication and no gas cost.

## Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special tooling — only the ability to send HTTP POST requests to the public JSON-RPC endpoint. The crafted call data can be a simple contract that reads many storage slots. The attack is repeatable and automatable. `maxGasEstimateRetriesCount = 20` and `requestTimeout = 10s` are default configuration values, making every default deployment vulnerable.

## Recommendation

1. **Reset `startTime` per binary search iteration.** Make `startTime` non-final and add a `resetTimer()` method to `ContractCallContext`, then call it inside `BinaryGasEstimator.search()` at the top of each iteration alongside `reset()`. This ensures each EVM execution gets its own fresh 10-second budget.

2. **Propagate `QueryTimeoutException` out of `safeCall`.** Narrow the catch clause in `BinaryGasEstimator.safeCall()` to exclude timeout exceptions (e.g., re-throw `QueryTimeoutException` or `RuntimeException` subtypes that indicate resource exhaustion), so the binary search loop terminates immediately on timeout rather than continuing silently.

3. **Apply rate limiting to estimate calls.** Remove or condition the `if (!estimate)` guard around `restoreGasToBucket` so that estimate calls are subject to the same gas token-bucket throttling as regular calls. Alternatively, add a separate per-IP or per-connection request rate limiter at the HTTP layer for `estimate=true` requests.

4. **Set a per-iteration wall-clock deadline.** Independently of the Hibernate inspector, track a per-iteration start time and abort the binary search loop if the total elapsed time across all iterations exceeds the configured `requestTimeout`.

## Proof of Concept

```http
POST /api/v1/contracts/call HTTP/1.1
Host: <mirror-node>
Content-Type: application/json

{
  "data": "<calldata for a contract that reads 100+ storage slots>",
  "to": "<deployed contract address>",
  "estimate": true,
  "gas": 15000000
}
```

Send 15 such requests concurrently (e.g., via `ab`, `wrk`, or a simple Python `asyncio` script). Each request will hold a DB connection for ~10 seconds. With a default HikariCP pool of 10 connections, legitimate requests will begin failing with connection acquisition timeout errors within seconds of the attack starting. The attack can be sustained indefinitely by repeating the concurrent burst every 10 seconds.
