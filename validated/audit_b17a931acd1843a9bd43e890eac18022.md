Audit Report

## Title
Historical Opcode Replay Uses Current Bytecode Instead of Historical Bytecode, Producing Fabricated Execution Traces

## Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint replays historical EVM transactions but always loads the **current** contract bytecode from the `contract` table, ignoring the historical timestamp context. When a contract's bytecode has been updated since the original transaction, the replay executes under the new code, returning an opcode trace that never actually occurred on-chain.

## Finding Description

The call chain is:

`OpcodeServiceImpl.processOpcodeCall()` (line 64) → `ContractDebugService.processOpcodeCall()` (line 48) → `callContract()`

`ContractDebugService.processOpcodeCall()` correctly establishes a historical context at line 51:

```java
ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1));
``` [1](#0-0) 

However, during EVM execution, `ContractBytecodeReadableKVState.readFromDataSource()` (lines 39–46) is invoked to load bytecode:

```java
return contractRepository
        .findRuntimeBytecode(entityId.getId())
        .map(Bytes::wrap)
        .map(Bytecode::new)
        .orElse(null);
``` [2](#0-1) 

`ContractRepository.findRuntimeBytecode()` executes:

```sql
select runtime_bytecode from contract where id = :contractId
``` [3](#0-2) 

**There is no timestamp predicate.** This always returns the current row from the `contract` table. `ContractBytecodeReadableKVState.readFromDataSource()` never calls `ContractCallContext.get().getTimestamp()` and never queries `contract_history`.

**Failed assumption:** The system assumes contract bytecode is immutable. In reality, the schema explicitly supports historical bytecode. Migration `V1.64.0__contract_action.sql` adds `runtime_bytecode` to both `contract` and `contract_history`:

```sql
alter table if exists contract
    add column if not exists runtime_bytecode bytea null;
alter table if exists contract_history
    add column if not exists runtime_bytecode bytea null;
``` [4](#0-3) 

`SidecarContractMigration` uses an upsert that overwrites the current bytecode when a new version is ingested:

```java
insert into contract (id, runtime_bytecode)
values (?, ?)
on conflict (id)
do update set runtime_bytecode = excluded.runtime_bytecode
``` [5](#0-4) 

The historical timestamp set in `ContractCallContext` is used for storage slot lookups and entity lookups elsewhere, but `ContractBytecodeReadableKVState` bypasses it entirely.

## Impact Explanation
The opcodes endpoint is the primary forensic tool for auditing historical EVM execution on Hedera. When a contract's bytecode has been updated, every call to this endpoint for a pre-update transaction returns a fabricated opcode trace — one that represents an execution that never occurred on-chain. The returned `opcodes` array, `return_value`, `failed` flag, and `gas` values all reflect the new bytecode's behavior, not the original. This corrupts forensic investigations, dispute resolution, and any tooling that relies on the mirror node for historical execution fidelity. The endpoint is read-only so no funds are moved, but the integrity of the historical record is completely undermined for all affected contracts. [6](#0-5) 

## Likelihood Explanation
The precondition — a contract with updated bytecode — is realistic. Upgradeable proxy patterns are standard practice in EVM development, and Hedera's native contract update mechanism explicitly supports bytecode replacement, as evidenced by the `contract_history.runtime_bytecode` column and the `ON CONFLICT DO UPDATE` upsert in `SidecarContractMigration`. No privilege is required: any user who knows a transaction ID or hash for a pre-update transaction can call the endpoint. The exploit is fully passive and repeatable. [7](#0-6) 

## Recommendation
`ContractBytecodeReadableKVState.readFromDataSource()` must be made timestamp-aware. When `ContractCallContext.get().useHistorical()` is true, the method should query `contract_history` for the bytecode valid at the historical timestamp (i.e., the most recent `contract_history` row with `timestamp_range` covering `consensusTimestamp - 1`), falling back to the `contract` table only for non-historical calls. A new repository method such as `findRuntimeBytecodeByTimestamp(Long contractId, Long timestamp)` should be added to `ContractRepository`, mirroring the pattern already used by storage slot lookups (`findStorageByBlockTimestamp`). [8](#0-7) 

## Proof of Concept

1. Deploy a contract `C` at address `A` with bytecode `B1` that implements function `foo()` returning `1`.
2. Record the transaction hash `TX1` of a call to `foo()` — this executes under `B1`.
3. Upgrade the contract (via proxy pattern or Hedera's native update) so the `contract` table row for `C` now holds bytecode `B2`, where `foo()` returns `2`.
4. Call `GET /api/v1/contracts/results/TX1/opcodes`.
5. **Observed:** The response contains the opcode trace for `B2` executing `foo()` → returns `2`, `failed: false`.
6. **Expected:** The response should contain the opcode trace for `B1` executing `foo()` → returns `1`.

The discrepancy is directly caused by `ContractRepository.findRuntimeBytecode()` returning `B2` (the current row) instead of `B1` (the historical row from `contract_history`), because no timestamp filter is applied. [9](#0-8)
