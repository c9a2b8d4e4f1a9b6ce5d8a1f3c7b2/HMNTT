# Audit Report

## Title
Cache Key Timestamp Normalization Missing for Address Book Files Causes Perpetual Cache Thrashing and Sustained DB Load

## Summary
`SystemFileLoader.load()` applies `roundDownToHour()` timestamp normalization only for exchange rate, fee schedule, and simple fee schedule files. Address book files 101/102 use the raw nanosecond wall-clock timestamp as the `CacheKey`, making the `defaultSystemFileCacheManager` perpetually ineffective. Every address book read results in a database query.

## Finding Description

In `FileReadableKVState.readFromDataSource()`, `getCurrentTimestamp()` returns the current wall-clock time converted to nanoseconds: [1](#0-0) 

This value is passed directly to `systemFileLoader.load()` for all system files including address book files 101/102: [2](#0-1) 

In `SystemFileLoader.load()`, `roundDownToHour()` is applied only for three specific file IDs. Address book files fall through to line 120 with the raw nanosecond timestamp: [3](#0-2) 

`CacheKey` is a record keyed on `(FileID, long timestamp)`: [4](#0-3) 

Since nanosecond wall-clock time advances on every call, every request produces a distinct `CacheKey` for address book files. The cache lookup at line 129 always misses, and `loadFromDB()` is called unconditionally: [5](#0-4) 

The `defaultSystemFileCacheManager` is configured via `cacheProperties.getSystemFile()` spec string: [6](#0-5) 

Address book files 101/102 are registered as system files in `loadAll()`: [7](#0-6) 

## Impact Explanation
The cache for address book files 101/102 is structurally non-functional. Every request that triggers an address book read issues a `fileDataRepository.getFileAtTimestamp()` database query with no possibility of a cache hit. Under sustained load this degrades database performance for all mirror node consumers and can cause query timeouts or connection pool exhaustion.

## Likelihood Explanation
No authentication or special privileges are required. Any client with access to the `/api/v1/contracts/call` endpoint can trigger address book reads by sending repeated contract call requests. The attacker does not supply the timestamp — the system's own use of nanosecond wall-clock time as the cache key guarantees a cache miss on every request. The attack is trivially repeatable with a simple HTTP loop.

## Recommendation
Apply the same `roundDownToHour()` (or an equivalent normalization such as `roundDownToMinute()`) to the `consensusTimestamp` for address book file IDs before constructing the `CacheKey`, mirroring the existing pattern for exchange rate and fee schedule files:

```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp);
} else if (fileId.equals(addressBookFile101Id)
        || fileId.equals(addressBookFile102Id)) {
    consensusTimestamp = roundDownToHour(consensusTimestamp); // add normalization
}
```

This ensures that all requests within the same hour share a single `CacheKey` for address book files, making the cache effective.

## Proof of Concept
```bash
# Send repeated contract calls that trigger address book lookups
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"data":"<address-book-reading-calldata>","to":"<contract>"}' &
done
wait
# Each request generates a unique nanosecond CacheKey for file 101/102,
# resulting in 500 uncached DB queries per second with zero cache reuse.
```
