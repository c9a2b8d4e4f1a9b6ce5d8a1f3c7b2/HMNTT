Audit Report

## Title
Gas Throttle Bypass via Non-Existent Block Lookup Enables Unauthenticated DB Query Flooding

## Summary
An unprivileged caller can supply a syntactically valid but non-existent block identifier (number or hash) in a `/api/v1/contracts/call` request. The resulting `BlockNumberNotFoundException` is not caught by `doProcessCall`'s catch blocks, causing the `finally` branch to restore 100 % of the consumed gas back to the throttle bucket, completely neutralising the gas-based rate limiter for this failure path. For hash-typed lookups, `RecordFileRepository.findByHash` carries no `@Cacheable` annotation, so every request unconditionally issues a database query. The sole remaining guard is the 500 req/s request-count bucket.

## Finding Description

**Code path (all references verified):**

**1. Throttle consumption and dispatch**
`ContractController.call()` calls `throttleManager.throttle(request)` (consuming one rate-limit token and `scaleGas(gas)` gas tokens), then delegates to `contractExecutionService.processCall(params)`. [1](#0-0) 

**2. Lazy block supplier registration**
`ContractCallService.callContract()` registers a memoized supplier that calls `recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)`. The supplier is lazy — it is only evaluated when `getRecordFile()` is first called during EVM execution. [2](#0-1) 

**3. Supplier invocation inside `transactionExecutionService.execute()`**
`BlockInfoSingleton.get()` (called during EVM state initialisation) invokes `ContractCallContext.get().getRecordFile()`, which calls `blockSupplier.get()`. For a non-existent block, `findByBlockType` returns `Optional.empty()` and `orElseThrow` fires `BlockNumberNotFoundException`. [3](#0-2) [4](#0-3) 

**4. Exception not caught by `doProcessCall`**
`doProcessCall` only catches `IllegalStateException | IllegalArgumentException` and `MirrorEvmTransactionException`. `BlockNumberNotFoundException` extends `InvalidInputException` → `Web3Exception` → `MirrorNodeException` → `RuntimeException` — none of those types. [5](#0-4) [6](#0-5) [7](#0-6) 

**5. `finally` block restores 100 % of gas**
Because `result` is still `null` when the `finally` block runs, `restoreGasToBucket(null, gasLimit)` takes the `result == null` branch and restores `gasLimitToRestoreBaseline = gasLimit × gasLimitRefundPercent / 100` (default 100 %) back to the bucket. Net gas consumed from the bucket: zero. [8](#0-7) [9](#0-8) 

**6. `findByHash` has no `@Cacheable` annotation**
`RecordFileRepository.findByIndex` is annotated `@Cacheable(unless = "#result == null")`. Because `Optional.empty()` is not `null`, a miss for a given index is cached after the first hit. `findByHash` has **no** `@Cacheable` annotation at all — every call executes `WHERE hash LIKE concat(:hash, '%')` against the database. [10](#0-9) 

**7. `scaleGas` makes the gas bucket irrelevant for gas ≤ 10 000**
`ThrottleProperties.scaleGas` returns `0` for any gas value ≤ `GAS_SCALE_FACTOR` (10 000). For such requests the gas bucket is never consumed and never restored — the gas throttle is structurally irrelevant regardless of the exception path. [11](#0-10) 

**8. Sole remaining guard: 500 req/s request-count bucket** [12](#0-11) 

**Root cause:** `doProcessCall` assumes that all costly failure paths produce a `MirrorEvmTransactionException` (or `IllegalStateException`/`IllegalArgumentException`), so that gas accounting is correct. `BlockNumberNotFoundException` escapes all catch blocks, leaving `result == null` and triggering unconditional full-gas restoration. The hash-lookup path additionally lacks any caching layer.

## Impact Explanation

Each attacker thread can sustain up to 500 database queries per second at zero effective gas cost. The `findByHash` query (`WHERE hash LIKE concat(:hash, '%')`) is a prefix-scan that may be slow on large `record_file` tables. The configured `db.statementTimeout` is 3 000 ms; at 500 req/s each query can hold a connection for up to 3 seconds, meaning up to ~1 500 concurrent connections can be in-flight simultaneously. A typical connection pool (10–50 connections) is exhausted well before that ceiling, causing all subsequent legitimate requests to queue or fail with connection-pool timeouts. This constitutes a denial-of-service against the entire web3 API surface. [13](#0-12) 

## Likelihood Explanation

No authentication or special privilege is required. The attack requires only the ability to POST to `/api/v1/contracts/call` with a `block` field set to a syntactically valid but non-existent value (e.g., `"0x" + 64 random hex chars`). This is trivially scriptable. The 500 req/s per-server rate limit can be circumvented by distributing requests across multiple source IPs. The attack is fully repeatable and stateless.

## Recommendation

1. **Catch `BlockNumberNotFoundException` (and other `InvalidInputException` subclasses) in `doProcessCall`** and either wrap it in a `MirrorEvmTransactionException` (so gas accounting is correct) or add a dedicated catch block that calls `restoreGasToBucket` with the full gas limit **without** restoring it to the bucket (i.e., treat it as a consumed request).
2. **Add `@Cacheable` to `RecordFileRepository.findByHash`** with an appropriate TTL and a `unless` condition that caches empty results (e.g., `unless = "#result == null"`), consistent with `findByIndex`.
3. **Evaluate whether the `ContractController` catch block for `InvalidParametersException`** should be broadened to cover all `InvalidInputException` subclasses, so that gas is not restored at the controller layer for any pre-execution validation failure.

## Proof of Concept

```
# Send 500 requests/second with a random non-existent block hash.
# Each request triggers one uncached DB LIKE-scan and restores 100% gas to the bucket.

for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{
      "from": "0x0000000000000000000000000000000000000001",
      "to":   "0x0000000000000000000000000000000000000002",
      "gas":  50000,
      "block": "0x'$(openssl rand -hex 32)'"
    }' &
done
wait
# Repeat in a loop from multiple IPs to exceed the 500 req/s per-server limit.
```

Each iteration: one `findByHash` DB query, zero net gas tokens consumed, one rate-limit token consumed. DB connection pool exhaustion occurs when concurrent slow queries exceed pool capacity.

    protected final EvmTransactionResult doProcessCall(
            CallServiceParameters params, long estimatedGas, boolean estimate) throws MirrorEvmTransactionException {
        EvmTransactionResult result = null;
        var status = ResponseCodeEnum.SUCCESS.toString();

        try {
            result = transactionExecutionService.execute(params, estimatedGas);

            if (!estimate) {
                validateResult(result, params);
            }
        } catch (IllegalStateException | IllegalArgumentException e) {
            throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
        } catch (MirrorEvmTransactionException e) {
            // This result is needed in case of exception to be still able to call restoreGasToBucket method
            result = e.getResult();
            status = e.getMessage();
            throw e;
        } finally {
            if (!estimate) {
                restoreGasToBucket(result, params.getGas());

                // Only record metric if EVM is invoked and not inside estimate loop
                if (result != null) {
                    updateMetrics(params, result.gasUsed(), 1, status);
                }
            }
        }
        return result;
    }

```java
    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
```
