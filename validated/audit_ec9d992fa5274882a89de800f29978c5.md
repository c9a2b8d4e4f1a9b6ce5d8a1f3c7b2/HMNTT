Audit Report

## Title
Unbounded Historical Scan via `timestamp=lte:X` on `/contracts/:contractId/state` Causes Full Cross-Partition Table Scan

## Summary
The `getContractStateById` handler in `rest/controllers/contractController.js` accepts a timestamp filter and routes to `contractStateTimestampQuery`, which executes `DISTINCT ON (slot)` over the partitioned `contract_state_change` table with only an upper-bound constraint and no lower-bound guard. Any unauthenticated caller can supply `timestamp=lte:9999999999` to force PostgreSQL to scan every historical partition for the target contract, merge and sort all rows, and apply `DISTINCT ON` before `LIMIT` takes effect, causing sustained DB I/O and CPU spikes.

## Finding Description

**`contractStateTimestampQuery` — confirmed at:** [1](#0-0) 

The query uses `DISTINCT ON (slot)` with `ORDER BY slot ASC, consensus_timestamp DESC`. For `DISTINCT ON` to correctly return the most-recent value per slot, PostgreSQL must process **all** rows for each slot before emitting any output row. The `LIMIT` clause only applies after `DISTINCT ON` completes — it does not short-circuit the cross-partition merge and sort.

**`getContractStateByIdAndFilters()` switches to this query when `timestamp = true`:** [2](#0-1) 

**`extractContractStateByIdQuery()` sets `timestampPresent = true` for any operator (`lte`, `lt`, `gte`, `gt`, `eq`→`lte`) with no lower-bound enforcement:** [3](#0-2) 

**`getContractStateById` handler calls these two functions directly with no `bindTimestampRange` guard:** [4](#0-3) 

`bindTimestampRange` is imported at line 32 of `contractController.js` and is used in `optimizeTimestampFilters()` (lines 384–404), but `optimizeTimestampFilters` is **never called** in the `getContractStateById` handler path. [5](#0-4) 

**`contract_state_change` is declared `PARTITION BY RANGE (consensus_timestamp)`:** [6](#0-5) 

Time-based partitions are created automatically for this table: [7](#0-6) 

**The only relevant index is `(contract_id, slot, consensus_timestamp)`:** [8](#0-7) 

This index stores `consensus_timestamp` ascending. The query requires `ORDER BY slot ASC, consensus_timestamp DESC`. PostgreSQL can use the index to filter by `contract_id` and scan by `slot`, but must reverse-scan `consensus_timestamp` per slot group. With only an upper bound (`<= 9999999999000000000`), it must scan **all** historical rows for the contract across every partition.

**The `stateTimestampQueryParam` schema pattern permits seconds up to `9999999999` (year 2286):** [9](#0-8) 

## Impact Explanation
A single HTTP GET request with `timestamp=lte:9999999999` against a high-activity contract (e.g., a DEX or stablecoin with thousands of storage slots and years of history) forces a full cross-partition sequential scan of `contract_state_change`, a cross-partition sort, and a `DISTINCT ON` aggregation — all before `LIMIT 25` (default) takes effect. Repeated requests from one or more unauthenticated clients can saturate DB I/O and CPU, degrading or denying service to all other API consumers. Impact scales with the age and activity of the targeted contract.

## Likelihood Explanation
No authentication is required. The endpoint is publicly documented in the OpenAPI spec. The exploit requires only knowledge of a high-activity contract ID (trivially discoverable from block explorers) and a single HTTP GET request. It is trivially scriptable and repeatable at high frequency. No per-IP rate limiting is present in the REST layer for this endpoint.

## Recommendation
1. **Enforce a lower-bound on timestamp in `extractContractStateByIdQuery()`**: When only an upper-bound timestamp is provided, automatically inject a lower-bound (e.g., `consensus_timestamp >= upper_bound - maxRange`) to limit partition scan scope.
2. **Apply `bindTimestampRange` to the state endpoint**: Call `optimizeTimestampFilters` (which already invokes `bindTimestampRange`) in the `getContractStateById` handler, consistent with how contract logs and results endpoints handle timestamp ranges.
3. **Reject open-ended upper-only timestamp ranges**: Require both a lower and upper bound when a timestamp filter is present, or cap the implicit lower bound to a configurable maximum lookback window.

## Proof of Concept

```
GET /api/v1/contracts/0.0.1234/state?timestamp=lte:9999999999
```

Against a contract with years of storage history, this triggers:
1. `extractContractStateByIdQuery()` sets `timestampPresent = true`, adds `consensus_timestamp <= 9999999999000000000` with no lower bound.
2. `getContractStateByIdAndFilters()` selects `contractStateTimestampQuery`.
3. PostgreSQL scans all partitions of `contract_state_change` for `contract_id = 1234`, merges and sorts by `(slot ASC, consensus_timestamp DESC)`, applies `DISTINCT ON (slot)`, then applies `LIMIT 25`.
4. DB CPU and I/O spike for the duration of the query. Repeated at high frequency, this exhausts DB resources and degrades service for all users.
