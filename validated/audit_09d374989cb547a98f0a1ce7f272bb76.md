Audit Report

## Title
UNION ALL Subquery Overlap Produces Duplicate Allowance Rows via Crafted `spender.id` Range

## Summary
When a caller supplies a degenerate closed range for `spender.id` where the lower and upper bound share the same value (e.g., `spender.id=gte:5&spender.id=lte:5`) together with a `token.id` range, `getLowerFilters` and `getUpperFilters` independently rewrite their respective primary bounds to equality predicates with the same value. Because `TokenAllowanceService.getQuery()` joins all non-empty subqueries with `UNION ALL`, every row satisfying both predicates is returned twice.

## Finding Description

**Exact code path (all line references verified against the actual source):**

`getAccountTokenAllowances` [1](#0-0)  calls `extractTokenMultiUnionQuery` [2](#0-1) , which calls `validateBounds` at line 49 [3](#0-2)  and then `getLowerFilters` / `getInnerFilters` / `getUpperFilters` in `baseController.js`. [4](#0-3)  The resulting `lower`, `inner`, and `upper` arrays are passed to `TokenAllowanceService.getQuery`, which joins non-empty subqueries with `UNION ALL` at line 74. [5](#0-4) 

**Root cause — `getLowerFilters` (line 141):**

When `primary.hasLower()` and `secondary.hasLower()` are both true, the lower subquery is built by rewriting the primary lower bound to an equality operator:

```js
filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
``` [6](#0-5) 

**Root cause — `getUpperFilters` (line 182):**

Independently, when `primary.hasUpper()` and `secondary.hasUpper()` are both true, the upper subquery is built by rewriting the primary upper bound to an equality operator:

```js
return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
``` [7](#0-6) 

When `lower_value === upper_value` (i.e., `gte:5` and `lte:5`), both rewrites produce `spender = 5`. The lower subquery matches `spender = 5 AND token_id >= 100`; the upper subquery matches `spender = 5 AND token_id <= 200`. Any row with `spender = 5` and `100 ≤ token_id ≤ 200` satisfies both, and `UNION ALL` returns it twice.

The inner subquery receives `[{spender, gt, 5}, {spender, lt, 5}]`, which is always empty (`spender > 5 AND spender < 5`), but it is non-empty as an array, so all three subqueries are included in the `UNION ALL`. [8](#0-7) 

**Why validation does not catch it:**

`validateBoundsRange` (lines 117–123) only rejects the combination of a range **and** an equality on the same key:

```js
if (bounds[key].hasBound() && bounds[key].hasEqual()) { throw ... }
``` [9](#0-8) 

A range where `gte_value === lte_value` has no `equal` field set, so this check passes.

`validateLowerBounds` (lines 83–92) only rejects the case where the primary lower operator is `gt` (not `gte`) when a secondary lower bound is present. It does not compare the numeric values of the lower and upper bounds. [10](#0-9) 

`validateUpperBounds` (lines 100–109) mirrors this logic for the upper side. [11](#0-10) 

No validation anywhere detects or rejects `gte:X lte:X` with the same `X`.

## Impact Explanation

The REST API returns a JSON `allowances` array containing duplicate entries for every `(spender, token_id)` pair where `spender = X` and `lower_token_id ≤ token_id ≤ upper_token_id`. Any client that iterates the list, sums amounts, or makes authorization decisions based on the presence of an allowance will process phantom duplicate records. Pagination is also corrupted: the outer `LIMIT` is consumed by duplicates, so the `next` link may skip real records. The data integrity of the read path is broken for the targeted account without any write access required.

## Likelihood Explanation

The endpoint is unauthenticated and publicly reachable. The trigger requires only two query parameters with the same numeric value (`spender.id=gte:X&spender.id=lte:X`) plus a `token.id` range — a trivially crafted URL. No account ownership, signing key, or elevated privilege is needed. The condition is deterministic and 100% reproducible for any account that has at least one token allowance for the targeted spender.

## Recommendation

Add a validation step in `validateBounds` (or within `extractTokenMultiUnionQuery`) that compares the numeric values of the primary lower and upper bounds. If `primary.lower.value === primary.upper.value` (after entity-ID normalization), the combination should either be rejected with a `400 Bad Request` or collapsed into a single equality predicate (`spender = X`) before the three filter arrays are computed. This prevents `getLowerFilters` and `getUpperFilters` from independently emitting the same equality predicate and feeding both into a `UNION ALL`.

Alternatively, replace `UNION ALL` with `UNION` (deduplication) in `TokenAllowanceService.getQuery` at line 74, though this is a heavier fix and may mask other latent overlap cases.

## Proof of Concept

**Request:**
```
GET /api/v1/accounts/1/allowances/tokens?spender.id=gte:5&spender.id=lte:5&token.id=gte:100&token.id=lte:200
```

**Generated SQL (derived from `getQuery` logic):**
```sql
(select * from token_allowance
   where owner = $1 and amount > 0 and spender = $3 and token_id >= $4
   order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance
   where owner = $1 and amount > 0 and spender > $5 and spender < $6   -- always empty
   order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance
   where owner = $1 and amount > 0 and spender = $7 and token_id <= $8
   order by spender asc, token_id asc limit $2)
order by spender asc, token_id asc limit $2
-- params: [1, limit, 5, 100, 5, 5, 5, 200]
```

Every row with `spender = 5` and `100 ≤ token_id ≤ 200` satisfies both the first and third subqueries and is returned twice in the `allowances` array. [12](#0-11)

```javascript
  getLowerFilters(bounds) {
    let filters = [];
    const {primary, secondary} = bounds;
    if (!secondary.hasBound()) {
      // no secondary bound filters or no secondary filters at all, everything goes into the lower part and there
      // shouldn't be inner or upper part.
      filters = [primary.equal, primary.lower, primary.upper, secondary.equal];
    } else if (primary.hasLower() && secondary.hasLower()) {
      // both have lower. If primary has lower and secondary doesn't have lower, the lower bound of primary
      // will go into the inner part.
      filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
    } else if (primary.hasEqual()) {
      filters = [primary.equal, primary.lower, primary.upper, secondary.lower, secondary.equal, secondary.upper];
    }
    return filters.filter((f) => !isNil(f));
  }

  /**
   * Gets filters for the inner part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getInnerFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasBound() || !secondary.hasBound()) {
      return [];
    }

    return [
      // if secondary has lower bound, the primary filter should be > ?
      {filter: primary.lower, newOperator: secondary.hasLower() ? utils.opsMap.gt : null},
      // if secondary has upper bound, the primary filter should be < ?
      {filter: primary.upper, newOperator: secondary.hasUpper() ? utils.opsMap.lt : null},
    ]
      .filter((f) => !isNil(f.filter))
      .map((f) => ({...f.filter, operator: f.newOperator || f.filter.operator}));
  }

  /**
   * Gets filters for the upper part of the multi-union query
   *
   * @param {Bound}[] Bounds
   * @return {{key: string, operator: string, value: *}[]}
   */
  getUpperFilters(bounds) {
    const {primary, secondary} = bounds;
    if (!primary.hasUpper() || !secondary.hasUpper()) {
      return [];
    }
    // the upper part should always have primary filter = ?
    return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
  }
```

  getQuery(query) {
    const {lower, inner, upper, order, ownerAccountId, limit} = query;
    const params = [ownerAccountId, limit];
    const accountIdCondition = `${TokenAllowance.OWNER} = $1`;
    const limitClause = super.getLimitQuery(2);
    const orderClause = super.getOrderByQuery(
      ...TokenAllowanceService.orderByColumns.map((column) => OrderSpec.from(column, order))
    );

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = this.getSubQuery([], params, accountIdCondition, limitClause, orderClause);
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
    }

    return {sqlQuery, params};
  }
