Audit Report

## Title
Unbounded Subscription Intake Exhausts Shared `boundedElastic()` Thread Pool in POLL Mode, Denying Service to New Subscribers

## Summary
When `hiero.mirror.grpc.listener.type` is set to `POLL`, each active subscriber schedules a repeating blocking JPA database poll on the global `Schedulers.boundedElastic()` singleton. Because no subscription cap is enforced, an attacker can open enough concurrent gRPC connections to saturate the bounded thread pool, causing new subscription setup tasks to queue indefinitely or be rejected outright.

## Finding Description

**Confirmed: `PollingTopicListener` uses the global `boundedElastic()` singleton per subscriber.**

`PollingTopicListener` stores `Schedulers.boundedElastic()` in a field and uses it for both `delaySubscription` and `repeatWhen`: [1](#0-0) 

Each call to `Schedulers.boundedElastic()` returns the same JVM-global singleton. Every subscriber in POLL mode therefore competes for threads on the same pool.

**Confirmed: `findByFilter` is a fully blocking JPA call.**

`TopicMessageRepositoryCustomImpl.findByFilter()` uses `EntityManager` (Hibernate/JDBC) and calls `typedQuery.getResultList().stream()`. The code comment explicitly acknowledges the blocking nature: [2](#0-1) 

Because `delaySubscription(interval, scheduler)` fires the subscription on a `boundedElastic()` thread, and `Flux.defer(() -> poll(context))` executes on that same thread, the blocking JPA call holds the thread for the full duration of the database query on every poll cycle. [3](#0-2) [4](#0-3) 

**Confirmed: `subscriberCount` is a metric gauge only — never enforced.**

`TopicMessageServiceImpl` tracks `subscriberCount` solely as a Micrometer gauge. It is never compared against a maximum before accepting a new subscription: [5](#0-4) [6](#0-5) 

**Confirmed: `safetyCheck` also uses `subscribeOn(Schedulers.boundedElastic())`.**

Each subscription also schedules a safety-check mono on the same global pool, compounding thread pressure: [7](#0-6) 

**Scope correction vs. the submitted claim.**

The claim incorrectly treats `SHARED_POLL` as equally vulnerable. `SharedPollingTopicListener` uses `.share()` to create a single upstream flux shared by all subscribers — the `subscribeOn(scheduler)` applies to one polling loop regardless of subscriber count: [8](#0-7) 

The `publishOn(Schedulers.boundedElastic())` in `SharedTopicListener.listen()` dispatches already-fetched messages to each subscriber; it does not hold a thread during a blocking DB call: [9](#0-8) 

The vulnerability is therefore specific to `POLL` mode. The default listener type is `REDIS`: [10](#0-9) 

**Partial mitigation: `maxConcurrentCallsPerConnection = 5`.**

`GrpcConfiguration` enforces a per-connection call limit of 5 (default): [11](#0-10) [12](#0-11) 

This limits calls per TCP connection but does not cap total connections or total subscriptions. An attacker simply opens more connections.

## Impact Explanation
In POLL mode, every active attacker subscription continuously occupies a `boundedElastic()` thread during each blocking JPA poll cycle. The `boundedElastic()` pool is capped at `10 × availableProcessors` threads with a task queue of 100,000 entries. Once threads are saturated, new legitimate subscribers' `delaySubscription` tasks queue indefinitely. Once the queue fills, tasks are rejected with `RejectedExecutionException`, hard-failing new subscriptions. The safety-check `subscribeOn(Schedulers.boundedElastic())` in `TopicMessageServiceImpl` also queues on the same exhausted pool, compounding the effect.

## Likelihood Explanation
The attack requires only the ability to open gRPC TCP connections — no credentials or special privileges. The `subscribeTopic` endpoint is publicly documented. The per-connection limit of 5 concurrent calls means an attacker needs `ceil(threadPoolSize / 5)` connections to saturate the pool, which is trivially achievable. The attack is repeatable and automatable with any gRPC client library. The vulnerability only manifests when `hiero.mirror.grpc.listener.type=POLL` is configured (not the default `REDIS`), which reduces real-world exposure but does not eliminate it for deployments that use POLL mode.

## Recommendation
1. **Enforce a global subscription cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` before accepting a new subscription, returning `RESOURCE_EXHAUSTED` when the cap is reached.
2. **Use a dedicated scheduler for POLL mode**: Replace the global `Schedulers.boundedElastic()` singleton in `PollingTopicListener` with a dedicated `Schedulers.newBoundedElastic(...)` instance with an explicit thread cap sized for the expected subscription load.
3. **Consider making `findByFilter` non-blocking**: Migrate `TopicMessageRepositoryCustomImpl` to R2DBC to eliminate the blocking JPA call entirely, removing the root cause of thread holding.
4. **Add per-IP or per-client connection rate limiting** at the load balancer or gRPC interceptor layer.

## Proof of Concept
```bash
# Open 50 connections, each with 5 concurrent subscribeTopic streams (250 total subscriptions)
# On a 4-core host this saturates the 40-thread boundedElastic() pool
for i in $(seq 1 50); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
      localhost:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
# New legitimate subscribers now queue indefinitely or receive RejectedExecutionException
```

```java
    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
