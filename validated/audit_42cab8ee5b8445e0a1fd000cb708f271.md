# Audit Report

## Title
Unrestricted `ne` Operator on `ACCOUNT_ID` Filter Enables Sequential Scan Griefing

## Summary
The `filterValidityChecks` function in `rest/utils.js` validates the `account.id` query parameter by checking only the value format, not the operator. This allows any unauthenticated user to supply `account.id=ne:0.0.1`, which generates an `account_id != $1` SQL predicate. PostgreSQL cannot efficiently use a B-tree index for a `!=` predicate on a high-cardinality column, causing full sequential scans on large tables. Repeated requests degrade query performance for all concurrent users.

## Finding Description

`basicOperators` is defined to explicitly exclude `ne`:

```js
// rest/utils.js, lines 264-266
const basicOperators = Object.values(constants.queryParamOperators).filter(
  (o) => o !== constants.queryParamOperators.ne
);
```

This restriction is correctly applied to `BLOCK_NUMBER` and `SLOT`:

```js
// line 302
ret = (isPositiveLong(val, true) || isHexPositiveInt(val, true)) && includes(basicOperators, op);

// line 357
ret = isValidSlot(val) && includes(basicOperators, op);
```

However, the `ACCOUNT_ID` case performs no operator check whatsoever:

```js
// lines 289-291
case constants.filterKeys.ACCOUNT_ID:
  ret = EntityId.isValidEntityId(val);
  break;
```

`opsMap` maps `ne` to the SQL `!=` operator:

```js
// lines 29-36
const opsMap = {
  lt: ' < ', lte: ' <= ', gt: ' > ', gte: ' >= ', eq: ' = ', ne: ' != ',
};
```

`parseAccountIdQueryParam` then injects this operator verbatim into the SQL clause:

```js
// lines 619-623
(op, value) => {
  return Array.isArray(value)
    ? [`${columnName} IN (?`.concat(', ?'.repeat(value.length - 1)).concat(')'), value]
    : [`${columnName}${op}?`, [value]];
},
```

For a `ne` operator with a non-array value, this produces: `account_id != $1`.

**Root cause:** The operator restriction pattern (`basicOperators`) was applied to `BLOCK_NUMBER` and `SLOT` but was omitted for `ACCOUNT_ID`. The `filterValidityChecks` switch case for `ACCOUNT_ID` performs no `op` check, so `ne` passes validation and reaches query construction.

## Impact Explanation
Any endpoint accepting `account.id` as a filter (e.g., `/api/v1/transactions`, `/api/v1/balances`, NFT endpoints) is affected. On a production mirror node with millions of rows, a sequential scan on `crypto_transfer` or `transaction` can take seconds per query. Concurrent legitimate queries relying on the same tables will experience I/O saturation and CPU contention. Service availability degrades for all users with no financial cost to the attacker.

## Likelihood Explanation
The attack requires zero authentication, zero tokens, and zero on-chain activity. The endpoint is publicly accessible. The attacker needs only a valid entity ID (e.g., `0.0.1`, which always exists on any Hedera network) and knowledge that `ne:` is an accepted operator prefix. The attack is trivially scriptable and repeatable at any rate the network allows.

## Recommendation
Apply the same `basicOperators` restriction to `ACCOUNT_ID` in `filterValidityChecks`:

```js
// rest/utils.js, lines 289-291 — change to:
case constants.filterKeys.ACCOUNT_ID:
  ret = EntityId.isValidEntityId(val) && includes(basicOperators, op);
  break;
```

This mirrors the existing pattern used for `BLOCK_NUMBER` (line 302) and `SLOT` (line 357) and prevents `ne` from reaching query construction for account ID filters.

## Proof of Concept
```
GET /api/v1/transactions?account.id=ne:0.0.1
```
1. `paramValidityChecks` calls `filterValidityChecks('account.id', 'ne', '0.0.1')`.
2. `EntityId.isValidEntityId('0.0.1')` returns `true`; no operator check is performed.
3. `parseAccountIdQueryParam` calls `parseParams`, which calls `parseOperatorAndValueFromQueryParam('ne:0.0.1')`, returning `{op: ' != ', value: '0.0.1'}`.
4. The `processQuery` callback produces `['account_id != $1', [<encoded_id>]]`.
5. PostgreSQL executes a full sequential scan on `crypto_transfer` / `transaction`.
6. Repeating at high frequency saturates I/O and CPU, degrading all concurrent indexed queries.
