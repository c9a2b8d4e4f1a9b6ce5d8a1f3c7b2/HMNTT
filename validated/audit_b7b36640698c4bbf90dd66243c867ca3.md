Audit Report

## Title
Deleted Account Data Exposed via Long-Zero EVM Address Lookup Bypassing `deleted` Filter

## Summary
`EntityServiceImpl.getByEvmAddressAndType()` contains a branching lookup that, when the supplied EVM address is in Hedera's long-zero format, falls through to `CrudRepository.findById()` — which carries no `deleted is not true` guard. Every other lookup path in the same service enforces the soft-delete filter at the SQL level. An unprivileged caller can retrieve full account data for any deleted account by encoding its numeric ID as a long-zero EVM address.

## Finding Description
**Confirmed code path:**

`AccountController.account()` → `EntityServiceImpl.getByEvmAddressAndType()` → long-zero branch → `entityRepository.findById()`

In `EntityServiceImpl.java` lines 37–38:
```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
}
``` [1](#0-0) 

`findById` is the standard Spring Data `CrudRepository` method inherited by `EntityRepository`; it issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate. [2](#0-1) 

Both custom repository queries carry the guard:
- `findByAlias`: `select * from entity where alias = ?1 and deleted is not true`
- `findByEvmAddress`: `select * from entity where evm_address = ?1 and deleted is not true` [3](#0-2) 

The only post-fetch filter applied after `findById` is `e.getType() == type`, which checks entity type but not deletion status. [4](#0-3) 

**Additional scope:** `getByIdAndType()` (line 25) also calls `findById` without a deleted check, meaning the `entityId` input path to `AccountController.account()` is equally affected. [5](#0-4) 

**Root cause:** The long-zero branch was written to resolve Hedera account-number aliases to their internal entity row, but the `deleted is not true` guard that all other branches enforce at the SQL level was omitted.

**Failed assumption:** The developer assumed `findById` would be safe because deleted accounts are "gone," but the `deleted` column is a soft-delete flag — rows remain in the table.

## Impact Explanation
Any deleted account's full record — including its admin key, memo, auto-renew settings, and balance snapshot — is readable by any unauthenticated caller who knows the account's numeric ID. Because Hedera account numbers are sequential and publicly observable on-chain, enumeration is trivial. This constitutes unintended information disclosure of data that the protocol explicitly marks as deleted/inaccessible. No funds are directly at risk, but key material exposure could facilitate targeted attacks on accounts whose keys were reused after deletion.

## Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication required. The long-zero EVM address format is documented Hedera behavior. Account numbers are sequential and publicly observable on-chain. An attacker needs only to know a deleted account's number, encode it as `0x` + 24 zero hex chars + 16-hex-char number, and issue a single POST request. Fully repeatable with zero privilege.

## Recommendation
Replace `entityRepository.findById()` in both the long-zero branch of `getByEvmAddressAndType()` and in `getByIdAndType()` with a custom repository method that includes the `deleted is not true` predicate, consistent with `findByAlias` and `findByEvmAddress`. For example, add to `EntityRepository`:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Then update both service methods to call `findByIdAndNotDeleted` instead of `findById`. [3](#0-2) 

## Proof of Concept
Given a deleted account with numeric ID `12345` (hex `0x0000000000003039`):

```
POST /graphql/alpha
Content-Type: application/json

{
  "query": "{ account(input: { evmAddress: \"0x000000000000000000000000000000000000000000003039\" }) { id alias memo adminKey { key } } }"
}
```

The long-zero check passes (`first 4 bytes == 0`, `next 8 bytes == 0`), `findById(12345)` returns the soft-deleted row, and the only post-fetch filter (`e.getType() == ACCOUNT`) passes — returning the full account record despite the `deleted = true` flag. [6](#0-5) [7](#0-6)
