Audit Report

## Title
Silent Sentinel-Value Collision in `getIntMetadataValue` Allows `valid_start_nanos: "0"` to Trigger Random Transaction ID Generation

## Summary
`getIntMetadataValue()` uses a `< 0` boundary check, allowing the value `0` to pass through without error. `transactionSetTransactionId()` treats `validStartNanos == 0` as a sentinel meaning "generate a random ID." Because no guard exists between these two functions on the `valid_start_nanos` code path, a caller supplying `valid_start_nanos: "0"` silently receives a server-generated transaction ID instead of an error or an epoch-0 timestamp.

## Finding Description

**`getIntMetadataValue()`** — `rosetta/app/services/construction_service.go` line 549:
```go
if metadataValue, err = tools.ToInt64(value); err != nil || metadataValue < 0 {
    return metadataValue, errors.ErrInvalidArgument
}
```
The guard is strictly `< 0`. The string `"0"` parses to `0`, which is not `< 0`, so the function returns `(0, nil)`. [1](#0-0) 

**`getTransactionTimestampProperty()`** — lines 424–427: After calling `getIntMetadataValue` for `valid_start_nanos`, there is no post-call check for `validStartNanos <= 0`. The function returns `(0, nil)` silently. [2](#0-1) 

By contrast, the `valid_until_nanos` branch at lines 407–412 **does** explicitly check `if validStartNanos <= 0` and returns `ErrInvalidArgument`. That guard is entirely absent for the direct `valid_start_nanos` path. [3](#0-2) 

**`transactionSetTransactionId()`** — lines 730–731: `0` is the sentinel for "no caller-supplied value; generate a random current-time ID."
```go
if validStartNanos == 0 {
    transactionId = hiero.TransactionIDGenerate(payer)
``` [4](#0-3) 

**Root cause:** The boundary check `< 0` in `getIntMetadataValue` and the sentinel `== 0` in `transactionSetTransactionId` share the value `0` with no guard between them on the `valid_start_nanos` code path. The `valid_until_nanos` path has the correct guard; the `valid_start_nanos` path does not.

## Impact Explanation
A caller supplying `valid_start_nanos: "0"` receives an unsigned transaction whose transaction ID was generated at server time, not at the epoch they specified. The mirror node indexes and returns this transaction under the server-generated `payer@<server_nanos>` identifier. Any downstream system that pre-computed or stored the expected transaction ID (`payer@0.000000000`) will fail to locate the transaction. In a Rosetta construction flow where the caller is expected to track the transaction ID for status polling, this silent substitution breaks the caller's ability to correlate the submitted transaction with mirror-node records without inspecting the raw bytes of the response. Note that a transaction with `valid_start = 0` (Unix epoch, 1970) would be rejected by the Hedera network as expired anyway, so the substitution produces a usable transaction — but the caller is never informed of the substitution, which is the integrity failure.

## Likelihood Explanation
No authentication or privilege is required. The `/construction/payloads` endpoint is reachable by any Rosetta API client. The trigger is a single field set to the string `"0"`. It is deterministically reproducible on every invocation.

## Recommendation
Change the boundary check in `getIntMetadataValue` from `metadataValue < 0` to `metadataValue <= 0`:

```go
// rosetta/app/services/construction_service.go, line 549
if metadataValue, err = tools.ToInt64(value); err != nil || metadataValue <= 0 {
    return metadataValue, errors.ErrInvalidArgument
}
```

This aligns the validation with the sentinel semantics in `transactionSetTransactionId` and matches the existing guard already present in the `valid_until_nanos` branch at line 408. Alternatively, add an explicit post-call check in `getTransactionTimestampProperty` after line 424 mirroring the guard at lines 408–412.

## Proof of Concept
1. Send a POST to `/construction/payloads` with metadata containing `"valid_start_nanos": "0"` alongside valid `valid_duration_seconds` and `node_account_id`.
2. Observe that the call succeeds (no error returned).
3. Decode the returned unsigned transaction bytes and inspect the transaction ID.
4. The transaction ID will be `payer@<current_server_time_nanos>`, not `payer@0.000000000`.
5. Any pre-computed reference to `payer@0.000000000` will fail to match the actual transaction ID in mirror-node records.
