Audit Report

## Title
Empty Transaction Hash Bypasses Validation and Triggers Unnecessary DB Query in `BlockTransaction()`

## Summary
Submitting `"0x"` as `TransactionIdentifier.Hash` to the `/block/transaction` endpoint causes `hex.DecodeString("")` to succeed silently (returning `[]byte{}`, `nil`), bypassing the `ErrInvalidTransactionIdentifier` guard and issuing a live DB query with an empty `bytea` parameter. The intended fast-fail path is never reached.

## Finding Description
The code path is fully confirmed in the repository:

**`rosetta/app/services/block_service.go`, lines 87–92** — `request.TransactionIdentifier.Hash` is forwarded verbatim to `FindByHashInBlock()`. Only the block hash receives `SafeRemoveHexPrefix` (line 81); the transaction hash does not. [1](#0-0) 

**`rosetta/app/tools/hex.go`, lines 18–22** — `SafeRemoveHexPrefix("0x")` returns `string[2:]`, which is `""` for the input `"0x"`. [2](#0-1) 

**`rosetta/app/persistence/transaction.go`, line 180** — `hex.DecodeString(tools.SafeRemoveHexPrefix(hashStr))` is called. With input `"0x"`, this becomes `hex.DecodeString("")`, which the Go standard library resolves to `([]byte{}, nil)` — no error. [3](#0-2) 

Because `err == nil`, the guard at lines 181–183 is never entered and `ErrInvalidTransactionIdentifier` is never returned. Execution continues to the ORM query at lines 188–193, which executes `selectTransactionsByHashInTimestampRange` with `sql.Named("hash", []byte{})` — a real DB round-trip with an empty `bytea` hash. [4](#0-3) 

The query returns zero rows, and `ErrTransactionNotFound` is returned at line 199.

**Root cause:** The developer assumed `hex.DecodeString` would reject a bare `"0x"` prefix with no hex digits. The Go standard library treats an empty string as valid hex, returning an empty byte slice without error. There is no minimum-length check on `transactionHash` before the query is issued.

## Impact Explanation
Every request with `"0x"` as the transaction hash causes a real database query against the `transaction` table (bounded by the block's timestamp range). The intended fast-fail path is skipped entirely. An unauthenticated caller can flood the `/block/transaction` endpoint with such requests to amplify DB load. There is no data exfiltration, no state mutation, and no privilege escalation; the impact is availability degradation / griefing.

## Likelihood Explanation
The `/block/transaction` endpoint is unauthenticated and publicly reachable in any standard Rosetta deployment. The payload is trivial: a valid block identifier plus `"0x"` as the transaction hash. The attack is fully repeatable and requires no special knowledge or privilege.

## Recommendation
Add a minimum-length check on `transactionHash` immediately after decoding, before the DB query is issued. In `FindByHashInBlock()` (`rosetta/app/persistence/transaction.go`), after line 180, add:

```go
if len(transactionHash) == 0 {
    return nil, hErrors.ErrInvalidTransactionIdentifier
}
```

This ensures that a bare `"0x"` (or any other input that decodes to zero bytes) is rejected before any DB interaction occurs.

## Proof of Concept
```
POST /block/transaction
{
  "network_identifier": { ... },
  "block_identifier": { "index": 1, "hash": "<valid block hash>" },
  "transaction_identifier": { "hash": "0x" }
}
```
Repeat indefinitely. Each request bypasses the `ErrInvalidTransactionIdentifier` guard and issues a live DB query, returning `ErrTransactionNotFound` after the round-trip completes.
