# Audit Report

## Title
Aborted HTTP Connections Permanently Inflate `api_all_request_in_processing_total` In-Flight Counter

## Summary
In `metricsMiddleware()` (`rest/middleware/metricsHandler.js`), `inFlightCounter.add(1)` is called synchronously on every non-metrics request, but the corresponding `inFlightCounter.add(-1)` is registered only inside a `res.on('finish')` handler. When a client aborts a TCP connection before the server sends a response, Node.js destroys the underlying socket, which causes `http.ServerResponse` to emit `'close'` — not `'finish'`. Because no `'close'` handler exists, the decrement never executes, permanently inflating the counter by +1 per aborted request for the lifetime of the process.

## Finding Description
**File:** `rest/middleware/metricsHandler.js`, function `metricsMiddleware()`

**Increment — always fires (line 169):**
```js
inFlightCounter.add(1);
```

**Decrement — only fires on `'finish'` (lines 185–186), never on `'close'`:**
```js
res.on('finish', () => {
  inFlightCounter.add(-1);
  ...
});
```

There is no `res.on('close', ...)` handler anywhere in the file.

**Root cause — failed assumption:** The code assumes `'finish'` is the terminal event for every HTTP response. This is incorrect. Per the Node.js stream contract:
- `'finish'` fires only after `res.end()` is called and all data is flushed to the OS buffer.
- `'close'` fires when the underlying socket is destroyed, including client-initiated aborts.

When a client sends a TCP RST or closes the socket before the server calls `res.end()`, the socket is destroyed, `ServerResponse` is destroyed, and `'close'` is emitted — `'finish'` is **never** emitted.

The `res.end` override at lines 180–183 only accumulates `responseSize`; it does not call `inFlightCounter.add(-1)` and is itself never invoked on an aborted connection.

## Impact Explanation
Every aborted request leaves `api_all_request_in_processing_total` permanently incremented by +1 for the lifetime of the process. The Grafana dashboard panel "Requests Processing" queries this metric directly (`sum(api_all_request_in_processing_total{...})`) and is configured to turn red at a threshold of 50. An attacker sending 50 aborted requests silently trips the alert threshold, masking the true concurrency level and potentially causing operators to ignore genuine overload conditions or take incorrect remediation actions (e.g., scaling down). The counter only resets on process restart.

## Likelihood Explanation
No authentication, rate limiting, or IP restriction is applied to any non-metrics endpoint before `metricsMiddleware` runs. Any unprivileged user with network access to the REST API can exploit this. The technique (send request, immediately close socket) is trivially scriptable. It is repeatable indefinitely and leaves no error in server logs because the server never errors — it simply never finishes the response.

## Recommendation
Add a `res.on('close')` handler that decrements the counter, guarded against double-decrement in case `'finish'` already fired:

```js
let decremented = false;
const decrement = () => {
  if (!decremented) {
    decremented = true;
    inFlightCounter.add(-1);
  }
};

res.on('finish', () => {
  decrement();
  // ... rest of metrics recording
});

res.on('close', () => {
  decrement();
});
```

This ensures the counter is decremented exactly once regardless of whether the connection completes normally or is aborted by the client.

## Proof of Concept
```bash
# Each invocation aborts immediately, incrementing the counter by +1 without decrement
for i in $(seq 1 50); do
  curl --max-time 0 http://<host>/api/v1/transactions &
done
# After 50 iterations, api_all_request_in_processing_total == 50
# Grafana "Requests Processing" panel turns red
```
