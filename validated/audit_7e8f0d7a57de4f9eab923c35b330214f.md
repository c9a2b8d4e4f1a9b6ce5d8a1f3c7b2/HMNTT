Audit Report

## Title
Unauthenticated DB Connection Pool Exhaustion via Concurrent EVM Address Lookups in `getAccountCryptoAllowances`

## Summary
The `/accounts/:idOrAliasOrEvmAddress/allowances/crypto` endpoint unconditionally issues a live database query for every EVM-address-style path parameter, with no result caching. Combined with a default connection pool of 10 and no application-level rate limiting, a single unauthenticated client can exhaust the pool and deny service to all API consumers.

## Finding Description

**Code path — verified line by line:**

`rest/controllers/cryptoAllowanceController.js` line 77 calls `EntityService.getEncodedId()` on every inbound request: [1](#0-0) 

`rest/service/entityService.js` lines 118–124: when the parsed `EntityId` has a non-null `evmAddress`, `getEncodedId()` always delegates to `getEntityIdFromEvmAddress()`: [2](#0-1) 

`rest/service/entityService.js` lines 90–91: `getEntityIdFromEvmAddress()` unconditionally executes `entityFromEvmAddressQuery` against the database on every call — no cache, no guard: [3](#0-2) 

**Failed assumption — the `quickLru` cache does NOT protect this path.** The `parseCached` function in `rest/entityId.js` (lines 314–332) caches only the *parsing* of the address string into an `EntityId` object (key = `${id}_${allowEvmAddress}_${evmAddressType}`). It stores and returns the `EntityId` struct, not a database result. `getEntityIdFromEvmAddress()` is invoked *after* parsing and always issues a fresh query regardless of whether the same address was seen before: [4](#0-3) 

**Pool size — verified:** `rest/dbpool.js` line 14 sets `max: config.db.pool.maxConnections`, and the documented default for `hiero.mirror.rest.db.pool.maxConnections` is **10**: [5](#0-4) [6](#0-5) 

**No application-level rate limiting — verified:** The full middleware stack registered in `rest/server.js` is: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, `responseCacheUpdateHandler`, `handleError`. No throttle or rate-limit middleware is present: [7](#0-6) 

The throttling found in the codebase (`web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`) applies exclusively to the separate Java `web3` service, not to this Node.js REST API. [8](#0-7) 

## Impact Explanation
With a pool of 10 connections, `connectionTimeoutMillis` of 20,000 ms, and `statement_timeout` of 20,000 ms, an attacker sending ≥10 concurrent requests — each with a distinct syntactically valid EVM address — occupies all pool connections simultaneously. Every subsequent legitimate request blocks for up to 20 seconds waiting for a free connection, then either times out or receives a 503. The attack is repeatable in waves: each wave holds connections for up to 20 seconds, making sustained denial of service trivially achievable with a simple loop.

## Likelihood Explanation
The endpoint is public and unauthenticated. The attack requires only a standard HTTP client capable of concurrent requests. EVM addresses need only be syntactically valid 40-hex-character strings — they do not need to resolve to real entities (a `NotFoundError` is thrown after the query completes, but the connection is already consumed during execution). No special knowledge beyond the public API specification is required.

## Recommendation
1. **Cache DB lookup results for EVM addresses** in `EntityService.getEntityIdFromEvmAddress()` using the existing `quickLru` infrastructure (or a similar TTL cache), keyed on the normalized EVM address string.
2. **Add application-level rate limiting** to the Node.js REST API (e.g., via `express-rate-limit` or an equivalent middleware), particularly for endpoints that trigger database lookups per request.
3. **Consider increasing the default pool size** or deploying a connection pooler (e.g., PgBouncer) in front of the database for the REST API, consistent with the production Helm chart configuration already used for other services. [9](#0-8) 

## Proof of Concept
```bash
# Send 15 concurrent requests with distinct valid EVM addresses
for i in $(seq 1 15); do
  addr=$(printf '%040x' $RANDOM$RANDOM$RANDOM)
  curl -s "http://<host>/api/v1/accounts/0x${addr}/allowances/crypto" &
done
wait
# All 10 pool connections are held; subsequent legitimate requests
# block for up to 20 seconds (connectionTimeoutMillis) then fail.
```

   * Gets the encoded entity id from the evm address.

   * @param {EntityId} entityId
   * @param {Boolean} requireResult

   */
  async getEntityIdFromEvmAddress(entityId, requireResult = true) {
    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();
      }

      return null;
    } else if (rows.length > 1) {
      logger.error(`Incorrect db state: ${rows.length} alive entities matching evm address ${entityId}`);
      throw new Error(EntityService.multipleEvmAddressMatch);
    }

    return rows[0].id;
  }
