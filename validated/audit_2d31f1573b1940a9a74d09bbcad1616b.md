Audit Report

## Title
Retry-Budget Exhaustion via Alternating HTTP 500 / HTTP 200 Empty-Node-List Responses Permanently Halts Pinger Startup

## Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` treats HTTP 5xx as retryable but HTTP 200 with an empty node list as unconditionally non-retryable. An attacker who can control responses from the mirror node REST endpoint can exhaust the full retry budget with HTTP 500 responses, then deliver a final HTTP 200 `{"nodes":[]}`, causing the function to return a non-retryable error that propagates to `log.Fatalf()` in `main()`, permanently terminating the pinger process.

## Finding Description
`buildNetworkFromMirrorNodes()` drives a retry loop with a configurable attempt count (default 11):

```go
attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)
for attempt := 1; attempt <= attempts; attempt++ {
    network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
    if err == nil { return network, nil }
    lastErr = fmt.Errorf("attempt %d/%d: %w", attempt, attempts, err)
    if !retry || attempt == attempts { break }
    ...
}
return nil, lastErr
``` [7](#0-6) 

`fetchMirrorNodeNetwork()` sets the `retry` flag as follows:
- **HTTP 5xx**: `retry = true` [2](#0-1) 
- **HTTP 200 + empty node list**: `retry = false`, error = `"no usable service_endpoints found"` [3](#0-2) 

The break condition `if !retry || attempt == attempts` exits immediately on any non-retryable error regardless of remaining budget. There is no distinction between a transient empty response and a permanent one, and no minimum-node-count retry path exists.

The error propagates through `newClient()` in `sdk_client.go` (only reached when `network == "other"`) [6](#0-5)  and terminates the process via `log.Fatalf` in `main.go`: [4](#0-3) 

## Impact Explanation
The pinger process is killed via `log.Fatalf` and does not self-recover. In a Kubernetes deployment the pod restarts, but the attacker can repeat the attack on every restart cycle, keeping the pinger permanently non-functional. The pinger's purpose is to submit periodic Hiero network transfers as a liveness/health signal; its sustained absence silently degrades observability and health-check coverage for the network.

## Likelihood Explanation
The attack requires the ability to serve controlled HTTP responses from the mirror node REST endpoint. Realistic vectors:

- **DNS hijacking**: if the mirror REST hostname resolves via an attacker-influenced resolver, the attacker can point it at a controlled server. No local network access required.
- **Compromised or misconfigured mirror node**: the mirror node REST API is often publicly exposed; if it can be induced to return empty node lists (e.g., during a maintenance window or via a separate bug), the pinger will not retry.
- **Network-adjacent MITM** (ARP poisoning, DNS spoofing on the same L2 segment): enabled by the plain HTTP example URL (`http://mirror-rest:5551`) in the config. [5](#0-4) 

No TLS enforcement, certificate pinning, or authentication is present in the HTTP client construction: [8](#0-7) 

## Recommendation
1. **Make empty-node-list responses retryable**: change line 128 to return `retry = true` for the empty-node case, treating it as a transient condition identical to a 5xx error.
2. **Enforce HTTPS** for the mirror REST URL and reject plaintext `http://` configurations in `loadConfig()`.
3. **Decouple startup failure from `log.Fatalf`**: instead of fatally exiting, loop with backoff at the `main()` level so a single bad startup sequence does not permanently kill the process without a supervisor restart.

## Proof of Concept
1. Configure pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<controlled-server>`.
2. On the controlled server, serve HTTP 500 for the first 10 requests to `GET /api/v1/network/nodes`.
3. On the 11th request, serve HTTP 200 with body `{"nodes":[]}`.
4. Observe: `buildNetworkFromMirrorNodes` returns `"no usable service_endpoints found"`, `newClient` returns the error, `main` calls `log.Fatalf("client error: %v", err)`, and the process exits with status 1.
