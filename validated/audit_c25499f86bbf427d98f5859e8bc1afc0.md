Audit Report

## Title
Unbounded `CaffeineCacheManager` Internal Cache Map Growth via Distinct `contractId` Requests Leads to Memory Exhaustion

## Summary
`cacheManagerSlotsPerContract` is configured as a `CaffeineCacheManager` without `setCacheNames()`, enabling dynamic cache creation mode. Every call to `getCache(contractId.toString())` for a previously unseen contract ID permanently inserts a new `CaffeineCache` into the manager's internal `ConcurrentHashMap<String, Cache>`. The level-1 `contractSlotsCache` (bounded at 3000 entries) evicts contract IDs but never removes the corresponding entries from `cacheManagerSlotsPerContract`'s internal map, resulting in unbounded heap growth exploitable via public JSON-RPC endpoints.

## Finding Description

**Root cause — `EvmConfiguration.java` lines 107–112:**

`cacheManagerSlotsPerContract` is the only `CaffeineCacheManager` bean in `EvmConfiguration.java` that omits `setCacheNames()`:

```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    return caffeineCacheManager;  // NO setCacheNames() → dynamic mode
}
``` [1](#0-0) 

Every other bean (e.g., `cacheManagerContract`, `cacheManagerContractSlots`, `cacheManagerEntity`, etc.) calls `setCacheNames(Set.of(...))`, which locks the manager to a fixed set of named caches and disables dynamic creation. [2](#0-1) 

In Spring's `CaffeineCacheManager`, omitting `setCacheNames()` enables dynamic mode: every `getCache(name)` call with a new name creates a `CaffeineCache` and stores it permanently in the manager's internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no size limit and no eviction.

**Trigger — `ContractStateServiceImpl.java` lines 86–87:**

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [3](#0-2) 

For each `contractId` not present in the level-1 `contractSlotsCache`, the lambda fires and calls `cacheManagerSlotsPerContract.getCache(contractId.toString())`, inserting a new `CaffeineCache` into `cacheMap` under the key `contractId.toString()`.

**Why the level-1 guard is insufficient:**

The level-1 `contractSlotsCache` is configured with `maximumSize=3000`: [4](#0-3) 

When it evicts a `contractId`, the `CaffeineCache` value is removed from the level-1 store. However, `cacheManagerSlotsPerContract`'s internal `cacheMap` retains its own reference to that same `CaffeineCache` under `contractId.toString()` — it is never notified of the eviction and never removes the entry. The two structures are completely decoupled with respect to eviction.

**Default enablement:**

The vulnerable code path is active by default: [5](#0-4) 

Each per-contract cache is configured with `maximumSize=1500`: [6](#0-5) 

## Impact Explanation
Each `CaffeineCache` wrapping a `BoundedLocalCache` with `maximumSize=1500` allocates internal data structures even when empty. With N distinct contract IDs, `cacheMap` holds N permanent `CaffeineCache` instances with no recovery path short of a JVM restart. Sustained growth leads to `OutOfMemoryError`, crashing the web3 service. Since mirror nodes are stateless replicas, an attacker targeting multiple nodes can degrade or eliminate JSON-RPC processing capacity across the cluster.

## Likelihood Explanation
The attack requires no authentication, no privileged access, and no on-chain transactions. The Hedera entity ID space is 64-bit, providing billions of distinct contract IDs. An attacker needs only an HTTP client and a list of distinct Ethereum-format addresses to call `eth_call` or `eth_estimateGas`. The `enableBatchContractSlotCaching` flag defaults to `true`, so the vulnerable code path is active in all default deployments. Rate limiting, if present, can be bypassed by distributing requests over time or source IPs.

## Recommendation
The core fix is to ensure that eviction from the level-1 `contractSlotsCache` also removes the corresponding entry from `cacheManagerSlotsPerContract`'s internal map. Options:

1. **Add a Caffeine removal listener** to the level-1 `contractSlotsCache` that calls `cacheManagerSlotsPerContract.getCacheNames()` / invalidation logic when a `contractId` is evicted, keeping the two structures in sync.
2. **Eliminate the two-level structure** entirely: replace the per-contract `CaffeineCache` lookup with a single bounded cache keyed by `(contractId, slotKey)` composite keys, removing the need for `cacheManagerSlotsPerContract` in dynamic mode.
3. **Use a `LoadingCache<EntityId, Cache>`** (Caffeine-native) with a configured `maximumSize` and `removalListener` that explicitly invalidates the per-contract sub-cache on eviction, rather than delegating to `CaffeineCacheManager`'s unmanaged `cacheMap`.

Option 2 is the simplest and eliminates the architectural mismatch entirely.

## Proof of Concept
```
for i in $(seq 1 100000); do
  ADDR=$(printf "0x%040x" $i)
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"data\":\"0x\",\"to\":\"$ADDR\",\"estimate\":false}" &
done
wait
```
Each request with a distinct `$ADDR` maps to a distinct `contractId`, triggering `cacheManagerSlotsPerContract.getCache(contractId.toString())` once per address. After 3000 requests the level-1 cache begins evicting, but `cacheMap` continues to grow. JVM heap exhaustion follows as N scales.
