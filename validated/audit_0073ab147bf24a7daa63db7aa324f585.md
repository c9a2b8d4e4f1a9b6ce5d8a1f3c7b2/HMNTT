# Audit Report

## Title
Silent `BlockType.LATEST` Fallback in Opcode Replay Produces Incorrect Historical Execution Environment

## Summary
`OpcodeServiceImpl.buildCallServiceParameters()` silently falls back to `BlockType.LATEST` when `recordFileService.findByTimestamp()` returns empty. Because `ContractCallContext.useHistorical()` gates all historical state reads on `block != BlockType.LATEST`, this fallback causes the entire EVM replay to execute against the current chain state rather than the state at the time of the original transaction, producing a fabricated opcode trace.

## Finding Description

In `OpcodeServiceImpl.java` lines 158–161, when no record file is found for the given consensus timestamp, the code silently assigns `BlockType.LATEST`:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);   // ← silent fallback
``` [1](#0-0) 

This `blockType` is passed into `ContractDebugParameters.block`. In `ContractDebugService.processOpcodeCall()`, the context timestamp is correctly pre-set to `consensusTimestamp - 1`:

```java
ctx.setTimestamp(Optional.of(params.getConsensusTimestamp() - 1));
``` [5](#0-4) 

However, the immediately following `callContract(params, ctx)` call unconditionally overwrites `callServiceParameters` in the context:

```java
ctx.setCallServiceParameters(params);  // params.getBlock() == LATEST
``` [6](#0-5) 

After this write, `useHistorical()` evaluates to `false`:

```java
public boolean useHistorical() {
    return callServiceParameters != null && callServiceParameters.getBlock() != BlockType.LATEST;
}
``` [7](#0-6) 

And `getTimestamp()` short-circuits to `Optional.empty()` regardless of the previously set `timestamp` field:

```java
public Optional<Long> getTimestamp() {
    if (useHistorical()) {
        return getTimestampOrDefaultFromRecordFile();
    }
    return Optional.empty();   // ← always reached when block == LATEST
}
``` [8](#0-7) 

Every KV-state accessor (contract storage, entities, balances, bytecode) branches on `ctx.getTimestamp()`: non-empty triggers historical queries; empty triggers latest queries. The explicitly set `consensusTimestamp - 1` is silently discarded.

The only guard before reaching this code is `contractResultRepository.findById(consensusTimestamp)`: [9](#0-8) 

This confirms the contract result exists but says nothing about whether the corresponding record file still exists — the two tables can have different retention/pruning policies.

## Impact Explanation
When the fallback fires, the opcode trace returned to the caller reflects the **current** EVM state (latest storage slots, latest account balances, latest bytecode) rather than the state at the time of the original transaction. A contract whose storage or code changed after the original transaction will produce a completely different execution path, different `SLOAD`/`SSTORE` values, and potentially a different success/failure outcome. This corrupts the forensic value of the `debug_traceTransaction`-equivalent endpoint and can be used to produce a trace that misrepresents what actually happened on-chain.

## Likelihood Explanation
An unprivileged user needs only:
1. A valid transaction hash or ID for a transaction whose record file has been pruned but whose `contract_result` row still exists (possible when retention policies differ between tables, or when the importer prunes `record_file` rows more aggressively than `contract_result` rows).
2. A standard HTTP GET to `/api/v1/contracts/results/{transactionIdOrHash}/opcodes`.

No elevated privileges, no special tokens, and no on-chain interaction are required. The condition is uncommon in default deployments but is realistic in long-running mirror nodes with storage-saving pruning enabled.

## Recommendation
Replace the silent `.orElse(BlockType.LATEST)` fallback with an explicit error:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElseThrow(() -> new EntityNotFoundException(
                "Record file not found for consensus timestamp: " + consensusTimestamp));
```

This ensures that if the record file required for accurate historical replay is unavailable (e.g., pruned), the request fails with a clear error rather than silently replaying against the wrong state. Additionally, consider adding a check that validates record file availability immediately after the `contractResultRepository.findById()` guard, so the failure is surfaced as early as possible. [10](#0-9) 

## Proof of Concept
1. Deploy a mirror node with a pruning policy that removes `record_file` rows after N days but retains `contract_result` rows longer.
2. Wait until a historical contract transaction's record file is pruned while its `contract_result` row survives.
3. Modify the contract's storage or redeploy it so its on-chain state differs from what it was at the time of the original transaction.
4. Issue `GET /api/v1/contracts/results/{transactionHash}/opcodes`.
5. Observe that the returned opcode trace reflects the **current** storage/bytecode state, not the historical state — `SLOAD` opcodes return current slot values, and execution may follow a completely different code path than the original transaction.
