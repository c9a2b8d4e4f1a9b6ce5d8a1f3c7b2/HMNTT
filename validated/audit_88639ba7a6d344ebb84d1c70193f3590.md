**Audit Report**

## Title
URL Encoding Cache Key Collision — Cache Bypass and Stale Data Inconsistency via `cacheKeyGenerator()`

## Summary
`cacheKeyGenerator()` in `rest/middleware/responseCacheHandler.js` hashes `req.originalUrl` directly with MD5 without any URL normalization or percent-decoding. Logically identical requests with different percent-encodings produce distinct MD5 hashes and therefore distinct Redis cache keys, enabling cache bypass, Redis pollution, and stale data inconsistency.

## Finding Description
The exact code path is confirmed at `rest/middleware/responseCacheHandler.js`, lines 151–153:

```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
``` [1](#0-0) 

The comment at line 149 explicitly acknowledges the missing normalization, confirming this is a known gap:
> `path?query - In the future, this will utilize Edwin's request normalizer (9113).` [2](#0-1) 

`responseCacheCheckHandler` calls `cacheKeyGenerator(req)` at line 42 with the raw request object, with no normalization applied beforehand: [3](#0-2) 

A `requestNormalizer.js` middleware does exist and performs query parameter sorting and default injection, but it operates on `req.query` (the parsed query object) and reconstructs a normalized string — it does **not** modify `req.originalUrl`, which remains the raw string Express received: [4](#0-3) 

Similarly, `requestQueryParser` in `requestHandler.js` lowercases keys and canonicalizes some values, but again only affects `req.query`, not `req.originalUrl`: [5](#0-4) 

**Root cause:** `cacheKeyGenerator` assumes `req.originalUrl` is already in canonical form. This assumption is false: HTTP allows any unreserved character to be percent-encoded, and `+` vs `%20` for spaces are both legal and semantically equivalent. Express's query parser normalizes these when populating `req.query`, but `req.originalUrl` is never touched.

## Impact Explanation
- **Stale/inconsistent records:** Two users querying the same logical resource with different percent-encodings receive responses from independent cache entries with independent TTLs and independent creation timestamps. If the underlying data changes between the two cache-population events, users observe data of different ages for the same resource.
- **Cache bypass / DB amplification:** Any client can force every request to be a cache miss by percent-encoding one character in the URL (e.g., `limit=%31` instead of `limit=1`), causing all such requests to hit the database directly. This degrades mirror node availability under load.
- **Redis cache pollution:** Each encoding variant creates a separate Redis entry. The default TTL is 1 second (`DEFAULT_REDIS_EXPIRY = 1`), which limits persistence of individual entries, but an automated attacker can continuously generate new variants to sustain the pollution. [6](#0-5) 

## Likelihood Explanation
No authentication or special privilege is required. Any HTTP client can percent-encode query parameter characters. The encoding variants are trivially generated (encode any digit or letter in a parameter value). The attack is fully repeatable and automatable. The code comment at line 149 confirms the developers are aware of the missing normalization, meaning it has been present since the cache was introduced.

## Recommendation
Before hashing in `cacheKeyGenerator`, normalize `req.originalUrl` by:
1. Percent-decoding the query string (`decodeURIComponent` on each parameter value).
2. Lowercasing parameter keys (already done in `requestQueryParser` for `req.query`).
3. Sorting query parameters canonically (already implemented in `normalizeRequestQueryParams` for `req.query`).

The existing `normalizeRequestQueryParams` function in `rest/middleware/requestNormalizer.js` already performs steps 2 and 3 on `req.query`. The intended fix is referenced in the comment itself (issue 9113 — "Edwin's request normalizer"). The fix should reconstruct the canonical URL from the already-normalized `req.query` (post-`requestQueryParser` and post-`normalizeRequestQueryParams`) rather than using `req.originalUrl`. [4](#0-3) 

## Proof of Concept
```
# Step 1: Populate cache under key K1
GET /api/v1/accounts?account.id=0.0.18&limit=1
# → cache miss → DB hit → stored under MD5("/api/v1/accounts?account.id=0.0.18&limit=1") + "-v1"

# Step 2: Same logical request, different encoding → different cache key K2
GET /api/v1/accounts?account.id=0.0.%31%38&limit=%31
# → cache miss (K2 ≠ K1) → second DB hit → stored under MD5("/api/v1/accounts?account.id=0.0.%31%38&limit=%31") + "-v1"

# Step 3: Repeat step 2 with a new variant on every request to ensure perpetual cache misses
GET /api/v1/accounts?account.id=0.0.%31%38&limit=%3%31
GET /api/v1/accounts?account.id=0.0.%31%38&lim%69t=1
# Each produces a unique MD5 hash → perpetual cache bypass
```

const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
};

const requestQueryParser = (queryString) => {
  const merge = (current, next) => {
    if (!Array.isArray(current)) {
      current = [current];
    }

    if (Array.isArray(next)) {
      current.push(...next);
    } else {
      current.push(next);
    }

    return current;
  };

  // parse first to benefit from qs query handling
  const parsedQueryString = qs.parse(queryString, queryOptions);

  const caseInsensitiveQueryString = {};
  for (const [key, value] of Object.entries(parsedQueryString)) {
    const lowerKey = key.toLowerCase();
    const canonicalValue = canonicalizeValue(lowerKey, value);
    if (lowerKey in caseInsensitiveQueryString) {
      // handle repeated values, merge into an array
      caseInsensitiveQueryString[lowerKey] = merge(caseInsensitiveQueryString[lowerKey], canonicalValue);
    } else {
      caseInsensitiveQueryString[lowerKey] = canonicalValue;
    }
  }

  return caseInsensitiveQueryString;
};
