# Audit Report

## Title
Timeout Bypass via Exception Swallowing in Binary Gas Estimation

## Summary
`BinaryGasEstimator.safeCall()` catches all `Exception` subtypes, including `QueryTimeoutException` thrown by the Hibernate `StatementInspector` timeout guard. This allows the binary search loop to silently continue after the timeout fires, running up to `maxGasEstimateRetriesCount` (default 20) iterations regardless of the configured `requestTimeout`. Any unprivileged caller can trigger this via `POST /api/v1/contracts/call` with `"estimate": true`.

## Finding Description

**Code path:**

- `ContractExecutionService.processCall()` → `estimateGas()` → `binaryGasEstimator.search()` [7](#0-6) 
- `estimateGas()` invokes `binaryGasEstimator.search(...)` with `gasUsedByInitialCall` as lower bound [8](#0-7) 
- `search()` calls `contractCallContext.reset()` then `safeCall(mid, call)` per iteration [9](#0-8) 
- `safeCall()` wraps the EVM call in `catch (Exception ignored)`, returning `null` on any exception [1](#0-0) 
- `HibernateConfiguration.statementInspector()` throws `QueryTimeoutException` (a `RuntimeException`) when `elapsed >= timeout` [10](#0-9) 

**Root cause:** `ContractCallContext.startTime` is `final` and set once at context creation. [3](#0-2)  `reset()` only clears `writeCache` and does not reset `startTime`. [4](#0-3)  Once the timeout fires, `elapsed >= timeout` remains true for all subsequent iterations, so every DB query in every remaining iteration immediately throws `QueryTimeoutException`. `safeCall`'s `catch (Exception ignored)` intercepts each one, logs a benign info message, and returns `null`. The loop treats `null` as `err=true` and continues. The timeout never propagates out of `search()`.

**Failed assumption:** The `StatementInspector` timeout guard assumes its exception will terminate the in-flight request. Instead, `safeCall` unconditionally swallows it, and there is no outer deadline check in `search()` itself.

## Impact Explanation

The timeout mechanism is completely neutralized for gas estimation requests. The binary search loop always runs to its full `maxGasEstimateRetriesCount` (default 20) iteration limit regardless of how long the request has been running. Each iteration still attempts DB queries (which immediately fail after the first timeout fires), holding the server thread for the full loop duration. An attacker sending many concurrent `eth_estimateGas` requests can exhaust the thread pool and degrade service for all users. The `gasPerSecond` throttle (`ThrottleProperties`) limits gas throughput but does not bound wall-clock execution time per request and does not mitigate this.

## Likelihood Explanation

The endpoint is public and unauthenticated. No special privileges, tokens, or on-chain assets are required. A single script sending concurrent `POST /api/v1/contracts/call` requests with `"estimate": true` targeting a historical block is sufficient to trigger the condition repeatedly.

## Recommendation

1. **Re-throw timeout exceptions in `safeCall`:** Distinguish between expected call-failure exceptions (e.g., EVM revert) and infrastructure exceptions (e.g., `QueryTimeoutException`). Only catch the former; let the latter propagate:
   ```java
   private EvmTransactionResult safeCall(long mid, LongFunction<EvmTransactionResult> call) {
       try {
           return call.apply(mid);
       } catch (QueryTimeoutException e) {
           throw e; // let timeout terminate the loop
       } catch (Exception ignored) {
           log.info("Exception while calling contract for gas estimation");
           return null;
       }
   }
   ```
2. **Add a deadline check inside `search()`:** At the top of the `while` loop, check `ContractCallContext.get().getStartTime()` against the configured timeout and break/throw if exceeded.
3. **Consider resetting `startTime` on `reset()`** only if per-iteration timeouts are the intended design; otherwise keep it as a request-scoped deadline (which is the correct model).

## Proof of Concept

```
POST /api/v1/contracts/call
Content-Type: application/json

{
  "data": "<calldata for an expensive contract function>",
  "to": "<contract address>",
  "estimate": true,
  "block": "<historical block number>"
}
```

1. The initial `callContract()` call succeeds within the timeout window.
2. `binaryGasEstimator.search()` begins; the first iteration that triggers a DB query after 10 s causes `HibernateConfiguration.statementInspector()` to throw `QueryTimeoutException`.
3. `safeCall` catches it, returns `null`; the loop sets `err=true` and continues.
4. All remaining iterations (up to 19 more) immediately throw `QueryTimeoutException` on their first DB query (since `startTime` is not reset and `elapsed >= timeout`); each is caught and swallowed.
5. The loop runs all 20 iterations and returns normally. The timeout never terminates the request.
6. Sending many such requests concurrently exhausts server threads and degrades availability.
