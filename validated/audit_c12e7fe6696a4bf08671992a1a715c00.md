## Audit Report

## Title
Unbounded Throttled Polling Loop Enables Unauthenticated DB Connection Pool Exhaustion DoS

## Summary
Any unauthenticated client can open an unlimited number of concurrent `subscribeTopic` gRPC subscriptions. Each subscription unconditionally enters an infinite polling loop against the database (via `PollingTopicMessageRetriever.retrieve(filter, true)`), with no global subscriber cap enforced. The only termination guard (`isComplete()`) fails to fire on a live node with continuous message ingestion, and the per-emission timeout does not bound total subscription lifetime. Flooding the server with such subscriptions exhausts the HikariCP database connection pool, causing a service-wide DoS.

## Finding Description

**Code path:**
`ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicMessageRetriever.retrieve(filter, true)` → `PollingContext(filter, throttled=true)`.

**1. Unbounded repeat loop:**
In `PollingContext` constructor, when `throttled=true`, `numRepeats` is unconditionally set to `Long.MAX_VALUE`: [1](#0-0) 

The repeat loop in `retrieve()` runs up to `Long.MAX_VALUE` iterations, terminating only when `isComplete()` returns `true`: [2](#0-1) 

**2. Failed termination assumption — `isComplete()` never fires on a live node:**
For the throttled path, `isComplete()` returns `true` only when the last DB page returned fewer rows than `maxPageSize` (default 1000), or a limit is hit: [3](#0-2) 

On a live mirror node with continuous message ingestion, every poll returns a full page of 1000 messages, so `isComplete()` never returns `true`. The loop runs indefinitely, polling every 2 seconds (default `pollingFrequency`): [4](#0-3) 

**3. Timeout is insufficient:**
The timeout applied is Reactor's per-emission idle timeout (default 60s), not a total subscription lifetime cap: [5](#0-4) 

Since each poll emits up to 1000 messages every 2 seconds, items flow continuously and the idle timeout never fires.

**4. No authentication or global subscriber cap:**
`ConsensusController.subscribeTopic()` has no authentication check: [6](#0-5) 

`subscriberCount` is a Micrometer Gauge metric only, never enforced as a limit: [7](#0-6) 

**5. Per-connection limit is bypassable:**
`maxConcurrentCallsPerConnection = 5` is per-connection: [8](#0-7) [9](#0-8) 

An attacker opens N TCP connections × 5 calls each = 5N concurrent indefinitely-polling subscriptions.

**6. Each poll acquires a HikariCP JDBC connection:**
`findByFilter()` is annotated `@Transactional(readOnly = true)` and uses JPA/Hibernate backed by HikariCP: [10](#0-9) [11](#0-10) 

With many concurrent subscriptions each polling every 2 seconds, the HikariCP pool is saturated. The Grafana alert rule `GrpcHighDBConnections` confirms the project already monitors for this condition: [12](#0-11) 

## Impact Explanation
Exhausting the HikariCP database connection pool blocks all other gRPC subscribers and internal services (importer, REST API) from acquiring DB connections, causing a complete service outage. The mirror node is critical infrastructure for the Hedera ecosystem; its unavailability affects all downstream consumers of topic messages, account queries, and transaction data.

## Likelihood Explanation
The attack requires zero privileges — the gRPC `subscribeTopic` endpoint is publicly accessible with no authentication. The attacker needs only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK), a topic ID with historical messages (publicly known on mainnet), and the ability to open multiple TCP connections. The attack is trivially scriptable. On mainnet, topics like the address book topic have millions of messages, guaranteeing full pages on every poll and ensuring `isComplete()` never fires.

## Recommendation
1. **Enforce a global subscriber cap**: Convert `subscriberCount` from a metric-only gauge to an enforced limit in `TopicMessageServiceImpl.subscribeTopic()`, rejecting new subscriptions when the cap is reached.
2. **Add a total subscription lifetime timeout**: Apply a wall-clock timeout (e.g., via `.take(Duration)`) in addition to the per-emission idle timeout, to bound the maximum lifetime of any single subscription.
3. **Add rate limiting / authentication**: Require authentication or apply per-IP rate limiting at the gRPC layer (e.g., via an interceptor) to prevent a single client from opening unbounded connections.
4. **Bound the throttled polling loop**: For the throttled path, consider a configurable `maxPolls` analogous to the unthrottled path's `maxPolls = 12`, or require clients to set a `consensusEndTime` for long-running subscriptions.

        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
