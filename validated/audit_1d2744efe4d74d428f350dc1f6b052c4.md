# Audit Report

## Title
Deleted Entity Disclosure via Long-Zero EVM Address Bypassing `deleted is not true` Filter

## Summary
`EntityServiceImpl.getByEvmAddressAndType()` routes long-zero EVM addresses (bytes 0–11 all zero, bytes 12–19 = numeric entity ID) to `entityRepository.findById()`, which carries no deletion filter. The `findByEvmAddress()` path explicitly enforces `deleted is not true`, but the `findById()` fallback does not, allowing any caller to retrieve a deleted entity's full record by supplying a crafted long-zero address.

## Finding Description
In `EntityServiceImpl.java`, the `getByEvmAddressAndType` method branches on whether the first 12 bytes of the decoded EVM address are zero:

```java
// graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java, lines 34-41
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {
        return entityRepository.findById(buffer.getLong())        // NO deleted check
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes)     // has "deleted is not true"
                           .filter(e -> e.getType() == type);
}
``` [1](#0-0) 

The `findByEvmAddress` query explicitly filters deleted rows:

```java
// graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java, lines 16-17
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
``` [5](#0-4) 

`findById` is Spring Data's generic `CrudRepository.findById(Long)`, which issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate. The `Entity` class has no Hibernate `@Where` annotation to inject that filter automatically. [3](#0-2) 

The only guard on the long-zero branch is `.filter(e -> e.getType() == type)`, which checks entity type but not deletion status.

**Root cause:** Inconsistent deletion filtering — the `findByEvmAddress` path enforces `deleted is not true` in SQL, but the `findById` fallback for long-zero addresses does not.

**Failed assumption:** The developer assumed long-zero addresses always correspond to live system accounts (`0.0.X`), but any numeric entity ID — including IDs of deleted accounts, contracts, or tokens — can be encoded in this format.

## Impact Explanation
An attacker can retrieve the full `Entity` row (memo, keys, auto-renew settings, balance snapshot, `deleted = true` flag, etc.) for any entity that has been deleted from the ledger. Through the GraphQL API, this entity object is the root resolver for related data (e.g., transaction history queries keyed on entity ID). Exposing deleted entities violates the ledger's soft-delete semantics and leaks historical account/contract metadata to unauthenticated callers.

## Likelihood Explanation
Exploitation requires zero privileges — only knowledge of a target entity's numeric ID, which is publicly derivable from any prior transaction referencing that entity. The long-zero address format is documented Hedera behavior (EVM alias for `0.0.<num>`). The attack is trivially repeatable via a single GraphQL query and requires no special tooling.

## Recommendation
Add an explicit deletion check on the `findById` branch. The simplest fix is to add a dedicated repository method with the deletion filter:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Then replace the `findById` call in `getByEvmAddressAndType`:

```java
return entityRepository.findByIdAndNotDeleted(buffer.getLong())
                       .filter(e -> e.getType() == type);
```

The same fix should be applied to `getByIdAndType` at line 25, which has the identical omission. [6](#0-5) 

## Proof of Concept
1. Identify a deleted entity with numeric ID `N` (e.g., `N = 12345`).
2. Construct the long-zero EVM address: 12 zero bytes followed by the 8-byte big-endian encoding of `N` → `0x000000000000000000000000000000000000000000003039` (for N=12345).
3. Send a GraphQL query:
```graphql
{
  account(input: { evmAddress: { address: "0x0000000000000000000000000000000000003039" } }) {
    id
    memo
    deleted
    key
  }
}
```
4. The response returns the full entity record including `deleted: true`, confirming the filter bypass.
