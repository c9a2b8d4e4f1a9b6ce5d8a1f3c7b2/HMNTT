Audit Report

## Title
Redis Failure Causes `TypeError` in `getSingleWithTtl`, Turning All Requests into 500 Errors (Effective DoS)

## Summary
When Redis fails after `this.ready === true`, the `.catch()` handler in `getSingleWithTtl` swallows the Redis error by returning `undefined` (the implicit return of `logger.warn()`). The immediately following unconditional `result[1][1]` access throws a `TypeError`. This propagates through `wrap()` in `extendExpress.js` and is converted to HTTP 500 by `handleError`, causing every API request to fail for the duration of the Redis outage instead of gracefully falling through to the database handler.

## Finding Description

**Verified code path â€” `rest/cache.js`, `getSingleWithTtl()`, lines 64â€“83:**

```js
async getSingleWithTtl(key) {
  if (!this.ready) {          // guard â€” bypassed during race window
    return undefined;
  }

  const result = await this.redis
    .multi()
    .ttl(key)
    .get(key)
    .exec()
    .catch((err) => logger.warn(`Redis error during ttl/get: ${err.message}`));
    //              ^^^ logger.warn() returns void â†’ result === undefined

  const rawValue = result[1][1];   // TypeError: Cannot read properties of undefined
``` [1](#0-0) 

**Root cause:** The `.catch()` at line 74 absorbs the Redis error but returns `undefined` (the return value of `logger.warn()`). `result` is therefore `undefined`. The unconditional property access `result[1][1]` at line 77 throws `TypeError: Cannot read properties of undefined (reading '1')`. There is no null-guard on `result` before this access.

**Why `this.ready` does not fully protect:** `this.ready` is set to `false` inside the `retryStrategy` callback at line 31, which fires asynchronously only when ioredis begins a reconnection attempt. [2](#0-1) 
Between the moment a Redis connection drops and the moment `retryStrategy` fires, `this.ready` remains `true`. Every request in that window passes the guard, issues the pipeline, receives an error, and hits the crash path.

**Propagation through `wrap()`:** `responseCacheCheckHandler` is registered at `server.js` line 97 via `app.useExt()`. [3](#0-2) 
`useExt` wraps every handler with `wrap()` from `extendExpress.js`. `wrap()` catches the `TypeError` at lines 42â€“43 and calls `next(err)`. [4](#0-3) 

**`handleError` maps to HTTP 500:** A `TypeError` does not match `NotFoundError`, `InvalidArgumentError`, `RangeError`, `DbError`, or `HttpError`, so it falls through to `defaultStatusCode = httpStatusCodes.INTERNAL_ERROR`. [5](#0-4) 

## Impact Explanation
Any Redis failure (network partition, Redis restart, connection exhaustion) causes 100% of API requests to return HTTP 500 for the entire duration of the outage. The cache layer was designed to degrade gracefully (return `undefined` on a miss and fall through to the database handler), but the missing null-guard inverts this: instead of a cache miss, every request becomes a hard 500 error. No database or application logic is involved.

## Likelihood Explanation
Non-malicious events (Redis restart, network blip, connection timeout) trigger the same outcome without any attacker involvement, making this a high-likelihood reliability bug. An external attacker can also deliberately trigger it by exhausting the Redis connection pool (e.g., opening many TCP connections to the Redis port), requiring no authentication. The attack is repeatable and sustainable.

## Recommendation
Add a null-guard on `result` immediately after the `.catch()` in `getSingleWithTtl`, consistent with the pattern already used in the `get()` method at line 106:

```js
const result = await this.redis
  .multi()
  .ttl(key)
  .get(key)
  .exec()
  .catch((err) => {
    logger.warn(`Redis error during ttl/get: ${err.message}`);
    return null;  // explicit sentinel
  });

if (!result) {
  return undefined;  // graceful cache-miss on Redis error
}

const rawValue = result[1][1];
``` [6](#0-5) 

Note the existing `get()` method already applies this pattern correctly with `|| new Array(keys.length)` as a fallback, confirming the fix is straightforward and consistent with the codebase's own conventions.

## Proof of Concept
1. Start the mirror node REST service with Redis enabled (`config.redis.enabled = true`, `config.cache.response.enabled = true`).
2. Send any valid API request (e.g., `GET /api/v1/accounts`) â€” it returns 200.
3. Kill or network-partition the Redis instance while the service is running (`this.ready` remains `true` until `retryStrategy` fires).
4. Immediately send another valid API request.
5. **Observed:** HTTP 500 with `Internal Server Error` instead of a normal database-backed response.
6. **Expected:** HTTP 200 with a database-backed response (graceful cache miss).

The TypeError is logged server-side as an unhandled error, confirming the crash path is reached.
