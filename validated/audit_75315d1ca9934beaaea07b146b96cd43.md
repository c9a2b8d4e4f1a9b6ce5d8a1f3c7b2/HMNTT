## Audit Report

## Title
Soft-Deleted Hook Records Returned with `contractId` via Unauthenticated REST API

## Summary
`HookRepository` omits a `deleted = false` predicate on both its derived-query methods, causing `GET /api/v1/accounts/{ownerId}/hooks` to return soft-deleted `Hook` records — including their `contractId` field — to any unauthenticated caller.

## Finding Description

**`HookRepository` — missing `AndDeletedIsFalse`**

Both query methods in `HookRepository` lack the soft-delete filter:

```java
List<Hook> findByOwnerIdAndHookIdIn(long ownerId, Collection<Long> hookIds, Pageable pageable);
List<Hook> findByOwnerIdAndHookIdBetween(long ownerId, long lowerBound, long upperBound, Pageable pageable);
``` [1](#0-0) 

Spring Data JPA derives the SQL solely from the method name, producing `WHERE owner_id = ? AND hook_id BETWEEN ? AND ?` with no `AND deleted = false`. The contrast with `HookStorageRepository`, which correctly appends `AndDeletedIsFalse` to both its methods, confirms this is an omission rather than intentional design: [2](#0-1) 

**`HookServiceImpl.getHooks()` — no post-fetch filtering**

The service calls the unfiltered repository method directly and returns the result without checking the `deleted` flag: [3](#0-2) 

**`AbstractHook` — `contractId` and `deleted` are first-class fields**

Both fields are present on the domain object with no `@JsonIgnore` or similar suppression: [4](#0-3) 

**`HookMapper` — maps all fields with no suppression**

The mapper applies only a timestamp qualifier; no `@Mapping(target = "contractId", ignore = true)` or `@Mapping(target = "deleted", ignore = true)` is present: [5](#0-4) 

**Soft-delete mechanics confirmed**

`EVMHookHandler.processHookDeletion()` sets `deleted = true` and upserts the record; the row is never physically removed: [6](#0-5) 

## Impact Explanation
After an account owner deletes a hook, the `contractId` — identifying the smart contract that implements the hook's executing bytecode — remains retrievable by any unauthenticated caller via `GET /api/v1/accounts/{ownerId}/hooks`. This leaks business-sensitive contract associations (e.g., proprietary hook implementations) that the owner believed were no longer visible. The `deleted` flag itself is also exposed in the response, confirming the hook's lifecycle state to external observers.

## Likelihood Explanation
Preconditions are minimal: the attacker only needs a valid `ownerId`, which is a public account identifier on the ledger. The endpoint is unauthenticated, paginated, and supports range filters (`hook.id=gte:0`), making bulk enumeration trivial. The vulnerability is deterministic — every soft-deleted hook is always returned.

## Recommendation
Add `AndDeletedIsFalse` to both `HookRepository` query methods, mirroring the pattern already used in `HookStorageRepository`:

```java
List<Hook> findByOwnerIdAndHookIdInAndDeletedIsFalse(long ownerId, Collection<Long> hookIds, Pageable pageable);
List<Hook> findByOwnerIdAndHookIdBetweenAndDeletedIsFalse(long ownerId, long lowerBound, long upperBound, Pageable pageable);
``` [7](#0-6) 

Update `HookServiceImpl` call sites accordingly. Optionally, add a `@Where(clause = "deleted = false")` annotation on the `Hook` entity as a defence-in-depth measure.

## Proof of Concept
1. Create a hook for account `0.0.12345` — note the returned `contractId` (e.g., `0.0.9999`).
2. Delete the hook (triggering `EVMHookHandler.processHookDeletion()`, which sets `deleted = true`).
3. Without any credentials, query:
   ```
   GET /api/v1/accounts/0.0.12345/hooks?hook.id=gte:0
   ```
4. The response includes the soft-deleted hook record with `contractId: "0.0.9999"` and `deleted: true`, confirming the leak. [8](#0-7)
