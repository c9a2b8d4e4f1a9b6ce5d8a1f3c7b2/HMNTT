Audit Report

## Title
Race Condition in `RedisTopicListener`: `computeIfAbsent` / `remove()` Interleaving Causes Orphaned Flux Instances and Cascading Map Corruption (Resource-Exhaustion DoS)

## Summary
`RedisTopicListener.getSharedListener` uses `ConcurrentHashMap.computeIfAbsent` to cache a shared `Flux` per topic, while `unsubscribe` calls `topicMessages.remove(topic.getTopic())` keyed only on the topic string. A concurrent cancel from one subscriber can race with a new `computeIfAbsent` call, causing the freshly-inserted `Flux` to be immediately evicted. The orphaned `Flux` remains active and, when eventually cancelled, its `doOnCancel` removes whatever entry is *currently* in the map, corrupting the registry for all future subscribers on that topic.

## Finding Description

**Exact code path:**

- `getSharedListener`, line 61: `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))`
- `subscribe`, line 74: `.doOnCancel(() -> unsubscribe(topic))`
- `unsubscribe`, line 83: `topicMessages.remove(topic.getTopic())`

**Flaw 1 — Post-insert removal race.**
`ConcurrentHashMap.computeIfAbsent` holds a per-bin lock while the mapping function executes. A concurrent `remove()` for the same key blocks on that lock, then executes immediately after it is released — removing the value that was just inserted. The caller of `computeIfAbsent` receives the new `Flux F` and subscribes to it, but the map entry is already gone.

**Flaw 2 — Non-instance-specific removal.**
`unsubscribe()` calls `topicMessages.remove(topic.getTopic())` — the single-argument form — which removes *any* entry for that key, not the specific `Flux` instance that was cancelled. An orphaned `Flux` (evicted from the map but still active) will, when its last subscriber cancels, fire `doOnCancel` and remove whatever `Flux G` is currently registered in the map for that topic.

**Note on `.share()` placement:** `doOnCancel` is placed upstream of `.share()` (line 74 vs line 79), so it fires when `.share()` cancels its upstream — i.e., when the refCount drops to zero (last subscriber cancels). This is the correct trigger point, but it does not prevent the race described above.

**Exploit flow:**
```
T1: Flux E in map, last subscriber cancels
    → .share() cancels upstream → doOnCancel fires
    → unsubscribe() → remove("topic.X") [blocks on bin lock]

T2: new subscriber calls computeIfAbsent("topic.X")
    → key absent → calls subscribe() → creates Flux F
    → inserts Flux F → releases lock → returns Flux F to caller

T1 (unblocked): remove("topic.X") executes → removes Flux F from map

T2 caller: subscribes to Flux F (active, receives messages, NOT in map)

T3: next subscriber calls computeIfAbsent("topic.X")
    → key absent → creates Flux G → inserts Flux G

T2 caller cancels: Flux F's doOnCancel fires
    → remove("topic.X") → removes Flux G

... cascade repeats indefinitely
```

## Impact Explanation
Each race cycle leaks one `ReactiveRedisMessageListenerContainer` subscription and one `Flux` pipeline. Because `unsubscribe()` removes the *current* map entry rather than the specific instance, every orphaned `Flux` that is later cancelled removes a legitimately-registered `Flux`, forcing the next subscriber to create yet another new upstream subscription. Under sustained cycling this exhausts Redis connections, thread-pool resources, and heap. The sharing invariant is permanently broken for the affected topic: subscribers that should share one upstream connection each hold independent ones, multiplying load linearly with attacker-controlled cancel/subscribe cycles.

## Likelihood Explanation
No authentication or privilege is required beyond the ability to open a gRPC topic subscription (the normal public API). The race window is narrow on a lightly loaded system but widens under concurrent scheduler threads and GC pauses. An attacker can widen it deliberately by issuing many concurrent subscribe/cancel pairs in a tight loop. The attack is repeatable and automatable with a simple gRPC client.

## Recommendation
Replace the single-argument `remove` with the two-argument form that only removes the entry if the value matches the specific `Flux` instance being cancelled:

```java
// In subscribe(), capture the Flux reference before returning
Flux<TopicMessage> shared = container
    .flatMapMany(...)
    ...
    .share();

// Pass the specific instance to unsubscribe
shared.doOnCancel(() -> topicMessages.remove(topic.getTopic(), shared));
```

This eliminates Flaw 2 entirely. For Flaw 1, consider replacing the `computeIfAbsent` + `remove` pattern with a `compute`-based approach that atomically checks and cleans up stale entries, or use a dedicated lock around the lookup-and-subscribe sequence.
