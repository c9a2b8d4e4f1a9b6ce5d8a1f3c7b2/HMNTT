# Audit Report

## Title
Unbounded Concurrent Subscriptions Exhaust Shared `boundedElastic()` Scheduler in `PollingTopicListener` (POLL Mode DoS)

## Summary
When `hiero.mirror.grpc.listener.type=POLL` is configured, each call to `PollingTopicListener.listen()` schedules perpetual DB polling on the global shared `Schedulers.boundedElastic()` thread pool. Because there is no global subscription cap, no per-IP rate limit, and no authentication on the gRPC endpoint, an unauthenticated attacker can open an unbounded number of connections to saturate the shared thread pool, halting topic message delivery for all subscribers.

## Finding Description

**Verified code path:**

`ConsensusController.subscribeTopic()` (lines 43–53) accepts any unauthenticated request and calls `topicMessageService::subscribeTopic` with no credential check. [1](#0-0) 

`TopicMessageServiceImpl.subscribeTopic()` constructs a live flux via `Flux.defer(() -> incomingMessages(topicContext))`, which calls `topicListener.listen(newFilter)` at line 120 inside `incomingMessages()`. [2](#0-1) [3](#0-2) 

`PollingTopicListener` stores a reference to the **global** shared `Schedulers.boundedElastic()` scheduler as an instance field: [4](#0-3) 

Every `listen()` call:
1. Allocates a new `PollingContext` (line 35).
2. Schedules an initial delay on the shared scheduler via `delaySubscription(interval, scheduler)` (line 39).
3. Schedules a perpetual repeat (`RepeatSpec.times(Long.MAX_VALUE)`) on the same scheduler (lines 40–43).
4. On each tick, executes `poll()` → `Flux.fromStream(topicMessageRepository.findByFilter(...))` — a synchronous JDBC stream — on a scheduler thread, holding that thread for the full DB query duration. [5](#0-4) 

**No global subscription cap exists.** The only server-side guard is `maxConcurrentCallsPerConnection = 5` (default, confirmed in `NettyProperties`), which is a per-connection limit, not a global one. [6](#0-5) 

`GrpcConfiguration`'s `grpcServerConfigurer` bean sets only `executor` and `maxConcurrentCallsPerConnection` — no `maxInboundConnections` or equivalent is configured. [7](#0-6) 

A grep across all Java sources under `grpc/src/main/java/` for `interceptor`, `auth`, `security`, and `authentication` returns no authentication interceptors — only an unrelated match in `TopicMessageFilter.java`. The endpoint is fully unauthenticated.

The same global `boundedElastic()` scheduler is also used by `TopicMessageServiceImpl`'s safety-check path, meaning scheduler exhaustion blocks gap-recovery for all subscribers simultaneously: [8](#0-7) 

**Precondition confirmed:** `POLL` is a non-default but fully documented, supported listener type. The default is `REDIS`. [9](#0-8) 

## Impact Explanation
On a 4-core server, `Schedulers.boundedElastic()` caps at 40 threads with a 100,000-task queue. If each DB poll takes ~100 ms at the default 500 ms interval, the pool sustains roughly 200 concurrent subscriptions before poll tasks begin queuing. An attacker opening hundreds of HTTP/2 connections (each carrying 5 streams) can exceed this threshold trivially. Once the queue fills, `RejectedExecutionException` terminates subscriptions. Between saturation and queue exhaustion, all legitimate subscribers experience growing poll delays — effectively halting real-time topic message delivery. Because the safety-check `Mono.delay` also runs on the same scheduler, gap-recovery for all subscribers is blocked simultaneously, compounding the outage.

## Likelihood Explanation
- **Precondition:** `hiero.mirror.grpc.listener.type=POLL` must be set. This is non-default but is a documented, supported option. Any operator selecting POLL mode is immediately exposed.
- **Attacker capability:** Zero privileges required. The gRPC port (5600) is publicly reachable. A single machine can open hundreds of HTTP/2 connections, each carrying 5 streams, yielding thousands of concurrent subscriptions using a trivial loop with `grpcurl` or `grpc-java`. No special knowledge beyond the public proto definition is needed. The attack is stateless and immediately repeatable.

## Recommendation

1. **Add a global subscription cap** in `TopicMessageServiceImpl` using the existing `subscriberCount` `AtomicLong` (already tracked at lines 89–90). Reject new subscriptions when the count exceeds a configurable threshold. [10](#0-9) 

2. **Configure `maxInboundConnections`** in `GrpcConfiguration`'s `grpcServerConfigurer` bean via `NettyServerBuilder.maxConnectionsTotal()` or equivalent, exposed as a configurable `NettyProperties` field. [7](#0-6) 

3. **Use a dedicated scheduler for `PollingTopicListener`** instead of the global `Schedulers.boundedElastic()`, so polling exhaustion cannot affect the safety-check path or other reactive pipelines. [4](#0-3) 

4. **Document the DoS risk** of POLL mode in operator-facing configuration documentation and recommend connection-level network controls (e.g., firewall rate limiting on port 5600) when POLL mode is used.

## Proof of Concept

```bash
# Requires grpcurl and the mirror node proto definitions
# Opens 200 connections × 5 streams = 1000 concurrent subscriptions

for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}, "consensusStartTime": {"seconds": 0}}' \
      <MIRROR_NODE_HOST>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done

# After saturation: legitimate subscribers will observe no messages delivered.
# Server logs will show poll tasks queuing; eventually RejectedExecutionException.
```

Each background process holds a gRPC stream open indefinitely (no `endTime` set), continuously consuming a `boundedElastic()` thread every 500 ms per subscription. The attack requires no authentication and no prior knowledge beyond the public proto definition.

```java
    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)
                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```
