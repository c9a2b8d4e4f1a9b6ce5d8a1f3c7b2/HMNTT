Audit Report

## Title
Unauthenticated DB Amplification via `findStorageBatch()` Growing Batch Query Bypasses Per-Node Gas Throttle

## Summary
The gas throttle in `ThrottleManagerImpl` is a per-node, in-memory token bucket that limits gas consumption before EVM execution but places no bound on the number of database queries generated per unit of gas consumed. A contract that performs many unique SLOAD operations causes `ContractStateServiceImpl.findStorageBatch()` to issue a growing `findStorageBatch` query on every cache miss — each query including all previously accumulated slot keys (up to 1,500). The shared PostgreSQL backend can be saturated by an unprivileged attacker exploiting this amplification path.

## Finding Description

**Confirmed code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` → `ContractExecutionService.processCall()` → EVM executes SLOADs → `ContractStorageReadableKVState.readFromDataSource()` → `ContractStateServiceImpl.findStorage()` → `ContractStateServiceImpl.findStorageBatch()` → `ContractStateRepository.findStorageBatch()` (uncached DB query).

**Gas throttle is per-node and in-memory:**

`ThrottleManagerImpl.throttle()` deducts `scaleGas(request.getGas())` tokens from the in-memory `gasLimitBucket` before execution begins: [1](#0-0) 

`scaleGas` divides by `GAS_SCALE_FACTOR = 10_000`: [2](#0-1) 

The bucket is a JVM-local `Bucket` with `SynchronizationStrategy.SYNCHRONIZED` — there is no cross-node coordination: [3](#0-2) 

**Note on default values:** The claim states `gasPerSecond = 1,500,000,000` (docs value), but the actual code default is `7_500_000_000L`. The documentation lists `1500000000`. This discrepancy affects the exact request-rate numbers but does not affect the core vulnerability mechanism. [4](#0-3) 

**DB amplification in `findStorageBatch()`:**

`findStorage()` checks `contractStateCache` (TTL = 2s). On a miss it calls `findStorageBatch()`: [5](#0-4) 

`findStorageBatch()` accumulates the current key into `contractSlotsCache`, then retrieves **all** accumulated keys and issues a single batch query with the full set: [6](#0-5) 

`findStorageBatch` on the repository has **no `@Cacheable` annotation** — it always hits the DB: [7](#0-6) 

**Cache TTL asymmetry creates a repeating attack window:**

`contractState` (value cache) TTL = 2s: [8](#0-7) 

`slotsPerContract` (key accumulator) TTL = 5 min, max 1,500 entries: [9](#0-8) 

**Corrected amplification behavior (important deviation from the claim):**

The claim states "1,500 `findStorageBatch` calls/req" sustained. The actual behavior is more nuanced:

- **First request** with N unique slots: N DB queries are issued, with `IN` clause sizes growing as 1, 2, 3, …, N (O(N²) total keys across all queries for that request).
- **Subsequent requests within the 2s TTL**: 0 DB queries — all values are in `contractStateCache`.
- **After 2s TTL expiry**: The first SLOAD in the next request triggers 1 `findStorageBatch` call with all N keys (since `slotsPerContract` still holds all N keys), repopulating the entire value cache. Remaining SLOADs in the same request hit the cache.
- **Thundering herd**: Multiple concurrent requests arriving simultaneously after the 2s TTL can each independently miss the cache and each issue a `findStorageBatch` with N keys before any of them repopulates the cache.

The claim's figure of "150,000 DB queries/second sustained" is overstated. The realistic worst case is a burst of large-IN-clause queries every 2 seconds, amplified by concurrent request concurrency.

**Gas restore does not help the attacker here:**

With `gasLimitRefundPercent = 100` (default) and a contract that consumes all 15M gas, `gasRemaining = 0` and `restore(0)` is called — nothing is returned to the bucket: [10](#0-9) 

## Impact Explanation

- **O(N²) DB key exposure on first access**: A contract with 1,500 unique storage slots causes 1,500 DB queries on first access, with `IN` clause sizes 1, 2, …, 1,500 — totaling 1,125,750 key lookups in a single request.
- **Thundering herd every 2 seconds**: After each 2s `contractStateCache` expiry, concurrent requests simultaneously miss the cache and each issue a `findStorageBatch` with up to 1,500 byte-array keys. The number of simultaneous large queries equals the number of concurrent requests that arrive in that window.
- **Shared DB, per-node throttle**: The gas throttle is JVM-local. Each mirror node has its own independent bucket. The shared PostgreSQL backend (`statementTimeout = 3,000 ms`) is the common resource under attack.
- **No authentication required**: `eth_call` is a read-only simulation; no on-chain transaction or HBAR fee is required.

## Likelihood Explanation

- **Precondition**: A contract with many storage slots exists (trivially true on mainnet/testnet; attacker can deploy one).
- **Skill required**: Sending HTTP POST requests to `/api/v1/contracts/call` with `gas: 15000000` and calldata that loops over SLOAD opcodes. Standard Solidity/web3 tooling.
- **Repeatability**: Fully automated and repeatable. The 2s cache TTL creates a predictable attack window.
- **No cost**: Read-only simulation, no on-chain fees.

## Recommendation

1. **Bound `findStorageBatch` query size**: Cap the `IN` clause to a configurable maximum (e.g., 50–100 keys) regardless of how many keys are accumulated in `contractSlotsCache`. This eliminates the O(N²) growth on first access.
2. **Add per-request slot-key accumulation reset**: Clear or scope `contractSlotsCache` per-request rather than sharing it across requests, so the batch query only includes keys accessed in the current request.
3. **Add `@Cacheable` or result-level caching to `findStorageBatch`**: Cache the batch query results keyed on `(contractId, sortedSlotSet)` to avoid redundant DB hits on thundering herd.
4. **Add per-IP or per-source rate limiting**: The current throttle is gas-based and per-node. A network-level or application-level per-IP rate limit would reduce the thundering herd window.
5. **Increase `contractStateCache` TTL or add negative caching**: A longer TTL reduces the frequency of cache expiry windows that trigger batch re-fetches.

## Proof of Concept

1. Deploy a contract on testnet with 1,500 storage slots pre-populated with unique values.
2. Send a POST to `/api/v1/contracts/call` with `gas: 15000000` and calldata that executes `SLOAD` on all 1,500 unique slots sequentially.
3. Observe: the first request triggers 1,500 calls to `ContractStateRepository.findStorageBatch()`, with `IN` clause sizes growing from 1 to 1,500 (confirmed by `ContractStateServiceImpl.findStorageBatch()` lines 90–103).
4. Wait 2 seconds for `contractStateCache` to expire (TTL confirmed at `CacheProperties.java` line 28).
5. Send 50+ concurrent requests simultaneously. Each concurrent request that arrives before the first one repopulates the cache will independently issue a `findStorageBatch` with 1,500 keys (since `slotsPerContract` TTL is 5 min, confirmed at `CacheProperties.java` line 39).
6. Observe PostgreSQL query load spike with large `IN` clause queries every 2 seconds, bounded only by request concurrency and the `requestsPerSecond = 500` RPS limit.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
