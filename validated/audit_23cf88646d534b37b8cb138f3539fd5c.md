# Audit Report

## Title
Unbounded In-Memory Opcode Trace Accumulation Leads to Heap Exhaustion via EVM Replay

## Summary
The `/api/v1/contracts/results/{id}/opcodes` endpoint replays historical EVM transactions and accumulates every executed opcode into an unbounded `ArrayList<Opcode>` inside `OpcodeContext`. No per-request cap on opcode count or heap consumption exists. A single unauthenticated HTTP request replaying a high-gas tight-loop transaction can exhaust the JVM heap of the web3 service process.

## Finding Description

**Code path:**

`OpcodesController.getContractOpcodes()` (lines 61–64) calls `throttleManager.throttleOpcodeRequest()` and then `opcodeService.processOpcodeCall(request)` with no per-request memory guard. [1](#0-0) 

`OpcodeServiceImpl.processOpcodeCall()` (line 67) constructs `new OpcodeContext(opcodeRequest, (int) params.getGas() / 3)`. The second argument is passed to `new ArrayList<>(opcodesSize)` — an *initial capacity hint*, not a cap. For a 15 M-gas transaction this pre-allocates 5 M slots, but the list grows without bound. [2](#0-1) [8](#0-7) 

`OpcodeContext.addOpcodes()` (lines 54–56) performs an unconditional `opcodes.add(opcode)` with no size check. [3](#0-2) 

`OpcodeActionTracer.tracePostExecution()` (lines 44–62) is invoked once per executed EVM opcode and calls `addOpcodes()` unconditionally. With `stack=true` (the default), each `Opcode` object carries a full copy of the EVM stack as a `List<String>`. [4](#0-3) 

`ThrottleManagerImpl.throttleOpcodeRequest()` (lines 52–56) enforces only a global rate limit (`opcodeRequestsPerSecond = 1` by default). It does nothing to bound the memory consumed by a single request. [5](#0-4) [6](#0-5) 

**Root cause:** The failed assumption is that gas-bounded EVM execution implies memory-bounded tracing. A tight `JUMPDEST` loop costs 1 gas/iteration, so a 15 M-gas transaction can execute up to 15 M iterations, producing up to 15 M `Opcode` objects — each with stack data — in a single unbounded `ArrayList`. At ~500 bytes per object this approaches several gigabytes of heap allocation from one request.

## Impact Explanation
A single replayed high-gas tight-loop transaction exhausts the JVM heap of the web3 service process, causing an `OutOfMemoryError` that crashes or severely degrades the service for all users — not just the opcodes endpoint. Because the throttle is global (not per-source), a single attacker occupying the 1 req/sec slot keeps the service under continuous memory pressure. This constitutes shutdown/severe degradation of the service without brute-force volume.

## Likelihood Explanation
The endpoint is disabled by default (`enabled = false`), which is a meaningful mitigating control. [7](#0-6) 

However, when enabled (required for any opcode-debugging use case), exploitation requires only:
1. A one-time on-chain expenditure to submit a high-gas tight-loop contract transaction.
2. A single unauthenticated HTTP GET with `Accept-Encoding: gzip` — no credentials, no special role.

The transaction hash is publicly visible on-chain. The attack is fully repeatable at 1 req/sec. No brute force is needed.

## Recommendation
1. **Hard cap on opcode count:** Enforce a configurable maximum (e.g., `maxOpcodes`) in `OpcodeContext.addOpcodes()` and throw a controlled exception when the limit is reached, preventing unbounded growth.
2. **Gas-based pre-check:** Reject replay requests whose recorded `gasLimit` exceeds a configurable threshold before EVM execution begins.
3. **Request timeout:** Enforce a wall-clock timeout on the EVM replay execution so that long-running replays are interrupted before heap is exhausted.
4. **Per-source rate limiting:** Supplement the global `opcodeRequestsPerSecond` bucket with per-IP or per-authenticated-client limiting to prevent a single source from monopolizing the slot.

## Proof of Concept

1. Deploy a contract containing a tight `JUMPDEST` loop to Hedera testnet with the maximum allowed gas limit (~15 M gas). Record the transaction hash.
2. Ensure the mirror node has `hiero.mirror.web3.opcode.tracer.enabled=true`.
3. Send the following request:
```
GET /api/v1/contracts/results/<TX_HASH>/opcodes?stack=true
Accept-Encoding: gzip
```
4. The EVM replays the transaction, `OpcodeActionTracer.tracePostExecution()` is called up to ~15 M times, each call appending an `Opcode` object (with stack) to the unbounded `ArrayList` in `OpcodeContext`.
5. The JVM heap is exhausted; the web3 service process throws `OutOfMemoryError` and becomes unavailable to all clients.
6. Repeat at 1 req/sec (the global throttle limit) to maintain continuous pressure.
