Audit Report

## Title
Cache Pollution via Unnormalized `evmAddress` Literal Variants in `CachedPreparsedDocumentProvider`

## Summary
`CachedPreparsedDocumentProvider` keys its Caffeine cache on the verbatim GraphQL query string. Because the `evmAddress` field accepts both `0x`-prefixed and unprefixed hex strings with mixed case, an attacker who inlines the address as a query literal can produce thousands of distinct cache keys for semantically identical queries, saturating the 1,000-entry cache and forcing repeated re-parsing for all legitimate users.

## Finding Description

**Cache key is the raw query string with no normalization.** [1](#0-0) 

The key passed to `cache.get(...)` is `executionInput.getQuery()` — the verbatim text sent by the client.

**Schema permits both `0x`-prefixed and bare hex, and mixed case.** [2](#0-1) 

The regex `^(0x)?[a-fA-F0-9]{40}$` allows `0xAbCd...`, `abcd...`, `ABCD...`, and all mixed-case variants — all semantically identical.

**`decodeEvmAddress` normalizes the address at execution time, making all variants equivalent.** [3](#0-2) 

The `0x` prefix is stripped and the hex is decoded, but this happens during data-fetcher execution — after the cache lookup.

**`@Pattern` validation runs as a data-fetcher interceptor (execution time), not during parse/validate.** [4](#0-3) [5](#0-4) 

`ValidationSchemaWiring` is registered as a `directiveWiring` in the runtime wiring, meaning it wraps data fetchers. It does not run inside `parseAndValidateFunction`. Therefore, any syntactically valid GraphQL document — regardless of the `evmAddress` string value — produces a cache entry.

**Cache is bounded at 1,000 entries with a 1-hour TTL.** [6](#0-5) 

Once saturated, every legitimate query misses the cache for up to one hour.

**Parser limits do not prevent the attack.** [7](#0-6) 

The `maxCharacters(10000)` and `maxTokens(1000)` limits are not triggered by a short account lookup query with an inlined address.

## Impact Explanation
Once the 1,000-entry cache is saturated with attacker-controlled entries, every legitimate query misses the cache and must be fully re-parsed and re-validated on every request for the duration of the 1-hour TTL window. This increases CPU load and latency for all users. Caffeine's `maximumSize` prevents unbounded memory growth, but the cache's protective purpose — amortizing parse cost — is completely defeated. The attack is repeatable: the attacker can continuously rotate new address variants to keep the cache polluted indefinitely.

## Likelihood Explanation
No authentication or privilege is required. The GraphQL endpoint is publicly accessible. The 40 hex characters `a–f` can each be upper or lower case, yielding up to 2²⁴ ≈ 16 million syntactically distinct strings per address. An attacker needs only ~1,000 unique strings to saturate the cache. The attack is trivially scriptable with sequential HTTP requests from a single IP.

## Recommendation

1. **Normalize the cache key before lookup.** Before calling `cache.get(...)`, normalize the query string by lowercasing all string literals and stripping `0x` prefixes. Alternatively, use the parsed AST's canonical string representation as the cache key instead of the raw query text.

2. **Encourage/enforce variable usage.** Document or enforce that address values must be passed as GraphQL variables rather than inlined literals. Variables are not part of the query document string and therefore do not affect the cache key.

3. **Add per-IP rate limiting** on the `/graphql/alpha` endpoint at the application or infrastructure layer to limit the rate at which new unique query strings can be submitted.

## Proof of Concept

Send the following ~1,000 requests sequentially from a single IP, varying the case of hex characters in the inlined `evmAddress`:

```
POST /graphql/alpha
{ "query": "{ account(input: { evmAddress: \"0xaAbBcCdDeEfF0000000000000000000000000001\" }) { entityId { num } } }" }

POST /graphql/alpha
{ "query": "{ account(input: { evmAddress: \"0xAABBCCDDEEFF0000000000000000000000000001\" }) { entityId { num } } }" }

POST /graphql/alpha
{ "query": "{ account(input: { evmAddress: \"aabbccddeeff0000000000000000000000000001\" }) { entityId { num } } }" }

... (repeat with ~997 more case variants)
```

Each request produces a distinct cache key, fills one cache slot, and evicts a legitimate entry. After ~1,000 requests the cache is saturated. All subsequent legitimate queries (using variables or any other query shape) will miss the cache and incur full re-parse cost on every request for up to one hour.
