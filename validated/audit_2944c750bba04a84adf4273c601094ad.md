Audit Report

## Title
Unbounded Cumulative Initialization Delay via Network-Path Interception in `buildNetworkFromMirrorNodes`

## Summary
`buildNetworkFromMirrorNodes` is invoked with `context.Background()` (no deadline, never cancelled) and retries up to `mirrorNodeClientMaxRetries + 1` times. Each attempt can block for the full per-request `http.Client.Timeout`, followed by exponential backoff. An attacker controlling the network path to the mirror REST endpoint can force every attempt to consume its full timeout, producing a cumulative initialization delay of ~621 seconds with default settings, during which zero transactions are submitted.

## Finding Description

**`pinger/sdk_client.go` line 18** — `buildNetworkFromMirrorNodes` is called with an unbounded context: [1](#0-0) 

**`pinger/mirror_node_client.go` lines 46–48** — the HTTP client and attempt count are derived from config defaults (10s timeout, 11 attempts): [2](#0-1) 

**`pinger/mirror_node_client.go` lines 63–68** — the backoff `select` contains a `ctx.Done()` branch that is permanently dead because `context.Background()` never cancels: [3](#0-2) 

**`pinger/mirror_node_client.go` lines 79–84** — each HTTP request is bound only by the per-request `http.Client.Timeout`, with no independent overall deadline: [4](#0-3) 

**Default configuration values confirmed in `pinger/config.go`:**
- `mirrorNodeClientMaxRetries` = 10 → 11 attempts
- `mirrorNodeClientTimeout` = 10s
- `mirrorNodeClientBaseBackoff` = 500ms [5](#0-4) 

**Cumulative worst-case delay:**

| Phase | Duration |
|---|---|
| 11 attempts × 10s timeout | 110s |
| Backoff: 500ms × (2⁰ + 2¹ + … + 2⁹) | ~511.5s |
| **Total** | **~621.5s (~10.4 min)** |

## Impact Explanation

`newClient()` is called synchronously in `main()` at line 41. The readiness file `/tmp/ready` is only written **after** `newClient()` returns successfully (line 47). Until then, the pinger submits zero transactions. [6](#0-5) 

The liveness heartbeat goroutine writes `/tmp/alive` every 15 seconds independently of `newClient()`, so the pod is never killed by the liveness probe — it silently stalls. [7](#0-6) 

This constitutes a complete freeze of all transaction submissions for the full duration of the attack (~10 minutes with defaults). The attack is repeatable on every pod restart.

## Likelihood Explanation

The precondition is controlling the network path between the pinger pod and the mirror REST endpoint (`HIERO_MIRROR_PINGER_REST`). This is achievable via:
- **DNS poisoning** if the REST URL uses a hostname
- **ARP/BGP-level interception** in cloud or on-prem environments
- **A compromised intermediate proxy or load balancer**

No credentials, operator key, or authentication are required. The attacker only needs to accept the TCP connection and withhold the HTTP response. This is only triggered when `cfg.network == "other"`, but that is the exact deployment mode that uses `buildNetworkFromMirrorNodes`. [8](#0-7) 

## Recommendation

1. **Wrap the entire `buildNetworkFromMirrorNodes` call with an overall deadline context**, e.g.:
   ```go
   ctx, cancel := context.WithTimeout(context.Background(), cfg.mirrorNodeClientOverallTimeout)
   defer cancel()
   netmap, err := buildNetworkFromMirrorNodes(ctx, cfg)
   ```
   This makes the `ctx.Done()` branch in the backoff `select` functional and caps total initialization time.

2. **Add a configurable `mirrorNodeClientOverallTimeout`** (e.g., default 60s) distinct from the per-request timeout.

3. **Cap the maximum backoff** to prevent exponential growth from dominating the total delay.

## Proof of Concept

1. Set `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>/`.
2. On the attacker-controlled host, run a TCP listener that accepts connections on port 80 but never sends any bytes.
3. Start the pinger pod. Observe that `newClient()` blocks for ~621 seconds (11 × 10s hangs + exponential backoff).
4. Confirm `/tmp/ready` is never written during this window, and `/tmp/alive` is updated every 15s (pod not killed).
5. Confirm zero transactions are submitted during the stall period.
