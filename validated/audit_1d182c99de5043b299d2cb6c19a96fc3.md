Audit Report

## Title
Batch DB Query Amplification via Stale `contractSlotsCache` Key Accumulation

## Summary
An unauthenticated caller can accumulate up to 1,500 unique slot keys in the per-contract `contractSlotsCache` (TTL: 5 minutes) by issuing standard `eth_getStorageAt` JSON-RPC calls. Because `contractStateCache` expires every 2 seconds, every subsequent cache miss for that contract triggers `contractStateRepository.findStorageBatch()` with all 1,500 accumulated keys in the SQL `IN` clause, producing a sustained 1,500Ă— DB query amplification ratio with a single request every 2 seconds.

## Finding Description

**Root cause and failed assumption:**

The design assumes that once a batch query populates `contractStateCache`, subsequent requests will be cache hits and the large `IN` clause will not be re-issued. This assumption fails because of a 150Ă— TTL mismatch:

- `contractState` cache: `expireAfterWrite=2s,maximumSize=25000`
- `slotsPerContract` cache: `expireAfterAccess=5m,maximumSize=1500` [4](#0-3) 

The key registry outlives the value cache by 150Ă—. After every 2-second expiry of `contractStateCache`, the next single request for any slot of that contract re-triggers the full batch query with all K accumulated keys.

**Why existing checks are insufficient:**

The early-return at lines 65â€“67 only fires on a `contractStateCache` hit (`cachedValue != null && cachedValue != EMPTY_VALUE`). It cannot fire after the 2-second TTL expires. Additionally, for slots that do not exist in the DB, `findStorageBatch()` never writes anything into `contractStateCache` (line 109 only stores values for slots returned by the DB), so those slots will **never** produce a cache hit â€” the full batch query is re-issued on every single request for those slots, indefinitely. [5](#0-4) [6](#0-5) 

`maximumSize=1500` on `slotsPerContract` caps K but does not prevent the amplification â€” it merely bounds the worst-case factor.

## Impact Explanation

Each attacker request after the warm-up phase causes the DB to evaluate a `WHERE slot IN (k1, k2, ..., k1500)` predicate instead of a single-key lookup. At 1,500 keys this is a 1,500Ă— increase in per-request DB work. Sustained at one request per 2 seconds (the `contractStateCache` TTL), this yields 750 full-batch queries per 25-minute window per targeted contract. For non-existent slots, the amplification is permanent (not bounded by the 2-second TTL) because empty slots are never written into `contractStateCache`. An attacker targeting multiple contracts multiplies this linearly. This is sufficient to significantly increase DB CPU/IO on a lightly loaded node without any brute-force volume.

## Likelihood Explanation

`eth_getStorageAt(contractAddress, slotIndex, "latest")` is a standard, unauthenticated JSON-RPC call available on every public mirror-node endpoint. No wallet, token, or privileged credential is required. The attacker needs only to enumerate 1,500 slot indices for one contract â€” sequential integers `0x0` through `0x5DC` suffice even if most slots are empty, because empty-slot keys are still registered in `contractSlotsCache` at line 90 and included in every subsequent batch query. The attack is fully repeatable and scriptable.

## Recommendation

1. **Store negative results in `contractStateCache`**: After `findStorageBatch()` returns, write a distinct sentinel (e.g., a dedicated `MISSING_VALUE` constant) into `contractStateCache` for every key in `cachedSlots` that was not returned by the DB. Update the early-return check in `findStorage()` to also short-circuit on this sentinel, returning `Optional.empty()`. This prevents non-existent slots from bypassing the cache on every request.

2. **Align TTLs**: Consider reducing `slotsPerContract` TTL to be closer to `contractState` TTL, or switching `slotsPerContract` to `expireAfterWrite` semantics, so that the key registry does not outlive the value cache by orders of magnitude. [7](#0-6) 

3. **Limit batch size per invocation**: Cap the number of keys passed to `findStorageBatch()` to a configurable maximum (e.g., 50â€“100) to bound the worst-case amplification factor regardless of how many keys have accumulated.

4. **Rate limiting on `eth_getStorageAt`**: Apply per-IP or per-contract rate limiting on this endpoint. The existing `requestsPerSecond` throttle is global; a per-contract or per-slot-key limit would directly constrain the warm-up phase. [8](#0-7) 

## Proof of Concept

```
# Phase 1: Warm-up â€” accumulate 1,500 slot keys in contractSlotsCache (TTL: 5m)
for i in $(seq 0 1499); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_getStorageAt\",
         \"params\":[\"<contractAddress>\",\"$(printf '0x%x' $i)\",\"latest\"],
         \"id\":$i}"
done

# Phase 2: Sustained amplification â€” one request every 2 seconds triggers
# findStorageBatch() with all 1,500 keys in the SQL IN clause
while true; do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","method":"eth_getStorageAt",
         "params":["<contractAddress>","0x0","latest"],"id":9999}'
  sleep 2
done
```

After Phase 1, every iteration of Phase 2 causes the DB to execute:
```sql
SELECT slot, value FROM contract_state
WHERE contract_id = <id>
AND slot IN (/* 1,500 keys */)
```
instead of a single-key lookup, with no authentication or elevated privilege required.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
