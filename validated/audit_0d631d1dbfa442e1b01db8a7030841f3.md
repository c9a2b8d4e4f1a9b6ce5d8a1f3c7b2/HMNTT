Audit Report

## Title
Unbounded `json_agg` in `signatureQuery` Enables DB Amplification via `account.id` Filter

## Summary
In `getSchedules()` (`rest/schedules.js`), a secondary `signatureQuery` is issued against `transaction_signature` using `json_agg` with no `LIMIT` on the number of signatures aggregated per `entity_id`. Because the `transaction_signature` table intentionally has no unique constraint, an attacker can accumulate arbitrarily many rows per schedule and trigger expensive unbounded aggregation via a public, unauthenticated API call.

## Finding Description

**Verified code path — `rest/schedules.js`:**

The `transactionSignatureJsonAgg` template is defined at lines 39–45 with no inner `LIMIT`: [1](#0-0) 

Inside `getSchedules()`, after the primary `schedulesQuery` returns up to `limit` schedule rows (line 241), `entityIds` is extracted (line 250) and the following `signatureQuery` is built at lines 255–259: [2](#0-1) 

There is **no `LIMIT` clause** on the number of `transaction_signature` rows aggregated per `entity_id`. The query will process every matching row in the table for all up to 100 returned schedule IDs.

**Root cause — confirmed by design document:**

`docs/design/scheduled-transactions.md`, line 65 explicitly states: [3](#0-2) 

> *"There's no unique constraint/primary key since the client can potentially sign multiple times with the same key."*

The `transaction_signature` table schema confirms no primary key or unique constraint: [4](#0-3) 

**Why existing checks are insufficient:**

- The `schedulesQuery` `LIMIT` (capped at `responseLimit.max`, default 100) only bounds the number of schedule IDs in the `IN` clause — it does not bound the number of `transaction_signature` rows fetched per schedule. [5](#0-4) 

- `buildAndValidateFilters` / `extractSqlFromScheduleFilters` cap the `limit` parameter but have no effect on the inner `json_agg` aggregation. [6](#0-5) 

- The index `transaction_signature (entity_id desc, consensus_timestamp desc)` speeds up the lookup but does not reduce the number of rows aggregated. [7](#0-6) 

**Note:** The same unbounded `transactionSignatureJsonAgg` is also used in `getScheduleByIdQuery` (lines 46–64), making the single-schedule endpoint similarly affected. [8](#0-7) 

## Impact Explanation
An attacker who controls an account can:
1. Create N schedules with `creator_account_id = attacker_account`.
2. Repeatedly submit `ScheduleSign` transactions to accumulate many `transaction_signature` rows per schedule (permitted by design, as confirmed by the design doc).
3. Issue `GET /api/v1/schedules?account.id=0.0.ATTACKER&limit=100`, causing the mirror node to execute a `json_agg` over potentially millions of rows in a single query.
4. Replay the same HTTP request indefinitely to sustain elevated DB CPU/memory load.

The result is degraded mirror node availability (slow or timed-out responses) for all users. No user funds or on-chain assets are affected; this is a griefing/DoS classification.

## Likelihood Explanation
- **No privilege required**: `/api/v1/schedules` is public and unauthenticated.
- **Attacker cost**: Each `ScheduleSign` transaction costs a small HBAR fee, but fees are low enough that accumulating thousands of signatures per schedule is economically feasible for a motivated attacker.
- **Repeatability**: Once signatures are accumulated on-chain, the same HTTP request can be replayed indefinitely at no additional cost.
- **No rate-limiting visible in the reviewed code path**: No per-IP or per-account request throttling is present in `getSchedules()` or its callees. [9](#0-8) 

## Recommendation
1. **Add a per-entity signature cap in the query**: Rewrite `signatureQuery` to use a subquery or `LATERAL` join that limits the number of signatures fetched per `entity_id` (e.g., `FETCH FIRST N ROWS ONLY` per group), where N is a configurable maximum (e.g., 25–50).
2. **Alternatively, paginate signatures separately**: Remove signatures from the list endpoint entirely and only return them on the single-schedule endpoint (`/schedules/:id`), where the scope is already one entity.
3. **Apply a statement timeout**: Ensure `db.pool.statementTimeout` is configured to a low value so runaway aggregation queries are killed before exhausting DB resources.
4. **Consider rate-limiting**: Apply per-IP or per-account rate limiting at the API gateway level for the `/schedules` endpoint.

## Proof of Concept

```
# Step 1: Attacker creates many schedules and accumulates signatures
# (done on-chain via ScheduleCreate + repeated ScheduleSign transactions)
# Assume attacker account = 0.0.12345, with 100 schedules each having 10,000 signature rows

# Step 2: Trigger the amplified query
curl "https://mainnet-public.mirrornode.hedera.com/api/v1/schedules?account.id=0.0.12345&limit=100"

# This causes the mirror node to execute:
# SELECT entity_id, json_agg(...) FROM transaction_signature
# WHERE entity_id IN ($1, ..., $100)   -- up to 100 schedule IDs
# GROUP BY entity_id
# ORDER BY entity_id ASC
#
# With 100 schedules × 10,000 signatures each = 1,000,000 rows aggregated in one query.

# Step 3: Repeat the request in a loop to sustain DB load
while true; do
  curl -s "https://.../api/v1/schedules?account.id=0.0.12345&limit=100" > /dev/null
done
```

```javascript
const extractSqlFromScheduleFilters = (filters) => {
  const filterQuery = {
    filterQuery: '',
    params: [defaultLimit],
    order: constants.orderFilterValues.ASC,
    limit: defaultLimit,
  };

  // if no filters return default filter of no where clause, defaultLimit and asc order
  if (filters && filters.length === 0) {
    return filterQuery;
  }

  const pgSqlParams = [];
  let whereQuery = '';
  let applicableFilters = 0; // track the number of schedule specific filters
  let paramCount = 1; // track the param count used for substitution, not affected by order and executed params

  for (const filter of filters) {
    if (filter.key === constants.filterKeys.LIMIT) {
      filterQuery.limit = filter.value;
      continue;
    }

    if (filter.key === constants.filterKeys.ORDER) {
      filterQuery.order = filter.value;
      continue;
    }

    const columnKey = filterColumnMap[filter.key];
    if (columnKey === undefined) {
      continue;
    }

    // add prefix. 'where' for the 1st param and 'and' for subsequent
    whereQuery += applicableFilters === 0 ? `where ` : ` and `;
    applicableFilters++;

    whereQuery += `${filterColumnMap[filter.key]}${filter.operator}$${paramCount}`;
    paramCount++;
    pgSqlParams.push(filter.value);
  }

  // add limit
  pgSqlParams.push(filterQuery.limit);

  filterQuery.filterQuery = whereQuery;
  filterQuery.params = pgSqlParams;

  return filterQuery;
};
```

**File:** rest/schedules.js (L234-282)
```javascript
const getSchedules = async (req, res) => {
  // extract filters from query param
  const filters = utils.buildAndValidateFilters(req.query, acceptedSchedulesParameters);

  // get sql filter query, params, order and limit from query filters
  const {filterQuery, params, order, limit} = extractSqlFromScheduleFilters(filters);
  const schedulesQuery = getSchedulesQuery(filterQuery, order, params.length);
  const {rows: schedules} = await pool.queryQuietly(schedulesQuery, params);

  const schedulesResponse = {schedules: [], links: {next: null}};
  res.locals[constants.responseDataLabel] = schedulesResponse;

  if (schedules.length === 0) {
    return;
  }

  const entityIds = schedules.map((s) => s.schedule_id);
  const positions = range(1, entityIds.length + 1)
    .map((i) => `$${i}`)
    .join(',');
  const entityQuery = `select ${entityFields} from entity where id in (${positions}) order by id ${order}`;
  const signatureQuery = `select entity_id, ${transactionSignatureJsonAgg} as signatures
    from transaction_signature ts
    where entity_id in (${positions})
    group by entity_id
    order by entity_id ${order}`;

  const [{rows: entities}, {rows: signatures}] = await Promise.all([
    pool.queryQuietly(entityQuery, entityIds),
    pool.queryQuietly(signatureQuery, entityIds),
  ]);

  schedulesResponse.schedules = mergeScheduleEntities(schedules, entities, signatures).map(formatScheduleRow);

  // populate next link
  const lastScheduleId =
    schedulesResponse.schedules.length > 0
      ? schedulesResponse.schedules[schedulesResponse.schedules.length - 1].schedule_id
      : null;

  schedulesResponse.links.next = utils.getPaginationLink(
    req,
    schedulesResponse.schedules.length !== limit,
    {
      [constants.filterKeys.SCHEDULE_ID]: lastScheduleId,
    },
    order
  );
};
```
