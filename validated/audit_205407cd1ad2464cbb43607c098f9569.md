# Audit Report

## Title
Unbounded Infinite DB Polling via `limit=0` Subscription in `PollingTopicListener`

## Summary
When `ListenerType.POLL` is configured, any unauthenticated client can open a gRPC `subscribeTopic` call with `limit=0` and an arbitrarily old `startTime`. This causes `PollingTopicListener` to issue a database query every 500 ms indefinitely per subscription, with no server-side termination. Enough concurrent subscriptions exhaust the DB connection pool and deny service to all legitimate users.

## Finding Description

**Infinite repeat loop — confirmed:**

`PollingTopicListener.listen()` constructs the Flux with:

```java
.repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
        .jitter(0.1)
        .withFixedDelay(interval)
        .withScheduler(scheduler))
``` [1](#0-0) 

`Long.MAX_VALUE` iterations at 500 ms each is ~9.2 × 10¹⁸ iterations — effectively forever. There is no conditional termination based on whether the filter has a limit.

**`limit=0` is valid and treated as "no limit" — confirmed:**

`TopicMessageFilter` declares `limit` with `@Min(0)`, so `limit=0` passes validation: [2](#0-1) 

`hasLimit()` returns `limit > 0`, so `limit=0` → `hasLimit() == false`: [3](#0-2) 

**`poll()` uses `Integer.MAX_VALUE` when no limit — confirmed:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize()); // = 5000
``` [4](#0-3) 

Each poll fetches up to `maxPageSize=5000` rows (default): [5](#0-4) 

**`startTime` validation is insufficient — confirmed:**

The only constraint is `startTime <= DomainUtils.now()`: [6](#0-5) 

Epoch 0 (or any past timestamp) is accepted. With no `last` message seen yet, every poll queries from that old timestamp forward, potentially scanning a large historical range.

**No per-subscriber connection cap — confirmed:**

`TopicMessageServiceImpl` tracks `subscriberCount` only as a Micrometer gauge metric: [7](#0-6) 

There is no enforcement of a maximum concurrent subscriber count anywhere in the reviewed code path.

## Impact Explanation

Each malicious subscription fires one DB query every 500 ms indefinitely. With `maxPageSize=5000`, each query can return up to 5,000 rows from a potentially large historical range. N concurrent subscriptions generate N DB queries every 500 ms. Because the scheduler is `Schedulers.boundedElastic()` and each poll holds a DB connection for its duration, a modest number of concurrent subscriptions (proportional to the DB connection pool size) saturates the pool. This causes all legitimate queries — from other gRPC subscribers and the importer — to time out or fail, constituting a complete application-level DoS against the mirror node's database tier. [8](#0-7) 

## Likelihood Explanation

- **No authentication required**: The gRPC `subscribeTopic` endpoint is publicly accessible.
- **Trivial to trigger**: A single gRPC call with `limit=0` and `startTime=0` is sufficient per connection.
- **Scalable**: One attacker can open many connections from a single host, multiplying DB load linearly.
- **Persistent**: The attack sustains itself as long as the TCP connection is held open; no repeated action is needed.
- **Condition**: Deployment must use `ListenerType.POLL`. The default is `REDIS`, but `POLL` is a documented, supported, and production-viable configuration option: [9](#0-8) 

## Recommendation

1. **Reject `limit=0` or treat it as requiring a minimum positive value.** Change `@Min(0)` to `@Min(1)` on `TopicMessageFilter.limit`, or redefine the semantics so that `limit=0` means "use a server-enforced default cap" rather than "no limit." [2](#0-1) 

2. **Bound the repeat loop when no limit is set.** In `listen()`, if `filter.hasLimit()` is false, consider using a finite `RepeatSpec` or adding a server-side maximum subscription duration/iteration count. [1](#0-0) 

3. **Enforce a maximum concurrent subscriber cap.** The existing `subscriberCount` gauge should be converted into an enforced limit that rejects new subscriptions when the cap is reached. [7](#0-6) 

4. **Add a minimum `startTime` bound.** Reject `startTime` values older than a configurable retention window to prevent full historical table scans. [6](#0-5) 

## Proof of Concept

```python
import grpc
# Pseudocode — adapt to the actual proto definitions
channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = ConsensusServiceStub(channel)

request = ConsensusTopicQuery(
    topic_id=TopicID(topic_num=1),
    consensus_start_time=Timestamp(seconds=0, nanos=0),  # epoch 0
    limit=0  # no limit — triggers Integer.MAX_VALUE path
)

# Open N concurrent subscriptions from one host
for _ in range(N):
    threading.Thread(target=lambda: list(stub.subscribeTopic(request))).start()
# Each subscription now fires a DB query every 500 ms indefinitely.
# N subscriptions → N queries/500ms → DB pool exhaustion.
```

The attack requires no credentials, no special tooling, and sustains itself passively as long as the TCP connections remain open.
