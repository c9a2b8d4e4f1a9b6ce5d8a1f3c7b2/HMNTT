## Audit Report

## Title
Connection Pool Exhaustion via Unbounded Concurrent Requests to `TopicRepository.findById()` During Database Degradation

## Summary
The `rest-java` service exposes `/api/v1/topics/{id}` with no application-level rate limiting or circuit breaker on the database call path. During database degradation (slow queries not yet timing out), each request holds a HikariCP connection for up to the full `statementTimeout` (10 seconds). An unprivileged attacker flooding this endpoint during a degradation event can exhaust the connection pool. The Traefik retry policy (`attempts: 3`) amplifies pool pressure by a factor of 3, and the Traefik circuit breaker only trips on error ratios — not latency — so it provides no protection during the slow-query window.

## Finding Description

**Code path:**

`TopicController.getTopic()` → `TopicServiceImpl.findById()` → `TopicRepository.findById()` (Spring Data `CrudRepository`, no fallback, no timeout override at the application layer beyond the DB-level `statementTimeout`). [1](#0-0) [2](#0-1) [3](#0-2) 

**Gap 1 — No application-level rate limiting in rest-java.** The `web3` module has a full `ThrottleConfiguration` with bucket4j (`requestsPerSecond`, `gasPerSecond`, per-request filters). The `rest-java` module has no equivalent — no `@RateLimiter`, no `ThrottleManager`, no request-count guard. [4](#0-3) 

**Gap 2 — No application-level circuit breaker.** No `CircuitBreaker`, `resilience4j`, or equivalent is present anywhere in `rest-java`. The repository call is a raw Spring Data proxy with no fallback.

**Gap 3 — Traefik retry amplifies pool pressure.** The Helm chart configures `attempts: 3` on the retry middleware. Each user-visible request that times out is retried 3 times by Traefik before the circuit breaker trips, tripling the number of concurrent HikariCP connections consumed per attacker request. [5](#0-4) 

**Why the Traefik circuit breaker is insufficient:**

The circuit breaker expression `ResponseCodeRatio(500, 600, 0, 600) > 0.25` only trips *after* 25% of responses are already 5xx. During the degradation window — slow queries that have not yet hit the 10-second `statementTimeout` — responses are not yet 5xx; they are simply slow. The circuit breaker does not trip on latency. During this window, connections accumulate in the HikariCP pool. [11](#0-10) 

**HikariCP pool size:** `CommonConfiguration` creates HikariCP from `spring.datasource.hikari` properties. No `maximumPoolSize` override exists in `rest-java` resources, so the HikariCP default of 10 connections applies. [6](#0-5) 

**Statement timeout:** `hiero.mirror.restJava.db.statementTimeout` defaults to `10000` ms (10 seconds). [7](#0-6) 

**Monitoring gap:** The `prometheusRules.enabled` parent flag defaults to `false` in the Helm chart, meaning the `RestJavaHighDBConnections` PrometheusRule CRD is not deployed by default. The Grafana/Terraform-managed alert has `is_paused = false` but a `for: 5m` evaluation window — the attack can succeed and cause an outage before the alert fires. [12](#0-11) [10](#0-9) 

## Impact Explanation

When the HikariCP pool (default 10 connections) is exhausted, all subsequent requests to **any endpoint** in the `rest-java` service queue waiting for a connection. Once the queue fills, requests fail with 503/500. This converts a partial database degradation (slow queries) into a complete service outage for all users of the `rest-java` service. The Traefik retry policy means each attacker connection attempt consumes up to 3 pool slots, so only 3–4 concurrent attacker connections are needed to saturate the default pool. The outage persists until the database recovers AND the pool drains.

## Likelihood Explanation

The `/api/v1/topics/{id}` endpoint is publicly accessible with no authentication. Any unprivileged user can send concurrent HTTP GET requests. The attacker does not need to cause the database degradation — they only need to observe it (e.g., via elevated latency on public responses) and then flood the endpoint. The GCP backend policy `maxRatePerEndpoint: 250` provides some protection but applies per-endpoint at the load balancer level and does not prevent pool exhaustion from a moderate number of concurrent slow-query-holding connections. [8](#0-7) 

## Recommendation

1. **Add application-level rate limiting to `rest-java`** — mirror the `ThrottleConfiguration`/`ThrottleManager` pattern from the `web3` module, or add a Tomcat/Spring `server.tomcat.max-connections` + `server.tomcat.accept-count` cap.
2. **Add a Traefik `inFlightReq` middleware** — the `graphql` and `rosetta` charts already use `inFlightReq.amount` per source IP; add the same to `rest-java`.
3. **Add a latency-based circuit breaker** — use `LatencyAtQuantileMS(50, 0) > 2000` in the Traefik circuit breaker expression, or add a Resilience4j `TimeLimiter` on the service layer.
4. **Explicitly configure `maximumPoolSize`** — set `spring.datasource.hikari.maximum-pool-size` to a value appropriate for the expected concurrency, and configure `connectionTimeout` to fail fast when the pool is exhausted rather than queuing indefinitely.
5. **Enable `prometheusRules` by default** or ensure the Grafana alert `for` duration is reduced so operators are alerted before pool exhaustion causes a full outage.

## Proof of Concept

```bash
# During a database degradation event (slow queries):
# 4 concurrent connections × 3 Traefik retries = 12 pool slots (exceeds default pool of 10)

for i in $(seq 1 4); do
  curl -s "https://<mirror-node>/api/v1/topics/0.0.1234" &
done
wait

# All subsequent requests to any rest-java endpoint will queue/fail:
curl -v "https://<mirror-node>/api/v1/network/supply"
# Expected: 503 Service Unavailable or connection timeout
```

The attack requires no authentication, no special tooling, and no ability to cause the underlying database degradation — only the ability to observe elevated response latency and send parallel HTTP GET requests.

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }

    @Bean(name = GAS_LIMIT_BUCKET)
    Bucket gasLimitBucket() {
        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()
                .withSynchronizationStrategy(SynchronizationStrategy.SYNCHRONIZED)
                .addLimit(limit)
                .build();
    }

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
