# Audit Report

## Title
Cache-Amplified Batch DB Query DoS via Slot Key Accumulation in `findStorageBatch()`

## Summary
An unauthenticated attacker can fill the per-contract `slotsPerContract` Caffeine cache (TTL: 5 min, max 1,500 entries) with unique slot keys using only 1,500 HTTP requests. Because `contractStateCache` values expire every 2 seconds while slot keys persist for 5 minutes, every subsequent request for the targeted contract — including from legitimate users — triggers a `SELECT ... WHERE slot IN (<1500 keys>)` database query instead of a single-slot equality lookup, achieving a 1,500× amplification of per-request DB work that persists for up to 5 minutes per setup cycle.

## Finding Description

**Affected file:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`

`findStorage()` (lines 58–70) checks `contractStateCache` (`expireAfterWrite=2s`). On a miss it calls `findStorageBatch()` (lines 85–122). [1](#0-0) 

Inside `findStorageBatch()`:
- **Line 90**: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — unconditionally registers every new slot key into the per-contract Caffeine cache (`slotsPerContract`, `expireAfterAccess=5m`, `maximumSize=1500`).
- **Lines 91–101**: reads **all** currently cached slot keys for the contract into `cachedSlots`.
- **Line 103**: issues `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — a native SQL `SELECT slot, value FROM contract_state WHERE contract_id = :contractId AND slot IN (:slots)` with the full accumulated list. [7](#0-6)

**Root cause — TTL mismatch + unbounded accumulation:**

| Cache | TTL | Max size |
|---|---|---|
| `contractStateCache` | `expireAfterWrite=2s` | 25,000 |
| `slotsPerContract` | `expireAfterAccess=5m` | 1,500 | [9](#0-8) 

Slot *keys* accumulate for 5 minutes; slot *values* expire every 2 seconds. After the 2-second value TTL, `contractStateCache.get(key)` returns `null`, the condition on line 65 is false, and execution falls through to `findStorageBatch()`, which re-issues the full batch query with all 1,500 accumulated keys. There is no mechanism to limit the batch size relative to the number of slots actually needed by the current request. [10](#0-9) 

**Exploit flow:**

1. **Setup (≤3 seconds at 500 RPS):** Attacker sends 1,500 `eth_call` requests targeting the same contract, each with a distinct 32-byte slot key (`0x0000...0001` through `0x0000...05DC`). Slot keys need not correspond to real storage — missing slots return no rows. Each request adds one key to `slotsPerContract` and triggers a growing batch query (1, 2, 3, … 1,500 slots).

2. **Amplification phase (persists 5 minutes):** After 2 seconds, `contractStateCache` values expire. Any subsequent request for this contract — including from legitimate users — triggers `SELECT ... WHERE slot IN (<1500 keys>)`. The attacker need not send further traffic; legitimate load sustains the amplified DB pressure.

3. **Scale-out:** `contractSlots` holds up to 3,000 contracts (`maximumSize=3000`). The attacker repeats setup for multiple contracts, multiplying impact. [6](#0-5) 

**Why existing mitigations fail:**

- `maximumSize=1500` on `slotsPerContract`: caps the batch at 1,500 but does not prevent the attack — 1,500 keys in an `IN` clause is the attack goal, not a mitigation.
- Global rate limit (`requestsPerSecond=500`): not per-IP; attacker consumes the shared budget during setup, degrading legitimate traffic simultaneously.
- Gas throttle bypass: `scaleGas()` returns `0` for `gas ≤ 10,000`, so `tryConsume(0)` always succeeds — the gas bucket is completely bypassed for minimal-gas calls, leaving only the 500 RPS limit as a constraint during setup. [4](#0-3) 
- `db.statementTimeout=3000ms`: limits individual query duration but does not prevent sustained high-complexity queries within that window. [11](#0-10) 

## Impact Explanation
Every post-setup request to the targeted contract causes the DB to evaluate a `slot IN (1500 keys)` predicate instead of a single equality lookup — a 1,500× increase in per-request DB work. At 500 RPS (the global cap), this translates to 750,000 slot comparisons per second per targeted contract, versus 500 under normal load. With multiple contracts targeted simultaneously (up to 3,000), the DB load increase can cause query queue saturation, connection pool exhaustion, and cascading latency for all users of the mirror node web3 API. Critically, the amplification effect persists for 5 minutes after the setup phase completes, meaning legitimate user traffic sustains the DB pressure without any further attacker involvement.

## Likelihood Explanation
The attack requires zero privileges — only the ability to send HTTP POST requests to `/api/v1/contracts/call`, a public, unauthenticated endpoint. The attacker needs no valid contract, no funds, and no special knowledge: any contract address and any 32-byte slot keys suffice. The setup phase (1,500 requests) completes in 3 seconds at the allowed rate. The gas throttle bypass (`gas ≤ 10,000 → 0 tokens consumed`) further reduces the cost of each setup request. The attack is fully scriptable, repeatable after each 5-minute `slotsPerContract` TTL window, and can be distributed across IPs to avoid IP-level firewall rules. [12](#0-11) 

## Recommendation

1. **Bound batch size to the current request context:** In `findStorageBatch()`, do not issue a batch query with all accumulated slot keys on every cache miss. Instead, limit the batch to a configurable maximum (e.g., 50–100 keys) or only include keys that were accessed within the current request's execution context.

2. **Align TTLs:** Set `slotsPerContract` TTL to match or be shorter than `contractStateCache` TTL (2 seconds), so slot keys do not outlive their corresponding values. Alternatively, evict slot keys from `slotsPerContract` when their values are evicted from `contractStateCache`.

3. **Fix the gas throttle bypass:** Change `scaleGas()` to return at least `1` for any positive gas value, or enforce a minimum gas floor on incoming requests (e.g., reject requests with `gas < 21,000`). [4](#0-3) 

4. **Add per-contract or per-IP rate limiting** to prevent a single source from filling the slot key cache for any given contract.

5. **Consider disabling batch caching by default** (`enableBatchContractSlotCaching=false`) until the TTL mismatch is resolved, as the feature flag exists precisely for this purpose. [13](#0-12)

    t.start()

import time

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
