Audit Report

## Title
Indefinite Subscription Resource Exhaustion via Long.MAX_VALUE consensusEndTime

## Summary
An unauthenticated client can send a `subscribeTopic()` gRPC request with `consensusEndTime.seconds >= 9223372035L`. `convertTimestamp()` silently promotes this to `Long.MAX_VALUE`, which causes `isComplete()` to permanently return `false` and `pastEndTime()` to never emit a terminal signal. The subscription never self-terminates. Because the `maxConcurrentCallsPerConnection` limit is per-connection only, an attacker can open arbitrarily many connections and inject up to `5N` permanent subscriptions, exhausting the shared `applicationTaskExecutor` thread pool.

## Finding Description

**Code path 1 — timestamp coercion**

`convertTimestamp()` in `ConsensusController.java` maps any client-supplied `seconds >= 9223372035L` to `Long.MAX_VALUE` without error or rejection: [1](#0-0) 

**Code path 2 — `isComplete()` never returns `true`**

`TopicContext.isComplete()` in `TopicMessageServiceImpl.java` evaluates `Instant.ofEpochSecond(0, Long.MAX_VALUE)`, which normalises to epoch-second `9223372036` (~year 2262). Adding `endTimeInterval` (30 s) still yields ~year 2262. `.isBefore(Instant.now())` is permanently `false`: [2](#0-1) 

**Code path 3 — `pastEndTime()` never emits**

Because `endTime != null`, the `Flux.never()` fast-path is skipped. The returned flux repeats `Long.MAX_VALUE` times at 30-second intervals but never emits an item, so `takeUntilOther(pastEndTime(...))` never terminates the live stream: [3](#0-2) 

**Code path 4 — `takeWhile` never terminates**

`t.getConsensusTimestamp() < Long.MAX_VALUE` is always `true` for any real message: [4](#0-3) 

**Validation bypass**

`isValidEndTime()` passes because `Long.MAX_VALUE > startTime` is always `true`. `isValidStartTime()` is unaffected. Neither constraint rejects the crafted value: [5](#0-4) 

## Impact Explanation
Each crafted subscription holds an executor thread indefinitely and schedules periodic work on `Schedulers.boundedElastic()` every 30 seconds. The `maxConcurrentCallsPerConnection = 5` limit is enforced per-connection only: [6](#0-5) [7](#0-6) 

There is no global cap on total active subscriptions. An attacker opening `N` connections injects up to `5N` permanent subscriptions, exhausting the `applicationTaskExecutor` pool and starving legitimate subscribers of threads (availability loss / DoS).

## Likelihood Explanation
No authentication is required. The gRPC port (default 5600) is publicly reachable. The crafted field is a standard protobuf `Timestamp` with `seconds` set to any value `>= 9223372035`. Any scripted client (`grpcurl`, a custom gRPC stub) can reproduce this. The attack is repeatable and scalable with minimal resources.

## Recommendation
1. **Reject, don't coerce.** In `convertTimestamp()`, return an error (throw a `StatusRuntimeException` with `INVALID_ARGUMENT`) instead of silently mapping overflow timestamps to `Long.MAX_VALUE`.
2. **Add a global subscription cap.** Enforce a server-wide limit on concurrent active subscriptions (e.g., via an `AtomicInteger` counter checked at subscription entry) in addition to the per-connection limit.
3. **Add an upper-bound validator.** In `TopicMessageFilter`, add a constraint that `endTime` must be `<= some reasonable future bound` (e.g., current time + N years in nanoseconds).

## Proof of Concept
```bash
# Using grpcurl with a crafted consensusEndTime.seconds >= 9223372035
grpcurl -plaintext -d '{
  "topicID": {"topicNum": 1},
  "consensusEndTime": {"seconds": 9223372035}
}' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Repeat across N connections to exhaust the thread pool
```
Each connection holds up to 5 permanent subscriptions. With enough connections the `applicationTaskExecutor` pool is saturated and legitimate `subscribeTopic` calls are denied.
