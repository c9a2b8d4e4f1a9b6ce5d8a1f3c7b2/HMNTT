Audit Report

## Title
Negative Caching of `Optional.empty()` in `findRuntimeBytecode` Permanently Poisons Contract Bytecode Lookup for Up to 1 Hour

## Summary
`ContractRepository.findRuntimeBytecode()` uses `@Cacheable` with `unless = "#result == null"`. When a contract does not yet exist in the DB, Spring Data JPA returns `Optional.empty()` — a non-null singleton — which passes the `unless` guard and is written into the Caffeine cache with a 1-hour `expireAfterAccess` TTL. Any request that arrives before the importer writes the contract row permanently poisons the cache entry for that contractId, causing all subsequent calls to return empty bytecode for up to one hour.

## Finding Description
**Exact location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18 [1](#0-0) 

The `@Cacheable` annotation uses `unless = "#result == null"`. Spring evaluates this SpEL expression after the method returns. When no contract row exists, Spring Data JPA returns `Optional.empty()`. Because `Optional.empty()` is a non-null singleton object, `#result == null` evaluates to `false`, and the empty `Optional` is stored in the cache.

**Cache TTL:**
`CacheProperties.java` line 22 defines: [2](#0-1) 

`expireAfterAccess=1h` means every cache read resets the expiry window, potentially extending the poisoned entry indefinitely under repeated access.

**Cache manager wiring:**
`EvmConfiguration.java` lines 67–73 builds the `CaffeineCacheManager` for the contract cache using this specification: [3](#0-2) 

**Downstream consumer:**
`ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode()` and maps the result with `.orElse(null)`: [4](#0-3) 

When the cache serves `Optional.empty()`, `.orElse(null)` returns `null`, and the EVM treats the address as having no code.

## Impact Explanation
Any contract deployed on Hedera becomes inaccessible through the mirror node's web3 API for up to 1 hour (or longer under repeated reads). `eth_getCode` returns `0x`, and all `eth_call`/`eth_estimateGas` requests to the contract fail as if the address is an EOA. This breaks every dApp or integration relying on the mirror node's JSON-RPC endpoint to interact with newly deployed contracts.

## Likelihood Explanation
No privileges are required. Hedera entity IDs are sequential integers, making the target `contractId` trivially predictable from the pending `ContractCreate` transaction visible on-chain. The attacker only needs to win a race against the importer's processing latency (typically seconds). The attack is repeatable: after the cache entry expires, the attacker can re-poison it. A single HTTP request is sufficient.

## Recommendation
Change the `unless` condition to also exclude empty `Optional` values:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only a present (non-empty) `Optional` is stored in the cache, while a missing contract causes a fresh DB query on every call until the row is written.

## Proof of Concept
The existing test `ContractRepositoryTest.findRuntimeBytecodeFailCall()` at line 34–38 already demonstrates that `findRuntimeBytecode` returns `Optional.empty()` for a non-existent contract ID: [5](#0-4) 

To confirm cache poisoning:
1. Call `findRuntimeBytecode(newContractId)` before the contract row exists → returns `Optional.empty()`, which is cached.
2. Insert the contract row into the DB.
3. Call `findRuntimeBytecode(newContractId)` again → cache hit returns `Optional.empty()` instead of querying the DB.
4. `ContractBytecodeReadableKVState.readFromDataSource()` maps this to `null`, causing the EVM to treat the address as having no code for up to 1 hour.
