Audit Report

## Title
Unbounded Memory Growth via `cacheManagerSlotsPerContract` Internal Cache Map in `findStorageBatch()`

## Summary
`ContractStateServiceImpl.findStorageBatch()` uses `contractSlotsCache` (bounded at 3000 entries) as a loader cache whose values are `CaffeineCache` instances retrieved from `cacheManagerSlotsPerContract`. Because `cacheManagerSlotsPerContract` is configured in dynamic mode (no `setCacheNames()`), Spring's `CaffeineCacheManager` permanently stores every created `CaffeineCache` in its internal `ConcurrentHashMap<String, Cache> cacheMap` with no eviction. Eviction from `contractSlotsCache` does not remove entries from `cacheManagerSlotsPerContract.cacheMap`, causing unbounded heap growth proportional to the number of distinct contract IDs ever seen.

## Finding Description

**Exact code path — verified:**

    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [1](#0-0) 

`EvmConfiguration.java` lines 107–112:
```java

``` [2](#0-1) 

`setCacheNames()` is **not** called — confirmed. Every other `CaffeineCacheManager` bean in the same file calls `setCacheNames()` to restrict to a fixed set of names; `cacheManagerSlotsPerContract` is the sole exception. [3](#0-2) 

**Root cause:**

Spring's `CaffeineCacheManager.getCache(name)` in dynamic mode (no pre-declared names) creates a new `CaffeineCache` and inserts it into an internal `ConcurrentHashMap<String, Cache> cacheMap` that has no size bound and no eviction policy. `contractSlotsCache` is configured as `expireAfterAccess=5m,maximumSize=3000`: [4](#0-3) 

When `contractSlotsCache` evicts an entry for a given `contractId`, the `CaffeineCache` object that was its value is dereferenced from `contractSlotsCache`, but `cacheManagerSlotsPerContract.cacheMap` retains a strong reference under the key `contractId.toString()`. The `CaffeineCache` is never garbage-collected.

Each per-contract cache is configured as `expireAfterAccess=5m,maximumSize=1500`: [5](#0-4) 

So each retained `CaffeineCache` carries a Caffeine `BoundedLocalCache` with its own `ConcurrentHashMap`, `MpscGrowableArrayQueue`, and scheduler references — even after the entries inside it have expired.

**Failed assumption:** The design assumes that bounding `contractSlotsCache` to 3000 entries bounds total memory. It does not, because the *manager-level* map that holds the `CaffeineCache` wrappers is separate from and unaffected by `contractSlotsCache` eviction.

## Impact Explanation

Each permanently retained `CaffeineCache` object (with its Caffeine internals) consumes on the order of 1–10 KB. With `contractSlotsCache` cycling through unique contract IDs, `cacheManagerSlotsPerContract.cacheMap` accumulates one entry per unique ID ever seen, growing without bound. At sustained load with many distinct contract IDs, this leads to progressive heap exhaustion and eventual `OutOfMemoryError`, crashing the web3 node process.

## Likelihood Explanation

Exploitation requires only unauthenticated HTTP POST requests to `/api/v1/contracts/call` with distinct contract addresses. Contract addresses can be enumerated sequentially. The global rate limit (500 req/s) is not per-source, so a single client can saturate it. The `contractSlotsCache` `maximumSize=3000` means that after 3000 unique IDs, eviction begins and the divergence between `contractSlotsCache` size and `cacheManagerSlotsPerContract.cacheMap` size starts growing immediately.

## Recommendation

1. **Call `setCacheNames()` on `cacheManagerSlotsPerContract`** with a fixed set of names to disable dynamic mode. This prevents `CaffeineCacheManager` from accepting arbitrary new cache names at runtime.
2. **Alternatively**, manage the per-contract `CaffeineCache` lifecycle manually (outside of `CaffeineCacheManager`) using a Caffeine cache of caches, so that eviction of a contract entry also releases the inner cache object.
3. **Add a removal listener** to `contractSlotsCache` that explicitly calls a cleanup method on `cacheManagerSlotsPerContract` when a contract entry is evicted, preventing stale `CaffeineCache` objects from accumulating.

## Proof of Concept

```
for i in $(seq 1 100000); do
  ADDR=$(printf "0x%040x" $i)
  curl -s -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"data\":\"0x\",\"to\":\"$ADDR\",\"estimate\":false}" &
done
```

Each request with a new `ADDR` triggers `cacheManagerSlotsPerContract.getCache(contractId.toString())`, adding a permanent entry to `cacheManagerSlotsPerContract.cacheMap`. After `contractSlotsCache` cycles past 3000 entries, the map divergence is observable via JVM heap dumps: `CaffeineCacheManager.cacheMap` will contain far more entries than `contractSlotsCache.maximumSize`, and the count grows monotonically with unique addresses seen.
