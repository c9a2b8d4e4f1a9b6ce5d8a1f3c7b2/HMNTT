# Audit Report

## Title
Log Injection via Unsanitized `X-Forwarded-For` / `X-Real-IP` Headers in `TracingMiddleware`

## Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` returns the raw, unvalidated value of the `X-Forwarded-For` or `X-Real-IP` header. `TracingMiddleware` interpolates this value directly into a log message and emits it via `log.Info()` without any sanitization of newline characters, carriage returns, or ANSI escape sequences. Any unauthenticated HTTP client can inject arbitrary content into the log stream, forging, overwriting, or suppressing log entries.

## Finding Description
**Code path:**

- `rosetta/app/middleware/trace.go`, `getClientIpAddress()`, lines 63–74: reads `X-Real-IP` then `X-Forwarded-For` verbatim. No length cap, no character-class check, no control-character stripping is performed. [1](#0-0) 

- `TracingMiddleware()`, lines 52–53: the raw return value is interpolated directly into the log message string via `fmt.Sprintf`. [2](#0-1) 

- Lines 55–59: for any path not in `internalPaths` (which covers only `/health/liveness`, `/health/readiness`, `/metrics`), the message is emitted at `log.Info` level. `/construction/submit` always falls into this branch. [3](#0-2) 

**Root cause:** The code implicitly assumes `X-Forwarded-For` / `X-Real-IP` contain only a valid IP address. This assumption is never enforced. Logrus's `log.Info(string)` emits the string as-is; it performs no escaping of embedded newlines or ANSI sequences.

**Failed check:** The test suite (`trace_test.go` lines 26–85) only exercises well-formed IP strings. There are no tests for control characters, ANSI sequences, or multi-line values. No middleware, no HTTP framework layer, and no logrus hook performs sanitization. [6](#0-5) 

## Impact Explanation
An attacker submitting a tampered Hedera transaction via `/construction/submit` can simultaneously inject a forged "success from a benign IP" log entry and corrupt or suppress the real entry. Automated SIEM rules keyed on IP address, path, or status code will fire on the fake entry — or not fire at all if the real entry is truncated. This directly undermines the audit trail for the highest-risk Rosetta endpoint (the one that broadcasts signed transactions to the network), making forensic investigation and real-time alerting unreliable.

## Likelihood Explanation
The precondition is trivially met: any HTTP client reachable by the Rosetta node can set arbitrary headers. No authentication, no API key, and no network privilege is required beyond the ability to reach the Rosetta port — the same requirement as submitting a transaction at all. The attack is repeatable on every request and leaves no reliable trace in the log. Standard HTTP libraries (`curl`, Python `requests`, Go `net/http`) all allow setting `X-Forwarded-For` freely.

## Recommendation
Sanitize the IP address value returned by `getClientIpAddress()` before it is used in any log message:

1. **Validate format:** After reading the header, attempt to parse the value with `net.ParseIP()`. If parsing fails, substitute a safe placeholder (e.g., `"invalid"`) rather than using the raw string.
2. **Strip control characters:** At minimum, remove or replace `\n`, `\r`, and ANSI escape sequences (`\x1b[...`) before logging.
3. **Add test coverage:** Add test cases to `trace_test.go` for headers containing embedded newlines, carriage returns, and ANSI sequences, asserting that the log output does not contain the injected content.

Example fix for `getClientIpAddress()`:
```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }
    // Validate: accept only if it parses as a valid IP
    if net.ParseIP(strings.TrimSpace(ipAddress)) == nil {
        return "invalid"
    }
    return strings.TrimSpace(ipAddress)
}
```

## Proof of Concept
Send the following request to the Rosetta node:

```
POST /construction/submit HTTP/1.1
Host: <rosetta-host>
X-Forwarded-For: 10.0.0.1\ntime="2024-01-01T00:00:00Z" level=info msg="10.0.0.1 POST /construction/submit (200) in 1ms"
Content-Type: application/json

{ ... }
```

**Expected log output (two lines written to the log sink):**
```
time="..." level=info msg="10.0.0.1
time="2024-01-01T00:00:00Z" level=info msg="10.0.0.1 POST /construction/submit (200) in 1ms" POST /construction/submit (200) in ...ms"
```

A log parser splitting on newlines sees a plausible, benign-looking entry on line 1 and a malformed/truncated real entry on line 2. For terminal-based viewers, a carriage-return payload (`\r192.168.1.1`) causes the real IP to be overwritten on-screen, hiding the attacker's address entirely.
