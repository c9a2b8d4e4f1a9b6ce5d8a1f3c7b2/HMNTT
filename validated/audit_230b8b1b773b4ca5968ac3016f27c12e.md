Audit Report

## Title
Missing `return` After Block Number Mismatch in `BlockAssembler.onEndOfBlock()` Allows Block Identity Corruption

## Summary
In `BlockAssembler.onEndOfBlock()` within `BlockNode.java`, when a mismatch is detected between the `BlockHeader` number and the `EndOfBlock` block number, `Utility.handleRecoverableError()` is called but no `return` follows. In the default configuration (`haltOnError=false`), execution falls through and the attacker-supplied `blockNumber` from `EndOfBlock` is used to construct the block's filename, causing the block to be stored under the wrong number and permanently skipping the legitimate block. In the `haltOnError=true` configuration, a `ParserException` propagates through the generic exception handler, triggering `onError()` and eventually marking the block node inactive.

## Finding Description

In `BlockNode.java`, `BlockAssembler.onEndOfBlock()` at lines 239–272:

```java
void onEndOfBlock(final BlockEnd blockEnd) {
    final long blockNumber = blockEnd.getBlockNumber();  // attacker-controlled
    if (pending.isEmpty()) {
        Utility.handleRecoverableError(...);
        return;  // ← return IS present here
    }

    final var blockHeader = pending.getFirst().getFirst().getBlockHeader();
    if (blockHeader.getNumber() != blockNumber) {
        Utility.handleRecoverableError(
                "Block number mismatch in BlockHeader({}) and EndOfBlock({})",
                blockHeader.getNumber(),
                blockNumber);
        // ← NO return — falls through unconditionally
    }

    // ... block assembly ...

    final var filename = BlockFile.getFilename(blockNumber, false);  // uses attacker-controlled blockNumber
    blockStreamConsumer.accept(new BlockStream(block, null, filename, loadStart));
}
```

The mismatch is detected at line 248, `handleRecoverableError` is called at lines 249–252, but there is no `return` at line 253. Execution unconditionally falls through to line 270, where `blockNumber` — sourced from `blockEnd.getBlockNumber()` — is used as the block's filename. The `pending.isEmpty()` guard at lines 241–245 does have a `return`, making the asymmetry a clear oversight.

`Utility.handleRecoverableError()` at lines 220–231 has two behaviors:
- `haltOnError=false` (default): logs the error and returns normally → execution continues.
- `haltOnError=true`: throws `ParserException` → caught by the generic `catch (Exception ex)` at lines 162–164 → `onError()` called → `BlockStreamException` thrown. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

## Impact Explanation

**Default mode (`haltOnError=false`) — primary impact:** Silent, persistent data corruption. The block's content (assembled from `pending`, which contains the correct `BlockHeader` with number N) is dispatched under filename derived from M (the attacker-supplied `EndOfBlock` number). The mirror node's sequence pointer advances to M+1, permanently skipping block N and all its transactions. All downstream consumers (REST API, gRPC API, event streams) serve incorrect or missing data for block N. This is undetected in normal operation since the error is only logged.

**`haltOnError=true` mode:** `ParserException` propagates through `streamBlocks`, `onError()` increments the error counter. After `maxSubscribeAttempts` consecutive triggers, the block node is marked inactive. Block N's transactions are never ingested. Repeated triggering across consecutive blocks can stall the entire importer pipeline. [5](#0-4) 

## Likelihood Explanation

**Preconditions:**
- Attacker must control a block node endpoint the mirror node subscribes to. This is achievable by registering a malicious block node (if HIP-1081 allows permissionless registration), performing MITM on an unencrypted gRPC channel, or compromising a legitimate block node operator.
- No cryptographic binding between `EndOfBlock.blockNumber` and the `BlockHeader` number is enforced by the mirror node.

**Repeatability:** One malformed `EndOfBlock` per block is sufficient. An attacker can corrupt or skip every block indefinitely with no additional effort.

**Likelihood: Medium** — requires control of a subscribed block node endpoint, but no authentication of `EndOfBlock.blockNumber` against the `BlockHeader` is performed.

## Recommendation

Add a `return` statement immediately after `handleRecoverableError` in the mismatch branch, mirroring the pattern already used in the `pending.isEmpty()` guard:

```java
if (blockHeader.getNumber() != blockNumber) {
    Utility.handleRecoverableError(
            "Block number mismatch in BlockHeader({}) and EndOfBlock({})",
            blockHeader.getNumber(),
            blockNumber);
    return;  // ← add this
}
```

Alternatively, use `blockHeader.getNumber()` instead of `blockNumber` on line 270 to ensure the filename always reflects the authoritative block header number, not the unverified `EndOfBlock` value. The `return` approach is preferable as it avoids processing a structurally inconsistent block entirely. [6](#0-5) 

## Proof of Concept

1. Configure the mirror node to subscribe to an attacker-controlled block node endpoint.
2. Attacker sends a valid `BlockItemSet` containing a `BlockHeader` with `number = N` (e.g., `N = 100`).
3. Attacker sends `EndOfBlock` with `blockNumber = M` where `M ≠ N` (e.g., `M = 200`).
4. `onEndOfBlock` is called: mismatch detected at line 248, `handleRecoverableError` logs the error (default mode), no `return` is executed.
5. Execution reaches line 270: `BlockFile.getFilename(200, false)` is called — block 100's content is stored as block 200.
6. The mirror node's next requested block is 201. Block 100 is permanently skipped.
7. Repeat for every subsequent block to corrupt the entire ingested chain sequence. [7](#0-6)
