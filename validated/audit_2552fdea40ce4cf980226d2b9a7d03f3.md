## Audit Report

## Title
Zero-pageSize Unbounded Database Query in `PollingTopicListener.poll()` When Delivered Count Reaches Filter Limit

## Summary
In `PollingTopicListener.poll()`, when the number of messages delivered by the listener equals the filter's limit, the computed `limit` becomes zero. `Math.min(0, maxPageSize)` yields `pageSize = 0`, which is set as the new filter's `limit`. Because `TopicMessageFilter.hasLimit()` returns `false` for `limit == 0`, `TopicMessageRepositoryCustomImpl.findByFilter()` never calls `setMaxResults()`, issuing an unbounded SQL query that loads all matching rows — bypassing the `maxPageSize` design constraint entirely.

## Finding Description

**Step 1 — Zero limit computed in `poll()`**

In `PollingTopicListener.poll()`, the remaining limit is computed as:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
``` [1](#0-0) 

When `context.getCount().get() == filter.getLimit()` (the listener has delivered exactly N messages for a limit of N), `limit = 0` and `pageSize = Math.min(0, 5000) = 0`. The `newFilter` is then built with `limit(pageSize)` = `limit(0)`.

**Step 2 — `hasLimit()` returns `false` for zero**

```java
public boolean hasLimit() {
    return limit > 0;
}
``` [2](#0-1) 

`limit = 0` → `hasLimit() = false`.

**Step 3 — `setMaxResults()` never called, unbounded query issued**

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [3](#0-2) 

Because `hasLimit()` is `false`, `setMaxResults()` is never called. The JPA `TypedQuery` executes with no `LIMIT` clause, returning every matching row from `startTime` onward for the given topic.

**Step 4 — How the listener receives its filter**

`TopicMessageServiceImpl.incomingMessages()` computes the remaining limit and passes it to `topicListener.listen(newFilter)`:

```java
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();
return topicListener.listen(newFilter).concatMap(t -> missingMessages(topicContext, t));
``` [4](#0-3) 

So `PollingTopicListener` receives a filter with `limit = remaining_messages_needed`. Its internal `context.getCount()` tracks messages delivered by that listener instance. When the listener delivers all `remaining_messages_needed` messages, `context.getCount() == filter.getLimit()`, and the next `poll()` computes `pageSize = 0`.

**Step 5 — Race condition enabling the extra poll**

The listener uses `repeatWhen(RepeatSpec.times(Long.MAX_VALUE)...)` with a fixed delay on `Schedulers.boundedElastic()`: [5](#0-4) 

The outer `flux.take(filter.getLimit())` in `subscribeTopic()` cancels after N total messages: [6](#0-5) 

Because the next `poll()` is scheduled asynchronously on the bounded-elastic scheduler, it can fire before the cancel signal propagates through the operator chain. That poll call computes `pageSize = 0` and issues the unbounded query.

**Root cause:** The code assumes `filter.getLimit() - context.getCount().get()` is always strictly positive when `filter.hasLimit()` is true. It does not guard against the zero case. `Math.min(0, maxPageSize) = 0` silently produces a filter value that `hasLimit()` treats as "no limit."

## Impact Explanation

The unbounded JPA query (`getResultList()` with no `setMaxResults()`) loads all `topic_message` rows from `startTime` onward for the subscribed topic into the JVM heap. [7](#0-6) 

For a high-volume topic this can be millions of rows, causing:
- **Memory exhaustion** on the mirror node gRPC server (OOM).
- **Database CPU/IO spike** from a full table scan with no LIMIT.
- **Denial of service** for other subscribers sharing the same JVM and database connection pool.

The user does not receive more data than their limit (the outer `take(N)` cancels after the first element of the unbounded result), so this is a **server-side resource exhaustion issue**, not a data exfiltration issue. Severity: **Medium**.

## Likelihood Explanation

- Requires `listenerProperties.type = POLL` (non-default; default is `REDIS`). [8](#0-7) 
- Any subscriber can trigger this by setting any positive `limit` and waiting for messages to arrive — no special privileges required.
- The race window is bounded by the polling interval (≥50 ms, default 500 ms) — small but reproducible under load or with crafted subscriptions.
- An attacker can open many concurrent subscriptions with small limits to amplify the effect.

## Recommendation

Guard against the zero case in `poll()` before issuing the query. The simplest fix is to return `Flux.empty()` immediately when the computed `limit` is ≤ 0:

```java
private Flux<TopicMessage> poll(PollingContext context) {
    TopicMessageFilter filter = context.getFilter();
    TopicMessage last = context.getLast();
    int limit = filter.hasLimit()
            ? (int) (filter.getLimit() - context.getCount().get())
            : Integer.MAX_VALUE;
    if (limit <= 0) {
        return Flux.empty(); // limit exhausted; cancel will arrive shortly
    }
    int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
    long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
    var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
    return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
}
``` [9](#0-8) 

This ensures that even if `poll()` fires after the limit is exhausted, no database query is issued.

## Proof of Concept

1. Deploy the mirror node with `hiero.mirror.grpc.listener.type=POLL`.
2. Subscribe via gRPC `subscribeTopic` with `limit = N` (any positive integer) for a topic that has at least N messages.
3. Wait for N messages to be delivered. The outer `take(N)` sends a cancel upstream.
4. Due to the 500 ms polling interval and the bounded-elastic scheduler, the next scheduled `poll()` fires before the cancel propagates.
5. `poll()` computes `limit = N - N = 0`, `pageSize = 0`, builds `newFilter` with `limit = 0`.
6. `findByFilter(newFilter)` is called; `hasLimit()` returns `false`; `setMaxResults()` is never called.
7. The database executes `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with no `LIMIT` clause, loading all matching rows into the JVM heap.
8. Repeat with many concurrent subscriptions to amplify memory and database pressure.
