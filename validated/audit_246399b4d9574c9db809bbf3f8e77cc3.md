Audit Report

## Title
Missing `@Validated` on `AllowancesController` Renders `@Max(MAX_LIMIT)` Unenforced, Allowing Unbounded DB Result Sets

## Summary
`AllowancesController` declares `@Max(MAX_LIMIT)` and `@Positive` on the `limit` request parameter, but the class lacks `@Validated`. Spring MVC requires `@Validated` on the controller class to activate `MethodValidationInterceptor` for method-level Bean Validation. Without it, all JSR-380 annotations on method parameters are ignored at runtime. An unauthenticated caller can supply `?limit=2147483647`, which flows unmodified into a raw jOOQ `.limit(request.getLimit())` call, causing the database to return an unbounded result set.

## Finding Description

**Controller — constraint declared but never enforced:**

`AllowancesController` at line 61 declares `@Positive @Max(MAX_LIMIT)` on the `limit` parameter: [1](#0-0) 

The class-level annotations are only `@CustomLog`, `@RequestMapping`, `@RequiredArgsConstructor`, and `@RestController` — `@Validated` is absent: [2](#0-1) 

A grep across the entire controller package confirms `@Validated` appears nowhere in any controller class.

**DTO — no independent guard:**

`NftAllowanceRequest.limit` carries no validation annotation; it is a plain `int` with a default of 25: [3](#0-2) 

**Service — no limit check:**

`NftAllowanceServiceImpl.getNftAllowances()` performs only parameter-relationship checks and passes the request object directly to the repository with no limit capping: [4](#0-3) 

**Repository — raw limit propagation:**

`NftAllowanceRepositoryCustomImpl.findAll()` calls `.limit(request.getLimit())` with no capping or sanity check before issuing the SQL query: [5](#0-4) 

**Root cause:** `@Max(MAX_LIMIT)` is a Bean Validation annotation that requires `@Validated` on the declaring class to be activated by Spring's `MethodValidationPostProcessor`. Its absence means the annotation is compile-time decoration only. `MAX_LIMIT = 100` is completely bypassed: [6](#0-5) 

## Impact Explanation

An attacker can force the mirror node to execute a query equivalent to `SELECT * FROM nft_allowance WHERE owner = ? AND approved_for_all = true LIMIT 2147483647`, returning every matching row. This causes:
- **Memory exhaustion** in the JVM — all rows are materialized via `fetchInto` into a Java collection
- **Database CPU/IO spike** on every such request
- **Effective DoS** against the REST API with no authentication required

## Likelihood Explanation

- The endpoint `/api/v1/accounts/{id}/allowances/nfts` is public and requires no authentication
- The exploit is a single HTTP GET with `?limit=2147483647` — trivially repeatable
- Account IDs are enumerable from public ledger data
- The attack can be parallelized across many accounts to amplify resource exhaustion

## Recommendation

Add `@Validated` to `AllowancesController` at the class level:

```java
@CustomLog
@RequestMapping(value = "/api/v1/accounts/{id}/allowances", produces = APPLICATION_JSON)
@RequiredArgsConstructor
@RestController
@Validated  // <-- add this
public class AllowancesController {
```

As a defense-in-depth measure, also add an explicit cap in the service or repository layer:

```java
int safeLimit = Math.min(request.getLimit(), Constants.MAX_LIMIT);
```

## Proof of Concept

```
GET /api/v1/accounts/0.0.1234/allowances/nfts?limit=2147483647 HTTP/1.1
Host: <mirror-node-host>
```

Without `@Validated`, the `@Max(100)` constraint is not enforced, the value `2147483647` is bound directly into `NftAllowanceRequest.limit`, and the repository executes `.limit(2147483647)` against the database, returning an unbounded result set.
