Audit Report

## Title
Race Between `computeIfAbsent` and `unsubscribe()` in `RedisTopicListener` Causes Orphaned Flux and Incorrect Map Eviction

## Summary
`RedisTopicListener.getSharedListener()` uses `ConcurrentHashMap.computeIfAbsent` to retrieve or create a shared `Flux` per topic. The `unsubscribe()` cleanup callback, triggered asynchronously via `doOnCancel`, removes the map entry using the single-argument `remove(key)` rather than the atomic `remove(key, value)`. This means a stale cancellation callback can evict a newly-inserted, currently-active `Flux` from the map, orphaning its subscribers and opening a duplicate Redis pub/sub connection per race cycle.

## Finding Description
In `RedisTopicListener.java`:

- **`getSharedListener` (line 61):** `topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic))` atomically inserts a new `Flux` only if the key is absent, but does not prevent a concurrent `doOnCancel` from removing a freshly-inserted entry.
- **`subscribe` / `doOnCancel` (line 74):** `.doOnCancel(() -> unsubscribe(topic))` is placed upstream of `.share()`. When `share()` detects zero subscribers and cancels its upstream, the cancel signal propagates through the operator chain and fires `doOnCancel` asynchronously in the cancelling thread — not under any lock coordinated with `computeIfAbsent`.
- **`unsubscribe` (line 83):** `topicMessages.remove(topic.getTopic())` performs a key-only removal. There is no check that the value being removed is the same `Flux` instance that triggered the callback.

**Root cause:** The `ConcurrentHashMap` guarantees atomicity per individual operation, but the compound sequence — "last subscriber cancels → `share()` cancels upstream → cancel signal propagates → `doOnCancel` fires → `remove(key)`" — is not atomic with respect to a concurrent "new subscriber → `computeIfAbsent` → insert new entry." The missing primitive is `remove(key, value)` (the two-argument form), which would make the eviction conditional on the value still being the triggering instance.

**Exploit flow:**
1. `Flux_A` is the active shared stream for `"topic.1"` in `topicMessages`.
2. The last subscriber to `Flux_A` cancels. `share()` detects zero subscribers and begins cancelling its upstream. The cancel signal is propagating through the operator chain toward `doOnCancel` in Thread A.
3. Before Thread A reaches `doOnCancel`, Thread B calls `computeIfAbsent("topic.1", ...)` — finds `Flux_A` still in the map — returns it. Thread B's subscriber subscribes to `Flux_A`; `share()` re-subscribes to the upstream, opening a new Redis connection.
4. Thread A's cancel signal reaches `doOnCancel` → `topicMessages.remove("topic.1")` — `Flux_A` is evicted from the map, even though it now has an active subscriber.
5. Thread C calls `computeIfAbsent("topic.1", ...)` — key absent — creates `Flux_B` and inserts it.
6. Thread B's subscriber eventually cancels `Flux_A`. `Flux_A`'s `doOnCancel` fires → `topicMessages.remove("topic.1")` — this removes `Flux_B` (the current canonical entry), not `Flux_A`.
7. All subsequent subscribers create `Flux_C`, and the cycle repeats. Subscribers on `Flux_A` and `Flux_B` are orphaned, each holding an independent Redis subscription invisible to the map and to each other.

## Impact Explanation
- **Message loss / delivery integrity:** Orphaned `Flux` instances each hold their own private Redis pub/sub subscription. Messages published after the split are not visible across all subscribers for the same topic — a core correctness guarantee of the HCS streaming feature is broken.
- **Resource exhaustion:** Each race cycle opens an additional Redis pub/sub connection for the same topic channel. Under repeated triggering, this exhausts Redis connection limits and server-side file descriptors, degrading or denying service to all users.
- **Cascading eviction:** Each orphaned `Flux`'s eventual cancellation incorrectly evicts the current canonical map entry, perpetuating fragmentation indefinitely without any self-healing mechanism.

## Likelihood Explanation
Exploitation requires only the ability to open and cancel gRPC streams to the publicly accessible `subscribeTopic` endpoint — no credentials or privileged access needed. The race window is narrow under normal conditions but widens significantly during Redis partition recovery (elevated scheduler latency, reconnection backoff), which is precisely when the `retryWhen` logic is active and cancellation/resubscription events are most frequent. The attack is trivially scriptable with any gRPC client by repeatedly subscribing and immediately cancelling.

## Recommendation
Replace the key-only `remove` in `unsubscribe` with the atomic two-argument form `ConcurrentHashMap.remove(key, value)`, capturing the specific `Flux` instance at creation time:

```java
private Flux<TopicMessage> subscribe(Topic topic) {
    Duration interval = listenerProperties.getInterval();
    // Use an array to capture the self-reference before return
    @SuppressWarnings("unchecked")
    Flux<TopicMessage>[] ref = new Flux[1];
    ref[0] = container
            .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
            .map(Message::getMessage)
            .doOnCancel(() -> topicMessages.remove(topic.getTopic(), ref[0]))
            .doOnComplete(() -> topicMessages.remove(topic.getTopic(), ref[0]))
            .doOnError(t -> log.error("Error listening for messages", t))
            .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
            .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
            .share();
    return ref[0];
}
```

`ConcurrentHashMap.remove(key, value)` is atomic and only removes the entry if the current value equals the provided instance, eliminating the stale-eviction race entirely.

## Proof of Concept
```
Thread A (last subscriber cancels Flux_A):
  share() → cancel upstream → [propagating through retryWhen, doOnSubscribe, doOnError, doOnComplete, doOnCancel...]

Thread B (new subscriber, concurrent):
  computeIfAbsent("topic.1") → Flux_A still in map → returns Flux_A
  Flux_A.subscribe(newSubscriber) → share() re-subscribes upstream → new Redis connection opened

Thread A (doOnCancel fires):
  topicMessages.remove("topic.1")  // removes Flux_A — but Flux_A has an active subscriber!

Thread C (another new subscriber):
  computeIfAbsent("topic.1") → key absent → creates Flux_B → inserts Flux_B

Thread B (subscriber eventually cancels Flux_A):
  doOnCancel fires → topicMessages.remove("topic.1")  // removes Flux_B, not Flux_A!

// Result: Flux_A and Flux_B are both orphaned with independent Redis subscriptions.
// Flux_C will be created next, and the cycle repeats.
```
