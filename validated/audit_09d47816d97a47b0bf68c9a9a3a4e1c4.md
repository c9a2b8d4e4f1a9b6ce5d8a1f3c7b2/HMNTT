Audit Report

## Title
Binary Search Gas Estimation Multiplier Bypasses Throttle: Up to 21× EVM Execution Per Single Throttle-Deducted Request

## Summary
`ThrottleManagerImpl.throttle()` deducts gas tokens from the bucket exactly once per request based on `request.getGas()`. When `estimate=true`, `ContractExecutionService.estimateGas()` performs one initial EVM execution followed by up to 20 additional binary-search iterations via `BinaryGasEstimator.search()`. Each binary-search iteration calls `doProcessCall(params, gas, true)`, which — by design of the `if (!estimate)` guard — skips `restoreGasToBucket` and never deducts additional tokens from the throttle bucket. A single unauthenticated request therefore causes up to 21 full EVM executions while consuming only one request's worth of throttle tokens.

## Finding Description

**Entry point** — `ContractController.call()`: [1](#0-0) 

`throttleManager.throttle(request)` is called once, consuming `scaleGas(request.getGas())` tokens from the gas bucket: [2](#0-1) 

`scaleGas` divides by `GAS_SCALE_FACTOR = 10_000`, so `gas=15,000,000` → 1,500 tokens consumed: [3](#0-2) 

**Estimate path** — `ContractExecutionService.estimateGas()`:

- Step 1: `callContract(params, context)` → `doProcessCall(params, params.getGas(), false)` — `estimate=false`, so `restoreGasToBucket` runs and returns some tokens.
- Step 2: `binaryGasEstimator.search(gas -> doProcessCall(params, gas, true), ...)` — up to `maxGasEstimateRetriesCount = 20` iterations: [4](#0-3) 

**Root cause** — `doProcessCall` with `estimate=true` skips the `restoreGasToBucket` call entirely, and no additional tokens are deducted from the throttle bucket for any binary-search iteration: [5](#0-4) 

The `maxGasEstimateRetriesCount` default is 20: [6](#0-5) 

The binary search loop condition confirms up to 20 iterations: [7](#0-6) 

**Failed assumption**: The throttle assumes one gas deduction ≈ one EVM execution. For estimate requests, the ratio is 1 deduction : up to 21 executions.

## Impact Explanation

With `gas=15,000,000` (the configured maximum via `maxGasLimit`) and bytecode that consumes gas proportional to the gas provided:

| Metric | Value |
|---|---|
| Throttle tokens deducted | 1,500 (net ~1,400 after initial-call restore) |
| Actual EVM executions | up to 21 (1 initial + up to 20 binary search) |
| Throttle-equivalent work | ~14,000,000 gas |
| Actual gas-equivalent work | significantly higher (each binary search iteration executes the EVM with a midpoint gas value) |
| Effective multiplier | up to ~21× in execution count |

Note: The binary search uses midpoint gas values (between `gasUsedByInitialCall` and `params.getGas()`), so individual iterations consume less than the maximum gas. However, the total EVM work across all iterations still substantially exceeds what the throttle accounts for. Repeated at the rate limit (500 req/s default), this can saturate node CPU and memory, causing denial of service for all other users. [8](#0-7) 

## Likelihood Explanation

- **No authentication required**: any external user can POST to `/api/v1/contracts/call`
- **No special knowledge needed**: setting `estimate=true` and providing any gas-consuming bytecode is sufficient
- **Repeatable**: the rate-limit bucket (500 req/s default) still allows 500 such amplified requests per second
- **Bytecode is not validated** for computational complexity before the binary search begins
- The `requestTimeout = 10000ms` provides some bound, but 10 seconds of EVM execution per request at 500 req/s is still a significant amplification vector

## Recommendation

1. **Deduct throttle tokens per binary-search iteration**: Before each call inside `BinaryGasEstimator.search()`, deduct `scaleGas(mid)` tokens from the gas bucket. Restore them after the iteration completes (mirroring the non-estimate path).

2. **Alternatively, deduct a multiplier upfront**: In `ThrottleManagerImpl.throttle()`, when `request.isEstimate() == true`, deduct `scaleGas(request.getGas()) * maxGasEstimateRetriesCount` tokens upfront to account for the worst-case binary search cost.

3. **Cap binary search iterations more aggressively**: Reduce `maxGasEstimateRetriesCount` from 20 to a lower value, or make it configurable with a lower default for unauthenticated callers.

4. **Add per-iteration gas accounting in `doProcessCall`**: When `estimate=true`, still call `throttleManager.restore()` (or a deduct equivalent) proportional to the gas used in that iteration, so the throttle accurately reflects actual EVM work.
