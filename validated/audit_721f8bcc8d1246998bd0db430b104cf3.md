# Audit Report

## Title
Default Gas Value Equals Maximum Gas Limit, Enabling Throttle Bucket Exhaustion with Minimal Request Crafting

## Summary
`ContractCallRequest` defaults `gas` to `15_000_000`, which is identical to the configured `maxGasLimit`. Because `ThrottleManagerImpl.throttle()` deducts gas tokens from the bucket **before** execution and **before** per-request gas validation, every request omitting the `gas` field silently claims the maximum possible tokens. The restore path only runs after execution completes, leaving a window where the bucket is fully drained.

## Finding Description

**Root cause — default gas equals the maximum:**

`ContractCallRequest.java` line 37 sets the default: [1](#0-0) 

`EvmProperties.java` line 69 sets the ceiling: [2](#0-1) 

Both values are `15_000_000L`. Any request omitting `gas` silently claims the maximum allowed gas.

**Token consumption path — throttle before validation:**

In `ContractController.call()`, throttling occurs first, then validation: [3](#0-2) 

`ThrottleManagerImpl.throttle()` immediately deducts scaled tokens from the gas bucket: [4](#0-3) 

**Why `validateContractMaxGasLimit` never triggers an early restore for the default value:**

The validation uses a strict `>` comparison: [5](#0-4) 

Since `15_000_000 > 15_000_000` is `false`, `InvalidParametersException` is never thrown for the default value. The `catch` block that restores tokens: [6](#0-5) 
...is never reached for these requests.

**Why the post-execution restore does not prevent exhaustion:**

`restoreGasToBucket()` in `ContractCallService.java` only runs in the `finally` block after execution completes: [7](#0-6) 

The restore logic itself: [8](#0-7) 

During the execution window — which can span tens to hundreds of milliseconds — all tokens are already consumed. Any concurrent legitimate request arriving during that window receives `GAS_PER_SECOND_LIMIT_EXCEEDED`. The restore is sequential relative to the execution, not relative to incoming concurrent requests.

## Impact Explanation
Any unauthenticated user can send a flood of syntactically valid, minimal POST requests (e.g., `{"to":"0x00000000000000000000000000000000000004e2"}`) and monopolize the entire gas throttle budget. All other users — including legitimate callers who specify a reasonable gas value — receive HTTP 429 `Gas per second rate limit exceeded.` for the duration of the attack. The global rate-limit bucket does not protect against this because each attacker request is individually valid and consumes a rate-limit slot along with the maximum gas tokens simultaneously.

## Likelihood Explanation
No authentication, API key, or IP-based rate limiting is required. The attacker needs only a valid `to` address (any 20-byte hex value) and the ability to sustain high-frequency HTTP POST requests, achievable from a single machine or small botnet. The attack is trivially scriptable, repeatable indefinitely, and requires zero knowledge of contract internals.

## Recommendation
1. **Decouple the default from the maximum.** Set the default `gas` in `ContractCallRequest` to a conservative value (e.g., `50_000` or `300_000`) rather than `15_000_000`. The maximum should remain a ceiling, not a default.
2. **Move validation before throttling.** Reorder `ContractController.call()` so `validateContractMaxGasLimit(request)` is called before `throttleManager.throttle(request)`, eliminating the need for the post-validation restore path entirely.
3. **Use `>=` in the gas limit check** if the intent is that `gas` must be strictly less than `maxGasLimit`, or document clearly that equality is permitted.

## Proof of Concept
```bash
# Send 500 minimal requests per second with no gas field (defaults to 15_000_000)
for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x00000000000000000000000000000000000004e2"}' &
done
wait
# All concurrent legitimate requests with explicit gas values will receive:
# HTTP 429: "Gas per second rate limit exceeded."
```

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
