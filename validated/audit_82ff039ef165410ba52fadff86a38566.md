# Audit Report

## Title
Gas Throttle Bypass via Deliberate Out-of-Gas Failure in `restoreGasToBucket()`

## Summary
`ContractCallService.restoreGasToBucket()` fully restores the gas throttle bucket for any transaction where `gasUsed == gasLimit` (i.e., an Out-of-Gas failure). With the default `gasLimitRefundPercent = 100`, the net gas cost to the throttle bucket for every OOG transaction is zero. An unauthenticated attacker can craft OOG transactions at the maximum gas limit, completely neutralizing the gas throttle and submitting far more EVM computation per second than the throttle is designed to permit.

## Finding Description

**File:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractCallService.java`
**Method:** `restoreGasToBucket()`, lines 140–152; `doProcessCall()`, lines 109–138

On every request, `ThrottleManagerImpl.throttle()` consumes `scaleGas(gasLimit)` tokens from the gas bucket:

```java
// ThrottleManagerImpl.java:40
gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))
```

In the `finally` block of `doProcessCall()`, `restoreGasToBucket(result, params.getGas())` is called unconditionally (line 129). Inside that method:

```java
// ContractCallService.java:143–145
final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
    throttleManager.restore(gasLimitToRestoreBaseline);
```

With `gasLimitRefundPercent = 100` (the default in `ThrottleProperties.java:22`), `gasLimitToRestoreBaseline == gasLimit`. The `restore()` method applies the same `scaleGas()` scaling as `throttle()`, so the exact number of tokens consumed on entry is returned to the bucket. **Net gas cost to the bucket = 0.**

The condition `gasLimit == result.gasUsed()` is precisely the definition of an Out-of-Gas exception — the EVM ran until it consumed every unit of gas provided.

The secondary `result == null` branch (line 144) also triggers a full restore when pre-EVM validation throws `IllegalStateException` or `IllegalArgumentException` (line 120–121), though that path involves no EVM computation and is a lesser concern.

## Impact Explanation

The gas throttle (`gasLimitBucket`) is the primary mechanism for capping total EVM computation per second. With the bypass active, an attacker submitting OOG transactions at the maximum gas limit (e.g., 15M gas) causes the EVM to execute 15M gas worth of opcodes per request while the gas bucket is fully refunded after each one. The only remaining constraint is the RPS throttle (500 req/s), which is independent and does not account for per-request computational cost. This enables a sustained CPU exhaustion attack at 500 req/s × 15M gas = 7.5B gas/s of actual EVM execution, regardless of the configured `gasPerSecond` limit (default 7.5B gas/s — but that limit is now entirely unenforced for OOG traffic). The mirror node's EVM execution threads can be saturated with no special privileges.

## Likelihood Explanation

The `/api/v1/contracts/call` endpoint requires no authentication, API key, or request signing. Crafting an OOG transaction requires only targeting any contract with a loop or providing a gas value just below what a known contract requires. This is trivially scriptable. The default `gasLimitRefundPercent = 100` means the bypass is active in all default deployments without any configuration change.

## Recommendation

1. **Change the default `gasLimitRefundPercent`** to a value significantly below 100 (e.g., 0 or 25). A failed OOG transaction still consumed real EVM computation; refunding 100% of the gas limit is incorrect.
2. **Decouple the refund condition from `gasUsed == gasLimit`**: OOG failures should not be treated as "free" requests. Consider charging the actual `gasUsed` regardless of success/failure, and only refunding the *unused* portion (`gasLimit - gasUsed`), which for OOG is zero.
3. **Apply the same restore cap to failed transactions as to successful ones**: the `else` branch at line 149 already correctly caps the restore at `Math.min(gasRemaining, gasLimitToRestoreBaseline)`. The OOG branch should follow the same logic — `gasRemaining = 0` for OOG, so the restore should be 0.

## Proof of Concept

1. Deploy or identify any contract with an infinite loop (or any contract call that exhausts all provided gas).
2. Send repeated `eth_call` requests to `/api/v1/contracts/call` with `gas = 15000000` targeting that contract.
3. Each call causes the EVM to execute until OOG (`gasUsed == gasLimit`).
4. `restoreGasToBucket()` restores `15000000 * 100 / 100 = 15000000` gas to the bucket — exactly what was consumed.
5. The gas bucket level is unchanged. Repeat immediately at 500 req/s (the RPS limit).
6. Observe that the gas throttle never triggers (`GAS_PER_SECOND_LIMIT_EXCEEDED` is never thrown) regardless of how many requests are sent, while the EVM executes at full capacity on every request.

```java
    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
```
