Audit Report

## Title
Unbounded `data` Field Serialization in LOG Action Causes Heap Exhaustion via Repeated Large Requests

## Summary
`ThrottleManagerImpl.throttle()` passes requests through rate and gas bucket checks that are blind to payload size. When a matching `RequestProperties` filter with `action = LOG` is configured, `log.info("{}", request)` is called unconditionally, forcing full serialization of `ContractCallRequest` including a `data` field bounded only at ~1 MB. At the default rate of 500 req/s this creates ~500 MB/s of short-lived `String` allocations, driving GC pressure and heap occupancy well above normal operating levels.

## Finding Description

**Verified code path:**

**Step 1 — `ThrottleManagerImpl.throttle()`**: After both bucket checks pass, every configured `RequestProperties` whose `test()` returns `true` triggers `action()`. [1](#0-0) 

**Step 2 — `ThrottleManagerImpl.action()`**: The `LOG` branch executes `log.info("{}", request)`. SLF4J parameterized logging calls `request.toString()` (Lombok `@Data`-generated, includes all fields) after confirming INFO level is enabled — which it is in typical production deployments. [2](#0-1) 

**Step 3 — `ContractCallRequest.data`**: Annotated `@Hex` with no explicit `maxLength`, so `HexValidator` enforces the annotation default of `1_048_576L` hex characters (~512 KB binary / ~1 MB as a Java `String`). [3](#0-2) [4](#0-3) 

**Step 4 — Default `action = LOG`**: Any operator-added `RequestProperties` that omits an explicit action silently defaults to `LOG`. [5](#0-4) 

**Root cause:** Neither the rate-limit bucket (counts requests) nor the gas-limit bucket (counts gas units) accounts for the byte length of `data`. The LOG action unconditionally serializes the entire request object, including the arbitrarily large `data` string, on every matching request. No truncation or size guard exists anywhere in the logging path.

**Failed assumption:** The code assumes requests passing the two bucket checks are "cheap" to log. This is false when `data` approaches its maximum allowed size.

## Impact Explanation

At the default rate limit of 500 req/s with `data` at the 1 MB ceiling:

- **String allocation rate:** 500 req/s × ~1 MB/req ≈ 500 MB/s of short-lived `String` objects created solely for log formatting.
- **GC pressure:** The JVM young-generation heap fills and is collected at a rate proportional to this allocation, causing frequent minor GCs and potential promotion to old-gen, triggering major/full GCs.
- **Heap inflation:** Concurrent in-flight log strings and their backing `byte[]` arrays inflate live heap well beyond normal operating levels.

The default `requestsPerSecond = 500` and `request = List.of()` are confirmed in the source. [6](#0-5) 

## Likelihood Explanation

- **No authentication required** to POST to the contract-call endpoint.
- **Precondition — LOG filter configured:** The default `request` list is empty, so the attack only activates when an operator has added at least one `RequestProperties` entry. However, the default `action` is `LOG`, making it easy for operators to inadvertently enable this path when adding any filter for observability purposes.
- **Staying within rate limits:** The attacker need only send ≤500 req/s with large `data` values — well within the capability of a single machine — to sustain the attack indefinitely.
- **Default `rate = 100`** in `RequestProperties` means 100% of matching requests are logged with no sampling reduction. [7](#0-6) 

## Recommendation

1. **Truncate `data` in log output**: Override `toString()` in `ContractCallRequest` (or use a custom Lombok `@ToString` exclude/include) to cap the logged `data` field at a safe length (e.g., 256 characters).
2. **Add a size guard before logging**: In `ThrottleManagerImpl.action()`, check `request.getData()` length before calling `log.info`.
3. **Use lazy log suppliers**: Replace `log.info("{}", request)` with `log.info("{}", () -> truncate(request))` to avoid any allocation when INFO is disabled.
4. **Reduce the `@Hex` default `maxLength`** on `ContractCallRequest.data` to a value consistent with actual EVM calldata limits rather than the generic validator default.

## Proof of Concept

```bash
# Requires: operator has configured at least one RequestProperties with action=LOG (or default)
# Sends 500 req/s with ~1 MB data payloads
DATA=$(python3 -c "print('0x' + 'aa' * 524288)")  # 1,048,576 hex chars = 512 KB binary

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"$DATA\",\"to\":\"0x$(printf '%040x' 1)\",\"gas\":15000000}" &
done
wait
```

Each request passes the rate-limit and gas-limit bucket checks, then triggers `log.info("{}", request)` which materializes a ~1 MB `String` per request via Lombok's `toString()`. [8](#0-7)
