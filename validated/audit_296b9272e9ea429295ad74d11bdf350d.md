Audit Report

## Title
Long Overflow in `getInclusiveValue()` Causes Semantic Bypass: `GT:Long.MAX_VALUE` Returns All Nodes Instead of Empty Set

## Summary
`NumberRangeParameter.getInclusiveValue()` performs unchecked `value + 1` arithmetic on a user-supplied `long`. Supplying `node.id=gt:9223372036854775807` (Long.MAX_VALUE) causes the addition to silently overflow to `Long.MIN_VALUE`. `Math.max(0L, Long.MIN_VALUE)` clamps `lowerBound` back to `0`, the guard `lowerBound > upperBound` never fires, and the repository query executes with `lowerBound=0, upperBound=Long.MAX_VALUE` — returning every node in the table instead of the semantically correct empty set.

## Finding Description

**Overflow site** — `NumberRangeParameter.getInclusiveValue()`:

```java
// NumberRangeParameter.java, line 36-37
if (operator == RangeOperator.GT) {
    return value + 1;   // Long.MAX_VALUE + 1 == Long.MIN_VALUE (silent wrap)
``` [1](#0-0) 

**Missing upper-bound check** — `getNumberValue()` only rejects negative values; `Long.MAX_VALUE` (9223372036854775807) passes cleanly:

```java
// NumberRangeParameter.java, lines 26-33
private static long getNumberValue(String number) {
    var value = Long.parseLong(number);
    if (value < 0) {
        throw new IllegalArgumentException("Invalid range value");
    }
    return value;
}
``` [2](#0-1) 

**`hasLowerBound()` returns `true` for `GT`**, confirmed in `RangeParameter.java`: [3](#0-2) 

**Consumption in `getNetworkNodes()`** — step-by-step trace:

```java
// NetworkServiceImpl.java, lines 107-121
long lowerBound = 0L;
long upperBound = Long.MAX_VALUE;

for (final var nodeIdParam : nodeIdParams) {
    ...
    } else if (nodeIdParam.hasLowerBound()) {
        lowerBound = Math.max(lowerBound, nodeIdParam.getInclusiveValue());
        // GT:Long.MAX_VALUE → getInclusiveValue() = Long.MIN_VALUE
        // Math.max(0L, Long.MIN_VALUE) = 0L  ← overflow clamped back to 0
    }
}

if (lowerBound > upperBound) {   // 0 > Long.MAX_VALUE → false → no exception
    throw new IllegalArgumentException(...);
}
// Query fires: lowerBound=0, upperBound=Long.MAX_VALUE → full-table scan
``` [4](#0-3) 

The same overflow path exists in `resolveRegisteredNodeIdBounds()` for the registered-nodes endpoint. [5](#0-4) 

## Impact Explanation
- **Semantic correctness:** The filter `GT:Long.MAX_VALUE` has no valid result in the domain; the correct response is an empty set. The overflow silently converts it into an unfiltered full-table scan.
- **Resource exhaustion:** Any unauthenticated client can issue a single crafted request (`?node.id=gt:9223372036854775807`) to force a full-table scan on the `address_book_entry` / node tables, consuming database CPU and I/O with no predicate filtering. This is an application-logic bug causing resource exhaustion — distinct from volumetric DDoS (which is excluded by SECURITY.md) — and is therefore in scope as a griefing vector.
- Network node data is public, so there is no confidentiality breach.

## Likelihood Explanation
- Zero privileges required; the `/api/v1/network/nodes` endpoint is publicly accessible.
- Payload is a single, trivially constructed query parameter.
- Fully repeatable and scriptable with no authentication or rate-limit bypass needed.

## Recommendation
1. **Add an upper-bound check in `getNumberValue()`**: reject values that would overflow when incremented, e.g., reject `value == Long.MAX_VALUE` when the operator is `GT` (or use `Math.addExact` and catch `ArithmeticException`).
2. **Use `Math.addExact` / `Math.subtractExact`** in `getInclusiveValue()` so overflow throws `ArithmeticException` rather than wrapping silently.
3. Apply the same fix to `resolveRegisteredNodeIdBounds()` which contains an identical pattern.

## Proof of Concept

```
GET /api/v1/network/nodes?node.id=gt:9223372036854775807
```

Expected: `[]` (empty array — no node id can exceed Long.MAX_VALUE).  
Actual: Full list of all network nodes returned (lowerBound=0, upperBound=Long.MAX_VALUE passed to the repository query).
