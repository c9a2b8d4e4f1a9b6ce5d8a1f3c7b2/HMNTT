Audit Report

## Title
Unbounded Base32 Alias Input Bypasses Index, Enabling Unauthenticated DoS via Full Table Scan

## Summary
The GraphQL `account` query accepts an `alias` input validated only by a character-set regex with no maximum length constraint. An attacker can submit an arbitrarily long valid Base32 string that decodes to a byte array exceeding PostgreSQL's B-tree index key size limit (~2712 bytes), forcing the `findByAlias` query to fall back to a sequential scan on the `entity` table. Repeated unauthenticated requests can exhaust database resources.

## Finding Description

**Verified code path:**

**Step 1 — No length bound on `alias` in the GraphQL schema.**
`account.graphqls` line 90 applies only a character-set pattern; there is no `@Size(max=...)`:
```graphql
alias: String @Pattern(regexp: "^[A-Z2-7]+$")
``` [1](#0-0) 

**Step 2 — `decodeBase32` performs no length check on input or output.** [2](#0-1) 

**Step 3 — Decoded bytes are passed directly to the repository with no guard.** [3](#0-2) 

**Step 4 — Native query performs an equality comparison against the full byte array.** [4](#0-3) 

**Step 5 — The `entity__alias` index is a standard B-tree partial index.**
```sql
create index if not exists entity__alias
    on entity (alias) where alias is not null;
``` [5](#0-4) 

**Root cause:** PostgreSQL B-tree indexes cannot store or match keys larger than ~2712 bytes (for the default 8 KB page size). When `?1` is a `bytea` value exceeding this limit, PostgreSQL cannot use `entity__alias` and falls back to a sequential scan of the entire `entity` table. Base32 encodes 5 bytes per 8 characters, so a string of ~4340 valid `[A-Z2-7]` characters decodes to >2712 bytes — well within what the unbounded `@Pattern` allows. No existing check (`@Pattern`, `decodeBase32`, `getByAliasAndType`) imposes any length restriction.

## Impact Explanation
Every sequential scan on a large `entity` table consumes significant CPU and I/O on the database server. Because the GraphQL endpoint is publicly accessible with no authentication, an attacker can flood the service with alias-lookup requests carrying oversized payloads. This exhausts the database connection pool and query executor threads, causing legitimate queries that depend on the same `entity` table to time out or queue indefinitely — a sustained, targeted DoS against the mirror node's read path.

## Likelihood Explanation
No authentication or API key is required to reach the GraphQL `account` query. The attack requires only knowledge of the public GraphQL schema (introspectable by default) and the ability to send HTTP POST requests. The payload is trivially constructed: a string of repeated `A` characters of length ≥ 4340. The attack is fully repeatable and scriptable.

## Recommendation
Add a `@Size(max = N)` constraint alongside the existing `@Pattern` on the `alias` field in `AccountInput`. A valid Hedera account alias is derived from a public key (ED25519 = 32 bytes → 56 Base32 chars; ECDSA secp256k1 = 33 bytes → 56 Base32 chars), so a conservative maximum of 100 characters is more than sufficient and eliminates the index-bypass vector entirely:

```graphql
alias: String @Pattern(regexp: "^[A-Z2-7]+$") @Size(max = 100)
```

Optionally, add a defensive length check in `decodeBase32` or `getByAliasAndType` as a defense-in-depth measure.

## Proof of Concept
```bash
# Construct a payload of 4340 'A' characters (decodes to >2712 bytes)
ALIAS=$(python3 -c "print('A' * 4340)")

curl -X POST https://<mirror-node>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}"
```

Sending this request repeatedly (e.g., via a simple loop or tool like `ab` or `wrk`) will force repeated sequential scans on the `entity` table, saturating database I/O and blocking legitimate queries.
