Audit Report

## Title
Missing `callDataId` Resolution in `addDefaultEthereumTransactionContractResult()` Causes Empty `functionParameters` for Failed Ethereum Transactions with Offloaded Call Data

## Summary
`ContractResultServiceImpl.addDefaultEthereumTransactionContractResult()` does not resolve the `callDataId` file reference when building a default `ContractResult` for failed Ethereum transactions. When a transaction uses the call-data-offloading mechanism (`callDataId` set, inline `callData` null), the function silently stores `DomainUtils.EMPTY_BYTE_ARRAY` as `functionParameters`, producing a permanently incorrect record in the `contract_result` table.

## Finding Description

**Vulnerable code path** — `ContractResultServiceImpl.addDefaultEthereumTransactionContractResult()`:

```java
var ethereumTransaction = recordItem.getEthereumTransaction();
var functionParameters = ethereumTransaction.getCallData() != null
        ? ethereumTransaction.getCallData()
        : DomainUtils.EMPTY_BYTE_ARRAY;   // ← callDataId is never consulted
``` [1](#0-0) 

This function is reached when the transaction is not `CONTRACTCALL`/`CONTRACTCREATEINSTANCE` **and** the `ContractFunctionResult` in the record is the default (empty) instance: [2](#0-1) 

The guard at lines 100–107 only skips `DUPLICATE_TRANSACTION`, `WRONG_NONCE`, and successful transactions — all other failure codes proceed to the buggy assignment: [3](#0-2) 

**Correct code path (not used here)** — `EthereumTransactionHandler.updateContractResult()` correctly resolves `callDataId` for blockstream records:

```java
byte[] callData = ethereumTransaction.getCallData();
var callDataId = ethereumTransaction.getCallDataId();
if (ArrayUtils.isEmpty(callData) && !EntityId.isEmpty(callDataId)) {
    callData = contractBytecodeService.get(callDataId);
    ...
}
contractResult.setFunctionParameters(callData != null ? callData : ArrayUtils.EMPTY_BYTE_ARRAY);
``` [4](#0-3) 

However, this path is only reached when `isContractFunctionResultSet(functionResult)` is true (i.e., a non-default `ContractFunctionResult` exists), which is not the case for the failure scenarios that route to `addDefaultEthereumTransactionContractResult()`. [5](#0-4) 

**Root cause:** `ContractResultServiceImpl` does not inject `ContractBytecodeService` at all: [6](#0-5) 

The `callDataId` is set during `EthereumTransactionHandler.doUpdateTransaction()` when `body.hasCallData()` is true: [7](#0-6) 

## Impact Explanation

The `contract_result.function_parameters` column is written once and never corrected. Any downstream consumer of the mirror node REST API (`/api/v1/contracts/results/{id}`) will receive `"function_parameters": "0x"` for these transactions instead of the actual ABI-encoded call data.

Additionally, `OpcodeServiceImpl.getCallDataBytes()` falls back to reading `contractResult.getFunctionParameters()` when `ethereumTransaction` is null: [8](#0-7) 

If `functionParameters` is empty due to this bug, opcode debugging/replay for these transactions will also receive incorrect (empty) call data.

**Severity: Medium** — permanent data integrity corruption for a well-defined class of transactions; no fund loss.

## Likelihood Explanation

Any unprivileged Hedera account can trigger this:
1. Create a Hedera file containing ABI-encoded call data (`FileCreate`/`FileAppend`).
2. Submit an `EthereumTransaction` with the `callData` FileID set in the body (the `hasCallData()` branch in `doUpdateTransaction()` at line 107).
3. Ensure the transaction fails with any status other than `DUPLICATE_TRANSACTION` or `WRONG_NONCE` (e.g., `INSUFFICIENT_GAS`, `CONTRACT_REVERT_EXECUTED`).
4. The mirror node importer processes the record, calls `addDefaultEthereumTransactionContractResult()`, and persists `functionParameters = []`.

This is repeatable, requires no elevated privileges, and costs only standard Hedera transaction fees.

## Recommendation

Inject `ContractBytecodeService` into `ContractResultServiceImpl` and add `callDataId` resolution inside `addDefaultEthereumTransactionContractResult()`, mirroring the logic already present in `EthereumTransactionHandler.updateContractResult()` (lines 78–92):

```java
var ethereumTransaction = recordItem.getEthereumTransaction();
byte[] functionParameters = ethereumTransaction.getCallData();
var callDataId = ethereumTransaction.getCallDataId();
if (ArrayUtils.isEmpty(functionParameters) && !EntityId.isEmpty(callDataId)) {
    functionParameters = contractBytecodeService.get(callDataId);
}
if (functionParameters == null) {
    functionParameters = DomainUtils.EMPTY_BYTE_ARRAY;
}
``` [1](#0-0) 

## Proof of Concept

1. Call `FileCreate` on Hedera to create a file containing ABI-encoded call data (e.g., `0xa9059cbb...`).
2. Construct an `EthereumTransaction` (EIP-1559 or legacy) with the `to` field set to a contract address and `callData` field in the Hedera transaction body set to the FileID from step 1. Leave the inline `data` field in the RLP-encoded Ethereum transaction empty or minimal.
3. Submit the transaction such that it fails (e.g., target a non-existent contract, or use insufficient gas).
4. After the mirror node processes the record, query `/api/v1/contracts/results/{transactionId}`.
5. Observe `"function_parameters": "0x"` in the response instead of the actual ABI call data from the file.

The mirror node will have called `addDefaultEthereumTransactionContractResult()` (confirmed by the guard at lines 100–107 passing for non-`DUPLICATE_TRANSACTION`/`WRONG_NONCE` failures), stored `DomainUtils.EMPTY_BYTE_ARRAY` as `functionParameters`, and the record is now permanently incorrect.     private void addDefaultEthereumTransactionContractResult(RecordItem recordItem, Transaction transaction) {
        if (!entityProperties.getPersist().isContractResults()) {
            return;
        }

        var status = recordItem.getTransactionRecord().getReceipt().getStatus();
        if (recordItem.isSuccessful()
                || status == ResponseCodeEnum.DUPLICATE_TRANSACTION
                || status == ResponseCodeEnum.WRONG_NONCE
                || !recordItem.getTransactionBody().hasEthereumTransaction()) {
            // Don't add default contract result for the transaction if it's successful, or the result is
            // DUPLICATE_TRANSACTION, or the result is WRONG_NONCE, or it's not an ethereum transaction
            return;
        }

        var ethereumTransaction = recordItem.getEthereumTransaction();
        var functionParameters = ethereumTransaction.getCallData() != null
                ? ethereumTransaction.getCallData()
                : DomainUtils.EMPTY_BYTE_ARRAY;
        var payerAccountId = transaction.getPayerAccountId();
        var contractResult = ContractResult.builder()
                .callResult(DomainUtils.EMPTY_BYTE_ARRAY)
                .consensusTimestamp(transaction.getConsensusTimestamp())
                .contractId(0)
                .functionParameters(functionParameters)
                .gasLimit(ethereumTransaction.getGasLimit())
                .gasUsed(0L)
                .payerAccountId(payerAccountId)
                .senderId(payerAccountId)
                .transactionHash(ethereumTransaction.getHash())
                .transactionIndex(transaction.getIndex())
                .transactionNonce(transaction.getNonce())
                .transactionResult(transaction.getResult())
                .build();
        entityListener.onContractResult(contractResult);
    }
```
