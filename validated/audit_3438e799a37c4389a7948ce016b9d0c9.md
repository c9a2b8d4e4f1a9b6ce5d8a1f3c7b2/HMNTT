Audit Report

## Title
Unbounded Timestamp Range Enables Full Historical Scan of `hook_storage_change` via Unauthenticated GET Endpoint

## Summary
The `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` endpoint accepts a user-controlled `timestamp` parameter that is forwarded to `Bound.of()` with no minimum-range or maximum-span enforcement. Supplying `timestamp=gte:0` sets an effective scan window of `[0, Long.MAX_VALUE]`, triggering a full historical scan of the `hook_storage_change` table. No authentication is present on the controller or anywhere in the application's security configuration.

## Finding Description

**Step 1 — Parameter acceptance with only cardinality limit.**

`HooksController.getHookStorage()` accepts `TimestampParameter[] timestamps` constrained only by `@Size(max = 2)` (at most two values), with no value-range enforcement. [1](#0-0) 

**Step 2 — Verbatim forwarding to `Bound.of()`.**

At line 186, the timestamps array is passed directly to `Bound.of()` with the `CONSENSUS_TIMESTAMP` field. [2](#0-1) 

**Step 3 — `Bound.of()` always passes `primarySortField = false`.**

`Bound.of()` constructs the `Bound` with `primarySortField = false`. [3](#0-2) 

**Step 4 — Range-validity check is gated on `primarySortField`.**

Inside the `Bound` constructor, the only range-validity check is:
```java
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException(...);
}
```
Because `primarySortField` is `false` for timestamp bounds, **no range validation fires**. [4](#0-3) 

**Step 5 — Missing upper bound defaults to `Long.MAX_VALUE`.**

When only a lower bound is supplied (`timestamp=gte:0`), `adjustUpperBound()` returns `Long.MAX_VALUE`. [5](#0-4) 

**Step 6 — Non-empty `Bound` activates historical mode.**

`HookStorageRequest.isHistorical()` returns `true` whenever `timestamp.isEmpty()` is `false`, i.e., whenever `lower != null || upper != null`. [6](#0-5) [7](#0-6) 

**Step 7 — Repository query performs full scan with unbounded range.**

The native query in `HookStorageChangeRepository` uses `consensus_timestamp between :timestampLowerBound and :timestampUpperBound`. With `[0, Long.MAX_VALUE]`, the database must scan every row for the given `(owner_id, hook_id)` pair. The `DISTINCT ON (key)` clause requires evaluating all matching rows to find the most recent value per key before the `LIMIT` can be applied. [8](#0-7) 

**Step 8 — No authentication anywhere in the application.**

A grep for `@PreAuthorize`, `SecurityFilterChain`, `permitAll`, and `authenticated` across all Java source files returns zero matches. No `SecurityConfig` class exists. The controller carries no security annotations. [9](#0-8) 

**Note on the 19-digit literal:** `Long.MAX_VALUE` (19 digits) is rejected by `TimestampParameter`'s regex `\d{1,17}`. However, `timestamp=gte:0` achieves an identical effective range `[0, Long.MAX_VALUE]` and is fully accepted. [10](#0-9) 

## Impact Explanation
Any caller can force the service into historical mode with an unbounded time window `[0, Long.MAX_VALUE]`. The database must scan the entire `hook_storage_change` table for the given `(owner_id, hook_id)` pair. The `DISTINCT ON (key)` aggregation requires evaluating all rows before the `LIMIT` is applied, so the `limit` parameter does not prevent the full index/table scan. Repeated requests constitute a low-cost denial-of-service against the database. Additionally, the response exposes the complete storage-change history for any `(ownerId, hookId)` pair to any caller.

## Likelihood Explanation
The endpoint requires no authentication — no `@PreAuthorize`, no `SecurityFilterChain`, and no security configuration exists anywhere in the `rest-java` module. The exploit requires a single HTTP GET request with one query parameter (`timestamp=gte:0`). It is trivially repeatable by any external user with network access, making automated abuse straightforward.

## Recommendation
1. **Enforce a maximum timestamp span.** Reject requests where `upper - lower` exceeds a configurable maximum (e.g., 30 days in nanoseconds) when `primarySortField` is `false`.
2. **Require an upper bound when a lower bound is supplied.** If only `gte:X` is provided, either reject the request or implicitly cap the upper bound at `now()`.
3. **Add authentication/authorization.** Apply `@PreAuthorize` or a `SecurityFilterChain` rule to restrict access to this endpoint to authenticated callers.
4. **Add a database-level query timeout** to bound the worst-case cost of any single request.

## Proof of Concept
```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=25
```
This single unauthenticated request sets `timestampLowerBound = 0` and `timestampUpperBound = Long.MAX_VALUE` (9223372036854775807), activating historical mode and triggering a full scan of `hook_storage_change` for `(owner_id=1234, hook_id=1)`. Repeating this request in a loop constitutes a denial-of-service against the backing database.
