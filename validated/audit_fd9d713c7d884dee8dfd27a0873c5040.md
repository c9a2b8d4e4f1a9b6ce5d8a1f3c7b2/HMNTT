Audit Report

## Title
Missing `deleted` Filter on `entity_history` Branch in `findActiveByEvmAddressAndTimestamp` Returns Deleted Entities as Active at Historical Timestamps

## Summary
`findActiveByEvmAddressAndTimestamp` and `findActiveByEvmAddressOrAliasAndTimestamp` apply `deleted is not true` only to the `entity` branch of their UNION queries, but omit any deleted check on the `entity_history` branch. This is directly inconsistent with `findActiveByIdAndTimestamp`, which correctly applies `deleted is not true` to both branches. As a result, an `entity_history` record with `deleted = true` can be returned as "active" at a historical timestamp when queried by EVM address.

## Finding Description

**Confirmed code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85 (`findActiveByEvmAddressAndTimestamp`): [1](#0-0) 

The `entity` branch at line 70 has `where e.deleted is not true`, but the `entity_history` branch at lines 76–81 has only a timestamp filter (`lower(eh.timestamp_range) <= ?2`) and an ID filter — **no deleted check**. [2](#0-1) 

The same omission exists in `findActiveByEvmAddressOrAliasAndTimestamp`, lines 109–116: [3](#0-2) 

Compare with `findActiveByIdAndTimestamp` (lines 136–155), which correctly applies `and deleted is not true` to **both** branches: [4](#0-3) 

The `entity_cte` (lines 60–65) also has no deleted check — it only requires `evm_address = ?1 and created_timestamp <= ?2` — so it resolves the entity ID even for currently-deleted entities: [5](#0-4) 

**Why the existing test (line 126–132) does not catch this:**

```java
void findHistoricalEntityByEvmAddressAndTimestampRangeAndDeletedTrueCall() {
    final var entityHistory = persistEntityHistoryWithDeleted();
    assertThat(entityRepository.findActiveByEvmAddressAndTimestamp(
                    entityHistory.getEvmAddress(), entityHistory.getTimestampLower() - 1))
            .isEmpty();
}
``` [6](#0-5) 

`persistEntityHistoryWithDeleted()` creates **only** an `EntityHistory` row — no `entity` table row: [7](#0-6) 

Because `entity_cte` selects from the `entity` table only, it returns nothing when no entity row exists, making the entire query return empty regardless of any deleted check. The test also uses `getTimestampLower() - 1` (a timestamp **before** the record), which would exclude the record by timestamp even if the CTE did return something. Neither condition exercises the actual deleted-check gap.

**Exploit flow:**

1. Entity A (EVM address `0xABCD`) is created at T1. `entity` table: `deleted = NULL`.
2. Entity A is modified at T2. Old state written to `entity_history` with `deleted = NULL`, `timestamp_range = [T1, T2)`.
3. Entity A is deleted at T3. `entity` table: `deleted = true`. Deletion record written to `entity_history` with `deleted = true`, `timestamp_range = [T3, ∞)`.
4. Caller queries `findActiveByEvmAddressAndTimestamp(0xABCD, T3)`.
5. `entity_cte`: finds entity A's ID (`created_timestamp T1 <= T3`, no deleted check) — returns the ID.
6. Entity branch: `deleted = true` → filtered out → **no row**.
7. `entity_history` branch: most recent record with `lower(timestamp_range) <= T3` is the deletion record (`deleted = true`, `lower = T3`) → **returned with no deleted check**.
8. `ORDER BY timestamp_range DESC LIMIT 1` → deletion record selected.
9. The function returns the deleted entity as if it were active.

## Impact Explanation
An unprivileged caller can issue historical EVM queries (`eth_call`, `eth_getBalance`, `eth_estimateGas` at a past block) targeting the EVM address of a deleted account or contract. The mirror node will resolve the entity as active, returning stale state (balance, bytecode, storage) that should be inaccessible. This enables:
- Incorrect historical balance reporting for deleted accounts, which can mislead off-chain systems relying on mirror node historical data.
- Incorrect historical contract existence/bytecode resolution, enabling replays of historical calls against contracts that were deleted at the queried timestamp.
- Manipulation of transaction history reconstruction tools that rely on `findActiveByEvmAddressAndTimestamp` to determine entity liveness at a given block.

Severity: **Medium**. No direct on-chain fund movement is possible, but the integrity of historical state served by the mirror node is compromised.

## Likelihood Explanation
Any unprivileged user with knowledge of a deleted entity's EVM address can trigger this by issuing a standard JSON-RPC historical query. No special credentials, private keys, or privileged access are required. The EVM addresses of deleted entities are publicly visible on-chain and in the mirror node's REST API. The attack is trivially repeatable and requires no timing precision beyond knowing the deletion timestamp (also publicly available).

## Recommendation
Add `and eh.deleted is not true` to the `entity_history` branch in both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- findActiveByEvmAddressAndTimestamp (and findActiveByEvmAddressOrAliasAndTimestamp)
select *
from entity_history eh
where lower(eh.timestamp_range) <= ?2
and eh.id = (select id from entity_cte)
and eh.deleted is not true          -- ADD THIS
order by lower(eh.timestamp_range) desc
limit 1
```

Additionally, add a test that:
1. Persists both an `entity` row **and** an `entity_history` row with `deleted = true` for the same ID.
2. Queries `findActiveByEvmAddressAndTimestamp` at a timestamp `>= lower(entity_history.timestamp_range)`.
3. Asserts the result is empty.

This mirrors the existing test `findHistoricalEntityByIdAndTimestampRangeAndDeletedTrueCall` (line 244–250) which correctly covers `findActiveByIdAndTimestamp`: [8](#0-7) 

## Proof of Concept

```java
@Test
void findHistoricalEntityByEvmAddressAndTimestampRangeAndDeletedTrueCallActualGap() {
    // 1. Persist an entity row (so entity_cte resolves the ID)
    final var entity = domainBuilder.entity()
            .customize(e -> e.deleted(true))
            .persist();

    // 2. Persist an entity_history row with deleted=true for the same ID
    final var entityHistory = domainBuilder.entityHistory()
            .customize(e -> e.id(entity.getId())
                             .evmAddress(entity.getEvmAddress())
                             .deleted(true))
            .persist();

    // 3. Query at a timestamp >= lower(entity_history.timestamp_range)
    //    Without the fix, this returns the deleted entity_history record.
    //    With the fix, this must return empty.
    assertThat(entityRepository.findActiveByEvmAddressAndTimestamp(
                    entity.getEvmAddress(), entityHistory.getTimestampLower()))
            .isEmpty(); // FAILS on unpatched code
}
```
