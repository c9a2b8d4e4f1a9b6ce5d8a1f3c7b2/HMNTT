## Audit Report

## Title
32-bit `Arrays.hashCode` Cache Key Collision Enables Alias Cache Poisoning in `findByEvmAddressOrAliasAndDeletedIsFalse`

## Summary
`findByEvmAddressOrAliasAndDeletedIsFalse` and `findByEvmAddressAndDeletedIsFalse` use `@spelHelper.hashCode(#alias)` — which resolves to Java's `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole Caffeine cache key. Two distinct byte arrays that produce the same 32-bit integer share a cache slot. An unprivileged caller who queries entity `X` whose address hash-collides with address `B` will cause the cache to serve entity `X` to any subsequent caller querying `B`, silently redirecting alias resolution without a database query.

## Finding Description
**Exact locations:**
- `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–43
- `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22

`SpelHelper.hashCode(byte[])` delegates to `Arrays.hashCode(value)`:
```java
// SpelHelper.java:20-22
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```
`Arrays.hashCode` computes `result = 31*result + b[i]` for each byte, producing a 32-bit integer. The `@Cacheable` annotation stores the result under this integer key:
```java
// EntityRepository.java:39-43
@Cacheable(
    cacheNames = CACHE_NAME_ALIAS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
```
Spring autoboxes the returned `int` to `Integer` and uses it as the Caffeine map key. Two different `byte[]` inputs with the same `Arrays.hashCode` value collide on the same cache slot. The `unless = "#result == null"` guard only prevents caching null results; it does not verify that a cache hit's entity matches the queried `#alias`.

**Collision construction:** For any byte array `A` of length ≥ 2, a colliding array `B` can be constructed in O(1) by modifying two adjacent bytes at positions `i` and `i+1`:
```
B[i]   = A[i]   + 1
B[i+1] = A[i+1] - 31
```
This satisfies `31*(B[i]-A[i]) + (B[i+1]-A[i+1]) = 31 - 31 = 0`, so `Arrays.hashCode(A) == Arrays.hashCode(B)` exactly, provided both bytes remain in `[0, 255]`.

**Exploit flow:**
1. Attacker identifies entity `X` with EVM address `A` (public on-chain data).
2. Attacker analytically constructs `B` such that `Arrays.hashCode(A) == Arrays.hashCode(B)`, where `B` is the address a victim will query.
3. Attacker sends an `eth_call` resolving address `A` → `findByEvmAddressOrAliasAndDeletedIsFalse(A)` executes → entity `X` is stored in cache under key `K = hashCode(A)`.
4. Victim queries address `B` → `findByEvmAddressOrAliasAndDeletedIsFalse(B)` → cache key `K = hashCode(B) = K` → **cache hit returns entity `X`** instead of the correct entity, no DB query is made.

## Impact Explanation
Any caller of `CommonEntityAccessor.get(Bytes alias, Optional.empty())` hits the poisoned cache path. A poisoned entry causes the EVM execution layer to resolve the wrong contract or account for a given alias. Concretely: a `CALL` or `STATICCALL` targeting address `B` will execute against the code and storage of entity `X` instead. This enables silent misdirection of contract calls, incorrect balance/nonce reads, and wrong authorization checks — all without any privileged access. The poisoned entry persists until TTL expiry (governed by `cacheProperties.getEntity()`), after which the attacker can trivially re-poison by re-querying `A`.

## Likelihood Explanation
The attack requires no special privileges — only the ability to send JSON-RPC requests (e.g., `eth_call`). Hedera entity addresses are public and enumerable via the mirror node REST API. The collision construction is analytical and O(1): given any known address `A`, a colliding address `B` is computed by adjusting two adjacent bytes. No brute-force search is needed. The attack is repeatable after cache eviction.

## Recommendation
Replace the 32-bit integer hash with a collision-resistant key. The simplest correct fix is to use the full byte array as the cache key by relying on Spring's default key generation (which uses `Arrays.equals` semantics via `SimpleKey`) or by encoding the byte array as a hex string:

```java
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"
// or simply remove the key attribute and let Spring use the byte[] directly
// via a KeyGenerator that wraps it in a ByteArrayWrapper with proper equals/hashCode
```

Alternatively, implement a `KeyGenerator` bean that wraps `byte[]` in a type with a proper `equals`/`hashCode` contract (e.g., `ByteBuffer.wrap(alias)`), and register it on the `@Cacheable` annotation via `keyGenerator`. The same fix must be applied to `findByEvmAddressAndDeletedIsFalse` at lines 32–37.

## Proof of Concept
```java
// Construct a colliding address analytically
byte[] A = realEntity.getEvmAddress(); // e.g., {0x01, 0x02, ...}
byte[] B = Arrays.copyOf(A, A.length);
B[0] = (byte)(A[0] + 1);   // +1
B[1] = (byte)(A[1] - 31);  // -31

// Verify: Arrays.hashCode(A) == Arrays.hashCode(B)
assert Arrays.hashCode(A) == Arrays.hashCode(B);

// Step 1: poison the cache

// Step 2: victim queries B — gets entity X from cache instead of entity Y
Optional<Entity> result = entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(B);
assert result.get().getId().equals(entityX.getId()); // wrong entity returned
```
