# Audit Report

## Title
Unbounded Timestamp Range in `getHookStorageChange` Enables DB Exhaustion via Full Historical Scan

## Summary
`HookServiceImpl.getHookStorageChange` accepts user-supplied timestamp bounds with no maximum-range validation. A caller who supplies a maximally wide timestamp range (e.g., `timestamp=gte:0` with no upper bound) causes the underlying `DISTINCT ON (key)` native SQL query to sort and process every matching row in `hook_storage_change` for the targeted `owner_id`/`hook_id` pair before the `LIMIT` clause is applied, making per-request DB cost proportional to the full history of that pair.

## Finding Description
`HookServiceImpl.getHookStorageChange` (lines 84–118) extracts timestamp bounds from the user-controlled `Bound` object without any span-width check:

```java
final var timestamp = request.getTimestamp();
final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue(); // returns 0 if no lower bound
final long timestampUpperBound = timestamp.adjustUpperBound();           // returns Long.MAX_VALUE if no upper bound
``` [9](#0-8) 

These values are forwarded verbatim to both repository methods: [10](#0-9) 

Both repository queries use `DISTINCT ON (key)` with `consensus_timestamp between :timestampLowerBound and :timestampUpperBound`: [5](#0-4) 

`HookStorageRequest.getPageRequest()` injects `ORDER BY key ASC, consensus_timestamp DESC` for the historical path: [11](#0-10) 

PostgreSQL's `DISTINCT ON` requires the engine to sort **all** rows matching the `WHERE` clause by `(key, consensus_timestamp DESC)` before it can emit the first distinct-key row. The `LIMIT` (default 25) imposed by `PageRequest` is applied only *after* this full sort completes.

**Why existing checks fail:**
- The `limit` field caps output rows, not the rows the DB must read/sort.
- `Bound` validates operator cardinality and lower ≤ upper ordering only for primary sort fields; timestamp is passed as `primarySortField = false` in `Bound.of()`, so no lower ≤ upper check is enforced for it either. [12](#0-11) 
- `QueryProperties` exposes only `maxFileAttempts`; there is no `maxTimestampRange` or equivalent. [13](#0-12) 

## Impact Explanation
An attacker who knows any valid `ownerId`/`hookId` pair can issue repeated requests with a maximally wide timestamp range. Each request forces a full sort of `hook_storage_change` rows for that owner/hook, consuming DB I/O, CPU (sort), and connection pool slots. Sustained at modest request rates this degrades or denies service to all other API consumers sharing the same database. Confidentiality and integrity are unaffected; availability impact is **High**.

## Likelihood Explanation
No authentication or authorization is required on the endpoint. [7](#0-6)  Valid `ownerId` and `hookId` values are observable from any public block explorer or the mirror node's own `/hooks` endpoint. The attack requires only a single `curl` loop and is fully repeatable. No rate limiting is visible in the service or repository layer.

## Recommendation
1. **Enforce a maximum timestamp span** in `HookStorageRequest` or `HookServiceImpl`. Reject or clamp requests where `timestampUpperBound - timestampLowerBound` exceeds a configurable maximum (e.g., add a `maxTimestampRange` to `QueryProperties`).
2. **Require an explicit upper timestamp bound** when the historical path is taken (`isHistorical() == true`), returning HTTP 400 if only a lower bound is supplied.
3. **Add a composite index** on `(owner_id, hook_id, key, consensus_timestamp DESC)` so that even wide-range queries can use an index skip scan rather than a full sort, reducing the blast radius of any remaining unbounded queries.
4. **Apply API-level rate limiting** at the gateway or filter layer for the `/hooks/{hookId}/storage` endpoint.

## Proof of Concept
```bash
# Obtain any valid ownerId and hookId from the public /hooks endpoint
OWNER=0.0.12345
HOOK_ID=1
BASE_URL=https://<mirror-node-host>

# Issue repeated requests with a maximally wide timestamp range
while true; do
  curl -s "${BASE_URL}/api/v1/accounts/${OWNER}/hooks/${HOOK_ID}/storage?timestamp=gte:0&limit=25" &
done
```

Each request triggers `findByKeyBetweenAndTimestampBetween` with `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`, forcing PostgreSQL to sort the entire `hook_storage_change` history for that owner/hook pair on every call. [5](#0-4)

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
