## Audit Report

## Title
BufferUnderflowException via Short EVM Address in `getByEvmAddressAndType()`

## Summary
`EntityServiceImpl.getByEvmAddressAndType()` unconditionally reads 20 bytes from a `ByteBuffer` wrapping the decoded EVM address. Because `decodeEvmAddress()` enforces no length constraint, a crafted short even-length hex string (e.g., 32 hex chars = 16 bytes) that begins with 12 zero bytes passes the first two buffer reads but causes the third `buffer.getLong()` to throw an unchecked `BufferUnderflowException`, crashing the GraphQL resolver for every such request.

## Finding Description

**`decodeEvmAddress()`** in `GraphQlUtils.java` strips the optional `0x` prefix and delegates directly to `Hex.decodeHex()` with no length validation: [1](#0-0) 

The returned byte array of any even length is immediately wrapped in a `ByteBuffer` in `getByEvmAddressAndType()`, which then performs three sequential reads requiring exactly 20 bytes: [2](#0-1) 

For a 16-byte (32 hex char) all-zero input:
- `buffer.getInt()` → reads bytes 0–3, returns 0 → condition `== 0` passes
- `buffer.getLong()` → reads bytes 4–11, returns 0 → condition `== 0` passes
- `buffer.getLong()` → attempts to read bytes 12–19, but only 4 bytes remain → **`BufferUnderflowException`**

The test suite confirms that `decodeEvmAddress()` accepts inputs as short as 5 bytes without error: [3](#0-2) 

The `invalidEvmAddress` test only rejects odd-length and non-hex strings — short even-length strings are not rejected: [4](#0-3) 

The only upstream guard is `@Valid` on `AccountInput` in `AccountController`, but no `AccountInput` class with a `@Pattern` or `@Size` constraint on `evmAddress` exists in the codebase to enforce a 40-character minimum: [6](#0-5) 

## Impact Explanation
Every GraphQL `account` query supplying a crafted short all-zero hex EVM address throws an unchecked `BufferUnderflowException` inside the resolver. Spring GraphQL propagates this as an unhandled resolver error. Because the condition is trivially and repeatably triggerable with zero authentication, an attacker can flood the endpoint to degrade the GraphQL service's thread pool, causing effective denial of service for all account/contract lookups via the GraphQL API.

## Likelihood Explanation
The GraphQL endpoint is publicly reachable with no authentication required. The exploit requires only a single malformed GraphQL query with a 32-character all-zero hex string. No special knowledge, credentials, or tooling beyond a standard HTTP client is needed. It is fully repeatable and automatable.

## Recommendation
Add a length check in `decodeEvmAddress()` or at the start of `getByEvmAddressAndType()` to reject any input that does not decode to exactly 20 bytes:

```java
// In decodeEvmAddress() or getByEvmAddressAndType():
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException("EVM address must be exactly 20 bytes");
}
```

Additionally, add a `@Pattern(regexp = "^(0x)?[0-9a-fA-F]{40}$")` constraint on the `evmAddress` field of `AccountInput` to enforce the 40-character (20-byte) requirement at the input validation layer before the service is ever reached. [2](#0-1)

  account(input: { evmAddress: "0x00000000000000000000000000000000" }) {

  }

**Execution trace:**
1. `AccountController.account()` receives `evmAddress = "0x00000000000000000000000000000000"` (32 hex chars after prefix strip)
2. `decodeEvmAddress()` returns a 16-byte zero array — no exception
3. `ByteBuffer.wrap(16 bytes)` created
4. `buffer.getInt()` → 0 ✓ (12 bytes remain)
5. `buffer.getLong()` → 0 ✓ (4 bytes remain)
6. `buffer.getLong()` → **`java.nio.BufferUnderflowException`** (needs 8, only 4 remain)
7. Spring GraphQL returns an error response; repeated at scale → DoS

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);

            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
