Audit Report

## Title
Error Counter Reset Bypass via Intra-Stream `errors.set(0)` Allows Indefinite Griefing of Block Node Deactivation

## Summary
In `BlockNode.streamBlocks()`, `errors.set(0)` is placed inside the per-message `while` loop at line 157, causing it to fire after any successfully processed `BLOCK_ITEMS` or `END_OF_BLOCK` message — even when the overall subscription call subsequently fails. A malicious block node operator can exploit this by sending one valid `BLOCK_ITEMS` message per call to permanently reset the cross-call error counter, preventing `errors` from ever reaching `maxSubscribeAttempts` and thus preventing the node from ever being marked inactive.

## Finding Description
**Exact location**: `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`

The `errors` field is an `AtomicInteger` declared at line 54 that persists across calls to `streamBlocks()`. Its intended semantic is to count *consecutive failed subscription attempts* so that after `maxSubscribeAttempts` (default: 3, confirmed in `StreamProperties.java` line 27) consecutive failures, `active` is set to `false`.

However, `errors.set(0)` is placed at line 157, inside the `while` loop body, after the `switch` statement: [1](#0-0) 

This means `errors.set(0)` executes after any successful processing of a `BLOCK_ITEMS` or `END_OF_BLOCK` message — before the call has completed. If the call subsequently throws (e.g., a `NOT_AVAILABLE` status arrives next), `onError()` increments from 0 to 1, not from the previously accumulated value.

`onError()` at lines 196–207 only deactivates the node when `errors` reaches `maxSubscribeAttempts`: [2](#0-1) 

**Exploit flow** (with default `maxSubscribeAttempts = 3`):
1. `errors` starts at 0.
2. **Call N**: attacker sends one valid `BLOCK_ITEMS` (with a `BLOCK_HEADER` as first item, satisfying the `append()` guard at line 284). `assembler.onBlockItemSet()` succeeds, `errors.set(0)` fires — resetting `errors` to 0 regardless of prior accumulated value.
3. Attacker then sends `NOT_AVAILABLE` status → `BlockStreamException` thrown → `onError()` runs → `errors` becomes 1.
4. **Call N+1**: repeat steps 2–3. `errors` is reset to 0 again before `onError()` can push it to 2.
5. `errors` never reaches 3; `active` is never set to `false`.

**Test gap confirmed**: The existing `onError` test at lines 163–199 only exercises the scenario where `NOT_AVAILABLE` is sent as the *first and only* response (no preceding valid `BLOCK_ITEMS`). It does not cover the interleaved-reset scenario. [3](#0-2) 

## Impact Explanation
A malicious block node that is listed in the mirror node's configuration can permanently avoid deactivation. The mirror node will keep selecting it as the streaming source on every scheduling cycle, wasting connection and processing resources. If the malicious node is the highest-priority or only configured node, block ingestion stalls indefinitely. This is a griefing/availability degradation issue.

## Likelihood Explanation
Any operator of a configured block node can execute this attack. It requires only the ability to control the gRPC response stream — the normal capability of a block node operator. No special cryptographic privilege, timing precision, or race condition is required. One valid `BLOCK_ITEMS` frame per call is sufficient, and the attack is trivially repeatable.

## Recommendation
Move `errors.set(0)` out of the per-message `while` loop and place it after the loop exits normally (i.e., after line 158, still inside the `try` block but before the `catch` clauses). This ensures the counter is only reset when an entire `streamBlocks()` call completes without throwing an exception, which matches the intended semantic of counting *consecutive failed calls*. [4](#0-3) 

## Proof of Concept
1. Configure a mirror node with a single block node pointing to a controlled gRPC server.
2. On each incoming `subscribeBlockStream` RPC, the server sends:
   - One `SubscribeStreamResponse` with a `BLOCK_ITEMS` payload containing a single `BlockItem` with a `BLOCK_HEADER` (satisfying the `append()` precondition at line 284).
   - Immediately followed by one `SubscribeStreamResponse` with `STATUS = NOT_AVAILABLE`.
3. Observe that after each call, `errors` is incremented to 1 by `onError()` but then reset to 0 on the next call's first valid message.
4. After any number of calls (well beyond `maxSubscribeAttempts = 3`), `node.isActive()` remains `true` and the mirror node never stops selecting this node. [5](#0-4) [6](#0-5)
