# Audit Report

## Title
Integer Overflow in `poll()` Bypasses `maxPageSize` Cap, Enabling Unbounded Database Queries

## Summary
In `PollingTopicMessageRetriever.poll()`, a narrowing cast from `long` to `int` at line 69 allows a caller supplying `limit = Long.MAX_VALUE` to produce `pageSize = -1`, which bypasses the `maxPageSize` guard and causes `findByFilter` to execute SQL queries with no `LIMIT` clause against the `topic_message` table.

## Finding Description

**Location:** `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`, `poll()`, lines 68–71.

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())  // narrowing cast
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());       // guard bypassed
```

**Root cause:** `filter.getLimit()` and `context.getTotal().get()` are both `long`. Their difference is computed as a `long` and then silently truncated to `int`. When `filter.getLimit() = Long.MAX_VALUE` and `total = 0`:

```
(int)(Long.MAX_VALUE - 0) == (int)(0x7FFFFFFFFFFFFFFF) == -1
```

`Math.min(-1, maxPageSize)` evaluates to `-1`, so `pageSize = -1`.

**Why existing checks fail:**

- `@Min(0)` on `TopicMessageFilter.limit` only rejects negative values. `Long.MAX_VALUE` is positive and passes validation at the `subscribeTopic` entry point.
- The internally rebuilt filter at line 75 (`filter.toBuilder().limit(-1).startTime(startTime).build()`) is a direct builder call — Spring AOP `@Validated` is **not** invoked on it.
- `hasLimit()` returns `limit > 0`. With `limit = -1`, it returns `false`, so `findByFilter` receives a filter with no effective limit, omitting the `LIMIT` clause from the SQL query.
- `isComplete()` checks `filter.getLimit() == total.get()`, i.e., `Long.MAX_VALUE == total`, which is never true, so the retriever keeps polling indefinitely (throttled: until `pageSize < maxPageSize`; unthrottled: until `numRepeats` exhausted) — each poll issuing an unbounded query.

## Impact Explanation
Each invocation of `poll()` issues a SQL query with no `LIMIT` clause against the `topic_message` table, potentially causing a full table scan. A single subscription with `limit = Long.MAX_VALUE` triggers this on every polling interval (default 2 s throttled, 20 ms unthrottled). Multiple concurrent subscriptions amplify database load linearly. The 60-second stream timeout is the only backstop, but the attack is trivially repeatable on every new subscription, constituting a sustained denial-of-service against the database tier.

## Likelihood Explanation
The gRPC `ConsensusService.subscribeTopic` RPC maps the protobuf `uint64 limit` field directly to `TopicMessageFilter.limit` (a Java `long`). No authentication or authorization is required. Any network-reachable client can send a `ConsensusTopicQuery` with `limit = Long.MAX_VALUE` (a valid positive `uint64`). The attack requires zero privileges, zero knowledge of topic ownership, and is trivially scriptable and repeatable.

## Recommendation
Replace the unchecked narrowing cast with a bounds-safe computation:

```java
long remaining = filter.hasLimit()
        ? filter.getLimit() - context.getTotal().get()
        : Integer.MAX_VALUE;
// Clamp to int range before applying maxPageSize guard
int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
int pageSize = Math.min(limit, context.getMaxPageSize());
```

Additionally, add an upper-bound constraint on `TopicMessageFilter.limit` (e.g., `@Max(Integer.MAX_VALUE)` or a domain-appropriate maximum) to reject pathological values at the entry point before they reach `poll()`.

## Proof of Concept

```java
// Craft a filter with limit = Long.MAX_VALUE
TopicMessageFilter filter = TopicMessageFilter.builder()
        .topicId(EntityId.of(0, 0, 1))
        .limit(Long.MAX_VALUE)   // passes @Min(0), valid positive long
        .build();

// Inside poll():
// filter.hasLimit() == true  (Long.MAX_VALUE > 0)
// (int)(Long.MAX_VALUE - 0L) == -1
// Math.min(-1, 1000) == -1  → pageSize = -1

// Rebuilt filter: limit = -1
// hasLimit() == false  (-1 > 0 is false)
// findByFilter issues: SELECT * FROM topic_message WHERE ... (no LIMIT clause)

// isComplete(): Long.MAX_VALUE == 0 → false → keeps polling every 2s
// Each poll = full table scan until 60s timeout
// Attack is repeatable on every new subscription
```
