# Audit Report

## Title
Unauthenticated Redis Channel Injection Allows Delivery of TopicMessages with Arbitrary `validStartTimestamp` to gRPC Subscribers

## Summary
`RedisTopicListener.subscribe()` deserializes and forwards every `TopicMessage` received from a Redis pub/sub channel to gRPC subscribers without any integrity or provenance check. Because the channel naming scheme is predictable and the serialization format is fully public, an attacker with network access to the Redis instance can publish a crafted `TopicMessage` carrying an arbitrary `validStartTimestamp` (or any other field), and it will be delivered as a legitimate live message to all active subscribers on that topic.

## Finding Description

**Verified code path:**

`RedisTopicListener.subscribe()` (lines 68–80) subscribes to a Redis channel and maps every received payload directly to a `TopicMessage` via `messageSerializer` with no integrity check:

```java
return container
    .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
    .map(Message::getMessage)   // raw deserialized object, no validation
    ...
    .share();
``` [1](#0-0) 

The channel name is constructed as `topic.<numericId>`: [2](#0-1) 

The serializer is `Jackson2JsonRedisSerializer` backed by `MessagePackFactory` — a fully public, open-source format with no signing: [3](#0-2) 

The only downstream filter applied by `CompositeTopicListener.filterMessage()` is a `topicId` equality check and a `consensusTimestamp >= startTime` lower-bound check: [4](#0-3) 

`TopicMessageServiceImpl.subscribeTopic()` adds only a sequence-number deduplication guard (`sequenceNumber > last.sequenceNumber`): [5](#0-4) 

`validStartTimestamp` is a plain, unconstrained `Long` field on `TopicMessage` with no validation annotation: [1](#0-0) [6](#0-5) 

The legitimate publisher (`RedisPublisher`) writes to the identical channel format with no MAC or signature: [7](#0-6) [8](#0-7) 

**Root cause:** The system assumes only the trusted importer's `RedisPublisher` writes to Redis channels. There is no HMAC, message signing, or any other integrity mechanism. Any process that can reach the Redis port and knows the channel name (trivially derived from open-source code) can publish a structurally valid `TopicMessage`.

**Why checks fail:** No check in the entire pipeline (`subscribe()` → `filterMessage()` → `subscribeTopic()`) validates `validStartTimestamp` against the topic's `createdTimestamp`. The topic creation timestamp exists in the database (`Topic.createdTimestamp`) but is never fetched or compared during live message delivery.

## Impact Explanation
Any gRPC subscriber using the REDIS listener type receives the injected message as a legitimate live Hashgraph event. The `validStartTimestamp` field is surfaced to clients as `chunk_info.initial_transaction_id.transaction_valid_start`. A value predating the topic's creation is logically impossible on a real Hashgraph network, so delivering it falsifies the chronological metadata of the topic for all connected subscribers. Because the shared `Flux` is multicasted via `.share()`, a single injected message reaches every concurrent subscriber on that topic simultaneously. Beyond `validStartTimestamp`, the attacker controls all other fields in the injected `TopicMessage` (e.g., `runningHash`, `sequenceNumber`, `message` payload), enabling broader data integrity violations. A crafted `sequenceNumber` gap can also trigger the `missingMessages` gap-fill path, causing spurious database queries. [1](#0-0) [9](#0-8) 

## Likelihood Explanation
Redis is frequently deployed without authentication or with broad internal network exposure in containerized/cloud environments. The channel naming scheme (`topic.<id>`) is trivially discoverable from the open-source code. The MessagePack serialization format and `TopicMessage` schema are fully public. An attacker with any internal network path to Redis (misconfigured firewall, compromised sidecar, SSRF in another service) can craft and publish a valid payload with no application-level credentials. The attack is repeatable and requires no timing precision. The SECURITY.md exclusion for "local network access" applies only to the "Websites and Apps" category; this is a backend gRPC service, so that exclusion does not apply. [10](#0-9) 

## Recommendation
1. **Authenticate Redis connections**: Require Redis `AUTH` and restrict network access to the Redis port to only the importer and gRPC service pods via network policy.
2. **Add message-level integrity**: Sign each published `TopicMessage` with an HMAC (shared secret between importer and gRPC service) and verify the signature in `RedisTopicListener.subscribe()` before forwarding. Reject messages with invalid or missing signatures.
3. **Validate `validStartTimestamp` bounds**: After deserialization, compare `validStartTimestamp` against the topic's `createdTimestamp` fetched from the database (already available via `entityRepository`) and drop messages where `validStartTimestamp < topic.createdTimestamp`.
4. **Validate `consensusTimestamp` plausibility**: Reject messages whose `consensusTimestamp` is implausibly far in the past or future relative to the current wall clock.

## Proof of Concept

```python
import msgpack, redis

# TopicMessage schema from open-source code
# Channel: topic.<topicId>
# Fields: @type, chunk_num, chunk_total, consensus_timestamp,
#         initial_transaction_id, message, payer_account_id,
#         running_hash, running_hash_version, sequence_number,
#         topic_id, valid_start_timestamp

TOPIC_ID = 1001          # target topic numeric ID
FAKE_VALID_START = 1     # nanoseconds — predates any real topic creation

payload = {
    "@type": "TopicMessage",
    "chunk_num": 1,
    "chunk_total": 1,
    "consensus_timestamp": 9999999999999999999,  # passes >= startTime filter
    "initial_transaction_id": None,
    "message": b"injected",
    "payer_account_id": 3,
    "running_hash": b"\x00" * 48,
    "running_hash_version": 3,
    "sequence_number": 9999,                     # passes > last.sequenceNumber
    "topic_id": TOPIC_ID,
    "valid_start_timestamp": FAKE_VALID_START,   # arbitrary, pre-creation value
}

r = redis.Redis(host="<redis-host>", port=6379)
r.publish(f"topic.{TOPIC_ID}", msgpack.packb(payload))
# All active gRPC subscribers on topic 1001 receive this message immediately
# via the .share() multicast Flux.
```

The injected message passes `filterMessage()` (correct `topicId`, `consensusTimestamp >= startTime`) and the sequence-number guard (sequence 9999 > any prior last), and is delivered to all subscribers with `chunk_info.initial_transaction_id.transaction_valid_start = "0.000000001"` — a timestamp that predates the Hashgraph network itself.     private Flux<TopicMessage> missingMessages(TopicContext topicContext, @Nullable TopicMessage current) {
        final var last = topicContext.getLast();

        // Safety check triggered
        if (current == null) {
            long startTime = last != null
                    ? last.getConsensusTimestamp() + 1
                    : topicContext.getFilter().getStartTime();
            var gapFilter =
                    topicContext.getFilter().toBuilder().startTime(startTime).build();
            log.info("Safety check triggering gap recovery query with filter {}", gapFilter);
            return topicMessageRetriever.retrieve(gapFilter, false);
        }

        if (last == null || topicContext.isNext(current)) {
            return Flux.just(current);
        }

        long numMissingMessages = current.getSequenceNumber() - last.getSequenceNumber() - 1;

        // ignore duplicate message already processed by larger subscribe context
        if (numMissingMessages <= -1) {
            log.debug("Encountered duplicate missing message to be ignored, last: {}, current: {}", last, current);
            return Flux.empty();
        }

        TopicMessageFilter newFilter = topicContext.getFilter().toBuilder()
                .endTime(current.getConsensusTimestamp())
                .limit(numMissingMessages)
                .startTime(last.getConsensusTimestamp() + 1)
                .build();

        log.info(
                "[{}] Querying topic {} for missing messages between sequence {} and {}",
                newFilter.getSubscriberId(),
                topicContext.getTopicId(),
                last.getSequenceNumber(),
                current.getSequenceNumber());

        return topicMessageRetriever.retrieve(newFilter, false).concatWithValues(current);
    }
