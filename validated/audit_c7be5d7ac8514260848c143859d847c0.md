# Audit Report

## Title
`BlockType` Reference-Equality Bypass via Decimal `Long.MAX_VALUE` Block Parameter Causes `BlockNumberNotFoundException`

## Summary
`BlockType.of("9223372036854775807")` (and `BlockType.of("0x7fffffffffffffff")`) returns a **new** `BlockType` instance whose `number` equals `Long.MAX_VALUE` but whose reference is distinct from the `BlockType.LATEST` singleton. `RecordFileServiceImpl.findByBlockType()` guards the `findLatest()` path with a `==` identity check, so this new instance bypasses it, falls through to `recordFileRepository.findByIndex(Long.MAX_VALUE)`, receives `Optional.empty()`, and causes `BlockNumberNotFoundException` — making any `eth_call` or `eth_estimateGas` with that block parameter fail unconditionally with HTTP 400.

## Finding Description

**Step 1 — Singleton definition.**
`BlockType.LATEST` is a single static instance:

```java
public static final BlockType LATEST = new BlockType("latest", Long.MAX_VALUE);
``` [1](#0-0) 

**Step 2 — Decimal branch allocates a new object.**
The regex `(\d{1,20})` matches the 19-digit string `"9223372036854775807"`. `Long.parseLong` succeeds (no overflow), and the branch unconditionally constructs a fresh instance:

```java
final var decimal = matcher.group(GROUP_DECIMAL);
if (decimal != null) {
    return new BlockType(value, Long.parseLong(decimal, 10));  // new object, NOT LATEST
}
``` [2](#0-1) 

The resulting object has `name="9223372036854775807"` and `number=Long.MAX_VALUE` — structurally different from `LATEST` (`name="latest"`), and a different heap reference.

**Step 3 — Identity check in `findByBlockType()` fails.**
`RecordFileServiceImpl` uses `==` (reference equality) for both sentinel checks:

```java
if (block == BlockType.EARLIEST) {
    return recordFileRepository.findEarliest();
} else if (block == BlockType.LATEST) {   // FALSE — different reference
    return recordFileRepository.findLatest();
} else if (block.isHash()) { ... }
return recordFileRepository.findByIndex(block.number()); // findByIndex(Long.MAX_VALUE)
``` [3](#0-2) 

`findByIndex(Long.MAX_VALUE)` executes `select r from RecordFile r where r.index = ?1` with `Long.MAX_VALUE` — a block index that will never exist — and returns `Optional.empty()`. [4](#0-3) 

**Step 4 — Exception thrown in `ContractCallService`.**
The supplier wraps the result with `orElseThrow`:

```java
ctx.setBlockSupplier(Suppliers.memoize(() ->
    recordFileService.findByBlockType(params.getBlock()).orElseThrow(BlockNumberNotFoundException::new)));
``` [5](#0-4) 

`Optional.empty().orElseThrow(...)` throws `BlockNumberNotFoundException`, which the controller maps to HTTP 400 "Unknown block number."

**The same path is triggered by the hex form** `0x7fffffffffffffff`: the `GROUP_HEX_NUM` branch parses it to `Long.MAX_VALUE` and also returns a new instance. [6](#0-5) 

**Root cause:** The `blockTypeForTag` switch correctly returns the `LATEST` singleton for string tags (`"latest"`, `"pending"`, `"safe"`, `"finalized"`), but neither the decimal nor the hex-number branch checks whether the parsed value equals `Long.MAX_VALUE` before allocating a new object. The `==` guard in `findByBlockType` then silently misroutes it. [7](#0-6) 

## Impact Explanation
Every `eth_call` (or `eth_estimateGas`) submitted with `block: "9223372036854775807"` or `block: "0x7fffffffffffffff"` fails with HTTP 400 instead of executing against the latest block. Any client or DApp that encodes "latest" as its numeric sentinel value (`Long.MAX_VALUE` decimal) receives a permanent, deterministic error. No funds are directly at risk, but smart contract execution is silently denied for a valid-looking input. [8](#0-7) 

## Likelihood Explanation
No privileges are required. The attack surface is the public JSON-RPC `eth_call` endpoint. The input passes all validation: the regex matches and `Long.parseLong` succeeds without exception. It is trivially repeatable and requires no special knowledge beyond knowing `Long.MAX_VALUE = 9223372036854775807`. The same issue applies to the hex form `0x7fffffffffffffff`. [9](#0-8) 

## Recommendation
In `BlockType.of()`, after parsing the decimal (and hex-number) branch, check whether the result equals `Long.MAX_VALUE` and return the `LATEST` singleton instead of a new instance:

```java
// decimal branch
long parsed = Long.parseLong(decimal, 10);
if (parsed == Long.MAX_VALUE) return LATEST;
return new BlockType(value, parsed);

// hex-number branch — same guard
long parsed = Long.parseLong(hexNum, 16);
if (parsed == Long.MAX_VALUE) return LATEST;
return new BlockType(hexNum, parsed);
```

Alternatively, replace the `==` identity checks in `RecordFileServiceImpl.findByBlockType()` with value-based comparisons (`block.number() == Long.MAX_VALUE` for the LATEST path and `block.number() == 0L` for the EARLIEST path), which is more robust regardless of how `BlockType` instances are constructed. [10](#0-9) 

## Proof of Concept

```
POST /api/v1/contracts/call HTTP/1.1
Content-Type: application/json

{
  "data": "0x...",
  "to": "0x<contract>",
  "block": "9223372036854775807"
}
```

**Expected:** executes against the latest block (same as `"block": "latest"`).

**Actual:** `BlockType.of("9223372036854775807")` returns a new instance → `findByBlockType` falls through to `findByIndex(9223372036854775807)` → `Optional.empty()` → `BlockNumberNotFoundException` → HTTP 400 `{"_status":{"messages":[{"message":"Unknown block number"}]}}`.

The same result is reproduced with `"block": "0x7fffffffffffffff"`. [11](#0-10)
