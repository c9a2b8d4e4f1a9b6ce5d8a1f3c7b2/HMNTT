Audit Report

## Title
Unbounded Per-Topic Subscription Flood Enables DoS via Resource Exhaustion in `subscribeTopic()`

## Summary
`TopicMessageServiceImpl.subscribeTopic()` imposes no cap on concurrent subscriptions. The sole counter, `subscriberCount`, is wired exclusively to a Micrometer gauge and is never checked against any maximum. An unauthenticated attacker opening many TCP connections can exhaust DB connections (POLL mode) or heap/thread-pool resources (SHARED\_POLL/REDIS mode), denying service to all legitimate subscribers.

## Finding Description

**`subscriberCount` is metrics-only — never enforced.**

`subscriberCount` is declared as a plain `AtomicLong` and registered only as a Micrometer `Gauge`: [1](#0-0) 

In `subscribeTopic()`, it is incremented and decremented purely for observability — no comparison against any configured maximum ever occurs: [2](#0-1) 

**`topicExists()` performs no authentication or rate-limiting.**

The only pre-subscription check validates that the entity exists and has type `TOPIC`. There is no authentication, no per-topic subscriber cap, and no per-caller rate limit: [3](#0-2) 

**The only connection-level guard is trivially bypassed.**

`NettyProperties` defaults `maxConcurrentCallsPerConnection` to 5: [4](#0-3) 

This is applied in `GrpcConfiguration`: [5](#0-4) 

An attacker opens N TCP connections × 5 calls each, yielding 5N concurrent subscriptions with no further barrier.

**POLL mode — independent DB polling loop per subscription.**

`PollingTopicListener.listen()` creates a `Flux.defer(() -> poll(context)).repeatWhen(...)` loop scheduled on `Schedulers.boundedElastic()` for every subscriber independently: [6](#0-5) 

Each subscription issues periodic `SELECT` queries, directly exhausting the DB connection pool.

**SHARED\_POLL/REDIS mode — per-subscriber 16 384-element backpressure buffer.**

`SharedTopicListener.listen()` applies `onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)` and `publishOn(Schedulers.boundedElastic())` **per subscriber**: [7](#0-6) 

The default `maxBufferSize` is 16 384: [8](#0-7) 

Thousands of subscribers each holding a 16 384-element buffer causes heap exhaustion; `publishOn(Schedulers.boundedElastic())` allocates a delivery thread per active subscriber, exhausting the bounded-elastic scheduler.

## Impact Explanation

- **POLL mode**: DB connection pool saturation. Every attacker subscription issues periodic `SELECT` queries; with hundreds of subscriptions the pool is exhausted, causing legitimate subscribers' historical retrieval and live polling to time out or fail.
- **SHARED\_POLL/REDIS mode**: Heap exhaustion from per-subscriber 16 384-element backpressure buffers. Thread-pool exhaustion from `publishOn(Schedulers.boundedElastic())` per subscriber. Legitimate subscribers experience message delivery delays or `OutOfMemoryError`-induced JVM crashes affecting all topics.
- **Severity: High** — complete denial of service to all subscribers of the targeted topic, with potential JVM crash affecting the entire node.

## Likelihood Explanation

- **Precondition**: Network access to gRPC port 5600. No credentials required; `ConsensusController` and `TopicMessageServiceImpl` contain zero authentication or authorization logic.
- **Bypass**: 200 TCP connections × 5 calls = 1 000 concurrent subscriptions. Trivially scriptable with `grpcurl` or any gRPC client library.
- **Persistence**: Subscriptions with no `endTime` and no `limit` never terminate (`pastEndTime()` returns `Flux.never()` when `endTime == null`), so attacker subscriptions persist indefinitely. [9](#0-8) 

## Recommendation

1. **Enforce a configurable global subscriber cap**: compare `subscriberCount.get()` against a `maxSubscribers` property before allowing a new subscription in `subscribeTopic()`, returning `RESOURCE_EXHAUSTED` when the cap is reached.
2. **Enforce a per-topic subscriber cap**: maintain a `ConcurrentHashMap<EntityId, AtomicLong>` of per-topic counts and reject subscriptions that exceed a configurable per-topic limit.
3. **Add per-source-IP rate limiting** at the gRPC interceptor layer to limit new subscription attempts per time window.
4. **Increase `maxConcurrentCallsPerConnection`** only if justified, but pair it with the caps above; the current value of 5 is not a meaningful defence on its own.

## Proof of Concept

```bash
# Open 200 TCP connections, 5 subscriptions each = 1000 concurrent subscriptions
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext \
      -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
```

Each background process holds an open, non-terminating subscription (no `endTime`, no `limit`). In POLL mode, 1 000 independent `SELECT` loops saturate the DB pool within seconds. In SHARED\_POLL/REDIS mode, 1 000 × 16 384-element buffers (~1 000 × object overhead) rapidly exhaust heap, and 1 000 `boundedElastic` threads exhaust the scheduler, starving all legitimate subscribers.

    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
    }

        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)

                        .withFixedDelay(interval)
                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
