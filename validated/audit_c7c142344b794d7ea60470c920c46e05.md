# Audit Report

## Title
Cache Miss DoS via Non-Existent Block Numbers in `findByIndex` — Empty Results Never Cached

## Summary
`RecordFileRepository.findByIndex(long index)` is annotated with `@Cacheable(unless = "#result == null")`. Spring Framework unwraps `Optional<T>` return values before evaluating SpEL expressions, so `Optional.empty()` becomes `null` in `#result`. This makes the `unless` condition `true` for every non-existent block index, permanently suppressing negative-result caching. An attacker can issue requests with unique non-existent block numbers and force a live database query on every single request.

## Finding Description
**Code path:**

`RecordFileServiceImpl.java` lines 19–28 — for any numeric `BlockType`, the call falls through to:
```java
return recordFileRepository.findByIndex(block.number());
``` [1](#0-0) 

`RecordFileRepository.java` lines 27–29:
```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
           unless = "#result == null")
Optional<RecordFile> findByIndex(long index);
``` [2](#0-1) 

**Root cause:** Spring's caching abstraction (since 4.3) unwraps `Optional<T>` before evaluating `unless`/`condition` SpEL expressions and before storing the value. When the repository returns `Optional.empty()` (index not in `record_file`), `#result` is `null`, so `unless = "#result == null"` is `true`, and the result is never written to the Caffeine cache. The cache manager is configured with `expireAfterWrite(10 minutes)` and `maximumSize(10000)`, but this capacity is irrelevant because negative results are never stored. [3](#0-2) 

**Failed assumption:** The developer intended `unless = "#result == null"` to avoid caching absent rows, but this condition also silently prevents caching of `Optional.empty()` — the exact case that requires negative caching to protect the database.

## Impact Explanation
Every unique non-existent block number generates an uncached database round-trip executing `SELECT r FROM RecordFile r WHERE r.index = ?1`. Under sustained parallel load with unique values, the DB connection pool saturates, query latency rises, and legitimate requests are starved. The attacker has an effectively unbounded space of non-existent indices (any value above the current chain tip), and the `record_file` table grows only as new blocks are ingested (~1 block every few seconds on Hedera).

## Likelihood Explanation
The JSON-RPC API is public and unauthenticated. No rate limiting is present at the service or repository layer. `BlockType.of()` accepts both decimal and hex strings, making it trivial to generate millions of unique non-existent block numbers programmatically. A single attacker with a modest HTTP client can sustain thousands of requests per second, each bypassing the cache entirely.

## Recommendation
Change the `unless` condition on `findByIndex` to also cache `Optional.empty()` results (negative caching). The correct SpEL expression is:

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
           unless = "#result != null && !#result.isPresent() == false")
```

Or more simply, remove the `unless` guard entirely (Spring will store `null` for empty Optionals, which is the desired negative-cache behavior), or use:

```java
unless = "#result != null && #result.isEmpty()"
```

Wait — the correct fix is to **allow** `Optional.empty()` to be cached. Since Spring stores the unwrapped value (`null`) in the cache, the fix is to drop the `unless` clause or change it to a condition that only excludes truly unexpected nulls (i.e., a raw `null` return, not an empty Optional). In practice, since the method signature is `Optional<RecordFile>`, a raw `null` return should never occur, so the `unless` clause can be removed entirely:

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX)
Optional<RecordFile> findByIndex(long index);
```

This allows Spring to store `null` (unwrapped from `Optional.empty()`) in the cache, so subsequent requests for the same non-existent index are served from cache without hitting the database.

Additionally, consider adding rate limiting at the API gateway or web layer for block-parameterized JSON-RPC endpoints.

## Proof of Concept
```
# Generate unique non-existent block numbers (above current chain tip)
for i in $(seq 1 10000); do
  HEX=$(printf "0x%X" $((0xFFFFFFF0 + i)))
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"block\":\"$HEX\",\"data\":\"0x\",\"to\":\"0x0000000000000000000000000000000000000001\"}" &
done
wait
```
Each request resolves the block via `findByBlockType` → `findByIndex(N)` → cache miss → DB query → `Optional.empty()` → not cached. With 10,000 concurrent unique values, all 10,000 hit the database simultaneously, with zero cache benefit.
