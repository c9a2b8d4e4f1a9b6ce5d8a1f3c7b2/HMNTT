Audit Report

## Title
Missing Deleted-Status Check in `findRuntimeBytecode` Allows EVM Simulation of Logically Deleted Contracts

## Summary
`ContractRepository.findRuntimeBytecode()` queries only the `contract` table and never consults `entity.deleted`. `ContractBytecodeReadableKVState.readFromDataSource()` invokes this method unconditionally for the `contractNum` and long-zero EVM address resolution paths. As a result, any caller of the unauthenticated web3 API can obtain EVM simulation results for a logically deleted contract, contradicting actual Hedera network state.

## Finding Description

**`ContractRepository.findRuntimeBytecode()`** — the sole query:

```sql
select runtime_bytecode from contract where id = :contractId
```

This touches only the `contract` table. The `deleted` flag lives exclusively in the `entity` table. When a contract is deleted on Hedera, `entity.deleted` is set to `true`, but the `contract` row (including `runtime_bytecode`) is never removed. [1](#0-0) 

**`ContractBytecodeReadableKVState.readFromDataSource()`** calls `findRuntimeBytecode` unconditionally after resolving the entity ID: [2](#0-1) 

**`toEntityId()` has three branches with asymmetric deleted-status handling:**

- **`contractNum` path (lines 50–51):** converts directly to `EntityId` via `entityIdFromContractId()` — **no deleted check**.
- **Long-zero EVM address path (lines 54–55):** converts directly via `DomainUtils.fromEvmAddress()` — **no deleted check**.
- **Non-long-zero EVM address path (lines 57–60):** calls `commonEntityAccessor.getEntityByEvmAddressAndTimestamp()`, which internally calls `findByEvmAddressAndDeletedIsFalse` — **deleted check present**. [3](#0-2) 

**Root cause:** The `contractNum` and long-zero EVM address paths bypass the `entity` table entirely, so `entity.deleted` is never consulted before bytecode is returned to the EVM.

**Contrast with the correct pattern** — `NftRepository.findActiveById()` explicitly joins the entity table and filters `e.deleted is not true`: [4](#0-3) 

`CommonEntityAccessor.get(EntityId, timestamp)` also enforces the deleted check for all other entity lookups via `findByIdAndDeletedIsFalse`: [5](#0-4) 

## Impact Explanation
The web3 module is the mirror node's EVM simulation layer. When `readFromDataSource` returns a non-null `Bytecode` for a deleted contract, the EVM treats that address as a live contract and executes its code. This produces simulation results from `eth_call`, `eth_estimateGas`, etc. that contradict actual Hedera network state, where calls to deleted contracts are rejected. Applications that rely on mirror-node simulation to gate real transactions will receive false-positive results. Additionally, the `@Cacheable` annotation on `findRuntimeBytecode` means stale bytecode for a deleted contract can persist in the cache indefinitely after deletion, amplifying the window of incorrect responses. [1](#0-0) 

## Likelihood Explanation
No authentication or privilege is required to call the web3 API endpoints (`eth_call`, `eth_estimateGas`). Deleted contracts are a normal, frequent occurrence on Hedera (self-destruct, admin delete). The contract ID of any deleted contract is public information visible on the network. The exploit requires only knowing a deleted contract's numeric ID and issuing a standard JSON-RPC call — trivially repeatable by any external user. The `contractNum` path is the most common way contracts are addressed in Hedera's JSON-RPC relay, making this the default code path for the vast majority of callers.

## Recommendation
Fix the asymmetry by adding a deleted-status check for the `contractNum` and long-zero EVM address paths. Two equivalent approaches:

1. **At the repository layer:** Modify `findRuntimeBytecode` to join the `entity` table:
   ```sql
   select c.runtime_bytecode
   from contract c
   join entity e on e.id = c.id
   where c.id = :contractId
   and e.deleted is not true
   ```

2. **At the service layer:** In `readFromDataSource`, resolve the entity through `commonEntityAccessor.get(entityId, timestamp)` (which already enforces `findByIdAndDeletedIsFalse`) before calling `findRuntimeBytecode`, and return `null` if the entity is absent or deleted — mirroring the pattern used by `ScheduleReadableKVState.readFromDataSource()`. [6](#0-5) 

## Proof of Concept

1. Deploy a contract on Hedera (e.g., contract ID `0.0.12345`).
2. Delete the contract via an admin-delete or self-destruct transaction. After deletion, `entity.deleted = true` for entity `12345`, but the `contract` row with `runtime_bytecode` remains.
3. Issue an `eth_call` to the mirror node web3 endpoint using the contract's numeric address (long-zero form `0x000000000000000000000000000000000000303900` for `0.0.12345`):
   ```json
   {
     "jsonrpc": "2.0",
     "method": "eth_call",
     "params": [{"to": "0x0000000000000000000000000000000000003039", "data": "0x..."}, "latest"],
     "id": 1
   }
   ```
4. **Expected (correct) behavior:** The call returns an error or empty result, consistent with the contract being deleted on Hedera.
5. **Actual behavior:** `ContractBytecodeReadableKVState.readFromDataSource()` takes the long-zero path, calls `findRuntimeBytecode(12345)`, finds the bytecode in the `contract` table (no deleted check), and returns a non-null `Bytecode` to the EVM. The EVM executes the contract's code and returns a result as if the contract were live. [7](#0-6)

    protected Bytecode readFromDataSource(@NonNull ContractID contractID) {
        final var entityId = toEntityId(contractID);

        return contractRepository
                .findRuntimeBytecode(entityId.getId())
                .map(Bytes::wrap)
                .map(Bytecode::new)
                .orElse(null);
    }

    private EntityId toEntityId(@NonNull final ContractID contractID) {
        if (contractID.hasContractNum()) {
            return entityIdFromContractId(contractID);
        } else if (contractID.hasEvmAddress()) {
            final var evmAddress = contractID.evmAddress().toByteArray();
            if (isLongZeroAddress(evmAddress)) {
                return DomainUtils.fromEvmAddress(evmAddress);
            } else {
                return commonEntityAccessor
                        .getEntityByEvmAddressAndTimestamp(evmAddress, Optional.empty())
                        .map(Entity::toEntityId)
                        .orElse(EntityId.EMPTY);
            }
        }
        return EntityId.EMPTY;
    }
