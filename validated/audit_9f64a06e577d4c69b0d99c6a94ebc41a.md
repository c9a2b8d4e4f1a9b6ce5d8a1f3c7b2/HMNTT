Audit Report

## Title
`incrementByteArray` Overflow on All-0xFF Key Causes Full Storage Slot Disclosure

## Summary
When a caller supplies `key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF` to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, Apache Tuweni's `MutableBytes32.increment()` silently wraps the all-0xFF value to all-0x00. The resulting inclusive lower bound is identical to `MIN_KEY_BYTES`, so the range guard in `hookStorageChangeRequest()` never tightens the bound, and the query returns every storage slot for the target hook.

## Finding Description

**Step 1 — Input accepted by the regex.**
`SlotRangeParameter.valueOf()` uses the pattern `[0-9a-fA-F]{1,64}`, which accepts exactly 64 hex `F`s. The subsequent `leftPad` to 64 chars is a no-op. [1](#0-0) 

**Step 2 — `getInclusiveValue()` calls `incrementByteArray` on all-0xFF.**
For `operator == GT`, the static helper decodes the hex to a 32-byte all-0xFF array and passes it to `incrementByteArray`. [2](#0-1) 

**Step 3 — Tuweni wraps silently to all-0x00.**
`BytesUtil.incrementByteArray` delegates entirely to `MutableBytes32.wrap(bytes).increment().toArrayUnsafe()`. Tuweni's `increment()` performs modular big-endian arithmetic with no overflow guard: `0xFF…FF + 1 = 0x00…00`. [3](#0-2) 

**Step 4 — The `SlotRangeParameter` record stores operator=`GTE`, value=all-0x00.**
`operator.toInclusive()` converts `GT` → `GTE`; the already-wrapped all-0x00 array is stored as `value()`. [4](#0-3) [5](#0-4) 

**Step 5 — The range guard in `hookStorageChangeRequest()` fails.**
`lowerBound` is initialised to `MIN_KEY_BYTES` (all-0x00). The guard `Arrays.compareUnsigned(value, lowerBound) > 0` evaluates `compareUnsigned(all-zeros, all-zeros) > 0` → `0 > 0` → **false**. `lowerBound` is never updated. [6](#0-5) 

**Step 6 — Full key space is forwarded to the service.**
`HookStorageRequest` is built with `keyLowerBound = all-zeros` and `keyUpperBound = MAX_KEY_BYTES` (all-0xFF), causing the repository query to return every storage slot. [7](#0-6) [8](#0-7) 

**Root cause:** `incrementByteArray` has no overflow guard, and its caller has no post-condition check that the returned value is strictly greater than the input.

**Failed assumption:** The code assumes `incrementByteArray(x) > x` always holds. For `x = 0xFF…FF` this is false.

## Impact Explanation
Any caller can enumerate the complete storage of any hook by issuing a single crafted GET request. Storage slots may contain sensitive contract state (balances, access-control mappings, private configuration). The endpoint carries no visible privilege requirement in the controller. [9](#0-8) 

## Likelihood Explanation
The exploit requires zero authentication, zero cryptographic material, and a single HTTP GET request with a trivially constructed query parameter (`key=gt:0xFFFF…FFFF`). It is deterministic and repeatable. Any attacker who reads the API documentation or fuzzes the `key` parameter will discover it.

## Recommendation
Add an overflow guard in `incrementByteArray` (or its caller) that detects when the input is all-0xFF and either throws an `IllegalArgumentException` or returns a sentinel that the caller treats as "no valid lower bound exists":

```java
// In BytesUtil or SlotRangeParameter.getInclusiveValue()
public static byte[] incrementByteArray(byte[] bytes) {
    for (byte b : bytes) {
        if (b != (byte) 0xFF) {
            return MutableBytes32.wrap(bytes).increment().toArrayUnsafe();
        }
    }
    throw new IllegalArgumentException(
        "Storage key 0xFF…FF has no valid GT successor; use lte/lt instead.");
}
```

Alternatively, add a post-condition check in `hookStorageChangeRequest()`: after computing the inclusive value for a `GT` filter, verify that the result is strictly greater than the original input before using it as a lower bound. If it is not, the filter matches nothing and the request should return an empty result.

## Proof of Concept

```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF
```

1. `SlotRangeParameter.valueOf()` accepts the input (64 hex Fs match the regex).
2. `getInclusiveValue(GT, "FFFF…FFFF")` → `incrementByteArray(all-0xFF)` → Tuweni wraps to `all-0x00`.
3. Record stored: `operator=GTE`, `value=all-0x00`.
4. In `hookStorageChangeRequest()`: `compareUnsigned(all-zeros, MIN_KEY_BYTES) > 0` → false → `lowerBound` stays `all-0x00`.
5. `HookStorageRequest` built with `keyLowerBound=all-zeros`, `keyUpperBound=all-0xFF`.
6. Service executes `findByOwnerIdAndHookIdAndKeyBetweenAndDeletedIsFalse(ownerId, hookId, all-zeros, all-0xFF, page)` — returns all storage slots.
