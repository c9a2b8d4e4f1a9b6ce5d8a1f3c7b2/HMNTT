## Audit Report

## Title
Unbounded In-Memory Transaction Accumulation in `Block()` Before `maxTransactionsInBlock` Cap Is Applied

## Summary
In `rosetta/app/services/block_service.go`, the `Block()` function unconditionally loads all transactions for a block into memory via `FindBetween()` before the `maxTransactionsInBlock` cap is applied. The underlying `FindBetween()` in `rosetta/app/persistence/transaction.go` accumulates batches of 2,000 rows into a single unbounded in-memory slice with no total-count ceiling. Concurrent unauthenticated requests targeting a high-transaction block can exhaust server memory.

## Finding Description
In `rosetta/app/services/block_service.go` at line 56, `Block()` calls:

```go
if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
```

The response-size cap is only applied afterward at line 61:

```go
if len(block.Transactions) > s.maxTransactionsInBlock {
```

In `rosetta/app/persistence/transaction.go`, `FindBetween()` uses a loop that fetches `batchSize = 2000` rows per iteration and appends them all to a single in-memory slice with no upper bound on total accumulation:

```go
const batchSize = 2000
...
transactions := make([]*transaction, 0)
for start <= end {
    transactionsBatch := make([]*transaction, 0)
    err := db.Raw(...).Limit(batchSize).Find(&transactionsBatch).Error
    ...
    transactions = append(transactions, transactionsBatch...)
    if len(transactionsBatch) < batchSize {
        break
    }
    start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
}
```

**Root cause:** `maxTransactionsInBlock` is a response-size cap, not a query or memory cap. The full transaction set for the block is always materialized in memory before truncation. There is no `LIMIT` applied at the database level to the total number of transactions fetched across all batches.

**Failed assumption:** The design assumes blocks will never contain enough transactions to cause memory pressure before the cap is applied. This assumption is not enforced in code.

## Impact Explanation
Hedera can sustain thousands of transactions per second. A single record-file block covering a ~2-second window during a high-throughput period can contain tens of thousands of transactions. Each transaction row includes JSON-encoded crypto transfers, staking reward payouts, and itemized transfers — all deserialized into Go structs in memory. With concurrent requests from multiple source IPs targeting the same high-transaction block, server memory can be exhausted, causing an OOM kill or severe degradation of the Rosetta API process. The `/block` endpoint requires no authentication.

## Likelihood Explanation
The `/block` endpoint is publicly accessible with no authentication requirement. An attacker only needs to identify a block with a large transaction count — trivially done by querying the Hedera mirror node REST API or monitoring the network during high-throughput periods. The block index or hash is then used directly in the `BlockRequest`. The Traefik-level rate limit (`average: 10`, `inFlightReq: 5` per IP) exists only in the optional Helm chart deployment and is not enforced at the application layer; it is bypassable with multiple source IPs.

## Recommendation
Apply the `maxTransactionsInBlock` limit at the database query level rather than post-fetch. Pass the limit into `FindBetween()` (or a new variant) and enforce it via a `LIMIT` clause in the SQL query, so that at most `maxTransactionsInBlock + 1` rows are ever fetched from the database. The `+1` allows detection of overflow (to populate `OtherTransactions`) without materializing the full set. This eliminates the unbounded accumulation loop as the attack surface.

## Proof of Concept
1. Identify a high-transaction block using the public mirror node REST API (e.g., `GET /api/v1/blocks?order=desc&limit=1` during a high-throughput period).
2. Extract the block index or hash from the response.
3. Send concurrent POST requests to `/block` with the identified `block_identifier` from multiple source IPs (to bypass per-IP Traefik rate limits):
```json
{
  "network_identifier": { "blockchain": "Hedera", "network": "mainnet" },
  "block_identifier": { "index": <high_tx_block_index> }
}
```
4. Each request causes `FindBetween()` to fetch all transactions for that block into memory. With sufficient concurrency, the Rosetta API process exhausts available memory and is OOM-killed or severely degraded.
