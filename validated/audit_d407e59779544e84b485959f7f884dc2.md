Audit Report

## Title
Missing `return` After Block Number Mismatch in `BlockAssembler.onEndOfBlock()` Allows Data Integrity Corruption or DoS

## Summary
In `BlockNode.java`, the inner class `BlockAssembler.onEndOfBlock()` detects a mismatch between `BlockHeader.number` and `EndOfBlock.blockNumber` and calls `Utility.handleRecoverableError()`, but omits a `return` statement afterward. In the default configuration (`HALT_ON_ERROR=false`), execution falls through and the block is dispatched using the attacker-supplied `blockNumber` from `EndOfBlock` as the filename, corrupting the mirror node's block index. Under `HALT_ON_ERROR=true`, a `ParserException` is thrown, terminating the streaming session.

## Finding Description

**Exact location**: `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`, inner class `BlockAssembler`, method `onEndOfBlock()`.

The `blockNumber` variable is sourced from the attacker-controlled `BlockEnd` message at line 240: [1](#0-0) 

The mismatch check at lines 248–253 calls `handleRecoverableError()` but has **no `return` statement**: [2](#0-1) 

Contrast this with the `pending.isEmpty()` guard at lines 241–245, which correctly uses `return` after the error call: [3](#0-2) 

After the mismatch check, execution unconditionally reaches line 270, where the attacker-controlled `blockNumber` is used to generate the filename: [4](#0-3) 

**`handleRecoverableError()` behavior** (`Utility.java` lines 220–231):
- `HALT_ON_ERROR=false` (default, `HALT_ON_ERROR_DEFAULT = "false"`): logs the error and **returns normally**, allowing fall-through.
- `HALT_ON_ERROR=true`: throws `ParserException`. [5](#0-4) [6](#0-5) 

The thrown `ParserException` is not a `BlockStreamException`, so it is caught by the generic `catch (Exception ex)` block in `streamBlocks()`, which calls `onError()` and re-throws as `BlockStreamException`: [7](#0-6) 

`onError()` increments the error counter and, after `maxSubscribeAttempts` consecutive errors, marks the node inactive with a readmit delay: [8](#0-7) 

**Root cause**: The failed assumption is that `handleRecoverableError()` always halts execution. It only does so under a non-default system property. The mismatch check is purely diagnostic in the default configuration.

## Impact Explanation

**Default configuration (`HALT_ON_ERROR=false`)**: Block N's content (assembled from `pending`, which is correct) is dispatched with a filename derived from the attacker-supplied block number M (`BlockFile.getFilename(blockNumber, false)`). The mirror node records block N's transactions as belonging to block M. The sequential block tracker advances past M, causing block N to be silently skipped. This is a **persistent data integrity corruption** — transactions are either lost or misattributed to the wrong block number, affecting all downstream consumers of the mirror node API.

**`HALT_ON_ERROR=true`**: Repeated triggering across multiple blocks causes the block node to be marked inactive after `maxSubscribeAttempts` errors. If the attacker controls all configured block nodes, the mirror node cannot process any blocks until the readmit delay expires, at which point the attack can be repeated indefinitely. This is a **repeatable DoS** against block ingestion.

## Likelihood Explanation

An attacker must operate a block node that the mirror node is configured to subscribe to. Block nodes are permissioned network participants, but the mirror node's trust model does not cryptographically authenticate the `EndOfBlock.blockNumber` field against the `BlockHeader.number` in the block items — it only logs the discrepancy. The default configuration (`HALT_ON_ERROR=false`) makes the data integrity path exploitable without any special operator configuration. The `HALT_ON_ERROR=true` path requires a non-default but documented and supported configuration. The attack is trivially repeatable on every block.

## Recommendation

Add a `return` statement immediately after the `handleRecoverableError()` call in the block number mismatch branch, mirroring the pattern already used in the `pending.isEmpty()` guard:

```java
if (blockHeader.getNumber() != blockNumber) {
    Utility.handleRecoverableError(
            "Block number mismatch in BlockHeader({}) and EndOfBlock({})",
            blockHeader.getNumber(),
            blockNumber);
    return; // ADD THIS
}
```

This ensures that regardless of the `HALT_ON_ERROR` setting, a mismatched block is never dispatched. In the default mode, the error is logged and the malformed block is discarded. In `HALT_ON_ERROR=true` mode, the existing `ParserException` throw behavior is preserved.

## Proof of Concept

1. Configure the mirror node to subscribe to an attacker-controlled block node.
2. The attacker sends valid `BlockItemSet` messages for block N (with `BlockHeader.number = N`).
3. The attacker sends an `EndOfBlock` message with `blockNumber = M` where `M ≠ N` (e.g., a future block number).
4. `onEndOfBlock()` is called: `blockNumber = M` (line 240), `blockHeader.getNumber() = N` (line 247).
5. The mismatch is detected (line 248), `handleRecoverableError()` logs the error and returns normally (default mode).
6. Execution falls through to line 270: `BlockFile.getFilename(M, false)` generates filename for block M.
7. `blockStreamConsumer.accept(new BlockStream(block, null, filename_M, loadStart))` dispatches block N's content under block M's identity (line 271).
8. The mirror node records block N's transactions as belonging to block M, and `getNextBlockNumber()` advances past M, permanently skipping block N.

        if (haltOnError) {
            var formattingTuple = MessageFormatter.arrayFormat(message, args);
            var throwable = formattingTuple.getThrowable();
