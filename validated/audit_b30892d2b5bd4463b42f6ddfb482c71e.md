Audit Report

## Title
Unauthenticated Forced Routing to Expensive Historical Query Path via Trivial Timestamp Parameter in `getHookStorage()`

## Summary
Any unauthenticated caller can supply a single timestamp query parameter (e.g., `timestamp=gte:0`) to force `getHookStorage()` to route to `getHookStorageChange()`, which executes a `DISTINCT ON (key)` full-range scan across the unbounded, time-partitioned `hook_storage_change` table. No rate limiting exists on the rest-java hooks endpoint, and the result-set `LIMIT` cap does not bound the underlying database scan cost.

## Finding Description

**Routing decision — `HookServiceImpl.java` lines 56–58:** [1](#0-0) 

`request.isHistorical()` delegates to `HookStorageRequest.isHistorical()`: [2](#0-1) 

Which delegates to `Bound.isEmpty()`: [3](#0-2) 

`Bound.isEmpty()` returns `true` only when both `lower` and `upper` are `null`. Providing any timestamp operator (e.g., `timestamp=gte:0`) sets `lower` to a non-null value, making `isHistorical()` return `true` and unconditionally routing to `getHookStorageChange()`.

**Default bounds when no key filter is supplied — `HooksController.java` lines 167–168:** [4](#0-3) 

`keyLowerBound = MIN_KEY_BYTES` (32 × `0x00`) and `keyUpperBound = MAX_KEY_BYTES` (32 × `0xFF`). With `timestamp=gte:0`, `getAdjustedLowerRangeValue()` returns `0` (GTE operator, no increment) and `adjustUpperBound()` returns `Long.MAX_VALUE` (null upper): [5](#0-4) [6](#0-5) 

**The expensive query — `HookStorageChangeRepository.java` lines 15–39:** [7](#0-6) 

`DISTINCT ON (key)` with `key BETWEEN 0x00…00 AND 0xFF…FF` and `consensus_timestamp BETWEEN 0 AND Long.MAX_VALUE` forces PostgreSQL to scan every row in `hook_storage_change` for the given `owner_id`/`hook_id` across all time partitions, sort them, and deduplicate — before applying the `LIMIT` from the `Pageable`. The `LIMIT` bounds the result set, not the scan.

**Contrast with the cheap path** — `HookStorageRepository` uses derived Spring Data methods that resolve to indexed primary-key range scans on the current-state `hook_storage` table: [8](#0-7) 

**No rate limiting on the rest-java hooks endpoint.** `ThrottleManagerImpl` and `ThrottleConfiguration` are scoped exclusively to the `web3` module: [9](#0-8) [10](#0-9) 

`HooksController` has no equivalent guard. [11](#0-10) 

## Impact Explanation
The `hook_storage_change` table is partitioned by `consensus_timestamp` and grows without bound as hooks are updated over time. A wide-range `DISTINCT ON` query must cross partition boundaries, sort all matching rows, and deduplicate before applying `LIMIT`. This is orders of magnitude more expensive than the indexed primary-key lookup on `hook_storage`. An attacker sending a sustained stream of requests with `timestamp=gte:0` and no key filter forces repeated full-history scans, consuming database CPU, I/O, and connection pool resources, degrading service for all users of the mirror node.

## Likelihood Explanation
No special privileges, credentials, or knowledge of internal state are required. The exploit requires only a valid `ownerId` and `hookId`, both discoverable via the public `/api/v1/accounts/{id}/hooks` endpoint. The attack is trivially scriptable, repeatable, and requires no brute force. The attacker controls the exact parameters that maximize scan cost (full key range, full timestamp range) and can sustain the attack indefinitely.

## Recommendation
1. **Require authentication or apply rate limiting** on `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` before allowing historical queries. Integrate a throttle guard analogous to the `web3` module's `ThrottleManagerImpl` into `HooksController`.
2. **Restrict the historical query path**: Require an explicit, bounded timestamp range (both lower and upper bounds) before routing to `getHookStorageChange()`. Reject or redirect to the cheap path when only a lower bound is provided with no upper bound.
3. **Add a database-side guard**: Enforce a maximum timestamp range width (e.g., reject ranges wider than a configurable threshold) in `HookStorageRequest` or `HookServiceImpl` before executing the `DISTINCT ON` query.
4. **Index optimization**: Ensure `hook_storage_change` has a composite index on `(owner_id, hook_id, key, consensus_timestamp DESC)` to reduce the cost of the `DISTINCT ON` scan even when it is legitimately used.

## Proof of Concept
```
# Trigger the expensive DISTINCT ON full-range scan with a single parameter:
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0

# No authentication header required.
# keyLowerBound defaults to 0x00...00, keyUpperBound to 0xFF...FF
# timestampLowerBound = 0, timestampUpperBound = Long.MAX_VALUE
# Routes to getHookStorageChange() -> findByKeyBetweenAndTimestampBetween()
# PostgreSQL executes: SELECT DISTINCT ON (key) ... FROM hook_storage_change
#   WHERE owner_id=1234 AND hook_id=1
#   AND key >= '\x0000...00' AND key <= '\xffff...ff'
#   AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
# Full cross-partition scan, sort, dedup — then LIMIT applied on result.
# Repeat in a tight loop to exhaust DB resources.
```

    public long adjustUpperBound() {
        if (this.upper == null) {

            upperBound--;
        }
