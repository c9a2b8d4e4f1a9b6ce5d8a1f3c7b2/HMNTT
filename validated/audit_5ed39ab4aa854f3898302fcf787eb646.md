Audit Report

## Title
Unbounded Unthrottled Historical Query via Safety-Check with `last == null` and User-Controlled `startTime`

## Summary
In `TopicMessageServiceImpl.missingMessages()`, the safety-check branch (`current == null`, `last == null`) constructs a gap-recovery filter directly from the subscriber-supplied `filter.getStartTime()` with no `endTime` and no `limit`, then issues it to `topicMessageRetriever.retrieve(gapFilter, false)` with throttling disabled. A subscriber who sets `startTime = 0` (epoch) with no `endTime` and no `limit` deterministically triggers a full, unthrottled table scan of `topic_message` from epoch start, one second after connecting.

## Finding Description

**Exact code path — `TopicMessageServiceImpl.java`:**

The safety check is scheduled unconditionally at subscription time:

```java
// lines 67–70
Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
        .filter(_ -> !topicContext.isComplete())          // always true when endTime == null
        .flatMapMany(_ -> missingMessages(topicContext, null))  // current = null
        .subscribeOn(Schedulers.boundedElastic());
``` [2](#0-1) 

Inside `missingMessages()`, when `current == null` and `last == null`:

```java
// lines 142–149
if (current == null) {
    long startTime = last != null
            ? last.getConsensusTimestamp() + 1
            : topicContext.getFilter().getStartTime();   // attacker-controlled: 0
    var gapFilter =
            topicContext.getFilter().toBuilder().startTime(startTime).build();
    // inherits endTime = null, limit = 0 from original filter
    return topicMessageRetriever.retrieve(gapFilter, false);  // throttled = false
}
``` [7](#0-6) 

**Why each existing guard fails:**

| Guard | Location | Failure reason |
|---|---|---|
| `isValidStartTime()` | `TopicMessageFilter.java:49–51` | Only checks `startTime <= now()`; `0` satisfies this |
| `flux.takeWhile(endTime)` | `TopicMessageServiceImpl.java:79–81` | Skipped when `endTime == null` |
| `flux.take(limit)` | `TopicMessageServiceImpl.java:83–85` | Skipped when `hasLimit()` is false (`limit == 0`) |
| `isComplete()` | `TopicMessageServiceImpl.java:203–205` | Returns `false` unconditionally when `endTime == null` |
| Normal gap-fill path | `TopicMessageServiceImpl.java:164–168` | Always sets bounded `endTime` and computed `limit`; safety-check path sets neither | [5](#0-4) [4](#0-3) [3](#0-2) [8](#0-7) 

**Root cause:** The safety-check branch assumes `filter.getStartTime()` is a recent timestamp, but `isValidStartTime()` only enforces an upper bound (`<= now()`), not a lower bound. The value `0` (nanoseconds since epoch, 1970-01-01) is fully valid. The branch also fails to impose any `endTime` or `limit` before delegating to the retriever, unlike the normal gap-fill path at lines 164–168 which always sets both.

## Impact Explanation
Each subscription with `startTime = 0`, no `endTime`, and no `limit` causes the server to issue a full table scan of `topic_message` from epoch start, unthrottled, one second after the connection is opened. On a node with significant topic history this is a multi-million-row query per connection. Multiple concurrent connections multiply the database load linearly. The database connection pool, heap, and `Schedulers.boundedElastic()` thread pool are all targets for exhaustion, causing denial of service for all subscribers on the affected node.

## Likelihood Explanation
The preconditions require no authentication, no special role, and no knowledge of internal state. Any gRPC client can set `startTime = 0` in a `ConsensusTopicQuery`. The trigger is deterministic: wait one second. The condition `last == null` is guaranteed when subscribing to a topic with no messages yet, or when the importer is temporarily unavailable. The attack is trivially repeatable and scriptable with any HCS SDK or raw gRPC client.

## Recommendation

1. **Bound the safety-check filter explicitly.** Before calling `retrieve`, set `endTime` to `DomainUtils.now()` and a reasonable `limit` (e.g., the retriever's configured page size) on `gapFilter`, mirroring the bounds applied by the normal gap-fill path at lines 164–168.
2. **Enforce a minimum `startTime` age.** Add a lower-bound validation in `isValidStartTime()` or at the gRPC ingress layer (e.g., reject `startTime` older than a configurable maximum lookback window).
3. **Apply the outer `takeWhile`/`take` guards unconditionally** (or with safe defaults) rather than only when the subscriber supplies an `endTime`/`limit`, so that internal gap-recovery queries are always bounded regardless of the original filter.

## Proof of Concept

```python
import grpc
from proto import consensus_service_pb2_grpc, consensus_query_pb2, timestamp_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# startTime = 0 (epoch), no endTime, no limit
query = consensus_query_pb2.ConsensusTopicQuery(
    topicID=...,                          # any existing topic
    consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
    # consensusEndTime omitted → null
    # limit omitted → 0
)

# After ~1 second the server issues:
# SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= 0
# with no LIMIT and no upper-bound predicate, unthrottled.
for msg in stub.subscribeTopic(query):
    pass  # repeat N times concurrently to amplify DB load
```

Each concurrent connection issues one unbounded full-table scan after one second. No authentication is required.
