Audit Report

## Title
Unbounded Growth of `cacheManagerSlotsPerContract` Internal Cache Map via Dynamic Cache Creation Leads to Memory Leak

## Summary
`cacheManagerSlotsPerContract` is the only `CaffeineCacheManager` bean in `EvmConfiguration.java` that omits `setCacheNames()`, placing it in Spring's dynamic cache creation mode. Every distinct `contractId` passed through `findStorageBatch()` permanently registers a new named `CaffeineCache` in the manager's internal `ConcurrentHashMap`. The outer `contractSlotsCache` eviction does not remove entries from this inner map, causing it to grow without bound.

## Finding Description

**Root cause — missing `setCacheNames()` on `cacheManagerSlotsPerContract`:**

Every other `CaffeineCacheManager` bean in `EvmConfiguration.java` calls `setCacheNames()` to lock the manager to a fixed set of named caches. The `slotsPerContract` bean does not:

```java
// EvmConfiguration.java lines 107-112
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    // ← NO setCacheNames() call
    return caffeineCacheManager;
}
``` [1](#0-0) 

Compare with every other bean, e.g. `cacheManagerContractSlots` at line 78: [2](#0-1) 

Without `setCacheNames()`, Spring's `CaffeineCacheManager` operates in *dynamic mode*: any call to `getCache(name)` with a previously unseen name creates a new `CaffeineCache` and stores it permanently in the manager's internal `ConcurrentHashMap<String, Cache> cacheMap`.

**Trigger site — `findStorageBatch()`:**

```java
// ContractStateServiceImpl.java lines 85-87
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [3](#0-2) 

The loader `() -> cacheManagerSlotsPerContract.getCache(contractId.toString())` fires on every cache miss in the outer `contractSlotsCache`. Each invocation with a new `contractId` string permanently registers a new named cache in `cacheManagerSlotsPerContract.cacheMap`.

**Why the outer cache does not protect the inner map:**

`contractSlots` is configured as `"expireAfterAccess=5m,maximumSize=3000,recordStats"`: [4](#0-3) 

`slotsPerContract` is configured as `"expireAfterAccess=5m,maximumSize=1500"`: [5](#0-4) 

When an entry is evicted from `contractSlotsCache` (outer, max 3000), the `CaffeineCache` *value* is removed from the outer Caffeine cache. However, the named cache entry in `cacheManagerSlotsPerContract.cacheMap` is **never removed**. Spring's `CaffeineCacheManager` has no eviction listener or cleanup hook tied to the outer cache's eviction events. The `cacheMap` is a plain `ConcurrentHashMap` that only grows.

**Existing mitigations and why they fail:**

- `contractSlotsCache` `maximumSize=3000`: limits active per-contract caches in the outer cache, but does **not** bound `cacheManagerSlotsPerContract.cacheMap`.
- `slotsPerContract` `maximumSize=1500`: limits entries *within* each per-contract cache, not the number of named caches in the manager.
- `expireAfterAccess=5m` on `slotsPerContract`: expires entries inside each per-contract Caffeine cache, but the `CaffeineCache` wrapper object and its internal Caffeine data structures remain allocated in `cacheMap` indefinitely. [6](#0-5) 

## Impact Explanation
Each `CaffeineCache` wrapping a Caffeine `BoundedLocalCache` with `maximumSize=1500` allocates internal data structures (ring buffers, eviction policy arrays, segment locks) even when empty. With tens of thousands of distinct `contractId` strings accumulated over time, `cacheManagerSlotsPerContract.cacheMap` grows monotonically with no upper bound and no cleanup mechanism. This leads to progressive heap exhaustion and eventual `OutOfMemoryError` on the web3 service JVM, crashing the mirror node's JSON-RPC layer. The leak persists across normal operation and resets only on process restart. [3](#0-2) 

## Likelihood Explanation
No authentication is required for `eth_call` or `eth_estimateGas`. The list of deployed contracts is publicly available via the mirror node REST API (`/api/v1/contracts`). An attacker can iterate through distinct deployed contracts, sending one `eth_call` per contract containing an `SLOAD` opcode. Each call that misses `contractSlotsCache` permanently adds one entry to `cacheManagerSlotsPerContract.cacheMap`. The attack is low-cost, repeatable, and accumulates damage persistently. [7](#0-6) 

## Recommendation
Add `setCacheNames()` to `cacheManagerSlotsPerContract` to disable dynamic mode. However, since the design intentionally uses per-contract named caches, the correct fix is to **not use `CaffeineCacheManager` as the per-contract cache registry at all**. Instead, maintain a bounded `LoadingCache<EntityId, Cache>` (a Caffeine cache of Caffeine caches) directly in `ContractStateServiceImpl`, with an eviction listener that calls `cache.invalidateAll()` on eviction. This ensures the number of live per-contract caches is bounded by the same `maximumSize` as the outer map, and evicted caches are properly cleaned up. [1](#0-0) 

## Proof of Concept
1. Deploy or identify N > 3000 distinct contracts on the target network that contain `SLOAD` opcodes.
2. For each contract address, send: `eth_call { to: <contractAddress>, data: <SLOAD-triggering calldata> }`.
3. Each call that misses `contractSlotsCache` invokes `cacheManagerSlotsPerContract.getCache(contractId.toString())`, permanently adding one entry to `cacheMap`.
4. After 3000 calls, `contractSlotsCache` begins evicting old entries. For each new contractId beyond 3000, a new permanent entry is added to `cacheMap`.
5. Monitor JVM heap via JMX/Actuator: `cacheManagerSlotsPerContract.cacheMap` size grows monotonically. After sufficient distinct contractIds, heap exhaustion and `OutOfMemoryError` occur.
6. Restart the service; the leak resumes immediately on the next batch of requests. [8](#0-7)
