Audit Report

## Title
Unauthenticated HTTP Node-List Fetch in `buildNetworkFromMirrorNodes` Enables MITM-Driven Network Routing Takeover

## Summary
`buildNetworkFromMirrorNodes()` in `pinger/mirror_node_client.go` fetches the Hiero consensus node list over a plain HTTP connection with no scheme enforcement, no TLS requirement, and no response integrity verification. A network-path attacker can substitute a crafted node list, causing the Hiero SDK client to be initialized exclusively against attacker-chosen nodes for the lifetime of the process.

## Finding Description

**`pinger/config.go` line 37** — the `mirrorRest` field is populated from the `-mirror-rest` flag or `HIERO_MIRROR_PINGER_REST` env var. The documented example is `http://mirror-rest:5551` — an explicit HTTP URL. [1](#0-0) 

**`pinger/config.go` line 133** — the only validation is a non-empty check when `network == "other"`. No `strings.HasPrefix(cfg.mirrorRest, "https://")` guard exists. [2](#0-1) 

**`pinger/mirror_node_client.go` lines 36–44** — `buildNetworkFromMirrorNodes` takes `cfg.mirrorRest` verbatim, trims trailing slashes, and appends the API path. No scheme check, no HTTPS upgrade. [3](#0-2) 

**`pinger/mirror_node_client.go` line 46** — a plain `http.Client` is constructed with only a timeout. No `Transport` override, no TLS configuration, no certificate pinning. [4](#0-3) 

**`pinger/mirror_node_client.go` lines 79–98** — the HTTP GET is issued and the response body is decoded directly into `nodesEnvelope` with no signature check, no HMAC, and no comparison against a known-good baseline. [5](#0-4) 

**`pinger/mirror_node_client.go` lines 100–124** — every `service_endpoint` from the (potentially attacker-controlled) response is inserted into the `network` map without any allowlist or sanity check beyond non-empty host and non-zero port. [6](#0-5) 

**`pinger/sdk_client.go` line 22** — the poisoned map is passed directly to `hiero.ClientForNetwork(netmap)`, configuring the SDK client exclusively with attacker-supplied node addresses. [7](#0-6) 

**`pinger/main.go` line 63** — `submitWithRetry` is called on every ticker tick against this client for the entire process lifetime. [8](#0-7) 

**Root cause:** The code assumes the mirror REST endpoint is reachable only over a trusted, secure channel. Neither assumption is enforced in code. The node list is fetched once at startup; a single successful interception permanently poisons the client.

## Impact Explanation

A successful MITM attack redirects 100% of the pinger's transaction traffic to attacker-chosen nodes. The concrete impacts are:

1. **Monitoring disruption** — the pinger believes it is exercising the real network but is communicating only with attacker-controlled endpoints. Liveness and readiness signals become meaningless.
2. **Transaction misdirection** — all signed transfers are submitted to nodes the attacker controls, not to legitimate consensus nodes. The attacker can observe transaction patterns (account IDs, amounts, timing) without ever obtaining the private key.
3. **Selective node load concentration** — if the injected list names only a small subset of real consensus nodes, those nodes absorb all pinger-originated gRPC traffic. At the default 1-second interval this alone is insufficient to exhaust node resources; however, if the pinger interval is reduced or multiple pinger instances are deployed, the concentrated load becomes more significant.

The claim that this alone can "shut down ≥30% of network processing nodes" is overstated for a single default-configured pinger instance. The primary confirmed impacts are monitoring integrity loss and transaction misdirection.

## Likelihood Explanation

**Required preconditions:**
1. `HIERO_MIRROR_PINGER_NETWORK=other` — required to invoke `buildNetworkFromMirrorNodes`.
2. `HIERO_MIRROR_PINGER_REST` is an HTTP URL — the documented default example is HTTP.
3. The attacker can intercept traffic between the pinger and the mirror REST service.

**Feasibility:** In a Kubernetes deployment, an attacker with a foothold in the same namespace can perform DNS spoofing of the internal service name `mirror-rest`. If the mirror REST URL resolves to a public endpoint over HTTP, any on-path network attacker (BGP hijacker, cloud provider network) can intercept without cluster access. The attack is a one-shot, persistent compromise: the node list is fetched once at startup, so a single successful interception poisons the client for the entire process lifetime with no further attacker interaction required.

## Recommendation

1. **Enforce HTTPS at config validation** — in `pinger/config.go` after line 133, add:
   ```go
   if !strings.HasPrefix(cfg.mirrorRest, "https://") {
       return cfg, fmt.Errorf("HIERO_MIRROR_PINGER_REST must use https://")
   }
   ```
2. **Use a TLS-enforcing transport** — in `pinger/mirror_node_client.go` line 46, replace the plain `http.Client` with one that sets `Transport: &http.Transport{TLSClientConfig: &tls.Config{MinVersion: tls.VersionTLS12}}` and optionally pins the CA.
3. **Update documentation** — change the example in the flag description from `http://mirror-rest:5551` to `https://mirror-rest:5551` to prevent operators from inadvertently configuring HTTP.
4. **Consider response integrity** — for defense-in-depth, validate that returned `node_account_id` values fall within the expected shard/realm/account range for the target network.
