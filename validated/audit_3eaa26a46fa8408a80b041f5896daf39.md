## Audit Report

## Title
Deleted Entity Disclosure via Long-Zero EVM Address in GraphQL `getByEvmAddressAndType()`

## Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, the long-zero EVM address branch routes to the unguarded `CrudRepository.findById()`, which has no `deleted is not true` predicate. Any caller who knows a deleted entity's numeric ID can craft a long-zero EVM address and retrieve the deleted entity through the GraphQL API, bypassing the deletion filter that protects the `findByEvmAddress()` path.

## Finding Description
In `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, the method `getByEvmAddressAndType()` branches on whether the first 12 bytes of the decoded EVM address are all zero:

```java
// lines 37–40
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())   // NO deleted filter
                           .filter(e -> e.getType() == type);
}
return entityRepository.findByEvmAddress(evmAddressBytes) // has deleted filter
                       .filter(e -> e.getType() == type);
``` [4](#0-3) 

The `findById()` call is the standard Spring Data `CrudRepository` method, generating `SELECT * FROM entity WHERE id = ?` with no additional predicate. In contrast, `findByEvmAddress()` is a custom `@Query` that explicitly appends `AND deleted is not true`:

```java
// lines 16–17
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
``` [5](#0-4) 

The `deleted` field is a nullable `Boolean` on `AbstractEntity` (the parent of `Entity`), meaning it can be `true`, `false`, or `null`. The `deleted is not true` idiom correctly handles all three states; `findById()` handles none of them. [6](#0-5) 

Note: `getByIdAndType()` at line 24–26 has the same structural issue (bare `findById()` with no deleted guard), but that is a separate code path not addressed by this report. [7](#0-6) 

## Impact Explanation
Any deleted Hashgraph entity — accounts, contracts, tokens, topics — whose numeric ID is known can be retrieved in full via the GraphQL endpoint. The `deleted` flag is set to `true` by transaction handlers such as `FileDeleteTransactionHandler`, `ContractDeleteTransactionHandler`, etc., meaning these entities are intentionally marked as no longer active. Exposing them violates the mirror node's intended view of network state and leaks historical entity data that operators have marked as deleted.

## Likelihood Explanation
No authentication or privileges are required. Entity IDs are sequential integers, publicly observable from the Hashgraph network and the mirror node REST API. An attacker constructs the long-zero address by zero-padding the 8-byte big-endian entity ID to 20 bytes (e.g., entity `1234` → `0x000000000000000000000000000004D2`) and submits a standard GraphQL query. The attack requires only an HTTP client and is fully repeatable.

## Recommendation
Add a deleted-entity guard to the `findById()` branch in `getByEvmAddressAndType()`. The simplest fix is to add a custom repository method with an explicit filter:

```java
// In EntityRepository.java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Then replace line 38 in `EntityServiceImpl.java`:
```java
return entityRepository.findByIdAndNotDeleted(buffer.getLong())
                       .filter(e -> e.getType() == type);
```

The same fix should be applied to `getByIdAndType()` at line 24–26 for consistency.

## Proof of Concept
1. Identify a deleted entity with numeric ID `N` (observable from REST API or network history).
2. Construct the long-zero EVM address: 12 zero bytes + 8-byte big-endian `N`. For `N = 1234`: `0x000000000000000000000000000004D2`.
3. Send a GraphQL query:
```graphql
query {
  account(input: { evmAddress: { address: "0x000000000000000000000000000004D2" } }) {
    id
    deleted
    # ... other fields
  }
}
```
4. The `buffer.getInt() == 0 && buffer.getLong() == 0` check passes, routing to `findById(1234)`.
5. Spring Data executes `SELECT * FROM entity WHERE id = 1234` — no `deleted` filter — and returns the deleted entity.
6. The only remaining check `.filter(e -> e.getType() == type)` validates entity type only, not deletion status, so the deleted entity is returned to the caller.
