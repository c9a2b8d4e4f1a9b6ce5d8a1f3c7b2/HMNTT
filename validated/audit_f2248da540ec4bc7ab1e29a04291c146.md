## Audit Report

## Title
DB Connection Pool Exhaustion via Unauthenticated Concurrent Alias-Based Requests to `getAccountCryptoAllowances`

## Summary
The `getAccountCryptoAllowances()` handler in `rest/controllers/cryptoAllowanceController.js` performs two sequential database queries per alias-based or EVM-address-based request. The REST API has no per-client rate limiting or in-flight request cap at either the application or ingress layer. An unauthenticated attacker can multiplex many concurrent requests over a single HTTP/2 connection, exhausting the DB connection pool and denying service to all users.

## Finding Description
**Code path:**

`rest/controllers/cryptoAllowanceController.js`, `getAccountCryptoAllowances()`, lines 76–98: [9](#0-8) 

Line 77 calls `EntityService.getEncodedId()`. When the path parameter is an alias or non-parsable EVM address, this issues a live DB query against the `entity` table: [10](#0-9) 

Line 80 then calls `CryptoAllowanceService.getAccountCryptoAllowances()`, issuing a second DB query against `crypto_allowance`. Each in-flight request holds a DB pool connection for the duration of both sequential queries. [11](#0-10) 

**DB pool:** Configured from `config.db.pool.maxConnections` and `config.db.pool.connectionTimeout`. The pool is bounded and shared across all concurrent requests on a pod. [12](#0-11) 

**No application-level rate limiting:** A search across all `rest/**/*.js` for `rateLimit`, `rateLimiter`, `express-rate-limit`, `throttle`, and `inFlightReq` returns zero hits. The middleware stack exports no throttling component. [4](#0-3) 

**No ingress-level per-client limiting:** The REST Traefik middleware block contains only `circuitBreaker` and `retry` — no `inFlightReq` and no `rateLimit`: [5](#0-4) 

Compare to the Rosetta chart, which explicitly configures both `inFlightReq` (per-IP, amount: 5) and `rateLimit` (per-host, average: 10): [6](#0-5) 

**GCP Gateway `maxRatePerEndpoint: 250`** is a per-backend-pod aggregate throughput cap, not a per-client concurrency control, and does not prevent a single client from holding many simultaneous connections. [7](#0-6) 

**Aggravating factor — retry amplification:** The `retry: attempts: 10` Traefik middleware causes each failed request (e.g., due to pool exhaustion) to be retried up to 10 times, multiplying pool pressure during an attack. [8](#0-7) 

## Impact Explanation
With the DB pool bounded per pod and a `connectionTimeout` that causes queued requests to time out, an attacker sending enough concurrent alias-based requests saturates the pool. All subsequent requests from any user queue and eventually time out, returning errors. Because the REST API is the primary read interface for Hedera Mirror Node consumers (wallets, explorers, dApps), sustained pool exhaustion constitutes a full REST API denial of service for the duration of the attack. The `statementTimeout` means each attacker request holds a connection for up to that duration, giving the attacker a favorable amplification ratio.

## Likelihood Explanation
The attack requires zero authentication, zero special privileges, and only a standard HTTP/2 client (`curl --http2`, Python `httpx`, `h2load`). The alias-based path (`/api/v1/accounts/{alias}/allowances/crypto`) is publicly documented and forces the two-query code path. HTTP/2 multiplexing allows all concurrent streams to share a single TCP connection, bypassing any IP-level connection-count controls. The attack is trivially repeatable and scriptable. No existing control detects or blocks it.

## Recommendation
1. **Add `inFlightReq` and `rateLimit` to the REST Traefik middleware**, mirroring the Rosetta chart configuration. This is the most direct fix and requires only a values change. [5](#0-4) 
2. **Add application-level concurrency limiting** (e.g., `express-rate-limit` or a semaphore on DB-bound handlers) so protection is not solely dependent on the ingress layer.
3. **Review the `retry: attempts: 10` setting** — retrying on pool exhaustion amplifies the problem; consider reducing attempts or scoping retries to network errors only.
4. **Consider a dedicated, smaller connection pool** for alias-resolution queries to prevent a single endpoint from starving the entire pool.

## Proof of Concept
```bash
# Send 50 concurrent alias-based requests over HTTP/2 using h2load
h2load -n 500 -c 1 -m 50 \
  "https://<mirror-node-host>/api/v1/accounts/AAAQEAYEAUDAOCAJCAIREEYUCULBOGAZ/allowances/crypto"
```
With 50 concurrent streams over a single TCP connection, all DB pool connections on the target pod are held simultaneously. Legitimate requests from other users begin timing out immediately. The single-connection nature of HTTP/2 means IP-based connection limits are irrelevant.
