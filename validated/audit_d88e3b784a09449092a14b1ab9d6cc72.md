# Audit Report

## Title
`ne`-Only Timestamp Filter Bypasses `bindTimestampRange` Enforcement in `optimizeTimestampFilters()`, Enabling Unbounded Contract Result Enumeration

## Summary
When a caller supplies only `timestamp=ne:X` to `/api/v1/contracts/results`, the `optimizeTimestampFilters()` function produces a SQL condition with no range bounds — only a bare `consensus_timestamp != X` exclusion — because `bindTimestampRange` is a no-op in its default configuration. This allows any unauthenticated user to paginate through the entire `contract_result` table, bypassing the `maxTransactionsTimestampRange` window that is supposed to limit query scope.

## Finding Description

**Code path in `rest/controllers/contractController.js`, `optimizeTimestampFilters()` (lines 384–404):**

`parseTimestampFilters` is called with `allowNe=true`, `allowOpenRange=true`, `validateRange=false`, and `strictCheckOverride=false`:

```js
const {range, eqValues, neValues} = utils.parseTimestampFilters(
  timestampFilters, false, true, true, false, false
);
```

With only `timestamp=ne:X` as input, this returns `{range: null, eqValues: [], neValues: [X]}` — confirmed by the test spec at line 1388–1393 of `rest/__tests__/utils.test.js`. [7](#0-6) 

Because `eqValues.length === 0`, the code delegates to `bindTimestampRange(null, order)`:

```js
const {range: optimizedRange, next} =
  eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
``` [8](#0-7) 

In `rest/timestampRange.js` lines 19–22, when `queryConfig.bindTimestampRange` is `false` (the default), the function immediately returns `{range}` unchanged — i.e., `{range: null}`:

```js
const bindTimestampRange = async (range, order) => {
  if (!queryConfig.bindTimestampRange) {
    return {range};   // range is null → no bounding
  }
  ...
};
``` [3](#0-2) 

With `optimizedRange === null`, neither the `begin` nor `end` guards fire, so no range conditions are added. Only the `ne` filter is pushed:

```js
neValues.forEach((value) =>
  filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value})
);
``` [9](#0-8) 

`updateConditionsAndParamsWithInValues` in `baseController.js` emits this directly as `consensus_timestamp != $N` with no range guard:

```js
} else {
  existingParams.push(filter.value);
  existingConditions.push(`${fullName}${filter.operator}$${position}`);
}
``` [5](#0-4) 

**Why existing checks fail:**

- `filterValidityChecks` for `TIMESTAMP` only validates the value via `isValidTimestampParam(val)`, not the operator — so `ne` is accepted at the input layer. [1](#0-0) 
- `parseTimestampFilters` with `allowOpenRange=true` skips the "must have both bounds or eq" check at line 1651. [10](#0-9) 
- The skip-on-empty guard at line 506 does not trigger because `optimizedTimestampFilters` contains the `ne` filter. [4](#0-3) 
- `strictCheckOverride=false` means strict checks defer to `config.strictTimestampParam`, which does not block a lone `ne` filter when `allowNe=true`. [11](#0-10) 

## Impact Explanation

The generated SQL becomes effectively:
```sql
SELECT … FROM contract_result WHERE consensus_timestamp != <X> ORDER BY … LIMIT <default>
```

With standard pagination (cursor + `limit`), an attacker can retrieve every row in the `contract_result` table. This:

1. **Bypasses `maxTransactionsTimestampRange`** — the intended time-window guard protecting both data confidentiality and database performance is completely circumvented.
2. **Causes full sequential table scans** on potentially very large partitioned tables, enabling a low-cost denial-of-service against the mirror node database.
3. **Exposes the complete contract execution history** (sender, calldata, results, gas) to any unauthenticated caller.

## Likelihood Explanation

- No authentication or special privilege is required; `/api/v1/contracts/results` is a public endpoint.
- The `ne` operator is explicitly accepted by `optimizeTimestampFilters` (`allowNe=true`) and passes `filterValidityChecks` for `TIMESTAMP`.
- `bindTimestampRange` defaults to `false` in standard deployments — the feature is opt-in.
- The attack requires a single HTTP request and standard pagination, making it trivially repeatable.

## Recommendation

1. **Reject lone `ne`-only timestamp filters at the input layer** for the `/api/v1/contracts/results` endpoint. Either extend `contractResultsFilterValidityChecks` to reject `ne` for `TIMESTAMP`, or add an explicit check in `optimizeTimestampFilters` that throws when `neValues` is non-empty and no range or `eq` values are present.

2. **Add a fallback bound in `optimizeTimestampFilters`** when `bindTimestampRange` is disabled: if `optimizedRange` is `null` after the call and `neValues` is non-empty, either reject the request or apply a default time window (e.g., last N days) before emitting the `ne` filter.

3. **Restrict the `TIMESTAMP` operator in `filterValidityChecks`** to `basicOperators` (which excludes `ne`) unless the endpoint explicitly opts in, consistent with how `SLOT` is handled. [12](#0-11) 

## Proof of Concept

```
GET /api/v1/contracts/results?timestamp=ne:1000000000000000000
```

With `bindTimestampRange=false` (default), this produces:
```sql
SELECT * FROM contract_result
WHERE consensus_timestamp != 1000000000000000000
ORDER BY consensus_timestamp DESC
LIMIT 25
```

Paginate using the returned `next` link to enumerate all rows in the table.
