Audit Report

## Title
Unauthenticated gRPC Topic Subscription Exhausts Shared `boundedElastic()` Thread Pool, Causing DoS for Legitimate Subscribers

## Summary
`PollingTopicMessageRetriever` schedules all per-subscription polling and timeout operations on the global shared `Schedulers.boundedElastic()` instance. Because the gRPC endpoint has no authentication, no per-IP connection limit, and no global subscription cap, an attacker can open an unbounded number of connections to saturate the finite thread pool, stalling legitimate subscribers' polling tasks.

## Finding Description

**Verified code references:**

`PollingTopicMessageRetriever` constructor assigns the global shared scheduler at line 41: [1](#0-0) 

Every subscription's repeat delay and timeout both execute on this shared scheduler (lines 52–59): [2](#0-1) 

The `poll()` method issues a blocking JDBC call via `Flux.fromStream(topicMessageRepository.findByFilter(newFilter))` at line 78. Because `RepeatSpec.withScheduler(scheduler)` fires re-subscriptions on the `boundedElastic()` thread, this blocking call holds that thread for the full query duration: [3](#0-2) 

The safety-check path in `TopicMessageServiceImpl` adds further pressure on the same global pool at line 70: [4](#0-3) 

`subscriberCount` is wired only as a Micrometer gauge (lines 52–55) and is never consulted to reject new subscriptions: [5](#0-4) 

`GrpcConfiguration` configures only `maxConcurrentCallsPerConnection`; no `maxConnections`, `maxConnectionAge`, or global subscriber cap is set: [6](#0-5) 

The only `GrpcInterceptor` in the repository lives in the **test** package and performs no authentication or authorization — there are zero production-path interceptors under `grpc/src/main/`: [7](#0-6) 

**Root cause:** `Schedulers.boundedElastic()` is a global singleton capped at `10 × CPU_cores` threads (with a 100,000-task queue). Each active subscription's polling loop occupies one of those threads for the duration of a blocking JDBC call. With no connection limit and no subscription quota, an attacker trivially exceeds the thread cap, causing all subsequent polling tasks — including those of legitimate subscribers — to queue behind the attacker's tasks.

## Impact Explanation
When the thread pool is saturated:
1. Legitimate subscribers' `repeatWhen` callbacks queue behind attacker tasks; historical message delivery stalls indefinitely.
2. The `.timeout(retrieverProperties.getTimeout(), scheduler)` check runs on the same pool; if the pool is full, timeout enforcement is delayed, keeping stalled subscriptions alive longer and worsening the backlog.
3. The `Mono.delay(1s).subscribeOn(Schedulers.boundedElastic())` safety-check also queues, breaking gap-recovery for live subscribers.
4. Result: complete or near-complete denial of the HCS topic subscription service for all legitimate clients, with no self-recovery until the attacker disconnects.

## Likelihood Explanation
Preconditions: none. The gRPC port (default 5600) is publicly reachable, requires no credentials (no production `ServerInterceptor` exists), and imposes no per-IP connection limit. A single attacker machine can open hundreds of TCP connections, each carrying up to 5 concurrent `subscribeTopic` calls (the only enforced limit) with `startTime=0` to maximize historical polling load. This is a trivial, repeatable, low-cost attack requiring only a gRPC client library.

## Recommendation
1. **Introduce a global subscription cap**: check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` when the limit is exceeded.
2. **Isolate the polling scheduler**: replace `Schedulers.boundedElastic()` with a dedicated, bounded `Schedulers.newBoundedElastic(...)` instance so attacker load cannot starve other Reactor work.
3. **Add server-level connection limits**: configure `NettyServerBuilder.maxConnectionAge`, `maxConnectionAgeGrace`, and a `maxConnections` limit in `GrpcConfiguration`.
4. **Add per-IP rate limiting**: implement a production `ServerInterceptor` that enforces per-IP connection and call-rate limits.
5. **Make the JDBC call non-blocking**: use R2DBC or offload via `subscribeOn` on the dedicated scheduler so threads are not held for the full query duration.

## Proof of Concept
```python
import grpc
import concurrent.futures
# proto-generated stub for ConsensusService
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc as cs
from com.hedera.mirror.api.proto import consensus_service_pb2 as cs_pb2

TARGET = "mirror-node-host:5600"

def flood():
    channel = grpc.insecure_channel(TARGET)
    stub = cs.ConsensusServiceStub(channel)
    req = cs_pb2.ConsensusTopicQuery(
        topicID=...,          # any valid topic
        consensusStartTime=0, # maximise historical polling load
    )
    # blocks until server closes; holds one boundedElastic() thread per call
    for _ in stub.subscribeTopic(req):
        pass

# Open 200 connections × 5 concurrent calls = 1000 simultaneous polling tasks
with concurrent.futures.ThreadPoolExecutor(max_workers=1000) as pool:
    futures = [pool.submit(flood) for _ in range(1000)]
    concurrent.futures.wait(futures)
# Legitimate subscribers now queue behind 1000 attacker tasks on the shared
# boundedElastic() pool (default cap: 10 × CPU cores threads).
```
