Audit Report

## Title
Unbounded Timestamp Range in Historical Hook Storage Query Causes Full-Table Scan DoS

## Summary
Supplying a single one-sided timestamp parameter (e.g., `timestamp=gte:0`) to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage` triggers the historical query path with an effective range of `[0, Long.MAX_VALUE]`, forcing PostgreSQL to scan and sort every `hook_storage_change` row for the targeted `(owner_id, hook_id)` pair before emitting any result. No intermediate row-count guard exists.

## Finding Description

**`isHistorical()` trigger:**

`HookStorageRequest.isHistorical()` delegates to `!timestamp.isEmpty()`. [1](#0-0) 

`Bound.isEmpty()` returns `true` only when **both** `lower` and `upper` are `null`. [2](#0-1) 

Supplying `timestamp=gte:0` sets `lower` to the `GTE:0` parameter and leaves `upper` as `null`, so `isEmpty()` returns `false` and `isHistorical()` returns `true`.

**Default bounds when one side is absent:**

`getAdjustedLowerRangeValue()` returns `0` when `lower == null`; with `gte:0` it also returns `0`. [3](#0-2) 

`adjustUpperBound()` returns `Long.MAX_VALUE` when `upper == null`. [4](#0-3) 

**Service dispatches to the expensive query:**

With no `key` filters, `HookServiceImpl.getHookStorageChange()` calls `findByKeyBetweenAndTimestampBetween` with `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE`. [5](#0-4) 

**The unbounded native SQL query:**

```sql
select distinct on (key)
     owner_id, hook_id, key, value_written as "value", ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id  = :hookId
  and key >= :keyLowerBound
  and key <= :keyUpperBound
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
``` [6](#0-5) 

PostgreSQL's `DISTINCT ON (key)` requires a full sort of all matching rows by `(key, consensus_timestamp DESC)` (injected by `getPageRequest()`) before the first deduplicated row can be emitted. [7](#0-6) 

The `LIMIT` from `Pageable` is applied **after** this full sort; it does not reduce the intermediate scan.

**Why existing checks are insufficient:**

- `@Size(max = 2)` on `timestamps` limits the number of parameters to 2, not the width of the range.
- `@Max(MAX_LIMIT)` (`limit ≤ 100`) caps output rows, not the intermediate sort set. [8](#0-7) 

The table is range-partitioned by `consensus_timestamp`, meaning `BETWEEN 0 AND Long.MAX_VALUE` forces a scan across **all** partitions. [9](#0-8) 

## Impact Explanation
Every request with a wide timestamp range forces PostgreSQL to sort the entire `hook_storage_change` dataset for the targeted `(owner_id, hook_id)` across all time partitions. A hook with millions of storage-change records will cause repeated disk spills when the sort exceeds `work_mem`, consuming I/O, CPU, and memory. Sending this request in a loop from a single client can significantly increase DB processing load. Because the query is scoped to one `(owner_id, hook_id)` pair, the attacker only needs to identify one high-activity hook — information that is publicly observable on-chain.

## Likelihood Explanation
No authentication or authorization is required. `ownerId` and `hookId` are public blockchain identifiers discoverable by any observer. The endpoint is reachable by any HTTP client. The exploit requires a single crafted GET request and is trivially repeatable in a loop. No special tooling, credentials, or elevated privileges are needed.

## Recommendation
1. **Require both bounds when a timestamp is supplied**: reject requests where only one side of the timestamp range is provided, or default the missing side to a narrow window (e.g., ±1 hour from the supplied bound).
2. **Enforce a maximum timestamp range width**: validate that `adjustUpperBound() - getAdjustedLowerRangeValue()` does not exceed a configurable maximum (e.g., 1 week in nanoseconds) in `Bound` or in the controller before the request is built.
3. **Add a DB-level statement timeout** for this query class so runaway sorts are killed before exhausting `work_mem`.
4. **Consider a covering index** on `(owner_id, hook_id, key, consensus_timestamp DESC)` so PostgreSQL can satisfy `DISTINCT ON (key)` via an index scan rather than a full sort.

## Proof of Concept
```
GET /api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0
```
This single unauthenticated request sets `timestampLowerBound=0` and `timestampUpperBound=9223372036854775807` (Long.MAX_VALUE), triggering a full scan and in-memory sort of every `hook_storage_change` row for `(owner_id=123, hook_id=1)` across all time partitions. Repeating this in a loop against a high-activity hook exhausts PostgreSQL sort memory and saturates DB CPU.
