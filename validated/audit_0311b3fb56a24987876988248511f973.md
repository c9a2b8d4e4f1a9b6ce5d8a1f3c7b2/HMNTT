Audit Report

## Title
Missing Non-Negative Validation on EVM Long-Zero Address Entity Num Allows Cross-Shard Entity Confusion

## Summary
`getByEvmAddressAndType()` in `EntityServiceImpl` identifies a "long-zero" EVM address by verifying that bytes 0–3 (shard) and bytes 4–11 (realm) are zero, then passes bytes 12–19 raw via `ByteBuffer.getLong()` directly to `entityRepository.findById()` without validating the value is non-negative. Because the entity `id` column stores a bit-packed encoded long that is legitimately negative for entities with non-zero shard/realm, a crafted address with bytes 12–19 set to a negative encoded ID will resolve to a completely different entity than the address implies.

## Finding Description
**Exact code path** — `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {          // bytes 0-3 = shard, bytes 4-11 = realm
        return entityRepository.findById(buffer.getLong())         // bytes 12-19 — NO non-negative check
                .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
``` [1](#0-0) 

**Root cause:** The long-zero branch is intended to resolve `0.0.<num>` entity IDs. The code correctly verifies shard (bytes 0–3) and realm (bytes 4–11) are zero, but never validates that the extracted `num` (bytes 12–19) is `>= 0` before passing it to `findById()`.

**Why negative IDs exist in the database:** `EntityId.encode()` packs shard (10 bits), realm (16 bits), and num (38 bits) into a single `long`. When `shard == 0 && realm == 0`, the encoded ID equals `num` directly (always non-negative). For non-zero shard/realm, the bit-shift arithmetic produces negative longs: [7](#0-6) 

This is confirmed by `EntityIdTest.java`:
- `EntityId.of(1023, 65535, 274877906943)` → encoded id = **-1**
- `EntityId.of(1023, 0, 0)` → encoded id = **-18014398509481984** [3](#0-2) 

`EntityRepositoryTest.java` line 18 explicitly persists and retrieves an entity with ID `-2`, confirming negative IDs are valid, queryable rows in the database: [4](#0-3) 

**Why existing checks are insufficient:**
- The `@Pattern(regexp = "^(0x)?[a-fA-F0-9]{40}$")` annotation only validates hex format and length — it does not constrain the semantic value of bytes 12–19. [6](#0-5) 
- `decodeEvmAddress()` only strips the `0x` prefix and hex-decodes — no range validation. [5](#0-4) 
- `EntityId.encode()` validates `num >= 0`, but it is **never called** in this code path; the raw `long` from `ByteBuffer` bypasses it entirely. [8](#0-7) 

## Impact Explanation
An unprivileged attacker can resolve any entity with a non-zero shard or realm (i.e., any entity whose encoded ID is negative) by supplying a crafted "long-zero" EVM address. The GraphQL response will present that entity's account/contract data as if it were the entity at `0.0.<negative-num>`, which is not a valid entity. This constitutes entity confusion: the caller receives authoritative-looking state for an entity that does not correspond to the supplied address. In multi-shard/multi-realm deployments (which the codebase explicitly supports), this allows an attacker to enumerate and retrieve any entity by its encoded ID rather than its EVM address, bypassing the intended address-to-entity resolution semantics.

## Likelihood Explanation
No privileges are required. The attack requires only the ability to send a GraphQL query with a crafted hex string — a capability available to any unauthenticated user of the public mirror node API. The encoded IDs of target entities are deterministically computable from their `shard.realm.num` values using the documented encoding formula. The attack is trivially repeatable and scriptable. It is constrained to deployments where entities with non-zero shard/realm exist (negative encoded IDs in the DB), which is the case for any multi-shard deployment.

## Recommendation
Add a non-negative check on the extracted `num` before calling `findById()`. If the value is negative, it cannot represent a valid `0.0.<num>` entity and the long-zero branch should not be taken:

```java
long num = buffer.getLong();
if (num < 0) {
    return Optional.empty(); // or fall through to findByEvmAddress
}
return entityRepository.findById(num).filter(e -> e.getType() == type);
```

Alternatively, reconstruct the lookup through `EntityId.of(0, 0, num)` which will throw `InvalidEntityException` for negative values, enforcing the same constraint as the rest of the entity ID resolution paths.

## Proof of Concept
1. Ensure an entity `1023.65535.274877906943` (encoded ID = `-1`) exists in the database.
2. Send the following GraphQL query:
```graphql
{
  account(input: { evmAddress: "000000000000000000000000FFFFFFFFFFFFFFFF" }) {
    id
  }
}
```
3. The long-zero branch is taken because bytes 0–3 = `0` and bytes 4–11 = `0`.
4. `buffer.getLong()` returns `-1L`.
5. `entityRepository.findById(-1L)` executes `SELECT * FROM entity WHERE id = -1`.
6. The entity `1023.65535.274877906943` is returned — a completely different entity from what the address format implies.

    private static long encode(long shard, long realm, long num) {
        if (shard > SHARD_MASK || shard < 0 || realm > REALM_MASK || realm < 0 || num > NUM_MASK || num < 0) {
            throw new InvalidEntityException("Invalid entity ID: " + shard + "." + realm + "." + num);
