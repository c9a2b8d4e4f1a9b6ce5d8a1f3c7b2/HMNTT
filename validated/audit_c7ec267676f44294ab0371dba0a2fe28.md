# Audit Report

## Title
Unbounded Alias Length Enables Unauthenticated CPU/DB Exhaustion via `getEncodedId()`

## Summary
The `accountAliasRegex` in `rest/accountAlias.js` imposes no upper bound on the alias component length. OpenAPI request validation is both hardcoded off (`validateRequests: false`) and disabled by default at the middleware-loading level. Any unauthenticated caller can submit arbitrarily long alias strings to any endpoint routing through `EntityService.getEncodedId()`, forcing unbounded regex evaluation, base32 decoding, and a fresh database query per request with no alias-level caching.

## Finding Description

**Code path:**

1. `rest/accountAlias.js` line 10 — `accountAliasRegex = /^(\d{1,5}\.){0,2}[A-Z2-7]+$/` — the `[A-Z2-7]+` quantifier is unbounded. [1](#0-0) 

2. `rest/accountAlias.js` lines 41–43 — `isValid()` runs this unbounded regex against attacker-controlled input. [10](#0-9) 

3. `rest/accountAlias.js` lines 51–64 — `fromString()` splits on `.` and passes the alias component to the constructor. [11](#0-10) 

4. `rest/accountAlias.js` line 24 — constructor calls `base32.decode(base32Alias)` with no length guard, allocating a `Uint8Array` proportional to input length. [12](#0-11) 

5. `rest/base32.js` line 13 — `decode` delegates to `rfc4648`'s `base32.parse()` with `{loose: true}`, no length check. [13](#0-12) 

6. `rest/service/entityService.js` lines 17–20 and 43 — the decoded blob is sent as `$1` in a parameterized DB query. [14](#0-13) [15](#0-14) 

7. `rest/service/entityService.js` lines 125–126 — `getEncodedId()` calls `AccountAlias.fromString()` then `getAccountIdFromAlias()` with no alias-level caching. [5](#0-4) 

**OpenAPI validation is disabled at two levels:**

- `validateRequests: false` is hardcoded in `openApiValidator()`, so even when the middleware is loaded, request bodies/params are never validated against the spec's bounded alias pattern. [16](#0-15) 

- The middleware is not loaded at all unless `openapi.validation.enabled` is set to `true` (default: `false`). [3](#0-2) [17](#0-16) 

The OpenAPI spec defines a bounded pattern for `AccountAlias` that would reject oversized inputs, but it is never enforced: [4](#0-3) 

**Response cache does not protect alias resolution:**

The cache key is `md5(req.originalUrl)`, so varying the alias by one character per request produces a distinct key and forces a fresh DB query every time. The response cache is also disabled by default. [7](#0-6) [8](#0-7) 

The `parseCached` LRU cache in `entityId.js` covers only `EntityId` (numeric/EVM) parsing, not alias resolution, so it provides no protection here. [6](#0-5) 

## Impact Explanation
Each request with a large alias string forces: O(n) regex evaluation, O(n) string split, O(n) base32 decode allocating a `Uint8Array`, and a DB query transmitting a binary blob parameter. With concurrent requests using distinct alias strings (cache bypass), an attacker can exhaust the DB connection pool (default: 10 connections) and saturate Node.js CPU and heap. This constitutes a non-network DoS against the REST API process and its backing database. Note: practical alias size is bounded by Node.js's HTTP parser URL limit (~8 KB by default), so "megabyte-scale" strings are not achievable, but even kilobyte-scale aliases with high concurrency are sufficient to exhaust the DB pool. [9](#0-8) 

## Likelihood Explanation
No authentication is required. All affected endpoints (`/api/v1/accounts/:id`, `/api/v1/balances`, `/api/v1/tokens/:id`, etc.) are public. The attack requires only an HTTP client sending requests in a loop with per-request alias variation. It is trivially scriptable. [18](#0-17) 

## Recommendation

1. **Add a length cap to `accountAliasRegex`**: Change `[A-Z2-7]+` to `[A-Z2-7]{2,128}` (or a tighter bound matching the OpenAPI spec's effective maximum of ~70 characters as seen in the Java implementation). [19](#0-18) [20](#0-19) 

2. **Enable OpenAPI request validation** or, at minimum, change `validateRequests: false` to `true` so the spec-level alias pattern is enforced at the middleware layer. [21](#0-20) 

3. **Add an alias-level in-process cache** in `EntityService.getAccountIdFromAlias()` (similar to the existing `parseCached` LRU for entity IDs) to avoid a DB round-trip for repeated alias lookups. [22](#0-21) 

4. **Deploy infrastructure-level rate limiting** (e.g., via the existing Traefik middleware support) to bound request rates per IP on public endpoints. [23](#0-22) 

## Proof of Concept

```bash
# Generate a large but HTTP-valid alias (~7000 chars, all valid base32 chars)
ALIAS=$(python3 -c "print('A' * 7000)")

# Flood with distinct aliases to bypass cache and force a DB query per request
for i in $(seq 1 500); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.${ALIAS}${i}" &
done
wait
```

Each request passes `isValid()`, proceeds through `fromString()` → `base32.decode()` → `getAccountFromAlias()` → DB query. With 500 concurrent requests and a pool of 10 DB connections, the pool is saturated and legitimate requests are queued or rejected. [24](#0-23)
