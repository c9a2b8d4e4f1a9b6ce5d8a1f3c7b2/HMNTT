Audit Report

## Title
UNION ALL Subquery Overlap Produces Duplicate Rows When Primary Bound Lower == Upper Value

## Summary
When `spender.id=gte:N&spender.id=lte:N` (identical bounds) is combined with a secondary `token.id` range, `getLowerFilters()` and `getUpperFilters()` both independently emit `spender = N` as their primary condition. Because `getQuery()` joins subqueries with `UNION ALL`, any row satisfying both the lower-subquery token condition and the upper-subquery token condition is returned twice. No existing validation rejects this degenerate range.

## Finding Description

**Verified code path:**

`extractTokenMultiUnionQuery()` in `tokenAllowanceController.js` calls `validateBounds()`, then `getLowerFilters()`, `getInnerFilters()`, and `getUpperFilters()` from `baseController.js`. The results are passed to `TokenAllowanceService.getQuery()` which assembles them with `union all`.

**Trace with `spender.id=gte:5&spender.id=lte:5&token.id=gte:10&token.id=lte:20`:**

After `Bound.parse()` in `bound.js`:
- `primary.lower = { operator: '>=', value: 5 }` (gte → `lower` field)
- `primary.upper = { operator: '<=', value: 5 }` (lte → `upper` field)
- `primary.equal = null` [1](#0-0) 

**`validateBoundsRange`** only rejects `hasBound() && hasEqual()`. A pure gte+lte range has `hasEqual() = false`, so it passes silently. [2](#0-1) 

**`validateLowerBounds`** rejects only when `primary.lower.operator === gt`. Since the operator is `gte`, the condition evaluates to false and no error is thrown. [3](#0-2) 

**`validateUpperBounds`** similarly only rejects `lt`, not `lte`. [4](#0-3) 

**`getLowerFilters`**: `primary.hasLower()` and `secondary.hasLower()` are both true, so the branch at line 138 fires and emits `[{spender = 5}, {token_id >= 10}]`. [5](#0-4) 

**`getUpperFilters`**: `primary.hasUpper()` and `secondary.hasUpper()` are both true, so it emits `[{spender = 5}, {token_id <= 20}]`. [6](#0-5) 

**`getInnerFilters`**: Both `primary.hasBound()` and `secondary.hasBound()` are true, so it emits `[{spender > 5}, {spender < 5}]` — a logically impossible but non-empty array. [7](#0-6) 

In `getQuery()`, all three filter arrays have `length !== 0`, so all three subqueries are included and joined with `union all`: [8](#0-7) 

The resulting SQL is:
```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ORDER BY spender ASC, token_id ASC LIMIT $2)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ORDER BY spender ASC, token_id ASC LIMIT $2)
ORDER BY spender ASC, token_id ASC LIMIT $2
```
where `$3=$7=5`, `$4=10`, `$5=$6=5`, `$8=20`. Any row `(spender=5, token_id=15)` satisfies subquery 1 (`token_id >= 10`) and subquery 3 (`token_id <= 20`), appearing **twice** in the result. Subquery 2 is vacuously empty.

## Impact Explanation
`GET /accounts/{id}/allowances/tokens` returns duplicate allowance rows. Clients summing or displaying allowances see inflated values. Because each subquery and the outer query both apply `LIMIT $2`, duplicates consume result slots and can suppress legitimate rows from the page, causing silent data omission in pagination. [9](#0-8) 

## Likelihood Explanation
The endpoint requires no authentication. The trigger is a single HTTP request with two query parameters set to the same numeric value (e.g., `?spender.id=gte:5&spender.id=lte:5&token.id=gte:10&token.id=lte:20`). It is deterministically reproducible by any caller who reads the API documentation. [10](#0-9) 

## Recommendation
Add a check in `validateBounds` (or a dedicated `validatePrimaryBoundRange`) that rejects the case where `primary.hasLower() && primary.hasUpper() && primary.lower.value === primary.upper.value`. This degenerate range should either be rejected with an `InvalidArgumentError` or collapsed into a single `eq` filter before the subquery split logic runs, eliminating the overlapping lower/upper subqueries. [11](#0-10) 

## Proof of Concept
```
GET /api/v1/accounts/0.0.100/allowances/tokens
    ?spender.id=gte:5&spender.id=lte:5&token.id=gte:10&token.id=lte:20
```
Any `token_allowance` row with `owner=100`, `spender=5`, `10 <= token_id <= 20`, and `amount > 0` will appear twice in the `allowances` array of the response. The middle subquery (`spender > 5 AND spender < 5`) returns zero rows and does not mask the duplication. [12](#0-11)

  getQuery(query) {
    const {lower, inner, upper, order, ownerAccountId, limit} = query;
    const params = [ownerAccountId, limit];
    const accountIdCondition = `${TokenAllowance.OWNER} = $1`;
    const limitClause = super.getLimitQuery(2);
    const orderClause = super.getOrderByQuery(
      ...TokenAllowanceService.orderByColumns.map((column) => OrderSpec.from(column, order))
    );

    const subQueries = [lower, inner, upper]
      .filter((filters) => filters.length !== 0)
      .map((filters) => this.getSubQuery(filters, params, accountIdCondition, limitClause, orderClause));

    let sqlQuery;
    if (subQueries.length === 0) {
      // if all three filters are empty, the subqueries will be empty too, just create the query with empty filters
      sqlQuery = this.getSubQuery([], params, accountIdCondition, limitClause, orderClause);
    } else if (subQueries.length === 1) {
      sqlQuery = subQueries[0];
    } else {
      sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
    }

    return {sqlQuery, params};
  }
