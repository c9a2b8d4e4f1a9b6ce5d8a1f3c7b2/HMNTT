# Audit Report

## Title
Connection Pool Exhaustion DoS via Unbounded Concurrent Requests in Rosetta API

## Summary
The `Pool` struct carries `MaxOpenConnections int` with no minimum-value enforcement. `ConnectToDb()` passes this value directly to `SetMaxOpenConns()`, which Go's `database/sql` interprets as **unlimited connections** when the value is `0`. Combined with the absence of application-level rate limiting and the optional nature of the Traefik middleware, this creates a realistic DoS vector.

## Finding Description

**Pool struct — no validation:**
The `Pool` struct in `rosetta/app/config/types.go` defines `MaxOpenConnections` as a plain `int` with no minimum-value constraint or validation tag. [1](#0-0) 

**Raw value passed to `SetMaxOpenConns`:**
`ConnectToDb()` in `rosetta/app/db/db.go` passes the config value directly to `sqlDb.SetMaxOpenConns()` without any guard. [2](#0-1) 

In Go's `database/sql`, `SetMaxOpenConns(0)` means **no limit** on open connections. An operator who omits the field or sets it to `0` silently enables an unbounded pool.

**No application-level rate limiting:**
The middleware chain in `rosetta/main.go` consists only of `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — none of which throttle requests or limit concurrency. [3](#0-2) 

The `rosetta/app/middleware/` directory contains only `health.go`, `metrics.go`, and `trace.go` — no rate-limiting or in-flight-request-limiting middleware exists at the application layer.


**Traefik middleware is strictly optional:**
The Helm chart's `values.yaml` sets `global.middleware: false` by default, meaning the `inFlightReq` (amount: 5) and `rateLimit` (average: 10) Traefik middleware rules are not applied in default or non-Kubernetes deployments. [4](#0-3) [5](#0-4) 

**Note on default config:** The application loads an embedded `application.yml` as its baseline defaults (`//go:embed application.yml`). If that file sets `MaxOpenConnections` to a non-zero value, the zero-value risk applies only to operators who explicitly override it to `0`. The worst-case scenario (unbounded pool) is therefore conditional on misconfiguration, but the bounded-pool exhaustion + indefinite connection-wait scenario applies regardless. [6](#0-5) 

## Impact Explanation
- With `MaxOpenConnections=0`: the PostgreSQL server can be overwhelmed as each concurrent request opens a new connection, eventually hitting the server's own `max_connections` limit and causing rejections across all mirror-node services sharing the database.
- With a bounded pool (e.g., default 30 or 100): Go's `database/sql` has no built-in connection-acquisition timeout. Goroutines queue indefinitely waiting for a free connection, causing the service to hang under sustained concurrent load. The HTTP `WriteTimeout` provides a partial mitigation, but only if explicitly configured to a low value.
- The `statementTimeout` (default 20 s) limits query execution time but does **not** limit how long a goroutine waits to *acquire* a connection from an exhausted pool.

## Likelihood Explanation
The Rosetta API is a public, unauthenticated HTTP endpoint. Sending hundreds of concurrent requests is trivially achievable with standard tools (`wrk`, `hey`, `ab`). The Traefik middleware that would otherwise throttle requests is disabled by default (`global.middleware: false`), meaning most non-Kubernetes and default Kubernetes deployments have no infrastructure-level protection either.

## Recommendation
1. **Enforce a minimum value for `MaxOpenConnections`**: Add startup validation that rejects or corrects a value of `0` (or any value below a safe minimum), logging a clear error.
2. **Set a connection-acquisition timeout**: Use `db.SetConnMaxIdleTime()` and consider wrapping connection acquisition with a context deadline to prevent indefinite goroutine queuing.
3. **Add application-level concurrency limiting**: Implement an in-flight request limiter middleware (e.g., using a semaphore or `golang.org/x/net/netutil`) so the protection is present regardless of deployment environment.
4. **Enable Traefik middleware by default**: Change `global.middleware` default to `true` in the Helm chart, or document clearly that it must be enabled for production deployments.

## Proof of Concept
```
# With MaxOpenConnections=0 or a bounded pool:
wrk -t 50 -c 500 -d 30s -s post.lua http://<rosetta-host>:5700/block

# post.lua sends a valid /block POST body
# Each request triggers a DB query; with 500 concurrent connections
# and no rate limiting, the pool is exhausted and new requests queue
# indefinitely (bounded pool) or open new DB connections until the
# PostgreSQL server hits max_connections (unbounded pool).
```
