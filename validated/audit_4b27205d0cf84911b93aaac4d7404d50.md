Audit Report

## Title
Missing `errorMessage` Sanitization Allows Null-Byte DoS in COPY-Based Batch Insert

## Summary
`ContractResultServiceImpl.processContractResult()` assigns `functionResult.getErrorMessage()` directly to `contractResult.errorMessage` without calling `DomainUtils.sanitize()`. The `BatchInserter` writes `ContractResult` rows to PostgreSQL via `COPY FROM STDIN WITH CSV`. PostgreSQL rejects null bytes (`\u0000`) in `text` columns at the wire level, so a single poisoned row aborts the entire in-flight COPY batch and propagates a `ParserException`.

## Finding Description

**Exact code path confirmed:**

`ContractResultServiceImpl` line 251 sets `errorMessage` with no sanitization: [1](#0-0) 

`ContractResult.errorMessage` is a plain `String` field with no custom setter and no sanitization: [2](#0-1) 

By contrast, `AbstractEntity.setMemo()` and its builder both call `DomainUtils.sanitize()`: [3](#0-2) [4](#0-3) 

`DomainUtils.sanitize()` is explicitly documented and implemented to replace the null character (`\u0000`) with the Unicode replacement character (`\uFFFD`) for PostgreSQL compatibility: [5](#0-4) 

`BatchInserter` serializes the entire POJO collection via `CsvMapper` and streams it into a `PGCopyOutputStream` using `COPY … FROM STDIN WITH CSV`: [6](#0-5) [7](#0-6) 

Any exception during the COPY stream cancels the in-flight `copyIn` and is re-thrown as a `ParserException`, aborting the entire batch: [8](#0-7) [9](#0-8) 

**Root cause:** `ContractResult.errorMessage` has no custom setter, no `@Column` constraint, and no sanitization call in the service layer, unlike every other user-influenced string field in the domain model. The `BatchInserter` has no per-field filtering; it serializes the entire POJO graph.

**Why existing checks fail:** `DomainUtils.sanitize()` is never invoked for `errorMessage`. The `BatchInserter` performs no per-field null-byte filtering before streaming to PostgreSQL.

## Impact Explanation
A single `ContractResult` row with a null byte in `errorMessage` causes the entire COPY batch for that flush cycle to be aborted. The `ParserException` propagates upward through the importer's record-stream parser, stalling or causing indefinite retries on the same poisoned record file. This halts all contract-result writes, breaking downstream REST API queries for contract results, Ethereum-compatible JSON-RPC calls, and any application relying on mirror-node contract data.

## Likelihood Explanation
The protobuf `string` type in Java permits `\u0000` (it is valid UTF-8 / U+0000). The Hedera consensus node records the raw EVM revert reason in `ContractFunctionResult.error_message`; if the revert payload contains null bytes (e.g., from ABI-encoding padding or an assembly-crafted revert), those bytes reach the mirror node verbatim. No special privilege is required — a standard `ContractCall` transaction is sufficient. The attack is repeatable: the attacker can re-submit the transaction every time the importer recovers.

## Recommendation
Apply `DomainUtils.sanitize()` to `errorMessage` in `ContractResultServiceImpl.processContractResult()`, consistent with how `memo` and other user-influenced string fields are handled:

```java
// Before (line 251):
contractResult.setErrorMessage(functionResult.getErrorMessage());

// After:
contractResult.setErrorMessage(DomainUtils.sanitize(functionResult.getErrorMessage()));
```

Alternatively, add a custom `setErrorMessage(String errorMessage)` setter in `ContractResult` that calls `DomainUtils.sanitize()` internally, matching the pattern used in `AbstractEntity.setMemo()`. [3](#0-2) 

## Proof of Concept
1. Deploy a Solidity contract that uses inline assembly to revert with a payload containing `\x00`:
   ```solidity
   assembly { mstore(0x00, 0x00) revert(0x00, 1) }
   ```
2. Submit a `ContractCall` transaction targeting this contract on the Hedera network.
3. The consensus node records the revert reason (containing `\x00`) in `ContractFunctionResult.error_message`.
4. The mirror-node importer reads the record file, calls `ContractResultServiceImpl.processContractResult()`, and sets `contractResult.errorMessage` to the unsanitized string.
5. When `BatchInserter.persistItems()` streams the batch via `PGCopyOutputStream`, PostgreSQL rejects the null byte, cancels the COPY, and throws an exception.
6. The exception is caught at `BatchInserter.persist()` line 110–111 and re-thrown as a `ParserException`, aborting the entire flush cycle for `ContractResult` rows.
7. The importer stalls on the same record file, preventing all subsequent contract-result writes.
