Audit Report

## Title
Global Shared `gasLimitBucket` Exhaustion via Max-Gas Burst — Unauthenticated DoS

## Summary
The `gasLimitBucket` is a single process-wide token bucket with no per-user or per-IP partitioning. An unauthenticated attacker sending requests at the maximum permitted rate (500 req/s), each with the default/maximum gas value (15,000,000), consumes exactly 750,000 scaled tokens per second — equal to the bucket's full capacity and full refill rate. This keeps the bucket perpetually near-zero, causing all concurrent legitimate users to receive HTTP 429 `ThrottleException("Gas per second rate limit exceeded.")` for the duration of the attack.

## Finding Description

**Verified code path:**

`ContractController.call()` calls `throttleManager.throttle(request)` at line 40 before `validateContractMaxGasLimit` at line 41: [1](#0-0) 

Inside `ThrottleManagerImpl.throttle()`, the gas bucket is consumed at line 40 with no per-caller isolation: [2](#0-1) 

`ThrottleConfiguration` creates the `gasLimitBucket` as a single Spring `@Bean` (singleton) with capacity and refill rate both equal to `scaleGas(gasPerSecond)`: [3](#0-2) 

**Verified arithmetic:**

- `gasPerSecond` default = `7_500_000_000L`, `GAS_SCALE_FACTOR` = `10_000L` → `getGasPerSecond()` = `scaleGas(7_500_000_000)` = **750,000 tokens** (bucket capacity and refill rate): [4](#0-3) 

- Default `gas` in `ContractCallRequest` = `15_000_000L`; `scaleGas(15_000_000)` = **1,500 tokens per request**: [5](#0-4) 

- `requestsPerSecond` default = **500 req/s**: [6](#0-5) 

- Attacker throughput: **500 × 1,500 = 750,000 tokens/s = full bucket capacity = full refill rate**. The bucket is perpetually drained.

**Why `validateContractMaxGasLimit` does not restore tokens:**

`maxGasLimit` defaults to `15_000_000L` and the check is strict greater-than (`gas > maxGasLimit`): [7](#0-6) [8](#0-7) 

A request with `gas = 15_000_000` satisfies `15_000_000 > 15_000_000 == false`, so no `InvalidParametersException` is thrown and `throttleManager.restore()` in the catch block is never reached. Tokens are consumed and not returned at the controller layer.

**Why `restoreGasToBucket` does not save it:**

`restoreGasToBucket` runs in the `finally` block of `doProcessCall`, i.e., only *after* execution completes: [9](#0-8) 

During the burst window — before any execution finishes — the bucket is already empty. The attacker pipelines new requests faster than old ones complete. Even with `gasLimitRefundPercent = 100` restoring tokens for failed calls, the attacker immediately re-consumes them with the next wave. [10](#0-9) 

## Impact Explanation
All concurrent legitimate users receive HTTP 429 `ThrottleException("Gas per second rate limit exceeded.")` for the duration of the attack. The `/api/v1/contracts/call` endpoint is effectively unavailable for all EVM call and estimate workloads. No authentication, funds, or special privileges are required.

## Likelihood Explanation
Any external actor with a basic HTTP client can execute this. The default `gas` value in `ContractCallRequest` is already `15_000_000L`, so no special request crafting is needed — the attacker simply sends the default request body at the maximum allowed rate. The attack is stateless, requires no on-chain state, and has zero cost to the attacker. The arithmetic relationship between the rate limit and gas bucket capacity makes this self-sustaining at exactly the permitted request rate.

## Recommendation
1. **Per-IP or per-client gas bucket partitioning:** Introduce per-source-IP (or authenticated-client) gas sub-buckets so a single caller cannot exhaust the global pool.
2. **Decouple rate limit and gas bucket capacity:** Ensure `requestsPerSecond × scaleGas(maxGasLimit) < gasPerSecond` (scaled). Currently they are exactly equal, leaving zero headroom.
3. **Pre-consume validation ordering:** Move `validateContractMaxGasLimit` *before* `throttleManager.throttle()` so invalid requests never consume gas tokens.
4. **Restore on gas-bucket throttle failure:** If `gasLimitBucket.tryConsume()` fails, the `rateLimitBucket` token has already been consumed but no gas was consumed — this is correct, but the inverse (gas consumed, then validation fails) should also restore promptly, which the current catch block only does for `InvalidParametersException`.

## Proof of Concept
```bash
# Send 500 requests/second with default gas (15,000,000) — no auth required
# Each request consumes 1,500 scaled tokens; 500/s × 1,500 = 750,000 tokens/s = full bucket
for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":15000000}' &
done
wait
# Legitimate concurrent users will receive:
# HTTP 429: {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```
Sustaining this loop keeps the `gasLimitBucket` perpetually at or near zero, blocking all legitimate `/call` and `/estimateGas` traffic for the duration of the attack.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
