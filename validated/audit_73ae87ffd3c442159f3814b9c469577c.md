# Audit Report

## Title
IP Spoofing via Unvalidated `X-Real-IP`/`X-Forwarded-For` Headers in `TracingMiddleware` Corrupts Audit Trail for `/construction/submit`

## Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` unconditionally accepts the client-supplied `X-Real-IP` and `X-Forwarded-For` headers without verifying the request originated from a trusted reverse proxy. Because `TracingMiddleware` wraps every request, any external client can forge the source IP recorded in every audit log entry, including those for `/construction/submit`.

## Finding Description
**Exact code path:**

`getClientIpAddress()` (lines 63–74, `rosetta/app/middleware/trace.go`) resolves the client IP in this priority order:

1. `X-Real-IP` header — taken verbatim from the request
2. `X-Forwarded-For` header — taken verbatim from the request
3. `r.RemoteAddr` — the actual TCP peer address (only used as last resort)

```go
// rosetta/app/middleware/trace.go lines 63-74
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)
    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)
    }
    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }
    return ipAddress
}
```

`TracingMiddleware` calls `getClientIpAddress()` and logs the result at `Info` level for every non-internal path:

```go
// rosetta/app/middleware/trace.go lines 52-59
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, ...)
if internalPaths[path] {
    log.Debug(message)
} else {
    log.Info(message)
}
```

In `main.go`, the middleware chain is assembled with no proxy-stripping layer between the internet and `TracingMiddleware`:

```go
// rosetta/main.go lines 217-221
metricsMiddleware := middleware.MetricsMiddleware(router)
tracingMiddleware := middleware.TracingMiddleware(metricsMiddleware)
corsMiddleware := server.CorsMiddleware(tracingMiddleware)
httpServer := &http.Server{
    Addr: fmt.Sprintf(":%d", rosettaConfig.Port), ...
```

**Root cause:** The function assumes `X-Real-IP`/`X-Forwarded-For` are only present when set by a trusted upstream proxy. There is no allowlist of trusted proxy CIDRs, no check that `RemoteAddr` belongs to a known proxy, and no stripping of these headers for direct connections.

**Failed assumption:** The code assumes these headers are only reachable after passing through a proxy that controls their values. In reality the server binds directly to a TCP port, so any client controls these headers.

## Impact Explanation
Every `/construction/submit` request is logged at `Info` level with the forged IP as the attributed source. Operators relying on these logs to investigate malicious transaction submissions will see a fabricated IP (e.g., a Tor exit node, a competitor's IP, or `127.0.0.1`) instead of the real attacker's address. This destroys the audit trail for the highest-risk endpoint in the Rosetta API — the one that broadcasts signed transactions to the Hedera network. No authentication is required by the Rosetta spec, so the logs are the primary forensic artifact available to operators.

## Likelihood Explanation
The precondition is only that the attacker can reach the Rosetta port, which is the default behavior with no authentication required. The exploit requires adding a single HTTP header, trivially repeatable with any HTTP client. No special privileges, credentials, or network position are needed. Every submission request can carry a different forged IP, making cross-request correlation impossible.

## Recommendation
Implement trusted-proxy validation before accepting forwarded headers:

1. **Allowlist trusted proxy CIDRs** in configuration (e.g., `trustedProxies: ["10.0.0.0/8"]`).
2. In `getClientIpAddress()`, check `r.RemoteAddr` against the allowlist **before** reading `X-Real-IP` or `X-Forwarded-For`. If `RemoteAddr` is not in the allowlist, ignore both headers and use `RemoteAddr` directly.
3. Alternatively, use a well-tested library such as [`realip`](https://github.com/tomasen/realip) or [`xff`](https://github.com/sebest/xff) that implements this logic correctly.
4. Document the expected deployment topology (mandatory reverse proxy) and add a startup warning if the server is bound to a public interface without proxy validation configured.

## Proof of Concept
```bash
# Forge source IP as 1.2.3.4 on a /construction/submit call
curl -s -X POST http://<rosetta-host>:<port>/construction/submit \
  -H "Content-Type: application/json" \
  -H "X-Real-IP: 1.2.3.4" \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"signed_transaction":"..."}'

# The resulting log line will read:
# time="..." level=info msg="1.2.3.4 POST /construction/submit (200) in 12ms"
# instead of the attacker's real IP.
```
