Audit Report

## Title
Global Rate Limit Token Consumed Without Rollback When Per-Filter THROTTLE Bucket Rejects

## Summary
In `ThrottleManagerImpl.throttle()`, the global `rateLimitBucket` token is consumed unconditionally before the per-filter `THROTTLE` bucket is evaluated. When the per-filter bucket is exhausted and rejects the request, the already-consumed global token is never returned. An attacker can exploit this to drain the global rate limit capacity for all other users, causing legitimate requests to be rejected even though the THROTTLE filter was intended to protect only a narrow request class.

## Finding Description

**Verified code path in `ThrottleManagerImpl.java`:**

In `throttle()`: [1](#0-0) 

The global `rateLimitBucket.tryConsume(1)` is called at line 38 and the gas bucket at line 40 — both **before** the per-filter loop at lines 44–48.

In `action()`: [2](#0-1) 

When `filter.getBucket().tryConsume(1)` fails at line 71, a `ThrottleException` is thrown at line 72. At this point the global token consumed at line 38 is permanently lost.

In `restore()`: [3](#0-2) 

`restore()` only calls `gasLimitBucket.addTokens(tokens)` — there is no `rateLimitBucket.addTokens(1)` call anywhere in the exception path. The `ThrottleManager` interface itself only exposes `restore(long gas)`, confirming no mechanism exists to return rate limit tokens. [4](#0-3) 

The `ContractCallService.restoreGasToBucket()` call path also only restores gas tokens, not rate limit tokens. [5](#0-4) 

**Per-filter bucket capacity:**

In `RequestProperties.createBucket()`, the per-filter THROTTLE bucket capacity equals `rate` (range 0–100, default 100), refilling at `rate` tokens/second — far below the global limit. [6](#0-5) 

**Root cause:** The ordering of checks in `throttle()` assumes that any request passing the global check will either succeed or fail in a way that does not invalidate the global token consumption. This assumption breaks for the `THROTTLE` action, where the per-filter bucket is a secondary, more restrictive gate that can reject requests that already passed the global gate.

## Impact Explanation

An unprivileged attacker can consume the entire global rate limit budget (e.g., 500 tokens/s by default) using requests that are all ultimately rejected by the per-filter THROTTLE bucket. Legitimate users receive `ThrottleException("Requests per second rate limit exceeded")` even though the THROTTLE filter was intended to protect only a narrow request class. The attacker receives no useful responses (all rejected), but fully starves the global rate limit for all other users. This is a targeted availability degradation against a public-facing JSON-RPC endpoint where availability is a core service guarantee.

## Likelihood Explanation

No privileges are required. The attacker only needs to identify a filter expression matching a configured THROTTLE rule. Filter fields (`DATA`, `TO`, `BLOCK`, `FROM`, `GAS`, `VALUE`) and match types (`EQUALS`, `CONTAINS`) are documented in `docs/configuration.md`. The attack is trivially repeatable with a simple HTTP client loop. The per-filter bucket refills at `rate`/s (max 100), so the attacker can re-exhaust it each second with minimal traffic while sustaining global token drain indefinitely.

## Recommendation

Add a rollback of the global `rateLimitBucket` token when a per-filter `THROTTLE` rejection occurs. The cleanest fix is to restructure `throttle()` to evaluate all per-filter checks **before** consuming the global token, or to catch the `ThrottleException` thrown from `action()` and call `rateLimitBucket.addTokens(1)` before re-throwing. The `restore()` method (or a new overload) should be extended to also accept a rate-limit token count to return. The same issue applies to the `REJECT` action case at line 69, which should be evaluated similarly.

## Proof of Concept

```
# Assume: THROTTLE filter matches requests with DATA prefix "0xdeadbeef", rate=10
# Global rateLimitBucket capacity: 500 tokens/s

# Step 1: Exhaust the per-filter bucket (10 requests)
for i in {1..10}; do
  curl -s -X POST http://<host>/api/v1/contracts/call \
    -d '{"data":"0xdeadbeef...","gas":50000}' &
done

# Step 2: Continue sending matching requests at global rate
# Each request: consumes 1 global token (line 38), consumes gas token (line 40),
# then is rejected at line 71-72 — global token permanently lost.
while true; do
  for i in {1..500}; do
    curl -s -X POST http://<host>/api/v1/contracts/call \
      -d '{"data":"0xdeadbeef...","gas":50000}' &
  done
  sleep 1
done

# Result: legitimate users receive HTTP 429 "Requests per second rate limit exceeded"
# even though their requests do not match the THROTTLE filter.
```

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
