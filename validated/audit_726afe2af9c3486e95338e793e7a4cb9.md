Audit Report

## Title
Integer Overflow in `poll()` Causes Unlimited Database Queries via Crafted `limit` Value

## Summary
A silent Java narrowing-cast overflow in `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()` converts a crafted `long` limit value into a negative `int`, which then bypasses the `hasLimit()` guard in `TopicMessageRepositoryCustomImpl.findByFilter()`. The result is a JPA query with no `LIMIT` clause that materialises every matching row in the topic on every poll cycle. A single unauthenticated gRPC subscription is sufficient to trigger this.

## Finding Description

**Input path — no upper-bound validation**

`ConsensusController.toFilter()` passes the protobuf `uint64 limit` field directly to `TopicMessageFilter.limit` (a Java `long`) with no upper-bound check: [1](#0-0) 

`TopicMessageFilter.limit` carries only `@Min(0)` — no `@Max` constraint — so any value ≥ 0 passes validation: [2](#0-1) 

**Integer overflow in `PollingTopicMessageRetriever.poll()`**

The remaining limit is computed with a narrowing cast from `long` to `int`: [3](#0-2) 

With `filter.getLimit() = 2147483648L` and `context.getTotal().get() = 0`:
- `2147483648L - 0L = 2147483648L`
- `(int) 2147483648L = -2147483648` (Integer.MIN_VALUE — silent narrowing overflow)
- `Math.min(-2147483648, maxPageSize) = -2147483648`

The overflowed `pageSize` is written into a new filter as a `long` (value remains `-2147483648L`): [4](#0-3) 

**Negative limit bypasses `hasLimit()` in `findByFilter()`**

`hasLimit()` is defined as `limit > 0`: [5](#0-4) 

`-2147483648 > 0` is `false`, so `setMaxResults()` is never called and the JPA query executes with no `LIMIT` clause: [6](#0-5) 

**`isComplete()` never terminates the loop**

The termination check compares `total.get()` against the *original* `filter.getLimit()` (`2147483648L`). `total` would need to reach 2 billion for `limitHit` to become `true`: [7](#0-6) 

In throttled mode the secondary check `pageSize.get() < retrieverProperties.getMaxPageSize()` is also `false` if the unlimited query returns more rows than `maxPageSize`, so polling continues indefinitely.

**Identical overflow in `PollingTopicListener.poll()`**

The same narrowing cast pattern exists in the listener: [8](#0-7) 

## Impact Explanation

Each poll cycle issues a full-table scan (scoped to the given topic) against PostgreSQL with no row limit. On a topic with millions of messages, the entire result set is materialised into JVM heap on every poll interval. A single malicious subscription can exhaust the database connection pool, JVM heap, and I/O bandwidth, causing denial of service for all other subscribers. The `flux.take(filter.getLimit())` guard in `TopicMessageServiceImpl` uses the original large limit (`2147483648L`) and does not prevent the unbounded database fetch that occurs inside each `poll()` call.

## Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication required. The trigger value (`2147483648`) is trivially constructable in any gRPC client by setting the `limit` field of `ConsensusTopicQuery` to `Integer.MAX_VALUE + 1`. The protobuf field is `uint64` and the value is well within its range. No special knowledge of internals is needed. The attack is repeatable and can be sustained with a single persistent subscription.

## Recommendation

1. **Clamp the limit before the cast.** In both `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`, replace the unchecked narrowing cast with a safe clamp:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Add an upper-bound constraint on `TopicMessageFilter.limit`.** Add `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) alongside the existing `@Min(0)` annotation so that oversized values are rejected at the validation layer before reaching `poll()`.
3. **Validate at the controller boundary.** In `ConsensusController.toFilter()`, explicitly reject or clamp `query.getLimit()` values that exceed `Integer.MAX_VALUE`.

## Proof of Concept

Using any standard gRPC client (e.g., `grpcurl` or a generated Java stub), send a `subscribeTopic` request with `limit = 2147483648`:

```protobuf
ConsensusTopicQuery {
  topicID: { topicNum: <any valid topic> }
  limit: 2147483648          // Integer.MAX_VALUE + 1; valid uint64
}
```

**Execution trace:**
1. `ConsensusController.toFilter()` stores `2147483648L` in `TopicMessageFilter.limit`. `@Min(0)` passes.
2. `PollingTopicMessageRetriever.poll()` computes `(int)(2147483648L - 0L)` → `-2147483648`.
3. `Math.min(-2147483648, maxPageSize)` → `-2147483648` is written as `pageSize`.
4. `newFilter.limit = -2147483648L`. `hasLimit()` returns `false`.
5. `findByFilter()` skips `setMaxResults()`. PostgreSQL returns all rows for the topic.
6. `isComplete()` never returns `true` (total can never reach `2147483648`). Polling repeats indefinitely at the configured interval.
