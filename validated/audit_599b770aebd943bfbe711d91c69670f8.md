Audit Report

## Title
Unauthenticated `getNodes()` gRPC Endpoint Enables Bounded Elastic Scheduler Exhaustion via Unbounded Connection Flooding

## Summary
`NetworkController.getNodes()` accepts unlimited unauthenticated gRPC connections. Each subscription drives blocking JDBC page fetches scheduled on Reactor's JVM-global `Schedulers.boundedElastic()` singleton, which is shared with all topic-message listeners and retrievers. An attacker opening enough concurrent connections can saturate that shared thread pool, starving topic-message processing across targeted mirror node replicas.

## Finding Description

**`NetworkController.getNodes()`** (lines 33–43) accepts any unauthenticated `AddressBookQuery` and immediately subscribes to `networkService::getNodes` with no throttle guard, no IP check, and no connection counter. [1](#0-0) 

**`NetworkServiceImpl.getNodes()`** (lines 68–76) builds a paged `Flux` that re-executes a blocking DB call (`transactionOperations.execute()`) on every page, with the inter-page delay and repeat subscription scheduled via `Schedulers.boundedElastic()`. When the delay fires, the next `page()` call runs on the same bounded-elastic thread, holding it for the duration of the synchronous JDBC query. [2](#0-1) [3](#0-2) 

**`Schedulers.boundedElastic()` is a JVM-global singleton.** The same pool is used by:
- `PollingTopicListener` (line 31, `scheduler` field; lines 39, 43 `withScheduler`) [4](#0-3) 
- `SharedPollingTopicListener` (line 37, `subscribeOn` and `withScheduler`) [5](#0-4) 
- `SharedTopicListener.publishOn` (line 25) [6](#0-5) 
- `PollingTopicMessageRetriever` (line 41, `withScheduler` line 55) [7](#0-6) 

**The only connection-level guard** is `maxConcurrentCallsPerConnection = 5`, which limits calls *per TCP connection* but places no ceiling on the number of connections or on total concurrent calls across all connections. [8](#0-7) [9](#0-8) 

**No rate-limiting exists in the grpc module.** `ThrottleConfiguration` / `ThrottleProperties` live exclusively in the `web3` module and are never applied to any gRPC endpoint. [10](#0-9) 

**Root cause:** The design assumes `maxConcurrentCallsPerConnection` is sufficient to bound total scheduler load, but it only limits calls per connection. An attacker opens arbitrarily many connections, each contributing up to 5 concurrent blocking page-fetch tasks to the shared bounded-elastic pool.

## Impact Explanation

Reactor's `boundedElastic` pool defaults to `10 × availableProcessors` threads (e.g., 40 threads on a 4-core pod). Each concurrent `getNodes()` page fetch holds one thread for the duration of a synchronous JDBC query inside `transactionOperations.execute()`. With N connections × 5 calls each, once N exceeds ~8 on a 4-core pod, the pool is saturated. All subsequent `boundedElastic` tasks — including the periodic polls in `PollingTopicListener`, `PollingTopicMessageRetriever`, and the `publishOn` dispatch in `SharedTopicListener` — queue behind the flood. Topic-message delivery to all subscribers on that replica halts. [2](#0-1) 

## Likelihood Explanation

No authentication, no API key, no IP allowlist, and no rate limit are required. A single attacker machine can open thousands of TCP connections to the gRPC port and issue 5 concurrent `getNodes()` calls on each. Because each call completes in a few seconds (small address book, `pageSize=10`, `pageDelay=250ms`), the attacker simply reconnects in a tight loop to maintain sustained pressure. This requires only a standard gRPC client library and a loop — no exploit tooling, no credentials, no brute force. [1](#0-0) 

## Recommendation

1. **Dedicated scheduler for `NetworkServiceImpl`:** Replace `Schedulers.boundedElastic()` in `NetworkServiceImpl.getNodes()` with a dedicated, capacity-bounded `Schedulers.newBoundedElastic(...)` instance so that address-book page fetches cannot starve topic-listener threads.
2. **Global connection limit:** Configure `NettyServerBuilder.maxConnectionAge` and a global `maxConnections` ceiling on the gRPC server in `GrpcConfiguration` to cap total concurrent connections.
3. **Per-IP connection rate limiting:** Add a gRPC `ServerInterceptor` that tracks and limits connections per source IP.
4. **Concurrency cap on `getNodes()`:** Use a `Semaphore` or Reactor's `flatMap(maxConcurrency)` inside `NetworkServiceImpl.getNodes()` to bound the total number of in-flight address-book subscriptions.
5. **Authentication/API key:** Require an API key or mTLS for `getNodes()` consistent with the authentication model applied elsewhere.

## Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import network_service_pb2_grpc, mirror_network_service_pb2

TARGET = "mirror-node-grpc:5600"
NUM_CONNECTIONS = 50   # 50 connections × 5 calls = 250 concurrent bounded-elastic threads
CALLS_PER_CONN = 5

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = network_service_pb2_grpc.NetworkServiceStub(channel)
    threads = []
    for _ in range(CALLS_PER_CONN):
        # Each call triggers paged DB fetches on Schedulers.boundedElastic()
        t = threading.Thread(
            target=lambda: list(stub.getNodes(mirror_network_service_pb2.AddressBookQuery()))
        )
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

while True:  # sustain pressure
    workers = [threading.Thread(target=flood, args=(i,)) for i in range(NUM_CONNECTIONS)]
    for w in workers: w.start()
    for w in workers: w.join()
    # With 50 connections × 5 calls = 250 concurrent tasks on a pool of ~40 threads,
    # PollingTopicListener and SharedTopicListener tasks queue indefinitely.
```
