Audit Report

## Title
Integer Narrowing Cast Overflow in `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()` Enables Unbounded Database Queries

## Summary
When a gRPC client submits a `ConsensusTopicQuery` with `limit = Long.MAX_VALUE`, an unsafe narrowing cast `(int)(filter.getLimit() - context.getCount().get())` produces `-1`. This causes `findByFilter()` to execute SQL queries with no `LIMIT` clause on every poll interval, enabling a denial-of-service via unbounded database reads.

## Finding Description

**Primary location:** `PollingTopicListener.java`, `poll()`, lines 54–57.

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())  // narrowing cast
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
``` [1](#0-0) 

**Root cause:** `Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF`. Truncating to 32 bits yields `0xFFFFFFFF = -1` (signed). So `limit = -1` and `Math.min(-1, maxPageSize) = -1`.

**Propagation:** The value `-1` is set on a rebuilt filter at line 59 and passed to `findByFilter()`: [2](#0-1) 

Inside `TopicMessageRepositoryCustomImpl.findByFilter()`, the guard `filter.hasLimit()` calls `limit > 0`. With `limit = -1`, this returns `false`, so `typedQuery.setMaxResults()` is never invoked and the JPA query executes with no SQL `LIMIT` clause: [3](#0-2) 

`hasLimit()` is defined as: [4](#0-3) 

**Why existing checks fail:**
- `@Min(0)` on `TopicMessageFilter.limit` only rejects negative values; `Long.MAX_VALUE` is positive and passes. [5](#0-4) 
- `@Validated` on `TopicMessageFilter` is a Spring AOP marker. Bean Validation is not triggered at Lombok builder `.build()` time — only when the object is passed to a `@Validated` Spring-managed method with `@Valid` on the parameter, which does not occur here. [6](#0-5) 
- `flux.take(filter.getLimit())` in `TopicMessageServiceImpl.subscribeTopic()` caps total items emitted to the subscriber at `Long.MAX_VALUE` (effectively unlimited), but does not prevent per-poll unbounded DB reads. [7](#0-6) 

**Secondary location:** The identical overflow pattern exists in `PollingTopicMessageRetriever.poll()` lines 68–71, which is invoked for all subscriber types: [8](#0-7) 

## Impact Explanation
On every poll interval (default 500 ms), the polling loop issues a SQL query against the `topic_message` table with no row limit. For a high-volume topic, this returns millions of rows per query, consuming database CPU, I/O, network bandwidth, and JVM heap. A single malicious subscription can saturate the database connection pool and degrade or crash the mirror node for all other users.

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. Any client can set `limit` to `Long.MAX_VALUE` in a `ConsensusTopicQuery` message using any standard gRPC client. The protobuf field is `uint64`, so `Long.MAX_VALUE` is a valid wire value. No special privileges or internal knowledge are needed. The attack is trivially repeatable and can be parallelized across many connections.

## Recommendation
Replace the unsafe narrowing cast with a safe clamped conversion. In both `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()`, replace:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
```

with:

```java
long remaining = filter.hasLimit()
        ? filter.getLimit() - context.getCount().get()
        : Integer.MAX_VALUE;
int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
```

Additionally, add an upstream cap in the gRPC request-to-filter conversion to reject or clamp `limit` values exceeding a reasonable maximum (e.g., `Integer.MAX_VALUE`) before the filter is constructed.

## Proof of Concept
```
# Using grpcurl against a running mirror node
grpcurl -plaintext -d '{
  "topicID": {"topicNum": 1},
  "limit": 9223372036854775807
}' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

With `limit = Long.MAX_VALUE = 9223372036854775807`:
1. `filter.hasLimit()` → `true` (since `Long.MAX_VALUE > 0`)
2. `(int)(Long.MAX_VALUE - 0)` → `(int)(0x7FFFFFFFFFFFFFFF)` → `0xFFFFFFFF` → `-1`
3. `Math.min(-1, maxPageSize)` → `-1`
4. `newFilter.limit = -1`, `newFilter.hasLimit()` → `false`
5. `setMaxResults()` never called → unbounded SQL query fires every 500 ms
