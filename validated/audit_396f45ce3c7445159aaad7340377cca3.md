Audit Report

## Title
Missing `deleted is not true` Filter in `entity_history` Branch of `findActiveByEvmAddressAndTimestamp` and `findActiveByEvmAddressOrAliasAndTimestamp`

## Summary
`findActiveByEvmAddressAndTimestamp()` and `findActiveByEvmAddressOrAliasAndTimestamp()` in `EntityRepository.java` apply `deleted is not true` only to the `entity` branch of their `UNION ALL` queries, but omit this filter from the `entity_history` branch. When a historical `eth_call` is made at a `blockTimestamp` at or after an entity's deletion, the `entity` branch is correctly suppressed, but the `entity_history` branch returns the most recent pre-deletion snapshot (which has `deleted = false`), causing the deleted entity to be surfaced as active.

## Finding Description

**File:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`

**Affected methods:**
- `findActiveByEvmAddressAndTimestamp` (lines 59–85)
- `findActiveByEvmAddressOrAliasAndTimestamp` (lines 95–121)

In both methods, the `entity_history` branch has no `deleted` predicate:

```sql
-- entity branch: correctly filtered
where e.deleted is not true
and e.id = (select id from entity_cte)

-- entity_history branch: NO deleted filter
where lower(eh.timestamp_range) <= ?2
and eh.id = (select id from entity_cte)
order by lower(eh.timestamp_range) desc
limit 1
``` [5](#0-4) 

Contrast with `findActiveByIdAndTimestamp` (lines 136–155), which correctly applies `and deleted is not true` to both branches: [6](#0-5) 

**Data model mechanics:** When an entity is deleted at timestamp `T_del`:
- `entity` table: `deleted = true`, `timestamp_range = [T_del, null)`
- `entity_history` table: pre-deletion snapshot with `deleted = false`, `timestamp_range = [T_created, T_del)`

**Exploit flow for `blockTimestamp = T_query` where `T_query >= T_del`:**
1. `entity_cte` resolves the entity ID from `entity` (no deletion filter in the CTE).
2. `entity` branch: `deleted is not true` fails → zero rows.
3. `entity_history` branch: `lower([T_created, T_del)) = T_created <= T_query` → the pre-deletion snapshot (`deleted = false`) is returned with no filter blocking it.
4. Outer `ORDER BY timestamp_range DESC LIMIT 1` picks this `entity_history` row.
5. The function returns the deleted entity as if it were active.

**Why existing tests do not catch this:** The test `findHistoricalEntityByEvmAddressAndTimestampRangeAndDeletedTrueCall` (line 130) passes `entityHistory.getTimestampLower() - 1` as `blockTimestamp`. Since the `entity_history` branch filters `lower(timestamp_range) <= blockTimestamp`, using a value *before* the lower bound means the branch returns nothing regardless of the missing `deleted` filter. The case `blockTimestamp >= entityHistory.getTimestampLower()` with a pre-deletion (`deleted = false`) history record has no test coverage. [4](#0-3) 

## Impact Explanation

Any caller of `findActiveByEvmAddressAndTimestamp` — primarily `CommonEntityAccessor.getEntityByEvmAddressTimestamp()` (lines 80–83) and `getEntityByEvmAddressAndTimestamp()` (lines 66–71) — will receive a non-empty `Optional<Entity>` for a deleted entity during historical queries. [7](#0-6) 

This propagates into the EVM execution layer for historical `eth_call` requests:
- A deleted smart contract appears to have code and is treated as callable, producing incorrect execution results.
- A deleted account appears to have its last-known balance and key material, corrupting historical balance/state queries.
- All mirror node consumers relying on these historical responses receive incorrect data, breaking the integrity guarantee of the mirror node's historical state export.

## Likelihood Explanation

**Preconditions (all achievable by an unprivileged external user):**
1. Know the EVM address of any previously deleted contract or account (publicly observable on-chain).
2. Know any block number whose consensus timestamp is at or after the deletion timestamp (also publicly observable).

**Trigger:** Submit an `eth_call` JSON-RPC request with `blockNumber` set to any block at or after the entity's deletion. No authentication, no special keys, no privileged access required.

**Repeatability:** Fully deterministic and repeatable for any deleted entity.

## Recommendation

Add `and eh.deleted is not true` to the `entity_history` branch in both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- entity_history branch fix for findActiveByEvmAddressAndTimestamp
select *
from entity_history eh
where lower(eh.timestamp_range) <= ?2
and eh.id = (select id from entity_cte)
and eh.deleted is not true          -- ADD THIS LINE
order by lower(eh.timestamp_range) desc
limit 1
```

Apply the same fix to `findActiveByEvmAddressOrAliasAndTimestamp`. Additionally, add test cases that use `blockTimestamp >= entityHistory.getTimestampLower()` with a pre-deletion (`deleted = false`) history record to cover this scenario.

## Proof of Concept

**Setup:**
1. Create an entity with `evm_address = 0xABCD`, `created_timestamp = 100`, `deleted = false`.
2. Delete the entity at `T_del = 200`. This results in:
   - `entity` table: `evm_address = 0xABCD`, `deleted = true`, `timestamp_range = [200, null)`
   - `entity_history` table: `evm_address = 0xABCD`, `deleted = false`, `timestamp_range = [100, 200)`

**Query:** Call `findActiveByEvmAddressAndTimestamp(0xABCD, 250)` (blockTimestamp = 250 >= T_del = 200).

**Expected result:** `Optional.empty()` (entity is deleted).

**Actual result:** The `entity_history` branch returns the pre-deletion snapshot (`deleted = false`, `timestamp_range = [100, 200)`) because `lower([100, 200)) = 100 <= 250` passes and there is no `deleted is not true` guard. The outer `LIMIT 1` returns this row, yielding a non-empty `Optional<Entity>` for the deleted entity.
