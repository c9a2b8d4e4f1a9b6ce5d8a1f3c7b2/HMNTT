Audit Report

## Title
Unauthenticated EVM Address Parameter Triggers Mandatory DB Lookup That Fails Independently, Causing Full Request Failure

## Summary
Any unauthenticated caller can supply `account.id=<evm_address>` to `GET /api/v1/balances`. This causes `parseAccountIdQueryParam()` to inject a live `Promise` (the entity-lookup DB call) into the params array. That promise is resolved via `Promise.all()` before the main balance query runs. If the entity-lookup query throws a `DbError` (statement timeout, connection exhaustion, transient network error), the entire request is aborted with HTTP 503 — even though the main balance query was never attempted and would have succeeded. The `requireResult = false` flag passed at the call site suppresses only `NotFoundError`; it has no effect on `DbError`.

## Finding Description

**Step 1 — EVM address triggers a live DB Promise**

In `rest/balances.js`, `parseAccountIdQueryParam()` returns `EntityService.getEncodedId(value, false)` — a raw, unresolved `Promise` — as a value in the params array when the supplied `account.id` value is a valid EVM address: [1](#0-0) 

**Step 2 — `getBalances()` awaits that Promise before the main query**

`getBalances()` resolves all params (including the entity-lookup Promise) via `Promise.all()` at line 88, before the main balance SQL is ever constructed or executed: [2](#0-1) 

The main balance query is not issued until line 153: [3](#0-2) 

**Step 3 — `requireResult = false` does not suppress `DbError`**

Inside `getEncodedId()`, the `requireResult` flag is only consulted inside `getEntityIdFromEvmAddress()` to decide whether to throw `NotFoundError` when zero rows are returned. It has no bearing on errors thrown by the underlying DB call. The `catch` block in `getEncodedId()` re-maps `InvalidArgumentError` but unconditionally rethrows everything else, including `DbError`: [4](#0-3) 

The `requireResult` guard in `getEntityIdFromEvmAddress()` only fires on the empty-rows path: [5](#0-4) 

**Step 4 — `getRows()` delegates to `pool.queryQuietly()`, which wraps all DB errors as `DbError`** [6](#0-5) 

**Step 5 — `DbError` maps to HTTP 503** [7](#0-6) 

## Impact Explanation
The `/api/v1/balances` endpoint becomes selectively unavailable for any request that includes an EVM address as `account.id`. Requests using numeric account IDs bypass the entity-lookup entirely and are unaffected. During any period of DB stress (statement timeouts, connection pool exhaustion, transient network errors), an unauthenticated attacker can guarantee HTTP 503 responses for EVM-address-filtered balance queries with a single GET request — no credentials, no rate-limit bypass, no special tooling required.

## Likelihood Explanation
The precondition (DB under stress) is a realistic operational condition, not an exotic scenario. The EVM address format is documented in the public API. The trigger is a single unauthenticated HTTP GET. The attack is trivially repeatable and can be automated to amplify the impact of any DB degradation event.

## Recommendation
The `requireResult = false` contract should be extended to cover DB-level failures, not just "not found" outcomes. Concretely:

1. In `getEntityIdFromEvmAddress()` (and `getAccountIdFromAlias()`), wrap the `getRows()` call in a try/catch; when `requireResult` is `false`, catch `DbError` and return `null` instead of rethrowing.
2. Alternatively, in `getBalances()`, wrap the `Promise.all(accountParamsPromise)` call so that a rejected entity-lookup promise resolves to `null` (treating a DB failure as "entity not found") rather than aborting the entire request.

Either fix ensures that a transient entity-lookup failure degrades gracefully (omitting the EVM address filter) rather than failing the whole request.

## Proof of Concept
```
GET /api/v1/balances?account.id=0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef
```
During any period where the DB statement timeout fires on the `entity` table lookup (e.g., `statement_timeout = 500ms` and the entity table is under load), this single unauthenticated request returns:
```json
HTTP 503 Service Unavailable
{"_status":{"messages":[{"message":"..."}]}}
```
while a concurrent request to:
```
GET /api/v1/balances?account.id=0.0.1234
```
succeeds normally, confirming the selective failure.
