Audit Report

## Title
Missing `COALESCE` in `entityFromEvmAddressQuery` Causes `NotFoundError` for Entities with `deleted = NULL`

## Summary
`EntityService.entityFromEvmAddressQuery` filters with `deleted <> true` instead of `coalesce(deleted, false) <> true`. In SQL, `NULL <> true` evaluates to `NULL` (falsy), so any entity whose `deleted` column is `NULL` — the standard state for a never-deleted entity — is silently excluded from query results. This causes `getEntityIdFromEvmAddress()` to receive zero rows and unconditionally throw `NotFoundError`, making every such entity permanently unreachable via its EVM address.

## Finding Description

**Vulnerable query** (`entityFromEvmAddressQuery`, lines 22–25):
```sql
where deleted <> true        -- NULL <> true → NULL (falsy) → row excluded
  and evm_address = $1
``` [1](#0-0) 

**Correct reference query** (`entityFromAliasQuery`, lines 17–20):
```sql
where coalesce(deleted, false) <> true   -- NULL → false → row included
  and alias = $1
``` [2](#0-1) 

The presence of `COALESCE` in the alias query and its absence in the EVM address query is a clear inconsistency. The developers demonstrably knew `NULL` was a valid `deleted` state.

**Call chain:**

`getEncodedId()` receives a user-supplied EVM address string, parses it into an `EntityId` with a non-null `evmAddress`, and routes to `getEntityIdFromEvmAddress()`: [3](#0-2) 

`getEntityIdFromEvmAddress()` executes the broken query. When `rows.length === 0` and `requireResult` is `true` (the default), `NotFoundError` is thrown unconditionally: [4](#0-3) 

## Impact Explanation
Any entity with a non-null `evm_address` and `deleted = NULL` (the default, never-deleted state) is permanently inaccessible through every API endpoint that calls `EntityService.getEncodedId()` with an EVM address. This includes NFT queries, staking reward queries, allowance queries, and token queries. Because `deleted = NULL` is the normal state for the vast majority of live entities, this affects most EVM-addressed entities in a production deployment.

## Likelihood Explanation
No privileges are required. EVM addresses are public on-chain data. Any user who supplies a valid EVM address for a live entity will deterministically receive a 404 response. The triggering condition (`deleted = NULL`) is the default state, not a rare edge case, making this trivially and repeatedly reproducible against real production data.

## Recommendation
Apply `COALESCE` to the `entityFromEvmAddressQuery` to match the pattern already used in `entityFromAliasQuery`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

## Proof of Concept
1. Insert an entity row with `evm_address = '\xABCD...'` and `deleted = NULL` (default).
2. Send `GET /api/v1/accounts/0xABCD.../nfts` (or any endpoint calling `getEncodedId`).
3. `getEncodedId` → `getEntityIdFromEvmAddress` → SQL returns 0 rows (because `NULL <> true` is `NULL`) → `NotFoundError` thrown → API returns HTTP 404.
4. The same request against the alias path (`GET /api/v1/accounts/<alias>/nfts`) succeeds for the same entity, confirming the EVM address path is uniquely broken.
