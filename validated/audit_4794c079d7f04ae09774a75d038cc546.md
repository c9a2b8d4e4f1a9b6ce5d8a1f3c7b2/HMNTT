# Audit Report

## Title
Cache Poisoning via `Arrays.hashCode()` Collision in `CACHE_NAME_ALIAS` and `CACHE_NAME_EVM_ADDRESS` Cache Keys

## Summary
`findByEvmAddressOrAliasAndDeletedIsFalse()` and `findByEvmAddressAndDeletedIsFalse()` use `@spelHelper.hashCode(#alias)` — which delegates to `Arrays.hashCode(byte[])`, a 32-bit non-cryptographic hash — as the sole cache key for the `CACHE_NAME_ALIAS` and `CACHE_NAME_EVM_ADDRESS` Caffeine caches respectively. Two distinct byte arrays that produce the same `Arrays.hashCode` value share a single cache slot with no secondary equality check. An unprivileged attacker can analytically construct such a collision in O(1), causing a lookup for alias B to return the cached entity for alias A, misrepresenting the current ledger entity state.

## Finding Description

**Exact code path:**

`SpelHelper.hashCode(byte[])` delegates directly to `Arrays.hashCode(value)`: [1](#0-0) 

`findByEvmAddressOrAliasAndDeletedIsFalse()` uses the 32-bit integer result as the only cache key: [2](#0-1) 

`findByEvmAddressAndDeletedIsFalse()` has the identical pattern: [3](#0-2) 

Both caches are Caffeine caches under `CACHE_MANAGER_ENTITY` with no custom key comparator: [4](#0-3) 

**Root cause:** Spring's `@Cacheable` stores and retrieves entries using the SpEL key expression result verbatim — here an `int`. The original `byte[]` is never stored as part of the cache key. When two distinct byte arrays `aliasA` and `aliasB` satisfy `Arrays.hashCode(aliasA) == Arrays.hashCode(aliasB)`, the second lookup hits the first entry with no further comparison.

**Collision construction (analytical, zero brute-force):**
`Arrays.hashCode` computes `h = 31*h + (b & 0xff)` per byte. For any `aliasA` of length ≥ 2, construct `aliasB` identical to `aliasA` except:
```
aliasB[n-2] = (aliasA[n-2] + 1) & 0xFF
aliasB[n-1] = (aliasA[n-1] - 31 + 256) & 0xFF
```
Net hash delta: `31² × (+1) + 31 × (−31) = 961 − 961 = 0`. Guaranteed collision, single arithmetic operation.

**Why the existing guard fails:**
The `unless = "#result == null"` condition only prevents caching absent results; it does not prevent a collision from serving a wrong cached result to a subsequent caller. [5](#0-4) 

## Impact Explanation
A caller querying entity state by alias receives a completely different entity's data — balance, contract bytecode, key material, deleted status. In the Hashgraph context this directly misrepresents the current ledger state: smart contract calls, balance checks, and account resolution can all be fed wrong entity data. The flaw exists in both `CACHE_NAME_ALIAS` (via `findByEvmAddressOrAliasAndDeletedIsFalse`) and `CACHE_NAME_EVM_ADDRESS` (via `findByEvmAddressAndDeletedIsFalse`).

## Likelihood Explanation
No authentication or special privilege is required — only the ability to send HTTP requests to the web3 JSON-RPC endpoint. The collision is computed analytically in O(1) from any known alias; no brute force is needed. The attack is repeatable for the duration of the cache TTL and can be re-triggered after expiry. Any public-facing mirror-node deployment is exposed.

## Recommendation
Replace the 32-bit `Arrays.hashCode()` cache key with the full byte array wrapped in a type that implements correct `equals`/`hashCode` semantics, such as `java.nio.ByteBuffer` or a dedicated wrapper. For example:

```java
// In SpelHelper.java
public ByteBuffer cacheKey(byte[] value) {
    return ByteBuffer.wrap(value);
}
```

Then update the `@Cacheable` key expressions:
```java
key = "@spelHelper.cacheKey(#alias)"
```

`ByteBuffer.equals()` performs element-wise comparison of the wrapped bytes, so two distinct byte arrays with the same `Arrays.hashCode` value will correctly map to different cache entries.

## Proof of Concept

```java
// aliasA: any registered alias, e.g. {0x01, 0x02, 0x03}
byte[] aliasA = new byte[]{0x01, 0x02, 0x03};

// Construct aliasB with same Arrays.hashCode as aliasA
byte[] aliasB = aliasA.clone();
aliasB[aliasB.length - 2] = (byte)((aliasA[aliasA.length - 2] + 1) & 0xFF);
aliasB[aliasB.length - 1] = (byte)((aliasA[aliasA.length - 1] - 31 + 256) & 0xFF);

assert Arrays.hashCode(aliasA) == Arrays.hashCode(aliasB); // always true

// Step 1: Request with aliasA → DB queried, entity A cached under key k
entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(aliasA);

// Step 2: Request with aliasB → cache hit on key k → entity A returned instead of entity B
Optional<Entity> result = entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(aliasB);
// result contains entity A's data, not entity B's
```
