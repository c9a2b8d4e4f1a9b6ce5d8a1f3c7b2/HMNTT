## Audit Report

## Title
Unauthenticated Full-History Scan via Timestamp Parameter Triggers Expensive `DISTINCT ON` Query in Hook Storage API

## Summary
Any unauthenticated caller can supply `timestamp=gte:0.000000000` to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`. This unconditionally routes execution to `getHookStorageChange()`, which issues a `SELECT DISTINCT ON (key)` query spanning the entire `hook_storage_change` table for the targeted `(owner_id, hook_id)` pair. Because no rate limiting exists in the `rest-java` module, an attacker can repeatedly trigger this expensive code path to exhaust database CPU and I/O.

## Finding Description

**Verified code path:**

**Step 1 — Unauthenticated controller entry.**
`HooksController.getHookStorage()` accepts `timestamps` with only a `@Size(max = 2)` constraint. No authentication annotation, no minimum-value check. [1](#0-0) 

**Step 2 — Raw timestamp passed to `Bound.of()` with no floor validation.**
`hookStorageChangeRequest()` passes the raw `timestamps` array directly to `Bound.of()`. No check that the timestamp value is above any minimum. [2](#0-1) 

**Step 3 — `isHistorical()` gate.**
`isHistorical()` returns `true` for any non-empty `Bound`. `Bound.isEmpty()` returns `false` whenever `lower != null`. With `gte:0`, `lower` is set to a `GTE(0)` parameter, so `isHistorical()` returns `true` and execution is routed to `getHookStorageChange()`. [3](#0-2) [4](#0-3) 

**Step 4 — Timestamp bounds span entire history.**
`getAdjustedLowerRangeValue()` returns `0` when the lower value is `0` with `GTE`. `adjustUpperBound()` returns `Long.MAX_VALUE` when no upper bound is set. The resulting range is `[0, Long.MAX_VALUE]`, covering all partitions. [5](#0-4) [6](#0-5) 

**Step 5 — Expensive `DISTINCT ON` query.**
`findByKeyBetweenAndTimestampBetween()` issues a `SELECT DISTINCT ON (key)` with no `ORDER BY` in the SQL body. The sort (`key ASC, consensus_timestamp DESC`) is injected via `Pageable` by Spring Data JPA. PostgreSQL must scan **all** rows matching `(owner_id, hook_id, key BETWEEN min AND max, consensus_timestamp BETWEEN 0 AND MAX_LONG)` before deduplicating and applying `LIMIT`. The primary key `(owner_id, hook_id, key, consensus_timestamp)` stores `consensus_timestamp` ascending; the required descending order within each key group prevents an efficient index-only skip scan. [7](#0-6) [8](#0-7) 

**Step 6 — No rate limiting in `rest-java`.**
`ThrottleConfiguration` (bucket4j-based rate limiter) exists exclusively in the `web3` module. The `rest-java` module has no equivalent throttle for this endpoint. [9](#0-8) 

**Why existing checks fail:**
- `@Size(max = 2)` limits the *count* of timestamp parameters, not their *values*.
- `@Max(MAX_LIMIT)` on `limit` caps output rows at 100 but does not reduce the number of rows PostgreSQL must scan before deduplication.
- There is no `@Min` or custom validator on `TimestampParameter` values.

## Impact Explanation
A single request with `timestamp=gte:0.000000000` forces PostgreSQL to scan every row in `hook_storage_change` for the targeted `(owner_id, hook_id)` pair across all timestamp partitions, deduplicate by key, and only then apply `LIMIT 100`. The `hook_storage_change` table schema confirms it is partitioned by `consensus_timestamp` with primary key `(owner_id, hook_id, key, consensus_timestamp)`. [10](#0-9) 

For a hook with a large write history, this scan is orders of magnitude more expensive than the non-historical path, which uses a simple primary-key index scan on `hook_storage` (`(owner_id, hook_id, key)`). [11](#0-10) 

Repeated unauthenticated requests can saturate database CPU and I/O, degrading availability for all users of the mirror node.

## Likelihood Explanation
The exploit requires zero privileges — no API key, no authentication, no special account. The endpoint is publicly documented in the OpenAPI spec and the `timestamp` parameter explicitly accepts `gte:` operators. [12](#0-11) 

The attack is trivially repeatable with a simple HTTP client loop. Any attacker who can reach the mirror node's REST API can execute this.

## Recommendation

1. **Add a minimum timestamp floor.** Reject or clamp `timestamp` values below a configurable minimum (e.g., current time minus a maximum lookback window) in `hookStorageChangeRequest()` or in `Bound.of()`.

2. **Add rate limiting to `rest-java`.** Apply a per-IP or global request-rate limiter (analogous to `ThrottleConfiguration` in `web3`) to the `rest-java` module, specifically for endpoints that trigger historical queries. [13](#0-12) 

3. **Add a query timeout.** Configure a statement timeout on the database connection used by `rest-java` to bound the maximum wall-clock time any single query can consume.

4. **Optimize the query.** Consider rewriting the `DISTINCT ON` query to use a lateral join or a recursive CTE that leverages the `(owner_id, hook_id, key, consensus_timestamp)` index for a skip-scan pattern, allowing early exit once `LIMIT` rows are found.

## Proof of Concept

```bash
# Trigger a full-history scan for hook 1 owned by account 0.0.123
# Repeat in a loop to amplify database load

while true; do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.123/hooks/1/storage?timestamp=gte:0.000000000&limit=100" \
    -o /dev/null &
done
```

The request requires no authentication header. The `timestamp=gte:0.000000000` value passes `@Size(max=2)` (only one parameter), passes `TimestampParameter` parsing (valid format), and results in `Bound.getAdjustedLowerRangeValue() == 0` and `Bound.adjustUpperBound() == Long.MAX_VALUE`, causing `findByKeyBetweenAndTimestampBetween()` to scan the entire `hook_storage_change` table for the targeted hook. [14](#0-13)

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
