Audit Report

## Title
Malicious Block Node Can Cause Indefinite Block Ingestion Latency via Slow-Drip gRPC Streaming

## Summary
`BlockAssembler.timeout()` resets its `Stopwatch` after every completed block, granting a fresh full timeout budget per block rather than per connection. A malicious block node can exploit this to hold the importer's synchronous `streamBlocks()` call blocked indefinitely by completing each block just before the timeout expires and immediately starting the next.

## Finding Description

**Root cause — per-block stopwatch reset:**

`onEndOfBlock()` calls `stopwatch.reset()` at line 268, which stops and zeroes the stopwatch: [1](#0-0) 

The next call to `timeout()` sees `!stopwatch.isRunning()` as `true`, restarts the stopwatch, and returns the **full** `timeout.toMillis()` again: [2](#0-1) 

This value is consumed as the per-read deadline in the streaming loop: [3](#0-2) 

There is no wall-clock deadline applied to the entire `streamBlocks()` call — only a per-read deadline that resets with every new block.

**Why existing checks are insufficient:**

- `onError()` / `maxSubscribeAttempts` / `readmitDelay` — only triggered by exceptions in the catch blocks, never by a slow-but-non-erroring stream: [4](#0-3) 

- `maxBlockItems` — limits item count per block, not delivery speed: [5](#0-4) 

- `maxStreamResponseSize` — limits individual inbound message size, not delivery rate: [6](#0-5) 

- `responseTimeout` (400 ms default) — only applied to the `serverStatus` RPC in `getBlockRange()`, **not** to `streamBlocks()`: [7](#0-6) [8](#0-7) 

**Synchronous blocking:** `streamBlocks()` is called synchronously from `BlockNodeSubscriber.doGet()`, which is called from `AbstractBlockSource.get()`. No parallelism exists; while one node's `streamBlocks()` is executing, no other node can be polled: [9](#0-8) [10](#0-9) 

## Impact Explanation
A malicious block node holds the importer's single streaming thread indefinitely, one block at a time, each block consuming up to the full configured timeout. Because `streamBlocks()` is synchronous and the importer selects one node per block, legitimate block nodes cannot be used while the malicious node is active. Block ingestion latency spikes proportionally, causing the mirror node to fall behind the chain tip and degrading data freshness for all downstream consumers. Severity is **medium** (availability degradation / griefing; no direct fund loss).

## Likelihood Explanation
The attacker must control a block node reachable by the importer — either via static configuration or via `BlockNodeDiscoveryService` auto-discovery. No authentication of block nodes is visible in the reviewed code path. An operator of any node the importer is pointed at can execute this attack without any privileged access to the importer itself. The attack is repeatable and requires only network-level participation.

## Recommendation
Apply a **wall-clock deadline to the entire `streamBlocks()` call**, not just to individual reads. One approach: record `Instant connectionStart` before the streaming loop and check `Duration.between(connectionStart, Instant.now())` against a configurable `maxStreamDuration` at each iteration. If exceeded, cancel the gRPC call and invoke `onError()` so the node is penalized via the existing `maxSubscribeAttempts` / `readmitDelay` mechanism. The per-read timeout can remain for detecting stalled individual reads, but it must not be the only guard.

## Proof of Concept

1. Configure a malicious block node reachable by the importer.
2. On `SUBSCRIBE_BLOCK_STREAM` request, the malicious node sends `BLOCK_HEADER` for block N — `stopwatch.start()` is called, full timeout T is returned.
3. The node sends subsequent `BlockItemSet` messages at a rate that keeps `timeout.toMillis() - stopwatch.elapsed()` just above zero (e.g., one item every `T - ε` ms).
4. Just before the timeout expires, the node sends `END_OF_BLOCK` for block N → `stopwatch.reset()` is called at line 268.
5. The node immediately sends `BLOCK_HEADER` for block N+1 → `timeout()` returns full T again.
6. Steps 3–5 repeat indefinitely. The `streamBlocks()` call never returns, the importer thread is held, and no other block node can be polled. [11](#0-10) [2](#0-1)

            final var blockHeader = pending.getFirst().getFirst().getBlockHeader();

            final List<BlockItem> block;

                block = pending.getFirst();

                // assemble when there are more than one BlockItemSet
                block = new ArrayList<>();
                for (final var items : pending) {
                    block.addAll(items);
                }
            }

            pending.clear();
            pendingCount = 0;
            stopwatch.reset();
