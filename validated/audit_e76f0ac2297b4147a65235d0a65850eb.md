Audit Report

## Title
Unsanitized `X-Real-IP` / `X-Forwarded-For` Headers Allow Audit Log Injection in `getClientIpAddress()`

## Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` accepts the `X-Real-IP` and `X-Forwarded-For` HTTP headers verbatim with no IP format validation. The raw value is written directly into the mirror-node trace log, allowing any reachable client to inject arbitrary strings — including fake IPs, hostnames, or crafted log lines — as the authoritative client address in audit records.

## Finding Description
**File:** `rosetta/app/middleware/trace.go`
**Function:** `getClientIpAddress()`, lines 63–74

```go
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64 – raw header, no validation

    if len(ipAddress) == 0 {                           // line 66 – only emptiness check
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67 – same issue
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // line 71 – only fallback uses real addr
    }

    return ipAddress                                   // line 74 – arbitrary string returned
}
```

The returned value is immediately interpolated into the log message at lines 52–53:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, tracingResponseWriter.statusCode, time.Since(start))
```

**Root cause:** The sole guard is `len(ipAddress) == 0`. There is no call to `net.ParseIP()`, no regex check, and no allowlist. Whatever the client supplies in `X-Real-IP` is returned and logged without sanitization.

**Failed assumption:** The code assumes `X-Real-IP` is always set by a trusted reverse proxy that overwrites the header with `$remote_addr`. In practice, many proxy configurations append or pass through client-supplied headers rather than replacing them, and the service may be directly reachable in dev/staging environments.

## Impact Explanation
- **Audit trail corruption:** IP-based forensics, rate-limiting analysis, and incident response become unreliable because the logged "client IP" is fully attacker-controlled.
- **Log injection / log forgery:** A crafted value such as `1.2.3.4 POST /admin 200 in 0s\n` can inject fake log lines, potentially hiding real attacker activity or framing legitimate users.
- **Spoofed IP attribution:** An attacker can impersonate internal IPs (e.g., `127.0.0.1`, `10.0.0.1`) to make malicious requests appear to originate from trusted infrastructure.

## Likelihood Explanation
- No authentication is required; any HTTP client can set arbitrary headers.
- Directly exploitable whenever the Rosetta service is exposed without a proxy that unconditionally overwrites `X-Real-IP`.
- Even behind a proxy, many default configurations pass through or append client-supplied headers rather than replacing them.
- Fully repeatable and automatable; requires zero privileges.

## Recommendation
Validate the extracted IP address using `net.ParseIP()` before trusting it. If the parsed result is `nil`, fall back to `r.RemoteAddr`. Example:

```go
func getClientIpAddress(r *http.Request) string {
    for _, header := range []string{xRealIpHeader, xForwardedForHeader} {
        raw := r.Header.Get(header)
        // X-Forwarded-For may be a comma-separated list; take the first entry
        if idx := strings.IndexByte(raw, ','); idx != -1 {
            raw = raw[:idx]
        }
        raw = strings.TrimSpace(raw)
        if ip := net.ParseIP(raw); ip != nil {
            return ip.String()
        }
    }
    ipAddress, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ipAddress
}
```

Additionally, ensure the reverse proxy is configured to unconditionally overwrite `X-Real-IP` with `$remote_addr` (e.g., nginx `proxy_set_header X-Real-IP $remote_addr;`).

## Proof of Concept
```bash
# Inject a fake log line as the client IP
curl -H "X-Real-IP: 10.0.0.1 POST /admin 200 in 0s" \
     http://<rosetta-host>/network/status

# Expected log output (corrupted):
# INFO 10.0.0.1 POST /admin 200 in 0s GET /network/status (200) in 3ms
```

The injected string `10.0.0.1 POST /admin 200 in 0s` appears verbatim in the trace log, prepended to the real request details, producing a forged log entry.
