Audit Report

## Title
Missing `deleted is not true` Filter in `entity_history` Branch of `findActiveByEvmAddressOrAliasAndTimestamp` and `findActiveByEvmAddressAndTimestamp`

## Summary
The `entity_history` sub-query in both `findActiveByEvmAddressOrAliasAndTimestamp` and `findActiveByEvmAddressAndTimestamp` omits the `deleted is not true` guard that is present in the `entity` branch of those same queries and in both branches of the equivalent `findActiveByIdAndTimestamp`. This allows a deleted entity to be returned as active for historical timestamp queries, causing incorrect entity state to be surfaced through the mirror node's EVM execution path.

## Finding Description

**Exact code location**

`findActiveByEvmAddressOrAliasAndTimestamp` (lines 95–121) and `findActiveByEvmAddressAndTimestamp` (lines 59–85) in `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`.

The `entity_history` sub-query in `findActiveByEvmAddressOrAliasAndTimestamp`:

```sql
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2
    and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
``` [1](#0-0) 

There is no `deleted is not true` predicate. The `entity` branch of the same query does include it: [2](#0-1) 

By contrast, `findActiveByIdAndTimestamp` correctly applies `and deleted is not true` to **both** branches: [3](#0-2) 

**Root cause**

The `entity_cte` resolves the entity ID from the `entity` table without checking `deleted`: [4](#0-3) 

So even when the current entity row has `deleted = true`, the CTE still returns its ID. The `entity` branch then correctly returns nothing due to `e.deleted is not true`, but the `entity_history` branch has no such guard and returns the most recent history row whose `lower(timestamp_range) <= blockTimestamp` — regardless of whether that row is marked deleted.

**Exploit flow (simplest variant)**

| Time | Event | `entity` table | `entity_history` table |
|------|-------|----------------|------------------------|
| T0 | Created | `[T0,∞) deleted=false` | — |
| T1 | Deleted | `[T1,∞) deleted=true` | `[T0,T1) deleted=false` |

Caller submits `blockTimestamp = T1` (or any value ≥ T0):
1. `entity_cte` → finds entity ID (`created_timestamp=T0 ≤ T1`).
2. `entity` branch → `deleted=true` → returns **empty**.
3. `entity_history` branch → `[T0,T1)` has `lower=T0 ≤ T1` → returns this row (`deleted=false`).
4. Final `ORDER BY timestamp_range DESC LIMIT 1` → returns the `[T0,T1) deleted=false` row.

Result: **a deleted entity is returned as active**.

**Why existing tests fail to catch this**

The tests at lines 366–382 only persist an `entity_history` row via `persistEntityHistoryWithDeleted()` *without* a corresponding `entity` row: [5](#0-4) 

Without an `entity` row, `entity_cte` (which queries the `entity` table) returns `NULL`, so both branches return empty and the test passes. Additionally, the tests use `timestampLower - 1`, making `lower(T) <= T-1` false, so the history row would never be reached even if the CTE did return an ID. The real attack path — a deleted `entity` row paired with a history row — is never exercised.

## Impact Explanation

`findActiveByEvmAddressOrAliasAndTimestamp` is called by `CommonEntityAccessor.get(Bytes alias, Optional<Long> timestamp)`: [6](#0-5) 

This feeds directly into `AliasesReadableKVState.readFromDataSource()`, which resolves `AccountID` for EVM state: [7](#0-6) 

Concrete consequences:
- A deleted entity is resolved as active in historical `eth_call`, `eth_getBalance`, and contract-call queries, returning stale or incorrect balances and state.
- A `deleted=true` `entity_history` row can be returned as a non-empty `Optional`, causing callers that do not re-check the `deleted` flag to treat the entity as present.
- Mirror node exports incorrect historical entity state, undermining the integrity guarantees of the mirror node API.

## Likelihood Explanation

No privilege is required. Any user of the public JSON-RPC endpoint can issue an `eth_call` or `eth_getBalance` with an arbitrary `blockNumber` (which maps to a `blockTimestamp`). The attacker only needs to know the EVM address or alias of a previously deleted account (publicly visible on-chain) and the block number at which the deletion occurred (also publicly visible). Entity deletion is a normal Hedera operation (account expiry). The attack is fully repeatable and deterministic.

## Recommendation

Add `and eh.deleted is not true` to the `entity_history` sub-query in both `findActiveByEvmAddressOrAliasAndTimestamp` and `findActiveByEvmAddressAndTimestamp`, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2
    and eh.id = (select id from entity_cte)
    and eh.deleted is not true          -- ADD THIS
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

Additionally, add test cases that persist both an `entity` row (with `deleted=true`) and a corresponding `entity_history` row, and assert that the query returns empty for a `blockTimestamp` within the history row's range.

## Proof of Concept

1. Insert into `entity`: `id=1, evm_address=X, created_timestamp=T0, deleted=true, timestamp_range=[T1,∞)`.
2. Insert into `entity_history`: `id=1, evm_address=X, deleted=false, timestamp_range=[T0,T1)`.
3. Call `entityRepository.findActiveByEvmAddressOrAliasAndTimestamp(X, T0)`.
4. Observe: returns the `entity_history` row with `deleted=false` instead of `Optional.empty()`. [8](#0-7)
