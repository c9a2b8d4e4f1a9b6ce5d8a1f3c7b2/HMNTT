# Audit Report

## Title
O(n²) Sibling Chain Traversal in `parsePossiblyHookContractRelatedParent()` Enables Mirror Node CPU Exhaustion

## Summary
`RecordItem.java`'s inner `RecordItemBuilder.parsePossiblyHookContractRelatedParent()` contains an unbounded backward walk over the `previous` linked chain. For a parent transaction P followed by n sibling child transactions C1…Cn, the k-th sibling walks k−1 nodes, yielding O(n²) total work across the sibling group. An unprivileged attacker can trigger this by submitting a single `ContractCall` that spawns many internal child calls to contracts whose entity number is not 365.

## Finding Description

**Location**: `common/src/main/java/org/hiero/mirror/common/domain/transaction/RecordItem.java`, inner class `RecordItemBuilder`, method `parsePossiblyHookContractRelatedParent()`, lines 350–375.

**`previous` chain wiring**: Every record reader sets `previous` to the immediately preceding `RecordItem` in stream order. `AbstractRecordFileItemReader.onRecordStreamItem()` (line 173) sets `.previous(items.isEmpty() ? null : items.getLast())`, and `AbstractPreV5RecordFileReader.readBody()` (line 110) sets `.previous(lastRecordItem)`. For parent P followed by siblings C1…Cn the chain is: Cn.previous = C(n−1), …, C2.previous = C1, C1.previous = P.

**`parent` resolution**: `parseParent()` (lines 330–348) resolves the parent for any sibling Ck (k ≥ 2) via `previous.parent` (which equals P), so `parent = P` for every sibling. [6](#0-5) 

**The loop**:
```java
while (candidateRecord != null && candidateRecord != parent) {
    var candidateTxnRecordContractResult = candidateRecord.getContractResult();
    if (candidateTxnRecordContractResult == null) { break; }
    if (candidateTxnRecordContractResult.getContractID().getContractNum() == HOOK_CONTRACT_NUM) {
        return candidateRecord;
    }
    candidateRecord = candidateRecord.previous;
}
``` [7](#0-6) 

For sibling Ck the loop starts at C(k−1) and walks backward until it hits P (== `parent`). If every sibling carries a non-null `ContractFunctionResult` with `contractNum != 365`, no early exit fires and the loop runs exactly k−1 iterations. Summing over all n siblings: 0+1+…+(n−1) = n(n−1)/2 = **O(n²)**.

**Entry conditions the attacker fully controls**:
1. `hasParentConsensusTimestamp()` — set by the consensus node for every internal child call; triggered by submitting a `ContractCall`.
2. `currentTxnRecordContractResult != null` — present on every child contract call record.
3. `contractNum != HOOK_CONTRACT_NUM (365)` — all user-deployed contracts have entity numbers other than 365 by default.
4. `previous != null` — always true for non-first items in a file.

**Why existing checks are insufficient**: The only early-exit conditions inside the loop are (a) `candidateRecord == parent` (terminates correctly but only after full traversal), (b) `candidateTxnRecordContractResult == null` (never fires if all siblings are contract calls), and (c) `contractNum == 365` (never fires if attacker avoids 365). There is no depth cap, no visited-set, and no maximum sibling count enforced at this layer. [7](#0-6) 

## Impact Explanation
With Hedera's maximum gas per transaction (~15,000,000 gas) and a minimal CALL costing ~700–1,000 gas, an attacker can produce hundreds to low thousands of child transactions per parent. At n = 1,000 siblings the loop executes ~500,000 iterations per parent transaction. Submitting a sustained stream of such transactions (each costing the attacker gas but imposing quadratic work on the mirror node's importer thread) can drive the importer's CPU well above sustainable thresholds. Because `parsePossiblyHookContractRelatedParent()` is called synchronously during `RecordItem.build()` for every item in the stream, CPU saturation directly stalls all record file ingestion, causing the mirror node to fall behind the live chain.

## Likelihood Explanation
No special privileges are required. Any Hedera account holder can deploy a fan-out contract and submit `ContractCall` transactions. The attack is repeatable and can be automated. The cost to the attacker scales linearly with n while the cost to the mirror node scales quadratically, making the attack economically asymmetric. The attack produces no errors — only elevated CPU — making it silent and difficult to distinguish from legitimate load.

## Recommendation
Replace the backward linear scan with an O(1) lookup. The simplest fix is to record the first hook-contract child on the parent `RecordItem` when it is first encountered, and have subsequent siblings read that cached reference directly. Alternatively, cap the traversal depth (e.g., break after a fixed maximum of iterations) as a short-term mitigation. The root cause is that the algorithm re-traverses already-visited siblings for each new sibling; the fix is to avoid repeated traversal entirely.

## Proof of Concept
1. Deploy contract A (entity number ≠ 365) that, when called, issues N internal `CALL` opcodes to contract B (also entity number ≠ 365).
2. Submit a `ContractCall` to contract A with sufficient gas to produce N child transactions (e.g., N = 500).
3. The mirror node importer reads the record file: parent P followed by C1…C500, each with a `ContractFunctionResult` whose `contractNum` ≠ 365.
4. During `build()` for each Ck, `parsePossiblyHookContractRelatedParent()` walks k−1 nodes backward.
5. Total iterations: 0+1+…+499 = 124,750 for a single parent transaction.
6. Repeat this transaction in a tight loop; the importer's parse thread saturates, stalling ingestion of all subsequent record files.

        private RecordItem parseParent() {
            if (transactionRecord.hasParentConsensusTimestamp() && previous != null) {
                var parentTimestamp = transactionRecord.getParentConsensusTimestamp();
                if (parentTimestamp.equals(previous.transactionRecord.getConsensusTimestamp())) {
                    return previous;
                } else if (previous.parent != null
                        && parentTimestamp.equals(previous.parent.transactionRecord.getConsensusTimestamp())) {
                    // check older siblings parent, if child count is > 1 this prevents having to search to parent
                    return previous.parent;

                    // transaction so following items in batch may need to look at the grandparent

                }
            }
            return this.parent;
        }

        private RecordItem parsePossiblyHookContractRelatedParent() {
            final var currentTxnRecordContractResult = getContractResult();

            if (transactionRecord.hasParentConsensusTimestamp()
                    && currentTxnRecordContractResult != null
                    && currentTxnRecordContractResult.getContractID().getContractNum() != HOOK_CONTRACT_NUM
                    && previous != null) {
                var candidateRecord = previous;

                while (candidateRecord != null && candidateRecord != parent) {
                    var candidateTxnRecordContractResult = candidateRecord.getContractResult();
                    if (candidateTxnRecordContractResult == null) {
                        break;
                    }

                    if (candidateTxnRecordContractResult.getContractID().getContractNum() == HOOK_CONTRACT_NUM) {
                        // we found the first hook child item, which has a ContractResult and is executed via a hook
                        return candidateRecord;
                    }

                    candidateRecord = candidateRecord.previous;
                }
            }

            return null;
        }
