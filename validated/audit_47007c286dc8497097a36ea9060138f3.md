Audit Report

## Title
Unbounded `json_agg` on `transaction_signature` in `getScheduleById` Enables Memory Exhaustion DoS

## Summary
`getScheduleByIdQuery` in `rest/schedules.js` performs an uncapped `json_agg` over all rows in `transaction_signature` matching a given `schedule_id`. Because the table intentionally has no unique constraint, an attacker can accumulate an arbitrary number of rows for a single schedule and force the mirror node to materialize an unbounded JSON payload in Node.js heap memory on every `GET /api/v1/schedules/{id}` request.

## Finding Description
**Exact code path — `rest/schedules.js` lines 39–64:**

```js
const transactionSignatureJsonAgg = `
  json_agg(json_build_object(
    'consensus_timestamp', ts.consensus_timestamp,
    'public_key_prefix', encode(ts.public_key_prefix, 'base64'),
    'signature', encode(ts.signature, 'base64'),
    'type', ts.type
  ) order by ts.consensus_timestamp)`;   // ← no LIMIT

const getScheduleByIdQuery = `
  ...
  ${transactionSignatureJsonAgg} as signatures
  from schedule s
  left join entity e on e.id = s.schedule_id
  left join transaction_signature ts on ts.entity_id = s.schedule_id  // ← unbounded join
  where s.schedule_id = $1
  group by s.schedule_id, e.id`;
``` [1](#0-0) 

**Root cause:** The `json_agg` aggregates every row in `transaction_signature` for the given `schedule_id` with no row cap. The failed assumption is that the number of signatures per schedule is bounded by the network protocol.

**Why the assumption fails:** The database schema design explicitly documents: *"There's no unique constraint/primary key since the client can potentially sign multiple times with the same key."* [3](#0-2) 

The `transaction_signature` table has no primary key or unique constraint: [2](#0-1) 

The previously-unique index on `(consensus_timestamp, public_key_prefix)` was explicitly dropped and recreated as non-unique: [4](#0-3) 

The importer inserts one `TransactionSignature` row per `sigPair` entry in the `sigMap` of every `ScheduleCreate` or `ScheduleSign` transaction: [10](#0-9) 

**Why existing checks are insufficient:**

- `response.limit.default` (25) and `response.limit.max` (100) apply only to paginated list endpoints. `getScheduleById` is a single-resource endpoint with no pagination and no limit applied to the signatures sub-array. [7](#0-6) 

- The `statementTimeout` of 20,000 ms limits query *execution time*, not the size of the result set materialized in memory. A query aggregating tens of thousands of rows can complete well within 20 seconds while producing a multi-hundred-MB payload. [8](#0-7) 

- There is no HTTP response body size cap in the REST middleware. `responseHandler.js` calls `res.send()` unconditionally after `JSONStringify`: [6](#0-5) 

## Impact Explanation
A single attacker-controlled schedule can force the mirror node to allocate arbitrarily large amounts of heap memory per request. Because Node.js is single-threaded and heap allocation is synchronous during JSON serialization (`JSONStringify`), a sufficiently large payload will stall the event loop and/or trigger an OOM crash, making the entire REST API unavailable. The attack targets a public, unauthenticated endpoint (`GET /api/v1/schedules/{id}`), so no credentials are required beyond HBAR to fund the setup transactions.

## Likelihood Explanation
Hedera transaction fees are fractions of a US cent; on testnet they are free. A `ScheduleSign` transaction can carry multiple `sigPair` entries per submission. An attacker can automate thousands of `ScheduleSign` submissions over the lifetime of a schedule (up to ~62 days) at negligible cost. The attack is repeatable: a new schedule can be created after each mitigation attempt. No privileged access to the mirror node or its database is required.

## Recommendation
1. **Add a `LIMIT` clause to the `transaction_signature` join** inside `getScheduleByIdQuery`, capping the number of signatures returned to a configurable maximum (e.g., 1,000 or the existing `response.limit.max`).
2. **Alternatively, paginate the signatures sub-array** using a separate query with `LIMIT`/`OFFSET` or keyset pagination, consistent with how other list endpoints are handled.
3. **Add a response body size guard** in `responseHandler.js` (or at the reverse-proxy layer) to reject or truncate responses exceeding a configured byte threshold.
4. **Consider adding a unique partial index** on `transaction_signature (entity_id, public_key_prefix, consensus_timestamp)` to prevent duplicate rows from accumulating for the same schedule/key/timestamp combination.

## Proof of Concept
```
# 1. Create a scheduled transaction on Hedera (ScheduleCreate)
#    → scheduleId = 0.0.XXXXX

# 2. Submit N ScheduleSign transactions, each with M sigPairs
for i in $(seq 1 10000); do
  hedera schedule sign 0.0.XXXXX --sig-pairs M
done

# 3. Query the mirror node single-resource endpoint
curl https://<mirror-node>/api/v1/schedules/0.0.XXXXX
# → mirror node executes getScheduleByIdQuery, aggregates N×M rows via json_agg,
#   materializes the entire array in PostgreSQL, transfers to Node.js heap,
#   and calls JSONStringify on the unbounded payload before sending.
```
