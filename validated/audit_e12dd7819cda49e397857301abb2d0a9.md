Audit Report

## Title
Cache Bypass for Address Book Files Due to Missing Timestamp Normalization Causes Unbounded DB + Protobuf Parse Per Request

## Summary
`SystemFileLoader.load()` applies `roundDownToHour()` timestamp normalization only for exchange-rate, fee-schedule, and simple-fee-schedule files. Address book files (`addressBookFile101`, `addressBookFile102`) receive a raw nanosecond wall-clock timestamp as their cache key. Because `FileReadableKVState.readFromDataSource` always supplies `Utils.getCurrentTimestamp()` — unique per call — the Caffeine cache is permanently bypassed for address book files. Every read triggers a live PostgreSQL query and a `NodeAddressBook.PROTOBUF.parse()` allocation.

## Finding Description

**Step 1 — Caller always passes wall-clock nanoseconds for system files**

`FileReadableKVState.readFromDataSource` obtains the `ContractCallContext` timestamp at line 51 but ignores it for system files. Instead, it unconditionally calls `getCurrentTimestamp()` at line 54 and passes that value to `systemFileLoader.load()` at line 57: [1](#0-0) 

`Utils.getCurrentTimestamp()` returns `Instant.now()` converted to nanoseconds — a value that is unique for every invocation: [2](#0-1) 

**Step 2 — `roundDownToHour` is not applied to address book files**

Inside `SystemFileLoader.load()`, the normalization branch at lines 113–118 covers only `exchangeRateFileId`, `feeSchedulesFileId`, and `simpleFeeSchedulesFileId`. Address book file IDs fall through with the raw nanosecond value, which is then embedded directly into the `CacheKey`: [3](#0-2) 

**Step 3 — Cache miss on every request**

Because `CacheKey(fileId, consensusTimestamp)` embeds a nanosecond timestamp that is unique per request, `cache.get(cacheKey, File.class)` always returns `null` for address book files. Execution falls through to `loadFromDB` → `loadWithRetry`: [4](#0-3) 

**Step 4 — DB query + protobuf parse on every miss**

`loadWithRetry` unconditionally calls `fileDataRepository.getFileAtTimestamp()` and, when data is found, calls `codec.parse()` (`NodeAddressBook.PROTOBUF` for address book files): [5](#0-4) 

The address book codec is confirmed as `NodeAddressBook.PROTOBUF` in `loadAll()`: [6](#0-5) 

**Step 5 — Cache configuration confirms zero reuse**

The `systemFile` Caffeine cache is configured `maximumSize=20,expireAfterWrite=10m`. With a unique nanosecond key per request, the 20 slots fill instantly with non-reusable entries and provide no protection: [7](#0-6) 

## Impact Explanation
Every request that causes the EVM to read `addressBookFile101` or `addressBookFile102` unconditionally issues a PostgreSQL query (`getFileAtTimestamp`) and allocates a full `NodeAddressBook` protobuf object. The intended caching layer provides zero protection for these files. Under the default throttle of 500 req/s, address-book-triggering traffic produces 500 DB queries/s and 500 short-lived `NodeAddressBook` allocations/s sustained indefinitely. The DB connection pool is the hard limit: sustained traffic of this kind can saturate the pool and degrade or deny service to all other queries. GC pressure from the short-lived allocations compounds the problem under concurrent load.

## Likelihood Explanation
No privilege is required. Any caller who submits an `eth_call` or `eth_estimateGas` that internally reads the address book (e.g., a precompile or system-contract call that resolves node addresses) triggers this path. The attack requires no special knowledge beyond identifying which contract call causes an address-book read. It is trivially repeatable and scriptable. This is not a volumetric DDoS — the cache is broken by design for these files, so even a single sustained caller at modest rates causes unbounded DB load.

## Recommendation
Apply the same `roundDownToHour` normalization (or an equivalent coarser-grained normalization) to `addressBookFile101` and `addressBookFile102` inside `SystemFileLoader.load()`. The address book changes infrequently; rounding the timestamp down to the hour (or even to the day) is semantically correct and would allow the existing `defaultSystemFileCacheManager` cache to absorb repeated reads. Alternatively, since address book files are static for the duration of a request, consider bypassing the DB entirely and returning the genesis/mock address book directly (as is already done for `networkPropertyFile`). [8](#0-7) 

## Proof of Concept
1. Identify a precompile or system-contract call that reads `addressBookFile101` or `addressBookFile102` (e.g., a call to a Hedera system contract that resolves node addresses).
2. Submit the same `eth_call` request repeatedly (e.g., 100 times in rapid succession).
3. Enable DB query logging on the PostgreSQL instance. Observe that `getFileAtTimestamp` is called for the address book entity ID on every single request — no cache hits occur.
4. Confirm by adding a debug log or breakpoint at `SystemFileLoader.java` line 137 (`log.info("Updating cache for key {}", cacheKey)`): it fires on every request because the nanosecond-keyed entry written on request N is never matched by request N+1.
