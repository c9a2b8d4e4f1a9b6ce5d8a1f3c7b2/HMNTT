Audit Report

## Title
Missing EVM Address Length Validation Causes Unhandled `BufferUnderflowException` in `getByEvmAddressAndType()`

## Summary
`decodeEvmAddress()` in `GraphQlUtils.java` performs no length check on the decoded byte array. Any user can supply a valid even-length hex string shorter than 40 hex characters (< 20 bytes), causing `ByteBuffer.getInt()` or `getLong()` in `EntityServiceImpl.getByEvmAddressAndType()` to throw a `java.nio.BufferUnderflowException`. Because `CustomExceptionResolver` does not handle this exception type, Spring GraphQL returns a generic `INTERNAL_ERROR` response instead of a `ValidationError`.

## Finding Description
**Code path, verified line-by-line:**

1. `AccountController.account()` passes the raw user-supplied `evmAddress` string directly to `entityService.getByEvmAddressAndType()` with no prior length validation. [1](#0-0) 

2. `EntityServiceImpl.getByEvmAddressAndType()` calls `decodeEvmAddress()`, wraps the result in a `ByteBuffer`, then unconditionally calls `buffer.getInt()` and `buffer.getLong()` (the third `getLong()` is conditional, but the first two reads are not guarded). [2](#0-1) 

3. `decodeEvmAddress()` strips the optional `0x` prefix and calls `Hex.decodeHex()`. It catches `DecoderException` (odd-length or non-hex input) and rethrows as `IllegalArgumentException`, but **performs no check on the length of the resulting byte array**. [3](#0-2) 

4. `CustomExceptionResolver.resolveToSingleError()` only handles `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException`. `BufferUnderflowException` (a `RuntimeException`) is not in this list, so the method returns `null`, and Spring GraphQL falls back to a generic `INTERNAL_ERROR` response. [4](#0-3) 

**Root cause:** `decodeEvmAddress()` assumes it will always return exactly 20 bytes. There is no guard enforcing this invariant before the `ByteBuffer` reads in `getByEvmAddressAndType()`.

## Impact Explanation
Every such request returns HTTP 200 with a GraphQL `INTERNAL_ERROR` classification instead of a `ValidationError`. This:
- Creates an error-type oracle: clients can distinguish a real internal failure from a bad-input failure, which is not the intended behavior.
- Pollutes server logs and metrics with unhandled exception stack traces on every crafted request.
- Prevents proper client-side error handling.

Severity: **Medium**. No sensitive data is directly exposed, but the unhandled exception path is fully attacker-controlled and repeatable.

## Likelihood Explanation
No authentication or special privilege is required. Any user who can send a GraphQL query to the `account` endpoint can trigger this with a single crafted request. The input constraint (even-length hex string shorter than 40 chars) is trivially satisfied. The attack is fully repeatable and requires no prior knowledge of the system.

## Recommendation
Add a length check in `decodeEvmAddress()` (or at the call site in `getByEvmAddressAndType()`) to enforce that the decoded byte array is exactly 20 bytes, throwing an `IllegalArgumentException` if not. For example, in `GraphQlUtils.java`:

```java
public static byte[] decodeEvmAddress(String evmAddress) {
    if (evmAddress == null) {
        return ArrayUtils.EMPTY_BYTE_ARRAY;
    }
    try {
        evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
        byte[] decoded = Hex.decodeHex(evmAddress);
        if (decoded.length != 20) {
            throw new IllegalArgumentException("evmAddress must be 20 bytes");
        }
        return decoded;
    } catch (DecoderException e) {
        throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
    }
}
```

This ensures `IllegalArgumentException` is thrown for short inputs, which `CustomExceptionResolver` already handles as a `ValidationError`. [3](#0-2) 

## Proof of Concept
Send the following GraphQL query to the `account` endpoint:

```graphql
{
  account(input: { evmAddress: "aabb" }) {
    id
  }
}
```

- `"aabb"` decodes to 2 bytes. `ByteBuffer.getInt()` requires 4 bytes and throws `BufferUnderflowException`.
- `CustomExceptionResolver` returns `null` for this exception type.
- Spring GraphQL responds with HTTP 200 and `"errorType": "INTERNAL_ERROR"` instead of `"errorType": "ValidationError"`.

Variants:
- `"aabbccdd"` (4 bytes): `getInt()` succeeds, `getLong()` throws.
- Any even-length hex string with fewer than 40 hex characters triggers the underflow at some read. [5](#0-4)
