## Audit Report

## Title
Unbounded Prometheus Label Cardinality via Unmatched Routes in `metricsMiddleware`

## Summary
The `metricsMiddleware` in `rest/middleware/metricsHandler.js` records per-route Prometheus metrics using a `path` label derived by `toOpenApiPath()`. For requests that match no registered Express route, `toOpenApiPath` falls back to the raw `req.path` value. Because the OpenTelemetry `MeterProvider` holds strong references to every unique label combination and never evicts them, an unauthenticated attacker can exhaust Node.js heap memory by sending requests with unique paths, crashing the REST API.

## Finding Description
`toOpenApiPath` resolves the `path` label in priority order:
1. `res.locals[requestPathLabel]` — set only by `recordRequestPath` for matched routes inside `AccountRoutes`, `BlockRoutes`, and `ContractRoutes`.
2. `req.route.path` — set only when Express matched a registered route.
3. **Fallback: `req.path`** — the raw, un-normalized URL path.

For any request that does not match a registered route, both (1) and (2) are absent, so line 18 of `metricsHandler.js` assigns `path = req.path`. On `res.on('finish')` (line 185), this raw path is used as a label across all four per-route instruments (`requestTotalCounter`, `durationHistogram`, `requestSizeHistogram`, `responseSizeHistogram`). The `metricsMiddleware` registers this hook before `responseHandler` runs, so the `NotFoundError` thrown at `responseHandler.js:30` does not prevent the label from being recorded.

The `MeterProvider` accumulates a separate time-series entry for every unique `{method, path, code}` triple. There is no cardinality cap, TTL, or eviction mechanism anywhere in the metrics pipeline.

## Impact Explanation
Each unique path value creates new in-memory time-series entries across four instruments. Histograms additionally allocate per-bucket counters (4 duration buckets, 4 response-size buckets by default). The `MeterProvider` holds strong references to all registered time-series, preventing garbage collection. Sending requests with unique paths (e.g., `/api/v1/<counter>`) causes monotonically growing heap consumption until the Node.js process is OOM-killed or becomes unresponsive, denying service to all legitimate users of the mirror node REST API.

## Likelihood Explanation
The attack requires no authentication, no special headers, and no prior knowledge beyond the public API hostname. Any HTTP client can generate unique paths by appending a counter or random string. The REST API is publicly exposed by design, and `server.js` contains no rate-limiting middleware of any kind.

## Recommendation
1. **Normalize unmatched-route paths before recording metrics.** Replace the `req.path` fallback in `toOpenApiPath` with a fixed sentinel label (e.g., `"unmatched"` or `apiPrefix + "/unknown"`) so all 404 traffic collapses to a single time-series entry.
2. **Add a cardinality guard.** Maintain a bounded set of known `path` label values; discard or bucket any value not in the set.
3. **Deploy rate-limiting middleware** (e.g., `express-rate-limit`) in `server.js` to limit the request rate per IP, reducing the feasibility of rapid label-space enumeration.

## Proof of Concept
```bash
# Send N requests with unique paths; each creates a new time-series entry
for i in $(seq 1 1000000); do
  curl -s "http://<mirror-node-host>/api/v1/nonexistent/$i" &
done
```
Monitor Node.js heap via `/swagger/metrics/` (`nodejs_process_memory_heap_used_bytes`); it will grow monotonically with no bound until the process is exhausted.
