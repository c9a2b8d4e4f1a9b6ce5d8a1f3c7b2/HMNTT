Audit Report

## Title
Cache Key Collision via `Arrays.hashCode()` on EVM Addresses Enables Cross-Address Entity Confusion in `CACHE_NAME_EVM_ADDRESS`

## Summary
`findByEvmAddressAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` — which delegates to `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole cache key for 20-byte EVM addresses. Because 2^160 possible EVM addresses map into a 2^32 key space, an attacker can trivially compute a second address B that collides with any existing cached address A. When the cache is hit for B, it returns Entity A's data, causing the EVM to execute with incorrect entity state for address B.

## Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 32–37: [1](#0-0) 

```java
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
```

`common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java`, lines 20–22: [2](#0-1) 

```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
```

**Root cause:** `Arrays.hashCode(byte[])` computes a 32-bit polynomial hash (`result = 31 * result + (b & 0xFF)` for each byte), mapping the 2^160 EVM address space into 2^32 possible integer keys. Spring's `@Cacheable` uses the SpEL expression result — an `int` — as the literal cache key stored in the underlying Caffeine `ConcurrentHashMap`. There is no secondary equality check against the original byte array. Two distinct addresses A and B with `Arrays.hashCode(A) == Arrays.hashCode(B)` will map to the same cache entry.

The same flaw exists identically in `findByEvmAddressOrAliasAndDeletedIsFalse` at lines 39–49. [3](#0-2) 

The entity cache is configured with `expireAfterWrite=1s,maximumSize=10000`: [4](#0-3) 

The `CACHE_MANAGER_ENTITY` bean registers both `CACHE_NAME_EVM_ADDRESS` and `CACHE_NAME_ALIAS` under this same spec: [5](#0-4) 

**Why existing checks fail:**
- `unless = "#result == null"` only prevents caching null results; it does not prevent a collision-based false hit from being returned.
- The 1-second TTL limits the window but does not eliminate it — the attacker controls the timing of both requests programmatically.
- There is no equality check between the cached key and the actual input bytes; the integer hash is the only discriminator.

## Impact Explanation
When address B resolves to Entity A's data via cache collision, the EVM executes with incorrect state: wrong bytecode, wrong balance, wrong account type. A contract call to address B may execute Entity A's code, read Entity A's storage, or report Entity A's balance. This is a cross-address integrity failure. In a scenario where address B is expected to be an EOA or non-existent, but the cache returns a contract entity, the EVM may incorrectly execute contract logic, leading to wrong call results, incorrect gas accounting, or state corruption visible to callers. The effect is local to each node (Caffeine is in-memory per-node), but any node serving the attacker's requests is affected.

## Likelihood Explanation
The attacker requires no privileged access — only the ability to submit standard `eth_call` or contract call RPC requests. Computing a collision in `Arrays.hashCode` for a 20-byte array is trivial: fix bytes 0–17 arbitrarily, then enumerate all 65,536 combinations of bytes 18–19 until `Arrays.hashCode(B) == Arrays.hashCode(A)`. This requires at most 65,536 arithmetic operations and no cryptographic capability. The 1-second TTL requires both requests within the same second, which is trivially achievable programmatically. The attack is repeatable on demand.

## Recommendation
Replace the 32-bit hash with a collision-resistant key derived from the full byte array. The simplest correct fix is to use the hex-encoded string of the address as the cache key:

```java
key = "T(org.hiero.mirror.common.util.DomainUtils).bytesToHex(#alias)"
```

Alternatively, wrap the byte array in a type with proper structural equality (e.g., `java.nio.ByteBuffer.wrap(#alias)` has `equals`/`hashCode` based on content). The same fix must be applied to `findByEvmAddressOrAliasAndDeletedIsFalse`.

## Proof of Concept

```java
// Find address B that collides with address A under Arrays.hashCode
byte[] A = /* any known EVM address, e.g. from on-chain data */;
int targetHash = Arrays.hashCode(A);

// Fix first 18 bytes of B arbitrarily (e.g., same as A)
byte[] B = Arrays.copyOf(A, 20);
outer:
for (int b18 = 0; b18 < 256; b18++) {
    for (int b19 = 0; b19 < 256; b19++) {
        B[18] = (byte) b18;
        B[19] = (byte) b19;
        if (!Arrays.equals(A, B) && Arrays.hashCode(B) == targetHash) {
            break outer; // B is a collision address
        }
    }
}

// Step 1: Trigger caching of Entity A
// GET /api/v1/contracts/call  { "to": hex(A), ... }

// Step 2 (within 1 second): Query with address B
// GET /api/v1/contracts/call  { "to": hex(B), ... }
// Spring @Cacheable computes Arrays.hashCode(B) == targetHash → cache HIT → returns Entity A's data
```
