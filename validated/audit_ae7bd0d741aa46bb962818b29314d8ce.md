Audit Report

## Title
`SPENDER_ID` Filter Accepts Opaque EVM Address But Resolves to `null`, Causing Silent Zero-Result Query on Token Allowances

## Summary
When an opaque (non-parsable) EVM address is submitted as the `spender.id` query parameter to `/api/v1/accounts/{id}/allowances/tokens`, it passes input validation but is silently converted to `null` during formatting. This `null` is then bound directly into the SQL query as `spender = $N`, which in PostgreSQL always evaluates to false, causing the endpoint to return zero allowance records even when allowances exist for that spender.

## Finding Description

**Step 1 — Validation passes for opaque EVM addresses.**

In `filterValidityChecks`, the `SPENDER_ID` case calls `EntityId.isValidEntityId(val)` with no additional arguments, meaning defaults `allowEvmAddress = true` and `evmAddressType = ANY` are used: [1](#0-0) 

`isValidEntityId` with `evmAddressType = ANY` accepts any 40-hex-char string, including opaque EVM addresses like `0x71eaa748d5252be68c1185588beca495459fdba4`: [2](#0-1) 

This is confirmed by the existing test suite: [3](#0-2) 

**Step 2 — Formatting calls `getEncodedId()` which returns `null` for opaque addresses.**

In `formatComparator`, the `SPENDER_ID` case calls `EntityId.parseString(comparator.value).getEncodedId()` with no options: [4](#0-3) 

For an opaque EVM address, `piecesFromString` detects it via `isEvmAddressAlias` and prepends `[null, null]` as shard/realm: [5](#0-4) 

After filtering nulls and re-parsing, `parseFromString` reaches the EVM address branch. Since the prefix is not the 24-zero `longFormEvmAddressPrefix`, it returns `[shard, realm, null, numOrEvmAddress]` — `num` is `null`: [6](#0-5) 

`getEncodedId()` then returns `null` when `this.num === null`: [7](#0-6) 

So `comparator.value` is silently set to `null`.

**Step 3 — `null` is bound into the SQL query.**

`TokenAllowanceService.getSubQuery` pushes `filter.value` (now `null`) directly into `params` and generates `spender = $N`: [8](#0-7) 

In PostgreSQL, `spender = NULL` is never true (NULL comparison requires `IS NULL`), so the query returns zero rows.

**Contrast with `TOKEN_ID`:** The `TOKEN_ID` case explicitly passes `evmAddressType: EvmAddressType.NUM_ALIAS`, which only accepts parsable (long-zero) EVM addresses that map to a numeric entity ID. `SPENDER_ID` has no such restriction: [9](#0-8) 

## Impact Explanation
Any unprivileged external user can submit a valid-looking opaque EVM address as `spender.id` and receive an empty `allowances` array (HTTP 200), regardless of how many allowances exist for that spender. The API silently returns incorrect results instead of an error or the correct records. Downstream clients or integrations relying on this endpoint to check allowance existence will be misled. The same bug applies to the crypto allowances endpoint which shares the same `formatComparator` path.

## Likelihood Explanation
No authentication or special privilege is required. Any user who knows or discovers a spender's EVM address (publicly observable on-chain) can trigger this. The EVM address format is widely used in EVM-compatible tooling, making accidental or deliberate use highly likely. The attack is trivially repeatable with a single HTTP GET request.

## Recommendation
Restrict `SPENDER_ID` validation to only accept parsable (long-zero) EVM addresses, mirroring the `TOKEN_ID` approach. In `filterValidityChecks` (`rest/utils.js`), change:

```js
case constants.filterKeys.SPENDER_ID:
  ret = EntityId.isValidEntityId(val);
  break;
```

to:

```js
case constants.filterKeys.SPENDER_ID:
  ret = EntityId.isValidEntityId(val, true, constants.EvmAddressType.NUM_ALIAS);
  break;
```

This ensures opaque EVM addresses are rejected at validation time with a proper `400 Bad Request`, rather than silently producing a null-bound SQL query. Correspondingly, `formatComparator` for `SPENDER_ID` should also pass `evmAddressType: EvmAddressType.NUM_ALIAS` to `parseString` for consistency.

## Proof of Concept

```
GET /api/v1/accounts/0.0.1000/allowances/tokens?spender.id=0x71eaa748d5252be68c1185588beca495459fdba4
```

1. `filterValidityChecks` accepts the value (opaque EVM address passes `isValidEntityId` with default args).
2. `formatComparator` calls `EntityId.parseString('0x71eaa748d5252be68c1185588beca495459fdba4').getEncodedId()`, which returns `null`.
3. `TokenAllowanceService.getSubQuery` pushes `null` into params and generates `spender = $3`.
4. PostgreSQL evaluates `spender = NULL` as false for every row.
5. Response: HTTP 200 `{"allowances": [], "links": {"next": null}}` — even if allowances exist for that spender.
