Audit Report

## Title
Thread Pool Exhaustion via Unauthenticated Flood on `GET /api/v1/topics/{id}` During Database Degradation

## Summary
`TopicController.getTopic()` issues three sequential, synchronous, blocking repository calls per request with no application-level rate limiting, no per-IP throttle, and no HikariCP pool size cap. Under database degradation, an unauthenticated attacker can exhaust all Tomcat worker threads, rendering the entire rest-java service unresponsive.

## Finding Description

**Three sequential blocking DB calls — confirmed:**

`TopicController.getTopic()` at lines 32–37 executes three sequential blocking calls:

```java
var topic     = topicService.findById(id.id());
var entity    = entityService.findById(id.id());
var customFee = customFeeService.findById(id.id());
``` [1](#0-0) 

Each call blocks the Tomcat worker thread until the database responds or a statement timeout fires. The `statementTimeout` value for rest-java was **not found** in the rest-java source tree — the claim of exactly 10,000 ms is unverified from the code, though the blocking behavior itself is confirmed. [2](#0-1) 

**No rate limiting in rest-java — confirmed:**

A search across the entire `rest-java/` source tree for `RateLimiter`, `bucket4j`, `throttle`, and `RequestRateLimiter` returns zero hits. This is in direct contrast to the `web3` module, which has a full `ThrottleConfiguration` using `bucket4j` with a default `requestsPerSecond` of 500. [3](#0-2) [4](#0-3) 

**No explicit HikariCP pool size — confirmed:**

A search for `maximumPoolSize` and `hikari` in `rest-java/` returns zero hits, meaning HikariCP defaults to 10 connections. Requests beyond 10 concurrent DB users queue inside HikariCP, extending thread hold time beyond any statement timeout. [5](#0-4) 

**Partial inaccuracy in the claim — Traefik circuit breaker exists at ingress level:**

The claim states "no circuit breaker." This is inaccurate at the ingress layer. A Traefik circuit breaker middleware is configured:

```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 3
      initialInterval: 100ms
``` [6](#0-5) 

However, this circuit breaker operates at the ingress/proxy level and does **not** prevent thread pool exhaustion inside the pod. It only trips after errors are already occurring. Worse, the `retry: attempts: 3` setting amplifies the attack — each original request can trigger up to 3 retries, multiplying the DB load threefold.

Additionally, the GCP backend policy has `maxRatePerEndpoint: 250` but is gated behind `global.gateway.enabled: false` (disabled by default), so it provides no protection in default deployments. [7](#0-6) [8](#0-7) 

## Impact Explanation

When Tomcat's worker thread pool (default 200) is saturated, Spring/Tomcat queues incoming connections up to `acceptCount` (default 100) and then refuses new TCP connections. All endpoints become unreachable. The Kubernetes readiness probe at `/actuator/health/readiness` with a 2-second timeout will fail, causing the pod to be removed from the load-balancer, potentially amplifying the outage across the replica set. [9](#0-8) 

The `RestJavaHighDBConnections` alert fires only after **5 minutes** sustained above 75% utilization — far too slow to prevent the outage. [5](#0-4) 

## Likelihood Explanation

The endpoint requires no authentication. Any internet-accessible deployment is reachable. The attack requires only a standard HTTP load tool. A network partition is not required; any transient DB slowdown is sufficient. The Traefik retry middleware (3 attempts) amplifies the attack automatically. The attack is repeatable and sustainable. [6](#0-5) 

## Recommendation

1. **Add application-level rate limiting** to rest-java, mirroring the `bucket4j`-based `ThrottleConfiguration` already present in the `web3` module. [10](#0-9) 

2. **Set an explicit HikariCP `maximumPoolSize`** in rest-java configuration, sized to match expected concurrency.

3. **Enable the GCP backend policy** `maxRatePerEndpoint` or configure an equivalent ingress-level rate limit that is active by default. [11](#0-10) 

4. **Remove or reduce the Traefik retry count** for this endpoint to avoid amplifying DB load under degradation. [12](#0-11) 

5. **Parallelize the three DB calls** in `TopicController.getTopic()` (e.g., using `CompletableFuture`) to reduce per-request thread hold time. [2](#0-1) 

## Proof of Concept

```bash
# Simulate DB degradation by introducing artificial latency, then flood the endpoint:
hey -n 10000 -c 250 https://<host>/api/v1/topics/0.0.1234
```

With 250 concurrent requests each holding a thread on 3 sequential blocking DB calls, and HikariCP limited to 10 connections by default, threads queue inside HikariCP far beyond any statement timeout. Within seconds, the Tomcat thread pool is exhausted. The readiness probe (2 s timeout) begins failing, and the pod is removed from the load balancer service. [9](#0-8)

        final var limit = Bandwidth.builder()

                .refillGreedy(rateLimit, Duration.ofSeconds(1))

    }
