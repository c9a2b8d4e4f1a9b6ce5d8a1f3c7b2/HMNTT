Audit Report

## Title
Unauthenticated `X-Real-IP` Header Spoofing in `TracingMiddleware` Enables Log Poisoning

## Summary
`getClientIpAddress` in `rosetta/app/middleware/trace.go` unconditionally trusts the client-supplied `X-Real-IP` and `X-Forwarded-For` headers without verifying the request originated from a trusted reverse proxy. Because port 5700 is bound directly to the host in `docker-compose.yml`, an attacker bypassing nginx can inject arbitrary IP addresses into the persistent access log.

## Finding Description
`getClientIpAddress` (lines 63–74 of `trace.go`) reads `X-Real-IP` first, then `X-Forwarded-For`, and only falls back to `r.RemoteAddr` if both are absent — with no check that the request arrived via a trusted upstream. [1](#0-0) 

`TracingMiddleware` embeds the resolved IP verbatim into a `log.Info` message for every non-internal API path. [2](#0-1) 

The nginx proxy does overwrite `X-Real-IP` with `$$remote_addr` for requests routed through port 8080. [3](#0-2) 

However, the rosetta container independently binds port 5700 to the host, so requests arriving directly on that port bypass nginx entirely. [4](#0-3) 

The test suite explicitly confirms the intended (but insecure) behavior: a client-supplied `X-Real-IP: 10.0.0.100` is accepted and logged without any proxy-origin check. [5](#0-4) 

**Failed assumption:** The code assumes `X-Real-IP` is always set by a trusted reverse proxy. This assumption breaks whenever port 5700 is reachable directly.

Note: the `docker-compose.yml` sets `deploy.replicas: 0` for the rosetta service, meaning it is disabled by default under Docker Swarm (`docker stack deploy`). However, under plain `docker-compose up` the `deploy` key is ignored and the container starts with port 5700 exposed. [6](#0-5) 

## Impact Explanation
Every Rosetta API call (construction, block, account, network endpoints) is logged at `INFO` level with the forged IP. An operator relying on these logs to audit which network participants are querying transaction construction or submission endpoints will see fabricated attribution. This constitutes **persistent plain text injection** into the audit log — permanently corrupting the only available record of API access patterns. No on-chain state or transaction data is altered; the impact is confined to the logging/audit layer.

## Likelihood Explanation
The precondition is network reachability to port 5700. In any deployment where the host is internet-facing and port 5700 is not firewalled (the default `docker-compose` configuration adds no firewall rules), any unauthenticated attacker can exploit this with a single HTTP request. No credentials, tokens, or special knowledge are required. The attack is trivially repeatable.

## Recommendation
1. **Restrict direct port exposure:** Remove or bind port 5700 to `127.0.0.1` (e.g., `127.0.0.1:5700:5700`) so only the nginx proxy can reach it.
2. **Validate proxy origin in `getClientIpAddress`:** Only trust `X-Real-IP` / `X-Forwarded-For` when `r.RemoteAddr` matches a configured set of trusted proxy CIDRs. Fall back to `r.RemoteAddr` for all other connections.
3. **Defense in depth:** Apply both mitigations; network-level restriction alone is insufficient if the proxy configuration changes.

## Proof of Concept
```bash
# Direct connection to port 5700, bypassing nginx
curl -H "X-Real-IP: 192.0.2.1" http://<host>:5700/network/list
```
The rosetta access log will record `192.0.2.1` as the client IP regardless of the actual source address, confirming the forged attribution is written persistently to the audit trail. [7](#0-6)

```go
func getClientIpAddress(r *http.Request) string {
	ipAddress := r.Header.Get(xRealIpHeader)

	if len(ipAddress) == 0 {
		ipAddress = r.Header.Get(xForwardedForHeader)
	}

	if len(ipAddress) == 0 {
		ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
	}

	return ipAddress
```
