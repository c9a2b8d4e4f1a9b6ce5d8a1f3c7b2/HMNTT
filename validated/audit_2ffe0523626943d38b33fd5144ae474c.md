# Audit Report

## Title
GraphQL `account` Balance Reflects Spurious `errata=DELETE` Transfers Due to Missing Errata Check in `SqlEntityListener.onCryptoTransfer`

## Summary
`SqlEntityListener.onCryptoTransfer()` unconditionally applies every crypto transfer amount to `entity.balance` when `trackBalance=true`, including transfers that `EntityRecordItemListener.insertTransferList()` has already flagged as `errata=DELETE`. Because the GraphQL `account` query reads `entity.balance` directly with no post-query errata correction, any caller can receive an incorrect balance for the 1,177 accounts affected by the "Failed Transfers in Record" mainnet errata (2019-09-14 – 2019-10-03).

## Finding Description

**Code path — confirmed line by line:**

**Step 1 — `insertTransferList()` marks transfers `errata=DELETE` but still forwards them:**

In `EntityRecordItemListener.java`, the `failedTransfer` flag is set for failed crypto-transfer transactions before 2020-01-01: [1](#0-0) 

When `failedTransfer=true` and the account amount is found in the transaction body, the transfer is marked `errata=DELETE`: [2](#0-1) 

Immediately after, the transfer — now carrying `errata=DELETE` — is unconditionally forwarded to the entity listener: [3](#0-2) 

**Step 2 — `SqlEntityListener.onCryptoTransfer()` applies the amount to `entity.balance` without inspecting `errata`:** [4](#0-3) 

There is no `cryptoTransfer.getErrata()` check anywhere in this method. The spurious amount is merged into the entity's balance unconditionally.

**Step 3 — GraphQL reads `entity.balance` directly:**

`EntityServiceImpl.getByIdAndType()` performs a plain `findById` with no errata adjustment: [5](#0-4) 

`AccountController.balance()` returns whatever `account.getBalance()` holds, which is mapped 1-to-1 from `entity.balance`: [6](#0-5) 

**Step 4 — The errata guard exists elsewhere but not in the streaming path:**

`InitializeEntityBalanceMigration` correctly excludes `errata=DELETE` transfers when initialising balances from snapshots: [7](#0-6) 

`ErrataMigration.doMigrate()` temporarily sets `trackBalance=false` to protect its own SQL updates: [8](#0-7) 

`BalanceReconciliationService` and `AccountBalanceRepository` (web3) both filter `errata <> 'DELETE'` in their queries: [9](#0-8) [10](#0-9) 

None of these guards are present in `SqlEntityListener.onCryptoTransfer()`.

**Root cause:** The `errata` field on the `CryptoTransfer` domain object is set correctly by `insertTransferList()`, but `SqlEntityListener` never reads it before updating `entity.balance`. The failed assumption is that marking a transfer `errata=DELETE` is sufficient to prevent it from affecting the denormalized balance column — it is not, because the balance update happens in the listener before the row is persisted.

## Impact Explanation

For a fresh mirror-node deployment streaming mainnet data from genesis, `InitializeEntityBalanceMigration` runs against an empty database and initialises nothing. The importer then streams the errata period (2019-09-14 – 2019-10-03) and `SqlEntityListener.onCryptoTransfer()` applies the spurious non-fee transfer amounts to `entity.balance` for all 1,177 affected transactions. Any unauthenticated caller of the public GraphQL endpoint receives the inflated or deflated balance directly from `entity.balance` with no further filtering. The errata documentation confirms the scope: [11](#0-10) 

## Likelihood Explanation

The condition is triggered on any mirror-node deployment that streams mainnet data from genesis with `trackBalance=true` (the default). The GraphQL query requires no authentication and is trivially constructed, as shown in the project's own smoke-test documentation: [12](#0-11) 

A second trigger exists for any future errata scenario where `insertTransferList` marks transfers `errata=DELETE` while `trackBalance=true`, since the guard is structurally absent from the streaming path.

## Recommendation

Add an errata guard at the top of `SqlEntityListener.onCryptoTransfer()` before the balance update:

```java
@Override
public void onCryptoTransfer(CryptoTransfer cryptoTransfer) throws ImporterException {
    if (entityProperties.getPersist().isTrackBalance()
            && cryptoTransfer.getErrata() != ErrataType.DELETE) {  // add this guard
        var entity = new Entity();
        entity.setId(cryptoTransfer.getEntityId());
        entity.setBalance(cryptoTransfer.getAmount());
        entity.setBalanceTimestamp(cryptoTransfer.getConsensusTimestamp());
        onEntity(entity);
    }
    context.add(cryptoTransfer);
}
```

This mirrors the guard already applied in `InitializeEntityBalanceMigration`, `BalanceReconciliationService`, `AccountBalanceRepository`, and `rosetta/app/persistence/account.go`.

## Proof of Concept

1. Start a mirror-node importer against a fresh PostgreSQL database with `hiero.mirror.importer.parser.record.entity.persist.trackBalance=true` (default).
2. Configure the importer to stream mainnet data from genesis.
3. Allow the importer to process records through the errata period (2019-09-14 – 2019-10-03).
4. Query the GraphQL endpoint for any of the 1,177 affected accounts:

```bash
curl -X POST http://localhost:8083/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query": "{account(input: {entityId: {shard: 0, realm: 0, num: <affected_account>}}) { balance }}"}'
```

5. Compare the returned `balance` against the value computed by `BalanceReconciliationService` (which correctly excludes `errata=DELETE` transfers). The values will diverge by the sum of the spurious transfer amounts for that account.

### Failed Transfers in Record

- Period: 2019-09-14 to 2019-10-03
- Scope: Affected the records of 1177 transactions.
- Problem: When a crypto transfer failed due to an insufficient account balance, the attempted transfers were

- Solution: Fixed in Consensus Node v0.4.0 late 2019. Fixed in Mirror Node in v0.53.0 by adding an `errata` field to

  omitted.
