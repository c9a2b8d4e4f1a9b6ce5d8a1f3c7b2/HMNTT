Audit Report

## Title
Gas Throttle Amplification via Binary Search in `estimateGas`: Up to 21x EVM Executions Per Single Throttle Token

## Summary
An unprivileged caller can send `eth_estimateGas` requests that trigger a binary search loop executing up to 20 additional full EVM calls inside `BinaryGasEstimator.search()`, on top of the initial `callContract()` invocation, while the `gasLimitBucket` is charged for only the gas consumed by the initial call. The `!estimate` guard in `doProcessCall()` that skips `restoreGasToBucket` for binary search iterations was intended to avoid double-accounting, but it also makes those iterations completely invisible to the throttle, amplifying actual EVM work by up to 21x per throttle token deducted.

## Finding Description

**Throttle consumption — once per HTTP request:**
`ThrottleManagerImpl.throttle()` consumes `scaleGas(request.getGas())` tokens from `gasLimitBucket` once at the request boundary. [1](#0-0) 

**`ContractExecutionService.processCall()` — estimate branch:**
When `params.isEstimate()` is `true`, `estimateGas(params, ctx)` is called. [2](#0-1) 

**`estimateGas()` — two-phase execution:**
Phase 1 calls `callContract(params, context)`, which internally calls `doProcessCall(params, params.getGas(), false)` — `estimate=false`, so `restoreGasToBucket` IS invoked and the net throttle cost is the gas actually used by this call. [3](#0-2) 

Phase 2 passes `gas -> doProcessCall(params, gas, true)` — `estimate=true` — to `binaryGasEstimator.search()` for every iteration. [4](#0-3) 

**`BinaryGasEstimator.search()` — up to `maxGasEstimateRetriesCount` iterations:**
The loop runs `while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount())`, executing a full EVM call each iteration via `safeCall`. The default `maxGasEstimateRetriesCount` is 20. [5](#0-4) [6](#0-5) 

**`doProcessCall()` — throttle accounting gated on `!estimate`:**
The `finally` block skips both `restoreGasToBucket` and `updateMetrics` when `estimate=true`. All 20 binary search iterations pass `estimate=true`, so no gas is restored and no additional tokens are consumed.

```java
} finally {
    if (!estimate) {
        restoreGasToBucket(result, params.getGas());
        // Only record metric if EVM is invoked and not inside estimate loop
        if (result != null) {
            updateMetrics(params, result.gasUsed(), 1, status);
        }
    }
}
``` [7](#0-6) 

**`gasLimitBucket` — single-dimension throttle:**
The bucket is filled at `gasPerSecond` (default 7.5 billion, scaled to 750,000 tokens by the `GAS_SCALE_FACTOR` of 10,000). It has no awareness of whether a request is an estimate that will multiply EVM work. [8](#0-7) [9](#0-8) 

**Root cause:** The throttle model assumes 1 request = 1 EVM execution. For estimate requests, the actual ratio is 1 request = 1 + up to 20 EVM executions. The `!estimate` guard that skips `restoreGasToBucket` was intended to avoid double-accounting gas restoration inside the search loop, but it also makes the binary search iterations completely invisible to the throttle.

## Impact Explanation

With `maxGasEstimateRetriesCount = 20` and `maxGasLimit = 15,000,000`:

- A normal `eth_call` consumes throttle tokens proportional to gas used → 1 EVM execution.
- An `eth_estimateGas` consumes throttle tokens proportional to gas used by the initial call → up to 21 EVM executions (1 initial + 20 binary search).

An attacker sending estimate requests at the maximum allowed rate saturates the node's EVM execution capacity at up to 21x the rate the throttle was designed to permit. This directly increases CPU, memory, and I/O consumption far beyond the throttle's intended ceiling. Under sustained attack, legitimate `eth_call` requests will be starved or severely delayed. [10](#0-9) 

## Likelihood Explanation

- **No authentication required.** `eth_estimateGas` is a standard, publicly accessible JSON-RPC endpoint.
- **No special contract needed.** Any deployed contract with a non-trivial function maximizes binary search iterations.
- **Trivially automatable.** A single script sending concurrent `eth_estimateGas` requests with `gas=15000000` is sufficient.
- **Repeatable indefinitely.** The throttle refills every second; the attacker simply re-sends at the refill rate. [11](#0-10) 

## Recommendation

Account for the binary search multiplier in the throttle. Two concrete approaches:

1. **Charge upfront for estimate multiplier:** In `ThrottleManagerImpl.throttle()`, when `request.isEstimate()` is `true`, multiply the consumed tokens by `maxGasEstimateRetriesCount + 1` (or a configurable factor) before calling `gasLimitBucket.tryConsume()`.

2. **Restore gas after each binary search iteration:** Remove the `!estimate` guard from the `restoreGasToBucket` call in `doProcessCall()`, or add a separate restoration path for estimate iterations that charges the actual gas used per iteration back against the bucket.

Either approach ensures the throttle accurately reflects the actual EVM work performed per request. [1](#0-0) [7](#0-6) 

## Proof of Concept

```python
import requests, concurrent.futures

NODE_URL = "http://<mirror-node>:8545"

# Deploy or use any existing contract with a non-trivial function
CONTRACT_ADDR = "0x<deployed_contract>"
CALL_DATA     = "0x<function_selector_with_args>"

payload = {
    "jsonrpc": "2.0",
    "method":  "eth_estimateGas",
    "params":  [{"to": CONTRACT_ADDR, "data": CALL_DATA, "gas": "0xE4E1C0"}],  # 15,000,000
    "id":      1
}

def send():
    return requests.post(NODE_URL, json=payload, timeout=30).json()

# Send at the maximum allowed rate (requestsPerSecond = 500)
with concurrent.futures.ThreadPoolExecutor(max_workers=50) as ex:
    futures = [ex.submit(send) for _ in range(500)]
    for f in concurrent.futures.as_completed(futures):
        print(f.result())
```

Each of the 500 requests triggers up to 21 EVM executions while the `gasLimitBucket` is charged for only 1. The node processes up to 10,500 EVM executions per second while the throttle believes it is processing 500. [5](#0-4) [12](#0-11)
