Audit Report

## Title
Soft-Deleted Entity Disclosure via Zero-Prefix EVM Address in GraphQL `getByEvmAddressAndType()`

## Summary
`EntityServiceImpl.getByEvmAddressAndType()` routes "long-zero" EVM addresses (first 12 bytes all zero) to `CrudRepository.findById()`, which carries no `deleted is not true` predicate. An unauthenticated caller can retrieve the full record of any soft-deleted entity by encoding its numeric ID into a crafted EVM address and querying the public GraphQL endpoint.

## Finding Description

The method at issue is `getByEvmAddressAndType()` in `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`: [1](#0-0) 

When the first 12 bytes of the decoded address are zero, the code calls `entityRepository.findById(buffer.getLong())` and applies only a type-equality filter — no deletion check.

By contrast, both custom repository queries explicitly guard against deleted records: [2](#0-1) 

`findByEvmAddress` and `findByAlias` both include `deleted is not true`, but `findById` is inherited from `CrudRepository<Entity, Long>` and issues a plain `SELECT … WHERE id = ?` with no deletion predicate. [3](#0-2) 

The same missing deletion filter is present in `getByIdAndType()`: [4](#0-3) 

This confirms the pattern is systemic across both ID-based lookup paths.

## Impact Explanation
The GraphQL endpoint returns the full entity record, including fields such as `key` (admin key), `memo`, `balance`, `alias`, `autoRenewAccount`, and `obtainer` for the matched entity. Soft-deleted entities represent accounts/contracts explicitly deleted on the Hedera network. Returning them violates the access-control invariant enforced by every other lookup path in the same repository and leaks historical key material, memo content, and balance data for entities that should no longer be visible.

## Likelihood Explanation
- The GraphQL endpoint (`/graphql/alpha`) requires no authentication.
- Hedera entity IDs are sequential integers, publicly observable from on-chain transaction history and other mirror node endpoints.
- Exploitation requires a single HTTP POST with a crafted 20-byte hex string — no special tooling or privileges needed.
- The zero-prefix ("long-zero") address format is a well-known Hedera convention, making the branch trivially discoverable.

## Recommendation
Add a `deleted is not true` guard to the `findById` branch. The cleanest fix is to add a dedicated repository method analogous to the existing ones:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Replace both `entityRepository.findById(...)` calls in `EntityServiceImpl` — in `getByIdAndType()` (line 25) and in the zero-prefix branch of `getByEvmAddressAndType()` (line 38) — with this new method.

## Proof of Concept

1. Identify a deleted entity's numeric ID (e.g., `12345` / `0x3039`) from public transaction history.
2. Construct the 20-byte long-zero EVM address: bytes 0–11 = `0x00`, bytes 12–19 = ID as big-endian `long`:
   ```
   0x000000000000000000000000000000000000003039
   ```
3. Send an unauthenticated GraphQL query:
   ```graphql
   query {
     account(input: { evmAddress: "000000000000000000000000000000000000003039" }) {
       deleted key memo balance alias
     }
   }
   ```
4. `getByEvmAddressAndType()` detects the zero prefix at line 37, calls `findById(12345)` at line 38, passes the type filter, and returns the full soft-deleted entity record including `key`, `memo`, and `balance`. [5](#0-4)
