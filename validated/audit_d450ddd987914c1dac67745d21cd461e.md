Audit Report

## Title
SSRF via Unguarded HTTP Redirect Following in `fetchMirrorNodeNetwork()`

## Summary
The `http.Client` constructed in `buildNetworkFromMirrorNodes()` has no `CheckRedirect` policy. Go's default behavior silently follows up to 10 consecutive 3xx redirects. A mirror REST server returning an HTTP 301/302 pointing to an internal address causes the pinger to issue a full HTTP GET to that internal target, constituting Server-Side Request Forgery.

## Finding Description
**Exact location:** `pinger/mirror_node_client.go`, `buildNetworkFromMirrorNodes()` line 46 and `fetchMirrorNodeNetwork()` lines 74–132.

At line 46, the HTTP client is constructed with only a timeout — no `CheckRedirect` field is set:

```go
httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}
``` [1](#0-0) 

At line 84, the request is executed with this client, which will transparently follow any 3xx redirect returned by the server:

```go
resp, err := httpClient.Do(req)
``` [2](#0-1) 

The status-code guard at lines 90–93 is evaluated only on the **final** response, after all redirects have already been followed and any internal service has already received the request:

```go
if resp.StatusCode < 200 || resp.StatusCode >= 300 {
    retry := resp.StatusCode == http.StatusTooManyRequests || resp.StatusCode >= 500
    return nil, retry, fmt.Errorf("GET %s returned %s", url, resp.Status)
}
``` [3](#0-2) 

If the internal service returns HTTP 200, this guard passes entirely and the response body is fed to the JSON decoder. The JSON decode failure at line 96–98 is the only thing that surfaces an error, but by then the internal request has already completed. [4](#0-3) 

This code path is only reached when `cfg.network == "other"`, which is the operator-supplied custom deployment path: [5](#0-4) 

## Impact Explanation
In a cloud-hosted deployment (AWS, GCP, Azure, Kubernetes), the pinger process has network access to link-local metadata services and internal cluster endpoints unreachable from the public internet. A successful redirect to `http://169.254.169.254/latest/meta-data/iam/security-credentials/<role>` (AWS IMDSv1, no token required) leaks temporary IAM credentials. Even without credential theft, the attacker gains the ability to probe and interact with any HTTP service reachable from the pinger's network namespace.

## Likelihood Explanation
The precondition — influencing what the mirror REST server returns — is achievable via: supply-chain compromise of a shared mirror node, DNS hijacking of the configured hostname, BGP/ARP poisoning on the network path, or operator misconfiguration pointing at an attacker-controlled URL. The `network=other` code path is explicitly designed for custom/private deployments where the mirror URL is operator-supplied, broadening the attack surface. The exploit requires no authentication and is repeatable on every startup or retry cycle.

## Recommendation
Set a `CheckRedirect` policy on the `http.Client` that rejects all redirects (or at minimum rejects redirects to RFC-1918 / link-local addresses):

```go
httpClient := &http.Client{
    Timeout: cfg.mirrorNodeClientTimeout,
    CheckRedirect: func(req *http.Request, via []*http.Request) error {
        return http.ErrUseLastResponse // never follow redirects
    },
}
```

This causes `httpClient.Do` to return the 3xx response directly rather than following it, and the existing status-code guard at lines 90–93 will then correctly reject it as a non-2xx response. [1](#0-0) 

## Proof of Concept
1. Stand up an HTTP server that responds to `GET /api/v1/network/nodes` with:
   ```
   HTTP/1.1 301 Moved Permanently
   Location: http://169.254.169.254/latest/meta-data/iam/security-credentials/
   ```
2. Configure the pinger with `HIERO_MIRROR_PINGER_REST=http://<attacker-server>` and `network=other`.
3. Start the pinger. `buildNetworkFromMirrorNodes` constructs the unguarded `http.Client` at line 46 and calls `fetchMirrorNodeNetwork`.
4. `httpClient.Do(req)` at line 84 follows the 301; the pinger issues `GET http://169.254.169.254/latest/meta-data/iam/security-credentials/` from within the cloud instance.
5. The metadata service returns HTTP 200 with a JSON-like body; the status guard at line 90 passes.
6. The JSON decoder at line 96 fails on the non-conforming body, but the internal request has already completed and any credentials in the response body have been processed by the Go runtime.
