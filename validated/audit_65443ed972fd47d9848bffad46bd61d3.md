Audit Report

## Title
Integer Narrowing Cast Overflow in `PollingTopicListener.poll()` Bypasses `maxPageSize` Cap, Enabling Unlimited DB Queries

## Summary
In `PollingTopicListener.poll()`, a narrowing cast from `long` to `int` at line 55 overflows when `filter.getLimit()` is `Long.MAX_VALUE`. The resulting value of `-1` causes `Math.min(-1, maxPageSize)` to return `-1`, which is then stored as the `limit` in a rebuilt `TopicMessageFilter`. Because `hasLimit()` returns `false` for `-1`, `setMaxResults()` is never called in the repository, and the JPA query returns all matching rows on every poll cycle.

## Finding Description

**Narrowing cast — `PollingTopicListener.java` lines 54–56:** [1](#0-0) 

`filter.getLimit()` is a primitive `long` and `context.getCount()` is an `AtomicLong`. The subtraction is a `long` operation, but the result is immediately narrowed to `int`. Java's narrowing primitive conversion discards the upper 32 bits:

```
Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF
lower 32 bits  = 0xFFFFFFFF  →  (int) = -1
```

**`Math.min` does not save it — line 57:** [2](#0-1) 

`Math.min(-1, 5000)` returns `-1`. The `maxPageSize` cap (default 5000) is bypassed entirely.

**`hasLimit()` returns `false` for `-1` — `TopicMessageFilter.java` lines 39–41:** [3](#0-2) 

`-1 > 0` is `false`.

**`setMaxResults()` is never called — `TopicMessageRepositoryCustomImpl.java` lines 51–53:** [4](#0-3) 

Without `setMaxResults()`, the JPA query has no row limit.

**Full result set is materialized — line 60:** [5](#0-4) 

`getResultList()` loads the entire result into heap before returning.

**Why `@Min(0)` does not block the initial input:** [6](#0-5) 

`@Min(0)` only rejects values `< 0`. `Long.MAX_VALUE` is positive and passes validation. The overflow occurs *after* validation, inside `poll()`.

**Why `@Min(0)` does not block the internally-built filter with `limit = -1`:** [7](#0-6) 

Lombok's `@Builder` does not invoke Bean Validation. `filter.toBuilder().limit(-1).build()` constructs the object directly without triggering the `@Min(0)` constraint.

**Call chain from user input:**

1. User sends gRPC `subscribeTopic` with `limit = Long.MAX_VALUE`.
2. `TopicMessageServiceImpl.subscribeTopic()` validates the filter — passes (`Long.MAX_VALUE > 0`).
3. `incomingMessages()` computes `limit = Long.MAX_VALUE - 0 = Long.MAX_VALUE` and calls `topicListener.listen(newFilter)`. [8](#0-7) 
4. `PollingTopicListener.listen()` schedules repeated calls to `poll()` every 500 ms. [9](#0-8) 
5. `poll()` computes `(int)(Long.MAX_VALUE) = -1`; `pageSize = -1`.
6. `findByFilter()` receives `limit = -1`; `hasLimit()` is `false`; no `setMaxResults()`; full table scan per poll.

**Note on `flux.take(Long.MAX_VALUE)` in `subscribeTopic()`:** [10](#0-9) 

This limits what is *emitted to the subscriber*, but does not prevent the DB from materializing the entire result set via `getResultList()` before the reactive pipeline applies the `take`. The heap and DB load occur regardless.

## Impact Explanation

Every 500 ms, the attacker's subscription triggers an unbounded `SELECT` against the `topic_message` table for the chosen topic. On a production node with millions of messages, each query can return gigabytes of data, consuming:

- Database CPU and I/O (full index scan or sequential scan)
- JPA/Hibernate heap (entire result list materialized via `getResultList()`)
- Network bandwidth between DB and mirror node

Multiple concurrent subscriptions multiply the effect linearly. Because the `RepeatSpec.times(Long.MAX_VALUE)` loop persists indefinitely, a single connection sustains the attack without re-authentication. This constitutes a practical denial-of-service against the database and the mirror node JVM.

## Likelihood Explanation

**Scope condition:** `PollingTopicListener` is only active when the deployment is explicitly configured with `type = POLL`. [11](#0-10) 

The default listener type is `REDIS`, so deployments using the default configuration are not affected. However, deployments using `POLL` or `SHARED_POLL` (which may also delegate to polling logic) are fully exposed. The gRPC `subscribeTopic` endpoint requires no authentication beyond a valid topic ID (public on-chain data). The exploit requires only setting `limit = Long.MAX_VALUE` in a standard `ConsensusTopicQuery` protobuf message, reproducible with any gRPC client.

## Recommendation

Replace the narrowing cast with a bounds-safe computation in `PollingTopicListener.poll()`:

```java
// Before (vulnerable):
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;

// After (safe):
int limit = filter.hasLimit()
        ? (int) Math.min(filter.getLimit() - context.getCount().get(), Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
```

`Math.min(long, long)` clamps the result to `Integer.MAX_VALUE` before the cast, preventing negative overflow. Additionally, add a guard `if (limit <= 0) return Flux.empty();` immediately after the computation to terminate the poll when the remaining limit is exhausted or invalid.

## Proof of Concept

```java
// gRPC client pseudocode
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setLimit(Long.MAX_VALUE)   // passes @Min(0) validation
    .build();

// Subscribe — triggers PollingTopicListener.poll() every 500ms
// Each poll executes: SELECT * FROM topic_message WHERE topic_id = 1
//                     AND consensus_timestamp >= <startTime>
//                     ORDER BY consensus_timestamp ASC
// with NO setMaxResults() — full table scan, repeated indefinitely
stub.subscribeTopic(query, responseObserver);
```
