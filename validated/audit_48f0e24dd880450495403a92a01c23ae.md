**Audit Report**

## Title
Integer Overflow in `EntityIdRangeParameter.getInclusiveValue()` / `Bound.getAdjustedLowerRangeValue()` Causes Filter Bypass via Crafted GT/LT Entity ID

## Summary
Three unchecked Java `long` arithmetic operations in the REST Java layer allow a crafted entity ID at the encoding boundary (`511.65535.274877906943` → `Long.MAX_VALUE`) to silently overflow when a GT/LT operator is applied. The `Bound` constructor's range-validity check does not detect the overflow, so the resulting query bound is `Long.MIN_VALUE` instead of an out-of-range sentinel, causing the `account.id` filter on the NFT allowances endpoint to match every row.

## Finding Description

**Overflow sites (all confirmed, no overflow guard present):**

1. `EntityIdRangeParameter.getInclusiveValue()` — `value + 1` (GT) / `value - 1` (LT): [1](#0-0) 

2. `Bound.getAdjustedLowerRangeValue()` — `lowerBound++` (GT): [2](#0-1) 

3. `Bound.getInclusiveRangeValue()` — `value += 1L` (GT): [3](#0-2) 

**Encoding boundary (confirmed by unit tests):**

`EntityId.encode()` uses a 10-bit shard / 16-bit realm / 38-bit num layout. The maximum valid entity ID `511.65535.274877906943` encodes to exactly `Long.MAX_VALUE` (9223372036854775807), and `512.0.0` encodes to `Long.MIN_VALUE` (-9223372036854775808). Both pass all validation in `encode()` because 511 ≤ SHARD_MASK (1023), 65535 ≤ REALM_MASK (65535), 274877906943 ≤ NUM_MASK (274877906943): [4](#0-3) 

Confirmed by the Go encoding test: [5](#0-4) 

And the JavaScript encoding test: [6](#0-5) 

**Failed guard in `Bound` constructor:**

The constructor calls `getAdjustedLowerRangeValue()` (which overflows to `Long.MIN_VALUE` for GT + `Long.MAX_VALUE` input), then checks `adjustedLower > adjustedUpper`. With no upper bound supplied, `adjustedUpper = Long.MAX_VALUE`. The comparison `Long.MIN_VALUE > Long.MAX_VALUE` is `false`, so no exception is thrown: [7](#0-6) 

**Controller wires `primarySortField = true`:**

`AllowancesController.getNftAllowances()` constructs the `ownerOrSpenderIds` bound with `primarySortField = true`, meaning the guard is active but still bypassed: [8](#0-7) 

**GT exploit flow:**
1. `GET /api/v1/accounts/0.0.1000/allowances/nfts?account.id=gt:511.65535.274877906943`
2. `getEntityId("511.65535.274877906943")` → `EntityId.of(511, 65535, 274877906943)` → `id = Long.MAX_VALUE`. Passes all validation.
3. `EntityIdRangeParameter(GT, Long.MAX_VALUE)` stored.
4. `Bound` constructor calls `getAdjustedLowerRangeValue()` → `Long.MAX_VALUE + 1 = Long.MIN_VALUE` (silent overflow).
5. Check: `Long.MIN_VALUE > Long.MAX_VALUE` → `false` → no exception.
6. Query executes with `WHERE spender_id >= -9223372036854775808`, matching every row.

**LT mirror case:** `account.id=lt:512.0.0` → `value = Long.MIN_VALUE` → `value - 1 = Long.MAX_VALUE` → `WHERE spender_id <= 9223372036854775807`, matching every row.

## Impact Explanation
The `account.id` filter on `GET /api/v1/accounts/{id}/allowances/nfts` is completely bypassed: the query returns all NFT allowances for the target account regardless of the intended spender/owner filter. Because this is a public read API, the underlying data is already accessible without any filter, so there is no direct privilege escalation or confidential data exposure. However, the corrupted bounds also feed into pagination cursor generation (`links.next` is derived from the last returned item), which can produce incorrect cursor values and cause clients to enter infinite or incorrect pagination loops. Severity: **Medium** — logic correctness violation with filter bypass and pagination corruption on a public endpoint.

## Likelihood Explanation
Any unauthenticated HTTP client can trigger this with a single crafted GET request. The entity ID `511.65535.274877906943` is syntactically valid and passes all input validation in `EntityIdRangeParameter.getEntityId()` and `EntityId.encode()`. No special privileges, tokens, or prior knowledge of system state are required. The attack is trivially repeatable.

## Recommendation
1. **Guard `getInclusiveValue()` in `EntityIdRangeParameter`:** Replace `value + 1` / `value - 1` with `Math.addExact(value, 1)` / `Math.subtractExact(value, 1)` and catch `ArithmeticException`, converting it to `IllegalArgumentException("Entity ID out of range for operator")`.
2. **Guard `Bound.getAdjustedLowerRangeValue()` and `Bound.getInclusiveRangeValue()`:** Apply the same `Math.addExact`/`Math.subtractExact` pattern.
3. **Alternatively, reject boundary-value inputs at parse time:** In `EntityIdRangeParameter.valueOf()`, after encoding, check that a GT operator is not paired with `Long.MAX_VALUE` and an LT operator is not paired with `Long.MIN_VALUE`, throwing `IllegalArgumentException` immediately.

## Proof of Concept
```
# GT overflow — filter matches all rows
curl "https://<mirror-node>/api/v1/accounts/0.0.1000/allowances/nfts?account.id=gt:511.65535.274877906943"

# LT overflow — filter matches all rows
curl "https://<mirror-node>/api/v1/accounts/0.0.1000/allowances/nfts?account.id=lt:512.0.0"
```

Expected (correct) behavior: `400 Bad Request` — the adjusted bound is out of the valid entity ID range.

Actual behavior: `200 OK` with all NFT allowances for the account, ignoring the filter. The `links.next` cursor in the response will contain a corrupted `account.id` value derived from the overflowed bound.
