Audit Report

## Title
Unbounded Metric Label Cardinality via User-Controlled Path in `metricsMiddleware()`

## Summary
`toOpenApiPath()` in `rest/middleware/metricsHandler.js` falls back to the raw, attacker-controlled `req.path` for every unmatched route. This value is used as the `path` label on four per-route OpenTelemetry instruments. An unauthenticated attacker can permanently exhaust the SDK's default cardinality limit (~2000 per instrument) with as few as ~2000 unique-path requests, corrupting all per-route metrics for the lifetime of the process.

## Finding Description
In `toOpenApiPath()`, when `res.locals[requestPathLabel]` is unset and `req.route` is `undefined` (every 404 request), the function assigns the raw request path directly:

```js
// metricsHandler.js lines 16-21
if (!path) {
  if (!req.route) {
    path = req.path;   // ← raw, attacker-controlled value
  } else {
    path = (req.baseUrl ?? '') + req.route?.path;
  }
}
``` [1](#0-0) 

`res.locals[requestPathLabel]` is only written by matched route handlers. For unmatched routes, `responseHandler.js` throws `NotFoundError` immediately, before any label is stored: [2](#0-1) 

The raw path then flows into all four per-route instruments: [3](#0-2) 

The `MeterProvider` is constructed with no `cardinalityLimit` override, leaving the SDK default (2000 per instrument) as the only backstop: [4](#0-3) 

**Root cause / failed assumption:** `toOpenApiPath()` assumes it always returns a bounded, template-style path (e.g., `/api/v1/transactions/{id}`). This holds only for matched routes. For unmatched routes it silently returns raw user input, creating an unbounded label space.

## Impact Explanation
- **Metrics corruption:** Once the cardinality limit is saturated with attacker-generated paths, legitimate route labels (e.g., `/api/v1/network/fees`) are silently dropped into the overflow bucket, making per-route latency and error-rate data invisible to operators.
- **Memory pressure:** Each unique label set allocates histogram bucket arrays retained for the lifetime of the `MeterProvider`. With four instruments × 2000 sets each, heap growth is measurable.
- **Persistent effect:** `initMetrics()` is called once at startup and never reset; poisoned label sets persist until process restart.
- **Monitoring blindness:** Operators relying on `durationHistogram` to detect fee-API degradation lose visibility exactly when an attack is in progress.

## Likelihood Explanation
- No authentication is required; any external client can send HTTP requests to arbitrary paths.
- Only ~2000 requests per instrument are needed to saturate the limit — trivially automatable with a simple loop generating unique path segments.
- Each request returns a 404 quickly, so the attacker's cost is negligible.
- The effect is permanent until process restart, requiring no sustained traffic.

## Recommendation
1. **Normalize unmatched paths:** In `toOpenApiPath()`, replace the `path = req.path` fallback with a fixed sentinel string such as `"unmatched"` or `"unknown"`:
   ```js
   if (!req.route) {
     path = 'unmatched';  // never use req.path
   }
   ```
2. **Set an explicit cardinality limit** on the `MeterProvider` to enforce a hard cap regardless of label values.
3. **Optionally**, apply a rate-limit or path-length cap on incoming requests before they reach the metrics middleware.

## Proof of Concept
```bash
# Saturate cardinality limit with ~2001 unique paths
for i in $(seq 1 2001); do
  curl -s "http://<mirror-node-host>/api/v1/x/UNIQUE${i}" > /dev/null
done
# After this, scrape /metrics and observe that legitimate route labels
# (e.g., api_request_duration_milliseconds{path="/api/v1/network/fees",...})
# are absent or collapsed into the overflow bucket.
```
