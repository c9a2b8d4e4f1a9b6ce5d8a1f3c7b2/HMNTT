# Audit Report

## Title
Unbounded In-Memory Opcode Accumulation in EVM Replay Enables JVM Heap Exhaustion via `getContractOpcodes()`

## Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint replays a historical EVM transaction and accumulates every executed opcode as a heap-allocated `Opcode` object in an unbounded `ArrayList`. The per-request throttle enforces only a request-rate limit (one token consumed regardless of transaction gas), so a single request referencing a maximum-gas transaction can force the JVM to allocate hundreds of millions of bytes of opcode trace data, exhausting heap and crashing the node.

## Finding Description

**Entry point — `OpcodesController.getContractOpcodes()`**

The only guards applied before delegating to `opcodeService.processOpcodeCall()` are a feature-flag check, a gzip header validation, and `throttleManager.throttleOpcodeRequest()`. No gas-aware limit is applied. [1](#0-0) 

**Throttle is request-rate only — `ThrottleManagerImpl.throttleOpcodeRequest()`**

`throttleOpcodeRequest()` consumes exactly one token from `opcodeRateLimitBucket` (default: 1 req/s) regardless of the gas limit of the replayed transaction. The `gasLimitBucket` used by the regular `throttle()` path is never consulted for opcode requests. [2](#0-1) 

**Unbounded accumulation — `OpcodeContext`**

`opcodes` is a plain `ArrayList` with no size cap. `addOpcodes()` appends unconditionally. [3](#0-2) 

**Called once per EVM opcode — `OpcodeActionTracer.tracePostExecution()`**

Every opcode executed during the replay appends a new `Opcode` object. When `stack=true` (the default) or `memory=true`, each object also carries a full snapshot of the EVM stack/memory at that point. [4](#0-3) 

**Initial capacity hint does not cap growth — `OpcodeServiceImpl.processOpcodeCall()`**

`gas / 3` is only the initial `ArrayList` capacity hint; the list grows without bound beyond it. [5](#0-4) 

**Root cause / failed assumption**: The design assumes the response size is bounded by gzip compression and that a rate limit is sufficient. Neither is true: gzip compresses the serialised response but the in-memory `List<Opcode>` is fully materialised before serialisation; and the rate limit does not account for the per-request work proportional to gas consumed.

## Impact Explanation
Hedera's maximum transaction gas is 15,000,000. With cheap opcodes (JUMPDEST = 1 gas, ADD = 3 gas) a single transaction can execute ~5–15 million opcodes. With `stack=true` (the default), each `Opcode` object holds a `List<String>` of 32-byte hex stack items. A conservative estimate of 500 bytes per entry × 5 million entries = ~2.5 GB of heap per request. A single such request can trigger an `OutOfMemoryError`, crashing the JVM and taking the entire web3 mirror-node service offline. Concurrent requests (permitted at the configured rate) multiply the effect.

## Likelihood Explanation
**Precondition**: The opcode tracer must be enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`; it defaults to `false`). [6](#0-5) 

When enabled for production debugging or public use, any unauthenticated caller who knows (or can enumerate) the hash of any high-gas historical transaction can trigger the attack. No special privileges, no on-chain action, and no brute force are required — only a valid transaction hash and the `Accept-Encoding: gzip` header. The attack is repeatable at the configured request rate (default: 1 req/s). [7](#0-6) 

## Recommendation
1. **Enforce a hard opcode-count cap**: In `OpcodeContext.addOpcodes()`, reject or truncate once `opcodes.size()` exceeds a configurable maximum (e.g., 100,000 entries), throwing a descriptive error to the caller.
2. **Gas-aware throttling for opcode requests**: In `ThrottleManagerImpl.throttleOpcodeRequest()`, also consume tokens from `gasLimitBucket` proportional to the gas limit of the replayed transaction, consistent with how `throttle()` handles regular contract calls.
3. **Streaming / lazy serialisation**: Instead of materialising the full `List<Opcode>` in memory before serialisation, stream opcode objects directly to the HTTP response writer.
4. **Configurable response size limit**: Add a maximum response byte-size guard (e.g., via a `ContentCachingResponseWrapper` or a custom `HttpMessageConverter`) that aborts serialisation once a threshold is reached.

## Proof of Concept
1. Deploy or identify any historical Hedera transaction that consumed close to the 15,000,000 gas limit.
2. Enable the opcode tracer: `hiero.mirror.web3.opcode.tracer.enabled=true`.
3. Send the following request (the `Accept-Encoding: gzip` header is required by the controller):
```
GET /api/v1/contracts/results/<HIGH_GAS_TX_HASH>/opcodes?stack=true
Accept-Encoding: gzip
```
4. Observe JVM heap usage climbing rapidly during EVM replay as `OpcodeContext.addOpcodes()` is called millions of times without bound.
5. The JVM throws `OutOfMemoryError`, crashing the web3 mirror-node service process.
