# Audit Report

## Title
Unbounded `keys` Collection in `getHookStorageChange` Causes IN-Clause Explosion and Database DoS

## Summary
`HookServiceImpl.getHookStorageChange()` forwards the full, user-supplied `keys` collection — with no size cap — directly to a native SQL query that expands it into one bind parameter per element (`key in (?,?,?,...)`). A single unauthenticated HTTP request with thousands of `key=` parameters generates an arbitrarily large SQL statement, exhausting database parse, plan, and memory resources.

## Finding Description

**1. No size constraint on `keys` in `HookStorageRequest`**

The `keys` field carries no `@Size`, `@Max`, or any other validation annotation: [1](#0-0) 

**2. `getHookStorageChange()` passes the raw `keys` collection, not `keysInRange`**

The guard at line 93 only short-circuits when *all* supplied keys fall outside the range (`keysInRange.isEmpty() && requestHasKeys`). If even one key is in range, execution falls through to line 103–105, where the **full, unfiltered `keys`** collection is forwarded to the repository — not the already-filtered `keysInRange`: [2](#0-1) 

This is a direct inconsistency with the non-historical path in `getHookStorage()`, which correctly passes `keysInRange`: [3](#0-2) 

**3. Native SQL query expands the collection unboundedly**

`HookStorageChangeRepository.findByKeyInAndTimestampBetween()` uses a native query with `key in (:keys)`. Spring Data JPA expands a `Collection` parameter in a native query into one bind placeholder per element: [4](#0-3) 

**Root cause / failed assumption:** The developer likely assumed the `limit` field (defaulting to 25) would bound query cost. It does not — `limit` controls result-set size via `Pageable`, not the size of the IN clause used to filter rows. The `keysInRange` filtering exists but is bypassed in the historical path.

## Impact Explanation
A single HTTP request with thousands of `key=<hex>` query parameters causes the database to parse, plan, and execute a query with a proportionally large IN clause. Repeated requests can saturate database CPU, memory, and connection pool resources, making the mirror-node REST API unavailable to all users. The historical (timestamp-filtered) code path is the trigger, so any endpoint accepting both `timestamp` and `key` parameters is affected.

## Likelihood Explanation
The mirror-node REST API is a public read API requiring no authentication. The attack is trivially scriptable with a single `curl` or Python `requests` call generating thousands of `key=` parameters. It is repeatable at will and requires no special knowledge beyond the public API schema.

## Recommendation
1. **Fix the immediate bug:** Pass `keysInRange` instead of `keys` at line 104 of `HookServiceImpl.java`, mirroring the non-historical path.
2. **Add an input size cap:** Annotate the `keys` field in `HookStorageRequest` with `@Size(max = N)` (e.g., `max = 100`) to enforce an upper bound at the DTO layer.
3. **Consider a hard limit in the repository layer** as a defense-in-depth measure.

## Proof of Concept
```bash
# Generate a request with 5000 distinct key= parameters
python3 -c "
import requests, os
keys = '&'.join(f'key={os.urandom(32).hex()}' for _ in range(5000))
url = 'http://<mirror-node>/api/v1/hooks/<ownerId>/<hookId>/storage'
r = requests.get(f'{url}?timestamp=gte:0&{keys}')
print(r.status_code, r.elapsed)
"
```
This generates a SQL statement with 5000 IN-clause entries. Repeated invocations will exhaust database resources.
