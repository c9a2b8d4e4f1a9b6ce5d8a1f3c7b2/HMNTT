# Audit Report

## Title
Cache-Amplified Database Query Inflation via Unbounded Slot Key Accumulation in `findStorageBatch()`

## Summary
`ContractStateServiceImpl.findStorageBatch()` inserts every requested slot key into a per-contract Caffeine cache (`contractSlotsCache`) without first verifying that the key exists in the contract. It then issues a single `IN`-clause DB query containing **all** keys ever accumulated in that cache. Because non-existent slot keys are never written to the short-lived `contractStateCache`, every subsequent request for a non-existent key bypasses the fast-path check and re-triggers the full batch query indefinitely, for the entire 5-minute `expireAfterAccess` lifetime of the per-contract cache. An unprivileged caller can pre-seed the cache with up to 1,500 distinct non-existent slot keys, after which every single `eth_call` touching that contract issues a 1,500-entry `IN`-clause against `contract_state`.

## Finding Description

**Code path:**

`findStorage()` checks `contractStateCache` first (line 63). On a miss it calls `findStorageBatch()` (line 69). [10](#0-9) 

Inside `findStorageBatch()`:
- Line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — inserts the incoming key into the per-contract slot cache with no existence check.
- Line 91: snapshots the **entire** key-set of that cache.
- Line 103: passes the full key-set to `contractStateRepository.findStorageBatch()`. [11](#0-10) 

The repository executes an unbounded `slot IN (:slots)` query with no server-side size guard: [6](#0-5) 

**Why non-existent keys create permanent amplification:**

`contractStateCache.put()` is called only inside the loop over DB-returned results (line 109). Non-existent slot keys are never returned by the DB, so they are never written to `contractStateCache`. [5](#0-4) 

Therefore, for a non-existent slot key:
1. `contractStateCache.get(...)` always returns `null` → `findStorageBatch()` is always invoked.
2. The key is already in `contractSlotsCache` → the full accumulated batch is always sent to the DB.
3. No cache expiry is needed to sustain the amplification; it persists for the full 5-minute `expireAfterAccess` window of the per-contract cache.

**Cache size bounds (confirmed):**
- `slotsPerContract` (per-contract slot key cache): `expireAfterAccess=5m, maximumSize=1500` — this is the actual ceiling for the `IN`-clause.
- `contractSlots` (outer cache, maps `contractId → per-contract CaffeineCache`): `expireAfterAccess=5m, maximumSize=3000` — this is the number of distinct contracts tracked, not slots per contract.
- `contractState` (slot value cache): `expireAfterWrite=2s, maximumSize=25000`. [12](#0-11) 

## Impact Explanation
After the one-time seeding phase, every `eth_call` or `eth_estimateGas` request that touches any slot of the poisoned contract causes the database to evaluate a `WHERE slot IN (1500 entries)` predicate against the `contract_state` table. At the default 500 req/s rate limit this translates to up to 750,000 slot comparisons per second from a single attacker, a 1,500× amplification of DB I/O per request compared to a clean workload. Sustained over the 5-minute `expireAfterAccess` window this can saturate DB CPU and I/O, degrading or denying service to legitimate users. The `db.statementTimeout` provides a partial backstop but does not prevent the amplification from accumulating across concurrent requests.

## Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special tooling — only the ability to call `POST /api/v1/contracts/call`, the standard public JSON-RPC endpoint. The seeding phase (1,500 calls with distinct non-existent slot keys) is achievable within the default rate limit. The attacker can target any high-traffic contract to maximize collateral impact. The non-existent-key variant requires no timing coordination and no waiting for `contractStateCache` expiry. The attack is fully repeatable after the 5-minute window expires.

## Recommendation
1. **Validate before inserting:** Before calling `contractSlotsCache.putIfAbsent()`, check whether the key exists in the contract (e.g., via a lightweight existence query or by only inserting keys that the DB confirms exist).
2. **Cap the batch size:** Enforce a hard upper bound on the number of keys passed to `findStorageBatch()` (e.g., reject or truncate the list if it exceeds a safe threshold such as 100–200 entries).
3. **Write sentinel for non-existent keys:** After a batch query returns no result for a requested key, write a sentinel value (e.g., `EMPTY_VALUE`) into `contractStateCache` for that key with a short TTL, so subsequent requests for the same non-existent key are served from the fast-path cache and do not re-trigger the batch query.
4. **Separate caches for existent vs. non-existent keys:** Maintain a small, short-lived negative-result cache for slot keys confirmed absent, preventing them from accumulating in `contractSlotsCache`.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
