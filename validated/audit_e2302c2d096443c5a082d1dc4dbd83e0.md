## Audit Report

## Title
Unauthenticated Unbounded Pagination DoS via Absent Application-Level Rate Limiting on `/api/v1/blocks`

## Summary
The `/api/v1/blocks` endpoint accepts unauthenticated requests with `limit=100` and issues unbounded paginated SQL queries against the `record_file` table. No application-level rate limiting exists in the Express middleware stack, and the Helm chart for the REST service neither enables Traefik middleware by default nor defines a `rateLimit` middleware entry (unlike the Rosetta chart, which does). A single attacker can sustain hundreds of requests per second, exhausting the database connection pool.

## Finding Description

**1. `getBlocks()` handler — confirmed code path**

`blockController.js` `getBlocks()` at lines 101–112 calls `extractSqlFromBlockFilters()` then `RecordFileService.getBlocks()` with no rate-check interposed: [1](#0-0) 

**2. `extractLimitFromFilters()` — limit=100 passes through unchanged**

Lines 57–61 implement the following logic: when `limit.value > maxLimit` the function falls back to `defaultLimit` (25); when `limit.value <= maxLimit` it returns `limit.value` unchanged. Because `100 > 100` is false, `limit=100` is accepted as-is and reaches the query layer: [2](#0-1) 

Note: the fallback-to-`defaultLimit` branch is actually *more* restrictive than capping at `maxLimit`, so it does not amplify the attack — but it does not prevent it either.

**3. `RecordFileService.getBlocks()` — limit interpolated directly into SQL**

Lines 149–162 build the query by string-interpolating `filters.limit` (a validated integer, so no SQL injection) with no secondary guard: [3](#0-2) 

**4. `generateNextLink()` — self-describing cursor enables trivial scripting**

Lines 90–99 always emit a `next` link anchored to `last(blocks).index`, making full-table traversal with `order=asc` trivially scriptable: [4](#0-3) 

**5. No rate-limiting middleware in `server.js`**

The Express middleware stack registers `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, and an optional `responseCacheCheckHandler`. No rate-limiting middleware is present for any route, including `/api/v1/blocks`: [5](#0-4) 

**6. `ThrottleManagerImpl` is scoped exclusively to web3 `/contracts/call`**

The only in-process throttle in the codebase lives in the web3 Java module and operates on `ContractCallRequest` objects; it has no effect on the REST Node.js service: [6](#0-5) 

**7. Helm chart: `global.middleware: false` by default; REST middleware has no `rateLimit` entry**

`global.middleware: false` means the Traefik `Middleware` CRD is never created: [7](#0-6) 

Even if `global.middleware` were set to `true`, the REST chart's `middleware` block defines only `circuitBreaker` and `retry` — no `rateLimit`: [8](#0-7) 

By contrast, the Rosetta chart explicitly includes a `rateLimit` entry (`average: 10`), confirming the omission in the REST chart is not intentional parity: [9](#0-8) 

**8. Confirmed defaults**

`responseLimit.default = 25`, `responseLimit.max = 100`: [10](#0-9) 

## Impact Explanation
Each request with `limit=100` causes the database to execute `SELECT … FROM record_file ORDER BY consensus_end ASC LIMIT 100` (or with a `WHERE index > N` cursor). While individual queries are index-backed and fast, a single attacker issuing hundreds of such requests per second saturates the database connection pool, increases I/O wait, and degrades or denies service for all other API consumers. The `record_file` table on mainnet contains tens of millions of rows; full traversal via pagination is trivially scriptable. No credentials, tokens, or special network position are required.

## Likelihood Explanation
The endpoint is publicly documented, requires zero authentication, and the pagination cursor is self-describing in the `next` link. A single HTTP client in a loop (`curl`, Python `requests`, etc.) can sustain hundreds of requests per second. The attack is repeatable, requires no prior knowledge of the system, and leaves no persistent state that would block re-entry. The default Helm deployment provides no rate-limiting protection at any layer.

## Recommendation
1. **Application layer:** Add a rate-limiting middleware (e.g., `express-rate-limit`) to `server.js` before the route handlers, keyed on client IP (respecting `X-Forwarded-For` via `trust proxy`).
2. **Infrastructure layer:** Add a `rateLimit` entry to the REST Helm chart's `middleware` block (mirroring the Rosetta chart) and set `global.middleware: true` in the default values, or document clearly that operators must enable it.
3. **Limit logic:** Fix `extractLimitFromFilters` to cap at `maxLimit` rather than fall back to `defaultLimit` when `limit.value > maxLimit`, for consistent and predictable behavior.

## Proof of Concept
```bash
# Obtain the first page
NEXT=$(curl -s 'https://<host>/api/v1/blocks?order=asc&limit=100' \
  | jq -r '.links.next')

# Follow pagination in a tight loop — no credentials required
while [ -n "$NEXT" ]; do
  NEXT=$(curl -s "https://<host>${NEXT}" | jq -r '.links.next')
done
```
Parallelising this loop across even a handful of concurrent processes will rapidly exhaust the database connection pool.
