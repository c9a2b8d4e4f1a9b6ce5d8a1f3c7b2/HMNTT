# Audit Report

## Title
Unbounded Redis Channel Subscription Growth via Distinct Topic ID Flooding in `RedisTopicListener`

## Summary
`RedisTopicListener.getSharedListener()` stores one shared `Flux` per distinct topic ID in an unbounded `ConcurrentHashMap`. Each unique topic ID causes a new Redis channel subscription via `ReactiveRedisMessageListenerContainer.receive()`. An unauthenticated attacker can open many gRPC connections, each subscribing to different valid topic IDs, growing the map and Redis subscriptions without bound until Redis server-side resources are exhausted, causing node-wide pub/sub delivery failure.

## Finding Description

`topicMessages` is declared as an unbounded `ConcurrentHashMap` with no size cap: [7](#0-6) 

`getSharedListener()` calls `computeIfAbsent`, creating one new Redis subscription per distinct topic ID: [2](#0-1) 

Each distinct topic ID maps to a unique Redis channel name: [8](#0-7) 

`subscribe()` issues a Redis `SUBSCRIBE` command for that channel and retries indefinitely on failure: [9](#0-8) 

The `unsubscribe()` cleanup only fires when the last subscriber to a given topic disconnects (due to `.share()`). While connections are held open, entries accumulate without bound: [4](#0-3) 

**Why `topicExists` is insufficient:** The check requires the topic to exist in the entity repository: [5](#0-4) 

On a live Hedera network, topic IDs are public, sequential, and thousands of valid topics exist. They are freely enumerable via the public REST API with no authentication. This check does not limit the number of distinct valid topics an attacker can subscribe to simultaneously.

`ListenerProperties` contains no field limiting the number of concurrent topic subscriptions: [6](#0-5) 

## Impact Explanation
Redis pub/sub channel subscriptions consume server-side memory proportional to the number of subscribed channels. Redis's default `client-output-buffer-limit` for pub/sub clients (32 MB hard / 8 MB soft) can be breached by flooding the listener container with thousands of channel subscriptions. Since `this.container` is a singleton `Mono` shared across all topics, its disconnection stops message delivery for **all** topics on the node. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` loop then causes continuous reconnect attempts, amplifying load and prolonging disruption.

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is unauthenticated and publicly accessible in standard deployments. Valid topic IDs are publicly enumerable with no rate limiting on the REST API. Opening thousands of gRPC connections is trivial with standard tooling. No special privileges, credentials, or insider knowledge are required. The attack is repeatable and self-amplifying due to the retry loop.

## Recommendation

1. **Cap the `topicMessages` map size.** Add a configurable `maxTopicSubscriptions` property to `ListenerProperties` and enforce it in `getSharedListener()` before calling `computeIfAbsent`. Reject new subscriptions with an appropriate error when the cap is reached.

2. **Add per-IP or per-client rate limiting** on the number of distinct topic subscriptions allowed concurrently.

3. **Add a global subscription limit** enforced at the `SharedTopicListener` level, independent of the Redis backend.

4. **Consider a subscription TTL or idle timeout** to evict stale entries from `topicMessages` even if clients hold connections open indefinitely.

## Proof of Concept

```python
import grpc
import threading

# Enumerate valid topic IDs from public REST API (omitted for brevity)
valid_topic_ids = list(range(1, 10001))  # 10,000 valid topics

def flood(topic_ids):
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(channel)
    for tid in topic_ids:
        req = ConsensusTopicQuery(topic_id=TopicID(topic_num=tid))
        # Non-blocking: just initiate the stream, keep it open
        threading.Thread(target=lambda r=req: list(stub.subscribeTopic(r)), daemon=True).start()

# Open M connections, each subscribing to 5 distinct topics
chunk = 5
for i in range(0, len(valid_topic_ids), chunk):
    threading.Thread(target=flood, args=(valid_topic_ids[i:i+chunk],)).start()

# Result: topicMessages map grows to 10,000 entries;
# ReactiveRedisMessageListenerContainer issues 10,000 SUBSCRIBE commands;
# Redis client-output-buffer-limit is breached; container disconnects;
# all topic message delivery on the node halts.
```

```java
    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
```

```java
    private Mono<?> topicExists(TopicMessageFilter filter) {
        var topicId = filter.getTopicId();
        return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
                .switchIfEmpty(
                        grpcProperties.isCheckTopicExists()
                                ? Mono.error(new EntityNotFoundException(topicId))
                                : Mono.just(Entity.builder()
                                        .memo("")
                                        .type(EntityType.TOPIC)
                                        .build()))
                .filter(e -> e.getType() == EntityType.TOPIC)
                .switchIfEmpty(Mono.error(new IllegalArgumentException("Not a valid topic")));
```
