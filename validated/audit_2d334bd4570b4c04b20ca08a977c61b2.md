Audit Report

## Title
Race Condition in `RedisTopicListener.unsubscribe()` Allows Key-Only Removal to Evict Active Shared Flux

## Summary
`RedisTopicListener.unsubscribe()` removes a topic entry from `topicMessages` using a key-only `ConcurrentHashMap.remove(key)`. Because `getSharedListener()` returns a `Flux` reference via `computeIfAbsent` before the caller subscribes, a concurrent cancellation can cause `unsubscribe()` to evict a newer, still-active `Flux` instance (F2) that was placed in the map after the cancelling `Flux` (F1) was already superseded. This fragments the shared subscription graph and creates windows during which messages are not delivered to affected subscribers.

## Finding Description

**File:** `grpc/src/main/java/org/hiero/mirror/grpc/listener/RedisTopicListener.java`

**`getSharedListener()` — line 59–62:**
```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
```
Returns a `Flux` reference to the caller. The caller has not yet subscribed. This is not atomic with the subsequent subscription.

**`subscribe()` / `doOnCancel` hook — line 68–79:**
```java
return container
    .flatMapMany(...)
    .map(Message::getMessage)
    .doOnCancel(() -> unsubscribe(topic))   // line 74
    .doOnComplete(() -> unsubscribe(topic)) // line 75
    ...
    .share();                               // line 79
```
The `doOnCancel` callback fires asynchronously on the Reactor scheduler when `.share()`'s upstream refcount drops to 0. It is not executed under any lock.

**`unsubscribe()` — line 82–85:**
```java
private void unsubscribe(Topic topic) {
    topicMessages.remove(topic.getTopic()); // key-only removal
    log.info("Unsubscribing from {}", topic);
}
```
**Root cause:** `ConcurrentHashMap.remove(key)` unconditionally removes whatever is currently stored under that key. The two-argument form `remove(key, value)` — which only removes the entry if the stored value equals the expected value — is not used. This means if a newer `Flux` (F2) has already been stored in the map by the time `unsubscribe()` runs for the old `Flux` (F1), F2 is evicted instead.

**Exploit flow:**
```
T=0  Victim:   computeIfAbsent → receives reference to F1 (not yet subscribed)
T=1  Attacker: sole subscriber to F1 cancels
               → .share() refcount → 0 → upstream cancel propagates
               → doOnCancel fires → unsubscribe() removes F1 from map
T=2  New sub:  computeIfAbsent creates F2, stores F2 in map
T=3  Victim:   subscribes to F1 (stale reference)
               → .share() refcount 0→1 → F1 reconnects upstream to Redis
               → F1 is NOT in the map; F2 is
T=4  Attacker: subscribes to F2, then immediately cancels
               → F2 refcount → 0 → doOnCancel fires
               → unsubscribe() removes F2 from map (key-only: removes F2, not F1)
T=5  Any new subscriber: computeIfAbsent creates F3
     Victim is isolated on F1; new subscribers are on F3
     Messages published between T=1 and T=3 are missed by victim
```

**Why existing checks are insufficient:** `ConcurrentHashMap` makes individual operations atomic, but `computeIfAbsent` + deferred subscription is not a single atomic unit. The `doOnCancel` callback fires asynchronously on the Reactor scheduler, not under any lock, so the window between "last subscriber cancels" and "new subscriber calls `computeIfAbsent`" is real and exploitable under load.

## Impact Explanation
Legitimate subscribers monitoring a topic are silently isolated on a stale, map-orphaned `Flux`. They continue to receive a Redis stream, but it is a reconnected stream that missed messages published during the cancellation/reconnection gap. Additionally, the attacker's cycling creates repeated Redis `SUBSCRIBE`/`UNSUBSCRIBE` churn, causing resource exhaustion on the broker. Because the mirror node is the authoritative real-time delivery path for HCS topic messages, silent message loss is a high-severity availability and correctness issue for downstream applications.

## Likelihood Explanation
No authentication or rate-limiting on gRPC subscription calls is visible in this code path. Any unauthenticated client can open and immediately cancel a `subscribeToTopic` stream. The race window (between `computeIfAbsent` returning and the caller subscribing) is narrow but widens under load; automated tooling can fire hundreds of subscribe/cancel cycles per second, making reliable exploitation straightforward. The attack is stateless and repeatable.

## Recommendation
Replace the key-only removal in `unsubscribe()` with the two-argument form of `ConcurrentHashMap.remove`, passing the specific `Flux` instance being cancelled:

```java
// In subscribe(), capture the Flux reference:
private Flux<TopicMessage> subscribe(Topic topic) {
    Flux<TopicMessage>[] ref = new Flux[1];
    ref[0] = container
            .flatMapMany(r -> r.receive(...))
            .map(Message::getMessage)
            .doOnCancel(() -> topicMessages.remove(topic.getTopic(), ref[0]))
            .doOnComplete(() -> topicMessages.remove(topic.getTopic(), ref[0]))
            ...
            .share();
    return ref[0];
}
```

This ensures that `unsubscribe()` only removes the entry if the stored value is still the same `Flux` instance that is being cancelled, preventing eviction of a newer active `Flux`.

## Proof of Concept
1. Client A calls `getSharedListener()` for topic `T`, receives reference to `F1` but does not yet subscribe (simulating a slow thread or scheduler delay).
2. Client B subscribes to `F1` (the only subscriber), then immediately cancels. `doOnCancel` fires, `unsubscribe()` removes `F1` from `topicMessages`.
3. Client C calls `getSharedListener()` for topic `T`. `computeIfAbsent` creates `F2` and stores it in the map.
4. Client A now subscribes to `F1` (stale reference). `F1.share()` refcount goes 0→1, reconnecting upstream. `F1` is live but absent from the map.
5. Client B subscribes to `F2` (from the map), then immediately cancels. `doOnCancel` fires, `unsubscribe()` calls `topicMessages.remove("topic.T")` — this removes `F2` (the currently stored value), not `F1`.
6. Client D calls `getSharedListener()` for topic `T`. `computeIfAbsent` creates `F3`.
7. Client A is now isolated on `F1`; Client D is on `F3`. Any messages published to Redis between steps 2 and 4 are missed by Client A. The map now contains `F3`, and `F1` is a live but orphaned subscription.
