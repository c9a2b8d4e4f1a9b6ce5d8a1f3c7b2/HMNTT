# Audit Report

## Title
Unbounded Sub-Cache Accumulation in `cacheManagerSlotsPerContract` Leads to Memory Exhaustion DoS

## Summary
In `ContractStateServiceImpl.findStorageBatch`, every unique `contractId` causes `cacheManagerSlotsPerContract.getCache(contractId.toString())` to register a new named `CaffeineCache` in the `CaffeineCacheManager`'s internal `ConcurrentHashMap`. Because `cacheManagerSlotsPerContract` is configured in dynamic mode (no `setCacheNames()` call), this internal registry is never evicted. The outer `contractSlotsCache` (bounded at 3000 entries) evicts entries over time, but the sub-caches remain permanently referenced in `cacheManagerSlotsPerContract`, causing unbounded heap growth. An unprivileged attacker querying distinct contract addresses will exhaust JVM heap memory and crash the service.

## Finding Description

**Exact code path:**

`ContractStateServiceImpl.java`, `findStorageBatch`, lines 85–87: [7](#0-6) 

```java

    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause — two-layer cache design with one-sided eviction:**

1. The outer `contractSlotsCache` is backed by `CACHE_MANAGER_CONTRACT_SLOTS`, configured with `"expireAfterAccess=5m,maximumSize=3000,recordStats"`. It maps `EntityId → CaffeineCache` and is bounded at 3000 entries. [3](#0-2) 

2. The `cacheManagerSlotsPerContract` bean is a `CaffeineCacheManager` created **without** a `setCacheNames()` call, placing it in *dynamic mode*. [1](#0-0)  In dynamic mode, Spring's `CaffeineCacheManager.getCache(name)` creates a new `CaffeineCache` for any previously unseen name and stores it in its internal `ConcurrentHashMap<String, Cache> cacheMap`. **This map has no eviction mechanism.**

3. When the outer `contractSlotsCache` evicts an entry (size > 3000 or TTL expiry), the `CaffeineCache` object is removed from the outer cache. However, `cacheManagerSlotsPerContract` still holds a strong reference to it in its `cacheMap` under the key `contractId.toString()`. The sub-cache is never garbage-collected.

4. Each new unique `contractId` permanently adds one `CaffeineCache` instance (backed by a Caffeine cache with `expireAfterAccess=5m,maximumSize=1500`) to `cacheManagerSlotsPerContract.cacheMap`. [8](#0-7)  Over time, this map grows without bound.

**Why existing checks fail:**

- The outer cache's `maximumSize=3000` only limits the outer mapping; it does not remove entries from `cacheManagerSlotsPerContract.cacheMap`. [2](#0-1) 
- The global rate limiter enforces a per-second request cap globally, not per-IP. An attacker can sustain the allowed rate indefinitely across multiple IPs or over time, continuously adding new sub-caches. [5](#0-4) 
- `enableBatchContractSlotCaching` defaults to `true`, so the vulnerable path is active by default. [4](#0-3) 
- There is no validation that `contractId` corresponds to an existing contract before the sub-cache is created; the cache is allocated before the DB query returns. [9](#0-8) 

## Impact Explanation
Each `CaffeineCache` wrapping a Caffeine instance with `maximumSize=1500` allocates internal data structures (timer wheel, node arrays, stats counters) even when empty — conservatively 2–10 KB per instance. At a sustained rate of N requests/second with distinct `contractId`s, `cacheManagerSlotsPerContract.cacheMap` grows at N entries/second. With a global rate limit of hundreds of requests/second, an attacker can accumulate millions of sub-caches within hours, exhausting JVM heap and triggering `OutOfMemoryError`, crashing the web3 service. This prevents all contract call processing.

## Likelihood Explanation
The attack requires no authentication, no privileged access, and no valid on-chain contracts — any address value accepted by the JSON-RPC layer suffices as a `contractId`. The attacker only needs to send `eth_call` or `eth_getStorageAt` requests with distinct `to` addresses at the permitted rate. This is trivially scriptable. The attack is persistent: sub-caches accumulate across the full uptime of the service, not just within a single rate-limit window.

## Recommendation

1. **Fix the root cause**: Call `setCacheNames()` on `cacheManagerSlotsPerContract` with a fixed set of names, or switch to a different design that does not use `CaffeineCacheManager` in dynamic mode. [1](#0-0) 

2. **Alternatively, manage the sub-cache lifecycle explicitly**: Instead of delegating to `CaffeineCacheManager`, maintain a bounded `LoadingCache<EntityId, CaffeineCache>` directly in `ContractStateServiceImpl`, with a removal listener that invalidates the sub-cache on eviction. This ensures the sub-cache map is bounded by the same limit as the outer cache.

3. **Add per-IP rate limiting**: The current global token bucket does not prevent a single attacker from consuming the full rate budget. Per-IP rate limiting would reduce the rate at which new sub-caches can be created. [5](#0-4) 

4. **Validate `contractId` before cache allocation**: Check that the `contractId` corresponds to an existing contract in the database before calling `cacheManagerSlotsPerContract.getCache()`. [7](#0-6) 

## Proof of Concept

```python
import requests
import random

RPC_URL = "http://<mirror-node-web3>:8545"

def random_address():
    return "0x" + "".join(random.choices("0123456789abcdef", k=40))

# Each request uses a unique contract address, causing a new CaffeineCache
# to be permanently registered in cacheManagerSlotsPerContract.cacheMap
while True:
    addr = random_address()
    payload = {
        "jsonrpc": "2.0",
        "method": "eth_getStorageAt",
        "params": [addr, "0x0", "latest"],
        "id": 1
    }
    requests.post(RPC_URL, json=payload)
    # Each call: cacheManagerSlotsPerContract.cacheMap grows by 1 entry (never removed)
    # After ~3000 calls: outer contractSlotsCache starts evicting, but inner cacheMap
    # retains all CaffeineCache instances. Memory grows without bound.
```

After sufficient iterations, the JVM heap is exhausted and the service crashes with `OutOfMemoryError`. The outer cache's `maximumSize=3000` provides no protection because it does not trigger cleanup of `cacheManagerSlotsPerContract.cacheMap`. [7](#0-6)
