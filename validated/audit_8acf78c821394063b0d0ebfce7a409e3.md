# Audit Report

## Title
Integer Narrowing Cast in `PollingTopicListener.poll()` Bypasses `maxPageSize` Guard, Enabling Unlimited DB Query

## Summary
When the POLL listener type is active, a single gRPC subscription with `limit = Long.MAX_VALUE` causes `(int)(Long.MAX_VALUE)` to overflow to `-1`. `Math.min(-1, maxPageSize)` returns `-1`, which is stored as the new filter's `limit`. Because `hasLimit()` tests `limit > 0`, the value `-1` is treated as "no limit," causing `findByFilter()` to execute an unbounded `SELECT` and load the entire `topic_message` table into heap memory via `getResultList()`. The poll repeats every 500 ms indefinitely.

## Finding Description

**Entry point — `ConsensusController.toFilter()`** [1](#0-0) 

`query.getLimit()` (proto `uint64`, full 64-bit range) is passed directly to the filter builder with no upper-bound check.

**No upper-bound validation — `TopicMessageFilter`** [2](#0-1) 

Only `@Min(0)` is present. `Long.MAX_VALUE` passes validation.

**Limit forwarded unchanged — `TopicMessageServiceImpl.incomingMessages()`** [3](#0-2) 

With `filter.getLimit() = Long.MAX_VALUE` and `count = 0`, `limit = Long.MAX_VALUE`. This is forwarded to `topicListener.listen()`.

**Root cause — `PollingTopicListener.poll()`** [4](#0-3) 

```
(int)(Long.MAX_VALUE - 0L) = -1   // narrowing overflow
Math.min(-1, 5000)         = -1   // maxPageSize guard bypassed
newFilter.limit            = -1
```

**Guard failure — `TopicMessageRepositoryCustomImpl.findByFilter()`** [5](#0-4) 

`hasLimit()` returns `false` for `-1` (`-1 > 0` is false), so `setMaxResults` is never called. `getResultList()` then loads the entire result set into JVM heap.

**Reactive `take()` does not mitigate:** [6](#0-5) 

`flux.take(Long.MAX_VALUE)` is effectively no limit, and it operates after the database query has already executed and loaded all rows into memory.

## Impact Explanation
On a busy mainnet topic with millions of messages, each poll cycle executes a full table scan and loads the entire result set into heap memory. This can cause `OutOfMemoryError` or severe GC pressure, crashing or severely degrading the gRPC service for all subscribers. The poll repeats on `RepeatSpec.times(Long.MAX_VALUE)` with a default 500 ms interval, so the unbounded query fires continuously until the subscription is cancelled. [7](#0-6) 

## Likelihood Explanation
The `subscribeTopic` gRPC endpoint is publicly accessible with no authentication. Setting `limit = Long.MAX_VALUE` requires only a standard gRPC client and knowledge of the proto schema. No special privileges, tokens, or internal state knowledge are needed. Multiple simultaneous connections can amplify the impact.

## Recommendation

1. **Fix the narrowing cast** in `PollingTopicListener.poll()`: clamp the remaining limit to `[0, Integer.MAX_VALUE]` before casting:
   ```java
   long remaining = filter.getLimit() - context.getCount().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```

2. **Add an upper-bound constraint** on `TopicMessageFilter.limit`, e.g. `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate maximum, to reject pathological values at the controller layer.

3. **Guard `findByFilter()` unconditionally**: always call `typedQuery.setMaxResults(listenerProperties.getMaxPageSize())` as a safety net, regardless of whether `hasLimit()` is true.

## Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from com.hedera.hashgraph.sdk.proto import consensus_topic_query_pb2, basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1234),
    limit=9223372036854775807  # Long.MAX_VALUE
)

# Each poll cycle will execute: SELECT * FROM topic_message WHERE topic_id=1234
# with no LIMIT clause, loading the entire table into heap memory.
for msg in stub.subscribeTopic(query):
    pass
```

With `hiero.mirror.grpc.listener.type=POLL`, this single subscription triggers an unbounded `SELECT` every 500 ms, exhausting heap memory on any topic with a large message history.
