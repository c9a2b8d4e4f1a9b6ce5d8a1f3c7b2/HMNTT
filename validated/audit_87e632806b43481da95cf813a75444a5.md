Audit Report

## Title
Unbounded `nodeIds` Array Input Enables Resource Exhaustion DoS on `/api/v1/network/nodes`

## Summary
`NetworkNodeRequest.nodeIds` accepts an unlimited number of `node.id` query parameters with no `@Size` constraint. An unauthenticated attacker can submit thousands of `node.id=eq:X` values per request, forcing unbounded `Long[]` allocation on the Java heap and serialization of that array into a PostgreSQL `any(:nodeIds)` expression on every call.

## Finding Description

**[1] `NetworkNodeRequest.nodeIds` — no `@Size` constraint:** [1](#0-0) 

The field carries only `@RestJavaQueryParam` and `@Builder.Default`. No `@Size` annotation is present.

**[2] `RegisteredNodesRequest.registeredNodeIds` — correctly constrained with `@Size(max = 2)`:** [2](#0-1) 

The analogous field in the sibling DTO applies `@Size(max = 2)`, confirming the pattern exists in the codebase and was simply omitted from `NetworkNodeRequest`.

**[3] `RequestParameterArgumentResolver.validateAndAddParameter()` — no count check for multi-value fields:** [3](#0-2) 

When `field.getType()` is a `Collection`, the entire `paramValues` array is forwarded to `propertyValues` without any length check. The only guard (`paramValues.length > 1`) applies only to *non*-collection fields.

**[4] `NetworkServiceImpl.getNetworkNodes()` — unbounded `HashSet` → `Long[]` conversion:** [4](#0-3) 

Every `EQ` parameter is added to a `HashSet<Long>` with no size cap. The set is then streamed into a `Long[]` and passed directly to the repository.

**[5] SQL query — `abe.node_id = any(:nodeIds)` with the unbounded array:** [5](#0-4) 

PostgreSQL receives and evaluates the full array on every request.

**Why existing checks fail:**
- `@Min(1)` on `limit` bounds result count, not input array size. [6](#0-5) 
- `getEffectiveLimit()` caps results at 25 but does not touch `nodeIds`. [7](#0-6) 
- `HashSet` deduplication reduces duplicates but does not cap the number of distinct values an attacker can supply. [8](#0-7) 

## Impact Explanation
Each malicious request forces: (1) allocation of a `Long[~10,000]` array (~80 KB) on the Java heap, (2) JDBC serialization of that array into a PostgreSQL array literal, and (3) PostgreSQL evaluation of `any({...10,000 elements...})` against every row in the address book. Concurrent flooding causes heap pressure, increased GC pauses, and elevated DB connection hold time. The address book is small so per-query DB work is bounded, but the Java/JDBC overhead is real and repeatable. Severity: **Medium**.

## Likelihood Explanation
No authentication is required. The endpoint is public. The attack requires only a standard HTTP client capable of sending a URL with thousands of repeated query parameters — trivially scriptable. Tomcat's default `maxParameterCount` is 10,000, so up to ~10,000 distinct `node.id=eq:X` values can be submitted per request. The inconsistency with `RegisteredNodesRequest` confirms this is an oversight, not an intentional design choice.

## Recommendation
Add `@Size(max = N)` to `NetworkNodeRequest.nodeIds`, consistent with the pattern already applied to `RegisteredNodesRequest.registeredNodeIds`:

```java
// NetworkNodeRequest.java
@RestJavaQueryParam(name = NODE_ID, required = false)
@Builder.Default
@Size(max = 25)   // or a smaller bound matching business requirements
private List<NumberRangeParameter> nodeIds = List.of();
```

The `RequestParameterArgumentResolver` already invokes `binder.validate()`, so the `@Size` constraint will be enforced automatically before the service layer is reached. [9](#0-8) 

## Proof of Concept
```
GET /api/v1/network/nodes?node.id=eq:1&node.id=eq:2&...&node.id=eq:9999
```
Repeat with ~10,000 distinct `node.id=eq:X` values. Each request allocates a `Long[10000]` array, serializes it to JDBC, and evaluates `any({1,2,...,9999})` in PostgreSQL. Flood at high concurrency to exhaust heap and DB connection pool.
