# Audit Report

## Title
Missing Entity Type Validation in `isValidAccount()` Allows Contract Addresses to Pass Account Check in `getTokenRelationships()`

## Summary
`EntityService.isValidAccount()` fetches the `type` column from the `entity` table but discards it, checking only for row existence. As a result, `TokenController.getTokenRelationships()` — which relies exclusively on `isValidAccount()` as its entity-type gate — will serve token relationship data for contract entities via the `/accounts/{id}/tokens` endpoint. No authentication is required; any publicly known contract EVM address is sufficient to trigger the behavior.

## Finding Description
`entityExistenceQuery` in `rest/service/entityService.js` (lines 28–30) selects `Entity.TYPE` from the `entity` table:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;
``` [1](#0-0) 

`isValidAccount()` (lines 60–63) executes this query but only checks whether a row was returned — the fetched `type` value is never read:

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity);
}
``` [2](#0-1) 

`getTokenRelationships()` in `rest/controllers/tokenController.js` (lines 66–74) uses `isValidAccount()` as its only guard before querying `token_account`:

```js
getTokenRelationships = async (req, res) => {
  const accountId = await EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]);
  const isValidAccount = await EntityService.isValidAccount(accountId);
  if (!isValidAccount) {
    throw new NotFoundError();
  }
  ...
  const tokenRelationships = await TokenService.getTokenAccounts(query);
``` [3](#0-2) 

`entityFromEvmAddressQuery` (lines 22–25) resolves EVM addresses with no `type` filter:

```sql
select ${Entity.ID}
from ${Entity.tableName}
where ${Entity.DELETED} <> true
  and ${Entity.EVM_ADDRESS} = $1
``` [4](#0-3) 

**Exploit flow:**
1. Attacker supplies a known contract EVM address (e.g., `0xcef2a2c6c23ab8f2506163b1af55830f35c483ca`) to `GET /api/v1/accounts/0xcef2a2c6c23ab8f2506163b1af55830f35c483ca/tokens`.
2. `getEncodedId()` calls `getEntityIdFromEvmAddress()`, which resolves the address to the contract's encoded entity ID with no type check.
3. `isValidAccount()` finds the contract entity row exists → returns `true`.
4. `TokenService.getTokenAccounts()` queries `token_account` using the contract's entity ID as `account_id` and returns real token association data.
5. The endpoint returns `200 OK` with token relationship data for a contract entity under the `/accounts/` namespace.

The `isValidAccount` integration tests confirm this gap: they only assert existence, never type correctness. [6](#0-5) 

## Impact Explanation
The `/accounts/{id}/tokens` endpoint is documented and consumed as an account-scoped API. Returning token relationships for contract entities violates this contract. Off-chain systems and smart contracts that consume mirror node data to determine account token balances may receive data for a contract entity, leading to incorrect balance reads, incorrect authorization decisions, or incorrect state assumptions in dependent logic. Severity is bounded because the mirror node is read-only and no funds are directly manipulated, but the data integrity violation can propagate into on-chain decisions via smart contracts that consume mirror node data.

## Likelihood Explanation
Exploitation requires zero privileges — the REST API is public. The attacker only needs a contract EVM address, which is trivially discoverable from block explorers or the `/contracts` endpoint. The attack is fully repeatable and deterministic. Any Hedera network with deployed contracts that hold token associations is affected.

## Recommendation
Add an entity type check inside `isValidAccount()`. The `type` column is already fetched; it only needs to be inspected:

```js
async isValidAccount(accountId) {
  const entity = await super.getSingleRow(EntityService.entityExistenceQuery, [accountId]);
  return !isNil(entity) && entity.type === 'ACCOUNT';
}
```

Alternatively, add a `type = 'ACCOUNT'` predicate directly to `entityExistenceQuery` to enforce the constraint at the database level, which also reduces the risk of future callers forgetting to check the field. The `entityFromEvmAddressQuery` should similarly be restricted to account-type entities when used in account-scoped resolution paths.

## Proof of Concept

GET /api/v1/accounts/0xcef2a2c6c23ab8f2506163b1af55830f35c483ca/tokens
```

3. `TokenService.getTokenAccounts()` queries `token_account` with `account_id = 95623`.
4. If the contract holds any token associations, the response is `200 OK` with token relationship data — served under the `/accounts/` namespace for a contract entity.
