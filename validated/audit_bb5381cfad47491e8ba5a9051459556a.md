## Audit Report

## Title
Historical Hook Storage Query Bypasses Key Range Filter via Raw `keys` Instead of `keysInRange`

## Summary
In `HookServiceImpl.getHookStorageChange()`, when a historical query includes explicit keys and at least one of them falls within the declared key range, the method passes the raw (unfiltered) `keys` collection to `HookStorageChangeRepository.findByKeyInAndTimestampBetween()` instead of the range-filtered `keysInRange`. This allows a caller to retrieve historical `hook_storage_change` records for arbitrary storage keys outside the declared `[keyLowerBound, keyUpperBound]` range.

## Finding Description

**Non-historical path (correct):**
In `getHookStorage()`, `keysInRange` is computed at line 72 and correctly passed to the repository at lines 78–79: [1](#0-0) 

**Historical path (buggy):**
In `getHookStorageChange()`, `keysInRange` is computed at line 91, but the guard at line 93 only short-circuits when `keysInRange` is entirely empty AND the request has keys: [2](#0-1) 

When at least one supplied key is within range, `keysInRange` is non-empty, the guard does not trigger, and execution falls through to line 104–105 where the raw `keys` (not `keysInRange`) is passed to the repository: [3](#0-2) 

The SQL query in `findByKeyInAndTimestampBetween` applies no key-range bounds — only `key in (:keys)` and a timestamp range: [4](#0-3) 

`getKeysInRange()` correctly filters keys to `[keyLowerBound, keyUpperBound]`, but its output is never used in the historical branch: [5](#0-4) 

**Root cause:** Line 105 passes `keys` where `keysInRange` should be used, making the `keyLowerBound`/`keyUpperBound` range constraints entirely ineffective in the historical code path whenever at least one key is in range.

## Impact Explanation
A caller can retrieve historical `hook_storage_change` records for arbitrary storage keys belonging to any hook, as long as they include at least one legitimately in-range key to defeat the guard. This allows reading storage slot values at past timestamps for keys the caller was never authorized to access, violating the confidentiality of historical hook storage state. The `hook_storage_change` table may contain sensitive historical state transitions that should only be visible within the declared key range.

## Likelihood Explanation
No special privilege is required beyond the ability to issue a historical hook storage query (i.e., supply a `timestamp` parameter). An attacker crafts a `keys` list containing one in-range key and any number of out-of-range keys. The exploit is deterministic, trivially repeatable, and requires no race condition or timing dependency.

## Recommendation
In `getHookStorageChange()` at line 104–105 of `HookServiceImpl.java`, replace the raw `keys` argument with `keysInRange`:

```java
// Before (buggy):
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);

// After (fixed):
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keysInRange, timestampLowerBound, timestampUpperBound, page);
```

This mirrors the correct behavior already implemented in the non-historical path at lines 78–79. [3](#0-2) 

## Proof of Concept
1. Identify a hook with storage keys. Suppose the declared range is `[0x00...01, 0x00...0A]` and the target out-of-range key is `0xFF...FF`.
2. Issue a historical hook storage query with:
   - `timestamp` set to any valid historical range (making `isHistorical()` return `true`)
   - `keyLowerBound = 0x00...01`, `keyUpperBound = 0x00...0A`
   - `keys = [0x00...01, 0xFF...FF]` — one in-range key, one out-of-range key
3. `getKeysInRange()` returns `[0x00...01]` (non-empty), so the guard at line 93 does **not** trigger.
4. `requestHasKeys` is `true`, so the branch at line 103 is taken.
5. The raw `keys = [0x00...01, 0xFF...FF]` is passed to `findByKeyInAndTimestampBetween`.
6. The SQL query returns records for both `0x00...01` and `0xFF...FF` — the out-of-range key is returned despite being outside the declared bounds. [6](#0-5)
