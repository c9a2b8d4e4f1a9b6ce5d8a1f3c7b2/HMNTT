Audit Report

## Title
`deleted <> true` SQL Predicate Silently Excludes Entities with `deleted=NULL` from Alias Lookups

## Summary
The `findByAlias()` (and `findByEvmAddress()`) queries in both the `importer` and `rest-java` modules use the SQL predicate `deleted <> true`. In standard SQL, `NULL <> true` evaluates to `NULL` (not `TRUE`), so rows where `deleted IS NULL` are silently excluded. Because `CryptoCreateTransactionHandler.doUpdateEntity()` never sets `deleted`, and the `deleted` column is intentionally nullable, every account created via a standard CryptoCreate transaction is stored with `deleted=NULL`. This causes alias-based lookups to fail in `rest-java` immediately (no cache) and in the importer after any restart (cold cache).

## Finding Description

**Vulnerable queries — `deleted <> true` in both modules:**

`importer/EntityRepository.java` line 16: [1](#0-0) 

`rest-java/EntityRepository.java` line 13: [2](#0-1) 

Both use `deleted <> true`. In SQL, `NULL <> true` yields `NULL`, which is falsy — rows with `deleted=NULL` are excluded from results.

**Root cause 1 — `deleted` column is intentionally nullable:**

Migration `V1.39.1__upsert_support.sql` explicitly dropped the `NOT NULL` constraint and `DEFAULT false` from the `deleted` column with the comment *"allow nullable on entity deleted as transaction cannot make this assumption on updates"*. [3](#0-2) 

**Root cause 2 — `CryptoCreateTransactionHandler.doUpdateEntity()` never sets `deleted`:**

The method sets `type`, `alias`, `key`, `evmAddress`, `balance`, `memo`, staking info, etc., but never calls `entity.setDeleted(false)`: [4](#0-3) 

**Cache masks the bug during the same importer run:**

`EntityIdServiceImpl.notify()` only skips entities when `deleted != null && deleted == true`, so entities with `deleted=null` are cached normally: [5](#0-4) 

After any restart the in-memory cache is cold, `findByAlias()` hits the DB, and returns `Optional.empty()` for every account created via CryptoCreate.

**Direct REST-API impact — `rest-java/EntityServiceImpl` has no cache:**

It calls `entityRepository.findByAlias()` directly and throws `EntityNotFoundException` on empty result: [6](#0-5) 

**Tests confirm the broken behavior as correct:**

The `rest-java` integration test explicitly asserts that `deleted=null` entities return empty from `findByAlias`: [7](#0-6) 

The importer's `findByAlias` test only exercises `domainBuilder.entity()` (which defaults to `deleted=false`) and never tests the `deleted=null` case: [8](#0-7) 

**Inconsistency with correct implementations:**

`graphql` and `web3` both use `deleted is not true`, which correctly treats `NULL` as "not deleted": [9](#0-8) [10](#0-9) 

## Impact Explanation

Any account created via a standard CryptoCreate transaction is stored with `deleted=NULL`. The `rest-java` mirror node API immediately returns HTTP 404 for any alias-based or EVM-address-based lookup of such an account — no restart required. In the importer, after any restart (routine during upgrades/maintenance), alias-based entity ID resolution silently fails for those accounts, causing all downstream transaction records that reference those accounts by alias to be recorded with a null/empty entity ID, corrupting the mirror node's view of the ledger.

## Likelihood Explanation

Every CryptoCreate transaction (the most common Hedera operation) triggers this condition. No special permissions or protocol knowledge are required. The `rest-java` impact is immediate and permanent. The importer impact manifests on every restart. The condition is highly repeatable by any network participant.

## Recommendation

Replace `deleted <> true` with `deleted is not true` in both affected repositories, consistent with the already-correct implementations in `graphql` and `web3`:

- `importer/src/main/java/org/hiero/mirror/importer/repository/EntityRepository.java` — lines 16 and 19
- `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java` — lines 13 and 16

Update the `rest-java` integration test at line 28 of `EntityRepositoryTest.java` to assert that `deleted=null` entities **are** returned (not empty), and add a corresponding test case to the importer's `EntityRepositoryTest.findByAlias`.

## Proof of Concept

1. Submit a CryptoCreate transaction with an alias on any Hedera network.
2. The importer stores the resulting entity with `deleted=NULL` (confirmed: `CryptoCreateTransactionHandler` never calls `setDeleted`).
3. Immediately query the `rest-java` API using the alias: `GET /api/v1/accounts/<alias>`. The service calls `entityRepository.findByAlias()`, which executes `WHERE alias = ? AND deleted <> true`. Since `NULL <> true` is `NULL`, the row is excluded, `Optional.empty()` is returned, and `EntityServiceImpl` throws `EntityNotFoundException` → HTTP 404.
4. For the importer path: restart the importer process (clearing the in-memory cache). Any subsequent transaction referencing the account by alias will call `findByAlias()` via `EntityIdServiceImpl.findByAlias()` → `entityRepository.findByAlias()` → returns empty → entity ID is unresolved.

    protected void doUpdateEntity(Entity entity, RecordItem recordItem) {

        var transactionBody = recordItem.getTransactionBody().getCryptoCreateAccount();
        var alias = DomainUtils.toBytes(
                transactionRecord.getAlias() != ByteString.EMPTY
                        ? transactionRecord.getAlias()
                        : transactionBody.getAlias());
        boolean emptyAlias = ArrayUtils.isEmpty(alias);
        var key = transactionBody.hasKey() ? transactionBody.getKey().toByteArray() : null;
        boolean emptyKey = ArrayUtils.isEmpty(key);
        entity.setType(EntityType.ACCOUNT);

        if (!emptyAlias) {
            entity.setAlias(alias);
            if (emptyKey && alias.length > EVM_ADDRESS_LENGTH) {
                entity.setKey(alias);
            }
        }

        if (!emptyKey) {
            entity.setKey(key);
        }

        var evmAddress = transactionRecord.getEvmAddress();
        if (evmAddress != ByteString.EMPTY) {
            entity.setEvmAddress(DomainUtils.toBytes(evmAddress));
        } else if (!emptyAlias) {
            entity.setEvmAddress(Utility.aliasToEvmAddress(alias));
        }

        if (transactionBody.hasAutoRenewPeriod()) {
            entity.setAutoRenewPeriod(transactionBody.getAutoRenewPeriod().getSeconds());
        }

        if (transactionBody.hasProxyAccountID()) {
            var proxyAccountId = EntityId.of(transactionBody.getProxyAccountID());
            entity.setProxyAccountId(proxyAccountId);
            recordItem.addEntityId(proxyAccountId);
        }

        entity.setBalance(0L);
        entity.setBalanceTimestamp(recordItem.getConsensusTimestamp());
        entity.setMaxAutomaticTokenAssociations(transactionBody.getMaxAutomaticTokenAssociations());
        entity.setMemo(transactionBody.getMemo());
        entity.setReceiverSigRequired(transactionBody.getReceiverSigRequired());

        updateStakingInfo(recordItem, entity);
        entityListener.onEntity(entity);
        evmHookHandler.process(recordItem, entity.getId(), transactionBody.getHookCreationDetailsList(), List.of());
