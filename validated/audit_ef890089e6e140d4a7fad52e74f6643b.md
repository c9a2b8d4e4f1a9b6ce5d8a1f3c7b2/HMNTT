## Audit Report

## Title
Unauthenticated, Unthrottled `getNodes()` gRPC Endpoint Enables Database Connection Pool Exhaustion DoS

## Summary
`NetworkController.getNodes()` accepts `AddressBookQuery` requests from any unauthenticated caller with no rate limiting at the gRPC application layer. Each request drives multiple database queries and repeated `transactionOperations.execute()` calls in `page()`, each of which acquires a HikariCP connection. A flood of concurrent requests across many connections can exhaust the finite connection pool, starving all legitimate callers of database access.

## Finding Description

**Code path:**

`NetworkController.getNodes()` (lines 33–43) accepts any `AddressBookQuery` with no authentication check, no IP-based throttle, and no per-client rate limit. [1](#0-0) 

The only per-connection limit is `maxConcurrentCallsPerConnection = 5` (default), which limits calls *per connection* but places no cap on the total number of connections an attacker may open. [2](#0-1) [10](#0-9) 

The call delegates to `NetworkServiceImpl.getNodes()`, which performs two uncached database queries before the paging loop begins:
- `addressBookRepository.findLatestTimestamp()` — no `@Cacheable`
- `nodeStakeRepository.findLatestTimestamp()` — no `@Cacheable`

(`findAllStakeByConsensusTimestamp` is `@Cacheable` and is not a contributing factor after the first call.) [11](#0-10) 

The paging loop then calls `page()` repeatedly. Each invocation calls `transactionOperations.execute()`, which opens a read-only transaction and acquires a HikariCP connection for the duration of the lambda — even when the inner `findByConsensusTimestampAndNodeId` result is served from the Caffeine cache. The `TransactionTemplate.execute()` opens the connection before the cache proxy on the repository method is consulted. [6](#0-5) [12](#0-11) 

The `withScheduler(Schedulers.boundedElastic())` applies only to the delay between repeat pages, not to the initial two uncached DB queries or the first `page()` call. [13](#0-12) 

**Root cause:** No rate limiting, authentication, or connection-count cap exists at the gRPC controller layer for `getNodes()`. The throttling infrastructure (`ThrottleManagerImpl`) is present only in the `web3` module and is never applied to the gRPC `NetworkService`. [7](#0-6) 

**Why existing checks fail:**
- `maxConcurrentCallsPerConnection = 5` is per-connection; an attacker opens N connections for N×5 concurrent calls.
- `Schedulers.boundedElastic()` bounds only the inter-page delay scheduling, not the initial DB queries or the first `page()` call per request.
- `@Cacheable` on `findByConsensusTimestampAndNodeId` reduces SQL execution time but does not prevent the transaction (and thus the HikariCP connection acquisition) from occurring.
- The optional GCP backend policy `maxRatePerEndpoint: 250` is infrastructure-level and not enforced in all deployments. [14](#0-13) 

## Impact Explanation
When the HikariCP pool is exhausted, all gRPC calls requiring a DB connection queue and eventually time out with `SQLTransientConnectionException`. Smart contract clients (e.g., Hedera SDK instances bootstrapping their node list) that rely on `getNodes()` to discover network topology will fail to obtain address book data, breaking their ability to submit transactions. The mirror node's address book service becomes unavailable, indirectly disrupting smart contract client connectivity to the Hedera network. The existing Prometheus alert fires only after 75% pool utilization is sustained for 5 minutes — well after a DoS is already in effect. [8](#0-7) 

## Likelihood Explanation
The attack requires no credentials, no special protocol knowledge beyond the publicly documented gRPC endpoint, and no on-chain funds. A single attacker with a modest number of TCP connections (e.g., 20 connections × 5 concurrent calls = 100 concurrent requests) is sufficient to saturate a default 10-connection HikariCP pool. The attack is trivially repeatable and scriptable. Deployments without the GCP gateway rate limit are fully exposed.

## Recommendation
1. **Application-level rate limiting:** Apply a per-IP or global rate limit to `getNodes()` at the gRPC interceptor layer, analogous to the `ThrottleManagerImpl` in the `web3` module.
2. **Decouple transaction from cache:** Restructure `page()` so that the `@Cacheable` repository method is called *outside* the `transactionOperations.execute()` block when a cache hit is expected, or use a connection-less read path for cached results.
3. **Cache the pre-loop timestamp queries:** Add `@Cacheable` to `addressBookRepository.findLatestTimestamp()` and `nodeStakeRepository.findLatestTimestamp()` with an appropriate TTL.
4. **Enforce connection limits:** Set an explicit `maximumPoolSize` in the gRPC application configuration and configure a global `maxConcurrentCallsPerConnection` cap at the server level (not just per-connection).
5. **Enforce the GCP rate limit in all deployments:** Make `maxRatePerEndpoint` a required configuration rather than an optional one.

    private Flux<AddressBookEntry> page(AddressBookContext context) {
        return transactionOperations.execute(t -> {
            var addressBookTimestamp = context.getAddressBookTimestamp();
            var nodeStakeMap = context.getNodeStakeMap();
            var nextNodeId = context.getNextNodeId();
            var pageSize = addressBookProperties.getPageSize();
            var nodes = addressBookEntryRepository.findByConsensusTimestampAndNodeId(
                    addressBookTimestamp, nextNodeId, pageSize);
            var endpoints = new AtomicInteger(0);

            nodes.forEach(node -> {
                // Override node stake
                node.setStake(nodeStakeMap.getOrDefault(node.getNodeId(), 0L));
                // This hack ensures that the nested serviceEndpoints is loaded eagerly and voids lazy init exceptions
                endpoints.addAndGet(node.getServiceEndpoints().size());
            });

            if (nodes.size() < pageSize) {
                context.completed();
            }

            log.info(
                    "Retrieved {} address book entries and {} endpoints for timestamp {} and node ID {}",
                    nodes.size(),
                    endpoints,
                    addressBookTimestamp,
                    nextNodeId);
            return Flux.fromIterable(nodes);
        });
