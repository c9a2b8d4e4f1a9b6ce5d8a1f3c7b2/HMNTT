Audit Report

## Title
Long Overflow in `getInclusiveValue()` Causes `GT:Long.MAX_VALUE` Filter to Return All Registered Nodes

## Summary
In `rest-java/src/main/java/org/hiero/mirror/restjava/service/NetworkServiceImpl.java`, the method `resolveRegisteredNodeIdBounds()` calls `NumberRangeParameter.getInclusiveValue()` to convert a `GT` operator into an inclusive lower bound. When the input value is `Long.MAX_VALUE`, the arithmetic `value + 1` silently overflows to `Long.MIN_VALUE`. `Math.max(0L, Long.MIN_VALUE)` then evaluates to `0L`, leaving the lower bound unchanged. The range guard `lowerBound > upperBound` is never triggered, and the repository is queried with `[0, Long.MAX_VALUE]`, returning every registered node.

## Finding Description

**Step 1 ‚Äî Input passes validation.**
`NumberRangeParameter.getNumberValue()` only rejects negative values: [1](#0-0) 
`Long.MAX_VALUE` (9223372036854775807) is non-negative, so it is accepted without error.

**Step 2 ‚Äî Overflow in `getInclusiveValue()`.**
For the `GT` operator, the method returns `value + 1` using plain Java `long` arithmetic: [2](#0-1) 
`Long.MAX_VALUE + 1` wraps silently to `Long.MIN_VALUE` (-9223372036854775808). No `Math.addExact` or overflow guard is used.

**Step 3 ‚Äî `Math.max` absorbs the overflow.**
In `resolveRegisteredNodeIdBounds()`, the overflowed value is passed to `Math.max`: [3](#0-2) 
`Math.max(0L, Long.MIN_VALUE)` = `0L`. The `lowerBound` variable is never updated from its initial value of `0`.

**Step 4 ‚Äî Range guard is bypassed.**
The guard `lowerBound > upperBound` evaluates to `0 > Long.MAX_VALUE`, which is `false`: [4](#0-3) 
No exception is thrown.

**Step 5 ‚Äî Full-table query is issued.**
The repository is called with `lowerBound=0`, `upperBound=Long.MAX_VALUE`: [5](#0-4) 
This returns every non-deleted registered node, regardless of type filter.

**Note:** The identical overflow pattern exists in `getNetworkNodes()` (line 114) and `HooksController.hooksRequest()` (line 142), both of which call `getInclusiveValue()` under the same `Math.max` pattern. [6](#0-5) 

## Impact Explanation
A semantically empty filter (`GT:Long.MAX_VALUE` ‚Äî no node ID can satisfy this) is silently converted into a full-table scan. All registered nodes are returned to the caller instead of an empty result set. This violates data-integrity guarantees for downstream consumers of the mirror node API and may expose the complete registered node list to any unauthenticated caller.

## Likelihood Explanation
Exploitation requires only a single unauthenticated HTTP query parameter: `registerednode.id=gt:9223372036854775807`. The value is a valid non-negative `long` that passes all existing input validation. No special privileges, tokens, or internal access are needed. The attack is trivially repeatable and deterministic.

## Recommendation
Replace the plain arithmetic in `NumberRangeParameter.getInclusiveValue()` with overflow-safe alternatives:

1. **Use `Math.addExact`** and catch `ArithmeticException`, returning an error to the caller:
   ```java
   public long getInclusiveValue() {
       try {
           if (operator == RangeOperator.GT) return Math.addExact(value, 1);
           if (operator == RangeOperator.LT) return Math.subtractExact(value, 1);
       } catch (ArithmeticException e) {
           throw new IllegalArgumentException("Range value out of bounds for operator " + operator);
       }
       return value;
   }
   ```
2. **Or cap the input** in `getNumberValue()` to reject `Long.MAX_VALUE` when the operator is `GT` (and `0` when the operator is `LT`). The same fix should be applied to `EntityIdRangeParameter.getInclusiveValue()` and `Bound.getAdjustedLowerRangeValue()` / `getInclusiveRangeValue()`, which contain the same unchecked arithmetic.

## Proof of Concept
```
GET /api/v1/network/registeredNodes?registerednode.id=gt:9223372036854775807
```

**Expected:** Empty result (no node ID can be greater than `Long.MAX_VALUE`).

**Actual:** All registered nodes are returned.

Trace:
1. `NumberRangeParameter.valueOf("gt:9223372036854775807")` ‚Üí `new NumberRangeParameter(GT, 9223372036854775807L)` ‚Äî passes `getNumberValue` validation.
2. `range.getInclusiveValue()` ‚Üí `9223372036854775807L + 1` ‚Üí `-9223372036854775808L` (overflow).
3. `Math.max(0L, -9223372036854775808L)` ‚Üí `0L` ‚Äî `lowerBound` stays `0`.
4. `0 > Long.MAX_VALUE` ‚Üí `false` ‚Äî no exception.
5. Repository called with `[0, Long.MAX_VALUE]` ‚Üí full table returned.
