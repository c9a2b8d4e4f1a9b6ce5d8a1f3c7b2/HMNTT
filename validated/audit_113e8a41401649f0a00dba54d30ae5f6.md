# Audit Report

## Title
Cache-Bypass DoS via Unbounded Token Cache Queries on `/accounts/{id}/tokens` Endpoint

## Summary
The `getTokenAccounts()` function in `rest/service/tokenService.js` issues a secondary database query for every token ID not present in the in-memory LRU cache. Because the cache has no TTL and the endpoint has no rate limiting, an unauthenticated attacker can continuously bypass the cache using novel token IDs, sustaining a high rate of secondary `token_id = any($1)` queries that can exhaust the database connection pool and deny service to all users.

## Finding Description

**Verified code path:**

`rest/service/tokenService.js`, `getTokenAccounts()` (lines 96–115): [1](#0-0) 

1. A primary query fetches rows from `token_account` for the given account ID.
2. Token IDs from those rows are collected into a `Set`.
3. `getCachedTokens()` (lines 136–161) is called: [2](#0-1) 
   - For each token ID, the `quickLru` cache is checked.
   - Cache misses are collected into `uncachedTokenIds`.
   - If any misses exist, a secondary query `WHERE token_id = any($1)` is issued with the full array of uncached IDs — **no LIMIT clause**. [3](#0-2) 

**Three compounding weaknesses confirmed:**

1. **No rate limiting.** `rest/server.js` registers the route via `AccountRoutes.router` with no throttle middleware. The full middleware chain contains no IP-based or request-rate limiter. [8](#0-7) 

2. **`quickLru` cache has no TTL.** Constructed with only `maxSize`. Entries are only evicted by LRU pressure, never by expiry. An attacker rotating through accounts associated with tokens not yet cached will always produce cache misses. [4](#0-3) 

3. **Secondary query has no LIMIT.** `tokenCacheQuery` is unbounded; the array passed can contain up to `limit` IDs (the primary query's user-controlled limit parameter). [3](#0-2) 

## Impact Explanation

Each request to `/api/v1/accounts/{id}/tokens?limit=100` with uncached token IDs causes two DB queries: one against `token_account` and one `token_id = any($1)` against the `token` table with up to 100 IDs. Sustained parallel requests can exhaust the database connection pool, causing all subsequent queries — including unrelated endpoints — to queue or timeout. This degrades or denies service for all users of the mirror node REST API without any network-layer amplification.

## Likelihood Explanation

- **No authentication required.** `authHandler` only sets a custom response limit for known users; unauthenticated requests are passed through unconditionally. [6](#0-5) 
- **No privileges needed.** Any HTTP client can call the endpoint.
- **Trivially repeatable.** A single script cycling through account IDs with diverse token associations (publicly discoverable via the same API) is sufficient.
- **Low attacker cost.** Each request is a standard GET; the cost is asymmetric — cheap for the attacker, expensive for the database.

## Recommendation

1. **Add rate limiting middleware** (e.g., `express-rate-limit`) to the REST API, applied globally or specifically to token-related endpoints.
2. **Add a TTL to the `quickLru` cache** using the `maxAge` option so stale entries expire and the cache cannot be permanently saturated by an attacker.
3. **Add a LIMIT clause to `tokenCacheQuery`** to cap the number of rows returned by the secondary query, regardless of input array size.
4. **Cap the input array size** in `getCachedTokens()` to enforce an upper bound on `uncachedTokenIds.length` independent of the primary query's limit.
5. **Consider a connection pool size** appropriate for the expected concurrency, and add query timeouts at the application layer in addition to the DB-level `statementTimeout`.

## Proof of Concept

```
# Rotate through accounts associated with diverse, uncached token IDs
for i in $(seq 1 1000); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.$i/tokens?limit=100" &
done
wait
```

Each parallel request that hits uncached token IDs triggers a secondary `WHERE token_id = any($1)` query. With sufficient concurrency, the database connection pool is exhausted and all API endpoints become unresponsive.

  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });
