Audit Report

## Title
Unbounded In-Memory Opcode Trace Accumulation via Unauthenticated Request Causes JVM OOM

## Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint, when enabled, accepts `stack`, `memory`, and `storage` query parameters from any unauthenticated caller. Re-executing a historically large contract transaction with all three flags set causes the JVM to accumulate unbounded `Opcode` objects — each carrying full stack, memory, and storage snapshots — into a heap-resident `ArrayList` with no per-request size cap, potentially exhausting JVM heap memory.

## Finding Description

**Entry point — `OpcodesController.getContractOpcodes()`**

The endpoint at `GET /api/v1/contracts/results/{transactionIdOrHash}/opcodes` requires no authentication. The only guards are:

1. `validateAcceptEncodingHeader` — trivially satisfied by including `Accept-Encoding: gzip` in the request.
2. `throttleManager.throttleOpcodeRequest()` — a rate limiter, not a memory/size guard. [1](#0-0) 

Note: the endpoint is gated by `OpcodesProperties.enabled`, which defaults to `false`. [2](#0-1) 
When the feature is enabled in a deployment, the vulnerability is fully exposed.

**Pre-allocation proportional to gas — `OpcodeServiceImpl.processOpcodeCall()`**

`OpcodeContext` is constructed with initial capacity `gasLimit / 3`. For a 15 M-gas transaction this pre-allocates a 5,000,000-slot `ArrayList` before a single opcode is traced. [3](#0-2) [4](#0-3) 

**Per-opcode unbounded capture — `OpcodeActionTracer.tracePostExecution()`**

For every EVM opcode executed, `captureMemory`, `captureStack`, and `captureStorage` are called and the resulting `Opcode` object is appended to the list with no bound check. [5](#0-4) 

The capture methods in `AbstractOpcodeTracer` allocate new `ArrayList` and `TreeMap` instances per opcode:
- `captureStack`: up to `frame.stackSize()` hex strings (EVM max 1024 × 32 bytes each).
- `captureMemory`: full copy of EVM memory word-by-word; grows during execution.
- `captureStorage`: all modified storage slots accumulated in a `TreeMap`. [6](#0-5) 

There is no maximum opcode count, no maximum per-opcode data size, and no heap-budget check anywhere in the call chain.

## Impact Explanation
A single request targeting a historically large transaction (e.g., a complex DeFi interaction consuming 10–15 M gas) with `stack=true&memory=true&storage=true` forces the JVM to allocate several gigabytes of heap for the in-flight `ArrayList<Opcode>`. This triggers an `OutOfMemoryError`, crashing the mirror-node web3 process. Because the endpoint is unauthenticated (when enabled) and the throttle is rate-based rather than size-based, one request per second is sufficient to keep the node in a crash-restart loop, denying service to all web3 API consumers.

## Likelihood Explanation
- **No privilege required**: any HTTP client can call the endpoint when enabled.
- **Trivial bypass of gzip check**: add `Accept-Encoding: gzip` header.
- **Throttle is ineffective as a memory guard**: 1 req/s is a rate limit; one large request is enough to trigger OOM.
- **Historical transactions are permanent**: an attacker identifies a known large transaction once and replays the attack indefinitely.
- **Repeatability**: after the process restarts, the same request can be re-sent immediately.

## Recommendation
1. **Cap the opcode list size**: enforce a configurable maximum opcode count (e.g., 100,000) in `OpcodeContext.addOpcodes()`, throwing a `ResponseStatusException(429/413)` when exceeded.
2. **Cap per-opcode data**: limit stack depth captured, memory words captured, and storage entries captured per opcode to configurable maximums.
3. **Remove gas-proportional pre-allocation**: replace `gasLimit / 3` with a fixed reasonable initial capacity (e.g., 1,000) to avoid the large upfront allocation.
4. **Add authentication/authorization**: restrict the endpoint to privileged callers, since it is explicitly a debug/tracing tool.
5. **Add a request timeout**: enforce a wall-clock timeout on the EVM re-execution to bound CPU and memory usage.

## Proof of Concept
```
# 1. Identify a historically large contract transaction hash (e.g., 15M gas DeFi tx)
TX_HASH="0x<known_large_tx_hash>"

# 2. Send a single request with all flags enabled
curl -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"

# Expected result: JVM OOM / process crash on the mirror-node web3 service
# The same request can be repeated every second to prevent recovery
```

    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
    }
