Audit Report

## Title
Missing Timestamp Bound on Entity Branch in `findActiveByEvmAddressAndTimestamp` Returns Current State for Historical Queries

## Summary
The entity branch of `findActiveByEvmAddressAndTimestamp` (and its sibling `findActiveByEvmAddressOrAliasAndTimestamp`) omits the `lower(e.timestamp_range) <= ?2` predicate that is correctly present in `findActiveByIdAndTimestamp`. For any entity that has been updated after the queried block timestamp, the `ORDER BY timestamp_range DESC LIMIT 1` tie-break always selects the current `entity` table row over the correct `entity_history` row, silently returning the entity's present state instead of its historical state.

## Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`

**Affected methods:**
- `findActiveByEvmAddressAndTimestamp` — entity branch, lines 67–72
- `findActiveByEvmAddressOrAliasAndTimestamp` — entity branch, lines 103–108

The entity branch in both methods reads:
```sql
(
    select *
    from entity e
    where e.deleted is not true
    and e.id = (select id from entity_cte)
    -- MISSING: and lower(e.timestamp_range) <= ?2
)
```

The sibling method `findActiveByIdAndTimestamp` (lines 139–141) correctly includes the bound:
```sql
select *
from entity
where id = ?1 and lower(timestamp_range) <= ?2
and deleted is not true
```

**Root cause:** The CTE (lines 60–66) only guarantees `created_timestamp <= ?2`, i.e., the entity existed at some point before the queried timestamp. It does **not** guarantee that the entity's current row in the `entity` table was valid at `?2`. When an entity is updated, the old state is moved to `entity_history` with a closed range `[T0, T1)`, and the new state stays in `entity` with an open range `[T1, ∞)`. Without `lower(e.timestamp_range) <= ?2`, the entity branch returns the `[T1, ∞)` row even when `T1 > blockTimestamp`.

## Impact Explanation
Any historical EVM query resolved via an EVM address (rather than a numeric entity ID) returns the entity's current on-chain state regardless of the requested block. Concretely:
- Contract bytecode, balance, key material, and other fields reflect the *present* state, not the state at the queried block.
- An entity deleted at block B but later re-created will appear active when queried at any block between its creation and deletion.
- Audit tools, block explorers, and smart-contract replay engines relying on the mirror node's historical EVM API receive silently corrupted data, undermining the integrity of Hashgraph history.

**Severity: Medium** — read-only data-integrity corruption; no direct fund loss, but historical correctness is a core guarantee of the mirror node.

## Likelihood Explanation
- **No privilege required.** Any caller of the public web3 JSON-RPC API (`eth_call`, `eth_getCode`, `eth_getBalance`, etc.) with a `blockNumber` parameter triggers this path via `CommonEntityAccessor.getEntityByEvmAddressTimestamp` → `findActiveByEvmAddressAndTimestamp`.
- **Affects all updated entities.** Every entity that has ever been updated (i.e., has at least one `entity_history` row with a lower `timestamp_range` lower-bound than the current `entity` row) is vulnerable.
- **Trivially repeatable.** The attacker only needs a known EVM address and any historical block number predating the entity's last update.

## Recommendation
Add `and lower(e.timestamp_range) <= ?2` to the entity branch of both affected methods, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- findActiveByEvmAddressAndTimestamp, entity branch (fix)
(
    select *
    from entity e
    where e.deleted is not true
    and e.id = (select id from entity_cte)
    and lower(e.timestamp_range) <= ?2        -- ADD THIS
)
```

Apply the identical fix to `findActiveByEvmAddressOrAliasAndTimestamp`.

## Proof of Concept
**Setup:**
- Entity E with `evm_address = A` created at `T0=100`. `entity` row: `timestamp_range=[100,∞)`.
- Entity E updated at `T1=200`. `entity_history` row: `timestamp_range=[100,200)`. `entity` row: `timestamp_range=[200,∞)`, `deleted=NULL`.

**Query:** Call `eth_getCode` (or any EVM endpoint) with `blockNumber` mapping to `blockTimestamp=150` (i.e., `T0 < 150 < T1`).

**Execution path:**
1. `CommonEntityAccessor.get(address, Optional.of(150))` → `getEntityByEvmAddressTimestamp` → `findActiveByEvmAddressAndTimestamp(A, 150)`.
2. CTE resolves entity ID: `created_timestamp=100 ≤ 150` ✓.
3. Entity branch: `deleted IS NULL` passes `deleted is not true`; no timestamp guard → returns row with `timestamp_range=[200,∞)`.
4. History branch: `lower([100,200))=100 ≤ 150` → returns row with `timestamp_range=[100,200)`.
5. `ORDER BY timestamp_range DESC`: `[200,∞) > [100,200)` → entity branch wins.
6. **Result:** T1 (current) state returned for a T0.5 historical query — incorrect.

**Expected result:** The `[100,200)` history row should be returned. With the fix applied (`lower(e.timestamp_range) <= 150`), the entity branch would be excluded (`lower([200,∞))=200 > 150`), and the history row would correctly win.
