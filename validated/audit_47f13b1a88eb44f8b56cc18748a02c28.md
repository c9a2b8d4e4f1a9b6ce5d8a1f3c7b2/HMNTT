Audit Report

## Title
Unbounded Protobuf Request Body in `estimateFees()` Enables Unauthenticated DoS

## Summary
The `POST /api/v1/network/fees` endpoint in `NetworkController.java` accepts a raw `byte[]` request body with no size constraint. Spring MVC's `ByteArrayHttpMessageConverter` buffers the entire body into a heap-allocated array before the controller is invoked. The subsequent PBJ protobuf parse call may allocate additional heap objects. No application-layer size enforcement exists for this path, making the endpoint vulnerable to heap exhaustion via large unauthenticated requests.

## Finding Description

**Why existing defenses do not apply:**

1. **`@NotNull` only** — rejects a null body, not an oversized one. [2](#0-1) 

2. **`JacksonConfiguration` limits** (`maxDocumentLength(1000)`, `maxStringLength(1000)`) — apply exclusively to Jackson JSON deserialization. The protobuf binary path never touches Jackson; the body is read as raw `byte[]` via `ByteArrayHttpMessageConverter` and then parsed by PBJ (`com.hedera.pbj.runtime`), completely bypassing these constraints. [3](#0-2) 

3. **`RestJavaConfiguration`** — registers a `ProtobufHttpMessageConverter` for `com.google.protobuf.Message` subtypes and a `ShallowEtagHeaderFilter`. Neither imposes a body-size limit, and the converter does not apply to `byte[]` parameters anyway. [4](#0-3) 

4. **`MetricsFilter`** — reads the `Content-Length` header for observability only; it enforces nothing and passes the request through unconditionally. [5](#0-4) 

5. **Tomcat `maxPostSize`** — Tomcat's default 2 MB `maxPostSize` applies only to `application/x-www-form-urlencoded` bodies, not to raw binary `application/protobuf` content. No `server.tomcat.max-swallow-size` or equivalent property was found anywhere in the module.

## Impact Explanation
The Kubernetes pod for `rest-java` is configured with a hard memory limit of `2048Mi`. [6](#0-5) 

A single request with a body of several hundred megabytes, or a small number of concurrent large requests, can exhaust available JVM heap. This causes `OutOfMemoryError` or sustained GC pressure, rendering the entire pod — including all other endpoints served by it — unavailable for the duration of the outage.

## Likelihood Explanation
The endpoint is publicly reachable with no authentication requirement. No rate-limiting or body-size enforcement exists at the application layer. The attack requires only a standard HTTP client capable of streaming a large body; it is trivially scriptable, requires no protocol knowledge beyond the content type, and is repeatable at will.

## Recommendation
Enforce a maximum body size at the application layer for this endpoint. Options include:

- Add a `ContentCachingRequestWrapper`-based servlet filter that rejects requests exceeding a configured threshold (e.g., 64 KB, which is far larger than any valid signed Hedera transaction) before the body is read into heap.
- Configure `spring.servlet.multipart.max-request-size` is not applicable here, but a custom `OncePerRequestFilter` checking `Content-Length` and/or counting bytes read from the stream can be registered with higher priority than the dispatcher.
- Alternatively, replace `@RequestBody byte[] body` with a streaming approach that reads up to a fixed limit and rejects the request if the limit is exceeded, avoiding full heap allocation.
- At the infrastructure layer, configure an ingress/gateway `maxRequestBodySize` policy (the GCP `BackendPolicy` in the Helm chart does not currently set one). [7](#0-6) 

## Proof of Concept

```bash
# Generate a ~200 MB payload of random bytes
dd if=/dev/urandom bs=1M count=200 of=/tmp/large_payload.bin

# Send it to the endpoint
curl -X POST https://<host>/api/v1/network/fees \
     -H "Content-Type: application/protobuf" \
     --data-binary @/tmp/large_payload.bin

# Repeat concurrently to accelerate heap exhaustion
for i in $(seq 1 5); do
  curl -X POST https://<host>/api/v1/network/fees \
       -H "Content-Type: application/protobuf" \
       --data-binary @/tmp/large_payload.bin &
done
wait
```

Each request causes Tomcat to allocate a `byte[]` of the full body size on the JVM heap before the controller is invoked. With the 2048 MiB pod memory limit, five concurrent 200 MB requests exceed the heap budget, triggering OOM or continuous GC and making the pod unresponsive.
