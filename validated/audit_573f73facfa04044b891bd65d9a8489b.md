Audit Report

## Title
Integer Overflow in `buildNetworkFromMirrorNodes` Backoff Calculation Causes Tight Retry Loop (DoS)

## Summary
`buildNetworkFromMirrorNodes` in `pinger/mirror_node_client.go` computes exponential backoff with an uncapped bit-shift multiplication. When `HIERO_MIRROR_PINGER_MIRROR_NODE_CLIENT_MAX_RETRIES` is set to ≥ 36 (default 500 ms base) or ≥ 35 (1 s base), the `time.Duration` multiplication overflows `int64`, producing a negative duration. Go's `time.After` fires immediately for any non-positive duration, collapsing the intended backoff into a tight loop that issues unthrottled HTTP requests to the mirror node REST API.

## Finding Description
**Vulnerable code — `pinger/config.go` lines 123–128:**
```go
if cfg.mirrorNodeClientMaxRetries < 0 {
    cfg.mirrorNodeClientMaxRetries = 0
}
if cfg.mirrorNodeClientBaseBackoff <= 0 {
    cfg.mirrorNodeClientBaseBackoff = 500 * time.Millisecond
}
```
Only lower-bound checks are applied; no upper bound is enforced on either field. [1](#0-0) 

**Vulnerable code — `pinger/mirror_node_client.go` line 63:**
```go
backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
``` [5](#0-4) 

`time.Duration` is `int64` (nanoseconds). With the default base of 500 ms:

| attempt | `2^(attempt-1)` | product (ns) | fits int64? |
|---------|----------------|--------------|-------------|
| 35 | 2^34 = 17,179,869,184 | 8.59 × 10^18 | ✓ |
| 36 | 2^35 = 34,359,738,368 | 1.72 × 10^19 | **overflow → negative** |

**Why existing checks fail:** The guards in `config.go` are purely lower-bound. There is no `min(backoff, maxBackoff)` clamp in `buildNetworkFromMirrorNodes`. By contrast, `transfer.go`'s `backoff()` helper correctly caps at 30 s:
```go
func backoff(base time.Duration, attempt int) time.Duration {
    d := base * time.Duration(1<<(attempt-1))
    return min(d, 30 * time.Second)
}
``` [3](#0-2) 

`buildNetworkFromMirrorNodes` does not call this helper and has no equivalent cap. [6](#0-5) 

## Impact Explanation
After overflow, `backoff` is negative. `time.After(negative)` fires immediately (documented Go behavior: non-positive duration → channel fires at once). The `select` never blocks, and the loop spins through all remaining attempts with zero delay, issuing rapid, unthrottled HTTP GET requests to `/api/v1/network/nodes`. With `MAX_RETRIES=63`, up to ~28 attempts (36 through 63) fire with no delay at process startup. If the mirror node returns retryable errors (5xx or 429), the full burst repeats on every pinger restart or pod reschedule. At scale (multiple pinger replicas), this can exhaust mirror node REST capacity.

## Likelihood Explanation
The trigger requires setting `HIERO_MIRROR_PINGER_MIRROR_NODE_CLIENT_MAX_RETRIES` to ≥ 36 (with the default 500 ms base). Environment variables for the pinger are set in the Kubernetes Deployment spec or a ConfigMap. Any principal with `patch`/`update` RBAC on the Deployment or ConfigMap in the pinger's namespace — a role commonly granted to developers and CI pipelines — can inject these values. No application-level authentication is required; the pinger reads env vars unconditionally at startup. The misconfiguration is persistent across pod restarts and is not self-correcting.

## Recommendation
1. **Add an upper-bound cap in `buildNetworkFromMirrorNodes`** — reuse the existing `backoff()` helper from `transfer.go`, or apply an equivalent `min(backoff, maxBackoff)` clamp:
   ```go
   backoff := min(cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1)), 30*time.Second)
   ```
2. **Add an upper-bound validation in `loadConfig()`** — reject or clamp `mirrorNodeClientMaxRetries` to a safe maximum (e.g., 62 for 500 ms base, or lower for operational sanity):
   ```go
   if cfg.mirrorNodeClientMaxRetries > 62 {
       cfg.mirrorNodeClientMaxRetries = 62
   }
   ```
3. **Prefer the shared `backoff()` helper** — consolidate all backoff logic to a single, tested, capped function to prevent divergence.

## Proof of Concept
Set the following environment variables before starting the pinger:
```
HIERO_MIRROR_PINGER_MIRROR_NODE_CLIENT_MAX_RETRIES=63
HIERO_MIRROR_PINGER_MIRROR_NODE_CLIENT_BASE_BACKOFF=500ms
```
Configure the mirror REST endpoint to return HTTP 503 on every request. Observe that:
- Attempts 1–35 sleep for the expected exponential backoff durations.
- Starting at attempt 36, `backoff` is negative; `time.After` fires immediately.
- Attempts 36–63 (28 requests) are issued with no delay, constituting an unthrottled burst against the mirror node REST API.

	if strings.HasSuffix(base, "/api/v1") {
		url = base + "/network/nodes"
	} else {
		url = base + "/api/v1/network/nodes"
	}

	httpClient := &http.Client{Timeout: cfg.mirrorNodeClientTimeout}

	attempts := max(cfg.mirrorNodeClientMaxRetries + 1, 1)

	var lastErr error

	for attempt := 1; attempt <= attempts; attempt++ {
		network, retry, err := fetchMirrorNodeNetwork(ctx, httpClient, url)
		if err == nil {
			return network, nil
		}

		lastErr = fmt.Errorf("attempt %d/%d: %w", attempt, attempts, err)
		if !retry || attempt == attempts {
			break
		}

		backoff := cfg.mirrorNodeClientBaseBackoff * time.Duration(1<<(attempt-1))
		select {
		case <-ctx.Done():
			return nil, ctx.Err()
		case <-time.After(backoff):
		}
	}

	return nil, lastErr
}
