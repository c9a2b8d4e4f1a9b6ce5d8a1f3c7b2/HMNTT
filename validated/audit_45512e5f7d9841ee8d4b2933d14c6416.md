Audit Report

## Title
Gas Token Leak via Unrestored Tokens When `action()` Throws `ThrottleException` After `gasLimitBucket` Consumption

## Summary
In `ThrottleManagerImpl.throttle()`, gas tokens are consumed from `gasLimitBucket` before per-request filter actions are evaluated. If a configured `REJECT` or exhausted `THROTTLE` filter subsequently throws a `ThrottleException`, `ContractController.call()` does not catch it and never calls `restore()`, permanently losing those gas tokens. An attacker who can craft requests matching a `REJECT` filter can repeatedly drain the shared gas budget, starving legitimate users.

## Finding Description
**`ContractController.call()`** [1](#0-0)  wraps `throttleManager.throttle(request)` in a try-block that catches **only** `InvalidParametersException` and calls `restore()` exclusively for that exception type. `ThrottleException` is not caught, so `restore()` is never invoked when it propagates.

**`ThrottleManagerImpl.throttle()`** executes in this order: [2](#0-1) 

1. **Line 38–39**: `rateLimitBucket.tryConsume(1)` — throws before any gas is consumed if false. Safe.
2. **Line 40–42**: `gasLimitBucket.tryConsume(scaleGas(request.getGas()))` — if **true**, gas tokens are **permanently consumed** and execution falls through to the filter loop.
3. **Lines 44–48**: Iterates `requestFilter` list and calls `action()`.

**`action()`** throws `ThrottleException` unconditionally for `REJECT`, and for `THROTTLE` when the per-filter bucket is exhausted: [3](#0-2) 

When either branch throws, execution unwinds through `throttle()` back to `call()`. Because `ThrottleException` is not caught there, `restore()` is never invoked, and the gas tokens consumed at step 2 are permanently lost.

**Root cause**: The failed assumption is that a `ThrottleException` from `action()` means "no resources were consumed." In reality, gas tokens were already consumed at line 40 before the filter loop runs.

The `restore()` method itself is correct — it properly adds tokens back to `gasLimitBucket` [4](#0-3)  — it is simply never called on the `ThrottleException` path.

## Impact Explanation
The `gasLimitBucket` is a shared, global resource. The default `gasPerSecond` is `7_500_000_000L`, which after `scaleGas` (÷ 10,000) yields **750,000 tokens/s**. [5](#0-4)  Each request with gas = 15,000,000 drains 1,500 tokens. At the default 500 req/s rate limit, a single attacker can drain the entire gas budget without any EVM execution occurring. Legitimate users then receive `"Gas per second rate limit exceeded."` errors, constituting a complete DoS of the `/api/v1/contracts/call` endpoint.

## Likelihood Explanation
Default deployments are **not affected** because `request` defaults to `List.of()` (empty filter list). [6](#0-5)  However, operators who configure `REJECT` filters (e.g., to block known-bad contract addresses or calldata patterns) are vulnerable. Any attacker who can observe or guess those patterns — a blocked contract address is public on-chain — can exploit this with no credentials, no special role, and no prior state, using only repeated HTTP POST requests to a public endpoint.

## Recommendation
Catch `ThrottleException` in `ContractController.call()` and call `restore()` before re-throwing, mirroring the existing `InvalidParametersException` handling:

```java
} catch (ThrottleException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
```

Alternatively, restructure `ThrottleManagerImpl.throttle()` so that gas tokens are consumed **after** all filter actions have been evaluated without throwing, ensuring no tokens are consumed for requests that will be rejected.

## Proof of Concept
1. Configure a `REJECT` filter matching a specific `to` address (e.g., a known contract address).
2. Send repeated POST requests to `/api/v1/contracts/call` with `gas: 15000000` and `to` set to the blocked address.
3. Each request passes `rateLimitBucket.tryConsume(1)` (consuming a rate token), then passes `gasLimitBucket.tryConsume(1500)` (consuming 1,500 gas tokens), then hits the `REJECT` filter and throws `ThrottleException`.
4. `restore()` is never called. After ~500 such requests in one second, the 750,000-token gas budget is exhausted.
5. All subsequent legitimate requests receive `"Gas per second rate limit exceeded."` until the bucket refills, and the attacker can repeat indefinitely to maintain the DoS.
