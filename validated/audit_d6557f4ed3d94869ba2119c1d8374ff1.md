Audit Report

## Title
Persistent Double DB Query Triggered by Attacker-Controlled Successful SCHEDULECREATE Transaction

## Summary
`mayMissLongTermScheduledTransaction()` in `rest/transactions.js` lacks a time-based expiry guard. For any successful SCHEDULECREATE (type=42) transaction that has no corresponding `scheduled=true` execution row, the function returns `true` indefinitely — causing `getTransactionsByIdOrHash()` to fire a second, wider-range `pool.queryQuietly()` call on every request. An attacker who submits one such transaction can permanently double (or worsen) DB load per request for that transaction ID.

## Finding Description

**Double-query trigger** — `getTransactionsByIdOrHash()`: [1](#0-0) 

When `mayMissLongTermScheduledTransaction()` returns `true`, the upper consensus timestamp bound is replaced with `lowerConsensusTimestamp + maxScheduledTransactionConsensusTimestampRangeNs` and a second query fires.

**The missing time guard** — `mayMissLongTermScheduledTransaction()`: [2](#0-1) 

When `scheduled === undefined`, the function loops through transactions. If it finds a successful SCHEDULECREATE (`type === 42`, `result` in `SUCCESS_PROTO_IDS`) and no `transaction.scheduled === true` sibling, it returns `true`. There is **no check on how old the SCHEDULECREATE is**. The function will return `true` for a 3-year-old expired schedule just as readily as for a brand-new one.

**The time guard that exists — but only for cache headers:** [3](#0-2) 

`getTransactionsByIdOrHashCacheControlHeader()` performs the identical SCHEDULECREATE check but adds `elapsed < maxScheduledTransactionConsensusTimestampRangeNs` before acting. This guard was never ported to `mayMissLongTermScheduledTransaction()`, creating a permanent divergence: cache headers expire correctly, but the second DB query fires forever.

**Bypassed guards:**
- `isTransactionHash` guard (line 976): bypassed — attacker queries by transaction ID, not hash.
- `scheduled === false` guard (line 976): bypassed — attacker omits `?scheduled=`, leaving `scheduled` as `undefined`.
- `transaction.scheduled` loop check (line 985): bypassed — the scheduled execution never occurred, so no `scheduled=true` row exists in the result set.

## Impact Explanation
Every HTTP request to `GET /api/v1/transactions/{payerAccount}-{validStart}` for the attacker's transaction ID issues two database queries instead of one. The second query uses `maxScheduledTransactionConsensusTimestampRangeNs` as its upper bound — a value intended to cover long-term scheduled transactions (default configurable, test value shown as `86400000000000n` ns = 24 hours, but production values can be much larger) — making it a heavier scan than the first. [4](#0-3) 

With multiple crafted transactions and a sustained request rate, the attacker multiplies DB load, degrading mirror node availability for all users. The condition persists indefinitely — a single on-chain transaction enables unlimited future amplification with no further on-chain cost.

## Likelihood Explanation
The precondition is minimal: one Hedera account and one successful SCHEDULECREATE transaction (fractions of a cent on mainnet, free on testnet). The schedule can be crafted to never execute (e.g., `wait_for_expiry=true` with required signatures never gathered, or the schedule simply expires). No privileged access, no insider knowledge, and no ongoing on-chain cost is required after the initial setup. The attack is fully repeatable and scriptable.

## Recommendation
Add the same time-based guard that exists in `getTransactionsByIdOrHashCacheControlHeader()` to `mayMissLongTermScheduledTransaction()`. Specifically, inside the `scheduled === undefined` branch, when a successful SCHEDULECREATE is found, check whether `utils.nowInNs() - transaction.consensus_timestamp < maxScheduledTransactionConsensusTimestampRangeNs` before setting `scheduleExists = true`. If the elapsed time exceeds the maximum scheduled transaction range, the schedule has definitively expired and no second query is warranted.

```js
// Proposed fix in mayMissLongTermScheduledTransaction(), lines 982-984
if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
  const elapsed = utils.nowInNs() - transaction.consensus_timestamp;
  if (elapsed < maxScheduledTransactionConsensusTimestampRangeNs) {
    scheduleExists = true;
  }
}
```

This mirrors the logic already present in `getTransactionsByIdOrHashCacheControlHeader()` at lines 906–908 and closes the permanent amplification window.

## Proof of Concept

1. Create a Hedera account on testnet.
2. Submit a `SCHEDULECREATE` transaction with `wait_for_expiry=true` and a required signer whose signature will never be provided. The transaction lands on-chain with `type=42`, `result=22` (SUCCESS), and no `scheduled=true` execution row ever appears.
3. Note the transaction ID (e.g., `0.0.XXXX-<validStart>`).
4. In a loop, send `GET /api/v1/transactions/0.0.XXXX-<validStart>` to the mirror node REST API (no `?scheduled=` parameter).
5. Observe via DB query logs or metrics that each request triggers two `pool.queryQuietly()` calls — the second with an expanded upper timestamp bound — while an equivalent request for a non-SCHEDULECREATE transaction ID triggers only one.
6. Scale to multiple crafted transaction IDs and/or higher request rates to amplify DB load proportionally.

```javascript
const mayMissLongTermScheduledTransaction = (isTransactionHash, scheduled, transactions) => {
  // Note scheduled may be undefined
  if (isTransactionHash || scheduled === false) {
    return false;
  }

  if (scheduled === undefined) {
    let scheduleExists = false;
    for (const transaction of transactions) {
      if (transaction.type === scheduleCreateProtoId && SUCCESS_PROTO_IDS.includes(transaction.result)) {
        scheduleExists = true;
      } else if (transaction.scheduled) {
        return false;
      }
    }

    return scheduleExists;
  } else if (scheduled && transactions.length === 0) {
    return true;
  }

  return false;
};
```
