Audit Report

## Title
Deleted/Expired Contract Bytecode Returned via `findRuntimeBytecode()` — Missing Entity Status Check

## Summary
`ContractBytecodeReadableKVState.readFromDataSource()` returns stored `runtime_bytecode` for contracts whose corresponding `entity` row is marked `deleted = true` or has expired, because two of the three resolution paths in `toEntityId()` never consult the `entity` table before the bytecode fetch. The non-long-zero EVM address path correctly filters deleted entities; the `contractNum` and long-zero-address paths do not.

## Finding Description

**`ContractRepository.findRuntimeBytecode()`** issues a bare query against the `contract` table with no join to `entity`:

```sql
select runtime_bytecode from contract where id = :contractId
``` [1](#0-0) 

The `Contract` domain object carries no lifecycle fields (`deleted`, `expiration_timestamp`); those columns exist only in the `entity` table: [2](#0-1) 

**`ContractBytecodeReadableKVState.readFromDataSource()`** calls `findRuntimeBytecode()` unconditionally after resolving the entity ID — no deleted/expired guard anywhere in the method: [3](#0-2) 

**Inside `toEntityId()`, two paths perform no entity-table lookup:**

- **`contractID.hasContractNum()` path** (line 50–51): resolves the ID purely from the protobuf contract number via `entityIdFromContractId()` — no `entity` row is read, so `deleted` is never checked.
- **Long-zero EVM address path** (line 54–55): derives the ID from the address bytes via `DomainUtils.fromEvmAddress()` — again, no `entity` row is read. [4](#0-3) 

**Contrast with the non-long-zero EVM address path** (lines 57–60): `commonEntityAccessor.getEntityByEvmAddressAndTimestamp()` is called, which delegates to `findByEvmAddressAndDeletedIsFalse()` — deleted contracts are correctly excluded for this path only. [5](#0-4) [6](#0-5) 

The asymmetry is confirmed: `findByIdAndDeletedIsFalse` and `findByEvmAddressAndDeletedIsFalse` exist and are used elsewhere, but are never invoked for the `contractNum` / long-zero-address resolution paths before bytecode is fetched. [7](#0-6) 

## Impact Explanation

`ContractBytecodeReadableKVState` is the authoritative bytecode source for the mirror node's EVM state layer, used during `eth_getCode`, `eth_call`, and `eth_estimateGas` processing.

- **`eth_getCode`** returns non-empty bytecode for a deleted contract, violating EVM semantics (deleted contracts must return `0x`).
- **`eth_call` / `eth_estimateGas`** targeting a deleted contract's long-zero address will find bytecode present and attempt to execute the contract's logic, producing results that diverge from canonical network state.
- Downstream tooling (DeFi protocols, wallets, indexers) that uses `eth_getCode != 0x` as a liveness check will incorrectly treat deleted contracts as live.

## Likelihood Explanation

Exploitation requires only a standard JSON-RPC call with no authentication:
```
eth_getCode(<deleted_contract_long_zero_address>, "latest")
```
Contract IDs are sequential and publicly enumerable on the network. Additionally, the result is cached via `@Cacheable` on `findRuntimeBytecode()`, meaning a single successful call poisons the cache for all subsequent callers until eviction. [8](#0-7) 

## Recommendation

Before calling `findRuntimeBytecode()`, validate entity liveness for all three resolution paths. For the `contractNum` and long-zero-address paths, add a call to `entityRepository.findByIdAndDeletedIsFalse(entityId.getId())` (or the equivalent `commonEntityAccessor.get(entityId, timestamp)`) and return `null` if the entity is absent or deleted. This mirrors the guard already applied in the non-long-zero EVM address path via `getEntityByEvmAddressAndTimestamp()`. [9](#0-8) 

## Proof of Concept

1. Deploy a contract on the network; note its numeric ID (e.g., `0.0.12345`).
2. Delete the contract via a `ContractDelete` transaction.
3. Confirm the `entity` table row has `deleted = true`.
4. Call `eth_getCode` with the long-zero address `0x000000000000000000000000000000000000303d` (encoding of `12345`).
5. The mirror node resolves the entity ID via `DomainUtils.fromEvmAddress()` (no entity-table check), then calls `findRuntimeBytecode(12345)`, which returns the stored bytecode from the `contract` table.
6. The response is non-empty bytecode instead of `0x`, confirming the bug. [10](#0-9)

    private EntityId toEntityId(@NonNull final ContractID contractID) {
        if (contractID.hasContractNum()) {
            return entityIdFromContractId(contractID);
        } else if (contractID.hasEvmAddress()) {
            final var evmAddress = contractID.evmAddress().toByteArray();
            if (isLongZeroAddress(evmAddress)) {
                return DomainUtils.fromEvmAddress(evmAddress);
            } else {
                return commonEntityAccessor
                        .getEntityByEvmAddressAndTimestamp(evmAddress, Optional.empty())
                        .map(Entity::toEntityId)
                        .orElse(EntityId.EMPTY);
            }
        }
        return EntityId.EMPTY;
    }
```
