# Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` via Unchecked Narrowing Cast of `Long.MAX_VALUE` Limit

## Summary
`PollingTopicListener.poll()` performs an unchecked narrowing cast `(int)(filter.getLimit() - context.getCount().get())` at line 55. When a client supplies `limit = Long.MAX_VALUE`, the cast silently produces `-1`. This value is propagated into a rebuilt `TopicMessageFilter`, where `hasLimit()` returns `false`, causing `findByFilter` to issue an unbounded database query against the topic message table.

## Finding Description

In `PollingTopicListener.java` lines 54â€“59:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [1](#0-0) 

**Root cause:** `filter.getLimit()` is a `long`. When it equals `Long.MAX_VALUE` and `context.getCount().get()` is `0`:

- `Long.MAX_VALUE - 0 = 0x7FFFFFFFFFFFFFFF`
- Truncated to 32 bits: `0xFFFFFFFF` â†’ signed `int`: **`-1`**

`Math.min(-1, maxPageSize)` = `-1`, so `pageSize = -1`.

**Why validation does not catch this:** `TopicMessageFilter.limit` carries only `@Min(0)`: [2](#0-1) 

`Long.MAX_VALUE` is positive and passes `@Min(0)`. The `@Validated` annotation on the class triggers Spring AOP method-level validation; it does **not** validate objects constructed via Lombok's `toBuilder().limit(-1).build()`, so the corrupted `newFilter` is created without any constraint violation.

**`hasLimit()` with `-1`:** [3](#0-2) 

`-1 > 0` is `false`, so `hasLimit()` returns `false`.

**Downstream in `findByFilter`:** [4](#0-3) 

Because `hasLimit()` is `false`, `typedQuery.setMaxResults(...)` is never called. The JPA query is issued with no row cap, returning the entire matching topic message history.

## Impact Explanation

On every polling interval for the lifetime of the subscription, `findByFilter` executes an unbounded `SELECT` against the `topic_message` table for the targeted topic. A topic with a large message history will return all rows, potentially exhausting JVM heap and causing an out-of-memory condition or severe latency degradation for all subscribers sharing the same mirror node process. This directly affects the availability and integrity of the consensus topic stream.

## Likelihood Explanation

The Hedera gRPC `ConsensusTopicQuery` proto exposes a `uint64 limit` field. `Long.MAX_VALUE` (`9223372036854775807`) is a valid `uint64` value. Any unauthenticated gRPC client can set this field. No privilege is required. The attack is trivially repeatable: one subscription request per target topic triggers the overflow on every polling interval for the lifetime of that subscription.

## Recommendation

Replace the unchecked narrowing cast with a safe clamped computation:

```java
long remaining = filter.getLimit() - context.getCount().get();
// Clamp to [0, Integer.MAX_VALUE] before narrowing
int limit = filter.hasLimit()
        ? (int) Math.min(Math.max(remaining, 0L), Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
```

Additionally, add an `@Max` constraint to `TopicMessageFilter.limit` to reject unreasonably large values at the API boundary:

```java
@Min(0)
@Max(Long.MAX_VALUE / 2)   // or a domain-appropriate cap
private long limit;
``` [2](#0-1)
