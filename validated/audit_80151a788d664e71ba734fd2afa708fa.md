Audit Report

## Title
Gas Throttle Bypass via Full Restoration on Out-of-Gas Exceptional Halt

## Summary
The mirror node's gas bucket throttle is completely neutralized for out-of-gas transactions. `restoreGasToBucket()` unconditionally returns 100% of the consumed gas to the bucket whenever `gasUsed == gasLimit` (the out-of-gas condition), resulting in zero net gas bucket consumption per request. Combined with the intentional disabling of consensus-node-level gas throttles, the only remaining defense is the 500 req/s rate limit, allowing an attacker to force maximum EVM computation on every request.

## Finding Description

**1. Consensus-node gas throttles are intentionally disabled.**

`buildTransactionProperties()` in `EvmProperties.java` hardcodes:

```java
props.put("contracts.throttle.throttleByGas", "false");
props.put("executor.disableThrottles", "true");
``` [1](#0-0) 

This is by design — the mirror node implements its own gas throttle via bucket4j. However, this makes the mirror node's own gas bucket the **sole** gas-level defense.

**2. The gas bucket throttle is fully bypassed for out-of-gas transactions.**

`restoreGasToBucket()` in `ContractCallService.java`:

```java
final var gasLimitToRestoreBaseline = (long)(gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
// gasLimitRefundPercent defaults to 100 → gasLimitToRestoreBaseline = gasLimit

if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
    throttleManager.restore(gasLimitToRestoreBaseline);  // restores 100% of gas limit
``` [2](#0-1) 

`gasLimitRefundPercent` defaults to `100`: [3](#0-2) 

In standard EVM, an out-of-gas exceptional halt sets `gasUsed = gasLimit`. The condition `!result.isSuccessful() && gasLimit == result.gasUsed()` evaluates to `true`, and `restore(gasLimit)` is called. `ThrottleManagerImpl.restore()` calls `gasLimitBucket.addTokens(scaleGas(gas))`: [4](#0-3) 

**Net gas bucket consumption per out-of-gas request: zero tokens.**

**3. Token arithmetic confirms the bypass.**

`scaleGas(15_000_000) = 15_000_000 / 10_000 = 1_500` tokens consumed on entry, then 1,500 tokens restored on exit. [5](#0-4) 

Bucket capacity is `scaleGas(7_500_000_000) = 750_000` tokens. [6](#0-5) 

**4. Root cause — failed assumption.**

The code comment reads: *"If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent of the gasLimit value back in the bucket."* [7](#0-6) 

The failed assumption is that a failed transaction represents "wasted" or "cheap" work. In reality, an out-of-gas transaction is the **most expensive** case — the EVM executed the full gas budget before halting. Restoring 100% of the gas limit eliminates the throttle's protective effect for exactly this class of transactions.

Note: for reverts (`gasUsed < gasLimit`), the else branch correctly restores only the remaining gas (`gasLimit - gasUsed`), so reverts are not affected. [8](#0-7) 

## Impact Explanation
With the gas bucket fully bypassed for out-of-gas transactions, an attacker can force the node to execute up to `500 × 15,000,000 = 7.5 billion gas units of EVM computation per second` (bounded only by the 500 req/s rate limit). This causes CPU exhaustion, thread pool saturation, and denial of service to legitimate users. The `maxGasLimit` has no effective upper bound beyond `@Min(21_000)` from the user's perspective: [9](#0-8) 

## Likelihood Explanation
`POST /api/v1/contracts/call` is a public, unauthenticated endpoint. The attack requires only a request body with `gas=15000000` and bytecode for an infinite loop (e.g., `JUMPDEST JUMP`, two opcodes). No funds, no privileged account, and no prior knowledge beyond the public API spec are required. It is trivially repeatable at 500 req/s from a single machine.

## Recommendation
1. **Fix `restoreGasToBucket()`**: Do not restore gas for out-of-gas transactions. The condition should distinguish between reverts (which return remaining gas and deserve partial restoration) and out-of-gas halts (which consumed all gas and deserve no restoration). Change the condition to only restore gas when `result == null` (pre-execution failure), not when `gasUsed == gasLimit`.
2. **Reduce `gasLimitRefundPercent` default**: Lower the default from 100% to a value that reflects realistic unused-gas ratios (e.g., 25–50%).
3. **Enforce a tighter `maxGasLimit`**: Consider a lower operator-configurable ceiling to reduce per-request EVM work.

## Proof of Concept
```
POST /api/v1/contracts/call
{
  "data": "0x5b56",   // JUMPDEST JUMP — infinite loop, exhausts all gas
  "gas": 15000000,
  "to": "0x0000000000000000000000000000000000000001"
}
```
Send this at 500 req/s. Each request:
1. Consumes 1,500 tokens from the gas bucket (pre-execution).
2. EVM runs the infinite loop until out-of-gas; `gasUsed = gasLimit = 15,000,000`.
3. `restoreGasToBucket()` detects `!isSuccessful() && gasLimit == gasUsed` → restores 1,500 tokens.
4. Net bucket consumption: **0 tokens**.

The gas bucket never depletes. The node performs full 15,000,000-gas EVM execution on every request indefinitely, limited only by the 500 req/s rate limit.

        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);

        }
