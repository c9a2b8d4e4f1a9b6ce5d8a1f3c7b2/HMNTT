# Audit Report

## Title
Deleted Contract Bytecode Returned in EVM Simulation Due to Missing Entity Deletion Check in `findRuntimeBytecode`

## Summary
`ContractRepository.findRuntimeBytecode()` queries only the `contract` table with no join to the `entity` table and no `deleted` predicate. Because migration `V1.64.2` moved the `deleted` column exclusively to `entity` and dropped it from `contract`, the query unconditionally returns bytecode for logically deleted contracts. The `contractNum` and long-zero EVM address resolution paths in `ContractBytecodeReadableKVState.readFromDataSource()` invoke this query with no prior deletion guard, while the non-long-zero EVM address path is correctly protected. Any caller can trigger incorrect EVM simulation results for a deleted contract via a standard `eth_call` or `eth_estimateGas` request.

## Finding Description
**Vulnerable query** — `ContractRepository.findRuntimeBytecode` issues:
```sql
select runtime_bytecode from contract where id = :contractId
```
There is no `JOIN entity ON entity.id = contract.id AND entity.deleted IS NOT TRUE`. Because `deleted` was dropped from `contract` by migration `V1.64.2`, the `contract` table has no lifecycle flag at all, so the query always returns bytecode if a row exists.

**Schema root cause** — Migration `V1.64.2__merge_contract_entity.sql` migrated `deleted` (and all other lifecycle metadata) from `contract` into `entity`, then executed `DROP COLUMN IF EXISTS deleted` on `contract`. The `Contract` domain object confirms no `deleted` field remains.

**Unguarded call chain** — `ContractBytecodeReadableKVState.readFromDataSource()` resolves a `ContractID` to an `EntityId` via `toEntityId()`, then immediately calls `contractRepository.findRuntimeBytecode(entityId.getId())`. For the `contractNum` branch (`contractID.hasContractNum()`) and the long-zero EVM address branch (`isLongZeroAddress(evmAddress)`), `toEntityId()` performs a pure arithmetic conversion with no database lookup and no deletion check before the bytecode query is issued.

**Asymmetric protection** — The non-long-zero EVM address branch calls `commonEntityAccessor.getEntityByEvmAddressAndTimestamp(evmAddress, Optional.empty())`, which resolves to `entityRepository.findByEvmAddressAndDeletedIsFalse(addressBytes)` — a deletion-aware query. The `contractNum` and long-zero EVM address branches have no equivalent guard.

## Impact Explanation
The mirror node's EVM simulation (`eth_call`, `eth_estimateGas`, opcode tracing) returns non-empty `Bytecode` for deleted contracts when addressed by numeric ID or long-zero EVM address. Consequently:
- `EXTCODESIZE` returns a non-zero value for a deleted contract.
- `EXTCODECOPY` copies the deleted contract's code.
- Direct calls execute the deleted contract's logic within the simulation.

This causes the mirror node to export simulation results that diverge from consensus-node behavior, which returns empty bytecode for deleted contracts. Downstream consumers (wallets, dApps, indexers) relying on mirror node simulation will receive incorrect state information about deleted contracts, violating the mirror node's core correctness guarantee.

## Likelihood Explanation
Contract deletion is a normal, observable on-chain event requiring no attacker involvement. The deleted contract's numeric ID is publicly visible on-chain. The trigger is a standard, unauthenticated JSON-RPC call (`eth_call`). The `contract` row is never cleaned up after deletion, so the window is indefinite and the condition is trivially repeatable by any user.

## Recommendation
Add a deletion check before returning bytecode for the `contractNum` and long-zero EVM address paths. The fix should be applied at one of two levels:

1. **Query level**: Modify `findRuntimeBytecode` to join `entity` and filter on `deleted IS NOT TRUE`:
   ```sql
   SELECT c.runtime_bytecode
   FROM contract c
   JOIN entity e ON e.id = c.id
   WHERE c.id = :contractId
     AND e.deleted IS NOT TRUE
   ```

2. **Application level**: In `ContractBytecodeReadableKVState.readFromDataSource()`, resolve the entity via `commonEntityAccessor` (which already enforces deletion checks) for all three branches before calling `findRuntimeBytecode`, mirroring the protection already present in the non-long-zero EVM address branch.

## Proof of Concept
1. Deploy contract `0.0.X` on-chain; confirm `entity.deleted = false`, `contract.runtime_bytecode` is populated.
2. Delete contract `0.0.X` on-chain; confirm `entity.deleted = true`. The `contract` row is unchanged.
3. Issue `eth_call` targeting the contract by numeric ID (resolves to long-zero address `0x000...0X`):
   ```json
   {"jsonrpc":"2.0","method":"eth_call","params":[{"to":"0x000000000000000000000000000000000000000X","data":"0x"},"latest"],"id":1}
   ```
4. `ContractBytecodeReadableKVState.readFromDataSource()` enters the `contractNum` or long-zero EVM address branch, calls `findRuntimeBytecode(X)`, and receives the stored bytecode.
5. The EVM simulation treats the address as a live contract and executes its code, returning a non-empty result instead of the empty response that the consensus network would produce.
