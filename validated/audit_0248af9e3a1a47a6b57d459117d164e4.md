# Audit Report

## Title
Missing `close` Event Handler Causes Permanent `inFlightCounter` Inflation via Abrupt Client Disconnection

## Summary
In `metricsMiddleware()` (`rest/middleware/metricsHandler.js`), `inFlightCounter.add(1)` is called unconditionally for every non-metrics request, but `inFlightCounter.add(-1)` is placed exclusively inside a `res.on('finish', ...)` handler. Because Node.js does not emit `'finish'` when the underlying TCP socket is destroyed before `res.end()` completes, any client that sends a valid HTTP request and immediately issues a TCP RST causes the counter to be permanently incremented without a corresponding decrement.

`http.ServerResponse` extends `stream.Writable`. The `'finish'` event is emitted only after `end()` is called on a non-destroyed stream. When a client sends a TCP RST before the server calls `res.end()`, the socket is destroyed, `res.destroyed` becomes `true`, any subsequent `res.end()` call is a no-op, and `'finish'` is never emitted. The `'close'` event fires instead, but there is no `res.on('close', ...)` handler in this file, and no `on-finished` package is used anywhere in the REST middleware layer.

## Impact Explanation
The `api_all_request_in_processing_total` UpDownCounter is exported via the OpenTelemetry Prometheus exporter and consumed by the mirror node's observability pipeline. A permanently inflated value produces incorrect in-flight request records. The counter never resets (it is a process-lifetime accumulator), so corruption persists until process restart. Operators relying on this metric for autoscaling, alerting, or capacity planning receive false data. All other per-request metrics (`allRequestCounter`, `durationHistogram`, `requestSizeHistogram`, `responseSizeHistogram`) are also skipped for abruptly-closed connections since they share the same `'finish'` handler.

## Likelihood Explanation
No privileges, credentials, or special network position are required. Any internet-reachable instance is exploitable with a standard TCP socket using `SO_LINGER` with `l_linger=0` on socket close. The attack is trivially scriptable, repeatable at high frequency, and leaves no meaningful application-layer trace. The only prerequisite is that `config.metrics.enabled` is `true`, which is the default operational configuration.

## Recommendation
Replace the `res.on('finish', ...)` listener with a handler that fires on both `'finish'` and `'close'`, ensuring the decrement and all per-request metric recordings execute regardless of how the connection terminates. The standard approach is to use the `on-finished` npm package (already common in Express middleware ecosystems), or to listen on both events with a guard flag:

```js
let settled = false;
const settle = () => {
  if (settled) return;
  settled = true;
  inFlightCounter.add(-1);
  // ... rest of metric recording
};
res.on('finish', settle);
res.on('close', settle);
```

## Proof of Concept
```js

const socket = net.createConnection({port: 3000, host: 'target'}, () => {
```
