# Audit Report

## Title
Inconsistent Deleted-Entity Filter in `getByEvmAddressAndType()` Exposes Deleted Account State via Long-Form EVM Address

## Summary
`getByEvmAddressAndType()` in `EntityServiceImpl` routes long-form EVM addresses (first 12 bytes zero) through the unfiltered `CrudRepository.findById()`. Every other lookup path enforces `deleted is not true` at the SQL level. An unauthenticated caller can retrieve the full persisted state of a deleted account by supplying a crafted 20-byte address whose first 12 bytes are zero and whose last 8 bytes encode the target entity ID.

## Finding Description

**Vulnerable code path — `EntityServiceImpl.java` lines 34–41:** [1](#0-0) 

When `buffer.getInt() == 0 && buffer.getLong() == 0` (line 37), the remaining 8 bytes are read as a `long` entity ID and passed directly to `entityRepository.findById()` (line 38). No deleted-entity filter is applied before or after this call.

**Root cause — `findById()` carries no deleted predicate:** [2](#0-1) 

`findByAlias()` (line 13) and `findByEvmAddress()` (line 16) both embed `deleted is not true` in their native SQL. `findById()` is inherited from `CrudRepository<Entity, Long>` and issues a plain `SELECT * FROM entity WHERE id = ?` with no such predicate.

**No post-fetch guard in `AccountController`:** [3](#0-2) 

The result of `getByEvmAddressAndType()` is mapped directly to the view model with no check on the `deleted` flag.

## Impact Explanation
Any unauthenticated caller who supplies a crafted long-form EVM address receives the full persisted `Entity` row for a deleted account — including `deleted = true`, `pendingReward`, balance, keys, memo, and staking metadata. The API is explicitly designed to suppress deleted-entity state (evidenced by the `deleted is not true` guards on `findByAlias` and `findByEvmAddress`). API consumers that rely on a null/empty response to infer account non-existence are misled. Because the mirror node is read-only, no funds can be moved; the concrete impact is unauthorized information disclosure of deleted-account state.

## Likelihood Explanation
Preconditions are minimal:
- The attacker needs only a numeric entity ID, which is trivially enumerable from public ledger data.
- The crafted address is deterministic: 12 zero bytes followed by the 8-byte big-endian encoding of the entity ID.
- No authentication, no special role, and no on-chain transaction are required.
- The attack is repeatable for every deleted account ID on the network.

## Recommendation
Add a `deleted is not true` predicate to the long-form branch. The simplest fix is a post-fetch filter mirroring what the other paths already do:

```java
// EntityServiceImpl.java — line 38
return entityRepository.findById(buffer.getLong())
        .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))  // add this
        .filter(e -> e.getType() == type);
```

Alternatively, add a custom repository method `findByIdAndDeletedFalse` with an explicit `deleted is not true` predicate, consistent with the style of `findByAlias` and `findByEvmAddress`.

Note: `getByIdAndType()` (line 24–26) uses the same unfiltered `findById()` and should receive the same treatment. [4](#0-3) 

## Proof of Concept

1. Identify a deleted account with numeric entity ID `N` (e.g., `12345`).
2. Encode `N` as an 8-byte big-endian value: `0x0000000000003039` for `12345`.
3. Construct the 20-byte EVM address: `0x000000000000000000000000 0000000000003039` (12 zero bytes + 8-byte ID).
4. Send the GraphQL query:
   ```graphql
   query {
     account(input: { evmAddress: "0x0000000000000000000000000000000000003039" }) {
       id
       balance
       pendingReward
       deleted
     }
   }
   ```
5. The response returns the full deleted account row, including `deleted: true` and `pendingReward`, instead of `null`.

        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
