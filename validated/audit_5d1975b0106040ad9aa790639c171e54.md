Audit Report

## Title
Malicious Block Node Can Cause Intermittent Ingestion Stalls via Deliberate `serverStatus()` Response Timing Near Deadline

## Summary
`BlockNode.getBlockRange()` applies a configurable gRPC deadline (default 400 ms) to `serverStatus()` calls but catches all resulting exceptions silently, returning `EMPTY_BLOCK_RANGE` without invoking `onError()`. Because `onError()` is the sole mechanism that increments the error counter and eventually sets `active = false`, a block node that deliberately delays every `serverStatus()` response to just past the deadline is permanently treated as active, bypassing the `maxSubscribeAttempts`/`readmitDelay` circuit-breaker entirely.

## Finding Description

**Root cause:** `getBlockRange()` in `BlockNode.java` (lines 99–112) catches `Exception` and returns `EMPTY_BLOCK_RANGE` without calling `onError()`:

```java
// BlockNode.java:99-112
public Range<Long> getBlockRange() {
    try {
        final var blockNodeService = BlockNodeServiceGrpc.newBlockingStub(channel)
                .withDeadlineAfter(streamProperties.getResponseTimeout()); // 400 ms
        final var response = blockNodeService.serverStatus(SERVER_STATUS_REQUEST);
        ...
    } catch (Exception ex) {
        log.error("Failed to get server status for {}", this, ex);
        return EMPTY_BLOCK_RANGE;   // no onError() here
    }
}
``` [2](#0-1) 

`onError()` (lines 196–207) is only reachable from the two catch blocks inside `streamBlocks()`: [7](#0-6) 

A node that never reaches the streaming phase — because it always times out at the status check — never increments `errors` and never sets `active = false`.

**Call chain:**
1. `BlockNodeSubscriber.getNode()` iterates all active nodes and calls `hasBlock()` for each. [5](#0-4) 
2. `hasBlock()` calls `node.getBlockRange()` and returns `false` when the range is empty. [4](#0-3) 
3. If no active node satisfies `hasBlock()`, the loop falls through to inactive nodes; if still none, throws `BlockStreamException("No block node can provide block …")`.

**Failed assumption:** The design assumes that a `serverStatus()` timeout is a transient network event that will eventually resolve, so no penalty is applied. A deliberate, repeatable delay invalidates this assumption.

**Why the circuit-breaker fails:** `maxSubscribeAttempts` and `readmitDelay` are checked exclusively inside `onError()`, which is only called from `streamBlocks()`. A node that always times out at `getBlockRange()` never enters `streamBlocks()` and is therefore invisible to the circuit-breaker. [8](#0-7) 

## Impact Explanation

- **Single malicious node (only configured node):** Every `getNode()` call blocks for the full `responseTimeout` (400 ms by default), then throws `BlockStreamException`. Ingestion stalls indefinitely.
- **Malicious node among multiple nodes:** The malicious node is tried first if it sorts highest by priority. Each `getNode()` call wastes up to 400 ms before falling through to a healthy node, imposing a sustained latency floor of one `responseTimeout` per block.
- **Intermittent variant:** If the response arrives at exactly the deadline boundary, behavior is non-deterministic under system load — sometimes the node appears to have the block, sometimes not — producing unpredictable node selection and sporadic `BlockStreamException` throws indistinguishable from legitimate failures.

## Likelihood Explanation

An attacker must control a block node that is either statically configured in the importer's config file or registered in the `registered_node` table (auto-discovery path, enabled via `blockProperties.isAutoDiscoveryEnabled()`). [9](#0-8) 

In a permissioned deployment this requires admin action. In a semi-open deployment where node registration is accessible to participants, any registered node operator can execute this attack. The timing manipulation is trivial: the server-side gRPC handler sleeps for `responseTimeout - ε` ms before responding. No cryptographic material, no special network position, and no memory-safety exploit is required. The attack is fully repeatable and requires no ongoing effort once configured.

## Recommendation

1. **Call `onError()` on `getBlockRange()` failure.** The simplest fix is to invoke `onError()` inside the `catch` block of `getBlockRange()`, so that repeated status-check timeouts accumulate against the same error counter and eventually trigger the `active = false` / `readmitDelay` circuit-breaker.

2. **Separate timeout for status checks vs. streaming.** Consider a shorter, non-configurable deadline for `serverStatus()` (e.g., 100 ms) distinct from the streaming timeout, to limit the wall-clock cost of each failed probe.

3. **Bounded retry before exclusion.** Track consecutive `getBlockRange()` failures per node independently of `streamBlocks()` failures, and apply the same `maxSubscribeAttempts` threshold before marking the node inactive.

## Proof of Concept

1. Configure a block node endpoint that, on every `serverStatus()` gRPC call, sleeps for `responseTimeout - 1 ms` (e.g., 399 ms) before returning a deadline-exceeded error.
2. Set this node as the sole (or highest-priority) configured node in the importer.
3. Observe that `getNode()` blocks for ~400 ms on every invocation, always returns `EMPTY_BLOCK_RANGE`, and throws `BlockStreamException("No block node can provide block …")`.
4. Observe that `node.isActive()` remains `true` throughout — the node is never marked inactive, and the error counter stays at zero — confirming the circuit-breaker is never triggered.

        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
