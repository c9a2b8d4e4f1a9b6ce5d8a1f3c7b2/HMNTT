# Audit Report

## Title
Missing EVM Address Length Validation Causes Unhandled `BufferUnderflowException` in `getByEvmAddressAndType()`

## Summary
`decodeEvmAddress()` in `GraphQlUtils.java` imposes no length constraint on the decoded byte array. `getByEvmAddressAndType()` in `EntityServiceImpl.java` unconditionally reads 4 + 8 + 8 = 20 bytes from the resulting `ByteBuffer` without a prior length check. Supplying a valid hex string shorter than 20 bytes triggers a `java.nio.BufferUnderflowException` that is not handled by `CustomExceptionResolver`, causing Spring GraphQL's default handler to return an `INTERNAL_ERROR` classification instead of the expected `ValidationError`.

## Finding Description

**`decodeEvmAddress()` — no length check:**

`GraphQlUtils.decodeEvmAddress()` strips the `0x` prefix and decodes hex, but never asserts that the resulting byte array is exactly 20 bytes. [1](#0-0) 

**`getByEvmAddressAndType()` — unconditional reads:**

The method wraps the decoded bytes in a `ByteBuffer` and immediately calls `getInt()` (4 bytes) and two `getLong()` calls (8 bytes each), totalling 20 bytes, with no guard on `evmAddressBytes.length`. [2](#0-1) 

**`AccountController.account()` — no pre-validation:**

The raw `evmAddress` string is forwarded directly to `getByEvmAddressAndType()` without any length check. [3](#0-2) 

**`CustomExceptionResolver` — gap in exception handling:**

Only `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException` are mapped to `ValidationError`. `BufferUnderflowException` (a `RuntimeException`) matches none of these, so `resolveToSingleError()` returns `null`. [4](#0-3) 

**Test evidence confirming arbitrary-length output:**

The existing test suite explicitly exercises `decodeEvmAddress()` with 5-byte inputs (e.g., `"0x0000AaAaAa"`), confirming the function makes no length guarantee. [6](#0-5) 

## Impact Explanation

- **Error oracle**: An attacker can distinguish "invalid hex" (returns `ValidationError`) from "valid hex but wrong length" (returns `INTERNAL_ERROR`), leaking internal parsing behavior.
- **Unhandled exception path**: The exception bypasses the application's own error-handling contract defined by `CustomExceptionResolver`, violating the intended security boundary.
- **Server-side log pollution**: Each crafted request writes a full stack trace to server logs, which can obscure legitimate errors. *(Note: per `SECURITY.md`, stack trace disclosure is listed as out of scope; the error oracle and unhandled exception path remain in scope.)*

## Likelihood Explanation

- No authentication is required; the GraphQL `account` query is publicly accessible.
- The trigger is trivial: any valid hex string shorter than 20 bytes with leading zero bytes suffices (e.g., `"0x00000000"`).
- The condition is fully deterministic and requires only a single request.

## Recommendation

1. **Enforce length in `decodeEvmAddress()`**: After decoding, assert `bytes.length == 20`; throw `IllegalArgumentException` otherwise. This ensures the exception is caught by `CustomExceptionResolver` and returned as a `ValidationError`.
2. **Add a guard in `getByEvmAddressAndType()`**: Check `evmAddressBytes.length == 20` before wrapping in `ByteBuffer`, throwing `IllegalArgumentException` if the check fails.
3. **Optionally extend `CustomExceptionResolver`**: Add `BufferUnderflowException` to the handled types as a defence-in-depth measure, so any future similar oversight still produces a `ValidationError` rather than `INTERNAL_ERROR`.

## Proof of Concept

**Trigger `BufferUnderflowException` on `getLong()` (4-byte input):**
```
POST /graphql
{ "query": "{ account(input: { evmAddress: \"0x00000000\" }) { id } }" }
```
Flow: `decodeEvmAddress("0x00000000")` → `[0x00,0x00,0x00,0x00]` (4 bytes) → `buffer.getInt()` returns `0` (buffer now empty) → `buffer.getLong()` throws `BufferUnderflowException` → `CustomExceptionResolver` returns `null` → response: `INTERNAL_ERROR`.

**Trigger on second `getLong()` (12-byte input):**
```
POST /graphql
{ "query": "{ account(input: { evmAddress: \"0x000000000000000000000000\" }) { id } }" }
```
Flow: 12 bytes → `getInt()` = 0 (8 remaining) → first `getLong()` = 0 (0 remaining) → second `getLong()` throws `BufferUnderflowException` → `INTERNAL_ERROR`.

**Contrast with invalid hex (returns `ValidationError`, not `INTERNAL_ERROR`):**
```
{ "query": "{ account(input: { evmAddress: \"0xZZZZ\" }) { id } }" }
```
`decodeEvmAddress()` throws `IllegalArgumentException` → caught by `CustomExceptionResolver` → `ValidationError`.

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);
        } catch (DecoderException e) {
            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
        }
    }

        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);

    protected GraphQLError resolveToSingleError(Throwable ex, DataFetchingEnvironment env) {
        if (ex instanceof IllegalStateException
                || ex instanceof IllegalArgumentException
                || ex instanceof MirrorNodeException) {
            return GraphqlErrorBuilder.newError()
                    .errorType(ErrorType.ValidationError)
                    .message(ex.getMessage())
                    .path(env.getExecutionStepInfo().getPath())
                    .location(env.getField().getSourceLocation())
                    .build();
        } else {
            return null;
        }
    }
