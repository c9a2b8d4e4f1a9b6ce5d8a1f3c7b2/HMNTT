Audit Report

## Title
`getHookStorageChange()` Passes Raw `keys` Instead of `keysInRange` to Repository, Bypassing Key-Range Filter

## Summary
In `HookServiceImpl.getHookStorageChange()`, the code correctly computes `keysInRange` (the subset of caller-supplied keys that fall within `[keyLowerBound, keyUpperBound]`) but then passes the original, unfiltered `keys` collection to `hookStorageChangeRepository.findByKeyInAndTimestampBetween()`. The SQL query for that method contains no key-range predicate, so every key in the raw list — including those outside the declared range — is fetched and returned to the caller. The non-historical code path (`getHookStorage()`) correctly uses `keysInRange`; the historical path does not.

## Finding Description
**File:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/HookServiceImpl.java`
**Function:** `getHookStorageChange()`, lines 84–118

```java
// line 89 – raw user-supplied keys
final var keys = request.getKeys();
final boolean requestHasKeys = !keys.isEmpty();

// line 91 – correctly filtered subset (never used below)
final var keysInRange = request.getKeysInRange();

// line 93 – guard only fires when ALL keys are out of range
if (keysInRange.isEmpty() && requestHasKeys) {
    return new HookStorageResult(ownerId, List.of());
}

// lines 103-105 – BUG: passes raw `keys`, not `keysInRange`
if (requestHasKeys) {
    changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
            ownerId.getId(), hookId, keys,   // <-- should be keysInRange
            timestampLowerBound, timestampUpperBound, page);
}
```

`getKeysInRange()` (`HookStorageRequest.java` lines 43–52) filters keys using `Arrays.compareUnsigned` against `keyLowerBound` and `keyUpperBound`. The guard at line 93 only short-circuits when `keysInRange` is *completely* empty. When at least one key is in range, the guard passes, but the subsequent repository call still uses the full `keys` collection.

The SQL query `findByKeyInAndTimestampBetween` (`HookStorageChangeRepository.java` lines 41–63) contains no key-range predicate:

```sql
where owner_id = :ownerId
  and hook_id  = :hookId
  and key in (:keys)          -- no range check
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
```

By contrast, the non-historical path at lines 72–79 correctly passes `keysInRange` to its repository method.

## Impact Explanation
Any caller can retrieve historical `HookStorage` change records for arbitrary keys belonging to any `(ownerId, hookId)` pair. The attacker includes one key that genuinely falls within `[keyLowerBound, keyUpperBound]` (to pass the `keysInRange.isEmpty()` guard) alongside any number of out-of-range keys they wish to read. All of those out-of-range keys are returned by the database because the SQL has no range predicate. This leaks historical key-value state that the key-range filter was designed to restrict.

## Likelihood Explanation
No privileges are required — only the ability to send an HTTP request to the mirror-node REST API with a `timestamp` parameter (making `isHistorical()` return `true`) and a `keys` list. The condition is trivially satisfied: include one key inside the declared range and any number of keys outside it. The bug is deterministic and fully reproducible.

## Recommendation
Replace `keys` with `keysInRange` on line 105 of `HookServiceImpl.java`:

```java
// Before (buggy)
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keys, ...);

// After (fixed)
changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
        ownerId.getId(), hookId, keysInRange, ...);
```

This mirrors the already-correct non-historical path at lines 78–79.

## Proof of Concept
1. Identify a valid `(ownerId, hookId)` pair and a key `K_in` known to satisfy `keyLowerBound ≤ K_in ≤ keyUpperBound`.
2. Choose any key `K_out` outside that range whose historical value you wish to read.
3. Send a request with `timestamp=<some range>`, `keys=[K_in, K_out]`, `keyLowerBound=<lower>`, `keyUpperBound=<upper>`.
4. `getKeysInRange()` returns `[K_in]` — non-empty, so the guard at line 93 does not fire.
5. `findByKeyInAndTimestampBetween()` is called with the full list `[K_in, K_out]`.
6. The database returns rows for both `K_in` and `K_out`; `K_out`'s historical data is disclosed despite being outside the declared range.

        if (keys.isEmpty()) {
            return List.of();

                .toList();
