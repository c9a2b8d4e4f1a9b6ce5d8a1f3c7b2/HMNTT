# Audit Report

## Title
Reference Identity Check Bypass in `findByBlockType()` Allows Crafted `BlockType` to Skip `findLatest()` and Return Empty

## Summary
`RecordFileServiceImpl.findByBlockType()` guards the `findLatest()` path using Java reference equality (`==`) against the `BlockType.LATEST` singleton. Because `BlockType` is a Java `record`, `BlockType.of("9223372036854775807")` constructs a **new** instance with `number = Long.MAX_VALUE` that is not the same object reference as `LATEST`. The guard fails, `findByIndex(Long.MAX_VALUE)` is called instead, and `Optional.empty()` is returned to every caller expecting the latest block.

## Finding Description

**`BlockType.LATEST` singleton definition:** [1](#0-0) 

**Decimal path in `BlockType.of()` constructs a new instance, not the singleton:** [2](#0-1) 

The regex group `(\\d{1,20})` matches `9223372036854775807` (19 digits). `Long.parseLong("9223372036854775807", 10)` equals `Long.MAX_VALUE` without overflow, so no exception is thrown and `new BlockType("9223372036854775807", Long.MAX_VALUE)` is returned — a distinct object from `LATEST`. [3](#0-2) 

**Reference equality check in `findByBlockType()`:** [4](#0-3) 

The crafted instance passes neither the `EARLIEST` nor the `LATEST` guard, and is not a hash, so execution falls through to `findByIndex(Long.MAX_VALUE)`.

**`findByIndex` query:** [5](#0-4) 

No real record file will ever have `index = 9223372036854775807`, so the query always returns `Optional.empty()`.

**Root cause:** Java `record` types auto-generate value-based `equals()`, but the guard uses `==` (reference identity). The correct check would be `.equals()` or a numeric comparison (`block.number() == Long.MAX_VALUE`).

## Impact Explanation
Every caller of `findByBlockType()` that receives `Optional.empty()` when it expects the latest block will either throw an exception, return incorrect data, or silently fail. Ethereum-compatible JSON-RPC calls (`eth_call`, `eth_getBlockByNumber`, etc.) that pass the numeric equivalent of `"latest"` will not return the latest block state, producing incorrect or missing responses to mirror node consumers. Severity: **Medium** — data integrity issue; no funds are at risk, but protocol-level responses are incorrect.

## Likelihood Explanation
Any unprivileged external user can trigger this by passing the decimal string `"9223372036854775807"` as the block parameter in a JSON-RPC request. The `@JsonCreator` annotation on `BlockType.of()` means this path is directly reachable from deserialized HTTP request bodies. [6](#0-5) 

No authentication, special role, or prior knowledge of internal state is required. The condition is trivially and repeatably triggerable.

## Recommendation
Replace the reference equality checks in `findByBlockType()` with value-based comparisons. The cleanest fix is to change the guard to use `.equals()`:

```java
} else if (BlockType.LATEST.equals(block)) {
    return recordFileRepository.findLatest();
}
```

Alternatively, normalize in `BlockType.of()`: when the parsed decimal value equals `Long.MAX_VALUE`, return the `LATEST` singleton directly (mirroring how `blockTypeForTag` returns `LATEST` for `"latest"`). This is consistent with how `EARLIEST` is handled for `0`. [7](#0-6) 

## Proof of Concept

1. Send an HTTP request to any JSON-RPC endpoint that accepts a block parameter, e.g.:
   ```json
   {"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["9223372036854775807", false],"id":1}
   ```
2. `BlockType.of("9223372036854775807")` is invoked via `@JsonCreator`, returning `new BlockType("9223372036854775807", Long.MAX_VALUE)`.
3. `RecordFileServiceImpl.findByBlockType()` evaluates `block == BlockType.LATEST` → `false` (different object reference).
4. `recordFileRepository.findByIndex(9223372036854775807L)` is called; the JPQL query `select r from RecordFile r where r.index = ?1` matches no row.
5. `Optional.empty()` is returned; the caller fails to retrieve the latest block.
