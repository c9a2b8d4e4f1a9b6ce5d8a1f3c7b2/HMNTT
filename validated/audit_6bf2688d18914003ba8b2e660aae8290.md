## Audit Report

## Title
Intentional Statement-Timeout Suppression Creates Silent DoS Blind Spot in REST API

## Summary
Any unauthenticated user can send requests that trigger PostgreSQL statement timeouts. Two independent, intentional suppressions compound: the error handler reduces log detail for timeout errors, and the Loki `RestLogErrors` alert rule explicitly filters them out. A sustained flood of timeout-inducing requests exhausts the connection pool and renders the REST API unavailable with no automated operator alert.

## Finding Description

**Code path 1 — reduced log detail on statement timeout:**

`rest/middleware/httpErrorHandler.js` defines the suppression regex at line 10: [1](#0-0) 

`shouldPrintStacktrace()` returns `false` when the error message matches: [2](#0-1) 

For a `DbError`, `shouldReturnMessage()` returns `false` (it only returns `true` for client errors or `BAD_GATEWAY`): [3](#0-2) 

So execution falls into the `else` branch, where `shouldPrintStacktrace()` controls whether the full error object or only `err.message` is logged: [4](#0-3) 

**Code path 2 — DB error wrapping:**

`DbError` is imported in `rest/utils.js` and the pool configuration reads `statement_timeout` and `maxConnections` from config: [5](#0-4) 

**Code path 3 — Loki alert suppression:**

The `RestLogErrors` alert rule explicitly excludes log lines containing the PostgreSQL statement-timeout string: [6](#0-5) 

**Root cause:** The design treats statement timeouts as routine noise and intentionally suppresses both log detail and alerting for them. This assumption fails when an attacker deliberately and continuously triggers timeouts to exhaust the connection pool, causing all legitimate queries to also time out — with no alert firing.

## Impact Explanation

With the connection pool exhausted by attacker-controlled slow queries, all legitimate REST API requests return `503 Service Unavailable`. Because the Loki alert rule filters out exactly these errors, operators receive no automated notification. The only signal is manual log inspection. The reduced log detail (message only, no stack trace) also makes post-incident forensics harder.

## Likelihood Explanation

No authentication is required. The attacker only needs to identify REST API endpoints that generate expensive DB queries (e.g., wide timestamp range scans). Sending concurrent requests to such endpoints is trivial and can be sustained indefinitely. The attack is repeatable, cheap, and requires no special knowledge beyond the public API documentation.

## Recommendation

1. **Decouple the alert suppression from the error type.** The `RestLogErrors` Loki rule should not unconditionally exclude statement-timeout errors. Instead, alert when the *rate* of statement-timeout errors exceeds a threshold (e.g., more than N per minute), distinguishing organic slow queries from a flood.
2. **Add a dedicated alert for sustained statement-timeout rates.** A separate Loki rule scoped to `"canceling statement due to statement timeout"` with a rate threshold would catch pool-exhaustion attacks without generating noise from occasional slow queries.
3. **Implement rate limiting for unauthenticated requests** on endpoints known to generate expensive queries, or enforce tighter query parameter bounds (e.g., reduce `maxTimestampRange`).
4. **Restore stack-trace logging for statement timeouts** or at minimum log the originating query/endpoint, so post-incident forensics can distinguish an attack from organic load.

## Proof of Concept
