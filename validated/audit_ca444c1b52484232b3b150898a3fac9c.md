Audit Report

## Title
Unbounded Growth of `cacheManagerSlotsPerContract` Internal Cache Registry via Unique `contractId` Flooding

## Summary
In `ContractStateServiceImpl.findStorageBatch()`, each unique `contractId` that misses the outer bounded `contractSlotsCache` triggers `cacheManagerSlotsPerContract.getCache(contractId.toString())`. Because `cacheManagerSlotsPerContract` is a `CaffeineCacheManager` configured without `setCacheNames()` (dynamic mode), Spring permanently registers every new cache name in its internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no eviction mechanism. The outer `contractSlotsCache` (bounded at `maximumSize=3000`) silently evicts entries without triggering any cleanup in `cacheManagerSlotsPerContract`, causing the internal registry to accumulate `CaffeineCache` objects indefinitely.

## Finding Description

**Exact code location:**

`ContractStateServiceImpl.java` lines 85–87: [1](#0-0) 

```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause — two-layer cache mismatch:**

- `contractSlotsCache` (from `cacheManagerContractSlots`) is configured at `EvmConfiguration.java` lines 75–81 with `setCacheNames(Set.of(CACHE_NAME))` and `maximumSize=3000` from `CacheProperties.java` line 25. It is a **bounded** Caffeine cache mapping `contractId → CaffeineCache`. [2](#0-1) [3](#0-2) 

- `cacheManagerSlotsPerContract` is configured at `EvmConfiguration.java` lines 107–112 **without** `setCacheNames()`, placing it in dynamic mode. In dynamic mode, `CaffeineCacheManager.getCache(name)` creates a new `CaffeineCache` and stores it permanently in an internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has **no size limit and no eviction**. [4](#0-3) 

- The `slotsPerContract` spec `"expireAfterAccess=5m,maximumSize=1500"` (line 39 of `CacheProperties.java`) applies only to entries *within* each `CaffeineCache` instance, not to the `CaffeineCache` objects themselves in `cacheManagerSlotsPerContract`'s internal registry. [5](#0-4) 

**No contract existence validation:**

`ContractStorageReadableKVState.readFromDataSource()` only checks `slotKey.hasContractID()` (line 33) before forwarding to `contractStateService.findStorage()`. There is no validation that the contract address corresponds to an existing on-chain entity. [6](#0-5) 

**Exploit flow:**

1. Attacker sends `eth_call` or `eth_getStorageAt` with a unique `contractId` not in `contractSlotsCache`.
2. The value loader `() -> cacheManagerSlotsPerContract.getCache(contractId.toString())` is invoked.
3. `CaffeineCacheManager.getCache()` creates a new `CaffeineCache` and inserts it into `cacheManagerSlotsPerContract.cacheMap` under `contractId.toString()`.
4. The `CaffeineCache` reference is stored in `contractSlotsCache` (bounded at 3000).
5. When `contractSlotsCache` reaches `maximumSize=3000` and evicts an old entry, the corresponding `CaffeineCache` object in `cacheManagerSlotsPerContract.cacheMap` is **never removed** — it remains permanently referenced, preventing GC.
6. Repeating with N unique `contractId` values causes `cacheManagerSlotsPerContract.cacheMap` to accumulate N entries indefinitely.

## Impact Explanation
Each entry permanently retained in `cacheManagerSlotsPerContract.cacheMap` holds a `String` key and a `CaffeineCache` wrapping a Caffeine cache instance configured with `expireAfterAccess` and `maximumSize`, which allocates internal data structures (timer wheel, write buffer, etc.) on first use. With millions of unique `contractId` values, the JVM heap is exhausted, causing `OutOfMemoryError` and crashing the mirror node web3 service. This denies EVM query service (`eth_call`, `eth_estimateGas`, contract storage reads) to all users.

## Likelihood Explanation
An attacker needs only to send repeated JSON-RPC requests referencing different contract addresses. Since the mirror node does not validate contract existence before invoking `findStorage`, the attacker can enumerate arbitrary 20-byte EVM addresses at zero on-chain cost. No privileged access, no HBAR spend, and no special tooling is required — a simple HTTP client loop suffices. The attack is persistent across JVM restarts as the cache is re-populated on each new request.

## Recommendation
1. **Register cache names eagerly**: Call `setCacheNames(Set.of(...))` on `cacheManagerSlotsPerContract` with a fixed set of names, or switch to a design that does not use `CaffeineCacheManager` in dynamic mode for per-contract caches.
2. **Add a removal listener**: Register a Caffeine removal listener on `contractSlotsCache` that calls a cleanup method to remove the corresponding entry from `cacheManagerSlotsPerContract` when an entry is evicted.
3. **Validate contract existence**: In `ContractStorageReadableKVState.readFromDataSource()`, reject requests for non-existent contract addresses before they reach `findStorageBatch()`.
4. **Use a single bounded cache**: Replace the two-layer design with a single `Cache<EntityId, Cache<ByteBuffer, byte[]>>` backed by a Caffeine cache that supports bounded nested structures, or use a composite key `(contractId, slotKey)` in a single flat cache.

## Proof of Concept
```python
import requests, itertools, struct

RPC_URL = "http://<mirror-node-web3>:8545"

def eth_get_storage_at(address):
    payload = {
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [address, "0x0", "latest"], "id": 1
    }
    requests.post(RPC_URL, json=payload, timeout=5)

# Enumerate unique contract addresses (0x0000...0001, 0x0000...0002, ...)
for i in itertools.count(1):
    addr = "0x" + format(i, '040x')  # unique 20-byte address
    eth_get_storage_at(addr)
    # Each call causes cacheManagerSlotsPerContract.cacheMap to grow by 1 entry
    # After >3000 calls, entries are orphaned in cacheMap but never GC'd
    # JVM heap exhaustion occurs after sufficient iterations
```

Each iteration permanently adds one `CaffeineCache` object to `cacheManagerSlotsPerContract.cacheMap`. After the outer `contractSlotsCache` fills (3000 entries) and begins evicting, the orphaned `CaffeineCache` objects accumulate in the registry, consuming heap until `OutOfMemoryError`.

    protected SlotValue readFromDataSource(@NonNull SlotKey slotKey) {
        if (!slotKey.hasContractID()) {
            return null;
        }

        final var timestamp = ContractCallContext.get().getTimestamp();
        final var contractID = slotKey.contractID();
        final var entityId = EntityIdUtils.entityIdFromContractId(contractID);
        final var keyBytes = slotKey.key().toByteArray();
        return timestamp
                .map(t -> contractStateService.findStorageByBlockTimestamp(
                        entityId, Bytes32.wrap(keyBytes).trimLeadingZeros().toArrayUnsafe(), t))
                .orElse(contractStateService.findStorage(entityId, keyBytes))
                .map(byteArr ->
                        new SlotValue(Bytes.wrap(leftPadBytes(byteArr, Bytes32.SIZE)), Bytes.EMPTY, Bytes.EMPTY))
                .orElse(null);
    }
