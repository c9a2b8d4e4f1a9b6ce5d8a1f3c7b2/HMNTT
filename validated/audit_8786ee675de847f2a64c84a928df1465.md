# Audit Report

## Title
Cache Key Excludes Authorization Header — Authenticated User's Elevated-Limit Response Served to Unauthenticated Users

## Summary
`responseCacheCheckHandler` generates Redis cache keys using only an MD5 hash of `req.originalUrl`. Because `authHandler` can raise the effective per-request result limit above the global maximum for authenticated users, a response produced under elevated privileges is stored and later served verbatim to any unauthenticated client that issues the same URL, completely bypassing the configured result-count tier.

## Finding Description

**Middleware execution order** (`rest/server.js` lines 85–98):
1. `authHandler` (line 86) — on valid Basic credentials, calls `httpContext.set(userLimitLabel, user.limit)`, raising the effective limit above `responseLimit.max`.
2. `responseCacheCheckHandler` (line 97) — on a Redis hit, immediately writes the stored response to the client and returns, skipping all downstream route handlers.

**Cache key generation** (`rest/middleware/responseCacheHandler.js` lines 151–153):
```js
const cacheKeyGenerator = (req) => {
  return crypto.createHash('md5').update(req.originalUrl).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```
The `Authorization` header is never read or included.

**Limit enforcement** (`rest/utils.js` lines 533–553): `getEffectiveMaxLimit()` reads `userLimitLabel` from `httpContext`. When an authenticated user with `user.limit = 10` requests `/api/v1/transactions?limit=10` against a system whose global `responseLimit.max = 2`, the route handler returns up to 10 records. `responseCacheUpdateHandler` stores this response in Redis under the URL-only key.

**Cache hit for unauthenticated user**: A subsequent unauthenticated request to the same URL hits the cache. `responseCacheCheckHandler` reads the `CachedApiResponse`, sets `cache-control: public, max-age=<ttl>` (line 60), and sends the full 10-record body. No limit re-evaluation occurs.

## Impact Explanation
The `users[].limit` configuration is the sole mechanism for differentiating data-volume access between user tiers. The cache completely nullifies it for any URL a privileged user has previously requested within the Redis TTL window. The `cache-control: public` header additionally instructs downstream proxies and CDNs to cache and re-serve the elevated response to further unauthenticated clients, amplifying the exposure beyond the originating server.

## Likelihood Explanation
Exploitation requires no special tooling or privileges. An attacker needs only to:
- Know or guess a URL a privileged user has recently requested — trivially achievable since the API URL space is documented and finite.
- Issue an identical GET request within the Redis TTL window.

The attack is fully passive, repeatable on every cache refresh cycle, and leaves no distinguishing trace beyond a normal HTTP request in access logs.

## Recommendation
Include the authenticated user's tier (or a boolean `elevated` flag derived from the presence of a valid `Authorization` header) in the cache key. A minimal fix:

```js
const cacheKeyGenerator = (req) => {
  const userLimit = httpContext.get(userLimitLabel) ?? 'default';
  const base = req.originalUrl + '|limit-tier:' + userLimit;
  return crypto.createHash('md5').update(base).digest('hex') + CACHE_KEY_VERSION_SUFFIX;
};
```

Alternatively, disable caching entirely for requests that carry an `Authorization` header, or change `cache-control` from `public` to `private` for authenticated responses so downstream proxies do not re-serve them.

## Proof of Concept
1. Configure the server with `responseLimit.max = 2` and one user `{ username: "premium", password: "secret", limit: 10 }` (matching the integration test setup in `rest/__tests__/integration/authentication.test.js` lines 19–23).
2. Send an authenticated request:
   ```
   GET /api/v1/transactions?limit=10
   Authorization: Basic cHJlbWl1bTpzZWNyZXQ=
   ```
   Response: 10 transaction records. Redis now stores this response under `md5("/api/v1/transactions?limit=10") + "-v1"`.
3. Within the Redis TTL, send an unauthenticated request:
   ```
   GET /api/v1/transactions?limit=10
   ```
   Response: the cached 10-record body, with `cache-control: public, max-age=<ttl>` — 5× the permitted unauthenticated limit.

const responseCacheCheckHandler = async (req, res) => {
  const startTime = res.locals[requestStartTime] || Date.now();
  const responseCacheKey = cacheKeyGenerator(req);
  const cachedTtlAndValue = await getCache().getSingleWithTtl(responseCacheKey);

  if (!cachedTtlAndValue) {
    res.locals[responseCacheKeyLabel] = responseCacheKey;
    return;
  }

  const {ttl: redisTtl, value: redisValue} = cachedTtlAndValue;
  const cachedResponse = Object.assign(new CachedApiResponse(), redisValue);
  const conditionalHeader = req.get(CONDITIONAL_HEADER);
  const clientCached = conditionalHeader && conditionalHeader === cachedResponse.headers[ETAG_HEADER]; // 304
  const statusCode = clientCached ? httpStatusCodes.UNMODIFIED.code : cachedResponse.statusCode;
  const isHead = req.method === 'HEAD';

  let body;
  const headers = {
    ...cachedResponse.headers,
    ...{[CACHE_CONTROL_HEADER]: `public, max-age=${redisTtl}`},
  };

  if (isHead || clientCached) {
    if (clientCached) {
      delete headers[contentTypeHeader];
    } else {
      // For HEAD requests when status code is not 304, negotiate the encoding and set corresponding headers
      negotiate(cachedResponse, req, res);
    }
  } else {
    const useCompressed = negotiate(cachedResponse, req, res);
    body = useCompressed ? cachedResponse.getBody() : cachedResponse.getUncompressedBody();
  }

  res.set(headers);
  res.status(statusCode);
  if (body !== undefined) {
    res.send(body);
  } else {
    res.end();
  }

  const elapsed = Date.now() - startTime;
  logger.info(
    `${req.ip} ${req.method} ${req.originalUrl} from cache (ttl: ${redisTtl}) in ${elapsed} ms: ${statusCode}`
  );
};

const getEffectiveMaxLimit = () => {
  const userLimit = httpContext.get(userLimitLabel);
  return userLimit !== undefined ? userLimit : responseLimit.max;

/**
 * Gets the limit param value, if not exists, return the default; otherwise cap it at max.
 * Note if values is an array, the last one is honored.
 * @param {string[]|string} values Values of the limit param
 * @return {number}
 */
const getLimitParamValue = (values) => {
  let ret = responseLimit.default;
  if (values !== undefined) {
    const value = Array.isArray(values) ? values[values.length - 1] : values;
    const parsed = Number(value);
    const maxLimit = getEffectiveMaxLimit();
    ret = parsed > maxLimit ? maxLimit : parsed;
  }
  return ret;
};
