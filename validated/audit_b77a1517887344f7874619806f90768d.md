Audit Report

## Title
`Optional.empty()` Incorrectly Cached in `CACHE_MANAGER_RECORD_FILE_INDEX` Due to Flawed `unless` Condition on `findByIndex()`

## Summary
`RecordFileRepository.findByIndex()` is annotated with `@Cacheable(unless = "#result == null")` but returns `Optional<RecordFile>`. Spring Data JPA never returns a raw `null` from an `Optional<T>` method — it returns `Optional.empty()`, which is a non-null singleton. The `unless` guard therefore never fires on a cache miss, and `Optional.empty()` is stored in the Caffeine-backed `CACHE_MANAGER_RECORD_FILE_INDEX` for the full 10-minute TTL. Any caller that queries a block index before it is ingested will receive a stale "not found" response for up to 10 minutes after the block lands in the database.

## Finding Description

**Exact location:**
`web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 27–29: [1](#0-0) 

```java
@Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")
@Query("select r from RecordFile r where r.index = ?1")
Optional<RecordFile> findByIndex(long index);
```

**Root cause:** Spring's `@Cacheable` `unless` SpEL expression is evaluated against the actual return value. For a method declared `Optional<T>`, the return value is never `null`; it is either `Optional.of(value)` or `Optional.empty()`. `Optional.empty()` is a non-null singleton, so `#result == null` is always `false` on a miss, the `unless` condition never suppresses caching, and the empty result is written into the cache.

**Cache configuration confirmed** at `EvmConfiguration.java` lines 156–158: [2](#0-1) 

`expireAfterWrite(10, TimeUnit.MINUTES)` / `maximumSize(10000)`.

**Call path confirmed** at `RecordFileServiceImpl.java` line 28: [3](#0-2) 

`findByBlockType(BlockType)` delegates to `recordFileRepository.findByIndex(block.number())` for any numeric or hex block identifier.

**Why `@CachePut` on `findByTimestamp` does not mitigate this:**
The `@CachePut` at line 47 writes to `CACHE_MANAGER_RECORD_FILE_INDEX` using the default key, which is the method argument — a nanosecond-precision consensus timestamp. [4](#0-3) 

`findByIndex` is keyed by block index (e.g. `50000000`). These are entirely different numeric domains; a `findByTimestamp` call never overwrites a poisoned `findByIndex` cache entry.

## Impact Explanation
Any block queried by index before ingestion is invisible to all callers for up to 10 minutes after ingestion completes. Downstream consumers relying on `eth_getBlockByNumber` or any service path through `findByBlockType` receive an empty response and may skip or misreport the block. With `maximumSize(10000)`, an attacker can pre-poison up to 10,000 distinct future block indices in a single sweep, extending the denial-of-visibility window across a large range of upcoming blocks.

## Likelihood Explanation
No authentication or special privilege is required. The JSON-RPC endpoint is publicly accessible. Block numbers are strictly sequential and trivially predictable from the current chain tip. A single automated script issuing `eth_getBlockByNumber` for the next N block numbers is sufficient to poison the cache. After each 10-minute expiry the attacker can re-query to re-poison, making the attack continuously repeatable.

## Recommendation
Change the `unless` condition on `findByIndex` (and all other `Optional<T>`-returning `@Cacheable` methods in this repository) from:

```java
unless = "#result == null"
```

to:

```java
unless = "#result == null || !#result.isPresent()"
```

This prevents `Optional.empty()` from being stored in the cache while still allowing populated `Optional` values to be cached normally. The same fix should be applied to `findEarliest()` and `findLatest()`, which share the identical flawed pattern. [5](#0-4) 

## Proof of Concept
1. Identify the current chain tip block number `N` (e.g. via `eth_blockNumber`).
2. Issue `eth_getBlockByNumber` for blocks `N+1` through `N+10000`. Each call reaches `RecordFileServiceImpl.findByBlockType` → `recordFileRepository.findByIndex(index)`. The DB returns no row; Spring Data JPA returns `Optional.empty()`; the `unless` guard does not fire; `Optional.empty()` is written into `CACHE_MANAGER_RECORD_FILE_INDEX` with a 10-minute TTL.
3. Wait for the importer to ingest blocks `N+1` … `N+10000`.
4. Re-issue `eth_getBlockByNumber` for any of those block numbers. Spring serves the cached `Optional.empty()` — the block appears not to exist — until the TTL expires, regardless of what is now in the database.
