Audit Report

## Title
Gas Throttle Bucket Fully Bypassed via Deliberate Out-of-Gas Exceptional Halt

## Summary
`restoreGasToBucket()` in `ContractCallService` unconditionally restores 100% of `gasLimit` back to the `gasLimitBucket` whenever a transaction result is unsuccessful **and** `gasUsed == gasLimit`. An out-of-gas EVM halt satisfies both conditions by definition. With the default `gasLimitRefundPercent = 100`, the net token consumption from the gas throttle bucket is zero for every such request, completely neutralizing the gas-per-second throttle. The only remaining protection is the request-rate bucket (default 500 req/s).

## Finding Description

**Throttle entry point** — `ThrottleManagerImpl.java` line 40: [1](#0-0) 

Tokens consumed = `scaleGas(request.getGas())` = `floor(gasLimit / 10_000)`.

**Restore logic** — `ContractCallService.java` lines 140–152: [2](#0-1) 

With `gasLimitRefundPercent = 100` (the default): [3](#0-2) 

`gasLimitToRestoreBaseline = gasLimit`. When the branch at line 144 fires, `restore(gasLimit)` is called.

**Restore implementation** — `ThrottleManagerImpl.java` lines 59–64: [4](#0-3) 

`scaleGas(gasLimit)` tokens are added back — identical to what was consumed — leaving the bucket at the same level as before the request.

**Root cause:** The condition `!result.isSuccessful() && gasLimit == result.gasUsed()` is satisfied by **two distinct scenarios** that the code conflates:

1. **Pre-check failures** (no EVM execution): `gasUsed` is reported as `gasLimit` by convention because no result was produced. Restoring gas here is correct — no computation occurred.
2. **Out-of-gas EVM halts** (full EVM execution): the EVM runs until it exhausts the supplied gas, setting `gasUsed = gasLimit` and marking the result as failed. Restoring gas here is incorrect — real CPU and memory were consumed.

The code comment at line 141 ("If the transaction fails, gasUsed is equal to gasLimit") conflates both cases. The existing test `ercPrecompileExceptionalHaltReturnsExpectedGasToBucket` explicitly confirms that the full-refund path fires for exceptional halts: [5](#0-4) 

The test verifies `restore(TRANSACTION_GAS_LIMIT * gasLimitRefundPercent / 100)` is called — i.e., the full gas limit is restored — treating an out-of-gas halt identically to a pre-check failure.

**Call path:** `ContractController.call()` → `throttleManager.throttle(request)` → `contractExecutionService.processCall(params)` → `doProcessCall()` → `restoreGasToBucket()` [6](#0-5) [7](#0-6) 

## Impact Explanation

The `gasLimitBucket` is the primary defense against high-gas-cost request floods on the mirror node's EVM simulation endpoint (`/api/v1/contracts/call`). With this bypass, an attacker can send requests at the full request-rate limit (default 500 req/s) with gas limits up to `maxGasLimit` (default 15,000,000) without ever depleting the gas bucket: [8](#0-7) 

The EVM still executes until it runs out of gas on each call, consuming real CPU and memory on the server. The gas throttle provides zero protection; only the weaker request-rate bucket (500 req/s) remains. This enables sustained resource-exhaustion (DoS) of the endpoint by any unauthenticated caller.

## Likelihood Explanation

No privileges, authentication, or special contract deployment is required. Any caller can send a standard `eth_call`-style POST to `/api/v1/contracts/call` with a `gas` value that:
- Is greater than `GAS_SCALE_FACTOR` (10,000) so that `scaleGas` returns a non-zero value and the gas bucket check is triggered, and
- Is insufficient to complete the target contract's execution, causing an out-of-gas halt.

The condition is trivially reproducible, stateless, and repeatable at 500 req/s. No on-chain state is required. [9](#0-8) 

## Recommendation

Distinguish between pre-check failures and out-of-gas EVM halts in `restoreGasToBucket`. One approach:

- **Pre-check failure** (`result == null`): restore `gasLimitToRestoreBaseline` (current behavior is correct — no EVM work was done).
- **Out-of-gas EVM halt** (`!result.isSuccessful() && gasLimit == result.gasUsed()`): treat identically to a successful execution — restore only `min(gasRemaining, gasLimitToRestoreBaseline)`. Since `gasRemaining = gasLimit - gasUsed = 0` in this case, **no tokens are restored**, correctly charging the full gas cost to the bucket.
- **Revert** (`!result.isSuccessful() && gasLimit != result.gasUsed()`): current else-branch behavior is correct.

Additionally, consider lowering the default `gasLimitRefundPercent` from 100 to a value that limits the maximum refundable gas even in the pre-check-failure case. [2](#0-1) 

## Proof of Concept

```
POST /api/v1/contracts/call
Content-Type: application/json

{
  "to": "<address of any deployed contract with non-trivial execution>",
  "gas": 50000,        // > 10_000 (passes scaleGas), but too low to complete execution
  "data": "<calldata that requires > 50000 gas to execute>"
}
```

1. `throttle()` consumes `scaleGas(50000) = 5` tokens from `gasLimitBucket`.
2. EVM executes until it exhausts 50,000 gas → out-of-gas halt → `result.isSuccessful() = false`, `result.gasUsed() = 50000 = gasLimit`.
3. `restoreGasToBucket()` fires the full-refund branch: `restore(50000 * 100 / 100 = 50000)`.
4. `restore()` adds `scaleGas(50000) = 5` tokens back → net bucket change = 0.
5. Repeat at 500 req/s indefinitely. The gas bucket is never depleted. The EVM executes fully on every request.

```java
    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
```
