Audit Report

## Title
Unbounded Batch DB Query Amplification via Cached Slot Key Accumulation in `findStorageBatch()`

## Summary
`ContractStateServiceImpl.findStorageBatch()` issues a `SELECT ... WHERE slot IN (...)` query containing every slot key ever accumulated in the per-contract Caffeine cache — not just the one being requested. Because non-existent slot keys are never written to `contractStateCache`, the early-return guard in `findStorage()` never fires for them, causing `findStorageBatch()` to be invoked on every single call and the IN-clause to grow up to the configured maximum of 1,500 elements.

## Finding Description

**Entry point — `findStorage()` (lines 58–70):**

`findStorage()` checks `contractStateCache` first. The guard at line 65 returns early only when `cachedValue != null && cachedValue != EMPTY_VALUE`. [1](#0-0) 

For a slot key that does not exist in the database, `contractStateCache.get(...)` always returns `null`, so the guard never fires and execution always falls through to `findStorageBatch()`.

**Inside `findStorageBatch()` (lines 85–122):**

Line 90 unconditionally inserts the requested key into the per-contract Caffeine cache (`contractSlotsCache`). Line 91 then snapshots **all** keys currently in that cache. [2](#0-1) 

Line 103 passes the entire snapshot to the repository, producing `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (?, ?, … up to 1500 elements)`. [3](#0-2) 

**Root cause — non-existent slots are never written to `contractStateCache`:**

After the batch query returns, only rows that actually exist in the DB are stored in `contractStateCache` (lines 106–113). A slot key with no matching row produces no iteration, so nothing is ever written. [4](#0-3) 

This means the guard at line 65 will always evaluate to `false` for non-existent slots on every subsequent call, and `findStorageBatch()` — with the full accumulated IN-clause — is invoked 100% of the time for those keys.

**Cache size ceiling:**

The per-contract slot cache is configured with `maximumSize=1500`, so each contract accumulates at most 1,500 slot keys, producing an IN-clause of up to 1,500 elements on every cache-miss call. [5](#0-4) 

## Impact Explanation

Each 1,500-element `IN` clause forces the PostgreSQL planner to evaluate a large parameter list against the `contract_state` table. On a production table with millions of rows this materially increases per-query execution time and I/O. The throttle is gas-based and request-rate-based — not DB-query-complexity-based — so an attacker can sustain this at 500 req/s (the default `requestsPerSecond` limit) with minimal gas cost, making the database the bottleneck rather than the application tier. Sustained amplification of this kind can push DB CPU and I/O above the degradation threshold for legitimate traffic.

## Likelihood Explanation

No authentication or special privilege is required — `eth_call` is a public, unauthenticated endpoint. The attacker needs only a valid contract address (publicly discoverable on-chain) and 1,500 arbitrary 32-byte slot keys. The cache-poisoning phase requires only 1,500 HTTP requests, well within the 500 req/s rate limit (achievable in approximately 3 seconds). The attack is fully repeatable and stateless from the attacker's perspective. The `enableBatchContractSlotCaching` flag defaults to `true`, so the vulnerable code path is active in all default deployments. [6](#0-5) 

## Recommendation

After the batch query completes, write a sentinel value (e.g., the existing `EMPTY_VALUE = new byte[0]`) into `contractStateCache` for every slot key in `cachedSlots` that did **not** appear in the query results. Then update the guard in `findStorage()` to treat `EMPTY_VALUE` as a definitive "not found" answer and return `Optional.empty()` immediately, rather than falling through to `findStorageBatch()`.

Concretely:
- After the loop at lines 106–114, iterate over `cachedSlots` and for any key not found in the results, call `contractStateCache.put(generateCacheKey(contractId, key), EMPTY_VALUE)`.
- Change the guard at line 65 from `cachedValue != null && cachedValue != EMPTY_VALUE` to two separate branches: if `cachedValue == EMPTY_VALUE` return `Optional.empty()`; if `cachedValue != null` return `Optional.of(cachedValue)`.

This ensures that a non-existent slot is cached as "absent" in `contractStateCache` (which expires after 2 seconds per its `expireAfterWrite=2s` configuration), preventing repeated full-batch queries for the same non-existent key within that window. [7](#0-6) 

## Proof of Concept

1. Identify any valid contract address `C` on the network (publicly visible on-chain).
2. Send 1,500 `eth_call` requests to contract `C`, each specifying a distinct, non-existent 32-byte slot key (e.g., `0x0000...0001` through `0x0000...05DC`). Each call reaches `findStorageBatch()`, adds the key to the per-contract `contractSlotsCache` via `putIfAbsent` (line 90), and issues a growing IN-clause query (1-element, 2-element, …, 1,500-element). After 1,500 calls the cache is saturated at `maximumSize=1500`. [2](#0-1) 
3. From this point forward, every `eth_call` to contract `C` for any non-existent slot triggers a 1,500-element IN-clause query. Because non-existent slots are never written to `contractStateCache`, the guard at line 65 never fires and `findStorageBatch()` is called on 100% of such requests. [1](#0-0) 
4. Repeat step 3 at the maximum allowed rate (500 req/s by default), sustaining continuous 1,500-element batch queries against the DB indefinitely. [8](#0-7)
