## Audit Report

## Title
Missing `maxInboundMessageSize` on gRPC Server Allows Unauthenticated Resource Exhaustion via Oversized `AddressBookQuery` Payloads

## Summary
`GrpcConfiguration.grpcServerConfigurer` configures the Netty gRPC server with only `executor` and `maxConcurrentCallsPerConnection`, never calling `maxInboundMessageSize()`. This leaves the gRPC-Java framework default of 4 MB per message in place. Because `AddressBookQuery` is a semantically tiny message (~40 bytes), an unauthenticated attacker can send near-maximum-size frames across many concurrent connections to drive heap pressure toward OOM conditions.

## Finding Description

**Proto schema** — `AddressBookQuery` contains exactly two fields: `file_id` (a `FileID` sub-message: three `int64` varints) and `limit` (an `int32` varint). Maximum legitimate payload is approximately 40 bytes. [1](#0-0) 

**Server configuration** — `GrpcConfiguration.grpcServerConfigurer` customises the `NettyServerBuilder` with only `executor` and `maxConcurrentCallsPerConnection`. `maxInboundMessageSize()` is never called, so the gRPC-Java default of 4 MB per message applies. [2](#0-1) 

**Per-connection concurrency cap** — `NettyProperties` caps concurrent calls per connection at 5 but places no limit on the number of connections or total in-flight memory. [3](#0-2) 

**Controller entry point** — `NetworkController.getNodes()` passes the fully-deserialized `AddressBookQuery` directly to `toFilter()` with no size or content pre-check. Deserialization (including unknown-field retention, which protobuf-java 3.5+ performs by default) occurs before the method body executes. [4](#0-3) 

**Root cause** — The failed assumption is that the gRPC-Java 4 MB default is an acceptable bound for all endpoints. For `AddressBookQuery`, the legitimate payload is ~100,000× smaller than the permitted maximum, making the default a resource-exhaustion amplifier rather than a meaningful guard.

## Impact Explanation
Each in-flight `getNodes` RPC can hold up to 4 MB of heap (gRPC-Java buffers the full frame before deserialization). With `maxConcurrentCallsPerConnection = 5` and N attacker-controlled connections, total heap pressure is `N × 5 × 4 MB = 20N MB`. At 250 connections this is 5 GB; at 500 connections it is 10 GB. Sustained heap pressure causes long GC pauses or JVM OOM, making the gRPC service unresponsive to legitimate address-book queries. No authentication is required to reach port 5600. [5](#0-4) 

## Likelihood Explanation
Port 5600 is the documented public gRPC port. No authentication, API key, or application-layer rate limit is enforced. Any external actor with a TCP stack and a tool such as `grpcurl` or a custom gRPC client can execute this attack. The attack is trivially scriptable and repeatable.

## Recommendation
1. **Set a tight `maxInboundMessageSize`** in `GrpcConfiguration.grpcServerConfigurer`. Given the proto schema, a limit of 1 KB (1024 bytes) is more than sufficient for any legitimate `AddressBookQuery`:
   ```java
   serverBuilder.maxInboundMessageSize(1024); // AddressBookQuery max ~40 bytes
   ``` [6](#0-5) 

2. **Add a `maxInboundMessageSize` field to `NettyProperties`** so the limit is configurable via `hiero.mirror.grpc.netty.maxInboundMessageSize` and validated with `@Min`. [7](#0-6) 

3. **Consider a connection-count limit** (`maxConnectionsTotal` / `maxConnectionsPerIp` via Netty or an upstream proxy) to bound the number of simultaneous attacker connections.

## Proof of Concept
```bash
# Generate a 4 MB AddressBookQuery payload stuffed with unknown fields (wire type 2, field 15)
python3 -c "
import struct, sys
# Unknown field tag: field 15, wire type 2 (length-delimited)
tag = (15 << 3) | 2
payload = b''
chunk = b'A' * 65536
while len(payload) < 4 * 1024 * 1024 - 10:
    payload += struct.pack('BB', tag, len(chunk) & 0x7F | 0x80) + struct.pack('B', len(chunk) >> 7) + chunk[:128]
sys.stdout.buffer.write(payload)
" > big_query.bin

# Send 5 concurrent RPCs per connection across 250 connections
for i in $(seq 1 250); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d @ -proto network_service.proto \
      mirror-node:5600 com.hedera.mirror.api.proto.NetworkService/getNodes \
      < big_query.bin &
  done
done
wait
# Expected result: JVM heap exhaustion / OOM / severe GC pauses on the mirror node gRPC process
```
