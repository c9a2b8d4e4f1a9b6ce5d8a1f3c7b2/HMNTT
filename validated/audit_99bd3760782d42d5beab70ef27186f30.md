Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()` Causes Unbounded DB Queries via `limit = Long.MAX_VALUE`

## Summary
A narrowing cast from `long` to `int` in the remaining-limit calculation inside `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()` causes integer overflow when a gRPC subscriber sets `limit = Long.MAX_VALUE`. The overflow produces `-1`, which causes `hasLimit()` to return `false`, resulting in every subsequent poll cycle issuing a fully unbounded SQL query against the `topic_message` table for the lifetime of the subscription.

## Finding Description

**Step 1 — No upper-bound cap at ingestion.**
`ConsensusController.toFilter()` copies the raw protobuf `uint64` limit directly into the filter builder with no cap: [1](#0-0) 

**Step 2 — Bean Validation permits `Long.MAX_VALUE`.**
`TopicMessageFilter.limit` carries only `@Min(0)`, so `Long.MAX_VALUE` passes validation at the `subscribeTopic` boundary: [2](#0-1) 

**Step 3 — Narrowing cast overflows in `PollingTopicListener.poll()`.**
```
(int)(Long.MAX_VALUE - 0L)
  = (int)(0x7FFFFFFFFFFFFFFF)
  = 0xFFFFFFFF
  = -1
``` [3](#0-2) 

`Math.min(-1, maxPageSize)` evaluates to `-1`, and `filter.toBuilder().limit(-1).build()` is a plain Lombok builder call — it does **not** go through Spring's AOP proxy, so `@Min(0)` is never re-enforced.

**Step 4 — `hasLimit()` returns `false` for `-1`.** [4](#0-3) 

**Step 5 — Repository issues an unbounded query.**
`setMaxResults` is only called when `hasLimit()` is `true`; with `-1` it is skipped entirely: [5](#0-4) 

**Step 6 — Polling loop repeats indefinitely.** [6](#0-5) 

On every tick, `count` increments but `Long.MAX_VALUE - count` always produces a negative `int` for any `count` in `[0, Integer.MAX_VALUE]`, so the condition never self-corrects.

**Step 7 — Identical overflow in `PollingTopicMessageRetriever.poll()`.** [7](#0-6) 

## Impact Explanation
Every poll cycle issues a query against `topic_message` with no `LIMIT` clause. On a busy topic with millions of rows, each tick loads the entire result set into the JVM heap. Sustained across the polling interval for the lifetime of the subscription, this exhausts heap memory and database connection/CPU resources, degrading or crashing the mirror node for all subscribers.

## Likelihood Explanation
Exploitation requires only a valid gRPC client and a known topic ID. No authentication, privileged role, or special network position is needed. The protobuf `limit` field is a plain `uint64`; setting it to any value in `[Long.MAX_VALUE - Integer.MAX_VALUE, Long.MAX_VALUE]` triggers the overflow. A single malicious subscription is sufficient; multiple concurrent subscriptions amplify the impact linearly.

## Recommendation
1. **Cap the limit at ingestion** in `ConsensusController.toFilter()` — clamp `query.getLimit()` to a configurable maximum (e.g., `Integer.MAX_VALUE` or a lower operator-defined bound) before passing it to the filter builder.
2. **Eliminate the narrowing cast** in both `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()`. Compute the remaining limit as a `long`, then clamp to `maxPageSize` using `long` arithmetic before casting to `int`:
   ```java
   long remaining = filter.hasLimit()
       ? filter.getLimit() - context.getCount().get()
       : Integer.MAX_VALUE;
   int pageSize = (int) Math.min(remaining, listenerProperties.getMaxPageSize());
   ```
3. **Add a positive-value guard** before calling `filter.toBuilder().limit(pageSize).build()` to ensure `pageSize >= 1`.

## Proof of Concept
```java
// Attacker constructs a gRPC ConsensusTopicQuery with:
//   topicID = <any valid topic>
//   limit   = Long.MAX_VALUE  (0x7FFFFFFFFFFFFFFF as uint64)
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setLimit(Long.MAX_VALUE)   // passes @Min(0) validation
    .build();

// Inside PollingTopicListener.poll() on first tick (count = 0):
//   (int)(Long.MAX_VALUE - 0) = -1
//   Math.min(-1, maxPageSize) = -1
//   newFilter.limit = -1  → hasLimit() = false
//   → TopicMessageRepositoryCustomImpl issues SELECT ... WHERE topic_id=? AND consensus_timestamp>=?
//     with NO setMaxResults() call → full unbounded result set loaded every poll interval
```
