Audit Report

## Title
Unauthenticated Mirror REST Response with Empty `nodes` Array Causes Non-Retryable Pinger Shutdown

## Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` treats an HTTP 200 response containing an empty `nodes` array as a permanent, non-retryable failure. An attacker who controls the configured mirror REST endpoint (or can MITM a plaintext HTTP connection to it) can serve `{"nodes":[],"links":{"next":null}}` once, causing the pinger to exit via `log.Fatalf` without ever attempting a retry, permanently halting all transaction-confirmation monitoring.

## Finding Description
**Code path:** `pinger/mirror_node_client.go` → `fetchMirrorNodeNetwork()` (lines 74–131) → `buildNetworkFromMirrorNodes()` (lines 36–72) → `newClient()` in `pinger/sdk_client.go` (lines 17–22) → `main()` in `pinger/main.go` (line 43).

**Root cause:** When the decoded `payload.Nodes` slice is empty, the loop at lines 102–125 never executes, leaving `network` empty. The guard:

```go
// mirror_node_client.go:127-129
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

returns the second value as `false` (no retry). Back in `buildNetworkFromMirrorNodes()`:

```go
// mirror_node_client.go:58-61
lastErr = fmt.Errorf("attempt %d/%d: %w", attempt, attempts, err)
if !retry || attempt == attempts {
    break
}
```

`retry=false` causes an immediate `break` on the very first attempt, bypassing all `mirrorNodeClientMaxRetries` (default 10, env `HIERO_MIRROR_PINGER_MIRROR_NODE_CLIENT_MAX_RETRIES`). `newClient()` returns this error, and `main()` calls `log.Fatalf`, terminating the process.

**Failed assumption:** The code assumes an empty-nodes 200 response is a definitive permanent condition rather than a potentially adversarial or transient one, and therefore skips all retry logic.

**Contrast with transient errors:** Network errors (line 86) and 5xx/429 responses (lines 90–92) correctly return `retry=true`. Only the empty-nodes case and JSON decode errors return `retry=false`, meaning adversarial valid JSON is treated as more fatal than a server crash.

## Impact Explanation
The pinger is the sole component that continuously submits transactions to verify the Hiero network is live. If `newClient()` fails, `main()` calls `log.Fatalf` (line 43), terminating the process. No SDK client is created, no transactions are submitted, and the `/tmp/ready` readiness file is never written (line 47), so Kubernetes readiness probes also fail. In deployments where the pinger's transaction success rate gates alerts or automated responses, this constitutes a complete denial-of-service against network observability. The attack is permanently effective: the pinger never recovers because there is no periodic re-initialization after startup failure.

**Severity: Medium** — DoS on the pinger monitoring service; does not directly affect consensus nodes.

## Likelihood Explanation
**Preconditions:**
1. `HIERO_MIRROR_PINGER_NETWORK=other` must be configured (the only mode that calls `buildNetworkFromMirrorNodes`).
2. The attacker must control the mirror REST endpoint OR be able to MITM the HTTP connection. The default example URL in `config.go` line 37 is `http://mirror-rest:5551` — plaintext HTTP, making MITM trivial on the same network segment (e.g., a shared Kubernetes cluster namespace or compromised sidecar).
3. A malicious or compromised mirror node operator can serve the empty-nodes payload on every request.

The attack requires only a single malicious response and is trivially repeatable. Because the retry loop is bypassed unconditionally on `retry=false`, there is no window for recovery.

## Recommendation
1. **Treat empty-nodes as retryable:** Change the guard at line 127–129 to return `true` for the retry flag:
   ```go
   if len(network) == 0 {
       return nil, true, fmt.Errorf("no usable service_endpoints found from %s", url)
   }
   ```
   This ensures all configured retries are exhausted before giving up.

2. **Use TLS for the mirror REST endpoint:** Enforce `https://` in the URL or add a validation step in `loadConfig()` that rejects plaintext `http://` URLs, eliminating the MITM vector.

3. **Decouple startup failure from process exit:** Instead of `log.Fatalf`, retry `newClient()` in a loop with backoff in `main()`, so a transient or adversarial startup failure does not permanently kill the pinger.

4. **Add a minimum-nodes threshold:** Require at least N usable nodes before accepting the response as valid, making it harder for an attacker to serve a minimal but non-empty response that still degrades monitoring.

## Proof of Concept
Serve the following response from the configured mirror REST endpoint at `GET /api/v1/network/nodes`:

```
HTTP/1.1 200 OK
Content-Type: application/json

{"nodes":[],"links":{"next":null}}
```

**Expected result:** `fetchMirrorNodeNetwork()` returns `(nil, false, err)` → `buildNetworkFromMirrorNodes()` breaks on attempt 1/11 → `newClient()` returns error → `main()` calls `log.Fatalf("client error: ...")` → pinger process exits immediately, never writing `/tmp/ready`, never submitting transactions, and never recovering.
