Audit Report

## Title
Unbounded HTTP Response Body Read in `fetchMirrorNodeNetwork` Enables Memory Exhaustion DoS

## Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` decodes `resp.Body` directly via `json.NewDecoder` with no byte-size cap. A mirror node returning an oversized JSON payload can exhaust the pinger container's memory, triggering an OOM kill and a crash loop.

## Finding Description
**Exact location:** `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()`, line 96:

```go
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil {
``` [1](#0-0) 

`resp.Body` is the raw HTTP response stream. No `io.LimitReader`, `http.MaxBytesReader`, or `Content-Length` guard is applied before this call ā€” confirmed by grep returning zero matches for either primitive across the entire pinger package.

**Why existing checks are insufficient:**

- **HTTP timeout** (default 10 s, `config.go` line 90): limits wall-clock time, not bytes transferred. A local or fast network can deliver hundreds of MiB within 10 s. [2](#0-1) 

- **Empty-string guard** (`mirror_node_client.go` line 103) and **`AccountIDFromString` check** (lines 107ā€“110): both execute only after the entire JSON body is already decoded and resident in heap memory. [3](#0-2) 

## Impact Explanation
The pinger container has a hard Kubernetes memory limit of **20 Mi** and a Go runtime soft limit (`GOMEMLIMIT`) of **10 MiB** ā€” both set in `charts/hedera-mirror-pinger/values.yaml`: [4](#0-3) [5](#0-4) 

A response body exceeding these thresholds causes the Linux OOM killer to terminate the pinger process. Because the pinger submits periodic HBAR transfers to verify network liveness, a crash loop means:
- Kubernetes liveness/readiness probes fail, triggering repeated restarts.
- A sustained gap in transaction submission, breaking end-to-end network health monitoring.
- Permanent unavailability if the malicious mirror node remains in rotation.

## Likelihood Explanation
The attacker must be able to serve a crafted HTTP response to the pinger. Realistic vectors include:

1. **`network=other` mode**: `HIERO_MIRROR_PINGER_NETWORK=other` is the **default** in `values.yaml` (line 18), and `HIERO_MIRROR_PINGER_REST` is operator-configurable ā€” a misconfigured or attacker-controlled URL is a plausible deployment scenario. [6](#0-5) 

2. **Network-level MITM**: No TLS pinning is enforced in the pinger HTTP client. [7](#0-6) 

3. **Compromised mirror node instance** in a permissioned deployment.

The attack requires a single HTTP response, is trivially repeatable, and needs no authentication beyond the ability to respond to the pinger's GET request.

## Recommendation
Wrap `resp.Body` with a size-capped reader before passing it to the JSON decoder. A reasonable cap (e.g., 1 MiB) is far above any legitimate mirror node response:

```go
const maxBodyBytes = 1 << 20 // 1 MiB
limited := io.LimitReader(resp.Body, maxBodyBytes+1)
if err := json.NewDecoder(limited).Decode(&payload); err != nil {
    ...
}
```

Alternatively, use `http.MaxBytesReader(nil, resp.Body, maxBodyBytes)` which also causes the read to return an error if the limit is exceeded, preventing silent truncation.

## Proof of Concept
1. Stand up an HTTP server that returns a 200 OK with `Content-Type: application/json` and a body of the form:
   ```json
   {"nodes":[{"node_account_id":"0.0.3","service_endpoints":[{"domain_name":"node.example.com","port":50211}]},
   ... (repeated millions of times or with a single entry whose node_account_id is a 50 MB string) ...
   ]}
   ```
2. Set `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-server>`.
3. Start the pinger. `json.NewDecoder(resp.Body).Decode(&payload)` at line 96 allocates the full payload on the Go heap.
4. With `GOMEMLIMIT=10MiB` and a container limit of 20 Mi, a response body of ~15ā€“20 MiB triggers OOM termination before `Decode` returns. [8](#0-7)
