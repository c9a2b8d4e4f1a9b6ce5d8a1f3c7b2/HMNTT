Audit Report

## Title
Adversarial Block Node Can Permanently Evade Block Requests via Deliberate `serverStatus()` Timeout

## Summary
`BlockNode.getBlockRange()` catches all exceptions—including gRPC `DEADLINE_EXCEEDED` from a deliberately slow `serverStatus()` call—and silently returns `EMPTY_BLOCK_RANGE`. Unlike `streamBlocks()`, `getBlockRange()` never calls `onError()`, so a node that consistently times out is never penalized or marked inactive. If that node is the sole provider of certain blocks, those blocks are permanently unindexed by the mirror node.

## Finding Description

**`StreamProperties.java` line 35** sets the default `responseTimeout` to 400ms with a minimum of 100ms: [1](#0-0) 

**`BlockNode.getBlockRange()`** (lines 99–112) applies this deadline to `serverStatus()` and catches **all** exceptions, returning `EMPTY_BLOCK_RANGE = Range.closedOpen(0L, 0L)` on any failure — including deliberate timeouts. Critically, it never calls `onError()`: [2](#0-1) [3](#0-2) 

**`BlockNodeSubscriber.hasBlock()`** (lines 133–145) checks `blockRange.isEmpty()` and returns `false` for any node whose `getBlockRange()` timed out: [4](#0-3) 

**`getNode()`** (lines 108–131) iterates all active nodes, then all inactive nodes. If none pass `hasBlock()`, it throws `BlockStreamException("No block node can provide block N")`. A timing-out node stays permanently in the **active** pool (never penalized), consuming a slot in every iteration while never serving blocks: [5](#0-4) 

**Root cause — asymmetric error handling:** `streamBlocks()` calls `onError()` on any exception, eventually marking the node inactive after `maxSubscribeAttempts`. `getBlockRange()` does not: [6](#0-5) [7](#0-6) 

**Auto-discovery surface:** When `autoDiscoveryEnabled` is true, `BlockNodeDiscoveryService.discover()` reads all on-chain registered nodes with `STATUS + PUBLISH + SUBSCRIBE_STREAM` APIs and adds them to the pool automatically: [8](#0-7) [9](#0-8) 

## Impact Explanation
If the adversarial node is the only configured or discovered node holding a particular block range, `getNode()` throws `BlockStreamException` on every polling cycle and the mirror node fails to index those blocks indefinitely. Applications relying on the mirror node for transaction history (explorers, wallets, compliance tools) will see those blocks as permanently missing. While the mirror node holds no funds, failure to index fund-transfer transactions causes incorrect balance and history reporting for downstream consumers.

## Likelihood Explanation
- **With manual config:** Requires the attacker to already be a trusted, configured block node — low likelihood.
- **With auto-discovery enabled:** Any entity that can register a tier-1 block node on-chain (STATUS + PUBLISH + SUBSCRIBE_STREAM APIs) is automatically added to the pool. The attack is trivially repeatable: simply never respond to `serverStatus()` within 400ms. No special runtime access to the mirror node is needed after initial registration.

## Recommendation
Apply the same `onError()` penalty logic to `getBlockRange()` failures as is already applied in `streamBlocks()`. After a configurable number of consecutive `getBlockRange()` failures (e.g., reusing `maxSubscribeAttempts`), call `onError()` to mark the node inactive and enforce the `readmitDelay` cooldown. This closes the asymmetry between the two code paths and ensures a consistently unresponsive node is eventually removed from the active pool. [7](#0-6) 

## Proof of Concept
1. Register a block node on-chain advertising STATUS, PUBLISH, and SUBSCRIBE_STREAM APIs with `autoDiscoveryEnabled = true`.
2. Configure the node's `serverStatus()` handler to sleep for longer than `responseTimeout` (default 400ms) before responding — or simply never respond.
3. Ensure this is the only node in the pool that advertises the target block range.
4. On every `BlockNodeSubscriber.doGet()` call, `getBlockRange()` will time out, return `EMPTY_BLOCK_RANGE`, `hasBlock()` returns `false`, `getNode()` throws `BlockStreamException`, and the node's `errors` counter is never incremented.
5. The node remains permanently active. The target blocks are never indexed. The test `getBlockRangeTimeout` in `BlockNodeTest.java` (lines 139–155) already demonstrates that a timeout produces an empty range with no side effects on node state. [10](#0-9)

    private BlockNode getNode(final AtomicLong nextBlockNumber) {
        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
                inactiveNodes.add(node);
                continue;
            }

            if (hasBlock(nextBlockNumber, node)) {
                return node;
            }
        }

        // find the first inactive node with the block and force activating it
        for (final var node : inactiveNodes) {
            if (hasBlock(nextBlockNumber, node)) {
                node.tryReadmit(true);
                return node;
            }
        }

        throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
    }
