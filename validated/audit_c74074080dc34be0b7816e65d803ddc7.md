Audit Report

## Title
Cache Miss DoS via Uncached Negative Lookups in `findByIdAndDeletedIsFalse`

## Summary
`EntityRepository.findByIdAndDeletedIsFalse` uses `unless = "#result == null"` to skip caching negative results. Because Spring Cache unwraps `Optional<T>` before evaluating SpEL — treating `Optional.empty()` as `null` — lookups for non-existent entity IDs are never cached. An unauthenticated attacker can repeatedly query `/api/v1/contracts/call` with crafted long-zero EVM addresses encoding non-existent entity IDs, forcing every call to issue a live DB query. The only protection is a global, in-memory, non-per-IP rate limiter.

## Finding Description

**Confirmed code location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 20–30:

```java
@Caching(cacheable = {
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_ENTITY,
               unless = "#result == null"),
    @Cacheable(cacheNames = CACHE_NAME, cacheManager = CACHE_MANAGER_SYSTEM_ACCOUNT,
               condition = "#entityId < 1000 && ...",
               unless = "#result == null")
})
Optional<Entity> findByIdAndDeletedIsFalse(Long entityId);
``` [1](#0-0) 

**Root cause:** Spring's `CacheAspectSupport` unwraps `Optional<T>` return values before evaluating `unless` SpEL expressions and before storing values in cache. When the entity does not exist, the method returns `Optional.empty()`, which Spring unwraps to `null`. The condition `unless = "#result == null"` then evaluates to `true`, so the empty result is **never stored in cache**. Every subsequent request for the same (or any other) non-existent entity ID hits the database.

**Exploit path confirmed in `CommonEntityAccessor.java` lines 73–78:**
```java
private Optional<Entity> getEntityByMirrorAddressAndTimestamp(Address address, final Optional<Long> timestamp) {
    final var entityId = entityIdNumFromEvmAddress(address);
    return timestamp
            .map(t -> entityRepository.findActiveByIdAndTimestamp(entityId, t))
            .orElseGet(() -> entityRepository.findByIdAndDeletedIsFalse(entityId));
}
``` [2](#0-1) 

A long-zero EVM address (12 zero bytes + 8-byte entity num) is detected by `ConversionUtils.isLongZeroAddress`, routed to `getEntityByMirrorAddressAndTimestamp`, and the entity num is passed directly to `findByIdAndDeletedIsFalse`. If no entity exists, the result is never cached. [3](#0-2) 

**Cache config does not help:** `CacheProperties.java` line 19 sets `expireAfterWrite=1s,maximumSize=10000` — but this only applies to entries that are actually stored. Negative results are never stored. [4](#0-3) 

## Impact Explanation

Every request targeting a non-existent entity ID issues a live SQL query (`SELECT * FROM entity WHERE id = ? AND deleted IS NOT TRUE`). Under sustained attack, this generates DB queries at the full rate the attacker can sustain (up to the rate limit). In a multi-instance deployment, each instance has its own independent in-memory rate-limit bucket, so the effective DB query rate scales linearly with the number of running instances. This can exhaust DB connection pools or degrade query latency for legitimate users. [5](#0-4) 

## Likelihood Explanation

The `/api/v1/contracts/call` endpoint requires no authentication. [6](#0-5) 

The only throttle is a global in-memory bucket of 500 req/s (`requestsPerSecond = 500`), shared across all callers — not per-IP, not distributed. [7](#0-6) 

A single attacker can consume the full 500 req/s budget. The attack requires no special knowledge: any 20-byte address with a zero 12-byte prefix and a non-existent entity num in the last 8 bytes suffices. The attack is trivially automatable.

## Recommendation

1. **Cache negative results explicitly.** Change `unless = "#result == null"` to `unless = "false"` (or remove the `unless` clause) so that `Optional.empty()` is also stored in cache. Spring will store the unwrapped `null` and return a cached empty `Optional` on subsequent calls. Alternatively, use a dedicated sentinel value or a separate negative-result cache with a short TTL (e.g., 1–2 seconds).

2. **Add per-IP rate limiting.** The current global in-memory bucket does not prevent a single client from consuming the entire budget. Introduce per-IP throttling (e.g., using Bucket4j with a distributed store such as Redis, or a servlet filter keyed on `X-Forwarded-For`/`RemoteAddr`).

3. **Validate entity ID range.** Reject requests where the long-zero address encodes an entity num that exceeds the known maximum entity ID (`findMaxId()`), returning a 400 before any DB query is issued. [8](#0-7) 

## Proof of Concept

```
# Craft a long-zero address with a non-existent entity num (e.g., 0xFFFFFFFFFFFF)
TO="0x000000000000000000000000FFFFFFFFFFFF0001"

# Send repeated requests to the unauthenticated endpoint
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d "{\"to\":\"$TO\",\"data\":\"0x\"}" &
done
wait
```

Each request resolves the long-zero address to entity num `0xFFFFFFFFFFFF0001`, finds no entity, returns `Optional.empty()`, Spring unwraps to `null`, `unless = "#result == null"` fires, result is not cached, and a DB query is issued for every single request. Rotating through different non-existent IDs (e.g., incrementing the last bytes) prevents any incidental cache hits. [1](#0-0) [2](#0-1)
