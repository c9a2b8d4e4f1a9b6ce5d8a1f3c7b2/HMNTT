Audit Report

## Title
`TransferTransaction` Token/NFT Transfer Blind Spot in `Parse` Enables Deceptive On-Chain Execution

## Summary
`cryptoTransferTransactionConstructor.Parse()` reads only HBAR transfers from a `TransferTransaction`, silently discarding any token or NFT transfer legs encoded in the same protobuf body. Because `isSupportedTransactionType()` gates only on Go struct type and `ConstructionSubmit` performs no content-level validation, an attacker can craft a transaction that appears HBAR-only to `/construction/parse` but executes hidden token/NFT drains when submitted to the network.

## Finding Description

**Type gate — confirmed at `construction_service.go` lines 676–685:** [1](#0-0) 

Any `hiero.TransferTransaction` value passes unconditionally, regardless of what transfer legs are encoded in its protobuf body.

**Parse blind spot — confirmed at `crypto_transfer_transaction_constructor.go` lines 68–78:** [2](#0-1) 

Only `GetHbarTransfers()` is called. `GetTokenTransfers()` and `GetNftTransfers()` are never invoked anywhere in the construction code (grep across all `construction*.go` files returns zero matches for those symbols). Token and NFT transfer legs are therefore invisible to `/construction/parse`.

**Submit path — confirmed at `construction_service.go` lines 340–355:** [3](#0-2) 

No content-level validation occurs between `unmarshallTransactionFromHexString` and `TransactionExecute`. The full protobuf body — including hidden token/NFT transfers — is forwarded to the network verbatim.

**Root cause:** The `Parse` implementation assumes a `TransferTransaction` can only carry HBAR transfers, which is false. The Hedera `CryptoTransfer` protobuf body supports HBAR, fungible token, and NFT transfers simultaneously in the same transaction.

## Impact Explanation
`/construction/parse` is the canonical human-readable verification step in the Rosetta construction workflow. Its failure to surface token/NFT transfers breaks the security contract that signers rely on for informed consent. A victim who follows the standard Rosetta signing flow is deceived into authorizing arbitrary token/NFT drains. The `frozenBodyBytes` that the victim signs cover the full protobuf body (including the hidden token transfer), so the signature is cryptographically valid for the malicious transaction. Severity is **High**: tokens and NFTs can be stolen from any account whose owner uses the Rosetta API to verify transactions before signing.

## Likelihood Explanation
The attack requires no special privileges — only the ability to craft a hex-encoded Hedera transaction (trivially done with the public Hiero SDK). In exchange or wallet integrations built on Rosetta, calling `/construction/parse` before signing is the standard workflow, making this a realistic attack surface. The attack is repeatable and requires no on-chain state beyond the victim holding tokens.

## Recommendation
In `cryptoTransferTransactionConstructor.Parse()`, after processing HBAR transfers, also call `GetTokenTransfers()` and `GetNftTransfers()` on the `transferTransaction`. If either returns a non-empty map, return an error (e.g., `ErrTransactionInvalidType` or a new `ErrUnsupportedTransferType`) to reject transactions containing transfer legs that the Rosetta layer cannot represent and verify. This ensures that any `TransferTransaction` submitted to `/construction/parse` is fully accounted for in the returned operations. [4](#0-3) 

## Proof of Concept
1. Using the Hiero SDK, construct a `TransferTransaction` containing:
   - A nominal HBAR transfer (e.g., −1 tinybar attacker → victim) to appear legitimate.
   - A large fungible-token or NFT transfer from victim → attacker.
2. Hex-encode the unsigned transaction and send it to the victim as part of a Rosetta-based integration.
3. Victim calls `POST /construction/parse` — response shows only the HBAR operation; the token drain is invisible because `Parse()` only iterates `GetHbarTransfers()`. [2](#0-1) 
4. Victim signs the `frozenBodyBytes` (which cover the full protobuf body including the token transfer) and returns the signature.
5. Attacker calls `POST /construction/combine` then `POST /construction/submit`.
6. `ConstructionSubmit` calls `hiero.TransactionExecute` with no content validation, forwarding the full body to the Hedera network. [5](#0-4) 
7. Hedera network executes the full `CryptoTransfer`, draining the victim's tokens.

```go
func isSupportedTransactionType(transaction hiero.TransactionInterface) *rTypes.Error {
	switch transaction.(type) {
	case hiero.AccountCreateTransaction:
	case hiero.TransferTransaction:
	default:
		return errors.ErrTransactionInvalidType
```

```go
func (c *cryptoTransferTransactionConstructor) Parse(_ context.Context, transaction hiero.TransactionInterface) (
	types.OperationSlice,
	[]types.AccountId,
	*rTypes.Error,
) {
	transferTransaction, ok := transaction.(hiero.TransferTransaction)
	if !ok {
		return nil, nil, errors.ErrTransactionInvalidType
	}

	if transferTransaction.GetTransactionID().AccountID == nil {
		return nil, nil, errors.ErrInvalidTransaction
	}

	hbarTransferMap := transferTransaction.GetHbarTransfers()
	numOperations := len(hbarTransferMap)
	operations := make(types.OperationSlice, 0, numOperations)

	for accountId, hbarAmount := range hbarTransferMap {
		var err *rTypes.Error
		amount := &types.HbarAmount{Value: hbarAmount.AsTinybar()}
		if operations, err = c.addOperation(accountId, amount, operations); err != nil {
			return nil, nil, err
		}
	}

	senderMap := senderMap{}
	for _, operation := range operations {
		if operation.Amount.GetValue() < 0 {
			senderMap[operation.AccountId.String()] = operation.AccountId
		}
	}

	return operations, senderMap.toSenders(), nil
}
```
