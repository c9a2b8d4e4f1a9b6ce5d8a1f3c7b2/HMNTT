Audit Report

## Title
Unbounded Sequential DB Lookups via Multiple `from` EVM Address Parameters in `extractContractResultsByIdQuery`

## Summary
The `extractContractResultsByIdQuery` function in `rest/controllers/contractController.js` issues one sequential `await EntityService.getEncodedId()` database call per `from` filter value that is a valid EVM address. Because the `from` parameter is repeatable up to `config.query.maxRepeatedQueryParameters` (default: 100), an unauthenticated attacker can force up to 100 sequential DB lookups per single HTTP request with no authentication required.

## Finding Description

**Verified code path â€” sequential awaited DB calls:**

The loop at lines 439â€“454 of `rest/controllers/contractController.js` iterates over all filters and, for each `FROM` filter whose value is a valid EVM address, issues a blocking `await EntityService.getEncodedId(filter.value)` call: [1](#0-0) 

**Verified â€” `getEncodedId` executes a raw SQL query per call with no caching:**

`EntityService.getEncodedId` delegates to `getEntityIdFromEvmAddress`, which runs `entityFromEvmAddressQuery` â€” a direct `SELECT` against the `entity` table â€” for every invocation. There is no in-process cache, deduplication, or batching: [2](#0-1) [3](#0-2) 

**Verified â€” HTTP layer accepts up to 100 repeated values:**

The `qs` parser in `requestHandler.js` is configured with `arrayLimit: config.query.maxRepeatedQueryParameters`, confirming that up to 100 values for the same parameter are accepted at the HTTP layer: [4](#0-3) 

**Verified â€” DB pool is bounded:**

The connection pool is configured with `max: config.db.pool.maxConnections` and `statement_timeout: config.db.pool.statementTimeout`, confirming the pool is finite and queries are subject to a timeout: [5](#0-4) 

**Root cause:** The code assumes `from` will be supplied at most once per request. The `buildFilters` utility explicitly supports repeated parameters as arrays, and the only guard (`isRepeatedQueryParameterValidLength`) permits up to 100 values. No deduplication or batching is applied before the sequential `await` loop.

## Impact Explanation

Each request with 100 unique EVM address `from` values causes 100 sequential DB queries against the `entity` table before the main contract results query runs. Because the queries are sequential (each `await` blocks the next), a single request holds a DB connection for the duration of all 100 round-trips. With a small number of concurrent attackers, the bounded connection pool (`maxConnections`, default 10) becomes saturated, causing legitimate requests to queue or time out. This is a low-cost, unauthenticated denial-of-service against the database layer.

**Correction to the impact claim in the report:** The report states "1,000 in-flight sequential queries" for 10 concurrent requests. Sequential queries do not run in parallel â€” each request uses one connection at a time. The actual effect is that each connection is held for 100Ă— longer than a normal request, which still exhausts the pool under modest concurrency.

## Likelihood Explanation

The attack requires no credentials, no account, and no special knowledge beyond the public API documentation. Both `/api/v1/contracts/results` and `/api/v1/contracts/:contractId/results` are unauthenticated endpoints. The attacker only needs to craft a URL with up to 100 `from=<unique_40_hex_char_address>` query parameters, which is trivially scriptable and repeatable at high frequency.

## Recommendation

1. **Deduplicate `from` filter values** before entering the loop so that identical EVM addresses result in only one DB lookup.
2. **Batch the EVM address lookups** into a single `WHERE evm_address = ANY($1)` query instead of N sequential queries.
3. **Add an in-process cache** (e.g., a short-lived LRU cache) for EVM address â†’ encoded ID mappings in `EntityService`.
4. **Reduce `maxRepeatedQueryParameters`** for the `from` parameter specifically, or add a per-parameter override for parameters that trigger DB lookups.

## Proof of Concept

```
GET /api/v1/contracts/results?from=0xaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa
  &from=0xbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb
  ... (repeat up to 100 unique 40-hex-char addresses)
```

Each unique EVM address triggers one `SELECT id FROM entity WHERE evm_address = $1` call sequentially before the main query executes. Sending this request concurrently from multiple clients saturates the DB connection pool and degrades service for all users.
