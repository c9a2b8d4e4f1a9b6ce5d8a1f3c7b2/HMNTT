Audit Report

## Title
Unauthenticated Connection Pool Exhaustion DoS in `getTransactions()` via Unique Timestamp Range Flooding

## Summary
The `getTransactions()` handler in `rest/transactions.js` issues at least one unconditional `pool.queryQuietly()` call per request for the timestamp-range query, with no rate-limiting middleware on the endpoint. The default DB pool is capped at 10 connections and the full-response Redis cache is disabled by default. An unauthenticated attacker sending concurrent requests with unique timestamp ranges can saturate the pool, causing all subsequent requests to queue and time out, resulting in a complete denial-of-service of the mirror node REST API.

## Finding Description

**Verified code path:**

- `getTransactions()` at [1](#0-0)  calls `doGetTransactions()`.
- `doGetTransactions()` at [2](#0-1)  calls `getTransactionTimestamps()`.
- `getTransactionTimestamps()` issues an unconditional `pool.queryQuietly(query, params)` at [3](#0-2) .
- A second `pool.queryQuietly()` is issued at [4](#0-3)  for `getTransactionsDetails` on any cache-missing transaction.
- `queryQuietly` in `rest/utils.js` at line 1481, when `preQueryHint` is absent (the case for both calls above), falls through to `this.query(query, params)` at [5](#0-4) , which checks out a pool connection for the full duration of the query.
- The pool is configured with `max: config.db.pool.maxConnections` at [6](#0-5) , defaulting to **10** connections per [7](#0-6) .

**Two failed assumptions confirmed:**

1. **Timestamp-range query is never cached.** The Redis transaction-detail cache at line 705 is keyed via `keyMapper` as `transaction:${timestamp}` per individual `consensus_timestamp` — not per the incoming timestamp-range filter. [8](#0-7)  The full-response Redis cache (`responseCacheCheckHandler`) is **disabled by default** (`cache.response.enabled: false`). [9](#0-8)  Every unique `?timestamp=gt:X&timestamp=lt:Y` combination always hits the database.

2. **No rate limiting on the REST API.** `server.js` registers `getTransactions` at line 132 with no throttle middleware. [10](#0-9)  The middleware stack (lines 68–98) contains only: URL encoding, JSON parsing, CORS, optional compression, httpContext, requestLogger, authHandler (optional Basic Auth for response-limit overrides only), optional metrics, and optional response-cache check — no request-rate limiter. [11](#0-10) 

**Why existing mitigations are insufficient:**

- `statementTimeout = 20000 ms` limits individual query duration but does not prevent the attacker from immediately re-filling released connection slots. [12](#0-11) 
- `connectionTimeoutMillis = 20000 ms` causes queued legitimate requests to fail with a timeout error rather than queue indefinitely — this means legitimate users receive errors, not that the attack is mitigated. [13](#0-12) 
- `authHandler` is entirely optional; unauthenticated requests proceed normally. [14](#0-13) 
- `maxRepeatedQueryParameters = 100` limits repeated params per request, not concurrent requests. [15](#0-14) 

## Impact Explanation
The mirror node REST API becomes unavailable to all legitimate consumers. Queries for transaction history, account balances, and all other endpoints sharing the same pool return timeout errors for the duration of the attack. The mirror node is a read-only service and does not participate in Hedera consensus; transaction confirmation on the network itself is unaffected. However, the REST API is the primary interface for wallets, explorers, and dApps to verify transaction status, so sustained unavailability constitutes a significant service-level impact. **Severity: High** (DoS of public API, no authentication required, trivially repeatable).

## Likelihood Explanation
No special privileges, accounts, or credentials are required. The attack requires only HTTP access to the public endpoint and the ability to open ~10–20 concurrent TCP connections. Any commodity machine can sustain the required request rate. With a pool of 10 connections and a `statementTimeout` of 20 s, the attacker needs to send only ≥1 new request every 2 s to maintain saturation. The attack is repeatable indefinitely and leaves no persistent state to clean up.

## Recommendation
1. **Add a rate-limiting middleware** (e.g., `express-rate-limit`) to the `rest/` middleware stack in `server.js`, applied globally or specifically to the `/transactions` route, to cap requests per IP per time window.
2. **Increase the default pool size** (`maxConnections`) or deploy a connection pooler (e.g., PgBouncer) in front of PostgreSQL to decouple HTTP concurrency from DB connection slots.
3. **Enable the full-response Redis cache** (`cache.response.enabled: true`) for high-traffic endpoints so repeated identical queries are served from cache without touching the DB pool.
4. **Consider query-level concurrency limits** per client IP at the application layer to bound the number of in-flight DB queries from a single source.

## Proof of Concept
```bash
# Send 15 concurrent requests with unique nanosecond-precision timestamp ranges
for i in $(seq 1 15); do
  curl -s "http://<mirror-node>/api/v1/transactions?timestamp=gt:169000000000${i}000&timestamp=lt:169000000000${i}999" &
done
wait
# Expected: pool saturated (10 connections held), subsequent legitimate requests
# receive connection timeout errors after 20 s (connectionTimeoutMillis default).
# Attacker sustains saturation by repeating the loop every ~2 seconds.
```
