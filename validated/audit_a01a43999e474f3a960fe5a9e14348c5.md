# Audit Report

## Title
Malicious Block Node Can Exploit `BlockAssembler.append()` to Cause Repeated Error Loop, Disrupting Block Streaming

## Summary
A block node operator included in the importer's node list can send a malformed `BlockItemSet` (first item is not a `BLOCK_HEADER`) on every connection attempt. This triggers a `BlockStreamException` on each attempt, increments the per-node error counter, and eventually marks the node inactive. Because `getNode()` unconditionally force-readmits the only available inactive node, the importer is trapped in a permanent error loop and block processing halts indefinitely.

## Finding Description

`BlockAssembler.append()` enforces protocol ordering by throwing a `BlockStreamException` when the first item of a `BlockItemSet` is not a `BLOCK_HEADER` and `pending` is empty:

```java
} else if (firstItemCase != BLOCK_HEADER && pending.isEmpty()) {
    throw new BlockStreamException("Incorrect first block item case " + firstItemCase);
}
``` [7](#0-6) 

This exception propagates to the `streamBlocks()` catch block, which calls `onError()`:

```java
} catch (BlockStreamException ex) {
    onError();
    throw ex;
}
``` [8](#0-7) 

`onError()` increments the error counter and, after `maxSubscribeAttempts` consecutive failures, marks the node inactive with a readmit delay:

```java
if (errors.incrementAndGet() >= streamProperties.getMaxSubscribeAttempts()) {
    active = false;
    errors.set(0);
    readmitTime.set(Instant.now().plus(streamProperties.getReadmitDelay()));
``` [9](#0-8) 

The error counter reset (`errors.set(0)` at line 157) only fires after a successful response is processed inside the `while` loop — an attacker who never sends a valid first item prevents this reset from ever occurring. [6](#0-5) 

When `getNode()` is called and the malicious node is the only one, the force-readmit fallback unconditionally re-activates it:

```java
for (final var node : inactiveNodes) {
    if (hasBlock(nextBlockNumber, node)) {
        node.tryReadmit(true);
        return node;
    }
}
``` [10](#0-9) 

`tryReadmit(true)` bypasses the readmit delay entirely:

```java
public BlockNode tryReadmit(final boolean force) {
    if (!active && (force || Instant.now().isAfter(readmitTime.get()))) {
        active = true;
    }
``` [11](#0-10) 

The root cause is that `append()` correctly validates protocol ordering but assumes the remote node is honest. There is no authentication or integrity check on `SubscribeStreamResponse` content, so a malicious node can send any item sequence it chooses.

## Impact Explanation

If the malicious node is the sole configured block node (or all configured nodes are malicious), the importer is trapped in a tight loop: connect → malformed response → `BlockStreamException` → `onError()` → `active = false` → force-readmit → repeat. Block processing halts entirely for the duration of the attack. Even with multiple nodes, if the attacker controls all of them, the same outcome applies. The mirror node falls behind on block ingestion, degrading data freshness for all downstream consumers.

## Likelihood Explanation

The precondition is that the attacker's node must appear in the importer's node list. With `autoDiscoveryEnabled = true` as the default in `BlockProperties`, nodes can be added dynamically without static configuration changes. [5](#0-4) 

A block node operator with no special network privileges who is included in the discovery set can execute this attack trivially by modifying their node software to return a malformed first `BlockItemSet`. The attack requires no cryptographic material, is fully repeatable, and survives the readmit delay because `tryReadmit(true)` bypasses it unconditionally.

## Recommendation

1. **Permanent blacklisting after repeated malformed responses:** Introduce a separate counter for protocol-violation errors (distinct from transient network errors). After a configurable threshold of consecutive protocol violations, permanently exclude the node from the node list rather than cycling through inactive/force-readmit.

2. **Do not force-readmit a node that was deactivated due to protocol violations:** Distinguish between nodes deactivated for transient reasons (network timeout, server-side `SUCCESS` close) and those deactivated for malformed data. Only the former should be eligible for force-readmit.

3. **Rate-limit reconnection attempts:** Even for transient failures, introduce exponential backoff rather than immediate reconnection, so that a malicious node cannot drive a tight CPU/network loop.

4. **Operator-level authentication:** Where the deployment model permits, require mutual TLS or a signed handshake so that only authorized block nodes can serve responses to the importer.

## Proof of Concept

A malicious block node implementation overrides the `subscribeBlockStream` gRPC handler to always respond with a single `BlockItemSet` whose only item is a `SIGNED_TRANSACTION` (i.e., `ItemCase != BLOCK_HEADER`). The importer is configured (or auto-discovers) only this node.

**Expected sequence:**
1. Importer calls `streamBlocks(blockNumber, ...)`.
2. Malicious node returns `BlockItemSet{items: [SIGNED_TRANSACTION]}`.
3. `onBlockItemSet()` calls `append(items, SIGNED_TRANSACTION)`.
4. `append()` throws `BlockStreamException("Incorrect first block item case SIGNED_TRANSACTION")`.
5. `onError()` increments `errors` (1 → 2 → 3).
6. At `errors == maxSubscribeAttempts (3)`: `active = false`, `errors = 0`, `readmitTime = now + 1 min`.
7. `getNode()` finds no active node, enters the inactive-node loop, calls `node.tryReadmit(true)`, re-activates the malicious node, returns it.
8. Go to step 1. Loop is infinite; `nextBlockNumber` never advances.
