Audit Report

## Title
Raw `IllegalArgumentException` Message Leaked to Unprivileged Users via `doProcessCall` Exception Handler

## Summary
`ContractCallService.doProcessCall` catches `IllegalArgumentException` thrown during EVM execution and places the raw `.getMessage()` string directly into the `detail` field of a `MirrorEvmTransactionException`. The dedicated `@ExceptionHandler` in `GenericControllerAdvice` unconditionally serializes `e.getDetail()` into the HTTP response body with no sanitization, exposing internal implementation strings to any unauthenticated caller.

## Finding Description

**Step 1 — Exception catch in `doProcessCall`:**

`ContractCallService.doProcessCall` at lines 120–121 catches both `IllegalStateException` and `IllegalArgumentException` and wraps them:

```java
} catch (IllegalStateException | IllegalArgumentException e) {
    throw new MirrorEvmTransactionException(e.getMessage(), EMPTY);
}
``` [1](#0-0) 

**Step 2 — Constructor stores raw message as `detail`:**

The two-argument constructor of `MirrorEvmTransactionException` delegates to the four-argument constructor with `message=null`:

```java
public MirrorEvmTransactionException(final String detail, final String hexData) {
    this(null, detail, hexData, null);
}
``` [2](#0-1) 

`e.getMessage()` is stored as `this.detail`; `responseCode` is never set and remains `null`. [3](#0-2) 

**Step 3 — `GenericControllerAdvice` returns `detail` unconditionally:**

The dedicated handler checks `SERVER_RESPONSE_CODES.contains(e.getResponseCode())`. Because `responseCode` is `null`, `contains(null)` is `false`, so the `BAD_REQUEST` branch is always taken and `e.getDetail()` — the raw exception message — is written into the JSON response: [4](#0-3) 

**Step 4 — Sources of `IllegalArgumentException` with internal strings:**

`MapReadableStates.get()` throws messages such as `"Unknown k/v state id: " + stateId` and `"State is not an instance of ReadableKVState: " + stateId`: [5](#0-4) 

`MapWritableStates.get()` throws analogous messages (`"Unknown k/v state key: "`, `"State is not an instance of WritableKVState: "`): [6](#0-5) 

The same pattern applies to `getSingleton` and `getQueue` in both classes. [7](#0-6) 

**Step 5 — Why the existing 5xx suppression does not help:**

`handleExceptionInternal` suppresses details for server errors (`!statusCode.is5xxServerError() ? ex.getMessage() : StringUtils.EMPTY`), but `MirrorEvmTransactionException` is handled by its own `mirrorEvmTransactionError` handler which has no such suppression and always returns `e.getDetail()` regardless of content: [8](#0-7) 

## Impact Explanation
An unauthenticated caller receives internal implementation strings in the HTTP response body, including:
- Internal integer state IDs (e.g., `"Unknown k/v state id: 42"`)
- Java class names (`ReadableKVState`, `WritableKVState`, `WritableSingletonState`, `ReadableQueueState`)
- Analogous strings from `MapWritableStates` and `MapReadableStates` for singleton and queue states

This constitutes an information-disclosure vulnerability that aids reconnaissance by confirming internal architecture, state key numbering, and data layout. **Severity: Medium.**

## Likelihood Explanation
Any unauthenticated user can trigger this via `POST /api/v1/contracts/call` with a historical block parameter. The trigger requires either:
- **(a)** A historical block whose state map is missing a state key the EVM tries to access — possible with edge-case block numbers near chain genesis or during state migrations, and user-triggerable without any DB precondition by probing early block numbers.
- **(b)** A malformed entity row in the DB due to importer bugs or data corruption — opportunistic but realistic in production.

Condition (a) is directly user-triggerable, making this a realistic attack vector.

## Recommendation
Sanitize the exception message before placing it in `detail`. Options:
1. **Preferred:** In the `catch` block in `doProcessCall`, replace `e.getMessage()` with a generic, non-revealing string (e.g., `"EVM execution error"`) and log the original message server-side at `ERROR` level.
2. **Alternative:** In `mirrorEvmTransactionError`, apply the same suppression logic already present in `handleExceptionInternal` — return `StringUtils.EMPTY` as `detail` when the exception carries no `responseCode` (i.e., when it originated from an `IllegalArgumentException`/`IllegalStateException` wrap).

## Proof of Concept
```
POST /api/v1/contracts/call
Content-Type: application/json

{
  "block": "earliest",
  "data": "0x...",
  "to": "0x0000000000000000000000000000000000000001"
}
```

If the EVM accesses a state key absent from the historical state map, the response will be:
```json
HTTP/1.1 400 Bad Request
{
  "_status": {
    "messages": [{
      "message": null,
      "detail": "Unknown k/v state id: 7",
      "data": ""
    }]
  }
}
```

The integer `7` (or whichever state ID is missing) is an internal implementation detail of the mirror node's state management layer, confirming internal architecture to the caller.
