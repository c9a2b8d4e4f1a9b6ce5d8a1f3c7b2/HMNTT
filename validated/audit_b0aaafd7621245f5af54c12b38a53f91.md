## Audit Report

## Title
Cache Key Collision via 32-bit `Arrays.hashCode` Allows Entity Data Confusion in EVM Address Lookups

## Summary
`findByEvmAddressAndDeletedIsFalse` and `findByEvmAddressOrAliasAndDeletedIsFalse` in `EntityRepository` use `@spelHelper.hashCode(#alias)` as the sole Caffeine cache key for 20-byte EVM addresses. `SpelHelper.hashCode(byte[])` returns `Arrays.hashCode(value)` — a plain 32-bit `int`. Two distinct EVM addresses that share the same `Arrays.hashCode` value are indistinguishable to the cache, causing the mirror node's web3 API to return entity A's data for entity B's address lookup.

## Finding Description

**`SpelHelper.hashCode` returns a 32-bit `int`:**

`SpelHelper.hashCode(byte[])` delegates directly to `Arrays.hashCode(value)`, which produces a 32-bit integer — a lossy, non-injective mapping from a 20-byte (160-bit) input space. [1](#0-0) 

**This `int` is the complete cache key for both cached methods:** [2](#0-1) 

**The backing store is a `CaffeineCacheManager` keyed on the boxed `Integer`:** [3](#0-2) 

**Root cause:** The Caffeine cache has no knowledge of the original byte array — it only sees the `Integer` key. Two distinct byte arrays `A` and `B` where `Arrays.hashCode(A) == Arrays.hashCode(B)` are indistinguishable to the cache. When entity A's result is stored under key `k`, a subsequent lookup for entity B (same `k`) returns entity A's cached `Optional<Entity>` without a database round-trip.

**Why `unless = "#result == null"` is insufficient:** This guard only prevents caching of `null`/empty results. It does nothing to prevent a cache hit from returning a previously stored result for a *different* address that happens to share the same 32-bit hash. [4](#0-3) 

## Impact Explanation
The mirror node's web3 API (`eth_getBalance`, `eth_getCode`, `eth_call`, contract execution context) relies on `findByEvmAddressAndDeletedIsFalse` and `findByEvmAddressOrAliasAndDeletedIsFalse` to resolve entities. [5](#0-4) 

Serving entity A's record for entity B's address causes: incorrect balance reporting, incorrect bytecode returned for smart contract lookups, incorrect account metadata (deleted status, type) used in EVM execution, and corrupted responses to EVM-compatible tooling (MetaMask, Hardhat, ethers.js). This is a data integrity issue affecting the mirror node's read API — the primary interface for EVM-compatible clients.

## Likelihood Explanation
Finding a 32-bit `Arrays.hashCode` collision against a specific 20-byte target requires approximately 65,536 key-pair generations on average (birthday paradox against a 32-bit space), achievable in milliseconds on commodity hardware. Creating a Hedera account is permissionless and costs a small HBAR fee. The attack is repeatable and automatable. No privileged access is required.

## Recommendation
Replace the lossy 32-bit hash with a collision-free key derived from the full byte array content. Options in order of preference:

1. **Change `SpelHelper.hashCode(byte[])` to return a hex or Base64 string** — a lossless, unique string representation of the byte array — and update the SpEL key expression accordingly.
2. **Use `T(java.util.Arrays).toString(#alias)` directly in the SpEL `key` expression** — produces a unique string like `[0, 1, 2, ...]` for each distinct byte array.
3. **Wrap in a `ByteBuffer`** — `ByteBuffer.wrap(#alias)` has content-based `equals`/`hashCode`, but this still reduces to a 32-bit hash internally in the cache map; a string encoding is safer.

The simplest, lowest-risk fix is to change `SpelHelper` to return a `String`:

```java
// SpelHelper.java
public String toKey(byte[] value) {
    return java.util.HexFormat.of().formatHex(value);
}
```

And update the repository annotations:
```java
key = "@spelHelper.toKey(#alias)"
``` [1](#0-0) 

## Proof of Concept

```java
// Demonstrate collision: find two 20-byte arrays with the same Arrays.hashCode
import java.util.Arrays;
import java.util.Random;

public class CollisionDemo {
    public static void main(String[] args) {
        // Target: any known entity's EVM address
        byte[] target = new byte[20];
        new Random(42).nextBytes(target);
        int targetHash = Arrays.hashCode(target);

        // Brute-force a colliding address (~65K attempts on average)
        Random rng = new Random();
        byte[] candidate = new byte[20];
        int attempts = 0;
        do {
            rng.nextBytes(candidate);
            attempts++;
        } while (Arrays.hashCode(candidate) != targetHash
                 || Arrays.equals(candidate, target));

        System.out.printf("Collision found after %d attempts%n", attempts);
        System.out.printf("Target:    %s (hash=%d)%n",
            java.util.HexFormat.of().formatHex(target), targetHash);
        System.out.printf("Collider:  %s (hash=%d)%n",
            java.util.HexFormat.of().formatHex(candidate),
            Arrays.hashCode(candidate));
        // Both addresses map to the same Caffeine cache key.
        // Step 1: Query mirror node for `collider` address → cache miss → DB hit → cached under key `targetHash`
        // Step 2: Query mirror node for `target` address  → cache HIT  → returns collider's Entity record
    }
}
