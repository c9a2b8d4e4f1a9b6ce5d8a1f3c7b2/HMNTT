Audit Report

## Title
Empty-Credential Authentication Bypass in Metrics Endpoint `authenticate()`

## Summary
The `authenticate()` function in `rest/middleware/metricsHandler.js` performs no validation that the configured `username` and `password` are non-empty strings before comparing them via `tsscmp`. When both values are empty strings (e.g., from unset environment variables), an attacker can send `Authorization: Basic Og==` (base64 of `:`) to produce `{name: '', pass: ''}` from `basicAuth()`, causing both `tsscmp` calls to return `true` and granting unauthenticated access to the Prometheus metrics endpoint.

## Finding Description
**File:** `rest/middleware/metricsHandler.js`, function `authenticate()`, lines 137–144.

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

The function reads `username` and `password` from `config.metrics.config` and immediately passes them to `tsscmp` with no guard against empty-string values. [1](#0-0) 

The only existing guard — `credentials &&` — prevents `null`/`undefined` from `basicAuth()` but does **not** prevent a truthy object `{name: '', pass: ''}`. A header of `Authorization: Basic Og==` is syntactically valid RFC 7617 Basic Auth with empty username and empty password, so `basicAuth()` returns a truthy object, bypassing the only guard.

The default values for `username` and `password` are `mirror_api_metrics` and `mirror_api_metrics_pass` respectively. [2](#0-1) 

However, if an operator overrides these via environment variables (e.g., `HIERO_MIRROR_REST_METRICS_CONFIG_USERNAME=` and `HIERO_MIRROR_REST_METRICS_CONFIG_PASSWORD=`), both become empty strings. The `tsscmp('', '')` call returns `true` for both comparisons, and `authenticate()` returns `true`, serving the metrics payload at lines 161–164. [3](#0-2) 

## Impact Explanation
An attacker gains unauthenticated read access to the full Prometheus metrics output, which includes per-route request counts, durations, error rates, in-flight request counts, and Node.js process memory/CPU telemetry. This leaks internal operational state (traffic patterns, error spikes, resource usage) that can be used to fingerprint the deployment, time attacks, or confirm the success of denial-of-service attempts. The metrics endpoint is documented as intentionally access-controlled (`authentication: true` by default), so this bypass directly violates the intended security boundary. [4](#0-3) 

## Likelihood Explanation
The precondition — empty credentials — is realistic in container/Kubernetes deployments where secrets are injected via environment variables. An unset or accidentally blank secret value silently produces an empty string in the config. The attacker needs no privileges, only network access to the REST API port (default 5551). The exploit is a single HTTP request with a known, fixed header value (`Authorization: Basic Og==`). It is repeatable and leaves no distinguishing log entry beyond a normal 200 response. [5](#0-4) 

## Recommendation
Add an explicit non-empty guard for `username` and `password` before performing the `tsscmp` comparisons:

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {
    return true;
  }
  if (!username || !password) {
    return false;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
```

This ensures that a misconfigured (empty) credential pair causes authentication to fail closed rather than open. Additionally, consider adding startup validation (similar to the `parseUsersConfig` pattern already used in `rest/config.js`) that rejects an empty `username` or `password` when `authentication` is `true`. [6](#0-5) 

## Proof of Concept
**Precondition:** Operator sets `HIERO_MIRROR_REST_METRICS_CONFIG_USERNAME=` and `HIERO_MIRROR_REST_METRICS_CONFIG_PASSWORD=` (empty), causing `username = ''` and `password = ''` in `config.metrics.config`.

**Request:**
```
GET /swagger/metrics/ HTTP/1.1
Host: <mirror-node-host>:5551
Authorization: Basic Og==
```

(`Og==` is the base64 encoding of `:`, representing empty username and empty password per RFC 7617.)

**Execution trace:**
1. `authentication` is `true` (default) → early-return bypass not triggered.
2. `basicAuth(req)` parses `Basic Og==` → returns `{name: '', pass: ''}` (truthy object).
3. `tsscmp('', '')` → `true` for both username and password comparisons.
4. `authenticate()` returns `true`.
5. Server responds with HTTP 200 and the full Prometheus metrics payload. [7](#0-6)
