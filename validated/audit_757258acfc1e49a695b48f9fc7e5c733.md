Audit Report

## Title
Insufficient Node Count Validation Allows Poisoned Mirror REST Response to Reduce Pinger's Network View to 1–2 Nodes

## Summary
`fetchMirrorNodeNetwork` in `pinger/mirror_node_client.go` accepts any non-empty node map as valid. When `network=other` is configured, the pinger fetches its consensus node list over an operator-supplied URL (which may be plain HTTP). An attacker with MITM capability on that connection — or a compromised mirror REST service — can return a response with only 1–2 nodes. The map passes the sole validation guard and is handed directly to `hiero.ClientForNetwork(netmap)`, causing the SDK to route all subsequent transactions exclusively to those nodes for the entire lifetime of the process.

## Finding Description

**Verified code path in `pinger/sdk_client.go` lines 17–22:**

```go
case "other":
    netmap, err := buildNetworkFromMirrorNodes(context.Background(), cfg)
    if err != nil {
        return nil, err
    }
    client = hiero.ClientForNetwork(netmap)
```

No size check is performed on `netmap` between `buildNetworkFromMirrorNodes` returning and `hiero.ClientForNetwork` consuming it. [1](#0-0) 

**Verified sole validation guard in `pinger/mirror_node_client.go` lines 127–129:**

```go
if len(network) == 0 {
    return nil, false, fmt.Errorf("no usable service_endpoints found from %s", url)
}
```

A response containing exactly 1 or 2 nodes produces `len(network) >= 1`, passes this check, and is returned as a valid network map. [2](#0-1) 

**Plain HTTP is explicitly shown as the example in `pinger/config.go` line 37:**

```go
flag.StringVar(&cfg.mirrorRest, "mirror-rest", envOr("HIERO_MIRROR_PINGER_REST", ""), "mirror node REST base URL (required for other), e.g. http://mirror-rest:5551")
``` [3](#0-2) 

No URL scheme validation is performed anywhere in `loadConfig()`. [4](#0-3) 

## Impact Explanation

`newClient` is called exactly once at startup in `pinger/main.go` line 41, and the returned `client` is reused in the ticker loop indefinitely: [5](#0-4) [6](#0-5) 

Once `hiero.ClientForNetwork(netmap)` is initialized with a 1–2 node map, the Hiero SDK routes every `TransferTransaction.Execute()` call exclusively to those nodes for the entire process lifetime. The pinger becomes blind to the omitted consensus nodes: failures, latency spikes, or outages on those nodes go undetected, producing false-healthy liveness signals. If the attacker controls the injected nodes, they also observe every transaction the pinger submits, including the operator account ID and the gRPC channel traffic.

## Likelihood Explanation

**Preconditions:**
- `network=other` must be configured — this is the required mode for private/custom networks, which is the exact deployment scenario this code path serves.
- The `mirrorRest` URL uses plain HTTP — the official example in the flag description (`http://mirror-rest:5551`) normalizes this pattern, and internal Kubernetes service-to-service traffic commonly uses HTTP with TLS termination at the ingress.
- The attacker needs network-level MITM capability: ARP spoofing on the same subnet, DNS poisoning, a compromised sidecar/proxy, or a rogue mirror REST service. No credentials or privileged access to the pinger process are required.

**Repeatability:** The attack only needs to succeed once — at pinger startup — because the client is never rebuilt. A brief MITM window at boot is sufficient for a persistent effect.

## Recommendation

1. **Enforce a minimum node count** in `fetchMirrorNodeNetwork` (e.g., `len(network) < 3`) and return an error if the threshold is not met.
2. **Validate the URL scheme** in `loadConfig` when `network=other`: reject any `mirrorRest` URL that does not begin with `https://`.
3. **Periodically refresh the client** rather than building it once at startup, so a transient MITM at boot does not have a permanent effect.

## Proof of Concept

1. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://<attacker-controlled-host>`.
2. Serve a crafted JSON response from that host:
   ```json
   {
     "nodes": [
       {
         "node_account_id": "0.0.3",
         "service_endpoints": [{"ip_address_v4": "1.2.3.4", "port": 50211}]
       }
     ],
     "links": {"next": null}
   }
   ```
3. `fetchMirrorNodeNetwork` builds `network = {"1.2.3.4:50211": AccountID{0,0,3}}`, `len(network) == 1 > 0`, passes the guard, and returns.
4. `hiero.ClientForNetwork(netmap)` is initialized with this single-node map.
5. Every subsequent `TransferTransaction.Execute()` in the ticker loop is routed exclusively to `1.2.3.4:50211` for the entire process lifetime.
