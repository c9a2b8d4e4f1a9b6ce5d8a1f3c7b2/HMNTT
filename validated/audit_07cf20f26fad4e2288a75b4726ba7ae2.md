Audit Report

## Title
Unbounded Memory Accumulation in EVM Opcode Replay Causes JVM OOM via `memory=true` Parameter

## Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint allows any unauthenticated caller to replay a historical transaction with `memory=true`. During replay, `AbstractOpcodeTracer.captureMemory()` snapshots the entire EVM memory frame at every single opcode step with no size cap, and all snapshots are accumulated in an unbounded `ArrayList` inside `OpcodeContext`. For a transaction that executed a memory-intensive loop, this causes the JVM to allocate gigabytes of heap in a single request, crashing the mirror node process.

## Finding Description

**Verified code path:**

`OpcodesController.getContractOpcodes()` accepts `memory=true` from any caller with no authentication. The only guards are a gzip header check and a global rate-limit bucket: [1](#0-0) 

The rate limit defaults to `opcodeRequestsPerSecond = 1` — one request per second **globally**, not per-IP: [2](#0-1) 

`ThrottleManagerImpl.throttleOpcodeRequest()` only checks the global bucket with no per-IP or per-user enforcement: [3](#0-2) 

`OpcodeServiceImpl.processOpcodeCall()` creates an `OpcodeContext` with initial `ArrayList` capacity of `gas / 3` (up to millions of slots) and passes it to the EVM replay: [4](#0-3) 

`OpcodeActionTracer.tracePostExecution()` is called after **every single opcode** and unconditionally invokes `captureMemory()`: [5](#0-4) 

`AbstractOpcodeTracer.captureMemory()` reads the **entire EVM memory** at each opcode step with no size cap, converting every 32-byte word into a 66-character hex string and adding it to a new `ArrayList`: [6](#0-5) 

All resulting `Opcode` objects (each carrying the full memory snapshot) are appended to an unbounded `ArrayList` in `OpcodeContext` with no cap: [7](#0-6) 

**Root cause:** There is no cap on (a) the EVM memory size captured per opcode step, (b) the number of opcode snapshots accumulated, or (c) the total heap consumed per request. The `hexCache` in `AbstractOpcodeTracer` (max 1600 entries) applies only to stack/storage, not to memory — memory uses `HexFormat.of()` directly, creating fresh `String` objects for every word at every step. [8](#0-7) 

## Impact Explanation

A single HTTP GET request with `Accept-Encoding: gzip` and `memory=true` targeting a memory-intensive historical transaction causes unbounded heap growth inside the JVM. The JVM throws an unrecoverable `OutOfMemoryError`, crashing the mirror node process and taking down all co-hosted services (REST API, gRPC, web3 API) for all users. Recovery requires a manual process restart. Because the rate limit is global (not per-IP) and allows 1 req/sec, a single attacker can continuously crash the node at 1-second intervals, constituting a sustained denial of service.

## Likelihood Explanation

**Preconditions:**
1. The opcode tracer feature must be enabled (`hiero.mirror.web3.opcode.tracer.enabled=true`). This defaults to `false` but is the intended configuration for debug/trace nodes. [9](#0-8) 
2. A memory-intensive transaction must exist on-chain (dynamic arrays, sorting, data-processing loops). The attacker does not need to deploy the contract — only a valid transaction hash is needed, which is publicly visible on-chain.
3. No credentials are required.

The attacker needs only a valid transaction hash and the ability to send one HTTP GET request with `Accept-Encoding: gzip`. This is trivially achievable by any external party.

## Recommendation

1. **Cap EVM memory snapshot size:** In `AbstractOpcodeTracer.captureMemory()`, add a hard limit on `frame.memoryWordSize()` (e.g., reject or truncate snapshots exceeding a configurable threshold such as 1,000 words / 32 KB).
2. **Cap total opcode accumulation:** In `OpcodeContext.addOpcodes()`, enforce a maximum list size and throw or truncate beyond it.
3. **Add per-IP rate limiting:** Replace or supplement the global `opcodeRateLimitBucket` with per-IP token buckets to prevent a single attacker from monopolizing the 1 req/sec global allowance.
4. **Add a gas limit cap for opcode replay:** Reject replay requests for transactions whose gas limit exceeds a configurable maximum (e.g., 1,000,000 gas) when `memory=true` is requested.

## Proof of Concept

1. Identify any historical transaction hash that used a contract performing large memory allocations (e.g., a Solidity contract that `mstore`s into a large dynamic array and then loops).
2. Send the following request to a mirror node with `opcode.tracer.enabled=true`:
```
GET /api/v1/contracts/results/<TX_HASH>/opcodes?memory=true&stack=false&storage=false
Accept-Encoding: gzip
```
3. The mirror node JVM begins allocating heap for each opcode step's full memory snapshot. For a transaction with ~10,000 EVM memory words and hundreds of thousands of loop iterations, heap consumption grows into the tens-to-hundreds of gigabytes range.
4. The JVM throws `OutOfMemoryError`; the mirror node process crashes.
5. Repeat at 1-second intervals (the global rate limit) to sustain the denial of service.

    protected final List<String> captureMemory(final MessageFrame frame, final OpcodeContext options) {
        if (!options.isMemory()) {
            return Collections.emptyList();
        }
        var size = frame.memoryWordSize();
        var hexMemoryEntries = new ArrayList<String>(size);

        final var wordSize = 32;
        final var memory = frame.readMutableMemory(0L, (long) wordSize * size).toArrayUnsafe();
        final var hex = HexFormat.of();

        for (var i = 0; i < size; i++) {
            var startIndex = i * wordSize;
            final var result = HEX_PREFIX + hex.formatHex(memory, startIndex, startIndex + wordSize);
            hexMemoryEntries.add(result);
        }

        return hexMemoryEntries;
    }
