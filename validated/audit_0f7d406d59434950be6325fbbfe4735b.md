# Audit Report

## Title
Unauthenticated DoS via DB Connection Pool Exhaustion on Unrate-Limited Airdrop Endpoints

## Summary
The `/api/v1/accounts/{id}/airdrops/outstanding` and `/api/v1/accounts/{id}/airdrops/pending` endpoints in the `rest-java` module accept unauthenticated requests, execute synchronous blocking database queries, and have no rate limiting. An attacker can exhaust the HikariCP connection pool (default size: 10) with concurrent requests, causing the service to queue indefinitely and become unresponsive.

## Finding Description

`TokenAirdropsController` exposes two unauthenticated `@GetMapping` endpoints with no throttle guard: [1](#0-0) 

Each request reaches `TokenAirdropRepositoryCustomImpl.findAll()`, which issues a synchronous, blocking JOOQ query: [8](#0-7) 

The only rate-limiting infrastructure in the codebase — `ThrottleConfiguration` using bucket4j — is scoped exclusively to the `web3` module: [9](#0-8) 

The `rest-java` module has no `application.yml` (only `banner.txt`), so no `spring.datasource.hikari` properties override Spring Boot defaults. `CommonConfiguration` wires HikariCP from `@ConfigurationProperties("spring.datasource.hikari")` with no rest-java-specific values: [5](#0-4) 

The `rest-java` filter chain registers only `MetricsFilter`, `LoggingFilter`, and `ShallowEtagHeaderFilter` — none enforce rate limits: [6](#0-5) 

Existing input guards (`@Max(MAX_LIMIT)`, `@Positive`, `@Size(max = 2)`) cap result rows and validate parameter format only — they do not limit concurrent request volume: [7](#0-6) 

## Impact Explanation

With a default HikariCP pool of 10 connections, 10 concurrent requests that each hold a connection for the duration of a DB query saturate the pool. All subsequent requests block waiting for a connection for up to 30 seconds (default `connectionTimeout`). Tomcat's default 200-thread executor fills up, causing the JVM to queue or reject new requests entirely. Because each `rest-java` instance shares one pool, a sustained flood renders the entire service on each targeted instance unresponsive.

## Likelihood Explanation

Preconditions are minimal: no credentials, no special tokens, no privileged access. Any valid account ID (e.g., `0.0.1`) is sufficient. The attack is trivially repeatable with standard HTTP load tools (`wrk`, `ab`, `hey`) from a single machine. The endpoints are publicly documented in the OpenAPI spec.

## Recommendation

1. **Add rate limiting to `rest-java`**: Introduce a `ThrottleFilter` or Spring MVC interceptor in `rest-java` analogous to the `ThrottleConfiguration` already present in `web3`, using bucket4j or a similar token-bucket implementation.
2. **Configure HikariCP explicitly**: Add an `application.yml` to `rest-java/src/main/resources/` with explicit `spring.datasource.hikari.maximum-pool-size` and `spring.datasource.hikari.connection-timeout` values appropriate for the expected load.
3. **Deploy an API gateway or reverse proxy** (e.g., nginx, Envoy) in front of `rest-java` instances with per-IP and global request-rate limits as a defense-in-depth measure.

package org.hiero.mirror.web3.config;

import io.github.bucket4j.Bandwidth;
import io.github.bucket4j.Bucket;
import io.github.bucket4j.local.SynchronizationStrategy;
import java.time.Duration;
import lombok.RequiredArgsConstructor;
import org.hiero.mirror.web3.throttle.ThrottleProperties;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration(proxyBeanMethods = false)
@RequiredArgsConstructor
public class ThrottleConfiguration {

    public static final String GAS_LIMIT_BUCKET = "gasLimitBucket";
    public static final String RATE_LIMIT_BUCKET = "rateLimitBucket";
    public static final String OPCODE_RATE_LIMIT_BUCKET = "opcodeRateLimitBucket";

    private final ThrottleProperties throttleProperties;

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
