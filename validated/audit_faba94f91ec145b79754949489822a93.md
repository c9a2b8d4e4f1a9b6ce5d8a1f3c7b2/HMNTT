Audit Report

## Title
Unbounded Mirror-Node Response Body Causes Memory Exhaustion in `fetchMirrorNodeNetwork()`

## Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` passes `resp.Body` directly to `json.NewDecoder` with no byte-limit guard. An attacker who controls the mirror node response can return a crafted JSON payload that causes unbounded heap allocation, OOM-killing the pinger process.

## Finding Description

**Verified code path — `pinger/mirror_node_client.go`:**

At line 96, `resp.Body` is decoded with no `io.LimitReader` wrapper: [1](#0-0) 

The `IPAddressV4` field on the `serviceEndpoint` struct is an unconstrained `string`: [2](#0-1) 

The processing loop at lines 113–124 calls `strings.TrimSpace`, `net.JoinHostPort`, and map insertion — all O(n) in the length of the string — with no length check on `ep.IPAddressV4`: [3](#0-2) 

**Why existing guards are insufficient:**

- `http.Client{Timeout: cfg.mirrorNodeClientTimeout}` (default 10 s) enforces wall-clock time, not bytes. A fast local server can stream gigabytes within that window. [4](#0-3) 
- The HTTP status check at lines 90–93 only rejects non-2xx; a 200 response with a huge body passes through. [5](#0-4) 
- The `if host == "" || ep.Port == 0` guard skips empty hosts but does not bound non-empty ones. [6](#0-5) 
- No `net.ParseIP` or length validation of `IPAddressV4` is performed anywhere in the file. [2](#0-1) 

**Retry amplification:** The retry loop in `buildNetworkFromMirrorNodes` re-invokes `fetchMirrorNodeNetwork` up to `mirrorNodeClientMaxRetries + 1` times (default: 11 attempts), re-reading the full body on each attempt: [7](#0-6) 

## Impact Explanation

The pinger writes `/tmp/alive` (liveness probe) and `/tmp/ready` (readiness probe) to signal Kubernetes health: [8](#0-7) [9](#0-8) 

An OOM-killed pinger causes the pod to restart, disrupting the heartbeat and triggering false liveness/readiness failures. In a deployment monitoring mainnet or testnet health, repeated crashes degrade observability of the entire network. The attack requires only a single crafted HTTP response.

## Likelihood Explanation

**Precondition:** The attacker must control what the mirror node returns. Achievable via:

1. **Compromise of the mirror node** — realistic for `network=other` internal deployments.
2. **MITM on plain HTTP** — `http://mirror-rest:5551` is the documented example in `config.go`; no TLS is enforced or required: [10](#0-9) 
3. **DNS poisoning** of the mirror node hostname.

Crafting the payload requires no special privileges — a standard HTTP server returning a large JSON body suffices. The attack is repeatable on every startup or reconnect cycle due to the retry loop.

The `mirrorNodeClientMaxRetries` default is 10 (11 total attempts): [11](#0-10) 

## Recommendation

Wrap `resp.Body` with `io.LimitReader` before passing it to `json.NewDecoder`, e.g.:

```go
const maxBodyBytes = 10 * 1024 * 1024 // 10 MiB
limited := io.LimitReader(resp.Body, maxBodyBytes)
if err := json.NewDecoder(limited).Decode(&payload); err != nil { ... }
```

Additionally:
- Validate `IPAddressV4` with `net.ParseIP` before use.
- Cap the number of nodes and endpoints processed per response.
- Enforce TLS for the mirror node REST URL in production deployments.

## Proof of Concept

A minimal HTTP server returning the following JSON, served at the configured mirror REST URL, triggers the issue:

```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        {
          "ip_address_v4": "<100MB string of digits and dots>",
          "port": 50211
        }
      ]
    }
  ],
  "links": {}
}
```

`json.NewDecoder(resp.Body).Decode(&payload)` at line 96 allocates the full string into the Go heap. With 11 retry attempts and a sufficiently large payload, the process is OOM-killed before the retry loop exits. [1](#0-0)
