Audit Report

## Title
`BufferUnderflowException` in `getByEvmAddressAndType()` via Under-Length EVM Address Input

## Summary
`decodeEvmAddress()` in `GraphQlUtils.java` performs no length validation on the decoded byte array. `getByEvmAddressAndType()` in `EntityServiceImpl.java` unconditionally performs three sequential binary reads (`getInt()` + `getLong()` + `getLong()`) on the resulting `ByteBuffer`, assuming exactly 20 bytes. A crafted input of exactly 12 zero bytes satisfies the first two reads and causes an uncaught `BufferUnderflowException` on the third, which propagates out of the service layer on every such request.

## Finding Description

**Root cause — no length guard in `decodeEvmAddress()`:**

`GraphQlUtils.decodeEvmAddress()` strips the optional `0x` prefix and delegates directly to `Hex.decodeHex()`, returning a byte array of whatever length the caller supplied. There is no assertion or check that the result is exactly 20 bytes. [1](#0-0) 

The existing test suite explicitly confirms sub-20-byte inputs are accepted without error — e.g., `"0000AaAaAa"` decodes to 5 bytes and `"0x0000AaAaAa"` also decodes to 5 bytes: [2](#0-1) 

**Vulnerable consumer — `getByEvmAddressAndType()`:**

```java
// EntityServiceImpl.java lines 34-41
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {       // reads 4 + 8 = 12 bytes
        return entityRepository.findById(buffer.getLong())      // reads 8 more bytes
                .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
``` [3](#0-2) 

The three reads require a cumulative 20 bytes. No `buffer.remaining()` check exists anywhere in this path.

**Exact trigger — 12-byte input (24 hex zeros):**

Input: `"000000000000000000000000"` → 12 bytes, all zero.

| Step | Operation | Bytes needed | Bytes remaining after | Result |
|------|-----------|-------------|----------------------|--------|
| 1 | `buffer.getInt()` | 4 | 8 | `0` → condition continues |
| 2 | `buffer.getLong()` (condition) | 8 | 0 | `0` → condition true, enters `if` block |
| 3 | `buffer.getLong()` (entity ID) | 8 | 0 available | **`BufferUnderflowException`** |

`BufferUnderflowException` extends `RuntimeException` and is unchecked. There is no `try/catch` in this call chain. Spring GraphQL's framework-level error handler catches it at the boundary and returns a generic error response, but the exception is thrown and logged on every such request.

**Note on short-circuit evaluation:** Due to Java's `&&` short-circuit, the `getLong()` in the condition is only evaluated when `getInt() == 0`. Inputs where the first 4 bytes are non-zero take the `else` branch safely. The vulnerable path requires the first 4 bytes to be zero (triggering the `if` branch) and total length between 12 and 19 bytes.

## Impact Explanation

Any unauthenticated caller who can reach the GraphQL endpoint can send a malformed `evmAddress` value (e.g., `"000000000000000000000000"`) and force an unhandled `BufferUnderflowException` on every such request. The GraphQL service returns an error response for that request; the JVM and Spring thread pool are not affected. The realistic impact is a **reliable per-request failure on the GraphQL account/contract lookup path** — every query using a crafted `evmAddress` will fail with an unhandled exception. The "total network shutdown" framing in the original claim is overstated; this is a partial, targeted DoS on the affected resolver, not a full service outage.

## Likelihood Explanation

- No authentication is required; the GraphQL endpoint is public-facing.
- The payload is trivial: any even-length hex string of 12–19 chars whose first 12 bytes are zero.
- The exploit is fully deterministic, repeatable, and scriptable with no race condition or timing dependency.
- The existing test suite does not cover this case; only exactly-20-byte addresses and empty input are tested. [2](#0-1) 

## Recommendation

Add a length guard in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()`:

```java
// Option A: guard in decodeEvmAddress()
byte[] decoded = Hex.decodeHex(evmAddress);
if (decoded.length != 20) {
    throw new IllegalArgumentException("evmAddress must be exactly 20 bytes");
}
return decoded;
```

```java
// Option B: guard in getByEvmAddressAndType()
byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException("evmAddress must be exactly 20 bytes");
}
```

Option A is preferred as it enforces the invariant at the source and protects all future callers of `decodeEvmAddress()`. [1](#0-0)
