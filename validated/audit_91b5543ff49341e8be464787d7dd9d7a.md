Audit Report

## Title
Permanent Stale Long-Zero Address Caching in `SyntheticLogListener` Due to No-TTL Caffeine Cache

## Summary
`SyntheticLogListener.updateTopicField()` unconditionally writes the long-zero address into a no-TTL Caffeine `LoadingCache` when no EVM address is found for an entity. If that entity later acquires a real EVM address (e.g., via `CRYPTO_UPDATE` adding an ECDSA key), all future synthetic log topics referencing that entity will permanently serve the stale long-zero address for the lifetime of the importer process.

## Finding Description

**Verified code references:**

`CacheProperties.java` line 24 — `evmAddress` cache has no expiry policy: [1](#0-0) 

`SyntheticLogListener.java` lines 225–231 — the `else` branch unconditionally writes the long-zero address (`defaultValue`) into the permanent cache: [2](#0-1) 

`EntityRepository.java` lines 22–23 — the DB query filters out entities with no EVM address, so entities without one are never returned: [3](#0-2) 

**Root cause and failed assumption:**

The comment at lines 226–229 encodes a false assumption: *"we can safely assume the entity does not have an evm address."* This is only true at the moment of processing. Hedera accounts can acquire EVM addresses after creation via `CRYPTO_UPDATE` with an ECDSA key. [4](#0-3) 

Once `getEvmCache().put(key.getId(), defaultValue)` executes, Caffeine's `getAll()` in `onEnd()` will return the cached long-zero address for that key on all subsequent calls — it never calls `loadAll()` for a key already present in the cache. [5](#0-4) 

The `parserContext` fallback check at lines 220–224 is also bypassed in the stale-cache scenario: `entityEvmAddresses.get(key.getId())` returns the cached long-zero address (non-null), so the `else` branch — and with it the `parserContext` check — is never reached, even if the entity was updated in the same record file. [6](#0-5) 

There is no `onStart()` or cache-invalidation hook anywhere in `SyntheticLogListener`.

The test `cachesNoDbResults` (lines 96–120) explicitly validates and locks in this behavior — confirming it is intentional but based on the incorrect assumption: [7](#0-6) 

## Impact Explanation
Synthetic contract logs are the EVM-compatible representation of HTS token transfers, consumed by `eth_getLogs` and EVM indexers. Corrupted `topic1`/`topic2` fields mean Transfer events show the wrong sender/receiver EVM address. DeFi protocols, wallets, and block explorers relying on these events will misattribute token movements. The bloom filter (`contractLog.setBloom`) is also computed from the stale address, causing false negatives in log filtering. [8](#0-7) 

This constitutes permanent, silent corruption of historical synthetic log data for any entity that transitions from no-EVM-address to having one, persisting until the importer process restarts.

## Likelihood Explanation
Any unprivileged Hedera user can trigger this with three self-controlled steps: create an ED25519 account, perform one HTS transfer involving that account, then update the account with an ECDSA key. No special permissions, no admin access, and no race condition exploitation is required. The window is not time-bounded — the cache entry persists until the importer process restarts. On mainnet, where the importer runs continuously for months, the stale entry can corrupt logs indefinitely. The attack is repeatable for any number of entity IDs up to the cache's 500,000-entry limit. [1](#0-0) 

## Recommendation
1. **Remove the unconditional `put` of the long-zero address** from `updateTopicField`. Instead of caching a negative result, leave the key absent from the cache so it is re-queried on the next record file.
2. **Add a TTL** to the `evmAddress` cache spec (e.g., `expireAfterWrite=1h`) so stale entries are eventually evicted, consistent with how other caches in `CacheProperties` are configured. [9](#0-8) 
3. **Invalidate the cache entry** in any code path that processes a `CRYPTO_UPDATE` which adds an EVM address to an existing entity, so the next lookup fetches the fresh value.
4. **Update the test** `cachesNoDbResults` to assert that a subsequent record file *does* re-query the DB for entities whose long-zero address was previously returned, rather than asserting the DB is never re-queried. [10](#0-9) 

## Proof of Concept
1. Create account `0.0.N` with an ED25519 key (no EVM address).
2. Perform an HTS token transfer involving `0.0.N`. The importer processes the record file: `findEvmAddressesByIds` returns no row for `N` (filtered by `length(evm_address) > 0`); `parserContext.get(Entity.class, N)` returns null. Fallback fires: `getEvmCache().put(N, longZeroAddress)`. [2](#0-1) 
3. Submit `CRYPTO_UPDATE` for `0.0.N` adding an ECDSA key, giving it real EVM address `0xABCD...`.
4. Perform another HTS transfer involving `0.0.N`. The importer calls `getEvmCache().getAll(keys)` — the cache already contains `N → longZeroAddress`, so `loadAll()` is never called. `entityEvmAddresses.get(N)` returns the stale long-zero address (non-null), `setter.accept(cachedResult)` is called, and the synthetic log topic is permanently set to the wrong address. [11](#0-10)
