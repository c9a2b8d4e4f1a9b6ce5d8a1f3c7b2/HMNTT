# Audit Report

## Title
Unvalidated `X-Forwarded-For` Header Logged Verbatim Enables IP Spoofing in Rosetta Tracing Middleware

## Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` returns the raw `X-Forwarded-For` header value without splitting on commas or validating IP format. The value is interpolated directly into every Rosetta request log line. Both the Traefik (Kubernetes) and nginx (docker-compose) proxy configurations actively allow client-controlled content to reach this header, making the audit log's source-IP field fully attacker-controlled.

## Finding Description
**File:** `rosetta/app/middleware/trace.go`

```go
// lines 63–74
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)          // line 64

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader)  // line 67 — raw, unvalidated
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr)
    }

    return ipAddress   // returned and logged as-is
}
```

`r.Header.Get()` returns the entire header value as a single string. When the value is `victim_ip, attacker_ip`, the function returns that whole string. The value is then interpolated verbatim at lines 52–53:

```go
message := fmt.Sprintf("%s %s %s (%d) in %s",
    clientIpAddress, request.Method, path, tracingResponseWriter.statusCode, time.Since(start))
```

**Proxy layer amplifies the issue:**

- **Traefik** (`charts/hedera-mirror-common/values.yaml`, lines 643–645): configured with `--entryPoints.web.forwardedHeaders.insecure` for both `web` and `websecure` entry points. This instructs Traefik to trust and pass through `X-Forwarded-For` from *all* sources, including external clients, unmodified.
- **nginx** (`docker-compose.yml`, line 205): uses `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for`. A client sending `X-Forwarded-For: victim_ip` causes nginx to forward `victim_ip, <real_client_ip>`. Rosetta receives and logs the entire string.

**Test gap:** `trace_test.go` lines 56–58 only test `xForwardedForHeader` with a single clean IP (`"10.0.0.100"`). Multi-value and malformed inputs were never considered.

## Impact Explanation
Every Rosetta API request log entry can have its source IP replaced with an arbitrary attacker-controlled string. This directly corrupts the audit trail used to attribute which IP address initiated which Rosetta transaction query. SIEM systems, forensic pipelines, and compliance dashboards ingesting these logs will record the spoofed IP as the originator. Additionally, if the injected string contains newline characters (`\n`), an attacker can forge entirely new log lines (log injection), further corrupting the audit record. The impact is confined to log integrity — no funds are at risk — but the audit trail becomes unreliable for any security or compliance purpose.

## Likelihood Explanation
No authentication or special privilege is required. Any client that can reach the Rosetta HTTP endpoint can send an arbitrary `X-Forwarded-For` header. The Traefik `insecure` forwarded-headers configuration is the default in the provided Helm chart values (`charts/hedera-mirror-common/values.yaml`), making this exploitable in standard Kubernetes deployments out of the box. The attack is trivially repeatable with a single `curl` command.

## Recommendation
1. **Split and take the first IP:** Per RFC 7239 and common proxy convention, the leftmost value in `X-Forwarded-For` is the original client IP. Replace the raw `r.Header.Get(xForwardedForHeader)` call with:
   ```go
   raw := r.Header.Get(xForwardedForHeader)
   if len(raw) > 0 {
       ipAddress = strings.TrimSpace(strings.SplitN(raw, ",", 2)[0])
   }
   ```
2. **Validate the extracted value:** After extraction, call `net.ParseIP(ipAddress)` and fall back to `r.RemoteAddr` if the result is `nil`.
3. **Sanitize before logging:** Strip or escape `\r`, `\n`, and other control characters from `ipAddress` before it is passed to `fmt.Sprintf` to prevent log injection.
4. **Traefik:** Replace `forwardedHeaders.insecure` with `forwardedHeaders.trustedIPs` scoped to known load-balancer CIDR ranges so that only trusted upstream proxies can set `X-Forwarded-For`.

## Proof of Concept

**IP spoofing via nginx (docker-compose):**
```bash
# Client sends a spoofed X-Forwarded-For header
curl -H "X-Forwarded-For: 192.168.1.1" http://localhost:8080/rosetta/network/list

# nginx forwards: X-Forwarded-For: 192.168.1.1, <real_client_ip>
# Rosetta logs:   "192.168.1.1, <real_client_ip> POST /network/list (200) in 3ms"
# The spoofed IP appears as the attributed source.
```

**Log injection (newline injection):**
```bash
curl -g -H $'X-Forwarded-For: 1.2.3.4\ntime="2024-01-01T00:00:00Z" level=info msg="10.0.0.1 POST /construction/submit (200) in 1ms"' \
  http://localhost:8080/rosetta/network/list

# Rosetta logs a forged second line attributing a construction/submit call to 10.0.0.1
```

**IP spoofing via Traefik (Kubernetes):**
```bash
# With --entryPoints.web.forwardedHeaders.insecure, Traefik passes the header through as-is
curl -H "X-Forwarded-For: victim_ip" https://<rosetta-host>/network/list
# Rosetta logs: "victim_ip POST /network/list (200) in 5ms"
```
