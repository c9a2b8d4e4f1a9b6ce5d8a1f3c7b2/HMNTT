# Audit Report

## Title
Zero-Length `initialTransactionId` Bypasses Null Guard, Streams Default `TransactionID` to gRPC Subscribers Breaking Chunk Reassembly

## Summary
An attacker can submit a `ConsensusSubmitMessage` with `chunkInfo.initialTransactionID` set to a default (all-zero) protobuf `TransactionID`. The importer stores this as a zero-length byte array. The gRPC controller's `parseTransactionID` guards only against `null`, so `TransactionID.parseFrom(new byte[0])` silently returns the default instance. Every subscriber receives `chunkInfo.initialTransactionID = {0.0.0, 0.0, 0, false}` as the correlation key, causing incorrect chunk grouping and data loss for multi-chunk messages.

## Finding Description

**Step 1 — Importer stores zero-length bytes**

In `ConsensusSubmitMessageTransactionHandler.doUpdateTransaction()`, when an attacker explicitly sets `chunkInfo.initialTransactionID` to a default `TransactionID`, `hasInitialTransactionID()` returns `true` (protobuf3 message-field presence is set), and `toByteArray()` on the default instance returns `[]` because all fields are at their default values and are omitted during serialization:

```java
if (chunkInfo.hasInitialTransactionID()) {          // true — field was explicitly set
    topicMessage.setInitialTransactionId(
            chunkInfo.getInitialTransactionID().toByteArray()); // returns []
}
``` [1](#0-0) 

The result is `initialTransactionId = new byte[0]` persisted to the database.

**Step 2 — `poll()` streams the record without validation**

`PollingTopicListener.poll()` emits every `TopicMessage` from the repository as-is, with no content validation on `initialTransactionId`:

```java
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [5](#0-4) 

**Step 3 — `parseTransactionID` null-only guard is bypassed**

```java
private TransactionID parseTransactionID(byte[] transactionIdBytes, long topicId, long sequenceNumber) {
    if (transactionIdBytes == null) {   // only null check — empty array passes through
        return null;
    }
    try {
        return TransactionID.parseFrom(transactionIdBytes); // succeeds on [], returns default instance
    } catch (InvalidProtocolBufferException e) { ... }
}
``` [3](#0-2) 

`TransactionID.parseFrom(new byte[0])` does not throw; it returns `TransactionID.getDefaultInstance()` — a non-null object with `accountID = 0.0.0`, `transactionValidStart = 0.0`, `nonce = 0`, `scheduled = false`.

**Step 4 — Default `TransactionID` is set on the chunk and streamed**

Because the returned value is non-null, the `if (transactionID != null)` branch is taken, the fallback path using `payerAccountEntity`/`validStartInstant` is bypassed, and the all-zero key is set:

```java
if (transactionID != null) {                              // true — default instance is non-null
    chunkBuilder.setInitialTransactionID(transactionID);  // sets {0.0.0, 0.0, 0, false}
} else if (payerAccountEntity != null && validStartInstant != null) {
    // fallback — never reached
}
``` [4](#0-3) 

**Root cause / failed assumption**: The code assumes `transactionIdBytes != null` implies a valid, non-empty serialized `TransactionID`. This is false: protobuf allows a message with all-default fields to serialize to zero bytes, and `parseFrom` on zero bytes returns the default instance without error.

## Impact Explanation
SDK clients (Hedera Java SDK and compatible) use `initialTransactionID` as the correlation key to group chunks of a multi-chunk message into a single logical message. When an attacker submits multiple chunk transactions each carrying an empty `initialTransactionID`, all such chunks share the same correlation key (`0.0.0 / 0.0`). The SDK incorrectly groups them together regardless of which logical message they belong to. Legitimate multi-chunk messages whose chunks are interleaved with tampered ones will be mis-assembled or dropped entirely, causing **data corruption and data loss** for all subscribers of the affected topic.

## Likelihood Explanation
Preconditions are minimal: the attacker needs only a funded Hedera account and a target topic without a `submitKey` (the majority of public HCS topics). The attack is fully repeatable and scriptable. No privileged access, no key material beyond the attacker's own account key, and no race condition is required. The Hedera network does not validate the semantic content of `chunkInfo.initialTransactionID`.

## Recommendation
In `ConsensusController.parseTransactionID`, add an explicit length check before attempting to parse:

```java
private TransactionID parseTransactionID(byte[] transactionIdBytes, long topicId, long sequenceNumber) {
    if (transactionIdBytes == null || transactionIdBytes.length == 0) {
        return null;
    }
    try {
        return TransactionID.parseFrom(transactionIdBytes);
    } catch (InvalidProtocolBufferException e) {
        log.error("Failed to parse TransactionID for topic {} sequence number {}", topicId, sequenceNumber);
        return null;
    }
}
```

This ensures that a zero-length byte array (produced by serializing a default `TransactionID`) is treated the same as a missing `initialTransactionID`, allowing the fallback path (lines 105–109) to reconstruct a meaningful key from `payerAccountId` and `validStartTimestamp` instead.

Additionally, consider adding a validation step in `ConsensusSubmitMessageTransactionHandler.doUpdateTransaction()` to reject or skip storing a zero-length `initialTransactionId`:

```java
if (chunkInfo.hasInitialTransactionID()) {
    byte[] bytes = chunkInfo.getInitialTransactionID().toByteArray();
    if (bytes.length > 0) {
        topicMessage.setInitialTransactionId(bytes);
    }
}
``` [1](#0-0) 

## Proof of Concept

1. Create a Hedera account and obtain a topic ID for a topic without a `submitKey`.
2. Construct a `ConsensusSubmitMessage` transaction with:
   - `chunkInfo.number = 1`, `chunkInfo.total = 2`
   - `chunkInfo.initialTransactionID = TransactionID.getDefaultInstance()` (all-zero)
3. Submit the transaction. The importer stores `initialTransactionId = []`.
4. Subscribe to the topic via the mirror node gRPC API.
5. Observe that the streamed `ConsensusTopicResponse` contains `chunkInfo.initialTransactionID = {accountID: 0.0.0, transactionValidStart: 0.0}`.
6. Repeat for any number of unrelated chunk transactions — all will share the same correlation key, causing the SDK to mis-assemble them into a single logical message.
