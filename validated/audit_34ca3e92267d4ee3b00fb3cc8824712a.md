# Audit Report

## Title
Cache Slot-Key Seeding Amplification: Unprivileged DB Batch Query Inflation in `findStorageBatch()`

## Summary
An unprivileged caller can permanently seed the per-contract `slotsPerContract` Caffeine cache (up to `maximumSize=1500`) with attacker-chosen slot keys by issuing 1500 `eth_call`/`eth_getStorageAt` requests. Because non-existent slots are never written back to the short-lived `contractState` cache (`expireAfterWrite=2s`), every subsequent legitimate request for the targeted contract — after each 2-second expiry — triggers a PostgreSQL `IN (...)` query over all 1500 seeded keys instead of a single-slot lookup. This constitutes a sustained 1500× amplification of database work per affected request, maintainable at ~6 req/s.

## Finding Description

**File:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`

**Root cause — three cooperating lines:**

1. **Line 90** — every call to `findStorageBatch()` unconditionally inserts the requested slot key into the shared per-contract cache with no caller validation or per-caller quota:
   ```java
   contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
   ``` [8](#0-7) 

2. **Line 91** — immediately after insertion, the *entire* key-set of that per-contract cache is read:
   ```java
   final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
   ``` [9](#0-8) 

3. **Line 103** — all collected keys are forwarded to PostgreSQL in a single `IN (...)` batch query:
   ```java
   final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
   ``` [2](#0-1) 

**Cache configuration that enables the attack:**

| Cache | Config | Effect |
|---|---|---|
| `slotsPerContract` | `expireAfterAccess=5m, maximumSize=1500` | Holds up to 1500 slot keys per contract; attacker resets timer by re-querying |
| `contractSlots` (outer) | `expireAfterAccess=5m, maximumSize=3000` | Holds per-contract caches for up to 3000 contracts |
| `contractState` | `expireAfterWrite=2s, maximumSize=25000` | Expires every 2 s, forcing re-entry into `findStorageBatch()` | [10](#0-9) 

**Why non-existent slots sustain the attack:** Lines 106–113 only write back slots that the DB actually returns. Attacker-chosen fake slots produce no DB row, so they are never cached in `contractStateCache`. On every 2-second expiry, the next legitimate request re-enters `findStorageBatch()` and re-issues the full 1500-slot batch query. [11](#0-10) 

**Failed assumption:** The design assumes the per-contract slot-key set grows organically from legitimate traffic and stays small. There is no validation that the caller has any relationship to the contract, no per-caller injection limit, and no distinction between attacker-injected and organically-observed keys.

## Impact Explanation

Every legitimate request for the targeted contract, after each 2-second `contractState` cache expiry, causes PostgreSQL to evaluate a `WHERE slot IN (...)` clause over 1500 entries instead of 1. On a node serving hundreds of requests per second for a popular contract, this translates to a sustained multiplicative increase in DB CPU, I/O, and connection pool pressure. An attacker targeting multiple contracts simultaneously (up to 3000 fit in the outer `contractSlots` cache) multiplies the impact further. The amplification is bounded only by `maximumSize=1500` — a hard ceiling that still represents a severe per-request cost increase without any brute-force volume from the attacker.

## Likelihood Explanation

The attack requires zero privileges — only access to the public `eth_call` or `eth_getStorageAt` JSON-RPC endpoints. The seeding phase is a one-time cost of 1500 requests (completable in seconds). Maintenance requires only ~6 req/s to reset `expireAfterAccess` timers, far below any rate limit. The attacker needs no knowledge of actual contract storage layout; arbitrary, non-existent slot keys are accepted and cached identically to real ones. The attack is fully automatable with a trivial script and is repeatable indefinitely.

## Recommendation

1. **Cap per-caller slot-key injection:** Track slot keys per (contractId, callerAddress) and refuse to insert into the shared per-contract cache beyond a small per-caller quota (e.g., 10–20 keys).

2. **Cache negative results:** After `findStorageBatch()` returns, explicitly write `EMPTY_VALUE` into `contractStateCache` for every queried slot that had no DB result. This prevents the batch query from re-firing on the next 2-second expiry for slots already known to be absent. [12](#0-11) 

3. **Separate attacker-injected keys from organically-observed keys:** Only add a slot key to the shared per-contract batch set if it was observed in a prior *successful* (non-empty) DB lookup, not on every incoming request.

4. **Reduce `slotsPerContract` maximumSize** or tie it to observed contract storage density rather than a fixed global cap.

## Proof of Concept

```python
import requests, hashlib

NODE_URL = "http://<mirror-node>/api/v1/contracts/call"
TARGET_CONTRACT = "0x<popular_contract>"

# Phase 1: Seed 1500 fake slot keys (one-time, ~3 seconds at 500 RPS)
for i in range(1500):
    slot = "0x" + hashlib.sha256(i.to_bytes(4, "big")).hexdigest()
    requests.post(NODE_URL, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [TARGET_CONTRACT, slot, "latest"], "id": i
    })

# Phase 2: Maintain seeded state (~6 req/s, resets expireAfterAccess timer)
import time, itertools
for i in itertools.cycle(range(1500)):
    slot = "0x" + hashlib.sha256(i.to_bytes(4, "big")).hexdigest()
    requests.post(NODE_URL, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [TARGET_CONTRACT, slot, "latest"], "id": i
    })
    time.sleep(1/6)  # 6 req/s total across 1500 slots = each slot re-queried every ~4 min
```

After Phase 1, every legitimate user querying any slot of `TARGET_CONTRACT` (after the 2-second `contractState` expiry) triggers a 1500-slot `IN (...)` PostgreSQL query instead of a single-slot lookup. The DB amplification is sustained indefinitely by Phase 2.
