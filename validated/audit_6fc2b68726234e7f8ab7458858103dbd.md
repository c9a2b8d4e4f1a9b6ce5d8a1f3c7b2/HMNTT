# Audit Report

## Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` Produces Negative `pageSize`, Bypassing `hasLimit()` Guard and Enabling Unbounded Database Queries

## Summary
A `long`-to-`int` narrowing cast in `poll()` converts a user-supplied `limit` value greater than `Integer.MAX_VALUE` into a large negative integer. That negative value is written into a new `TopicMessageFilter` via Lombok's `build()` — which never invokes Bean Validation — causing `hasLimit()` to return `false` and `setMaxResults` to be skipped, issuing a fully unbounded SQL query.

## Finding Description

**Unchecked narrowing cast** in `PollingTopicMessageRetriever.poll()`: [1](#0-0) 

`filter.getLimit()` is a `long`; `context.getTotal().get()` is an `AtomicLong`. Their difference is a `long` that is silently narrowed to `int`. When `filter.getLimit() = 2147483648L` (Integer.MAX_VALUE + 1) and `total = 0`, the result is `(int) 2147483648L = -2147483648` (Integer.MIN_VALUE). `Math.min(-2147483648, positiveMaxPageSize)` returns the negative value, not the bounded page size.

**`limit` field is `long` with `@Min(0)` only:** [2](#0-1) 

Any non-negative `long` — including values far above `Integer.MAX_VALUE` — satisfies `@Min(0)` and passes the `@Valid` check on `subscribeTopic`. [3](#0-2) 

**Builder bypasses Bean Validation.** The `@Validated` annotation on `TopicMessageFilter` activates Spring AOP method-level validation only when the object is passed through a Spring proxy as a `@Valid` parameter. Lombok's `build()` is plain Java — it never invokes the Jakarta Validation engine. The negative `pageSize` (widened back to a negative `long`) is stored in `newFilter.limit` unchecked. [4](#0-3) 

**`hasLimit()` returns `false` for negative values:** [5](#0-4) 

**`setMaxResults` is never called when `hasLimit()` is `false`:** [6](#0-5) 

The JPA query is issued with no `LIMIT` clause, returning every row matching the topic/timestamp predicate.

**`isComplete()` never terminates in unthrottled mode** because `filter.getLimit() == total.get()` (`2147483648L == n`) is never true for realistic row counts: [7](#0-6) 

Polling repeats for the full `maxPolls` count (default 12), each issuing an unbounded query.

## Impact Explanation

Each unbounded query loads all `topic_message` rows for the targeted topic via `getResultList()`: [8](#0-7) 

For a high-volume topic this can be millions of rows loaded into JVM heap. In unthrottled mode (default `maxPolls = 12`, `pollingFrequency = 20 ms`), 12 unbounded queries fire in ~240 ms per subscription. [9](#0-8) 

Multiple concurrent subscriptions multiply the effect. Consequences: heap exhaustion / OOM on the gRPC service, database connection pool saturation, and denial of service to all other subscribers. The only backstop is `retrieverProperties.getTimeout()` (default 60 s), but damage accumulates within that window. [10](#0-9) 

## Likelihood Explanation

The gRPC `uint64 limit` field maps to a signed Java `long`. Any value above `2,147,483,647` — trivially expressible in any gRPC client — triggers the overflow. No authentication or special privilege is required; `subscribeTopic` is the public API. The overflow fires on the very first `poll()` invocation when `total == 0`. The attack is trivially repeatable.

## Recommendation

1. **Replace the narrowing cast** with a bounds-safe calculation:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Add an explicit upper-bound validation** on `TopicMessageFilter.limit` (e.g., `@Max(Integer.MAX_VALUE)`) so the original filter is rejected before it ever reaches `poll()`.
3. **Validate the internally-constructed filter** by calling the Jakarta Validator explicitly after `build()`, or add a factory method that enforces invariants.

## Proof of Concept

```python
import grpc
from mirror_node_pb2 import ConsensusTopicQuery
from mirror_node_pb2_grpc import ConsensusServiceStub

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = ConsensusServiceStub(channel)

query = ConsensusTopicQuery()
query.topicID.topicNum = 1234
query.limit = 2147483648  # Integer.MAX_VALUE + 1; valid uint64, triggers long→int overflow

# Each message received triggers an unbounded DB query (no LIMIT clause)
for msg in stub.subscribeTopic(query):
    print(msg)
```

On the first `poll()` call with `total = 0`:
- `(int)(2147483648L - 0L)` → `-2147483648`
- `Math.min(-2147483648, 5000)` → `-2147483648`
- `newFilter.limit = -2147483648L`
- `hasLimit()` → `false`
- `setMaxResults` never called → full table scan per poll, repeated 12 times
