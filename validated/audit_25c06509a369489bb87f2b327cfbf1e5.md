Audit Report

## Title
Silent Importer Stall via Malformed Data Files Uploaded to Cloud Storage Bucket

## Summary
In `Downloader.verifySignatures()`, when `streamFileReader.read()` throws `InvalidStreamFileException` or `StreamFileReaderException`, the exception falls through to a generic `catch (Exception e)` block that logs an error and silently advances to the next node. If all CONSENSUS_REACHED nodes serve malformed data files, every iteration fails, `onVerified()` is never called, `lastStreamFile` is never advanced, and the importer retries the same batch indefinitely.

## Finding Description

**Verified code path in `Downloader.java`:**

`verifySignatures()` spans lines 320–383. The critical sequence is:

```java
// line 336
T streamFile = streamFileReader.read(streamFileData);
// line 338 — never reached if read() throws
verify(streamFile, signature);
// line 361 — never reached either
onVerified(streamFileData, streamFile);
``` [1](#0-0) 

The catch structure at lines 363–379:

```java
} catch (FileOperationException | HashMismatchException | TransientProviderException e) {
    log.warn(...);   // named: warn + continue
} catch (Exception e) {
    log.error("Error downloading data file from node {} ...", nodeId, ...);
    // generic: error + continue
}
``` [2](#0-1) 

`InvalidStreamFileException` and `StreamFileReaderException` both extend `ImporterException` → `MirrorNodeException` → `RuntimeException`. Neither appears in the named multi-catch, so both land in the generic handler. [3](#0-2) [4](#0-3) [5](#0-4) 

**Trigger in `CompositeRecordFileReader.java`:**

Any version integer not in `{1, 2, 5, 6}` causes an `InvalidStreamFileException` at line 55–56, before any hash verification occurs:

```java
default:
    throw new InvalidStreamFileException(
        String.format("Unsupported record file version %d in file %s", version, filename));
``` [6](#0-5) 

**Stall mechanism:**

After all CONSENSUS_REACHED nodes are exhausted, `verifySignatures()` returns `false` (line 382). Back in `verifySigsAndDownloadDataFiles()`, this only logs `"None of the data files could be verified"` — no exception is thrown, no state is advanced. [7](#0-6) 

`lastStreamFile` (line 86) is only updated inside `onVerified()` at line 401. Since `onVerified()` is never reached, `getStartAfterFilename()` (lines 253–263) returns the same filename on every subsequent scheduled invocation, causing the same batch to be re-downloaded and re-failed indefinitely. [8](#0-7) [9](#0-8) [10](#0-9) 

## Impact Explanation
The importer permanently stalls on the targeted batch file. No subsequent record files, balance files, or event files are processed. The mirror node falls arbitrarily far behind the network. Because the failure is only logged (not alerted), operators may not notice until the mirror node's data is visibly stale. This constitutes a sustained denial-of-service against the mirror node's stream processing pipeline.

## Likelihood Explanation
The precondition is write access to the cloud storage bucket. This is achievable via misconfigured bucket ACLs (public write), compromised cloud credentials, or a supply-chain/insider attack on the storage layer. The attacker does not need any Hedera network keys or node access. Once write access is obtained, the attack is trivially repeatable: upload a 4-byte file (e.g., `\x00\x00\x00\x07` — version 7, unsupported) to the data file path for each CONSENSUS_REACHED node. The attack persists until an operator manually removes the malformed files and restarts the importer.

## Recommendation
1. **Promote parser exceptions to the named catch**: Add `InvalidStreamFileException` and `StreamFileReaderException` (or their common parent `ImporterException`) to the explicit multi-catch at line 363 so they are handled with the same `warn`-and-continue semantics as `HashMismatchException`.
2. **Circuit-break on repeated parse failures**: Track consecutive parse failures per filename. After a configurable threshold (e.g., all CONSENSUS_REACHED nodes fail with parse errors for the same file), throw a `SignatureVerificationException` or halt polling rather than silently looping.
3. **Alert on `verifySignatures()` returning `false`**: Replace the `log.error` at line 310 with a metric increment or an alertable event so operators are notified promptly when no data file can be verified.
4. **Bucket write access hardening**: Enforce least-privilege IAM policies on the cloud storage bucket; the importer should only require read access.

## Proof of Concept
1. Identify the data file path for the next expected batch (e.g., `recordstreams/record0.0.3/2024-01-01T00_00_00.000000000Z.rcd`).
2. For each CONSENSUS_REACHED node's bucket path, upload a 4-byte file: `\x00\x00\x00\x07` (big-endian integer 7 — an unsupported version).
3. On the next scheduled invocation, `CompositeRecordFileReader.read()` reads version `7`, hits the `default` branch, and throws `InvalidStreamFileException("Unsupported record file version 7 in file ...")`.
4. The generic `catch (Exception e)` at line 373 logs the error and continues to the next node.
5. After all nodes are exhausted, `verifySignatures()` returns `false`; `lastStreamFile` is unchanged.
6. `getStartAfterFilename()` returns the same filename; the next invocation downloads and fails on the same batch.
7. The importer stalls indefinitely with only `ERROR` log output and no state advancement.
