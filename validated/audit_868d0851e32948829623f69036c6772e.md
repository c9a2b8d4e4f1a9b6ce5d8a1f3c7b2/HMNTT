# Audit Report

## Title
Non-JSON HTTP 200 Response Silently Kills Retry Loop, Preventing Pinger Startup

## Summary
In `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()` returns `retry=false` on any JSON decode failure, regardless of whether the failure is transient. Because `buildNetworkFromMirrorNodes()` breaks its retry loop immediately when `retry=false`, a single HTTP 200 response with a non-JSON body (e.g., an HTML error page) bypasses all configured retry attempts and causes the pinger process to exit via `log.Fatalf`.

## Finding Description

**`pinger/mirror_node_client.go` — `fetchMirrorNodeNetwork()`**

The HTTP status check at lines 90–93 only gates on non-2xx responses. An HTTP 200 with any body passes through unconditionally: [7](#0-6) 

The JSON decode at lines 96–98 returns `(nil, false, err)` on failure — `false` meaning "do not retry": [1](#0-0) 

There is no `Content-Type: application/json` validation between these two points.

**`pinger/mirror_node_client.go` — `buildNetworkFromMirrorNodes()`**

Line 59 breaks the retry loop immediately when `retry=false`, regardless of how many attempts remain (`mirrorNodeClientMaxRetries`, default 10): [2](#0-1) 

**`pinger/sdk_client.go` — `newClient()`**

The error from `buildNetworkFromMirrorNodes` is returned directly to the caller: [3](#0-2) 

**`pinger/main.go` — `main()`**

The error from `newClient` triggers `log.Fatalf`, terminating the process: [4](#0-3) 

**Root cause:** The code conflates a transient/attacker-induced decode error with a permanent, non-retryable failure by hardcoding `retry=false` at line 97. The retry gate at line 59 is correctly implemented but is never given the opportunity to function for decode errors.

## Impact Explanation
When `network=other` is configured, the pinger depends entirely on `buildNetworkFromMirrorNodes` to construct the Hiero node map at startup. A single poisoned HTTP 200 response with a non-JSON body causes `newClient` to return an error, which triggers `log.Fatalf` in `main()`, killing the pinger process. The result is complete cessation of all transfer submissions for the duration of the outage. The `/tmp/ready` readiness file is never written, so Kubernetes readiness probes will also fail, preventing the pod from receiving traffic or being considered healthy.

## Likelihood Explanation
The default mirror REST URL is plaintext HTTP (`http://mirror-rest:5551`): [5](#0-4) 

In a Kubernetes deployment (evidenced by `/tmp/alive` and `/tmp/ready` probe files), an attacker with any foothold in the cluster network can perform ARP spoofing or DNS poisoning to redirect `mirror-rest` to a server returning HTTP 200 with a non-JSON body. Additionally, a temporary misconfiguration of the mirror node (e.g., a CDN or WAF returning an HTML error page with status 200) triggers the same code path with no retry. The attack is repeatable on every pinger restart.

## Recommendation
Change `fetchMirrorNodeNetwork()` to return `retry=true` on JSON decode failure, treating it as a transient error:

```go
// line 97 — change false to true
return nil, true, fmt.Errorf("decode mirror nodes: %w", err)
```

Additionally, validate the `Content-Type` header before attempting to decode:

```go
if ct := resp.Header.Get("Content-Type"); !strings.Contains(ct, "application/json") {
    return nil, true, fmt.Errorf("unexpected Content-Type %q from %s", ct, url)
}
```

This ensures that transient non-JSON responses (HTML error pages, WAF interstitials, MITM-injected responses) are retried up to `mirrorNodeClientMaxRetries` times before the pinger gives up.

## Proof of Concept
1. Configure the pinger with `HIERO_MIRROR_PINGER_NETWORK=other` and `HIERO_MIRROR_PINGER_REST=http://mirror-rest:5551`.
2. Stand up a server at `mirror-rest:5551` (or redirect DNS/ARP to one) that responds to `GET /api/v1/network/nodes` with HTTP 200 and body `<html>error</html>`.
3. Start the pinger.
4. Observe: `fetchMirrorNodeNetwork` returns `retry=false` on the first attempt; `buildNetworkFromMirrorNodes` breaks immediately; `newClient` returns an error; `main` calls `log.Fatalf("client error: ...")` and the process exits. All 10 configured retries are skipped.
