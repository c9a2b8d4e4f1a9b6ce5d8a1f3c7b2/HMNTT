Audit Report

## Title
`ConstructionParse` Returns Unverified `AccountIdentifierSigners` Without Validating Embedded Signatures

## Summary
`ConstructionParse` with `request.Signed = true` populates `AccountIdentifierSigners` from transaction body fields (sender accounts / payer account ID) without verifying that the cryptographic signatures embedded in the transaction actually correspond to those accounts. An unauthenticated caller can craft a transaction naming arbitrary victim accounts as senders, attach attacker-controlled signatures, and receive a response that falsely asserts those victim accounts authorized the transaction.

## Finding Description

**Exact code path:**

In `ConstructionParse` (`rosetta/app/services/construction_service.go`, lines 183–215), the accounts returned by `transactionHandler.Parse()` are directly promoted to `AccountIdentifierSigners` when `request.Signed == true`, with no signature verification: [1](#0-0) 

For `cryptoTransferTransactionConstructor.Parse()` (`rosetta/app/services/construction/crypto_transfer_transaction_constructor.go`), signers are derived entirely from the transaction body's `HbarTransfers` map — any account with a negative transfer amount is returned as a signer: [2](#0-1) 

For `cryptoCreateTransactionConstructor.Parse()` (`rosetta/app/services/construction/crypto_create_transaction_constructor.go`), the signer is the payer from `GetTransactionID().AccountID`: [3](#0-2) 

**Root cause:** Neither `ConstructionParse` nor the underlying `Parse()` implementations inspect the `sigMap` (signature map) embedded in the serialized transaction. `AccountIdentifierSigners` is populated purely from transaction body fields that the attacker fully controls.

**Why existing checks fail:** `ConstructionCombine` does perform `ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, signature.Bytes)` before embedding signatures: [4](#0-3) 

However, `ConstructionParse` performs no equivalent check. An attacker bypasses `ConstructionCombine` entirely by constructing the signed transaction hex directly (e.g., using the Hiero SDK or raw protobuf), embedding their own valid-but-irrelevant signatures, and submitting it straight to `ConstructionParse`.

## Impact Explanation
Any system that relies on `ConstructionParse`'s `AccountIdentifierSigners` to confirm multi-party authorization — such as an exchange's withdrawal pipeline, a multi-sig coordinator, or an automated compliance monitor — can be deceived into believing a victim account has approved a transaction it never signed. This enables fraudulent authorization attestation at the Rosetta API layer. The Rosetta spec explicitly states that when `signed=true`, `AccountIdentifierSigners` must reflect accounts that *have* signed; violating this contract breaks the trust model of the entire construction workflow.

## Likelihood Explanation
The attack requires zero privileges. The `/construction/parse` endpoint is a public, unauthenticated, offline endpoint by design. The attacker needs only the Hiero SDK or raw protobuf tooling to construct a valid serialized transaction. The attack is fully repeatable and requires no on-chain interaction. Any operator exposing the Rosetta API publicly is immediately exposed to this deception.

## Recommendation
In `ConstructionParse`, when `request.Signed == true`, verify that each account returned by `Parse()` has a corresponding valid signature in the transaction's `sigMap` before including it in `AccountIdentifierSigners`. Specifically:

1. Extract `frozenBodyBytes` from the signed transaction (reusing the existing `getFrozenTransactionBodyBytes` helper).
2. Retrieve the `sigMap` from the deserialized `SignedTransaction` proto.
3. For each account in the `accounts` slice returned by `transactionHandler.Parse()`, confirm that the `sigMap` contains at least one signature entry whose public key corresponds to that account and whose signature bytes pass `ed25519.Verify(pubKey.Bytes(), frozenBodyBytes, sigBytes)`.
4. Only include accounts that pass this check in `AccountIdentifierSigners`; return an error or omit unverified accounts.

This mirrors the verification already performed in `ConstructionCombine` and closes the gap between the two endpoints.

## Proof of Concept

```
1. Generate an attacker Ed25519 keypair (attacker_priv, attacker_pub).

2. Using the Hiero SDK, construct a TransferTransaction:
     - AddHbarTransfer(0.0.VICTIM, -100 HBAR)
     - AddHbarTransfer(0.0.ATTACKER, +100 HBAR)
     - SetTransactionID with payer = 0.0.VICTIM

3. Freeze the transaction and sign it with attacker_priv:
     tx.FreezeWith(nil)
     tx.Sign(attacker_priv)

4. Serialize to bytes and hex-encode:
     txBytes, _ := tx.ToBytes()
     txHex = hex.EncodeToString(txBytes)

5. POST to /construction/parse:
     {
       "network_identifier": {...},
       "signed": true,
       "transaction": "<txHex>"
     }

6. Server response:
     {
       "account_identifier_signers": [{"address": "0.0.VICTIM"}],
       "operations": [...]
     }

The server returns 0.0.VICTIM as a signer despite the transaction being
signed exclusively by attacker_priv. No valid signature from 0.0.VICTIM
exists in the sigMap, yet the endpoint attests to their authorization.
```
