Audit Report

## Title
Ghost Entity Resurrection: Deleted Entity Returned as Active via Missing Upper-Bound Check in `findActiveByEvmAddressAndTimestamp`

## Summary
The `entity_history` branch of the `UNION ALL` in `findActiveByEvmAddressAndTimestamp` checks only `lower(eh.timestamp_range) <= blockTimestamp` but never verifies `blockTimestamp < upper(eh.timestamp_range)`. When queried at a timestamp after an entity's deletion, the query returns the pre-deletion history record (with `deleted=false`) as if the entity were still active.

## Finding Description

**File:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 59–85

The query is:

```sql
with entity_cte as (
    select id from entity
    where evm_address = ?1 and created_timestamp <= ?2
    order by created_timestamp desc limit 1
)
(select * from entity e
 where e.deleted is not true and e.id = (select id from entity_cte))
union all
(select * from entity_history eh
 where lower(eh.timestamp_range) <= ?2          -- ← only lower bound checked
 and eh.id = (select id from entity_cte)
 order by lower(eh.timestamp_range) desc limit 1)
order by timestamp_range desc limit 1
```

**Confirmed data model** (`SqlEntityListenerTest.onEntityHistory`, lines 1076–1134):

After create (T1) → update (T2) → delete (T3):
- `entity` table: `id=100`, `deleted=true`, `timestamp_range=[T3, ∞)`
- `entity_history` table: `id=100`, `deleted=false`, `timestamp_range=[T2, T3)` ← pre-deletion snapshot

**Query trace with `blockTimestamp = T_query` where `T3 < T_query`:**

1. **CTE** — `entity` table has `created_timestamp=T1 ≤ T_query` → returns `id=100`.
2. **First branch** — `entity` has `deleted=true` → filtered out, returns nothing.
3. **Second branch** — `lower([T2,T3)) = T2 ≤ T_query` ✓, no upper-bound check → returns the pre-deletion record with `deleted=false`.
4. **Final `ORDER BY timestamp_range DESC LIMIT 1`** — only one row in the union; returns the stale record.

**Root cause:** Line 77 checks only `lower(eh.timestamp_range) <= ?2`. A history record with range `[T2, T3)` is valid only for timestamps in `[T2, T3)`. When `T_query ≥ T3`, the record is expired but still matched because `upper(eh.timestamp_range)` is never tested.

**Contrast with `findActiveByIdAndTimestamp`** (lines 136–155): that query adds `and deleted is not true` to the history branch, but it also lacks the upper-bound check. The `deleted is not true` guard is irrelevant here because the pre-deletion snapshot has `deleted=false`; the missing upper-bound check is the actual defect in both queries.

**No existing test covers this scenario.** The closest test, `findByEvmAddressAndTimestampRangeLessThanBlockTimestampAndDeletedTrueCall` (lines 117–123), only persists a deleted entity in the `entity` table with no corresponding history record, so the history branch returns nothing and the test passes vacuously. ## Impact Explanation
`CommonEntityAccessor.getEntityByEvmAddressAndTimestamp` (line 69) and `getEntityByEvmAddressTimestamp` (line 82) both delegate directly to `findActiveByEvmAddressAndTimestamp`. Any historical `eth_call` or equivalent JSON-RPC call that resolves a contract/account by EVM address at a block after the entity's deletion will receive the pre-deletion record with `deleted=false`. The EVM simulation layer then treats the non-existent entity as live, producing incorrect execution results: calls to a deleted contract succeed, balance/nonce/bytecode lookups return stale data, and existence checks return `true` when they should return `false`. [4](#0-3) [5](#0-4) 

## Likelihood Explanation
Any unprivileged caller can trigger this with a standard `eth_call` at a historical block number. No special credentials or on-chain access are required. The attacker only needs to know an EVM address that was deleted at some point — this is publicly observable from the ledger. The vulnerable window spans from the deletion timestamp to either re-creation or indefinitely if the address is never re-created, making the condition reliably and repeatably triggerable.

## Recommendation
Add an upper-bound check to the `entity_history` branch in both `findActiveByEvmAddressAndTimestamp` and `findActiveByEvmAddressOrAliasAndTimestamp`:

```sql
where lower(eh.timestamp_range) <= ?2
  and ?2 < upper(eh.timestamp_range)   -- add this line
  and eh.id = (select id from entity_cte)
```

This mirrors the semantics of a PostgreSQL range containment check (`eh.timestamp_range @> ?2::int8`) and ensures that only history records whose validity window actually contains `blockTimestamp` are considered. Additionally, add `and deleted is not true` to the history branch (consistent with `findActiveByIdAndTimestamp`) to guard against returning explicitly-deleted snapshots. ## Proof of Concept

**Setup:**
```sql
-- Entity created at T1=100, updated at T2=200, deleted at T3=300
INSERT INTO entity (id, evm_address, created_timestamp, deleted, timestamp_range)
  VALUES (100, '\xDEAD...', 100, true, '[300,)');

INSERT INTO entity_history (id, evm_address, created_timestamp, deleted, timestamp_range)
  VALUES (100, '\xDEAD...', 100, false, '[200,300)');  -- pre-deletion snapshot
```

**Trigger:**
```java
// blockTimestamp = 400 (after deletion at T3=300)
Optional<Entity> result = entityRepository
    .findActiveByEvmAddressAndTimestamp(evmAddress, 400L);

// result.isPresent() == true  ← BUG: should be empty
// result.get().isDeleted()    == false  ← BUG: entity was deleted at T=300
```

**Expected:** `Optional.empty()` — entity was deleted at T=300, queried at T=400.
**Actual:** Returns the `entity_history` record with `deleted=false` and `timestamp_range=[200,300)`, because `lower([200,300)) = 200 ≤ 400` passes and no upper-bound check exists. [9](#0-8) [10](#0-9)
