Audit Report

## Title
Unbounded `CaffeineCache` Instance Accumulation in `cacheManagerSlotsPerContract` via Unauthenticated `eth_call` Requests

## Summary
`cacheManagerSlotsPerContract` is configured in dynamic mode (no `setCacheNames()`), causing Spring's `CaffeineCacheManager` to permanently accumulate `CaffeineCache` instances in its internal unbounded `ConcurrentHashMap` — one per unique `contractId` — with no eviction path. The outer `contractSlotsCache` (max 3000 entries) evicts its own entries but never removes the corresponding `CaffeineCache` objects from `cacheManagerSlotsPerContract`'s internal map. An unauthenticated attacker can drive unbounded heap growth via the public `/api/v1/contracts/call` endpoint.

## Finding Description

**`EvmConfiguration.java` lines 107–112** — `cacheManagerSlotsPerContract` is created without `setCacheNames()`, placing it in dynamic mode:

```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    return caffeineCacheManager;  // no setCacheNames() → dynamic mode
}
``` [1](#0-0) 

Every other `CaffeineCacheManager` bean in the same file calls `setCacheNames()` to fix the set of allowed cache names, which prevents dynamic creation. [2](#0-1) 

**`ContractStateServiceImpl.java` lines 85–87** — for every outer-cache miss, `getCache(contractId.toString())` is called, which creates and permanently registers a new `CaffeineCache` in `cacheManagerSlotsPerContract`'s internal `ConcurrentHashMap<String, Cache>`:

```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [3](#0-2) 

**`CacheProperties.java`** confirms the relevant bounds:
- `contractSlots = "expireAfterAccess=5m,maximumSize=3000,recordStats"` — the outer cache holding `CaffeineCache` references, max 3000 entries. [4](#0-3) 
- `slotsPerContract = "expireAfterAccess=5m,maximumSize=1500"` — governs entries *within* each per-contract cache, not the number of cache instances. [5](#0-4) 

**Root cause:** Spring's `CaffeineCacheManager.getCache(name)` in dynamic mode inserts into an internal `ConcurrentHashMap<String, Cache>` and never removes entries from it. When the outer `contractSlotsCache` evicts a `(contractId → CaffeineCache)` entry (due to `maximumSize=3000` or `expireAfterAccess=5m`), the `CaffeineCache` object is removed from the outer cache but its entry in `cacheManagerSlotsPerContract`'s internal map is untouched. The `expireAfterAccess=5m` / `maximumSize=1500` on `slotsPerContract` apply to key-value entries *inside* each `CaffeineCache`, not to the lifecycle of the `CaffeineCache` instances themselves.

## Impact Explanation
Each `CaffeineCache` backed by Caffeine's `BoundedLocalCache` with `maximumSize=1500` pre-allocates internal arrays and bookkeeping structures (several KB to tens of KB per instance). With tens of thousands of unique contractIds, `cacheManagerSlotsPerContract`'s internal map accumulates a proportional number of `CaffeineCache` objects that are never garbage-collected (the map holds strong references). This causes sustained heap growth, increased GC pressure, GC pauses, and eventual OOM, directly degrading or halting EVM simulation request handling.

## Likelihood Explanation
`/api/v1/contracts/call` requires no authentication and accepts any valid 20-byte hex address as the `to` field. An attacker can trivially script thousands of requests with distinct addresses. The `ThrottleManager` limits gas per request but does not bound the cumulative number of distinct contractIds processed over time. The attack is low-cost, requires no on-chain activity, and is cumulative — each request permanently adds to the leaked cache instances.

## Recommendation
1. **Fix (preferred):** Call `setCacheNames(Set.of())` or pre-populate a fixed set of cache names on `cacheManagerSlotsPerContract`. Instead of using the `CaffeineCacheManager` as a per-contract cache factory, maintain a single `Cache` instance and use composite keys `(contractId, slotKey)`.
2. **Alternative:** Replace the two-level caching design with a single `CaffeineCache` keyed by `(contractId, slotKey)` with a global `maximumSize`, eliminating the need for dynamic cache instance creation entirely.
3. **Mitigation:** If the two-level design is kept, track created cache names and evict them from `cacheManagerSlotsPerContract` using a custom eviction listener on the outer `contractSlotsCache`.

## Proof of Concept
```python
import requests, itertools, threading

BASE = "http://<mirror-node>/api/v1/contracts/call"
BODY = lambda addr: {
    "data": "0x",
    "to": addr,
    "estimate": False
}

def make_address(n):
    return "0x" + hex(n)[2:].zfill(40)

for i in itertools.count(1):
    addr = make_address(i)
    # Each unique address → unique contractId → new CaffeineCache in cacheManagerSlotsPerContract
    requests.post(BASE, json=BODY(addr))
    # After >3000 requests, outer contractSlotsCache starts evicting,
    # but cacheManagerSlotsPerContract.cacheMap keeps growing unboundedly.
```

After sending N requests with distinct addresses, `cacheManagerSlotsPerContract`'s internal map holds N `CaffeineCache` instances. Monitor JVM heap with `jmap -histo` to observe `com.github.benmanes.caffeine.cache.BoundedLocalCache` instance count growing linearly with N, never decreasing.
