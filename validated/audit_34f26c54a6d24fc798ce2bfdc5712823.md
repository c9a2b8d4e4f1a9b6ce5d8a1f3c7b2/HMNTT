# Audit Report

## Title
Unauthenticated GraphQL Request Storm via Missing Rate Limiting and Caching on `getByIdAndType()`

## Summary
The graphql module's `EntityServiceImpl.getByIdAndType()` issues an uncached, unthrottled `SELECT` to the database on every invocation. The grpc and web3 modules both apply `@Cacheable` to their equivalent repository lookups, and the web3 module additionally applies bucket4j rate limiting. The graphql module has neither. A public attacker can flood the GraphQL endpoint with concurrent low-complexity queries, exhausting the HikariCP connection pool and causing service-wide unavailability.

## Finding Description

**Verified code path:**

`AccountController.account()` → `EntityService.getByIdAndType()` → `EntityRepository.findById()` → DB

`EntityServiceImpl.getByIdAndType()` at lines 24–25 calls `entityRepository.findById()` with no caching annotation: [1](#0-0) 

The graphql `EntityRepository` extends `CrudRepository<Entity, Long>` and declares no `@Cacheable` override for `findById()`: [2](#0-1) 

By contrast, the grpc module's `EntityRepository` explicitly overrides `findById()` with `@Cacheable`: [3](#0-2) 

The web3 module's `EntityRepository` applies `@Caching`/`@Cacheable` across multiple cache managers: [4](#0-3) 

`GraphQlConfiguration` configures only query-complexity (200), query-depth (10), and JSON parsing caps — no per-IP rate limit, no global RPS throttle, no in-flight deduplication: [5](#0-4) 

A `CachedPreparsedDocumentProvider` exists in the graphql module and caches the parsed query AST, but this only avoids re-parsing the query string — it does **not** cache database results and does not prevent a DB hit per request. [5](#0-4) 

No `bucket4j`, `ThrottleManager`, `@RateLimit`, or `@Cacheable` annotation exists anywhere in the graphql module's Java sources (confirmed by exhaustive grep).

**Root cause:** The graphql module was never given the same defensive layers (caching + rate limiting) that the web3 and grpc modules received. The failed assumption is that query-complexity and query-depth limits alone are sufficient to protect the DB tier.

## Impact Explanation
An attacker sending N concurrent `account(input: { entityId: "0.0.1" })` queries (complexity = 1, depth = 1, well within limits) causes N simultaneous `SELECT` statements against the database. With no caching, identical queries are not deduplicated. With no rate limiting, there is no per-IP or global cap. The HikariCP connection pool is exhausted; all subsequent graphql queries fail with connection-timeout errors. If the graphql service shares a PostgreSQL instance with other mirror-node modules, those are also impacted.

Note: the "network partition" scenario described in the claim is an embellishment — the attack succeeds without any partition. Concurrent flooding alone is sufficient to exhaust the pool.

## Likelihood Explanation
The GraphQL endpoint is publicly reachable with no authentication required. [6](#0-5) 
The attack requires only an HTTP client capable of concurrent requests. The query is trivially scriptable and requires no credentials or special knowledge of the system.

## Recommendation
1. **Add `@Cacheable` to the graphql `EntityRepository`** — override `findById()` with a cache annotation backed by a short-TTL cache (matching the pattern in the grpc module).
2. **Add rate limiting to the graphql module** — wire in a bucket4j `ThrottleManager` (or equivalent) at the HTTP filter layer, mirroring the web3 module's `ThrottleManagerImpl`. [7](#0-6) 
3. **Consider connection-pool partitioning** — assign a dedicated HikariCP pool or schema to the graphql module so pool exhaustion cannot cascade to other modules.

## Proof of Concept
```bash
# Send 500 concurrent identical queries to the public GraphQL endpoint
seq 500 | xargs -P500 -I{} curl -s -X POST https://<mirror-node>/graphql/alpha \
  -H 'Content-Type: application/json' \
  -d '{"query":"{ account(input: { entityId: \"0.0.1\" }) { id } }"}'
```
Each request passes complexity (1 ≤ 200) and depth (1 ≤ 10) checks, reaches `EntityServiceImpl.getByIdAndType()`, and issues an uncached `SELECT` against the DB. With 500 concurrent requests and a default HikariCP pool of 10 connections, 490 threads queue waiting for a connection. DB connection pool is saturated; all other queries time out for the duration of the storm. [8](#0-7)
