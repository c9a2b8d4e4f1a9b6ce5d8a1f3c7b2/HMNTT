Audit Report

## Title
Integer Narrowing Overflow in `PollingTopicMessageRetriever.poll()` Bypasses DB Query Limit, Enabling Unbounded Query DoS

## Summary
A `uint64 limit` value of `Long.MAX_VALUE` sent via the unauthenticated gRPC `subscribeTopic` endpoint passes `@Min(0)` validation, then is narrowed to `int` via an unchecked cast in `poll()`, producing `-1`. This causes `hasLimit()` to return `false` on the internally-built filter, removing the `setMaxResults` guard in `findByFilter()` and issuing an unbounded `SELECT` against the `topic_message` table. The synchronous `getResultList()` call loads the entire result set into JVM heap before any reactive timeout can intervene.

## Finding Description

**Entry point — no upper-bound check on `limit`:**

`ConsensusController.toFilter()` maps the proto `uint64 limit` field directly to `TopicMessageFilter.limit` with no cap: [1](#0-0) 

The proto definition confirms `limit` is `uint64`, so any value `0 … 2^64-1` is wire-legal: [2](#0-1) 

**Validation passes for `Long.MAX_VALUE`:**

`TopicMessageFilter.limit` carries only `@Min(0)`. `Long.MAX_VALUE` satisfies `>= 0`, so the constraint is not violated: [3](#0-2) 

**Overflow in `poll()`:**

```
filter.getLimit()          = Long.MAX_VALUE  (0x7FFFFFFFFFFFFFFF)
context.getTotal().get()   = 0               (first poll)
difference                 = 0x7FFFFFFFFFFFFFFF
(int) 0x7FFFFFFFFFFFFFFF   = 0xFFFFFFFF = -1  ← negative
Math.min(-1, maxPageSize)  = -1
``` [4](#0-3) 

**Internal filter built with `limit = -1` bypasses `hasLimit()`:**

`filter.toBuilder().limit(-1).build()` is a raw Lombok builder call — Spring's `@Validated` AOP interceptor is never invoked on a plain value object, so `@Min(0)` is not re-enforced here: [5](#0-4) 

`hasLimit()` returns `limit > 0`; with `limit = -1` this is `false`: [6](#0-5) 

**`setMaxResults` is never called — unbounded query executes:** [7](#0-6) 

`getResultList()` is synchronous and loads the entire result set into JVM heap before any reactive timeout can cancel it: [8](#0-7) 

**Same pattern exists in `PollingTopicListener.poll()`:** [9](#0-8) 

**`isComplete()` does not save the situation:**

With `filter.getLimit() = Long.MAX_VALUE`, `limitHit` is `Long.MAX_VALUE == total.get()` — practically never true. For throttled mode the only other exit is `pageSize.get() < maxPageSize`, which only fires *after* the unbounded query already returns and after the next poll cycle: [10](#0-9) 

## Impact Explanation
A single gRPC `subscribeTopic` call with `limit = Long.MAX_VALUE` (9223372036854775807) causes the mirror node to issue a `SELECT * FROM topic_message WHERE topic_id = ?` with no `LIMIT` clause, loading potentially millions of rows into JVM heap via the synchronous `getResultList()` call. On a busy HCS topic this causes OOM or severe GC pressure, crashing or stalling the gRPC service for all subscribers. The `retrieverProperties.getTimeout()` wraps the reactive stream but cannot interrupt the synchronous `getResultList()` call mid-execution. Multiple concurrent attackers amplify the effect. The `isComplete()` guard also fails to terminate the polling loop, causing repeated unbounded queries.

## Likelihood Explanation
The gRPC endpoint is publicly accessible with no authentication required. The attacker needs only a valid `topicId` (trivially discoverable from the public ledger) and a single gRPC call with `limit` set to `9223372036854775807`. No special privileges, no rate-limit bypass, and no prior knowledge of internals is needed. The attack is trivially repeatable with `grpcurl` or any gRPC client library.

## Recommendation
1. **Cap the limit at the controller layer** in `ConsensusController.toFilter()` — clamp `query.getLimit()` to a safe maximum (e.g., `Integer.MAX_VALUE` or a configurable property) before setting it on the filter.
2. **Fix the narrowing cast** in both `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()` — use `Math.min(filter.getLimit() - total, (long) maxPageSize)` and keep the result as `long`, or guard with `if (difference > Integer.MAX_VALUE)` before casting.
3. **Add a `@Max` constraint** on `TopicMessageFilter.limit` to enforce an upper bound at the validation layer.
4. **Add a fallback `setMaxResults`** in `findByFilter()` using the configured `maxPageSize` when `hasLimit()` is false, as a defense-in-depth measure.

## Proof of Concept
```bash
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1234},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```
On a topic with a large message history, this triggers an unbounded `SELECT` in `TopicMessageRepositoryCustomImpl.findByFilter()` with no `LIMIT` clause, loading all rows into heap. Repeating this call concurrently from multiple clients will exhaust JVM heap and crash or severely degrade the gRPC service.
