Audit Report

## Title
Cache Key Hash Collision in `findByEvmAddressOrAliasAndDeletedIsFalse` and `findByEvmAddressAndDeletedIsFalse` Enables Cache Poisoning and Wrong Entity Return

## Summary
Both `findByEvmAddressOrAliasAndDeletedIsFalse` and `findByEvmAddressAndDeletedIsFalse` in `EntityRepository` use a single 32-bit `Arrays.hashCode(byte[])` integer as the sole Spring Cache key. Because the hash space is only 2^32 and collisions are analytically computable in O(1), an unprivileged attacker can craft an alias byte array that collides with a legitimate cached entry, either poisoning the cache slot with `Optional.empty()` (DoS) or receiving a different entity's object (wrong-entity return).

## Finding Description

**Exact code path:**

`SpelHelper.hashCode` at `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java` lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);  // 32-bit int
}
``` [1](#0-0) 

`findByEvmAddressOrAliasAndDeletedIsFalse` and `findByEvmAddressAndDeletedIsFalse` at `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java` lines 32–49:
```java
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",   // ← sole key = 32-bit int
        unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);

@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",   // ← sole key = 32-bit int
        unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
``` [2](#0-1) 

**Root cause:** `Arrays.hashCode(byte[])` is a polynomial hash `result = 31*result + b` over a 32-bit integer space. The entire cache key is this single integer. Two distinct byte arrays that produce the same integer are treated as the same cache entry. Spring Cache with Caffeine stores and retrieves by this key with no secondary equality check on the actual byte array.

**Failed assumption:** The code assumes `Arrays.hashCode()` is collision-free for the alias inputs it will receive. It is not — collisions are analytically computable in O(1).

**Collision example (verified):**
- `Arrays.hashCode([0, 31])` = 31×(31×1 + 0) + 31 = **992**
- `Arrays.hashCode([1, 0])` = 31×(31×1 + 1) + 0 = **992**

**Why `unless = "#result == null"` fails:** When no entity is found, Spring Data JPA returns `Optional.empty()` — a non-null Java object. The condition `#result == null` evaluates to `false`, so the empty Optional **is** cached under the colliding hash key. This is the mechanism that enables the DoS scenario. [5](#0-4) 

**Exploit flow — cache poisoning DoS:**
1. Attacker identifies target alias `X` of a real entity (aliases are public on-chain).
2. Attacker computes alias `Y` (no real entity) such that `Arrays.hashCode(Y) == Arrays.hashCode(X)`.
3. Attacker sends a request using alias `Y` before any legitimate request for `X` has populated the cache.
4. DB lookup for `Y` returns no entity → `Optional.empty()` is returned and cached under key `hashCode(Y) == hashCode(X)`.
5. All subsequent legitimate requests for alias `X` within the 1-second TTL window hit the cache and receive `Optional.empty()`.

**Exploit flow — wrong entity return:**
1. Entity A (alias `X`) is already cached under key `H = Arrays.hashCode(X)`.
2. Attacker sends a request with alias `Y` where `Arrays.hashCode(Y) == H`.
3. Cache hit → Entity A is returned for alias `Y`.

**Affected call path:** `CommonEntityAccessor.get(Bytes, Optional<Long>)` → `entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(alias.toByteArray())` for the non-historical (latest-block) path. [6](#0-5) 

## Impact Explanation
- **Cache poisoning DoS:** An attacker can make any entity unreachable via alias lookup for the duration of the cache TTL (default: 1 second, configurable) by pre-populating the cache slot with a colliding non-existent alias. This affects `eth_call`, balance queries, and any EVM execution path that resolves accounts by alias through `CommonEntityAccessor`. The attack must be repeated each TTL cycle to maintain the DoS, but this is trivially automatable.
- **Wrong entity returned:** A query with a crafted alias returns a different entity's full object, causing incorrect EVM execution results (wrong balance, wrong contract state) for callers relying on alias-based resolution.
- The 1-second default TTL (`expireAfterWrite=1s`) limits the persistence of each poisoned entry but does not eliminate the vulnerability, as the attack is repeatable and automatable. [4](#0-3) 

## Likelihood Explanation
- Hedera account aliases are public (visible in transaction records and the mirror node REST API), so an attacker can enumerate targets.
- Computing a colliding alias is O(1) arithmetic — no brute force needed.
- The attacker only needs to send a single HTTP request with the crafted alias to poison the cache slot within the TTL window.
- The attack must be repeated each 1-second TTL cycle to maintain the DoS, but this is trivially automatable with a simple loop.
- No special privileges, credentials, or network position are required.

## Recommendation
Replace the 32-bit hash key with the full byte array as the cache key. Since Caffeine/Spring Cache uses `equals()` and `hashCode()` on the key object, wrap the byte array in a type that implements proper equality:

```java
// Use a key that includes the full byte content, not just a 32-bit hash
key = "T(java.util.Arrays).toString(#alias)"
// or
key = "T(java.util.Base64).getEncoder().encodeToString(#alias)"
```

Alternatively, wrap the byte array in a `ByteArrayWrapper` class that implements `equals()` and `hashCode()` using `Arrays.equals()` and `Arrays.hashCode()` respectively — but use it as the full key object so Caffeine's internal map performs proper equality checks.

Additionally, fix the `unless` condition to also suppress caching of empty Optionals:
```java
unless = "#result == null || !#result.isPresent()"
```

## Proof of Concept
```java
// Collision: Arrays.hashCode([0, 31]) == Arrays.hashCode([1, 0]) == 992
byte[] aliasX = new byte[]{0, 31};  // real entity's alias
byte[] aliasY = new byte[]{1, 0};   // crafted alias, no entity

// Step 1: Send request with aliasY first (cache miss → DB returns empty → Optional.empty() cached under key 992)
entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(aliasY);
// → Optional.empty() now cached under key 992

// Step 2: Send request with aliasX (cache HIT → returns Optional.empty() instead of real entity)
Optional<Entity> result = entityRepository.findByEvmAddressOrAliasAndDeletedIsFalse(aliasX);
// → result == Optional.empty()  ← WRONG: real entity not returned
```
