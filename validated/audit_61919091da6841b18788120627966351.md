## Audit Report

## Title
Unauthenticated Historical Balance Path Triggers Expensive `DISTINCT ON` Query, Enabling DB Connection Pool Exhaustion

## Summary
Any unauthenticated caller can supply a `timestamp` query parameter to `GET /api/v1/accounts/:id`, which unconditionally activates a historical balance code path that executes a `DISTINCT ON (token_id) … ORDER BY token_id ASC, consensus_timestamp DESC` subquery against the `token_balance` table. Because the available indexes store `consensus_timestamp` ascending while the query demands it descending, PostgreSQL cannot satisfy `DISTINCT ON` via a pure index scan and must sort all matching rows before applying `LIMIT`. With no rate limiting or in-flight request cap on the REST API and a finite pgbouncer connection ceiling of 250 for `mirror_rest`, concurrent requests from a single attacker can exhaust available connections and deny service.

## Finding Description

**Step 1 — Unauthenticated timestamp parameter accepted**

`getOneAccount()` in `rest/accounts.js` accepts `timestamp` as a valid filter with no authentication:

```js
const acceptedSingleAccountParameters = new Set([
  constants.filterKeys.LIMIT,
  constants.filterKeys.ORDER,
  constants.filterKeys.TIMESTAMP,   // ← no auth gate
  ...
]);
``` [1](#0-0) 

**Step 2 — First DB query: `getAccountBalanceTimestampRange()`**

When `timestampFilters.length > 0`, `getAccountBalanceTimestampRange()` is called immediately, consuming one DB connection:

```js
if (timestampFilters.length > 0) {
  const {lower, upper} = await balances.getAccountBalanceTimestampRange(...);
``` [2](#0-1) 

`getAccountBalanceTimestampRange()` issues a `SELECT … FROM account_balance … ORDER BY consensus_timestamp DESC LIMIT 1` query: [3](#0-2) 

**Step 3 — `accountBalanceQuery.query` set to non-empty string when `upper !== undefined`**

If a valid balance snapshot exists, `accountBalanceQuery.query` is set to a non-empty subquery string: [4](#0-3) 

**Step 4 — `needHistoricalBalanceInfo` becomes `true`**

In `getEntityBalanceQuery()`, the non-empty `accountBalanceQuery.query` triggers the historical branch:

```js
const needHistoricalBalanceInfo = accountBalanceQuery.query || accountBalanceQuery.forceUnionEntityHistory;
``` [5](#0-4) 

**Step 5 — Expensive `DISTINCT ON` subquery constructed and executed**

The historical branch builds and executes:

```sql
select distinct on (token_id) token_id, balance
from token_balance
where account_id = $1
  and consensus_timestamp >= $lower
  and consensus_timestamp <= $upper
order by token_id ASC, consensus_timestamp desc
limit <tokenBalanceResponseLimit.singleAccount>   -- default 1000
``` [6](#0-5) 

**Step 6 — Index mismatch forces a full sort**

The `token_balance` primary key is `(account_id, token_id, consensus_timestamp)` — all ascending: [7](#0-6) 

The secondary index is `(token_id, account_id, consensus_timestamp)` — also all ascending: [8](#0-7) 

The query requires `ORDER BY token_id ASC, consensus_timestamp DESC`. Because `consensus_timestamp` is stored ascending in both indexes but the query demands it descending within each `token_id` group, PostgreSQL cannot satisfy `DISTINCT ON` using a pure index scan. It must fetch all rows for the account within the timestamp window and sort them in memory or on disk. The `LIMIT 1000` is applied only *after* the full sort completes — it does not bound the sort work.

**Step 7 — No rate limiting on the REST API**

The REST API Helm chart middleware contains only `circuitBreaker` and `retry` — no `rateLimit` or `inFlightReq`: [9](#0-8) 

Compare with the Rosetta chart, which explicitly configures both: [10](#0-9) 

**Step 8 — Connection pool ceiling**

The pgbouncer configuration caps `mirror_rest` at 250 server-side connections (`max_user_connections: 250`): [11](#0-10) 

The only server-side query guard is `statement_timeout` (default 20 s), which kills individual long queries but does not prevent many concurrent medium-duration queries from saturating the pool simultaneously: [12](#0-11) [13](#0-12) 

## Impact Explanation
An attacker who floods `GET /api/v1/accounts/<high-token-count-account>?timestamp=<valid-past-ts>` with concurrent requests will hold DB connections for the duration of each `DISTINCT ON` sort. Each request consumes at least two connections (one for `getAccountBalanceTimestampRange()`, one for the main entity query). Once the 250-connection pgbouncer ceiling for `mirror_rest` is saturated, all REST API queries queue or fail with connection timeout errors, causing a full denial of service for all users. Because `token_balance` is a historical accumulation table that grows unboundedly, the cost per query increases over time.

## Likelihood Explanation
The attack requires zero privileges — no API key, no account, no special knowledge beyond a valid account ID (publicly enumerable via `GET /api/v1/accounts`) and any past timestamp that falls within a balance snapshot window. The endpoint is publicly documented. A single attacker with a modest number of concurrent HTTP connections is sufficient to exhaust the pool. The attack is trivially repeatable and scriptable.

## Recommendation

1. **Add `inFlightReq` and `rateLimit` middleware to the REST API Helm chart**, mirroring the Rosetta chart configuration, to cap concurrent requests per source IP.
2. **Add a composite index on `token_balance (account_id, token_id, consensus_timestamp DESC)`** so PostgreSQL can satisfy `DISTINCT ON (token_id) ORDER BY token_id ASC, consensus_timestamp DESC` via an index scan without a full sort.
3. **Consider requiring authentication or an API key** for the historical balance code path (`timestamp` parameter on `/api/v1/accounts/:id`), or at minimum enforce a stricter per-IP rate limit on this specific endpoint.
4. **Reduce the default `tokenBalanceResponseLimit.singleAccount`** (currently 1000) or enforce a tighter cap for unauthenticated callers.

## Proof of Concept

```bash
# Enumerate a high-token-count account (publicly available)
ACCOUNT="0.0.1234"
# Find any valid past timestamp (any balance snapshot timestamp works)
TS="1680000000.000000000"

# Flood with concurrent requests — no authentication required
for i in $(seq 1 300); do
  curl -s "https://<mirror-node>/api/v1/accounts/${ACCOUNT}?timestamp=lte:${TS}" &
done
wait
# Result: DB connection pool for mirror_rest exhausted; all REST API endpoints
# return connection timeout errors for legitimate users.
```
