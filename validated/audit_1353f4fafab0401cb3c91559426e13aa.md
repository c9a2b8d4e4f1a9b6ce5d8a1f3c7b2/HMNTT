# Audit Report

## Title
`SharedPollingTopicListener` Timestamp Reset on Re-subscription Causes Permanent Message Loss After Subscriber Drain

## Summary
`SharedPollingTopicListener` uses a single shared `PollingContext` whose `onStart` callback unconditionally resets `lastConsensusTimestamp` to wall-clock `now()` every time the upstream Flux is re-subscribed. Because the upstream is wrapped with `.share()` (i.e., `publish().refCount()`), it is cancelled whenever the subscriber count drops to zero. Any subsequent subscriber causes a fresh upstream subscription, fires `doOnSubscribe → onStart → lastConsensusTimestamp = DomainUtils.now()`, and permanently skips every message whose `consensus_timestamp` falls between the last successful poll and the reconnection instant. No privilege is required.

## Finding Description
**Exact code path:**

`SharedPollingTopicListener` constructor: [1](#0-0) 

A single `PollingContext` is created at construction time and captured in the closure. The chain ends with `.share()`, which is `publish().refCount()`: the upstream is live only while at least one subscriber is present. When the last subscriber cancels, the upstream is torn down. The next subscriber triggers a brand-new upstream subscription, which fires `doOnSubscribe(context::onStart)`.

`PollingContext.onStart`: [2](#0-1) 

`onStart` unconditionally overwrites `lastConsensusTimestamp` with the current wall-clock time. There is no guard checking whether a previous value exists or whether messages may have been written since the last poll.

`poll()` uses that value as the exclusive lower bound: [3](#0-2) 

`findLatest` SQL: [7](#0-6) 

The strict `>` predicate means any row with `consensus_timestamp ≤ lastConsensusTimestamp` is permanently excluded from all future queries.

**Why `filterMessage` in `CompositeTopicListener` does not compensate:** [5](#0-4) 

This filter operates on messages already emitted by the shared Flux. Because `poll()` never fetches the partition-window rows from the database in the first place, the filter never sees them.

**Root cause:** The design assumes the upstream is never cancelled while messages of interest are being produced. `.share()` violates this assumption whenever all subscribers disconnect simultaneously. The single `PollingContext` is never re-created, so the reset permanently advances the polling cursor.

## Impact Explanation
Any deployment configured with `type: SHARED_POLL` permanently loses every topic message whose `consensus_timestamp` falls between the last successful poll before the subscriber drain and the reconnection instant. The loss is silent: the subscriber's sequence numbers will have gaps with no error signal, no retry, and no catch-up mechanism. For a consensus-message streaming service this constitutes undetectable data loss.

## Likelihood Explanation
Network partitions that drain all active gRPC streams simultaneously are routine operational events: rolling pod restarts, load-balancer timeout flushes, Kubernetes node evictions. No attacker capability is required for accidental triggering. An adversary who can cause even a brief network interruption — a targeted TCP RST injection or a short-lived connection flood against the gRPC port — can reliably trigger the reset on demand. The default listener type is `REDIS`, which limits exposure, but any deployment explicitly configured with `type: SHARED_POLL` is fully affected.

## Recommendation
Replace the unconditional reset in `onStart` with a conditional initialisation: set `lastConsensusTimestamp` to `DomainUtils.now()` only when the value has never been set (e.g., when it is `0` or a sentinel). On re-subscription after a subscriber drain, the existing cursor value should be preserved so that polling resumes from where it left off rather than from the reconnection instant. Alternatively, consider replacing `.share()` with `.publish().autoConnect(0)` (which never cancels the upstream regardless of subscriber count) if always-on polling is acceptable for the deployment.

## Proof of Concept
1. Configure the mirror node with `hedera.mirror.grpc.listener.type: SHARED_POLL`.
2. Connect one or more gRPC subscribers to a topic stream.
3. Disconnect all subscribers simultaneously (simulating a network partition or rolling restart).
4. While no subscribers are connected, allow topic messages to be written to the database (their `consensus_timestamp` values fall in the partition window).
5. Reconnect a subscriber.
6. Observe that `onStart` fires, `lastConsensusTimestamp` is set to the reconnection instant, and `findLatest` is called with `consensus_timestamp > <reconnection time>`.
7. Confirm that the messages written during step 4 are never delivered to the subscriber and are not present in any subsequent poll result — the gap is permanent.
