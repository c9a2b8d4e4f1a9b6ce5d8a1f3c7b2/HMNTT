Audit Report

## Title
Unbounded Retry-Amplified DB Query DoS via Corrupt Timestamp Targeting on GET /api/v1/network/fees

## Summary
`GET /api/v1/network/fees` in the `rest-java` module invokes `FileServiceImpl.getSystemFile()` twice per request — once for the fee schedule and once for the exchange rate — each wrapped in a `RetryTemplate` configured for up to `maxFileAttempts` (default: **12**) sequential DB calls. An unauthenticated attacker who supplies a timestamp pointing to a partially-written file version can force up to **24 correlated-subquery DB calls** per single HTTP request. No rate limiter is applied to this endpoint in the `rest-java` module.

## Finding Description

**Exact code path:**

`NetworkController.getFees()` calls `fileService.getFeeSchedule(bound)` and `fileService.getExchangeRate(bound)` sequentially with the same attacker-controlled `bound`: [1](#0-0) 

Each delegates to `FileServiceImpl.getSystemFile()`. The `RetryTemplate` is configured with `maxRetries = queryProperties.getMaxFileAttempts() - 1`, where `maxFileAttempts` defaults to **12**, yielding 12 total attempts per invocation: [2](#0-1) [3](#0-2) 

On each attempt, `fileDataRepository.getFileAtTimestamp(entityId.getId(), lowerBound, upperBound.get())` is called. If the returned bytes fail protobuf parsing, `upperBound` is decremented by 1 nanosecond and the exception is re-thrown, triggering the next retry: [4](#0-3) 

The underlying SQL is a correlated subquery with `string_agg` aggregation over the `file_data` table: [5](#0-4) 

**Root cause:** The retry loop is designed to recover from corrupt on-chain file versions, but there is no per-request cap on total DB work, no rate limiting on the `rest-java` network fees endpoint, and no authentication requirement. `getFees()` doubles the amplification by calling `getSystemFile()` for two different system files with the same timestamp bound.

**Why existing checks are insufficient:**

- `@Size(max = 2)` on the `timestamp` parameter only limits the number of timestamp query parameters, not the request rate or DB work per request.
- The `statementTimeout` applies per individual SQL statement, not to the aggregate work of 24 sequential statements.
- There is no `maxFileAttempts` guard that accounts for the double invocation in `getFees()`.
- The `ThrottleConfiguration`/`ThrottleManagerImpl`/bucket4j rate limiter exists only in the `web3` module and operates on `ContractCallRequest` objects — it is not wired into the `rest-java` module at all: [6](#0-5) [7](#0-6) 

## Impact Explanation
Each request to `GET /api/v1/network/fees` with a carefully chosen timestamp can consume up to 24× the DB resources of a normal single-attempt request. The `getFileAtTimestamp` query uses a correlated subquery and `string_agg` aggregation over the `file_data` table, which is CPU- and I/O-intensive. A modest number of concurrent attackers (e.g., 50 concurrent connections each firing requests at ~1 req/s) can sustain hundreds of complex DB queries per second, exhausting the DB connection pool and CPU without any brute-force volume.

## Likelihood Explanation
The precondition — a timestamp pointing to a partially-written file version — is realistic and observable. Hedera system file updates are multi-transaction operations (FILECREATE + multiple FILEAPPENDs) whose individual transaction timestamps are publicly visible via the mirror node's own transaction history API. An attacker can trivially identify such timestamps by querying `/api/v1/transactions?account.id=0.0.111&type=FILEAPPEND` and selecting a timestamp between the first and last append. No credentials, special access, or on-chain capability is required. The attack is repeatable and automatable.

## Recommendation

1. **Add a rate limiter to `rest-java`** for `GET /api/v1/network/fees` (and `/exchangerate`), analogous to the bucket4j throttle in the `web3` module.
2. **Reduce `maxFileAttempts`** or introduce a separate, lower cap specifically for the `getFees()` path that accounts for the double invocation (e.g., cap total DB calls per request at `maxFileAttempts`, not `2 × maxFileAttempts`).
3. **Cache the result** of `getSystemFile()` for a short TTL (e.g., 1 minute) keyed on the resolved timestamp, so repeated requests for the same timestamp do not re-execute the retry loop.

## Proof of Concept

1. Query the public mirror node to find a timestamp `T` between a FILEUPDATE and its subsequent FILEAPPEND for file `0.0.111`:
   ```
   GET /api/v1/transactions?account.id=0.0.111&type=FILEAPPEND&order=asc&limit=1
   ```
   Take the `consensus_timestamp` of the first FILEAPPEND result. The timestamp immediately before it (`T - 1ns`) points to an incomplete protobuf blob.

2. Send the crafted request (no credentials required):
   ```
   GET /api/v1/network/fees?timestamp=lte:<T>
   ```

3. The server executes up to 12 `getFileAtTimestamp` calls for the fee schedule file, each decrementing `upperBound` by 1 nanosecond. It then executes up to 12 more for the exchange rate file. Total: up to **24 correlated-subquery DB calls** per HTTP request.

4. Flood the endpoint with concurrent requests. No rate limiter in `rest-java` will intervene: [1](#0-0)
