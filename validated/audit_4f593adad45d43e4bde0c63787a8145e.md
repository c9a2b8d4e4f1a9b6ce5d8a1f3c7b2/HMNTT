# Audit Report

## Title
Connection Pool Exhaustion DoS via Unbounded Concurrent Requests in Rosetta API

## Summary
The Rosetta API combines an unvalidated `MaxOpenConnections` config field (Go zero-value `0` = unlimited pool) with a complete absence of application-level rate limiting and a Traefik middleware stack that is **disabled by default**. An unauthenticated attacker can exhaust the PostgreSQL connection pool or cause indefinite goroutine queuing by flooding any data-fetching endpoint with concurrent requests.

## Finding Description

**1. Unvalidated `MaxOpenConnections` in `Pool` struct**

The `Pool` struct carries `MaxOpenConnections int` with no validation tag, no minimum-value enforcement, and no default value: [1](#0-0) 

Go's zero-value for `int` is `0`. Any deployment that omits this field in its `application.yaml` silently enables an unbounded pool.

**2. Raw config value passed directly to `SetMaxOpenConns`**

`ConnectToDb()` passes the config value without any guard: [2](#0-1) 

Go's `database/sql` documents that `SetMaxOpenConns(0)` means **no limit** on open connections.

**3. No application-level rate limiting**

The entire middleware chain in `main.go` is: [3](#0-2) 

The `rosetta/app/middleware/` package contains only `metrics.go`, `trace.go`, and `health.go` — no rate-limiting or in-flight-request-limiting middleware exists in the application itself.

**4. Traefik middleware disabled by default**

The Helm template gates all Traefik middleware (including `inFlightReq: amount: 5` and `rateLimit: average: 10`) behind: [4](#0-3) 

The default value of `global.middleware` in the chart is explicitly `false`: [5](#0-4) 

This means the `inFlightReq` and `rateLimit` entries defined in the `middleware` block are **never applied** in a default deployment: [6](#0-5) 

## Impact Explanation
With `MaxOpenConnections=0` (the default when the field is omitted), every concurrent HTTP request can open a new DB connection. The PostgreSQL server will begin rejecting connections once its own `max_connections` limit is reached, rendering the Rosetta API completely unresponsive. Even with a finite pool size, Go's `database/sql` has no built-in wait timeout for acquiring a connection from an exhausted pool — goroutines queue indefinitely, causing the service to hang under sustained load. Because the Rosetta API and other mirror-node services share the same PostgreSQL instance, a successful attack can cascade to affect the entire mirror node.

## Likelihood Explanation
The Rosetta API is a public, unauthenticated HTTP endpoint. Sending hundreds of concurrent requests is trivially achievable with standard tools (`wrk`, `hey`, `ab`) from a single machine. The misconfiguration risk is elevated because Go's zero-value for `int` is `0`, meaning any deployment that does not explicitly set `maxOpenConnections` in its config silently enables an unlimited pool. The Traefik-based rate limiting that would otherwise mitigate this is disabled by default (`global.middleware: false`), making the unprotected state the out-of-the-box behavior.

## Recommendation

1. **Enforce a safe default and minimum for `MaxOpenConnections`**: Add config validation that rejects `0` or negative values and sets a documented safe default (e.g., `20`–`50`).
2. **Add a connection-wait timeout**: Call `sqlDb.SetConnMaxIdleTime` and consider wrapping pool acquisition with a context deadline so requests fail fast rather than queuing indefinitely.
3. **Enable Traefik middleware by default**: Change `global.middleware` default to `true` in `charts/hedera-mirror-rosetta/values.yaml`, or add application-level in-flight request limiting that does not depend on infrastructure configuration.
4. **Add application-level concurrency limiting**: Implement a middleware in `rosetta/app/middleware/` that caps in-flight requests independently of the deployment environment.

## Proof of Concept

```
# Assuming MaxOpenConnections is 0 (default when field is omitted)
# and global.middleware=false (chart default)

# Send 200 concurrent requests to a data-fetching endpoint
wrk -t 20 -c 200 -d 30s \
  -s post.lua \
  http://<rosetta-host>/block

# post.lua sends a valid Rosetta /block request body

# Expected result:
# - PostgreSQL hits max_connections, starts rejecting new connections
# - Rosetta API returns 500 errors or hangs indefinitely
# - Other mirror-node services sharing the DB become unavailable
```
