Audit Report

## Title
Missing Entity Existence Check for Numeric Account IDs in Token Allowance Lookup

## Summary
`GET /api/v1/accounts/{id}/allowances/tokens` returns HTTP 200 with an empty allowances list when the supplied account ID is a numeric/shard.realm.num value that does not exist in the database. The OpenAPI contract specifies HTTP 404 for this case. The EVM-address and alias paths for the same endpoint correctly return 404, making the behavior inconsistent across ID formats.

## Finding Description
In `rest/controllers/tokenAllowanceController.js` line 69, `getAccountTokenAllowances()` calls `EntityService.getEncodedId()` with the raw path parameter and uses the returned value directly as the owner filter for the allowance query, with no subsequent existence check. [7](#0-6) 

Inside `rest/service/entityService.js`, `getEncodedId()` branches on the input type. For a plain numeric or `shard.realm.num` ID (`entityId.evmAddress === null`), it returns `entityId.getEncodedId()` immediately — a pure arithmetic encoding with zero database interaction: [2](#0-1) 

The EVM-address branch calls `getEntityIdFromEvmAddress()`, which throws `NotFoundError` on a DB miss. The alias branch calls `getAccountIdFromAlias()`, which also throws `NotFoundError` on a DB miss. Only the numeric branch is exempt. The method's own JSDoc comment at line 110 explicitly acknowledges this asymmetry: *"Throws NotFoundError if the account is not present when retrieving by alias or the entity is not present and requireResult is true when retrieving by evm address"* — the numeric case is conspicuously absent. [8](#0-7) 

`isValidAccount()` exists on the same service and performs exactly the needed check (`SELECT type FROM entity WHERE id = $1`), but it is never called in this code path: [3](#0-2) 

The integration test named `"EntityService.getEncodedId - No match"` (line 161–163) loads **no entity** into the database yet asserts that `getEncodedId` resolves successfully, explicitly confirming the missing check is intentional in the current implementation: [4](#0-3) 

`TokenAllowanceService.getAccountTokenAllowances()` then executes `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 ...`, returns zero rows for the non-existent owner, and the controller writes `{ allowances: [], links: { next: null } }` with HTTP 200: [5](#0-4) 

The OpenAPI spec at lines 327–337 explicitly declares a `404` response for this endpoint, making this a documented contract violation: [6](#0-5) 

## Impact Explanation
Any off-chain system that interprets `200 + empty allowances list` as "account exists but has no allowances" will be silently misled when the account does not exist. Concrete consequences include:

- **Incorrect authorization decisions**: a relay or bridge concluding a spender has no allowance and skipping a required approval step.
- **Phantom account enumeration**: an attacker can probe arbitrary numeric IDs and always receive 200, making it impossible for callers to distinguish "account exists, zero allowances" from "account does not exist."
- **Inconsistency**: the EVM-address and alias paths for the identical endpoint correctly return 404, breaking any client that assumes uniform behavior across ID formats.

## Likelihood Explanation
- No authentication is required; any unauthenticated HTTP client can trigger this.
- Trivially reproducible: a single `GET /api/v1/accounts/999999999/allowances/tokens` reproduces the issue every time.
- No rate-limit or CAPTCHA barrier is present on mirror-node REST endpoints by default.
- No prior knowledge of the system beyond the public API spec is needed.

## Recommendation
In `getEncodedId()` (or in `getAccountTokenAllowances()` before the allowance query), add a DB existence check for the numeric-ID path, consistent with what the alias and EVM-address paths already do. The `isValidAccount()` method already implements the required query (`SELECT type FROM entity WHERE id = $1`) and should be called when `entityId.evmAddress === null` and `requireResult` is `true`. If the entity is not found, throw `NotFoundError` to produce the specified HTTP 404 response. [3](#0-2) 

## Proof of Concept
```
# Account ID 999999999 does not exist in the database.
# Expected per OpenAPI spec: HTTP 404
# Actual: HTTP 200 with empty allowances list

curl -s -o /dev/null -w "%{http_code}" \
  "http://<mirror-node-host>/api/v1/accounts/999999999/allowances/tokens"
# → 200

curl -s "http://<mirror-node-host>/api/v1/accounts/999999999/allowances/tokens"
# → {"allowances":[],"links":{"next":null}}

# Contrast with EVM address path (correctly returns 404):
curl -s -o /dev/null -w "%{http_code}" \
  "http://<mirror-node-host>/api/v1/accounts/0x000000000000000000000000000000003B9AC9FF/allowances/tokens"
# → 404
```
