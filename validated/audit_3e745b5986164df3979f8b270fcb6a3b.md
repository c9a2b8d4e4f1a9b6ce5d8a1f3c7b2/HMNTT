## Audit Report

## Title
Unbounded Historical Scan via `DISTINCT ON` in `GET /contracts/:contractId/state?timestamp=lte:<far_future>`

## Summary
The `getContractStateById` endpoint accepts a single `timestamp=lte:<value>` filter with no lower-bound requirement. When a timestamp is present, the service switches to `contractStateTimestampQuery`, which uses `DISTINCT ON (slot) ... ORDER BY slot ASC, consensus_timestamp DESC LIMIT N`. PostgreSQL must sort all matching rows across all time partitions before applying `LIMIT`, making the limit ineffective as a scan guard. Any unauthenticated caller can trigger a full historical scan of `contract_state_change` for any contract.

## Finding Description

**Code path:**

`rest/routes/contractRoute.js:15` → `ContractController.getContractStateById` → `extractContractStateByIdQuery` (lines 894–956, `rest/controllers/contractController.js`) → `ContractService.getContractStateByIdAndFilters` (lines 245–267, `rest/service/contractService.js`).

**Root cause 1 — No lower-bound enforcement on timestamp:**

In `extractContractStateByIdQuery`, the `TIMESTAMP` filter case (lines 909–918) directly appends the condition to `conditions` and sets `timestampPresent = true` with no lower-bound check. The `parseTimestampFilters` utility — which enforces that both a lower and upper bound exist (or an `eq` value) at `rest/utils.js:1651–1653` — is never called for this endpoint. A single `timestamp=lte:<far_future>` is accepted, generating `consensus_timestamp <= <far_future>` with no lower bound. [2](#0-1) [5](#0-4) 

**Root cause 2 — `DISTINCT ON` prevents LIMIT early-exit:**

When `timestamp=true`, `getContractStateByIdAndFilters` (lines 256–263) uses `contractStateTimestampQuery` with `ORDER BY slot ASC, consensus_timestamp DESC`. This is the standard PostgreSQL `DISTINCT ON (slot)` pattern. PostgreSQL evaluates `DISTINCT ON` before `LIMIT`, requiring it to read and sort every row satisfying `contract_id = X AND consensus_timestamp <= <far_future>` across all partitions before truncating to `LIMIT`. The `LIMIT 100` cap does not reduce rows read from disk. [3](#0-2) 

**Root cause 3 — `contract_state_change` is partitioned by `consensus_timestamp`:**

With `consensus_timestamp <= 99999999999999999999`, the query spans every time partition, forcing a full cross-partition scan. [4](#0-3) 

## Impact Explanation
A single request against a heavily-used contract forces the database to perform a full sequential scan across all `contract_state_change` partitions for that contract. Repeated or parallelized requests saturate DB I/O and CPU, degrading or denying service for all mirror-node consumers. No authentication is required.

## Likelihood Explanation
The exploit requires only a known contract ID with a large history (publicly discoverable via the same API). No credentials, tokens, or special network position are needed. The request is a standard HTTP GET, trivially repeatable and parallelizable. Active DeFi contracts on Hedera accumulate large numbers of historical state-change records, making the scan cost high.

## Recommendation
1. **Enforce a lower bound on timestamp**: In `extractContractStateByIdQuery`, call `parseTimestampFilters` (or equivalent logic) to require both a lower and upper bound when a timestamp filter is present, consistent with the range validation already implemented at `rest/utils.js:1651–1653`.
2. **Cap the timestamp range**: Apply a maximum allowed range (e.g., `maxTimestampRangeNs` from config) to bound the number of partitions scanned.
3. **Add rate limiting**: Apply per-IP or per-client rate limiting at the API gateway or Node.js middleware layer for this endpoint.

## Proof of Concept
```
GET /api/v1/contracts/0.0.1234/state?timestamp=lte:99999999999999999999&limit=100
```
No authentication required. For a contract with millions of historical state changes, this forces PostgreSQL to scan all `contract_state_change` partitions for `contract_id = 1234`, sort all results by `(slot, consensus_timestamp DESC)` to compute `DISTINCT ON (slot)`, and only then apply `LIMIT 100`. Issuing this request in parallel from multiple clients will saturate database resources.
