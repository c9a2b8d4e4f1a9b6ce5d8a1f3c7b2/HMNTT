# Audit Report

## Title
Unauthenticated DoS via Uncached Alias Lookups Causing Unbounded Database Queries in REST Service

## Summary
`getAccountFromAlias()` in `rest/service/entityService.js` issues a direct database query on every invocation with no result caching. The Redis response cache only stores HTTP 2xx responses, so 404 replies for non-existent aliases are never cached. The REST service middleware stack contains no rate-limiting middleware, allowing any unauthenticated attacker to flood the database with alias lookup queries at arbitrary request rates.

## Finding Description

**Code path — no-cache DB query:**

`getAccountFromAlias()` unconditionally executes `entityFromAliasQuery` against the `entity` table on every call with no in-process cache, no negative-result cache, and no deduplication. [1](#0-0) 

**Failed assumption 1 — `EntityId.parseCached` does not protect alias lookups:**

`parseCached` in `rest/entityId.js` uses a `quickLru` cache, but it only caches the *parsing* of entity ID strings (shard.realm.num, encoded IDs, EVM addresses). It never touches the database and provides zero protection for alias-to-entity DB lookups. [2](#0-1) 

**Failed assumption 2 — response cache does not cover 404 misses:**

`responseCacheUpdateHandler` in `rest/middleware/responseCacheHandler.js` only persists a response when `httpStatusCodes.isSuccess(res.statusCode)` is true. A 404 returned for a non-existent alias is never stored in Redis, so every repeated request for the same non-existent alias re-hits the database. [3](#0-2) 

**Failed assumption 3 — no rate limiting in REST middleware:**

The REST middleware stack exported from `rest/middleware/index.js` contains no rate-limiting middleware. [4](#0-3) 

The `ThrottleConfiguration` / `ThrottleManagerImpl` throttle infrastructure exists exclusively in the `web3` Java module and does not protect the REST service.

**Exploit flow:**

`GET /api/v1/accounts/{idOrAliasOrEvmAddress}` accepts aliases. When an alias is supplied, `getEncodedId()` detects it via `AccountAlias.isValid()` and calls `getAccountIdFromAlias()` → `getAccountFromAlias()` → `entityFromAliasQuery` DB call. [5](#0-4) 

Each request issues an index lookup against the `entity` table, returns 404, and is never cached. An attacker rotating unique valid-format but non-existent aliases guarantees every request reaches the database.

## Impact Explanation
Sustained alias flooding exhausts the database connection pool and saturates DB CPU/I/O. This degrades or completely blocks all other REST API operations (transactions, balances, tokens) for legitimate users. The attack requires no authentication and no on-chain state, making it a full availability-class vulnerability against the public mirror node REST API.

## Likelihood Explanation
The attack requires zero privileges, zero on-chain accounts, and only a standard HTTP client. Valid alias formats (base32-encoded public keys) are publicly documented in the Hedera API spec. An attacker can generate an infinite supply of unique valid-format aliases that will never match any entity, guaranteeing every request reaches the database. The attack is trivially scriptable with `curl` or any load-testing tool and is repeatable indefinitely.

## Recommendation
1. **Negative-result cache:** Cache alias-lookup misses in-process (e.g., a bounded LRU/TTL map keyed on the alias string) so repeated requests for the same non-existent alias do not hit the database.
2. **Response cache for 404s:** Extend `responseCacheUpdateHandler` to also cache 404 responses for alias lookups with a short TTL (e.g., 5–30 seconds).
3. **Rate limiting:** Add a rate-limiting middleware to the REST service middleware stack (e.g., `express-rate-limit` or an equivalent), scoped at minimum to the `/api/v1/accounts/:id` endpoint.
4. **Query timeout:** Enforce a short per-query timeout on `entityFromAliasQuery` to bound the DB resource cost of each individual request.

## Proof of Concept
```bash
# Generate unique non-existent base32 aliases and flood the endpoint
for i in $(seq 1 10000); do
  ALIAS=$(head -c 32 /dev/urandom | base32 | tr -d '=' | head -c 32)
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}" &
done
wait
```
Each request passes `AccountAlias.isValid()`, reaches `getAccountFromAlias()`, executes a full DB query, receives a 404, and is never cached. At sufficient concurrency this exhausts the DB connection pool and degrades all REST API endpoints for legitimate users.
