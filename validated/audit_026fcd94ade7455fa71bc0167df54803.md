## Audit Report

## Title
`BlockAssembler.timeout()` Returns Negative Value Passed to `grpcCall.read()`, Causing Silent Premature Stream Termination

## Summary
`BlockAssembler.timeout()` computes remaining time as `timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS)` with no lower-bound clamping. When elapsed time meets or exceeds the configured timeout — achievable by a malicious block node operator timing responses near the deadline — the method returns a negative `long`. This negative value is passed directly to `grpcCall.read(timeout, MILLISECONDS)`, which returns `null` immediately per `LinkedBlockingQueue.poll(long, TimeUnit)` semantics when given a non-positive timeout, causing the streaming `while` loop to exit silently as if the stream ended normally.

## Finding Description

**File:** `importer/src/main/java/org/hiero/mirror/importer/downloader/block/BlockNode.java`

`BlockAssembler.timeout()` at lines 274–281 computes remaining time with no floor guard:

```java
long timeout() {
    if (!stopwatch.isRunning()) {
        stopwatch.start();
        return timeout.toMillis();
    }
    return timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS);
}
``` [1](#0-0) 

The stopwatch starts on the **first** call to `timeout()` — which occurs before the first `grpcCall.read()` — and is only reset in `onEndOfBlock()` via `stopwatch.reset()` at line 268, which is never reached if the block is not fully assembled. [2](#0-1) 

The unclamped return value is passed directly to `grpcCall.read()` at line 138:

```java
while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
``` [3](#0-2) 

`BlockingClientCall.read(long timeout, TimeUnit unit)` delegates internally to `LinkedBlockingQueue.poll(timeout, unit)`. Java's `LinkedBlockingQueue.poll(long timeout, TimeUnit unit)` converts the argument to nanoseconds; if `nanos <= 0`, it returns `null` immediately without blocking. A negative value from `timeout()` therefore causes `grpcCall.read()` to return `null` immediately if no message is already buffered, which the `while` condition interprets as end-of-stream.

**Root cause:** No `Math.max(0, ...)` or minimum-value guard on the return path at line 280. The failed assumption is that `stopwatch.elapsed()` will never exceed `timeout.toMillis()` between successive calls to `timeout()`.

**Why existing checks fail:**

The `catch (BlockStreamException ex)` and `catch (Exception ex)` blocks at lines 159–164 are never reached because `grpcCall.read()` returning `null` is treated as a normal end-of-stream, not an error. [4](#0-3) 

`errors.set(0)` at line 157 is inside the loop body and is not executed on the iteration where `null` is returned, so the error counter is neither incremented (via `onError()`) nor reset. [5](#0-4) 

## Impact Explanation
A malicious block node can force the mirror node into a perpetual retry loop: each subscription attempt terminates silently after the first `BlockItemSet`, the block is never fully assembled, and `onEndOfBlock()` is never called. Because no exception is thrown and `onError()` is never invoked, the node's error counter is not incremented and the node is never marked inactive via the `maxSubscribeAttempts` / `readmitDelay` path. [6](#0-5) 

Block ingestion stalls for all blocks served by that node. If no other active node can provide the required block, `getNode()` throws `BlockStreamException("No block node can provide block …")`, halting block processing entirely. [7](#0-6) 

## Likelihood Explanation
Any block node operator — a role requiring no special privileges beyond running a node — can deliberately time their first `BlockItemSet` response to arrive near the timeout boundary. The attack requires no cryptographic material, no authentication bypass, and no insider access. It is fully repeatable: the attacker keeps the connection open and delays each response to just before the deadline. Under adverse but realistic network conditions (high latency, jitter), this can also be triggered unintentionally, making it hard to distinguish from a legitimate timeout.

## Recommendation
Clamp the return value of `timeout()` to a minimum of `0`:

```java
long timeout() {
    if (!stopwatch.isRunning()) {
        stopwatch.start();
        return timeout.toMillis();
    }
    return Math.max(0, timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS));
}
```

Additionally, treat a `null` return from `grpcCall.read()` as a timeout error when `pending` is non-empty (i.e., a block is partially assembled but not completed), so that `onError()` is invoked and the node's error counter advances correctly:

```java
if (response == null && !assembler.isPending()) {
    // normal end-of-stream
} else if (response == null) {
    throw new BlockStreamException("Stream ended with a partially assembled block (timeout)");
}
``` [1](#0-0) 

## Proof of Concept

**Setup:** Configure the mirror node with a block timeout of `T` ms (e.g., `T = 5000`). Operate a malicious block node that the mirror node subscribes to.

**Steps:**

1. Mirror node calls `streamBlocks(blockNumber, ...)`. The `while` loop evaluates `assembler.timeout()` for the first time: stopwatch starts, returns `T`.
2. `grpcCall.read(T, MILLISECONDS)` blocks for up to `T` ms.
3. Malicious server deliberately delays the first `BlockItemSet` until `T − δ` ms (e.g., `δ = 1 ms`), so the read returns successfully with a valid `BLOCK_ITEMS` response.
4. `assembler.onBlockItemSet()` is called; `pending` is now non-empty. `errors.set(0)` executes.
5. The loop iterates; `assembler.timeout()` is called again. Elapsed ≈ `T` ms → returns `T − T = 0` or, with any scheduling overhead, a **negative value**.
6. `grpcCall.read(≤0, MILLISECONDS)` calls `LinkedBlockingQueue.poll(≤0, MILLISECONDS)`. Since no second message is buffered yet, `poll` returns `null` immediately.
7. The `while` condition evaluates `null != null → false`; the loop exits. No exception is thrown. `onError()` is not called.
8. The `finally` block cancels the call. The block is never completed; `onEndOfBlock()` is never called.
9. The caller retries the same block number. The attack repeats indefinitely with zero error-counter advancement against the malicious node. [8](#0-7)
