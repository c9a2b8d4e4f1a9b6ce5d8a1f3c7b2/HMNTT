Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()` Causes Unbounded Database Query via `limit > Integer.MAX_VALUE`

## Summary
An unchecked narrowing cast from `long` to `int` in `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()` allows any gRPC client to supply a `limit` value of `2147483648L` (one above `Integer.MAX_VALUE`), causing the computed `pageSize` to overflow to `Integer.MIN_VALUE` (`-2147483648`). This negative value is passed to an internally-built `TopicMessageFilter` whose `hasLimit()` returns `false`, causing `TopicMessageRepositoryCustomImpl.findByFilter()` to execute an unbounded `getResultList()` call — loading every matching row into JVM heap — on every poll cycle.

## Finding Description

**Entry point** — `ConsensusController.toFilter()` reads `query.getLimit()` directly from the protobuf `uint64` field and stores it as a Java `long` with no upper-bound check:

```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

A value of `2147483648L` is positive, passes the `@Min(0)` constraint on `TopicMessageFilter.limit`, and is semantically valid on the wire. [2](#0-1) 

**Overflow site** — Inside `PollingTopicListener.poll()`:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())  // line 55
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());  // line 57
``` [3](#0-2) 

With `filter.getLimit() = 2147483648L` and `context.getCount().get() = 0L`:
- `(int) 2147483648L` → `-2147483648` (Java narrowing-cast overflow to `Integer.MIN_VALUE`)
- `Math.min(-2147483648, 5000)` → `-2147483648` (any positive `maxPageSize` loses)

The identical pattern exists in `PollingTopicMessageRetriever.poll()` (lines 68–71), which is always active for historical message retrieval regardless of listener type: [4](#0-3) 

**Validation bypass** — The negative `pageSize` is used to build an internal `newFilter` via `toBuilder()`:

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [5](#0-4) 

This builder call is internal and never passes through Spring's `@Validated` AOP proxy, so the `@Min(0)` Bean Validation constraint on `limit` is never evaluated. `newFilter.getLimit()` is `-2147483648L`.

**`hasLimit()` returns `false` for negative limit** — The guard in `TopicMessageFilter`:

```java
public boolean hasLimit() {
    return limit > 0;
}
``` [6](#0-5) 

`-2147483648 > 0` is `false`.

**Unbounded query** — In `TopicMessageRepositoryCustomImpl.findByFilter()`:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [7](#0-6) 

`setMaxResults` is never called. The query then executes:

```java
return typedQuery.getResultList().stream();
``` [8](#0-7) 

`getResultList()` materializes the entire result set into a `List` in JVM heap before returning.

**Why `flux.take()` does not mitigate this** — `TopicMessageServiceImpl.subscribeTopic()` applies `flux.take(filter.getLimit())` using the *original* filter:

```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
``` [9](#0-8) 

This limits downstream *emission* but does not prevent `getResultList()` from loading all rows into memory on each poll cycle before the Flux layer can cancel.

## Impact Explanation
Each poll cycle (default interval: 500 ms per `ListenerProperties`) executes a full table scan for the topic with no `LIMIT` clause. On a topic with millions of messages this causes:
- Heap exhaustion / OOM in the mirror node JVM (all rows loaded via `getResultList()`)
- Excessive database load (full sequential scan repeated every poll interval)
- Denial of service for all other subscribers sharing the same JVM and database connection pool

A single malicious subscription is sufficient to trigger this continuously until the connection is dropped. The `PollingTopicMessageRetriever` path is always active (for historical messages), making this exploitable regardless of the configured listener type. The `PollingTopicListener` path is additionally affected when the listener type is set to `POLL`. [10](#0-9) 

## Likelihood Explanation
The precondition is trivial: any unauthenticated gRPC client can send a `ConsensusTopicQuery` with `limit = 2147483648` (one above `Integer.MAX_VALUE`). No credentials, no special role, and no prior knowledge beyond the public proto schema is required. The proto field is `uint64`, so the value is semantically valid and passes all wire-level checks. The attack is repeatable and persistent for the lifetime of the subscription.

## Recommendation
Replace the unchecked narrowing cast with a safe bounded computation in both `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()`:

```java
// Before (unsafe):
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;

// After (safe):
long remaining = filter.hasLimit()
        ? filter.getLimit() - context.getCount().get()
        : (long) Integer.MAX_VALUE;
int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
```

Additionally, add an upper-bound validation on `TopicMessageFilter.limit` (e.g., `@Max(Long.MAX_VALUE)` is already satisfied, but a practical cap such as `@Max(Integer.MAX_VALUE)`) or validate that the computed `pageSize` is positive before building `newFilter`.

## Proof of Concept
```java
// gRPC client sends:
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
    .setLimit(2147483648L)  // Integer.MAX_VALUE + 1
    .build();

// In PollingTopicListener.poll():
// filter.getLimit() = 2147483648L, context.getCount().get() = 0L
// (int)(2147483648L - 0L) = (int)2147483648L = -2147483648  ← overflow
// Math.min(-2147483648, 5000) = -2147483648
// newFilter.getLimit() = -2147483648L
// newFilter.hasLimit() → (-2147483648 > 0) → false
// → setMaxResults() never called
// → getResultList() returns all rows for the topic, every 500ms
```
