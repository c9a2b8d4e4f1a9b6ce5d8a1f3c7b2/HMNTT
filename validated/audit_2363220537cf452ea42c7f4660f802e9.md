# Audit Report

## Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Unique ContractId Cycling — Heap-Exhaustion DoS

## Summary
In `ContractStateServiceImpl.findStorageBatch()`, each unique `contractId` that misses the outer `contractSlotsCache` triggers a call to `cacheManagerSlotsPerContract.getCache(contractId.toString())`. Because `cacheManagerSlotsPerContract` is a `CaffeineCacheManager` configured in dynamic mode (no `setCacheNames()` call), Spring's implementation inserts a new `CaffeineCache` into an internal, unbounded `ConcurrentHashMap<String, Cache> cacheMap` for every new name. The outer cache's eviction of a `contractId` entry does not remove the corresponding `CaffeineCache` from `cacheManagerSlotsPerContract.cacheMap`, causing permanent heap retention. An unauthenticated attacker cycling through unique contract addresses accumulates an unbounded number of live `CaffeineCache` objects in the JVM heap, leading to OOM or severe GC pressure.

## Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 86–87:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
```

**Root cause — two-layer cache design with a broken eviction assumption:**

**Layer 1 — outer cache (`contractSlotsCache`):**
Configured as `CACHE_MANAGER_CONTRACT_SLOTS` with `expireAfterAccess=5m,maximumSize=3000`. It maps `EntityId contractId → CaffeineCache`. This cache is bounded and will evict entries. [5](#0-4) 

**Layer 2 — inner cache manager (`cacheManagerSlotsPerContract`):**
Configured at `EvmConfiguration.java` lines 107–112 with only `setCacheSpecification()` and **no `setCacheNames()` call**. This places Spring's `CaffeineCacheManager` in dynamic mode. [1](#0-0) 

In dynamic mode, Spring's `CaffeineCacheManager.getCache(name)` creates a new `CaffeineCache` and inserts it into an internal `ConcurrentHashMap<String, Cache> cacheMap` for every previously-unseen name. **This map has no size bound and no eviction mechanism.**

**The broken assumption:**
When the outer `contractSlotsCache` evicts an entry for `contractId_X` (due to `maximumSize=3000` or TTL), the `CaffeineCache` object for `contractId_X` is no longer referenced by the outer cache. However, `cacheManagerSlotsPerContract.cacheMap` still holds a strong reference to it under the key `contractId_X.toString()`. The `CaffeineCache` is never GC'd.

The `slotsPerContract` spec `expireAfterAccess=5m,maximumSize=1500` applies only to entries *within* each per-contract `CaffeineCache` (slot keys), not to the number of `CaffeineCache` objects held in `cacheManagerSlotsPerContract.cacheMap`. [6](#0-5) 

**No contractId validation before cache creation:**
`findStorage()` does not validate that the `contractId` corresponds to an existing contract before calling `findStorageBatch()`. Any contractId — including non-existent ones — that misses the `contractStateCache` check proceeds directly to `findStorageBatch()` and triggers cache creation. [4](#0-3) 

## Impact Explanation
Each retained `CaffeineCache` object carries internal Caffeine data structures (segment arrays, scheduler references, etc.) — even an empty cache has non-trivial overhead. At the default throttle of 500 RPS, an attacker can register ~500 new caches per second. Over one hour that is ~1.8 million `CaffeineCache` objects permanently held in heap. This causes progressive heap exhaustion, triggering `OutOfMemoryError` or sustained full-GC pauses that render the web3 service unresponsive. No contract state data needs to exist; the attacker only needs to reference distinct contract addresses.

## Likelihood Explanation
- **No authentication required**: the JSON-RPC endpoint (`eth_call`, `eth_estimateGas`) is public.
- **No special knowledge required**: Hedera contract IDs are sequential integers; iterating them is trivial.
- **Rate limiting is insufficient**: the 500 RPS throttle slows but does not stop accumulation; the leak is permanent and cumulative until JVM restart.
- **Repeatable**: the attacker can sustain the attack indefinitely from a single machine.
- **No on-chain cost**: `eth_call` is a read-only simulation; no gas or HBAR is spent by the attacker.

## Recommendation
Fix the broken eviction assumption by removing the two-layer design. Instead of storing `CaffeineCache` objects as values in the outer cache and delegating to a dynamic `CaffeineCacheManager`, maintain a single bounded `Cache<EntityId, Cache<ByteBuffer, byte[]>>` using Caffeine's native support for nested/hierarchical caches, or use a `LoadingCache<EntityId, com.github.benmanes.caffeine.cache.Cache<ByteBuffer, byte[]>>` directly (bypassing Spring's `CaffeineCacheManager` entirely for the inner layer). This ensures that when the outer cache evicts an entry for a `contractId`, the inner cache object is also eligible for GC. Alternatively, register a Caffeine eviction listener on the outer cache that explicitly calls `cacheManagerSlotsPerContract` to remove the corresponding entry from its internal map when an outer entry is evicted.

## Proof of Concept
```
# Cycle through N unique contract IDs via eth_call
for i in $(seq 1 1000000); do
  curl -s -X POST http://<web3-host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"0x\",\"to\":\"0x$(printf '%040x' $i)\",\"gas\":21000}" &
done
```
Each request with a new `to` address (mapping to a unique Hedera contract ID) that misses the `contractStateCache` will call `cacheManagerSlotsPerContract.getCache(contractId.toString())`, inserting a new `CaffeineCache` into the unbounded internal map. After 3000+ unique IDs, the outer cache begins evicting entries while `cacheManagerSlotsPerContract.cacheMap` retains all previously-seen `CaffeineCache` objects. Memory grows linearly and without bound with the number of unique contract IDs seen.
