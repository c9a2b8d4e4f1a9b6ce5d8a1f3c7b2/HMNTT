Audit Report

## Title
Accept-Encoding Substring Check Bypass Allows Uncompressed Opcode Replay Responses

## Summary
`validateAcceptEncodingHeader()` in `OpcodesController` uses a naive `String.contains("gzip")` check. A header value of `gzip;q=0, deflate` satisfies the substring test but, per RFC 7231, explicitly opts out of gzip encoding. Tomcat honors `q=0` and returns the full uncompressed opcode trace, defeating the compression gate that exists specifically to limit bandwidth on this expensive endpoint.

## Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, line 76

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
        throw HttpClientErrorException.create(...);
    }
}
```

The check only tests for the substring `"gzip"` anywhere in the header value. RFC 7231 §5.3.4 defines `q=0` as "not acceptable." The value `gzip;q=0, deflate` contains the substring `"gzip"` and passes the guard, but signals to the HTTP stack that gzip is explicitly rejected. Spring's embedded Tomcat correctly honors `q=0` and will not apply gzip compression, returning the full uncompressed opcode trace payload.

The test suite at line 507 only covers `{"deflate", "identity", "br", ""}` as rejection cases — `gzip;q=0` and `gzip;q=0, deflate` are not tested and would not be rejected by the current implementation. [1](#0-0) [2](#0-1) 

## Impact Explanation
An attacker can bypass the compression enforcement gate and force the server to:
1. Perform the full, expensive EVM transaction replay (CPU/DB intensive).
2. Serialize and transmit the full uncompressed opcode trace (potentially megabytes of stack/memory/storage data per request).

The per-request bandwidth and serialization cost is significantly higher than intended. The throttle (`opcodeRequestsPerSecond = 1` by default) still applies, but each allowed request now consumes substantially more network resources than the compression gate was designed to permit. [5](#0-4) 

## Likelihood Explanation
Any unauthenticated external user can craft this header. No privileges, tokens, or special access are required. The bypass requires only knowledge of HTTP `q`-values (public RFC 7231 knowledge) and a single crafted `Accept-Encoding` header. It is trivially repeatable up to the rate limit.

## Recommendation
Replace the substring check with proper RFC 7231 quality-value parsing. Spring provides `MediaType`-style utilities; alternatively, parse the `Accept-Encoding` header to extract the effective quality for `gzip` and reject the request if `q=0` or if `gzip` is absent:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw ...;
    }
    boolean gzipAccepted = Arrays.stream(acceptEncodingHeader.split(","))
        .map(String::trim)
        .anyMatch(token -> {
            String[] parts = token.split(";");
            if (!parts[0].trim().equalsIgnoreCase("gzip")) return false;
            for (int i = 1; i < parts.length; i++) {
                String[] kv = parts[i].trim().split("=");
                if (kv.length == 2 && kv[0].trim().equalsIgnoreCase("q")) {
                    return Double.parseDouble(kv[1].trim()) > 0;
                }
            }
            return true; // no q param means q=1
        });
    if (!gzipAccepted) {
        throw ...;
    }
}
```

Also add `"gzip;q=0"` and `"gzip;q=0, deflate"` to the `@ValueSource` in `rejectsWhenAcceptEncodingIsNotGzip` to prevent regression. [2](#0-1) 

## Proof of Concept
```
GET /api/v1/contracts/results/{txHash}/opcodes?stack=true&memory=true&storage=true HTTP/1.1
Host: <mirror-node>
Accept-Encoding: gzip;q=0, deflate
```

1. The `contains("gzip")` check passes because the substring `"gzip"` is present.
2. Tomcat sees `q=0` and does not compress the response.
3. The server performs the full EVM replay and returns the complete uncompressed opcode trace.
4. The response is transmitted without gzip compression, consuming full bandwidth — the exact scenario the gate was designed to prevent. [1](#0-0)
