# Audit Report

## Title
Deleted Account Data Exposed via Long-Zero EVM Address Lookup Bypassing `deleted` Filter

## Summary
The `account(input: {evmAddress: ...})` GraphQL query contains a branching lookup in `EntityServiceImpl.getByEvmAddressAndType()` that, when the supplied address is in Hedera's "long-zero" format, falls through to `CrudRepository.findById()` — which carries no `deleted is not true` guard. Every other lookup path (`findByEvmAddress`, `findByAlias`) explicitly filters deleted entities at the SQL level. An unprivileged caller can therefore retrieve full account data (keys, memo, auto-renew settings, balance snapshot) for any deleted account by encoding its numeric ID as a long-zero EVM address.

## Finding Description
**Vulnerable code path:**

`AccountController.account()` → `EntityServiceImpl.getByEvmAddressAndType()` → long-zero branch → `entityRepository.findById()`

In `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`:

```java
// lines 37–38
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
}
``` [1](#0-0) 

`findById` is the standard Spring Data `CrudRepository` method; it issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate.

The two custom repository queries both carry the guard in `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`:

```java
// line 13
@Query(value = "select * from entity where alias = ?1 and deleted is not true", nativeQuery = true)
// line 16
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
``` [6](#0-5) 

The only post-fetch filter applied after `findById` is `e.getType() == type`, which checks entity type but not deletion status.

**Additional note:** `getByIdAndType()` (line 25) has the identical omission — it also calls bare `findById` — meaning the `entityId` input path is equally affected. [7](#0-6) 

## Impact Explanation
Any deleted account's full record — including its admin key, memo, auto-renew settings, and balance snapshot — is readable by any unauthenticated caller who knows the account's numeric ID. Because Hedera account numbers are sequential and publicly observable on-chain, enumeration is trivial. This constitutes unintended information disclosure of data that the protocol explicitly marks as deleted/inaccessible. Key material exposure could facilitate targeted attacks on accounts whose keys were reused after deletion.

## Likelihood Explanation
The GraphQL endpoint is publicly accessible with no authentication required. The long-zero EVM address format is documented Hedera behavior, well-known to any ecosystem developer. An attacker needs only to: know a deleted account's number, encode it as `0x` + 24 zero hex chars + 16-hex-char num, and issue a single POST request to `/graphql/alpha`. Fully repeatable with zero privilege.

## Recommendation
Add a `deleted` check to every `findById` call in the graphql `EntityServiceImpl`:

1. **`getByEvmAddressAndType` (line 38):** Replace `entityRepository.findById(buffer.getLong())` with a custom repository method that includes `AND deleted IS NOT TRUE`, consistent with `findByAlias` and `findByEvmAddress`.

2. **`getByIdAndType` (line 25):** Apply the same fix — add a `deleted is not true` predicate to the `findById` call used for the `entityId` input path.

The cleanest approach is to add a named query to `EntityRepository`:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

and replace both bare `findById` calls with `findByIdAndNotDeleted`.

## Proof of Concept
Given a deleted account with numeric ID `12345` (hex `0x0000000000003039`):

```
POST /graphql/alpha
Content-Type: application/json

{
  "query": "{ account(input: { evmAddress: \"0x000000000000000000000000000000000000000000003039\" }) { key memo deleted } }"
}
```

The long-zero detection in `EntityServiceImpl` reads the first 4 bytes (`int` = 0) and next 8 bytes (`long` = 0), then reads the final 8 bytes as the entity ID (`12345`). `findById(12345)` returns the row regardless of `deleted = true`. The response will contain the full account record including `key` and `memo` for the deleted account. [8](#0-7)
