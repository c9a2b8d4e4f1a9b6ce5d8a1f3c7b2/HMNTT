Audit Report

## Title
Offloaded-Calldata Ethereum Transactions Replayed with Empty Input, Producing Fabricated Opcode Traces

## Summary
`OpcodeServiceImpl.getEthereumDataBytes()` feeds the replay engine the stored raw RLP bytes that contain an **empty** calldata field for transactions whose calldata was offloaded to a Hedera file. The EVM re-executes the transaction with zero input, producing an opcode trace that is entirely different from the original on-chain execution. The endpoint is unauthenticated and accessible to any caller.

## Finding Description

**Confirmed DB state for offloaded-calldata transactions** — proven by the integration test:

```
ethereumTransaction.getCallData()   → EMPTY_BYTE_ARRAY
ethereumTransaction.getData()       → RAW_TX_TYPE_1_CALL_DATA_OFFLOADED (RLP with "80" = empty calldata)
ethereumTransaction.getCallDataId() → <file entity ID holding the real calldata>
``` [1](#0-0) 

**Root cause — `getEthereumDataBytes()` blindly returns the offloaded-variant RLP:**

```java
private byte[] getEthereumDataBytes(EthereumTransaction ethereumTransaction) {
    if (ethereumTransaction == null) { return new byte[0]; }
    final var data = ethereumTransaction.getData();
    return data != null ? data : new byte[0];
}
``` [2](#0-1) 

It never checks `callDataId` or reconstructs the full transaction. `getCallDataBytes()` returns `EMPTY_BYTE_ARRAY` for offloaded transactions and is never consulted in the ethereum-transaction branch anyway: [3](#0-2) 

**`buildCallServiceParameters()` assembles `ContractDebugParameters` with both broken values:** [4](#0-3) 

**`TransactionExecutionService.execute()` takes the ethereum-transaction branch because `ethereumData.length > 0`, and `callData` is never consulted:**

```java
if (params instanceof ContractDebugParameters debugParams
        && debugParams.getEthereumData() != null
        && debugParams.getEthereumData().length > 0) {
    transactionBody = buildEthereumTransactionBody(debugParams);
}
``` [5](#0-4) 

**`buildEthereumTransactionBody()` wraps the offloaded-variant RLP directly as `ethereumData`:** [6](#0-5) 

**Contrast with the correct pattern:** `AbstractEthereumTransactionParser.getHash()` explicitly detects the offloaded case (`callData` empty + `callDataId` non-empty), loads the real calldata from the file, re-encodes the full RLP, and then hashes it. The replay path performs no equivalent reconstruction: [7](#0-6) 

Similarly, `EthereumTransactionHandler.updateContractResult()` correctly loads calldata from the file when `callDataId` is set — the replay path is the only place this reconstruction is missing: [8](#0-7) 

**No authentication gate:** `OpcodesController.getContractOpcodes()` has no `@PreAuthorize` or any other access control. The only guards are a feature-flag check and a `gzip` header requirement, both trivially satisfied: [9](#0-8) 

## Impact Explanation
For every historical Ethereum transaction whose calldata was offloaded to a Hedera file, the opcode endpoint returns a trace that reflects execution of an **empty-input call** rather than the actual transaction. The fabricated trace will show a different sequence of opcodes, different SLOAD/SSTORE operations, different return data, and a different success/failure outcome. Auditors, investigators, and tooling that rely on opcode traces to reconstruct what a contract did will receive false evidence. Because the endpoint is unauthenticated, the attack surface is the entire public internet.

## Likelihood Explanation
Offloaded calldata is a documented, production-used Hedera feature evidenced by testnet data in the test suite. Any caller who can enumerate transaction hashes — trivially done via the public mirror-node REST API — can identify qualifying transactions and repeatedly trigger misleading traces. No special account, key, or privilege is required. The attack is fully repeatable and stateless.

## Recommendation
In `OpcodeServiceImpl`, mirror the reconstruction logic already present in `AbstractEthereumTransactionParser.getHash()` and `EthereumTransactionHandler.updateContractResult()`:

1. In `getCallDataBytes()`, check whether `ethereumTransaction.getCallData()` is empty **and** `ethereumTransaction.getCallDataId()` is non-empty. If so, load the real calldata from the file (via `FileDataRepository.getFileAtTimestamp` or `ContractBytecodeService.get`).
2. In `getEthereumDataBytes()`, after loading the real calldata, re-encode the full RLP (using the existing `encode()` method on the appropriate parser) so that `ethereumData` contains the complete transaction bytes with the real calldata field.
3. Alternatively, reconstruct the full `EthereumTransaction` object (with real calldata populated) before passing it to `buildCallServiceParameters()`, following the same pattern used in `AbstractEthereumTransactionParser.getHash()` lines 42–58.

## Proof of Concept

1. Identify any historical Ethereum transaction on the network where calldata was offloaded to a file (i.e., `callDataId` is set in the `ethereum_transaction` table and `call_data` is empty).
2. Obtain its transaction hash via `GET /api/v1/contracts/results/{hash}` (public, unauthenticated).
3. Call `GET /api/v1/contracts/results/{hash}/opcodes` with `Accept-Encoding: gzip`.
4. The returned opcode trace reflects execution of an empty-input call. Compare it against the actual on-chain execution (e.g., via a full node or Etherscan equivalent) — the opcode sequences, storage accesses, and return values will differ, confirming the fabricated trace.
