Audit Report

## Title
Unauthenticated Timestamp Boundary Bypass Causes Forced `EntityNotFoundException` on `/api/v1/network/supply`

## Summary
Supplying `timestamp=lt:0.000000001` to `GET /api/v1/network/supply` collapses both the lower and upper timestamp bounds to `0`, bypasses the existing range guard, and causes the database query to return no rows, triggering an `EntityNotFoundException` (HTTP 404) regardless of actual data availability.

## Finding Description

Each step of the exploit was verified against the actual source code:

**Step 1 — Input parsing**
`TimestampParameter.valueOf("lt:0.000000001")` matches the regex [1](#0-0) 
with seconds=`0`, nanos=`1`. `DomainUtils.convertToNanos(0, 1)` returns `1L`. [2](#0-1) 
Result: `TimestampParameter(LT, 1L)`.

**Step 2 — `Bound.of` passes `primarySortField=false`**
The controller calls `Bound.of(timestamp, TIMESTAMP, ...)`, which always constructs the `Bound` with `primarySortField=false`. [3](#0-2) 
This means the constructor's range guard (`if (primarySortField && adjustedLower > adjustedUpper)`) is **never evaluated** for timestamp bounds. [4](#0-3) 

**Step 3 — `adjustUpperBound()` returns `0`**
With `upper.operator() == LT` and `upper.value() == 1`, the method computes `upperBound = 1 - 1 = 0`. [5](#0-4) 

**Step 4 — `getAdjustedLowerRangeValue()` returns `0`**
No lower bound is provided (`lower == null`), so the method returns `0`. [6](#0-5) 

**Step 5 — Range check in `getSupply` is bypassed**
`minTimestamp=0`, `maxTimestamp=0`. The guard `if (minTimestamp > maxTimestamp)` evaluates `0 > 0 = false`; no exception is thrown. [7](#0-6) 

**Step 6 — `getFirstDayOfMonth(0, -1)` produces a large negative epoch**
`Instant.ofEpochSecond(0, 0)` is 1970-01-01. `plusMonths(-1)` yields 1969-12-01, whose `toEpochSecond()` is negative. Multiplied by `NANOS_PER_SECOND`, the result is a large negative `long`. `Math.max(0, negative) = 0`, so `minTimestamp` stays `0`. [8](#0-7) [9](#0-8) 

**Step 7 — Database query returns `consensusTimestamp = 0`**
`getSupplyHistory(..., 0, 0)` executes `WHERE ab.consensus_timestamp BETWEEN 0 AND 0`. No real balance record has `consensus_timestamp = 0`, so `coalesce(max(consensus_timestamp), 0)` returns `0`. [10](#0-9) 

**Step 8 — `EntityNotFoundException` is thrown**
The sentinel check `if (networkSupply.consensusTimestamp() == 0L)` fires, throwing `EntityNotFoundException("Network supply not found")`. [11](#0-10) 

**Root cause:** The range check `minTimestamp > maxTimestamp` is insufficient because it does not reject the degenerate case where both bounds collapse to `0`. The `Bound.of` factory always passes `primarySortField=false` for timestamp parameters, so the constructor-level guard is never applied.

## Impact Explanation
Any unauthenticated caller can force `GET /api/v1/network/supply` to return HTTP 404 for any request that includes `timestamp=lt:0.000000001`, regardless of whether valid supply data exists in the database. Clients or downstream systems that depend on this endpoint for circulating/total supply data will receive persistent errors when using this parameter. The impact is limited to availability of this specific endpoint with this specific parameter; no data is leaked or modified, and requests without this parameter are unaffected.

## Likelihood Explanation
The exploit requires zero privileges, zero authentication, and a single crafted HTTP GET request. The parameter format is publicly documented in the OpenAPI spec. [12](#0-11) 
The regex accepts `0.000000001` as a syntactically valid timestamp. The attack is trivially repeatable and scriptable.

## Recommendation
Add a minimum-value validation on the resolved `maxTimestamp` before proceeding with the historical query. For example, in `NetworkServiceImpl.getSupply`, after computing `maxTimestamp`, reject values that are at or below `0`:

```java
if (maxTimestamp <= 0) {
    throw new IllegalArgumentException("Invalid range provided for timestamp");
}
```

Alternatively, strengthen the range check to also reject the degenerate `[0, 0]` case:

```java
if (minTimestamp > maxTimestamp || maxTimestamp == 0) {
    throw new IllegalArgumentException("Invalid range provided for timestamp");
}
```

This would return HTTP 400 (Bad Request) instead of HTTP 404, which is semantically correct for an invalid input.

## Proof of Concept

```
GET /api/v1/network/supply?timestamp=lt:0.000000001 HTTP/1.1

```

Actual response: HTTP 404 Not Found — `{"_status":{"messages":[{"message":"Network supply not found"}]}}`
