Audit Report

## Title
Liveness Probe Decoupled from Transaction Submission — Indefinite Silent Failure Window

## Summary
The heartbeat goroutine in `pinger/main.go` writes `/tmp/alive` every 15 seconds regardless of whether transaction submission is succeeding. The Kubernetes liveness probe (`pinger/cmd/healthcheck/main.go`) only checks the age of that file. Persistent `submitWithRetry` failures are silently logged and never reflected in the health signal, so the pod is never restarted and gaps in Hashgraph history accumulate indefinitely.

## Finding Description
**Heartbeat goroutine** (`pinger/main.go` lines 28–39): runs independently of the transaction loop. It writes `/tmp/alive` unconditionally on every 15-second tick as long as the process is alive.

**Transaction loop** (`pinger/main.go` lines 57–70): on `submitWithRetry` failure, the error is only logged. There is no mechanism to stop the heartbeat, delete `/tmp/alive`, or otherwise degrade the health signal.

**Liveness check** (`pinger/cmd/healthcheck/main.go` lines 26–39): the sole criterion is `age > 2*time.Minute`. Because the heartbeat goroutine is alive whenever the process is alive (a goroutine panic in Go crashes the whole process; the failure mode here is the transaction loop returning errors, not panicking), the file stays fresh and the probe always exits 0.

**Root cause**: the failed assumption is that a live heartbeat implies live transaction submission. These are two independent code paths with no coupling.

## Impact Explanation
Every tick of `cfg.interval` (default 1 s) that fails to submit a transaction is a missing entry in Hashgraph history. Because the liveness probe never fires, Kubernetes never restarts the pod. The gap accumulates for as long as the underlying failure condition persists — hours or days — with no automated remediation. This directly undermines the integrity guarantee the pinger exists to provide: a continuous, verifiable record of activity on the ledger.

## Likelihood Explanation
No privileged access is required. Any condition that causes `submitWithRetry` to exhaust its retries — network partition to the Hashgraph node, operator account drained of hbar, SDK-level TLS failure, DNS resolution failure for the node endpoint — triggers the silent failure. The default `maxRetries=10` with exponential backoff means a sustained outage of roughly 5 minutes per tick exhausts retries; after that every tick fails and is only logged. These conditions can arise naturally or be induced by a network-adjacent attacker.

## Recommendation
Couple the liveness signal to transaction health. One approach: maintain an atomic timestamp of the last *successful* transaction submission and have the heartbeat goroutine write that timestamp (or refuse to write `/tmp/alive`) if the last success is older than a configurable threshold. Alternatively, on consecutive `submitWithRetry` failures exceeding a threshold, explicitly delete `/tmp/alive` or call `os.Exit(1)` to let Kubernetes restart the pod.

## Proof of Concept
1. Deploy the pinger pod normally; confirm `/tmp/alive` is written every 15 s and the liveness probe passes.
2. Simulate a sustained `submitWithRetry` failure (e.g., block egress to the Hashgraph node at the network level, or drain the operator account of hbar).
3. Observe that `pinger/main.go` logs `transfer failed: ...` on every tick but the process continues running.
4. Run `healthcheck live` (or observe the Kubernetes liveness probe result): it exits 0 because `/tmp/alive` was written within the last 2 minutes by the heartbeat goroutine.
5. Confirm the pod is never restarted and no transactions appear on the ledger for the duration of the failure.
