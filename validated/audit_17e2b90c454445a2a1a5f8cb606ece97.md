# Audit Report

## Title
Unbounded `DISTINCT ON` Historical Query in `getHookStorageChange` Enables DoS via Full Table Scan

## Summary
`HooksController.getHookStorage` accepts timestamp and key parameters with no range-width validation. When no filters are supplied, the resulting query against `hook_storage_change` uses `DISTINCT ON (key)` over an unbounded timestamp range, forcing PostgreSQL to sort and deduplicate every historical row for the targeted `(owner_id, hook_id)` pair before applying `LIMIT`. A single unauthenticated HTTP request can exhaust database resources and starve the connection pool.

## Finding Description

**Verified code path:**

`HooksController.getHookStorage` at lines 104–130 accepts `key` and `timestamp` parameters. When no `key` filter is provided, `lowerBound` defaults to `MIN_KEY_BYTES` (32×`0x00`) and `upperBound` defaults to `MAX_KEY_BYTES` (32×`0xFF`). [1](#0-0) [2](#0-1) 

`Bound.getAdjustedLowerRangeValue()` returns `0` when `lower == null`, and `adjustUpperBound()` returns `Long.MAX_VALUE` when `upper == null`. [3](#0-2) [4](#0-3) 

`HookServiceImpl.getHookStorageChange` (lines 84–118) passes these unbounded values directly to `HookStorageChangeRepository.findByKeyBetweenAndTimestampBetween`. [5](#0-4) 

The repository executes the following native query:

```sql
SELECT DISTINCT ON (key)
     owner_id, hook_id, key, value_written as "value", ...
FROM hook_storage_change
WHERE owner_id = :ownerId
  AND hook_id = :hookId
  AND key >= :keyLowerBound
  AND key <= :keyUpperBound
  AND consensus_timestamp BETWEEN :timestampLowerBound AND :timestampUpperBound
``` [6](#0-5) 

**Root cause:** PostgreSQL's `DISTINCT ON` requires the engine to sort and deduplicate **all rows** satisfying the `WHERE` predicate before `LIMIT` is applied. With `key` spanning the full 32-byte space and `consensus_timestamp BETWEEN 0 AND Long.MAX_VALUE`, every historical row for the targeted `(owner_id, hook_id)` pair is read, sorted, and deduplicated in a single query. The `LIMIT 100` (from `MAX_LIMIT = 100`) restricts only the output rows, not the work performed.

**Why existing checks fail:**

- `@Max(MAX_LIMIT)` on `limit` caps returned rows at 100 but does not limit the scan work. [7](#0-6) 

- `@Size(max = 2)` on `timestamps` allows at most two timestamp parameters but imposes no minimum range width and no maximum span. [8](#0-7) 

- `HIGH_VOLUME_THROTTLE` is defined in `Constants` but is only applied in `NetworkController`, never in `HooksController`. [9](#0-8) 

## Impact Explanation
A single crafted request forces a full sort and deduplication of all historical rows for a targeted `(owner_id, hook_id)` pair, holding a database connection open for an extended period. With a small number of concurrent requests, the database connection pool is exhausted, all other REST API queries queue and time out, and the mirror node becomes unavailable. Because the mirror node is the canonical read path for clients verifying transaction finality, this constitutes a denial of service for transaction confirmation.

## Likelihood Explanation
The endpoint is publicly accessible with no authentication requirement. The attacker needs only a valid `ownerId` (an account ID, publicly enumerable from the ledger) and any `hookId` (a small integer, trivially brute-forced or enumerated via the `getHooks` endpoint). The exploit requires a single HTTP GET request with no special privileges, tools, or credentials, and is trivially repeatable and scriptable.

## Recommendation

1. **Enforce a maximum timestamp range width.** Require that when a timestamp range is provided, `upper - lower` does not exceed a configurable maximum (e.g., 1 hour in nanoseconds). Reject requests where no upper timestamp bound is supplied, or default the upper bound to `now()` rather than `Long.MAX_VALUE`.

2. **Rewrite the query to avoid unbounded `DISTINCT ON`.** Replace the `DISTINCT ON` pattern with a lateral join or a recursive CTE that uses an index skip scan, so that `LIMIT` is applied per-key rather than after full deduplication. Alternatively, ensure a covering index on `(owner_id, hook_id, key, consensus_timestamp DESC)` exists so PostgreSQL can satisfy the query with an index-only scan.

3. **Apply `HIGH_VOLUME_THROTTLE` to `HooksController`.** The constant already exists in `Constants` and is used in `NetworkController`; apply the same rate-limiting annotation or filter to the `/hooks/{hookId}/storage` endpoint. [9](#0-8) 

4. **Set a database-level `statement_timeout`** for the REST API database role to bound the maximum duration of any single query.

## Proof of Concept

```
GET /api/v1/accounts/0.0.1234/hooks/1/storage
    ?timestamp=gte:0
    &limit=100
```

With no `timestamp` upper bound and no `key` filter, `timestampLowerBound = 0` and `timestampUpperBound = Long.MAX_VALUE`, `keyLowerBound = 0x00...00`, `keyUpperBound = 0xFF...FF`. PostgreSQL must sort and deduplicate every row in `hook_storage_change` for `(owner_id=1234, hook_id=1)` before returning 100 rows. Repeating this request concurrently (e.g., 10–20 parallel connections) exhausts the connection pool and renders the API unavailable. [6](#0-5) [10](#0-9)

    private HookStorageResult getHookStorageChange(HookStorageRequest request) {
        final var page = request.getPageRequest();
        final var ownerId = entityService.lookup(request.getOwnerId());
        final long hookId = request.getHookId();

        final var keys = request.getKeys();
        final boolean requestHasKeys = !keys.isEmpty();
        final var keysInRange = request.getKeysInRange();

        if (keysInRange.isEmpty() && requestHasKeys) {
            return new HookStorageResult(ownerId, List.of());
        }

        final var timestamp = request.getTimestamp();
        final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
        final long timestampUpperBound = timestamp.adjustUpperBound();

        List<HookStorage> changes;

        if (requestHasKeys) {
            changes = hookStorageChangeRepository.findByKeyInAndTimestampBetween(
                    ownerId.getId(), hookId, keys, timestampLowerBound, timestampUpperBound, page);
        } else {
            changes = hookStorageChangeRepository.findByKeyBetweenAndTimestampBetween(
                    ownerId.getId(),
                    hookId,
                    request.getKeyLowerBound(),
                    request.getKeyUpperBound(),
                    timestampLowerBound,
                    timestampUpperBound,
                    page);
        }

        return new HookStorageResult(ownerId, changes);
    }
