# Audit Report

## Title
Stale `CACHE_MANAGER_RECORD_FILE_INDEX` Cache Serves Pruned Block Metadata, Causing Silent Incorrect EVM Simulation Results

## Summary
`RecordFileRepository.findByIndex()` caches `Optional<RecordFile>` results for up to 10 minutes under the `CACHE_MANAGER_RECORD_FILE_INDEX` Caffeine cache. When the importer's `RetentionJob` deletes the corresponding `record_file` row, the web3 module's in-memory cache is never invalidated. Any `eth_call` or `eth_estimateGas` request for that block number within the TTL window receives the stale `RecordFile`, causing the EVM to execute against a pruned block context and silently return incorrect (empty/default) state instead of a proper `BlockNumberNotFoundException`.

## Finding Description

**Code path:**

1. A numeric block type in `RecordFileServiceImpl.findByBlockType()` falls through to `recordFileRepository.findByIndex(block.number())`. [7](#0-6) 

2. `findByIndex` is annotated `@Cacheable(cacheManager = CACHE_MANAGER_RECORD_FILE_INDEX, unless = "#result == null")`. The `unless` guard only prevents caching a Java `null`; it does not prevent caching `Optional.empty()`, and it does not evict an already-cached non-empty entry when the underlying row is later deleted. [2](#0-1) 

3. The cache manager is configured with a hard-coded 10-minute write-expiry TTL, a maximum of 10,000 entries, and no removal/eviction listener that could signal downstream logic. [3](#0-2) 

4. In `ContractCallService.callContract()`, the block supplier is set as a memoized lambda that calls `orElseThrow(BlockNumberNotFoundException::new)`. Because the cached `Optional` is non-empty, `orElseThrow` never fires — the stale `RecordFile` is silently accepted. [4](#0-3) 

5. `ContractCallContext.getTimestampOrDefaultFromRecordFile()` resolves the historical timestamp from the stale `RecordFile.consensusEnd`. All downstream state queries (contract storage, account balances, token state) use this timestamp. [8](#0-7) 

6. The importer's `RetentionJob` deletes rows via `prune(long consensusTimestamp)` with no cross-service cache-invalidation signal. The importer and web3 are separate JVM processes sharing only the database. [6](#0-5) 

**Root cause:** The Caffeine cache is purely in-memory and time-bounded. There is no eviction hook, no distributed invalidation, and no re-validation against the database within the TTL window. A `RecordFile` cached before pruning survives in memory for up to 10 minutes after the database row is deleted.

**Failed assumption:** The code assumes that a cached non-empty `Optional<RecordFile>` remains valid for the full TTL. This assumption breaks when retention is enabled and the importer deletes the row mid-TTL.

## Impact Explanation
When the stale `RecordFile` is served, `ContractCallContext.getTimestamp()` resolves to the pruned block's `consensusEnd`. All historical-state queries execute against that timestamp. Because the associated data rows have been pruned, those queries return empty/default values: zero balances, empty storage slots, absent tokens. The EVM simulation completes with HTTP 200 and fabricated zero-state results instead of a `404 BlockNumberNotFoundException`. Off-chain consumers (dApps, indexers, bridges) that rely on `eth_call` against historical blocks for correctness checks or balance proofs receive silently wrong data. No direct fund movement is possible from the mirror node itself, but incorrect simulation output can feed into off-chain decision logic.

## Likelihood Explanation
- Retention is disabled by default but is explicitly documented and recommended for production deployments on public networks to control storage costs.
- No credentials are required: a single unauthenticated HTTP POST to `/api/v1/contracts/call` with a numeric `block` field populates the cache.
- The 10-minute TTL is long relative to the retention job's batch cadence; the stale-cache window is reliably present.
- An attacker does not need to know when pruning occurs — repeated queries to the same block number after the expected pruning window will reveal stale-cache responses (no 404 returned).
- The cache holds up to 10,000 entries, allowing pre-warming of many block indices before pruning occurs.

## Recommendation
1. **Reduce the TTL** for `CACHE_MANAGER_RECORD_FILE_INDEX` to a value shorter than the retention job's minimum batch interval, or switch to `expireAfterAccess` with a short window.
2. **Add a Caffeine removal listener** that logs or alerts on eviction, and consider a `refreshAfterWrite` policy that re-validates against the database before serving a cached entry.
3. **Emit a cache-eviction event** from the importer's `RetentionJob` (e.g., via a shared Redis pub/sub channel or a database NOTIFY trigger) so the web3 module can proactively invalidate affected entries.
4. **Add a post-cache DB check** in `RecordFileServiceImpl.findByBlockType()` when retention is enabled: after receiving a cached result, verify the row still exists before returning it.
5. As a short-term mitigation, annotate `findByIndex` with `@CacheEvict` triggered by the retention boundary, or disable caching for this method entirely when retention is active.

## Proof of Concept
```
# Step 1: Pre-warm the cache for block N (before pruning)
POST /api/v1/contracts/call
{ "block": "0x<N>", "to": "0x...", "data": "0x..." }
# → 200 OK, correct result; RecordFile for block N is now cached.

# Step 2: Retention job runs and prunes record_file rows up to block N.
# (No cache invalidation occurs in the web3 process.)

# Step 3: Within 10 minutes of Step 1, re-query the same block:
POST /api/v1/contracts/call
{ "block": "0x<N>", "to": "0x...", "data": "0x..." }
# → 200 OK (expected: 404 BlockNumberNotFoundException)
# Result contains zero balances / empty storage — fabricated zero-state.

# Step 4: Confirm the block is truly pruned:
GET /api/v1/blocks/<N>
# → 404 Not Found (REST layer queries DB directly, no cache)
# The discrepancy between the REST 404 and the eth_call 200 confirms the stale cache path.
```
