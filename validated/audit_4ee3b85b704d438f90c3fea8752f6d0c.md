Audit Report

## Title
Rate-Based Throttle Without Concurrency Limit Enables DB Connection Pool Exhaustion via Precompile-Heavy EVM Replay

## Summary
When the opcode tracer is enabled, `OpcodesController.getContractOpcodes()` admits requests at a rate of 1/sec via a token-bucket throttle that tracks admission rate only. Because no concurrency limit exists, long-running EVM replays accumulate in-flight, each holding HikariCP connections for the duration of their historical DB queries. Sustained admission at the allowed rate can exhaust the shared connection pool and deny service to all other endpoints on the node.

## Finding Description

`OpcodesController.getContractOpcodes()` is a public, unauthenticated `GET` endpoint at `/api/v1/contracts/results/{transactionIdOrHash}/opcodes`. Its only admission controls are:
1. `properties.isEnabled()` — static flag, `false` by default.
2. `validateAcceptEncodingHeader()` — trivially satisfied with `Accept-Encoding: gzip`.
3. `throttleManager.throttleOpcodeRequest()` — the sole rate control. [7](#0-6) 

`ThrottleManagerImpl.throttleOpcodeRequest()` calls `opcodeRateLimitBucket.tryConsume(1)` and returns immediately. Once the token is consumed the request proceeds; there is no tracking of how many requests are currently executing. [2](#0-1) 

The bucket is configured with `capacity = opcodeRequestsPerSecond` (default **1**), refilling at 1 token/second, and is built **without** `SynchronizationStrategy.SYNCHRONIZED` (unlike `gasLimitBucket`), with no per-IP partitioning. [3](#0-2) [4](#0-3) 

After admission, `OpcodeServiceImpl.processOpcodeCall()` delegates to `ContractDebugService.processOpcodeCall()`, which sets the historical timestamp to `consensusTimestamp - 1` and calls `callContract()` — a full synchronous EVM re-execution. [6](#0-5) 

For transactions involving Hedera precompile token operations, each precompile call during replay issues multiple historical-timestamp DB queries (UNION queries across `*_history` tables, multi-CTE queries joining balance/transfer tables), each holding a HikariCP connection for the duration of the query. With T seconds per replay and admission at 1 req/sec, T requests are simultaneously in-flight after T seconds, each consuming one or more connections from the shared pool.

## Impact Explanation
When the opcode tracer is enabled, a single unauthenticated attacker sending requests at the admitted rate (1/sec) against precompile-heavy transactions can progressively fill the shared HikariCP connection pool. Once exhausted, every other endpoint on the same web3 node that requires a DB connection will time out or throw connection-pool-exhausted errors. Because the web3 module shares one DB pool per node instance, this effectively takes the node offline for all users. The attack requires no privileged access and is fully repeatable.

## Likelihood Explanation
The opcode tracer is disabled by default (`enabled = false`), which is the primary mitigating factor. However, operators who enable it for debugging or developer tooling expose the endpoint publicly with no authentication. The attack requires only a valid transaction ID or hash (publicly available on-chain), the `Accept-Encoding: gzip` header, and requests sent at the admitted rate. No special knowledge or privileged access is required.

## Recommendation
1. **Add a concurrency semaphore** in `ThrottleManagerImpl` (or `OpcodesController`) that limits the number of simultaneously executing opcode replays (e.g., `Semaphore(N)` with a non-blocking `tryAcquire()`), independent of the rate bucket.
2. **Isolate the DB connection pool** used by opcode replay from the pool used by all other endpoints, so pool exhaustion in the tracer path cannot affect other requests.
3. **Add per-IP rate limiting** to the opcode bucket to prevent a single client from consuming the entire global budget.
4. **Set an execution timeout** on `callContract()` within the debug path so that runaway replays release their connections promptly.

## Proof of Concept
```
# Requires: opcode tracer enabled, valid transaction hash with precompile calls
for i in $(seq 1 60); do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<node>/api/v1/contracts/results/<precompile_heavy_tx_hash>/opcodes" \
    --compressed &
  sleep 1
done
# After ~10-30 seconds, all other endpoints begin returning DB connection errors
```
Each iteration is admitted by the token bucket (1/sec). After T seconds (replay duration), T requests are simultaneously executing, each holding HikariCP connections. Once the pool is exhausted, all other DB-dependent endpoints on the node fail.

    @Bean(name = OPCODE_RATE_LIMIT_BUCKET)
    Bucket opcodeRateLimitBucket() {
        long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder().addLimit(limit).build();
    }
