Audit Report

## Title
HTTP 206 "Partial Mirror Node" Responses Incorrectly Cached and Served to All Subsequent Users

## Summary
`responseCacheUpdateHandler()` uses `httpStatusCodes.isSuccess(res.statusCode)` to gate caching decisions. Because `isSuccess` is defined as `code >= 200 && code < 300`, it returns `true` for status code 206 (`PARTIAL_CONTENT`). The project's 206 is a domain-specific code meaning the mirror node has incomplete records — not the standard HTTP range-request 206. Once a 206 response is cached, every subsequent request for the same URL is served the stale, partial payload until the TTL expires, even after the mirror node has fully caught up.

## Finding Description

**Exact code locations (all confirmed by direct file inspection):**

`rest/middleware/responseCacheHandler.js` line 95 — the caching gate: [1](#0-0) 

```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```

`rest/constants.js` line 159 — `isSuccess` definition: [2](#0-1) 

```js
isSuccess: (code) => code >= 200 && code < 300,
```

`rest/constants.js` line 155 — `PARTIAL_CONTENT` definition: [3](#0-2) 

```js
PARTIAL_CONTENT: new StatusCode(206, 'Partial mirror node'),
```

`rest/middleware/responseCacheHandler.js` lines 114–116 — 206 stored verbatim: [4](#0-3) 

```js
const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
```

`rest/middleware/responseCacheHandler.js` line 54 — 206 replayed from cache to every client: [5](#0-4) 

```js
const statusCode = clientCached ? httpStatusCodes.UNMODIFIED.code : cachedResponse.statusCode;
```

**Root cause:** The `isSuccess` predicate was designed to admit only fully successful responses into the cache. The range `[200, 300)` inadvertently includes 206. The 304 branch is correctly normalised to 200 before storage (line 114), but 206 receives no equivalent treatment and is stored as-is.

**Why existing checks fail:**
- The `isUnmodified` (304) branch is handled separately and correctly — 206 is not.
- The `ttl > 0` guard (line 97) only prevents caching when `max-age=0`; it does not filter on status code.
- `DEFAULT_REDIS_EXPIRY = 1` (line 24) means even a 206 response with no `Cache-Control` header is cached for 1 second; if the response carries a real `max-age`, it is cached for that full duration.
- There is no explicit exclusion of 206 anywhere in the caching pipeline. [6](#0-5) [7](#0-6) 

## Impact Explanation
Any paginated or time-sensitive endpoint (transactions, contract logs, token balances, etc.) that returns 206 during a mirror node lag window will have its incomplete payload frozen in Redis for the full `max-age` TTL. All users querying the same URL during that window receive missing records and a misleading 206 status, even after the mirror node has fully caught up. This causes persistent data integrity loss for consumers relying on the mirror node for complete ledger history and is not self-healing until TTL expiry.

## Likelihood Explanation
Mirror nodes routinely lag behind the consensus network during high-throughput periods, node restarts, or network upgrades — all conditions that naturally produce 206 responses. No authentication, special headers, or elevated privileges are required. Any anonymous HTTP client that is first to query a given URL during such a lag window triggers the cache poisoning. The condition is repeatable: an adversary can monitor for 206 responses and immediately re-query to confirm the cache has been poisoned, then wait for TTL expiry and repeat.

## Recommendation
Explicitly exclude 206 from the caching gate in `responseCacheUpdateHandler()`. The simplest fix is to change the condition to only cache responses with status code exactly 200 (or an explicit allowlist), rather than the broad `[200, 300)` range:

```js
// Before
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {

// After
if (responseBody && responseCacheKey && (isUnmodified || res.statusCode === httpStatusCodes.OK.code)) {
```

Alternatively, add an explicit exclusion:

```js
const isPartial = res.statusCode === httpStatusCodes.PARTIAL_CONTENT.code;
if (responseBody && responseCacheKey && !isPartial && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```

## Proof of Concept

1. Start the mirror node REST API with Redis caching enabled.
2. Arrange for the mirror node to be in a lag state (e.g., stop ingestion briefly).
3. Send `GET /api/v1/transactions?limit=25` — observe the response is `HTTP 206` with a partial body.
4. Immediately send the same request again — observe the response is served from Redis cache with `HTTP 206` and the same partial body.
5. Resume ingestion so the mirror node catches up fully.
6. Send the same request again before TTL expiry — observe the response is still `HTTP 206` with the stale partial body, even though the mirror node now has complete data.
7. Wait for TTL expiry and repeat step 3 to re-poison the cache.
