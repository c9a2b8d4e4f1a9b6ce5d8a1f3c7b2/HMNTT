# Audit Report

## Title
Missing `deleted` Filter in `EntityServiceImpl.getByIdAndType()` Allows Retrieval of Deleted Entities via Numeric EntityId on GraphQL Endpoint

## Summary
`EntityServiceImpl.getByIdAndType()` in the GraphQL module resolves entities using Spring Data JPA's inherited `CrudRepository.findById()`, which issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate. The two custom query methods in the same `EntityRepository` — `findByAlias` and `findByEvmAddress` — both explicitly exclude deleted entities. This creates an inconsistency: a deleted entity is invisible under alias or EVM address lookups but fully visible under numeric ID lookup via the public GraphQL endpoint.

## Finding Description
**Exact code path:**

`AccountController.account()` dispatches to `entityService.getByIdAndType(toEntityId(entityId), EntityType.ACCOUNT)` when the caller supplies an `entityId` input. [3](#0-2) 

`EntityServiceImpl.getByIdAndType()` resolves the entity with:
```java
return entityRepository.findById(entityId.getId()).filter(e -> e.getType() == type);
``` [1](#0-0) 

`findById()` is the inherited `CrudRepository` method — it issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` predicate.

The two custom query methods in the same repository both carry the guard:
```sql
select * from entity where alias = ?1 and deleted is not true
select * from entity where evm_address = ?1 and deleted is not true
``` [2](#0-1) 

**Root cause:** The developer added `deleted is not true` to the two custom query methods but relied on the inherited `findById()` for the ID path, omitting the same guard. No downstream check in `AccountController` or the mapper filters on `entity.getDeleted()` before returning the result. The only filter applied after `findById()` is a type check (`.filter(e -> e.getType() == type)`), which does not inspect the `deleted` flag.

**Secondary instance:** `getByEvmAddressAndType()` also falls through to `findById()` for long-zero EVM addresses (when the first 12 bytes are zero), carrying the same missing guard: [5](#0-4) 

**Test gap confirmed:** The test suite has no test case for `getByIdAndType` with a deleted entity: [4](#0-3) 

## Impact Explanation
The GraphQL mirror node API is a public, unauthenticated read endpoint. A deleted Hedera entity is supposed to be absent from active-entity queries; the design intent is explicit in the `findByAlias`/`findByEvmAddress` filters. Returning a deleted entity via the numeric ID path means:

- Downstream applications that query by numeric ID and check only for a non-null response (rather than inspecting `deleted: true`) will incorrectly treat the entity as live.
- The inconsistency between lookup methods creates a split-brain view: the same deleted entity is invisible by alias/EVM address but visible by ID, violating the mirror node's correctness guarantee of accurately reflecting consensus-node state.
- Severity: **Medium** — no private data is leaked (all fields are public blockchain data), but protocol-level data integrity is violated and dependent integrations can be misled.

## Likelihood Explanation
Exploitation requires zero privileges and zero special knowledge beyond knowing (or enumerating) a numeric entity ID, which is sequential and publicly observable on-chain. The GraphQL endpoint is publicly accessible. Any developer or automated tool querying `account(input: {entityId: {num: X}})` for a deleted account will trigger the behavior deterministically and repeatably.

## Recommendation
Add a custom query method to the graphql `EntityRepository` that includes the `deleted is not true` predicate for ID-based lookup, analogous to the existing `findByAlias` and `findByEvmAddress` methods:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(long id);
```

Then update `EntityServiceImpl.getByIdAndType()` (and the `findById` fallback in `getByEvmAddressAndType()`) to use this method instead of the inherited `CrudRepository.findById()`. Add corresponding test cases for deleted entities in `EntityServiceTest` and `AccountControllerTest`.

## Proof of Concept
1. Insert an account entity into the mirror node database with `deleted = true`.
2. Send the following GraphQL query to the public endpoint:
```graphql
query {
  account(input: { entityId: { num: <deleted_account_num> } }) {
    id
    deleted
    type
  }
}
```
3. Observe that the response returns the full entity record with `deleted: true`, confirming the entity is returned despite being deleted.
4. Send the same entity's alias or EVM address via `account(input: { alias: "..." })` or `account(input: { evmAddress: "..." })` — observe that the response returns `null`, confirming the inconsistency.
