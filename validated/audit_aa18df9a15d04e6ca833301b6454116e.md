Audit Report

## Title
Unauthenticated gRPC Subscriber Can Force Full Historical Topic Scan via Epoch-Zero consensusStartTime

## Summary
An unauthenticated external caller can send a `subscribeTopic` gRPC request with `consensusStartTime = {seconds:0, nanos:0}` and no `limit`, causing the server to execute a full per-topic historical scan on the `topic_message` table and stream every result indefinitely. No authentication, no rate limiting, and no enforced minimum start time exist on the gRPC endpoint.

## Finding Description

**Step 1 — No authentication or rate limiting on the gRPC endpoint.**

`ConsensusController.subscribeTopic()` has no authentication guard, no per-IP limit, and no connection-level throttle. [1](#0-0) 

The only registered `@GlobalServerInterceptor` sets an `EndpointContext` for table-usage tracking and immediately delegates to the next handler — it performs no auth or rate-limiting. [2](#0-1) 

Rate-limiting infrastructure (`ThrottleConfiguration`, `ThrottleManagerImpl`) lives exclusively in the `web3` module and is not wired into the `grpc` module. [3](#0-2) 

**Step 2 — `{seconds:0, nanos:0}` is treated as an explicit timestamp.**

In proto3, `Timestamp` is a message type. Setting `{seconds:0, nanos:0}` explicitly causes `hasConsensusStartTime()` to return `true`, so `toFilter()` calls `convertTimestamp()` and overwrites the safe default (`DomainUtils.now()`). [4](#0-3) 

**Step 3 — `convertTimestamp()` has no lower-bound guard.**

The method only guards against positive overflow (`seconds >= 9223372035L`). For `seconds=0, nanos=0` it falls through to `DomainUtils.timestampInNanosMax(timestamp)`, which returns `0`. [5](#0-4) 

**Step 4 — `TopicMessageFilter` validation permits `startTime = 0`.**

`@Min(0)` explicitly allows zero. `isValidStartTime()` only rejects future timestamps (`startTime > now()`); epoch 0 trivially satisfies `0 <= now()`. [6](#0-5) 

**Step 5 — The repository query is unbounded.**

`TopicMessageRepositoryCustomImpl.findByFilter()` builds `WHERE topic_id = ? AND consensus_timestamp >= 0`. `setMaxResults` is only called when `filter.hasLimit()` is true. [7](#0-6) 

**Step 6 — No flux-level cap without a limit.**

`TopicMessageServiceImpl` applies `flux.take(filter.getLimit())` only when `filter.hasLimit()` returns true. `hasLimit()` is `limit > 0`; the proto3 default for `uint64 limit` is `0`, so no cap is applied. [8](#0-7) 

**Step 7 — `PollingTopicMessageRetriever` pages through all historical messages.**

In throttled mode (used for historical retrieval), `isComplete()` returns `true` only when a page returns fewer rows than `maxPageSize` (default 1000). For a topic with millions of messages, the retriever issues repeated DB queries until the last page is smaller than the page size, streaming all results. [9](#0-8) 

The 60-second `timeout` is a per-emission idle timeout, not a total-duration cap; it does not fire while messages are being continuously emitted. [10](#0-9) 

**Step 8 — Own test suite confirms the behavior.**

Multiple integration tests explicitly use `startTime(0)` with no limit and expect all messages to be returned, confirming this is the actual runtime behavior. [11](#0-10) [12](#0-11) 

## Impact Explanation
For any high-volume HCS topic (e.g., a price-feed topic with millions of messages), a single unauthenticated request with `consensusStartTime={seconds:0, nanos:0}` and no `limit` forces the server to:
1. Execute repeated paginated queries against `topic_message` (page size up to 1000 rows each) until all historical rows for the topic are exhausted.
2. Stream all results over the gRPC connection.
3. Concurrently, the live listener path remains open indefinitely (no `endTime`, no `limit`).

Multiple concurrent connections (each allowed up to 5 concurrent calls via `maxConcurrentCallsPerConnection`) can saturate the database connection pool and exhaust JVM heap, constituting a practical application-level Denial-of-Service.

## Likelihood Explanation
The gRPC endpoint is publicly reachable with no authentication. The attack requires only a standard gRPC client (e.g., `grpcurl`) and knowledge of any active topic ID (topic IDs are public on the Hedera network). The exploit is trivially repeatable and scriptable. The own test suite demonstrates the behavior works exactly as described.

## Recommendation
1. **Enforce a minimum `startTime`**: In `convertTimestamp()` or `TopicMessageFilter.isValidStartTime()`, reject or clamp `startTime` values older than a configurable floor (e.g., 30 days before `now()`).
2. **Enforce a maximum result window**: Require either a `limit > 0` or a bounded `consensusEndTime` on every request; reject open-ended historical scans.
3. **Add gRPC-level rate limiting**: Implement a `ServerInterceptor` in the `grpc` module that applies per-IP or per-connection request rate limits, analogous to the `ThrottleManagerImpl` in the `web3` module.
4. **Cap concurrent subscriptions per client**: Enforce a global or per-IP cap on active `subscribeTopic` streams beyond the existing per-connection limit.

## Proof of Concept
```bash
# Requires grpcurl and knowledge of an active topic ID (e.g., 0.0.12345)
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 12345},
    "consensusStartTime": {"seconds": 0, "nanos": 0}
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

With `consensusStartTime={seconds:0, nanos:0}` and no `limit` field, the server will begin streaming every message ever posted to topic `0.0.12345` from the beginning of time. Repeating this across multiple connections exhausts DB and memory resources.
