Audit Report

## Title
NullPointerException in `evmAddressFromId` When `ContractResult.senderId` Is Null

## Summary
`OpcodeServiceImpl.getSenderAddress()` passes `contractResult.getSenderId()` directly to `CommonEntityAccessor.evmAddressFromId()` without a null check. Because `senderId` is a nullable JPA-mapped field on `ContractResult`, any stored row with a null `sender_id` column causes `entityId.getId()` inside `evmAddressFromId` to throw a `NullPointerException`, aborting the request with an unhandled 500 error.

## Finding Description

**`OpcodeServiceImpl.java` lines 177–180** — `getSenderAddress` passes the result of `getSenderId()` directly, with no null guard before the call: [1](#0-0) 

**`CommonEntityAccessor.java` lines 86–93** — `evmAddressFromId` has no `@NonNull` annotation on its `entityId` parameter and performs no null check. Because `timestamp` is `Optional.empty()`, the `.orElseGet()` lambda always executes, calling `entityId.getId()` on a null reference: [2](#0-1) 

**`ContractResult.java` line 71** — `senderId` is declared with no `@NonNull`, no default value, and no NOT NULL database constraint, making null a valid persisted value: [3](#0-2) 

The post-call guard `address != null ? address : EMPTY_ADDRESS` in `getSenderAddress` is never reached because the NPE fires inside `evmAddressFromId` before it returns. [4](#0-3) 

## Impact Explanation
Any HTTP GET to `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` that resolves to a `ContractResult` row with a null `sender_id` column receives an unhandled 500 Internal Server Error. Request processing is fully aborted for that call. There is no data leakage, no privilege escalation, and no economic damage — impact is limited to denial of the opcode-tracing service for all affected transactions.

## Likelihood Explanation
The `senderId` column is nullable in the domain model with no enforcement of a non-null value at the persistence layer. Historical or system-initiated contract executions may legitimately have no sender recorded. An unprivileged user requires only a single HTTP GET with a known transaction hash or ID that maps to such a row — no write access, no credentials, and no special privileges are needed to reproduce the crash.

## Recommendation
Add a null guard in `getSenderAddress` before calling `evmAddressFromId`:

```java
private Address getSenderAddress(ContractResult contractResult) {
    final var senderId = contractResult.getSenderId();
    if (senderId == null) {
        return EMPTY_ADDRESS;
    }
    final var address = commonEntityAccessor.evmAddressFromId(senderId, Optional.empty());
    return address != null ? address : EMPTY_ADDRESS;
}
```

Alternatively, annotate the `entityId` parameter of `evmAddressFromId` with `@NonNull` and add a null-return fast-path at the top of the method, so all callers are protected defensively.

## Proof of Concept
1. Identify (or wait for) a `contract_result` row in the database where `sender_id IS NULL`.
2. Obtain the `transaction_hash` or transaction ID for that row.
3. Issue:
   ```
   GET /api/v1/contracts/results/<hash>/opcodes
   ```
4. The call stack reaches `OpcodeServiceImpl.getSenderAddress` → `CommonEntityAccessor.evmAddressFromId(null, Optional.empty())` → `null.getId()` → `NullPointerException`.
5. The server returns HTTP 500. No authentication or special privileges required.
