Audit Report

## Title
`sync.Once` Misplacement in `initGenesisRecordFile` Causes Unbounded Repeated `selectGenesis` CTE Execution

## Summary
In `rosetta/app/persistence/block.go`, `initGenesisRecordFile()` contains two compounding bugs: (1) `sync.Once` guards only the *assignment* of genesis data, not the expensive DB query that precedes it, and (2) the error return path at line 254 exits without ever calling `once.Do`, leaving `genesisBlock.ConsensusStart` permanently at `genesisConsensusStartUnset`. The result is that every call to any of the five public block-lookup methods re-executes the full `selectGenesis` CTE for the entire duration of the startup window — and under concurrent load, N goroutines execute it simultaneously.

## Finding Description

**File:** `rosetta/app/persistence/block.go`, function `initGenesisRecordFile`, lines 243–263.

**Bug 1 — `sync.Once` placed after the query, not around it:**

```go
// line 244 — unsynchronized read, no mutex, no atomic
if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
    return nil
}
// lines 252–254 — expensive CTE runs here, BEFORE once.Do
if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
    return handleDatabaseError(err, hErrors.ErrNodeIsStarting)  // once.Do never reached
}
// lines 257–259 — once.Do guards only the assignment
br.once.Do(func() { br.genesisBlock = rb })
``` [1](#0-0) 

N goroutines that all observe `ConsensusStart == genesisConsensusStartUnset` at line 244 simultaneously will all dispatch `selectGenesis` to the database. `once.Do` fires for the first to finish; the remaining N-1 queries are already in flight and cannot be cancelled.

**Bug 2 — Error path skips `once.Do` entirely:**

When genesis data is absent (the normal "node is starting" state), `db.Raw(...).First(&rb)` returns `gorm.ErrRecordNotFound`. The function returns at line 254 via `handleDatabaseError`. `once.Do` is never invoked. `genesisBlock.ConsensusStart` remains `genesisConsensusStartUnset`. Every subsequent call to `FindByHash`, `FindByIndex`, `FindByIdentifier`, `RetrieveGenesis`, or `RetrieveLatest` re-enters the expensive path unconditionally — not just under concurrent load, but for every single request for the entire startup window. [2](#0-1) 

**The `selectGenesis` query:**

```sql
with genesis as (
  select consensus_timestamp from account_balance
  where account_id = @treasury_entity_id
  order by consensus_timestamp limit 1
)
select ... from record_file rf
join genesis on rf.consensus_end > genesis.timestamp
order by rf.consensus_end limit 1
``` [3](#0-2) 

This CTE scans `account_balance` (potentially millions of rows) and joins against `record_file` on every invocation.

**Middleware does not mitigate this.** The Traefik middleware chain (`inFlightReq: 5`, `rateLimit: average: 10`) is disabled by default (`global.middleware: false`). [4](#0-3) 

Even when enabled, `rateLimit` uses `sourceCriterion: requestHost: true` (keyed on the HTTP `Host` header, not client IP), so a single attacker host is limited to 10 req/s — still sufficient to sustain continuous expensive queries given typical CTE latency. [5](#0-4) 

## Impact Explanation

During the startup window (which can last minutes while the importer populates genesis data), every block-lookup request — regardless of concurrency — executes the full `selectGenesis` CTE against the database. Under any meaningful request rate, DB CPU spikes significantly, degrading or denying service to all legitimate users. Because the state is never set on the error path, the window remains open for the entire startup period. No credentials are required.

## Likelihood Explanation

The five affected endpoints (`/block`, `/block/transaction`, etc.) are standard, publicly documented Rosetta API paths requiring no authentication. Any client that can reach the Rosetta HTTP port during node startup can trigger this. The attack is trivially scriptable with standard HTTP load tools and repeatable on every node restart.

## Recommendation

Restructure `initGenesisRecordFile` so that `sync.Once` wraps the entire operation — both the DB query and the assignment — and so that a failed query does not permanently prevent future attempts. A correct pattern:

```go
func (br *blockRepository) initGenesisRecordFile(ctx context.Context) *rTypes.Error {
    // Fast path: atomic or mutex-protected read
    br.mu.RLock()
    initialized := br.genesisBlock.ConsensusStart != genesisConsensusStartUnset
    br.mu.RUnlock()
    if initialized {
        return nil
    }

    br.mu.Lock()
    defer br.mu.Unlock()
    // Double-checked locking
    if br.genesisBlock.ConsensusStart != genesisConsensusStartUnset {
        return nil
    }

    var rb recordBlock
    if err := db.Raw(selectGenesis, ...).First(&rb).Error; err != nil {
        return handleDatabaseError(err, hErrors.ErrNodeIsStarting)
        // Do NOT set genesis; allow retry on next request
    }
    br.genesisBlock = rb
    return nil
}
```

The existing `sync.Once` field should be removed; it is not fit for a retriable initialization pattern. A `sync.RWMutex` correctly serializes writers while allowing concurrent readers after initialization.

## Proof of Concept

1. Start the mirror node before genesis data is present in the database.
2. Send any HTTP POST to `/block` (or `/block/transaction`, `/network/status`, etc.) repeatedly:
   ```
   while true; do
     curl -s -X POST http://<rosetta-host>/block \
       -H 'Content-Type: application/json' \
       -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},"block_identifier":{"index":0}}' &
   done
   ```
3. Observe in the database that `selectGenesis` executes on every request. DB CPU rises proportionally to request rate.
4. After genesis data is populated, the first successful call sets `genesisBlock` via `once.Do` and subsequent calls take the fast path — confirming the vulnerability is confined to the startup window but is fully persistent within it.
