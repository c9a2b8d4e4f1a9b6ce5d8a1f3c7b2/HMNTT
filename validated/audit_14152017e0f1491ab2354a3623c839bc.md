# Audit Report

## Title
Safety-Check Path in `missingMessages()` Allows Unauthenticated Users to Force Expensive Historical Scans via `startTime=0`

## Summary
Any unauthenticated gRPC client can subscribe to a valid topic with `consensusStartTime = {seconds:0, nanos:0}`. After exactly one second, the unconditional safety-check timer fires and — because no messages have been received (`last == null`) — calls `topicMessageRetriever.retrieve()` with `startTime = 0` and `throttled = false`. This triggers the unthrottled polling retriever to issue up to 12 sequential queries, each fetching up to 5,000 rows, scanning all historical messages for that topic from the beginning of time.

## Finding Description

**Step 1 — Input accepted without lower-bound guard**

`ConsensusController.toFilter()` passes the client-supplied `consensusStartTime` directly to `TopicMessageFilter.startTime`. The only validation is `@Min(0)` and `startTime <= DomainUtils.now()`, both of which `0` trivially satisfies. [1](#0-0) [2](#0-1) 

**Step 2 — Safety check fires unconditionally after 1 s**

`subscribeTopic()` schedules a `Mono.delay(Duration.ofSeconds(1L))` that calls `missingMessages(topicContext, null)` for every subscription where `!topicContext.isComplete()`. Without an `endTime`, `isComplete()` always returns `false`. [3](#0-2) [4](#0-3) 

**Step 3 — `missingMessages(ctx, null)` uses original `startTime` when `last == null`**

Because no messages have been delivered, `topicContext.getLast()` is `null`, so the branch falls through to `topicContext.getFilter().getStartTime()`, which is `0`. [5](#0-4) 

**Step 4 — Unthrottled retriever issues up to 12 × 5,000-row queries**

`retrieve(gapFilter, false)` constructs a `PollingContext` with `numRepeats = maxPolls = 12` and `maxPageSize = 5000`. The unthrottled `isComplete()` only returns `true` when a limit is hit; without a limit it always returns `false`, so all 12 polls execute, each scanning `WHERE topic_id = X AND consensus_timestamp >= 0`. [6](#0-5) [7](#0-6) [8](#0-7) 

**Root cause:** The safety-check path was designed for the case where the importer is temporarily down and messages are delayed. It does not account for an attacker deliberately subscribing with `startTime = 0` to a quiet topic, making `last == null` a permanent condition and guaranteeing the expensive fallback query fires every time the safety check triggers.

**Why existing checks are insufficient:**
- `@Min(0)` explicitly permits `startTime = 0`.
- `isValidStartTime()` only checks `startTime <= now()`, which `0` trivially satisfies.
- There is no authentication, no per-client rate limit, and no cap on concurrent subscriptions.
- `isComplete()` only returns `true` when an `endTime` is set and has elapsed; without `endTime`, it always returns `false`.

## Impact Explanation
Each subscription with `startTime = 0` to a high-volume topic causes up to 60,000 rows to be read from the `topic_message` table (12 polls × 5,000 rows) within the first two seconds of connection. Because the query is `WHERE topic_id = X AND consensus_timestamp >= 0`, it performs a full index range scan over the entire history of that topic. With N concurrent connections, the database receives N × 12 queries in rapid succession, competing directly with the importer's write path and with other subscribers' read queries, degrading overall transaction-processing throughput.

## Likelihood Explanation
The attack requires no credentials, no special knowledge beyond a valid topic ID (which is public on-chain), and no sophisticated tooling — a standard gRPC client suffices. The trigger is deterministic (fires after exactly 1 second) and each new connection resets the timer. An attacker can open hundreds of connections from a single host. The attack is therefore highly feasible and easily automated.

## Recommendation
1. **Enforce a minimum `startTime` floor** — Reject or clamp `startTime` values that are unreasonably old (e.g., more than N days before `now()`). This prevents epoch-zero scans without breaking legitimate historical queries.
2. **Guard the safety-check path** — Before calling `retrieve(gapFilter, false)`, verify that `gapFilter.getStartTime()` is within a reasonable recent window (e.g., `>= DomainUtils.now() - maxHistoricalLookback`).
3. **Rate-limit safety-check retrievals per subscriber** — Track how many unthrottled retrievals a given subscriber has triggered and apply back-pressure or reject after a threshold.
4. **Add a connection/subscription rate limit** — Enforce a maximum number of concurrent subscriptions per source IP at the gRPC layer.

from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2
from google.protobuf.timestamp_pb2 import Timestamp

# Open N concurrent subscriptions with startTime=0 to a known quiet topic
TOPIC_NUM = 12345  # any valid topic ID
N = 100            # number of concurrent connections

channels = []
for _ in range(N):
    channel = grpc.insecure_channel("mirror-node-grpc:5600")
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        consensusStartTime=Timestamp(seconds=0, nanos=0),  # epoch zero
        # no endTime, no limit -> isComplete() always False
    )
    # Subscribe but don't read — keep connection open
    response_iter = stub.subscribeTopic(query)
    channels.append((channel, response_iter))

# After ~1 second, each subscription's safety check fires:
# missingMessages(ctx, null) -> retrieve(filter{startTime=0}, throttled=false)
# -> 12 polls x 5000 rows = 60,000 rows per connection
# -> N * 12 = 1,200 queries hit the database simultaneously
import time
time.sleep(5)
