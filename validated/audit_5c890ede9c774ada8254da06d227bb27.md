Audit Report

## Title
Null Dereference (NPE) in `ContractBytecodeReadableKVState` via Crafted Long-Zero EVM Address with Out-of-Range `num`

## Summary
`DomainUtils.fromEvmAddress()` silently returns `null` when the 8-byte `num` field of a long-zero EVM address exceeds `EntityId.NUM_MASK` (274,877,906,943), swallowing the resulting `InvalidEntityException` at DEBUG level. `ContractBytecodeReadableKVState.toEntityId()` propagates this `null` without a guard, and `readFromDataSource()` immediately dereferences it via `entityId.getId()`, producing an unhandled `NullPointerException` reachable by any unauthenticated caller of the web3 API.

## Finding Description

**Step 1 â€” Silent null production**

`EntityId.encode()` throws `InvalidEntityException` when `num > NUM_MASK || num < 0`: [1](#0-0) 

`NUM_MASK` is `(1L << 38) - 1 = 274,877,906,943`: [2](#0-1) 

`DomainUtils.fromEvmAddress()` catches that exception at DEBUG level and returns `null`: [3](#0-2) 

`isLongZeroAddress()` only checks that bytes 0â€“11 are zero; it places no constraint on bytes 12â€“19. Any 8-byte value > 274,877,906,943 or any negative value triggers the suppressed exception and the `null` return.

**Step 2 â€” Null propagated without guard**

`ContractBytecodeReadableKVState.toEntityId()` returns the result of `fromEvmAddress()` directly when `isLongZeroAddress()` is true, with no null check and no fallback to `EntityId.EMPTY`: [4](#0-3) 

**Step 3 â€” Null dereference**

`readFromDataSource()` calls `entityId.getId()` unconditionally on the value returned by `toEntityId()`: [5](#0-4) 

When `entityId` is `null`, this throws `NullPointerException`.

## Impact Explanation
The NPE is thrown inside the mirror node's web3 service during contract bytecode lookup (e.g., `eth_getCode`, `eth_call`). Depending on the framework's exception handling, this produces a 500 error for every such request or, if the exception escapes a shared handler thread, can destabilize the web3 service. The Hedera consensus network itself is unaffected (the mirror node is read-only), but the web3 API endpoint becomes unreliable or unavailable for any contract address matching the crafted pattern â€” a targeted, repeatable denial-of-service.

## Likelihood Explanation
The precondition is trivially met: the attacker needs only the ability to send an HTTP request to the mirror node's public web3 endpoint â€” no authentication, no privileged account, no on-chain transaction required. The crafted address (12 zero bytes + 8 bytes encoding a value > 274,877,906,943, e.g., `0x000000000000000000000000ffffffffffffffff`) is a single static input. The attack is deterministic and repeatable at zero cost.

## Recommendation
Two complementary fixes should be applied:

1. **In `DomainUtils.fromEvmAddress()`**: Instead of returning `null` on `InvalidEntityException`, either re-throw the exception or return `EntityId.EMPTY`. Returning `null` from a utility method that callers routinely dereference is the root design flaw.

2. **In `ContractBytecodeReadableKVState.toEntityId()`**: Add a null guard after calling `fromEvmAddress()`:
   ```java
   if (isLongZeroAddress(evmAddress)) {
       final var entityId = DomainUtils.fromEvmAddress(evmAddress);
       return entityId != null ? entityId : EntityId.EMPTY;
   }
   ``` [6](#0-5) 

## Proof of Concept
Send the following JSON-RPC request to the mirror node's web3 endpoint:

```json
{
  "jsonrpc": "2.0",
  "method": "eth_getCode",
  "params": ["0x000000000000000000000000ffffffffffffffff", "latest"],
  "id": 1
}
```

The address `0x000000000000000000000000ffffffffffffffff` satisfies `isLongZeroAddress()` (bytes 0â€“11 are zero) and encodes `num = 0xffffffffffffffff = 18446744073709551615`, which far exceeds `NUM_MASK`. This causes `EntityId.encode()` to throw `InvalidEntityException`, which is swallowed in `fromEvmAddress()`, returning `null`. `toEntityId()` returns that `null`, and `readFromDataSource()` throws `NullPointerException` on `entityId.getId()`.
