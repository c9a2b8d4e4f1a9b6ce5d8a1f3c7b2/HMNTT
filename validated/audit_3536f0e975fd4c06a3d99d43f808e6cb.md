# Audit Report

## Title
Unauthenticated 1-Nanosecond Timestamp Range Triggers Guaranteed-Empty DB Query in `getBalances()`, Enabling Connection Pool Exhaustion

## Summary
Any unauthenticated caller can supply `timestamp=gt:<X>&timestamp=lt:<X+2>` to `/api/v1/balances`. After `getOptimizedTimestampRange()` converts the operators, the effective range collapses to `[X+1, X+1]`. This passes the only guard (`lowerBound > upperBound`), causing `getAccountBalanceTimestampRange()` to unconditionally fire a PostgreSQL query that virtually never matches a real balance-snapshot timestamp. At scale, repeated requests exhaust the connection pool and degrade or deny service.

## Finding Description

**Call chain:** `getBalances()` → `getTsQuery()` → `getAccountBalanceTimestampRange()` → `getOptimizedTimestampRange()`

**Step 1 — Validation only checks individual values.**
`parseTimestampQueryParam` at [1](#0-0)  calls `parseTimestampParam` per value (format check only) and never invokes `parseTimestampFilters`. `filterValidityChecks` for `TIMESTAMP` at [2](#0-1)  only calls `isValidTimestampParam(val)` — a regex format check with no cross-value range validation.

**Step 2 — `getOptimizedTimestampRange` collapses the range to a single nanosecond.**
For `timestamp=gt:<X>&timestamp=lt:<X+2>`:
- `lt:<X+2>` branch at [3](#0-2) : `upperBound = (X+2) - 1n = X+1`
- `gt:<X>` branch at [4](#0-3) : `lowerBound = X + 1n`

The only guard is:
```js
if (lowerBound > upperBound) {   // line 260
  return {};
}
```
`X+1 > X+1` is **false**, so execution continues. [5](#0-4) 

The lower-bound optimization at lines 270–275 sets `lowerBound = max(X+1, firstDayOfPreviousMonth(min(X+1, now)))`. Since `X+1` falls within its own month and `optimalLowerBound` is the first day of the *previous* month, `X+1 > optimalLowerBound`, so `lowerBound` remains `X+1`. The function returns `{lowerBound: X+1, upperBound: X+1, neParams: []}`. [6](#0-5) 

**Step 3 — `getAccountBalanceTimestampRange` fires the DB query.**
Because `lowerBound !== undefined`, the function builds and executes: [7](#0-6) 
```sql
SELECT consensus_timestamp
FROM account_balance
WHERE account_id = $1
  AND consensus_timestamp >= $2   -- X+1
  AND consensus_timestamp <= $3   -- X+1
ORDER BY consensus_timestamp DESC
LIMIT 1
```
Balance snapshots are taken at most every few minutes; the probability of one landing on an arbitrary single nanosecond is effectively zero. The query returns 0 rows, `getAccountBalanceTimestampRange` returns `{}`, `getTsQuery` returns `{}`, and `getBalances` exits early at lines 115–117 — but only **after** the DB round-trip has already consumed a connection-pool slot. [8](#0-7) 

**Why existing checks fail:**
- `isEmptyRange` (utils.js:901–946) uses strict `upper < lower`; equal bounds return `false`. Moreover, it is never called in the balances validation path. [9](#0-8) 
- `parseTimestampFilters` (utils.js:1583–1681), which has the `difference <= 0n` guard at line 1661, is used by `contractController.js`, `accounts.js`, and `transactions.js` — but **not** by `balances.js`. [10](#0-9) 
- No application-level rate limiting exists in the REST API JavaScript layer (confirmed: zero matches for `rateLimit` in `rest/**/*.js`).

## Impact Explanation
Each crafted request causes one synchronous PostgreSQL query to execute and hold a connection from the pool (`max: config.db.pool.maxConnections`, `connectionTimeoutMillis` configured in `dbpool.js`). [11](#0-10)  With enough concurrent requests the pool is saturated; subsequent legitimate requests queue or time out. Because the query is a fast indexed PK lookup, the attacker needs high concurrency rather than long-running queries, but this is trivially achievable with async HTTP clients. The result is degraded or complete denial of the `/api/v1/balances` endpoint for all users.

## Likelihood Explanation
The endpoint is public and requires no credentials. The payload is a standard HTTP GET with two query parameters. Any HTTP client capable of sending concurrent requests (`curl`, `ab`, `wrk`, Python `asyncio`) can execute this. The attacker needs no knowledge of internal state; any value of `X` works. The attack is repeatable indefinitely and stateless, making it trivially scriptable.

## Recommendation
1. **Add a cross-value range check in the balances validation path.** Before calling `getAccountBalanceTimestampRange`, check that the effective range (after operator normalization) has a positive width — i.e., `upperBound > lowerBound` (strict). A range of zero nanoseconds should be rejected with a `400 Bad Request`.
2. **Reuse `parseTimestampFilters`** (which already enforces `difference > 0n` at utils.js:1661) in the balances endpoint, replacing the older `parseTimestampQueryParam` path, to unify timestamp range validation across all endpoints.
3. **Apply infrastructure-level rate limiting** (e.g., via an API gateway or reverse proxy) to all public REST endpoints to bound the maximum request rate per IP.

## Proof of Concept
```bash
# Pick any timestamp X (e.g., current time in nanoseconds)
X=1700000000000000000

# Craft the 1-nanosecond range: gt:X and lt:(X+2)
URL="http://<host>/api/v1/balances?timestamp=gt:${X}&timestamp=lt:$((X+2))"

# Fire 200 concurrent requests to exhaust the connection pool
seq 200 | xargs -P200 -I{} curl -s "$URL" -o /dev/null
```
Each request causes one guaranteed-empty DB query. With `maxConnections` concurrent requests in flight, the pool is saturated and legitimate requests begin timing out with `connectionTimeoutMillis` errors.
