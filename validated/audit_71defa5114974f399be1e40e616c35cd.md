Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses Query Limit, Enabling Unbounded Database Reads

## Summary
In `PollingTopicListener.poll()`, the remaining limit is computed with a narrowing `(int)` cast: `(int)(filter.getLimit() - context.getCount().get())`. A user-supplied `limit` of `2147483648L` (Integer.MAX_VALUE + 1) causes this cast to produce `Integer.MIN_VALUE`. Because `TopicMessageFilter.hasLimit()` returns `false` for any value ≤ 0, `TopicMessageRepositoryCustomImpl.findByFilter()` never calls `setMaxResults()`, executing a fully unbounded SQL query on every poll cycle for the lifetime of the subscription.

## Finding Description

**Step 1 — Overflow in `PollingTopicListener.poll()`** [1](#0-0) 

With `filter.getLimit() = 2147483648L` and `context.getCount().get() = 0`:
- `(int)(2147483648L - 0L)` → `Integer.MIN_VALUE` (`-2147483648`)
- `Math.min(Integer.MIN_VALUE, 5000)` → `Integer.MIN_VALUE`
- `newFilter` is built with `limit = -2147483648L` (int widened to long)

**Step 2 — `hasLimit()` silently drops the limit** [2](#0-1) 

`-2147483648L > 0` is `false`, so `hasLimit()` returns `false` for the derived filter.

**Step 3 — `setMaxResults()` is never called** [3](#0-2) 

Because `hasLimit()` is false, `typedQuery.setMaxResults()` is skipped. The JPA query executes with no SQL `LIMIT`, and `getResultList()` materializes the entire matching result set into JVM heap before streaming. [4](#0-3) 

**Step 4 — Service-level guard uses the original (unadjusted) limit** [5](#0-4) 

`filter.getLimit()` here is the original `2147483648L`. `hasLimit()` returns `true` (positive), so `flux.take(2147483648L)` is applied — effectively no bound in practice.

**Why existing checks fail:**

- `@Min(0)` on `TopicMessageFilter.limit` permits any positive `long`, including values above `Integer.MAX_VALUE`. [6](#0-5) 

- No `@Max` constraint exists to cap the value at `Integer.MAX_VALUE`.
- The internal `filter.toBuilder().limit(pageSize).build()` call in `poll()` does not pass through Spring's `@Valid` AOP interception, so the `@Min(0)` annotation is never re-evaluated on the derived filter carrying the negative value.
- `hasLimit()` returning `false` for the overflowed negative value causes `findByFilter` to silently omit the limit rather than reject or clamp it.

## Impact Explanation

Every poll cycle (default interval: 500 ms) issues a full table scan against `topic_message` for the targeted topic with no row limit:

1. **DoS via database exhaustion** — repeated unbounded scans consume database I/O and CPU, degrading service for all users.
2. **Full topic history disclosure** — all `TopicMessage` rows matching the topic and start time are streamed to the attacker, up to ~2 billion messages per subscription per the service-level `take()`.
3. **JVM heap exhaustion** — `getResultList()` materializes the entire result set in memory before the reactive stream processes it, risking OOM on large topics.

The combination of DoS and information disclosure places this outside the "impacts that only require DDoS" exclusion in SECURITY.md.

## Likelihood Explanation

- Any gRPC client can call `subscribeTopic` with an arbitrary `limit` value; no authentication is required beyond a valid gRPC connection.
- The trigger value (`2147483648L = Integer.MAX_VALUE + 1`) is trivially derivable from standard Java integer overflow knowledge.
- The attack is repeatable: each new subscription with the crafted limit re-triggers the unbounded query on every poll interval for the subscription's lifetime.
- Requires `listenerProperties.type = POLL`; the default is `REDIS`. [7](#0-6) 

`POLL` and `SHARED_POLL` are first-class documented deployment options, and `PollingTopicListener` is a `@Named` Spring component.

## Recommendation

1. **Add `@Max(Integer.MAX_VALUE)` to `TopicMessageFilter.limit`** to reject any value that would overflow on cast.
2. **Replace the silent `(int)` cast** in `PollingTopicListener.poll()` with `Math.toIntExact()` (throws `ArithmeticException` on overflow) or an explicit clamp: `(int) Math.min(remaining, Integer.MAX_VALUE)`.
3. **Add a defensive guard in `findByFilter`** to treat any negative limit as "no limit requested" with a log warning, rather than silently omitting `setMaxResults()`.

## Proof of Concept

```java
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setLimit(2147483648L)   // 0x80000000 — one above Integer.MAX_VALUE
    .build();

// Subscribe via any standard HCS gRPC client.
// With listenerProperties.type=POLL, every 500 ms poll will execute:
//   SELECT * FROM topic_message
//   WHERE topic_id = 1 AND consensus_timestamp >= <startTime>
//   ORDER BY consensus_timestamp ASC
// with NO LIMIT clause, materializing the full result set into heap.
```

Execution trace:
- `filter.getLimit()` = `2147483648L` → `hasLimit()` = `true`
- `(int)(2147483648L - 0L)` = `-2147483648` (Integer.MIN_VALUE)
- `Math.min(-2147483648, 5000)` = `-2147483648`
- `newFilter.limit` = `-2147483648L` → `hasLimit()` = `false`
- `setMaxResults()` skipped → unbounded SQL query executed every 500 ms
