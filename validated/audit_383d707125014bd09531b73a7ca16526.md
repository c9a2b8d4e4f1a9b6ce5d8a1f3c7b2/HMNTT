Audit Report

## Title
Missing `lower(timestamp_range) <= blockTimestamp` Guard in Entity Branch of `findActiveByEvmAddressAndTimestamp` Returns Wrong Historical State

## Summary
The `entity` branch of the `UNION ALL` in `findActiveByEvmAddressAndTimestamp` and `findActiveByEvmAddressOrAliasAndTimestamp` omits the `lower(timestamp_range) <= ?2` predicate that is present in the `entity_history` branch and in the analogous `findActiveByIdAndTimestamp` query. Because the final `ORDER BY timestamp_range DESC LIMIT 1` always prefers the row with the highest `timestamp_range` lower bound, the current-state row from `entity` (whose `timestamp_range` starts at the entity's last-modification time) will win over the correct historical row from `entity_history` whenever the entity was modified after the queried block timestamp.

## Finding Description

**File:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`

**Affected methods:**
- `findActiveByEvmAddressAndTimestamp` (lines 59–85)
- `findActiveByEvmAddressOrAliasAndTimestamp` (lines 95–121)

**Entity branch — missing guard (lines 67–72 and 103–108):**
```sql
(
    select *
    from entity e
    where e.deleted is not true          -- ← no lower(timestamp_range) <= ?2
    and e.id = (select id from entity_cte)
)
```

**History branch — correctly guarded (lines 74–81):**
```sql
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2   -- ← guard present
    and eh.id = (select id from entity_cte)
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

**Correctly written `findActiveByIdAndTimestamp` for contrast (lines 138–141):**
```sql
select *
from entity
where id = ?1 and lower(timestamp_range) <= ?2   -- ← guard present
and deleted is not true
```

**Root cause:** `entity_cte` only checks `created_timestamp <= ?2`, confirming the entity existed at the queried block. It does not constrain the entity's *current* `timestamp_range` lower bound. The `entity` table always holds `timestamp_range = [T_last_modified, ∞)`. If `T_last_modified > blockTimestamp`, that row must be excluded, but it is not.

**Test gap:** The test `findHistoricalEntityByEvmAddressAndTimestampRangeEqualToBlockTimestampAndDeletedIsFalse` (lines 157–168) calls `persistEntityHistoryWithId` *after* `persistEntity`, so `domainBuilder`'s monotonically increasing timestamp generator gives `entityHistory` the *higher* timestamp. The test queries at `entityHistory.getTimestampLower()`, so the history row wins by timestamp ordering — the reverse of the real bug scenario — and the test passes while the defect goes undetected.

## Impact Explanation
Any historical `eth_call` simulation that resolves an account or contract by EVM address will receive the entity's **current** state (balances, keys, admin settings, freeze/KYC flags, etc.) rather than the state valid at the requested block. Off-chain systems, audit tools, or dApps that rely on historical EVM simulation accuracy (replaying past governance votes, verifying historical token ownership, auditing past access-control state) will receive silently incorrect data. The same flaw exists in `findActiveByEvmAddressOrAliasAndTimestamp`, widening the affected surface to alias-based lookups.

## Likelihood Explanation
The `/api/v1/contracts/call` endpoint is publicly documented and requires no authentication. The `block` field accepts any historical block number. Any entity that has ever been modified (key rotation, balance change, freeze/unfreeze, etc.) is a valid target. The attacker needs only a valid EVM address and any block number prior to the entity's last modification — both are trivially discoverable from public mirror-node REST APIs. The attack is fully repeatable and requires no special privileges.

## Recommendation
Add `and lower(e.timestamp_range) <= ?2` to the entity branch in both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- findActiveByEvmAddressAndTimestamp entity branch (fix)
(
    select *
    from entity e
    where e.deleted is not true
    and lower(e.timestamp_range) <= ?2      -- ← add this
    and e.id = (select id from entity_cte)
)
```

Apply the identical fix to `findActiveByEvmAddressOrAliasAndTimestamp`. Additionally, add a test that persists an entity first (lower timestamp) and then an entity_history record with a *lower* timestamp than the entity, queries at a block between the two, and asserts the history record is returned — the scenario the current test suite does not cover.

## Proof of Concept

**Data setup:**
```
entity table:
  id=42, evm_address=0xABCD, created_timestamp=100, timestamp_range=[200,∞), key=KEY_NEW

entity_history table:
  id=42, evm_address=0xABCD, timestamp_range=[100,200), key=KEY_OLD
```

**Request:**
```
POST /api/v1/contracts/call
{ "to": "0xABCD", "block": "<block at timestamp 150>" }
```

**Query execution with blockTimestamp=150:**
1. `entity_cte`: finds id=42 (created_timestamp=100 ≤ 150 ✓)
2. Entity branch: returns row with `timestamp_range=[200,∞)` — no guard, included despite 200 > 150
3. History branch: returns row with `timestamp_range=[100,200)` — lower(100) ≤ 150 ✓
4. `ORDER BY timestamp_range DESC LIMIT 1`: picks `[200,∞)` (lower bound 200 > 100)

**Result:** `KEY_NEW` (current state) is returned instead of `KEY_OLD` (correct state at T=150).
