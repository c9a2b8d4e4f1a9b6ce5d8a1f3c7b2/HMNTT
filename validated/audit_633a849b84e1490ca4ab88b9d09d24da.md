Audit Report

## Title
Integer Overflow in `NumberRangeParameter.getInclusiveValue()` Bypasses `registeredNodeId` Filter

## Summary
Supplying `registeredNodeId=gt:9223372036854775807` (`Long.MAX_VALUE`) causes `getInclusiveValue()` to compute `Long.MAX_VALUE + 1`, which silently wraps to `Long.MIN_VALUE` in Java. `Math.max(0L, Long.MIN_VALUE)` then resets `lowerBound` to `0`, and the subsequent `lowerBound > upperBound` guard does not fire. The repository query executes with `registered_node_id >= 0 AND registered_node_id <= Long.MAX_VALUE`, returning all non-deleted registered nodes instead of zero results.

## Finding Description

**Step 1 — Parsing (`NumberRangeParameter.getNumberValue()`):**
`Long.parseLong("9223372036854775807")` succeeds and returns `Long.MAX_VALUE`. The only guard is `value < 0`; `Long.MAX_VALUE` is positive, so it passes. [1](#0-0) 

**Step 2 — Overflow (`NumberRangeParameter.getInclusiveValue()`):**
For `RangeOperator.GT`, the method returns `value + 1`. Java `long` arithmetic wraps silently: `Long.MAX_VALUE + 1 = Long.MIN_VALUE` (-9223372036854775808). No exception is thrown. [2](#0-1) 

**Step 3 — `hasLowerBound()` routes the overflowed value into `Math.max`:**
`RangeParameter.hasLowerBound()` returns `true` for `GT`. [3](#0-2) 

In `resolveRegisteredNodeIdBounds()`, the branch `range.hasLowerBound()` is taken and executes `lowerBound = Math.max(0L, Long.MIN_VALUE)` = `0L`. The overflowed value is silently discarded. [4](#0-3) 

**Step 4 — Guard fails:**
`lowerBound (0) > upperBound (Long.MAX_VALUE)` is `false`, so no `IllegalArgumentException` is thrown. [5](#0-4) 

**Step 5 — Repository query returns all non-deleted nodes:**
`findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs(0L, Long.MAX_VALUE, ...)` executes `WHERE registered_node_id >= 0 AND registered_node_id <= 9223372036854775807 AND deleted IS FALSE`. [6](#0-5) 

**Why the existing test does not catch this:**
The test marks `"9223372036854775808"` (Long.MAX_VALUE + 1 as a *string*) as invalid — that throws `NumberFormatException` from `Long.parseLong`. The actual trigger `"9223372036854775807"` (Long.MAX_VALUE as a string) is not in the invalid list and is never tested with a `GT` operator. [7](#0-6) 

## Impact Explanation
An unauthenticated caller can bypass the `registeredNodeId` range filter entirely. A request semantically intended to return zero results (no node can have an ID greater than `Long.MAX_VALUE`) instead returns all non-deleted registered nodes up to the configured pagination `limit`. If registered node records contain operationally sensitive data (e.g., pending or pre-production nodes, internal configuration), this constitutes unauthorized data disclosure. The `deleted IS FALSE` filter and pagination `limit` remain in effect, partially bounding the exposure.

## Likelihood Explanation
No authentication or special privilege is required. The parameter is a standard query parameter on the public REST API. The exact trigger value (`9223372036854775807`) is trivially discoverable by any attacker familiar with Java `Long` boundaries. The attack is stateless, repeatable, and requires a single HTTP GET request.

## Recommendation
Add an upper-bound check in `getNumberValue()` or `getInclusiveValue()` to reject values that would overflow on `GT`/`LT` arithmetic:

- In `getNumberValue()`: reject `value == Long.MAX_VALUE` (or more generally, any value where `value + 1` would overflow) when the operator is `GT`.
- Alternatively, in `getInclusiveValue()`: use `Math.addExact(value, 1)` for `GT` and `Math.subtractExact(value, 1)` for `LT`, which throw `ArithmeticException` on overflow, and catch/rethrow as `IllegalArgumentException`.
- Add a test case for `"gt:9223372036854775807"` asserting it throws `IllegalArgumentException`.

The same pattern exists in `EntityIdRangeParameter.getInclusiveValue()` and `Bound.getAdjustedLowerRangeValue()` / `adjustUpperBound()` and should be reviewed for the same class of overflow. [8](#0-7) [9](#0-8) 

## Proof of Concept

```
GET /api/v1/network/nodes/registered?registeredNodeId=gt:9223372036854775807
```

**Expected behavior:** 0 results (no node can have an ID > `Long.MAX_VALUE`).

**Actual behavior:** All non-deleted registered nodes are returned (up to the configured `limit`), because the overflow resets `lowerBound` to `0` and `upperBound` remains `Long.MAX_VALUE`.

Arithmetic trace:
```
Long.parseLong("9223372036854775807") = 9223372036854775807  (Long.MAX_VALUE)
getInclusiveValue() for GT: 9223372036854775807 + 1 = -9223372036854775808  (Long.MIN_VALUE, overflow)
Math.max(0L, -9223372036854775808L) = 0L
lowerBound=0 > upperBound=9223372036854775807 → false  (no exception)
SQL: WHERE registered_node_id >= 0 AND registered_node_id <= 9223372036854775807 AND deleted IS FALSE
```
