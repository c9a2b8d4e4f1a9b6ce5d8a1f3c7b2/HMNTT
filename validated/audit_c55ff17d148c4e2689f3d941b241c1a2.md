Audit Report

## Title
`failed` Field in Opcodes Response Derived Exclusively from EVM Re-execution Result, Not Original On-Chain Consensus Status

## Summary
`OpcodeServiceImpl.buildOpcodesResponse()` sets the `failed` field solely from the EVM re-execution result (`txnResult.isSuccessful()`). The original on-chain failure status stored in `ContractResult.transactionResult` is fetched but never consulted when constructing the response. For transactions that failed at the Hedera consensus layer (e.g., `CONSENSUS_GAS_EXHAUSTED`) rather than at the EVM level, the re-execution can succeed, causing the endpoint to return `failed: false` for a transaction that definitively failed on-chain.

## Finding Description

**Exact code path:**

`OpcodeServiceImpl.buildOpcodesResponse()` at line 146:
```java
.failed(txnResult == null || !txnResult.isSuccessful())
``` [8](#0-7) 

`txnResult` is the `EvmTransactionResult` from the EVM re-execution. `isSuccessful()` returns `true` only when `responseCodeEnum == ResponseCodeEnum.SUCCESS` (re-execution success): [2](#0-1) 

The original on-chain status is stored in `ContractResult.transactionResult` (an integer proto status code): [3](#0-2) 

This field is fetched at lines 154–156 of `OpcodeServiceImpl` to build `ContractDebugParameters`: [4](#0-3) 

But it is **never read** when constructing the `OpcodesResponse`. There is no code in `buildOpcodesResponse()` or `buildCallServiceParameters()` that cross-checks the re-execution outcome against `contractResult.transactionResult`.

**Why queryable records exist for consensus-layer failures:**

`ContractResultServiceImpl.addDefaultEthereumTransactionContractResult()` creates a `ContractResult` record for any failed Ethereum transaction (excluding `DUPLICATE_TRANSACTION` and `WRONG_NONCE`): [7](#0-6) 

This record stores the Hedera failure code via `transactionResult(transaction.getResult())` and includes the transaction hash, making it queryable via the opcodes endpoint. The `contractId` is set to `0`, but the receiver address is correctly resolved from `ethereumTransaction.getToAddress()` for contract calls.

**Why existing checks are insufficient:**

`ContractDebugService.validateResult()` catches and silently swallows `MirrorEvmTransactionException`, ensuring the re-execution result is always returned regardless of EVM-level failure: [5](#0-4) 

`TransactionExecutionService.handleFailedResult()` correctly returns a failed `EvmTransactionResult` in the opcode-trace path when the EVM itself fails: [6](#0-5) 

However, this path is only reached when the EVM re-execution also fails. When the EVM re-execution succeeds (consensus-layer-failed transaction), `buildSuccessResult()` is called and returns `ResponseCodeEnum.SUCCESS`, which propagates to `failed: false`.

## Impact Explanation
The opcodes endpoint is the only Mirror Node API providing EVM execution traces for historical transactions. Consumers (block explorers, auditing tools, forensic debuggers) rely on the `failed` field to determine whether a historical transaction succeeded. A response of `failed: false` for a transaction that definitively failed on-chain constitutes a false representation of Hashgraph history. This can mislead users or automated systems into believing a transaction succeeded when it did not — for example, falsely confirming a token transfer or contract state change that never occurred. The impact is bounded to read-only misrepresentation (no state mutation), but it directly undermines the integrity guarantee of the historical debug endpoint.

## Likelihood Explanation
The precondition — a transaction that failed at the Hedera consensus layer rather than the EVM layer — is common in production. `CONSENSUS_GAS_EXHAUSTED` and similar codes occur regularly under network load. The endpoint is publicly accessible with no authentication. The divergence is deterministic and repeatable for any qualifying transaction. No special privileges, keys, or network access beyond standard HTTP are required.

## Recommendation
In `buildOpcodesResponse()`, after obtaining `txnResult`, cross-check the re-execution outcome against the original on-chain status. The `ContractResult` is already fetched in `buildCallServiceParameters()` and can be threaded through to `buildOpcodesResponse()` (e.g., via `OpcodesProcessingResult` or a wrapper). The fix should be:

```java
// Determine failed: true if original on-chain status was not SUCCESS,
// OR if re-execution itself failed.
boolean originalFailed = contractResult.getTransactionResult() != null
    && contractResult.getTransactionResult() != ResponseCodeEnum.SUCCESS_VALUE;
boolean reexecFailed = txnResult == null || !txnResult.isSuccessful();
.failed(originalFailed || reexecFailed)
```

This ensures that any transaction that failed on-chain — regardless of whether the EVM re-execution succeeds — is reported as `failed: true`.

## Proof of Concept

1. Identify a historical Ethereum transaction that failed on-chain with a Hedera-specific consensus-layer code (e.g., `CONSENSUS_GAS_EXHAUSTED`). Such transactions have a `ContractResult` record created by `addDefaultEthereumTransactionContractResult()` and a `ContractTransactionHash` record, making them queryable.
2. Note the transaction hash from the `ContractTransactionHash` record.
3. Send:
   ```
   GET /api/v1/contracts/results/{hash}/opcodes
   Accept-Encoding: gzip
   ```
4. `OpcodeServiceImpl.processOpcodeCall()` fetches the `ContractResult`, builds `ContractDebugParameters` using the original gas limit, and re-executes via the EVM at the historical block state.
5. The EVM re-execution succeeds (no consensus-layer throttle in the EVM path).
6. `buildOpcodesResponse()` evaluates `txnResult.isSuccessful()` → `true` → sets `failed: false`.
7. The HTTP 200 response contains `"failed": false`, contradicting the Hashgraph record which shows the transaction failed with `CONSENSUS_GAS_EXHAUSTED`.
