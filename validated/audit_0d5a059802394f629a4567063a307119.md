# Audit Report

## Title
206 Partial Content Response Incorrectly Cached, Poisoning Subsequent Users with Incomplete Transaction History

## Summary
`responseCacheUpdateHandler()` uses `httpStatusCodes.isSuccess(res.statusCode)` as its cache-admission guard. `isSuccess` is defined as `code >= 200 && code < 300`, which includes HTTP 206. The application intentionally emits 206 for certain endpoints when the mirror node's data is incomplete. Once a 206 response is cached, every subsequent request for the same URL is served the partial body with status 206 for the full cache TTL — even after the mirror node has caught up and would return a complete 200 OK.

## Finding Description

**`isSuccess` includes 206:** [8](#0-7) 

**Cache-admission condition in `responseCacheUpdateHandler`:** [5](#0-4) 

**Status code stored verbatim for non-304 responses:** [6](#0-5) 

**NFT transaction history emits 206 when mint transaction is not yet ingested:** [9](#0-8) 

**Contract results emit 206 when `callResult` is nil:** [10](#0-9) 

**`responseHandler` applies `res.locals.statusCode` to `res.statusCode` before the cache update handler runs:** [4](#0-3) 

The root cause is the failed assumption that every 2xx response is a complete, cacheable result. HTTP 206 is explicitly used by this application to signal *incomplete* data (mirror node still syncing). The `isSuccess()` predicate does not distinguish between 200 and 206. There is no exclusion for `PARTIAL_CONTENT` (206) anywhere in the cache-update path, and no short-TTL override for partial responses. The `isUnmodified` branch correctly remaps 304→200 before storing, but no analogous guard exists for 206.

## Impact Explanation
Any user who queries `/api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions` or `/api/v1/contracts/results/{transactionIdOrHash}` while the mirror node has incomplete data causes a 206 partial response to be cached under the normal URL-hash key. All subsequent users querying the identical URL receive the cached partial body with status 206 for the full cache TTL — even after the mirror node has fully ingested the missing data and would return a complete 200 OK. For NFT transaction history, the mint transaction is absent from the cached response, presenting a falsified provenance record to every consumer of that cached entry. This constitutes data-integrity corruption of transaction history served at scale.

## Likelihood Explanation
Mirror nodes routinely have transient windows of incomplete data during normal sync operation. No special privileges, authentication, or crafted payload are required — a plain unauthenticated GET request is sufficient. An attacker who monitors mirror-node lag (e.g., by comparing the latest consensus timestamp in responses against the network's known head) can reliably time a request to land during an incomplete-data window, guaranteeing cache poisoning for the duration of the TTL. The attack is repeatable: after TTL expiry the attacker can re-trigger it with another GET request if the data is still incomplete, or simply wait for the next sync gap.

## Recommendation
Exclude HTTP 206 from the cache-admission condition in `responseCacheUpdateHandler`. The minimal fix is to change the guard in `rest/middleware/responseCacheHandler.js` line 95 to explicitly reject partial-content responses:

```js
if (
  responseBody &&
  responseCacheKey &&
  (isUnmodified || (httpStatusCodes.isSuccess(res.statusCode) && res.statusCode !== httpStatusCodes.PARTIAL_CONTENT.code))
) {
```

Alternatively, redefine `isSuccess` to exclude 206, or add a dedicated `isCacheable` predicate that only admits status codes that represent complete, stable responses (200, 204, etc.). A short-TTL override for 206 responses is insufficient — partial responses should not be cached at all under the normal cache key.

## Proof of Concept
1. Identify a token/NFT whose mint transaction has not yet been ingested by the mirror node (e.g., during a sync lag window).
2. Send a GET request to `/api/v1/tokens/{tokenId}/nfts/{serialNumber}/transactions`.
3. The mirror node returns HTTP 206 with a partial transaction list (mint transaction absent). `responseCacheUpdateHandler` stores this 206 response in Redis under the URL-hash key with the full `max-age` TTL.
4. Wait for the mirror node to fully ingest the mint transaction (it would now return 200 with complete data).
5. Send the same GET request again. The response is served from cache: HTTP 206, partial body, mint transaction still absent — for the remainder of the TTL.
6. Repeat step 2 after TTL expiry to re-poison the cache.
