Audit Report

## Title
`incrementByteArray()` Integer Overflow on Max-Value `gt` Parameter Exposes Full Hook Storage

## Summary
Supplying `key=gt:0xFFFF...FFFF` (64 `F`s) to the hook storage endpoint causes `BytesUtil.incrementByteArray()` to silently wrap the all-`0xFF` value to all-`0x00` via Tuweni's `MutableBytes32.increment()`. The wrapped value is equal to `MIN_KEY_BYTES`, so the lower-bound guard in `hookStorageChangeRequest()` never updates the bound, and the repository query returns every non-deleted storage slot for the targeted hook.

## Finding Description

**Step 1 — Parsing (`SlotRangeParameter.java`)**

The `GT` branch calls `incrementByteArray` on the decoded bytes: [1](#0-0) 

**Step 2 — Silent overflow (`BytesUtil.java`)**

`incrementByteArray` delegates directly to Tuweni's `MutableBytes32.increment()`, which performs modular (wrapping) big-endian arithmetic with no overflow check or exception: [2](#0-1) 

`[0xFF×32] + 1` wraps to `[0x00×32]`. The method returns silently.

**Step 3 — Operator normalisation (`SlotRangeParameter.java:38`)**

`GT` is converted to `GTE`, so the resulting parameter is `SlotRangeParameter(GTE, [0x00×32])`: [3](#0-2) 

**Step 4 — Lower-bound guard fails (`HooksController.java`)**

`lowerBound` is initialised to `MIN_KEY_BYTES = [0x00×32]`: [4](#0-3) 

The guard requires `value > lowerBound` (strictly): [5](#0-4) 

`Arrays.compareUnsigned([0x00×32], [0x00×32])` returns `0`, not `> 0`. The branch is skipped. `lowerBound` remains `[0x00×32]` and `upperBound` remains `MAX_KEY_BYTES = [0xFF×32]`.

**Step 5 — Full-range repository query (`HookServiceImpl.java`)**

With no `EQ` key filters, the service issues an unbounded range query: [6](#0-5) 

This returns every non-deleted storage slot for the hook — identical to issuing no `key` filter at all.

## Impact Explanation
An attacker can enumerate the complete current storage of any smart contract hook. Storage slots may contain balances, access-control mappings, or other sensitive application state. The endpoint carries no visible authentication requirement (`HooksController` has no `@PreAuthorize` or equivalent annotation), making every deployed hook reachable. [7](#0-6) 

## Likelihood Explanation
The exploit requires zero privileges and zero on-chain interaction. A valid `ownerId` and `hookId` are enumerable via the `/hooks` endpoint. The payload is a single well-formed query parameter (`key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF`). It is trivially repeatable and automatable.

## Recommendation
Add an explicit overflow guard in `BytesUtil.incrementByteArray` (throw `IllegalArgumentException` when the input is `[0xFF×32]`), or add a post-increment check in `getInclusiveValue` before returning:

```java
// SlotRangeParameter.java – getInclusiveValue
if (operator == RangeOperator.GT) {
    if (Arrays.equals(bytes, MAX_BYTES)) {
        throw new IllegalArgumentException("GT value cannot be the maximum slot key");
    }
    return incrementByteArray(bytes);
}
```

Alternatively, validate in `hookStorageChangeRequest` that the incremented lower bound is strictly greater than the original decoded value before accepting it.

## Proof of Concept
```
GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage
    ?key=gt:0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF
```

Expected (correct) behaviour: 400 Bad Request or empty result set (no key can be greater than the maximum value).

Actual behaviour: returns all non-deleted storage slots for the hook, equivalent to omitting the `key` parameter entirely.
