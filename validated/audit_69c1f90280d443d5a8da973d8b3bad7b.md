Audit Report

## Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` Bypasses Query Limit, Enabling Unbounded Database Reads

## Summary
A narrowing cast from `long` to `int` in `PollingTopicMessageRetriever.poll()` causes integer overflow when a client supplies `limit = Long.MAX_VALUE`. The overflowed value `-1` is used to construct an internal `TopicMessageFilter` whose `hasLimit()` returns `false`, causing `findByFilter()` to execute with no `LIMIT` clause and potentially return the entire `topic_message` table.

## Finding Description

**File:** `grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`

The overflow occurs at lines 68–75:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())  // line 69
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());       // line 71
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build(); // line 75
``` [1](#0-0) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getTotal().get() = 0`:
- `(int)(Long.MAX_VALUE - 0L)` = `(int)(0x7FFFFFFFFFFFFFFF)` = `0xFFFFFFFF` = **`-1`** (signed int overflow)
- `Math.min(-1, maxPageSize)` = **`-1`** regardless of configured page size
- `filter.toBuilder().limit(-1)` widens `-1` (int) to `-1L` (long)

**Why `hasLimit()` fails on the internal filter:**

`TopicMessageFilter.hasLimit()` at line 39 returns `limit > 0`. [2](#0-1) 

For the internal filter with `limit = -1L`, this evaluates to `false`.

**Why `@Min(0)` does not prevent this:**

The `@Min(0)` constraint on `TopicMessageFilter.limit` only rejects negative values. `Long.MAX_VALUE` is a valid positive `long` and passes validation. [3](#0-2) 

**Why Spring `@Validated` does not catch `-1` on the internal filter:**

The `@Validated` annotation on `TopicMessageFilter` is enforced by Spring only on Spring-managed bean method parameters. The internal `filter.toBuilder().limit(-1).build()` is a plain Lombok builder invocation — Spring's validation interceptor is never invoked, so `-1` is silently accepted. [4](#0-3) 

**How `findByFilter()` skips the `LIMIT` clause:**

In `TopicMessageRepositoryCustomImpl.findByFilter()`, `setMaxResults()` is only called when `filter.hasLimit()` is `true`. Since the internal filter has `limit = -1`, `hasLimit()` returns `false` and `setMaxResults()` is never called, resulting in an unbounded JPA query. [5](#0-4) 

## Impact Explanation
An unprivileged gRPC subscriber can force the mirror node to load the entire `topic_message` table for a given topic into memory per poll cycle. This causes unbounded heap allocation in the Flux pipeline, saturates database I/O, and can exhaust JVM heap on the mirror node. Because the polling loop in throttled mode repeats with `numRepeats = Long.MAX_VALUE`, the attack is automatically sustained on every poll interval. [6](#0-5) 

## Likelihood Explanation
Any client that can open a gRPC `subscribeTopic` stream can trigger this. No authentication, elevated privilege, or special knowledge is required beyond a valid topic ID. `Long.MAX_VALUE` is trivially constructable in any gRPC client (protobuf `uint64` field). The overflow is deterministic and reproducible on every poll interval.

## Recommendation
Replace the unchecked narrowing cast with a saturating conversion. Before casting, clamp the computed remaining limit to `Integer.MAX_VALUE`:

```java
long remaining = filter.getLimit() - context.getTotal().get();
int limit = filter.hasLimit()
        ? (int) Math.min(remaining, Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
```

This ensures `limit` is always a non-negative `int`, so `pageSize` is always bounded, and the internal filter's `hasLimit()` always returns `true` when a limit was originally set.

Additionally, add a defensive guard in `findByFilter()` to enforce a hard maximum even if a negative limit somehow reaches it:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) Math.max(1, filter.getLimit()));
}
```

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1

        }
