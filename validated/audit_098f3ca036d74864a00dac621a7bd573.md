Audit Report

## Title
Unbounded Timestamp Scan via Missing Upper Bound Cap in `getHookStorageChange`

## Summary
When a caller supplies only a lower-bound timestamp (e.g., `timestamp=gte:0`) with no upper bound, `Bound.adjustUpperBound()` returns `Long.MAX_VALUE`, which is forwarded verbatim to both `HookStorageChangeRepository` queries. The resulting `DISTINCT ON (key)` SQL must process every row from timestamp X to `Long.MAX_VALUE` for the given owner/hook before the page limit is applied, enabling an unprivileged caller to trigger arbitrarily large database scans with a single HTTP request.

## Finding Description

**Verified code path:**

`HookServiceImpl.getHookStorage` delegates to `getHookStorageChange` at line 57–58 whenever `request.isHistorical()` is true. [1](#0-0) 

`isHistorical()` returns `!timestamp.isEmpty()`, and `isEmpty()` is `lower == null && upper == null`. Supplying only `timestamp=gte:X` sets `lower` but leaves `upper` null, so `isHistorical()` returns `true`. [2](#0-1) [3](#0-2) 

Inside `getHookStorageChange`, lines 97–99 extract the timestamp bounds: [4](#0-3) 

`adjustUpperBound()` returns `Long.MAX_VALUE` when `upper` is null — no cap is applied: [5](#0-4) 

`Long.MAX_VALUE` is then passed directly to both repository queries: [6](#0-5) 

Both SQL queries use `consensus_timestamp between :timestampLowerBound and :timestampUpperBound` combined with `DISTINCT ON (key) … ORDER BY key, consensus_timestamp DESC`. PostgreSQL must read and sort **all** rows matching `owner_id`, `hook_id`, key range, and the open-ended timestamp range before it can emit the first page of results. The `Pageable` limit caps output rows, not the rows the engine must examine. [7](#0-6) 

The only guard in `Bound`'s constructor checks `adjustedLower > adjustedUpper`, but the timestamp `Bound` is constructed with `primarySortField = false` (via `Bound.of()`), so the guard is never evaluated: [8](#0-7) [9](#0-8) 

## Impact Explanation
An attacker who knows a valid `ownerId`/`hookId` pair for a hook with a large history can repeatedly issue requests with `timestamp=gte:0`, forcing the database to perform a full historical scan of the `hook_storage_change` table on every request. This can exhaust database CPU and I/O, degrade response times for all users, and interfere with transaction processing that shares the same database.

## Likelihood Explanation
The endpoint is reachable by any unprivileged external caller. The only precondition is knowing a valid `ownerId` and `hookId`, which are enumerable via the public `/hooks` endpoint. The attack requires a single HTTP parameter, no authentication, and no special tooling. The SECURITY.md exclusion for "Impacts that only require DDoS" does not apply here — this is a single-request application-level resource exhaustion caused by a missing input validation/capping defect, not a volumetric network flood.

## Recommendation
Cap `timestampUpperBound` to a sensible maximum (e.g., current consensus timestamp or `now()` in nanoseconds) when `upper` is null, either inside `adjustUpperBound()` or at the call site in `getHookStorageChange`. Alternatively, reject requests that supply a lower-bound timestamp without a corresponding upper bound, or enforce a maximum scan window (e.g., no more than 24 hours of history per request). [10](#0-9) 

## Proof of Concept
```
GET /api/v1/hooks/{ownerId}/{hookId}/storage?timestamp=gte:0&limit=25
```
With `upper == null`, `adjustUpperBound()` returns `Long.MAX_VALUE`. The generated SQL becomes:
```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = <ownerId>
  AND hook_id  = <hookId>
  AND key BETWEEN <min> AND <max>
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
ORDER BY key, consensus_timestamp DESC
LIMIT 25;
```
PostgreSQL must sort the entire matching partition before returning 25 rows. Repeating this request drives sustained full-table-scan load on the database with no rate-limiting or query cost guard in the application layer. [11](#0-10)

    public long adjustUpperBound() {
        if (this.upper == null) {
            return Long.MAX_VALUE;
        }

        long upperBound = this.upper.value();
        if (this.upper.operator() == RangeOperator.LT) {
            upperBound--;
        }

        return upperBound;
    }
