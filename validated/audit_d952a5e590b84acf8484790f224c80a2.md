# Audit Report

## Title
Uncached, Unthrottled `NetworkStakeRepository.findLatest()` Enables Thread Pool Exhaustion via DB Slowdown Cascade

## Summary
The `GET /api/v1/network/stake` endpoint in the rest-java service invokes `networkStakeRepository.findLatest()` on every request with no caching, no per-IP rate limiting, and no application-level bulkhead or circuit breaker. The Traefik middleware for rest-java omits both `inFlightReq` and `rateLimit` directives (present in Rosetta), and its circuit breaker only fires on network errors or 5xx responses — not on slow-but-successful queries. An unprivileged attacker flooding this endpoint can hold Tomcat threads blocked against the DB for up to 20 seconds each, exhausting the thread pool and causing service-wide unavailability.

## Finding Description

**Exact code path:**

`NetworkController.getNetworkStake()` (no rate-limit annotation, no throttle guard): [1](#0-0) 

calls `NetworkServiceImpl.getLatestNetworkStake()` (no `@Cacheable`, no Resilience4j `@CircuitBreaker`/`@Bulkhead`): [2](#0-1) 

which calls `NetworkStakeRepository.findLatest()` — a native correlated subquery executed synchronously on every request: [3](#0-2) 

**Root cause — failed assumptions:**

1. **No caching**: The grpc module's `NodeStakeRepository.findAllStakeByConsensusTimestamp()` is annotated `@Cacheable` with a dedicated `CacheManager`: [4](#0-3) 
The rest-java `NetworkStakeRepository` has no equivalent — every HTTP request hits the DB.

2. **No per-IP rate limiting or `inFlightReq` at the proxy**: The rest-java Traefik middleware only configures `circuitBreaker` and `retry`: [5](#0-4) 
Compare with Rosetta, which adds `inFlightReq` (5 concurrent per IP) and `rateLimit` (10 req/s per host): [6](#0-5) 

3. **Traefik circuit breaker is blind to slow 200 OK responses**: The expression `NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25` only trips on network-layer errors or HTTP 5xx. A DB query that takes 19 seconds and returns HTTP 200 never triggers it: [7](#0-6) 

4. **Retry middleware amplifies the attack**: With `attempts: 3`, each attacker request can generate up to 3 DB queries, tripling thread and connection consumption: [8](#0-7) 

5. **HPA is disabled by default**: No auto-scaling to absorb the load: [9](#0-8) 

6. **Statement timeout is 20 seconds**: The DB-level bound is long enough to make the attack effective with modest concurrency: [10](#0-9) 

## Impact Explanation
Thread pool exhaustion causes the rest-java service to stop processing all requests (not just `/network/stake`), affecting every endpoint routed through it including `/api/v1/network/`, account allowances, airdrops, and topics. The `prometheusRules` are defined but `enabled: false` by default, meaning alerts may not fire: [11](#0-10) 
The Prometheus alert thresholds themselves are set at 80% CPU and 80% memory, confirming the operational significance of the resource exhaustion scenario: [12](#0-11) 

## Likelihood Explanation
The endpoint is unauthenticated, publicly documented in the OpenAPI spec, and requires only standard HTTP GET requests: [13](#0-12) 
The attack is trivially repeatable with any HTTP load tool. The absence of per-IP rate limiting or `inFlightReq` means a single IP can open hundreds of concurrent connections. The 20-second statement timeout is the only bound, and it is long enough to make the attack effective with modest concurrency (~50–100 concurrent requests).

## Recommendation
1. **Add caching** to `NetworkStakeRepository.findLatest()` in the rest-java module (analogous to the grpc module's `@Cacheable` on `NodeStakeRepository.findAllStakeByConsensusTimestamp()`). The `network_stake` table is updated once per day, making it an ideal cache candidate.
2. **Add `inFlightReq`** to the rest-java Traefik middleware (matching the graphql and Rosetta configurations).
3. **Enable HPA** by default or lower the default replica count floor.
4. **Enable `prometheusRules`** by default so that CPU/memory/DB-connection alerts fire in all deployments.
5. **Consider a Resilience4j `@Bulkhead`** on `getLatestNetworkStake()` to cap concurrent DB calls at the application level.

## Proof of Concept
```bash
# Flood the endpoint with 200 concurrent connections, each making 1000 requests
wrk -t200 -c200 -d60s https://<mirror-node-host>/api/v1/network/stake
```
With 200 concurrent requests each holding a Tomcat thread for up to 20 seconds (the DB statement timeout), the thread pool is saturated. Subsequent requests to any rest-java endpoint (e.g., `/api/v1/network/nodes`, `/api/v1/accounts/{id}/allowances/nfts`) will queue in the accept backlog and eventually time out, causing full service unavailability. The retry middleware (`attempts: 3`) means each attacker connection generates up to 3 DB queries, tripling the effective load on the connection pool.
