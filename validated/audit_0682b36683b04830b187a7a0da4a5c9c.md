Audit Report

## Title
Unprivileged gRPC Subscriber Can Reset `lastConsensusTimestamp` to Skip Unprocessed Topic Messages Permanently

## Summary
`SharedPollingTopicListener.PollingContext.onStart()` unconditionally overwrites `lastConsensusTimestamp` with the current wall-clock time (`DomainUtils.now()`) every time the shared Flux receives a new upstream subscription. Because the Flux is built with `.share()` (Reactor's `publish().refCount()`), the upstream — including `.doOnSubscribe(context::onStart)` — is re-subscribed every time the active-subscriber count transitions from 0 to 1. Any unauthenticated caller of the public `subscribeTopic` gRPC endpoint who arrives when no other subscribers are active triggers this reset, advancing the polling cursor past messages that have achieved consensus on the network but have not yet been written to the database by the importer.

## Finding Description

**Exact code path**

`PollingContext.onStart` unconditionally sets the cursor to the current wall-clock nanoseconds: [1](#0-0) 

`DomainUtils.now()` returns `convertToNanosMax(Instant.now())` — the current wall-clock instant in nanoseconds: [2](#0-1) 

`findLatest` issues `WHERE consensus_timestamp > ?1` — a strict greater-than predicate against the cursor: [3](#0-2) 

The shared Flux is assembled with `.doOnSubscribe(context::onStart)` placed before `.share()`, meaning every upstream re-subscription (0→1 refcount) invokes `onStart`: [4](#0-3) 

**Root cause and failed assumption**

The design assumes `onStart` is called exactly once at application startup, making `DomainUtils.now()` a safe initial cursor (no live messages exist before "now"). The failed assumption is that `.share()` guarantees a single upstream subscription for the process lifetime. In reality, Reactor's `publish().refCount()` cancels the upstream when the last subscriber leaves and creates a fresh upstream subscription when the next subscriber arrives. Because `PollingContext` is a single shared instance constructed once in the constructor, each re-subscription overwrites the same `lastConsensusTimestamp` field with the new wall-clock time, which may be ahead of messages that have achieved consensus but have not yet been ingested.

## Impact Explanation

Any message whose `consensusTimestamp` falls between the previous cursor value and the new `DomainUtils.now()` value at the moment of re-subscription is permanently excluded from all future `findLatest` polls for the lifetime of the process (or until the next re-subscription advances the cursor even further). Because `lastConsensusTimestamp` is only ever moved forward (by `onNext`), there is no self-healing path for the skipped message. All gRPC subscribers — current and future — sharing the single `PollingContext` are affected. Applications that rely on receiving every HCS topic message (e.g., governance freeze instructions, cross-chain coordination signals) will silently miss the skipped message. [5](#0-4) 

## Likelihood Explanation

The attack requires no credentials. The `subscribeTopic` gRPC endpoint is public and unauthenticated. The attacker needs only to subscribe when the active-subscriber count is zero — a realistic condition during low-traffic periods, after server restarts, or after all previous subscribers have timed out. The importer ingestion lag (typically seconds to minutes) provides a wide timing window between a message achieving consensus on the Hedera network and its appearance in the `topic_message` table. The attack is repeatable: every time the subscriber count drops to zero and the attacker re-subscribes, the cursor is reset further forward.

**Why existing mitigations are insufficient:**

The 1-second safety check in `TopicMessageServiceImpl` fires once and queries from the subscriber's `filter.getStartTime()` via `topicMessageRetriever.retrieve(gapFilter, false)`: [6](#0-5) 

This only catches messages already present in the database at the moment the check fires. If the targeted message has not yet been ingested at that instant, it is not recovered. The sequence-number gap-filling logic in `missingMessages` detects gaps between *consecutively received* messages: [7](#0-6) 

It cannot recover a message that was skipped before the listener's cursor was established, because there is no "previous message" to compare against. If no subsequent messages arrive on the topic, the gap is never detected.

## Recommendation

1. **Initialize the cursor once and never reset it on re-subscription.** Move the `lastConsensusTimestamp` initialization out of `onStart` and into the `PollingContext` constructor, setting it to `DomainUtils.now()` at construction time only. `onStart` should be a no-op or a logging-only callback.
2. **Alternatively, use `publish().refCount(1, Duration.ZERO)` with a grace period**, or switch to `publish().autoConnect()`, so the upstream is not cancelled when the last subscriber leaves, preserving the cursor across subscriber transitions.
3. **If re-subscription is unavoidable**, `onStart` should set `lastConsensusTimestamp` to `Math.min(currentValue, DomainUtils.now())` — i.e., never advance the cursor backward but also never reset it forward past its current position.

## Proof of Concept

```
1. Deploy mirror node with SHARED_POLL listener type.
2. Wait until all gRPC subscribers disconnect (subscriber count = 0).
   Confirm via the hiero.mirror.grpc.subscribers gauge.
3. Observe a pending HCS topic message M with consensusTimestamp T_freeze
   on the Hedera network (e.g., via a consensus node API or block explorer).
   Confirm the importer has NOT yet written M to topic_message
   (SELECT COUNT(*) FROM topic_message WHERE consensus_timestamp = T_freeze returns 0).
4. Subscribe to the gRPC subscribeTopic endpoint for the relevant topic.
   This is the first subscription; Reactor's refCount triggers upstream re-subscription,
   invoking onStart → lastConsensusTimestamp.set(DomainUtils.now()) = T_now > T_freeze.
5. Wait for the importer to ingest M (poll until
   SELECT COUNT(*) FROM topic_message WHERE consensus_timestamp = T_freeze returns 1).
6. Observe that the gRPC stream never delivers M.
   Confirm by querying findLatest(T_now, 100): returns 0 rows for T_freeze
   because T_freeze < T_now and the predicate is consensus_timestamp > T_now.
7. Subscribe a second, independent gRPC client to the same topic.
   It also never receives M, confirming the cursor reset affects all subscribers.
```
