Audit Report

## Title
Cache Poisoning via `Optional.empty()` Caching in `findRuntimeBytecode()` Causes Persistent Bytecode Unavailability

## Summary
`ContractRepository.findRuntimeBytecode()` is annotated with `@Cacheable(unless = "#result == null")`, but the method returns `Optional<byte[]>`. When the contract does not yet exist in the database, Spring Data JPA returns `Optional.empty()`, which is a non-null Java object. The `unless` guard evaluates to `false`, so the empty `Optional` is stored in the Caffeine cache. All subsequent calls for the same `contractId` return the cached empty result, even after the importer has written the correct bytecode to the database.

## Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16â€“18

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

Spring's `@Cacheable` evaluates the `unless` SpEL expression against the actual return value. `Optional.empty()` is a non-null singleton (`Optional.EMPTY`), so `#result == null` is `false`, and the empty `Optional` is stored in the Caffeine cache under the `contractId` key.

The cache is configured with `expireAfterAccess=1h` (`CacheProperties.java`, line 22), meaning each access resets the TTL. An attacker who periodically re-queries the same `contractId` can keep the poisoned entry alive indefinitely.

The consumer of this method, `ContractBytecodeReadableKVState.readFromDataSource()` (lines 42â€“46), calls `.orElse(null)` on the result, so the EVM layer receives `null` bytecode for the poisoned contract.

## Impact Explanation
Any `eth_call`, `eth_estimateGas`, or contract simulation routed through the web3 module that targets the poisoned `contractId` will receive `null` bytecode. The EVM treats a contract with no bytecode as non-existent, causing all calls to revert or return empty results. This is a data-integrity and availability impact: the mirror node serves incorrect state for a real on-chain contract for up to 1 hour per access, or indefinitely if the attacker periodically re-queries the same `contractId` (resetting the `expireAfterAccess` timer each time).

## Likelihood Explanation
No privileges are required. The attacker only needs to:
1. Observe a new contract deployment on-chain (public mempool/block data).
2. Issue a single unauthenticated HTTP request to the mirror node's web3 endpoint referencing that `contractId` before the importer finishes processing the block.

The importer always has a non-zero processing lag (typically seconds to tens of seconds). The attack window covers the entire lag period and is easily exploitable by automated tooling. The attack is repeatable for every new contract deployment.

## Recommendation
Change the `unless` condition to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only non-empty `Optional` values (i.e., contracts whose bytecode was actually found in the database) are stored in the cache. The same fix should be audited and applied to all other `@Cacheable`-annotated methods in the repository layer that return `Optional<T>` with `unless = "#result == null"` (e.g., `EntityRepository`, `RecordFileRepository`).

## Proof of Concept
1. A new contract with numeric ID `12345` is deployed on-chain. The importer has not yet written it to the database.
2. Attacker sends: `POST /api/v1/contracts/call` with `{"to": "0x0000000000000000000000000000000000003039", ...}` (long-zero address for ID `12345`).
3. `ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode(12345L)`.
4. The DB returns no rows â†’ Spring Data JPA returns `Optional.empty()`.
5. `unless = "#result == null"` evaluates to `false` â†’ `Optional.empty()` is stored in the Caffeine cache under key `12345`.
6. The importer writes the correct bytecode to the database seconds later.
7. All subsequent calls to `findRuntimeBytecode(12345L)` return the cached `Optional.empty()` â†’ `.orElse(null)` â†’ `null` bytecode â†’ EVM treats contract as non-existent â†’ all calls revert.
8. Attacker re-queries every ~50 minutes to reset the `expireAfterAccess` timer, keeping the poisoned entry alive indefinitely.
