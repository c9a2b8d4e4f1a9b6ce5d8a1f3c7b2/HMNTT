Audit Report

## Title
Unthrottled DB Query Storm via Concurrent gRPC Topic Subscriptions

## Summary
Every `subscribeTopic` call schedules a safety-check that fires after 1 second. If no data has arrived, it invokes `topicMessageRetriever.retrieve(gapFilter, false)` — the unthrottled path. In unthrottled mode, `PollingContext` issues up to `maxPolls` (default 12) DB queries at 20 ms intervals with a page size of 5,000 rows. Because there is no global per-IP subscription limit, an unprivileged attacker opening many concurrent gRPC connections can trigger hundreds of simultaneous unthrottled retrieval bursts, causing a DB query storm that degrades service for all users.

## Finding Description

**Code path 1 — Unthrottled `PollingContext` construction**

The `PollingContext` constructor at lines 94–108 of `PollingTopicMessageRetriever.java` sets aggressive parameters when `throttled=false`: `numRepeats=12`, `frequency=20ms`, `maxPageSize=5000`. [1](#0-0) 

These defaults are confirmed in `RetrieverProperties.UnthrottledProperties`: [2](#0-1) 

**Code path 2 — `isComplete()` never short-circuits the unthrottled path without a limit**

When `throttled=false`, `isComplete()` returns only `limitHit`. Without a client-supplied limit, `limitHit` is always `false`, so all 12 polls execute unconditionally. [3](#0-2) 

**Code path 3 — Safety check triggers the unthrottled path for every subscription to a quiet topic**

In `TopicMessageServiceImpl.subscribeTopic`, a `Mono.delay(1s)` fires for every subscription. If `topicContext.isComplete()` is false (which it always is for open-ended subscriptions since `TopicContext.isComplete()` returns `false` when `endTime == null`), it calls `missingMessages(topicContext, null)`. [4](#0-3) [5](#0-4) 

`missingMessages` with `current == null` unconditionally calls `topicMessageRetriever.retrieve(gapFilter, false)` — the unthrottled path: [6](#0-5) 

**Code path 4 — Per-connection cap is the only guard, and it is insufficient**

`maxConcurrentCallsPerConnection` defaults to 5 and is the only concurrency control applied: [7](#0-6) [8](#0-7) 

There is no IP-level connection cap, no global subscription ceiling, and no rate limiting in the gRPC layer.

**Root cause / failed assumption**: The design assumes the unthrottled path is rare (gap recovery only). In practice, any open-ended subscription to a topic with no recent messages always triggers the safety check and always enters the unthrottled path. The per-connection call limit does not bound the number of connections an attacker can open.

## Impact Explanation

Each concurrent subscription to a quiet topic generates 12 DB queries at 20 ms intervals (up to 5,000 rows/query). With 200 concurrent subscriptions (40 connections × 5 calls each), the attacker drives ~2,400 DB queries in a ~240 ms window (~10,000 queries/second). This saturates the DB connection pool and statement-execution capacity, causing query timeouts and latency spikes for all legitimate users. The `statementTimeout` of 10,000 ms means each query can hold a DB connection for up to 10 seconds under load, amplifying pool exhaustion.

## Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC port (5600) is publicly exposed. Opening 40 TCP connections with 5 streams each is trivially achievable with any gRPC client library in a single script. The attack is repeatable: after the 12-poll burst completes (~240 ms), the attacker simply re-subscribes to restart the cycle. No economic cost is incurred.

## Recommendation

1. **Global subscription cap**: Enforce a configurable maximum number of concurrent active subscriptions globally (and optionally per source IP) in `TopicMessageServiceImpl`, rejecting new subscriptions when the cap is reached.
2. **Throttle the safety-check path**: Pass `throttled=true` (or a dedicated "safety-check" mode with conservative parameters) to `topicMessageRetriever.retrieve` in the safety-check branch of `missingMessages`, rather than using the unthrottled path.
3. **IP-level connection limiting**: Configure Netty with a maximum number of connections per remote IP in `GrpcConfiguration` to prevent a single attacker from opening an unbounded number of TCP connections.
4. **Backpressure on unthrottled bursts**: Introduce a global semaphore or rate limiter guarding entry into the unthrottled `PollingContext` so that concurrent unthrottled retrievals are bounded regardless of subscription count.

## Proof of Concept

```python
import grpc
import threading
from mirror_node_pb2 import ConsensusTopicQuery
from mirror_node_pb2_grpc import ConsensusServiceStub

TARGET = "mirror-node-grpc:5600"
CONNECTIONS = 40
STREAMS_PER_CONN = 5  # matches maxConcurrentCallsPerConnection default

def attack_connection():
    channel = grpc.insecure_channel(TARGET)
    stub = ConsensusServiceStub(channel)
    threads = []
    for _ in range(STREAMS_PER_CONN):
        def subscribe():
            # Subscribe to a quiet/non-existent topic; no messages will arrive,
            # so the 1-second safety check always fires and triggers 12 unthrottled
            # DB queries at 20ms intervals (5000 rows/query each).
            query = ConsensusTopicQuery(topicID=..., consensusStartTime=...)
            try:
                for _ in stub.subscribeTopic(query):
                    pass
            except Exception:
                pass
        t = threading.Thread(target=subscribe)
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

attack_threads = [threading.Thread(target=attack_connection) for _ in range(CONNECTIONS)]
for t in attack_threads:
    t.start()
for t in attack_threads:
    t.join()
# Result: 200 concurrent subscriptions × 12 DB queries = 2,400 queries in ~240ms
```

    private Flux<TopicMessage> missingMessages(TopicContext topicContext, @Nullable TopicMessage current) {

                    topicContext.getFilter().toBuilder().startTime(startTime).build();
