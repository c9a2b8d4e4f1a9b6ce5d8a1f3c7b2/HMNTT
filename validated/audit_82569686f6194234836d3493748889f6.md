# Audit Report

## Title
Cache Poisoning via `Arrays.hashCode` Collision in SpEL Cache Key for EVM Address Lookups

## Summary
`SpelHelper.hashCode(byte[])` returns a 32-bit integer via `Arrays.hashCode`. This value is used as the **sole** Spring `@Cacheable` key for two EVM address lookup methods in `EntityRepository`. Two distinct byte arrays that produce the same 32-bit hash share the same cache slot, allowing a precomputed collision to cause one address's cached entity to be returned for a different address.

## Finding Description

**`SpelHelper.hashCode`** delegates directly to `Arrays.hashCode`: [1](#0-0) 

This 32-bit integer is used as the exclusive cache key in two repository methods: [2](#0-1) [3](#0-2) 

**Root cause:** `Arrays.hashCode` over a `byte[]` of length N computes a degree-N polynomial over a 32-bit ring (multiplier 31). The output space is 2^32. For 20-byte EVM addresses, finding two distinct inputs A and B such that `Arrays.hashCode(A) == Arrays.hashCode(B)` is analytically trivial — the hash is a public, linear, deterministic function. The cache key is assumed to uniquely identify the input byte array; it does not.

**Failed guard:** The `unless = "#result == null"` condition only prevents caching null results. It performs no collision detection and no secondary key verification. [4](#0-3) 

**Cache configuration:** The entity cache manager registers both `evmAddress` and `alias` caches with a default spec of `expireAfterWrite=1s,maximumSize=10000`: [5](#0-4) [6](#0-5) 

## Impact Explanation
A caller of the web3 JSON-RPC API (no authentication required) who precomputes a collision pair (address A → real entity E_A; address B → hash collision with A) can:
1. Continuously send requests for A, keeping the cache slot populated with E_A.
2. Any concurrent or subsequent request for B within the same 1-second TTL window receives E_A instead of the correct entity for B.

In the context of `eth_call`, `eth_getBalance`, or contract simulation, this produces silently wrong execution results — wrong account balances, wrong contract bytecode, wrong key material — for the colliding address. Because the attacker can repeat the priming request every second, the poisoned state can be maintained continuously for as long as the attacker sends traffic.

## Likelihood Explanation
`Arrays.hashCode` collision pairs for 20-byte arrays are trivially computable offline: the hash is a public, deterministic, linear function over Z/2^32Z. No privileges, no special network position, and no authentication are required. The only precondition is knowing one valid EVM address in the system, which is public on-chain data. The 1-second TTL narrows the per-entry window but does not prevent the attack, since the attacker can continuously re-prime the cache. The attack is repeatable and scriptable.

## Recommendation
Replace the lossy 32-bit hash with a collision-resistant cache key that encodes the full byte array content. Options:

1. Use hex encoding in the SpEL expression:
   ```java
   key = "T(java.util.HexFormat).of().formatHex(#alias)"
   ```
2. Add a `toHex(byte[])` method to `SpelHelper` that returns a hex `String`, and use `@spelHelper.toHex(#alias)` as the key.
3. Use `T(java.util.Arrays).toString(#alias)` (produces a decimal representation, unique per content).

Any of these produces a `String` key that is unique per distinct byte array content, eliminating the collision surface entirely. [1](#0-0) 

## Proof of Concept

```java
// Offline collision search (pseudocode)
// Find two distinct 20-byte arrays A and B where Arrays.hashCode(A) == Arrays.hashCode(B)
// Arrays.hashCode for byte[]: result = 1; for each b: result = 31*result + b
// This is a linear equation over Z/2^32Z — solvable analytically or by birthday search.

byte[] A = /* known valid EVM address with entity E_A in the system */;
byte[] B = /* precomputed collision: Arrays.hashCode(B) == Arrays.hashCode(A), B != A */;

// Step 1: Prime the cache
httpClient.post("/api/v1/contracts/call", body(A));  // caches E_A under key hashCode(A)

// Step 2: Query colliding address — returns E_A instead of correct entity for B
httpClient.post("/api/v1/contracts/call", body(B));  // cache hit: returns E_A

// Step 3: Repeat step 1 every <1 second to maintain poisoned state continuously
```

The attacker needs no credentials. The collision is computed entirely offline before any network interaction.
