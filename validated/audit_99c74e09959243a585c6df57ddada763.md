# Audit Report

## Title
Rate-Limit Token Permanently Consumed on Gas-Check Failure Enables DoS of `rateLimitBucket`

## Summary
In `ThrottleManagerImpl.throttle()`, a rate-limit token is consumed from `rateLimitBucket` before the gas check against `gasLimitBucket`. When the gas check fails, a `ThrottleException` is thrown but the already-consumed rate-limit token is never restored. An attacker can craft requests with a `gas` value that permanently exceeds `gasLimitBucket`'s maximum capacity, causing every request to consume a rate-limit token while consuming zero gas tokens, rapidly exhausting the `rateLimitBucket` and denying service to legitimate users.

## Finding Description

**Vulnerable code — `ThrottleManagerImpl.throttle()`:**

```java
// ThrottleManagerImpl.java lines 38–42
if (!rateLimitBucket.tryConsume(1)) {              // token consumed here
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(             // gas check second
        throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED); // rate token NOT restored
}
``` [1](#0-0) 

**Root cause:** The design assumes any request passing the rate-limit check will also pass the gas check. There is no rollback of the rate-limit token when the gas check fails.

**`restore()` does not help:** `restore()` only adds tokens back to `gasLimitBucket`:

```java
// ThrottleManagerImpl.java lines 59–64
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);  // rateLimitBucket never touched
    }
}
``` [2](#0-1) 

Furthermore, `restore()` is only called in the `InvalidParametersException` catch block in `ContractController`, not for `ThrottleException`:

```java
// ContractController.java lines 46–49
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());
    throw e;
}
``` [3](#0-2) 

**`validateContractMaxGasLimit` is bypassed:** This check runs *after* `throttleManager.throttle(request)` at line 40–41, so it cannot prevent the rate-limit token from being consumed before the gas check fails. [8](#0-7) 

**No `@Max` on `gas`:** `ContractCallRequest.gas` only has `@Min(21_000)`, so arbitrarily large values pass Bean Validation: [4](#0-3) 

**Bucket capacities (defaults):**
- `rateLimitBucket`: capacity = `requestsPerSecond` = **500 tokens**, refills 500/second. [9](#0-8) 
- `gasLimitBucket`: capacity = `scaleGas(7_500_000_000)` = `7_500_000_000 / 10_000` = **750,000 tokens**, refills 750,000/second. [10](#0-9) 

**Why `tryConsume(751_000)` always fails:** Bucket4j's `tryConsume(n)` returns `false` when `n` exceeds the bucket's maximum capacity, regardless of current fill level. A gas value of `7_510_000_000` scales to `751_000`, which permanently exceeds the `750_000` capacity — this request can never succeed.

## Impact Explanation
Any unauthenticated user can continuously deny service to all legitimate callers of `/api/v1/contracts/call`. With default settings (500 req/s limit), the attacker needs only 500 HTTP requests per second — trivially achievable — to keep `rateLimitBucket` at zero indefinitely. Legitimate users receive HTTP 429 errors for all contract call and estimate requests. The `gasLimitBucket` is entirely unaffected, meaning the attacker wastes no gas budget while executing the attack.

## Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge of deployed contracts. The only requirement is the ability to send HTTP POST requests with a crafted `gas` field. The gas value needed (`> 7_500_000_000` with defaults) is a valid `long` that passes Bean Validation (`@Min(21_000)` only). The attack is fully repeatable, automatable with a simple script, and sustainable indefinitely. Likelihood is **high**.

## Recommendation
Apply one or more of the following fixes:

1. **Reorder the checks:** Perform the gas check before the rate-limit check in `throttle()`. If the gas check fails first, no rate-limit token is consumed.
2. **Restore the rate-limit token on gas failure:** Add `rateLimitBucket.addTokens(1)` in the gas-check failure branch, or extend `restore()` to also restore the rate-limit token and call it from a broader catch block.
3. **Add `@Max` on `ContractCallRequest.gas`:** Constrain `gas` to at most `gasPerSecond` (e.g., `7_500_000_000`) via Bean Validation so that requests with gas exceeding the bucket capacity are rejected before `throttle()` is ever called.
4. **Move `validateContractMaxGasLimit` before `throttle()`:** This would reject oversized gas values before any tokens are consumed.

The cleanest fix is option 1 (reorder checks) combined with option 3 (add `@Max`), which eliminates the race condition entirely and prevents the crafted input from reaching the throttle logic.

## Proof of Concept

```bash
# Drain rateLimitBucket (500 tokens) in under 1 second
# gas = 7_510_000_000 → scaleGas = 751_000 > gasLimitBucket capacity (750_000)
# → rateLimitBucket.tryConsume(1) succeeds (token consumed)
# → gasLimitBucket.tryConsume(751_000) always fails (exceeds max capacity)
# → ThrottleException thrown, rate token permanently lost

for i in $(seq 1 500); do
  curl -s -o /dev/null -X POST http://<host>/api/v1/contracts/call \
    -H "Content-Type: application/json" \
    -d '{"to":"0x0000000000000000000000000000000000000001","gas":7510000000}' &
done
wait

# All subsequent legitimate requests now receive HTTP 429:
# "Requests per second rate limit exceeded"
# Repeat every second to sustain indefinitely.
```
