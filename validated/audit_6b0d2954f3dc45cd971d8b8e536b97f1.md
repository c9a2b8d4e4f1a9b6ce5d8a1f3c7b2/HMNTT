Audit Report

## Title
Integer Overflow in `getInclusiveValue()` Causes `gt:Long.MAX_VALUE` to Return All Registered Nodes

## Summary
`NumberRangeParameter.getInclusiveValue()` performs an unchecked `value + 1` when the operator is `GT`. Supplying `registerednode.id=gt:9223372036854775807` causes a silent Java integer overflow to `Long.MIN_VALUE`. `Math.max(0L, Long.MIN_VALUE)` resets `lowerBound` to `0`, the bounds guard passes, and the repository query executes over the full range `[0, Long.MAX_VALUE]`, returning every registered node instead of an empty result.

## Finding Description

**Step 1 — Input accepted without overflow guard**

`getNumberValue` only rejects negative values. `Long.MAX_VALUE` (9223372036854775807) is ≥ 0 and passes: [1](#0-0) 

The test suite at `NetworkControllerTest.java:1964` only rejects `"9223372036854775808"` (which overflows `Long.parseLong` itself), not `"gt:9223372036854775807"`: [2](#0-1) 

**Step 2 — Silent overflow in `getInclusiveValue()`**

```java
// Long.MAX_VALUE + 1 = Long.MIN_VALUE in Java (no exception)
return value + 1;
``` [3](#0-2) 

**Step 3 — `Math.max` absorbs the overflow, guard never fires**

```java
lowerBound = Math.max(lowerBound, range.getInclusiveValue());
// Math.max(0L, Long.MIN_VALUE) = 0L
```

`lowerBound` stays `0L`, so `lowerBound > upperBound` (`0 > Long.MAX_VALUE`) is `false` and no exception is thrown: [4](#0-3) 

**Step 4 — Repository query returns all rows**

The resolved range `[0, Long.MAX_VALUE]` is passed directly to the native query: [5](#0-4) 

```sql
select * from registered_node
where registered_node_id >= 0
  and registered_node_id <= 9223372036854775807
  and deleted is false
  ...
``` [6](#0-5) 

This matches every row in the table.

## Impact Explanation
A semantically empty query (`gt:Long.MAX_VALUE` — no node ID can ever exceed `Long.MAX_VALUE`) is silently rewritten into a full-table scan. The caller receives all registered nodes instead of an empty list. Depending on what `registered_node` contains (admin keys, service endpoints, internal topology), this is an information-disclosure vulnerability. It also breaks any caller logic that relies on an empty response to mean "no such node exists."

## Likelihood Explanation
No authentication or special privilege is required. The endpoint `GET /api/v1/network/registered-nodes` is publicly accessible. The payload is a single, well-formed query parameter using the documented `Long.MAX_VALUE` constant (9223372036854775807). The attack is deterministic, repeatable, and leaves no side effects.

## Recommendation
Replace the bare arithmetic in `getInclusiveValue()` with `Math.addExact`, which throws `ArithmeticException` on overflow:

```java
// NumberRangeParameter.java
public long getInclusiveValue() {
    if (operator == RangeOperator.GT) {
        return Math.addExact(value, 1);  // throws on Long.MAX_VALUE
    } else if (operator == RangeOperator.LT) {
        return Math.subtractExact(value, 1);  // symmetric fix for LT:0
    } else {
        return value;
    }
}
```

The same fix should be applied to the `LT` branch to prevent an analogous underflow when `value = 0`. The thrown `ArithmeticException` should be caught at the parameter-parsing layer and translated into a `400 Bad Request`.

## Proof of Concept

```
GET /api/v1/network/registered-nodes?registerednode.id=gt:9223372036854775807
```

**Expected:** HTTP 200 with an empty `registeredNodes` array (no node ID can exceed `Long.MAX_VALUE`).

**Actual:** HTTP 200 with all registered nodes in the database returned, up to the configured pagination limit.
