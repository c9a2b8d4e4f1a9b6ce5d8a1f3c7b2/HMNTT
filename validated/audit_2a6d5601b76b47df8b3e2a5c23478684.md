Audit Report

## Title
32-bit Hash Collision in `CACHE_NAME_ALIAS` Cache Key Enables Cache Poisoning via Unprivileged Alias Lookup

## Summary
`findByEvmAddressOrAliasAndDeletedIsFalse` uses `@spelHelper.hashCode(#alias)` — which resolves to `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole cache key in `CACHE_NAME_ALIAS`. Two distinct byte arrays can produce the same 32-bit key. An attacker who registers a Hedera account whose alias hash collides with a target entity's hash can poison the cache so that subsequent lookups for the target return the attacker's entity instead.

## Finding Description

**`SpelHelper.hashCode` returns a Java `int` (32 bits):** [1](#0-0) 

**This `int` is used verbatim as the cache key for `CACHE_NAME_ALIAS`:** [2](#0-1) 

**`CACHE_MANAGER_ENTITY` registers `CACHE_NAME_ALIAS` as one of its named caches:** [3](#0-2) 

**The entity cache TTL is 1 second:** [4](#0-3) 

**The cached result feeds directly into alias resolution in `AliasesReadableKVState` and `AccountReadableKVState`:** [7](#0-6) [8](#0-7) 

**Root cause:** `Arrays.hashCode(byte[])` is a non-cryptographic 32-bit polynomial hash (`result = 31*result + b`). Its output space is only ~4 billion values. Two distinct byte arrays `A` and `B` can satisfy `Arrays.hashCode(A) == Arrays.hashCode(B)`. Spring `@Cacheable` stores the result under the integer key with no secondary equality check on the actual byte array, so a collision causes the wrong entity to be returned on a cache hit.

**Failed assumption:** The design assumes that `Arrays.hashCode` produces a unique key per distinct alias byte array. This is false for a 32-bit hash over variable-length byte arrays.

**Why existing checks are insufficient:**
- `unless = "#result == null"`: `Optional.of(attackerEntity)` is not null, so the attacker's result is unconditionally cached. `Optional.empty()` is also not null, so even a non-existent alias poisons the cache with an empty result. [5](#0-4) 
- There is no post-cache verification that the returned entity's actual alias/evm_address matches the queried byte array.
- The cache has no collision-detection mechanism (e.g., storing the original key alongside the value).

## Impact Explanation
When the wrong entity is returned for a fee-collecting account lookup, the EVM execution layer uses the attacker's account data to compute and route fees. This causes fees to be undercharged or misdirected outside of design parameters. The impact is bounded by the 1-second TTL — the attacker must continuously re-poison — but within each poisoning window every request that resolves the target alias is affected. The `feeCollectionAccount`, `networkAdminFeeAccount`, `nodeRewardAccount`, and `stakingRewardAccount` are all resolved through this path. [9](#0-8) 

## Likelihood Explanation
Creating a Hedera account is permissionless (costs only HBAR). For both ED25519 aliases and EVM addresses, finding a colliding byte array requires brute-forcing ~2³² hash evaluations (the report's claim of a single arithmetic step for ED25519 is incorrect because alias bytes are constrained by cryptographic key generation, not freely chosen). ~2³² attempts is feasible for a motivated attacker with commodity hardware. Sustaining the attack requires sending one request per second, which is trivially automated.

## Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with the full byte array itself (or a cryptographic digest such as SHA-256) so that the cache key uniquely identifies the alias:

```java
// Instead of:
key = "@spelHelper.hashCode(#alias)"

// Use the byte array directly (Caffeine handles byte[] keys correctly via Arrays.equals/hashCode internally):
// Remove the key attribute entirely and let Spring use the default SimpleKey,
// or use a hex-encoded string key:
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"
```

Alternatively, store the original alias alongside the cached value and verify equality on cache retrieval, or use a `Cache` implementation that performs full key equality checks on the raw byte array.

## Proof of Concept
```java
// Target entity has alias A with Arrays.hashCode(A) == T
byte[] A = targetFeeCollectionAccount.getAlias();
int T = Arrays.hashCode(A);

// Attacker brute-forces a byte array B with Arrays.hashCode(B) == T
// For a 32-bit hash, expected ~2^32 attempts
byte[] B = null;
Random rng = new SecureRandom();
byte[] candidate = new byte[A.length];
while (true) {
    rng.nextBytes(candidate);
    if (Arrays.hashCode(candidate) == T) {
        B = candidate.clone();
        break;
    }
}

// Attacker registers a Hedera account with alias B (permissionless)
// Then sends: eth_call triggering findByEvmAddressOrAliasAndDeletedIsFalse(B)
// Cache stores attacker's entity under key T
// Next legitimate lookup for alias A hits cache under key T → returns attacker's entity
```
