Audit Report

## Title
`BufferUnderflowException` via Under-Length Hex `evmAddress` in `getByEvmAddressAndType()`

## Summary
`decodeEvmAddress()` performs no length validation on the decoded byte array. `getByEvmAddressAndType()` unconditionally wraps the result in a `ByteBuffer` and reads 4 + 8 + 8 = 20 bytes. A crafted hex input whose first 4 bytes are zero but whose total length is between 5 and 11 bytes causes `buffer.getLong()` to throw an unchecked `BufferUnderflowException`. `CustomExceptionResolver` does not handle this exception type, so Spring GraphQL falls back to a generic `INTERNAL_ERROR` response.

## Finding Description

**`GraphQlUtils.decodeEvmAddress()`** — `graphql/src/main/java/org/hiero/mirror/graphql/util/GraphQlUtils.java`, lines 87–98:

Only hex-format validity is checked via `Hex.decodeHex()`. Any even-length hex string of any byte length is returned without a length assertion. [1](#0-0) 

**`EntityServiceImpl.getByEvmAddressAndType()`** — `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41:

The method unconditionally wraps the decoded bytes in a `ByteBuffer` and performs three sequential reads totalling 20 bytes, with no prior length guard:
- `buffer.getInt()` — consumes bytes 0–3 (requires ≥ 4 bytes)
- first `buffer.getLong()` — consumes bytes 4–11 (requires ≥ 12 bytes total)
- second `buffer.getLong()` — consumes bytes 12–19 (requires ≥ 20 bytes total)

Due to Java's short-circuit `&&`, the first `getLong()` is only reached when `getInt() == 0`. An input whose first 4 bytes are `0x00000000` but whose total length is 5–11 bytes passes the `getInt()` check and then throws `BufferUnderflowException` on the `getLong()` call. [2](#0-1) 

**`CustomExceptionResolver`** — `graphql/src/main/java/org/hiero/mirror/graphql/config/CustomExceptionResolver.java`, lines 16–29:

Only `IllegalStateException`, `IllegalArgumentException`, and `MirrorNodeException` are mapped to a `ValidationError` response. `BufferUnderflowException` (which extends `RuntimeException` directly) is not listed; `resolveToSingleError` returns `null`, causing Spring GraphQL's default handler to emit a generic `INTERNAL_ERROR`. [3](#0-2) 

## Impact Explanation
Every request carrying a crafted short `evmAddress` triggers an unhandled `BufferUnderflowException` that bypasses the application's own error-handling layer. The caller receives an opaque `INTERNAL_ERROR` GraphQL response, leaking the fact that an unhandled exception occurred. At scale, flooding this endpoint with such requests produces a continuous stream of unhandled exceptions logged at ERROR level, polluting observability infrastructure and potentially masking legitimate errors. Note: thread exhaustion is not a consequence — Spring GraphQL's framework-level handler catches the exception and completes the request thread normally.

## Likelihood Explanation
No authentication is required. The GraphQL endpoint is publicly reachable. The crafted input is trivially constructed: any even-length hex string starting with `00000000` and shorter than 24 hex characters reliably triggers the bug. It is repeatable, requires no special knowledge, and can be automated.

## Recommendation
Add an explicit length check in `decodeEvmAddress()` or at the top of `getByEvmAddressAndType()` before wrapping in a `ByteBuffer`:

```java
if (evmAddressBytes.length != 20) {
    throw new IllegalArgumentException("evmAddress must be exactly 20 bytes");
}
```

This converts the `BufferUnderflowException` path into an `IllegalArgumentException`, which `CustomExceptionResolver` already handles and maps to a `ValidationError` response. [4](#0-3) 

Alternatively, add `BufferUnderflowException` to the handled types in `CustomExceptionResolver` as a defence-in-depth measure. [5](#0-4) 

## Proof of Concept
Send the following GraphQL query (adjust the field/type name to match the schema):

```graphql
{
  account(input: { evmAddress: "0000000001" }) {
    id
  }
}
```

- `"0000000001"` decodes to 5 bytes: `[0x00, 0x00, 0x00, 0x00, 0x01]`
- `buffer.getInt()` reads bytes 0–3 → returns `0` → short-circuit condition is true
- `buffer.getLong()` attempts to read bytes 4–11 → only 1 byte remains → throws `java.nio.BufferUnderflowException`
- `CustomExceptionResolver.resolveToSingleError` returns `null`
- Response: `{"errors":[{"message":"INTERNAL_ERROR","extensions":{"classification":"INTERNAL_ERROR"}}]}`

    public static byte[] decodeEvmAddress(String evmAddress) {
        if (evmAddress == null) {
            return ArrayUtils.EMPTY_BYTE_ARRAY;
        }

        try {
            evmAddress = Strings.CS.removeStart(evmAddress, HEX_PREFIX);
            return Hex.decodeHex(evmAddress);

            throw new IllegalArgumentException("Unable to decode evmAddress: " + evmAddress);
