# Audit Report

## Title
Unbounded Timestamp Range in `getHookStorage()` Enables Full Historical Table Scan DoS

## Summary
`HooksController.getHookStorage()` accepts user-supplied `timestamp` parameters with no range-width validation. `Bound.of()` is called with `primarySortField=false`, which bypasses the only inversion guard. The resulting `[0, Long.MAX_VALUE]` range is passed directly to a `DISTINCT ON (key)` SQL query against `hook_storage_change`. No rate limiting exists in the `rest-java` module, making this endpoint trivially abusable by any unauthenticated caller.

## Finding Description

**Step 1 — Controller accepts timestamps with no range-width check.**

`getHookStorage()` accepts `@Size(max = 2) TimestampParameter[] timestamps`. The only constraint is that at most 2 values are supplied; no minimum/maximum span is enforced. [1](#0-0) 

**Step 2 — `Bound.of()` is called with `primarySortField=false`.**

`hookStorageChangeRequest()` at line 186 calls `Bound.of(timestamps, TIMESTAMP, ...)` which hard-codes `false` as the `primarySortField` argument on line 181. [2](#0-1) [3](#0-2) 

**Step 3 — The only range-inversion guard is dead code for this call site.**

The constructor's guard `if (primarySortField && adjustedLower > adjustedUpper)` is never evaluated when `primarySortField=false`, so no `IllegalArgumentException` is ever thrown for an inverted or unbounded timestamp range. [4](#0-3) 

**Step 4 — Missing upper bound resolves to `Long.MAX_VALUE`.**

When only a lower bound is supplied (e.g., `timestamp=gte:0`), `adjustUpperBound()` returns `Long.MAX_VALUE` because `this.upper == null`. [5](#0-4) 

**Step 5 — Unbounded values flow directly into the repository.**

`HookServiceImpl.getHookStorageChange()` extracts `timestampLowerBound` and `timestampUpperBound` from the `Bound` object without any additional validation and passes them to the repository. [6](#0-5) 

**Step 6 — `DISTINCT ON (key)` query scans the full timestamp range.**

Both repository methods execute a `DISTINCT ON (key)` native query filtering on `consensus_timestamp between :timestampLowerBound and :timestampUpperBound`. PostgreSQL must evaluate all rows in the range to determine the latest `consensus_timestamp` per key before applying `LIMIT`. [7](#0-6) 

**Step 7 — No rate limiting in `rest-java`.**

A grep for `RateLimiter`, `ThrottleManager`, and `RateLimit` across the entire `rest-java` module returns zero matches. The `ThrottleConfiguration` infrastructure exists only in the `web3` module. [8](#0-7) 

**Contrast with the Node.js REST module**, which enforces `maxTimestampRange` / `maxTimestampRangeNs` and rejects overly wide ranges. No equivalent exists in the `rest-java` path. [9](#0-8) 

## Impact Explanation
Each request with `timestamp=gte:0` causes PostgreSQL to scan every row in `hook_storage_change` for the given `(owner_id, hook_id)` pair across the full timestamp domain `[0, Long.MAX_VALUE]`. As the table grows with network history, each such query consumes proportionally more DB CPU, I/O, and memory. Concurrent flooding from multiple source IPs can saturate DB connection pools and query workers, degrading query processing across all services sharing the same DB cluster.

## Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. A valid `ownerId` and `hookId` are discoverable via `GET /api/v1/accounts/{ownerId}/hooks`. No tokens, credentials, or cryptographic material are required. The attack is trivially scriptable with standard HTTP tooling (e.g., `curl` in a loop).

## Recommendation
1. **Enforce a maximum timestamp range width** in `hookStorageChangeRequest()` or in `Bound.of()` when called for the timestamp field. Reject requests where `adjustedUpper - adjustedLower` exceeds a configurable `maxTimestampRangeNs` (mirroring the Node.js REST module's existing guard).
2. **Require an upper bound** when a lower bound is supplied for the timestamp parameter on this endpoint, or default the upper bound to `lower + maxTimestampRangeNs` rather than `Long.MAX_VALUE`.
3. **Add rate limiting** to the `rest-java` module (e.g., via a servlet filter or Spring `HandlerInterceptor`) analogous to the `ThrottleConfiguration` in `web3`.

## Proof of Concept
```
# Trigger a full cross-partition scan for a known owner/hook pair
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=100

# Flood concurrently
for i in $(seq 1 200); do
  curl -s "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=gte:0&limit=100" &
done
```

With `timestampLowerBound=0` and `timestampUpperBound=Long.MAX_VALUE` (9223372036854775807), the resulting SQL is:

```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = 1234
  AND hook_id = 1
  AND key >= '\x0000...'
  AND key <= '\xffff...'
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
ORDER BY key ASC, consensus_timestamp DESC
LIMIT 100;
```

This forces a full scan of all historical rows for the targeted hook before the `LIMIT` can be applied.
