Audit Report

## Title
Request Timeout Bypass via Pre-Execution-Only `statementInspector` Check Allowing Multi-Query Overshoot

## Summary
The `statementInspector` in `HibernateConfiguration.java` enforces `requestTimeout` only as a pre-flight gate before each SQL statement. Once a query is dispatched, no application-layer mechanism can interrupt it. Combined with `BinaryGasEstimator.safeCall()` silently swallowing `QueryTimeoutException`, a single `eth_estimateGas` request can hold a server thread for up to `requestTimeout + (N × statementTimeout)` milliseconds — far beyond the intended 10-second ceiling.

## Finding Description

**Root cause:** The `statementInspector` lambda in `HibernateConfiguration.java` (lines 31–46) checks elapsed time before each SQL statement and throws `QueryTimeoutException` only if the deadline has already passed. Once `return sql` is reached, the query executes with no further wall-clock enforcement from the application layer. [9](#0-8) 

`startTime` is set once at `ContractCallContext` construction and is never reset between binary-search iterations, so the elapsed-time calculation is correct in intent but the guard fires only at the *start* of each new query. [2](#0-1) 

**Amplifier — `safeCall` swallows `QueryTimeoutException`:** `BinaryGasEstimator.safeCall()` catches `Exception` broadly and returns `null`, treating a timed-out iteration as a failed gas estimate and allowing the loop to continue for up to `maxGasEstimateRetriesCount = 20` more iterations. [4](#0-3) [3](#0-2) 

**Concrete timeline** (`requestTimeout = 10 000 ms`, DB `statement_timeout = 10 000 ms`):
- t = 0 ms: inspector fires (elapsed = 0 < 10 000), Query-1 starts, runs 9 900 ms.
- t = 9 900 ms: Query-1 completes. Inspector fires for Query-2 (elapsed = 9 900 < 10 000), Query-2 starts, runs 9 900 ms.
- t = 19 800 ms: Query-2 completes. Inspector fires (elapsed = 19 800 ≥ 10 000) → `QueryTimeoutException` thrown — but `safeCall` catches it, loop continues.
- Total wall time per iteration pair ≈ 19.8 s; with up to 20 iterations the theoretical ceiling is `requestTimeout + 20 × statementTimeout`.

**Why existing checks are insufficient:**
- The `statementInspector` is the only application-level wall-clock guard; there is no `Thread.interrupt()`, reactive timeout, or Servlet async timeout configured for the web3 service.
- The DB-level `statement_timeout` (3 000 ms application default, 10 000 ms Helm) caps individual queries, not the aggregate request.
- `safeCall` swallows `QueryTimeoutException`, so a timed-out iteration is treated as a failed gas estimate and the loop continues. [10](#0-9) 

## Impact Explanation
A single malicious `eth_estimateGas` request can hold a server thread for significantly longer than the intended 10-second ceiling. With the default Helm `statement_timeout = 10 000 ms` and `maxGasEstimateRetriesCount = 20`, the theoretical maximum per-request wall time is `10 s + 20 × 10 s = 210 s`. Even with the application-default 3 000 ms `statementTimeout`, the overshoot is `10 s + 20 × 3 s = 70 s`. Repeated at the allowed 500 RPS throttle, this exhausts the thread pool and causes denial of service. Additionally, for `LATEST`-block calls, successive queries within one logical request execute against different PostgreSQL MVCC snapshots (no explicit transaction wraps the full request), producing reads across inconsistent database states. [7](#0-6) [6](#0-5) 

## Likelihood Explanation
No authentication is required; `eth_estimateGas` is a public endpoint. The attacker needs only a deployed contract with many SLOAD operations or token precompile calls to slow individual queries. The binary-search loop (up to 20 iterations) is a built-in amplifier, and `safeCall` ensures the loop never terminates early on timeout. The attack is repeatable and scriptable. [8](#0-7) 

## Recommendation
1. **Propagate `QueryTimeoutException` out of `safeCall`:** Re-throw `QueryTimeoutException` (or any `Spring` `QueryTimeoutException`) before the generic `catch (Exception ignored)` block so that a timed-out request terminates the binary-search loop immediately.
2. **Add an aggregate request deadline:** Wrap the entire `ContractCallContext.run(...)` scope with a hard wall-clock deadline (e.g., a `ScheduledExecutorService` that interrupts the executing thread, or a Spring MVC async timeout) so that no combination of queries can exceed `requestTimeout` regardless of individual query durations.
3. **Align `statementTimeout` with `requestTimeout`:** Ensure the DB-level `statement_timeout` is strictly less than `requestTimeout` so that a single slow query cannot itself consume the entire request budget.

## Proof of Concept
1. Deploy a contract on a local fork that performs a large number of SLOAD operations (e.g., reads 500 distinct storage slots in a loop).
2. Submit `eth_estimateGas` targeting that contract with `gas = 15_000_000`.
3. Observe via thread-dump or metrics that the handling thread remains active well beyond 10 seconds (up to the `requestTimeout + N × statementTimeout` ceiling).
4. Confirm that `BinaryGasEstimator.safeCall` logs `"Exception while calling contract for gas estimation"` on each timed-out iteration and that the loop continues rather than propagating the exception. [4](#0-3) [1](#0-0)

    StatementInspector statementInspector() {
        long timeout = web3Properties.getRequestTimeout().toMillis();
        return sql -> {
            if (!ContractCallContext.isInitialized()) {
                return sql;
            }

            var startTime = ContractCallContext.get().getStartTime();
            long elapsed = System.currentTimeMillis() - startTime;

            if (elapsed >= timeout) {
                throw new QueryTimeoutException("Transaction timed out after %s ms".formatted(elapsed));
            }

            return sql;
        };
