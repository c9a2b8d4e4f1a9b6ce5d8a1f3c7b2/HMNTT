Audit Report

## Title
Missing `deleted is not true` Filter in `entity_history` Sub-query of `findActiveByEvmAddressOrAliasAndTimestamp` Returns Soft-Deleted Entities as Active in Historical Queries

## Summary
The `entity_history` branch of the UNION query inside `findActiveByEvmAddressOrAliasAndTimestamp()` omits the `deleted is not true` predicate present in the sibling `entity` branch and in the ID-based peer method `findActiveByIdAndTimestamp()`. Any caller submitting a historical query using the alias or `evm_address` of a soft-deleted entity with a `blockTimestamp` at or after the deletion timestamp receives the deleted entity record as if it were active. The same defect exists in `findActiveByEvmAddressAndTimestamp()`.

## Finding Description

**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`

In `findActiveByEvmAddressOrAliasAndTimestamp` (lines 95â€“121), the `entity` branch correctly carries the guard:

```sql
where e.deleted is not true
and e.id = (select id from entity_cte)
``` [1](#0-0) 

The `entity_history` branch immediately below it has no such guard:

```sql
where lower(eh.timestamp_range) <= ?2
and eh.id = (select id from entity_cte)
order by lower(eh.timestamp_range) desc
limit 1
``` [2](#0-1) 

By contrast, the ID-based peer `findActiveByIdAndTimestamp()` applies `deleted is not true` to **both** branches: [3](#0-2) 

The same omission exists in `findActiveByEvmAddressAndTimestamp()`: [4](#0-3) 

**Failed assumption:** The query assumes `ORDER BY timestamp_range DESC LIMIT 1` will always prefer a non-deleted record. It does not. When the entity is currently deleted, the `entity` branch returns zero rows (correctly filtered), leaving only the `entity_history` branch. The most recent `entity_history` row at or after the deletion timestamp is the deletion record itself (`deleted = true`), which wins the final `LIMIT 1` and is returned unfiltered.

**Exploit flow:**
1. Entity A (alias `X`, `evm_address` `E`) is created at `T1`. The `entity` table holds the live record.
2. Entity A is deleted at `T2 > T1`. The importer writes a row to `entity_history` with `deleted = true` and `timestamp_range = [T2, ...)`, and sets `deleted = true` in the `entity` table.
3. An unprivileged caller issues a historical `eth_call` with `alias = X` and `blockTimestamp = T2` (or any value â‰Ą T2).
4. Query execution:
   - `entity_cte`: finds entity A (no `deleted` filter in CTE) â†’ returns entity ID.
   - `entity` branch: `deleted is not true` filters out the deleted row â†’ **zero rows**.
   - `entity_history` branch: `lower(timestamp_range) = T2 â‰¤ T2` matches the deletion record â†’ **one row returned, unfiltered**.
   - `UNION ALL + ORDER BY + LIMIT 1` â†’ the deletion record is the only candidate â†’ **returned**.
5. `CommonEntityAccessor.get(Bytes alias, Optional<Long> timestamp)` passes the result straight through with no `deleted` check: [5](#0-4) 

**Why existing tests are insufficient:** The two tests covering the deleted-history case (`findHistoricalEntityByEvmAddressOrAliasAndTimestampRangeAndDeletedTrueCallWithAlias` and its `EvmAddress` variant) only assert emptiness at `entityHistory.getTimestampLower() - 1` â€” i.e., strictly *before* the deletion record's lower bound. There is no test for `getTimestampLower()` or `getTimestampLower() + 1`, which are the timestamps that trigger the bug. The tests pass today precisely because they avoid the vulnerable timestamp range. [6](#0-5) 

## Impact Explanation
Any historical query resolved by alias or `evm_address` at a timestamp â‰Ą the entity's deletion time returns the deleted entity as active. Downstream consumers (indexers, wallets, dApps, other mirror nodes) receive incorrect historical state: a deleted contract or account appears to exist, potentially with stale balance, key, or type fields. This directly violates the data integrity guarantee for historical state exported by the mirror node.

## Likelihood Explanation
Preconditions are trivially met by any unprivileged external user. The alias and `evm_address` of deleted entities are permanently visible on-chain and via mirror node REST APIs. Historical `eth_call` (specifying a block number) is a standard, unauthenticated RPC endpoint. No keys, credentials, or internal access are required. The condition is deterministic and repeatable for every deleted entity in the network's history.

## Recommendation
Add `and eh.deleted is not true` to the `entity_history` branch of both affected queries, mirroring the pattern already used in `findActiveByIdAndTimestamp()`:

```sql
(
    select *
    from entity_history eh
    where lower(eh.timestamp_range) <= ?2
    and eh.id = (select id from entity_cte)
    and eh.deleted is not true          -- add this line
    order by lower(eh.timestamp_range) desc
    limit 1
)
```

Apply the same fix to `findActiveByEvmAddressAndTimestamp()`. Add regression tests asserting that `findActiveByEvmAddressOrAliasAndTimestamp` returns empty at `entityHistory.getTimestampLower()` and `getTimestampLower() + 1` when the history record has `deleted = true`.

## Proof of Concept
1. Insert an entity row with `alias = X`, `evm_address = E`, `created_timestamp = 100`, `deleted = false`.
2. Insert an `entity_history` row for the same `id` with `timestamp_range = [200, )`, `deleted = true`.
3. Update the `entity` row to `deleted = true`.
4. Call `entityRepository.findActiveByEvmAddressOrAliasAndTimestamp(X, 200)`.
5. **Expected:** `Optional.empty()` (entity is deleted).
6. **Actual:** `Optional` containing the `entity_history` row with `deleted = true` â€” the deleted entity is returned as active.
