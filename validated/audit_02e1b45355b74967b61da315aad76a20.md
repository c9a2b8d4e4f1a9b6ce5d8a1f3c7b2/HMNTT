Audit Report

## Title
Integer Overflow in `poll()` Bypasses DB Query LIMIT via Negative `pageSize`

## Summary
A narrowing cast from `long` to `int` in `PollingTopicMessageRetriever.poll()` (and identically in `PollingTopicListener.poll()`) converts a user-supplied limit value of `2147483648L` (2³¹) to `Integer.MIN_VALUE`. `Math.min` then selects the negative value as the page size. The resulting `TopicMessageFilter` has a negative `limit`, causing `hasLimit()` to return `false` and `findByFilter()` to omit `setMaxResults` entirely, issuing an unbounded SQL query that loads the full topic message table partition into JVM heap memory.

## Finding Description

**Step 1 — User input reaches `TopicMessageFilter.limit` as a positive `long`.**

`ConsensusController.toFilter()` reads the protobuf `uint64 limit` field directly:

```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

The value `2147483648L` (2³¹) is a valid positive `long` and passes the `@Min(0)` constraint on `TopicMessageFilter.limit`. [2](#0-1) 

**Step 2 — Narrowing cast produces `Integer.MIN_VALUE`.**

In `PollingTopicMessageRetriever.poll()`:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
``` [3](#0-2) 

With `filter.getLimit() = 2147483648L` and `context.getTotal().get() = 0L`:
- `(int)(2147483648L)` = `Integer.MIN_VALUE` = `-2147483648`
- `Math.min(-2147483648, 1000)` = `-2147483648`

The identical overflow exists in `PollingTopicListener.poll()`. [4](#0-3) 

**Step 3 — Negative `pageSize` is written into a new filter.**

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [5](#0-4) 

The `@Min(0)` annotation on `TopicMessageFilter.limit` is not enforced here because the builder is invoked directly in private code, not through a Spring AOP proxy.

**Step 4 — `hasLimit()` returns `false` for the negative value.**

```java
public boolean hasLimit() {
    return limit > 0;
}
``` [6](#0-5) 

`-2147483648 > 0` is `false`.

**Step 5 — `setMaxResults` is never called; unbounded query executes.**

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
...
return typedQuery.getResultList().stream();
``` [7](#0-6) 

`getResultList()` materializes the entire result set into JVM heap before any reactive `take()` operator can discard elements.

**Step 6 — The outer `flux.take(filter.getLimit())` in `subscribeTopic()` does not prevent heap exhaustion.**

```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());
}
``` [8](#0-7) 

The DB query and `getResultList()` have already completed and loaded all rows into memory before the reactive `take` operator processes any element.

## Impact Explanation
An attacker can force the mirror node to execute an unbounded `SELECT` against the `topic_message` table for any topic, loading the full result set into JVM heap. On a high-volume topic this causes heap exhaustion and an OOM crash, taking down the gRPC service for all subscribers. The bug also exposes all stored topic messages for the targeted topic to the attacker's stream, constituting a data exfiltration risk for topics intended to be private or rate-limited.

## Likelihood Explanation
The gRPC endpoint is publicly accessible with no authentication. The only precondition is a valid `topicId`, which is trivially discoverable from the public ledger. A single `ConsensusTopicQuery` with `limit = 2147483648` is sufficient to trigger the bug. No credentials, no special role, and no prior state are required. The attack is trivially scriptable and repeatable.

## Recommendation
1. **Clamp the limit before the cast.** In both `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`, guard the narrowing cast:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = remaining > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) remaining;
   ```
2. **Cap user-supplied limits at a server-side maximum** in `ConsensusController.toFilter()` (e.g., reject or clamp any `limit` exceeding a configured `maxLimit`).
3. **Enforce `@Min(0)` on the internal filter builder path** by adding an explicit non-negative check before calling `filter.toBuilder().limit(pageSize)`.

## Proof of Concept
```java
// gRPC client snippet
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1234).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
    .setLimit(2147483648L)   // 2^31 — overflows to Integer.MIN_VALUE in poll()
    .build();

Iterator<ConsensusTopicResponse> it = blockingStub.subscribeTopic(query);
// The first poll() call issues SELECT ... FROM topic_message WHERE topic_id=1234
// with no LIMIT clause, loading all rows into heap.
it.hasNext(); // triggers OOM if topic has large message history
```
