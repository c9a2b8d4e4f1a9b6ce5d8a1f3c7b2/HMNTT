Audit Report

## Title
Unauthenticated Connection Pool Exhaustion via Public Key Query Dedicated Client Path in `getAccounts()`

## Summary
The `getAccounts()` function in `rest/accounts.js` unconditionally checks out a dedicated PostgreSQL client from the connection pool for any request containing `account.publickey`. The client is held for the full duration of the query. With a finite pool and no rate limiting on the REST API, an attacker can exhaust all available connections with a small number of concurrent requests, causing all subsequent public-key queries to block or fail.

## Finding Description
In `rest/accounts.js` line 366, `preQueryHint` is set to `constants.zeroRandomPageCostQueryHint` (a non-empty string) whenever `pubKeyQuery.query !== ''`, which is true for any valid `account.publickey` parameter. [1](#0-0) 

`pool.queryQuietly()` in `rest/utils.js` lines 1519–1527 then takes the dedicated-client branch: it calls `this.connect()` to check out a client from the pool, issues `BEGIN; SET LOCAL random_page_cost = 0`, runs the main query, and commits. The client is only released in the `finally` block after the query completes or errors. [2](#0-1) [3](#0-2) 

The pool is configured with `max: config.db.pool.maxConnections` (default 10) and `statement_timeout` set as a startup parameter. In deployments using pgbouncer (the standard Helm chart deployment), `statement_timeout` appears in `ignore_startup_parameters` in `charts/hedera-mirror/values.yaml`, meaning pgbouncer strips it before forwarding to PostgreSQL, so no server-side query timeout is enforced. [6](#0-5) 

No rate-limiting middleware exists for the Node.js REST API's `/api/v1/accounts` endpoint.

## Impact Explanation
With a pool of 10 connections, 10 concurrent `GET /api/v1/accounts?account.publickey=<key>` requests exhaust the pool. All subsequent requests hitting the dedicated-client path block until `connectionTimeoutMillis` (default 20 s) elapses, then fail with a connection timeout error. Legitimate users receive errors for the duration of the attack. Because the attack is repeatable (re-send 10 requests as each batch times out), the REST API can be kept in a continuously degraded state. Even if `statement_timeout` is enforced (20 s), the attacker simply re-sends requests every 20 seconds.

## Likelihood Explanation
The attack requires zero authentication, zero privileges, and only a syntactically valid public key string. It requires only 10 concurrent HTTP GET requests with no body. It is fully automatable with standard HTTP tools. There is no server-side defense (rate limiting, authentication, or connection-per-IP cap) to overcome.

## Recommendation
1. **Add rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit`) targeting the `/api/v1/accounts` endpoint, particularly for requests containing `account.publickey`.
2. **Enforce `statement_timeout` at the PostgreSQL server level** (via `ALTER ROLE` or `postgresql.conf`) rather than relying on a startup parameter that pgbouncer may strip.
3. **Increase the pool size** or introduce a separate, smaller sub-pool for dedicated-client queries to limit blast radius.
4. **Consider query cost controls**: require pagination limits or restrict the `account.balance=gte:0` wildcard predicate when combined with `account.publickey`.
