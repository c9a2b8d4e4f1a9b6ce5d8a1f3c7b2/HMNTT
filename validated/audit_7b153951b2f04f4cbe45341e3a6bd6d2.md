Audit Report

## Title
`isValidAccount()` Missing Deleted-Account Filter Causes 200 OK for Deleted Accounts Instead of 404

## Summary
`EntityService.isValidAccount()` uses `entityExistenceQuery`, which queries the `entity` table by ID only with no filter on the `deleted` column. A deleted account retains its row in the `entity` table with `deleted = true`, so `isValidAccount()` returns `true`. The subsequent `TokenService.getTokenAccounts()` call finds no active associations and returns `[]`, causing the API to respond `200 OK {"tokens": [], "links": {"next": null}}` instead of `404 Not Found`.

## Finding Description
In `rest/service/entityService.js`, the existence-check query is:

```js
// lines 28-30
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where ${Entity.ID} = $1`;
``` [1](#0-0) 

There is no `deleted` filter. By contrast, `entityFromAliasQuery` (lines 17-20) and `entityFromEvmAddressQuery` (lines 22-25) both correctly exclude deleted entities:

```js
where coalesce(${Entity.DELETED}, false) <> true
  and ${Entity.ALIAS} = $1
``` [2](#0-1) 

`isValidAccount()` calls `getSingleRow` with `entityExistenceQuery` and returns `!isNil(entity)`. For a deleted account the row still exists, so `isNil(entity)` is `false` and the method returns `true`: [3](#0-2) 

In `getTokenRelationships()` of `rest/controllers/tokenController.js`:
- Line 68: `isValidAccount()` returns `true` for the deleted account — no `NotFoundError` is thrown.
- Line 74: `TokenService.getTokenAccounts()` queries `token_account` where `associated = true`; a deleted account has no active associations, so it returns `[]`.
- Lines 86-91: The response is `200 OK` with `{"tokens": [], "links": {"next": null}}`. [4](#0-3) 

The `tokenRelationshipsQuery` in `rest/service/tokenService.js` confirms the `associated = true` filter that produces the empty result: [5](#0-4) 

## Impact Explanation
Any caller querying `/accounts/{deletedAccountId}/tokens` receives `200 OK` with an empty token list, which is semantically indistinguishable from a valid account that holds no tokens. Clients and downstream systems — wallets, explorers, compliance tools — cannot distinguish "account does not exist / is deleted" from "account exists but has no token relationships." This breaks the documented API contract and can cause incorrect business logic in any system that relies on HTTP status codes to gate further processing.

## Likelihood Explanation
Trivially exploitable by any unprivileged user with knowledge of a previously-deleted account ID (public information on a mirror node). No special timing, network conditions, or elevated access is required. The trigger is a single HTTP GET request and is 100% reproducible as long as the account row remains in the `entity` table with `deleted = true`.

## Recommendation
Add a `deleted` filter to `entityExistenceQuery` in `rest/service/entityService.js`, consistent with the pattern already used in `entityFromAliasQuery` and `entityFromEvmAddressQuery`:

```js
static entityExistenceQuery = `select ${Entity.TYPE}
                               from ${Entity.tableName}
                               where coalesce(${Entity.DELETED}, false) <> true
                                 and ${Entity.ID} = $1`;
``` [1](#0-0) 

## Proof of Concept
1. Identify any account that has been deleted (its row in `entity` has `deleted = true`). This is public information.
2. Send: `GET /api/v1/accounts/{deletedAccountId}/tokens`
3. Observed response: `200 OK` — `{"tokens": [], "links": {"next": null}}`
4. Expected response: `404 Not Found`

The root cause is that `isValidAccount()` at line 61 executes `entityExistenceQuery` (no `deleted` filter), finds the row, and returns `true`, bypassing the `NotFoundError` guard at line 69-71 of `tokenController.js`. [6](#0-5)
