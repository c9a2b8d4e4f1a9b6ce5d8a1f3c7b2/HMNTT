# Audit Report

## Title
Unauthenticated DoS via Non-Existent Numeric Account ID Flooding Exhausts DB Connection Pool in `AccountBalance()`

## Summary
`AccountBalance()` in `rosetta/app/services/account_service.go` performs no account-existence pre-check before invoking `RetrieveBalanceAtBlock()`. For any syntactically valid but non-existent numeric account ID (e.g., `0.0.999999999`), the persistence layer unconditionally executes four sequential database queries. The Traefik middleware chain (circuit breaker, in-flight limiter, rate limiter) is disabled by default via `global.middleware: false`. An unauthenticated attacker can flood the endpoint to exhaust the shared PostgreSQL connection pool, degrading or halting the mirror node's gossip ingestion writer.

## Finding Description

**Code path:**

`AccountBalance()` parses the address string (format-validation only), resolves a block, then unconditionally calls `RetrieveBalanceAtBlock()` with no existence check: [8](#0-7) 

Inside `RetrieveBalanceAtBlock()`, the early-exit guard is:

```go
if entity == nil && accountId.HasAlias() {
    return types.AmountSlice{&types.HbarAmount{}}, entityIdString, nil, nil
}
``` [2](#0-1) 

This guard fires **only for alias-type account IDs**. For a numeric ID like `0.0.999999999`, `HasAlias()` is `false`. Even when `getCryptoEntity()` returns `nil` (account not found), execution falls through to `getLatestBalanceSnapshot()` and `getBalanceChange()`.

**Four DB queries execute per request for a non-existent numeric account:**

1. `selectCryptoEntityById` — UNION ALL across `entity` + `entity_history` with `timestamp_range @> @consensus_end`: [9](#0-8) 

2. `selectPreviousPartitionLowerBound` — CTE scan over `mirror_node_time_partitions`: [10](#0-9) 

3. `latestBalanceBeforeConsensus` — correlated subquery scanning `account_balance` partitions. Critically, this query anchors on the **treasury entity's** balance timestamp, not the target account's. In a running system the treasury always has balance records, so `ab.ConsensusTimestamp != 0` and execution continues: [11](#0-10) 

4. `balanceChangeBetween` — `SUM` over `crypto_transfer` filtered by `entity_id` and timestamp range: [12](#0-11) 

All four return empty/zero results but still acquire a DB connection and execute query plans against large partitioned tables.

**Existing protections are insufficient:**

The Traefik middleware chain is gated by:

```yaml
{{ if and .Values.global.middleware .Values.middleware -}}
``` [5](#0-4) 

with the default:

```yaml
global:
  middleware: false
``` [4](#0-3) 

So the entire middleware chain — including `inFlightReq: amount: 5` and `rateLimit: average: 10` — is **not deployed by default**. [13](#0-12) 

Even when manually enabled, the rate limiter is keyed on `requestHost` (the `Host` header), not source IP. An attacker rotating the `Host` header value bypasses the per-host bucket entirely: [6](#0-5) 

Note: the `inFlightReq` middleware correctly uses `ipStrategy.depth: 1` (IP-based), but this protection is irrelevant since the entire chain is off by default.

The response for a non-existent account is HTTP 200 with a zero balance, so the circuit breaker (`ResponseCodeRatio(500, 600, 0, 600) > 0.25`) never triggers. [14](#0-13) 

## Impact Explanation

The Rosetta API and the mirror node's gossip importer share the same PostgreSQL instance. Each flood request holds a DB connection for the duration of four sequential queries against `entity`, `entity_history`, `account_balance` (partitioned), and `crypto_transfer` (partitioned). At sufficient request rate, the connection pool is exhausted: the gossip importer's write transactions queue behind the pool limit, stalling consensus record ingestion. Even below pool exhaustion, I/O saturation from scans on large partitioned tables degrades write throughput. Because responses are HTTP 200, no error-ratio circuit breaker triggers.

## Likelihood Explanation

The attack requires zero privileges, zero authentication, and zero knowledge beyond the public Rosetta API spec. The attacker needs only a valid block index (obtainable from `/network/status`) and any syntactically valid but non-existent account number. Default deployments via the published Helm chart have no rate limiting active (`global.middleware: false`). The attack is trivially scriptable and can be distributed across multiple source IPs to defeat even correctly-configured per-IP limits.

## Recommendation

1. **Add a numeric-account existence short-circuit:** Extend the early-exit guard in `RetrieveBalanceAtBlock()` to also return early when `entity == nil` for numeric IDs, returning a zero balance (matching current behavior) without executing the remaining three queries. [2](#0-1) 

2. **Enable middleware by default:** Change `global.middleware` default to `true` in the rosetta chart values, or document clearly that production deployments must enable it. [4](#0-3) 

3. **Fix rate limiter source criterion:** Change `requestHost: true` to `ipStrategy: depth: 1` in the `rateLimit` middleware configuration so the rate limit cannot be bypassed by rotating the `Host` header. [6](#0-5) 

4. **Add application-level concurrency control:** Implement a semaphore or token-bucket limiter in the Go service layer as a defense-in-depth measure independent of the Traefik middleware chain.

## Proof of Concept

```bash
# 1. Obtain a valid block index
BLOCK=$(curl -s -X POST https://<rosetta-host>/network/status \
  -H 'Content-Type: application/json' \
  -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"}}' \
  | jq '.current_block_identifier.index')

# 2. Flood with non-existent numeric account IDs
for i in $(seq 1 10000); do
  curl -s -X POST https://<rosetta-host>/account/balance \
    -H 'Content-Type: application/json' \
    -H "Host: host-$i.example.com" \
    -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},
         \"account_identifier\":{\"address\":\"0.0.$((RANDOM + 900000000))\"},
         \"block_identifier\":{\"index\":$BLOCK}}" &
done
wait
```

Each request triggers four DB queries against large partitioned tables. With `global.middleware: false` (the default), no rate limiting or in-flight limiting is active. The `Host` header rotation defeats the `requestHost`-keyed rate limiter even if middleware is manually enabled. The endpoint returns HTTP 200 for all requests, preventing circuit-breaker activation.

func (ar *accountRepository) getLatestBalanceSnapshot(ctx context.Context, accountId, timestamp int64) (
	int64,
	*types.HbarAmount,
	*rTypes.Error,
) {
	db, cancel := ar.dbClient.GetDbWithContext(ctx)
	defer cancel()

	var partitionLowerBound int64
	if err := db.Raw(
		selectPreviousPartitionLowerBound,
		sql.Named("timestamp", timestamp),
	).Scan(&partitionLowerBound).Error; err != nil {
		log.Errorf(
			databaseErrorFormat,
			hErrors.ErrDatabaseError.Message,
			fmt.Sprintf("%v looking for previous account_balance partition before %d", err, timestamp),
		)
	}

	// gets the most recent balance at or before timestamp
	ab := &accountBalance{}
	if err := db.Raw(
		latestBalanceBeforeConsensus,
		sql.Named("account_id", accountId),
		sql.Named("lower_bound", partitionLowerBound),
		sql.Named("timestamp", timestamp),
		sql.Named("treasury_entity_id", ar.treasuryEntityId.EncodedId),
	).First(ab).Error; err != nil {
		log.Errorf(
			databaseErrorFormat,
			hErrors.ErrDatabaseError.Message,
			fmt.Sprintf("%v looking for account %d's balance at or before %d", err, accountId, timestamp),
		)
		return 0, nil, hErrors.ErrDatabaseError
	}

	if ab.ConsensusTimestamp == 0 {
		return 0, nil, hErrors.ErrNodeIsStarting
	}

	hbarAmount := types.HbarAmount{Value: ab.Balance}

	return ab.ConsensusTimestamp, &hbarAmount, nil
}
