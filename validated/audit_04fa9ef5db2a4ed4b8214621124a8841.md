Audit Report

## Title
Integer Overflow in `poll()` Causes Unlimited Database Query via `hasLimit()` Bypass

## Summary
In `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`, a narrowing cast `(int)(filter.getLimit() - context.getTotal().get())` silently overflows to a negative integer when `filter.getLimit()` exceeds `Integer.MAX_VALUE`. The resulting negative `pageSize` is embedded into a derived `TopicMessageFilter` via Lombok's `toBuilder().build()`, which does not trigger Spring Bean Validation. In `TopicMessageRepositoryCustomImpl.findByFilter()`, `hasLimit()` returns `false` for negative limits (since `limit > 0` fails), so `setMaxResults()` is never called and the JPA query executes with no SQL `LIMIT` clause — loading the entire topic message table for the targeted topic into JVM heap on every poll cycle.

## Finding Description

**Entry point — `ConsensusController.toFilter()`:** [1](#0-0) 

`query.getLimit()` is a protobuf `uint64` mapped to Java `long`. It is assigned directly to `TopicMessageFilter.limit` with no upper-bound check. The only constraint on `TopicMessageFilter.limit` is `@Min(0)`: [2](#0-1) 

A value such as `3_000_000_000L` (> `Integer.MAX_VALUE`) passes `@Min(0)` without error.

**Overflow site — `PollingTopicMessageRetriever.poll()` lines 68–71:** [3](#0-2) 

With `filter.getLimit() = 3_000_000_000L` and `context.getTotal().get() = 0`:
- `3_000_000_000L - 0L = 3_000_000_000L`
- `(int) 3_000_000_000L = -1_294_967_296` (Java silent narrowing overflow)
- `Math.min(-1_294_967_296, maxPageSize) = -1_294_967_296`

The negative `pageSize` is embedded into a new filter: [4](#0-3) 

Lombok's `build()` is a plain constructor — it does **not** trigger Spring Bean Validation. The negative limit reaches `findByFilter()` unchecked.

**Limit bypass — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51–53:** [5](#0-4) 

`hasLimit()` is defined as `return limit > 0`: [6](#0-5) 

A negative limit fails `limit > 0`, so `setMaxResults()` is never called. The JPA query executes with **no SQL LIMIT**, returning every row for the topic.

**Partial mitigation that does NOT prevent the DB attack — `TopicMessageServiceImpl.subscribeTopic()`:** [7](#0-6) 

`flux.take(filter.getLimit())` caps what is *delivered to the client* at 3 billion messages using the original filter, but it does not prevent each individual `poll()` invocation from issuing an unbounded DB query and loading all results into JVM heap. In throttled mode, `numRepeats = Long.MAX_VALUE`: [8](#0-7) 

The `isComplete()` check `filter.getLimit() == total.get()` will never fire (3 billion messages would need to exist), so polling repeats indefinitely, each time issuing an unbounded full-table scan: [9](#0-8) 

The identical overflow exists in `PollingTopicListener.poll()` lines 54–57: [10](#0-9) 

## Impact Explanation
Each poll cycle fetches the entire `topic_message` table for the targeted topic into JVM heap with no row limit. On a topic with millions of messages this causes severe memory pressure or OOM crash of the gRPC service. Multiple concurrent subscriptions with this crafted limit amplify the effect. The database connection pool is also saturated by long-running full-table scans, degrading service for all users.

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication required. The protobuf `uint64 limit` field accepts any 64-bit unsigned value. Setting `limit = 2_147_483_648` (`Integer.MAX_VALUE + 1`) is the minimum value to trigger the overflow and requires no special privileges or knowledge beyond the public API schema. The attack is trivially repeatable with any gRPC client.

## Recommendation
1. **Clamp the limit before the cast** in both `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Add an upper-bound validation** on `TopicMessageFilter.limit`, e.g. `@Max(Integer.MAX_VALUE)` or a custom constraint, so that values exceeding `Integer.MAX_VALUE` are rejected at the API boundary in `ConsensusController.toFilter()`.
3. **Validate derived filters**: ensure that any `TopicMessageFilter` constructed via `toBuilder().build()` is also validated before use, or add a defensive check in `findByFilter()` to treat non-positive limits as no-limit or throw.

## Proof of Concept
```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

request = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    limit=2_147_483_648,  # Integer.MAX_VALUE + 1 — minimum overflow trigger
)

# Each message received triggers an unbounded DB query in poll()
for response in stub.subscribeTopic(request):
    pass
```
With `limit = 2_147_483_648`, the cast `(int)(2_147_483_648L - 0L)` yields `-2_147_483_648`, `hasLimit()` returns `false`, and every poll issues a full-table scan with no `LIMIT` clause.
