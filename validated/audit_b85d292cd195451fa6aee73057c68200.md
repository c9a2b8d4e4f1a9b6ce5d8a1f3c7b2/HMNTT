# Audit Report

## Title
Accept-Encoding Substring Check Bypass Allows Uncompressed Opcode Replay Responses

## Summary
`validateAcceptEncodingHeader()` in `OpcodesController` uses a naive `String.contains("gzip")` check. An attacker can send `Accept-Encoding: gzip;q=0, deflate`, which contains the substring `"gzip"` and passes validation, but signals to the HTTP stack (Tomcat) that gzip is explicitly rejected (`q=0`). The server then performs the full EVM replay and returns the full uncompressed opcode trace, defeating the compression enforcement gate that exists specifically to protect server and network resources.

## Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/controller/OpcodesController.java`, line 75–76

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null || !acceptEncodingHeader.toLowerCase().contains("gzip")) {
```

The check tests only for the substring `"gzip"` anywhere in the header value. Per RFC 7231 §5.3.4, a quality value of `q=0` means the encoding is explicitly **not acceptable**. The value `gzip;q=0, deflate` contains `"gzip"` as a substring and passes the Java-level check, but Tomcat correctly honors `q=0` and will not apply gzip compression, returning the full uncompressed payload.

The test at line 507 only covers `{"deflate", "identity", "br", ""}` as rejection cases. `gzip;q=0` and `gzip;q=0, deflate` are not tested and are not rejected by the current implementation.

The Javadoc comment at lines 71–72 explicitly states the gzip requirement exists because "the response from this endpoint is huge and without compression this will result in big network latency," confirming this is a resource-protection control.

## Impact Explanation
An attacker bypassing this gate forces the server to:
1. Perform the full, expensive EVM transaction replay (CPU/DB intensive).
2. Serialize and transmit the full uncompressed opcode trace (potentially megabytes of stack/memory/storage data per request).

The per-request bandwidth and resource cost is significantly higher than intended. The rate throttle (`opcodeRequestsPerSecond`) still applies, but each allowed request now consumes substantially more network resources than the compression gate was designed to permit.

## Likelihood Explanation
Any unauthenticated external user can craft this header. No privileges, tokens, or special access are required. The bypass requires only knowledge of HTTP `q`-values (public RFC 7231 knowledge) and a single crafted `Accept-Encoding` header. It is trivially repeatable up to the rate limit.

## Recommendation
Replace the naive substring check with proper RFC 7231 `Accept-Encoding` header parsing. Specifically, parse the quality values and reject the request if `gzip` is absent or has `q=0`. Example fix:

```java
private void validateAcceptEncodingHeader(String acceptEncodingHeader) {
    if (acceptEncodingHeader == null) {
        throw ...;
    }
    boolean gzipAccepted = Arrays.stream(acceptEncodingHeader.split(","))
        .map(String::trim)
        .anyMatch(token -> {
            String[] parts = token.split(";");
            if (!parts[0].trim().equalsIgnoreCase("gzip")) return false;
            for (int i = 1; i < parts.length; i++) {
                String[] kv = parts[i].trim().split("=");
                if (kv.length == 2 && kv[0].trim().equalsIgnoreCase("q")) {
                    return Double.parseDouble(kv[1].trim()) > 0;
                }
            }
            return true; // no q= means q=1.0
        });
    if (!gzipAccepted) {
        throw HttpClientErrorException.create(...);
    }
}
```

Additionally, add test cases for `gzip;q=0` and `gzip;q=0, deflate` to `rejectsWhenAcceptEncodingIsNotGzip`.

## Proof of Concept
```
GET /api/v1/contracts/results/{transactionId}/opcodes HTTP/1.1
Host: <mirror-node>
Accept-Encoding: gzip;q=0, deflate
```

1. The `contains("gzip")` check passes because `"gzip;q=0, deflate"` contains the substring `"gzip"`.
2. The throttle check passes (first request within rate limit).
3. The EVM replay executes in full.
4. Tomcat honors `q=0`, does not apply gzip compression.
5. The full uncompressed opcode trace is returned to the client — bypassing the compression enforcement gate entirely.
