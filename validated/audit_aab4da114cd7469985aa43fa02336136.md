# Audit Report

## Title
Binary Search Compute Amplification via Unthrottled `estimateGas` Iterations

## Summary
The `estimateGas` path in `ContractExecutionService` performs 1 initial EVM execution plus up to 20 additional binary-search EVM executions, but the throttle is consumed exactly once per HTTP request at the controller layer. The sanity check intended to short-circuit the binary search structurally never fires for successful calls, creating a verified ~15× compute amplification factor exploitable by any unauthenticated caller.

## Finding Description

**Throttle is consumed once, at the controller layer.**

`ContractController.call()` calls `throttleManager.throttle(request)` at line 40, consuming 1 rate-bucket token and `scaleGas(request.getGas())` gas-bucket tokens. No further throttle tokens are consumed for the remainder of the request lifecycle. [1](#0-0) [2](#0-1) 

**The sanity check never fires for successful calls.**

`ContractExecutionService.estimateGas()` line 86 checks `gasUsedByInitialCall >= params.getGas()`. For any successful EVM execution, the EVM returns `gasUsed < gasLimit` by definition. This condition is structurally only true on an out-of-gas (OOG) failure. For every normal successful call, the binary search always proceeds. [3](#0-2) 

**Binary search runs up to 20 unthrottled EVM executions.**

`BinaryGasEstimator.search()` loops `while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount())`, calling `doProcessCall(params, gas, true)` on each iteration. The `estimate=true` flag causes `doProcessCall` to skip both `restoreGasToBucket` and any throttle interaction entirely. [4](#0-3) [5](#0-4) [6](#0-5) 

**Gas bucket restore further neutralizes the gas throttle for cheap contracts.**

After the initial call (which runs with `estimate=false`), `restoreGasToBucket` restores `min(gasRemaining, gasLimitRefundPercent% of gasLimit)` tokens. For a cheap contract using ~21,000 gas against a 15M gas limit, nearly all 1,500 scaled tokens are refunded (≈1,497), leaving a net gas-bucket cost of ~3 tokens. The rate-bucket (1 token/request) becomes the sole binding constraint. [7](#0-6) 

**Maximizing iterations is confirmed by the project's own test suite.**

With `lo=21000` (cheap contract) and `hi=15_000_000` (default `maxGasLimit`), the test suite explicitly asserts 14 binary-search iterations. The `estimateIterationThreshold` early-exit (`lo * 0.10 = 2,100`) is negligible relative to the 15M search space and does not prevent near-maximum iterations. [8](#0-7) [9](#0-8) 

**Default configuration values.** [10](#0-9) [11](#0-10) 

## Impact Explanation

With default settings (`requestsPerSecond=500`, `maxGasLimit=15_000_000`, `maxGasEstimateRetriesCount=20`):

- The rate throttle permits 500 `estimateGas` requests/second.
- Each request with `gas=15_000_000` targeting a cheap contract (~21,000 gas) triggers 1 initial + 14 binary-search = **15 full EVM executions**.
- Actual compute: **500 × 15 = 7,500 EVM executions/second**, while the throttle models only 500.
- **Amplification factor: ~15×** above what the throttle accounts for.

This can saturate CPU and database resources on the mirror node web3 service, causing degraded or denied service for all users. The gas bucket is effectively bypassed for cheap contracts due to the `restore()` refund mechanism, making the rate bucket the sole (and insufficient) guard.

## Likelihood Explanation

No privileges, deployed contracts, or authentication are required. An attacker can target any existing cheap contract (e.g., a standard ERC-20 `balanceOf` call). The attack is:

- **Stateless**: each request is independent.
- **Fully repeatable**: at the 500 RPS ceiling.
- **Trivially scriptable**: standard JSON-RPC `eth_estimateGas` calls with `gas=15000000`.
- **Unauthenticated**: the `/api/v1/contracts/call` endpoint requires no credentials.

## Recommendation

1. **Account for binary search iterations in the throttle.** Before entering `BinaryGasEstimator.search()`, pre-consume `maxGasEstimateRetriesCount` additional rate tokens (or a configurable multiplier), or consume one rate token per binary-search iteration inside `search()`.
2. **Charge gas-bucket tokens per binary-search iteration.** Pass the throttle manager into `BinaryGasEstimator` and consume `scaleGas(mid)` tokens on each iteration, restoring unused gas after convergence.
3. **Reduce `maxGasEstimateRetriesCount` or add a separate `estimateGas`-specific rate limit** lower than the general `requestsPerSecond` to bound worst-case compute per second.
4. **Consider capping the `hi/lo` ratio** accepted for binary search to limit the number of iterations structurally, independent of throttle configuration.

## Proof of Concept

```python
import requests, json, time

# Target: any cheap contract (e.g., ERC-20 balanceOf)
ENDPOINT = "https://<mirror-node>/api/v1/contracts/call"
PAYLOAD = {
    "to":       "0x<cheap_contract_address>",
    "data":     "0x70a08231000000000000000000000000<any_address>",  # balanceOf
    "gas":      15000000,   # maxGasLimit → maximizes hi/lo ratio → 14 iterations
    "estimate": True
}

# Send 500 requests/second (the rate limit ceiling)
# Each triggers 1 initial + 14 binary-search EVM executions = 15 total
# Actual EVM load: 500 × 15 = 7,500 executions/second
start = time.time()
for _ in range(500):
    requests.post(ENDPOINT, json=PAYLOAD)
elapsed = time.time() - start
print(f"500 requests in {elapsed:.2f}s → ~{int(500*15/elapsed)} EVM exec/s")
```

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
