Audit Report

## Title
Prototype-Safe Object Injection via `qs` Depth-1 Parsing Causes Unhandled TypeError in `buildComparatorFilter`

## Summary
Sending `ORDER[x]=asc` in any query string causes `qs.parse()` with `depth: 1` / `strictDepth: true` to produce `{ORDER: {x: 'asc'}}`. The object value passes through `lowerCaseQueryValue` unchanged, and `buildComparatorFilter` then calls `.split(':')` on it, throwing a `TypeError`. Every endpoint routing through `buildAndValidateFilters` returns HTTP 500 for any such request. No authentication is required.

## Finding Description

**Step 1 — `qs.parse` with `depth: 1` and `strictDepth: true` does not throw**

`rest/middleware/requestHandler.js` configures `qs` with `depth: 1` and `strictDepth: true`. [1](#0-0) 

`strictDepth: true` throws a `RangeError` only when depth **exceeds** the limit. `ORDER[x]=asc` produces exactly one level of nesting (`{ORDER: {x: 'asc'}}`), which is at the limit — no exception is raised.

**Step 2 — `canonicalizeValue` dispatches to `lowerCaseQueryValue` with an object**

After lowercasing the key to `order`, `canonicalizeValue` looks up `lowerCaseQueryValue` in `queryCanonicalizationMap` and calls it directly because the value is not an array. [2](#0-1) 

**Step 3 — `lowerCaseQueryValue` silently passes the object through**

`typeof {x:'asc'} === 'string'` is `false`, so the object is returned unchanged. `req.query.order` is now `{x: 'asc'}`. [3](#0-2) 

**Step 4 — `buildFilters` passes the object to `buildComparatorFilter`**

`{x: 'asc'}` is not an array, so the `else` branch calls `buildComparatorFilter('order', {x: 'asc'})` with no type guard. [4](#0-3) 

**Step 5 — `buildComparatorFilter` calls `.split(':')` on the object**

`filter.split(':')` where `filter = {x: 'asc'}` throws `TypeError: filter.split is not a function`. [5](#0-4) 

## Impact Explanation

Every API endpoint that processes query parameters via `buildAndValidateFilters` (transactions, contracts, tokens, topics, blocks, schedules, etc.) will throw an unhandled `TypeError` for any request containing `ORDER[x]=<any>`. [6](#0-5) 

The `TypeError` propagates up through the route handler, is caught by `extendExpress`'s try/catch wrapper, and forwarded to `handleError` via `next(err)`. [7](#0-6) 

`handleError` does not recognise `TypeError` as a client error (`InvalidArgumentError`, `RangeError`, `NotFoundError`, `DbError`, `HttpError`), so it falls through to the default `INTERNAL_ERROR` (500) status code. The generic server-error message is returned to the client; the detailed error is logged server-side. [8](#0-7) 

**Correction to the submitted claim**: the process does **not** crash. `handleUncaughtException` is only triggered for truly uncaught exceptions; this error is caught by Express's error handler. The impact is a reliable, repeatable HTTP 500 DoS against all affected endpoints, not a process-level crash.

## Likelihood Explanation

No authentication is required. The payload is a single, trivially crafted HTTP query parameter (`?ORDER[x]=asc`). It is repeatable at will, requires no special knowledge, and affects all REST API endpoints that accept an `order` parameter via `buildAndValidateFilters`. The OpenAPI validator (`openApiValidatorEnabled`) is a conditional mitigation — if disabled (non-test environments may have it off), no upstream guard exists. [9](#0-8) 

## Recommendation

1. **Add a type guard in `buildComparatorFilter`**: reject or stringify non-string `filter` values before calling `.split(':')`. [10](#0-9) 

2. **Add a type guard in `buildFilters`**: before calling `buildComparatorFilter`, verify that `values` (and each element of an array) is a `string`; push a `badParams` entry otherwise. [11](#0-10) 

3. **Harden `lowerCaseQueryValue` / `canonicalizeValue`**: when the canonicalization function receives a non-string, non-array value (i.e., a nested object produced by `qs`), treat it as an invalid parameter rather than passing it through silently. [2](#0-1) 

4. **Consider `depth: 0`**: setting `depth: 0` in `queryOptions` would prevent `qs` from producing nested objects entirely, eliminating the root cause. [1](#0-0) 

## Proof of Concept

```
GET /api/v1/transactions?ORDER[x]=asc HTTP/1.1
Host: <mirror-node-host>
```

Expected (correct) behaviour: `400 Bad Request` — invalid query parameter.

Actual behaviour: `500 Internal Server Error` — `TypeError: filter.split is not a function` thrown in `buildComparatorFilter`, caught by Express error handler, logged server-side, generic 500 returned to client. Repeatable on every request; no authentication required.

const buildFilters = (query) => {
  const badParams = [];
  const filters = [];

  for (const [key, values] of Object.entries(query)) {
    // for repeated params val will be an array
    if (Array.isArray(values)) {
      if (!isRepeatedQueryParameterValidLength(values)) {
        badParams.push({
          code: InvalidArgumentError.PARAM_COUNT_EXCEEDS_MAX_CODE,
          key,
          count: values.length,
          max: config.query.maxRepeatedQueryParameters,
        });
        continue;
      }

      for (const val of values) {
        filters.push(buildComparatorFilter(key, val));
      }
    } else {
      filters.push(buildComparatorFilter(key, values));
    }
  }

  return {badParams, filters};
};
