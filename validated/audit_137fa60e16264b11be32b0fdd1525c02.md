Audit Report

## Title
Token Range Validity Check Bypassed When `ownerOrSpenderParams` Has Both Lower and Upper Bounds

## Summary
In `NftAllowanceServiceImpl.checkOwnerSpenderParamValidity()`, the only service-layer check that rejects an impossible token range (`lower > upper`) is guarded by `!ownerOrSpenderParams.hasLowerAndUpper()`. When a caller supplies both a lower and upper bound for `account.id`, this guard evaluates to `false`, short-circuiting the `&&` and silently skipping the validation. The `Bound` constructor does not compensate because `tokenIds` is always built with `primarySortField = false`, so its constructor-level range check is also skipped. The repository then executes a query with an unsatisfiable predicate and returns empty results with no error.

## Finding Description
**File:** `rest-java/src/main/java/org/hiero/mirror/restjava/service/NftAllowanceServiceImpl.java`, lines 41–44

```java
if (!ownerOrSpenderParams.hasLowerAndUpper()
        && tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {
    throw new IllegalArgumentException("Invalid range provided for %s".formatted(Constants.TOKEN_ID));
}
```

When `ownerOrSpenderParams.hasLowerAndUpper()` returns `true` (both GTE/GT and LTE/LT supplied for `account.id`), the left operand of `&&` is `false` and Java's short-circuit evaluation prevents the right operand from ever being evaluated. The `IllegalArgumentException` is therefore never thrown regardless of the token range values.

The `Bound` constructor at lines 56–60 of `Bound.java` performs the same adjusted-lower > adjusted-upper check, but only when `primarySortField == true`. The controller at line 72 of `AllowancesController.java` always passes `false` for `tokenIds`:

```java
.tokenIds(new Bound(tokenIds, false, TOKEN_ID, NFT_ALLOWANCE.TOKEN_ID))
```

Every test fixture in `NftAllowanceServiceTest.java` does the same. There is therefore no path through which an inverted token range is caught when a two-sided account range is also present.

## Impact Explanation
Any unauthenticated caller can cause the NFT allowance endpoint to silently return an empty result set for an account that actually has allowances, by pairing a valid two-sided account range with an inverted token range. The API responds HTTP 200 with an empty collection instead of HTTP 400. Downstream consumers (indexers, wallets, dApps) receive missing records with no indication that the query parameters were invalid, compromising data integrity of exported records. No data is leaked or modified; severity is medium.

## Likelihood Explanation
No privileges are required. The parameters are ordinary HTTP query parameters accepted from any external caller. The combination (two-sided account range + inverted token range) is trivially constructable by anyone who reads the API documentation or fuzzes the endpoint. The condition is stable and repeatable.

## Recommendation
Remove the `!ownerOrSpenderParams.hasLowerAndUpper()` guard so the token range validity check is unconditional:

```java
// Before
if (!ownerOrSpenderParams.hasLowerAndUpper()
        && tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {

// After
if (tokenParams.getAdjustedLowerRangeValue() > tokenParams.adjustUpperBound()) {
```

Alternatively, move the inverted-range check into the `Bound` constructor and apply it regardless of `primarySortField`, or add a dedicated `verifyRange()` call inside `verifyRangeId()` so it is applied uniformly to both bounds.

## Proof of Concept
```
GET /api/v1/accounts/0.0.500/allowances/nfts
    ?account.id=gte:0.0.100
    &account.id=lte:0.0.200
    &token.id=gte:0.0.9000
    &token.id=lte:0.0.1
```

1. `ownerOrSpenderParams` has both lower (`gte:100`) and upper (`lte:200`) → `hasLowerAndUpper()` = `true`.
2. `!ownerOrSpenderParams.hasLowerAndUpper()` = `false` → the `&&` short-circuits; the token range check is skipped.
3. `tokenParams` lower = 9000, upper = 1; the `Bound` constructor does not throw because `primarySortField = false`.
4. `repository.findAll(request, id)` executes with predicate `token_id >= 9000 AND token_id <= 1`.
5. Database returns zero rows; API responds `200 OK` with `{"allowances": [], "links": {...}}` instead of `400 Bad Request`.
