Audit Report

## Title
Cache Key Bypass via Unnormalized `req.originalUrl` Enables Repeated Database Queries

## Summary
`cacheKeyGenerator` in `rest/middleware/responseCacheHandler.js` produces Redis cache keys by MD5-hashing the raw `req.originalUrl`. Because `normalizeRequestQueryParams` from `rest/middleware/requestNormalizer.js` is never registered as middleware, semantically identical requests with different parameter orderings or explicit default values each produce a distinct cache key, causing permanent cache misses and repeated PostgreSQL queries. No authentication is required.

## Finding Description

**`cacheKeyGenerator` — confirmed at lines 151–153:** [1](#0-0) 

The comment immediately above it (lines 148–150) explicitly acknowledges the missing normalization, referencing a future fix (issue 9113): [2](#0-1) 

**`normalizeRequestQueryParams` is never wired into production middleware.** A search across all `rest/**/*.js` files confirms it appears only in `rest/middleware/requestNormalizer.js` and its test file `rest/__tests__/middleware/requestNormalizer.test.js` — never imported or called in any route or middleware registration path. [3](#0-2) 

**`openApiValidator` has `validateRequests: false`**, so it performs no request-level rejection that could limit the parameter space: [4](#0-3) 

**Middleware ordering in `server.js`:** `responseCacheCheckHandler` is registered at line 97, before all route handlers (lines 101+). The cache key is computed and the Redis lookup performed before any application-level validation runs: [5](#0-4) 

**`listStakingRewardsByAccountId` in `accountController.js` calls `EntityService.isValidAccount` (a DB query) at line 172, before `buildAndValidateFilters` at line 176.** This means even a request with an unknown query parameter triggers a DB round-trip before the 400 is returned: [6](#0-5) 

**`responseCacheUpdateHandler` does not cache 400 responses**, so each bypass attempt is a permanent cache miss — the attacker's variants are never absorbed into the cache: [7](#0-6) 

## Impact Explanation
An attacker can generate an unbounded number of distinct Redis cache keys for any cached endpoint by varying parameter order or supplying explicit default values (e.g., `balance=true`, `limit=25`, `order=asc`). Each variant bypasses Redis and hits PostgreSQL directly. For high-cardinality endpoints such as `/api/v1/accounts`, `/api/v1/transactions`, or `/api/v1/contracts/results`, each DB query is expensive. The Redis cache — the primary scalability mechanism — is rendered ineffective, constituting a low-cost, high-impact cache-exhaustion / database DoS vector.

## Likelihood Explanation
No authentication, API key, or special knowledge is required. The attack is trivially scriptable: iterate over all permutations of valid query parameters and their default values. The parameter space for endpoints like `/api/v1/accounts` yields hundreds of distinct orderings. The attack is repeatable indefinitely and is not rate-limited by any mechanism visible in the codebase.

## Recommendation
Wire `normalizeRequestQueryParams` from `rest/middleware/requestNormalizer.js` into the cache key path — either by calling it inside `cacheKeyGenerator` before hashing, or by registering it as middleware that rewrites `req.originalUrl` before `responseCacheCheckHandler` runs. This is already planned (issue 9113 referenced in the comment at line 149 of `responseCacheHandler.js`). Until it is deployed, the cache affords no protection against trivially varied but semantically identical requests.

## Proof of Concept
```
# All three are semantically identical for /api/v1/accounts (balance=true, limit=25, order=asc are defaults)
# but each produces a different MD5 hash and a separate DB query:

curl "https://host/api/v1/accounts"
curl "https://host/api/v1/accounts?balance=true"
curl "https://host/api/v1/accounts?limit=25&balance=true&order=asc"
curl "https://host/api/v1/accounts?order=asc&limit=25&balance=true"
curl "https://host/api/v1/accounts?balance=true&order=asc&limit=25"
# ... (6! / duplicates = hundreds of distinct cache keys for 6-parameter endpoints)
```

Each request misses the Redis cache, hits PostgreSQL, and — because `responseCacheUpdateHandler` only stores successes keyed to the exact URL used — the cache is never populated for the other variants.

```javascript
const normalizeRequestQueryParams = (openApiRoute, path, query) => {
  const openApiParameters = openApiMap.get(openApiRoute);
  if (isEmpty(openApiParameters)) {
    return isEmpty(query) ? path : path + '?' + querystring.stringify(query);
  }

  let normalizedQuery = '';
  for (const param of openApiParameters) {
    const name = param.parameterName;
    const value = query[name];
    let normalizedValue = '';
    if (value !== undefined) {
      normalizedValue = Array.isArray(value) ? getNormalizedArrayValue(name, value) : value;
    } else if (param?.defaultValue !== undefined) {
      // Add the default value to the query parameter
      normalizedValue = param.defaultValue;
    }

    if (!isEmpty(normalizedValue)) {
      normalizedQuery = appendToQuery(normalizedQuery, name + '=' + normalizedValue);
    }
  }

  return isEmpty(normalizedQuery) ? path : path + '?' + normalizedQuery;
};
```
