**Analysis of the `unless = "#result == null"` claim:**

The report is technically correct: `Optional<byte[]>` is never `null`, so the condition is always `false` and every result — including `Optional.empty()` for non-existent contracts — is cached. Audit Report

## Title
Cache Thrashing via Predictable contractId Cycling Forces Sustained DB Polling in `findRuntimeBytecode()`

## Summary
The `findRuntimeBytecode()` method in `ContractRepository` is backed by a Caffeine cache capped at 1,000 entries with a 1-hour access expiry. The global rate limiter is a single shared `Bucket` with no per-source-IP partitioning. An unprivileged attacker who cycles contract call requests across more than 1,000 distinct `contractId` values continuously evicts legitimate entries, keeping the cache perpetually cold and forcing every legitimate lookup to hit the database.

## Finding Description

**Code locations:**

- Cache annotation on `findRuntimeBytecode()`: [1](#0-0) 
- Cache spec `maximumSize=1000`: [2](#0-1) 
- Cache manager bean wiring: [3](#0-2) 
- Global rate limiter (single shared `Bucket`): [4](#0-3) 
- Call site in `readFromDataSource()`: [6](#0-5) 

**Root cause:**

The Caffeine cache for contract bytecode has a hard cap of 1,000 entries. Hedera entity IDs are sequential `Long` values starting from 1 and are publicly enumerable. Any caller who supplies more than 1,000 distinct `to` addresses in contract call requests will continuously evict the oldest cache entry via Caffeine's LRU-like eviction policy, keeping the cache perpetually cold for any ID outside the current working set.

The `unless = "#result == null"` condition is always `false` because the return type is `Optional<byte[]>`, which is never `null`. This means both hits (`Optional.of(bytes)`) and misses (`Optional.empty()`) are cached. While this is the intended behavior for preventing repeated DB hits on the same ID, it also means the attacker does not need valid contract IDs — any 1,001+ distinct `Long` values will fill and continuously evict the cache.

The rate limiter is a single global `Bucket` shared across all source IPs. A single attacker can consume the entire request budget without any per-IP throttling. [7](#0-6) 

**Exploit flow:**

1. Attacker sends POST `/api/v1/contracts/call` requests with `to` addresses mapping to contractIds 1, 2, 3, …, N (N >> 1,000).
2. Each request reaches `ContractBytecodeReadableKVState.readFromDataSource()`, which calls `contractRepository.findRuntimeBytecode(entityId.getId())`.
3. With N = 10,000 unique IDs and the global rate limit consumed by the attacker, the Caffeine cache (size 1,000) holds at most 10% of the working set.
4. Approximately 90% of requests per second bypass the cache and execute `SELECT runtime_bytecode FROM contract WHERE id = :contractId` against the database.
5. The attack is self-sustaining: as long as the attacker cycles IDs faster than the cache can stabilize, legitimate users' lookups are also cache-cold.

## Impact Explanation
Under normal operation with a warm cache, the node issues a small number of DB queries per second for bytecode lookups. The attacker drives this to a large fraction of the global RPS limit — a many-fold increase in read I/O on the `contract` table. This degrades query latency for all other users sharing the same database connection pool, increases CPU load on the DB server, and can cause cascading timeouts across the web3 service. The attack is repeatable indefinitely because the cache eviction is a structural property of the fixed `maximumSize=1000` setting, not a transient condition.

## Likelihood Explanation
The attack requires no credentials, no special knowledge, and no brute force. Hedera entity IDs are publicly enumerable sequential integers. The attacker needs only a standard HTTP client and a list of sequential integers. The global rate limiter is not partitioned per source IP, so a single attacker can consume the entire request budget. The attack is trivially automatable and indefinitely repeatable.

## Recommendation

1. **Increase cache size** or make it configurable via `CacheProperties` to match the realistic working set of active contracts. [2](#0-1) 
2. **Add per-IP rate limiting** to `ThrottleConfiguration` so a single source cannot consume the entire global request budget. The existing `RequestProperties` filter mechanism in `ThrottleManagerImpl` can be extended for this purpose. [8](#0-7) 
3. **Fix the `unless` condition** to correctly express intent: `unless = "#result != null && !#result.isPresent()"` if caching empty results is undesirable, or document that caching misses is intentional. [9](#0-8) 
4. Consider adding a **circuit breaker** on the DB connection pool for the `contract` table to prevent cascading timeouts under sustained load.

## Proof of Concept

```python
import requests, itertools, threading

BASE = "https://<mirror-node>/api/v1/contracts/call"
HEADERS = {"Content-Type": "application/json"}

def make_long_zero_address(n):
    # Hedera long-zero EVM address: 0x000...0<n as 8-byte hex>
    return "0x" + "00" * 12 + format(n, '016x')

def attack():
    for contract_id in itertools.cycle(range(1, 10001)):
        payload = {
            "to": make_long_zero_address(contract_id),
            "data": "0x",
            "gas": 30000
        }
        try:
            requests.post(BASE, json=payload, headers=HEADERS, timeout=2)
        except Exception:
            pass

# Launch threads to saturate the global rate limit
threads = [threading.Thread(target=attack) for _ in range(20)]
for t in threads:
    t.start()
```

**Expected outcome:** The Caffeine cache for contract bytecode (size 1,000) is continuously evicted by the 10,000-ID working set. DB query rate for `SELECT runtime_bytecode FROM contract WHERE id = :contractId` rises to a large fraction of the global RPS limit. Legitimate users experience increased latency and potential timeout errors on contract call requests.
