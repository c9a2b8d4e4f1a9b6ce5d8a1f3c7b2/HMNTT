Audit Report

## Title
O(n²) Sibling Chain Traversal in `parsePossiblyHookContractRelatedParent()` Enables Mirror Node CPU Exhaustion

## Summary
`RecordItem.java`'s inner `RecordItemBuilder.parsePossiblyHookContractRelatedParent()` contains an unbounded backward walk over the `previous` linked chain. For a parent transaction P followed by n sibling child transactions C1…Cn, the k-th sibling walks k−1 nodes, producing O(n²) total CPU work across the sibling group. An unprivileged attacker triggers this by submitting a `ContractCall` that spawns many internal child calls to contracts whose entity number is never 365.

## Finding Description

**Exact location**: `common/src/main/java/org/hiero/mirror/common/domain/transaction/RecordItem.java`, inner class `RecordItemBuilder`, method `parsePossiblyHookContractRelatedParent()`, lines 350–375. [1](#0-0) 

**`HOOK_CONTRACT_NUM` value**: Defined as `365` at line 57. [2](#0-1) 

**How `previous` is wired**: Every record reader sets `previous` to the immediately preceding `RecordItem` in stream order.

- `AbstractRecordFileItemReader.onRecordStreamItem()` line 173: `.previous(items.isEmpty() ? null : items.getLast())` [3](#0-2) 

- `AbstractPreV5RecordFileReader.readBody()` line 110: `.previous(lastRecordItem)` [4](#0-3) 

For a parent P followed by siblings C1…Cn the chain is: `Cn.previous = C(n−1)`, …, `C2.previous = C1`, `C1.previous = P`.

**How `parent` is resolved**: `parseParent()` (lines 330–348) resolves `parent = P` for every sibling Ck (k ≥ 2) via the `previous.parent` shortcut, so `parent` is always P for all siblings. [5](#0-4) 

**The loop** (lines 359–371):

```java
while (candidateRecord != null && candidateRecord != parent) {
    var candidateTxnRecordContractResult = candidateRecord.getContractResult();
    if (candidateTxnRecordContractResult == null) { break; }
    if (candidateTxnRecordContractResult.getContractID().getContractNum() == HOOK_CONTRACT_NUM) {
        return candidateRecord;
    }
    candidateRecord = candidateRecord.previous;
}
``` [6](#0-5) 

For sibling Ck the loop starts at `C(k−1)` and walks backward until it hits P (== `parent`). If every sibling has a non-null `ContractFunctionResult` with `contractNum != 365`, no early exit fires and the loop runs exactly k−1 iterations. Summing over all n siblings: 0+1+…+(n−1) = n(n−1)/2 = **O(n²)**.

**Entry conditions an attacker fully controls**:
1. `hasParentConsensusTimestamp()` — set by the consensus node for every internal child call; attacker triggers this by submitting a `ContractCall`.
2. `currentTxnRecordContractResult != null` — present on every child contract call record.
3. `contractNum != HOOK_CONTRACT_NUM (365)` — attacker deploys target contracts with any entity number other than 365; this is the default for all user-deployed contracts.
4. `previous != null` — always true for non-first items in a file.

**Why existing checks are insufficient**: The only early-exit conditions inside the loop are (a) `candidateRecord == parent` (terminates correctly but only after full traversal), (b) `candidateTxnRecordContractResult == null` (never fires if all siblings are contract calls), and (c) `contractNum == 365` (never fires if attacker avoids 365). There is no depth cap, no visited-set, and no maximum sibling count enforced at this layer.

## Impact Explanation
With Hedera's maximum gas per transaction (~15,000,000 gas) and a minimal CALL costing ~700–1,000 gas, an attacker can produce hundreds to low thousands of child transactions per parent. At n = 1,000 siblings the loop executes ~500,000 iterations per parent transaction. Submitting a sustained stream of such transactions (each costing the attacker gas but imposing quadratic work on the mirror node's importer thread) can drive the importer's CPU well above sustainable thresholds. The importer processes record files sequentially, so CPU saturation directly stalls all record file ingestion, causing the mirror node to fall behind the network.

## Likelihood Explanation
No special privileges are required. Any Hedera account holder can deploy a fan-out contract and submit `ContractCall` transactions. The attack is repeatable and can be automated. The cost to the attacker scales linearly with n while the cost to the mirror node scales quadratically, making the attack economically asymmetric. The attack is also silent — it produces no errors, only elevated CPU.

## Recommendation
Replace the backward linear scan with an O(1) lookup. The simplest fix is to record the first hook-contract sibling on the parent itself during processing, or to pass it forward through the `previous` reference so each new sibling can inherit it in O(1). Alternatively, cap the traversal depth (e.g., break after a fixed maximum of iterations) as a short-term mitigation. The root cause is that the algorithm re-traverses already-visited siblings on every new sibling instead of maintaining a running pointer.

## Proof of Concept
1. Deploy n contracts on Hedera (all will receive entity numbers far from 365).
2. Deploy a fan-out contract that, when called, issues `CALL` opcodes to each of the n deployed contracts.
3. Submit a `ContractCall` to the fan-out contract. The consensus node emits one parent record P and n child records C1…Cn, each with a `ContractFunctionResult` whose `contractNum != 365`.
4. The mirror node importer reads the file. When building Ck, `parsePossiblyHookContractRelatedParent()` walks k−1 nodes backward. Total iterations across all siblings: n(n−1)/2.
5. Repeat step 3 in a loop. Each submission costs O(n) gas to the attacker but imposes O(n²) CPU iterations on the importer. At n ≈ 1,000, a sustained submission rate of a few transactions per second is sufficient to saturate the importer thread.

        private RecordItem parsePossiblyHookContractRelatedParent() {
            final var currentTxnRecordContractResult = getContractResult();

            if (transactionRecord.hasParentConsensusTimestamp()

                        // we found the first hook child item, which has a ContractResult and is executed via a hook
                        return candidateRecord;
                    }

                    candidateRecord = candidateRecord.previous;
                }

            return null;
        }
