Audit Report

## Title
Connection Pool Exhaustion DoS via Unauthenticated `GET /api/v1/network/registered-nodes` Flooding

## Summary
The `GET /api/v1/network/registered-nodes` endpoint in the `rest-java` module requires no authentication and has no application-level rate limiting. All endpoints in the `rest-java` service share a single HikariCP connection pool. An unprivileged attacker can flood this endpoint with concurrent requests, exhausting the shared pool and starving other endpoints of database connections, causing them to return errors.

## Finding Description

**Code path:**

`NetworkController.getRegisteredNodes()` (lines 173–187) is mapped to `GET /api/v1/network/registered-nodes` with no security annotation. [1](#0-0) 

It delegates to `NetworkServiceImpl.getRegisteredNodes()` (lines 139–152), which calls `RegisteredNodeRepository.findByRegisteredNodeIdBetweenAndDeletedIsFalseAndTypeIs()` — a native SQL query that holds a HikariCP connection for its duration. [2](#0-1) [3](#0-2) 

**Shared pool with no size override:**

The pool is constructed in `CommonConfiguration.dataSource()` (lines 66–95), binding `HikariConfig` from `spring.datasource.hikari` properties. [4](#0-3) 

The `rest-java/src/main/resources/` directory contains only `banner.txt` — no `application.yml` or `application.properties` exists to override `maximumPoolSize`. HikariCP's documented default is **10**.


**No application-level rate limiting:**

`RestJavaConfiguration` registers only `ShallowEtagHeaderFilter` — no throttle, no per-IP limiter, no global rate limiter. [5](#0-4) 

By contrast, the `web3` module has a full bucket4j `ThrottleConfiguration` (rate, gas, opcode buckets). No equivalent exists in `rest-java`. [6](#0-5) 

**Infrastructure mitigations are inactive by default:**

- `global.gateway.enabled: false` disables the GCP backend policy (which carries `maxRatePerEndpoint: 250`) in a default deployment. [7](#0-6) [8](#0-7) 

- The Traefik circuit breaker (`NetworkErrorRatio() > 0.10`) only activates after errors have already occurred. [9](#0-8) 

- HPA is disabled by default (`hpa.enabled: false`), so no automatic horizontal scaling occurs. [10](#0-9) 

## Impact Explanation

All `rest-java` endpoints (`/api/v1/network/nodes`, `/api/v1/network/stake`, `/api/v1/network/supply`, `/api/v1/network/fees`, etc.) draw from the same HikariCP pool. Once the pool is exhausted, any endpoint requiring a DB connection blocks in HikariCP's wait queue. After the default 30-second `connectionTimeout`, requests throw `SQLTimeoutException`, surfacing as 500 errors to callers — indistinguishable from a database outage. The `RestJavaHighDBConnections` Prometheus alert fires only after 75% utilization sustained for 5 minutes, well after the DoS is effective. [11](#0-10) 

## Likelihood Explanation

No privileges, credentials, API keys, or special knowledge are required. The endpoint is publicly documented. A single attacker with ≥10 concurrent HTTP connections can trigger the condition using any standard HTTP load tool (`curl`, `wrk`, `ab`). The threshold is trivially low. The attack is repeatable and scriptable with no preconditions.

## Recommendation

1. **Set an explicit `maximumPoolSize`** in the `rest-java` Helm chart's `config` section (e.g., `spring.datasource.hikari.maximum-pool-size: 50`) sized to match expected concurrency, and configure a per-endpoint or per-IP connection limit.
2. **Add application-level rate limiting** to `rest-java` equivalent to the `web3` `ThrottleConfiguration` — a bucket4j global rate limiter registered as a servlet filter in `RestJavaConfiguration`.
3. **Enable HPA** (`hpa.enabled: true`) so the service can scale horizontally under load.
4. **Enable the GCP backend policy** in production deployments to enforce `maxRatePerEndpoint` at the gateway layer.
5. **Set a query timeout** on the `RegisteredNodeRepository` native query (e.g., via `@QueryHints`) to bound the maximum connection hold time.

## Proof of Concept

```bash
# Exhaust the default HikariCP pool (10 connections) with 15 concurrent requests
seq 15 | xargs -P15 -I{} curl -s "https://<mirror-node-host>/api/v1/network/registered-nodes" &

# Simultaneously probe another endpoint — expect 500 errors once pool is exhausted
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "%{http_code}\n" "https://<mirror-node-host>/api/v1/network/stake"
done
```

Expected result: once the 10-connection pool is saturated by the first command, the second command returns HTTP 500 responses for `/api/v1/network/stake` and other endpoints that require a DB connection.

    public Collection<RegisteredNode> getRegisteredNodes(RegisteredNodesRequest request) {
        final var sort = Sort.by(request.getOrder(), REGISTERED_NODE.REGISTERED_NODE_ID.getName());

    @Bean
    @ConditionalOnMissingBean(DataSource.class)
    @Lazy
    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
