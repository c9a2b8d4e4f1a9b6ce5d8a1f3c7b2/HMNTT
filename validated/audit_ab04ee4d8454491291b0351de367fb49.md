# Audit Report

## Title
Unauthenticated EVM Address Lookup Causes Unconditional DB Miss on Every Request in `getNftAllowances()`

## Summary
The NFT allowances endpoint accepts EVM address strings as the `accountId` path parameter. When such an address does not correspond to any entity, `EntityServiceImpl.lookup()` executes a full `findByEvmAddress()` database query on every request with no negative-result caching and no application-level rate limiting. An attacker can trivially generate syntactically valid but non-existent EVM addresses and flood the endpoint, causing guaranteed DB misses at scale and degrading performance for all legitimate users.

## Finding Description

**Entry point:** `AllowancesController.getNftAllowances()` accepts `@PathVariable EntityIdParameter id`, which is parsed by `EntityIdParameter.valueOf()`. [7](#0-6) 

**Parse chain:** `EntityIdParameter.valueOf()` tries `EntityIdNumParameter` first, then `EntityIdEvmAddressParameter`. A 40-hex-character string (e.g. `deadbeefdeadbeefdeadbeefdeadbeefdeadbeef`) does not match the numeric pattern and resolves to `EntityIdEvmAddressParameter`. [8](#0-7) 

**Service call:** `NftAllowanceServiceImpl.getNftAllowances()` calls `entityService.lookup(request.getAccountId())` unconditionally at line 28, after only validating owner/spender range parameters. [1](#0-0) 

**DB hit with no cache:** Inside `EntityServiceImpl.lookup()`, the `EntityIdEvmAddressParameter` branch calls `entityRepository.findByEvmAddress(p.evmAddress())` directly. The `rest-java` `EntityRepository` declares this method with no `@Cacheable` annotation, so every call hits the database. [9](#0-8) [4](#0-3) 

**Asymmetry:** The `EntityIdNumParameter` branch returns `Optional.of(p.id())` with zero DB access. Only the EVM address (and alias) branches touch the database, making EVM address inputs disproportionately expensive. [10](#0-9) 

**Root cause / failed assumption:** The design assumes EVM address inputs will mostly correspond to real entities. There is no negative-result cache (unlike the `importer` module's `EntityIdServiceImpl`, which wraps all EVM address lookups in `cacheLookup()`) and no guard before the DB call.

## Impact Explanation
Each request with a non-existent EVM address causes one guaranteed index-scan miss on the `entity` table (`evm_address` column). At scale, this saturates DB connection pools and index-scan capacity, increasing query latency for all concurrent legitimate requests. The impact is proportional to request rate and requires no authentication or special privileges.

## Likelihood Explanation
Preconditions: none. The endpoint is public and unauthenticated. Generating syntactically valid 40-hex-character strings is trivial. The attack is fully repeatable and automatable with a simple HTTP client loop (e.g., `curl` or `wrk`). No knowledge of the system internals is required beyond the public API documentation.

## Recommendation
1. **Add negative-result caching** to `EntityServiceImpl.lookup()` for the `EntityIdEvmAddressParameter` branch, mirroring the pattern used in `importer`'s `EntityIdServiceImpl.cacheLookup()`. Cache misses (empty results) with a short TTL to prevent cache poisoning while still absorbing repeated lookups for the same non-existent address.
2. **Apply application-level rate limiting** on the `/api/v1/accounts/{id}/allowances/nfts` endpoint, keyed on the `id` path parameter or client IP, to bound the per-client request rate.
3. **Consider short-circuiting** the DB call for EVM addresses that match the long-zero prefix pattern (shard/realm encoded as zeros), which can be resolved to a numeric ID without a DB lookup, as the `graphql` module's `EntityServiceImpl.getByEvmAddressAndType()` already does.

## Proof of Concept
```bash
# Generate a random non-existent EVM address and hammer the endpoint
while true; do
  ADDR=$(openssl rand -hex 20)
  curl -s "https://<mirror-node>/api/v1/accounts/${ADDR}/allowances/nfts" -o /dev/null &
done
```
Each iteration causes `EntityRepository.findByEvmAddress()` to execute `SELECT id FROM entity WHERE evm_address = $1 AND deleted <> true` against the database with a value that will never match, with no caching of the empty result.
