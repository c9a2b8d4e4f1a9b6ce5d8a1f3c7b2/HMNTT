Audit Report

## Title
Offloaded Call Data Not Reconstructed in EVM Replay — Opcode Trace Diverges from Original Execution

## Summary
When an Ethereum transaction has its calldata offloaded to a Hedera file (`callDataId` set, `callData` empty in the stored RLP), `OpcodeServiceImpl` passes the truncated RLP and empty calldata to `TransactionExecutionService.buildEthereumTransactionBody()`. The hedera-app `TransactionExecutor` then executes with empty calldata, producing an opcode trace that is entirely different from the original on-chain execution.

## Finding Description
**Affected file:** `web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`

`buildCallServiceParameters()` calls two helpers:

**`getCallDataBytes()` (lines 221–226):**
```java
private byte[] getCallDataBytes(EthereumTransaction ethereumTransaction, ContractResult contractResult) {
    final var callData = ethereumTransaction != null
            ? ethereumTransaction.getCallData()
            : contractResult.getFunctionParameters();
    return callData != null ? callData : new byte[0];
}
```
When `ethereumTransaction != null`, it returns `ethereumTransaction.getCallData()`. For offloaded-calldata transactions this field is `EMPTY_BYTE_ARRAY` — the actual calldata lives in a Hedera file identified by `callDataId`. The code never falls back to `contractResult.getFunctionParameters()`, which *does* hold the reconstructed calldata after import (populated by `EthereumTransactionHandler.updateContractResult()`).

**`getEthereumDataBytes()` (lines 228–234):**
```java
private byte[] getEthereumDataBytes(EthereumTransaction ethereumTransaction) {
    if (ethereumTransaction == null) { return new byte[0]; }
    final var data = ethereumTransaction.getData();
    return data != null ? data : new byte[0];
}
```
Returns `ethereumTransaction.getData()` — the raw RLP as submitted to Hedera, which contains `0x80` (empty) in the calldata slot.

**`buildEthereumTransactionBody()` (lines 201–212 of `TransactionExecutionService.java`):**
```java
private TransactionBody buildEthereumTransactionBody(final ContractDebugParameters params) {
    final var txnBody = defaultTransactionBodyBuilder(params)
            .ethereumTransaction(EthereumTransactionBody.newBuilder()
                    .ethereumData(Bytes.wrap(params.getEthereumData()))
                    .maxGasAllowance(Long.MAX_VALUE)
                    .build())
            .transactionFee(CONTRACT_CREATE_TX_FEES)
            .build();
    patchSenderNonce(params);
    return txnBody;
}
```
The truncated RLP is wrapped verbatim. `EthereumTransactionBody` carries only `ethereumData` and `maxGasAllowance` — no `callData` file reference. The executor parses the RLP, finds empty calldata, and executes with it.

This branch is taken because `ethereumData` is non-empty (the truncated RLP is still valid bytes), satisfying the guard at lines 82–85.

**Root cause:** `getCallDataBytes()` assumes `ethereumTransaction.getCallData()` always contains the full calldata. This is false for transactions where calldata was offloaded to a file. No reconstruction step re-inserts the file-backed calldata before replay. `patchSenderNonce()` patches the nonce but does nothing about calldata.

## Impact Explanation
Any opcode trace for a transaction whose calldata was offloaded to a Hedera file will reflect execution with empty calldata. The resulting trace — opcodes, stack, memory, storage — reflects a fabricated execution context, not the original one. Developers and security auditors relying on `debug_traceTransaction` (the `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint) to understand historical contract behavior will receive systematically misleading output. This is a data integrity failure in the trace layer with no direct fund risk.

## Likelihood Explanation
Offloaded calldata is a documented, production-used Hedera feature, evidenced by the `RAW_TX_TYPE_1_CALL_DATA_OFFLOADED` constant and associated test fixtures in the test suite, as well as real testnet transactions in `ethereum_transaction.json` (e.g., `callDataId: 4675057`, `callDataId: 4675853`, `callDataId: 2192120`). Any unprivileged user can trigger the divergence by supplying the hash of any such transaction to the public endpoint. No special privileges, credentials, or prior knowledge beyond the transaction hash are required. The condition is repeatable for every affected transaction.

## Recommendation
In `OpcodeServiceImpl.getCallDataBytes()`, add a fallback: when `ethereumTransaction != null` but `ethereumTransaction.getCallData()` is empty and `ethereumTransaction.getCallDataId()` is non-empty, return `contractResult.getFunctionParameters()` instead of empty bytes. The importer already populates this field with the reconstructed calldata.

Additionally, `getEthereumDataBytes()` should reconstruct the RLP with the actual calldata re-inserted before passing it to `buildEthereumTransactionBody()`, so the executor receives a complete, valid Ethereum transaction. The `AbstractEthereumTransactionParser.encode()` infrastructure already exists in the importer for this purpose and could be reused or exposed.

## Proof of Concept
1. Find any transaction in the mirror node DB where `ethereum_transaction.call_data_id IS NOT NULL` and `ethereum_transaction.call_data = '\x'` (empty). Real examples exist in testnet data (e.g., `consensusTimestamp: 1723510869859467009`, `callDataId: 4675853`).
2. Call `GET /api/v1/contracts/results/{hash}/opcodes` with the transaction hash.
3. Observe that the returned opcode trace shows execution with empty calldata (`CALLDATASIZE` returns 0, `CALLDATALOAD` returns 0, etc.) rather than the actual calldata that was present during original execution.
4. Compare against `GET /api/v1/contracts/results/{hash}` which correctly shows the reconstructed `function_parameters` (populated from the file during import), confirming the calldata was available but not used during replay.
