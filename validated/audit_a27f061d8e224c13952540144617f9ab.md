**Architectural note the report glosses over:**

`SidecarContractMigration` lives in the **importer** module; `ContractRepository`'s Caffeine cache lives in the **web3** module. These are separate Spring Boot JVMs sharing only a PostgreSQL database. The importer physically cannot call `cache.evict()` on the web3 JVM's in-process Caffeine instance. The report frames this as the importer "bypassing" the cache, but it is an inherent property of the two-service architecture. There is no shared/distributed cache layer between them.

Despite that framing issue, the underlying data-consistency gap is real and all technical claims check out.

Audit Report

## Title
Stale Bytecode Cache Poisoning via SidecarContractMigration Bypassing Spring Cache Eviction

## Summary
`SidecarContractMigration.migrate()` in the importer service writes `runtime_bytecode` directly to PostgreSQL via raw JDBC. The web3 service caches the result of `ContractRepository.findRuntimeBytecode()` in a local Caffeine cache with `expireAfterAccess=1h`. Because the two services share only the database and there is no cross-service cache invalidation, any DB update made by the importer is invisible to the web3 cache. If incorrect bytecode is written and then a user query populates the cache before the DB is corrected, the web3 service will serve the stale incorrect bytecode for up to one hour — indefinitely under continuous query load.

## Finding Description

**Code path 1 — importer writes without any cache interaction:**

`SidecarContractMigration.migrate()` executes a raw JDBC upsert:

```java
// SidecarContractMigration.java lines 26-30 / 44-48
private static final String UPDATE_RUNTIME_BYTECODE_SQL = """
        insert into contract (id, runtime_bytecode)
        values (?, ?)
        on conflict (id)
        do update set runtime_bytecode = excluded.runtime_bytecode""";

jdbcOperations.batchUpdate(
        UPDATE_RUNTIME_BYTECODE_SQL, contractBytecodes, BATCH_SIZE, (ps, contractBytecode) -> {
            ps.setLong(1, EntityId.of(contractBytecode.getContractId()).getId());
            ps.setBytes(2, DomainUtils.toBytes(contractBytecode.getRuntimeBytecode()));
        });
``` [1](#0-0) 

This is called during live record stream ingestion from `ContractResultServiceImpl.processSidecarRecords()` whenever a sidecar record carries `migration = true`. [7](#0-6) 

**Code path 2 — the cached query in the web3 service:**

```java
// ContractRepository.java lines 16-18
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [2](#0-1) 

`unless = "#result == null"` only suppresses caching a Java `null`. An `Optional.empty()` (non-null object) and any non-null `Optional<byte[]>` — including one wrapping malformed bytes — is unconditionally stored in the cache.

**Cache TTL:**

```java
// CacheProperties.java line 22
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
``` [3](#0-2) 

`expireAfterAccess=1h` resets the TTL on every read. Under continuous query load the incorrect entry never expires.

**No eviction anywhere for contract bytecode:**

A codebase-wide search for `@CacheEvict` returns only two hits, both in `AddressBookServiceImpl`, entirely unrelated to contract bytecode. [5](#0-4) 

**Downstream consumer:**

`ContractBytecodeReadableKVState.readFromDataSource()` passes the cached bytes directly into EVM execution with no validation:

```java
return contractRepository
        .findRuntimeBytecode(entityId.getId())
        .map(Bytes::wrap)
        .map(Bytecode::new)
        .orElse(null);
``` [4](#0-3) 

## Impact Explanation
The web3 service serves incorrect `runtime_bytecode` for EVM simulation:
- `eth_getCode` returns wrong bytecode.
- `eth_call` and `eth_estimateGas` execute against wrong bytecode, producing incorrect results or spurious reverts.
- EIP-1167 proxy detection and `EXTCODEHASH` return wrong values.
- The incorrect state persists for the full cache lifetime — up to indefinitely under load — with no operator-visible signal and no self-healing path short of a service restart or cache expiry.

## Likelihood Explanation
The precondition is a malformed sidecar record reaching the importer. This can occur via a bug in a consensus node producing a malformed `TransactionSidecarRecord`, a compromised or Byzantine consensus node injecting bad bytecode, or a replay/fuzzing attack against the importer's record stream ingestion. Once the malformed record is processed and a single unauthenticated `eth_getCode` or `eth_call` is issued for the affected contract, the incorrect bytecode is locked into the cache. The trigger requires zero privilege from the external user.

## Recommendation
1. **Add a cross-service cache invalidation channel.** Publish a cache-invalidation event (e.g., via Redis pub/sub or a lightweight notification table) from the importer after each `SidecarContractMigration.migrate()` call; have the web3 service subscribe and evict the affected contract IDs.
2. **Switch the contract cache to `expireAfterWrite` instead of `expireAfterAccess`.** `expireAfterWrite=N` bounds the staleness window regardless of query traffic, preventing indefinite persistence.
3. **Reduce the contract cache TTL** to a value consistent with the expected migration frequency (e.g., 60 s instead of 1 h).
4. **Add a `@CacheEvict` or programmatic eviction** in any code path that writes `runtime_bytecode` to the database, if the importer and web3 are ever co-located in the same JVM.

## Proof of Concept
1. Start the importer and web3 services against a shared PostgreSQL instance.
2. Inject a `TransactionSidecarRecord` with `migration = true` and a truncated/zeroed `runtime_bytecode` for contract ID `X` into the record stream.
3. `ContractResultServiceImpl.processSidecarRecords()` calls `sidecarContractMigration.migrate()`, which upserts the malformed bytecode for `X` into the `contract` table.
4. Issue `eth_getCode` for `X` against the web3 service. The cache misses, the DB returns the malformed bytes, and they are stored in the Caffeine cache under key `X`.
5. Inject a second `TransactionSidecarRecord` with `migration = true` and the correct `runtime_bytecode` for `X`. The importer upserts the correct bytes into the DB.
6. Issue `eth_getCode` for `X` again. The web3 service returns the malformed bytes from cache, not the corrected DB value.
7. Continue issuing `eth_getCode` for `X` at any interval shorter than 1 hour. The `expireAfterAccess` TTL resets on each hit; the malformed bytecode is served indefinitely.
