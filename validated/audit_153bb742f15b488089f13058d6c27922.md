Audit Report

## Title
Unprivileged User Can Silently Suppress Balance Snapshot Results via `timestamp=ne:` Operator in `/api/v1/balances`

## Summary
The `/api/v1/balances` endpoint accepts the `ne` (not-equal) operator for the `timestamp` query parameter. Because `filterValidityChecks` for `TIMESTAMP` only validates the value format and not the operator, and because `getBalances` uses `parseTimestampQueryParam` (which bypasses the `parseTimestampFilters` guard that would reject `ne`), an attacker can craft a request that excludes the only balance snapshot in a given range, causing the API to return a silent empty response indistinguishable from a genuine data gap.

## Finding Description

**Validation gap — `filterValidityChecks` does not block `ne` for `TIMESTAMP`:**

In `rest/utils.js`, the `filterValidityChecks` function handles `TIMESTAMP` as:

```js
case constants.filterKeys.TIMESTAMP:
  ret = isValidTimestampParam(val);
  break;
``` [1](#0-0) 

This only validates the value format. The `validateOpAndValue` check at the top of `filterValidityChecks` accepts `ne` because `isValidOperatorQuery` matches `/^(gte?|lte?|eq|ne)$/`: [2](#0-1) [3](#0-2) 

The `balanceFilterValidator` in `rest/balances.js` delegates to `filterValidityChecks` for all non-`ACCOUNT_ID` parameters, including `TIMESTAMP`: [4](#0-3) 

**`parseTimestampFilters` guard is bypassed:** `getBalances` uses `parseTimestampQueryParam`, not `parseTimestampFilters`. The guard in `parseTimestampFilters` that throws `'Not equals operator not supported for timestamp param'` when `allowNe=false` is never reached for this code path: [5](#0-4) [6](#0-5) [7](#0-6) 

**`getOptimizedTimestampRange` accepts and propagates `ne`:** [8](#0-7) 

**`getAccountBalanceTimestampRange` injects the exclusion into SQL:** [9](#0-8) 

If `Z` is the only snapshot timestamp for the treasury account in `[lowerBound, upperBound]`, the query returns zero rows: [10](#0-9) 

**Silent empty response propagation:** `getTsQuery` returns `{}` → `!tsQueryResult.query` is truthy → `getBalances` returns early, leaving `res.locals` as `{timestamp: null, balances: [], links: {next: null}}`: [11](#0-10) [12](#0-11) 

**Contrast with the Java REST layer**, which explicitly rejects `ne` for timestamp: [13](#0-12) 

The same `getAccountBalanceTimestampRange` is also called from `rest/accounts.js` (single-account balance lookup) and `rest/tokens.js` (token balances), making those endpoints equally affected: [14](#0-13) [15](#0-14) 

No test spec for the balances endpoint covers `ne` timestamp rejection, confirming the gap is untested:


## Impact Explanation
The attacker causes the API to return `{timestamp: null, balances: [], links: {next: null}}` — the same response as when no balance data exists. There is no error, no warning, and no indication that a `ne` filter suppressed valid data. Any downstream consumer (monitoring dashboard, alerting system, client application) treating this as a genuine data absence will draw incorrect conclusions. During a network partition or incident investigation, this manufactured empty response is indistinguishable from a real data gap, undermining the integrity of balance data availability signals.

## Likelihood Explanation
No privileges are required. The attacker only needs to know a valid snapshot timestamp `Z` in the target range, which is trivially discoverable by first querying the same endpoint without the `ne` filter to observe the returned `timestamp` field. The attack is repeatable, requires a single HTTP request, and leaves no distinguishing trace in the response.

## Recommendation
Add an explicit operator check for `TIMESTAMP` in `filterValidityChecks` in `rest/utils.js`, restricting it to `basicOperators` (which excludes `ne`), consistent with how `BLOCK_NUMBER` and `SLOT` are handled:

```js
case constants.filterKeys.TIMESTAMP:
  ret = isValidTimestampParam(val) && includes(basicOperators, op);
  break;
```

Alternatively, add a dedicated check in `getBalances` (and the affected paths in `accounts.js` and `tokens.js`) that rejects `ne` for `timestamp` before calling `parseTimestampQueryParam`, mirroring the Java layer's behavior in `TimestampParameter.java`.

## Proof of Concept

1. Query the endpoint to discover a snapshot timestamp `Z`:
   ```
   GET /api/v1/balances?timestamp=lte:1700000000.000000000
   → Response: {"timestamp": "1699999800.000000000", ...}
   ```

2. Craft a request that excludes `Z` as the only snapshot in the range:
   ```
   GET /api/v1/balances?timestamp=gte:1699999700.000000000&timestamp=lte:1699999900.000000000&timestamp=ne:1699999800.000000000
   ```

3. Observe the silent empty response:
   ```json
   {"timestamp": null, "balances": [], "links": {"next": null}}
   ```

This is identical to the response when no balance data exists, with no error or indication that valid data was suppressed.
