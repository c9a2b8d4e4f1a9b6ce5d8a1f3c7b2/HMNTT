**Audit Report**

## Title
SQL NULL Handling Bug in `findByEvmAddress` Causes Active Entities with `deleted = NULL` to Be Silently Skipped, Triggering Recoverable Errors on Cache-Cold Lookups

## Summary
The `findByEvmAddress` query in the importer's `EntityRepository` uses `deleted <> true`, which in PostgreSQL evaluates `NULL <> true` as `NULL` (not `TRUE`), causing entities with `deleted IS NULL` to be excluded from results. This is inconsistent with `EntityIdServiceImpl.notify()`, which explicitly treats `deleted = NULL` as an active entity and caches it. After a mirror node restart (cold cache), any transaction referencing such an entity by EVM address will fail to resolve it, triggering `handleRecoverableError` and producing incorrect transaction processing or a full importer halt.

## Finding Description

**Confirmed vulnerable query** — `importer/src/main/java/org/hiero/mirror/importer/repository/EntityRepository.java`, line 19:
```java
@Query(value = "select id from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Long> findByEvmAddress(byte[] evmAddress);
``` [1](#0-0) 

In PostgreSQL, `NULL <> true` evaluates to `NULL`, not `TRUE`. A `WHERE` clause only passes rows where the predicate is `TRUE`. Therefore, any entity row where `deleted IS NULL` is silently excluded.

**Correct predicate used in other modules** — `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 16:
```java
@Query(value = "select * from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
``` [2](#0-1) 

And in `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, line 47:
```java
where (evm_address = ?1 or alias = ?1) and deleted is not true
``` [3](#0-2) 

**Inconsistency with `notify()`** — `importer/src/main/java/org/hiero/mirror/importer/domain/EntityIdServiceImpl.java`, line 138:
```java
if (entity == null || (entity.getDeleted() != null && entity.getDeleted())) {
    return;  // only skip if deleted == true
}
// ... cache the entity
``` [4](#0-3) 

This means on a warm cache (populated via `notify()`), entities with `deleted = NULL` are found correctly. On a cold cache (post-restart), the DB query is used and these entities are invisible.

**Call chain to error** — `EntityIdServiceImpl.java`, lines 169–177:
```java
private Optional<EntityId> findByEvmAddress(byte[] evmAddress, boolean throwRecoverableError) {
    var id = Optional.ofNullable(DomainUtils.fromEvmAddress(evmAddress))
            .or(() -> entityRepository.findByEvmAddress(evmAddress).map(EntityId::of));
    if (id.isEmpty() && throwRecoverableError) {
        Utility.handleRecoverableError("Entity not found for EVM address {}", Hex.encodeHexString(evmAddress));
    }
    return id;
}
``` [5](#0-4) 

**Test gap confirmed** — `importer/src/test/java/org/hiero/mirror/importer/repository/EntityRepositoryTest.java`, lines 97–107 only tests `deleted=true` and `deleted=false`; no `deleted=NULL` case: [6](#0-5) 

The `rest-java` module's equivalent test **does** cover `deleted=NULL` and explicitly asserts it returns empty — confirming the behavior is real and reproducible: [7](#0-6) 

## Impact Explanation

When `handleRecoverableError` is invoked with the default configuration (`HIERO_MIRROR_IMPORTER_PARSER_HALTONERROR=false`), the importer logs an ERROR and continues, processing the transaction with a missing/empty entity ID. This corrupts the mirror node's view of that transaction — contract call results, token transfers, and contract actions that reference the affected entity will be stored with a null/empty entity reference instead of the correct one. [8](#0-7) 

If `HIERO_MIRROR_IMPORTER_PARSER_HALTONERROR=true`, the importer throws `ParserException` and halts entirely, causing a denial-of-service for all subsequent transaction ingestion until the node is manually restarted. [9](#0-8) 

## Likelihood Explanation

**Precondition:** At least one entity in the database has `deleted = NULL`. This is a realistic steady-state: the `rest-java` integration test explicitly constructs such entities, confirming the schema allows it. Newly ingested entities whose creation transaction has been processed but whose `deleted` column was never explicitly set to `false` satisfy this condition.

**Trigger:** The mirror node restarts (cache is cold). This is a routine operational event (deployments, crashes, rolling restarts).

**Repeatability:** The bug fires on every cache-miss lookup for every affected entity, on every restart, indefinitely until the SQL is fixed.

## Recommendation

Change the `deleted <> true` predicate in the importer's `EntityRepository` to `deleted is not true`, consistent with the `graphql` and `web3` modules:

```java
// importer EntityRepository.java
@Query(value = "select id from entity where evm_address = ?1 and deleted is not true", nativeQuery = true)
Optional<Long> findByEvmAddress(byte[] evmAddress);

@Query(value = "select id from entity where alias = ?1 and deleted is not true", nativeQuery = true)
Optional<Long> findByAlias(byte[] alias);
```

Note that `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java` line 16–17 has the same `deleted <> true` bug and should be fixed identically. [10](#0-9) 

Add a test case for `deleted=NULL` to `importer/src/test/java/org/hiero/mirror/importer/repository/EntityRepositoryTest.java` mirroring the existing `rest-java` test pattern.

## Proof of Concept

1. Insert an entity into the mirror node DB with `deleted = NULL` and a known `evm_address` (e.g., via normal Hedera contract creation where the `deleted` column is never explicitly set to `false`).
2. Restart the mirror node importer (cold cache — `notify()` has not been called for this entity).
3. Submit a `ContractCall` transaction on the Hedera network targeting the entity's EVM address as a `ContractID.EVM_ADDRESS`.
4. The importer processes the transaction: `lookup(ContractID)` → `cacheLookup(...)` → `findByEvmAddress(bytes, true)` → `entityRepository.findByEvmAddress(evmAddress)` returns `Optional.empty()` (SQL `NULL <> true` excludes the row).
5. `Utility.handleRecoverableError(...)` is invoked.
   - With default config: ERROR is logged, transaction is stored with `contractId = 0` (corrupted).
   - With `HIERO_MIRROR_IMPORTER_PARSER_HALTONERROR=true`: `ParserException` is thrown, importer halts.
6. Verify by querying the DB: the stored contract result has a null/empty entity reference instead of the correct entity ID.

    public static void handleRecoverableError(String message, Object... args) {
        var haltOnError = Boolean.parseBoolean(System.getProperty(HALT_ON_ERROR_PROPERTY));

        if (haltOnError) {
            var formattingTuple = MessageFormatter.arrayFormat(message, args);
            var throwable = formattingTuple.getThrowable();
            var formattedMessage = formattingTuple.getMessage();
            throw new ParserException(formattedMessage, throwable);
        } else {
            log.error(RECOVERABLE_ERROR + message, args);
        }
    }
