Audit Report

## Title
Unbounded Concurrent Subscriptions Trigger Unthrottled DB Polling via Safety-Check Path, Enabling DoS

## Summary
Any unauthenticated gRPC client can open an unlimited number of `subscribeTopic` streams. Each stream without an `endTime` unconditionally fires a one-shot safety check after one second that invokes the unthrottled retriever path, which polls the database up to 13 times at 20 ms intervals with a page size of 5,000 rows. Because there is no cap on concurrent subscribers, an attacker multiplies this load linearly across many TCP connections, exhausting database connections and CPU.

## Finding Description

**1. Unthrottled path parameters confirmed** (`RetrieverProperties.UnthrottledProperties`): [1](#0-0) 

Defaults: `maxPageSize = 5000`, `maxPolls = 12`, `pollingFrequency = 20ms`.

**2. Safety check is unconditionally scheduled for every subscription:** [2](#0-1) 

`Mono.delay(Duration.ofSeconds(1L))` fires once per subscription. The guard `!topicContext.isComplete()` does not prevent it for the common case (see point 3).

**3. `TopicContext.isComplete()` always returns `false` when `endTime == null`:** [3](#0-2) 

A client omitting `consensusEndTime` (the default per the proto definition) guarantees the safety check fires.

**4. Safety check calls the unthrottled retriever path:** [4](#0-3) 

When `current == null` (safety-check trigger), `topicMessageRetriever.retrieve(gapFilter, false)` is called — the unthrottled path.

**5. Unthrottled `PollingContext.isComplete()` only terminates on `limitHit`:** [5](#0-4) 

Without a client-supplied `limit`, `hasLimit()` returns `false`, so `limitHit` is always `false`. The retriever polls all `maxPolls` (12) iterations at 20 ms with 5,000-row pages — 13 total DB queries per safety-check invocation.

**6. `subscriberCount` is a metric gauge only — no enforcement:** [6](#0-5) 

There is no check against `subscriberCount` before accepting a new subscription.

**7. The only per-connection cap is `maxConcurrentCallsPerConnection = 5`:** [7](#0-6) [8](#0-7) 

An attacker opens N TCP connections, each carrying 5 streams, bypassing this limit entirely.

## Impact Explanation
Each subscription without an `endTime` generates up to 13 database queries (1 initial + 12 repeats) at 20 ms intervals, each fetching up to 5,000 rows. With N concurrent connections × 5 streams each, the attacker produces `N × 5 × 13 = 65N` database queries in approximately 260 ms. At N=100 connections (trivially achievable), that is 6,500 queries in a quarter-second burst, capable of saturating the HikariCP connection pool and driving CPU to 100%, denying service to legitimate users. The `timeout` of 60 s means each subscription holds resources for up to a minute before expiring. [9](#0-8) 

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint requires no authentication: [10](#0-9) 

The proto definition confirms `consensusEndTime` is optional and defaults to indefinite streaming: [11](#0-10) 

Any attacker with network access can script hundreds of concurrent `grpcurl` or SDK calls. No special knowledge, credentials, or chain state is required. The attack is repeatable and can be sustained indefinitely by reconnecting after the 60 s timeout.

## Recommendation
1. **Enforce a global subscriber cap**: Check `subscriberCount` against a configurable maximum before accepting new subscriptions in `TopicMessageServiceImpl.subscribeTopic()`, returning `RESOURCE_EXHAUSTED` when exceeded.
2. **Rate-limit the safety check per source IP or connection**: Prevent a single client from triggering many concurrent unthrottled retrievals.
3. **Apply per-IP connection limits** at the Netty/load-balancer layer to complement `maxConcurrentCallsPerConnection`.
4. **Reduce unthrottled defaults or gate them**: Consider requiring a client-supplied `limit` before allowing the unthrottled path, or reduce `maxPolls`/`maxPageSize` defaults.

## Proof of Concept
```bash
# Open 100 parallel connections, each with 5 concurrent streams (500 total subscriptions)
# Each stream omits consensusEndTime, guaranteeing the safety check fires
for i in $(seq 1 100); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{"topicID": {"topicNum": 1}}' \
      <mirror-node-host>:5600 \
      com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
# After ~1 second: 500 safety checks fire, each triggering up to 13 unthrottled DB queries
# = 6,500 queries in ~260ms, saturating the connection pool
```

    @Override
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }
