Audit Report

## Title
Unvalidated `X-Forwarded-For` / `X-Real-IP` Headers Allow Arbitrary String Injection into Client IP Logs in `TracingMiddleware`

## Summary
`getClientIpAddress()` in `rosetta/app/middleware/trace.go` returns the raw value of `X-Real-IP` or `X-Forwarded-For` without any IP format validation. An unauthenticated attacker can supply any string — including `attacker:8080`, a spoofed localhost address, or a newline-containing payload — which `TracingMiddleware` logs verbatim as the authoritative client IP for every Rosetta API request, including `/construction/submit`.

## Finding Description
**File:** `rosetta/app/middleware/trace.go`

```go
// lines 63–74
func getClientIpAddress(r *http.Request) string {
    ipAddress := r.Header.Get(xRealIpHeader)        // raw, no validation

    if len(ipAddress) == 0 {
        ipAddress = r.Header.Get(xForwardedForHeader) // raw, no validation
    }

    if len(ipAddress) == 0 {
        ipAddress, _, _ = net.SplitHostPort(r.RemoteAddr) // only fallback validates
    }

    return ipAddress
}
```

`net.SplitHostPort` — which strips the port and validates structure — is called **only** on `r.RemoteAddr`, the OS-provided TCP address the attacker cannot forge. For both header paths the raw string is returned after a sole `len == 0` guard. The result is consumed at line 46 and emitted at lines 52–58 via `log.Info` / `log.Debug`.

The test suite (`trace_test.go` lines 52–58) only exercises a well-formed IP (`10.0.0.100`) in `X-Forwarded-For`; the malformed case is untested.

## Impact Explanation
- **Log injection (persistent):** A value containing `\n` can inject synthetic log lines (e.g., a fake successful submission from a spoofed IP), corrupting audit trails stored in any log aggregator or SIEM.
- **Security monitoring bypass:** IP-based threat-intelligence lookups, geolocation, and rate-limit correlation all fail silently when the "IP" field contains a non-IP token such as `attacker:8080`.
- **Real IP never recorded:** The TCP `RemoteAddr` fallback is bypassed entirely whenever either header is present, so the attacker's actual IP is never logged.

## Likelihood Explanation
- No authentication or privilege is required; the Rosetta Construction API is designed for external callers.
- Trivially reproducible with any HTTP client (`curl -H "X-Forwarded-For: attacker:8080" ...`).
- Repeatable at will — every request with the header set produces a poisoned log entry.
- No existing middleware control (rate limiting, WAF) is assumed or present in the code path.

## Recommendation
In `getClientIpAddress()`, validate the extracted string with `net.ParseIP` before accepting it. For multi-value `X-Forwarded-For` lists, split on `,` and take the first token before parsing. Fall through to `r.RemoteAddr` if the header value does not parse as a valid IP:

```go
func getClientIpAddress(r *http.Request) string {
    for _, header := range []string{xRealIpHeader, xForwardedForHeader} {
        raw := r.Header.Get(header)
        if raw == "" {
            continue
        }
        // X-Forwarded-For may be a comma-separated list; take the first entry
        candidate := strings.TrimSpace(strings.SplitN(raw, ",", 2)[0])
        if ip := net.ParseIP(candidate); ip != nil {
            return ip.String()
        }
    }
    ip, _, _ := net.SplitHostPort(r.RemoteAddr)
    return ip
}
```

Add test cases for malformed values (`"attacker:8080"`, `"1.2.3.4, 5.6.7.8"`, `"127.0.0.1\nINJECTED"`) to prevent regression.

## Proof of Concept
```bash
curl -s -X POST http://<rosetta-host>/construction/submit \
     -H "Content-Type: application/json" \
     -H "X-Forwarded-For: attacker:8080" \
     -d '{"network_identifier":{},"signed_transaction":""}'
```

Expected log output (before fix):
```
level=info msg="attacker:8080 POST /construction/submit (200) in 3ms"
```

The real TCP `RemoteAddr` is never consulted; the attacker's actual IP is absent from the log.
