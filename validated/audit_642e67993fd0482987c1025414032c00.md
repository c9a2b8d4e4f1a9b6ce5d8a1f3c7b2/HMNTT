Audit Report

## Title
32-bit `Arrays.hashCode` Cache Key Collision Enables Alias Cache Poisoning in `findByEvmAddressOrAliasAndDeletedIsFalse`

## Summary
`SpelHelper.hashCode(byte[] value)` returns `Arrays.hashCode(value)`, a 32-bit integer, which is used as the sole cache key for the `CACHE_NAME_ALIAS` Caffeine cache. Two distinct byte-array aliases that produce the same 32-bit hash value will map to the same cache slot. An attacker who registers a Hedera account whose alias hash-collides with a target entity's alias can poison the cache so that subsequent lookups for the target return the attacker's entity instead.

## Finding Description

**Exact code locations:**

`SpelHelper.hashCode` delegates directly to `Arrays.hashCode`: [1](#0-0) 

This 32-bit integer is used verbatim as the cache key for `CACHE_NAME_ALIAS`: [2](#0-1) 

The entity cache manager registers `CACHE_NAME_ALIAS` with a shared Caffeine spec of `expireAfterWrite=1s,maximumSize=10000`: [3](#0-2) [4](#0-3) 

**Root cause:**

The SpEL expression `@spelHelper.hashCode(#alias)` evaluates to a Java `int` (autoboxed to `Integer`). The Caffeine cache (backed by `ConcurrentHashMap`) uses `Integer.equals()` for key comparison. Because the key stored in the cache IS the integer value itself, two distinct byte arrays `A` and `B` satisfying `Arrays.hashCode(A) == Arrays.hashCode(B)` will map to the same cache entry. Whichever is cached first will be returned for both lookups. There is no post-retrieval check comparing the returned entity's `evm_address` or `alias` field against the requested byte array, and `unless = "#result == null"` only prevents caching null results.

**Exploit flow:**

1. Attacker identifies target entity with alias `A` (e.g., a 20-byte EVM address, publicly visible on-chain).
2. Attacker generates ECDSA key pairs, computing derived EVM addresses until finding alias `B ≠ A` such that `Arrays.hashCode(B) == Arrays.hashCode(A)`. The 32-bit output space makes this feasible (~2³² evaluations, achievable in minutes on commodity hardware).
3. Attacker registers a Hedera account with alias `B` (small HBAR fee).
4. Attacker submits a contract call causing the mirror node to invoke `findByEvmAddressOrAliasAndDeletedIsFalse(B)`. This hits the DB, returns entity B, and stores it in `CACHE_NAME_ALIAS` under key `hash(B) = hash(A)`.
5. Within the same 1-second TTL window, any call resolving the target entity by alias `A` receives entity B from cache without a DB round-trip.
6. `CommonEntityAccessor.get(Bytes alias, Optional<Long> timestamp)` returns entity B for alias A: [5](#0-4) 

## Impact Explanation

When a fee-collecting or otherwise significant entity is resolved by alias during EVM simulation (e.g., via `AliasesReadableKVState.readFromDataSource` or `AccountReadableKVState.readFromDataSource` calling `commonEntityAccessor.get`), the wrong entity is returned: [6](#0-5) [7](#0-6) 

Consequences include incorrect EVM simulation results (wrong account balances, wrong fee parameters, wrong `allCollectorsAreExempt` flags) returned to callers of `eth_call` and `eth_estimateGas`. The window is bounded by the 1-second TTL, but an attacker can continuously re-trigger the poisoning request to sustain the effect.

## Likelihood Explanation

- **Unprivileged**: No special role required; any Hedera account holder can submit contract calls and register accounts.
- **Computational cost**: Finding a 32-bit `Arrays.hashCode` preimage for a 20-byte EVM address target requires ~2³² hash evaluations, achievable in under a minute on a GPU.
- **On-chain cost**: Registering the colliding account costs a small HBAR fee, making the attack economically viable.
- **Repeatability**: The attacker must re-poison every ~1 second due to the TTL, but a single automated script can sustain this indefinitely.
- **Targeting**: The target entity's alias bytes are publicly visible on-chain.

Overall likelihood: **low-medium** — technically feasible for a motivated attacker, but requires upfront computation and continuous re-triggering.

## Recommendation

Replace the 32-bit `Arrays.hashCode` cache key with a collision-resistant key. Options:
1. Use the full byte array as the cache key directly (Spring's `SimpleKeyGenerator` handles arrays by value if wrapped, or use `java.util.Arrays.toString(alias)` / `HexFormat.of().formatHex(alias)` as a string key).
2. Use a cryptographic hash (e.g., `Arrays.toString` of a SHA-256 digest) to make collisions computationally infeasible.
3. Add a post-cache equality check that validates the returned entity's `evm_address`/`alias` field matches the requested byte array before returning it.

The same issue applies to `findByEvmAddressAndDeletedIsFalse` which uses the identical key expression: [8](#0-7) 

## Proof of Concept

```java
// Demonstrate collision: find B such that Arrays.hashCode(B) == Arrays.hashCode(A)
byte[] A = hexToBytes("deadbeefdeadbeefdeadbeefdeadbeef01020304"); // target 20-byte alias
int targetHash = Arrays.hashCode(A);

// Brute-force: generate random 20-byte EVM addresses (from ECDSA key pairs)
// until Arrays.hashCode(candidate) == targetHash
byte[] B = null;
Random rng = new Random();
for (long i = 0; i < 0xFFFFFFFFL; i++) {
    byte[] candidate = new byte[20];
    rng.nextBytes(candidate);
    if (Arrays.hashCode(candidate) == targetHash && !Arrays.equals(candidate, A)) {
        B = candidate;
        break;
    }
}
// B != A but Arrays.hashCode(B) == Arrays.hashCode(A)
// Register Hedera account with alias B, then trigger mirror node lookup for B
// before lookup for A — A's lookup will return B's entity from cache.
```
