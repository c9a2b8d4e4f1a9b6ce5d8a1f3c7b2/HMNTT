# Audit Report

## Title
Blocking JDBC Call in Reactive Context Enables Thread Pool Exhaustion DoS During Network Partition

## Summary
`TopicMessageServiceImpl.topicExists()` executes a blocking JDBC call (`entityRepository.findById()`) synchronously on the gRPC executor thread before any reactive pipeline is assembled. Because the gRPC server uses Spring Boot's finite `applicationTaskExecutor` thread pool and imposes no total connection limit, an attacker can exhaust all threads by opening many connections and flooding `subscribeTopic()` with unique, uncached topic IDs — especially effective during a network partition where TCP-level hangs far outlast the DB statement timeout.

## Finding Description

**Exact code path:**

In `TopicMessageServiceImpl.java`, `topicExists()` at line 96:
```java
return Mono.justOrEmpty(entityRepository.findById(topicId.getId()))
```
`entityRepository.findById(topicId.getId())` is passed as an **eager argument** to `Mono.justOrEmpty()`. Java evaluates method arguments before the method call, so the blocking JDBC query executes synchronously on the calling thread — the gRPC executor thread — before the `Mono` is even constructed.

`subscribeTopic()` calls `topicExists(filter)` directly at line 87 with no `Mono.defer()` or `subscribeOn(Schedulers.boundedElastic())` wrapper, so there is no deferred or off-thread execution.

**Thread model:** `GrpcConfiguration.java` line 32 binds all gRPC calls to `applicationTaskExecutor` (Spring Boot's `ThreadPoolTaskExecutor`, finite platform threads). Line 33 sets `maxConcurrentCallsPerConnection` but no total connection limit is configured.

**Cache bypass:** `EntityRepository.findById` is `@Cacheable` with `unless="#result == null"`. Because the return type is `Optional<Entity>`, the result is never `null` — `Optional.empty()` is cached after the first miss. An attacker using a rotating set of unique, never-before-seen topic IDs ensures every request is a cache miss and hits the DB.

**Network partition amplification:** The `statementTimeout=10000ms` is a PostgreSQL-level directive requiring a live DB connection to enforce. During a full TCP-level partition, the JDBC driver's socket hangs until OS TCP keepalive/retransmit fires (typically 2–15 minutes), far exceeding the statement timeout. Each blocked thread holds its `applicationTaskExecutor` slot for the full TCP timeout duration.

## Impact Explanation
Complete DoS of the gRPC mirror node service. Because all gRPC calls share the same `applicationTaskExecutor`, thread pool exhaustion starves unrelated operations (address book queries, health probes) as well. Downstream clients (wallets, dApps, consensus monitors) lose real-time topic message delivery. No authentication is required. Severity: **High** (full availability impact, no data exfiltration).

## Likelihood Explanation
The attacker requires only: (1) network access to the gRPC port, (2) a gRPC client library (freely available), and (3) the ability to open many TCP connections — standard for any internet-connected host. The network partition is a precondition the attacker does not need to cause; degraded DB connectivity is observable via increased response latency. The attack is repeatable and requires no credentials or knowledge of the topic namespace beyond knowing sequential IDs will be cache misses.

## Recommendation
Wrap the blocking `entityRepository.findById()` call inside `Mono.fromCallable()` and subscribe it on `Schedulers.boundedElastic()` to move the blocking I/O off the gRPC executor thread:
```java
private Mono<?> topicExists(TopicMessageFilter filter) {
    var topicId = filter.getTopicId();
    return Mono.fromCallable(() -> entityRepository.findById(topicId.getId()))
            .subscribeOn(Schedulers.boundedElastic())
            .flatMap(Mono::justOrEmpty)
            ...
}
```
Additionally, consider:
- Configuring a total gRPC connection limit (`maxConnectionAge`, `maxConnectionIdle`, or a Netty connection limit).
- Adding a per-IP rate limit on `subscribeTopic()` calls.
- Setting an explicit JDBC socket connect timeout (separate from statement timeout) to bound TCP-level hangs.

## Proof of Concept
1. Induce or wait for degraded DB connectivity (observable via increased `subscribeTopic` latency).
2. Open K gRPC connections to port 5600 (no server-side connection limit).
3. On each connection, send `maxConcurrentCallsPerConnection` concurrent `subscribeTopic` RPCs, each with a distinct, never-before-seen topic ID (e.g., sequential integers beyond any real topic ID to guarantee cache misses).
4. Each call blocks its `applicationTaskExecutor` thread in `entityRepository.findById()` waiting for a TCP timeout (2–15 minutes during full partition).
5. With K × `maxConcurrentCallsPerConnection` threads blocked, the thread pool is exhausted.
6. All subsequent gRPC requests (including health checks) queue indefinitely — service appears hung.
