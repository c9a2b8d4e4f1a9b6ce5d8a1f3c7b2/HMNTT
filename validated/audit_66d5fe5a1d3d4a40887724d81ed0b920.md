Audit Report

## Title
Empty `Optional` Cache Poisoning in `ContractRepository.findRuntimeBytecode()` Enables Persistent Contract DoS

## Summary
The `@Cacheable` annotation on `findRuntimeBytecode()` uses `unless = "#result == null"`, which only prevents caching of literal `null`. Because `Optional.empty()` is a non-null Java singleton, a query for a not-yet-ingested contract stores an empty `Optional` in the Caffeine cache. The contract cache is configured with `expireAfterAccess=1h`, so the poisoned entry persists — and resets its TTL on every access — making the contract invisible to the EVM for at least one hour per attacker request.

## Finding Description

**Exact code location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18: [1](#0-0) 

The `unless` SpEL expression `#result == null` evaluates to `false` when the result is `Optional.empty()`, because `Optional.empty()` is a non-null singleton object. Spring therefore stores the empty `Optional` in the Caffeine cache.

**Cache configuration:**

`web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`, line 22: [2](#0-1) 

The policy is `expireAfterAccess=1h` (not `expireAfterWrite`), so every subsequent read of the poisoned entry resets the 1-hour clock.

**Downstream impact path:**

`web3/src/main/java/org/hiero/mirror/web3/state/keyvalue/ContractBytecodeReadableKVState.java`, lines 42–46: [3](#0-2) 

`readFromDataSource()` calls `contractRepository.findRuntimeBytecode(entityId.getId())` and maps the result to a `Bytecode` object, returning `null` when the `Optional` is empty. A `null` return causes the EVM to treat the contract as non-existent.

**Why the existing check fails:**

`unless = "#result == null"` is the only guard. There is no check for `!#result.isPresent()`. The test `ContractRepositoryTest.findRuntimeBytecodeFailCall()` confirms that `findRuntimeBytecode` returns `Optional.empty()` (not `null`) for a missing contract: [4](#0-3) 

Additionally, `findRuntimeBytecodeSuccessfulCall()` demonstrates that querying a non-existent contract (after `deleteAll()`) returns `isEmpty()`, confirming the empty-Optional path is live: [5](#0-4) 

## Impact Explanation
Any contract whose `contractId` is queried before the mirror node has ingested its `runtime_bytecode` row will have an empty `Optional` cached. All subsequent `eth_call` / `eth_estimateGas` requests for that contract will receive EVM execution failures (contract treated as non-existent) for up to 1 hour — or indefinitely if the attacker periodically re-queries the same ID. This is a targeted, per-contract denial-of-service with no authentication required. [6](#0-5) 

## Likelihood Explanation
Hedera contract IDs are sequential and publicly observable on the network. An attacker can:
1. Watch the consensus stream for a `ContractCreate` transaction.
2. Immediately send an `eth_call` to the mirror node for the new contract ID before the mirror node's ingestion pipeline has written the `runtime_bytecode` row.
3. The mirror node's ingestion lag (typically seconds to tens of seconds) provides a reliable race window.

No privileged access, no special tooling, and no on-chain funds are required. The attack is repeatable and scriptable. [7](#0-6) 

## Recommendation
Change the `unless` condition in `ContractRepository.findRuntimeBytecode()` to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only a populated `Optional<byte[]>` (i.e., a contract with actual bytecode) is stored in the cache. A miss-query for a not-yet-ingested contract will fall through to the database on every request until the bytecode is available. [1](#0-0) 

## Proof of Concept

```
1. New contract C is created on-chain with ID = X.
   Mirror node ingestion lag: ~5–30 seconds.

2. Attacker sends:
     POST /api/v1/contracts/call
     { "to": "0x<long-zero-address-of-X>", "data": "0x..." }
   before the mirror node writes the runtime_bytecode row.

3. ContractRepository.findRuntimeBytecode(X) hits the DB → row not found
   → returns Optional.empty()
   → unless="#result == null" evaluates to FALSE (Optional.empty() is non-null)
   → Spring caches Optional.empty() in the Caffeine "contract" cache under key X.

4. Mirror node ingests the contract; runtime_bytecode row now exists in DB.

5. All subsequent eth_call / eth_estimateGas for contract X:
   → Spring cache HIT → returns cached Optional.empty()
   → ContractBytecodeReadableKVState.readFromDataSource() returns null
   → EVM treats contract X as non-existent → REVERT on every call

6. Attacker sends a lightweight read request for X every ~55 minutes
   → resets the expireAfterAccess=1h TTL
   → cache poisoning persists indefinitely.
``` [3](#0-2) [8](#0-7)
