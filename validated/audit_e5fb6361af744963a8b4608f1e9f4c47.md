# Audit Report

## Title
Stale Contract Bytecode Cache Causes `eth_call` Simulations to Execute Against Pre-Upgrade Bytecode

## Summary
`ContractRepository.findRuntimeBytecode()` is cached with `expireAfterAccess=1h`. Because `expireAfterAccess` resets on every read, a frequently queried contract's bytecode entry never expires after an on-chain upgrade. No `@CacheEvict` or invalidation mechanism exists for this cache anywhere in the codebase. Any caller submitting an `eth_call` to a recently upgraded contract will receive simulation results computed against the old bytecode for an unbounded duration.

## Finding Description

**Verified code path:**

`TransactionExecutionService.execute()` at line 78 calls `transactionExecutorFactory.get()` to obtain an executor. [1](#0-0) 

During EVM execution, bytecode is resolved through `ContractBytecodeReadableKVState.readFromDataSource()`: [2](#0-1) 

`ContractRepository.findRuntimeBytecode()` is declared with `@Cacheable` using the contract cache manager, with `unless = "#result == null"` as the only guard: [3](#0-2) 

The backing Caffeine cache is configured as `expireAfterAccess=1h,maximumSize=1000,recordStats`: [4](#0-3) 

The cache manager is wired in `EvmConfiguration.cacheManagerContract()`: [5](#0-4) 

**Confirmed absence of invalidation:** A full codebase search for `@CacheEvict` returns hits only in `AddressBookServiceImpl.java` (importer module). There is zero `@CacheEvict` usage anywhere for the contract bytecode cache. [6](#0-5) 

**Root cause:** `expireAfterAccess` resets the idle timer on every cache hit. For any contract receiving steady `eth_call` traffic, the cache entry is perpetually refreshed and never evicted after an upgrade. The `unless = "#result == null"` guard only prevents caching null results; it does not handle bytecode replacement.

## Impact Explanation
Any caller using `eth_call` to simulate contract interactions against a recently upgraded contract will receive incorrect return values, incorrect revert/success signals, and incorrect gas estimates for the full duration the stale entry remains in cache — indefinitely for contracts with ongoing traffic. This can cause callers to submit real transactions with incorrect parameters, approve incorrect token amounts, or incorrectly conclude a function will succeed when it will revert under the upgraded logic.

## Likelihood Explanation
No attacker capability is required beyond the ability to send an HTTP request to the mirror node's JSON-RPC endpoint. The condition — a contract upgrade followed by an `eth_call` within the cache window — is routine in production. The `expireAfterAccess` policy makes the stale window unbounded for any contract with ongoing traffic. The behavior is deterministic and repeatable.

## Recommendation
Replace `expireAfterAccess=1h` with `expireAfterWrite=Xs` (a short TTL, e.g., `expireAfterWrite=10s`) for the contract bytecode cache. `expireAfterWrite` does not reset on reads, so the cache entry will expire a fixed time after it was last populated, bounding the staleness window regardless of traffic volume. Alternatively, add a `@CacheEvict` on the contract bytecode cache triggered by the importer's contract update processing path to proactively invalidate entries when new bytecode is written to the `contract` table. [7](#0-6) 

## Proof of Concept
1. Deploy contract `C` on the Hedera network; confirm `eth_call` returns value `V1` under the original bytecode.
2. Submit a `ContractUpdateTransaction` upgrading contract `C`; the importer writes new `runtime_bytecode` to the `contract` table.
3. Immediately after the upgrade, submit an `eth_call` targeting contract `C` at least once per hour to keep the cache entry alive.
4. Observe that `ContractRepository.findRuntimeBytecode()` returns the cached pre-upgrade bytecode (cache hit, no DB query).
5. The EVM simulation executes against the old bytecode and returns `V1` instead of the post-upgrade value `V2`.
6. Confirm the DB now contains the new bytecode while the cache still serves the old one — the discrepancy persists indefinitely as long as the contract receives at least one `eth_call` per hour.
