Audit Report

## Title
Unbounded `ServiceEndpoints` from Mirror REST Server Causes SDK Connection Exhaustion

## Summary
`fetchMirrorNodeNetwork()` in `pinger/mirror_node_client.go` places no upper bound on the number of nodes or `ServiceEndpoints` entries returned by the mirror REST server, and applies no body-size limit to the HTTP response. A malicious or compromised mirror REST server can return a response with thousands of synthetic endpoints, inflating the `network` map passed to `hiero.ClientForNetwork()` and causing the Hiero SDK to attempt gRPC connections to every entry, exhausting OS file descriptors and connection-pool resources.

## Finding Description
In `pinger/mirror_node_client.go`, `fetchMirrorNodeNetwork()` (lines 74–131) decodes the mirror REST response with no body-size cap:

```go
// lines 95–98
var payload nodesEnvelope
if err := json.NewDecoder(resp.Body).Decode(&payload); err != nil { ... }
```

`resp.Body` is read directly with no `io.LimitReader`, so an arbitrarily large JSON body is fully consumed into memory. The inner loop (lines 113–124) then iterates every `ServiceEndpoint` for every node with no bounds check:

```go
for _, ep := range n.ServiceEndpoints {          // line 113 — unbounded
    host := strings.TrimSpace(ep.DomainName)
    if host == "" { host = strings.TrimSpace(ep.IPAddressV4) }
    if host == "" || ep.Port == 0 { continue }   // line 118 — trivially bypassed
    addr := net.JoinHostPort(host, fmt.Sprintf("%d", ep.Port))
    network[addr] = nodeAccountId               // line 122 — map grows without limit
}
```

The only guard (`host == "" || ep.Port == 0`) is trivially bypassed by supplying valid-looking but attacker-controlled `host:port` pairs. The resulting `network` map — potentially containing tens of thousands of entries — is returned to `newClient()` in `sdk_client.go` (line 22) and passed directly to `hiero.ClientForNetwork(netmap)`, which causes the SDK to attempt gRPC connections to every entry.

**Root cause:** The code assumes the mirror REST server is trusted and returns a reasonable number of endpoints. There is no `io.LimitReader` on the response body, no cap on `len(payload.Nodes)`, and no cap on `len(n.ServiceEndpoints)`.

## Impact Explanation
When the SDK receives a map with thousands of fake `host:port` entries, it attempts to establish gRPC connections to all of them. This exhausts OS-level file descriptors and goroutine/connection-pool resources. Real consensus nodes are either never reached (all connection slots consumed by fake endpoints) or are reached only after extreme latency, constituting a de facto network partition. The pinger's health-check purpose is completely defeated. Severity: **High** — full denial of service of the pinger's network connectivity with no recovery until restart with a trusted mirror URL.

## Likelihood Explanation
The precondition is control over the mirror REST endpoint configured in `cfg.mirrorRest`. This is achievable by:
- Operating a malicious or misconfigured mirror node that the pinger is pointed at (zero privilege on the pinger host required).
- DNS poisoning or BGP hijacking of the mirror REST hostname.
- A MITM on an unencrypted or improperly validated TLS connection to the mirror REST server.

No authentication or signature verification is performed on the mirror REST response. The attack is repeatable on every startup or retry cycle (lines 52–69 in `buildNetworkFromMirrorNodes`).

## Recommendation
1. **Limit response body size** — wrap `resp.Body` with `io.LimitReader` before decoding, e.g.:
   ```go
   const maxBodyBytes = 1 << 20 // 1 MiB
   if err := json.NewDecoder(io.LimitReader(resp.Body, maxBodyBytes)).Decode(&payload); err != nil { ... }
   ```
2. **Cap the number of nodes** — after decoding, reject or truncate `payload.Nodes` if `len(payload.Nodes)` exceeds a reasonable constant (e.g., 100).
3. **Cap endpoints per node** — inside the loop, break after a maximum number of `ServiceEndpoints` per node (e.g., 10).
4. **Validate port range** — reject endpoints whose port is outside the valid gRPC range (1–65535) or not in an expected set.
5. **Pin or verify the mirror REST URL** — use TLS with certificate pinning or at minimum enforce HTTPS for `cfg.mirrorRest`.

## Proof of Concept
A controlled mirror REST server returns the following JSON at `/api/v1/network/nodes`:

```json
{
  "nodes": [
    {
      "node_account_id": "0.0.3",
      "service_endpoints": [
        {"domain_name": "attacker1.example.com", "port": 50211},
        {"domain_name": "attacker2.example.com", "port": 50211},
        ... // repeated 50,000 times
      ]
    }
  ],
  "links": {"next": null}
}
```

When the pinger starts with `cfg.network = "other"` and `cfg.mirrorRest` pointing to this server:
1. `buildNetworkFromMirrorNodes` calls `fetchMirrorNodeNetwork`, which decodes the full 50,000-entry response into memory.
2. The inner loop populates `network` with 50,000 `attacker*.example.com:50211 → 0.0.3` entries.
3. `newClient()` passes this map to `hiero.ClientForNetwork(netmap)`.
4. The SDK attempts gRPC connections to all 50,000 endpoints, exhausting file descriptors and goroutine resources.
5. The pinger fails to reach any real consensus node and becomes permanently non-functional until restarted.
