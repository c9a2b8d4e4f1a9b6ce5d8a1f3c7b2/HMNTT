# Audit Report

## Title
Undeclared `statementTimeout` for rest-java DSLContext Enables Connection-Pool Starvation via Unbounded Query Duration

## Summary
The `hiero.mirror.restJava.db.statementTimeout` property is documented with a default of 10 000 ms but is never read or applied anywhere in the rest-java application. Every jOOQ query issued through the injected `DSLContext` — including `NftAllowanceRepositoryCustomImpl.findAll` — runs without an application-enforced time bound. The sole real guard is a PostgreSQL role-level `statement_timeout` set by the Helm chart, which is absent in non-Helm deployments and is 20 seconds even when present.

## Finding Description
**Exact code path:**

`NftAllowanceRepositoryCustomImpl.java` lines 41–46 issues a jOOQ query directly via the injected `DSLContext` with no timeout:

```java
return dslContext
        .selectFrom(NFT_ALLOWANCE)
        .where(condition)
        .orderBy(SORT_ORDERS.get(new OrderSpec(byOwner, request.getOrder())))
        .limit(request.getLimit())
        .fetchInto(NftAllowance.class);
```

**Root cause — `statementTimeout` is a dead property for rest-java:**

`RestJavaConfiguration.java` line 38 is the only `DefaultConfigurationCustomizer` registration:

```java
return c -> c.set(domainRecordMapperProvider).settings().withRenderSchema(false);
```

No `Settings.withQueryTimeout`, `ExecuteListener` with a timeout, or `connectionInitSql` carrying `SET statement_timeout` is configured anywhere. A search for `statementTimeout`, `connectionInitSql`, `statement_timeout`, `queryTimeout`, or `setQueryTimeout` across the entire `rest-java` source tree returns **zero matches**.

`CommonConfiguration.java` lines 60–95 builds `HikariDataSource` from `spring.datasource.hikari` properties only; no `connectionInitSql` is present. The `rest-java/src/main/resources/` directory contains only `banner.txt` — no `application.yml` that could supply one.

**Failed assumption:** Operators and reviewers reading `docs/configuration.md` line 630 assume `hiero.mirror.restJava.db.statementTimeout` is enforced at the application layer. It is not. The only real guard is the PostgreSQL role-level `statement_timeout` set by `secret-passwords.yaml` line 125 (`alter user ... set statement_timeout to '20000'`), which is absent in non-Helm deployments and is 20 seconds even when present.

## Impact Explanation
HikariCP's default maximum pool size is 10 connections. With no application-enforced timeout, each request to the NFT allowances endpoint holds a connection for the full duration of the query — up to 20 s in Helm deployments, or indefinitely in bare deployments. Sending 10–15 concurrent slow requests exhausts the pool. Subsequent requests block waiting for a connection until HikariCP's `connectionTimeout` (default 30 s) expires, at which point they throw `SQLTransientConnectionException`. This partitions the rest-java service from the database for **all** endpoints sharing the same pool — a full service-level DoS, not limited to the NFT allowances endpoint.

## Likelihood Explanation
No authentication is required to call the NFT allowances endpoint. The `limit` parameter caps the result-set row count but does not bound query execution time; complex `getBoundConditions` predicates on a large `nft_allowance` table can produce slow sequential scans regardless of the row limit. A single attacker with a script sending ~15 concurrent HTTP requests can reliably trigger pool exhaustion. The attack is repeatable and requires no special knowledge beyond the public API schema.

## Recommendation
1. **Apply the documented property**: Read `hiero.mirror.restJava.db.statementTimeout` and set it on the jOOQ `Settings` via `withQueryTimeout(...)` inside the `DefaultConfigurationCustomizer` in `RestJavaConfiguration.java`, or set it on the `HikariConfig` via `connectionInitSql = "SET statement_timeout = <value>"` in `CommonConfiguration.java`.
2. **Verify Helm-only deployments**: Ensure the PostgreSQL role-level `statement_timeout` is also enforced in non-Helm deployment paths.
3. **Add integration test**: Assert that a query exceeding the configured timeout raises `QueryTimeoutException` and is handled by `GenericControllerAdvice.queryTimeout`.

## Proof of Concept
```bash
# Send 15 concurrent requests to the NFT allowances endpoint
# (replace ACCOUNT_ID with a valid account that has many NFT allowances)
for i in $(seq 1 15); do
  curl -s "http://<host>/api/v1/accounts/<ACCOUNT_ID>/allowances/nfts?limit=100" &
done
wait
# Subsequent requests will fail with 503 Service Unavailable
# (SQLTransientConnectionException mapped by GenericControllerAdvice)
curl -v "http://<host>/api/v1/accounts/<ACCOUNT_ID>/allowances/nfts"
```
In a non-Helm deployment (no PostgreSQL role-level timeout), connections are held indefinitely until the pool is fully exhausted and all subsequent requests time out at the HikariCP `connectionTimeout` boundary.

    DataSource dataSource(
            DataSourceProperties dataSourceProperties,
            HikariConfig hikariConfig,
            DatabaseWaiter databaseWaiter,
            ObjectProvider<JdbcConnectionDetails> detailsProvider) {

        var jdbcUrl = dataSourceProperties.determineUrl();
        var username = dataSourceProperties.determineUsername();
        var password = dataSourceProperties.determinePassword();

        final var connectionDetails = detailsProvider.getIfAvailable();
        if (connectionDetails != null) {
            jdbcUrl = connectionDetails.getJdbcUrl();
            username = connectionDetails.getUsername();
            password = connectionDetails.getPassword();
        }

        databaseWaiter.waitForDatabase(jdbcUrl, username, password);

        final var config = new HikariConfig();
        hikariConfig.copyStateTo(config);
        config.setJdbcUrl(jdbcUrl);
        config.setUsername(username);
        config.setPassword(password);

        return new HikariDataSource(config);
    }
