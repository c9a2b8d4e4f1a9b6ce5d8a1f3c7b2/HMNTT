Audit Report

## Title
Shared Poll Flux Re-subscription Resets `lastConsensusTimestamp` to `now()`, Causing Live Message Loss

## Summary
In `SharedPollingTopicListener`, the single shared `topicMessages` Flux places `.doOnSubscribe(context::onStart)` upstream of `.share()`. Because `.share()` (`publish().refCount()`) cancels the upstream when the last subscriber disconnects and re-subscribes when the first new subscriber arrives, `onStart` fires on every such re-subscription and unconditionally resets `lastConsensusTimestamp` to `DomainUtils.now()`. Any messages written to the database during the gap between the last subscriber's disconnect and the next subscriber's connect are permanently skipped by the shared poll loop.

## Finding Description

**Verified code path:**

`SharedPollingTopicListener` constructor (lines 41–52) builds a single `topicMessages` Flux:

```java
topicMessages = Flux.defer(() -> poll(context).subscribeOn(scheduler))
        .repeatWhen(...)
        ...
        .doOnSubscribe(context::onStart)   // fires on every upstream subscription
        .retryWhen(...)
        .share();                           // publish().refCount()
``` [1](#0-0) 

`getSharedListener` returns this single Flux for every caller regardless of filter: [2](#0-1) 

`PollingContext.onStart` unconditionally resets `lastConsensusTimestamp` to `DomainUtils.now()` with no "already initialized" guard: [3](#0-2) 

`poll` uses `lastConsensusTimestamp` as the lower bound for every DB query: [4](#0-3) 

`ConsensusController.subscribeTopic` has no authentication: [5](#0-4) 

**Root cause:** `.doOnSubscribe` is placed before `.share()` in the operator chain. In Reactor, `.share()` (`publish().refCount()`) cancels the upstream when the subscriber count drops to zero and re-subscribes when the first new subscriber arrives. Each re-subscription triggers `doOnSubscribe` → `onStart` → `lastConsensusTimestamp = now()`. There is no guard to skip the reset if the context was already initialized.

**Why existing mitigations are insufficient:**

1. **Safety check** (`TopicMessageServiceImpl` lines 67–70): fires once after 1 second and calls `missingMessages(topicContext, null)`. When the new subscriber has received no messages yet (`last == null`), `missingMessages` uses `filter.getStartTime()` as the query lower bound. [6](#0-5) 
The default `startTime` in `TopicMessageFilter` is `DomainUtils.now()`: [7](#0-6) 
So the safety check queries from `now()`, not from before the gap, and does not recover the skipped messages.

2. **Historical retriever**: covers messages up to the subscriber's `startTime` (= `now()` by default). The gap messages fall between historical's end and the shared listener's new cursor and are never delivered.

3. **`missingMessages` gap-fill** (line 120): triggered only when a live message is received from the shared listener. If the shared listener skips the gap entirely, no trigger fires. [8](#0-7) 

## Impact Explanation
Any messages written to the database during the window between the last subscriber's disconnect and the next subscriber's connect are permanently skipped by the shared poll loop. Subscribers relying on `SHARED_POLL` mode for live delivery will silently miss those messages with no error, no retry, and no indication of data loss. HCS consumers will observe a gap in sequence numbers that the per-subscriber gap-fill mechanism cannot recover because it is never triggered.

## Likelihood Explanation
The gRPC endpoint requires no authentication. Any external network client can open and immediately close a `subscribeTopic` stream. The attack is trivially repeatable: the attacker cycles connect/disconnect to continuously reset the cursor, maximizing the gap window. No special knowledge beyond a valid topic ID (public on-chain data) is required.

## Recommendation
Add an "already initialized" guard in `PollingContext.onStart` so that `lastConsensusTimestamp` is only set to `now()` on the very first subscription, not on re-subscriptions:

```java
void onStart(Subscription subscription) {
    lastConsensusTimestamp.compareAndSet(0L, DomainUtils.now()); // only set if uninitialized
    log.info("Starting to poll every {}ms", listenerProperties.getInterval().toMillis());
}
```

Alternatively, move `.doOnSubscribe(context::onStart)` to inside the `Flux.defer(...)` lambda so it only fires on each individual poll cycle rather than on upstream re-subscription, or persist `lastConsensusTimestamp` outside the `PollingContext` so it survives upstream cancellation and re-subscription.

## Proof of Concept

```bash
# 1. Attacker becomes sole subscriber
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' <host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
ATTACKER_PID=$!

# 2. Wait for poll loop to advance lastConsensusTimestamp
sleep 5

# 3. Attacker disconnects — subscriber count drops to 0, upstream cancelled
kill $ATTACKER_PID

# 4. Messages arrive in DB during this gap (normal network activity)
sleep 3

# 5. Legitimate subscriber connects — upstream re-subscribed, onStart fires,
#    lastConsensusTimestamp = now(), gap messages are permanently skipped
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' <host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

The legitimate subscriber will never receive messages that arrived during step 4. No error is reported.
