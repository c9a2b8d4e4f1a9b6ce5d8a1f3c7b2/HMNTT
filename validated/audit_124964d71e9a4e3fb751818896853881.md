# Audit Report

## Title
Cancel-Before-Retry Race Condition in `RedisTopicListener.subscribe()` Enables Subscription Storm During Network Partition

## Summary
In `RedisTopicListener.subscribe()`, `doOnCancel(() -> unsubscribe(topic))` is placed upstream of `retryWhen(...)` in the reactive operator chain. Because `retryWhen` only intercepts `onError` signals and does not absorb cancellation signals, a cancel from `.share()` (triggered when all gRPC subscribers leave) propagates upstream through `retryWhen`, fires `unsubscribe(topic)`, and removes the topic from `topicMessages`. This defeats the retry logic entirely and creates a remove-then-reinsert cycle on every client reconnect attempt during a Redis outage.

## Finding Description

The operator chain in `subscribe()` is confirmed as:

```java
return container
    .flatMapMany(r -> r.receive(...))
    .map(Message::getMessage)
    .doOnCancel(() -> unsubscribe(topic))    // line 74 — upstream of retryWhen
    .doOnComplete(() -> unsubscribe(topic))  // line 75 — upstream of retryWhen
    .doOnError(t -> log.error(...))
    .doOnSubscribe(...)
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L))) // line 78
    .share();                                                                                  // line 79
``` [1](#0-0) 

`getSharedListener()` caches the shared Flux via `computeIfAbsent`: [2](#0-1) 

`unsubscribe()` removes the entry from the map: [3](#0-2) 

**Root cause:** In Project Reactor, `retryWhen` only intercepts `onError` terminal signals from upstream and converts them into re-subscription attempts. It does **not** intercept or absorb cancellation signals. A `cancel` propagated from downstream (e.g., from `.share()` when its subscriber count drops to zero) flows upstream unconditionally, past `retryWhen`, and reaches `doOnCancel`. The developer's assumption — that `retryWhen` would keep the Flux alive in `topicMessages` during Redis reconnection — is incorrect for the cancellation path.

**Exploit flow:**
1. gRPC clients subscribe to a topic; `computeIfAbsent` creates and stores a shared Flux.
2. Redis connection fails; `retryWhen` begins exponential backoff — the Flux is still in `topicMessages`.
3. The gRPC transport layer detects the partition and cancels all active streams for that topic.
4. `.share()` observes zero remaining subscribers and cancels its upstream subscription.
5. The cancel signal propagates upstream through `retryWhen` → reaches `doOnCancel` → fires `unsubscribe(topic)` → `topicMessages.remove(topic.getTopic())`.
6. A gRPC client retries. `computeIfAbsent` finds no entry and calls `subscribe(topic)` again, creating a new Flux.
7. The new Flux immediately attempts to connect to Redis (still partitioned), fails, and `retryWhen` starts another backoff.
8. If the gRPC layer cancels again before Redis recovers, step 5 repeats. Each cycle creates a new Flux, a new Redis connection attempt, and a new backoff timer with no deduplication or rate-limiting at the map level.

**Why existing checks are insufficient:**
- `retryWhen(Retry.backoff(...))` only intercepts `onError`. A `cancel` signal is not an error; it flows upstream unconditionally and is not absorbed. [4](#0-3) 
- `ConcurrentHashMap` at line 43 ensures thread-safe map operations but provides no protection against the remove-then-reinsert cycle. [5](#0-4) 
- There is no subscriber-count guard or cooldown on `computeIfAbsent` re-entry after `unsubscribe`. [6](#0-5) 

## Impact Explanation

During a Redis network partition, the server enters a tight loop: each client reconnect attempt creates a new `ReactiveRedisMessageListenerContainer.receive()` call, which fails, which (upon the next cancel) removes the map entry, which the next retry recreates. With `N` clients each retrying aggressively, this produces `N` concurrent Flux lifecycles per topic. This exhausts Redis connection pool resources, thread pool capacity in `Schedulers.boundedElastic()`, and generates log spam that can mask the real partition event. After the partition heals, the storm continues until all in-flight retry cycles settle, delaying message delivery resumption. Severity: **High** — service degradation during the exact failure mode (network partition) where reliability is most critical.

## Likelihood Explanation

No attacker privileges are required beyond being a valid gRPC subscriber. The trigger (network partition) can be induced by any attacker with network-layer access (BGP manipulation, firewall rule injection, ARP poisoning on the Redis segment). Even without a malicious actor, this fires naturally during routine Redis maintenance, rolling restarts, or cloud provider network events. The gRPC SDK supports aggressive retry behavior (as demonstrated by `setMaxAttempts(Integer.MAX_VALUE)` usage in the test acceptance client at `StartupProbe.java` line 116), making the storm self-sustaining once triggered. [7](#0-6) 

## Recommendation

Move `doOnCancel` and `doOnComplete` to be placed **after** `retryWhen` in the operator chain, so that cancellation only triggers `unsubscribe` after the retry logic has had a chance to run. Alternatively, use `doFinally` after `retryWhen` to handle both cancel and complete in a single operator that is correctly positioned downstream of the retry logic:

```java
return container
    .flatMapMany(r -> r.receive(...))
    .map(Message::getMessage)
    .doOnError(t -> log.error("Error listening for messages", t))
    .doOnSubscribe(s -> log.info("Creating shared subscription to {}", topic))
    .retryWhen(Retry.backoff(Long.MAX_VALUE, interval).maxBackoff(interval.multipliedBy(4L)))
    .doFinally(signal -> {
        if (signal == SignalType.CANCEL || signal == SignalType.ON_COMPLETE) {
            unsubscribe(topic);
        }
    })
    .share();
```

Additionally, consider adding a minimum cooldown or reference-count guard before allowing `computeIfAbsent` to re-create a Flux entry immediately after removal, to prevent rapid cycling during sustained outages.

## Proof of Concept

1. Start the mirror node with `listener.type=REDIS`.
2. Subscribe one or more gRPC clients to a topic.
3. Simulate a Redis network partition (e.g., `iptables -A OUTPUT -p tcp --dport 6379 -j DROP`).
4. Observe `retryWhen` begin backoff in logs.
5. Simulate gRPC transport timeout/cancel (e.g., let the gRPC keepalive timeout expire, or kill the client and immediately restart it).
6. Observe in logs: `Unsubscribing from topic.X` followed immediately by `Creating shared subscription to topic.X` — confirming the map entry was removed and recreated.
7. Repeat step 5 rapidly. Each cycle produces a new `receive()` call against the still-partitioned Redis, a new backoff timer, and a new log entry — with no rate-limiting between cycles.
8. Restore the Redis connection (`iptables -D OUTPUT -p tcp --dport 6379 -j DROP`) and observe delayed recovery as all in-flight retry cycles settle.
