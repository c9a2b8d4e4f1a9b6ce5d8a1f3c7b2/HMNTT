Audit Report

## Title
Unauthenticated DB Amplification via `findStorageBatch()` Batch Query Growth Bypasses Per-Node Gas Throttle

## Summary
The gas throttle in `ThrottleManagerImpl` is a per-node, in-memory token bucket that limits gas consumption before EVM execution but places no bound on the number of database queries generated per unit of gas consumed. A contract that performs many unique SLOAD operations causes `ContractStateServiceImpl.findStorage()` to invoke `findStorageBatch()` on every value-cache miss — each call issuing a DB query whose `IN (...)` clause grows with every new slot key accumulated in the `slotsPerContract` cache (up to 1,500 entries). At the default throttle of `gasPerSecond = 7,500,000,000` with `maxGasLimit = 15,000,000`, an unprivileged attacker can sustain up to 500 requests/second, each generating up to 1,500 DB queries, producing a sustained load that can saturate the shared PostgreSQL backend.

## Finding Description

**Exact code path:**

`ContractController.call()` → `ThrottleManagerImpl.throttle()` (gas consumed pre-execution) → `ContractExecutionService.processCall()` → EVM executes SLOADs → `ContractStorageReadableKVState.readFromDataSource()` → `ContractStateServiceImpl.findStorage()` → `ContractStateServiceImpl.findStorageBatch()` → `ContractStateRepository.findStorageBatch()` (uncached DB query).

**Gas throttle does not bound DB queries per gas unit:**

`ThrottleManagerImpl.throttle()` deducts `scaleGas(request.getGas())` tokens from the in-memory `gasLimitBucket` before execution begins: [1](#0-0) 

`scaleGas` divides by `GAS_SCALE_FACTOR = 10_000`: [2](#0-1) 

The bucket is built from `getGasPerSecond()` which itself calls `scaleGas(gasPerSecond)`: [3](#0-2) 

With the code default `gasPerSecond = 7_500_000_000L`, the bucket holds `scaleGas(7_500_000_000) = 750,000` tokens. Each request at `gas = 15,000,000` consumes `scaleGas(15,000,000) = 1,500` tokens → **500 requests/second** pass the throttle. [4](#0-3) 

**DB amplification in `findStorageBatch()`:**

`findStorage()` checks `contractStateCache` (TTL = 2s). On a miss it calls `findStorageBatch()`: [5](#0-4) 

`findStorageBatch()` accumulates the new slot key into `contractSlotsCache`, then reads **all** accumulated keys and issues a single batch DB query with all of them: [6](#0-5) 

`findStorageBatch` on the repository has **no `@Cacheable` annotation** — it always hits the DB: [7](#0-6) 

Compare with `findStorage` (single-key), which is `@Cacheable`: [8](#0-7) 

Each unique SLOAD in a request that misses `contractStateCache` triggers one `findStorageBatch` call whose `IN (...)` clause grows with every new slot key accumulated in `slotsPerContract` (max 1,500 entries, `expireAfterAccess=5m`). For a contract with 1,500 pre-seeded unique slots, every cache-miss SLOAD issues a query with an ever-growing key list.

**`contractState` cache TTL = 2s creates a repeating attack window:** [9](#0-8) 

Every 2 seconds the value cache expires. The slot-key cache (`slotsPerContract`, TTL = 5 min) remains warm: [10](#0-9) 

So every 2 seconds the attacker's requests re-trigger the full batch-query load against a cold value cache.

**Gas restore does not mitigate the attack:**

For a successful call that consumes all gas, `gasRemaining = 0` and `restore(Math.min(0, gasLimitToRestoreBaseline)) = restore(0)` — nothing is returned to the bucket: [11](#0-10) 

**The throttle is per-node and in-memory:** [12](#0-11) 

This is a JVM-local `Bucket`. Each mirror node has its own independent bucket. There is no per-IP rate limiting in the controller or filter chain. [13](#0-12) 

## Impact Explanation

At 500 req/s (actual default) with a contract that accesses 1,500 unique storage slots per call (cost: 1,500 × 2,100 = 3,150,000 gas, well within 15M):

- **DB queries per second**: 500 req/s × 1,500 `findStorageBatch` calls/req = **750,000 DB queries/second**, with `IN` clauses growing from 1 to 1,500 byte-array keys.
- The shared PostgreSQL backend (`statementTimeout = 3,000 ms`) will begin queuing and timing out queries, causing cascading failures across all mirror nodes sharing the same DB instance.
- Because the throttle is per-node and the DB is shared, a single attacker targeting one node can degrade all nodes. Targeting multiple nodes multiplies the effect linearly.
- No authentication, API key, or account is required.

## Likelihood Explanation

- **Precondition**: A contract with many storage slots exists on the network (trivially true on mainnet/testnet; attacker can also deploy one).
- **Skill required**: Sending HTTP POST requests to `/api/v1/contracts/call` with `gas: 15000000` and calldata that loops over SLOAD opcodes. This is standard Solidity/web3 tooling.
- **Repeatability**: Fully automated and repeatable. The attack sustains itself as long as the attacker sends requests at the allowed rate.
- **No cost**: `eth_call` is a read-only simulation; no on-chain transaction or HBAR fee is required.

## Recommendation

1. **Add `@Cacheable` to `findStorageBatch`** or cache its results in `contractStateCache` after execution to prevent repeated DB hits for the same slot set within the TTL window.
2. **Bound DB queries per request**: Limit the number of `findStorageBatch` calls per single EVM execution (e.g., cap at a configurable maximum per request).
3. **Increase `contractState` cache TTL** or use a write-through strategy so that slot values populated by a batch query survive longer than 2 seconds.
4. **Add per-IP or per-source rate limiting** at the HTTP layer (e.g., via a servlet filter or API gateway) to complement the per-node gas throttle.
5. **Consider a distributed throttle** (e.g., Redis-backed Bucket4j) so that the gas budget is shared across all mirror nodes rather than being independent per JVM.

## Proof of Concept

1. Deploy a contract on testnet with 1,500 unique storage slots pre-populated.
2. Write a Solidity function that SLOADs all 1,500 slots in a loop.
3. Send POST requests to `/api/v1/contracts/call` at the maximum allowed rate (up to 500 req/s at default config):
```json
{
  "to": "<contract_address>",
  "data": "<calldata_for_sload_loop>",
  "gas": 15000000
}
```
4. After the first request, the `slotsPerContract` cache is warm with 1,500 keys. After 2 seconds, `contractStateCache` expires.
5. Each subsequent request triggers 1,500 `findStorageBatch` DB queries (one per SLOAD miss), each with a growing `IN` clause up to 1,500 keys.
6. Monitor PostgreSQL `pg_stat_activity` to observe query queue depth and statement timeouts increasing, degrading all mirror nodes sharing the DB.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
