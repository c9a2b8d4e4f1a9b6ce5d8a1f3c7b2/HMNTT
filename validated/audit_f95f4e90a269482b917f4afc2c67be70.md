## Audit Report

## Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Unique `contractId` Requests Causes JVM Heap Exhaustion

## Summary
`ContractStateServiceImpl.findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` for every unique `contractId` not already in the outer `contractSlotsCache`. Because `cacheManagerSlotsPerContract` is a `CaffeineCacheManager` configured without `setCacheNames()` (dynamic mode), its internal `ConcurrentHashMap<String, Cache> cacheMap` grows without any bound. The outer `contractSlotsCache` (capped at 3,000 entries) does not trigger any cleanup in `cacheManagerSlotsPerContract.cacheMap` when it evicts entries, so every unique `contractId` ever seen permanently occupies a slot in that map. An unauthenticated attacker can exhaust JVM heap by sending requests referencing a large number of distinct contract addresses.

## Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorageBatch()` at lines 85‚Äì87:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [1](#0-0) 

The outer `contractSlotsCache` is the `"default"` cache from `cacheManagerContractSlots`, configured with `maximumSize=3000`: [2](#0-1) 

The inner `cacheManagerSlotsPerContract` bean is declared **without** `setCacheNames()`: [3](#0-2) 

Omitting `setCacheNames()` leaves `CaffeineCacheManager` in *dynamic* mode. Spring's `CaffeineCacheManager.getCache(name)` stores every newly created cache in an internal `ConcurrentHashMap<String, Cache> cacheMap` with no capacity limit. Once a `CaffeineCache` is inserted for a given name it is **never removed** from that map ‚Äî there is no eviction listener, no TTL on the map itself, and no size bound.

**Root cause and failed assumption:**

The design assumes the outer `contractSlotsCache` (bounded to 3,000) acts as the sole growth limiter. It does not. When the outer cache evicts an entry for `contractId=X`, it removes the `EntityId ‚Üí CaffeineCache` mapping from its own Caffeine table, but `cacheManagerSlotsPerContract.cacheMap` still holds the strong reference `"X" ‚Üí CaffeineCache`. The `slotsPerContract` spec `"expireAfterAccess=5m,maximumSize=1500"` governs entries *within* each per-contract cache, not the number of caches registered in the manager: [4](#0-3) 

**Exploit flow:**

1. Attacker sends `POST /api/v1/contracts/call` with a unique `to` address per request.
2. Each request reaches `ContractStorageReadableKVState.readFromDataSource()`, which calls `contractStateService.findStorage(entityId, keyBytes)`: [5](#0-4) 
3. `findStorage` checks `contractStateCache` (miss for a new contractId), then calls `findStorageBatch`: [6](#0-5) 
4. `findStorageBatch` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())`, inserting a new `CaffeineCache` object into the unbounded internal map.
5. After the outer `contractSlotsCache` fills to 3,000 and starts evicting, the inner map continues to grow monotonically ‚Äî one entry per unique contractId, never reclaimed.

**Why existing checks are insufficient:**

- The outer `contractSlotsCache` `maximumSize=3000` only bounds the outer map; it has no eviction listener wired to `cacheManagerSlotsPerContract`.
- The `slotsPerContract` TTL/size spec applies to slot-key entries *inside* each per-contract cache, not to the cache objects themselves.
- The rate limiter (500 req/s) slows accumulation but does not cap the total number of unique contractIds seen over the lifetime of the process: [7](#0-6) 
- No authentication is required on the `/api/v1/contracts/call` endpoint.

## Impact Explanation
Each `CaffeineCache` instance allocated by `cacheManagerSlotsPerContract.getCache()` carries its own Caffeine internal data structures (segment arrays, reference queues, stats counters). At 500 req/s sustained with unique contract addresses, an attacker accumulates cache objects at the maximum allowed rate. This leads to progressive JVM heap exhaustion, ultimately triggering `OutOfMemoryError` and crashing the web3 service ‚Äî a complete denial of service for all users of the mirror node's EVM/JSON-RPC interface. [3](#0-2) 

## Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special knowledge beyond the public API. The attacker only needs to vary the `to` field across requests. The default rate limit of 500 req/s is generous; even at a fraction of that rate the leak is persistent and cumulative across the process lifetime. The attack is fully repeatable and automatable with a simple script. [1](#0-0) 

## Recommendation
Register a fixed set of cache names in `cacheManagerSlotsPerContract` using `setCacheNames()`, or replace the two-level cache design with a single bounded Caffeine cache keyed by `(contractId, slotKey)`. If dynamic per-contract caches are required, add an eviction listener on the outer `contractSlotsCache` that calls `cacheManagerSlotsPerContract` to invalidate/remove the corresponding inner cache entry when the outer entry is evicted, ensuring the inner map is bounded by the same 3,000-entry limit as the outer cache. [8](#0-7) 

## Proof of Concept
```python
import requests, threading

BASE = "http://<mirror-node-host>/api/v1/contracts/call"
HEADERS = {"Content-Type": "application/json"}

def send(i):
    # Each request uses a unique 'to' address derived from counter i
    addr = "0x" + hex(i)[2:].zfill(40)
    requests.post(BASE, json={"to": addr, "data": "0x", "gas": 300000}, headers=HEADERS)

# Flood with unique contract addresses at the allowed rate
threads = []
for i in range(1_000_000):
    t = threading.Thread(target=send, args=(i,))
    threads.append(t)
    t.start()
    if len(threads) % 500 == 0:
        for t in threads: t.join()
        threads = []
```

Each iteration inserts a new `CaffeineCache` into `cacheManagerSlotsPerContract.cacheMap`. The outer `contractSlotsCache` evicts entries after 3,000 are reached, but the inner map never shrinks. Sustained execution causes monotonically increasing heap usage until `OutOfMemoryError`. [9](#0-8)
