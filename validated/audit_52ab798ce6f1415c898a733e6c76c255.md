Audit Report

## Title
Unauthenticated Alias-Miss DoS via Missing Null Guard and Double DB Query in `getBalances()`

## Summary
Any unauthenticated caller can supply a syntactically valid but non-existent account alias in `account.id=eq:<alias>` to `/api/v1/balances`. Due to a missing null guard after alias resolution, `getBalances()` unconditionally executes two sequential DB queries per request even when the alias does not exist. With no rate limiting and no alias-lookup caching, flooding this endpoint with concurrent requests doubles the effective DB load per request, exhausting the 10-connection default pool and starving legitimate users.

## Finding Description

**Alias resolution — first DB query:**

In `rest/balances.js`, `parseAccountIdQueryParam` (lines 320–357) handles the `account.id` filter. When the value passes `AccountAlias.isValid(value, true)`, it calls:

```js
return EntityService.getAccountIdFromAlias(AccountAlias.fromString(value), false);
``` [10](#0-9) 

In `rest/service/entityService.js`, `getAccountIdFromAlias` (lines 71–81) calls `getAccountFromAlias`, which issues a live DB query:

```js
const rows = await super.getRows(EntityService.entityFromAliasQuery, [accountAlias.alias]);
``` [11](#0-10) 

`BaseService.getRows` calls `pool.queryQuietly` directly with no cache: [3](#0-2) 

When no row is found, `getAccountFromAlias` returns `null`, and `getAccountIdFromAlias` propagates `null` (no exception because `requireResult=false`): [12](#0-11) 

**Missing null guard — second DB query unconditionally executed:**

Back in `getBalances()`, the resolved promise is awaited at line 88:

```js
const accountParams = await Promise.all(accountParamsPromise);
```

There is **no null check** on `accountParams` after this line. Compare with other early-exit guards present in the same function: [13](#0-12) 

The code unconditionally proceeds to build and execute the main SQL query with `null` as the bound parameter: [14](#0-13) 

In PostgreSQL, `ab.account_id = NULL` evaluates to `NULL` (never `true`), so the query returns zero rows — a fully wasted second DB round-trip.

**No caching of alias lookups:**

The `quickLru` cache in `rest/entityId.js` (lines 301–303) caches parsed `EntityId` objects for shard.realm.num strings only. The alias-to-id DB lookup path in `EntityService.getAccountIdFromAlias` is not cached. [9](#0-8) 

**No rate limiting:**

`server.js` registers no rate-limiting middleware for any route, including `/balances`: [6](#0-5) 

**DB pool defaults:**

Default `maxConnections = 10`, `connectionTimeout = 20000 ms`: [7](#0-6) [15](#0-14) 

## Impact Explanation

Each request with a non-existent alias makes two sequential DB queries instead of one. Each query holds one pool connection for its duration, then releases it. Because the total connection-hold time per request is doubled, under concurrent load the pool is occupied for twice as long per request. With a pool capped at 10 connections and a `connectionTimeoutMillis` of 20,000 ms, an attacker sending ~10 concurrent requests keeps the pool saturated. Legitimate requests queue and eventually time out with a connection-pool error, effectively taking the `/api/v1/balances` endpoint (and any other endpoint sharing the same pool) offline for real users. No on-chain state or user funds are affected; this is a mirror-node availability (griefing) impact.

## Likelihood Explanation

The endpoint is public and unauthenticated. The alias format is documented in the OpenAPI spec. A valid-format alias string (e.g., a base32-encoded public key) is trivially generated. No special privileges, tokens, or accounts are required. The attack is trivially scriptable with any HTTP client (`curl`, `ab`, `wrk`) and is repeatable indefinitely at zero cost to the attacker.

## Recommendation

1. **Add a null guard / early exit in `getBalances()`**: After `await Promise.all(accountParamsPromise)`, check whether any resolved value is `null`. If so, return immediately (empty result), mirroring the existing early-exit pattern at lines 115–122.
2. **Cache alias lookups**: Introduce a short-lived in-process or Redis-backed cache for alias-to-entity-id resolutions in `EntityService.getAccountIdFromAlias`, keyed on the alias bytes.
3. **Add rate limiting**: Apply a per-IP or global rate-limiting middleware (e.g., `express-rate-limit`) to the REST API, particularly for endpoints that trigger DB lookups during parameter parsing.
4. **Increase pool size or add a connection-wait queue limit**: Tune `maxConnections` and `connectionTimeoutMillis` to match expected concurrency, or reject requests early when the pool is saturated rather than queuing them for 20 seconds.

## Proof of Concept

```bash
# Generate a syntactically valid but non-existent alias (base32-encoded 20 bytes)
ALIAS="AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA"  # valid format, no matching entity

# Flood with 10 concurrent requests
for i in $(seq 1 10); do
  curl -s "http://<mirror-node>/api/v1/balances?account.id=eq:${ALIAS}" &
done
wait

# Subsequent legitimate requests will time out or receive 503/pool-exhaustion errors
curl -v "http://<mirror-node>/api/v1/balances"
```

Each of the 10 concurrent requests triggers:
1. A live DB query against the `entity` table for the alias (returns 0 rows, `null` propagated).
2. A second live DB query for the main balance result with `null` as the account parameter (returns 0 rows, wasted).

With the default pool of 10 connections and no rate limiting, the pool is saturated for the duration of both queries across all concurrent requests, blocking legitimate traffic.
