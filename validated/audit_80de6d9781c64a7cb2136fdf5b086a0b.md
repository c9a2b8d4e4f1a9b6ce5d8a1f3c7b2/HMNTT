# Audit Report

## Title
Unauthenticated Content-Length Header Injection Permanently Inflates `api_request_size_bytes` Histogram Sum

## Summary
The `metricsMiddleware()` function reads the raw `content-length` request header and passes it directly to `requestSizeHistogram.record()` with no upper-bound validation. A single unauthenticated GET request carrying `Content-Length: 9007199254740991` permanently corrupts the cumulative `api_request_size_bytes_sum` Prometheus counter until process restart.

## Finding Description
In `rest/middleware/metricsHandler.js` at line 209, inside the `res.on('finish')` callback:

```javascript
requestSizeHistogram.record(parseInt(req.headers['content-length'] ?? '0', 10) || 0, labels);
``` [1](#0-0) 

The only sanitization is `parseInt(...) || 0`, which guards against `NaN` and empty string but imposes no upper bound. `parseInt('9007199254740991', 10)` returns `9007199254740991` (truthy), so the `|| 0` fallback never fires and the attacker-controlled value is passed directly to `record()`.

The body-parser limit does not protect this path. `express.json()` (line 73 of `server.js`) only enforces its size limit when the request has `Content-Type: application/json` and an actual body to parse. For a GET request with no body, `express.json()` skips parsing entirely. The request flows through to `metricsMiddleware` (registered at lines 89–92), the `res.on('finish')` callback is registered and fires normally, and the attacker-supplied header is read verbatim. [5](#0-4) 

## Impact Explanation
Prometheus histogram `_sum` counters are strictly monotonically increasing and never reset during process lifetime. A single injected value of `9007199254740991` bytes (~9 PB) dwarfs any realistic traffic volume, making the computed average (`_sum / _count`) meaningless for the affected `{method, path, code}` label set. Capacity-planning dashboards and alerting rules that rely on `api_request_size_bytes` (rate-of-change, percentile approximations derived from sum/count) are rendered permanently unreliable until the process is restarted. The attacker can repeat the injection immediately after each restart with zero cost.

## Likelihood Explanation
The attack requires no authentication, no special privileges, and no knowledge beyond a public API path. A single HTTP GET request is sufficient. The exploit is trivially scriptable and repeatable against any internet-accessible deployment of the mirror-node REST API.

## Recommendation
Cap the recorded value to a realistic maximum before passing it to `record()`. For example:

```javascript
const MAX_REQUEST_SIZE = 100 * 1024 * 1024; // 100 MB upper bound
const contentLength = Math.min(
  parseInt(req.headers['content-length'] ?? '0', 10) || 0,
  MAX_REQUEST_SIZE
);
requestSizeHistogram.record(contentLength, labels);
``` [1](#0-0) 

Alternatively, validate that the parsed value does not exceed the configured body-size limit (e.g., the `express.json()` limit) before recording.

## Proof of Concept
```
GET /api/v1/accounts HTTP/1.1
Host: <mirror-node-host>
Content-Length: 9007199254740991
```

1. `express.json()` skips the request (no body / no matching content-type).
2. `metricsMiddleware` runs, registers `res.on('finish')`.
3. Response completes; callback fires at line 209.
4. `parseInt('9007199254740991', 10)` → `9007199254740991`; `|| 0` does not trigger.
5. `requestSizeHistogram.record(9007199254740991, labels)` executes.
6. `api_request_size_bytes_sum` for the `{GET, /api/v1/accounts, 200}` label set is permanently inflated by ~9 PB.
