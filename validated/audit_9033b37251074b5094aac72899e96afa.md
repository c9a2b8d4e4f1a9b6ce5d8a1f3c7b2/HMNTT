Audit Report

## Title
Stale Token Airdrop Cache Causes Smart Contract Simulations to Read Non-Existent PENDING State

## Summary
`CACHE_NAME_TOKEN_AIRDROP` within `CACHE_MANAGER_TOKEN` uses a 1-second write-expiry TTL. `TokenAirdropRepository.findById()` is annotated `@Cacheable(unless = "#result == null")`, but the method returns `Optional<TokenAirdrop>` — never a literal `null` — so every result is cached unconditionally. After the DB reflects a `CLAIMED` or `CANCELLED` state transition, the cache continues to serve the stale `PENDING` result for up to 1 second, causing `AirdropsReadableKVState.readFromDataSource()` to return a non-existent pending airdrop to the EVM simulation layer.

## Finding Description
`EvmConfiguration.cacheManagerToken()` registers `CACHE_NAME_TOKEN_AIRDROP` under `CACHE_MANAGER_TOKEN` and applies `cacheProperties.getToken()`, which resolves to `ENTITY_CACHE_CONFIG = "expireAfterWrite=1s,maximumSize=10000,recordStats"`. [1](#0-0) [2](#0-1) 

`TokenAirdropRepository.findById()` is annotated:
```java
@Cacheable(cacheNames = CACHE_NAME_TOKEN_AIRDROP, cacheManager = CACHE_MANAGER_TOKEN, unless = "#result == null")
```
The query filters `state = 'PENDING'`. The return type is `Optional<TokenAirdrop>`. Because `Optional.of(...)` and `Optional.empty()` are both non-null objects, `#result == null` is always `false`, and every result is written to the cache. When the airdrop transitions to `CLAIMED`/`CANCELLED`, the DB row no longer matches the `state = 'PENDING'` filter, but the cache still holds the previously stored `Optional.of(pendingAirdrop)`. [4](#0-3) 

`AirdropsReadableKVState.readFromDataSource()` dispatches to `findById()` for all latest-block calls (no timestamp context), directly consuming the cached value: [5](#0-4) 

No `@CacheEvict` or `@CachePut` is applied anywhere in the airdrop claim/cancel transaction handlers. The 1-second TTL is the sole eviction mechanism.

## Impact Explanation
Smart contract simulations (`eth_call`, `eth_estimateGas`) served by the web3 module return incorrect airdrop state for up to 1 second after a claim or cancel is processed by the importer. A contract that branches on whether an airdrop is PENDING will receive a simulated `SUCCESS` outcome for an operation that would revert on the actual Hedera network. This produces incorrect gas estimates, misleading `eth_call` return values, and dApp pre-flight check errors. No funds are at direct risk because the mirror node is a read-only simulation layer; actual on-chain state is maintained independently by the Hedera consensus network.

## Likelihood Explanation
The stale window is bounded by the 1-second TTL. An unprivileged user needs only to issue an `eth_call` for a known airdrop key within the window between the DB update and cache expiry. Airdrop parameters (sender, receiver, token, serial) are publicly observable on-chain. The scenario occurs naturally under normal network activity (concurrent claim and query) without any deliberate attack.

## Recommendation
1. **Fix the `unless` condition**: Change `unless = "#result == null"` to `unless = "#result != null && !#result.isPresent()"` (or equivalently `unless = "#result == null || !#result.isPresent()"` inverted) so that empty Optionals are also cached and stale PENDING entries are not re-served after expiry. More precisely, the intent should be to cache only non-empty results: `unless = "#result == null || !#result.isPresent()"`.
2. **Add cache eviction on state transition**: Apply `@CacheEvict(cacheNames = CACHE_NAME_TOKEN_AIRDROP, cacheManager = CACHE_MANAGER_TOKEN)` in the importer's airdrop claim/cancel transaction handlers so the cache entry is invalidated immediately when the DB state changes.
3. **Consider a shorter TTL or no caching** for airdrop state, given that airdrop state transitions (PENDING → CLAIMED/CANCELLED) are one-way and time-sensitive for simulation correctness.

## Proof of Concept
1. Submit a token airdrop on-chain; the importer writes `state=PENDING` to `token_airdrop`.
2. Issue `eth_call` targeting a contract that reads the airdrop via the `claimAirdrops`/`cancelAirdrops` precompile. `AirdropsReadableKVState.readFromDataSource()` calls `tokenAirdropRepository.findById(senderId, receiverId, tokenId, 0)`, which hits the DB, returns `Optional.of(pendingAirdrop)`, and stores it in the Caffeine cache with a 1-second expiry.
3. The receiver claims the airdrop on-chain. The importer processes the record file and updates the DB row to `state=CLAIMED` (or deletes/moves it to history).
4. Within the remaining TTL window (up to ~1 second), issue another `eth_call` for the same airdrop key. `findById()` returns the cached `Optional.of(pendingAirdrop)` without hitting the DB.
5. `readFromDataSource()` maps the stale result to `AccountPendingAirdrop` and returns it to the EVM simulation layer, which treats the airdrop as still PENDING and simulates `SUCCESS` for a `cancelAirdrops`/`claimAirdrops` call that would revert on-chain.
