Audit Report

## Title
Unbounded `endTime` Allows Unprivileged Users to Hold gRPC Subscriptions Open Indefinitely (Resource Exhaustion DoS)

## Summary
`TopicMessageFilter.isValidEndTime()` imposes no upper bound on the user-supplied `endTime` field. Because `TopicContext.isComplete()` only returns `true` once wall-clock time has passed `endTime + endTimeInterval`, setting `endTime` to `Long.MAX_VALUE` keeps `isComplete()` returning `false` for ~292 years. The `RepeatSpec` inside `pastEndTime()` therefore never terminates, holding every such subscription open permanently and exhausting server-side resources.

## Finding Description

**`TopicMessageFilter.isValidEndTime()` — no upper-bound check:**

The only constraint on `endTime` is that it must be greater than `startTime`. A caller may supply any arbitrarily large value and pass validation. [1](#0-0) 

**`ConsensusController.convertTimestamp()` — explicitly maps overflow timestamps to `Long.MAX_VALUE`:**

When a gRPC client sends a `consensusEndTime` with `seconds >= 9223372035`, the controller explicitly sets `endTime = Long.MAX_VALUE` and passes it through to the filter without rejection. [2](#0-1) 

**`TopicContext.isComplete()` — never returns `true` for `Long.MAX_VALUE`:**

With `endTime = Long.MAX_VALUE`, `Instant.ofEpochSecond(0, Long.MAX_VALUE).plus(30s).isBefore(Instant.now())` evaluates to `false` until approximately year 2262. The method never returns `true` during any realistic server lifetime. [3](#0-2) 

**`pastEndTime()` — `RepeatSpec` predicate is permanently `true`:**

`RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)` fires every 30 s for up to `Long.MAX_VALUE` iterations. Because the predicate `!isComplete()` is permanently `true`, this `Flux` never emits a terminal signal. [4](#0-3) 

**`subscribeTopic()` — `takeUntilOther(pastEndTime(...))` never fires:**

The merged live+safetyCheck flux is terminated only by `pastEndTime(topicContext)`. Since that flux never completes, the entire gRPC stream stays open indefinitely. [5](#0-4) 

**No compensating timeout on the outer subscription flux:**

`retrieverProperties.getTimeout()` (default 60 s) is applied only inside `PollingTopicMessageRetriever.retrieve()`, not to the outer `subscribeTopic()` flux. [6](#0-5) 

`NettyProperties` only configures `maxConcurrentCallsPerConnection = 5`; it sets no per-stream deadline. [7](#0-6) 

## Impact Explanation
Each malicious subscription permanently occupies a Reactor scheduler slot, a `subscriberCount` gauge increment (never decremented until stream ends), and periodic DB polling every 30 s via the safety-check path. With `maxConcurrentCallsPerConnection = 5`, an attacker opens N TCP connections × 5 streams each, exhausting the bounded-elastic thread pool and database connection pool. This is a server-wide Denial-of-Service against all legitimate subscribers. The mirror node is a read-only service so no funds are directly at risk, but full service unavailability is achievable. [8](#0-7) 

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication requirement. The exploit requires only a standard gRPC client (e.g., `grpcurl`) and a single timestamp field set to a far-future value. The `convertTimestamp` method in `ConsensusController` explicitly maps any timestamp with `seconds >= 9223372035` to `Long.MAX_VALUE`, meaning the attacker does not even need to know the exact `Long.MAX_VALUE` nanosecond value — any timestamp in the far future suffices. It is trivially scriptable, repeatable, and requires zero privileges. [9](#0-8) 

## Recommendation
1. **Add an upper-bound validator in `TopicMessageFilter`**: Reject `endTime` values that exceed `now + some_maximum_window` (e.g., 30 days or a configurable cap).
2. **Add a server-side stream deadline**: Apply a `timeout()` operator on the outer `subscribeTopic()` flux, or configure a gRPC per-call deadline via `NettyServerBuilder`.
3. **Cap `convertTimestamp` output**: Instead of mapping overflow timestamps to `Long.MAX_VALUE`, map them to a bounded maximum (e.g., `now + maxAllowedWindow`).

## Proof of Concept
```bash
# Using grpcurl — sends consensusEndTime far in the future (seconds >= 9223372035 maps to Long.MAX_VALUE)
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "consensusEndTime": {"seconds": 9999999999}
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
# Repeat across N connections × 5 streams each to exhaust server resources
```

Each such stream holds open indefinitely because `isComplete()` never returns `true` and `pastEndTime()` never emits a terminal signal, keeping the merged flux — and the gRPC stream — alive permanently. [10](#0-9)
