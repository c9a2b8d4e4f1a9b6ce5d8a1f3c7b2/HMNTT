Audit Report

## Title
`Error`-Subclass Exceptions Bypass `onError()` in `BlockNode.streamBlocks()`, Keeping Malicious Node Permanently Active

## Summary
`BlockNode.streamBlocks()` catches only `BlockStreamException` and `Exception`, leaving `java.lang.Error` subclasses (e.g., `OutOfMemoryError`) uncaught by the error-accounting path. The inner `BlockAssembler` accumulates `List<List<BlockItem>>` in `pending` across multiple gRPC responses with no byte-budget enforcement — only an item-count limit (`maxBlockItems`, default 800,000). A malicious block node can stream many large `BlockItemSet` responses (each up to the 36 MB per-message limit) for a single block, exhausting heap before the item-count guard fires. The resulting `OutOfMemoryError` propagates without calling `onError()`, so the node is never marked inactive and is re-selected on every subsequent polling cycle.

## Finding Description

**Catch structure in `streamBlocks()` (lines 159–164):**

```java
} catch (BlockStreamException ex) {
    onError();
    throw ex;
} catch (Exception ex) {
    onError();
    throw new BlockStreamException(ex);
} finally { ... }
```

`java.lang.Error` is a direct subclass of `java.lang.Throwable`, not of `java.lang.Exception`. Any `Error` thrown inside the `try` block (lines 121–158) skips both catch clauses, skips `onError()`, and propagates through the `finally` block, which only cancels the gRPC call. [1](#0-0) 

**`doGet()` has no `Throwable` catch (lines 58–70):**

`BlockNodeSubscriber.doGet()` calls `node.streamBlocks(...)` with no surrounding `Throwable` or `Error` catch, so any `Error` propagates freely up the call stack. [2](#0-1) 

**Unbounded byte accumulation in `BlockAssembler.append()` (lines 283–297):**

`pending.add(blockItems)` accumulates each gRPC response's items without any byte-size check. Only `pendingCount` (item count) is checked against `maxBlockItems` (default 800,000). The gRPC channel enforces `maxInboundMessageSize` (default 36 MB) per individual `SubscribeStreamResponse`, but cross-message accumulation in `pending` is unbounded in bytes. [3](#0-2) 

**Default limits confirmed in `StreamProperties`:**
- `maxBlockItems = 800_000`
- `maxStreamResponseSize = DataSize.ofMegabytes(36)` [4](#0-3) 

**`onError()` is the sole mechanism for deactivating a node (lines 196–207):**

Without `onError()` being called, `errors` is never incremented, `active` remains `true`, and no `readmitTime` cooldown is scheduled. [5](#0-4) 

**`errors` is reset to 0 on every successful response (line 157):**

A malicious node can serve preceding blocks normally to keep `errors` at 0, then launch the attack on the targeted block. [6](#0-5) 

| Check | What it limits | What it misses |
|---|---|---|
| `maxInboundMessageSize` (36 MB) | Single gRPC message | Cross-message accumulation in `pending` |
| `maxBlockItems` (800,000) | Item count | Item byte size |
| `catch (Exception ex)` | Checked + unchecked exceptions | `java.lang.Error` hierarchy |

## Impact Explanation

A block node operator (no mirror-node credentials required) can target any specific block number:

1. Serve all preceding blocks normally, keeping `errors` reset to 0 on each successful response.
2. For the targeted block, stream enough large `BlockItemSet` responses (each ≤36 MB, each containing a single large `BlockItem`) to exhaust the JVM heap.
3. `OutOfMemoryError` fires inside `streamBlocks()`; both catch clauses are bypassed; `onError()` is never called; `active` remains `true` and `errors` stays at 0.
4. On the next polling tick, `getNode()` selects the same node (still active, errors = 0); the attack repeats indefinitely.

The targeted block — and every transaction it contains — is never persisted to the mirror node database. Downstream API consumers, explorers, and compliance tools see those transactions as non-existent. This constitutes selective transaction suppression with no automatic recovery. [7](#0-6) 

## Likelihood Explanation

**Preconditions:**
- Attacker controls or compromises one block node that the mirror node is configured to use (no mirror-node credentials needed).
- The mirror node's JVM heap is finite (always true).

**Feasibility:** The attack requires only a modified gRPC server that streams oversized `BlockItemSet` responses for a chosen block number. With a 4 GB JVM heap and 36 MB per response, `OutOfMemoryError` fires after roughly 111 responses — far below the 800,000-item threshold that would throw a `BlockStreamException` and correctly call `onError()`.

**Repeatability:** Because `onError()` is never called, the node is never cooled down. The attack repeats on every polling cycle with no operator intervention required.

## Recommendation

1. **Catch `Throwable` (or at minimum `Error`) in `streamBlocks()`** to ensure `onError()` is always called:
   ```java
   } catch (BlockStreamException ex) {
       onError(); throw ex;
   } catch (Exception ex) {
       onError(); throw new BlockStreamException(ex);
   } catch (Error err) {
       onError(); throw err;
   }
   ```
   Alternatively, use a single `catch (Throwable t)` block with re-throw logic.

2. **Enforce a byte budget in `BlockAssembler.append()`** by tracking the serialized size of accumulated `BlockItem`s and throwing a `BlockStreamException` when a configurable `maxBlockBytes` threshold is exceeded — before heap exhaustion can occur.

3. **Add a `maxBlockBytes` property to `StreamProperties`** (e.g., defaulting to a value well below typical JVM heap size) to make the byte budget configurable and observable.

## Proof of Concept

```
Attacker's malicious block node gRPC server:
  - For blocks 0..N-1: respond normally (keeps errors=0 on mirror node)
  - For block N (target):
      Loop:
        Send SubscribeStreamResponse { block_items: BlockItemSet { items: [<35MB BlockItem>] } }
      // Each response passes maxInboundMessageSize check (35MB < 36MB)
      // pendingCount increments by 1 per response (far below 800,000)
      // pending list grows by ~35MB per iteration
      // After ~115 iterations on a 4GB heap: OutOfMemoryError thrown
      //   -> bypasses catch(BlockStreamException) and catch(Exception)
      //   -> onError() never called
      //   -> node.active remains true, errors remains 0
      //   -> next poll selects same node, attack repeats
```

Block N is never assembled, never passed to `blockStreamConsumer`, never persisted. [8](#0-7)

    private BlockNode getNode(final AtomicLong nextBlockNumber) {
        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
                inactiveNodes.add(node);
                continue;
            }

            if (hasBlock(nextBlockNumber, node)) {
                return node;
            }
        }

        // find the first inactive node with the block and force activating it
        for (final var node : inactiveNodes) {
            if (hasBlock(nextBlockNumber, node)) {
                node.tryReadmit(true);
                return node;
            }
        }

        throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
    }
