Audit Report

## Title
Unbounded Concurrent `subscribeTopic` Streams Exhaust Shared `boundedElastic()` Scheduler in `PollingTopicListener`, Causing Severe Polling Delays for All Subscribers

## Summary
When `hiero.mirror.grpc.listener.type=POLL`, each `subscribeTopic` call creates a new per-subscriber infinite polling loop scheduled on the application-wide `Schedulers.boundedElastic()` singleton. Because there is no global subscriber cap and `maxConcurrentCallsPerConnection` only limits streams per TCP connection, an attacker can open arbitrarily many connections and saturate the bounded thread pool, delaying all polling cycles well beyond the 500ms interval.

## Finding Description

**`PollingTopicListener.java` ã per-subscriber scheduler allocation:**

Line 31 assigns the application-wide singleton: [1](#0-0) 

Every `listen()` call schedules an infinite repeat loop on that shared scheduler: [2](#0-1) 

`poll()` at line 61 calls `topicMessageRepository.findByFilter(newFilter)` via `Flux.fromStream()`, which iterates a JDBC-backed `Stream` synchronously, blocking the `boundedElastic` worker thread for the full duration of the DB query: [3](#0-2) 

**`TopicMessageServiceImpl.java` ã no subscriber cap:**

`subscriberCount` is registered only as a metrics `Gauge` and is never checked against a maximum: [4](#0-3) [5](#0-4) 

**`NettyProperties.java` / `GrpcConfiguration.java` ã per-connection limit only:**

`maxConcurrentCallsPerConnection` defaults to 5 and is applied per TCP connection, not globally: [6](#0-5) [7](#0-6) 

**Shared scheduler across the application:**

`SharedPollingTopicListener` and `TopicMessageServiceImpl`'s safety-check flux also call `Schedulers.boundedElastic()`, sharing the same singleton thread pool: [8](#0-7) [9](#0-8) 

**Root cause:** `BoundedElasticScheduler` defaults to `10 û CPU_count` worker threads. With N concurrent `POLL` subscribers, N JDBC queries run simultaneously every 500ms. Once N exceeds the thread cap, tasks queue (up to 100,000 deep) and are delayed proportionally. The failed assumption is that the per-connection stream limit (`maxConcurrentCallsPerConnection=5`) constitutes a global subscriber cap ã it does not, because there is no limit on the number of TCP connections.

**`ListenerType.POLL` is a valid, documented configuration:** [10](#0-9) 

## Impact Explanation
With the thread pool saturated, every subscriber's polling cycle is delayed proportionally to queue depth. On a 4-core host (40 `boundedElastic` threads), 500 concurrent attacker streams cause legitimate subscribers to experience polling delays of `(500/40) û query_time` per cycle ã easily 10ã25û the 500ms baseline. Because the same `Schedulers.boundedElastic()` singleton is shared by `SharedPollingTopicListener`, `TopicMessageServiceImpl`'s safety-check flux, and other components, starvation cascades across the entire gRPC application. Topic message delivery to all subscribers is effectively frozen for the duration of the attack.

## Likelihood Explanation
`subscribeTopic` requires no authentication and is publicly reachable. An attacker needs only a standard gRPC client (e.g., `grpcurl`) and the ability to open many TCP connections ã trivial from any host with outbound connectivity. Streams stay open indefinitely (no server-side timeout on live subscriptions), so the attacker does not need to continuously reconnect. The prerequisite is `listener.type=POLL`; while non-default (default is `REDIS`), it is a documented and supported configuration option.

## Recommendation
1. **Enforce a global subscriber cap**: Check `subscriberCount` against a configurable maximum in `TopicMessageServiceImpl.subscribeTopic()` and reject excess subscriptions with `RESOURCE_EXHAUSTED`.
2. **Isolate the `POLL` scheduler**: Use a dedicated, bounded `Schedulers.newBoundedElastic(...)` instance in `PollingTopicListener` rather than the application-wide singleton, preventing cross-component starvation.
3. **Limit total TCP connections**: Configure Netty's `maxConnectionAge` and a global connection limit at the server level.
4. **Require authentication**: Gate `subscribeTopic` behind authentication/authorization to raise the cost of the attack.

## Proof of Concept
```bash
# Open 200 TCP connections û 5 streams each = 1000 concurrent POLL subscribers
for i in $(seq 1 200); do
  grpcurl -plaintext -d '{"topicId": {"topicNum": 1}}' \
    -H 'content-type: application/grpc' \
    <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# Legitimate subscriber now experiences polling delays >> 500ms
grpcurl -plaintext -d '{"topicId": {"topicNum": 1}}' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```
With 1000 concurrent attacker streams on a 4-core host (40 `boundedElastic` threads), each legitimate poll cycle is queued behind ~25 attacker polls, causing delays of `25 û query_time` ã well beyond the 500ms interval.
