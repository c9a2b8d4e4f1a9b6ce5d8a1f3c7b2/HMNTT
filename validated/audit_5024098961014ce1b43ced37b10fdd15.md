## Audit Report

## Title
Cache TTL Asymmetry in `findStorageBatch()` Enables Low-Rate DB Amplification Attack

## Summary
A structural asymmetry between the `contractStateCache` TTL (`expireAfterWrite=2s`) and the `slotsPerContract` cache TTL (`expireAfterAccess=5m`, max 1500 entries) allows an unprivileged attacker to trigger a full 1500-slot batch database query with a single HTTP request every 2 seconds. Because the slot-key accumulator outlives the slot-value cache by 150×, each cache miss causes `findStorageBatch()` to re-query every previously seen slot for a contract, creating up to a 1500× DB amplification ratio at 0.5 RPS — well below all rate-limiting thresholds.

## Finding Description

**Confirmed cache TTL values** in `CacheProperties.java`:
- `contractState = "expireAfterWrite=2s,maximumSize=25000,recordStats"` [1](#0-0) 
- `slotsPerContract = "expireAfterAccess=5m,maximumSize=1500"` [2](#0-1) 

**Code path in `ContractStateServiceImpl.findStorage()`** (lines 58–70): checks `contractStateCache`; on any miss (including expiry after 2 s), unconditionally calls `findStorageBatch()`. [3](#0-2) 

**`findStorageBatch()`** (lines 85–103): retrieves the per-contract `CaffeineCache` from `slotsPerContract`, then at line 91 collects **all** slot keys currently in that cache regardless of whether their values are warm in `contractStateCache`: [4](#0-3) 

At line 103, a single `findStorageBatch` SQL query is issued for all collected slots: [5](#0-4) 

**SQL query** in `ContractStateRepository.findStorageBatch()` (lines 23–28):
```sql
select slot, value from contract_state
where contract_id = :contractId
and slot in (:slots)   -- up to 1500 entries
``` [6](#0-5) 

**Root cause:** The design assumes `contractStateCache` misses are rare (i.e., the cache is warm). It provides no guard against the case where the slot-key accumulator (`slotsPerContract`) is full but the slot-value cache (`contractStateCache`) is cold. The `slotsPerContract` cache is kept alive by access (`expireAfterAccess`), so it grows to its maximum (1500) and stays there. The `contractStateCache` expires every 2 seconds regardless of access. A single request arriving just after the 2-second expiry re-triggers a batch query for all 1500 accumulated keys.

## Impact Explanation

Each attacker request at 0.5 RPS causes PostgreSQL to execute a query scanning up to 1500 rows via an `IN` clause on the `contract_state` table — the hot path for all EVM `SLOAD` operations. The `db.statementTimeout` is 3000 ms, which is long enough for a 1500-row `IN` query to complete and return, so no timeout protection applies. [7](#0-6) 

Multiple attackers targeting different contracts multiply this linearly. Sustained large `IN` queries increase I/O, connection pool pressure, and query planner overhead on the `contract_state` table.

## Likelihood Explanation

No authentication is required. The attacker only needs a publicly accessible `eth_call` endpoint and knowledge of any contract with many storage slots (e.g., any ERC-20 token contract). Slot keys for standard contracts are deterministic and enumerable. The attack requires no special tooling beyond a script sending one HTTP request every 2 seconds — well below the 500 RPS rate limit. [8](#0-7) 

## Recommendation

1. **Align TTLs**: Set `slotsPerContract` to use `expireAfterWrite` with a TTL equal to or shorter than `contractState`, so the slot-key accumulator cannot outlive the slot-value cache.
2. **Guard batch size on cache miss**: Before issuing `findStorageBatch`, check how many of the accumulated slot keys are actually missing from `contractStateCache` and only query those, rather than re-querying all accumulated keys unconditionally.
3. **Cap batch query size**: Enforce a hard upper bound on the number of slots passed to `findStorageBatch` per invocation, independent of the accumulator size.
4. **Consider disabling batch caching under load**: The `enableBatchContractSlotCaching` flag [9](#0-8)  provides a kill switch; document it as a mitigation for this scenario.

## Proof of Concept

```
# Step 1: Populate slotsPerContract accumulator with 1500 distinct slot keys
for i in $(seq 1 1500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"$(slot_sload_calldata $i)\",\"to\":\"<erc20_contract>\"}" &
done
wait

# Step 2: Every 2 seconds, send ONE request to trigger a 1500-slot IN query
while true; do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"<any_slot_sload>","to":"<erc20_contract>"}'
  sleep 2
done
```

After step 1, `slotsPerContract` holds 1500 keys with a 5-minute access-based TTL. Each iteration of step 2 arrives after `contractStateCache` has expired (2 s), causing `findStorageBatch()` to collect all 1500 keys at line 91 [4](#0-3)  and issue a single 1500-row `IN` query at line 103. [5](#0-4)  The access in step 2 resets the `slotsPerContract` TTL, keeping the accumulator full indefinitely.
