# Audit Report

## Title
Integer Narrowing Overflow in `PollingTopicMessageRetriever.poll()` Bypasses Query Limit, Enabling Unbounded Database Queries via Unprivileged gRPC Request

## Summary
When a gRPC client supplies a `limit` value greater than `Integer.MAX_VALUE` in a `subscribeTopic` request, an unchecked narrowing cast from `long` to `int` in `PollingTopicMessageRetriever.poll()` produces a large negative integer. This negative value is stored as the `limit` of a new `TopicMessageFilter`, causing `hasLimit()` to return `false` and `findByFilter()` to execute with no `LIMIT` clause ā€” returning every matching row in the `topic_message` table on every poll cycle.

## Finding Description

**Step 1 ā€” No upper-bound validation on ingress.**

`ConsensusController.toFilter()` passes the protobuf `uint64` field directly to `TopicMessageFilter.builder().limit(...)`:

```java
// ConsensusController.java:56
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

`TopicMessageFilter.limit` carries only `@Min(0)`, no `@Max`:

```java
// TopicMessageFilter.java:25-26
@Min(0)
private long limit;
``` [2](#0-1) 

Although `TopicMessageServiceImpl` is annotated `@Validated`, the `subscribeTopic` method signature carries no `@Valid` on the `filter` parameter, so Bean Validation is never triggered:

```java
// TopicMessageServiceImpl.java:59
public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
``` [3](#0-2) 

**Step 2 ā€” Narrowing cast overflows.**

In `poll()`, the remaining limit is computed with an unchecked `(int)` cast:

```java
// PollingTopicMessageRetriever.java:68-71
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
``` [4](#0-3) 

With `filter.getLimit() = 2_147_483_648L` and `context.getTotal().get() = 0`:
- `(int)(2_147_483_648L)` ā†’ `-2_147_483_648` (two's-complement wrap)
- `Math.min(-2_147_483_648, maxPageSize)` ā†’ `-2_147_483_648`

**Step 3 ā€” Corrupted limit propagates to the DB query.**

The negative `pageSize` is widened back to `long` and stored in a new filter:

```java
// PollingTopicMessageRetriever.java:75
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [5](#0-4) 

**Step 4 ā€” `hasLimit()` returns `false`, `setMaxResults()` is never called.**

```java
// TopicMessageFilter.java:39-41
public boolean hasLimit() {
    return limit > 0;
}
``` [7](#0-6) 

```java
// TopicMessageRepositoryCustomImpl.java:51-53
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [6](#0-5) 

Since `-2_147_483_648 > 0` is `false`, `setMaxResults()` is never called and the JPA query executes with no row limit.

**Step 5 ā€” Polling loop does not terminate.**

`isComplete()` compares against the original filter's limit (2,147,483,648), which `total.get()` will never reach:

```java
// PollingTopicMessageRetriever.java:122
boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
``` [8](#0-7) 

Polling continues until the configured timeout (default 60 seconds), issuing an unbounded query on every cycle.

**Partial mitigation (insufficient):** `TopicMessageServiceImpl` applies `flux.take(filter.getLimit())` using the original long limit, which caps messages delivered to the client. However, this does not prevent the unbounded `SELECT *` queries from being issued to the database on each poll cycle. [9](#0-8) 

## Impact Explanation
Each poll cycle issues an unbounded `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with no `LIMIT` clause. For a topic with a large message history, the database returns all matching rows, consuming database CPU, I/O, JVM heap, and gRPC thread pool resources. Multiple concurrent such subscriptions can exhaust the database connection pool and cause cascading failures across the mirror node's gRPC service, constituting a practical denial-of-service.

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly exposed and requires no authentication. The protobuf `ConsensusTopicQuery.limit` field is `uint64`, so any client can trivially set it to `2_147_483_648` or any value above `Integer.MAX_VALUE`. No special knowledge, credentials, or race conditions are required. The exploit is deterministic and repeatable on every request.

## Recommendation
1. **Add an upper-bound cap in `ConsensusController.toFilter()`** before building the filter:
   ```java
   long rawLimit = query.getLimit();
   long safeLimit = rawLimit == 0 ? 0 : Math.min(rawLimit, Integer.MAX_VALUE);
   final var filter = TopicMessageFilter.builder().limit(safeLimit);
   ```
2. **Replace the unchecked cast in `poll()`** with a safe narrowing:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
3. **Add `@Max(Integer.MAX_VALUE)` to `TopicMessageFilter.limit`** and add `@Valid` to the `subscribeTopic` parameter in `TopicMessageServiceImpl` to enforce validation at the service boundary.
