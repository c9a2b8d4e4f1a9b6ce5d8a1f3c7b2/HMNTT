# Audit Report

## Title
Silent Fallback to `BlockType.LATEST` in Opcode Replay Produces Historically Inaccurate EVM Trace

## Summary
In `OpcodeServiceImpl.buildCallServiceParameters()`, when `recordFileService.findByTimestamp()` returns empty for a given `consensusTimestamp`, the code silently falls back to `BlockType.LATEST`. This causes all downstream EVM state reads (storage, balances, bytecode) to be performed against the **current** database state rather than the historical state at the time of the original transaction, producing an opcode trace that misrepresents the original execution — with no error raised and a `200 OK` response returned to the caller.

## Finding Description

**Step 1 — Silent fallback in `OpcodeServiceImpl`:**

The `blockType` is resolved at lines 158–161:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);   // ← silent fallback
``` [1](#0-0) 

This `blockType` is then passed directly into `ContractDebugParameters.builder().block(blockType)`. [2](#0-1) 

**Step 2 — `useHistorical()` gates all historical reads:**

`ContractCallContext.useHistorical()` returns `false` whenever `block == BlockType.LATEST`:

```java
public boolean useHistorical() {
    return callServiceParameters != null && callServiceParameters.getBlock() != BlockType.LATEST;
}
``` [3](#0-2) 

**Step 3 — `getTimestamp()` returns empty when not historical:**

```java
public Optional<Long> getTimestamp() {
    if (useHistorical()) {
        return getTimestampOrDefaultFromRecordFile();
    }
    return Optional.empty();   // ← no historical timestamp pinned
}
``` [4](#0-3) 

**Step 4 — Storage reads fall through to latest state:**

`ContractStorageReadableKVState.readFromDataSource()` branches on the timestamp:

```java
final var timestamp = ContractCallContext.get().getTimestamp();
...
return timestamp
        .map(t -> contractStateService.findStorageByBlockTimestamp(..., t))
        .orElse(contractStateService.findStorage(entityId, keyBytes))  // ← reads latest
``` [5](#0-4) 

When `timestamp` is `Optional.empty()`, `findStorage()` is called with no time bound, reading the most recent row in the database.

**Step 5 — The query that produces the empty optional:**

`RecordFileRepository.findByTimestamp()` uses:

```sql
select r from RecordFile r where r.consensusEnd >= ?1 order by r.consensusEnd asc limit 1
``` [6](#0-5) 

This returns empty whenever `consensusTimestamp` exceeds the highest `consensusEnd` in the table — which occurs during ingestion lag or after record-file pruning.

**Root cause:** The code assumes a `contract_result` row always has a corresponding `record_file` row reachable via `findByTimestamp`. When this assumption fails, no exception is thrown, no warning is logged, and the fallback to `BlockType.LATEST` silently corrupts the historical context. The `consensusTimestamp` field is still present in `ContractDebugParameters` but is never used to pin state when `useHistorical()` is `false`. [7](#0-6) 

## Impact Explanation
Any caller of `GET /api/v1/contracts/results/{txHash}/opcodes` receives an opcode trace that reflects **current** contract storage, balances, and bytecode rather than the state at the time the transaction was originally executed. For contracts whose state has changed since the original transaction (e.g., storage slots updated, tokens transferred, contract upgraded), the replayed trace will show different `SLOAD` values, different branch outcomes, and potentially a different execution path than what actually occurred. The HTTP response is `200 OK` with no indication of the data being inaccurate, directly enabling distortion of forensic or audit evidence about historical contract execution.

## Likelihood Explanation
The precondition — a `contract_result` row existing while its `record_file` is absent — is reachable by any unprivileged user who:
1. Queries a transaction during a period of ingestion lag (the importer writes `contract_result` before the corresponding `record_file` is committed), or
2. Queries a transaction whose record file has been pruned from the mirror node's database (a common operational practice for older data).

No special credentials, admin access, or on-chain privileges are required. Only a valid transaction hash is needed, which is publicly observable on-chain.

## Recommendation
Replace the silent `.orElse(BlockType.LATEST)` fallback with an explicit exception:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElseThrow(() -> new EntityNotFoundException(
                "Record file not found for consensus timestamp: " + consensusTimestamp));
```

This ensures that when the historical context cannot be established, the request fails fast with a clear error rather than silently returning misleading data. Additionally, consider adding a guard in `ContractDebugParameters` processing to verify that `consensusTimestamp` is used to pin state even when `blockType` is resolved, as a defense-in-depth measure.

## Proof of Concept
1. Identify a transaction hash for a contract whose storage has changed since the original transaction (e.g., a mapping updated after the call).
2. Ensure the mirror node's `record_file` table does not contain a row with `consensusEnd >= consensusTimestamp` for that transaction (achievable during ingestion lag or on a node with pruned record files).
3. Call `GET /api/v1/contracts/results/{txHash}/opcodes`.
4. Observe that the response is `200 OK` with an opcode trace.
5. Compare `SLOAD` values in the trace against the known historical storage values — they will reflect the **current** storage state, not the state at the time of the original transaction, demonstrating the silent fallback to latest state.
