Audit Report

## Title
Missing `coalesce()` on Nullable `deleted` Column in `entityFromEvmAddressQuery` Causes Silent Entity Omission

## Summary
In `rest/service/entityService.js`, the static query `entityFromEvmAddressQuery` filters with `deleted <> true` without a `coalesce()` guard. Because the `entity.deleted` column is explicitly nullable (per migration `V1.39.1__upsert_support.sql`), any entity with `deleted IS NULL` is silently excluded by SQL three-valued logic, producing false 404 responses for valid, non-deleted entities looked up by EVM address.

## Finding Description

**Vulnerable query** — `entityFromEvmAddressQuery`, lines 22–25: [1](#0-0) 

```sql
WHERE deleted <> true AND evm_address = $1
```

When `deleted IS NULL`, `NULL <> true` evaluates to `NULL` in SQL (three-valued logic), which is treated as falsy in a `WHERE` clause — the row is excluded.

**Correct sibling query** — `entityFromAliasQuery`, lines 17–20, uses `coalesce()` properly: [2](#0-1) 

**Schema proof** — Migration `V1.39.1__upsert_support.sql` intentionally drops both `NOT NULL` and `DEFAULT false` from `entity.deleted`: [3](#0-2) 

`NULL` is therefore a documented, valid, production-reachable value for `entity.deleted`.

**Consumer** — `getEntityIdFromEvmAddress()` at line 91 executes the broken query; the only guard is `rows.length === 0` at line 92, which is the symptom of the bug, not a mitigation: [4](#0-3) 

## Impact Explanation
Any entity with `evm_address` set and `deleted IS NULL` is permanently invisible to all EVM-address-based REST lookups. Callers receive a 404 / `null` for an entity that genuinely exists and is not deleted. Downstream consumers (wallets, explorers, dApps) will display incorrect "not found" states, fail contract interactions, and produce broken address-to-entity mappings. Severity: **Medium** (data availability / integrity; no fund loss, but persistent incorrect state visible to all users).

## Likelihood Explanation
No privileges are required. Any caller who knows an EVM address of an affected entity can trigger the bug deterministically with a standard REST GET request (e.g., `GET /api/v1/contracts/0x<evmAddress>`). The `deleted IS NULL` condition is an explicitly supported schema state, so affected entities can exist in any production deployment without any special setup.

## Recommendation
Apply the same `coalesce()` pattern already used in `entityFromAliasQuery`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

## Proof of Concept

1. Insert a row into `entity` with `evm_address = '\xdeadbeef...'` and `deleted = NULL` (valid per schema).
2. Issue `GET /api/v1/contracts/0xdeadbeef...` to the mirror node REST API.
3. The request reaches `getEntityIdFromEvmAddress()`, which executes:
   ```sql
   SELECT id FROM entity WHERE deleted <> true AND evm_address = $1
   ```
4. `NULL <> true` → `NULL` → row excluded → zero rows returned.
5. `rows.length === 0` → `requireResult=true` → `NotFoundError` thrown → HTTP 404 returned.
6. The entity exists and is not deleted, but the API reports it as not found.

Compare: the same entity looked up via alias would succeed because `entityFromAliasQuery` uses `coalesce(deleted, false) <> true`. [2](#0-1)
