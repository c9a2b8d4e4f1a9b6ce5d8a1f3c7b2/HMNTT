Audit Report

## Title
Error Counter Reset Bypass Allows Malicious Block Node to Evade Inactivity Marking Indefinitely

## Summary
In `BlockNode.streamBlocks()`, the `errors.set(0)` call at line 157 executes after every successfully processed message within the streaming loop — including messages received in sessions that ultimately fail. A malicious block node can exploit this by sending one valid `BLOCK_ITEMS` message (resetting the counter) before each deliberate failure, keeping the cross-invocation error counter perpetually at 1 and never reaching `maxSubscribeAttempts` (default: 3). The node is never marked inactive and remains in the active pool indefinitely.

## Finding Description

**Exact code path — `streamBlocks()`, lines 138–158:**

```java
while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
    switch (response.getResponseCase()) {
        case BLOCK_ITEMS -> assembler.onBlockItemSet(response.getBlockItems());
        case END_OF_BLOCK -> assembler.onEndOfBlock(response.getEndOfBlock());
        case STATUS -> {
            var status = response.getStatus();
            if (status == SubscribeStreamResponse.Code.SUCCESS) {
                serverSuccess = true;
                break;
            }
            throw new BlockStreamException(...);  // jumps to onError()
        }
        default -> throw new BlockStreamException(...);
    }
    errors.set(0);  // LINE 157 — resets BEFORE the next iteration can fail
}
``` [1](#0-0) 

**`onError()`, lines 196–207:**

```java
private void onError() {
    errorsMetric.increment();
    if (errors.incrementAndGet() >= streamProperties.getMaxSubscribeAttempts()) {
        active = false;
        errors.set(0);
        readmitTime.set(Instant.now().plus(streamProperties.getReadmitDelay()));
    }
}
``` [2](#0-1) 

**Root cause:** The `errors` counter is intended to count *consecutive failed `streamBlocks()` invocations* across calls. However, `errors.set(0)` is placed inside the per-message loop body, not after a fully successful streaming session. Any single successfully parsed response message within a doomed session resets the accumulated cross-invocation error count to zero before `onError()` increments it back to 1.

**`maxSubscribeAttempts` default is 3**, confirmed in `StreamProperties.java`: [3](#0-2) 

**Exploit flow:**
- Invocation 1: Malicious node sends `BLOCK_ITEMS` (valid) → `errors.set(0)` fires (errors = 0) → sends `STATUS=NOT_AVAILABLE` → exception → `onError()` → `errors.incrementAndGet()` = 1. Since 1 < 3, node stays active.
- Invocation 2: Same pattern → `errors.set(0)` resets to 0 → `onError()` → errors = 1. Node stays active.
- Invocation N: Same pattern → errors never reaches 3. Node is never marked inactive.

## Impact Explanation

The concrete impact is **availability disruption**. The malicious block node remains permanently active in the node pool. `getNode()` in `BlockNodeSubscriber` iterates the node list in order and returns the first active node that has the block: [4](#0-3) 

A malicious node that is never deactivated will be selected on every invocation, send one valid message, then fail — stalling or retrying block ingestion indefinitely. Legitimate nodes at lower priority are bypassed. In a single-node configuration this is a complete denial of block stream ingestion.

`BlockStreamVerifier.verify()` calls `verifyBlockNumber()`, `verifyHashChain()`, and `verifySignature()` (TSS or wrapped record file): [5](#0-4) 

These cryptographic checks prevent actual block content manipulation, so the impact is **availability only, not integrity**. Severity: **Medium**.

## Likelihood Explanation

Precondition: the attacker must control a host registered as a block node in the mirror node's configuration (`hiero.mirror.importer.block.nodes`) or discoverable via auto-discovery (`autoDiscoveryEnabled=true`, the default). This is not zero-privilege, but with auto-discovery enabled and no authentication on the registration path, the bar is lower than traditional privileged access. The attack pattern is trivially repeatable: one valid gRPC `BLOCK_ITEMS` message per invocation is sufficient, and the pattern is stable across all `maxSubscribeAttempts` values.

## Recommendation

Move `errors.set(0)` **outside** the while loop so it only executes after a fully successful streaming session (i.e., after the loop exits normally without throwing an exception):

```java
// current (buggy): errors.set(0) inside the loop body at line 157
// fixed: place errors.set(0) after the while loop, before the catch blocks

while (!serverSuccess && (response = grpcCall.read(...)) != null) {
    switch (response.getResponseCase()) { ... }
    // remove errors.set(0) from here
}
errors.set(0);  // only reached if the loop exits without exception
``` [6](#0-5) 

This ensures the counter is only reset when an entire streaming session completes without error, which is the intended semantics.

## Proof of Concept

1. Register a malicious gRPC server as a block node (or rely on auto-discovery).
2. On each `SubscribeBlockStream` call, respond with:
   - One `SubscribeStreamResponse` with `BLOCK_ITEMS` containing a syntactically valid (but otherwise useless) block item set.
   - Immediately follow with a `SubscribeStreamResponse` with `STATUS = READ_STREAM_NOT_AVAILABLE` (or any non-SUCCESS status).
3. Observe that `errors.set(0)` fires after the `BLOCK_ITEMS` message is processed (line 157), then `onError()` increments `errors` from 0 to 1.
4. Repeat steps 2–3 indefinitely. The `errors` counter never reaches `maxSubscribeAttempts` (3), the node is never marked inactive, and block ingestion stalls permanently on the malicious node.

    private BlockNode getNode(final AtomicLong nextBlockNumber) {
        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
                inactiveNodes.add(node);
                continue;
            }

            if (hasBlock(nextBlockNumber, node)) {
                return node;
            }
        }

        // find the first inactive node with the block and force activating it
        for (final var node : inactiveNodes) {
            if (hasBlock(nextBlockNumber, node)) {
                node.tryReadmit(true);
                return node;
            }
        }

        throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
    }
