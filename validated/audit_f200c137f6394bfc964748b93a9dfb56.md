Audit Report

## Title
Gas Token Non-Restoration via Dead Try-Catch in `constructServiceParameters()` Enables Gas Bucket Depletion DoS

## Summary
In `ContractController.call()`, the `catch (InvalidParametersException e)` block is the only path that calls `throttleManager.restore()`. The `constructServiceParameters()` method contains a dead try-catch that wraps a Lombok getter incapable of throwing, while the actual `hexToBytes(data)` call — which throws `IllegalArgumentException` on odd-length hex input — sits outside that catch. Because `IllegalArgumentException` is not `InvalidParametersException`, it propagates through `call()` without triggering `restore()`, permanently consuming gas tokens from `gasLimitBucket` for every such request.

## Finding Description

**Dead try-catch — confirmed:**

In `ContractController.java` lines 67–72, the `try` block wraps only `request.getData()`, a plain Lombok getter that cannot throw: [1](#0-0) 

The developer's intent (evidenced by the error message "contains invalid odd length characters") was to catch the exception from `hexToBytes(data)`, but that call is at line 80, **outside** the try-catch: [2](#0-1) 

**`hexToBytes()` throws `IllegalArgumentException`, not `InvalidParametersException` — confirmed:**

`BytesDecoder.hexToBytes()` catches `DecoderException` from `org.apache.commons.codec.binary.Hex.decodeHex()` and re-wraps it as `IllegalArgumentException`: [3](#0-2) 

**`restore()` is never called for this exception — confirmed:**

The only `restore()` call in `call()` is inside `catch (InvalidParametersException e)`. `IllegalArgumentException` is not a subtype of `InvalidParametersException`, so it propagates uncaught: [4](#0-3) 

**`restore()` only refunds `gasLimitBucket` — confirmed:** [5](#0-4) 

**`throttle()` consumes both `rateLimitBucket` and `gasLimitBucket` before `constructServiceParameters()` is reached — confirmed:** [6](#0-5) 

**`@Hex` on `data` does not enforce even-length — confirmed:**

The `data` field carries `@Hex` with no `minLength` or `maxLength` constraints: [7](#0-6) 

`HexValidator` compiles a pattern `^(0x)?[0-9a-fA-F]{minLength,maxLength}$` that matches any sequence of valid hex characters regardless of parity: [8](#0-7) 

An odd-length string such as `0x123` passes `@Valid`, passes `throttle()`, passes `validateContractMaxGasLimit()`, then causes `hexToBytes()` to throw `IllegalArgumentException` — consuming gas tokens permanently.

## Impact Explanation

Each malicious request with `gas` near `maxGasLimit` consumes `scaleGas(gas)` tokens from `gasLimitBucket` with no restoration. With default settings (`gasPerSecond = 7,500,000,000`, `requestsPerSecond = 500`, `GAS_SCALE_FACTOR = 10,000`): [9](#0-8) 

- Bucket capacity: `scaleGas(7,500,000,000)` = 750,000 tokens
- Attacker rate: 500 req/s × `scaleGas(15,000,000)` = 500 × 1,500 = 750,000 tokens/s

The attacker drains the gas bucket at exactly the refill rate, holding it at zero and causing every legitimate contract call to receive `ThrottleException("Gas per second rate limit exceeded.")`. This is a complete, sustained DoS on `/api/v1/contracts/call`.

## Likelihood Explanation

- **No privileges required**: any HTTP client can POST to `/api/v1/contracts/call`.
- **Trivially repeatable**: a single script sending `{"data":"0x123","gas":15000000}` in a loop at 500 req/s is sufficient.
- **No detection barrier**: requests pass Spring `@Valid` and only fail deep inside `constructServiceParameters()` after tokens are consumed.
- **Sustainable**: the attacker's own `rateLimitBucket` tokens refill at 500/s, matching their send rate, so the attack runs indefinitely.

## Recommendation

Move the `hexToBytes(data)` call inside the existing try-catch block in `constructServiceParameters()`, or catch `IllegalArgumentException` in `call()` and call `restore()` before rethrowing. The simplest fix:

```java
// In constructServiceParameters():
String data;
try {
    data = request.getData() != null ? request.getData() : HEX_PREFIX;
    return ContractExecutionParameters.builder()
            ...
            .callData(hexToBytes(data))   // moved inside try
            ...
            .build();
} catch (final Exception e) {
    throw new InvalidParametersException(
            "data field '%s' contains invalid odd length characters".formatted(request.getData()));
}
```

Alternatively, add an even-length constraint to the `@Hex` validator or add a dedicated `@Pattern` constraint on `data` to reject odd-length hex strings at the validation layer, before `throttle()` is ever called.

## Proof of Concept

```bash
# Drain gasLimitBucket with odd-length hex data (passes @Hex validation, fails hexToBytes)
while true; do
  curl -s -o /dev/null -X POST https://<host>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"data":"0x123","gas":15000000,"to":"0x0000000000000000000000000000000000000001"}' &
done
# After ~1 second the bucket is empty; legitimate callers receive:
# 429 {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

    @Min(1)

    // Necessary since bucket4j has a max capacity and fill rate of 1 token per nanosecond
    public long getGasPerSecond() {
        return scaleGas(gasPerSecond);
    }

    public long scaleGas(long gas) {
        if (gas <= GAS_SCALE_FACTOR) {
            return 0L;
        }
        return Math.floorDiv(gas, GAS_SCALE_FACTOR);
