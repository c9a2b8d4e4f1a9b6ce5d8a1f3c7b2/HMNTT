Audit Report

## Title
Cache-Poisoning Slot Amplification: Unprivileged Inflation of Batch DB Query Size via `findStorageBatch` Slot Registration

## Summary
An unauthenticated attacker can permanently register up to 1,500 arbitrary slot keys into the per-contract `slotsPerContract` Caffeine cache by issuing `eth_call` requests that trigger SLOAD opcodes for distinct storage slots of a target contract. Once seeded, every subsequent `findStorageBatch` invocation for that contract issues a DB batch query carrying all 1,500 registered keys instead of the single key actually needed, constituting a sustained amplification of database load that persists for the full 5-minute `expireAfterAccess` window.

## Finding Description

**Exact code path — verified against source:**

In `ContractStateServiceImpl.findStorage()` (lines 58–70), the `contractState` cache is checked first. On a miss (or when the cached value equals `EMPTY_VALUE`), `findStorageBatch()` is called:

```
// ContractStateServiceImpl.java line 63–69
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);
if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
return findStorageBatch(contractId, key);
```

Inside `findStorageBatch()` (lines 85–122):

```
// line 90: unconditionally registers attacker-supplied key
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);

// line 91: snapshot of ALL keys — including attacker-seeded ones
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

// line 103: full batch query issued with every registered key
contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

**Root cause — failed assumption:** `putIfAbsent` at line 90 unconditionally writes the caller-supplied key into the per-contract cache before the batch query is built. There is no validation of whether the key was ever legitimately returned by the database. The batch query at line 103 then blindly uses the entire key-set snapshot, so an attacker-inflated registry directly inflates every subsequent DB query.

**Cache configuration — verified against `CacheProperties.java` (lines 25–39) and `docs/configuration.md` (lines 688–693):**
- `contractSlots` (outer, contractId → per-contract cache): `expireAfterAccess=5m, maximumSize=3000`
- `slotsPerContract` (inner, per-contract, slotKey → EMPTY_VALUE): `expireAfterAccess=5m, maximumSize=1500`
- `contractState` (slot values): `expireAfterWrite=2s, maximumSize=25000`

**Why `expireAfterAccess` does not self-heal:** Caffeine's `asMap().keySet()` iteration at line 91 does **not** count as an access for `expireAfterAccess` purposes. Only explicit `Cache.get()` / `Cache.put()` calls reset the timer. The only key whose timer is reset on each legitimate request is the one key passed to `putIfAbsent` (line 90). The 1,499 attacker-seeded keys are only iterated, so their timers run independently from the moment of seeding and expire 5 minutes after their last explicit access.

**Why the `contractState` cache does not protect:** Its `expireAfterWrite=2s` TTL means that within any 2-second window the batch result is served from cache. But every ≥2 seconds any new request for the target contract misses the `contractState` cache and re-enters `findStorageBatch`, re-issuing the full 1,500-key batch query. Critically, attacker-seeded slots that do not exist in the DB are never written into `contractState` cache (line 109 only stores values returned by the DB), so they never contribute to cache hits — they only inflate the batch query size.

## Impact Explanation

Each `findStorageBatch` call with 1,500 keys issues a single SQL query with a 1,500-element `IN` clause against the `contract_state` table:

```sql
select slot, value from contract_state
where contract_id = :contractId
and slot in (:slots)   -- 1,500 elements
```

Compared to the baseline single-key lookup, this query transfers ~1,500× more key data to the DB and forces the DB to evaluate 1,500 index lookups per call. For any contract receiving sustained traffic, the DB receives a 1,500-key batch query every ≥2 seconds per concurrent request thread. With the global 500 RPS limit and a popular contract handling even 50 RPS, this translates to 50 × 1,500 = 75,000 slot lookups per second instead of 50 — a >1,000× amplification of DB read load for that contract's queries. The attack degrades service for all users sharing the DB connection pool.

## Likelihood Explanation

- **No authentication required.** The `/api/v1/contracts/call` endpoint is public.
- **Seeding cost is trivial.** 1,500 requests at the 500 RPS global limit takes 3 seconds. Each request need only call a contract function that reads a mapping/array slot at an attacker-controlled index (e.g., `balanceOf(address)` on any ERC-20).
- **Maintenance cost is low.** To keep the seeded keys alive past 5 minutes, the attacker re-seeds every ~4.5 minutes (5 requests/second sustained, well within rate limits).
- **No per-IP rate limiting exists.** The `rateLimitBucket` in `ThrottleConfiguration.java` (lines 24–32) is a single global token bucket (`requestsPerSecond=500`); there is no per-source throttle.
- **Fully repeatable and automatable** with a simple script.

## Recommendation

1. **Register keys only after DB confirmation:** Move `putIfAbsent` to after the DB query returns, and only register keys that were actually returned by `findStorageBatch`. This ensures the slot registry only grows with keys that have real on-chain state.

2. **Alternatively, store `EMPTY_VALUE` in `contractState` for non-existent slots:** After the batch query, for any key in `cachedSlots` that was not returned by the DB, store `EMPTY_VALUE` in `contractState` cache. This causes subsequent requests for those slots to hit the `contractState` cache and bypass `findStorageBatch` entirely, preventing re-registration.

3. **Short-term mitigation:** Set `hiero.mirror.web3.cache.enableBatchContractSlotCaching=false` to disable the batch caching feature entirely, falling back to single-key lookups.

4. **Defense in depth:** Add per-contract-per-IP rate limiting at the API gateway layer to limit the rate at which new slot keys can be registered for a single contract.

## Proof of Concept

```
# Step 1: Seed 1,500 distinct slot keys for target contract C
# Each request reads balanceOf(address_i) — a different mapping slot
for i in range(1500):
    eth_call(
        to=CONTRACT_C,
        data=encode("balanceOf(address)", attacker_address_i)
    )
# After 1,500 requests: slotsPerContract[C] contains 1,500 attacker keys

# Step 2: Wait for contractState cache to expire (>2s)
sleep(3)

# Step 3: Any legitimate user request for contract C now triggers:
#   findStorageBatch(C, [slot_1, slot_2, ..., slot_1500])
# → DB receives: SELECT slot, value FROM contract_state
#                WHERE contract_id = C AND slot IN (1500 elements)
# instead of: SELECT slot, value FROM contract_state
#             WHERE contract_id = C AND slot = <single_key>

# Step 4: Maintain amplification — re-seed every ~4.5 minutes
# (5 requests/second, well within the 500 RPS global limit)
```

```java
    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```
