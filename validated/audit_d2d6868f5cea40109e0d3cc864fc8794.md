Audit Report

## Title
GraphQL `findByAlias` Leaks Entities with `deleted = NULL` Due to `IS NOT TRUE` vs `<> true` SQL Predicate Inconsistency

## Summary
The GraphQL `EntityRepository.findByAlias` uses the SQL predicate `deleted is not true`, which under SQL three-valued logic returns rows where `deleted` is `NULL`. Every other service (importer, rest-java) uses `deleted <> true`, which correctly excludes `NULL` rows. This inconsistency allows an unprivileged caller of the public GraphQL endpoint to retrieve entities that all other services treat as non-existent.

## Finding Description
**Verified code locations:**

- **GraphQL** — `graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java`, line 13: [1](#0-0) 

- **Importer** — `importer/src/main/java/org/hiero/mirror/importer/repository/EntityRepository.java`, line 16: [2](#0-1) 

- **Rest-Java** — `rest-java/src/main/java/org/hiero/mirror/restjava/repository/EntityRepository.java`, line 13: [3](#0-2) 

**Root cause — SQL three-valued logic:**

| `deleted` value | `deleted IS NOT TRUE` | `deleted <> true` |
|---|---|---|
| `false` | TRUE (row returned) | TRUE (row returned) |
| `NULL` | **TRUE (row returned)** | NULL → FALSE **(row excluded)** |
| `true` | FALSE (row excluded) | FALSE (row excluded) |

`IS NOT TRUE` is a boolean test that treats `NULL` as "not true"; `<> true` is a comparison that propagates `NULL` as unknown/false. The GraphQL query therefore surfaces any entity whose `deleted` column is `NULL`.

**Note:** The same `IS NOT TRUE` predicate is also used in `findByEvmAddress` on line 16 of the GraphQL repository, making that lookup equally affected. [4](#0-3) 

**Failed assumption:** The GraphQL repository assumes `deleted is not true` is semantically equivalent to `deleted <> true`. It is not when `deleted` can be `NULL`.

**No application-layer guard:** `EntityServiceImpl.getByAliasAndType` only filters the result by `EntityType` after retrieval; it does not check the `deleted` field. [5](#0-4) 

**Test gap confirmed:** The GraphQL `EntityRepositoryTest.findByAlias` test (lines 22–26) only tests a default entity (where `deleted = false`). There is no test case for `deleted = null`. [6](#0-5) 

By contrast, the rest-java `EntityRepositoryTest` explicitly creates an entity with `deleted(null)` and asserts `findByAlias` returns empty for it. [7](#0-6) 

## Impact Explanation
An attacker can observe entities via GraphQL that the importer and rest-java treat as non-existent. This creates a split-brain view of ledger state: GraphQL reports an account as resolvable by alias while the transaction-processing pipeline (importer) cannot resolve the same alias to any entity. Downstream consumers relying on GraphQL for account lookups will see a different set of active accounts than consumers relying on the REST or importer APIs, violating ledger-state integrity guarantees for affected entities. No funds can be moved, but the inconsistency is observable and exploitable for information disclosure of entities in an indeterminate deletion state.

## Likelihood Explanation
Entities with `deleted = NULL` are a realistic database state: any entity row written before the `deleted` column was introduced, or any entity whose import processing was interrupted before a definitive deletion status was committed, will have `deleted = NULL`. No special privilege is required — the GraphQL endpoint is public. The attacker only needs to know or enumerate a valid alias, which is derivable from public key material. The condition is repeatable for every such entity in the database.

## Recommendation
Replace `deleted is not true` with `deleted <> true` in both queries in the GraphQL `EntityRepository`:

```java
// graphql/src/main/java/org/hiero/mirror/graphql/repository/EntityRepository.java

@Query(value = "select * from entity where alias = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByAlias(byte[] alias);

@Query(value = "select * from entity where evm_address = ?1 and deleted <> true", nativeQuery = true)
Optional<Entity> findByEvmAddress(byte[] evmAddress);
```

Add test cases to `EntityRepositoryTest` (graphql) mirroring the rest-java test suite: persist an entity with `deleted = null` and assert that `findByAlias` and `findByEvmAddress` return empty for it. [8](#0-7) 

## Proof of Concept
1. Insert a row into the `entity` table with a known `alias` value and `deleted = NULL` (e.g., an entity mid-import or pre-migration).
2. Call the public GraphQL endpoint:
   ```graphql
   { account(alias: "<base32_alias>") { id balance } }
   ```
   via `EntityServiceImpl.getByAliasAndType` → `EntityRepository.findByAlias`.
3. Observe the entity is returned in full — `deleted is not true` matches the `NULL` row.
4. Query the same alias through the importer's `EntityIdServiceImpl.findByAlias` → importer `EntityRepository.findByAlias` (`deleted <> true`): returns empty.
5. Query the same alias through rest-java's `EntityRepository.findByAlias` (`deleted <> true`): returns empty.
6. GraphQL returns the entity; importer and rest-java return nothing — confirming the split-brain visibility.
