Audit Report

## Title
Missing Timestamp Bound in `getLatestFileDataContents` Allows Post-Transaction File Mutation to Corrupt Historical `call_data` in API Response

## Summary
`getContractResultsByTimestamp` and `getContractResultsByTransactionIdOrHash` both extract a consensus timestamp from the request but never forward it to `FileDataService.getLatestFileDataContents`. The call uses `{whereQuery: []}`, causing the underlying SQL to resolve the **current** latest version of the referenced file rather than its state at the transaction's consensus timestamp. Any account holding the key to the referenced Hedera file can issue a `FileUpdate` after their transaction reaches consensus, causing the mirror node API to return fabricated `call_data` for that historical transaction.

## Finding Description
In `rest/controllers/contractController.js`, `getContractResultsByTimestamp` (lines 1002–1042):

```js
// line 1010 — timestamp extracted from path param
const {contractId, timestamp} = await getAndValidateContractIdAndConsensusTimestampPathParams(req);
...
// lines 1022-1023 — timestamp is never forwarded
if (utils.isValidUserFileId(ethTransaction?.callDataId)) {
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
}
``` [1](#0-0) 

The same defect is present in `getContractResultsByTransactionIdOrHash` at line 1184: [2](#0-1) 

**Root cause — SQL has no upper timestamp bound:**

`FileDataService.latestFileContentsQuery` (lines 19–42 of `rest/service/fileDataService.js`) works by finding `max(consensus_timestamp)` of the most recent `FileCreate`/`FileUpdate` (transaction types 17/19) for the given file ID, then aggregating all subsequent file chunks. With `{whereQuery: []}`, neither the inner CTE nor the outer aggregation has an upper bound on `consensus_timestamp`, so the query always resolves to the **current** latest version of the file. [3](#0-2) 

**Contrast with the correct pattern:**

`FileDataService.getFileDataQuery` (lines 44–63) and `getFileData` (lines 71–75) are the timestamp-aware variants. They enforce `consensus_timestamp <= $2` in both the inner subquery and the outer filter, correctly bounding the result to the file's state at the moment of the transaction. [4](#0-3) 

**Failed assumption:** The code assumes the file referenced by `callDataId` is immutable after the transaction, or that "latest" equals "at-transaction-time." Neither is guaranteed on Hedera; files are mutable by their key holders.

## Impact Explanation
The `call_data` field in the API response for `/contracts/{contractId}/results/{timestamp}` and `/contracts/results/{transactionIdOrHash}` will reflect the **post-mutation** file content rather than the bytes that were actually submitted in the transaction. A verifier, auditor, or dApp relying on the API to reconstruct what was called in a past Ethereum transaction will receive falsified input data. An attacker can make it appear that a different function selector or different ABI-encoded arguments were used, enabling retroactive misrepresentation of on-chain activity without any privileged access. Note: the `function_parameters` field is stored immutably at import time by the importer, so the impact is scoped to the `call_data` supplementary field in the REST response. [5](#0-4) 

## Likelihood Explanation
Preconditions are minimal and require no special privilege:
1. The attacker holds the key to a Hedera file — achievable by any account via a standard `FileCreate` transaction.
2. The attacker submits an Ethereum transaction referencing that file as `callDataId` — a normal user operation.
3. After the transaction reaches consensus, the attacker issues a `FileUpdate` or `FileAppend` to overwrite the file content — again a normal user operation requiring only the file key.

The attack is fully repeatable, costs only normal Hedera transaction fees, and requires no knowledge of internal mirror node state.

## Recommendation
Replace the unbounded `getLatestFileDataContents` calls with the timestamp-aware `getFileData` method, passing the transaction's consensus timestamp as the upper bound. Specifically:

- In `getContractResultsByTimestamp`, replace:
  ```js
  fileData = await FileDataService.getLatestFileDataContents(ethTransaction.callDataId, {whereQuery: []});
  ```
  with:
  ```js
  fileData = await FileDataService.getFileData(ethTransaction.callDataId, timestamp);
  ```
- Apply the same fix in `getContractResultsByTransactionIdOrHash` at line 1184, using `ethTransaction.consensusTimestamp` (or the resolved `transactionDetails.consensusTimestamp`) as the bound.

This mirrors the correct pattern already used in `getFileDataQuery`, which enforces `consensus_timestamp <= $2` in both the inner subquery and the outer filter. [6](#0-5) 

## Proof of Concept
1. Create a Hedera file (`FileCreate`) with content `0xdeadbeef`. Note the file ID (e.g., `0.0.12345`).
2. Submit an Ethereum transaction with `callDataId = 0.0.12345` (no inline `callData`). Note the consensus timestamp `T`.
3. Query `GET /api/v1/contracts/{contractId}/results/{T}` — observe `call_data: 0xdeadbeef`.
4. Issue a `FileUpdate` on `0.0.12345` changing content to `0xcafebabe`.
5. Re-query `GET /api/v1/contracts/{contractId}/results/{T}` — observe `call_data: 0xcafebabe`, despite the transaction at time `T` having used `0xdeadbeef`.

The same result is reproducible via `GET /api/v1/contracts/results/{transactionIdOrHash}`. [7](#0-6) [3](#0-2)
