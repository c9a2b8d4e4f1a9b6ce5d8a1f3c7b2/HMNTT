# Audit Report

## Title
Unbounded Per-Topic Subscription Flood Enables DoS via Resource Exhaustion in `subscribeTopic()`

## Summary
`TopicMessageServiceImpl.subscribeTopic()` accepts an unlimited number of concurrent subscriptions from any unauthenticated caller. The global `subscriberCount` counter is registered solely as a Micrometer `Gauge` and is never compared against any configured maximum. An unprivileged attacker can open thousands of subscriptions to a single topic, exhausting thread pool capacity, heap memory (per-subscriber backpressure buffers), and database connections, starving legitimate subscribers.

## Finding Description

**Code path:** `ConsensusController.subscribeTopic()` → `TopicMessageServiceImpl.subscribeTopic()` → `topicListener.listen()` (via `incomingMessages()`)

In `TopicMessageServiceImpl.java`, `subscriberCount` is declared and registered exclusively as an observability gauge:

```java
// lines 48, 52–55
private final AtomicLong subscriberCount = new AtomicLong(0L);
Gauge.builder("hiero.mirror.grpc.subscribers", () -> subscriberCount)
    .description("The number of active subscribers")
    .tag("type", TopicMessage.class.getSimpleName())
    .register(meterRegistry);
``` [9](#0-8) 

In `subscribeTopic()`, it is incremented/decremented for observability only — never checked against any maximum:

```java
// lines 87–91
return topicExists(filter)
    .thenMany(flux.doOnNext(topicContext::onNext)
        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
        .doFinally(s -> subscriberCount.decrementAndGet())
        .doFinally(topicContext::finished));
``` [10](#0-9) 

`topicExists()` only validates entity existence and type `TOPIC` — no authentication, no per-topic subscriber cap, no per-caller rate limit: [11](#0-10) 

The only connection-level guard is `maxConcurrentCallsPerConnection = 5` in `NettyProperties.java`: [6](#0-5) 

Applied in `GrpcConfiguration.java` line 33: [12](#0-11) 

This is trivially bypassed by opening multiple TCP connections.

**Per listener type, resource impact differs:**

- **`POLL` mode (`PollingTopicListener`)**: Each subscription creates an independent `Flux.defer(() -> poll(context)).repeatWhen(...)` polling loop on `Schedulers.boundedElastic()`. N subscriptions = N independent DB polling loops, directly exhausting the DB connection pool and bounded elastic thread pool. [13](#0-12) 

- **`SHARED_POLL` / `REDIS` mode (`SharedTopicListener`)**: `listen()` applies `onBackpressureBuffer(listenerProperties.getMaxBufferSize(), BufferOverflowStrategy.ERROR)` **per subscriber**. With the default `maxBufferSize = 16384`, thousands of subscribers each holding a 16384-element buffer causes heap exhaustion. Additionally, `publishOn(Schedulers.boundedElastic(), false, listenerProperties.getPrefetch())` allocates a bounded elastic thread slot per active subscriber delivery. [3](#0-2) [14](#0-13) 

## Impact Explanation

- **POLL mode**: Direct DB connection pool exhaustion. Each attacker subscription issues periodic `SELECT` queries via `topicMessageRepository.findByFilter()`. With hundreds of subscriptions, the DB pool is saturated, causing legitimate subscribers' historical retrieval and live polling to time out or fail.
- **SHARED_POLL/REDIS mode**: Heap exhaustion from per-subscriber 16384-element backpressure buffers. Thread pool exhaustion from `publishOn(Schedulers.boundedElastic())` per subscriber. Legitimate subscribers experience message delivery delays or `OutOfMemoryError`-induced JVM crashes affecting all topics.
- **Severity: High** — complete denial of service to all subscribers of the targeted topic, with potential JVM crash affecting the entire service.

## Likelihood Explanation

- **Precondition**: Network access to gRPC port 5600. No credentials required. The gRPC endpoint is public-facing by design.
- **Bypass of `maxConcurrentCallsPerConnection=5`**: An attacker opens 200 TCP connections × 5 calls = 1000 concurrent subscriptions. Trivially scriptable with `grpcurl` or any gRPC client library.
- **Repeatability**: Fully repeatable. Subscriptions with no `endTime` and no `limit` never terminate (`pastEndTime()` returns `Flux.never()` when `endTime == null`), so attacker subscriptions persist indefinitely.
- **No authentication barrier**: `ConsensusController` and `TopicMessageServiceImpl` contain zero authentication or authorization logic. [7](#0-6) [15](#0-14) 

## Recommendation

1. **Enforce a global subscriber cap**: Compare `subscriberCount.get()` against a configurable `maxSubscribers` property before accepting a new subscription in `subscribeTopic()`, returning `RESOURCE_EXHAUSTED` gRPC status if exceeded.
2. **Enforce a per-topic subscriber cap**: Track per-topic subscription counts in a `ConcurrentHashMap<EntityId, AtomicLong>` and reject subscriptions exceeding a configurable per-topic limit.
3. **Add per-IP/caller rate limiting**: Implement a token-bucket or sliding-window rate limiter at the `ConsensusController` level, keyed on the remote peer address from the gRPC context.
4. **Require subscriptions to have an `endTime` or `limit`**: Reject or auto-expire open-ended subscriptions after a configurable maximum duration.
5. **Tune `maxConcurrentCallsPerConnection`**: Consider lowering the default further and adding a global `maxConnectionAge` to prevent indefinite connection holding.

## Proof of Concept

```bash
# Open 200 TCP connections × 5 subscriptions each = 1000 concurrent subscriptions
# to topic 0.0.12345 with no endTime (never terminates)
for i in $(seq 1 200); do
  for j in $(seq 1 5); do
    grpcurl -plaintext -d '{
      "topicID": {"shardNum": 0, "realmNum": 0, "topicNum": 12345},
      "consensusStartTime": {"seconds": 0}
    }' <mirror-node-host>:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
  done
done
wait
```

In **POLL mode**: monitor DB connection pool saturation — legitimate queries begin timing out after pool exhaustion.

In **SHARED_POLL/REDIS mode**: monitor JVM heap via `jconsole` or Micrometer metrics — heap climbs at ~`16384 × sizeof(TopicMessage)` per attacker subscription until `OutOfMemoryError` or GC thrashing renders the service unresponsive.
