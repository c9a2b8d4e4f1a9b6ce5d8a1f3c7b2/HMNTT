# Audit Report

## Title
Unbounded Operation Count in `getOperationSlice()` Enables Memory Exhaustion DoS on Rosetta Mirror Node

## Summary
`getOperationSlice()` in `rosetta/app/services/construction_service.go` allocates and iterates over every element of the caller-supplied `operations` slice with no upper bound on count. Both `ConstructionPreprocess` and `ConstructionPayloads` pass raw, unauthenticated request operations directly to this function before any count validation. A single POST request with tens of thousands of crafted operations causes unbounded heap growth, OOM-killing the Rosetta process.

## Finding Description

**Code path:**

`ConstructionPreprocess` (line 289) and `ConstructionPayloads` (line 232) both call `getOperationSlice(request.Operations)` with no prior count check:

```go
// construction_service.go lines 432–465
func (c *constructionAPIService) getOperationSlice(operations []*rTypes.Operation) (
    types.OperationSlice,
    *rTypes.Error,
) {
    operationSlice := make(types.OperationSlice, 0, len(operations)) // cap = attacker-controlled
    for _, operation := range operations {
        accountId, err = types.NewAccountIdFromString(...)
        amount, rErr = types.NewAmount(...)
        operationSlice = append(operationSlice, types.Operation{...})
    }
    return operationSlice, nil
}
```

**Root cause:** No check on `len(operations)` before the loop. The slice is pre-allocated with `cap = len(operations)` (attacker-controlled), and each iteration performs string parsing and struct construction.

**Why downstream checks are insufficient:**

`validateOperations` is called only *after* `getOperationSlice()` returns, inside `transactionHandler.Preprocess`/`Construct`. For `cryptoTransfer`, it is called with `size=0` (no count limit):

```go
// crypto_transfer_transaction_constructor.go line 125
if err := validateOperations(operations, 0, c.GetOperationType(), false); err != nil {
```

`cryptoTransferTransactionConstructor.preprocess()` then allocates a second slice (`transfers := make([]transfer, 0, len(operations))`) and a `totalAmounts` map, iterating every operation again.

**No application-level body size limit exists.** A search of all rosetta Go source finds zero uses of `http.MaxBytesReader`, `ReadLimit`, or any equivalent. The HTTP server sets timeout fields only — no body byte cap:

```go
// main.go lines 220–227
httpServer := &http.Server{
    Addr:              fmt.Sprintf(":%d", rosettaConfig.Port),
    Handler:           corsMiddleware,
    IdleTimeout:       rosettaConfig.Http.IdleTimeout,
    ReadHeaderTimeout: rosettaConfig.Http.ReadHeaderTimeout,
    ReadTimeout:       rosettaConfig.Http.ReadTimeout,
    WriteTimeout:      rosettaConfig.Http.WriteTimeout,
}
```

The `Http` config struct contains only duration fields and no body size field.

**The Traefik middleware** (`inFlightReq: 5`, `rateLimit: average: 10`) is gated on `{{ if and .Values.global.middleware .Values.middleware }}` and `global.middleware` defaults to `false`. It is not deployed by default, is bypassable by direct pod access, and does not limit request body size in any case.

## Impact Explanation
A single malformed POST to `/construction/preprocess` or `/construction/payloads` (no authentication required, available in both online and offline modes) with a large number of `cryptoTransfer` operations (amounts balanced to sum to zero to pass the `totalAmounts` check) causes the Rosetta Go process to allocate unbounded heap. The container memory limit of `200Mi` means the OOM threshold is low — a modest payload is sufficient to trigger an OOM kill. Repeated requests, or a single sufficiently large one, crash the process and disrupt the mirror node's ability to forward transactions via the Rosetta Construction API.

## Likelihood Explanation
The endpoint requires no credentials, no API key, and no prior state. The exploit is a single HTTP POST with a crafted JSON body. Any external user who can reach the Rosetta port (default 5700) can trigger it. The attack is repeatable and automatable. The only infrastructure-level mitigation (Traefik middleware) is disabled by default and bypassable by direct pod access.

## Recommendation
1. **Add an operation count limit** at the start of `getOperationSlice()`:
   ```go
   const maxOperations = 1000 // or a configurable value
   if len(operations) > maxOperations {
       return nil, errors.ErrInvalidOperations
   }
   ```
2. **Add a request body size limit** using `http.MaxBytesReader` in a middleware wrapping the router, e.g.:
   ```go
   http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
       r.Body = http.MaxBytesReader(w, r.Body, maxBodyBytes)
       next.ServeHTTP(w, r)
   })
   ```
3. **Enable the Traefik middleware by default** or document it as a required security control.
4. **Set a non-zero `ReadTimeout`** in the HTTP server configuration to bound the time spent reading large request bodies.

func (c *constructionAPIService) getOperationSlice(operations []*rTypes.Operation) (
	types.OperationSlice,
	*rTypes.Error,
) {
	operationSlice := make(types.OperationSlice, 0, len(operations))
	for _, operation := range operations {
		var accountId types.AccountId
		if operation.Account != nil {
			var err error
			accountId, err = types.NewAccountIdFromString(operation.Account.Address, c.systemShard, c.systemRealm)
			if err != nil || accountId.IsZero() {
				return nil, errors.ErrInvalidAccount
			}
		}

		var amount types.Amount
		var rErr *rTypes.Error
		if operation.Amount != nil {
			if amount, rErr = types.NewAmount(operation.Amount); rErr != nil {
				return nil, rErr
			}
		}

		operationSlice = append(operationSlice, types.Operation{
			AccountId: accountId,
			Amount:    amount,
			Index:     operation.OperationIdentifier.Index,
			Metadata:  operation.Metadata,
			Type:      operation.Type,
		})
	}

	return operationSlice, nil
}
