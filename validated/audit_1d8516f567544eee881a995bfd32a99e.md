## Audit Report

## Title
Unbounded O(N×M) CPU Exhaustion via Shared Flux Fan-out in `SharedPollingTopicListener` (SHARED_POLL mode)

## Summary
In `SHARED_POLL` mode, `SharedPollingTopicListener.getSharedListener()` unconditionally returns a single shared `topicMessages` Flux for every subscriber regardless of `topicId`. `CompositeTopicListener.listen()` attaches a per-subscriber `.filter()` operator to that shared Flux, meaning every emitted message is evaluated against every active subscriber's filter. With no per-IP connection cap anywhere in the stack, an unprivileged attacker can open an arbitrary number of subscriptions and drive CPU usage to O(N × M) per poll cycle.

## Finding Description

**`SharedPollingTopicListener.getSharedListener()` ignores the filter entirely** and returns the single pre-built `topicMessages` Flux for every caller, regardless of `topicId`: [1](#0-0) 

This is in direct contrast to `RedisTopicListener.getSharedListener()`, which correctly uses `computeIfAbsent` keyed on the per-topic channel name to maintain separate shared Fluxes per topic: [2](#0-1) 

The single shared Flux is constructed once at bean creation time and polls `topicMessageRepository.findLatest()` — which fetches messages across **all topics** — up to `maxPageSize` (default 5,000) messages every `interval` (default 500 ms): [3](#0-2) [4](#0-3) 

`SharedTopicListener.listen()` wraps the shared Flux with backpressure buffering and `publishOn`, then `CompositeTopicListener.listen()` chains a per-subscriber `.filter(t -> filterMessage(t, filter))` on the result: [5](#0-4) [6](#0-5) 

The `filterMessage` predicate performs a `topicId` equality check and a `consensusTimestamp` comparison for every (message, subscriber) pair: [7](#0-6) 

**`startTime=0` passes all validation.** `TopicMessageFilter` accepts `startTime=0` via `@Min(0)` and `isValidStartTime()` (which checks `startTime <= DomainUtils.now()`, always true for 0): [8](#0-7) 

**`subscriberCount` is a metrics gauge only, not a guard.** There is no check that rejects new subscriptions when the count is too high: [9](#0-8) 

**The only connection-level limit is `maxConcurrentCallsPerConnection = 5` per TCP connection**, which limits calls per connection but places no cap on the number of connections from a single IP: [10](#0-9) [11](#0-10) 

**Root cause:** `getSharedListener()` was designed to share one Flux across subscribers of the *same* topic (reducing DB load), but it returns the identical Flux for *all* topics. Reactor's `.share()` fan-out delivers each emitted item to every subscriber's operator chain before moving to the next item, so N subscribers each paying O(1) per message yields O(N×M) total work per poll cycle.

## Impact Explanation
With default settings (500 ms poll interval, 5,000 messages/poll), an attacker holding N open subscriptions forces N × 5,000 `filterMessage` evaluations every 500 ms on the gRPC process's bounded-elastic thread pool. At sufficient N, this saturates CPU cores, stalls the shared Flux's emission loop, and causes backpressure buffer overflows (`maxBufferSize` default 16,384) for legitimate subscribers — effectively denying service to all users of the mirror node's topic subscription API. This is a process-internal CPU/memory exhaustion, not a network-layer flood.

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly reachable with no authentication. Opening many long-lived gRPC streaming connections is trivial with any standard gRPC client library. The attack is repeatable and requires no special knowledge beyond the public API schema. It is effective only when the deployment uses `SHARED_POLL` mode (not the default `REDIS`), but operators who switch to `SHARED_POLL` to reduce Redis dependency are fully exposed with no in-process safeguard.

## Recommendation
1. **Fix `getSharedListener()`** to maintain a per-`topicId` map of shared Fluxes, mirroring the pattern already used in `RedisTopicListener` (`ConcurrentHashMap` + `computeIfAbsent` keyed on topic ID). This reduces fan-out from O(all_subscribers) to O(subscribers_for_that_topic).
2. **Add a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()` that rejects new subscriptions when `subscriberCount` exceeds a configurable threshold, converting the existing metrics gauge into an enforcement guard.
3. **Add a per-IP or per-connection subscription limit** at the gRPC interceptor layer to prevent a single client from monopolizing the shared Flux.

## Proof of Concept
```java
// Attacker opens N connections, each with maxConcurrentCallsPerConnection (5) subscriptions
// targeting different or the same topicId, all with startTime=0
ManagedChannel channel = ManagedChannelBuilder.forAddress(host, 5600).usePlaintext().build();
ConsensusServiceGrpc.ConsensusServiceStub stub = ConsensusServiceGrpc.newStub(channel);

for (int i = 0; i < 2000; i++) {
    stub.subscribeTopic(
        ConsensusTopicQuery.newBuilder()
            .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
            // consensusStartTime omitted → defaults to epoch 0, passes @Min(0) and isValidStartTime()
            .build(),
        new StreamObserver<>() { /* no-op */ }
    );
}
// Result: every poll cycle emits up to 5,000 messages × 2,000 active filter lambdas
// = 10,000,000 filterMessage() evaluations per 500ms on the bounded-elastic pool
```
