# Audit Report

## Title
UNION ALL Double-Counting of Token Allowance Records via Degenerate `spender.id gte X lte X` Range with `token.id` Bounds

## Summary
When a caller supplies identical lower and upper bounds for `spender.id` (e.g., `spender.id=gte:5&spender.id=lte:5`) together with a `token.id` range, the multi-union query builder in `tokenAllowanceController.js` / `baseController.js` produces three subqueries joined with `UNION ALL`. The lower and upper subqueries both match `spender = N`, causing every row in the intersection to be returned twice.

## Finding Description

**Code path:**
- `rest/controllers/tokenAllowanceController.js` `extractTokenMultiUnionQuery` (lines 22–60) parses filters into `Bound` objects and calls `getLowerFilters`, `getInnerFilters`, `getUpperFilters`. [8](#0-7) 
- `rest/controllers/baseController.js` `validateBoundsRange` (lines 117–123) only rejects the combination of a range operator *and* an `eq` operator on the same field. It does not reject `gte X` paired with `lte X` when X is identical. [1](#0-0) 
- `validateLowerBounds` (lines 83–92) only rejects the case where `primary.lower.operator === gt`; it never compares the actual bound values. [2](#0-1) 
- `validateUpperBounds` (lines 100–109) is symmetric and equally blind to value equality. [3](#0-2) 
- `getLowerFilters` (line 141) emits `[{spender, eq, N}, {token_id, gte, T1}]` when both primary and secondary have lower bounds. [4](#0-3) 
- `getUpperFilters` (lines 181–182) emits `[{spender, eq, N}, {token_id, lte, T2}]` when both primary and secondary have upper bounds. [6](#0-5) 
- `getInnerFilters` (lines 160–167) emits `[{spender, gt, N}, {spender, lt, N}]` — a vacuous (always-false) filter, but its non-zero length causes it to be included as a subquery. [5](#0-4) 
- `rest/service/tokenAllowanceService.js` `getQuery` (lines 63–74) joins all non-empty filter arrays with `UNION ALL`. [7](#0-6) 

**Root cause:** The validation layer assumes that `gte X` and `lte X` on the same field with the same value `X` is a semantically distinct case from `eq X`. It is not: the degenerate range `[X, X]` is equivalent to equality, but the code treats it as a range, generating two non-empty subqueries that both resolve to `spender = N`. No guard compares `primary.lower.value` against `primary.upper.value`.

**Resulting SQL (simplified):**
```sql
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$3 AND token_id>=$4 ...)
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender>$5 AND spender<$6 ...)  -- always empty
UNION ALL
(SELECT * FROM token_allowance WHERE owner=$1 AND amount>0 AND spender=$7 AND token_id<=$8 ...)
ORDER BY ... LIMIT $2
```
With `$3=$7=N`, `$5=$6=N`, `$4=T1`, `$8=T2`. Every row satisfying `spender=N AND T1 <= token_id <= T2` is returned by both the first and third subqueries.

## Impact Explanation
Consumers of `/accounts/:id/allowances/tokens` receive duplicate records for any token allowance where `spender = N` and `T1 <= token_id <= T2`. Any downstream system that sums, counts, or aggregates allowances will compute inflated figures. In financial or authorization contexts this constitutes incorrect state exported from the mirror node, violating data integrity guarantees. The outer `LIMIT $2` is applied after the `UNION ALL`, so the effective result set before the limit is doubled, which can also cause pagination to behave incorrectly.

## Likelihood Explanation
The trigger is a single unauthenticated GET request with four query parameters. No account ownership, special role, or prior knowledge beyond a valid account ID is required. The degenerate range is not rejected by any existing validation. It is trivially scriptable and repeatable on demand.

## Recommendation
Add a value-equality check in `validateBoundsRange` (or a dedicated guard in `validateBounds`) that rejects the case where `primary.lower.value === primary.upper.value` when both a lower and upper bound are present. Concretely, in `baseController.js` after the existing `hasBound() && hasEqual()` check, add:

```js
if (bounds.primary.hasLower() && bounds.primary.hasUpper() &&
    bounds.primary.lower.value === bounds.primary.upper.value) {
  throw new InvalidArgumentError(
    `${bounds.primary.filterKey} gte and lte values must not be equal; use eq instead`
  );
}
```

This collapses the degenerate range into an error (or alternatively, the caller could be redirected to use `eq`), preventing the split into two overlapping subqueries.

## Proof of Concept

```
GET /api/v1/accounts/0.0.1234/allowances/tokens
    ?spender.id=gte:5&spender.id=lte:5&token.id=gte:100&token.id=lte:200
```

Pre-condition: at least one token allowance row exists with `owner=1234`, `spender=5`, `token_id` in `[100, 200]`, `amount > 0`.

Expected: each matching row appears once.
Actual: each matching row appears twice (once from the lower subquery `spender=5 AND token_id>=100`, once from the upper subquery `spender=5 AND token_id<=200`).
