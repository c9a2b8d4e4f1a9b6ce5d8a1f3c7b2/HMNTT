Audit Report

## Title
Missing Application-Level Rate Limiting on `getByEvmAddressAndType()` Enables Database Connection Pool Exhaustion DoS

## Summary
The `graphql` module exposes an unauthenticated `/graphql/alpha` endpoint that accepts `account(input:{evmAddress:"0x..."})` queries. Each such query triggers an uncached, live SQL query against the `entity` table with no application-level throttle. Infrastructure-level mitigations (Traefik `inFlightReq`, GCP gateway rate limit) are either disabled by default or trivially bypassed. A sustained flood of unique EVM address lookups can exhaust the HikariCP connection pool and deny service to all users.

## Finding Description

**Code path — confirmed line-for-line:**

`AccountController.account()` at lines 51–54 unconditionally calls `entityService.getByEvmAddressAndType(evmAddress, EntityType.ACCOUNT)` for any request supplying an `evmAddress` field. [1](#0-0) 

`EntityServiceImpl.getByEvmAddressAndType()` at lines 34–41 decodes the address and unconditionally calls either `entityRepository.findById()` or `entityRepository.findByEvmAddress()` — a live SQL query on every invocation. There is no `@Cacheable`, no counter, and no guard of any kind. [2](#0-1) 

`EntityRepository.findByEvmAddress()` executes a native SQL query `select * from entity where evm_address = ?1 and deleted is not true` against the database on every call. [3](#0-2) 

**No application-level rate limiting in the `graphql` module:**

A search for `ThrottleManager`, `RateLimiter`, `throttle`, or `RateLimit` across all of `graphql/src/main/java/**` returns zero matches. No `@Cacheable` annotation exists anywhere in the graphql module's Java source either.

**GraphQL structural protections do not address request rate:**

`GraphQlConfiguration` applies `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)`. These reject structurally abusive queries but impose no limit on the *rate* of simple, valid `account(input:{evmAddress:"0x..."})` queries. [4](#0-3) 

**Traefik middleware is disabled by default:**

The `middleware.yaml` template is gated by `{{ if and .Values.global.middleware .Values.middleware }}`. The `global.middleware` flag defaults to `false` in the graphql chart's `values.yaml`. [5](#0-4) [6](#0-5) 

When the middleware *is* enabled, the `inFlightReq` limit of 5 concurrent requests is per source IP and is trivially bypassed by distributing requests across multiple IPs. The `retry: attempts: 3` middleware in the same chain amplifies each attacker request threefold at the load-balancer layer. [7](#0-6) 

**GCP gateway rate limit requires explicit opt-in:**

The `maxRatePerEndpoint: 250` backend policy is per-pod and is only rendered when `global.gateway.enabled` is `true`, which defaults to `false`. [8](#0-7) [9](#0-8) 

## Impact Explanation

Each concurrent request to `getByEvmAddressAndType()` holds a HikariCP database connection for the duration of the SQL query. With unique EVM addresses per request, no result can be served from any cache. Once the pool is exhausted, all GraphQL queries — including those from legitimate users — block waiting for a connection, producing effective denial of service for the entire `/graphql/alpha` endpoint. The `GraphQLHighDBConnections` Prometheus alert fires only after 75% utilization sustained for 5 minutes, meaning the service is already degraded before any automated response. [10](#0-9) 

## Likelihood Explanation

No authentication or API key is required to reach `/graphql/alpha`. The query payload is minimal — a single GraphQL `account` query with a 40-hex-character `evmAddress`. An attacker needs only a script and a modest number of source IPs (or a single IP when the Traefik middleware is not deployed, which is the default) to sustain thousands of concurrent requests. The attack is fully repeatable and requires no special knowledge beyond the public GraphQL schema.

## Recommendation

1. **Add application-level rate limiting to the `graphql` module.** Integrate bucket4j (already used in the `web3` module) or Spring's `@RateLimiter` into the GraphQL request path, keyed by source IP or a token, before any database call is made.
2. **Add `@Cacheable` to `EntityServiceImpl.getByEvmAddressAndType()`.** Even a short TTL (e.g., 30 seconds) would dramatically reduce database load under repeated lookups of the same address.
3. **Enable the Traefik middleware by default** or document it as a required security configuration, and increase the `inFlightReq` amount to a value that reflects realistic concurrency while still bounding abuse.
4. **Reduce the `GraphQLHighDBConnections` alert threshold** (e.g., to 50%) and shorten the `for` duration so automated responses trigger before the pool is fully exhausted.
5. **Enable the GCP gateway `maxRatePerEndpoint`** as a default-on defense-in-depth measure for GCP deployments.

## Proof of Concept

```bash
# Flood /graphql/alpha with unique EVM addresses (no auth required)
for i in $(seq 1 10000); do
  ADDR=$(printf "0x%040x" $i)
  curl -s -X POST https://<host>/graphql/alpha \
    -H "Content-Type: application/json" \
    -d "{\"query\":\"{ account(input:{evmAddress:\\\"$ADDR\\\"}) { balance } }\"}" &
done
wait
```

Each request reaches `AccountController.account()` → `EntityServiceImpl.getByEvmAddressAndType()` → `EntityRepository.findByEvmAddress()` (native SQL). With unique addresses, no cache can absorb the load. Once the HikariCP pool is exhausted, all subsequent GraphQL requests — from attackers and legitimate users alike — stall waiting for a connection.

        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
