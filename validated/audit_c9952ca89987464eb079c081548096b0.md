# Audit Report

## Title
Rate-Limit Token Leaked on Gas-Bucket Rejection Enables Sustained DoS Against Legitimate Users

## Summary
`ThrottleManagerImpl.throttle()` consumes one `rateLimitBucket` token before checking the `gasLimitBucket`. When the gas check fails, a `ThrottleException` is thrown but the already-consumed rate-limit token is never restored. Because `ContractController.call()` only catches `InvalidParametersException` (not `ThrottleException`) to invoke `restore()`, and `restore()` itself only refills the gas bucket, an attacker can continuously drain the shared `rateLimitBucket` without any EVM execution occurring, starving legitimate callers of their request quota.

## Finding Description

**`ThrottleManagerImpl.throttle()`** — `web3/src/main/java/org/hiero/mirror/web3/throttle/ThrottleManagerImpl.java`, lines 37–42:

```java
if (!rateLimitBucket.tryConsume(1)) {           // token consumed here
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(          // may fail; token above is NOT returned
        throttleProperties.scaleGas(request.getGas()))) {
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

When `rateLimitBucket.tryConsume(1)` succeeds but `gasLimitBucket.tryConsume(...)` fails, a `ThrottleException` is thrown and the rate-limit token is permanently lost.

**`restore()`** — lines 59–64 — only refills `gasLimitBucket`; `rateLimitBucket` is never touched:

```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);   // rateLimitBucket untouched
    }
}
```

**`ContractController.call()`** — lines 39–50 — only catches `InvalidParametersException`; `ThrottleException` propagates without ever calling `restore()`:

```java
} catch (InvalidParametersException e) {
    throttleManager.restore(request.getGas());   // never reached for ThrottleException
    throw e;
}
```

**Root cause:** The two bucket checks are not atomic and there is no compensating restore for the rate-limit token when the second (gas) check fails.

## Impact Explanation
The `rateLimitBucket` is a single shared application-wide bean. An attacker who holds the gas bucket at zero can consume the entire rate-limit quota (e.g., 500 tokens/s) with zero EVM processing cost, completely blocking all other callers. The service becomes unavailable for legitimate users for as long as the attack is sustained.

## Likelihood Explanation
Any anonymous HTTP client can reach `POST /api/v1/contracts/call`. The attack requires only a sustained stream of requests with `gas` set to the maximum allowed value. Once the gas bucket is pinned at zero, every subsequent request costs the attacker nothing (no EVM work) while consuming legitimate users' rate-limit quota. No authentication or special privilege is required.

## Recommendation
Restore the rate-limit token when the gas check fails. The simplest fix is to add a `rateLimitBucket.addTokens(1)` call before throwing the `GAS_PER_SECOND_LIMIT_EXCEEDED` exception:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);  // compensate for the consumed token
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, update `restore()` to also accept a flag or separate method to restore the rate-limit token, and call it from `ContractController.call()` in a broader catch block covering `ThrottleException` as well.

## Proof of Concept
1. Send 500 requests with `gas = maxGasLimit` to exhaust the `gasLimitBucket`.
2. Continue sending requests at the full 500 req/s rate. Each request: passes `rateLimitBucket.tryConsume(1)` (consuming 1 token), fails `gasLimitBucket.tryConsume(...)` (bucket empty), throws `ThrottleException`. The rate-limit token is gone; no gas token is consumed; no EVM work is done.
3. The shared `rateLimitBucket` is continuously drained. Legitimate users receive HTTP 429 for every request for as long as the attack is sustained.
