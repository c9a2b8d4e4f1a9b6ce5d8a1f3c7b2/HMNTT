Audit Report

## Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Dynamic Cache Creation in `cacheManagerSlotsPerContract` Leads to JVM Heap Exhaustion DoS

## Summary
`cacheManagerSlotsPerContract()` in `EvmConfiguration.java` constructs a `CaffeineCacheManager` without calling `setCacheNames()`, placing it in Spring's "dynamic" mode. In this mode, every call to `getCache(name)` for a previously-unseen name permanently inserts a new `CaffeineCache` into an internal `ConcurrentHashMap` with no size bound or eviction. Because `findStorageBatch()` in `ContractStateServiceImpl.java` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` for every distinct contract entity ID, an attacker who issues `eth_call` requests against a large number of distinct contract addresses causes this internal map to grow without bound, eventually exhausting JVM heap memory.

## Finding Description

**Exact code references:**

`EvmConfiguration.java` lines 107–112 — the bean is created with no `setCacheNames()`: [1](#0-0) 

Every other `CaffeineCacheManager` bean in the same file calls `setCacheNames()` to lock the manager into static mode (e.g., lines 67–73, 75–81, 83–89), but `cacheManagerSlotsPerContract` does not: [2](#0-1) 

`ContractStateServiceImpl.java` lines 85–87 — every cache miss on the outer `contractSlotsCache` triggers a call into the dynamic manager: [3](#0-2) 

**Root cause:**

Spring's `CaffeineCacheManager.getCache(name)` in dynamic mode stores every newly created `CaffeineCache` in an internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no size limit, no TTL, and no eviction mechanism. Once an entry is inserted it is never removed.

The outer `contractSlotsCache` (bean `CACHE_MANAGER_CONTRACT_SLOTS`) is configured with `maximumSize=3000`: [4](#0-3) 

When the outer cache evicts a `contractId` entry, the corresponding `CaffeineCache` object is removed from the outer cache's value, but **remains permanently in `cacheManagerSlotsPerContract.cacheMap`**. For every new distinct `contractId`, a fresh `CaffeineCache` is inserted and never freed.

The `slotsPerContract` cache spec is: [5](#0-4) 

Each retained `CaffeineCache` wraps a Caffeine `BoundedLocalCache` configured with `maximumSize=1500, expireAfterAccess=5m`, which allocates internal data structures (write-buffer queue, striped read buffer, segment arrays).

The vulnerable path is active by default: [6](#0-5) 

**Why existing checks fail:**

- The outer `contractSlotsCache` `maximumSize=3000` bounds only the outer cache, not `cacheManagerSlotsPerContract.cacheMap`.
- The gas-per-second bucket and the 500 req/s rate limiter slow the attacker but do not cap the *total* number of distinct contract IDs ever seen across the lifetime of the process.
- No authentication or IP-based per-client limit is enforced at the application layer.

## Impact Explanation
Sustained heap growth leads to `OutOfMemoryError`, crashing the JVM or causing severe GC pressure that renders the mirror-node web3 service completely unresponsive. Because the mirror node is the sole EVM query endpoint for applications relying on `eth_call`/`eth_estimateGas`, a successful attack takes down all smart-contract read operations. Recovery requires a process restart; the attack can be immediately resumed.

## Likelihood Explanation
The attack requires no credentials, no on-chain funds, and no special knowledge beyond knowing that the endpoint accepts `eth_call`. Any contract address that reads storage suffices. The attacker needs only a script that cycles through distinct `to` addresses at the permitted rate. The default 500 req/s throttle means a large number of distinct cache entries are achievable from a single client over time; multiple clients multiply this. The attack is fully repeatable after a restart.

## Recommendation
Call `setCacheNames()` on the `cacheManagerSlotsPerContract` bean to place it in static mode, preventing dynamic cache creation. However, since the design intentionally uses one `CaffeineCache` per contract ID, the correct fix is to **remove `CaffeineCacheManager` entirely** for this role and instead manage the per-contract `CaffeineCache` instances directly within `ContractStateServiceImpl`, using a `Caffeine`-backed `LoadingCache<EntityId, CaffeineCache>` with a bounded `maximumSize` matching the outer `contractSlotsCache` limit (3000). This ensures the number of live `CaffeineCache` instances is always bounded by the same eviction policy as the outer cache.

Alternatively, replace the two-level cache design with a single `Cache<(contractId, slotKey), value>` with a combined `maximumSize` limit, eliminating the structural mismatch entirely.

## Proof of Concept

```
for i in $(seq 1 100000); do
  # Each address is a distinct contract entity ID
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d "{\"data\":\"0x\",\"to\":\"0x$(printf '%040x' $i)\",\"gas\":100000}" &
done
```

Each request for a new `to` address causes `findStorageBatch` to call `cacheManagerSlotsPerContract.getCache("<entityId>")`, inserting a new `CaffeineCache` into the unbounded internal map. After the outer `contractSlotsCache` exceeds 3000 entries and begins evicting, the internal map continues to grow. JVM heap exhaustion follows as the number of permanently retained `CaffeineCache` instances (each with `maximumSize=1500` internal structures) accumulates without bound. [1](#0-0) [3](#0-2)
