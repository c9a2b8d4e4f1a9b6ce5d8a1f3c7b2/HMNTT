# Audit Report

## Title
Unbounded Timestamp Query via Missing Default Enforcement in `optimizeTimestampFilters()` When `bindTimestampRange=false`

## Summary
When the `query.bindTimestampRange` config flag is `false` (the default), calling `GET /api/v1/contracts/results` with no timestamp parameters causes `optimizeTimestampFilters()` to return an empty filter set. The resulting SQL query executes against the `contract_result` table with no timestamp `WHERE` clause. The only mitigations are a `LIMIT` of at most 100 rows and a 20-second statement timeout, which are insufficient to prevent repeated, expensive full-table-scan queries from exhausting the connection pool.

## Finding Description
The code path in `rest/controllers/contractController.js` `optimizeTimestampFilters()` (lines 384â€“405) proceeds as follows when no timestamp filters are supplied:

1. `parseTimestampFilters(timestampFilters, false, true, true, false, false)` is called with an empty array and `validateRange=false`. It returns `{range: null, eqValues: [], neValues: []}`. [1](#0-0) 

2. `range?.isEmpty()` evaluates to `undefined` on `null` â€” the early-return guard is skipped. [2](#0-1) 

3. Because `eqValues.length === 0`, `bindTimestampRange(null, order)` is called. [3](#0-2) 

4. Inside `bindTimestampRange`, the first check is `if (!queryConfig.bindTimestampRange)`. With the default `false`, it immediately returns `{range}` â€” the `null` passed in, unchanged. [4](#0-3) 

5. Back in `optimizeTimestampFilters`, `optimizedRange` is `null`. Both `optimizedRange?.begin` and `optimizedRange?.end` are `undefined`, so no timestamp filters are pushed. The function returns `{filters: [], next: undefined}`. [7](#0-6) 

6. The caller's skip-guard at line 506 is conditioned on `timestampFilters.length !== 0`. Since the user supplied no timestamp filters, this is `false` and the query is **not** skipped. [6](#0-5) 

7. The SQL query is built and executed with no timestamp `WHERE` clause, performing a full sequential scan of `contract_result`.

**Root cause:** The code assumes either (a) `bindTimestampRange` will be enabled in production, or (b) users will always supply at least one timestamp filter. Neither assumption is enforced. The `validateRange=false` argument explicitly disables the `maxTimestampRange` guard, and the skip-guard at line 506 is conditioned on the user having provided filters in the first place.

## Impact Explanation
An unauthenticated user can repeatedly issue `GET /api/v1/contracts/results` with no query parameters. Each request triggers a full sequential scan of the `contract_result` table. On a production mirror node with millions of rows, this is a high-cost query. With a default connection pool of 10 and a statement timeout of 20 seconds, a small number of concurrent requests can saturate the pool and deny service to all other API consumers. Additionally, up to 100 contract result records are returned per request with no time-based access scoping, exposing the full historical dataset to any caller.

## Likelihood Explanation
No authentication or special privilege is required. The endpoint is publicly reachable. The exploit is a single HTTP GET with no parameters â€” trivially repeatable by any script. The default configuration (`bindTimestampRange=false`) means every standard deployment is affected unless an operator has explicitly opted in to the binding feature. [4](#0-3) 

## Recommendation
1. **Change the default** of `query.bindTimestampRange` to `true`, or enforce a mandatory timestamp range when no filters are provided in `optimizeTimestampFilters()`.
2. **Require at least one timestamp filter** for the no-`contractId` path (i.e., `GET /api/v1/contracts/results`), returning a `400 Bad Request` if none is supplied.
3. **Adjust the skip-guard** at line 506 to also fire when `optimizedTimestampFilters.length === 0` regardless of whether the user supplied filters, preventing unbounded queries from reaching the database. [8](#0-7)
