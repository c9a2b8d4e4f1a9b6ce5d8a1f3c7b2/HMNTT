# Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Bypasses DB Page Size Cap, Enabling Unbounded Query DoS

## Summary
An unprivileged gRPC client can supply `limit = Long.MAX_VALUE` in a `ConsensusTopicQuery`. An unchecked `(int)` narrowing cast in `PollingTopicListener.poll()` wraps the value to `-1`, which causes `hasLimit()` to return `false` in the repository layer, removing the `setMaxResults` guard entirely and allowing the database to return every matching row on every poll cycle.

## Finding Description

**Entry point — no upper-bound validation:**

`ConsensusController.toFilter()` copies the proto `uint64 limit` field directly into `TopicMessageFilter` with no ceiling check: [1](#0-0) 

`TopicMessageFilter.limit` carries only `@Min(0)` — there is no `@Max` constraint, so `Long.MAX_VALUE` passes bean validation: [2](#0-1) 

**Overflow site — `PollingTopicListener.poll()` lines 54–59:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())   // ← narrowing cast
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [3](#0-2) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getCount() = 0`:
- `Long.MAX_VALUE - 0 = 0x7FFFFFFFFFFFFFFF`
- `(int) 0x7FFFFFFFFFFFFFFF = 0xFFFFFFFF = -1` (lower 32 bits, two's-complement)
- `Math.min(-1, 5000) = -1`
- `newFilter` is built with `limit = -1`

The `@Min(0)` constraint on `TopicMessageFilter.limit` does **not** fire here because `TopicMessageFilter` is a plain Lombok value object, not a Spring bean. The `@Validated` annotation on the class only activates method-level validation when the class is managed by a Spring proxy; `toBuilder().build()` is a direct constructor call and is never intercepted. [4](#0-3) 

**Guard bypass — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51–53:**

```java
if (filter.hasLimit()) {          // hasLimit() = (-1 > 0) = false
    typedQuery.setMaxResults(...); // ← never reached
}
``` [5](#0-4) 

`hasLimit()` returns `false` for any non-positive limit: [6](#0-5) 

`setMaxResults` is never called, so the JPA query fetches **all** rows matching the topic/timestamp predicate. The `maxPageSize` default of 5000 is completely bypassed: [7](#0-6) 

**Note on listener type:** The default listener type is `REDIS`, not `POLL`: [8](#0-7) 

The overflow only triggers when `hiero.mirror.grpc.listener.type=POLL` (or `SHARED_POLL`). Deployments using the default `REDIS` type are not affected via this specific path. However, `POLL` mode is a supported and documented configuration option.

## Impact Explanation
Every poll cycle (default every 500 ms) issues an unbounded `SELECT *` against the `topic_message` table for the chosen topic: [9](#0-8) 

On a busy topic with millions of rows, this exhausts database memory, connection pool resources, and JVM heap on the mirror node. A single persistent gRPC subscription is sufficient to degrade or crash the service for all other subscribers.

## Likelihood Explanation
The gRPC endpoint is publicly reachable with no authentication. The `ConsensusTopicQuery` proto field `uint64 limit` accepts the full 64-bit range; setting it to `Long.MAX_VALUE` requires only a standard gRPC client. No special privileges, tokens, or knowledge of internal state are required. The attack is trivially repeatable and persistent for the lifetime of the subscription. The vulnerability is only exploitable when the listener is configured to `POLL` or `SHARED_POLL` mode. [10](#0-9) 

## Recommendation

1. **Add an upper-bound cap in `ConsensusController.toFilter()`** — clamp `query.getLimit()` to `Integer.MAX_VALUE` (or a configurable maximum) before storing it in `TopicMessageFilter`.
2. **Fix the narrowing cast in `PollingTopicListener.poll()`** — use `Math.min(filter.getLimit() - context.getCount().get(), Integer.MAX_VALUE)` before casting to `int`, or keep the intermediate value as `long` and only cast after the `Math.min` with `maxPageSize`.
3. **Add a `@Max` constraint on `TopicMessageFilter.limit`** — add `@Max(Integer.MAX_VALUE)` alongside `@Min(0)` and ensure the filter is validated at the service boundary via `@Valid` on the `subscribeTopic` parameter.
4. **Defensive guard in `findByFilter()`** — treat any `limit <= 0` as "apply `maxPageSize`" rather than "no limit".

## Proof of Concept

```bash
# Using grpcurl against a mirror node running in POLL listener mode
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

With `limit = Long.MAX_VALUE = 9223372036854775807`:
- `ConsensusController.toFilter()` stores `Long.MAX_VALUE` in `TopicMessageFilter.limit` (passes `@Min(0)`)
- `PollingTopicListener.poll()`: `(int)(Long.MAX_VALUE - 0) = -1`; `Math.min(-1, 5000) = -1`
- `newFilter.limit = -1`; `hasLimit()` returns `false`
- `setMaxResults` is never called; every poll fetches all rows for the topic
- Poll repeats every 500 ms indefinitely, issuing unbounded DB queries until the subscription is cancelled
