Audit Report

## Title
Integer Overflow in `poll()` Limit Cast Causes Unbounded Database Query via `limit=Long.MAX_VALUE` Subscription

## Summary
In `PollingTopicMessageRetriever.poll()`, the cast `(int)(filter.getLimit() - context.getTotal().get())` silently overflows to `-1` when a subscriber sets `limit=Long.MAX_VALUE`. The resulting negative `pageSize` is passed to `topicMessageRepository.findByFilter()` via a rebuilt filter, where `hasLimit()` returns `false` for negative values, causing `setMaxResults()` to be skipped entirely. The JPA query executes with no `LIMIT` clause and loads all matching topic messages into heap memory. No authentication is required.

## Finding Description

**Entry point — `ConsensusController.toFilter()`:**

The proto `uint64 limit` field is mapped directly to Java `long` with no upper-bound check. [1](#0-0) 

The only validation constraint on `limit` is `@Min(0)`, which only rejects negative values. `Long.MAX_VALUE` passes this constraint. `hasLimit()` returns `true` since `Long.MAX_VALUE > 0`. [2](#0-1) 

**Overflow site — `PollingTopicMessageRetriever.poll()` lines 68–71:**

```
filter.getLimit()          = Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF
context.getTotal().get()   = 0
Long.MAX_VALUE - 0         = 0x7FFFFFFFFFFFFFFF
(int) 0x7FFFFFFFFFFFFFFF   = lower 32 bits = 0xFFFFFFFF = -1
Math.min(-1, maxPageSize)  = -1
``` [3](#0-2) 

**Rebuilt filter bypasses Bean Validation — line 75:**

`filter.toBuilder().limit(-1).build()` is a direct Lombok builder call. `TopicMessageServiceImpl` carries `@Validated` but `subscribeTopic()` has no `@Valid` on its parameter, so Bean Validation is never triggered for this internal filter rebuild. [4](#0-3) 

**Guard bypassed in repository — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51–53:**

`filter.hasLimit()` evaluates `limit > 0` → `-1 > 0` → `false`. `setMaxResults()` is never called. `getResultList()` then loads all matching rows into heap memory with no SQL `LIMIT` clause. [5](#0-4) 

**Outer `flux.take()` does not help — `TopicMessageServiceImpl.subscribeTopic()` lines 83–85:**

`filter.hasLimit()` returns `true` for the original filter (Long.MAX_VALUE > 0), so `flux.take(Long.MAX_VALUE)` is called — which is a no-op limit at the reactive layer. [6](#0-5) 

**Termination condition never fires — `isComplete()` line 122:**

`filter.getLimit() == total.get()` requires `total` to reach `Long.MAX_VALUE`, which is practically impossible. The subscription polls indefinitely. [7](#0-6) 

**Identical overflow in `PollingTopicListener.poll()` lines 54–57:** [8](#0-7) 

## Impact Explanation

Each poll cycle issues a `SELECT * FROM topic_message WHERE topic_id = ? AND consensus_timestamp >= ?` with no `LIMIT`, loading the entire message history for the targeted topic into JVM heap. For a high-traffic topic with millions of messages this causes:

- **OOM / heap exhaustion** on the mirror node gRPC process
- **Full index scan** on the database for every poll interval, saturating I/O
- **Indefinite subscription lifetime** — the connection is never terminated by the limit logic, compounding resource consumption across concurrent connections

This is a server-side Denial of Service reachable without any authentication or special role. This is distinct from a volumetric DDoS (which is out of scope per SECURITY.md); this is a single-request resource exhaustion triggered by a code logic bug.

## Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly exposed. The attacker needs only a valid topic ID (trivially discoverable from the public ledger) and a standard gRPC client. Setting `limit = Long.MAX_VALUE` (or any `uint64` value whose lower 32 bits are negative, e.g., `2147483648 = 0x80000000` → `(int) = Integer.MIN_VALUE`) in the `ConsensusTopicQuery` proto message is a single-field change. The attack is repeatable, requires no credentials, and can be parallelized across many connections to amplify impact.

## Recommendation

1. **Clamp the limit at the entry point** in `ConsensusController.toFilter()`: cap `query.getLimit()` to `Integer.MAX_VALUE` (or a configurable maximum) before storing it in `TopicMessageFilter`.
2. **Use safe narrowing in `poll()`**: replace `(int)(filter.getLimit() - context.getTotal().get())` with `(int) Math.min(filter.getLimit() - context.getTotal().get(), Integer.MAX_VALUE)` to prevent overflow.
3. **Add an upper-bound constraint** on `TopicMessageFilter.limit`, e.g., `@Max(Integer.MAX_VALUE)`, so the field-level validation rejects oversized values.
4. Apply the same fix to `PollingTopicListener.poll()` which contains the identical overflow pattern.

## Proof of Concept

```java
// Java gRPC client
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1234).build())
    .setLimit(Long.MAX_VALUE)   // 9223372036854775807 — valid uint64
    .build();

ConsensusServiceGrpc.ConsensusServiceBlockingStub stub = ...;
stub.subscribeTopic(query).forEachRemaining(response -> {
    // server executes unbounded SELECT on every poll cycle
});
```

With `limit = Long.MAX_VALUE`:
- `(int)(Long.MAX_VALUE - 0)` = `-1` (overflow)
- `Math.min(-1, maxPageSize)` = `-1`
- `newFilter.hasLimit()` = `false` (`-1 > 0` is false)
- `setMaxResults()` skipped → full table scan per poll cycle
