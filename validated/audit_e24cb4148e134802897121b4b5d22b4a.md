# Audit Report

## Title
Unauthenticated Full-Table Scan via Unindexed `e.balance` Filter Enables DB Connection Pool Exhaustion (DoS)

## Summary
The `GET /api/v1/accounts?account.balance=<op>:<value>` endpoint accepts an unauthenticated balance filter that is injected as a WHERE predicate on `entity.balance` — a column with no database index. Combined with a default connection pool of 10 connections and a 20-second statement timeout, an attacker can saturate all pool connections with concurrent requests that force full sequential scans of the `entity` table.

## Finding Description

**Confirmed code path:**

`getAccounts()` unconditionally parses `account.balance` and maps it to `e.balance`: [1](#0-0) 

This `balanceQuery` is passed as the `entityBalanceQuery` argument to `getAccountQuery()` with `accountBalanceQuery = undefined` (defaults to `{query:'', params:[]}`): [2](#0-1) 

Inside `getEntityBalanceQuery()`, because `accountBalanceQuery.query` is empty, `needHistoricalBalanceInfo` is `false`, so `entityTable = 'entity'` and `entityBalanceQuery.query` is injected directly into the WHERE clause: [3](#0-2) [4](#0-3) 

**No index on `entity.balance`:**

The `balance` column was added to the `entity` table by migration `V1.65.0__entity_balance.sql` with no accompanying index: [5](#0-4) 

A search across all `*.sql` migration files returns zero hits for any index on `entity.balance`. No subsequent migration creates such an index.

**Connection pool defaults:**

The default pool is 10 connections with a 20-second statement timeout and 20-second connection timeout: [6](#0-5) 

The pool is configured in `dbpool.js` directly from these values: [7](#0-6) 

**No rate limiting on the REST API:**

The `server.js` middleware stack registers `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, `responseHandler`, and `handleError` — no HTTP-level rate limiting middleware: [8](#0-7) 

The throttling infrastructure (`ThrottleManagerImpl`, `ThrottleConfiguration`, `RequestFilter`) belongs exclusively to the `web3` module and is not present in the REST API: [9](#0-8) 

The Helm chart's `rateLimit`/`inFlightReq` Traefik middleware is only configured for the `hedera-mirror-rosetta` and `hedera-mirror-graphql` charts, not for the main REST API chart.

The endpoint is publicly documented in the OpenAPI spec: [10](#0-9) [11](#0-10) 

## Impact Explanation

With 10 pool connections and a 20-second statement timeout, an attacker maintaining 10 concurrent requests ties up every available connection for up to 20 seconds per wave. All other REST API consumers (legitimate users, monitoring, health checks) receive connection-timeout errors (`connectionTimeoutMillis = 20000`). The attack is repeatable indefinitely with no authentication, no API key, and no account registration. On a production mainnet mirror node with millions of entity rows, each sequential scan is CPU- and I/O-intensive, compounding the impact on the database host.

Note: The production Helm chart configures pgbouncer with `max_user_connections: 250` for `mirror_rest`, which raises the effective ceiling. However, the Node.js application pool itself is still bounded at 10 connections by default, and the statement timeout still allows each query to hold a connection for up to 20 seconds. [12](#0-11) 

## Likelihood Explanation

The preconditions are zero: no account, no token, no special header. A single `curl` loop or a trivial script sending 10–20 concurrent HTTP GET requests is sufficient. The endpoint is publicly documented. The attack is repeatable, stateless, and requires no knowledge of internal state.

## Recommendation

1. **Add a database index on `entity.balance`** — a B-tree index on `(balance)` or a composite index on `(type, balance, id)` would allow the planner to satisfy both the filter and the `ORDER BY id LIMIT 25` efficiently.
2. **Add HTTP-level rate limiting middleware** to the Node.js REST API (e.g., `express-rate-limit`) to cap concurrent or per-IP request rates, consistent with the throttling already present in the `web3` module.
3. **Add Traefik `inFlightReq` and `rateLimit` middleware** to the REST API Helm chart, mirroring the configuration already present for the Rosetta API.
4. **Consider increasing `maxConnections`** or adding a query-complexity guard that rejects or caps requests that would produce unindexed full-table scans.

## Proof of Concept

```bash
# Saturate the pool with 10 concurrent requests using a selective balance filter
for i in $(seq 1 10); do
  curl -s "https://<mirror-node>/api/v1/accounts?account.balance=eq:999999999" &
done
wait
# All 10 pool connections are held for up to 20 seconds.
# Any subsequent request during this window receives a connection timeout error.
```

final class ThrottleManagerImpl implements ThrottleManager {

    static final String REQUEST_PER_SECOND_LIMIT_EXCEEDED = "Requests per second rate limit exceeded";
    static final String GAS_PER_SECOND_LIMIT_EXCEEDED = "Gas per second rate limit exceeded.";

    @Qualifier(GAS_LIMIT_BUCKET)
    private final Bucket gasLimitBucket;

    @Qualifier(RATE_LIMIT_BUCKET)
    private final Bucket rateLimitBucket;

    @Qualifier(OPCODE_RATE_LIMIT_BUCKET)
    private final Bucket opcodeRateLimitBucket;

    private final ThrottleProperties throttleProperties;

    @Override
    public void throttle(ContractCallRequest request) {
        if (!rateLimitBucket.tryConsume(1)) {
            throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
        } else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
            throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
        }

        for (var requestFilter : throttleProperties.getRequest()) {
            if (requestFilter.test(request)) {
                action(requestFilter, request);
            }
        }
    }
