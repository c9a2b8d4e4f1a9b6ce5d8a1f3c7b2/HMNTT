## Audit Report

## Title
Soft-Deleted Entity Disclosure via Zero-Prefix EVM Address in GraphQL `getByEvmAddressAndType()`

## Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, a zero-prefix EVM address (Hedera long-form numeric encoding) routes to `entityRepository.findById()`, which carries no `deleted` filter. The alternate branch calls `findByEvmAddress()`, which explicitly enforces `deleted is not true` in SQL. An unauthenticated caller can craft a zero-prefix EVM address encoding any deleted entity's numeric ID and retrieve that entity's full data through the public GraphQL API.

## Finding Description

**Vulnerable method** — `EntityServiceImpl.getByEvmAddressAndType()`:

```java
// graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {
        // findById() — no deleted filter
        return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
    }
    // findByEvmAddress() — enforces deleted is not true
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
``` [6](#0-5) 

**Repository layer:**

- `findByEvmAddress()` issues `SELECT * FROM entity WHERE evm_address = ?1 AND deleted is not true`. [2](#0-1) 
- `findById()` is inherited from Spring Data's `CrudRepository<Entity, Long>` and issues a plain `SELECT * FROM entity WHERE id = ?` — no `deleted` predicate. [3](#0-2) 

The only post-retrieval guard in the zero-prefix branch is `.filter(e -> e.getType() == type)`. There is no `.filter(e -> !Boolean.TRUE.equals(e.getDeleted()))` anywhere in that branch. The `deleted` field on `Entity` can be `true` (explicitly deleted) or `null` (treated as deleted in the SQL guards), and neither value is rejected. [7](#0-6) 

**Note:** The same missing guard exists in `getByIdAndType()` (line 25), which also calls `findById()` directly. The zero-prefix EVM address path is the more exploitable vector because it is reachable via the `evmAddress` input field, which callers may not associate with an ID-based lookup. [8](#0-7) 

## Impact Explanation
Any soft-deleted entity (account, contract, token, etc.) whose numeric ID is known or guessable can be retrieved in full through the public GraphQL endpoint. The `deleted is not true` guards on `findByAlias()` and `findByEvmAddress()` demonstrate the system's clear intent to suppress deleted entities from API responses. The zero-prefix EVM address path silently bypasses this intent, exposing the full entity record including all fields defined in the GraphQL schema. [9](#0-8) 

## Likelihood Explanation
- No authentication is required; the GraphQL endpoint is public.
- Hedera entity IDs are sequential integers, making enumeration trivial.
- The zero-prefix EVM address encoding (`0x000000000000000000000000<8-byte-id>`) is documented Hedera behavior, so any informed user knows the format.
- The attack is fully scriptable and repeatable with a single GraphQL query.

## Recommendation
Add a `deleted` guard in the zero-prefix branch of `getByEvmAddressAndType()`, consistent with the existing guards on `findByAlias()` and `findByEvmAddress()`:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
        .filter(e -> !Boolean.TRUE.equals(e.getDeleted()))
        .filter(e -> e.getType() == type);
}
```

Apply the same fix to `getByIdAndType()` for consistency. Alternatively, add a custom `findByIdAndDeletedIsNotTrue(Long id)` query method to `EntityRepository` (mirroring the pattern already used in the `web3` module's `EntityRepository.findByIdAndDeletedIsFalse()`) and use it in both service methods. [1](#0-0) 

## Proof of Concept
Given a deleted account with numeric ID `0xABCD` (43981 decimal):

1. Encode as zero-prefix EVM address: `0x000000000000000000000000000000000000ABCD`
2. Submit the following GraphQL query to the public endpoint:
   ```graphql
   query {
     account(input: { evmAddress: "0x000000000000000000000000000000000000ABCD" }) {
       deleted
       entityId { shard realm num }
       memo
       key
     }
   }
   ```
3. `buffer.getInt() == 0 && buffer.getLong() == 0` evaluates to `true` → `findById(0xABCD)` is called.
4. The deleted entity is returned with `deleted: true` in the response body, bypassing the `deleted is not true` guard that would have blocked the `findByEvmAddress()` path. [1](#0-0) [4](#0-3)
