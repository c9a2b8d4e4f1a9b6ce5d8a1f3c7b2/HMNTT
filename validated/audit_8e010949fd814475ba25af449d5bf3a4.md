# Audit Report

## Title
Rate-Limit Token Consumed Without Refund When Gas Check Fails, Enabling Unauthenticated DoS

## Summary
In `ThrottleManagerImpl.throttle()`, a rate-limit token is irrevocably consumed from `rateLimitBucket` before the gas check against `gasLimitBucket` is performed. When the gas check fails, the consumed rate-limit token is never returned. An unauthenticated attacker can exploit this logical flaw to exhaust the entire `rateLimitBucket` using requests that are ultimately rejected by the gas check, starving all legitimate users for the remainder of each 1-second refill window.

## Finding Description

**Exact code path — `ThrottleManagerImpl.java`, `throttle()`, lines 38–42:**

```java
if (!rateLimitBucket.tryConsume(1)) {                                          // line 38 – token consumed
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {  // line 40
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);                // line 41 – token NOT refunded
}
``` [1](#0-0) 

The two-bucket check is non-atomic with no rollback. A rate-limit token is permanently consumed even when the request is ultimately rejected by the gas check.

**`restore()` — lines 59–64:**

```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);   // only gasLimitBucket is refunded
    }
}
``` [2](#0-1) 

There is no `rateLimitBucket.addTokens(1)` call anywhere in the codebase.

**Secondary asymmetry — `ThrottleConfiguration.java`:**
- `rateLimitBucket` (line 31): built without `SynchronizationStrategy` → default `LOCK_FREE` (CAS-based, thread-safe but not the core issue).
- `gasLimitBucket` (line 42): built with `SynchronizationStrategy.SYNCHRONIZED`. [3](#0-2) 

**Root cause:** The vulnerability is purely logical, not a race condition. Even sequential requests can drain the rate-limit bucket through gas-failed requests.

## Impact Explanation
An unauthenticated attacker can completely block all legitimate web3 API traffic for the full duration of each 1-second refill window, repeatable indefinitely. With default settings (`requestsPerSecond = 500`, `gasPerSecond = 7_500_000_000`), the attacker needs only 500 HTTP requests per second — trivially achievable from a single machine. All subsequent legitimate requests within that second are rejected with `REQUEST_PER_SECOND_LIMIT_EXCEEDED`. Both buckets refill after 1 second (`refillGreedy(capacity, Duration.ofSeconds(1))`), so the attack is repeatable every second. [6](#0-5) 

## Likelihood Explanation
No authentication, API key, or special privilege is required. The attacker only needs network access to the web3 endpoint. The attack requires no state, is fully repeatable every second, and can be automated with a simple HTTP client loop sending requests with `gas >= gasPerSecond` (to deplete the gas bucket on the first request) followed by requests with any `gas > 10,000` (to consume rate-limit tokens via gas-failed rejections).

## Recommendation
Refund the rate-limit token when the gas check fails. The fix should be applied atomically in `throttle()`:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
} else if (!gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))) {
    rateLimitBucket.addTokens(1);  // refund the consumed token
    throw new ThrottleException(GAS_PER_SECOND_LIMIT_EXCEEDED);
}
```

Alternatively, restructure the check to verify gas availability before consuming the rate-limit token, or use a single atomic check if Bucket4j supports multi-bucket transactions.

Additionally, `restore()` should be reviewed to determine whether a rate-limit token refund is also appropriate in the gas-restoration path. [7](#0-6) 

## Proof of Concept

```
# Step 1: Deplete gasLimitBucket with one request
POST /api/v1/contracts/call
{ "gas": 7500000000 }
# → scaleGas(7_500_000_000) = 750_000 tokens consumed from gasLimitBucket (full capacity)
# → 1 token consumed from rateLimitBucket

# Steps 2–500: Send 499 more requests (gas > 10_000 so scaleGas > 0)
for i in {1..499}; do
  POST /api/v1/contracts/call { "gas": 100000 }
  # → rateLimitBucket.tryConsume(1) succeeds → 1 token consumed (irrevocably)
  # → gasLimitBucket.tryConsume(...) fails (bucket empty) → ThrottleException thrown
done

# Result: rateLimitBucket is now at 0 tokens
# All subsequent legitimate requests within this second receive:
# HTTP 429: "Requests per second rate limit exceeded"

# Repeat every second indefinitely.
``` [8](#0-7)

    @Bean(name = RATE_LIMIT_BUCKET)
    Bucket rateLimitBucket() {
        long rateLimit = throttleProperties.getRequestsPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(rateLimit)
                .refillGreedy(rateLimit, Duration.ofSeconds(1))
                .build();

    @Bean(name = GAS_LIMIT_BUCKET)

        long gasLimit = throttleProperties.getGasPerSecond();
        final var limit = Bandwidth.builder()
                .capacity(gasLimit)
                .refillGreedy(gasLimit, Duration.ofSeconds(1))
                .build();
        return Bucket.builder()

                .addLimit(limit)
                .build();
    }
