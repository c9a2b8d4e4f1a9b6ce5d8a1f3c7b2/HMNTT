Audit Report

## Title
`merge()` in `requestQueryParser` Bypasses `qs` `arrayLimit` via Mixed-Case Key Variants

## Summary
`requestQueryParser` in `rest/middleware/requestHandler.js` relies on `qs`'s per-key `arrayLimit` + `throwOnLimitExceeded: true` as its sole array-size guard. Because `qs` is case-sensitive, mixed-case variants of the same parameter name (e.g. `transaction.hash`, `transaction.Hash`, `transaction.HASH`) are treated as independent keys, each allowed up to `arrayLimit` values. The subsequent lowercasing loop merges all variants through `merge()`, which contains no secondary length check, producing a merged array that is a multiple of `arrayLimit` in size.

## Finding Description
**Exact code path:**

1. `queryOptions` configures `qs` with `arrayLimit` and `throwOnLimitExceeded: true` (`requestHandler.js` lines 15–20). [1](#0-0) 
2. `qs.parse(queryString, queryOptions)` at line 54 enforces the limit per-key. [2](#0-1) 
3. The loop at lines 57–66 lowercases every key; on collision it calls `merge()`. [4](#0-3) 
4. `merge()` at lines 39–51 unconditionally appends elements with no length check. [3](#0-2) 

**Root cause:** The `throwOnLimitExceeded` guard fires only when a *single qs-level key* accumulates more than `arrayLimit` values. Mixed-case variants are independent qs keys, each passing the guard individually. After `qs.parse()` returns, `merge()` combines them with no secondary size enforcement.

**Test-suite confirmation:** The test at lines 21–39 of `requestHandler.test.js` explicitly expects and validates this merging behaviour across case variants. [5](#0-4) 

## Impact Explanation
1. **Array materialisation before secondary check.** The secondary guard in `buildFilters()` (`rest/utils.js` lines 1241–1248) does exist, but the oversized array is fully materialised in the Node.js heap before it runs. [6](#0-5) 
2. **Single-value constraint bypass.** Any handler or middleware that reads `req.query['transaction.hash']` directly — before `buildFilters()` is invoked — receives the already-inflated array. Sending `transaction.hash=v1&transaction.Hash=v2` produces a 2-element array that bypasses validators expecting a scalar.
3. **Handlers that skip `buildFilters()` entirely** receive the inflated array with no guard at all.

## Likelihood Explanation
- No authentication is required; any HTTP client can craft the query string.
- Mixed-case variants require no special tooling.
- The condition is deterministic and reproducible on every request.
- Practical upper bound on array size is constrained by the HTTP query-string length limit (e.g. nginx default 8 KB), limiting but not eliminating the risk.

## Recommendation
Add a length check inside `merge()` (or immediately after it is called) that throws the same `throwOnLimitExceeded` error when the merged array exceeds `config.query.maxRepeatedQueryParameters`. For example:

```js
const merge = (current, next) => {
  if (!Array.isArray(current)) current = [current];
  if (Array.isArray(next)) current.push(...next);
  else current.push(next);

  // NEW: enforce the same limit post-merge
  if (current.length > queryOptions.arrayLimit) {
    throw new RangeError(`Array limit of ${queryOptions.arrayLimit} exceeded`);
  }
  return current;
};
```

This ensures the `arrayLimit` is respected regardless of how many case variants are used.

## Proof of Concept
```
GET /api/v1/transactions?transaction.hash=h1&transaction.hash=h2&...&transaction.hash=h8
  &transaction.Hash=h9&transaction.Hash=h10&...&transaction.Hash=h16
  &transaction.HASH=h17&transaction.HASH=h18&...&transaction.HASH=h24
```
`qs.parse()` returns three independent keys, each with `arrayLimit` values — no exception thrown. The lowercasing loop merges all three into `transaction.hash` with `3 × arrayLimit` elements. The test suite at `requestHandler.test.js` lines 31–39 demonstrates the identical merge pattern with `account.id`/`account.Id`/`account.ID`. [7](#0-6)
