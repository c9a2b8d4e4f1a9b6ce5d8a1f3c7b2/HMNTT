# Audit Report

## Title
Cache Pollution via Unbounded Slot Key Injection Causes DB Query Amplification and LRU Eviction of Legitimate Keys

## Summary
Any unauthenticated caller can inject up to 1500 arbitrary, non-existent slot keys into the per-contract `slotsPerContract` Caffeine cache by issuing 1500 `findStorage()` calls. Once saturated, every subsequent `findStorage()` call for the targeted contract — from any user — triggers a `findStorageBatch` DB query with a 1500-element `IN` clause, amplifying DB load by up to 1500×. Continuous injection of new unique keys also causes LRU eviction of legitimate slot keys, forcing additional individual DB queries via the fallback path.

## Finding Description

**File:** `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, method `findStorageBatch()`, lines 85–122.

**Root cause:** At line 90, any caller-supplied `key` is unconditionally inserted into the per-contract slot cache with no validation that the key corresponds to an existing DB slot and no per-caller rate limit on distinct key insertions:

```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);   // line 90
``` [7](#0-6) 

The cache is configured with `maximumSize=1500`:

```
slotsPerContract = "expireAfterAccess=5m,maximumSize=1500"
``` [4](#0-3) 

Immediately after insertion, line 91 snapshots **all** keys currently in the cache:

```java
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();  // line 91
``` [8](#0-7) 

Line 103 then issues a single batch DB query for **every** key in that snapshot:

```java
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
``` [2](#0-1) 

Non-existent slots are never stored in `contractStateCache` (line 109 only caches values returned by the DB), so the short-circuit check at line 63–67 never fires for junk keys, ensuring every call reaches `findStorageBatch`. [5](#0-4) 

**Exploit flow:**

1. **Phase 1 — Cache saturation:** Attacker sends 1500 `eth_call` requests targeting a specific `contractId`, each with a distinct non-existent 32-byte slot key. Each call inserts a junk key via `putIfAbsent` until the per-contract cache reaches its 1500-entry maximum.

2. **Phase 2 — Sustained amplification:** Each subsequent attacker call inserts a new junk key (evicting the LRU entry), snapshots all 1500 cached keys, and triggers `findStorageBatch` with a 1500-element `IN` clause. Result: **1 API call → 1500-slot DB query** (1500× amplification). After initial saturation, only ~1 RPS is needed to maintain eviction pressure given the 5-minute `expireAfterAccess` TTL.

3. **Phase 3 — Legitimate key eviction / fallback:** When a legitimate slot key is evicted by the attacker's continuous injection and a legitimate user concurrently calls `findStorage()` for that key, the loop at lines 96–101 does not find `wrappedKey` in `cachedSlotKeys`, leaving `isKeyEvictedFromCache = true`. The code falls back to an individual DB query at lines 118–119 on top of the already-inflated batch query:

```java
if (isKeyEvictedFromCache) {
    return contractStateRepository.findStorage(contractId.getId(), key);
}
``` [3](#0-2) 

**Why existing checks are insufficient:**
- `putIfAbsent` prevents duplicate keys per call but does not prevent a single caller from inserting 1500 distinct junk keys across 1500 calls.
- The `isKeyEvictedFromCache` fallback preserves correctness but does not prevent amplification or eviction.
- Global RPS throttling limits total throughput but does not bound the number of distinct slot keys a single caller can inject.

## Impact Explanation

**DB query amplification (primary):** Every `findStorage()` call for the targeted contract — from any user — triggers a `WHERE slot IN (...)` query with up to 1500 parameters. This can saturate DB connection pools and degrade or deny service for all contracts sharing the same DB instance.

**Legitimate key eviction (secondary):** In concurrent scenarios, legitimate slot keys are evicted from the per-contract cache, forcing additional individual DB queries per evicted key. Correctness is preserved but latency increases and DB load compounds.

**Severity:** Medium — no funds are at risk and no on-chain state is corrupted, but the attack causes measurable service degradation (increased DB latency, potential query timeouts) for all users of the mirror node web3 service.

## Likelihood Explanation

The attack requires no credentials, no tokens, and no special network position. Any user with access to the public JSON-RPC endpoint can issue `eth_call` requests with arbitrary `data` fields that cause arbitrary slot keys to be read. The attacker needs only ~1500 initial requests to saturate the cache and then ~1 request every 5 minutes per junk key to maintain eviction pressure. This is trivially automatable with a simple script.

## Recommendation

1. **Validate slot keys before caching:** Before calling `putIfAbsent` at line 90, check whether the slot key exists in the DB (or in `contractStateCache`). Only insert keys that correspond to known-existing slots.

2. **Per-caller slot key injection limit:** Introduce a per-IP or per-caller rate limit on the number of distinct slot keys that can be inserted into the per-contract cache within a time window.

3. **Cap batch query size independently of cache size:** Decouple the batch query size from the full cache snapshot. Instead of querying all cached keys on every call, limit the batch to a configurable maximum (e.g., 50–100 slots) and prioritize recently accessed keys.

4. **Separate junk-key TTL from legitimate-key TTL:** Consider a shorter TTL for keys whose DB lookup returned no value, preventing long-lived pollution of the cache.

## Proof of Concept

```python
import requests, os

TARGET_RPC = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT_ADDRESS = "0x<target_contract>"

# Phase 1: Saturate the slotsPerContract cache with 1500 unique junk keys
for i in range(1500):
    slot_key = hex(i).zfill(66)  # unique 32-byte slot key
    requests.post(TARGET_RPC, json={
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": CONTRACT_ADDRESS, "data": slot_key}, "latest"],
        "id": i
    })

# Phase 2: Maintain eviction pressure with 1 new unique key per request
# Each call now triggers findStorageBatch with 1500-slot IN clause
i = 1500
while True:
    slot_key = hex(i).zfill(66)
    requests.post(TARGET_RPC, json={
        "jsonrpc": "2.0", "method": "eth_call",
        "params": [{"to": CONTRACT_ADDRESS, "data": slot_key}, "latest"],
        "id": i
    })
    i += 1
    # Every legitimate user's findStorage call for this contract now
    # triggers a 1500-slot DB query instead of a targeted single-slot query
```
