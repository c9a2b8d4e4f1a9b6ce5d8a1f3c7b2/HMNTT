## Audit Report

## Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Unique ContractId Cycling — Heap-Exhaustion DoS

## Summary
In `ContractStateServiceImpl.findStorageBatch()`, each unique `contractId` triggers a call to `cacheManagerSlotsPerContract.getCache(contractId.toString())`. Because `cacheManagerSlotsPerContract` is configured in Spring's "dynamic" mode (no `setCacheNames()` call), every new name creates a `CaffeineCache` and inserts it permanently into `CaffeineCacheManager`'s internal unbounded `ConcurrentHashMap<String, Cache>`. The outer `contractSlotsCache` (bounded at `maximumSize=3000`) evicts entries but does **not** remove the corresponding `CaffeineCache` from `cacheManagerSlotsPerContract`'s internal map, causing permanent heap retention proportional to the total number of unique contract IDs ever seen.

## Finding Description

**Exact code location:**
`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`, lines 86–87:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [1](#0-0) 

**Root cause — two-layer cache design with a broken eviction assumption:**

1. `this.contractSlotsCache` is a Caffeine cache (`CACHE_MANAGER_CONTRACT_SLOTS`) configured with `expireAfterAccess=5m,maximumSize=3000`. It maps `EntityId contractId → CaffeineCache`. [2](#0-1) 

2. On a cache miss, the lambda calls `cacheManagerSlotsPerContract.getCache(contractId.toString())`. The `cacheManagerSlotsPerContract` bean is a `CaffeineCacheManager` configured **without** a `setCacheNames()` call, placing it in Spring's "dynamic" mode. [3](#0-2) 

3. In dynamic mode, Spring's `CaffeineCacheManager` maintains an internal `ConcurrentHashMap<String, Cache> cacheMap`. Every call to `getCache(name)` for a new name creates a `CaffeineCache` and inserts it into `cacheMap`. **This map is never bounded and never evicts entries.**

4. When the outer `contractSlotsCache` evicts an entry for `contractId_X` (due to `maximumSize=3000` or TTL), the `CaffeineCache` object for `contractId_X` is no longer referenced by the outer cache. However, `cacheManagerSlotsPerContract.cacheMap` still holds a strong reference to it under the key `contractId_X.toString()`. The `CaffeineCache` is never GC'd.

5. The `slotsPerContract` spec `expireAfterAccess=5m,maximumSize=1500` applies to entries *within* each per-contract `CaffeineCache` (slot keys), not to the number of caches held in `cacheManagerSlotsPerContract.cacheMap`. [4](#0-3) 

All other `CaffeineCacheManager` beans in `EvmConfiguration` correctly call `setCacheNames()` to prevent dynamic growth; `cacheManagerSlotsPerContract` is the sole exception. [5](#0-4) 

## Impact Explanation
Each retained `CaffeineCache` object carries internal Caffeine data structures (segment arrays, scheduler references, etc.) — even an empty cache has non-trivial overhead. Memory grows linearly and without bound with the number of unique contract IDs ever seen. Over a sustained attack or even normal long-running operation with many contracts, this causes progressive heap exhaustion, triggering `OutOfMemoryError` or sustained full-GC pauses that render the web3 service unresponsive. No contract state data needs to exist; the attacker only needs to reference distinct contract addresses that trigger a SLOAD opcode path reaching `findStorage`. [6](#0-5) 

## Likelihood Explanation
- **No authentication required**: the JSON-RPC endpoint (`eth_call`, `eth_estimateGas`) is public.
- **No special knowledge required**: Hedera contract IDs are sequential integers; iterating them is trivial.
- **Rate limiting is insufficient**: the 500 RPS throttle slows but does not stop accumulation; the leak is permanent and cumulative.
- **No on-chain cost**: `eth_call` is a read-only simulation; no gas or HBAR is spent by the attacker.
- **Also triggered by normal traffic**: in a production environment with many distinct contracts, the leak occurs organically without any attacker. [3](#0-2) 

## Recommendation
The `cacheManagerSlotsPerContract` bean should not be used in dynamic mode. The correct fix is to **not use `CaffeineCacheManager` as a registry of per-contract caches at all**. Instead, the outer `contractSlotsCache` (already bounded at `maximumSize=3000`) should store the `CaffeineCache` objects directly, and an eviction listener on the outer cache should explicitly call `cacheManagerSlotsPerContract` to remove the corresponding entry when an outer-cache entry is evicted. Alternatively, replace the two-layer design with a single Caffeine cache keyed by `(contractId, slotKey)` pairs, eliminating the need for `cacheManagerSlotsPerContract` entirely.

## Proof of Concept
```
1. Start the web3 service with default configuration.
2. Send N eth_call requests, each targeting a distinct contract address
   (e.g., Hedera IDs 0.0.1 through 0.0.N), where each call triggers a SLOAD
   (e.g., call a contract that reads its own storage slot 0).
3. After 3001 requests, the outer contractSlotsCache begins evicting entries.
4. Inspect cacheManagerSlotsPerContract via JMX/heap dump:
   - cacheMap.size() == N (never decreases)
   - Each entry is a live CaffeineCache object
5. Continue until JVM heap is exhausted or GC overhead limit is exceeded.
``` [7](#0-6)
