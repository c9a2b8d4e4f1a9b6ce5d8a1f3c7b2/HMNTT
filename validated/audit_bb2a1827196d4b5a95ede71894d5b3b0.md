Audit Report

## Title
Gas Throttle Amplification via `estimate=true` Binary Search — Up to 21× EVM Executions Per Single Rate-Limit Token

## Summary
`ContractController.call()` consumes throttle tokens exactly once per HTTP request. When `estimate=true`, `ContractExecutionService.estimateGas()` performs one initial EVM execution followed by up to 20 additional EVM executions inside `BinaryGasEstimator.search()`, none of which consume additional throttle tokens. This creates a deterministic 21× amplification of EVM CPU work per rate-limit token consumed.

## Finding Description
**Code path:**

1. `ContractController.call()` calls `throttleManager.throttle(request)` once at line 40, consuming 1 `rateLimitBucket` token and `scaleGas(request.getGas())` `gasLimitBucket` tokens. [1](#0-0) 

2. `ThrottleManagerImpl.throttle()` deducts tokens only at request entry via `rateLimitBucket.tryConsume(1)` and `gasLimitBucket.tryConsume(throttleProperties.scaleGas(request.getGas()))`. No further deductions occur during processing. [2](#0-1) 

3. `ContractExecutionService.estimateGas()` first calls `callContract(params, context)` (one full EVM execution), then passes `gas -> doProcessCall(params, gas, true)` to `binaryGasEstimator.search()`. [3](#0-2) 

4. `BinaryGasEstimator.search()` loops `while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount())`, calling `safeCall(mid, call)` (i.e., `doProcessCall`) on every iteration with no throttle interaction. [4](#0-3) 

5. `doProcessCall` with `estimate=true` explicitly skips `restoreGasToBucket` and all throttle interactions inside the `finally` block, confirming zero throttle accounting occurs during the binary search. [5](#0-4) 

**Root cause:** The throttle gate is a one-time check at the HTTP boundary. The `gasLimitBucket` is charged for `request.getGas()` once, but the binary search can execute up to `maxGasEstimateRetriesCount` (default: 20) full EVM simulations, each consuming CPU proportional to the gas value at that iteration. The failed assumption is that one HTTP request equals one EVM execution for throttle accounting purposes.

## Impact Explanation
With `maxGasEstimateRetriesCount = 20` and `gas = 15,000,000` (the default maximum), one HTTP request triggers up to 21 EVM executions (1 initial + 20 binary search) while consuming only 1 `rateLimitBucket` token and 1,500 `gasLimitBucket` tokens (`15,000,000 / 10,000`). [6](#0-5) [7](#0-6) 

The actual CPU cost is up to 21× what the throttle accounts for. At the default 500 req/s rate limit, an attacker can sustain approximately 10,500 EVM executions/second against the server's effective throttle of 500. This enables server-side resource exhaustion (CPU saturation, thread pool starvation) with a modest request rate from a single HTTP client. [8](#0-7) 

## Likelihood Explanation
No authentication or special privileges are required. Any external caller can set `estimate: true` and `gas: 15000000` in a standard JSON POST to `/api/v1/contracts/call`. The attack is trivially repeatable, requires no on-chain state, and is effective from a single HTTP client. The amplification factor is deterministic and bounded only by `maxGasEstimateRetriesCount`. [9](#0-8) 

## Recommendation
Charge throttle tokens proportionally to the actual number of EVM executions performed, not just the initial request. Two concrete approaches:

1. **Charge per binary search iteration:** Inside `BinaryGasEstimator.search()`, call `throttleManager.throttle(...)` or a dedicated `throttleManager.throttleEstimateIteration(gas)` on each iteration, consuming additional `gasLimitBucket` tokens proportional to the `mid` gas value used in that iteration.

2. **Charge a multiplier upfront:** Before entering `estimateGas()`, pre-consume `maxGasEstimateRetriesCount × scaleGas(request.getGas())` tokens from `gasLimitBucket`, and restore unused tokens after the search completes (similar to the existing `restoreGasToBucket` pattern already used for non-estimate calls). [10](#0-9) 

## Proof of Concept
```
POST /api/v1/contracts/call
Content-Type: application/json

{
  "to": "0x<any_deployed_contract>",
  "data": "0x<function_selector>",
  "gas": 15000000,
  "estimate": true
}
```

Send this request at the rate limit (500 req/s). Each request:
1. Passes `throttleManager.throttle()` consuming 1 rate token and 1,500 gas tokens.
2. Executes `callContract()` once (full EVM run at 15,000,000 gas).
3. Enters `BinaryGasEstimator.search()` and executes `doProcessCall()` up to 20 more times with no additional throttle cost.

Result: 500 HTTP requests/second → up to 10,500 EVM executions/second, while the throttle only accounts for 500. The server's CPU and thread pool are exhausted at a fraction of the intended throttle load. [11](#0-10)

    public long search(
            final ObjIntConsumer<Long> metricUpdater, final LongFunction<EvmTransactionResult> call, long lo, long hi) {
        long prevGasLimit = lo;
        int iterationsMade = 0;
        long totalGasUsed = 0;

        // Now that we also support gas estimates for precompile calls, the default threshold is too low, since
        // it does not take into account the minimum threshold of 5% higher estimate than the actual gas used.
        // The default value is working with some calls but that is not the case for precompile calls which have higher
        // gas consumption.
        // Configurable tolerance of 10% over 5% is used, since the algorithm fails when using 5%, producing too narrow
        // threshold. Adjust via estimateGasIterationThresholdPercent value.
        final long estimateIterationThreshold = (long) (lo * properties.getEstimateGasIterationThresholdPercent());

        ContractCallContext contractCallContext = ContractCallContext.get();
        while (lo + 1 < hi && iterationsMade < properties.getMaxGasEstimateRetriesCount()) {
            contractCallContext.reset();

            long mid = (hi + lo) / 2;

            // If modularizedServices is true - we call the safeCall function that handles if an exception is thrown
            final var transactionResult = safeCall(mid, call);

            iterationsMade++;

            boolean err =
                    transactionResult == null || !transactionResult.isSuccessful() || transactionResult.gasUsed() < 0;
            long gasUsed = err ? prevGasLimit : transactionResult.gasUsed();
            totalGasUsed += gasUsed;
            if (err || gasUsed == 0) {
                lo = mid;
            } else {
                hi = mid;
                if (Math.abs(prevGasLimit - mid) < estimateIterationThreshold) {
                    lo = hi;
                }
            }
            prevGasLimit = mid;
        }

        metricUpdater.accept(totalGasUsed, iterationsMade);
        return hi;
    }

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
