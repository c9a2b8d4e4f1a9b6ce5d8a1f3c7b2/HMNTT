# Audit Report

## Title
Contract Filter Bypass via Unresolved Recipient Entity Enables Trace Log Flooding

## Summary
In `MirrorOperationActionTracer.tracePostExecution()`, the contract allowlist filter is guarded by a short-circuit `&&` that requires `recipientNum.isPresent()`. When `commonEntityAccessor.get()` returns `Optional.empty()` for an unknown recipient address, the filter is never evaluated and `log.info()` fires unconditionally for every EVM opcode in the call frame. An unprivileged caller can exploit this to bypass the operator-configured contract filter and flood trace logs by targeting any address absent from the entity store.

## Finding Description
**File:** `web3/src/main/java/org/hiero/mirror/web3/evm/contracts/execution/traceability/MirrorOperationActionTracer.java`, lines 59–68 [1](#0-0) 

```java
final var recipientAddress = frame.getRecipientAddress();
final var recipientNum = recipientAddress != null
        ? commonEntityAccessor.get(recipientAddress, Optional.empty())
        : Optional.empty();

if (recipientNum.isPresent()
        && traceProperties.contractFilterCheck(
                CommonUtils.hex(toEvmAddress(((Entity) recipientNum.get()).getId())))) {
    return;
}
log.info(...);
```

**File:** `web3/src/main/java/org/hiero/mirror/web3/evm/properties/TraceProperties.java`, lines 33–35 [2](#0-1) 

```java
public boolean contractFilterCheck(String contract) {
    return !getContract().isEmpty() && !getContract().contains(contract);
}
```

**Root cause:** The guard `recipientNum.isPresent() && contractFilterCheck(...)` uses short-circuit evaluation. When `commonEntityAccessor.get()` returns `Optional.empty()` (address not in the entity store), `isPresent()` is `false`, the `&&` short-circuits, `contractFilterCheck` is never invoked, and execution falls through to `log.info()`. The failed assumption is that every recipient address will resolve to a known entity — but the EVM allows calls to arbitrary 20-byte addresses that may not exist in the mirror node's entity store.

**Existing checks are insufficient:**
- The `isEnabled()` check at line 51 only gates the entire feature; it does not protect against filter bypass when tracing is on.
- The `stateFilterCheck` at line 55 only filters by EVM frame state, not by contract address.
- The `contractFilterMismatch` test only covers the path where the entity **is** found (`Optional.of(recipientEntity)`). The `contractFilter` test inadvertently exercises the bypass: `commonEntityAccessor.get()` is not stubbed, so Mockito returns `Optional.empty()`, and the test asserts logging fires — but for the wrong reason (filter bypassed, not filter matched). [5](#0-4) 

## Impact Explanation
When tracing is enabled with a non-empty contract filter, an attacker can generate unbounded `INFO`-level log entries by repeatedly calling non-existent addresses via `eth_call`. Each call produces one log line per EVM opcode executed in the call frame. This can exhaust disk space, overwhelm log aggregation pipelines (e.g., Elasticsearch, Splunk), degrade observability for legitimate operators, and potentially cause the mirror node process to become unresponsive due to I/O saturation. The contract filter — the operator's primary tool for scoping trace verbosity — is rendered completely ineffective against calls to unresolvable addresses.

## Likelihood Explanation
The precondition is that an operator has enabled tracing (`enabled=false` by default) and configured a non-empty contract filter. Any deployment that enables tracing with a contract filter is immediately vulnerable. The attacker requires no credentials, no special role, and no on-chain funds, since `eth_call` on mirror nodes simulates calls without consuming gas. The non-existent address space is effectively infinite (2^160 addresses), making blocklisting impractical. The attack is trivially repeatable via scripted `eth_call` requests.

## Recommendation
The filter logic should treat an unresolvable recipient address as a non-match against the allowlist, not as a bypass. When `recipientNum` is empty and a non-empty contract filter is configured, the call should be **suppressed** (early return), not logged. Replace the current guard:

```java
// Current (buggy): empty Optional bypasses the filter
if (recipientNum.isPresent()
        && traceProperties.contractFilterCheck(...)) {
    return;
}
```

With logic that explicitly handles the empty case:

```java
// Fixed: if a contract filter is configured and the recipient is unknown, suppress logging
if (!traceProperties.getContract().isEmpty()) {
    if (recipientNum.isEmpty()) {
        return; // unknown address: not in allowlist, suppress
    }
    if (traceProperties.contractFilterCheck(
            CommonUtils.hex(toEvmAddress(((Entity) recipientNum.get()).getId())))) {
        return;
    }
}
```

Additionally, add a test case asserting that when a non-empty contract filter is configured and `commonEntityAccessor.get()` returns `Optional.empty()`, `log.info()` is **not** called. [7](#0-6) 

## Proof of Concept
1. Configure the mirror node with:
   ```
   hiero.mirror.web3.evm.trace.enabled=true
   hiero.mirror.web3.evm.trace.contract=0x000000000000000000000000000000000000abcd
   ```
2. Submit an `eth_call` targeting any address not present in the entity store, e.g., a randomly generated 20-byte address `0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef`.
3. `commonEntityAccessor.get(recipientAddress, Optional.empty())` returns `Optional.empty()`.
4. The condition `recipientNum.isPresent() && contractFilterCheck(...)` evaluates to `false`; no early return.
5. `log.info(...)` fires for **every EVM opcode** executed in the call frame, regardless of the configured allowlist.
6. Repeat in a tight loop via scripted `eth_call` requests (no gas cost on mirror nodes) to generate unbounded log volume. [8](#0-7)
