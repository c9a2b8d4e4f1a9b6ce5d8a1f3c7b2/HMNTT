Audit Report

## Title
DUPLICATE_TRANSACTION Timestamp Shadowing Suppresses Contract Action Trace in `/results/:transactionIdOrHash/actions`

## Summary
When an Ethereum transaction hash maps to multiple rows in `contract_transaction_hash` — a failed original execution at T1 and a later `DUPLICATE_TRANSACTION` at T2 — the hash-lookup query returns T2 (newer timestamp wins the tiebreaker). `getContractActions` then queries contract actions at T2, which has none, returning an empty action list and hiding the original execution's trace.

## Finding Description
`ethereumTransactionsByHashQuery` in `rest/service/contractService.js` (lines 208–212) orders results by `(transaction_result = SUCCESS) desc, consensus_timestamp desc`. When neither row is `SUCCESS`, the tiebreaker is the latest timestamp. A `DUPLICATE_TRANSACTION` entry submitted after the original failed transaction (T2 > T1) is therefore returned.

`getContractActions` in `rest/controllers/contractController.js` (lines 1256–1269) takes `tx[0].consensusTimestamp` without inspecting the transaction result code and passes it directly to `getContractActionsByConsensusTimestamp`. Because `DUPLICATE_TRANSACTION` is never executed by the EVM, no `contract_action` rows exist at T2, so the endpoint returns `{"actions": [], "links": {"next": null}}`.

The importer's `addDefaultEthereumTransactionContractResult` (lines 100–107 of `ContractResultServiceImpl.java`) explicitly skips inserting a `contract_result` for `DUPLICATE_TRANSACTION`, confirming these entries have no EVM execution record. The test fixture `transaction-hash-match-multiple-tx-latest-failed.json` confirms the system already handles this multi-entry scenario and that the DUPLICATE_TRANSACTION timestamp is selected.

## Impact Explanation
Any call to `/api/v1/contracts/results/{hash}/actions` for an affected hash returns an empty action list, misrepresenting that no EVM execution occurred. Block explorers, auditors, and tooling relying on this endpoint see a blank trace instead of the real one. The ledger itself is unaffected, but the mirror node — the canonical read API — returns incorrect data for that hash permanently (the duplicate record persists).

## Likelihood Explanation
Submitting a duplicate Ethereum transaction to Hedera requires only the original raw signed transaction bytes and a funded Hedera account. The raw bytes may be publicly observable (e.g., captured from a public RPC relay). The standard transaction fee is the only cost. No special keys or permissions are required. The attack is repeatable and the corrupted state is permanent.

## Recommendation
In `getContractActions`, after resolving the transaction by hash, filter out `DUPLICATE_TRANSACTION` (and similarly `WRONG_NONCE`) results before selecting the `consensusTimestamp`. Specifically, the `ethereumTransactionsByHashQuery` ordering should deprioritize `DUPLICATE_TRANSACTION` and `WRONG_NONCE` results (similar to how `SUCCESS` is prioritized), or the controller should skip any resolved transaction whose result code indicates it was never executed by the EVM. A secondary fix is to add an explicit `transaction_result NOT IN (DUPLICATE_TRANSACTION, WRONG_NONCE)` filter when the intent is to find the actionable execution.

## Proof of Concept
1. Submit a valid Ethereum transaction `T` to Hedera; it fails with `CONSENSUS_GAS_EXHAUSTED` at consensus timestamp T1. Contract actions are recorded at T1.
2. Resubmit the identical raw signed transaction bytes. Hedera records it as `DUPLICATE_TRANSACTION` at T2 > T1.
3. Call `GET /api/v1/contracts/results/{hash(T)}/actions`.
4. `ethereumTransactionsByHashQuery` returns T2 (neither is SUCCESS; T2 > T1).
5. `getContractActionsByConsensusTimestamp(T2, ...)` finds no rows.
6. Response: `{"actions": [], "links": {"next": null}}` — the original execution trace is invisible.
