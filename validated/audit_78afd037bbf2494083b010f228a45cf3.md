Audit Report

## Title
Cache Key Hash Collision in `findByEvmAddressOrAliasAndDeletedIsFalse` Enables Cache Poisoning and Wrong Entity Return

## Summary
The `CACHE_NAME_ALIAS` and `CACHE_NAME_EVM_ADDRESS` caches in `EntityRepository` use a 32-bit `Arrays.hashCode(byte[])` integer as the sole cache key. Because this hash space has only 2^32 possible values and collisions are analytically trivial to compute, an unprivileged attacker can craft an alias byte array that collides with a cached entry, either receiving a wrong entity object or pre-poisoning the cache so that a legitimate alias lookup returns `Optional.empty()` instead of the real entity.

## Finding Description

**Exact code path:**

`SpelHelper.hashCode` at `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java` lines 20–22:
```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);  // 32-bit int
}
``` [1](#0-0) 

`findByEvmAddressOrAliasAndDeletedIsFalse` and `findByEvmAddressAndDeletedIsFalse` at `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java` lines 32–49:
```java
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",   // ← sole key = 32-bit int
        unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);

@Cacheable(
        cacheNames = CACHE_NAME_ALIAS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",   // ← sole key = 32-bit int
        unless = "#result == null")
Optional<Entity> findByEvmAddressOrAliasAndDeletedIsFalse(byte[] alias);
``` [2](#0-1) 

**Root cause:** `Arrays.hashCode(byte[])` is a polynomial hash `result = 31*result + b` over a 32-bit integer space. The entire cache key is this single integer. Two distinct byte arrays that produce the same integer are treated as the same cache entry. Spring Cache with Caffeine stores and retrieves by this key with no secondary equality check on the actual byte array.

**Failed assumption:** The code assumes `Arrays.hashCode()` is collision-free for the alias inputs it will receive. It is not — collisions are analytically computable in O(1).

**Collision example:**
- `Arrays.hashCode([0, 31])` = 31×(31×1 + 0) + 31 = **992**
- `Arrays.hashCode([1, 0])` = 31×(31×1 + 1) + 0 = **992**

**Why `unless = "#result == null"` fails:** Spring Data JPA returns `Optional.empty()` (not Java `null`) when no entity is found. `Optional.empty() != null`, so the `unless` condition evaluates to `false` and the empty result **is** cached under the colliding hash key. This allows a "not found" result to poison the slot for a legitimate alias. [4](#0-3) 

**Exploit flow — cache poisoning:**
1. Attacker identifies target alias `X` of a real entity (aliases are public on-chain).
2. Attacker computes alias `Y` (no real entity) such that `Arrays.hashCode(Y) == Arrays.hashCode(X)`.
3. Attacker sends a request using alias `Y` before any legitimate request for `X` has populated the cache.
4. DB lookup for `Y` returns no entity → `Optional.empty()` is returned and **cached** under key `hashCode(Y) == hashCode(X)`.
5. All subsequent legitimate requests for alias `X` hit the cache and receive `Optional.empty()` — the real entity is never fetched from DB until the cache entry expires (TTL: 1 second).

**Exploit flow — wrong entity return:**
1. Entity A (alias `X`) is already cached under key `H = Arrays.hashCode(X)`.
2. Attacker sends a request with alias `Y` where `Arrays.hashCode(Y) == H`.
3. Cache hit → Entity A is returned for alias `Y`, causing incorrect EVM execution for the caller.

The same flaw exists identically in `findByEvmAddressAndDeletedIsFalse` (lines 32–37, `CACHE_NAME_EVM_ADDRESS`). [6](#0-5) 

## Impact Explanation
- **Wrong entity returned:** A query with a crafted alias returns a different entity's full object, causing incorrect EVM execution results (wrong balance, wrong contract state) for callers relying on alias-based resolution through `CommonEntityAccessor.get(Bytes, Optional<Long>)`. [5](#0-4) 
- **Cache poisoning DoS:** An attacker can make any entity unreachable via alias lookup for the duration of the cache TTL (1 second, configurable) by pre-populating the cache slot with a colliding non-existent alias. This affects `eth_call`, balance queries, and any EVM execution path that resolves accounts by alias. The 1-second default TTL limits the window per poisoning attempt, but the attack is trivially repeatable. [3](#0-2) 
- Severity: **Medium**. No authentication is required; the attack is repeatable; it affects correctness and availability of the web3 API for targeted accounts. The short default TTL (1s) reduces the DoS window but does not eliminate the wrong-entity-return scenario, which requires only a single request.

## Likelihood Explanation
- Hedera account aliases are public (visible in transaction records and the mirror node REST API), so an attacker can enumerate targets.
- Computing a colliding alias is O(1) arithmetic — no brute force needed.
- The attacker only needs to send a single HTTP request with the crafted alias to trigger the wrong entity return.
- For the cache poisoning DoS, the attack must be repeated each cache TTL cycle (default 1 second), but this is trivially automatable and is a low-volume targeted attack, not a volumetric DDoS.
- No special privileges, credentials, or network position are required.

## Recommendation
Replace the 32-bit hash sole key with the full byte array. Spring Cache supports composite keys and byte-array-safe keys. Options:
1. Use `key = "T(java.util.Arrays).toString(#alias)"` to produce a string key that is collision-free for distinct byte arrays.
2. Use `key = "T(org.springframework.util.DigestUtils).md5DigestAsHex(#alias)"` for a compact but collision-resistant key.
3. Wrap the byte array in a `ByteBuffer` or `String` before using it as a key, ensuring `equals()` and `hashCode()` are based on content.
4. Add a secondary post-cache-hit verification: after a cache hit, verify that the returned entity's alias/evm_address actually matches the queried `alias` bytes before returning it.
5. Change `unless = "#result == null"` to `unless = "#result == null || !#result.isPresent()"` to prevent caching of empty results (mitigates the poisoning scenario, though not the wrong-entity-return scenario).
