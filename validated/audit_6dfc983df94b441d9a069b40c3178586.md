Audit Report

## Title
Unbounded In-Memory Opcode Response Buffering Enables Heap Exhaustion DoS via Trivially-Satisfied gzip Check

## Summary
`OpcodesController.getContractOpcodes()` gates access behind an `Accept-Encoding: gzip` header check and a 1 req/sec token-bucket throttle, but neither prevents the full `OpcodeContext.opcodes` `ArrayList` from being materialized in JVM heap before serialization. A single request against a high-gas historical transaction with `stack=true&memory=true&storage=true` can exhaust heap and crash the JVM.

## Finding Description

**`OpcodesController.getContractOpcodes()`** performs two checks before processing:

1. `validateAcceptEncodingHeader(acceptEncoding)` rejects requests lacking `gzip` in `Accept-Encoding`. The Javadoc comment explicitly acknowledges the response is "huge." However, this check only gates on a header value the attacker fully controls — any attacker simply sends `Accept-Encoding: gzip`. It does not prevent heap allocation. [1](#0-0) 

2. `throttleManager.throttleOpcodeRequest()` is implemented in `ThrottleManagerImpl` as a token-bucket check against `opcodeRateLimitBucket`. It limits throughput to `opcodeRequestsPerSecond = 1` (default) but places zero constraint on the size of any individual response. [2](#0-1) [3](#0-2) 

After both checks pass, `OpcodeServiceImpl.processOpcodeCall()` constructs an `OpcodeContext` whose `opcodes` field is an `ArrayList` initialized with capacity `gas/3` — a hint, not a cap. [4](#0-3) 

`OpcodeContext.addOpcodes()` appends to this list without any bound check. The EVM re-executes the full historical transaction, appending one `Opcode` entry per executed instruction. With `memory=true`, each entry carries a full snapshot of EVM memory at that step; with `stack=true` (the default), each carries up to 1024×32 bytes of stack; with `storage=true`, all modified storage slots. [5](#0-4) 

The entire list is returned as a Java object and serialized to JSON in-process before any compression occurs. [6](#0-5) 

**Root cause:** The failed assumption is that requiring `Accept-Encoding: gzip` prevents large responses from being buffered. It does not — it only affects transport encoding. The actual heap allocation happens unconditionally during EVM re-execution and response construction, with no size limit at any layer.

## Impact Explanation
A single request against a high-gas historical transaction (Hedera allows up to 15 M gas; `PUSH1` costs 3 gas, yielding up to ~5 M opcodes per transaction) with all three trace options enabled can produce hundreds of MB to multiple GB of live Java objects in heap. This causes `OutOfMemoryError`, crashing the JVM and taking down the entire web3 API service. All concurrent requests are dropped. The impact is a complete denial-of-service of the web3 API node.

## Likelihood Explanation
Preconditions: (1) the operator has set `hiero.mirror.web3.opcode.tracer.enabled=true` (disabled by default per `OpcodesProperties`, but required for the endpoint to function at all); (2) the attacker knows or can discover a historical transaction hash with high gas consumption (publicly visible on-chain). [7](#0-6) 

Both preconditions are realistic for any production deployment of this feature. The exploit requires no credentials, no special protocol knowledge, and no prior access — only an HTTP client that includes `Accept-Encoding: gzip`. The 1 req/sec throttle means a single attacker connection is sufficient; no flood is needed.

## Recommendation
1. **Enforce a hard cap on opcode count.** Add a configurable `maxOpcodes` property to `OpcodesProperties` and check it inside `OpcodeContext.addOpcodes()` — throw a `ResponseStatusException` (e.g., `413 Payload Too Large`) when the limit is exceeded. [8](#0-7) 

2. **Enforce a gas-based pre-flight limit.** In `OpcodeServiceImpl.processOpcodeCall()`, reject requests where `params.getGas()` exceeds a configurable threshold before EVM re-execution begins. [4](#0-3) 

3. **Add a gas-based throttle token cost.** Extend `throttleOpcodeRequest()` to consume tokens proportional to the gas limit of the requested transaction (similar to how `throttle(ContractCallRequest)` consumes from `gasLimitBucket`), so high-gas replays are throttled more aggressively. [9](#0-8) 

4. **Stream the response.** Consider streaming the opcode list incrementally (e.g., via `StreamingResponseBody` or reactive types) rather than materializing the full object graph before serialization, to avoid peak heap usage.

## Proof of Concept
```
# 1. Find a high-gas historical transaction hash (publicly visible on-chain)
TX_HASH="0x<high_gas_tx_hash>"

# 2. Send a single request with all trace options and the trivially-satisfied gzip header
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-web3>/api/v1/contracts/results/${TX_HASH}/opcodes?stack=true&memory=true&storage=true"

# Result: JVM heap exhaustion -> OutOfMemoryError -> web3 API service crash
# No flood required; a single request suffices at the default 1 req/sec throttle.
```
