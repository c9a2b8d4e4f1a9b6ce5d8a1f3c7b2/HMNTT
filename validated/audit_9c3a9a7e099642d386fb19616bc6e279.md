Audit Report

## Title
Integer Narrowing Overflow in `poll()` Bypasses DB Query Limit, Enabling Unbounded Query DoS

## Summary
A `uint64 limit` value of `Long.MAX_VALUE` sent via the `subscribeTopic` gRPC endpoint passes `@Min(0)` validation, then is narrowed to `int` via an unchecked cast in `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`, producing `-1`. This causes `hasLimit()` to return `false` on the internally-built filter, which skips the `setMaxResults` guard in `findByFilter()`, issuing an unbounded `SELECT` against the `topic_message` table. The same overflow recurs on every subsequent poll because `Long.MAX_VALUE - N` remains a large positive `long` whose lower 32 bits are always negative for any practical `N`.

## Finding Description

**Entry point — no upper-bound check on `limit`:**

`ConsensusController.toFilter()` maps the proto `uint64 limit` field directly to `TopicMessageFilter.limit` with no cap: [1](#0-0) 

**Validation passes for `Long.MAX_VALUE`:**

`TopicMessageFilter.limit` carries only `@Min(0)`: [2](#0-1) 

`Long.MAX_VALUE` (`9223372036854775807`) satisfies `>= 0`, so Spring's `@Validated` AOP on `TopicMessageServiceImpl` does not reject it.

**Overflow in `PollingTopicMessageRetriever.poll()`:**

```
filter.getLimit()        = Long.MAX_VALUE  (0x7FFFFFFFFFFFFFFF)
context.getTotal().get() = 0               (first poll)
difference               = 0x7FFFFFFFFFFFFFFF
(int) 0x7FFFFFFFFFFFFFFF = 0xFFFFFFFF = -1  ← negative
Math.min(-1, maxPageSize) = -1
``` [3](#0-2) 

**Internal filter built with `limit = -1` bypasses `hasLimit()`:**

`filter.toBuilder().limit(pageSize)` is a raw Lombok builder call — Spring's `@Validated` AOP interceptor is never invoked, so `@Min(0)` is not re-enforced: [4](#0-3) 

`hasLimit()` returns `limit > 0`; with `limit = -1` this is `false`: [5](#0-4) 

**`setMaxResults` is never called — unbounded query executes:** [6](#0-5) 

`getResultList()` is synchronous and loads the entire result set into JVM heap before any reactive timeout can cancel it: [7](#0-6) 

**Same pattern exists in `PollingTopicListener.poll()`:** [8](#0-7) 

**`isComplete()` does not save the situation:**

With `filter.getLimit() = Long.MAX_VALUE`, `limitHit` requires `total.get() == Long.MAX_VALUE` — practically never true. For throttled mode the only other exit is `pageSize.get() < maxPageSize`, which only fires *after* the unbounded query already returns. Moreover, on every subsequent poll `Long.MAX_VALUE - N` (for any realistic `N`) still has lower 32 bits that are negative, so the overflow repeats indefinitely: [9](#0-8) 

## Impact Explanation
A single `subscribeTopic` call with `limit = Long.MAX_VALUE` causes the mirror node to issue a `SELECT * FROM topic_message WHERE topic_id = ?` with no `LIMIT` clause, loading potentially millions of rows into JVM heap. On a busy HCS topic this causes OOM or severe GC pressure, crashing or stalling the gRPC service for all subscribers. The `retrieverProperties.getTimeout()` wraps the reactive stream but cannot interrupt the synchronous `getResultList()` call mid-execution. The database-level `statementTimeout` (default 10 s) limits per-query duration but does not prevent the attack — each connection can issue repeated unbounded queries, and multiple concurrent attackers amplify the effect. The `maxConcurrentCallsPerConnection = 5` limit is easily bypassed by opening multiple connections.

## Likelihood Explanation
The gRPC endpoint is publicly accessible with no authentication required. The attacker needs only a valid `topicId` (trivially discoverable from the public ledger) and a single gRPC call with `limit = 9223372036854775807`. No special privileges, no rate-limit bypass, and no prior knowledge of internals is needed. The attack is trivially repeatable with `grpcurl` or any gRPC client library.

## Recommendation
1. **Cap the limit at the controller layer** in `ConsensusController.toFilter()`: reject or clamp any `limit` value exceeding a configured maximum (e.g., `Long.MAX_VALUE` or a reasonable business cap) before building the `TopicMessageFilter`.
2. **Use safe narrowing in `poll()`**: replace `(int)(filter.getLimit() - context.getTotal().get())` with `(int) Math.min(filter.getLimit() - context.getTotal().get(), Integer.MAX_VALUE)` to prevent negative overflow.
3. **Add a `@Max` constraint** on `TopicMessageFilter.limit` to enforce an upper bound at the validation layer.
4. **Add a defensive guard in `findByFilter()`**: if `filter.getLimit() <= 0` after the internal rebuild, treat it as `hasLimit() == false` and apply a hard fallback `setMaxResults(maxPageSize)`.

## Proof of Concept
```bash
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1234},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```
This sends `limit = Long.MAX_VALUE`. The mirror node will:
1. Pass `@Min(0)` validation (value is positive).
2. Enter `PollingTopicMessageRetriever.poll()` where `(int)(Long.MAX_VALUE - 0) = -1`.
3. Build an internal filter with `limit = -1`, causing `hasLimit()` to return `false`.
4. Issue `SELECT * FROM topic_message WHERE topic_id = 1234 ORDER BY consensus_timestamp ASC` with no `LIMIT` clause.
5. Repeat this unbounded query on every poll interval until the reactive timeout fires (default 60 s), with each query bounded only by the DB statement timeout (default 10 s).
