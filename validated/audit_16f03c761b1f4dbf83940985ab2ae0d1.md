Audit Report

## Title
Silent Transaction Omission in `FindBetween()` Pagination Due to Shared `consensus_timestamp` Boundary Skip

## Summary
The `FindBetween()` pagination loop in `rosetta/app/persistence/transaction.go` advances its cursor via `start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1`. Because the `transaction` table permits multiple rows per `consensus_timestamp` (differentiated by `nonce`), any rows sharing the boundary timestamp with the last row of a full 2000-row batch are permanently skipped for that query. No error is returned; the response appears normal.

## Finding Description

**Location**: `rosetta/app/persistence/transaction.go`, `FindBetween()`, lines 112–147.

The constant `batchSize = 2000` is declared at line 22. [1](#0-0) 

The pagination loop fetches ordered batches and advances the cursor: [2](#0-1) 

The cursor advancement at line 146 assumes `consensus_timestamp` is unique per row. It is not.

**Schema evidence — composite PK**: Migration `V2.0.3__index_init.sql` line 261 established a composite primary key of `(consensus_timestamp, payer_account_id)`: [3](#0-2) 

**Schema evidence — PK dropped**: Migration `V2.5.1__update_indexes.sql` line 2 drops that constraint entirely, leaving no uniqueness guarantee on `consensus_timestamp`: [4](#0-3) 

**Schema evidence — `nonce` column**: The `transaction` table carries a `nonce integer not null default 0` column (line 677 of `V2.0.0__create_tables.sql`), explicitly designed to accommodate internal/scheduled transactions that share the same `consensus_timestamp` as their parent: [5](#0-4) 

This was also added to the v1 schema via `V1.50.2__hts_precompiles.sql`: [6](#0-5) 

**Termination check does not detect mid-timestamp truncation**: The only loop exit condition is `len(transactionsBatch) < batchSize` (line 142), which correctly detects end-of-data but cannot detect that a full batch ended in the middle of a group of rows sharing the same timestamp: [7](#0-6) 

**Call path**: `block_service.go` line 56 calls `FindBetween` directly in response to a `/block` request, with no additional safeguard: [8](#0-7) 

## Impact Explanation
An incomplete transaction set is returned for any block where the batch boundary falls mid-group on a shared `consensus_timestamp`. The omission is silent — no error is returned and the response is structurally valid. Downstream Rosetta consumers (exchange reconciliation systems, block explorers, audit tools) receive a silently truncated ledger presented as authoritative. This violates ledger correctness for any block exhibiting this condition.

## Likelihood Explanation
- **Precondition 1**: A block must contain ≥ 2000 transactions. This occurs on Hedera mainnet during high-throughput periods.
- **Precondition 2**: The 2000th (or 4000th, …) row must share its `consensus_timestamp` with at least one subsequent row. Internal transactions triggered by HTS precompile calls and scheduled transaction executions produce exactly this pattern, as they share the parent's `consensus_timestamp` and are differentiated only by `nonce`.
- **Attacker capability**: Zero privilege required. Any caller of `GET /block` with a valid block identifier triggers the code path.
- **Repeatability**: Deterministic — once a block with this property exists, every `/block` request for it returns the same incomplete result.
- **Detectability**: No error is surfaced.

## Recommendation
Replace the single-field `+1` cursor with a proper keyset (composite) cursor. The `ORDER BY` clause should include `consensus_timestamp, payer_account_id, nonce` as a stable tiebreaker, and the `WHERE` predicate should advance past the full composite key of the last row seen:

```sql
WHERE (consensus_timestamp, payer_account_id, nonce) > (@last_ts, @last_payer, @last_nonce)
  AND consensus_timestamp <= @end
ORDER BY consensus_timestamp, payer_account_id, nonce
```

This ensures no row sharing the boundary timestamp is ever skipped, regardless of batch size or timestamp collisions.

## Proof of Concept
1. Identify (or construct on a local fork) a block containing ≥ 2000 transactions where at least one internal/scheduled transaction shares its `consensus_timestamp` with the 2000th row.
2. Issue `GET /block` with that block's identifier.
3. Observe that the response omits the transactions whose `consensus_timestamp` equals the boundary value but whose `(payer_account_id, nonce)` tuple places them after position 2000 in the ordered result.
4. Confirm by querying the database directly:
   ```sql
   SELECT consensus_timestamp, payer_account_id, nonce
   FROM transaction
   WHERE consensus_timestamp BETWEEN <block_start> AND <block_end>
   ORDER BY consensus_timestamp, payer_account_id, nonce;
   ```
   The rows missing from the API response will be visible in the direct query result.
