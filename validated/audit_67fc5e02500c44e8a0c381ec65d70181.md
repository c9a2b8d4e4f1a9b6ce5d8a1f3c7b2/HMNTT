# Audit Report

## Title
Malicious Block Node Can Persistently Evade Inactivity Penalty via Deliberate `serverStatus()` Timeout

## Summary
`BlockNode.getBlockRange()` catches all exceptions (including gRPC deadline exceeded) from a timed-out `serverStatus()` call and silently returns `EMPTY_BLOCK_RANGE` without invoking `onError()`. Because the error counter is never incremented through this path, a block node that deliberately delays every status response beyond the 400 ms `responseTimeout` default is never marked inactive. It causes a 400 ms stall and an `ERROR`-level log entry on every block-selection cycle, indefinitely, with no self-healing.

## Finding Description

**`StreamProperties.java` — default timeout**

`responseTimeout` defaults to 400 ms: [1](#0-0) 

**`BlockNode.getBlockRange()` — exception swallowed, `onError()` never called**

The method applies the deadline, then catches every `Exception`, logs at `ERROR`, and returns `EMPTY_BLOCK_RANGE`. `onError()` is not called: [2](#0-1) 

**`onError()` is only reachable via `streamBlocks()`**

The inactivity mechanism (`errors` counter → `active = false` → `readmitTime`) is exclusively wired into the catch blocks of `streamBlocks()`: [3](#0-2) [4](#0-3) 

**`BlockNodeSubscriber.hasBlock()` — `EMPTY_BLOCK_RANGE` treated as "no blocks"**

`EMPTY_BLOCK_RANGE` is `Range.closedOpen(0L, 0L)`, which satisfies `blockRange.isEmpty()`, so `hasBlock()` returns `false`: [5](#0-4) 

**`BlockNodeSubscriber.getNode()` — throws if no node qualifies**

After iterating all active and inactive nodes without finding one that passes `hasBlock()`, the method throws `BlockStreamException`: [6](#0-5) 

**Auto-discovery amplifies reach**

When `autoDiscoveryEnabled=true`, `BlockNodeDiscoveryService.discover()` pulls every `BLOCK_NODE`-typed registered node from the database and adds it to the active node list: [7](#0-6) 

**Root cause — failed assumption:** The design assumes that a node failing `serverStatus()` is merely "empty" (has no blocks yet) rather than "misbehaving." No penalty is applied, so a deliberately slow node is permanently retained in the active pool.

## Impact Explanation

| Scenario | Effect |
|---|---|
| One malicious node among healthy peers | Every `getNode()` call incurs a 400 ms stall + `ERROR` log. Repeats every scheduled cycle with no self-healing. |
| Sole configured/discovered node | `getNode()` throws `BlockStreamException` every cycle. `CompositeBlockSource` counts 3 consecutive failures and switches away from `BLOCK_NODE` source (AUTO mode) or halts ingestion entirely (BLOCK_NODE-only mode). |
| Multiple malicious nodes | Stalls are additive: N × 400 ms per cycle. |

`CompositeBlockSource.SourceHealth` threshold of 3 errors before switching source: [8](#0-7) 

## Likelihood Explanation

The attacker needs only to operate a reachable block node that delays every `serverStatus()` gRPC response beyond 400 ms (trivially implemented with `Thread.sleep` in the server handler). Entry is possible via:

1. **Manual operator configuration** — no network access required beyond reachability.
2. **Auto-discovery** — registering a tier-1 `BLOCK_NODE` service endpoint in the network registry is a standard, unprivileged network operation. Once registered, every mirror-node importer with `autoDiscoveryEnabled=true` will automatically include the node.

The attack is repeatable indefinitely, requires no ongoing authentication, and survives importer restarts because the node is never penalized.

## Recommendation

Call `onError()` inside the `catch` block of `getBlockRange()` so that repeated `serverStatus()` timeouts are treated the same as repeated `streamBlocks()` failures:

```java
// BlockNode.java — getBlockRange()
} catch (Exception ex) {
    log.error("Failed to get server status for {}", this, ex);
    onError();               // <-- add this
    return EMPTY_BLOCK_RANGE;
}
```

This ensures a node that consistently times out on status checks accumulates errors, is eventually marked inactive, and is subject to the existing `readmitDelay` cooldown — the same self-healing path already used by `streamBlocks()`. [9](#0-8) 

## Proof of Concept

1. Stand up a gRPC server implementing `BlockNodeService` whose `serverStatus()` handler sleeps for 500 ms before responding (exceeding the 400 ms deadline).
2. Register this server as the sole block node (manually or via auto-discovery).
3. Start the mirror-node importer targeting this node.
4. Observe: every scheduled `get()` cycle logs `ERROR Failed to get server status for BlockNode(...)`, the node's `active` field remains `true`, `errors` stays at 0, and the stall repeats indefinitely with no fallback or cooldown applied.
5. Confirm: replacing the `catch` block with an `onError()` call causes the node to be marked inactive after `maxSubscribeAttempts` (default 3) failures and triggers the `readmitDelay` cooldown. [10](#0-9) [11](#0-10)
