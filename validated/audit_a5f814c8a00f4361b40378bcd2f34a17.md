Audit Report

## Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` via Unbounded `uint64` Limit Allows Repeated Full-Table Scans (DoS)

## Summary
`ConsensusController.toFilter()` passes `query.getLimit()` (a protobuf `uint64` decoded as a Java `long`) directly to `TopicMessageFilter.limit` with no upper bound. In `PollingTopicMessageRetriever.poll()`, the narrowing cast `(int)(filter.getLimit() - context.getTotal().get())` overflows to `-1` when `limit = Long.MAX_VALUE`, causing `Math.min(-1, maxPageSize)` to produce `-1`. This negative value is forwarded to the repository as the new filter's limit; because `hasLimit()` returns `false` for `-1`, no SQL `LIMIT` clause is applied, resulting in full-table scans on every poll cycle for up to 60 seconds per connection.

## Finding Description

**Step 1 — No upper bound in `toFilter()`**

`ConsensusController.toFilter()` sets the filter limit directly from the protobuf field with no cap:

```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
```

Any value from `1` to `Long.MAX_VALUE` is a valid positive Java `long` and passes the downstream `hasLimit()` check (`limit > 0`). [1](#0-0) 

**Step 2 — `@Min(0)` does not catch `Long.MAX_VALUE`**

`TopicMessageFilter.limit` carries `@Min(0)`, which only rejects negative values. `Long.MAX_VALUE` is positive and passes. The `@Valid` annotation **is** present on the interface method (`TopicMessageService.subscribeTopic(@Valid TopicMessageFilter filter)`), and `TopicMessageServiceImpl` is annotated `@Validated`, so Bean Validation is triggered — but it cannot reject a value that satisfies `@Min(0)`. [2](#0-1) [3](#0-2) 

**Step 3 — Integer overflow in `poll()`**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```

With `filter.getLimit() = Long.MAX_VALUE` and `total = 0`:
- `filter.hasLimit()` → `true` (`Long.MAX_VALUE > 0`)
- `(int)(Long.MAX_VALUE - 0)` = `(int)(0x7FFFFFFFFFFFFFFF)` = **`-1`**
- `Math.min(-1, 1000)` = **`-1`**

The overflow persists on subsequent polls: `Long.MAX_VALUE - N` for any realistic `N` still truncates to a negative `int`. [4](#0-3) 

**Step 4 — Negative page size forwarded to repository with no SQL LIMIT**

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
```

`pageSize = -1` (int) is widened to `-1L` for the `long limit` field. In `findByFilter()`:

```java
if (filter.hasLimit()) {          // -1 > 0 → false
    typedQuery.setMaxResults(...); // never reached
}
```

`setMaxResults()` is never called, so the JPA query carries no `LIMIT` clause and returns all matching rows. [5](#0-4) [6](#0-5) 

**Step 5 — Polling continues for the full 60-second timeout**

`isComplete()` checks `filter.getLimit() == total.get()`, i.e., `Long.MAX_VALUE == total` — practically never true. The `flux.take(filter.getLimit())` guard in `TopicMessageServiceImpl` resolves to `flux.take(Long.MAX_VALUE)`, which is also effectively unlimited. With the default `pollingFrequency = 2s` and `timeout = 60s`, approximately 30 unlimited queries execute before the connection is dropped. [7](#0-6) [8](#0-7) [9](#0-8) 

## Impact Explanation

Each malicious subscription triggers ~30 full-table scans against the `topic_message` table over 60 seconds. With `N` concurrent connections the database receives `N × 30` unlimited queries. This constitutes a practical Denial-of-Service against the database backend. All matching topic messages are also streamed to the caller, constituting unintended bulk data exfiltration beyond any operator-intended limit.

## Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly accessible with no authentication. The exploit requires only a standard gRPC client setting `limit = 9223372036854775807` (`Long.MAX_VALUE`). No special privileges or internal knowledge are required. The attack is trivially scriptable and repeatable.

## Recommendation

1. **Cap the limit in `toFilter()`**: Apply an operator-configurable maximum (e.g., `Math.min(query.getLimit(), maxAllowedLimit)`) before constructing the filter.
2. **Guard the cast in `poll()`**: Before the narrowing cast, clamp the computed remaining limit to `[0, Integer.MAX_VALUE]`:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
3. **Add an upper-bound constraint**: Add `@Max(someReasonableMax)` alongside `@Min(0)` on `TopicMessageFilter.limit` to reject absurdly large values at the validation layer. [10](#0-9) [11](#0-10)


**Trace through the code:**
1. `toFilter()` sets `filter.limit = Long.MAX_VALUE` — no cap applied.
2. `subscribeTopic(@Valid filter)` — `@Min(0)` passes since `Long.MAX_VALUE > 0`.
3. First `poll()`: `(int)(Long.MAX_VALUE - 0)` = `-1`; `Math.min(-1, 1000)` = `-1`.
4. `newFilter.limit = -1L`; `hasLimit()` = `false`; `setMaxResults()` skipped → full table scan.
5. `isComplete()` = `Long.MAX_VALUE == 0` = `false`; polling repeats every 2 s for 60 s → ~30 unlimited queries per connection.
