Audit Report

## Title
Cache Poisoning via 32-bit Hash Collision in SpelHelper-Keyed Entity Cache

## Summary
`SpelHelper.hashCode(byte[])` delegates to `Arrays.hashCode()`, producing a 32-bit integer used as the **sole** cache key for EVM address and alias entity lookups in `EntityRepository`. Because the 32-bit output space (2^32) is trivially collided against 20-byte EVM address inputs (2^160), an attacker who can create Hedera accounts can poison the shared Caffeine cache, causing subsequent lookups for a victim's EVM address to return a different entity's data — or an empty result — within the cache TTL window.

## Finding Description

**`SpelHelper.hashCode`** at `common/src/main/java/org/hiero/mirror/common/util/SpelHelper.java` lines 20–22 returns `Arrays.hashCode(value)` — a non-cryptographic 32-bit polynomial hash (`result = 31 * result + b` per byte).

This value is used as the **sole** cache key in two repository methods in `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`:
- `findByEvmAddressAndDeletedIsFalse` (line 35): `key = "@spelHelper.hashCode(#alias)"`
- `findByEvmAddressOrAliasAndDeletedIsFalse` (line 42): `key = "@spelHelper.hashCode(#alias)"`

Both caches (`evmAddress`, `alias`) are backed by a shared Caffeine instance registered as `CACHE_MANAGER_ENTITY` in `web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java` lines 99–105, configured with `expireAfterWrite=1s, maximumSize=10000`.

**Critical detail on `unless` guard:** The annotation `unless = "#result == null"` does **not** prevent caching of `Optional.empty()` — an empty Optional is a non-null object. This means a cache miss (entity not found) is also stored under the colliding hash key, enabling a false-negative attack path: if address B (non-existent) is looked up first and cached as `Optional.empty()` under key `hashCode(B)`, any subsequent lookup for address A where `hashCode(A) == hashCode(B)` will return `Optional.empty()` from cache, denying resolution of a legitimate entity.

**Root cause:** The design assumes `Arrays.hashCode` produces unique keys for distinct byte arrays. For 20-byte EVM addresses, the input space is 2^160 but the key space is only 2^32. Collisions are guaranteed and computationally trivial (~2^16 attempts via birthday attack).

## Impact Explanation

The entity cache is shared across all requests. A poisoned entry persists for up to 1 second (the default `expireAfterWrite` TTL). Any caller resolving address B during that window — including EVM contract execution, balance checks, and alias resolution via `CommonEntityAccessor` — receives the wrong `Entity` object or an empty result. This can cause:
- Wrong contract bytecode or account data returned for EVM calls
- Incorrect balance or key information served to users
- False-negative entity resolution (legitimate entity appears non-existent) if the empty-Optional path is exploited

The 1-second TTL is a meaningful mitigating factor that narrows the attack window significantly compared to longer-lived caches, but does not eliminate the vulnerability — the collision is deterministic and the attacker controls timing.

## Likelihood Explanation

The attack requires no special privilege — only the ability to create Hedera accounts (standard network access with HBAR). Finding a collision pair takes ~65,536 key-pair generations, trivial on commodity hardware. The attacker can pre-generate a collision table offline and register accounts for one side of each pair. The attack must be timed within the 1-second TTL window, which requires coordinated request injection but is feasible for an attacker with network access to the mirror node API. The attack is repeatable.

## Recommendation

Replace the 32-bit `Arrays.hashCode` cache key with the full byte array as the cache key. Spring's Caffeine cache supports arbitrary `Object` keys; `byte[]` keys can be wrapped in `ByteBuffer` (which implements `equals`/`hashCode` correctly based on content) or a dedicated value type:

```java
// Instead of:
key = "@spelHelper.hashCode(#alias)"

// Use:
key = "T(java.nio.ByteBuffer).wrap(#alias)"
```

Alternatively, encode the byte array as a hex string key:
```java
key = "T(org.apache.commons.codec.binary.Hex).encodeHexString(#alias)"
```

This eliminates the collision entirely by using the full input as the key.

## Proof of Concept

The following demonstrates finding a collision pair for `Arrays.hashCode` on 20-byte inputs:

```java
import java.util.Arrays;
import java.util.HashMap;
import java.util.Random;

public class CollisionDemo {
    public static void main(String[] args) {
        HashMap<Integer, byte[]> seen = new HashMap<>();
        Random rng = new Random();
        int attempts = 0;
        while (true) {
            byte[] addr = new byte[20];
            rng.nextBytes(addr);
            int h = Arrays.hashCode(addr);
            if (seen.containsKey(h)) {
                byte[] collision = seen.get(h);
                System.out.println("Collision after " + attempts + " attempts:");
                System.out.println("A: " + Arrays.toString(collision));
                System.out.println("B: " + Arrays.toString(addr));
                System.out.println("hashCode(A) == hashCode(B): " + (Arrays.hashCode(collision) == h));
                break;
            }
            seen.put(h, addr);
            attempts++;
        }
    }
}
```

Expected output: collision found in ~65,000 attempts (birthday bound for 32-bit space). Once addresses A and B are identified:
1. Register a Hedera account for address A.
2. Trigger `findByEvmAddressAndDeletedIsFalse(A)` — entity A is cached under key `hashCode(A)`.
3. Within 1 second, trigger `findByEvmAddressAndDeletedIsFalse(B)` — cache hit returns entity A's data for address B.
