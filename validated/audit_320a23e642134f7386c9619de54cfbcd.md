Audit Report

## Title
Unprivileged User Can Force Contract-Create Simulation Path via Explicit Zero-Address `to` Field, Bypassing Initcode Validation

## Summary
Submitting `/api/v1/contracts/call` with `"to": "0x0000000000000000000000000000000000000000"` and arbitrary `data` bypasses the `hasTo()` bean-validation guard and routes the request into `buildContractCreateTransactionBody()`, using the caller-supplied `data` verbatim as EVM initcode. The root cause is that `ContractController` uses `Address.ZERO` as a dual-purpose sentinel for both "no `to` supplied" and "contract create", but never normalises an explicitly supplied zero address before the downstream `isZero()` branch check.

## Finding Description

**Step 1 — Controller routing (`ContractController.constructServiceParameters()`):** [1](#0-0) 

The guard at line 60 only catches `null` or `""`. The string `"0x0000000000000000000000000000000000000000"` is neither, so execution falls into the `else` branch and `Address.fromHexString(...)` returns `Address.ZERO` — identical to the sentinel value set for a missing `to`.

**Step 2 — Contract-create branch (`TransactionExecutionService.execute()`):** [2](#0-1) 

`params.getReceiver().isZero()` is `true` for both a missing `to` and an explicit zero address, so `buildContractCreateTransactionBody()` is called.

**Step 3 — Caller data used as initcode:** [3](#0-2) 

`params.getCallData()` (the attacker-controlled `data` field) is placed directly into the `initcode` field with no further validation.

**Step 4 — Bypassed `hasTo()` guard (`ContractCallRequest`):** [4](#0-3) 

The guard is intended to reject requests where `value > 0`, `from != null`, `to` is absent, and `data` is not valid initcode. With `to = "0x0000000000000000000000000000000000000000"`, `StringUtils.isNotEmpty(to)` returns `true`, so `isValidToField = true` and `BytecodeUtils.isValidInitBytecode(data)` is never evaluated — even though the downstream code will treat this as a contract-create and use `data` as initcode regardless. [5](#0-4) 

## Impact Explanation

1. **Validation bypass** — the `hasTo()` guard that was specifically designed to require well-formed initcode when no real destination is provided is circumvented entirely by supplying the zero address as a string. Malformed or maximally gas-consuming bytecode can be submitted as initcode.

2. **Semantic divergence from Ethereum JSON-RPC spec** — per the spec, `to = 0x0` is a *call* to the zero address, not a deployment. Clients relying on spec-compliant behaviour receive incorrect results.

3. **Amplified resource consumption** — contract-create simulation uses `CONTRACT_CREATE_TX_FEES = 1_000_000_000L` and runs initcode up to the full gas limit (default 15,000,000). [6](#0-5) 

4. **Blast radius is bounded** — the mirror node is simulation-only; no real Hedera transactions are submitted and no persistent state is mutated. Gas throttling limits overall throughput. [7](#0-6) 

Severity: **Medium**.

## Likelihood Explanation

- The `/api/v1/contracts/call` endpoint requires no authentication. [8](#0-7) 
- A single HTTP POST with `"to": "0x0000000000000000000000000000000000000000"` and arbitrary `"data"` is sufficient to trigger the path.
- The gas throttle bucket partially restores tokens after a failed or reverted simulation, allowing sustained repeated requests. [9](#0-8) 

## Recommendation

1. **Normalise the zero address in the controller**: after parsing `to`, check whether the resulting `Address` is `Address.ZERO` and treat it identically to a missing `to` — or explicitly reject it with a `400 Bad Request` if the intent is to disallow calling the zero address.

2. **Fix `hasTo()` to check the parsed address, not the raw string**: the guard should evaluate whether the resolved receiver is the zero address (i.e., contract-create semantics apply), not merely whether the raw `to` string is non-empty.

3. **Align with Ethereum JSON-RPC semantics**: `to = 0x0` should be routed as a call to the zero address, not as a contract deployment, to match `eth_call`/`eth_estimateGas` spec behaviour.

## Proof of Concept

```http
POST /api/v1/contracts/call HTTP/1.1
Content-Type: application/json

{
  "to":    "0x0000000000000000000000000000000000000000",
  "from":  "0xabcdef1234567890abcdef1234567890abcdef12",
  "data":  "0xdeadbeef",
  "value": 1,
  "gas":   15000000
}
```

**Expected (spec-compliant):** routed as a call to the zero address; `hasTo()` passes because `to` is non-empty; `buildContractCallTransactionBody()` is invoked.

**Actual:** `hasTo()` passes (bypass), `Address.fromHexString("0x000...000")` returns `Address.ZERO`, `isZero()` is `true`, `buildContractCreateTransactionBody()` is invoked with `0xdeadbeef` as initcode — despite `0xdeadbeef` failing `BytecodeUtils.isValidInitBytecode()`.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
