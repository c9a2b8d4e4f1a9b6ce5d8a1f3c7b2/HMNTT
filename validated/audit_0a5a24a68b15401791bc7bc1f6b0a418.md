Audit Report

## Title
Unbounded Subscription Lifetime via Far-Future `endTime` Causing `isComplete()` to Never Return True

## Summary
`ConsensusController.convertTimestamp()` maps any gRPC timestamp with `seconds >= 9223372035` to `Long.MAX_VALUE`. This value is stored as `endTime` in `TopicMessageFilter`. Because `TopicContext.isComplete()` evaluates `Instant.ofEpochSecond(0, Long.MAX_VALUE).plus(endTimeInterval).isBefore(Instant.now())` — a condition that resolves to year ~2262 and is never true — the `RepeatSpec` loop in `pastEndTime()` fires every 30 seconds indefinitely, and `takeUntilOther(pastEndTime(topicContext))` never terminates the subscription.

## Finding Description

**Entry point — `ConsensusController.convertTimestamp()`:** [1](#0-0) 

Any client sending `consensusEndTime.seconds >= 9223372035` receives `endTime = Long.MAX_VALUE` in the resulting `TopicMessageFilter`.

**Validation gap — `TopicMessageFilter.isValidEndTime()`:** [2](#0-1) 

The only constraint is `endTime > startTime`. `Long.MAX_VALUE > startTime` is always true, so the far-future value passes validation without error.

**`isComplete()` never returns true:** [3](#0-2) 

With `endTime = Long.MAX_VALUE` nanoseconds, `Instant.ofEpochSecond(0, Long.MAX_VALUE)` resolves to approximately year 2262. Adding `endTimeInterval` (default 30 s, confirmed in `GrpcProperties`) still yields year 2262. `Instant.now()` never reaches that value, so the method permanently returns `false`. [4](#0-3) 

**Infinite repeat loop in `pastEndTime()`:** [5](#0-4) 

`RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)` fires every `endTimeInterval` (30 s) and never stops because the predicate is always `true`. `Flux.empty().repeatWhen(...)` never emits an element and never completes, so `takeUntilOther(pastEndTime(topicContext))` at line 73 never terminates the subscription. [6](#0-5) 

**Per-connection limit is insufficient:** [7](#0-6) 

`maxConcurrentCallsPerConnection` (default 5) limits streams per TCP connection, but there is no global cap on total concurrent subscriptions. An attacker opens many connections, each with 5 streams, accumulating arbitrarily many zombie subscriptions. The `subscriberCount` gauge reflects the inflation but no enforcement action is tied to it. [8](#0-7) 

## Impact Explanation
Each zombie subscription holds a live Reactor timer entry (firing every 30 s), a gRPC server stream, and in-memory `TopicContext` state. With no global subscription ceiling, an attacker can accumulate thousands of these, exhausting Reactor scheduler timer queues, JVM heap, and Netty connection resources, leading to denial of service for legitimate subscribers.

## Likelihood Explanation
The exploit requires only a standard gRPC client (e.g., `grpcurl` or the Hedera SDK) and the ability to open TCP connections to port 5600. No authentication, no special privileges, and no knowledge of internal state is needed. The trigger value (`seconds >= 9223372035`) is trivially discoverable from the source or by fuzzing. The attack is repeatable and automatable.

## Recommendation
1. **Reject or clamp far-future `endTime` values at the controller layer.** In `convertTimestamp()`, instead of returning `Long.MAX_VALUE`, reject timestamps beyond a reasonable horizon (e.g., 100 years from now) with an `INVALID_ARGUMENT` gRPC status.
2. **Fix `isComplete()` to handle `Long.MAX_VALUE` explicitly.** Add a guard: if `filter.getEndTime() == Long.MAX_VALUE`, treat it as no end time (return `false` and route through `Flux.never()` in `pastEndTime()`), or reject it upstream.
3. **Add a global subscription cap.** Enforce a configurable maximum on `subscriberCount` and reject new subscriptions when the limit is reached.
4. **Add a `maxConcurrentConnections` limit** at the Netty/gRPC server level in addition to the per-connection stream limit.

## Proof of Concept
```bash
# Using grpcurl against the mirror node gRPC endpoint (port 5600)
# Send consensusEndTime.seconds = Long.MAX_VALUE (9223372036854775807)
# which is >= 9223372035, triggering the Long.MAX_VALUE path

for i in $(seq 1 200); do
  grpcurl -plaintext \
    -d '{
      "topicID": {"topicNum": 1},
      "consensusStartTime": {"seconds": 0},
      "consensusEndTime": {"seconds": 9223372036854775807}
    }' \
    localhost:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done

# Each of the 200 background streams:
# - passes isValidEndTime() because Long.MAX_VALUE > startTime
# - causes pastEndTime() to fire a Reactor timer every 30s indefinitely
# - never terminates (isComplete() always returns false)
# - accumulates as a zombie subscription, exhausting scheduler/heap/Netty resources
```

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
