Audit Report

## Title
Cache Key Collision for Alias-Only Accounts Causes Incorrect Account Substitution in `/block` Response

## Summary
In `updateOperationAccountAlias()` in `rosetta/app/services/block_service.go`, the `entityCache` is keyed by `int64` via `accountId.GetId()`. For every alias-only account, `GetId()` returns `0` because no numeric entity ID is assigned. When a block contains two or more distinct alias-only accounts, all share cache key `0`, causing the first alias resolved to be silently substituted for every subsequent alias-only account in the same request.

## Finding Description
**Exact code path:**

`rosetta/app/services/block_service.go`, `updateOperationAccountAlias()`:

```go
if cached, found = s.entityCache.Get(accountId.GetId()); !found {   // line 115
    result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
    ...
    s.entityCache.Set(result.GetId(), result)                        // line 121
    cached = result
}
operations[index].AccountId = cached                                 // line 125
```

The cache is declared as `*cache.Cache[int64, types.AccountId]`, keyed by `int64`. [1](#0-0) 

`GetId()` returns `a.accountId.EncodedId`: [2](#0-1) 

`NewAccountIdFromAlias` constructs `AccountId` with `domain.EntityId{ShardNum: shard, RealmNum: realm}` — `EntityNum` is zero-valued, so `EncodedId` == `0`: [3](#0-2) 

The test suite explicitly confirms this: `ed25519AliasAccountId.GetId()` is expected to be `0`: [4](#0-3) 

In `GetAccountAlias`, the query uses `accountId.GetId()` (= 0) as the lookup key. Since no entity has `id = 0`, `gorm.ErrRecordNotFound` is returned and the original `accountId` is returned unchanged — still with `GetId()` == 0: [5](#0-4) 

**Root cause:** The cache key `int64(0)` is shared by all alias-only accounts. There is no guard that skips caching when `GetId() == 0`, and no richer key (e.g., alias bytes) is used.

## Impact Explanation
The Rosetta `/block` and `/block/transaction` endpoints return wrong `account_identifier` values for alias-only accounts. Any downstream system (exchange, indexer, wallet) consuming the Rosetta API will misattribute balance changes and transfers to the wrong account. Funds credited to `alias_B` appear attributed to `alias_A` in the Rosetta view, causing incorrect balance accounting and potential financial misreporting. [6](#0-5) 

## Likelihood Explanation
The Rosetta API requires no authentication; any external user can call `/block`. Hedera supports auto-account-creation via alias (sending HBAR to a public key alias), so blocks containing multiple alias-only account operations are a normal network occurrence. An attacker only needs to identify (or wait for) a block containing two or more alias-only account operations and issue a standard `/block` request. The bug is deterministic and 100% reproducible once such a block exists. [7](#0-6) 

## Recommendation
In `updateOperationAccountAlias`, skip caching (or use a richer cache key) when the account is alias-only. The simplest fix is to check `accountId.HasAlias()` before the cache lookup: if the account already carries an alias, bypass the cache entirely and call `GetAccountAlias` directly (or skip the lookup altogether, since `GetAccountAlias` returns the original `accountId` unchanged for alias-only inputs). Alternatively, change the cache key type to `string` and use `accountId.String()` as the key, which is unique per alias. [8](#0-7) 

## Proof of Concept
1. Identify (or construct) a block `B` containing two transactions: `Tx1` referencing alias-only account `alias_A` (public key PK1) and `Tx2` referencing alias-only account `alias_B` (public key PK2). Both have `GetId()` == 0.
2. Issue `POST /block` for block `B`.
3. `updateOperationAccountAlias` processes `alias_A` first: cache miss at key `0` → `GetAccountAlias` queries `entity` table with `id=0` → no record found → returns `alias_A` → `entityCache.Set(0, alias_A)`.
4. Processing `alias_B`: cache **hit** at key `0` → returns `alias_A` → `operations[j].AccountId = alias_A`.
5. The response JSON shows `alias_A`'s address for both operations. `alias_B` is never looked up and is silently replaced with `alias_A`. [9](#0-8)

```go
// Block implements the /block endpoint.
func (s *blockAPIService) Block(
	ctx context.Context,
	request *rTypes.BlockRequest,
) (*rTypes.BlockResponse, *rTypes.Error) {
	block, err := s.RetrieveBlock(ctx, request.BlockIdentifier)
	if err != nil {
		return nil, err
	}

	if block.Transactions, err = s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos); err != nil {
		return nil, err
	}

	var otherTransactions []*rTypes.TransactionIdentifier
	if len(block.Transactions) > s.maxTransactionsInBlock {
		otherTransactions = make([]*rTypes.TransactionIdentifier, 0, len(block.Transactions)-s.maxTransactionsInBlock)
		for _, transaction := range block.Transactions[s.maxTransactionsInBlock:] {
			otherTransactions = append(otherTransactions, &rTypes.TransactionIdentifier{Hash: transaction.Hash})
		}
		block.Transactions = block.Transactions[0:s.maxTransactionsInBlock]
	}

	if err = s.updateOperationAccountAlias(ctx, block.Transactions...); err != nil {
		return nil, err
	}

	return &rTypes.BlockResponse{Block: block.ToRosetta(), OtherTransactions: otherTransactions}, nil
}
```

```go
func (s *blockAPIService) updateOperationAccountAlias(
	ctx context.Context,
	transactions ...*types.Transaction,
) *rTypes.Error {
	for _, transaction := range transactions {
		operations := transaction.Operations
		for index := range operations {
			var cached types.AccountId
			var found bool

			accountId := operations[index].AccountId
			if cached, found = s.entityCache.Get(accountId.GetId()); !found {
				result, err := s.accountRepo.GetAccountAlias(ctx, accountId)
				if err != nil {
					return err
				}

				s.entityCache.Set(result.GetId(), result)
				cached = result
			}

			operations[index].AccountId = cached
		}
	}

	return nil
```
