Audit Report

## Title
Zombie Reactive Pipeline Resource Exhaustion via Network Partition (No gRPC Keepalive Configured)

## Summary
`ConsensusController.subscribeTopic()` registers `disposable::dispose` as the sole pipeline teardown mechanism via `setOnCancelHandler`. Because `GrpcConfiguration` configures no HTTP/2 PING-based keepalive on the `NettyServerBuilder`, a network partition leaves TCP connections appearing `ESTABLISHED` on the server indefinitely. The gRPC layer never fires `onCancel`, `disposable::dispose` is never called, and the server-side reactive pipeline — which runs forever when no `endTime` is set — continues consuming scheduler threads and database connections for every zombie subscription.

## Finding Description

**Cancel handler is the sole teardown path.** [1](#0-0) 
`disposable::dispose` is only invoked when the Netty/gRPC transport delivers an `onCancel` event. That event requires a TCP FIN/RST or HTTP/2 RST_STREAM frame, neither of which arrives during a silent network partition.

**No keepalive configured.** [2](#0-1) 
The `ServerBuilderCustomizer<NettyServerBuilder>` sets only `executor` and `maxConcurrentCallsPerConnection`. There is no `keepAliveTime`, `keepAliveTimeout`, or `permitKeepAliveWithoutCalls` call. Without HTTP/2 PING probes the server has no out-of-band mechanism to detect a dead peer.

**Pipeline runs forever without `endTime`.** [3](#0-2) 
The live `Flux` is terminated only by `takeUntilOther(pastEndTime(topicContext))`. [4](#0-3) 
`pastEndTime` returns `Flux.never()` when `filter.getEndTime() == null`, so there is no server-side timeout on the live phase. `endTime` and `limit` are both optional in the protocol: [5](#0-4) 

**`subscriberCount` never decrements for zombie connections.**
`doFinally` (which decrements `subscriberCount`) only fires when the pipeline terminates via a signal — which never happens for a zombie subscription: [6](#0-5) 

**Exploit flow:**
1. Attacker opens N TCP connections to the gRPC port (no authentication required).
2. Each connection issues up to `maxConcurrentCallsPerConnection` (default 5) `subscribeTopic` calls with no `endTime` and no `limit`.
3. Attacker uses `iptables -A OUTPUT -p tcp --dport <grpc_port> -j DROP` to silently drop outgoing packets — no FIN/RST is sent.
4. Server sees all connections as `ESTABLISHED`. Netty never fires `onCancel`. `disposable::dispose` is never called.
5. Each zombie subscription keeps the live `Flux` alive, issuing periodic `topicMessageRepository.findByFilter()` queries against the database indefinitely.
6. `subscriberCount` grows monotonically; `boundedElastic` scheduler threads are consumed; DB connection pool is exhausted.

**Why existing mitigations fail:**
- `setOnCancelHandler` — correct pattern, but entirely dependent on the transport delivering the cancellation signal, which a partition prevents.
- `maxConcurrentCallsPerConnection = 5` — limits calls per connection, but there is no limit on the number of connections, so N connections × 5 = 5N zombie pipelines.
- `retrieverProperties.getTimeout()` — applies only to the historical retrieval phase, not to the live `topicListener.listen()` phase.
- The one-shot safety check (`Mono.delay(1s)`) fires once and does not terminate the pipeline.

## Impact Explanation
Each zombie subscription holds a `boundedElastic` scheduler thread slot and issues periodic SQL queries. With enough connections the database connection pool is exhausted, causing all legitimate subscribers and internal importer operations to fail. The `subscriberCount` metric grows monotonically, masking the true subscriber count. Severity is **High** (availability impact, no authentication required, no rate limiting on connections).

## Likelihood Explanation
Any unprivileged client with network access to the gRPC port can trigger this. The partition can be simulated locally with `iptables` or by killing the network interface without a graceful TCP close. No special knowledge of the application is needed beyond knowing the gRPC endpoint. Default OS TCP keepalive timers (typically 2 hours) mean zombie connections persist for hours without intervention. The attack is repeatable and scalable.

## Recommendation
1. **Enable HTTP/2 PING-based keepalive** in `GrpcConfiguration.java`:
   ```java
   serverBuilder.keepAliveTime(30, TimeUnit.SECONDS);
   serverBuilder.keepAliveTimeout(10, TimeUnit.SECONDS);
   serverBuilder.permitKeepAliveWithoutCalls(true);
   ```
   This causes Netty to send HTTP/2 PING frames and close the connection (firing `onCancel`) if the peer does not respond within the timeout.

2. **Add a server-side maximum subscription duration** in `TopicMessageServiceImpl.subscribeTopic()` as a defense-in-depth measure, e.g. `flux.timeout(maxSubscriptionDuration)`, so pipelines are forcibly terminated even if the transport-level detection fails.

3. **Limit the number of concurrent connections** at the load balancer or via a Netty `ChannelHandler` to bound the total number of zombie pipelines an attacker can create.

## Proof of Concept
```bash
# 1. Open a gRPC stream with no endTime and no limit
grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    <host>:<grpc_port> \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &

# 2. Silently drop outgoing packets to simulate a partition
iptables -A OUTPUT -p tcp --dport <grpc_port> -j DROP

# 3. Observe on the server: subscriberCount metric never decrements,
#    DB query rate remains constant, connection stays ESTABLISHED
watch -n5 'ss -tnp | grep <grpc_port>'
# Expected: connection remains ESTABLISHED indefinitely;
# DB connection pool exhausted after scaling to N connections × 5 calls
```
