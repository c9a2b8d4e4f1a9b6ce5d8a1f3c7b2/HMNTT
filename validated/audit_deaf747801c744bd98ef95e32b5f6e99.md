Audit Report

## Title
Permanent Cache Poisoning of `CACHE_MANAGER_RECORD_FILE_EARLIEST` via `Optional.empty()` Caching on Empty DB

## Summary
`RecordFileRepository.findEarliest()` uses `@Cacheable` with `unless = "#result == null"`, but returns `Optional<RecordFile>`. When the `record_file` table is empty, the method returns `Optional.empty()` — a non-null object — causing the `unless` guard to evaluate to `false` and the empty result to be permanently cached. The backing Caffeine cache has `maximumSize(1)` and **no TTL**, so the poisoned entry persists for the entire process lifetime with no eviction path.

## Finding Description

**Affected files:**

- `web3/src/main/java/org/hiero/mirror/web3/evm/config/EvmConfiguration.java`, lines 190–197: `cacheManagerRecordFileEarliest` configured with `maximumSize(1)` and no `expireAfterWrite`/`expireAfterAccess`. [1](#0-0) 

- `web3/src/main/java/org/hiero/mirror/web3/repository/RecordFileRepository.java`, lines 23–25: `@Cacheable(... unless = "#result == null")` on `Optional<RecordFile> findEarliest()`. [2](#0-1) 

**Root cause:** Spring's `@Cacheable` `unless` SpEL expression evaluates `#result` against the actual Java return value. When the DB has no rows, Spring Data JPA returns `Optional.empty()` — a non-null object. `Optional.empty() == null` is `false`, so the `unless` condition does not suppress caching. Spring's `CacheAspectSupport` then unwraps the `Optional`, stores `NullValue.INSTANCE` in Caffeine, and all subsequent calls return `Optional.empty()` from cache without querying the DB. [2](#0-1) 

**Why existing checks fail:**
- `unless = "#result == null"` is the only guard and is semantically incorrect for `Optional`-returning methods. The correct guard is `unless = "#result == null || !#result.isPresent()"`.
- There is no `@CacheEvict` annotation anywhere in the codebase targeting `CACHE_MANAGER_RECORD_FILE_EARLIEST` — confirmed by grep returning zero matches.
- No TTL is set; `maximumSize(1)` with a single fixed key means the poisoned entry is never displaced by size-based eviction. [3](#0-2) 

## Impact Explanation
Once poisoned, `eth_getBlockByNumber('earliest')` returns "block not found" for the entire process lifetime. `RecordFileServiceImpl.findByBlockType(BlockType.EARLIEST)` calls `recordFileRepository.findEarliest()` directly with no additional guard, so the cached `Optional.empty()` propagates immediately to all callers. [4](#0-3) 

Any downstream logic using the earliest block as a lower-bound anchor — including `eth_getLogs` range validation, historical state queries, and block range iteration — will fail or return incorrect results. Recovery requires a full process restart. The impact is a persistent, operator-invisible DoS on the genesis block anchor affecting all clients of the node.

## Likelihood Explanation
The precondition (empty `record_file` table) exists on every fresh deployment and every process restart before the importer writes the genesis record file. No authentication is required to call `eth_getBlockByNumber`. An attacker who monitors for node restarts (observable via connection resets or latency spikes) can reliably trigger this with a single unauthenticated HTTP request. The window may be seconds to minutes depending on importer lag. The attack is repeatable on every restart.

## Recommendation
Fix the `unless` condition on `findEarliest()` to correctly handle `Optional` return types:

```java
@Cacheable(
    cacheNames = CACHE_NAME,
    cacheManager = CACHE_MANAGER_RECORD_FILE_EARLIEST,
    unless = "#result == null || !#result.isPresent()")
Optional<RecordFile> findEarliest();
```

Additionally, consider adding a short TTL (e.g., `expireAfterWrite(30, TimeUnit.SECONDS)`) to `cacheManagerRecordFileEarliest` as a defense-in-depth measure, consistent with how `cacheManagerRecordFileLatest` is configured with `expireAfterWrite(500, TimeUnit.MILLISECONDS)`. [5](#0-4) 

The same `unless = "#result == null"` pattern on `findByIndex()` and `findByTimestamp()` should also be reviewed, though those caches have TTLs that limit the blast radius. [6](#0-5) 

## Proof of Concept
1. Start the web3 service against a fresh database (empty `record_file` table).
2. Before the importer writes the genesis record file, send:
   ```
   POST /
   {"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["earliest",false],"id":1}
   ```
3. The DB query returns no rows → `Optional.empty()` → `unless` guard passes → `NullValue.INSTANCE` stored in Caffeine cache.
4. Wait for the importer to write the genesis record file to the DB.
5. Send the same request again. Despite the DB now containing the genesis block, the response is still "block not found" — the cache is never invalidated and the DB is never queried again.
6. The node must be restarted to recover.
