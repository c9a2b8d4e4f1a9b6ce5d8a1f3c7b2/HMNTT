Audit Report

## Title
Unbounded Concurrent `subscribeTopic()` Streams Exhaust Shared `Schedulers.boundedElastic()` Thread Pool via `PollingTopicMessageRetriever`

## Summary
`PollingTopicMessageRetriever` stores a reference to the global shared `Schedulers.boundedElastic()` instance and uses it to schedule blocking JDBC poll tasks for every active subscriber. No global subscriber cap is enforced anywhere in the call chain. An unauthenticated attacker opening arbitrarily many concurrent `subscribeTopic()` gRPC streams — across multiple connections to bypass the per-connection limit — can saturate the shared thread pool, stalling all polling, timeout signaling, and safety-check recovery for legitimate users.

## Finding Description

**`PollingTopicMessageRetriever`** (`grpc/src/main/java/org/hiero/mirror/grpc/retriever/PollingTopicMessageRetriever.java`)

At construction time, the retriever captures the global shared `boundedElastic` scheduler: [1](#0-0) 

When `throttled=true`, `PollingContext` unconditionally sets `numRepeats = Long.MAX_VALUE` with a 2-second polling frequency: [2](#0-1) 

The `repeatWhen` operator dispatches re-subscriptions (and thus `poll()` invocations) on that shared scheduler: [3](#0-2) 

`poll()` calls `topicMessageRepository.findByFilter()` — a blocking JDBC `Stream` — directly on the scheduler thread, holding it for the duration of the DB query: [4](#0-3) 

The `.timeout()` call also uses the same shared scheduler, so timeout signals are queued behind the saturated pool: [5](#0-4) 

**`TopicMessageServiceImpl`** (`grpc/src/main/java/org/hiero/mirror/grpc/service/TopicMessageServiceImpl.java`)

Every new subscriber immediately triggers `retrieve(filter, true)` (throttled=true → `numRepeats=Long.MAX_VALUE`): [6](#0-5) 

`subscriberCount` is registered only as a Micrometer gauge metric — it is never read back to enforce a cap: [7](#0-6) 

The safety-check path also schedules work on the same global `boundedElastic` pool: [8](#0-7) 

**Partial mitigation that does not prevent the attack:**

`GrpcConfiguration` sets `maxConcurrentCallsPerConnection = 5` (default from `NettyProperties`): [9](#0-8) [10](#0-9) 

This is a **per-connection** limit only. An attacker opens N TCP connections, each carrying 5 streams, yielding 5N concurrent scheduler-blocking tasks. No global connection count limit or global stream count limit is configured.

## Impact Explanation
The global `Schedulers.boundedElastic()` pool (default: `10 × availableProcessors` threads, e.g. 80 on an 8-core host) is the single thread pool backing all polling, timeout signaling, and safety-check subscriptions for the entire gRPC process. Once all threads are occupied with attacker-controlled blocking DB calls:

- Legitimate subscribers stop receiving messages — their poll tasks queue indefinitely.
- Timeout signals are delayed or never fired (they are scheduled on the same saturated pool), keeping zombie subscriptions alive and compounding exhaustion.
- The safety-check gap-recovery path (`subscribeOn(Schedulers.boundedElastic())`) also stalls.
- When the internal task queue (default 100,000 entries) fills, new tasks are rejected with `RejectedExecutionException`, breaking all scheduler-dependent operations.

## Likelihood Explanation
- **No authentication**: `ConsensusController.subscribeTopic()` processes requests with no auth check. [11](#0-10) 
- **No global subscriber or connection cap**: only a per-connection limit of 5 streams, trivially bypassed by opening multiple TCP connections.
- **Low attacker cost**: each stream is a single gRPC streaming RPC with a minimal filter (`startTime=0`, no `endTime`, no `limit`). No data needs to be sent after the initial request.
- **Sustained attack**: the attacker reopens streams as old ones time out (every 60 s per `RetrieverProperties.timeout` default), maintaining saturation indefinitely. [12](#0-11) 

## Recommendation
1. **Enforce a global subscriber cap** in `TopicMessageServiceImpl.subscribeTopic()`: read `subscriberCount` before incrementing and return a gRPC `RESOURCE_EXHAUSTED` error if it exceeds a configurable maximum.
2. **Move blocking DB work off the shared scheduler**: use a dedicated, bounded `Schedulers.newBoundedElastic(...)` instance in `PollingTopicMessageRetriever` with a thread cap and queue depth sized for expected load, isolating retriever threads from timeout and safety-check scheduling.
3. **Add a global concurrent-stream limit** at the Netty server level (e.g., `NettyServerBuilder.maxConnectionAge` / a global call limiter interceptor) in addition to the existing per-connection limit.
4. **Consider non-blocking DB access** (e.g., R2DBC) for `findByFilter` to avoid holding scheduler threads during DB I/O.

## Proof of Concept

```python
import grpc
import threading
from com.hedera.mirror.api.proto import consensus_service_pb2, consensus_service_pb2_grpc
from proto.com.hedera.hashgraph.sdk.proto import basic_types_pb2

TARGET = "mirror-node-host:5600"
CONNECTIONS = 20   # 20 TCP connections
STREAMS_PER_CONN = 5  # maxConcurrentCallsPerConnection default

def flood(conn_id):
    channel = grpc.insecure_channel(TARGET)
    stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=1),
        # no endTime, no limit → throttled=true, numRepeats=Long.MAX_VALUE
    )
    streams = [stub.subscribeTopic(query) for _ in range(STREAMS_PER_CONN)]
    for s in streams:
        try:
            for _ in s:  # drain indefinitely
                pass
        except Exception:
            pass

threads = [threading.Thread(target=flood, args=(i,)) for i in range(CONNECTIONS)]
for t in threads: t.start()
for t in threads: t.join()
# Result: 100 concurrent blocking JDBC polls on the shared boundedElastic pool
# → legitimate subscribers stall, timeouts delayed, safety checks stall
```
