# Audit Report

## Title
Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode()` Causes Persistent Bytecode Blackout

## Summary
`ContractRepository.findRuntimeBytecode()` is annotated with `@Cacheable(unless = "#result == null")`, but the method returns `Optional<byte[]>`. When no bytecode row exists in the database, the method returns `Optional.empty()` — a non-null Java object — so the `unless` guard does not fire and the empty result is written into the Caffeine cache with a TTL of `expireAfterAccess=1h`. An attacker who queries a contract ID in the brief window before the importer indexes it permanently poisons that instance's cache, causing all subsequent `eth_getCode`/`eth_call` requests for that contract to return empty bytecode for the full cache TTL.

## Finding Description
**Exact code path:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18:
```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [1](#0-0) 

**Root cause:** Spring's `@Cacheable` evaluates the `unless` SpEL expression against the actual return value. The return type is `Optional<byte[]>`. When the DB query finds no row, Spring Data JPA wraps the result as `Optional.empty()`. `Optional.empty()` is a non-null singleton, so `#result == null` evaluates to `false`, and the empty `Optional` is stored in the Caffeine cache. [2](#0-1) 

**Cache configuration:** `CACHE_MANAGER_CONTRACT` is a `CaffeineCacheManager` configured with `expireAfterAccess=1h,maximumSize=1000,recordStats` — a JVM-local, in-process cache with a 1-hour access-based TTL. [3](#0-2) [4](#0-3) 

**No invalidation path:** A `grep` for `@CacheEvict` against `CACHE_NAME_CONTRACT` returns zero results — there is no eviction or invalidation mechanism anywhere in the codebase.

**Downstream impact in `ContractBytecodeReadableKVState`:** The cached `Optional.empty()` is consumed at lines 42–46, where `.orElse(null)` maps it to `null`, causing the EVM to treat the address as having no code. [5](#0-4) 

## Impact Explanation
Any valid deployed contract can be made to appear as a zero-bytecode (non-existent) account on a poisoned instance for up to 1 hour (or longer, since `expireAfterAccess` resets on each read):
- `eth_getCode` returns `0x` for a live contract.
- `eth_call` and `eth_estimateGas` fail or return incorrect results because the EVM treats the address as an EOA.
- DApps and tooling relying on the mirror node's web3 API malfunction for that contract.
- In a multi-instance deployment, only the poisoned instance is affected, creating inconsistent responses across the cluster.

Severity: **Medium** — targeted, persistent DoS on specific contracts; no data exfiltration or privilege escalation.

## Likelihood Explanation
Hedera network transactions are publicly observable in near-real-time via the mirror REST API and gRPC streams. The window between on-chain `ContractCreate` finality and importer indexing is typically a few seconds. An attacker with a script monitoring for new contract entity IDs can reliably race the importer with a single HTTP request to the public web3 endpoint. No credentials or special access are required. The attack is repeatable for every new contract deployment.

## Recommendation
Change the `unless` condition to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

This ensures that only a populated `Optional` (i.e., a row with actual bytecode) is written into the cache. Optionally, consider also reducing the `expireAfterAccess` TTL or switching to `expireAfterWrite` to bound the staleness window for legitimate cache entries.

## Proof of Concept
1. Monitor the Hedera mirror REST API (`/api/v1/transactions?transactiontype=CONTRACTCREATEINSTANCE`) for a new `ContractCreate` transaction and extract the assigned contract entity ID (e.g., `0.0.12345`).
2. Before the importer processes the record, send:
   ```
   POST /api/v1/contracts/call
   { "to": "0x000000000000000000000000000000000000303d", "data": "0x" }
   ```
   or an `eth_getCode` JSON-RPC call targeting the long-zero EVM address for entity `12345`.
3. The DB has no `runtime_bytecode` row yet; `findRuntimeBytecode(12345L)` returns `Optional.empty()`, which is cached.
4. Wait for the importer to index the contract (a few seconds).
5. Repeat the same `eth_getCode` call — the response is still `0x` (empty bytecode) because the cache serves the poisoned `Optional.empty()`.
6. The poisoned state persists for up to 1 hour (or indefinitely if the entry is accessed repeatedly, due to `expireAfterAccess` semantics).
