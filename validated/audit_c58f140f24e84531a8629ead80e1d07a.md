## Audit Report

## Title
Unauthenticated Per-Request DB Query Flood via Alias Resolution in `getTokenRelationships()`

## Summary
The `/api/v1/accounts/{alias}/tokens` endpoint resolves the account identifier on every cache-miss request by issuing two uncached database queries with no rate limiting applied to unauthenticated callers. An attacker can flood the endpoint with syntactically valid but distinct account aliases, causing each concurrent request to consume DB pool connections and denying service to all other API consumers.

## Finding Description

`getTokenRelationships()` unconditionally calls two `EntityService` methods per request: [1](#0-0) 

When the identifier is an alias, `getEncodedId()` routes to `getAccountIdFromAlias()` → `getAccountFromAlias()`, which issues a raw SQL query against the `entity` table with no caching: [2](#0-1) [3](#0-2) 

`isValidAccount()` then issues a second independent SQL query: [4](#0-3) 

Both queries go directly to the DB pool with no intermediate cache layer: [5](#0-4) 

A response-level Redis cache exists (`responseCacheHandler.js`) keyed by `md5(req.originalUrl)`, but it only serves cache hits for previously seen identical URLs. An attacker using distinct alias values per request — or any alias not yet cached — produces a cache miss on every request, triggering the full two-query DB path each time. [6](#0-5) 

The `authHandler` middleware simply returns (allows the request through) when no `Authorization` header is present, imposing zero restriction on unauthenticated callers: [7](#0-6) 

No rate-limiting or IP-throttling middleware exists anywhere in the production middleware stack:


Unlike `transactions.js` which uses an in-process `cache.get()` to batch and deduplicate DB lookups, `EntityService` has no equivalent caching mechanism: [8](#0-7) 

## Impact Explanation
An attacker with no credentials can exhaust the PostgreSQL connection pool by sending a high volume of concurrent requests to `/api/v1/accounts/{alias}/tokens`, each with a distinct alias value. Each request consumes two sequential DB pool checkouts for alias resolution and existence validation. Once the pool is saturated, all other API endpoints requiring DB access (transactions, balances, contracts, etc.) will queue or fail, constituting a full denial of service for all API consumers.

## Likelihood Explanation
The attack requires no authentication, no special knowledge, and no on-chain funds. The alias format (`AccountAlias.isValid()`) is publicly documented. An attacker can generate an arbitrary number of syntactically valid alias strings and issue them with standard HTTP load tools (`wrk`, `ab`, `hey`). Even non-existent aliases still consume a full DB round-trip before returning `null`. The response cache provides no protection when alias values are varied across requests.

## Recommendation
1. **Rate limiting**: Add an IP-based rate-limiting middleware (e.g., `express-rate-limit`) to the middleware stack, applied before route handlers, to cap unauthenticated request volume per IP per time window.
2. **Alias resolution caching**: Introduce a short-lived in-process or Redis-backed cache in `EntityService.getAccountFromAlias()` keyed on the alias string, similar to the `cache.get()` pattern already used in `transactions.js`.
3. **Connection pool protection**: Configure a query timeout and a maximum pool wait time so that pool exhaustion causes fast-fail errors rather than indefinite queuing.
