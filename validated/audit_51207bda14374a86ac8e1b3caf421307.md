Audit Report

## Title
Unsynchronized Torn Read of `genesisBlock` Struct in `blockRepository.initGenesisRecordFile`

## Summary
`blockRepository` uses `sync.Once` to write `br.genesisBlock` but performs a plain, unsynchronized read of `br.genesisBlock.ConsensusStart` at line 244 as an early-exit guard. A concurrent goroutine that observes the partially-written `ConsensusStart` field bypasses `once.Do` entirely and proceeds to use `br.genesisBlock.Index` and `br.genesisBlock.Hash` while they still hold zero/nil values. This is a confirmed data race under Go's memory model.

## Finding Description

The `initGenesisRecordFile` function contains two conflicting memory accesses with no synchronization between them:

**Unsynchronized read (line 244):** [1](#0-0) 

**Struct write inside `once.Do` (lines 257–259):** [2](#0-1) 

The `recordBlock` struct is a multi-field value type: [3](#0-2) 

The assignment `br.genesisBlock = rb` compiles to multiple sequential store instructions — at minimum one per field, with `string` requiring two (data pointer + length). `sync.Once` provides a happens-before guarantee **only** for goroutines that themselves call `once.Do`. Goroutine B, which exits at line 245 after observing `ConsensusStart != -1`, never enters `once.Do` and therefore has **no happens-before guarantee** for `Index`, `Hash`, `ConsensusEnd`, or `PrevHash`.

**Race sequence:**
1. Goroutine A passes the line-244 check (sees `-1`), queries the DB, enters `once.Do`, and begins writing `br.genesisBlock = rb`. The compiler emits stores in field order; `ConsensusStart` (offset 0) is stored first.
2. Goroutine B reads `br.genesisBlock.ConsensusStart` at line 244 concurrently, observes the new non-`-1` value, and returns `nil` — genesis is "initialized."
3. Goroutine B proceeds to call `findBlockByHash` or `findBlockByIndex`, which read `br.genesisBlock.Index` (still `0`) and `br.genesisBlock.Hash` (still `""` or a torn fat-pointer).

The Go race detector will flag the concurrent read at line 244 and write at line 258 as a **data race** (undefined behavior per the Go memory model).

## Impact Explanation

Three security-relevant call sites consume the torn `br.genesisBlock`:

**`findBlockByIndex` — line 211:** [4](#0-3) 
With `br.genesisBlock.Index == 0` (torn), the guard `index < 0` is effectively the only check, allowing any non-negative index — including pre-genesis blocks — to be served.

**`findBlockByHash` — line 235:** [5](#0-4) 
Same bypass: pre-genesis blocks whose `Index >= 0` pass the check and are returned as valid.

**`ToBlock` — line 102:** [6](#0-5) 
With `genesisBlock.Index == 0`, any block at index 0 is misidentified as genesis, receiving wrong `ConsensusStart`, `parentHash`, and `parentIndex`.

Additionally, `Hash` is a Go `string` (fat pointer: 8-byte data pointer + 8-byte length). A torn read — e.g., new data pointer with old length `0`, or a null pointer with a non-zero length — causes an **immediate panic** (nil dereference or out-of-bounds slice access) when the string is used, crashing the Rosetta API process.

## Likelihood Explanation

- **Precondition:** The server has just started and `genesisBlock` has not yet been initialized — the normal state on every cold start.
- **Attacker capability:** Any unauthenticated HTTP client can send concurrent Rosetta API requests (`/block`, `/block/transaction`). No credentials or special privileges are required.
- **Feasibility:** On a multi-core host the race window is real. An attacker sends a burst of concurrent requests immediately after the node becomes reachable. Modern CPUs with store-buffer visibility make the partial write observable across cores. The Go race detector confirms this as a data race.
- **Repeatability:** The window exists once per process lifetime, but a crash caused by the torn `Hash` panic resets the state on restart, making the attack repeatable.

## Recommendation

Replace the `sync.Once`-plus-unsynchronized-read pattern with a `sync.RWMutex` or an atomic pointer:

**Option A — `sync.RWMutex`:** Protect all reads of `br.genesisBlock` (including the early-exit check at line 244) and the write at line 258 with a `RWMutex`. Readers acquire `RLock`; the writer acquires `Lock`.

**Option B — atomic pointer:** Store `genesisBlock` as an `atomic.Pointer[recordBlock]`. Initialize it to a sentinel value. The early-exit check becomes an atomic load; the write becomes an atomic store. This eliminates the torn-read risk entirely.

Either option must also protect the downstream reads of `br.genesisBlock` in `findBlockByIndex` (line 211), `findBlockByHash` (line 235), `RetrieveLatest` (line 203), and `ToBlock` (line 102). [7](#0-6)

func TestGenesisRace(t *testing.T) {

}
