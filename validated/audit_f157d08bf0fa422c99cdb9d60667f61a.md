Audit Report

## Title
Unauthenticated Access to Prometheus Metrics Endpoint via Falsy `authentication` Config

## Summary
In `rest/middleware/metricsHandler.js`, the `authenticate()` function returns `true` (grants access) whenever `config.metrics.config.authentication` is falsy — including `undefined`. Because no default config file (`rest/config/application.yml`) exists in the repository, any deployment that does not explicitly set `authentication: true` in a custom config silently disables authentication, allowing any unauthenticated caller to retrieve the full Prometheus metrics payload.

## Finding Description

**Exact location:** `rest/middleware/metricsHandler.js`, `authenticate()`, lines 137–144.

```js
const authenticate = (req) => {
  const {authentication, username, password} = config.metrics.config;
  if (!authentication) {   // true for undefined, null, 0, "", false
    return true;
  }
  const credentials = basicAuth(req);
  return credentials && tsscmp(credentials.name, username) && tsscmp(credentials.pass, password);
};
``` [1](#0-0) 

The guard at line 157 is `if (!authenticate(req))` — when `authenticate()` returns `true`, the request proceeds directly to `exporter.collect()` and the full Prometheus payload is serialized and returned. [2](#0-1) 

**Root cause:** The `!authentication` branch conflates two distinct states:
- (a) operator explicitly disabled auth (`authentication: false`) — intentional bypass
- (b) property was never set (`authentication: undefined`) — unintentional bypass

Both produce `!authentication === true`, both return `true`, and both grant unauthenticated access.

**Missing default config:** The documentation declares the default as `true`: [3](#0-2) 

However, `rest/config/application.yml` does not exist in the repository (confirmed: glob search for `**/application.yml` returns no results). The config loader at `rest/config.js` silently skips missing files: [4](#0-3) 

The documented default of `true` is therefore never loaded into the running config. Any operator who supplies a minimal config (e.g., only DB credentials) without explicitly setting `authentication: true` will have `config.metrics.config.authentication === undefined`.

**Metrics handler registration:** The handler is registered unconditionally when `config.metrics.enabled` is truthy (default: `true` per docs): [5](#0-4) 

**Exploit flow:**
1. Attacker identifies the metrics path. Per documented config defaults, `uriPath = '/swagger'`, so `metricsPath = '/swagger/metrics/'`. [6](#0-5) 
2. Attacker sends `GET /swagger/metrics/` with no `Authorization` header.
3. `authenticate(req)` reads `authentication` as `undefined` → `!undefined` → `true` → returns `true`.
4. The `if (!authenticate(req))` guard is skipped; `exporter.collect()` is called and the full Prometheus payload is returned.

**Why existing checks fail:** There is no secondary guard, no IP allowlist, and no network-layer restriction in this code path. The sole protection is `authenticate()`, which is bypassed entirely when the config property is absent.

## Impact Explanation
The metrics payload exposes:
- `api_request{method, path, code}` — per-route request counts, revealing which API paths exist and their traffic volumes
- `api_request_duration_milliseconds` — per-route latency histograms, enabling timing-based inference of backend behavior
- `api_all_errors`, `api_all_server_error`, `api_all_client_error` — error rates revealing instability windows
- `nodejs_process_cpu_usage_percentage`, `nodejs_process_memory_*` — process-level resource telemetry [7](#0-6) 

An attacker can use this data to identify under-monitored or high-error routes, infer deployment topology, and time attacks to coincide with high-load or high-error windows. **Severity: Medium** (information disclosure with no direct data exfiltration, but meaningful reconnaissance value).

## Likelihood Explanation
- **Precondition:** No default config file exists in the repository. Any deployment that does not explicitly supply `authentication: true` in a custom config is vulnerable by default.
- **Attacker capability:** Zero — no credentials, no special network position, no prior knowledge beyond the default `uriPath` value (`/swagger`) which is publicly documented.
- **Repeatability:** Fully repeatable; the endpoint is stateless and returns fresh metrics on every request.
- **Feasibility:** A single unauthenticated HTTP GET suffices.

## Recommendation
1. **Fix the logic:** Replace `if (!authentication)` with `if (authentication === false)` so that only an explicit `false` disables auth, while `undefined` (absent config) falls through to credential checking.
2. **Add a default config file:** Create `rest/config/application.yml` with `hiero.mirror.rest.metrics.config.authentication: true` (and default `username`/`password`) so the documented defaults are actually enforced at runtime.
3. **Fail-closed:** Consider defaulting `authentication` to `true` in code (e.g., `const authentication = config.metrics.config.authentication ?? true`) rather than relying solely on a config file.

## Proof of Concept
