# Audit Report

## Title
Cache TTL Mismatch Enables Low-Rate DB Amplification via Batch Slot Query

## Summary
A TTL mismatch between `contractStateCache` (2-second `expireAfterWrite`) and `slotsPerContract` (5-minute `expireAfterAccess`, up to 1,500 keys) in `ContractStateServiceImpl` allows an unprivileged attacker to trigger a full 1,500-slot batch DB query every 2 seconds using a single HTTP request — a 1,500× DB amplification factor at 0.5 RPS, well below the global 500 RPS rate limit.

## Finding Description

**Code path — `findStorage()` (lines 58–70):**

```java
final var cachedValue = contractStateCache.get(generateCacheKey(contractId, key), byte[].class);
if (cachedValue != null && cachedValue != EMPTY_VALUE) {
    return Optional.of(cachedValue);
}
return findStorageBatch(contractId, key);
```

Any miss on `contractStateCache` unconditionally invokes `findStorageBatch()`. [6](#0-5) 

**Code path — `findStorageBatch()` (lines 85–103):**

```java
contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();
// ... builds cachedSlots list from ALL keys ...
final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
```

Every invocation reads **all** slot keys currently in the per-contract `slotsPerContract` cache and issues a single `SELECT ... WHERE slot IN (all_cached_slots)` query. [7](#0-6) 

Results are then written back to `contractStateCache` (line 109), all sharing the same write timestamp and expiring together ~2 seconds later. [8](#0-7) 

**Root cause — TTL mismatch:**

| Cache | Config | Effect |
|---|---|---|
| `contractState` | `expireAfterWrite=2s,maximumSize=25000` | All slot values for a contract expire together every 2 seconds |
| `slotsPerContract` | `expireAfterAccess=5m,maximumSize=1500` | Slot keys survive 5 minutes per access | [9](#0-8) 

After `findStorageBatch()` populates `contractStateCache`, all entries expire together at T+2s. The next request after that window finds `contractStateCache` cold while `slotsPerContract` still holds all 1,500 keys, triggering another full batch query.

**Why existing checks fail:**

The throttle is a single global token bucket at 500 RPS with no per-IP, per-contract, or per-batch-size dimension:

```java
if (!rateLimitBucket.tryConsume(1)) {
    throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
}
``` [10](#0-9) 

At 0.5 RPS the attacker consumes 0.1% of the global budget. The gas throttle applies only to EVM execution gas, not to storage slot query volume. [11](#0-10) 

## Impact Explanation

Each attacker request at 0.5 RPS causes a DB query scanning up to 1,500 rows. With N pre-loaded contracts, N × 0.5 RPS generates N × 1,500 DB row reads per second. For example, 20 contracts → 10 RPS → 30,000 DB row reads/second, sustained indefinitely. This can saturate DB I/O, increase query latency for all users, and degrade overall node throughput.

An additional amplification vector exists for non-existent slot keys: since `findStorageBatch()` only writes to `contractStateCache` for slots returned by the DB (line 109), a non-existent slot key will **never** populate `contractStateCache`, meaning every single request for that key bypasses the 2-second TTL entirely and always triggers the full batch query immediately. [12](#0-11) 

## Likelihood Explanation

The attack requires no credentials, no special network position, and no brute force. The setup phase (1,500 requests) is a one-time cost executable with any standard Ethereum JSON-RPC client. The sustain phase is a single HTTP call every 2 seconds per contract. The attack is fully repeatable and automatable with a simple script.

## Recommendation

1. **Cap batch size**: Enforce a hard upper bound on the number of slot keys passed to `findStorageBatch()` (e.g., 50–100) regardless of how many keys are in `slotsPerContract`.
2. **Align TTLs**: Set `slotsPerContract` TTL to be equal to or shorter than `contractState` TTL, or switch `contractState` to `expireAfterAccess` so that actively-used entries are not mass-evicted simultaneously.
3. **Per-contract rate limiting**: Add a per-contract (or per-IP) rate limit on `findStorageBatch()` invocations to prevent a single actor from repeatedly triggering full batch queries.
4. **Negative caching**: Store a sentinel value in `contractStateCache` for non-existent slots so that repeated lookups for missing keys do not bypass the cache entirely.

## Proof of Concept

```python
import requests, time

NODE_URL = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract>"

# Phase 1: Populate slotsPerContract cache with 1500 distinct slot keys
for i in range(1500):
    slot = hex(i).zfill(66)  # distinct slot keys
    requests.post(NODE_URL, json={
        "to": CONTRACT,
        "data": f"0x{slot[2:]}",  # eth_getStorageAt equivalent
        "block": "latest"
    })

# Phase 2: Every 2 seconds, one request triggers a 1500-row batch DB query
while True:
    requests.post(NODE_URL, json={
        "to": CONTRACT,
        "data": "0x" + "00" * 32,  # any slot key
        "block": "latest"
    })
    time.sleep(2)
    # Each iteration: contractStateCache miss -> findStorageBatch() ->
    # SELECT ... WHERE slot IN (1500 keys) against the DB
```

            }
        }
