Audit Report

## Title
Integer Overflow in `PollingTopicListener.poll()` Causes Unbounded Database Queries via `Long.MAX_VALUE` Limit

## Summary
An unauthenticated gRPC client can send `limit = Long.MAX_VALUE` in a `ConsensusTopicQuery`. This value propagates through the service layer into `PollingTopicListener.poll()`, where a narrowing cast `(int)(Long.MAX_VALUE)` produces `-1`. The resulting filter with `limit = -1` bypasses the `hasLimit()` guard in `TopicMessageRepositoryCustomImpl.findByFilter()`, issuing unbounded `SELECT *` queries against the database on every poll cycle for the lifetime of the subscription.

## Finding Description

**Step 1 — User input reaches the filter without an upper-bound check.**

`ConsensusController.toFilter()` passes `query.getLimit()` directly to the builder with no upper-bound cap:

```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

`TopicMessageFilter.limit` carries only `@Min(0)`. `Long.MAX_VALUE > 0` passes. `hasLimit()` returns `true` since `Long.MAX_VALUE > 0`: [2](#0-1) 

**Step 2 — The limit is forwarded unchanged into the listener.**

In `TopicMessageServiceImpl.incomingMessages()`, when `topicContext.getCount().get()` is 0:

```java
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
...
var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();
return topicListener.listen(newFilter)...
``` [3](#0-2) 

`Long.MAX_VALUE - 0 = Long.MAX_VALUE` is stored in `newFilter.limit` and passed to `topicListener.listen(newFilter)`.

**Step 3 — Integer overflow in `poll()`.**

In `PollingTopicListener.poll()`:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
...
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [4](#0-3) 

- `(int)(Long.MAX_VALUE - 0)` = `(int) 0x7FFFFFFFFFFFFFFF` = `(int) 0xFFFFFFFF` = **`-1`** (Java narrowing cast truncates to low 32 bits)
- `Math.min(-1, 5000)` = **`-1`** (default `maxPageSize` is 5000)
- `filter.toBuilder().limit(-1).build()` is a direct Lombok builder call — Spring's `@Validated` AOP proxy is **not** involved, so `@Min(0)` is **not** enforced

The same overflow exists identically in `PollingTopicMessageRetriever.poll()`: [5](#0-4) 

**Step 4 — Negative limit bypasses the guard in the repository.**

In `TopicMessageRepositoryCustomImpl.findByFilter()`:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
``` [6](#0-5) 

`hasLimit()` checks `limit > 0`. With `limit = -1`, this returns `false`. `setMaxResults()` is never called. The JPA query executes with **no LIMIT clause**, returning every matching `topic_message` row for the topic. [7](#0-6) 

**Step 5 — This repeats indefinitely.**

The poll loop uses `RepeatSpec.times(Long.MAX_VALUE)`, meaning the unbounded query fires on every poll interval for the lifetime of the subscription: [8](#0-7) 

The outer `flux.take(filter.getLimit())` in `subscribeTopic` is `flux.take(Long.MAX_VALUE)`, providing no practical bound: [9](#0-8) 

## Impact Explanation
Every poll cycle issues a full table scan of `topic_message` for the given topic with no row limit. On a topic with millions of messages, each poll returns the entire dataset, consuming database CPU, I/O, and network bandwidth. A single connection held open by an attacker causes this to repeat at the configured poll interval (default 500ms) indefinitely. Multiple concurrent connections amplify the effect linearly. This is a practical denial-of-service against the database tier.

## Likelihood Explanation
The gRPC endpoint is publicly accessible. The proto field `limit` is `uint64`; sending `Long.MAX_VALUE` requires only a standard gRPC client (e.g., `grpcurl`) with a single field set. No credentials, no special knowledge, and no prior access are needed. The attack is trivially repeatable and scriptable. This vulnerability is only triggered when the listener type is configured to `POLL` (default is `REDIS`), but `POLL` mode is a supported and documented configuration. [10](#0-9) 

## Recommendation

1. **Cap the limit at the controller layer** in `ConsensusController.toFilter()` before building the filter — reject or clamp any `limit` value exceeding a configured maximum (e.g., `Integer.MAX_VALUE` or a domain-specific cap).

2. **Fix the narrowing cast** in `PollingTopicListener.poll()` and `PollingTopicMessageRetriever.poll()`. Replace:
   ```java
   int limit = filter.hasLimit()
           ? (int) (filter.getLimit() - context.getCount().get())
           : Integer.MAX_VALUE;
   ```
   with a safe clamp:
   ```java
   long remaining = filter.hasLimit()
           ? filter.getLimit() - context.getCount().get()
           : Integer.MAX_VALUE;
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```

3. **Add a defensive guard in `findByFilter()`** to treat any non-positive limit as "no limit" or throw, rather than silently issuing an unbounded query.

## Proof of Concept

```bash
# Using grpcurl against a mirror node with POLL listener type configured
grpcurl -plaintext \
  -d '{
    "topicID": {"topicNum": 1},
    "consensusStartTime": {"seconds": 0},
    "limit": 9223372036854775807
  }' \
  localhost:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

With `limit = 9223372036854775807` (`Long.MAX_VALUE`):
- `hasLimit()` returns `true` → listener path is taken
- `(int)(Long.MAX_VALUE - 0)` = `-1` in `PollingTopicListener.poll()`
- `Math.min(-1, 5000)` = `-1`
- `findByFilter()` receives `limit = -1`, `hasLimit()` returns `false`, no `setMaxResults()` is called
- Every 500ms, a full table scan of `topic_message` for topic 1 is issued indefinitely
