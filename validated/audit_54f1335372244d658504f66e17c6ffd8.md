## Audit Report

## Title
Unbounded DB Queries via Non-Existent Historical Block Numbers Due to Cache Miss on Empty Optional

## Summary
`RecordFileRepository.findByIndex()` uses `@Cacheable(unless = "#result == null")`, but Spring Cache (4.3+) unwraps `Optional` return values before evaluating the `unless` condition — so `Optional.empty()` is treated as `null` and is **never stored in the cache**. Combined with the gas throttle being fully restored for failed calls (when `result == null`), an unauthenticated attacker can sustain a continuous stream of uncached DB queries at the full `requestsPerSecond` rate limit with near-zero gas cost.

## Finding Description

**Verified code path:**

`ContractController.call()` throttles the request and calls `contractExecutionService.processCall()`. [1](#0-0) 

`ContractCallService.callContract()` (lines 100–107) sets a lazy, memoized `blockSupplier` that calls `recordFileService.findByBlockType()` → `recordFileRepository.findByIndex()` when first accessed during EVM execution: [2](#0-1) 

`RecordFileServiceImpl.findByBlockType()` routes any numeric block to `findByIndex()`: [3](#0-2) 

`RecordFileRepository.findByIndex()` is annotated: [4](#0-3) 

**Root cause — cache miss on empty Optional:**
Spring Cache 4.3+ unwraps `Optional` return values before evaluating `unless`. When `findByIndex()` returns `Optional.empty()` (non-existent block), the unwrapped value is `null`, so `#result == null` is `true` and the result is **never stored in the cache**. Every subsequent request for the same non-existent block number re-executes `SELECT r FROM RecordFile r WHERE r.index = ?1` against the database. The `CACHE_MANAGER_RECORD_FILE_INDEX` cache (10-minute TTL, max 10,000 entries) only ever holds successful hits. [5](#0-4) 

**Gas throttle bypass:**
`BlockNumberNotFoundException` extends `InvalidInputException` → `Web3Exception`, which is **not** caught by any of the `catch` blocks in `doProcessCall()` (`IllegalStateException | IllegalArgumentException` or `MirrorEvmTransactionException`). It propagates uncaught, but the `finally` block still executes and calls `restoreGasToBucket(result, params.getGas())` with `result == null`: [6](#0-5) 

In `restoreGasToBucket()`, when `result == null`, the full `gasLimitToRestoreBaseline` is restored. With the default `gasLimitRefundPercent = 100`, the entire gas amount is returned to the bucket: [7](#0-6) [8](#0-7) 

This is confirmed by the existing test `pureCallWithOutOfRangeCustomBlockThrowsException`, which demonstrates that `BlockNumberNotFoundException` is thrown and propagates out of the service layer: [9](#0-8) 

## Impact Explanation
Each request with a distinct non-existent block number (e.g., `Long.MAX_VALUE - N`) executes `SELECT r FROM RecordFile r WHERE r.index = ?1` against the database with no cache benefit. At the default `requestsPerSecond` ceiling of 500 RPS, an attacker can sustain 500 uncached DB queries per second indefinitely. Because gas is fully restored on each failed call, the attacker's effective gas cost per DB query is zero — only the per-request rate limit applies. On a node where the `record_file` table is large or the DB connection pool is constrained, this produces measurable I/O amplification. [10](#0-9) 

## Likelihood Explanation
No authentication or special privilege is required — `/api/v1/contracts/call` is a public endpoint. The attacker only needs to supply a valid JSON body with a numeric `block` field set to a non-existent block number. Rotating through a large space of non-existent block numbers (e.g., `Long.MAX_VALUE - N`) trivially avoids any future cache population. The attack is fully scriptable and repeatable. [11](#0-10) 

## Recommendation

1. **Fix the cache miss:** Change the `unless` condition on `findByIndex()` to cache empty Optional results. Use `unless = "#result != null && !#result.isPresent()"` or, more idiomatically, use a sentinel/null-object pattern. Alternatively, use `unless = "false"` to always cache (including misses), or switch to a `@Cacheable` condition that explicitly handles `Optional.empty()`.

2. **Validate block existence before EVM execution:** In `ContractCallService.callContract()`, eagerly resolve the block supplier for numeric block types before entering `doProcessCall()`. If the block does not exist, throw `BlockNumberNotFoundException` before any gas is consumed from the throttle bucket — so no gas restoration is needed and the gas throttle remains effective.

3. **Do not restore gas for pre-execution failures:** In `doProcessCall()`, catch `BlockNumberNotFoundException` (or its parent `InvalidInputException`) explicitly and re-throw without triggering `restoreGasToBucket()`, or restructure the `finally` block to skip gas restoration when the failure occurred before EVM execution began.

    private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
        // If the transaction fails, gasUsed is equal to gasLimit, so restore the configured refund percent
        // of the gasLimit value back in the bucket.
        final var gasLimitToRestoreBaseline = (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
        if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
            throttleManager.restore(gasLimitToRestoreBaseline);
        } else {
            // The transaction was successful or reverted, so restore the remaining gas back in the bucket or
            // the configured refund percent of the gasLimit value back in the bucket - whichever is lower.
            final var gasRemaining = gasLimit - result.gasUsed();
            throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
        }
    }
