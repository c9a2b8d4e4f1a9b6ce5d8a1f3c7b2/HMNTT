Audit Report

## Title
Log Injection via Unsanitized Exception Message in GraphQL `LoggingFilter`

## Summary
The `LoggingFilter` in the `graphql` module logs raw exception messages without stripping newline characters (`\n`, `\r`). An unauthenticated attacker can craft a GraphQL request that triggers a parse/validation error whose message contains attacker-controlled newlines, injecting arbitrary fake log lines into the application log stream.

## Finding Description
**File:** `graphql/src/main/java/org/hiero/mirror/graphql/config/LoggingFilter.java`

`doFilterInternal()` (lines 27–38) catches any exception and passes it to `logRequest()` (lines 40–60), which calls `getMessage()` (lines 62–72):

```java
// lines 62-72
private String getMessage(HttpServletRequest request, Exception e) {
    if (e != null) {
        return e.getMessage();   // raw, unsanitized
    }
    if (request.getAttribute(ERROR_EXCEPTION_ATTRIBUTE) instanceof Exception ex) {
        return ex.getMessage();  // raw, unsanitized
    }
    return SUCCESS;
}
```

The returned `message` is placed directly into the log params array at line 51:

```java
var params = new Object[] {request.getRemoteAddr(), request.getMethod(), uri, elapsed, status, message};
```

and emitted via SLF4J at lines 54–58 with format `"{} {} {} in {} ms: {} {}"`. SLF4J/Logback does not strip embedded newlines from `{}` arguments.

**Root cause:** No newline/CRLF stripping is applied anywhere in the logging path. `DomainUtils.sanitize()` (line 239–241) only replaces null bytes (`\0`). The `web3` module's `LoggingFilter` applies `StringUtils.deleteWhitespace()` to the request body content but not to exception messages, and the `graphql` module's `LoggingFilter` does not apply it at all.

**URI vector (secondary, largely mitigated):** `request.getRequestURI()` at line 41 is also logged raw, but Tomcat's default configuration rejects `%0a`/`%0d` in URI paths (returning 400 before the filter runs).

**Exception-message vector (primary, unmitigated):** GraphQL-Java includes the offending input token in many parse/validation error messages (e.g., `"Invalid syntax with offending token '<attacker string>' at line 1 column N"`). A POST body containing a string literal with an embedded newline character produces a parse error whose `.getMessage()` contains that newline verbatim.

## Impact Explanation
An attacker can inject arbitrary text as additional log lines, for example:

```
INFO  ... 127.0.0.1 POST /graphql in 3 ms: 400 Invalid syntax ... 'foo
INFO  ... 10.0.0.1 GET /admin/secret in 0 ms: 200 Success
```

This enables: (1) forging successful-access or privileged-path entries to mislead SIEM/SOC analysts; (2) hiding real attack traffic by flooding logs with noise; (3) corrupting structured log parsers (e.g., JSON log pipelines) that split on newlines. Severity is **Medium** — no direct data exfiltration, but meaningful damage to log integrity and security monitoring.

## Likelihood Explanation
Preconditions: none beyond network access to the GraphQL endpoint (unauthenticated by design). Crafting a POST body with embedded newlines in a string literal is trivial. The attack is repeatable at will and leaves no distinguishable trace in the HTTP layer.

## Recommendation
In `getMessage()`, strip or replace `\n` and `\r` characters before returning the message string. For example:

```java
private static String sanitizeLogValue(String value) {
    return value == null ? null : value.replace('\n', '_').replace('\r', '_');
}
```

Apply this to both `e.getMessage()` return paths in `getMessage()`, and also to `uri` in `logRequest()` as a defense-in-depth measure. Consider applying the same fix to the analogous `getMessage()` in `web3/src/main/java/org/hiero/mirror/web3/config/LoggingFilter.java` and other modules.

Alternatively, configure Logback with a `replace` conversion pattern or a `PatternLayoutEncoder` that encodes control characters, providing a centralized fix across all log statements.

## Proof of Concept
Send the following HTTP request to the GraphQL endpoint:

```
POST /graphql HTTP/1.1
Content-Type: application/json

{"query":"{ field(arg: \"foo\nINFO  10.0.0.1 GET /admin/secret in 0 ms: 200 Success\") }"}
```

GraphQL-Java will produce a parse/validation error whose message contains the embedded newline. The `LoggingFilter` will write two separate log lines: the real 400 error line and the injected fake line.
