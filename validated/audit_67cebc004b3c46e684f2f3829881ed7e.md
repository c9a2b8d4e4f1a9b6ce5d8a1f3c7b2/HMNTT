Audit Report

## Title
Unbounded `CaffeineCacheManager` Registry Growth via Unique `contractId` Flooding in `findStorageBatch()`

## Summary
`ContractStateServiceImpl.findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` on every cache miss. Because `cacheManagerSlotsPerContract` is a `CaffeineCacheManager` in dynamic mode (no `setCacheNames()`), its internal `ConcurrentHashMap<String, Cache> cacheMap` grows without bound. The outer `contractSlotsCache` (maximumSize=3000) evicts entries but never removes the corresponding `CaffeineCache` objects from `cacheManagerSlotsPerContract`'s registry, creating a permanent memory leak exploitable via unauthenticated `eth_call` flooding.

## Finding Description
`EvmConfiguration.java` lines 107–112 configure `cacheManagerSlotsPerContract` without calling `setCacheNames()`:

```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    return caffeineCacheManager; // dynamic mode — no setCacheNames()
}
``` [1](#0-0) 

In dynamic mode, Spring's `CaffeineCacheManager.getCache(name)` calls `cacheMap.computeIfAbsent(name, this::createCaffeineCache)`. This `cacheMap` is a plain `ConcurrentHashMap` with no eviction mechanism.

`ContractStateServiceImpl.java` lines 86–87 invoke this on every outer cache miss:

```java
final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
        contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [2](#0-1) 

The outer `contractSlotsCache` is bounded at `maximumSize=3000` with `expireAfterAccess=5m`. [4](#0-3) 

When `contractSlotsCache` evicts an entry for a `contractId`, the `CaffeineCache` object is removed from `contractSlotsCache`'s value set, but **it remains permanently in `cacheManagerSlotsPerContract.cacheMap`** keyed by `contractId.toString()`. There is no removal callback or listener wired to clean up the registry. The registry only grows.

Each `CaffeineCache` instance wraps a Caffeine cache configured with `maximumSize=1500`, [5](#0-4)  which pre-allocates internal structures including a `FrequencySketch` table (sized to the next power-of-two ≥ `maximumSize`), a `BoundedLocalCache` node array, and associated bookkeeping — totalling tens of kilobytes per instance.

## Impact Explanation
Each unique `contractId` permanently allocates a `CaffeineCache` object in the JVM heap inside `cacheManagerSlotsPerContract.cacheMap`. With sustained flooding at the global rate limit, the heap fills progressively. The leaked objects are not eligible for GC because `cacheManagerSlotsPerContract` (a Spring bean) holds a strong reference to `cacheMap`, which holds strong references to every `CaffeineCache` ever created. This leads to continuous full GC cycles, CPU exhaustion from GC threads competing with request threads, and ultimately `OutOfMemoryError`, crashing the web3 service. The attack requires no privileged access, no on-chain transactions, and no special tooling.

## Likelihood Explanation
The `eth_call` JSON-RPC endpoint is publicly accessible and requires no authentication. An attacker needs only to iterate through distinct Ethereum-format addresses in the `to` field of `eth_call` requests — trivially scriptable with `curl` or any JSON-RPC library. The attack is persistent: even after the attacker stops, the leaked `CaffeineCache` objects remain in the heap until the JVM is restarted.

## Recommendation
1. **Pre-declare cache names**: Call `caffeineCacheManager.setCacheNames(Set.of(...))` in `cacheManagerSlotsPerContract()` with a fixed set of names, and set `caffeineCacheManager.setAllowNullValues(false)`. This disables dynamic cache creation entirely, preventing unbounded registry growth.
2. **Eliminate the two-level cache design**: Instead of storing `CaffeineCache` objects as values in `contractSlotsCache`, use a single flat cache keyed by `(contractId, slotKey)` tuples, removing the need for `cacheManagerSlotsPerContract` altogether.
3. **Add a removal listener**: If the two-level design is retained, register a Caffeine removal listener on `contractSlotsCache` that calls a custom eviction hook to remove the corresponding entry from `cacheManagerSlotsPerContract`'s registry via reflection or a wrapper.
4. **Per-IP rate limiting**: Enforce rate limits per source IP at the ingress/API gateway layer to prevent a single client from exhausting the global budget.

## Proof of Concept
```python
import requests, json, threading

TARGET = "http://<web3-endpoint>/api/v1/contracts/call"

def flood(start, end):
    for i in range(start, end):
        addr = f"0x{i:040x}"
        payload = {
            "jsonrpc": "2.0", "method": "eth_call",
            "params": [{"to": addr, "data": "0x"}, "latest"],
            "id": i
        }
        requests.post(TARGET, json=payload, timeout=5)

# Launch threads to generate unique contractId cache entries
threads = [threading.Thread(target=flood, args=(i*10000, (i+1)*10000)) for i in range(50)]
for t in threads: t.start()
for t in threads: t.join()
# After >3000 unique addresses, contractSlotsCache evicts entries
# but cacheManagerSlotsPerContract.cacheMap retains all CaffeineCache objects.
# Heap grows monotonically until OOM.
```
