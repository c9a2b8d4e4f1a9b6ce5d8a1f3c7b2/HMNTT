Audit Report

## Title
Unbounded Cache Eviction via Unauthenticated Contract Calls Enables DB Resource Exhaustion

## Summary
The `CACHE_MANAGER_CONTRACT` Caffeine cache used by `ContractRepository.findRuntimeBytecode()` has a hard `maximumSize=1000`. An unauthenticated attacker cycling through 1001+ distinct contract addresses via `POST /api/v1/contracts/call` can continuously evict legitimate cached bytecodes, forcing every subsequent legitimate request to re-query the database. The global (not per-IP) rate limit of 500 req/s does not prevent this.

## Finding Description

**Cache configuration** — `CacheProperties.java:22`:
```
private String contract = "expireAfterAccess=1h,maximumSize=1000,recordStats";
```
The cache holds at most 1000 entries. When the 1001st unique `contractId` is inserted, Caffeine evicts the least-recently-accessed entry. [1](#0-0) 

**Unauthenticated public endpoint** — `ContractController.java:37-51`:
The `@PostMapping(value = "/call")` handler has no authentication guard. Any caller can trigger EVM execution. [2](#0-1) 

**Cache population path** — `ContractBytecodeReadableKVState.java:39-47`:
Each call reaches `readFromDataSource()`, which calls `contractRepository.findRuntimeBytecode(entityId.getId())`, populating the cache keyed by `contractId`. [3](#0-2) 

**`unless` condition gap** — `ContractRepository.java:16`:
```java
@Cacheable(..., unless = "#result == null")
```
`findRuntimeBytecode` returns `Optional<byte[]>`. `Optional.empty()` is **not** `null`, so cache entries for non-existent contracts are stored. The attacker does not need valid contract IDs to fill all 1000 slots. [4](#0-3) 

**Global-only rate limiting** — `ThrottleProperties.java:35` and `ThrottleManagerImpl.java:37-42`:
The throttle is a single shared `rateLimitBucket` of 500 req/s with no per-IP or per-sender partitioning. An attacker can consume the entire budget. [5](#0-4) [6](#0-5) 

## Impact Explanation
Under normal operation the contract bytecode cache absorbs repeated lookups for the same contracts. With the cache continuously thrashed, every legitimate `eth_call` or `eth_estimateGas` that touches a contract incurs a live `SELECT runtime_bytecode FROM contract WHERE id = :contractId` query. The `runtime_bytecode` column stores large blobs, amplifying I/O cost per miss. On a node serving hundreds of requests per second this translates to a sustained, measurable increase in database query volume and I/O load.

## Likelihood Explanation
No credentials or special knowledge are required. Contract addresses are publicly enumerable via the Mirror Node REST API (`/api/v1/contracts`). The attack requires only an HTTP client capable of ~500 req/s — achievable from a single machine. Because `Optional.empty()` results are also cached, the attacker does not even need valid contract IDs; any 1001 distinct numeric IDs suffice. The attack is self-sustaining and repeatable indefinitely.

## Recommendation

1. **Fix the `unless` condition** to also exclude empty Optionals:
   ```java
   unless = "#result == null || !#result.isPresent()"
   ```
   This prevents non-existent contract lookups from consuming cache slots. [7](#0-6) 

2. **Increase `maximumSize`** in `CacheProperties.java` to a value that reflects the realistic number of active contracts, or switch to a size-weighted eviction policy. [8](#0-7) 

3. **Add per-IP rate limiting** in `ThrottleManagerImpl` or at the ingress/load-balancer layer so a single source cannot consume the entire global budget. [6](#0-5) 

## Proof of Concept

```python
import requests, itertools

BASE = "https://<mirror-node>/api/v1"
CALL = f"{BASE}/contracts/call"

# Step 1: enumerate 1001 contract addresses (no auth required)
contracts = requests.get(f"{BASE}/contracts?limit=1000").json()["contracts"]
addresses  = [c["evm_address"] for c in contracts]
# pad with synthetic long-zero addresses if needed
addresses += [f"0x{i:040x}" for i in range(1001 - len(addresses))]

# Step 2: cycle through all 1001+ addresses continuously
for addr in itertools.cycle(addresses):
    requests.post(CALL, json={"to": addr, "data": "0x", "gas": 30000}, timeout=2)
    # After 1000 unique requests the cache is full.
    # Each subsequent request evicts the LRU entry, preventing cache stabilisation.
```

Every legitimate user request for a previously-cached contract now misses the cache and executes a live DB query for the `runtime_bytecode` blob.
