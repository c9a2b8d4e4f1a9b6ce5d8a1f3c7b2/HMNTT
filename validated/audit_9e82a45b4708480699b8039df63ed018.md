Audit Report

## Title
Unauthenticated HTTP Fetch of Node Topology Enables Transaction Misdirection via DNS Spoofing or MITM

## Summary
When `network=other` is configured, `buildNetworkFromMirrorNodes` fetches consensus node gRPC addresses from `cfg.mirrorRest` over plain HTTP with no transport security enforcement and no response authentication. An attacker who can poison DNS or intercept the HTTP response can inject arbitrary gRPC endpoints, causing every subsequent signed `CryptoTransfer` to be submitted to attacker-controlled infrastructure instead of real consensus nodes.

## Finding Description
**`pinger/config.go` line 37** documents the default example URL as `http://mirror-rest:5551` — plain HTTP — and no validation anywhere in the config loading enforces an `https://` scheme. [1](#0-0) 

**`buildNetworkFromMirrorNodes`** (`pinger/mirror_node_client.go` lines 36–72) constructs the URL verbatim from `cfg.mirrorRest` and issues the request through a plain `http.Client` with no TLS configuration. [2](#0-1) 

**`fetchMirrorNodeNetwork`** (lines 74–132) performs three checks on the response, all of which an attacker-controlled server trivially satisfies:
- HTTP 2xx status (lines 90–93) — the attacker's server returns 200.
- JSON decode success (lines 96–98) — the attacker returns well-formed JSON.
- Non-empty network map (lines 127–129) — the attacker includes at least one valid-looking entry. [3](#0-2) 

The `host:port` pairs from `service_endpoints` are placed directly into the network map with no IP/domain allowlist and no cryptographic verification. [4](#0-3) 

**`newClient`** (`pinger/sdk_client.go` lines 17–22) passes the poisoned map straight to `hiero.ClientForNetwork(netmap)`, making the attacker's endpoints the sole gRPC targets for the SDK client. [5](#0-4) 

**`submitWithRetry`** (`pinger/transfer.go` lines 33–42) then calls `cryptoTransfer.Execute(client)`, sending every operator-signed transaction to whatever nodes the poisoned client contains. [6](#0-5) 

**Root cause:** `buildNetworkFromMirrorNodes` assumes the HTTP response from `cfg.mirrorRest` is trustworthy. There is no mechanism — TLS certificate pinning, response signing, or IP allowlist — to verify that the returned node addresses belong to the legitimate network.

## Impact Explanation
The attacker receives fully operator-signed `CryptoTransfer` transactions. They can:
1. **Suppress** them entirely — health-check transactions never reach consensus, causing the pinger to report false failures or masking real network outages.
2. **Delay and selectively replay** — hold transactions and submit them to the real network at a chosen time, altering their ordering in consensus history relative to other transactions.
3. **Observe** transaction content, operator account identity, and timing patterns.

Because the pinger is a liveness/health monitoring tool, sustained suppression blinds operators to real network degradation.

## Likelihood Explanation
The attack is scoped to `network=other` deployments (custom/private networks). In Kubernetes or Docker Compose environments — the typical deployment target for a self-hosted mirror node stack — DNS is resolved via an internal resolver. An attacker who has compromised any workload in the same namespace or network segment can poison DNS for the pinger's mirror REST hostname without any privileges on the pinger pod itself. If `cfg.mirrorRest` uses a plain `http://` URL, any on-path attacker (e.g., a compromised sidecar or network device) can perform a trivial MITM. No credentials or host-level access to the pinger are required.

## Recommendation
1. **Enforce HTTPS at config load time**: In `loadConfig` (`pinger/config.go`), reject any `cfg.mirrorRest` value that does not begin with `https://` when `network=other`.
2. **Use TLS with certificate verification**: Ensure the `http.Client` in `buildNetworkFromMirrorNodes` uses the default TLS transport (it already does, but only if the URL scheme is `https`). Do not disable `InsecureSkipVerify`.
3. **Optional defense-in-depth**: Maintain a configurable allowlist of valid node account IDs or IP ranges and reject any `service_endpoints` entry not on the list.
