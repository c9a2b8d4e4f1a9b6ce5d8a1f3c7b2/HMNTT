Audit Report

## Title
Stale Contract Bytecode Cache (1h TTL) Causes Incorrect Gas Estimates After Contract Deletion/Self-Destruct

## Summary
`ContractRepository.findRuntimeBytecode()` is annotated with `@Cacheable` using `CACHE_MANAGER_CONTRACT`, which is configured with `expireAfterAccess=1h`. There is no `@CacheEvict` annotation, no deletion-event listener, and no deletion-status guard in the bytecode lookup path. Once a contract's bytecode is cached, it remains cached for up to one hour after last access, even if the contract is deleted or self-destructed on the consensus network. `ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode()` directly without checking entity deletion status, and for long-zero addresses it bypasses `commonEntityAccessor` entirely.

## Finding Description

**Root cause:** `findRuntimeBytecode` is cached for 1 hour after last access (`expireAfterAccess=1h`). The `unless = "#result == null"` guard only prevents caching a null return value; Spring Cache unwraps `Optional`, so `Optional.of(bytecode)` is cached and `Optional.empty()` is not. A contract that existed at cache-population time and is subsequently deleted will have its bytecode remain in cache for the full TTL window. There is no `@CacheEvict` on any deletion path, no Spring `ApplicationEvent` listener, and no deletion-status check in `readFromDataSource`.

**Exact locations:**
- Cache TTL: `CacheProperties.java` line 22 — `expireAfterAccess=1h,maximumSize=1000,recordStats`
- Cache annotation: `ContractRepository.java` lines 16–18 — `@Cacheable(..., unless = "#result == null")`, no `@CacheEvict`
- No deletion check: `ContractBytecodeReadableKVState.java` lines 39–47 — calls `findRuntimeBytecode` unconditionally
- Long-zero bypass: `ContractBytecodeReadableKVState.java` lines 50–55 — skips `commonEntityAccessor` for long-zero addresses
- Confirmed by test: `ContractRepositoryTest.java` lines 25–29 — after `deleteAll()`, cached bytecode is still returned

**Why the entity cache does not help:** The entity cache (`expireAfterWrite=1s`) correctly reflects deletion within one second, but `readFromDataSource` in `ContractBytecodeReadableKVState` does not consult the entity cache before calling `findRuntimeBytecode`. The two caches are completely independent.

## Impact Explanation
`eth_estimateGas` and `eth_call` requests targeting a self-destructed or deleted contract address will execute EVM simulation against stale bytecode and return a non-zero gas estimate as if the contract still exists. Wallets and dApps that use this estimate to set transaction gas limits will submit transactions to the consensus network that fail (the contract no longer exists there), wasting the submitted gas fee. The staleness window is up to 1 hour and can be extended indefinitely by periodically querying the mirror node to reset the `expireAfterAccess` TTL before it expires.

## Likelihood Explanation
The preconditions are minimal: any Hedera account holder can deploy a contract, trigger cache population via a mirror node `eth_estimateGas` call, and then self-destruct the contract. No special privilege is required. The self-destruct path is exercised in the existing test suite, confirming it is a supported and reachable operation. The 1-hour `expireAfterAccess` window is long enough to be practically exploitable.

## Recommendation
1. **Add `@CacheEvict` on deletion paths** or replace `expireAfterAccess` with a shorter `expireAfterWrite` TTL for the contract bytecode cache.
2. **Add a deletion-status guard in `readFromDataSource`**: before returning cached bytecode, verify the entity is not deleted via `commonEntityAccessor` (consistent with how other `ReadableKVState` implementations such as `ScheduleReadableKVState` check entity status before returning data).
3. **Apply the deletion check to long-zero addresses** as well, since the current code bypasses `commonEntityAccessor` for that path.
4. Consider aligning the bytecode cache TTL with the entity cache TTL (`expireAfterWrite=1s`) or using `expireAfterWrite` instead of `expireAfterAccess` to bound the maximum staleness window.

## Proof of Concept
The existing integration test `ContractRepositoryTest.findRuntimeBytecodeSuccessfulCall()` serves as a direct proof of concept:

```java
Contract contract1 = domainBuilder.contract().persist();
// Populate cache
assertThat(contractRepository.findRuntimeBytecode(contract1.getId()))
        .get().isEqualTo(contract1.getRuntimeBytecode());

// Delete all contracts from DB (simulates self-destruct/deletion)
contractRepository.deleteAll();

// Cache still returns stale bytecode for contract1
assertThat(contractRepository.findRuntimeBytecode(contract1.getId()))
        .get().isEqualTo(contract1.getRuntimeBytecode()); // PASSES — stale cache hit
```

This test explicitly asserts that stale bytecode is returned after deletion, confirming the behavior. Any `eth_estimateGas` call to the mirror node for the deleted contract's address within the 1-hour window will execute EVM simulation against this stale bytecode and return an incorrect gas estimate.
