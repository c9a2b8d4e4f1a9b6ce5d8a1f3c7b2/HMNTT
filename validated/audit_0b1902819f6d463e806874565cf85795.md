Audit Report

## Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` Enables Unbounded Database Query via gRPC `limit` Field

## Summary
A narrowing cast from `long` to `int` in `PollingTopicMessageRetriever.poll()` allows a caller to supply a `limit` value (e.g., `Long.MAX_VALUE`) that overflows to `-1` after the cast. The resulting negative value bypasses the `hasLimit()` guard in the repository layer, causing `setMaxResults` to never be called and the JPA query to return every matching row from the `topic_message` table.

## Finding Description

**Overflow site** — `PollingTopicMessageRetriever.java`, lines 68–71:

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
```

`filter.getLimit()` is a `long`. [1](#0-0) 

When `filter.getLimit()` is `Long.MAX_VALUE` and `context.getTotal().get()` is `0`, the subtraction yields `Long.MAX_VALUE`. The narrowing cast `(int)(0x7FFFFFFFFFFFFFFF)` retains only the lower 32 bits: `0xFFFFFFFF` = **`-1`**. `Math.min(-1, maxPageSize)` = `-1`, so `pageSize = -1`. [2](#0-1) 

**Rebuilt filter** — line 75 passes `pageSize = -1` directly into a new filter with no validation:

```java
var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [3](#0-2) 

**`hasLimit()` bypass** — `hasLimit()` is `return limit > 0;`. With `limit = -1` this returns `false`. [4](#0-3) 

**No `setMaxResults` applied** — `TopicMessageRepositoryCustomImpl.findByFilter()` lines 51–53:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());
}
```

With `hasLimit()` returning `false`, `setMaxResults` is never called and the JPA query returns every matching row. [5](#0-4) 

**Why existing checks fail:**

1. `TopicMessageFilter.limit` carries `@Min(0)` but no `@Max`, so `Long.MAX_VALUE` passes Bean Validation at the service boundary. [1](#0-0) 

2. The `@Valid` annotation on `subscribeTopic()` validates only the user-supplied filter, not the internally rebuilt `newFilter`. [6](#0-5) 

3. The `limit` field is sourced directly from `query.getLimit()` (a protobuf `uint64`) with no upper-bound check before building the filter. [7](#0-6) 

4. The `isComplete()` check uses `filter.getLimit() == total.get()`, which can never be satisfied when `filter.getLimit()` is `Long.MAX_VALUE`, so the polling loop never self-terminates for throttled subscribers (`numRepeats = Long.MAX_VALUE`). [8](#0-7) [9](#0-8) 

## Impact Explanation
An unprivileged gRPC client can trigger an unbounded `SELECT` against the `topic_message` table on every polling interval. For a high-volume topic this can return millions of rows per poll cycle, causing heap exhaustion or severe GC pressure on the mirror node's JVM, degrading or crashing the gRPC service for all subscribers. The polling loop repeats with `numRepeats = Long.MAX_VALUE` for throttled subscribers, so the unbounded query fires continuously until the connection times out (default 60 s). [10](#0-9) 

## Likelihood Explanation
The exploit requires only a standard gRPC client and knowledge of any valid topic ID. No credentials, elevated roles, or special network access are needed. The `limit` field is a plain `uint64` in the protobuf wire format; setting it to `Long.MAX_VALUE` is a single-field change. The condition is 100% reproducible and requires no race or timing dependency. [7](#0-6) 

## Recommendation

1. **Add an upper-bound constraint** on `TopicMessageFilter.limit`, e.g. `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate maximum, to reject extreme values at the service boundary. [1](#0-0) 

2. **Eliminate the narrowing cast** in `poll()`. Compute `limit` as a `long`, clamp it to `[0, maxPageSize]` using `long` arithmetic, and only cast to `int` after the clamp is confirmed safe (i.e., after verifying the value fits in `int`). [2](#0-1) 

3. **Guard against non-positive `pageSize`** before building `newFilter`: if `pageSize <= 0`, return `Flux.empty()` immediately rather than issuing a query. [3](#0-2) 

4. **Validate the rebuilt filter** or add a defensive check in `findByFilter` to treat any non-positive limit as "no results" rather than "no cap". [5](#0-4)
