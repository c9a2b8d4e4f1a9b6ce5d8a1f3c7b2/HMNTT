Audit Report

## Title
Reference Equality Bypass on `BlockType.EARLIEST` Allows `findByIndex(0)` to Be Called Instead of `findEarliest()`, Causing `BlockNumberNotFoundException` in Partial-History Deployments

## Summary
`RecordFileServiceImpl.findByBlockType()` uses Java reference equality (`==`) to detect the `EARLIEST` sentinel. `BlockType.of()` only returns the `EARLIEST` singleton for the string `"earliest"` (and its aliases). Any numeric representation of block 0 — including `"0x0"`, `"0"`, or `"0x0000000000000000"` — constructs a **new** `BlockType` instance, causing the identity check to fail silently and falling through to `recordFileRepository.findByIndex(0)`. In partial-history deployments where the earliest stored block has `index > 0`, this returns an empty `Optional`, and the caller throws `BlockNumberNotFoundException` (HTTP 400).

## Finding Description

**Root cause — singleton definition:**

`BlockType` is a Java `record`. The `EARLIEST` singleton is defined as: [1](#0-0) 

**`BlockType.of("0x0")` does not return the singleton:**

The `of()` factory method only returns `EARLIEST` via the `GROUP_TAG` branch when the input is the literal string `"earliest"`: [2](#0-1) 

Any hex-numeric input (e.g. `"0x0"`, `"0x0000000000000000"`) matches `GROUP_HEX_NUM` and returns a **new heap object**: [3](#0-2) 

Similarly, decimal `"0"` matches `GROUP_DECIMAL` and also returns a new instance: [4](#0-3) 

**Guard uses `==` (reference equality):**

`findByBlockType()` checks identity, not value equality: [5](#0-4) 

Because the new instance is not the same object reference as `EARLIEST`, and `isHash()` is false (number is `0`, not `BLOCK_HASH_SENTINEL = -1`), the call falls through to `recordFileRepository.findByIndex(0)`.

**Semantic difference between the two repository methods:**

- `findEarliest()` — `SELECT * FROM record_file ORDER BY index ASC LIMIT 1` — always returns the lowest-indexed block present: [6](#0-5) 

- `findByIndex(0)` — `SELECT r FROM RecordFile r WHERE r.index = 0` — returns a block only if one with **exactly** `index = 0` exists: [7](#0-6) 

**Exception propagation:**

The caller in `ContractCallService` wraps the result with `.orElseThrow(BlockNumberNotFoundException::new)`. When `findByIndex(0)` returns empty, `BlockNumberNotFoundException` is thrown: [8](#0-7) 

Controller tests confirm this maps to HTTP 400 with body `"Unknown block number"`: [9](#0-8) 

## Impact Explanation
In a partial-history mirror node deployment (where syncing began after genesis, so the earliest stored block has `index > 0`), `findByIndex(0)` returns empty for every request specifying block `0x0`, `0`, or any zero-valued hex string. Every `eth_call` or `eth_estimateGas` request using those block parameters receives HTTP 400 `"Unknown block number"`, while the semantically equivalent `"earliest"` tag succeeds. This violates the Ethereum JSON-RPC specification, which treats block number `0x0` and tag `"earliest"` as equivalent. In full-history deployments the behavior is accidentally correct because `index = 0` exists, but the logic is structurally broken regardless.

## Likelihood Explanation
Any unprivileged external caller can trigger this with a standard JSON-RPC request. The input `"0x0"` is a valid, commonly used block parameter. No authentication, special role, or knowledge of internal state is required. The condition is fully repeatable and deterministic. The existing test suite (`RecordFileServiceTest.testFindByBlockTypeEarliest`) only tests with the `BlockType.EARLIEST` singleton directly and does not cover `BlockType.of("0x0")`, so the bug is not caught by current tests. [10](#0-9) 

## Recommendation
Replace the reference equality checks in `findByBlockType()` with value-based comparisons. Since `BlockType` is a Java `record`, its compiler-generated `equals()` compares fields structurally. The fix is:

```java
// RecordFileServiceImpl.java
if (block.equals(BlockType.EARLIEST)) {
    return recordFileRepository.findEarliest();
} else if (block.equals(BlockType.LATEST)) {
    return recordFileRepository.findLatest();
}
```

Alternatively, compare by `number()` value directly:

```java
if (block.number() == BlockType.EARLIEST.number() && !block.isHash()) {
    return recordFileRepository.findEarliest();
}
```

The `blockTypeForTag` method already correctly returns the singleton for the `"earliest"` string; the fix only needs to be applied in `findByBlockType()`. [11](#0-10)

```

// This succeeds — returns the block at index 5
POST /api/v1/contracts/call

→ HTTP 200

// This fails — BlockType.of("0x0") creates a new instance,
// == check fails, findByIndex(0) returns empty, HTTP 400 thrown
POST /api/v1/contracts/call
{ "block": "0x0", ... }
→ HTTP 400 "Unknown block number"

// Reproducer in Java:
BlockType b = BlockType.of("0x0");
assert b != BlockType.EARLIEST;          // true — different object reference
assert b.number() == 0L;                 // true — same numeric value

// But RecordFileServiceImpl uses ==, so the EARLIEST branch is never taken
```
