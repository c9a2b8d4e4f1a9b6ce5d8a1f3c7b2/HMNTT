Audit Report

## Title
Malicious Block Node Operator Can Cause Sustained Block Ingestion Blackout via Controlled Inactivation and Force-Readmit Bypass

## Summary
A configured block node operator can deliberately trigger the `onError()` inactivation path by returning error status codes, then prevent the force-readmit fallback in `getNode()` by returning an empty block range during `serverStatus` queries. When all configured block nodes are simultaneously held inactive this way, the importer cannot ingest blocks for the full `readmitDelay` duration — a minimum of 10 seconds per cycle. The attack is repeatable indefinitely.

## Finding Description

All referenced code locations are confirmed accurate:

**`StreamProperties.java` lines 29–31** enforces a minimum `readmitDelay` of 10 seconds (default 60 seconds): [1](#0-0) 

**`BlockNode.java` lines 196–206** — `onError()` marks a node inactive after `maxSubscribeAttempts` (default 3) consecutive errors and sets `readmitTime`: [2](#0-1) 

**`BlockNode.java` lines 99–112** — `getBlockRange()` returns `EMPTY_BLOCK_RANGE` on any exception or when `firstAvailableBlock == -1`: [3](#0-2) 

**`BlockNodeSubscriber.java` lines 108–131** — `getNode()` fallback loop force-readmits inactive nodes **only if** `hasBlock()` returns `true`; if all inactive nodes return empty block range, throws `BlockStreamException`: [4](#0-3) 

**`BlockNodeSubscriber.java` lines 133–145** — `hasBlock()` calls `getBlockRange()` and returns `false` on empty range, preventing `tryReadmit(true)` from ever being reached: [5](#0-4) 

**Root cause:** The design assumes that when all nodes are inactive, the force-readmit fallback will always find at least one node that can serve the block. This assumption fails when the attacker controls the block node's `serverStatus` responses. The critical gate is at line 124:

```java
if (hasBlock(nextBlockNumber, node)) {
    node.tryReadmit(true);   // ← never reached if attacker returns empty range
    return node;
}
``` [6](#0-5) 

**Exploit flow:**

1. **Phase 1 — Inactivation**: The attacker's block node returns a non-`SUCCESS` status on `subscribeBlockStream`. Each such response triggers `onError()`. After 3 consecutive errors, `active = false` and `readmitTime = now + readmitDelay`. [7](#0-6) 

2. **Phase 2 — Force-readmit bypass**: When `getNode()` enters the fallback loop and calls `hasBlock()` → `getBlockRange()` → `serverStatus`, the attacker's node returns an error or sets `firstAvailableBlock = -1`. `getBlockRange()` catches the exception and returns `EMPTY_BLOCK_RANGE`. `hasBlock()` returns `false`. `tryReadmit(true)` is never called. [8](#0-7) 

3. **Result**: `getNode()` throws `BlockStreamException("No block node can provide block X")`. `CompositeBlockSource` catches it and calls `sourceHealth.onError()`, then waits `frequency` (default 100ms) before retrying — but every retry hits the same dead end until `readmitDelay` expires. [9](#0-8) 

4. **Cycle repeats**: Once `readmitDelay` expires, `tryReadmit(false)` re-activates the node. The attacker immediately causes 3 more errors → node inactive again → repeat indefinitely. [10](#0-9) 

## Impact Explanation

**In `BLOCK_NODE` source mode**: No fallback exists. Block ingestion halts for the full `readmitDelay` per cycle (minimum 10 seconds, default 60 seconds). The attack is repeatable with no rate limiting, causing indefinite periodic blackouts. [11](#0-10) 

**In `AUTO` source mode**: `CompositeBlockSource` switches to `BlockFileSource` after 3 consecutive failures of the block node subscriber (`errors.get() < 3` threshold). This mitigates the impact only if cloud storage is available and up-to-date. [12](#0-11) 

The `@DurationMin(seconds=10)` constraint means the operator cannot reduce the blackout window below 10 seconds even if they wanted to. [1](#0-0) 

## Likelihood Explanation

**Preconditions:**
- The attacker must operate a block node that is configured in the importer's `nodes[]` list.
- For a single-node deployment (common in testing and small production setups), one attacker is sufficient.
- For multi-node deployments, the attacker must control all configured nodes simultaneously — a stronger precondition that reduces likelihood.

**Feasibility**: The attack requires only standard gRPC protocol responses (returning a non-`SUCCESS` status and `firstAvailableBlock=-1`). No cryptographic bypass or privileged access is needed. The attacker is simply a block node operator behaving maliciously. The attack is fully repeatable with no cooldown on the attacker's side. [3](#0-2) 

## Recommendation

The force-readmit fallback should not be gated solely on `hasBlock()` returning `true`. Two complementary fixes:

1. **Decouple force-readmit from `hasBlock()`**: When all active nodes fail and all inactive nodes also fail `hasBlock()`, force-readmit the least-recently-failed inactive node unconditionally and attempt to stream from it. This prevents the attacker from permanently blocking re-admission by controlling `serverStatus` responses.

2. **Track `serverStatus` failures separately from stream failures**: `getBlockRange()` failures during the force-readmit check should not silently prevent re-admission. Consider counting `serverStatus` failures independently and force-readmitting after a bounded number of such failures regardless of the returned range. [13](#0-12) 

## Proof of Concept

```
1. Configure importer with sourceType=BLOCK_NODE and a single attacker-controlled block node.
2. Attacker's node returns SubscribeStreamResponse with status=NOT_AVAILABLE on subscribeBlockStream.
3. After 3 such responses, BlockNode.onError() sets active=false, readmitTime=now+60s.
4. getNode() enters the fallback loop for inactiveNodes.
5. hasBlock() calls getBlockRange() → serverStatus RPC.
6. Attacker's node returns firstAvailableBlock=-1 (or throws an error).
7. getBlockRange() returns EMPTY_BLOCK_RANGE.
8. hasBlock() returns false → tryReadmit(true) is never called.
9. getNode() throws BlockStreamException("No block node can provide block X").
10. CompositeBlockSource.get() catches the exception, calls sourceHealth.onError(), waits 100ms, retries.
11. Steps 4–10 repeat every 100ms for the full 60-second readmitDelay.
12. After 60s, tryReadmit(false) re-activates the node → attacker causes 3 more errors → repeat from step 3.
``` [4](#0-3) [14](#0-13)

    private BlockNode getNode(final AtomicLong nextBlockNumber) {
        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
                inactiveNodes.add(node);
                continue;
            }

            if (hasBlock(nextBlockNumber, node)) {
                return node;
            }
        }

        // find the first inactive node with the block and force activating it
        for (final var node : inactiveNodes) {
            if (hasBlock(nextBlockNumber, node)) {
                node.tryReadmit(true);
                return node;
            }
        }

        throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
    }
