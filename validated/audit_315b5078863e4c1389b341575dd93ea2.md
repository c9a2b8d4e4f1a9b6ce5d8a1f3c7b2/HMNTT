Audit Report

## Title
Cache-Amplified Database DoS via Unbounded Batch Slot Query Accumulation in `findStorageBatch()`

## Summary
`ContractStateServiceImpl.findStorageBatch()` unconditionally accumulates every distinct slot key ever requested for a given contract into a per-contract Caffeine cache (`slotsPerContract`, TTL 5 min, max 1500 entries). On every miss in `contractStateCache` (TTL 2 s), it issues a single SQL `IN`-clause query containing **all** accumulated slot keys for that contract — up to 1500. An unprivileged attacker can prime this cache with fabricated (non-existent) slot keys, then repeatedly trigger 1500-slot batch queries, multiplying database work by up to 1500× per request with no per-user, per-IP, or per-contract rate control.

## Finding Description

**Exact code path:**

`ContractStateServiceImpl.findStorage()` checks `contractStateCache` (TTL `expireAfterWrite=2s`, max 25,000): [1](#0-0) 

On any miss it calls `findStorageBatch()`: [2](#0-1) 

Inside `findStorageBatch()`:
- **Line 86–87**: retrieves or lazily creates a per-contract `CaffeineCache` via `cacheManagerSlotsPerContract.getCache(contractId.toString())`. Because `cacheManagerSlotsPerContract` is configured without `setCacheNames()`, it uses dynamic cache creation — the named cache object is stored permanently in `CaffeineCacheManager`'s internal `ConcurrentHashMap` and is never evicted from the manager itself. [3](#0-2) 

- **Line 90**: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — unconditionally registers the requested slot key into the per-contract cache, including fabricated non-existent keys.
- **Line 91**: collects **all** currently cached slot keys for this contract into `cachedSlots`.
- **Line 103**: issues `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — a native SQL query `WHERE slot IN (:slots)` with up to 1500 parameters. [4](#0-3) 

**Root cause — two mismatched TTLs with no IN-clause size cap:**

| Cache | TTL | Max size |
|---|---|---|
| `contractStateCache` (values) | `expireAfterWrite=2s` | 25,000 |
| `slotsPerContract` (keys) | `expireAfterAccess=5m` | 1,500/contract | [5](#0-4) 

Slot keys accumulate for 5 minutes; their corresponding values expire every 2 seconds. After each 2-second window, every new request to a primed contract misses `contractStateCache` and triggers a full 1500-slot IN query. There is no cap on the IN-clause size before the DB call is issued.

**Why existing checks fail:**

The only throttle is a global token-bucket at `requestsPerSecond=500` and `gasPerSecond` — neither is per-IP, per-contract, nor per-user: [6](#0-5) 

The `gasPerSecond` throttle applies to EVM execution gas, not to the number of DB rows scanned. The `contractSlots` outer cache (`maximumSize=3000`) limits active outer entries but does **not** evict the underlying per-contract caches from `cacheManagerSlotsPerContract` — those persist in the manager's internal map. When an outer entry is evicted and re-created, `cacheManagerSlotsPerContract.getCache(contractId.toString())` returns the existing (still-populated) cache object. [7](#0-6) 

## Impact Explanation

**Baseline DB work**: 500 req/s × 1 slot lookup = 500 slot lookups/s.

**Under attack**: 500 req/s × 1,500 slot lookups = 750,000 slot lookups/s — a **1,500× amplification** in DB row scans. Each IN-clause query forces the DB planner to evaluate a 1,500-element parameter list, increasing query planning overhead. With N primed contracts and concurrent requests, the DB connection pool and I/O bandwidth are saturated. The `db.statementTimeout=3000ms` provides partial protection but does not prevent the amplification from accumulating across many concurrent queries before timeouts fire. [8](#0-7) 

## Likelihood Explanation

No authentication, no special privilege, and no on-chain cost is required. The priming phase costs N×1,500 HTTP requests; at the global limit of 500 req/s, priming 100 contracts takes 300 seconds. The attack is fully repeatable every 2 seconds thereafter for the full 5-minute slot key TTL window. Any public Hedera mirror-node web3 endpoint is reachable by any internet user. The attack is scriptable with a simple HTTP client loop.

## Recommendation

1. **Cap the IN-clause size before issuing the batch query**: In `findStorageBatch()`, limit `cachedSlots` to a configurable maximum (e.g., 100) before calling `contractStateRepository.findStorageBatch()`. Keys beyond the cap should fall back to individual `findStorage()` calls or be silently dropped from the batch.

2. **Align TTLs**: Set `slotsPerContract` TTL equal to or shorter than `contractStateCache` TTL (currently 2s), so slot keys do not outlive their cached values by orders of magnitude.

3. **Evict non-existent slot keys**: After the batch query returns, evict from `contractSlotsCache` any slot key for which no value was found in the DB. This prevents fabricated keys from permanently inflating the batch size.

4. **Add per-contract or per-IP rate limiting**: The global 500 req/s bucket does not prevent a single attacker from consuming the entire budget against a single primed contract.

## Proof of Concept

```python
import requests, threading, time

TARGET = "https://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract_address>"

# Phase 1: Prime — send 1500 requests with distinct fabricated slot keys
# Each request triggers findStorageBatch() which calls putIfAbsent() for the new key
def prime():
    for i in range(1500):
        slot = hex(i).zfill(66)  # 32-byte fabricated slot key
        requests.post(TARGET, json={
            "to": CONTRACT,
            "data": "0x<sload_selector>" + slot[2:],
            "gas": 30000
        })

prime()
time.sleep(3)  # Wait for contractStateCache (2s TTL) to expire

# Phase 2: Trigger — every request now issues WHERE slot IN (1500 params)
# Repeat every 2 seconds for 5 minutes (slotsPerContract TTL window)
def trigger():
    while True:
        requests.post(TARGET, json={
            "to": CONTRACT,
            "data": "0x<sload_selector>" + "00" * 32,
            "gas": 30000
        })
        time.sleep(2)

threads = [threading.Thread(target=trigger) for _ in range(50)]
for t in threads: t.start()
```

Each trigger request causes a `WHERE slot IN (…1500 params…)` query against `contract_state`, amplifying DB load by up to 1500× compared to a normal single-slot lookup.
