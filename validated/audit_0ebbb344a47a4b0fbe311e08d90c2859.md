## Audit Report

## Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` Bypasses Page-Size Cap, Enabling Unbounded DB Queries via `Long.MAX_VALUE` Limit

## Summary
An unauthenticated gRPC caller can set `limit = Long.MAX_VALUE` in a `ConsensusTopicQuery`. This value passes all input validation. Inside `PollingTopicMessageRetriever.poll()`, a narrowing cast `(int)(Long.MAX_VALUE - 0)` overflows to `-1`. `Math.min(-1, maxPageSize)` evaluates to `-1`, which is then used to build a new `TopicMessageFilter` with `limit = -1`. Because `hasLimit()` returns `false` for any value ≤ 0, the repository omits `setMaxResults` entirely, issuing a full-table scan against `topic_message` with no `LIMIT` clause.

## Finding Description

**Step 1 — No upper-bound check in `toFilter()`**

`ConsensusController.toFilter()` assigns the raw protobuf `uint64` field directly to the filter builder with no clamping or rejection of values above a safe threshold:

```java
final var filter = TopicMessageFilter.builder().limit(query.getLimit());
``` [1](#0-0) 

**Step 2 — `TopicMessageFilter` only enforces `@Min(0)`**

`Long.MAX_VALUE` satisfies `@Min(0)` and passes Spring validation without error. There is no `@Max` or upper-bound constraint. [2](#0-1) 

**Step 3 — Integer overflow in `poll()` destroys the page-size cap**

When `filter.getLimit() = Long.MAX_VALUE` and `context.getTotal().get() = 0`:

```
(int)(Long.MAX_VALUE - 0)  →  (int)0x7FFFFFFFFFFFFFFF  →  -1
Math.min(-1, maxPageSize)  →  -1
```

`pageSize` becomes `-1`. Line 75 then builds `newFilter` with `limit(-1)`. [3](#0-2) 

The `@Min(0)` constraint on `TopicMessageFilter.limit` is **not re-enforced** for this internally-constructed `newFilter` because it is created and consumed entirely within `poll()`, never passing through Spring's validation proxy.

**Step 4 — Repository receives a filter with no effective limit**

`hasLimit()` returns `false` when `limit ≤ 0`: [4](#0-3) 

With `hasLimit() == false`, `TopicMessageRepositoryCustomImpl.findByFilter()` skips `setMaxResults` entirely, issuing an unbounded JPA query:

```java
if (filter.hasLimit()) {
    typedQuery.setMaxResults((int) filter.getLimit());  // never reached
}
``` [5](#0-4) 

**Step 5 — `isComplete()` never fires on the limit branch**

`filter.getLimit() == total.get()` evaluates to `Long.MAX_VALUE == <small counter>`, which is permanently `false`. The limit-hit termination path is disabled for the lifetime of the subscription: [6](#0-5) 

## Impact Explanation
Each malicious subscription triggers at least one unbounded `SELECT` against the `topic_message` table with no `LIMIT` clause, holding a JDBC connection for the full duration of the scan. On a mainnet-scale topic with millions of rows, each such query can run for seconds to minutes. A small number of concurrent malicious subscriptions exhausts the connection pool, causing all legitimate `subscribeTopic` calls — including those used by wallets and exchanges to confirm HCS messages — to queue or fail with connection-timeout errors. This is a practical, unauthenticated denial-of-service against the mirror node's gRPC tier.

## Likelihood Explanation
The `subscribeTopic` gRPC endpoint is publicly reachable with no authentication. Setting `limit` to `Long.MAX_VALUE` in a protobuf `uint64` field requires only a standard gRPC client and a single line of code. The attack is trivially repeatable and scriptable; a single attacker with a modest number of parallel connections can sustain the DoS indefinitely.

## Recommendation

**Primary fix — clamp the limit in `toFilter()` before it enters the system:**

```java
private static final long MAX_LIMIT = 10_000L; // or a configurable property

private TopicMessageFilter toFilter(ConsensusTopicQuery query) {
    long rawLimit = query.getLimit();
    long safeLimit = (rawLimit == 0) ? 0 : Math.min(rawLimit, MAX_LIMIT);
    final var filter = TopicMessageFilter.builder().limit(safeLimit);
    // ...
}
``` [7](#0-6) 

**Secondary fix — add an `@Max` constraint to `TopicMessageFilter.limit`:**

```java
@Min(0)
@Max(10_000)
private long limit;
``` [2](#0-1) 

**Tertiary fix — guard the narrowing cast in `poll()` against overflow:**

```java
long remaining = filter.getLimit() - context.getTotal().get();
int limit = filter.hasLimit()
        ? (int) Math.min(remaining, Integer.MAX_VALUE)
        : Integer.MAX_VALUE;
``` [8](#0-7) 

## Proof of Concept

```java
// Standard gRPC client — no authentication required
ManagedChannel channel = ManagedChannelBuilder.forAddress("mirror-node-host", 5600)
        .usePlaintext()
        .build();

ConsensusServiceGrpc.ConsensusServiceBlockingStub stub =
        ConsensusServiceGrpc.newBlockingStub(channel);

ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
        .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
        .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
        .setLimit(Long.MAX_VALUE)   // 0x7FFFFFFFFFFFFFFF — passes @Min(0)
        .build();

// Each call triggers poll() → (int)(Long.MAX_VALUE - 0) = -1
// → hasLimit() false → no LIMIT clause → full table scan
Iterator<ConsensusTopicResponse> it = stub.subscribeTopic(query);
// Repeat in parallel threads to exhaust the DB connection pool
```

**Trace through the overflow:**
1. `filter.getLimit()` = `9223372036854775807L` (`Long.MAX_VALUE`)
2. `context.getTotal().get()` = `0L`
3. `(int)(9223372036854775807L - 0L)` = `(int)(0x7FFFFFFFFFFFFFFFL)` = `-1`
4. `Math.min(-1, maxPageSize)` = `-1`
5. `newFilter.limit` = `-1` → `hasLimit()` = `false`
6. `typedQuery.setMaxResults(...)` is never called → full-table scan issued
