Audit Report

## Title
Integer Overflow in `poll()` Bypasses Database Query Limit, Enabling Unbounded Result Set Fetch

## Summary
In `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`, a narrowing cast from `long` to `int` silently overflows when `filter.getLimit()` exceeds `Integer.MAX_VALUE`. The resulting negative `pageSize` is written into an internally-constructed `TopicMessageFilter` via the Lombok builder, bypassing Bean Validation. In `TopicMessageRepositoryCustomImpl.findByFilter()`, the `hasLimit()` guard evaluates to `false` for the negative value, so `setMaxResults()` is never called and the JPA query executes without any `LIMIT` clause, materializing the entire matching result set into JVM heap.

## Finding Description

**Step 1 — No upper-bound cap on the incoming `limit`:**

`ConsensusController.toFilter()` maps the raw proto `uint64 limit` field directly to `TopicMessageFilter.limit` with no cap: [1](#0-0) 

The proto definition confirms `limit` is `uint64`, accepting any value from 0 to 2^64−1: [2](#0-1) 

**Step 2 — `@Min(0)` does not reject `Long.MAX_VALUE`:**

`TopicMessageFilter.limit` carries only `@Min(0)`, which `Long.MAX_VALUE` satisfies: [3](#0-2) 

**Step 3 — Narrowing cast overflows in `PollingTopicMessageRetriever.poll()`:**

```
(int)(Long.MAX_VALUE - 0L)
= (int) 0x7FFFFFFFFFFFFFFFL
= 0xFFFFFFFF (lower 32 bits)
= -1 (signed int)
``` [4](#0-3) 

`Math.min(-1, maxPageSize)` returns `-1` because `-1 < maxPageSize` for any valid `maxPageSize` (minimum 32 per `@Min(32)` on `RetrieverProperties.maxPageSize`).

**Step 4 — Negative `pageSize` written into `newFilter` without re-validation:**

The internally-constructed filter is built via the Lombok builder directly; no Spring AOP intercepts this call, so Bean Validation is not triggered: [5](#0-4) 

**Step 5 — `hasLimit()` returns `false` for negative limit:**

`hasLimit()` uses a `> 0` check. With `limit = -1`, it returns `false`: [6](#0-5) 

**Step 6 — `setMaxResults()` is never called; unbounded `getResultList()` executes:** [7](#0-6) 

**Step 7 — Identical overflow in `PollingTopicListener.poll()`:** [8](#0-7) 

**Why existing checks fail:**
- `@Min(0)` on `TopicMessageFilter.limit` only rejects negative user-supplied values; `Long.MAX_VALUE` passes.
- `@Validated` on `TopicMessageServiceImpl` validates the original filter (which has a positive `Long.MAX_VALUE`), not the internally-constructed `newFilter` with `limit = -1`.
- `Math.min(limit, maxPageSize)` does not protect against a negative `limit`; it returns the negative value unchanged.
- `flux.take(filter.getLimit())` in `TopicMessageServiceImpl` applies `Long.MAX_VALUE` as the take count, which is effectively unlimited: [9](#0-8) 
- `isComplete()` in `PollingContext` checks `filter.getLimit() == total.get()`, which is never true when `filter.getLimit() = Long.MAX_VALUE`, so polling continues indefinitely (or until timeout): [10](#0-9) 

## Impact Explanation
Each call to `poll()` with an overflowed limit causes `findByFilter()` to execute a `SELECT` against the `topic_message` table with no `LIMIT` clause. The entire matching result set is materialized into JVM heap via `getResultList()`. On a network with a large message history for a given topic, this can exhaust server memory, degrade database performance for all concurrent users, and cause cascading failures. The `retrieverProperties.getTimeout()` provides a time-based backstop, but heap pressure and database load occur before the timeout fires. Multiple concurrent requests amplify the impact into a reliable denial of service. [11](#0-10) 

## Likelihood Explanation
The gRPC `subscribeTopic` endpoint is publicly accessible. Any client can craft a `ConsensusTopicQuery` protobuf message with `limit` set to any value ≥ `2147483648L` (i.e., greater than `Integer.MAX_VALUE`) to trigger the overflow. The exploit requires no special knowledge beyond the public HCS gRPC API specification and is trivially scriptable. The proto field is `uint64`, so the full range of large values is reachable from any standard gRPC client. [12](#0-11) 

## Recommendation
1. **Cap the limit before the cast.** In `PollingTopicMessageRetriever.poll()` and `PollingTopicListener.poll()`, clamp the remaining limit to `Integer.MAX_VALUE` before the narrowing cast:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = (int) Math.min(remaining, Integer.MAX_VALUE);
   ```
2. **Add an upper-bound constraint on `TopicMessageFilter.limit`.** Add `@Max(Integer.MAX_VALUE)` (or a domain-appropriate maximum) alongside `@Min(0)` to reject oversized values at the validation boundary.
3. **Guard `hasLimit()` against negative values.** Change `hasLimit()` to `return limit > 0` is already correct, but the root issue is that a negative value should never reach it; the cast must be fixed first.
4. **Add a regression test** that passes `limit = Long.MAX_VALUE` through the full stack and asserts that `setMaxResults()` is called with a bounded positive value.

## Proof of Concept
```java
// Attacker sends a gRPC ConsensusTopicQuery with limit = Long.MAX_VALUE
ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).build())
    .setLimit(Long.MAX_VALUE)  // uint64: passes @Min(0), overflows to int -1
    .build();

// Inside PollingTopicMessageRetriever.poll():
// filter.getLimit() = Long.MAX_VALUE, context.getTotal().get() = 0
// (int)(Long.MAX_VALUE - 0L) = (int)0x7FFFFFFFFFFFFFFFL = 0xFFFFFFFF = -1
// Math.min(-1, maxPageSize) = -1
// newFilter.limit = -1
// newFilter.hasLimit() = (-1 > 0) = false
// → setMaxResults() never called
// → SELECT * FROM topic_message WHERE topic_id=1 AND consensus_timestamp>=0 ORDER BY consensus_timestamp
//   (no LIMIT clause — fetches entire table partition into heap)
```
