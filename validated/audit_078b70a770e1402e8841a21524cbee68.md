Audit Report

## Title
Unauthenticated Distributed Resource Exhaustion via Unthrottled Triple-DB-Query Endpoint at `/api/v1/topics/{id}`

## Summary
`TopicController.getTopic()` issues three independent database queries per HTTP request. The rest-java Helm chart ships with no `rateLimit` or `inFlightReq` middleware, and the entire middleware chain is disabled by default (`global.middleware: false`). No application-level throttling exists in the rest-java Spring context. A distributed attacker can collectively exhaust the HikariCP connection pool with no per-source-IP guard firing.

## Finding Description

**Exact code path — confirmed:**

`TopicController.getTopic()` at lines 31–37 executes three sequential, blocking repository calls per single HTTP request:

```java
var topic     = topicService.findById(id.id());    // DB query 1
var entity    = entityService.findById(id.id());   // DB query 2
var customFee = customFeeService.findById(id.id()); // DB query 3
``` [1](#0-0) 

**Missing rate-limit middleware in rest-java — confirmed:**

The rest-java `values.yaml` middleware block contains only `circuitBreaker` and `retry`; there is no `rateLimit` or `inFlightReq` entry:

```yaml
middleware:
  - circuitBreaker:
      expression: NetworkErrorRatio() > 0.10 || ResponseCodeRatio(500, 600, 0, 600) > 0.25
  - retry:
      attempts: 3
      initialInterval: 100ms
``` [2](#0-1) 

**Middleware disabled by default — confirmed:**

The guard condition in the middleware template is `{{ if and .Values.global.middleware .Values.middleware }}`, and `global.middleware` defaults to `false`: [3](#0-2) [4](#0-3) 

This means that even the `circuitBreaker` and `retry` are inactive in a default deployment. When an operator does enable middleware (`global.middleware: true`), there is still no rate-limit protection because neither `rateLimit` nor `inFlightReq` is present in the chain.

**Rosetta has both guards — confirmed (disparity is intentional gap, not design parity):**

The rosetta chart explicitly includes `inFlightReq` (per-IP) and `rateLimit` (per-host) in its middleware list: [5](#0-4) 

**No application-level throttling in rest-java — confirmed:**

`ThrottleConfiguration` and `ThrottleManagerImpl` exist exclusively under the `web3` module: [6](#0-5) 

They are not present in the rest-java Spring context.

**GCP backend policy is insufficient — confirmed:**

`maxRatePerEndpoint: 250` is a per-pod aggregate cap, not a per-source-IP limit. It is also gated behind `global.gateway.enabled: false` by default: [7](#0-6) [8](#0-7) 

## Impact Explanation

Each request to `getTopic()` holds up to 3 HikariCP connections for the duration of the three sequential queries. Under distributed load, the connection pool saturates, causing cascading timeouts and `EntityNotFoundException` errors for all users of the rest-java service (topics, network, accounts endpoints). The `RestJavaHighDBConnections` Prometheus alert fires at 75% pool utilization but triggers no automatic mitigation: [9](#0-8) 

When middleware is enabled, `retry: attempts: 3` worsens the situation by re-issuing failed requests up to 3× per original request under transient failures, amplifying effective DB load.

## Likelihood Explanation

The endpoint `/api/v1/topics/{id}` is publicly routable with no authentication, API key, or CAPTCHA. Cloud function platforms (AWS Lambda, GCP Cloud Functions) trivially provide hundreds of distinct source IPs at negligible cost. The attack requires only the ability to issue HTTP GET requests and is repeatable indefinitely with no persistent attacker-side state.

## Recommendation

1. **Add `inFlightReq` and `rateLimit` to the rest-java middleware list** in `charts/hedera-mirror-rest-java/values.yaml`, mirroring the rosetta configuration.
2. **Enable middleware by default** (`global.middleware: true`) or document the security implication of leaving it disabled.
3. **Parallelize the three DB calls** in `TopicController.getTopic()` (e.g., using `CompletableFuture`) to reduce the per-request connection hold time from 3 sequential slots to 1 concurrent window.
4. **Add a per-user-IP connection limit** at the HikariCP or pgBouncer layer so a single source cannot monopolize the pool.
5. **Wire a throttle mechanism** (analogous to `ThrottleConfiguration`/`ThrottleManagerImpl` in `web3`) into the rest-java Spring context.

## Proof of Concept

```bash
# Distribute across N IPs (cloud functions, proxies, etc.)
# Each IP sends moderate traffic; collectively saturates the pool

for i in $(seq 1 100); do
  curl -s "https://<mirror-node-host>/api/v1/topics/0.0.$i" &
done
wait

# Each request triggers 3 HikariCP connections.
# 100 concurrent requests = up to 300 simultaneous DB connections.
# Default HikariCP pool size (10) saturates immediately;
# even a larger pool (e.g., 50) saturates at ~17 concurrent requests.
# No per-IP guard fires because rateLimit/inFlightReq middleware is absent.
```
