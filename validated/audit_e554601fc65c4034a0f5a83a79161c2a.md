# Audit Report

## Title
Spoofed `X-Forwarded-For` Header Poisons `req.ip` in Audit Log via Unrestricted Proxy Trust

## Summary
`rest/server.js` sets `trust proxy: true`, causing Express to unconditionally accept the leftmost value of any `X-Forwarded-For` header as `req.ip`. The Helm chart simultaneously configures Traefik with `forwardedHeaders.insecure`, disabling header stripping for external clients. The resulting attacker-controlled `req.ip` is written verbatim into the access log by `responseHandler.js`, enabling both IP spoofing and newline-based log injection against the audit trail.

## Finding Description

**Root cause 1 — Express trusts all proxies unconditionally**

`rest/server.js` line 58:
```js
app.set('trust proxy', true);
```
Per Express documentation, the boolean `true` causes `req.ip` to be resolved as the **leftmost** entry in the `X-Forwarded-For` chain, regardless of how many hops are present or whether they are trusted. [4](#0-3) 

**Root cause 2 — Traefik passes client-supplied forwarded headers unmodified**

`charts/hedera-mirror-common/values.yaml` contains `forwardedHeaders.insecure` (4 occurrences). This Traefik flag disables the built-in protection that would otherwise strip `X-Forwarded-For` headers arriving from untrusted external sources. Instead, Traefik appends the real client IP to whatever value the client already sent. [3](#0-2) 

**Root cause 3 — Tainted value logged without sanitization**

`rest/middleware/responseHandler.js` line 57:
```js
logger.info(`${req.ip} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${code}`);
```
`req.ip` is interpolated directly into the log string with no IP-format validation, no escaping, and no newline stripping. [2](#0-1) 

**Exploit flow:**
```
Attacker sends:
  X-Forwarded-For: 10.0.0.1\nINFO fake-entry GET /admin 200

Traefik (forwardedHeaders.insecure):
  passes header through, appends real IP → X-Forwarded-For: 10.0.0.1\nINFO ..., <real-ip>

Express (trust proxy: true):
  req.ip = "10.0.0.1\nINFO fake-entry GET /admin 200"

responseHandler.js line 57:
  logger.info("10.0.0.1\nINFO fake-entry GET /admin 200 GET /api/v1/... 200")

Log output: two lines — one fully forged, one real
```

## Impact Explanation
Every route handled by `responseHandler` (accounts, balances, contracts, blocks, schedules, tokens, topics, transactions) is affected. An attacker can:
- Attribute any API request to an arbitrary IP address, undermining IP-based forensic investigation.
- Inject synthetic log lines (via `\n`) to fabricate or obscure audit events, making post-incident analysis unreliable.
- Cause log-parsing pipelines (SIEM, alerting) to misinterpret injected lines as legitimate structured log entries.

## Likelihood Explanation
No authentication, special network position, or prior access is required. Any client that can reach the public API endpoint can send a crafted `X-Forwarded-For` header with a single `curl` command. Both misconfigurations (`trust proxy: true` and `forwardedHeaders.insecure`) are present in the default server bootstrap and the default Helm chart values, meaning every standard deployment is affected out of the box.

## Recommendation

1. **Restrict proxy trust to known infrastructure.** Replace the boolean `true` with the specific number of trusted proxy hops or the explicit IP/CIDR of the Traefik ingress:
   ```js
   // Trust exactly one proxy hop (the Traefik ingress)
   app.set('trust proxy', 1);
   ``` [1](#0-0) 

2. **Remove `forwardedHeaders.insecure` from the Traefik configuration** in `charts/hedera-mirror-common/values.yaml` so Traefik strips client-supplied forwarded headers from untrusted sources. [3](#0-2) 

3. **Sanitize `req.ip` before logging.** Strip or escape control characters (especially `\n`, `\r`) before interpolating into log strings in `responseHandler.js` and `httpErrorHandler.js`:
   ```js
   const safeIp = (req.ip ?? '').replace(/[\r\n]/g, '_');
   logger.info(`${safeIp} ${req.method} ${req.originalUrl} in ${elapsed} ms: ${code}`);
   ``` [5](#0-4) 

## Proof of Concept
```bash
curl -H "X-Forwarded-For: 1.2.3.4"$'\n'"INFO spoofed GET /admin HTTP/1.1 200" \
     https://<mirror-node-host>/api/v1/contracts/results/0x<hash>
```
The access log will contain a fabricated line attributed to `1.2.3.4` followed by the injected `INFO spoofed GET /admin HTTP/1.1 200` entry, with the real request appearing on a separate line attributed to the actual client IP appended by Traefik.
