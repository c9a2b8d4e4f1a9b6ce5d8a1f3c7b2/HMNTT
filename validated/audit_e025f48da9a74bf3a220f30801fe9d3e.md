Audit Report

## Title
Cache TTL Asymmetry in `findStorageBatch()` Enables Unauthenticated DB Query Amplification DoS

## Summary
An unauthenticated attacker can exploit the TTL mismatch between the `slotsPerContract` cache (5-minute expiry) and the `contractStateCache` (2-second expiry) in `ContractStateServiceImpl`. By first populating a per-contract slot key list with up to 1,500 entries, then issuing one request every ≥2 seconds, the attacker forces a DB `SELECT ... WHERE slot IN (...)` with up to 1,500 parameters on every cache miss — indefinitely — with no per-contract or per-slot rate control.

## Finding Description

**Verified code location:**
`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`

**Verified cache configuration** (`web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`):
- `contractSlots` (outer map: contractId → per-contract CaffeineCache): `expireAfterAccess=5m, maximumSize=3000`
- `slotsPerContract` (per-contract slot key list): `expireAfterAccess=5m, maximumSize=1500`
- `contractState` (slot value cache): `expireAfterWrite=2s, maximumSize=25000` [1](#0-0) 

**Root cause — verified failed assumption:**
`findStorageBatch()` at line 90 adds the requested slot key to the per-contract `contractSlotsCache`, then at line 91 reads back **all** currently cached slot keys for that contract, and at line 103 issues a single batch DB query with all of them. The design assumes the slot key list is only useful while the corresponding `contractStateCache` entries are alive. In reality, `slotsPerContract` entries survive for 5 minutes while `contractStateCache` entries expire every 2 seconds. There is no mechanism to shrink or evict the slot key list when `contractStateCache` entries expire. [2](#0-1) 

**Exploit flow:**

*Phase 1 — Slot list inflation (~1,500 requests):*
The attacker calls the public `eth_call` endpoint cycling through 1,500 distinct slot keys for contract C. Each call misses `contractStateCache` (line 63), falls through to `findStorageBatch()` (line 69), and adds the key to the per-contract cache (line 90). After 1,500 requests, the per-contract `contractSlotsCache` holds 1,500 slot keys, persisting for 5 minutes from last access. [3](#0-2) 

*Phase 2 — Sustained amplification (1 request every ≥2 seconds):*
The attacker sends a single request for any slot of contract C. Because `contractStateCache` expired (2s TTL), `findStorage()` misses at line 63 and calls `findStorageBatch()`. `findStorageBatch()` reads all 1,500 keys from the per-contract cache (line 91) and issues one DB query with a 1,500-element `IN` clause (line 103). Results are written back to `contractStateCache` (line 109), which expires again in 2 seconds. The cycle repeats indefinitely. [4](#0-3) 

**Why existing checks are insufficient:**
`ThrottleManagerImpl.throttle()` consumes exactly 1 token from `rateLimitBucket` per HTTP request and checks gas via `gasLimitBucket`. It has no awareness of how many DB slot lookups a single request triggers. One allowed request can trigger a 1,500-slot batch query. [5](#0-4) 

The `findStorageBatch` repository method issues an unbounded `slot IN (:slots)` query with no server-side cap on list size. [6](#0-5) 

## Impact Explanation
Each attacker request (within the global rate limit) triggers a DB `SELECT ... WHERE slot IN (...)` with up to 1,500 parameters — a **1,500x amplification factor** per request. With the global rate limit at 500 req/s, the realistic worst-case is 500 × 1,500 = **750,000 effective slot lookups per second** against PostgreSQL from a single attacker. Even a fraction of this load is sufficient to saturate the DB connection pool and degrade or halt contract call processing for all users.

Note: The claim's figure of "2.25 billion slot lookups per second" is arithmetically inconsistent with the 500 req/s global limit and should be disregarded; the 750,000/s figure is the realistic ceiling.

## Likelihood Explanation
The web3 JSON-RPC endpoint is publicly accessible with no authentication. Phase 1 requires ~1,500 requests (achievable in 3 seconds at the 500 req/s limit). Phase 2 requires one request every 2 seconds. The attack is trivially automatable with standard HTTP tooling, requires no privileged access, no on-chain transactions, and no special contract knowledge. Slot keys can be guessed (sequential integers are common EVM storage patterns) or enumerated from public chain data.

## Recommendation
1. **Cap batch query size:** Enforce a configurable maximum on the number of slot keys passed to `findStorageBatch()` (e.g., 100), regardless of how many are in the per-contract cache.
2. **Align TTLs or add coordinated eviction:** Either reduce `slotsPerContract` TTL to match `contractState` (2s), or implement a listener that evicts slot keys from the per-contract cache when their corresponding `contractStateCache` entries expire.
3. **Per-contract rate limiting:** Add a per-contract (or per-IP-per-contract) rate limit in `ThrottleManagerImpl` to prevent a single client from repeatedly triggering batch queries for the same contract.
4. **Disable by default or gate behind auth:** Consider setting `enableBatchContractSlotCaching=false` by default until the above mitigations are in place, since the flag at line 59 already provides a clean kill switch. [7](#0-6) 

## Proof of Concept

```python
import requests, time

RPC = "http://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract>"

# Phase 1: populate 1500 slot keys
for i in range(1500):
    slot = hex(i).zfill(66)  # 32-byte zero-padded slot key
    requests.post(RPC, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [CONTRACT, slot, "latest"], "id": i
    })

# Phase 2: one request every 2s forces a 1500-slot batch query indefinitely
while True:
    requests.post(RPC, json={
        "jsonrpc": "2.0", "method": "eth_getStorageAt",
        "params": [CONTRACT, "0x0", "latest"], "id": 9999
    })
    time.sleep(2)
```

Each iteration of Phase 2 causes `findStorageBatch()` to execute:
```sql
SELECT slot, value FROM contract_state
WHERE contract_id = <id>
AND slot IN (<1500 slot keys>)
```
against the database, with no throttle on the DB-side amplification.
