# Audit Report

## Title
Unbounded Timestamp Range in `getHookStorage()` Enables Full Historical Table Scan DoS

## Summary
`HooksController.getHookStorage()` accepts user-supplied `timestamp` parameters with no range-width validation. `Bound.of()` is invoked with `primarySortField=false`, permanently bypassing the only range sanity check. The resulting `[0, Long.MAX_VALUE]` range is passed directly to a `DISTINCT ON (key)` SQL query that must scan every matching row in `hook_storage_change` before applying `LIMIT`. No rate limiting exists in the `rest-java` module, so any unauthenticated caller can repeatedly trigger full cross-partition scans.

## Finding Description

**Root cause — `Bound.of()` hardcodes `primarySortField=false`:**

`HooksController.hookStorageChangeRequest()` calls:
```java
Bound.of(timestamps, TIMESTAMP, HookStorageChange.HOOK_STORAGE_CHANGE.CONSENSUS_TIMESTAMP);
``` [2](#0-1) 

`Bound.of()` always passes `false` as `primarySortField`:
```java
return new Bound(timestamp, false, parameterName, field);
``` [9](#0-8) 

Inside the `Bound` constructor, the only range-width guard is:
```java
if (primarySortField && adjustedLower > adjustedUpper) {
    throw new IllegalArgumentException(...);
}
``` [10](#0-9) 

Because `primarySortField` is always `false` here, this check is dead code for this path. No other range-width validation exists.

**Open upper bound defaults to `Long.MAX_VALUE`:**

When only a lower bound is supplied (e.g., `timestamp=gte:0`), `adjustUpperBound()` returns `Long.MAX_VALUE`:
```java
if (this.upper == null) {
    return Long.MAX_VALUE;
}
``` [5](#0-4) 

**Unbounded values flow directly to the repository:**

`HookServiceImpl.getHookStorageChange()` extracts these values without any additional validation:
```java
final long timestampLowerBound = timestamp.getAdjustedLowerRangeValue();
final long timestampUpperBound = timestamp.adjustUpperBound();
``` [11](#0-10) 

**`DISTINCT ON (key)` forces a full scan before `LIMIT`:**

The repository executes:
```sql
select distinct on (key) ...
from hook_storage_change
where owner_id = :ownerId
  and hook_id = :hookId
  and key >= :keyLowerBound
  and key <= :keyUpperBound
  and consensus_timestamp between :timestampLowerBound and :timestampUpperBound
``` [12](#0-11) 

PostgreSQL's `DISTINCT ON` must read **all** rows matching the predicate to determine the latest `consensus_timestamp` per `key` before it can apply `LIMIT`. With a range of `[0, Long.MAX_VALUE]`, every time-partition of `hook_storage_change` is touched for the given `(owner_id, hook_id)`.

**No rate limiting in `rest-java`:**

`ThrottleConfiguration` and `ThrottleManager` are confined to the `web3` module. No equivalent HTTP-level throttling exists for `HooksController` or anywhere else in `rest-java`. The Node.js REST API enforces `maxTimestampRange` / `maxTimestampRangeNs` and throws `InvalidArgumentError` when the range is too wide, but no such check exists in the `rest-java` path.

## Impact Explanation
Each request with `timestamp=gte:0` forces a full cross-partition sequential scan of `hook_storage_change` for the targeted `(owner_id, hook_id)`. As the table grows with network history, each such query consumes proportionally more DB CPU, I/O, and memory. Concurrent flooding from multiple source IPs can saturate DB connection pools and query workers, degrading or halting consensus-timestamp-indexed query processing across all services sharing the same DB cluster.

## Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. Both `ownerId` and `hookId` are discoverable via `GET /api/v1/accounts/{ownerId}/hooks`. No special privileges, tokens, or cryptographic material are required. The attack is trivially scriptable with standard HTTP tooling (e.g., `curl` in a loop or any load-testing tool).

## Recommendation
1. **Enforce a maximum timestamp range width** in `hookStorageChangeRequest()` or inside `Bound.of()` when called for the timestamp field. Reject requests where `adjustedUpper - adjustedLower` exceeds a configurable `maxTimestampRangeNs` (mirroring the Node.js REST API behavior in `rest/utils.js`).
2. **Require an upper bound** when a lower bound is supplied for the timestamp parameter on this endpoint, or default the upper bound to `lower + maxTimestampRangeNs` rather than `Long.MAX_VALUE`.
3. **Add rate limiting** to the `rest-java` module for endpoints that trigger historical queries, analogous to the `ThrottleConfiguration`/`ThrottleManager` infrastructure in `web3`.
4. **Consider a covering index** on `(owner_id, hook_id, key, consensus_timestamp DESC)` to allow PostgreSQL to satisfy `DISTINCT ON (key)` via an index scan rather than a full table scan, reducing the blast radius of wide-range queries.

## Proof of Concept
```bash
# Step 1: Discover a valid ownerId and hookId
curl "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks"
# Response contains hookId values, e.g., 42

# Step 2: Trigger full cross-partition scan repeatedly
while true; do
  curl "https://<mirror-node>/api/v1/accounts/0.0.1234/hooks/42/storage?timestamp=gte:0&limit=100" &
done
```
Each request causes PostgreSQL to scan all partitions of `hook_storage_change` for `(owner_id=1234, hook_id=42)` with `consensus_timestamp BETWEEN 0 AND 9223372036854775807`. Concurrent requests from multiple IPs exhaust DB CPU and connection pool resources.
