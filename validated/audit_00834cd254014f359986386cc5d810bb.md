Audit Report

## Title
Address Book Files Lack Timestamp Normalization, Rendering Cache Ineffective and Enabling Per-Request DB Amplification

## Summary
`SystemFileLoader.load(FileID, long)` normalizes the consensus timestamp to the hour boundary only for exchange-rate and fee-schedule files. Address-book files 101 and 102 receive no such normalization. Because `FileReadableKVState.readFromDataSource()` always supplies the current wall-clock nanosecond as the timestamp for every system-file lookup, every request produces a unique cache key for address-book files, making the `defaultSystemFileCacheManager` permanently ineffective for those files. Each request that touches the address book therefore unconditionally issues one SQL query and one protobuf deserialization.

## Finding Description

**`FileReadableKVState.readFromDataSource()`** always calls `systemFileLoader.load(key, currentTimestamp)` for system files, where `currentTimestamp` is obtained from `Utils.getCurrentTimestamp()`:

```java
public static long getCurrentTimestamp() {
    final var now = Instant.now();
    return DomainUtils.convertToNanos(now.getEpochSecond(), now.getNano());
}
``` [1](#0-0) 

This is called unconditionally for every system-file lookup, regardless of whether a historical timestamp is available in `ContractCallContext`: [2](#0-1) 

Inside `SystemFileLoader.load()`, the timestamp-rounding branch fires **only** for `exchangeRateFileId`, `feeSchedulesFileId`, and `simpleFeeSchedulesFileId`:

```java
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp);
}
``` [3](#0-2) 

`addressBookFile101` and `addressBookFile102` fall through without rounding. The cache key is then built from the raw nanosecond value: [4](#0-3) 

Because wall-clock nanoseconds advance with every request, `cache.get(cacheKey, File.class)` always returns `null`, and execution falls through to `loadFromDB()` → `loadWithRetry()`: [5](#0-4) 

`loadWithRetry()` unconditionally calls `fileDataRepository.getFileAtTimestamp()` and, when data is present, invokes `NodeAddressBook.PROTOBUF.parse()`: [6](#0-5) 

The address-book files are registered with `NodeAddressBook.PROTOBUF` as their codec in `loadAll()`: [7](#0-6) 

The `defaultSystemFileCacheManager` is backed by the `systemFile` cache spec `expireAfterWrite=10m,maximumSize=20,recordStats`: [8](#0-7) [9](#0-8) 

`maximumSize=20` prevents unbounded memory growth but does nothing to prevent the per-request DB + parse cost, since the cache is never hit for address-book files.

## Impact Explanation
Every HTTP request that causes the EVM to read `addressBookFile101` or `addressBookFile102` unconditionally issues one SQL query (`getFileAtTimestamp`) and, when DB data exists, one full `NodeAddressBook.PROTOBUF` deserialization. The intended caching layer provides zero relief. At the documented default rate limit of 500 RPS, a sustained flood of such requests produces up to 500 DB queries/second and 500 protobuf parses/second for address-book data alone, amplifying both DB connection pool pressure and CPU load. The `statementTimeout=3000 ms` and `maximumSize=20` provide no mitigation against this throughput. [10](#0-9) [11](#0-10) [12](#0-11) 

## Likelihood Explanation
No privilege is required. Any caller able to submit `eth_call` or equivalent JSON-RPC requests can trigger address-book reads. The exploit requires no special knowledge beyond knowing that the mirror-node exposes a public JSON-RPC endpoint. It is trivially repeatable and scriptable, and the rate limit (500 RPS) is the only throttle standing between the attacker and sustained DB amplification.

## Recommendation
Apply the same `roundDownToHour` normalization (or an equivalent coarser-grained normalization) to `addressBookFile101` and `addressBookFile102` before constructing the `CacheKey`. Since address-book data changes infrequently, rounding to the hour (or even to the day) is semantically safe and consistent with the treatment of the other high-frequency system files. Alternatively, address-book files could be served from an in-memory genesis/mock value (as is already done for `networkPropertyFile`) and bypass the DB entirely, since their content is stable within a deployment.

## Proof of Concept
1. Identify any `eth_call` request path that causes the EVM to read `FileID` 0.0.101 or 0.0.102.
2. Send repeated requests at the rate limit (500 RPS).
3. Observe via DB metrics that `getFileAtTimestamp` is called once per request for the address-book entity IDs, with no cache hits, confirming the cache key space is unbounded due to the nanosecond timestamp.
4. Contrast with exchange-rate file lookups at the same rate: those produce at most one DB query per hour due to `roundDownToHour` normalization.
