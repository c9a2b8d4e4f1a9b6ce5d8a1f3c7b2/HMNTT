# Audit Report

## Title
Unbounded Concurrent Throttled Subscriptions Exhaust Shared `boundedElastic()` Scheduler, Starving Legitimate Topic Message Delivery

## Summary
`PollingTopicMessageRetriever` stores a reference to the JVM-wide shared `Schedulers.boundedElastic()` instance. Every throttled subscription (`throttled=true`) sets `numRepeats = Long.MAX_VALUE` and indefinitely re-executes a blocking JDBC poll on that shared scheduler every 2 seconds. Because the only server-side guard is a per-connection call limit (not a global subscription or connection cap), an attacker can open an unbounded number of TCP connections and saturate the shared scheduler, preventing legitimate subscribers from ever scheduling their poll tasks.

## Finding Description

**Exact code path:**

`PollingTopicMessageRetriever` constructor stores the shared scheduler:
```java
// PollingTopicMessageRetriever.java:41
scheduler = Schedulers.boundedElastic();
``` [1](#0-0) 

When `throttled=true`, `numRepeats` is set to `Long.MAX_VALUE`:
```java
// PollingTopicMessageRetriever.java:98-99
if (throttled) {
    numRepeats = Long.MAX_VALUE;
``` [11](#0-10) 

The `repeatWhen` operator re-subscribes on that scheduler after each 2-second delay:
```java
// PollingTopicMessageRetriever.java:52-55
.repeatWhen(RepeatSpec.create(r -> !context.isComplete(), context.getNumRepeats())
        .jitter(0.1)
        .withFixedDelay(context.getFrequency())
        .withScheduler(scheduler))
``` [3](#0-2) 

Each re-subscription executes a blocking JDBC call on the scheduler thread:
```java
// PollingTopicMessageRetriever.java:78
return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
``` [4](#0-3) 

Infinite retry amplifies pressure when the scheduler is full:
```java
// PollingTopicMessageRetriever.java:58
.retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1)))
``` [5](#0-4) 

The same shared scheduler is also used by `TopicMessageServiceImpl` for the safety check:
```java
// TopicMessageServiceImpl.java:70
.subscribeOn(Schedulers.boundedElastic());
``` [12](#0-11) 

The only server-side guard is per-connection, not global:
```java
// GrpcConfiguration.java:33
serverBuilder.maxConcurrentCallsPerConnection(nettyProperties.getMaxConcurrentCallsPerConnection());
``` [13](#0-12) 

No `maxConnectionAge`, `maxConnectionIdle`, per-IP limit, or total connection limit is configured anywhere in `GrpcConfiguration`. [14](#0-13) 

**Root cause:** `Schedulers.boundedElastic()` returns the **same singleton** `BoundedElasticScheduler` instance on every call (this is Reactor's documented behavior). The design assumes `maxConcurrentCallsPerConnection` bounds total scheduler load — it does not. It only limits calls per TCP connection. There is no global concurrent-subscription cap.

**Failed assumption:** The per-connection limit of `maxConcurrentCallsPerConnection` is assumed to be a sufficient global guard. It is not: an attacker can open `C` connections and get `C × maxConcurrentCallsPerConnection` concurrent throttled subscriptions, each permanently and periodically occupying a `boundedElastic()` worker thread.

**Condition for indefinite looping:** The `repeatWhen` continues as long as `!context.isComplete()`. For throttled subscriptions, `isComplete()` returns `false` when the last page returned exactly `maxPageSize` (1000) messages. An attacker subscribing to any high-traffic topic with a large message backlog will satisfy this condition, keeping subscriptions alive indefinitely. [10](#0-9) 

## Impact Explanation
The `BoundedElasticScheduler` default thread cap is `10 × CPU_cores` (e.g., 40 threads on a 4-core host) with a task queue of 100,000. Once saturated by attacker subscriptions, legitimate subscribers' `poll()` tasks queue behind attacker tasks. When the queue fills, tasks are rejected with `RejectedExecutionException`, which the `retryWhen(Retry.backoff(Long.MAX_VALUE, ...))` catches and retries — further amplifying scheduler pressure. Legitimate subscribers receive no `ConsensusTopicResponse` messages or experience severe delays, breaking the gossip/delivery guarantee. The `timeout = 60s` does not protect against this: it fires only when no message is emitted for 60 consecutive seconds, and attacker subscriptions on active topics keep emitting messages, preventing timeout. [15](#0-14) 

## Likelihood Explanation
No authentication or authorization is required to call `subscribeTopic`. The gRPC port is publicly exposed. Opening hundreds of TCP connections from a single host is trivial with any gRPC client library. The attack is fully automatable with minimal client code. No special network position or privileged account is needed. The attacker only needs knowledge of one active, high-traffic topic ID to maintain indefinite subscriptions.

## Recommendation

1. **Dedicate an isolated scheduler** for `PollingTopicMessageRetriever` instead of using the shared `Schedulers.boundedElastic()`. Use `Schedulers.newBoundedElastic(threadCap, queueSize, "polling-retriever")` with a bounded thread cap and queue sized to the expected legitimate subscription load.

2. **Add a global concurrent-subscription cap** in `TopicMessageServiceImpl` using the existing `subscriberCount` `AtomicLong` — reject new subscriptions when the count exceeds a configurable threshold.

3. **Add global connection limits** in `GrpcConfiguration`: configure `maxConnectionAge`, `maxConnectionIdle`, and a total `maxConnections` on the `NettyServerBuilder` to prevent unbounded TCP connection accumulation.

4. **Add per-IP rate limiting** at the load balancer or ingress layer to limit the number of concurrent gRPC connections per source IP.

## Proof of Concept

```java
// Pseudocode: open C connections, 5 streams each, targeting a high-traffic topic
ManagedChannel[] channels = new ManagedChannel[C];
for (int i = 0; i < C; i++) {
    channels[i] = ManagedChannelBuilder.forAddress("mirror-node", 5600)
                      .usePlaintext().build();
    ConsensusServiceGrpc.ConsensusServiceStub stub =
        ConsensusServiceGrpc.newStub(channels[i]);
    for (int j = 0; j < 5; j++) {  // maxConcurrentCallsPerConnection = 5
        stub.subscribeTopic(
            ConsensusTopicQuery.newBuilder()
                .setTopicID(HIGH_TRAFFIC_TOPIC_ID)
                .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0))
                .build(),
            new StreamObserver<>() { /* discard responses */ }
        );
    }
}
// C * 5 throttled subscriptions now permanently occupy boundedElastic() threads.
// With C = 10 on a 4-core host: 50 subscriptions vs. 40 available threads.
// Legitimate subscribers' poll tasks queue and are eventually rejected.
```

With `C = ceil(threadCap / 5) + 1` connections (e.g., 9 connections on a 4-core host), the attacker saturates all `boundedElastic()` threads. Legitimate subscribers observe no `ConsensusTopicResponse` delivery until the attack stops.
