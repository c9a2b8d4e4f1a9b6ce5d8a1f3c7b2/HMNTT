Audit Report

## Title
Unbounded Base32 Alias Input Enables DoS via Unconstrained Memory Allocation and DB Parameter Binding

## Summary
The GraphQL `AccountInput.alias` field is validated only by a character-set regex with no upper bound on string length. An unauthenticated attacker can submit an arbitrarily large valid Base32 string that passes validation, causing proportional heap allocation and JDBC serialization overhead on every request. Repeated concurrent requests can exhaust JVM heap memory.

## Finding Description

Every referenced line was confirmed against the actual source:

**Step 1 — Schema validation (character-set only, no length):**
`account.graphqls` line 90 defines:
```
alias: String @Pattern(regexp: "^[A-Z2-7]+$")
```
No `@Size`, `@Length`, or equivalent constraint exists. [1](#0-0) 

**Step 2 — Controller passes alias directly to service:**
`AccountController.java` line 47–48 forwards the raw string with no additional check. [2](#0-1) 

**Step 3 — Service calls `decodeBase32` unconditionally:**
`EntityServiceImpl.java` line 30 passes the alias directly to `decodeBase32`. [3](#0-2) 

**Step 4 — `decodeBase32` has no length guard:**
`GraphQlUtils.java` lines 83–85 calls `BASE32.decode(base32)` directly, allocating `ceil(inputLen * 5 / 8)` bytes with no size check. [4](#0-3) 

**Step 5 — Decoded byte array bound to native JDBC query:**
`EntityRepository.java` line 13–14 binds the full `byte[]` as a native query parameter. [5](#0-4) 

**Contrast with `rest-java`:**
`EntityIdAliasParameter.java` line 13 enforces `{40,70}` characters via its regex, bounding the decoded byte array to at most ~44 bytes. The GraphQL module has no equivalent bound. [6](#0-5) 

No HTTP-level `max-request-size` configuration was found in any `application.yml` or `.properties` file in the repository.

## Impact Explanation

Each request with an N-character alias allocates approximately `N * 5/8` bytes on the JVM heap. A 10 MB Base32 string (all valid `[A-Z2-7]` characters, passes the regex) decodes to ~6.25 MB. With tens of concurrent requests, heap exhaustion is achievable on a default-configured JVM. The JDBC driver must also serialize the full byte array into the PostgreSQL wire protocol for every request, adding CPU and network overhead. The service is publicly accessible with no authentication required for GraphQL queries.

## Likelihood Explanation

The exploit requires no credentials, no special knowledge, and no race condition. Any HTTP client can POST a GraphQL query with a megabyte-scale alias string. The attack is trivially repeatable and scriptable. The only natural mitigating factor is a reverse-proxy or load-balancer request-body size limit, which is not configured in the application itself and may not be present in all deployments.

## Recommendation

1. **Add a length constraint on the `alias` field** in `account.graphqls`. Mirror the `rest-java` bound of 40–70 characters:
   ```
   alias: String @Pattern(regexp: "^[A-Z2-7]{40,70}$")
   ```
   This bounds the decoded byte array to at most ~44 bytes, consistent with `EntityIdAliasParameter.ALIAS_REGEX`. [7](#0-6) 

2. **Add a defensive length check in `decodeBase32`** in `GraphQlUtils.java` as a defense-in-depth measure, rejecting inputs exceeding the maximum expected alias length before any allocation occurs. [4](#0-3) 

3. **Configure an HTTP request body size limit** at the Spring Boot level (e.g., `server.tomcat.max-http-form-post-size` or a `WebFilter`) to provide an application-level backstop independent of infrastructure configuration.

## Proof of Concept

```bash
# Generate a 1 MB valid Base32 string (all 'A' characters, valid [A-Z2-7])
ALIAS=$(python3 -c "print('A' * 1048576)")

# Send the GraphQL query
curl -s -X POST http://<host>/graphql \
  -H 'Content-Type: application/json' \
  -d "{\"query\": \"{ account(input: { alias: \\\"$ALIAS\\\" }) { entityId { num } } }\"}"
```

Each such request causes `GraphQlUtils.decodeBase32` to allocate ~655 KB on the heap. Sending this concurrently from multiple clients (e.g., 50 parallel requests with a 10 MB alias each) will allocate ~312 MB per batch, rapidly exhausting heap on a default JVM configuration. The request passes the `@Pattern(regexp: "^[A-Z2-7]+$")` check because every character is a valid Base32 character.
