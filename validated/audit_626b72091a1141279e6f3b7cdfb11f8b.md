Audit Report

## Title
Gas Estimation Binary Search Bypasses Throttle: Up to 21x EVM Execution Amplification per Request

## Summary
`ThrottleManagerImpl.throttle()` deducts gas-bucket tokens exactly once per HTTP request, but a request with `estimate=true` triggers 1 initial EVM execution plus up to 20 binary-search EVM executions — all invisible to the throttle — yielding a 21x EVM-execution amplification factor at the same throttle cost as a single `eth_call`.

## Finding Description

**[1] `ContractController.call()` calls `throttleManager.throttle(request)` exactly once per HTTP request:** [1](#0-0) 

**[2] `ThrottleManagerImpl.throttle()` deducts `scaleGas(request.getGas())` tokens from `gasLimitBucket` and 1 token from `rateLimitBucket` — both consumed exactly once:** [2](#0-1) 

`scaleGas()` divides gas by `GAS_SCALE_FACTOR = 10_000`. Default `gasPerSecond = 7_500_000_000` (scaled to 750,000 tokens/s), default `requestsPerSecond = 500`: [3](#0-2) 

**[3] For `estimate=true`, `ContractExecutionService.estimateGas()` first calls `callContract()` (EVM execution #1), then calls `binaryGasEstimator.search()` passing `gas -> doProcessCall(params, gas, true)` for up to 20 more executions:** [4](#0-3) 

**[4] `BinaryGasEstimator.search()` loops up to `maxGasEstimateRetriesCount` (default 20) times, each iteration invoking `transactionExecutionService.execute()` via the passed lambda:** [5](#0-4) 

`maxGasEstimateRetriesCount` defaults to 20: [6](#0-5) 

**[5] Root cause — `doProcessCall()`'s `finally` block gates `restoreGasToBucket()` behind `!estimate`. When `estimate=true` (all 20 binary-search iterations), no gas restoration and no additional throttle deduction occurs:** [7](#0-6) 

**[6] `validateContractMaxGasLimit()` only enforces `gas ≤ maxGasLimit` (default 15,000,000), which is the exact value that maximises binary-search range and iteration count:** [8](#0-7) [9](#0-8) 

## Impact Explanation
At the default rate limit of 500 req/s with `gas=15_000_000`, the throttle consumes `500 × 1500 = 750,000` gas tokens/s — exactly at the `gasLimitBucket` capacity — while the actual EVM executions reach `500 × 21 = 10,500/s`. Even at 25 req/s (5% of the rate limit), an attacker injects 525 extra EVM executions/s on top of a legitimate baseline of 500, exceeding a 30% CPU increase threshold. Each EVM execution at 15M gas is CPU-intensive (bytecode interpretation, storage reads, precompile calls), degrading response times for all users and potentially causing the mirror node to fall behind on its duties.

## Likelihood Explanation
The attack requires no credentials, no on-chain assets, and no special knowledge beyond the public REST API. The `POST /api/v1/contracts/call` endpoint is publicly documented. The attacker needs only an HTTP client and knowledge that `estimate=true` triggers binary search. The attack is trivially automatable and indefinitely repeatable from a single modest connection.

## Recommendation
1. **Charge throttle tokens proportionally for estimate requests.** Before invoking `binaryGasEstimator.search()`, deduct additional `gasLimitBucket` tokens proportional to the expected number of binary-search iterations (e.g., multiply by `maxGasEstimateRetriesCount`), or deduct one token per iteration inside the search loop.
2. **Separate rate limit for estimate requests.** Introduce a dedicated `estimateRateLimitBucket` (analogous to `opcodeRateLimitBucket`) with a lower per-second cap for `estimate=true` requests, enforced in `ThrottleManagerImpl.throttle()` when `request.isEstimate()` is true.
3. **Reduce `maxGasEstimateRetriesCount` or cap it dynamically** based on the ratio `(gas - gasUsedByInitialCall) / gas` to limit iterations when the search range is wide.
