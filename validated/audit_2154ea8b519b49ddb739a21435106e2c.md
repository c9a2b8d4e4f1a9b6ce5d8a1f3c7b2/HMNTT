Audit Report

## Title
Negative Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode()` — Newly Deployed Contracts Permanently Invisible on Web3 API

## Summary
The `@Cacheable` guard `unless = "#result == null"` on `ContractRepository.findRuntimeBytecode()` fails to suppress caching of absent results because the method returns `Optional<byte[]>`, not `byte[]`. When no contract row exists, Spring Data JPA returns `Optional.empty()` — a non-null singleton — so the guard evaluates to `false` and the empty `Optional` is stored in the Caffeine cache under `CACHE_NAME_CONTRACT`. Any unauthenticated caller can pre-poison this cache for an arbitrary `contractId`, causing the mirror node's web3 API to treat a subsequently deployed contract as non-existent for up to one hour, and indefinitely with periodic re-access.

## Finding Description
**Exact location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18: [1](#0-0) 

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** The SpEL expression `#result == null` was written assuming the method could return a raw `null`. However, the declared return type is `Optional<byte[]>`. Spring Data JPA never returns a raw `null` for an `Optional`-typed query method — it returns `Optional.empty()` when no row matches. `Optional.empty()` is a non-null singleton, so `#result == null` is always `false`, and the empty `Optional` is unconditionally stored in the cache. [2](#0-1) 

**Cache configuration** (`CacheProperties.java`, line 22): [3](#0-2) 

```
expireAfterAccess=1h, maximumSize=1000, recordStats
```

Because the policy is `expireAfterAccess`, every read of a poisoned entry resets the 1-hour timer, making the poisoning indefinitely renewable.

**Call chain from public API:**

**Why existing tests do not catch this:** `ContractRepositoryTest.findRuntimeBytecodeFailCall()` (line 34–38) confirms `Optional.empty()` is returned for a missing contract but makes no assertion about whether that result is stored in the cache. [9](#0-8) 

## Impact Explanation
Once an attacker pre-poisons the cache for a target `contractId`:

- All subsequent `eth_call` / `eth_estimateGas` requests to that contract address return as if no bytecode exists, regardless of whether the contract has since been deployed and written to the mirror node database.
- `ContractBytecodeReadableKVState.readFromDataSource()` calls `.orElse(null)` on the cached `Optional.empty()`, returning `null` to the EVM, which treats the address as an EOA (no code). Any call that depends on the contract's logic silently returns empty data or reverts, producing incorrect results for dApps and tooling relying on the mirror node.
- With `expireAfterAccess=1h` and a cache size of 1000, an attacker can keep up to 1000 contract addresses poisoned indefinitely by periodically re-querying them, effectively blocking the web3 API from ever seeing those contracts' bytecode.
- No funds are directly at risk on the consensus layer, but smart contract behavior observed through the mirror node is incorrect.

## Likelihood Explanation
The attack requires zero privileges — only the ability to send an HTTP POST to `/api/v1/contracts/call` with a `to` address that maps to a not-yet-existing `contractId`. Hedera entity IDs are sequential and predictable (shard.realm.num format), making it straightforward to target the next `contractId` before deployment. The attack is repeatable, automatable, and requires no on-chain transactions.

## Recommendation
Fix the `unless` SpEL expression to correctly handle the `Optional<byte[]>` return type. Replace:

```java
unless = "#result == null"
```

with:

```java
unless = "#result == null || !#result.isPresent()"
```

or equivalently:

```java
unless = "#result?.isEmpty() ?: true"
```

This ensures that `Optional.empty()` results are never stored in the cache, while `Optional` values containing actual bytecode are cached as intended. [1](#0-0) 

## Proof of Concept

```bash
# Step 1: Determine the next sequential contractId on the network (e.g., 0.0.5001)
# and compute its long-zero EVM address: 0x0000000000000000000000000000000000001389

# Step 2: Pre-poison the cache BEFORE the contract is deployed
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to": "0x0000000000000000000000000000000000001389",
    "data": "0x",
    "gas": 21000,
    "estimate": false,
    "block": "latest"
  }'
# Returns empty/EOA result; Optional.empty() is now cached for contractId=5001

# Step 3: Contract 0.0.5001 is deployed to Hedera and written to the mirror node DB

# Step 4: Query the now-deployed contract — still returns empty due to poisoned cache
curl -X POST https://<mirror-node>/api/v1/contracts/call \
  -H "Content-Type: application/json" \
  -d '{
    "to": "0x0000000000000000000000000000000000001389",
    "data": "0x<function_selector>",
    "gas": 100000,
    "estimate": false,
    "block": "latest"
  }'
# Returns empty data as if no contract exists — cache hit on Optional.empty()

# Step 5: Keep the entry alive indefinitely by re-querying every ~55 minutes
# (resets the expireAfterAccess=1h timer)
```

The cache entry for `contractId=5001` will never be refreshed from the database as long as the attacker periodically re-accesses it, because `expireAfterAccess` resets on every read and the `unless` guard never evicts the empty result. [10](#0-9)
