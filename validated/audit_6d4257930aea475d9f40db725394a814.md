Audit Report

## Title
Missing Account Existence Check in `getAccountCryptoAllowances` Returns HTTP 200 for Non-Existent Numeric Account IDs

## Summary
`CryptoAllowanceController.getAccountCryptoAllowances` omits the `EntityService.isValidAccount()` guard that sibling handlers `listStakingRewardsByAccountId` and `getTokenRelationships` both perform. A numeric account ID (e.g. `0.0.99999`) that does not exist in the entity table causes the endpoint to return HTTP 200 with an empty allowances array instead of 404. The same omission exists in `TokenAllowanceController.getAccountTokenAllowances`.

## Finding Description

**Route registration** — `allowances/crypto` is wired to `CryptoAllowanceController.getAccountCryptoAllowances`: [1](#0-0) 

**Vulnerable handler** — `getAccountCryptoAllowances` calls `EntityService.getEncodedId` but never calls `EntityService.isValidAccount`: [2](#0-1) 

**Root cause** — `getEncodedId` takes a pure-arithmetic fast path for numeric IDs (`shard.realm.num`), returning the encoded integer with zero database interaction: [3](#0-2) 

Only alias and EVM-address inputs hit the database and throw `NotFoundError` when absent. Numeric IDs bypass all existence checks entirely.

**Correctly guarded sibling — `listStakingRewardsByAccountId`:** [4](#0-3) 

**Correctly guarded sibling — `getTokenRelationships`:** [5](#0-4) 

**`isValidAccount` implementation** (the guard that is available but unused): [6](#0-5) 

**Identical omission in `getAccountTokenAllowances`:** [7](#0-6) 

**Test gap** — `alias-not-found.json` confirms a missing alias returns 404: [8](#0-7) 

The misleadingly-named `not-found.json` does **not** test a non-existent numeric account — account `3001` is present in its `setup.accounts` array; it only tests an existing account with no matching allowances, and expects 200: [9](#0-8) 

No test spec exercises a numeric account ID that is absent from the entity table against this endpoint.

## Impact Explanation

Any consumer that calls `GET /api/v1/accounts/{numericId}/allowances/crypto` for a non-existent account receives `200 {"allowances":[]}`, which is indistinguishable from a valid account that has no allowances. The same mirror node returns 404 for the same non-existent numeric ID on `/rewards` and `/tokens`, creating an inconsistent API contract. Systems that cross-check endpoints or act on "account has no allowances" state can be misled into treating phantom accounts as real ones with zero allowances.

## Likelihood Explanation

No authentication is required. Any unauthenticated HTTP client can trigger this by supplying any syntactically valid but non-existent numeric account ID (e.g. `0.0.99999999`). The condition is stable across restarts and trivially repeatable.

## Recommendation

Add the `isValidAccount` guard to both affected handlers, mirroring the pattern already used in `listStakingRewardsByAccountId` and `getTokenRelationships`:

In `rest/controllers/cryptoAllowanceController.js`, after line 77:
```js
const isValidAccount = await EntityService.isValidAccount(accountId);
if (!isValidAccount) { throw new NotFoundError(); }
```

Apply the same fix in `rest/controllers/tokenAllowanceController.js` after the `getEncodedId` call on line 69.

Add a test spec (e.g. `numeric-id-not-found.json`) that supplies a numeric account ID absent from `setup.accounts` and asserts `responseStatus: 404`.

## Proof of Concept

```
# 1. Confirm a non-existent numeric account returns 404 on /rewards (guarded)
GET /api/v1/accounts/0.0.99999999/rewards
→ 404 Not Found

# 2. Same non-existent account returns 200 on /allowances/crypto (unguarded)
GET /api/v1/accounts/0.0.99999999/allowances/crypto
→ 200 OK  {"allowances":[],"links":{"next":null}}

# 3. Same non-existent account returns 200 on /allowances/tokens (unguarded)
GET /api/v1/accounts/0.0.99999999/allowances/tokens
→ 200 OK  {"allowances":[],"links":{"next":null}}
```

Steps 2 and 3 are indistinguishable from a real account with no allowances, confirming the inconsistency.
