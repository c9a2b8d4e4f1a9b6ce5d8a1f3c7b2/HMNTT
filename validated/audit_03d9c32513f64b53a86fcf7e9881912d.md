# Audit Report

## Title
Unbounded Global Subscription Count Exhausts Shared `boundedElastic` Scheduler in `PollingTopicMessageRetriever.retrieve()`

## Summary
`PollingTopicMessageRetriever` allocates a single private `Schedulers.boundedElastic()` instance at construction time and routes every subscriber's repeat-delay and timeout tasks through it. Because there is no global cap on concurrent subscriptions — only a per-connection limit of 5 — an attacker who opens many TCP connections can create an unbounded number of concurrent subscriptions, exhausting the scheduler's thread pool and eventually its task queue, causing `RejectedExecutionException` for legitimate subscribers.

## Finding Description

**Constructor — single shared scheduler:**

`PollingTopicMessageRetriever` creates one `boundedElastic` scheduler shared by all subscriptions: [8](#0-7) 

**`retrieve()` — every subscription uses that scheduler, no guard:**

Both the repeat-delay and the timeout are scheduled on the same instance: [9](#0-8) 

**Throttled path runs with `Long.MAX_VALUE` repeats:**

When `throttled=true` (the path taken by `subscribeTopic`), the subscription polls indefinitely until the 60-second timeout fires: [10](#0-9) 

**`subscribeTopic()` calls `retrieve(filter, true)` unconditionally:** [4](#0-3) 

**`subscriberCount` is a metrics gauge only — no enforcement:** [11](#0-10) 

**Only per-connection limit exists; no `maxConnections` is configured:** [12](#0-11) [13](#0-12) 

**Root cause:** The application assumes that `maxConcurrentCallsPerConnection=5` is a sufficient guard. It is not — it is a per-connection Netty limit. With `C` connections an attacker holds `5C` concurrent subscriptions, each enqueuing periodic tasks on the single shared `boundedElastic` scheduler. Reactor's `BoundedElasticScheduler` caps threads at `10 × CPU_cores`; once all threads are busy, tasks queue up, and once the per-worker queue (100,000 entries) fills, new scheduling calls throw `RejectedExecutionException`.

## Impact Explanation

When the shared scheduler's thread pool is saturated, legitimate subscribers' repeat-delay and timeout tasks cannot be scheduled. This manifests as `RejectedExecutionException` propagated as terminal errors to honest clients. The result is complete unavailability of the `subscribeTopic` gRPC endpoint for all users while the attack is sustained. There is no financial loss, but the service is fully degraded for honest clients. Severity: **Medium** (DoS, no economic damage).

## Likelihood Explanation

The gRPC port (default 5600) requires no authentication to call `subscribeTopic`. An attacker needs only a standard gRPC client and the ability to open many TCP connections — trivially achievable from a single host. The 60-second `RetrieverProperties.timeout` limits each subscription's duration, but the attacker simply reconnects after timeout, making the attack continuously repeatable with no special knowledge, credentials, or on-chain resources. [14](#0-13) 

## Recommendation

1. **Add a global subscription cap** in `TopicMessageServiceImpl.subscribeTopic()`: check `subscriberCount` against a configurable maximum before proceeding, and return `RESOURCE_EXHAUSTED` if the cap is exceeded.
2. **Configure `maxConnections`** on the `NettyServerBuilder` in `GrpcConfiguration` to bound the total number of TCP connections at the transport layer.
3. **Consider a per-IP rate limit** on new subscription attempts to slow reconnect-after-timeout attacks.
4. **Optionally use a dedicated, size-bounded scheduler** for `PollingTopicMessageRetriever` with an explicit thread cap and queue size, so scheduler exhaustion cannot affect other Reactor pipelines.

## Proof of Concept

```python
import grpc
import threading
from concurrent.futures import ThreadPoolExecutor

# Attacker opens N connections, each with 5 concurrent subscribeTopic calls
# (maxConcurrentCallsPerConnection=5 is the per-connection limit)

NUM_CONNECTIONS = 500   # 500 connections × 5 calls = 2500 concurrent subscriptions
CALLS_PER_CONN  = 5

def flood_connection(_):
    channel = grpc.insecure_channel("mirror-node:5600")
    stub = ConsensusServiceStub(channel)
    threads = []
    for _ in range(CALLS_PER_CONN):
        t = threading.Thread(
            target=lambda: list(stub.subscribeTopic(
                ConsensusTopicQuery(topic_id=TopicID(topic_num=1))
            ))
        )
        t.start()
        threads.append(t)
    for t in threads:
        t.join()

with ThreadPoolExecutor(max_workers=NUM_CONNECTIONS) as ex:
    list(ex.map(flood_connection, range(NUM_CONNECTIONS)))

# Result: 2500 concurrent subscriptions each scheduling tasks on the single
# boundedElastic scheduler every 2 s (pollingFrequency) plus a 60 s timeout task.
# On a 4-core host: 40 scheduler threads, queue fills → RejectedExecutionException
# for legitimate subscribers.
```
