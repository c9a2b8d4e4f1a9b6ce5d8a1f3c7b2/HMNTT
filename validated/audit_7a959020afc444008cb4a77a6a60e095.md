Audit Report

## Title
Unbounded DB Lookup Amplification via Account Alias Path Parameter on All Account Sub-Routes

## Summary
Every account sub-route registered in `rest/routes/accountRoute.js` accepts an alias string as the `:idOrAliasOrEvmAddress` path parameter. When an alias is supplied, `EntityService.getEncodedId()` unconditionally executes a `SELECT` against the `entity` table on every request with no per-alias result caching. Because numeric-ID requests require zero DB queries for ID resolution (pure in-memory computation), an attacker flooding the sub-routes with unique valid alias strings forces at least one extra DB query per request, bypassing the response cache entirely.

## Finding Description

**Route registration** — `rest/routes/accountRoute.js` lines 15–19 register five sub-routes, all parameterized by `:idOrAliasOrEvmAddress`: [1](#0-0) 

**Handler entry points** — Each handler immediately calls `EntityService.getEncodedId()` with the raw path parameter. Verified in `accountController.js`: [2](#0-1) [3](#0-2) 

**Branching logic in `getEncodedId()`** — `rest/service/entityService.js` lines 118–137:
- If `EntityId.isValidEntityId()` is true and no EVM address → encoded ID computed **in memory**, zero DB queries.
- If `AccountAlias.isValid()` is true → `getAccountIdFromAlias()` → `getAccountFromAlias()` → executes `entityFromAliasQuery`. [4](#0-3) 

**The DB query executed for every alias request:** [5](#0-4) 

**No alias→ID caching exists** anywhere in `entityService.js`. The entire file contains no cache layer between `getAccountFromAlias()` and the DB. [6](#0-5) 

**Response cache does not help** — `responseCacheHandler.js` generates its cache key from `req.originalUrl` (MD5 hash). Each unique alias string produces a unique URL → guaranteed cache miss on every novel alias. [7](#0-6) 

**Alias validity regex** — `rest/accountAlias.js` line 10 accepts any string matching `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/`. An attacker can trivially generate an unbounded number of syntactically valid alias strings (`AAAAAAA`, `AAAAAAB`, …) with no knowledge of the system. [8](#0-7) [9](#0-8) 

**Additional amplification on `/rewards`** — `listStakingRewardsByAccountId` calls `EntityService.isValidAccount()` immediately after `getEncodedId()`, adding a second DB query (`entityExistenceQuery`) for alias-parameterized requests, making the amplification 2× on that endpoint. [10](#0-9) [11](#0-10) 

## Impact Explanation

For every numeric-ID request, ID resolution costs 0 DB queries. For every alias request, it costs at least 1 additional `SELECT` against the central `entity` table (2 on the `/rewards` sub-route). An attacker directing all traffic to alias-parameterized URLs with unique alias strings bypasses the response cache entirely, forcing the DB query on every single request. The `entity` table is central to the mirror node; sustained extra load degrades all other queries sharing the same DB connection pool and I/O bandwidth. The increase in DB I/O is ≥100% over the numeric-ID baseline, well above any reasonable threshold.

## Likelihood Explanation

No authentication or privilege is required. All five affected sub-routes are public REST endpoints. Generating valid base32 alias strings requires no knowledge of the system. The `ThrottleConfiguration` found in the codebase exists only in the `web3` Java module; no rate limiting is present in the REST JavaScript module. [12](#0-11) 

A single attacker with a modest HTTP client can sustain thousands of unique-alias requests per second, each forcing a DB query.

## Recommendation

1. **Add an in-process LRU/TTL cache** in `EntityService` for alias→ID resolution results (e.g., using `lru-cache`). A short TTL (e.g., 5–30 seconds) is sufficient to absorb burst traffic while keeping data reasonably fresh.
2. **Add rate limiting** to the REST module (e.g., `express-rate-limit`) at the application or reverse-proxy layer, consistent with the throttling already applied in the `web3` module.
3. **Consider input validation at the routing layer** to reject alias strings that do not correspond to any known account before they reach the DB, or enforce a minimum alias length to reduce the trivial generation surface.

## Proof of Concept

```
# Generate unique valid alias strings and flood the /nfts sub-route
for i in $(seq 1 10000); do
  ALIAS=$(python3 -c "import string; chars='ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'; print(''.join([chars[(${i}>>j)&31] for j in range(0,15,5)]))")
  curl -s "https://<mirror-node>/api/v1/accounts/${ALIAS}/nfts" &
done
```

Each request carries a unique alias string, produces a unique URL (cache miss in `responseCacheHandler`), and forces one `SELECT id FROM entity WHERE alias = $1` query. At 1,000 req/s, this generates 1,000 extra DB queries per second against the `entity` table with zero authentication required.
