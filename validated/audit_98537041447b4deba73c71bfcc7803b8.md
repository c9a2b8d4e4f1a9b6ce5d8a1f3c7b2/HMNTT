## Audit Report

## Title
Gas Throttle Bypass via Minimum-Gas Requests: `scaleGas` Returns Near-Zero Tokens for `gas=21000`

## Summary
The `scaleGas` function in `ThrottleProperties` divides request gas by `GAS_SCALE_FACTOR = 10_000`. A request with `gas=21_000` (the enforced `@Min` on `ContractCallRequest.gas`) consumes only **2 tokens** from the gas bucket per request. With a default gas bucket capacity of 750,000 tokens/second, an attacker sending minimum-gas requests exhausts only 0.13% of the gas bucket per second at the 500 RPS ceiling, rendering the gas throttle completely ineffective as a computational-load defense for this class of requests.

## Finding Description

**Code path:**

`ContractCallRequest.gas` is annotated `@Min(21_000)`: [1](#0-0) 

In `ThrottleManagerImpl.throttle()`, the gas bucket is consumed via `throttleProperties.scaleGas(request.getGas())`: [5](#0-4) 

`scaleGas` is defined as: [7](#0-6) 

With `GAS_SCALE_FACTOR = 10_000` and `gas = 21_000`:
- Guard condition: `21_000 <= 10_000` → **false**
- Returns: `Math.floorDiv(21_000, 10_000)` = **2 tokens** consumed per request

The gas bucket is initialized with capacity `scaleGas(gasPerSecond)`: [8](#0-7) 

Default `gasPerSecond = 7_500_000_000L`: [9](#0-8) 

So `getGasPerSecond()` = `scaleGas(7_500_000_000)` = **750,000 tokens/second** bucket capacity.

**Root cause:** The `GAS_SCALE_FACTOR = 10_000` was introduced to work around bucket4j's 1-token-per-nanosecond fill-rate ceiling (noted in the comment at line 37 of `ThrottleProperties.java`). This creates a structural asymmetry: the minimum allowed gas (21,000) maps to only 2 tokens, while the bucket holds 750,000 tokens/second. The gas throttle is inert for minimum-gas requests.

**Why existing checks fail:**

The `rateLimitBucket` (default 500 RPS) is the only binding constraint: [10](#0-9) 

At 500 RPS with `gas=21_000`: gas bucket consumption = 500 × 2 = 1,000 tokens/second out of 750,000 capacity (0.13%). The `gasLimitBucket.tryConsume()` check **never returns false** for this workload.

The `gasLimitRefundPercent=100` restore path also does not help — `scaleGas(gasRemaining)` for small remaining-gas values (≤ 10,000) returns 0, so net consumption per request remains near zero: [6](#0-5) 

## Impact Explanation
The gas throttle's stated purpose is to cap total computational load (gas/second) processed by the node. With minimum-gas requests, this protection is entirely absent. An attacker can sustain the full 500 RPS indefinitely — each request still triggers a full EVM execution path through `ContractExecutionService.processCall()` — without the gas throttle ever activating. If an operator raises `requestsPerSecond` (e.g., to 5,000 for higher-throughput deployments), the gas throttle remains the only intended backstop for computational load, and it still does not trigger for `gas=21_000` requests (10,000 tokens/second vs. 750,000 capacity). This enables sustained EVM-execution DoS at the maximum permitted request rate with zero gas-bucket cost.

## Likelihood Explanation
No authentication, API key, or IP restriction is required. Any external user can POST to `/api/v1/contracts/call` with `{"gas": 21000, "to": "0x...", "data": "0x"}`. The attack is trivially scriptable with any HTTP client. The minimum gas value is publicly discoverable via the `@Min(21_000)` validation error message. The attack is fully repeatable and stateless.

## Recommendation
1. **Raise the effective minimum token cost**: Ensure `scaleGas(minimumGas)` yields a meaningful token count relative to bucket capacity. For example, set a minimum return value of `Math.max(1, Math.floorDiv(gas, GAS_SCALE_FACTOR))` and ensure the bucket capacity is sized so that `requestsPerSecond × scaleGas(minGas)` approaches the bucket capacity.
2. **Decouple scaling from minimum gas**: Introduce a separate `minGasTokenCost` configuration that sets a floor on tokens consumed per request regardless of the gas value supplied, preventing the asymmetry introduced by the scale factor.
3. **Align bucket capacity with RPS ceiling**: Ensure `scaleGas(minGas) × requestsPerSecond` is at least a significant fraction (e.g., ≥ 50%) of `getGasPerSecond()` so the gas throttle can actually fire before the RPS limit is reached.

## Proof of Concept
```bash

# Gas throttle never fires; only the RPS bucket (500/s) is the constraint
for i in $(seq 1 500); do

done
```
