Audit Report

## Title
Unbounded `endTime` + `startTime=0` Enables Unauthenticated Resource Exhaustion via Unlimited Concurrent Subscriptions

## Summary
Any unauthenticated caller can open an arbitrary number of `subscribeTopic` gRPC sessions with `startTime=0` and `endTime=Long.MAX_VALUE`. Each session triggers a full historical database scan from epoch and holds a live reactive pipeline open until approximately year 2262, with no per-user or global subscription cap enforced anywhere in the service layer.

## Finding Description

**Validation gap — no upper bound on `endTime`:**

`TopicMessageFilter.isValidEndTime()` only checks `endTime > startTime`: [1](#0-0) 

`Long.MAX_VALUE` trivially satisfies `endTime > 0`. `startTime=0` passes `isValidStartTime()` because `0 <= DomainUtils.now()`: [2](#0-1) 

**Subscription never terminates:**

`TopicContext.isComplete()` computes `Instant.ofEpochSecond(0, filter.getEndTime()).plus(endTimeInterval).isBefore(Instant.now())`: [3](#0-2) 

With `endTime=Long.MAX_VALUE`, this resolves to ~year 2262, so `isComplete()` returns `false` for the entire practical lifetime of the service. The `pastEndTime` flux therefore polls every 30 seconds indefinitely: [4](#0-3) 

**Full historical scan per subscription:**

`subscribeTopic` unconditionally calls `topicMessageRetriever.retrieve(filter, true)` (throttled=true): [5](#0-4) 

With `throttled=true`, `numRepeats=Long.MAX_VALUE` and the retriever pages through all messages from timestamp 0 until a page is smaller than `maxPageSize` (1000 rows), bounded only by a 60-second inactivity timeout: [6](#0-5) [7](#0-6) 

**Per-subscription polling loop (POLL mode):**

`PollingTopicListener.listen()` creates an independent polling loop with `RepeatSpec.times(Long.MAX_VALUE)` per subscription: [8](#0-7) 

**No subscription limit:**

`subscriberCount` is only a Micrometer gauge — it is never checked against any cap: [9](#0-8) 

There is no per-IP, per-user, or global subscription ceiling in `TopicMessageServiceImpl`, `CompositeTopicListener`, or `SharedTopicListener`: [10](#0-9) [11](#0-10) 

The only partial friction is `NettyProperties.maxConcurrentCallsPerConnection = 5`, which limits concurrent calls per connection but does not prevent an attacker from opening multiple connections: [12](#0-11) 

## Impact Explanation

Each malicious subscription consumes:
- A slot in the `boundedElastic` scheduler for historical retrieval and `pastEndTime` polling
- Repeated DB queries scanning from timestamp 0 (up to 60-second inactivity timeout per retrieval phase)
- A 16,384-element in-memory backpressure buffer per subscription (`maxBufferSize` in `ListenerProperties`)
- An ongoing DB poll every 500 ms (POLL mode) or a Redis channel slot (REDIS mode)

Stacking many such sessions exhausts the scheduler thread pool, saturates the database connection pool, and causes memory pressure via buffer accumulation — degrading or denying service to legitimate subscribers.

## Likelihood Explanation

No credentials, tokens, or special network access are required. A single attacker with a gRPC client library can script the attack. The filter parameters (`startTime=0`, `endTime=Long.MAX_VALUE`) pass all server-side validation as confirmed above. The only partial friction is the per-connection call limit of 5, which requires the attacker to open multiple connections — a trivial requirement.

## Recommendation

1. **Enforce a maximum `endTime` horizon**: Reject or cap `endTime` values beyond a configurable maximum (e.g., `now + 30 days`).
2. **Add a global subscription cap**: Check `subscriberCount` against a configurable maximum before accepting a new subscription in `TopicMessageServiceImpl.subscribeTopic()`.
3. **Add a per-connection or per-IP subscription cap**: Enforce limits at the Netty/gRPC layer in addition to the service layer.
4. **Limit historical scan depth**: Reject or warn on `startTime=0` or `startTime` values older than a configurable threshold.

## Proof of Concept

```python
import grpc
from hedera.mirror.api.proto import consensus_service_pb2_grpc
from hedera.mirror.api.proto import consensus_service_pb2
from hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

def open_exhausting_subscription(stub, topic_num):
    query = consensus_service_pb2.ConsensusTopicQuery(
        topicID=basic_types_pb2.TopicID(topicNum=topic_num),
        consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
        consensusEndTime=timestamp_pb2.Timestamp(seconds=9223372035, nanos=0),  # triggers Long.MAX_VALUE path
    )
    return stub.subscribeTopic(query)  # returns a streaming iterator, holds resources open

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

# Open many subscriptions across multiple channels to bypass per-connection limit of 5
streams = []
for _ in range(200):
    streams.append(open_exhausting_subscription(stub, topic_num=1))
# Each stream triggers a full historical scan from epoch and holds a live pipeline open until ~2262
```

Each stream passes `isValidEndTime()` (since `Long.MAX_VALUE > 0`) and `isValidStartTime()` (since `0 <= now()`), triggers `topicMessageRetriever.retrieve(filter, true)` with `numRepeats=Long.MAX_VALUE`, and keeps `pastEndTime` polling every 30 seconds indefinitely because `isComplete()` always returns `false`.

    @Override
    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));
    }

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)
                        .withFixedDelay(grpcProperties.getEndTimeInterval()));
    }
