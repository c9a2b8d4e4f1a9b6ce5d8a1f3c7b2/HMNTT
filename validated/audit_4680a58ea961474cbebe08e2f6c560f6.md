Audit Report

## Title
Deleted Account Data Disclosed via Long-Form EVM Address Lookup in GraphQL `account` Query

## Summary
The `account(input: {evmAddress: ...})` GraphQL query is publicly accessible with no authentication. When a long-form EVM address is supplied (first 12 bytes are zero, encoding a Hedera numeric entity ID), `EntityServiceImpl.getByEvmAddressAndType()` falls through to `entityRepository.findById()`, which issues a plain `SELECT * FROM entity WHERE id = ?` with no `deleted` filter. This is inconsistent with the `findByEvmAddress()` and `findByAlias()` paths, both of which explicitly exclude deleted entities.

## Finding Description

The inconsistency is confirmed in `EntityServiceImpl.getByEvmAddressAndType()`: [1](#0-0) 

When the first 4 bytes (int) and next 8 bytes (long) of the decoded EVM address are zero, the code calls `entityRepository.findById(buffer.getLong())`. This is the inherited Spring Data `CrudRepository.findById`, which issues a plain `SELECT * FROM entity WHERE id = ?` with **no deleted filter**. [2](#0-1) 

By contrast, both custom query methods in `EntityRepository` explicitly exclude deleted records:
- `findByEvmAddress`: `select * from entity where evm_address = ?1 and deleted is not true`
- `findByAlias`: `select * from entity where alias = ?1 and deleted is not true`

The only post-fetch filter in the long-form path is `e.getType() == type`, which does not check `deleted`. [3](#0-2) 

The `AccountController` carries no `@PreAuthorize` or equivalent annotation, and no `SecurityConfig` class exists in the graphql module. [4](#0-3) 

The `AccountInput` schema accepts `evmAddress` matching `^(0x)?[a-fA-F0-9]{40}$`, which permits long-form addresses such as `0x0000000000000000000000000000000000004D2`. [5](#0-4) 

**Additional note:** The `getByIdAndType()` method (used when `entityId` input is provided) also calls `findById` without a deleted filter, exhibiting the same behavior. The claim's assertion that "every other lookup path" filters deleted accounts is therefore slightly overstated — the inconsistency is specifically between the address/alias lookup paths and the ID-based lookup paths. [6](#0-5) 

## Impact Explanation

A deleted account's full record — including `key`, `memo`, `alias`, `balance`, `autoRenewAccount`, `receiverSigRequired`, and all other fields exposed by the `Account` type — is returned to any unauthenticated caller. The `deleted: Boolean` field in the response will show `true`, confirming the account is deleted, but the full data is still disclosed. [7](#0-6) 

## Likelihood Explanation

Preconditions are minimal: the attacker needs only a known numeric account ID (obtainable by iterating small integers or from public ledger history) and network access to the GraphQL endpoint. No credentials are required. The long-form EVM address encoding is a documented Hedera convention. The attack is fully repeatable and automatable.

## Recommendation

Add a `deleted is not true` filter to the `findById` path within `getByEvmAddressAndType()`. The simplest fix is to introduce a custom repository method, e.g.:

```java
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(long id);
```

and replace the `entityRepository.findById(buffer.getLong())` call with `entityRepository.findByIdAndNotDeleted(buffer.getLong())`.

The same fix should be applied to `getByIdAndType()` if the intended invariant is that deleted accounts must never be returned by any lookup path. [1](#0-0) 

## Proof of Concept

1. Identify a deleted account with numeric ID, e.g., `1234` (decimal `0x4D2`).
2. Construct the long-form EVM address: `0x0000000000000000000000000000000000004D2`.
3. Send the following GraphQL query to `/graphql/alpha` (no authentication required):

```graphql
query {
  account(input: { evmAddress: "0x0000000000000000000000000000000000004D2" }) {
    entityId
    deleted
    key
    memo
    alias
    balance
  }
}
```

4. The response returns the full account record with `deleted: true`, confirming the deleted account's data is disclosed.

    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
