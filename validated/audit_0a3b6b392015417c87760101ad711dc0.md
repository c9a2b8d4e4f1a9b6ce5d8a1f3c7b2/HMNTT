Audit Report

## Title
Integer Overflow in `PollingTopicMessageRetriever.poll()` Bypasses DB Query Limit, Enabling Unbounded Data Retrieval DoS

## Summary
When a gRPC client supplies `limit = Long.MAX_VALUE` in a `ConsensusTopicQuery`, a Java narrowing cast in `poll()` silently produces `-1`. Because `TopicMessageFilter.hasLimit()` returns `false` for negative values, the downstream JPA query omits `setMaxResults` entirely, loading every row for the topic on every poll cycle. The termination check `isComplete()` never fires, so polling repeats until the subscription timeout.

## Finding Description

**Entry point — proto field maps directly to Java `long` with no upper-bound guard:**

The protobuf field is `uint64`: [1](#0-0) 

`ConsensusController.toFilter()` assigns it without any upper-bound check: [2](#0-1) 

The only guard on `TopicMessageFilter.limit` is `@Min(0)`, which `Long.MAX_VALUE` trivially satisfies: [3](#0-2) 

**Overflow site — `poll()` in `PollingTopicMessageRetriever`:**

```java
int limit = filter.hasLimit()
        ? (int) (filter.getLimit() - context.getTotal().get())
        : Integer.MAX_VALUE;
int pageSize = Math.min(limit, context.getMaxPageSize());
``` [4](#0-3) 

With `filter.getLimit() = Long.MAX_VALUE` and `context.getTotal().get() = 0`:
- `Long.MAX_VALUE = 0x7FFFFFFFFFFFFFFF`
- Java narrowing cast `(int)` retains only the low 32 bits: `0xFFFFFFFF` = **`-1`** (signed)
- `Math.min(-1, 1000) = -1`

The `newFilter` is then built with `limit = -1`: [5](#0-4) 

**Limit bypass — `findByFilter()` in `TopicMessageRepositoryCustomImpl`:**

`hasLimit()` is defined as `limit > 0`. With `limit = -1`, it returns `false`: [6](#0-5) 

So the `setMaxResults` guard is never reached: [7](#0-6) 

The query materialises the entire result set into the JVM heap: [8](#0-7) 

**`isComplete()` never terminates the loop:**

```java
boolean limitHit = filter.hasLimit() && filter.getLimit() == total.get();
``` [9](#0-8) 

Here `filter` is the **original** filter (limit = `Long.MAX_VALUE`). `filter.hasLimit()` is `true` (since `Long.MAX_VALUE > 0`), but `total.get()` can never reach `Long.MAX_VALUE`. In throttled mode, if the DB returns more rows than `maxPageSize` (which it does, since there is no limit), `pageSize.get() >= maxPageSize` and `limitHit` is `false`, so `isComplete()` returns `false` and polling repeats indefinitely.

On every subsequent poll, `(int)(Long.MAX_VALUE - N)` remains negative for any small `N` (the high bits of `Long.MAX_VALUE` are never cleared), so the overflow and missing limit persist across all poll cycles.

## Impact Explanation

Each poll cycle issues an unbounded `SELECT` against the `topic_message` table for the targeted topic. The entire result set is materialised into the JVM heap via `getResultList().stream()`, repeated every `pollingFrequency` (default 2 s). For a high-volume topic this can exhaust heap memory and saturate DB I/O, causing an effective denial of service for all subscribers. The subscription `timeout` (default 60 s) provides a per-connection ceiling, but the attacker can immediately reconnect and repeat, sustaining the attack indefinitely.

## Likelihood Explanation

No authentication or privilege is required — any gRPC client can set `limit` to `Long.MAX_VALUE` in a `ConsensusTopicQuery`. The protobuf field is `uint64`; `Long.MAX_VALUE` (`9223372036854775807`) is a valid, encodable value. The exploit is deterministic, requires a single RPC call, and is trivially repeatable. Topics with large message histories amplify the impact.

## Recommendation

1. **Clamp the limit before the cast.** In `poll()`, cap the computed remaining limit to `Integer.MAX_VALUE` before narrowing:
   ```java
   long remaining = filter.getLimit() - context.getTotal().get();
   int limit = filter.hasLimit()
           ? (int) Math.min(remaining, Integer.MAX_VALUE)
           : Integer.MAX_VALUE;
   ```
2. **Add an upper-bound validator on `TopicMessageFilter.limit`.** Add `@Max(Long.MAX_VALUE / 2)` or a domain-appropriate maximum (e.g., `@Max(10_000)`) alongside the existing `@Min(0)`.
3. **Validate at the controller boundary.** In `ConsensusController.toFilter()`, reject or clamp `query.getLimit()` values that exceed a configured maximum before constructing the filter.

## Proof of Concept

```
# Send a single gRPC subscribeTopic call with limit = Long.MAX_VALUE (9223372036854775807)
# using grpcurl or any gRPC client targeting a topic with a large message history.

grpcurl -d '{
  "topicID": {"topicNum": 1234},
  "limit": 9223372036854775807
}' <mirror-node-host>:5600 \
  com.hedera.mirror.api.proto.ConsensusService/subscribeTopic
```

**Expected result:** Every 2 seconds, the mirror node issues an unbounded `SELECT` on `topic_message` for the given topic, materialising all rows into heap. Heap exhaustion and DB I/O saturation occur proportionally to the topic's message volume. The subscription never self-terminates until the 60 s timeout, after which the client reconnects and repeats.
