Audit Report

## Title
Silent Overwrite of Lower Bound in `Bound` Constructor Bypasses `verifySingleOccurrence()` Validation in Token Airdrop Endpoints

## Summary
When two same-direction range operators (e.g., `receiver.id=gt:100&receiver.id=gt:200`) are supplied to the outstanding or pending airdrop endpoints, the `Bound` constructor silently discards the first lower bound and retains only the second. Because `TokenAirdropServiceImpl` never calls `verifySingleOccurrence()`, no error is raised and the query executes with a narrower filter than intended, silently omitting valid airdrop records.

## Finding Description

**`TokenAirdropsController.java` line 71** accepts `@Size(max = 2) EntityIdRangeParameter[] receiverIds`, explicitly permitting two values. [1](#0-0) 

**Line 97** passes the array directly to the `Bound` constructor with `primarySortField = true`: [2](#0-1) 

**`Bound.java` lines 47–54**: the constructor iterates over all params and silently overwrites `lower` on each iteration that satisfies `hasLowerBound()` (true for `GT`, `GTE`, and `EQ`), while still incrementing the cardinality counter for every param: [3](#0-2) 

With `receiver.id=gt:100&receiver.id=gt:200`:
- Iteration 1: `lower = gt:100`, `cardinality[GT] = 1`
- Iteration 2: `lower = gt:200` (overwrites silently), `cardinality[GT] = 2`

The only guard that follows is the `primarySortField` range check at line 58: [4](#0-3) 

With no upper bound, `adjustUpperBound()` returns `Long.MAX_VALUE` (line 65), so `201 > Long.MAX_VALUE` is false — no exception is thrown. [5](#0-4) 

`verifySingleOccurrence()` at lines 155–159 would catch `cardinality[GT] = 2` and throw, but it is **never called** in the TokenAirdrop path: [6](#0-5) 

`TokenAirdropServiceImpl.getAirdrops()` proceeds directly to `repository.findAll()` with zero validation: [7](#0-6) 

By contrast, `NftAllowanceServiceImpl.verifyRangeId()` explicitly calls `verifySingleOccurrence()` on every bound: [8](#0-7) 

## Impact Explanation
The query executes with `receiver_account_id > 200` instead of `receiver_account_id > 100`, silently dropping all airdrops with receiver IDs in the range (100, 200]. No error is returned to the caller, so there is no indication the result set is incomplete. For the outstanding airdrops endpoint a sender misses entries in the dropped range; for the pending airdrops endpoint a receiver misses claimable airdrops in the dropped range. While the endpoint is read-only and no funds are moved, the silent omission of records the caller is entitled to see can cause missed claim or cancel actions.

## Likelihood Explanation
No authentication or privileges are required. Any unauthenticated HTTP client can craft the request. The `@Size(max = 2)` annotation is part of the declared API contract, so the input is within the accepted parameter space. The behavior is deterministic and reproducible on every such request.

## Recommendation
Add the same validation that `NftAllowanceServiceImpl` already applies. In `TokenAirdropServiceImpl.getAirdrops()` (or in `TokenAirdropsController.processRequest()` immediately after constructing the `Bound`), call the following on each `Bound` that accepts two values:

```java
entityIdsBound.verifySingleOccurrence();
```

This mirrors the existing pattern in `NftAllowanceServiceImpl.verifyRangeId()`: [8](#0-7) 

The same guard should be applied to the `serialNumbers` and `tokenIds` bounds in `processRequest()` for consistency.

## Proof of Concept

```
GET /api/v1/accounts/0.0.500/airdrops/outstanding
    ?receiver.id=gt:100&receiver.id=gt:200
```

Expected behavior: HTTP 400 with a message such as `"Only one range operator from [GT, GTE] is allowed"`.

Actual behavior: HTTP 200 with results filtered to `receiver_account_id > 200`, silently omitting all airdrops where `receiver_account_id` is in the range (100, 200].

The `cardinality` map inside the constructed `Bound` will contain `{GT=2}`, which `verifySingleOccurrence()` would reject, but that method is never invoked in this code path. [9](#0-8)
