Audit Report

## Title
Unauthenticated DB Query Exhaustion via Unbounded Alias Flooding in `getNftsByAccountId`

## Summary
Any unauthenticated caller can flood `GET /api/v1/accounts/{id}/nfts` with syntactically valid but non-existent base32 aliases. Each request unconditionally executes a live `SELECT` against the `entity` table. Because the REST server registers no rate-limiting middleware and 404 responses are never cached, an attacker can saturate the PostgreSQL connection pool and degrade service for all users.

## Finding Description

Every code claim in the report was verified against the actual source:

**Step 1 â€” Raw parameter forwarded to `getEncodedId`**
`getNftsByAccountId` passes `req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS]` directly to `EntityService.getEncodedId` with no pre-validation or sanitisation. [1](#0-0) 

**Step 2 â€” Alias branch triggers immediate DB call**
`getEncodedId` checks `AccountAlias.isValid()` and, if true, immediately calls `getAccountIdFromAlias` with no guard, cache check, or short-circuit. [2](#0-1) 

**Step 3 â€” Every alias lookup hits the database**
`getAccountFromAlias` calls `super.getRows(EntityService.entityFromAliasQuery, ...)` on every invocation. There is no in-process or Redis cache wrapping this call. [3](#0-2) 

**Step 4 â€” No alias length cap in the validation regex**
`accountAliasRegex` uses `[A-Z2-7]+` (unbounded `+`), so arbitrarily long base32 strings pass validation. [4](#0-3) 

**Step 5 â€” Response cache skips 404 results**
`responseCacheUpdateHandler` only stores a response when `httpStatusCodes.isSuccess(res.statusCode)` is true. A non-existent alias returns a `NotFoundError` (404), which is never cached, so every miss re-hits the DB. [5](#0-4) 

**Step 6 â€” No rate-limiting middleware in the REST server**
`server.js` registers: `urlencoded`, `json`, `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, route handlers, `responseHandler`, `responseCacheUpdateHandler`, `handleError`. No rate-limiting middleware is present anywhere in this stack. [6](#0-5) 

**Step 7 â€” `authHandler` does not block unauthenticated callers**
When no `Authorization` header is present, `authHandler` calls `next()` without rejecting the request. Authentication is optional and only affects result-set limits, not access control. [7](#0-6) 

## Impact Explanation
An attacker sending concurrent requests with unique valid base32 aliases (e.g., `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA`, `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAB`, â€¦) forces a new `SELECT` on the `entity` table for each request. Once the PostgreSQL connection pool is exhausted, every REST endpoint that requires DB access begins queuing or rejecting requests, causing broad availability loss for all legitimate users. No account, funds, or on-chain state is modified.

## Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero on-chain resources. An attacker needs only an HTTP client and the ability to generate unique base32 strings, which is trivial. The endpoint is publicly documented in the OpenAPI spec. The attack is fully repeatable and can be sustained indefinitely from a single machine or amplified across multiple source IPs to bypass any upstream network-level throttle.

## Recommendation
1. **Negative-result cache**: Cache alias-miss results (e.g., in Redis with a short TTL such as 60 s) so repeated lookups for the same non-existent alias do not hit the DB.
2. **Alias length cap**: Add a maximum length to `accountAliasRegex` (e.g., `[A-Z2-7]{1,100}`) to reject pathologically long inputs before any DB interaction.
3. **Rate limiting**: Add a per-IP rate-limiting middleware (e.g., `express-rate-limit`) to `rest/server.js` for all `/api/v1/` routes, consistent with the rate-limit middleware already configured for the Rosetta service in its Helm chart.
4. **DB-level protection**: Configure a connection pool maximum and a statement timeout so that even if the pool is saturated, individual queries do not hold connections indefinitely.

## Proof of Concept
```bash
# Generate and send 1000 concurrent requests with unique non-existent aliases
for i in $(seq 1 1000); do
  ALIAS=$(python3 -c "import random,string; print(''.join(random.choices('ABCDEFGHIJKLMNOPQRSTUVWXYZ234567', k=40)))")
  curl -s "https://<mirror-node-host>/api/v1/accounts/${ALIAS}/nfts" &
done
wait
```
Each request passes `AccountAlias.isValid()`, enters `getAccountFromAlias`, executes a live `SELECT` against `entity`, returns a 404 (not cached), and the cycle repeats indefinitely. Under sufficient concurrency the PostgreSQL connection pool is exhausted and all subsequent DB-backed endpoints return errors.
