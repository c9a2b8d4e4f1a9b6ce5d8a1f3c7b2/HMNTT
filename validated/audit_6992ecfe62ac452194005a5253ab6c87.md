Audit Report

## Title
Cache Key Nanosecond Timestamp Bypass Causes Permanent Cache Miss and DB Query Amplification for Non-Exchange-Rate System Files

## Summary
`SystemFileLoader.load()` uses a nanosecond-precision wall-clock timestamp as part of the cache key for four system file types (`addressBookFile101`, `addressBookFile102`, `hapiPermissionFile`, `throttleDefinitionFile`). Because this timestamp is unique on every invocation, the `systemFile` cache (`expireAfterWrite=10m,maximumSize=20`) never produces a hit for these files, causing every request that reads them to unconditionally execute a correlated DB query — the exact load the cache was designed to prevent.

## Finding Description

**`Utils.getCurrentTimestamp()`** returns `Instant.now()` converted to nanoseconds:

```java
// web3/src/main/java/org/hiero/mirror/web3/state/Utils.java:57-60
public static long getCurrentTimestamp() {
    final var now = Instant.now();
    return DomainUtils.convertToNanos(now.getEpochSecond(), now.getNano());
}
``` [1](#0-0) 

**`FileReadableKVState.readFromDataSource()`** always passes this nanosecond value to `systemFileLoader.load()` for every system file lookup:

```java
// web3/src/main/java/org/hiero/mirror/web3/state/keyvalue/FileReadableKVState.java:50-58
final var currentTimestamp = getCurrentTimestamp();
if (systemFileLoader.isSystemFile(key)) {
    return systemFileLoader.load(key, currentTimestamp);
}
``` [2](#0-1) 

**`SystemFileLoader.load()`** applies `roundDownToHour()` only for the three exchange-rate-related files. The remaining four system file types receive the raw nanosecond value as the cache key:

```java
// web3/src/main/java/org/hiero/mirror/web3/state/SystemFileLoader.java:113-120
if (fileId.equals(exchangeRateFileId)
        || fileId.equals(feeSchedulesFileId)
        || fileId.equals(simpleFeeSchedulesFileId)) {
    cacheManager = exchangeRatesCacheManager;
    consensusTimestamp = roundDownToHour(consensusTimestamp);  // ← only these 3
}
final var cacheKey = new CacheKey(fileId, consensusTimestamp); // nanosecond for the other 4
``` [3](#0-2) 

Because `cache.get(cacheKey, File.class)` at line 129 always returns `null` for these four types, `loadFromDB()` is called unconditionally on every request: [4](#0-3) 

`loadFromDB()` delegates to `loadWithRetry()`, which calls `fileDataRepository.getFileAtTimestamp()` — an expensive correlated subquery against the `file_data` table: [5](#0-4) 

The `systemFile` cache is configured `expireAfterWrite=10m,maximumSize=20`. With nanosecond-unique keys, the 20-entry limit is saturated immediately and entries are evicted before they can ever be reused: [6](#0-5) 

**Root cause:** The design assumes `(fileId, consensusTimestamp)` will repeat across requests for the same file. The hour-rounding mitigation applied to exchange-rate files was not applied to `addressBookFile101`, `addressBookFile102`, `hapiPermissionFile`, and `throttleDefinitionFile`.

## Impact Explanation
Every HTTP request that causes the EVM to read any of the four affected system files unconditionally executes at least one correlated subquery against the `file_data` table. At the default rate limit of 500 req/s, this produces up to 2,000 unmitigated DB queries/second for system files alone — a sustained amplification that the cache was explicitly designed to eliminate. This can significantly increase DB CPU and I/O load, degrading service for all users.

## Likelihood Explanation
No authentication or special privilege is required. The `/api/v1/contracts/call` endpoint is publicly accessible. The default rate limit is 500 requests/second: [7](#0-6) 

Any valid `eth_call` request that triggers a Hedera precompile reading one of the four affected system files is sufficient. No brute-force or credential is needed. The attack is trivially repeatable and sustainable indefinitely within the rate limit.

## Recommendation
Apply the same `roundDownToHour()` (or an equivalent time-bucketing strategy) to the `consensusTimestamp` for `addressBookFile101`, `addressBookFile102`, `hapiPermissionFile`, and `throttleDefinitionFile` before constructing the `CacheKey`. The rounding interval can be tuned to the acceptable staleness window for each file type. Alternatively, for files that change very infrequently (address books, permissions, throttle definitions), a fixed TTL-based key (e.g., round to the minute or use a constant sentinel) would be sufficient.

## Proof of Concept
1. Send any `eth_call` request that invokes a Hedera precompile requiring throttle or permission data (e.g., a token association precompile call).
2. Observe via DB query logs or metrics that `file_data.getFileAtTimestamp()` is called for file IDs 101, 102, 122, or 123 on **every** request.
3. Confirm that the `systemFile` Caffeine cache hit rate remains at 0% for these file IDs while the exchange-rate cache shows near-100% hit rate after the first request within an hour window.
4. Scale to 500 req/s (within the default rate limit) and observe sustained DB query amplification of up to 2,000 additional queries/second with no cache relief.
