## Audit Report

## Title
Unbounded Storage Trace Response Causes Server-Side Memory Exhaustion in `getContractOpcodes()`

## Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes?storage=true` endpoint re-executes a historical transaction and attaches a freshly allocated `TreeMap<String, String>` of all cumulative storage accesses to **every single opcode entry** in the response. There is no cap on the number of opcodes, storage slots, or total response size. The mandatory `Accept-Encoding: gzip` header mitigates network bandwidth but does not prevent the server from constructing an arbitrarily large in-memory object graph before serialization, enabling server-side heap exhaustion with a single well-crafted request.

## Finding Description

**`OpcodesController.getContractOpcodes()`** accepts `storage=true` from any caller with no authentication. [1](#0-0) 

**`ThrottleManagerImpl.throttleOpcodeRequest()`** consumes one token from a single shared global bucket — there is no per-IP or per-user isolation. The default rate is `opcodeRequestsPerSecond = 1`. [2](#0-1) [3](#0-2) 

During EVM replay, **`AbstractOpcodeTracer.captureStorage()`** is invoked on every opcode via `tracePostExecution`. Each invocation allocates a **new `TreeMap<String, String>`** containing all storage slots accessed so far in the transaction, with no upper bound on map size. [4](#0-3) 

This map is embedded into every `Opcode` object appended to the unbounded `ArrayList` in `OpcodeContext`. The `addOpcodes()` method performs no size check. [5](#0-4) 

The `OpcodeActionTracer.tracePostExecution()` calls `captureStorage()` and appends the result to the opcode list on every single opcode execution. [6](#0-5) 

The final `OpcodesResponse` is built from this list with no truncation or size check. [7](#0-6) 

The gzip enforcement comment explicitly acknowledges the response is "huge" but only addresses **network latency**, not server-side memory. [8](#0-7) 

**Root cause:** The storage snapshot is O(M) per opcode and is replicated N times (once per opcode), yielding O(N × M) total heap allocation before any serialization or compression occurs. The `hexCache` (bounded at 1,600 entries) mitigates string duplication but does not prevent the creation of N separate `TreeMap` objects each holding M entry references. [9](#0-8) 

**Failed assumption:** The design assumes gzip compression is sufficient protection. It is not — compression happens after the full object graph is materialized in the JVM heap.

## Impact Explanation

For a transaction consuming 15 M gas (the configured `maxGasLimit`): [10](#0-9) 

- Maximum unique cold `SSTORE` operations: ~7,142 (at 2,100 gas each)
- Cheap opcodes (e.g., `ADD` at 3 gas) can produce up to ~5 M opcode entries

A realistic worst-case (e.g., 50,000 opcodes × 500 storage slots) produces 25 M `TreeMap` entries. Even with string deduplication via `hexCache`, the TreeMap objects themselves are not reused — N separate TreeMap instances are allocated, each with M entries. This can trigger JVM `OutOfMemoryError`, crash the web3 service pod, or cause GC pauses that degrade all concurrent users. Even at the 1 req/sec global rate, a single attacker can sustain continuous heap pressure.

## Likelihood Explanation

- No authentication is required; any internet user can call the endpoint when enabled.
- The attacker only needs the hash of one historical high-gas transaction (publicly visible on-chain).
- The `Accept-Encoding: gzip` header is trivially supplied by any HTTP client.
- The global rate limit of 1 req/sec does not prevent a single request from exhausting heap; it only limits request frequency.
- The feature flag (`enabled = false` by default) reduces exposure in default deployments, but any operator who enables the feature is immediately exposed. [11](#0-10) 

## Recommendation

1. **Cap the opcode list size**: Enforce a configurable maximum number of opcodes (e.g., `maxOpcodes`) in `OpcodeContext.addOpcodes()` and return a 400/413 error or truncate with a warning when exceeded.
2. **Cap storage slots per opcode**: In `captureStorage()`, limit the number of entries written to the `TreeMap` per invocation.
3. **Share the storage map across opcodes**: Instead of allocating a new `TreeMap` per opcode, maintain a single shared map and pass a reference (or a snapshot only when the map changes), reducing allocation from O(N×M) to O(M).
4. **Add a response size budget**: Track cumulative estimated response size during EVM replay and abort early if a configurable threshold is exceeded.
5. **Per-IP rate limiting**: Replace the single global token bucket with per-IP rate limiting to prevent a single client from monopolizing the 1 req/sec budget.

## Proof of Concept

```bash
# 1. Find a high-gas transaction hash on-chain (e.g., a complex DeFi interaction)
TX_HASH="0x<high_gas_tx_hash>"

# 2. Send a single request with storage=true
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node-host>/api/v1/contracts/results/${TX_HASH}/opcodes?storage=true&stack=false&memory=false" \
  --output /dev/null

# Expected: server allocates O(N x M) TreeMap objects in heap before
# serialization begins; for a 15M gas transaction with many SSTORE ops,
# this can exhaust JVM heap and trigger OutOfMemoryError or severe GC pauses.
```

The attack requires no special tooling, no authentication, and only a single HTTP request targeting a publicly known transaction hash.
