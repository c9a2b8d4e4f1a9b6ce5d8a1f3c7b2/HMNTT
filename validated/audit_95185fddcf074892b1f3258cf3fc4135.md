Audit Report

## Title
Unhandled Plain `Error` on `multipleEvmAddressMatch` Causes HTTP 500 in `getAccountTokenAllowances`

## Summary
In `rest/service/entityService.js`, `getEntityIdFromEvmAddress()` throws a plain `new Error(...)` (not a `RestError` subclass) when multiple alive entities share the same EVM address. The error middleware in `rest/middleware/httpErrorHandler.js` has no handler for plain `Error`, so it falls through to the default `INTERNAL_ERROR` (500) status code. Any caller who supplies an EVM address that matches this DB state receives an opaque 500 response instead of a defined API error.

## Finding Description
**Exact code path:**

1. `rest/controllers/tokenAllowanceController.js` line 69 calls `EntityService.getEncodedId(req.params[filterKeys.ID_OR_ALIAS_OR_EVM_ADDRESS])`.
2. `rest/service/entityService.js` `getEncodedId()` line 124 calls `this.getEntityIdFromEvmAddress(entityId, requireResult)` when the input is an EVM address.
3. `getEntityIdFromEvmAddress()` lines 98–100:
   ```js
   } else if (rows.length > 1) {
     logger.error(`Incorrect db state: ...`);
     throw new Error(EntityService.multipleEvmAddressMatch); // plain Error, NOT RestError
   }
   ```
4. The `catch` block in `getEncodedId()` (lines 128–134) only intercepts `InvalidArgumentError`; all other exceptions — including this plain `Error` — are re-thrown unchanged.
5. `handleError()` in `httpErrorHandler.js` (lines 17–25) checks only `NotFoundError`, `InvalidArgumentError`, `RangeError`, `DbError`, and `HttpError`. A plain `Error` matches none, so `statusCode` stays at `defaultStatusCode = INTERNAL_ERROR` (500).
6. `shouldReturnMessage()` (line 57) returns `true` only for client errors (4xx) or `BAD_GATEWAY`. A 500 is not a client error, so the actual error message is suppressed and a generic 500 body is returned.

**Root cause:** The thrown object is `new Error(...)`, not a subclass of `RestError`. The error handler has no catch-all for unexpected DB integrity errors, so they silently become 500s with no diagnostic information surfaced to the caller.

**Failed assumption:** The code assumes the `multipleEvmAddressMatch` state is purely an ops/monitoring concern (logged via `logger.error`), but the thrown plain `Error` escapes into the HTTP response layer with no defined mapping.

## Impact Explanation
Any client calling `/accounts/:evmAddress/allowances/tokens` when the DB is in this state receives a 500 instead of a 404 or a structured error. The mirror node is a read-only service, so no funds are directly at risk. However, off-chain systems and smart contract integrations that depend on deterministic allowance responses cannot distinguish "address not found" from "server broken," which can lead to incorrect allowance assumptions in dependent logic.

## Likelihood Explanation
The precondition — multiple alive entities sharing one EVM address — is described in the code itself as "Incorrect db state" and should not occur under normal operation. Triggering this requires either a bug in the mirror node ingestion pipeline or a network-level edge case during contract creation that produces duplicate EVM address entries. Once the state exists, any caller who knows the affected EVM address can reliably and repeatedly trigger the 500 with a simple unauthenticated GET request.

## Recommendation
Replace the plain `new Error(...)` throw in `getEntityIdFromEvmAddress()` with a `RestError` subclass (e.g., a new `DbStateError` extending `RestError`) and add a corresponding handler in `handleError()` that maps it to an appropriate HTTP status code (e.g., 409 Conflict or 503 Service Unavailable). This ensures the error is handled deterministically and the response is meaningful to callers.

The same pattern exists in `getAccountFromAlias()` at line 49 and should be fixed consistently:
```js
// rest/service/entityService.js line 49
throw new Error(EntityService.multipleAliasMatch); // also plain Error
```

## Proof of Concept
1. Arrange the database to have two rows in the `entity` table with `deleted <> true` and the same `evm_address` value (simulating the "Incorrect db state").
2. Send a GET request:
   ```
   GET /api/v1/accounts/0x<affected_evm_address>/allowances/tokens
   ```
3. Observe the response:
   ```json
   HTTP/1.1 500 Internal Server Error
   { "_status": { "messages": [{ "message": "Internal error" }] } }
   ```
   instead of a 404 or a structured error with a meaningful message.
