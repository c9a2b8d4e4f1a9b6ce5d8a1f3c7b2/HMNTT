Audit Report

## Title
Cache TTL Asymmetry Enables Sustained Low-Rate Amplified Batch DB Query Attack

## Summary
A TTL asymmetry between the contract-state value cache (`expireAfterWrite=2s`) and the per-contract slot-key accumulator cache (`expireAfterAccess=5m`) in `ContractStateServiceImpl` allows an attacker to force large IN-clause DB queries (`WHERE slot IN (up to 1500 entries)`) at a rate of one per 2 seconds using a single HTTP request per trigger — well below any rate-limit threshold.

## Finding Description

**Verified code path:**

`ContractStateServiceImpl.findStorage()` checks `contractStateCache` first. On any miss (or `EMPTY_VALUE`), it unconditionally calls `findStorageBatch()`. [1](#0-0) 

Inside `findStorageBatch()`, the per-contract slot-key cache is read in full and every accumulated key is passed to the DB: [2](#0-1) 

**Verified TTL configuration (from `CacheProperties.java`):**

| Cache field | Config string | Effective lifetime |
|---|---|---|
| `contractState` | `expireAfterWrite=2s,maximumSize=25000` | 2 seconds |
| `slotsPerContract` | `expireAfterAccess=5m,maximumSize=1500` | up to 5 min, reset on access |
| `contractSlots` | `expireAfterAccess=5m,maximumSize=3000` | up to 5 min per contract entry | [3](#0-2) 

**Root cause:** Slot keys accumulate in the per-contract cache for up to 5 minutes (max 1500 entries). Their corresponding values in `contractStateCache` expire every 2 seconds. There is no mechanism that clears accumulated slot keys when their values expire. After every 2-second window, the full accumulated key set is re-queried against the DB.

**Exploit flow:**

1. **Seed phase (one-time):** Attacker sends 1500 requests, each for a distinct slot of a target contract. Each request adds a new key to the per-contract cache via `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)`. The `expireAfterAccess` TTL resets on every access, keeping keys alive. [4](#0-3) 

2. **Sustained phase:** Every ≥2 seconds, attacker sends one request for any slot of that contract. `contractStateCache` is expired, so `findStorageBatch()` is called. It reads all 1500 accumulated keys and issues:
   ```sql
   SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (1500 entries)
   ``` [5](#0-4) 

3. **Multi-contract multiplication:** The outer `contractSlots` cache allows up to 3000 contracts to be targeted simultaneously, multiplying the effect proportionally. [6](#0-5) 

**Existing mitigations shown insufficient:**
- `requestsPerSecond=500` throttle: The sustained attack requires only 0.5 req/s (1 request per 2s).
- `maximumSize=1500` on `slotsPerContract`: Caps batch size but does not prevent the attack.

## Impact Explanation

Each triggered `findStorageBatch()` call issues a parameterized SQL query with up to 1500 bind parameters against the `contract_state` table. [7](#0-6) 

On a production node with a large `contract_state` table, repeated 1500-slot IN-clause queries every 2 seconds per targeted contract constitute a sustained, amplified DB load increase achievable with a negligible HTTP request rate. An attacker targeting multiple contracts (up to 3000 simultaneously) multiplies this effect proportionally. The DB `statementTimeout` of 3000ms means each query can hold a DB connection for up to 3 seconds — longer than the 2-second re-trigger interval — enabling connection pool exhaustion as a secondary effect.

## Likelihood Explanation

The attack requires no authentication, no special tooling, and no precise timing. The seed phase (1500 requests) can be spread over minutes to avoid detection. The sustained phase requires only 1 request every 2 seconds per targeted contract. Any public-facing deployment of the web3 API is exposed. The attack is fully self-sustaining because each access resets the `expireAfterAccess` TTL on the slot-key accumulator, keeping the accumulated keys alive indefinitely. [8](#0-7) 

## Recommendation

1. **Align TTLs:** Set `slotsPerContract` to use `expireAfterWrite` (not `expireAfterAccess`) with a TTL equal to or shorter than `contractState`'s 2-second TTL, so slot keys do not outlive their cached values.
2. **Bound batch size per request:** Cap the number of slot keys passed to `findStorageBatch()` to a small constant (e.g., the keys accessed in the current request window only), rather than the full accumulated set.
3. **Evict slot keys on value expiry:** When a value expires from `contractStateCache`, also evict the corresponding key from the per-contract slot-key accumulator.
4. **Per-contract rate limiting:** Apply per-contract-ID rate limiting on `findStorageBatch()` invocations independently of per-IP HTTP rate limiting.

## Proof of Concept

```python
import requests, time

TARGET = "https://<mirror-node>/api/v1/contracts/call"
CONTRACT = "0x<target_contract>"

# Seed phase: fill slotsPerContract cache with 1500 distinct slots
for i in range(1500):
    slot = hex(i).zfill(66)  # distinct slot keys
    requests.post(TARGET, json={"to": CONTRACT, "data": "0x<read_slot_calldata_" + str(i) + ">"})

# Sustained phase: 1 request every 2s triggers a 1500-slot IN-clause DB query
while True:
    requests.post(TARGET, json={"to": CONTRACT, "data": "0x<any_slot_calldata>"})
    time.sleep(2)  # just above the contractStateCache TTL
```

Each iteration of the loop causes `findStorageBatch()` to execute `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (1500 entries)` against the database. [9](#0-8)

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
