Audit Report

## Title
Global Throttle Exhaustion via Default `stack=true` Enabling Unauthenticated Maximum-Cost EVM Re-execution DoS

## Summary
The `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` endpoint defaults `stack=true`, triggering the most expensive EVM re-execution path on every bare request. The sole rate-limiting guard is a single application-wide token bucket with no per-IP partitioning, allowing a single unauthenticated attacker to exhaust the entire endpoint quota and deny service to all legitimate users.

## Finding Description

`OpcodesController.java` line 55 sets `stack` to `true` by default, meaning every request that omits the parameter triggers full per-opcode stack capture:

```java
@RequestParam(required = false, defaultValue = "true") boolean stack,
``` [1](#0-0) 

This value is passed directly into `OpcodeRequest` and flows to `captureStack()` in `AbstractOpcodeTracer`, which iterates the entire EVM operand stack at every opcode step during re-execution:

```java
protected final List<String> captureStack(final MessageFrame frame, final OpcodeContext options) {
    if (!options.isStack()) {
        return Collections.emptyList();
    }
    var size = frame.stackSize();
    var stack = new ArrayList<String>(size);
    for (var i = 0; i < size; ++i) { ... }
``` [4](#0-3) 

The only guards before this expensive work are a trivially-satisfied `Accept-Encoding: gzip` header check and `throttleOpcodeRequest()`:

```java
validateAcceptEncodingHeader(acceptEncoding);
throttleManager.throttleOpcodeRequest();
``` [7](#0-6) 

`throttleOpcodeRequest()` consumes from a single application-wide `opcodeRateLimitBucket` Spring singleton:

```java
public void throttleOpcodeRequest() {
    if (!opcodeRateLimitBucket.tryConsume(1)) {
        throw new ThrottleException(REQUEST_PER_SECOND_LIMIT_EXCEEDED);
    }
}
``` [8](#0-7) 

This bucket is a plain Spring `@Bean` singleton with no IP or identity partitioning:

```java
@Bean(name = OPCODE_RATE_LIMIT_BUCKET)
Bucket opcodeRateLimitBucket() {
    long rateLimit = throttleProperties.getOpcodeRequestsPerSecond();
    ...
    return Bucket.builder().addLimit(limit).build();
}
``` [3](#0-2) 

**Root cause:** The failed assumption is that a global rate limit is sufficient to bound resource consumption. It is not, because (a) the limit is global — one source exhausts the entire quota — and (b) `stack=true` by default ensures every consumed token triggers the most expensive EVM re-execution path without the attacker needing any knowledge of tracing options.

## Impact Explanation

- **Availability (DoS):** A single attacker sending requests at the configured rate limit drains the global bucket entirely. All concurrent legitimate users receive HTTP 429 for the duration of the attack.
- **Resource exhaustion:** `stack=true` by default causes the server to allocate and serialize the full EVM operand stack at every opcode step. For complex contracts this creates significant CPU and heap pressure per request — the attacker achieves maximum damage per token consumed.
- **No privilege required:** The endpoint has no authentication, no IP-based limiting, and no proof-of-work. The only prerequisite is a valid transaction hash and an `Accept-Encoding: gzip` header, both trivially obtained.

## Likelihood Explanation

Any attacker who can reach the web3 service on a deployment where the feature is enabled (note: `enabled = false` by default, so this only affects operators who have explicitly turned it on) can execute this with a single `curl` loop. No special credentials or tooling are required. The `stack=true` default means the attacker does not need to know about tracing options — the most expensive path is taken automatically. The attack is fully repeatable and sustainable indefinitely.

## Recommendation

1. **Change the `stack` default to `false`** in `OpcodesController.java` line 55. Expensive tracing options should require explicit opt-in, not opt-out.
2. **Implement per-IP (or per-client) rate limiting** for the opcode endpoint. The current global `opcodeRateLimitBucket` singleton should be replaced or supplemented with a per-source bucket (e.g., using a `LoadingCache<String, Bucket>` keyed by client IP).
3. **Consider requiring authentication** for this endpoint, given its inherently expensive nature.
4. **Document the `enabled = false` default** as a security control and ensure operators understand the exposure before enabling the feature in public-facing deployments.

## Proof of Concept

```bash
# Attacker loop — no stack parameter needed, gzip header trivially satisfied
while true; do
  curl -s -H "Accept-Encoding: gzip" \
    "https://<host>/api/v1/contracts/results/<valid_tx_hash>/opcodes" \
    -o /dev/null
  sleep 1
done
```

This loop, running from a single IP with no credentials, consumes every token from the global `opcodeRateLimitBucket` at the configured rate. All other clients attempting the same endpoint receive HTTP 429 for the duration of the attack, while the server performs full EVM re-execution with per-opcode stack capture on each request.
