Audit Report

## Title
NullPointerException in `evmAddressFromId` When `ContractResult.senderId` Is Null

## Summary
`OpcodeServiceImpl.getSenderAddress()` passes `contractResult.getSenderId()` directly to `CommonEntityAccessor.evmAddressFromId()` without a null check. When a stored `ContractResult` has a null `senderId`, the call to `entityId.getId()` inside `evmAddressFromId` throws a `NullPointerException`, aborting the request with a 500 error.

## Finding Description
`ContractResult.senderId` is declared as a plain nullable field with no `@NonNull` annotation and no default value: [1](#0-0) 

`OpcodeServiceImpl.getSenderAddress()` passes this value directly to `evmAddressFromId` with no null guard: [2](#0-1) 

Inside `evmAddressFromId`, the `entityId` parameter has no `@NonNull` annotation (unlike the `get(EntityId, ...)` overload at line 60 which does). With `timestamp` being `Optional.empty()`, the `.orElseGet()` branch always executes and immediately dereferences `entityId.getId()`: [3](#0-2) 

The post-call guard `address != null ? address : EMPTY_ADDRESS` in `getSenderAddress` is never reached because the NPE fires inside `evmAddressFromId` before returning.

## Impact Explanation
Any call to `/api/v1/contracts/results/{transactionIdOrHash}/opcodes` that resolves to a `ContractResult` row with a null `sender_id` column will receive an unhandled 500 Internal Server Error. The full request processing pipeline for that call is aborted. There is no data leakage, no privilege escalation, and no economic damage — impact is limited to denial of the opcode-tracing service for affected transactions.

## Likelihood Explanation
The `senderId` column is nullable in the domain model with no `NOT NULL` constraint enforced at the Java layer. Historical or system-initiated contract executions may legitimately have no sender recorded. An unprivileged user only needs to discover one such transaction hash or ID (e.g., by iterating known contract transaction hashes) and submit a single HTTP GET to reproduce the crash. No special privileges, no write access, and no network-level attack are required.

## Recommendation
Add a null check in `getSenderAddress` before calling `evmAddressFromId`:

```java
private Address getSenderAddress(ContractResult contractResult) {
    final var senderId = contractResult.getSenderId();
    if (senderId == null) {
        return EMPTY_ADDRESS;
    }
    final var address = commonEntityAccessor.evmAddressFromId(senderId, Optional.empty());
    return address != null ? address : EMPTY_ADDRESS;
}
```

Alternatively, add a `@NonNull` guard or null check at the top of `evmAddressFromId` in `CommonEntityAccessor` to make it consistent with the `get(@NonNull EntityId, ...)` overload at line 60. [4](#0-3) 

## Proof of Concept
1. Identify any `contract_result` row in the database where `sender_id IS NULL`.
2. Obtain the `consensus_timestamp` for that row and look up its corresponding transaction hash.
3. Issue: `GET /api/v1/contracts/results/{hash}/opcodes`
4. The call stack reaches `OpcodeServiceImpl.getSenderAddress()` → `CommonEntityAccessor.evmAddressFromId(null, Optional.empty())` → `null.getId()` → `NullPointerException` → HTTP 500.
