## Audit Report

## Title
Deleted Entity Bypass via Long-Zero EVM Address in GraphQL `getByEvmAddressAndType`

## Summary
`EntityServiceImpl.getByEvmAddressAndType()` routes long-zero EVM addresses through `entityRepository.findById()`, which carries no `deleted is not true` filter. The two other lookup paths (`findByEvmAddress`, `findByAlias`) explicitly exclude soft-deleted entities. An unauthenticated caller can craft a long-zero EVM address encoding any known entity ID and retrieve a soft-deleted account or contract through the GraphQL API.

## Finding Description

**Confirmed code path** — `EntityServiceImpl.java` lines 34–41:

```java
public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
    byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
    var buffer = ByteBuffer.wrap(evmAddressBytes);
    if (buffer.getInt() == 0 && buffer.getLong() == 0) {
        return entityRepository.findById(buffer.getLong())   // no deletion filter
                               .filter(e -> e.getType() == type);
    }
    return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
}
``` [1](#0-0) 

**Asymmetric deletion filtering confirmed** in `EntityRepository.java`:

- `findByAlias`: `select * from entity where alias = ?1 and deleted is not true`
- `findByEvmAddress`: `select * from entity where evm_address = ?1 and deleted is not true`
- `findById` (inherited from `CrudRepository<Entity, Long>`): **no deletion predicate** [2](#0-1) 

**`ByteBuffer` read sequence**: 4 bytes (`getInt()` → shard), 8 bytes (`getLong()` → realm), 8 bytes (`getLong()` → entity num. When shard == 0 and realm == 0, the third read is passed directly to `findById`. [3](#0-2) 

**Additional scope**: `getByIdAndType` (line 24–25) also calls `findById` without a deletion filter, meaning the same bypass is reachable via the `entityId` GraphQL input field — a simpler attack vector the report did not mention. [4](#0-3) 

**GraphQL surface**: The `account` query accepts `evmAddress` as input and the `Account` type exposes `deleted`, `key`, `memo`, `expirationTimestamp`, and other metadata fields. [5](#0-4) [6](#0-5) 

## Impact Explanation
Any soft-deleted account or contract (marked `deleted = true` in the `entity` table) can be fully resolved through the public GraphQL mirror-node API. The response includes admin keys, memo, expiry timestamps, type, and creation timestamps. Because this is a read-only mirror-node API, no funds are at risk. The severity is **Medium** — information disclosure of entity metadata that the `deleted is not true` guards on alias/EVM-address paths are clearly intended to suppress.

## Likelihood Explanation
Exploitation requires zero privileges and zero on-chain interaction. All Hedera entity IDs are permanently visible in public transaction history. The crafted 20-byte long-zero address is a trivially computed hex string. The attack is fully repeatable and automatable against any entity ID ever deleted on the network.

## Recommendation
Replace the bare `findById` call in the long-zero branch of `getByEvmAddressAndType` with a custom repository method that includes the deletion guard, consistent with the other two lookup paths:

```java
// In EntityRepository (graphql module)
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(Long id);
```

Apply the same fix to `getByIdAndType` (line 25), which has the identical omission via the `entityId` input path.

## Proof of Concept

1. Identify a deleted contract with entity num `N` (e.g., from public Hedera transaction history).
2. Construct the 20-byte long-zero address: `000000000000000000000000` + zero-padded big-endian 8-byte encoding of `N` (hex).
3. Submit the following GraphQL query to `/graphql/alpha`:

```graphql
query {
  account(input: { evmAddress: "0x000000000000000000000000000000000000<N_HEX>" }) {
    deleted
    key
    memo
    expirationTimestamp
    entityId { shard realm num }
  }
}
```

4. The response returns the full entity record for the deleted entity. The `deleted: true` field confirms the bypass. The `findById` call at line 38 executes with no deletion predicate, returning the row that `findByEvmAddress` would have suppressed. [7](#0-6)
