Audit Report

## Title
404 Response Cache Poisoning via `Cache-Control: public, max-age=5` on `/api/v1/topics/{id}` Not-Found Responses

## Summary
The `@ModelAttribute` method in `GenericControllerAdvice` unconditionally writes `Cache-Control: public, max-age=5` onto the raw `HttpServletResponse` before the handler executes. When `TopicServiceImpl.findById()` throws `EntityNotFoundException`, the `notFound` `@ExceptionHandler` returns a `ResponseEntity` with `null` headers. Spring's response-writing path does not clear headers already committed to the underlying servlet response, so the 404 reply is emitted with `Cache-Control: public, max-age=5`. Any shared reverse proxy or CDN that honours this directive will cache the 404 for up to 5 seconds.

## Finding Description

**Step 1 — Controller invokes service:**
`TopicController.getTopic()` calls `topicService.findById(id.id())` as its first statement. [1](#0-0) 

**Step 2 — Service throws on missing topic:**
`TopicServiceImpl.findById()` calls `orElseThrow(() -> new EntityNotFoundException("Topic not found"))`. [2](#0-1) 

**Step 3 — `@ModelAttribute` pre-writes caching headers:**
Before the handler method runs, Spring MVC invokes `responseHeaders()`, which calls `response::setHeader` for every header configured for the matched path — including `Cache-Control: public, max-age=5` — directly on the `HttpServletResponse`. [3](#0-2) 

**Step 4 — Exception handler passes `null` headers:**
The `notFound` handler calls `handleExceptionInternal(e, null, null, NOT_FOUND, request)` — the third argument (`headers`) is `null`. [4](#0-3) 

**Step 5 — `handleExceptionInternal` returns a `ResponseEntity` with null headers:**
The overridden `handleExceptionInternal` builds `new ResponseEntity<>(errorResponse, headers, statusCode)` where `headers` is `null`. [5](#0-4) 

Spring's `HttpEntityMethodProcessor` writes the `ResponseEntity`'s (empty) header map to the response but does **not** call `response.reset()` or otherwise clear headers already set on the underlying `HttpServletResponse`. The `Cache-Control: public, max-age=5` header written in Step 3 therefore survives on the 404 reply.

**Step 6 — Test confirms `Cache-Control: public, max-age=5` on success but has no assertion for 404:**
The `success()` test explicitly asserts `getCacheControl().isEqualTo("public, max-age=5")`. [6](#0-5) 

The `entityNotFound` test only validates the error message body; it never checks the `Cache-Control` header on the 404 response. [7](#0-6) 

**Step 7 — The Node.js guard is irrelevant:**
The Node.js `responseCacheUpdateHandler` skips non-2xx responses via `httpStatusCodes.isSuccess(res.statusCode)`, but `/api/v1/topics/{id}` is served by the **Spring Boot rest-java** service, not the Node.js layer. No equivalent guard exists in the Spring layer. [8](#0-7) 

## Impact Explanation
Any reverse proxy or CDN deployed in front of the mirror node that honours `Cache-Control: public` will cache the 404 for up to 5 seconds. During that window every user routed through the same cache node receives the stale not-found error for a topic that may have just been created. Integrations that poll the endpoint immediately after a `ConsensusCreateTopic` transaction is ingested are broken for the TTL duration. There is no direct economic loss; the impact is availability degradation / griefing, severity Medium.

## Likelihood Explanation
The attack requires no credentials, no special network position, and no rate-limit bypass — a single unauthenticated HTTP GET suffices. Hedera topic IDs are monotonically increasing integers visible on the public ledger, making the next ID predictable. The 5-second TTL limits each poisoning window, but the attack is trivially repeatable every 5 seconds, extending the effective denial window indefinitely. Any production deployment behind a shared CDN (a common configuration) is affected.

## Recommendation
Override `handleExceptionInternal` (or add a `javax.servlet.Filter` / Spring `HandlerInterceptor`) to explicitly set `Cache-Control: no-store` on the `HttpServletResponse` whenever the response status is 4xx or 5xx. Alternatively, move the caching-header logic out of the `@ModelAttribute` hook and into a post-processing step that is aware of the final response status (e.g., a `ResponseBodyAdvice` or a `Filter` that inspects the committed status code). Add a test assertion in `entityNotFound` that verifies the 404 response carries `Cache-Control: no-store` (or equivalent non-cacheable directive).

## Proof of Concept
```
# 1. Identify the next topic ID from the public ledger (e.g., 0.0.5001)
# 2. Pre-fetch before the mirror node ingests the ConsensusCreateTopic transaction:
curl -v https://<mirror-node>/api/v1/topics/0.0.5001
# Response: HTTP/1.1 404  Cache-Control: public, max-age=5
# 3. Any CDN/reverse proxy caches this 404 for 5 seconds.
# 4. Within those 5 seconds, legitimate clients receive the cached 404
#    even after the topic has been ingested by the mirror node.
# 5. Repeat every 5 seconds to extend the denial window indefinitely.
```

const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
};
