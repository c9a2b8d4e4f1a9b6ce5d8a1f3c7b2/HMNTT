Audit Report

## Title
`@Cacheable` `unless` Guard Ineffective for `Optional.empty()` — Cache Pollution via Non-Existent Contract IDs

## Summary
`findRuntimeBytecode()` in `ContractRepository` uses `unless = "#result == null"` to suppress caching of missing-contract results, but the method returns `Optional<byte[]>`. `Optional.empty()` is a non-null singleton object, so the guard always evaluates to `false` and every "not found" lookup is stored in the cache. An attacker can saturate the 1,000-slot contract bytecode cache with empty entries using only 1,000 `eth_call` requests targeting non-existent contract IDs, evicting all legitimate cached bytecodes and forcing repeated database queries for every subsequent real contract call for up to one hour.

## Finding Description

**Exact location — `ContractRepository.java` lines 16–18:**

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
``` [1](#0-0) 

**Root cause:** The SpEL expression `#result == null` tests the `Optional` wrapper for nullity. `Optional.empty()` is never `null`; it is a well-known singleton. Spring's `@Cacheable` therefore stores every "not found" result under its `contractId` key. The correct guard would be `unless = "#result == null || !#result.isPresent()"`.

**Confirmed call chain:**

`ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode(entityId.getId())` at lines 42–43. For any `contractId` with no matching row, the native query returns zero rows; Spring Data JPA wraps this as `Optional.empty()`, which passes the broken guard and is cached. [2](#0-1) 

**Cache configuration — `CacheProperties.java` line 22:**

```
contract = "expireAfterAccess=1h,maximumSize=1000,recordStats"
``` [3](#0-2) 

The cache manager is wired in `EvmConfiguration.cacheManagerContract()`: [4](#0-3) 

**No eviction path exists:** A full-repository search for `@CacheEvict` returns zero matches in the `web3` module. Once an empty entry is stored, it persists for up to one hour with no mechanism to remove it.

## Impact Explanation

The contract bytecode cache has a hard cap of 1,000 entries (`maximumSize=1000`). An attacker who sends 1,000 `eth_call` requests each targeting a distinct non-existent contract address fills the entire cache with `Optional.empty()` entries. Caffeine's LRU eviction then removes all legitimate cached bytecodes. For the following hour (`expireAfterAccess=1h`), every `eth_call` to a real contract must execute a synchronous database query to fetch bytecode, increasing database load and response latency proportionally to traffic. The attack is repeatable: as legitimate entries re-populate the cache, the attacker can re-flood it. There is no data loss or unauthorized access, but infrastructure is continuously degradable.

## Likelihood Explanation

No authentication or rate-limiting specific to contract-ID diversity is enforced at the `eth_call` endpoint. Any 20-byte EVM address value is accepted. A single script generating 1,000 distinct long-zero addresses (e.g., `0x0000…0001` through `0x0000…03E8`) is sufficient to saturate the cache. The gas throttle limits throughput but does not prevent the attack — each call can use minimum gas (21,000), and only 1,000 requests are needed to fill the cache, far below any practical rate limit.

## Recommendation

Fix the `unless` SpEL expression to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only results containing actual bytecode are stored in the cache, while "not found" lookups are never persisted.

## Proof of Concept

1. Generate 1,000 distinct non-existent contract IDs (e.g., numeric IDs `1` through `1000` that have no rows in the `contract` table).
2. Send 1,000 `eth_call` JSON-RPC requests, each with `"to"` set to the corresponding long-zero EVM address (`0x0000000000000000000000000000000000000001` through `0x00000000000000000000000000000000000003E8`).
3. Each request causes `findRuntimeBytecode()` to execute the native query, receive zero rows, return `Optional.empty()`, and — because `Optional.empty() != null` — store the empty result in the cache.
4. After 1,000 requests, the cache is full of empty entries. Caffeine evicts all previously cached legitimate bytecodes.
5. Issue `eth_call` to any real contract: observe that the cache no longer contains its bytecode and a database query is executed on every call.
6. The state persists for up to one hour. Repeat step 2 to maintain the degraded state indefinitely.
