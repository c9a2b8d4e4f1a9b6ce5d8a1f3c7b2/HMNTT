# Audit Report

## Title
Unauthenticated Unbounded DB Query via `/contracts/results/logs` Without Timestamp Filters

## Summary
The `getContractLogs` handler at `GET /api/v1/contracts/results/logs` unconditionally enters the `optimizeTimestampFilters` → `bindTimestampRange` path when no `transaction_hash` is present and no `contractId` is provided. With the default configuration (`bindTimestampRange: false`), a request carrying no timestamp parameters produces a SQL query against `contract_log` with no timestamp `WHERE` clause, allowing any unauthenticated user to trigger expensive DB queries at will.

## Finding Description

**Code path (all confirmed against source):**

`getContractLogs` (line 835, `rest/controllers/contractController.js`) calls `extractContractLogsMultiUnionQuery` with no `contractId`: [1](#0-0) 

Inside `extractContractLogsMultiUnionQuery`, when `contractId === undefined` and `transactionHash === undefined`, the optimization branch is unconditionally entered: [2](#0-1) 

`optimizeTimestampFilters` (line 387) calls `parseTimestampFilters` with `filterRequired=false`, so an empty filter list returns `{range: null, eqValues: [], neValues: []}` without error. Because `eqValues.length === 0`, it calls `bindTimestampRange(null, order)`: [7](#0-6) 

`bindTimestampRange` (line 20, `rest/timestampRange.js`) returns `{range: null}` immediately when `queryConfig.bindTimestampRange` is `false` (the default): [4](#0-3) 

Back in `optimizeTimestampFilters`, with `optimizedRange === null`, neither the `begin` nor `end` guard is truthy, so no timestamp filters are pushed: [5](#0-4) 

The result is an empty `timestampFilters` list returned to `extractContractLogsMultiUnionQuery`, and `bounds.primary` remains empty. The generated SQL query against `contract_log` carries no timestamp `WHERE` clause.

**Why existing checks fail:**

`checkTimestampsForTopics` only enforces timestamp presence when at least one `topic*` filter is present. A bare request with no topics and no timestamp passes through without error: [6](#0-5) 

There is no other guard requiring a timestamp or `contractId` for this endpoint. [8](#0-7) 

## Impact Explanation
Every unauthenticated `GET /api/v1/contracts/results/logs` request with no `timestamp`, `topic*`, or `transaction_hash` parameters executes a query against `contract_log` with no timestamp bounds. While the `LIMIT` clause caps the number of rows returned, the database must still locate those rows — on a large table without a selective filter, this requires scanning from the index root across potentially millions of entries. Concurrent flooding of this endpoint can saturate DB I/O and CPU, degrading query processing for all mirror-node consumers.

## Likelihood Explanation
The endpoint is public, documented in the OpenAPI spec, and requires zero authentication. The exploit is a single `GET` with no parameters, trivially scriptable. Any attacker with network access to the REST API can execute it repeatedly.

## Recommendation
Enforce at least one of the following for requests to `/contracts/results/logs` that lack a `contractId` or `transaction_hash`:
- Require at least one `timestamp` filter (reject requests with no timestamp bounds).
- Enable `bindTimestampRange` by default, or make it non-optional for this endpoint, so that an implicit bounded range is always applied.
- Add a dedicated guard in `getContractLogs` analogous to `checkTimestampsForTopics` that rejects requests with no timestamp and no `transaction_hash`.

## Proof of Concept
```

Host: <mirror-node-host>
```
No authentication, no parameters. With `bindTimestampRange: false` (default), the resulting SQL contains no timestamp `WHERE` clause on `contract_log`. Repeating this request concurrently stresses the database.

const optimizeTimestampFilters = async (timestampFilters, order) => {
  const filters = [];

  const {range, eqValues, neValues} = utils.parseTimestampFilters(timestampFilters, false, true, true, false, false);
  if (range?.isEmpty()) {
    return {filters};
  }

  const {range: optimizedRange, next} = eqValues.length === 0 ? await bindTimestampRange(range, order) : {range};
  if (optimizedRange?.begin) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.gte, value: optimizedRange.begin});
  }

  if (optimizedRange?.end) {
    filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.lte, value: optimizedRange.end});
  }

  eqValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.eq, value}));
  neValues.forEach((value) => filters.push({key: filterKeys.TIMESTAMP, operator: utils.opsMap.ne, value}));

  return {filters, next};
};
