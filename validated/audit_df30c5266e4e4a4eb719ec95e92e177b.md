# Audit Report

## Title
Hash Collision in `CACHE_NAME_ALIAS` Cache Allows Wrong Entity Resolution via 32-bit `Arrays.hashCode` Key

## Summary
`findByEvmAddressOrAliasAndDeletedIsFalse` uses `Arrays.hashCode(byte[])` — a 32-bit integer — as the sole cache key for the `CACHE_NAME_ALIAS` cache. Because this single method handles both alias-based and EVM-address-based lookups (via `WHERE evm_address = ?1 OR alias = ?1`), two distinct byte arrays belonging to different entities that produce the same `Arrays.hashCode()` value will collide: the second lookup returns the first entity from cache instead of querying the database.

## Finding Description
`findByEvmAddressOrAliasAndDeletedIsFalse` is annotated with:

```java
@Cacheable(
    cacheNames = CACHE_NAME_ALIAS,
    cacheManager = CACHE_MANAGER_ENTITY,
    key = "@spelHelper.hashCode(#alias)",
    unless = "#result == null")
``` [5](#0-4) 

`spelHelper.hashCode(byte[])` delegates to `Arrays.hashCode(value)`, which computes a 32-bit signed integer via the polynomial `result = 31*result + b[i]`. [2](#0-1) 

The underlying SQL query accepts both an alias and an EVM address as the same parameter:

```sql
WHERE (evm_address = ?1 OR alias = ?1) AND deleted IS NOT TRUE
``` [6](#0-5) 

This means the same `CACHE_NAME_ALIAS` cache stores results for both alias-keyed and EVM-address-keyed lookups. Spring's `@Cacheable` performs no secondary equality check on the actual byte array before returning a cached value — it returns whatever is stored under the matching integer key. Two distinct byte arrays `X` (alias of entity A) and `Y` (EVM address of entity B) satisfying `Arrays.hashCode(X) == Arrays.hashCode(Y)` will collide: a lookup for `Y` returns entity A from cache.

The `CACHE_NAME_ALIAS` and `CACHE_NAME_EVM_ADDRESS` caches are separate (no cross-cache collision), but the intra-cache collision within `CACHE_NAME_ALIAS` is real because `findByEvmAddressOrAliasAndDeletedIsFalse` is the single entry point for both input types. [3](#0-2) 

**Failed assumption:** The cache assumes `Arrays.hashCode(X) == Arrays.hashCode(Y)` implies `X == Y`. This is false for a 32-bit hash over arbitrary-length byte arrays.

## Impact Explanation
The mirror node's web3 JSON-RPC API is used by DApps and wallets to resolve Hedera aliases and EVM addresses to account entities (balance, EVM address, account ID). A cache collision causes the API to return entity A's data in response to a query for entity B's address. DApps that use this resolution to determine transfer destinations will receive incorrect entity data, potentially misdirecting HBAR or token transfers.

## Likelihood Explanation
The attacker requires no privileges beyond:
1. The ability to register a Hedera account (permissionless on mainnet/testnet).
2. Two sequential unauthenticated HTTP requests to the mirror node's web3 API within the 1-second cache TTL (`expireAfterWrite=1s`). [4](#0-3) 

Finding a 20-byte EVM address `Y` such that `Arrays.hashCode(Y) == Arrays.hashCode(X)` is a preimage search over a 32-bit output space, requiring ~2³² ≈ 4 billion iterations of a 5-operation polynomial loop — feasible in under a minute on commodity hardware. On Hedera, an account can be created with an arbitrary 20-byte EVM address as its alias via `CryptoCreate`, so the attacker can register entity B with the crafted address `Y` directly.

## Recommendation
Replace the 32-bit `Arrays.hashCode` cache key with a collision-resistant key. Options in increasing robustness:

1. **Use the full byte array as the cache key directly** (if the cache implementation supports it and performs structural equality): remove the `key = "@spelHelper.hashCode(#alias)"` expression and let Spring use the default parameter-based key, which uses `Arrays.equals` semantics via `SimpleKey`.
2. **Use a cryptographic hash** (e.g., `Arrays.toString(MessageDigest.getInstance("SHA-256").digest(alias))`) as the cache key string — collision-resistant and still compact.
3. **Separate the cache by lookup type**: use distinct methods (and distinct caches) for alias-only vs. EVM-address-only lookups, eliminating the shared key space entirely. A separate `findByAliasAndDeletedIsFalse` method with its own cache would prevent cross-type collisions by construction.

Option 3 is the most architecturally sound, as it also aligns with the existing separation between `findByEvmAddressAndDeletedIsFalse` (using `CACHE_NAME_EVM_ADDRESS`) and `findByEvmAddressOrAliasAndDeletedIsFalse` (using `CACHE_NAME_ALIAS`). [7](#0-6) 

## Proof of Concept

// Offline: find Y (20 bytes) such that Arrays.hashCode(Y) == Arrays.hashCode(X)
// where X is the known alias of entity A (e.g., 32-byte Ed25519 public key)
byte[] X = /* entity A's alias */;
int targetHash = Arrays.hashCode(X);

byte[] Y = new byte[20];
Random rng = new Random();
while (true) {
    rng.nextBytes(Y);
    if (Arrays.hashCode(Y) == targetHash) break; // found collision ~2^32 iterations
}

// Register Hedera account with EVM address = Y (entity B)
// CryptoCreate transaction with alias = Y

// Step 1: trigger caching of entity A
httpGet("/api/v1/contracts/" + toHex(X)); // calls findByEvmAddressOrAliasAndDeletedIsFalse(X)
                                           // → cached under key targetHash

// Step 2: within 1 second, query entity B via its EVM address Y
httpGet("/api/v1/contracts/" + toHex(Y)); // calls findByEvmAddressOrAliasAndDeletedIsFalse(Y)
                                           // → Arrays.hashCode(Y) == targetHash → cache HIT
                                           // → returns entity A's data instead of entity B's
```
