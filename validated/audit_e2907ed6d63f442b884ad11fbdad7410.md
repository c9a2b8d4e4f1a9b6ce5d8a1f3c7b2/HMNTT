Audit Report

## Title
Deleted-Entity Bypass via Crafted EVM Address Exposes Account Data Through `getByEvmAddressAndType()`

## Summary
`getByEvmAddressAndType()` in `EntityServiceImpl` routes a crafted 20-byte EVM address (first 12 bytes zero, last 8 bytes encoding any entity ID) to `entityRepository.findById()`, which is Spring Data's `CrudRepository.findById()` and carries no `deleted is not true` filter. Both sibling lookup paths (`findByEvmAddress`, `findByAlias`) explicitly filter deleted entities at the SQL level, but the `findById` branch does not. An unauthenticated caller can therefore retrieve the full `Entity` record of a deleted account, including `pendingReward`, `balance`, `key`, and `memo`.

## Finding Description

**Exact code path — `EntityServiceImpl.getByEvmAddressAndType()`:** [1](#0-0) 

When bytes 0–3 and 4–11 of the decoded EVM address are all zero, the method calls `entityRepository.findById(buffer.getLong())`. The only post-retrieval guard is `.filter(e -> e.getType() == type)`, which enforces entity type but not deletion status.

**Root cause — asymmetric deleted filtering in `EntityRepository`:**

The graphql-module `EntityRepository` defines explicit `deleted is not true` guards on both custom queries: [2](#0-1) 

`findById` is inherited from `CrudRepository<Entity, Long>` and issues a plain `SELECT * FROM entity WHERE id = ?` with no deleted predicate. The `findByAlias` and `findByEvmAddress` branches are safe; the `findById` branch is not.

**Note:** The same missing deleted filter also affects `getByIdAndType()` at line 25, which also calls `entityRepository.findById()` directly: [3](#0-2) 

**`pendingReward` and other sensitive fields are exposed:**

`pendingReward` is a first-class field on the `Account` GraphQL type: [4](#0-3) 

It is also declared on the `Accountable` interface: [5](#0-4) 

The `AccountMapper` maps the `Entity` domain object directly to the `Account` view model, meaning all fields present on the returned `Entity` row — including `pendingReward`, `balance`, `key`, `memo`, and `deleted: true` — are resolved and returned to the caller: [6](#0-5) 

## Impact Explanation

An unauthenticated attacker can read the full `Entity` record of any deleted account, including `pendingReward`, `balance`, `key`, `memo`, and `deleted: true`. The design intent is clear — `findByAlias` and `findByEvmAddress` both apply `deleted is not true` at the SQL level — but the `findById` branch bypasses this intent. This is an **information disclosure** vulnerability. No funds can be moved or redirected through this endpoint; the impact is data leakage of financial state that the protocol intends to hide after account deletion. Severity is **Medium**: no funds are at risk, but sensitive financial data of deleted accounts is exposed to any caller with network access to the GraphQL endpoint.

## Likelihood Explanation

Exploitation requires no authentication, no special privileges, and no on-chain interaction. The attacker only needs to know a deleted account's numeric entity ID, which is public information derivable from the mirror node's own REST API or block explorers. The crafted EVM address format (12 zero bytes followed by the 8-byte big-endian entity ID) is trivially constructable. The attack is fully repeatable and scriptable.

## Recommendation

Add a deleted-entity filter in `EntityServiceImpl` after `findById` returns, consistent with the SQL-level guards already applied by `findByAlias` and `findByEvmAddress`:

```java
// In getByEvmAddressAndType(), replace:
return entityRepository.findById(buffer.getLong())
    .filter(e -> e.getType() == type);

// With:
return entityRepository.findById(buffer.getLong())
    .filter(e -> e.getType() == type)
    .filter(e -> !Boolean.TRUE.equals(e.getDeleted()));
```

Apply the same fix to `getByIdAndType()` at line 25, which has the identical missing guard. Alternatively, add a `findByIdAndDeletedIsNotTrue(Long id)` derived query method to the graphql-module `EntityRepository` (mirroring the pattern used in the `web3` module's `EntityRepository.findByIdAndDeletedIsFalse`) and use it in both service methods.

## Proof of Concept

1. Identify a deleted account's numeric entity ID (e.g., `12345`).
2. Construct a 20-byte hex EVM address: `0x000000000000000000000000000000000000003039` (bytes 0–11 = zero, bytes 12–19 = big-endian `12345`).
3. Send the following GraphQL query to `/graphql/alpha`:

```graphql
{
  account(input: { evmAddress: "0x000000000000000000000000000000000000003039" }) {
    pendingReward
    balance
    key
    memo
    deleted
  }
}
```

4. `buffer.getInt()` reads bytes 0–3 → `0`; `buffer.getLong()` reads bytes 4–11 → `0`; condition at line 37 is true.
5. `entityRepository.findById(12345)` returns the deleted `Entity` row with no deleted filter applied.
6. The `.filter(e -> e.getType() == type)` check passes (type is still `ACCOUNT`).
7. The GraphQL framework resolves and returns `pendingReward`, `balance`, `key`, `memo`, and `deleted: true` for the deleted account. [7](#0-6)

    @Override
    public Optional<Entity> getByEvmAddressAndType(String evmAddress, EntityType type) {
        byte[] evmAddressBytes = decodeEvmAddress(evmAddress);
        var buffer = ByteBuffer.wrap(evmAddressBytes);
        if (buffer.getInt() == 0 && buffer.getLong() == 0) {
            return entityRepository.findById(buffer.getLong()).filter(e -> e.getType() == type);
        }
        return entityRepository.findByEvmAddress(evmAddressBytes).filter(e -> e.getType() == type);
    }
