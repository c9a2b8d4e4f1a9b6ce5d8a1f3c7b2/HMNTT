# Audit Report

## Title
Cache TTL Mismatch Enables Sustained DB Amplification via `findStorageBatch()` Slot Key Accumulation

## Summary
A mismatch between the 2-second write-expiry of `contractStateCache` and the 5-minute access-expiry of `slotsPerContract` allows an unprivileged attacker to accumulate up to 1,500 slot keys per contract in the key cache, then sustain a 1,500-slot `IN` query against the database every 2 seconds indefinitely, achieving up to a 1,500x DB query amplification factor per attacker request.

## Finding Description

**File**: `web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`

**Cache configuration mismatch (root cause)**:

| Cache | Config | Expiry type |
|---|---|---|
| `contractState` | `expireAfterWrite=2s, maximumSize=25000` | Write-based |
| `slotsPerContract` | `expireAfterAccess=5m, maximumSize=1500` | Access-based |
| `contractSlots` | `expireAfterAccess=5m, maximumSize=3000` | Access-based |

In `findStorage()` (line 63), a cache miss on `contractStateCache` unconditionally calls `findStorageBatch()` (line 69). Inside `findStorageBatch()`:

1. Line 90: `contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE)` — accumulates the requested slot key into the per-contract `slotsPerContract` cache. In Caffeine `expireAfterAccess` mode, any access (read or write) resets the TTL, so `putIfAbsent` keeps all previously accumulated keys alive.
2. Line 91: `contractSlotsCache.getNativeCache().asMap().keySet()` — retrieves **all** accumulated slot keys for the contract, with no size cap applied before the DB call.
3. Line 103: `contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots)` — issues a single `SELECT ... WHERE slot IN (...)` query with all accumulated keys.

**Exploit flow**:

1. **Setup**: Attacker sends 1,500 requests to `/api/v1/contracts/call`, each accessing a distinct slot key of the same contract. Each request misses `contractStateCache` (empty) and calls `findStorageBatch()`, which accumulates all 1,500 slot keys into `slotsPerContract` via `putIfAbsent`.

2. **Trigger**: After 2 seconds, all slot values expire from `contractStateCache`. A single attacker request for any slot of that contract misses the cache, calls `findStorageBatch()`, which reads all 1,500 still-cached keys from `slotsPerContract` and issues a 1,500-slot `IN` query to the DB.

3. **TTL maintenance**: The attacker cycles through all 1,500 slot keys at ~5 req/s (1,500 keys / 300 seconds), resetting their `expireAfterAccess` TTL via `putIfAbsent` at line 90. This is well within the 500 req/s rate limit.

4. **Sustained state**: Every 2 seconds, `contractState` expires, and the next request triggers a fresh 1,500-slot batch query. The attacker sustains this indefinitely.

**Why throttle checks fail**: `ThrottleManagerImpl.throttle()` enforces only `requestsPerSecond=500` and a gas-per-second bucket. Neither is proportional to the number of DB slots queried per request. There is no per-contract rate limit, no cap on the number of slot keys passed to `findStorageBatch()`, and no check on the accumulated size of `slotsPerContract` before issuing the DB query.

## Impact Explanation
Each attacker request causes a DB `IN` query over up to 1,500 slots — a 1,500x amplification factor relative to a single-slot query. The `contract_state` table is repeatedly queried with large `IN` clauses, degrading query planner performance, consuming DB connection pool resources, and increasing I/O load. Even at a fraction of the rate limit (5 req/s to maintain TTLs plus periodic bursts), the sustained batch queries represent a disproportionate DB load increase. At the full 500 req/s rate limit, the attacker can force up to 750,000 DB slot lookups per second from a single client.

## Likelihood Explanation
No privileges are required — any caller of the public `/api/v1/contracts/call` endpoint can trigger this. The setup phase (1,500 requests) is a one-time cost. The sustain phase requires only ~5 req/s, which is invisible against the 500 req/s rate limit. The attack is fully automatable with a simple script using any valid contract address and 1,500 distinct storage slot indices (e.g., sequential integers starting from 0).

## Recommendation

1. **Cap batch size before the DB call**: In `findStorageBatch()`, limit the number of slot keys passed to `contractStateRepository.findStorageBatch()` to a configurable maximum (e.g., 100–200), regardless of how many keys are in `slotsPerContract`.

2. **Align TTLs**: Change `slotsPerContract` from `expireAfterAccess=5m` to `expireAfterWrite=2s` (matching `contractState`), so slot keys are evicted at the same rate as their values. This eliminates the window during which stale keys accumulate.

3. **Bound key accumulation per contract**: Before calling `putIfAbsent` at line 90, check the current size of the per-contract cache and skip accumulation if it exceeds a threshold, preventing unbounded growth.

4. **Gas-proportional throttling**: Consider making the gas throttle account for the number of storage slots accessed per request, so large batch queries consume proportionally more gas tokens.

    private Optional<byte[]> findStorageBatch(final EntityId contractId, final byte[] key) {
        final var contractSlotsCache = ((CaffeineCache) this.contractSlotsCache.get(
                contractId, () -> cacheManagerSlotsPerContract.getCache(contractId.toString())));
        final var wrappedKey = ByteBuffer.wrap(key);
        // Cached slot keys for contract, whose slot values are not present in the contractStateCache
        contractSlotsCache.putIfAbsent(wrappedKey, EMPTY_VALUE);
        final var cachedSlotKeys = contractSlotsCache.getNativeCache().asMap().keySet();

        final var cachedSlots = new ArrayList<byte[]>(cachedSlotKeys.size());
        boolean isKeyEvictedFromCache = true;

        for (var slot : cachedSlotKeys) {
            cachedSlots.add(((ByteBuffer) slot).array());
            if (wrappedKey.equals(slot)) {
                isKeyEvictedFromCache = false;
            }
        }

        final var contractSlotValues = contractStateRepository.findStorageBatch(contractId.getId(), cachedSlots);
        byte[] cachedValue = null;

        for (final var contractSlotValue : contractSlotValues) {
            final byte[] slotKey = contractSlotValue.getSlot();
            final byte[] slotValue = contractSlotValue.getValue();
            contractStateCache.put(generateCacheKey(contractId, slotKey), slotValue);

            if (Arrays.equals(slotKey, key)) {
                cachedValue = slotValue;
            }
        }

        // If the cache key was evicted and hasn't been requested since, the cached value will be null.
        // In that case, fall back to the original query.
        if (isKeyEvictedFromCache) {
            return contractStateRepository.findStorage(contractId.getId(), key);
        }
        return Optional.ofNullable(cachedValue);
    }
