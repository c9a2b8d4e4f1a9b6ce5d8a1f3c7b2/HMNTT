Audit Report

## Title
Ghost NFT 206 Response via Token Deletion Timestamp Leakage in `getNftTransferHistoryRequest()`

## Summary
The timestamp-gathering query in `extractSqlFromNftTransferHistoryRequest()` unions the `entity` table for token-deletion events using only `id = $1 AND deleted is true`, with no `serial_number` filter. For any deleted token, this branch returns a row for **any** requested serial number ŌĆö including ones never minted. The subsequent `nftTransactionHistoryDetailsQuery` then synthesizes a `TOKENDELETION` nft_transfer entry stamped with the attacker-supplied serial, and the `NftService.getNft` null-check triggers HTTP 206, falsely implying the NFT existed with incomplete history.

## Finding Description

**Step 1 ŌĆö Timestamp query lacks serial_number guard for entity table**

In `extractSqlFromNftTransferHistoryRequest()`, the `tokenDeleteConditions` array is built with only two predicates:

```js
const tokenDeleteConditions = [`${Entity.ID} = $1`, `${Entity.DELETED} is true`];
``` [1](#0-0) 

The `nftConditions` array correctly includes `serial_number = $2`, but `tokenDeleteConditions` does not. The resulting UNION branch against the `entity` table is:

```sql
select lower(timestamp_range) as timestamp
from entity
where id = $1 AND deleted is true
``` [2](#0-1) 

For any deleted token (e.g., `0.0.1500`), this returns the deletion timestamp regardless of what serial number was requested.

**Step 2 ŌĆö Early-exit check is bypassed**

```js
const {rows} = await pool.queryQuietly(timestampQuery, timestampParams);
if (rows.length === 0) {
  return;   // ŌåÉ never reached; entity row is always present for a deleted token
}
``` [3](#0-2) 

Because the entity branch always returns a row for a deleted token, `rows.length` is at least 1 and the early return is never taken.

**Step 3 ŌĆö TOKENDELETION synthesizes a fake nft_transfer for the requested serial**

`nftTransactionHistoryDetailsQuery` special-cases transaction type 35:

```sql
when type = 35 then
    jsonb_build_array(jsonb_build_object(
        'is_approval', false,
        'receiver_account_id', null,
        'sender_account_id', null,
        'serial_number', $2::bigint,   ŌåÉ attacker-controlled serial
        'token_id', $1::bigint))
``` [4](#0-3) 

The serial number injected here is whatever the caller supplied ŌĆö it is never validated against the `nft` or `nft_history` tables.

**Step 4 ŌĆö 206 is set because `NftService.getNft` returns null**

```js
const nft = await NftService.getNft(tokenId, serialNumber);
if (nft?.createdTimestamp == null) {
  res.locals.statusCode = httpStatusCodes.PARTIAL_CONTENT.code;
}
``` [5](#0-4) 

`NftService.getNft` executes:

```js
static nftByIdQuery = `select * from nft where token_id = $1 and serial_number = $2`;
async getNft(tokenId, serialNumber) {
  const {rows} = await pool.queryQuietly(NftService.nftByIdQuery, [tokenId, serialNumber]);
  return isEmpty(rows) ? null : new Nft(rows[0]);
}
``` [6](#0-5) 

For a never-minted serial, `rows` is empty, so `null` is returned. In JavaScript, `null?.createdTimestamp` evaluates to `undefined`, and `undefined == null` is `true`, so the 206 branch is taken unconditionally. The code cannot distinguish between "NFT exists but has partial data" and "NFT never existed."

The test fixture `partial-data-missing-mint.json` confirms the 206 path is exercised and expected when `created_timestamp` is null: [7](#0-6) 

## Impact Explanation
An unprivileged external user can query `GET /api/v1/tokens/{deletedTokenId}/nfts/{anySerial}/transactions` for any serial number (e.g., `999999`) that was never minted on a deleted token. The response is HTTP 206 with a synthesized `TOKENDELETION` transaction entry that includes `serial_number: 999999` and `token_id: <deleted token>`. Integrators and downstream systems that treat 206 as "partial history of a real NFT" will incorrectly conclude the NFT with serial 999999 existed and was deleted as part of a token deletion event. This can be exploited to fabricate apparent on-chain history for NFT serials that never existed, enabling fraud in systems that rely on the mirror node API for NFT provenance verification (e.g., marketplaces, custody systems, compliance tools).

## Likelihood Explanation
- **Precondition**: Any deleted NFT token on the network. Token deletion is a standard operation.
- **Attacker capability**: Zero privileges required ŌĆö this is a public read-only REST endpoint.
- **Repeatability**: Fully repeatable for any deleted token ID and any arbitrary serial number.
- **Detection difficulty**: The 206 status code is the intended signal for partial data, so monitoring systems are unlikely to flag it as anomalous.

## Recommendation
Add a `serial_number` existence check before returning any result. The simplest fix is to add a guard in `getNftTransferHistoryRequest()`: after the timestamp query returns rows, verify that at least one row came from the `nft` or `nft_history` tables (i.e., the serial actually existed). If the only matching timestamp is from the `entity` table and no `nft`/`nft_history` rows exist for that serial, return an empty 200 response rather than synthesizing a TOKENDELETION entry.

Alternatively, add `serial_number = $2` to `tokenDeleteConditions` in `extractSqlFromNftTransferHistoryRequest()` ŌĆö but note this would suppress the TOKENDELETION entry even for serials that genuinely existed, so the guard approach is preferable.

## Proof of Concept
1. Identify any deleted NFT token on the network, e.g., token `0.0.1500` (deleted, `entity.deleted = true`).
2. Choose an arbitrary serial number that was never minted, e.g., `999999`.
3. Send: `GET /api/v1/tokens/0.0.1500/nfts/999999/transactions`
4. The timestamp query returns the deletion timestamp from the `entity` table (no serial filter).
5. The details query synthesizes a `TOKENDELETION` entry with `serial_number: 999999`.
6. `NftService.getNft(1500, 999999)` returns `null` ŌåÆ `null?.createdTimestamp == null` is `true`.
7. Response: HTTP **206** with body:
```json
{
  "transactions": [{
    "type": "TOKENDELETION",
    "serial_number": 999999,
    "token_id": "0.0.1500",
    "receiver_account_id": null,
    "sender_account_id": null,
    "is_approval": false
  }],
  "links": { "next": null }
}
```
This falsely implies NFT serial 999999 of token 0.0.1500 existed and was destroyed in the token deletion event.
