Audit Report

## Title
SharedPollingTopicListener Timestamp Reset on Re-subscription Causes Permanent Message Loss After Network Partition

## Summary
`SharedPollingTopicListener` uses a single shared `PollingContext` whose `lastConsensusTimestamp` is unconditionally reset to wall-clock `now` every time the shared upstream Flux is re-subscribed. Because `.share()` cancels the upstream when all subscribers leave, any reconnection after a complete subscriber drain permanently advances the polling cursor past all messages written during the outage window.

## Finding Description

Every claim in the report is confirmed by the actual source.

**Single shared context and `.share()`:** [1](#0-0) 

`PollingContext` is constructed once at line 39 and captured in the constructor closure. `.share()` at line 52 is `publish().refCount()`: when the subscriber count drops to 0 the upstream is cancelled; the next subscriber triggers a brand-new upstream subscription.

**`doOnSubscribe` fires `onStart` on every upstream re-subscription:** [2](#0-1) 

**`onStart` unconditionally resets the cursor to wall-clock now:** [3](#0-2) 

**`poll()` uses that cursor as the exclusive lower bound:** [4](#0-3) 

**`findLatest` enforces a strict `>` predicate — rows at or before the reset timestamp are never fetched:** [5](#0-4) 

**`filterMessage` in `CompositeTopicListener` cannot compensate** because it only filters messages already emitted by the shared Flux; messages from the partition window are never queried from the database at all: [6](#0-5) 

**Root cause:** The design assumes the upstream is never cancelled while messages of interest are being produced. `.share()` violates this assumption the moment all subscribers leave. The single `PollingContext` is never re-created, so the reset permanently advances the cursor past the partition window with no catch-up path.

## Impact Explanation
Any deployment configured with `type: SHARED_POLL` permanently loses every `TopicMessage` whose `consensus_timestamp` falls between the last successful poll before the drain and the reconnection instant. The loss is silent — no error is raised, sequence numbers simply have gaps, and downstream consumers have no signal that data was skipped.

## Likelihood Explanation
A complete subscriber drain is a routine operational event: rolling pod restarts, load-balancer timeout flushes, and Kubernetes node evictions all simultaneously drop every active gRPC stream. No attacker capability is required for accidental triggering. An adversary who can cause even a brief network interruption (TCP RST injection, short-lived port-level disruption) can trigger the reset on demand and reliably cause targeted message loss. The default listener type is `REDIS`, which limits exposure to deployments that have explicitly set `type: SHARED_POLL`.

## Recommendation
1. **Do not reset `lastConsensusTimestamp` in `onStart`.** The cursor should only advance via `onNext`. `onStart` should be a no-op with respect to the timestamp, or should initialize it only when it has never been set (e.g., `compareAndSet(0, DomainUtils.now())`).
2. **Alternatively, re-create `PollingContext` per upstream subscription** by moving its construction inside the `Flux.defer(...)` lambda, so each upstream subscription starts with a fresh context that is initialized from the filter's `startTime` rather than wall-clock now.
3. **Add a catch-up query on reconnection** that back-fills from the last known good timestamp rather than from now.

## Proof of Concept
1. Start the mirror node with `hedera.mirror.grpc.listener.type=SHARED_POLL`.
2. Connect one or more gRPC subscribers to a topic stream.
3. Disconnect all subscribers simultaneously (simulating a network partition or pod restart).
4. While no subscribers are connected, allow topic messages to be written to the database.
5. Reconnect a subscriber.
6. Observe that the subscriber receives no messages from the partition window — `lastConsensusTimestamp` was reset to the reconnection instant by `onStart` at line 102, and `findLatest` at line 16 of `TopicMessageRepository.java` uses `consensus_timestamp > <reconnection_time>`, permanently skipping all rows written during the outage.
