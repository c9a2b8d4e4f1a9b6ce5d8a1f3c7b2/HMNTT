# Audit Report

## Title
Unconditional Counter Increment in `RequestProperties.test()` Enables Filter Exhaustion by Unprivileged Users

## Summary
`RequestProperties.test()` increments its `AtomicLong counter` on every invocation, regardless of whether the incoming request matches any configured `RequestFilter`. When an operator sets a finite `limit`, an unauthenticated caller can exhaust the counter by sending traffic that does not match the filter criteria. Once `counter >= limit`, `test()` permanently returns `false`, disabling the filter for all subsequent requests until the process is restarted.

## Finding Description

**File:** `web3/src/main/java/org/hiero/mirror/web3/throttle/RequestProperties.java`
**Method:** `test(ContractCallRequest)`
**Line 46:**
```java
if (rate == 0 || counter.getAndIncrement() >= limit) {
    return false;
}
```

`counter.getAndIncrement()` executes before:
- The random-rate sampling check (line 50)
- The `RequestFilter` loop (lines 54–58)

A request that matches zero filters still increments the counter and returns `false` at line 58. The counter is never decremented and never resets at runtime.

**Call path:**
`ThrottleManagerImpl.throttle()` (lines 44–48) iterates over all `RequestProperties` entries and calls `requestFilter.test(request)` for each one on every request that passes the global `rateLimitBucket` and `gasLimitBucket` checks. Every such request increments the counter of every configured filter, whether or not the request matches that filter's criteria.

**Failed assumption:**
The design intent is that `limit` bounds "how many times this filter fires (matches)." The implementation counts "how many times `test()` is called," which includes all non-matching traffic.

## Impact Explanation

| Action | Effect after exhaustion |
|--------|------------------------|
| `REJECT` | Requests matching the filter are no longer rejected — security control bypassed |
| `THROTTLE` | Per-filter rate-limiting bucket is never consulted — rate limiting bypassed |
| `LOG` | Audit/logging coverage silently dropped |

The `REJECT` case is the most severe. `action()` is only invoked when `test()` returns `true`; once the counter is exhausted, `test()` always returns `false`, so `action()` is never reached and the `ThrottleException` is never thrown for matching requests.

## Likelihood Explanation

- **No privileges required.** Any caller that can reach the web3 JSON-RPC endpoint can trigger this.
- **Rate-limited but not prevented.** The global `rateLimitBucket` defaults to 500 req/s (`ThrottleProperties.requestsPerSecond`, line 35). A `limit` of 50,000 is exhausted in ~100 seconds at the default rate.
- **Permanent until restart.** The counter never resets at runtime; once exhausted the filter stays disabled.
- **Requires non-default config.** The default `limit` is `Long.MAX_VALUE` (line 35 of `RequestProperties.java`), which is unreachable in practice. The vulnerability is only exploitable when an operator explicitly sets a finite `limit` — a documented, valid configuration option.

## Recommendation

Move `counter.getAndIncrement()` to after the filter-matching logic so it only increments when the filter actually matches (i.e., when `test()` would return `true`). Concretely, restructure `test()` so that the counter is incremented only on the path that returns `true`:

```java
@Override
public boolean test(ContractCallRequest contractCallRequest) {
    if (rate == 0) {
        return false;
    }
    if (action != ActionType.THROTTLE && RandomUtils.secure().randomLong(0L, 100L) >= rate) {
        return false;
    }
    boolean matched = filters.isEmpty() || filters.stream().anyMatch(f -> f.test(contractCallRequest));
    if (!matched) {
        return false;
    }
    // Only increment when the filter actually matches
    return counter.getAndIncrement() < limit;
}
```

This ensures `limit` correctly bounds the number of times the filter fires, not the number of times it is evaluated.

## Proof of Concept

**Setup:** Configure one `RequestProperties` entry with `action = REJECT`, `limit = 1000`, and a `RequestFilter` that matches requests whose `data` field contains the selector `0xdeadbeef`.

**Step 1 — Exhaust the counter (non-matching traffic):**
Send 1,001 requests with `data` set to any value that does NOT contain `0xdeadbeef`. Each call to `test()` increments the counter. After 1,001 requests, `counter (1001) >= limit (1000)` is true.

**Step 2 — Bypass the REJECT filter:**
Send a request with `data` containing `0xdeadbeef`. Under normal operation this would be rejected. After exhaustion, `test()` returns `false` at line 46 before reaching the filter loop, so `action()` is never called and the request is processed normally.

**Constraint:** At the default global rate limit of 500 req/s, step 1 completes in ~2 seconds for `limit = 1000`. The attacker requires no credentials and no special network position.
