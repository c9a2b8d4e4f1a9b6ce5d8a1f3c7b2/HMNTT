# Audit Report

## Title
Count-Based `maxBlockItems` Guard Allows Heap Exhaustion Before DoS Protection Activates in `BlockAssembler.append()`

## Summary
`BlockAssembler` (inner class of `BlockNode`) accumulates deserialized `BlockItem` objects into the `pending` list and only checks item **count** against `maxBlockItems` (default 800,000) after items are already added to the heap. Because there is no cumulative byte-size guard on the `pending` list, a malicious block node can stream many maximally-sized messages and exhaust JVM heap long before the integer count threshold fires.

## Finding Description

**Confirmed code path:**

`StreamProperties.java` defines both limits: [1](#0-0) 

`BlockNode` constructor applies `maxStreamResponseSize` as a per-message gRPC cap only: [2](#0-1) 

`BlockAssembler.append()` (inner class of `BlockNode`) adds items to `pending` and increments `pendingCount` **before** the count check: [3](#0-2) 

There is no byte-size accumulator anywhere in `BlockAssembler`. The `pending` field is an unbounded `List<List<BlockItem>>` that grows across successive `BlockItemSet` messages until `pendingCount` exceeds 800,000: [4](#0-3) 

**Root cause:** The design assumes item count is a proxy for memory consumption. This is false: each `BlockItem` can carry arbitrary-length protobuf byte/string fields up to the 36 MB per-message ceiling. With one large item per message, 800,000 messages would be needed to trigger the count guard, but OOM occurs far earlier.

**Why existing checks are insufficient:**
- `maxStreamResponseSize` (36 MB): limits each individual gRPC frame, not cumulative in-memory accumulation.
- `maxBlockItems` (800,000): a pure integer counter with no relationship to byte size.
- `responseTimeout` (400 ms): a per-message read deadline; it does not bound total accumulation across messages.

## Impact Explanation
A malicious block node can crash the importer process with `OutOfMemoryError`. This halts all block ingestion, causing the mirror node to fall behind the chain and making all downstream APIs (REST, gRPC) serve stale data. The attack is repeatable: after restart the importer reconnects to the same registered block node and the crash can be triggered again immediately. After `maxSubscribeAttempts` (default 3) errors the node is marked inactive with a `readmitDelay` (default 1 minute), but an OOM crash terminates the process before this circuit-breaker can engage. [5](#0-4) 

## Likelihood Explanation
`autoDiscoveryEnabled` defaults to `true`: [6](#0-5) 

`BlockNodeDiscoveryService.discover()` reads from `registeredNodeRepository` and adds any registered tier-1 block node endpoint to the active node list without additional credential or administrator approval: [7](#0-6) 

Any Hedera account holder who can pay fees for a `RegisteredNodeCreate` transaction can register a malicious endpoint. The attack requires only a TCP server speaking the block-node gRPC protocol that sends maximally-sized items.

## Recommendation
Add a cumulative byte-size accumulator alongside `pendingCount` in `BlockAssembler`. Introduce a `maxBlockBytes` property in `StreamProperties` (e.g., defaulting to a multiple of `maxStreamResponseSize`) and throw a `BlockStreamException` in `append()` when the running byte total exceeds it. The byte size of a `BlockItemSet` can be obtained via `blockItemSet.getSerializedSize()` before deserialization or by summing `blockItem.getSerializedSize()` for each item. This provides a size-proportional guard that fires well before heap exhaustion regardless of item count.

## Proof of Concept
1. Register a block node endpoint via an on-chain `RegisteredNodeCreate` transaction advertising `STATUS`, `PUBLISH`, and `SUBSCRIBE_STREAM` APIs.
2. Start a gRPC server at that endpoint implementing `BlockStreamSubscribeService`.
3. On `subscribeBlockStream`, respond with one `BlockItemSet` containing a valid `BLOCK_HEADER` item, then stream a continuous sequence of `BlockItemSet` messages each containing exactly one `BlockItem` whose bytes field is padded to ~36 MB.
4. Each message passes the `maxInboundMessageSize` check. Each call to `append()` adds the list to `pending` and increments `pendingCount` by 1.
5. With a 4 GB JVM heap, `OutOfMemoryError` occurs after approximately 111 messages (4 GB / 36 MB). The `maxBlockItems` count check would not fire until message 800,001 â€” approximately 7,200Ă— too late.
