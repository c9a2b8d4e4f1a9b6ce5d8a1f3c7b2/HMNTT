Audit Report

## Title
Stale Entity Lookup in `getReceiverAddress()` Produces Divergent Opcode Trace for Deleted Long-Zero Targets

## Summary
`OpcodeServiceImpl.getReceiverAddress()` resolves a long-zero `toAddress` to its EVM alias using a **current-state** query (`Optional.empty()`) instead of the historical query that uses `consensusTimestamp`. For any entity deleted after the original transaction, the lookup returns empty, the receiver falls back to the raw long-zero address, and the EVM replay targets a different (codeless) address — producing a trace that diverges from the actual historical execution.

## Finding Description

**Exact location:** `web3/src/main/java/org/hiero/mirror/web3/service/OpcodeServiceImpl.java`, `getReceiverAddress()`, line 191.

The defective call: [1](#0-0) 

`commonEntityAccessor.get(address, Optional.empty())` dispatches to `getEntityByMirrorAddressAndTimestamp(address, Optional.empty())`: [2](#0-1) 

With `Optional.empty()`, the `orElseGet` branch executes `entityRepository.findByIdAndDeletedIsFalse(entityId)` — a live-state query that returns `Optional.empty()` for any entity whose `deleted` flag is `true`: [3](#0-2) 

When the entity is absent, `entity` is `null`, the guard is skipped, and the method falls through to `return address` — returning the raw long-zero address instead of the entity's EVM alias: [4](#0-3) 

`consensusTimestamp` is in scope in the calling method `buildCallServiceParameters` and is correctly set on the `ContractDebugParameters` builder, but is **never forwarded** to `getReceiverAddress`: [5](#0-4) 

The historical query `findActiveByIdAndTimestamp` correctly retrieves entities from `entity_history` that existed at `consensusTimestamp` even if they are now deleted: [6](#0-5) 

Note: the same `Optional.empty()` pattern also affects `getSenderAddress` (line 178) and the non-Ethereum path at line 203, but the receiver path is the most impactful because it determines the EVM call target. [7](#0-6) 

## Impact Explanation
The opcodes API (`GET /api/v1/contracts/results/{txHash}/opcodes`) is intended to faithfully replay historical EVM execution. When the receiver is resolved to the wrong address, the EVM replay targets an address with no code, producing a failed or empty trace instead of the original successful one. Downstream consumers — auditors, block explorers, compliance tools, dispute-resolution systems — that rely on this endpoint to reconstruct what a transaction did will receive a false account. The attacker controls which traces are corrupted and when.

## Likelihood Explanation
The precondition (entity deletion) is fully within the attacker's control: any account or contract owner can delete their own entity on Hedera using the admin key they hold. No privileged access, governance approval, or third-party action is required. The attack is repeatable for any transaction whose `toAddress` is a long-zero address pointing to a now-deleted entity. The opcodes endpoint is publicly accessible.

## Recommendation
Pass `consensusTimestamp` into `getReceiverAddress` and replace `Optional.empty()` with `Optional.of(consensusTimestamp)` at line 191 (and symmetrically at lines 178 and 203). This causes `getEntityByMirrorAddressAndTimestamp` to take the `timestamp.map(...)` branch and call `findActiveByIdAndTimestamp`, which correctly retrieves the entity's historical state from `entity_history` even if the entity is now deleted.

```java
// Before (line 191)
commonEntityAccessor.get(address, Optional.empty())

// After
commonEntityAccessor.get(address, Optional.of(consensusTimestamp))
```

`getReceiverAddress` must also accept `consensusTimestamp` as a parameter, and `buildCallServiceParameters` must forward it at the call site (line 171).

## Proof of Concept
1. Deploy a contract. It receives entity ID `0.0.N` (long-zero address `0x0000…000N`) and EVM alias `0xALIAS`.
2. Send an `EthereumTransaction` with `toAddress = 0x0000…000N`. The execution resolves the entity, uses `0xALIAS` as the canonical receiver, and executes the contract code — producing trace T₁.
3. Delete the contract using the admin key.
4. Call `GET /api/v1/contracts/results/{txHash}/opcodes`.
5. `getReceiverAddress` calls `findByIdAndDeletedIsFalse` → returns empty → falls through → receiver is set to `0x0000…000N`.
6. The EVM replay targets `0x0000…000N`. Contract code is stored under `0xALIAS`; the long-zero address has no code. The replay produces trace T₂ (empty/failed execution).
7. T₁ ≠ T₂: the API returns a fabricated trace contradicting the actual historical execution.
