Audit Report

## Title
Cache Collision via 32-bit `Arrays.hashCode()` Key in `CACHE_NAME_EVM_ADDRESS` Allows Wrong Entity Resolution for EVM Address Lookups

## Summary
`findByEvmAddressAndDeletedIsFalse()` uses `@spelHelper.hashCode(#alias)` — delegating to `Arrays.hashCode(byte[])`, a 32-bit polynomial hash — as the sole Caffeine cache key. Two distinct 20-byte EVM addresses that produce the same 32-bit hash map to the same cache slot. A lookup for address B will return the cached entity of address A with no post-hit equality check.

## Finding Description
In `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java` lines 32–37:

```java
@Cacheable(
        cacheNames = CACHE_NAME_EVM_ADDRESS,
        cacheManager = CACHE_MANAGER_ENTITY,
        key = "@spelHelper.hashCode(#alias)",
        unless = "#result == null")
Optional<Entity> findByEvmAddressAndDeletedIsFalse(byte[] alias);
``` [1](#0-0) 

The key expression calls `SpelHelper.hashCode(byte[])`:

```java
public int hashCode(byte[] value) {
    return Arrays.hashCode(value);
}
``` [2](#0-1) 

`Arrays.hashCode(byte[])` produces a signed 32-bit `int` — only ~4.3 billion distinct values for a 20-byte (160-bit) input space. The `CaffeineCacheManager` for `CACHE_MANAGER_ENTITY` stores and retrieves entries using this integer as the exact key: [3](#0-2) 

No secondary equality check on the actual byte content is performed after a cache hit. The `unless = "#result == null"` guard only prevents caching null results; it does not prevent a non-null result for address A from being served for a colliding address B. The same flaw exists in `findByEvmAddressOrAliasAndDeletedIsFalse` (lines 39–49, `CACHE_NAME_ALIAS`). [4](#0-3) 

The entity cache TTL is `expireAfterWrite=1s`: [5](#0-4) 

This short TTL reduces the exploitation window but does not eliminate it — a 1-second window is sufficient for a targeted, timed attack.

## Impact Explanation
Any EVM address lookup through `findByEvmAddressAndDeletedIsFalse` can be poisoned to return a different entity. `CommonEntityAccessor.getEntityByEvmAddressAndTimestamp()` (lines 66–71) returns the wrong entity with no post-cache address verification: [6](#0-5) 

This affects: entity type resolution (account vs. contract), balance and nonce reads, contract code existence checks, and any downstream logic relying on the returned `Entity` object. An attacker can make the mirror node report that address T is a different entity, disrupting `eth_call` results, `eth_getBalance`, and contract interaction simulations.

## Likelihood Explanation
The attack requires no special privileges — only the ability to submit transactions (deploy a contract via CREATE2) and trigger RPC calls. Finding a 32-bit hash collision against a target address requires ~65,536 hash computations on average (birthday paradox), achievable in milliseconds on commodity hardware. The attacker controls the CREATE2 salt and bytecode, making the search space entirely under their control. The 1-second TTL requires timing the poisoning call and the victim's query within the same window, which is feasible for a targeted attack against a known RPC endpoint. The attack is repeatable after each TTL expiry.

## Recommendation
Replace the lossy 32-bit hash with the full byte array as the cache key. Spring Cache supports composite keys or string-encoded keys:

```java
key = "T(java.util.Arrays).toString(#alias)"
```

This ensures the Caffeine cache performs exact byte-array equality on the key, eliminating collision entirely. Alternatively, use `T(java.util.HexFormat).of().formatHex(#alias)` for a human-readable hex string key. Apply the same fix to `findByEvmAddressOrAliasAndDeletedIsFalse`.

## Proof of Concept
```java
// Find two 20-byte addresses with the same Arrays.hashCode
byte[] addrT = hexToBytes("target_address_here");
int targetHash = Arrays.hashCode(addrT);

// Brute-force a colliding address using CREATE2 salt iteration
for (int salt = 0; salt < Integer.MAX_VALUE; salt++) {
    byte[] addrA = computeCreate2Address(deployerAddress, salt, bytecodeHash);
    if (Arrays.hashCode(addrA) == targetHash) {
        // addrA collides with addrT in the cache
        // Deploy contract at addrA, then call findByEvmAddressAndDeletedIsFalse(addrA)
        // Cache is now poisoned: key=targetHash -> entity_A
        // Next call to findByEvmAddressAndDeletedIsFalse(addrT) returns entity_A
        break;
    }
}
```
