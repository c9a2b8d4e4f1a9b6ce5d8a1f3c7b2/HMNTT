Audit Report

## Title
Unbounded `findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc` Query Enables DB Resource Exhaustion via Opcode Endpoint

## Summary
`TransactionRepository.findByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc` issues an unbounded SQL query with no `LIMIT` clause, returning every transaction row sharing a given `(payerAccountId, validStartNs)` pair. `OpcodeServiceImpl.buildCallServiceParameters` fetches the full result set but discards all rows except `getFirst()`. The opcode endpoint's rate-limit bucket is a single global in-memory token (1 RPS, not per-IP, not distributed), so an attacker can monopolize that slot with maximally expensive queries, causing sustained unnecessary DB I/O and denying legitimate users access to the endpoint.

## Finding Description

**Unbounded query:**

`TransactionRepository` declares a Spring Data JPA derived query method that returns `List<Transaction>` with no row limit: [1](#0-0) 

Spring Data JPA derives `SELECT * FROM transaction WHERE payer_account_id = ? AND valid_start_ns = ? ORDER BY consensus_timestamp ASC` with **no LIMIT clause**.

**Only first row used:**

`OpcodeServiceImpl.buildCallServiceParameters` fetches the full list and immediately discards all rows beyond the first: [2](#0-1) 

**Child transactions confirmed to share the same key:**

In Hedera, every synthetic child transaction produced by a contract call shares the parent's `(payerAccountId, validStartNs)` with an incrementing `nonce`. This is confirmed by test data and domain model usage throughout the codebase: [3](#0-2) [4](#0-3) 

**Global in-memory rate limiter:**

`ThrottleConfiguration` creates the `opcodeRateLimitBucket` as a plain local `Bucket` with no distributed backend and no per-IP keying: [5](#0-4) 

`ThrottleManagerImpl.throttleOpcodeRequest()` simply tries to consume one token from this single shared bucket: [6](#0-5) 

The default rate is 1 RPS: [7](#0-6) 

**Endpoint is disabled by default** (reduces default attack surface but does not eliminate the vulnerability when enabled): [8](#0-7) 

**Why existing mitigations are insufficient:**

- `statementTimeout = 3000 ms` limits wall-clock query time, not row count. A query returning hundreds of rows easily completes within 3 seconds. [9](#0-8) 

- The 1 RPS global bucket limits request frequency but does not bound the number of rows fetched per request. In a multi-replica deployment, each pod has its own independent in-memory bucket, so the effective global rate is `replicas × 1 RPS`.

## Impact Explanation

Each attacker request causes the DB to read, serialize, and transmit an unbounded number of `Transaction` rows over the JDBC connection, consuming DB CPU, memory, and network bandwidth proportional to the child-transaction count. All rows beyond the first are deserialized into `Transaction` objects and immediately garbage-collected. At 1 RPS (or N RPS across replicas), this produces sustained amplified DB load. Simultaneously, legitimate users are denied access to the opcode endpoint because the global token bucket is consumed by the attacker's heavy requests.

## Likelihood Explanation

The endpoint must be explicitly enabled (`hiero.mirror.web3.opcode.tracer.enabled = true`), which reduces the default attack surface. However, when enabled (e.g., on a debug or testnet deployment), the preconditions are trivially met: parent transactions with many children are publicly visible on-chain, the transaction ID format is well-documented, and no authentication is required. The attack requires only an HTTP client and knowledge of one high-child-count transaction ID.

## Recommendation

1. **Add `LIMIT 1` to the query** — either via a `@Query` annotation or by changing the return type to `Optional<Transaction>` using a Spring Data method that fetches only the minimum required row (e.g., `findFirstByPayerAccountIdAndValidStartNsOrderByConsensusTimestampAsc`). Since only the first row (lowest `consensus_timestamp`) is ever used, there is no reason to fetch the full result set.

2. **Per-IP rate limiting** — replace or supplement the global in-memory bucket with per-IP throttling so a single client cannot monopolize the global slot.

3. **Distributed rate limiting** — in multi-replica deployments, use a distributed rate-limit backend (e.g., Redis-backed Bucket4j) so the 1 RPS limit is enforced globally across all pods rather than per-pod.

## Proof of Concept

1. Identify (from public chain data) a parent transaction whose contract execution produced a large number of child/synthetic transactions (e.g., a batch token-transfer contract). All children share the same `payerAccountId` and `validStartNs`.
2. Enable the opcode endpoint (`hiero.mirror.web3.opcode.tracer.enabled = true`).
3. Send `GET /api/v1/contracts/results/{payerAccountId}-{validStartSec}-{validStartNano}/opcodes` with `Accept-Encoding: gzip`.
4. Observe via DB query logs that the full set of N+1 matching rows is returned, while the application only uses row 0. Repeat at 1 RPS to sustain the amplified DB load and deny the endpoint to other users.
