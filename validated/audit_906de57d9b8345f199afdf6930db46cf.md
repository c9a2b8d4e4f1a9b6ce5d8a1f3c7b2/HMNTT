Audit Report

## Title
Silent Overwrite of Lower Bound Allows Duplicate GTE Parameters to Omit Pending Airdrop Records

## Summary
`TokenAirdropServiceImpl.getAirdrops()` never calls `verifySingleOccurrence()` on the constructed `Bound` objects. When a caller supplies two lower-bound operators for the same parameter (e.g., `sender.id=gte:0.0.1&sender.id=gte:0.0.1000`), the `Bound` constructor silently discards the first value and retains only the last, causing the query to execute with a narrower range than requested. No error is returned to the caller.

## Finding Description

**Root cause — `Bound` constructor unconditional overwrite:**

In `Bound.java` the constructor loop at lines 47–54 assigns `lower` on every iteration that satisfies `hasLowerBound()`, with no duplicate check:

```java
for (var param : params) {
    if (param.hasLowerBound()) {
        lower = param;          // unconditional overwrite
    } else if (param.hasUpperBound()) {
        upper = param;
    }
    cardinality.merge(param.operator(), 1, Math::addExact);
}
``` [1](#0-0) 

With `params = [GTE:1, GTE:1000]`, the loop sets `lower = GTE:1` then immediately overwrites it with `lower = GTE:1000`. The `cardinality` map correctly records `{GTE: 2}`, but `lower` retains only the last value.

**The inversion guard does not catch this:**

After the loop, the only constructor-level check is `adjustedLower > adjustedUpper`. When no upper bound is supplied, `adjustUpperBound()` returns `Long.MAX_VALUE`, so the check never fires regardless of the GTE values. [2](#0-1) [3](#0-2) 

**The guard that should prevent this — `verifySingleOccurrence()` — is never called for airdrop requests:**

`Bound.verifySingleOccurrence()` throws `IllegalArgumentException` when the cardinality of any operator group exceeds 1: [4](#0-3) [5](#0-4) 

`NftAllowanceServiceImpl` correctly invokes it via `verifyRangeId()`: [6](#0-5) 

`TokenAirdropServiceImpl.getAirdrops()` performs no such validation — it goes directly to the repository: [7](#0-6) 

**Entry point — `@Size(max = 2)` explicitly permits two parameters:** [8](#0-7) 

The `Bound` is then constructed with `primarySortField = true` and the two-element array, with no subsequent validation: [9](#0-8) 

## Impact Explanation
An unprivileged external caller can craft a request that silently narrows the effective lower bound of the sender (or receiver) filter. All pending airdrop records for senders between the discarded lower GTE value and the surviving higher GTE value are omitted from the API response with no error signal. Clients relying on this endpoint for auditing or reconciliation of pending airdrops will receive an incomplete, misleading result set. The severity is moderate — no data is written or deleted, but the mirror node's read-path correctness guarantee is violated for an unauthenticated caller.

## Likelihood Explanation
No authentication or privilege is required. The endpoint is public. The `@Size(max = 2)` annotation is documented behavior that explicitly invites two-parameter usage (it is the intended mechanism for supplying both a lower and upper bound). A misconfigured client or a deliberate actor can trigger this with a single crafted HTTP GET request, repeatably and at will.

## Recommendation
Add the same validation that `NftAllowanceServiceImpl` applies. In `TokenAirdropServiceImpl.getAirdrops()` (or in `processRequest` in the controller before constructing the `Bound`), call `verifySingleOccurrence()` on each `Bound` that is constructed from user-supplied parameters (`entityIds`, `tokenIds`, `serialNumbers`). This will cause the API to return a `400 Bad Request` when duplicate lower-bound or upper-bound operators are supplied, consistent with the behavior of the NFT allowance endpoint.

## Proof of Concept
```
GET /api/v1/accounts/0.0.5000/airdrops/pending
    ?sender.id=gte:0.0.1
    &sender.id=gte:0.0.1000
```

1. Spring binds both values into `EntityIdRangeParameter[] senderIds` (permitted by `@Size(max = 2)`).
2. `new Bound(senderIds, true, ACCOUNT_ID, ...)` iterates: sets `lower = GTE:1`, then overwrites `lower = GTE:1000`.
3. `cardinality = {GTE: 2}` but `verifySingleOccurrence()` is never called.
4. The inversion check passes because `adjustedUpper = Long.MAX_VALUE`.
5. The repository query executes with `sender_account_id >= 1000`, silently omitting all records for senders `0.0.1` through `0.0.999`.
6. The caller receives a `200 OK` with an incomplete result set and no indication that the lower bound was discarded.
