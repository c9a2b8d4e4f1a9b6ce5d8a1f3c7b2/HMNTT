# Audit Report

## Title
Thundering Herd / Cache Stampede on `getAccountTokenAllowances()` Enables Unauthenticated DB Amplification DoS

## Summary
The `/api/v1/accounts/{id}/allowances/tokens` endpoint has no request deduplication, no single-flight coalescing, and no rate limiting. When the Redis response cache is cold or expired, all concurrent identical requests simultaneously bypass the cache and each independently execute the same database query. This allows an unauthenticated attacker to amplify database load proportionally to the number of concurrent requests, exhausting the connection pool and degrading service for all users.

## Finding Description

**Route registration** is confirmed at the exact line cited: [1](#0-0) 

**Controller handler** `getAccountTokenAllowances` at lines 68–81 calls `TokenAllowanceService.getAccountTokenAllowances(query)` with no deduplication or in-service caching: [2](#0-1) 

**Service layer** at lines 86–90 unconditionally builds and executes a DB query on every invocation: [3](#0-2) 

**DB execution path** in `BaseService.getRows` goes directly to `pool.queryQuietly`: [4](#0-3) 

**The cache does not prevent the stampede.** `responseCacheCheckHandler` performs a simple Redis lookup and, on a miss, sets a local key and returns — no mutex, no lock, no promise coalescing: [5](#0-4) 

The update handler only stores the result *after* the query completes, so all N concurrent requests that arrived before the first response was stored all independently hit the database: [6](#0-5) 

**Default Redis TTL is 1 second**, meaning the cache window expires every second and the stampede recurs continuously: [7](#0-6) 

**When Redis is disabled**, `getSingleWithTtl` returns `undefined` immediately because `this.ready = false` (set via `lazyConnect: !redisConfig.enabled`), and the cache middleware is not even registered (`applicationCacheEnabled` requires both flags), so every single request hits the database with zero caching: [8](#0-7) [9](#0-8) [10](#0-9) 

**No rate limiting exists anywhere in the middleware stack.** A search of `rest/**/*.js` for `express-rate-limit`, `rateLimit`, `rate_limit`, and `throttle` returned zero matches. The full middleware chain registered in `server.js` is: `cors`, `compression`, `httpContext`, `requestLogger`, `authHandler`, `metricsHandler`, `responseCacheCheckHandler`, routes, `responseHandler`, `responseCacheUpdateHandler`, `handleError` — no throttle or rate-limit middleware at any position: [11](#0-10) 

**The DB connection pool is finite**, bounded by `config.db.pool.maxConnections`: [12](#0-11) 

## Impact Explanation

An attacker sending N concurrent identical GET requests to `/api/v1/accounts/0.0.X/allowances/tokens` causes N simultaneous `SELECT * FROM token_allowance WHERE owner = $1 AND amount > 0 ...` queries. Each query is non-trivial for accounts with large token allowance sets. At N = 1000 concurrent requests (trivially achievable with `ab`, `wrk`, or any async HTTP client), the database receives 1000 identical queries per 1-second cache-miss window. This exhausts the connection pool (`max: config.db.pool.maxConnections`), causes query queuing and timeouts (`connectionTimeoutMillis`), and degrades or denies service to all legitimate users of the mirror node REST API. No authentication or special privilege is required.

## Likelihood Explanation

The endpoint is fully public — no authentication check exists in the route, controller, or any middleware specific to this path. Any attacker with internet access and a basic HTTP flood tool can exploit this. The attack is repeatable indefinitely (every ≥1 second cache window), requires no prior knowledge beyond the public API spec, and is not mitigated by any existing control in the codebase.

## Recommendation

1. **Single-flight / promise coalescing**: Implement a request deduplication layer (e.g., using a `Map` of in-flight promises keyed by cache key) so that N concurrent identical requests share a single DB query and its result.
2. **Distributed lock on cache miss**: Use a Redis `SET NX PX` lock around the cache-miss path so only one request executes the DB query while others wait and then read from cache.
3. **Rate limiting**: Add `express-rate-limit` (or equivalent) middleware to the global middleware stack, with per-IP and optionally per-account-ID limits.
4. **Increase default TTL**: The 1-second `DEFAULT_REDIS_EXPIRY` is extremely short and maximizes stampede frequency. Increase it to a value appropriate for the data's freshness requirements (e.g., 5–30 seconds).
5. **Infrastructure-level protection**: Deploy a WAF or API gateway with rate limiting and request coalescing in front of the REST API.

## Proof of Concept

```bash
# Send 500 concurrent requests to the same endpoint
ab -n 500 -c 500 \
  "https://<mirror-node-host>/api/v1/accounts/0.0.1234/allowances/tokens"
```

Each request arrives within the same 1-second cache window. Because `responseCacheCheckHandler` performs no locking, all 500 requests find the cache empty simultaneously and each independently calls `TokenAllowanceService.getAccountTokenAllowances`, which calls `BaseService.getRows` → `pool.queryQuietly` → `pg.Pool.query`. The database receives 500 identical queries simultaneously. Repeating this every second sustains the load. With `maxConnections` exhausted, legitimate requests begin receiving connection timeout errors.

const responseCacheUpdateHandler = async (req, res) => {
  const responseCacheKey = res.locals[responseCacheKeyLabel];
  const responseBody = res.locals[responseBodyLabel];
  const isUnmodified = res.statusCode === httpStatusCodes.UNMODIFIED.code;

  if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
    const ttl = getCacheControlExpiryOrDefault(res.getHeader(CACHE_CONTROL_HEADER));
    if (ttl > 0) {
      // There's no content-type header when code is 304, so get it from the default headers and override with the
      // optional headers from response.locals
      const headers = !isUnmodified
        ? res.getHeaders()
        : {
            ...config.response.headers.default,
            ...res.getHeaders(),
            ...(res.locals[responseHeadersLabel] ?? {}),
          };

      // Delete headers that will be re-computed when response later served by cache hit
      delete headers[CACHE_CONTROL_HEADER];
      delete headers[CONTENT_ENCODING_HEADER];
      delete headers[CONTENT_LENGTH_HEADER];
      delete headers[VARY_HEADER];

      const statusCode = isUnmodified ? httpStatusCodes.OK.code : res.statusCode;
      const cachedResponse = new CachedApiResponse(statusCode, headers, responseBody, shouldCompress(responseBody));
      await getCache().setSingle(responseCacheKey, ttl, cachedResponse);
    }
  }
};
