Audit Report

## Title
Silent Treasury Substitution as `msg.sender` When Sender Is a Smart Contract in `getSenderAccountID()`

## Summary
When a caller supplies a non-long-zero EVM address that resolves to an account with `smartContract() == true`, `getSenderAccountID()` silently returns the treasury `AccountID` instead of throwing `PAYER_ACCOUNT_NOT_FOUND`. That treasury ID is placed into `transactionID.accountID`, which the Hedera `TransactionExecutor` uses to derive the EVM-level sender, causing the simulation to run with the treasury's long-zero address as `msg.sender` rather than the user-supplied contract address.

## Finding Description
In `TransactionExecutionService.java`, `getSenderAccountID()`:

```java
// lines 239–265
} else if (account.smartContract()) {
    return EntityIdUtils.toAccountId(systemEntity.treasuryAccount()); // line 251
}
```

The returned treasury `AccountID` is placed into the transaction body:

```java
// line 167
.accountID(getSenderAccountID(params))
```

The `transactionID.accountID` field is the Hedera payer, and in a `ContractCall` transaction the payer is the EVM-level sender. The treasury account (e.g. `0.0.2`) maps to the long-zero EVM address `0x0000000000000000000000000000000000000002`, so `msg.sender` inside the simulation becomes the treasury address rather than the supplied contract address.

The code's own comment at lines 287–288 acknowledges that the real consensus node (`SolvencyPreCheck#getPayerAccount()`) returns `PAYER_ACCOUNT_NOT_FOUND` when the payer is a smart contract. For non-long-zero addresses the mirror node does not replicate this rejection — it silently substitutes the treasury instead.

The test `testExecuteContractCallSenderIsSmartContract` (non-long-zero address, `smartContract=true`) confirms the substitution path succeeds and returns `SUCCESS`.

## Impact Explanation
Any `eth_call` or `eth_estimateGas` request where `from` is set to a deployed contract's EVM address will execute with `msg.sender = <treasury EVM address>` instead of the supplied address. Contracts that branch on `msg.sender` (ownership checks, access control, allowance checks, role-based guards) will return results that diverge from what would happen on the real consensus node. This can cause:

- dApps to receive incorrect simulation results and make wrong decisions (e.g., believing a call would succeed when it would fail on-chain, or vice versa)
- Incorrect gas estimates for transactions that depend on `msg.sender`-gated branches
- Misleading opcode traces that show the treasury as the initiator

Severity is medium: no real funds are at risk (mirror node is read-only), but the simulation fidelity guarantee is broken in a user-controllable way.

## Likelihood Explanation
Exploitation requires no privileges. Any user of the JSON-RPC API (`eth_call`, `eth_estimateGas`) can set the `from` field to any deployed contract address. Deployed contracts are publicly enumerable on-chain. The trigger is trivially repeatable and requires no special knowledge beyond knowing a contract address exists on the network.

## Recommendation
In `getSenderAccountID()`, replace the silent treasury substitution with the same rejection used for other invalid payer cases:

```java
} else if (account.smartContract()) {
    throwPayerAccountNotFoundException(SENDER_NOT_FOUND); // mirror consensus-node behavior
}
```

This aligns the mirror node with the documented consensus-node behavior (`SolvencyPreCheck#getPayerAccount()` returns `PAYER_ACCOUNT_NOT_FOUND` for smart-contract payers) and eliminates the divergent simulation path.

## Proof of Concept
1. Deploy any contract `C` at non-long-zero EVM address `0xABCD...`.
2. Send an `eth_call` with `from: "0xABCD..."` and `to: <some contract that returns msg.sender>`.
3. Observe the returned `msg.sender` is `0x0000000000000000000000000000000000000002` (treasury) instead of `0xABCD...`.
4. The same call on the real consensus node would return `PAYER_ACCOUNT_NOT_FOUND`, not a successful execution with a substituted sender.

    private AccountID getSenderAccountID(final CallServiceParameters params) {
        // Set a default account to keep the sender parameter optional.
        if (params.getSender().isZero() && params.getValue() == 0L) {
            return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
        }
        final var senderAddress = params.getSender();
        final var accountIDNum = getSenderAccountIDAsNum(senderAddress);

        final var account = accountReadableKVState.get(accountIDNum);
        if (account == null) {
            throwPayerAccountNotFoundException(SENDER_NOT_FOUND);
        } else if (account.smartContract()) {
            return EntityIdUtils.toAccountId(systemEntity.treasuryAccount());
        } else if (!account.hasKey() || account.key().equals(IMMUTABILITY_SENTINEL_KEY)) {
            // If the account is hollow, complete it in the state as a workaround
            // as this happens in HandleWorkflow in hedera-app but calling the
            // transaction executor directly skips this account completion and
            // this results in failed transactions that would otherwise succeed
            // against the consensus node.
            final var writableAccountCache =
                    ContractCallContext.get().getWriteCacheState(AccountReadableKVState.STATE_ID);
            final var completedAccount = account.copyBuilder().key(DEFAULT_KEY).build();
            writableAccountCache.put(account.accountId(), completedAccount);
        }

        return accountIDNum;
    }
