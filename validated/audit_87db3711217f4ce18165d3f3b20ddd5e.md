Audit Report

## Title
UNION ALL Subquery Overlap Produces Duplicate Allowance Rows via Crafted `spender.id` Degenerate Range

## Summary
When a caller supplies a degenerate closed range for `spender.id` where the lower and upper bound share the same value (e.g., `spender.id=gte:5&spender.id=lte:5`) together with a closed `token.id` range, the `getLowerFilters` and `getUpperFilters` helpers independently rewrite the primary bound to `spender = X`. Because `TokenAllowanceService.getQuery()` joins the resulting subqueries with `UNION ALL`, every row satisfying both predicates is returned twice.

## Finding Description

**Verified code path:**

`getAccountTokenAllowances` (`tokenAllowanceController.js` line 68–81) → `extractTokenMultiUnionQuery` (line 22–60) → `validateBounds` (line 49) → `getLowerFilters` / `getInnerFilters` / `getUpperFilters` (`baseController.js` lines 131–183) → `TokenAllowanceService.getQuery` (`tokenAllowanceService.js` lines 54–78) → `UNION ALL` at line 74.

**Root cause — verified against source:**

`getLowerFilters` at line 138–141 enters the branch `primary.hasLower() && secondary.hasLower()` and rewrites the primary lower bound to equality:

```js
filters = [{...primary.lower, operator: utils.opsMap.eq}, secondary.lower];
``` [1](#0-0) 

`getUpperFilters` at line 182 independently rewrites the primary upper bound to equality:

```js
return [{...primary.upper, operator: utils.opsMap.eq}, secondary.upper];
``` [2](#0-1) 

When `lower_value === upper_value` (i.e., `gte:5` and `lte:5`), both subqueries carry the identical predicate `spender = 5`. Any row also satisfying both `token_id >= lower` and `token_id <= upper` is matched by both the lower and upper subquery. `UNION ALL` at line 74 concatenates without deduplication:

```js
sqlQuery = [subQueries.map((q) => `(${q})`).join('\nunion all\n'), orderClause, limitClause].join('\n');
``` [3](#0-2) 

**Why validation does not catch it:**

`validateBoundsRange` only rejects the combination of a range **and** an equality on the same key:

```js
if (bounds[key].hasBound() && bounds[key].hasEqual()) { throw ... }
``` [4](#0-3) 

A degenerate range `gte:X lte:X` has no `equal` field set, so this check passes. `validateLowerBounds` and `validateUpperBounds` only verify operator tier compatibility (gte vs gt, lte vs lt), not whether the numeric values of the lower and upper bounds are equal: [5](#0-4) 

**Resulting SQL (concrete example):**

Request: `GET /api/v1/accounts/1/allowances/tokens?spender.id=gte:5&spender.id=lte:5&token.id=gte:100&token.id=lte:200`

```sql
(select * from token_allowance
   where owner = $1 and amount > 0 and spender = $3 and token_id >= $4
   order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance
   where owner = $1 and amount > 0 and spender > $5 and spender < $6   -- ∅: 5>5 AND 5<5
   order by spender asc, token_id asc limit $2)
union all
(select * from token_allowance
   where owner = $1 and amount > 0 and spender = $7 and token_id <= $8
   order by spender asc, token_id asc limit $2)
order by spender asc, token_id asc limit $2
-- params: [1, limit, 5, 100, 5, 5, 5, 200]
```

The inner subquery is vacuous (`spender > 5 AND spender < 5`). Every row with `spender=5` and `100 ≤ token_id ≤ 200` satisfies both the lower and upper subqueries and is returned twice. This is consistent with the existing test fixture for the `spender closed range and token closed range` case, which uses distinct lower/upper values (2 and 8) and thus does not expose the overlap: [6](#0-5) 

## Impact Explanation
The REST API returns a JSON `allowances` array containing duplicate entries for every `(spender, token_id)` pair where `spender = X` and `token_id` falls within the supplied range. Clients that iterate the list, sum amounts, or make authorization decisions based on allowance presence will process phantom duplicate records. Pagination is also corrupted: the outer `LIMIT` is partially consumed by duplicates, causing the `next` link to skip real records. The data integrity of the read path is broken for any targeted account that has at least one token allowance for the targeted spender, with no write access required.

## Likelihood Explanation
The endpoint is unauthenticated and publicly reachable. The trigger requires only two query parameters with the same numeric value (`spender.id=gte:X&spender.id=lte:X`) plus a `token.id` range — a trivially crafted URL. No account ownership, signing key, or elevated privilege is needed. The condition is deterministic and 100% reproducible for any account with at least one token allowance for the targeted spender.

## Recommendation
In `extractTokenMultiUnionQuery` (or in `validateBounds`), add a check that rejects a degenerate range where `primary.lower.value === primary.upper.value`. Alternatively, detect this case and collapse it to a single equality subquery (`spender = X AND token_id >= lower AND token_id <= upper`) rather than generating three subqueries joined by `UNION ALL`. This eliminates the overlap without changing the semantics for legitimate non-degenerate ranges. [7](#0-6) 

## Proof of Concept
1. Ensure the target account (e.g., owner=1) has at least one token allowance with `spender=5` and `token_id` between 100 and 200.
2. Issue: `GET /api/v1/accounts/1/allowances/tokens?spender.id=gte:5&spender.id=lte:5&token.id=gte:100&token.id=lte:200`
3. Observe that the `allowances` array in the JSON response contains each matching `(spender, token_id)` entry exactly twice.
4. The duplication is caused by the lower subquery (`spender=5, token_id>=100`) and the upper subquery (`spender=5, token_id<=200`) both matching the same rows, joined by `UNION ALL` with no deduplication. [8](#0-7)
