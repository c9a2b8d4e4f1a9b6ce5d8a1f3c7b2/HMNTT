Audit Report

## Title
Inconsistent Entity Visibility: `deleted = NULL` Entities Returned by Alias Lookup but Silently Dropped by EVM Address Lookup

## Summary
In `rest/service/entityService.js`, the two SQL queries used for entity resolution apply different NULL-handling logic to the `deleted` column. `entityFromAliasQuery` uses `coalesce(deleted, false) <> true`, correctly treating `NULL` as "not deleted," while `entityFromEvmAddressQuery` uses a bare `deleted <> true`, which evaluates to SQL `NULL` (not `TRUE`) when `deleted IS NULL`, silently excluding the entity. Since migration `V1.39.1` explicitly made `entity.deleted` nullable by design, `deleted = NULL` is a normal, expected database state — not a corrupt one.

## Finding Description
**Exact locations:**

`entityFromAliasQuery` — [1](#0-0) 

```sql
where coalesce(deleted, false) <> true   -- NULL → false → false <> true → TRUE → entity returned
  and alias = $1
```

`entityFromEvmAddressQuery` — [2](#0-1) 

```sql
where deleted <> true                    -- NULL <> true → NULL → WHERE fails → entity silently dropped
  and evm_address = $1
```

These back `getAccountFromAlias()` and `getEntityIdFromEvmAddress()` respectively. [3](#0-2) [4](#0-3) 

**Root cause confirmed by schema migration:**

Migration `V1.39.1` explicitly dropped the `NOT NULL` constraint and default from `entity.deleted`: [5](#0-4) 

The comment reads: *"allow nullable on entity deleted as transaction cannot make this assumption on updates."* This confirms `deleted = NULL` is the normal state for entities upserted without an explicit deletion event.

**Dispatcher with no reconciliation:**

`getEncodedId()` routes to one path or the other based solely on input format, with no fallback or cross-check between the two paths. [6](#0-5) 

## Impact Explanation
The mirror node is the authoritative read layer for Hedera network state. For any entity with `deleted = NULL`, a non-null `alias`, and a non-null `evm_address` (common for ECDSA secp256k1 accounts):

- Alias-based lookup returns the entity as active.
- EVM address-based lookup silently drops it, returning `null` or throwing `NotFoundError`.
- Downstream consumers (wallets, explorers, dApps) resolving the same account by EVM address will see it as non-existent while alias-based resolvers see it as active.
- Any authorization or existence check performed via EVM address lookup will incorrectly report the entity as absent. The inconsistency is silent — no error is logged, no warning is surfaced.

**Severity: Medium.** Data integrity violation in the mirror node's core entity resolution.

## Likelihood Explanation
- **No privileges required.** Any user with knowledge of an account's alias and EVM address can trigger the inconsistency via standard public REST API calls.
- **Precondition is common.** Entities inserted via the upsert path without an explicit deletion event will have `deleted = NULL`. This is the designed behavior per migration `V1.39.1`.
- **Repeatable and deterministic.** The SQL behavior of `NULL <> true` evaluating to `NULL` is a fixed property of PostgreSQL's three-valued logic; the inconsistency reproduces on every query for every affected entity.

## Recommendation
Apply the same `coalesce` guard to `entityFromEvmAddressQuery` as is already used in `entityFromAliasQuery`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
```

This aligns both query paths with the schema's intentional nullability and matches the existing correct pattern. A test covering the `deleted = NULL` case for EVM address lookup should also be added.

## Proof of Concept
1. Insert an entity into the `entity` table with `deleted = NULL`, a non-null `alias`, and a non-null `evm_address`.
2. Query the mirror node REST API using the entity's alias → `getAccountFromAlias()` → `entityFromAliasQuery` → `coalesce(NULL, false) <> true` = `TRUE` → **entity returned**.
3. Query using the entity's EVM address → `getEntityIdFromEvmAddress()` → `entityFromEvmAddressQuery` → `NULL <> true` = `NULL` → **entity silently dropped**, `rows.length === 0` → `NotFoundError` thrown or `null` returned.
4. The same entity is visible via one lookup path and invisible via the other, with no error or warning surfaced.
