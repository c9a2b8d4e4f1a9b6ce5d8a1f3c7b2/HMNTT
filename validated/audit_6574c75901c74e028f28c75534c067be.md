Audit Report

## Title
Negative Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode` Allows DoS Against Newly Deployed Contracts

## Summary
The `@Cacheable` guard `unless = "#result == null"` on `ContractRepository.findRuntimeBytecode` fails to prevent caching of absent-contract results. When no contract row exists, Spring Data JPA returns `Optional.empty()` — a non-null singleton — so the SpEL expression evaluates to `false` and the empty `Optional` is stored in the Caffeine cache with a 1-hour `expireAfterAccess` TTL. An unprivileged attacker who queries a predictable contract address before deployment poisons the cache and causes all subsequent bytecode lookups for that address to return empty for at least one hour.

## Finding Description
**Exact location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18: [1](#0-0) 

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** The SpEL expression `#result == null` is evaluated against the `Optional` wrapper object returned by the method, not the value inside it. `Optional.empty()` is a non-null singleton (`Optional.EMPTY`), so `#result == null` is always `false` for a miss, and Spring's caching infrastructure proceeds to store the empty `Optional` in the cache. The intended guard — "do not cache a miss" — is never triggered for the absent-contract case.

**Cache TTL:** The contract cache is configured as `expireAfterAccess=1h,maximumSize=1000,recordStats`: [2](#0-1) 

`expireAfterAccess` resets the expiry clock on every read. An attacker who periodically re-queries the poisoned ID keeps the stale entry alive indefinitely.

**Consuming call path:** `ContractBytecodeReadableKVState.readFromDataSource` calls `contractRepository.findRuntimeBytecode(entityId.getId())` and chains `.orElse(null)`: [3](#0-2) 

When the cache returns the poisoned `Optional.empty()`, `.orElse(null)` returns `null`, making the EVM treat the address as a non-contract account (no bytecode).

**Test evidence of the gap:** The existing test `ContractRepositoryTest.findRuntimeBytecodeFailCall` confirms that a miss returns `Optional.empty()` but does not assert that the result is excluded from the cache: [4](#0-3) 

## Impact Explanation
Any contract whose numeric entity ID can be predicted before deployment (via CREATE2 deterministic addressing, or by observing the monotonically increasing entity ID sequence) can be rendered non-functional for at least one hour after deployment. All `eth_call`, `eth_estimateGas`, and direct bytecode queries routed through `ContractBytecodeReadableKVState` will receive empty bytecode, causing every call to the contract to revert or behave as if the address is an EOA. This is a targeted, per-contract denial-of-service with no authentication requirement and no rate-limiting barrier.

## Likelihood Explanation
The attack requires no credentials or special privileges — only the ability to send a single JSON-RPC request (e.g., `eth_call` to a not-yet-deployed address). Contract addresses are predictable: CREATE2 addresses are fully deterministic from public parameters, and standard CREATE addresses are derivable from the deployer address and nonce. One successful pre-deployment query is sufficient to poison the cache. The attack is repeatable and can be automated to continuously refresh the cache entry via periodic re-queries, extending the DoS window beyond one hour indefinitely.

## Recommendation
Change the `unless` SpEL expression to also exclude empty `Optional` results:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This ensures that only a genuinely present bytecode result (i.e., `Optional` containing a non-null byte array) is stored in the cache, while cache misses always fall through to the database.

## Proof of Concept
1. Observe the current maximum entity ID from the public mirror node API to predict the next contract's numeric ID (e.g., `N+1`).
2. Before the contract is deployed, send an `eth_call` to the long-zero EVM address corresponding to entity ID `N+1` (e.g., `0x00000000000000000000000000000000000<N+1>`).
3. `ContractBytecodeReadableKVState.readFromDataSource` calls `contractRepository.findRuntimeBytecode(N+1)`. The DB returns no row; Spring Data JPA returns `Optional.empty()`. Because `Optional.empty() != null`, the `unless` guard passes and `Optional.empty()` is stored in the Caffeine cache under key `N+1`.
4. The contract is deployed and receives entity ID `N+1`.
5. All subsequent `eth_call` or `eth_estimateGas` requests for that contract hit the cache, receive `Optional.empty()`, and `readFromDataSource` returns `null` — the EVM sees no bytecode and every call reverts.
6. Optionally, send a re-query every 59 minutes to reset the `expireAfterAccess` timer, keeping the poisoned entry alive indefinitely.
