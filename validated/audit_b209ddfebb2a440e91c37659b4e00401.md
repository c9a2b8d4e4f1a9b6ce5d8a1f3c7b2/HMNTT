Audit Report

## Title
Missing `coalesce()` on Nullable `deleted` Column in `entityFromEvmAddressQuery` Causes Silent Entity Omission

## Summary
In `rest/service/entityService.js`, the static query `entityFromEvmAddressQuery` filters with `deleted <> true` without a `coalesce()` guard. Because the `entity.deleted` column is explicitly nullable (per migration `V1.39.1__upsert_support.sql`), any entity with `deleted IS NULL` is silently excluded by SQL three-valued logic, producing false 404 responses for valid, non-deleted entities looked up by EVM address.

## Finding Description
**File:** `rest/service/entityService.js`, lines 22ā€“25

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where ${Entity.DELETED} <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
```

The predicate `deleted <> true` evaluates to SQL `NULL` (not `TRUE`) when `deleted IS NULL`, causing the row to be excluded from results. The sibling query `entityFromAliasQuery` (lines 17ā€“20) correctly uses `coalesce(${Entity.DELETED}, false) <> true`, making this an inconsistency with a concrete impact.

The `entity.deleted` column is intentionally nullable: migration `V1.39.1__upsert_support.sql` explicitly drops both the `NOT NULL` constraint and the `DEFAULT false`, with the comment *"allow nullable on entity deleted as transaction cannot make this assumption on updates"*. This is a documented, supported schema state.

`getEntityIdFromEvmAddress()` (line 90) executes this query and, when `rows.length === 0`, either throws `NotFoundError` (line 93ā€“94) or returns `null` (line 97) ā€” both incorrect outcomes for an entity that exists and is not deleted.

## Impact Explanation
Any entity with `evm_address` set and `deleted IS NULL` ā€” a valid, schema-permitted state ā€” is permanently invisible to all EVM-address-based REST lookups. Callers receive incorrect 404 / `null` responses for accounts and contracts that genuinely exist and are not deleted. Downstream consumers (wallets, explorers, dApps) relying on EVM address resolution will receive false negatives, leading to incorrect balance displays, failed contract interactions, and broken address-to-entity mappings.

## Likelihood Explanation
No privileges are required. Any user who knows an EVM address of an affected entity can trigger the bug deterministically by issuing a standard REST GET request (e.g., `GET /api/v1/contracts/0x<evmAddress>`). The condition (`deleted IS NULL`) is an explicitly supported schema state that can exist in any production deployment. The bug is stable and requires no race condition or timing window.

## Recommendation
Apply the same `coalesce()` guard used in `entityFromAliasQuery`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
```

This aligns the two sibling queries and correctly treats `NULL` as `false` (not deleted), matching the intended semantics.

## Proof of Concept
1. Insert a row into `entity` with `evm_address = '\xABCD...'` and `deleted = NULL` (valid per schema after migration `V1.39.1`).
2. Issue `GET /api/v1/contracts/0xABCD...` to the mirror node REST API.
3. The request reaches `getEntityIdFromEvmAddress()` ā†’ executes `entityFromEvmAddressQuery`.
4. SQL evaluates `NULL <> true` ā†’ `NULL` ā†’ row excluded ā†’ `rows.length === 0`.
5. `requireResult=true` path throws `NotFoundError` ā†’ API returns HTTP 404.
6. The entity exists and is not deleted, but the API permanently reports it as not found.

    const rows = await this.getRows(EntityService.entityFromEvmAddressQuery, [Buffer.from(entityId.evmAddress, 'hex')]);
    if (rows.length === 0) {
      if (requireResult) {
        throw new NotFoundError();

      throw new Error(EntityService.multipleEvmAddressMatch);

    return rows[0].id;
