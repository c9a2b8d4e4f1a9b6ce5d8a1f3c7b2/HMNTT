## Audit Report

## Title
Silent Fallback to `BlockType.LATEST` in Opcode Replay Produces Historically Inaccurate EVM Trace

## Summary
In `OpcodeServiceImpl.buildCallServiceParameters()`, when `recordFileService.findByTimestamp()` returns empty for a given `consensusTimestamp`, the code silently substitutes `BlockType.LATEST`. This causes `ContractCallContext.useHistorical()` to return `false`, which in turn causes every KV-state reader (storage, balances, bytecode) to read the **current** database state rather than the state at the original consensus timestamp. The HTTP response remains `200 OK` with no indication that the trace is inaccurate.

## Finding Description

**Confirmed code path — `OpcodeServiceImpl.java` lines 158–161:**

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElse(BlockType.LATEST);   // ← silent fallback
``` [1](#0-0) 

The resulting `blockType` is stored in `ContractDebugParameters.block` (line 166). The `consensusTimestamp` field is also stored (line 169) but is **never consulted** for state reads — only `block` matters. [2](#0-1) 

**`ContractCallContext.useHistorical()` — lines 103–105:**

```java
public boolean useHistorical() {
    return callServiceParameters != null && callServiceParameters.getBlock() != BlockType.LATEST;
}
``` [3](#0-2) 

When `block == BlockType.LATEST`, `useHistorical()` is `false`.

**`ContractCallContext.getTimestamp()` — lines 111–116:**

```java
public Optional<Long> getTimestamp() {
    if (useHistorical()) {
        return getTimestampOrDefaultFromRecordFile();
    }
    return Optional.empty();   // ← no historical timestamp pinned
}
``` [4](#0-3) 

**`ContractStorageReadableKVState.readFromDataSource()` — lines 37–44:**

```java
final var timestamp = ContractCallContext.get().getTimestamp();
...
return timestamp
        .map(t -> contractStateService.findStorageByBlockTimestamp(..., t))
        .orElse(contractStateService.findStorage(entityId, keyBytes))
``` [5](#0-4) 

When `timestamp` is `Optional.empty()`, the branch falls to `findStorage(entityId, keyBytes)` — the **latest** row — instead of `findStorageByBlockTimestamp(..., t)`.

**Root cause / failed assumption:** The code assumes that every `contract_result` row will always have a corresponding `record_file` row reachable via `findByTimestamp`. No error is raised when this assumption is violated; the fallback is completely silent.

## Impact Explanation
Any caller of `GET /api/v1/contracts/results/{txHash}/opcodes` may receive an opcode trace that reflects **current** contract storage, balances, and bytecode rather than the state at the time the transaction was originally executed. For contracts whose state has changed since the original transaction (storage slots updated, tokens transferred, contract upgraded), the replayed trace will show different `SLOAD` values, different branch outcomes, and potentially a different execution path than what actually occurred. This directly distorts forensic and audit evidence about historical contract execution, with no error signal to the caller.

## Likelihood Explanation
The precondition — a `contract_result` row existing while its `record_file` is absent — is reachable without any special credentials:

1. **Ingestion lag:** The importer may write `contract_result` before the corresponding `record_file` is committed. Any query during this window triggers the fallback.
2. **Record-file pruning:** Mirror nodes commonly prune older record files to manage storage. Any transaction whose record file has been pruned will permanently trigger the fallback.

Only a valid transaction hash is required, which is publicly observable on-chain.

## Recommendation
Replace the silent `.orElse(BlockType.LATEST)` fallback with an explicit error:

```java
final var blockType = recordFileService
        .findByTimestamp(consensusTimestamp)
        .map(recordFile -> BlockType.of(recordFile.getIndex().toString()))
        .orElseThrow(() -> new EntityNotFoundException(
                "Record file not found for consensus timestamp: " + consensusTimestamp));
``` [1](#0-0) 

This ensures the API returns a clear `404` rather than a silently inaccurate `200 OK` trace. Additionally, consider whether `consensusTimestamp` stored in `ContractDebugParameters` should be used as a direct fallback pin for state reads when a record file is unavailable, so that at minimum the timestamp-based state queries remain anchored to the correct point in time. [6](#0-5) 

## Proof of Concept

1. Identify any contract transaction hash `T` whose `record_file` row is absent (e.g., during importer lag or after pruning).
2. Call `GET /api/v1/contracts/results/{T}/opcodes`.
3. `contractResultRepository.findById(consensusTimestamp)` succeeds (the `contract_result` row exists).
4. `recordFileService.findByTimestamp(consensusTimestamp)` returns `Optional.empty()`.
5. `blockType` is set to `BlockType.LATEST` (line 161).
6. `ContractDebugParameters.block == BlockType.LATEST` → `useHistorical()` returns `false`.
7. `ContractCallContext.getTimestamp()` returns `Optional.empty()`.
8. `ContractStorageReadableKVState.readFromDataSource()` calls `findStorage(entityId, keyBytes)` (latest state) instead of `findStorageByBlockTimestamp(entityId, key, t)`.
9. The response is `200 OK` with an opcode trace reflecting current state, not the state at the time of the original transaction.
