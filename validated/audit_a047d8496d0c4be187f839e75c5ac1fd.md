Audit Report

## Title
Unauthenticated Alias Lookup Forces Uncached DB Query Per Request, Enabling Sustained DB Load (Griefing)

## Summary
Any unauthenticated caller can supply syntactically valid but non-existent account aliases to `/api/v1/accounts/<ALIAS>/nfts` (and sibling routes). Because format validation is the only pre-filter before a live DB query, and 404 responses are never cached, each unique alias forces one full DB lookup with no absorption by the cache layer.

## Finding Description

**Verified code path:**

**Step 1 – Unauthenticated route registration.**
`rest/routes/accountRoute.js` line 15 registers the route with no authentication or rate-limiting middleware: [1](#0-0) 

**Step 2 – Immediate call to `EntityService.getEncodedId()`.**
`AccountController.getNftsByAccountId` at line 91 passes the raw path parameter directly: [2](#0-1) 

**Step 3 – Alias branch in `getEncodedId()`.**
Lines 125–126 of `entityService.js` check `AccountAlias.isValid()` and, on success, call `getAccountIdFromAlias()` → `getAccountFromAlias()`: [3](#0-2) 

**Step 4 – `AccountAlias.isValid()` is a pure regex check.**
The regex `/^(\d{1,5}\.){0,2}[A-Z2-7]+$/` accepts any uppercase base32 string of any length. No DB lookup is performed: [4](#0-3) 

**Step 5 – `getAccountFromAlias()` unconditionally queries the DB.**
`entityFromAliasQuery` is executed on every call; there is no in-process or Redis cache for alias misses: [5](#0-4) 

**Step 6 – `responseCacheUpdateHandler` skips non-2xx responses.**
Line 95 gates caching on `httpStatusCodes.isSuccess(res.statusCode)`. A 404 (alias not found) is never stored, so every repeated or novel non-existent alias hits the DB fresh: [6](#0-5) 

## Impact Explanation
An attacker generating a stream of distinct valid-format aliases (e.g., `AAAAAAAAAA`, `AAAAAAAAAB`, …) causes one `SELECT` against the `entity` table per request, filtered by the `alias` column. Under sustained concurrent load this degrades DB performance for all legitimate users of the mirror node REST API.

## Likelihood Explanation
No privileges are required. The endpoint is publicly reachable. Generating valid base32 strings matching `[A-Z2-7]+` is trivial. A single attacker with a modest HTTP client (`ab`, `wrk`, or a simple script) can sustain hundreds of unique alias queries per second. Application-level rate limiting is absent from the REST middleware stack; the only documented protection is optional infrastructure-level Traefik configuration, which may not be deployed in all environments. [7](#0-6) 

## Recommendation
1. **Cache negative alias lookups.** Introduce a short-lived (e.g., 5–30 s) in-process or Redis cache for alias-miss results in `getAccountFromAlias()` / `getAccountIdFromAlias()`. This converts repeated identical-alias attacks into cache hits at near-zero cost.
2. **Apply application-level rate limiting.** Add a rate-limit middleware (e.g., `express-rate-limit`) to the REST router, keyed on IP, before alias resolution occurs.
3. **Bound alias length.** Extend `AccountAlias.isValid()` to enforce a maximum alias length consistent with the protocol's actual key sizes, reducing the effective attack surface. [8](#0-7) 

## Proof of Concept
```bash
# Generate 10 000 unique valid-format aliases and fire concurrent requests
python3 -c "
import itertools, string
chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'
for i, combo in enumerate(itertools.product(chars, repeat=5)):
    if i >= 10000: break
    print(''.join(combo))
" > aliases.txt

# Send 200 concurrent requests, cycling through unique aliases
while read alias; do
  curl -s "http://<mirror-node>/api/v1/accounts/${alias}/nfts" &
done < aliases.txt
wait
```
Each request passes `AccountAlias.isValid()`, reaches `getAccountFromAlias()`, executes `entityFromAliasQuery` against the DB, returns HTTP 404, and is **not** stored in the response cache — guaranteeing a fresh DB hit for every subsequent request with the same or a new alias. [9](#0-8) [10](#0-9)
