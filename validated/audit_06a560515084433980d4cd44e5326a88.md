Audit Report

## Title
Unauthenticated DB Saturation via `findStorageBatch()` Amplification Through Global 500 RPS Throttle

## Summary
A single unauthenticated attacker can consume the entire global 500 RPS budget and exploit a permanent cache-miss condition for non-existent contract storage slots to drive up to 500 uncached `SELECT ... WHERE slot IN (1500 slots)` queries per second against the `contract_state` table, exhausting the DB connection pool and degrading EVM contract call processing for all users.

## Finding Description

**1. Global, non-partitioned rate limiter**

`ThrottleManagerImpl.throttle()` checks a single shared `rateLimitBucket` with no per-IP or per-user partitioning: [1](#0-0) 

The bucket capacity is the default `requestsPerSecond = 500`: [2](#0-1) 

One attacker can consume all 500 tokens/second.

**2. `findStorage()` unconditionally calls `findStorageBatch()` on any cache miss**

`ContractStateServiceImpl.findStorage()` checks `contractStateCache` first; if the value is absent or equals `EMPTY_VALUE`, it falls through to `findStorageBatch()`: [3](#0-2) 

`contractStateCache` is configured `expireAfterWrite=2s,maximumSize=25000`: [4](#0-3) 

**3. Non-existent slots are never written to `contractStateCache`**

Inside `findStorageBatch()`, only rows actually returned by the DB are written to `contractStateCache`: [5](#0-4) 

Non-existent slots produce an empty result set, so nothing is ever written to `contractStateCache` for them. Every subsequent call to `findStorage()` for a non-existent slot returns `null` from the cache and re-enters `findStorageBatch()` — a permanent cache miss.

**4. `findStorageBatch()` accumulates slot keys and issues a growing IN-clause query**

Each invocation of `findStorageBatch()` registers the requested key in `contractSlotsCache` (TTL `expireAfterAccess=5m`, max 1500 slots per contract), then queries the DB with **all accumulated keys**: [6](#0-5) 

`slotsPerContract` cache configuration: [7](#0-6) 

The repository method has no `@Cacheable` annotation — every call hits the database directly: [8](#0-7) 

**5. No synchronization between concurrent calls**

`findStorageBatch()` is an unsynchronized private method. All 500 concurrent requests issue independent DB queries simultaneously with no coalescing.

**Exploit flow:**

- **Phase 1 (slot poisoning):** Attacker sends ~1500 requests targeting unique non-existent slot keys for a single contract. Each request passes the global throttle, misses `contractStateCache`, calls `findStorageBatch()`, and registers the non-existent key in `contractSlotsCache` (5-minute TTL). These slots are never written to `contractStateCache` because they don't exist in the DB.
- **Phase 2 (sustained amplification):** Any subsequent request for any slot of that contract calls `findStorageBatch()` with up to 1500 slots in the `IN` clause. The 2-second `contractStateCache` TTL means even legitimately cached slots re-trigger the batch query every 2 seconds. At 500 RPS, this produces 500 × `SELECT ... WHERE slot IN (1500 slots)` queries per second.

## Impact Explanation

500 uncached `SELECT ... WHERE slot IN (1500 slots)` queries per second against the `contract_state` table will exhaust the DB connection pool and CPU. The `db.statementTimeout` of 3000 ms means each query holds a connection for up to 3 seconds, causing connection pool exhaustion and cascading timeouts across all web3 API nodes. [9](#0-8) 

This degrades or halts EVM contract call processing for all users. The attack is sustained as long as the attacker maintains 500 RPS, which is trivially achievable from a single machine.

## Likelihood Explanation

No authentication, API key, or account is required. The attacker only needs HTTP access to the `/api/v1/contracts/call` endpoint. Sending 500 HTTP requests/second is achievable with standard load-testing tools (`wrk`, `k6`, `hey`) from a single machine. The attack is repeatable indefinitely because the `contractSlotsCache` TTL is 5 minutes and non-existent slots are never evicted by successful DB lookups. The attacker does not need to know valid slot keys — any 32-byte values work as non-existent slots.

## Recommendation

1. **Cache negative results:** In `findStorageBatch()`, after the DB query returns, write a sentinel value (e.g., `EMPTY_VALUE`) to `contractStateCache` for every slot key that was queried but not found. Update `findStorage()` to treat a cached `EMPTY_VALUE` as a definitive "not found" result, skipping `findStorageBatch()`. This eliminates the permanent cache-miss condition for non-existent slots.

2. **Per-IP rate limiting:** Introduce a per-source-IP token bucket in `ThrottleManagerImpl` so a single client cannot consume the entire global 500 RPS budget.

3. **Cap the IN-clause size:** Add a hard limit on the number of slot keys passed to `findStorageBatch()` to bound the per-query cost regardless of how many keys have accumulated in `contractSlotsCache`.

4. **Synchronize/coalesce concurrent batch calls:** Use a per-contract lock or a loading cache (e.g., Caffeine's `AsyncLoadingCache`) to ensure that concurrent requests for the same contract share a single in-flight DB query rather than issuing independent ones.
