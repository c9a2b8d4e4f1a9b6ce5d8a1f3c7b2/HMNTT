Audit Report

## Title
Unprivileged Users Trigger Unthrottled DB Polling via Automatic Safety Check

## Summary
Every gRPC `subscribeTopic` call without an `endTime` or `limit` unconditionally fires a safety-check after 1 second that invokes the unthrottled retrieval code path. The unthrottled path exhausts all 12 configured polls at 20 ms intervals with a 5,000-row page size, issuing up to 12 DB queries per subscription regardless of whether any data exists. This is a code-level amplification flaw, not a volumetric DDoS, and is exploitable by any unauthenticated client.

## Finding Description

**Safety check always fires for limitless subscriptions.**

In `TopicMessageServiceImpl.java`, `TopicContext.isComplete()` returns `false` whenever `filter.getEndTime() == null`: [1](#0-0) 

The safety check at lines 67–70 filters on `!topicContext.isComplete()`, so for any subscription without `endTime` it always passes and fires: [2](#0-1) 

**Safety check calls the unthrottled retriever with no limit.**

`missingMessages(topicContext, null)` (the safety-check branch) builds `gapFilter` by copying the original filter and setting only `startTime` — no limit is added. It then calls `topicMessageRetriever.retrieve(gapFilter, false)` (`throttled=false`): [3](#0-2) 

**Unthrottled `PollingContext` never completes without a limit.**

In `PollingTopicMessageRetriever.java`, the unthrottled `PollingContext` constructor sets `numRepeats = unthrottled.getMaxPolls()` (default **12**), `frequency = 20 ms`, and `maxPageSize = 5,000`: [4](#0-3) 

`isComplete()` for the unthrottled path returns only `limitHit`, which requires `filter.hasLimit()` to be true. Without a limit, it is always `false`, so all 12 polls are exhausted: [5](#0-4) 

Default values confirmed in `RetrieverProperties.java`: [6](#0-5) 

**Contrast with throttled path.** Under `throttled=true`, `isComplete()` returns `true` as soon as a partial page is received (line 125), capping DB queries to one per 2 s with page size 1,000. The unthrottled path has no such early-exit. [7](#0-6) 

## Impact Explanation

Each limitless subscription triggers exactly 12 DB queries at 20 ms intervals with page size 5,000 — up to 60,000 rows fetched per subscription — regardless of whether any rows exist. Under the throttled path the same subscription would issue at most one query per 2 s with page size 1,000. The per-connection DB amplification factor is approximately **300×** in query frequency and **5×** in page size. With N concurrent connections the total DB load scales linearly with N, causing CPU/IO saturation and degraded service for all users.

## Likelihood Explanation

The gRPC `subscribeTopic` endpoint is publicly reachable. An attacker needs only to open many long-lived gRPC streams to a valid topic with no `endTime` and no `limit`, then wait 1 second. The attack is trivially scriptable, requires zero privileges, and is repeatable indefinitely. The 1-second delay is not a meaningful barrier.

## Recommendation

1. **Add a limit to the safety-check `gapFilter`.** When building `gapFilter` in `missingMessages()`, apply a reasonable cap (e.g., `maxPageSize` of the unthrottled retriever) so the retriever's `isComplete()` can terminate early on `limitHit`.
2. **Make `isComplete()` for the unthrottled path mirror the throttled path's partial-page check.** Return `true` when `pageSize < maxPageSize`, not only on `limitHit`. This ensures the retriever stops as soon as the DB returns fewer rows than requested.
3. **Rate-limit safety-check invocations per subscriber** to prevent repeated triggering across reconnects.

## Proof of Concept

```
# Open N concurrent gRPC streams to a valid topic, no endTime, no limit
for i in $(seq 1 100); do
  grpcurl -plaintext -d '{"topicID":{"topicNum":1}}' \
    mirror-node:5600 \
    com.hedera.mirror.api.proto.ConsensusService/subscribeTopic &
done
# After 1 second, each stream triggers 12 unthrottled DB queries at 20ms intervals
# = 1,200 DB queries in ~240ms, each fetching up to 5,000 rows
```

Each connection independently enters `missingMessages(topicContext, null)` → `retrieve(gapFilter, false)` → `PollingContext(throttled=false)` → 12 polls × 20 ms, exhausting `maxPolls` because `isComplete()` never returns `true` without a limit. [8](#0-7) [9](#0-8)
