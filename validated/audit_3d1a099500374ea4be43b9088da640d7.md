Audit Report

## Title
Non-Idempotent Retry Loop in `submitWithRetry()` Enables Duplicate HBAR Transfers

## Summary
`submitWithRetry()` in `pinger/transfer.go` constructs a new `TransferTransaction` — and therefore a new `TransactionID` — on every retry iteration. If `Execute()` succeeds (transaction reaches consensus) but `GetReceipt()` subsequently fails for any reason, the retry loop submits a second, independently-committed transaction. Because Hedera deduplicates solely by `TransactionID`, both transfers are permanently finalized. A malicious consensus node injected via the unauthenticated HTTP call in `fetchMirrorNodeNetwork()` can reliably trigger this condition, but even ordinary transient network failures are sufficient.

## Finding Description

**Non-idempotent retry — `pinger/transfer.go` lines 23–57:**

`hiero.NewTransferTransaction()` is called at line 29, inside the `for i := 1; i <= attempts; i++` loop. Each call produces a fresh SDK object whose `TransactionID` is auto-generated at `Execute()` time (operator account ID + current wall-clock nanoseconds). There is no frozen transaction reused across iterations, no explicit `SetTransactionID()`, and no idempotency key. [5](#0-4) 

The only early-exit paths are `ctx.Err()` (line 24, cancellation only) and a successful receipt (line 36–39). Any error from `GetReceipt()` — including a transient timeout after the transaction has already been committed — causes the loop to fall through, sleep, and submit a completely new transaction. [6](#0-5) 

**Unauthenticated node injection surface — `pinger/mirror_node_client.go` lines 74–131:**

`fetchMirrorNodeNetwork()` uses a plain `http.Client` with no TLS enforcement, no certificate pinning, and no integrity check on the returned JSON. The resulting node map is passed directly to `hiero.ClientForNetwork()` and becomes the sole gRPC node pool for all subsequent SDK calls. Any party who can MITM the plain-HTTP mirror REST connection can inject an arbitrary gRPC endpoint as a "legitimate" consensus node. [2](#0-1) [7](#0-6) [8](#0-7) 

## Impact Explanation

Each duplicate transfer is a consensus-finalized, irreversible HBAR debit from the operator account. With `maxRetries=10` (default), up to 11 transfers can be committed per pinger tick instead of one. [9](#0-8) 

Even without a malicious actor, a transient `GetReceipt()` timeout after a successful `Execute()` causes an unintended duplicate transfer. With a malicious injected node that deliberately returns receipt errors, the attacker can reliably drain up to 11× the intended amount per tick. The duplicate records also corrupt the mirror node's transfer history, undermining downstream analytics and reconciliation.

## Likelihood Explanation

The non-idempotent retry requires no attacker at all — ordinary transient network conditions (gRPC deadline exceeded, node restart, load-balancer reset) are sufficient to trigger it. The MITM amplification path is scoped to `network=other` deployments, which is the exact mode that calls `fetchMirrorNodeNetwork()`. In Kubernetes environments (evidenced by `/tmp/ready` and `/tmp/alive` probe files), a compromised sidecar, rogue DNS entry, or misconfigured service mesh can trivially MITM the plain-HTTP mirror REST call without any cryptographic material. [10](#0-9) 

## Recommendation

1. **Fix the retry loop:** Freeze the transaction before the loop and reuse the same object (and thus the same `TransactionID`) across all retry attempts:
   ```go
   cryptoTransfer, _ := hiero.NewTransferTransaction().
       AddHbarTransfer(...).
       AddHbarTransfer(...).
       FreezeWith(client)
   for i := 1; i <= attempts; i++ {
       resp, err := cryptoTransfer.Execute(client)
       ...
   }
   ```
   This ensures Hedera's network-level deduplication window covers all retries.

2. **Check prior receipt before retrying:** After a `GetReceipt()` error, query `resp.TransactionID` directly to confirm whether the transaction was committed before submitting a new one.

3. **Enforce HTTPS for the mirror REST endpoint:** Reject plain-HTTP `mirrorRest` URLs at config validation time, or use a TLS-configured `http.Client` with certificate verification. [11](#0-10) 

## Proof of Concept

1. Deploy the pinger with `network=other` and a mirror REST URL pointing to an attacker-controlled HTTP server.
2. The attacker's server returns a `nodes` JSON array containing a single entry whose `service_endpoints` point to an attacker-controlled gRPC server.
3. The attacker's gRPC server, upon receiving a `CryptoTransfer`, forwards the exact transaction bytes to a real Hedera node (transaction commits), then returns a gRPC `DEADLINE_EXCEEDED` error to the SDK's `GetReceipt()` call.
4. `submitWithRetry()` falls through to iteration 2, constructs a new `TransferTransaction` with a new `TransactionID`, and submits it. The second transaction also commits.
5. Both transfers appear as distinct records in Hashgraph history. Repeat for all 11 attempts.

Even without step 1–3, reproducing with a real network: submit a transfer, then kill the gRPC connection immediately after `Execute()` returns but before `GetReceipt()` completes. Observe two transfer records for the operator account in the mirror node's REST API.
