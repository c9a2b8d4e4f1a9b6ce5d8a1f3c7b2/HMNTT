# Audit Report

## Title
Unbounded Alias Processing in `resolveAccountAliases()` Enables DB Query Amplification via Unauthenticated Request

## Summary
The `resolveAccountAliases()` function in `constructionAPIService` processes a caller-controlled, comma-separated list of account aliases with no upper bound on count and no deduplication. An unauthenticated attacker can repeat a single valid alias thousands of times in one POST to `/construction/metadata`, triggering one synchronous database query per entry and growing an unbounded `accountMap` slice, with no server-side guard to prevent this.

## Finding Description

`ConstructionMetadata()` delegates to `resolveAccountAliases()` at line 159: [6](#0-5) 

Inside `resolveAccountAliases()`, the loop at line 575 iterates over every comma-separated token in the attacker-supplied string with no guard on count: [1](#0-0) 

For every token that represents an alias-based account ID, `GetAccountId()` issues a real synchronous DB query: [2](#0-1) 

Numeric account IDs skip the DB query (line 142–143), but alias-based IDs always hit the database. There is no deduplication: the same alias repeated N times produces exactly N DB queries.

**Root cause — two compounding failures:**
1. No alias count limit anywhere in the call chain.
2. No deduplication before querying.

**Why existing mitigations are insufficient:**
- `main.go` configures `ReadTimeout`, `WriteTimeout`, and `ReadHeaderTimeout` but applies no `http.MaxBytesReader`, so a large body (e.g., 3.5 MB of repeated aliases) is accepted in full before processing begins. [3](#0-2) 
- The Traefik middleware caps in-flight requests at 5 per IP and rate at 10 req/s per host, but: (a) these are Kubernetes-only controls bypassed by direct access; (b) they do not limit the alias count within a single request; (c) 5 concurrent requests × N aliases each = 5N simultaneous DB queries. [7](#0-6) 

## Impact Explanation

**Database exhaustion:** Each alias-based token causes one synchronous call to `selectCurrentCryptoEntityByAlias`. With N = 50,000 aliases per request and 5 concurrent requests (Traefik in-flight limit), this produces up to 250,000 synchronous DB queries, saturating the connection pool and starving all other operations including block ingestion and balance queries.

**Memory pressure:** `accountMap` grows to N entries; `strings.Join` at line 588 allocates a single contiguous result string proportional to N. At N = 50,000 this is approximately 4 MB per request. While `strings.SplitSeq` is lazy (O(1) for the split itself), the accumulated `accountMap` slice and the final joined string are both heap-allocated and live until the response is serialised.

**No authentication required.** The `/construction/metadata` endpoint is publicly reachable.

## Likelihood Explanation

A valid alias is trivially discoverable via the public Hedera mirror REST API — any account that has submitted a transaction with an alias is visible. The attacker needs only one such alias. The attack payload is a single HTTP POST with a comma-separated string of ~3.5 MB (50,000 repetitions of a ~70-char alias), well within default HTTP client and reverse-proxy body limits. It is repeatable at the Traefik rate limit of 10 req/s, making sustained exhaustion straightforward with no special tooling.

## Recommendation

1. **Enforce an alias count limit** in `resolveAccountAliases()` before the loop — reject requests where the number of comma-separated tokens exceeds a small constant (e.g., 10–20). [8](#0-7) 
2. **Deduplicate aliases** before querying — use a `map[string]struct{}` to skip repeated tokens.
3. **Apply `http.MaxBytesReader`** in the HTTP handler or server middleware to cap request body size (e.g., 64 KB), preventing oversized payloads from reaching application logic. [3](#0-2) 
4. **Add a Traefik `buffering` middleware** with `maxRequestBodyBytes` to enforce body size limits at the ingress layer as a defence-in-depth measure.

## Proof of Concept

```bash
# 1. Discover a valid alias from the public mirror REST API
ALIAS="0.0.AABBCCDDEEFF..."  # any alias-based account

# 2. Build a payload with 50,000 repetitions
ALIASES=$(python3 -c "print(','.join(['$ALIAS']*50000))")

# 3. POST to /construction/metadata
curl -s -X POST https://<mirror-node>/construction/metadata \
  -H "Content-Type: application/json" \
  -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"mainnet\"},\"options\":{\"account_aliases\":\"$ALIASES\"}}"

# Result: 50,000 synchronous DB queries issued server-side per request.
# Repeat with 5 concurrent processes to reach 250,000 simultaneous queries.
```
