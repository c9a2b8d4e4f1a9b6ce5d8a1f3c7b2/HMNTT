# Audit Report

## Title
Gas Throttle Bypass via Deliberate Gas-Exhausting Failures Allows Sustained Maximum-Load EVM Execution

## Summary
The `restoreGasToBucket()` method in `ContractCallService` restores 100% of the gas limit back to the throttle bucket whenever a call fails with full gas exhaustion. Because the EVM still performs the full computation in such cases, the gas bucket is never net-depleted by these requests. An unprivileged attacker can craft `eth_call` requests that always exhaust all gas, causing the gas-per-second throttle to be completely ineffective while the node performs maximum-gas EVM computation on every request.

## Finding Description

**Exact code path:**

`ContractCallService.java`, `restoreGasToBucket()`, lines 140–152:

```java
private void restoreGasToBucket(EvmTransactionResult result, long gasLimit) {
    final var gasLimitToRestoreBaseline =
        (long) (gasLimit * throttleProperties.getGasLimitRefundPercent() / 100f);
    if (result == null || (!result.isSuccessful() && gasLimit == result.gasUsed())) {
        throttleManager.restore(gasLimitToRestoreBaseline); // restores full gasLimit (default: 100%)
    } else {
        final var gasRemaining = gasLimit - result.gasUsed();
        throttleManager.restore(Math.min(gasRemaining, gasLimitToRestoreBaseline));
    }
}
``` [1](#0-0) 

`ThrottleManagerImpl.java`, `restore()`, lines 59–64:

```java
public void restore(long gas) {
    long tokens = throttleProperties.scaleGas(gas);
    if (tokens > 0) {
        gasLimitBucket.addTokens(tokens);
    }
}
``` [3](#0-2) 

**Root cause:** The code assumes that a gas-exhausting failure means the user's transaction was "wasted" and the gas should be returned to the bucket. This assumption is false: the EVM still performed the full computation up to the gas limit, consuming real CPU resources. The gas bucket is supposed to meter that computation, but restoring 100% of the gas after a gas-exhausting failure means the bucket is never actually depleted by such requests.

**Token arithmetic (with defaults):**

- `gasLimit` = 15,000,000 (max per call)
- `scaleGas(15,000,000)` = `15,000,000 / 10,000` = **1,500 tokens consumed** by `throttle()`
- `gasLimitRefundPercent` = 100 → `gasLimitToRestoreBaseline` = 15,000,000
- `restore(15,000,000)` → `scaleGas(15,000,000)` = **1,500 tokens restored**
- **Net tokens consumed from bucket: 0**

## Impact Explanation

The gas-per-second throttle (`gasPerSecond = 7,500,000,000`, bucket capacity = 750,000 scaled tokens) is the primary mechanism limiting EVM CPU consumption on the mirror node. [9](#0-8)  By bypassing it, an attacker forces the node to execute maximum-gas EVM computations (up to 15M gas per call) at the full RPS rate (up to 500 RPS). [10](#0-9)  This yields up to 7.5B gas/second of sustained EVM computation — the designed maximum — regardless of what legitimate traffic is doing. The mirror node is a shared read-only service; sustained maximum load degrades response times for all users and can render the service unavailable.

## Likelihood Explanation

No privileges, authentication, or special accounts are required. Any caller of the public `eth_call` or `eth_estimateGas` endpoints can trigger this. Crafting a gas-exhausting call is trivial: target any existing contract with calldata that causes an infinite loop or very expensive computation, or deploy a minimal contract with `while(true){}`. The attack is fully repeatable and automatable with a simple HTTP client loop. The attacker does not need to pay any fees (this is a mirror node simulation, not a real Hedera transaction). The only remaining constraint is the RPS limit (500 req/s), which does not prevent the gas throttle bypass.

## Recommendation

The fix should ensure that gas-exhausting failures are **not** treated as free operations. Two complementary approaches:

1. **Reduce `gasLimitRefundPercent` default:** Change the default from `100` to a lower value (e.g., `0` or `25`). [2](#0-1)  With `gasLimitRefundPercent = 0`, gas-exhausting failures restore nothing, so the full gas cost is charged to the bucket. This directly closes the bypass.

2. **Distinguish gas-exhausting failures from other failures:** The current condition `!result.isSuccessful() && gasLimit == result.gasUsed()` treats gas exhaustion as a special case deserving a full refund. [11](#0-10)  Instead, gas-exhausting failures should restore `0` (or a minimal amount), since the EVM performed the full computation. Only reverts that did not exhaust gas (where `gasUsed < gasLimit`) should receive a partial refund of the unused gas.

## Proof of Concept

import requests, time

# Target any mirror node eth_call endpoint
URL = "http://<mirror-node>/api/v1/contracts/call"

# Payload: call a contract that always exhausts gas (e.g., infinite loop)

    "data": "0x<calldata-for-infinite-loop>",
    "to": "0x<target-contract>",
    "gas": 15000000,  # maximum gas
    "estimate": False
}

# Send at maximum RPS — gas bucket is never depleted
while True:
    requests.post(URL, json=payload)
    # No sleep needed; RPS limit (500/s) is the only constraint
    # Gas bucket stays full because each gas-exhausting failure
    # restores 100% of the gas limit back to the bucket
```

With `gasLimitRefundPercent = 100` (default), each iteration consumes and immediately restores 1,500 scaled tokens, leaving the bucket perpetually full. The node performs 15M-gas EVM computations at up to 500 RPS = 7.5B gas/s sustained load, matching the designed maximum and starving legitimate users of capacity.
