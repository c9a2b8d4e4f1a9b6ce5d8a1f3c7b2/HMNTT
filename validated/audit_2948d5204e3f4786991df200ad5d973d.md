All claims in the report are verified against the actual code. The discrepancy between the two queries is real, the migration is real, and the SQL NULL semantics are correct.

Audit Report

## Title
Missing COALESCE on Nullable `deleted` Column in `entityFromEvmAddressQuery` Causes Live Entities with `deleted = NULL` to Be Silently Excluded

## Summary
`entityFromEvmAddressQuery` filters with `deleted <> true` without `COALESCE`. Because migration `V1.39.1__upsert_support.sql` explicitly made `deleted` nullable, any entity with `deleted = NULL` evaluates the predicate to `NULL` (unknown) in PostgreSQL, causing the row to be silently dropped. The parallel alias-lookup query already applies the correct `COALESCE` fix; the EVM address path does not.

## Finding Description
In `rest/service/entityService.js`, the two static query strings diverge in how they handle the nullable `deleted` column:

**Vulnerable query** (lines 22–25):
```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where ${Entity.DELETED} <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
``` [1](#0-0) 

**Correct query** (lines 17–20):
```js
static entityFromAliasQuery = `select ${Entity.ID}
                               from ${Entity.tableName}
                               where coalesce(${Entity.DELETED}, false) <> true
                                 and ${Entity.ALIAS} = $1`;
``` [2](#0-1) 

The `deleted` column was made nullable by design in migration `V1.39.1__upsert_support.sql`:
```sql
alter table entity
    alter column deleted drop default,
    alter column deleted drop not null;
``` [3](#0-2) 

In PostgreSQL, `NULL <> true` evaluates to `NULL`, which is falsy in a `WHERE` clause. Any entity with `deleted = NULL` is therefore excluded from `entityFromEvmAddressQuery` results.

`getEntityIdFromEvmAddress()` (lines 90–104) throws `NotFoundError` when zero rows are returned: [4](#0-3) 

This is called by `getEncodedId()` (line 124) for all EVM-address-based lookups: [5](#0-4) 

## Impact Explanation
Any live entity whose `deleted` column is `NULL` — a state the importer explicitly produces during partial upserts — becomes completely invisible to all EVM-address-based API lookups. Endpoints for accounts, contracts, tokens, and transaction routing that resolve identifiers via `getEncodedId()` → `getEntityIdFromEvmAddress()` will return 404 for these entities. The same entity remains reachable by alias (since `entityFromAliasQuery` uses `COALESCE`), creating an inconsistent identity resolution: the entity exists by one lookup path but not the other.

## Likelihood Explanation
No privileges are required. The attacker only needs the EVM address of an affected entity, which is public information on a mirror node. The `deleted = NULL` state is not an edge case — it is the documented, intentional result of the upsert mechanism used during normal importer operation (as stated in the migration comment). Any entity last written via a partial update may be in this state. The attack is trivially repeatable.

## Recommendation
Apply the same `COALESCE` pattern already used in `entityFromAliasQuery` to `entityFromEvmAddressQuery`:

```js
static entityFromEvmAddressQuery = `select ${Entity.ID}
                                    from ${Entity.tableName}
                                    where coalesce(${Entity.DELETED}, false) <> true
                                      and ${Entity.EVM_ADDRESS} = $1`;
```

This mirrors the fix already present at line 19 and correctly treats `NULL` as `false` (not deleted). [6](#0-5) 

## Proof of Concept
1. Insert an entity into the `entity` table with a known `evm_address` and `deleted = NULL` (the state produced by a partial upsert).
2. Call `GET /api/v1/accounts/{evmAddress}` (or any endpoint that resolves by EVM address).
3. Observe a `404 Not Found` response.
4. Call `GET /api/v1/accounts/{alias}` for the same entity (if it has an alias).
5. Observe a `200 OK` response — confirming the entity exists but is invisible via EVM address lookup.

The discrepancy is caused solely by the missing `COALESCE` in `entityFromEvmAddressQuery`.
