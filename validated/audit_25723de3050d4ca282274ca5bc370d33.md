Audit Report

## Title
Unauthenticated Multi-Path Amplification of Expensive `ethereum_transaction` JOIN via `getDetailedContractResults()` with No REST-Layer Rate Limiting

## Summary
`getEthTransactionByTimestampAndPayerId()` executes an `ethereum_transaction JOIN transaction` query on every invocation with no service-layer caching. Two structurally distinct public endpoints both invoke `getDetailedContractResults()`, which always fires this JOIN. Because the response cache key is derived from the full request URL, the same underlying data produces separate cache entries per endpoint, allowing an attacker to double the DB hit rate per timestamp. No rate-limiting middleware exists in the REST API layer.

## Finding Description

**Root cause — unbounded JOIN on every cache miss:**

`rest/service/transactionService.js` lines 74–85 build and execute the JOIN query unconditionally on every call: [1](#0-0) 

The static query it references (lines 28–55) performs a full `JOIN` between `ethereum_transaction` and `transaction` on `(consensus_timestamp, payer_account_id)`: [2](#0-1) 

**Both public endpoints always invoke this JOIN:**

`getDetailedContractResults()` (lines 1205–1220) unconditionally calls `TransactionService.getEthTransactionByTimestampAndPayerId()` as one of five parallel DB queries: [3](#0-2) 

It is called from:
- `getContractResultsByTimestamp` — `/api/v1/contracts/:contractId/results/:timestamp` (line 1017): [4](#0-3) 
- `getContractResultsByTransactionIdOrHash` — `/api/v1/contracts/results/:transactionIdOrHash` (line 1166): [5](#0-4) 

**Why the cache does not protect against this:**

The cache key is `MD5(req.originalUrl)`: [6](#0-5) 

The two endpoints have structurally different URLs for the same underlying ethereum transaction (one uses `contractId/timestamp`, the other uses `transactionIdOrHash`), so they produce independent cache entries. Each endpoint independently triggers a full DB round-trip on cache miss. Additionally, query parameters such as `?hbar=true` vs `?hbar=false` produce further distinct cache entries for the same data.

**No rate limiting in the REST API:**

The complete middleware directory contains: `authHandler.js`, `httpErrorHandler.js`, `metricsHandler.js`, `openapiHandler.js`, `requestHandler.js`, `requestNormalizer.js`, `responseCacheHandler.js`, `responseHandler.js` — no rate-limiting or throttling middleware:


A grep across all `rest/**/*.js` for `rate.?limit|throttle|rateLimit` returns zero matches in production code (only a test utility). The `ThrottleManagerImpl`/`ThrottleConfiguration` found elsewhere in the codebase are scoped to the `web3` Java module and do not apply here.

## Impact Explanation

Each unique `(timestamp, contractId/hash)` pair that reaches either endpoint before its cache warms triggers 5 parallel DB queries, including the `ethereum_transaction JOIN transaction`. An attacker rotating through N distinct timestamps across both endpoints generates up to 10N DB queries (5 per endpoint × 2 endpoints) before any caching takes effect. With a 600-second cache window and no rate limiting, a sustained low-rate flood of unique timestamps can continuously saturate DB I/O, causing service degradation or unavailability for legitimate users.

## Likelihood Explanation

- **No authentication required**: both endpoints are fully public.
- **No rate limiting**: the REST API has zero throttling middleware in production.
- **Timestamps are enumerable**: block/transaction timestamps are publicly discoverable via `/api/v1/blocks` and `/api/v1/transactions`.
- **Repeatability**: each new timestamp resets the cache-miss window; the attacker simply iterates.
- **Low sophistication**: a simple script cycling through timestamps across both endpoint paths is sufficient.

## Recommendation

1. **Service-layer cache for `getEthTransactionByTimestampAndPayerId()`**: add an in-process or Redis-backed cache keyed on `(consensusTimestamp, payerAccountId)` so repeated calls for the same data — regardless of which endpoint triggered them — are served from cache.
2. **Normalize cache keys**: strip or canonicalize query parameters that do not affect the DB query (e.g., `hbar`) before computing the cache key in `cacheKeyGenerator`, so `?hbar=true` and `?hbar=false` share a single cache entry.
3. **Add REST-layer rate limiting**: introduce a per-IP (and optionally global) rate-limiting middleware in `rest/middleware/` (e.g., `express-rate-limit` backed by Redis) and register it in `rest/middleware/index.js`.
4. **Consider query-level protection**: add a DB-side index on `(consensus_timestamp, payer_account_id)` on `ethereum_transaction` if not already present, and consider a connection-pool query queue to bound concurrent JOIN executions.

## Proof of Concept

```bash
# 1. Enumerate known timestamps from the public blocks endpoint
TIMESTAMPS=$(curl -s "https://<mirror-node>/api/v1/blocks?limit=100" \
  | jq -r '.blocks[].timestamp.from')

# 2. For each timestamp, hit both endpoints (different cache keys, same DB hit)
for TS in $TIMESTAMPS; do
  # Endpoint 1: by contractId + timestamp
  curl -s "https://<mirror-node>/api/v1/contracts/0.0.1234/results/$TS" &
  # Endpoint 2: by transactionId (resolve timestamp -> txId via /api/v1/transactions)
  TXID=$(curl -s "https://<mirror-node>/api/v1/transactions?timestamp=$TS&limit=1" \
    | jq -r '.transactions[0].transaction_id')
  curl -s "https://<mirror-node>/api/v1/contracts/results/$TXID" &
  # Variant: add ?hbar=false for a third distinct cache key
  curl -s "https://<mirror-node>/api/v1/contracts/results/$TXID?hbar=false" &
done
wait
```

Each iteration fires up to 3 cache-miss requests per timestamp, each triggering 5 parallel DB queries (15 DB queries per timestamp). With 100 timestamps and no rate limiting, this is 1,500 DB queries in a single pass, repeatable every 600 seconds indefinitely.
