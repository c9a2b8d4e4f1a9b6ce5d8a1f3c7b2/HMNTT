Audit Report

## Title
Partial Content (206) Response Incorrectly Cached in Redis, Serving Stale Incomplete Data to All Subsequent Clients

## Summary
`httpStatusCodes.isSuccess()` in `rest/constants.js` returns `true` for any 2xx status code including 206. The `responseCacheUpdateHandler` middleware has no exclusion for 206 responses. When a transient partial-data condition is triggered (e.g., `callResult` is nil for a contract result), any unauthenticated user who queries the affected endpoint during that window causes the incomplete 206 response to be stored in Redis under a stable URL-derived cache key with a TTL up to 600 seconds, blocking all subsequent clients from receiving the complete 200 response even after the mirror node finishes ingesting the full data.

## Finding Description

**Root cause — `isSuccess()` is too broad:**

`rest/constants.js` line 159 defines:
```js
isSuccess: (code) => code >= 200 && code < 300,
```
This returns `true` for 206 `PARTIAL_CONTENT`. [1](#0-0) 

**Cache update gate — no 206 exclusion:**

`rest/middleware/responseCacheHandler.js` line 95:
```js
if (responseBody && responseCacheKey && (isUnmodified || httpStatusCodes.isSuccess(res.statusCode))) {
```
The only guards are: non-empty body, cache key present, and `isSuccess` or 304. There is no check to exclude 206. [2](#0-1) 

**206 is triggered by deterministic data conditions, not operator action:**

- `rest/controllers/contractController.js` lines 1026–1029: sets 206 when `contractResults[0].callResult` is nil. [3](#0-2) 

- `rest/controllers/contractController.js` lines 1198–1201: sets 206 when `contractResult.callResult` is nil for `getContractResultsByTransactionIdOrHash`. [4](#0-3) 

- `rest/tokens.js` lines 1048–1051: sets 206 when `nft.createdTimestamp == null` in NFT transfer history. [5](#0-4) 

These are transient states during normal mirror node data ingestion — no privilege is required to encounter them.

**Cache key is stable and TTL is long:**

The key is purely URL-derived (MD5 of `req.originalUrl`): [6](#0-5) 

The `cache-control` header for `/api/v1/contracts/{id}/results/{id}` is `public, max-age=600`: [7](#0-6) 

**Stored status code is 206:**

`res.statusCode` (206) is stored directly into `CachedApiResponse`: [8](#0-7) 

On cache hit (`responseCacheCheckHandler` line 54), `cachedResponse.statusCode` (206) is replayed to all subsequent clients: [9](#0-8) 

## Impact Explanation
All clients querying the affected endpoint during the 600-second cache window receive incomplete contract result data (e.g., null `gas_used`, empty `call_result`, missing state changes). Smart contract developers, block explorers, and automated systems relying on this data will observe incorrect results. For NFT transfer history endpoints the same applies when `createdTimestamp` is null. The impact is data integrity corruption at the API layer, affecting all consumers of the cached key simultaneously for up to 600 seconds per occurrence.

## Likelihood Explanation
No privileges are required. The attacker only needs to send a standard GET request at the right moment — immediately after a contract transaction is submitted. Since contract transactions are publicly visible on the Hedera network, an attacker can automate this: watch the network for new transactions, immediately query the mirror node REST API, and reliably hit the transient partial-data window. The attack is repeatable for every new contract transaction and requires no authentication, no special headers, and no rate-limit bypass.

## Recommendation
Exclude 206 responses from being cached in `responseCacheUpdateHandler`. The simplest fix is to change the condition at `rest/middleware/responseCacheHandler.js` line 95 to explicitly exclude `PARTIAL_CONTENT`:

```js
if (
  responseBody &&
  responseCacheKey &&
  (isUnmodified || (httpStatusCodes.isSuccess(res.statusCode) && res.statusCode !== httpStatusCodes.PARTIAL_CONTENT.code))
) {
```

Alternatively, change `isSuccess` to only return `true` for exactly 200 and 201 (or whatever codes are appropriate for caching), or introduce a separate `isCacheable(code)` helper that explicitly excludes 206.

## Proof of Concept
1. Observe a new contract transaction on-chain (public information on Hedera).
2. Immediately send `GET /api/v1/contracts/{contractId}/results/{timestamp}` before the mirror node has finished ingesting the full record file.
3. `contractResults[0].callResult` is nil → `res.locals.statusCode = 206` is set.
4. `responseHandler` sets `cache-control: public, max-age=600` and sends the body.
5. `responseCacheUpdateHandler` runs: `isSuccess(206)` → `true`, TTL=600 → `getCache().setSingle(key, 600, cachedResponse206)`.
6. For the next 600 seconds, `responseCacheCheckHandler` serves the stale 206 with `call_result: "0x"`, `gas_used: null`, etc. to every client querying the same URL.
7. After the mirror node completes ingestion and would return 200 with full data, the cache still serves 206 until TTL expires.
