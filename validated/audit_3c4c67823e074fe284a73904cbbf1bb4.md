Audit Report

## Title
Unbounded Redundant Work in `FindBetween()` Enables Resource Exhaustion via Repeated `/block` Requests

## Summary
`FindBetween()` in `rosetta/app/persistence/transaction.go` performs a full multi-batch database query and in-memory processing pipeline on every invocation with no result cache. The `/block` HTTP endpoint calls `FindBetween()` unconditionally on every request. The application-level middleware stack contains no rate limiting, and the Traefik-based rate limiting in the Helm chart is disabled by default (`global.middleware: false`), leaving any unprivileged caller free to drive repeated full-cost re-executions for the same block.

## Finding Description
**Verified code path:**

`POST /block` → `blockAPIService.Block()` at `block_service.go:47-74` → `s.FindBetween(ctx, block.ConsensusStartNanos, block.ConsensusEndNanos)` at `base_service.go:54-60` → `transactionRepository.FindBetween()` at `transaction.go:112-171`.

**Root cause — no result cache in `FindBetween()`:**

Every call to `FindBetween()` unconditionally:

1. Opens a DB connection and executes `selectTransactionsInTimestampRangeOrdered` in a loop with `LIMIT 2000` per iteration until all rows are consumed. [1](#0-0) 

2. Builds an in-memory `sameHashMap` and `hashes` slice over all returned rows. [2](#0-1) 

3. Calls `constructTransaction()` for every unique hash. [3](#0-2) 

There is no memoization, no TTL cache, and no deduplication of identical `(start, end)` pairs across concurrent or sequential requests. The only guard is the trivial `start > end` check at line 116–118, which does not limit repeated identical calls. [4](#0-3) 

**`entityCache` does not help:** The LRU cache in `blockAPIService` caches only entity-ID-to-alias mappings used in `updateOperationAccountAlias()`, not block transaction results. [5](#0-4) [6](#0-5) 

**`FindBetween()` is called unconditionally on every `/block` request:** [7](#0-6) 

**Rate limiting is disabled by default:** The Traefik middleware chain (including `rateLimit: average: 10` and `inFlightReq: amount: 5`) is rendered only when `global.middleware: true`. [8](#0-7) 

The default value is `global.middleware: false`. [9](#0-8) 

The application's own middleware stack contains only `MetricsMiddleware`, `TracingMiddleware`, and `CorsMiddleware` — no rate limiting. [10](#0-9) 

## Impact Explanation
An attacker targeting a block with a large transaction count (requiring multiple 2000-row DB batches) can force the node to re-execute the full query-and-processing pipeline on every request. Because the Rosetta `/block` endpoint is a public, unauthenticated POST endpoint, there is no credential barrier. Sustained parallel requests for the same high-transaction block will saturate DB connection pool slots, increase PostgreSQL CPU and I/O, and consume Go goroutine and memory resources for JSON unmarshaling and map construction. This can raise node resource consumption significantly above baseline without any brute-force volume — a modest rate of concurrent requests against a large block is sufficient. [11](#0-10) 

## Likelihood Explanation
The exploit requires no credentials, no special knowledge beyond a valid block index or hash (publicly observable on-chain), and no tooling beyond a standard HTTP client. The attacker can identify a historically large block and issue concurrent POST requests to `/block` with that block's identifier. The attack is repeatable indefinitely. Default deployments without the Traefik middleware enabled are fully exposed, as `global.middleware: false` is the shipped default. [9](#0-8) 

## Recommendation
1. **Add an application-level result cache** in `FindBetween()` or at the `blockAPIService.Block()` layer, keyed on `(consensusStartNanos, consensusEndNanos)`, with a short TTL (e.g., 30–60 seconds). This eliminates redundant DB work for repeated identical requests. [12](#0-11) 

2. **Enable rate limiting by default** by changing `global.middleware` to `true` in the Helm chart defaults, or add an application-level rate limiter in the middleware chain in `rosetta/main.go`. [9](#0-8) [10](#0-9) 

3. **Add a DB-level query timeout** per request to bound the maximum cost of a single `FindBetween()` call.

## Proof of Concept
```bash
# Identify a block with many transactions (e.g., block index 12345678)
# Issue concurrent POST requests to /block

for i in $(seq 1 50); do
  curl -s -X POST http://<rosetta-host>/block \
    -H "Content-Type: application/json" \
    -d '{"network_identifier":{"blockchain":"Hedera","network":"mainnet"},
         "block_identifier":{"index":12345678}}' &
done
wait
```

Each of the 50 concurrent requests independently triggers the full `FindBetween()` pipeline — multiple `SELECT` batches of 2000 rows each against `transaction`, `crypto_transfer`, and `staking_reward_transfer`, followed by in-memory map construction and `constructTransaction()` calls — with no shared result and no rate gate in a default deployment. func (tr *transactionRepository) FindBetween(ctx context.Context, start, end int64) (
	[]*types.Transaction,
	*rTypes.Error,
) {
	if start > end {
		return nil, hErrors.ErrStartMustNotBeAfterEnd
	}

	db, cancel := tr.dbClient.GetDbWithContext(ctx)
	defer cancel()

	transactions := make([]*transaction, 0)
	for start <= end {
		transactionsBatch := make([]*transaction, 0)
		err := db.
			Raw(
				selectTransactionsInTimestampRangeOrdered,
				sql.Named("start", start),
				sql.Named("end", end),
			).
			Limit(batchSize).
			Find(&transactionsBatch).
			Error
		if err != nil {
			log.Errorf(databaseErrorFormat, hErrors.ErrDatabaseError.Message, err)
			return nil, hErrors.ErrDatabaseError
		}

		transactions = append(transactions, transactionsBatch...)

		if len(transactionsBatch) < batchSize {
			break
		}

		start = transactionsBatch[len(transactionsBatch)-1].ConsensusTimestamp + 1
	}

	hashes := make([]string, 0)
	sameHashMap := make(map[string][]*transaction)
	for _, t := range transactions {
		h := t.getHashString()
		if _, ok := sameHashMap[h]; !ok {
			// save the unique hashes in chronological order
			hashes = append(hashes, h)
		}

		sameHashMap[h] = append(sameHashMap[h], t)
	}

	res := make([]*types.Transaction, 0, len(sameHashMap))
	for _, hash := range hashes {
		sameHashTransactions := sameHashMap[hash]
		transaction, err := tr.constructTransaction(sameHashTransactions)
		if err != nil {
			return nil, err
		}
		res = append(res, transaction)
	}
	return res, nil
}
