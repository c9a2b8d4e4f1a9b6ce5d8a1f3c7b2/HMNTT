# Audit Report

## Title
Unauthenticated High-Frequency Flooding of `/api/v1/network/stake` Causes Repeated SQL Subquery Execution and Database CPU Exhaustion

## Summary
The `GET /api/v1/network/stake` endpoint in the `rest-java` module is publicly accessible with no rate limiting, no per-IP throttling, and no result caching. Every request triggers `NetworkStakeRepository.findLatest()`, which executes a native SQL query containing a correlated subquery against the live database. An unprivileged attacker can flood this endpoint to exhaust database CPU and connection pool resources.

## Finding Description

**Verified code path:**

`NetworkController.getNetworkStake()` (lines 126–130) calls `networkService.getLatestNetworkStake()` with no throttle guard: [1](#0-0) 

`NetworkServiceImpl.getLatestNetworkStake()` (lines 52–55) directly delegates to the repository with no `@Cacheable` annotation: [2](#0-1) 

`NetworkStakeRepository.findLatest()` executes the following native SQL on every invocation:
```sql
select *
from network_stake
where consensus_timestamp = (
    select max(consensus_timestamp) from network_stake
)
``` [3](#0-2) 

**Three compounding failures confirmed:**

1. **No rate limiting in `rest-java`**: The `ThrottleManager` infrastructure exists only in the `web3` module and is wired only into `ContractController`. `NetworkController` has no throttle dependency whatsoever. [4](#0-3) 

2. **No result caching**: Neither `NetworkServiceImpl.getLatestNetworkStake()` nor `NetworkStakeRepository.findLatest()` carries a `@Cacheable` annotation. By contrast, the `grpc` module's `NodeStakeRepository.findAllStakeByConsensusTimestamp()` uses `@Cacheable` for an equivalent pattern: [5](#0-4) 

3. **`ShallowEtagHeaderFilter` does not protect the database**: `RestJavaConfiguration` registers a `ShallowEtagHeaderFilter` for `/api/*`. This filter buffers the full response body to compute an ETag *after* the controller and repository have already executed — meaning the database query runs unconditionally on every request before any ETag comparison can occur. [6](#0-5) 

## Impact Explanation
The `network_stake` table is updated only once per staking period (approximately daily), yet the correlated `max()` subquery re-executes on every HTTP request. Under a sustained flood, the database connection pool is saturated and CPU spikes, degrading all API endpoints that share the same PostgreSQL instance — not just `/network/stake`. This is a medium-severity availability issue (griefing/DoS).

## Likelihood Explanation
The attack requires zero privileges, zero authentication, and zero knowledge beyond the public OpenAPI specification. Any attacker with a basic HTTP flood tool (`wrk`, `ab`, `k6`) can reproduce this. There is no IP-based or global rate limiting in the `rest-java` stack at the application layer.

## Recommendation
1. **Add caching**: Annotate `NetworkServiceImpl.getLatestNetworkStake()` with `@Cacheable` using a TTL appropriate for the staking period (e.g., 1 hour), following the same pattern already used in `grpc`'s `NodeStakeRepository`.
2. **Add rate limiting**: Introduce a rate-limiting filter or interceptor in `rest-java` analogous to the `ThrottleManager` in `web3`, applied globally or at least to high-cost endpoints.
3. **Rewrite the query**: Replace the correlated subquery with `ORDER BY consensus_timestamp DESC LIMIT 1` to reduce per-request database work even before caching is applied.

## Proof of Concept
```bash
# Flood the endpoint at high frequency (no auth required)
wrk -t 10 -c 100 -d 60s http://<mirror-node-host>/api/v1/network/stake
```
Each request unconditionally executes:
```sql
select * from network_stake
where consensus_timestamp = (select max(consensus_timestamp) from network_stake)
```
with no cache hit possible and no rate limit enforced, directly against the live PostgreSQL instance.
