Audit Report

## Title
Unprivileged gRPC Subscriber Can Reset `lastConsensusTimestamp` to Skip Unprocessed Topic Messages Permanently

## Summary
`SharedPollingTopicListener.PollingContext.onStart()` unconditionally overwrites `lastConsensusTimestamp` with `DomainUtils.now()` (current wall-clock nanoseconds) every time the shared Flux is re-subscribed. Because the `topicMessages` Flux uses Reactor's `.share()` (publish/refCount), any subscriber who arrives when the active-subscriber count is zero triggers a fresh upstream subscription and therefore a fresh `onStart` call. If the new wall-clock value is ahead of a message that has achieved consensus on the network but has not yet been written to the database by the importer, that message is permanently excluded from every subsequent `findLatest` poll, which uses a strict `>` predicate on `lastConsensusTimestamp`.

## Finding Description

**Exact code path**

`onStart` unconditionally resets the shared cursor to wall-clock time on every upstream re-subscription: [1](#0-0) 

`DomainUtils.now()` returns the current wall-clock instant converted to nanoseconds: [2](#0-1) 

`findLatest` issues a strict `>` predicate against `lastConsensusTimestamp`, permanently excluding any message whose `consensus_timestamp` is at or below the cursor: [3](#0-2) 

`.doOnSubscribe(context::onStart)` is placed upstream of `.share()`, so it fires on every 0→1 subscriber-count transition: [4](#0-3) 

The single `PollingContext` instance (and its `lastConsensusTimestamp` field) is created once in the constructor and shared across all re-subscriptions: [5](#0-4) 

**Root cause and failed assumption**

The design assumes `onStart` is called exactly once at application startup, making `DomainUtils.now()` a safe initial cursor (no live messages exist before "now"). The failed assumption is that `.share()` guarantees a single upstream subscription for the lifetime of the process. In reality, `.share()` (refCount semantics) cancels the upstream when the last subscriber leaves and re-subscribes when the next subscriber arrives. Because `lastConsensusTimestamp` belongs to the single shared `PollingContext`, each re-subscription overwrites the cursor with the current wall-clock time, which may be ahead of messages that have achieved consensus on the network but have not yet been ingested by the importer.

**Why existing checks are insufficient**

The `missingMessages` gap-filling logic in `TopicMessageServiceImpl` detects gaps between *consecutively received* messages by comparing sequence numbers: [6](#0-5) 

It cannot recover a message that was skipped *before* the listener's cursor was established, because there is no "previous message" to compare against. The one-second safety-check path calls `missingMessages(topicContext, null)`, which queries from the *subscriber's* `startTime` forward through the retriever: [7](#0-6) 

This safety check fires only once, one second after subscription. If the importer ingests the skipped message after that one-second window, neither the safety check nor the listener will ever return it. The listener's internal cursor is opaque to the service layer and is already set past the message's timestamp.

## Impact Explanation

All gRPC subscribers sharing the single `PollingContext` permanently miss the skipped message. Because `lastConsensusTimestamp` only ever moves forward (updated by `onNext` to the latest delivered message's timestamp), there is no self-healing path: the skipped message's timestamp is forever below the cursor. Any application relying on receiving this message — for example, one monitoring an HCS topic for coordination instructions — will never receive it for the lifetime of the process (or until the next re-subscription resets the cursor even further forward). The impact is process-lifetime-persistent and affects all current and future subscribers sharing the single `PollingContext`. [8](#0-7) 

## Likelihood Explanation

The attack requires no credentials. The gRPC `subscribeTopic` endpoint is public and unauthenticated. The attacker needs only to subscribe when the active-subscriber count is zero — a realistic condition during low-traffic periods, after server restarts, or after all previous subscribers have timed out. The importer ingestion lag (typically seconds to minutes) provides a wide timing window between a message achieving consensus on the network and being written to the database. The attack is repeatable: each time the subscriber count drops to zero and the attacker re-subscribes, the cursor is reset further forward. [9](#0-8) 

## Recommendation

`onStart` should only initialize `lastConsensusTimestamp` if it has not already been set (i.e., on the very first subscription). A simple guard suffices:

```java
void onStart(Subscription subscription) {
    lastConsensusTimestamp.compareAndSet(0L, DomainUtils.now());
    log.info("Starting to poll every {}ms",
             listenerProperties.getInterval().toMillis());
}
```

`AtomicLong` is initialized to `0L` by default, so `compareAndSet(0L, DomainUtils.now())` sets the cursor only on the first call and is a no-op on all subsequent re-subscriptions, preserving the cursor value that was accumulated before the last subscriber disconnected.

Alternatively, the cursor could be initialized to `0L` and left at `0L` on re-subscription, relying on the existing `onNext` updates to advance it. This would cause a brief re-scan of already-delivered messages on re-subscription, but the service-layer deduplication (sequence-number filter) would suppress duplicates. [10](#0-9) 

## Proof of Concept

1. Start the mirror node with `SHARED_POLL` listener type.
2. Subscribe one gRPC client to any topic; confirm it receives messages. Then cancel that subscription so the subscriber count drops to zero. The `.share()` operator cancels the upstream.
3. Observe (via Hedera network APIs or a block explorer) that a topic message `M` has achieved consensus at timestamp `T_freeze`.
4. Before the importer writes `M` to the `topic_message` table, subscribe a new gRPC client. This is the first subscription (count 0→1), so Reactor re-subscribes the upstream and calls `onStart`, setting `lastConsensusTimestamp = DomainUtils.now() = T_now > T_freeze`.
5. Allow the importer to ingest `M` (it will write a row with `consensus_timestamp = T_freeze`).
6. Observe that every subsequent `findLatest(T_now, pageSize)` call issues `WHERE consensus_timestamp > T_now`, which permanently excludes `T_freeze < T_now`.
7. Confirm that `M` is never delivered to any subscriber — current or future — for the lifetime of the process.

The one-second safety check does not help if `M` is ingested after the one-second window, or if the subscriber's `startTime` is set to "now" (the default), because the retriever will complete immediately with no results before `M` is available. [11](#0-10)
