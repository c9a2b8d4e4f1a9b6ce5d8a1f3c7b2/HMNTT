Audit Report

## Title
Unbounded DB Query Amplification via EXTCODESIZE Loop in `eth_call` — Single Unauthenticated Request Triggers Thousands of `findRuntimeBytecode()` Queries

## Summary
A single unauthenticated `POST /api/v1/contracts/call` request containing EVM bytecode that loops `EXTCODESIZE` over thousands of unique contract addresses can generate thousands of live SQL queries against the database. The two-layer cache (per-request `HashMap` + cross-request Caffeine) only deduplicates repeated lookups for the **same** address; it provides no protection when the attacker uses a large set of **distinct** addresses. No authentication, no on-chain state, and no special privileges are required.

## Finding Description

**Confirmed code path:**

1. `ContractController.call()` receives the request, validates gas against `maxGasLimit = 15,000,000`, and calls `contractExecutionService.processCall()`. [1](#0-0) [2](#0-1) 

2. During EVM execution, each `EXTCODESIZE` opcode triggers `ReadableKVStateBase.get()`. The per-request cache is a `HashMap` stored in `ContractCallContext` (a `ScopedValue`, i.e., scoped to a single request and empty at the start of every call). For each unique `ContractID` key not yet seen in this request, `readFromDataSource()` is called. [3](#0-2) [4](#0-3) [5](#0-4) 

3. `ContractBytecodeReadableKVState.readFromDataSource()` calls `contractRepository.findRuntimeBytecode(entityId.getId())` for every cache miss. [6](#0-5) 

4. `findRuntimeBytecode()` is backed by a Caffeine cache with `maximumSize=1000`. For any address not already in this 1,000-entry cache, the annotated method executes `SELECT runtime_bytecode FROM contract WHERE id = :contractId`. [7](#0-6) [8](#0-7) 

**Root cause:** There is no limit on the number of distinct DB queries a single `eth_call` execution can generate. The gas limit bounds EVM opcode count, not DB round-trips. The two-layer cache only deduplicates repeated lookups for the **same** address; it provides no protection when the attacker uses a large set of **distinct** addresses.

**Why `unless = "#result == null"` does not fully mitigate:** The return type is `Optional<byte[]>`. `Optional.empty()` is not `null`, so the `unless` condition evaluates to `false` and the empty result is cached — but only up to the 1,000-entry limit. Once the cache is full, Caffeine evicts entries and subsequent lookups for evicted addresses hit the DB again. [7](#0-6) 

## Impact Explanation

With `maxGasLimit = 15,000,000` and `EXTCODESIZE` costing 700 gas (pre-EIP-2929) or 2,600 gas (EIP-2929 cold access), a single request can issue between ~5,769 and ~21,428 `EXTCODESIZE` calls against unique addresses. The Caffeine cache absorbs the first 1,000 unique addresses; the remaining thousands hit the database directly.

The gas throttle (`gasPerSecond = 7,500,000,000`) does not prevent this: 15M gas is only 0.2% of the per-second gas budget, so the throttle does not reject the request. [9](#0-8) 

The 10-second request timeout (`requestTimeout = 10000`) is the only practical upper bound on execution, but thousands of simple indexed lookups easily fit within it. [10](#0-9) 

This constitutes a severe DB-layer amplification attack (1 HTTP request → up to ~21,000 DB queries) that can exhaust database connection pools and degrade service for all users.

## Likelihood Explanation

No authentication, no special account, and no on-chain state is required. The attacker only needs access to the public `POST /api/v1/contracts/call` endpoint. The attack is trivially repeatable: the attacker can send one request per second (limited by the gas bucket refill), each generating thousands of DB queries. Sequential long-zero addresses (e.g., `0x0000…0001` through `0x0000…5265`) are valid inputs and are not filtered. The attack is stateless and requires no prior contract deployment. [1](#0-0) 

## Recommendation

1. **Introduce a per-request DB query budget**: Track the number of `readFromDataSource()` calls within a single `ContractCallContext` and abort execution (or return a synthetic empty result) once a configurable threshold (e.g., 1,000) is exceeded.

2. **Increase the Caffeine cache size**: Raise `maximumSize` for the contract cache well above the maximum number of unique addresses reachable in a single call (e.g., to 25,000, consistent with `contractState` cache sizing). [8](#0-7) 

3. **Lower `maxGasLimit`**: Reducing the maximum gas limit per request directly reduces the maximum number of `EXTCODESIZE` calls and thus the maximum DB amplification factor. [2](#0-1) 

4. **Gas throttle tightening**: Reduce `gasPerSecond` or add a per-IP rate limit to make repeated attacks more costly. [9](#0-8) 

## Proof of Concept

Construct EVM bytecode that:
1. Initializes a counter `i = 0`.
2. In a loop: pushes address `0x0000…0000 + i` onto the stack, executes `EXTCODESIZE` (opcode `0x3B`), pops the result, increments `i`.
3. Repeats until gas is exhausted.

Submit via:
```
POST /api/v1/contracts/call
{
  "data": "<loop_bytecode_hex>",
  "gas": 15000000,
  "estimate": false
}
```

Each iteration of the loop with a unique address causes one call to `ContractBytecodeReadableKVState.readFromDataSource()` → `ContractRepository.findRuntimeBytecode()` → one SQL query `SELECT runtime_bytecode FROM contract WHERE id = :contractId`. With 15M gas and ~700–2600 gas per `EXTCODESIZE`, this generates between ~5,769 and ~21,428 SQL queries from a single HTTP request. [6](#0-5) [7](#0-6)
