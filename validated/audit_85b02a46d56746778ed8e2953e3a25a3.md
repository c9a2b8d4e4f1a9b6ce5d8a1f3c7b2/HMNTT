Audit Report

## Title
Pinger Persistent DoS via Empty `nodes` Array from Mirror REST Endpoint (Retry Bypass + Fatal Exit)

## Summary
When `network=other`, a single HTTP 200 response containing an empty `nodes` array causes `fetchMirrorNodeNetwork` to return `retry=false`, which immediately breaks the retry loop in `buildNetworkFromMirrorNodes`, propagates an error through `newClient`, and terminates the process via `log.Fatalf`. An attacker able to serve one poisoned response at startup can keep the pinger in a permanent crash loop.

## Finding Description

**`pinger/mirror_node_client.go` lines 127–129** — empty network map returns the non-retryable sentinel: [1](#0-0) 

**`pinger/mirror_node_client.go` lines 59–61** — the retry loop checks the flag before sleeping; `retry=false` causes an immediate break regardless of `mirrorNodeClientMaxRetries` (default 10): [2](#0-1) 

**`pinger/sdk_client.go` lines 18–21** — error propagates unchanged: [3](#0-2) 

**`pinger/main.go` lines 41–44** — `log.Fatalf` calls `os.Exit(1)`: [4](#0-3) 

**Root cause:** The code classifies "no usable endpoints in response" as a permanent, non-retryable failure. This is incorrect — an empty-nodes response is indistinguishable from a transient or adversarial response, yet it silently discards the entire retry budget and terminates the process.

## Impact Explanation

- The pinger's monitoring function is completely disabled; no transfers are submitted.
- `/tmp/ready` is never written (line 47 is never reached), so the readiness probe permanently fails. [5](#0-4) 
- The liveness heartbeat goroutine never runs long enough to write `/tmp/alive`.
- Kubernetes restarts the pod; the attacker serves the same response again → persistent crash loop with no self-healing.

Severity: **Medium** (complete availability loss of the pinger component; no direct economic impact on network users).

## Likelihood Explanation

Only reachable when `HIERO_MIRROR_PINGER_NETWORK=other` (custom deployment mode). [6](#0-5) 

The `mirrorRest` URL is operator-configured and may use plain `http://`: [7](#0-6) 

Realistic attacker positions:
- **DNS hijacking** of the mirror REST hostname — no local network access required.
- **Network MITM** (ARP/BGP poisoning on the same L2 segment) against an unencrypted `http://` URL.
- **Compromised mirror node** — the REST service itself serves the empty response.

A single poisoned response at startup is sufficient; the attacker does not need to sustain the attack beyond the pod's startup window.

## Recommendation

Change the classification of an empty-nodes response from non-retryable to retryable. In `fetchMirrorNodeNetwork`, return `retry=true` instead of `retry=false` when `len(network) == 0`:

```go
// pinger/mirror_node_client.go line 128
return nil, true, fmt.Errorf("no usable service_endpoints found from %s", url)
```

This ensures the full retry budget (`mirrorNodeClientMaxRetries`, default 10) is consumed before the process gives up. Additionally, consider whether `log.Fatalf` is the right terminal action, or whether the process should enter a retry loop at a higher level (e.g., in `main`) rather than exiting immediately.

## Proof of Concept

1. Set `HIERO_MIRROR_PINGER_NETWORK=other` and point `HIERO_MIRROR_PINGER_REST` at an attacker-controlled HTTP server.
2. Serve the following response for `GET /api/v1/network/nodes`:
   ```json
   {"nodes":[],"links":{"next":null}}
   ```
3. Start the pinger. `fetchMirrorNodeNetwork` decodes successfully, finds `len(network)==0`, returns `(nil, false, error)`.
4. The retry loop at line 59 evaluates `!retry` as `true` and breaks after the first attempt.
5. `buildNetworkFromMirrorNodes` returns the error; `newClient` returns it; `log.Fatalf` calls `os.Exit(1)`.
6. `/tmp/ready` is never created; the readiness probe fails; Kubernetes restarts the pod.
7. Repeat from step 3 — persistent crash loop.
