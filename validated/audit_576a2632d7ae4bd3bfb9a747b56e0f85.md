Audit Report

## Title
Unbounded `CaffeineCacheManager` Internal Map Growth via Unique `contractId` Flooding in `findStorageBatch`

## Summary
`ContractStateServiceImpl.findStorageBatch()` calls `cacheManagerSlotsPerContract.getCache(contractId.toString())` for every cache-miss on a new `contractId`. The `cacheManagerSlotsPerContract` bean is a `CaffeineCacheManager` configured without `setCacheNames()`, placing it in dynamic mode where each `getCache(name)` call creates a new `CaffeineCache` and stores it permanently in an internal unbounded `ConcurrentHashMap<String, Cache>`. The outer `contractSlotsCache` (bounded at 3000 entries) evicts entries over time, but the inner manager's map never removes them, creating a permanent memory leak exploitable via the public `/api/v1/contracts/call` endpoint.

## Finding Description

**Confirmed code path:**

`EvmConfiguration.java` lines 107–112 — `cacheManagerSlotsPerContract` is created with no `setCacheNames()` call, unlike every other `CaffeineCacheManager` bean in the same file (all of which call `setCacheNames()`), enabling dynamic (unbounded) cache registration: [1](#0-0) 

Compare with a correctly bounded peer bean: [2](#0-1) 

`ContractStateServiceImpl.java` lines 86–87 — on every outer-cache miss for a new `contractId`, `getCache()` is called on this dynamic manager, passing `contractId.toString()` as the cache name: [3](#0-2) 

**Root cause:** Spring's `CaffeineCacheManager.getCache(name)` in dynamic mode stores the newly created `CaffeineCache` in an internal `ConcurrentHashMap<String, Cache> cacheMap`. This map has no eviction, no size bound, and no TTL. The outer `contractSlotsCache` is configured `expireAfterAccess=5m,maximumSize=3000`: [4](#0-3) 

Each inner `CaffeineCache` is configured `expireAfterAccess=5m,maximumSize=1500`: [5](#0-4) 

When the outer cache evicts a `contractId` entry after 5 minutes of inactivity, the corresponding `CaffeineCache` object in `cacheManagerSlotsPerContract`'s internal `cacheMap` is **not** removed. The next request for that same `contractId` calls `getCache()` again, but since the name already exists in the inner map, it returns the existing (now-empty) instance — it does not create a duplicate. However, every **new** unique `contractId` ever seen accumulates permanently. An attacker cycling through N unique `contractId` values causes N permanent entries in the inner map, regardless of the outer cache's 3000-entry bound.

**Failed assumption:** The design assumes the outer `contractSlotsCache` bounds total memory usage. It does not — it only bounds the number of *active* per-contract caches. The inner manager's map is the actual long-lived store and has no bound.

## Impact Explanation

The `cacheManagerSlotsPerContract` internal map grows proportionally to the number of unique `contractId` values ever seen, holding strong references to `CaffeineCache` objects that cannot be garbage collected. Each `CaffeineCache` with `maximumSize=1500` carries non-trivial fixed overhead (Caffeine's frequency sketch, eviction arrays, internal node arrays) even when empty — on the order of tens of KB per instance. At scale this causes severe GC pressure (frequent full GCs stalling all threads) and ultimately `OutOfMemoryError`, crashing the web3 service and making `eth_call` / `eth_estimateGas` completely unavailable.

## Likelihood Explanation

The attack requires no authentication, no on-chain funds, and no special knowledge beyond the public JSON-RPC API. The attacker only needs to cycle through different `to` addresses in `eth_call` requests. Hedera entity IDs are sequential integers, making enumeration trivial. The global rate limiter is a single shared bucket — not per-IP: [6](#0-5) 

There is no per-IP throttle, no validation that `contractId` refers to an existing contract before entering `findStorageBatch`, and no cleanup path for the inner manager's map. The leaked objects are never reclaimed across the lifetime of the process.

## Recommendation

Fix the `cacheManagerSlotsPerContract` bean by removing dynamic mode. Since the per-contract caches are created programmatically at runtime (not by name at startup), the correct fix is to **not use `CaffeineCacheManager` as a factory** for per-contract caches at all. Instead, maintain a bounded `LoadingCache<EntityId, com.github.benmanes.caffeine.cache.Cache<ByteBuffer, byte[]>>` directly in `ContractStateServiceImpl`, configured with the same `maximumSize=3000` and `expireAfterAccess=5m` as the outer cache. This collapses the two-level structure into one, eliminates the unbounded inner map, and ensures eviction of the outer entry also releases the inner cache.

If the two-level structure must be kept, at minimum call `setCacheNames(Set.of())` on the `CaffeineCacheManager` and implement a `RemovalListener` on the outer Caffeine cache that calls an explicit cleanup method to remove the evicted name from the inner manager's map. [1](#0-0) 

## Proof of Concept

```python
import requests, threading

TARGET = "http://<mirror-node>/api/v1/contracts/call"

def flood(start, count):
    for i in range(start, start + count):
        # Each unique 'to' address maps to a different contractId
        addr = "0x" + hex(i)[2:].zfill(40)
        requests.post(TARGET, json={
            "data": "0x",
            "to": addr,
            "estimate": False
        })

# Spawn threads to exhaust the 500 req/s global bucket with unique contractIds
threads = [threading.Thread(target=flood, args=(i*500, 500)) for i in range(20)]
for t in threads: t.start()
for t in threads: t.join()
# After ~6 seconds the outer cache (3000 entries) is full.
# Every subsequent unique contractId adds a permanent CaffeineCache to the inner map.
# Monitor JVM heap: growth is unbounded.
```

Each iteration adds one permanent `CaffeineCache` entry to `cacheManagerSlotsPerContract`'s internal map. After the outer cache saturates at 3000 entries, net-new permanent entries accumulate at the rate of unique `contractId` values submitted. Heap growth is observable via JMX/`jmap` by watching the count of `com.github.benmanes.caffeine.cache.BoundedLocalCache` instances.
