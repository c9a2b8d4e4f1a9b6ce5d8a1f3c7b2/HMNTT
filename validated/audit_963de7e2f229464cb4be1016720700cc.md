# Audit Report

## Title
Unbounded Historical Timestamp Scan DoS on Hook Storage Change Endpoint

## Summary
An unauthenticated caller can supply only an upper-bound timestamp (e.g., `timestamp=lte:9223372036.854775807`) to `GET /api/v1/accounts/{ownerId}/hooks/{hookId}/storage`, causing the mirror node to execute a full cross-partition scan of the `hook_storage_change` table with `consensus_timestamp BETWEEN 0 AND Long.MAX_VALUE`. The `limit` cap does not bound the amount of data the database must read to satisfy `DISTINCT ON (key)`. Repeated requests can saturate database I/O and degrade service for all users.

## Finding Description

**Step 1 — Controller builds the timestamp `Bound`:**

`HooksController.java` line 186 calls `Bound.of(timestamps, TIMESTAMP, ...)`: [1](#0-0) 

**Step 2 — `Bound.of()` passes `primarySortField=false`, disabling range validation:**

`Bound.java` lines 169–182 construct the `Bound` with `false` for `primarySortField`: [2](#0-1) 

The range-validity check at line 58 is guarded by `primarySortField`: [3](#0-2) 

Because `primarySortField=false`, the check `adjustedLower > adjustedUpper` is **never evaluated** for the timestamp parameter on this endpoint.

**Step 3 — `lower` is `null`; `getAdjustedLowerRangeValue()` silently returns `0`:**

When only an upper-bound timestamp is supplied, `lower` is never set. `getAdjustedLowerRangeValue()` returns `0` unconditionally: [4](#0-3) 

**Step 4 — `adjustUpperBound()` returns `Long.MAX_VALUE` for `lte:9223372036.854775807`:**

`TimestampParameter`'s regex allows up to 17 digits for seconds and 9 for nanoseconds: [5](#0-4) 

`9223372036 * 1_000_000_000 + 854775807 = Long.MAX_VALUE`. With operator `LTE`, `adjustUpperBound()` returns the value unchanged: [6](#0-5) 

**Step 5 — Service passes `0` and `Long.MAX_VALUE` directly to the repository:**

`HookServiceImpl.java` lines 98–99: [7](#0-6) 

**Step 6 — Repository executes a full-table scan with `DISTINCT ON (key)`:**

Both query methods use `consensus_timestamp between :timestampLowerBound and :timestampUpperBound` with `DISTINCT ON (key)`: [8](#0-7) 

With bounds `0` to `Long.MAX_VALUE`, partition pruning on `consensus_timestamp` is entirely ineffective. PostgreSQL must scan every partition and merge results before applying `DISTINCT ON`.

**Step 7 — `isHistorical()` routes to the expensive path whenever any timestamp is present:**

Supplying only an upper-bound timestamp makes `Bound.isEmpty()` return `false` (since `upper != null`), so `isHistorical()` returns `true` and the code routes to `getHookStorageChange()`: [9](#0-8) 

## Impact Explanation

Any unauthenticated caller can force the mirror node to execute a full cross-partition historical scan of `hook_storage_change` for any `(owner_id, hook_id)` pair. The `limit` cap of 100 rows limits the response size but does **not** limit the amount of data the database engine must read — PostgreSQL must gather and merge all matching rows across all partitions before it can apply `DISTINCT ON (key)` and return the top-N. Repeated requests from one or more clients can saturate database I/O, degrade query performance for all other users, and exhaust connection pool resources.

## Likelihood Explanation

The endpoint requires no authentication or authorization. The exploit requires a single HTTP GET request with one crafted query parameter. It is trivially repeatable in a loop. The only precondition is knowing a valid `(ownerId, hookId)` pair, which is discoverable via the public `GET /api/v1/accounts/{ownerId}/hooks` endpoint.

## Recommendation

1. **Require a lower-bound timestamp when a timestamp filter is present on this endpoint.** If `lower == null` and `upper != null`, reject the request with HTTP 400.
2. **Enforce a maximum time window.** Validate that `adjustedUpper - adjustedLower` does not exceed a configurable maximum (e.g., 7 days in nanoseconds) to prevent arbitrarily wide scans even when both bounds are supplied.
3. **Enable the range-validity check for timestamp bounds.** Either pass `primarySortField=true` in `Bound.of()` for this endpoint, or add a dedicated validation step in `HookServiceImpl.getHookStorageChange()` before calling the repository.
4. **Consider rate-limiting** the `/hooks/{hookId}/storage` endpoint for unauthenticated callers.

## Proof of Concept

```
GET /api/v1/accounts/0.0.1234/hooks/1/storage?timestamp=lte:9223372036.854775807
```

This single unauthenticated request causes the database to execute:

```sql
SELECT DISTINCT ON (key) ...
FROM hook_storage_change
WHERE owner_id = 1234
  AND hook_id = 1
  AND key >= '\x0000...00'
  AND key <= '\xffff...ff'
  AND consensus_timestamp BETWEEN 0 AND 9223372036854775807
ORDER BY key, consensus_timestamp DESC
LIMIT 100
```

With `BETWEEN 0 AND 9223372036854775807`, every time partition of `hook_storage_change` for this `(owner_id, hook_id)` pair must be scanned. Issuing this request in a tight loop from one or more clients will progressively degrade database performance for all users of the mirror node.
