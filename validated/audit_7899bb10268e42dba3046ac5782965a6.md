Audit Report

## Title
Retry Loop Creates Fresh Transaction IDs on Each Attempt, Enabling Multiple On-Chain Transfers Per Tick via gRPC Error Injection

## Summary
`submitWithRetry()` in `pinger/transfer.go` calls `hiero.NewTransferTransaction()` on every loop iteration, generating a distinct transaction ID each time. An attacker with a MITM position on the gRPC channel can intercept the response leg after a transaction has already been accepted on-chain, causing `Execute()` to return an error. The function then retries with a brand-new transaction ID that the Hedera network treats as an independent transfer, resulting in multiple on-chain fund debits per scheduled tick.

## Finding Description

`submitWithRetry()` constructs a new `TransferTransaction` unconditionally on every loop iteration: [1](#0-0) 

`hiero.NewTransferTransaction()` generates a fresh transaction ID (operator account + nanosecond timestamp) on each call. `Execute(client)` signs and submits that transaction to a Hedera consensus node over gRPC. The gRPC call is two-phase: the node receives and processes the transaction, then sends back a response. If the attacker intercepts the response leg and substitutes a transient error (`UNAVAILABLE`, `DEADLINE_EXCEEDED`), `Execute()` returns a non-nil error even though the transaction was already finalized on-chain.

The error path logs the failure and sleeps for a backoff period, then the loop continues: [2](#0-1) 

On the next iteration, a new `NewTransferTransaction()` is called, producing a new transaction ID. The Hedera network has no basis to deduplicate this against the prior accepted transaction and processes it as an independent transfer.

**Failed assumption:** The code assumes a failed `Execute()` means the transaction was not submitted. This is false when the failure occurs on the response path after the node has already accepted the transaction.

**No deduplication guard exists.** The only early-exit paths are `ctx.Err()` (line 24) and a successful receipt (lines 36–39). There is no idempotency key, no transaction ID reuse across retries, and no check for a prior successful submission.

Default `maxRetries = 10`: [3](#0-2) 

giving `attempts = 11` per tick: [4](#0-3) 

## Impact Explanation

Each successful on-chain transfer debits `cfg.amountTinybar` (default 10,000 tinybar = 0.0001 HBAR) from the operator account and credits the destination. With up to 11 distinct transaction IDs per tick, an attacker sustaining the attack can cause multiple independent on-chain debits per tick. Even a single injected error per tick doubles the transfer rate. Over time this constitutes direct, repeated financial loss from the operator account with no on-chain safeguard.

**Note on rate claim:** The report's "11× per tick" figure is an overstatement. The exponential backoff (`base * 2^(attempt-1)`, capped at 30s, default base 2s) means 11 full attempts require ~210 seconds minimum, not 1 second: [5](#0-4) 

However, the attacker need not inject errors on every attempt — a single injected error on attempt 1 causes 2 independent on-chain transfers per tick, which is still a direct financial loss.

## Likelihood Explanation

The precondition is a MITM position on the gRPC channel between the pinger pod and the Hedera consensus nodes. Realistic vectors include:

- **Compromised container network / service mesh**: An attacker who has compromised another pod in the same Kubernetes namespace can intercept gRPC traffic.
- **BGP hijacking / rogue DNS**: Redirecting the consensus node address to an attacker-controlled endpoint that forwards the transaction to the real node and returns a synthetic error.
- **Compromised load balancer or ingress**: Infrastructure-level MITM without requiring application credentials.

The attack requires no knowledge of the operator private key, no Hedera account, and no privileged application access. It is repeatable on every tick.

## Recommendation

Construct the `TransferTransaction` and its transaction ID **once**, before the retry loop, and reuse the same transaction object on every attempt. This ensures that if the transaction was already accepted on-chain, any retry carries the same transaction ID and the Hedera network will deduplicate it (returning `DUPLICATE_TRANSACTION`), preventing double-spending. Move lines 29–31 to before the `for` loop:

```go
// Build once, outside the loop
cryptoTransfer := hiero.NewTransferTransaction().
    AddHbarTransfer(client.GetOperatorAccountID(), hiero.HbarFromTinybar(-cfg.amountTinybar)).
    AddHbarTransfer(toID, hiero.HbarFromTinybar(cfg.amountTinybar))

for i := 1; i <= attempts; i++ {
    // reuse cryptoTransfer — same transaction ID on every attempt
    resp, err := cryptoTransfer.Execute(client)
    ...
}
```

Additionally, treat `DUPLICATE_TRANSACTION` receipt status as a success (the transfer landed) rather than an error.

## Proof of Concept

1. Deploy the pinger against a testnet with `maxRetries=1`.
2. Place a transparent TCP proxy between the pinger and the consensus node gRPC endpoint.
3. Configure the proxy to forward the first request normally but drop/replace the response with a gRPC `UNAVAILABLE` status.
4. Observe the pinger logs: attempt 1 reports failure; attempt 2 succeeds.
5. Query the operator account transaction history on the mirror node REST API — two distinct `CryptoTransfer` transactions with different transaction IDs will appear, each debiting `amountTinybar` from the operator account.
