Audit Report

## Title
Integer Overflow in `poll()` Bypasses Page-Size Cap, Enabling Unbounded DB Query DoS

## Summary
In `PollingTopicMessageRetriever.poll()`, the expression `(int)(filter.getLimit() - context.getTotal().get())` performs an unchecked narrowing cast from `long` to `int`. When a client supplies `limit = Long.MAX_VALUE` (a valid proto `uint64`), the cast overflows to `-1`. `Math.min(-1, maxPageSize)` evaluates to `-1`, which is then written into a new `TopicMessageFilter` via a direct Lombok builder call that bypasses Spring Bean Validation. In the repository layer, `hasLimit()` is defined as `limit > 0`, so `-1` causes it to return `false` and `setMaxResults()` is never called, issuing a fully unbounded JPA query.

## Finding Description

**Step 1 — No upper-bound check on the incoming `limit`.**

`ConsensusController.toFilter()` passes `query.getLimit()` (a raw proto `uint64` mapped to Java `long`) directly to the filter builder with no cap: [1](#0-0) 

**Step 2 — Spring validation passes for `Long.MAX_VALUE`.**

`TopicMessageFilter.limit` carries `@Min(0)`. `Long.MAX_VALUE >= 0`, so the constraint is satisfied and no exception is thrown: [2](#0-1) 

**Step 3 — Narrowing cast overflows in `poll()`.**

```
(int)(Long.MAX_VALUE - 0L)
= (int)(0x7FFFFFFFFFFFFFFF)
= (int)(0xFFFFFFFF)   // lower 32 bits
= -1
Math.min(-1, maxPageSize) = -1
``` [3](#0-2) 

The same pattern exists identically in `PollingTopicListener.poll()`: [4](#0-3) 

**Step 4 — `toBuilder().limit(-1).build()` bypasses `@Min(0)`.**

The Lombok `toBuilder()` call constructs a new `TopicMessageFilter` directly, not through Spring's AOP validation proxy. The `@Min(0)` constraint on `limit` is never re-evaluated: [5](#0-4) 

**Step 5 — `hasLimit()` returns `false` for `-1`.**

`hasLimit()` is defined as `limit > 0`. With `limit = -1` this is `false`: [6](#0-5) 

**Step 6 — `setMaxResults()` is skipped; unbounded query executes.**

`findByFilter()` only calls `setMaxResults()` when `hasLimit()` is `true`. With `limit = -1`, the branch is skipped and `getResultList()` materialises the entire matching result set into a Java `List`: [7](#0-6) [8](#0-7) 

The outer `flux.take(filter.getLimit())` in `TopicMessageServiceImpl` does not prevent this because `filter.getLimit()` is still `Long.MAX_VALUE` at that layer, making `take()` effectively unlimited: [9](#0-8) 

## Impact Explanation
A single gRPC `subscribeTopic` call with `limit = Long.MAX_VALUE` causes the mirror node to issue a `SELECT *` against `topic_message` with no `LIMIT` clause. Because `getResultList()` materialises the full result set before streaming, both the database connection and the JVM heap are exhausted. This can render the gRPC service unavailable for all other subscribers. No authentication is required.

## Likelihood Explanation
The `subscribeTopic` endpoint is publicly accessible. The `limit` field in `ConsensusTopicQuery` is `uint64`; any gRPC client can trivially set it to `Long.MAX_VALUE`. The attack is deterministic, requires a single RPC call, and is trivially repeatable. The SECURITY.md exclusion for "Impacts that only require DDoS" refers to volumetric network-layer attacks; this is an application-level resource exhaustion triggered by a single request exploiting a code-level integer overflow, which is distinct from that exclusion.

## Recommendation
1. **Cap the limit at the controller layer.** In `ConsensusController.toFilter()`, clamp `query.getLimit()` to a safe maximum (e.g., `Math.min(query.getLimit(), MAX_ALLOWED_LIMIT)`) before passing it to the filter builder.
2. **Use safe narrowing in `poll()`.** Replace `(int)(filter.getLimit() - context.getTotal().get())` with `(int) Math.min(filter.getLimit() - context.getTotal().get(), Integer.MAX_VALUE)` to prevent overflow.
3. **Re-validate on `toBuilder().build()`.** Either run the validator explicitly after building, or add a guard in `hasLimit()` / `findByFilter()` to treat any negative limit as "no limit requested" and substitute `maxPageSize` instead of skipping `setMaxResults()` entirely.
