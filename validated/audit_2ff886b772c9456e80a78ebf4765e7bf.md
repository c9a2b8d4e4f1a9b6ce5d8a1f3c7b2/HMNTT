# Audit Report

## Title
Unbounded `endTime` Allows Unprivileged Users to Hold Subscriptions Open Indefinitely (DoS)

## Summary
`TopicMessageFilter` places no upper bound on the `endTime` field. An attacker can supply `endTime = Long.MAX_VALUE`, which causes `TopicContext.isComplete()` to permanently return `false`, preventing `pastEndTime()` from ever emitting a terminal signal. The reactive subscription is held open indefinitely, consuming server-side resources. Repeating this across many connections exhausts the bounded-elastic thread pool, heap, and database polling capacity.

## Finding Description

**`TopicMessageFilter.java` — missing upper-bound constraint on `endTime`:**

`endTime` is a plain nullable `Long` with only one validation constraint: it must be greater than `startTime`. There is no `@Max`, no cap relative to `Instant.now()`, and no maximum-future-offset check. `Long.MAX_VALUE` (≈ year 2262 in nanoseconds) trivially satisfies `endTime > startTime`. [1](#0-0) [2](#0-1) 

**`isComplete()` — always returns `false` with `endTime = Long.MAX_VALUE`:**

The method has three exit conditions. With `Long.MAX_VALUE`:
- `filter.getEndTime() == null` → `false`
- `filter.getEndTime() < startTime` → `false` (`Long.MAX_VALUE` exceeds any current nanosecond timestamp)
- `Instant.ofEpochSecond(0, Long.MAX_VALUE).plus(Duration.ofSeconds(30)).isBefore(Instant.now())` → `false` (year ~2262 is not before now)

Result: `isComplete()` **permanently returns `false`**. [3](#0-2) 

**`pastEndTime()` — never emits a terminal signal:**

Because `endTime != null`, the method does not return `Flux.never()`. Instead it enters a `repeatWhen` loop gated on `!topicContext.isComplete()`. Since `isComplete()` is always `false`, the predicate is always `true`, the loop repeats up to `Long.MAX_VALUE` times, and the flux never completes. `takeUntilOther(pastEndTime(topicContext))` therefore never fires. [4](#0-3) 

**`takeWhile` — never terminates:**

The `takeWhile` guard only terminates the stream when a message with `consensusTimestamp >= endTime` arrives. With `endTime = Long.MAX_VALUE`, no real message will ever satisfy this condition. [5](#0-4) 

**No subscriber cap:**

`subscriberCount` is registered solely as a Micrometer gauge for observability. There is no enforcement of a maximum subscriber count anywhere in the service. [6](#0-5) 

**`endTimeInterval` default confirmed at 30 seconds:**

Each open subscription polls for missing messages every 30 seconds, amplifying database load linearly with attacker-controlled subscription count. [7](#0-6) 

## Impact Explanation
Each persistent subscription holds a Reactor bounded-elastic scheduler thread slot, heap memory for the `TopicContext` and message buffers, and a periodic database polling task via `missingMessages`. With no subscriber cap and no connection-level rate limiting in this service, an attacker can exhaust the bounded-elastic thread pool and heap, causing the gRPC service to become unresponsive for all legitimate users. This is a resource-exhaustion DoS affecting the mirror node's HCS topic streaming capability.

## Likelihood Explanation
The `subscribeTopic` gRPC endpoint is unauthenticated by default (`checkTopicExists = true` only validates the topic exists, not the caller's identity). The attack requires only a valid topic ID and setting `endTime = Long.MAX_VALUE`. It is trivially scriptable using any gRPC client, requires no special knowledge beyond the public protobuf API, and is repeatable from a single machine across many concurrent connections. [8](#0-7) 

## Recommendation

1. **Add an upper-bound validator on `endTime` in `TopicMessageFilter`:** Reject requests where `endTime` exceeds `Instant.now()` plus a configurable maximum window (e.g., 30 days).

```java
// In TopicMessageFilter.java
@AssertTrue(message = "End time must not be too far in the future")
public boolean isValidEndTimeBound() {
    return endTime == null || endTime <= DomainUtils.now() + MAX_FUTURE_OFFSET_NS;
}
```

2. **Enforce a maximum subscriber count:** Check `subscriberCount` before accepting a new subscription and return `RESOURCE_EXHAUSTED` if the limit is exceeded.

3. **Add per-IP or per-client connection rate limiting** at the gRPC interceptor layer.

4. **Set a server-side maximum stream duration** via Netty/gRPC idle timeout configuration so that even a subscription with a far-future `endTime` is eventually reaped.

## Proof of Concept

```python
import grpc
# Using the Hedera/Hiero mirror node proto definitions
from com.hedera.mirror.api.proto import consensus_service_pb2_grpc, consensus_service_pb2
from com.hederahashgraph.api.proto.python import basic_types_pb2

channel = grpc.insecure_channel("mirror-node-host:5600")
stub = consensus_service_pb2_grpc.ConsensusServiceStub(channel)

request = consensus_service_pb2.ConsensusTopicQuery(
    topicID=basic_types_pb2.TopicID(topicNum=1),
    consensusStartTime=...,
    consensusEndTime=9223372036854775807,  # Long.MAX_VALUE
    limit=0
)

# Each of these holds a subscription open indefinitely
for _ in range(1000):
    stub.subscribeTopic(request)  # fire and forget, never terminates
```

With 1,000 concurrent connections, the server holds 1,000 `TopicContext` objects, 1,000 bounded-elastic thread slots, and issues a database query every 30 seconds per subscription (33 queries/second at steady state), exhausting resources and degrading service for legitimate users.
