Audit Report

## Title
Missing `deleted` Filter in `isValidAccount()` Allows Stale Reward Data Return for Deleted Accounts

## Summary
`EntityService.isValidAccount()` uses `entityExistenceQuery`, which queries the `entity` table by `id` only — with no filter on the `deleted` column. A soft-deleted account (row present with `deleted = true`) passes the validity gate in `listStakingRewardsByAccountId()`, causing `StakingRewardTransferService.getRewards()` to execute and return historical reward data with a `200 OK` instead of the correct `404 Not Found`.

## Finding Description
In `rest/controllers/accountController.js`, `listStakingRewardsByAccountId()` calls `EntityService.isValidAccount(accountId)` as its guard: [1](#0-0) 

`isValidAccount()` in `rest/service/entityService.js` delegates to `entityExistenceQuery`: [2](#0-1) 

`entityExistenceQuery` is: [3](#0-2) 

```sql
select type from entity where id = $1
```

There is no `deleted <> true` predicate. A soft-deleted account's row remains in the table with `deleted = true`; the query returns a row; `!isNil(entity)` is `true`; the guard passes.

This is inconsistent with the sibling queries in the same file, which correctly exclude deleted entities: [4](#0-3) 

- `entityFromAliasQuery` uses `coalesce(deleted, false) <> true`
- `entityFromEvmAddressQuery` uses `deleted <> true`

After the guard passes, `StakingRewardTransferService.getRewards()` executes a query filtered only by `account_id = $1` with no deletion check: [5](#0-4) 

## Impact Explanation
Any caller can send `GET /api/v1/accounts/{deleted_account_id}/rewards` and receive a `200 OK` with the full historical staking reward transfer list for a deleted account, instead of `404 Not Found`. While staking reward data is public ledger data (no confidentiality breach), the API contract is violated: deleted accounts are presented as active. Downstream integrations, wallets, and monitoring tools that rely on `404` to detect account deletion will receive incorrect signals. When `config.db.primaryHost` is set, `global.pool` points to a read replica and `global.primaryPool` to the primary: [6](#0-5) 

Replication lag extends the window during which this incorrect `200` is returned after deletion, but the bug fires on the primary DB itself regardless.

## Likelihood Explanation
Exploitation requires zero privileges — only knowledge of a previously valid account ID, which is trivially obtainable from public ledger history. No race condition, network partition, or special timing is required. The bug fires unconditionally on every request for any soft-deleted account.

## Recommendation
Add a `deleted` filter to `entityExistenceQuery` in `rest/service/entityService.js`, consistent with the sibling queries:

```sql
select type from entity where id = $1 and coalesce(deleted, false) <> true
``` [3](#0-2) 

## Proof of Concept
1. Identify any soft-deleted account ID (e.g., `0.0.12345`) from public ledger history.
2. Send: `GET /api/v1/accounts/0.0.12345/rewards`
3. `EntityService.getEncodedId()` resolves the numeric ID.
4. `EntityService.isValidAccount(accountId)` runs `select type from entity where id = $1` — returns a row because the deleted row is still present.
5. `!isNil(entity)` → `true` → guard passes, no `NotFoundError` thrown.
6. `StakingRewardTransferService.getRewards()` executes and returns historical reward rows.
7. Response: `200 OK` with `{"rewards": [...], "links": {"next": null}}` instead of `404 Not Found`.
