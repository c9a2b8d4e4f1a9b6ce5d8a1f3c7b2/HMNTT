# Audit Report

## Title
Unauthenticated Fee Estimation Endpoint Triggers Uncached Double DB Query Per Request via FeeTopicStore

## Summary
The public `POST /api/v1/network/fees?mode=STATE` endpoint accepts arbitrary protobuf transactions with no authentication or rate limiting. For `ConsensusSubmitMessage` transactions, `FeeTopicStore.getTopic()` issues two sequential, uncached database queries per call. Any unprivileged caller can flood this endpoint to generate 2N database queries for N concurrent requests, saturating the connection pool and degrading the mirror node for all consumers.

## Finding Description
**Code path:**

`NetworkController.estimateFees()` (lines 110–124) carries no authentication or rate-limiting annotation. When `mode=STATE`, `FeeEstimationService.estimateFees()` (line 101–103) constructs a `FeeEstimationFeeContext` backed by the singleton `FeeTopicStore`.

For `ConsensusSubmitMessage` transactions, the fee calculator invokes `FeeTopicStore.getTopic()`:

```java
// FeeTopicStore.java:28-33
public Topic getTopic(@NonNull final TopicID id) {
    return topicRepository
            .findById(id.topicNum())                          // DB query 1
            .map(topic -> toTopic(id, topic, customFeeRepository))
            .orElse(null);
}
// toTopic → getCustomFees → customFeeRepository.findById(topicId)  // DB query 2
```

Both `TopicRepository` and `CustomFeeRepository` are plain `CrudRepository<T, Long>` interfaces with no `@Cacheable` or any caching decoration. `getCustomFees()` (lines 51–58) unconditionally calls `customFeeRepository.findById()` whenever the topic row exists.

A grep across all of `rest-java/src/main/java/` finds zero uses of `@Cacheable`, `RateLimiter`, `@PreAuthorize`, or `@Secured`. The only registered filters are `LoggingFilter` and `MetricsFilter` — neither enforces any request throttle. The `highVolumeThrottle` request parameter (lines 116–117) is a caller-supplied fee-calculation input, not a server-side rate limiter.

The live-read design is intentional and confirmed by the test comment at `FeeEstimationServiceTest.java:361`: *"store reads live from DB, no refresh needed."*

**Root cause:** `FeeTopicStore` is intentionally designed to read live from the database on every call, but no compensating control (caching, rate limiting, or authentication) exists on the public endpoint that drives it.

## Impact Explanation
Each `POST /api/v1/network/fees?mode=STATE` request carrying a `ConsensusSubmitMessage` targeting a valid topic causes exactly 2 synchronous database round-trips with no result reuse across requests. N concurrent requests generate 2N database queries. At sufficient volume this saturates the database connection pool, degrades query latency for all other mirror node consumers (REST, gRPC, importer), and can render the mirror node unresponsive. The damage is confined to mirror node infrastructure availability; the Hedera consensus network itself is unaffected.

## Likelihood Explanation
The exploit requires zero privileges, zero tokens, and zero on-chain activity. A valid topic ID is obtainable from any public mirror node query. The attack is trivially scriptable, repeatable at any rate, and requires no special knowledge beyond the public OpenAPI spec. The attacker must explicitly set `mode=STATE` (the default is `INTRINSIC`), but this is a one-parameter change.

## Recommendation
1. **Application-level rate limiting:** Apply a rate limiter (e.g., Bucket4j, Resilience4j, or a servlet filter) to `POST /api/v1/network/fees` keyed by client IP or API key.
2. **Caching:** Add `@Cacheable` (with a short TTL) to `FeeTopicStore.getTopic()` or to the underlying repository calls to avoid redundant DB round-trips for the same topic within a time window.
3. **Infrastructure-level throttling:** Enforce request-rate limits at the ingress layer (API gateway, load balancer) as a defense-in-depth measure independent of application code.
