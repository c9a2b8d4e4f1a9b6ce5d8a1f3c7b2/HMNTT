# Audit Report

## Title
Stale Token-Level Freeze Status Permanently Applied to Accounts via Unbounded LRU Cache in `getTokenAccounts()`

## Summary
`getTokenAccounts()` in `rest/service/tokenService.js` uses a nullish-coalescing fallback (`??`) to substitute a cached token-level `freezeStatus` onto any `token_account` row whose `freeze_status` column is `null` — a valid post-V1.94.0 state. The in-process `tokenCache` (`quickLru`) has no TTL and no invalidation path. Once a token's freeze status is cached, it is never refreshed until evicted by LRU size pressure. An unprivileged user can anchor a FROZEN entry in the cache with a single read-only GET request; after the token is unfrozen on-chain, every subsequent query for an account with `freeze_status = null` will receive the stale FROZEN status indefinitely.

## Finding Description

**Cache construction — no TTL:**

`tokenCache` is constructed with `maxSize` only; the `quick-lru` library's `maxAge` option is not used. [1](#0-0) 

**Fallback application in `getTokenAccounts()`:**

Line 109 applies the cached token-level `freezeStatus` to any row where `freeze_status` is `null` via the `??` operator. [2](#0-1) 

**Cache write with no refresh in `getCachedTokens()`:**

On a cache hit, the cached entry is returned directly and the DB is never re-queried. The DB is only queried for `uncachedTokenIds`; the result is written once and never updated. [3](#0-2) 

**`putTokenCache()` also never updates existing entries:** [4](#0-3) 

**No invalidation path exists** outside of the test-only `clearTokenCache` helper: [5](#0-4) 

**Migration V1.94.0 makes `null` a valid state for `token_account.freeze_status`:**

Lines 36–37 explicitly drop the `NOT NULL` constraint and default from `token_account.freeze_status`, meaning any account associated after this migration (without an explicit freeze/unfreeze transaction) will have `freeze_status = null`. [6](#0-5) 

**`CachedToken` stores `freezeStatus` directly from the DB row at cache-population time:** [7](#0-6) 

**`TokenRelationshipViewModel` serializes the computed `freeze_status` directly into the API response:** [8](#0-7) 

**Root cause:** The code assumes `token.freeze_status` (the token-level default) is immutable after first cache population. It is not — a token admin can call `TokenUpdate` to change `freeze_default`, which updates `token.freeze_status` in the DB. The cache is never notified of this change.

## Impact Explanation

The mirror node REST API is the canonical read layer used by wallets, exchanges, and DeFi integrations to gate token transfers. `TokenRelationshipViewModel` directly exposes the computed `freeze_status` field to clients. An account incorrectly reported as FROZEN will be refused by any application that checks this field before submitting a `CryptoTransfer`. While the Hedera consensus layer enforces the true state, most client-side UIs and SDKs short-circuit on the mirror node response and never attempt the transfer. The result is a denial-of-service on token transfers for affected accounts, with no on-chain action required from the attacker and no self-healing until the LRU entry is evicted.

## Likelihood Explanation

- **Attacker capability:** zero — a single unauthenticated GET request suffices to anchor the cache entry.
- **Precondition:** the token must have been frozen at some point (a routine administrative action). The attacker does not need to control or influence the token admin.
- **Persistence:** indefinite in the absence of LRU eviction. In production deployments with large `maxSize` and a popular token, the entry will remain hot and never be evicted.
- **Repeatability:** if the entry is eventually evicted, the attacker re-anchors it with another GET request during any future freeze window.

## Recommendation

1. **Add a TTL to the cache.** Pass `maxAge` (in milliseconds) to the `quickLru` constructor, e.g.:
   ```js
   const tokenCache = new quickLru({
     maxSize: config.cache.token.maxSize,
     maxAge: config.cache.token.maxAge, // e.g. 60_000 ms
   });
   ```
   This bounds the maximum staleness of any cached freeze/kyc status.

2. **Invalidate on write.** Any code path that processes a `TokenUpdate` transaction and updates `token.freeze_status` in the DB should call `tokenCache.delete(tokenId)`.

3. **Re-evaluate the `??` fallback.** If `token_account.freeze_status = null` is intended to mean "inherit the token default," the inheritance should be resolved at query time from a fresh DB read, not from a potentially stale in-memory cache.

## Proof of Concept

1. Token T is created with `freeze_default = true` → `token.freeze_status = 1` (FROZEN) in DB.
2. Attacker (no privileges) calls `GET /api/v1/accounts/{any}/tokens?token.id=T`. `getCachedTokens` finds T uncached, queries DB, stores `CachedToken{freezeStatus: 1}` in `tokenCache`.
3. Token admin calls `TokenUpdate` to set `freeze_default = false` → `token.freeze_status = 2` (UNFROZEN) in DB. Cache is **not** invalidated.
4. Account B associates with token T. Because no explicit freeze/unfreeze transaction has been issued for B, `token_account.freeze_status = null`.
5. Any user calls `GET /api/v1/accounts/B/tokens`. `getCachedTokens` returns the cached entry (cache hit). Line 109 evaluates `null ?? 1 → 1`. `TokenRelationshipViewModel` line 24 serializes this as `"freeze_status": "FROZEN"` in the JSON response.
6. Account B is permanently reported as FROZEN to all API consumers until the LRU entry is evicted by size pressure.

```javascript
  async getCachedTokens(tokenIds) {
    const cachedTokens = new Map();
    const uncachedTokenIds = [];
    tokenIds.forEach((tokenId) => {
      const cachedToken = tokenCache.get(tokenId);
      if (cachedToken) {
        cachedTokens.set(tokenId, cachedToken);
      } else {
        uncachedTokenIds.push(tokenId);
      }
    });

    if (uncachedTokenIds.length === 0) {
      return cachedTokens;
    }

    const rows = await super.getRows(TokenService.tokenCacheQuery, [uncachedTokenIds]);
    rows.forEach((row) => {
      const tokenId = row.token_id;
      const cachedToken = new CachedToken(row);
      tokenCache.set(tokenId, cachedToken);
      cachedTokens.set(tokenId, cachedToken);
    });

    return cachedTokens;
  }
```
