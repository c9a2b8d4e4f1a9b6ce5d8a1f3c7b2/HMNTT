Audit Report

## Title
Gas Token Leak on REJECT Filter Path Enables `gasLimitBucket` Exhaustion DoS

## Summary
`ThrottleManagerImpl.throttle()` consumes tokens from `gasLimitBucket` before evaluating request filters. When a `REJECT` filter matches, a `ThrottleException` is thrown with no preceding `restore()` call. `ContractController.call()` only catches `InvalidParametersException` (and restores there), leaving `ThrottleException` to propagate to `GenericControllerAdvice`, which returns HTTP 429 without restoring the consumed gas tokens. The tokens are permanently lost until the bucket naturally refills.

## Finding Description

**Step 1 — Gas consumed before filter evaluation.**
In `ThrottleManagerImpl.throttle()`, line 40 unconditionally consumes scaled gas tokens from `gasLimitBucket`, and only then (lines 44–48) iterates over request filters. [1](#0-0) 

**Step 2 — REJECT throws with no restore.**
`action()` line 69 throws `ThrottleException("Invalid request")` for a `REJECT` filter with no call to `restore()` anywhere in the method or its callers before the exception propagates. [2](#0-1) 

**Step 3 — `ContractController.call()` does not catch `ThrottleException`.**
The only `catch` block in `call()` is for `InvalidParametersException` (lines 46–49), which explicitly calls `throttleManager.restore(request.getGas())`. `ThrottleException` is not caught here and propagates upward. [3](#0-2) 

**Step 4 — `GenericControllerAdvice` returns 429 with no restore.**
`throttleException()` at line 135 simply returns `TOO_MANY_REQUESTS`. No restore is performed. [4](#0-3) 

**Step 5 — `restoreGasToBucket()` in `ContractCallService` is unreachable on this path.**
`restoreGasToBucket()` is called only from `doProcessCall()`, which is only reached via `contractExecutionService.processCall(params)` at line 44 of `ContractController.call()`. Because `throttle()` throws before line 44 is reached, `restoreGasToBucket()` is never invoked. [5](#0-4) 

**Root cause:** The developer correctly added a restore on `InvalidParametersException` but omitted an equivalent restore for `ThrottleException` originating from REJECT (and THROTTLE) filter actions, creating an asymmetric gas-token leak.

Note: the `THROTTLE` action branch (line 72) has the identical defect — it also throws `ThrottleException` without restoring gas tokens. [6](#0-5) 

## Impact Explanation

Every request that matches a REJECT filter with `gas > 10,000` (the `GAS_SCALE_FACTOR` threshold below which `scaleGas` returns 0) permanently removes `floor(gas / 10,000)` tokens from the shared `gasLimitBucket` until the per-second refill recovers them. [7](#0-6) 

With default settings (`gasPerSecond = 7,500,000,000` → 750,000 scaled tokens/s; `requestsPerSecond = 500`; max gas = 15,000,000 → 1,500 scaled tokens/request):
- 500 req/s × 1,500 tokens = 750,000 tokens/s consumed — exactly the refill rate.
- The bucket is kept perpetually empty.
- Legitimate callers receive `GAS_PER_SECOND_LIMIT_EXCEEDED` and cannot execute any contract calls. [8](#0-7) 

## Likelihood Explanation

**Preconditions:**
- Operator has configured at least one `REJECT` filter (non-default, but a documented and intended use case; the default `request` list is empty).
- Attacker can identify filter criteria by probing the API or reading public configuration.

**Feasibility:** No authentication or special privilege is required. Any HTTP client can `POST /api/v1/contracts/call` with arbitrary `gas` and `to` fields. The attack is trivially scriptable. The rate-limit bucket (500 req/s) does not prevent gas-bucket drain because the two buckets are independent and the math produces exact exhaustion at the rate limit. [1](#0-0) 

## Recommendation

Add a `restore()` call before throwing from the `REJECT` (and `THROTTLE`) branches in `ThrottleManagerImpl.action()`, or — preferably — restructure `throttle()` to evaluate request filters **before** consuming gas tokens from `gasLimitBucket`. The latter approach is cleaner because it avoids the need for any restore on the rejection path entirely.

Alternatively, add a `catch (ThrottleException e)` block in `ContractController.call()` that calls `throttleManager.restore(request.getGas())` before re-throwing, mirroring the existing `InvalidParametersException` handler. [9](#0-8) 

## Proof of Concept

```
# Assume REJECT filter is configured for to=0xdeadbeef...
for i in $(seq 1 500); do
  curl -s -X POST https://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0xdeadbeef...","gas":15000000}' &
done
wait
# Repeat every second. After ~1 second the gasLimitBucket is empty.
# Legitimate callers now receive:
# HTTP 429 {"_status":{"messages":[{"message":"Gas per second rate limit exceeded."}]}}
```

Each iteration consumes 1,500 scaled tokens (15,000,000 / 10,000). At 500 req/s the bucket (capacity 750,000) is drained in one second and the refill rate exactly matches consumption, keeping it empty indefinitely. [10](#0-9)
