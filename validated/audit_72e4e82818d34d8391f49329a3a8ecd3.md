# Audit Report

## Title
Deleted Entity Bypass via Long-Form EVM Address in `getByEvmAddressAndType()`

## Summary
`EntityServiceImpl.getByEvmAddressAndType()` applies inconsistent deletion filtering across its two branches. The long-form EVM address branch calls the inherited `CrudRepository.findById()`, which issues a plain `SELECT … WHERE id = ?` with no `deleted` predicate. The regular EVM address branch calls a custom `findByEvmAddress()` query that explicitly appends `and deleted is not true`. An unauthenticated caller can craft a long-form EVM address encoding any deleted entity's numeric ID and retrieve its full record through the public GraphQL endpoint.

## Finding Description
In `EntityServiceImpl.java`, `getByEvmAddressAndType()` decodes the supplied EVM address and checks whether the first 12 bytes are all zero (shard=0, realm=0). If so, it treats the last 8 bytes as a raw entity ID and delegates to `entityRepository.findById()`:

```java
// EntityServiceImpl.java lines 37–40
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())   // ← no deletion filter
                           .filter(e -> e.getType() == type);
}
return entityRepository.findByEvmAddress(evmAddressBytes) // ← deleted is not true
                       .filter(e -> e.getType() == type);
``` [5](#0-4) 

`EntityRepository` defines `findByEvmAddress` with an explicit guard:

```sql
select * from entity where evm_address = ?1 and deleted is not true
``` [2](#0-1) 

`findById` is inherited from `CrudRepository<Entity, Long>` and is not overridden. The `Entity` class carries no `@Where`, `@Filter`, or `@SQLRestriction` annotation that would automatically inject a deletion predicate. [3](#0-2) 

The only downstream check after `findById` is `.filter(e -> e.getType() == type)`, which validates entity type but not deletion status.

**Additional note:** `getByIdAndType()` (line 25) has the same pattern — it also calls `findById` without a deletion filter — but that path requires the caller to supply a structured `entityId` input, making it a separate but related inconsistency. [6](#0-5) 

## Impact Explanation
A deleted account's full record — including balance, alias, EVM address, and associated metadata — can be read by any unauthenticated caller. This violates the intended invariant that deleted entities are invisible to API consumers, and may expose account state that operators expected to be suppressed after deletion.

## Likelihood Explanation
No credentials or special network access are required. The GraphQL `account` query mapping in `AccountController` carries no authentication annotation and is publicly reachable. [4](#0-3) 
Entity IDs are sequential integers starting from 1, so enumerating deleted IDs requires only a loop of single HTTP requests. The exploit is fully repeatable.

## Recommendation
Replace the `findById` call in the long-form branch with a custom repository method that includes the deletion guard, consistent with `findByEvmAddress`:

```java
// EntityRepository.java — add:
@Query(value = "select * from entity where id = ?1 and deleted is not true", nativeQuery = true)
Optional<Entity> findByIdAndNotDeleted(long id);
```

Then update `EntityServiceImpl`:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findByIdAndNotDeleted(buffer.getLong())
                           .filter(e -> e.getType() == type);
}
```

Apply the same fix to `getByIdAndType()` if deleted entities should also be suppressed when looked up by structured entity ID. Alternatively, add a `@SQLRestriction("deleted is not true")` annotation on the `AbstractEntity` class to enforce the filter globally across all derived queries, including `findById`.

## Proof of Concept

```graphql
# Replace <entity_id_hex> with the big-endian hex of any deleted account's numeric ID
# e.g., entity ID 1234 → 00000000000004D2
{
  account(input: { evmAddress: "0x000000000000000000000000<entity_id_hex>" }) {
    id
    balance
    alias
    evmAddress
  }
}
```

1. Enumerate entity IDs starting from 1 (sequential integers).
2. For each candidate, encode as 8 big-endian bytes and prepend 12 zero bytes to form a 20-byte long-form EVM address.
3. Submit the GraphQL query above to the public `/graphql` endpoint.
4. A non-null response with a matching `ACCOUNT` type confirms the deleted entity was returned.
