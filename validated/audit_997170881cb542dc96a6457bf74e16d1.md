Audit Report

## Title
Deleted Entity Exposure via Long-Zero EVM Address Bypass of `deleted is not true` Filter

## Summary
In `EntityServiceImpl.getByEvmAddressAndType()`, a long-zero EVM address (first 12 bytes all zero) triggers a fallback to `entityRepository.findById()`, which carries no `deleted is not true` predicate. All other lookup paths (`findByEvmAddress`, `findByAlias`) explicitly filter deleted entities. An unauthenticated attacker can retrieve fully deleted accounts or contracts through the GraphQL API by encoding any known numeric entity ID as a long-zero EVM address.

## Finding Description

**Exact location:** `graphql/src/main/java/org/hiero/mirror/graphql/service/EntityServiceImpl.java`, lines 34–41.

The long-zero branch at line 37–38 calls `entityRepository.findById(buffer.getLong())` — a `CrudRepository`-inherited method that issues a plain `SELECT * FROM entity WHERE id = ?` with no deleted predicate: [1](#0-0) 

The non-long-zero branch at line 40 calls `entityRepository.findByEvmAddress()`, which is defined with an explicit `deleted is not true` guard: [2](#0-1) 

`findByAlias` carries the same guard: [3](#0-2) 

The `Entity` / `AbstractEntity` domain class has a `deleted` field but no JPA-level `@SQLRestriction` or `@Where` annotation, so no automatic filtering occurs at the ORM layer: [4](#0-3) 

The only post-query check in the long-zero branch is `.filter(e -> e.getType() == type)`, which tests entity type but never tests `deleted`. The `AccountController` then maps and returns the result unconditionally: [5](#0-4) 

Note: `getByIdAndType` (line 25) has the same missing deleted filter when looked up by `entityId`, making the inconsistency systemic across lookup paths that use `findById`. [6](#0-5) 

## Impact Explanation
A deleted account or contract is fully returned by the GraphQL API — including key material, balance, memo, and all other fields — when queried via a crafted long-zero EVM address. Callers receive a non-null `Account` object for an entity that has been deleted, misrepresenting the current state of the ledger. This can mislead downstream systems about account existence, key ownership, and balance. No authentication is required.

## Likelihood Explanation
Hedera account and contract IDs are sequential integers publicly broadcast in every transaction. Any network observer can enumerate deleted entity IDs. Constructing the corresponding long-zero EVM address requires only basic hex encoding (zero-pad 12 bytes, then big-endian encode the numeric ID into the remaining 8 bytes). The attack requires zero privileges, no special tooling, and is trivially repeatable for any known deleted entity ID.

## Recommendation
Add a `deleted` check in the long-zero branch of `getByEvmAddressAndType`, consistent with the other lookup paths:

```java
if (buffer.getInt() == 0 && buffer.getLong() == 0) {
    return entityRepository.findById(buffer.getLong())
            .filter(e -> e.getType() == type && !Boolean.TRUE.equals(e.getDeleted()));
}
```

Alternatively, define a custom `findByIdAndDeletedIsNotTrue` query in `EntityRepository` (analogous to `findByEvmAddress` and `findByAlias`) and use it in both `getByEvmAddressAndType` (long-zero branch) and `getByIdAndType`. This would make the deleted-filtering contract uniform across all repository lookups.

## Proof of Concept

1. Identify a deleted Hedera account with numeric ID, e.g. `12345` (publicly visible on-chain).
2. Construct the 20-byte long-zero EVM address: bytes 0–11 = `0x00`, bytes 12–19 = big-endian `12345` → `0x0000000000000000000000000000000000003039`.
3. Send the following GraphQL query to the mirror node:
   ```graphql
   query {
     account(input: { evmAddress: "0000000000000000000000000000000000003039" }) {
       id
       deleted
       balance
       key
       memo
     }
   }
   ```
4. `decodeEvmAddress` decodes the hex; `buffer.getInt()` = 0, `buffer.getLong()` = 0 → long-zero branch taken at line 37.
5. `entityRepository.findById(12345)` returns the deleted entity with no filter applied.
6. `.filter(e -> e.getType() == ACCOUNT)` passes if the entity is an account.
7. The full deleted entity — including `deleted: true`, key material, balance, and memo — is returned to the unauthenticated caller.
