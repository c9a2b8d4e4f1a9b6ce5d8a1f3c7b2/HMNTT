## Audit Report

## Title
Unauthenticated Timestamp Filter on `/contracts/:contractId/state` Triggers Unbounded `DISTINCT ON` Scan of `contract_state_change`, Enabling DoS

## Summary
Any unauthenticated caller can supply a `timestamp` query parameter to `GET /api/v1/contracts/:contractId/state`. This switches the query from the cheap `contract_state` table to an unbounded `DISTINCT ON (slot)` scan over the entire `contract_state_change` table for that contract, with no timestamp range cap enforced. Repeated requests against a high-activity contract can saturate database CPU/IO.

## Finding Description

**Verified code path:**

In `rest/controllers/contractController.js`, `extractContractStateByIdQuery()` processes filters in a `switch` statement. Any `TIMESTAMP` filter (except `ne`) sets `timestampPresent = true` and pushes a `consensus_timestamp` condition into the WHERE array: [1](#0-0) 

The function returns `timestamp: timestampPresent`: [2](#0-1) 

`getContractStateById()` then passes this flag directly to `ContractService.getContractStateByIdAndFilters()`: [3](#0-2) 

When `timestamp === true`, the service switches from `contractStateQuery` (reads the small, current-state `contract_state` table) to `contractStateTimestampQuery`, which performs a `DISTINCT ON (slot)` scan over the append-only, unbounded `contract_state_change` table: [4](#0-3) 

For comparison, the cheap path reads only from `contract_state`: [5](#0-4) 

**Index/sort mismatch makes LIMIT useless:**

The index `contract_state_change__id_slot_timestamp` is defined on `(contract_id, slot, consensus_timestamp)` — all columns ascending: [6](#0-5) 

The `DISTINCT ON (slot)` query requires `ORDER BY slot ASC, consensus_timestamp DESC` to correctly select the latest row per slot. Because `consensus_timestamp` is stored ascending in the index but the query needs it descending, PostgreSQL cannot satisfy the ordering from the index. It must:
1. Scan **all** rows matching `contract_id = $1 AND consensus_timestamp <= $2`
2. Sort them in memory by `(slot ASC, consensus_timestamp DESC)`
3. Apply `DISTINCT ON (slot)` to pick the latest row per slot
4. Only then apply `LIMIT`

The `LIMIT` clause is applied **after** the full scan and sort, providing no early-termination protection.

**No timestamp range validation:**

`extractContractStateByIdQuery` does not call `parseTimestampFilters()` (which enforces a configurable max range). It only validates timestamp format. The `parseTimestampFilters` utility exists and is used in other controllers: [7](#0-6) [8](#0-7) 

**No authentication required:** The endpoint is public REST API with no privilege check on `getContractStateById`.

## Impact Explanation

The `contract_state_change` table is append-only and grows without bound. On a production network with popular DeFi contracts, this table can contain tens or hundreds of millions of rows. Each crafted request with `timestamp=lte:<large_value>` forces a full-table-equivalent scan plus in-memory sort for that contract. Concurrent requests from multiple IPs can saturate PostgreSQL CPU and I/O, causing query timeouts and cascading failures across all mirror-node API endpoints sharing the same DB pool.

## Likelihood Explanation

- **Precondition:** None. No account, API key, or token required.
- **Knowledge required:** Any publicly visible high-activity contract ID (available on block explorers).
- **Repeatability:** Fully repeatable; each request is stateless and independently expensive.
- **Detection evasion:** The request is syntactically valid and indistinguishable from a legitimate historical-state lookup.

## Recommendation

1. **Enforce a maximum timestamp range** in `extractContractStateByIdQuery` by calling `parseTimestampFilters()` (already used elsewhere in the codebase) or adding an explicit max-range guard before setting `timestampPresent = true`.
2. **Fix the index/sort mismatch** by adding a partial index on `(contract_id, slot, consensus_timestamp DESC)` or restructuring the query to use a lateral join or subquery that can leverage the existing index for early termination.
3. **Apply rate limiting** at the API gateway or middleware layer specifically for the `/contracts/:contractId/state` endpoint when a `timestamp` parameter is present.
4. **Consider requiring authentication** or a lower default `LIMIT` cap for the timestamp-filtered path.

## Proof of Concept

```http
GET /api/v1/contracts/0.0.1234567/state?timestamp=lte:9999999999999999999&limit=100
```

No authentication headers required. Against a high-activity contract, this triggers a full scan + in-memory sort of all rows in `contract_state_change` for that contract before returning 100 results. Sending this request concurrently from multiple clients will progressively degrade database performance for all mirror-node consumers.

  async extractContractStateByIdQuery(filters, contractId) {
    let limit = defaultLimit;
    let order = orderFilterValues.ASC;
    let timestampPresent = false;
    const conditions = [this.getFilterWhereCondition(ContractState.CONTRACT_ID, {operator: '=', value: contractId})];
    const slotFilters = [];

    for (const filter of filters) {
      switch (filter.key) {
        case filterKeys.LIMIT:
          limit = filter.value;
          break;
        case filterKeys.ORDER:
          order = filter.value;
          break;
        case filterKeys.TIMESTAMP:
          if (utils.opsMap.ne === filter.operator) {
            throw new InvalidArgumentError(`Not equals (ne) operator is not supported for ${filterKeys.TIMESTAMP}`);
          }

          if (utils.opsMap.eq === filter.operator) {
            filter.operator = utils.opsMap.lte;
          }
          conditions.push(this.getFilterWhereCondition(ContractStateChange.CONSENSUS_TIMESTAMP, filter));
          timestampPresent = true;
          break;
        case filterKeys.SLOT:
          slotFilters.push(filter);
          break;
        default:
          break;
      }
    }
