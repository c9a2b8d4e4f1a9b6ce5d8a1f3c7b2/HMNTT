# Audit Report

## Title
Malicious Block Node Can Cause Indefinite Block Ingestion Latency via Slow-Drip gRPC Streaming

## Summary
`BlockAssembler.timeout()` in `BlockNode.java` resets its `Stopwatch` after every completed block via `onEndOfBlock()`. This grants a fresh, full timeout budget per block rather than per connection. A malicious block node can exploit this to hold the importer's synchronous streaming thread indefinitely — one block at a time — without ever triggering an error path.

## Finding Description

**`timeout()` method (lines 274–281):**

```java
long timeout() {
    if (!stopwatch.isRunning()) {
        stopwatch.start();
        return timeout.toMillis();
    }
    return timeout.toMillis() - stopwatch.elapsed(TimeUnit.MILLISECONDS);
}
``` [1](#0-0) 

This return value is passed directly to the blocking gRPC read at line 138:

```java
while (!serverSuccess && (response = grpcCall.read(assembler.timeout(), TimeUnit.MILLISECONDS)) != null) {
``` [2](#0-1) 

**Root cause — `stopwatch.reset()` in `onEndOfBlock()` at line 268:** [3](#0-2) 

`Stopwatch.reset()` both stops and zeroes the stopwatch. The next call to `timeout()` sees `!stopwatch.isRunning() == true`, restarts the stopwatch, and returns the **full** `timeout.toMillis()` again. The per-block timeout is never cumulative across blocks.

**Why existing mitigations do not apply:**

- `onError()` / `maxSubscribeAttempts` / `readmitDelay` — only triggered inside `catch` blocks; a slow-but-non-erroring stream never reaches them. [4](#0-3) 

- `maxBlockItems` — throws only when item *count* per block exceeds the limit, not when delivery is slow. [5](#0-4) 

- `maxStreamResponseSize` — limits individual gRPC message size at the channel level, not delivery rate. [6](#0-5) 

- `responseTimeout` (400 ms default) — used exclusively in `getBlockRange()` via `withDeadlineAfter(streamProperties.getResponseTimeout())`; it is **not** applied to `streamBlocks()`, which uses `commonDownloaderProperties.getTimeout()` instead. [7](#0-6) [8](#0-7) 

**Synchronous call chain:** `streamBlocks()` is called synchronously from `BlockNodeSubscriber.doGet()`, which is called from `AbstractBlockSource.get()`. No parallelism exists; while one node's `streamBlocks()` is executing, no other node can be polled. [9](#0-8) [10](#0-9) 

## Impact Explanation

A malicious block node holds the importer's single streaming thread indefinitely, one block at a time, each block consuming up to the full configured timeout. Because `streamBlocks()` is synchronous and the importer selects one node per block, legitimate block nodes cannot be used while the malicious node is active. Block ingestion latency spikes proportionally, degrading mirror node data freshness for all downstream consumers. Severity is **medium** (availability degradation / griefing; no direct fund loss or data corruption).

## Likelihood Explanation

The attacker must control a block node reachable by the importer — either via static configuration or via the `BlockNodeDiscoveryService` auto-discovery path. Auto-discovery reads tier-1 block nodes from `RegisteredNodeRepository` and merges them with configured nodes. [11](#0-10) 

No authentication of block nodes is visible in the reviewed code path. Any operator of a node the importer is pointed at (or that is auto-discovered) can execute this attack without privileged access to the importer itself. The attack is repeatable and requires only network-level participation.

## Recommendation

1. **Add a cumulative wall-clock deadline per streaming session.** Record `Instant sessionStart` when `streamBlocks()` begins and enforce a maximum total duration (e.g., `N × timeout`) across all blocks in a single call. Abort and call `onError()` if the deadline is exceeded.

2. **Track per-block elapsed time independently of the stopwatch reset.** Instead of resetting the stopwatch in `onEndOfBlock()`, maintain a separate session-level elapsed counter so that slow-drip across multiple blocks is detectable.

3. **Apply `responseTimeout` (or a dedicated streaming timeout) to the `streamBlocks()` gRPC call** via `CallOptions.withDeadlineAfter(...)`, giving the entire streaming RPC a hard wall-clock limit.

4. **Count consecutive slow-but-successful blocks** and treat them as a soft error, incrementing the `errors` counter and eventually triggering `onError()` / `readmitDelay` even without an exception.

## Proof of Concept

```
1. Attacker controls a block node reachable by the importer.
2. Importer calls streamBlocks(N, ...) → grpcCall.read(timeout_ms) blocks.
3. Attacker sends BLOCK_HEADER for block N → stopwatch starts in timeout().
4. Attacker sends one BlockItemSet every (timeout - ε) ms, keeping remaining
   timeout just above zero.
5. Attacker sends END_OF_BLOCK for block N just before timeout expires
   → onEndOfBlock() calls stopwatch.reset() (line 268).
6. Next grpcCall.read() call invokes timeout() → stopwatch not running →
   stopwatch.start() → returns full timeout.toMillis() again.
7. Attacker immediately sends BLOCK_HEADER for block N+1.
8. Steps 4–7 repeat indefinitely.
Result: streamBlocks() never returns; the importer's streaming thread is
blocked; no onError() is ever called; legitimate nodes are never polled.
```
