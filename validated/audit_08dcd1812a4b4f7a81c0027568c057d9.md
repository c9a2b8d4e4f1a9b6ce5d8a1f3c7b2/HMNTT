Audit Report

## Title
Silent Exception Swallowing in `BinaryGasEstimator.safeCall()` Causes Gas Estimate Inflation Without Error Surfacing

## Summary
`BinaryGasEstimator.safeCall()` catches all `Exception` subtypes and returns `null`. The binary search loop treats `null` identically to an out-of-gas result by advancing `lo = mid`. Because `TransactionExecutionService.execute()` throws `MirrorEvmTransactionException` for any contract revert before returning a result — and `doProcessCall` re-throws it regardless of the `estimate` flag — a contract that reverts at every intermediate gas value causes every binary search probe to silently fail. The estimator converges on `hi` (the caller-supplied gas ceiling, up to 15 M gas) and returns that inflated value with HTTP 200 and no error indication.

## Finding Description

**`BinaryGasEstimator.safeCall()` — lines 67–74:** Catches all `Exception` subtypes and returns `null`, with only an `INFO`-level log entry. [1](#0-0) 

**`BinaryGasEstimator.search()` — lines 45–56:** `null` result sets `err = true`, which unconditionally advances `lo = mid`. `hi` is only reduced in the `else` branch, which is never reached when every probe returns `null`. [2](#0-1) 

**`ContractCallService.doProcessCall()` — lines 120–126:** The `estimate=true` flag suppresses `validateResult` but does not affect the `catch (MirrorEvmTransactionException e)` block, which always re-throws. [6](#0-5) 

**`TransactionExecutionService.handleFailedResult()` — lines 140–153:** For any non-SUCCESS transaction (including contract reverts), this method throws `MirrorEvmTransactionException` before returning a result. This throw occurs inside `execute()`, before `doProcessCall` can apply the `estimate` flag logic. [4](#0-3) 

**`ContractExecutionService.estimateGas()` — lines 91–95:** The lambda `gas -> doProcessCall(params, gas, true)` is passed to `binaryGasEstimator.search()`, connecting all the above components. [5](#0-4) 

**Root cause:** The failed assumption is that any exception thrown during a binary search probe means "gas was insufficient." In reality, `TransactionExecutionService.execute()` throws `MirrorEvmTransactionException` for contract reverts, precompile failures, and other non-gas-related errors. `safeCall` cannot distinguish between an OOG exception and a revert exception; it maps both to `null`, which the loop treats as `err = true` → `lo = mid`.

## Impact Explanation
The caller receives a massively inflated gas estimate (up to the configured `maxGasLimit`, default 15 M) with HTTP 200 and no error indication. On Hedera, gas limits directly determine transaction fees; a user who trusts this estimate and submits a transaction with a 15 M gas limit will pay the maximum possible fee for what may be a cheap operation. Client libraries (ethers.js, web3.js) and wallets unconditionally trust `eth_estimateGas` responses and have no signal that the estimate is unreliable. A malicious dApp operator can use this to systematically inflate fees paid by users who interact with a crafted contract.

## Likelihood Explanation
No special privileges are required. Any account holding enough HBAR to deploy a contract can execute this attack. The contract logic required (succeed at max gas, revert at lower gas using `gasleft()`) is trivial to write in Solidity. The attack is fully repeatable and deterministic. The initial probe at max gas succeeds, so `estimateGas()` does not surface any error to the caller.

## Recommendation

1. **Distinguish exception types in `safeCall`:** Only treat OOG-specific exceptions as "gas insufficient." Re-throw (or propagate as a proper error response) any `MirrorEvmTransactionException` that represents a contract revert or pre-check failure unrelated to gas.

2. **Return a failed `EvmTransactionResult` instead of throwing for estimate probes:** Modify `TransactionExecutionService.handleFailedResult()` or `doProcessCall` so that when `estimate=true`, a failed execution returns a result object with `isSuccessful() == false` rather than throwing. This allows `safeCall` to be removed entirely and the loop to inspect the actual failure reason.

3. **Propagate revert errors to the caller:** If all binary search probes fail with a non-OOG exception, the estimator should abort and return an error response rather than silently returning `hi`.
