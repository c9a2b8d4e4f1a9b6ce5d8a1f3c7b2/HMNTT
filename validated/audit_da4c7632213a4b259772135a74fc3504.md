# Audit Report

## Title
Unbounded Alias Loop in `resolveAccountAliases` Exhausts Database Connection Pool via Single Unauthenticated Request

## Summary
The `resolveAccountAliases()` function in `rosetta/app/services/construction_service.go` issues one synchronous database query per comma-separated alias token with no upper bound on alias count and no HTTP body size cap. An unauthenticated caller of the `/construction/metadata` endpoint can supply an arbitrarily large alias list, causing prolonged connection pool saturation and denying service to all other consumers of the mirror node.

## Finding Description

**Verified code path:**

`ConstructionMetadata` at line 159 calls `resolveAccountAliases`: [1](#0-0) 

Inside `resolveAccountAliases`, the function splits the raw string on commas and issues one blocking DB round-trip per token with no alias count cap: [2](#0-1) 

`GetAccountId` in `account.go` acquires a connection from the pool and executes a raw SQL query for every call, regardless of whether the alias resolves to a real account: [3](#0-2) 

The connection pool ceiling is enforced in `db.go` via `SetMaxOpenConns`: [4](#0-3) 

**Why existing mitigations fail:**

| Mitigation | Why insufficient |
|---|---|
| `NewAccountIdFromString` format check | Only validates format; trivially satisfied with syntactically valid alias strings that do not correspond to real accounts — the DB query still executes and returns `ErrAccountNotFound` only *after* the round-trip completes |
| `ReadTimeout` on HTTP server | Configurable but no `http.MaxBytesReader` is set anywhere in `main.go` — confirmed absent from the entire codebase |
| `statementTimeout` context per query | Limits individual query duration, not the total number of sequential queries or concurrent connections consumed across requests |
| Traefik rate-limit middleware | Defined in `values.yaml` lines 157–161 but gated on `global.middleware: true`; the default is `global.middleware: false` at line 95, so it is **off by default** | [5](#0-4) [6](#0-5) [7](#0-6) 

**Clarification on connection usage:** Within a single request, queries are sequential (each connection is released before the next is acquired). The attack surface is therefore: many concurrent requests, each carrying many aliases, where each request holds a connection for the full `statementTimeout` duration per alias. This still saturates the pool effectively.

## Impact Explanation
The database connection pool is bounded by `Pool.MaxOpenConnections`. Multiple concurrent requests each carrying a large alias list will hold connections for extended periods (up to `statementTimeout` seconds per alias, sequentially). Once the pool is saturated, all subsequent database-backed endpoints — balance queries, block queries, and any other mirror-node service sharing the same pool — queue indefinitely or return errors, directly degrading the mirror node's ability to serve its consumers. [8](#0-7) [9](#0-8) 

## Likelihood Explanation
The `/construction/metadata` endpoint is part of the public Rosetta API, reachable by any HTTP client with no authentication. The `account_aliases` option key is visible in the codebase. The attack requires only `curl` and knowledge of the endpoint path. With Traefik middleware absent by default, there is no per-IP rate limiting or in-flight request cap in a standard deployment. Alias strings need only satisfy the format check — they do not need to correspond to real accounts. [10](#0-9) 

## Recommendation

1. **Enforce an alias count cap** inside `resolveAccountAliases` before entering the loop — e.g., reject requests where `len(strings.Split(accountAliases, ",")) > MAX_ALIASES` (a small constant such as 20).
2. **Add `http.MaxBytesReader`** in `main.go` to cap the request body size before any handler processes it.
3. **Enable Traefik middleware by default** (`global.middleware: true`) or document clearly that deployments without it are unprotected against this class of abuse.
4. **Batch alias lookups** into a single `IN (...)` query rather than one query per alias, eliminating the linear connection-time growth entirely.

## Proof of Concept

```bash
# Generate a payload with 500 syntactically valid alias tokens
ALIASES=$(python3 -c "
import base64, json
# A valid alias is a base32-encoded public key; use a dummy but format-valid value
alias = '0.0.' + '0x' + 'aa' * 32  # adjust to match NewAccountIdFromString format
print(','.join([alias] * 500))
")

curl -s -X POST http://<mirror-node>/construction/metadata \
  -H 'Content-Type: application/json' \
  -d "{\"network_identifier\":{\"blockchain\":\"Hedera\",\"network\":\"testnet\"},\"options\":{\"account_aliases\":\"$ALIASES\"}}"
```

Send this request from several concurrent terminals. Each request drives 500 sequential DB queries. With enough concurrent senders, `Pool.MaxOpenConnections` is reached and all other DB-backed endpoints begin returning errors or hanging.
