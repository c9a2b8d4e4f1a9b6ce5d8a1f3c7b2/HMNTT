# Audit Report

## Title
Missing Entity-Type Validation in `getNftsByAccountId()` Allows Contract EVM Address to Return NFT Records via Accounts Endpoint

## Summary
`AccountController.getNftsByAccountId()` resolves the path parameter via `EntityService.getEncodedId()`, which accepts any syntactically valid EVM address — including those belonging to contract entities — and returns their numeric ID without checking the entity type. The resolved ID is passed directly to `NftService.getNfts()`, which queries the `nft` table purely by `account_id`. As a result, NFT records owned by a contract are returned through the `/accounts/{evmAddress}/nfts` endpoint with no authentication required.

## Finding Description

**Verified code path:**

`GET /api/v1/accounts/{evmAddress}/nfts` → `AccountController.getNftsByAccountId()` → `EntityService.getEncodedId()` → `EntityService.getEntityIdFromEvmAddress()` → `NftService.getNfts()`

**`getNftsByAccountId()` — no entity-type check:**

`getNftsByAccountId` calls `EntityService.getEncodedId()` and immediately passes the result to `extractNftMultiUnionQuery()` and then `NftService.getNfts()`. There is no `isValidAccount()` call and no type predicate anywhere in the path. [1](#0-0) 

Compare with `listStakingRewardsByAccountId()`, which at least calls `EntityService.isValidAccount()` before proceeding (though that check is also type-blind, as shown below): [2](#0-1) 

**`entityFromEvmAddressQuery` — no `type` predicate:**

The SQL used by `getEntityIdFromEvmAddress()` selects `id` from `entity` filtered only by `deleted <> true` and `evm_address = $1`. There is no `and type = 'ACCOUNT'` clause, so any entity — account or contract — with a matching `evm_address` column will have its ID returned. [3](#0-2) 

The function returns `rows[0].id` unconditionally for any matching entity: [4](#0-3) 

**`isValidAccount()` — checks existence only, not type:**

`entityExistenceQuery` does select `type`, but `isValidAccount()` only checks `!isNil(entity)` — it never inspects the `type` field. A contract entity passes this check identically to an account entity. [5](#0-4) [6](#0-5) 

**`isValidEntityId()` / `isValidEvmAddress()` — format-only validation:**

`isValidEntityId` delegates to `isValidEvmAddress`, which matches `^(0x)?[A-Fa-f0-9]{40}$`. A contract's EVM address satisfies this trivially. No entity-type semantics are involved. [7](#0-6) [8](#0-7) 

**`NftService.getNfts()` — purely ID-based query:**

The `accountIdCondition` used in every subquery is `account_id = $1` with no join or filter on entity type. [9](#0-8) 

## Impact Explanation

1. **API boundary violation** — The `/accounts/{id}/nfts` endpoint is documented and intended for account entities. Supplying a contract's EVM address causes it to return contract NFT holdings, violating the API contract and any downstream assumptions about the entity type of results.
2. **Information disclosure** — NFT holdings of contracts (e.g., DAO treasuries, escrow contracts) are exposed through an unintended surface. While NFT data is public on-chain, the mirror node's endpoint segregation is part of its access model; collapsing that boundary is an unintended disclosure path.
3. **Downstream consumer confusion** — Clients that assume `/accounts/.../nfts` only returns data for user accounts will silently process contract NFT records as if they belong to a user account, potentially causing incorrect application behavior.

## Likelihood Explanation

Exploitation requires zero privileges and zero authentication. The attacker only needs a contract's EVM address, which is public on-chain data. The attack is a single unauthenticated HTTP GET request and is trivially scriptable against any deployed mirror-node instance.

## Recommendation

Add an entity-type check in `getNftsByAccountId()` after resolving the ID, analogous to (but stricter than) the pattern used in `listStakingRewardsByAccountId()`. Specifically:

1. Extend `EntityService.isValidAccount()` to verify that the resolved entity's `type` field equals `'ACCOUNT'` (the `entityExistenceQuery` already selects `type` — it just isn't used). [5](#0-4) 

2. Call this corrected `isValidAccount()` at the start of `getNftsByAccountId()`, throwing `NotFoundError` if the check fails, mirroring the pattern in `listStakingRewardsByAccountId()`. [2](#0-1) 

3. Optionally, add a `and type = 'ACCOUNT'` predicate to `entityFromEvmAddressQuery` so that EVM-address resolution itself rejects non-account entities at the SQL level. [3](#0-2) 

## Proof of Concept

```
# 1. Obtain the EVM address of any contract that holds NFTs (public on-chain data).
CONTRACT_EVM=0x<40-hex-chars>

# 2. Query the accounts/nfts endpoint with the contract's EVM address.
curl -s "https://<mirror-node>/api/v1/accounts/${CONTRACT_EVM}/nfts"

# Expected (correct) behaviour: 404 Not Found — no account with that address.
# Actual behaviour:             200 OK — returns NFT records owned by the contract.
```

No authentication, no special tooling, and no prior exploitation are required.
