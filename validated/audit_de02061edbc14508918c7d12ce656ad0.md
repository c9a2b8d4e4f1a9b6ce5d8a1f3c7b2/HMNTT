Audit Report

## Title
Cache Stampede (Thundering Herd) in `findStorageBatch()` Due to Mismatched TTLs Between `contractStateCache` and `slotsPerContract` Cache

## Summary
`ContractStateServiceImpl.findStorageBatch()` has no stampede protection. When `contractStateCache` entries expire (every 2 seconds), all concurrent threads that miss the cache independently issue the same large batch DB query — up to 500 simultaneous `IN (1500 slots)` queries — saturating the DB connection pool and causing cascading latency or timeout failures for all users. The attack is repeatable every 2 seconds with no authentication required.

## Finding Description

**Cache configuration** (`web3/src/main/java/org/hiero/mirror/web3/repository/properties/CacheProperties.java`):

- Line 28: `contractState = "expireAfterWrite=2s,maximumSize=25000,recordStats"` — global `(contractId, slotKey) → slotValue` cache with a 2-second hard expiry.
- Line 39: `slotsPerContract = "expireAfterAccess=5m,maximumSize=1500"` — per-contract slot-key registry with a 5-minute idle TTL. [1](#0-0) [2](#0-1) 

**Code path** (`web3/src/main/java/org/hiero/mirror/web3/service/ContractStateServiceImpl.java`):

1. `findStorage()` (lines 58–70): checks `contractStateCache`; on any miss (including post-expiry), unconditionally calls `findStorageBatch()` with no guard. [3](#0-2) 

2. `findStorageBatch()` (lines 85–122):
   - Line 91: snapshots **all** slot keys from the per-contract `slotsPerContract` Caffeine cache (up to 1,500 keys) via `getNativeCache().asMap().keySet()`.
   - Line 103: issues a single `SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (...)` with all collected keys — **no lock, no loading sentinel, no deduplication across threads**.
   - Lines 106–114: writes results back into `contractStateCache`. [4](#0-3) [5](#0-4) [6](#0-5) 

The `findStorageBatch()` method is also confirmed in the repository interface: [7](#0-6) 

**Root cause**: `contractStateCache` uses `expireAfterWrite`, meaning all entries written in the same batch expire simultaneously after 2 seconds. The `slotsPerContract` cache uses `expireAfterAccess=5m`, so slot keys remain registered long after the state values expire. There is no `AsyncLoadingCache`, no per-key lock, and no "loading" flag to coalesce concurrent misses into a single DB call. Every thread that calls `findStorage()` after the 2-second expiry independently enters `findStorageBatch()` and issues the full batch query.

**Correction on slot count**: The global `contractStateCache` holds up to 25,000 entries across all contracts, but `findStorageBatch()` reads slot keys from the per-contract `slotsPerContract` cache, which is capped at 1,500 entries per contract. Each thundering herd event therefore produces `IN (≤1500 slots)` queries, not `IN (25000 slots)` queries. The vulnerability is real within this bound.

## Impact Explanation

Each 2-second expiry cycle can produce up to 500 simultaneous `SELECT ... IN (1500 slots)` queries against the database (bounded by the default `requestsPerSecond = 500` throttle). [8](#0-7) 

The DB connection pool (typically 10–50 connections) becomes saturated, causing:
- Request queuing and latency spikes across all users of the mirror node web3 service.
- DB thread exhaustion leading to query timeouts (default `statementTimeout = 3000 ms`) and HTTP 500 errors for all concurrent callers.
- The effect repeats every 2 seconds as long as the attacker sustains traffic, making it a continuous, repeatable DoS. [9](#0-8) 

## Likelihood Explanation

An unprivileged user needs only:
1. A contract with ≥1 storage slot (any active DeFi/NFT contract on Hedera mainnet qualifies; the attack scales with slot count up to 1,500).
2. To issue `eth_call` or `eth_estimateGas` requests that read those slots, seeding the `slotsPerContract` cache (5-minute TTL).
3. To wait 2 seconds for `contractStateCache` to expire.
4. To send up to 500 concurrent requests immediately after expiry, repeating every 2 seconds.

No authentication, no privileged access, and no on-chain transactions are required. The throttle (`requestsPerSecond = 500`) is a global token-bucket refilled every second, not a per-contract or per-IP limit, so a single client can consume the entire budget against one contract. [10](#0-9) 

## Recommendation

1. **Replace the manual cache-aside pattern with Caffeine's `LoadingCache` or `AsyncLoadingCache`**: Caffeine's loading cache coalesces concurrent misses for the same key into a single loader invocation, eliminating the stampede by design.

2. **Per-contract lock/semaphore**: Introduce a `ConcurrentHashMap<EntityId, Lock>` and acquire a per-contract lock before entering `findStorageBatch()`, releasing it after the results are written back to `contractStateCache`. Threads that lose the race re-check the cache after the lock is released.

3. **Caffeine `refreshAfterWrite` instead of `expireAfterWrite`**: Using `refreshAfterWrite=2s` causes Caffeine to serve the stale value while asynchronously refreshing it in the background, eliminating the expiry window during which all threads miss simultaneously.

4. **Reduce the default `requestsPerSecond`** or add per-IP/per-contract rate limiting to bound the worst-case fan-out even if the stampede is not fully fixed.

## Proof of Concept

```
# Step 1: Seed the slotsPerContract cache for contract 0x<TARGET>
# Send eth_call requests reading 1500 distinct storage slots of the target contract.
# After these requests, slotsPerContract cache holds 1500 slot keys (TTL: 5 min).

# Step 2: Wait 2 seconds for contractStateCache to expire.
sleep 2

# Step 3: Send 500 concurrent eth_call requests targeting any slot of the same contract.
# All 500 threads simultaneously miss contractStateCache, enter findStorageBatch(),
# snapshot the same 1500 slot keys, and issue 500 identical
# SELECT slot, value FROM contract_state WHERE contract_id = ? AND slot IN (<1500 keys>)
# queries against the database.

for i in $(seq 1 500); do
  curl -s -X POST http://<mirror-node>/api/v1/contracts/call \
    -H 'Content-Type: application/json' \
    -d '{"to":"0x<TARGET>","data":"0x<SLOAD_SLOT_0>","estimate":false}' &
done
wait

# Step 4: Repeat every 2 seconds to sustain the DoS.
```

The DB receives 500 simultaneous large `IN`-clause queries every 2 seconds, exhausting the connection pool and causing timeouts for all concurrent users.
