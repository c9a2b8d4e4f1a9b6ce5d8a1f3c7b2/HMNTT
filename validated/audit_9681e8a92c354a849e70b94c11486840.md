Audit Report

## Title
Unauthenticated DB Connection Pool Exhaustion via Unbounded Alias Lookups in GraphQL `account` Query

## Summary
The GraphQL `account` endpoint accepts an `alias` field with no result caching and no application-level rate limiting. Each request with a distinct alias value triggers a direct database query via `EntityServiceImpl.getByAliasAndType()`. An unauthenticated attacker can flood the endpoint with concurrent requests carrying distinct valid alias values, exhausting the HikariCP connection pool and denying service to all users.

## Finding Description

**Cache scope — parsed documents only, not results:**

`CachedPreparsedDocumentProvider` caches by `executionInput.getQuery()` — the raw query string — not by query results. [1](#0-0) 

Default cache spec is `expireAfterWrite=1h,maximumSize=1000,recordStats`. [2](#0-1) 

When an attacker inlines the alias value directly in the query body (e.g., `{ account(input: { alias: "ABCDE" }) { id } }` vs `{ account(input: { alias: "ABCDF" }) { id } }`), each unique alias string produces a unique cache key → cache miss → re-parse + re-validate + resolver execution. Even when using GraphQL variables (same query string → cache hit), the resolver still executes and a DB call is made.

**No result caching in the service layer:**

`EntityServiceImpl.getByAliasAndType()` calls `entityRepository.findByAlias(decodeBase32(alias))` with no `@Cacheable` annotation or any in-memory result cache. [3](#0-2) [4](#0-3) 

**No rate limiting in the GraphQL module:**

`ThrottleConfiguration` / `ThrottleManagerImpl` exist only in the `web3` module and are scoped to `ContractCallRequest`. [5](#0-4) 

`GraphQlConfiguration` only applies `MaxQueryComplexityInstrumentation(200)` and `MaxQueryDepthInstrumentation(10)` — these limit structural query complexity, not request rate. [6](#0-5) 

The Helm chart defines a Traefik `inFlightReq` middleware (5 concurrent requests per IP), but it is **disabled by default** (`global.middleware: false`). The middleware template only renders when `global.middleware` is explicitly set to `true`. [7](#0-6) [8](#0-7) 

**Input validation is insufficient:**

The `@Pattern(regexp: "^[A-Z2-7]+$")` directive on `alias` only enforces character set — it does not bound alias length and does not limit request rate. [9](#0-8) 

The Jackson `maxStringLength` is 11000, permitting very long alias strings. [10](#0-9) 

**Exploit flow:**

1. Attacker generates N distinct Base32 strings matching `^[A-Z2-7]+$` (trivially: `AAAA`, `AAAB`, `AAAC`, …).
2. Sends N concurrent POST requests to `/graphql/alpha` with inlined alias values.
3. Each request: document cache miss (unique query string) → parse → validate → `AccountController.account()` → `EntityServiceImpl.getByAliasAndType()` → `entityRepository.findByAlias()` → DB query. [11](#0-10) 
4. HikariCP's default pool (10 connections for Spring Boot) is saturated; subsequent legitimate requests block or time out.

## Impact Explanation

Complete denial of service for the GraphQL API. All users — including legitimate ones querying by `entityId` or `evmAddress` — are blocked when the connection pool is exhausted. The `statementTimeout` of 10000ms means each attacker connection holds a pool slot for up to 10 seconds, requiring only ~10 concurrent requests to saturate the default pool. The attack is amplified by the document cache eviction: with `maximumSize=1000`, sustained flooding with >1000 unique query strings continuously evicts cached entries, adding CPU re-parsing overhead on top of DB pressure. [12](#0-11) 

## Likelihood Explanation

No authentication is required. The alias alphabet (`A-Z`, `2-7`) provides 32 characters; a 4-character alias yields 32⁴ = 1,048,576 unique valid values — far more than needed. The attack requires only an HTTP client capable of concurrent requests (e.g., `curl`, `ab`, `wrk`). It is repeatable indefinitely and requires no special knowledge of the system beyond the public GraphQL schema. The Traefik `inFlightReq` middleware that could partially mitigate this is disabled by default and, even when enabled, limits only 5 concurrent requests per source IP — easily bypassed with multiple source IPs or distributed attack. [13](#0-12) 

## Recommendation

1. **Add a result cache** to `EntityServiceImpl.getByAliasAndType()` using Spring's `@Cacheable` with a short TTL (e.g., 1–5 seconds), similar to the `web3` module's entity cache (`expireAfterWrite=1s,maximumSize=10000`).
2. **Add application-level rate limiting** to the GraphQL module, analogous to `ThrottleConfiguration` / `ThrottleManagerImpl` in the `web3` module.
3. **Add a `@Size` or `@Length` constraint** on the `alias` field in `account.graphqls` to bound input length to a realistic maximum (e.g., 100 characters).
4. **Enable the Traefik middleware** by default (`global.middleware: true`) and consider adding a `rateLimit` middleware entry (as done for the Rosetta chart) in addition to `inFlightReq`. [14](#0-13) 

## Proof of Concept

```bash
# Generate 20 distinct valid alias values and send concurrent requests
for i in $(seq 1 20); do
  ALIAS=$(python3 -c "import string; chars='ABCDEFGHIJKLMNOPQRSTUVWXYZ234567'; print(''.join([chars[(${i}*j)%32] for j in range(1,9)]))")
  curl -s -X POST http://<host>/graphql/alpha \
    -H 'Content-Type: application/json' \
    -d "{\"query\": \"{ account(input: { alias: \\\"AAAA${i}BBBB\\\" }) { id } }\"}" &
done
wait
# With default HikariCP pool of 10, legitimate requests will time out
```

Each request carries a unique inlined alias → unique query string → cache miss → `findByAlias()` DB call → pool slot held for up to 10 seconds. [3](#0-2) [1](#0-0)
