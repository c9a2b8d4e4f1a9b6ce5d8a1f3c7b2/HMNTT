Audit Report

## Title
Historical State Bypass via Missing `timestamp_range` Upper-Bound in `findActiveByEvmAddressOrAliasAndTimestamp` Entity Branch

## Summary
`findActiveByEvmAddressOrAliasAndTimestamp` resolves an entity by evm_address or alias using `created_timestamp <= ?2` in the CTE, but the `entity` branch of the UNION fetches the current live row with no `lower(timestamp_range) <= ?2` guard. When an entity has been updated after the queried `blockTimestamp`, the live row's `timestamp_range` lower-bound is later than any matching `entity_history` row, so `ORDER BY timestamp_range DESC LIMIT 1` selects the post-timestamp state. The sibling method `findActiveByIdAndTimestamp` correctly applies `lower(timestamp_range) <= ?2` to the entity branch, confirming this is an unintentional omission. The same defect is present in the closely related `findActiveByEvmAddressAndTimestamp` method.

## Finding Description

**Affected methods and exact lines:**

`findActiveByEvmAddressOrAliasAndTimestamp` — `web3/src/main/java/org/hiero/mirror/web3/repository/EntityRepository.java`, lines 95–121: [1](#0-0) 

The `entity` branch (lines 103–108) has no `lower(timestamp_range) <= ?2` predicate: [2](#0-1) 

The `entity_history` branch (lines 109–117) correctly applies `lower(eh.timestamp_range) <= ?2`: [3](#0-2) 

The same defect exists in `findActiveByEvmAddressAndTimestamp` (lines 59–85), whose `entity` branch is also unguarded: [4](#0-3) 

**Contrast with the correct implementation** — `findActiveByIdAndTimestamp` (lines 136–155) applies `lower(timestamp_range) <= ?2` to the `entity` branch: [5](#0-4) 

**Root cause:** The `entity_cte` only proves the entity *existed* at `blockTimestamp` via `created_timestamp <= ?2`. It does not constrain which *version* of the entity is returned. The `entity` table always holds the most-recent live state; its `timestamp_range` lower-bound equals the timestamp of the last update. If that update occurred after `blockTimestamp`, the live row has a higher `lower(timestamp_range)` than any `entity_history` row valid at `blockTimestamp`, so it wins the `ORDER BY timestamp_range DESC LIMIT 1` sort.

**Exploit flow:**

1. Entity `E` is created at `T1 = 100` with state `S1` (e.g., `balance = 1000`, `key = K1`). Live row: `timestamp_range = [100, ∞)`.
2. At `T3 = 300`, `E` is updated (`balance = 500`, `key = K2`). The old state `S1` is moved to `entity_history` with `timestamp_range = [100, 300)`. The live row now has `timestamp_range = [300, ∞)`.
3. Caller queries with `blockTimestamp = T2 = 200` (i.e., `T1 < T2 < T3`), targeting `E`'s evm_address or alias.
4. `entity_cte` resolves `E`'s ID (`created_timestamp = 100 ≤ 200`). ✓
5. `entity` branch returns `S2` (`balance=500`, `timestamp_range=[300,∞)`). No timestamp check blocks it.
6. `entity_history` branch returns `S1` (`balance=1000`, `timestamp_range=[100,300)`). `lower(100) ≤ 200`. ✓
7. `ORDER BY timestamp_range DESC LIMIT 1`: `lower(300) > lower(100)` → `S2` is selected.
8. **Result:** The API returns `balance=500, key=K2` for block `T2=200`, but the correct answer is `balance=1000, key=K1`.

**Test coverage gap:** The existing tests do not cover this scenario. The tests at lines 431–455 set up entity history with a *higher* `lower(timestamp_range)` than the live entity (an artifact of `domainBuilder`'s auto-incrementing timestamps), which is the inverse of the production scenario. No test exercises the case where `T1 < blockTimestamp < T3` with a live row updated after `blockTimestamp`. [6](#0-5) 

The call site in `CommonEntityAccessor.get(Bytes, Optional<Long>)` routes alias/evm_address lookups directly to `findActiveByEvmAddressOrAliasAndTimestamp`: [7](#0-6) 

## Impact Explanation

Any caller of `eth_call`, `eth_getBalance`, `eth_getCode`, or any EVM simulation that resolves an account or contract by evm_address or alias at a historical block will receive the **current** entity state instead of the state valid at the requested block:

- **Balance misrepresentation:** A contract or account's balance at a past block is reported as its current balance, enabling false proof-of-funds assertions.
- **Key/admin misrepresentation:** The admin key, auto-renew settings, or contract bytecode at a past block is reported as the current value, corrupting historical audit trails.
- **Smart-contract simulation integrity:** `eth_call` at a past block is the foundation for off-chain verification in bridges, oracles, and governance systems. Returning wrong state silently breaks these guarantees.
- **Scope:** Every entity looked up by evm_address or alias (not by numeric ID) via a historical timestamp is affected. This covers the majority of EVM-compatible address lookups.

The returned data is silently wrong with no error signal to the caller (HTTP 200, well-formed response).

## Likelihood Explanation

- **No authentication required.** The `blockTimestamp` parameter is derived from the user-supplied block number in standard JSON-RPC calls. Any anonymous user can supply an arbitrary past block number.
- **Precondition is common.** Any entity that has ever been updated (key rotation, balance change, contract upgrade) satisfies the precondition. On a live network this is the majority of active accounts and contracts.
- **Trivially repeatable.** The attacker only needs a valid evm_address or alias and a block number prior to the entity's last update. No special privileges, no gas, no on-chain transaction required.
- **Undetectable by the caller.** The API returns a well-formed result with no indication that the state is from the wrong point in time.

## Recommendation

Add `and lower(e.timestamp_range) <= ?2` to the `entity` branch in both affected methods, mirroring the pattern already used in `findActiveByIdAndTimestamp`:

```sql
-- findActiveByEvmAddressOrAliasAndTimestamp, entity branch (fix)
select *
from entity e
where e.deleted is not true
  and e.id = (select id from entity_cte)
  and lower(e.timestamp_range) <= ?2   -- ADD THIS
```

Apply the same fix to `findActiveByEvmAddressAndTimestamp`. Add integration tests that explicitly cover the scenario where `T1 < blockTimestamp < T3` (entity created before, updated after the queried timestamp) to prevent regression.

## Proof of Concept

**Setup (SQL):**
```sql
-- Entity created at T1=100
INSERT INTO entity (id, evm_address, alias, balance, key, created_timestamp, timestamp_range, deleted)
VALUES (1, '\xABCD', '\xABCD', 1000, 'K1', 100, '[100,300)', false);

-- At T3=300, entity is updated; old state moved to history
INSERT INTO entity_history (id, evm_address, alias, balance, key, created_timestamp, timestamp_range, deleted)
VALUES (1, '\xABCD', '\xABCD', 1000, 'K1', 100, '[100,300)', false);

UPDATE entity SET balance = 500, key = 'K2', timestamp_range = '[300,)'
WHERE id = 1;
```

**Query (simulating blockTimestamp = T2 = 200):**
```sql
with entity_cte as (
    select id from entity
    where created_timestamp <= 200 and (evm_address = '\xABCD' or alias = '\xABCD')
    order by created_timestamp desc limit 1
)
(select *, 'live' as src from entity e
 where e.deleted is not true and e.id = (select id from entity_cte))
union all
(select *, 'history' as src from entity_history eh
 where lower(eh.timestamp_range) <= 200 and eh.id = (select id from entity_cte)
 order by lower(eh.timestamp_range) desc limit 1)
order by timestamp_range desc limit 1;
```

**Actual result:** Returns the live row (`balance=500, key=K2, timestamp_range=[300,∞)`) — the state from `T3=300`.

**Expected result:** Should return the history row (`balance=1000, key=K1, timestamp_range=[100,300)`) — the state valid at `T2=200`.
