Based on my code investigation, I can verify the key claims against the actual source.

**Confirmed facts:**

1. `getContractOpcodes()` accepts `storage` with no auth check, only gzip header + 1 RPS throttle. [1](#0-0) 

2. `captureStorage()` builds a **new** `TreeMap<String,String>` on every opcode step by iterating all cumulative `getTxStorageUsage(true).accesses()`. [2](#0-1) 

3. Each `Opcode` object stores its own storage map reference; all opcodes are accumulated in an unbounded `ArrayList`. [3](#0-2) 

4. `server.compression` is absent from the repository entirely — no `application.yml` exists under `web3/src/main/resources/` (only `banner.txt`), and `grep_search` for `server.compression` returns zero matches.

5. The 1 RPS throttle is frequency-only, not size-bounded. [4](#0-3) [5](#0-4) 

**Important mitigations the report understates:**

- `hexCache` (max 1600 entries) means key/value `String` objects are **shared** across all `TreeMap` instances — only `TreeMap.Entry` node objects (~48 bytes each) are newly allocated per opcode step, not full string copies. [6](#0-5) 

- `maxGasLimit` is 15,000,000 gas, which naturally bounds both opcode count and SSTORE count simultaneously (each cold SSTORE costs ≥2,100 gas). The report's "50,000 opcodes × 500 slots" scenario is unrealistic within a single gas budget. [7](#0-6) 

- The endpoint is **disabled by default** (`enabled = false`). [8](#0-7) 

**SECURITY.md scope check:**

The SECURITY.md lists "Impacts that only require DDoS" as out of scope and "Any denial-of-service attacks that are executed against project assets" as a prohibited activity. However, this vulnerability is a **single-request server-side resource exhaustion** (code-level allocation bug), not a volumetric DDoS. Single-request OOM from unbounded allocation is a distinct class of vulnerability that is not excluded by those clauses. [9](#0-8) [10](#0-9) 

The core vulnerability is real and confirmed. The severity numbers are overstated but the mechanism is valid.

# Audit Report

## Title
Unbounded Per-Opcode Storage `TreeMap` Allocation in `captureStorage()` Without Server-Side Compression

## Summary
`OpcodesController.getContractOpcodes()` with `storage=true` causes `captureStorage()` in `AbstractOpcodeTracer` to allocate a new `TreeMap<String,String>` on every EVM opcode step, containing all cumulative storage accesses seen so far. There is no cap on the number of opcodes, no cap on storage slots, no pagination, and no server-side HTTP compression configured. The `Accept-Encoding: gzip` header check validates only the client's declaration; it does not activate server-side compression. The full response object is materialized in JVM heap before any serialization.

## Finding Description

**Exact code path:**

`OpcodesController.getContractOpcodes()` (lines 52–68) accepts `storage` as an unauthenticated boolean query parameter. The only size-related guard is the `Accept-Encoding: gzip` header check at line 60, which validates only that the client declares it accepts gzip. [1](#0-0) 

**Root cause — `captureStorage()` in `AbstractOpcodeTracer` (lines 75–129):**

For every opcode step during EVM replay, `captureStorage()` calls `getTxStorageUsage(true).accesses()` — the cumulative set of every storage slot touched so far — and builds a **new** `TreeMap<String,String>`. This map is embedded in each `Opcode` object and accumulated in an unbounded `ArrayList`. [11](#0-10) 

`tracePostExecution()` calls `captureStorage()` on every single opcode step: [3](#0-2) 

**No server-side compression:** No `application.yml` exists under `web3/src/main/resources/` and no `server.compression.enabled` property is set anywhere in the repository. The full response is serialized to JSON in heap before any network transmission.

**Throttle is rate-only:** The 1 RPS `opcodeRateLimitBucket` limits frequency but a single request can still trigger large allocation. [4](#0-3) 

## Impact Explanation

- **Server-side heap exhaustion**: O(opcodes × storage_slots) `TreeMap.Entry` objects are allocated in heap before serialization. With `maxGasLimit = 15,000,000`, a transaction using ~3-gas opcodes in a loop with repeated SSTORE operations can produce tens of thousands of opcodes each carrying a growing storage map, potentially exhausting JVM heap and triggering `OutOfMemoryError`.
- **Large uncompressed response**: Without server-side gzip, the full JSON payload traverses the network, saturating TCP buffers.
- **Severity: Medium–High** when the endpoint is enabled. The `hexCache` (max 1600 entries) means string objects are shared across `TreeMap` instances, reducing raw memory pressure compared to the worst-case estimate in the report, but `TreeMap.Entry` node objects (~48 bytes each) are still freshly allocated per opcode per storage slot.

## Likelihood Explanation

- The endpoint is **disabled by default** (`hiero.mirror.web3.opcode.tracer.enabled=false`), which significantly reduces exposure.
- When enabled, no authentication or authorization is required.
- The attacker only needs a valid transaction hash for a storage-heavy contract (publicly visible on-chain).
- The `Accept-Encoding: gzip` header is trivially added by any HTTP client.
- The 1 RPS throttle does not prevent a single oversized request from causing heap pressure.
- The same transaction hash can be replayed indefinitely. [8](#0-7) 

## Recommendation

1. **Enable server-side compression**: Add `server.compression.enabled=true`, `server.compression.mime-types=application/json`, and `server.compression.min-response-size=1024` to the web3 `application.yml`.
2. **Cap the storage map size**: In `captureStorage()`, limit the number of entries in the returned `TreeMap` (e.g., to a configurable `maxStorageEntriesPerOpcode`).
3. **Cap total opcodes**: Add a configurable `maxOpcodes` limit in `OpcodeContext.addOpcodes()` that throws or truncates beyond the threshold.
4. **Stream the response**: Instead of materializing the full `OpcodesResponse` in heap, use streaming JSON serialization (e.g., `StreamingResponseBody` or `ResponseBodyEmitter`) to write opcodes incrementally.
5. **Consider authentication**: Gate the endpoint behind an API key or IP allowlist given its resource-intensive nature.

## Proof of Concept

```bash
# 1. Find a historical transaction hash for a contract with many SSTOREs
TX_HASH="0x<hash_of_storage_heavy_tx>"

# 2. Send request with storage=true and the required gzip header
curl -v \
  -H "Accept-Encoding: gzip" \
  "https://<mirror-node>/api/v1/contracts/results/${TX_HASH}/opcodes?storage=true&stack=false&memory=false"

# 3. The EVM replay fires captureStorage() at every opcode step.
# Each step allocates a new TreeMap containing all cumulative storage accesses.
# With a transaction touching N storage slots across K opcodes:
#   - K * N TreeMap.Entry objects (~48 bytes each) are allocated in heap
#   - The full JSON response is built before any network write
# Monitor JVM heap: jcmd <pid> VM.native_memory or observe GC logs for OOM
```

    protected Map<String, String> captureStorage(
            final MessageFrame frame, final OpcodeContext options, final ContractCallContext context) {
        if (!options.isStorage()) {
            return Collections.emptyMap();
        }

        try {
            if (context.getOpcodeContext().getRootProxyWorldUpdater() == null) {

                var worldUpdater = frame.getWorldUpdater();
                var parent = worldUpdater.parentUpdater().orElse(null);
                while (parent != null) {
                    worldUpdater = parent;
                    parent = worldUpdater.parentUpdater().orElse(null);
                }

                if (!(worldUpdater instanceof RootProxyWorldUpdater rootProxyWorldUpdater)) {
                    // The storage updates are kept only in the RootProxyWorldUpdater.
                    // If we don't have one -> something unexpected happened and an attempt to
                    // get the storage changes from a ProxyWorldUpdater would result in a
                    // NullPointerException, so in this case just return an empty map.
                    return Collections.emptyMap();
                }

                context.getOpcodeContext().setRootProxyWorldUpdater(rootProxyWorldUpdater);
            }

            final var rootProxyWorldUpdater = context.getOpcodeContext().getRootProxyWorldUpdater();
            final var updates = rootProxyWorldUpdater
                    .getEvmFrameState()
                    .getTxStorageUsage(true)
                    .accesses();

            if (updates.isEmpty()) {
                return Collections.emptyMap();
            }

            final var result = new TreeMap<String, String>();
            for (final var storageAccesses : updates) {
                for (final var access : storageAccesses.accesses()) {
                    final var key = hexCache.get(access.key(), Bytes::toHexString);
                    if (!result.containsKey(key)) {
                        final var value = access.writtenValue() != null
                                ? hexCache.get(access.writtenValue(), Bytes::toHexString)
                                : hexCache.get(access.value(), Bytes::toHexString);
                        result.put(key, value);
                    }
                }
            }
            return result;

        } catch (final ModificationNotAllowedException e) {
            return Collections.emptyMap();
        }
    }

    public void tracePostExecution(@NonNull final MessageFrame frame, @NonNull final OperationResult operationResult) {
