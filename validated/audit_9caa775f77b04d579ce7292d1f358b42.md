Audit Report

## Title
Shared Singleton `LinkedList` in `MirrorNodeState` Causes Cross-Request Transaction Receipt Leakage

## Summary
`MirrorNodeState` is a Spring singleton (`@Named`) that allocates a single `LinkedList` for `TRANSACTION_RECEIPTS_STATE_ID` at application startup. Both the readable and writable queue-state wrappers operate directly on this shared, unsynchronized `LinkedList` with no per-request isolation, no clearing between requests, and no synchronization. Any concurrent or subsequent request that reads the queue observes receipts written by all prior requests.

## Finding Description

`MirrorNodeState.initQueueStates()` creates one `LinkedList` for the entire process lifetime: [1](#0-0) 

`getReadableStates()` uses `computeIfAbsent` on a `ConcurrentHashMap`. On the first call for `RecordCacheService.NAME` it constructs a `ListReadableQueueState` wrapping the shared `LinkedList` and caches the entire `MapReadableStates`: [6](#0-5) 

`ListReadableQueueState.iterateOnDataSource()` returns a live iterator directly over the backing `LinkedList`: [3](#0-2) 

`getWritableStates()` also uses `computeIfAbsent` and creates a `ListWritableQueueState` wrapping the same shared `LinkedList`. Its commit callback only removes the *readable* states cache entry, leaving the writable states permanently cached: [7](#0-6) 

`ListWritableQueueState.addToDataSource()` writes directly to the shared `LinkedList` with no buffering: [5](#0-4) 

`MapWritableStates.commit()` calls `queue.commit()` on the `WritableQueueStateBase`, but this only fires the `onCommit` runnable (which evicts only the readable cache entry). The `LinkedList` itself is never cleared: [8](#0-7) 

**Root cause:** The failed assumption is that each request gets an isolated queue. In reality the `LinkedList` is a process-wide singleton. `addToDataSource()` mutates it immediately and `iterateOnDataSource()` reads it live, so every caller of `getReadableStates(RecordCacheService.NAME).getQueue(TRANSACTION_RECEIPTS_STATE_ID)` sees the accumulated receipts of all prior and concurrent requests. The queue grows monotonically and is never cleared.

## Impact Explanation
Any request that reads `TRANSACTION_RECEIPTS_STATE_ID` (e.g., during `eth_sendRawTransaction` or `eth_getTransactionReceipt` processing) will observe receipts belonging to other concurrent or prior requests. Transaction receipts contain transaction IDs, account IDs, and status information, constituting cross-user information disclosure. Additionally, the `LinkedList` grows without bound, leading to unbounded memory growth and eventual OOM. Concurrent unsynchronized access (`add()` from one thread, `iterator()` from another) on a non-thread-safe `LinkedList` can throw `ConcurrentModificationException`, crashing request-handling threads.

## Likelihood Explanation
Any unprivileged user submitting JSON-RPC requests to the mirror node's web3 endpoint triggers the write path. A second concurrent or subsequent request reading the same queue immediately observes the first request's receipt. No special privileges, authentication bypass, or precise race-condition timing is required beyond sending two overlapping requests. This is trivially repeatable under any load.

## Recommendation
1. **Per-request isolation:** Replace the singleton `LinkedList` with a per-request queue. The simplest approach is to not cache the writable states at all for `RecordCacheService.NAME`, or to create a fresh `LinkedList` on each call to `getWritableStates()` for that service and discard it after the request completes.
2. **Clear on commit:** If a shared queue is intentional (e.g., as a bounded recent-receipt cache), the commit callback must also clear the queue (or drain consumed entries) and the `writableStates` cache entry must be evicted alongside the readable one.
3. **Thread safety:** Replace `LinkedList` with a thread-safe structure such as `ConcurrentLinkedQueue` or protect all accesses with explicit synchronization.
4. **Evict writable states on commit:** The `onCommit` runnable at line 114 should also call `writableStates.remove(serviceName)` so that a fresh `ListWritableQueueState` (and a fresh backing queue) is created for each logical transaction.

## Proof of Concept

                }
