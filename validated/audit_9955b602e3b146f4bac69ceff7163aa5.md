Audit Report

## Title
Redundant Timestamp Predicate Injection via Blocks Endpoint Bypasses Strict Filter Checks

## Summary
The `/api/v1/blocks` endpoint accepts up to 100 identical `timestamp=gte:0` query parameters, each of which is mapped to a separate SQL predicate without deduplication, producing a WHERE clause with 100 redundant `consensus_end >= $N` conditions sent to PostgreSQL on every such request.

## Finding Description

All code references in the claim are accurate.

**Step 1 — Entry point:** `getBlocks` at [1](#0-0)  calls `utils.buildAndValidateFilters(req.query, acceptedBlockParameters)`. Inside `buildFilters`, the only cardinality check is `isRepeatedQueryParameterValidLength` at [2](#0-1) , which enforces `values.length <= config.query.maxRepeatedQueryParameters`. Exactly 100 identical values satisfies `≤ 100` and passes.

**Step 2 — No deduplication in filter extraction:** `extractSqlFromBlockFilters` at [3](#0-2)  maps **every** filter whose key is in `blockWhereFilters` to a WHERE condition via `getFilterWhereCondition`. There is no call to `parseTimestampFilters()` and no deduplication. 100 identical `{key:'timestamp', operator:'>=', value:'0'}` filters produce 100 identical entries.

**Step 3 — Blind iteration in SQL builder:** `buildWhereSqlStatement` at [4](#0-3)  iterates the full `whereQuery` array and emits one `consensus_end >= $N` condition per entry, producing a 100-predicate WHERE clause with all parameters bound to `0`.

**Step 4 — The guard that would prevent this is never called:** `parseTimestampFilters` at [5](#0-4)  throws `"Multiple gt or gte operators not permitted"` when `lowerBoundFilterCount > 1`, but it is only invoked in `contractController.js`, `accounts.js`, and `transactions.js` — never in `blockController.js`.

**Step 5 — Parser-level cap is not a mitigation:** `requestHandler.js` sets `arrayLimit: config.query.maxRepeatedQueryParameters` with `throwOnLimitExceeded: true` at [6](#0-5) , which rejects requests with *more than* 100 values. A payload of exactly 100 values is the maximum allowed and passes cleanly.

## Impact Explanation
Each malicious request causes PostgreSQL to receive and plan a query with 100 redundant range predicates on the `consensus_end` indexed column. While PostgreSQL can optimize simple redundant conditions, planning overhead scales with predicate count and is non-trivial under concurrent load. An attacker sending a sustained stream of such requests (no authentication required) can elevate DB CPU usage and degrade query planning for legitimate traffic. The impact is a non-network-based resource exhaustion against the database tier.

## Likelihood Explanation
The `/api/v1/blocks` endpoint is fully public and requires zero privileges. The payload is trivially constructed (a URL with 100 repeated `timestamp=gte:0` parameters). It is fully repeatable, can be parallelized across multiple connections, and requires no special knowledge beyond knowing the endpoint accepts `timestamp` as a query parameter.

## Recommendation
1. **Deduplicate in `extractSqlFromBlockFilters`:** Before mapping filters to WHERE conditions, deduplicate entries with the same `key` and `operator`, retaining only the most restrictive value (e.g., the maximum `gte` value).
2. **Apply operator-count validation:** Invoke `parseTimestampFilters()` (or an equivalent check) within the blocks endpoint flow to reject requests with multiple same-direction timestamp operators, consistent with how other endpoints handle this.
3. **Reduce the ceiling:** Consider lowering `maxRepeatedQueryParameters` for timestamp-type filters specifically, or enforce a per-operator limit of 1 for `gt`/`gte` and 1 for `lt`/`lte` at the `buildFilters` level.

## Proof of Concept
```
GET /api/v1/blocks?timestamp=gte:0&timestamp=gte:0&[...repeat 98 more times...]
```
This produces the following query sent to PostgreSQL:
```sql
SELECT ... FROM record_file
WHERE consensus_end >= $1 AND consensus_end >= $2 AND ... AND consensus_end >= $100
ORDER BY consensus_end DESC
LIMIT 25
```
with all 100 parameters bound to `0000000000`. No authentication is required. The request passes all existing validation checks.
