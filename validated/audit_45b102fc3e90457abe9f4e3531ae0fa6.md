Audit Report

## Title
Malicious Block Node Can Persistently Evade Inactivity Penalty via Deliberate `serverStatus()` Timeout

## Summary
`BlockNode.getBlockRange()` catches all exceptions — including gRPC deadline-exceeded exceptions from a timed-out `serverStatus()` call — and silently returns `EMPTY_BLOCK_RANGE` without invoking `onError()`. Because the error counter is never incremented through this path, a block node that deliberately delays every status response beyond the 400 ms `responseTimeout` is never marked inactive. It causes a 400 ms stall and an `ERROR`-level log entry on every block-selection cycle, indefinitely, with no self-healing.

## Finding Description

**`StreamProperties.java` line 35** — default `responseTimeout` is 400 ms: [1](#0-0) 

**`BlockNode.getBlockRange()` lines 99–112** — applies the deadline, then catches every exception and returns `EMPTY_BLOCK_RANGE` without calling `onError()`: [2](#0-1) 

**`onError()` lines 196–207** — the inactivity/penalty mechanism is only reachable from `streamBlocks()`'s catch blocks (lines 159–164), never from `getBlockRange()`: [3](#0-2) [4](#0-3) 

**`BlockNodeSubscriber.hasBlock()` lines 133–145** — `EMPTY_BLOCK_RANGE` (`Range.closedOpen(0L, 0L)`) is empty, so `blockRange.isEmpty()` returns `true` and the node is silently skipped: [5](#0-4) 

**`BlockNodeSubscriber.getNode()` lines 108–131** — if no node passes `hasBlock()`, throws `BlockStreamException`: [6](#0-5) 

**`BlockNodeDiscoveryService.discover()` lines 73–92** — when `autoDiscoveryEnabled=true`, any `BLOCK_NODE`-typed registered node is pulled from `RegisteredNodeRepository` and added to the active node list: [7](#0-6) 

**Root cause — failed assumption:** The design assumes that a node failing `serverStatus()` is merely "empty" (has no blocks yet) rather than "misbehaving." Consequently, no penalty is applied. A malicious node exploits this gap to remain permanently in the active node list while being permanently useless.

## Impact Explanation

- **Single malicious node among healthy peers:** Every `getNode()` call incurs a 400 ms stall waiting for the deadline to expire, plus an `ERROR`-level log entry. Healthy nodes still serve blocks, but the stall repeats on every scheduled block-fetch cycle with no self-healing.
- **Sole configured or discovered node:** `getNode()` throws `BlockStreamException` every cycle. `CompositeBlockSource.SourceHealth` counts 3 consecutive failures (`errors.get() < 3` threshold at line 88) and switches away from `BLOCK_NODE` source in `AUTO` mode, or fails completely in `BLOCK_NODE`-only mode, halting block ingestion. [8](#0-7) 
- **Multiple malicious nodes:** Stalls are additive (N × 400 ms per cycle), compounding the delay proportionally.

## Likelihood Explanation

The attacker needs only to operate a reachable block node that delays every gRPC `serverStatus` response beyond 400 ms (trivially implemented with `Thread.sleep` in the server handler). Entry is possible either via manual operator configuration or — more accessibly — by registering a tier-1 block node endpoint (STATUS + PUBLISH + SUBSCRIBE_STREAM) in the network's node registry when auto-discovery is enabled. The attack is repeatable indefinitely because the node is never penalized, requires no ongoing authentication, and survives importer restarts.

## Recommendation

1. **Call `onError()` on timeout in `getBlockRange()`:** Distinguish between a timeout/connection error (misbehaving node) and a legitimate empty range response. On any exception from `serverStatus()`, call `onError()` so the inactivity counter is incremented and the node is eventually marked inactive.
2. **Separate "no blocks available" from "unreachable/timed out":** Return a distinct sentinel (e.g., `null` or a typed result) from `getBlockRange()` to allow callers to differentiate between a healthy-but-empty node and a failing node.
3. **Cap cumulative timeout per `getNode()` cycle:** Bound the total time spent probing all nodes in a single `getNode()` call to prevent additive stalls with multiple malicious nodes.

## Proof of Concept

1. Stand up a gRPC server implementing `BlockNodeService` that, on every `serverStatus()` call, sleeps for 500 ms before responding (or never responds, letting the deadline expire).
2. Register this server as a block node (manually in config, or via the network registry with `autoDiscoveryEnabled=true`).
3. Observe that on every scheduled `BlockNodeSubscriber.doGet()` invocation:
   - `getBlockRange()` blocks for 400 ms, then logs `ERROR: Failed to get server status for BlockNode(...)`.
   - `hasBlock()` returns `false` (empty range).
   - If this is the only node, `getNode()` throws `BlockStreamException`.
   - `BlockNode.errors` remains `0`; `active` remains `true`.
4. Restart the importer — the node is immediately re-added to the active list and the cycle repeats without any cooldown.

    private BlockNode getNode(final AtomicLong nextBlockNumber) {
        final var nodeList = getBlockNodes();
        final var inactiveNodes = new ArrayList<BlockNode>();
        for (final var node : nodeList) {
            if (!node.tryReadmit(false).isActive()) {
                inactiveNodes.add(node);
                continue;
            }

            if (hasBlock(nextBlockNumber, node)) {
                return node;
            }
        }

        // find the first inactive node with the block and force activating it
        for (final var node : inactiveNodes) {
            if (hasBlock(nextBlockNumber, node)) {
                node.tryReadmit(true);
                return node;
            }
        }

        throw new BlockStreamException("No block node can provide block " + nextBlockNumber.get());
    }
