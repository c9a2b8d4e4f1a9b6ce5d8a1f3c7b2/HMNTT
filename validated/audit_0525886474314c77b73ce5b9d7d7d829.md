Audit Report

## Title
Unbounded Inner Transaction Fetch and In-Memory Accumulation in `getTransactionsByIdOrHash` Enables Memory-Based DoS

## Summary
`getTransactionsByIdOrHash()` in `rest/transactions.js` fetches all inner transactions of an atomic batch transaction from the database with no application-level cap, spreads the result array directly into the existing `rows` array via `rows.push(...innerTransactionRows)`, and passes the entire combined array to `formatTransactionRows()` for synchronous in-memory processing. An attacker who creates a large atomic batch transaction can then repeatedly trigger unbounded memory allocation by querying the corresponding transaction ID — a completely unprivileged read-only operation.

## Finding Description

In `rest/transactions.js`, `getTransactionsByIdOrHash()` (lines 923–972):

```js
// lines 944–947: collect ALL inner_transactions from all rows, no size cap
const innerTransactions = rows
  .map((row) => row.inner_transactions)
  .filter((innerTransactions) => innerTransactions)
  .flat();

// lines 949–958: second DB query for ALL inner transactions, no LIMIT
if (!isTransactionHash && innerTransactions.length > 0) {
  const {query: innerTransactionQuery, params} = getTransactionsByTransactionIdsSql(
    innerTransactions,
    filters,
    Transaction.VALID_START_NS
  );
  const {rows: innerTransactionRows} = await pool.queryQuietly(innerTransactionQuery, params);

  rows.push(...innerTransactionRows);   // ← spread of unbounded array
}

const transactions = await formatTransactionRows(rows);  // ← full in-memory processing
``` [6](#0-5) 

`getTransactionsByTransactionIdsSql` builds one OR condition per inner transaction pair and returns `getTransactionQuery(mainConditions, subqueryConditions)` with **no LIMIT clause**. [7](#0-6) 

The `inner_transactions` column is a raw `bigint[]` with no application-level size enforcement at the REST layer. [4](#0-3) 

`acceptedSingleTransactionParameters` only permits `nonce` and `scheduled` — there is no `limit` parameter available for this endpoint. [3](#0-2) 

**Root cause:** No guard on `innerTransactions.length` before issuing the second query, no `LIMIT` in `getTransactionsByTransactionIdsSql` for this code path, and no cap on the combined `rows` array before `formatTransactionRows`.

**Failed assumption:** The code assumes the number of inner transactions in a batch is small enough to be safely fetched and held in memory in a single request. The design document for HIP-551 explicitly states the intent to "query all `payer_account_id` and `valid_start_ns` combinations contained in `transaction.inner_transactions`" with no mention of a limit. [8](#0-7) 

**Why existing checks are insufficient:**
- The `!isTransactionHash` guard (line 949) only prevents the inner-transaction fetch for hash-based lookups; it does not limit the count for ID-based lookups.
- The `scheduled !== undefined` check (line 964) is unrelated to inner transaction count.
- `acceptedSingleTransactionParameters` (line 1014) allows no `limit` parameter for this endpoint.

**Note on `RangeError` claim:** The claim that `rows.push(...innerTransactionRows)` could throw `RangeError: Maximum call stack size exceeded` (requiring ~65,536 elements) is overstated. Hedera's outer transaction byte budget (~6 KB) practically limits the number of inner transactions to tens, not tens of thousands. This specific sub-claim does not hold under realistic network constraints.

## Impact Explanation

Each inner transaction row returned from the DB includes `jsonb_agg`-aggregated `crypto_transfer_list`, `token_transfer_list`, and `assessed_custom_fees` subqueries. [9](#0-8) 

A batch with N inner transactions causes the Node.js process to allocate memory proportional to N × (row size including all aggregated transfer lists). Under concurrent requests targeting the same large batch transaction ID, memory consumption multiplies linearly. The combined effect is elevated per-request memory pressure and increased response latency, degrading service for all API consumers. The `RangeError` crash vector is not realistic given Hedera's transaction size constraints, but the memory amplification per request is real.

## Likelihood Explanation

**Setup phase (one-time, requires fees):** Any Hedera account holder can submit an atomic batch transaction. The cost is proportional to the number of inner transactions. This is a realistic capability for a motivated attacker.

**Exploit phase (fully unprivileged, repeatable):** Once the batch transaction exists in the database, any external user — with no credentials, no HBAR, and no special access — can trigger the memory amplification by issuing GET requests to the public REST API. The endpoint requires no authentication. The exploit is trivially repeatable and scriptable. Any large batch transaction created by any user (including legitimate ones) becomes a permanent amplification vector. [10](#0-9) 

## Recommendation

1. **Add an application-level cap** on `innerTransactions.length` before issuing the second query. If the count exceeds a configurable threshold (e.g., 100), return an error or paginate.
2. **Add a `LIMIT` clause** in `getTransactionsByTransactionIdsSql` when called for inner transaction lookups, or enforce it at the call site in `getTransactionsByIdOrHash`.
3. **Use `Array.prototype.concat` or `push` in a loop** instead of `rows.push(...innerTransactionRows)` to avoid any spread-related stack concerns, even if the `RangeError` is not currently reachable.
4. **Consider pagination** for the inner transactions sub-response, consistent with how the main `/api/v1/transactions` endpoint handles large result sets.

## Proof of Concept

1. Submit an atomic batch transaction to the Hedera network containing the maximum number of inner transactions (each with the maximum number of crypto/token transfers).
2. Wait for the mirror node importer to write the batch parent row (with a large `inner_transactions bigint[]`) and all inner transaction rows to the database.
3. Send repeated GET requests to `/api/v1/transactions/<batchTransactionId>`.
4. Each request triggers: initial DB query → extraction of all inner transaction keys (no cap) → second DB query returning N full rows with JSON-aggregated transfer lists (no LIMIT) → `rows.push(...innerTransactionRows)` → `formatTransactionRows(rows)` holding all N+1 rows in memory simultaneously.
5. Under concurrent load, multiple such requests multiply the memory pressure linearly, degrading or exhausting the Node.js process heap.

const getTransactionsByTransactionIdsSql = (transactionKeys, filters, timestampField) => {
  let payerAccountParams = [];
  let params = [];

  let minTimestamp = constants.MAX_LONG;
  let maxTimestamp = -1;
  let idConditions = [];

  if (transactionKeys.length % 2 !== 0) {
    throw new InvalidArgumentError('transaction keys must be in sequenced pairs of [payer,timestamp]');
  }

  for (let index = 0; index < transactionKeys.length; index += 2) {
    let paramIndex = params.length;
    const payer = transactionKeys[index];
    const timestamp = BigInt(transactionKeys[index + 1]);

    if (payer) {
      payerAccountParams.push(`$${++paramIndex}`);
      idConditions.push(`(${Transaction.PAYER_ACCOUNT_ID} = $${paramIndex} and 
                               ${timestampField} = $${++paramIndex})`);
      params.push(payer);
    } else if (timestampField === Transaction.VALID_START_NS) {
      throw new InvalidArgumentError('payer is required when timestamp is valid_start_ns');
    } else {
      idConditions.push(`${timestampField} = $${++paramIndex}`);
    }
    params.push(timestamp);

    if (timestamp < minTimestamp) {
      minTimestamp = timestamp;
    }
    if (timestamp > maxTimestamp) {
      maxTimestamp = timestamp;
    }
  }

  const maxConsensusTimestamp =
    timestampField === Transaction.CONSENSUS_TIMESTAMP
      ? transactionKeys[transactionKeys.length - 1]
      : maxTimestamp + maxTransactionConsensusTimestampRangeNs;
  const commonConditions = [
    `${Transaction.CONSENSUS_TIMESTAMP} >= $${params.length + 1}`,
    `${Transaction.CONSENSUS_TIMESTAMP} <= $${params.length + 2}`,
  ];
  params.lowerConsensusTimestampIndex = params.length;
  params.push(
    timestampField === Transaction.VALID_START_NS
      ? minTimestamp - config.query.maxValidStartTimestampDriftNs
      : minTimestamp
  );
  params.upperConsensusTimestampIndex = params.length;
  params.push(maxConsensusTimestamp);

  const {conditions: filterConditions, scheduled} = getTransactionByIdQueryParamConditions(filters, params);
  const mainConditions = [...commonConditions, `(${idConditions.join(' or ')})`, ...filterConditions].join(' and ');
  const subqueryConditions = payerAccountParams.length
    ? [...commonConditions, `${Transaction.PAYER_ACCOUNT_ID} in (${payerAccountParams.join(',')})`].join(' and ')
    : commonConditions.join(' and ');

  return {
    query: getTransactionQuery(mainConditions, subqueryConditions),
    params,
    scheduled,
  };
};
```
