Audit Report

## Title
Unbounded Subscription via `startTime=0` and `limit=Long.MAX_VALUE` Causes Indefinite Pipeline Stage Exhaustion

## Summary
An unauthenticated gRPC client can open a topic subscription with `startTime=0`, no `endTime`, and `limit=Long.MAX_VALUE`. The server accepts all three values without restriction, causing the historical retriever to scan all stored messages from epoch, then seamlessly transitioning to a live subscription that never terminates. No server-side mechanism caps either stage or limits concurrent subscribers.

## Finding Description

**Filter validation — `TopicMessageFilter.java`:**

`limit` is annotated only with `@Min(0)`, so `Long.MAX_VALUE` is a valid value. [1](#0-0) 

`startTime` is annotated `@Min(0)` and validated by `isValidStartTime()` which only checks `startTime <= DomainUtils.now()`. The value `0` (Unix epoch) satisfies both constraints. [2](#0-1) 

`endTime` is a plain nullable `Long` with no `@NotNull`, making it entirely optional. [3](#0-2) 

**Stage 1 — historical retriever (`TopicMessageServiceImpl.java` line 63):**

The raw filter (with `startTime=0` and no `endTime`) is passed directly to the retriever with no server-side floor or cap applied beforehand. [4](#0-3) 

**Stage 2 — live subscription never terminates:**

`pastEndTime()` explicitly returns `Flux.never()` when `endTime == null`, so `takeUntilOther` never fires. [5](#0-4) 

`take(filter.getLimit())` is called when `hasLimit()` is true. Since `hasLimit()` returns `limit > 0`, and `Long.MAX_VALUE > 0`, `take(Long.MAX_VALUE)` is invoked — which in Project Reactor is functionally equivalent to no limit. [6](#0-5) 

`isComplete()` unconditionally returns `false` when `endTime == null`, keeping the safety-check polling loop alive indefinitely. [7](#0-6) 

**No subscriber cap:**

`subscriberCount` is registered only as a Micrometer `Gauge` metric. It is never compared against a maximum to reject new subscriptions. [8](#0-7) 

**No rate limiting in `GrpcProperties`:**

`GrpcProperties` exposes no `maxSubscribers`, no per-IP rate limit, and no `maxStartTimeAge` property. The only relevant properties are `endTimeInterval`, `entityCacheSize`, `checkTopicExists`, and `netty`. [9](#0-8) 

## Impact Explanation
Each malicious subscription holds open one or more DB connections (paginated historical queries from epoch), a `boundedElastic` scheduler thread (safety-check polling), and a live-listener slot — all indefinitely. An attacker opening N such subscriptions exhausts the DB connection pool, starves legitimate subscribers, and renders the gRPC service unresponsive. Cleanup only occurs if the client sends a gRPC cancel signal (`doFinally` at line 90–91), which an attacker simply never does. [10](#0-9) 

## Likelihood Explanation
The service is publicly reachable with no authentication. No per-IP rate limit, no maximum subscriber count, no server-enforced `endTime`, and no upper bound on `limit` exist anywhere in the codebase. The attack requires only a standard gRPC client library and a single call. It is trivially scriptable to open hundreds of such subscriptions simultaneously.

## Recommendation
1. **Cap `startTime` lookback**: Enforce a maximum age for `startTime` (e.g., reject requests where `DomainUtils.now() - startTime` exceeds a configurable threshold).
2. **Enforce a server-side `limit` ceiling**: Add a `@Max` constraint or clamp `limit` to a configurable maximum (e.g., 10,000) in `TopicMessageFilter`.
3. **Require or default `endTime`**: Either make `endTime` mandatory or apply a server-side default maximum subscription duration.
4. **Enforce a maximum concurrent subscriber count**: Compare `subscriberCount` against a configurable maximum in `subscribeTopic()` and reject new subscriptions when the limit is reached.
5. **Add per-IP or per-client rate limiting** at the gRPC interceptor layer.

## Proof of Concept
```java
// Using grpc-java
ManagedChannel channel = ManagedChannelBuilder.forAddress("mirror-node-host", 5600)
    .usePlaintext().build();
ConsensusServiceGrpc.ConsensusServiceStub stub =
    ConsensusServiceGrpc.newStub(channel);

ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(1).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).setNanos(0).build())
    // No consensusEndTime set
    .setLimit(Long.MAX_VALUE)
    .build();

// Open N such subscriptions — each holds DB connections and threads indefinitely
for (int i = 0; i < 500; i++) {
    stub.subscribeTopic(query, new StreamObserver<ConsensusTopicResponse>() {
        public void onNext(ConsensusTopicResponse r) {}
        public void onError(Throwable t) {}
        public void onCompleted() {}
        // Never cancel — attacker simply holds the stream open
    });
}
// DB connection pool and boundedElastic thread pool are now exhausted
```
