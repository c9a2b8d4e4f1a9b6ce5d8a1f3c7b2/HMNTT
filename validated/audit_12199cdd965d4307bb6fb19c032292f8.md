Audit Report

## Title
Unauthenticated `timestamp=ne:` Operator Accepted on `/api/v1/balances`, Enabling Silent Empty 200 Response via Treasury Snapshot Exclusion

## Summary
The `/api/v1/balances` endpoint accepts the `ne` (not-equals) operator for the `timestamp` parameter because `filterValidityChecks` validates only the value format for `TIMESTAMP`, not the operator. An unauthenticated attacker can use `timestamp=gte:T&lte:T&ne:T` to make `getAccountBalanceTimestampRange` return zero rows, causing `getBalances` to silently return `{timestamp: null, balances: [], links: {next: null}}` with HTTP 200 — indistinguishable from a legitimate empty result.

## Finding Description

**Validation gap in `filterValidityChecks`:**

The `TIMESTAMP` case in `filterValidityChecks` only validates the value format:

```js
case constants.filterKeys.TIMESTAMP:
  ret = isValidTimestampParam(val);
  break;
``` [1](#0-0) 

It does not restrict the operator. By contrast, `BLOCK_NUMBER` and `SLOT` explicitly use `basicOperators` (which excludes `ne`) to prevent this:

```js
case constants.filterKeys.BLOCK_NUMBER:
  ret = (isPositiveLong(val, true) || isHexPositiveInt(val, true)) && includes(basicOperators, op);
  break;
// ...
case constants.filterKeys.SLOT:
  ret = isValidSlot(val) && includes(basicOperators, op);
  break;
``` [2](#0-1) [3](#0-2) 

`basicOperators` is explicitly defined to exclude `ne`: [4](#0-3) 

**`ne` flows into `getOptimizedTimestampRange`:**

`getOptimizedTimestampRange` collects `ne` values into `neParams`:

```js
} else if (query.includes(utils.opsMap.ne)) {
  neParams.push(value);
}
``` [5](#0-4) 

**Treasury exclusion in `getAccountBalanceTimestampRange`:**

The `neParams` are appended to the treasury account query:

```js
if (neParams.length) {
  condition += ' and not consensus_timestamp = any ($4)';
  params.push(neParams);
}
``` [6](#0-5) 

When the treasury query returns 0 rows, the function returns `{}`: [7](#0-6) 

**Silent empty 200 path:**

`getTsQuery` propagates the empty object: [8](#0-7) 

Back in `getBalances`, a missing `query` property causes an early `return`:

```js
if (!tsQueryResult.query) {
  return;
}
``` [9](#0-8) 

This exits with the pre-initialized `res.locals` value:

```js
res.locals[constants.responseDataLabel] = {
  timestamp: null,
  balances: [],
  links: { next: null },
};
``` [10](#0-9) 

No error is raised, no log distinguishes this from a legitimate empty result, and HTTP 200 is returned.

## Impact Explanation
The silent empty 200 response is protocol-level ambiguous. Monitoring systems, operators, and downstream consumers cannot distinguish "no balance data exists" from "balance data was excluded by a crafted query." During a network partition — when operators most need accurate balance data — an attacker can suppress all visible balance information, preventing detection. This constitutes an availability/integrity impact on the observability layer of the network.

## Likelihood Explanation
Any unauthenticated HTTP client can trigger this. The `gte:T&lte:T&ne:T` variant requires no prior knowledge and always succeeds. The network-partition variant requires one prior API call to discover the current snapshot timestamp T. The attack is trivially repeatable and scriptable with no special privileges, credentials, or network position required.

## Recommendation
In `filterValidityChecks`, restrict the `TIMESTAMP` case to `basicOperators` (which excludes `ne`), consistent with how `BLOCK_NUMBER` and `SLOT` are handled:

```js
case constants.filterKeys.TIMESTAMP:
  ret = isValidTimestampParam(val) && includes(basicOperators, op);
  break;
``` [1](#0-0) 

This ensures `ne` is rejected at the validation layer before it can reach `getOptimizedTimestampRange` or the treasury query.

## Proof of Concept

**Step 1 — Always-unsatisfiable variant (no prior knowledge needed):**
```
GET /api/v1/balances?timestamp=gte:1000000000.000000000&timestamp=lte:1000000000.000000000&timestamp=ne:1000000000.000000000
```
The treasury query becomes:
```sql
WHERE account_id = <treasury>
  AND consensus_timestamp >= 1000000000000000000
  AND consensus_timestamp <= 1000000000000000000
  AND NOT consensus_timestamp = ANY (ARRAY[1000000000000000000])
```
This is always unsatisfiable. Result: HTTP 200 `{"timestamp":null,"balances":[],"links":{"next":null}}`.

**Step 2 — Network-partition variant:**
1. `GET /api/v1/balances` → note the returned `timestamp` value T.
2. `GET /api/v1/balances?timestamp=ne:T` → if T is the only snapshot in the ~2-month optimized window, the treasury row is excluded and the same empty 200 is returned.
