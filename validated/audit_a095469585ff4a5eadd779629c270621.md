Audit Report

## Title
Unbounded Historical Scan via Epoch-Zero `consensusStartTime` in gRPC `subscribeTopic`

## Summary
Any unauthenticated gRPC client can set `consensusStartTime` to Unix epoch (`seconds=0, nanos=0`) in a `ConsensusTopicQuery`. The value passes all validation checks and causes `PollingTopicMessageRetriever` to repeatedly scan the full topic history from timestamp 0, with no per-client rate limiting on the gRPC API. This enables a griefing attack that saturates DB I/O and degrades service for all users.

## Finding Description

**Code path 1 — `ConsensusController.toFilter()` accepts epoch-zero:**

`ConsensusController.toFilter()` checks `query.hasConsensusStartTime()` and calls `convertTimestamp()`. For `{seconds:0, nanos:0}`, the guard `timestamp.getSeconds() >= 9223372035L` is false, so `DomainUtils.timestampInNanosMax(timestamp)` is called and returns `0 * 1_000_000_000 + 0 = 0`. [1](#0-0) [2](#0-1) 

**Code path 2 — `TopicMessageFilter` validation passes for `startTime=0`:**

- `@Min(0)` on `startTime`: value `0` satisfies `>= 0`.
- `isValidStartTime()`: checks only `startTime <= DomainUtils.now()`. Unix epoch `0` is always before now, so this passes unconditionally.
- No minimum lookback window is enforced anywhere. [3](#0-2) 

**Code path 3 — `TopicMessageRepositoryCustomImpl.findByFilter()` scans from timestamp 0:**

The JPA criteria query builds `WHERE topic_id = ? AND consensus_timestamp >= 0 ORDER BY consensus_timestamp ASC LIMIT <pageSize>`. For a topic with millions of messages, this index range scan starts at the very beginning of chain history. [4](#0-3) 

**Code path 4 — `PollingTopicMessageRetriever` polls indefinitely in throttled mode:**

In throttled mode (`retrieve(filter, true)` called from `TopicMessageServiceImpl`), `numRepeats = Long.MAX_VALUE`, `frequency = 2s`, `maxPageSize = 1000`. The `isComplete()` check returns `true` only when the last page returned fewer than `maxPageSize` rows. For a topic with a large history, every page is full (1000 rows), so `isComplete()` never returns `true` — polling continues until the `.timeout(retrieverProperties.getTimeout(), scheduler)` fires. The default timeout is 60 seconds **between emissions**, not total duration. If each page is returned quickly, the subscription can run well beyond 60 seconds. [5](#0-4) [6](#0-5) 

**Code path 5 — No gRPC-level rate limiting:**

The only concurrency control is `maxConcurrentCallsPerConnection = 5` (per connection), configured in `NettyProperties` and applied in `GrpcConfiguration`. An attacker opens many connections (each with 5 streams), multiplying DB load linearly. The `ThrottleConfiguration` (bucket4j rate limiter) lives exclusively in the `web3` module and does not apply to the gRPC service. [7](#0-6) [8](#0-7) [9](#0-8) 

## Impact Explanation

Each malicious subscription with `startTime=0` on a large topic causes repeated paginated DB queries (1000 rows each, every 2 seconds) until the no-emission timeout fires. With N connections × 5 streams each, the attacker generates `150N` DB queries per minute, all performing large index range scans from timestamp 0. This saturates DB I/O, increases query latency for legitimate users, and can cascade into subscriber errors. The `db.statementTimeout = 10000ms` does not help because each individual page query completes well within 10 seconds — the harm is cumulative across repeated polls. [10](#0-9) 

## Likelihood Explanation

The attack requires no authentication, no special privileges, and no knowledge beyond the public gRPC proto definition. The `ConsensusTopicQuery.consensusStartTime` field is documented and standard. Any client library (Hedera Java/Go/JS SDK) can set it to epoch zero in one line. The attack is trivially repeatable and scriptable. A single attacker with a modest number of connections can sustain continuous DB pressure indefinitely by reconnecting after the timeout. [11](#0-10) 

## Recommendation

1. **Enforce a minimum `startTime` lookback window** in `TopicMessageFilter.isValidStartTime()` or in `ConsensusController.toFilter()`. For example, reject `startTime` values older than a configurable maximum lookback (e.g., 7 days before `DomainUtils.now()`). [12](#0-11) 

2. **Add per-IP or global gRPC subscription rate limiting** in `GrpcConfiguration` or via a gRPC interceptor, analogous to the `ThrottleConfiguration` in the web3 module. [13](#0-12) 

3. **Add a global cap on concurrent active subscriptions** (not just per-connection) to bound total DB load regardless of connection count.

4. **Consider requiring a `limit` or `endTime`** for historical-only queries (where `startTime` is far in the past) to prevent unbounded open-ended scans.

## Proof of Concept

```java
// Using the Hedera Java SDK or any gRPC client:
ManagedChannel channel = ManagedChannelBuilder.forAddress("mirror-node-host", 5600)
    .usePlaintext()
    .build();

ConsensusServiceGrpc.ConsensusServiceBlockingStub stub =
    ConsensusServiceGrpc.newBlockingStub(channel);

ConsensusTopicQuery query = ConsensusTopicQuery.newBuilder()
    .setTopicID(TopicID.newBuilder().setTopicNum(TARGET_TOPIC_NUM).build())
    .setConsensusStartTime(Timestamp.newBuilder().setSeconds(0).setNanos(0).build())
    // No limit set — subscription runs until timeout
    .build();

// Open 5 streams per connection, repeat across many connections
stub.subscribeTopic(query); // triggers full-history scan from timestamp 0
```

Each stream drives `PollingTopicMessageRetriever` to issue `findByFilter` queries with `consensus_timestamp >= 0` every 2 seconds, reading 1000 rows per query, for the duration of the subscription. With N connections × 5 streams, DB load scales as `150N` queries/minute. [14](#0-13) [15](#0-14)

    public Stream<TopicMessage> findByFilter(TopicMessageFilter filter) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<TopicMessage> query = cb.createQuery(TopicMessage.class);
        Root<TopicMessage> root = query.from(TopicMessage.class);

        Predicate predicate = cb.and(
                cb.equal(root.get(TOPIC_ID), filter.getTopicId()),
                cb.greaterThanOrEqualTo(root.get(CONSENSUS_TIMESTAMP), filter.getStartTime()));

        if (filter.getEndTime() != null) {
            predicate = cb.and(predicate, cb.lessThan(root.get(CONSENSUS_TIMESTAMP), filter.getEndTime()));
        }

        query = query.select(root).where(predicate).orderBy(cb.asc(root.get(CONSENSUS_TIMESTAMP)));

        TypedQuery<TopicMessage> typedQuery = entityManager.createQuery(query);
        typedQuery.setHint(HibernateHints.HINT_READ_ONLY, true);

        if (filter.hasLimit()) {
            typedQuery.setMaxResults((int) filter.getLimit());
        }

        if (filter.getLimit() != 1) {
            // only apply the hint when limit is not 1
            entityManager.createNativeQuery(TOPIC_MESSAGES_BY_ID_QUERY_HINT).executeUpdate();
        }

        return typedQuery.getResultList().stream(); // getResultStream()'s cursor doesn't work with reactive streams
    }

    private Flux<TopicMessage> poll(PollingContext context) {
        TopicMessageFilter filter = context.getFilter();
        TopicMessage last = context.getLast();
        int limit = filter.hasLimit()
                ? (int) (filter.getLimit() - context.getTotal().get())
                : Integer.MAX_VALUE;
        int pageSize = Math.min(limit, context.getMaxPageSize());
        var startTime = last != null ? last.getConsensusTimestamp() + 1 : filter.getStartTime();
        context.getPageSize().set(0L);

        var newFilter = filter.toBuilder().limit(pageSize).startTime(startTime).build();

        log.debug("Executing query: {}", newFilter);
        return Flux.fromStream(topicMessageRepository.findByFilter(newFilter));
    }
