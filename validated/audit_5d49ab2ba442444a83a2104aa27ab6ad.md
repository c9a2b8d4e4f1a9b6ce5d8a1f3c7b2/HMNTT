Audit Report

## Title
Cache Poisoning via `Optional.empty()` in `findRuntimeBytecode` Allows Stale Empty Bytecode to Be Served for Up to 1 Hour

## Summary
`ContractRepository.findRuntimeBytecode` is annotated with `@Cacheable(unless = "#result == null")` but returns `Optional<byte[]>`. `Optional.empty()` is a non-null singleton object, so the `unless` guard never fires for absent results. When a contract row exists in the database with `runtime_bytecode = NULL` — a legitimate, explicitly supported state — the first query caches `Optional.empty()` in the Caffeine contract cache (TTL: `expireAfterAccess=1h`). All subsequent queries return the stale empty result, causing the EVM layer to treat the contract as having no deployed code.

## Finding Description

**Exact location:**

`web3/src/main/java/org/hiero/mirror/web3/repository/ContractRepository.java`, lines 16–18: [1](#0-0) 

```java
@Cacheable(cacheNames = CACHE_NAME_CONTRACT, cacheManager = CACHE_MANAGER_CONTRACT, unless = "#result == null")
@Query(value = "select runtime_bytecode from contract where id = :contractId", nativeQuery = true)
Optional<byte[]> findRuntimeBytecode(final Long contractId);
```

**Root cause:** Spring's SpEL expression `#result == null` evaluates `#result` as the `Optional<byte[]>` wrapper. `Optional.empty()` is never `null` — it is a well-known singleton. The condition is always `false` for an absent result, so Spring caches it unconditionally. The developer's intent (suppress caching of absent results) is not achieved.

**How `runtime_bytecode = NULL` enters the database:**

`ContractCreateTransactionHandler.createContract()` (lines 136–145) only sets `runtimeBytecode` when a matching non-migration sidecar bytecode record is found. If no such sidecar is present (failed/reverted transaction, missing sidecar file), `runtimeBytecode` remains `null`, yet the contract is still persisted via `entityListener.onContract(contract)` at line 152: [2](#0-1) 

This is an explicitly supported state, confirmed by the `migrationBytecodeNotProcessed` test asserting `.returns(null, Contract::getRuntimeBytecode)`: [3](#0-2) 

**Cache TTL:**

The contract cache is configured with `expireAfterAccess=1h` in `CacheProperties.java` line 22: [4](#0-3) 

The `cacheManagerContract` bean wires this spec directly: [5](#0-4) 

**Downstream consumer:**

`ContractBytecodeReadableKVState.readFromDataSource()` calls `findRuntimeBytecode` and chains `.orElse(null)`, returning `null` `Bytecode` to the EVM when the cached `Optional.empty()` is served: [6](#0-5) 

## Impact Explanation
Any contract whose `runtime_bytecode` column is `NULL` at the time of first query becomes permanently invisible to the EVM layer for up to 1 hour (or indefinitely, since `expireAfterAccess` resets the TTL on every read). `eth_call`, `eth_estimateGas`, and `eth_getCode` requests to that contract will behave as if the contract has no deployed code — returning empty results or behaving as a STOP opcode — causing DeFi integrations, wallets, and tooling relying on the mirror node's JSON-RPC endpoint to receive incorrect results. If the importer later populates `runtime_bytecode` (e.g., via a migration or delayed sidecar), the cache will continue serving the stale empty result until expiry. [7](#0-6) 

## Likelihood Explanation
No privileges are required. Any user who can send a JSON-RPC request to the mirror node's web3 endpoint can trigger the caching of `Optional.empty()` for any contract that currently has `runtime_bytecode = NULL`. The precondition is a normal operational state: it occurs whenever a contract creation transaction is processed before its sidecar file arrives, or when the transaction is a migration (`sidecar.getMigration() == true`). This is not a contrived edge case — it is explicitly handled in production code and covered by tests. [8](#0-7) 

## Recommendation
Change the `unless` SpEL expression to also exclude empty Optionals:

```java
@Cacheable(
    cacheNames = CACHE_NAME_CONTRACT,
    cacheManager = CACHE_MANAGER_CONTRACT,
    unless = "#result == null || !#result.isPresent()"
)
```

This mirrors the pattern already used correctly in `AddressBookEntryRepository` via `@spelHelper.isNullOrEmpty(#result)`, which handles both null and empty collection results. [9](#0-8) 

## Proof of Concept

1. Ensure a contract row exists in the DB with `runtime_bytecode = NULL` (e.g., by processing a contract creation transaction whose sidecar is marked as migration, or whose sidecar is absent).
2. Send any `eth_call` or `eth_estimateGas` JSON-RPC request targeting that contract ID to the mirror node web3 endpoint.
3. `findRuntimeBytecode(contractId)` executes the native query, gets a row with NULL → Spring Data JPA returns `Optional.empty()`.
4. `Optional.empty()` passes the `unless = "#result == null"` guard (it is not null) and is stored in the Caffeine cache with a 1-hour access-based TTL.
5. The importer subsequently populates `runtime_bytecode` for that contract.
6. Re-send the same `eth_call` request. Observe that `ContractBytecodeReadableKVState.readFromDataSource()` returns `null` (from `.orElse(null)` on the cached `Optional.empty()`), and the EVM treats the contract as having no code.
7. The stale result persists for up to 1 hour after the last access; active polling extends the outage indefinitely.

The existing `ContractRepositoryTest.findRuntimeBytecodeFailCall` test (line 34–38) demonstrates that `Optional.empty()` is returned for a missing contract, but does not test the caching behavior for a contract row with a NULL bytecode column — the gap that allows this issue to go undetected. [10](#0-9)
